Abstract
In this project different approaches to determining the length of a longest common subsequence between
two strings A and B are introduced, some of them were implemented in Java and tested on test files with
differing compositions. It was shown that dynamic programming can be used to solve the problem in both
quadratic space and linear space. The so-called pebbles algorithm, which only uses the positions in B which
contain the same character as in A to determine the length of the LCS, is then introduced. Two different
improvements to pebbles are explored theoretically. An approach based on a similar notion, leftmost longest
subsequence, is also introduced. Different characteristics of pairs of tests files were measured, such as the
length of the LCS for the pair, the size of the alphabet, length of the files, number of matches, and more. In
testing these files with differing characteristics was used to show that the implementations performed better
under depending on certain characteristics relative to the other implementations.

Chapter 1
Preliminaries
1.1
Introduction
Solutions to the LCS problem is applied in bioinformatics to identify similarities in DNA and protein se-
quences. Finding the longest common subsequence can be used to find the difference between two texts.
In computer science this application is more widely known in version control software to find differences
between commits and similarly standardized by POSIX as the diff utility.
1.2
Definitions
Consider three strings A, B, and C, let m, n, and l denote their lengths respectively, where l ≤m ≤n.
Where the characters used are taken from the alphabet Σ = (σ1, σ2, ..., σs).
C is a subsequnce of A if all characters in C appear in A in the same order, but not necessarily consecu-
tively. A string C is a common subsequence for A and B iff C is both a subsequence of A and B. C is the
longest common subsequence (LCS) of A and B if it is a maximal common subsequence of A and B.
1.3
Project
In this project a survey of selected solutions to the LCS problem, with focus on finding the length of the
LCS, will be reviewed. This inlcludes description, theoretical analysis of their space and time complexities,
implementation, and gathering data on test strings.
Through this the strength and weaknesses of the
algorithms will be mapped out in regard to the parameters of their theoretical time complexities.
1.3.1
About examples
Some of the algorithms in the report has been provided with small examples of their execution. The same
strings have been used for these examples: A = abcdbb and B = cbacbaaba. These two strings have two
LCSs of length 4. On figure 1.1 this has been visualized. In many programming languages strings and lists
have their first element at index 0. In the examples provided here they start at index 1 , to make it easier
to read and refer to. The same principles apply either way, but when making the conversion from text to
code this should be kept in mind. In most cases it is sufficient to remember that the strings start at index
0 and end at index n −1 in the code.

Chapter 2
Dynamic Programming
In this solution each character of A is compared to each character of B. Each character can at most contribute
one to the length of the LCS. By considering the contribution of each character in relation to the previous
the length can be calculated using dynamic programming. This solution to the LCS problem is described in
[CLRS].
Figure 2.1: Finding length of the LCS of two strings A = abcdbb and B = cbacbaaba, using an m + 1 by
n + 1 table.
2.1
Tabulation
Given the two strings A of length m and B of length n, create a table with m + 1 rows and n + 1 columns.
Initially all values in the table are zero. Let ai and bj denote the i’th and j’th characters in A and B
respectively. Go through the characters of A, and for each of them compare it to every character in B. If
ai = bj then there is a match between the characters at position i of A and the position j of B. This match
can contribute to the length of the LCS if a previous character ai′ had a match with a previous character
bj′, where i′ < i and j′ < j. If ai and bj match, record the length of the LCS in (j, i) as one plus the length
recorded at (j −1, i −1). If there is no match between two characters record the maximum of (j −1, i) and
(j, i −1), which is the longest length of the LCS recorded up to this point. Once all characters of A has
been compared to all characters of B the length of the LCS will be recorded as the last element of the last
row. An example of this is shown on figure 2.1, note that the zero’th row and column only contains zeroes,

2.2
Space optimized
From the description of the tabulation method above it is clear that in order to fill out a row in the table
only the previous row and the current one is needed. Hence, a more space optimized solution using the same
approach can be achieved by utilizing two rows instead of the full table. Take two lists of length n + 1 with
all values initially set to zero. Use one of them to store the recordings for the current character in A and
the other for the previous. Make the same comparisons for A and B as before, but when all comparisons for
one character has been done switch the two lists. Now the one where the contributions of ai was recorded in
becomes the previous row to use for comparisons for ai+1, and the what was the previous before is now used
for recording. Since the comparisons are done from left to right, and the lookups in the lists use the previous
index, the recordings from the last round can simply be overwritten without being taken into consideration.
An example of this is shown on figure 2.2.

2.3
Theoretical Analysis
2.3.1
Tabulation
Both A and B needs to be stored for traversal, and since m ≤n this gives O(n). Storing the table requires
O(mn) for the m rows and the n columns. In total the space used is O(mn).
Filling out the table requires comparing each character in A with each character in B. Hence, the run
time becomes O(mn).
2.3.2
Space optimized
It is still required to store both of the strings for traversal. But now there is only need for two lists of length
n + 1, hence, the space used becomes O(n).
The runtime does not change, the exact same steps are performed in both case, so the runtime is still
O(mn).

Chapter 3
Pebbles
In this solution only the matches for each character in A are considered. By performing l iterations through
A, l ranks of dominant matches are determined, where l is the length of the LCS. On each iteration the
dominant matches are determined by comparing the previously found dominant match of this rank, with
the position of the pebble for the current character. If the pebble points to a match before the previous
dominant match, then that match is also dominant and can be registered. Then the pebble is moved so that
it points to the first match after the previously found one. Once all pebbles become inactive the algorithm
is done. This is the so-called pebbles algorithm introduced in [1].
Let k denote the position in the LCS of strings A and B, where 1 ≤k ≤l. Then a k-dominant match is
a match ai = bj, where for that k all other matches ai′ = bj′ either equation 3.1 or equation 3.2 holds [1].
I.e. all other k-dominant matches are either found between characters before in A and later in B or vice
versa. On figure 3.1 the dominant matches has been highlighted in example used in the previous section on
dynamic programming.

In this approach for every character in A all distinct matches in B are precomputed, and from these all
dominant matches are determined.
3.1
Preprocessing
3.1.1
Positions in B of characters in A
Let ci be the i’th character in the set of unique characters occurring in A. Let pci denote the list of all
positions in B that match ci. The goal is to preprocess a dictionary using the unique characters of A as
keys, and for each of those store a list of with all the positions that character occurs at in B, such a list will
be referred to as a p list, and the p list for a character c as pc. As such pc will contain all the matches for c
from A in B. Starting with an empty dictionary, go through A for each character if it has not been added
to the dictionary add it with an empty list. Then go through B and for each character bi lookup bi in the
dictionary, if there is a list add i to the list. In each list the last value to be appended is n + 1. On figure
3.2 the p lists from the previous example has been prepared.
Figure 3.2: Occurrence lists for the characters a, b, c, and d in the string B given as cbacbaaba. Notice that
while the character d has no occurrences in B it has still been given a p list containing n + 1 = 10.
3.1.2
Lookup table
Create a table that allows one to look up the position of a character bi in the list pbi. Let vi denote the i’th
value in a p list. Go through the values in each p list, at index vi of the lookup table insert i. The result is a
list where the indeces correspond to the characters in B that occur at that index, and the value corresponds
to how many times that character has been seen in B at that index. This corresponds to the index in the p
list for that character, where this character occurs at.
Figure 3.3: For a character bi in B, its position in pbi is saved at position i of the lookup table. E.g. consider
b5 = b, the value at index 5 of the lookup table is 2, going to pb the value at index 2 is 5.

3.1.3
Pebbles
For each character in B one pebble is used to keep track of the next possible position for a dominant match
for that character. As such all pebbles start at the first position in the corresponding p list, and are then
advanced as the dominant matches are determined.
Figure 3.4: p lists with the corresponding pebbles for each character in A at their initial position.
Let Pebbles be a list of length m containing integers to keep track of the next possible match in B for
the characters in A. For each ai the i’th index of Pebbles contains the index in the p list of ai for the next
occurrence of ai in B to be checked. Initially all are set to one, corresponding to checking the first occurrence
in B of each character in A. A pebble is inactive if the index it points to has the value n + 1, corresponding
to there not being any matches left in B for that character.
3.1.4
Ranks of dominant matches
Storing the dominant matches found for each rank in a list, such that once the algorithm terminates there
is l ranks, where each rank k stores all the k-dominant matches. I.e. a list of lists containing pairs on the
form (i, j), denoting the dominant match ai = bj.
3.2
Finding dominant matches
Let t denote the index of the previously found k-dominant match in B and let T denote the index of the
current k-dominant match in B, initially both are set to n + 1. Find the k-dominant matches by starting at
ai, where i = k. If the pebble for ai is active proceed, otherwise go to the next character in A. For the next
match ai = bj check if j < t: Retrieve the p list for ai, find the current position of the pebble of ai using
Pebbles to find j in the p list. If j < t store the pair (i, j) as a k-dominant match in the list for the k’th
level and update T to j. If the pebble was active and ai = bt the pebble is advanced to the next occurrence
bj where j > t. Since ai = bt both use the same p list, so advancing the pebble of ai past that of bt is done
by going to the lookup table and finding the position of bt’s pebble by looking up index t, adding one to
its position places the pebble of ai at the next occurrence after bt. Note that when t = n + 1 the pebble of
ai simply becomes inactive. Otherwise if ai ̸= bt and the pebble is not currently at an index that lies after
t, the pebble is advanced until it does. Once all the k-dominant matches have been found the process is
repeated until all l levels of k-dominant matches have been found. The length of the LCS then corresponds
to the number of levels of dominant matches.

3.3
Optimizing for space
For determining the length of the LCS only the number of ranks of dominant matches is needed. By omitting
the storage of dominant matches, and instead just counting the ranks the same result can be achieved with
less space.
For this to work the algorithm has to proceed as before advancing the pebbles in the same
manner. Whenever a dominant match would be registered, if it is the first dominant match for that rank
it is counted as one representing that entire rank. In the end the number of ranks has been counted, which
will be the same number as the length of the LCS.
3.4
Theoretical Analysis
Space
Storing the p lists requires storing O(n) integers in total. The lookup table requires storing an integer for
each position in B, for a total of O(n). Pebbles has m indices, taking up O(m) space. Storing the levels of
dominant matches requires storing a total of O(d), where d denotes the total number of dominant matches.
Storing A and B is also required, taking up O(m) and O(n) respectively. All in all O(n+d) space is required.
Time
Preprocessing the p lists takes O(m) to determine all the unique characters in A. Retrieving all of their
positions in B then takes O(n). Preprocessing of the pebbles requires initialising O(m) pebbles, one for each
character in A. Going through all of the p lists takes O(n). Hence, all of the preprocessing takes O(n).
Finding all k-dominant matches for k = {1, 2, ..., l}. In order to determine all the dominant matches, l
iterations are required, at each iteration m pebbles are checked. Checking a pebble requires some constant
time operations, and retrieving the p list from the dictionary. Retrieval takes expected constant time. Let r
denote the number of distinct matches between A and B, then at most the pebbles can be advanced r times.
Finding the dominant matches takes expected O(lm + r).
The total running time with preprocessing is then expected O(lm + r + n).

Successor table
This solution aims to improve the previous solution by allowing constant time advancement of pebbles. A
table of successor positions in B for the unique characters in A is created as a part of the preprocessing.
Whenever a pebble is to be advanced past the previous found dominant match instead of going through
the p list, accessing the successor positions gives the next position for the pebble to be advanced to. This
improvement to the pebbles solution is introduced in [1].
4.1
Preprocessing - Successor lists
Start by preprocessing in the same manner as in section 3.1. Go through each p list creating a successor
list of length n by doing the following: Starting at the first value in a p list insert that value into the list
through the index corresponding to the value, then repeat for the remaining values of the p list. On figure
Figure 3.4 an example of successor lists are shown with the p lists.
Figure 4.1: Example of the successor lists created from the p lists from figure 3.4.
4.2
Advancing pebbles
The procedure for the algorithm is the same as in section 3.2, but instead of advancing the pebble until the
value pointed to in the p list is greater than t, the successor table is used in combination with the lookup
16
==PAGE:18==
table to directly access the value in p. Figure 4.2 gives examples of the two cases for advancing pebbles using
this new approach.
If the previous dominant match was for the same character as the current one, then the advancement of
the pebble is done as previously, accessing the lookup table to get the position in the p list for that match
and adding one. However, if they are not the same character then the successor list for the character is used.
Given that the previous match was bt, then looking up t in the successor list for the current character gives
the successor of t. This change in advancing pebbles in the case where the current character is different from
that of the previously registered dominant match is the only change to the algorithm.
Figure 4.2: Advancing pebbles using successor table: String B is shown at the bottom with the previous
match highlighted in red and the closest occurrence of ai to the left and right of the previous match highlighted
in purple. The t’th index of the successor list of ai will give the successor of ai. To the left the general case
is shown, instead of having to traverse pc until the value larger than t is found, looking up t in the successor
list gives the index in pc that holds the index of the successor position of c at t in B. To the right the
previous match found was for a5 = b, i.e. the previous match was the same character as the one currently
being checked, hence, the pebble has to be advanced to the next value in the successor list.
4.3
Theoretical Analysis
In the pebble solution the total space usage was O(n+d), the same data structures are used with the addition
of the successor table. The successor table has exactly s rows and for each row it has n indices, so the space
usage is O(sn). Hence, the total space usage of this solution is O(sn + d)
All of the preprocessing from pebbles is done, which required O(n), with the addition of the preprocessing
of the successor matrix, which takes O(sn), since all n indices has to be filled out for every unique character
in Σ. Hence, the preprocessing is now dominated by this term.
Finding all k-dominant matches still requires performing l iterations, and at most considering all m
pebbles. Recall that for the previous solution O(r) advancements of pebbles was needed. Using the successor
table at most each of the m pebble will require exactly one advancement to get to the value larger than the
last match found. Hence, finding all of the k-dominant matches requires at most O(lm) advancements. The
total running time with preprocessing is then O(lm + sn)

Chapter 5
log-Close
In this solution a different approach to improving the advancements of pebbles is taken. By enumerating the
characters from the alphabet a single table of size n containing the successors for each character is prepared,
such that each successor for a character is s indices apart, where s denotes the size of the alphabet. Given a
character and an index the upper and lower index that the successor occurs at in the p list can be determined
in constant time. Once the the interval is found a search is performed in it to find the successor. This is
different improvement to the pebbles solution introduced in [1].
5.1
Preprocessing
The preprocessing is similar as for pebbles, an additional table Close of size n is prepared to hold the
successors. Enumerate the unique characters that occur in both A and B from 0 to s −1. The characters
that occur in both strings will be those characters for which there initially are active pebbles. Hence, finding
these characters can be integrated into the preprocessing of the p lists. Let j denote the indices of Close,
then to fill out Close start at the first index and calculate c = j mod s, where c denotes the number for
the character from the enumeration. Taking the p list for the character represented by c, j in Close is set
to the leftmost match equal or greater than j. Now keep moving s indices ahead in Close and each time
inserting the match from the p list that is equal or greater than the index, repeating this process until it is
not possible to move s indices ahead. Moving to the second index of Close, finding the next character and
repeating the process until all positions has been filled in. An example of the filled out Close is shown on
figure 5.1.
Figure 5.1: The table Close. The colors correspond to the characters a = orange, b = red, c = blue. In this
example s = 3 and the characters have been enumerated as a = 0, b = 1, c = 2. In this example Close has
its first index at one, when the first index for an a in Close is calculated zero is returned, here zero is set to
three since there is not index zero. In implementation this would be different, but the same principles would
apply.

==PAGE:20==
5.2
Description
The procedure is the same as described in section 3.2 for Pebbles, with a different approach to advancing
pebbles in general. The process of advancing a pebble now requires accessing Close based on the previously
found dominant match, t, then accessing the lookup table to find the two matches in the p list that the
successor lies between.
If there is a successor of t it can now be found as follows: First the position in the p list of the match is
determined by accessing j′ in Close, where j′ = ⌊t/s⌋s + c. c is determined from the current character in A
and the enumeration of characters from earlier. j′ is one end of the interval that the successor is located in
within the p list. Consider the following three cases for j′ in relation to t:
• Case 1: [j′ = t] The closest match for the character was at t, hence the next character from the p list
is the next one.
• Case 2: [j′ < t] The successor lies between the position of j′ and j′ + s in the p list.
• Case 3: [j′ > t:] The successor lies between the position of j′ and j′ −s in the p list.
Once j′ is found the position of the p list can be found by taking the value at j′ in Close and going to
that index in the lookup table which gives the position of that match in the p list. From here the three cases
from above are applied. In the two latter cases a binary search can be performed to determined the successor
in the interval between the two end points. On figure 5.2 examples of determining j′ and the intervals are
shown.
Figure 5.2: Two examples of looking up in Close using j′ = ⌊t/s⌋s + c. On the left a lookup has been made
for a with t = 7, the result is 6. The case is j′ < t, so j′ + s is taken as the other end of the interval. Then
the two indices are accessed in the lookup table to find the locations in the p list, and from that the interval
to search in the p list is found. The case where j′ > t the procedure is similar, but j′ −s is taken as the
other end of the interval. On the right a lookup for b on 7 has been made. The case is now j′ = t, if the
match in Close is larger than t it can be used directly, if not, the p list can be accessed as before.
5.3
Theoretical Analysis
The space used by the new data structure is O(n), hence, in the asymptotic notation the space usage is not
changed and is still O(n + d).
Preprocessing Close requires going through all of the p lists which have a total length of O(n). Finding
the dominant matches requires l iterations, at most m pebbles have to be advanced. Advancing a pebble
now requires at most two constant time lookups and performing a binary search between the two end points.
From the calculation of j′ the nearest index for the character with the value c is found, between the two
points there are at most s matches. Hence, the binary search takes O(log s) . The total running time is then
O(lm log s + n).

Leftmost Longest Subsequence
This algorithm computes the length of the LCS by attempting to assemble the leftmost longest subsequence
(LLS) by going through A and replacing previously registered matches in the LLS with matches found more
to the left in B. Once all matches in the LLS, it may be extended with the leftmost match in the p list that
is greater than the last match in the LLS.
6.1
Preprocessing
To quickly access the matches for a character in A the preprocessing of p lists from the pebbles implementation
will be used again. However, since this algorithm does not involve advancing pebbles the last value of n + 1
is not needed. Hence, the p lists here refers to a list associated with a character from A and all the positions
that character occurs at in B. Besides the p lists only two lists are needed, initially containing a zero. One
list to store the previous LLS used for reading, and another list to store the LLS of the current iteration
used for writing.
6.2
Description
The goal is to end up with a sequence of ascending matches, where the length of the sequence is one longer
than the length of the LCS. The LLS is built continuously by taking each character of A and finding if any
of the matches recorded in the LLS may be replaced by a match from the p list that lies more to the left,
and finally if the LLS can be extended by a match. Any match can only occur once in the LLS, representing
that a character at a position in B can only appear once in the LCS. Going through a p list starting from the
leftmost match determine its predecessor in the LLS, once the predecessor is found, if the match is already
registered move to the next match in the p list. Otherwise if the match from the p list is smaller than the
one registered after the predecessor, then this new match from the p list lies more to the left and replaces
the old match. In this case the next match to be examined from the p list needs to lie after the old match
that was just replaced. Note that the replacement must happen after all the matches from the p list has
been checked. Otherwise a newly replaced match may be examined in the same round, this error would be
the same as reading a character twice from A when it only occurred once. On figure ?? an example of the
correct replacement and the erronous replacement are given. A similar principle as from the space optimized
dynamic programming solution from section 2.2 is used, so that the matches that would replace in the LLS
does not have to be stored until the end of each step.
Going through A recover the p list for the current character, starting at the first match in the p list
determine if it has a predecessor in the LLS. If it has one, check if the match after is the one currently being
examined from the p list, if it is not the match is written to the index after the predecessor. The next match
taken from the p list has to be greater greater than the one that was just overwritten. Once found repeat
the process of finding a predecessor and inserting after. Once there are no more matches in the p list, or the

6.3
Finding the matches to replace
This section focuses on different approaches to finding the two matches to insert into the LLS, and provide
a theoretical time analysis based on the given approach. These approaches leverage that both the LLS and
any p list are sorted.
6.3.1
Merge-like approach
Based on the merge from merge-sort this approach aims to find the two matches by going through the LLS
while potentially moving to the next match in the p list. Below the algorithm using the merge-like apporach
is described in bullet points. An example of the merge approach for a single p list is given on figure 6.6:
Figure 6.6: Finding the first match to replace in the LLS using the merge-like approach. Starting at the first
match in the p list, go through LLS until the first character greater than or equal to the match is found.
The match three has been registered in the LLS previously, hence, the next match in the p list is checked.
Similarly it is found that 6 was also registered in the LLS. When 10 is checked the first match greater or
equal is 14, hence, it should be replaced in the LLs for writing (not shown). Since the previous match to be
replaced was 14, the next match checked from the p list needs to be greater than 14. Hence, the next match
picked from the p list is 16.
1. Start at the first character in A.
2. Retrieve the p list for the current character in A, let a denote the current character in A.
3. Let i and j denote the indices in p list for a (pa), and LLS, both initially set to 0.
• pa[i] > LLS[j]: The match in the LLS is more to the left than the one from pa, move to the next
match in the LLS (increment j by one).
• pa[i] < LLS[j]:
– The match from pa is more to the left than the one in the LLS, save pa[i] to be stored at
LLS[j]. E.g. tuple (i, j).
– Move to the next match in pa which is larger than LLS[j].
– Move to next match in LLS (increment j by one).
• pa[i] = LLS[j]: Then the LLS[j] already contains this match and cannot be pushed further left.
Move to the next match in both pa and LLS (increment j and i by one).
• i > pa.length: The current character in A cannot alter the LLS further.
• j > LLS.length: Take last match in LLS and proceed from pa[i] through the pa list until first
value larger is encountered or the pa list runs out of values.
4. Take the next character in A and go to step 2, repeat until all characters in A has been checked.

Theoretical Analysis
Going through the full LLS takes at most O(l), since the LLS can never be longer than the LCS. At worst
all of the matches in the p lists are checked, and at most there can be r matches in a p list. Hence in the
worst case finding one of the matches takes O(l + r). This has to be done for every character in A, hence
the total running time is:
Merge approach: O(m(l + r))

6.3.2
Binary Search
In this approach a binary search is used to search in the LLS for a match in the p list, and when a replacement
in the LLS would happen it is also used to search for the next match in the p list greater than the match just
replaced. The binary searches are used when the next match to check is not the neighbour of the previously
checked match, in which case the neighbour is just taken directly.
1. Start at the first character in A
2. Retrieve the p list for the current character in A
3. Let i denote the i’th match in pa and let j denote the j’th match in LLS. Initially i = 1 and j = 2.
• The first element of LLS is skipped, as this is always the smallest and does not occur in any p
lists, since it represents the zero’th position.
4. If i < pa.length and j < LLS.length proceed to next step, otherwise go to step 6.
5. Take pa[i] and search for it in LLS using BS. Let m denote the index of the result in LLS.
• LLS[m] > pa[i]: Save pa[i] to replace LLS[m]. Perform a binary search for LLS[m] in pa, update
the i to this index, if the exact value was found increment i. Increment j and go to step 4.
– Since LLS[m] > pa[i], LLS[m] will be replaced and the match from pa to consider has to be
larger than what was replaced. If pa[m] = LLS[m], increment i, taking the neighbour.
• LLS[m] = pa[i]: Already contained, increment i and increment j and go to step 4.
– Searching for LLS[m] in pa to find the next match to search for in LLS is not necessary.
Since LLS[m] = pa[i], pa[i+1] will be the smallest match larger than the previous one found,
and it can be used directly in the next search in the LLS.
• LLS[m] < pa[i]: increment j, go to step 4.
– If LLS[m] < pa[i] either m is at the end of LLS and pa[i] is larger than all elements in LLS,
in which case incrementing j terminates the loop in step 4. Or there was an element larger
than pa[i] which forced the search to go to the left of it, and all elements in that part was
smaller than pa[i]. Hence, the neighbour of LLS[m] should be greater than pa[i], which will
then be used for the next search in LLS.
6. Take the last match in LLS, go through pa and try to find a match greater than it to extend LLS.
7. Move to next character in A, if possible, and go to step 2.

Theoretical Analysis
Given that the only thing that has been changed is how the matches are found in the two list, t will still be
the case that the maximum length of LLS is l and the maximum length of a p list is r. Performing a binary
search in the LLS takes O(log l), and similarly it would take O(log r) for a search in the p list. In the worst
case each match from the p list replace a match in the LLS. This would require performing a binary search
in LLS for each match in the p list, and when the match to be replaced was found in LLS, a binary search
would be made back in the p list. Hence, at worst the running time is:
Binary approach: O(m(l log(r) + r log(l)))
(6.2)
6.3.3
Exponential Search
In this approach exponential search is used to look ahead in the list to determine the interval in which the
next match appears. Let d denote the number of indices ahead in the list that are being looked and let i
denote an index in a list. For searching in LLS having checked the i’th match in LLS previously, by looking
at index i + d, if is larger than the match from the p list then the match from the p list is either between i
and i + d or there is a match it can replace. If the match at i + d is smaller, then there is no need to look
at the maches in that interval, and i can be set to i + d, d can be double, and the procedure repeats. The
approach for looking up in the p is similar.
6.3.4
Exponential Searches
1. Start at the first character in A.
2. Retrieve the p list for the current character in A, let a denote the current character in A.
3. Let i and j denote the indices in p list for a (pa), and LLS, both set to 0.
4. Comparing pa[i] to LLS[d], Let d be an integer, initially 2:
• LLS[d] > pa[i]: Between the indices j and d in LLS there is a match larger than pa[i], which also
has a neighbor that is smaller or equal to pa[i]. Iterate through LLS to determine which, save
the last index checked.
(a) If equal, then pa[i] is already recorded, move to next match in pa and set j to the last checked
index plus one. Go to step 4.
(b) If smaller, then there are two matches next to each other in LLS, where the one is smaller
than pa[i] and the other is larger. The larger one should be overwritten by pa[i], save index
and value for overwriting later. Set j to the last index checked plus one, go to step 4.
• LLS[d] < pa[i]: There are no matches between indeces j and d in LLS smaller than pa[i], update
j to d, then double d and go to step 4.
• j > LLS.length: No more matches to compare to in the LLS. Take the last one in LLS and go
through pa from i to find a match to append to LLS. Then go to step 5
• i > pa.length: There are no more matches in pa that could alter LLS. Go to step 5
5. Go to the next character in A, if there are no more characters the algorithm is done. Otherwise, go to
step 2.

Chapter 8
Closing
8.1
Further work
It would have been interesting to test out the implementations of the remaining solutions that were introduced
in the report. Pebbles may have benefited greatly from either the successor or Close table. Similarly the
exponential search approach for LLS could have made a small enhancement to merge-like, but it could then
have been further combined with binary searches. Through testing determining a distance where binary
searches are better than merge.
In regards to the implementation improvements could be made to reduce some constant time factors.
Since p lists are unchanged once preprocessed, using primitive arrays of integers would come with a smaller
memory overhead. This could be achieved by going through B once to count up occurrences of each character,
initializing arrays, and on a second run through insert the indices.
For LLS instead of copying a list at the end of each round it could have been interesting to test if using
two constant size arrays of length m would be worthwhile and switching between them, this could work since
l ≤m. Switching between the lists could also be tested as a standalone alternative to copying the LLS after
each character in A had been checked.
On the binary searches the end of the interval is set to the end of the list that is being looked into.
However, this means that when the value that is found is close to the start of the interval many iterations
are needed to reach that value. One could consider either using the last value recorded in the LLS as an
upper bound, or using the distance from the lower bound to the index of the last value in the LLS when
searching in the LLS.
In testing LLS one thing that stood out was that often it may be the case that many of the matches in
a p list was already recorded, essentially leading to spending time figuring out where in the p list the first
match that was not already recorded was. Hence, it could be interesting to have a way to determine where
in the p list the last one was recorded. This is not as straightforward as it seems, since it is possible to have
gaps between the matches that are recorded from a p list. In addition to this there should be some way
of determining where this match is recorded. This would eliminate unnecessary traversal of both p list and
LLS.
35
==PAGE:37==
8.2
Conclusion
The solution with using dynamic programming performed well in situations where the two strings being
compared have much in common, i.e. a long LCS and many matches. Preprocessing to find the locations
that results in matches between two characters into p lists is useful for reducing time, if the alphabet is not
small with many matches. If the alphabet is small with many matches, such as DNA, the number of matches
to check is only reduced by a small fraction compared to checking all positions in B. However, if the alphabet
is large and the occurrences in B are somewhat evenly distributed among them, then more time is saved,
which is the case for the custom files. For this reason both pebble and both LLS solution performs well in
testing with the custom files. The LLS solution with binary searches performs poorly when the matches to
be searched for appears closely together, this is seen in is relatively poor performance for DNA files.
