Abstract
This thesis describes and examines a relatively new computation model ﬁrst introduced by Ben-
der et al in 2018 based on combining the streaming model and the external memory (I/O) model.
It discusses some challenges inherent to and techniques available in this model, including dictio-
nary queries and query batching. A new model for the family of on-line event detection problems
is introduced, and the time-stretch ﬁlter is described and analyzed in this model in more de-
tail and more exhaustively than in its original introduction. Several extensions, including more
predictable behavior in case of very high frequency items, weighted items, and string preﬁx
matching are introduced. Each of these data structures can for time-stretch α > 0 be main-
tained in O((1/α + 1) logM/B(T/B)/B) amortized I/Os per item/symbol for window size T, or
O((1/α + 1) logN/B(N/B)/B) amortized I/Os per item/symbol without windowing. To facili-
tate preﬁx matching, operations on a certain serialization of tries are introduced and analyzed.
Finally, a novel algorithm for an on-line version of the classic union–ﬁnd/connectivity problem
is introduced and analyzed. This algorithm in particular supports solving the on-line union–
ﬁnd/connectivity problem for query delay Θ(V ) with the same I/O-complexity as the oﬀ-line
minimal spanning tree problem.

Introduction
This thesis examines the external memory streaming model introduced in [5], where a large
stream of items or operations is processed using the external memory model. In particular, it
examines event detection problems, where we want to detect high-frequency items, and an on-
line version of the union–ﬁnd problem. For both kinds of problems, we will be achieving better
I/O performance by allowing queries to be answered after a bounded delay.
This model and these problems have applications in situations where a high-throughput stream
of items or operations must be processed and handled on the ﬂy, especially where the latency
of performing an external memory transfer for each item is unacceptable. Possible applications
thus include anomaly detection (whether of networked communication or of devices, such as in
industrial settings) and processing of social data.
1.1
Structure of the report
This report is divided into three main parts.
• Section 2 describes the algorithm examined in this thesis. It also discusses some of the
challenges inherent to working in this model, and some of the techniques available.
• Section 3 presents the family of problems related to doing on-line event detection of streams
of items, and gives an adapted description of the time-stretch ﬁlter from [5]. It then de-
scribes and examines some extensions to this data structure, including more predictable
handling of very heavy events, and a version of the time-stretch ﬁlter for strings with
preﬁx matching.
• Section 4 describes the well-known union–ﬁnd problem, albeit in a way adapted to the
particular approach used in the section, and introduces a novel on-line external memory
union–ﬁnd algorithm with delayed responses based on a batched algorithm from [1].
Finally, Section 5 summarizes the main ﬁndings of this project.
1.2
Contributions
The primary contributions of this thesis are the following:
• An adapted and more exhaustive description of the time-stretch ﬁlter data structure from
[5].
• An extension of the time-stretch ﬁlter with more predictable handling of very heavy events.
• Algorithms for merging and subtracting serialized tries.
• An extension of the time-stretch ﬁlter to allow string events with preﬁx matching and
diﬀerent thresholds for diﬀerent preﬁx lengths.
• A novel algorithm based on an algorithm from [1] for answering union–ﬁnd problems
on-line with ﬁxed delay for responses.

External streaming model
In the streaming model [3], an input stream S consisting of N items is processed sequentially
to produce some result, either in real time or after the fact, maintaining only O(polylog N) bits
of memory. In other words, the streaming model is focused on processing S with less memory
than it takes to store S.
In the external streaming model introduced in [5], on the other hand, we are still interested in
processing an input stream S of N items, but instead of strictly limiting the available memory,
we will measure external memory use as in the I/O model of [2]. In other words, we assume an
internal memory of M words and an unlimited external memory with memory transfer oper-
ations (I/Os) that move contiguous blocks of B words at a time between internal and external
memory. Based on the assumption that memory is much slower than computation, we will only
count the number of I/Os used by an algorithm and thus consider computation to be free.
As argued in [5], when processing high-throughput streams, data ingestion is not the bottleneck.
Rather, several write-optimized data structures exist, where writes are batched such that they
required amortized less than one I/O per update, but generally require at least one I/O per query.
In this thesis, however, we are in particular interested in trying to solve certain problems with a
possible less-than-one amortized I/Os per operation, even for queries.
While problems in this model are very similar to dynamic problems in the ordinary I/O model,
a major diﬀerence between the two is, that in the external streaming model we will be allowing
queries to be answered, and events to be reported, with some bounded delay. This is the principal
reason why it is possible to answer queries in less-than-one amortized I/Os.
We will generally assume N > M, as otherwise the problem can be solved without using external
memory at all.
2.1
Parallel scanning and merging
One of the most fundamental building blocks used in this thesis is that of parallel scanning of
two or more arrays. It is a well known technique and the deﬁning building block of merge-based
sorting algorithms, but I will nonetheless deﬁne the technique semi-formally here.
Let R be some totally ordered set of values, e.g., integers or bit strings. Then consider two sets
A1 and A2 equipped with sort-key functions r1 : A1 →R and r2 : A2 →R and stored as arrays
sorted according to these functions. If a1 ∈A1 and a2 ∈A2 satisfy r1(a1) = r2(a2), then we
will call (a1, a2) a matching pair. If a1 ∈A1 for which no a2 ∈A2 satisﬁes r2(a2) = r1(a1),
then (a1, ⊥) is the matching pair for a1, and similarly (⊥, a2) is the matching pair in the opposite
situation. This is illustrated in Figure 1.
Then it is possible to construct a new array by applying some function f : (A1 ∪{⊥}) × (A2 ∪
{⊥}) →C∪{⊥} (with a returned ⊥indicating an element should be left out of the new array) to
the matching pairs of A1 and A2 in O((|A1|+|A2|)/B) I/Os. This is done by simply maintaining
an index i1 into A1, an index i2 into A2, and a pointer to the end of the array being constructed.
Then at each step, if r1(A1[i1]) = r2(A2[i2]), then f(a1, a2) is written to the new array (unless
⊥) and i1 and i2 are both incremented. If, on the other hand, r1(a1) < r2(a2), then f(a1, ⊥) is
written and only i1 is incremented, and otherwise f(⊥, a2) is written and only i2 is incremented.
This technique is also readily extensible to more than two input arrays, as long as one block of
memory for each array plus one block of memory for the resulting array can ﬁt in main memory.
This means that it in general is only possible to merge up to M/B −1 arrays at a time.

2.2
Dictionaries
Many data structures depends on some sort of dictionary look-up to function, whether this be
trough direct indexing in large arrays, or through binary search trees or hash maps. The simple
problem of looking up entries in a large enough section of memory, however, proves to be more
troublesome to do eﬃciently in the external streaming model. Given a dictionary of N items
stored in Θ(N) words (i.e., without signiﬁcant duplication or compression), at most a ratio of
O(M/N) of it can be loaded into internal memory at any given time. In the worst case, a query
will thus require at least one I/O.
While several write-optimized dictionary solutions exist, like Bε-trees [7], cache-oblivious looka-
head arrays (COLAs) [6], and external hash maps [10], these are still too slow to query, if a query
is needed for every item of the stream.

3
Event detection
In the event-detection family of problems we are interested in processing a stream of items over
some universe of types U and reporting those item types whose count reaches some threshold.
We will a little atypically model the stream S as a set of stream items. Each stream item is a tuple
(t, s) consisting of a timestamp t and a type s ∈U. The timestamps are unique and sequential.
Thus the sequence of types s1, s2, . . . , sN gives the stream S = {(1, s1), (2, s2), . . . , (N, sN)}.
While this scheme can model inﬁnite streams, doing so in our external streaming model would
imply inﬁnite bit-width words, so this is not particularly useful. It is, however, a useful limitation
to assume that the exact length of the stream is not known at processing time, even if an upper
bound is implied by the bit-width of the pointers used.
Let S(t1, t2] denote {(t, s) ∈S | t1 < t ≤t2}, that is, the substream of items strictly after time
t1 up to and including time t2.
Deﬁnition 3.1. A (τ, T)-event is a tuple (t, s) for integer t and s ∈U, such that S(t −T, t]
contains at least τ items of type s. A τ-event is a (τ, T)-event with T = N.
In other words, each (τ, T)-event is a witness for a set of τ items of the same type within a
window of T items, while each τ-event does the same but without the windowed restriction.
The objective for the event-detection problems we will be considering in these section is then to
process the stream and report (τ, T)-events. To make the association between items and (τ, T)-
events more concrete, we will use the following deﬁnition as well:
Deﬁnition 3.2. For an item type s ∈U and integers τ, T ≥0, an (s, τ, T)-substream is a
substream S(t1, t2] for integers 0 ≤t2 −t1 ≤T such that:
(i) S(t1, t2] contains exactly τ items of type s; and
(ii) S(t1, t2] is minimal in the sense that any strict substream S(t′

fewer than τ items of type s.
A (s, τ)-substream is a (s, τ, T)-substream with T = N.
When t2 ̸= t1 (which it is for τ ̸= 0), condition (ii) is equivalent to the boundary items S[t1 + 1]
and S[t2] both having type s.

Problem 3.1 (Windowed time-stretch event detection with overlap). Given integers τ > 0 and
T ≥0, and real number α > 0, process a stream S sequentially, and for each (s, τ, T)-substream
S(t1, t2], report at some time t ≥t2 a (τ, T)-event (t, s) such that t −t2 < α(t2 −t1).
Problem 3.2 (On-line time-stretch event detection with overlap). Given integer τ > 0 and real
number α > 0, process a stream S sequentially, and for each (s, τ)-substream S(t1, t2], report
at some time t ≥t2 a τ-event (t, s) such that t −t2 < α(t2 −t1).
Note that for these versions of the problem, the same item type may be reported multiple times.
We will look into strategies to mitigate this in Section 3.2, including how the original article
handles it, after presenting the basic data structure.
3.1
Time-stretch ﬁlter
The article [5] introduces a few diﬀerent event detection data structures (“ﬁlters”) with the same
general structure but diﬀerent properties.
Each ﬁlter consists of a sequence of levels C0, C1, . . . , CL−1 of increasing size, where only C0 ﬁts
in main memory. Events are ﬁrst inserted into C0, then later ﬂushed from C0 to C1, from C1 to C2,
and so on. As a consequence, no I/O operations are needed until a ﬂush is triggered. The exact
size, structure, and ﬂushing behavior diﬀers between ﬁlters.
We will in this thesis describe and examine the time-stretch ﬁlter anew, in diﬀerent terms and
with diﬀerent parameters than in the original formulation in [5]. In particular, we will here prefer
the use of concrete, integer parameters, with a straight-forward structural interpretation, to the
more abstract real parameters used in [5].
Thus, the time-stretch ﬁlter takes as parameters integers T, τ, q > 0. We will additionally use
derived integer parameters L, m > 0 and real constant r ≥2, all of which we will determine
later. The integer T gives a lower bound for the number of stream items that can ﬁt in the
ﬁlter, τ gives the reporting threshold, q gives the number of subdivisions of each level (as will be
explained shortly), L gives the number of levels, and m is a derived parameter, that ensures the
ﬁrst level C0 ﬁts in main memory.
3.1.1
Structure and operation
The time-stretch ﬁlter then consists of L levels C0, . . . , CL−1 of increasing capacity stored as
arrays of stream items sorted by type. Stream items are inserted into C0, and when a level becomes
full, some of its items are transferred to the next level. The proportion of how many items
get transferred at a time is what determines our time-stretch guarantee. For simplicity, we will
pretend that there is an extra level CL. Moving items to this ﬁctitious level then amounts to
discarding them. Note that this is the only way items can be deleted from the ﬁlter.
Let ki for i = 0, . . . , L−1, L be the number of stream items transferred at a time to Ci, and let the
capacity of each level be |Ci| = qki+1. The general structure is illustrated in Figure 3. Since single
items are inserted into C0, we set k0 = 1. Then we will have each level receive roughly r times
more stream items than the previous level. To have each level evenly hold an integer number
of transfers from the previous level, however, we will set ki+1 = ⌈rim/ki⌉ki. We thus (since
m ≥1, r ≥2, and thus rim/ki ≥1) have rim ≤ki+1 < 2rim and hence ki+1 = Θ(rim).
Thus |Ci| = Θ(qrim). We then ﬁx a positive integer m = Θ(M/q) such that C0 ﬁts in main
memory, and get |Ci| = Θ(riM). Note that this restricts the possible value of q in relation to
M, as suﬃciently large q can make it impossible to ﬁnd a suitable m. This is discussed further
in Section 3.1.4.

or greater. Otherwise, let k be the index such that C0, . . . , Ck−1 are all full, and Ck is not. Then

1. Merge C0, . . . , Ck−1, Ck into sorted array R.
2. Scan through R and count each type. If a type has count τ or greater, report it.
3. For i = k −1, . . . , 0, scan through Ci to move the ki+1 oldest items to Ci+1.
After this, C0 is no longer full and the next stream item can be inserted.
3.1.2
Correctness
Lemma 3.3. For s ∈U, given an (s, τ, T)-substream S(t1, t2], at some time t a (τ, T)-event (t, s)
is reported, where t2 ≤t < t2 + (t2 −t1)/(q −1).
Proof. When a ﬂush of levels C0, . . . , Ci−1 occurs, all of these levels are full. They thus together
contain |C0| + · · · + |Ci−1| stream items. After a ﬂush, only the ki items transferred from Ci−1
leave these levels, so the next ﬂush of C0, . . . , Ci−1 occurs after ki new items are inserted. Thus,
after being ﬁlled for the ﬁrst time, each level Ci−1 experiences a ﬂush every ki inserted stream
items, meaning the items in C0, . . . , Ci−1, Ci are counted every ki inserted stream items. At any
time, the maximum time before an item in Ci is included in a count is thus ki.
Since ki+1 stream items are removed from level Ci per ﬂush, and Ci has capacity qki+1, a stream
item inserted into Ci experiences q ﬂushes before being moved to Ci+1. Since the ﬁrst ﬂush may
occur immediately after its insertion into Ci, at least (q −1)ki+1 further items must be inserted
before an item is moved from Ci to Ci+1. Hence, for a newly inserted item to end up in Ci, at least
(q −1)k1 + (q −1)k2 + · · · + (q −1)ki ≥(q −1)ki further insertions are needed.
Since t2 −t1 ≤T, and the ﬁlter contains at least T stream items, immediately before inserting
(t2, s) into the C0, the stream item (t1 + 1, s) must still be in the ﬁlter, and must thus be in some

3.1.4
Results
For suﬃciently large q, it impossible for C0 to ﬁt in main memory, if it is to contain qm items for
some positive integer m. To reason about this situation, consider the boundary case m = 1. If
m = 1, then k1 = ⌈r0m/k0⌉k0 = ⌈m⌉= 1, meaning that items are moved individually from C0
to C1. A ﬂush of C0 thus occurs every stream item (after it has been ﬁlled once), so the levels C0
and C1 are eﬀectively always counted together. The clear solution is thus, that if q is too great
for C0 to ﬁt in main memory, we just ﬁx m = 1, place both C0 and C1 in external memory, and
insert item into both by ﬂushing.
Unfortunately, this means that we require at least one I/O per stream item, somewhat defeating
the purpose. Expanding the internal memory of the machine should thus be considered instead,
if greater values of q are required.

3.2
Reducing the number of reported items
So far, we have been considering only the requirement, that if a (s, τ, T)-substream exists, an
event with type s must be reported. However, this means, that once an event has been reported,
it may be re-reported repeatedly—in fact, the ﬁlter as described so far will cause the event to be
re-reported every time a ﬂush of the levels containing the corresponding items occurs. In other
words, there may be reported many more than one event per τ items.
To avoid this and to give some guarantees on the event throughput, we will now consider the
event detection problems with the additional requirement, that the substreams corresponding to
each reported (τ, T)-event must be non-overlapping.
We will however, even further require that the algorithm must report as many such events as
possible, and thus get as close as possible to having every τ items of the same type produce a
event—a goal that will, in fact, be achieved in the non-windowed version. In other words, we want
the reported events to correspond to a maximum family of non-overlapping (s, τ, T)-substreams,
in the sense, that no other other such family contains more substreams.

Problem 3.4 (On-line time-stretch event detection (maximum non-overlapping)). Given integer
τ > 0, and real number α > 0, process a stream S sequentially. For each s ∈U, consider
windowed maximum set of non-overlapping (s, τ)-substreams, and for each of these substreams
S(t1, t2], report a τ-event (t, s) at time t such that t −t2 ≤α(t2 −t1).
It should be noted, that this approach diﬀers from that of [5]; in the original article, items are
simply marked in a way only temporarily prevents re-reporting.
To ﬁrst satisfy the non-overlapping criteria, we do the following: Each stream item in each level
C0, . . . , CL−1 is stored with an additional bit indicating whether the item counted towards an
event that has already been reported. Stream items that have this bit set are not included when
counting items for each item type in R. Initially, the bit is unset for all items. When a (τ, T)-
event (t, s) is reported, perform an additional scan of the items of type s and mark the oldest
τ of them as used by setting the bit. As this at worst consists of a second scan of R, the total
I/O complexity is unchanged. This partial solution corresponds closely to the one in [5]. It does,
however, not satisfy the maximum criteria of Problems 3.3 and 3.4.

To combine weighted stream items with the previous idea of marking stream items as “used”,
it is necessary to be able to mark stream items as partially used. We can do so quite simply by
simply replacing the “used” bit with a number indicating how much of the item’s weight has
contributed to an event. When transferring which stream items are considered used during a
ﬂush, these values are then accumulated and consolidated into the oldest relevant items.
When scanning R and counting whether an event should be reported, it is important to do so
while scanning, as otherwise negative item weights can bring the count back down below the
threshold, even if a substream should trigger an event.
We can thus conclude that we can solve the weighted versions of the problems in the same I/O
complexity as the ordinary versions, i.e., in O((1/α + 1) log(T/M)/B), respectively O((1/α +
1) log(N/M)/B), amortized I/Os per stream item.
3.4
Preﬁx matching
It is also possible to extend the time-stretch ﬁlter to support small string types with preﬁx match-
ing. This can be useful for situations where more speciﬁc (longer) preﬁxes should be weighted
more than less speciﬁc (shorter) preﬁxes. It is important to note, that in this extension preﬁxes
rather than full types are reported.
In other words, if we let Σ be some alphabet, then our type universe U will be the set of strings
over Σ, i.e., U = Σ∗. We will, however, in the following generally assume that the strings are
nonempty. This is not a problem, as we can simply handle empty strings separately—either with
a single counter, or, if strict handling of window widths is important, an array of counts, such
that the exact count for any given interval can be determined as a simple diﬀerence.
There are diﬀerent ways the concept of events can be extended to preﬁx matches, but one simple
approach is the following: Rather than considering the count for each full type, counts are done
per preﬁx. Thus, a stream item with type abc counts towards a counter for a, a counter for ab,
and a counter for abc. This can be combined with the above extension of weighted stream items.
Instead of a single threshold value τ, each preﬁx length ℓis assigned a threshold τℓ, such that
a preﬁx P is reported if the (possibly weighted) count for P reaches or exceeds τ|P|. We will
assume that these thresholds are non-increasing τ1 ≥τ2 ≥· · · , so that shorter preﬁxes don’t
get reported before longer ones with the same count.
A direct but powerful generalization, which is the approach the rest of this section will use,
is to let the thresholds depend arbitrarily on the preﬁx being considered. In other words, let
τ : Σ∗→Z≥0. Then a preﬁx P should be reported if the count for P reaches or exceeds
15
==PAGE:16==
τ(P). The previous scheme is then given by τ(P) = τ|P|. Again, we will typically want shorter
preﬁxes to have higher reporting thresholds than longer preﬁxes, which we can then formalize
as τ(s1s2) ≤τ(s1) for all s1, s2 ∈Σ∗.
In order to apply the previous idea of marking items as “used” for a report, so that repeated events
are less common and more consistent, it is necessary to store the length of the preﬁx that was
reported together with the bit/value used to indicate that/how much of the item has been used.
Otherwise, it is impossible to determine whether it, for example, is sound to transfer the used
bit/value from an item with type aabc to an item with type aabde. This also makes it possible to
make more longer preﬁxes being reported, even if the corresponding stream items have already
been used for a shorter preﬁx. Doing so is both more consistent (since which exact stream items
that get used for an event becomes less important), and useful, since longer and more speciﬁc
preﬁxes would likely be more interesting to report than shorter and less speciﬁc ones.
We will be modelling the stream slightly diﬀerently, since we for the sake of the analysis want
timestamps to be in terms of symbols rather than stream items. Thus the stream of nonempty
item types s1, . . . , sN ∈Σ+ with weights w1, . . . , wN are then represented by the set of tuples
S given by
S = {(|s1|, s1, w1), (|s1| + |s2|, s2, w2), . . . , (|s1| + · · · + |sN|, sN, wN)}.
The timestamps are thus still unique and increasing, but now indicate at which time in terms of
symbols the stream item has fully been read. We will also update the interpretation of substreams
S(t1, t2], such that the individual symbols must be completely included in the half-open interval.
In other words, since the ﬁrst symbol of an item (t, s, w) must correspond to the time t−|s|+1,
we set
S(t1, t2] = {(t, s, w) | t1 < t −|s| + 1, t ≤t2}.
The deﬁnitions of (weighted) (τ, T)-event and (weighted) (s, τ, T)-substream are then adapted
to ﬁt this new model.
We will then be solving problems analogous to Problems 3.3 and 3.4, but with these string preﬁx
matching scheme, and with both timestamps and the parameters N, T, α expressed in terms of

Let A be a preﬁx-free set of nonempty strings over an alphabet Σ, and T be the (uncompressed)

the number of strings in A with π(v) as preﬁx).

following steps for each outgoing edge e = (u, v) in order of their label γ(e):
1. Write (γ(e), ν(v)).
2. Recursively serialize v.
3. Write the special value ♦.

3.4.2
Merging serialized tries
In order to use tries in the time-stretch ﬁlter, we must be able to merge tries together eﬃciently.
To do so, we will ﬁrst consider a subprocedure:
copySubtrie(Z, i):
1. Set d ←0.
2. Repeatedly:
(a) If Z[i] = ♦and d = 0: Return i.
(b) If Z[i] = ♦and d > 0: Set d ←d −1, write Z[i], and increment i.
(c) If Z[i] ̸= ♦: Set d ←d + 1, write Z[i], and increment i.
Lemma 3.12. Let A be a preﬁx-free set of nonempty strings over Σ, T its trie, and Z = σ(T) the
serialization of T. Let p be a node in T with subtrie serialized to Z[a, b). Then copySubtrie(Z, a)
copies Z[a, b) to the output and returns b.
Proof. First, we will show that at the start of each iteration, d is the depth of the node before Z[i]
relative to p (such that p has depth 0). Initially this holds because the node before Z[i = a] is p
with depth d = 0. At the beginning of an iteration, let u and v be respectively the node before
and after Z[i], and assume the claim holds such that d is the depth of u relative to p. Case (a)
does not impact the loop-invariance of the claim as the loop terminates here. In case (b), v must
be the parent of u, and hence have depth d −1, so the claim is maintained. In case (c), v must
be a child of u, and thus have depth d + 1, so the claim is maintained. The claim is thus in fact a
loop invariant.
As d is only decremented in case (b), before which it must be positive, d ≥0 is also a loop
invariant. Thus no strict ancestors of p is ever the node before Z[i], why we can conclude, by

the Euler-walk nature of the serialization, that at the beginning of an iteration the node before
Z[i] is in the subtrie rooted at p. Thus, since p itself is the only node in this subtrie for which
d = 0, we can conclude, that if case (a) applies, then the node before Z[i] is p and that either
i = |Z| or the node after Z[i] is the parent of p. In either case, i = b, so we can conclude that if
the procedure terminates b is returned.
Since thus a ≤i ≤b and i is incremented for each iteration of the loop until it is terminated, that
the loop eventually does terminate. As exactly each entry in Z[a, b) is thus written to output,
the proof is complete.

3.4.4
Using tries with the time-stretch ﬁlter
Given we now know how to merge tries and remove strings from a trie, we can now extend the
time-stretch ﬁlter to support preﬁx matching by storing each level as a serialized trie.
To satisfy the preﬁx-freedom and uniqueness conditions, we can store each stream item (t, w),
where t is the timestamp and w is the string, as w$t, where $t is a unique symbol.1 With each
serialized leaf node, the used bit/value and weight for the corresponding stream item is also
stored.
In the ﬂush step, the levels C0, . . . , Ck−1, Ck are then ﬁrst merged as usual, but into a serialized trie
R rather than a sorted array. Since merging two tries has the same asymptotic I/O-complexity
as merging two sorted arrays, the proof Lemma 3.4 (for step 1) applies.
Scanning through R to report events—and to mark stream items as (partially) used—is then some-
what more involved. We still scan through R linearly, leveraging the Euler walk nature of the
serialization. We then also maintain a stack for the current preﬁx P (by pushing symbols when
going down the trie and popping them when going back up), as well as calculating τ(P) for each
node. In order to avoid the evaluation of τ to aﬀect the overall I/O-complexity of the algorithm,
it should either be evaluated using only main memory, or by maintaining some state based on
the evaluation of τ in the parent node.
In any case, we can then maintain the number of unused stream items with a given preﬁx, or,
more accurately, the summed unused weight of the items, with a stack: When going down in the
trie, a 0 is pushed to the stack, indicating that we have so far visited no leaf nodes under that
node with unused weight. When visiting a leaf node, its unused weight is added to the top-most
value on the stack. And ﬁnally, when going up in the trie, the top-most value is popped from the
stack and added to the value beneath it, indicating that the sum of unused weights for the preﬁx
of the node we just ﬁnished processing also counts towards the one symbol shorter preﬁx of the
parent node we are now visiting.
In order to mark stream items as used, a trie over the used amounts of weight can be output, and
after ﬁnishing the ﬁrst scan of R, a parallel scan of it can mark the correct leaf nodes.

4
On-line union–ﬁnd
In this section we will be considered a streaming version of the classical union–ﬁnd problem. In
particular, I will present a new algorithm based loosely on a simple algorithm from [1] (speciﬁ-
cally Algorithm 1) for solving a batched version of the same problem.
A partition of a set V is a family S ⊆P(V ) of subsets of V such that each element in V is in
exactly one set in S. For each u ∈V , denote this unique set by Su.
The union–ﬁnd problem then consists of maintaining a partition S of V together with a map
r : S →V mapping each set A ∈S to its representant r(A) such that r(A) ∈A. Initially, S is
the partition into singleton sets S = {{u} | u ∈V }, and r is given by r({u}) = u. We must
then support two operations:
• union(a, b) for a, b ∈V . Replace the sets Sa and Sb in S by their union Sa ∪Sb and choose
an element w ∈Sa ∪Sb to be its representant.
• ﬁnd(c) for c ∈V . Return the representant r(Sc) of Sc.
We can also formulate the union–ﬁnd problem in terms of undirected graphs. In this interpre-
tation, we want to answer connectivity queries on an initially empty graph while adding edges.
We maintain an undirected graph G = (V, E) and a representant function ρ : V →V mapping
vertices to connected component representants. For each connected component C ⊆V in G,
every vertex in C must map to the same representant, which must itself be in C. Because of this,
ρ(u) = ρ(v) holds if and only if u and v are connected for u, v ∈V . Such a function always
exists, because each an arbitrary element can just be selected for each connected component as
its representant.
The two operations then become:
• union(u, v) for u, v ∈V . Add the edge {u, v} to E and augment ρ as necessary.
• ﬁnd(u) for u ∈V . Return ρ(u).
It is possible to support these operations storing only ρ.
For I/O analyses, we will abuse notation slightly and denote the size of V simply by V .
For a sequence S of union operations, let GS = (V, ES) be the graph resulting from exe-
cuting S in the initial state. Consider then a sequence Σ of union operations. Now split Σ
into subsequences A and B such that Σ = AB, and let ρ be a valid representant function for
GA. Let then ρ(B) be the sequence produced by replacing each operation union(u, v) in B by
union(ρ(u), ρ(v)). Then we have the following two lemmas:
Lemma 4.1. Two vertices s, t ∈V are connected in GΣ if and only if ρ(s) and ρ(t) are connected
in Gρ(B).
Proof. Assume s and t are connected in GΣ. Then there exists a path π from s to t. Consider
then the sequence of edges ρ(π) given by replacing each edge (u, v) on π with (ρ(u), ρ(v)) and
removing loops (u, u). For each edge (u, v) on π that is in EA, u and v are trivially connected in
GA, so ρ(u) = ρ(v), and the produced loop (ρ(u), ρ(v)) is thus removed. Every edge (ρ(u), ρ(v))
on ρ(π) thus corresponds to an edge (u, v) not in EA and thus from an operation union(u, v) in
B. But then ρ(B) contains the operation union(ρ(u), ρ(v)), so (ρ(u), ρ(v)) is in Eρ(B). Thus,
ρ(π) is actually a path in Gρ(B) from ρ(s) to ρ(t), why ρ(s) and ρ(t) are connected in Gρ(B).

Assume ρ(s) and ρ(t) are connected in Gρ(B). Then there exists a path π in Gρ(B) from ρ(s)
to ρ(t). Each edge (x, y) on π must then come from a union(u, v) operation in B with ρ(u) =
x and ρ(v) = y. For each pair of neighboring edges (x, y), (y, z) on π, we must thus have
(u1, v1), (u2, v2) ∈EΣ such that ρ(u1) = x, ρ(u2) = ρ(v1) = y, and ρ(v2) = z. But then u2 and

We can thus construct a path π′ in GΣ from s to t, showing that s and t are connected.
Lemma 4.2. Let σ be a valid representant function for Gρ(B). Then σ ◦ρ is a valid representant

Proof. Let C ⊆V be a connected component in GΣ. Then, by the above lemma, ρ(C) = {ρ(u) |

since σ(u) ∈ρ(C) for u ∈ρ(C), we also have σ(ρ(u)) ∈C for u ∈C. Both conditions are thus

4.1
External streaming algorithm
We will now consider a union–ﬁnd algorithm in the external memory streaming model, where we
will batch sequences of operations in order to sacriﬁce timeliness for a reduced number of I/Os.
Batching k operations can be done by straightforwardly writing them to an array in O(scan(k))
I/Os, so we will focus on what happens once the operations have been batched.
Given a representant function ρ : V →V and a sequence B of k union and ﬁnd operations, we
can perform the operations in B in the state of ρ with the following abstract steps:
1. Apply ρ to B, replacing each union(u, v) operation with union(ρ(u), ρ(v)) and each ﬁnd(u)
operation with ﬁnd(ρ(u)).
2. Solve the union–ﬁnd problem for B to answer the ﬁnd operations and produce represen-
tant function σ.
3. Calculate new representant function ρ′ = σ ◦ρ.
That this abstract approach is correct follows directly from Lemma 4.2.
We will concretely represent ρ by the set R = {(u, ρ(u)) | u ∈V, ρ(u) ̸= u}, stored as a list
sorted by the ﬁrst entry of each pair. Thus ρ(u) = v for u ̸= v if and only if (u, v) ∈R, while
ρ(u) = u if and only if no pair in R has u as its ﬁrst entry. By construction, R is initially empty.
Similarly, σ is represented by an analogous set S.
4.1.1
Step 1: Apply ρ to B
Step 1 is handled with the following sub-steps:
1a. Scan through the batched operations to produce a sequence A by handling each operation:
• For a union(a, b) operation, output a then b.
• For a ﬁnd(c) operation, output c.
Store each element u of A as a tuple (i, u), where i is its index in A so we can restore the
original order after sorting.

==PAGE:26==
1b. Sort A by the elements (second entry of each pair), then scan through it and R in parallel,
and replace each pair (i, u) in A by (i, ρ(u)) using that ρ(u) = w if (u, w) ∈R and
ρ(u) = u otherwise.
1c. Restore the original order of A by sorting it by the the indices (ﬁrst entry of each pair),
then scan through it and the batched operations in parallel and updating each operation:
• For a union(a, b) operation, let x and y be the next two elements in A, then replace
the operation with union(x, y).
• For a ﬁnd(c) operation, let z be the next element in A, then replace the operation
with ﬁnd(z).
Correctness.
Since step 1c matches step 1a, the elements extracted from A in step 1c corre-
spond exactly to those added in step 1a. Since step 1b replaces each element u by ρ(u), we thus
have x = ρ(a), y = ρ(b), and z = ρ(c), why these steps correctly apply ρ to the operations as
described.
Analysis.
Since A contains at most 2k elements, steps 1a and 1c use O(scan(k)) I/Os. Since
R contains O(V ) pairs, step 1b uses O(sort(k) + scan(V )) I/Os. The total I/O-complexity of
these steps is thus O(sort(k) + scan(V )).
4.1.2
Step 3: Combine ρ and σ
Step 3 is handled with the following sub-steps:
3a. Sort R by the second element of each pair. Scan through S and R in parallel, matching
the second entries of R with the ﬁrst entries of S, and construct a new sequence R′ by for
each v ∈V such that v is the ﬁrst entry of a pair in S or the second entry of a pair in R
doing:
• If (u, v) ∈R and (v, w) ∈S for some v, w ∈V , then output (u, w).
• If (u, v) ∈R for some v ∈V and no pair in S matched, then output (u, v).
• If (v, w) ∈S for some w ∈V and no pair in R matched, then output (v, w).
3b. Sort R′ by the ﬁrst entry of each pair. It is now the new R.
Correctness.
In the ﬁrst case of step 3a, ρ(u) = v and σ(v) = w, so σ(ρ(u)) = w, and it is
correct to output (u, w). In the second case, ρ(u) = v and σ(v) = v, so σ(ρ(u)) = v, and it is
correct to output (u, v). In the third case, ρ(v) = v and σ(v) = w, so σ(ρ(v)) = w, and it is
correct to output (v, w). For any other v ∈V , ρ(v) = v and σ(v) = v, so σ(ρ(v)) = v, and it is
correct not to output anything. The sequence R′ thus correctly encodes ρ′ = σ ◦ρ.
Analysis.
Step 3a takes O(sort(V ) + scan(k)) I/Os, while step 3b takes O(sort(V )) I/Os.
The total I/O complexity of step 3 is thus O(sort(V ) + scan(k)).

5
Conclusion
We have now in this thesis examined the external streaming model, some of its challenges, as
well as some of the techniques available. We have considered the potential and possibility of
less-than-one amortized I/O per item for two diﬀerent kinds of problems. To do so, we have seen
the role allowing delayed answering plays in making the batching of queries possible, as well as
the importance the merging of data structure plays in the external streaming model.
This thesis has examined two particular, relevant problems modelled in the external streaming
model, and an approach for solving each. For the event detection problem, we have considered
the time-stretch ﬁlter data structure, as well as several diﬀerent extensions that still achieve the
same less-than-one amortized I/O per item possibility (for suﬃcient high block transfer sizes B).
For the external streaming union–ﬁnd/connectivity problem, we have considered a particular
ﬁxed-cadence solution, that for suﬃciently long delays also can achieve less-than-one amortized
I/Os per item (but can also eschew this to instead beneﬁt from more timely query replies).
We have in particular seen, that the maximally non-overlapping windowed and non-windowed
time-stretch event detection problems (Problems 3.3 and 3.4) can be solved in amortized O((1/α+
1) log(T/M)/B) and O((1/α + 1) log(N/M)/B) I/Os per stream item respectively, where α
is the time-stretch factor and T is the window width. We have also seen, that the string-based
preﬁx-match versions of the same problems can also achieve the same bounds, but with α, T,
and N corresponding to the total size of the input (i.e., including the sizes of strings), rather than
just the number of items.
We have further seen, that the on-line union–ﬁnd/connectivity problem over a set of size V can
be solved with query delay k in O( V

mum spanning tree problem.
5.1
Future work
As the combination of the streaming and external memory model has little been explored so
far, especially in the context of achieving write-like performance for queries by allowing reply
delays, the directions for possible future work are many.
In particular, many streaming problems and algorithms can likely be adapted to the external
streaming model almost directly and possibly made eﬃcient by allowing delayed responses (es-
pecially, if the algorithms can be reinterpreted as sequences of merges of data structures). Fur-
ther, external memory dynamic (possibly as well as non-dynamic problems) can likely be made
streaming in a similar way to the union–ﬁnd/connectivity algorithm presented in Section 4, i.e.,
by batching suﬃciently many items/operations before applying them all at once. Furthermore,
while many graph problems will likely tricky to solve in the external streaming model, due to
matters such as non-global eﬀects of added edges, it would likely be both interesting and fruitful
to look further into these.
Additionally, the time-stretch ﬁlter in particular seems readily extensible, and its general tiered
structure—with older/less relevant items further down in larger levels and newer/more relevant
items available “closer” to main memory—is likely possible to adapt for other streaming problems.
Finally, implementing and practically evaluating the algorithms and data structures described
herein is an obvious and interesting path forward.
