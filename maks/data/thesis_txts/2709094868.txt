Abstract
A plethora of services make sharing files easy, but security requirements are higher than ever. The ease
of use often comes at the cost of placing unbounded trust in the service provider and giving up control of
file security. In this thesis we propose a design that uses cryptographic access control, an implicit access
control mechanism built into a file, to enable secure file sharing on a commodity file storage service.
Files are protected by state-of-the-art cryptographic primitives and the solution allows users to grant
read- and/or write-access at file granularity. A virtual file system implemented with FUSE abstracts
away the access control mechanism and exposes files like any other file system would.
Encryption
and decryption are transparently performed on-the-fly with support for random-access. This enables
end users to interact with their files through any application. Evaluation shows that the file system
performs well enough to support everyday usage. At the same time, a daemon (background program)
runs in concert across all users and quickly detects and reverts unauthorized changes to files. The thesis
is accompanied by fully functioning proof-of-concept implementations of both the file system and the
daemon.

Glossary & Abbreviations
• Attribute-Based Encryption (ABE): A suite of public key cryptographic schemes that allow en-
cryption under a set of attributes, which must be satisfied for an actor to decrypt the ciphertext.
• Authenticated Encryption with Additional Data (AEAD): A form of encryption that guarantees
both confidentiality and authenticity of the encrypted data using MACs.
• Ciphertext Policy Attribute Based Encryption (CP-ABE): A subset of ABE where a ciphertext
is associated with a policy and a user’s key is associated with attributes.
• Command Line Interface (CLI): A simple interface between a user and a computer.
• Cryptographic Access Control (CAC): An access control system in which the mechanism is based
solely on cryptographic primitives.
• Cryptographic File System (CFS): A file system with built in cryptographic measures typically
for integrity and confidentiality.
• Distributed Denial of Service (DDoS): A Denial of Service attack where the attacker acts through
a swarm of distributed computers.
• Distributed Hash Table (DHT): A key-value data structure distributed across several computers
in a network.
• Graphical User Interface (GUI): The visual interface through which the user interacts with ther
system.
• File Storage Provider (FSP): An external commodity service on which one can store files. Exam-
ples of FSP’s are Dropbox, OneDrive and Google Drive.
• Filesystem in Userspace (FUSE): An interface for user space programs to export and control a
virtual file system.
• Identity Based Encryption (IBE): A form of public key cryptography in which an agent can choose
a custom string as their public key, typically their email or other human-readable identifier.
• iff: If and only if.
• Key Policy Attribute Based Encryption (KP-ABE): A subset of ABE where a ciphertext is asso-
ciated with attributes and a users key is associated with a policy.
• Message Authentication Code (MAC): A short blob of data associated with a message used to
authenticate the message. The MAC also guarantees the integrity of the message.
• Node Package Manager (npm): A program that makes it easy to manage and use open source
modules written in JavaScript.
• Object: An entity representing a resource in the system. Access to the object may be controlled
by the access control mechanism of the system.

CHAPTER 1
Introduction
This thesis documents the analysis, design, implementation and evaluation of a system that utilizes a
cryptographic access control mechanism to provide a secure and flexible file sharing solution on top of
a commodity file storage provider.
1.1
Motivation
Data is an integral part of computing. We build computers to perform electronic processing of data.
It is hard to imagine any company today that does not rely on processing, manipulating, storing and
exchanging data in some way, regardless of what area they operate in. Even most individuals rely on
computers and more than half of the world’s population use the internet 1.
Exchanging data with others over the internet or through other mediums has enabled unprecedented
means of collaboration between humans. One of the most ubiquitous abstractions of data in computing
is the file. It is an encapsulation of data to which we assign metadata and attributes such as a file name
and type. File systems create the necessary structure to allow us to organize and retrieve files in ways
we find intuitive, such as grouping files together in directories and establishing a hierarchy.
During the relatively brief yet extensive history of computers, the way we share files has changed
almost continuously. Newer technologies, methods and protocols emerge, allowing us to move increas-
ingly larger amounts of data more easily than ever. Different methods such as floppy disks, thumb
drives, FTP (file transfer protocol) servers, file servers, peer to peer networks, AirDrop and file stor-
age providers, to name just a few, provide different trade-offs with regards to speed, latency, security,
convenience, cost, availability and other areas. Especially the latter, file storage providers (FSP), have
become the method of choice for many individuals as well as companies. They offer a low cost and
highly convenient solution for storing files and synchronizing changes across networks.
As much good as the sharing of files can do, it can also cause harm when files fall into the wrong
hands. This has been recognized in the last few years with an increasing focus on data sovereignty and
the introduction of data protection laws such as the EU’s General Data Protection Regulation (GDPR)
2. Privacy is, according to the UN 3, a human right, and many believe this extends to our data.
To protect our data from unauthorized access we employ access control systems. These are mecha-
nisms that allow us to control which actors get which capabilities (such as being able to read or update
a file). As we explain later, there are many different ways to model access control. The access control
present in FSP solutions rely on a centralized reference monitor to enforce policies. A way to think of
this is that our data is open, so we build a perimeter with a gate around it. The reference monitor is
the gatekeeper. The problems with this approach is a lack of transparency and control. We trust the
reference monitor, often on the basis of legal agreements such as privacy policies, to behave as agreed
upon. Experience has shown this approach to be fragile, as disgruntled employees, governments or
other bad actors may circumvent protections and find themselves on the inside of the perimeter.
This motivates us to consider a novel approach called cryptographic access control that uses cryp-
tography to embed the protection of files into the files themselves. How this works will be covered in
detail later but the result is a clear separation between the storage layer and the access control layer.

1.2 Goals & Requirements
2
The FSP is relegated to ensuring just the availability of files, while confidentiality and integrity are in
the control of file owners.
The main goal of this project is to provide a convenient solution that allows users, whether they
are individuals or large organizations, to use a commodity file storage provider while at the same time
remaining in control of their own data security through the use of cryptographic access control.
1.2
Goals & Requirements
We will list technical and user experience requirements followed by a list of assumptions that we make
about any file storage provider used in the system.
1.2.1
Technical Requirements
The requirements of technical nature are listed below:
• The system should be able to run on macOS and Linux.
• The system should implement an access control mechanism purely dependant on cryptographic
primitives. There should be no dependency on a central reference monitor.
• The access control mechanism should protect the confidentiality of files. Unauthorized actors,
including the FSP, must not be able to view the contents of files.
• The access control mechanism should guarantee integrity of resources. Unauthorized actors, in-
cluding the FSP, must not be able to modify files without this being detected.
• The system should effectively deny unauthorized operations by reverting the system to a prior
state when such an operation is detected.
• The system must have a flat user hierarchy. There should be no notion of root-users, administra-
tors or similar, and the system must not depend on any one user.
• It should be possible to use any kind of file storage provider as long as it satisfies a number of
requirements (specified in section 4.1).
1.2.2
User Experience Requirements
The user experience requirements listed below describe how the solution should behave from the per-
spective of an end-user. The intended end-user is a somewhat technically adept and privacy concerned
individual or member of an organization.
• The system should behave like a regular file system and attempt to support normal operations
including creating, reading, writing, deleting, renaming and moving files and directories through
the native file browser application (Finder an macOS and GNOME Files or similar on Linux).
• Reading from and writing to files should be suﬀiciently performant and have no appreciable lag
in normal workflows.
• New users in an organization should be able to join the system easily.
• It should be possible to share read-access and write-access at the granularity of files.
• All files are seamlessly and continuously synchronized between all users in the system.
• The Cryptographic Access Control (CAC) should be transparent to the user. Thus the imple-
mentation details of key management, integrity checks, encryption/decryption and similar are all
completely hidden from the users point of view.

1.2.3
Non-project goals
As will be seen in our review of existing work in chapter 3, some of the previous implementations have
relied on proprietary technologies that are now obsolete and no longer maintained. This makes it more
diﬀicult for others to continue research and work on the topic. Based on this we decide to rely on open

the open source ethos is giving back any changes and improvements made for others to benefit. For
this reason we have made it a goal to contribute any changes we make.
1.3
Out of Scope
The following things are relevant but not something this project will attempt to provide a solution for.
• Real-world user identity: Being able to tie the identity of users in the system with their
real world identity can improve the overall friendliness and experience of the system and enable
interesting features. For example, in an organization it is much more useful to refer to another user
as Benny from Accounting than by an opaque identifier (such as their public key): 22bb...73feb.
How to perform this mapping from identifier to identity is an ongoing research topic and different
solutions, such as web-of-trust in PGP or public-key infrastructure (PKI), have different trade-offs.
We will leave it to others to provide a suitable identity management layer but will discuss the
benefits of having such a layer in section 8.3.
• File metadata protection: While our solution protects file contents, metadata such as the
filename and size is still revealed, which could be of interest to an adversary. Implications and a
possible solution will be discussed in section 8.7.
• Revocation: A major challenge in any kind of system that uses cryptographic access control is
how to revoke permissions that were previously granted to a user. Our solution does not support
this presently but we will look into possible approaches in section 8.6.
• Advanced keyring structures: Our solution uses keyrings, a simple key data structure intro-
duced in section 5.6, and we will explain how keyrings can be used to model various access control
modes. Other contributions to the topic of cryptographic access control have demonstrated im-
plementations of this so we will focus our efforts on other areas. This means our implementation
will only support the local keyring of a user.
• Alternative FSP support: Our solution has been designed to work with any file storage provider
that meets a set of requirements (specified in section 4.1), however, we will focus on integrating
with a single FSP. This can then be used as a reference for those interested in creating an inte-
gration with their preferred FSP. A non-exhaustive list of providers can be seen in [Wikb].

CHAPTER 2
Theory
In this chapter, we will cover theory relevant for the problem and our solution. We expect the reader to
already be familiar with basic concepts in computer science, cryptography and data security. Section 2.1
introduces classic access control systems and compares various access control modes to cryptographic
access control. Section 2.2 elaborates on the cryptographic primitives and specific algorithms used in
our solution. Lastly, section 2.6 describes the role of file systems and what FUSE is and how it works.
2.1
Access Control Systems
Access control systems are used to determine and enforce authorization (i.e. whether permission to
do something is granted). A policy specifies what actions a given subject (typically a user) is allowed
to carry out on a given object (for example a file). This can imply that some form of authentication
(i.e. knowing the identity of the subject) is necessary. To understand how cryptographic access control
works, it is relevant to first consider how a typical access control system looks and what its components
are. Figure 2.1 shows a common architecture for an access control system.
We will in the following elaborate on three different access control models relevant for future refer-
ence.
Access Matrix
An abstract model of a policy is the access matrix (illustrated by figure 2.2). It
records for each subject and each object exactly which actions are permitted. In practice, this model
quickly becomes too diﬀicult to scale and maintain.
Access Control List (ACL)
A list associated with an object that specifies which actions each
subject is allowed to perform on that object. This can be thought of as a row in the access matrix.

Capability List
A list associated with a subject that specifies which actions the subject is allowed
to perform on each object. This can be thought of as a column in the access matrix.
Access control systems can often be designated as either ACL-based or capability-based. There are
a few subtle differences between them. In ACL-based systems, there is typically a central entity that
keeps track of the list for each object and acts as an authority. They will be responsible for modifying the
access control lists by adding and removing subjects. ACL-based systems are often a type of Mandatory
Access Control. In contrast, capability-based systems enable subjects themselves to manage their own
list of capabilities. A capability is some token of authority, that when possessed grants a permission
to perform an action on an object. This allows for systems where authorization can happen without
authentication. In other words, the subject is only required to prove that they posses a capability and
not their identity. 1 Subjects can delegate their access to other subjects by transferring the capability to
them, which is not possible in ACL-based systems. This results in what is called Discretionary Access
Control. On the other hand, revoking a permission is trivial in ACL-based systems since all that needs
to happen is for the authority to update the ACL. In capability-based systems, reclaiming a capability
from a subject and ensuring they have not made copies of it is impossible to do reliably. In [Mar03], the
authors suggest a form of proxy-mechanism, in which adding a level of indirection can allow a subject
to delegate a capability and later revoke it.
2.1.1
Access Control Modes
There exist many different access control modes (also referred to as models or frameworks). These are
different ways of modelling the access control policy, and each carries its own trade-offs. In the following
sections we cherry-pick and compare a number of commonly used modes. The purpose of these modes
is to increase flexibility and scalability of the access control system.
Role Based Access Control (RBAC)
A model in which permissions are associated with a prede-
fined set of roles. Subjects are assigned one of these roles. This abstraction of hiding users behind roles
simplifies permission management greatly at the cost of less granularity. Roles are typically created
to match certain job functions in an organization such as finance manager or IT worker. The setup,
granting and management of the roles is typically handled by a central authority, like an IT/Policy
department in the organization.
Attribute Based Access Control (ABAC)
A model in which policies are formed as boolean expres-
sion that is evaluated over sets of attributes. The attributes can relate not only to the subject but also
the object, the action being carried out and other contextual information (such as time or location). An
example of policy could be ((“Finance” AND (“Copenhagen” OR “Aarhus”)) OR (clearance-level

A mandatory access control variant where security levels

2.1.2
Cryptographic Access Control
Cryptographic access control (CAC) was proposed by [HJ03] and involves building an access control
system from cryptographic primitives. While it might make sense in other contexts, we will primarily
focus on the case where objects are files and subjects are users that wish to interact with these files.
CAC is an example of a capability-based access control system where cryptographic keys represent
capabilities. It typically involves a symmetric key and an asymmetric key pair for each object, which
correspond to three different capabilities 2.
The symmetric key represents the read capability, the
private key of the pair represents the write capability and the public key of the pair represents the
verify capability (being able to verify that a write was authorized). Authorization happens through
the cryptographic operations of encryption/decryption and signing (these are covered in more detail in
section 2.2). It can be seen clearly how delegating capabilities is as simple as (securely) transferring a
key to another subject.
It can be diﬀicult to reason about CAC systems because the typical architectural components (as
shown in 2.1) are more scattered and the boundaries between them less clear. For example, enforcement
of policies related to the read action happens directly between the subject and the object. If the subject
has a read-capability for an object, they can decrypt it, otherwise they can not. Permission to write,
on the other hand, is enforced elsewhere.
Depending on the system architecture, this could be by
a centralized storage server, that serves as a reference monitor, or by other peers in the case of a
decentralized system. This distributed enforcement of policies will be covered in more detail as we look
at existing work within CAC and design our own solution.
The upside of using CAC is that it eliminates the need for a centralized reference monitor, which
improves scalability and allows for federation between multiple systems, since there is no longer a need
for one authority to maintain global knowledge of the system. The ease of delegation is desirable for
file sharing as it makes it easy for one user with access to grant the same access to another user.
Some known challenges that arise in systems using CAC are:
• Auditing: It becomes impossible to audit some actions in the system (such as reads), since this
action occurs between subject and object with no other entities involved. During our research we
have not come across any proposals on how to audit actions in CAC.
• Delegation: The right to delegate is not governed by any control mechanisms. This means any
subject can transfer their capabilities to other subjects, which can be both a positive and negative,
depending on the desired properties of the system. This is mostly a concern in systems that do
not have non-repudiation of actions. In section 8.4 we discuss this further.
• Key distribution & management: Since capabilities are represented by keys, the security of
the entire system is heavily reliant on how these keys are distributed and stored.
• Revocations: As discussed already, revocation is more complex in capability-based systems. In
the case of cryptographic access control, revocation often involves re-keying (using new keys to
encrypt and sign files) and distributing these new keys. In other words, everyone but the subject

2.1.3
Cryptographic Access Control Modes
CAC as described hitherto corresponds to a simple capability list system. Since the capabilities are
keys, the list is fittingly referred to as a keyring. For reasons already discussed, we want to support
more elaborate access control modes, even when using CAC.
Other projects, which will be detailed in chapter 3 on existing work, have used an approach that
allows for this. It has been covered extensively, so we will only briefly summarize the idea behind it.
By treating keyrings as files, they can be protected by CAC and stored alongside other files (using
whatever storage backend is available). At the same time, this allows keyrings to refer to other keyrings.
This means a hierarchy of keyrings can be built. Each user has their own keyring which in turn unlocks
shared keyrings. These shared keyrings can correspond to roles, attributes, identities or something else.
This system is so flexible that it even allows for different modes to be combined without interfering.
Figure 2.3 demonstrates how an Lattice Based Access Control (LBAC) system can be modelled using
keyrings. The challenge with keyrings, especially with an elaborate hierarchy, is to handle deduplication
of keys. If many copies of the same key exists across various keyrings, changing the path that the key
points to becomes practically impossible. [Thy06] proposes Link Keys, which are keys that reference
other keys rather than files. Additionally, keyrings suffer from the same challenges with regards to
revocation. For example in an RBAC system, once a role has been granted it is diﬀicult to demote or
even fire the subject.

Cryptographic Primitives
A cryptographic primitive is a basic cryptographic building block that is typically combined with other
primitives to form a cryptographic system. Our proposed solution relies on a number of cryptographic

2.2 Cryptographic Primitives

Cryptography is a means to ensuring a system provides a number of properties. For our solution we
are interested in the following:
• Confidentiality: keeping data secret for all but those who are authorized to see it.
• Integrity: ensuring that data has not been tampered with (either deliberately or accidentally).
• Authenticity: being able to verify a source of data.
• Non-repudation: is the inability for an actor in the system to deny or dispute having performed
some action.
2.2.1
Encryption
Encryption is the process of encoding information, typically called plaintext, using a secret key, to
produce a ciphertext. The goal is to ensure the confidentiality of the information such that only parties
in possession of the key can reverse the encryption (decrypt), returning the information to its plaintext
representation. An ideal encryption scheme is one that encodes the data in such a way that the resulting
ciphertext is indistinguishable from random noise. It should remain secure even if everything about the
system except the key is known (this is known as Kerchoff’s Principle).
There are generally two types of encryption schemes: symmetric encryption, in which the key to
encrypt and decrypt the message is the same, and asymmetric (or public-key) encryption in which one
key is used to encrypt the message and another related key is used to decrypt it.
Symmetric key ciphers can work either as a stream cipher or a block cipher. A stream cipher encrypts
a single bit at a time, typically by performing an exclusive or (XOR) operation between the plaintext
and a pseudorandom keystream. Block ciphers operate on fixed-size blocks, requiring the last input
block to sometimes be padded. It also means the size of the ciphertext is a multiple of the block size,
which somewhat obscures the size of the plaintext. As we will discuss later, this can have important
implications for our system.
Block ciphers are used in a mode of operation, which determines how each block operation is
performed and how it relates to other blocks.
These modes all have different properties, security
guarantees and levels of performance. In general, we want previously encrypted blocks to affect how
future blocks are encrypted.
This causes whats known as an avalanche effect, increasing diffusion
in the ciphertext, which makes cryptanalysis much more diﬀicult to carry out. A common example
of the importance of the avalanche effect is ECB mode (electronic code book) in which blocks are
encrypted independently. This not only allows certain structure of the plaintext to come through in the
ciphertext but also makes the ciphertext vulnerable to blocks being substituted, removed or reordered
(as demonstrated with figure 2.4). Many applications instead use CBC (cipher block chaining), which
adds diffusion but has the disadvantage that in order to decrypt block N, each prior block 0.. < N
must also be decrypted. Other modes such as AES-CTR (counter), can make a block cipher behave as
a stream cipher.
Some modes provide authenticated encryption with additional data (AEAD) by combining an en-
cryption algorithm with a message authentication code (MAC) function. This provides not only con-
fidentiality but also integrity, allowing a recipient to determine if a message has been corrupted or
tampered with. The encryption and authentication are performed using the same key. The additional
data refers to a header that is included with the ciphertext and covered by authenticity guarantees
but not confidentiality. Authenticated encryption protects against chosen ciphertext attacks since a
decryption oracle will detect improperly constructed ciphertexts and refuse decryption. Authenticated
encryption constructions differ depending on the order in which the ciphertext and MAC are produced.
Today, it is considered best practice to use authenticated encryption, as evidenced by which algorithms
and modes are supported in modern security protocols such as TLS 1.3 3.

In most modes, encryption requires the use of an additional input, besides the key and the plaintext,
called an initialization vector (IV). Whether this value should be chosen randomly or just be unique
depends on the mode. One thing that is critical to note is that some modes require the IV to only ever
be used once for a given key. In these cases, the IV is often called a nonce (number used once), which
is also the term we will use going forward. If a nonce is ever reused with the same encryption key, the
cryptosystem fails catastrophically. To see why, consider an example where we have plaintext messages
M1 and M2, a key K and a nonce N. KS is the function that derives a keystream from a key and a
nonce. We encrypt the two messages, producing ciphertexts C1 and C2:
C1 = M1 ⊕KS(K, N)
C2 = M2 ⊕KS(K, N)
If an attacker obtains both ciphertexts C1 and C2 they can derive the XOR of the plaintext messages:
C1 ⊕C2 = (M1 ⊕KS(K, N)) ⊕(M2 ⊕KS(K, N)) = M1 ⊕M2
While it does not reveal M1 and M2 directly, it reveals a certain structure about the inputs as
demonstrated by figure 2.5.
This structure might allow the attacker to make an educated guess about the contents of either M1
or M2. Correctly guessing either message allows for the recovery of KS(K, N). To obtain the key K
from this result would be a known-plaintext attack, which modern ciphers are resistant against, thus
requiring a brute-force search.
Some popular algorithms that suffer from failure if a nonce is reused are AES-CTR, AES-GCM
(Galois counter mode) and ChaCha. Notice that all are, or behave like, stream ciphers. As will be seen
later, this is one of the most critical considerations in the design of the cryptographic file system layer
in which file blocks are continuously re-written.
There are numerous encryption algorithms that have undergone varying levels of scrutiny by the
community. We will focus on two modern algorithms that are presently considered secure.

ChaCha is family of stream ciphers that have evolved from the Salsa family of ciphers [Ber08]. There
exists 8, 12 and 20 round versions of the cipher with the 20-round version, known as ChaCha20, ap-

generator in the Linux kernel since version 4.8 and it is currently the only supported cipher in the
WireGuard VPN protocol ([Don15]).
Performance wise, ChaCha20 is significantly faster than AES when both are implemented purely in
software and slightly slower when AES CPU instructions are available [Ber]. ChaCha20 has very little
to gain from specialized hardware instructions and its implementation in software is less susceptible
to side-channel attacks. This makes it a more portable encryption standard that can be used on all
devices ranging from low-power embedded systems to desktop workstations.

2.2.2
Hashing
A hash function is a function that maps values from an arbitrarily (and potentially infinitely) large
domain to a fixed size range. There are different types of hash functions serving different purposes. Of
particular interest to us are cryptographic hash functions, which we will subsequently refer to as just
hash functions. We want a hash function f to have the following properties:
• Avalanche effect: a small change to the input (such as flipping a single bit) should significantly
change the output.
==PAGE:23==
2.3 Identity-Based Encryption
14
• Collision resistance: it should be diﬀicult to find two different inputs M1 and M2 such that
f(M1) = f(M2)
• Pre-image resistance: given a hash value H, it should be diﬀicult to find an M such that
H = f(M).
• Second pre-image resistance: given M1, it should be diﬀicult to find an M2 such that f(M1) =
f(M2).
Depending on the area of usage we might want the hash function to be either very fast or very
slow to compute.
Common hashing algorithms are MD5, SHA-(1/2/3) and BLAKE(1/2/3).
MD5
and SHA-1 are considered broken and obsolete at this point, since collisions can be computed more
trivially through brute-force search. They should not be used in any new deployments and existing
implementations should migrate to something more secure.
SHA-2 is likely the most widely used
hash function today. SHA-3 uses a different internal construction from previous hash functions and is
intended as a fallback in case flaws are discovered in SHA-2 more than an immediate replacement for
SHA-2 4. BLAKE is based on the previously described ChaCha stream cipher and is considered to be
secure, while being much faster to compute than SHA-2 and SHA-3.
2.2.2.1
Message Authentication Codes
A Message Authentication Code (MAC) is used to verify the integrity and authenticity of a message.
It is somewhat related to hashing but also involves a secret key and is therefore sometimes called keyed
hashing. The sender produces a MAC for a given message using a key. A receiver of the message that
knows the same secret key can produce the MAC themselves and compare it to the one provided by
the sender. Since both sender and receiver share the key, MACs do not provide non-repudiation. As
explained in a prior section, MACs are combined with ciphers to provide authenticated encryption.
2.2.3
Digital Signatures
Digital signature schemes are used to verify the integrity and authenticity of a message similarly to
MACs. Instead of using the same secret key for both the signature generation and verification, digital
signatures rely on asymmetric cryptographic.
A private key is used to produce the signature and
the corresponding public key can be used by anyone to verify the signature. This also provides non-
repudiation if and only if the signer is the only one in possession of the private key.
The two most common public-key cryptosystems are RSA (Rivest-Shamir-Adleman) and Elliptic-
curve cryptography (ECC), which derive their security from the diﬀiculty of solving the prime factoring
problem and the discrete-logarithm problem, respectively. Both are considered secure but the latter
requires much smaller keys leading to lower storage and computational costs. Digital signature scheme
are often used in combination with a hash function.
Instead of signing an entire and potentially
large message, a cryptographic hash function is used to compute a digest of the file which in turn is
signed. When computing digital signatures, we will use Ed25519 introduced in [Ber+12], which uses
the Edwards-curve Digital Signature Algorithm (EdDSA) signature scheme and SHA-512 hash function
with the Curve25519 elliptic curve.

Generator, a trusted third party) of that identifier obtains the corresponding private key, allowing them

2.4
Attribute-Based Encryption
Attribute-Based Encryption (ABE) is an encryption scheme that supports an attribute-based access
control (ABAC) mode through a cryptographic access control mechanism.
ABE was introduced in
[SW05] and builds upon the idea of IBE. It uses the concept of attributes and policies (also called
access control trees). A policy is a predicate made up of the available attributes. For ABE schemes we
discern between key-policy attribute-based encryption (KP-ABE) and ciphertext-policy attribute-based
encryption (CP-ABE).
CP-ABE
In this type, a set of attributes is encoded in each key and objects are encrypted under a
policy. Any key that matches the policy that was used to encrypt an object can decrypt it.
Assume we have attributes A, B and C, which are used to form the policy P.
P = (A ∧B) ∨C
An object encrypted under P can be encrypted by any key that has the appropriate attributes to
satisfy P. Three such keys could be:
• K1 = {A, B}
• K2 = {C}
• K3 = {A, B, C}
The key K4 = B would not be able to decrypt the object as its attributes do not satisfy the policy.
KP-ABE
KP-ABE is the dual of CP-ABE. The policy is encoded in the secret key of a user and the
message is encrypted under a set of attributes. Using the same attributes as above, we could issue the
following key:
K1 = (A ∧B) ∨C
Any object encrypted with the sets of attributes {A, B} or {C} (or the union of these two) can be
decrypted using K1.
Which of the two schemes is most appropriate depends on the system at hand. CP-ABE is ideal
when access policies are known at the time of object creation (such as the department to which an
object belongs).
Imagine then that the need for an external auditor arises.
Unless this possibility
was considered at the time of object-creation, and the appropriate "auditor" attribute was set during
encryption, the only way for the auditor to get access is for them to obtain keys that model the necessary
attributes. It might be inappropriate to claim that the external auditor should now have the attribute
"Department = Finance", when they are, in fact, not in this department. With KB-ABE on the other
hand, we can create a key for the auditor that has a policy that fits exactly the attributes that they
were brought in to audit (such as "Legal document").
2.4.1
Advantages
ABE has the advantage of providing highly granular access control built right into the keys and the
ciphertext. As demonstrated previously, attribute-based access control is one of the most flexible modes
on top of which other modes such as role-based access control can be built. We have demonstrated
a simple example of a policy using just the and and or logical operators.
[BSW07] describes how
an access structure can be modelled to allow for more elaborate policies that can even contain range
==PAGE:25==
2.4 Attribute-Based Encryption
16
queries. This would allow for the query previously mentioned in the discussion of ABAC: ((“Finance”
AND (“Copenhagen” OR “Aarhus”)) OR (clearance-level > 8) OR “name = Alice”).
Similar to IBE, public key distribution between subjects is not required, because encryption of files
happens under a policy or set of attributes rather than a key. Unlike IBE, however, we only need to
produce a single ciphertext even when we want to share an object with many different recipients. Key
management is also much more simple because there is exactly one key for each subject, instead of one
key for each object.
Another advantage is how ABE allows for sophisticated revocation schemes. The scheme introduced
in [Yam+15] is non-monotonic and allows for policies to include negated attributes. This means the
policy can explicitly exclude certain attributes. This effectively allows for a blacklist of users to be
built into the policy. Imagine each user is granted a unique attribute ID, IDuser. Then that user u can
be revoked from an object encrypted under a policy P by extending the policy with the clause ¬IDu:
P ′ = P ∧¬IDu and re-encrypting the cipher under P ′. No key re-distribution is needed and only the
user u will be revoked. A problem with this approach is that the policy will keep growing as users are
revoked, and in most of the ABE schemes cited, the number of clauses in the policy appears to affect
the size of the ciphertext and the speed at which it can be decrypted.
2.4.2
Challenges and Disadvantages
• Collusion resistance: A major challenge with ABE is to ensure the scheme is collusion resistant,
meaning it should not be possible for two or more users to combine their secret keys and obtain
a new key that has the union of their individual attributes. This challenge has been solved in all
schemes that we cite.
• Eﬀiciency: Current schemes are very ineﬀicient both in terms of speed and ciphertext size. As
with other public-key encryption schemes (i.e. RSA), the throughput of encryption and decryption
will likely never rival that of symmetric encryption algorithms. A hybrid approach, in which ABE
is used to encrypt a symmetric key, which in turn encrypts the data, can be used instead. One
such scheme involving ABE is proposed in [LLC18].
• Private Key Generator: Similar to IBE, a central trusted authority is involved in the setup
of the system and the issuing of secret keys. This introduces a single-point-of-failure, and if the
trusted party were to be compromised, the entire system would collapse.
Another thing to note is that a user can not delegate access to another user (other than by giving
up their own secret key), meaning they can not ”give” an attribute that their key has. Whether this is
a disadvantage depends on the context of the system.
2.4.3
Implementations
Unlike the cryptographic primitives previously introduced in 2.2, there are no standardized algorithms
and widely used libraries available when it comes to ABE.
During our research we discovered OpenABE [LLC], which is an open source library that provides a
number of ABE (both CP and KP) implementations. Unfortunately, it appears the company behind it
is defunct and development has stopped, based on the lack of updates to the library.
With considerable effort we managed to make the library build and pass its own test suite. This
revealed a number of issues with the implementation. Any attempt to encrypt files large than approx-
imately 250 MB with CB-ABE failed. This limitation could likely be circumvented by using a hybrid
scheme as described above.

2.5
Attribute-Based Signatures
In cryptographic access control we separate read and write capabilities by using both encryption and
signatures. If we opt for attribute-based encryption, we also need a attribute-based signature scheme.
Otherwise, if we use ABE with a regular signature scheme, we have the disadvantages of both and the
complexity of having to combine two orthogonal approaches.
[KPR10] proposes a scheme for attribute-based signatures (ABS), which allows a signer that has a
key with certain attributes to sign a message while at the same time proving that their set of attributes
satisfies a given predicate. Anyone will be able to verify that the signature on the message was indeed
made by someone with the appropriate attributes.
For our purpose, we thought of a design where each ciphertext would have two policies. One deter-
mines whether a given key can decrypt the ciphertext (i.e. read capability) and the other determines
whether a signature was produced by someone with the right attributes (i.e. write capability). For
example, we might have the following two policies that allow anyone in the engineering department to
read a file but only the department head to write to it.
• Read policy: Department: Engineering
• Write policy: Department: Engineering ∧Role: head of engineering
Only a limited amount of research has been done on Attribute-Based Signatures (ABS) and we were
unable to find any schemes that simultaneously allow for ABE and ABS, nor did we find any available
implementations of ABS.

Cryptographic Access Control
In the following, we introduce five prior contributions that use CAC and relate them to the [HJ03],
which originally introduced the idea. The five contributions are all authored by prior DTU students.
They share a number of traits, such as using the keyring concept previously described, but also differ in

point for students wishing to continue work in this area in the future.
3.1.1
Cryptographic Access Control in a Distributed File System
In 2003, [HJ03] introduced the idea of cryptographic access control in a distributed file system. Files
are encrypted and signed and stored on a modified NFS file server.
Each file is associated with a key, k and a key pair, (priv, pub). The key k is used for symmetric
en-/decryption, and thus provides read-access to the object. The private key, priv, models write-access,
as the possessor can sign their write-operations to the object. Likewise, the public key, pub, models the
verify access allowing the possessor to verify the integrity of incoming writes-operations by checking its
signature. Thus the access control mechanism is implicit in that objects are stored as signed ciphers1.
The paper states that CAC allows anyone to perform reads from and writes to the storage layer
without any form of reference monitor. The only requirement is that the storage layer uses some form of
journaling or log-structured file system, such that writes to a file produce copies instead of overwriting
the existing file. Otherwise, nothing prevents a malicious actor from writing to a file without permission,
which compromises its integrity and/or availability.
This, however, leads to the number of copies growing indefinitely. To counter this, the authors
propose a log compression operation on the server that uses the verify-capability of pub to verify that
a write was indeed authorized and remove any copies that were written without authorization. Besides
carrying out this operation, the server is very simple and does not have access to files, since it never
receives the key, k.
The paper also mentions that by having clients encrypt and sign files before they are transferred
to the server, the network that is used to transfer files does not have to be secure. This is true, but
we believe that this specific advantage is less of a concern today, since most communications happens
between internet-connected services and is protected by TLS.

Kryptografisk Adgangskontol i peer-to-peer netværk
In 2003, [HK03] investigate cryptographic access control in a peer-to-peer setting. They leverage a
protocol specification called JXTA for resource discovery (i.e where to find a file in the network). In
2010, the development of JXTA was stopped by Sun Microsystems, the main contributor.
The main take away from this thesis is their introduction of keyring data structures for key man-
agement. A keyring is a file containing keys (i.e. read/write/verify capabilities) for other files - which
might also be keyrings. This simple data structure is extremely flexible, and will serve as building
blocks for more complex access structures introduced later.
The remaining work mostly relates to typical problems when developing peer-to-peer networks. We
will not go into detail with these, as this category of problems is not relevant to our solution, and
because their work is (understandably) somewhat dated.
3.1.4
Key Management in Cryptographic Access Control
In [Thy06], Thyregod attempts to build CAC for CryptOS, an open problem to build an operating
system (OS) in which all mechanisms are solved with cryptography.
This contribution is closely related to the work of [HJ03], and mostly shows itself as an attempt
to implement the ideas originating in [HJ03]. Its main contribution is to model Role Based Access
Control (RBAC) with keyrings, where a keyring associates a role, and the keys in the keyring model
the capabilities of that role. Thus granting a user access to a keyring is equivalent to assigning a role
to that user (and allowing them to now delegate that role to others).
3.1.5
Cryptographic Access Control in a Cloud Based File Storage Environment
In [LF16], Lundt and Flamsted combine the ideas of CAC and keyrings with a MediaWiki backend. This
shows the first attempt to use a generic, non-purpose-built server as the storage backend. MediaWiki
supports revisioning, allowing one to download old versions. This is similar to the log-structured file
system mentioned in [HJ03]. Since the server is an off-the-shelf solution that has no notion of the CAC
system being used, it does not do any form of log compression (removing invalid revisions).
Furthermore, Lundt and Flamsted try to implement a virtual file system using FUSE. However, the
authors later conclude this to be too complex. Instead, they provide users with a simple CLI, which
they acknowledge it a sub-optimal user experience.
They also start the work of a (naive and manual, yet working) access revocation protocol: To revoke
one subject’s access to an object, another subject copies the object, encrypts it under fresh keys and
distribute these keys to all but the subject to be revoked. However, their work on this matter is rather
superficial and they leave it as future work.
Lastly, they are the first contribution to consider and acknowledge the problems of keeping keyrings
updated when files are moved or renamed. Keyrings keep references to the object each key refers to,
but these references become invalid when files are moved, renamed or deleted. They suggest a simple
protocol for updating keyring entries when objects are renamed locally. Unfortunately, they do not
consider how local file renaming leave all replicas of the file (and corresponding keyrings) in an invalid
state, as other keyrings are not aware of the renaming. Interestingly, this leaves the question of who is
responsible for updating keyrings when objects are moved/deleted/renamed. We discuss this problem
in 8.5.

validating author tokens without some kind of PKI or similar identity management system. We discuss
identity management in 8.3.

CHAPTER 4
Analysis
In this chapter we will identify the major problems that need to be addressed by our solution. Recall
that our overall project goal is to provide a solution that allows for files to easily be stored on commodity
file storage providers while the file owner remains in control of data security.
4.1
Storage Backend
An essential problem to consider is how and where to store files. This affects everything from the security
requirements and threat model to how files should be shared and how file operations are carried out. The
subsequent system design and implementation are heavily influenced by the choice of storage backend.
As our literature review in chapter 3 shows, there have been many different approaches to this.
Our project goal reveals that we have already directed our focus towards cloud based file storage
providers such as Dropbox, however, we would like to share our reasoning behind this decision.
Going back a few years, the preferred way of sharing files was either through removable media (thumb
drives, discs) or self-hosted file servers. The former has a kind of physical access control mechanism
since you have to be in possession of the media to access its contents. File servers on the other hand are
always online and available to everyone simultaneously, hence they require some form of access control.
Today, as with many things in information technology, organizations have outsourced their file
storage.
We can no longer require organizations to self-host their own solutions, whether they are
Network File Systems (NFS) or Server Message Block (SMB) file shares or MediaWiki instances. Setting
up, monitoring and continuously patching a service is expensive. Especially patching is laborious with
the number of critical vulnerabilities being discovered regularly, yet it has to be done because employees
demand being able to access files remotely, and thus the service needs to be exposed. Instead, we should
embrace the managed solutions and design our system to work with these file storage providers. We
choose to use Dropbox as the FSP for our solution. Dropbox is but one offering among many. This is
why we refer to these as commodity file storage providers. We are not partial to any specific provider
and our system reflects this by having providers be interchangeable as long as they satisfy the following
requirements:
• The FSP stores data and serves as the source of truth for the entire network.
• The data is expected to have high availability and changes should be synchronized to clients that
are online without any considerable delay.
• There should be a client application running on each users machine that provides a local 1-to-1
replica of what is stored at the FSP. The client continuously synchronizes changes between the
local replica and the remote source of truth.
• Access control is not provided by the FSP. All users of the system could potentially use the same
credentials for the FSP instead of each having to create their own user.
• The FSP is honest but curious: It will comply with all agreed-upon protocols, but it might attempt
to look at the data it stores.
==PAGE:35==
4.1 Storage Backend
26
• The FSP should store revisions files, meaning whenever a file is modified, a revision is recorded to
the history of that file. [HJ03] refers to this as journaling or a log-structed file system and notes
that it is absolutely necessary for ensuring integrity and availability of files.
• There should be an application programming interface (API) that allows for prior revisions of a
file to be listed and restored. Restoring a file to a prior revision creates a new revision (so the list
of revisions is append-only).
• It should be impossible for subjects to permanently delete objects and their revisions. Otherwise
it circumvents the mechanism that protects the integrity and availability of files.
Log compression, introduced in [HJ03] as the act of removing invalid revisions in order to minimize
the amount of data stored, is less of concern when using a FSP (at least the providers we have looked
at) for two reasons:
1. The FSP only counts the most recent revision of a file towards the storage budget of the orga-
nization. This means the FSP bears the burden of having to store several revisions of the same
file.
2. The FSP will typically only keep the n most recent revisions (where n depends on the specific
FSP and level of service purchased). This gives users enough time to detect invalid revisions and
revert them but it also prevents the number of revisions from growing indefinitely.
An important thing to note is that we have absolutely no control over the functionality of the FSP.
We can not modify it to check signatures on files or perform any other kind of validation. And we would
not want to, since it has been established that the FSP is not to be relied upon for confidentiality and
integrity. This is a stark contrast to previous work in this area (chapter 3). Some projects have created
their own server implementation or modified an existing one, with the intent of having the server check
signatures on files. This is very notable, because it means that the server now has to assume the role
of a reference monitor on at least some operations (such as writing to files). This adds unnecessary
complexity and it completely subverts the motivation for using cryptographic access control in the first
place.
Our design should reflect that access control will have to be enforced and the integrity of the system
maintained only through the efforts of the clients.

4.2 Security

4.2
Security
We need to determine what it means for the system to be secure. Security is not a binary property
but more like a spectrum. It requires us to consider the context in which the system will be used to
understand what the threats are. Let us consider which actors are in play:
• Users: These are the users that will use the solution to store files and share them with each other,
for example employees in a company. The users can be placed according to some organizational
hierarchy (i.e. various departments, oﬀices etc.)
• File Storage Provider: This is the company (and its employees and subcontracters) that
provides the storage facilities.
• Others: Any outside actor not aﬀiliated with either the group of users sharing files or the storage
provider.
Security can be considered according to the CIA triad: confidentiality, integrity and availability.
The first two have already been described in section 2.2. Availability means that users are able to
access files in a timely and uninterrupted way. Typical threats to availability are unintentional such
as hardware failures or intentional disturbances called denial-of-service attacks (DoS). We assume the
storage backend has the necessary architecture and level of redundancy to provide high availability.
Our concern is primarily with confidentiality and integrity. Not only do we want to ensure that
users within an organization can only access and modify objects if authorized to do so but we also
want complete protection from the FSP. Access control mechanisms, in one form or another, help us
accomplish the former but not the latter. When storing data with an external and potentially untrusted
provider, the only option for ensuring confidentiality and integrity is cryptography.
In practice, we want security measures to be as transparent as possible to the users of the solution.
Otherwise, if users find the measures to be a hassle, they will attempt to circumvent them. For example,
if sharing a file with a colleague through our solution is too complicated, the employee might instead
send the file via email without knowing or considering the security implications.
4.3
Access Control
Access control is a means of enforcing the security level we want. The motivation for using cryptographic
access control has already been covered in detail and it aligns perfectly with the requirement that we
encrypt all files, since they are going to be stored with a FSP. We consider two completely different
CAC designs for our solution.

building a cryptographic access control system. Its policies allow for both flexible and granular levels
of access control and the attribute system is easier to reason about than the model of using keyrings.
It also does not require the distribution of public keys.
It has the drawbacks of requiring a trusted private key generator as well as diﬀiculties surrounding
the revocation of keys, for which it may be possible to devise solutions. At the moment it is still a very
immature primitive and the lack of standardized algorithms and implementations hinders its adoption.
4.3.2
Keyrings
Keyrings have been shown to be a flexible way of supporting various access control structures using
cryptography. Because of the immaturity of ABE we choose to focus on implementing a keyring based
system using modern cryptographic primitives.
4.4
Key Distribution
In cryptographic access control, granting capabilities means having to distribute a key. In the keyring
based system, each user has their own personal keyring and keyrings can link to each other. Assume a
new employee is hired for a department in an organization. How do we onboard this employee and give
them the keys they should have?
One option is to transmit keys out-of-band using a separate channel. This channel has to be secure
in order not to compromise the security of objects related to the keys. Alternatively, we could establish
a shared secret between all pairs of users so that when Alice wants to share a file with Bob, she encrypts
the file key with their shared secret and transmits it to Bob (via email for example). This, however, is
not very scalable as a system with n users requires n2 shared secrets. It also becomes very laborious to
add new users. Another possible solution is to use public-key cryptography. In this model, every user
has their own public/private key pair. To share a key with Bob, Alice encrypts it under Bob’s public
key and only Bob can then decrypt it using his private key.
Depending on the channel, it might also require both the sender and receiver to be online simultane-
ously, which is an inconvenience. We already have a persistent file storage and syncing system in place,
so we propose using the FSP as the key distribution channel, onto which we will add relevant security
measures.
4.5
User Experience
A non-functional requirement for the solution is user friendliness. Not only is it important for security, as
discussed above, but it can also be the deciding factor between whether an individual or an organization
would want to adopt the solution or not. To support the user’s need to browse and interact with the
files, perform operations on them and share files with other users, we need an interface. We consider
the following three options:
• Command Line Interface (CLI): A CLI allows the user to interact with the solution through
their terminal or command prompt. While very powerful yet simple, it will only be familiar to
users with a certain level of technical knowledge.
• Graphical User Interface (GUI): A purpose built application that runs on the users machine
and presents a graphical interface in which the user can download and upload files and share them
with others.
==PAGE:38==
4.5 User Experience
29
• Native file manager: A solution that leverages the native file manager (such as Finder on
macOS or Explorer on Windows). This allows the user to browse their files in a familiar interface.
The secure file system created by the solution will be presented as a directory alongside the users
other files.
The selection criteria is that the solution should be as transparent and require as little training as
possible. It should be noted that having one interface does not necessarily preclude others.
We decide on utilizing the native file manager as the preferred interface to interact with our solution.
It provides a high degree of transparency and familiarity for users and requires little to no adjustment
in their workflows.
As we see it, the best way to integrate with the native file manager is by exposing a file system,
which the file manager can inspect like any other file system. This allows for more portability and
means that the user is free to use the file manager they prefer. One way of doing that is to implement
a userspace file system using FUSE.

example, using the native file manager means the solution also supports the native load and save

4.6
File Operations
While most users understand the general concept of a file, there is a lot of hidden complexity that they
do not consider. Besides reading from and writing to a file, what other operations can be performed?
Not only that, file systems have also introduced additional complexity in the form of links and attributes.
It also begs the question of how directories (also called folders) fit in in relation to files.
These are important aspects to analyze before we design a solution because it affects several com-
ponents including the file system that applications interact with, the FSP and the access control mech-
anism.
It is notable that prior work within this area has given very little attention to supporting other
operations than reading & writing. Our literature review in chapter 3 indicates that this is largely due
to the very high diﬀiculty of reliably and securely supporting other operations when using cryptographic
access control. The following sections cover some of the operations that we will attempt to support or
at least discuss further.
4.6.1
Truncating a File
Truncating a file means reducing its contents to a certain size and discarding everything else. It is an
often overlooked operation that is used a lot by applications. We consider truncation to be a form of
write operation, since it modifies the contents of a file.
4.6.2
Deleting a File
Having write capability for a file should also allow the user to delete that file. The rationale behind
this is that with the write capability, they can already truncate the file, removing all of its contents.
In cryptographic access control, users prove their write capability by signing the files that they modify
and including this signature with the file. This presents a challenge when we want to allow the deletion
of files, because how, then, is the proof communicated to other users?
4.6.3
Moving a File
Moving a file means changing its path from one place to another. It should be noted that renaming
a file is treated as a special case of a move, in which only the last part of the path, the file name, is
changed.
==PAGE:40==
4.7 Summary
31
Move-operations are surprisingly diﬀicult to support in a keyring based system. Consider the ex-
ample where a keyring kr contains capabilities for a file F at path /dir1/file1.txt. Now someone
moves F to path /dir2/file2.txt. This raises several questions:
1. Who is responsible for updating the capabilities in kr so they reflect the new path for F?
2. How are all other subjects interested in F notified of its changed path so they can update their
own keyrings?
3. Which users have permission to move a file? None of the current capabilities cover this operation.
Is everyone allowed to move all files? If yes, how does the system enforce that they comply with
the protocol for how to properly move a file. If not, how is the rule enforced?
Handling file moves is a challenging task that we discuss further in section 8.5.
4.6.4
Attributes & Links
Some operating systems assign additional metadata to files, which are known as extended attributes.
This is not the data of the file itself, but rather some additional data associated with the file and used
by the operating system. Since this is not something that is universally supported, and considering
that most FSPs discard most (if not all) extended attributes of files, we do not see the need to support
this file system feature.
Similarly, the concept of links, in which one file is a link (also called alias or shortcut) that points
to another actual file, is not well supported with FSPs. As such, we believe it can be omitted from our
solution.
4.6.5
Directories
Directories are a way to group files in a common namespace. They are mostly a practical tool that
improves the user experience when dealing with files. There are a lot of file-related tools that do not
consider a directory to be an entity in itself. Instead, directories are simply part of the path that
points to a file in the file system. This can be seen, for example, with git, which does not allow empty
directories to be committed to a repository. We will take a similar approach in our system. This means
that directories themselves have no access control. Any user can create a directory, list its contents and
even delete it. Deleting or moving a directory essentially means carrying out that operation on all of
its descendants.
4.7
Summary
In order to reach our project goal and solve the subproblems that it poses, we propose building a
solution that uses strong cryptographic primitives to implement cryptographic access control, modelled
using keyrings, to ensure confidentiality and integrity of files. The design should be storage provider
agnostic, as long as the requirements specified in section 4.1 are met. The access control policies are
enforced in a decentralized manner as a result of users working to uphold the integrity of the system.
Access control capabilities are granted by distributing keys using a public-key system with the FSP as
a communication channel. To provide convenience and interoperability, the solution should expose files
through a file system that can be browsed in the native file manager application. Finally, it should
attempt to support file operations besides reads and writes such as deletes and moves.

5.2
File Storage Provider Client
The FSP client provides bi-directional synchronization between the FSP and a directory on the local
machine. The client program runs in the background and monitors a directory for changes. When a
new file is added, or an existing one is modified, the client will upload the file to the FSP. Similarly, if a
file is changed elsewhere by another user and updated on the FSP, the client will be notified and begin
to download the new or modified file. In other words, the client is responsible for ensuring that a local
directory remains a mirror of what is on the FSP, which is the source of truth.
Some clients attempt to minimize the amount of data that needs to be transferred by computing
the delta when a file changes. It should not affect our solution but its benefits could be undermined
somewhat due to the use of encryption.
5.2.1
Replicating the functionality of the FSP client
It should be stressed that the FSP client is supplied by the FSP and not created as part of this project.
Most FSPs provide an API that supports file uploads and downloads.
We initially planned to
incorporate this functionality into the file system component, such that it would handle synchronization
with the FSP. We created an initial implementation but quickly concluded that it would add enormous
complexity and remove focus from the problems the project attempts to solve. Instead, we would be
much better served by relying on the existing FSP client and not having to implement networking, retry
logic, file system observation, caching, diﬀing and all the other amenities that the client provides. The
system architecture was revised to include the FSP client and let it handle synchronization.

• Mountpoint: The path to where the file system should be mounted. Our examples will use the
mountpoint /mnt.
• Keypair: The user’s personal asymmetric keypair.

5.4
File Integrity
Recall that clients are responsible for verifying the integrity of files and the validity of writes made to
them, since this is not performed by the FSP. This implies an integrity check occurs on a file after it
has already been stored by the FSP.
==PAGE:44==
5.4 File Integrity
35
Our idea is to have all honest users continuously monitors changes made to files. Changes such as
creating, modifying or deleting a file will automatically propagate to all users that synchronize with the
FSP. Each user will then for themselves check the validity of the file (whether the file was written to
by someone with permission to do so). If an invalid file is detected by a user, they will issue a request
to the FSP to revert that file to a prior revision, such that it is once again valid 1.
This distributed housekeeping process ensures that invalid file operations are detected and reverted
as quickly as possible. While the process happens in cooperation between all users, it does so without
any explicit coordination between them.
Previous projects either use the server as a reference monitor to validate changes, an option we have
ruled out because it contradicts the use of CAC, or have clients check the signature of files. In the
case of the letter, the mechanism is designed such that users only check the integrity of files for their
own benefit. They make no attempt to restore the file on the storage backend for the benefit of every
other user. The check also does not happen until a user explicitly attempts to access the file instead
of happening whenever a file change occurs, which means there can be a significant delay between the
unauthorized operation and detection of it.

Conclusion
The Only Future Updates-proposal is susceptible to race conditions, since it has to
always run before the FSP client does in order to pick up on every change that occurs. The operating
system might be able to guarantee this, but it is a fragile promise, and it does not cover cases where
the daemon crashes or is terminated and has to be restarted.
As our final solution is a proof-of-concept, we choose to implement the Only Future Updates-
proposal for its simpler design and limited trade-offs. We encourage future work on topic to look into
the other proposals as a way to optimize the start up times of the daemon.
5.4.2
Integrity Daemon
The daemon should be designed to tolerate a large amount of concurrent file changes, as multiple remote
file changes can occur simultaneously. There are several critical considerations to make when designing
the verify-and-restore algorithm that the daemon will run:
1. The algorithm spans across the FSP and the client. Thus in a system with n honest users, an
invalid write to a file will trigger n instances of the algorithm.
2. There is no coordination between instances of the algorithm. This means the environment is
highly likely to experience race conditions, when the effects of one algorithm instance may ”spill
over” into another instance.
A received response is likely already deemed outdated from the
changed made by another algorithm instance. The algorithm should take appropriate measures
to correctly handle the outdated data it receives.
We will detail two design proposals for the verify-and-restore algorithm.
After implementation
and evaluation of the first proposal, we discovered a number of ineﬀiciencies that would hamper the
effectiveness of the first proposal in some cases. This will be elaborated on in section 5.4.2.2. The
second proposal attempts to account for this and is used in the final solution.
5.4.2.1
Algorithm Proposal 1: Iteratively Restore
Recall that revisions are maintained by the FSP and that the history is append-only. Each revision,
rev, is a blob of metadata containing (among other properties) a unique revision ID, revID and a hash
of the corresponding file’s content, revID.hash. The size of a rev is constant, it does not depend on
the size of the file it represents. It is important to note that this content hash is computed by the
FSP and associated with a revision. As we will detail in section 6.4.2.5, we also compute the hash of
the file for the purpose of signing it. These two hashes, the FSP content hash and our hash, are not
necessarily the same (unless the FSP uses exactly the same cryptographic hash function that we do for
in implementation, which is not the case for Dropbox).
==PAGE:47==
5.4 File Integrity
38
The algorithm starts when an invalid revision of a file, F, is encountered. It works by iteratively
restoring F to its prior revision until a valid revision is encountered.
The algorithm is as follows:
1. Initialize a counter k = 1 to be incremented for every subsequent encounter of invalid revision of
F.
2. Fetch a list of F’s revisions [revx, revx−1, revx−2, ...], where revx is the current revision.
3. Request the FSP to restore F to the 2k −1’th revision.
4. If the restored revision is also invalid, increment k by 1 and repeat from step 2.
Because of step 3 and 4, we first jump 1 revision back in the revision log, then 3, then 5 and so on.
This is necessary to account for the fact that a restore also makes a new revision, so for each iteration
of the algorithm, the revision history has changed. Figure 5.7 illustrates the flawed case where we only
jump 1 revision back for each invalid file encountered. It illustrates how two illegal sequential write-
operations would force the flawed algorithm into an infinite loop, where it would alternate between
restoring revision 1 and 2 forever. We avoid this by jumping back by 2k −1 for each sequential invalid
revision.
Unfortunately, the algorithm stated above is prone to race conditions, as it incorrectly assumes
that the head, revx, of the fetched revision log in step 2, corresponds to the revision of F currently
downloaded. We illustrate the race condition in figure 5.8.

5.4.2.2
Algorithm Proposal 1 Evaluation
This proposed algorithm has two issues:
1. Performance: It turns out that the restore-API of the FSP can take upwards of 5-10 seconds to
complete, after which the new revision needs to be downloaded by all clients. If there are several
invalid revisions in sequence, this adds considerable overhead.
The algorithm does not cache
results, meaning it might download the same revision multiple times across separate invocations
of the algorithm.
2. Replay-attack: Any user is allowed to request that the FSP restores a file to a previous revision.
Restoring an invalid revision will be detected by other users and they will revert the change.
Restoring a prior but valid revision, on the other hand, will be seen as okay by users (because the
revision is valid). This means that a malicious user, even if they do not have the write capability
for a file, can restore the file to a previous revision made by someone that do. While the malicious
user can not change the file contents to something of their own choice, they still violate the
integrity of the file by changing its contents to something that it is not supposed to be.

5.4.2.4
Algorithm Proposal 2 Evaluation
The proposed algorithm 2 resolves the issues that were discovered in the first algorithm. It restores a
file if an invalid change is detected but also if someone, whether with malicious intent or not, attempts
to replay an older revision.
The fact that a file is not allowed to have two revisions with the same content hash, unless it is a
result of restoring the file because of an invalid change, seems to indicate that the contents of the file
can never change back to something that it was previously. This however, is not the case due to our
use of probabilistic encryption, as we will detail further in section 6.4.2. In short, encrypting the same
file multiple times will always yield a different ciphertext output and thus a different content hash.
With regards to performance, this algorithm is expected to be more eﬀicient since it only needs to
download and verify an invalid revision once.

5.4.5
Remarks
Since there is no coordination between instances of the daemon, the result is that every online user that
sees an invalid change will attempt to revert it simultaneously. There is an amplification effect in play
where one malicious user can direct every honest user towards the FSP.
While we have made the assumption that the FSP is highly available and can tolerate a lot of
requests, things might be different in practice. If the FSP sees a lot of traﬀic using the same access
token they might be inclined to rate limit that traﬀic, which affects the availability of their service (and
thus the files). The severity of this is hard to estimate as it depends on a lot of factors. A practical
solution might be to introduce some kind of coordination or leader-election scheme such that only one
or a few daemons attempt to revert any given invalid change. This, however, is out of scope for our
proof of concept solution.
Another concern with this design is that it puts malicious users and honest users into a ”tug of war”
with each other where one side might overwhelm the other. Ideally, we want honest users to be the
victors, but if the malicious users produce so many invalid changes that the honest users can not keep
up with their validation, it can compromise the availability of files until the honest users are able to
catch up. We will discuss this in more detail in chapter 8.
5.5
Capabilities
Capabilities are stored separately from the object they refer to, and should encapsulate the necessary
information for a subject to know what object the capability relates to and what operation it permits.
Capabilities should also be small in size, as they will be transferred between users. The verify-capability
(the ability to verify the validity of a file) in particular are meant to be broadcast to all other users.
Based on these requirements we propose the following tuple-based schema for an object O:
(P, T, K)
where P is the (remote) full path of O, T is the type of the capability and K is the key. T can be
either read, write or verify. From the file-extension of P, the subject will be able to determine the type
of O, i.e. whether it is a keyring, or a text-file etc.
A file’s capabilities are generated upon the file’s creation and immediately put into the user’s keyring.
5.6
Keyrings
A keyring is a list of capabilities. The list is stored in its own file denoted with the .keyring extension.
Figure 5.12 illustrates an example keyring consisting of three capabilities, where one of them refers to
another keyring. This depicts the hierarchical nature of keyrings: entries can refer to other keyrings,
allowing the subjects to build graphs of keyrings.
Each user maintains a local keyring containing all their capabilities. This keyring works as a database
for retrieving capabilities based on their file path or type. Any capabilities shared with them can be
added to this local keyring.
5.7

FSP and its synchronization mechanism is an obvious communications channel between users. Our
design for doing this is based on a concept of postal boxes. These are remote directories located under
/users/ID(S) where ID(S) denotes a unique identifier for subject S.
Protocol 1 demonstrates the postal box setup performed by a subject once when they join the

Description: One subject (S2) sends a capability to another subject (S1) securely.

2. S2 encrypts [cap]: c = Enc([cap], pub1).
3. S2 chooses a random identifier r and stores ciphertext c as a file at /users/pub1/r.
4. /users/pub1/r is synchronized to S1 who detects this change and and decrypts it: [(p, t, k)] =

5. S1 checks that the capability works for the indicated operation for the file stored at p. If successful,

6. S1 may choose to remove the file uploaded to its postal box afterwards.
Note that protocol 2 makes the assumption that the sender knows the public key of the recipient.
This is reasonable for a proof of concept solution but an inconvenience in practice. We will come back
to this in section 8.3.
Verifying whether a capability works can be done in different ways depending on what operation
the capability permits:
• read: Attempt to decrypt (parts of) the file. If this succeeds, the capability works.
• verify: Verify the signature on the file. If it succeeds, the capability works. Otherwise the
capability or the signature is wrong, but we can not know which.
• write: This is more tricky as we want to be able to non-destructively check the capability,
meaning we can not make changes to the file to do so. A solution is to utilize the fact that in
elliptic curve cryptography, which we use for signatures, the public key can be derived from the
private key. This means we can check a write-capability by first deriving the verify-capability and
then checking it as described above.
Recall that integrity checks are a joint effort. Therefore, it is in an honest user’s interest to broadcast
their files verify capabilities to all others so that they can also make integrity checks on the files. To
support this we propose a reserved path, /users/shared, for a shared postal box that all users observe
for changes. When a new file is created, the user can then distribute the file’s verify capability to the
entire network without knowing the public keys of everyone. Note, that all capabilities saved under
/users/shared are unencrypted since it is a broadcast intended for everyone. This is of no concern,
since verify capabilities provide no way to compromise either the confidentiality or integrity of files.

1. No external channels: Note that both protocols rely on the existing channels between clients

2. Prone to malicious deletes: Nothing stops a malicious user from deleting files in another user’s

deletes, the postal box owner (and anyone else) can restore files that were deleted assuming they
know the paths of the deleted files. We will come back to this assumption in section 7.1.1.
3. Sending a capability is partially traceable: All postal boxes in the network are public,
meaning that it is public knowledge how many capabilities someone receives (lower bounded). It
is not public who the sender of the capability is, nor what the capability is for. If a user wishes
to hide the number of capabilities they receive, we propose they can add arbitrary noise to their
own postal box by continuously sending encrypted noise to their own postal box. This ”smoke
screen” makes it infeasible for outsiders to guess the actual number of capabilities received.
4. Sender must know public key of receiver To send a capability requires prior knowledge of
the recipients ID (i.e. their public key). This in itself is not unreasonable (how else would the
sender address the receiver?) but public keys are hard to memorize and not user friendly. In
section 8.3 we discuss possible solutions to this.
5. Receiver does not know sender: In the postal box design the sender is anonymous to the
receiver and anyone else in the system. The capabilities that are sent, however, can be verified
(as desrcibed previously) such that the receiver knows that they are valid.
5.8
File Deletions
As discussed previously, having a write capability for a file also means being able to delete the file. The
deletion of a file is a change that should be verified similarly to when a file is modified. This implies we
can not let a file be deleted entirely, as there would then be nothing to perform the verification on.
We specify the requirements for file deletes as follows:
1. Only users with a write capability for a file are allowed to delete that file.
2. Users must be able to verify that the deletion was authorized.
3. If a file is invalidly deleted, it should be restored.
4. A deleted file should no longer be presented to the user through the cryptographic file system.
5. The solution must not be susceptible to replay-attacks, meaning the proof of deletion must be
tied to the file that was deleted. 3
Our proposal for how to delete a file is formalized in protocol 3. In short, the idea is to delete a file
but leave behind a note saying that the file was here, and sign the note using the write capability for
the file that was deleted.
3We find this particularly important to highlight since prior contributions that introduce a delete-operation, are
susceptible to this kind of attack.

5.9 Design Summary

Protocol 3 Deleting a File
Description: A prover wants to prove to a verifier that their deletion of a file, F, is valid. The prover
possesses the write capability of F, and the verifier possesses the verify capability F.
Definitions: The verify capability is pub and the write capability is priv. We denote s = sign(m, priv)
as a signing function to create a signature s for a message m using the private key priv. verify(s, m, pub)
returns true if and only if s is a valid signature for m made with priv. The path of F is denoted by
path.
The protocol:
Prover:
1. The prover signs the path of F with their write-capability: s = sign(path, priv).
2. The prover stores s at path || .deleted (i.e. it adds the deleted extension to the file path).
3. The prover deletes the file.
Verifier:
1. The verifier receives the file with the .deleted extension and reads s from it.
2. The verifier confirms that verify(s, path, pub) is true. If it is, the delete is accepted. Otherwise,
the delete is invalid and the verifier will request the FSP to restore F to revision rev.
The verification of file deletes works very similar to that of file modifications, which is why we let the
daemon handle both. The steps carried out by the prover in protocol 3 are handled by the cryptographic
file system.
We do not allow .deleted files to be deleted. If this happens, the daemon should automatically
restore the file. Note that the .deleted file does not share revision log with its non-deleted counterpart,
since the .deleted file makes up a new path and revision logs are keyed by path. In other words, if
thesis.pdf.deleted were to be physically deleted, the daemon should restore the file based on the
revision log of thesis.pdf.deleted and not thesis.pdf.
Since deletes are tied to a specific file path it prevents malicious users from taking a signature and
using it to mark other files as deleted.
Adding the deleted extension to files that have been deleted does two things:
1. It allows the daemon to know where to look for the signature of a deleted file when it needs to
verify the delete.
2. It means the cryptographic file system can hide these deletion-markers from the view of the user.
The result is a protocol that allows an authorized user to delete a file and for other users to verify
and ensure that the delete is valid. It should be noted that our definition of deleted means that the
contents of a file are gone and the file is hidden from view. There is still a trace in the form of the
deleted marker indicating that a file with a given name once existed. Another thing to be aware of is
that Dropbox will permanently delete files after 30 days so an invalid file delete has to be detected and
reverted within this timeframe. It seems unlikely that not a single honest user would be online in the
system for 30 days so we do not see this as a major concern.

6.1
Programming Language
The entire code base is written in JavaScript and runs on the Node.js runtime.
Node.js is single-
threaded and enables parallelism by dispatching work to its event-loop. The event-loop allows for easy
development of applications dependent on asynchronous control flows, which is characteristic for the
applications that we are developing. Node.js supports the async/await programming paradigm, which
allows us to write asynchronous code as if it were synchronous. This makes it much easier to reason
about asynchronous code and reduces the number of bugs.
The Node Package Manager (npm) accompanies the Node.js runtime.
It gives access to a rich
package ecosystem, allowing developers to easily re-use the work of others. All our direct dependencies
are listed in the package.json file and includes wrappers for the Dropbox API (dropbox), FUSE
(fuse-native) and libsodium (sodium-native). Node.js also has a number of built-in packages such
as fs, which has an extensive interface for interacting with the local file system. Node.js has a foreign
function interface (FFI) called N-API that allows JavaScript code to interact with native modules written
in C, C++ or some other systems programming language. Further, Node.js runs on all major platforms,
including macOS, Linux and Windows, and architectures.
We chose to build the application in Node.js because its extensive package ecosystem, asynchronous
model and our prior experience enables a rapid development cycle and allows us to quickly iterate on
ideas.
Node.js has two versions of its filesystem package: fs and fs/promises. The promise-based version
supports the async/await paradigm but, compared to the older callback-based version, it also eschews
the use of file descriptors. Instead it wraps them in its own type, hiding the underlying file descriptor
(which is simply an integer). This hampers its interoperability with FUSE, which largely works with
file descriptors. Our solution is to build a small wrapper around the fs package such that it can be
used with async/await while still using plain integer-based file descriptors. The wrapper can be found
in ./fsFns.js.

6.4.1
Handlers
This section lists various handlers that respond to file system requests from FUSE. The list only covers
the handlers our solution implements. Most handlers in FUSE are optional and can be implemented as
needed while other handlers have been deprecated and should be replaced.
getattr
Returns metadata about a file such as its size and timestamps. The attributes come from the
file in the FSP directory but we adjust the size so that it reflects the size of the file before encryption
and not after encryption. See section 6.4.2.1 for a discussion about this.
readdir
Returns the contents of a directory. Our handler hides some files and directories from the
user such as postal boxes, keyrings and deleted files.
mkdir
Creates a new directory in the FSP directory.
unlink
Deletes a file. This involves replacing the file with a signed ”delete marker”.
rmdir
Removes a directory. The operation is not recursive, meaning the directory must be empty.
rename
As covered previously, renaming files in a keyring-based system is highly complex and not
something the solution currently supports. This handler will throw an EIO (input/output) error to
indicate that the operation is not supported.
chmod
Changes the permission flags of the files. This call is forwarded to the underlying file in the
FSP directory but it has nothing to do with the cryptographic access control system that our solution
provides. It only affects the permissions of the file on the local machine.
chown
Changes the owner (user and group) of a file. Similarly to chmod, this only affects the permis-
sions of the file on the local machine.
truncate
Reduces a file to a given size (which can be 0) by discarding any excess content. Since this
means modifying the file, write capabilities are required. After truncating, the file signature must be
updated. We can not truncate on the ciphertext, so unless the size is 0, the file must first be read and
decrypted before the plaintext can be truncated, encrypted and written back to disk.
open
Opens a file descriptor to a file and returns it. FUSE will later pass this file descriptors to other
handlers such as read or write.
read
Reads a number of bytes from a file from a given position. The handler will decrypt the data
as it is being read. If the user does not have the appropriate capabilities, an error is thrown. Section

When FUSE calls the read handler, it supplies a length and a position argument, indicating which
section of the file it is requesting to be read. The maximum read size is configurable and may vary
between systems. We make absolutely no assumptions about either argument. Our read handler must
support random access reads. This means being able to read any number of bytes from an arbitrary
position in the file (until the end of the file).
The same is the case for the write handler, which is given a buffer of data and a position in the file
to which it should be written.

While programs usually read and write files sequentially starting from the beginning of the file, we
have to support random access for cases where the application ”skips”. For example, consider watching
a video. Initially, the file is read sequentially starting from the first byte. If the user decides to skip
ahead, the video player application will request to read from the point in the video to which the user
has skipped. Video, however, is still a very linear file format. Other applications, such as databases,
have entirely non-linear file formats, which makes them even more reliant on random access reads and
writes.
This is fairly trivial to support when dealing only with plaintext files, as illustrated by figure 6.1.
In our solution, however, the cryptographic file system is responsible for transparently decrypting and
encrypting files on-the-fly, as they are read and written, respectively. This is not trivial and necessitates
a careful choice of which cipher and mode to use. Since FUSE has a maximum read and write size, it
often needs to call the read and write handlers multiple times to read or write a file. For example,
FUSE reads 4096 bytes at a time by default, meaning in order to read a 1 MiB file, it issues 255 calls
to read.
The following sections detail some of the challenges that were encountered during the development
of the file system. They help provide an understanding of the ideas behind the final implementation.
6.4.2.1
Determining file size
With some ciphers, the size of the ciphertext is not necessarily the same as the plaintext. This is because
the size of the input to the cipher must be a multiple of its block size. To work with inputs that do
not align perfectly, padding is added to the file before encryption and removed again during decryption.
This is a problem when implementing a file system, since we are often interested in determining the size
of the plaintext without having to (or maybe not being able to) decrypt it. FUSE calls the getattr
handler to get information about a file including its size. It uses this information both so that the file
manager can report the size of the file but also when reading. If we return an incorrect file size, FUSE
might call the read handler with a range that is outside the bounds of the file.
Older attempts at cryptographic file systems such as eCryptfs ([Kir06]) try to work around this by
storing the size of the plaintext in a large 8 KB header in the file. In order to determine the size of
the plaintext, the file must be opened for reading and the first number of bytes read. This is highly
ineﬀicient and not appropriate for the getattr, since it is called frequently and for every file in a
directory. Querying the metadata of a file (such as its size on disk) is much quicker than reading from
the file.
The newer EncFS ([Gou]) fares better and cites this as one of its major advantages over eCryptfs. It
circumvents the need for padding by combining a block cipher with a stream cipher. The block cipher

This was initially implemented using a concept in libsodium called secretstream. This is a stream
of related messages which are encrypted and decrypted individually using ChaCha20-Poly1305. Each
message, in our case a chunk, is encrypted using the same key and a nonce that is computed determin-
istically from the authentication tag of the previous block and a counter value that corresponds to the
message number (although offset by a randomly chosen constant). The idea behind computing each
consecutive nonce using both a monotonically increasing counter as well as a part of the authentication
tag of the prior block is that messages in the stream are tied together. Otherwise it is possible to violate
the integrity of the file by reordering, duplicating or removing messages from the stream. This is similar
to the issue that occurs when encrypting files in ECB mode. As we will see later, this vulnerability is
not present in our solution because we also employ digital signatures.
In this approach, random reads are possible with the caveat that the tags (but not the entire
encrypted chunk) of any prior blocks must be read from disk. No decryption of prior blocks is necessary.
The tags can be cached in RAM as an optimization. While this is much more eﬀicient than having to
read and decrypt the entire ciphertext up to the read point (as would be necessary with something like
AES in CBC mode), it still adds memory and CPU overhead.
Splitting the file in chunks greatly limits how much of a file that needs to be re-encrypted and it
ties in much better with the way FUSE issues calls to write. The best case scenario is when complete
chunks are appended to a file, since no re-encryption of prior chunks is necessary.
The worst case
scenario is when a write is made to the first chunk of the file, since that chunk and every subsequent
one needs to be re-encrypted because of the way secretstream ties them together. This means the file
system performs well for sequential writes but suffers from write amplification when writes are random.
More importantly, the use of a counter-mode scheme has a major flaw. When a chunk is written to
it gets re-encrypted using not only the same key but also the same nonce. The nonce used for a chunk
at a given position in the file remains unchanged as long as the chunks that precede it are not written
to and thus re-encrypted. As demonstrated in chapter 2, re-encrypting two different messages with the
same key and nonce causes stream ciphers to catastrophically fail.
This leads to the realization that we can never have both a predictable nonce for a chunk at a given
position and at the same time support random writes. Random writes means chunks get re-encrypted,
and re-encrypting under the same key must be done with a new nonce. The only option is to use a
randomly generated nonce for every chunk and to choose a new one every time a chunk is re-encrypted.
This is possible but requires a carefully considered implementation.
The only encryption scheme available to us that safely allows the use of a randomly chosen nonce
for each message encrypted under the same key is XChaCha20. In the next section we will describe the
final implementation based on this scheme.

6.4.2.5
Signing files
So far we have covered how the cryptographic file system uses encryption to protect the confidentiality
of files. In this section, we will detail how digital signatures are used to protect the integrity of files.
In the cryptographic access control scheme, each file has an associated private key that represents a
write capability. A file has been written to legally if it is signed by the correct key. Implementing the
signing of files in the cryptographic file system requires the following three problems to be considered:
• Where is the signature stored?
• When is the signature produced?
• How is the signature produced?
Before describing when and how the signature is produced, we look at where the signature is stored.
If a user has a file, they should also have the signature of that file. We considered a number of options.
At first we looked at using extended attributes. These are small fields of metadata that can be associated
with a file. While most operating systems support it, file storage provider clients will often ignore these
attributes (or only support a predefined set of attributes) and not synchronize them. This would mean
that the signature is lost. Another option is to store the signature in a separate file alongside the file
that it represents. The file can be hidden so that the user does not see it in their file manager. This
approach, however, can quickly introduce race conditions in the system since we have no control over
the order in which files are synchronized. It could be that the file is synchronized and the signature-file
only follows seconds later. Finally, there is the option of storing the signature directly in the encrypted
file. The signature is prepended to the file so that it is in a predictable range of the file (i.e. always
the first 64 bytes). A prepended signature is arguably preferable to a appended signature as appended
signatures don’t lie in a predictable range. Also, it will be (potentially only partly) overwritten when
new writes are appended to the file.
While it requires some care in how the file is read, this approach seems like the most reliable in a
system where files are synchronized between users. Figure 6.3 illustrates how the signature is stored in
the file.
This leads to the introduction of an invariant in the system:
A file must always have a signature for its content
The reason for mandating this is two-fold. First, if the file has a signature only in some cases (such
as after being written to, but not after it is created) it makes it diﬀicult to reliably determine the
correct file size. Secondly, we have no control over when the FSP client begins to synchronize a file.
Our testing shows that the Dropbox client appears to have a debounce of a few seconds. This, however,
is an implementation detail that should not be relied upon since Dropbox can change it without notice.
Initially, we would compute and prepend the signature when the release handler was called. This
was to allow for multiple sequential write-calls before signing the content. However, this could cause
problems if the FSP client starts synchronizing the file after we have handled any number of calls to
write but before release has been invoked. The result would be that the invariant is broken. Instead,
we have to sign the file in every call to write to ensure that the signature always reflects the contents
of the file.
Thus, the write handler makes two writes to the underlying file: one being the ciphertext and the
other being the signature. As depicted in figure 6.6 this leaves a small window in between the two
writes where the invariant is not met. We solve this problem by locking the file in the FUSE write
handler and releasing the lock before returning from the call.
Another important thing to consider is that the signature should be computed whenever any file
modification is carried out. This means it must also be done in handlers such as create and truncate,
which in turn should also lock the file for the same reasons as for the write handler.
The signature needs to be computed over the entire file, excluding the first 64 bytes that hold the
signature. As described above, this has to happen on every call to write, so the implementation must

Ephemeral Files and Resource Forks
Some applications (like Microsoft Word, Apple Preview or Apple TextEdit) make use of what we will
call ephemeral files and directories.
These are typically hidden and only exist for a short moment.
To our knowledge, applications use these files to auto-save or cache file changes or to make their own
copy-on-write mechanism.
Another case where these show up is on macOS, which for legacy reasons use something called
resource forks. On file systems that do not natively support these, they are represented using a hidden
file. Through research and our own experimentation we conclude that resource forks can safely be
ignored on macOS. Therefore, our handlers throw an ENOENT (no entry) error for every resource fork to
indicate to the caller that they are not supported. The base name of a resource fork always starts with
”._” which we use to distinguish them from regular files.
These ephemeral files are generally essential for applications to function properly and need to be
supported by the file system. We have to consider which capabilities should apply to these files. Since
an ephemeral file is tied to some other file, our solution is to re-use the capabilities used for the main
file (i.e the one the user opened in the application) for all related ephemeral files and directories. A
real-world example is shown below where the file /egon.jpg is open and manipulated in Apple Preview.

7.1
Qualitative Evaluation
We will evaluate our solution against the previously stated requirements.
The solution uses cryptographic access control to protect the confidentiality and integrity of files
against any unauthorized actor, including the file storage provider. We have demonstrated the use of
Dropbox as a file storage provider, and while the solution has been designed to be provider agnostic,
we have not attempted to use the system with another provider. We do believe, however, that any
provider that supports the specified requirements can be integrated in a matter of days.
Capabilities are modelled as read, write and verify and can be shared between users. While it is our
intention to enable this through the native file manager, the solution currently only supports it through
a command line script.
Invalid file operations are mitigated by daemons run on client machines, that continuously react to
all file changes. They determine a revision’s validity based on its signature and restore an invalid file
back to its most recent valid revision. This effectively denies unauthorized operations.
The solution exposes a file system that transparently handles the details surrounding the crypto-
graphic access control mechanism. It works well with the native file manager and other applications
and supports most operations in a manner that feels performant to the user. The exact performance
figures will be quantified in section 7.3. The abilities of the file system are listed in table 7.1. While the
system supports deletion of files, it does not support moves. In section 8.5 we will discuss the challenges
of this issue.
One requirement states that the user hierarchy should be completely flat (i.e. no group of users
with special privileges). This is the case within the system, where every user holds the same access
token and nothing else. There is, however, an external factor in that at least one user has to sign up for
an account with the FSP. This means they have login credentials (username and password) and thus
permission to take down the account at any point. Unfortunately, we see no way to avoid this with any
of the FSPs that we have encountered.
The solution is built using technologies such as FUSE and Node.js that are available on both macOS
and Linux. While it works on both platform, our testing has been most extensive on macOS so we
expect there to be quirks and unknown issues on Linux.

7.1.1
Known Issues
During the evaluation of the solution a few issues have been discovered. We believe these can mostly
be resolved by slight changes to the design and/or implementation. None of these have an impact on
file confidentiality or integrity.
Reuse of paths
Once a file has been created at a path, the path is reserved for the capabilities of
that file. Imagine a file F with path p and capabilities capsF which are distributed to users in the
system. When F is deleted, a marker with the .deleted extension is put in its place, hence freeing p.
Then a new file F ′ with capabilities capsF ′ is put on p, and capsF ′ are distributed. Since we can not
guarantee that all users receive the new capabilities, it partitions users between those expecting p to
have capabilities capsF and those expecting it to have capsF ′. This could trigger an infinite loop where
the file at p is restored between F ′ and F.
The immediate solution would be to let users delete their capabilities when the associated file is
deleted. We see this solution as closely related to file moves and revocation to be discussed in section
8.5 and 8.6.
Dropbox revision list limitations
The Dropbox API documentation ([Droa]) states that no more
than the 100 most recent revision IDs for a file can be retrieved. While our testing (see the script in
1See section 8.5 for a discussion on this matter
==PAGE:75==
7.2 Software Testing

appendix D) shows that it is possible to restore to earlier revisions, it requires knowing the ID of that
revision since it can no longer be retrieved from Dropbox. This could cause problems with the daemon
in situations where a lot of invalid writes are performed but no honest users are online to revert them.
Dropbox offers a 10-year revision history (at en extra cost) ([Drob]), but it is unclear if this is helpful
as long as the API will only return the 100 most recent IDs.
Deleting postal box contents
The FSP only provides a revision history for files but not for direc-
tories. If a capability is placed in a user’s postal box (a directory), and then deleted before the user
sees it, they will not know that it was ever there. The problem arises because capabilities are placed
under a random path not known to the recipient beforehand, so the recipient is unaware of what path
to look up revisions for. To solve this, we propose turning postal boxes into files. Instead of placing
messages within the postal box, the postal box file is overwritten with the message. The revision history
allows the recipient to retrieve previous messages that were overwritten, and the file is now located at
a predictable path.
Deleting a directory sometimes not possible
As previously explained, a directory can be deleted
freely as long as it does not contain files. Once it contains files, each file must be deleted before the
directory itself can be deleted. The problem is that file deletes, as they are currently designed, leave
behind a marker, which itself is a file. Removing this marker is an integrity violation and the system
will work to prevent such an action. This prevents a directory from being deleted once a file has been
created in it. A possible solution might be to store all markers in a common directory separate from
where the deleted file that they refer to resided.

7.3
Performance Testing
In this section we test the performance of the cryptographic file system using a number of benchmarks.
There is some overhead expected for a multitude of reasons including the fact that the file system adds
an extra layer of abstraction and runs in userspace and the on-the-fly encryption, decryption and signing
that occurs. While this is expected, we wish to determine how much overhead there is and also see if
adjusting certain parameters will influence the results.
It should be stressed that we are testing the local performance of the file system and not how quickly
it is able to perform actions that involve the FSP (such as synchronizing files or rolling back invalid
writes). These are largely dependent on network performance and the performance of the FSP, both of
which are factors that are out of our hands.
7.3.1
Methodology
The tests are performed on a 2018 MacBook Pro 15” running macOS 12.2 (21D49). The machine has a
2.6 GHz Intel Core i7 8850H, 32 GB of DDR4 memory running at 2400 MHz and a PCIe 3.0 x4 NVMe
SSD (solid state disk) formatted with APFS (Apple File System). The runtime environment is Node.js
v17.4.0.
Tests are performed using AmorphousDiskMark version 4.0(2). We test the read & write speeds for
the following scenarios:
• Sequential, 1 MiB blocks, queue depth 8 (3)
• Sequential, 1 MiB blocks, queue depth 1
• Random, 4 KiB blocks, queue depth 64
• Random, 4 KiB blocks, queue depth 1 (4)
Each test is run on a 64 MiB file. The result of each test is an average over five consecutive runs
of that test. For the sake of comparison, all tests are also performed directly against the computer’s
disk. This allows us to see how well the cryptographic file system performs and to determine whether
the SSD of the computer is a bottleneck.
A queue depth greater than 1 allows more than one request to be issued to the drive at once, which
can enable the application to exploit parallelism. Modern SSDs consist of multiple flash memory dies
and the controller will read from and write to multiple (often 8 or 16) simultaneously, which is what
helps make these drives fast. The FUSE bindings we use are multithreaded and the CFS interacts with
the SSD, so we expect the CFS to also benefit somewhat from a queue depth that is greater than 1.

Reflections on results
It should be noted that this kind of test is whats called a synthetic benchmark, which acknowledges that
the way it tests the system does not reflect the workloads encountered during real-world usage. As we
have already established, the CFS supports many tasks such as editing documents or images or playing
back a high resolution video. With that said, the results show raw performance numbers that are lower
than expected. Or, put differently, there is a gap between the performance of the underlying SSD and
our CFS that is fairly significant. The write speed on the SEQ1M QD8 test is also unexpectedly low.
By looking into how the benchmark application works, we learn that it reuses an existing file for that
particular benchmark. Instead of sequentially writing to a new file, it sequentially writes to a location
within a file that has already been allocated. In the CFS, writing to any location within the file (and
not at the end of the file) counts as a random write, regardless of whether the write calls are issued in
sequence. This explains the low performance on this particular test and it means one should instead
refer to the SEQ1M QD1 test to get a sense of the sequential write speed of the CFS, as this test does
not exhibit the same behavior.
As previously detailed, the cipher we use, ChaCha20, does not have hardware acceleration. This
might lead to concerns that it hampers the performance of the system. To see whether this is a valid
concern, we test the throughput against AES. We use the OpenSSL (version 1.1.1m) cryptographic
library since it has a built in benchmarking tool and its performance should comparable to that of
libsodium. It only supports ChaCha20 and not XChaCha20 but the size of the nonce is not expected to
make a difference in terms of throughput. The benchmark is performed using the command: openssl
speed -evp {aes-256-gcm, chacha20-poly1305} [-decrypt]. Table 7.2 shows the results. While

8.1
Integrity Check Remarks
In section 5.4 we proposed a design that actively maintains file integrity by reverting invalid changes.
Recall that the idea is to handle unauthorized changes as they happen instead of only discovering them
when accessing an affected file. We did not discuss on whether it is acceptable to do so in a reactive
manner (as opposed to rejecting invalid operations before they can occur).
First, we turn towards other established distributed systems with a integrity scheme similar to ours.
Perhaps most noticeable is the Bitcoin network [Nak08] in which peers co-operate to derive new blocks
on the blockchain1. Any peer can broadcast what they think the next block should be, which causes
the chain to branch. To decide which branch to go with, peers use a consensus mechanism based on
proof-of-work, meaning that the attacker need control of a majority of the network’s computational
power if they were to overthrow the network. Over time, peers will reach a majority vote and the
network converges towards a block. Peers are financially incentivized to follow this decision.
A somewhat similar dynamic between users is at play in our system. Honest users are incentivized
to keep files in a valid state. This includes not only their own files but also those of others, because
there is an expectation of solidarity from everyone.
Malicious users are kept at bay as long as the honest users that are online are able to verify and
revert invalid changes as fast as the malicious users can produce them. It is diﬀicult to determine who
has the upper hand here, because the FSP adds a considerable delay between one user making a change
and other users seeing it. It is not just a matter of available computing power. One thing that can make
this tug of war challenging are Sybil Attacks [Dou02]. In a Sybil Attack, the attacker undermines the
reputation mechanism of a network by creating a large army of identities to obtain a disproportionately
large influence. There is nothing preventing a malicious user from using tens or hundreds of clients to
produce invalid changes. To counter this, users might want to run always-on instances of the daemon.
We will call them verification servers. The idea is for there to always be one or more daemon instances
that will immediately detect when an invalid change is made. Similar to how the daemon works on other
users machines, there is no need for the verification servers to have read or write capabilities. They
also help alleviate concerns about invalid file deletes not being detected before 30 days have passed (as
described in section 5.8).
The reason for mentioning Bitcoin is to show that there are large scale open networks where oper-
ations are validated after they occur. The idea itself is a viable solution but comes with its own set
of challenges. Introducing validation servers could help alleviate some of these. While these might
appear to be similar to a reference monitor, it is important that they do not function as gatekeepers
that proactively prevent an operation from being performed. Instead, they react to changes and revert
them if necessary.

8.1.1
Beyond a 1-to-1 Mirror
Until now we have assumed that the FSP client always locally mirrors what is stored remotely. This
reduces lag when opening a file since it has already been downloaded. To work around scenarios where
the user has limited local storage, most FSPs provide some kind of selective sync feature. 2 It uses a
heuristic to keep frequently used files around locally while the rest are downloaded on-demand.
We have not investigated the consequence of this mechanism on the integrity daemon but we expect
that it might limit its eﬀicacy. If a user runs out of storage, it is equivalent to them not being online
and participating in the integrity checks. A verification server with enough storage capacity would be a
possible solution for this. Another approach would be for users to continuously shuffle which files they
have locally but this could become very bandwidth intensive.
8.2
FUSE optimization
As the performance testing in section 7.3 showed, FUSE adds considerable overhead while the crypto-
graphic layer also contributes some. In this section we will detail an idea that might be worth exploring.
Writes currently happen in chunks of 4096 bytes to match the write size used by FUSE. This,
however, appears to be configurable using the max_write option. The manual for FUSE states the
following:
Set the maximum number of bytes in a single write operation. The default is 128 kilobytes. Note,
that due to various limitations, the size of write requests can be much smaller (4 kilobytes).
This
limitation will be removed in the future.
Similarly, there is a max_read option. We have tried experimenting with other values but were unable
to make FUSE heed the option. Instead we leave it as an option for future work. We expect larger
values to result in faster sequential reads & writes but also that it could have an adverse performance
on random reads & writes. Only by testing various values can we determine what the right balance is.
8.3
Identity Management
As stated in the introduction, we believe that an identity management solution that maps user identity
in the system (represented by a public key in our solution) to their real-world identity can benefit the
system in a number of ways. For example, it makes it easier to share capabilities with others by allowing
the lookup of the recipients public key based on something like their name or email. These pieces of
information are easier to memorize. It also makes it more meaningful to have non-repudiable actions
(more on this in section 8.4). These two examples indicate the need for the mapping to be bi-directional:
sometimes we want to determine the system identity from the real world identity and in other cases we
want to do the opposite.
The challenge is guaranteeing the authenticity of the mapping between identities. Previous contri-
butions (chapter 3) have attempted to use the web-of-trust model. This requires no third parties but
is subject to spoofing attacks, especially in the infancy of the network. The opposite approach is using
a third party trusted by everyone. These come in the form of private key generators and certificate
authorities. The choice between the decentralized web of trust and the centralized TTP should be made
depending on the context in which the system is used.
8.4
Non-repudiable Actions
With non-repudiable actions we refer to making a file-changing operation (i.e. writes, truncates, deletes)
attributable to the user that performed it. This means that once an action has been performed, the

not record who performed what actions. This anonymity may or may not be desirable.
In practice, this could involve performing two signatures instead of one. First, the user signs using
their write capability to ensure that they are authorized to perform the action. Then they sign using
their personal private key to authenticate themselves and include their public key in the file next to the
two signatures. For a change to be valid, both signatures must be valid.
Non-repudiable actions makes most sense when we are able to map the system identity (i.e. public
key) to the real world identity of the user (as described in the previous section). Otherwise, there is
the possibility that a user simply generates a new identity for every action that they perform.
8.5
File & Directory Moves
In the following section we will analyze and discuss moving a file or directory from its source path to
a destination path. The goal of this section is to highlight the diﬀiculties in supporting moves when
using keyrings.
Note that moving a resource is equivalent to renaming it, since both operations are an alteration of
the resource’s path. We will refer to both as move throughout the section.
A move in a typical file system is an easy operation to perform. Figure 8.1 illustrates a move in
which DirA is moved from the root-directory (/) to be a sub-directory of DirC. Now imagine a keyring

After:

If the keyring is not updated, its capabilities are left invalid, since the their paths point to non-existing
files; they still point to the source paths instead of the destination paths.
The root problem is to prevent dangling paths in keyrings. This can be broken down to a number
of sub-problems that must be addressed in order to support moves:
1. Update responsibility: A solution should clearly state who is responsible for updating keyrings
after a file is moved, and how these rules are enforced. If the responsibility lies with the initiator
of the operation, then how do they know which keyrings need to be updated? If a keyring is
stored locally on another users machine, the initiator is unaware of it and does not have access to
it. If it is stored at the FSP, the initiator might not have the appropriate capabilities to modify
it.

must be able to deduce that the two separate events of a file disappearing from one location and

maintain some form of index that allows for files to be looked up based on an ID instead of a path.
Lastly, we want to highlight a key management data structure named Cryptree [Gro+06] that we
have come across in our research. It is closely related to keyrings but supports both file/directory moves
and revocations (although the authors do not explain how their rules are enforced by the system). Like
keyrings, Cryptree leverages the idea of a recursive data structure in which keys can derive other
keys through decryption. Keys and derivation links are used to build a read-access tree and a write-
access tree, whose structures closely follow that of the accompanying file system.
For example, a
directory’s read access is modelled with five keys and seven links which are used to control access for
its contents, directory metadata, parent directory and child directories. Their paper illustrates how the
strict structures of the read/write access trees allow them to eﬀiciently determine which keys and links
need update when an object is moved. We find their invention to be a compelling starting point, and
see no theoretical obstacles in building Cryptrees from keyrings that point to each other.
8.6
Revocation
Revocation is often required in an access control system but as we have described it is challenging to
support when using capabilities. This is because it involves several steps, all of which are expensive to
carry out and require the cooperation of users. To revoke capabilities from a user, one has to:
1. Replace existing capabilities with new capabilities and distribute these to current holders, except
for the user that is being revoked.
2. Re-encrypt and re-sign file using new capabilities.
Cryptree makes the first step easier as its rigid tree structure makes it possible to deduce which keys
need to be replaced to revoke a user from all files and directories inside a directory.
What is interesting is that the attribute-based encryption (ABE) scheme that was described in
section 2.4 is able to skip the first step. To revoke a user, all that needs to happen is for the file
to be re-encrypted using a policy explicitly excluding the user (this is the blacklist concept that was
discussed).
In [BCO05], the authors introduce two interesting ideas: lazy revocation and key rolling.
Lazy
revocation is a performance optimization in which the re-encryption of a file under a new key as a
consequence of revocation is postponed until the next write to the file. Note that it is impossible to
prevent a user from accessing a revision they have previously had access to. We can not force a user
to delete capabilities they already have and the revisioned nature of the file storage provider allows for
the retrieval of older revisions. Revoking only prevents access to future revisions, which implies that
lazy revocation comes with no negative security implications.
Furthermore, the authors introduce a key derivation scheme in which only the owner of a file can
derive new keys from the file’s prior keys (forward rolling), but everyone else can derive prior keys from
==PAGE:89==
8.7 Metadata Security
80
the current keys (backward rolling). This idea removes the need to store old keys to access old revisions,
but it requires a notion of file ownership which is something we do not currently have.
While some of the techniques described above help make revocation easier, we have not even dis-
cussed who is allowed to initiate a revocation in the first place. This complicates the solution further
but there is likely no way around it. This is because if anyone can revoke access from anyone else, it
becomes an easy venue for a denial-of-service attack.
8.7
Metadata Security
Even when a file is encrypted, some information about the file is still available.
This is known as
metadata and includes the name and extension of the file, timestamps and its size. As covered previously,
being able to determine the size of the plaintext from the size of the ciphertext is desirable for a virtual
file system like the one we have proposed. At the same time, it reveals very little useful information to
an adversary.
File names on the other hand can be sensitive. We believe that it could be possible to obscure these
by encrypting the file name using the read-key of the file and encoding the ciphertext such that it can
be stored in place of the plain file name. File systems have a limit on the length of file names so the
overhead added by encryption will reduce the maximum possible length of the name. It is important not
to store metadata (encrypted or otherwise) within the file, as reading the file from disk is comparatively
slow.

CHAPTER 9
Conclusion
The overall project goal is to provide a solution that allows for files to easily and securely be stored on
commodity file storage providers and shared with others, while unauthorized users, including the file
storage provider, do not have the ability to access or modify files.
Our solution uses Dropbox as a file storage provider and access control is built around the idea
of cryptographic access control. On top of the FSP sits a cryptographic file system, implemented in
FUSE, that transparently handles all encryption, decryption and signing operations, using keyrings.
The operations are based on state-of-the-art cryptographic primitives. The encryption scheme used by
the file system has been designed to support random read and write access - a first for any of the CAC
projects we have encountered, and also a major challenge to accommodate. The result is that users can
interact with their files as they normally would through their native file manager. The performance of
the cryptographic file system is good enough to support normal workflows such as editing documents
and playing a video. Besides supporting read and write operations, our solution also supports deletion
of files in a manner that is resistant to replay attacks observed in other projects. Additionally, we
discuss how the system could allow for files to be renamed or moved.
The access control system supports read, write and verify capabilities at the granularity of files.
In contrast to typical file sharing solutions, there is no centralized reference monitor, which has been
demonstrated to have both advantages and disadvantages. An advantage is that a user can delegate
their capabilities to another user through postal boxes hosted at the FSP. When a file is created, its
verify-capability is automatically broadcast to everyone, which allows everyone to participate in the
verification of the integrity of files. Invalid file operations are detected and reverted in a distributed
manner by a daemon program run by all users.
Our evaluation shows that the solution works as intended and fulfills the most important requirement
of protecting the confidentiality and integrity of files. Most threats affect the availability of the system,
which is to be expected in a large and open network (especially one in which actions are repudiable).
Performance figures for the cryptographic file system are lower than we hoped but our testing indicates
that this can primarily be attributed to FUSE. While the numbers are not great on paper, in practice
the performance is good enough for typical work.
The evaluation also identified a few issues and
shortcomings (none of which affect security) that we believe can mostly be resolved by our proposed
design changes.
In the project we have learned that file systems are much more challenging to work with than initially
anticipated. FUSE works in strange and sometimes undocumented ways, and the same is the case for
applications, which all seem to have their own peculiar way of interacting with files (such as creating
ephemeral files). With a file system, every detail has to be just right or undefined behavior quickly
ensues.
We also did not fully appreciate the consideration required in designing a cryptographic scheme that
allows for not only random access reads but also writes in files, but it we think the final solution strikes
a good balance between security, overhead and design complexity.
As demonstrated there are several examples of features, such as revocation and file moves, that
are diﬀicult to support in a system like ours. This is a consequence of several factors including the
somewhat simplistic nature of keyrings and the diﬀiculties around how to keep paths up to date.
While not part of our final solution, we spent a notable amount of time researching attribute-based
encryption (ABE). Our conclusion is that it is still too immature for any kind of use but it could very
well serve as a compelling alternative to key rings in the future.

We would recommend our solution to anyone, organizations or individuals, that have security con-
cerns and requirements suﬀiciently high to outweigh the concessions they make in regards to usability.
