Abstract
When generating textures for computer graphics, a skilled artist has to mod-
ify UV-maps to achieve as little distortion and tearing of the texture on the
model as possible, as well as manually paint the desired texture. This is a
time-consuming and laborious process. For many types of textures it isn’t
needed, since they have a random distribution of texture-details, and are not
directly bound to predetermined positions of details - for example a stone
need variations in colour and structure, but the position and distribution of
these details don’t need to be strictly positioned.
This project will look into noise-based texture synthesis, by estimating input-
parameters to a Fractal Brownian Motion generator from a supplied image ex-
emplar. From these parameters, it will generate a solid-texture with roughly
the same look as the source image. The focus is on diﬀuse, naturally looking
grainy textures, like stone, as they are the prime candidate for uniformly
distributed texture-details, which noise-based procedural textures excels at.
The report covers a possible way to do the input-parameter estimation au-
tonomously, by comparing the Fourier space of the input-image and a gen-
erated noise-map, as well as comparing the gradients of the textures. The
comparison of the Fourier-spaces are done by calculating the Structural Sim-
ilarity Index of the two Fourier-spaces after these have been blurred with a
Gaussian blur. The comparison of the gradients are done separately on each
axis of the textures. By minimizing the diﬀerences between the input-image
and the generated noise-map, a visual similarity between the input-image and
the noise-map can be achieved. In addition to generating input-parameters,
a histogram-matching on the input-image and the noise-map is also per-
formed, such that the colouration of the ﬁnal noise-map closely matches the
input-image.
The generator is able to create noise-maps, with an isotropic structure, and
colour-match images which has most of the colour distribution in the same
hue. It is not possible with this solution to generate intricate details like the
streaks in marble, nor is it possible to colour-match materials with several,
separated hues.
This report shows that it is possible to autonomously determine input-
parameters to a noise-generator which, depending on the structure of the
input-image, can replicate the look of the given input-image.

Glossary
Aggregates: Solid discrete parts inside a volume, like rocks and stone in
concrete.
Anisotropic: The property of being directionally dependent. For exam-
ple the veins in a wooden plank usually run parallel to the length and not
perpendicular.
Exemplar, image exemplar: An input image to a procedural generator.
Can for example be a photograph of a stone texture which is then replicated.
fBm (Fractal Brownian Motion): The process of stacking noise-maps
of varying frequencies and amplitudes to create more complex textures than
what a single noise-map can do.
Fourier: The transformation from time, or space, to the frequency domain.
Used to extract information about textures to be able to compare them.
Gabor Kernel: A falloﬀfunction which combines a Gaussian Kernel with
a sinusoidal plane wave.
Gaussian Kernel: A falloﬀfunction with a peak in the center, and smooth
falloﬀfrom the center which converges towards zero.
Gradient-noise: A noise created by interpolating gradients in a lattice.
Grainy structure: The word used throughout the report to describe a
texture with much high frequency noise.
Histogram-matching: Mapping the distribution of colours of one image,
to the likeness of another image.
Hue: Which combination of the primary colours (Red, Green, and Blue), a
colour has.
Input-image: The word used throughout the report in place of image ex-
emplar. An image which the look is to be replicated from.
Input-parameters:
The word used throughout the report to represent
noise-parameters.
Isotropic: The property of being directionally independent. For example,
if a metal plate is heated in the center, the heat will expand equally in all
directions on the metal plate.
Kernel: A falloﬀfunction, for example a Gaussian kernel.
Lattice: In the scope of this project, a lattice is a set of equally spaced
points in a system.
Macro-structure: The word used throughout the report to represent a

==PAGE:8==
texture with large areas of the same colour / texture. Opposite of micro-
structure.
Micro-structure: A texture with small high frequency details. Opposite of
macro-structure.
Noise: Random values though with a cohesive structure, where the neigh-
bouring values have a relation to each other.
Noise-function: A function which can create random values, but with some
similarity to the neighbouring coordinates.
Noise-map: The word used throughout the report for the generated 2D
texture. The noise-map is generated from a set of input-parameters.
Noise-parameters: The variables used as input to the noise-function, to
adjust the smoothness, frequency, etc.
Noise-value: The pseudo-random value the noise-function returns from a
given coordinate.
Perlin-noise: A noise-generator, made by Ken Perlin.
Procedural generation: To produce something mathematically, usually
from a noise function. Can be used to create semi random structures.
Procedural texture: A texture generated by algoritms, rather than an
artist
Seed: A single value, can be used to replicate the results from a noise-
generator, to get the exact same outcome every time. Is generally used to
oﬀset the part of the solid-texture which is generated.
Simplex-noise: A newer, simpler, version of Perlin-noise, which has a lower
computational complexity, scales to higher dimensions at lower cost, and has
less directional artifacts.
Solid texture, volumetric texture, 3D texture: A texture which spans
into three dimensions.
SSIM (Structural similarity): A numerical representation of how similar
two diﬀerent images are.
Texture, 2D texture: A texture which spans two dimensions.
Tone: The lightness/darkness of a colour.
UV-coordinates: The coordinates in a texture, used to map a texture onto
a 3D model.
UV-mapping, UV-wrapping: The process of mapping a 3D mesh onto a
2D texture. Each vertex is assigned a 2D coordinate on the texture called

Voxel: A voxel is a value in a regular 3D grid. The 3D equivalent to 2D

Wavelet-noise: An alternative to perlin-noise. It is almost perfectly band-
limited, which can be utilized to precisely control the weighting of each band,

Introduction
Procedural-generation with noise-functions is a strong tool to create diverse
textures, but it is not inherently easy to use it, unless you have experience
with it. There is no direct logic between the input-parameters to the func-
tions, for an artist which are used to tools like Z-Brush or Photoshop, and
the resulting texture. Thus a tool which bridges this gap by estimating these
parameters for the artists could be useful. Procedural-generation has the
added function of working in several dimensions, and are not bound to only
generate 2D textures as is traditionally done, it can easily be extended to
create textures in three dimensions. This is called solid-textures.
A solid-texture is a 3D texture, where a point on the surface of a model
can be mapped to a coordinate in the texture. If a suitable solid-texture is
used, a coherency of the surface is achieved, with no need to do UV-mapping.
This has the bonus of also having data for the inner parts of the model which
in surface wrapped textures, is hollow. So if a model, with a solid-texture
applied is split open, the inner parts along the breach would have a texture
on the split surface as well. An example of this can be seen in Fig. 1 where
a spherical shape has been cut out from a cube.

The data for the solid-texture has to be represented in some way. This is
often done as input-parameters to a procedural texture, since it can be easily
expanded into three dimensions, and does not have any wrapping or tiling
as a normal texture would have. To give variation to the texture, a gradient
noise like Simplex noise or Perlin noise is often used. Some solutions exist
which generate aggregates of certain shapes, which can be used to simulate
for example concrete, other solutions handle heterogeneous textures like jade.

material, and match the pattern, colours and general look to that picture.
Traditionally when texturing 3D models, a 3D model has to be UV mapped
to be able to correctly receive a 2D texture.
Think of a globe, how the

This requires a skilled person to sit and semi-manually adjust every polygon
and vertex of the model, in the UV-plane, so that as little distortion and
tearing as possible is achieved. This is a laborious process, which for plenty
of models, for example naturally occurring materials like stone, dirt, grass,
etc., and human made materials like cloth, concrete, roof-tiling, etc., isn’t
really needed, since the surfaces in general have the same basic texture.
This project aims at providing a solution to generate a set of autonomously
generated set of input-parameters to a noise-generator, based on an input-
image, in order to lower the introductory gap of using noise-based solid-
textures. This tool could for instance oﬀer a lower cost for games or movie
creators who are seeking to create 3D textures similar to real-wold material
without using repetition/tiling.

Scope
The scope of the project is to estimate parameters to a procedural noise-
generator to create textures like granite or marble, which have several diﬀer-
ent looks, from almost homogeneous looking textures, to more complex tex-
tures with streaks of diﬀerently coloured internal structures. The scope does
not involve materials other than diﬀuse ones with grainy surfaces. Therefore
materials such as metals, gemstones, smooth fabrics etc will not be explored.
The result should have similar looking features to the supplied source texture.
There is no real way to intuitively set these input-parameters to the noise-
generator, and as such a solution which could generate these parameters
based on a input-image, could go a long way to help the artists in gener-
ating textures with a speciﬁc look. The goal is to have the procedure be
completely autonomous where the only interaction from the user is present-
ing a photograph/picture, and the end result being the parameters for a
solid-texture which could be used on any given 3D model.

Related Work
The idea of solid-textures was proposed in the 1985 paper Solid Texturing
of Complex Surfaces[31] by Darwyn R. Peachey, where examples of wood
and stone materials are presented, though they are very synthetic in ap-
pearance. Simultaneously also in 1985 Ken Perlin in the paper An Image
Synthesizer[32] introduces the concept of solid texturing. This is the pa-
per where Perlin Noise is introduced, and various examples of bumpy and
coloured solid-textures are presented. Later Ken Perlin introduces Simplex
noise[6] in 2001, where it is described and the code to implement it is sup-
plied. Simplex noise is another way to do gradient noise, but is more readily
scalable to higher dimensions. This text was further explained by Stefan
Gustavson in 2005 in Simplex noise demystiﬁed[16]. Other types of noise
is often used in procedurally generation, for example Sparse Convolution
Noise[29, 23], Spot-noise[29, 34] and Wavelet noise[21]. Simplex noise is the
noise-generator which will be used in this project.
There are in general two diﬀerent approaches to generating textures proce-
durally from real world examples. Either by simulating real-world properties
and physically distribute these properties in the material, or by approximat-
ing the looks of the material by intelligently combining noise textures. Noise
is often-time used in physically based procedurally generated content to ran-
domly place properties, or give variations to the structure and colouration.
I will brieﬂy go over how real-world properties has been simulated by others,
since this isn’t the direction of this project. In the 2015 paper simulating wood
structure[30], Liu et al shows how the ﬁbrous details in a tree trunk has a
great impact in how a plank of wood is presented. They go about simulating
pores, vessels, rays, etc. in a trunk, to create very realistic looking 3D models
of planks, cubes, and intricate carvings. They also show generating renders
based on photographs of real-world wood species. Another type of material
which have been attempted to simulate is what could be concrete, terrazzo
or igneous rock. These types of material all have the structure of smaller
aggregates inside a uniform medium. This is attempted by Robert Jagnow et
al in the 2004 paper Stereological Techniques for Solid Textures[25], where
they generate assorted shapes of 3D particles, which are placed at random
inside a volume. Afterwards the volume is coloured along with the particles,
and a noise texture is added to create variations in the colouration. The
result is a very life-like looking stone texture which strongly resembles the
source images. This idea was expanded upon by Kopf et al[26], Dong et
al[22], and Yue Shu et al[33] in respectively 2007, 2008, and 2014. Starting
12
==PAGE:14==
with Kopf et al in Solid Texture Synthesis from 2D Exemplars[26], they
used an input image which could have had aggregates in the real world, and
instead of generating aggregates, made a feature-map of this texture, where
the location of the aggregates was marked with white, and where there is
the encompassing volume was marked with black, such that this feature-map
has the same eﬀect as an alpha-map. This feature-map is then used to guide
the simulation to where the boundaries inside the material is, and from that
get crisp edges where it is needed with neatly separated aggregates from the
encompassing volume. The generator also works without the feature-map,
but can’t replicate the sharp transitions in the material as well as when a
feature map is supplied. This feature map gives a bit more of an overhead
when supplying input to the generator, where a person has to sit and trace
the input texture, and manually ﬁnd where the details in the texture are,
and where the transitions seems to be. Dong et al in Lazy Solid Texture
Synthesis[22] from 2008 looks into using voxels to only calculate the part
of the texture which can be seen, but otherwise end up with results which
are very close to Kopf et al. In 2014 Yue Shu et al with Eﬃcient texture
synthesis of aggregate solid material[33], looks into how the placement of
these aggregates can be controlled by a cellular structure and spring systems.
The spring systems are used to evenly distribute particles. Their results also
resemble the results of Kopf et al.
As for approximating looks by using noise, there is a number diﬀerent ways
this can be done. One of the ways is by splatting Gabor Kernels either on
the surface of models, or in the 3D space. This is described in the 2009
paper Procedural Noise using Sparse Gabor Convolution[27] by Ares Lagae
et al.
With this, they were able to create various materials like marble,
cloth-ﬁbers, leather, and wood, although they were all slightly synthetic in
the appearance, and not exactly natural looking. It was done by controlling
the frequency-range and radial spread in the power spectrum. The colour
of the material is obtained by feeding the noise into a colour map. This
is expanded upon in the 2012 paper Gabor Noise by Example[24] by Bruno
Galerne et al, where the power spectrum is intricately constructed by several
layers of ﬁner and ﬁner details. This gives more precise control of the ﬁnal
look of the texture, which mostly looks like textiles like denim or linen. The
construction of the power spectrum is created to have the same structure
as the power spectrum of a supplied image exemplar. A diﬀerent approach
to creating textures with noise, is using wavelet-noise, as shown in the 2010
paper Procedural isotropic stochastic textures by example[28] by Ares Lagae
et al.
The noise is generated from diﬀerent weights for each noise band.
By ﬁnely adjusting the weights of these bands, a span of diﬀerent looks can
be generated. These weights can be autonomously determined by utilizing
the frequency space, so by providing an image exemplar, the weights for the
noise generation can be determined, a similarly looking texture can be pro-
cedurally generated. This can then be coloured by histogram-matching the
generated texture with the provided image exemplar. The results of this can
be quite convincing, as long as the provided exemplar is isotropic in structure.
Lastly Jingjie Yang et al wrote a paper in 2016 called An Inverse Rendering
Approach for Heterogeneous Translucent Materials[35], where they estimate
13
==PAGE:15==
optical parameters to create similar looking materials to a heterogeneous
translucent input material, taken from photographs of various types of jade.
This, along with histogram matching the colours to the input image, results
in very lifelike renders of jade models. The input parameters for the 3D sim-
plex noise generation seems to be manually determined, and not taken from
the input photograph.
This project will mainly take inspiration from the two papers mentioned
last in the previous paragraph, Procedural isotropic stochastic textures by
example[28] by Ares Lagae et al and An Inverse Rendering Approach for
Heterogeneous Translucent Materials[35] by Jingjie Yang et al, which covers
autonomously determined noise parameters, and histogram-matched coloura-
tion.

Method
This chapter will cover the design of automatically generating parameters
from an input image, along with describing the method of how these designs
were realized. The start of the chapter describes the main challenges with
the project, followed by an iterative description of each design component.
The challenges of the project can be summerized as:
• Understand how procedural noise generation can be done, and how to
manipulate the results depending on the input parameters.
• Determine how the colouration of the input image can be replicated on
the generated noise-map. The challenge is two-fold, as there is both
the challenge of extracting the colours from the input image, and how
to map them onto the generated noise-map.
• Explore a way to automatically determine the noise-parameters from
the input image.
• Present the results such that visual comparisons can be made, and to
see if the expected outcome in 3D was achieved.

Input Images
Since the idea is to replicate/re-construct textures from a source image,
proper source material is needed. When researching the diﬀerent kinds of
marble that exist, www.marble.com[4] came up. This website has a quite im-
pressive assortment of diﬀerent marble examples (and other types of stone),
in a nice uniform size of 300 by 300 pixels. From the entire library of diﬀer-
ent stone-textures, the marble and limestone textures were acquired. They
can be seen in the appendix, in section Appendix - Marble and Appendix
- Limestone. The selection of these types of textures is based on the fact
that limestones seem to have a bit more uniform, isotropic, look, where the
marble textures in general has more intricate details, as well as anisotropic

the development phase and as ﬁnal examples.

Noise Generation
When deciding between the various noise generation solutions, Simplex noise
was chosen.
The main reason for this choice is that the project isn’t an
exercise in implementing a noise generation system, but rather utilizing what
already exists, and when a version of Perlin noise and Simplex noise already
exists as a framework for Python, the challenge of getting up and running
was handled. The choice between Simplex noise and Perlin noise came down
to the fact that Simplex noise is newer, slightly faster and gives a bit more
utilization of the colour-range[8]. Wavelet noise was rejected due to it not
being readily available for Python, and that there already exists a good
solution for estimating parameters for wavelet noise[28], as described in the
related works chapter.
Simplex noise is of the type gradient noise, where pseudo-random gradients
are placed in a lattice. These gradients are then interpolated on, between
the lattice points, which gives smooth transitions with neighbour values re-
sembling each other. The only caveat this brings, is that the noise-value in
the lattice points will always be zero.
The advantage of gradient-noise compared to randomly chosen values, is this
neighbourhood resemblance, which gives a coherent structure in the textures.
Example of 1D gradient noise can be seen in Fig. 5.
Figure 5: 1D gradient noise example. Black line is the interpolated
gradient values, the red circles are the lattice points, and the red lines
are the gradients.
The output of a call to a noise-function will return a value between -1.0
and 1.0.
This is scaled throughout the project, such that the values lie
between 0.0 and 1.0.
This is done such that there is a clear connection
to colour values as a ﬂoat, which goes from 0.0 as completely black, and
1.0 as completely white. The translation is a simple calculation as follows:
noisevalue = (noise[x, y] + 1.0) · 0.5. When the fBm is implemented, these
values can end up lower than 0.0 or higher than 1.0, if two separate octaves
constructively or destructively interferes. This is not an issue, since values
outside of the range are clipped, and thus just get the darkest, or brightest,
colour.
The challenge is then to ﬁnd suitable ranges for the input variables to the
fBm generator.
It is assumed that the supplied input image is a square.
When generating and comparing to an input image, the width of the image
19
==PAGE:21==
is saved as the Size, and thus the generated noise-map will end up with the
same dimensions as the input-image.
This implementation of the fBm generator has range of input variables: point,
seed value, scale, octaves, persistance, lacunarity, oﬀset, amplitude, and fre-
quency.
• point is the coordinate plugged into the noise generator.
• seed value is the seed, to be able to reproduce the result. Used in
conjunction with a randomizer to adjust the oﬀset for each octave.
• scale is used to scale the noise-map, smaller values zoom out, and
larger values zoom in. The coordinate is divided by this scale factor.
• octaves is used to layer noise-maps on top of each other. Each oc-
tave use the lacunarity and persistance to change the frequency and
amplitude.
• persistance is multiplied onto the amplitude for each octave, such
that the amplitude is reduced for each octave. This is done so that
the higher frequency noise, has a lesser impact on the structure of the
noise-map than the low frequency noise.
• lacunarity is multiplied onto the frequency for each octave, such that
the frequency is increased for each octave.
• offset is used to oﬀset the coordinate from the given point.
• amplitude is the initial amplitude of the noise, set to the same value
as the persistance. For each octave, the amplitude is multiplied by the
persistance, to reduce the amplitude for each subsequent octave.
• frequency is the initial frequency, set to 1.0/size. For each octave, the
frequency is multiplied by the lacunarity, to increase the frequency for
each subsequent octave. The frequency is multiplied onto the scaled
coordinate, such that each octave has a larger distance between each
coordinate sample.
When determining the optimal noise-parameters, for each comparison and
generation of noise-maps, the Seed, Scale, and Oﬀset is non-changing. The
Scale is in the project always set to 1.0, but is supplied to the generator,
such that it is possible to give another Scale if wanted. The Seed is set to an
integer. It is used to be able to reproduce the image results. The Oﬀset is
adjusted in the function by the Seed value to oﬀset each Octave from each
other.
Inspired by Jingjie Yang et al[35], the Frequency was set to always be a factor
of the width of the input image, frequency = 1/width, and thus always a
value lesser than 1.0. The initial Amplitude is set to the same value as the
Persistance, same as Jingjie Yang et al[35] did.

Histogram Matching
Another solution is needed, since simply moving and scaling the colours
didn’t work.
The chosen solution is histogram-matching[18].
Histogram-
matching is chosen due how it innately reaches the same lower and higher
boundaries, as well as the correct shape of the distribution, as the input-
image.
The process is as follows: For each pixel value, sum the total amount of
pixels in the image with the given value. Then do a cumulative sum of these
counts. When this sum is normalized in regards to the total pixel-count in
the image, two diﬀerently sized images can be easily compared and adjusted,
since the cumulative sum will sum to 1.0. This action is performed for both
the input image, and the generated noise-map.
Then for each pixel-value in the noise-map, the corresponding cumulative
sum-value is found. The nearest cumulative sum-value for the input image
is then found, and the corresponding pixel-value is determined. This pixel-
value from the input-image is then used instead of the original pixel-value in
the noise-map. When all the unique pixel-values have been traversed, and
replaced, the colour-matching is complete.
This is done for each of the three channels, red, green, and blue. Results of
this can be seen in Fig. 16 and Fig. 17, where subﬁgure 16(b) and 17(b), is the
histogram for the input-image. Subﬁgure 16(d) and 17(d) is the histogram
for the corresponding linearly scaled image from last section, to give a direct
comparison. Subﬁgure 16(f) and 17(f) is the histogram for the histogram-
matched result. It can clearly be seen that the colour-distribution is much
closer in structure to the input-image with this solution. The plots of the
histogram matching can be seen in Fig. 18 and Fig. 19, where a direct compar-
ison between the input-image, the generated noise-map, and the histogram-
matched noise-map can be done.
While the visual result in Fig. 16(e) is very close to the input image, the
second result Fig. 17(e) isn’t as exact, even though the colour-distribution
are just as close to the input image. I believe this is due to the input image
having separated ”colour-areas”, as in separate yellow, red and blueish areas.
It is my assumption that this cannot be correctly replicated with the current
solution of generating a single black and white noise-map, which is then
duplicated to the three colour channels. But the solution should work with
textures with colours in the same shade or hue throughout the texture.

Automatic Parameter Estimation
When autonomously determining the noise-parameters, some sort of image
comparison has to be done, such that the input image can be compared to
the generated noise-map, other than just comparing pixel to pixel. This is
mainly due to the premise of replicating the look of a texture, and not to
make a carbon copy of the input image, such that for example, two diﬀerent
statues would look like they were carved from two diﬀerent parts of the same
marble chunk.
An idea is to compare the 2D Fourier space of the input image, to the 2D
Fourier space of the generated noise-map. A problem with this is that if the
two Fourier spaces are equal, the non-Fourier space images are also equal,
but if the Fourier spaces are blurred slightly before comparing, two images
which are not exactly equal, but have frequency distributions of the same
caliber, would be more alike, since there would be more overlap of values in
the blurred Fourier-space.
Then a metric for comparison has to be established. One way to do this is to
pixel by pixel compare the two Fourier spaces, and ﬁnd where the diﬀerence
is smallest. This could work, but a bit more lenient solution would probably
be better, such that small areas of big diﬀerences won’t impact the ﬁnal score
too much.
A way to do this, could be to do a Structural Similarity comparison[20],
where a comparison between two images is given a value between -1.0 and
1.0. If the result is 1.0, then the two images are a perfect match, and a result
of -1.0 means that they are perfectly unmatched. Examples of this can be
seen in Fig. 21, which shows examples of the SSIM value by combining the
two images shown in Fig. 20.

Minimizer
To do the minimizing, a Python Library called SciPy[13] is used. The mini-
mizing function can take several input parameters and returns the optimized
value for these input parameters, along with a scalar which gives an estimate
for how good those return parameters are.
Since I knew that there was some logical boundaries for the Persistance and
the Lacunarity, as stated earlier, I ﬁrst explored some of the solvers which can
take boundaries as input, namely SLSQP and COBYLA. They did okay, but
seemed like they didn’t search far enough before settling on a local minimum.
The next thing I explored, was the solver Nelder-Mead, which doesn’t have
the possibility to set boundaries, but it had a much wider initial search
radius compared to the other two. This had the eﬀect of some of the results
being outside of what are useful parameters, but when accepting a set of
parameters as best candidate, a simple check to see if they lie within the
allowed boundaries was enough to discard unwanted parameters in favor of
parameters which were inside the boundaries.
The parameters adjusted by the minimizer is the lacunarity, persistance and
indirectly the octaves. The octaves are used in a for loop, with hard-coded
values, {4, 6, 8}, and the lacunarity and persistance is given to the minimizer
as input parameters. Initial values for lacunarity and persistance is set to
{1.2 , 1.7 , 2.0} for lacunarity and {0.47, 0.57, 0.67, 0.77} for persistance. The
lacunarity and persistance values are used as nested loops inside the octave
loop, such that there in total, with these settings, are performed 3· 3 · 4 = 36
attempts at minimizing the input parameters. As mentioned in the Noise
Generation section, the boundaries when determining the best candidate are
values between (1.0, 4.0) for lacunarity, and (0.1, 1.2) for persistance.
These initial input parameters are estimated by trial and error, to see which
breakpoints give nice results, and how far and wide the minimizer searches.
With every loop, the estimated input parameters from the minimizer, as well
as the octaves and the reached SSIM is saved in an array. When the loops are
done, the function values for each minimizer is analyzed, and the one with
the lowest function value, as well as values of Persistance and Lacunarity
within boundaries, is then accepted as the best input parameters.

Gradient Comparison
When looking at the input image in Subﬁgure 25(c), There is a general
structure where there are some medium size areas where the gradient from
pixel to pixel hardly change, both in the white and the brown area, and some
sharp changes in the gradient where the pixel goes from a white to a brown
area, or vice versa. So these gradient changes over both the input image, and
the generated noise-map, can be compared and used in the minimizer. This
could both have an eﬀect of better representation of the type of structure in
Subﬁgure 25(c), but should also be a good addition to images with a more
grainy structure.
How these gradients were calculated is as follows. The image is sliced up
in horizontal and vertical slices.
Each slice is then pixel-by-pixel looped
through. The ﬁrst and last pixels are special cases. For each pixel it is found
how much the pixel-value changes from the pixel just before and the pixel
just after, such that it is purely the change happening over the pixel, not
counting the pixels own interference on the ﬂow. For the ﬁrst and last pixel,
it is not the change over the pixel which is found, but rather the change from
the ﬁrst pixel to the next, and the change from the second-to-last to the last
pixel. This is due to how there is no data for the non-existing pixel before
the ﬁrst, and the non-existing pixel after the last.
An example with fabricated values, which shows pixel values on the y-axis
and pixel indices on the x-axis, and the gradients as lines between the values,
can be seen in Fig. 27. This graph is purely to visualize how the gradients
are determined between pixel-values. The special cases are marked with a
red line.
This implementation is a variation on forward-, backward-, and central-
diﬀerences, where the oﬀset distance, h, for forward and backward is set
to 1, and for central-diﬀerence is set to h = 2. x is the pixel-index, and f() is
the pixel-value. See Eq. (3), (4), and (5).
gradientforward = f(x + h) −f(x)
(3)
gradientbackward = f(x) −f(x −h)
(4)
gradientcentral = f(x + 0.5 · h) −f(x −0.5 · h)
(5)

Implementation
There are two parts to the implementation.
The part which covers the
Python implementation to estimate the noise-parameters and the colour-
matching, and the part in Unity which is used to present the created solid-
textures on a 3D model.
Python
The Python section is divided into two diﬀerent sub-sections.
A section
explaining how the noise-generation is performed, and a section describing
how the autonomous noise-parameter estimation is achieved.
Noise-Generation
Since I’ve chosen to mainly work with Python for this project, I found a
library to supply functions to calculate noise from. The library in question
is Perlin noise library for Python (PNLFP)[7] by Casey Duncan[1], which
supplies functions for Simplex Noise, in both 2 and 3 dimensions, which is
just ﬁne for what I need it to do.
There is two kinds of noise generating functions in the project, one for 2D
and one for 3D. The main diﬀerence is whether they create an entire noise-
map of for example size n x n for 2D, or if there is created just a single
noise-value from a single [x, y, z] coordinate.
The ﬁrst type is used when determining the noise-parameters, and the second
is used to create the solid-texture used in Unity.
The biggest diﬀerence
between them, asides from the extra dimension in the 3D generator, is that
the 2D generator has the loop which loops over the n x n pixels inside the
function, and returns the entire noise-map. The 3D function only calculates
for a single point, and returns that value.

7
noise height = noise height + noise value * amplitude ;
8
amplitude *= persistance ;
9
frequency *= lacunarity ;
10 end
11 return (noise height + 1.0) * 0.5;
The following happens when calling the function to get a noise-value for at
coordinate in three dimensions.
The return value noise-height is initialized to zero. Then the scale value
is checked to make sure that it is larger than zero. If not, then it is set to a
small value of 0.001. This is purely to avoid division by zero.
A loop then starts to loop through the provided octaves. Each octave then
gets a random oﬀset, and is added to the supplied offset. The random oﬀ-
set is created by using the Python built-in library random. The seed value
is used to prime the random number generator. The random oﬀset is val-
ues between -10 000 and 10 000, which should be wide enough to avoid two
octaves using the same offset.
The provided point is then scaled and multiplied with the frequency. The
randomized offset is added on to this and is saved in a sample coordi-
nate. The 3D simplex noise function from PNLFP is called, with the gener-
ated sample coordinate. The return value is multiplied with the amplitude
of the current octave, and added to the current noise-height. Then the
amplitude is updated by multiplying with the persistance, and the frequency
is multiplied with the lacunarity.
After all the octaves has run, the noise-height is returned. It is returned
with the scaling to place it roughly between 0.0 and 1.0, by adding 1.0 to
the value and multiplying by 0.5. Going from the normalized values between
0.0 and 1.0 to the ﬁnal color ranges of 0 to 255 for each channel, is simply a
matter of multiplying with 255, and casting the value to int.

Histogram matching
A set of functions to do the histogram matching was found on Stack Over-
ﬂow[14], created by user ali m[15], instead of implementing the histogram-
matching myself. This worked ﬂawlessly out of the box, and works on diﬀer-
ently sized images.
The code in the histogram-matching works as follows:
The shape of the noise-map is saved.
Then both the noise-map and the
input-image is unraveled, each to a separate list. The unique pixel values of
the noise-map and the input-image are saved, along with the count of each
pixel-value. For the noise-map the indices for the pixel-values are also stored.
Then a cumulative sum of the counts are created for both the noise-map and
the input-image. This cumulative sum is then normalized to values between
0.0 and 1.0 for both images.
The noise-map is then interpolated, with the use of a Numpy function called
interp(), which take the two cumulative sums as input as well as the pixel
values of the input-image.
Lastly the original indices for the noise-map is then used to distribute these
newly interpolated pixel-values, back to the equivalent position in the list,
and the original shape is reconstructed. This resulting map is then returned
from the function.

both directions. Lastly the ADJ SSIM is constructed by taking 1 - SSIM and

Unity
For presenting the solid-textures on a 3D model, Unity was chosen. It is a free
game engine, which easily handles import of model ﬁles and manipulation of
shaders.
Instead of implementing a ”true” solid-texture shader in Unity, which could
take the noise-parameters, along with information on the colouration, and
then generate the solid-texture on the ﬂy, a simpler solution was chosen,
which used the existing output textures from implementing the Python code.
Since the Python code already creates a 2D noise-map for comparing to
the input image, this function was extended to save slices with the distance
between each slice as the distance between each pixel, such that a 300 by 300
by 300 solid-texture, resulted in 300 separate 2D slices with a resolution of
300 by 300. These slices could then be loaded into a 3D texture in Unity, and
shown with a shader. The slices are created by using a triple nested for-loop,
where the size for both x, y, and z, are 300, and passing these coordinates to
the 3D noise solution shown in the previous section. The distance between
each neighbouring pixel will then be kept the same, and the same coherency
on all three axis are present.
To present the solid-texture, two things had to be created. A shader which
can take a solid-texture as data and convert to colour fragments, and a way
to convert 2D slices to a solid-texture.
The lighting setup in Unity is set to a ﬂat blueish-grey environment colour
(to give a bit of a colour to reﬂect in the material, as well as supply some
ambient lighting) and a white directional light at 80% intensity, to keep the
environment as neutral as possible. There is a slight reﬂection in the material
to help give shape to the model along with the lighting.
Script to load the textures
The sorting of the 2D slices could easily be handled by giving the textures
a naming convention which could be handled sequentially in the code. This
was done such that, when the 2D texture is saved from Python, the last part
of the ﬁle-name is the z-coordinate of the slice, with padding zeroes. Then
it is trivial to use this part of the ﬁle-name to ﬁnd the correct placement for
the 2D slice in the solid-texture.

Discussion
The project set out to create a tool which could determine input-parameters
to an fBm noise generator, and determine the colours for the end result. The
aim was to provide the tool with a single input-image, and the tool would
derive from this input-image the parameters needed in order to create proce-
dural textures. This chapter covers the discussion of some of the challenges
which were overcome during the Method, and Implementation chapters, as
well as discussing the Results chapter. The end of the chapter shows some
ideas on how to improve on the solution.
70
==PAGE:72==
Noise-generation
Ranges for the input-parameters for the fBm, was established by ﬁrst looking
into what would be mathematically usable ranges, especially for persistance
and lacunarity. Then these ranges were visualized by generating noise-maps
with varying combinations of octaves, persistance, and lacunarity, to verify
that the ranges were correct, and gave usable results.
The octaves were determined to work the best with values between 4 and 8.
Less than 4 did not use the full range of what the fBm is capable of, and
more than 8 did not return visually diﬀerent looks than what 8 could create.
The persistance had to be a value other than zero. If persistance was zero,
the entire noise-map would end up being a pitch black texture, since the
persistance determines the base amplitude, and the scalar multiplied onto
the amplitude for each octave. The persistance also made most sense to cap
out at about 1.0, since values greater than this would increase the amplitude
for each octave, instead of reducing the amplitude as would be expected.
By generating noise-maps with persistance greater than 1.0, it was noticed
that values up to 1.2 for persistance could work for certain texture-looks,
but higher than that reduced the noise-map to something akin to white
noise. A negative value for persistance would have the eﬀect of making the
amplitude be a negative value for every other octave.
This could maybe
work for certain textures but was chosen to disregard negative values for
persistance, to narrow the scope of the minimizer. In the end the range from
values greater than 0.0 to values smaller than 1.2 was chosen.
As for lacunarity, it was noticed that the lower bound had to be a value
greater than 1.0. If the lacunarity is set to 1.0, the same frequency is used
in all the octaves, since lacunarity is a multiplier onto the frequency. With a
lacunarity smaller than 1.0, the frequency would be reduced for each octave,
and thus instead of generating higher and higher frequency details, each
octave instead created more and more low frequency details, which made
no discernible change to the overall visual structure. For the upper bound,
various noise-maps were generated to see how the lacunarity aﬀected the
look of the texture. It was established that values greater than 4.0 had a
tendency to create wormlike structures in the texture, which for the use-case
of the tool, to create grainy textures, was not ideal. This gives the range for
lacunarity of values greater than 1.0 and smaller than 4.0.
With these ranges, boundaries for the minimizer was established.

==PAGE:73==
Colour-matching
It was quickly determined that just linearly scaling the colours of the noise-
map, to match the boundaries of the input-image in the three colour channels,
was not a usable solution, since the distribution of the colours did not match
the input-image, other than the boundary values.
This gave washed out
colours, compared to the input-image.
Thus histogram-matching was chosen as a solution. This gave a good repre-
sentation of the colour distribution from the input-image in the noise-map.
A limitation with doing the colour-matching like this, is the way the noise-
map is constructed. Since all three colour channels are a copy of the same
noise-map, it is not possible to replicate separate areas of diﬀerent hues in
the same texture, and as such the tool is limited to textures of a single hue
throughout the image.

Presenting the results
The choice of creating slices of 2D noise-maps, and stacking them to a solid-
texture, when presenting the result in 3D, was made on the basis that it
would take signiﬁcant time to recreate the Simplex noise generator inside
Unity, such that the exact same outcome, as in Python, would be created with
the input-parameters. This would have complicated the project immensely,
and the idea was discarded, such that the focus would only be on estimating
the input-parameters and generate 2D slices in Python. And since Python
already created 2D textures as a part of the development, it was a small
change to extend the creation to make a given amount of 2D slices, which
can be stacked to a solid-texture, and save them to disk .
To be able to use this tool in real life projects in the industry it would be
necessary to create a noise-generation function in Unity such that the input-
parameter estimation will give the exact same results in both Unity and
Python.
A material for the shader was created with a slight reﬂection and the rough-
ness of the reﬂecting surface was increased a bit to give more diﬀuse reﬂec-
tions. The environment which is reﬂected is a neutral blue colour, such that
it doesn’t interfere with the colour of the texture too much, but gives a bit
of a dynamic look, such that the shape of the model is easier seen, instead
of having a totally ﬂat look. The lighting model was done such that the
brightest part of the 3D model was just a tad brighter than the same part of
the texture, since most of the model would not be in direct lighting.

As mentioned in the Method chapter, the limestone input-images has a bit
more uniform structure compared to the marble textures, and thus had
greater success at visually replicating the looks compared to the marble
input-images.
To illustrate what was a success and what wasn’t, I’ve grouped the results
up in three categories. The good, the mediocre and the unsuccessful.
The good results are representations of where both the histogram-matching
and the noise-parameter estimation gives results which visually are very close
to the input-images.
See Fig. 37 for examples of this.
The histogram-
matching succeeded in replicating the colouration of the input-image, in all
four examples. It worked ﬂawlessly because the hues of the input-images
in general stay in the same range, and there isn’t areas in the image with
diﬀerent hues of the same tone. Likewise, the structure, and the general
distribution of the low and high frequency noise, is also represented in the
resulting noise-maps. This is because the input-images in general has an
grainy texture without streaks or other anisotropic details.
These grainy
textures are where fBm’s excel.
The mediocre results are representations of situations where the histogram-
matching worked as it should, but the input-image has structures which an
fBm with this implementation can’t replicate.
The colour-representation,
both in colours and in contrasts, over the entirety of the generated noise-
map, is visually similar to the input image. The noise-parameter estimation
gave results which has visually similar looking results in regards to frequency,
but the input-image has structures, like streaks and anisotropic details, which
can’t be replicated with only using a rather simple fBm-generator which this
project utilizes. See Fig. 38 for examples of this.
Lastly the unsuccessful results are where the estimated noise-parameters
didn’t give usable results, and/or the histogram-matching failed to give an
accurate representation of the input image. See Fig. 39 for examples of this.
A reason why the colour-matching failed, is that the same noise-map is gen-
erated for each colour-channel. If the input image has a clear separation in
colours, as can be seen in Subﬁgure 39(d), where there is separated blue,
green and yellow-brownish areas, then this separation can not be replicated.
This would require some sort of separation of the colour-channels where, the
blue areas had to be kept separated from the red and green channel in some
areas, and in other areas that some overlap between the green and the red
81
==PAGE:83==
had to occur to create yellow colours, and thirdly areas where the green had
to dominate. This is not possible with this solution.
The failures in estimating the noise-parameters can have several reasons.
For the example in Subﬁgure 39(b), where the generated noise-map has a
completely diﬀerent micro-structure compared to the input-image, is due to
how little diﬀerence there is in the contrast. All of the colours are almost
the same value. This give very little wiggle-room for the minimizer to hit
a good combination of lacunarity and persistance with the chosen starting
guesses. For this speciﬁc example, the minimizer had a tendency to quickly
default to try negative values of persistance, and values lower than 1.0 for
lacunarity, which both has the opposite eﬀect of what is wanted out of this
kind of fBm-usage. Maybe if a broader span of starting guesses was used, a
proper minimum would have been found.

Shortcomings
The biggest shortcoming of this solution is in the inherently grainy struc-
ture of the output from the fBm-generator, with no real way to replicate
anisotropic structures. Complex structures as those seen in the input-images
in subﬁgure 38(a), 38(b), and 38(c), are not possible to create with this solu-
tion. Thus the generator is restricted to generate grainy, granite-like textures
rather than intricate marble-like textures.
Another shortcoming of the current solution, is how the minimizer some-
times has a tendency to focus on completely wrong values when minimizing,
and thus sometimes completely fail to create noise-maps which resemble the
input-image, such as the example in subﬁgure 38(j). This mostly seem to
happen in images with a very low colour range, which give very little contrast
for the minimizer to work with.
Since this wasn’t an exercise in eﬃcient (time-wise) noise-parameter estima-
tion, optimization of the code has not been performed. Thus noise-parameter
estimation for an input-image, with the chosen starting guesses take some-
where between 8 minutes for the fastest generation and 20 minutes for the
slowest, with a size of 300 by 300 pixels. The generation of solid-textures
used in Unity take about 20 minutes for a 300 by 300 by 300 texture, with
already estimated input-parameters.
The time spent estimating the input-parameters is not inherently an issue,
since it is parameters for a generator which are estimated and not directly
a texture to be used on a 3D model. Thus, if the shader where the input-
parameters are 100% aligned with the Python code, the generation of input-
parameters could take hours, and then used in the shader, where each frag-
ment would use milliseconds to convert the input-parameters to a colour to
be displayed on the fragment.

Ideas for improvement
In this section I’ve shown some possible improvements to the tool which due
to time-constraints was out of scope. The ﬁrst is Domain Warping which
can create complex anisotropic looking textures. The second is mapping the
histogram-matching to a colour ramp, which can be used to map the colour
of a fragment, to the correct histogram-matched colour, without having to
do the expensive histogram-matching calculation when running in a shader.
Domain Warping
An idea which could be expanded upon is Domain-Warping. Inigo Quilez
give a quick idea on how it could be implemented on his website[3]. The
idea is to stack fBm’s on top of each other with a varying oﬀset and scale
the result from each subsequent fBm. This would extend the complexity of
the autonomous noise-parameter estimation though, where the scaling has
to be autonomously determined as well. A quick test, with a single layer
domain-warp, to see how it could work with manually tweaked parameters
give promising results, see Fig. 50. First row is created with the same scaling
factor on X and Y. Second row has a factor 10 larger scaling on X than
Y. Both results in the rightmost column could be perceived as naturally
occurring materials, either a variant of marble, or precious stone, like jade.
Solid-texture examples can be seen in Fig. 51, where the convincing results
in 2D, is just as convincing in 3D.
With a one layer Domain-Warping, there is a massive time usage with the
code as it is right now. It takes roughly three times as long to generate a
single 2D texture with domain-warping, as it does without. With multiple
layer Domain-Warping this would grow exponentially. This tripling of run-
time is also present when creating the solid-textures, where without Domain-
warping, a 300 x 300 x 300 texture takes about 20 minutes. With domain-
warping the same resolution texture take about 63 minutes.

Map noise-map colours to a histogram-matched colour-

Since the created noise-map has a tendency to always follow roughly the
same curve when summing for the histogram-matching, a colour ramp can
be created from the input-image, and then always map the noise-values to
the corresponding colour on the ramp, such that values ≤0.0 is mapped to
the ﬁrst colour and values ≥1.0 is mapped to the last colour. The rest of the
values would be linearly mapped to the corresponding value on the ramp.
The ramp can be created as a list of 256 colours such that all the possible
colours in an 8-bit image can be represented. Colour-ramp examples for the
four input-images in Fig. 38 can be seen below in Fig. 52
This can be used when using the solution in a shader, where every pixel can’t
be histogram-matched to the input-image due to the performance overhead
it would cause, where a simple mapping to a ramp instead is practically
cost-less. The ramp is created by histogram-matching the input-image to a
smooth gradient, and is then saved as an image, but could just as well be
exported as a list along with the octaves, lacunarity, and persistance from
the noise-parameter estimator. The generated gradient should probably have
the same structure as the generated noise-map, so not a linear gradient, but
a curve which follows the same sine-like structure as the curve created by
the cumulative sum.
An example of this sine-like structure can be seen
in the histogram-matching section, Fig. 18 and 19, as the grey line, where
comparatively, a linear gradient would have created a straight line instead.

Varying lacunarity and persistance, based on the octave
An idea is to look into how the results would be if the lacunarity and per-
sistance was changed for each octave, instead of staying the same. If for
example the persistance was high for the ﬁrst two octaves, very low for the
middle octaves and high again for the last two, maybe some prominent macro
structures with small details of high frequency could occur. This would, like
domain warping, increase the complexity of the autonomous noise-parameter
estimation. The reason for the increased complexity is due to the increase of
parameters which needs to be estimated. It is no longer a single value for la-
cunarity and a single value for persistance, but a set for each, corresponding
to the amount of octaves.
Random choice of parameters for the minimizer
If time for generating the input-parameters isn’t an issue, an idea could be
to, instead of following a strict set of starting guesses as is done in the current
solution, perform a range of randomly chosen starting guesses, which would
lie in the valid range for the lacunarity and persistance. This range of guesses
could be in the hundreds or thousands of guesses, and from this maybe reach
a set of input-parameters which give a better result than the current set.
89
==PAGE:91==
Conclusion
This study investigated the process of analyzing an input-image, to use as a
base, to recreate the general texture and colouration of the input-image. The
texture and colouration was recreated in 2D as a noise-map by using Simplex
noise and Fractal Brownian Motion. The 2D image was then extended to
3D by stacking slices of 2D noise-maps to create a solid-texture. The overall
goal was to create a tool that digital artist in movies and games could use to
create convincing 3D textures of stony materials, without having much prior
knowledge about procedural generation.
The project is divided into two parts. Estimating the input-parameters to a
noise-generator from an input-image, and how to extract information of the
colour structure, also from the input-image.
For estimating the colours, two diﬀerent solutions where tried. First solution
was to do a simple linear scaling of the three channels in the noise-map,
such that the boundaries of the colours follows the same boundaries as the
input-image. This solution was not a success since the noise-map ended up
with a washed out look, due to the distribution of the colours between the
boundaries didn’t follow the same distribution as the input-image. Next,
another solution was tried, called histogram-matching.
This gave results
which perfectly replicated the colour distribution, for input-images which
has the same hue throughout the texture. If diﬀerent hues are present in
the same tones, this can’t be replicated with the current solution of stacking
gray-scale noise-maps to represent the three colour channels.
For estimating the input-parameters, a solution using a structural similarity
comparison, was tried. The structural similarity comparison was done be-
tween the Fourier-spaces of the input-image and the noise-map. The Fourier-
spaces were blurred before comparing, such that a bit of a leniency between
the images was introduced. This structural similarity index is then used in a
minimizer to ﬁnd the set of input-parameters which gives the closest look be-
tween the input-image and the noise-map. This worked with grainy uniform
input-images, but could not capture macro-structures in the input-images.
Then, this solution was extended to also look at the gradient progression of
the two images. If two images have the same gradients, there is a higher
chance for the images to have the same texture, compared to two images
with completely diﬀerent gradient structures. This helped tremendously to
replicate the macro-structures in the input-images.
90
==PAGE:92==
The project is implemented in Python, which has tools and libraries to handle
a lot of the calculations which needs to be performed during both the colour
matching and the input-parameter estimation. The generated noise-maps
are exported from Python to 2D textures. The presentations in 3D is done
in Unity, by creating 2D slices of textures in Python which can be stacked in
Unity to create a solid-texture.
The best parameters for blurring the Fourier-spaces, and the factor which the
gradients are multiplied with in the minimizer, are determined empirically,
by trying diﬀerent factors, until results which gave the visually best results
are found.
There are more or less two kinds of results from the estimator and colour-
matching:
If the colours in the input-image follows the same hue, a perfect replication of
the colour structure can be created, though if there are diﬀerent hues present
in the same tone, these hues will end up being mixed, and from this create a
washed out look, which resemble neither of the hues in the input-image.
The input-parameter estimator can successfully ﬁnd a set of input-parameters
which replicate the structure of a provided input-image, if there is suﬃcient
contrast / colour range in the input-image and the structure of the input-
image has an isotropic presentation. If there is anisotropic details in the
input-image, then these cannot be replicated with this implementation of a
simple fractal Brownian Motion generator.
This project shows that it is possible to create a tool which can autonomously
estimate input-parameters, from a supplied input-image, to a noise-generator,
such that the structure of the generated noise-map follows the structure of
the input-image, if the input-image is of an isotropic structure with suﬃcient
colour-range. It is also possible to replicate the colouration of the input-image
in the noise-map, if the input-image has colours in the same hue throughout
the texture.
