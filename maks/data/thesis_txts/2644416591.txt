Summary
Cloud computing continues its adoption and because of that software architectures are changing. Cloud
native application architectures take advantage of the cloud and allow applications to scale to larger
extents with less manual effort. These architectures are inspired by other relevant trends of DevOps,
microservice architectures and infrastructure as code methodologies. These architectures embrace the
elastic, dynamic cloud platform and achieve high deployment frequency with minimal lead time and high
operational excellence. In the realisation of such architectures, a high degree of automation is desired.
This requires non-trivial assessments and investment in open source, commercial or custom developed
tools and technologies. However, the configurations of these as well as the necessary infrastructure
resources are non-standardised and engineers utilise several graphical and programming interfaces to
configure and share configurations throughout organisations.
This thesis introduces configuration patterns that address the lack of reuseablility, composability and
maintainability of these configurations. Configuration patterns are defined in object-oriented code that
encapsulate configuration of tools, technologies, infrastructures and software processes in provider agnos-
tic constructs. Configuration patterns can be realised using several underlying providers and describe
infrastructure resources such as managed databases and continuous integration and continuous delivery
pipelines. Configuration patterns are defined and configured through common object-oriented code ab-
stractions that enable additional object-oriented programming concepts such as inheritance, delegation
and compositions. The developed CNOps framework hides internal code details of provider specific con-
figuration languages and configuration schemes and replace these with executable definitions of provider
agnostic configurations. The methodology and interface to define these architectures are demonstrated
with a case study of a non-trivial example of a cloud native application that demonstrates that we
can build technical architectures that exhibit minimal vendor lock-in and maximal provider portability.
The results of this thesis demonstrates that a provider agnostic strategy to cloud native application
architectures is feasible while still achieving complex and highly automated technical architectures.

Cloud computing has, in the past 10 years, seen significant adoption and because of that software archi-
tectures are changing. Flexera’s State of the Cloud Report 20201 shows that respondents are running
53 % of their workloads in public cloud environments and that the number is expected to reach 60 %
within the next 12 months. Software architectures changing is driven by the fact that operating appli-
cations in traditional data centres is different from operating applications in the cloud. New software
architectures are required to take advantage of the cloud environment and the cloud ecosystem. These
architectures allow applications to scale globally in short timelines with little manual effort. Hence, it
must be possible to allocate more resources or distribute applications to other places of the world with
the click of the button.
With cloud computing adoption, new software architectures and ways of organising IT teams have
emerged. These include DevOps, cloud computing and microservice architectures. These paradigms
originate from the software industry where large corporations such as Netflix, Amazon Web Services
(AWS) and Spotify with immense speed have disrupted their respective industries. By embracing cloud
computing and building distributed systems with autonomous teams, they have demonstrated exponen-
tial growth and global reach2. Furthermore, they have done so with previously unimaginable timelines.
A new architecture that combines key ideas and features from DevOps and microservice architectures
running in an cloud environment has emerged: the cloud native application architecture.
Cloud native application architectures combine the underlying elastic, dynamic platform of the cloud
with autonomous DevOps teams operating a series of applications or services in a highly automated
way. Adopters of these architectures strive for high deployment frequency, large scale of users and can
release new features to users frequently. Their applications must appear fast and reliable for their users
even when their applications are utilised in large scale. In addition, engineering teams should be able to
deploy changes with high frequency and with high success rate without compromising uptime. However,
achieving this is not trivial and organisations often utilise a mix of software as a service offerings, open
source projects and internally developed tools and technologies to help this.
In order to define cloud infrastructure and configuration of tools, the methodology of infrastructure as
code (IaC) and configuration as code (CaC) have emerged. IaC allows engineers to define resources such
as computing or network resources in definition files that are subject to software engineering practises
such as version control systems. However, infrastructure and configuration definition files are typically
expressed in domain specific languages or tool specific schemes in human-machine readable formats such
as YAML and JSON. These formats are frequently used since they allow flexible configuration schemes
that can be utilised by most programming languages. On the other hand, the format of the definition
files limit the composability and reusability of the definition files. These limitations are apparent when
several tools and technologies in various platforms are configured in several languages and formats with
redundant and copy-pasted configurations. The lack of composability and lack of reusability originates
from the fact that formats such as YAML and JSON do not allow modularisation as we know from
other object-oriented programming languages.

1 Introduction
In larger organisations, tools and technologies may be configured differently in different teams due to
the different challenges faced by the various teams. Maintaining different configurations become harder
over time since best practises and configurations are not shared and reused throughout the organisa-
tion. In addition, configurations may be specified through graphical user interfaces that are difficult to
document and share with other teams. This potentially leads to diverging configurations. Instead, we
want to have reusable and consistent configurations of the tools, infrastructure and technologies utilised
in the organisation. Organisations want to support a variety of projects developed throughout their
organisation to run with the same security principles and best practises thereby helping application
developers to achieve scaleable applications without requiring the individual teams to specify all infras-
tructure and tool configurations.
As an extension of the configuration- and infrastructure as code methodologies, we want to utilise
concepts from object-oriented programming to our infrastructure and configuration definitions. These
definitions can much easier be shared and adopted gradually by the different teams of the organisation.
An example of this is how teams can share pipelines that allows for continuous integration and contin-
uous delivery for a given type of service or how infrastructure resources can be configured to maximise
uptime and data consistency and data redundancy. This approach also allows standards to be defined
throughout the organisation that each application team can adhere to.
This thesis introduces configuration as object-oriented code patterns that address the lack of reuseabil-
ity and composability of the configuration- and infrastructure as code methodologies. Configuration
as object-oriented code patterns are from here on referred to as configuration patterns. Configuration
patterns define configurations of tools and technologies relevant for cloud native applications such as
pipelines, infrastructure and containerisation. This thesis enables, through object-oriented code, even
more software engineering concepts such as extendability, reuseability and inheritance. This approach
enables configurations to be composed without the limitations of domain specific languages or languages
such as YAML and JSON. Configuration patterns use a single language of object-oriented code to define
configurations of all underlying tools and technologies in a well-known programming language. This
simplifies several workflows where previously changes required several interfaces such as graphical and
tool specific configuration files to be used when one or more tools are to be configured. In addition,
the configuration pattern definitions are executable. The executability allows configuration patterns
to synthesise raw configuration files used by appropriate tools without requiring any changes to these.
The configuration patterns are defined through a framework, CNOps, short for Cloud Native Operations
Framework developed throughout this thesis. The name of the framework is inspired by the many -Ops
buzzword such as DevOps and DevSecOps that combines development and operation processes.
This thesis identifies configuration patterns relevant in cloud native application architectures. Con-
figuration patterns outline key architectural aspects, configurations and processes that combined can
help architects design, operate and scale cloud native applications. In addition to the identification of
these configuration patterns, this thesis puts forward a proof of concept framework that allows these
configuration patterns to be defined in executable object-oriented code which contains all the relevant
configurations as code. This object-oriented code can through tools developed in this thesis, synthesise
definition files in the desired format of the underlying provider such as a YAML configuration files.
Configuration patterns can be subject to software engineering practises and can be distributed through
package managers such as npm. This approach greatly reduces the redundantly copy-pasted config-
urations that are standard practise today when using configuration languages such as YAML, JSON
or other domain specific languages are used. Furthermore, the object-oriented configuration patterns
reduce the manual effort in maintaining and conforming to best practises defined in the organisational
context.
This proof of concept software framework, in combination with the case study presented later, demon-
strates how a well known programming language can be used to define, utilise and version configuration
patterns relevant in development and operations of cloud native applications. We show how config-

1.1
Research Methodology
In this thesis, the research methodology of design science is used [10]. This methodology outlines what
steps should be taken when some artefact is designed. The outcome of the methodology is often an
artefact that in the case of this thesis will be a proof of concept software framework. Before this artefact
is designed and implemented, the underlying problem is explicated and generalised to determine its rel-
evance. Next, requirements for an appropriate solution to the relevant use cases are put forward. The
requirements should not be bound to a specific solution but should include testable criteria that can be
used in a subsequent evaluation phase. The design and development of the artefact will be carried out
in an agile manner to ensure features are implemented systematically and continuously. Finally, the
artefacts are demonstrated and evaluated. The developed software artefacts are demonstrated through
the case study of the thesis. This case study will demonstrate the benefits of the artefact through an
example application with sufficient complexity to demonstrate that the solution can provide value for
large complex systems.
This thesis will use the design science methodology in a Requirements- and Development-Focused De-
sign Science Research [10] approach. This thesis will also perform evaluation focused research with
demonstration and realisation of a complex case study. This research approach is different from a large
focus on the problem explication that would outline and discuss existing approaches in greater details.
The design science methodology requires the definition of the problem statement, hypothesis and re-
search questions of the thesis. These are defined next.

2.3.2
Pipelines
Each microservice of the eshop application has its own distinct pipeline for continuous integration and
continuous delivery. Each pipeline ensures that changes to the code of the services are subject to the
same practises regardless of the content and programming language of the service. The steps of the
pipeline specify source code checkout, fetch of dependencies, build, static code analysis, run of unit
tests and finally a build and push of a docker image.
For this case study, a simple pipeline policy is used: all code changes to main line branches are done
through pull requests 6. This implies that changes are committed to a separate branch and pull requests
are opened targeting the main line branch, in this case the master branch. Pull requests are reviewed
by peers before they can be merged. The docker images are only pushed to the external docker registry
in the event the code changes are merged to the mainline branches, but are built when a pull request
is opened to ensure that these successfully build.
The eshop application includes definitions of pipelines expressed in the format of Azure Pipelines7, the
continuous integration service of Microsoft. The pipelines are expressed as code, more specifically in
YAML format in the associated configuration scheme. Since all the microservices are written in the
same programming language the steps of the pipeline for each microservice are identical with the excep-
tion of the service name and application code. Hence, it is desirable to utilise only a single definition of
these steps. Azure Pipelines offers this capability through pipeline templates8, which is utilised in the
eshop pipeline definitions. A single outline of steps are defined in the template shown in Listing 2.19.
The template uses a number of parameters to determine which service to build. In addition, as line 25
and 36 indicates, the template uses the defined docker- and docker compose files to perform the steps
of the pipeline.
In addition to pipeline template in Listing 2.1, the eshop application also has a pipeline configuration
file for each microservice. This file inputs the necessary parameters to the template such as the service
name. Moreover, this configuration file also specifies the trigger of the pipeline in question. The trigger
is a set of branch and path filters that determine if the pipeline should run. We show the configuration
file for the basket microservice in Listing 2.2. An almost identical file is also in place for the remaining
microservices. It is noted that the line 17 refers to the template defined in Listing 2.1 where the relative
file path is shown.

Monitoring and Observability
In this thesis, the case study will be instrumented with several observability techniques that allow us
to monitor our cloud native application. We must be able to determine the health and uptime of our
entire application and the microservices that this consists of.
Observability is a key practise in the operations and development of cloud native applications. This is
a larger topic that will be discussed in Chapter 3, but for now we note that we need to monitor the
microservices of our eshop application. We do this to ensure that our microservices are running, are
available for users so that users are not experiencing undesired errors. Observability and monitoring can
address various metrics such as error rate, response time and logs. Metrics and insights can be relative
to how each individual service behave or can describe and monitor the traffic that span more than one

In the deployment and operations of the eshop application, we must consider relevant infrastructure
definitions to run, scale and monitor our microservices. In operations of cloud native applications, an
orchestration platform is often used as the infrastructure on which the microservices are run. This
platform allows containerised microservices to be deployed, scaled and managed through tools available
to the platform. One of the most popular platforms to achieve this is the open source platform of
Kubernetes13. Kubernetes is used in this thesis since it allows for highly scaleable applications that
can run in most environments: private or public clouds. The eshop application is also designed to be

concepts will be described in later chapters. In addition, we note that the Kubernetes platform is a

we note that the self-healing nature of Kubernetes, where services are restarted in case of failures, as
well as the many tools and technologies that can run in Kubernetes are reasons to choose this platform.
Capabilities such as these are possible with monitoring of individual microservices run on the platform.

Kubernetes objects are also referred to as Kubernetes manifests throughout this thesis.

Backing Services
Backing services are services that can be consumed by microservices over the network. Examples of
backing services include database, message queues and caching systems. Backing services can either run
on platforms such as Kubernetes, but in other cases these are run external from Kubernetes clusters.
Backing services and infrastructure are often used interchangeably, but in this thesis we define backing
services as particular types of infrastructure resources. Infrastructure configurations refer to relevant
platform, network and security configuration that can be referenced by backing services, but is not a
part of the backing service scope. In the context of the eshop application, the different instances and
types of databases, message queues and storage services are the backing services.
The provisioning and configurations of these backing services are enabled through multiple scripts and
configurations files. Some expressed in executable PowerShell format and others in the format of Azure
Resource Manager18 language that uses JSON as the underlying file format. The defined scripts utilise
the Azure cloud command line interface19 and requires options to be passed and changed in order to
fit the identity and access rights of the caller. We note that the backing service definition files included
in the eshop application violate several best practises including the security measures for firewalls of
databases and message queues. Hence, these definitions should be avoided in the realisation of the
eshop application. We also note that the Azure Resource Manager language is yet another language
the engineers must grasp to develop and operate the eshop application. This language also has domain
specific functions, imperative statements and syntax20.
The method in which microservices can utilise and attach the backing services after these have been
provisioned is another interesting property of the eshop application. The eshop application repository
includes a local secret file where connection strings, DNS names and IP addresses can be specified.
These values are used in connection with the Helm templates to inject the appropriate configuration
parameters to the config map of the utilising microservices. The fact that config maps are used to store
the connection string with secrets inside violates yet another security practise. Instead, the connection
string should be stored in a Secret in the Kubernetes platform. Moreover, it is noted that this method
of attaching a backing service requires the provisioning and configuration to be carried out sequentially.
Hence, the provisioning process is disjoint from the development process and requires manual effort
that is not easily repeatable.

CHAPTER 3
Microservice Architectures
In this chapter, a shared understanding of microservice architectures is established. This chapter ad-
dresses key points relevant when discussing microservice practises and how these should be built and
operated. These points are compared to the architecture of traditional monolithic application architec-
tures. Finally, this chapter discusses challenges that appear in large scale microservice architectures
and how these can be addressed.
We initially outline a definition of monolithic applications.
3.1
Monolithic Applications
In this thesis, we define monolithic applications as applications that are deployed and run as one unit.
Even though monoliths are deployed as one unit, they may consists of distinct software modules. These
modules can have boundaries that help different development teams to work on various parts of the
application in parallel. Most commonly, the entire monolithic application is written in a single pro-
gramming language, but may also include several tiers such as the client-side, service-side and database
tier.
A monolithic architecture can be a conscious choice, but more commonly the monolithic architecture
has been a standard way of designing software for many years. Hence, many existing applications are
written in this way.
We list some of the common issues in monolithic architectures as outlined by Dragoni, Nicola et al. [2].
• Scaling: Due to the fact that the monolithic application run as one process, they also have to be
scaled as one. This requires that the entire application is replicated, which may require significant
wasteful resources.
• Technology Homogeneous: Monolithic applications are very often limited in their ability to
mix programming languages, framework and data storage technologies. Instead few technologies
are used in an one-size-fits-all approach.
• Deployed as one: Monolithic applications are deployed as one and requires that the entire code
base of the monolith is built and deployed even though changes only affect certain functionalities.
The speed in which this process can be complete can also increase as the size of the application
grows.
Newer architectures such as microservice architectures described here addresses these weaknesses, but
these come with costs.
3.2
Microservice Architectures
Microservice architectures define a particular type of architecture to design software applications. Mi-
croservice architectures consist of independently deployable services [13]. Each microservice runs in
==PAGE:30==
18
3 Microservice Architectures
its own process and environment and communicates through mechanisms such as HTTP or messaging.
The architecture of microservices models the collaboration of these microservices in order to address the
requirement of the application. The overall architecture describes where and how data can be retrieved
and lists the responsibilities of each microservice.
Each microservice contributes to the overall application by implementing a well defined amount of
functionalities. Typically, these functionalities are modelled around a certain business capability that
can modelled using Domain Driven Design (DDD) [3]. This approach is relevant as domain experts
and software engineers and architects all use the same ubiquitous language of a given context. These
context are defined as Bounded Contexts and provides IT architects with boundaries that are relevant
in large applications, especially if these are implemented using microservice architectures.
Successful implementation of microservice architectures achieve microservices that are complete inde-
pendent in their development and deployment processes. In addition, they exhibit loose coupling and
high cohesion by using microservices to achieve modularity [2].

3.3
Design for Failure
With increasing complexity of microservice architectures, we have to consider our microservices to be
designed for failure. Failure should in this context consider microservices that may fail, contain bugs
or fail to receive or send traffic. If we consider the eshop case study application, we can imagine one
or more of the microservices not servicing traffic. If for instance the order microservice is down, we
should still allow users to see the catalogue of purchasable item even though they may not be able to
finalise orders1. The fact that some of the microservices are not affected by a partial degration of the
application is a key benefit of microservice architectures. In many monolith applications, bugs or other
issues may affect the entire application rendering the entire application unusable.
In microservice architectures, we must continuously monitor the readiness and liveness of the individual
microservices. A simple endpoint can be exposed from each microservices that in turn executes a trivial

3.4
Microservice Observability and Monitoring
Observability and monitoring of microservices are essential techniques in larger microservice architec-
tures. In traditional monolithic applications, it may be possible to inspect logs on the individual hosts
running the application, but this is not feasible for microservice architectures. Here, each microser-
vice generates their own logs and metrics that should be correlated across microservices and hosts. In
addition, since microservices may call other microservice dependencies, we must be able to trace calls
through microservices and aggregate these to a centralised and single tool that can be used to inspect
these easily. These techniques are demonstrated in Chapter 8.
The metrics that we want to monitor for our microservice architecture can include both metrics from
the underlying host such as the memory and CPU usage but can also include business relevant metrics
such as how many users have logged in during the day or how many orders have been placed. These
metrics are also of interest for business stakeholders and should be made available for these without
requiring them to have technical knowledge of individual service log streams.
3.5

When developing and operating a non-trivial number of microservices, their architecture and topology
become even more relevant. The topology outlines which microservices interact. Several approaches
can be taken here and a mix of synchronous and asynchronous communication can be used.

From this example, it is important to consider the dependencies between microservices. A microservice
may interact with one or several other microservices. While these microservices may be independently
deployable, they are not necessarily useful as single units. Rather, the interaction of several microser-
vices combined address requirements and implement interesting use cases of the overall application.
If we consider the independent deployability of each service, it is clear that with a complexity of this scale,
the described technique of observability is essential. In addition, with a larger number of data storage
applications or databases, we must handle these tasks with a degree of automation. Key techniques for
ensuring that we continuously roll out changes are described in Chapter 4 and are demonstrated for the
eshop case study in Chapter 8.

DevOps
In this chapter, we describe DevOps in the context of cloud native application development and opera-
tions. DevOps is a mindset and a set of practises that allow organisations to organise their resources to
empower independent teams. DevOps is increasingly adopted in IT organisations and, in this thesis, the
high degree of automation in continuous deployment and continuous monitoring of applications of these
practitioners are relevant for the development and operations processes of cloud native applications.
DevOps shares many principles of agile development with key elements described in the Agile Manifesto
[1]. In particular, DevOps enables organisation to make continuous changes to their software through
efficient software delivery processes and high operational performance.
In this chapter, we outline some of the important practises known from agile development and DevOps
that ensure that engineers of cloud native applications can continuously deliver features to users in
a consistent and reliable way. This is achieved through a high degree of automation and automated
processes that can measured by metrics such as lead time, deployment frequency and availability.
We initially describe continuous integration and continuous delivery (CI/CD) that are key practises to
adopt in constantly changing cloud native application development and operations. Combined, these
outline automated steps and processes that help us deploy changes to our users fast and consistently.
4.1
Continuous Integration
Continuous integration is the practise of integrating changes from several developers into a single shared
code repository [8]. This repository should contain all the relevant files, scripts and configurations re-
quired for new developers to run and contribute new features to the software. In addition, engineers
should agree on a mainline of the repository, which in most version control systems is a branch that
specifies the newest version of the integrated changes of all the contributing engineers1.
Another important practise is an automated build and test process.
Projects must include scripts,
instructions or similar interfaces that allow the software to be built and execute relevant unit- or inte-
gration tests with as simple steps as possible. It is important to note that these should be consistent
instructions that can be run repeatably without requiring any manual configuration or interaction to
succeed. The specific tools to achieve this are often relative to the programming language of the soft-
ware, but build tools also exist that support a wide range of programming languages.
Martin Fowler recommends that developers commit to the mainline of the repository daily [8]. This
frequency is important to improve communication within the team and help detect any conflicting work
being done in similar areas of the application. When these changes are committed to the mainline, it
is important that a continuous integration server automatically checks out the new changes and subse-
quently builds and tests the new changes. In case any errors occur, the contributing developer must be
made aware of this and fix the issue as soon as possible. Typically, this is done by an automatic email.

A slightly different approach rather than committing changes directly to the mainline, is to run test the
build and unit tests of a merged version of the developer branch and the mainline branch before the
changes are committed to the main line of the repository. This is possible through pull/merge requests
that are supported by modern version control system hosting services such as GitHub and GitLab. With
these additional tools, developers propose changes to the mainline that are automatically tested and

In order to provide fast feedback to the developers, the speed in which the continuous integration server
take to build and test the changes are important to keep to a minimum. The time should be in minutes,
ideally a few minutes, thus giving developers great confidence in the correctness of their changes with

Continuous integration is a well known and proven method to improve the collaboration of code changes
of developers working on the same software artefacts. Automated build and test steps ensure that
changes are integrated often and help catch issues earlier in their development. Continuous integration
works well with test driven development techniques, which dictate that code changes should include
their associated unit- and, if relevant, integration tests.
4.2
Continuous Delivery
Continuous delivery extends continuous integration by extending pipelines to include steps that make
every code change ready for deployment. In addition to the test steps from continuous integration
stages, continuous delivery builds and stores artefacts that can be deployed to any environment with
no to little manual effort. The process and steps of the continuous delivery process is defined as a
deployment pipeline [9]. Deployment pipelines consist of multiple stages and can deliver the software
changes to the various environments of the organisation.
Figure 4.1 outlines important stages of a deployment pipeline. The commit stage is the first stage of
the pipeline where code changes are committed to version control systems such as Git. This starts
the deployment pipeline when changes reach the mainline of the repository. In this stage, the code is
compiled, analysed and binaries are published to a repository. The artefacts published should be tagged
with an unique version tag such as the Git SHA. The next stage of the pipeline is the acceptance stage.
This state can deploy the versions to a development or pipeline environment where integration-, smoke-
and acceptance tests can be performed. Subsequently, the same code artefact can be deployed to vari-
ous other environments. The number and difference of these environments depend on the organisation
and the software developed. However, shared for all of these is that these should be as similar in their
setup as possible.
In Figure 4.1, after the acceptance stage, test and operations employees can request new code build to
be released to the various environments such as UAT or production. This is very common as features
are possibly scheduled to be released at certain points in time, but in some cases we want to immediately
release our changes to the production environment when they pass our acceptance stage. This is known
as continuous deployment, which is frequently confused with continuous delivery that only states that
it is always possible to deploy a new version at any given time.
4.3
Continuous Deployment
Continuous deployment describes the practise of releasing every change to production immediately. In
order to achieve this, organisations must have successful continuous integration and continuous delivery
processes and pipelines. In addition, they must have highly automated acceptance tests that ensure

Progressive Delivery
Progressive delivery is the software delivery process of gradually deploying changes to growing subsets
of users. The purpose of this strategy is to minimise the risk of faulty all-or-nothing deployments where
a new version is rolled out to all users at the same time. Faulty deployments can be caused by various

issues, but in continuous deployment setups they can have big consequences for the availability and

The next few sections describe various patterns and techniques that are useful in progressive delivery
setups.
4.3.2
Blue-Green Deployments
Blue-green deployments is a technique that can be used to minimise the downtime when a new version
is released [9, 6]. This is achieved by having two copies of the production infrastructure, services and
possibly also databases. At any given time, either the green or blue version is active and serving users.
As new software is getting ready to be released, the non-active resources are updated, warmed and
smoke tests are run. When the new version has successfully been validated, a router changes the desti-
nation of the traffic to the updated version. The previously inactive colour is now active and serving the
traffic. If the new deployment is determined to be faulty, we can very quickly do the opposite operation
and switch the traffic back to the previously running version.
In this technique, databases and data pose unique challenges. These challenges originate from the fact
that there may be a short windows of time where several versions of an application are running. This
is the case where a single database tier is used by both the green and blue deployment targets. This
may be required for large databases where copying database from one deployment target to another is
simply infeasible [9]. With a single database tier, the two versions must be able to serve traffic from
the same database scheme. This is something that engineers should pay attention to and this issue is
also apparent in the next described technique of canary releases.
4.3.3
Canary release
Canary releases resemble blue-green deployments in the fact that new versions of applications are de-
ployed to infrastructure identical to the active resources. With canary releases, the new version can also
be smoke tested and warmed before proceeding with the rollout. The next step is what differentiates
canary releases to blue-green deployments. Canary releases start by routing a percentage of the traffic
to the new version. The remaining traffic is still served by the existing infrastructure. It is with this
technique possible to gradually increase the size of the subset of users on the new version leading to
a phased or incremental rollout - finally routing all the traffic to the canary release promoting that
release to the new stable version. The duration of the incremental rollout can vary, but it is important
that performance and key metrics from the canary release are monitored. If metrics are violated, rolling
back is trivial, as users can be routed back to the stable version. These processes can even be part of
the deployment pipeline and automatically trigger a rollback in case of potential issues.
By embracing canary releases, we get a relatively low risk test of our production capacity and the
performance of our newest changes [9]. Furthermore, this technique can also be used to test how users
interact with new features in a controlled manner.
4.3.4
Feature Toggles
Finally, we describe the technique of feature toggles, also known as feature flags. Feature toggles can
be described as a set of functionalities that can turned on and off or whose thresholds can change
dynamically. Feature toggles are typically toggle points or decision points whose decisions depend on
which feature flags are en- or disabled.

4.4 Organisation
27
Typically, feature toggles are configured on an environment basis through a set of configurations. These
configurations can be stored in the version control systems or in external systems. Shared for both these
approaches is the fact that our applications can utilise these from their runtime through environment
variables. This again highlights that the same source code is used regardless of the external feature
toggle configurations.
Feature toggles can be used to control work-in-progress features that can be enabled in private envi-
ronments but disabled in the production environment. Feature toggles come with a potential increased
test cost. This occurs if several feature flags interact with one another and certain functionality behave
differently for distinct combinations. It is therefore recommended to isolate feature flags from each
other to avoid this explosion of test space complexity.
4.4
Organisation
As hinted in the introduction of this chapter, DevOps also has an organisation aspect to its techniques.
The organisation should should be designed such that teams are empower to make changes relevant
for their projects. It should not be the case that several other teams are required to develop, setup
or release new features to users. Instead, teams should be empowered to make required changes to
infrastructure, databases and application source code in the same manner. The organisational aspect
should also bring developers, operations engineers and business analysts closer together. Teams often
include both technical and business representatives working with the same processes to foster fast
feedback and deliver new features faster.
4.5
DevOps Summary
These sections have shown several techniques and practises that can be adopted by practitioners of
DevOps. Techniques such as continuous integration and continuous delivery are of increasing importance
as business pushes for zero down time releases that can be carried out with great speed when features
are done. More advanced deployment strategies such as progressive delivery should be in place for
practitioners that have high availability objectives. Progressive rollout strategies using canary releases
with environment specific feature toggles enable high deployment frequency while still minimising the
deployment risk of all-or-nothing releases.

CHAPTER 5
Infrastructure as Code
In this chapter, we outline and describe the methodology of Infrastructure as Code (IaC). This method-
ology is growing in popularity with the growing adoption cloud computing platforms. We highlight
principles, best practises and goals of this methodology.
Furthermore, this chapter analyses and compares patterns and practises used to achieve the most
efficient workflows for continuously provisioning and configuring infrastructure resources. Workflows
here should enable continuous changes as also outlined in Chapter 4.
5.1
What is Infrastructure as Code
The infrastructure as code methodology enables resources such as computing and network configu-
rations and resources to be defined through code.
These definitions can be used to provision and
configure infrastructure resources. Infrastructure definitions are subsequently subject to best practises
from software development such as software development tools such as version control systems, test-
driven development, continuous- integration and delivery techniques.
Infrastructure as code emphasises consistent and repeatable definitions that can be executed to allow en-
gineers to provision or change configuration of infrastructure resources. Techniques such as continuous
delivery and continuous deployment can subsequently be applied to automatically roll out these changes.
The aim of IaC is to make use of the increased virtualisation, cloud and software defined networks of
the increasingly popular cloud platforms. Traditional infrastructure teams should support and enable
changes rather than being perceived as an obstacle for application development teams. Development
teams should be able to independently define, provision and manage the various resources of their appli-
cations. This will ultimately free IT staff from mundane repetitive manual tasks. Finally, changes and
improvements to infrastructure can be made continuously rather than large all-or-nothing deployments
with high risk.
5.2
Common Issues
Before expanding what principles are practises are recommended to following in the IaC methodology,
we first highlight common problems when not doing so. These are issues that have been known for
decades and occur in all sizes of IT organisations. Traditional in this context describes the setting
where organisations own and operate the hardware of their datacenter.
5.2.1
Snowflake Server
When operating a production server, a variety of actions are taken regularly to keep that server up to
date. Configurations are continuously changed and tweaked. This can happen through a mix of com-
mand line tools, graphical user interfaces as well as editing of text files. The issue of snowflake servers
occur when all these steps are not consistently tracked, documented or in other means abstracted from
==PAGE:42==
30
5 Infrastructure as Code
that specific server. The outcome of these workflows is described as aunique snowflake server by Martin
Fowler1.
The main issue of the snowflake server is the lack of reproduceability. There simply is no apparent way
to reproduce the identical setup of that server without significant manual effort. This is troublesome
if the setup is to be migrated to another target machine. The issue is also apparent as the number
of manually handled servers and resources increase. It requires significant effort to setup a mirrored
environment that can be used for testing purposes or to extend the compute resources of the existing
environments.
Continuously upgrading and tweaking these servers are risky. Upgrades and changes can have unpre-
dictable and undesired consequences. Configuration can have been in place for several years without
any auditability as to why or how a certain change is in place.
5.3
Principles of Infrastructure as Code
This section describes a set of principles put forward by Kief Morris [12] that ensure the dynamic
infrastructure are handled in the most efficient way. These principles address configuration drifts and
reduces the risk of creating snowflake servers.
5.3.1
Reproduceability
Any element of the infrastructure can with minimal effort be reliably be reproduced. In case of issues
this means that resources can be easily be replaced and recreated. New services and environments can
be provisioned with little effort. This principle addresses the issue of snowflake servers.
5.3.2
Disposability
Dynamic infrastructure platforms allow resources to be easily, created, configured, replaced or even
disposed. To take advantage of this, applications and systems running on this infrastructure should
continue to run even when resources disappear, change or are resized.
Former Microsoft employee Randy Bias coined the expression ”Cattle, not pets”2.
The expression
describes that infrastructure resources and in particular servers should not be treated like pets and
given memorable names, but should instead be disposed off in case of issues.
5.3.3
Consistency
Considering two infrastructure resources of identical or similar kind, these should be nearly identical.
These should only differentiate in the configuration that are required to distinguish these such as their
IP address. It should also be the case that with the same configuration of our infrastructure, we should
consistently reach that state over individual executions.
5.3.4
Repeatable Processes
Extending the reproduceability practise, any actions taken on the infrastructure resources should be
repeatable. Actions should here be considered steps taken to configure resources. This practise further

emphasises the use of scripts and configuration tools rather than performing changes manually.
This principle is most frequently violated in one-off tasks. However, even with a short term benefit of
doing this manually will eventually lead to configuration drift and lack of documentation of steps taken
to reproduce the system in case of disaster.
5.3.5
Design is Always Changing
Since requirements and the use of applications change over time, it is important that our dynamic infras-
tructure embraces constant change. Resources and their definitions must be designed to make frequent
changes with high confidence. This also points to the continuous provision of infrastructure resources
where new infrastructure resources can be provisioned in automated ways rather than engineers doing
this manually.
5.4
IaC Practises
Complementing the above high-level principles, Kief Morris also puts forward several practises that
should be followed when adopting infrastructure as code. These practises are not specific to any tools
but outline key practises that we should consider when assessing infrastructure as code tools and
providers.
5.4.1
Definition Files
The most important practise of IaC is to store the relevant configuration in definition files. These files
are meant as input for tools to carry out the work of provisioning or configuring the actual resources.
The definition files should be part of the code base and can be controlled with version control systems.
Different tools utilise different configuration file schemes and languages. These may be expressed in
popular formats such as JSON, YAML or these can utilise their own domain specific languages exclu-
sively usable by that tool. We remind the reader that in Chapter 2 Azure Resource Manager is an
example of a provider specific language to define Azure infrastructure resources.
5.4.2
Self-Documented systems and Processes
It is a common issue in IT organisations to have up to date, accurate documentation. Before adopting
IaC it is common to write comprehensive documentation for processes such as how to manually setup
some infrastructure resource.
These steps can however quickly become obsolete and leave room for human errors when performed.
With IaC, the steps necessary to carry out a process such as setting up a database are captured
in scripts and definitions files. This greatly reduces the documentation required to repeat a certain
process.
Finally, it is possible to generate more accessible documentation from the definition files
themselves reducing the risk of having obsolete steps documented.
5.4.3
Version everything
Version control systems (VSC) are a core part of IaC. The VSC constitute the source of truth of the
desired state of the infrastructure. Changes to the desired state should be driven by changes tracked
and versioned in the VSC.

Some of the key benefits of this include traceability, rollback capabilities, correlation through versioning,
visibility and actionability. Additionally, it is possible to enforce review policies if desirable. Policies
can control which engineers are allowed to make infrastructure changes. This is useful since these can
infer significant cost.
5.4.4
Continuously Test Systems and Processes
Best practises for testability should be applied to the infrastructure code. Automated test suites should
be a core practise of the teams that interact with the infrastructure definitions files. These tests should
be run continuously in a continuous integration fashion ensuring that the definitions behave as expected.
This is a key practise that prevents any unexpected changes to be rolled out as well as providing a fast
feedback for developers when changes are made. This practise resembles what is already described in
Chapter 4.
5.4.5
Incremental Changes
In agile development, features are often split in many distinct smaller tasks. This practise should also
apply to infrastructure code. The smaller changes can be deployed without necessarily being in use.
By embracing small changes, the risk of each change is reduced and if a change introduces an error it
is easier to find the change that resulted in that error, hence allowing the change to be rolled back or
corrected.
5.4.6
Availability
Infrastructure changes should ideally not impose reduced availability of that application or service.
Advanced deployment strategies can mitigate this. More simple alternative are also possible, which may
create new instances of infrastructure resources before provisioning old ones to ensure that rollback is
possible.
5.5
IaC Summary
This chapter outline key principles and principles of the infrastructure as code methodology. IaC defi-
nitions greatly reduce the risk of creating snowflake server by utilising versioned source code definition
files to consistently reproduce the state of infrastructure resources. In the assessment of tools used
to help engineers adopt the infrastructure as code methodology, one must consider how these enable
changes to be rolled out fast and consistently. Finally, we note that the methodology in itself does
not consider any underlying provider in which our infrastructure can be provisioned on, but that with
increasing setups and number of resource handled it is clear that these principles and practises become
even more important.

In this chapter we outline key characteristics of cloud native application architectures. We define high-

Famous examples of large IT organisation that have adopted the paradigm of cloud native applications

6.1
Cloud Computing
The National Institute of Standard and Technology, NISTs, definition of cloud computing [11] is a pop-
ular reference for essential characteristics that are important when addressing the vague cloud concept.
In this definition, cloud computing is outlined as on-demand access to a shared pool of configurable
computing resources.
The cloud computing era has gained huge popularity in the last 10 years and organisations as well as
private users are increasingly provisioning computing resources as their needs evolve. Flexera’s State
of the Cloud Report 20201 shows that respondents are are running 53 % of their workloads in public
cloud environments and that that number is expected to reach 60 % within the next 12 months. The
cloud computing paradigm has transformed our understanding of how computing resources are thought
of. Rather than physical components being purchased, installed in a physical location and manually
configured - storage and computing power is now offered as a service analogous to how we think of
utility such as power or water. The price of the utility is typically billed based on the usage.
From the NIST definition [11], we list the following characteristics that are essential in the cloud
computing definition:
• On-demand self-service Consumers are able to provision computing capabilities/resources with-
out requiring human interaction at all. This service is not limited to business operating hours.
• Broad network access Resources hosted in cloud environments are accessible through standard
protocols over the internet.
• Resource pooling The resources of the provider are pooled to serve several consumers through a
multi-tenant model. Physical and virtual resources are dynamically assigned. Consumers generally

The main takeaway from the above definition is the increased frequency in which our software appli-

6.4
Cloud Native Application Design Patterns
The cloud native definition from CNCF describes techniques such as immutable infrastructure and mi-
croservices to build cloud native applications. There are however additional design patterns that could
be utilised in cloud native application development. We describe some of these here.
Fehling et. al propose that cloud native application should be IDEAL [4]. We briefly describe each
property:
• Isolated State Application and session state should be stored only in a limited number of
application components. These may include distributed data storage application with stateless
services interacting with these.
• Distribution The resources of the cloud native application should be distributed. Hence, multiple
IT resources can be used to support replicas of the same workloads.
• Elasticity Cloud and infrastructure resources can be added and removed from the application.
The application can scale out instead of scaled up. The same components are split between replicas
distributed to other IT resources conversely from increasing the resources of existing resources.
The complexity of this approach decreases if the state of the application is isolated.

• Automated Management Cloud native applications should independently handle elastic scaling
and failure resiliency. Hence, in the operations of cloud native application, we should avoid manual
scaling and resilience decisions.
• Loose Coupling Cloud native applications must exhibit loose coupling with respects to time,
scaling and internal details of other components.
Combined, Fehling et al. define these as the architectural baseline of the modern cloud native appli-
cations [4]. It is worth mentioning that the properties of the isolated state should be considered in
combination with microservices. Here, it is viable to isolate the state of the microservices to stateful
applications such as databases and design the microservices to not store any state. This allows replicas
of the microservices to be distributed and elastically scaled giving the overall application increased
throughput. Techniques also exist that allow distribution and elasticity of stateful components. This is
however more complex as synchronisation between new and existing resources have to be considered.
If we consider the eshop case study of this thesis, we can note that this application exhibit these char-

6.5
The Twelve-Factor Application Manifest
The Twelve-Factor Application manifest5 outlines a set of principles and practices meant for developers
of cloud native applications. The principles can be applied to any programming languages and are
suitable for deployment on modern cloud computing platforms. The manifest, originally published in
2012, was a few years later followed by an extended version of the manifest: Hoffman, Kevin: Beyond
the Twelve-Factor App. We discuss some of the most relevant principles.
6.5.1
Configuration
The configurations of applications are likely to vary between deployment environments (staging, pro-
duction, developer environment). Configuration are in this context include:
• Backing service addresses
• Endpoints of other backing services or applications
• Credentials of other backing services or applications
• Environment specific feature toggles
The manifest recommends that we enforce a strict separation of configuration and code. Configuration
values can vary across deploys but code should not.
When our applications actually utilise the configuration we should access these through environment
variables. These are easily interchangeable and most, if not all, languages allow values to be utilised
with this approach.
This approach is a prerequisite of the presented technique of feature toggles in Chapter 4 and for
attaching environment specific data stores.

6.5.2
Backing services
Backing services are defined as services that applications can consume over the network. These should
be treated as attached resources. Backing services such as databases with various engines are the most
widely used examples of this. The attached resources are accessed through URL’s with credentials
stored in the configuration of that environment.
The application code should not make any distinction as to whether the attached backing service is a
local or a third party service. This enables a deployment to swap out the specific database resource
for the various deployment target such as the staging environment and the production environment.
In order to achieve this it is important that the the application respect the earlier described strict
separation of configuration and code, since URL’s or even credentials embedded in the application can
not be swapped as easily.
6.5.3
Build, Release, Run
The Twelve-Factor application manifest outlines the following three stages of the deployment process
of a cloud native application:
• Build Stage The stage in which the code repository is converted into an executable bundle. The
bundle should unambiguously be versioned to a specific commit of the code. Popular tools such
as Docker also requires the immutable bundles to be stored in an separate registry such as Docker
Hub.
• Release Stage The release stage references the executable bundle produced by the build stage
and combines it with the configuration of the deployment target. The resulting release should be
ready for immediate execution in the target environment.
• Run stage This is the final stage where the application is run in the execution environment.
This separation of the different stages prevents code from being changed at runtime and it furthermore
promotes the principle of having a single build to be deployable many times to various environments. If
a history of application builds are stored in an immutable ledger it also enabled rollback to be performed
with great speed and agility. This principle is very similar to the already described continuous delivery
technique in Chapter 4.
6.5.4
Dev/prod parity
The various environments of the application should be kept as similar as possible. The authors of the
manifest outline three historic reasons that contribute to environments such as development, staging
and production having substantial gaps:
• Time gap Code may be written, but weeks or even months could go by before the code is deployed
to production.
• Personnel gap In traditional settings, developers write the code while an team of operation
engineers actually deploy the code.
• Tools gap Developers may not utilise the exact technologies of their target environment. For
instance, SQLite could be used by developers where other environments utilise MySQL or similar
tools.
The identified gaps all contribute to the increased risk of deploying new software and organisations are
encouraged to minimise these gaps. In order to minimise these gaps the manifest highlights that the
applications built should be designed for DevOps principles such as continuous deployment.

Service Mesh
As outlined in Section 3.5, microservices interact with one another - either through direct or indirect
means. As the number of microservices grow, the complexity of this interaction between them increases.
Several approaches can be taken to address the connectivity, discovery, security and failure handling be-
tween the microservices. A typical approach is to introduce an API gateway or to utilise code libraries.
This approach does however introduce complexity in the application code, which may be significant
in size if different programming languages are used in a microservice architecture. A programming
language agnostic approach is to utilise a service mesh to address these challenges.
A services mesh works by deploying a light-weight reverse proxy, also called sidecar, to each running
microservice. This sidecar then intercepts all in- and outgoing traffic to the microservice. The service
mesh provider such as Istio can then be considered an infrastructure resource and can easily enable
complex retry mechanisms and encryption of the traffic between the services. Another key benefit of
service meshes is found in their observability capabilities. Metrics such as response times, success rates
and uptime can easily be exported with a unified interface regardless of the programming language of
the microservice in question. These capabilities are demonstrated in later chapters.

Kief Morris defines a stack in his book on Infrastructure as Code [12], page 152, as the following:
”A stack is a collection of infrastructure elements that are defined as a unit (the inspiration
for choosing the term stack comes mainly from the term’s use by AWS CloudFormation). A
stack can be any size. It could be a single server. It could be a pool of servers with their
networking and storage. It could be all of the servers and other infrastructure involved in
a given application. Or it could be everything in an entire data center. What makes a set
of infrastructure elements a stack isn’t the size, but whether it’s defined and changed as a
unit.”
While this is a meaningful definition in the context of infrastructure, this thesis will extend this defi-
nition to include other components, namely configuration of other tools and technologies. We do this
to include descriptions of how we actually make use of and configure underlying tools and technologies
adopted by the organisation. These should however not be limited to infrastructure, but technologies
such as pipelines and observability techniques are also relevant here.
Another relevant concept and usage of the stack concept is technology stack. This term is often seen in
job postings where technologies, tools and approaches are included in engineering job listings. Technol-
ogy stacks are relevant in the assessment of jobs since a significant portion of engineering tasks are not
directly materialised in application source code. These tasks can include development and operations
processes that can be achieved through configuration of various tools and technologies.

mon use cases that can be utilised throughout the organisation. As with any software development,
these are subject to change as new and existing technologies evolve. Hence, it must be possible to

This section outlines the requirements to allow application stacks, configuration patterns and composi-

The methodology and framework developed must address reuseability and composeability of the config-
uration patterns to ensure that we improve the current state-of-the-art approaches and methodologies.
[Req. 1] Configuration patterns must allow provider mappings and configurations to be defined for
more than one provider such that the same configuration patterns is usable for several providers.
[Req. 2] Configuration patterns should be expressed in an appropriate general purpose, object-
oriented language contrary from a domain-specific language or machine-human readable languages
such as YAML/JSON. This requirement enables reusable and composeable configuration patterns
by allowing configuration patterns to be parameterised.
[Req. 3] Configuration patterns should be versionable and should be shareable through package
mangers such as npm, NuGet, Pip or similar.
[Req. 4] Configuration patterns should be independently testable.
[Req. 5] The core engine of the framework should allow for new technologies and tools to be
supported.
This is possible through development of new configuration patterns and provider
mappings in the framework.
[Req. 6] Configuration patterns must be executable and should not require changes to external
tools or technologies. This is achieved by the internals of each configuration pattern knowing the
format and configuration schemes of the underlying providers.
[Req. 7] Configuration patterns should output declarative state definitions rather than imperative
steps to reach that state.
[Req. 8] Configuration patterns should be able to include infrastructure and other external re-
sources that, through references, becomes usable for the application. This includes databases
URL’s, service addresses etc.
[Req. 9] The framework should allow provider specific services and configurations to be defined
acting as an escape hatch to access the full underlying configuration schema of the underlying
provider if necessary.

configuration patterns to be reusable and solve the lack of composeability and redundantly copy-pasting
configurations of existing approaches where several languages and configuration schemes are used. It

7.4
Provider Agnostic Configuration Patterns
The concept of provider agnostic configuration patterns put forward here abstracts provider specific
parameters and configurations to common, meaningful parameters in a common programming inter-
face. From the provider agnostic parameters, provider specific mappers are defined that map to the
specific configuration scheme of the underlying provider. An example of this is a managed database.
Managed databases are offered by most, if not all, public cloud providers. They are very similar in
their capabilities, but their underlying configuration schemes, names, format and tools, used to config-
ure and provision these, differ significantly. Instead of grasping the various provider specific tools and
configurations, we instead want to specify a common managed database configuration pattern. This
configuration pattern requires parameters such as desired capacity and name. The internal code of the
configuration can subsequently map from the common configuration parameters to the provider specific
schema and format.
The fact that we can use the same configuration code pattern for more than one provider is a key idea
of the methodology of this thesis as it allows infrastructure, processes and configuration of tools to be
expressed in a higher abstraction that can then be mapped to the individual formats of the provider
when instantiated. This allows patterns to be migrated from one provider to another with significantly
less manual effort. This is ideally achieved by changing the single parameter that refers to the provider
we wish to utilise.
We list this in a concise definition that we utilise later in the design of the framework:
• Configuration patterns allow abstractions of provider specific configuration parameters and allow
parameters to be exposed by the configuration pattern. This allows the same configuration pattern
to be mapped to several providers through object-oriented code.
It is important to note that this provider agnostic approach is not exclusive for infrastructure defi-
nitions such as a managed database. This thesis also shows that other technology providers such as
pipeline tools, observability technologies and containerisation of microservices exhibit the same traits.
By designing configuration patterns in this way, we can for instance define what steps to perform in con-
tinuous integration and continuous delivery of a given microservice, in an provider agnostic format, that
can subsequently be mapped to several underlying configuration schemes matching different providers
from the same input of steps. By changing the provider parameter of the configuration pattern, we can
through the CNOps framework change the output declarative definitions of database, infrastructure
and pipeline configurations.
This approach minimises the risk of vendor lock-in.
Vendor lock-in is in this thesis defined as the
engineering cost of migrating applications, configurations or infrastructure from one provider to another.
In the context of this thesis, this could be a change of the continuous integration and continuous delivery
provider or a change of cloud infrastructure provider. However, in order not to be constrained by the
lowest common denominator capabilities, the framework should allow managed services, technologies
and configurations exclusively offered by a single provider to be utilised. This is a necessary escape
hatch to the provider agnostic configuration patterns, where provider specific configurations may not
be feasible to abstract to provider agnostic configuration parameters. Hence, even though configuration
patterns should be provider agnostic, we recognise that this is nor feasible nor viable for all possible

8.1
Executable Configuration Patterns
When building large, complex IT systems, organisations can benefit greatly from defining and dis-
tributing best practises across projects. A common approach to share and document these practises
is written documentation, often in internal systems. Documentation can also include example code or
project templates that can assist engineers unfamiliar with the technology stack of the organisation
to get started. The purpose of the documentation is to standardise how applications should be setup,
developed and deployed and to share experiences learned from various teams. Example projects can
be audited for security, scalability and utilise new technologies as these emerge and are assessed and
adopted. However, conformance to documented best practises, guidelines and design patterns pose
a greater challenge for projects over time and requires expensive manual work effort. Projects may
be older than the newest version of the best practises and the projects may require significant effort
to migrate to different infrastructure or make fundamental changes to the applications. In addition,
best practises and the documentation has to be kept up-to-date. Failure to do so will lead to teams
copy-pasting or drawing inspiration from similar types of projects that may lead to diverging setups
and configurations of tools and technologies over time.
By utilising object-oriented programming concepts to define, version and distribute codified, reusable
architectural patterns and configurations of tools and technologies that can be used in various projects
of the organisation, we address the maintainability of configurations over time. The CNOps frame-
work enables configuration patterns to be defined and distributed throughout the organisation. These
configuration patterns replace example code and template projects with executable definitions of con-
figuration patterns.
A configuration pattern can for instance define the pipeline steps or infrastructure of a specific type of
microservice. By defining and sharing this configuration pattern, developers of such types of services
can consume the configuration pattern in the same way as they utilise external code libraries. The
organisation benefits from having a central definition that is utilised by all developers of such types
==PAGE:64==

of services. Any changes to the central definition can with little to no effort be distributed to all the
consumers, which instantly enforces the new definitions to have effect in all the consuming teams. This
is possible by releasing a new version of the configuration pattern library. A great example of this could
be to introduce a new automated step in all pipelines that ensures that possible security vulnerabilities
in applications are found before software is deployed. Comparing this process to example code being
updated and emails sent to developers, configuration patterns can be distributed through moduralised
code which reduces the manual effort of conforming to architectural best practises and configuration of
tools and technologies to, in most cases, a negligible amount.

Pipelines describe the automated process of the continuous integration and continuous delivery steps

Pipelines also describe continuous deployment steps or allow manual interactions to deploy changes
to a given environment. Pipeline definitions should allow different jobs and steps to be expressed and
must enable deployments to several environments. The techniques of continuous integration, continuous
delivery and continuous deployment are realised using pipelines and have been described in Chapter 4.

8.3.3
Infrastructure
Infrastructure definitions are a particular difficult concept to encapsulate in configuration patterns.
This is due to the fact that the number of services, configurations and offerings from cloud providers
are immense. As of 2020, AWS offers more than 175 distinct services including computing, storage,
networking, developer tools and much more5. The same applies for the other major cloud providers
such as Microsoft and Google. Because of the sheer number of services it is not feasible to build con-
figuration patterns for each service offered by each provider in this thesis. Instead, we want to build
meaningful abstractions, that may encapsulate several underlying resources that hide internal details,
but allow environments, platforms and databases to be defined and provisioned.
As opposed to infrastructure and containerisation specifications, the tools and languages in which in-
frastructure resources can be defined and provisioned differ. They differ in the sense that each of the
assessed cloud providers of AWS and Azure provide a graphical user interface and a provider specific
interface for infrastructure definitions to be defined. Here, CloudFormation6 for AWS and Azure Re-
source Manager7 for Microsoft Azure are mentioned. These are complex languages that are provider
specific and often utilise language specific constructs to implement logic or utilise external parameters.
It is not desirable to support these languages directly as this would require significant engineering ef-
fort. Furthermore, it is deemed infeasible to attempt to design another language that replaces these
languages, as these are very large and contain a significant amount of provider specific idiosyncrasies.
On the other hand from these provider specific languages, infrastructure as code tools exist that allow
the same workflows and languages to be used across providers. These include but are not limited to
Terraform8, Chef 9 and Ansible10. The advantage of such tools is in the standardisation of the defini-
tion and provisioning process. Hence, regardless of the underlying provider, the same actions should be
taken to define, provision and configure infrastructure resources.
These infrastructure as code tools do however not address the fact that we would like to build provider
agnostic configuration patterns in object-oriented code that map to underlying provider specific infras-
tructure resources. However, they do provide tools and languages for expressing most, if not all, distinct
resources offered by the majority of cloud providers. The capabilities to actually define and provision

Another aspect that affects the assessment of tools is the difference between an imperative or a declar-
ative approach of specifying infrastructure resources or steps to carry out to provisioning these. As
outlined in the infrastructure as code methodology, we should strive for repeatable executions of our
infrastructure provisioning tools. If we allow infrastructure resources to be defined declaratively we
can focus on specifying the configurations of the infrastructure resources without outlining the steps to
achieve that state. This is in contrast to an imperative approach that outlines what steps should be
taken to achieve an end state. An imperative approach must take repeatability and idempotence into
account whereas that responsibility is outsourced to the infrastructure as code tool in the declarative
approach. Because of this, we restrict the CNOps framework to support declarative infrastructure def-
initions.

application on, Kubernetes. A Kubernetes cluster is an essential configuration pattern that we wish
to abstract to a provider agnostic object-oriented representation that can be mapped to the supported
cloud providers. Each cloud provider considered in this thesis has their own managed kubernetes service
offering, which is viable to support, since the Kubernetes platform is central to provision in order to
run our cloud native applications.
8.3.4
Backing Service
As previously mentioned, backing services are particular types of resources that are accessible and
consumed by applications through web protocols. The databases, data storage applications and message
queue of the eshop application are examples of backing services considered. Since these backing services
are consumed over the network, the CNOps framework must address how properties such as endpoints
and ports of provisioned backing services can be referenced and utilised by microservices. It is often
the case that parameters such as endpoints are determined only after the backing service has been
successfully provisioned and configured.
It should ideally be the case that we can reference these
properties without consulting graphical user interfaces or similar tools to determine the actual value of
for instance the url of a provisioned backing service. We design the framework such that backing services
can output fields or functions that can be used to reference the generated configuration parameters such
as the endpoint of the provisioned backing service. This was shown earlier in this chapter in Listing 8.4
and Listing 8.5. These listings also demonstrated an example of how the backing service a Redis cache
could be defined. The same approach is taken for other backing services such as relational databases.
8.3.5
Conceptualisation Summary
The above sections describe the various concepts of the cloud native application stack that our frame-
work must support. As outlined, we use several concepts from object-oriented programming to define
and compose configuration patterns that can be used or extended in an organisational context. Each
configuration pattern is supported by a number of providers and the capabilities and services of each
provider are considered building blocks that allow the lowest levels of infrastructure and capabilities
to be utilised. These details are however encapsulated in the configuration patterns allowing the same
pattern to be mapped to several providers and ultimately reducing the risk of vendor lock-in. This
approach enables practitioners to build meaningful abstractions of architectural practises and reusable
definitions of executable configurations of tools and technologies.

CNOps Framework
We now consider the design of the CNOps framework to define configuration patterns in this thesis. Ini-
tially, we focus on the usage of the framework more than the internal implementation. We present the
central object-oriented classes of the framework. These are implemented and shown in the Typescript11
programming language, a subset of JavaScript that allow types to be defined and checked statically.
This is the chosen language of the CNOps framework primarily due to the code libraries that we want
to utilise. The TypeScript language has a very complex type system and IDEs help engineers to benefit
from auto-completions and in-line documentation.
The next sections describe central concepts of the CNOps framework. These concepts are specific to
the CNOps framework and should be understood before the framework can be successfully utilised.
CNOps Stack
The cloud native ops stack is the most top-level class of the CNOps framework. Instances of the stack
can be instantiated without any parameters or optionally with parameters for the output directory and
if this should be emptied before each execution. The significant method of this class is the synthesise
method.
Stack Configuration
The stack configuration class contains all application stack specific configurations. These include the
choice of the various providers and relevant configuration properties of these. The stack configuration
can also specify relevant parameters and properties that are utilised in various components of the
application stack. The stack configuration instance is accessible to all other configuration patterns of
the same application stack.
Environment
An environment is identified by its name and specifies relevant configuration parameters that are to be
used within that environment. Environments represent units of resources, microservices or infrastruc-
ture resources that should only be utilised within that environment. As an example from the eshop
case study application, a public production environment and a private development environment can
be defined. Parameters such as the capacity, public access or version numbers should be independently
defined for each environment.
Service Definition
Service definitions are used in combination with the source code of a microservice implemented in any
given programming language. Here, we require that the repository in which the application code and the
framework code is stored is the same. This means that we do not currently consider poly-repositories
where the individual source code of each microservice are stored and versioned separately. We only
support mono-repositories that contain all application and configuration code.
Service definition instances specify the relative path to the application code as well as the identifying
name of the microservice in question. In the context of the shop application, we define a class for each
microservice of the application that extends the service definition class. An example of this has already
been shown in Listing 8.4.

CNOps Command Line Interface
In addition to the framework enabling application stacks to be defined as construct trees, we define
the associated command line interface (CLI) of the CNOps framework. This command line interface is
necessary for most of the operations of CNOps framework. For instance, the source code of construct
tree have to execute in order to output the appropriate configuration files. The command line interface
allows this. Furthermore, it also acts as a wrapper for the infrastructure provisioning tool, cdktf, pre-
sented later in this chapter.
The CNOps command line interface enables the following commands:
• Synth: This command executes the defined CNOps application stack and outputs the necessary
raw configuration files to the file system in the specified output folder or to the provider specific
file location of the repository13. The output files can subsequently be applied or used by other
tools.
• Deploy: In order to perform the provisioning of the infrastructure definitions this command can
be invoked. Depending on the underlying provider, related tools may also be invoked to perform
the provisioning. This command also applies any Kubernetes manifests to the provisioned cluster.
• Destroy: The opposite of the deploy command. The destroy command decommissions all the
defined infrastructure resources.

which we have earlier motivated is desirable for the CNOps framework.

for which we wish to generate object-oriented code definitions for. The CNOps framework supports
and bundles definitions for Azure and AWS as infrastructure providers and allows future expansions of
other modules and providers such as Google Cloud Platform18. The object-oriented code definitions for
most, if not all, infrastructure resource for the supported providers are automatically generated using
the cdktf library.

Backing Service Connectivity
In this section, we revisit the connectivity challenges of our microservices and the attached backing
services. The microservices are run in a Kubernetes cluster with appropriate security measures and the
backing services are provisioned externally from the cluster but may share security boundary with the
Kubernetes cluster. This is achieved by specifying the the resources residing inside the same subnet
and virtual network. However, this only enables the network configuration to allow traffic between the
services. In addition to that, applications must utilise appropriate credentials and utilise generated
endpoints to connect to various services.
To address the storage and utilisation of secrets containing service credentials and endpoints we note a
few key requirements.
• Some secrets are required at provisioning/deploy time.
• Secrets used in applications should specify references for secrets rather than their values.
• Secrets should be stored in appropriate locations and should be subject to authorisation checks.
• Clear text secrets should never be a part of the version control systems.
We mention that infrastructure resources should not be provisioned from local development machines.
Instead, pipeline providers should contain necessary secrets and parameters to provision resources de-
fined in the application stack. The supported pipeline providers considered in this thesis, allow for such
secrets to be securely stored. The secrets can subsequently be utilised through references in application
code as environment variables. Secrets stored in providers are not be readable after these have been
defined.
A set of static secrets are required to authenticate and authorise the underlying cdktf plugin. These
are obtained through provider specific identity and access management systems that we do not go into
details with. More interestingly is the secret management of new infrastructure resources. An example
of this is the SQL Server used to manage the relational databases of the eshop application. SQL Servers
can utilise external identity and access management system such as Azure Active Directory. Identity
and access management is out of scope for this thesis, but we demonstrate how applications can refer-
ences credentials or endpoints determined in or after the provisioning process of infrastructure resources.

The CNOps framework supports a Kubernetes Operator that enables progressive delivery. We do not
explain the concept of Kubernetes Operators here, but refer to the official documentation31.
The progressive delivery capabilities of the CNOps framework are offered through Flagger32. Flagger
automates roll-outs and promotions of new versions of applications run in Kubernetes known as canary
releases and A/B releases. Canary release specifications specify what steps should be taken to gradually
roll out new versions of application while measuring and comparing metrics such as success rate, request
duration and pod health. The results of this continued analysis can be published to popular messaging
systems such as Microsoft Teams and Slack. In order to utilise the Flagger operator, engineers specify
which service mesh is used. In the case of the eshop application, Istio is used, but other service meshes
such as Linkerd33 are also supported. The Flagger tool uses three custom resource types to control the
underlying behaviour of the Kubernetes operator: AlertProvider, Canary and MetricTemplate. These
are all available as classes that can be included in any service definition scope with appropriate pa-
rameters. These can be used to setup application stack specific metrics or alerts to be included in the
continuous analysis. In turn, these metrics use the metric collected and stored in the Prometheus addon
of Istio.
As an alternative to the manual configuration for each canary release specification for each microser-

duration and request success rate. The roll-out progresses with steps of 10% point increments until
finally reaching 50% where the canary promotion is considered successful if none of the metrics are
violated. If the progressive roll-out is not successful, the stable version is instantly promoted to service

CNOps Methodology
In this section, we describe the methodology of the CNOps framework in more details. The method-
ology describes how the framework and command line interface are to be utilised in order to roll out
changes to either pipelines, infrastructures or application source code. First we consider the application
source code.
The CNOps framework does not impose restrictions on the source code of the individual microservices
of the application stack. It does however benefit from a structured approach such as a microservice
per folder. The folder structure can help configuration patterns to reuse definitions that benefit from a
structured approach. This was demonstrated in the eshop pipeline in Listing 8.7. Instead, the CNOps
framework is centred around the various types of configurations that complement the source code of the
microservices. The application source code is not modelled in the CNOps framework, but concepts of
the CNOps framework reference the application source code. This is captured by the service definition
class that specifies the relative path of the application source code from the repository root.
The CNOps service definition class encapsulates the configuration of the microservice. It is essential that
the application source code adheres to the architectural patterns outlined in Chapter 6. This ensures
that configuration and environment specific variables are not directly specified in the application code
and are strictly separated. In addition to the configuration of each microservice, the service definition
scope defines the pipeline of the microservice. The pipeline uses the reference for the application source
code and configures containerisation steps necessary to build, version and store executable code images
in external registries. The generated code images are tagged using best practises of immutable and
unique tags. We demonstrate the above methods to reference application source code and generated
code images stored in external registries in Figure 8.8.

mentioned here in more details in Section 8.10. We do not explicitly label the role of the engineer that

by instantiating the database configuration pattern in the desired scope. In addition, the microservice

first provisioning the database manually and subsequently adding relevant generated endpoint and con-
figuration to the environment of the microservice. This fact eliminates the manual effort and enable
infrastructure and source code changes to follow the same software processes of continuous integration
and continuous deployment. If the infrastructure change is to be rolled back, the commit can easily be
reverted or the instantiated database configuration pattern can be deleted. This will decommission the
database and remove this from the cloud provider. Security measures to minimise the risk of data loss
can be put in place if desirable.
If we consider the roles of the engineers that may interact with the CNOps framework, we have to
consider the organisation of the practitioners. The two types of changes described here demonstrate
typical workflows of application source code developers and DevOps or platform engineers as realised
in the CNOps framework. These two roles have historically utilised very different software processes,
but with the use of the CNOps framework, we have enabled greater collaboration between these roles
as both infrastructure and application source code changes can be done in the same code repository
and with the use of object-oriented code.

9.1
CNOps Framework
Initially, we consider the composeability and the language of the CNOps framework.
The CNOps
framework uses Typescript to define all configurations necessary for cloud native application stacks.
The general purpose programming language enables tools, technologies and infrastructure resources to
be defined in a well-known object-oriented language. This language replaces the numerous domain spe-
cific languages and configuration schemes of the underlying tools and technologies. By utilising object-
oriented programming concepts, we have shown how configuration scheme and tool configurations can
be composed, reused and combined in domain appropriate constructs. The language furthermore en-
ables combinations of configurations spanning multiple tools or frameworks in the same configuration
pattern. By doing this, we are able to build better abstractions of cross-tool integrations. Moreover, the
provisioning and utilisation of for instance databases can be done through a instantiating a database
class and by exposing the connection string to the appropriate microservice. This replaces previous
expensive manual processes where infrastructure resources were provisioned with other processes with
minimal interaction between application developers and infrastructure/platform engineers.
In addition, we are also able to distribute and version configuration patterns that express configurations
together. The distribution and versioning through package managers improve the maintainability of
the configurations. Updating pipelines, database configuration or observability configurations can be
done by updating the version of a library.
In addition to the composeability of the configuration patterns in the CNOps framework, we introduce
provider agnostic abstractions. By introducing configuration patterns as abstractions of executable,
codified best practises and configurations of tools and technologies, we are able to build layers in our
application stack in provider agnostic configuration patterns. We allow engineers to build their own
abstractions with their desired default values and the CNOps framework distributes examples of these
such as the the managed Microsoft SQL database configuration pattern. By utilising provider agnos-
tic configuration patterns and developing provider specific mapping from the agnostic configuration
scheme to the format and properties of the provider specific offers, we achieve minimal vendor lock-in
and maximise portability without compromising the use of vendor specific managed services.
This
allows software architects to design appropriate abstractions without expressing these in provider spe-
cific idiosyncrasies. This is a key result and also has the benefit of alleviating practitioners from the
manual engineering effort of manually provisioning databases through provider specific interface and
subsequently exposing these to microservices using manual workflows.
In the development and operations of cloud native applications, we consider key metrics of scalability,

To accommodate the increased adoption of cloud computing and associated risk of vendor lock-in, sev-

The Open Cloud Computing Interface3 (OCCI) has published

and OpenStack6. The goal of these standards is to abstract the underlying cloud computing providers
and offer a common interface for IaaS capabilities. This is achieved by replacing the interfaces of the
underlying providers and express configurations in portable formats.
We have not investigated these standards in depth, but mention that AWS supports at least the Open-

of cloud offerings from the largest public cloud providers such as AWS and Microsoft Azure. These

Conclusion
This thesis has demonstrated how organisations can develop and operate complex provider agnostic
cloud native infrastructures, applications, development and operations processes. The language of the
developed CNOps framework replaces several provider specific languages and configuration schemes
of cloud providers with a common object-oriented language and simple command line interface. In-
frastructure as code, configurations of pipelines and utilisation of backing services can be expressed in
the same higher order object-oriented language while still benefiting from underlying managed service
capabilities. Infrastructure changes are continuously and consistently deployed using an automated
pipeline. By adopting the methodology described in this thesis, practitioners are able to build more
scaleable, maintainable applications and infrastructures that can be realised on several cloud providers
with maximal provider portability and minimal vendor lock-in.
In this thesis, cloud computing and the associated architecture of cloud native application architectures
have been described. Cloud native application architectures combine key techniques and patterns from
DevOps, microservice architectures and infrastructure as code. Combined, these individual paradigms
contribute to the elastic scalability of cloud native applications. Adopters of these architectures achieve
a high deployment frequency with robust and reliable automated processes to develop, operate and scale
their applications. In addition, their organisations are designed to empower independent cross-functional
teams that can design, implement and operate features in all layers of the cloud native application stack.
This thesis has discussed that in the adoption of a large amount of tools and technologies required to
develop and operate cloud native applications, we wish to share, distribute and abstract best practises
and configurations. This is addressed through identification of configuration patterns that encapsulate
these configurations through object-oriented programming concepts. The CNOps framework, designed
and developed in this thesis, has demonstrated how configurations and appropriate abstractions can be
defined, versioned and distributed. Configuration patterns describe key techniques and architectural
patterns that are achievable regardless of the underlying providers. Each configuration pattern chooses
from the supported provider mappings that map from the provider agnostic configuration scheme to
the provider specific scheme and format of the underlying provider. This approach abstracts provider
specific idiosyncrasies to reusable provider agnostic configuration patterns using design patterns from
object-oriented programming.
The case study of this thesis has demonstrated how an example of a non-trivial cloud native application,
its infrastructure, software tools and technologies can be expressed in the object-oriented programming
language of the CNOps framework. The eshop application has shown reusable abstractions of infrastruc-
tures, pipelines, containerisation, backing services and observability capabilities. These components are
essential in the development and operation of large, scaleable cloud native applications. The compo-
nents of the eshop application demonstrate a high degree of automation with configurations of complex
DevOps techniques of progressive delivery, automated infrastructure provisioning and distributed trac-
ing and logging. The case study application can, through the use of the CNOps framework, be deployed
to the AWS and Azure public cloud offerings with minimal provider specific configurations. Hence, with
minimal manual effort, the choice of underlying infrastructure provider can be replaced by changing a
single parameter.

In conclusion, the CNOps framework and associated command line interface enable organisations to
build, distribute and version executable, codified architectural patterns, configurations and setups of
cloud native tools and frameworks. The methodology developed in this thesis achieves this with minimal

By specifying our configuration patterns
in common object-oriented programming code, we can update our infrastructure, processes and best
practises in the same way as updating a code library. The case study of this thesis has shown that this
