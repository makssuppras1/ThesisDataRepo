Abstract
The number of software developers are globally rising as world wide problems are increas-
ingly solved with the inclusion of software. With the increase of new software products,
the increase in malicious code rises as well, increasing the necessity of security analysis
of software.
A previous thesis worked on analysing dynamically linked libraries in PE and ELF binaries.
It successfully was able to print the dependencies in given binaries of those formats.
However the tools developed could not co-exist as one singular tool.
This thesis focuses on combining the previous developed tools and extending that tool to
somehow analyze statically linked code in binaries of the same format.
The combination of the two previous tools was hindered due to implemented native code
for the two operating systems associated with the formats of the binaries. The inclusion of
a wrapper component would resolve this issue by implementing similar functionality using
non-native code.
For the extension of the tool, regarding static analysis, the tool has implemented code
that analyzes symbols in the binaries to find information related to statically linked code.
This required the binary to have been compiled with debugging information. The result
showed some information associated with the statically linked code, however there was
no data that explicitly gave indication of code being statically linked.

Introduction
Over the last years, the number of software developers has seen a significant increase.
In 2021 there were supposedly 26,9 million software developers [1] and the number is
expected to grow [2]. More software is developed, for many different reasons, but in
general more software are included in solving everyday problems all around world.
Developers tend to use third-party libraries to speed up development process. After all,
if the wheel is already invented, why reinvent it? Developers can instead focus on devel-
oping what is important for their software.
But the inclusion of third-party libraries might be the open door for malicious code. Third-
party libraries may serve as gateway for malicious activity both in the form of passing
viruses like Trojan horses and worms with them, as well as using exploits like typosquat-
ting.
Two different software tools have already been developed as part of a previous M.Sc.
thesis project. The aim here is to make a combination of both existing programs, from
the project, into a single program. Additionally, the aim is to extend the existing analysis
functionality to include analysis of statically linked libraries, as the original work primar-
ily focused on analysis for dynamically linked libraries. The idea then is to integrate the
analysis of statically linked libraries into the proposed multi-platform combination of the
PE and ELF tools.
The remainder of the report will be split into the following sections:

basic areas that the project is based on, specifically linkers, compilers and toolchains.

tionally, a discussion is included about the integration of the existing programs into one
entity, the approaches behind it etc. Additionally, an emphasis is also put on the linking
process with regards to analysis of statically linked libraries. Finally, an overview of pos-
sible attacks from statically linked libraries will be listed, to give a better understanding of
the malicious activities that can find its way to a system through static libraries.
Chapter 4 lists the requirements of the system, as well as the component/extensions to
be developed for the tools. These will be based on the general points documented in the
previous section.
Chapter 5 will cover the design of the tool. This chapter will cover both the combined

Background
To identify the content of the static libraries used in an executable file, and whether the
content is malicious, one must first know how to retrieve the static libraries. It may be
impossible to retrieve the static libraries used in an executable file [3] directly, but there
may be other ways for one to identify and process the content of static libraries used in
executable files.
But to be able to identify static code in binary files, one must first understand binary files
themselves. How are they created? What are the steps involved in generating an exe-
cutable? The process of going from source code to a binary is complex but will be crucial
for understanding exactly what they contain. Specifically for this project, the inclusion of
static libraries is important. Recognizing how they are included in the final binary should
help in identifying them within the binary.
In this chapter the compiler tool chain will be covered, i.e. what does it consist of and
what role does each part in it play regarding generating the executable file.
2.1
Compiler Tool Chain
A compiler tool chain is a set of tools which is used for compiling source code into a pro-
gram that can be executed on a target system [4]. It involves the translation of source
code into the targeted machine language and optionally linking with libraries. An example
of the GCC tool chain can be seen in figure 2.1. Basically the output of one part of the
chain will be input for the next. It is important to note that the output of each part differs
depending on the input. For instance the programming language used for development
limits the number of applicable compilers, as compilers are designed to specifically trans-
late specific source code. The target machine plays a role as well, as the instruction set
architecture (ISA) might be different from machine to machine. The ISA is an abstraction
of what the CPU can do and essentially determines which set of commands the CPU can
perform to execute the program instructions [5, 6].
Although tool chains varies, they often include these components [7]:
• Compiler • Assembler • Linker • Debugger • Libraries

2.1.1
Compiler
Nowadays most developers will be coding in a high-level programming language that re-
sembles real languages. This enables them to code something that humans can interpret
and understand better than the code that machines requires to run. The product of a
program in a high-level language are typically multiple source code files. These files are
clear and concise for humans, but machines are unable to run these files directly. They
have to be translated into something that computers understand. Machine code is the
language that is understood by computers, [9]. and it is code that is read by the CPU.
It is basically a sequence of 0’s and 1’s. Depending on the CPU and sequence of bits
received, the CPU will perform some instruction. However to have the CPU perform what
was developed previously in the source code files, the translation from a high-level lan-
guage to machine code is necessary. This is what the compiler will do [10]. The compiler
takes source code files as input and translates it into machine code that targets a specific
computer architecture. The output of the process are corresponding object files (.obj) to
the source code files.
It has to be noted that there exists different compilers with different purposes [11] but
in most cases, the purpose of the compiler is to take the source code files that humans
develop and translate it into machine code that the target computer architecture can un-
derstand.
2.1.2
Linker
In the previous step the compiler generated multiple object files, based on the number of
source code files. It is common for developers today to use code from external libraries in
their projects. For instance if a developer uses a method foo in their source code, and the
method is implemented elsewhere, then they don’t have to declare the implementation
themselves.
However once the compiler reads a source code file that uses the function, it generates
an object file that contains a reference to that function. Since foo is not declared in that
specific source code file, it cannot generate the corresponding machine code for the func-
tion. The purpose of the linker is to resolve that issue [12]. The linker will resolve it by
extracting information from the necessary object files. Object files can either be single
object files or archive files.
The point is that the linker will extract information from the object file that contains the
implementation of foo, essentially combining that object file’s implementation with the
object file that is equivalent to the source code.
The way the Linker solves the issue is through linking; Linking is the process of combin-
ing the object files and the type of linking depends on when it happens [13]. There exists
two types Static and Dynamic linking. The types will be described in the following sections.
Static Linking
In static linking, all the external references in the object file are resolved before the exe-
cution of the program. Therefore the executable at the end of the process will contain a
copy of the code for the external libraries used.
The pros of this type is that the executable will be more portable since all external code
used will be stored in the executable itself. The linking process also only has to happen
once until the program is ready to be used [13].

The downside of static linking is that it will require more memory if multiple programs uses
the same library and if a library is updated, the program using that library has to be re-
compiled to use the updated library.
Dynamic Linking
The alternative to static linking is dynamic linking where the external references are re-
solved at the time of executing the program.
The pros of dynamic linking is that it will require less memory depending on the amount
of programs using a specific library and contrary to static linking, the executable file do
not need to be recompiled every time the library is updated [13].
The cons of dynamic linking is that the library has to be installed for the program to work.
There will also be a small performance hit since the linker has to link the library once the
program is being executed.
2.1.3
Loader
The last part of the compiler toolchain is the Loader. The loader is a system program
responsible for loading the binary into main memory, and preparing it to be executed
[14]. Sometimes the process of loading the program into main memory is simply copying
the data from disk to main memory, otherwise it involves other steps such as allocating
storage [15].
In addition, the loader is responsible for is Relocation. Relocation is the process of as-
signing load addresses to the various parts of the program, adjusting it to reflect the newly
assigned addresses [15]. Programs may have a preferably desired area of memory, but
if that memory space is occupied by another process then the program has to be loaded
elsewhere, thus it has its addresses adjusted by the loader.

The content of the analysis chapter primarily revolves around subjects that can help in
identifying and analyzing static libraries in binary files of PE or ELF format. The authors will

The first step is to identify statically linked code in the binary files. To do that, it was

Optional Header: Following the file header comes the optional header. The optional
header is the most important header of the NT headers since the PE loader will look for
information here to be able to load and run the file [20].
The optional header consists of three parts; Standard fields, Windows-specific-fields and
Data directories. Most of the content in the standard fields is not that relevant, but the first
two bytes describe what bit type the binary is. If the value of these two bytes is 0x10B then
the file is a 32-bit file and if the value is 0x20B then it is a 64-bit file. The content of the
standard-fields are similar despite the binary being of different bit types. However if the
binary is a 32-bit type then the standard-fields will contain another four bytes describing
information relevant for the loader.
Depending on the bit type of the binary, the windows-specific-fields will be slightly different
as well. Firstly the offset of the fields are different due to the additional four bytes present in
the standard-fields. Additionally some of the content in the windows-specific-fields are of
different sizes. For instance, the first field of the windows-specific-fields is the ImageBase
and it’s size is either four bytes (32-bit) or eight bytes (64-bit). The most relevant field will
be the NumberOfRvaAndSizes which is four bytes describing the number of data-directory
entries that remains in the last part of the optional header.
The last part of the optional header is about the content of the data directories in the
binary. There are at most 16 data directory entries in the file where each entry is eight
bytes large and consists of two fields; VirtualAddress which describes the RVA. The RVA
will be explained later in this chapter. The last four bytes store the Size of the specific
data directory. Since there are no fields describing what data directory it is, the order of

3.2
ELF Format
Like the previous section covering the PE format, this section will cover the content of
an ELF file. Files following the ELF format can be of three different types: relocatable,
shared object or executable. [24]
Through the process of linking, relocatable files can be combined with other ELF files to
create shared object files or executable files.
Linkers use shared object files with relocatables to create new object files. Dynamic link-
ers can, alternatively, use them the first time that an executable file needs the shared
object during the execution process. The fourth ELF type, which is core dumps, will not
be analyzed in this paper, as it has no relevance for the thesis goals. [24]
Depicted in figure 3.2 is an example of an ELF file. The linking and execution views are
almost identical. The main difference to note here is that linking view relies on the section
header table, while the execution windows instead relies on the program header table
and segments. The linker takes sections from object files that are structured similarly to
the linking view, and creates segments to create an executable file. Because of this, seg-
ments are necessary if a file is to be an executable.
To illustrate with an example, linkers identify those sections that should be loadable, read-
able and executable, and then put those into one segment. Usually these are the second
segment that, among other things, contains instructions and read-only data. This process
is also known more specifically as static linking. [25]

3.5.1
DLL & SO
DLL: The shared libraries in Windows are referred to as dynamic link libraries or in short
DLL [31]. As briefly mentioned in the PE section a PE file can be that of a DLL file and it
will contain the various headers that an executable contains [19]. Although both executa-
bles and DLL are PE files, they do not contain the same things 1:1 due to their different
purposes; An executable indicates something that is runnable on the OS and although
the DLL is considered an executable file [19] it cannot be run directly.
So what is the purpose of a DLL?
As briefly described earlier in this chapter there exists dynamic and static libraries. DLL
are essentially dynamic libraries and contains the code that other programs can use. For
instance much of the functionality of the Windows OS is provided by DLL [31] and other
programs might be dependent on other DLLs that are provided.
SO: The equivalent of a DLL in Linux is a shared library file or shared object file (SO file)
[32]. And just like DLL files they serve the purpose of having code shared among multiple
programs. Similar to a DLL is a PE file, a shared object file is of the ELF format. Although
there are differences between the two types of shared libraries [33] the important thing
is, that DLLs are associated with the Windows OS and SOs are associated with Linux
OS. The purpose of dependency identification in [16] is to not only find the dependencies
of the binary which is being analyzed but also find the nested dependencies recursively.
But it is not expected that, e.g. a Linux system should have the necessary DLLs for a
particular PE executable. As such the dependency identification developed in [16] will
require adjustments for cross-OS support.
3.5.2
Compiling Issues
Combining the two tools in [16] was not possible as they were. The two tools were us-
ing OS-specific macros, structures and functions thus preventing the compilation of both
tools at once on either systems. For Windows the issue was that pieces of the code
used macros and functions defined in different windows libraries, exclusively used for
only windows systems. The OS-specific implementations were used for both systems
when performing the dependency identifications. On Windows functions such as GetSys-
temDirectoryA and GetWindowsDirectoryA were used to locate nested dependencies,
but obviously Linux systems does not have such definitions. In addition macros and data
types such as WIN32_FIND_DATA and INVALID_HANDLE_VALUE were used, prevent-
ing Linux systems from compiling (and thus using) the PE tool.
Obviously there was also issues the other way. The Linux tool was also using OS-specific
implementations from exclusive header files. Notable can be mentioned the fstatat which
closely resembles the non-OS exclusive functions stat. Another issue was the dirfd, an-
other Linux specific function that returns a file descriptor of the file.
The authors identified two ways of resolving this issue;
1. The code that used OS-exclusive implementations had to be rewritten so it enabled
other OS to compile and run it.
2. The code that used OS-exclusive implementations had to be replaced with a wrap-
per implementation. The wrapper function would then run one of two implementa-
tions depending on the current OS.

Attacks
In this section, attacks will be covered that have a possibility of being executed through
executable files. More specifically, only attacks and viruses relevant in terms of the possi-
bility of spreading through shared libraries will be covered, and only the most well-known
and/or relevant ones.
3.6.1
Malicious DLL or SO
Arguably the simplest case of an attack, malicious shared libraries are the cases where
an attack is hidden in some arbitrary functionality. The type of attack made from mali-
cious DLL or SO file can vary, from exploiting dependency confusion to acting as normal
functionality.[35]
Since shared libraries can be downloaded from both package managers and anywhere
on the internet, the risk of incidentally downloading malicious shared libraries, especially
when visiting unfamiliar sites, is quite high. The sources of the malware is usually a devel-
oper overseeing a possible vulnerability in a system, meaning that the user being cautious
does not always guarantee avoiding malware. Although these types of attacks are cen-
tered around dynamically linked libraries, which has not been the focus of this paper, the
authors still considered it worth having in this section. The reason is, first and foremost,
that this thesis expands upon an already existing project that handles dynamic linking.
3.6.2
Typosquatting
Another common attack type is one known as typosquatting, also sometimes referred to
as URL hijacking. The basic idea revolves around exploiting string comparisons, making
obscure differences that at first glance would be difficult to overlook. The general idea is
that any user may type a specific url, and by typing something only slightly different, e.g.
gogle instead of ”google”, or ”you-tube” instead of ”youtube”, will be misled to websites
that at first glance seem indistinguishable from the page it tries to imitate.
Similarly, typosquatting is also a possibility when looking at packet managers. Similarly
to mistyping a url, whne using the packet manager, a user can easily mistype a cer-
tain packet name, accidentally downloading something posing as the intended download,
while containing malicious intent. An infamous example of this is the crossenv package
incident, which exploited the weakness of closely related package names[36]
3.6.3
Dependency confusion
In many ways similar to typosquatting, dependency confusion is a supply chain attack
that substitutes malicious third-party code for genuine internal software dependencies.
The key to these attacks is that many package managers check public code registries
for packages before they check the private ones. As a result, it would be possible, if
a package existed in a private registry, that an attacker could register a package with
the same name within the public registry. Whenever a new install occurs, the malicious
version will be pulled in.

One of the more infamous examples of this type of attack is the one Alex Birsan, a security
researcher, has documented, where he used dependency confusion to hack into dozens
of companies, including Apple and Microsoft [37].
3.6.4
Detection
To detect dependency based malware, a recommended place to start is the CVE database.
CVE is in this instance short for ’Common Vulnerabilities Exposure’. From the database,
one can search a given file, and, if registered in the repository, the database will provide
identification number, description and punlic reference. Per CVE’s site, vulnerabilities are
defined as as mistakes in software code that provides attackers with direct access to a
system and/or network. [38] [39]
3.6.5
Antivirus detection
While not all antivirus systems work the same, there are two general outlines that most
programs follow.
The most well known and widely used technique for detecting malware is through sig-
nature detection. When using signature detection, the system writes signatures for pat-
terns found in files containing malicious behavior. The Other well-known and widely used
method is through heuristic analysis. The idea of heuristic analysis is to detect viruses and
malware by examining code for suspicious behavior. Heuristic analysis has multiple tech-
niques. One of them is the static method, in which a suspicious program is decompiled
and its source code examined. The code gets compared to already known viruses from
the heuristic database, and if a certain percentage matches anything in the database, it
will be flagged as potentially malicious [40].
Dynamic heuristics is an alternative approach, in which a suspicious program is isolated
in a specialized VM where the antivirus tests the code and simulates the outcome when it
is allowed to perform its functionalities. By doing this, it can then examine each command
when activated and examine them individually for suspicious behavior [41].

6.2.2
ELF Implementation
To identify the symbols in a statically loaded ELF file, the process starts when the user
types ”symbols” in the TUI. The first step for the algorithm is to instantiate variables for
the input file’s ELF header, Section header, symbol entry as well as a char pointer for the
string table.
Next, it instantiates the index of the symbol table as the result of the function for getting
the symbol table index. Additionally, it sets the section header variable as the return value
of the method to get the symbol table section.
If the entry size of the section header is 0, the program returns an error message informing
that the entry size of the symbol table not being set. Otherwise, the program calculates
the number of symbols in the library by dividing the section header’s sh_size variable with
its entry size variable.
Next step for the algorithm is for the system to loop through the symbols in the table and
reads them one at a time. If the functions either are not functions, objects, notypes or GNU
indirect functions, then the loop will skip over printing them, as they in that case would not
have any significance for our purposes. For each symbol, its name and index is retrieved
using the get_symbol_name_64 function, and then printed to the TUI. The program uses
the print_sym65_static_2 function to print out its information about the symbol, if it has
any.
In case the symbol is a function, the program then calls the print_aux_64 function to print
additional info that may be available for about it.
When the program is finished with running, it frees the memory that was allocated to the
symbol.

8.1
IDA F.L.I.R.T
IDA Fast Library Identification and Recogniztion Technology (F.L.I.R.T) is a tool developed
by Hex Rays [46] which intention is to identify functions in assembly code. They only
consider C/C++ code and therein standard library functions.
The idea behind the the algorithm is that it checks each byte of a program being disas-
sembled, whether that byte marks the beginning of a standard library function. They have
the signature file which contains the patterns of functions. They specifically check the first
32 bytes of a function against the signature file. The structures of the patterns are a tree
structure since some patterns share identical bytes [46].
8.2
Static Analysis on Binary Code
The paper with the name as this section’s title, by Bin Zheng [47] covers specific problems
and solutions with regard to static analysis in binaries. Overall the problem is much larger
than anticipated by the authors and the paper touches a bit upon those challenges and
how to solve it.
Although it is a survey it goes through the steps of static analysis and according to it, the
steps involves:
1. Disassemblers
2. Control Flow Graph Construction
3. Flag Registers
4. Alias Analysis on Binary Level
The paper contains several sources that could help in solving the problem. But it is also
apparent that the requirements for solving the problem is extremely difficult and there are
a lot of sub-problems related to static analysis as a whole.
The paper does mention IDA Pro and IDA F.L.I.R.T as being popular in disassembling
and can be used for construction of control flow graphs [47].
8.3
Going Forward
Based on the new sources, and the experience gained throughout the project, it seems
that identifying any given statically linked code is practically impossible. There needs to
be some conditions that can ease the process of doing so. A condition could be to have
a pattern which can be matched against the content of binaries. As documented in [47]
there are some algorithms that specifically solves part of the core problem and it would

be smart to understand those algorithms (and their related problems) on a very detailed
level, especially considering that the problem is not solved as easily as looking inside the
binary content (i.e. looking at the raw data from PE or ELF binaries).
Apart from understanding some of the other associated problems, the inclusion of pattern
seems important to include. But the issue with patterns is which patterns should one be
looking for? With IDA F.L.I.R.T they are looking for standard C library functions, but truth
be told it is a company specialized in solving this issue.
It seems the best action from here on out is to first better understand why it is so difficult
to identify statically linked code. [47] is a good place to start, and associated sources
within the paper can be used to understand, e.g. disassembling binaries and how one
can identify patterns here. The second thing to do is to properly define the patterns to
search for in binaries. Without a proper target, the goal seems unclear; What is it that
is being looked for? Statically linked code is not easy to identify and properly defined
patterns will limit it but at the same time make it possible to identify some code.

Conclusion
The purpose of this thesis was two things; One was combining two previously developed
tools in [16] that were dependent on the OS, in which they were developed. The tools were
developed to analyze dynamic dependencies in PE and ELF binaries. The other part was
trying to identify and analyze statically linked code in binaries of the same format.
The first purpose of combining the tools resulted in the combined tools essentially com-
bining the two tools from the previous project. The issue was that in some places of
development, native code was implemented, preventing compilation on the other plat-
form. The native code was replaced with a wrapper component that could resolve that
issue. Some additional adjustments regarding dependency search on the opposing plat-
form were made, as it did not seem applicable to perform nested dependency search. The
result of the combined tools was a tool that allowed the functionality of both previous tools
to exist in the same tool. Additionally, a layered architecture was implemented to ease
the process of running the tool, and also to extend the same tool with the functionality
required for the second task.
The second task of identifying and analyzing statically linked code yielded an experiment
where analyzing symbols from binaries, was compiled at various debugging levels. This
involved additional implementations for reading various records in the binaries. Unfortu-
nately, the results of the experiment did not aid in identifying statically linked code. Upon
further investigation from new sources, the problem of identifying statically linked code ap-
pears far more difficult than that of dynamic libraries and it was proposed that additional
investigation, regarding different topics, were required to be studied further, to enable the
tool to, one day, identify and analyze statically linked code.
