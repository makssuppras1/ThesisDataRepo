Abstract
In these years production companies stand in front of a digital revolution. Many pro-
duction companies have old machinery and upgrading the existing machinery is very
expressive. Blackbird has made a plug and play solution to collect data from existing
production equipment in a non-invasive way. The data collected is often counts of parts
passing through on a production line. From this data it is possible to derive downtime
and efficiency of the production line. It is possible through a web application to enter
stop causes manually and hence derive statistic overview of the stop causes. The purpose
of this work is to develop an industrial IoT button which can be used to match stop
causes on the production line with stops registered in the Blackbird application. The
development is focused on the hardware and software and the concept will be based on
a previous project. The goal is that at the end of this project the product is ready to
be produced.
The approach is to first analyze the prototype developed in the previous project and
then formulate a requirement specification in collaboration with Blackbird. From the
requirement specification, an ambitious project plan is developed to set the course. As
the development is based on a previous project the project in planed as an iterative
design phase.
First solutions to the features described in the requirement specification is discussed
and the optimal solutions are selected using morphological charts. The solutions are im-
plemented in both hardware and software design. A custom PCB is designed to accom-
modate the requirements. The PCB design features an ATmega1284P microcontroller
from Atmel and a RN2483A LoRaWAN module from Microchip. After the product de-
velopment, the integration of the product into the Blackbird system is presented. The
product integration is made using a Middleware service which connects the devices to the
Blackbird system. Furthermore the Middleware handles the devices and a management
web application is developed.
After all the development tasks the system is tested and verified on each sub com-
ponent through a series of test programs. The hole system from device to Blackbird is
also tested. From the tests, it was clear that some faults could be easily detected by the
Middleware. Battery levels are send periodic and from these it is possible to estimate
when a battery needs to be replaced. The combination of periodic sends of battery level
and the data send from the device it is possible to detect if a device is offline or online.
A method to synchronize the clock across the devices has been developed. Through
analysis of the residual it is possible to detect if a real time clock is faulty. From the
frame counter in the LoRaWAN header it is possible to detect if a device is in an infinity
loop of reconnecting.

CHAPTER 1
Introduction
A student design team from DTU has developed a concept of a flexible and inexpensive
Internet of Things(IoT) data collection device for industrial applications called Black-
bird Checkin. The device is designed to register stop causes in production environments
and send the data to a cloud server. On the server, the registered stop cause is compared
to the production flow and from this information it is possible to quickly and in real time
analyse the cost of different stop causes. This information is very desired when produc-
tion management needs to decide of when to repair a machine or buy new equipment.
The tech start-up company Blackbird ApS desires to further develop the prototype into
a professional product and create a platform which fits to a broader range of use cases.
The electrical hardware and the system integration of this transformation is the main
objective of this thesis.
Blackbird is a spin-off company from the emendo consulting group. Blackbird pro-
vides real-time data monitoring and analytics in the cloud for production lines. The
idea of Blackbird began as the frustration over the cost of installing one data collection
system across a production line. Many modern machine manufacturers have some sort
of proprietary data collection which often is difficult to interface and many production
lines have very old but fully working machines which cannot collect data. Upgrading
machines and/or PLC equipment is very expensive. The consultants in emendo saw an
opportunity to develop a better data collection solution using modern technology. They
developed a ”plug and play” solution where a sensor is connected to a “blackbox” which
transmits the data through a mobile modem to a cloud server.
IoT in industrial production is forecasted to be the 4th industrial revolution. The
term Industry 4.0 originates from the high-tech strategy promoting computerization in
manufacturing of the German governments. A German saying is ”Be ready in 2025 or
be gone in 2035”. Cyper physical systems are gaining significant attention when manu-
factures are looking to improve the production. According to Forbes magazine General
Electric predicts investment in the Industrial Internet of Things (IIoT) is expected to
top $60 trillion during the next 15 years[7]. The potential for a company to enable
industrial IoT for old manufacturing equipment with a plug and play solution is huge.
To enable this technological transformation modern technology/products must fit within
the existing industrial environments and equipment which are often very old.
Blackbird has made a product named Factbird which is a box to witch a digital or
analog sensor can be connected and the data is automatically send to a cloud database.

No configuration is needed. The sensor can count objects parsing through on a conveyer
belt or measure the temperature in a gas oven. The data is instantly available through
a website in real time. On figure 1.1 a functional overview of the Blackbird System is
shown.

From a sensor counting object on a conveyer belt it is possible to derive information
such as machine efficiency, numbers of breakdowns and the frequency of breakdown. The
next level is to gather information about what the cause is when a breakdown on the
production line happens. At many manufacturers, this information is handwritten in a
logbook or typed manually into an Excel sheet. Often operators/ mechanics neglect to
register the minor stops and their causes due to the inconvenience or they simply do not
have time to prioritize it. Having accurate data of stop causes gives a clear and precise
picture of the situation on the product line in real time and enables managers to make
faster decisions. It is evident that there is a potential place for a product which can ease
the registration of stop causes on industrial production lines. What characterizes the
data is that it is event driven. The Factbird needs to collect data at all time – even when
nothing is parsing through the sensor it needs to log that nothing has been produced.
The Checkin only needs to be active when an event is occurring, which makes it obvious
to use several emerging IoT technologies and standards. On the short-term only stop
causes are desired to be logged, but on the long-term any event based information could
potentially be logged through such an industrial IoT platform. The objective of this
thesis is to develop an IoT platform where the main purpose on the short-term is to
ease the stop cause registration on production lines. The platform needs to be easily
reconfigured for other purposes such as monitoring if a machine is on or off.

1.1 Previous Study

1.1
Previous Study
A previous study performed in the fall of 2016 by two Design and Innovation students,
resulted in a concept of easy stop cause registration[2]. The concept consisted of a simple
wireless push button which is placed next to a typical stop cause on the production line.
E.g. next to a filling machine and every time the operator would fix the machine he
pushes the button to register that this machine was the cause of the production stop.
The study showed that this could significantly save time for the operator and more stop
causes were registered and classified. Blackbird was very interested in the concept and
this was a motivation for the development of a professional and industrialized solution.
Furthermore, Blackbird desires to create a single hardware and software platform for
event registration.
By event registration refers to any event in a production facility
which is of interest to correlate with production data. The aim is to develop a platform
for this type of data collection.
This product is designed to be used in real life production facilities where manage-
ment decisions have high economic impact, hence there is a high requirement of reliability
and robustness of the system. The devices will be shipped to remote locations and de-
ployed in harsh environments. This can be compared to designing fault detection and
handling and for remote autonomous systems.
1.2
Problem Formulation
The Blackbird Checkin prototype has been developed on an Arduino compatible proto-
type platform called TvB LoRaWAN Node. The TvB LoRaWAN Node is mounted on
a PCB breadboard and connected to a LoRa module for communication. This setup
is great for prototyping however it is not suited for industrial environments nor mass
production. The problem is to transform the prototype into a professional product of
industrial standards. The prototype needs to be integrated into a custom PCB layout
optimized for production and low power usage. The production ready version of the
Blackbird Checkin must have following upgrades compared to the prototype:
• Custom PCB board
• Real-time clock for timestamp on data
• Prevent data loss in case of network faults
• Optional uplink over Wi-Fi
• RGB LED
• Optimised fault handling to prevent system from freezing
• Battery measurement and predictive estimation
• Powered by standard batteries with option for external power supply

• External trigger signals which are optically isolated
• 6 months of battery life
• Platform based design
A custom PCB board is highly desired compared to the current layout where two
boards are soldered together. One of the boards in the current prototype has handmade
traces which are very inefficient. The data collected from the Checkin is matched to
timestamped data from the production line. It is of high importance that the time on
the Checkin and the production data is in sync. A Real-Time Clock is needed to ensure
the time is in sync between time synchronisations. In the current prototype, there is
no time information is send with the data. The data is first timestamped when arrives
at the gateway.
The delay between the node and the gateway can vary from a few
milliseconds to several minutes, resulting in inaccurate data.
The prototype currently utilises a LoRaWAN module to communicate. LoRaWAN
has advantages and disadvantages and due to the disadvantages, it is desired to option-
ally replace the LoRaWAN module with a WiFi module. To enhance the ease of use it
is desired to use a RGB LED as a status indicator to clearly distinguish between success
and failure of operation.
The Checkin is intended to be mass produced and deployed in remote locations,
hence fault detection and automatic recovery are of high importance.
When a device is deployed on battery power and the reliability must be very high,
reliable battery voltage measurements are very important. But what is more important
than just measuring the battery voltage is to know when the battery voltage is too low
to operate properly. By predicting when the battery voltage is too low for operation,
maintenance can be scheduled in time to avoid downtime on the system.
Many customers have requests on measuring the on/off state of a machine or maybe
an output of a machine. It is desired to use the Checkin platform to gather such signals.
It is of high importance that the Checkin cannot influence the signal by introducing
noise or pulling the signal up or down. It is required to be optically isolated.

2.1
Industry 4.0
Industry 4.0 is the newest generation of industrialization and reflects the current trend
of automation and data exchange in manufacturing technologies[4]. Industry 4.0 is often
associated with Cyber-physical systems (CPS), the Internet of Things (IoT) and Cloud
Computing. With Industry 4.0 the aim is to create smart factories where Cyber-Physical
Systems monitor physical processes and make decentralized decisions based on real-time
data. A key element in this is that everything is connected through communication lines
and data is exchanged.
2.2
Sensor Nodes
A sensor node, also known as a mote, is a sensing device typically in a network of sensors.
The sensor node can gather data and in some cases proces the data. The data is either
stored locally or send through a communication line. Sensor nodes can appear in many
distinct network topologies, but common for all is that the sensors are in some way
interacting or in other ways communicating. Embedded sensor nodes typically consist
of a microcontroller, a communication module, a memory block, a power source and an
input interface. The microcontroller runs the application, process the data and control
the functionality. Microcontrollers are very popular in embedded systems due to low cost,
flexibility and high availability. Especially in systems where low power consumption is
a requirement, the microcontrollers gain significant advantages over a general-purpose
microprocessor. A communication module/strategy is essential for the sensor node to
be a part of a network. Depending on the application, the communication can either be
==PAGE:18==
CONFIDENTIAL
8
2 Theory
wired or wireless, short or long range. Many modern IoT sensor nodes utilize wireless
long range communication. Depending on the application, most sensor nodes need some
sort of memory, volatile or non-volatile. Often sensor nodes log data over time and then
transmit the agreagated data. Many microcontrollers offer internal memory, such as flash
memory or SRAM for this purpose. External memory is also very popular. Data inputs
can come in many forms. Most common are readings of analog values with an analog to
digital converter (ADC), however many digital sensors can be connected via SPI or I2C
protocols. Power supplies for sensor nodes is very dependent on the application. For a
wireless sensor network the nodes are often powered by batteries. For battery supplied
applications, power management is essential to get maximal operation time.
2.3
Wireless Sensor Networks
Wireless sensor networks (WSN) is a distributed network of sensors which is connected
wirelessly to exchange data. Wireless sensor networks is built of multiple sensor nodes/-
motes. The sensors gather data and pass their data through the network to a centralised
location. Many modern networks are bidirectional, enabling the nodes to both transmit
and receive data. Recent advantages in microelectronics, sensors and wireless communi-
cation standards have enabled the development of low-power wireless sensor networks for
long duration and wide area deployments. Simultaneously the production cost has dra-
matically dropped. All applications are restricted on different parameters such as power,
weight, range and speed etcetera. There are many network architectures available which
fits almost every need. However there is not one golden solution, each network archi-
tecture will be a compromise of performance based on physics. Many wireless sensor
networks are battery powered, hence energy saving networks are of high interest.
Often wireless communication networks are using the star topology.
In the star
network topology, each node is connected to a central hub, often called Router or Switch.
The hub controls the connections to each node and the communication is directly between
the node and the hub. This type of network is known from etc. Wi-Fi or GSM. The
advantages of this topology is the simplicity and communication is always following a
known path. The disadvantage is that some nodes might be able to reach other nodes
but not the gateway. If they can’t reach the gateway they cannot connect to the network.
This motivated another network topology called Mesh network. In a Mesh network the
nodes are partially or fully connected to all nodes within reach. The advantage of this
topology is that there is no need for a centralized hub and the network coverage is
extended to the combined range of all the nodes in the network.
Star and Mesh networks are the most popular topologies of emerging IoT wireless
networks. The trends are often focusing on minimal power usage and maximal range,
hence compromising the data rate.
Two emerging technologies both fighting to win the battle of becoming the domi-
nant IoT communication technology are Sigfox and LoRa. Both technologies are closely

2.4
LoRa and LoRaWAN
LoRa is a wireless modulation technology, which is the physical layer designed for long
range communication. LoRa connects a node to a gateway in a similar way as Wi-Fi
connect a laptop to an access point or any other star topology network technology. LoRa
is characterized by low bandwidth, low power usage and long range. LoRa operates in
the license-free ISM bands all around the world(Almost). The ISM bands are 433MHz,
868MHz and 915MHz. Similar to Wi-Fi this means that one does not need to obtain
a license to build and operate a network using these frequencies. The license free ISM
bands are essential for the wide spread of LoRa in IoT communities. It is important to
note that license free does not mean unregulated. The ISM bands are firmly regulated
in terms of transmission power, duty-cycle and bandwidth.
LoRaWAN is the communication protocol and architecture that utilizes the LoRa
physical layer. It can be compared to the IP protocol which the internet is built on. Lo-
RaWAN defines data rates ranging from 300bps to 5.5kbps, which is extremely low data
rates. Two high-speed channels at 11kbps and 50kbps are also available. LoRaWAN
supports secure bidirectional communication, mobility and localization. Unfortunately
the localization is yet to be practically implemented, however theoretical results have
shown optimistic results.

header and forward the data to the right customer without being able to decrypt the
payload. Only the sensor and the customer can decrypt the payload data.

2.5
LoRa Modulation
LoRa uses a spread-spectrum technique and variation of chirp spread spectrum(CSS).
There are two ways to increase the range by manipulating the modulation rate or the
transmitted power. The transmitted power is limited by regulation hence the modulation
rate is the only option. Manipulating the spreading factors from a narrow band to a
wider band, LoRa can tune the range of the network while transmitting with a constant
power. The compromise is the data rate. In general the data rate is in the order of the
bandwidth.

To understand how this differs from other more common modulations it is important
to understand how basic modulation works. On figure 2.3 a spectrogram of the ISM
band at 868 MHz is shown.
From figure 2.3 it is seen that the spectrum of the license free band is very busy.
The fact that the band is license free is an advantage because there is no need to buy
expensive licenses. The disadvantage is that anyone can use the band, hence it can
be just as busy as morning traffics in major cities. On figure 2.3 multiple modulations
techniques are seen.
On figure 2.4 two basic modulations from figure 2.3 are magnified. On-keying fre-
quency modulation works by generating a wave at a specific frequency and then turning
it on/off to represent 1/0. It is not very efficient but very simple to implement. Fre-
quency shift keying frequency modulation (FSK) works by discrete changes in frequency
to represent 0 or 1. FSK is widely used in modern application and has enhanced noise
immunity compared to On-keying frequency modulation. LoRa uses chirps to modulate
the signal. The chirps go from a low frequency to a high frequency at a constant speed.
To encode the data LoRa phase shift the chirps.
Decoding the LoRa modulation is very difficult if not impossible for the human eye
using the spectrogram. Decoding the LoRa modulation is actually very easy using an
inverse chirp signal. On figure 2.5 an example of a LoRa signal is shown and added
together with an inverse chirp in the frequency domain or complex multiplied in the
frequency domain. The results is the decoded symbols. Symbols are one or more bits
within one chirp width.
One of the key benefits of using LoRa modulation is the robustness against narrow-
band interferance. As shown in figure 2.3 the spectrum is full of narrowband signals.
On figure 2.6 the narrowband interferance is easily seperated from the decoded signal.
With this technology we can actually decode signals under the noise floor as de-
scribend in B. Note that the noise floor for this kind of signals are much higher than for
narrowband signals, hence the fact that LoRa is able to decode signals much under the
noise floor it does not extend the range. The higher noise floor level is proportional to
the level LoRa can decode under the noise floor which basically cancels out. There is
no black magic in LoRa which breaks the laws of physics. The key benefit of LoRa is
the resistance to noise in and out of the band. Due to the nature of the chirps, LoRa is
very resistant to Doppler effect on moving nodes.
As mentioned previously LoRa uses several spreading factors(SF7 to SF12). Spread-
ing factors are the speed of the chirp. A high spreading factor results in a slow chirp and
a low spreading factor in a fast chirp. Every time the spreading factor increases with
one the chirp speed is reduced by half. The amount of symbols within one spreading
factor does not change. The data rate is increased by the double for each decrease in
spreading factor. On figure 2.7 the corrorlation between spreading factors, bit rate and
time on air is shown. Within short range the data rate is high and time on air is low. As
the node moves away from the centre the data rate decreases while time on air increases.
This scheme is called adaptive data rate.

2.6
LoRaWAN Classes
LoRaWAN has three classes of end-point devices to fit a wide range of applications. By
end-point device means node/mote[Lora-allianceTechnology]. All three classes allow
bidirectional communication, but the difference is when it is possible to send a downlink
to the end-point device.
Class A is where the uplink transmission of each end-point device is followed by two
short downlink receive windows. Downlink communication outside the downlink receive
windows will have to wait until an uplink transmission from the end-point device. Class
A operation is the lowest power operating mode for devices which require extremely low
power usage.
Class B extends class A with scheduled receive windows besides the receive windows
which are followed by an uplink transmission. To syncronise when the end-point devices
receive windows are open, a syncronisation beacon is transmitted from the gateway.
Downlink communications will again be scheduled until the next available receive win-
dow, which is either the scheduled receive windows or receive windows after an uplink
transmission.
Class C devices will have continuously open receive windows only interrupted when
an uplink is send. This class is significantly more power consuming than class A or class
B.
In practice only class A has been publicly implemented. Class B and class C are
not available yet, hence the LoRa Alliance has not released a LoRaWAN stack with
these features implemented. The LoRa alliance releases(proprietary) and maintain an
implementation of the LoRaWAN stack.

Software and Firmware
The firmware of the TvB LoRaWAN Node module is implemented using the Arduino
software framework. Programming is easily available through the Arduino IDE and can
utilise the advantages of a huge community and shared code. A finite state machine
for the firmware was developed by [2] to document the intended functionality, which is
shown on figure 3.3. The entry point of the state machine is at the black dot. First
action is to join the LoRa network. If the network joint fail, try again. If the joint
succeeds put the device into sleep state. In the actual implementation, a finite limit of
join attempts is allowed before making a soft reset of the module. In sleep state the
microcontroller is put to sleep with only the interrupt register still activated. From sleep
state the device wakes up in waking state, when the button is pressed. In waking state,
it checks if the button is pressed for a long time or short time. A long press will make
a soft reset of the microcontroller and a short press will start the data transmission
procedure.
In sending state the transmission procedure starts and if the procedure fails the device
will reinitialise itself. The actual implementation of this will go to sleep state instead of
resetting. If data transmission was successful it will flash the LED to give visual feedback
to the user. Then it will check if it is time to send the battery level,. If so, send battery
level and go to sleep state. Otherwise go directly to sleep state. The implementation of
this does not behave functionally as described. Instead any attempt to send data, even if
transmission fails, the level of battery is send. This is a significant unintended behaviour
hence it will result in close to the double amount of power consumption. There is no
fault handing if the send fails and the data is lost. In figure 3.3 the state machine shows
that an interruption on pin 2 will put the device into sleep mode. This is likely a fault,
hence an interruption on pin 2 is a break condition from sleep mode into waking state.

input signals from the peripherals, process those signals, store it to the memory and
communicate the results. To save energy, the controller operates in low power mode
and controls power management of the other components, such as the communication
components. Some controllers feature one or more components internally. Currently
available on market there are controllers which feature Wi-Fi, LoRa, RTC and memory.

5.2
LoRa
LoRa is the predetermined wireless communication physical layer. There are two types
of LoRa modules available on the market today, a module with only the radio modu-
lation and a module with radio modulation and the LoRaWAN stack implementation.
The LoRa radio modules is the barebone radio and the host MCU needs to implement
the LoRaWAN stack. The LoRa radio modules are often interfaced via SPI. The Lo-
RaWAN module has an internal MCU and the LoRaWAN stack is pre-implemented. The
LoRaWAN modules are often interfaced via UART and AT commands. The LoRaWAN
module often have hardware encryption as the LoRaWAN encryption is requiring a lot
of ”horse power” if implemented on a microcontroller. There is significant time sav-
ing in using the LoRaWAN modules compared to implementing the stack by yourself.
Because LoRaWAN modules use an internal microcontroller with the LoRaWAN stack
implementation, it will draw more current than the barebone radio modules. Another
drawback is that it is not possible to change or fix eventual bugs in the LoRaWAN
modules. Some LoRaWAN modules have the possibility to access and program the em-
bedded microcontroller, however in none of the reviewed modules are the microcontroller
specifications sufficient for the functionality described in the requirement specification.
Hence the solution of using combined LoRa modules and microcontrollers are in general
disregarded. To use LoRa on public telecom networks the module must be LoRaWAN
certified. The network setup for this application is always maintained and setup by
our self, hence LoRaWAN certification is not a must. As mentioned in the introduc-
tion to this chapter the time to market must be kept low, hence it is not feasible to
implement the LoRaWAN stack our self when a ready made module can be bought.

Real Time Clock
The Checkin must feature a real-time clock(RTC) to ensure accurate timestamps on
data.
Some microcontrollers offer an internal RTC, however most of them are real-
time counters.
The difference between a real-time counter and a real-time clock is
that the real-time clock can be powered from a small coin cell battery and keep the
time while the microcontrollers are powered down while the real-time counter will lose
track of time. Another issue with built in RTCs is if the microcontroller crashes and
restarts it is not clear for most microcontrollers what happens with the RTC time during
reset. When RTCs get very small some of them needs an external crystal. From the
requirement specification, the RTC is required to have an error of less than 10 seconds a
year, equivalent to 0.3 PPM. However this design requirement has shown to be overstated.
In this design process the requirements are changed to accept a 2 PPM tolerance and
then implement a software solution to ensure that time synchronisation does not get
outside the specified tolerance.
Besides keeping the time, the RTC is used to wake
the microcontroller at preprogrammed time intervals, hence the RTC needs to have
interrupts.
Seven commonly available RTCs where considered and they are shown in Appendix

The microcontroller often runs on lower power than the battery, hence a voltage reg-

the microcontroller, however this is very unlikely for a low power microcontroller. This

5.6.3
Voltage Divider Switch Off
Another simple solution to decreasing the power the voltage dividers draw while not
being in use, is to simply switch off the voltage divider. This can be realized using an
N channel transistor and an I/O pin. The transistor is placed between the top resistor
and the node which connects the Vout with the resistors. The transistor is activated
using the I/O pin. This enables us to cut off the circuit and eliminate the current draw.
Depending on the transistor a minimal leakage current will flow, often significantly less
than without the transistor. In this setup, it is important to pay attention to how much
voltage the transistor needs to be activated. Most transistors needs a few volts above
the drain voltage.
The drain voltage will be Vout and the I/O pin must be able to
deliver a few volts above this. The scaling factor between the resistors can lower the
drain voltage sufficiently, but meanwhile the resolution of the measurement decreases.
In practical applications, this means that this will limit the resolution of your battery
measurement. E.g. the analog to digital converter can measure the range between 0
and 3.0 volt in 1024 steps, corresponding to the resolution 0.0029 volt per step. If the
voltage divider scales the voltage down to the range 0-1 volt, there will only be 1/3 of the
steps available, resulting in a loss of measurement resolution. To sum up this method

6.1.2
Memory
The memory solution in this configuration is chosen to be the internal flash. There are
many discussions on whether it is good practice or not to store data on the internal
flash in embedded systems. The argument against using internal flash for storing data
is that usually it has limited write cycles and block erasure. The memory often has a
finite number of program erase-cycles, often mentioned as P/E cycles. Most common
available flash chips are guaranteed to last approximate 100.000 P/E cycles per block.
Most EEPROM products guarantee an order of magnitude more cycles. Flash is block
based, which means that writing data to the memory is done in blocks. If you want
to store a single bit you need to update the entire block. Block sizes are often in the
magnitude of a few kilo bytes.
When a block is erased all bits are set high, which
also in some cases can influence the power usage significantly. In the context of this
application with approximate maximum of 50 daily data savings, it is not assumed to
set a significant limitation for the application. A significant benefit of using the internal
flash is that it is very easy to use and implement.
6.1.3
Real Time Clock
The ATMEGA 1284P does not feature an internal RTC, hence an external RTC is
needed. The choice came down to the DS1342 hence it was easily available and well
known from the DIY communities. The DS1342 is a low-power RTC with both clock
and calendar functions. The RTC is interfaced via the I2C data bus. The RTC features
an automatic power fail detection and switch circuitry. Furthermore, the RTC takes
leap-year into account. The accuracy of the clock is dependent upon the accuracy of
the external crystal/resonator. Both the load balance and temperature are significant
factors in the accuracy. The Maxim Integrated RTCs includes internal load capacitors
and bias resistors, hence it is essential to use a crystal with a certain frequency and load
capacitance.
The accuracy of the RTC is heavily dependent upon the accuracy of the crystal
and the accuracy of the capacitive load balance. If the capacitive load is greater than
specified, the oscillator runs slower. If the capacitive load is less than specified, the
oscillator runs faster. This error is static, because the load capacitance will not change
significant over the life time of the product which means this factor can be calibrated
for each hardware version. Besides the error introduced from the capacitive load match,
the crystals shift frequency as a function of temperature.
The signal between the crystal and the RTC is very exposed to noise, hence extra
precautions must be taken in the design phase. The crystal inputs of the RTC are very
high impedance in the order of 109 Ohm, hence the leads act as an antenna.
High
frequency signals from other parts of the system or electromagnetic signals/noise is
likely to introduce high frequency noise coupling. On-board there are frequencies in the
magnitude of 8 MHz to 2.4 GHz, hence a coupling from any of these will result in added
pulses making the clock run faster than expected. Since the noise is most likely radiated
into the crystal pins, extra care must be taken when designing the PCB layout. It is

Keeping the traces short and small also reduces the stray capacitance, which results
in better accuracy. A ground layer around the crystal and the signal traces will help
isolating the crystal from noise. Avoid other high frequency signals running close to the
crystal or signal traces, both in the same layer and the layers surrounding.

6.4.1
Initialisation Sub-state Machine States
The initialization sub-state machine handles initialization of the device and its parame-
ters from a clean start or a reset/restart of the device. The states are shown on figure
6.9. When the device enters the initialize state it turns on the blue led to show the user
it is powered on. The EEPROM memory is reset to remove old data. The LoRa radio
module is hardware reset and a wake signal is send to ensure the radio module is up and
running. A join command is send to the LoRa radio module. The radio module will try
to join a network maximum tree attempts before returning a result. If the join was a
success the next state is INITIAL_TIME_SYNC. If the join was rejected, the device
turns on the red LED and sets the microcontroller and radio to sleep for one minute
before continuing to the top of the INIT state. To wake the device from sleep the RTC’s
interrupt in configured prior to the sleep command.
If the join was successful the device enters the INITIAL_TIME_SYNC state from
where the current time from the RTC is send in a time synchronisation message. The
message is parsed by a server side application (See middelware in chapter 7 section 7.2)
where the timestamp from the RTC is compared to the timestamp of the gateway. The
transmission delay between the device and the gateway is often less than one second.
Hence, we can utilise this knowledge to assume the time on the device is approximately

just as wrong as in the beginning.
6.4.2
Idle Sub-state Machine
When the initialise sub-state machine enters IDLE state it will check if the network is
joined. If the network is not joined it resets the LoRa module and try to join agin if it is
more than one minute since last try to connect. If the network state is joined it will check
the amount of faulty send attempts. If the number of faulty send attempts is higher
than 3, assume network is not joined any more and go back to IDLE state. In case the
send attempts was less than 3, check if there is data available in the memory. If there
is data available go to SEND_DATA state. If there is no data in memory, check if it is
time to send a hearbeat or initiate a time syncronisation. The heartbeat is the battery
level sent at a daily interval to monitor the battery and see if the device is alive. If it is
not time to send either heartbeat or time syncronisation, check if the button is pressed.
If the button is pressed go back to IDLE state else go to GO_TO_SLEEP state. In
case there is either data is memory or it is time to send heartbeat/time syncronisation
go to SEND_DATA state. The SEND_DATA state multiplexes the data transmission
between Wi-Fi and LoRa depending on which module is configured. In SEND_LORA
state the attempts to send the data is executed. After trying to send the data go back
to IDLE state.

6.4.4
Time Synchronisation of Nodes
Time synchronisation of notes in LoRa networks is not easy because the delay between
node and network service is unknown. The the device where to poll the last time from
the network.
It would need to send a message to the application and wait for the
application to respond with a timestamp. The response will when be schduled for the
next downlink window. Depending on the delay in the network, it would often not reach
the gateway in time to be send with the two downlink windows after the transmission.
Nodes could be syncronised when the software was uploaded to the device, but it would
not be a robust solution. Time synchronisation is essential for this application, hence
we needed to invent a time synchronisation procedure. The data available is the non
synchronised time on the device, a time stamp on the gateway and a timestamp from
the network service. The time on air delay between the device and the gateway can be
calculated by the information of data rate and the amount of data. If a timestamp is
send from the device to an application, the application can calculate how much the node
approximately is out of synchronisation. If the timestamp from the device is subtracted
from the timestamp on the gateway the result is the time on air plus the syncronisation
error. Hence the time on air can be estimated an approximate synchronisation error can
be calculated. The synchronisation error can now be scheduled in a downlink and the
Checkin device can correct the time.

7.1
Message Protocol Between Node and
Server
The message protocol between the nodes and server is about agreeing on a format which
the data will be send. Often such a protocol consists of a header and a payload. The
header contains information regarding the data and the payload contains the data itself.
The most logical way so send data is to send the data together with a description of
what the data in order to avoid misunderstandings. When bandwidth is limited each
byte counts to reduce the size of data and then it is often not feasible to send all meta
data regarding the data. If we do not have the bandwidth to send the full description,
we can agree on some standard descriptions and give these a nick name. The nick name
can then be send with the data and the full description of the data is still available
while using less bandwidth transmitting the data. However, we now limit ourselves to
only send data that we have agreed on a description. This is basically agreeing on a
transmission protocol. It is essential that when designing such a protocol that not only
supports the needs of today, but also the needs of tomorrow, this can be a very complex
design exercise to invent a smart protocol, but often the simpler the better. In this
application, it is important that the protocol is very efficient regarding the data size
due to the nature of the LoRa technology. This means that we need to strip everything
except the data from the transmission. Not all messages need the same number of bytes
and the types of messages and the data is well known. To accommodate these needs, we
can reduce the data size by agreeing in which order the data is send and with a fixed
size. To maintain flexibility to send messages of multiple types and lengths we can add
a byte to identify the message type and or length.

CHAPTER 9
Automatic Fault Diagnosis
In this chapter the issue of automatic fault detection is looked at from the data stream
generated by the Checkins.
The automatic fault detection is used to automatically
alerting when a fault is diagnosed and potentially initiate action for automatic fault
recovery.
The fault detection and fault recovery are important components for the
overall robustness of the product/platform. The algorithms described in this chapter
are implemented as a service, continuously monitoring the Checkins. When the service
detects a problem and is not able to resolve it, it sends a notification with instructions
on how to handle the problem.
9.1
Battery Level
The battery level is periodically send from the Checkin. From these battery measure-
ments we first check if the battery voltage is below 4 volts. If the battery voltage is
below 4 volts we immediately notify the user about the fact that it is time to change
the batteries immediately. If the battery voltage is between 4.5 to 4 volts we notify the
that it is time to schedule a battery replacement.
9.2
Heartbeat
If a Checkin device has no activity within the last 24 hours and there has been no
heartbeat messages or time synchronisation, we must assume that the device is offline
and/or not working as intended. If the last battery measurement is below 4.5 volts the
service will send a notification telling the user to replace the batteries and see if this
resolves the problem. If the last battery measurement is above 4.5 volts the issue is
likely a connection fault either between the Checkin and the gateway or the gateway
and the network service. The user is notified with an instruction on how to debug the
connection problem.
9.3
Time Synchronisation Residual
Time synchronisation on the Checkin devices are very important for the Checkin to op-
erate reliably. Time synchronisations are performed frequently and the time corrections
==PAGE:84==
CONFIDENTIAL
74
9 Automatic Fault Diagnosis
are logged. When a device is reset the synchronisation error is expected to be large.
When a device is in normal operation the synchronisation error is expected to be very
small, less than 10 seconds. If the time synchronisation error is higher than expected
it is very likely that there is a problem with the RTC. Such a fault would require an
intervention from a technician, hence the service notifies the Blackbird team and not
the user.
9.4
Frame Counter
If a gateway is very poorly connected to the internet and the delay between the gateway
and the network server is above 500ms, there would occur an asymmetric situation where
the Checkin device can send data and it gets picked up by the gateway and forwarded but
the transmission is never acknowledged. The transmission is not acknowledged because
the acknowledge arrives too late at the gateway and the receive on the end device slot
has passed. This will cause the device to try to resend the data again and again. After
three tries the device will believe that there is no or a faulty network connection and
hence restart. The restarts will continue until the network delay is small enough for
the acknowledgement to arrive within the receive slot. In practise device will reset in
infinity. The data and joins does still arrive at the network server, hence the application
will see the frame counter resets to zero after less than 10 frames. If this happens the
service notifies the Blackbird team.
9.5
Conclusion
In this chapter four algorithms were presented to automatically diagnose a fault in the
Checkin system, and in some cases take actions for fault recovery. These algorithms are
highly usable when managing many deployed devices and they deliver valuable informa-
tion for debugging eventual problems.

10.7
Stability and Reliability Evaluation
During the test period, the Checkin devices has been extensively monitored.
Some
of the devices have failed during the test period due to software faults or accidental
behaviour. One of the first issues during the first week was that the 2G connection
to the gateway was too slow for the package forwarder to connect to the TTN and
receive an acknowledgement before the two receive windows were closed. Hence many
messages failed and had to be resend several times. It did not lead to data loss but
extensively battery use. The issue was discovered by looking through the incoming data
and noticing several data packages had the same data but a new frame count. The
connection between gateway and TTN is very critical for the stability of the system.
To increase the robustness a dynamic delay between retries of sending data has been
implemented to avoid a continuous loop of retries. The issue was discovered after one
week of use and was corrected by putting the gateway on a wired internet connection.
From the battery measurements during the first two weeks it was clear that the
battery performance was less than expected. Through thorough debugging it was clear
that the RN2483A radio module had a significant bug. In the documentation, the module
accepts a 4-byte unsigned integer as seconds to sleep. Unfortunately, the implementation
of the sleep time only works with a max value similar to a 4-byte signed integer. The
discovery has been filed to Microchip as a bug and they are currently investigating the
issue. After adjusting the code on the microcontroller the expected battery performance
looked promising. The issue was discovered after two weeks and did not lead to data
loss.
One device had odd spikes in the battery measurements during the field test. The
battery voltage sometimes raised significantly and in some cases also above 7 volts. Ini-
tially investigations were focused on the voltage divider gain might be too high resulting

A.1
Level-sensitive Interrupt versus
Edge-sensitive Interrupts
Most common microcontrollers can be programmed to operate in level-sensitive or edge
sensitive mode. There are fundamental differences which will function differently. In
level-sensitive mode, the interrupt is activated when the interrupt signal is on. The
interrupt function is executed and then it will send an End of Interrupt signal(EOI).
The interrupt will look at the signal line again and if it is still active it would immedi-
ately reactivate the interrupt. Depending on the application this characteristic could be
undesired.
An alternative to level-sensitive interrupt is the edge-sensitive interrupt. The edge-
sensitive interrupt would only trigger on an active edge of the signal. The interrupt is
triggered by an edge, then the interrupt function runs. When the end of interrupt(EOI)
signal is send, the microcontroller will not re-interrupt before the signal has restored
and a new edge is detected.
A.2
Programming ATmega1284P
When programming a microcontroller a piece of code is compiled into a binary .hex file.
The binary file is then programmed into the microcontroller using a programmer. A
programmer is a piece of hardware which can transfer the binary file into the microcon-
trollers program memory. For the ATmega1284p the program memory is 128 KB flash
memory. Some chips support programming through a USB/UART connection, this is
widely available for many DIY modules. Most chips can achieve this functionality us-
ing a bootloader. The bootloader is a piece of code, also called firmware, which allow
the microcontroller to be programmed through UART. The bootloader basically control
when to start the main application and when to download new firmware.
When programming a brand-new microcontroller, there are essential settings which
needs to be configured, which cannot be controlled through the firmware. These settings
are called fuses. The fuses are settings stored in on-chip non-volatile memory. The fuses
==PAGE:100==
CONFIDENTIAL
90
A Advanced Microcontroller concepts
determine how the chip will act, speed, voltage and bootloader are some of the settings
available in most chips. The fuses can be reprogrammed many times with a programmer.
Some chips offers security protections and/or the possibility to disable reprogramming. If
these fuses are activated the chip cannot be reprogrammed, which for many is undesirable
in the development phase. Despite the name the fuses has nothing to do with protecting
the circuit from overpowering. The fuse settings and functionality are often different for
each manufacturer. Some very common fuse settings are described below.
A.2.1
Clock Selection
Essential for all microcontroller is the speed which they will run at. The clock keeps
track of time for the chip, in general one assembly instruction is executed for each clock
cycle.
There are two main parameters for the fuse settings, clock source and clock
startup. Clock source distinguish between internal clock, external clock and external
crystal. Many microcontrollers have an internal clock to eliminate extra components
for most application where timing specifications are not strict. The internal clock often
drifts a lot in time and are very temperature/voltage dependent.
To achieve better
performance an external clock or crystal can be added. External clock is a square wave
signal, often provides by a clock generating chip. This is very rarely used in practice
due to component cost. External crystal is very often used in cases where a specific
frequency is needed or better timing performance. Having a low speed clock can help on
the power consumption for low power applications. For external crystals, these are two
types of components; crystals and ceramic oscillators. They both work the same way by
mechanical vibration. Crystals are often more precise, cost more and often needs extra
capacitors to load balance. Resonators are a cheaper option and often smaller in size.
Depending on the chosen clock source and the clock specification the startup time
may vary. The startup time is how many clock cycles the clock source needs before
obtaining a stable frequency. For many applications, it is not important to wake up
quickly, hence a conservative choice would be good. If the wakeup time is important one
must choose an external clock source which obtain frequency stability very fast. The
minimum wake-up time can be determined from the datasheet of the clock source.
A.2.2
Clock Output
Most external clock sources are very sensitive to changes of capacitance and impedance,
hence debugging the crystal with an oscilloscope often results in bad measurements.
Most microcontrollers have a clock output from where it is possible to indirectly measure
the clock frequency without the risk of influencing the original signal. The signal can
also be used to provide a clock signal for another chip.

A.2.3
Clock Divide
Many microcontrollers have the possibility to divide the clock rate to obtain a lower
speed than the clock input. Lowering the clock speed is often used to lower the power
consumption on low power devices. When lowering the speed of the microcontroller one
must consider the sampling frequency of inputs/outputs and the communication speed
for other components.

Reset Disable
Some microcontrollers feature a functionality where the reset pin is disabled and turned
into a normal pin. If the reset fuse is disabled, it cannot be reprogrammed through
a normal programmer. These can be used as a simple tamper prevention; hence you
would not be able to reprogram or read out the program with a normal programmer.
As tamper prevention, it is not bullet prof. Some are having luck with applying a high
voltage programmer to reset the fuses.
A.2.5
Brown-out Detection (BOD)
The brown-out detection fuse set what voltage the brownout detection will trigger on.
A brown-out is when the supply voltage drop, resulting in malfunction of the microcon-
troller. When the voltage drops, the clock will start slowing down and the memory will
malfunction. The brown-out simply turn off the chip when the voltage is below a certain
threshold. When the voltage return, it will reset and start over. When determining the
brown-out voltage consider the speed and voltage grade from the datasheet.
A.2.6
Boot Flash Size
The flash memory can be configured in two modes. If the application does not use a
bootloader the entire flash is available for the application code. If a bootloader is used a
section of the flash memory can be reserved to the bootloader. The size of the reserved
flash and the position in the memory is configured in a fuse setting.

A.3.1
Ring Buffer / Circular Buffer
The circular buffer is an essential component of many software systems, especially in
embedded systems which have digital inputs such as communication. The circular buffer
is a method to store data in the right places and to access the data in a certain order.
The circular buffer is a first-in first-out(FIFO) system which can be compared to the
queuing system in the local grocery shop. Two variables contain the information on the
write and read position, in other words the head and tail of the circular buffer. Every
time a new value is written to the buffer the write variable increases and when the buffer
is red the read variable increases. If a variable exceeds the size of the circular buffer, the
variable is set to the start position.
A circular buffer is a good strategy for a queuing system with a fixed size memory
allocation.
The circular buffer size is easy to scale up or down, or shift position in
memory, while not operating. A circular buffer can be implemented using four pointers;
buffer start position, buffer end position or buffer size, data start position, data end
position or amount of data. When implementing a circular buffer, it is important to
define the behaviour of the buffer when the buffer is full. Depending on the application,
overwriting old data or discarding the new data might be desired. When determining
the amount of data in the buffer, it is evident to look at the start and end position of
the data. This will introduce a problem when the buffer is full or empty, because the
start and end position will be identical. The solution to this problem is to count the
amount of data in buffer instead of using the data end position.

APPENDIX E
Hardware Development
Iterations
Three versions of PCBs have been made through this project. In this section the issues
at the two first version are listed. The third version has no known issues.
Version 1 issues
• Dimensions needs to be adjusted for the inner extrudes of the casing. Each outer
corner of the PCB needs to be rounded off with a negative rounding. The 2.4 GHz
antenna needs to be replaced to accommodate the dimension change.
• ICS programming headers for MCU is needed.
• Position of LED is off.
• Hole for button needs 2mm more clearance.
• Reset button for MCU needs to be separated from reset button for ESP8285.
• Landing pattern for the Optocoupler has wrong dimensions. The package was
mistaken to be a 4 pin SOP4 package, but is a 4 pin SOP6 package.
• Missing silkscreen layer.
Version 2 issues
• ICS and FTDI programming headers needs to be moved out of the centre of the
PCB. ICS and FTDI are not used while the system is operating. Pogo pins are
intended to be used instead of soldering pin headers onto the PCB.
• ESP8285 needs to be replaced by an ESP-WROOM-02 module due to benefits of
compliance certification.
• Programming pin headers for firmware update of ESP-WROOM-02.
• Programming pin headers for firmware update of RN2483.

• UART signal lines to the RN2483 module must be swapped.
