Abstract 
The purpose of this particular master thesis is to give critical assessment of the different SDN 
technological concept implementations, to provide an evaluation of the current state of the network 
programmability and to suggest some logical probabilistic aspects for development.  The currently 
well adopted distributed control networks provide stability and forwarding efficiency but it finds a 
hard times with precise and effective traffic engineering as well to propose a universal framework for 
new service introduction. The main missing pieces of the distributed control plain approach are the 
lack of high level of process automatization and low level of functional programmability. That leads 
to a lethargic network deployment, manually driven extensibility and slow responding time to the 
traffic demands. All of the aforementioned issues could be overcome by following the SDN logical 
concept- separating the control from the forwarding plain and to establish it in centralized manner. 
Nevertheless SDN is not a single instance of a solution but an entire spectrum of possible 
implementations. Starting with the control plain logical definition, it could be carried out in strictly 
centralized or semi centralized mode, which spreads the availability of a different technical 
implementations on the market. Strictly speaking the SDN concept is logical centralization of a 
control entity outside of the network forwarding paradigm, but not all implementations which follow 
that could be classify as SDN networks. Network virtualization is a vivid example, while it offers all 
the functional benefits of the concept (even in some cases extra practical features), it cannot be 
considered as a SDN network establishment fulfillment unit. The reason of that is the way of 
implementing all of the SDN features – pushing the programmability and the forwarding plain 
outside of the physical network (on the edges of it), and using it only as a pool of resources ( in 
general it creates a virtual overlays). That means the virtualization do not have any control or 
management impact of the physical network, which is statically flow defined (bridges) or maintains 
the distributed control framework. Of course is a matter of perception if virtualization is SDN or not 
and actually is a vigorously debated topic, but in the current master thesis it is strictly considered as a 
SDN concept implementation but not as covering the full meaningful potential of the SDN 
technology. 
 
        As already mentioned the SDN can exist as semi- centralized control approach, in order to 
introduce the SDN features (could be claimed for smooth technological transition) to the current 
distributed control plain networks in the form of a external PCE, which  uses the mechanism of the 
current environment  to enforce additional purposely driven control state (hybrid SDN). However the 
most functionally proven and efficient SDN technology which entirely covers the conceptual 
requirements of the newly emerged trends in an entirely satisfaction manner is the purely dedicated 
SDN framework.  When refer to pure SDN solution the most related example is the OpenFlow 
project, which in general determines the implementational reflection of SDN. It is entirely functional 
relevant with the desired state enforcement politics and requirements of the future datacenter 
networks, which gives the perspective of further abstraction value extension through the cloud 
elastic computing concepts. The result is a total network functional entities exposure to the 
application layer, which consumes it (by application developers) as constructional primitives in order 
to create higher value network services. Considering the provided absolute network automatization 
even further extended into a general network service deployment framework guided by a higher 
level of abstraction, it can be strongly claimed that SDN combination with Cloud management 
platform is the future of the datacenter and why not of the broadband networks control paradigm.

      Chapter 1 
 
Introduction- Network Programmability 
 
The concept of programmable network is one of the cornerstones of SDN. With the evolution 
of the technology, emerged the need to dynamically modify and interact with the network in 
response to a progressively increasing data loads. The ability to create, deploy and manage different 
service in a rapid and reliable way is the main requirement that drives all the researchers in that field. 
The result of their work is bound to bring a huge impact on customers, vendors, service providers. 
This is valid for various fields in telecommunication like mobile, local and broadband networks. There 
is a fierce competition between Internet Service Providers and it is directly affected by the changes in 
SDN. 
     There is a vast amount of processing and switching that must be handled in relation with 
computations that are service specific.  New programming tools have to be designed in order to be 
able to improve the performance, as well to maintain the scalability and expandability features of the 
network. 
    Before all of this could be achieved, a better understanding of the limitations of the current 
network state, as well as the fundamentals how to make the network more programmable. These 
fundamentals are associated with the deployment of new network programming environments, 
based on the idea to support network system across all of its conceptual layers. For example a future 
programmable network operating system could be based on active network execution environments 
operating on a commonly defined node OSs or open signaling network kernels supporting the 
coexistence of multiple control architectures.  
     All of issues which SDN is trying to solve are around how to make the network available for 
programmability and accelerate this process in a controlled and secure manner for the deployment 
of new architectures, services and protocols. There is one fundamental requirement that makes the 
network more programmable -the separation of communicational forwarding paradigm (i.e. routing 
engines, switching fabrics etc.) from control plain function. 
 
 Usually, the service providers do not have full access to the proprietary switch/router control 
environments (e.g. Cisco’s IOS operating system), algorithms (e.g.routing protocols) or states (e.g., 
routing tables, flow states)[27]. The deployment of new network services is a main priority in order 
to functionally improve and automate the existing control systems, but it is still hardly possible due 
to the softwarely closed nature of network nodes. So the first logical step forward would be to open 
up the box, thus enabling the deployment of third party control software and services in addition to 
the existing control and management environment. 
 
      In the current network state definition there are some noticeable limitations. The first and 
foremost is the static control behavior of the network, based on predefined logic. That is the main 
reason for the current stasis concerning innovations and new technologies regarding the networking 
field. The current datacenter network paradigm consists of a discrete list of protocols or statically 
pre-defined control configuration that were designed to connect different hosts regardless of 
topology, distance or available bandwidth but to take them into algorithmic consideration before the 
flow routes establishment. But those protocols are a box like solutions. Most of them are ment to 
==PAGE:9==
4 
 
address only specific logic, which in reality is very small part of the whole functionality that a network 
capacity is able to provide. Furthermore due to such particular proprietary  box design and the lack of 
control management abstraction is the main reason for the aforementioned limitation.  
    Never the less, the static control behavior has some benefits in terms of predictability, but at the 
same time that means higher costs for extending due to the hardware specific requirements in the 
face of the purpose build hardware as well all of that related with hard manual configuration labor. 
New technologies following the goal of achieving more dynamic network behavior has emerged. 
Before the boom of virtualization, applications used to reside on a single server and they were able 
to communicate with a limited set of clients. Today applications are deployed over clusters of virtual 
machines that distribute the traffic between each other while maintaining active and synchronized 
communication in between. That appears to be the heart concept of the Cloud. Also if one virtual 
machine fails it is easily replaceable. The Cloud is a major challenge for traditional networks and puts 
their evolution to a test. With network dynamical behavior advancements a new services have 
emerged like SaaS, PaaS and IaaS [25].   
      The inconsistency of the network policies due to the lack of automatization is another major 
drawback. During a network wide policy deployment (depends of the scale) a huge amount of 
devices, access tables and firewalls have to be configured. Complexity and variety of networks and 
network devices makes it very hard in terms of precision to implement any consistent access, 
security, and policy rules, which sometimes leaves the whole network vulnerable to malicious actions 
from third parties. 
 
      Complexity of deployment and sometimes configurationally inconsistency lead to the inability to 
scale. There is a demand for growth based of the new capabilities offered by the data centers. But 
the statically defined physical networks can not keep up with the development pace. It is not just 
about adding up and configuring more devices in order to match the upcoming demand. What 
actually most matters for the business industry is the responding time, down time due to network 
upgrades and the service availability.Those issues have to be resolved in order to use and manage 
network resources in more optimized way not just to add scale. 
 
    There is an ever increasing demand of adding new network services and modifying existing 
networks to better suit new application requirements. Such example is the introduction of integrated 
and differentiated services to IP networks with enhanced IP QoS. This is almost entirely manual 
process, hugely time consuming and usually costs a lot of money. The main goal of the 
programmable networks is to provide a way for easy deployment. This in general should lead to 
entirely new family of networks that explicitly support development and delivery processes for 
service. Programmable network architecture should be highly customizable and should incorporate 
and utilize a number of open programmable interfaces/APIs together with a certain toolkits and 
methodologies. 
       Major IT companies as Google and Facebook are facing scalability issues more than everyone else 
on the market. They have constructed and deployed massive systems of algorithms that ensure 
parallel data processing as well as complex metaheuristic algorithms to ensure that this data can be 
accessed with a reasonable speed. Those companies employ so called hyperscale networks that can 
provide very high performance and efficiency when it comes to indexing and search results with an 
instant speed [1]. 
 
     To stay competitive, carriers must deliver ever-higher value, better-differentiated services to 
customers. Multi-tenancy further complicates their task, as the network must serve groups of users 
with different applications and different performance needs. Even the simplest of operations that 
==PAGE:10==

direct the data flow and operating with the capacity of the network devices are very complex to 
implement (in terms of a manually defined line process) with what currently exists as network 
automatization process. They require cutting edge technology that is expensive to purchase and very 
costly to maintain. So that’s why the investment needed by a company in order to introduce 
something innovative grows significantly. 
 
    Different companies compete on the market to release the latest cutting edge services. But they 
are usually held back by the SPs, who can not keep up with the balance between a new technological 
releases and investment returns. The previous lack of open framework and open interfaces was 
narrowing down vendor’s interoperability and was an obstacle on a common goal of open network 
service development. Furthermore there was a severe gap between the requirements of the market 
and the current capabilities of the network. In a response to those obstacles the industry has 
committed itself on development of the SDN architecture as well as unified standards. 
      The programmable network model drastically changes network economics thanks to 
customizable software, thus allowing massive scalability, great resource efficiency, and virtually 
endless extensibility – across networks, devices, and correspondingly optimize the cloud. 
      The higher pressure for instant network provisioning and process automatization is the main 
driving force for the network industry to take a turn and revise the traditional network control 
architecture. Most of the conventional networks are hierarchical and use tiers of ethernet switches in 
a layered structure. This design previously have fitted the needs of client server computing, but the 
provided distributed control framework is very static and not suited at all for the dynamics of the 
network requirement demands expressed by the network service consumers.  
    There is a list of key trends in computing that motivates the need of new network design.The first 
one is motivated by the changing traffic patterns. In a applicationaly communication environment 
most of the data delivery is with client server structure. Applications of the new generation need to 
access a lot of different databases, web services,  other functional services in order to serve their 
final product purpose and thus the generation of high load of internal for the network traffic is 
inevitable. The fluctuations of the traffic patterns are hardly predictable due to the fact that a user 
can access a service via different devices, on different time spans from different positional locations; 
as well the functional range of the mobile devices is growing exponentially and requires much more 
network dedication, which puts additional demand for a network evolution. The new network model 
should allow the services to move huge amounts of data, thus it should be able to suppress the 
unoptimized network usage to a minimum state in order to provide more available bandwidth. 
   There are four milestones that the network model should be able to achieve in order to introduce 
itself as a programmable network solution. Open data access is the first one - the network model 
should allow getting information about the network itself, traffic flows, tenant configuration data, 
topology and etc, in order to be able to provide the applications with information and to allow 
applications to enforce the network with control state.  
    With such an open data interexchange the second milestone should be achievable : creating a 
programmable interfaces for control data exchange, which enables a service deployment via 
application environment.  
 
  The third milestone is deeply concerned with the technically specification of protocol definitions. 
Fine grain control for services and content storage in the network as well as standard 
interconnection in-between.  
 
  The last step is the future innovations which defines the forth milestone. The network 
programmability should be language independent via set of APIs destined for inter-modular

automatization and will be accessible for general purpose developer’s products. The whole idea of 
the programmable networks is build around the automatic configuration and provisioning services 
triggered by events from user requests or provoked by changes in the network state.  
  In another perspective where the SDN concept contributes with a great value is the network 
virtualization. The general implementation architecture allows multiple structural overlays to exist 
within the same physical network and to share the same pool of physical server capacity.  
   Virtualization of the network in this manner presents new level of innovation in network 
programmability concept that have not been considered before as such possible powerful solution.

      Chapter 2 
 
The Network Plains 
 
  
 The process of transferring data across the network involves different layers of logically 
established control concepts. The possibility to navigate a generated data across the physical/virtual 
environment (network) is inevitably accompanied by a control mechanism.  Some of the 
responsibilities of the mechanism are to route the data through a predefined graph or tree in an 
optimized for the traffic requirement manner, from its source to a specified destination and to make 
sure the data will be delivered in consistent/sequential and lossless state (as possible). Such a control 
mechanism needs to support a state, which is based on specific algorithms and management 
requirements. The management entity is concerned for maintaining consistent control mechanism 
(element configuration) and responsible to initiates a signaling if a logical or physical failure is 
detected.  Failure at the management functionality can lead to an incoherence of the control entity, 
which will corrupt the data forwarding.  Those tree entities can be specified as Data, Control and 
Management plains and any data transferred between two or more points must have participating 
logical elements classified in those entities.  
 
1. Control plain implementation 
   Basically the control plain can be implemented in tree general ways:  
I. 
The canonical Fully Distributed control plain – every device in the network has to take part of the 
control plain state. Depends of the protocols involved (link state/distance vector) the size of the 
device database and the compute power requirement can vary. The state of the network is 
bounded to the fast convergence state of the individual elements. 
II. 
Semi-Centralized (logically centralized) controlled plain – in this approach every controller has an 
instance of controlled devices. All the controllers (instances) communicating among each other in 
order to create a consistent control plain for all the instances in the network.  
III. 
Strictly centralized - all the network elements are controlled by one physically and logically 
centralized unit. [1] 
      The most common and mature approach used in the majority of the industry data networks is the 
canonical Fully Distributed approach. This method appears to be highly resilient to failures and easy 
to configure in industry environment (due to default inbox configuration). Unfortunately there are 
possible convergences difficulties in terms of the time required and rarely control data inaccuracy.  
Also the scalability can be limited due to the nature of the used control protocols (e.g. RIP – diameter 
of 15 hops).  Part of those challenges can be overcome by separating the network into routing 
instances and connecting them through exterior gateway protocol.  In this model, the participating 
==PAGE:13==
8 
 
elements exchange control information in order to distribute reachability information with the aim to 
establish a localized view of a consistent, loop-free network. The drawback of the distributed control 
paradigm is the propagation delay of the control messages in large networks. Sometimes the slow 
convergence rate can bring the network in inconsistent state and create black holes. Trigger based 
updates get this time to shorter span than the timer based updates technique (those are used 
together for better optimization and stability) [27]. 
    The typical approach of the modern SDN is to logically centralize the control plain - which is often 
realized as a SDN controller. Such a point of control is responsible to orchestrate, facilitates, and 
mediates between applications needing to interact with network elements and vice versa. Each 
controller depends of the implementation, has a limit of the controlled devices. In case of a network 
expansion, more controllers have to be attached.  The controllers managing the instances are 
synchronizing between each other and there is no need of multi-device manual configuration.  It is 
highly resilient to failures but still susceptible to those around the time of synchronization of the 
state. The horizontal scalability is not a problem, if more expansion is required, a new instance is 
deployed. The network convergence after failure is considerably shorter than the distributed 
approach. The main advantage of the centralized control plain is the capability to provide an 
extensive view of the underlying network direct to the applications through APIs, so there is no need 
the apps to have a direct interaction with the network elements, but with a few control points 
(controllers) which have to take care of collecting the detailed network information. A weak spot is 
the single point of failure, although backup controllers can be deployed for redundancy purpose.  
The strictly centralized scenario is mostly viable for experimentation SDN controllers. It presents a 
single point of failure and is difficult for scaling. In this particular implementation, no control plain 
exists on the network devices. The device is only a forwarding element (data plain) under the total 
control of the centralized controller. 
 
2. Control plain in depths 
         In general the control plain creates a local database which is used by the data plain to forward 
traffic between the ingress and egress ports of the device. The Routing Information Base (RIB) is the 
data base used to store the information about the network topology. It should be consistent and 
loop-free. Once the RIB is consistent and stable, it is used to create FIB based on the pre-defined 
algorithms. Routing protocols were introduced to manage the reachability information in terms of 
distribution and algorithmic generation of optimized paths. A few challenges have been faced, such 
as the increasing growth of the information base and how to be managed in the most optimal way; 
as well the size highly influences the bandwidth consumption of the control plain due to the DB 
synchronization.  The time needed for an address look up match in a large database is too long for 
the optimized network demands. To alleviate this particular problem, use of lookup algorithms take 
place (binary trie, compressed binary trie, Multibit tries) and combined with sub-networking can give 
the demand efficiency [28]. Dividing network in sub-routing areas can considerably reduce the 
information bases. After the initial database information exchange, additional data is only triggered 
in case of network state change. Such a behavior would reduce the consumed bandwidth by the 
control plain [1]. 
 
==PAGE:14==
9 
 
     If the control plain functionality of the layers in the data center network have to be specified, layer 
2 control plain is concerned with the hardware or physical layer addressing such as IEEE MAC 
addresses, while the layer 3 is build to facilitate network layer addressing  (the IP protocol). The 
purpose of the layer 2 control plain is to create an acyclic graph (STP). The limitation would come in 
terms of scalability, graph forwarding optimization and host mobility.  The user moving problem is 
due to the time needed for fully network convergence (meanwhile the entire network is forwarding 
unavailable) and unsuspension of the traffic flows. If the network is statically defined (bridges), 
adding new hosts or moving current is manually performed.  In the networking layer the control plain 
is focused on the IP address forwarding reachability (IP prefixes). Furthermore the layer 3 device is 
used for layer 2 domain isolation.   
It is worth mentioning MPLS, which is somewhere in the middle of the layer 2 and 3. It gives the 
possibility of greater traffic control with very flexible and complex path signaling. Some of the 
benefits of the MPLS control plain are: 
 
Provides faster address lookups of the label tables than the standard IP, due to the less 
entries.  
 
 Quality of service (QoS) - guaranteed traffic prioritization. Assigning properties on the data 
packets streams based on labels.  
 
Faster recovery from failure  
 
Better security. 
       The main aspects of the MPLS operation would be concern with label allocation, address binding 
and label distribution.  The label distribution control protocols have a session behavior and perform 
information exchange in order to provide the service. The actual protocols used for the label 
distributing are LDP or RSVP, which supports two types of label distribution –downstream on-
demand and downstream unsolicited. The allocation of the labels is dynamic, but can be manipulated 
by the vendor for higher control and clearness of the network documentation.  
 
     A MPLS network element is capable of the following actions over the labels except forwarding: 
push, pop, swap and variations between them.  By introducing the label concept, the forwarding 
behavior of the data plain switches from the longest destination IP address match to a short label 
match. As far as this, benefits in a less time demanding routing address look up; it brings more 
complexity by the requirement to maintain additional tables and the need of additional reference 
between the IP tables and the label tables. Nevertheless, the label technology provides a traffic 
engineering capabilities, which allows for better path control than the IP prefix –based forwarding 
[29].  
 
3. Data Plain 
        The main responsibilities of the data plain is to collect the data packets from the physical 
transmission environment (electrical, optical or wireless), make a consistency check and forward it 
base on the pre-created FIB by the control plain.  Such a sequence of actions is called the “the fast 
path” for packet processing, because further interaction with any kind of processes is not required. If 
the needed entry in the FIB is not found, a further request to the control plain is placed and based on 
its response the packet is forwarded or dropped. 
 
==PAGE:15==
10 
 
     Depends of the used implementation the data plain might be software or hardware defined. The 
hardware driven approach based on commodity forwarding chip offers low cost/ performance index 
($20/10Gport), but is lacking in forwarding flexibility.  In order to modify the forwarding algorithm 
the chip would need to be repined. The flexibility limitation pushed the packet forwarding 
implemented into the hardware to evolve, but unfortunately too slow for the constantly developing 
networks and their demand. The needed flexibility is achieved by purely software-based switches 
where they run on commodity operating system, on general-purpose CPU. The design introduces the 
required flexibility, since a new forwarding algorithm can be applied as simple as generate additional 
programming code, but the downside of such implementation is the poor port density and 
cost/performance index ($1000/10Gport) [2].  
   Typically the actions of the data plain look-up process are forward the packet (in case of multicast – 
replicate), drop, re-mark, count, and queue. Some of those functions can be combined and execute 
in particular sequence. Sometimes the traffic is destined for locally running process of the control 
plain (OSPF, EIGRP, BGP, RIP... etc ;). Such data is headed to the punt path, which is an internal 
communication channel that leads to the route processor of the device (just for a contrast the data 
plain traffic is headed through the hardware-forwarding path.  The punt path has generally low-
throughput; however some designs allow the adding of an extra path to the internal switching fabric.  
       Additionally to the forwarding decisions, the data plain can be capable of some small services 
known as forwarding features. Those include ACL and QoS/policy routing (based on TOS/COS).  Some 
implementations apply those features by using their own discrete tables, while others are using 
extension to the forwarding tables.  The forwarding operation order gives some feature operations 
exclusive priority over the others (Figure 1-1). Of course the order can be altered, according to the 
implementation [1].

4. The whole picture 
      Figure 1-2 describes the whole process of data – control-management plains interaction, in order 
to provide a full network transition of user data from point A (source) to point B (destination).  
 
                                                      Figure 1-2 Control and Data plain implementation 
       This figure applies for both, centralized and decentralized control plain cases. The centralized 
approach would have less information stored into the data base memory (need less space) and the 
need of less powerful CPU, because the functions for the network management and path 
computation are seized by the controller. 
     The scalability of the routing and switching systems can be analyzed from many perspectives; for 
example - forwarding performance (network throughput, delay, etc.), power consumption, 
convergence time (distributed control plain), but eventually all of those scaling points of 
consideration are revolving around a number of trade-offs between cost and performance. As 
mentioned before, the distributed control plain scalability is achieved by separating the routing 
domain in sub- domains, in that way the routing tables would be kept in normal size state and the 
network bandwidth occupied by the control plain (data base exchanges) will be maintained in 
efficient boundaries.  The other potentiality – the centralized control plain (logically centralized, 
physically distributed), is expanded by adding controller which have to take care of the entrusted 
instance. The communication and coordination between the controllers is achieved through east-
west interfaces and purposely defined protocols [1] [27].

        The development of the control and data plain from hardware and software perspective is 
tightly connected. Separating them apart would allow them to evolve and scale independently, which 
will lead to more optimal and faster development. Beside that reason, upgrading the control plain to 
accommodate scale is much easier and cheaper to be accomplished in a general purpose compute 
environment, while the data plain can be left to the purpose build forwarding hardware.  
       The cost is one of the main motivators for the separations of the plains. Of course, the price itself 
is determined by its primitives: scale, complexity, performance and stability.  To put the price on 
discussion, those primitives shouldn’t be compromised and set under the service providers 
requirements.  The cost of initial capital investment (CAPEX) to purchase a generic compute (COTS) is 
much cheaper in comparison to the cost of the processing in the network elements. The price is 
getting even higher, because most of the vendors equipment and software cannot be licensed 
separately. In that sense, SDN will definitely reduce the hardware integration price and will lead to 
re-pricing of the intellectual property.  
     Decoupling the control and data plain could benefit the innovation in either plain, for example: it 
would be possible to introduce a new hardware into the forwarding plain without the need to refer 
changes to the control plain, except applying new drivers or APIs.  Another benefit of the separation 
would be leaving the network element with a smaller and less complex code and data base. 
Generally the small code/data bases are more stable and less susceptible to security risk exposures.

Network Virtualization 
 
The concept of the virtualization brings the whole new era of possibilities to dynamic and 
rapid development of the data storage, OS deployment and networking optimization. Respectively 
the next part of the progress chain would be service evolutions.  The versatility and the optimization 
of the data centers in terms of compute, storage and networking gained through virtualization, wide 
opens the door for better service provisioning, which can be described as faster deployment, cheaper 
maintenance and more redundant structured for failure recoveries.   
 
1. Purpose and Features 
          Before the virtualization of the data center each machine from the rack has a dedicated 
purpose (like mail server, backup server, web server, storage server and etc.).  Each physical machine 
has its own dedicated operation system. Obviously a change of the purpose of one physical machine 
would be accompanied by the long downtime, tedious configuration and sometimes unpredicted 
complications from hardware or software perspective (memory, compute, connectivity, bandwidth 
unsufficiency, driver crushes, etc.). That leads to high maintenance and design cost.  Going further, 
the user applications and services are deployed on top of the OS of the dedicated server.  High 
performance application relied on services which reside on separate physical machines positioned on 
different locations. Moving a service or application would be concerned with planning and scheduling 
of actions which would be time demanding and high engineering resource consuming. In multivendor 
environment the challenge of the different physical and management interfaces, as well as the 
differentiation in the configuration process are highly significant, not to mention the great security 
issues which their coexisting can represent. The need of dynamically defined physical network 
configuration in the multivendor network environment requires a much higher and wider range of 
engineering skills.  The process of deployment is highly complicated and susceptible to human 
mistakes, not to mansion vendor features (proprietary) incompatibility issues.  Another challenge is 
the flexibility of the bandwidth allocation in the physical data center network. During the rush hours 
some services and applications would require more bandwidth in order to keep their performance. A 
wise and effective bandwidth management would be able to allocate the needed resource 
dynamically. From a network perspective, offloading the core and the distribution/aggregation layer 
by pushing the traffic on the edges of the network would boost the performance (delay, jitter, loss 
rate etc.) and increase the capacity [1][30].  
 
      Virtualization is an abstraction, which separates the logical capabilities from the physical 
hardware. The segregation of the network resources to physical and logical entities enables the fast 
development and innovation above and under the border which the virtualized abstraction 
determines.  That means from device perspective, to allow the vendors to produce hardware with 
different instruction sets and to emphasize on the optimization of different features (like higher 
performance, low power consumption, better graphics etc) as long as it is functionally compatible 
==PAGE:19==
14 
 
with the hardware abstraction layer. Such a physically defined flexibility gives the option of market 
competitiveness and widely opens the door for innovations provoked by the decoupling. The 
abstraction represents the hardware as a pool of resources, automatically and dynamically consumed 
by the user application entity. Expansion the pool is possible by simply adding commodity hardware 
and applies a controlling hypervisor on it. The possible types of resources virtualization are: compute 
(CPU), memory (RAM), storage and networking. By combining those virtualized primitives a server, a 
desktop, storage or a network modules of virtualization can be achieved. Roughly stated, the server 
virtualization converts the hardware devices (conceptually perceived as a whole inseparable unit, 
before the age of the virtualization – app, service/OS/, server) into software resource, removing the 
dependencies between the different service types and the virtual machines (Figure 2-1).  
The desktop virtualization gives the possibility to access and benefit from different OS on a single 
physical machine, without the need of full OS installation. The multi-device workspace is most useful 
for developers of services/apps for multi OS environments. Briefly the benefits of the storage 
virtualization are mainly aimed for disk waste reduction, achieved by taking the underlying physical 
logical units numbers (LUNs) and aggregate them in logical storage pools. Thus, introduces great 
hardware interoperability, which diminishes the cost for future expansion. Different schemas for 
storage are applied, for redundancy and security of the stored data [1][3][30].

          Considering the high success of the virtualization concept, it is not a surprise the effort to 
implement it in the network module of the data center design.  Here, one of the main beneficial 
features –the flexibility, is expressed in a way that allows the workload of the network traffic data to 
be evenly distributed among the elements of the data center without the need of re-design on 
hardware level. It is not only about balancing the load, but the option to place the load anywhere in 
the network independently of the hardware. Also the avail for the network administrator is the 
==PAGE:20==
15 
 
provisioning simplification and the less time needed for deployment. Basically all the 
implementations of the network virtualization follows the same main, initial concept -abstracts the 
physical network into a unified pool of network capacity which has the possibility to be segmented 
into logical networks used to interconnect different applications and services situated on virtual 
machines. That gives the opportunity to provide considerable network resource utilization; also 
logical networks can be reconfigured and scaled without any kind of interaction of the physical 
network. The hardware independency achieved by completely decoupling of the network 
components from the underlying physical network infrastructure, makes the connection of the 
applications much faster and effective. Those network resources needed for the network 
connectivity are requested from the standard pool of resources, and after there is no need of them, 
they are released back to the pool. By that high level of utilization and performance is achieved, as 
well the down time (for network deployment or upgrade) is shortened from weeks to hours and 
minutes. Separating the physical and the virtual network also favors in a security perspective - by 
using one of the major concepts of the virtualization of creating an entire new overlay pushed out of 
the physical network, the user privacy is guaranteed by the VLANs of the overlay (VxLAN) and the 
virtual switch policies, which are more flexible to imply due to the programmable nature of the 
virtualization [1] [30][32].  
 
2. Modules and elements of the Network Virtualization 
     The line which separates the physical from the logical network concept starts with the underlying 
hypervisor architecture.  The hypervisor is software which is directly installed on top of a physical 
server also called “bare metal installation”. The purpose of it is dynamically to partition the physical 
resources of the server (computational power (CPU), networking bandwidth and operational/storage 
memory) and thus allows multiple types of operation systems and virtualized elements to run 
parallel on the single hardware host.  The function of the hypervisor is to control the resources 
needed from each OS or virtual element and to make sure that they do not disrupt each other 
processes.  All the management functionality is possible through remote management tools. 
Depends of the hypervisor and the extended required functionality the install footprint on the server 
can vary between 2 MB (XenHypervisor) to 150 MB (VMware ESXi) [1][3].  
       For every defined Virtual Machine (VM) on the physical server at least one Virtual Network 
Interface Card (vNIC) has to be created. A vNIC as the name suggest is virtual interface card which 
allows the virtual machine to exchange data with other virtual or physical hosts on the private or the 
global network. It is an abstraction, which supports all the network attributes and functionality of the 
physical network adapter and can be used as a re-usable virtualized template. The OS is 
communication with the vNIC through commonly available device drivers or specialized drivers 
developed for the particular virtualized platform, dedicated for optimal performance in the virtual 
environment. In both cases the communication appears the same way as it is in a physical device. 
Installing the drivers to a vNIC is less time demanding and less complicated due to the automatization 
of the application process. The virtual platform has a set of pre-defined vNICs which optimized 
drivers are two clicks away from full installation. In that case the main job of the network 
administrator is to focus on the service deploying, instead loosing time for technical details.  The vNIC 
is granted with a MAC address and one or more IP addresses. It has the total communication 
==PAGE:21==
16 
 
compatibility with the standard Ethernet protocol, like the standard physical network adapter. The 
hosts communicating with the VM cannot detect if they interact with virtualized environment or not 
[3]. In order a VM to have a network connection (private or public) it has to be connected (the vNIC 
of the VM) to a Virtual Switch (VSw). Generally the VSw is a model of PSw (physical switch) and 
connects virtual to physical devices. The VSw like the physical one has uplink and downlink ports. The 
VSws can be elements to construct private networks with or without external global access (with or 
without vGateways); as well can be used to establish point-to-point connectivity.  By using the same 
virtualization platform (the homogeneous environment gives the possibility to avail not only of the 
standard but also of the extra features provided by the particular vendor), the networks can be 
spread across different global locations (Data Centers) and at the same time to keep the full 
functionality of the provided features [30]. 
 
         The VSw is implemented in the software and resides on the physical server controlled by the 
hypervisor. The VSw is directly connected to the physical LAN card of the server. The pNIC is the first 
point, where the physical and the virtual network meet (Figure 2-2).  The VSw have the same 
operability functions like the physical ones (can forward L2 frames and separate traffic into VLANs 
(support 802.1q encapsulation)), but provides higher flexibility and dynamics from point of 
configuration and network integration.  Reconfiguring, updating and managing the switches are 
much simpler in the software domain in terms of flexibility and automatization. The VSw can be 
divided into two logical sections (in terms of plains): data and management. The data plane is 
concerned with the packet tagging, switching, filtering and header altering, while the management 
plane is the control structure used to configure and give operational sense of the data plane. The 
VSw can perform filtering (basic firewall functionality) and virtual VPN the same way as the physical 
switches are doing so [30].  
 
                                                                          Figure 2-2 Virtual Elements Interconnection 
 
       One main feature of the VSw is the port groups. It allows logically to group virtual machines 
together and to apply a range of policies. The port group is identified by a label, which is unique to 
the current host. Those groups can be assigned to VLANs, configured to participate in load balancing 
or restricted with specific personal and security policies. From traffic routing point of view the load 
balancing means to transmit the data on two or more preferably disjoint paths in order  to achieve a 
higher bitrate (normally the routing protocols (OSPF, RIP, IS-IS..) allow it when the routes are with 
==PAGE:22==

the same metric, but some proprietary protocols permit it despite the metric through an explicit 
configuration (EIGRP)) . In case of the virtualization, spreading the traffic (load balancing) into two or 
more vNICs gives higher portion of the pNIC bandwidth to the VM. Normally, the physical servers 
have more than one pNIC (which dedicates higher bandwidth and redundancy to the server), so by 
load balancing the traffic a huge bandwidth dedication can be provided to a single VM. That gives a 
new level of bandwidth flexibility achieved through virtualization. Nevertheless, the virtualization is 
working directly with the protocol stack, thus allows full interoperability of the network hardware in 
the hosts with dual homing features.  Other useful feature of the VSw is the traffic shaping (QoS), 
which allows the prioritization of the traffic on the VSw uplink ports, but cannot be guaranteed on 
the physical network (Figure 2-3)[1][30].

One interesting piece of software (module), which resides on the controller side, is destined to allow 
the convenience of consistent configuration among the switches in the network. The network virtual 
equipment must be in correct port configuration, namespace, IP subnetwork and many other initial 
configuration details before taking advantage of its services. So called the Virtual Distributed switch 
(VDSw) allows the simultaneous configuration of all VSw on the network. Every physical server has a 
local proxy switch that contains the network settings for the server. Those settings are synchronized 
or initially configured by those on the VDSw (for the small networks, which do not need controller or 
such synchronization module are configured by a manual client module).  Once it is configured, the 
data configuration is applied to all switches in the network. In such a way the port groups, the VLAN’s 
and the policies can be kept in consistent state all over the overlay.  The VDSw contains the same 
functionality like a normal VSw, supports traffic forwarding, filtering, port groups etc. Such 
functionality provides flexibility, time efficiency, accuracy and diminishes the factor of human 
mistakes. Any following configuration changes to the VDSw are applied to the subscribed VSw.

3. Virtual Network Architecture 
The main goal of the virtualization is to offer a programmable and mobile network as a service 
deployed on a non-vendor specific hardware.  That is achieved through the virtualization of 
forwarding elements, gateways, and other service elements like firewalls, load balancers and 
specially developed software modules to provide VPN, QoS, monitoring and security.

to communicate with the different virtualized modules by REST APIs (abstracts the underlying

aware of the underlying physical network.  The inter module communication (switching, routing,

4. Perspectives and Current State 
      The line where the physical and the virtual network face each other is in between the virtualized 
forwarding element and the physical access layer switch. The direct VLAN mapping of the 
physical/virtual network is done by the hypervisor. The joint solution provides seamless physical-to-
virtual connectivity by unifying the management plane. The VLANs on the physical network should be 
administratively predefined before the hypervisor can do the map. In order to introduce much higher 
level of flexibility and efficiency at the border, where the two networks (virtual and the physical) 
meet, two companies (VMware and Juniper) are working on a common project to present futuristic 
solution. For the purpose a special access layer switches are developed (also are in process of further 
development), which have an extra functional module. That module allows the administrator of the 
physical network automatically to discover, manage and monitor the entire virtual network and 
make sure there is a consistency in between. For the purpose, each access layer switch has to be 
registered with the controller of the virtualized environment. In this way the controller can 
coordinate the creation and mapping between the physical/virtual network VLANs. The process of 
future development is concerned with creating the availability of programmable core/aggregation 
layer Ethernet switches. If all the physical forwarding elements in the network have such functional 
module, the VxLAN overlay can be deployed on the physical network. This would allow a direct 
mapping of the VxLAN to the data center physical network and gives the benefits of the overlay 
(programmability, flexibility, efficiency and VLAN capacity). The result of such approach would 
provide a single pane-of-glass management for programming and configuring virtual to physical 
network connectivity [6] [7] (for a higher level of automatization additionally are used Junos 
scripting, Python scripts, and integration with Pupet and Chef ( those represent configuration 
management systems that allows automatically to enforce pre-defined state of the current IT 
infrastructure) ). 
        The virtualized data center gives the possibility dynamically to adjust the network capacity in 
order to face the rush hour traffic. Another way to alleviate the traffic is to move the VMs and the 
application resident on them, closer to the data server (to which the application is related and 
interacting with it) or to the network gateway if the stream is requested from outside the network. 
The approach permits dynamically workload distribution in the network, without the need of any 
additional configuration of the physical network. The live migrating of the VM to another server has 
an automated process line, which includes porting the network settings (labels, port groups, VLANS, 
policies etc.), the state and the storage data. During the moving process, the VM down time will be 
negligible (from less than a second to maximum couple of seconds, because such network traffic is 
tagged with the highest priority). With no downtime the data center customers can have services 
with higher availability, which will contribute to the business strategy (Customer Value Proposition 
model – Differentiation (of the service-higher availability)) of the service provider.  
 
          Moving a VM around the network is a feature which is acquired manually by the administrator. 
When it is noticed (from the network monitoring module) that some workload traffic can be 
offloaded by moving VM closer to its interaction point, the action has to be explicitly declared by a 
network administrator. The achieved efficiency is questionable and unoptimized due to the fact that 
it is based on the human judgment and the unpredictable nature of the network traffic. For future 
==PAGE:27==
22 
 
development the process can be automated based on algorithmic calculations. It can evaluate the 
traffic generated for the migration and the data exchange needed by the VM (VM-data farm server, 
VM-VM, VM-gateway… etc.) and the distance. Also it can be base on many contemporary 
considerations like the green routing protocols. VMs can be migrated between different data 
centers, so an application can be placed in data center where the route to it is composited by 
network infrastructure powered by green energy. Other example would be optimizing the OPEX. If 
the data center is not on full capacity the VMs can be organized and moved on range of servers, 
guaranteed of their uninterrupted working process. The rest of the servers and the network 
infrastructure interconnecting them can be put in hibernate/standby mode and brought back when 
they are needed. 
 
   The VM movement functionality can be useful if low latency and jitter are required for the 
connection. Moving the applications close as possible would fulfill such a demand. Latency and 
bandwidth degradation can be a result of general network traffic overload or segment bottlenecks. In 
both cases the flexibility of moving the VMs will introduce the desired solution by offloading the 
traffic from the core and the distribution (aggregation) layer of the physical network architecture and 
confines it in the less loaded access layer. 
    The virtualized forwarding elements can be used to forward traffic in case of a network failure

a back up path has to be pre-defined (protection technique) (for the current state of the provided 
equipment), dynamical creation (restoration technique) in case of failure it is still not possible. 
Logically, close to achieve it is the aforementioned common project of VMware and Juniper. If all the 
physical and virtual elements are registered with a common controller, it will have a full overview of 
all the network devices and will be able to react and recalculate the possible back up routes if a 
network fail appears in a real time.

SDN-Open flow 
 
The initial motivation for SDN development was to allow the testing of experimental 
protocols and network concepts on already existing network infrastructures (campus networks) [8]. 
The initial view of OpenFlow was to replace completely the control capabilities of layer 2 and 3 of the 
commercial switches and routers.  This would give the desired flexibility to control the network in 
higher level of precision and grant exclusive traffic isolation to test new ideas without disrupting the 
current network traffic. Considering the fact that the innovation and development of the data center 
network is slowed and disrupted due to so well adopted and widespread protocols, accompanied by 
the supported equipment (specially dedicated for the current implementation). Experimenting with 
new solutions would require an especially dedicated network, if tested on the current available 
infrastructure it would endanger the production traffic.  For such a reason most new ideas and 
networking solutions go untested, thus leading the future of development in a sluggish pace. By 
following the solving chain of this challenge the networking community developed programmable 
networks which unlock the experimental research process of new network architectures. This 
involved the development of programmable physical network elements (routers and switches) 
capable of perceiving dynamical management instructions followed by real time process executions, 
allowing independent, programmable, and isolated network solutions to co-exist on one physical 
network and unleash a new era of network control and dynamicity. Defining network 
programmability would give acceleration for new ideas, growth and innovation of the network 
infrastructure, but unfortunately it comes with costly facilities investment. 
     The SDN concept is widely defined by every external change (with centralized characteristics) of 
the forwarding elements’ FIB, no matter if it is done by controller side, automated script or a 
dedicate service provider application.  The SDN architecture opens the dynamics of the network and 
creates manageable, adaptable and cost-effective environment and satisfies the modern application 
requirements in terms of flexibility, lower latency and high bandwidth requirements (achieved by 
optimization). The OpenFlow has become a common definition of SDN, due to the provided 
functional coverage of the SDN concept and as well it has been adopted under open source license 
for distribution and development.

      A flow entry is identified by the match field criteria and its priority. The actions of the table logic 
can be generalized in three directions – forward the flow, drop it or to send a packet (from the flow) 
to the controller (this is one of the configurable reactions in case of table-miss flow entry, the other 
two options are to drop it or to table pass it) and wait for further instructions. A match can pass the 
data flow to another flow table. The processing pipeline can go only in one direction (forward, not 
vise versa), which means a table can redirect a flow to other table with numeric value higher than 
itself.   
     The network design, concerning the forward paradigm can be approached in two different ways –
proactive (the network forwarding state is pretty much predefined) or reactive (so called the data 
plane driven; the flow decisions are taken in real time). The same two concepts of behavior find 
application in different logical challenges for example building network resilience (protection and 
restoration techniques). The proactive mode is the desirable case of implementation but requires the 
knowledge of the flows and their network demanding, also it is much effective if the flow-time 
existence is considerably high.  In the other method, every flow is sent to the controller for further 
processing decision, which would contribute with considerable delay due to queue formations, 
especially during the high intensive network time periods.  
 
      Another available table, extending the functionality and the engineering value of the protocol is 
the Meter table. It consists of entries defining metrics per flow. Based on those per –flow metrics the 
OpenFlow can provide simple QoS operations (like rate limiting), if combined with per-port queues it 
can achieve more complex schemas of prioritization like DiffServ.  All the packets are assigned to a 
metric which measures and controls the rate of the packet flow.  The OpenFlow QoS capabilities 
highly excel those of the distributed control based networks. It could be considered as a very 
successful solution for Policy Based Routing. 
 
       The OpenFlow match process can be used only to forward flows based on their IP packet header 
(masking capability) and by that can emulate IP based routing. The same is true for the Ethernet 
network, but more challenging is the idea of using the OpenFlow architecture by specifying a 
forwarding element to act like a service appliance (behave like NAT or Firewall). This is achieved in 
the software domain, by an application running above the controller. The forward table is 
programmed base on the logic of the specially developed application, giving its instructions to the 
controller vie APIs.  Such implementation does not have defined standard roots and it is a product of 
vendor specific considerations. The wired protocol is capable of label manipulation actions, thus 
could emulate MPLS LSR platform behavior.  
 
       The controller and the switch communicating through OpenFlow channel, which represents a 
connecting interface. It is used by the controller to modify the state, get statistical information and 
receive events from the switch. All the data messages on the channel must be considered with the 
OpenFlow protocol requirements. The information which the controller may demand from the 
forwarding element is the switch identity and basic capabilities before considering the flow entries. 
The channel is used by the switch as well in order to replay to a controller’s request or to send 
asynchronies message, notifying for a new packet arrival (no available actions in the flow table) or a 
change in the switch state. The importance of the data on the control channel implies reliable 
message delivery established by TCP sessions and optionally secured by TLS (default on TCP port 
6653). The channel might be settled by multiple TCP connections in order to take advantage of 
parallelism. Additional integrity check is made during the switch massage validation stage. Every 
message from the controller has to be verified by the switch in terms that all the features are valid

     Typically, one OpenFlow controller has to maintain multiple OpenFlow channels, each for every 
controlled forwarding element. For reliability purposes, every switch might have more than one 
control channel, each for every controller in the network. All the control elements dedicated to one 
network maintain intercommunication processes via API interfaces (so called the east-west interface; 
if the controllers belong to a singular vendor they could communicate via vendor specific OpenFlow 
messages). By those interfaces they coordinate and manage the controlled switches among 
themselves. In case of failure of a control element the network architecture can react with fast 
recovery due to the inter-controller synchronization. Also the controllers can be utilized to perform 
load-balancing and thus equally to distribute the control plain burden of the network.  In order to 
serve the both cases, the control elements are organized in a hierarchy. For a load balancing purpose 
some of the controllers can be defined with equal rolls (OFPCR_ROLE_EQUAL message), while the 
others dedicated for fail recovery scenarios are configured as slaves (OFPCR_ROLE_SLAVE message; 
the slaves have read-only rights over the switches). Officially the OpenFlow channel must be initiated 
by the switch (for security purposes), but there might be vise versa implementations. During the 
connection the switch must send a message (OFPT_HELLO), announcing the highest OpenFlow 
supported version and other information elements required for establishing the connection setup 
(connection negotiation). Considering the received information the controller takes the decision 
which version of the protocol to use for communication with particular forwarding element. 
      The OpenFlow ports can be considered as interfaces which purpose is to connect the OpenFlow 
processing logic and the rest of the network. It is not mandatory all the physical ports of a switch to 
be dedicated as OpenFlow ports, some of them might be assigned to an old fashioned distributed 
control plane network, more trusted to take care of the production traffic due to its technological 
maturity (including commodity/off-the-shelf available equipment). OpenFlow switch defines tree 
types of ports- physical, logical and reserved. The logical ports correspond to a higher level of 
abstraction, and do not need to match the physical ones. They can be used to generalize link 
aggregations, tunnels or loopback interfaces and thanks to the logical port ID additional metadata 
can be associated with it (like Tunnel-ID). The reserved ports specify general forwarding behavior like 
flooding (used with port grouping to achieve multicasting) or sending packets to the controller for 
further investigation ( dedicated for the control channel) and etc [1][8][33][36].  
 
3. The Configuration protocol 
           The main idea around the of-config is to be an implementation of configuration and 
management protocol (in addition to OpenFlow) designed for remotely setting up OpenFlow network 
elements (physical or virtual). It implements a further level of abstraction of the OpenFlow switch by 
defining the OpenFlow logical switch. The job of the protocol is to configure various essential logical 
switch artifacts like managing the controller’s IP addresses (assign a controller to the forwarding 
element; remember that a switch is responsible for the connection establishment), remotely change 
the port states (port assigning, enabling/disabling), providing the establishment of the control 
communication process in-between the controller and the forwarding logical element (ensures 
certificates for secure control data exchange; also responsible for capability discovery of an 
OpenFlow forwarding element).   The fundamental structure of the protocol is encoded in XML (the 
data model is structured into classes and attributes of those classes), Yang data model and NETCOMF 
protocol for delivery [1].  
 
==PAGE:34==

      The of-config protocol could achieve a multiple virtual forwarding element abstraction into one 
physical switch by separating a set of OpenFlow dedicated resources like ports and queues between 
the pre-defined OpenFlow virtual switches (the separation here is more like conceptual classification 
for explicit isolation purposes, not a resource virtualization where the underneath hypervisor is 
taking care of the physical resource management, nor the functional proxy implementation done by 
the FlowVisor).  From architectural point of view, the service entity which sends the of-config 
messages (configuration messages) to the OpenFlow capable switch is approached with the 
definition of OpenFlow Configuration Point. There is no specifically indicated way of the element 
implementation. It can be done like a software module residing on the OpenFlow controller’s side or 
defined on a separated administrative entity on a separated physical machine. Official specification 
about the direct interaction/communication between the of-config v 1.2 and OpenFlow protocol 
(controller) are not available, although there a statement for probabilistic future development in that 
direction (possibly, will be available within the specifications of of-config v1.3). It is good to be noted 
that some OpenFlow switch functionalities can be configure via the both elements (the controller 
and the configuration point); nonetheless they have separately dedicated control channels by reason 
of their significant difference (although there is a proposition by the OpenFlow architecture group for 
wire/config protocols merging). The current architectural interrelation of the two protocols can be 
seen on (Figure 3-4).

4. OpenFlow controller 
         The OpenFlow protocol although is considerably following the exponentially rising evolution 
curve during the past few years and managed to close many questionable recommendation into 
functional specifications, still has a long way to prove (develop) itself with the functional stability, 
failure management predictability (OAM)  and secure responsibility. The right way to achieve it is by 
the conduct of researches in many academic and research facilities concerning the standardization 
and development of the open sourced protocol. Up to the current officially documented version of 
the OpenFlow (v1.4.0), there is not a proposed standard for either the northbound (application 
facing interface) or the east/westbound (inter-controller communication) API interface (of course the 
controller to element agent communication-the southbound interface is fully documented and is 
already afore-explained). There are available east/west distribution mechanisms based on the 
database distribution model (comparing differently located databases (data files in terms of 
forwarding tables) between two or more elements, method used in the distributed control plane 
networks (RIP table exchange, designates a change into the routing graph)). Unfortunately such a 
==PAGE:36==
31 
 
solution is available for a single vendor’s homogeneous network and does not support state 
exchange among different branded devices. The lack of the north, east/west interfaces 
standardization considerably slowdowns the protocol’s adoption rate and throwing a shadow over 
the initial motivation purpose of the OpenFlow project, respectively it openness (non-
proprietariness)[1]. 
 
    The main core element of the architecture is of course the OpenFlow controller (Figure 3-6). It 
provides the main set of services to the top layer of the architecture (application), like path 
computation, topology discovery (achieved by LLDP, which is non-proprietary link layer protocol, 
allowing the network elements to exchange identity and capability information; the topology 
discovery is limited to layer 2 possibility) and network provisioning. It can be implemented on any 
physical server with adapter for network connectivity. Upon receiving a request from an application 
via the APIs, the controller dynamically can reshape the whole network, from adding/removing 
particular flow to total network re-routing and policy driven flow behavior. The communication 
bridge between the controller and the application is devoted to vendor consideration, bringing the 
heterogeneous OpenFlow network (in terms of hardware specification) extremely complicated to 
manage or nearly impossible. That brought the OpenStack project to the world, aiming to provide 
standard set of APIs, ensures not only  the communication between the application and the 
OpenFlow controller but as well the most market available open or vendor specific controllers. 
Initially the project was about defining a cloud platform converting the storage and the computing 
power into customer service and subsequently evolved, spreading the cloud over the network and 
connecting it with the emerging SDN and virtualization concepts. If the view is narrowed only from 
network perspective, the OpenStack APIs are enabling the future OpenFlow data center 
independency of particularly uniform equipment. Further and more detailed OpenFlow controller 
architecture analysis is reflected in Chapter 4 [1][25][33].

5. Used Cases   
 
         The first example would be related with the initial idea for emerging of the OpenFlow – creating 
environment for experimental purposes. The protocol is able to slice the network and to provide 
complete isolation between those segments (OpenFlow can be viewed as transcendent 
generalization of the VLANs with major level of control management). Such a isolation allows the 
creation of many derivative networks from the physical graph and be used for testing purposes (new 
routing protocols, management tools, scientific theories, and new technologies in general) without to 
disrupt outside of the given boundaries. If OpenFlow is used in the campus networks, it would 
permits the use of the common network infrastructure for experiments (eliminates the need for 
creating special testbeds) and at the same time would not interfere with the network production 
traffic.  The flexibility of the protocol allows these experimental networks to be consumed during the 
traffic peak periods in favor of the networks production availability and to be restored after. 
      A major functional application of OpenFlow is to deploy powerful policy enforcement and access 
control [9]. It can be achieved in the software by an especially dedicated application architecturally 
positioned above the controller or by specifically designed implementation of a controller itself. The 
policies have to be predefined into the central unit, which is configured to make admission control 
decisions for every new flow. The level of controller’s function extensibility is so high that it can takes 
care of managing the bindings between names to addresses (addresses to address as well) and 
respectively can be used as DNS, DHCP or authentication point. A reasonable question here would 
be, by integrating all those network services directly in to the network management process (instead 
of separate logical/modal or physical elements) is it going to degrade the main and basic network 
functional responsibilities ? The network reliability is one of the most important network 
requirements, which cannot be compromised in the name of functional expansion. Answering that 
challenge would be substantial for the further sense of development in that direction.  
     The OpenFlow is not strictly bounded to the IP. It is capable to use its header primitives for 
routing/filtering purposes but it is no mandatory. As far as OpenFlow primitives (Figure 2-5) can be 
used as a match in the flow table, the forwarding process is totally available (for example if the 
protocol is encapsulated into an Ethernet frame it can be forwarded based on its MAC address). That 
feature allows inter-protocol independency at some extent and verifies OpenFlow as the closest 
protocol definition of SDN [33].   
        The network services as always been an important part of the network can greatly benefit (and 
thus the users) from the network automatization. Now, with the hugely possible flow control, opens 
the door for fully automatization service subscription. If a network user, decide to subscribe to a 
firewall/antivirus service offered by the service provider, there is no need of manual interaction by 
the network engineer. Right after the subscription an application via APIs would signalize the 
controller to direct the flow (of the subscriber) to pass through the service element (firewall, 
antivirus, antispyware, load balancer and etc) (Figure 3-7). The flow is not limited to only one service 
element, but it might pass by chain of devices (L4 to L7 services). Higher granularity in terms of time 
leasing can be offered by the SP due to the lack of manually control labor, and the possibility of the 
user to subscribe and unsubscribe by himself (services per tenant bases). By using the same 
==PAGE:38==
33 
 
mechanism for flow redirection to a service element, many new network services might to emerge. 
For example, a new service like automatic protocol format conversion is totally feasible. An 
OpenFlow controller is perfectly capable to detect a new network flow protocol type or version 
(network layer- IPv4/6, IPX, AppleTalk etc) and if needed to direct it to a special service device 
(NetFPGA-based programmable router), which would convert the headers of the flow to a desired 
outcome. That particular feature grants the OpenFlow with the ability to interconnect differently 
layer 3 implemented networks.

     The OpenFlow unarguably is the future of the data center networks or at least has a major part of 
its characteristics and functional ideals, but before the complete adoption of the OpenFlow, there 
would be a time when the transition will force the past and future (distribution and centralized 
control plane approaches) to coexist on the same network. For that purpose a special group has been 
formed (Migration Working Group) to help the smooth protocol adoption, but mainly to emphasize 
on a hybrid network solution. The outcome of those efforts came up with a general solution to bring 
together the IP routings stacks and OpenFlow with the help of virtualization. The proposed concept is 
to use a hypervisor/ virtual switch (on a physical host) and bind its ports with those of the OpenFlow 
switch and the distributed control plane network element. The virtualized forwarding element (Open 
vSwitch) appears like a bordering device between the two network implementation. From the 
distributed control network (native network) side it forms an IGP (or BGP) neighboring relation with 
the connected forwarding elements and advertises all the registered OpenFlow prefixes (accessible 
networks and subnetworks through the protocol data flows) to the native network (boundary 
switches) by following the IGP specific route-distributed message formats. For the whole 
decentralized control pane network the routes appears as learned from adjacent peer. On the other 
side the virtual element converts (base on internal logic and policies) the data streams incoming from 
the native network (derived from the IGP routing protocols) and inject them into the controller’s flow 
calculation process. In his turn, the controller configures the ports of the boundary connected       
==PAGE:39==
34 
 
 
 OpenFlow switches to face the incoming data flows from the native network. Another way of 
implementation is to leave the controller to take care of the decisions for the newly incoming data 
flows (at the border of the two control plane frameworks) by itself but the delay for the flow 
establishment would be longer. By receiving the prefixes form the virtual switch, the flow can be 
initialized immediately after the route is advertized in the native network, while in the other case the 
flow would be created right after a initial data packet is received (on a border element), analyzed and 
finally then settled up by the controller. The hybrid network architecture is modular and it is based 
on three pieces of software (Figure 3-8).  For every hybrid network, there is at least one or more 
OpenFlow controllers, and each controller must run a simple application agent called rout flow 
proxy. The purpose of the agent is to extract the flow entries from the controller and pass them to 
the Rout Flow Server (RFS) via APIs. The core module of the architecture is the RFS - a standalone 
application responsible for a registering and mapping OpenFlow devices to the routing protocol 
stack. In other words, it translates routing information to OpenFlow entries. That information is used 
by the Route Flow Client (RFCL) (a small user space daemon based agent) run on the virtualized 
switch (Linux machine) along with their routing protocols. The communication between those two 
modules is achieved by Jason RPC and information storage pool based on SQL database that also 
stores the state of the core system. The RFCL is used to configure the routing information needed by 
the virtual switch to advertize the routes in the OpenFlow and the native domains [11][38]. 
      This particular approach for enabling hybrid network deployment, logically approved brings up a 
lot of questionable details for consideration. The virtualized element used to merge the both 
network entities represents a single point of failure to the network.  The secure provisioning from 
breaching that essential network point is more than critical, due to the major control role which 
takes. There is still no official development in that direction. No commercial network administrator 
would allow such a weak spot in the mature and well securely defined native network framework.       
    Another thing for consideration is the OpenFlow port rolls (gives additional traffic flow 
characteristics) and their representation in the native network. There is still no official standard 
proposition for interface representation about the native network routing information to the open 
flow primitives. Considering the current phase of development, the hybrid network deployment 
outside the scope of the testing purposes would be highly unrecommendable [11].

       After the released implementation of the SDN concept in the face of OpenFlow protocol, many 
other extensions emerged, in addition to cultivate it functional paradigm by applying a higher level of 
abstraction. Such an extra functional logic is the FlowVisor and ensures logical multi network 
separation with preservation of the distinct forwarding logic behavior on the same physical network. 
FlowVisor works as a transparent layer between the switches and multiple controllers. So what is the 
difference between FlowVisor and OpenFlow by itself? Both of them performs logical network 
separation and enables isolation. The substation difference is that FlowVisor isolates the network to 
a next level of abstraction which means to separate the OpenFlow network on different OpenFlow 
sub-domains. Thereby allowing each sub-domain to be controlled by separate controller (control 
element per network slice) and prevent the interference in between, while in the OpenFlow of 
course, can maintain more than one controller as well but they unitedly manage the whole network 
and can influence on each other’s working process. Architecturally the FlowVisor is situated on top of 
the OpenFlow switch and represents an intermediate layer in between the forwarding elements and

the controllers (fig 3-10) [10].  It intercepts all the control messages from and to the control plane

it, makes a policy check and redirects it to the controller responsible for the packet.  In the opposite 
direction, when the controller sends the control rule back to the switch, the FlowVisor module do 
another policy check to make sure the control plane element is authorized to imply the rule. To be 
achieved a complete isolation, the flow table in each switch is partitioned and each segment is 
assigned to a controller responsible for a network slice. The isolation concept is broken when one 
slice is able to consume the resources of another.

be sent to the controller, FlowVisor is not so effective (in an aspect of latency). The reason for that is 
the additional delay as a result of the FlowVisor proxy based functionality. But never the less the 
architectural OpenFlow extension is a step forward to the future datacenter network functional 
capabilities [10][34].

infrastructure.  The shortcomings of the protocol are mainly related with the lack of standard 
northbound and east-west interface definition, which enables vendor controller interoperability and 
application mobility. The OpenFlow primitives give a great deal of flow control and can satisfy every

1. General Overview    
       A SDN controller represents a softwarely specified system which can be viewed as a modularly 
defined entity, and each segment express different functional meaning.  The main responsibility of 
the controller is to manage the network state and occasionally (if there is a cluster of controllers) to 
distribute that state (that state may be dynamically calculated or launched from predetermined 
database source). An external database server or integrated module with such functionality can be 
used to store information derived from the network control elements or from the interacting 
through APIs with the controller applications. The stored data may include: the network state, 
different ephemeral configurations, topology info, control session information and statistical reports 
(logs concerning the network behavior, state transitions, reaction to failures and etc). In less 
demanding cases in-memory solutions may be employed. All the functional modules within the 
controller must interact with each other base on some principles and declared rules. For the 
purpose, a high level data model is enforced by using Yang modeling language for NETCONF (IETF 
network management protocol). Together they form a tool capable to bring the automated 
configuration principles across heterogeneous network elements of the SDN network. The YANG 
modules define a hierarchy of data, which purpose is to trigger NETCONF-base operations [12].  
Most of the cases the interaction between the applications and the controller takes place over 
RESTful APIs (a stateless architecture that runs over HTTP). This interfacing element describes all the 
available services and feature to the overlying applications. In some implementations the controller 
and the northbound interface are part of the same development environment that means the 
generated API’s might be function or object calls based on the used programming language (no 
interface abstraction). The currently vendor specific network controllers available on the market are 
VSphere (VMware), Nicira (NVP), NEC (Trema) , Big Switch Networks (Floodlight/BNC), 
Juniper/Contrail and Cisco. Some of those vendors offer open source products in parallel with their 
commercial solutions, but of course with less supported features. They have all initially defined 
proprietary northbound interfaces but the need of interoperability forced a second wave of 
controllers to emerge bounded to support open source API’s like OpenStack and CloudStack.  The 
driving force of that particular subsequent wave of development is the expansion of the SDN 
applications and moving them out of the data center, which built the idea of multi controlling app, 
capable of management domination over different production entities, leading to the birth of the 
open source API services.  
 
      The expansion of the datacenter also brought new set of elements to the network’s paradigm in 
the face of the hypervisor based forwarding entities (switch, router, and service elements). The 
service elements embody the solid building blocks of the Network Service Virtualization (also 
referred as Network Function Virtualization) and intend to enrich the next generation datacenter 
architecture (the roll of the controller is highly enhanced in such future SDN framework). The 
hypervisor based forwarding construct is not capable to serve as much forwarding entries as the 
purpose build network hardware .  
 
    The services of the SDN controller spread over different functional instances from network 
provisioning and topology discovery to collecting analytical information and event notification aiming 
to provide full network manageability. In general, the controller is the element which is giving the 
evolutional image used to shape next generation datacenter networks [1].

2. VMware 
      The VMware is not providing only a controller implementation but the whole data center 
orchestration package solution. The company offers the entire cloud suite from the hypervisor 
(control environment for the VMs, virtual switches, vRouters and virtualized service elements) to the 
management cloud platform. The ESXi (VMware hypervisor) is very small in size, directly installs on 
“bare metal” and provides operation system independence, which makes it to appear like a  light 
layer (in terms of occupied capacity) which interconnects the instance of a operation system with the 
hardware resource. It receives configuration from VSphere or vCenter, which both represents 
controller solutions (but serves different functional packages; vCenter can controls function 
virtualized elements like firewalls, load balancers and etc). They manage everything in the VM 
network what is concerned as IaaS components (all the virtual network and service elements). The 
hypervisor and the vCenter (the VMware controller) communicate via internal APIs, besides them the 
hypervisor can be configured manually by CLI. The products provided by VMware are entirely 
dedicated in the virtualization concept widely covered in chapter 2 “Network Virtualization”. Shortly 
summarized, the network mainly exists on a created overlay, where the network devices and the 
attached hosts exist on physical server elements with the installed hypervisor on them. The physical 
network is used to provide the resource bandwidth by pre-established bridge connection or VLAN 
tunnels, if the virtualization is deployed on layer 2 on top of layer 3 overlay (specific advanced 
feature offered as a functional adjunct in the higher level cloud solution products) the routing 
through the physical network is performed by the IP level routing protocols (connecting the VTEP 
points in order to create the logically connected network despite the physically separated elements). 
The vCenter server is the actual management/control point for all the network virtual devises, 
leaving to them the forwarding data functionality (the control plain). Once the entire network is 
provisioned and managed the network will continue to function even if the vCenter server fails or is 
disconnected. The cluster communication is defined by east-west interfaces and in case of failure a 
controller from the cluster can take the functional responsibilities of “his fallen brother”. In highly 
populated datacenters, the network is often segmented into regions and for every region a controller 
is assigned. Such a separation introduces a better performance of the network in general, due to the 
distribution of the control plane load over several management units. Sometime network 
segmentation is required due to the limited number of controlled virtual devises by a single 
controller (v5.0 limit – 32 units; v5.1 limit– 128 units [13]).The elements in the control cluster are 
monitoring each other’s health by heartbeat messages. For resource data storage they use SQL 
Databases base on Microsoft of Oracle solutions.   
 
     The virtual switches, as already mentioned support VxLAN tunnel overlays (Chapter 3), enable the 
creation of isolated tenant networks. The communication in-between the virtualized forwarding 
elements and the controllers (southbound interfaces) are taking place through vendor specific APIs, 
but for interoperability purposes there are availed vendor-consumable APIs, which permits third 
party network devises (switches, routers, service elements and etc; or more in generally put – third 
party hypervisors) to be controlled by the VMware controller, by translating/mapping the vCenter 
parameterized APIs instructions to the other vendor specific or open source device configuration 
triggers. The VMware controller’s core is written on Java developing platform, more specifically by 
using the Spring Framework, which provides higher productivity, eases the programming process and 
creates environment that enables portability across different application server implementations.     
Considering the nature of the virtualization, such platform choice of flexibility and portability is 
==PAGE:46==

totally reasonable. One of the key elements of the framework is the dependency injection 
capabilities, a mechanism intended to simplify Java development and thus improving the cache 
management and database related services. The framework also maintains the creation of RESTful

framework, allowing the VMware to have fully Java based interfaces between the controller and the 
applications dedicated to serve some network management/orchestration purposes.   
      Despite all the great provided features which bring the model close to the idealized controller

be used at full potential of the packet transmission [1]{13)[30] [32].

5. Trema

8. Path Computational Element  
      The Path Computational Element emerged because of the SP’s demand for higher control into the 
MLPS based networks. Thus the purpose of existing commits the need briefly to mansion where the 
old school form of control and management fails in order to express more clearly the merits of PCE.  
Before that, the closest as a functionality to the traffic shaping and control concept was RSVP-TE, but 
still was not enough to satisfy the dynamically changing network requirements. The criteria on which 
RSVP-TE use to signalize a network path through the MPLS network are based on the requested 
bandwidth and the available such. If both of those factors are matched (and of course the requested 
path/size has the policy permission to exist) the LSP is signaled thought the network. The traffic 
management problems initialize when there is a need of two or more LSPs to be created, but the 
==PAGE:58==
53 
 
network resource is limited. In this case the LSPs must discriminate based on prioritization dogma, 
otherwise the paths would be reserved on first-come first-served principle and that would confer a 
pretty nondeterministic view of the network.  Even if taking advantage of giving a prioritization levels 
leads to an issues concerning about braking conflicts between the LSPs on the same level. Putting all 
the equally priority-leveled LSPs on disjoint pats is might be a solution, but hardly feasible 
considering the size of a common network and the number of the table routing entries in the ingress 
LSR, besides such s logical network shaping puts additional control plane restrictions and diminishes 
the management flexibility. However, extra issues like the inability of RSVP signaling to accumulate 
bandwidth vie load balancing or the logically unavoidable LSPs deadlocks despite the available 
network graph solution screamed the need for external routing logic to solve all of the above.  
      The PCE is external element taking the role of a network controller, enabling the label switch path 
calculation and supervision. In order to get active topology information the PCE is utilizing the BGP-LS 
protocol, which can provide global routing information accumulated from multiple domains (the 
routing data is stored in Traffic Engineering Database from which the PCE element can access it). 
The controlling element has evolved by initially planned to provide only stateless behavior now is 
fully capable of dynamically LSP creation in a stateful manner like equally basic-functional SDN 
controller. To receive most possible accurate live topology information it could use more than one 
source (in addition to BGP-LS) like ALTO (server base entity that generates a multilevel topology 
statistical/informational data which can be consumed by every application via service based APIs) 
and I2RS (constructing a standardized (digest the collected data by using standardized object model) 
multi-domain topology collections, based on data gained from multiple sources). The computational 
path algorithm is defined via core APIs, which creates an internal abstraction between the path 
computational and the network provisioning modules giving the possibility to the users to define 
their personally customized routing algorithms (allows the concept of creation specifically business 
driven protocols related to a specific individual requirements). The most important feature of the 
controlling element, for sure is the global topology view which can spreads across several SP 
networks giving the capability to calculate much precise inter domain LSPs.  
 
      The actual network control management by PCE is also known as segment routing and is executed 
via Explicit Route Object (ERO, which is a label stack). Based on the topology view, the PCE server 
calculates a LSP, creates an ERO and sends it to the PCE client, which resides on ingress LSR. There 
the client creates a forwarding entry for particular destination based on the provided label stack, 
which carries the path information through the overlay. In such particular way the routing is achieved 
without the need of creating a signaling state (disabling the dependencies related with RSVP-TE and 
LDP). The ERO objects are strictly bounded to the resident domain, thus an inter-domain LSP has to 
be created by two border ingress LSRs, one from each of the domains.    
     By deploying PCE in addition to their MPLS network the SP can use it to manipulate the 
environment as an abstraction with less dedicated state information containing in the core, which 
actually covers all the conceptual idea about a controller- to extrapolate the control plane from the 
forwarding capability hardware.  The PCE concept is considerable freshly –released solution on the 
market, thus there are minor details relevant to the functional and compatibility aspects to be 
clarified [1][29].

10. SDN Implementational Models 
     At the current SDN technological state there are many controller implementations deriving from 
the network needs in terms of size, scalability, protocols/ hardware utilization (compatibility), level of 
flexibility and functional extension possibility. Despite that at the initial stage of the SDN 
development most of the controllers emerged with tightly connected communicational and 
management protocols (proprietary or adding customization functionality to existing standards), 
however the trend has changed and now all the vendor specific or open source solutions are 
deploying standardized protocols (in addition) for southbound communication. For northbound 
==PAGE:61==
56 
 
interface in addition to the proprietary communicational channels in between the controller and the 
orchestration platform (or network driven applications), a RESTful APIs represent the unification for 
network state delivery which creates an abstraction layer in-between the controllers and the cloud 
management systems. However there isn’t official standard for northbound communication even 
though some workgroups are highly struggle on that particular project (Open Daylight Project). For 
the east-west interfaces standardization there is still no projects available, although some of the 
organizations/vendors have already presented plug-in solution for brand-hybrid networks and 
continuing to work for higher level feature compatibility, which can bring proprietary or specific 
features in the general network. To sum up, the controllers in general do not provide architectural 
flexibility in term of southbound protocols variety support (in some extend Cisco OnePK is an 
exception) nor do a northbound programmability standard.  
 
 
                                                                                           Figure 4-15 Different Architectural SDN Models 
    The SDN is a very broad term and defines any external centralized logic that injects directly or 
indirectly (explicitly route insertions or advertized via a control plane; could be distributed) 
forwarding entries into the FIBs. That flexible definition reasonably brought the outcome of different 
architectural SDN models (Figure 4-15), which highly influenced the control plane segregation limits 
and respectively the controller’s functionality and construction logical frame. In the first observed 
architectural design (1) the control and the data plane are not separated, but additional 
programmability and operational information is distributed through representational state transfer 
programming interfaces (RESTful APIs) by specific functionally defined application or GUI. The 
particular model is used in SP networks to provide additional control to the distributed control plane 
environment, although the concept is not sufficiently functional in management aspect to be fully 
called SDN, but it leverages the SDN deployment model.  
 
==PAGE:62==
57 
 
      The next model is purely SDN implementation (2) by completely separating the control and the 
data plane. This particular model is the only one which does not need to use translational gateways 
or to create additional packet encapsulation (overlays) to bring the SDN functional meaning to the 
network. The control logic is centralized in a cluster or singular entity but strictly separated from the 
forwarding plane, determined by specific or generally purposed (off-the-shelf) hardware.  Such 
explicitly defined controllers are NOX/POX, Trema and Floodlight. All three are realized following the 
model of separately defined component based framework, which means the controller’s major 
functional entity is broken down into interrelated utilitarian partitions, communicating via an internal 
model (direct or via an abstraction). Although the tree sharing the same modularity architectural 
approach Trema has a little more shifted view away from it. While NOX/POX and Floodlight defines a 
rich set of components constituting the controller’s logic, Trema specifies basic core modules and 
leaves the user to develop the desired extended functionality in network driven applications. The 
core module selection is so basic that functional entities like topology discovery and the data base 
storage structure are defined in user application state.  Floodlight as well do not represent the in-
memory database concept for state management, but has specially developed component which 
externally defines a database storing system (BigDB –NoSQL based), dedicated for configuration and 
network state conservation. That system can be shared among the devices into the cluster to 
provoke state synchronization. For inter-modular communication Trema uses interexchange message 
system while NOX/POX for contrast is employing query-able topology graph abstraction. In other 
words in Trema concept every module takes decisions (to answer a query or to trigger action) based 
on independent logic while at NOX/POX those decisions are focused on especially dedicated entity – 
the NetGraph. The three controllers are tightly bounded to OpenFlow as the only one supported 
southbound protocol. Before choosing a controller the supported OF version should be taken in 
consideration and the functional difference in between (Trema and Floodlight support OF 1.3 but 
NOX/POX 1.0). In fact many critical updates have been made in OFv1.3 like fixing some security 
issues and adding functional coverage. The vertical scalability of the OpenFlow open source 
controllers is in current progression form of development and thus not so reliable from security point 
of view since is based on DBMS backends synchronization concept which is considerably freshly 
emerged (they are not initially designed to share memory/state). However the exception comes from 
the Floodlight commercial solution, which offers completely state synchronization (the controllers 
are explicitly architecturally designed for the purpose) between clustered controllers.  
       In the Hybrid SDN model (3) the control plane shares centralized and distributed management 
aspects. The controller comes in addition to “old school” distributed control logic in order to give the 
network a centralized grasp of management.  Moreover, the centralized control plane appears as a 
hierarchically dominant supplement to distributed control plane in order to provide control for 
resource and route optimization or introduction of new features like QoS and policy based routing.  A 
clear example of that model is the implementation of the Path Computational Element server in 
addition to the MPLS network. The approach actually utilizes the centralized control concept (SDN) in 
the face of the PCE and creates MPLS-LSP overlays in a MPLS network environment. The architectural 
model existence has emerged from the demand to introduce the SDN centralized control paradigm 
to already existing distributed networks. It is considered as an intermediate step from the 
technologically transitional process, capable to achieve substantial proximity to the decoupling 
concept of the network logic without the need of a complete hardware shift. Thus the size of the 
financial investment would be tolerant comparable to the acquired network functional benefits 
==PAGE:63==
58 
 
which will reflect on the customer-client relationship in a positive way. All aforementioned cases     
play an excellent role in favor of the Hybrid SDN model adoption and afterwards the promotion to 
full SDN architectural transition. 
     The last model (4) achieves the SDN network programmability and flexibility features by 
establishing an interwork abstraction over the existing physical network. The abstracted network is 
totally dedicated in the virtual domain and using the underlying structure as a pool of resources. The 
underlying physical network is a totally operational under the distributed control plane concept and 
has to be manually configured in order to provide the needed bandwidth for centralized control 
management dedicated into the abstracted overlay. Although completely functional, the distributed 
control underlay is losing some value-added features in the physical network like executing services 
based on the packet header, which is not interrelated with the overlay network infrastructure 
(cannot read the overlay encapsulated control data). The aforementioned model is not considered as 
SDN technology despite the provided network automatization, because it is highly dependable of the 
underlying network and more or less is relaying on manual configuration to fuse the gap between the 
underlay and the overlay. Another disadvantage is the extra traffic overhead generated by the 
overlay control structures, of course is negligible considering the gigabit interrelated datacenter links, 
but it is worth mentioning.  Also the cost for the model maintenance is high due to the fact two 
teams of network engineers has to be employed, one for the physical network and one for the virtual 
one. Despite all of those facts, after a small research appears that a medium size Cloud SP prefers to 
use this particular model. The pointed reasons were the minimal hardware investment, the provided 
extensible call support and the higher level of security that the environment provides. The controllers 
based on the overlay concepts are those of VMware, Nicira and Juniper. While Juniper’s virtualized 
environment is based on standard protocols, seeking more natural way for interoperability the 
VMware/Nicira are based entirely on proprietary network standards pursuing compatibility via a 
standardized APIs and plug-in agents. Generally, VMware offers a great security environment, 
because it provides the whole datacenter cloud suite, from the hypervisor to the orchestration and 
management services while the Juniper’s framework propose only northbound API’s. Bringing the 
bottom-top datacenter solution allows the development of higher integrity between the different 
framework segments and leaves less room for bordering deficiencies. Generally speaking for now the 
virtualization can bring new functional entities in the network much faster and much cheaper than 
the SDN, since it is entirely defined in the software domain and do not require any kind of support 
from the physical underlay except the transportation. In addition if include the fact that an eventual 
deployment would not enforce the network to be shaken from its current security state, that gives 
enough reason for a bright future of the model [18].

1. OpenStack Architecture 
         As already mentioned the architecture of OpenStack is modularly defined, thus any additional 
services (future defined) may be added as a functional entities without the need to interfere with the 
already created functional structure. Up to now they are lots of available functional entities building 
the framework of OpenStack, but the main building blocks will be reviewed and the main focus will 
drop on the network modules. In the core of the architecture lays the compute (Nova) and the object 
storage (Swift), while later an image service (Glance) was introduced as an intersection of the 
compute and storage modules. Those images represent templates of virtual machines capable of 
immediate launch in case of need. The demand for central authentication mechanism across all 
OpenStack components brought the identity module (Keystone) into the core stack. The module 
which standardized and simplifies the GUI is the so called dashboard (Horizon) and has user defined 
instances, so it can be used by ordinary users of the cloud as well network administrators.  The 
networking service, initially defined in the computing module has been isolated into extended 
functional entity- Neutron, allowing the introduction of the network as a user consumable feature. In 
addition to the core package was included a block storage component called Cinder. A little bit more 
details about the heart components of OpenStack and their purpose argumentation as follows 
(Figure 5-2): 
 
OpenStack Compute (Nova) – the module has the control over the physical resource like CPU, 
RAM, NICs and storage drives. It is developed under Python and constitutes the abstraction 
layer which builds the foundations of the virtualization by imposing automatic utilization. The 
purpose of it is to provision and maintain the state of the virtual machines (launch, suspend, 
reboot and etc.) via the integrated set of hypervisors (do not specify particular but has a wide 
range supported hypervisors). The management of the VMs and their operational 
environment is achieved through APIs and internally defined plug-in agents in the 
hypervisors.   
 
Networking entity (Neutron; previously Quantum) – the purpose of this module is to manage 
the interconnection of the VMs by applying control capabilities over the VLANs. The users are 
granted with the possibility to define their own networks topologies, address spaces and 
network elements configurations. A major part of the provided functional services of the 
module highly depends of the appointed underlying control and forwarding plain 
technological implementation (controllers, forwarding elements, and hypervisors). 
 
OpenStack Identity (Keystone) – provides authentications by service integration. It allows 
centralized policy definition and distribution across users and systems. Information about the 
defined permissions and service availability can be acquired via a programmatic interface 
request or by the provided GUI environment integrated into the cloud platform- Dashboard.  
 
OpenStack Block storage (Cinder) – the block storage perfectly fits for serving the 
performance requirements of databases and file systems (despite more expensive and 
complex for deployment than the file storage, it provide greater performance and flexibility 
of data management). Most commonly it is used with standard Linux volumes but has 
available plug-ins for other platforms (Ceph, Netapp, SolidFire etc. [19][20][21]). It has 
manageable interfaces pointing the dashboard GUI, via which the user has a full modeling 
==PAGE:67==
62 
 
capabilities over the block storage capacity (create, assign and detach blocks of storage 
volumes to VM’s and servers).  
 
OpenStack Object Storage (Swift) – as the name suggests, the data is stored as objects with 
global ID and additional metadata.  Such a storage organizational method is primarily used 
for static data (VM images, archives and backups). The actual storage process is executed by 
software, functionally corresponding to a controlling unit which navigates the data 
distribution over multiple servers around one or more datacenters. It main responsibility is 
the data replication and integrity over the clusters. The object storage creates an abstraction 
layer which spreads across different sets of commodity hardware, eliminating the need of 
expensive equipment. Obviously the horizontal scalability is a feature supported by the 
architecture expressed in cluster storage extension by insertion of additional servers 
(multivendor or not).  
 
OpenStack Image service (Glance) – represents a VM functional templates used as a base 
ingredient for service deployment.  Those templates have designated characteristics adjusted 
for the requirements of the particular types of service. Via a REST APIs the users can 
manipulate all the stored images and assigned them to back-end storage capacities. There is 
optional possibility the images to be assigned for public or private use.

2. Internal and External Module Relations 
        The orchestration engine serves as a template-based application deployment patterns that 
utilize all the OpenStack components in order to create an automatization of the pre-defined state. 
Those last two modules are optional to the core OpenStack implementation.  The cloud management 
platform’s architecture is designed to make all of the service entities independent from each other as 
possible, which gives the flexibility of partial and customized deployment or permits the integration 
with other systems/technologies capable of functional replacement. However, those autonomous 
architectural designs do not imply that the platform will be completely functional without some of its 
core functional modules. All the additionally created services are based on that core functional 
entities- compute, storage and interconnection (networking). The interconnection between the 
architectural elements in the cloud framework is driven by the API interfaces. The users do not 
interact directly with those interfaces in order to access the services but through GUI, command line 
or specially developed app (the command line is useful for shell script integration and more direct 
logical structure access) (Figure5-3). The call of the APIs is done internally by those three user access 
methods. The APIs itself represent a HTTP encapsulation of JASON or XML payload, where JASON is 
the most preferable for new service implementation but XML is supported for backwards 
computability. When a programming interface receives an API request, it converts it to data and 
place it on the massage queue for further execution processing, in case the data is a persistent 
configuration information then, it is stored in the local database for later references. The further 
processing is done by the service daemons which receives their instructions from the queue and 
perform the service tasks with additional database references if needed (the daemons are python 
based implementations using the Paste libraries).

       The networking API module of the OpenStack suit, which provides the network as a service to the 
users by giving them the full flexibility to define and control the network elements of their own 
topology is simply based on creating virtual networks, IP network space definition and port 
abstraction in order to express networking resources. When a user creates a network he reserves a 
broadcast domain which is characterized by block of IPv4 or IPv6 address space and virtual switch 
connection points (ports). The creation of the network is actually a user definition network data

submitted by the APIs to the executing control information unit (controller), which is creating the 
desired network state. For the purpose the controller must have a functional entity that translates 
the API’s interface data to executable instruction common for the controller. Such a particular entity 
is achieved through plug-in and agent concept introduction. The plug-in architecture gives the 
prospect of network capability extension and customization in the form of newly established APIs, 
eventually later incorporated into the core OpenStack Networking API kit (if they prove their worth 
as a functional value of course). Also the concept is suitable to overcome the inter-vendor 
computability gap by adjusting the communicational control process, unfortunately not always at 
hundred percent matching rates due to the specific-vendor extra functional product differentiations.  
The actual functional responsibility of the plug-in is to interact with the underlying infrastructure by 
giving setup information used for traffic route behavior establishment (process API call and store all 
the persistent data). The plug-in can translate abstraction based logic from the APIs but the actual 
implementation is executed by the device (virtual or physical) specific drivers. It is the bordering 
architectural element between the API abstraction of OpenStack and the chosen datacenter 
centralized control/forwarding implementation, as can be conclude from further mentioned the 
plug-in element is specifically created to face particular SDN set of datacenter implementation. This 
is one of the or probably the main reason why OpenStack is so highly preferable that the other 
orchestration platforms and is profoundly stressed and suggested as a solution in the current critical 
assessment of the data center SDN implementation - the highly industry adoption rate and still the 
continues exponential growth of vendor’s expectance. Some of the large number of available 
dedicated plug-ins (of course they support different features and performance specifications meeting 
the underlying technology specifications) are belonging to the most used SDN equipment solutions 
which own the higher market shares and provides full compatibility and integration with  OpenStack: 
OVS plug-in, Linux Bridge plug –in, NVP plug-in, NEC OpenFlow plug-in, Ryu OpenFlow controller 
plug-in, Cisco UCS/Nexus Plug-in, VMware integration OpenStack plug-in (still beta version) and etc.  
The plug-in architecture is simple (Figure 5-4) and can logically extend any centralized control 
network into OpenStack functional cloud orchestration.

controller, which is going to open the full capacity of the network control management to the cloud 
users and represent it as a service. The communication in-between does not require an agent 
installation on the controller unit, but only a plug-in deployment on the Neutron server side (the 
same goes for all the OpenFlow controllers as well for the VMware NSX). The controller receives a 
REST APIs from the plug-in and executes them in order to achieve a desired network state. 
  
       The network design of a typical cloud datacenter network could be classified up to four physical 
layers (depends of the network scale- simple implementations has less vivid layer borders). The 
management layer network purpose is to interconnect the OpenStack components, which 
importance requires an extra matter of provisioning.  The connections in between the Neutron 
server or its components with the underlying network controllers are also part of the management 
network, while the forwarding plain which provides a linkage in-between the instances is designated 
into a separate layer. The API network layer handovers the OpenStack APIs to the users via GUI or 
network application. The last layer is represented by the gateway interconnection entity which 
maintains the external internet access selectively required by the VM instances.

4. Implication 
        As shown above virtual and physical switching elements can co-exist on the same network, 
which is controlled by a common controller and serving the same overlay, with a little help by 
OpenStack to achieve full automatization of the  process of launching a VM on the network. Of 
course OVS has full compatibility with most of the OpenFlow controllers without the mediation of 
the orchestration system, but still without it the VM creational and network integration process 
would be much more manually segmented and complicated (not only with the initial deployment but 
with the further usage a degree of automatization is lost). While there are available hybrid 
Virtualization/SDN networks serving the same OpenFlow created overlay, there is still no common 
controlling unit to interconnect the VxLAN and OpenFlow, of course excluding the feasible gateways 
which links the two network entities as a separate domains.  The solution of that enigma could hide 
into the OpenStack itself. Representative implementations of both overlays have interconnection 
with the cloud management plain via APIs. On both cases the neutron-l3-agent serves as a gateway 
for the external network access. By considering the fact that all the apps positioned architecturally 
over the OpenStack abstraction has a full overview of the both network overlays, can be created a 
functional controller in the form of an application, which can unite the two frameworks in to a single 
controlled environment via the internal Neutron l3 agent. Such a suggestion brings a huge amount of 
questions and uncertainties like from logical and technically-implementation nature.  Having in mind 
that the current progress of breaching the barrier of overlay unification by managed to obey the OVS 
to VMware NSX controller cluster via the OpenStack plug-in concept (the NSX L3 gateway service is 
given to take over routing/NAT neutron-l3-agent responsibilities) has reached a high challenge level 
and even functionally available has met many implementation issues [23].  
       More interesting and logically possible useful case of OpenStack would be to interconnect of a 
two overlays (VxLAN and SDN) like not to appear as a singular domain layer, but to obey the 
transport network to serve purpose to the higher layer. As so far we know the VxLAN overlay is 
created over the physical network and uses it as a pool of network resources. The underlay network 
is statically full bridge mesh configured in order to provide full physical element connectivity. In that 
==PAGE:74==
69 
 
way they are bandwidth dedicated paths for full disposal of the overlay even if they are not used at 
all.  If the transport network of the VxLAN overlay is dynamically controlled by OpenFlow instead by 
the static route definitions, opens the door for some interesting opportunities. The possibility of 
controlling the network bandwidth is giving the chance to shrink the network according to the virtual 
overlay demands. That means, building the paths when they are really needed. The rest of the 
unused network can be put to hibernate or shut it down in order to reduce the CAPEX expenses. 
Another useful case would be to serve the purpose of newly emerged green routing protocols. In a 
major scale the OpenFlow underlay can guide the virtual networks capacities through forwarding 
devices, which uses green energy sources. With a such precise control of the raw bandwidth and the 
powerful isolation concept created by the both overlays, can be followed the Amazon policy for 
network over provisioning. In some point the network capacity should be in front of the company’s 
demand and thus stay unused. By loosely leasing contract, that capacity can be sold to third parties 
and gained additional revenue.  To make this all possible, the logical communication in between the 
two overlays can be handled by specially developed application capable of relationally to 
interconnect the management information from the both networks and to be able to take coherent 
decisions in order to predict a traffic behavior (Figure 5-6).

         The easiest way (and the less expensive one in terms of investment) to implement a multi-

provided by OpenStack abstraction, which supplies management information to an applicationaly

extraction/injection of control and management network information. Nevertheless such a project 
for interconnecting different overlays (not exactly the aforementioned) already exists. On 27 of May 
2014 was officially released an OpenFlow extension to enable the SDN concept for the transport 
networks. It basically provides multilayer (Layer0 photonic and Layer1 SONET/SDH) centralized 
control frame. Further functional extensions enable OpenFlow to manipulate the switch ports and 
fabrics that serves fibers, wavelengths, and timeslots as well as packet headers by following the 
model of multilayer cross tables match and action trigger. In order to face some of the critical 
transport networks requirements the OpenFlow is extended to support protection, monitoring, OAM 
and administration. By providing all of this the OpenFlow is capable to achieve an application-driven 
provisioning of the transport network in real time and to create a multilayer logical control reference.      
The ONF transport SDN model designates two interfaces for network integration. The first one 
provides direct connection to the network provider’s SDN controller which has a direct relation with 
the transport network elements and allows the network customers to have full visibility over the 
internal physical elements. The second option is via a CVNI, where the client’s controller 
communicates with the provider’s controller by OpenFlow, in order to get a virtualized view of the 
sliced network resource dedicated for the customer use. The scope of visibility (and the 
corresponding network resource) is still under standard definitions for future consideration, but 
probably will be subject of policy agreements [24].  
    The possible used cases of the SDN concept are summarization of all its technological aspects 
discussed so far. Some of them are concerned with creating bandwidth on demand, service 
integration, tenant isolation, virtualization and etc. The network API’s abstraction concept extends 
the service creational process to the reach of the common developer. The basic set of reachable 
network services are used as core primitives to design a highly functional network utilities, which 
control logic is realized in an application. An example of such logical collection of basic primitives 
forming a functional service is a bandwidth calendaring. The purpose of which is about bandwidth 
forwarding reservation for particular time span in order to optimize network traffic events (like 
system back up). It could be combined with the network elasticity feature, which allows virtual 
machines migration without the need of or with negligible downtime .In such a way an entire data 
centers could be moved with a schedule bandwidth reservation. Another useful purpose of SDN is 
the establishment and maintenance of the service engineering path [1]. The concept was introduced 
as innovative way for network service delivery. In non-SDN networks a new network service 
introduction is followed by additional device installation and total network reconfiguration (manually 
performed). Within the down of the virtualization, any additional network functionality (load 
balancers, Intrusion detection systems, intrusion prevention systems, firewalls and etc.) can be 
delivered as virtualized instance instead of physical one. In addition with SDN the service delivery can 
reach full automatization process line. In a SDN/OpenStack perspective network, a user can request a 
service subscription via an application or web interface. Then the external logic will signalized the 
==PAGE:76==
71 
 
controller of the particular data flow change, in order to provide service reachability. By such a total 
automatization of the service process, a deployment of a new network fictional entity is directly 
dependable of a capacity, but not from purposely-defined hardware and manual configuration 
influence.   
       Building a SDN/OpenStack network is based on s specific workload requirements, self developed 
calculations and specifically pre-defined conditions. Everything already mentioned can be matched to 
an already existing vendor specific available solution. In order to create a cloud service network, the 
hardware is the main initial features to consider. There are interesting online calculators which could 
help to adjust the network desired requirements to the needed relevant hardware to support them 
[26]. Surprisingly the availability of plentiful tools revealing the details (software and hardware) 
about open source cloud creation is astonishing and emerging every day.  
 
 
 
 
 
 
 
 
 
==PAGE:77==
72 
 
Conclusion      
 
  Considering all the revolutionary features contributed to the datacenter network by SDN puts it 
ahead as a driving force of the future network development process. The concept gives the 
opportunity for gradual network technological transition as a hybrid SDN implementation before 
adopting the full SDN solution.  SDN already has proven it functional network value, evidence of 
which is the huge amount of project (open source and proprietary) dedicated for further functional 
development. That led to interoperability issues, limiting the network vendor diversity to a single 
marketing instance. A top market vendors and open organizations cooperation has started a 
common project (OpenDaylight) in order to establish a standard framework, which going to allowed 
the creation of a modular functional products with different company origins, interoperable with 
each other.  As it can be seen SDN is destined to become a common datacenter solution. Having in 
mind that it is inseparable part of the cloud computing, which further proves it future worth of 
development.  The combination of OpenStack and SDN reveals the network functional primitives to 
the application layer, where they are used for extended network service creation, limited only by the 
developer’s imagination and network perception (and of course of the core functional limits of the 
network itself).  In the end of May 2014, the SDN concept extended its prepuce outside of the 
datacenter network scope with the release of OpenFlow-enabled transport networks definitions [24]. 
It briefly explains how OpenFlow (SDN) can enable application driven programmable management 
behavior over optical and packet-optical transport networks (by a wavelength granularity and path 
control). A future extension of the SDN concept would be the break outside of the single overlay 
control and to unify a couple of layers control plains into a single controller or a system of such. But 
gradual step to such ambitious goal would be the unification of the network virtualization and SDN 
networks control layers. The particular step will reflect in higher network function efficiency and 
would introduce even higher level of network resource distribution automatization.  Logically 
speaking, a shortcut to such solution can be seen in the face of the applicational model (over the 
OpenStack API abstraction) capable of taking coherent management decisions in order to control the 
both layers for single overlay unification or hierarchically defined support purpose. Before going 
further in the network evolutionary demands from SDN a time should be given in order the currently 
achieved progress in terms of hardware and software solutions (and framework of standard 
unification) to grow some maturity and to release its full potential.
