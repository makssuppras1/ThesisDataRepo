Summary
A very popular and successful abstract-interpretation approach to security protocols is
based on encoding an over-approximation using Horn clauses, generated automatically
from a more high-level protocol notation like Pi-Calculus. Using a resolution based
theorem prover like SPASS or ProVerif one can then often quite efficiently check
whether a particular attack-fact is derivable from these Horn clauses. A limitation
built into this abstraction is the monotonicity, i.e., what is true cannot become false
later. This prevents, for example, any kind of revocation protocol. The AIF tool is an
approach to overcome this limitation by considering a different set-based abstraction
to model a little state information, while remaining true to the original Horn clause
approach. Resolution-based approaches to Horn clauses is not always the best choice.
For many purposes we would like to have a characterization of the least fixedpoint of
facts that can be derived: this can serve both for humans as a readable proof (when
the representation is compact enough) and for automated proof validation e.g. with
the theorem prover Isabelle.
This thesis explores an efficient way of computing the fixed point representations,
while proposing a method of reducing the number of facts in them. We show that these
reduced fixed points are 60-75% smaller, making them considerably more readable
for humans. We have shown that we can always restore the full fixed points from the
reduced ones. Additionally, we offer an implementation which is up to two orders of
magnitude faster and slightly more memory efficient than previous implementations.

1 Introduction
Every day billions of messages are exchanged by a large number of people over phys-
ically unsecured or uncontrolled public networks. Cryptography has allowed these
messages to be exchanged securely, even though the medium may act hostile towards
its users. Although cryptography is a vital tool, its ability to safeguard secrecy, au-
thenticity, and data integrity is strongly dependent on proper usage within protocols.
With this in mind research has not only been spent on cryptography itself, but also
on the verification of its usage using formal methods.
Formal models allow researchers and protocol developers to express protocols in a
precise manner, leveraging the ability to use automated verification tools. The most
well known and used formal model was introduced by Dolev-Yao in 1983[10]. This
algebraic model abstracts cryptographic primitives such that only its ideal properties
are assumed. In this model an attacker is not able to perform real-world attacks such
as guessing keys or exploiting side-channel attacks. Instead, he is allowed to initiate,
observe, replay, and intercept messages on the network.
Model checkers are introduced to check against certain attack goals, for protocols
written in this algebraic fashion. These automated tools often allow their users to
specify a symbol which should stay secret during one or more protocol runs, i.e. the
symbol should be unreachable for a Dolev-Yao intruder. Because users specify attack
goals, automated checkers make an excellent tool for showing that a protocol is broken,
but not necessarily secure. In practice this does not present a problem, as different
(new) protocols often share the same families of exploits despite their presence in
literature. Users should therefore be able to know which attacks to test against.
A typical protocol allows the exchange of arbitrary messages and an arbitrary
number of sessions, implying a great if not infinite state space to check. To limit
the size of this state space, researchers have been introducing approximations and
abstractions. A very successful abstraction is based on the ability to over-approximate
a protocol by representing it as a set of Horn clauses[4], often generated from a high-
level formal modelling language. It assumes that all information needed to participate
in a single session is bound to it or immutable and shared among a number of sessions.
This is true for a lot of protocols, hence the success of the approach. Unfortunately,
it is not true for all. For example, a server might keep a list of valid certificates
for its clients. Upon user request or after a deadline has been reached, the server
might decide to revoke the user certificate. Based on the validity of a certificate a

resource might be available or not, thus changing a state between sessions. Many
protocols have similar behavior, unrepresentable by Horn clauses at a first glance.
Indeed, anything derived at some point cannot become false later, according to the
monotonic behaviour of these clauses. Any resources available at some point will
therefore also be available in another.
AIF[12] was the first tool to tackle this problem, while retaining the ability to
represent protocols as Horn clauses. It allows its users to specify a fixed number of
sets, each of which may contain a certain value or not. To prevent another state space
explosion, every two values which are members of the same sets (and not of others) are
considered to belong to the same equivalence class. AIF then lets transitions in the
protocol be guarded by such memberships. These set memberships can be encoded
as subterms, allowing the user to express concepts such as the public key of Alice is
considered valid by her and server S. To determine whether a protocol is secure, the
user specifies an attack goal. For example, it might consider whenever an intruder
knows the private key of a valid key of Alice an attack. To determine whether the
protocol allows for this attack a set of all reachable facts is computed using the Horn
clause and checked for the attack fact.
Unfortunately, introducing set membership values has resulted in very large sets
of reachable facts. In the worst case, for N different sets 2N facts are present in
the set of reachable facts for each term with a single set membership as a subterm.
Even worse, for every term with M set memberships as a subterm 2M facts might be
present. The large number of facts make the set of reachable states hard to read and
interpret by humans and possibly hard to parse by machines. However, the structure
of AIF Horn clauses is such that certain facts in the set of reachable facts imply others.
For users reading the set it is easy to see which facts must follow from others given a
single one, if provided with a single and comparatively very small implication graph.
This thesis explores these sets of reachable sets and proposes methods to reduce
them such that easily derivable facts are removed. It will also propose a number of
performance improvements, based on the structure of AIF Horn clauses. In summary,
this thesis aims to make AIF an easier to use and more attractive product. We will
first explore concepts relevant for understanding the thesis and sketch an overview
of related work.
We will then explain the problem in more detail and propose a
computation method for eliminating redundant facts. We will conclude by providing
performance analysis on some practical protocols.

2 Preliminaries
It has been demonstrated that verification of security protocols can greatly benefit
from an over-approximating approach, considering a set of Horn-clauses instead of
a set of reachable states[4][16]. The approach eliminates an unbounded state space
growth by eliminating the need to check for multiple protocol sessions. Unfortunately,
this limits the number of expressible protocols to ones which only share immutable
data across different sessions, such as known agents or protocol constants. All other
data must expire within a single protocol run.
While this approach is valid for a large number of protocols, it does hinder pro-
tocols which rely on some global state being mutable. Examples include building
security protocols, where users may be allowed or denied access to some room based
on the expiration status of their electronic token. While scanning a token might allow
access for one session, it is perfectly valid to deny it for some other. For example, to-
ken holders might leave the company or shift roles within it, thereby losing or gaining
access. This behavior is not directly translatable to Horn clauses, as the latter behave
monotonically: anything derived true at some point cannot become false later.
AIF[12] was the first to tackle this problem by allowing the specification of a fixed
number of sets in a protocol. Each set stores one type of value and all values are
considered equal if they belong to the same sets, but not to some others. In other
words, all values are partitioned in 2N number of equivalence classes where N is the
number of sets present in the protocol. Each transition can be guarded by any of
these equivalence classes, effectively introducing a global state while still keeping the
advantages of the initial approach.
To develop an intuition for the idea of AIF, we will introduce a simple example
modelling a very simple security system. At the same time, we will sketch how this
state transition is translated into Horn clauses. Furthermore, we will introduce the
language and concepts used throughout this thesis.

2.2
Translation to Horn clauses
We will provide an intuition for the translation to Horn clauses, based on the example
provided earlier. AIF considers any fact bar set memberships persistent. Persistence
entails that if some fact is true in some state S1, then it is also valid in state S2
provided that S2 is reachable from S1. Any persistent fact in AIF is easily translatable
to Horn clauses, as facts in Horn clauses follow similar semantics. Set memberships
however are non-persistent facts, meaning that some value might be a member in
some set and some state S1, but not in state S2.
The tricky part is therefore compiling set-memberships or non-persistent facts.
AIF solves this by defining an equality relation for all values belonging to the same sets.
This allows the translation to encode a combination of memberships as a single fact.
Such a fact is a function called val/N where N is a number of booleans corresponding
to the number of sets we wish to encode. In the example we consider three sets: a
set holding all keys stored in ring(a), all keys considered valid by s, and all keys
considered revoked by s. The membership of a key k can be described by a triple
of booleans (k ∈ring(a), k ∈db(s, a, valid), k ∈db(s, a, revoked)) (see figure 2.1). A
revoked, non-valid key, not stored in the keyring of a would be encoded as (0, 0, 1).
In AIF this is denoted by the function val/N, where N is the number of sets implied
by the protocol.
In our example, AIF would define val/3 and encode the earlier
mentioned set membership as val(0, 0, 1).
As we have seen, val represents a key in a certain state and can be used as a
subterm. If we want to express that an intruder knows a valid, non-revoked key not
in the keyring of user a we would therefore denote iknows(val(0, 1, 0)). If we can
derive this term and the key would change membership to a new membership k′ for
any reason, then we must also be able to derive iknows(k′). For example, the key
might be moved from the previously mentioned state where it only is in the revoked
set. Because the key itself does not change (only the metadata) the intruder must
also know iknows(val(0, 0, 1)).
This intuition is formalized by the introduction of two types of Horn clauses: a
term implication clause and a context clause. Both directly follow from the other Horn
clauses. The former follows from any clause (conceptually) changing set membership.
That is, for any rule with some requirements ϕ where a set membership k is a subterm
of some term in ϕ and a consequence with some set membership k′ in the consequences.
For any such Horn clause, a term implication clause is added having the form ϕ →
k ↠k′ where k ↠k′ is a term implication fact1. The latter is added for any term
in any clause having a val as a subterm, f[·]. For any such term a context clause is
added, having the form (k ↠k′) ∧f[k] →f[k′].

2. PRELIMINARIES
2.3
Unification theory
Throughout this thesis we will use unification and substitutions extensively.
We
will adopt the notation of Baader and Snyder[2] and slightly extend it.
We will
first quickly summarize the theory. Unification considers the problem of substituting
variables in two terms in such a way that the two terms are equal after substitution.
More concretely, if we want to unify f = t(A, x) and g = t(c, x) we should replace the
variable A by t. This mapping is called a substitution and is written as σ = [A 7→t].
Applying the substitution to the term f is written as fσ. As a result, fσ = g. Solving
the unification problem is equal to finding a substitution σ such that fσ = gσ.
By convention we write variables with uppercase symbols, while constants are
written with lowercase symbols. We write every substitution as:
σ = [X1 7→Y1, X2 7→Y2, · · · , Xn 7→Yn]
The domain of σ is {X1, X2, · · · , Xn}, while its range is {Y1, Y2, · · · , Yn}. This is
written as Dom(σ) and Ran(σ) respectively. The domain can only consist of variables,
while the range consists of variables and ground terms. Substitutions are not only
applicable to terms, but also to other substitutions. Given two substitutions σ, τ the
semantics are such that (fσ)τ = f(στ) for any term f.
We extend these definitions slightly by introducing some new concepts. A sub-
stitution σ is minimal with respect to f if V ars(f) = Dom(σ). A substitution is
complete with respect to f if V ars(f) ⊆Dom(σ). We also consider γ the unification
symbol. If there is no valid substitution σ for two terms f and g such that fσ = gσ,
we say the unification is undefined and write this as γ(f, g) = ⊥. Of course, if there is
such a substitution then γ(f, g) = σ. A substitution is ground if it maps to constants
only.

2.4
Horn clauses
Variables in the clauses are universally quantified with the clause itself as a scope. A
Horn clause has the form p1 ∧p2 · · · pn →q, to be interpreted as: if p1 ∧p2 · · · pn hold,
then q also holds under a grounding substitutions. We call p1 ∧p2 · · · pn requirements
and q a consequence. We may use syntactic sugar p1 ∧p2 · · · pn →q1 ∧q2 · · · qn to
denote the set of Horn clauses {p1 ∧p2 · · · pn →q|q ∈{q1, q2, · · · , qn}}.
The following definitions define the language used throughout the rest of the thesis.
Definition 1. A fact of the form timplies(val(X1, · · · , Xn), val(Y1, · · · , Yn)) where
all variables are unique and boolean is called a term implication fact. A nicer way of
writing this is val(X1, · · · , Xn) ↠val(Y1, · · · , Yn).
Definition 2. A context clause is a Horn clause of the form X ↠Y ∧C[X] →C[Y ],
where C is a context.
This thesis considers the set of Horn clauses generated by an AIF compilation pro-
cess. The following assumptions hold for all cases considered in [12] and all examples
in the accompanying software:
Assumption 1. occurs, attack, val and timplies are reserved keywords.
Assumption 2. Variables only occur as a subterm of val and timplies.

2.6
Contexts
A context is a fact f such that it contains “holes”. These holes might be filled up
with concrete values. In this thesis holes always refer to set membership values. An
example of a context is:
f[·] = iknows(pair(·, a))
Its implicit, semantically equivalent notation is:
f = iknows(pair(·, a))
In both cases, a hole might be replaced by a concrete value by bracket notation:
f[b] = iknows(pair(b, a))
A context might have more than one hole, denoted by typical tuple notation. To
refer to a context with two holes we write:
g[(·, ·)] = iknows(pair(·, ·))
In this case, holes are replaced from left to right. That is:
g[(a, b)] = iknows(pair(a, b))
For a context with one or more holes we write (of course, such a context remains
abstract):
h[· · · ]
For this thesis we introduce the concept of maximum contexts.
A maximum
context does not contain any concrete set membership values. For example:
j[·] = iknows(pair(val(1, 0), ·))
is not considered a maximum context, because it contains the subterm val(1, 0).
In contrast:
j[(·, ·)] = iknows(pair(·, ·))
is a maximum context.

2.7
Fixed point
To determine if a protocol is secure, AIF allows the user to specify an attack state. The
reachability of this attack state determines if the protocol is secure. In terms of our
Horn clauses this entails one is interested in whether or not the symbol corresponding
to the attack state is derivable. For this purpose we define the concept of a fixed point:
Definition 3. Given a finite set R of Horn clauses, let FR be the least set such
that for every (p1, · · · , pn →q) ∈R and for every grounding substitution σ if
{p1σ, p2σ, · · · pnσ} ⊆FR then qσ ∈FR. Such an FR is called a fixed point.
To simplify subsequent proofs, we will first simplify the Horn clauses in such a
way that the only relevant types for the of this thesis are booleans.
Lemma 2.7.1. Context clauses can be rewritten such that only boolean variables
occur in it.
Proof. Context clauses are Horn clauses of the form X ↠Y ∧C[X] →C[Y ]. The only
matching facts are term implication facts of the form val(X1, · · · , Xn) ↠val(Y1, · · · , Yn).
We can therefore replace X with val(X1, · · · , Xn) and Y with val(Y1, · · · , Yn).
Lemma 2.7.2. All variables are boolean in a set of Horn clauses R.
Proof. Variables only occur as a subterm of val and timplies (assumption 2). We can
rewrite every fact timplies term to having only variables under val (lemma 2.7.1). Be-
cause every val term represents some set membership (assumption 1), every subterm
is either a 0 or 1 (boolean) or a variable.
Lemma 2.7.3. The fixed point FR is finite.
Proof. Any f ∈FR must have been a consequence of some R under some substitution
σ. This substitution can only map to booleans (lemma 2.7.2). A consequence q can
therefore only result in a maximum of 2V ars(q) facts. The maximum size of the fixed
point FR is therefore:

3 Related Work
In 1989 Burrows, Abadi and Needham introduced a method for analyzing the authen-
tication of actors in distributed computing system[9] dubbed BAN. BAN logic had
been successfully used to discover weaknesses in existing protocols, but quickly fell
out of favor due to it proving insecure protocols as secure[6].
After that, approaches based on propositional logic became very popular[1][3]
thanks to their ability to be used in conjunction with SAT solvers, which are widely
available and heavily optimized. One prominent checker is FDR[15], based on the
formal language for describing distributed systems communicating sequential pro-
cesses[11]. These checkers usually consider the complete state space of a protocol
and try to find some attack state in order to prove a protocol is insecure.
While successful in their ability to indicate security flaws, protocol checkers often
suffered from state space explosion. Even modest expansions to a protocol might
increase the number of reachable states considerably. Due to this, protocol checkers
computing all states require a lot of computation power for relatively modest protocols.
Researchers aimed to reduce the number of states by considering a different approach:
instead of really considering states they began to over-approximate by expressing
their methods in terms of derivable, state independent facts. Based on this concept,
Blanchet introduced a protocol checker based on Prolog rules in 2001[5], Weidenbach
in 1999[16], and Bozga et al. in 2003[7]. Of course, the listed number of approaches
is by no means complete. Because each fact is state independent in these problems,
protocols could be reduced to a set of Horn clauses. In turn, this meant a strong
backing of existing tools such as the SPASS theorem prover.
Although over-approximations did not suffer from state space explosion, their
approximations were unable to express data changing across different protocol runs
(i.e., a state). In 2010, Mödersheim proposed an extension, AIF, to existing methods
by adding set membership values[12]. Protocol transitions could now add and remove
values from sets, while retaining the ability to be represented as Horn clauses. In
2016, an extension on top of that one was proposed dubbed AIF-ω adding the ability
to represent families of sets.

to eliminate some of the computation steps executed by Π. We realize that most of
the computational time is spent in the γ, the unifying operator. For a given iteration
of Π we execute it between |R||C| and |R||C|p times where p indicates the sum of
the lengths of all requirements of all Horn clauses. A large chunk is spent on the
computation of context clauses, even though we know a lot about the structure of
the Horn clauses beforehand. Let us consider a context clause:
X ↠Y ∧C[X] →C[Y ]
If we track a life cycle during the computation, we realize this clause can only
derive new facts whenever the life cycle changes or upon the discovery of a new fact.
Other than non-context clauses it is sufficient to check a new fact independently of
others, as long as a life cycle updates the number of derived facts correctly as well.
Whenever we update the life cycle we might be able to derive a large number of new
facts in one go due to its transitive nature, reducing the number of iterations of Π
considerably. Without filling in the details yet, we have established an abstract form
of the new algorithm. A single iteration πred should given a set of facts C and a set
of Horn clauses R:
1. Establish a set of context clauses Rctx and all other clauses Rothers.

the preliminaries, using all non context clauses Rothers.
3. Update life cycle graph based on term implication facts in C∗. Yield all newly
derivable facts.
4. Yield all newly derivable facts based on any fact in C∗matching with any
context clause.

4.1.2.4
Algorithmic complexity
The optimization steps assume that testing for reachability is non-trivial operation in
terms of computational complexity. If we are only considering contexts with a single
hole the optimization has clear computational benefits for only a small increase in
algorithmic complexity. For facts with two or more set memberships as subterms this
case is not that clear. If the majority of cases fall through all the way to the last case
(4.4), it might even be a lot slower as it will do many reachability tests anyway.
Clearly, we could precompute reachabilities by calculating the transitive closure of
the life cycle every time we find a new edge. Transitive closures allow a constant time
lookup for every reachability query, at the expensive of spatial complexity. While a
nearest neighbor representation only needs O(|V | + |E|) storage, a transitive closure
needs O(|V |2). We will test both implementations in the last section of this thesis.

5 Measurements
Previous chapters have established a theoretical framework for computing (reduced)
fixed points, and offered various performance improvements. The preliminaries offered
a method of storing unification calculations and reduced the number of clauses which
need to be checked during a single iteration. The main chapter of this thesis looked
into the specific structure of the Horn clauses generated by AIF, and offered a way of
representing these in an efficient way. We will offer an implementation as an extension
of the existing AIF Framework and subject it to various real-world protocols already
specified in AIF.
5.1
Methodology
Measurements are taken on a machine containing an Intel Core i5-4590, using two
times 4 GiB 1600 MHz of system memory, running Ubuntu 16.04.2 i686. All software
is written in Haskell and was compiled using GHC 7.10.3, leveraging -O2. Reported
times are based on wall clock times as measured by the default time command. Maxi-
mum memory usage is measured by regularly checking the memory in use as reported
by the kernel, again using the semantics of the default time command. For really
short running times, both methods might be inaccurate. In addition to these statis-
tics, we have also taken note of the size of the reduced fixed point versus the size of
its full counterpart.
All optimizations are able to be switched on or off at a source code level. The state
of a “switch” depends on a single globally scoped, constant variable allowing GHC
to eliminate any unreachable code paths automatically. In total, five separate opti-
mizations have been implemented. In the following sections we will use abbreviations
whenever they are mentioned:
• Tabulation (TAB): precomputes unifications with rule facts and eliminates
(where possible) clause checking according to the preliminaries.
• Reduction (RED): stores a reduced context mapping representing set member-
ship values in an efficient manner. Is automatically turned on whenever any of
the optimizations below are.

• Special case multiple (SCN): similar to SC1, but operates on set memberships
of any length.
• Transitive closure (TC): stores the life cycle graph as a transitive closure
Measurements have been taken from protocols previously implemented in AIF.
The four protocols which have been tested are:
• Chinese Wall: a simple protocol based on the Chinese Wall Security Model[8].
• SEVECOM1: a protocol for secure vehicle communication, implemented and
verified by AIF earlier[14].
• ZebsKeyserverSingle: running example as specified in the preliminaries.
• ZebsKeyserver: an extended version of zebsKeyserverSingle.
To illustrate performance differences for large protocols zebsKeyserver have been
tested with a various number of principals.
That is, by default the only honest
principal in the protocol is a. We have extended this to include b, c, · · · . For every
number of principals a different measurement was taken.

In

Caching the
unification result for every term implication and fact with set membership value as
a subterm proves to need quite some memory (see figure 5.3). Enabling the reduced
context map optimization allows us to enable all others as well. Combining all those
gives us the best performance, no other combination beats it indicating that each
optimization contributes at least a little bit.
Concerns about the transitive closure becoming too big (as opposed to direct-
neighbor storage) seem to be void. In fact, storing the transitive closure actually
improved memory usage by a significant amount. A possible explanation for this
behavior is the fact that reachability queries for non-transitive closures need memory
on their own. As Haskell is a garbage collected language, these might linger in memory
for a little while. A reachability for a transitive closure does not need to allocate any
new memory however.
The improvement with the smallest impact is SC1, which has only yielded sub-
second improvements over all measurements. Although close to a measurement error,
we believe its slight improvement is still real due to it consistently showing up. Its
related optimization SCN booked greater success: enabling it approximately halved
the execution time for reasonably large fixed points.

6 Conclusion and future work
In this thesis we have explored a method for reducing the number of facts we need
to show in the set of derivable facts (fixed point) when considering Horn clauses
generated by AIF. We have shown that it is always possible to restore a full fixed
point based on the reduced fixed point our algorithm produces. We have proposed an
algorithm that in theory takes a modest amount of additional memory and provides
a notable performance improvement by exploiting the AIF specific structure of the
Horn clauses.
By rewriting the equations we have established another substantial performance
improvement, not directly related to the structure of the Horn clauses. In practice, we
have seen an order of magnitude of performance improvement purely by exploiting
the structure of the given Horn clauses. The most optimistic case we have tested
reduced the computation time from 17 minutes to 12 seconds, while unexpectedly
also slightly reducing memory requirements.
In the future we would like to see compatibility with AIF-ω[13], which would allow
us to use reduced fixed points in combination with countably infinite sets of constants.
Furthermore, we would like to see more experiments with alternative data types for
the algorithms this thesis proposed. Lastly, additional performance benefits might be
gained by considering the reduced context map in the function π, which maps over
non-context clauses.
