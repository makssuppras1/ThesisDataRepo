Summary
This thesis proposes a design for a back-end server system for IoT devices that pro-
vides scalability, security and geolocation capabilities. Scalability of the system is
ensured by modularising the design and by utilising non-relational databases for the
majority of the data. It is shown that geolocation is possible without changing the
current local LoRaWAN protocol. However, with the current gateways, the precise
location of a device is not possible due to limitations in the ToA reported by the
gateways.

CHAPTER 1
Introduction
This paper will explore how to design and develop a back-end system for internet of
things (IoT) devices deployed in industry. Several aspects important to industrial
usage will be examined such as the choice of communication technology, asset local-
isation, data management and security measures. For an enterprise to be able to
track their IoT devices the devices will have to be within their network. However,
it might not be feasible to acquire and maintain a single network large enough that
all the devices in their enterprise will always be within the network. Therefore the
system will need to have the capability of utilising external networks. Furthermore,
with the recent introduction of GDPR in the EU, the way that data is managed has
become an important issue. This leaves a number of challenges that the system needs
to overcome.
1.1
Challenges
The growth in the adoption of IoT is predicted to be so rapid that new solutions will
be needed to handle the enormous number of connected devices. An arbitrarily large
number of devices all communicating with the same system imposed great require-
ments to the scalability of the server architecture and the scalability of the database.
The servers and databases will need to be scalable in two ways, they will have to
be extendable when the load reaches the capacity of the hardware and such exten-
sions will have to not result in congestion of the messaging between the additional
components. If the system is to be of use to companies working with sensitive data
then the system will need to enact strict security measures and access control. The
confidentiality, authenticity and integrity of user data and messages will have to be
ensured throughout the system and the network. This will require strong encryption
and authentication mechanisms. The following section presents the state of the art
in the areas of scalability and security of systems and databases.
1.2
State of the Art
When choosing where best to apply your resources it is necessary to examine the state
of the industry that you are entering. The IoT industry is a maturing industry, for
this reason, there are many competing alternative technologies that will impact the

1.2.2
The Implemented System
Several proprietary alternatives exist for back end server systems designed to collect
data from IoT-devices, using the terminology of LoRaWAN the back end system
presented in this thesis would be similar to a network server however it will operate
as an application server to be able to utilise the preexisting LoRa networks all-ready
deployed around the world. This means that each LoRaWAN network provider will
essentially have a similar system already, however, most of these are not possible to
set up and run yourself as they are proprietary. By interfacing with the network
servers, it will be possible to limit the amount of additional hardware needed to get
coverage in an area at the cost of having to buy access to the all-ready established
network.
1.2.3
Scalability
A number of ways to improve scalability have been gaining traction in recent years.
In big data there has been a tendency away from relational databases and towards
key-value pair structures similar to associative arrays.[Pok11] Using key-value pairs
sacrifices several of the convenient properties of relational databases, such as ACID1
compliance and the structure of the data. An increase in scalability and efficiency is
gained by removing most of the structure and breaking ACID compliance.
Breaking ACID compliance leads to problems of consistency in the distributed
databases, a solution to this problem being used in the industry is called eventual
consistency. Eventual consistency employs strategies that guarantee that after some
time the databases will be consistent, that is agreed on the state of their data, however,
it may be favourable to instead partition the databases so that they don’t share any
data eliminating the need for consistency and allowing for parallel querying of the
databases.[Ye+02]
To handle a large user base websites have been utilising caching servers to dis-
tribute the handling of repeat requests, but this still leaves the problem of commu-
nicating with all the IoT devices. Scalable many to one communication is not as
well matured as one to many communication has become over the years driven by
the increasing needs of web-services, one approach is to use a fixed size messaging
protocol.[SD05] Fixed-size messages might, however, impose too much of a limitation
on the system.
1.2.4

There are several vectors of attack that needs to be considered when hosting a service
on the internet, They include loss or disclosure of sensitive information e.g. creden-
tials, GPS data, etc., DoS attacks, man in the middle attacks and network sniffing.
To alleviate the threat of DoS attacks on the server content delivery networks can

CHAPTER 2
Theory
In this section, the theory of several subjects that makes the basis for this project
will be presented. First the technology used by the external systems interfacing with
the server will be discussed, both in regards to how the IoT-devices function and
how messages arrive in the system.
Secondly, the theory behind the architecture
of the built system will be explained followed by the access control scheme and the
encryption standards used. Lastly, the geolocation of an IoT-device without the use
of GPS will be explained.
There are a number of alternative LPWAN technologies that would support the
use case of this project. Technologies like Sigfox and LTE-M, however, the proprietors
of Sigfox and LTE-M do not allow just anyone to use the technology. The proprietor
of LoRa, on the other hand, allows anyone with access to certified hardware to set
up their own network. This means that it is possible to extend the coverage without
having to negotiate with a single network provider but instead set up ones own net-
work where it is needed. Such a network would need two main components a physical
communication channel between the network and the IoT devices and a network of
servers and gateways. The physical channel uses LoRa while the networking is done
using LoRaWAN.
2.1
LoRa
LoRa is a wireless technology using radio waves to communicate between entities over
relatively long distances. LoRa operates in the license-free spectrum since parts of
the electromagnetic spectrum are reserved for different things around the world LoRa
has to use different bands in some regions. The 868 MHz frequency band is used in
Europe. The 915 MHz frequency band is used in Australia and North America. The
923 MHz frequency band is used in Asia. Some of these bands have a restriction on
how they can be used. For instance, a device is only allowed to transmit for a fraction
of the time it is operating when using the 868 MHz frequency band in the EU.
Being in a license-free spectrum LoRa transmissions are bound to encounter a lot
of interference for this reason it employs a number of strategies to mitigate the risk
of message corruption. One of these is using chirp spread spectrum.

2.2
LoRaWAN
LoRaWAN is a protocol that facilitates the use of LoRa over the internet and operates
on the 2nd (Data link) and 3rd (Network) layer of the OSI model, it is an open
specification which encourages companies to implement and support it. LoRaWAN
handles the routing in the network from the devices over the gateways until the
network server beyond which conventional internet protocols are used to contact the
relevant application servers.

first class is A all LoRaWAN devices are compatible with this class.
Device Class A supports bidirectional communication with downlink transmissions
(RX1, RX2) being limited to shortly after an up-link transmission.

2.3
System Design
A system needs to accommodate its users in as unobtrusive a way as possible while
still satisfying a set of criteria. When designing a system the most important aspect
is to decide on the priorities of the system. What expects are essential and must be at
a certain level, what aspects should be optimised and what aspects can be neglected.
A number of these depends on which target audience is desired and on how they
are expected to interact with the service the system provides. For instance, if the
target audience is small the system won’t need to work at large scales so the scalability
of the system can be ignored. Likewise, if the system won’t be handling confidential
data then there is no reason to invest heavily in data protection. Once the criteria of
the system have been determined a design can then be made that fits the criteria as
closely as possible.
There are a bunch of typical criteria that most systems need to consider, they
are, scalability, security, usability and feasibility. When it is decided that a system
needs to work at scale, some additional considerations will come up. Considerations
like what parts of the system need to scale well and what features should work when
the system is massive. Likewise, there are many things to consider when deciding on
the security of a system. Should users have anonymity so that what they do on the
system can’t be traced back to them thus protecting their personal data or should
they be accountable for their actions, this is a very important consideration as it is
not possible to have both. Usability can often be at odds with the other criteria, as it
can be hard to have certain features implemented securely while allowing for human
error. If a user losses their password or encryption key should they be able to get
their data back or is the confidentiality of the data too important to risk any such
measures. At the end, it is a good idea to evaluate if the resources to actualise the
systems are available or if some requirements must be cut or reduced.
2.3.1
Server Architecture
Having a high abstraction in the architecture is beneficial to the scalability, production
cost and maintainability of the system, because it makes it possible to change only
parts of the system without it affecting the rest of it. In a system with high abstraction
a component will only need to know how to interface with other components not what
they do or how they do it, this means that a new component that can handle a greater
audience can be added to replace or administer the previous component as long as it
satisfies the same interface.
==PAGE:18==

The separation of components has security benefits as well since an attacker that
gains access to part of the system won’t necessarily be able to get to other parts of the
system, as they are limited to the provided interface to other components. Stopping
an attacker from gaining access to restricted data through the interface can be done
by incorporating access control into the interface.
2.3.2
Access Control
Access control is the means by which to restrict access to only approved users. In
additions to the RSA encryption passwords give another mean to authenticate a user
by requiring the users to know a shared secret. The use of passwords has been dis-
couraged lately due to how they need to have sufficiently high entropy to be resistant
to attacks, this requirement results in passwords that are long and hard to remember.
So people’s passwords tend to not satisfy having a high entropy.
To support some users being able to manage a subset of users devices a role-based
access control scheme can be used to efficiently maintain an access hierarchy. Where
users will be given roles and devices will be accessible to a user only if the user has
one of the roles associated with the device and will only be granted the access that
the role has over the given device. This will make it possible to manage IoT-devices
of employees without the employees being able to change them but still see them.
2.3.3
Encryption
Multiple encryption schemes are used at different layers of the system. Asymmetric
encryption will be used to initiate communication with users, due to the triviality
of setting up secure channels using paired keys given a trustworthy authority. For
the IoT-devices sensory data symmetric encryption will be used, the exact scheme is
arbitrary, however as LoRaWAN is set up for AES this scheme will be used.
RSA is one of the most widely implemented asymmetric encryption schemes and is
sufficiently secure for the purpose. Due to it being computationally expensive it will
be limited to user identification and setting up secure connections using symmetric
encryption. The security of RSA relies on the computational complexity of prime
factorisation making it infeasible to reconstruct the private key from the public key.
In RSA a set of paired keys are constructed from two prime numbers and derived
values there off.

2.4
Trilateration
Trilateration is similar to the more known triangulation but instead of using the
angle relative to two points, it works using the distances to a number of locations
that are known. This can be done as there is only one possible position that an entity
can have that satisfy a number of specific distances to the reference points, however,
this process is complicated in the domain of IoT by the problem of determining
the true distance between a device without a known location and a fixed reference
point. The distances can be estimated in more than one way, two approaches have
been implemented one based on the signal path loss and one based on the send and
receive times of the message because they both rely on information available via the
LoRaWAN protocol.

The project will be comprised of several interconnected but distinct components,
handling the challenges imposed by the problem.
The components will be three
servers, the main database, a credential database, and interfaces to the different
major network operators. The first server will be responsible for the communication
with the IoT devices through the use of the interfaces and potential directly. The
second server is the webserver that will expose an interface for users over the web that
allows monitoring of IoT devices and configuration of them where possible. The last
server is the Join server that allows devices to join a network server without being
manually added to it. The main database will house all the collected data from the
IoT devices as well as the metadata need to keep them operational and restrict who
can view them. The credential database store passwords as hashes with their salt
and user names.

exchanges that can be made. The interfaces of the Web Server and the Main Server

the gateways reported since even the smallest delays in the hardware of the gateways
will result in an offset at a rate of 0.299 792 458 m

Different types of tests exist for a software system, two of these will be explained
in this section. The first type of test will cover tests of individual lines of code, the
second tests features.
5.1
Unit Test
Unit tests allow for early detection of bugs if care is taken and all the code is covered
by tests, that test for all wanted behaviour of the code. Meaning if any change in
the code courses any other part of the code to not do what it is expected it will be
caught and can be fixed. This is done using unit tests for each implemented function,
they consist of setting up small individual parts of the system so that the function
can be executed and then asserting that the state changed in the intended way after
the function was run.

CHAPTER 6
Conclusion
The finished system can handle most of the use cases that were decided on from the
start, with some exceptions mainly the secondary scenarios in the use-cases. The
mixed database types achieve some scalability by separating the largest majority
of the data related to a user in an associative database but still retains most of
the benefits of a rational database. For the service provided by the system to be
truly scalable some additional infrastructure is needed to handle high demand of the
webserver. What is needed is some ways to cache requests as discussed in section
(1.2.3) caching servers are used for this purpose in large websites and would work
well for this system as well.
Trilateration using ToF and RSSI is possible but both come with a downside.
The problem with using RSSI is that RSSI is designed to be used for informing
a user of the relative signal strength.
This means that it was not a priority to
clearly define what an RSSI value means and therefore it is not a surprise that RSSI
is not a perfect solution.
The implemented trilateration solution has shown that
the clocks in the gateways in the vicinity of the Technical University of Denmark
are not precise enough in the receive timestamp they provide when they receive a
message.
But that given more accurate timestamps it is possible to geolocate a
device in a LoRaWAN network without using additional hardware in the IoT device.
Some trilateration approaches that use an iterative approach have shown promise in
locating a device even with slightly inaccurate timestamps.[FP17] This means that
even with the limitations of the current gateways there might still be room to improve
the accuracy of the algorithm.
LoRa might lead to problems as it has no mitigation for congestion. This will likely
be a problem in the future since the adoption of IoT is projected to rise considerably.
However, there has been some suggestion on how to solve this problem. If this problem
is solved it might be worth it to extend the system so that it can function as a network
server that communicates directly with the gateways.
