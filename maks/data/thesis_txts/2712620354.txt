Introduction
Evolutionary Algorithms (EAs) are generic, population based metaheuristic search algo-
rithms. EAs use methods inspired by Darwin‚Äôs evolution theory: reproduction, mutation,
selection, crossover. These methods can generate strong individuals that help to solve
the optimization problem. The history of the evolutionary algorithms dates back to 1960s
and there are numerous evolutionary algorithms and benchmark problems defined.
One of the challenges of EAs are that they can get stuck in local optima. There are
different strategies overcoming this problem, for example the following:
‚ÄĘ Stagnation Detection
‚ÄĘ Crossover operators
‚ÄĘ Estimation-of-distribution algorithms
‚ÄĘ Diversity maintaining operators
I selected numerous EAs from recent literature to implement for each strategy, namely:
‚ÄĘ SD-RLS, SD-RLS*, (SD-(1+1) EA, SASD-( 1+őĽ) and SD-RLSm
‚ÄĘ Hybrid GA
‚ÄĘ compact Genetic Algorithm
‚ÄĘ (¬Ķ+1) GA with high mutation
Evaluation and benchmarking will be done in different type of Jump functions where the
global optima may be in the middle of the gap. Other benchmarking function will be Cliff,
and the Minimum Spanning Tree problem. The algorithms and these cost functions are
defined in chapter 2. The motivation for this project is to see how these algorithms run
on different problems and to compare them to each other a framework was created. The
framework‚Äôs design is explained in chapter 3 and the implementation detail in chapter 4.
Finally, the algorithms are evaluated in the framework, using statistical tests and reflecting
on their runtime in chapter 5.

2.1
EDA
Estimation of distribution algorithm(EDA) guides the search towards the optimum via cre-
ating and sampling promising candidate solutions. Optimization is done by updating these
candidate solutions which in the end is likely to generate global optimum. The difference
between an EA and an EDA is that EAs use an implicit distribution while EDAs use an
explicit probability distribution.

2.5
Stagnation Detection
Stagnation Detection(SD) was proposed by Rajabi and Witt in [6]. In Evolutionary al-
gorithms you can adjust the mutation rate, population size during runtime although most
algorithm use static settings. There is evidence that EAs should adjust their static settings
during runtime to achieve better results (e.g.
[7], [8], [9], and [10]). However, these
algorithms were not created for escaping local optima and cannot detect if they are in one.
Therefore, they can not, for example increase their mutation rate in a local optima. Stag-
nation detection can detect being stuck in a local optima and can increase the mutation
rate gradually to escape from it.

The (1+őĽ) EA with self-adjusting mutation rate was defined in [11]. Half of the offsprings
are created with a mutation strength of 2r and the other half with r

increase the mutation rate, and won‚Äôt mutate half the offsprings with r/2, just with r as we

in the original algorithm to initialize r, but I set it to 2 during my testing. There is no runtime

SD-RLS*
There is a higher then 0 chance that SD-RLS will miss the required strength to escape
local optima so it can run forever. In SD-RLS* which is an extension of SD-RLS [12] a
new variable r is introduced called the radius. This controls how far the algorithm should
explore. Variable s = r when the algorithm starts or when r is increased. If the threshold
exceeds it will decrease s if s is greater than 1. If s = 1 and the threshold is reached again
than r will increase. If r = n

3.1.1
OpenGL and graphics libraries
OpenGL [19] is a cross-platform application programming interface used for rendering
2D and 3D vector graphics, first released in 1992. The API defines a set of functions, that
are usually implemented in the GPU hardware by the vendor. To call these functions I
used the OpenGL Extension Wrangler Library (GLEW) [20]. To create the window where

can handle input by mouse or keyboard, but this functionality was not used in the project.

3.2.1
Cost Functions
The classes implemented for the Cost Function part can be seen on figure 3.1. These
classes represents the Problem Instances introduced in section 2.6. We have a base
class CostFunction and all the other classes inherits from this. All the methods that the
inherited classes have to define are virtual. This is to implement polymorphism and for
the derived classes to override it. We can create a CostFunction from N which is the
length of the bitstring or from an other CostFunction. The GetMaximumFitnessValue is
used as the stopping criterion for the algorithms and is also the global optimum. The
FitnessValueToSum is used to calculate the sum of 1s from the fitness value. This is
useful when displaying the algorithm.
Creating Cliff does not require any additional information. For Jump we have to specify
the size of the gap and optionally the type of the Jump. This can be the Original, Offset
or Offset Spike. There is a CheckGapSize function to check if the gap size we selected
are correct. In the case of the normal(Original) Jump it can be the maximum size of the
bitstring or a minimum size 1. In the Offset cases it can be a maximum of N/4 as the
offset starts at 3N/4.
In the case of MST we need to specify a file which contains the info of the graph we would
like to run the MST on. These files are automatically generated during program run and
saved automatically with an .mst extension. It contains the number of nodes, number
of edges, the weight of the minimum spanning tree, and the edges with their weights.
The format of the file is the following: first line is the number of nodes, second line is the
number of edges, third line is the minimum weight for the graph, and the following lines,
are the edges, where the first and second number are the nodes that the edge connects
and third number is the weight(0-100). The algorithm used to find the minimum weight is
the Kruskal‚Äôs algorithm [25]. ReadIn is used to read in this type of file and save the data
into the variables. The VisitEdge function is used along with the ConnectedComponents
to find out how many components does the current bitstring has in terms of the underlying
graph. Calculating the fitness value is done via ‚ąí((components ‚ąí1)mM2 + (edgesSum ‚ąí
mNodesNum+1)mM+weightSum) where the components are the number of connected
components, edgesSum is the number of edges selected by the bitstring(where the bit
is a 1), mNodesNum is the number of nodes, weightSum is the sum of the weight of the
selected edges and mM = 10000. While mM here is a constant and works fine on graphs
with small weights, depending on the edge weights this should be increased. The two
terms, other than the weightSum are used as a penalty. We can see that if the selected
edges make a spanning tree the algorithm will only calculate the weight of the edges.
This formula is designed to guide the algorithm towards connecting all nodes together
then getting rid of cycles and finally minimizing the weight of the tree. The Evolutionary
Algorithms were programmed to maximize the fitness value that is why the negation is
used. This algorithm and the objective function is from [26].

4.1
Delta Evaluation
When evaluating a bitstring‚Äôs fitness value we need to go over the whole bitstring. In
some cases it is possible to measure the change from one bitstring to an other and eval-
uate only the change. This is called delta evaluation, which is possible when using the
Jump or the Cliff cost functions as these functions fitness value depends on the number
of 1s so we can track the change and apply that. This can improve the algorithms speed
as it does not need to go over the bitstring at every fitness evaluation. This is achieved
by the GetFitnessValue function which can take the change or the bitstring. Since poly-
morphism was used, it gets abstracted when taking a CostFunction parameter that which
CostFunction it is, therefore it is not clear if we can use delta evaluation. It is possible
to create and use an id of some sort to differentiate the CostFunctions based on that,
but an other method is to use a dynamic cast. Dynamic cast will return a valid pointer
when succeeding or it will return a nullptr if not. Based on this, we can use this code
dynamic_cast<Jump*>(mCostFunction) != nullptr to check if a mCostFunction is a Jump
type.

ask both of this from the EA. The problem with this is that getting these two values are not
atomic, so we can have a fitness value not intended for the bitstring we get. This would
result the red circle not displayed on the line. We could recalculate the fitness from the
bitstring, but a faster operation is to calculate the sum of 1s from the fitness value as we
only need that. So the Jump and Cliff cost functions implement a FitnessValueToSum
function for this.
The MST needs the bitstring as that determines if the edges are displayed. Here, the
delay is fine as it does not matter as it the fitness value only being updated one frame
later as it is not noticeable. To get the bitstrings, mutexes are used so the EA does not
update the bitstring while it is being read by the graphics part. Fitness values read atomic
values, so read and write operation are atomic and there is no need for mutexes. This is
the reason, when used the GRAPHICS macro the program is slightly slower.
The colors are passed to a different buffer and has to be drawn as well. Here, it is important
to pass 3 color points for each vertex. These 3 points are the intensity of the red, green
and blue and has to be between 0 and 1, where 1 means it is displayed with full intensity.
The frames per second is also restricted to around 60, so if a frame drawing finishes in
less then 1000/60 ms it has to wait until its total will be 1000/60 ms, then the next frame

To see if they differ statistically we can do a Mann-Whitney U Test [31]. Mann-Whitney

files. Then following the calculation described in this video [32] it calculates the Z-score.

Conclusion
I created a program and with that I was able to investigate different Evolutionary Algo-
rithms on different problems and compare them to each other. Moreover, the program al-
lows to visualize how different algorithms solve the problems and can easily be extended
with new algorithms or cost functions. I was also able to see, how the different algorithms
can overcome local optimum in different cases. In the case of the regular Jump, Hybrid
GA was the most effective and in Cliff as well. On other Jump types cGA seem to perform
the best. In the case of MST the (¬Ķ + 1) GA and the SD algorithms performed well with
SD-RLSm being the best.
Future work could include for the program to be able to add benchmarking task via an
input file, opening multiple windows at the same time, creating re-sizeable windows and
using an efficient algorithm to generate graphs that has spanning trees in them.

How To Use The Program
A.1
Compiling
To compile the program the cmake file can be used. Simply go to the build folder, enter
cmake .. to the terminal then make and the program will be compiled, and the executable
will be named ThesisCode. To add the graphics option you need to use windows as oth-
erwise the cmake won‚Äôt let you do that. It should not be too hard to port to other platforms,
just need the necessary libraries for it. To specify the graphics pass -DGRAPHICS=1. In
the case you have a graphics one compiled just run cmake again with -DGRAPHICS=0.
It is always a good idea to specify the build type, for example the release one. In cmake
you can do that with passing the -DCMAKE_BUILD_TYPE=Release argument. This will
enable all optimization on the program, equivalent with gcc‚Äôs -o3 argument. In the case
of windows the cmake might produce a visual studio solution file. This is undesirable as
then in visual studio you manually have to select where the libraries are. In case you
would like to use that I recommend checking out [35] and [36]. Otherwise I recommend
to use MinGW [37], for that cmake and make works out of the box. To specify mingw,
you can pass cmake the -G ‚ÄĚMinGW Makefiles‚ÄĚ argument. So to summarize, to build
with cmake on windows with graphics you should run cmake .. -G ‚ÄĚMinGW Makefiles‚ÄĚ
-DGRAPHICS=1 -DCMAKE_BUILD_TYPE=Release and then make. There are sepa-
rate libraries for visual studio and mingw. In glfw this is marked with the folder names in
glew, the libs for visual studio are in the lib/Release folder where as for mingw its in the
lib/Release/Win32/mingw folder.
A.2
Menu
When running the program, a menu seen on figure A.1. On the first line we can see
if the program was compiled for graphics or testing mode. We can select the algorithm
with typing the digit displayed front of it and then pressing enter. Then, if the algorithm
requires more info the program will ask for that, for example the population. Then we
can specify the bitstring length or the number of edges in case of MST. After that, we
can select the cost function. Here additional question might be asked for example for
the jump type and gap size. If all is specified, two things could happen. If the program
was compiled without graphics enabled, the program will schedule the EA and run it. If
it is compiled for graphics we are asked if we would like it displayed. A delay can be
specified which delays the algorithm by the specified milliseconds. In the case of the
MST a question will be asked if we would like to display the edge weights as well. After
all this is specified the window will be opened and the EA will run there. In the case when
there is one window open, we have to close that to open an other one. If not closed the
window open question will not pop up and the program just schedules the algorithm. To
view two windows simultaneously the program should be started twice. The program will
run up to the total number of threads of algorithms at the same time. If more specified the
program will schedule them and run them once one other algorithm is finished. Exiting can
be done by pressing 9 in the main menu. If there are algorithms are running the program
will ask you if you would like to quit or go back. The run results are saved into a .result file.
In the case of MST the program will search for the graph specification and if not found it
will generate it and save it into to the local directory. The same is done if it needs to be
displayed, but the center of the nodes are not specified yet. Input checking is done and
also usually it is made sure, that only correct values can be passed, for example in a N=10
