Abstract
A resizable array is a fundamental data structure supporting a se-
quence of items with random access in constant time. The sequence may
be modified by adding or removing items from the end of it. We define
a new resizable array, capable of storing n items, using at most O(n1/3)
additional space, and temporarily O(n2/3) during rebuilding of the data
structure. We compare it to existing resizable arrays empirically. We fur-
ther propose a way to extend the new array to only use O(n1/k) additional
space and temporarily O(n1−1/k) additional space during rebuilding.

1
Introduction
The static array is a fundamental data structure. It is capable of providing
access in O(1) worst-case time to a fixed number of items. However, when the
total number of items to store is not known in advance, static arrays falter. A
na¨ıve solution would be to allocate all of the available memory in the system.
However, if only a few items need to be stored, the rest of the memory is wasted.
Hence, the need for dynamically allocate more memory when required – or a
resizable array data structure.
In this paper, we introduce new resizable array data structures, and elabo-
rate on existing one. They are analysed both theoretically and experimentally.
Baseline comparisons are made to the native implementation of ArrayLists in
Java.
1.1
Outline
The remainder of this section is dedicated to introducing the data structure
formally and give an overview of existing literature on the topic.
Section 2
goes into further detail with some of the existing data structures, and we will
introduce our main contribution, the VB–Array, in section 3. Section 4 provides
information regarding the implementation of resizable arrays, as well as how the
performance of the arrays have been measured – the results of which are shown

We will discuss these results in section 6 and suggest areas of
improvement, ending with some concluding remarks in section 7.
1.2
Problem description
The formal description of the resizable array is introduced in this section, as
well as some definitions and terminology to ease further explanations.
A resizable array is a dynamic data structure, storing a sequence of n items.
These items may be modified, and new items can be added or removed from
the end of the sequence. A resizable array A = a0a1 . . . an−1 must implement
the following operations:
length() Report n, the number of items stored in A.
get(i) Report the item stored at ai.
set(i,a) Update the item stored at ai to a.
grow(a) Add a to the end of A, increasing the number of items stored in A by 1.
shrink() Remove the last item of A, decreasing the number of items stored in A by
1.
All of these operations should be supported efficiently, while having a small
space consumption. We will consider the space required to store all n items
separately from the overhead of the data structure (eg. space complexity will
3
==PAGE:5==
be n + f(n), where f(n) denotes the overhead). The overhead will be counted
both between operations and temporary additional space used during opera-
tions. This overhead will be referred to as static and dynamic respectively. A
⟨f(n), g(n)⟩-resizable array has f(n) static overhead and g(n) dynamic overhead.
We will refer to get() and set() operations as indexing operations, as
they require access to a specific index in the data structure. Similarly, grow()
and shrink() operations are update operations as they modify the size of the
resizable array.
We will define both data and index blocks. A block is a chunk of consecutive
words in memory. The items in the resizable array are stored in the datablocks.
Index blocks store information required to maintain the data structure – often
pointers to the datablocks. Occasionally, a data structure must re-allocate mul-
tiple blocks and/or move data between them in one operation. This process is
called rebuilding.
1.3
Previous work
We will introduce existing literature on resizable arrays, and elaborate on a
generalization of them; the dynamic array.
1.3.1
Resizable arrays
A resizable array can be implemented na¨ıvely with a standard, fixed size array.
The array is reallocated every time an element is added or removed such that
it exactly fits all of the items. No excess space is required to maintain the data
structure, and length(), get() and set() all run in O(1) worst case time.
However, both grow() and shrink() take O(n) time. An overhead of O(n) is
caused while items are being moved between arrays. Another approach with
a single array increases the size of the array by some constant factor c each
time it is full and moves all items over in the new array (eg. see Cormen et
al. [4], section 17.4). If the array becomes ”too empty”, a new, smaller array is
allocated and items are moved over. This leads to a data structure supporting
update operations in O(1) amortized time and the others in O(1) worst case
time. It is possible to de-amortize these operations, such that all operations
are O(1) worst case time by moving items over individually in later update
operations.
Works by both Sitarski [16] and Brodnik et al. [2, 3] have introduced resizable
arrays where the overhead is O(√n), while supporting all operations in at most
O(1) amortized time. Both of these approaches utilize multiple smaller arrays
to store all the items, and employ an index block to maintain pointers to the
smaller arrays. Moffat et al. [12] introduced a variation of an array by Brodnik
et al., achieving the same time and space bounds.
Tarjan et al. [17] introduced resizable arrays, where all queries can be done in
at most O(1) amortized time with only O(n1/r) overhead, and at most O(n1−1/r)
space during update operations. Tarjan et al. also showed that any ⟨f(n), g(n)⟩–
resizable array must have f(n) · g(n) ≥n space overhead at some point, even if

we only need to support grow() and indexing operations.
1.3.2
Dynamic arrays
The dynamic array is a generalization of the resizable array. It supports all
of the operations of the resizable array, and an insert(i, a) operation. This
operation inserts item a at index i such that each old item at j ≥i now becomes
item j + 1. Resizable arrays can na¨ıvely support this operation in O(n) time
with O(n) space overhead by removing items (and storing them in a secondary
structure) until the new item can be added.
Then, the removed items can
be re–added to the resizable array. Naturally, data structures designed with
this in mind support insert(i,a) more efficiently.
However, it comes at a
cost. The best achievable time bound for all operations at the same time is
O(log n/ log log n) due to Dietz [5] and Fredman et al. [7]. One can implement
O(r) worst–case indexing operations, for r ≥1, at the cost update operations
increasing to O(n1/r) (See Goodrich et al. [8], Raman et al. [15], or Bille et
al. [1] for instance).
While these data structures provide additional functionality to that of resiz-
able arrays, the time bound for (some of) the operations are strictly worse.
2
Data structures
We will further explain the resizable arrays mentioned in section 1.3. For each
of these data structures, we will explain how they function and analyse both
the space occupied and the time to perform the operations.
For any resizable array, the number of stored items, n, can be explicitly
maintained.
With only O(1) space overhead, it allows for O(1) worst case
length() operations. As such, it will not be explained further, but one can
assume that any resizable array supports this operation.
2.1
Constant array
A simple resizable array structure is the (1 + α)constant array for a constant
α > 0. It is also known as a doubling array for α = 1. It is a standard feature in
many programming languages according to Wikipedia [19], with 1/4 ≤α ≤1.
A constant array uses a single datablock A of size N to store all n items. We
maintain A such that N/(1 + α)2 ≤n ≤N.
Indexing operations are straightforward, as the index of an item directly
corresponds to the index in the datablock. If the array is not full, a grow(a)
operation simply inserts a at A[n] and increments n. Otherwise, the constant
array is full, and it must be rebuilt; a new datablock A′ of size (1 + α)N is
allocated, all items from A are moved over to A′.
The resizable array now
stores A′ instead. Finally, A is de-allocated.
Similarly, if A is still suffificently full, a shrink() operation retrieves the
last item stored at A[n −1], decrementing n afterwards. If the array contains
less than N/(1 + α)2 items, it is rebuilt with a size of N/(1 + α).

2.2
Hashed Array Tree
The Hashed Array Tree (HAT) was introduced by Sitarski [16]. This resizable
array maintains a parameter √n ≤B ≤4√n and an index block of size B. The
index block has pointers to ⌈n/B⌉datablocks, each of size B. At most two of
the datablocks are not full.
An item with index i is stored in the ⌊i/B⌋th datablock at index i mod B.
This make indexing operations straightforward, especially since B can be kept
as a power of two.

A grow operation just places the new item in the first free entry, if there is
a non-full datablock. If all datablocks are full, a new datablock is allocated and
the new item is inserted there. Otherwise, if the entire structure is full (and thus
B = √n), it is rebuilt. Rebuilding allocates a new index block of size 2B. The
old datablocks are moved over in pairs, each datablock pair merged together to
create a new datablock of size 2B. Afterwards, B is doubled.
Shrinking removes an item from the last non-empty datablock.
The last
datablock is de-allocated if the last two datablocks are empty. If B > 4√n,
the data structure is rebuilt with B halved. The rebuilding is equivalent to the
rebuilding procedure due to a grow() operation, except the old datablocks are
now split into two new blocks.
Space consumption
At most two non-full datablocks are allocated, giving
an overhead of at most O(√n) since each block is of size B and B ≤4√n. The
size of the index block is also O(√n).
The overhead during rebuilding can also be limited to O(√n) if the dat-
ablocks are allocated and de-allocated one by one. Hence, the HAT data struc-
ture has a space overhead of O(√n).

2.3
Brodnik et al.
Brodnik et al. [2, 3] introduced two variations of resizable arrays. We will cover
both, but go into further detail with the latter variant as it lays the foundation
for our own data structure. Both variants are based on the same principle; an
index block is maintained by a (1 + 1)constant array, containing pointers to
datablocks of increasing size. A feature of this data structure is that it only
needs to rebuild the index block; items stored in datablocks are never moved.
The two variants only differ in how the size of datablocks increases – and thus
how it is indexed.
Common for both variants are the update operations. Growing the resizable
array is straightforward; the new item is immediately added to the last datablock
if it is not full. Otherwise, a new datablock is allocated first. As such, at most
one data block is non-full. Similarly, the array is shrunk by removing the last
item from the last non-empty datablock. If the last datablock is empty, it is
de-allocated.

It uses a two–stage approach, where 2 sizes for datablocks are used, stored
in two separate index blocks. The first index block I stores up to 2B datablocks
of size B, called small blocks, where n1/3 ≤B ≤4n1/3. All small blocks are
full, except from the last datablock which may contain any number of items.
The second index block L stores up to B blocks of size B2, called large blocks.
Large blocks are always full. The structure is also shown in fig. 5. Let |I| and
|L| be the number of datablocks in I and L respectively.
A new item is inserted with the grow() operation in the last datablock if it is
not full. If it is full, a new small block is allocated and the new item is inserted
there instead – unless 2B blocks are full. In that case, the B oldest small blocks
are removed and made into a new large block. Similarly, removing an item, when
there are still at least one item in the small blocks simply retrieves that item.
If a small block becomes empty, it is de-allocated. Then, if there are no small
datablocks, a large block is removed, creating B small datablocks. If n1/3 > B
or n3 > B due to a grow() operation, the entire data structure is rebuilt with
B doubled. Similarly, when n < 64B3, we rebuild the data structure with B

O(nn/3). When allocating or de-allocating a large block, it is empty at some
point, incurring an overhead of B2, or O(n2/3). Thus, the overall space overhead

VB–Array
We will now introduce our own resizable array, the VB–Array (Variable–Brodnik
Array). It combines aspects of the other resizable arrays. We will first introduce
and analyse the main data structure. Finally, we will show an application of
the VB–Array to achieve an

O(n1/3), O(n2/3)

resizable array.
We will re-use the notion of superblocks as introduced by Brodnik et al.[2];
the kth superblock SBk is a collection of datablocks that stores a total of 2k
items. Superblock SBk stores 2⌊ak⌋datablocks of size 2⌈(1−a)k⌉for a parameter
0 < a ≤1. An index block, maintained by a (1 + 1)constant array, has pointers
to all the datablocks. The data structure only needs to rebuild its index block.
Notably, for a = 1/2, this data structure is identical to that of Brodnik et al.
Most of the operations are implemented in a similar fashion.
A grow operation simply inserts the new item in the last datablock D. If
D is full, a new datablock of size 2⌈(1−a)k⌉is allocated and the new item is
inserted there instead. Afterwards, k is increased, if all 2⌊ak⌋datablocks in SBk
have been allocated. A shrink operation removes the last item from the last
datablock. If the last datablock becomes empty, it is de-allocated. Then, k is
decreased if the last datablock of SBk is de-allocated.

Let the capacity of I be the total number of items it can store, or 2B ·

superblock, we may need to rebuild I and its datablocks, such that the capacity
of I is greater than twice the capacity of the next datablock in V . Similarly,
shrink() operations remove items from I first. If there are no items in I, the
last datablock from V is removed, creating 2q datablocks in I – and then the
last item can be removed. We will also rebuild I when its capacity becomes
larger than 2B · 2q.

Discussion
We will discuss the measured results in this section, as well as propose areas of
further interest.
Aside for the ArrayList of Java, the measured memory is in line with expec-
tation, due to the defined memory measurement approach. An effort has also
been put into ensuring stability of the measured time by repeating the same
run multiple times. Some observations for each of the data structures have been
collected below:
ARL The ArrayList in Java was the fastest in shrink() operations. However,
this is in part due to the ArrayList never de–allocating memory, thus po-
tentially having an unbounded space overhead. Further checks in indexing
also impact the performance negatively.
CNA The (1+1)constant array CNA was the fastest resizable array in shrink()
operations, and both types of indexing operations. The performance of
indexing is to be expected, as it is equivalent to a standard fixed–size
array, and keeps all items in contiguous memory. Thus CNA offers great
performance in general, except that the space overhead might be O(n).
CBG The CBG array has a larger static overhead than CNA, takes much more
time to perform indexing (≈×2 or ≈×7) and shrinking, whilst being
slightly slower at growing. This might be due to the branching in indexing,
as well as having to access two large arrays simulataneously. Thus, all
characteristics of CBG are dominated by CNA, unless the rebuild in one
operation is too expensive.
HAT The HAT array was the fastest ⟨O(√n), O(√n)⟩array in random index-
ing, but the slowest (around ×1.5 times slower) in sequential indexing
together with CBG. Both update operations are among the fastest sub-
linear arrays.
BRD The constant increasing datablock size is the fastest at growing and among
the fastest at shrinking. However, it is slower than its counterpart BP2
at sequential indexing. This might be caused due to the computation of
a square-root per query. Both BP2 and BRD are the slowest of of the
sublinear arrays at random indexing.
BP2 The power-of-two based Brodnik array BP2 provides excellent sequential
indexing performance. However, it is slower than its counterpart in all
other operations. This might be due to a slightly larger space overhead,
and more frequent move of items.
TJN The TJN array is the fastest of the sublinear arrays at sequential index-
ing, and only slightly slower than VBA at random indexing. However, it
the slowest of the sublinear data structures at both grow() and shrink()
operations.

VBA VBA is the fastest array besides CNA in random indexing. A reason
that the

arrays generally outperform might be that,
given their smaller overhead, more of the index block can fit in a faster
level of cache compared to the larger data structures, allowing for quicker
access. This is less prevalent when doing sequential indexing, hence the
relative performance to the other data stuctures is worsened. The update
operations for VBA are faster than TJN and is competitive with the other
sublinear arrays.
Hence, if memory is not of primary concern, CNA offers the best performance
– performing better than the native implementation. However, if space is of
concern, both TJN and VBA are excellent coices, depending on the use case.
The former performs better in more static workloads and sequential indexing,
whereas the latter performs better in a dynamic context with random indexing.

Conclusion
We implemented and analysed a new type of resizable array, the VB–array. We
showed how to utilize it to achieve a

O(na), O(n1−a)

. Furthermore, we pro-
posed another application of it to get a

O(n1/k), O(n1−1/k)

resizable array
for k ≥3. In addition, we implemented existing resizable arrays and compared
their efficiency in a variety of benchmarks. Overall, both the new VB–Array
based datas tructure and recent resizable array by Tarjan et al. [17] provide
great performance, when space is to be kept low between operations. Other-
wise, the classic constant array provides excellent performance if space is not of
particular concern.
