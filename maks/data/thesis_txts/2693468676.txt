Abstract
In recent years, the transformer architecture has gained popularity and has become the
state-of-the-art model used for Natural Language Processing. In Information Retrieval and
Open Domain Question Answering, the BERT architecture has proven to provide highly
eﬀicient state-of-the-art results. However, BERT is still a large and complex architecture
which require a huge amount of flops for retrievals. To reduce the number of required flops
and lower the latency of retrievals, ColBERT has proposed a late interaction mechanism
built on BERT.
The goal of this thesis, is to use ColBERT for retrieval on FindZebra’s rare disease med-
ical domain. For this purpose, several datasets and models are used, e.g., MS MARCO,
MedQA, UMLS, MetaMap, and FindZebra’s own data corpus. To evaluate the model,
MedQA’s multiple choice data structure is utilised, requiring additional changes to Col-
BERT to make it possible for it to perform Question Answering.
I show that ColBERT can be used for multiple choice, though the results are not much
better than simply guessing. However, training ColBERT end-to-end to perform multiple
choice, does increase the retrieval performance with more than 100%, compared to training
only using triples.

Introduction
Over the past couple of years we have seen a surge in the development of Natural Language
Processing (NLP), with the use of Machine Learning (ML), and especially Deep Learning
(DL), which have resulted in major improvements to NLP model performance. Though
the use cases are not obviously apparent to the average person, the models are used by
billions of people world wide every day. In phones, NLP is used to predict the next words
when we are texting, or to understand speech when we are talking with an AI like Alexa,
Google Assistant, or Siri. In the industry, NLP is becoming a tool to analyse social media,
articles, etc., instead of having employees doing it manually. Yet another use case, which is
the one that this project focuses on, is Information Retrieval (IR) and Question Answering
(QA), which are widely used in search engines such as Google, Bing, Yahoo, etc.
Making a computer retrieve relevant documents given a query, is not a new discipline. For
years, researchers have developed algorithms such as Bag of Words (BoW) and TF-IDF
(term frequency-inverse document frequency) variants. However, all of these algorithms
consider each word separately, and it is therefore hard to get the proper meaning of a
passage. For example, consider the following two sentences:
1. Is the dog found?
2. The dog is found.
The two sentences contain exactly the same words, but their meanings are different. By
using Neural Networks (NN), and especially Deep Neural Networks (DNN), it is possible
to consider all the words in a passage in relation to one another, and with Attention
and Transformers introduced by (Vaswani et al. 2017), it became possible to treat parts
of a passage with less or more importance than other parts using attention mechanisms.
This property is very important in IR and QA, since a user might ask a long question,
with some words being significantly more important, or relying heavily on other words
in the text. Following the idea of attention mechanisms, (Devlin et al. 2019) developed
the state-of-the-art NLP model called BERT: Bidirectional Encoder Representations from
Transformers. BERT based models achieve state-of-the-art performance on tasks such as
information retrieval and question answering, while being computationally a lot faster than
competing models. However, the large DNN still comes with a significant computational
cost. The authors of ColBERT (Khattab and Zaharia 2020) suggests a late interaction
mechanism between the query and document encoders, which keeps the state-of-the-art
BERT performance, but with retrieval times that compare with simpler models like BM25,
KNRM (Xiong et al. 2017), and Duet (Near et al. 2019).
Commonly, BERT based models used for IR are trained on datasets such as MSMARCO
(Bajaj et al. 2018) and TREC-CAR (Dietz et al. 2017), datasets which contain queries
and passages from a broad spectrum of text data. This project aims to train a model to be
specifically good at retrieving data from a medical domain, and therefore new more specific
datasets such as MedQA (Zhang et al. 2018) and the FindZebra corpus (Dragusin et al.
2013) are used to fine-tune the model. In particular, the goal of this thesis is to implement
and use a version of ColBERT, which is specifically accurate in retrieving passages relevant
to queries about rare or orphan diseases. The model should work using the FindZebra
corpus as the retrieval source.
It can be hard to judge the relevance of retrieved documents, so I propose to use the
multiple choice (MC) data by MedQA to evaluate the model performance initially. In the
end, I will present one heuristic way to judge the relevance of retrieved documents directly.

Machine Learning
Machine Learning (ML) is the study of algorithms that can learn tendencies in data by
optimising function parameters. Besides a good model, the key ingredient is good and
relevant data. What is considered ’good’ data, depends on the task the ML algorithm
needs to solve. If the model should classify images of dogs and cats, then the data should
consists mainly of pictures of dogs and cats. The algorithm will never learn what a dog
and a cat looks like, if it only sees pictures of trucks and motorcycles. But before we can
define a good dataset, we need to define some key terms.
2.1
Types of Machine Learning
Unsupervised Learning problems are models that take unlabelled data, and where the
main task is to find tendencies in the dataset. Two examples of unsupervised learning
algorithms, are clustering and outlier/anomaly detection. With clustering, the goal is to
find clusters/groups where in each cluster, the data points are similar. In outlier detection,
the goal is to find data points that are different to the majority of the data points, which
might for example be a result of an error.
Supervised Learning models uses labelled data, i.e., data where the target is known.
An example of a supervised learning algorithm, is the classification task. Given a number
of features, the goal of a classification model, is to predict what the target is. A classic
example, is an image recognition model that predicts what is seen on the image. In such
a model, the features are the images (translated into arrays of numbers), and the target
can be for examples ’dogs’ or ’cats’. In labelled data, the training data includes the target,
so it is possible to tell the model whether it predicts correctly or incorrectly. Creating
accurate labelled data for supervised learning is usually a labour intensive task, but in
some cases, the labels do not need to be quite as accurate.
Weakly-supervised learning can be used in some cases, where it is alright for the labels
to not be completely accurate. One such example, is when using [query + positive docu-
ment + negative document] triples (referred to just as triples for simplicity) for training
IR BERT models. The positive documents do need to be somewhat related to the query,
but it is hard to determine a gold standard. The negative documents should be less related
than the positive, but are usually not guaranteed to be. With this kind of data, it can
be relatively easy to create labels (positive/negative documents), for example using an
algorithm like the BM25 (Manning, Raghavan, and Schütze 2009).
Validation of a ML model can be a bit tricky for unsupervised learning where the correct
result is not known. However, for supervised learning, it is less complicated since the
correct target is known. The precise method used, depends on the model and the data,
but usually the models have the training/validation/test-split in common. The idea
behind the data split, is that the training set is used to train the model, the validation set
is used to for training/choosing model hyperparameters, and the test set is used to measure
how good the final model is. The reason for the different splits, is to avoid over-fitting to

2.2
Neural Networks
In Machine Learning there are many different models, and the metaphor ”The right tool
for the right job” rightly applies. However, lately Neural Networks has become like a
Swiss Army Knife, a tool that can solve most tasks, given enough data. The success of
neural networks originates from their many optimisable parameters and complicated ar-
chitecture. But the many parameters requires a lot of data to train well, which is not
always available.
This section will go over the topics related to neural networks and deep learning. First
regression will be presented in its standard format, and weights and biases will be intro-
duced. Then using these building blocks, a standard neural network architecture will be
introduced. With the network, the forward pass will be presented, followed by a brief
presentation on activation functions and backpropagation. Loss will then be discussed in
the domain of neural networks, and lastly, optimisation will be presented.
2.2.1
Regression
A neural network is basically a complicated regression model, so it makes sense to look into
a standard regression model first. A simple regression example has already been presented
in section 2.1, but a more thorough description will be presented here. Regression is a
statistical method that aims to describe data with a function. In a platonic view, we
assume there must exit some function/relationship that is behind the creation of the data,
and it is exactly this function regression tries to estimate. In example 1, the model is a
linear regression model, and the data is randomly generated using a linear function with
some noise added through a Gaussian distribution.
y = x + 15 + N(0, 15)

briefly, the loss of a regression model can be considered as the distance between the real
data point and the estimated model value at that point, as illustrated on Figure 2. The

That is where neural networks come in. When speaking neural networks, the coeﬀicients
are called weights, and the intercepts are called biases.

2.2.5

2.2.6
Optimisation
Using all of the above methods, it is possible to optimise the neural network. In short:
given input data, make a forward pass, calculate the loss, compute the gradient, do back-
propagation to change the weights and biases, and repeat.

Natural Language Processing
Humans communicate through language, which can be written of spoken. Natural Lan-
guage Processing (NLP) is a sub-field of AI where the goal is for a model to understand
certain properties of human language. Some of these properties can be for example: knowl-
edge, meaning, or sentiment. In human language, we usually talk about letters, words,
and sentences. A computer takes some different building blocks. In the following section,
we will first go over some NLP basics like tokens, stemming and lemmatisation, and vec-
tor space representations, followed by a dive into some popular NLP models. Afterwards,
OpenQA will be presented, and lastly, we will touch upon how medical domain data differs
from normal language data.
3.1
NLP preprocessing
3.1.1
Tokenisation
Instead of words and letters, an NLP model sees Tokens. Tokens can be words, but can
also be symbols, or numbers, etc.
There is no golden truth to what a token is.
The
method of creating tokens from a string, is called tokenisation. The goal of tokenisation,
is to cluster the different parts of a sentence together such that every token carries a
meaning from the sentence. As an example, consider the sentence:
Mr. Hansen’s car is blue
The golden tokenisation here, would be the following split:
["Mr.","Hansen","'s","car","is","blue"]
Here the meaning is carried into the tokens. ”Mr.” is a title which puts a sex to the person
”Hansen”. ”’s” itself is a genitive article that is important to know that the following noun
is something belonging to Mr. Hansen. The meanings of the rest of the tokens should
be fairly simple. This example is fairly easy to understand, but what if we have another
example:
It’s a good day to go skiing!
If we use the same tokeniser as above, we would once again get a token ”’s”, but this
time the meaning is very different, and only makes sense together with ”It”. So a valid
2https://pytorch.org/
10

tokenisation here, would be to take ”It’s” all together. The point is, different ways to
tokenise a sentence, might lead to different results.
After having tokenised a sentence and having created the tokens, a common operation is
to uncapitalise all letters. In a computer, Q̸=q, but in language, using capitalised letters
are more of a formality that makes it easier to read the sentence, rather than it actually
changing the meaning. Tokens that are uncapitalised are also referred to as being uncased.
3.1.2
Lemmatisation and stemming
In the second example above, the word ”skiing” was used. It is a combination of the verb
”to ski” and the suﬀix ”-ing”, which controls the grammatical tense. The tense might not
be important for the model, and therefore there exists methods to remove suﬀixes etc.
Stemming simply removes suﬀixes (etc.) in order to quite easily get the stem of the word.
In the ski example, both ”ski” and ”skiing” would become just ”ski” after stemming, and
would be considered the same word by a computer model. However, some words change
structure when they change tense, such as ”swim”, ”swimming”, and ”swam”.
Using
stemming here, would leave us with ”swim”, ”swim”, ”swam”, which would be considered
two different words. If we want the words to be the same, we could use lemmatisation.
Lemmatisation tries to transform a word into its lemma, which is different from its stem.
The lemma of ”swimming” and ”swam” are both ”swim”. However, lemmatisation is a
much more complicated operation than stemming. A simple stemming algorithm could be
to just remove a certain string, if that string is found at the end of a word. A lemmatisation
model needs to know the lemma of every word, making it a much more complicated model.
One of the advantages of DNNs in NLP, is that the model itself learns the meaning of
the words, applying that the model knows implicitly that the words ”swim”, ”swimming”,
and ”swam”, are strongly related, and must have approximately the same meaning.

Information Retrieval and Open Domain Question Answering
Most commonly known as a Search Engine, Open-domain Question Answer (OpenQA) are

any prior context, which means that to find the answer, it first has to retrieve the context.
The retrieval is done with an Information Retrievel (IR) model, which in itself can be a
search engine, and the answer is then found with a Reader model, based on the retrieved
passages. The most popular search engine is Google3, so to illustrate the parts mentioned,

competition4.
3.2.2
Reading
Given a list of relevant passages, for example retrieved by an IR model, a reader model
will find an answer to the query (question). Depending on the kind of reader, it will either
find the answer string (passage) directly in the documents (Chen et al. 2017), or it will
generate the answer word by word from the implicit knowledge either located within the
reader it self (Raffel et al. 2020), located in the reader and in the retrieved context (Izacard
and Grave 2021).

3.3.2
RNN
A Recurrent Neural Network (RNN) is a neural network that in the context of NLP,
considers not only the present token, but puts it into context of its preceding tokens as
well. This procedure creates a kind of memory, which makes an RNN good at learning
how to predict the next word in a sentence for example. See Figure 8 for an illustration
of an RNN.
Figure 8: An illustration of a Recurrent Neural Network (RNN).
Figure borrowed from https://aditi-mittal.medium.com/understanding-rnn-and-lstm-f7cdf6dfc14e
One of the major shortcomings of RNNs, is that the more preceding tokens it has to
remember, the larger the model becomes, which in turn leads to vanishing or exploding
gradients, when doing backpropagation. In order to counter this problem, a new kind of
network was invented.
Long short-term memory networks (LSTM) are a type of RNN that can remember longer
sequences because they only keep the ’important’ contexts that are needed to solve the
problem, in memory. This means that even though a sentence might be very long, the
network knows which terms are important and keeps those in memory, and ’forgets’ the
rest, effectively eliminating the thread of vanishing or exploding gradients.
3.3.3
CNN
Convulutional Neural Networks (CNN) are best known in the context of image classifica-
tion, but have also been used in NLP. Recall that a token can be represented as a vector,
so if we want to represent a sentence, simply stack all token vector representation on top
of each other to form a 2D matrix. Similarly to images, which are also represented as 2D
matrices, a CNN can then run over the sentence 2D matrix and for example do sentiment
classification. One of the shortcomings of CNNs in NLP, is that CNNs learn local struc-
ture well. If there is an edge in an image, then it is found in adjacent pixels. However, in
sentences, there are often no such local structure. As an example, the two words in the
sentence ”The car” plays exactly the same role as they do in ”The blue expensive car”,
but the words are no longer adjacent.
To enable a neural network to focus only on the important parts of a sentence, eliminating
vanishing or exploding gradients, bringing down model complexity, and focus only on what
is important rather than local structure, transformers where developed.
3.4
Attention
From our spoken and written language, we know that some words of a sentence are much
more important than others, depending on the context. Take for example the sentence:
The girl ran fast because she was in a hurry

”The girl” and ”she” are referring to the same person, so if we consider just the word ”she”,
then ”girl” or ”the girl” should be more important than for example ”because”. Likewise,
”fast” is an adverbium that explains something about ”ran”, meaning that ”ran” should

3.5
BERT
In attention, the encoder itself learns many things about language when it is trained,
which it needs for better encoding. What if we could get this information? The Deep
Bidirectional Transformers for Language Understanding (BERT) is introduced in (Devlin
et al. 2019), and works exactly with this idea.
BERT can solve many tasks such as
translation, sentiment analysis, and of course, IR and QA. But in order to train the
BERT model, two steps are used: pretraining to understand language, and fine-tuning to
solve a specific task.
The pretraining is done using a Masked Language Model (MLM) and Next Sentence Pre-
diction (NSP). The MLM takes a sentence and masks some of the words, which it then
has to predict:

The task here is to predict the language structural word [MASK1] = ”the”, and the
knowledge token [MASK2] = ”Biden”. In NSP the encoder takes in two sentences and

4.1.2
Retrieval
The late interaction mechanism allows ColBERT to pre-compute all of the document
representations offline when used for retrieval, such that given a query, ColBERT encodes
the query representation, and finds the maximum similarity between it and all documents.
It does not actually compute the similarity to all of the documents in the corpus due to
FAISS, but more on that momentarily.
The indexing procedures runs through batches of documents from the corpus. It takes
a large set of documents at once, and splits them into batches such that the documents
in each batch have approximately the same length, and all documents within a batch,
are padded to the maximum document length within that batch. The batches are then
indexed over multiple GPUs (if available). The authors found that a non-trivial portion
of indexing time is spent on BERTs tokenisation, which is a CPU heavy but independent
operation, and is therefore distributed to available CPU cores for shorter wall time. To
keep the memory footprint down, the representations are saved using 32-bit or 16-bit
values to represent each dimension.
For ColBERT’s reranking procedure, the indexed documents are simply loaded into mem-
ory, and the top K (e.g. K=1000) documents are then scored and sorted. However, for

end-to-end retrieval, ColBERT utilises faiss (Johnson, Douze, and Jégou 2017), which
in short is an eﬀicient way to store and collect indexes. The main indexing features of
faiss includes the IVFPQ (inverted file with product quantisation), which clusters the
representations into partitions using a k-means algorithm, which it then uses for a prun-
ing algorithm during vector-similarity scoring. Using these features, it is possible to look
through a very large corpus of text based data (for example) in a short amount of time.
Due to the pruning, some accuracy is lost, but ColBERT raises the accuracy again by
retrieving more documents than needed, and reranking those.
ColBERT uses a two-stage approach when indexing with faiss to retreive the top-k
documents to a query q. The first stage is for filtering the document representations to
find approximate matches, while the second is a refinement stage.
Issuing Nq vector-
similarity queries corresponding to each of the embeddings of Eq, onto the faiss index,
retrieves the top-k’ (k’ = k/2) document embeddings for that query embedding. Each
of the document embeddings gets mapped to the ID of their document origin, producing
Nq × k′ document IDs, where K ≤Nq × k′ are unique. These K documents likely contain
one or more embeddings which are highly similar to the query embedding, and in the
second stage, these K documents are reranked using the reranking procedure to find the
most similar documents.
In summary, ColBERT achieves state-of-the-art results similar to those of BERT, while
using only using 7B flops per query for the reranking procedure, compared to the BERT
base that uses 97T flops (13.9k more flops).

Medical domain

UMLS
The Unified Medical Language System (UMLS)(Bodenreider 2004) is the main classifi-

classify which observations are disorders, and which are something else. In order to use
it, every disorder is classified via Concept Unique Identifiers (CUI). Each unique disorder
has one or more unique CUIs, and each CUI number is linked to some TUI in the UMLS.
Furthermore, a dataset containing semantic groups is then used to connect TUIs with
categories, e.g. a disorder if the CUI in question is a disease. Figure 15 shows the UMLS
data with semantic groups. All of the shown CUIs are disorders under the TUI: T048,

5.1.2
MetaMap
To connect the queries and answers from MedQA (see 5.3) with CUIs, the MetaMap Java
REST API5 is used. MetaMap has been developed at the National Library of Medicine
(NLM) to map biomedical text data to the UMLS. In addition to mapping the CUIs, it
also has mappings between terms, effectively creating a synonym mapping, which is useful
since a disease may have more than one name. The term names and CUIs are found using
the MetaMap Indexing (MMI) ranking function. The ranking function is the product of a
frequency factor and a relevance factor, which is a weighted average of four components:
a MeSH tree depth factor, a word length factor, a character count factor, and a MetaMap
score factor. The actual use of MetaMap on the MedQA queries, has been done by my
fellow students Andreas Motzfeldt Jensen and Benjamin Starostka.

5.5.1
Splitting the articles into smaller passages
To hopefully avoid confusion, an article in the this section refers to either one of the
MedQA textbook snippets or one of the FZ articles. An article might be quite long and
span over thousands of words, where some parts might be a lot more relevant to a query
than others, which means that even if (in the case of FZ) a CUI identifies an article as
relevant to the query, the majority might still be unrelevant and not help answer the
question or find the disease. I propose splitting each article into several passages, and
use the split passages to train and test the model. In the case of training, the model
then receives one highly relevant short passage instead of a large and more general one.
For testing, I hypothesise that a short highly relevant passage is better for classifying the
correct multiple choice answer as well.
To make these splits, I propose to use NLTK’s sentence tokeniser, which splits each article
into sentences. I then combine neighbouring sentences such that each passage contains
two sentences. This is done in order to avoid losing too much important context. Consider
the following general example of a text where both sentences are needed to understand
the context:
The man just won the lottery. He had never been so happy before.
The number of sentences to consider in each passage, is ultimately a hyperparameter. I
tried looking at passages including 1,2, and 3 sentences, but felt 1 sentence was too short,
and 3 sentences started to include unrelevant data. I have not formerly tested it, but it
might be a case to look into.
5.5.2
Creating training triples
There are several ways to create training triples, and there is uncertainty about how much
work is needed to be put into it. Recall, a training triple is of the form: query, positive
passage, negative passage. In this project, the queries come from the MedQA data. The
positive passages are usually found using some sort of ranking algorithm, for example the
BM25.
Since the retrieval corpi are large, it is a very heavy task to rank all of the documents
with BM25. Following the intuition mentioned in (Lee, Chang, and Toutanova 2019),
a positive passage should include the query answer. With this intuition, I suggest first
filtering the corpus to only include passages that contain the answer, and then afterwards
ranking the passages using BM25. Now, there is a trade-off between quality and quantity.
We want to include the most relevant passages, and as many as possible. But how do we
decide just how many are actually relevant enough to be a good candidate for positive
passages in triples? Since the BM25 score depends on the query, I decided to include
all passages with a score over the mean score between the passages and that particular
query. Furthermore, if this filtering results in more than 25 passages, I cut it off to only
include the top 25. This is done in order to not overfit to specific queries. Some queries
return hundreds of passages, while others return less than 5, so setting a max makes the
number of passages more evenly distributed between the queries. The code for retrieving
and ranking positives wont be shown here because of it is quite long and does some other
things too. It follows much the same process as the one for finding negatives, which can
be seen below. Everything is written in the script create_triples_from_corpus.py.
The negative passages are harder to determine.
As mentioned in (Qu et al. 2021), it
can be diﬀicult to be sure that the negatives are in fact not false negatives. Intuitively,
picking a random article from a large dataset has a high probability of being a negative,
there is a chance that it is positive. I propose an approach that should at least be better
than choosing at random, while still being fairly fast. First, choose a random sample of

5 articles. Secondly, using the intuition from the positive passages, filter out the articles
that might contains the answer (if any). Thirdly, rank the passages with BM25. Now we
could choose the passage with the lowest score, however, especially the MedQA textbooks

def get_negative(q,a,df):
"""
Query: q
Answer: a
Dataframe with passages (article): df
"""
got_one = False
while got_one == False:
docs = list(df["article"].sample(n=5))
# Exclude passages with the answer
docs = [x for x in docs if a not in x]
if len(docs) >= 1:
tokenised_corpus = [word_tokenize(x) for x in docs]
qt = word_tokenize(q)
bm25 = BM25Okapi(tokenised_corpus)
doc_scores = bm25.get_scores(qt)
doc_scores = np.array(doc_scores)
doc_scores[doc_scores.argmin()] = doc_scores.max()
doc_scores[doc_scores.argmin()] = doc_scores.max()
# This index is now the index of the mean value (if len(docs) = 5)
top = doc_scores.argmin()
return docs[top]

6.1
Evaluation methods
To accurately use ColBERT for rare disease retrieval, it needs to be fine-tuned. In order
to find the best model, several experiments have been created. To benchmark and test
the models’ performance, divers tests have been applied.
6.1.1
Validation on MS MARCO dev
First, ColBERT trained on general data is tested using the reranking task on the MS
MARCO dev top 1000 dataset, which makes the benchmark comparable to that of the
authors (Khattab and Zaharia 2020). In this reranking task, the correct order of the top
1000 passages for each query is known, so it is easy to check if ColBERT has ranked cor-
rectly. This first test is used as a sanity check to make sure the model performs as expected.
6.1.2
Multiple Choice task using MedQA queries and asnwers
Secondly, models trained on medical data are supposed to perform well on a medical
domain, not on a general domain as in the MS MARCO reranking task. Therefore, they
need to be tested on another task. For this purpose, I utilise the MedQA (Zhang et al.
2018) test dataset for queries + answer. We do not know the correct document ranking
as we did in the validation test. Recall, the MedQA data is built upon a multiple task
test where we have a query, a correct answer, and 5 possible options to choose from. This
framework can be used to evaluate the model, but ColBERT is not set up for that.
In order to solve the MC task, I have made an implementation using ColBERT’s reranking
approach. The input is all of the retrieved passages to the queries (1000 for each query),
connected with each of the answer options (the correct answer always as the first one).
This data structure of sextuples is just like the triples, but with each passage acting as a
query, and the 5 options acting as the passages. ColBERT then ranks each of the options
up against the passage. The result is 1000 · 5 scores to each query.
As we saw in 5.3 and 5.4, some queries / answers only have a few relevant passages that
can be retrieved. So if the model performs well and ranks all of the relevant passages in the
beginning, I hypothesise that the model should perform better using the top 10 documents,
than when using all of the top 1000. The model is therefore benchmarked on the top N =
[1, 3, 5, 10, 50, 100, 500, 1000] retrieved documents. Furthermore, two approaches are used:
first the majority vote which predicts the option most often predicted using the top N
documents, and second, a prediction made on the highest overall score over the top N
documents. The first approach assumes that each passage is related to only one option,
and hence only the top scored option is relevant. The second approach values that each
option might be related to a passage. For instance, if the correct answer is always rated
second highest for the top N documents, then the first approach will never predict it, but
the second might. The two approaches are summarised below.
The benchmark is made for the top N = [1, 3, 5, 10, 50, 100, 500, 1000] retrieved documents.
1. Majority vote: The final prediction is made as the most predicted answer to the top
N documents.
2. Highest overall score: The final prediction is made as the options with the highest
aggregated score over the top N documents.

6.1.4
Evaluating on FindZebra corpus with MC
Thirdly, the purpose is to use the model on the FindZebra corpus for rare disease retrieval,

data on rare and orphan diseases, so not all of the MedQA queries can be answered using

FZ corpus, are used, resulting in 394 out of the 611 test queries. The MC task is then
used for evaluation here as well.
6.1.5
Random guessing benchmark
A random guessing approach has been implemented, which just guesses for one of the five
options, for each of the test queries. The test has been has been run for 10000 iterations
to find the percentiles of each guess on the two different tasks.
Figure 24 shows that randomly guessing on the MedQA MC tasks predicts each option
between 15% (5% percentile) and 25% (95% percentile) of the time on the MedQA task,
and between 14.5% (5% percentile) and 25.5% (95% percentile) of the time on the FindZe-
bra task. Both the tasks have a median of 20% as expected, given that there are 5 options
to each query. In summary, randomly guessing on the MedQA evaluation task results
in a mean value of 0.2 and a spread of 0.05, while the FindZebra task which has fewer
queries, has a mean value of 0.2 and a spread of 0.055. This random guessing benchmark is
interesting to consider, to see if the models perform significantly better than just guessing.
6.1.6
Evaluating on FindZebra, only retrievals
IR and QA are quite different tasks, and if a model does not perform well on a QA task,
it does not necessarily mean that the retriever does not perform well. The goal is after
all to build an eﬀicient retriever using ColBERT, so it is relevant to figure out how well it
performs alone. For this, the MRR and MCRR scores are applied.
To evaluate the retriever models directly without a reader module, the standard method
Mean Reciprocal Rank (MRR) is used. But to use the MRR method, we first need some
way to determine which documents are correct, and which are not. Recall, the majority of
the documents in the FindZebra corpus contain a unique identifier in the form of a CUI.
The CUI for each document, identifies which disease the document is about. It makes it
possible to check whether the MedQA answer CUI corresponds to a CUI in the retrieved
documents, and if so, then the document is deemed relevant. Due to the sorting of the
MedQA queries on the FindZebra evaluation (contains only the queries with answers that
are found in FZ), we know that there is always at least 1 relevant article. As a baseline,

6.6.2
Multiple Choice end-to-end training
When training with triples, there is an underlying statement in that the triples have been
created accurately, though we have no way to actually test that, short of getting a medical
professional to go through the dataset. However, using the MC task to train the model
end-to-end, we know that if the the model chooses the correct answer, then the model
must be doing something right. I am still using the triples because I want the model to
get a feeling about what is a relevant passage, and what is an unrelevant passage, to each
query. In summary, the task is both to train on triples, but it also has to classify the
correct answer between between the 5 MedQA options. For this purpose, I use the MC
triples dataset structure presented in section 5.5.3.
I have created the ColBERT version ColBERT_MC that, using the MC triples, creates 5
queries, each with one of the options appended to it, and 5 positive + negative pairs. This
could probably be done more eﬀiciently and with a smaller memory footprint, but this
structure allows us to use the ColBERT implementation with minumum changes. Instead
of scoring one ⟨q, d+, d−⟩triple, it scores 5 such triples: ⟨q + opti, d+, d−⟩where opti is
one of the 5 options. Using standard ColBERT scoring that determines that the positive
passage is correct while the negative is false, so are the 5 triples scored. However, in the
MC task, only the target answer (in this case the first option) is correct, so out of the 10
Using ColBERT for information retrieval on the FindZebra rare disease domain
37
==PAGE:46==
scores found with the MaxSim operator, only the first is considered correct.
An alternative way to implement the task, would be to find the MaxSim of every ⟨q +
opti, d+, d−⟩triple and find the cross entropy loss as normal, and then find the cross entropy
loss of all the 5 triples. I have not looked further into this method, since I hypothesise that
the two methods should be equal in performance, though the this method could result in
a lower memory footprint - which my implementation do struggle with, resulting in small
batch sizes.
In the regular triples task, we know that the positive passage always contains the correct
answer. For the models that learn via only triples, it results in the models at least being
able to learn a similar embedding between the given query and the answer. In this task,
using the same triples, it might result in a downfall. Both the query with the correct
answer and the positive passage contains the answer, so ColBERT might overfit because
it learns to look for the last query in the passages.

7.1
Max length of query
As mentioned in section 4.1, each query is truncated down to the maximum query length
if it is longer than that. At first, I had not thought about it, so I trained model 1-8 with
the standard query size of 32. However, the average query length of the MedQA queries
is around 132 tokens and follows the distribution seen on figure 31.
I tried training the models again using a maximum query length of 250 (corresponding
to the 98th percentile). What I found, was that the test performance got consistently
around 1-2% worse. I find it a bit surprising, since a maximum length of 32 cuts off a lot
of the information, but then again, the pretrained models have both been trained on a
max length of 32, which can imply that the models performs well on the first 32 tokens,
but adding tokens means that the models now try and re-learn the parameters with this
new token length. If I were to redo anything in this regard, I would like to try and use
MetaMap to extract the medical terms, and re-create new queries using these terms, and
hopefully reach a query size of less than 32 tokens.
Note: In the case of Model9 and Model10 which are trained on the MC end-to-end task,
the queries are concatenated with the options, meaning that it is essential that they are
long enough. Therefore they have been trained with a maximum query size of 250.
