Abstract 
With the rapid adoption of 4G technologies in the mobile networks there 
has been an unprecedented increase in the bandwidth and number of con-
nections that the mobile backhaul has to cope with. Because of the ever 
increasing importance of the mobile networks to everyday life, it is cru-
cial to provide with the means that will help alleviate some of this in-
creased stress in the mobile backhaul. 
This thesis provides with an SDN based traffic aggregation algorithm for 
the Floodlight OpenFlow 1.3 controller, which performs dynamic aggre-
gation of traffic flows based on path and QoS criteria. By performing 
traffic aggregation the controller manages a substantial reduction in the 
number of flow entries that are required to be stored in the network’s 
switches. While at the same time preserves the performance of the net-
work in terms of connectivity and delay. The documented reduction in 
flow entries, ranges between 15% and 40% depending on the situation.

1.1 
In depth problem analysis 
The scope of this section is to provide with a detailed analysis of the problem that this thesis 
tries to address. 
1.1.1 TCAM related memory limitations 
For a network device to properly process incoming traffic, a number of entries has to be 
stored in what is called the routing/forwarding table of the device. Each of these entries is a 
number of criteria (that each incoming packet is checked against), paired with one or more 
actions. When a packet arrives in a network device it is checked against these entries until a 
match is found (the criteria in the entry match some aspects of the packet itself), when a 
match is found the set of action(s) that are paired to this entry is performed. To avoid any 
overlapping between different entries there are two main safeguards. If two or more entries 
match the same packet, then the one with the longest set of matching criteria is followed. Fi-
nally in the case this is also equal then the entry with the highest priority assign to it, is fol-
lowed.  
1.1.1.1 
The Ternary Content Addressable Memory 
In modern switches these tables are stored in what is called a Ternary Content Addressable 
Memory (TCAM), this type of memory has the benefit of being extremely fast when com-
pared to other memory technologies (e.g. RAM). This is because it is able to search the com-
plete memory space in a single search operation. This means that the search time is independ-
ent from the number of entries that reside within the TCAM (O (1)) [7] 
While this disassociation makes the search time independent of the number of unique traffic 
flows in the network, the same cannot be said for the monetary costs related to the TCAM 
itself. This is because TCAMs are very expensive to both obtain and power and these costs 
are directly proportional to the size of the TCAM [7].So the problem here is that the increase 
in connections in the mobile networks brings also an increase to the number of unique traffic 
flows that pass through the network devices. In order to store the entries required to process 
these flows, the TCAM memory of the network devices will need to get bigger and bigger 
==PAGE:15==
 
15 
 
which will bring an increase in the monetary cost of both setting up and maintaining the mo-
bile network.  
This problem scales even worse when considering that Software Defined Networking (SDN) 
is one of the main candidates to be the main architecture/technology in the future mobile net-
works. This is because the currently used version of OpenFlow (the protocol that enables 
SDN-controller to SDN-switch communication) uses a 15 field tuple (15 distinct criteria) to 
define each traffic flow (for comparison traditional switches use only 2-5 field tuples) [7]. 
This increase in traffic flow granularity inevitable brings an increase in the memory space that 
is required to store each unique entry. 
1.1.2 Complexity of flow management 
Apart from the aforementioned increase in the required memory space (and the monetary cost 
that goes with it), there is also another matter that raises considerable concerns. This is the 
inevitable increase in complexity for managing all these individual flows and their related 
entries. Two main network processes are particularly affected by this increase in the number 
of individual flows, these processes are the management of the buffering and re-routing of 
traffic flows [16]. 
1.1.2.1 
Management of Buffers 
In order to avoid packet loss when the packet arrival rate in a network interface is larger than 
the available bandwidth of this interface, the network devices temporarily store the incoming 
packets into buffers. Then depending on the scheduling algorithm in effect they forward the 
packets when it is appropriate. 
Apart from the obvious problem related to the buffer size itself, the solution to which is out of 
the scope of this thesis, there is also the matter of the increase in the complexity of the sched-
uling algorithm as this usually goes hand-in-hand with the number of unique flows that need 
to be scheduled. Even though there are scheduling algorithms like First Come Fist Served 
(FCFS) and Round Robin (RR) where the complexity of the algorithm is independent of the 
number of scheduled tasks (O (1) complexity), these algorithms are rarely used in networks 
due to their very limited functionality. This is because they cannot make any distinction be-
tween the buffered flows and thus do not allow for differentiated treatment between traffic 
flows with different requirements (the importance of differentiated treatment is discussed later 
in this document in section 1.1.3) [17]. More commonly used algorithms like the Weighted 
Fair Queueing (WFQ) and the Earliest Deadline First (EDF), that allow for differentiated 
treatment, have complexities directly associated with the number of unique buffered flows 
(O(log(n) and O(n) respectively). Even though from a complexity theory perspective these 
complexities are considered good, one has to take into account the sheer amounts of traffic 
==PAGE:16==
 
16 
 
flow entries (72,000 flows per second [7]) that need to be scheduled. If you add to this the 
fact that the processing power of network equipment is limited, it is clear that reducing the 
number of flow entries that have to be scheduled can prove beneficial for the network. 
1.1.2.2 
Traffic rerouting management 
Finally in the case that there is a node and/or link failure in the network all the individual 
flows that currently pass through this link/node will need to be somehow rerouted. This will 
add an overhead in both the network devices’ processing capabilities and to the network traf-
fic related to the control plane as the new routes will have to be re-calculated. This overhead 
is again directly associated with the number of entries that need to be rerouted. By reducing 
the number of flow entries this issue can be mitigated, as less flow entries means less com-
plexity in the rerouting process. 
1.1.3 Enforcing QoS 
It is well established that different network applications have different needs in terms of 
bandwidth, delay, jitter and packet loss. This has brought the need for a Per-Hop-Behavior 
(PHB) in the network devices which would take into account the different needs of the differ-
ent types of network traffic and try to meet them when possible. This need for Per-Hop Be-
havior has established the Differentiated Services (DiffServ) architecture as the status quo in 
all modern networks.  
Because of the undisputed importance of the DiffServ model any solution associated to prob-
lems mentioned in 1.1.1 and 1.1.2, will have to be able to work in conjunction with the 
DiffServ model.  
1.1.3.1 
The DiffServ Architecture Model 
The DiffServ architecture [3] allows for distinct service discrimination in the Internet without 
the need for per-flow state and signaling. To allow for this service discrimination the DiffServ 
architecture uses an 8 bit header field (the DS field [2]) that is inserted to all IP packets (IPv4 
and IPv6). Out of these 8 bits only the first 6 are used, whereas the other two are Currently 
Unused (CU).This 6 bits serve as an identifier and are called the Differentiated Service Code 
Point (DSCP). The structure of the DS field is visualized in Figure 5.

As can be observed the DSCP values distinguish traffic into 3 main classes. The Class Selec-
tor (CS) class, the Assured Forwarding class (AF) [4] and the Expedited Forwarding (EF) 
class [1]. 
The CS1-CS7 classes provide with only inner CS traffic conditioning. The higher the CS class 
number is, the higher the traffic priority. This however does not provide with any assurance in 
how the traffic will be treated in the network. The CS classes can be easily distinguished as 
they must conform to the XXX000 format in the binary representation of their DSCP value. 
By definition the CS0 class provides with absolutely no assurance and is based on best-effort 
==PAGE:19==
 
19 
 
PHB. CSO is considered the default PHB and any traffic that does not conform to the DSCP 
mapping used inside a DS Domain or region is remarked with this DSCP value.  
In contrast with the CS class the AF class can provide with some assurance regarding the 
traffic treatment inside the network. This is done by assigning AF traffic into 4 different inner 
classes AF1X, AF2X, AF3X and AF4X and then reserving resources (buffer size and band-
width) in each network node to be used only by a specific inner class. This however may still 
lead to inner class congestion, if the arrival rate or traffic size of an AF inner class is higher 
than the reserved resources for it. For this case each AF inner class is also divided into 3 sepa-
rate subclasses AFY1, AFY2, and AFY3 where each of these subclasses has a different pack-
et drop priority. 
Finally the EF class provides with complete assurance regarding the traffic treatment in terms 
of quantifiable values of delay, jitter and loss. In order to do so there needs to be ensured that 
on each output interface of each node the service rate of EF packets exceed their arrival rate, 
independent of the arrival of non-EF traffic. 
1.2 
Proposed Solution in a nutshell 
As has been made clear, all the problems that have been analyzed in the previous sections can 
be traced to a single root, which is the ever-increasing number of entries in the rout-
ing/forwarding tables of the network equipment. The scope of this thesis is to propose a via-
ble solution that can effectively reduce the number of entries, without hindering the connec-
tivity and performance of the network. What is proposed is a SDN-based traffic aggregation 
algorithm for the Floodlight [22] OpenFlow controller [13]-[14] 
The designed algorithm aggregates traffic flows based either on their path through the net-
work, or on both their path and their DSCP value. The latter enables the network administra-
tor to perform QoS differentiation between the different aggregates as each of them will only 
contain traffic that requires the same PHB. All the experimental results show that the algo-
rithm is effective at reducing the total number of flow entries in the network. More details on 
how the algorithm operates, how it helps alleviate the aforementioned problems and the re-
sults from the performance experiments are provided in the following chapters of this docu-
ment. 
As the proposed algorithm has been designed to operate within an SDN enabled environment, 
Chapter 2 is devoted to SDN. It provides with both a theoretical background of SDN and 
OpenFlow (its communication protocol) but also explains why SDN was selected as the main 
technology/architecture for this project.

1.3 
Diversions from the Initial Planning 
As with any project of this scale, during the implementation phase there is usually a level of 
diversion from the initial planning. The reason for this is that the implementation phase brings 
out either previously unpredicted issues/limitations or highlights new paths that are more 
suitable and have not been considered before.  
1.3.1 Non-Developed functionalities 
The initial plan was for the aggregation algorithm to be specific only for the mobile backhaul 
networks. To do so the planning was to handle the signaling of the mobile backhaul network 
(X2 and S1 interfaces), separately from the rest of the traffic. But due to the fact that the con-
troller that was selected to implement the algorithm was not able to handle this type of proto-
cols, this idea had to be abandoned. This has made the algorithm more generic, but because 
4G backhaul networks are primarily IP networks, this generic fashion does not hinder the 
functionality of the algorithm. It means though that the algorithm can also operate in any IP 
based network. 
1.3.2 New paths taken 
1.3.2.1 
From an event-triggered to an always on approach 
At first the idea was to create an event triggered aggregation algorithm. The algorithm was 
supposed to be idle at its default state and only be enabled when one or more switches 
reached a predefined limit in their flow entries. This was supposed to create localized aggre-
gation “islands”, thus minimizing the disruption to the network. But while developing the 
algorithm this approach was abandoned    because of the following reasons. 
The initial approach required a secondary Floodlight module to continuously poll all the 
switches in the network. This was because of the need to get the current number of flow en-
tries that they have in their tables. This approach added additional complexity to the project 
and also overhead traffic to the OpenFlow channel. At first it was thought that performing 
aggregation only when and where it was needed, was overcoming these complexity and traf-
fic overheads, but during the implementation it was revealed that aggregation could be an 
always enabled function without hindering the networks performance. 
1.3.2.2 
From a two-module design to a unified one 
As has been stated before the traffic aggregation is performed in one of two modes, one with 
path and DSCP based QoS criteria and one with only path criteria. The initial though was to 
have two separate modules each of which would be responsible for one of these modes. But 
very early in the implementation phase it was made clear that this approach would only add 
==PAGE:21==
 
21 
 
complexity to the design without offering any functionality whatsoever. So in return it was 
decided to create one Floodlight module that can operate in both of these modes.  
More details about the initial planning of the project can be found in Appendix B. There a 
rejected manuscript is provided, which outlines the initial design of the algorithm. The manu-
script was rejected publication mainly due to a lack of experimental results, which were not 
available at the time of submission. The current plan is to submit the current state of work in 
another journal, as now it includes the complete design of the algorithm together with promis-
ing results. 
1.4 
Related Works 
There has been some research regarding the problem that this thesis addresses. This section 
will provide with short reviews about these research attempts, focusing on what they propose 
and how that resembles and /or differs from the proposal of this thesis. 
 In [8], a per-port prediction circuitry is proposed that enables SDN switches, in some cases, 
to forward packets immediately after they have been received without the need to access the 
TCAM and perform a lookup.  The circuitry snoops constantly on the memory bus between 
the port interface and the memory and creates a packet signature for each incoming packet, 
while this packet is still in transmission. Then it accesses the per-port local cache and search-
es for a matching signature, if a single match is found then the packet is immediately redi-
rected to the associated output port. This effectively bypasses the TCAM and as such has both 
power and latency related benefits.  However this approach requires extensive hardware 
changes to the design of the SDN enabled switches and so is hard to implement as it requires 
for the hardware vendors to adopt this design. 
In [7] a two tag system is proposed to reduce the bit-size required to store flow entries. These 
two tags are a Path and a Flow tag. The Path Tag (PT) is used to identify the network path 
that the packet has to pass through, while the Flow Tag (FT) is used to identify the individual 
flows and provide them with any Differentiated treatment in a PHB basis. In order to add 
these two tags to the packet header, the PUSH action of the OpenFlow protocol is used. Even 
though this solution does not perform traffic aggregation, it is still a very similar approach 
with the solution proposed by this thesis. The main difference between the two approaches is 
that the one described in [7], reduces the size required to store each flow entry while the one 
described in the current document focuses on reducing the number of flows and not their size.

The SDN architecture and the 
OpenFlow Protocol 
The scope of this chapter is to provide with the theoretical background of the SDN architec-
ture [13] and the OpenFlow protocol [14]. 
2.1 
Introduction to SDN 
Software Defined Networking is an emerging network architecture. The main difference be-
tween traditional networks and SDN is that in SDN the control and data planes are decoupled. 
The control plane (which decides how traffic should be handled) is moved to a centralized 
entity called the controller. But the data plane (which forwards traffic based on instruction 
from the control plane) is still kept inside each individual switch (just as it does in traditional 
networks).  
The aforementioned controller is a server running a specialized software that is responsible 
for enabling the communication with the SDN enabled switches. This change in the network’s 
philosophy has a very big impact in the networks behavior, operation and capabilities. 
2.1.1 SDN related benefits  
Due to the programmable and centralized philosophy of SDN, there are some inherent bene-
fits that emerge and which are discussed below. 
2.1.1.1 
Easier network management and vendor independence 
 In traditional networks even trivial network management tasks (e.g. creating a new firewall 
policy), require a lot of time and effort. This is because all the network devices usually have 
to be manually and independently configured by the network administrator. Even though 
there are some tools that allow for centralized administration, these are vendor specific and 
cannot operate in networks with multi-vendor equipment, which is usually the case for most 
networks.  
With SDN on the other hand, network management is completely centralized and vendor in-
dependent as it only requires a connection to the controller (physical or remote) which is usu-
ally an Open Source software, based on open standard protocol(s). 
==PAGE:24==
 
24 
 
2.1.1.2 
Generalization of network equipment 
The norm in traditional networks is that depending on the use case, the administrator needs to 
obtain a number of different network devices, each built for a specific operation. For example 
in order to build even the simplest business network, the network administrator will have to 
obtain at least a number of routers and switches to allow for the packets to reach their destina-
tion, some firewalls to protect the sensitive parts of the network from unauthorized access and 
finally some Network Address Translators to enable the private network to communicate with 
the Internet (by translating local IP addresses to public ones). This status quo puts limitations 
when considering the versatility and scalability of the network and also the ability to redesign 
the network if need be. This is because a change in the network’s design may require a com-
pletely different set of network equipment, while on the same time making some of the cur-
rently used equipment obsolete. This makes network administrators reluctant in making 
changes, which may in return limit the network’s performance and security.  
In total contrast with this philosophy, in a SDN architecture there is no inherent difference 
between the different network equipment. Every SDN enabled network device is an “SDN 
switch” and depending on how the SDN controller has been programmed it can behave either 
as a switch, a router, a firewall etc. or any combination of these devices. This means that in 
SDN there are no limitations in network redesign and scalability as long as the hardware of 
the device can handle the traffic passing through it. 
2.1.1.3 
Programmability and innovation  
As previously mentioned because of the programmable fashion of the SDN controller, a net-
work administrator is able to make drastic changes to the networks behavior. In addition to 
generalizing the network equipment and thus its behavior, the programmability of the SDN 
controller also allows for extended innovation in traffic management, opening new roads in 
what is currently possible. This element of the SDN architecture was especially utilized for 
the implementation of this project, as it provided with the “tools” required to create a previ-
ously unimplemented network service. 
2.1.1.4 
Increased granularity in packet management 
Traditional network devices depending on their type can usually only distinguish network 
packets based on a limited number of criteria. For example source and destination MAC ad-
dresses for switches, and source and destination IP addresses and TCP/UDP ports for routers 
and firewalls. This is not the case however for SDN enabled switches, as every OpenFlow 
based SDN enabled switch is able to check 13 different packet headers. This means that SDN 
allows for a significantly higher granularity in packet filtering, and us such enabling for the 
creation of traffic rules with higher functionality and security.

25 
 
2.2 
Basic concepts of SDN and the SDN architecture 
As has been previously stated, SDN in nothing more than a new network architecture and us 
such the main concepts that need to be defined are the network elements that belong to this 
architecture and the protocols used for communication between these different elements. The 
SDN architecture has two distinct types of network elements, the SDN enabled switches and 
the SDN controller(s). Figure 7 illustrates how these network elements are connected to form 
the SDN architecture.

These devices are responsible for handling the packets that pass through them (data plane) 
based on the rules that are installed in their TCAM memory. This memory is divided into 
flow tables (these are the equivalent of routing/forwarding tables in traditional network devic-

2.2.2 SDN controller(s) 
This is the network entity responsible for managing the control plane of the network. Because 
of its centralized location (logically and in some cases also physically), it has a global view of 
the network’s topology and state at all times. This allows the controller to make fast and intel-
ligent decisions without the need for a complex distributed protocol, as is the case in tradi-
tional networks (with protocols like RIP, RIPv2 OSPF, BGP etc). The controller receives 
queries from the SDN enabled switches which ask the controller on how a specific packet 
should be treated. The controller will then send back replies to the switch-(es), which will 
contain the match and action tuple that will be installed in their flow tables. 
2.3 
The need for an SDN communication protocol 
As previously mentioned SDN promotes the separation of the control and data network planes 
by centralizing the control plane inside the SDN controller and keeping the data plane in the 
individual switches. This separation though brought the need for a new communication proto-
col, which would allow the controller to have secure and reliable bidirectional communication 
with every SDN enabled switch in the network.  
2.3.1 OpenFlow 
There were a few attempts to design a protocol within this scope, but the most accepted is the 
OpenFlow protocol. It was released for the first time in 2008 with OpenFlow v0.2.0. The first 
major release was OpenFlow v1.0 that was released in December 2009. The current version is 
OpenFlow v 1.4, which was released in October 2013. Because though there is still no con-
troller designed around OpenFlow v1.4 the most widely used OpenFlow version is OpenFlow 
v1.3 that was released in April 2012. From now and on any referral to the OpenFlow protocol 
will have the OpenFlow v1.3 in mind, as it was the version used to implement the algorithm 
discussed in this document.

2.4.2 OpenFlow actions 
An OpenFlow action is the instruction that the switch will follow to process an incoming 
packet that has just met the criteria of an OpenFlow match. An OpenFlow match can have a 
number of different OpenFlow actions paired with it, which are all enforced on the incoming 
packet in the order with which they have been installed by the controller.  
The following actions are currently supported by OpenFlow 
 
Output: Forwards the packet to a specific port.  
 
Set-Queue: Add the packet to a specific queue, this queue stores the packet if the 
specified output port is busy. 
 
Drop: Drop the packet. The drop action is denoted by not specifying an output port 
and not with a separately defined drop action. 
 
Group: Process the packet through a specified group. 
 
Push-Pop Tag: Add or remove VLAN and/or MPLS tags to the packet. 
 
Set-Field: Change a specific field from the packet’s header. 
 
Change-TTL: Change the IP or MPLS TTL field. 
2.4.3 OFPT_PACKET_IN OpenFlow message 
This message is sent from an SDN enabled switch to the SDN controller. It is sent whenever 
the switch has no information on how to process a specific incoming packet and its purpose is 
to request this information. The aforementioned packet is encapsulated within the 
OFPT_PACKET_IN message. 
2.4.4 OFPT_PACKET_OUT OpenFlow message 
These messages are sent from the SDN controller to SDN enabled switches and they are used 
to install flow entries within the switches’ flow tables. They are either sent as replies to 
OPFT_PACKET_IN messages (reactive operation) or they can be sent without any triggering 
from the SDN switches (proactive operation). These messages contain the match-action tuple 
that will be installed in the flow tables.

Proposed Solution 
In order to address the problems that were investigated in Chapter 1, a module for the Flood-
light SDN controller was designed and implemented. A Floodlight module is a network ser-
vice that operates within the controller itself and can provide with additional functionalities. 
During the implementation of this project there were two major SDN controllers Floodlight 
and OpenDayLight [23], both with support for the OpenFlow v1.3 protocol. For this project 
the Floodlight controller was used, mainly because of better acquaintance gained from previ-
ous SDN related projects. 
This module performs traffic aggregation within the network supervised by the controller, 
based on path and QoS criteria. Two or more traffic flows will be aggregated together in a sub 
path of their corresponding routes if the following criteria are met: 
 
They share at least 3 hops within their routes. 
 
If DSCP traffic differentiation is enabled then the flows must belong to the same 
DSCP group. 
The example of section 3.1 will further facilitate the understanding of the algorithm’s logic 
and operation. 
The aggregation of traffic flows into aggregates is done by tagging traffic with VLAN ids in 
the ingress nodes of the aggregation paths and then forwarding this traffic flows only based 
on their VLAN tag, until they reach their designated egress node at which the VLAN tag is 
finally removed. This aggregation of traffic flows leads inevitably to a reduction in flow en-
tries in all the middle switches of the aggregation paths, and thus to a reduction in the total 
number of flow entries in the network in general. 
This section gives a thorough overview of the Traffic Aggregation module, by analyzing both 
graphically and in text all the sub-processes that take place while this module operates. 
3.1 
General overview 
The main operation of the traffic aggregation module is to aggregate traffic flows that pass 
through the network, based primarily on the path that these flows follow and optionally also 
based on the DSCP value of the incoming packets, if QoS aware aggregation is enabled. So if 
==PAGE:33==
 
33 
 
more than one flows can be aggregated together based on the aforementioned criteria, then the 
traffic aggregation module performs the following actions: 
1. Aggregates these traffic flows together, into traffic aggregates, by tagging their pack-
ets with the same VLAN id in the ingress node of the aggregation path. The module 
uses one (unique) VLAN id per traffic aggregate to ensure that traffic is not mishan-
dled. 
2. Installs forwarding actions in all the middle switches of the aggregation paths, which 
are only based on the VLAN tag of the incoming packets. This means that only one 
forwarding action is required per aggregate per hop. 
3. De-aggregates these aggregates by installing pop VLAN actions in the egress switch 
of the path. And then forwards all the traffic flows independently.

The first action that the module takes, upon receiving an OFPT_PACKET_IN message, is to 
parse it and extract the source and destination IP addresses and the DSCP value of the packet. 
By using the source and destination IP addresses the module can calculate the route that this 
packet should follow within the network. If for any reason a route cannot be computed (e.g. 
no route exists, the destination host is unknown etc) then the Aggregation module terminates 
and passes the packet down the module chain. If a route is found then the corresponding data 
structures are updated. 
The next step is to search if any of the other active routes in the network can qualify to aggre-
gate with. To do so the module iterates through the routes data structure and searches for can-
didates. There are two ways another route can qualify as a candidate:  
1. Share at least 3 hops with the current flow. 
2. A flow can be a candidate if it is already aggregated, prior to the arrival of the current 
flow, and its aggregation path crosses the route of the current flow (again at least 3 hops 
must be shared). This will allow the flow to piggy-back over these aggregation paths and 
as such reduce the number of new aggregation paths that needs to be created. 
The reason that in both cases there must be at least 3 common hops for a flow to be consid-
ered as a candidate is because there needs to be at least one ingress, one middle and one 
egress node for a aggregation path to be constructed. By default the module has set the mini-
mum to 3 hops but it can be changed to any other value (>3). Having this value set higher 
would create fewer but longer aggregation paths in the network. Longer aggregation paths are 
in general more efficient, as the higher the number of middle switches in an aggregation path, 
the higher the reduction in the number of flow entries. It would be of interest to observe how 
the change of this value interferes with the performance of the aggregation process. But due 
to time limitations this has not been yet investigated. 
In order to search for common paths between two routes, the module uses a modified com-
mon substring algorithm [24]. The reason for this is because Floodlight stores routes in String 
format. This algorithm takes the two routes as input and returns the number of shared hops, 
together with the nodes at which this common path starts and finishes. If for any reason a 
candidate cannot be found then the module terminates and passes the control of the packet to 
the Forwarding module of Floodlight. This way the packet will processed as it would if no 
aggregation was enabled. 
After a candidate flow has been found and it is already aggregated, the corresponding VLAN 
id of this aggregate is retrieved and used to tag the current flow. Else the next one available is 
booked and used to tag both the candidate and current flows. The VLAN ids are managed 
through a custom request/release mechanism that ensures that the VLAN ids are used in a 
responsible manner that does not allow for misuse (e.g. two different aggregates using the 
==PAGE:41==
 
41 
 
same VLAN id). By default the VLAN manager has a pool of 1000 VLANs available, but this 
can be easily reprogrammed to allow for more or less VLAN ids. One other reason that a 
VLAN manager had to be used, was that it was important to ensure that the operator of the 
mobile network could “ban” some VLAN ids from being used in traffic aggregation and thus 
use them freely for other purposes. 
After the aggregation candidate, the aggregation path and the VLAN id have all been success-
fully retrieved/computed the next step that the algorithm follows is to install the required flow 
entries in the network’s switches, thus creating the aggregation path. To do so the algorithm 
install the flow entries in a per-switch basis. Here it is important to note that this per-switch 
flow entry installation begins with the egress switch and finishes with the ingress switch of 
the aggregation path. If the controller would instead install the flows first in the ingress 
switch, then this switch would immediately start tagging and forwarding the incoming pack-
ets, but the rest of the switches in the aggregation path would not know how to process these 
packets yet, and thus they would sent OPFT_PACKET_IN messages to the controller. This 
would in return mess the aggregation process as the controller would perceive these as new 
messages and it would try to create aggregation paths for them. So the controller first installs 
the VLAN pop and forward flow entries in the egress switch, then the VLAN based forward-
ing entries in the middle switches and finally the VLAN push and forwarding entries in the 
ingress switch. In the ingress switches’ entries, the controller also adds a FlOW_REMOVED 
flag (see section 3.2.2.2.1 for more information). This flag instructs the switch to send an 
OFPT_FLOW_REMOVED message back to the controller when the corresponding flow entry 
is expired or removed.

the previously identified aggregation path and be aggregated also there. If no candidate is 
found for the first part of the path then the packet will still reach the ingress node of the 
second part with help from the forwarding module of Floodlight. A detailed example of 
this case is described later in section 3.4.5 
3.2.2.2 
Arrival of OFPT_FLOW_REMOVED 
A message of this type can arrive in the controller only if a flow has expired or has been re-
moved, in any of the network’s switches.  The reason the flow has expired or was removed 
(inactivity, hard time out or remote removal) does not concern the aggregation module, so 
there is no distinction made between these different cases. Because of the design of the mod-
ule though, only two types of flow entries trigger an OFPT_FLOW_REMOVED message to 
be sent to the controller when they expire. They can either be IP forwarding flows which have 
been previously installed by the Forwarding module, or aggregation flow entries in ingress 
switches, installed previously by the Traffic Aggregation module itself. 
In order to distinct between an IP and an aggregation flow entry expiration, the controller 
parses the match that is encapsulated inside the OFPT_FLOW_REMOVED message. As the 
Forwarding module always adds the source MAC address as a match field (while the Aggre-
gation module does not), then the existence or not of this field in the 
OFPT_FLOW_REMOVED message is used to distinguish between non-aggregation and ag-
gregation flow entry expirations.  
==PAGE:43==

3.2.2.2.1 Expiration of an Aggregation flow entry 
After identifying successfully that the expired flow entry is an aggregation related one, the 
databases that keep track of the flows in the network have to be updated. Depending on the 
situation, some entries will need to be removed and some VLAN values might need to be 
released if they are no longer in use. A special case that can rise when an aggregation flow 
entry expires is explained in section 3.3.3. 
3.2.2.2.2 Expiration of an IP flow entry 
By default the flow entries that are installed by the Forwarding module do not trigger 
OFPT_FLOW_REMOVED messages, so some minor changes had to be made to the default 
state of Floodlight to enable this option as it was crucial for the functionality of the Traffic 
Aggregation module. That is because the reception of OFPT_FLOW_REMOVED messages, 
is the only way the controller can keep an up-to-date mapping of all the flows in the network, 
without the need for continuous polling which would add overhead in the controller-switch 
channel. 
When the controller receives an OFPT_FLOW_REMOVED message that was triggered by an 
IP flow entry, then it checks if this flow is aggregated or not. If it is not aggregated then the 
expiration of the flow entry was due to the flow stopping altogether, and as such the flow is 
immediately removed from any databases. On the other hand if the flow is currently aggregat-
ed, then the expiration of the IP flow entry does not mean that the flow itself has stopped and 
so the flow is kept in the databases. This is because the flow is now handled by the aggrega-
tion flow entries which have a higher priority than the IP based flow entry.

3.3 
General Remarks on the Traffic Aggregation module 
3.3.1 Why VLAN tagging was selected 
There were two major candidates for tagging incoming packets in the ingress nodes of the 
aggregation paths, these were VLAN tags and MPLS labels. Because of the simpler admin-
istration, VLAN tagging was selected for this project. But as it is also discussed in the Con-
clusion of this report, in hindsight MPLS labels might have been a more suitable option. This 
is because the VLAN header has only 12 bits to identify the VLAN tag which allows for only 
4096 unique VLAN tags. This number might be enough for experimental purposes but in a 
real deployment scenario, a migration to MPLS labeling might be at hand. This is because 
MPLS allows for 1.048.576 unique MPLS labels, more than enough to cover a single back-
haul network. 
3.3.2 Separation of DSCP values into classes 
As has been stated in section 1.1.3.1 the DiffServ model has a proposed traffic classification 
scheme, in which the different DSCP values are divided into specific classes. The developed 
algorithm follows this path by allowing for the DSCP values to be assigned into separate clas-
ses. The algorithm then uses these classes to create the different aggregates and not the DSCP 
values themselves. This means that an aggregate can contain traffic flows with different 
DSCP values as long as these values belong to the same DSCP group. Currently this assign-
ment of DSCP values to DSCP is hardcoded in the algorithms source code. As a future im-
provement this assignment could be migrated to a REST based service that would remotely 
and on-demand change these assignments while the controller is in operation. 
3.3.3 Behaviour when only one flow is using an aggregation path 
When an aggregation path is left with only one active traffic flow using it, there are two ways 
the module could react. It could either destroy the path and let the flow to be forwarded based 
on IP information or it could keep it active and let the flow be forwarded based on the already 
used VLAN tag. For the current implementation the latter option was selected. This was be-
cause destroying the paths generated unnecessary complexity to the algorithm and unneces-
sary overhead traffic between the controller and the switches.  
On the other hand this option comes with a disadvantage, as in some cases the constructed 
aggregation paths tend to be more complex than needed. This is because keeping these paths

active may lead to a highly fragmented environment, with a big number of small aggregation 
paths.

Experiments 
The next logical step after validating that the traffic aggregation module works as intended 
(through the simple yet extensive validation phase), is to monitor its performance in a series 
of experiments that can yield quantifiable results. In order to do so, a number of realistic net-
work topologies and traffic patterns were designed and implemented using Mininet [20]. 
Mininet is an Open source software that allows the creation of SDN-compatible virtualized 
networks. Because these networks were designed using the Mininet Python based API, they 
are highly programmable and configurable. 
4.1 
Key Performance indicators 
During the validation process, what was monitored and documented were the total number of 
flow entries in the network’s switches, how these flow entries were distributed within the 
network itself and if this distribution corresponded with the expected behavior of the algo-
rithm. 
In the case of the performance testing though a different set of metrics was selected. The total 
number of flow entries was kept, as it is the key performance indicator that has to be meas-
ured. The distribution of the flow entries in the network though, was left out. This is because 
the complexity of both the network topologies and traffic patterns, used in the following ex-
periments, made the processing of this metric quite troublesome. In addition a number of new 
metrics was introduced. These were the connection setup delay and the resource usage (in 
terms of CPU and RAM) within the controller. These two metrics were added to see if the 
aggregation process has any negative side-effects to the network’s and controller’s operation. 
The following sections describes each performance metric in detail by highlighting why it has 
been chosen and how it has been measured in the following experiments. 
4.1.1 The total number of IP-related flow entries within the network switches 
As previously stated this is the main performance indicator for the implemented algorithm. 
By measuring the total number of flow entries in aggregation and non-aggregation environ-
ments it was possible to quantify the reduction in number of flow entries, which the aggrega-
tion process yielded.  
To ensure that the number of flow entries was measured in a reliable way, a relatively simple 
Floodlight module was created, whose sole purpose was to accurately measure the total num-
==PAGE:56==
 
56 
 
ber of IP related flow entries, within the network, when requested. This allowed for faster and 
more accurate measurement than by “manually” observing the flow tables of all the network’s 
switches.  
The reason that only IP related flows were measured is because the traffic aggregation algo-
rithm was designed to work based on the source and destination IP addressing of incoming 
packets. This means that any other flow entries that don’t have IP addressing information (e.g. 
ARP (Address Resolution Protocol) related) have been omitted from the results of both ag-
gregation and non-aggregation environments.  To generate the traffic flows either the ping or 
iperf tools were used. 
4.1.2 The time delay for flow setup between two hosts 
Because the aggregation process adds some complexity to the flow setup process, it was de-
cided to measure how much time it takes (on average) for all initiated traffic flows to be suc-
cessfully setup. 
In order to measure this metric the ping tool was used. This was because ping, logs the Round 
Trip Time (RTT) time of all initiated packets. So by logging the RTT time of the first ping 
packet between any two hosts we can have an accurate approximation for the time it takes for 
a bi-directional flow to be setup. This measurement is not the exact value of the bidirectional 
flow setup time as it also contains the setup process in the communicating hosts. Nevertheless 
what is of interest is not the time value itself, but a comparison of the measurements between 
aggregation and non-aggregation environments. Also as the networks (and so the hosts) are 
virtualized using always the same resource allocation, it can me assumed that the time it takes 
for the hosts to process incoming flows and forward outgoing flows is a relatively stable value 
from experiment to experiment. For this reasons these measurements can be safely considered 
as reliable. 
 Because of the big number of hosts in the following experiments, this measuring process was 
automated through the use of simple Python scripts.    
4.1.3 The resource usage in terms of CPU and RAM of the Floodlight Controller. 
Apart from any overhead in time, the aggregation process might also increase the CPU and 
RAM usage of the Floodlight controller. In order to ensure that the aggregation algorithm was 
a scalable solution and not too resource intensive, the overtime CPU and RAM usage of the 
Floodlight controller was logged while the experiments were running. 
In order to do so the Linux terminal top command was used. Top allows for monitoring in real 
time any running process, the gathered information can be logged and later processed to ex-
tract the required results (RAM and CPU usage). 
==PAGE:57==
 
57 
 
4.1.4 The connectivity between communicating hosts 
Because the traffic aggregation module modifies the forwarding process of the Floodlight 
controller, it is important to ensure that there is still successful connectivity between any 
communicating hosts. To ensure this connectivity, the logged information from the iperf and 
ping tools was parsed for hints that could translate to connection failures.  
4.2 
Floodlight controller setups 
As previously mentioned the traffic aggregation module supports two modes of traffic aggre-
gation, one with support for QoS traffic differentiation and one without. In order to assess the 
benefits of aggregation from these two controller setups there is the need to also obtain some 
baseline results to compare them against. To compare the results from aggregation with no 
support for QoS traffic differentiation, the default Floodlight controller setup was used. As it 
forwards traffic without performing aggregation and without performing DSCP based distinc-
tion between the traffic flows. 
Finally in order to retrieve the baseline results to compare against the ones from traffic aggre-
gation with support for QoS traffic differentiation, a slightly modified version of the default 
Floodlight setup was used. In this modified setup the Floodlight controller uses the DSCP 
value of the incoming packets as an additional field, in the matches installed in the SDN ena-
bled switches flow tables. This creates, for packets that only differ in their DSCP value, sepa-
rate flow entries in the SDN enabled switches.  
So to sum up there are four distinct Floodlight controller setups. The names in the parentheses 
are the IDs used later to identify each of the controller setups in the experiments: 
1. Floodlight with traffic aggregation and support for DSCP based QoS (FL+QoS). 
2. Floodlight with traffic aggregation and no support for DSCP based QoS (Path). 
3. Floodlight with no traffic aggregation and no support for DSCP based QoS (FL). 
4. Floodlight with no traffic aggregation but with support for DSCP based QoS 
(FL+QoS).

4.5 
Experiments 
The following sections describe the experiments that were conducted in order to measure the 
performance of the aggregation algorithm. This is done by presenting the different variables 
used in each experiment, the performance metric that was monitored during the experiment 
and finally by presenting and explaining the results. The results are presented through linear 
or bar graphs, but they can also be observed in their raw data format in Appendix A. 
4.5.1 Procedure followed 
In order to ensure that the results gathered from the following experiments were trustworthy, 
the procedure described below was followed. 
1. During all the sets of experiments the hardware of the host machine, the resources dedi-
cated to the VMs and the versions of all the software remained stable. 
2. All the experiments were repeated multiple times, until the mean value of the monitored 
performance metric stabilized. This procedure strengthens the credibility of the final re-
sults as it minimizes the effects of any results that have peak values. 
In all the following experiments the different traffic flows were generated in the following 
way: 
1. Before any new experiment the hosts were randomized and their random order was kept 
for all the iterations of the same experiment. This randomization was needed to minimize 
the chances of having a specific pattern in the generated traffic flows.  
2. During each experiment each host initiated 3 traffic flows, each flow had a DSCP value 
that belong to a different DSCP group.  
3. All these traffic flows had as destination the exit node of the topology. This was done to 
simulate the behaviour of a real mobile backhaul network, where all packets have to reach 
the core network even if they are destined to hosts within the same mobile backhaul net-
work.

Conclusions and future work 
5.1 
Conclusions and Remarks 
The scope of this thesis was to design, implement and measure the performance of an SDN-
based traffic aggregation algorithm for the mobile backhaul networks. This report provided 
with a thorough analysis of all these steps, providing also with any background knowledge 
that might be required. Based on the experimental results, as showed in Chapter 4, it can be 
safely concluded that the final implementation of the traffic aggregation algorithm has met 
the vast majority of the initial criteria set in Chapter 1 and Chapter 3. As it manages to per-
form path and DSCP based traffic aggregation, without hindering in any way the operation of 
the network and by only slightly effecting the performance of the SDN controller. The results 
of the experiments also showed a very stable and predictable performance regarding the 
reduction of flow entries in the network. Taking all these into account it is safe to assume 
that traffic aggregation and this algorithm in particular are a viable solution to the problems 
that the increase in the number of individual traffic flows will eventually bring to the mobile 
backhaul networks.  
5.1.1 Effects of user mobility in Aggregation 
One of the major issues regarding traffic aggregation in mobile networks is how to cope with 
the mobility of the UEs and thus their corresponding traffic flows. When end users of the 
mobile networks move around there is chance their serving eNodeB will change, this means 
that their traffic flows will now originate from this eNodeB and thus any previous aggregation 
done will need to be reconfigured. While this issue does not inhibit the aggregation process, 
in cases of high user mobility it might increase the computational and traffic overhead of the 
aggregation process. Due to time and resource limitations, the developed algorithm has not 
been tested with mobile UEs. 
5.1.2 Effects of traffic aggregation in flow processing granularity 
In order to perform meaningful traffic aggregation some of the packets’ headers have to be 
ignored during the allocation of a packet to an aggregate. If all the headers where taken into 
account during the aggregation process then each aggregate would consist of packets of a 
==PAGE:81==
 
81 
 
single traffic flow, something that contradicts with the logic of aggregation. By ignoring some 
of the packets’ headers though, the ability to utilize the high level flow granularity of SDN is 
effectively lost and with it the ability to create diverse and intelligent traffic rules. This issue 
is an inherent drawback of traffic aggregation and can only be mitigated by changing the 
level of granularity when assigning traffic flows into traffic aggregates. This issue was the 
main reason that DSCP based traffic aggregation was selected for this project, as the DSCP 
values allow for very efficient and intelligent traffic differentiation without the need of using 
multiple packet headers.

improve its performance. This section will highlight some ideas that can be implemented in 
the future.

5.2.2 Migration from VLAN tagging to MPLS labelling 
As has been previously stated, migrating the project from VLAN tagging to MPLS labelling 
would allow for far more aggregates to be created. This is because the number of available 
MPLS labels is far greater that the number of VLAN tags. This migration will require some 
changes to the source code of the project, but it would not change the philosophy of the algo-
rithm itself. This makes it a relatively easy change to implement.

5.2.3 Implementing a rerouting scheme 
Previously it has been stated that the rerouting process can be significantly simplified in a 
network with traffic aggregation, as the number of flows that need to be rerouted is signifi-
cantly reduced. However the current version of the project still lacks a rerouting scheme 
meaning that it was not designed with coping with node or link failures. This section will 
provide with a very crude planning on how rerouting can be implemented in the aggregation 
environment of this project. 
The algorithm should continuously monitor the network for node and link failures. If a failure 
is detected and it is inside one or more aggregation paths then this paths should be recalculat-
ed. The fact that the aggregation mechanism is primarily based on a source-destination logic, 
means that there is no need to add or remove flows from the current aggregates. So the only 
actions that will need to be taken, is to diverge the aggregation paths away of the point of 
failure. The reason it is important to monitor for link and/or node failures is that it is a better 
practise to identify and respond to these failures before the aggregates that pass through them 
have a chance to expire. If an aggregate expires then it will need to be recalculated and re-
established in all the nodes of the aggregation path, whereas if the problem is identified soon 
enough it will only need to be rerouted in a portion of the aggregation path. 
5.2.4 Run more extended experiments 
Even though a sufficient number of experiments were conducted to reach some safe conclu-
sions regarding the effects of traffic aggregation and the performance of the aggregation algo-
rithm, there is still room for more extended experimentation. Primarily the Tree of Rings to-
pology should be debugged, so there can be results regarding this topology. Finally there 
should be more experiments with scaling number of flows so the scalability of the aggrega-
tion algorithm in different scenarios can be further tested.

6.2.1 Dynamic traffic aggregation for mobile backhaul networks 
Abstract— as the traffic that originates from the 
radio access network increases, so do the require-
ments that the mobile backhaul network has to 
meet. One of these requirements is the need for the 
network switches, to be able to cope with an in-
creasing number of traffic flows and their corre-
sponding entries in their memory. This paper tack-
les the issue by proposing a SDN-enabled dynamic 
traffic aggregation mechanism, which changes to 
accommodate the traffic requirements.  
 
Index Terms—LTE, LTE-A, Mininet, Mobile 
Backhaul, OF, OpenFlow, Radio Access Network, 
QoS, RAN, SDN, Software Defined Networking, 
Traffic aggregation, VLAN 
 
I. 
INTRODUCTION 
he big amounts of traffic coming from the 
radio access part of the mobile networks to 
the backhaul have a tendency to increase even 
further. This is mainly because of the increase of 
available bandwidth, which came with the inte-
gration of the new mobile technologies (e.g. 4G). 
This has raised considerations on whether the 
existing backhaul architecture will be able to 
cope with this rapid evolution. Apart from the 
obvious problem of bandwidth management, 
there is also the issue of how to manage the vast 
number of traffic flows. This increase of traffic 
multiplies the demand for resources within the 
network devices with respect to memory capacity 
and memory management. It is common to think 
of a network bottleneck only from the perspective 
of bandwidth, but it can also be related to the 
memory dedicated for storing and managing the 
forwarding entries, in the network devices. More 
entries means more occupied space in this 
memory, which can lead to either increased 
searching times for matching entries or even 
equipment failure if the memory runs out of 
space. 
Currently the most dominant solution to this 
problem, is to use traffic aggregation. This is 
done in order to reduce the number of forwarding 
entries as much as possible. The problem with 
current aggregation approaches is that every 
implementation has to be statically defined in all 
the network equipment and it cannot react to 
changes in the network’s state. The work present-
ed in this paper tries to solve this issue. To do so, 
it proposes a traffic aggregation strategy that is 
enabled by the integration of Software Defined 
Networking (SDN) technology into the existing 
backhaul network. Because of the use of SDN as 
the core technology, the aggregation strategy can 
be superior from the traditional implementations. 
This is because it will benefit from the program-
mability offered by SDN. The main advantage is 
that SDN allows for the dynamic reconfiguration 
of the network devices depending on the network 
state. This allows the proposed aggregation 
mechanism to adapt without the need for constant 
administrative control. 
The proposed strategy alleviates stress from

Similar work 
Currently there is considerable work around 
SDN and on how it can be utilized to improve 
existing network technologies. This is also true 
for the mobile backhaul network architecture in

SDN is provided. Also some basic but important 
information about the principles of SDN are pre-
sented there. In [2], the steps for migrating to an 
SDN–only mobile backhaul architecture are pre-
sented, with the target of proving the network for 
the future requirements. What is proposed is to 
migrate to a flattened network architecture. The 
idea is to move the functionality of the different 
network elements to a centralized location, by 
virtualizing them within the same network entity 
as the SDN controller (to allow for example for 
faster mobility support).  In [6], it is argued that 
SDN can be used to simplify the architecture and 
management of existing cellular networks, while

with non SDN enabled Mobile networks. 
 
Subscriber Based policies and not only

 
Switches are able to perform simple actions 
(e.g. comparing traffic counters with thresh-
olds) when instructed by the controller. 
 
Switches can perform deep-packet inspection 
and header compression. 
 
Support for flexible cell management, by 
remote control of virtualized wireless re-
sources. 
 
 In [5], the use of SDN is proposed, to enable 
different operators to share their physical infra-
structure. This can help an operator to route parts 
of its traffic through third party networks, when 
its own network either reaches its maximum 
capacity or when a failure occurs. A centralized 
SDN controller is proposed, which intercepts 
traffic from and to the individual operator con-

work. In [9] the integration of SDN, to facilitate

consideration user and application criteria. That 
proposal [9] improves on this issue by using SDN 
to also include real-time network condition crite-
ria. 
The herein presented work provides an addi-
tional step for migrating the existing backhaul 
architecture to its SDN enabled counterpart. This 
is achieved by introducing a new aggregation 
mechanism, which is based on SDN. 
III. 
SDN and traffic aggregation overview 
This section of the paper focuses on providing 
some basic information regarding the technolo-
gies related to this proposal. 
A. 
SDN and OpenFlow overview 
SDN promotes a new network architecture, in 
which the control and data planes are decoupled. 
All the network intelligence is moved, to one or 
more dedicated controllers. So the SDN enabled 
switches are responsible only for forwarding 
packets, based on instruction they receive from 
the controller(s).  
Whenever a switch receives a packet, if it has no 
knowledge on how to forward it, it contacts the 
controller (using the OpenFlow protocol [3]) and 
requests specific forwarding instructions. When 
an SDN controller receives a packet from a 
switch, it will process it and will instruct the 
switch (again using the OpenFlow protocol) on 
how to process this packet. These instructions can 
later apply to every other packet that meets spe-
cific shared criteria with this packet. These in-
structions are a pair of a match and an action and 
are called flow entries.  To reduce the conver-
gence time by reducing the flow setup delay, 
these instructions can also be pre-installed to a 
switch, before a matching packet arrives. In Fig-
ure  a simple example of OpenFlow operation can 
be observed. 
H1 sends a packet to H2. 
1. The switch receives the packet and since 
its flow table does not contain a matching 
flow, it has no knowledge on how to pro-
cess it. So it forwards the packet to the 
controller as a packet_in OpenFlow (OF) 
message and waits for instructions. 
2. After receiving the packet_in OF message, 
==PAGE:95==
 
95 
 
the controller decides on the proper action 
and sends the packet back to the switch 
together with a (match, action) tuple, as a 
flow_mod OF message. 
3. When the switch receives the flow_mod 
OF message it installs the flow entry 
(match + action) to its flow table and for-
wards the packet to 2 (5). These entries 
may have a limited life-cycle time, when 
this time expires they are flushed from the 
memory. This is done to prevent old un-
used entries from cluttering the memory. 
 
As long as there is a flow entry present in the 
switch, the switch will not need to contact the 
controller again, for traffic from H1 to H2. 
 
Figure 0: OpenFlow Operation 
B. 
Traffic Aggregation overview 
Traffic aggregation is the concept of grouping 
multiple traffic flows, which share specific prede-
fined criteria (destination, QoS etc.). This has the 
benefit of alleviating some of the stress from the 
network nodes, as it reduces the forwarding com-
putational overhead in the devices. This is mainly 
because the number of forwarding entries in the 
switch is reduced and consequently also is the 
time needed to find a matching entry in the 
switch forwarding table. 
The basic idea behind this grouping mecha-
nism is to put some identifier to each packet that 
arrives at an aggregation point. This will help 
identify to which flow entry each packet belongs 
to. This identifier can be for example a VLAN 
tag or a MPLS label. After the packet has been 
tagged, it is forwarded based on this tag/label and 
not based on its destination IP address. Finally 
before their delivery to their final destination, all 
aggregated (tagged) packets have to be at some 
point de-aggregated (untagged).  
In Figure 1 an example of how a simple aggre-
gation strategy operates can be observed. The 
network consists of 2 switches (S1 and S2). S1 is 
the aggregation point and switch 2 is the de-
aggregation point. The network has also two end 
hosts (H1 and H2). Consider 5 different traffic 
flows entering S1. Without any aggregation 
mechanism, these traffic flows will have to be 
treated independently and thus both S1 and S2 
would have 5 entries on their memory. But if 
traffic is aggregated on S1, depending on the 
destination, then the outgoing traffic from S1 has 
been reduced from 5 flows to only 2, as S2 can 
now forward based only on the VLAN tag. In 
Figure 1, aggregated flows are represented in the 
{x, y} format, where x and y are the aggregated 
flows. 
 
 
Figure 1: Aggregation Example 
IV. 
Traffic Aggregation with SDN 
This paper proposes two modules, for an exist-
ing open source controller. These modules will be 
responsible for aggregating traffic in a program-
matically enabled manner, as previously dis-
cussed in this paper. One module will be respon-
sible for deciding where and when in the network 
aggregation is needed. The second module will 
be responsible for implementing the aggregation 
strategy itself. The benefits from using SDN 
enabled aggregation, originate from the pro-
grammability features provided by the SDN con-
troller. Both the aggregation point and the aggre-
gation strategy can be dynamically changed at 
any point in time, depending on the network 
state. 
In the mobile networks, end hosts are not 
bound to a specific location. Additionally their 
bandwidth depends largely on the behavior and 
number, of other end hosts in the same cell. This 
mobility and fluctuation in cell bandwidth output 
[1], creates an inherent unpredictability to the 
mobile backhaul traffic. This makes backhaul 
network provisioning a difficult task. Because 
SDN is a highly dynamic technology, it is a great 
candidate to help mobile operators cope with this 
unpredictability, as it can adapt with the changes 
in the Radio access network.  
 
==PAGE:96==
 
96 
 
A. Deciding the time and place for ag-
gregation 
The basic idea behind the first module is that 
depending on the current network state (traffic 
load, switch memory etc.) the controller will 
decide if and where in the network, traffic aggre-
gation is needed. To have the required infor-
mation the controller will periodically poll the 
network switches for specific statistics and use 
them to reach a decision.  In a non-SDN enabled 
network, retrieving this traffic information from a 
network requires a lot of resources, both from 
traffic and hardware perspective and it needs 
either dedicated hardware and/or is vendor spe-
cific. However with SDN it is as trivial as a re-
quest-response scheme, from the controller to the 
switches. More specific information about this 
mechanism can be observed in Section V of the 
document.  
B. Aggregation strategies 
It is important to note that depending on the net-
work state it is possible to either have a single 
layer or a multi-layer aggregation policy. 
Single layer aggregation 
When the aggregation points receive un-
tagged traffic they will tag them with an ap-
propriate VLAN tag. 
When the aggregation points receive already 
aggregated traffic, they will behave as nor-
mal switches and just forward the traffic 
based solely on the VLAN tag. 
When a non-aggregation switch receives ag-
gregated traffic it will forward it based sole-
ly on the VLAN tag. Else normal IP for-
warding will occur. 
 
Multi-layer aggregation 
There might be the case that even though 
aggregation has been enabled in the network, 
due to high loads it might prove to be insuf-
ficient. To solve this issue without having to 
remove the existing aggregation related flow 
entries from the network switches, a multi-
layered aggregation policy is proposed. This 
means that already aggregated traffic could 
be re-aggregated with an additional VLAN 
tag, in an additional aggregation point. 
 
The single/multi-layer aggregation policy can 
be observed in greater detail in Figure 2. 
 
Figure 2: Aggregation flow chart 
As an example of the previously discussed ag-
gregation policies, consider the network topology 
of Figure 3 (Note: Aggregation nodes are repre-
sented with a red glow and the numbers appear-
ing in the links correspond to the number of indi-
vidual flows within the specific link). Initially the 
aggregation node is switch S1, which tags traffic 
based on a combination of destination IP address 
and traffic type and switch S6 is the de-
aggregation point. Let’s assume that at first there 
is HTTP and VoIP traffic only, this means that 
the initial number of traffic flows (40 in this ex-
ample), will be reduced to 4 and will be tagged 
accordingly. After tagging the traffic, switch S1 
can either forward based on the packet infor-
mation itself (destination IP, traffic type etc) or 
on the tag it has just assigned. Both forwarding 
mechanisms will add the same overhead in switch 
S1 as the corresponding matching entries are 
already present (for the VLAN push action) and 
they only need to be coupled with an additional 
forwarding action (which port to push the packet 
out from). When switches S2 and S3 receive the 
tagged traffic, they will forward it based on the 
VLAN tags, and as such they will have 4 flow 
entries each.  The same also goes for switches S4, 
S5. Switch S6 will also have the required VLAN 
pop actions paired with the forwarding ones. 
Now let’s consider that after some time there are 
20 different traffic types arriving at switch S1. 
The current aggregation strategy will create 40 
individual traffic flows, which switches S4 and 
S5 cannot cope with. Because flushing all the 
existing aggregation rules from the switches will 
cause instability to the network (there might be 
inconsistent network behavior until all the new 
flow entries have been installed), it is more opti-
mal to aggregate traffic a second time on switch-
es S2 and S3, this time based on destination IP 
address only.  This second aggregation will re-
duce the number of flows to just 2. Apart from 
the benefit of ensuring network stability, keeping 
the initial VLAN tagging enable us to reuse it for 
forwarding in another place in the network 
(where there is no issue with the number of 
flows), after the additional VLAN tag has been 
==PAGE:97==
 
97 
 
popped. This strategy however can only be ap-
plied because there is an intermediary switch 
between S1 and S4, S5. If that is not the case (S1 
is directly connected with either S4 or S5), then 
switch S1 should be flushed from its initial ag-
gregation policy and be applied the new one 
based on the destination IP address. Finally it 
should be noted that if multi-layer aggregation is 
in effect the de-aggregation point (S6) will pop 
both VLAN tags before forwarding traffic to the 
final destinations. 
In the following section of the paper two ag-
gregation strategies are discussed.  
 
 
Figure 3: Detailed example 
1) Destination and QoS based aggregation 
An inherent problem with aggregation is the 
fact that after some traffic has been aggregated, 
the network operator loses the ability to enforce 
QoS over it. This is because any forwarding ac-
tion is only based on the identifier added to the 
specific traffic flow and not on the type of traffic 
itself. One way to overcome this issue is to take 
QoS criteria into consideration, when enforcing 
the aggregation policy. This will make sure that 
the traffic that is aggregated together, is traffic 
that should be treated in a similar manner. An 
example could be aggregating live video stream-
ing and HTTP traffic into different groups as 
their needs on bandwidth and delay differ. 
When a switch has been selected as an aggre-
gation point, it will have to do the grouping of the 
different flows based on a combination of the 
packet’s destination and some QoS criteria. Be-
cause the OpenFlow switches operate up to the 
transport layer (layer 4 of the OSI model), this 
means that any information contained in the 
packet up to this layer can be used as criteria for 
grouping the packet to a specific flow. More 
specifically these criteria can be: 
The source and destination MAC address. 
The source and destination IP address. 
The source and destination TCP or UDP port. 
Any pre-existing VLAN tag or MPLS label. 
The protocol ID. 
 
A lot of applications nowadays use encryption 
to secure the privacy of their users. And the effect 
this has on the ability to aggregate traffic, using 
QoS criteria, depends on the network layer in 
which encryption is used.  
Encryption in the transport layer and below 
may obstruct key information (ports, protocol IDs 
etc.), that the controller requires in order to reach 
a decision. On the other hand encryption that is 
applied to higher layers (e.g. application layer) 
does not obstruct aggregation in any way, as the 
SDN-enabled switches do not have access to this 
layer’s information to begin with. 
2) Path based aggregation 
It is possible to use the controller’s global view 
of the network to find the forwarding path be-
tween any two switches, before having to actually 
forward the packet between them. This ability 
can be used to improve the efficiency of the ag-
gregation mechanism. When packets arrive at an 
aggregation point, their forwarding paths will be 
checked by the controller.  If a number of flows 
share a significant part of their forwarding path, 
they will be aggregated together during this path 
and the final node of this path will serve as the 
de-aggregation point.  
To better understand this mechanism, consider 
the topology in Figure 4. If there is a traffic flow 
from H1 to H5 and one from H2 to H6, it can be 
easily observed that these two flows share the 
path (S3 - S4 – S6). Thus it would be beneficial 
to aggregate them into one flow in S3 and de-
aggregate them at S6. 
 
Figure 4: Path based aggregation 
In contrast with the Destination and QoS based 
aggregation, this policy is not affected by encryp-
tion in any way. 
It is important to note that these two strategies 
do not have to work separately, as they can work 
==PAGE:98==
 
98 
 
at the same time to maximize the total efficiency 
of aggregation. 
C. Deciding on the protocol 
After careful consideration it was decided that 
the implementation of the two modules will be 
based on the OpenFlow 1.3 protocol. There are 
two main reasons for this decision. 
 
By using OF 1.3, the interoperability of the 
implemented modules with future SDN 
hardware switches is secured. This is be-
cause version 1.3 is the latest major release 
of the OpenFlow protocol. 
 
The OF 1.3 release offers a number of new 
possibilities that were not present in previous 
releases. Some examples are: 
o 
The support for multiple flow 
tables within the same switch, which 
provide the ability to segment big tables 
into smaller and more efficient ones. 
o 
The introduction of meters can 
be utilized to gather required infor-
mation and thus enhance QoS support. 
 
It must be noted that Mininet [7] (the selected 
testing tool) currently has limited support for the 
OF 1.3 protocol. Some of the newest additions to 
OF (e.g. meters) have not been implemented yet 
for the Open vSwitch, which Mininet is based 
upon. Even with this limited support, the use of 
OF 1.3 [3] is still a better option than reverting to 
previous releases. It is certain that at some point, 
the use of OF 1.3 compatible messages will be 
mandatory for communication with SDN enabled 
switches. Taking into consideration the big 
changes in the API for the selected controller, it 
was decided that using a limited set of the availa-
ble tools was less disruptive than having to mi-
grate the code to OF 1.3, when the previous re-
leases will become obsolete.  
V. 
Prototype Description 
This section presents a descriptive overview of 
the implementation’s functionality. This is done 
by providing a step by step analysis of the im-
plementation’s behavior and also a graphical 
representation of the architecture (Figure 5). Steps 
1-6 regard the initialization of the aggregation 
policy while Steps 7-9 regard the functionality of 
the aggregation after it has been initialized. 
1. The controller continuously polls the SDN 
switches for statistics, using the Polling 
Mechanism. More specifically the control-
ler can request for statistics by sending a 
Multipart message to a switch. To identify 
the type of statistics that are requested 
(port, flow table, flow etc), the switch has 
to set the type field of the Multipart mes-
sage to the corresponding value. The same 
strategy must be followed by the switch 
when constructing the Multipart message 
reply, which contains the requested infor-
mation (2). 
2. When receiving the responses back, the 
Polling Mechanism evaluates the state of 
the switch. This is done by comparing the 
received data, with some threshold values 
that have been predefined, based on the 
switches capabilities. 
3. If the switch is at risk (the values received 
exceed the thresholds), then the Check 
Network State block is alerted. The func-
tionality of this block is to select the best 
points for aggregating and de-aggregating 
the traffic inside the network, based on the 
network architecture and current flow ta-
bles. The current state of the flow tables is 
needed so that the controller can evaluate 
where the problem originates from, they 
are obtained through the statistics re-
quest/response mechanism.  
4. When a decision is reached, then the 
Check Network State block, informs the 
Aggregation Policy Selection block about 
it. This is done by sending the selected 
switches IDs (unique identifiers that help 
distinguish each switch in the network), 
together with the current flow tables. This 
information is sent so that the individual 
aggregation strategies, would not need to 
contact the network devices again for fur-
ther information (thus minimizing the 
overhead traffic required for the aggrega-
tion mechanism). 
5. Depending on the situation, either one or 
both of the available aggregation strategies 
will be enabled (5).  
6. Immediately after aggregation is enabled, 
the selected mechanisms enforce their log-
ic into the flow tables they received from 
the Time & Place Module. They later pass 
their results to the Flow Mod Generator & 
Pusher block, which sends the appropriate 
flow_mod messages to the appropriate 
switches (8). These flow_mod OF messag-
es contain an action, which can be: 
 
A push/pop for a VLAN tag and/or  
 
A forwarding action through a spe-
cific port. 
 
It will also contain a set of requirements 
(matches), the incoming packet will have 
to meet in order for the corresponding ac-
tions to take effect (Some examples of 
match fields can be: input port, VLAN id, 
IP address etc). As all switches receive 
flow_mod messages when a new aggrega-
tion policy is in effect, it is expected that 
only aggregation point switches will send 
packet_in messages to the controller. 
==PAGE:99==
 
99 
 
7. The next time a switch sends a packet_in 
message, the controller will apply the ag-
gregation policy related to this packet.  
8.  The corresponding flow_mod OF messag-
es are sent to all the relevant switches of 
the network and not only to the switch that 
sent the packet_in OF message. This is 
done to reduce the overhead traffic as the 
controller already has knowledge on 
which switches will receive this packet in 
the future. 
9. It is important to note again that the Poll-
ing Mechanism block never stops operat-
ing. This means that if there is a change in 
the network state, then the Time & Place 
Module may instruct the Aggregation 
Module to either change or even stop the 
aggregation policy. 
 
 
Figure 5: Internal Architecture 
VI. 
Deployment in the backhaul network 
One of the biggest considerations operators 
face when considering the deployment of new 
technologies, 
is 
the 
Capital 
Expenditures 
(CAPEX). Migrating a network to a new technol-
ogy, can be very expensive and also time con-
suming. This may lead a network operator to be 
reluctant to change its network architecture. This 
however does not necessarily have to be the case 
when considering SDN integration. SDN allows 
for changes targeted to only specific parts of the 
network, leaving the rest of the architecture un-
disturbed. In our case this means that any changes 
can be limited to only the backhaul part of the 
mobile network, leaving the core and radio access 
parts undisturbed.  
Also it is important to note that SDN enabled 
switches can be programmed to perform multiple 
different operations (aggregation, firewall, for-
warding etc), while in non SDN-enabled net-
works each operation is usually performed by a 
different network element. This makes network 
provisioning in SDN networks easier, as the ad-
ministrator can change the functionality of the 
devices on the fly and does not need to worry 
about buying the right type of equipment.  
Thus migrating to an SDN enabled network 
will lead to a simpler network architecture that 
consists of a smaller number of cheaper and more 
versatile network elements. 
VII. 
Conclusion and future work 
This paper proposed a new traffic aggregation 
mechanism for the mobile backhaul network, 
which is based on SDN. As previously analyzed, 
this is an improvement over current aggregation 
mechanisms as it takes full advantage of the ben-
efits offered by SDN. The proposed aggregation 
mechanism is based on an intelligent VLAN 
tagging mechanism, which tags traffic based on 
QoS, destination and forwarding path criteria. To 
maximize the efficiency of aggregation, these 
criteria can be either use independently or togeth-
er, depending on the severity of the situation 
(high aggregation granularity may lead to in-
creased number of flow entries but also low 
granularity may lead to reduced network perfor-
mance). Finally Section V gives a detailed de-
scription of the implementation’s functionality. 
The next step is to characterize the implemen-
tation in terms of performance and capacity and 
compare with other aggregation mechanisms. To 
do so, the implementation will be tested in a 
combination of different network topologies and 
traffic scenarios. These network topologies/ traf-
fic scenarios will be simulated using Mininet. 
What is expected is to observe a considerable 
reduction in the number of flow entries in the 
SDN-enabled switches, when aggregation is 
enabled. Thus a faster response time from the 
switches when searching for a matching flow 
entry.
