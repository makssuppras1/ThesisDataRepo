Abstract
Non-Interactive Zero-Knowledge (NIZK) proofs, are a cryptographic primitive that enables
secure authentication and verification without revealing any information about the secret
on which the proofs are based. This thesis presents the development and implementation
of a library for the Fischlin transformation, a transformation that converts Sigma Protocols
from Zero-Knowledge Proofs (ZKP) to NIZK proofs. The library is designed to facilitate
research and testing of the Fischlin transformation, offering flexibility for different Sigma
protocols and elliptic curve configurations.
The performance of the Fischlin transformation is analyzed by examining different input
variables and how they affect the performance of the transformation. The implementation
is written in Rust and supports the usage of different Sigma Protocols, elliptic curves and
hashing algorithms. In addition, support has been designed to evaluate the computational
cost of the transformation and to find ideal input variables.
Through testing and validation, the results highlight the performance difference between
different Sigma Protocols and compare Fischlins performance to Fiat-Shamirs perfor-
mance, where the results show that Fiat-Shamir is up to 22 times faster.

1
Introduction
This thesis concerns the implementation of a library that enables research into Non-
Interactive Zero Knowledge Proofs (NIZK). The library will be based on the algorithm
proposed by Yi-Hsiu Chen and Yehuda Lindell [1]. In this thesis, the theoretical basis for
NIZK proofs will be discussed and explained.
The Fischlin transformation will be discussed analytically and analyzed step by step in
order to determine the performance of the transformation based on its inputs. Individual
Sigma Protocols and the cost of hashing will be analyzed in order to gain an understand-
ing of the performance impacts of the input variables.
After the analysis, the implementation of the library itself is elaborated into further de-
tails where two different solutions are suggested, one using library structs to store data,
and a second solution where the users provides their own data structures. After coun-
terbalancing the strengths and limitations of the two solutions, the second solution was
chosen despite the additional work related to users having to provide functions that gen-
erate the Common_Hash and Hash_B. However, the additional flexibility of being able to
use any valid Sigma Protocol, was determined to outweigh the cost of the additional work
for the user.
Additional functionality is also introduced to enable the determination of optimal input vari-
ables for the Fischlin transformation. This is achieved by measuring the execution time of
the individual user-supplied functions. Further support for the Fiat-Shamir transformation
was also added to the library to make it possible to compare the Fischlin transformation
to the Fiat-Shamir transformation
1.1
Motivation and Objectives
The motivation of this thesis is to implement a library that allows testing different Sigma
Protocols and input variables for the Fischlin transformation. The transformation imple-
mented is based on the paper ”Optimizing and Implementing Fischlin’s Transform for UC-
Secure Zero Knowledge” [1]. In this paper two algorithm’s are presented for the Fischlin
Transformation. The first algorithm is the generation of a Proof and the second is the
Verification of a Fischlin proof. Further, the paper presented an improvement from the
naive Fischlin transformation using batch proofs, resulting in an up to 24% increase in
performance with the Discrete log Sigma Protocol.
1.2
Limitations
Due to the scope and time limits of a Masters Thesis, this project does not concern it-
self with the mathematical proofs for the Cryptographic primitives, transformations, and
protocols used in this thesis. If the reader is interested in the proofs for the Fischlin trans-
formation, it is presented in the paper by Mark Fischlin et al 2005 [2]. The purpose of the
project is to implement a library of already described transformations and protocols.

1.4
State of the art
Cryptography has existed for thousands of years, with the first evidence found in ancient
Egyptian culture approximately 1900 BC. A well-known example of early cryptography is
when the Romans, specifically Julius Caesar, used Caesar ciphers to send messages
on the battlefield without the enemies being able to read the messages. In more recent
history, encryption has also played a crucial role. In WW2 the Germans were using the
Enigma machine made by Arthur Scherbius. Breaking Enigma Machines encryption was
a vital part of the war for the Allies, as it helped them win the Battle of Britain [3].
Commercially encryption started gaining traction in the 1970’s. In 1973 IBM created a
cryptography group that made the Lucifer algorithm. The Lucifer algorithm is a block
cypher encryption algorithm supporting up to 128 bit block sizes. The Lucifer algorithm
was later adopted as a standard by the National Institute of Standards and Technology
(NIST) and was renamed Data Encryption Standard (DES) . DES has since been broken
by exhaustive search attacks [3].

In modern day, one of the next challenges cryptography is going to face is the effect of
quantum computing on cryptography. This is due to the computing potential enabled by
quantum computing, which could break state-of-the-art cryptographic functions through
sheer brute force [4].
Another current topic within the field of cryptography is Zero Knowledge Proofs (ZKP).
ZKP are a set of cryptographic primitives with the purpose of allowing users, a Verifier
and a Prover, to confirm the validity of information between two parties without revealing
the information itself. ZKP has some practical advantages in certain fields. An example
of this is the financial sector. In the financial sector, it might be advantageous to prove
that one is solvent without revealing their positions. NIZK is an expansion upon Zero
Knowledge Proofs where the aim is to make the primitive non-interactive. This allows
for no communication from the Verifier to the Prover. This is very useful in services that
for example utilise block chains, as individuals using the block chain do not necessarily
communicate.
ZKP’s have two participants, the Prover and the Verifier. The Prover part is to prove that
they know the secret that they do not want to share. The Verifiers part is to verify that the
Prover has sent valid proof that they know the secret. This is in practice achieved by hav-
ing the Verifier issuing Challenges to the Prover. These challenges can only be correctly
responded to if the Prover knows the secret. In order to make ZKP’s non-interactive, a
mathematical transformation is required. The purpose of these transformations is to con-
vert the generation of the Challenge to become non-interactive. This is done by having
the Prover generate the challenge. Currently, three different transformations can be used
for converting ZKPs into NIZK proofs:
• Fischlin [2]
• Fiat-Shamir [5]
• Pass [6]
The Fiat-Shamir and Pass transformations have different advantages and disadvantages
compared to the Fischlin transformation. The Pass transformation is similar to the Fis-
chlin transformation in that it is UC secure. However, Pass is significantly slower than the
Fischlin transformation. A disadvantage for the Fischlin transformation is that it is only
applicable to a limited set of Sigma Protocols. However, this thesis focuses on the Fis-
chlin transformation as it is possible that the advantage of being UC secure outweigh the
disadvantages of only working for a limited set of Sigma Protocols [7].
Fiat-Shamir has the advantage that its the most-efficient of the three transformations;
however, it faces an issue with security. This is because it needs to rewind for the proof
of knowledge , which is not allowed in the Universal Composability (UC) security frame-
work; this means that the Fiat-Shamir transformation is not UC secure [8].

Theory of Zero Knowledge Proofs
This chapter explains the theory and knowledge required to implement a library to use
and test the Fischlin transformation and its performance.
The Fischlin transformation is a cryptographic primitive that transforms a ZKP to a NIZK
proof. In Section 2.1, it will be explained what ZKPs are, what properties they have, the
different types of ZKPs, and where ZKPs are used in practice.
As part of the ZKP overview, the Sigma Protocols will be explored. Sigma Protocols are
the type of interactive ZKP that the Fischlin transformation can convert to a NIZK proof.
In this chapter, it will be explained what Sigma Protocols are, their properties, and the
definition of each step of communication involved.
After the Sigma protocols a section will go over the three different transformations that
take Sigma Protocols and transforms them a to non-interactive protocol. In this section,
the Fiat-Shamir transformation, the Fischlin transformation, and the Pass transformation
will be explained. The chapter will compare the security and performance between the
three transformations.
The last section will provide a short introduction to elliptic curves. This is required as
the Sigma Protocols used for testing will be based on elliptic curves.
2.1
Zero Knowledge Proofs
ZKPs are a cryptographic primitive that involves two parties: a Prover and a Verifier. The
Prover knows a secret and wants to be able to prove that it knows this secret to the Verifier,
without revealing any information about the secret. This is achieved by having the Prover
generate a statement based on the secret that can only be true if the Prover knows the
secret. The objective of generating this statement is to ensure that no information about
the secret is revealed. ZKPs are required to have the following three properties:
• Completeness: If the statement is true and both parties are honest, the Verifier will
be convinced by the statement
• Soundness: If a statement is false, a dishonest Prover can’t convince a honest
Verifier its true - except for with a very small chance
• Zero-Knowledge: That if a statement is true, a Prover reveals no information about
the statement to the Verifier
In practice, this is achieved by sending multiple challenges to the Prover that can only be
correctly answered if the Prover knows the secret.
A classic example of ZKP’s systems is the ”two balls and the color blind friend”. In this
example, Victor (the Verifier) is color blind and has two balls that are different colors. To
Victor these balls are identical. Peter (the Prover) in this case wants to prove to Victor that
the balls are different without revealing how they differ. In this case a proof system could
be Victor taking a ball in each hand and showing Peter in which hand the balls are in. After
Peter has seen which hand holds which ball, Victor takes the balls behind his back. While
having the balls behind his back, Victor then either swaps the balls or leaves them in the

same hand as before. After possibly having swapped them, he shows them to Peter and
asks him to confirm if the balls are in the same hand. Peter, knowing the previous position
of the colored balls, can now confirm or deny whether a swap of the balls occurred. This
process is then repeated a number of times until the Victor is convinced that the Peter is
telling the truth.
The example also showcases why soundness is defined as not being able to convince an
honest Verifier that a dishonest Prover knows the secret except for with a low chance. This
is because Peter could still fool Victor by simply guessing correctly every time Victor asks.
ZKP’s advantage in modern day cryptography is data privacy. As the dependence on
online services using potentially sensitive private data continues to grow, utilizing ZKPs
can be a major advantage. ZKPs allow the user to never part ways with their sensitive
data, which prevents both malicious service providers and data breaches from revealing
private data. Some practical examples of where using ZKPs could be advantageous can
be seen below [9]:
• Block chain: ZKPs are used in block chains and specifically for cryptocurrencies, al-
lowing users to remain anonymous while utilizing these currencies to perform trans-
actions.
• Web authentication: Using ZKPs for web authentication would allow for more secu-
rity as it would reduce application layer threats to internet users.
• Mobile Health systems: Smart phones are increasingly used as a central node for
wearable health devices. Using ZKPs would ensure safer storage and transmission
of sensitive health data.
Although ZKPs has advantages when it comes to data privacy, it also has some problems
that need to be solved before it can be widely used. One of the major issues it faces is
computing costs and scalability, as ZKP’s can be quite heavy computationally.

2.1.1
Sigma Protocols
Sigma Protocols (Σ-protocols) are a three round public coin zero-knowledge proof cryp-
tographic primitive. The protocol being a public coin protocol means that any random
information generated by the Verifier is public to everyone involved. In Sigma Protocols,
this refers to the challenge, where the Verifier sends a randomly generated challenge to
the Prover. More formally, a Sigma Protocol consists of five different steps, a statement,
three rounds of communication, and verification.
• Statement: The Prover is given a statement and a secret.
• Commit (ProverFirstMessage): The Prover generates and sends a Commitment to
the Verifier.
• Challenge: The Verifier generates and responds to a randomly generated challenge.
• Response (ProverSecondMessage): The Prover generates and responds with a
transcript based of the challenge.
• Verify (VerifyProof): The Verifier accepts or rejects the proof based on the response.
As Sigma Protocols are a ZKP they follow the three properties described in Section 2.1:
Completeness, Soundness, and Zero-Knowledge. An example of a Sigma Protocol is the
discrete logarithm problem Sigma Protocol. This protocol uses the discrete log problem
for elliptic curves to construct a ZKP.
Figure 2.1: Discrete log Sigma Protocol
Fig. 2.1 shows the discrete logarithm problem in practice. The statement in this protocol,
is the generation of a witness w (the secret) with order q and using the witness to generate
Q with the group generator G. Due to the discrete logarithmic problem, it is very hard to
compute w even with Q and G being known.
The commitment is made by generating another ”witness” r that is used to calculate R,
both of which are sent to the Verifier. The Verifier responds with a randomly generated
Challenge (e). The Challenge is then used to generate the Response z. The Response
is sent to the Verifier, who can then verify if the Statement is valid.

2.2
Non-interactive zero knowledge proofs
For certain technologies such as crypto currencies, it can be beneficial to have a proof
that is non-interactive. This is because the proofs can be used even when the two parties
cannot communicate directly. This allows for more flexibility when using digital currencies
such as e.g. bitcoin. In order to make ZKPs non-interactive, another cryptographic prim-
itive is needed: NIZK proofs. NIZK proofs exhibit the same properties as ZKPs except
that NIZK proofs, unlike ZKPs, are only allowed a single round of communication from the
Prover to the Verifier.
2.2.1
NIZK transformations
One way of creating a NIZK proof is to transform an existing Sigma Protocol from a ZKP
to a NIZK proof.
The trick in converting a Sigma Protocol to become non-interactive is for the Prover to
be able to generate the Challenge. This allows for the Prover to generate the response
without any interaction, which will make the Sigma Protocol non-interactive. This means
that the main problem for transformations is how to generate a valid challenge. A valid
challenge refers to when both the Prover and the Verifier trust the challenge to hold the
three properties described in Chapter 2.1. In practice, this implies that the challenge
needs to be incomputable unless the witness is known and that it is unrealistic to be able
to guess a valid challenge.
Three such transformations will be explored here:
• Fischlin
• Fiat-Shamir
• Pass
Fiat-Shamir
The Fiat-Shamir transformation changes the Sigma Protocols from ZKP to NIZK. Out of
the tree mentioned transformations, it is the simplest and cheapest in terms of perfor-
mance. However, it is not as secure as it does not fulfil UC security due to rewind being
required for the knowledge proof.
The Fiat-Shamir transformations work by taking the Commit and statement, hashing them,
and using the hash as the challenge for the response. This is very simple compared to
the Fischlin transformation as it only requires an extra hashing compared to the regular
interactive protocol; it might even be faster if the response time of the challenge is large
enough.

Fischlin Transformation
The Fischlin transformation is a more complicated transformation that involves generating
many challenges and running the same proof multiple times. The transformation requires
three different input variables: 1) t: that determines an upper bound for the amount of
challenges attempted, 2) b: that determines the difficulty of finding a valid challenge and
3) ρ: that determines how many times the proof is run in ”parallel”.
The first step of the Fischlin transformation is to generate the Commit normally.
The second step is to generate the response with a valid challenge for each ρ. Unlike
in the Fiat-Shamir transformation, multiple challenges are generated. The Fischlin trans-
formation generates the challenges incrementally from 0 to 2t −1, checking if each gen-
erated challenge is valid. If a challenge is not valid, it is discarded and the next challenge
is generated. A challenge is valid if the first b bits of a hash are zero. The hash (Hash_B)
is found by hashing an identifier of the entire proof called the Common_Hash, identifier
for the individual proof, the challenge itself, and the response based on the generated
challenge.
In order to guarantee that no dishonest Prover can guess a valid challenge, each of the
two steps above are repeated ρ times.
The resulting commits, responses, and challenges are then sent to the Verifier. The Ver-
ifier can then confirm the validity, through the original verification method from the Sigma
Protocol and checking if the first b bits in Hash_B is 0 in all ρ proofs.
The Fischlin transformation is more secure fulfilling the UC security definitions; however,
it is also significantly more expensive to run due to the amount of computations required.
It is also important to set b and ρ to suitable values as ρ · b determines the probability of
being able to guess the correct challenge.
Pass transformation
The last of the three transformations is the Pass transformation. The Pass transformation
like the Fischlin transformation also satisfy the requirements of UC security. However the
pass transformation is significantly more expensive compared to the Fischlin transforma-
tion, while offering no extra security. This means that in practice there is very little reason
to use the Pass transformation and Pass transformations are, thus, not elaborated further
in this thesis.

Elliptic curves are algebraic structures that plays a fundamental role in modern-day cryp-

with smaller keys compared to other cryptographic primitives.
Mathematically, an elliptic curve can be described with the following formula:
y2 = x3 + ax + b
(2.1)
An example of such an elliptic curve can be seen in Fig. 2.2.

Elliptic curves have some beneficial properties for cryptography. Two of these are
• Any non-vertical line will intersect an elliptic curve only 3 times.
• Elliptic curves are horizontally symmetric.
These two properties allow us to create the mathematical operation called dotting. This
operation takes two points A and B as input. Then a line is generated between A and B,
only intersecting at one other point C. As described earlier, the elliptic curves are horizon-
tally symmetric, so C will have a symmetric counterpart. This counterpart D is the result
of the operation. An illustration can be seen in Fig. 2.3.

3.2
Analysing Fischlin and its inputs
The Fischlin transformation takes 6 different inputs as described in Chapter 3.1. It is
important to understand how these inputs affect the transformations performance.
The inputs t, b and ρ are t, b, ρ ∈N+, and they affect how the transformation calculates
the proofs and its security. The inputs x and w are the input data for the Sigma Protocol
functions. Lastly, sid is a session id ensuring that the proof cannot be reused for a replay
attack.
3.2.1
Completeness and soundness based of inputs
The first set of inputs to analyze are t, b, and ρ. These three inputs determine the chance
for completeness to be achieved and for how low the probability to break soundness is.
The input t does not affect the performance of the transformation itself, as the purpose of
t is to limit the number of repetitions of the nested loop in ProveFischlin. It is worth noting
that if t is set to a ”wrong” value, it will either allow the attempt at finding a valid challenge
to become too expensive or it will not allow enough attempts to find a valid challenge. If
not enough attempts are allowed, the proof will fail. The probability that the proof fails can
be described using an equation from [1]:
ρ · e−2t−b
(3.1)
NIZK proofs completion trait can be defined as: if a statement is true and both parties are
honest, the verifier will be convinced by the statement. This trait is still valid, as even if
the proof fails, the Prover can simply rerun the proof using the same input variables. In
terms of deciding a fitting t value, it depends on b. According to [1] a fitting t value for
ρ ≥64 t should be b + 6 and for ρ ≤64 t should be b + 5. It should be noted that any t
large enough should not impact performance, so any t ≥b + 6 should suffice.
Two other inputs that have a large impact on both the performance and security of the
transformation are b and ρ. This is because larger b and ρ make it harder for an adver-
sary to guess valid challenges. The chance for correct challenges to be guessed can be
described by the following equation:
2−b·ρ
(3.2)
Soundness in the case of NIZKs proofs is defined as; if a statement is false an dishonest
Prover can’t convince an honest verifier the statement is valid except for with a very small

Implementation
This chapter will go through details of the implementation of the Fischlin transformation
library and some of the challenges in designing it.
The first section will state the design goals of the library, available Sigma Protocols, elliptic
curves, maintainability, use ability, and being able to estimate ideal input values for the
transformation.
The second section will briefly discuss the programming language Rust and why it was
chosen. In addition, this section will go through the selected libraries that are being used
for elliptic curves and hashing.
The third section will go over the challenge that arises from being able to use the library
with multiple different Sigma Protocols with varying amounts of data.
After explaining the challenge of multiple Sigma Protocols, the first attempt to solve this
issue will be explained. This will show why the proposed solution makes the Sigma Pro-
tocols significantly more tedious to implement, and thus, makes the library significantly
more difficult to maintain. Additionally this results in the library being less flexible both for
testing different Sigma Protocols and for using different types of hashes.
Section five presents the second suggested solution. This chapter will explain why the
solution will add flexibility for the user, reduce maintenance compared to the first solution,
and why it comes with the drawback of adding extra input functions required from the user.
Lastly, the chapter will show additional functionality of being able to help the user to
estimate ideal input pair values, and check the performance of their own implemented
functions.
4.1
Design goals for implementation of the library
The purpose of this project is to implement a library that works with any valid Sigma Pro-
tocol, to enable testing of their performance with the Fischlin transformation. The library
will run the naive version of the Fischlin transformation. This should be implemented in
such a way as to both make it easy to test different Sigma Protocols and to evaluate their
performance. The user should be able to freely choose how the Sigma protocol is con-
structed. This is illustrated by the chosen examples which use both the ED25519 and
SECP256K1 elliptic curve.
In addition, the library must be easy to use. The library should focus on ease of use
and should, when reasonable, even do so at the cost of performance.
The library should be able to help the user easily determine the ideal inputs. The users
should also be presented with data allowing them to choose different input variables to
optimize performance for the Verifier or the Prover.
The library should also aim to be as minimalistic as possible. This is to decrease the
amount of maintenance, and to have the library performance have as little of an impact
on testing as possible.

Rust and used libraries
The language choice of the library is Rust. Rust is chosen because it being a fairly effec-

4.2.1
Elliptic curves
Two of the more well known and utilized elliptic curves are ED25519 and SECP256K1.
ED25519 is used for many communication protocols such as TLS and SSH, while Secp256k

ED25519 and SECP256K1. However, as a choice needs to be made, the following two
libraries have been chosen.
• k256 [11]
• ed25519 curves [12]
These libraries were chosen because of their popularity and ease of use, making Sigma
Protocols faster and easier to implement.
4.2.2
Hashing
A library is also required to hash the data for the transformation. The SHA2 library was
chosen again due to its popularity, but also due to its support for SHA-256 and SHA-512,
without requiring much change to the implemented code [13].

In terms of implementing the Sigma Protocols themselves, it requires users to constantly
have the Option tag in mind and to ensure that the structures are initiated correctly, mak-

Overall, this means that another solution to the data structure is needed for the library
to be flexible and have better performance.
4.5
Second Solution for data structures
Another way to solve the issue of the data for the Sigma Protocols in the library is to treat
every single data structure as generic data. Treating the data structure as generic data
would result in the user having to create the data structures for the library to use. How-
ever, in order for the library to be able to generate the Common_Hash and Hash_B, it
needs to understand what data m, x and z contain.
One way to solve this issue is to require the user to provide a function that is able to
generate the Common_Hash and Hash_B. Thereby having the user implement their own
data structures. The data structures will have to follow the same convention and flow as
in the Fischlin transformation, but otherwise the user would be free to design the data

In terms of being able to provide users with estimated ideal inputs and data to optimize the

For maintenance purposes, this is also a better solution, as it reduces the size of the

4.6
The Fiat-Shamir transformation
In order to properly test the performance of the Fischlin transformation, it is important
to also test the Fiat-Shamir to have a comparison of the run time. This means that a
Fiat-Shamir implementation is needed. The Fiat-Shamir transformation using the same
notation as the Fischlin transformation and can be described as:
• Prover: π ←proveFiatShamir(ProverFirstMessage, ProverSecondMessage, x, w)
1. (m, σ) ←ProverFirstMessage(x, w)
2. e ←H(x, m)
3. z ←ProverSecondMessage(x, w, σ, e)
4. π ←(m, z)
5. Output π
• Verifier: (accept/reject) ←verifyFiatShamir(VerifyProof, x, π)
1. Parse π as (m, z)
2. e ←H(x, m)
3. Output VerifyProof(x, m, e, z)
As the library is unable to handle data specific to a Sigma Protocol, the user needs to
provide any functionality where the data of the structs are accessed or generated. To
use Fiat-Shamir the user would be required to implement a function that generates the
challenge. Having support for Fiat-Shamir also impacts how the Challenge is handled by
the library. With Fischlin it is unlikely that the challenge itself will be much larger than 2b
where as with Fiat Shamir the challenge is the output of a hash, and thus the size of the
challenge is unknown. This means that the way a challenge is represented in the Library
needs to be flexible to support up to potentially 512 bit data.
In order to accommodate large numbers, this means that challenge will be represented
as a Vec<u8>. This allows the challenge to have any potential size, and therefore sup-
port the use of both SHA256 and SHA512. The downside of using a Vec<u8> is that it
potentially adds additional overhead to the library, making it slightly slower.
An example of how to use the Fiat-Shamir functions can be seen in chapter 5.5.

performance gap between Fischlin and Fiat-Shamir from 52 and 41 to around 40 and 30.

7.2
Implementation of the library
An important discussion to have is the implementation of the library. The libraries algo-
rithms are based on the simplified description from [1] with some small changes. The
changes are specifically to require the user to provide functions for the hashes involved in
the transformations, specifically GenerateCommonHash and GenerateHashB. For Fiat-
Shamir this means that a function that calculates the challenge is required. This design
decision was made to make the library more flexible, enabling the library treating the five
data structures and the user-provided functions as black boxes. This does come with
some additional work for the user, as they are required to correctly implement functions
to generate the Common_Hash and Hash_B to be able to use the library and for the proofs
to remain secure. If the functions are not implemented correctly and, for example, have
the wrong data, the proofs might not be secure. An example of such a mistake would be
forgetting to include the sid in the CommonHash. This would mean that any generated
proof could be reused, making the entire proof vulnerable to replay attacks.
7.3
Additional functionality
In addition to the complete Fischlin transformation, the library also supports measuring the
performance of the user supplied functions using the calculate_executiontime API. These
measurements can be used to estimate the execution time of the complete transformation
for different values of ρ and b. This in turn can be used to find a good candidate for the
optimal b value. In [1] they also proposed a method to calculate the optimal b. Their es-
timation was based on the cost of hashing and dotting. However, since the library treats
the input data as a black box, the user would be required to provide additional functions to
specifically test the cost of hashes and dotting. Directly measuring the execution time of
the user input functions also allows users to determine how efficient their implementation
of their provided functions is.
The library also supports the Fiat-Shamir transformation. In order to reuse the same
user provided functions for both transformations, the representation of the Challenges
had to be changed. This was due to Fiat-Shamir using hashes as challenges, which are
much larger than the challenges used in Fischlin. This adds some additional overhead
to ProverSecondMessage and VP as it required them to be able to handle much larger
numbers. This was solved by representing e as a Vec<u8>. This did end up costing some
performance; however, it made it much easier to compare Fischlin to Fiat-Shamir as the
same functions can be used for both.
7.4
Future Work
In terms of future improvements for the library, an obvious choice is multi threading. Multi
threading would be able to significantly increase the speed of the transformation as it runs
ρ independent proofs. These proofs can be done in parallel, as they do not interact with
each other in any way. However, the question is if this is actual a performance increase
or simply allocating more resources to speed up the process. When running proofs in
parallel, multiple cores will be required. Using Amazons cloud service as a reference for
pricing, it can be seen that the price of individual cores scales linearly [14]. This would
mean that using multiple cores to run the individual proofs in parallel would not actually
decrease the cost of the proof, but rather just make it faster. It is also worth noting that
46
A library for Fischlin’s transform
==PAGE:57==
when adding support for multi-threading it also adds more overhead for the library, mak-
ing the performance per core worse, therefore potentially making the transformation more
expensive.
Further, as the library is meant to be able to test the performance of different Sigma Pro-
tocols and to compare the performance with Fiat-Shamir multi-threading is not a require-
ment. If using multi threading, the results would not be comparable, as the Fischlin trans-
formation easily lends itself to multi threading where as the Fiat-Shamir transformation
does not.
For practical implementations, using multi-threading could still be beneficial in specific
circumstances. If a use case requires the proofs to be computed within a certain time
frame, which is only achievable by using parallel processing, multi-threading would be an
advantage.
When using the library to test the performance of different Sigma Protocols and the overall
Fischlin transformation performance, it is worth noting that this is a naive implementation.
In [1] they achieved an increase in performance of up to 24 % by using batch proofs. They
achieved this by improving the Discrete log Sigma Protocol through Batch proofs. This is
a noticeable improvement on the overall cost. The batch proof improvement was not im-
plemented as the library is meant to be a general implementation. It is, however, possible
that some batch proof improvements might still be possible using the library. This is due
to the open data structure implementation, where the user designs the data structures
themselves. A further improvement of the library could be to implement an extra optional
data structure for users to store any data that the user needs. With the current imple-
mentation, no data is shared between individual proofs and between individual attempts
at finding a valid challenge. This data structure could be universally passed to all user
functions, such that if the user wished to implement batch proof optimizations, it could be
achieved through storing the data required in this struct.
7.5
Conclusion
The objective of this project was to create a library capable of performing the Fischlin
transformation. The library needs to be easy to use and flexible to allow users to test
different implementations of Sigma protocols. The library should also attempt to be as
minimalistic as possible to reduce its impact on overall performance.
A library with these traits was implemented. The library itself is easy to use while allowing
the user full control over the data and how it is structured. Additional functionality was also
implemented to help the user determine optimal input variables and compare the Fischlin
transformation with the Fiat-Shamir transformation.
The implementation along with the additional features means that the overall objective of
the project to implement a library using the Fischlin transformation has been fulfilled.
