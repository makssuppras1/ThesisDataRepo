Summary
Documentation is an essential aspect of understanding and utilizing APIs. Pre-
vious studies have addressed the role of documentation in learning and using
APIs. However, documentation is often missing, outdated, or filled with irrel-
evant information. There is a need to identify significant information patterns
sought by developers from the documentation and ways to keep them up-to-date
and consistent. This thesis performs a systematic literature review to identify
the most crucial knowledge types of APIs that should not be excluded from the
documentation to make it concise. Additionally, documentation is a resource-
intensive process where information about the API has to be systematically
arranged. This thesis also explores methods of creating and distributing docu-
mentation effectively when working in Continuous Software Development. This
thesis found that documentation should at least include functional, behavioral,
and conceptual information with explained usage examples. This thesis also
finds that using tools already used in Continuous Development, such as Version
Control and build tools, can optimize documentation management. This thesis
identified information patterns that can minimize the gap toward completeness
of API documentation.
Further, this thesis summarizes methods to manage
documentation in Continuous Software Development.

Chapter 1
Introduction
Software documentation can be defined as the process of recording information
about the software. The artifacts created as a result can communicate infor-
mation about the software. By implication, Application Programming Interface
(API) documentation provides information about that API. APIs have become
building blocks of modern software. As more and more software development
teams have started adopting Continuous Development, APIs’ significance has
become unquestionable for faster software development through code reuse [1].
As such, API documentation plays a significant role in imparting knowledge on
how to use the API. However, it is commonly accepted that API documentation
is inflicted with quality issues. Developers have often found API documentation
to be missing and outdated [2, 3]. Developers regard good documentation to be
complete, up-to-date, and correct. However, as simple as it sounds, this idea of
good documentation is challenging to attain. When attempting to resolve the
incomplete documentation issue, for example, it may lead to long unbrowsable
documentation content and create other quality issues.
One of the most prominent challenges in CSD is to manage documentation
properly. The tendency to value working software over comprehensive docu-
mentation is considered to be a significant cause of poor documentation. When
the effort put into creating and updating documentation is low, the resulting
documentation can be inconsistent, outdated, or even missing essential infor-
mation. This can make documentation untrustworthy. Some might argue that

Motivation and Goals
The primary motivation for this work is to identify what information about the
API in its documentation makes it ‘acceptable’ as complete while keeping it as
concise as possible. This will also help to keep the documentation short and
precise. This involves identifying the essential bits of knowledge developers seek
from documentation to help them learn and utilize the API effectively.
A secondary motivation for this work involves documentation management in
the Continuous Development paradigm. The perceived lack of quality documen-
tation implies shortcomings in how the documentation is created, distributed,
and maintained.
This thesis aims to identify the essential knowledge that should be included in
the API documentation so that the users can find most of their information
needs in the documentation. This thesis also aims to find ways to manage the
documentation in Continuous Software Development.

Chapter 2
Background
Recently, software development has moved towards Continuous Development
[4]. However, documentation has emerged as one of the biggest challenges in
Continuous Software Development. This section provides a brief background on
Continuous Software Development and documentation.
2.1
Continuous Software Development
Harmful disconnects between crucial tasks like planning, assessment, design, and
implementation have been a hallmark of the software development process [5].
The conventional Waterfall software development process is a blatant example
of this [6].
The term ‘disconnects’ refers to gaps between various phases or
teams of software development. For instance, the operations team may not be
aware of the process of deploying a new feature of the software developed by the
development team. As the size and complexity of the software grow, software
development teams need to have tighter integration in their roles to increase the
quality of the software.
Continuous Software Development (CSD) is an approach to reduce disconnects
between different phases and software development teams, such as planning
==PAGE:16==
4
Background
and development [7]. Fitzgerald and Stol [5] define Continuous Software De-
velopment as a generic concept that involves several continuous activities with
definitions and descriptions. Continuous Software Development refers to prac-
tices carried out during the life-cycle of software with frequent iterative de-
livery. It covers Continuous Planning, Continuous Development, Continuous
Integration/Continuous Deployment, and so on. The software is delivered in an
iterative process as continuous activities are carried out.
Theunissen et. al. [8] define CSD as:
“as an umbrella term to collectively refer to such development
processes and other processes that share the following characteristics:
(a) it covers the values, principles and practices from Agile, Lean
and DevOps.
(b) it embraces activities from the whole life cycle of a software
product, from concept to end-of-life. In addition to Agile and Lean
software development, it includes maintenance activities. In addition
to DevOps, it includes continuous architecting activities.
(c) it considers the continuously changing state of the software
product and progress, such as progressive insights (e.g.
regarding
process, design, implementation), changes in contextual factors, new
features, bug fixes, or other unforeseen factors.
(d) it distributes information about software development across
multiple tools, because of demands for fast time-to-market, as well as
the need for a software development ecosystem for automated tests,
deployment, and monitoring. Thus, no central repository for all in-
formation is available.”
CSD combines the core ideas of Agile [9], Lean [10] and DevOps [11]. With
iterative software delivery, the time to market the software shortens significantly
[12]. This also provides feedback for the next iterations where the success of the
current iterations can be integrated, and failures can be improved/minimized
on the next iteration. Continuously changing needs and requirements of the
end-users can be integrated into the product quicker with insights gained from
the iterative processes.
One intrinsic trait of CSD is its distribution of knowledge. Due to the reasons
mentioned above, information on different aspects of the software is recorded
in different tools and platforms.
As a result, it might be challenging to lo-
cate the information needed. This might be mitigated with the help of proper
documentation to some extent.

2.2
Application Programming Interface
Application Programming Interfaces (APIs) are connections between computer
programs. They are interfaces that offer some service to other software or pro-
grams. Reddy [13] describes API as follows:
“An Application Programming Interface (API) provides an abstrac-
tion for a problem and specifies how clients should interact with soft-
ware components that implement a solution to that problem. The
components themselves are typically distributed as a software library,
allowing them to be used in multiple applications. In essence, APIs
define reusable building blocks that allow modular pieces of function-
ality to be incorporated into end-user applications" [13]
There are several things of note in this definition. First, an API provides an
abstraction of the underlying implementation of the solution. Even when the
implementation of the solution changes, the client can easily invoke an API with
its communication guidelines without being too concerned about the underly-
ing details. Second, API components are usually distributed across a software
library. Thus, an API can range from a single function to complex classes and
data structures with hundreds of sub-components.
Finally, an API provides
modularity and facilitates software reuse.
Web APIs are the most familiar APIs which allow interactions between comput-
ers joined by the internet. Other APIs include APIs for programming languages,
operating systems, and so on.
An API offers many benefits to developers. It helps encapsulate the implemen-
tation details and only exposes components necessary for its users [14]. It can
hide the internal details of the system which is essential for some proprietary
systems. Thus, API creators and maintainers can keep it consistent and reliable
even when a change is required in the implementation. An API helps make the
software modular as API components can be reused. Modern software typically
consists of several APIs [15]. Availability of APIs pertaining to developers’ needs
can speed up the software development process, increasing efficiency.
APIs have vital roles, especially in proprietary software. Software owners can
expose APIs for the users without exposing the internal details. This can also
help to add a layer of security and protection. Further, APIs provide a way to
improve collaboration between apps, platforms, and teams [16].

2.3
Documentation
Documentation is an integral part of software development. Complete and con-
cise documentation speeds up the development of the software. Documentation
is needed for communication between all those involved in software development
[17]. Documentation helps from the planning and decision-making to testing,
developing, and maintaining the software. It is the primary means of knowledge
transfer between individuals and teams working on the same software. In ad-
dition, documentation of the software helps its end-users fully understand and
utilize it.
Documentation comes in different flavors. Each phase in software development
is associated with its documentation. The planning phase of software devel-
opment may result in software specifications and design decisions. The devel-
opment phase may result in inline comments, reference documents, git commit
messages, tutorials, and so on. The maintenance phase may be documented in
issue tracking systems like Jira 1.
The most significant issue in adopting CSD has been documentation [8]. Most
developers tend to ignore the documentation aspect as it does provide an im-
mediate contribution to working software. Agile Manifesto [9] even prioritizes
working software over comprehensive documentation. However, this does not
imply that documentation should be neglected completely.
As a result, the
documentation that is created lacks quality.
2.4
Related Works
There have been numerous studies on different aspects of software documenta-
tion. The importance of software documentation has been firmly established
[18, 19, 20], and the overall quality aspect of documentation has also been stud-
ied [3, 21, 22, 23].
Documentation aspects like accuracy, readability, struc-
turedness, accessibility, and understandability were the most crucial qualities of
documentation [24, 23]. Further, documentation is often incomplete, obsolete,
ambiguous, and even incorrect in many cases [2, 25]. From their mapping study
of literature, Theunissen et al. [8] identified a few challenges, practices, and
tools regarding documentation in CSD. They found that focus on a short-term
tasks and the tendency to consider documentation creation a waste are among
the most significant challenges. Additionally, they found that documentation is
often out-of-sync with the software.

Methods
This section provides an overview of the review process adopted. This step-by-
step overview of the review process describes how to replicate the results. The
review was conducted in stages. First, the objectives of the review work were
determined. Second, a pool of academic literature was formed to meet the aims
of the review. Third, all selected literature was analyzed. Finally, the results
and conclusions of the analysis were obtained.
Section 3.1 describes the main goals of the review as research questions. Section
3.2 details the method of searching and selecting the sources of relevant academic
literature for analysis.
Section 3.4 describes the analysis process used.
The
overview of the review is shown in Figure 3.2
3.1
Research Questions
The main objective of this review is to identify ways to create concise and effec-
tive API documentation in CSD. With the challenges faced in the documentation
process in CSD, this work aims to explore what patterns of knowledge literature
places importance on. This can lead to create documentation that can provide
the best value while keeping it as terse as possible.
==PAGE:28==
16
Methods
Based on these goals, the following Research Questions (RQs) emerge:
RQ1: How can API be documented in a concise manner? This RQ aims at
understanding ways to minimize unnecessary information in the documentation
and keeping it concise. It involves identifying types of knowledge without which
documentation would be considered incomplete.
RQ2: How can API documentation be managed effectively in CSD? This RQ
aims at understanding how errors and inconsistencies in API documentation can
be minimized. Further, it also explores methods to distribute API documenta-
tion in an effective way.
Systematic Literature Review (SLR) [40] was used for this literature study. The
topic of API documentation has been significantly researched, with extensive
literature available. A SLR helps to get a comprehensive state of the topic at
hand. It also helps to find gaps in existing research. This justifies the selection
of SLR for this thesis.

3.3
Inclusion Criteria
To decide whether the paper was included in the source pool, following criteria
were adopted.
1. The paper is related to either API documentation, or documentation man-
agement in CSD.
2. Paper talks about broader aspects of documentation.
For example, a
paper studying the importance of directives was excluded as it does not
reveal whether directives are more or less important than other patterns
of knowledge.
3. Language of the paper was English.
4. The paper was peer reviewed.
Any papers that did not match the inclusion criteria were excluded.

3.4
Analysis
Content analysis was chosen for this review. Since the goal of the review is to
find out what makes good API documentation in general, the contents of the
papers hold the answer to the research questions. Thus, it justifies the choice
of analysis method, which consisted of the following stages:
• Read the papers manually
• Find and mark evidence-based answers to research questions
• Draw conclusions and results as described subsequently
Atlas.ti5 was used for the analysis. It gives the tools to examine the literature
sources and annotate/tag parts of their content. This helps in organizing and
analyzing the vast content. Figure 3.4 shows how Atlas.ti was used to encode
text and create notes. On the left side is the content of the paper. The text
fragments from the content of the papers were encoded as seen on the right
side. Text in the red background is the code used to encode the text fragment
(quotation). The text segment on the right side was used to create notes. It
was used to record findings and document the insights drawn.
The analysis was performed in steps. First, the text of the literature was read
manually, encoding the text segments that could contribute to the answers to
the research questions. Second, the encoded texts were grouped into higher-level
abstract categories. Lastly, results and conclusions were drawn.
The text segments that highlighted the importance of a different kind of knowl-
edge were encoded accordingly. Then, the encoded text segments were grouped

5.1
How can API be documented in concise man-
ner? (RQ1)
Essential information that should be included in the API documentation is the
information API users demand. Probably the obvious inclusion is of function-
ality. API users regard the conceptual aspect (concepts and purpose) of API
and its elements as essential information together with their usage examples
and patterns. Developers would find it easier to use the API if these patterns
of knowledge are included in the API documentation. It is a challenge to deter-
mine how much is enough documentation. However, this review suggests
that even the most concise documentation needs these five patterns
of knowledge: functionality, concepts, purpose, patterns, and exam-
ples. This will make the documentation usable with higher degree of
satisfaction.
On the lesser side of importance are directives, quality, control, structure, envi-
ronment and references. Directives help to know what is acceptable in the API,
thereby, removing ambiguity. This also saves developers from experimenting
with the API to find what works and what does not. Surprisingly, the perceived
importance of directives is lower in the literature. This suggests that users might
==PAGE:48==
36
Discussion
be somewhat satisfied with the current state of availability of directives in API
documentation.
Information on control might only be necessary in rare cases not covered in the
documentation. As one advantage of using API is not being concerned with the
internal workings of API, the perceived lack of importance of this knowledge
type can be justified. Patterns environment and quality could be justified in
similar manner. The information about structure can be obtained from IDEs.
Hence, it might have lower demand for inclusion in the API documentation.
However, the lower demand for these patterns of knowledge does not disregard
their importance. For example, information on structure is necessary to un-
derstand the architectural aspects of the API. Similarly, the need for control,
environment and quality may be lower but they’re still needed when develop-
ers come across situations involving them. Since the literature demands some
degree of documentation on these patterns of knowledge, it would be best to
include them in the API documentation as far as possible.
This study found that developers’ information demand for internal API devel-
opment, i.e., developers who create APIs as opposed to those who use them as
end-product, has not been researched in finer detail. Only a handful of papers
explore the topic. Technical documentation of program code, requirement speci-
fications, and design specifications are all examples of internal documentation in
software engineering [L25]. However, the major patterns of knowledge needed by
internal developers and maintainers seems to be functionality, structure, quality,
reference, purpose and concepts.
“Maintainers and searchers mentioned the importance of describing
how a file relates to other files in the project (S17), the state of
the world when a method is called (S8), executable examples (M5,
M8), implementation comments for future maintainers of an API
(M5), explicit links to external documentation (M5), semantics of a
function (M8), main concepts that someone should understand and
know to use the API (M8), “what” the code is doing and “why” at a
statement level (M6), and even a proof of correctness (M6).” [L17]
“technical documentation in software projects such as embedded com-
ments within the source code, descriptive and/or explanatory notes
of APIs, interfaces and algorithms are thought to be beneficial to
project initiators as well as to the engineers who are to maintain or
expand the software in the future.” [L25]

5.2
How can API documentation be managed ef-
fectively in CSD? (RQ2)
Official API documentation has been plagued by issues like incompleteness, ob-
soleteness, incorrectness, and so on. It is apparent that tremendous work has
been done to improve it. API Crowd-documentation has emerged as an excel-
lent addition to the official documentation. Although some might question the
reliability of unofficial crowd-sourced documentation, they have proved to be an
indispensable source of knowledge of modern APIs. Knowledge and experiences
from these sources give developers many insights into using the API in a way
that even the creators of the API might not have envisioned. However, it can
take a long time for crowd-sourced documentation to be utilized, as it takes
time for the crowd-sourced knowledge to be accumulated after the API is re-
leased. Further, the creation of crowd-sourced documentation requires diligence
in mining, processing, and curating crowd knowledge.
Various tools and approaches that use crowd documentation to complement the
official API documentation benefit the developers.
Similarly, there are tools

and approaches to detect inconsistencies, obsoleteness, and errors in the API
documentation. However, it is important to note that most of them are platform-

used to create a single software product. Thus, there is a need to develop a

The management of documentation in CSD can be significantly facilitated by
already existing tools of CSD like Version Control System and automated build
tools. With these, consistent documentation can be delivered to various stake-

Documentation becomes more essential as the software development team grows.
Changes within the team are also inevitable, with new members joining and old
members leaving. With this in mind, the software product needs to retain knowl-
edge and experiences as the software development teams change. This creates
the need for internal documentation. The documentation artifacts created for
this purpose are geared towards the internal developers, involved stakeholders,
and maintainers. Information documented can thus, include project overviews,
backlogs, user-stories (including requirements for new features and bug reports),
test cases, lessons learned, knowledge about the application domain, and so on.
Internal documentation may hold no significance for the customers and end-
users of the product who would be interested in how to utilize it to its fullest.
As a result, software creators may not be encouraged to create and maintain
internal documentation and view it as an extra effort which does not help the
end-users.

Chapter 6
Conclusions
This review identified what patterns of knowledge must be included in the API
documentation for it to be considered complete and concise. A systematic liter-
ature review of 25 sources was conducted as presented in Chapter 3. The main
finding of this review was the identification of elementary patterns of knowledge
that should be present in even the most concise API documentation (RQ1). API
users need the knowledge of API’s functionality and behavior explained with ex-
planatory examples. Furthermore, conceptual knowledge of API’s application
domain and the rationale behind its design decision is also rudimentary. The
inclusion of these knowledge types, however, may not make the API documen-
tation complete but will cater to most of its users’ needs. Exclusion of these
knowledge types, on the other hand, might compel the users to discard the API
and seek other solutions.
This review also found that internal documentation of API development dif-
fers to a great extent from the external API documentation. This review tried
to identify the patterns of knowledge need for internal documentation. How-
ever, the results were inconclusive as internal documentation aspects were only
researched by the scientific community in broader detail.
Crowd-sourced knowledge has become crucial for complementing official API
documentation (RQ2).
They provide API users with information missing in
official documentation, and developers heavily rely on them to fulfill their needs.
==PAGE:54==
42
Conclusions
As a result, several tools and methodologies are emerging to incorporate crowd-
sourced knowledge to recommend relevant documentation, detect, and suggest
updates to outdated and erroneous official API documentation. However, the
current tools and methods are limited to a specific platform. Hence, further
work on generalizing them is necessary.
The creation of API documentation is a time and resource extensive process.
Nonetheless, it is vital for the success of the API. If the documentation is created
as an artifact of the API development process, it could change the developers’
burdensome perspective of creating documentation. The use of executable doc-
umentation (RQ2) is one way to accomplish this. Additionally, documentation
through executables helps remain consistent, correct, and up-to-date informa-
tion in the API documentation.
API documentation process can be optimized by using the tools used in CSD
(RQ2). The use of version control systems to track and oversee the changes and
build tools to automate the compilation and distribution of final documentation
can provide a more efficient means of managing documentation without extra
overhead, as these tools are already used in most of the continuous development
processes.
Management of API documentation externally and internally also differs. Ex-
ternal documentation is usually made available online in various formats. How-
ever, internal documentation is mostly recorded in issue-tracking systems, user-
stories, and so on. While the tools and crowd-sourced knowledge can be helpful
in managing external documentation, it is not clear how they may be useful for
internal documentation.
For future work, the need for internal documentation can be studied in terms of
already identified patterns of knowledge. This study found that the knowledge
types needed in internal documentation have not been explored in much detail.
Similarly, empirical studies on different artifacts needed for internal documen-
tation and the knowledge type they should contain can be carried out. This
would give us the concrete idea on internal documentation.
