Abstract
This thesis aimed to improve how users interact with Danish parliamentary information
by introducing enhanced search, streamlined navigation, and user-friendly document ac-
cess. Despite refining data structures and employing modern NLP approaches for seman-
tic search, the project did not achieve its intended improvements in practice. Technical
hurdles, fragmented data formats, and too broad of a scope prevented full implementation
of the proposed prototype. As a result, the challenges identified in accessing information
related to parliamentary proceedings–identified through interviews with Members of Par-
liament, advisors, and engaged citizens—remain unresolved. Nonetheless, the lessons
learned about data modeling, user-centered design, and potential NLP applications offer
a foundation for future efforts to make legislative information more accessible.

Introduction
The Open Parliament project is a master’s thesis that seeks to improve how users can
interact with the information regarding the Danish parliamentary process. The Danish Par-
liament’s website (ft.dk) provides comprehensive information on legislative proceedings,
as it is one of the primary tools for Members of Parliament and their advisors. The project
aims to improve upon user interactions by developing a prototype of an open-source ser-
vice that addresses pain-points raised by users.
1.1
Motivation

Winston Churchill
Since the turn of the millenia, the famous proclamation by Francis Fukuyama that western
electoral democracy is ”the end of history, has been shown to possibly be premature.
As can be seen in figure 1.1), the blue line, representing the world index for electoral
demorcracy, has been trending down.

And while party membership has declined from from 13% to 2.2% of the population, the
number of parties in the Danish parliament has increased from 5 to 13. In 1960 approx-
imately 600,000 citizens out of 4,500,000 total were a member of a political party, now
merely 131,000 out of 5,850,000 in 2021 [3][4].
Furthermore, the legislative process itself has been accelerating. Over the past 15 years,

1.2
Initial Problem Statment
The Danish Parliament’s digital infrastructure, while comprehensive and well-structured,
could be improved with a more user-centered UX design. Through preliminary observa-
tions during the author’s brief parliamentary service, several issues with gathering infor-
mation in conjunction with parliamentary proceedings have been identified:
• Inadequate Search: Limited search capabilities hinder users from quickly finding
relevant information.
• Excessive Navigation: When processing documents related to a bill, navigation is
cumbersome and multiple tabs have to be opened.
• Analog Paradigm: Most documents related to parliamentary proceedings are only
available as PDF downloads, or HTML conversions. The content of the documents
vary from official A4-size ”letters” with formal questions and answers to a minister, to
a list of emails from organizations during the consultation phase, to the consultation
note prepared in Microsoft Word and exported to PDF by the responsible ministry.
These observations lead to the following initial problem statement:
It is possible to improve user interactions with parliamentary proceedings by designing an
interface and system architecture which allows for processing information and displaying
relevant information according to user needs.
1.3
Research Questions
To further qualify this, the following main questions are explored:
1.3.1
User Experience Design
How can legislative interfaces be designed to enhance accessibility for its users, including
MPs, advisors, and citizens? This examines the intersection of user interface design
principles with the requirements and constraints of a parliamentary information system.
1.3.2
Data Processing Architecture
What approaches can effectively transform complex relationships between parliamentary
actors, meetings, cases, and documents, into a more accessible format? This involves
processing multiple document formats (PDF, HTML, XML), generating semantic embed-
dings, and preserving the metadata relationships in a format for effecient data retrieval.
1.3.3
Data Modeling
How can we build upon the existing metadata schema, which already captures detailed
relationships between parliamentary entities, to better support advanced search and re-
trieval while maintaining data integrity? In particular how can we effectively query the
content of the transcripts and documents?

Background
2.1
Parliamentary Information Systems
A comparative case study from 2019 of open data initiatives in the Norwegian, Swedish,
and Danish parliaments highlights that, although all three countries have embraced open
data to increase transparency, the actual uptake and use by external parties remain lim-
ited.[11] Journalists and academia do make use of these datasets, but most hackathon
projects and third-party applications tend to remain exploratory and short-lived. One cited
barrier is that certain parliaments, including Denmark’s, often release only metadata (e.g.,
votes, agendas) rather than more comprehensive document datasets, potentially con-
straining opportunities for deeper analysis.
Regardless, these efforts to open parliamentary data illustrate the democratic value of pro-
viding machine-readable legislative information. Projects like ”Do they keep their word?”
(”Holder de ord?”) in Norway show how metadata, can be turned into interactive services
that track how MPs vote relative to party platforms[12] [13].
2.2
The Legislative Process in the Danish Parliament
The formal legislative process in the Danish Parliament runs in sessions (perioder) that
span from the first tuesday in October to around June[10]. There are two primary types
of bills: Law proposals (lovforslag or A-forslag in Danish) and proposals for parliamentary
resolution (Beslutningsforslag or B-forslag in Danish). It is primarily the sitting govern-
ment that presents the bills in Parliament, as they have the capacity in the ministries for
drafting legislation. Members of Parliament (MPs) can propose both private law proposals
and proposals for parliamentary resolution. Private bills are less common due to limited
party resources for drafting comprehensive proposals, leading MPs to frequently opt for
proposals for parliamentary resolution as a way to start a debate and potentially influence
government action through public support.
Before formal introduction (phase 4 in fig.1.2), most government bills undergo a compre-
hensive consultation process (høring - phase 2 in fig 1.2), where stakeholders provide
feedback through consultation responses (høringssvar) This consultation phase results
in a consultation note (høringsnotat) that are important to the subsequent legislative dis-
cussions.
The processing of parliamentary proposals follows different paths depending on their type.
Law proposals requires three readings with formal committee stages, allowing for amend-
ments at multiple points, although amendments are typically processed during the second
reading. Proposals for parliamentary resolution undergo only two readings. The Standing
Orders typically mandate a minimum interval of 30 days between a proposal’s introduction
and final passage, though this can be expedited for urgent matters.
During the first reading in the plenary chamber (phase 5 in fig.1.2), party spokespeople
present their positions and potential desired changes. Following this, the proposal is re-
ferred to the relevant committee (phase 6 in fig. 1.2), where additional processing occurs.
This commonly includes asking written questions to the responsible minister or proposing
amendments. The committee can also ask the minister for a consultation (samråd) or
a public hearing. The committee produces a committee report (betænkning) summariz-
ing deliberations and each party gives their stance on the bill and possible amendments.

2.3
The website of the Danish Parliament
The official website of the Danish Parliament, ft.dk, serves as the primary interface for
accessing legislative information. Furthermore, it’s one of the primary tools that Mem-
bers of Parliament and their advisors use to get information. It contains weekly agendas,
information on the bills being processed, interface to search for documents along with
information on committees and MPs.
2.3.1
"Tinget"
Members of Parliament and their advisors have access to an applicaton on either their
iPad or computer called ”Tinget”. This application utilizes the same data as ft.dk, how-
ever is presented in a different manner. Due to the sensitive nature of the application,
screenshots are not provided, but the core functionality is:
• Tracking the agenda for a single politician.
• Accessing documents that are on the meeting agenda.
• A timeline over which bills are being processed and what stage they are in.
2.3.2
Open Data
The open data project from the Danish Parliament was launched in 2014 and is accessible
via oda.ft.dk. An API using OData Version 3 Protocol [14] is provided for public use, and
a nightly database copy is also made available [15]. This dataset is the primary source of
information for this project, with additional documents retrieved according to the policies
for the use of the data. As the documents can contain copyrighted material, the documents
directly tied to the parliamentary proceedings were used in this work.
Dataset Description
The dataset is primarily composed of metadata:
• Current and past ”cases” (proposals, questions)
• Individual votes.
• Metadata regarding documents (questions, consultation notes, reports).
• Parliamentary membership records.
• Agenda
With the minutes of official parliamentary meetings being available to download from an
FTP server in XML format.

Core Entities
The current data model is structured around four main entity types:
• Aktør (Actor): Represents people and organizations involved in the parliamentary
process, such as politicians, parties, committees, etc. Actors can have relationships
with each other, and play various roles in relation to meetings, documents, cases
and votes.
• Sag (Case): Represents a law proposal, a proposal for parliamentary resolution,
a parliamentary debate or a question to a minister (§20 spørgsmål) for verbal or
written response.
• Møde (Meeting): Represents parliamentary meetings and sessions, both in the ple-
nary chamber and committee meetings. Meetings have participating actors, agenda
items, and are related to time periods.
• Dokument (Document): Represents various documents used in the parliamentary
process, such as the bills themselves, consultation reports, questions, etc. Docu-
ments are categorized, have a status, and are related to actors, cases, meetings
and subject keywords.
The key relationships are:
• Actors: participate in Meetings, are involved in Cases, create and interact with
Documents, and cast individual Votes in Polls.
• Meetings: have Agenda Items which are related to Cases and Documents.
• Cases: go through Phases, each of which may be related to Actors and Docu-
ments. Cases may have related sub-cases, most often if a law proposal is split into

• Documents: are related to Actors (authors, recipients, etc), Cases, Meetings (via

Documents
The content of the documents themselves are oftentimes not available directly in the avail-

the original PDF document.
2.4
Natural Language Processing and Information Retrieval
Natural Language Processing is a subfield of Artificial Intelligence (AI) focused on en-
abling computers to understand, interpret, and respond to human language. Historically,
NLP revolved around rule-based systems, manually defined grammars, and symbolic
representations of language. Modern NLP emphasizes statistical and machine learning
techniques, with deep learning architectures dominating the field. Over the last decade,
transformer-based models such as BERT (Bidirectional Encoder Representations from
Transformers)[16] and more recently GPT (Generative Pre-trained Transformer) [17] have
significantly advanced core NLP tasks including text classification, sentiment analysis,
named entity recognition, and summarization.
In parallel, Information Retrieval research has focused on storing, indexing, and retrieving
textual data efficiently. Classical IR systems often rely on keyword-based methods such
as TF-IDF (Term Frequency – Inverse Document Frequency) and BM25, which match
user queries to document indexes by counting word overlaps. With the rise of deep learn-
ing, neural IR methods have increasingly incorporated semantic representations of text,
enabling retrieval systems to go beyond simple keyword matching and leverage contex-
tual similarities in information retrieval [16] [9]. This change is especially relevant when
searching large document collections, as it enables a better understanding of user intent
and textual meaning.

3
Methods
This project employs a mixed-methods approach combining qualitative user research with
quantitative system evaluation. The methodology encompasses three main areas:
1. User Experience Design: Using the double diamond framework, semi-structured
interviews were conducted, which informed the development of user personas and
user journeys to understand information retrieval needs in parliamentary proceed-
ings and the requirements for an improved system.
2. Software Engineering: Agile development practices for implementing a high-fidelity
prototype.
3. Natural Language Processing: Vector embeddings and semantic search tech-
niques for information retrieval
3.1
User Experience Design
As outlined earlier the user experience design was initially based on personal observa-
tions. To mitigate the risk of personal bias, user interviews were conducted with a group
of users for the Danish Parliament’s website so to better understand their workflows and
their pain points. Building upon the Double Diamond framework, the approach initally
explored user challenges and then converged on targeted solutions.
3.1.1
Double Diamond Methodology
This project utilized the Double Diamond methodology[21] as the foundation for the User
Experience Design process, to facilitate both the exploration of the various types of user
needs and the refinement of possible solutions. In the initial Discover phase, various
ideas for the project were explored based on the initial personal observations. These
findings were then synthesized in the Define phase through interviews with stakeholders
and turned into user personas and user journeys, enabling the prioritization of system
features based on user needs and data model requirements. The Develop phase involved
iterative prototyping and the Deliver phase focused on implementing and evaluating the
developed features.
3.1.2
Semi-Structured Interviews
The project began with semi-structured interviews to validate and refine the initial user ex-
perience observations. While a formal survey was not conducted, these interviews served
to provide insights into the challenges faced during parliamentary information retrieval and
were conducted with the following group of stakeholders:
• Participants: 4 Members of Parliament, 8 advisors (current or former), and 4 en-
gaged citizens.
• Process: Participants were asked open-ended questions about their workflows,
information retrieval habits, and the pain points they observed in tasks related to
parliamentary information retrieval. Then the participants were asked specific ques-
tions related to my observations to see if they agreed with my observations or had
any additional insights.
Interview Questions
The interviewees were asked the following questions:
• Do you use the Danish Parliament’s website to find information?

• Do you think it is easy to find information on the Danish Parliament’s website?
• Do you think it is slow to find information?
• How do you usually find information on the Danish Parliament’s website?

• Are you irritated that everything is in PDF format?

3.1.3
User Personas
Building upon the interviews, user personas with supplementary stories were developed
to represent the most common stakeholder types, such as Members of Parliament, policy
advisors, and politically engaged citizens. Journalists were also considered as a stake-
holder type, but were not interviewed in the process, so they were not included in the
personas. These personas were created to capture motivations, pain points and typical
tasks, which helped to prioritize the features and functionality of the system. By anchoring
design choices in realistic archetypes, the process ensured that functionality and interface
decisions were always mapped back to specific user requirements.
3.1.4
User Journeys
To complement the personas, user journey mapping outlined the typical steps a persona
might take to locate and synthesize parliamentary information in the current system. The
focus here was on a member of parliament and policy advisor, as they are among the
most active stakeholders in the parliamentary process. This was to further gain insight
into the current workflow and identify pain points.
3.2
Software Engineering
The implementation followed established software engineering practices to ensure main-
tainability, scalability, and reliability of the prototype system. Given the exploratory and it-
erative nature of this project, a combination of Agile principles[22] and prototyping method-
ologies were chosen over a traditional Waterfall approach as it allowed for a more flexible
and user-centric approach to the development of the prototype.
3.2.1
Agile Development Principles
The development process was conducted through iterative development for various parts
of the prototype. While not strictly adhering to a specific Agile framework like Scrum, the
project tried to incorporate Agile concepts.
The project used an iterative and incremental approach to development, allowing for ad-
justments in the scope based on user feedback and emerging insights. User needs were
central to defining the priorities, as user interviews and subsequent user personas and
journeys directly informed feature selection and prioritization. Throughout the process,
the project was flexible and adaptable, incorporating new findings and addressing shifting
requirements as feedback was collected and technical challenges arose.
3.2.2
Prototyping Methodology
Constructing a high fidelity prototype was the main method for validating specific user in-

3.3
Natural Language Processing and Information Retrieval
The search and retrieval capabilities here were built on semantic search through vec-
tor embeddings with the goal of enabling retrieval based on meaning rather than merely
keywords.
3.3.1
Vector Embeddings
Vector embeddings represent text as dense numerical vectors in a high-dimensional space.
In this project, each document (or document chunk) is encoded using the Danish BERT
model [23]. The model’s final-layer embeddings are mean-pooled to create a single vec-
tor representing the semantic content of the chunk. These embeddings are then stored
in a PostgreSQL database with the pgvector extension.
3.3.2
Preprocessing & Tokenization
Before embedding, each piece of text is:
1. Lowercased and stripped of HTML tags.
2. Cleaned of Danish stopwords.
3. Tokenized with the Danish BERT model’s tokenizer (WordPiece subword tokeniza-
tion [24]).
For longer documents, text is split into smaller chunks of up to 512 tokens (with op-
tional overlap for continuity). This helps preserve performance and stay within model
constraints.
3.3.3
Semantic Similarity
The semantic similarity is computed using the cosine distance:
cosineDistance(⃗a,⃗b) = 1 −

∥⃗a∥· ∥⃗b∥
A smaller cosine distance indicates stronger semantic similarity. When querying the Post-
greSQL database with pgvector, the operator <=> (cosine distance) can be used directly,
or the cosineDistance function can be integrated into the application to compare query
embeddings with the stored document embeddings.
3.4
LLM Usage
Throughout this project Large Language Models (LLMs) were used as assistive tools com-
prehensively. During UX design LLMs were used to generate the initial user personas
from spoken transcripts. During software development, LLMs were extensively utilized
for coding tasks through code generation, code completion and debugging assistance.
They were also used discuss the various choices for the system architecture and the
technology stack. The editor used for this was Cursor [25], a fork of the VSCode editor
developed by the company Anysphere that indexes the codebase along with library doc-
umentation and provides code generation, code completion and debugging assistance.
The LLMs used were GPT-4, GPT-4o, o1 and Claude 3.5 Sonnet.
In the writing process, they aided primarily with refining prose from spoken transcripts and
notes throughout the process. They also gave suggestions for rewriting text, summarizing
research papers and creating LaTeX figures. All text modified by LLMs was verified by
the author to ensure the accuracy of the content.
LLMs have many limitations, among which are hallucinations or factual inaccuracies.
They were attempted to be used more as advanced autocomplete instead of a substi-
tute for reasoning. Citation accuracy was also maintained through manual verification.

4
User Experience Design
This chapter presents the outcomes of the user experience design process undertaken
to improve the usability and access to parliamentary information. By applying a user-
centered approach and the Double Diamond framework, the design process tried to iden-
tify core user needs, derive requirements, and propose a prototype addressing the pain
points uncovered through interviews, workflow analyses, and user journey mapping.
4.1
Key Findings from User Interviews
User interviews confirmed several key challenges in using the Danish Parliament’s web-
site, for legislative information retrieval.
These issues broadly fall into problems with
search, navigation, and general information overview:
4.1.1
Difficulty in Locating Information
Interviewees reported difficulties using the existing ft.dk search capabilities, especially
struggling when unsure of the precise terminology. Furthermore, the search does not
extend to meeting transcripts, requiring users to first locate the relevant proposal before
finding related discussions. This makes it hard to find information quickly, especially when
users only know a bill by its number (e.g., L98).
4.1.2
Excessive Navigation
Once a proposal was found, the next challenge lay in accessing relevant supporting docu-
ments (amendments, written questions, consultation responses). Interviewees mentioned
opening multiple PDF files to trace related events, which meant a lot of clicks and uncer-
tainty about whether all key documents were located.
4.1.3
Limited Overview
Interviewees expressed frustration with the scattered nature of information across multi-
ple documents, emails, and web pages. This fragmentation makes it time-consuming to
understand the full context of a legislative issue and increases the risk of missing crucial
details. Users currently rely on direct website navigation, keyword searches, and man-
ual document handling (downloading and cross-referencing PDFs) – workflows that are
inefficient and prone to overlooking information.
From a User Experience (UX) Design perspective, this fragmentation of information raises
several concerns, as discussed in [26]:
• Discoverability: Users struggle to locate key documents quickly, as questions and
answers are buried in separate documents.
• Efficiency: Users must open and browse multiple files with limited navigation ca-
pabilities, resulting in complex workflows.
• Integration: Lack of an integrated view makes it difficult to connect themes or track
the progression of parliamentary discourse.
Overall, the interviews confirmed that the core pain points stem from suboptimal search
functionality, excessive navigation and an interface that hides information and necessi-
tates multiple tabs or travesing back and forth between pages and documents.

The user locates the bill on ft.dk, which is typically done by either using the search function

The user opens multiple documents containing the bill text, background material, and

All references and supporting documents are compiled manually into a speech for the

4.4
User Experience Improvements
Based on the interviews, user personas, and user journeys, several themes emerged as
priorities for the prototype development. The main being effecient information search and
retrieval. To achieve this the following core UX improvements were identified:
• Enhanced search: Improve upon the current search capabilities.
• Display content in context: Present documents, questions and answers directly
in the context, instead of in separate PDF documents.
• Streamlined navigation: Reduce clicks required to access common information.
These priorities guided the formulation of prototype requirements, influencing both the
underlying data model (described in Chapter 5) and the overall system architecture. The
sections that follow detail the proposed improvements and outline the functional require-
ments.
4.4.1
Enhanced Search
A key improvement is a more powerful search interface, enabling users to locate docu-
ments and transcripts efficiently. This includes:
Mentionable Search Interface.
An iterative design approach explored a “mentionable” search component, where users
can specify particular politicians, committees, or ministries via keyboard shortcuts, in ad-
dition to the regular search query. Figure 4.2 illustrates how users can quickly refine their
search by tagging the relevant actors. This functionality requires a robust data model
capable of storing relationships between actors and documents.
Searching in Transcripts.
Currently, transcripts are not accessible through the existing search interface. Extending
the new interface to include full-text transcript searching addresses this gap. Implement-
ing such functionality also entails a data model that can store and index the transcripts for
fast, responsive search queries.
Displaying Content in Context
To present parliamentary documents and transcripts where they are most relevant, the
system must be able to fetch and parse them efficiently. Rather than forcing users to
open separate PDFs, the interface would embed contextually relevant segments directly
within search results or the overview of a bill. This approach streamlines navigation and
improves comprehension by keeping users within the same workflow. Technical details
on how this content is stored and served are discussed in Chapter 5 and Chapter 6.
The data model that underpins these features is explained in detail in Chapter 5.

5.1
Reﬁned Data Model
To support the requirements of the system, the data model was refined to include the
following entities:
• Entity: taleSegmentRaw
The taleSegmentRaw table captures the raw text content of each speaker’s contri-
bution (speech segment) during parliamentary meetings. It stores references to the
meeting, the speaker (actor), and optional references to related cases (if applicable).
• Entity: taleSegmentChunk
The taleSegmentChunk table stores smaller, embedding-friendly chunks of each
speech segment. By splitting long speeches into chunks, we facilitate more efficient
vector-based similarity searches (e.g., for semantic retrieval).
• Entity: filContent
The filContent table fulfills a similar role as taleSegmentChunk but for uploaded
files or documents (rather than transcribed speeches).
• Entity: DocumentContent
The DocumentContent table provides a higher-level abstraction for storing core tex-
tual information derived from documents.
5.2
Data Processing Pipelines
Three main pipelines were constructed to handle different aspects of the data:
1. Metadata Pipeline: Processes structured data from the Open Data API, including
case information, actor relationships, and document metadata.
2. Document Pipeline: Handles the extraction and preprocessing of documents.
3. Meeting Transcript Pipeline: Processes the parliamentary transcripts from the
meetings, extracting the speech segments and storing them in the database.

6.1
System Architecture
The selection of the technology stack for the Open Parliament prototype was a critical
design decision, driven by the project’s requirements for performance, maintainability,
and type safety. Initially, Prisma[28] was implemented for database interactions. How-
ever, challenges were encountered with character encoding for table names, specifically
with Danish characters (æ, ø, å), and limitations were anticipated in handling complex
database queries involving numerous relational joins, which are inherent in the parliamen-
tary data model. Consequently, Drizzle ORM[29] was adopted as the preferred Object-
Relational Mapper. Drizzle ORM offered a more SQL-centric approach, providing closer
control over query construction while still ensuring full type safety, which is crucial for
managing complex relational data with integrity.
Furthermore, the choice of Nuxt.js for the frontend framework was motivated by its server-
side rendering (SSR) capabilities. Leveraging SSR allows for the pre-compilation of appli-
cation pages into static HTML documents. This approach offers significant performance
advantages by enabling efficient caching of content, particularly for the overview of a par-
liamentary case, while dynamically generated, can be cached and served statically for
improved responsiveness. This aimed to enhance user accessibility and reduce server
load, potentially mitigating the need for aggressive rate limiting or DDoS (Destributed De-
nial of Service) protections on the entire website–although appropriate rate limiting mech-
anisms would still be necessary for dynamic elements. This performance consideration
is particularly relevant for public-facing websites aiming for broad accessibility and rapid
information delivery.
To optimize application performance and responsiveness, asynchronous data loading
strategies were implemented throughout the Open Parliament prototype. Asynchronous
operations help preventing blocking of the main thread and to ensure a smooth user ex-
perience, particularly when fetching data from external APIs or performing computation-
ally intensive tasks. This asynchronous approach is further complemented by Nuxt.js’s
server-side rendering (SSR) capabilities. As previously mentioned, SSR allows for the
generation of static HTML pages, which can be efficiently cached at the server level or
even on a Content Delivery Network (CDN). This caching strategy is particularly ben-
eficial for legislative cases, meeting agendas, and other parliamentary information that
can be pre-rendered. By serving static HTML for these content types, the system could
achieve significantly faster page load times and reduce server-side processing overhead,
enhancing the overall user experience and system scalability. While rate limiting is still
a necessary consideration for managing API access and preventing abuse, the efficient
caching enabled by Nuxt.js’s SSR reduces the reliance on aggressive rate limiting mea-
sures for general website access.

Discussion
8.1
Project Outcomes and Limitations
Despite the initial ambition to improve user access to legislative information, the project
encountered significant challenges that limited its success:
• Too large of a scope, which was not clearly enough defined.
• Not enough iterative development with constant feedback from users on targeted
functionality
• An overreliance on LLMs for the coding tasks turned into an issue when the project
scope grew too large.
• Attempts at transformer-based embeddings yielded imprecise and often irrelevant
results.
The project further more lacked iterative development with constant feedback from users
on targeted functionality. This would have allowed for a more focused and effective de-
velopment process, ensuring that the prototype met the needs of the target users.
8.2
Possible technical improvements
8.2.1
Domain-Speciﬁc Model Training
While the current implementation uses a pre-trained Danish BERT model that is not yet
working, training or fine-tuning the model specifically on parliamentary documents could
improve performance. As noted by [9], domain-specific BERT models often outperform
general-purpose models for specialized tasks, and at a lower deployment cost compared
to larger language models. This could be particularly beneficial given the unique vocab-
ulary and discourse patterns present in legislative texts.
The specialized nature of parliamentary language, including formal procedures, legal ter-
minology, and structured debate formats, suggests that a fine-tuned model could better
capture these domain-specific nuances and improve retrieval accuracy.

Conclusion
In its current form, the Open Parliament project remains far from a reliable, functional
platform. Technical difficulties due to ambitious scope and overreliance on LLMs resulted
in underdeveloped features prevented the project from being a functional prototype.
Nevertheless, the project has laid groundwork for integrating parliamentary documents
into a single, searchable platform and has highlighted essential next steps. With ex-
panded data acquisition, iterative development, and enhanced semantic capabilities, a
refined version of the prototype could one day advance transparency and engagement in
Denmark’s parliamentary proceedings.

B.1
Extended Meeting Parser Explanation
B.1.1
Parser Flow and Data Extraction
The parser flows through the following steps:
1. File Discovery: Collects XML files from a specified directory.
2. XML Parsing: Utilizes the Fast XML Parser library (fast-xml-parser)[30] to con-
vert XML into nested JavaScript objects.
3. Meeting Metadata Extraction: Calls extractMeetingMetadata to retrieve over-
all meta-data from MetaMeeting, aligning the parliamentary session and date with
database IDs (periode, møde).
4. Agenda Items and Sub-items: Iterates through each DagsordenPunkt.
• Resolves a matching case ID (sagId) in the database (if it exists) by referencing
FTCaseNumber and FTCaseType.
• Recursively processes DagsordenUnderpunkt sub-items with the same struc-
ture.
5. Speech Segments (Taler): For each Tale node:
• Extracts the speaker (aktør) by ID or name.
• Retrieves and cleans speech text from TekstGruppe.
• Persists full-text data to the taleSegmentRaw table.
• Splits text into chunks and sends them for embedding to an external service.
These embeddings are then stored as taleSegmentChunk entries.
6. Stats Tracking: The parser maintains a ParsingStats object that records:
• Total processed meetings
• Agenda item successes/failures
• Lookup results for sag (case) and aktør (actor)
B.1.2
Key Data Structures
The central interfaces include:
• MeetingData: Holds the metadata for the meetingand the full list of agendaItems.
• AgendaItem & SubItem: Each sub-item references the same basic fields (e.g., FTCaseTingdokID,
ShortTitle) but may recurse further, depending on the meeting.
• Tale: Represents the final extracted speech content, along with any embedding
indices, timestamps, and references to the speaker or case.

B.1.3
Database Integration
Multiple functions query and populate the postgres database:
findMødeId: Locates the correct meeting row in the møde table by matching the parlia-
mentary session (periode) and meeting number.
findSagId: Resolves the corresponding case ID (sagId) from FTCaseNumber and FTCaseType.
processTaleSegments: Writes raw speech data, splits content into chunks, and records
embeddings for each chunk.
By modularizing the logic (extracting IDs, segmenting text, searching for matching database
records), the parser achieves a clear separation of concerns. This ensures it can be ex-
tended to handle new cases or more complex input formats in the future.
