Summary (English)
The goal of the thesis is to develop a modelling language, EVent-B Oriented
Language (EVOL), for providing an easier way to create models in the modelling
notation Event-B. It will be possible to formulate models of simpler algorithms
for concurrent programming, and to translate the model into an Event-B model,
that can be checked using the automated theorem provers in the Rodin Platform.
The report is structured in six parts:
The rst part will give an introduction, and motive the need of the development
of the language.
The second part gives a brief introduction to Event-B.
In the third part, a formal denition will be given of the syntax and semantics
of the EVOL language.
The fourth part will contain a description of how the EVOL model can be
translation into Event-B.
The fth part will describe an implementation of a program in Scala that will
automatically perform such an translation.
In the sixth part, some examples of models that are translated will be discussed
and how they can be checked in the Rodin platform.
At last, a conclusion of the project and thoughts on further work will be given.

Chapter 1
Introduction
The topic of this thesis is the development of a modelling language for Event-B.
This includes the specication of the modelling language, the translation of the
language into Event-B, the argument for the equivalence between the original
and the translated model, and an implementation of a program for automatic
translation of a model.
The language has been given the name EVOL, an abbreviation of EVent-B
Oriented Language.
Event-B is in itself a modelling notation for specifying models involving com-
puter software. It involves dening theorems and invariants for the model and
to show that the model satises these. Proving the correctness of the theorems
can be supported by automatic theorem proving, that makes use of computer
programs to perform many of the steps. Ultimately, the goal is the to develop
computer software without any faults,1 and to provide solid argumentation for
the correctness of the program.
From the beginning of the project, Event-B has been chosen as the basis for the
translation. The thesis will thus not include comparisons with other theorem
provers or any analysis of Event-B on a deeper level. The notation will, however,

Other tools are available for developing reliable software systems. One approach
is to use test-driven development, but limitations of tests are well known since, as
Dijkstra put it, testing shows the presence, not the absence of bugs.2 Especially
for programs running in parallel, the use of tests are limited, since many faults
due to race conditions are hard to reproduce in testing environments. Some
errors may also be very rare, meaning that even a well-tested program can
crash, though it might only happen rarely.
A more thorough way of developing software systems is by using model checking.
This approach is based on modelling the software as a number of states, that
describes the values of the variables and where the control point of the program
has reached. As a program executes, it will go through a number of dierent
states. For any state, where there is a path of executions leading to this from
the initial state, the state is said to be reachable.
Formal properties of the model can be tested by automatically running through
all reachable states and check if the property hold for a given state. The result
will either be that the property is valid for the model or some path of executions
leading to a state violating the property will be given.
The inherent limitation of model checking is the brute force manner in which
all reachable states must be checked. The number of states grows exponentially
to the number of variables in the program, which leads to the problem of state-
space explosion.3 For even moderate sized models the number of states will be so
large that the running time of the model checker makes is practically unusable.
There is therefore a limit to the size of models that are in practice checkable,
even if bigger and faster machines is used.
Another limitation is that all variables must have a specic value. It is not pos-
sible to operate with undened constants that denote e.g. n processes running
in parallel. Model checking can verify that given a specic number for n, the
processes can run without problems, but other arguments outside the scope of
model checking must be provided to verify, that is also holds for n + 1 processes.
An approach that does not suer from the limitations of model checking is to
base the proofs on theorems. This will neither provide some number of states to
be checked nor require that constants have a specic value. It only needs to be
shown that no event in the model will cause the theorems to be violated. This

The notation of Event-B is based on a mathematical approach to verication,

However, this means that the Event-B notation itself is not based on the tradi-

Given a program, that one wants to check, or an algorithm that needs to be
veried, it can be hard to create a corresponding model in Event-B, given the
dierences in notation styles.
The goal of the thesis is to provide a modelling language, EVOL, that will allow
formulation of simpler algorithms for concurrent programming. This language
will be given a formal denition. An approach for translating the models into
Event-B will be sketched, and a program for automatic translation will be de-
scribed and implemented.

Chapter 2
Event-B
This section will contain an introduction to Event-B and the functionality of
the notation. It will also look at Rodin, an integrated development environment
(IDE) for working with Event-B, that is implemented in the Eclipse Platform.1
Rodin makes it possible to perform automated theorem proofs for the models,
and is also providing addition tools for working with the model.
The introduction will focus on how Event-B is used in this thesis and not be
a general introduction. Important concepts like renement will only be men-
tioned briey, and not all of the many possibilities provided by Event-B will be
presented. For an general introduction, see [Abr10] and [JB14].
2.1
Overview
Event-B is based on B, a design method for software systems developed by
Jean-Raymond Abrial in the 1980's, with the goal of intertwining the design of
the programs with mathematical proofs for the correctness of the programs.2
The idea is that the requirements of the program are developed alongside the

2[Abr96], p. xi

mathematical proofs that shows that the model is satisfying these requirements,
an approach called 'correct by construction' by Abrial.3 The notation is based
on set theory, and an abstract machine notation is used to formulate the models
in B. The notation of Event-B is based on the notation used for this abstract
machine.
Event-B is both "a simplication as well as an extension"4 of B. One of the dier-
ences Abrial emphasises is that while B is centred around developing software,
Event-B is expanded to model the complete system, including surroundings.
In the models of Event-B, the software is only one part of the model among
others.5 This aspect reects the approach that Abrial is voicing for software
development, where the modelling is the essential part, while the programming
becomes a subtask, once the model is completed and proven to be working as
expected.6 Anything that happens in the model is a discrete event, and it does
not matter if it is a task performed in a software program, a user pushing a but-
ton or an input from the physical surroundings. Another dierence between B
and Event-B is that the latter supports a more extensive notion of renement.7
Event-B is a "discrete transition system"8, that consists of a state and a number
of events. A state is constituted by the values of the variables and constants of
the model, and an event is a scheme for transitions between two states. An event
has a guard that represents the necessary condition for the event to execute, and
any event can be executed in a state, where the guard evaluates to true. The
event atomically updates one or more variables and thus changes the state of
the system. It is a at model and there is no notion of sequentiality. If the
guards of all the events are false in a reachable state, a deadlock has occurred
and the model cannot leave the state.
For each model, a number of assumptions are assumed to hold.
These are
formulated as invariants, and reect the specications of the system. This is
what is proven by the "invariant preservation proofs"9, and what the Rodin
system will do automatically if possible. Invariants can also be used as helping
theorems for proving more complex invariants.

2.2
Overall Design
A model in Event-B consist of two types of components: Machines and Contexts.
Machines are used to model the parts of the model that are changing, like
variables, while the context is used for dening the static parts, that hold at all
times. This includes denitions of constants and sets. A model can consist of
contexts, machines or both. Here the focus will be on models consisting of one
machine and one context. Combining more components is used when working
with renements, which will be briey described later in this chapter.
2.2.1
Context
A context consists of four elements: sets, constants, axioms and theorems. The
axioms are used to dene the sets and constants, and the theorems should be
derivable from the axioms and already proven theorems.
Sets are dened by assigning some elements to the set or they can be derived
from other sets using set operations. A set can consist of simple elements or
relations between elements in sets. A special case of a relation between sets
is a mapping, where each element in one set maps to an element in the other.
Some sets are predened in Event-B, including natural numbers, integers, and
booleans.
Constants must be given a type by being dened to belong to a set.
The
denition of a constant is done by formulating one or more axiom that establishes
this using predicate logic and set theory. The constant does not need to be given
a specic value, so a constant x can be any of the elements in the set, which
makes it possible to makes models that operate with unspecied constants.
It is possible to let a set and constants in a context dene each other, so that the
denition simply is that set S contains the constants A and B, without assigning
the constants any primitive type like in a programming language.

2.2.2
Machine
A machine consists of variables, invariants, theorems, and events. It can see one
or more contexts, which makes it possible to use the constants and sets dened
there.
Variables must have an unique name and be given a type by dening an invariant
that species which set the variable belongs to. This is similar to the context.
The variable must, however, also be initialised by being given a specic value
in special event INITIALISATION, that must be included in all machines.
Events are the main part of the machine and describe a change of state. An
event has a guard that must be true for the event to execute and a guard cannot
have any side-eects. What happens in an event is one or more actions, where
variables are assigned, meaning the variable is mapped to an element. Each
event occurs atomically and only one event is executed at a time. Actions in an

event takes place simultaneously and each variable can only be assigned once.
If a variable is assigned a new value while also occurring in expressions used
to assign other variables, the assignment will be based on the value of variable
before the event is executed.
Event-B allows for non-deterministic assignments, meaning that instead of giv-
ing the variable an exact value, it can either be specied using a predicate that
the value must afterwards full or by a set, such that the value can be any
element in that set afterwards. In this thesis, only deterministic assignments
will be used.10
The special event INITIALISATION is always the rst event to be executed
and cannot have any guards. In this event all variables must be assign a value
that will function as the initial value. If a variable is not given an initial value, it
will automatically be assigned one. As an example, if the variable is an integer,
it will be set to zero and if it is a relation between two sets, it will be the empty
set. It is, however, encouraged that all variables are initialised, and for more
specialized sets it might not be obvious what will be the initial value. When
working in the Rodin platform, missing initialisations will be high lighted, so it
is clear that unexpected behaviour might occur.
Invariants are used to dene variables, and are in this way similar to the axioms
in the context, but they also need to be preserved by all events. This means that
if a variable is assigned a new value in an event, it must be possible to prove that
the invariant of the variable still holds. Invariants can also be used to formulate
other propositions, that are assumed to hold for the machine. Constants can be
included in the invariants for the machine, so denitions of variables can depend
on constants. Theorems are formulated in the same way as invariants, but it
must be possible to prove them from the invariant.

Renements
A model in Event-B can be rened.
This is supporting the general way of
developing models, that Abrial is advocating, and he is focusing on this use
of Event-B in his book Modeling in Event-B, which is the canonical work on
Event-B.
In this approach, the system is modelled in a very broad, abstract way in the
rst version and is step-wise rened to a more concrete model. In each step it
is proven that the invariants hold and that the renements of one step do not
invalidate the invariants of the previous. The idea is that this reects a problem
validation, where the problem is rened for each step and validated continually,
in contrast to a solution validation, where only the nal solution to the problem
is proven to be consistent.12
Abrial dierentiates between two types of renements: horizontal and vertical.13
Horizontal is used when more abstract model is rened to a more concrete
model. This could include adding more variables to expand the state, adding
events or strengthen the guards. Vertical renement is used when the horizontal
renements are done, so that the model is concrete enough to solve the initial
problem. It is now rened to reect a software approach, where abstract notions
from Event-B is replaced with concepts from programming languages such as
array etc. Vertical renement is also referred to as "data renement".14
Each renement of a model must match the abstract model, that it is a re-
nement of, and Event-B includes constructions that support this. When one

ProB
There are a number of plug-ins for Rodin, that are useful when working with

model in Event-B, and also provides an interactive way to test the model, where
the events can be executed event by event and the resulting states printed.
Especially when representing models of software programs in Event-B, it can be
useful to be able to execute them stepwise, to see how the model reects the

As described in the introduction, there are some limitations to model checking,
which means that not all models of Event-B can be checked using ProB. If the
number of states are too big, ProB will note this, but it is still possible to use
the interactive way of running through the steps of the model. This means that
it might be possible to prove the correctness of models in Event-B using the
automated theorem provers of Rodin, but not using ProB. It is in this way only
a helping tool for the general use of Rodin.
Since all variables must have a value in a state, ProB will assign values to all
unassigned variables. This includes the constants dened in the context that the
machine sees. Depending on the set the variable is in, it will be given a standard
value; for integers and natural numbers, this is zero. If some axiom species
requirements of the constant, these will be applied in ProB as well. That means,
that if e.g. a constant c is included in an axiom, where it is required that c is
greater than some other variable x, c will be set to x + 1 when running ProB.
This is another limitation of the use of ProB, and one should be careful not to
think that ProB has provided a proof for a general formulation of a model, if it
is only for a special case where constants has been assigned a specic value.

Chapter 3
Design of the Language
This chapter will specify the language developed in this project, rst using
an informal description and afterwards using formal syntax and semantics. A
motivation for the requirements for the language and inspiration from other
languages will be given rst. As mentioned in the introduction, the language
has been named EVOL, short for EVent-B Oriented Language.
3.1
Requirements of the Language
The purpose of the language is to provide an easy way to create models in
Event-B, and also to provide a language that is closer to a more classic way
of formulating programs in software development. This approach has two ad-
vantages. First of all, it makes it easier to create a model without having to
have a comprehensive understanding of Event-B. It will make it possible to use
the verication of Event-B without having to know all the specics and details,
since the model is generated automatically.
Second, when a model is formulated in a language that is closer to the language
of the implementation, that the program will eventually be formulated in, it
is easier to argue that the nal program is equivalent to the model. As Baier
==PAGE:26==
16
Design of the Language
and Katoen stress it: "Any verication using model-based techniques is only as
good as the model of the system".1 Since the verication is only applied to the
model, whatever properties that hold for the model will only hold for the nal
program to the extend that they are equivalent, so it is an important aspect of
model checking and theorem proving to be able to argue that this is indeed the
case.
The aim has been to make it possible to express models of simple algorithms
for concurrent programming in the EVOL language. An example of such an
algorithm is Peterson's Algorithm, that have been conrmed to ensure mutual
exclusion for a critical section for two processes.2 The language should also be
able to express a model of the expansion of the algorithm to include n process,
where n is some unspecied natural number. It should in general be possible to
operate with constants, that have some unspecied value that will not change
during the execution.
3.1.1
Types
A fundamental type that is necessary for formulating any interesting algorithm
is integers. Without this fundamental notion of numbering, most algorithms
would not be expressible, and it has been natural to include this as a type in
the language. Another important aspect, especially for algorithms containing a
number of processes, is arrays. These are often used for the structuring of an
algorithm, so the language will also include arrays of integers as a type.
Considering the sets already build into the Event-B notation, integers will in-
clude the subtypes non-negative numbers and positive numbers, and they will
generally be handled indiscriminately in the language. That e.g. a non-negative
integer will never be assigned a negative value must be checked afterwards,
which can be done using Event-B.
The types could be extended to include oating point number and other types
of numbers as well, but for many simple algorithms, this has not been deemed
necessary. It is not easily supported by Event-B either, and this have been taken
into consideration as well. Boolean types has also been considered, but since
these are easily implemented using integers, other aspects has been prioritized.
The language will be an imperative language, where a state will consist of the
value of all variables. This diers from Event-B's event-driven paradigm.

3.1.2
Control Structures
Most algorithms for concurrent programming assumes some control structure,
that will allow for sequentiality. Branching and loops are also commonly found
in most software programming languages, so it should be possible to use these
kind of control structures as well. Otherwise, the language would often fall short
when common structures were to be implemented.
Indeed, the at structure of Event-B without any sequentiality is one of the main
reasons that the formulation of models in Event-B diers so much from more
classic programming languages. The lack of this is one of the main motivations
for making a language language for Event-B. This means that EVOL should
support sequentiality between statements, branching and loops.
3.1.3
Concurrency
Another aspect is expression of concurrent execution in the model. It should be
possible to have dierent parts executing independently, and the ability to check
for lack of race conditions and other problems arising in a concurrent setting is
one of the main focuses of EVOL.
It is possible to formulated a language where the concurrency is allowed on a
very low level and even simple statements can be formulated as executing con-
currently. However, to reect a more commonly found approach in programming
languages and to simplify the handling of concurrent statements, only top-level
parallel compositions will be allowed in this language. It will be handled using
processes, where each process is executing independently.
Concurrent executions gives rise to several other aspects:
¬ą One part of the model should be able to be stopped while waiting for some
condition to be true. Here boolean statements are used, that will function
as await statements.
¬ą The dierent parts of the model that runs concurrently should be able
to communicate and synchronize. Generally there are two common way
to do this: either using channels or shared variables. With channels, the
dierent parts of the program may synchronize on the calls, and they
can be used to pass variables. However, in the EVOL language the last
approach has been chosen, where global variables can be declared that
==PAGE:28==
18
Design of the Language
all process can see. Dierent processes can communicate by reading and
writing to these variables.
3.1.4
Atomicity
In continuation of the aspect of concurrency is the aspect of atomicity in the
execution of the statements. The execution should happen in suciently small
steps, so that this can reect the atomicity found in computers running software
in concurrent executions, and will not suppress interleavings that will be possible
when dierent scheduling paradigms are applied to the program. It should, on
the other hand, be possible to specify statements to be executed atomically
when necessary. This will allow for implementation of semaphores or switch
statements and generally make it possible to express structures guarding critical
sections or other concepts found in concurrent programming.
One special case, where atomicity is important, is in the assignment of variables.
Here interleavings of concurrently executing processes might lead to race condi-
tions, and to avoid to suppress this the reading and writing of critical references
in an assignment statement should be split up.
To simplify the translation into Event-B, the atomicity has been limited to
letting evaluations of expressions being done in a single atomic step, so that
all variables will be read simultaneously. This construction is ignoring that in
software executed on a computer each reading of a variable will be an atomic
step, so the state could change meanwhile. To capture this, another choice would
have been to split up the reading of each variable into separate events, but this
would caused a lot of events to be created in Event-B. The reading of non-critical
variables or numbers would also be superuous to put into separate events.
Another solution would have been to split up the evaluation of the expression
into parts that only contained one critical reference each.
This would keep
down the number of generated events, while being consistent with the atomicity
expected to be found in a regular programming language. This would, however,
require an analysis of the arithmetic expression during to translation, and to
keep it simple this has been left out.
This simplication of the atomicity is also applied to reading elements in an
array, where the expression specifying the index of the array might contain
critical references. This is handled as a single step as well, to keep the number
of events down and keep the model clear. If the reading of the variables needs
to be split up, it can manually be done when creating the model, so there is no
limitation of the programs that can be created in the model.

One inspiration for the EVOL language is Guarded Command Language.3

statement can be placed inside an if or do statement, not just boolean state-

statements where the guard was trivially true.

executed, the do statements will not be skipped. To break out of a do statement,
the specic break statement must be used, and if no guarded command can be
execute the process will simply wait.

3.4.5
Comments on the Semantic Rules
Some special constructions are used to specify the semantics of specic state-
ments. These are notations that do not occur in the language itself, but are
used as intermediate step in the semantics.
The do-repeat construction in the rules for the do statement is specifying that a
guarded command inside the do statement is executing, and when it is nished,
the guarded commands should be repeated as a new do statement. However,
if a break statement is found inside the guarded commands, both the currently
executing guarded command and the following do statement are nished execut-
ing. The resulting state is simply the state that the break statement occurred
in.
The special construction for assignments is used to dierentiate between the
case where the variables of the arithmetic expression are read and the complete
expression is evaluated ([Ass1]), and the case where the assigned variable is
mapped to the result of this evaluation ([Ass2]). This is to capture this as two
dierent atomic events in the language.

4.1
Assumptions About the Model
When translating a model into Event-B, it is assumed that some basic criteria
of the model are fullled. These assumptions are checked when translation the
model, and how this is done in practice will be described in the next chapter
about the implementation.
Technically, this means that some models that can be expression using the
syntax of EVOL cannot be translated into Event-B using the method stated
here.
Some criteria, like requiring that all variables should be initialised, is
necessary, since translating a model not satisfying these would cause errors

statements, are to prevent translation of models, that might express undesired
behaviour. Such a model could potentially be formulated in Event-B, but here
they are handled as erroneous.
Most of the assumptions stated here are the same that was assumed to hold
for well-formed models in EVOL in the last chapter. The only dierence is two
more assumptions has been added about statements inside atomic statements:

Event-B is, after all, to get help to prove invariants of the model, so some
properties have been natural to place here. Other problems, like uninitialised
variables, would simply also cause an error in Event-B, so even though they
would in a way be caught, it seems strange to allow translations that are erro-
neous. Since Event-B in many cases rely on helping theorems, assumptions can
also be handled by creating invariants in the Event-B model. This is a point I
will return to in the discussion in the end of the thesis.
It is known that static analysis has some limitations, so for some assumptions
it is simply not possible to check if the model full these before doing the
translation. One limitation is the inability to decide if a program with a given
input terminates or not. Since the termination of a program might inuence
the value of variables, and these variables might be used to assign values to

4.2
Processes
A process consists of a number of instances, that each should be able to execute
independently. Since it is assumed that the denition of the process does not
contain any variables, the number of instances is a nal value. Each instance
will have a unique number in the process, which will be the process id, here
referred to as the PID.
To let the instances execute independently, a mapping is used, where each PID
is referring to a specic control point, that represents where in the execution
the instance is at. The mapping is implemented in Event-B as a variable and
given the name of the process. In the initialisation event of the model, all PID's
are set to point to zero, as this will always be the rst control point in the
process. As an instance progresses and executes the events of the process, the
PID is updated for this specic instance, while the others only change when they
themselves execute events. In this way the executions of the process instances
are completely independent.
An assignment is split up into two dierent events, one that evaluates the arith-
metic expression and one that assigns the variable. This means that each process
must have an temporary variable in which it can save the evaluated value. It is
only necessary with one such temporary value for each process instance, since
an instance can only perform one statement at a time, so for each process a
temp variable is dened at the same time. The PID of the process is used as
index to this variable, so each instance has its own value.

4.3
Denitions
All variables and constants dened in the model must also be dened in Event-

except for the local variables that need to be distinguishable from both the
global variables and variables with the same name in other processes.
The global variables and arrays are initialised using the special initialisation
event, that every model in Event-B must have. For the local variable and arrays,
they are initialised as the rst event of the process. One reason for this is that
the local variable and array might use the PID in the denitions, and since this
value is known when executing the initial event of the process, it is easier to
implement the initialisation as an independent event. More importantly, it is
also a question of interpreting when the initialisation takes place. If a local
denition is using a global variable and all initialisations would take place as
the rst event of the model, all local denitions would use the initial value of
the global variable. In this solution, the value of the global variable might have
changed before the initialisation event of the process is executed, so the depends
on the state of the system at that moment.

Since Event-B has a at structure, the guards of the events must be used to
structure the model in such a way, that the events can only be executed in the
right sequence.
It is important that even though the statements in each process should be ex-

every interleaving of the executions, that is possible in EVOL, must also be pos-

potentially be veried in Event-B, but not work when implemented in a real
concurrent system.

boolean guards for the event that represents this statement, so that it can only
execute when the conditions are satised. This approach requires exhaustive
analysis of the program and is hard to combine with atomicity in concurrent
executions, since it can be hard to dene unique pre- and postconditions for all
statements.

process id to a specic point in the process. Each event will have a parameter

4.4.2
Assignment statement
An assignment statement in EVOL is similar to an action in Event-B, where a
variable is assigned a value of an arithmetic expression. All actions in Event-B
are executed atomically, however, so in order to capture this aspect, an assign-
ment will be split up into two events. One that evaluates the arithmetic ex-
pression and saves the result in a temporary variable, and a second that writes
the value of the temporary value into the assigned value. This is to simulate
the non-atomic nature of a regular assignment, where the reading and writing
of variables does not take place at the same time, so the execution might be
stopped and another process get to execute in the meantime. The atomicity of

the events in Event-B requires that several events need to be used to simulate
this.
As mentioned in the specication of EVOL, all variables will be read simulta-
neously. Otherwise, an event would have to be created for each variable read in
the arithmetic expression, resulting in an assignment statement being translated
into as many events as there are variables. If it is needed that the reading of
variables should happen in dierent events, it can be stated explicitly in the
model in EVOL.
The assignment will only be split up, if there are a critical reference in both
the expression that is read and the variable that is being assigned. A critical
reference will in this context be a reference to a global variable. The reason
for this is that with only one critical reference the statement will appear to be
atomic, since the only important part for the atomicity is this reference. In this
case the assignment can simply be reduced to one event. The same goes for
assignments placed in an atomic statements. This will reduce the number of
events created in Event-B and make the model more clear.

To create two sequential statements, the last event of the rst statement should
update the process id to point to the rst event of the next statement. The
sequentiality is thus not implemented as an independent event in itself, but is
only concerning how the process id is set in the process.

The boolean expression is split up into two parts. First part is the event that
evaluates the boolean expression. The boolean expression is set as a boolean
guard of the event, so it is only possible to execute this event when the boolean
guard evaluates to true. The action of this event is simply to update the process
id to the point of the rst event in the boolean statement.
This splits the
evaluation of the boolean expression and the execution of the statement into
two dierent parts as it should be.

Like the sequential statement, an if statement is not implemented as an event

An if statement consists of a number of guarded commands, and all the start
events of each of these guarded commands is given the same point in the process.
This means that the state guard will allow any of the events to execute when a
process id points to this value.
If the rst statement in the guarded command is a boolean statement, the rst
event will also include a boolean guard similar to the boolean expression of this
statement, so the boolean expression must be true for this event to be able to
execute. This insures that guarded commands with boolean statements will only
be 'chosen' if the guard allows this. Other statements will not have a guard,
which will correspond to having a guard that is always true.
The last event in each of the guarded commands will update the process id to
the point after the if statement, so no matter which event of a guarded command
that has been executed, the process id will always point to the same value after
the if statement.
4.4.6
Do Statement
A do statement is almost identical to an if statement. The only dierence is how
the process id is updated in the last event in each of the guarded commands.
Where in the if statement it is set to the point following the statement, in the do
statement it will be set to the same value as it had when starting the statement.
This means that whatever events could be executed when starting the statement
are now enabled again.
4.4.7
Break Statement
The break statement is simply updating the process id to point to a new point
in the process array. This new point is whatever comes after the do statement,
that the break statement is inside of. If it is inside several do statements, it
will be the point after the innermost do statement. Since it is assumed that the

point.
4.4.8

When executing the atomic statement, only the events of this statement should
be allowed to execute. This is ensured by introducing a global lock in Event-B,
that is simply a boolean, initially set to false. This represents that the lock is
not taken.
For the atomic statement, a start event and an end event is created. The rst is
to grab the global lock. This should only be possible when the lock is free, so a
boolean guard is that the lock must be false to execute this step. This event will

race conditions. For an atomic statement in a process P, where the statement

Skip Statement
A skip statement is implemented as a simple event, that updates the process id
to point to the next number.
It seems tempting to simply ignore skip statements in Event-B, because they
seem to do nothing. There are two reason not to do this. First of all, the Event-
B model should reect the EVOL model, so if one contains a skip statement,
it might be confusing to leave it out in the other. It is also sometimes the case
that the program behaviour is changed by a skip statement. An example of
this is if the rst statement in a guarded command inside an if statement is a
skip statement, while the next is a boolean statement. This is the case in the
following if statement:

fi
When at the if statement, any of the guarded commands might be chosen to
be execute, except when starting with a boolean statement where the guard is
false. If the skip statement was not there, the second guarded command would
always be chosen if x was greater than 5. In this example, however, the rst
command can still be chosen, but if x is creater than 5, the process is simply
stuck. This execution should be similar in Event-B, and if the skip statement
was not implemented as an independent event, it would not be possible.

5.1.1
The Lexer
The lexer is iterating over the input stream and reading the characters into a
string. Whenever a special character (parenthesis, arithmetic comparison etc.)
is encountered, the string of what has been read so far will be processed as a
token along with the newly read character. The same happens if a whitespace
character is read, except this character is simply skipped.
If what has been read so far is a keyword, a special token will be created,
depending on the type of keyword.
Examples of keywords are 'if', '', and
'skip'. If it is not a keyword, but ts the pattern for a variable token, it will be
read as a variable. The pattern for a variable is any string starting with a letter
and followed by any combination of letters and numbers. For an input that does
not match any pattern, an 'unknownSymbols' token will be outputted.
A special EndOfFile token is added in the end, to be used in the parser.
5.1.2
The Parser
The parser constructs the abstract syntax tree from the stream of tokens. In this
implementation, it is not actually a tree, but a list of denitions and processes.
The denitions contains trees of arithmetic expressions while the processes con-
sists of a list of denitions (these are the local denitions of the process) and a
statement, which is a tree of statements. The syntax tree is in this way a list of
trees.
The reason for this implementation is that the denitions are not handled as
statements, since they are all placed in the beginning of the program. Imple-
menting them as a list makes it easier to get all the names of the variables and
constants that are dened. The processes could also be implemented as a tree,
but since each process is completely separated from the others, a list has been
deemed a better data structure.
Another way the generated abstract syntax tree diers from a traditional imple-
mentation is in the generation of arithmetic expressions. Normally it would be
required of compositional arithmetic expressions to be split up depending on the
precedence of the compositional sign, e.g. multiplication should have a higher
precedence than addition. This means that two expressions in a compositional
arithmetic expression consisting of a addition sign might contain multiplications,
but not the other way around, unless parenthesis were used. Since the purpose
of the language is to be translated, the arithmetic expressions are never eval-
==PAGE:56==
46
Implementation
uated, so it has not been necessary to use this construction. This means that
there is no structure of the arithmetic expressions based on precedence, except
that it is right associative. When translated into Event-B, the expression will
be exactly as formulated in EVOL, so the precedence of Event-B will be used
when evaluation the expression there.
This solution works as long as no evaluation of the expression in the abstract
syntax tree is performed.
If some static analysis was applied that involved
evaluating expressions - or if syntax were to be compiled to machine code - this
would cause problems and would have to be changed. For simplicity, however,
this solution has been chosen.
Boolean expressions are handled in the same way. Ambiguities in the precedence
of conjunctions and disjunctions will be left up to Event-B and parenthesis can
be used to avoid mistakes.

In Scala, the structure of the abstract syntax tree is created by dening objects.
These will themselves consist of references to other objects, e.g. a sequential

structure of a tree.
When an object is created it is passed some arguments. Arguments have the
structure <name> : <type> and are separated by commas. An argument can
be given a standard value, which will be the default value of this argument, if
none is passed.
For instance the object BooleanStatement can be created by passing only a
boolean expression and a statement, and the variables nr and next will auto-
matically be set to -1. By omitting these numbers when the object is created,
the code is simpler. The integers in the statements, that are set to -1, will be
used in the next step of the translation, when the numbering of the events in
Event-B will be set.

The two processes communicate setting two variables. Each process has a vari-

in1 and in2. They processes furthermore use a shared variable to determine who

6.2
Peterson's Algorithm for N Processes
An extension of Peterson's algorithm for n processes will be given as another
example of a translation of a model, and it uses the possibility of declaring
both multiple process instances and constants. It also gives some examples of
possibilities and problems with arrays when translated.5
The algorithm uses the same way of letting a process setting itself to be inter-
ested in entering the critical region, and a last ag to indicate that it is the
latest process to enter this request. The algorithm works for n process, and a
process needs to pass through n stages before it can enter the critical region.
An array in of size n keeps track of which stage a particular process instance is
in, so that in(x) maps to the stage of process instance x. The array last, also of
size n, will keep track of which instance has last entered a stage, so that last(x)
maps to the process instance that has last entered stage x.
A process waits at a stage as long as there are any processes at higher stages
and if it is the last to have entered its current stage. When a process enters a
stage, it sets itself to be in it, using in, and that it is the last to have entered
this stage, using last. Any other process at that stage can now move a stage up.
After a process has reached and left the critical section, it sets itself to be in
stage zero, meaning that it is no longer interested in the critical section. The
process can now start over.
As with the simple version of Peterson's algorithm, one process can access the
critical section an unlimited number of times, if no other processes are trying
to access it. If several process are trying to access it, they will all get to enter
before the rst one can enter again.

variable inCrit and a constant n. Each process instance has a local variable j
and k.

one will have index n - 1. The local variable i refers to the process id of the

6.2.3
Checking the Model in Rodin
Having imported the translation into Rodin, it is not possible for the automatic
theorem prover to verify all invariants at rst.
The problems are the preservation of the invariant for the array last in evt6 and
if the boolean guard in evt9 is well formed. In both cases, indices of arrays are
referenced by reading a local variable. The problem is that it is not possible for
the automatic prover to determine, that the value of the variables are inside the
domain of the arrays.
Take for instance the action in evt6: last(CS_j_(PID)) := PID. If the value
of CS_j_(PID) is not between 0 and n - 1, the invariant will be violated that
states that this is the domain of last.
Looking at the algorithm in EVOL, it can be seen that the lower bound of j is
zero, since it is only incremented from this value. The assignment, that evt6 is
a translation of, is last[j] := i, which can only be reached when j is less than
n. Since it can be veried like this, one way to solve the problem is to add it
manually as a guard to the event:
newGrd: 0 ‚Č§CS_j_(PID) ‚ąßCS_j_(PID) ‚Č§n - 1
Similar guards can be added to evt9 for both CS_j_(PID) and CS_k_(PID).
It is now possible for Rodin to verify automatically, that the events does not
reference an index outside the scope of the array, and all Proof Obligations will
be veried.

6.2 Peterson's Algorithm for N Processes

It is, however, important to be careful when using boolean guards in this way,
since they might suppress interleavings possible in EVOL from occurring in
Event-B. If the events would only be able to execute when everything was ne,
it would make the validation of the model useless. In this case it is fairly simple
to assure by hand, that the added guards are always true, but more complex
situations could easily arise. When translating the model it might in some cases
be possible to automatically generate invariants, which is a point that will be
taken up in the discussion.
As for the two process version of Peterson's Algorithm, the validations of the
Proof Obligations are not very helpful, since they do not show us that multiple
processes cannot be in the critical section at the same time. For this we would
have to verify, that inCrit could never be more than one. Such an invariant is,
however, not possible to get Rodin to verify automatically without adding some
more invariant. To determine which invariants is needed would require some
more inspection of the way the automatic prover of Rodin works. Again, this is
outside the scope of this project, but will be discussed in the next chapter.
Instead ProB can be used. This will not give a full proof for the algorithm,
since it contains a constant, and this will have to be given a specic value when
the model checker veries the properties. It will, however, give an argument for
the correctness of the translation and the algorithm being able to secure mutual
exclusion for a small number of process instances.
When opening ProB, a warning will be stated that an error occured, that states:
"Reason: A timeout occured when nding constants after nding 3 solution(s).
Typically this means, that your axioms are too complicated for automatical
solving. [...]". The error can be ignored and the model checker used, but it
shows that the model is in some ways too complex for Rodin to handle.
The constant n will be set to one, since this is the default value of a constant
in the set of positive integers. This means that only one process instance is
created, which is not very interesting to check.
To get around this, another axiom can be added to the context besides the one
that denes n to be in the set of positive integers. Adding the axiom n > 3 will
make ProB set n to four, as this is the smallest number that ts the criteria.
The LTL model checker can now be used to check the algorithm for four pro-
cesses. To check that two processes cannot enter the critical section at the same
time and increment inCrit, the following formula is checked, that states that for
all states, inCrit must be less than two:
G{inCrit < 2}

Depending on the computer used, this takes a while, and at least a lot longer
than checking the LTL formula for the simpler Peterson's algorithm.
It will be veried that no counter examples has been found and that all notes
has been visited. This means that safety property of the algorithm holds for at
least four processes. It should also be checked for liveness properties, which can
be done with the formula:
GF{inCrit > 0}
The formula states that for all states globally, it will eventually be true that
inCrit is greater than zero. This means that it will always be the case that
at least one process will eventually get access to the critical section and there
are no deadlocks. These tests could be expanded to check the process instances
individually etc.
It might not be the full proof covering all values of n, but it shows that the
algorithm as such is working, and satises the properties for a low number of
processes.

Chapter 7
Conclusion
The project presented in this thesis has shown how a modelling language, EVOL,
has been formulated for providing an easier way to create models in Event-B.
Thoughts has been given to what aspects were needed in such a language, and
how they could be translated correctly into the Event-B notation. The transla-
tion between the two languages has been discussed and an implementation of a
program for automatic translation has been described and implemented.
When checking the model in Rodin, it has however not been possible to use
the automatic provers to get a general verication of a model. Peterson's algo-
rithm, that was given as an example of an implementation, could not be veried
to ensure mutual exclusion. The main problem was that the model needed ad-
ditional theorems to prove the general invariant inserted that would verify the
algorithm.
Instead the model checker of the Rodin platform, ProB, was used, and it was
able to check the invariants for the translated models. However, the approach
of using model checkers has some limitations, and it was hoped that the use of
Event-B could provide a verication tool that was not limited by these.
The project has in this way been partly successful. It has not been possible
to use the full automatic proving that Rodin provides, but the rst steps in
creating a modelling language for Event-B has been taken. Some thoughts will
==PAGE:98==
88
Conclusion
be given to possible further work of the project.
7.1
Further Work
The problem is not that the automatic provers of Rodin cannot check the model,
but that additional invariants are needed. By looking further into the logics of
the provers in Event-B might provide some insight into how these invariants
should be formulated and one might hope that this could be done in a sys-
tematic way. The goal would then be to be able to generate these invariants
automatically along with the translation.
Some small examples of this approach have been given in this project.
For
instance the way that the process id in each event is required to be in the
domain of the process instances. By providing this additional guard, that can
be shown not to suppress interleavings in Event-B, that are possible in the same
model in EVOL, Rodin was able to verify some invariants automatically, that
it could not verify before.
Another aspect of the project, where the work could be continued, is in the
expansion of the EVOL language to allow for further use of the tools provided
by Event-B. Renement is a big part of the Event-B notation, so it would be
natural to expand EVOL to be able to make use of this as well.
The possibility of creating sets in the context les of Event-B could also be used.
As it is now, EVOL only makes use of integers and arrays, but the completely
general way a set can be dened could be used.
In this way EVOL could be expanded and be an alternative to the rather com-
plicated notation of Event-B, while still provide the automatic theorem proving,
that is the strength of Event-B.
