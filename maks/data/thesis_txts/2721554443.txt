Abstract
Multivariate estimation-of-distribution algorithms (EDA) have been studied in research
for several decades as they allow for effective optimization of difficult black-box problems.
This thesis presents the results of a literature survey on the state of the art of research
in the field of multivariate EDAs. Selected bit-string benchmark optimization problems
and algorithms are presented in detail, and modifications are proposed for the MIMIC
and P4 algorithms for improving their computational efficiency. Methods presented in
literature for changing these algorithms to work on permutation-based problems such as
the Travelling Salesman Problem (TSP) are also outlined and discussed.
In addition,
the selected problems and algorithms have been implemented in C# alongside a software
framework for building and running empirical performance tests. Empirical results on the
performance of the algorithms across a range of parameter values are presented, discussed
and used to create a series of guidelines on the selection of EDAs. The results indicate that
for bit-string problems the proposed modifications to MIMIC makes it more effective at
escaping local optima and require fewer fitness function calls than any of the other EDAs
selected from literature.
Additionally, the results for TSP showed that the algorithm
producing best-quality solutions is P4. This original P4 algorithm however shows to have
issues with memory management which are addressed by the modifications proposed in
this project.

Introduction
Optimization problems can in general terms be defined as finding the best solution from
the set of all possible solutions. Problems of this kind appears in many shapes and sizes
in academia, industry and everyday life. Examples include scheduling tasks to minimize
their makespan, packing a truck to maximize the number of items that it can carry and
finding the best seating arrangement for a dinner party. Needless to say, we are interested
in finding good solutions for each of these three problems.
For most practical applications we know how the fitness of a solution is determined,
allowing us to tailor our optimization strategy to the problem at hand.
When this is
not the case, however, we are dealing with a so-called black-box optimization problem
where we are instead given an opaque function that simply takes an input and returns its
corresponding fitness value.
For black-box optimization, the only strategy to find the best solution with certainty is
brute-forcing, where all possible solutions are tried out one at a time. This guarantees that
we always find the global optimum, which is the best input across all possible candidates.
Does this not settle the issue?
By going through all possible solutions any black-box
optimization problem can be solved, everyone should be happy and the rest of this thesis
should be irrelevant. Unfortunately, such an approach is not feasible. Some optimization
problems might have very large search spaces that are impossible to exhaust within a
realistic time frame. An example of this is the bit-string search space where inputs consist
of n values that can each have either value 0 or 1. For a given problem size n there will
be 2n possible inputs to consider, which at a relatively small size of n = 100 will be a
whopping 1.27 · 1030 possible candidates. If even only one nanosecond was used to test
each candidate, the entire search would take several orders of magnitude longer than the
age of the universe.
Instead of requiring that our algorithm always finds the global optimum we can instead
use search heuristics, which are strategies that attempt to find a good enough solution
within reasonable time. While this sacrifices some quality of the solution, it speeds up
the search process considerably by not requiring that every possible input is evaluated.
The family of search heuristics includes a vast amount of different algorithms used for
different purposes. Some heuristics are created for specific problems, such as Christofides’
algorithm for the Travelling Salesman Problem [1], while others are designed to optimize
any arbitrary fitness function.
This thesis will focus on a category of search heuristics called estimation-of-distribution
algorithms (EDA). These search heuristics are randomized algorithms that guide their
search by building and using models that encode information learned through attempting
to optimize the problem. Such models can e. g. be probabilistic distributions over the
search space or a clustering of correlated variables. The model is used to sample new
solutions and updated with any new information. This creates a feedback loop where the
model is refined in each iteration and thus creates better individuals, narrowing down the
search to interesting areas in the search space.
The intuition behind EDAs is that they themselves are able to identify the relevant pat-
terns of the fitness functions and incorporate these into the model. Since the definition of
EDAs is very broad, there is a large variation of algorithms presented in literature. While
1
==PAGE:8==
some algorithms use simple models with few variables, others encode more information
using larger models with higher complexity and computational costs. While many ways of
classifying EDAs have been presented in literature, a popular approach is to distinguish
them by the number of model variables per problem variable [2].
The simplest EDAs are univariate, which have only a single model variable for each prob-
lem variable. In such models, all variables are independent of each other. This has the
advantage of making univariate EDAs simpler to work and reason with, since each variable
can be viewed in isolation from the rest. Due to the limited number of model variables
and their independence, these algorithms usually also incur smaller computational costs.
The speed of univariate EDAs however comes at the price of being unable to model any
relationships between problem variables, which might be necessary to properly optimize
more complex problems. In contrast, multivariate EDAs model can use multiple model
variables for a problem variable. This arguably makes them better able to model the de-
pendencies between variables and thus also better able to find good solutions for difficult
problems. The focus of this thesis will be to examine the truth of such claims by compar-
ing the empirical performance of different selected multivariate EDAs with a benchmark
univariate algorithm on a series of different optimization problems of varying difficulties.
This is done by first examining the state of research through a literature review, followed
by the design, implementation and usage of a software framework for running empirical
performance testing.
1.1
Overview
The ordering of chapters in this thesis reflects the chronological order of work performed
for this project.
Chapter 2 will initially give a description of the key results from a literature review of
research in the field of estimation-of-distribution algorithms with a focus on multivariate
EDAs. This includes a presentation of relevant fitness functions used for benchmarking
the performance of algorithms in literature, as well as some that are used for real-world
applications. A selected set of algorithms will be chosen from literature and described in
further detail, with their inner workings laid out in pseudocode.
Subsequently, chapter 3 will describe the process of designing and implementing a frame-
work for running EDAs and gather statistics on these runs. This will be done by first
specifying the overarching goal and requirements of the framework, followed by a bottom-
up walkthrough of the main design points of the framework.
Chapter 4 then lays out the results of using the newly created framework for evaluating the
performance of the selected EDAs empirically. Results will be presented for benchmark
problems of increasing difficulty, followed by summarizations of the results. Finally, the
results are compiled into a number of observations on the performance of the selected
EDAs and a series of guidelines for the use of these algorithms for practical use.
The overarching results of the project are summarized and scrutinized in chapter 5. This
includes identifying sources of error for each phase of the project and discuss the impact
and generality of the results. Additionally, EDAs are put into a wider context by discussing
their application in real-world scenarios.
Chapter 6 then identifies the limitations of the project and discusses the potential lines
of future work for refining and expanding on the results previously presented. The main
work and contributions of this project are finally summed up in chapter 7 which concludes
the report and reflects on its results.

Literature Survey
All models are wrong, but some are
useful.
George E. P. Box
The first phase of the project was to perform a literature study of the current state of
research on multivariate estimation-of-distribution algorithms. Central to this pursuit is
mapping out and describe algorithms presented in research. A discussion of the differences
between these algorithms is also necessary to properly introduce the reader to their core
concepts.
In addition to examining EDAs, another goal of the literature survey is to identify relevant
optimization problems to use for testing the performance of algorithms. Some of these
optimization problems are so-called benchmark problems, where the fitness function has a
relatively simple definition. These problems test specific aspects of algorithms, and serve
to give a glimpse into the performance of the algorithms in highly contrived scenarios.
Due to their simplicity, however, benchmark problems are usually not themselves of any
practical interest.
Other problems can be classified as combinatorial problems, which are well-known prob-
lems with practical applications. Many of these problems are NP-complete, making them
more intricate and difficult reason with. While benchmark problems aim to test the algo-
rithm under very specific conditions, combinatorial problems test their ability to search
for good solutions in non-trivial optimization landscapes.
Apart from mapping out relevant algorithms and problems, the survey will also present the
current state of results in the area. These results include both empirical and theoretical
results, where these exist.
Several theoretical results exist for univariate EDAs [3, 4],
but due to the more complex nature of multivariate EDAs fewer mathematical proofs on
expected runtime exist for these.
Additionally, an important element of some optimization problems is the constraints that
these impose on the candidate solutions. An example of this is the well-known knapsack
problem, where the total weight of any candidate solution is not allowed to be larger than
an imposed limit. How multivariate EDAs can deal with such constraints is not trivial,
and the survey will map out several issues and potential solutions presented in literature.
2.1
Methodology
Given the large amount of research literature available online, it is important to adhere to
a systematic process when performing a literature survey. The main working process for
the literature survey was to examine articles one by one in different levels of detail. These
levels of detail were, in increasing order:
1. Considering title
2. Reading abstract
3. Skimming contents
4. Thoroughly reading contents
For a given article, the abstract was first read through to assess whether the article was
relevant. If so, the contents were skimmed to extract the main points and contributions.

2.2
Optimization problems
A vast array of optimization problems are presented in literature, spanning from the very
simple to the very complex. Some problems are designed to use for academic purposes
while others have their origins in practical applications. Central to black-box optimization
is the concept of deception [5], which is the notion of an optimization landscape that leads
greedy strategies towards local optima. A local optimum can be characterized as a solution
with better fitness than its neighbors in the search space and with worse fitness than the
global optima. Deception greatly increases the difficulty of an optimization problem since
optimization strategies might be fooled by gradients pointing towards local optima, making
them yield inferior solutions.
OneMax
The easiest problems to optimize are so-called linear problems [9], where no dependencies
exist between variables and optimization can thus be performed by evaluating each variable
in isolation. The arguably simplest and one of the most widespread linear optimization
problem in literature is the OneMax problem. The fitness function for OneMax simply
returns the number of ones in the given bit-string, resulting in the all-ones string being the
local optimum. In this situation, there are no dependencies between the values of different
bits, and a greedy search should be able to find the global optimum.
LeadingOnes
The fitness value of the LeadingOnes function is the number of consecutive ones starting
from the leftmost position. LeadingOnes is in some ways similar to OneMax, but adds
the additional requirement of optimizing the variables in a specific order. The leftmost
4

bit that is not set to 1 is called the active bit, and the algorithm must flip this bit before
progressing to the next bit.
This makes LeadingOnes more tricky to optimize than OneMax, since the algorithm must
flip the correct bit out of n total options. This problem can however still be solved by a
greedy approach that attempts to flip every bit in isolation until an improvement occurs
[9].

Travelling Salesman Problem
One of the most quintessential and perhaps also notorious combinatorial optimization
problems is the Travelling Salesman Problem (TSP). Briefly, TSP consists of finding the
shortest Hamiltonian tour in a graph with weighed edges. A Hamiltonian tour is a path
that visits each vertex exactly once and finally returns to the start vertex. The name of
the problem stems from an analogy to a salesman who wishes to find the shortest route
for visiting a set of cities in order to sell his product. Because of this, the word city is
often used interchangeably with the word vertex in literature on TSP.
TSP instances can be defined using any weighed graph for which at least one Hamilto-
nian tour is possible. Since this problem is commonly found in routing, many real-world
instances use the road network as the underlying graph, with edge weights being travel
time estimates. Another type of instances is also d-dimensional Euclidean TSP instances,
in which the graph is fully connected, each vertex has a corresponding set of d coordinates
and the edge weight between two vertices is the Euclidean distance between these points.
TSP is an NP-hard problem [12], which is a strong indication that this problem is compu-
tationally hard to solve. This means that any attempt to solve TSP must either rely on an
exact algorithm with super-polynomial running time or some heuristic that approximates
the optimal solution. Estimation-of-distribution algorithms fall into the latter category,
as they give no guarantees of finding the global optimum for an arbitrary instance of any
problem.
Additionally, TSP is a constrained optimization problem in the sense that it requires its
inputs to be permutations of the set of vertices. This means that each variable can take n
different values, and that each of these possible value must appear at exactly one variable.
By having this constraint, TSP has a different input space and thus stands out from the
other optimization problems considered for this project. How this affects the design of
EDAs for solving TSP will be seen in the next section.

2.3
Algorithms
Since the field of EDAs is quite diverse, there have been proposed several ways of classifying
these algorithms. Apart from discerning between univariate and multivariate algorithms,
a classification between population-based and incremental EDAs is presented in [13]. The
difference between these two types is whether the population or the learned model is passed
on to the next iteration. In population-based EDAs a new model is built in each iteration
from the population and then used to update the population with new individuals sampled
from the model. Incremental EDAs, on the other hand, sample a new population from
the model at the beginning of each iteration and then use these new individuals to update
the model.
Another way of distinguishing between EDAs presented in [13] is to group algorithms by
the manner in which they represent the underlying distribution. The most versatile format
is to encode the distribution using explicit probabilities for every possible outcome. This is
clearly only possible for discrete domains where the number of possible outcomes is within
the limits of storage and computation time. Most EDAs, however, encode their models
more efficiently, such as e.g. having a probability distribution for each variable.
2.3.1
Univariate EDAs
Though the aim of the project is to study multivariate EDAs, their univariate counterparts
are a good introduction to EDAs due to their simpler nature. For this project, a univariate
algorithm will also be implemented and tested as a baseline with which the multivariate
algorithms can be compared. Univariate EDAs were the first ones presented in literature
with the first ones reaching back to the mid-90s [14, 15, 16], and they have shown fine
theoretical and empirical results for simple functions such as OneMax.
All univariate EDAs on bit-strings work by assigning a probability to each variable, de-
noting the probability that its value will be 1. Sampling new individuals then consists
of simply sampling from each of these individual probabilities and combining the results
into a new bit-string.
The difference in the different algorithms then lies in how the
probabilities are updated by a new population.
One of the arguably most classic univariate EDAs is the compact Genetic Algorithm (cGA)
originally presented in [15]. Pseudocode for cGA is shown in listing 1. The algorithm
takes a single parameter, K, which determines the intensity with which the probabilities
are updated after each iteration.

The simplicity of univariate EDAs, however, comes at a price. When the problems include

the different variables interact, the algorithms must instead rely on chance to perform
meaningful improvements for more complex problems.
2.3.2
Multivariate EDAs
By modelling dependencies between problem variables, multivariate EDAs should in theory
be able to produce better solutions for more complex optimization problems. While we
cannot expect general multivariate EDAs to perform better than heuristics tailored for a
specific problem, they should be able to capture information of the problem and encode
this in its model. This however comes at a larger overhead cost of updating the model in
each iteration and using it to create new solutions.
One straightforward way for multivariate EDAs of modelling the problem at hand is to have
a random variable for each problem variable, similar to how their univariate counterparts
work. In addition, however, this class of multivariate EDAs must also include dependencies
between these random variables. As mentioned in [2], such dependencies can be visualized
as a directed acyclic graph, where each node is a random variable and an edge represents
a dependency.
This way, the outcome of sampling a random variable depends on the
outcomes of the upstream random variables of the dependency graph.
MIMIC
A key question when designing such multivariate EDA is which shape this dependency
graph should have and how it should be updated. One possible answer to this question can
be found in the mutual-information-maximization input clustering (MIMIC) [7] algorithm,
which is arguably one of the earliest multivariate EDAs presented in literature.
The
dependency model used by MIMIC is a directed path, where all except the start and end
variables have exactly one ingoing and one outgoing edge. MIMIC updates this model
using empirical univariate and joint entropies of all problem variables in each iteration.
The model is built in successive steps, where each step adds a variable by greedily selecting
the one with the lowest conditional entropy to the previously added variable.
Pseudocode of the MIMIC algorithm is shown in listing 2.
The notation used in the
pseudocode listing varies slightly from the notation used in the original paper. In listing
2, n denotes the problem size while N and k are parameters representing population size
and selection percentile, respectively. For this project, the selection percentile was fixed
to 50%, since this also ensures that the population size is invariant across iterations of
MIMIC.
Lines 3–6 of the algorithm are initial calculations performed in the beginning of each
iteration, where relative frequencies and entropies are found for the given population. The
algorithm then proceeds to find an ordering of the problem variables in lines 7–10 by
greedily selecting the next variable with minimal conditional entropy to the previously
chosen variable. The first variable is chosen as the one with the lowest univariate entropy.
New individuals are sampled in lines 11–17 by conditioning the probability on the sampled
value of the previous variable of the newly found ordering. Finally, the least fit individuals
are removed from the population and the next iteration begins.

The conditional entropy between two problem variables describes the certainty of the
outcome of one variable given the outcome of another. This way, the conditional entropy
measure can work as a proxy for determining degree of dependency between variables, with
smaller conditional entropies signalling closer dependency. MIMIC uses this for building
its model by identifying the two variables that are most closely related to each other in
each step of the building process. While this approach is greedy, finding the best possible
model is in itself a non-trivial optimization problem.
As stated in the original paper each iteration of MIMIC has complexity O(n2) [7]. This
is due to the fact that joint frequencies and entropies must be computed for each pair
of problem variables, resulting in O(n2) computations. Additionally, the model building
phase also requires O(n2) calculations since the minimum amongst O(n) entropies must
be found once for each variable.
GOMEA
Not all multivariate EDAs use random variables to model and solve the problem at hand.
A family of EDAs first introduced with the Linkage-Tree Genetic Algorithm (LTGA) in [17]
instead makes use of so-called hierarchical clustering. This works by identifying clusters
of variables which make up partial solutions, meaning that their values to a high extent
depend on each other. The notion of a partial solution comes from the fact that in many
optimization problems, certain groups of variables are highly coupled. An example of this
is blocks of DeceptiveLeadingBlocks, where we wish to consider and modify blocks as a
whole. Variables within such partial solution should therefore be clustered together in the
model and changes between these variables should to a great extent are coordinated.
The Gene-pool Optimal Mixing Evolutionary Algorithm [18] (GOMEA) works in exactly
this manner by creating a set of clusters each containing one or more problem variables.
The same variable can be contained in multiple clusters at once, and clusters can consist
of just a single variable. After clustering, GOMEA then performs optimal mixing of its

Similar to GOMEA, the Parameter-less Population Pyramid algorithm (P3) [19] uses
clustering to model the dependencies between problem variables. While GOMEA and

as it attempts to optimize the given problem. The main idea behind P3 is to split the
population into levels of a pyramid with each successive level containing solutions of higher
quality. Starting with only a single level, the pyramid is expanded with more levels as the
algorithm progresses.
Pseudocode for the main loop of the P3 algorithm is shown in listing 6. At the beginning

are reconsidered for flipping. This procedure moves the solution to a local optimum, since
the procedure terminates only when no local changes result in an improvement of the
bit-string.

are undone and are otherwise kept.
As the name suggests, the P3 algorithm takes no parameters and simply optimizes any

the other algorithms, as some degree of parameter tuning will be required for those for
any practical application. For large and complex problems this parameter tuning process
can be both slow and computationally expensive, as it requires mapping over a potentially
enormous parameter space.
2.3.3
Dealing with constraints
So far, we have only considered EDAs that work with unconstrained bit-strings. This type
of domain is a nice starting point since bit-strings are natural for computers to work with
and since our algorithms have no pesky constraints that make certain solutions invalid. In
the real world we however often encounter fitness functions with different domains, some
of which are not as gentle to work with as others. Thus, we must think about how to
design EDAs that can handle different types of domains with a multitude of constraints.
The bit-string domain is a special case of the so-called discrete integer domain [6], where
each variable d can take the values {0, . . . , rd}. Here, the entire space of possible solutions
is the Cartesian product of all possible variable values:
Ω= {0, . . . , r1} × · · · × {0, . . . , rn}
For bit-strings, the allowed values for all variables is simply {0, 1}. In this domain, any
solution is valid as long as its variables all take one of the allowed values.
Some optimization problems, however, impose constraints that make some solutions in-
valid. An example of this is the knapsack problem, in which only solutions with items
of total weight below the limit are allowed. While solutions of this problem can be rep-
resented effectively using bit-strings, none of the dependency models of the presented
algorithms take into consideration this constraint. Using EDAs that are not designed to
deal with such issues can result in EDAs that get stuck in regions where the constraint is
not satisfied, since these regions contain no gradient or other information on how to reach
valid solutions.
A straightforward way of dealing with constraints is to simply set the fitness value of
invalid solutions to some infinitely poor value such as −∞or ∞, depending on whether
the goal is maximization or minimization. Another related method would be to add a
so-called penalty function to the fitness function for every constraint, where a penalty
function contributes detrimentally to the fitness value of solutions that violate the given
constraint. Both approaches would nudge the selection process of algorithms to disregard
solutions that violate the constraint since these would have worse fitness. They however
do not prevent the algorithm from generating such invalid solutions in the first place, and
for problems where the invalid region dominates the solution space the algorithm might
only generate invalid solutions.
Another idea could be to discard invalid solutions and repeat the generation step until
enough valid solutions are produced.
While this ensures that only valid solutions are
considered, there are no guarantees of the time to generate a valid solution. For problems
13
==PAGE:20==
where valid solutions comprise only a small subset of the solution space this generation step
must be repeated many times before the algorithm can proceed, resulting in inefficiency.
Both strategies seem to be more a treatment of the symptoms rather than the root cause of
the problem, namely that the underlying model of the EDAs do not deal with constraints.
Instead, several authors propose instead to modify existing EDAs to support constraints,
or to create entirely new ones with this capability [20] [21] [22]. This is done both for
specific and generic constraints, with those being generated for specific constraints usually
guaranteeing that all solutions are valid, while those for generic constraints instead nudge
the algorithm towards the valid region.
This project will focus on the constraint imposed by the Travelling Salesman Problem
(TSP): namely that all solutions should be permutations of the set of vertices. In TSP,
each vertex must be visited exactly once, meaning that each possible value should appear
exactly once in a given solution. As presented in [6], this search space is defined as follows:
Ω= {(σ1, σ2, . . . , σn) | σi ∈{1, 2, . . . , n}, σi ̸= σj∀i ̸= j}
TourMIMIC
Some EDAs, such as MIMIC, generate new solutions by sampling the value of one variable
at a time. An obvious way to support permutations for such problems would be to simply
set the probability of already chosen vertices to zero and then normalize the remaining
non-zero probabilities. This results in a new distribution that only generates vertices of
the TSP that have not yet been visited.
The order in which the variables are sampled, however, now play a role. This is due to
the fact that the results of sampling a variable affects the outcome of any subsequently
sampled variable. For univariate EDAs such as cGA this is a problem, since all variables
are independent. In MIMIC, however, the order of sampling is already determined by the
path-shaped dependency model. Putting these pieces together for MIMIC results in a new
permutation-based EDA dubbed TourMIMIC.

any such solution with continuous values a permutation can be created by finding the
sorting order of the variables of the solution.

Mallows EDA
So far, we have only considered algorithms that have been modified to deal with permuta-
tion instances. An algorithm designed specifically for permutation-based problems is the
Mallows EDA presented in [26]. This algorithm is based on the Mallows model, a proba-
bility distribution introduced in 1957 [27] which is described as a Gaussian distribution on
permutation spaces. This distribution is defined by a central permutation, σ0 and a spread
parameter θ. Briefly, the distribution assigns a probability to a permutation σ based on
its distance to σ0. The higher the parameter θ the higher probabilities are assigned to
permutations far away from the central permutation.
The Mallows EDA creates a Mallows distribution from the data by learning the two pa-
rameters from the current population. To generate a new solution, the algorithm samples
a vector representing a mapping from σ0 to a new permutation and then applies this on σ0.
This generates solutions that always satisfy the constraint of being permutations, since
the sampled mappings ensure that this property holds invariantly. For this project, the
Mallows EDA will however not be implemented and tested due to time limitations.
2.3.4
Modifications
Most of the algorithms considered were introduced several years ago, since which their
authors and other researchers in the field have proposed a number of modifications. This
results that sometimes multiple variations of the same algorithm exist, with some varia-
tions having certain advantages and disadvantages over others. An example of this is the
suggestion in [24] to apply a population-sizing schema from [28] to GOMEA. This makes
the algorithm self-regulate the population size by beginning with a small population and
then increasing its size as the algorithm iterates. Such modification makes the algorithm
parameter-less, saving the user from having to tune this parameter.
Additionally, I suggest in this thesis a modification to the MIMIC algorithm presented
in literature. The modification consists of making MIMIC only recompute the joint en-
tropies for a random subset of the problem variables in each iteration, yielding a new
algorithm which will be called FastMIMIC. The intuition behind this change is that not
all relationships between problem variables are equally important, and that some of the
Θ(n2) entries in the joint entropy matrix are thus not necessary to recompute in each
iteration. We can expect this change to increase the speed of MIMIC’s iterations, since
the main computational bottleneck of the algorithm is to compute the joint entropies for
all Θ(n2) pairs of variables. This however should also come with the cost of reducing the
quality of solutions, since the joint entropies are delayed by a number of iterations. The
experimental phase of this project will reveal whether this quality/speed trade-off pays off
for the selected problems.
The size of the sampled variable subset is a parameter of FastMIMIC, which will be called
the sampling intensity. The maximal value for this parameter is n, with a sampling inten-
sity of exactly n making FastMIMIC identical to the original MIMIC. The experimental
phase will also reveal the impact of the sampling intensity on the performance of the new
algorithm.
Apart from introducing FastMIMIC, a modification to the P4 algorithm is also proposed
in this thesis. The original version of the algorithm presented in [19] has no mechanism for
limiting the population size, which will simply grow until termination. In the experimental
phase the system quickly ran out of memory when running P4 on TSP instances of size
n > 80, forcing the test to terminate.
Because of this, a modification is proposed to
P4 in which the lowest level of the pyramid is periodically dropped. The length of this
period between each reduction is called the shedding interval and is specified in numbers

algorithm since it requires the user to specify a shedding interval.
2.3.5
Termination
In literature, most algorithm definitions do not specify when to terminate but instead
make the algorithm check before the start of each iteration whether a generic termination
criteria is met. This allows the algorithms to use an arbitrary termination criteria, as long
as they return a boolean value used to control the conditional execution of the iteration
loop.
In literature, many papers examine the performance as the number of iterations or fitness
function calls until the optimal solution is sampled for the first time [7, 19]. Since no
better solutions exist, the algorithm will in its subsequent execution not produce any
improvements, and the test can be terminated. While this is an ideal termination criterion
for academic evaluations on simple benchmark problems where the optimum is already
known or easy to compute, it is not a feasible solution for instances of more complex
combinatorial problems where the optimum is unknown for arbitrary instances. It also
does not reflect real-world applications, where the entire purpose of using EDAs is to find
the unknown global optimum of a black-box function.
Another possibility is to examine whether the algorithm has produced any meaningful
improvements within a fixed number of iterations. If this is not the case, then this might
indicate that the algorithm has stagnated in either a global or local optimum and is unable
to proceed. While this is a more pragmatic approach to termination, it introduces the
risk of killing the optimization prematurely. Additionally, the user needs to specify the
number of stagnated iterations before termination is triggered, adding a new parameter
dimension.
While these two criteria will be used for this project, a number of other options exist.
Execution can be terminated after a specified number of iterations or when a solution of
sufficient quality is produced. Algorithm-specific criteria can also be used, such as when
the average entropy of MIMIC reaches a given level or when the pyramid of P3 grows too
large for the allocated memory. In practice, the proper choice of termination criteria is
related to the fitness function, choice of algorithm and the wider context.
2.3.6
Considerations
As with other types of search algorithms, the designers and users of EDAs are faced with
the trade-off of exploration versus exploitation. The question is whether to perform a wide
search, where the diversity of the population is large, or a narrow one focusing on a less
diverse population with higher quality. The goal is to perform a search that both does
not exclude a significant portion of the search space, but also focuses on the elements that
yield the best solutions.
In terms of EDAs, this happens by setting the correct parameters for how much new indi-
viduals affect the underlying distribution used to sample new solutions. If the distribution
is updated too little, good individuals to not affect the future sampling enough to narrow
down the search, preventing the algorithm from properly learning a distribution that can
sample good individuals. On the other side of the scale, updating the distribution too fast
might make the algorithm get stuck in a local optimum.
While univariate EDAs are quite lightweight in terms of computational complexity, mul-
tivariate EDAs incur much larger costs for building models of the problem. This is clearly
the case for e.g. MIMIC, where the calculation of joint entropies is Θ(n2). We can therefore
17
==PAGE:24==
expect the required CPU time for multivariate EDAs to be much larger in the empirical
performance testing phase of this project.
2.4
Performance results
Since estimation-of-distribution algorithms are randomized, bounds on their performance
are usually stated in expected values instead of exact values.
The most used metric
for measuring performance is the number of fitness function calls done by the algortihm.
Historically, the first theoretical results on EDAs were on whether these algorithms reached
the global optimum [2]. This is called convergence analysis, as opposed to runtime analysis
which yields bounds on the expected time before an EDA reaches the global optimum.
As they are simpler to understand and reason with, a number of convergence and runtime
proofs have been constructed for univariate EDAs, such as in [3] for cGA on OneMax. Mul-
tivariate EDAs, however, are more complex to implement and reason with, and therefore
nearly no theoretical results exist for this class of algorithms. This is due to the fact that
proofs must deal with more intricate probabilistic interactions between the population,
dependency model and problem function.
Only one theoretical proof for multivariate EDAs was identified in the literature survey,
namely in [9] for P3 on a number of problems. In that work, the authors show that the
hill-climbing component of P3 solves OneMax and Leading with an expected number of
fitness function calls of O(n) and O(n2), respectively. It additionally gives upper bounds
for the expected runtime of P3 on other more complex problems, such as Concatenated
Traps and H-IFF [9].
As theoretical results are mostly infeasible to produce, the performance of multivariate
EDAs must be analyzed using empirical results. This is done by gathering and comparing
statistics of the algorithms over many runs on a variety of configurations.
While this
does not give the mathematical certainty provided by theoretical results, empirical testing
gives a good indication of the performance of algorithms with a reasonable certainty if the
number of runs is large enough.
Measuring the performance in terms of the number of fitness function evaluations however
rests on the assumption that the calculation of the fitness value is the computational
bottleneck of the algorithm. As we will examine later, this is not necessarily the case for
multivariate EDAs such as MIMIC, P3 and GOMEA that in each iteration must build a
new dependency model.

3.1
Requirements
An important step of designing software is to initially make an overview of how the final
software product is expected to behave and which characteristics are required.
These
expectations are called the software’s requirements, and they act as guidance throughout
the development, implementation and testing process. Because of this, a crucial first step
is to compile a list of these requirements to guide the further steps towards a functioning
solution.
A natural starting point for the gathering of requirements is to reflect on the goal of the
system and how the user is expected to interact with it. The overarching purpose of the
system is to compose and run test suites of selected EDAs on selected fitness functions,
and then subsequently gather and visualize their results. Test suites should be specified
either by the user through a UI or programmatically through a command-line interface
that uses files for specifying test suites.
A test suite consists of one fitness function, one or more algorithms, and one or more
termination criteria. Each of these components has its own configuration, with the values
of parameters being defined as mathematical functions of the problem size, n. The frame-
work must support parsing arbitrary functions of n with basic mathematical operators
such as addition, subtraction, multiplication exponentiation and respect the precedence
of these. While this adds complexity to the required solution, it is a necessary feature for
tests where parameters must scale with the problem size since the alternative would be to
have users input constant values.
Additionally, the user is allowed to set either the problem size or one parameter as variable
across a range of values. When this is the case, the framework must run the test suite
with the selected parameter’s value set to each value in this range. Possible parameters
are the problem size and all parameters of the fitness function, algorithms and termination
criteria. If e.g. the population size parameter is variable in the range 1, . . . , 10 the test
suite will be executed 10 times with population sizes corresponding to each of the 10
possible values.
19
==PAGE:26==
The framework must also be able to display the results of a test suite with graphs of
the performance across the different values of the variable parameter.
Performance is
measured both in the best and average solution quality, CPU time and the number of
calls to the fitness functions.
If multiple different algorithms are used in a test, their
results should be displayed back-to-back in the same graphs.
Apart from the functional requirements that specify the features expected of the system,
a number of non-functional requirements have also been identified. These specify certain
characteristics and qualities that the program should have:
 Usability and ease of use: the system should have an intuitive interface both when
using though its UI and its command-line interface.
 Any errors encountered in a test run should be handled gracefully, as to not make the
execution of the entire test suite crash. This notion of fault-tolerance also extends to
data, such that any results are backed up to prevent loss of data in case the system
terminates abruptly.
 Since test runs can be executed in parallel, the system should be able to make use
of multiple cores at once to improve the performance.
 The system should contain a suite of unit tests and integration tests that ensure
that key features are properly supported without any errors.
 New algorithms, fitness functions and termination criteria should be addable without
major changes to the existing codebase.
 The system should be able to run on multiple operating systems. This requirement
stems from the fact that development takes place on a Windows machine, while the
DTU HPC clusters run Linux.
Due to the limited scope and use of the system, security will not be expected for the system.
While this is usually an indispensable requirement for enterprise software, security is not
necessary to consider for this project since it is neither safety critical nor handling any
sensitive data.
3.2
Architecture
C# was chosen as the implementation language for a number of reasons. Firstly, it sup-
ports building programs to multiple platforms including Windows and Linux. Secondly, it
is a statically-typed language designed specifically for writing object-oriented code. While
many opinions exist on the choice of programming languages, it is widely accepted that
object-oriented programming with a strongly enforced type-system evaluated at compile-
time are ideal for large-scale applications. This is reflected in the wide use of C# for
enterprise-scale applications in industry.
The overall system consists of four main components:
 Core, which contains central classes representing solutions, populations, algorithms,
fitness functions etc.
 Runner, which builds and orchestrates tests and gathers results from these.
 CLI, which runs tests from specification files using the command-line.
 UI, which allows the user to run tests and see their results using a graphical user
interface.
The dependency graph between these components is shown on figure (3.1). The Core
component is used by Runner, which is in turn used by the UI and CLI components.
The UI and CLI components use the same interfaces of Runner, thus avoiding some code

3.3
Core
The algorithms and fitness functions work with one of two input formats: either bit-strings
or Hamiltonian tours. These are represented by the classes BitString and Tour, which
both inherit from the abstract Instance class. This abstract class contains properties
and methods that are shared for all solution formats. These include properties for getting
the size and fitness value. Since some algorithms do not require that the fitness value be
known at all times for an individual, this property is nullable.
The subclasses of Instance each contains data for representing instances of that partic-
ular format. A bit-string is represented by an array of boolean values, while a tour is
represented by an array of integers. Additionally, these subclasses contain methods that
are useful when working with instances of the given format. For example, the BitString
class contains methods for flipping specific bits and generating uniform instances.
The generic Population class acts as a tailor-made collection of instances, which is
widely used by the algorithms. Apart from allowing for adding and removing instances,
Population objects also keeps track of the fittest instance of the population. The class
also implements the built-in IEnumerable interface of C#, which allows for iteration using
foreach loops.
Throughout the system, the C# notion of generic types is heavily used. This allows the
same class to work on different instance formats, since objects can be instantiated with
different types for the same class. In the case of e.g. Population, this means that objects
can be created with type Population<BitString> or Population<Tour>. C# also allows
for constraining the types supported by a generic class. All generic classes and interfaces
working with instances have been constrained to only allow subclasses of the abstract
Instance class.
Looking at the general problem of black-box optimization, we can realize that a fitness
function simply takes an instance and returns a fitness value for this particular instance.
This realization is reflected in the interface of the abstract generic FitnessFunction
class, which takes an instance as input and returns a double. Additionally, each fitness
function contains a FitnessComparison object, which serves to compare two instances
and determine which one is fittest. Two such comparisons exist, one for maximization
and one for minimization.
Another central class of the framework’s Core component is the generic Optimizer class,
which represents a configured algorithm. This class provides a BuildRun method, which
creates a Run object from a fitness function. This is done by first initializing the optimizer
21
==PAGE:28==
and then creating an enumerator that indefinitely computes the next iteration. Initial-
ization and computation of iterations is implemented by the non-abstract child classes by
overriding the abstract methods Initialize and NextIteration.
A Run object that is created and then run immediately will never terminate, since the
optimizers do not themselves contain any termination logic. They simply continue yielding
the next iteration in an infinite while-loop.
The returned Run-class however supports
adding termination criteria that are evaluated after each iteration. If any of these criteria
are met, the execution of the run is terminated and the enumerator stops.
This way, consumers of the enumerator need not have any information about termination.
Consuming the run can simply be done using a regular foreach loop on the Run-object,
which then yields results for each iteration until a termination criterion is met. Termina-
tion criteria are represented by the abstract generic TerminationCriterion class, which
exposes the ShouldTerminate method.
This method takes a population as input and
returns a boolean indicating whether the run should terminate or not.
3.3.1
Fitness functions
The four selected benchmark problems on bit-string are all fairly straightforward to com-
pute, since they are clearly defined mathematically. Implementing these problems thus
simply consists of implementing these mathematical expressions.
Implementing the fitness function for TSP, however, is more involved.
The task first
requires determining how to model the problem instances and candidate solution tours.
Additionally, the instances must be created and loaded into the framework so that the
user can select from a list of possible problem instances.
Fortunately, most of these problems can be solved by using TSPLIB [29], which is a
publicly available library of files containing pre-generated TSP instances. Even better yet,
an open-source C# library, TSPLib.Net1, exists for loading these instances and calculate
the fitness of tours. The library introduces the ITour interface for tours, which simply
contains a list of integer indices of vertices in the given instance.
TSPLIB contains instances with different types of distances.
Some instances contain
explicit edge weights between each pair of vertices, while others use the Euclidean distance
metric in either 2 or 3 dimensions.
From the perspective of an EDA, the method of
determining these distances is irrelevant as the only value considered is the final fitness of
a solution.
3.3.2
Algorithms
Implementing the algorithms consists of filling out the Initialize method for initializa-
tion and the NextIteration methods with logic for a single iteration of the given EDA.
Beginning with cGA as the simplest algorithm and progressing to the more complex mul-
tivariate EDAs, this implementation was performed incrementally.
Some operations are shared across different algorithms, such as computing joint entropies
and clusters. To avoid code duplication, these methods were placed in static utility classes,
which could then be called by the algorithm code. Although static classes are commonly
considered an antipattern in software design, these routines lent themselves to be static
due to their stateless nature.
As most of the CPU time will be spent executing the algorithms, performance was an im-
portant consideration when implementing these. Because of this, performant data struc-
tures such as arrays and hash maps were chosen over other slower ones such as linked lists
1Available at https://github.com/pdrozdowski/TSPLib.Net, last visited on January 27, 2023
22
==PAGE:29==
when possible. Especially arrays were widely used for computing entropies, since these
have much higher read and write speeds when iterating over the contents due to caching.
At times the specifications of the various EDAs were ambiguous or incomplete in the
original papers, such as in [7] where the method for computing entropy is not specified.
Because of this, the implementation of the algorithms were based on the interpretations
of the algorithms which are outlined in pseudocode on section 2.3. The implementations
used are those which were deemed most plausible of being the original author’s intent. In
case of the entropy for MIMIC, the implementation used the relative frequency of 1 in the
population as the empirical probabilities for calculating entropies.
3.3.3
Termination criteria
In the Core component, termination criteria inherit from the TerminationCriterion
class and override the abstract method ShouldTerminate. This method simply takes a
collection of fitness values and returns a boolean for whether the criterion is satisfied. As
such, these classes can be used for any instance format since they simply get a list of
doubles representing the fitness values of a given iteration.
Implementing the termination criteria then simply consisted of filling out a single method,
leaving the rest of the work up to the framework. This was quite straightforward, as the
two criteria simply consider the best fitness found in the population and then respectively
determine whether this is the global optimum or if any improvement has happened since
last iteration. This makes the implementation of the global optimum criteria stateless,
while the stagnation criteria simply must keep track of the best fitness in the previous
iteration.
3.4
Runner
As previously stated, the purpose of the Runner component is to build and run tests,
as well as the gathering of results when these runs terminate. The main class that the
Runner component exposes is TestProvider, which takes a TestSpecification object
and returns an object implementing the ITest interface.
A TestSpecification fully and unambiguously defines a test suite, including which al-
gorithms, fitness functions, termination criteria to use. In addition, the specification also
determines how these individual parts should be configured and provides higher-level pa-
rameters for the test suite, such as problem sizes and whether to use parallelization.
From a TestSpecification object an ITest object is built, which represents the test suite
laid out by the specification. The interface contains a number of properties and methods
for retrieving results and executing the test. Upon creation the test is not executed, and
its result properties will simply be empty. Execution is started using the Execute method
which runs all tests in the suite and populates the result properties.
Results are kept in two separate collections: Items and Series. The Items collection
contains one-dimensional data for which only one value exists, such as e.g. the best fitness
value or the total number of iterations. The Series collection, on the other hand, contains
two-dimensional data, such as e.g. the population size at each iteration or the best fitness
over various problem sizes.
The ITest interface implements the observer pattern first described in [30], which allows
for other objects to get notified when new results are computed. This is done by having
the ITest object keep a collection of delegates that are invoked whenever a new result
is added. By using the observer pattern, results can be consumed in real-time to e.g.
23
==PAGE:30==
update a UI with new information or save results to prevent data loss. Since the ITest
object delegates these responsibilities to the subscribers without any knowledge of how
the results are consumed a higher degree of decoupling is obtained.
3.4.1
Composing tests
Tests can be run with a multitude of combinations of settings that influence how the test
should be run and results are gathered. For instance, a test that is run across different
problem sizes should be displayed differently from a test that only has a single problem
size. The different dimensions are the following:
 One/multiple runs
 One/multiple problem sizes
 One/multiple optimizers
Instead of having a single class manage all of these different possible combinations, the
design solves this problem by using the composite pattern from [30]. This common design
pattern facilitates composing objects from smaller objects that all implement the same
interface as the composite. This allows a consumer to use both the composite objects
or the individual sub-objects in the same manner, provided that these share the same
interface.
In the case of this project, the shared interface between composites and sub-objects is the
ITest interface. Leaves of the composite tree are single runs of one optimizer on a single
problem size. These runs can then be composed using one of the three composite classes:
 RunsComposite, which combines single runs by averaging the item results.
 MultipleVariablesComposite, which combines tests on different variable values by
creating new series over these values for all item results of the sub-tests.
 OptimizersComposite, which combines independent tests on different algorithm
configurations into a single unified object.
A diagram of the test composition classes is shown on figure 3.2. As shown on the diagram,
the structure is slightly more involved than the classic composite pattern, with the abstract
Test class adding an extra layer. Each class and its role in the overall composition of tests
will be described in further detail.
Test is the shared abstract ancestor class for all objects in the composite structure. The
class contains properties and logic that is shared across all test scenarios, such as keeping
track of the fittest individual and objects for synchronizing data access to avoid race
conditions. It is also in this class that the result properties are kept.
SingleRun represents the test consisting of just a single run of a single configuration
of an algorithm on one problem size and. This is the simplest test scenario, and more
complex scenarios are composed of multiple SingleRun instances. The logic for starting
new iterations of a Run object and consuming their results is also found here. As each
iteration is computed, the results are added to the respective properties of the SingleRun
object.
TestComposite is the abstract class from which all composites inherit, and it contains
code for keeping track of its sub-tests and running these in parallel.
This is done by
creating and running a series of asynchronous Task objects provided by C# and awaiting
their completion. Since a composite can potentially have many sub-tests, only a subset of
these will be run at a given moment. As a sub-test is completed, another one is created
and run. The size of this subset is defined by the MaxParallelization parameter in the
test specification. The task of consuming the results of a finished sub-test is delegated to
24
==PAGE:31==
Figure 3.2: Diagram of ITest composite structure
the child classes through the abstract ConsumeResult method, since this logic varies from
composite to composite.
RunsComposite represents the case of having to run the same algorithm multiple times on
the same configuration and then average the results of these runs. For performance pur-
poses, the sums for each returned property are kept and consuming a finished run simply
consists of adding its result values to the existing sums. These can then be divided by the
total number of finished runs to get an average. This results in a constant computational
cost for consuming each run.
VariablesComposite is the case in which the same test is being run on a number of
different values for a certain variable. In this case the composite takes all one-dimensional
results of its sub-tests and transforms it into 2-dimensional results. The data on the x-
axis is the value of the variable for the given sub-test and the y-data is the previously
1-dimensional result. The original 2-d results of the sub-tests are then discarded, since
the goal of the test is now to compare results across different variable values.
OptimizersComposite simply combines the sub-tests into a single one by concatenating
all results.
This is used in the cases where the same test has been run across differ-
ent optimizers to compare their performances back-to-back. This is the simplest case of
composition, since no data is modified or thrown away.
Since the number of SingleRun leaves can potentially be very large, certain measures
have been taken to avoid having to spawn many objects that each require allocating a
25
==PAGE:32==
non-trivial chunk of memory. Instead of taking a list of already instantiated sub-tests,
the RunsComposite instead takes a function to generate sub-tests alongside an integer
specifying a number of runs. This allows the composite to create and run the sub-tests as
they are needed and let the garbage collector to free up their objects as their results have
been consumed. Additionally, the SingleRun and RunsComposite can also be configured
to not save 2-dimensional data in the cases where we know that it will be discarded later
anyway. Without these measures, large tests would require massive amounts of memory
since potentially millions of SingleRun objects would be instantiated up-front and each
require quite a bit of memory.
3.4.2
Building tests
As previously mentioned, the TestProvider class builds an ITest object from a specifica-
tion object. This specification defines which problem, algorithms and termination criteria
to use, as well as the values to use for the required parameters. This leaves the question of
how the TestProvider selects the correct problems and algorithms from this specification
object and instantiates these with the proper parameters.
For this kind of issue the abstract factory pattern from [30] comes in handy. The factory
method pattern allows for separating the concern of instantiating a class from the actual
class itself, and instead delegates this responsibility to a family of classes that share a
common interface.
Using this pattern, the ITestFactory interface specifies a method
to create a test object from a specification.
The abstract generic TestFactory then
contains the bulk of the logic for creating tests, where the type parameter determines
the instance format of the test.
Finally, the two child classes BitStringTestFactory
and TourTestFactory each specialize in an instance format and contain mappings from
options in the specification to factories for fitness functions, optimizers and termination
criteria.
This way, the TestProvider only needs to select the proper factory depending on the
instance format and then delegate the creation process to it. This is done by looking at
the mappings contained in each factory and see whether they contain the options selected
in the specification. The instance format is determined from the optimization problem,
since this determines which algorithms can be used for the test.
When adding a new optimization problem, algorithm or termination criteria a correspond-
ing factory must also be added and registered in the mappings of the test factory. While
this does not entirely eliminate having to add new code when adding new components, it
limits the scope of additions and makes the code more independent on the number and va-
riety of these components. For future work, more general factory classes could be created
to instantiate any type of fitness function, optimizer or termination criteria. Combining
this with reflection in C# to automatically add mappings to test factories would mean
that adding a new component to the framework only required implementing its class.
3.4.3
Expression parsing
When running the same problem across different problem sizes, we would sometimes like
the problem and algorithm parameters to depend on the problem size.
For instance,
when running MIMIC it would be nice to have the population size be equal to n or for
JumpOffsetSpike we would like the gap size to be √n. Two options exist for implementing
such a feature: either the user can choose from a list of hard-coded expressions or the user
is allowed to enter an arbitrary mathematical function that takes the problem size as
input.
26
==PAGE:33==
The latter option was chosen for this project, even though it requires the additional com-
plexity of parsing these expressions.
Firstly, it must be decided which mathematical
operators and expressions to support. To keep things simple, a relatively short list of the
most basic operators were chosen:
 Addition and subtraction
 Multiplication and division
 Exponentiation
 Square root
 Base-2 logarithm
 Exponential function (ex)
The parsing should additionally respect the precedence of operators. This means that
computations of square roots, exponentials and logarithms are performed first, followed
by multiplication and division, and finally addition and subtraction. Parentheses should
also be supported, which overrule all other precedence rules.
The overall procedure of the parsing is to first scan the input string for known terminal
symbols. From this ordered collection of symbols, the parser creates an abstract syntax
tree (AST). From the AST a final expression object should be created. This process is
shown on figure (3.3).
Figure 3.3: Overview of expression parsing process
The first step before parsing is to scan the input string for symbols, creating a list of
symbols. This is the domain of the Scanner class, which simply traverses the string for
left to right, yielding symbols as they are found.
The following terminal symbols are
supported by the scanner:
+, -, *, /, (, ), log, sqrt, exp, ^, n and EOF
The EOF signifies the end of the input string. A last symbol c is also scanned by matching
numbers written in integer, decimal or exponential format. Examples of such values are:
2, 3.1415, 10E3 and 4.2E-2
From this list of terminal symbols the parser then constructs an AST. An AST is a tree
structure describing how the input string has been parsed using a grammar. Each leaf of
the tree represents a terminal symbol found in the string, while internal nodes correspond
to non-terminal symbols from the grammar.

3.4.4
Running Tests
The ITest interfaces exposes an Execute method that runs the actual tests. This method
returns a Task, which is a built-in class in C# representing a piece of computational work
in progress. A Task can be awaited by the caller, which blocks the calling thread until the
computation is finished.
When the execution of a composite task is triggered, it simply passes the call down to the
Execute method of its sub-tests. This gives the composite a list of tasks, one for each
of its sub-tests. These tasks can then collectively be awaited using the built-in WhenAny
method, which blocks the composite until one of the tasks is complete. The results of this
finished task is then consumed, and the remaining tasks are awaited in the same manner
until no more unfinished sub-tests remain.
This concept of task-based asynchronous programming is used for the parallelization of
test execution. Since all sub-tests are independent of each other, their tasks can be run
in parallel. This is handled automatically by the .NET runtime, which executes Task
objects on a series of threads separate from the main application thread. This way, the
parallel execution of test runs can be abstracted to a series of Task objects awaited by the
composite test objects.
Some tests require hundreds of runs of the same sub-test, making it infeasible to run all
these at the same time. Because of this, the user can specify the maximal number of
tasks to run in parallel at any given time through the MaxParallelization parameter in
the specification. When provided this parameter, the TestComposite only runs a limited
number of sub-tests at a time. When one sub-test finishes, a new one is then executed
and its task object tracked with the other ones currently in progress.
29
==PAGE:36==
The Execute method takes as input a cancellation token, which allows for a more graceful
test cancellation. A cancellation token contains a flag that tells whether the tests have
been marked for cancellation. If this is the case, the tests will return control back to the
caller when the current iteration is completed. This ensures that the tests are not in an
invalid state, which could be the case if their execution was stopped by other means such
as killing their process or thread.
As previously stated, the ITest objects allow for other objects to subscribe to any new
results using the observer pattern. Observable collections are already implemented for 2-
dimensional data in C# through the ObservableCollection class, which triggers events
whenever items are added or removed from the collection.
This way, subscribers can
simply add a delegate method to the relevant events of the observable collections, which
will then be triggered when results are computed.
For 1-dimensional data, however, this functionality must be custom-made. This is done
in the generic ObservableValue class, which contains a shallow getter and setter method
for a Value property. The setter method however also invokes the delegate methods of
all the subscribers, letting them know that the value has now changed. This allows for a
clean implementation of the observer pattern for objects of arbitrary types. In the case of
1-dimensiional data, the type parameter will be double since this is the type of the results.
3.5
Command-line interface
The command-line interface (CLI) component is a relatively thin wrapper for accessing the
Runner component from the command-line. All code for the CLI component is contained
in a single static Main method of the Program class. The command line program takes two
arguments: a path to the output results directory and a path to the input specification
JSON file.
The specification is read from the file and deserialized to a TestSpecification object,
which is then passed to the Runner component to create an ITest object. This test is then
executed, and its results are afterwards written to the results directory in a new file with
the same name as the input JSON file. The execution of the tests blocks the terminal
window until the tests are completed, and results for individual runs are logged to the
terminal as they are computed.
This allows for programmatic access to the framework, meaning that tests can be run via
scripts. Since the HPC cluster does not have any UI this also makes the framework usable
on the cluster.

3.7
Testing
The Core and Runner components were designed to maximize the testability of its main
classes, since this is crucial for ensuring correctness and high quality. Several aspects were
considered as to make the classes were testable:
 Making the interfaces of the classes simple, with few methods that each has a limited
number of parameters. This is to reduce the different test cases and possible param-
eter combinations that have to be considered when testing the class interfaces. As
coined in [31], this means making the classes deep by providing a lot of functionality
through a small interface.
 Using dependency injection, where the dependencies of the classes are provided
through the constructor. By making use of this technique the dependencies can more
easily be mocked, enabling the test cases to only test the logic contained within the
given class.
 Though dependency injection by itself improves the testability by a substantial
amount, it is also crucial to reduce the number of dependencies to other classes
and libraries. Each added dependency adds complexity to the class, which must be
properly dealt with during testing.
Since the CLI and UI components act as relatively thin wrappers for the Runner compo-
nent, these were not thoroughly tested for this project. Unit tests were written for the
major classes, which tested both so-called happy and unhappy execution paths where the
provided inputs are correct and incorrect, respectively.
In addition to the unit test suite, an integration test suite was also created for running
the system on various test specifications. This integration test suite reads all specification
files located in a specific folder and runs each of these as an individual test case, where a
test case passes if no unhandled exceptions occur. The specifications were written to test
a series of different scenarios on relatively small problem sizes as to not take a long time
to execute.

Having a unit and integration suite helped substantially when changes were done on the

4
Empirical Performance Testing
Measure what is measurable, and
make measurable what is not so.
Galileo Galilei
After designing and implementing the software framework, we can use it to evaluate the
performance of the selected algorithms. Since this evaluation is empirical, the results are
only able to reveal the performance for a sub-set of possible configurations and problem
sizes. The goal of the analysis is to identify the most likely regions of the parameter spaces
that yield good performance and to provide rough estimates on the performance. These
estimates are then used to compile a series of guidelines on the proper choice of algorithms
for optimization problems.
4.1
Methodology
As in other branches of science and engineering, it is important to adhere to a systematic
process when performing experiments. This is to ensure a higher consistency across the
results and to make sure that these can more easily be reproduced. This section aims to
explain the method used when performing experiments on the performance of the different
algorithms across various problems.
Theoretically, the best experiment would be one that tests all combinations of algorithm
configurations on all possible configurations and sizes of fitness functions. Such experiment
is unfortunately not feasible due to the infinitely many possibilities to configure these
algorithms and problems. So we must settle with a practical compromise: a test suite
that tests a selected finite amount of combinations.
Since the goal of the experimentation is to produce a series of guidelines for how to
configure the algorithms, the experiments can be guided towards better performances.
The goal will initially be to identify the areas of the search space where performance is
best. The experimental work will then narrow the search to examine only the performance
of algorithms using parameters in these optimal areas.
The experimental process used for this project revolves around three phases. Each phase
is executed in an iterative fashion, where the results are refined until proceeding to the
next phase. The three phases are as follows:
1. Exploring the performance of the algorithms across their respective parameter land-
scapes. This creates an overview of how the performance depends on the parameters
and which regions in the parameter spaces that are best.
2. Regression analysis of the dependency of parameters on the problem size. The goal
of this phase is to produce optimal expressions of the algorithm parameters for each
problem as a function of the problem size.
3. Testing the performance of the algorithms using the newly found parameter expres-
sions. This produces insights into how the selected algorithms can be expected to
perform for various problem sizes.
Figure 4.1 illustrates the format of results from each of the three phases, where performance
is measured in terms of fitness function calls. Other performance measures can also be
used, such as the average best solution quality or the required CPU time.
33

After phase 1, the parameter values with the best performance are read from the graphs
and plotted as a function of the problem size for regression analysis in phase 2. When a
suitable parameter expression has been found that fits the observed points, this can then
be tested on a wider range of problem sizes to get an overview of the general performance
of the algorithm as a function of the problem size. These results from the last phase then
form the basis for the final series of guidelines.

Due to computational limitations only problem sizes and parameters within a certain

Apart from terminating the test runs when the global optimum was reached, a stagnation

4.1.1
HPC workflow
Since each test consisted of many individual runs, the execution of the empirical tests
had a high computational cost. To speed up the experimentation the high-performance
computing (HPC) cluster of DTU was used to run the tests. This cluster consists of a series
of nodes each with either 12, 16 or 24 cores. In order to utilize this powerful computational
resource, the HPC requires its jobs to be specified in a special script format.
For the experiment the convention was chosen that each test specification corresponds to a
single job for the HPC. This way, an HPC script must be created for each test specification.
This creation was automated using a custom PowerShell script, which generates an HPC
script for each specification file and then using SSH uploads this to the cluster.

4.2
Results for OneMax
OneMax is the simplest of the selected benchmark problems to optimize, since the opti-
mization can be performed by optimizing each variable in isolation. This makes OneMax
suited to show how well any of the selected EDAs can follow a gradient to an optimum.
All selected algorithms were able to effectively optimize OneMax, but this was with a
large difference in the required CPU time.
The multivariate algorithms took a much
longer time to reach the optimum than the univariate cGA. This goes to show that while
the multivariate EDAs may have an edge over their univariate counterparts in terms of
solving deceptive problems, they come with an increased computational cost which is not
worth it for easy fitness functions.
4.2.1
cGA on OneMax
The theoretical results from [3] maps out the expected performance of cGA on OneMax
depending on the parameter K. In the paper, the authors show that the best expected
runtime of O(n log n) is found in the two asymptotic classes where K is Θ(log n) and
Θ(√n log n).
No theoretical results however exist for the best K-value within these asymptotic classes.
The algorithm was run with a range of values for K across different problem sizes, starting
from 100 and ending at 800. Figure 4.2 shows the phase 1 results of these tests both in
terms of required CPU time and fitness calls to reach the global optimum.

of K.

values of n. From each of these two charts the optimal value of K for each problem size can
be read. These values are then plotted as a function of the problem size and a regression
analysis can be performed to find whether there is any statistically significant relationship
between n and K.
Figure 4.3 shows the results of this process both for CPU time and fitness function calls
on cGA for OneMax. As seen on the charts, the two regression analyses agree that the
best fit is Θ(√n log n), and also more or less on the constant scalar that yields the best
performance.

4.2.2
MIMIC on OneMax
For MIMIC, the variable parameter in the first phase is the population size. Results from

terms of both fitness calls and CPU time. For population sizes that are just a little too
small, MIMIC takes a very long time to find the global optimum. Once a certain threshold
has been reached, however, the algorithm can easily find the all-ones string efficiently. As
the population size then increases, the algorithm becomes slower and slower due to the
higher number of individuals to consider for the model building. The sweet spot seems to

ensure fast convergence but not large enough to carry too much redundancy.

4.2.3
FastMIMIC on OneMax
For FastMIMIC, we must obtain the optimal expression for not only the population size

performance is first mapped out as a function of sampling intensity, for which an optimal
expression is found through regression. Fixing the sampling intensity to this expression,
phases 1 and 2 are then repeated for determining the best population size. While this
approach does not guarantee the optimal combination of the two parameters, it seems to
yield reasonable results.

model building.

4.3
Results for LeadingOnes
The fitness function of LeadingOnes is a bit more complicated, requiring the algorithms to
also factor in the position of the bits in their optimization strategies. LeadingOnes requires
that the correct flipping to 1 happens at the active bit at each step, which intuitively should
make convergence take longer than for OneMax.
4.3.1
cGA on LeadingOnes
The cGA algorithm was run with a maximum of n log n stagnated iterations before a
given test run was terminated. Values of K between 1 and 300 were tried for cGA on
LeadingOnes, but the algorithm as unable to find the globally optimal all-ones string in
any test run before stagnating.
The values of K that resulted in the best fitness were found to be relatively small. Intu-
itively this can be explained by the fact that lower values of K mean that the probabilities
are changed by a larger value after each iteration of the algorithm since the size of this
change is inverse in K. The issue with cGA on LeadingOnes is most likely that the al-
gorithm has to optimize the active bit before being able to proceed. If this optimization
is too slow, the algorithm will not produce an improvement to the best fitness and the
stagnation criterion will kill the run. Small values of K result in cGA changing the active
bit to 1 more quickly, and thus result in farther convergence.
Another factor in the poor performance of cGA on LeadingOnes is that the probabilities
are also updated for the bits that have not yet been optimized, even though these play no
role in the fitness value of the solution. The cGA algorithm is tricked to believe that the
zero values in the irrelevant bits are part of the improvement, increasing the probability
39
==PAGE:46==
that these become zero again. When these bits are then the active one, their probabilities
are near the lower margin, making cGA take a long time before realizing that this bit
needs to be a 1.
The expression of K that was found to converge farthest was 0.2√n log n. Running this
on various problem sizes showed that cGA was on average able to reach about halfway to
the all-ones string before stagnating. The required CPU time and fitness calls for this to
happen are both super-linear in n.
4.3.2
MIMIC on LeadingOnes
MIMIC was able to consistently find the all-ones string for LeadingOnes with even small
population sizes. The algorithm also seems to be much less sensitive to variations in the
population size for LeadingOnes compared to for OneMax. While the population size still
has a significant impact on the performance of the algorithm, small changes to its value
no longer result in enormous changes to the required CPU time or fitness function calls.
Figure 4.8 shows the results of MIMIC on LeadingOnes. While the algorithm is possible
to find the global optimum even with small populations, the computational costs of doing
so seem quite large. The optimal population was found to be 0.0064n2, which makes the
computational cost scale with at least Θ(n4). While this is no issue for smaller problem
sizes, it indicates that MIMIC is not a very efficient algorithm for larger problem instances.
This is confirmed by the results of the third phase, where MIMIC was run on larger in-
stances of LeadingOnes. While the algorithm can solve LeadingOnes efficiently for smaller
sizes, the computational cost explodes for instances larger than 100 bits.
Figure 4.8: Average performance over 100 runs of MIMIC on LeadingOnes across various
population sizes.
4.3.3
FastMIMIC on LeadingOnes
FastMIMIC seems to have radically different performance across different sampling inten-
sities. Figure 4.9 shows the results of running FastMIMIC on LeadingOnes across different
sampling intensities while fixing the population size to n.
The algorithm is unsurprisingly unable to find good quality solutions when the sampling
intensity is small. However, as the sampling intensity nears n the quality of solutions seems
to drop again. This is the exact point at which FastMIMIC becomes equal to the original
MIMIC. This indicates that the original MIMIC algorithm somehow considers too many
variables when building its model. In other words, making MIMIC slightly ”dumber” not
only makes it faster but for some reason also makes it yield solutions of better quality.

The best sampling intensity was found to be 0.78n using regression analysis of the results

GOMEA on LeadingOnes

each variable has its own cluster in the linkage tree generated by GOMEA. Each cluster
is copied upon mixing, which is followed by a call to the fitness function for whether this
results in an improvement. This makes GOMEA perform at least one improvement in
each iteration, making the number of iterations bounded by the problem size.
The optimal population size for GOMEA on LeadingOnes was found to be 1.58n, and

When measuring the performance with the number of fitness

4.5
Results for DeceptiveLeadingBlocks
In DeceptiveLeadingBlocks (DLB), the goal of the optimization is to converge as far as
possible towards the all-ones bit-string without being caught in the local optima of one
of the blocks. Depending on the size of the blocks this can vary in difficulty, with larger
blocks requiring a bigger ”jump” to escape the current block. Essential to optimizing
this problem effectively is that the algorithm is able to identify and manage dependencies
between bits, which multivariate EDAs are specifically designed to do. This project has
focused on the performance of the selected algorithms on DLB with block size 2.
4.5.1
cGA on DeceptiveLeadingBlocks
As expected, cGA is unable to overcome the non-trivial dependencies between bits in DLB.
This results in cGA getting stuck in local optima and not being able to continue before
the stagnation criterion kills the runs, giving solutions that are far from the optimal all-

4.6
Results for TSP
Due to the much higher computational cost of optimizing permutation-based problems
compared to those working on bit-strings, the empirical tests were performed on much
smaller instances of TSP. The tests were mostly performed on randomly generated in-
stances of TSP consisting of vertices that were uniformly sampled from the unit square.
Distances between two vertices were then simply the Euclidean distance between these in
the plane. This class of TSP instances was used due to the fact that such instances have
similar structure across different sizes, which is not necessarily the case for all TSPLIB
instances. The TSPLIB library also only contained a limited set of instances that were
small enough to be feasible for the selected algorithms. Using uniformly generated in-
stances thus allowed for arbitrary problem sizes and better comparison of results across
instances of different sizes.
Uniform instances of TSP however have certain properties that might make them easier to
solve compared to arbitrary instances of TSP. Firstly, the worst-case optimal tour length
of instances sampled in the unit square is bounded by Θ(√n) [32]. Secondly, uniform
instances are symmetrical, which means that the distance between two points is the same
regardless of which point is the source and which one is the destination. The triangle
inequality also holds for such instances, arguably making them slightly less deceptive
since strategies guided by local improvements such as 2-opt and Christofides’ algorithm
[1, 33] yield reasonable results. In fact, an optimization heuristic with polynomial runtime
exists for computing tours that are at most longer than the global optimum by a factor
(1 + 1/c) for arbitrary constants c > 1 [34].
4.6.1
TourMIMIC
Unsurprisingly, larger populations for TourMIMIC result in shorter average tours at a
higher average computational cost. Figure 4.19 shows the performance of TourMIMIC on
various sizes of uniform TSP instances across different population sizes. Unlike in the case
of most bit-string problems, no optimal population size is apparent in terms of finding
good-quality solutions at a fast computation time. We are posed with two choices that are
mutually exclusive: having either a population size that gives good solutions or one that
gives fast execution. Thus, we cannot perform phase 2 regression since the best population
size depends on practical circumstances.
4.6.2
TourGOMEA
The performance of TourGOMEA was very similar to that of TourMIMIC. Figure 4.20
shows the performance of TourGOMEA across various population sizes. Again, there is
no clear-cut optimal value for the population size due to the trade-off between solution
quality and fitness function calls. In general, the TourMIMIC calls the fitness function
an order of magnitude less than TourGOMEA and produces solutions that on average are
slightly better.

is the best-performing of the selected EDAs for permutations, FastP4 seems to be the best

is able to limit its requirements for memory and computation time.
4.6.6
Effects of perturbation
Uniformly generated TSP instances have special properties that may not necessarily apply
to arbitrary instances. Their vertices are in expectation distributed evenly across the unit
square, which results in instances where the expected distance between all pairs of vertices
is the same. For real-world applications, however, we have no such guarantees and vertices
are often clustered.
A middle way between using TSPLIB instances and uniformly generated ones is so-called
smoothed analysis [35] of TSP. Beginning with a predefined instance from TSPLIB, Gaus-
sian noise is added to each point in order to add a factor of randomness to the instance.
The noise used for this type of perturbation is sampled from a Gaussian distribution with
mean 0, and the level of perturbation is defined by the standard deviation, σ, chosen. As
51
==PAGE:58==
Figure 4.22: Performance of FastP4 on uniform TSP instances across various lengths of
shedding interval.
the value of σ increases, the position of the points are to a higher extent defined by the
random noise added, until a uniform instance is approximated.
To ensure that the results can be compared across different levels of perturbation, the
newly scrambled points are confined to the unit square. If a point lies beyond the unit
square after adding noise, its coordinates are simply wrapped around. Otherwise, the
points would simply move farther away from each other as the value of σ went up, making
comparison between different noise levels harder.
For performance reasons, the multivariate EDAs were run on a reduced instance of the
berlin52 TSPLIB instance, which consists of the 30 first points of the original instance.
Figure 4.23 shows the performance of TourMIMIC and TourGOMEA across various levels
of perturbations. The performance changes greatly for small perturbation levels, but the
instance quickly becomes saturated with noise. After this point of saturation, adding more
perturbation does not have any effect on the performance, which is reflected in the graph
by the noisy stagnation in both solution quality and CPU time. This is most likely because
the instance becomes equivalent to a uniform TSP instance after enough perturbation is
added. As such, adding any more noise does not alter the distribution of vertices, making
any additional perturbation have no effect.
An interesting observation to point out is that as noise is increased the produced solutions
have worse quality while the required CPU time of the algorithms drops. This is most
likely due to the fact that high levels of perturbation eliminates any clustering of the
points, which otherwise results in shorter tours. As perturbation increases, the average
distance between points increases until saturation is reached. Perturbation also somehow
makes the required CPU drop. This is likely due to the algorithms stagnating quicker,
which may indicate that at least the reduced berlin52 instance has less deception than
uniform instances.

4.7
Guidelines
Each of the selected problems reveal certain strengths and weaknesses of optimization
heuristics. After having evaluated the performance of the selected EDAs on these prob-
lems, the main points of the results can be compiled into a series of guidelines on which
algorithms to use depending on elements of the problems at hand. The hope is that these
guidelines can be used in the future to help on the choice of multivariate EDAs for practical
applications and to reveal possible avenues for future work.
The four bit-string benchmark problems reveal certain properties that a fitness function
can have which play an important role in the optimization landscape and choice of pa-
rameter.
Unimodal/multimodal:
This terminology is borrowed from [9], where a unimodal function is defined as one in
which every non-optimal solution has at least one neighbor of higher quality. A neigh-
bor is defined as another individual that can be obtained by flipping a single bit. While
unimodality of a function does not guarantee that it can be solved efficiently by greedy
approaches, it might indicate that the function lends itself better to optimization through
local improvements. This is the case for OneMax and LeadingOnes, which can be effec-
53
==PAGE:60==
tively solved by the HillClimb routine of P3. Multimodal fitness functions, on the other
hand, contain some level of deception and can also be characterized by the fact that they
contain at least one or more local optima.
Symmetrical/asymmetrical: Some fitness functions are defined using values that are
derived from global properties of an individual, such as OneMax and JumpOffsetSpike that
both use the total number of ones. Here it does not matter which variables are in fact set to
1 and which are not, meaning that variables are completely irrelevant for determining the
fitness of an individual. This is in contrast to LeadingOnes and DeceptiveLeadingBlocks
which both require the correct variables to have the correct values. This way, a fitness
function can be characterized as being symmetrical if the fitness calculation does not
factor in variables but only depends on global properties regardless of which variables
contribute. If this is not the case, variables are relevant for the computation and the
function can instead be said to be asymmetrical.
Grouping the four benchmark problems by these aspects, as shown on table 4.5, reveals
that each problem has its own unique profile. Using the empirical performance of the
algorithms on each of the problems, we can then produce a series of guidelines that possibly
generalize to other problems with the same properties.

For unimodal problems we could see that the HillClimb routine of P3 could consistently

For multimodal problems, it is recommended to use multivariate EDAs only when these

This is due to the fact that they are orders of magnitude slower
than their univariate counterparts such as cGA. As we could see in the results, cGA could
consistently find the global optimum for both OneMax and JumpOffsetSpike within a much
faster time than the other algorithms. For LeadingOnes, however, cGA was unable to find

well on all unimodal function, but rather that they are actually best if the given fitness
function is symmetrical. For situations where the function is asymmetrical, multivariate
EDAs are instead recommended.

than the second-fastest algorithm, FastMIMIC. FastMIMIC, however, called the fitness
function the fewest times by an extremely wide margin. This indicates that for problems
such as DLB where the fitness function is computationally cheap GOMEA is the best

Evaluation
The literature review yielded initial insights of the main ideas behind optimization, search
heuristics and estimation-of-distribution algorithms, as well as an overview of selected fit-
ness functions and algorithms in the field. These functions and algorithms were described
in detail and their main ideas explained, with the algorithms’ pseudocode also being pre-
sented. A goal of the literature survey was to map out the state of research on theoretical
results of multivariate EDAs.
Nearly no such results were found, indicating that per-
formance in the field of multivariate EDAs is still mostly determined through empirical
means.
The design of the software framework made use of several design patterns, which are
methods commonly known to produce cleaner solutions. The framework was supplemented
with a testing suite which validates the behavior of certain core classes and runs a series
of small integration tests to evaluate the correctness of the framework as a whole. While
the implemented framework contains the main features needed for the project and satisfies
the identified requirements, it would not hurt for it to be more extensively documented.
The framework can however serve as a solid starting point for future projects on empirical
performance testing of EDAs and other search heuristics.
The results for bit-string-based problems did not show a single algorithm to be superior
in all aspects, but rather revealed that a trade-off exists between modelling power and
computational speed. The main factors for determining the best algorithm for a problem
was found to be the modality and locality of the problem. Simple unimodal problems such
as OneMax and LeadingOnes were effectively and efficiently handled by the greedy hill-
climbing routine, while this was not the case for more involved problems with multimodal
structure. For this class of problems the best algorithms were found to be univariate EDAs
for symmetrical problems and multivariate for asymmetrical ones.
The results of this project supports several theoretical results from research, namely for
cGA on OneMax [3] and P3 on OneMax and LeadingOnes [9]. These show that cGA with
K in the order for Θ(√n log n) takes O(n log n) fitness calls on OneMax while P3 takes
O(n) on OneMax and O(n2) on LeadingOnes.
Surprisingly, the results revealed that the proposed FastMIMIC algorithm was able to
optimize difficult problems effectively using a much lower asymptotic number of calls to
the fitness function than the other multivariate EDAs such as GOMEA and the original
MIMIC algorithm. This improvement was found when the number of sampled positions
was somewhere in the middle of the allowed range of values. While the intuition for this
is not clear, we can observe that there is a correlation between making the model consider
fewer positions and the model being better at escaping local optima.
In addition, the proposed FastP4 algorithm also showed promise for reducing its com-
putational requirements both in terms of CPU time and memory usage. This however
came with the cost of yielding solutions of lesser quality than the original P4 algorithm.
Since the original algorithm was prone to crashing due to running out of memory, the
modifications are however still useful for practical purposes.
The results were quite different for permutation-based algorithms compared to those work-
ing on bit-strings. Due to the nature of permutations, the time and space complexity of
the permutation-based algorithms were much higher than those of the others. On top of
56
==PAGE:63==
this TSP is also NP-hard, making it much more difficult to reason and work with com-
pared to the simple benchmark problems on bit-strings. Because of this, the only clear-cut
guideline for multivariate EDAs on TSP is that higher population sizes result in better
solutions but also require more time and memory for computation.
Since the experimental work is based on empirical analysis, only a finite set of values
have been considered for parameters and problem sizes. This inherent limitation in the
experimental work means that some parameter expressions obtained through phase 2
regression analysis are based on a limited number of sample points, meaning that there
might actually be better expressions for larger parameter values which have simply not
been considered. This is reflected in the fact that some parameter expressions seem quite
arbitrary and likely to be over-fitting a small set of samples. An example of this is the
expression of population size for MIMIC on JumpOffsetSpike, 0.11n + 15, which provides
no intuition for either of its constant scalars.
In addition, the limited number of considered problem sizes also means that the conclusions
and guidelines produced from the results are in fact extrapolations onto larger problem
sizes. While uncertainty will never be fully eliminated for empirical analysis, increasing the
number of evaluated problem sizes would reinforce the generalization of these guidelines.
Throughout the experimental work, focus has been on two performance metrics: CPU
time and number of fitness calls. While the CPU time can in many ways be considered a
measure of performance in practical terms, the number of fitness calls is used the most in
literature. For univariate EDAs and other simpler evolutionary algorithms, the number
of fitness functions goes hand-in-hand with the required CPU time. This is however not
always the case for multivariate EDAs, where the most computationally taxing operation
is often building the model. Thus, for benchmark problems with fitness functions that are
cheap to evaluate CPU time is the preferred metric for multivariate EDAs.
As we saw with TSP, however, the number of fitness calls and CPU time moves in lockstep
when the fitness function takes longer to compute. In such cases, the number of fitness
calls is preferable to use as performance metric since this does not rely on hardware and
other test-specific elements.
Some of the selected multivariate EDAs were able to consistently find the global optimum
for all the evaluated problems. This optimization however took a long time, with running
times that are bounded only by large polynomials of the problem size. This in effect makes
the multivariate EDAs very slow for practical applications, where problem sizes can be in
the thousands or beyond. The issues of the algorithms’ computational complexity are only
compounded by using different representations such as permutations, where the running
times get out of hand for even smaller sizes.
This poses the question of what is then the potential for using multivariate EDAs for
real-world applications. While their strength certainly has been showed to not lie in com-
putational efficiency, this class of algorithms has shown to be able to effectively optimize
difficult problems without any knowledge of the inner workings of a problem’s fitness
function. This generality is perhaps what makes EDAs able to come into play, being an
optimization tool in cases where the function is unknown and no other methods are effec-
tive. We can imagine scenarios where a function needs to be optimized with no immediate
knowledge of its internals.
In such cases where the development of a problem-specific
optimization strategy is not feasible or economical EDAs can be used as a general out-of-
the-box heuristic.

Future Work
As mentioned in the evaluation section, the literature review only examined a limited
number of works.
Future work on this project can expand on the existing literature
survey with more papers, including those published in the future. This will result in a
more comprehensive overview of the field and in addition keep the review up to date with
the state of the art.
This project examined the challenges and possibilities for modifying EDAs such that they
are able to work with permutations instead of bit-strings. Another proposed avenue for
future work is to build upon these modifications to create variations of the algorithms
that work with other instance representations and constraints.
This can be combined
with a more extensive review of literature on the topic, which this project has only briefly
examined.
While this project has presented modifications to the MIMIC and P4 algorithms that
evidently yield improvements in performance, other viable ideas for modifications exist to
implement and evaluate. One such idea might be to use different distance metrics and
clustering strategies for GOMEA and P3 which might have certain advantages over the
currently used joint entropy measure. FastMIMIC might also be extended with additional
modifications to the original algorithm which might give further speedups and abilities of
overcoming deception of fitness functions. An idea could be to reconsider the way that
joint entropy is stored and used, reducing the memory usage from O(n2) to something
asymptotically smaller.
The point of criticism on the experimental part of this project has been the limited range of
problem sizes and parameter values used. Future expansions of this experimental work can
perform a more comprehensive exploration of the performance of the selected algorithms
by testing them on larger problem sizes and with a wider number of parameters.
A
parameter whose influence has not been thoroughly examined in this project is the number
of stagnated iterations before termination, which determines how long tests are allowed
to run without improvements until they are killed.
Additionally, only a handful of problems and algorithms were considered for this project.
In particular, the permutation-based algorithms were only tested on TSP, which is in itself
a quite involved problem. A possibly low-hanging fruit for future work would be to evaluate
the performance of the permtuation-based EDAs on simple and well-defined benchmark
problems similar to those for bit-strings. Some of the permutation-based optimization
problems presented in [36] might prove useful for this purpose.
The reason behind the long computation time for multivariate EDAs mostly lies in the
design of the algorithms themselves and the way they scale asymptotically with the prob-
lem size. Part of the long execution time can however probably also be attributed to the
performance of the framework, since it introduces substantial overhead for the test runs.
An avenue of future work is to optimize the framework in terms of performance such that
it runs as smoothly and efficiently as possible.

Conclusion
The overarching goal of this project has been to create an overview of multivariate
estimation-of-distribution algorithms (EDA) and their performance on optimization prob-
lems. This was done by firstly conducting a survey of existing literature on the topic and
identifying the main contributions presented. Throughout the literature survey a critical
point of view has been taken, which resulted in the proposition of modifications to the
MIMIC and P4 algorithms for enhancing performance and eliminating memory issues,
respectively called FastMIMIC and FastP4. While one objective of the literature survey
was to find theoretical results for multivariate EDAs, only very few were found. This con-
firms the suspicion that constructing mathematical proofs for multivariate EDAs remains
a challenge for research. The possibilities of making EDAs work with permutations was
additionally examined, and several approaches found in literature were presented.
A selection of 4 benchmark fitness functions and 4 EDAs was chosen, of which one was a
univariate algorithm to use as baseline. These were implemented in C# alongside a frame-
work for specifying and running empirical performance tests, allowing for both program-
matic control through the command-line and for manual control through a user interface.
In addition to implementing the original algorithms, the permutation-based variations
presented in literature were also implemented to be run on instances of the travelling
salesman problem (TSP).
A suite of tests was defined an executed on the DTU high-performance computing cluster,
which allows for massive parallelization using ample computing resources. The results
showed that while the univariate baseline algorithm was able to effectively optimize easy
fitness functions, it was unable to do so for more complex functions with deceptive aspects.
Here, the multivariate algorithms showed themselves to be much stronger, albeit with a
high computational cost of optimization. On the most difficult of the selected problems
the best algorithm in terms of required CPU time was found to be GOMEA. The proposed
FastMIMIC algorithm, however, used much fewer calls to the fitness function, indicating
that this might be superior to existing algorithms on problems with more computationally
taxing fitness functions.
In addition to testing the performance of the algorithms on bit-string problems, the
permutation-based variations were run on uniformly generated instances of TSP. Results
from these tests showed that the best algorithm for finding the shortest paths was the
P4 algorithm, but also that this algorithm showed a tendency to run out of memory for
even small problem sizes. Because of this, the best algorithm on TSP in practical terms is
arguably the proposed FastP4, which produced slightly longer tours but also did not have
any memory issues. Finally, a series of guidelines were produced on the choice of EDAs
for optimization problems.
The main aspects of the project were summed up and discussed, with the main points of
criticism being the scope of the literature survey, the design of the testing framework and
the limitations of evaluating algorithm performance by empirical means. Several ideas
for future work were finally presented to extend upon the contributions of this project.
These concerned both the breadth and depth of the performance evaluation, suggesting
that more algorithms and problems be examined and that the existing tests be extended
to more problem sizes and parameter values.
