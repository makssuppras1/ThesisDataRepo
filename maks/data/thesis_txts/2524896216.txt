I	    
 
Abstract	  
 
SDN emerges as a technology to decouple the network control and forwarding functions, 
it enables network control to become programmable and the infrastructure layer to be separated 
for network services. SDN is being utilized in network innovation development, leveraging SDN 
in IP mobility management is also a promising trend to solve problem resulted from traditional 
network devices. 
This project is to propose a SDN-assisted mobility management approach for the inter-
domain scenario, and evaluate the feasibility and the performances using simulation and 
emulation approaches. The architecture and topology of the mobility management protocol will 
be depicted. Then the investigation on the communication modules of the extended PMIPv6 and 
SDN controllers will be carried out to accomplish the handover signaling procedure. 
Additionally, when the multidomain tunnel is established, the protocol performance including 
handover latency and traffic engineering will be investigated and analyzed. The protocol model 
will be created in virtual environment VirtualBox with Mininet and Floodlight controller. Finally 
the development on Floodlight module with mature cluster technology of Hazelcast is 
implemented, an application with a web management page has been implemented, the network 
topology is able to be discovered in multi domain, and the flow table can be forwarded across 
multi SDN domain.

1.1 Research	  Background	  
The eduroam[1]is a successful roaming network access federation with Wi-Fi based, it has 
been initiated to provide the secure and global Wi-Fi roaming access service for international 
research and education communities since 2003, it now has covered various universities and 
research institution over 74 territories. Once the eduroam account is registered in home 
institution, a word-wide accessible roaming service could be utilized for students, researchers 
and all education staffs from participating institutions to obtain wireless Internet connectivity 
across campus and while visiting other participated institutions with this account. The 
coordination of activities between eduroam participants is founded by GÉANT[2] project 
following adoption of eduroam as a service in September 2007.The GÉANT community consists 
of Europe’s National Research and Educational Networks (NRENs) and currently associated 
with DANTE[3] and TERENA[4].GN3plus, the latest terms of GÉANT project is having over 
250 project participants in more than 40 countries across Europe. 
With the latest optical and wireless technologies migration and development, GÉANT 
Innovation Programs have been working on innovative programs to investigate the next 
generation of networking technologies with a broader range of emerging services, in order to 
meet the support of high-speed mobile networking, sensor networks and cloud services 
etc.Currently21 GÉANT Open Call Projects of Innovation Programs initiates to work on 
different areas: authentication, future network architecture, exploring SDN (Software Defining 
Network) potential on networking etc. In order to provide innovative networking infrastructure 
and excellent service in Europe towards 2020[5], fresh ideas are proposed for developing and 
operating the future network infrastructures and related services. 
SDN is a system that can optimize and simplify network operation, it allows a closer 
combination of interaction of network service and device, despite it is physical or virtual. 
OpenFlow separates the control layer from the traditional two-layer switch and three-layer router 
via a logical centralized controller, it utilizes an abstract flow table concept to forward network 
traffic in OpenFlow switch. Adopting OpenFlow enables manage network more efficiently, as

well as to form a more flatten hierarchy to handle network data processing. At the same time, 
Adopting SDN in network is not only limited to the core network, it can also be used in the 
access network, Local Area Network (LAN) or wireless access network. SDN is able to make all 
various traditional network layers and devices to unified virtual resource, hence these resource 
could be utilized by controllers. 
Additionally, the development and advantages of Long Term Evolution (LTE) enables it 
to be an alternative to extend eduroam from Wi-Fi to 4G network[6]. LTE user spectrum is more 
efficiently, air link is more robust than UMTS, and the VoLTE. Moreover, the average wireless 
LTE speed could reach almost 40.5 mbps in downlink and 19.8 mbps in uplink[7], and LTE 
commercial usage is so

deliver high-speed mobile networking service and to realize a seamless mobile education and 
research at any time in any place on any device in future, considering the user high-speed request, 
4G/LTE can be feasible and effective complementary to Wi-Fi for eduroam[8].

The objective is to realize the multi-domain eduroam architecture across LTE and Wi-Fi. As 
shown in Figure	   2, this new architecture aggregates eduroam service via public Wi-Fi-based 
hotspots and via LTE radio access network provided by vendors, and so far the pilot project of 
4G eduroam have been implemented at SURFnet[9] in Netherland and Janet in UK which are 
supported by was granted by GÉANT3Plus in the framework for open call proposals, they are 
exploring to connect the mobile operator to the NREN’s infrastructure.  
To integrate LTE as alternative of eduroam access technology, the wireless campus 
network can be accessed almost everywhere. The advantages of this seamless integrated access 
network architecture of 4G and eduroam can be described as following: secure access based on 
eduroam credentials; Users could be always connected; It allows all students, researchers and 
staffs able to collaborate, research and learn anytime and anywhere they move to; It could create 
a technology agnostic campus of various access technology of Wi-Fi/4G or 3G; It enables the 
native integration with unified communication (voice call, e-mail,…), remote E-learning and 
local printing etc. It offers trusted and secure connectivity. 
Due to the vertical handover issued caused by multi-domain eduroam across heterogeneous 
network of LTE and Wi-Fi, the inter-domain mobility management protocol across multiple 
domain needs to be investigated.

1.2 Challenges	  and	  Motivation	  
 
It is so promising to have a high-speed and seamless heterogeneous wireless network for 
eduroam in future with Wi-Fi and 4G/LTE, yet besides the physical implementation issue needs 
to be overcome, there are several problems needs to be figured out. 
• Necessity on design of mobile IP architecture and mobility protocol 
Due to the fact that this new architecture with integration of different access technologies 
will be leading to heterogeneous network vertical handover, when user is moving from home 
domain with Wi-Fi-based to a visited domain with LTE-based or vice versa. Users should be 
unaware of the location difference and handover issue across heterogeneous network, 
consequently it is necessary to design and develop an accessible and seamless mobile IP 
architecture and regulated handover protocol, or handset vendor might need to support the usage 
of handover in different technologies.  
• Necessity on utilization of SDN 
Moreover due to the fact that the existing networking technologies are based on a systems 
which is actually designed over several decades ago, it is also challenging to implement this 
seamless integration across different hardware and various network protocol. Software Defined 
Networking (SDN) emerges today as a promising technology for decoupling network control and 
data plane, and it brings programmable solution to network vendors to reduce complexity and 
costs of managing heterogeneous networks and services. SDN could provide feasible network 
flexibility and the potential to meet latest networking demands[5]. GÉANT Open Calls has also 
supported to investigate the potential application of adoption of SDN technology in future 
network.  
Above all, the strength of integrated eduroam over Wi-Fi and 4G enables it could become 
a promising alternative in future, this ubiquitous wireless coverage allows all students, 
researchers and staffs able to collaborate, research and learn anytime and anywhere. Nevertheless, 
the inter-domain vertical handover occurs across heterogeneous eduroam domain, it is necessary

to investigate inter-domain IP mobility management protocol to satisfy the real-time 
communication request assisted with the latest technologies. 
1.3 Contribution	  and	  Structure	  
 
The main objective of this Master project is to research on proposing a new network-based 
inter-domain mobility management approach over SDN assisted based on PMIPv6 protocol, 
which is feasible and has better performance. This protocol is to keep the home IP address and to 
preserve the TCP connection in the entire mobility procedure; it is required to set up a multi-
domain tunnel between LMAs in different domains.  
The architecture and topology of mobility management protocol will be depicted firstly, 
and the investigation on communication signaling module of LMAs will be given, Furthermore,

It is expected that this work offer a feasible and scalable mobility management protocol 
for future eduroam heterogeneous network architecture with short handover latency, and provide 
a convenient and efficient management method for each network vendor.

2 IP	  mobility	  management	  
 
This section analyzed all the theoretical and technical part on current IP mobility management 
protocol.	  
2.1 Current	  work	  
	  
Mobility management for wireless network has actually been one of research focus 
recently. The most commonly used Internet protocol is IPv4, nevertheless, the mobility feature 
was not considered in designing at the beginning. Additionally, with the rapid increasing users, 
the IPv4 address is almost running out, then the next generation IPv6[10] has becoming the key 
core protocol in mobile Internet protocol.  The IPv6 inherits most of merits from IPv4, and it has 
enlarged each function IPv4, especially the expand function on mobility in IPv6 could provide 
multi-layer support to mobility management. 
Mobile IP technology [11] ensures the communication accessibility and connectivity when 
mobile device is travelling or roaming among networks without changing the IP address. The 
IETF has proposed MIPv6 [ 12 ] (Mobile IPv6) protocol in 2004 to enable consistent 
communication for Mobile Node (MN) in IPv6 networks.  As a host-based mobility management 
protocol, MIPv6 requires the mobile device to involved in the entire interaction of handover 
signaling procedure: network discovery, configure IP address, authentication, binding update etc. 
Yet the handover latency may occur in each stage and lead to packet loss during data 
transmission. To decrease the handover latency and enhance the handover performance, several 
global mobility management protocol have been proposed, e.g. HMIPv6[13] and FMIPv6[14] etc. 
Although these protocols have solved the mobility issue in the network layer, the long 
registration latency and large packet loss have not been fixed effectively. The host-based 
protocol is contradictory to the above request, it is necessary to propose network-based mobility 
protocol. 
Even if the wireless network vendor has done their best effort to provide secure and 
seamless access network, the communication QoS would be depressed largely definitely due to 
packet latency and packet loss during handover. In order to overcome the following cons of the  
==PAGE:22==
	  
8	    
 
host-based mobility management protocol: high handover latency, complicated terminal protocol 
and difficulty to manage etc. IETF has proposed Proxy Mobile IPv6 (PMIPv6[15]) in 2008. 
PMIPv6 is network-based mobility management protocol and introduced two new network 
entities: Mobile Access Gateway (MAG) and Local Mobile Anchor (LMA). MAG handles 
mobility related operation instead of MN, LMA possesses Home Prefix (HP) of MN and it could 
save the binding message of MN, forward routing message from or to MN.  One of best features 
of PMIPv6 is that the MN is agnostic to the different networks and it always access to home 
network alike, because that MAG handles all mobility related signaling. PMIPv6 decreased 
handover latency, reduced the signaling cost and enhanced the network management capability. 
However, the current PMIPv6 only defines the intra-domain handover, while the inter-domain 
handover protocol in heterogeneous network has not been standardized. 
2.2 IP	  Mobile	  management	  
	  
2.2.1 Background	  of	  Network-­‐Based	  
 
Although MIPv6 is able to solve the terminal mobility problem, there still exist a lot of problems 
to achieve the request on latency sensitive communication. The global host-based mobility 
management protocol has the following common problems[16]: 
• Large update latency 
Either when global mobility anchor or CN is moving away from the network which the MN 
access to, it takes relatively longer time to update global mobility routing. However, during this 
update procedure, the data packets are still be sending to the previous network which the MN 
access to, excessive data packets will be lost and it has a bad impact on communication quality. 
• Large handover signaling cost 
It takes large handover signaling steps when MN is moving from current link to another link, the 
IP address needs to be configured in new link, permanent address of HA has to be mapped into 
the new link temporary address etc. Due to the limited wireless network bandwidth resource, 
larger signaling cost would significantly impact the real-time communication. 
==PAGE:23==
	  
9	    
 
• Security issue  
When MN moves into a new link, in order to sustain the communication with CN and HA, 
MN has to change temporary address, notify CN and HA this new temporary address through 
Binding Update (BU) message. If there is any illegal attacker in this network, this message might 
be intercepted and analyzed by attacker, the network prefix in which MN is located will be 
exposed to attacker, which is a threaten to MN security and confidentiality. 
Overall, when it comes to this scenario of MN frequently moving or roaming across 
networks, the host-based global mobility management protocol is not a best choice, it is urgent to 
propose a low complexity localized mobility management protocol that is proper for this 
scenario, large numbers of MN are moving among local smaller range. Therefore, PMIPv6 
relieves the burden on MN, and MN could get rid of the support of mobility protocol.

2.3.1.2 Components and Terminology 
 
LMA and MAG are the most 2 significant core components in PMIPv6. 
• 
MAG: when MN enters in PMIPv6 domain at the first time, and attaches to MAG, it

obtain Network Authentication Identity (NAI) of MN to see if MN can be endorsed network

When MN has attached any domain, MAG obtained MN address and it sends to LMA the PBU 
message for registration request. Besides, MAG is responsible for detecting MN status rather 
than MN executing initiation of binding operation, without MN signaling involved in setup stage 
of binding procedure, the handover latency is decreased and wireless network bandwidth 
utilization is raised. 
• 
LMA: it is responsible for the MN accessibility, as well as topology anchor of MN HNP. 
Its basic function is to forward data packet from CN to MN and forward data packet from MN. 
In brief, LMA is similar to the Home Agent (HA) in host-based mobility management protocol. 
LMA has to complete the following functions: 
§ For every MN which has attached to this domain, it maintenances a Binding Cache (BCE) 
to log MN related message. 
§ Set up bi-direction tunnel with MAG to forward data packet to MN. 
§ Allocation HNP, when LMA received PBU from MAG representing MN, and made 
relevant handling, it will send a unique HNP to MN. 
• 
Mobile Node (MN): MN is an IPv6 node, it refers to portable device like laptop, mobile 
phone etc. MN is able to communicate via wireless interface while changing location. MAG is 
responsible for managing MN mobility completely. When MN attaches to IPv6 domain, it 
attaches to LMA logically and to MAG physically. 
• 
Local Mobile Anchor Address (LMAA): LMAA is address configured to LMA, in order 
to communicate with MAG, additionally LMAA is an end node in bi-directional tunnel between 
MAG and LMA.  
• 
Proxy Care of Address (pCoA): pCoA is address configured to MAG, it is another end 
node in bi-directional tunnel between MAG and LMA. LMA uses pCoA to forward MN. 
• 
Home Address of Mobile Node (MN_HoA): MN_HoA is address for MN 
communication, MN could use this address in any access node of PMIPv6 domain. 
•

• 
Home Network Prefix (MN_HNP):  MAG broadcasts this prefix for each MN, it is 
decided by LMA, MN configures its own IP address via this prefix.  
• 
Mobile Node Identifier (MN_NAI):MN_NAI is an identifier for MN to perform

•

2.3.2.1 MAG forwards data  
 
• 
Forward data that is sent to MN_HoA 
Once MAG receives packets from the bi-directional tunnel, MAG dismantle outer header, and it 
can determine the destination address of the packet according to the inner data header, and then 
forward the packet. If the CN also links to the MAG, MAG needs to check Enable 
MAGLocalRouting option flag before forwarding data. If the flag is true, it indicates that the 
local routing optimization is allowed, CN and MN can communicate with each other directly 
without using the bi-directional tunnel; if the flag is false, it is not allowed to route optimization, 
communication between MN and CN needs to be forwarded through the bi-directional tunnel. 
• 
Forward data that is sent by MN 
After MAG receives a packet from the mobile node MN, if the destination address is a CN which 
is linked to the same MAG, then the EnableMAGLocalRouting flag needs to be checked to 
determine whether the route optimization is needed. If MN and CN are not attached to the same 
MAG, MAG should firstly check if the MN is binding to LMA. If the confirmation is successful, 
MAG forwards data packets through a bi-directional tunnel after encapsulate package (src=pCoA, 
dst=LMAA).

When LMA receives the data packet from MN to CN, the tunnel header has be removed, LMA 
identifies the destination address of the inner header, it performs packet forwarding. The 
destination address of the inner header is CN's address, and the source address is the MN_HoA.

2.4 Leveraging SDN in IP mobility management 
	  
The Software Defined Network (SDN) provides a new network architecture with 
decoupling the data plane and the control plane of the device, which could be a new solution for 
the study of the future network. SDN enables the network to become a computer infrastructure 
which is able to be programmed and managed in large scale. It enhanced network capability and 
flexibility to manage and expand the network for adopting SDN either in vendor environment or 
data center or school campus net. In order to leveraging SDN in IP mobility management area, 
SDN technology will be investigated here. 
2.4.1 SDN definition and framework 
	  
2.4.1.1 SDN framework 
 
Due to different commercial participant has investigated on SDN from various opinions, 
SDN architecture has many types. ONF (Open Networking Foundation) has defined a SDN[18] 
definition, architecture and market requirement. The main features are: decoupling data plane 
and control plane, centralized control and direct programmability. The control capability is 
migrated from independent network device to computable device, enables network basic 
infrastructure become abstracted to a logical or virtualized entity to application and network 
service. 
This ONF-defined SDN architecture is shown in Figure	  7. This logical view is composed 
of 3 layers and 2 interfaces. From the top to the bottom, Application layer is a group of 
application which are led by user request; Control Layer is the core in SDN, it is responsible for 
global network topology, collecting and maintenance link status info, give control command etc; 
Infrastructure Layer in general is abstracted to a data plane and is in charge of forwarding data, it 
consists of either hardware forwarding device or software switch like Open vSwitch. The  
==PAGE:34==
	  
20	    
 
2 interfaces are: Southbound API connects Control Layer and Application Layer, which is 
defined as OpenFlow by ONF; Northbound API connects Control Layer and Application Layer, 
the interface requirement is not standardized due to high complexity. 
	  
Figure	  7	  ONF	  defined	  SDN	  architecture	  
This architecture helps vendors to configure the abstract network via programmable approach, it 
also supports APIs to realize ordinary network services like routing, broadcasting, secure, access 
control, bandwidth control, traffic engineering, QoS etc, as well as various strategy management, 
which are able to be customized to satisfy the request. 
	  
2.4.1.2 SDN Features 
	  
Despite the existence of various forms of SDN architecture, but it is only the different 
manifestations of the concept of SDN architecture. As can be seen from the commercial 
significance of SDN, although SDN architecture based on different definitions, its essence is 
inseparable from the following basic features[19-20]: 
1. Decoupling data plane and control plane: traditional network device integrated control 
and forwarding in a same physical device, it is a tight coupling relation. SDN decoupling makes 
these two functions to be independent to each other and be more flexible. Thus vendors or users  
==PAGE:35==
	  
21	    
 
could develop some automatic tool to replace manual management task, which will decrease 
operating cost, unstable network resulted by operation mistakes.  
2. Centralized control. The logically centralized control makes it more convenient to 
manage network resources. It is highly effective and efficient for centralization to allocate 
network resource according to the trade-off between global network resource and requested 
demand, at the same time, centralized control simplifies the individuality of hardware device, 
unifies the entire the network as a single device. 
3. Open interface. The fact that whether interface of application and networks is open 
determines the approach needed to be taken during the application of network requirements. 
Opening bi-directional interface is of great significance, since it makes the utilization of network 
resources more efficiently. It benefits for application developer, and also enhances the utilization 
of network resources. 
4. Network virtualization. Opening south interface has compensated the differences in 
physical layer devices, simplified the upper layer network control as a logically virtualized 
network. The separation of logical network and physical network, networks can be easily 
controlled, which makes network no longer limited by physical network. 
These are three basic common features that are spontaneously followed by SDN 
participants. Different architecture reflects the different perspectives of proposer, but these basic 
idea of SDN will not change. SDN will turn the static network into a flexible and programmable 
platform, it will support dynamic resource allocation, enormous data center, dynamic virtualized 
and highly automatic Cloud. With multiple advantages and promising market trend, SDN 
technology is motivating innovation on network rapidly.

3 Mobility management across HRAN 
 
According to the previous introduction on the heterogeneous eduroam network over Wi-Fi 
and 4G/LTE, it is necessary to investigate the inter-domain mobility management to implement 
this multi-domain architecture successfully. Currently, the research on mobility handover 
management has been focused on intra-domain, whereas there is no standardized inter-domain 
mobility handover protocol. This section is about to propose a scheme focusing on mobility 
management in multi-domain. Firstly, a SDN assisted inter-domain vertical handover approach 
based on PMIPv6 is introduced, then the multi-domain. According to this approach, the scheme 
on SDN assisted mobility management across multi domain will be given in this section. 
3.1 SDN assisted vertical handover in HRAN 
 
ASDN assisted cross-domain vertical handover approach based on a Proxy PMIPv6 has 
been proposed in reference [34], the basic conception is to make Mobile Node to have the same 
Home Network Prefix to keep TCP through established inter-domain tunnels across 
heterogeneous RAN. 
This approach can be applied to multi-domain, each domain is governed and managed by 
its own SDN controller, SDN controller acts as a centralized entity[35],LMA and MAGs are 
SDN-enabled. When MN moves from a local AP to a foreign AP which belongs to another LMA, 
i.e. from home domain to a new visited domain, data traffic is transported in established bi-
directional tunnel between home LMA and visited LMA.

4.3.3 Topology discovery in Multi-Domain 
 
Each domain has its own topology and components, it is necessary to for each domain to 
get to know the topology of the other domain, here is the scheme to discovery the global 
topology in multi-domain networks. 
As we discussed in section 2, in order to implement topology discovery function, the 
LLDP is utilized, it allows network device to notify its identity and performance in local sub 
network LLDP provides a standard approach for network devices like switches, routers and RAN 
access point to notify existing message for itself, and it also saves the discovery message of 
neighbor devices. 
The controller discovers the link information in networks is to ask all of its switches to 
send out LLDP (Link Layer Discovery Protocol),which can be realized via this method: each 
LLDP packet is encapsulated into a packet_out message, the destination address is an 
unreachable address, then the action should be output. Each LLDP Data Unit is a sequence 
of type-length-value (TLV) structures, the TLV is used for distinguishing different controllers. 
Once one of a neighbor switch receives a LLDP, it has no idea how to forward this LLDP, 
this switch will forward this LLDP to its own controller; when controller receives this  
==PAGE:72==
	  
58	    
 
LLDP, and it starts to check this LLDP’s TLV to see if it is sent out by itself. If not, controller 
just throw away this LLDP; if this packet is sent out by itself, controller will record the packet 
transmit port and packet receive port, which means one link is discovered, and then this link will 
be compared with all of the known link, if this link is existed, it is not necessary to add the link 
and to notify updating topology, the packet will be dropped by controller at the end. However, 
when it comes to the cluster scenario, the controller is starting to escalate to a cluster, due to the 
fact that each controller has a different TLV, it is difficult to discover the entire link topology 
information by identifying other controller’s TLV. 
Therefore a type of a global TLV is proposed in application, which makes all controllers 
is able to receive same LLDP message from each other, eventually the global link information is 
discovered. The specific steps are shown in Figure	  29. 
 
 
	  
Figure	  29	  Topology	  discovery	  in	  multi-­‐domain 
There are two independent domain which is governed by controller C1 and controller C2 
respectively, the basic topology discovery flow steps are described as following: 
§ The Controller C1 asked OF switch S1 to transmit LLDP message; 
§ OF switch S1 forwarded the LLDP message to OF switch S2; 
§ OF switch S2 transmitted the LLDP message to Controller C2 via Packet-in packet; 
==PAGE:73==
	  
59	    
 
§ Controller C2 verified the TLV identifier in LLDP message and add the link message. 
This is the simplest scenario for only 2 domain with 2 controllers in each, when it comes to 
multi-domain, it is possible to discover the global network topology. 
4.3.4 Message forwarding in Multi-Domain 
 
When it comes to multi controllers in multi-domain, one controller received a message 
that actually needed to be handled by the other controller in another domain, it requires an 
effective approach to deal with this message.  
A message forwarding scheme is proposed below by combining distributed data base 
Hazelcast and a publish/subscribe model, which utilize the listener event in Hazelcat database. 
The working steps are shown in Figure	   30.Each controller will search the database to check 
switch and controller information, if it finds out which controller is governing the relevant switch, 
it will make that controller as a subscription ad configure the listener event to it, once there is 
related message comes, it can forward flow table to the corresponding controller immediately. 
 
 
	  
Figure	  30	  Flow	  table	  forward	  in	  multi-­‐domain	  
	  
==PAGE:74==
	  
60	    
 
The working flow is depicted as following: 
§ When Controller C1 has received a flow table which actually needed to be forwarded to 
OF switch S1 00:00:00:00:00:00:00:01 under controller C2, then C1 is searching its 
distributed Hazelcast data base to find out which controller is governing S1 
00:00:00:00:00:00:00:01; 
§ When C1 has found out that the S2 00:00:00:00:00:00:00:01 is under controlled by C2, it 
will utilize the publish/subscribe model to forward this flow table to C2; 
§ After C2 has received the flow table from C1, it sends out the flow table to this OF switch 
S1. 
The above is the simplest scenario, which contains only two domains with two controllers, 
when it comes to multi-domain by using this scheme, the message could be forwarded to each 
controller globally.

the scheme of how to forward flow table in multi-domain is proposed by using the 
publish/subscription modeling.

5.7 Scalable network 
 
All the implementation has done above is with two neighbor domains, and to be more 
important, this application based on Floodlight can be applied to multi domains more than 2. The 
entire network topology consists of two type: centralized and distributed, just as the previous 
topology discussion. 
In the distributed network, all cluster members are at an equal position and are parallel, it is 
not difficult to add a new member or domain into existing network, which needs to setup the 
member configuration in each controller, whereas the links between controllers is complex 
correspondingly. In the centralized network, the central core controller is actually a master 
controller. Since SDN controller cluster technology is also providing the master/slave controller 
role, this topology is not difficult to roll-out in large scale, yet the load on central controller is 
increasing.  
 
Based on all the design and implementation, the entire network topology is discovered 
and flow table is forwarded across domains, it is possible and viable to solve SDN controller 
communication issue in inter-domain handover scheme.

of data structure and module. Finally, result of each function can be shown in web front page.

6 Discussion 
	  
There are still some issues need to be considered in future work for this project.  
In this SDN management application, due to the difficulties and complexities of 
implementing hosts, only controllers and switches are updated in Floodlight module and can be 
shown in web application. It is necessary to investigate a management application to govern all 
of the network components.  
Additionally, after the signaling procedure are realized in inter-domain handover 
procedure across heterogeneous domains, the data transmission procedure has to be considered. 
When new user moves from local network to a visited network while utilizing some P2P, real 
time network service, it is necessary to implement a traffic engineering function to limit visitor’s 
network behavior during the visiting period, to ensure local users’ service quality. There already 
exists a QoS module in Floodlight project that is able to limit and control traffic in data 
transmission procedure, which is a possible solution to realize traffic engineering model in inter-
doamin IP mobility management protocol. The categorization of traffic and the priority of users 
are needed to be taken into consideration.  
This application has been tested in only 2 neighbor domains, due to the limitation of host 
hardware, it is also necessary to test this application in large scale network to obtain the range 
and scope of controller numbers, and the different impacts from different topology type.

7 Conclusion 
	  
The promising future of integrated eduroam over Wi-Fi and 4G is able to lead to seamless 
wireless coverage inside and outside campus, which will allow all students, researchers and 
employees to collaborate, research and learn anytime and at anywhere. Therefore, in order to the 
realize the integrated architecture across heterogeneous eduroam domain, it is necessary to solve 
vertical handover issue by investigating on inter-domain IP mobility management protocol, while 
to satisfy the real-time communication request assisted with the latest technologies. 
Currently, SDN emerges as a technology to decouple the network control and forwarding 
functions, it enables network control to become programmable and the infrastructure layer to be 
separated for network services, leveraging SDN in IP mobility management is also supposed to 
be a viable solution. 
Therefore, this project is to propose a SDN-assisted mobility management for the inter-
domain scenario, and evaluate the feasibility and the performances using simulation and 
emulation approaches, it provides an application based on development on Floodlight to 
implement communication module across multi domain. 
Beginning with an introduction of the research background, due to the fact that vertical 
handover problem of heterogeneous network is aroused by different access technologies of 4G 
and Wi-Fi, the necessity on design of mobile IP architecture and mobility protocol is given, as 
well as the difficulties on controlling and managing network based on the traditional network 
architecture and network devices, the necessity on utilization of SDN is given. Thus, the 
investigation on mobile IP management protocol and leveraging SDN in protocol are given later. 
Section 2 has described and depicted all theory and technology used in this project. Based 
on the analysis of IP management protocol host-based and network_based, the advantages of 
PMIPv6 including decreasing handover latency, reducing the signaling cost and enhancing the 
network management capability, makes PMIPv6 a good choice to be utilized to solve vertical 
handover issue across heterogeneous network, yet there only is standardized intra-domain 
protocol, thus the investigation of inter-domain handover protocol in heterogeneous network will 
be given next in section 3.  
==PAGE:98==
	  
84	    
 
In section 2, the detail analysis on intra-domain PMIPv6 has been given. The basic 
principle of PMIPv6 is to complete the detection of MN, initialization of signaling, configuration 
of routing while without MN involved. The two core components in PMIPv6 LMA and MAG, 
MAG sends routing solicitation request to configure IP address, LMA is responsible for the MN 
accessibility, how LMA and MAG forward data are introduced. The signaling process of MN 
attach and MN handover are described.  
Additionally, in order to leverage SDN to IP mobility management, the details of SDN 
technology are introduced here. During the process of studying SDN framework, SDN features 
and key technology, SDN advantages includes decoupling data plane and control plane, 
centralized control, open interface and visualization network,  the research has been focusing at 
the following area: network abstract, centralized intelligence and programmable API, which all 
makes SDN an promising trend to solve existing network issue. Openflow is one of important 
southbound protocol in SDN controller, the analysis of flow table, packet processing procedure 
and data header matching process has indicated how SDN controller handles data forwarding. 
For discovery link and network topology management, LLDP and its TLV signaling format has 
been investigated. Since eastbound-westbound of SDN controller has not been standardized, the 
SDN cluster technology is introduced for scalability of controller in multi-domain. Accordingly, 
leveraging SDN in IP mobility management is supposed to be a good method to govern network.  
In section 3, the SDN assisted inter-domain vertical handover across heterogeneous RAN 
has been proposed, including network architecture and signaling procedure. This scheme is a 
possible solution to handle handover agnostically for user without re-establishment of TCP-IP 
connection, the analysis on signaling procedure during handover across domain has aroused two 
question of investigation on new inter-domain PBU/PBA message transmission and on the 
communication between controllers. Thus, an extension on PBU/PBA message format of 
PMIPv6 has been given to accomplish the inter-domain signaling message forwarding. 
Furthermore, based on the analysis of inter-domain scheme between 2 neighboring networks, the 
advantages and disadvantages of multi-domain topology are analyzed, the centralized network is 
easy to scale whereas has large burden on center controller, the distributed network is easy to 
corporate while with large number of links between controllers. Then with a basic requirement of  
==PAGE:99==
	  
85	    
 
wireless network the handover latency of this inter-domain scheme has been given, the analysis 
of mathematical modeling is to provide a direct view of scheme performance, this scheme is 
supposed to reach a handover latency less than 2 seconds. 
	  
In order to investigate SDN controller communication issue resulted from the previous 
investigation on inter-domain vertical handover proposal, In section 4, the mature cluster 
technology is utilized to propose a multi-domain SDN application based on Floodlight, the basic 
idea, architecture and function flow are designed and introduced in this section. The virtualized 
SDN network is established in virtual machine VirtualBox on host machine to simulate SDN 
network topology, Mininet is utilized to customize topology, and Floodlight controller in Ubuntu 
14.04 is to provide controller governance function in network. Then how to development an 
application in Floodlight is introduced, the program is needed to support Floodlight and has to 
utilize Floodlight dependency module, and the REST API could be adopted to give a better 
display and management tool for developers. Finally, the architecture of this application is 
designed according to Floodlight structure, and the basic scheme of how to discover multi-
domain topology is introduced, which utilized the LLDP protocol discover neighbor topology in 
network via an updated TLV message, and the scheme of how to forward flow table in multi-
domain is proposed by using the publish/subscription modeling.  
 
The implementation of this application has been depicted in Section 5, the experiment is 
done in development on Floodlight controller project in Eclipse inside virtualized environment, 
with a web application inside Tomcat. The Hazelcast module has been integrated into Floodlight 
project, and a web application is shown to manage network from front page manager platform. 
The implementation has been tested in 2 neighbor domains. By updating the generation of TLV 
to make all controllers have same and can recognize unified TLV format, the function of link 
discovery in multi-domain has successfully implemented and shown; additionally, by updating 
static	   flow	   entry	   Pusher module and combining with Hazelcast monitoring as well as 
publish/subscription model, function of forwarding table across multi-domain are successfully 
implemented. Additionally, this SDN management application could be also applied to multi-
domain larger than 2 domains, it is necessary to trade-off the advantages and disadvantages of 
centralized topology and distributed topology to deploy the entire network structure in reality. 
==PAGE:100==

Above all, this project has provide a SDN assisted inter-domain solution for solving 
vertical handover across heterogeneous network of eduroam over 4G/Wi-Fi, and has proposed an 
application based on development of Floodlight module, which is able to use web page manager 
platform to govern network topology and forwarding flow table. Therefore, it is viable to utilize 
SDN technology in inter-domain IP mobility management protocol, which also provides a 
probable promising solution to manage and handle next generation network.

§ Default SDN network in Mininet  
 
The simplest network is created by the command: 
$	  sudo	  mn	  
By the way, in order toclean all remaining configuration and free all components, the following 
command is used once the test is done. 
$	  sudo	  mn	  –c	  	  
This network consists of 1 local Mininet reference controller, 1 OpenFlow switch and 2 hosts, all 
hosts are linked to OF switch, which is controlled the Mininet reference controller. The dashed 
line from switch to controller means there exists one secure control channel using TLS from the 
switch to the controller [14]. This reference controller is created by Mininet by default, as well as 
if no external controller is found. OF switches means Open vSwitch in Mininet.
