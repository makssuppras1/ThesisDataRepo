Summary (English)
The goal of this thesis is to develop a translator that can translate a given
speciﬁcation of a RSL-SAL transition system into a semantically equivalent
nuXmv transition system speciﬁcation.
This report contains analyses and discussions about the considerations and
choices made to deﬁne the translation rules from RSL-SAL to nuXmv. Fur-
thermore, a discussion on the choice of compiler has been done.
This report also contains a description on how the translator has been imple-
mented. Several tests have been performed to verify that the translator transla-
tions was correct. Furthermore, a comparison between the SAL model checker
and the nuXmv model checker has been performed.
Lastly, this report contains a discussion about the future work. This discussion
is about how the data structures that were not considered in this project, could
be translated.

Chapter 1
Introduction
The overall goal of this project is to provide the RAISE Speciﬁcation Language
(RSL) with a functionality to perform model checking.
1.1
Background
Safety critical systems are systems whose failures could result in injuries, loss of
life, property damage or environmental damage. Examples of such systems are
medical devices, aircraft ﬂight controls, weapons, and cars. Such systems can
be speciﬁed and veriﬁed by using formal methods[Kni02][Metb][RAI95]. Formal
methods are mathematical ways of describing and analysing system properties
when developing computer systems. It is a practice that is becoming more and
more important because of our increased dependency of digital systems[Win96].
Formal methods can be used at any stage of the system development, but are
most beneﬁcial in the design phase, since ﬁnding errors in this state of the devel-
opment process is much cheaper than in the test phase[BB]. The value of formal
methods is that they provide the means to examine entire state space systems
of digital designs. Because of the size and complexity of most safety critical
systems, formal methods are primarily applied in critical parts of the systems
to ensure the correctness of the systems or the safety properties[Meta]. There
==PAGE:16==

exists many diﬀerent formal methods e.g. RAISE[RAI92], VDM[ABD+98], and
Spec#[Res]. The RAISE method and its language RSL are build with inspi-
ration from VDM. RAISE has been found quite useful in industrial software
engineering to specify and verify system properties. RAISE has e.g. been used
in the speciﬁcation of the MATRA Transports Automatic train protection sys-
tem. Space Software Italia has used it in the project "Tethered satellite" and
the RobustRailS project uses it to specify and verify the new Danish railway
interlocking systems[Hax10][VHP15].To verify the speciﬁcations it is possible to
use theorem proving or model checking. Theorem proving uses a mathematical
approach for the veriﬁcation. The advantage of theorem proving is that it can
be used on a system of any size. However, theorem proving is cumbersome.
The way model checkers work is that when verifying the speciﬁcation they will
terminate either with an indication that the model satisﬁes the speciﬁcation or
it will give a counter example, which shows why the speciﬁcation is not met.
Model checking has the advantage that it is fully automated, but is challenged
by the occurrence of state space explosions[Cla97][Kot03].
A state space ex-
plosion is when the state space is so big that it is impossible to explore the
entire state space with the amount of resources - such as time and memory -
available[Pel09]. Model checkers are still being researched to make them more
eﬀective. One of the newest model checkers is nuXmv. In 2006 Juan Ignacio
Perna and Chris George, both from UNU-IIST (The United Nations University
- International Institute for Software Technology) wrote the report [PG06], in
which they found that SAL[DMOS03] would be a better tool to model check
RAISE speciﬁcation than SMV would. The strength they found in SAL was
that its syntax is somewhat similar to RSL, SAL provides a whole set of model
checking veriﬁcation techniques, and it provides good quality when produc-
ing counter examples. Since then much has happen to SMV and it has been
developed into nuSMV and later on into nuXmv. At the time of the report
SMV could only model check ﬁnite-state transition systems. Now, in nuXmv,
it is possible to model check both ﬁnite- and inﬁnite-state transition systems.
nuXmv also reduces the state space design by using diﬀerent model transforma-
tion techniques[BCC+14]. With these new improvements it is interesting to see
if nuXmv would be a better tool, than SAL, to perform model checking.
1.2
Project description
The goal of this project is to implement a translator that takes a given RSL-
SAL (henceforth I will use RSL when referring to RSL-SAL) state transition
system speciﬁcation and translates it into a nuXmv speciﬁcation, which then can
be model checked using the nuXmv model checker. The tool-chain consisting
of the translator and the nuXmv model checker should be applied to model

1.3
Development Process
This project consisted of four steps. The ﬁrst step was analysing nuXmv in
terms of syntax and the representation of transition systems. The next step
was the identiﬁcation of the translation rules. The third step consists of the
implementation and testing of the translator. This step was done using agile
development where each iteration added a new translation rule and the transla-
tor was tested, to check if the rule performed the translation correct. The last
step was to perform experiments to compare SAL and nuXmv in terms of time
consumptions and size of the state space.

Chapter 2
Introduction to nuXmv
This chapter gives an introduction to nuXmv and the keywords used in this
thesis.
nuXmv is a language to specify and verify models to be model checked. nuXmv
is an extension to nuSMV. Some of the extensions made is the introduction
of the types real and integer, which enables the user to specify inﬁnite state
transition systems. Furthermore the keyword process has been removed from
nuXmv, it was used to deﬁne that the system was asynchronous. Now nuXmv
is only targeting synchronous systems. In section 3.3.1 I will discuss how it is
possible to make nuXmv asynchronous.
As all other languages the nuXmv language contains keywords which are applied
to describe models.
Below is a description of the keywords that have been
considered in this project. After the description of the each keyword is a small
example that will be extended with the use of the respective keyword.
The
example will be a variable of type boolean, that changes between being true
and false.

There are several compiler construction tools that can be applied for construct-

nuxmv_gen_code.g
The ﬁle nuxmv_gen_code.g contains the functions for generating the nuXmv
syntax from the abstract syntax.
The ﬁle nuxmv.g calls the action nuXmv_Out_Model in nuxmv_gen_code.g

be:
xmvmodel(mod_decl(<RSLFILENAME >,nil))

nuxmv_global_vars.g
The ﬁle nuxmv_global_vars.g contains the global variables used by the trans-

4.6
Emacs Plug-in
The rsltc tool is a plug-in to Emacs[GNU]. Emacs is an extensible, customisable
text editor. The ﬁles rsl-mode.el and rsltc.el have been extended so it is possible
to make the translation from an RSL speciﬁcation into an nuXmv speciﬁcation.
The ﬁle rsl-mode.el is extended with the line:
[" Translate to nuXmv" rsltc -nuxmv t]
which creates the menu option and binds a command to that menu option. The
ﬁle rsltc.el is extended with lines:
(defun rsltc -nuxmv () ""
(interactive)
(rsltc "-nuxmv "))
which deﬁnes a function "rsltc-nuxmv". This function runs rsltc-nuxmv which
is the translation to nuXmv.

Chapter 6
Experiments
This chapter reports on experiments that investigate three issues. The ﬁrst is a
comparison between SAL and nuXvm regarding time and memory consumption.
The second issue is to see if both nuXmv and SAL agrees on the validity of a
given speciﬁcation. The third issue is a comparison of the state space explosion
for both nuXmv and SAL.
6.1
Technical Speciﬁcation
All the experiments were performed on a computer with a 3.2 GHz quad core
processor and 4 GB DDR3 RAM. The model checkers used are nuXmv and SAL.
The amount of resources available for the experiments were 2 GB RAM. If the
experiments were not completed within 24 hours, the experiments were termi-
nated. To measure the time and memory consumption the tool runlim[fFMV]
was used. This is a tool to manage the resources of processes. In nuXmv the
command check_ltlspec was used. The check_ltlspec command uses a symbolic
model checker[BCC+14][CGH94]. The version of SAL model checker that was
used was sal-smc. The sal-smc is a symbolic model checker that uses binary
decision diagrams (BDD)[SRI].

6.4
Conclusion
The experiments shows, when the state space is less than 10,002 nuXmv con-
sumes less time and memory. The intersection of time and memory consumption
of nuXmv and SAL is between 10,002 and 100,002 states. When having 100,002
states nuXmv is slower than SAL and the amount of memory used is nearly 62%
more. However, with the experiments done so far it is not possible to say which
of the two model checkers is the best. To do this the amount and the variety
of experiments must be greatly increased. Both model checkers got the same
result in terms of validity of the speciﬁcations.

7.1.4
Sets, Lists, and Maps
The data structures sets, lists, and maps are widely used in RSL, and they are
important data structures in RSL. In nuXmv sets are prohibited to use except
when used as operands of union and in operations, as the right operand of case
and as the second and third operand of if expressions and assignments[BCC+14].
The data structures list and map are not supported in nuXmv. Therefore, if
sets, lists, and maps should be translated from RSL, then it will be done with a
deep embedding to ensure the semantic of the operations for the data structures.
One challenge that occurs with the translation of these data structures is that
they can dynamically change the size. A solution to this problem, could be to
only accept ﬁxed size data structures. With this assumption either arrays could
be used in the translation or the keys and values could be stored in variables.
A suggestion for the translation of set type expressions can be seen in Table
7.3 on page 80. With this suggestion the set type expression is divided into
two arrays, the ﬁrst array tells which elements are in the set. The other array
contains the values that can be in the set.

Type-set’ is the translation of Type-set, and idIn and idVal is the type id
with In or Val concatenated.

Chapter 8
Conclusion
In this project I have analysed and discussed how it is possible to translate an
RSL speciﬁcation into a nuXmv speciﬁcation. Furthermore, a translator based
on this analysis has been implemented.
The goal of this project was to give RSL speciﬁcations the possibility to be
model checked using nuXmv. This was done by developing a translator, that
could take an RSL speciﬁcation and translate it into a semantically equivalent
nuXmv speciﬁcation. However, it is only possible to translate transition systems,
containing basic transition rules as described in section 3.3.4.
The rsltc tool chain has been extended with a simple translator, which can
translate basic RSL speciﬁcations into a semantically equivalent nuXmv speciﬁ-
cation. I have chosen to use RSL as a front end to write the speciﬁcations. One
of the advantages in RSL is that it is very useful when writing formal methods.
Another advantage is that RSL provides the possibility of doing a step-wise ap-
proach during development. This approach is also used when developing systems
in e.g. java.
In chapter 5 the translator was tested regarding the syntactic output. The re-
sults of the test cases, where the RSL speciﬁcation complied with the translator
supported declaration and expressions, were that the speciﬁcations were trans-
lated correctly. For the test case where the value expression was unknown for
==PAGE:98==
84
Conclusion
the translator, the translation was wrong, but threw an error message as ex-
pected. For the test case where an unsupported declaration was applied to the
RSL speciﬁcation the compiler threw an error because of missing rules in the
ﬁle that generates the abstract syntax tree. This was also expected.
A comparison was made between the two tools nuXmv and SAL. The result
was that nuXmv was faster and required less space when working on state
spaces with a maximum of 10,002 states. If larger state spaces are to be model
checked, the SAL model checker was found to be faster and used less memory.
It was also discovered that nuXmv encountered a state space explosion before
SAL did. However, a clear conclusion can not be made because of the amount of
experiments. Also diﬀerent results might be obtained using other nuXmv model
checking techniques.
Experiments regarding validity of the speciﬁcations were performed. The goal of
these experiments were to compare the two model checkers ability to ﬁnd invalid
speciﬁcations. This was done by limiting a variable in the LTL speciﬁcation.
When the transitions in the transition system were ﬁred, the variable would
exceed the value speciﬁed in the LTL speciﬁcation. All the speciﬁcations were
invalid and both model checkers found this to be true.
In chapter 7 I described what could be done in the future regarding increasing
the subset of the RSL syntax to be translated. Among the considerations were
sets, lists, and maps. The translation of these data structures present some
challenges such as their dynamic size and their operators, but since they are
widely used in RSL it would be very useful to be able to translate them.
In the current state he translator is capable of translating basic RSL speciﬁcation
into a semantic equivalent nuXmv speciﬁcation, and it is possible to model check
these using the nuXmv tool.
