Abstract
Input reduction algorithms help find the cause of bugs in compilers. Such algorithms vary
and have different assumptions about the input they reduce. We implement four input
reduction strategies with a particular focus on binary reduction. We compare them with
two state­of­the­art input reduction tools, Perses and C­Reduce, where they reduce 10 C­
files with a time limit of 30 minutes for each file. We find that when using binary reduction
and C­Reduce separately, they, on average, reduce the bytes in the files by 60.8% and
32.2%, respectively. When combined, they reduce the bytes by 65.7% on average. HDD*
excels at reducing files causing compile­time bugs and reduces the bytes by 99.9% in
such files. Overall, Perses performs best and reduces the files by 72.2% on average.
The numbers show that binary reduction can be used to reduce C­files and improves the
performance of an existing input reduction tool, C­Reduce.

Introduction
If a compiler has a bug, it will fail to translate the source code or produce incorrect machine
code. It can be a tedious process to search for the bug, especially if the source code has
thousands of lines. The problem can be mitigated by reducing the amount of code, which
is a process that can be automated. An input reduction algorithm reduces the size of the
code while keeping the behavior that caused the crash. Input reduction tools are powerful
tools that help compiler developers find bugs in compilers and use such algorithms.
In this thesis, we will develop a tool capable of using three input reduction algorithms
to reduce C­files. The performances of the algorithms we intend to implement are yet
to be measured when reducing C­files. In particular, we will focus on binary reduction,
which is an algorithm used by JReduce to reduce Java bytecode. We will compare our
tool with two state­of­the­art input reduction tools, Perses and C­Reduce, by having the
tools reduce 10 C­files, which older versions of the Clang compiler cannot successfully
compile.
1.1
Structure of the Thesis
We will present the structure of the thesis. After this introduction, we will:
• present three input reduction algorithms, two input reduction tools, and one program
debloating tool in chapter 2.
• explain how the three input reduction algorithms works in chapter 3. We will also
give examples of their reduction process and analyze their worst­case running time.
• describe how we have implemented the three algorithms in chapter 4, where we
present the reasoning behind their features.
• have the implemented algorithms reduce 10 C­files in chapter 5, and present how
they performed.
• discuss how the algorithms performed in chapter 6, and present features that our
implementation of binary reduction lacks.
• conclude and summarize the performances of the tools in chapter 7.

Related Work
In this chapter, we introduce three input reduction algorithms, two state­of­the­art input
reduction tools, and one program debloating tool.
2.1
Delta Debugging
The delta debugging algorithm, proposed by Andreas Zeller and Ralf Hildebrandt [1],
partitions a file into chunks and tries to reduce the number of chunks greedily. A chunk
can be, e.g., a byte, line, or character, which means that the algorithm can reduce any
type of file. Delta debugging reduces a list of chunks by deleting elements from the list
and checking whether the given criteria still holds. We will explain more in­depth how the
algorithm works in section 3.2.
2.2
Hierarchical Delta Debugging
The hierarchical delta debugging algorithm, proposed by Ghassan Misherghi and Zhen­
dong Su [2], reduces an abstract syntax tree by reducing every level in the tree using delta
debugging. Therefore, the algorithm cannot reduce any file but requires a tree structure.
We will explain the algorithm more in­depth in section 3.3.
2.3
Binary Reduction
The binary reduction algorithm is proposed by Christian Kalhauge, and Jens Palberg [3].
The algorithm partitions a file into chunks and uses binary search to identify which chunks
are required to satisfy the criteria. Kalhauge and Palberg extend the algorithm to reduce
a dependency graph to avoid invalid files, e.g., where a call is made to a non­existing
function. Kalhauge and Palberg further extend the algorithm, called generalized binary
reduction [4], using propositional logic to define each chunk and its dependencies in more
detail. They find that generalized binary reduction is effective when reducing Java byte­
code. We will elaborate in section 3.4.
2.4
C­Reduce
C­Reduce [5], proposed by Regehr et al., is an input reduction tool written in C++ that
specializes in reducing C and C++ files, although it can also reduce JavaScript and Rust
files. The tool reduces a file by repeatedly transforming and testing the program to find
smaller versions of the file that satisfy the criteria. Transforming a program means creating
a variant of the program that does the same. C­Reduce transforms a file by, for example,
removing unused variables and inlining functions and variables. Inlining a function means
Binary Reduction of C Programs
3
==PAGE:12==
replacing all calls to the function with its content, allowing the function to be removed. C­
Reduce can spend many hours reducing C­files but achieves outstanding results in terms
of the size of the reduced file.
2.5
Perses
Perses [6], proposed by Sun et al., is a language­agnostic input reduction tool written in
Java. It can reduce any language provided it has access to the Backus­Naur Form (BNF)
of the language. It converts the BNF into Perses Normal Form (PNF), which it can reduce
into a smaller, syntactically valid subset. PNF has four types of nodes, each of which is
reduced in its own way. Perses has a priority queue with PNF nodes, where a node with
more tokens has higher priority. Perses then retrieves and tries to reduce the highest
priority node. Although Perses is typically not capable of reducing C­files as much as
C­Reduce, it has the advantage of being able to reduce the files of any language.
2.6
Chisel
Heo et al. propose Chisel [7], which is a program debloater and has a different goal
than the two previously mentioned tools. Their tool uses reinforcement learning to im­
prove performance, decrease security vulnerabilities, and generally reduce the program
complexity while maintaining readability. Using reinforcement learning, Chisel builds a
statistical model to determine which transformations are most likely to satisfy the given
criteria.

3
Input Reduction Algorithms
In this chapter, we introduce three input reduction algorithms, give examples of how they
reduce an input and analyze their worse­case running time.
Given an input to a program that causes some behavior, an input reduction algorithm
reduces the input size while keeping the behavior. These algorithms are practical when
trying to fix bugs in compilers. If we try to compile some source code consisting of thou­
sands of lines of code and the compiler crashes, it can be challenging to find and correct
the bug. Using an input reduction algorithm, we can reduce the number of lines while
keeping the behavior that causes the crash, effectively making it easier to identify and
correct the error in the compiler. In this chapter, we will introduce three input reduction
algorithms.
3.1
Defining Expressions and Notations
In the coming sections, we use expressions and notations that we will define here.
Test case
A subset of the input.
Test function
Given a test case, A, then test(A) returns true if the wanted crash
occurs and false otherwise. When returning true, we say that the
test succeeds or passes and fails otherwise.
Unit relevance
Given a test case consisting of units, we say that a unit, u, is relevant
if we know that u is required for the test to succeed.
1­minimality
A test case is 1­minimal if it passes the test, but removing any one
part causes the test to fail.
Local minimum
A 1­minimal test case may be a local minimum, meaning that better
solutions exist.

3.2
Minimizing Delta Debugging
Andreas Zeller and Ralf Hildebrandt describe the minimizing delta debugging algorithm
(ddmin) in their paper Simplifying and Isolating Failure­Inducing Input [1]. The ddmin
algorithm reduces a set of units and finds a 1­minimal test case. A unit can be anything
that makes sense regarding the input, for example, a single character, a line of text, or a
node in a tree.
The ddmin algorithm assumes that test(∅) fails and that test(input) succeeds. It starts
by partitioning the set of units, U, into two subsets, ∆1 and ∆2, and tests each subset.
If a subset passes the test, ddmin continues the search in that subset. However, if both
subsets fail the test, we can either increase or decrease the size of each subset. If we
increase the size, it is more likely that the test will succeed on a subset, but we will progress
slower towards a 1­minimal test case. On the other hand, if we decrease the size, it is
less likely that the test will succeed on a subset, but we will progress faster towards a 1­
minimal test case. The ddmin algorithm combines both approaches by first testing each
subset, ∆i, and then testing each complement, ∇i = U −∆i. Besides U, ddmin requires
a granularity, n, which is the number of subsets U is partitioned into. If no subset or
complement succeeds the test, ddmin is called recursively with an increased granularity.
The algorithm is described in equation 3.1, which is initially called with n = 2.

3.3
Hierarchical Delta Debugging
Ghassan Misherghi and Zhendong Su describe the hierarchical delta debugging algo­
rithm (HDD) in their paper HDD: Hierarchical Delta Debugging [2]. While ddmin needs
no understanding of the structure of the input, HDD requires that the input has a hierar­

each level of the AST, starting with level 0 containing only the root node. HDD will then
increment the level and reduce the nodes in that level using ddmin. Algorithm 1 shows
the pseudocode of HDD.

3.4
Binary Reduction
Christian Kalhauge and Jens Palberg describe the binary reduction (BR) algorithm in their
paper Binary Reduction of Dependency Graphs [3]. BR uses binary search to figure out
which units must be included in the final test case.
Algorithm 2 shows the pseudocode of BR. The algorithm maintains two sets, D and S. The
former set contains the units to be searched and is initialized to be the input, and the latter
contains units required for the test to succeed and is initialized to be ∅. Besides the input,
BR requires a cost function, C, which is used to sort D. BR calls FindSmallestPrefix,
which uses binary search to find the smallest prefix of D that together with S succeeds the
test. The auxiliary function returns the index of the prefix, r, so that Dr can be removed
from D and added to S. The Sort function sorts D using S and C. It needs S because

Complexity Analysis
BR uses binary search, which runs worst case O(log n). The worst­case scenario is when
BR reduces n relevant nodes, meaning that all nodes must be included in the final re­
sult. In such a scenario, we will use binary search n times, giving us a running time of
O(n log n). However, each time we use binary search, we also use the cost function, C,
which has a running time of $C. The running time of BR is O(n log n + n · $C).
3.5
Generalized Binary Reduction
Christian Kalhauge and Jens Palberg extend BR in their paper Logical Bytecode Reduc­
tion [4], where they describe the generalized binary reduction (GBR) algorithm. GBR
uses propositional logic to define the dependencies more in detail. In their paper, they
define four types of variables in Java, where a variable is denoted using square brack­
ets. The types are classes, methods, method code, and implementation. In the example
Java­program in figure 3.6 we find class C that contains method C.mi() and implements
I. These correspond to variables [C], [C.mi()], [C.mi()!code], and [C ◁I]. We use
x()!code to refer to the code belonging to method x and z ◁y denotes that class z imple­
ments interface y.
We have three types of dependencies where we use [x] ⇒[y] to denote that [x] de­
pends on [y]. First, we have syntactic dependencies which is the relation between a
parent and a child in the AST of the program. Such dependencies are: [C.mi()] ⇒[C]
and [C.mi()!code] ⇒[C.mi()]. Second, we have referential semantic dependencies

4
Implementation
In this chapter, we will describe the tool we have implemented in C++ capable of reducing
C­programs. We have implemented four algorithms: ddmin, HDD*, binary reduction, and
multi­layered binary reduction inspired by the generalized binary reduction algorithm. We
have implemented the algorithms to reduce the nodes in the abstract syntax tree (AST).
As mentioned in chapter 3, the algorithms use a test function to confirm whether or not the
wanted crash occurs. In our implementation, the test function is a shell script that returns
0 if the wanted crash occurs and 1 otherwise. Besides a file to reduce, our tool requires
such a shell script to confirm that the wanted behavior still exists in the C­file. To see an
example of what such a shell script may look like, see appendix C.
4.1
Third Party Libraries
Constructing an AST requires a C­parser, and writing a parser is time­consuming. Since
our goal is to develop the program reduction tool, we used Tree­Sitter, a parser gener­
ator tool. Given a JavaScript file describing the grammar of a language, Tree­Sitter will
generate a C­program capable of parsing that language. Luckily, third­party libraries that
describe the grammar of C and many other languages already exist and are free to use.
Tree­Sitter­C is one of such libraries that, as the name suggests, describes the grammar
of C and is meant to be used with Tree­Sitter.
4.2
Minimizing Delta Debugging
The ddmin algorithm reduces a list of units. The challenge of defining a unit is balancing
the time it takes for ddmin to finish and the amount ddmin can reduce a file. If we define a
unit as a single character, ddmin might reduce the given file by much more than defining
a unit as a line of code. However, it will also take a considerably longer time to reduce the
file. As mentioned in section 3, ddmin runs in quadratic time, which means that even a few
units may take a long time to reduce. Therefore, a unit cannot be a character as they occur
abundantly, even in small C­files of a few hundred lines of code. In our implementation of
ddmin, we choose to define a unit as a function for the following reasons:
• ddmin will be able to reduce large C­files in a reasonable amount of time
• by reducing functions ddmin can remove all types of nodes excepts global declara­
tions. That is not necessarily true about other types of nodes.
• unless everything is is the main function we are sure that the file will contain func­
tions.

As explained in section 3.2, ddmin assumes test(∅) fails, but that may not always be
the case. For example, there could be a bug­triggering statement outside a function,
allowing us to remove all functions. Therefore, we do not assume that test(∅) fails, and
our implementation may remove all given units.
We have implemented ddmin as seen in algorithm 3.1. The algorithm works inefficiently
in that it typically tests the same combination of units multiple times, as the example in
table 3.3 also shows. For example, iterations 6, 11, 15, and 24 test duplicate test cases.
In total, 12 of the 28 iterations are duplicates. We will present two possible solutions to
the problem.
Solution Proposal ­ Perfect Hash Function
Each unit will either be included in a test case or not. We implement a perfect hash
function that, given a list of units, U, returns |U| bits, where bit n is 1 if unit Un is included
in the test case. Before calling the test function, we hash the current combination of units
and see if we have stored a result. If there is not, we will call the function.
Solution Proposal ­ Hash Function With Collisions
A perfect hash function might be practical when testing many different combinations. How­
ever, ddmin reaches a solution quickly. The example in table 3.3 reduces eight units
meaning there are 28 = 256 combinations, but it finds a solution in just 28 iterations.
Therefore, using a perfect hash function might be excessive. Instead, we propose a hash
function that returns a 32­bit integer. Getting a list of units, U, we iterate them using
1 ≤i ≤|U| and setting bit i to be 1 if the unit is tested and 0 otherwise. To keep the 32­bit
integer from overflowing, we will modulo the hash with 231 −1, which is the largest prime
less than 231, after each iteration.

4.3
Hierarchical Delta Debugging
As mentioned in section 3.3, HDD does not necessarily reach a 1­minimal test case.
Therefore, we choose to extend the implementation to HDD*, which uses the already­

implemented ddmin.
The running time of HDD* is O(n3), which is problematic when
reducing a large AST. We present two ways to optimize by reducing the nodes in the
AST.
Solution Proposal ­ Collapse 1­Child Branches
If HDD* cannot remove a branch with one child, it cannot remove the child either. Never­
theless, HDD will test them on separate calls to ddmin. Before HDD* reduces the AST, we
will collapse such branches meaning that we keep only the child. The procedure is done
recursively, meaning if we have a branch with one branch­child which itself has one child,
then we will collapse both branches, keeping only the child. We refer to this proposal as
collapse.
Solution Proposal ­ Merge Semicolon Leaves
Every semicolon counts as one node but will unlikely be a relevant node alone. Since there
are many semicolons in a C­program, we may gain performance by reducing the number
of nodes by one for each semicolon. We will traverse the AST, and upon encountering a
semicolon­leaf, we will remove it and extend the value of the previously encountered leaf
with a semicolon. We refer to this proposal as merge.
Solution Proposal ­ Combination
Proposal collapse and merge are not incompatible and HDD* might work better when
using both. When applying merge, some branches will end up having one leaf child,
allowing collapse to delete more branch­nodes. We refer to this proposal as both.
Solution Choice
To compare the proposals, we cannot use the same C files that we generated in section
4.2 because most functions are empty. HDD* will be able to remove the non­relevant
empty functions with one ddmin call. To make HDD* reduce more levels, we extend our
C­file generator to include statements, declarations, and calls to other functions. To read
more about the extension, see appendix B.

4.4
Binary Reduction
Our implementation of BR will reduce the number of functions in a C­program. We define
that a function A depends on another function B if A calls B. To measure the impact
of using a dependency graph, we make a version that reduces a list of functions and a
version that reduces a dependency graph or a list of sets of functions. We create the
dependency graph by having each function maintain a list of identifiers of the functions it
calls. We create a set, S, containing the subsets. For each function, we create a new set
containing the function and add its dependencies recursively. If the new set is not already
in S then we will add it. Our cost function is the number of elements in a set, C(x) = |x|.

6
Discussion
In this chapter, we start by discussing the results acquired in chapter 5. We will also
discuss what our tests lacked and end the chapter by suggesting improvements to our
implementation of multi­layered BR, which we refer to as just BR in this chapter.
6.1
Results
In chapter 5, we saw that BR is effective when reducing C­files. Our implementation
reduces the files quickly but cannot reduce many types of nodes since we have not im­
plemented them. We mitigate the problem by combining BR with C­Reduce, which can
reduce many node types. The performance of the algorithms depends on where the bug
lies. If the compiler crashes while compiling, HDD reduces the bytes in the file by 99.9%
because it does not consider dependencies. However, when reducing C­files causing
wrong code generation or run time bugs, it improves the performance to consider depen­
dencies. When finding code generation bugs, combining BR and C­Reduce is efficient.
We had only one file causing a run­time bug, which is a low sample size and makes it
difficult to conclude anything, but Perses reduced the file by 99.9%.
6.2
Results Shortcomings
We allowed the tools to reduce each C­file for no more than 30 minutes. While some
of the algorithms could finish reducing the files within the given time, others were forced
to stop. It would be interesting to see how each tool performs without a time limit, but
acquiring such results is time­consuming, as some of the C­files are large. We saw that
C­Reduce performed poorly when given a short amount of time. If we were to allow the
tools to reduce for a longer time, we expect to see the performance of C­Reduce increase
dramatically, surpassed only by combining BR with C­Reduce.
6.3
Further Improvements
Our implementation of BR can be improved in several ways. We present two ideas for
extending the implementation to reduce C­files by more.
6.3.1
Defining Additional Variables and Dependencies
BR reduces the nodes in an AST based on which variables are defined. Defining more
variables would result in BR being able to reduce them. So far, we have defined the
following seven variables: function definition, function code, function call, if statement,
for loop, variable declaration, and variable assignment. Many types of nodes have not
been implemented, for example: while loop, do­while loop, struct, enum, and logical and
arithmetic calculations. Figure 6.1 shows a valid program that BR cannot reduce at all.

6.3.2
Inlining
BR reduces the AST by removing a node and its children. Removing function definitions,
variable declarations, and variable assignments requires BR to remove all their usages.
If it is impossible to remove the usages, BR cannot reduce them. BR could reduce the
input even further by inlining these types of nodes. The algorithm can only remove nodes,
and implementing inlining requires extending the algorithm.
For example, look at the program in figure 6.2 that calculates a value and crashes while
trying to print the value. BR cannot reduce the program any further. Removing the two
functions requires removing the variable, which is impossible as the bug­triggering print
statement uses it. By implementing function inlining, we can reduce the program to the
program in figure 6.3 and implementing variable inlining results in the program in figure
6.4. BR can achieve an even smaller result by combining both types of inlining. Inlining
may also allow BR to use the already implemented features to reduce more.

7
Conclusion
In this thesis, we implemented and evaluated four input reduction algorithms, ddmin,
HDD*, binary reduction, and multi­layered binary reduction. They are all capable of re­
ducing a C­file while keeping a specified behaviour. We compared the algorithms with
two state­of­the­art input reduction tools, Perses and C­Reduce. Each tool reduced ten
C­files that cause bugs in older versions of the Clang compiler and were given no more
than 30 minutes to reduce each file. We saw that it is inefficient to reduce only one type
of node in the abstract syntax tree, as ddmin and our first version of binary reduction do.
Overall, they reduce the number of bytes by 44.7% and 49.0%, respectively. Of all the
compared file reduction strategies, we saw that Perses performs best and, on average,
reduces the bytes by 72.7%. C­Reduce did not perform well with the given time limit and
reduced the bytes only by 32.2%, and multi­layered binary reduction reduces the bytes by
60.8%. When combining multi­layered binary reduction with C­Reduce they reduce the
bytes by 65.7%.
A C­file can cause three types of bugs in a compiler, and the performance of the tools
varies depending on the type of bug the file causes. When reducing a file that causes a
compile­time bug, HDD* performs best because it ignores dependencies between, e.g.,
declarations and usages of the declaration. Overall, HDD* reduces the bytes by 59.0%,
but if the file causes a compile­time error, HDD* reduces the bytes by 99.9%. However,
when reducing files causing code generation bugs or run time bugs, HDD* managed to
reduce the bytes only by 2.8% and 17.7%, respectively. If the file causes one of the two
last­mentioned bugs, it is advantageous to consider dependencies. If the file causes a
code generation bug, then combining multi­layered binary reduction and C­Reduce per­
forms best and reduces the bytes by 29.3%. We had only one file causing a run time bug,
but in that category of bugs, Perses performs best and reduces it by 99.9%.
The numbers show that multi­layered binary reduction is capable of reducing C­files ef­
fectively and improves the performance of C­Reduce when they are combined.

In section 4.2 we tested and compared features of the ddmin implementation by writing

called anywhere and contain no statements except the relevant ones, which are called
in the main function and have one statement that prints ’a’. For each C­file, the tool also
generates a test.bat file that returns 0 if file Cij prints ’a’ Rj times and 1 otherwise. The
extension .bat is for scripting files on Windows. To automate running an input reduction
tool on all n generated files, we also generate a run.bat file that runs all test files.
