Summary
This thesis argues about the usability of the Blockchain technology as a database, its
fault tolerance capabilities and why it is a good choice when high availability and
resilience are required. An analysis of both the currently available academic solutions
and the commercially available tools is conducted. It is concluded that there is no
available solution that employs together the concepts of Blockchain, Microservices
and containers in designing the data store layer.
The problem of achieving optimal fault tolerance of the database is solved by design-
ing a new system. The new design employs a Microservice-oriented architecture and
containerization, for rapid self-healing.
Based on the proposed design, a proof-of-
concept solution is implemented.
In order to prove the high availability of the system, multiple stress tests of faulty
scenarios are conducted, along with essential read/write performance tests. Finally,
drawbacks and flaws of the prototype are discussed, and despite its risks, it is con-
cluded that the system still stands for high fault tolerance of the data store layer.

CHAPTER 2
Introduction
The modern web systems are typically composed of three main entities: data source,
backend/server app, and a data store.
Information is obviously sent from the data source and received at the backend, which
then works with that data and propagates it to the database for persistent storage.
When it comes to fault tolerance, this system design has two major flaws: firstly,
the backend can be a bottleneck or a single point of failure (SPOF) in regards to
accepting and storing data. Secondly, a centralized database can also be a SPOF,
since backups and replications are high maintenance. This is where the Blockchain
comes in place.
The goal of this project is to analyze how the data produced and sent out from
eSmiley A/S sensor devices can be stored in the Blockchain and kept its integrity.
The main part is to find an optimal way of implementing a database layer on top
of Blockchain, with a Microservice-oriented architecture, and how the produced data
will be handled and utilized when stored in the Blockchain. The idea is to utilize the
Blockchain as a data store, directly accessed from the sensor devices, where the nodes
of the data store are handled in a distributed fashion. The project will analyze the
pros and cons of this approach, decide whether it is a viable solution, find the best
possible implementation and discuss the benefits and drawbacks of it.
2.1
Problem
The interesting part of the topic is - what if the server goes down? What if just a
part of the application as small as the API that receives the data breaks, even if the
rest is intact? Then the whole process is broken because our system simply cannot
accept data.
In these cases the backend is a SPOF. It is still not that crucial when the data is
generated from human users - they can repeat the process when the backend is up
and running again - the information will still be handled. But when it comes to the
IoT - if the devices do not have some sort of caching and ACK mechanism - the data
is being sent out once and forgotten. This leads to loss of viable system information.
Microservice-oriented architectures and horizontal scaling mitigate this problem and
the system becomes a lot more reliable. What about the data store?
==PAGE:16==
4
2 Introduction
Here comes the other point of concern: how do we make the data store be as re-
liable and resilient as a backend build with a SOA. The usual approaches are to make
use of database replication, sharding or other concepts for data backup and fault
tolerance of the database. Nevertheless, this brings an overhead of complexity in
regards to configuration of the systems, continuous maintenance and monitoring.
Another, smaller concern is the data mutability.
This project is done by the use
case of a sensor platform of eSmiley A/S. In the company’s interests we do not want
the data received from the sensors to be mutable. And the typical databases allow
data editing. This, of course, can be restricted by permissions and admin rights, but
that can always be jumped over.
The Blockchain technology is a distributed database by design, and distributed sys-
tems characterize with high availability. Furthermore, its nature enforces immutabil-
ity of the data records.
According to this short discussion, the problem of the thesis can be summarized
with the following questions:
• Can the Blockchain technology be used as a persistent data store?
• Does the Blockchain provide better robustness than a centralized database?
• Do we achieve higher fault tolerance by employing a Microservice-oriented ar-
chitecture and use of containers for the data store layer?
The thesis problem is to be researched in the context of a private business software
system - the sensor platform of eSmiley A/S.
2.2
Scope and Limitations
The project aims to analyze the feasibility, pros and cons of utilizing the Blockchain
as a persistent data store, and to find a system design solution in which the database
layer can achieve optimal fault tolerance. The scope extends to implementing a work-
ing system as a proof of concept.
The following constraints and assumption must be taken into consideration:
• The project does not focus on researching best practices, solutions or implemen-
tations regarding data transfer protocols for IoT devices.
• It is assumed that the data sources (the IoT sensors) have uninterrupted con-
nection to the Internet.

CHAPTER 3
Theory
3.1
Blockchain
When talking about a new technology, in order to understand what it does and how
it does it, one should first understand the problem that this technology tries to solve.
The purpose of a Blockchain is to enable a database or ledger to be directly and
safely shared by non-trusting parties. Centralized databases, on the contrary, are
stored and controlled by a single entity. A Blockchain has multiple “nodes”, each
of which maintains a copy of the database state. Each node could be controlled by
a different entity. The nodes connect to each other in peer-to-peer fashion, using a
“gossip protocol” in which each node is constantly sharing its knowledge. Because
of that, any node can rapidly broadcast a message to the entire network via many
alternative paths.
Both centralized and Blockchain-powered databases begin in an empty state, and
are progressively updated via “transactions”. A transaction is a set of changes to a
record (row) in the database and they are atomic, meaning that they succeed or fail
as a whole. This means that the database must ensure that no two transactions affect
the same row simultaneously.
In the Blockchain, each node processes the transactions independently on its own
copy of the database. These transactions can be created anywhere on the network
and are propagated to all nodes automatically. Since the entities running the nodes
can have different interests, they cannot trust each other’s transactions. Therefore,
Blockchains employ rules for validating transactions. This is where the Blockchain
term ”trustless ledger” comes from. [20]
3.1.1
Transactions (tx)
As determined already, the database consists of records (rows). A tx can do three
things: delete, create and modify/update rows. These can be narrowed down to two,
since updating a row is actually deleting it and creating a new one with updated
data. In Blockchain’s terminology, the rows deleted by a tx are defined as ”inputs”
and those created as ”outputs”. Any row deleted by a tx must have been created
by a previous tx. Thus, each tx input consumes the output of a previous tx. The
==PAGE:20==
8
3 Theory
current/latest state of the database is defined by the set of unconsumed outputs.
In a Blockchain, a tx is valid if it fulfills the following three conditions:
Correctness. The tx must be a legitimate transformation from an input to an out-
put. For example, in a financial ledger, the total quantity of funds in the inputs
must match the total in the outputs, to prevent money from magically appearing or
disappearing.
Authorization.
The tx must be authorized by the owner of every output that
is being consumed by the tx. In a financial ledger, this prevents participants from
spending each other’s money without permission. Asymmetric (public-private key)
cryptography is used to handle the tx authorization. Owners are identified by public
keys and have private keys kept secret. Every row has an owner and each tx must be
digitally signed by the owner of each of its inputs.
Figure 3.1: Authorization of transactions in Blockchain [42].
Uniqueness. No output can be consumed by two different transactions. In a
financial ledger, this is how entities are prevented from making conflicting/duplicate
payments. While the tx of each of multiple entities could be correct and authorized,
the uniqueness rule ensures that only one will be processed by the database.
In a conventional Blockchain, every node checks every transaction in terms of these
three rules.
It is important to mention two additional things. Firstly, even though a financial
ledger was used for giving examples, the input-output paradigm can be used for
broader variety of use cases. Eg. each row can contain a data object (JSON) with
different types of information within. This changes nothing about the three valida-
tion rules.
Secondly, some Blockchain databases employ the logic of only creating and never
deleting rows. These are data-only Blockchains, which also provide big array of use
==PAGE:21==

9
cases, especially in the commercial and enterprise areas. In these Blockchains, the
transactions add data in their outputs but do not consume any inputs. This allows
for simplification of the validation rules. [20]
3.1.2
Blocks
A Blockchain is, well... a chain of blocks. Each of these is linked to the previous one
by a ”hash”. A hash is the unique output of a hash function which has been fed with
some data (payload). Hashes cannot be reversed back into readable information. The
hash function will always yield the same hash for the same data. For example the
string ”Blockchains are awesome”, hashed with the SHA256 algorithm, will always
yield:
F7BC6C614825DDE983795EB7A39ADD05C8BEA09285562AB4FB0467A4CB553D27
But ”Blockchains are awesome.” (notice the additional dot) yields:
15B4F08F900BA5C0342C0AC995D749BE8CCEE15FAEC5E111140C449A5BA41103
Each block in the chain consists of an ordered set of transactions. These transactions
must not conflict with the other transactions in the current block or the previous
blocks. Just like a tx, a block propagates rapidly through the network to the other
nodes. Once the allowed memory size of the block is filled by transactions it is sealed.
Once a tx appears in a block, it is ”confirmed”, which makes the other nodes reject
any conflicting transactions.
Figure 3.2: Chaining of blocks in a Blockchain network [42].
How do we ensure that the nodes agree on the state of the authoritative chain
and who creates these blocks? This is the job of the Consensus Algorithm (CA) 1.
There are multiple variants of CAs, but here are some of the most popular ones: PoW
(Proof of Work), PBFT (Practical Byzantine Fault Tolerance) and DPoS (Delegated
Proof of Stake).
These are mostly used for non-permissioned public Blockchains.
Permissioned Blockchains for enterprises use a variation of a voting scheme where
votes are provided by ”validator nodes” which are responsible all together to come to
a consensus on the Blockchain state. This still ensures that confirmed transactions
cannot be reversed.

3 Theory
3.2
Microservices
Since the beginning of large software applications, the main architectural model has
been the Monolith - the whole application bundled together. Nevertheless, in the
past years a relatively new style of architecture has emerged and established itself -
the Microservices architecture [43].
It is a style of building software where the business logic is broken down into sep-
arate, stand-alone applications (known as services), preferably according to their
domain and purpose. The Microservices approach stands out in the fact that the ap-
plications are independent of each other, running on their own processes and can be
deployed separately, where the new version of one service does not affect negatively
(does not stagger or break down) the functionality of the other services in the system.
The microservices each have individual development lifecycle.
Nevertheless, they
communicate with each other over a network and depend on each-other’s functional-
ity [44]. In the overall picture, the combined network of microservices comprise one
big application.
Figure 3.3: Monolith vs. Microservices architecture model [37].
3.2.1
Why and how?
When talking about the Web aspect of software development, the Monolithic style of
architecture, as already mentioned, was the de-facto way of building software. It was
==PAGE:23==
3.2 Microservices
11
a good practice, until the Cloud became the new way of hosting and supporting Web
applications. The scaling and updating Monolithic type of applications became a bit
problematic. The issue arises when multiple instances of the application are running
in parallel and a new change has to be introduced/deployed. In that case the whole
application had to be upgraded, in all instances, no matter whether the change is big
or small.
The Microservice architecture overcomes this problem with ease.
An entity that
needs changes can be upgraded separately, independently of the other services, this
way leaving them unaffected. This has two benefits:
• The deployment of changes becomes much simpler, with a lot shorter change
cycle time
• The scaling of a particular service is simplified and easier to manage
In general, there are no strict rules about how Microservice applications should
designed and architectured. It is generally agreed in the industry that the good Mi-
croservice architecture is comprised of several small components that interact with
each other independently and without failing.
There are numerous other advantages and disadvantages in the Microservice archi-
tecture style compared to the Monolithic one, when talking about Web applications.
Some of them are:
• Different services can be written in different programming languages and with
different technologies, in case one tech proves to be better for the particular
domain the service is going to be utilized for. All that the services need to
know about each other, in order to work together, is the input and output of
each-other’s endpoints. That is achieved by using a common data representation
notation, eg. JSON.
• Monolith applications scaling is only efficient when done horizontally. When
scaling vertically, a Monolith app will needlessly use resources for parts of the
application that do not require scaling at that particular case. On the other
hand, a Microservice will utilize vertical scaling much more efficiently, since it
functions for a narrowed-down domain.
• The microservices approach introduces additional complexity for the developers
to create a distributed system. Some of the main challenges are testing (specifi-
cally integration), maintainability and designing an inter-service communication
mechanism.
• Microservices are utilizing a very decentralized topology. This means that a
change in the code-base has a much smaller chance to break a different domain
part of the overall system, since it is independent of the other services. In a
Monolith, the data is shared in the single running application.

3.3
Containers
Containers serve the purpose of packaging software into standardized units for devel-
opment, continuous integration and deployment. They are characterized with being
lightweight, stand-alone and executable packages of software. A container includes
everything that is needed to run the application: code, runtime environment, sys-
tem tools and libraries and other settings. One of the beauties of containerization
is that it is environment independent. This means that it will run the same on all
major operating systems like Windows, Linux and OS X. A developer writes the code,
packs into a container image and shares it with others. Development collaboration
has never been easier! [14]
Figure 3.4: Example of a basic structure of Docker containers [14].
As shown in figure 3.4, the containers isolate different applications. This particu-

on the specific use case of the applications. It is definitely not wrong to combine a
couple or set of applications within the same container.

3.3.1
Containers and Virtual Machines
Virtual Machines (VM) have been used for virtualization, sharing and deploying
software for years. They share some characteristics with containers, but there are
still great differences between the two. Here are some of them:
• VMs contain complete OS. They are resource intensive and could take up several
GBs, depending on the guest OS.
• VMs use hypervisors to talk to the hardware, whereas containers share the
kernel of the host OS. The VMs have their own kernel, which makes them
deeply isolated from the host OS.
• VMs running on the same server can run different OS, while containers are
bound to the host OS.
• Containers virtualize the underlying OS, whereas VMs virtualize the underlying
hardware.
So, with these differences on the table, what are the advantages of using contain-
ers?
• the average container is within the range of tens, rarely hundreds of MB in size,
which makes them extremely lightweight
• much less resource intensive than VMs
• building and executing a container happens in the matter of seconds, which
makes them ideal for scaling
• makes development within teams much, much easier since whole environments
are being shared easily
• testing and bug-tracking also becomes easier since there is no difference between
running the app within the container locally or on a test or production server
• they are the best option for Microservices oriented architectures and continuous
integration and development
Of course, there are a few disadvantages:
• Security. Since containers share the kernel of the host OS, alongside having
root access, they are less isolated from the host and the host can be easier to
manipulate negatively. Nevertheless, proper setup and administration mitigates
this risk significantly.
• They provide less flexibility in operating systems. In order to provide a different
OS environment for the containers, one needs to run them on a different machine,
with the specific different OS. This is rarely a problem, especially when using a
hosting provider. It could be a bottleneck for a large enterprise application.

3.3.2
Summing up the theory
Quick reference of Findings and Conclusions:
• Blockchain technology is essentially a distributed, append-only database
• Blockchain can be used both as a public or a private ledger
• The records (or rows) in Blockchain are called transactions (tx) which are
chained one after another through hashing and encrypting with the users’ public-
private key pairs
• The transactions in Blockchain are combined and sealed in so-called blocks
• The blocks in Blockchain are chained together through hashing and nonce. Once
a block is created, all its transactions are confirmed as the truthful state of the
database
• Microservices are small applications that target a specific domain of the overall
system. Together, a set of microservices form a complete system
• The Microservice-oriented architecture provides higher fault tolerance than a
Monolith, but also higher effort on CI/CD
• Containers are lightweight, stand-alone, executable packages of software
• Containers are used to encapsulate applications, along with all the needed con-
figuration and environment, including a minimized host OS
• Containers make rapid CI/CD and application self-healing possible
• Containers need to be executed within another host system, and are environ-
ment independent

failed service will continue to work and provide availability.

5.1
CAP theorem
The Blockchain is a distributed data store, therefore it falls under the CAP theorem
[56]. The theorem identifies three system properties for any distributed data store:
• Consistency - every read receives the most recent write or an error
• Availability - every request receives a positive response, without guarantee that
it is the latest write
• Partition tolerance - the system continues to operate regardless of the loss of
an arbitrary number of messages between the nodes
In essence the theorem states that a distributed database can only provide two out
of the three properties. Since Partitioning is an inherent property of any distributed
system, for the Blockchain we are left to choose between Consistency and Availability.
This suggests that the Blockchain is either a CP (Consistency + Partitioning) or a
AP (Availability + Partitioning) compliant system.
==PAGE:46==
34
5 Discussion
If we choose Availability over Consistency, any requests to the database do not guar-
antee to respond with the latest record. Even though there is a response from the
data store at all times, in this scenario the characteristic of the Blockchain being
always truthful is defied. On the other hand, since Blockchain is widely used for
monetary systems, Consistency is not a property we can compromise on. It takes
priority in the Blockchain technology.
If we prioritize Consistency and leave Availability as optional, then in case of network
unavailability (Partition happens) the Consensus Algorithm will be disrupted. But
the consensus is a key feature in Blockchain. Therefore, we cannot compromise on
Availability. [38][17]
We cannot compromise on either of CP or AP. This does not mean that the Blockchain
technology violates the CAP theorem. It means that Blockchain opts out for AP +
Strong/Eventual Consistency. [58] This can partially be explained as the fact that
after a transaction (the write) has appeared in an X number of blocks in the chain, it
becomes consistent. ”X” depends on the design and implementation of the particular
Blockchain platform.
Besides the CAP theorem, we take other aspects as measuring factors, presented
in section 5.2 below.

In order to compare the Blockchain to centralized databases, one must consider how
each of them is designed and maintained.
The first consists of multiple decentralized nodes, as mentioned before, where each of
these nodes does the same job and participates in the administration of data equally.
This means that each node verifies the new entry to the database, and in order for
that addition to be finalized, multiple nodes have to reach consensus between each
other, to agree that the transaction is valid.

5.2.1
Confidentiality
As stated earlier, every node in the Blockchain verifies the incoming transaction inde-
pendently. This is possible because the node has knowledge of the whole Blockchain
database’s current state, the digital signatures that proves the transaction’s origin
and the modification requested to be done by the incoming transaction. Nevertheless,
this absolute transparency of the database’s state and data is a no-go for some type
of systems, like financial, record history, private company assets, etc.
The centralized databases avoid this problem by restricting the transactions that a
user can perform. It must be noted, that these restrictions are enforced at a central
location, which implies that the contents of the database can and must be visible only
at that central location, instead of multiple nodes. Likewise, requests to read the data
also go through that central location. This leads to the conclusion that centralized
databases can be both read- and write-controlled, whereas Blockchains can only be
write-controlled.
There are implementation strategies available that impose read control on the Blockchain
database, like transacting under multiple Blockchain addresses or cryptographic tech-
niques to encrypt the data, but any of these steps makes the computational burden
on the Blockchain heavier, which in terms decreases the performance even further.
5.2.2
Performance
Performance is a feature that Blockchain steps aside when compared to central-
ized databases.
It must not be misunderstood, though, that this is because the
Blockchain technologies of today are slow and not optimized. This is simply a result
of Blockchain’s nature. The two types of databases perform more or less the same
steps of work when processing a transaction, but the Blockchain has three additional
burdens:
• Signature verification: every transaction going into the Blockchain has to be
digitally signed using a public-private cryptographic scheme. This is needed in
order to prove the transactions’ origin, because of the peer-to-peer nature of
the Blockchain.
On the other hand, centralized databases do not have to prove the origin of each
incoming transaction, once a connection to the database has been established.
• Consensus mechanism: because the Blockchain has such a distributed nature, in
order to ensure the nodes in the network can reach a consensus on a transaction’s
validity, effort must be spent.
This might involve significant back-and-forth
communication between the nodes.
Centralized databases also do put effort in ensuring the integrity and validity of
transactions, but it is far less likely for conflicts to happen, since the transactions
are queued and processed in a single location.
==PAGE:49==
5.2 Blockchain vs. Centralized Database

• Redundancy: centralized databases process a transaction once, while a Blockchain
brings the redundancy of a transaction being processed individually by each sin-
gle node in the network.
5.2.3
Robustness
It cannot even be argued that the Blockchain type of database is extremely fault
tolerant. This is a default feature by its nature. As mentioned multiple times, each
node in the network does the same job of processing transactions, which means that
the failure of a single node is not crucial whatsoever to the overall database. Further-
more, because of the consensus algorithm in the Blockchain technology, it is ensured
that failed nodes can catch up on the latest database state, once they are back up
and running.
There are many techniques available, that ensure robustness in centralized databases,
eg. replication, master-slave, disaster recovery, and others. Nevertheless, all these
techniques require setup and configuration procedures, as a first, which are usually
difficult to do correctly. Secondly, once set up, these mechanisms require constant
and close monitoring and maintenance.
5.2.4
Disintermediation
This is probably the core value of the Blockchain database.
It means mitigating
the intermediaries in the communication between entities (eg. removing third par-
ties). Essentially, this characteristic enables direct sharing across certain trust rules,
without requiring central administration. The Blockchain achieves that through the
transactions’ own proof of validity and authorization, instead of having a centralized
application logic enforcing these constraints, as it is in centralized databases.
The fact is that data stored in a disk can be destroyed or corrupted, no matter the
security measurements taken. This means that the moment one entrusts their data
in a centralized database, they also become dependent on the human organization
that administers this database. Of course, the world is full of examples of valuable
data entrusted in organizations, eg. Google, Facebook, banks, governments. But this
brings the burden of the need for having trained and skilled people to manage the
databases. The Blockchain technology provides a way to replace these organizations
by a distributed, self-managed database, secured by smart cryptography.

5.3
Private vs. Public Blockchain
The public nature of Blockchain stems from the free and unconditional participation
of everyone in the process of determining what blocks are added to the chain, and
what its current state is. These fully decentralized Blockchains rest on a consensus
mechanism of proof-of-work (or proof-of-stake) for validation purposes [47].
This statement gives the idea that one of the core values of the Blockchain technology
as a database is ”database sharing”. This means that the database is open for read-
ing and writing by anyone, but the writing happens in an environment of ”limited
trust”. Limited trust would mean that the input transactions would undergo certain
validation in order for the integrity of the data to be kept.
Now what if an organization wants to utilize the Blockchain as a database, but does
not want to have its assets revealed openly. In short - it does not want to provide read
access for everyone, but still wants to allow writing to the database from multiple
sources within a limited trust environment, without the need to manage a central
system that handles the security. That is the Private Blockchain.
A private Blockchain would be one that is hosted and maintained privately by an or-
ganization, instead of being open to the public. In that case writing can still be done
in the traditional way as it is done in the Blockchain, but reading off the database
would be done only by systems/applications that are allowed to access the Blockchain
nodes. This approach solves the confidentiality disadvantage of the Blockchain, dis-
cussed in the previous section. At the same time, the shared writing capabilities are
not affected.
Bellow are considered the two scenarios in which an organization utilizes a private or
a public Blockchain. In the scenarios, the organization:
• Private Blockchain
– has to (probably) design and then implement the Blockchain and the con-
sensus algorithm by itself. Alternatively, it can use an already available
private Blockchain solution, eg. Multichain [39], Hyperledger [23]
– has to create the blocks itself, but does not need Proof-of-work [42] mining
– host the Blockchain nodes itself (on the cloud), which means there will be
hosting expenses and maintenance
– on the other hand, it does not have to pay mining fees in the Blockchain
– should not worry about confidentiality, since the Blockchain nodes are
hosted privately
– can customize the Blockchain, the block and transactions according to its
needs and use cases
– has full control of the Blockchain and can optimize it according to its views
and needs
==PAGE:52==
40
5 Discussion
• Public Blockchain
– can simply hook to a Blockchain of its desire
– does not create the blocks itself
– has to pay fees for each transaction because the miners have to mine the
blocks.
– cannot have full confidentiality over its data
– is constrained by the configuration, technical implementation and nature
of the Blockchain that it chooses to use
More and more organizations nowadays are looking into utilizing the Blockchain
technology for their own purposes, since it removes a lot of hassle with data validation
and integrity, which otherwise should be done by custom maintained systems, which
in terms involves trained employees.
Of course, as any time a choice has to be done about which technology to be utilized,
there are trade-offs and the advantages and disadvantages have to weighed. In such
case, the question has to be asked: Are the data confidentiality and implementation
customization or are the ease of initialization and hosting maintenance more impor-

Comparing the features of private and public Blockchain, it seems to make more

time. If a public Blockchain is utilized, the costs for transaction fees will grow linearly

moves the fees as a cost, and the hosting expenses do not grow drastically with the
network traffic.
5.4
Consensus Algorithms in major Blockchains
The Consensus Algorithm (”CA” ch.1) is the logical way the nodes in a distributed
system reach an agreement on the common state of the system. Taking the Blockchain
in particular, it is needed in order to keep the integrity of the data. It makes sure
that the next block in the chain is the one and only version of the truth.
Let’s take as an example a Blockchain network in which there are a 100 nodes, but
a transaction only gets propagated to 70 of them. In such a scenario, all the nodes
need to come to a consensus about which state of the chain is the de-facto one. This
happens through the Consensus Algorithm [57]. After that, all the nodes get updated
with the chosen state of the chain.
==PAGE:53==
5.4 Consensus Algorithms in major Blockchains
41
In the following sub-sections are described the Consensus Algorithms used in some
of the major public Blockchains.
5.4.1
Proof of Work
In this algorithm, the miners of the chain (the entities that host the separate nodes
of the distributed network), compete to add the next block by competing to solve a
cryptographically hard problem. This problem is hard to solve but extremely easy
to check and verify that the found solution is the correct one. Therefore, the miner
needs to perform a lot of work in order to find the solution. That is where the name
of the algorithm comes from.
Once a node finds a solution, new coins (or whatever the value token of the particular
Blockchain is) are created and are rewarded with coins. The difficulty of the problem
scales with the amount of nodes in the network.
This CA is suitable for public Blockchains, where there is an incentive of gaining
reward for the spent computational effort. It is nevertheless criticized that it requires
a lot of energy (electricity and traffic) to complete the computations.
5.4.2
Proof of Stake
In this algorithm, no new value tokens are created, therefore there’s no ”mining”.
Instead, all of them exist from the creation of the particular Blockchain and the nodes
are rewarded in transaction fees. Here a node is picked to be the forger (creator) of
the next block in a pseudo-random way, depending on the tokens it has put at ”stake”,
combined with other rules. Once the block is created, it needs to be committed to
the Blockchain, f.e. through majority voting and signing off by other nodes.
Since the tokens are put at stake, the node does not have the incentive of faulting the
validation of the transactions, f.e. by double-committing them or corrupting them,
because the tokens put at stake are slashed (terminated/lost) and the node loses its
right to participate as a forger in the network in the future.
Nevertheless, criticisms about this CA are that there is nothing to deter the forger
from double signing the block.
5.4.3
Practical Byzantine Fault Tolerance (PBFT)
The previous two algorithms are built on top of PBFT. Therefore it is a basis for
most used CAs in the different Blockchain platforms.
Naturally, the transactions read the nodes in the network at different time, therefore

nevertheless, it would be able to corrupt the sequence of transactions since it will be

5.5
Communication between the nodes
The Blockchain technology is a perfect example of a Peer-to-Peer network [41]. The
P2P network is defined by the fact that each node is ”equal” and acts as both server
and client. This differs from the traditional client-server architecture where there is
a centralized server that communicates to its clients and if the clients want to com-
municate with each other - they do it through the central server.
In order for the nodes in a P2P network to discover each other and communicate,
typically a so-called ”overlay network” is implemented [24] on top of the physical net-
work topology. This overlay is implemented on the Application Layer and is used for
making the P2P network independent of the physical network, and for peer discovery.
Generally, the overlay networks are classified as structured and unstructured.
• Structured overlay network: here the overlay is organized in a specific topology.
The goal here is that the nodes connect to some neighbours (neighbouring nodes
in the structure) that satisfy specific criteria. When data is transmitted from
a node, it goes to the neighbours, then to their neighbours and so on. This
makes the network generate less traffic, but it is less robust and characterizes
with higher latency in data transmission, compared to unstructured overlays.
• Unstructured overlay network: as the name dictates, this type of overlay does
not impose a specific structure by design. Instead, the nodes form a random
topology and connect to each other. Because of the lack of structure, these
networks are easier to build and allow for localized optimization. Furthermore,
because the role of all peers is the same, the network is highly robust and is
suitable for high rates of churn (the scenario when peers are frequently join-
ing/leaving the network). This makes it a very suitable type of network for
the Blockchain scenario. A disadvantage of this overlay is that a lot of traffic
==PAGE:55==
5.5 Communication between the nodes
43
is generated between the nodes, because the data is randomly flooded through
out the network. There is a risk of data requests duplication.
Judging by these factors, the Unstructured overlay network is the choice for a
Blockchain network implementation.
In fact, in the most famous Blockchain im-
plementation - Bitcoin - the network nodes interconnect in an unstructured mesh
network with a ”flat” topology [1]. This leads to the question of what type of data
transmission topology is a better choice for eSmiley’s use case. Some topologies in-
clude:
• Tree network: The nodes here are arranged in root-leaf (or parent-child) struc-
ture, where each node has its neighbours and simply propagates the packets
received from its root to its leafs.
• Mesh network: is defined as each node connects directly, in a non-structured way
to as many other nodes as possible. This way each node participates equally
in the relay of data. The lack of dependency allows for the network to self-
organize and self-configure which in terms increases workload distribution and
robustness in the case when nodes fail.
It is an obvious choice that the Blockchain implementation must utilize the mesh
network topology, because of its extreme fault-tolerance capabilities. Yes, the traffic
amount is greater than the tree structure, but in the certain scenario robustness has
a higher priority.
The mesh network relay messages in two ways:
• Flooding: this approach utilizes a routing algorithm in which the incoming data
is sent out to every outgoing link node except the one it arrived from. Since a
node C, that is connected to node A and B, can receive the packet from B, it
will try to transmit it to A. But A is the node that has sent out the packet to
B at first place. This a scenario in which flooding fails by executing duplicate
deliveries and must be controlled by implementing additional precautions.
• Routing: in this approach the packet is propagated along a path by hopping
from node to node until it reaches its destination.
In case the routing mechanism is implemented, each node will send out the packets
with a destination being each of its known neighbours (the nodes it is connected to),
because the point of the Blockchain is for a maximum number of nodes to agree and
have the same state of the chain. Following this logic, upon receiving a message, the
receiver node will route the message to all of its known nodes again. At the end of
the day, this propagation actually becomes flooding. So, in a sense, the Blockchain
network enforces the flooding type of message propagation.

The complexity of a Blockchain’s design and implementation can be quite high, con-

5.6.1
Implementing own Blockchain solution
Having an own implementation of a Blockchain layer is the ideal scenario, from the
following points of view:
• range of features that the Blockchain provides
• input and output format of the data
That would mean that there will not be excessive amount of features where some
might be unneeded, as it would be the case with a ready solution. This would be
beneficial for ease of use by developers. Other example is that the Blockchain could
be optimized to handle a certain type of data. This might as well contribute to per-
formance.
The second point is more important - this means that the Blockchain can be imple-
mented with an API with a preferred format that will fit the data format of the source.
This will remove the need of introducing a middleware to accept input and direct it
to the Blockchain.
The cons of implementing a Blockchain solution from scratch are of a broader as-
pect:
• high complexity of implementing a proper consensus algorithm
• high complexity of implementing knowledge sharing and integrity
• managing the security part of the solution
• maintenance and support
• time consumption and constraints
Without going into details, writing own cryptography is discouraged.
Mainte-
nance and support would mean that features will have to be rolled out and/or fixed
continuously, whereas available solutions usually come with decent enough spectrum
of features, that are supported either by big organizations or large open-source com-

5.6.2
Multichain
A very interesting available solution is the Multichain open platform for building
Blockchains.
It enables separate Blockchains to be created, and is optimized for
permissioned rather than open Blockchains. In that sense it is just like a database
platform. There is no official ”Multichain Blockchain” [39].
That being said, this platform presents good opportunities to be utilized exactly
as a database. Its unique feature, named data streams resembles a NoSQL database
and is specifically designed for recording any type of data, but not just recording
simple small transactions.
The way the knowledge sharing is implemented is that as soon as a node creates
a transaction, that transaction is propagated across the network, even before it is
confirmed/mined in a block. Data loss would only be an issue if the node stops in
the split second between the new transaction entering the local memory pool, and it
propagating to other nodes’ memory pools.
Transactions are also stored locally on disk at the moment of creation, even before
they are propagated. This way, in case of a node shutdown, the transactions will be
reloaded from this store when the node restarts and propagated in the network [39].

The main idea is that anyone can make their own version of Blockchain, by writing in
the so-called by Hyperledger ”chaincode”, which would define the data input/output
format and storage.
5.6.4
BigchainDB
The other major open Blockchain platform is BigchainDB [3]. It is, as the name
says, a database built on top of a Blockchain. The platform supports some great
features like easy querying of the assets, has been benchmarked for low latency and
fault tolerance.
Its main aim is to use the online hosted network of nodes by BigchainDB itself, but
it can be also hosted privately. The official documentation states that the network
takes about a second to come to consensus on a new block.

Containerization
Following the conclusion from the Analysis chapter (Sec.4.2) and the aim to solve
the problem of increasing the fault tolerance of distributed databases, the design of
the current system could make use of encapsulation by containers. This sub-section
discusses a few available container solutions for applications.
Docker is by far the most famous and most adopted containerization tool out there.
Characterized with extreme ease of use and rapid deployment, as well as additional
features for setting up the environment of the application and networking between
different containers that have to talk to each other. The main idea of Docker is that
a piece of software is sealed as an image (all set up and ready to run) and then this
image is executed in a container, as a live application. The beauty comes from the
fact that the images can be easily distributed around and have very small disk foot-
print.
rkt (pronounced ”rocket”) is the most worthy alternative of Docker. The focus falls
on compatibility of container formats. It supports its own and multiple others, in-
cluding Docker images. It is optimized for application containers, not OS containers,
and does not have that many third-party integrations available.
LXD (pronounced ”lexdi”) is still a containerized environment but emulates the ex-
perience of operating Virtual Machines. Nevertheless, its beauty comes in the fact
that it does not emulate hardware resources.
OpenVZ is an extension of the Linux kernel and provides tools for virtualization
to the user. It hosts guest systems by utilizing Virtual Environments. This means
that it uses containers that are optimized for entire operating systems instead of in-
dividual applications.
Linux VServer provides virtualization on operating system-level. Each of the virtual
machines is run as an isolated process on the same host system, without emulation.
This allows it to provide high efficiency. Nevertheless, it is a bit old-school because
no updates have been provided since 2007. [15]

6.2.2.1

After reading about separation of concerns and decoupling the different applications

both in one container together, instead of each encapsulated in its own container?
Let us take a step back and look at the design problem in place. Firstly, once a
Multichain instance is up, it starts listening for connections on a randomly chosen
free IP from the host system. This means that we cannot pre-define the IP of the
multichain-service. But the front-service needs to know an IP address in order to con-
nect to the multichain-service. Secondly, in order to solve the problem of achieving

optimal fault tolerance of the data store, we need to have a setup that provides rapid
deployment and node recovery.
With this in mind, we can proceed to look into two approaches of designing the
internal setup of the database node. Let us call the approach of having each service
encapsulated in a separate container ”approach A” and having both services encap-
sulated in a single container ”approach B”. There are a few pros and cons.
Approach ”A” has the following Pros:
• leverages the use of SOA and its benefits of handling updates and deployments
in smaller aspects/domains
• quickly recover a single container with only a single application within
• the multichain-service instances, encapsulating the Blockchain nodes, can stay
always running with almost no deployments and restarts needed. This is due to
the fact that we use the Multichain platform, which provides us the Blockchain
node
and the following Cons:
• introduces the need for having a third, outer service that will have the role
of an IP seed between the multichain-service and front-service. Since the two
services do not share a host environment (container), they cannot easily share
and exchange environment data. Sharing environment variables between the
host machine and the actual containers goes against best practices and defies
the use of containers.
An important note is that this IP seed service will have to employ logic that
prevents the scenario of Blockchain nodes ”starvation”, when multiple instances
of the front-service connect to the same multichain-service and there are other
instances of it left alone. Spinning up multiple couples of services will be a
regular occasion in order to handle scalability and fault resilience issues
• increases administration, maintenance and deployment efforts, since the overall
amount of services in the fleet is increased by the factor of two
Now let us look at Approach ”B” and its Pros:
• the two services share an environment, therefore the multichain-service can
directly pass its IP to the front-service. Thus, no need of IP seed service.
• decreases the maintenance and deployment efforts
• easier administration in a fault recovery scenario. Since the two services are
tightly coupled, it is easier to kill and/or start them together in one container.
This removes the need of designing an error handling logic in case the multichain-
service dies and having the front-service re-instantiate a connection to another

and its Cons:
• if one of the coupled services dies, the other one has to be re-instantiated as

• since the front and multichain services are within one single container, updates

Approach ”A” comes with the drawback of increasing the complexity of node discov-

7.1
Programming languages
The multichain-service is basically a wrapper of a Multichain instance, therefore there
is not much programming needed from our side, since the provided Blockchain node
works and provides functionality out of the box. The only part needed to be imple-

using this programming language is because interaction with the Multichain instance
can only happen through the Unix shell (terminal/console) by executing commands,
as described in sub-section 7.2.1. The automation mechanism essentially employs
conditional execution of such shell commands, and Bash is the de-facto language to
use when working with the Unix shell.
The front-service and the ip-seeder are essentially web-server applications with REST-
ful API. This gives us several options in regards to the programming language. There
is a big variety of languages that can be used for implementation but we will discuss
the four most popular options: Python, PHP, Node.js and Ruby.
• Python is famous for its extremely easy code readability. This is due to it
enforcing indentation for code structuring and line breaks. For web development
==PAGE:74==

it can be used through a web server interface implementation such as Django
and Flask.
It is convenient for quick server-side scripts, but is mostly used
in non-web purposes, like machine learning, AI and other complex algorithm

• PHP is one of the first server-side scripting languages. It is very easy to set up
and start using. It needs an installation of a server interface such as WAMP or

• Node.js is the big thing nowadays.
In the past couple of years it has sky-

The biggest power of Node.js is its very effective handling of multiple

makes it very suitable for handling massive amounts of traffic with no efforts.

Considering the various options, the conclusion is that the web services will be

big popularity of Node.js, which gives a variety of tools and integrations to be used.
Secondly, it is the asynchronous nature of the environment, which can handle loads

Looked at from the outside, the two apps form one single service (a node of the

This is achieved by layering Docker images. Firstly, we compose a Dockerfile that

This way, by following best practices, we have one single three-layer Docker image
that contains all the configuration and environment setup, plus the applications, that
we can run with a single command. If we wish to spin up a new server in the cloud,
we simply run that Docker image, without the need to install Multichain, Node.js or
any other setup, thus achieving a rapid deployment process.
The ip-seeder, on the other hand, is a completely separate service, running in its

7.2.1
multichain-service
As mentioned earlier in the report, this service is essentially a Multichain instance (a
single node of the Blockchain network). Therefore the only implementation needed is
for a functionality that deals with automated execution of the node and connecting
it to the network. That is done in a single Bash startup script. Below follows a basic
explanation of the internals of a Multichain node and how we utilize them, followed
by a description of the process flow in the startup functionality. The highlighted
phrases depict the main, important characteristics of Multichain that we deal with.
A Multichain node can either establish a new Blockchain or connect to an exist-
ing one. In both cases a configuration directory for the Blockchain is generated.
This means that if such a directory does not exist in the running instance, then the
node is not part of any Blockchain.
A Multichain instance joins a Blockchain network by connecting to another node and
duplicating its configuration directory. This connection happens through an IP:port
pair. The port can be set as a runtime parameter, but the IP is randomly chosen
from the free IPs on the host. Additionally, Multichain provides two ports: one for
node-to-node communication and one for world-to-node communication. The second
==PAGE:76==
64
7 Implementation
one is the RPC port, which exposes Multichain’s API. In our setup we use the port
number 4000 for node-to-node communication and 4001 for the RPC API. These
ports are configured the same across all database nodes for general standards and
operational reasons. Furthermore, since we run only one Node per host, for resilience
reasons, there will be no port collision.
The RPC API requires username and password credentials for each request.
Upon the first start of the Multichain instance, a random password is generated,
along with the multichainrpc username, and saved in a configuration file. Since
we want to allow the front-service to connect to the Multichain node, and we want
the same username and password across all instances, for operational and standard
reasons, we need to modify that configuration file. For this purpose, if its a newly
instantiated node, we stop it right after it has been started, modify the password and
then start the node again. The password in the configuration file can be modified
while the node is running, but it will take no effect until before the node is restarted.
Since we use a permissioned Blockchain, each new (child) node needs to be granted
permissions in order to join the network. Granting permissions happens on the side
of the parent node. That is done through a request from the child node to the parent
node’s RPC API, with the child node’s Blockchain address as a request parameter.
Since we have a standard RPC port, username and password across all nodes, it is
easy to simply grab the parameters and make the request.
The last step of the startup process is to share the IP of the newly started node.
First, with the container environment, so the front-service can connect to the node.
Second, with the ip-seeder, so other new nodes can use the current node as a parent.

7.2.2.1
Storing data
The data in Multichain streams is stored in hexadecimal format. The data from the
source (the sensors) comes in JSON format.
The sensors send not only measurement data, but also metadata, in their request
object. We want to store only minimal, important data regarding the measurements
of the units and to disregard any other, redundant data. We pick out only specific
properties from the input object. From this is formed an object, ready to be stored
in the database.
It is first converted to hexadecimal string and then sent to the
Blockchain. All this is taken care of by the front-service.
Obviously, we need to differentiate between the data stored for each sensor.
For
this purpose, a stream is dedicated per sensor. The streams in Multichain work on
the key-value pair basis. Therefore, the sensor ID is used as a stream name, and the
value is the actual measurements data. Since we do not want to manually create a
stream for each sensor, the a new stream is automatically created on a try-and-error
basis, as depicted in the diagram bellow:
Figure 7.2: Publishing data to the Blockchain.
The request input contains the sensor ID, so we publish the data to a stream
==PAGE:79==
7.2 The Services
67
with name equal to the given sensor ID. There are only two possible outcomes of
this. First, a transaction ID can be returned, which means the data has been stored
successfully as a transaction in the Blockchain. Second, an error saying that ”no such
stream” exists, in which case we will make a request to the Blockchain to create a
new stream with the sensor ID as a name, and then repeat publishing the data to
that stream name.
All nodes are started with the autosubscribe=streams runtime parameter (provided
by Multichain by default). This means that once a stream is created in one node,
it is immediately propagated throughout the network. Thus, any next request with
data for the same sensor, to any other node in the network, will not need to create a
stream, but will directly publish the data.
A few other write endpoints are exposed, such as creating a stream and subscrib-
ing to a stream. These endpoints are used for manual testing, rather than taking
action in the automated process of storing data.
7.2.2.2
Reading data
Quite a few endpoints are available on the front-service, that expose Multichain API
endpoints for reading data from the Blockchain. Example of such endpoints are: list-
ing all streams (which means all sensors that have stored data), listing data for a
specific stream (sensor), etc.
Multichain API returns a lot of metadata accompanying the requested entity. The
read endpoints all have one thing in common - strip out redundant metadata and
format the needed information into a custom object to be returned as a response to
the requesting application.
The endpoint of most interest is the one for reading data for a specific sensor. It does
conversion from hexadecimal to JSON format for each data record and composing a
list of these for the response.
7.2.3
ip-seeder
The Blockchain nodes need to know of a running node in an existing network when
they spin up, in order to join the network. This is a basic mechanism used all around
the different Blockchain implementations nowadays. The role of the ip-seeder service
is exactly that - to provide information to a new node about how it can join the
network.
It is a Node.js webserver application, running in its own Docker container.
The
functionalities are pretty simple. It accepts an IP from a node that has just started
and stores it in the list of IPs. Over a certain interval, eg. 10 seconds, it pings all
IPs in this list, in order to check whether they are still alive. If a host of an IP is
not responding in two consecutive pings, the corresponding IP is removed from the
==PAGE:80==
68
7 Implementation
list. This way we maintain an accurate map of the liveliness of the Blockchain nodes.
Then, there is an endpoint that returns the first IP in from the list. This is the
endpoint that a new starting node requests, in order to obtain the address of a parent
node, through which it can connect to the existing Blockchain network.

7.3.1
Orchestration platform
It is a common practice to use an orchestration tool for Docker containers. Famous
examples of such tools are Kubernetes [27], Docker Swarm [13], and others. Their
function is providing automated deployment, scaling and management of a fleet of
containers.
It is easy to manually mingle with couple of containers, but when one needs to main-
tain a larger set of containers, running multiple instances of different applications,
the manual work because tedious, if not impossible.
But probably the the most
important features for achieving high fault tolerance are the automated scaling and
self-healing.
The first means that new instances of an application can be started
with the execution of a single command or based on CPU load, in order to handle
the larger traffic. The self-healing is the feature that makes sure failed containers
are restarted or killed and replaced with a duplicate, in order to maintain a certain,
defined amount of application instances.

#1 When transactions are received in a node but not broadcasted yet

Risks & Future work
This section describes the risks in the current system design and the drawbacks of

work, that is to improve on those drawbacks and mitigate the risks.
9.1
Risks and Drawbacks
9.1.1
Data types
It is important to note that using the Blockchain as a database comes with its draw-
backs. Despite defined as out of the project scope, as mentioned in sec.2.2, we should
mention one of the major drawbacks of the Blockchain is that currently it is only
suitable for storing key-value pairs type of data and is impractical for relational data.
9.1.2
Loss of write acknowledgement
Another flaw, though small, in the current prototype is that acknowledgement re-
sponses of successful writing of data in the database drain out at around ten thousand
concurrent requests. Nevertheless, it was concluded that this is a problem with the
front-service, rather than the database platform itself. Furthermore, the case of such
amount of simultaneous write requests is not likely in eSmiley’s use case. Last but
not least, we must clarify that the storing of data is unaffected. It is only the actual
response ”data has been written” that drains out in that situation.
9.1.3
Failure of the Multichain instance
With the current design of the front-service and multichain-service being in the same
container, if the latter fails, the container will not stop and self-heal by itself. This
is because Multichain runs in a background process and the Node.js app is the one
running in foreground, dictating the container’s lifecycle. Nevertheless, this can easily
be solved by operations setup or through architecture design changes, both discussed
in section 9.2 ”Future Work” below.
The system design was approached in this way with the purpose of building a proof-
==PAGE:90==
78
9 Risks & Future work
of-concept prototype faster. The system still complies with the three major design
approaches for achieving optimal fault tolerance, as mentioned earlier.
9.1.4
Failure of the IP-seed
The ip-seeder service only serves the purpose of sharing knowledge about Blockchain
nodes in the network upon starting a new node. Therefore its failure would only
affect the process of spinning up new nodes. This is deemed a very tolerable fail case
and easy to recover from.
9.2
Future work
The implemented prototype serves the purpose of proof of concept and solving the
thesis problem, but it is not a perfect system as it is. That is why we look into
improving it for stable utilization in production environment.
First and foremost, the risk mentioned in subsection 9.1.3 must be removed.
As
mentioned, this can be done in two ways.
We can approach it by setting up a health-check mechanism in Rancher, executing
every X seconds, to check the liveliness of the multichain-service. Upon Y number of
unsuccessful responses, Rancher can restart the whole container. This way we achieve
automatic self-healing through operations setup.
The second approach is to make changes in the system architecture. In particular
this means the front-service and multichain-service will be decoupled into their own
containers each. Nevertheless, this will bring the con of a more complicated logic
regarding connecting a front-service with a multichain-service upon startup. Never-
theless, the benefits of separating the services in separate containers in the long term
are greater than this con.
The particular risk mitigation will be done in two steps: first the operational setup
for short term, and then the architecture changes for the long term.
The next thing to look into will be the DNS-seed and a Blockchain node crawler.
These two mechanisms will replace the current ip-seeder service. Employing these
technologies is much preferable than the in-house developed IP-seed, since they are
”battle-tested” in live systems, provide safety and have been developed with the sole
purpose of node discovery in the Blockchain and monitoring of their liveliness. Fur-
thermore, the startup process of the Blockchain node will be strengthened in regards
to error resilience.

CHAPTER 10
Conclusion
This thesis argued for the usability of the Blockchain technology as a database and
its fault tolerance. The goal was to get away from centralization of the data store
and remove possible single points of failure. A thorough investigation of academic pa-
pers, proposed solutions and available commercial tools was conducted. The analysis
concluded that very few solutions use the Blockchain as a primary database and no
solution has employed system design techniques for achieving high availability and
resilience of the data store.
In order to solve the problem at hand, a new design was proposed. It makes use
of Microservices, encapsulated in Docker containers, and architecture that allows for
rapid deployment and self-healing: three major approaches for achieving optimal fault
tolerance.
The proposed design takes a step further and removes any business-logic related ser-
vices between the data source and the store, thus opening the database for direct
writing from the source. This approach removes the back-end application as a SPOF,
as it is in a majority of web systems.
Following this design, a prototype was implemented.
At the current state of the
project, the system works as to prove the concept of using the Blockchain as a con-
tainerized, distributed database. The prototype follows all the aforementioned design
approaches for achieving optimal fault tolerance and has successfully withstood the
stress and performance tests. With this said, it can be concluded that the imple-
mented system is a fully functional minimal viable product.
Noted earlier, there are some drawbacks running the current prototype.
Despite
that, it is concluded that they do not pose fatal risks to the system and the benefits
of the current design overcome its flaws, for the project’s purpose.
As a final note, it is concluded that the project successfully solves the thesis problem.

APPENDIX E
Code comment
The following appendix depicts a comment which is an excerpt from the code in the
multichain-startup.sh script. It describes in nice wording the steps in the process of
starting up the Multichain node within the Blockchain Node container. The flowchart
in Fig.7.1 describes the process in a technical way.
1 Check if the multichain blockchain node dir exists.
2 -> If yes:
3 --> start up the node
4 --> publish its IP to the container environment and the IP-seeder
5 -> If not:
6 --> then it is a fresh container with no BC node config on it. Proceed to
the steps described below...
7
8 Then get an IP of an existing node on the network from the IP-seeder.
9 -> If the response is "NO-IP", it means there are no IPs in the list, which
means it is the first (root) node in the chain. Therefore:
10 --> create a new blockchain configuration
11 --> start it in the background, so the RPC credentials are generated
12 --> wait 2 seconds for it to start and then stop it
13 --> update the RPC credentials
14 --> start the node
15 --> publish its IP to the container environment and the IP-seeder
16
17 -> Otherwise, the response should be an IP of an existing (parent) node.
Then:
18 --> connect to that parent node. If unsuccessful , do 3 extra requests to the
IP-seeder + connection attempts to parent node. If none successful -
kill node
19 --> grant permissions for the current node
20 --> update the RPC credentials (the `multichain.conf` has been created
already)
21 --> start the node
22 --> publish its IP to the container environment and the IP-seeder
.
