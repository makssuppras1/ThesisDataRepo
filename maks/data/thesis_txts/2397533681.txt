Abstract
In this report the focus is mobile app development. The goal is to ﬁnd out, if
storyboards can be utilized in developing mobile applications. For this we set up a
hypothesis, that will test the statement that it is possible to develope app with
storyboards without writing any code.
To prove the hypothesis, exiting tools are examined alongside technologies, that are used
in diﬀerent areas of developing apps, as well as automatically generating code from
diagrams, with Model Based Software Engineering, and editors to allow for user
interfaces.
A new storyboard language, supporting that storyboarding apps is possible, is deﬁned.
This language provides the notation for an application storyboard and the guidelines for
how apps is deﬁned in the storyboard.
We work on implementing a proof of concept compiler, which can support the
transformation of a storyboard into source code for a mobile application.
In the end, it is partially achieved to create a compiler, which can generate an app from
a storyboard. While a runnable app is not achieved from the compiler, the possibility of
it working is there.

Introduction
In this report, you will read about a method for how automating development of mobile
phone applications (apps) can be achieved and how it can help solve problems regarding
the development process, by introducing a more visual way to develop apps.
Coming up with the idea for an app, is not the problem. However ﬁnalizing that idea, is.
Today, most app development lies in the hands of teams with skilled developers, designers
and programmers. It is possible to develop apps, without having a team, but the less of a
talented developer the person are, the harder the job becomes, and the longer time it will
take. That means, it can be necessary to hire a team to develop the app, which can get
very costly, depending on how it is done and how much should be developed.
The problem looked at in this report, is the lack of complete solutions to develop apps,
available for small business and persons with little resources and programming knowledge.
There exist a large number of tools to develope applications for various purposes. These
tools span from the use of basic templates, to drag and drop technologies for the UI and
then for the logic, to code from diagrams and more. Some of these tools are desktop
speciﬁc and some can be found in app form and used directly on a mobile phone.
However most require great programming knowledge or lack customization options or
needed functionality.
It has for a number of years been possible to generate code from diagrams, by designing
the structure in a diagram. This has made it possible for developers, to deﬁne complete
programs in diagrams and then be able to automatically generate the code for the program.
This will in this report be used to ﬁnd a
On the UI side program interfaces can be developed with the use of editors, with the drag
n drop feature, which can automatically generate the UI code. Here complete applications
UI can be designed, without the need of writing the code.
The problem occurs when we try to combine the two sides, of Logic and UI.
App development has however evolved to a point, where it is accessible to more people
than before, with online guides and templates to use. But creating apps of your own, still
requires basic knowledge of programming and app development, not mentioning designing
a good app. It is therefore still not everyone, who can create good or successful apps.
The problem is then, how can app development become easier accessible to individuals,
whether the need is for commercially or personally use.
The project will ﬁrst look into existing solutions and their limitations. What do they oﬀer
and what do they not oﬀer? Where after a proposed solution will be introduced. The
solution will focus on the use of storyboarding to develop apps. A user of the solution
should not have to write any app code.
The project will end with a proof of concept.
The aim is to ﬁnd out how much of a
given hypothesis are feasible. It will therefore not be the aim to develop a fully functional
program, but to create a minimum viable prototype, that shows what the goal is possible.
The project will result in a partial deﬁnition of a language notation for the storyboard

creation. It will also result in a way to test the idea, by means of a example and a simple
compiler to transform the storyboard into code.
At the end there will be discussed new concepts and what limitations the concept have.
From the result, there will be discussed how the road ahead looks and what the concept
of storyboarding mobile apps can be used for.

2.1
Problem
To better understand what the problem is and where it is applicable, we will ﬁrst start
with two scenarios.
Say that you have a small service company within the marketing and sales department.
You have a few employees, whom have mixed backgrounds working with marketing
and design. In order to get your service out to more potential clients and to provide
better service for your existing ones, the company needs and app. However, you and
your team of employees have no experience in app development or programming. De-
veloping the app in-house is therefore not an possibility, so you need to hire another
company to do this. When looking at prices and oﬀers, for what the app will cost to
be developed, it is clear, that the company can not aﬀord this. Your team can carry
out the app UI design and you only need the remaining development to be dealt with.
In another scenario you might want to build an app just for personal use. The apps pur-
pose might be to share between family or friends, either to showcase your holiday
experience or it might be a more feature rich app, which will be used to plan or
share stuﬀin between a small group of people.
For either one of these apps, you know how it should look and what features it
should build up the app. Because this will be a small fun project, you do not want
to spend money on expensive development and want to do it yourself. This way you
can at the same time acquirer some knowledge about app development, as you have
no experience in the area.
In the above scenarios, we have the problem, that app development ends up being to ex-
pensive for some companies and individuals or in fact to time consuming. Another problem
comes from lack of technical skills needed to carry out the development process, which
leads to extended development time or in the worst case, not even a complete product.
The latter, could result in lost time and resources and important revenue.
Lack of skills, could be a problem that could be addressed. However diﬀerent initiatives
are trying to ﬁx this, by e.g. focusing on the goal, that programming should be taught
at an early stage, for instance Code Pirates [1] and tools such as Code Kingdoms [2] and
Scratch [3], which all aim to teach children to code early on. Instead, in this report the
two scenarios will be addressed as the problem needed to be solved.
For an individual person, creating apps for them self, the money issue might not be of a
great concern, since there might be no cost associated with the app in the ﬁrst place, since
they are developing app. However, lack of skills and the prober tools, might lead to badly
build applications or no application at all, when the project fails. On a larger plain, this
might not seam as a mayor problem and might seam irrelevant for some. So why should

For small companies and individuals, getting an app developed can therefore be a great

2.2
Goal
The goal of the thesis is to tackle the two problems from the [2.1(Problem)]. The two
problems can be extracted as:
1. App development are not possible for some small business and individuals, lacking
the needed resources and technical skills
2. MBSE lack the possibility to combine Logic and UI in one design
In this report, it will be examined how a solution for the speciﬁed problem, can be devel-
oped.
The solution will address problem number 1 as the main problem. Then, for future work,
the solution and ﬁndings from this report can be applied to problem number 2, to examine
if a viable solution can be developed.
We deﬁne a hypothesis, that it is possible to combine storyboard technology with app
development, by creating a storyboard that deﬁnes a complete app. App Completeness
means that every aspect of an app is deﬁned, that is all the UI elements and all the Logic
elements, i.e. deﬁning app behavior, ui controls, data deﬁnitions, user controls and general
app parameters.
Hypothesis
It is possible to develop an app, by only designing a storyboard!
The approach is to combine storyboards and application development, with model driven
architecture and a drag and drop UI design.
In the report it will be investigated, if the above hypothesis is can be true, by analysis
og existing technologies and tools and by examining what is needed to develop a solution,
that solves the problem.

3.1
How are apps developed today
The app development start as an idea, which might be deﬁned in written text or sketched
on paper. From that, the basis of the app is deﬁned with decisions about where the app
should aim for and possible understanding of the target group. All this is the inception
phase. In general, the idea is deﬁned in detail, with what functions it will have, how it
should look, and who the users are etc. This is often referred to as the design phase. The
design can then be prototyped and tested to ﬁnd out what works and what doesn’t. From
testing the prototype app, the design phase can then be revisited or not, depending on the
outcome of the test. When the design has been tested to satisfy functionality and have
the right look, the development phase can begin. Here the app idea is turned from design
into functionally code, which can be then be deployed as an app.
Dependent on the scale of an app project and the resources available, diﬀerent sizes of
teams can be assembled. These teams can range from only one developer developing the
app, to multiple individuals of the type designers, programmers, testers, etc.
3.1.1
Who can do it?
Developing apps are not just for commercial purposes, whether it is productive apps, me-
dia apps or games, but have become much of a hobby for some individual developers. This
is mainly due to programming knowledge have become widely spread and more accessible.
This is then supported with wide range of tutorial and guides found online.
That has resulted in more and more people being able to program apps and more people
have the desire to develope apps, either for fun or for commercial purpose.

If a person have a general programming knowledge, then creating an app is not an unob-
tainable goal.
3.2
Automation
Automation have had a huge inﬂuence on software technology. The possibility to auto-
mate boring human task, has pushed the development of automated systems in a direction,
where it today is more common to use software for many tasks. Task automation can ei-
ther mean automating something physical in the real world or it can be automation of
computer tasks, where instead of letting the user handle areas of some software process,
the process is automated and handled by the software.
In this report, automation could be seen as, the process of programming the app. As the
goal for the report is to ﬁnd a solution, which requires no programming, then a solution
could do the programming automatically.
3.2.1
Limitations
The limitations of automated systems are it often is created to solve a speciﬁc problem.
The developer does therefore need to know the whole area the automation aﬀects and they
need to think of all scenarios. An automated system are limited to the scenarios covered
in it’s creation.
3.3
Diagram-to-code
In the MBSE, Model Based Software Engineering, technology, it is possible to create
software without writing any code. This is achieved by deﬁning models of the logic for
the software that is being developed.
The idea works because we develope a piece of software, which can handle the automatic
code generation. The only thing this automatic generator needs are a diagram or model
of the desired software. The generator can then, from the given model, produce the code
for the new software. This can however only be achieved, because the generator knows
what models are, i.e. what the look like and how the work. Given a syntax, the model has
to follow, it is then possible to create any software with the generator, by feeding it a model.
One issue with such softwares are that it often only is good at one thing, that is either
creating logic or interfaces. That is not because the software couldn’t do it, but because
it is diﬃcult to deﬁne both elements in one model.

Visual Programming Language: No coding needed
In contrast to textual programming, where code is written, visual programming uses pre-
deﬁned code elements, which are linked together in a visual representation.
VPL are often seen used in coding programs for children, as it provides an easy to under-
stand interface, from where they can learn how to create applications.

Scratch is a tool aimed at creating simple ﬂash games and is developed with the
purpose to teach programming skills to kids in the age 8 to 16. The logic in Scratch
is deﬁned using a visual programming language and builds on drag and drop of
code blocks. The logic is ordered in scripts, which is designed with the vpl and
is organized in a single workspace and gives the user an overview of the game in
just one place. In the workspace, the user can add comments, which is displayed as
a post-it note. A ﬁgure of the Scratch editor can be seen in Figure 29 in Appendix[C].

that uses drag and drop code blocks to code the logic. These blocks are a replacement
for traditional code lines and as so does not change any structure of the program,
but changes how it is displayed.
The codekingdoms tool provides the user with tutorials, which provides better un-
derstanding of the editor and not the least of programming and language used.
Codekingdoms uses java for minecraft modding and lua scripting for Roblox games.
A ﬁgure of the CodeKingdoms editor can be seen in Figure 30 in Appendix[C].
The codekingsom editor gives users the choice to change how the code is displayed.

to display clear java code, giving a total of four diﬀerent ways to see and write the
code.

3.6
Storyboards and Storyboarding
Storyboards is a way to plan the ﬂow of a movie or a visual story, i.e. planning the shots
or frames of which the movie consists. Storyboards consists of a series of linked images
accompanied by some textual description, e.g. the story and dialogue or sometimes notes
about the image. Storyboards are often drawn on paper and starts as a fast way to simu-
late the ﬁnal product, as it quickly can be changed, to ﬁt a new direction, without having
to change everything. It a storyboard is correctly drawn, it should from that storyboard
be possible, to visualize the whole picture of how the ﬁnal movie or product will look like.
With a storyboard it is also easy to ﬁnd out if the story are cohesive or an image or two
needs to be added somewhere. Storyboards are an important tool in movie writing, as the
story can be layed out by mayor key scenes, and show the complete story.
Storyboards are most commonly used in production of video-sequences, such as movies,
and the storyboard format used today was originally designed by Walt Disney in the
1930’s.
Storyboards can however also be used to sketch games, as most games builds around a
story. Just as with movie stories, a story for a game can be sketch in a storyboard, with
the dialogue or player choices written as notes to frames. This can be part of, writing a
more cohesive an engaging story, that the player wants to follow.
When creating a storyboard it is described as the process of storyboarding.
By storyboarding a product, the complete picture can be visualized before any actual work
is performed, i.e. ﬁlming, designing, producing, etc, which ensures a more complete ﬁnal
product, as corrections can be made in an early stage.
3.6.1
Use of storyboards in applications
Use of the storyboards technology and storyboarding process has recently been imple-
mented, in the development process of software engineering, i.e. user interface design, UI
design, web-sites and apps alongside logic and functionality, as a form of planning the
visual and logical components in the design phase. In software design storyboards can be
used to link diﬀerent wireframes together. In this form a storyboard shows a ﬂow, from
navigating between wireframes, which might be screens in a phone application or diﬀerent
states of a screen, e.g. a new state could be when a popup overlay is activate through an
action.
For an example, this is used in the program Visual Paradigm [7], where it is possible
to design storyboard layouts for diﬀerent target applications platforms, such as websites
and apps. The program links the applications wireframes together to show the complete
overview and user navigation.
Storyboarding is also used in the process of user experience design, UX design [note2],
because a ux designer can create a ﬂow of the user-story.

Proposed Solution
In the previous sections [3(Analysis)] and [4(Existing Solutions)], we have looked at exist-

we have looked at some existing alternative tools which provides am app design platform.
The knowledge gained from the available technologies and from the strengths and weak-
nesses of existing tools, will in this section be used to design a solution for storyboarding
mobile apps.
The purpose of the solution is to design a tool to automatically generate apps. A user of
the tool should be able to create a functional app, by storyboarding the app design and
logic.
The aim is to give the tools from with a user can storyboard the whole app. With the so-
lution a user can design the visuals as well as the logic behind and the app user interaction.

5.4
Writing a storyboard
In order for a storyboard to be complete, i.e. an app are fully deﬁned, it needs to include
the elements discussed before in [5.2(From app to Storyboard! What is Required)]. That
would be the user interface, the app logic and app parameters.
From the above mentioned elements, the ui can be compared to the scenes in a storyboard
of a movie and the logic are then the story and comments, that bind the movie together.
When writing the storyboard, each of these elements can be considered as a separate layer,
as illustrated in Figure 6. Each layer deﬁnes a domain of a speciﬁc storyboard component.
The ﬁrst important layer are the Logic Layer. Components in this layer are also the ﬁrst a
storyboard needs to know. This layer can be seen, as the general structure of the app. In
this layer, we do not consider actual content, but instead consider how the main compo-
nents of the app is connected and interact with one another. Main components are then
screens, data sources and data containers. The logic is then, where does data come from
and where is data containers used, i.e. what screens or other data containers are they
used in. The logic is also where is transitions in the app, i.e. what screens can transition
to other and which. The logic layer must be deﬁned in such a way, that a developer can
see how the app will work, even without any content.
The next two layers are the UI layer and the Data Layer. These layers are located within
23
==PAGE:30==
the logic layer, as part of the deﬁnition of screens and data containers accordingly. In
contrast to the logic layer, we here consider the actual content of the app. This is then
how is data represented and how does a screen appear to the user. Components in these
layers are children of their associated logic component.
The UI layer is in focus, whenever content in screen components are being deﬁned. This
is when the appearance of a screen is being deﬁned, i.e. which ui controls are being used
and how are it presented. It is color, size and font deﬁnitions, etc.
Additionally data layers are in focus, when data in data containers are deﬁned. This is
deﬁning how the data is stored in the app, by deﬁning the structure of the containers, i.e.
which ﬁelds does a container have.
When the three layers are all present, that is when the storyboard shows a complete
picture of the app it deﬁnes. It is then, where the storyboard can be used and is sound,
as a viable development process.

How to deﬁne the Logic
Before, we looked at the Logic layer and discussed how it deﬁnes the structure of an app

Logic components are deﬁnes using unique symbols. How these symbols can be used to

The ﬁrst logical component to consider is the screen. Screens are the main focus of most
apps, as this is what an app user will see. In app development screens are a combination
of ui and logic, and are often deﬁned in at least two separate ﬁles. However, when writing
the storyboard, these will be combined, and are denoted by a single screen. A way to
understand ui and logic in the storyboard, is to see the actual screen symbol as the screen
ui part, while the screen logic is located around the screen. For instance, a navigation
from screen 1 to screen 2, cast by a button tap, is shown with a connection between the
two screens in the logic layer. Looking at the code, this action would be located within
the logic part of screen 1. Because screens are the main focus, these can have multiple

Additionally screens can have one or more gesture symbols attached. These symbols de-

ui components, e.g. attaching a swipe left and right on a list view component.
One of the components can be the connector, which was mentioned before. There it was
used as a transition between screens. Such connectors need to contain some parameter,
deﬁning how to behave. The ﬁrst is where to start from.
In the logic layer it is enough, that a connector just has a screen as source. But when in
the ui layer, this connector needs to be extended to the actual control, that will cast the
transition, e.g. a connector goes from button A in screen 1, and points to screen 2, e.g.
the connector links a ui component to an external logic component. This shows that there
are some linking between layers, which in the end gives the complete picture of the app.
Other than acting as a transition, connectors should also be used be linking data to screens.
This means, that a connector between a data container and a screen or ui component, de-
notes the usage of that data in the given target, e.g. a ui component of type image is
set to the image ﬁeld in a data containing. This is equivalent to initialize variables in the
logic part of a screen. Assignments can either point to initialization of class ﬁelds or can
point to methods called at a later point in the runtime.

The previous sections has discussed app development using storyboards as a concept and
how the design could be achieved. With the storyboards we have seen, that it is possi-
ble to deﬁne an app with a storyboard, but this is still far from being called a complete
runnable app. To actually develop an app from the storyboard, and that is without the
developer programming it, then we need an application, which can do this. The solution
must therefore incorporate a tool, that can transform a storyboard into an app.
The solution can be described as a combination of three parts, the storyboard, the com-
piler and the result.
The ﬁrst part is the actual storyboard, which we have already discussed and includes the

==PAGE:36==

generate an app from the storyboard with no additional input from the developer.

5.6.1
The Visual Programming Language: StoryLapp
For the storyboard and compiler to have a common understanding, it is essential that the
solution has a general notation describing the storyboard structure. This notation will be
in the form of a programming language which we will name StoryLapp.
The language StoryLapp will, in the ﬁnal solution, be the building ground for the story-
board and be the language with which the compiler will interpret the storyboard, in order
to create a given app.
Since the workspace, which the user will be using, will be a storyboard and since the
solution addresses non-programming skilled users, it is then best to deﬁne a VPL, which
can be used to create the storyboard.
31
==PAGE:38==
The VPL is designed as a diagram language, with block-based ﬂow elements. In contrast
to other VPL’s, StoryLapp should focus more on visualizing the logic with symbols rather
than code blocks. Instead of drag and drop’ the logic into a code block, e.g. like Scratch
and Codekingdoms [see ﬁgure 29] and [see ﬁgure 30] in Appendix[C], the language
should provide the possibility to form the logic via visual elements. This was discussed in
[5.4.2(The Storyboard)] Later, the future of the language will be discussed [see section 7.1].
To achieve all this, the language must support both user interface programming and
logic programming. The storyboard design can be split into the three layers discussed in
[5.4(Writing a storyboard)]nd is here summarized and out into context.
The Logic Layer
In this layer, the language deﬁnes screen elements the app can have and the transi-
tions between them. Futhermore, the source of data is deﬁned here, alongside app
parameters, transition functions, where data is assigned, e.g. linking a data source
to a screen.
The logic layer is the outermost layer, and is the essential picture of the storyboard.
Within this the two other layers are located.
The Data Layer
In the logic layer, the source of data is deﬁned. However, in the data layer, we deﬁne
how that source data should be represented in the app, i.e. how the data is used.
This is similar to deﬁning a data class in any object-oriented programming language.
An example was seen in [see section 5.4.2] .
The UI Layer
This layer resides in the screen elements. Here the individual screens’ UI is deﬁned.
Using the logic of the three layers and what was discussed in [5.4.2(The Storyboard)] the
StoryLapp language can be deﬁned. In Figure 14, we deﬁne a meta model of the language.
The three layers can then be outlined as, storyboard and logic fragments, data fragments
and screen fragments accordingly.
In the language we can deﬁned, that a storyboard is required to consist of one story-
board element, which then can have one or more parameters. These parameters describes
app speciﬁc settings, such as screen orientation and app name, which was discussed in
[5.2.3(App parameters)].
The storyboard element can then compose one or more logic fragments. This corresponds
directly to the logic components in the logic layer, [see ﬁgure 6]. These fragments describe
what elements the app consists of and in what relation, i.e. how they interact and are
connected with other elements. Logic Fragments are Screens, DataSources, DataBlocks,
Gestures and Connectors. The logic fragments interact with other fragments using a set
of Parameter’s, which can initialize any fragment. An example could be, if an image on
the MainScreen [see section 8] is tapped, then that image’s ID would be sent to a new
instance of the ProjectScreen, within a Parameter.
Screens and Datablocks, are logic fragments, which can compose multiple of their own set
of fragments respectively, i,e. a screen can compose one of more Screen fragments and a
DataBlock can compose one or more Data Fragments.
A screen is build of one to many Screen Fragments, which deﬁnes the screen UI design.
Screen fragments can take to form of a number of elements, describing what corresponding

5.6.4
Compiler
When a storyboard ﬁle has been generated from an editor, it can be feed to the compiler
as an input.
The Compiler is designed around 4 components, which can be examined in the illustra-
tion, Figure 17. First the compiler needs to open and store the ﬁle. This will occur in the
Reader component. This phase copies the storyboard ﬁle into the compiler, so it can be
processed. Further understanding of the storyboard happens in the next phase.
That phase are the Interpreter.
As the storyboard are in multiple parts, i.e.
all the
screens and connectors are separated alongside data assignment, the Interpreter then tries
to merge these into the correct locations. The purpose of the merge, is to prepare the sto-
ryboard for the code generation, as this could otherwise create problems in the compiler.
Merging the storyboard includes eliminating connectors and instead including them in the
target or source screens. For instance, in Figure 8 the navigation from ProjectScreen to
MainScreen, would be included in the ProjectScreen as an action. This action can then
be additionally deﬁned during the generation.
The reason why the compiler is split into these diﬀerent components instead of creating
one component, which could handle everything, is to design a solution which is more acces-
sible for testing and to provide a design, which are easy to modify in future development,
so additional functions can be added.
When the Interpreter has run through the storyboard and merged necessary parts, the

6.2.5
Compiler Components: Interpreter
This component interprets connectors and datablocks. Each element are placed in either
their deﬁned target or source screen, depending on the type. The target and source screens
must have been created before this component are run.
In the interpreter, connectors and datablocks are retrieved from the storyboard. These
are then processed one at a time in the following order.
Connectors
A connector contain a ﬁeld type and a ﬁeld action.
Action denotes how the connector is used and can either be activated with a button click
or at screen initialization.
A button click action, will in the interpreter be translated
into a method of the type onButtonClick, which can later be generated into code by the
Generator [see section 6.2.6] . This method will be stored in the screen, with the relevant
connection info from the connector. An initialization action instead means, that the con-
nection will be activated during a screen initialization. As a button click this is stored in
the screen as a method of the type void and is stored to be called in the initialization of
that screen.
The type ﬁeld can be either navigate or assign. If the type is a navigate, then the in-
terpreter stores the connection in the source screen and if the type is assign, then it is
stored in the target screen. The navigate tells the interpreter, that when the connection
is activated, a transition from the source screen to the target screen should be performed.
This transition is then called from the source screen.
An assign type tells the interpreter, that the target screen should use some data in the
screen code. This can either be at initialization or on a button click, determined by the
action ﬁeld.
DataBlocks
DataBlocks contain info of how data is represented in the app. To deter-
mine where in an app DataBlocks are used, the interpreter ﬁnds the info from previously
processed connectors, and adds the info to the relevant target or source. In this process,
the interpreter tells the target or source, where to assign given data and how to utilize it.

Evaluation and Conclusion
7.1
Future development
The report has resulted in an implementation, which supports a basic storyboard. The
storyboard contains a few app elements and therefore lack more divers and complex ele-
ments. In order for the solution, discussed in this report, to be a useful tool, it needs some
future work. That means, that a complete version of the tool, would require development
towards more support.
Improvements
Before developing new features, the Producer in the compiler must be further devel-
oped. This should be able to produce some sort of functional app, based on a given
storyboard.
Additional support
Storyboard support
The support mentioned in the previous paragraph would be more storyboard fea-
tures. Here could be added more ui controls and layouts. A feature complete tool
should support the controls and layouts provided by the target OS, e.g. ui provided
by android should also be supported in the storyboard.
Device and Platform support
Next it could be a good idea to add support for more platforms, i.e. the OS where
the app will run on. The implementation has focussed on Android so future work
could include IOS and Windows. Then it could be an idea to work on diﬀerent
target devices. So instead of only supporting phone applications, the tool could also
support tablets, tv etc.
SDK versions
Additionally to supporting multiple platforms it would perhaps be necessary to sup-
port multiple versions of the SDKs provided by the diﬀerent OS’.
Programming support
To not only provide a tool attractive to users, who are not going to write code, it
would in a future version be good to support custom functions. These could be de-
ﬁned in a FunctionBlock and assigned to other Logic Fragments, in the same matter
as DataBlocks. Writing functions could then either be in a traditional language,
either C# or the language used by the target platform or there could be developed
a VPL for the purpose. The VPL could the use the same technology as some of the
mentioned tools in [3(Analysis)] and [4(Existing Solutions)], see Appendix[C]. The
FunctionBlocks should be part of the written storyboard.
Developing on the solution
When a more feature complete version of the tool is
developed, the focus could be to develop an editor from where to deﬁne the storyboard.
Additionally or instead there could be developed and integrated tool, containing the editor
and compiler.

7.2
Possible applications
Beyond using the solution for app development, it could be used to produce working
prototypes. Then it could be possible to test a design idea or test an app idea on a small
test group, before developing a bigger app with either the tool or in an existing IDE or
tool.
7.3
Conclusion
The goal for the report was to prove the hypothesis, [see section 2.2], which was:
It is possible to develop an app, by only designing a storyboard!
In order to prove this, the concept of storyboards and storyboarding, [see section 3.6], was
examined, alongside technologies [see section 3] which could be applied to a solution, for
it to work. Existing solutions was examined, [see section 4], and valuable technics could
be applied to the solution.
To ﬁgure out what was needed, the working example was created,[see section 5.1], which
was used to determine what an app needs to be fully deﬁned, and the best way to utilize
storyboards,[see section 5.2].
From determining the best way of utilizing storyboards, it was determined how apps should
be sketch within a storyboard. This resulted in a storyboard structure,[see section 5.4.2],
which could be used to fully describe any application. From that a storyboard language,
StoryLapp,[see section 5.6.1], was successfully created. Then, in order to automatically
generate code from a designed storyboard, a compiler was deﬁned as part of the solution.
Although the compiler did not meet a full implementation,[see section 6.2], it did show
that it is possible to generate app code from a deﬁnition given in a storyboard. The com-
piler can successfully read a xml deﬁnition, then interpret on it in order to transform it
into the two app parts, ui code and logic code.
A storyboard editor was neither created, which meant that the storyboard only could be
deﬁned in a textual xml ﬁle. On a basis of an editor, it can not be concluded that this
would work. However, on the basis of how existing IDEs ,[see section 3.4], utilizes this
technology, it can then be assumed the the idea would work.
From the work included in this report, I would say that the hypothesis holds in some
instance.
It can not be said, if a functional app can be generated from storyboards,
since the implemented compiler doesn’t succeed in this part. It is however possible to
generate the needed source code used to build and app, in the small scenario tested.
The implemented compiler should allow for addition of new functionalities, with future
development. If the android SDK or similar is used instead of the deﬁned library,[see
section 6.2.8], then it might be possible to add support for more complex app generation.
In conclusion, for the hypothesis to fully hold, more work needs to be put into the concept.
