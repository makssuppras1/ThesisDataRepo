Abstract
Stroke is a serious medical condition that can result in patients losing various abilities.
These loses can be recovered through rehabilitation therapy. The ReHyb project aims to
help therapists with the use of a human digital twin system. The system will allow thera-
pists to plan exercises, which will be subsequently performed with the help of exoskele-
tons. This thesis describes the implementation and testing of the therapist dashboard.
The implementation of the user interface and visualizations was completed. However, sig-
nificant limitations were encountered while integrating the dashboard with other systems.
The evaluations of the user interface did not reveal any bigger issues, and the evaluation
of 3D visualization by therapists revealed limitations in their intuitiveness. In future work,
the dashboard should be integrated with functionalities that are currently missing in the
server app implementation. Implementation of dashboards for rehabilitation systems.

Introduction
1.1
Motivation
A stroke is a medical condition that occurs when there is a sudden interruption of blood
flow to the brain. This can happen due to a blockage in a blood vessel (ischemic stroke) or
when a blood vessel bursts and causes bleeding in the brain (hemorrhagic stroke). When
the brain does not receive enough blood and oxygen, brain cells begin to die, leading to
various impairments depending on the area of the brain affected. [1] It is the second-
biggest cause of death and disability worldwide. Moreover, even though the incidence
declined, the total number of people who have had a stroke increased almost twofold
between 1990 and 2016. [2]
To recover from a stroke, rehabilitation is crucial. Usually, the patients go to a therapist,
where they learn to perform exercises and are later asked to exercise at home. At-home
rehabilitation is problematic, as it is difficult to ensure that patients perform exercises with
correct form and that they stick to their exercise schedule. Moreover, the therapists do
not have any data on how much the patients improve.
A system using novel IoT technologies might help with monitoring patients’ at-home exer-
cise sessions and their motivation. In such systems, it is important for therapists to gather
both historical data about patients’ performance and real-time information when patients
exercise. Moreover, the system should display data from multiple patients, display predic-
tions to help form further decisions, and be generic enough to support a variety of exercise
types and data sources. [3] The development of human-digital-twin (HDT) could help with
patients’ motivation and customization of exercise sessions. [4]
This project’s aim is to create a dashboard that provides those functionalities to the ther-
apist.
1.2
ReHyb Project
The ReHyb project aims to develop a hybrid upper-limb exoskeleton that will have sensing
and actuation capabilities that will allow for the creation of a digital twin of the user. It is
founded by the European Union and has partners in 11 universities in 6 different countries.
DTU is responsible for creating the following parts: the therapist dashboard (this thesis),
the patient’s app, the app for the person’s partners (relatives, children, that could be of
help during rehabilitation), and the backend that aggregates this data. [5]
There are many components to the project. Exoskeletons will be used to gather data
about users’ movements, and some models will be able to assist with the movements as
well. Each patient will have access to their patient’s dashboard, which will display basic
information about their progress, see their schedule, and perform exercises. This dash-
board will show the patient’s digital twin. Third-party dashboards allow sharing patients’
data with their relatives or other people who could play an important role in the patient’s
recovery and motivation. The therapists will be able to manage their patients, plan their
exercises, and display their data on the therapist dashboard, which is the main focus of
this thesis. The patients will perform exercises with the help of serious games. Games
whose main purpose is other than pure entertainment. In the context of this project, the
games are connected to the exoskeletons. The movement of the exoskeleton controls
the game, which improves the patient’s motivation to perform the exercises.

HTML
Hypertext Markup Language (HTML) is a language defining the structural elements of
websites. Originally, it was indented to define the website’s visual using HTML as well,

<element attribute="value">content</element>} or <element/> for elements with

• <div/> - a block
• <span/> - inline block
• <p/> - paragraph

Generally, some of these elements can be replicated with other blocks and CSS, but that

2.1.4
Web workers
Web Workers are a feature of modern web browsers that allow JavaScript code to run in
the background without blocking the main user interface thread. This allows the execution
of computationally intensive operations without blocking the user interface. This can be
useful for web apps with a lot of data processing.
Communication between the main UI thread and web workers is achieved through mes-
sage passing. The main thread can send messages to a web worker and vice versa,
allowing them to exchange the necessary data.
Their limitation is the inability to update the user interface. [15]

2.2.1
HTML + CSS
The most basic solution for drawing graphs would be to create them using HTML and CSS.
With this approach, each part of the visualization would be a DOM element, making the
implementation of click and hover handlers trivial. Moreover, it makes the development
very simple, as it allows for the use of all of HTML & CSS advanced features, such as
grid. A major drawback of this approach is the difficulty of drawing elements that are not
rectangles or circles, even though that can be achieved using CSS transformations.
2.2.2
SVG
SVG - Scalable Vector Graphics format is an XML format that can be included in HTML
and is supported by modern browsers. It allows defining custom shapes using elements
such as path, polygon, rectangle etc. These elements are part of the DOM, so it is
easy to define event handlers similarly to HTML. Its drawback is that the SVG format
allows specifying only elements absolute or relative positions, which makes the creation
of responsive elements very difficult. [16]
2.2.3
Canvas
Canvas is a HTML element, which provides a various APIs to draw onto the canvas.
2d Context
The simplest canvas API is the ”2d” context API, which can accessed by calling:
canvas.getContext("2d")
This API provides methods such as fillRect(...), arc(...), lineTo(...), which are
very performant. The drawback of using canvas for 2D visualizations is that elements
are not DOM elements, and therefore handlers such as onHover and onClick have to be
implemented by hand. I.e., by taking the position of a mouse on every move and checking
whether certain elements are focused by running some ”point-in-polygon” algorithm. An
example library that could take this approach is Chart.js. [17]
WebGPU & WebGL
For 3D graphics, there are currently two different APIs that can be used - WebGPU and
WebGL.
canvas.getContext("webgl2")
canvas.getContext("webgpu")
These APIs allow developers to run fast, GPU-optimized code in the browser. WebGL is
the older and more supported API, whereas WebGPU is very modern (made available in
the default Chrome release in April 2023), has more features, and is more performant.
[18]
Three.js
To simplify working with complex GPU APIs, a very popular library THREE.js
provides various helper methods for working with and manipulating 3D objects while using
either WebGL or WebGPU in the background. [19]
Performance of SVG, HTML and Canvas

between drawing a lot of HTML divs, SVG elements, or to a canvas, a small benchmark

second to finish the rendering, whereas canvas was consistently at about 150 ms. To
make a proper benchmark, multiple factors, such as HTML attributes and styles, different
shapes, different browsers, operating systems, and devices, would have to be taken into

HTTPS
To ensure that the communication between client and server is private and secure, it is
common to use https - HTTP over TLS. To establish a connection, the following steps
need to be performed.
• The client requests the server’s public key and certificate.
• The server presents its public key and certificate.
• The client verifies the validity of the certificate.

2.3
API communication
There are many different approaches to sending data over HTTP.
• SOAP
– SOAP (Simple Object Access Protocol) defines the format of messages that
are sent over HTTP between services. A SOAP service would often provide a
Web Services Description Language (WSDL), which can allow code generation
on the client.
– SOAP is a ”single-endpoint” protocol, meaning that the SOAP service exposes
only a single URL, to which SOAP commands are issued using a POST re-
quest.
– Its main disadvantage is its overall complexity and verbosity due to the use of
XML. [22]
• REST
– The definition of REST is often confused and mixed up with other terms, but a
brief summary will be attempted.
– REST - (Representational State Transfer) - is a software architecture pattern
for designing web services. It requires the following constraints:
* Client-server communication - For websites, the web-browser would be
the client.
* stateless communication: No information is retained on the server between
requests.
That means all information (such as which user is currently
logged in) needs to be included in the request so that it can be understood
on its own.
* caching - Responses from the server must be defined as to whether they
can be cached or not.
* layered system - In REST APIs, the request can go through multiple layers,
such as proxies and load balancers, while both the client and the server
should not have the ability to recognize these layers.
* code on demand - Server responses can optionally include code to execute
on the client.
* uniform interface - This constraint defines four more additional constraints
on the design of the API.

∙Resource-Based - Resources must be identified in requests, for exam-
ple by URL for web services.
∙Manipulation of Resources Through Representations: The client should
be able to modify a resource that it already holds if they have sufficient

∙Self-descriptive messages: The message should include information

– HTTP methods: The HTTP standard defines methods that perform opera-

multiple network calls into one.
It also provides a strongly typed schema and tools such as graphiql, which makes

2.4
General web application development approaches
There are a few different approaches to web application development. The following
section contains a comparison of the most common ones.
2.4.1
Server-side rendering
In the historically oldest approach, every single web page is rendered on the server. Every
time the client (i.e., web browser) requests a page, the server has to take the HTML for
the desired page, fill in the dynamic values, and send everything to the client.
Server-side rendering usually uses JavaScript sparingly or not at all.
There are a few advantages to this approach:
• Fast initial page load: great for search engine optimization and user experience for
static websites.
• Simple: There is no need to handle state changes on the client side, which reduces
complexity and subsequently the number of bugs.
10
Implementation of dashboards for rehabilitation systems
==PAGE:19==
• No JavaScript: Not using JavaScript can improve security, platform support, and
accessibility.
However, in recent years, most websites have abandoned this approach due to the fol-
lowing disadvantages:
• A reload is necessary for each page.
– This problem can be demonstrated in the following example: Let’s assume we
have a website with two pages, Home and Detail, both of which share the same
navigation menu. Now, each time the user navigates one of those pages, the
server will have to send the code for the navigation menu every single time,
even though the component is the same and already present on the client.
• Non-interactive
– To get the result of an action, server-side rendered websites need to yet again
reload the whole website. This causes an unpleasant user experience because
the user can perform only one action at a time, often followed by an unpleasant
flicker caused by the page re-render.
Examples of server-side rendering frameworks are: Spring MVC [26], Ruby On Rails [27],
Laravel [28], Flask [29]. These frameworks usually use the MVC pattern with a template
processor, which allows the simple injection of dynamic variables into HTML templates.
[30]
2.4.2
Client-side rendering
Client-side rendering web frameworks, also called single-page-applications (SPA), work in
a fundamentally different way as opposed to server-side rendered ones. The server sends
an empty HTML placeholder element and JavaScript files containing the application. [30]
The received JavaScript code then dynamically updates HTML, styles, and URL paths.
This comes with various advantages in comparison to the server-side rendering approach.
• Minimal loading after the initial page initialization
– After the page loads, all that is required is to retrieve dynamic data, which is
more efficient than loading whole pages.
• Interactivity
– When a user performs an action, the page does not need to be reloaded, and
only the relevant part of the UI is updated. This allows users to perform multiple
actions simultaneously waiting for the previous actions to finish.
However, this approach also has its disadvantages.
• Longer initial page load
– Loading and executing the JavaScript file, which includes all the application
views and business logic, can take some time, making users wait. This can
also badly affect search engine optimization.
• Additional complexity
– With application state management happening on the client, error handling,
loading, navigation, etc. have to be implemented. These would be traditionally
handled by the browser itself. For example, many React applications suffer

from artifacts such as infinite loading indicators or no error handling because
the developers forget or do not bother implementing them.
• Difficulty testing
– When a server-side rendered application crashes, it’s usually very easy to get
notified about the crash or find it in the logs. On the other hand, when a SPA
crashes, additional logging mechanisms need to be implemented to notify the
developers.
• Accessibility
– SPA frameworks don’t enforce accessibility conventions as strongly as server-
side rendered websites, which can lead to poorer accessibility for impaired
users.
The most popular SPA frameworks are currently React, Vue and Angular. All of which use
a different architecture:
• React [31]
– Splits the app into small, self-contained, re-usable components. It is common
for a component to contain its styles, business logic, and HTML in a single file.
This approach was a source of controversy when React was first introduced
because it broke existing ideas of how apps should be structured.
– Updating state of a React component causes the whole component to reload.
• Vue [32]
– A Vue component consists of a model (the data), a view (HTML and CSS) and a
ViewModel. The ViewModel uses data-binding so that the data is synchronized
between the model and the view.
• Angular [33]
– Angular uses a similar approach to Vue, when it comes to data-view synchro-
nization. The main difference is that Angular is a highly opinionated framework,
meaning many decisions about how to structure the app are made by Angular,
with the disadvantage of less flexibility. Moreover, it provides a set of standard
components for common tasks such as routing, animations, and international-
ization.
2.4.3
Server-side-rendering - Hybrid approach
To take the best of both worlds, in recent years a new group of frameworks appeared.
These frameworks first generate the HTML on the server - just like server side rendering
frameworks - and then inject the client-side code into the loaded HTML. It has the following
advantages: [30]
• Fast initial loads
– Since the HTML is pre-rendered on the server, the user does not have to wait
for the JavaScript code to load and execute. This can also help with SEO.
• Interactivity
– When the client-side JavaScript loads, the website gains all of the advantages
of SPAs.

2.5.5
State management
State management is a big concern when developing client-side rendered websites. His-
torically, server-side rendered websites had relatively straightforward state management
since all the state was stored on the server, with only login information saved in the
browser. Loading and error states were handled solely by the browser. Now, however,
the client - the browser - is responsible for managing the state. This presents a number
of challenges, such as how to share the global state between components and how to
modify it.
Similarly to the styling problem, React itself provides a solution that does not scale to
larger projects, and then there are several competing libraries. In this section, the most
popular approaches will be compared.
Types of state management solutions
First, let’s consider a web app that simply displays data from the server. An example
of such a web app would be a news website, a housing website, and even the ReHyb
therapist dashboard. For such websites, there isn’t that much to store globally, except the
authentication state, the user’s website settings such as language and color theme, and
a cache for server responses to avoid unnecessary reloads. All changes in state usually
come from the server, and not much is calculated directly in the browser.
A second type of website would be an app that works offline first, for example, Figma,
Google Docs and Draw.io. These applications have different requirements for state man-
agement solutions because all actions can trigger a set of calculations that affect the
change of the global state in more complex ways.
State management in React
The state primitive in React is the setState hook. Most of the time, it is beneficial to keep
state scoped to the component in which it is set. Sometimes, more than one component
needs to share state. The basic solution is to lift-the-state-up. This technique includes
putting the state in the top-level component, in which the children share the state. Then,
the state and setters for the state are passed to the children in the form of props.
That solution works, as long as the components sharing the state aren’t too far apart. For
example, let’s assume the app has a dark mode setting. That way, all components should
be able to access this information to know how to render themselves. We could pass the
dark-mode property through the whole app, but that would be very verbose and difficult to
work with. For that reason, React provides a useContext hook, which allows data from
the parent to be accessed in all of its children.
However, when we want to change the data in the children as well, it does not work
Implementation of dashboards for rehabilitation systems
17
==PAGE:26==
very well. First, we would need to pass the setter in the context, which requires unclean
TypeScript overrides, and calling the setter from the children causes a re-render of the
whole tree, which is not good for performance.
Redux
The most popular state management solution is the Redux library. In a nutshell, the state is
saved in a so-called ”store”. This store is immutable and accessible from anywhere within
the React app. The change in values in the store is achieved through the combination of
actions and reducers. Action is a simple object that includes the action name and optional
data values, and reducer is a function, that takes two parameters. The current state and
the action. Then, it returns the new modified state. The advantage of this is that all state
changes are trackable, and it is even possible to ”travel back in time” using the Redux
devtools.
The drawback is its verbosity, especially in combination with TypeScript and the lack of
solutions for handling request states. However, these problems were eliminated by an
additional library: Redux-toolkit. This library provides functions for request handling and
also simplifies the creation of actions and reducers. [42]
Zustand
Zustand is the second most popular state management library. Just like the Redux-toolkit,
it provides an easy way of creating actions that change global immutable state. Its main
advantages are minimal setup and overall simplicity. Unfortunately, it does not have any
support for request handling, and therefore all of that must be written by the programmer.
[43]
TanStack query
TanStack query provides a solution for caching requests, handling re-fetching, errors, and
loading state by providing convenient hooks. The library does not provide any solution for
handling any other global state. [44]
SWR
SWR very similary to TanStack query, but provides fewer features in return for a smaller
size. [45]
MobX
MobX is a state management library inspired by the model–view–viewmodel frameworks.
The state is held in a class, whose instance is later observed in a React component
wrapped in an observer function. Yet again, this library does not provide a solution for
handling request states. [46]

There are multiple ways to include third-party libraries in a project. The most basic is the
addition of the library URL in the head of the HTML file. For example, the following code
can be used to include the Rambda.js [56] library:
<script src="//cdnjs.cloudflare.com/ajax/libs/ramda/0.20.0/ramda.min.js"></script>
However, this approach has a major drawback. We are not in control of the server, so
when it goes down, our website will stop working.

We can improve on it by downloading the library and including it in our code project:
<script src="libraries/ramda.min.js"></script>
However, this still isn’t perfect.
• We have to manually update the library when a new version comes out.
• The whole library will load, whereas we might only need a few functions.
• Library might use features not supported by some browsers.
To solve this problem, we can use a package manager program. First, we specify the
libraries and their versions in package.json file. Then, the package manager locates
these libraries in a package registry, downloads them, checks for vulnerabilities, and
recursively downloads dependencies. Then, the dependencies can be included in the
source code just like the rest of the codebase; therefore, all optimizations mentioned in
the previous section will be applied. Moreover, the package manager will make it easy to
upgrade all dependencies and also show a notification when vulnerabilities are detected.
For example, we can include the Chart.js library by putting it in the package.json and
running the package manager’s install command.
"dependencies": {
"chart.js": "^4.2.1"
}
Package registry
When we run the installation command, the package manger has to download the de-
pendencies from a server - the package registry. The default package registry for
JavaScript apps is npmjs.com [57], however, it is possible to host and use private reg-
istries as well, for increased security.
Package managers
The most popular package managers in the JS ecosystem are npm [57], pnpm[58] and
yarn[59]. Their job is essentially the same. Look at the required dependencies, down-
load them, and include them in the project. The challenging part is handling duplicate
dependencies and sub-dependencies without naively downloading the same files multi-
ple times. PNPM seems to have solved the problem in the most efficient way by using
hard links and symbolic links, which allow for the sharing of libraries between projects and
dependencies.[60]
Let’s take a look at the previous example of including chart.js and see how the package
managers handle the dependency.
First, we initialize the package.json file using the npm init, pnpm init and yarn init
commands. Then, we add the chart.js dependency. Upon examination of the directory,
we can see that the node_modules folder looks the same for both yarn and npm. It con-
tains a folder with chart.js code but also code from @kurkle library, on which chart.js
depends. That is not ideal, because now our project could use code from the @kurkle
library without being explicitly dependent on it.
Now, looking at the contents of node_modules created by pnpm we see only one directory,
chart.js, which is actually a directory alias. This folder then points to a hidden .pnpm
directory in the project folder, which contains the files for the libraries. This is benefi-
cial because now the programmer cannot use libraries without explicitly adding them to
package.json. An additional benefit of using pnpm, is that instead of copying all depen-
dencies, the contents are saved in a global store and then hard-linked to the project, so if

disk space.

Docker is a tool that allows packaging an application and all of its dependencies into a
single unit called a ”container”. The created container can be subsequently run on any

By using Docker, developing, testing, and deploying applications becomes easier be-

Short introduction to Dockerfiles
Docker files (simply named Dockerfile) are files that specify the contents of each Docker
image. The basic instructions are as follows:
FROM node:19-alpine
The FROM specifies the parent image of the Docker file, which usually is some linux distri-
bution, plus some libraries. In this example, we use the alpine Linux distribution with the
nodejs library, which is required for building React applications.
RUN npm install -g pnpm
Another important Docker command is the RUN command. Using this command simply
runs the program written after it. The important thing to note is that Docker images are
made out of layers, and the RUN command creates a new layer of the image. This gives
us two advantages.
First, the layers can be cached and reused if none of the layers above them change.
Therefore, only the initial build of the image will install pnpm. Second, layers can be shared
between images.
COPY package.json ./
The copy command simply takes the files in the current directory and copies them to the
Docker image. This creates a new layer just like the RUN command.
CMD ["npx" "http-server" "dist" "-p" "0"]
The final command to note is the CMD command. There can be only one CMD command per
Dockerfile, and it specifies which code should be executed after the container launches.
Optimizing Dockerfiles
Many tools for automated testing and deployment depend on the speedy creation of
Docker images; therefore, a few things should be considered when creating them.
• Size of the base image
– There are many dockerized Linux distributions: various versions of Debian,
Ubuntu, Alpine, and more, which vary greatly in their sizes. Choosing the right
parent image can reduce the size of the resulting Docker image by a significant
amount.
• Ensuring proper caching of layers
– It is common for applications to have some kind of package manager that takes
care of downloading dependencies. To ensure that we don’t need to download
dependencies again if they don’t change, we can separate the addition code
and dependencies into different Docker layers. For example, for JavaScript
applications, we first copy only package.json and run the pnpm install com-
mand. Then, we run the COPY command to copy the rest of the code. This way,
if dependencies aren’t changed, Docker can reuse the layer from the previous
build.
• Separating build image and run image.
24
Implementation of dashboards for rehabilitation systems
==PAGE:33==
– Docker supports multi-stage builds. Often, it is the case that the tools re-
quired for building the application are not required for running the application.
In multi-stage builds, the result of the build phase can be transferred to another
image, which can greatly reduce the size.
– This can be useful because, in most workflows, the server for building the ap-
plication is different from the server running the application. The server that
runs the application needs to download the Docker image to run, which can be
hundreds of megabytes and take a long time.

result, they claim that both patients and therapists had a positive attitude toward using
the technology. The therapist emphasized their ability to set personal goals for patients
and see their progress based on the gathered data. As with other studies, some therapists
would like to have more context for the presented data, such as seeing normal values for
a healthy population. [72]
More insight was obtained in the ArmSleeve project, which consisted of a wearable device
that measured arm data and a dashboard for the therapists. In their dashboard implemen-
tation, they have displayed the following data: number of movements, time spent exercis-
ing, quality of the movement, range of motion, and postures of the arm. These data were
presented in an overview of the patient, timelines showing the patient’s progress, range
of motion visualizations, and heat maps showing arm positions. The therapists found
the timeline showing the patient’s exercises useful. The range of motion visualizations
were useful as well, as the therapist could see a discrepancy between the patient’s move-
ments in exercise and daily life. The heat-maps were useful for assessing the natural
arm positions that were preferred. Therapists expressed the need to know the contextual
information of what action is currently being performed with the gathered data. [73]
Another similar project was GEAR - GamE Assisted Rahabilitation system. In this project,
a combination of a wearable device and a smart-phone game was used to gather data
about the patient’s rehabilitation. In the therapist’s dashboard, the data was presented
in the form of a time series at various levels of detail - all sessions, single sessions, and
a single time point. Once again, the ability to track the patient’s progress over-time was
emphasized as very useful. [74]
Finally, a systematic review of 69 papers concluded, that real-time visual analytics for
rehabilitation require the following features: real-time, allow the clinician to monitor many
patients at many different locations, have a predictive component that could aid decision-
making, be independent and not created for one specific rehabilitation system, be able
to handle different types of data, the visualizations should be interactive, and be simple

Existing work conclusion
Based on the work already done in the field, there seem to be two reoccurring patterns.
Therapists need context for the gathered data and the time series data showing patients’

serious games. Time series data showing patient data will be shown for exercises. In
ReHyb, data is not gathered continuously, as the exoskeletons cannot be worn all day.

Handling reuqest states
The most straight-forward way of managing the state of a
network request would be to use the useState hook in the combination with the fetch
function. This solution, albeit simple at first sight, makes implementation of error prone
request fetching quite difficult. The following information about a request needs to be
included:
• data - response if the request succeeds
• isLoading - whether the data is loading for the first time
• isFetching - whether the request is loading but the data from a previous successful
request is already present
• error - in the event of an error, the error must be saved in the application state
Moreover, it might be desirable to refetch some requests at intervals; the results of re-
quests might be cached for faster performance, and in case of an error, we might want
to try re-fetching the request again. All of this functionality would be quite challenging to
write by hand, with all edge cases handled properly.
Handling request type checking
To properly display the data from the server in the
front-end application, the front-end needs to know the format in which the server returns
data. Yet again, there is a simple solution and a more advanced but more error-prone
solution.
The simple solution is to write out every single request and response type by hand. This
works for a small project and in cases in which the developer works on both the server
and client simultaneously. With this approach, changing the server code does not trigger
any warnings on the client, and the only way to detect the change is either through good
communication between the developers or by letting the client app error out.
To avoid these problems, we can use an OpenAPI code generator tool to generate a Type-
Script client from the OpenAPI documentation of the server API. The best solution might
be generating the documentation from the server code, which is possible through various
libraries available for most programming languages. However, in the ReHyb project, the
server did not implement any such library, and therefore we had to keep updating the API
documentation by hand, which can lead to errors.

Authentication
Each therapist needs to be authenticated when they log in to the app. The most common
way to perform authentication in a React app is to retrieve the authentication token from
a login request and save it in the application state and the browser’s session storage, or
local storage. Another option is to use cookies.
Cookies were chosen as the authentication method as they are more secure, thanks
to the secure and httpOnly flags[79]. The secure flag makes cookies available only
on https connections, preventing man-in-the-middle attacks, and the httpOnly prevents
JavaScript from reading its contents. This, however, prevents the React application from
knowing whether the user is logged in or not on the initial page load. We have solved

4.8.1
Joints visualization
To visualize the range of movement for the elbow, a set of points on a sphere was created.
First, the points used spherical coordinates: SHFE for azimuthal and SFE for polar. These
points were translated to Cartesian coordinates using the following equation:
x = r sin θ cos ϕ
y = r sin θ sin ϕ
z = r cos θ
Finally, the shoulder position vector was added to shift the sphere’s origin to the shoulder.

The hand’s reachability is calculated similarly. SIE is used as the azimuthal coordinate
and EFE as the polar. Then, the calculated points need to be rotated along the X and Y
axes and shifted by elbow location to put the sphere’s origin at the elbow.
4.8.2
Reachibility visualization
Implementation of the reachability visualization is a bit more involved and uses the follow-
ing steps:
1. Generate all reachable points
• First, four nested loops go over all the possible combinations of SHFE, SFE,
SIE and EFE to generate all points that are reachable by a healthy person,
based on the hard-coded min and max values of each variable.
• Then, the same function is used to generate points reachable by the patient.
• In the current implementation, the accuracy is set to 10°. This still generates
around 80 000 points. This is because each degree of freedom has possible
values between 0°and 180°. We can estimate the number of points using the
following equation:
(180/10)4 = 104976
The actual number is a bit lower because not all joints can move in the 0°to
180°range.
2. Merge points
• The next step is to reduce the number of points so that they can be rendered.
This is done by taking all the coordinates of a given point, rounding each of them
to some chosen accuracy constant, and putting them into a set to eliminate
duplicates. For example, if we set the accuracy constant to 0.5:
– (1.2,1.4,1.9) maps to (1.0,1.5,2.0)
– (1.1,1.3,2.1) maps to (1.0,1.5,2.0)
– (1.0,1.6,1.8) maps to (1.0,1.5,2.0)
In the current version, accuracy is set to 0.4, which reduces the number of
points to 786, which is low enough to render.
3. Draw points as cubes
• Then, the points are drawn around the person as cubes, with the side length
of the accuracy constant. As can be seen in 4.8b, this naive approach is prob-
lematic since the outer cubes make the inner cubes almost invisible.
4. Drawing only the necessary cube side
• To overcome the issue of inner cubes being invisible, we can render only the
cube’s side on the border. To do this, the for-cycle that draws the cubes first
checks for neighbors on all six sides before drawing only the necessary wall.
Algorithmic improvements
It is clear that this algorithm is not very efficient, and the visualization makes use of the
performance of modern browsers. The approach of generating all points and then merg-
ing them into groups could be replaced by calculating whether each group is reachable
using inverse kinematics. This would be a bit more involved and would require the imple-
mentation of an inverse kinematics solver or the use of some solver library.

4.10
Image upload
Normally, images would be uploaded to a special endpoint on the server, usually using
HTTP multipart request, so that larger files could be uploaded gradually. This was not
implemented on the backend, and the provided endpoints only supported a string value
for the image. The two choices considered were:
• Use a third-party image upload server and pass the image’s URL in the form. This
would enable proper upload of high-quality images, but it would be a security risk
as the images could be accessible from the internet. Moreover, having to manage
multiple servers would make deployment more difficult.
• Encoding the image as a string - After uploading a file, a browser provides a con-
venient class FileReader with the method readDataAsUrl. This method allows
converting an image into a string. For example, a PNG image will be converted to a
string looking like this: data:image/png;base64,iVBORw0KGg...OzzPQAAAABJRU5ErkJggg==.
The main disadvantages of this approach are that:
– Uploading too large images onto a server without multipart might fail.
– Storing images in a database is generally not recommended. File storage is
cheaper and makes image processing easier.
The second option was used due to its simple implementation. In the future, it would make
sense to have a separate image server.

Then, the ”at home” mode has widgets ”Events”, ”Patient Activities” and ”Patient today”.
• Events - The idea behind this widget was to have a synchronization with the ther-
apist’s calendar. A basic proof-of-concept was implemented in Calendar.ts file,
which parses events from the therapist’s calendar. However, it is not clear what
kind of events would be displayed, and therapists at different clinics might even use
different kinds of calendars (Microsoft Teams calendar, Google calendar) and plan-
ning tools, so proper parsing might be very difficult. Moreover, the designs include
the option to edit events, which is not technically feasible for some of the calendars.
• Patient activities - This widget is supposed to show the recent activities of users, but
it does not seem to be implemented on the backend.
• Patient today - Should be showing patients doing exercises today. However, since
the calendar’s days are selectable, it could also be used to show which patients
have exercises scheduled on a given day.
Conclusion
I believe that the current design of the page introduces a great deal of complexity, which
should be reduced by removing the dual view and most of the widgets.

Performance testing
A common complaint about ReactJS apps is that rendering a web app using JavaScript
(instead of getting HTML directly) can be slower. That’s because the browser first down-
loads the JavaScript code and then needs to run it to create the HTML and CSS, which
can take time. Therefore, it is necessary to measure the performance to ensure a good
user experience.
One way to test the performance of a website is to use Google’s Page Speed Insights
(PSI). It is a tool that loads websites, runs various analyses, and finally gives a score for
categories such as performance, accessibility, best practices, and SEO. For the ReHyb
project, SEO is not relevant, as the website does not need to be optimized for search
engines. Accessibility is relevant only partly, since therapists do not usually need acces-
sibility features.
PSI also gives a score for mobile devices, which are not relevant for the ReHyb project.
However, most of the layouts are responsive and could be made mobile-friendly with little
effort.
The PSI performance score is calculated based on the following metrics:
• First Contentful Paint (FCP)- The time at which the first text or image is painted.
• Total Blocking Time - Time from FCP to page becoming interactive.
• Speed Index - How quickly is the page content visibly populated.
• Largest Contentful Paint - Time for the largest text or image to get painted.
• Cumulative Layout Shift - Measures how much elements shift on page load.

6
Discussion
The purpose of this thesis was the implementation of the therapist dashboard for the
ReHyb project. Based on the results, the front-end web app includes all components that
were specified in the requirements, performs well, and works in all major browsers and
resolutions.
Performance
The performance results with PSI scores over 90 indicate a good user
experience. The reason for the scores not being 100 is the nature of SPA (single-page-
applications). The website is created using JavaScript (not HTML), so a small delay is
inevitable. Using an SSR (server-side rendering) solution (such as Next.js) would in-
crease these scores, but it would also increase complexity, and it is not suitable for this
type of application, as discussed in the section 2.4.4. [81]
The low score of 63 on the Patient data tab does not indicate a good user experience.
The profiler points to Plotly.js as being the culprit. It shows that the majority of the time
is spent executing Plotly.js code after it is loaded and drawing the graphs. Searching
through Plotly.js’s documentation, there does not seem to be a way to improve the
performance. Moreover, Plotly.js examples themselves have similar issues. A possible
future work could include replacing the Plotly.js library with another, more performant,
chart library.
Survey
The results of this survey are only indicative, as a survey has many biases.
People who are filling out the survey, whether friends, colleagues, or classmates, will be
biased towards more positive feedback. Moreover, none of the users will be able to use
the app as intended, as the required infrastructure is not in place. (such as a working
exoskeleton, a finished backend, prediction models, etc.) In the industry, web app testing
is done on real users, analyzing their behavior and finding better implementations using
techniques such as A/B testing. [82] This is unfortunately not possible for the ReHyb
project.
Based on the feedback on the UI, the most problematic were the 3D visualizations. This
was expected, as no exercise data were retrieved from the server, so the visualization
serves more as a proof-of-concept. In the future, the visualization should show data from
the last few, if not all, exercise sessions, whereas currently only two trials are available.
A problem with getting help from the therapist was that during Panpan’s design evaluation
session, we found out that therapists are not able to share screens due to security. More-
over, only 3 out of 5 therapist were available to give feedback on the 3D visualizations.
However, the results indicate that such visualizations might be useful. Still, features like
these should be tested on real usage data rather than surveys.
Integration with server
Only a subset of features were integrated with the server. The
problem was that to successfully integrate a feature, it needed to be 1. designed, 2.
API specification needs to be created in collaboration, 3. integrated in the back-end, 4.
integrated in the app. Usually, between step 3. and 4., bugs are found, and the process
needs a few iterations. Due to Panpan’s and Weijun’s thesis ending at similar times,
there was no time left for integrating the last design changes. Moreover, the design was
Implementation of dashboards for rehabilitation systems
49
==PAGE:58==
rapidly iterating over the whole course of the project, which meant a lot of time was spent
implementing components and APIs, which were later abandoned.
Visualisations
Components for visualizing the data were implemented; however, it was
not often clear what exactly the components should be displaying, where the data was
coming from, or who was responsible for the data source. These problems stem from
the fact that there are project partners with whom we are not in direct contact, and a
lot of those components are still in development. We did not even have access to the
exoskeletons, which are the most important part of the project.
The 3D visualizations were implemented using three.js. This proved to be the right
choice, as the library is more lightweight than Unity. However, the testing data of the
exoskeleton movements did not match perfectly with the 3D human animation. For a
more accurate result, creating a custom rig of the human body might be required.
Future work
Future work should focus on cutting features to create a minimal-viable-
product (MVP). It would be beneficial to get the whole system working end-to-end with-
out any mocks. This could be done by creating first feature flags, which would disable
all features that cannot or do not have to be integrated, such as spasticity graphs, in-
home/at-clinic settings, and cognitive scores. Then focusing all effort on the most im-
portant workflow, which is planning an exercise, presenting the exercise to the user, and
finally displaying the data gathered by the exoskeleton after the user performs the exer-
cise.

7
Conclusion
The purpose of this thesis was the implementation and evaluation of a therapist dashboard
for the ReHyb project. On the client-side - the web app - all designed components were
implemented in HTML, CSS and JavaScript. An admin interface was implemented for the
basic management of clinics and therapists. A new way of visualizing joint movements in
3D was implemented.
Feedback on the user interface suggests that the web app works as intended and does not
have any significant issues. Based on the feedback on 3D visualizations, more work might
be needed to make the 3D visualizations more intuitive, but feedback on their usefulness
is promising.
There were two major limitations during development. The first was the rapid iteration of
designs, which caused implementations to be abandoned. This could have been avoided
by getting feedback more frequently and clearly communicating what is ready to be imple-
mented. From my personal experience in the industry, feedback from clients was given
every one or two weeks. For this project, we only held two feedback sessions. The sec-
ond issue was that a lot of crucial components for development were not ready at the time
of implementation.
Future work should focus on integrating the system with the other components. Specifi-
cally, the data from the games, the movements from exoskeletons, and the data from the
patient’s and contact person’s apps.
