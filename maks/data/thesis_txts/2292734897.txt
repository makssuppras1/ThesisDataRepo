Executive Summary
The goal of this thesis is to develop a proof of concept pluggable solution that
works for any Drupal project. It is especially useful for Drupal deployments that
consist of many drupal installations, typically referred to as Drupal Multi Site
Installations. The problem with Drupal Multi Site Installations is the fact that
each of them does not share databases. This in terms means that using system
cron to execute background tasks, endures a huge overhead, since it needs to
connect to each and every installation, and invoke the Drupal cron. This thesis
will analyze and develop a proof of concept solution that handles the above
problem of background tasks, and at the same time allows easy integration into
a Drupal environment, not seen before in the Drupal community.
To avoid
reinventing the wheel, so to speak and ensuring future scalability, the proof of
concept solution will rely on a state of the art queuing backend system.
The proof of concept was developed for ChurchDesk and many of the decisions
made in this project come from discussions with the ChurchDesk team, and take
into account how the proposed solution would integrate in an optimal way. This
thesis makes an in depth analysis of the infrastructure of ChurchDesk, and the
application design, both in terms of the product right now, and the product in
the future.

Chapter 1
Introduction
In this chapter the company ChurchDesk is introduced for whom this project
is made in collaboration with. This chapter also explains the motivations for
doing this project and why it is needed by ChurchDesk. A list of goals will be
described to allow the reader to understand the goals needed for this project
becoming a success. Furthermore the development process will be described to
explain how the project has been developed and how the collaboration works
between ChurchDesk and the author. Lastly an overview of the report structure
will be given.
1.1
ChurchDesk
ChurchDesk is a young Danish startup that aims to modernize the workﬂow of
churches. The aim of the startup is to bring modern technologies to churches,
which in the end will empower the employees in their daily work. The focus is
on making one click become many, by automating key aspects of the churches
workﬂow. With more than 500 churches and 7000 active users, ChurchDesk is
oﬀering a software as a service (SaaS) model, which means that customers are
licensed on a subscription fee basis.The application oﬀered, makes it possible for
the church to manage employees and resources, and at the same time automate
the process of maintaining the churchs website, newsletters, and automatically

export events to 3rd party services: Kirku and the Danish Church Ministry. The
company is still very young, but it is already the biggest player on the Danish
market. Furthermore, it is gaining traction in Germany and England.
1.2
Motivation
At present ChurchDesk is run on a PHP stack partly based on the PHP frame-
work Drupal. The application is available both on the internet, via a browser,
and on mobile as a native application for Android and iOS. A shift from tradi-
tional blog websites to applications, running on Drupal, creates new demands
that Drupal or PHP traditionally were not built to handle and handles very
poorly, due to the fact that operations are executed in a linear fashion. Web-
applications today are treated by users merely as locally installed applications,
and users demand high performance.The limitations that Drupal imposes on
the application running in Drupal, can be traced all the way back to the way
the language PHP is implemented. HTTP-requests in PHP can only run tasks
in serial execution, with no possibility of multitasking.This means that the user
has to wait for all the tasks to complete in the background, when interacting
with the application. An example of this could be operations, where the user
wants to save information that requires an API call to other systems. In this
case the user has to wait until all operations are done before being able to con-
tinue. Furthermore, the application handles business critical API calls, which
from time to time fail silently to a distributed log with no means of debugging
or re-running the tasks.
Looking into the future development of the ChurchDesk application, it seems
only natural to use a micro service architecture [Mica], which is also what the
current architects agree on. Breaking the application up into smaller services,
has many beneﬁts and disadvantages. One of the key points is that services
have to communicate via strong APIs or some sort of intermediate. This com-
munication could slow down the application, due to authentication and network
overhead, which is why the new architecture only emphasizes the need for a
strong queuing infrastructure and framework. The communication has to hap-
pen without blocking main tasks, but at the same time should be monitored to
make sure that jobs that are critical, eventually succeed.
Current queuing systems, built by the Drupal community, have many limita-
tions. The biggest limitation is that they do not support multi site deployments,
which is what ChurchDesk is using. Other problems are that they are imple-
mented in a way that does not follow standards or paradigms used in the Drupal
framework, which makes it harder for Drupal developers to adapt.

1.3 Vision
3
The motivation for ChurchDesk to implement this queuing infrastructure system
is that it will allow them to continue using their existing PHP/Drupal platform,
and at the same time add the following beneﬁts to the company:
Faster experience for the users, since they do not have to wait for tasks to be
completed.
Uniﬁed way of creating and handling background jobs for the developers.
Better auditing possibilities of background tasks.
Performance monitoring of background tasks, so developers can plan when
to upgrade infrastructure.
Makes tasks robust by making them recoverable from errors.
The
individual tasks can be retried, if they failed.
Secure. Sites should only be able to post tasks for itself. Data stored in a task
should only be accessible by the site that created the task.
Priority can be given to tasks that need a fast turn around time.
Scaling the application becomes easier, since execution of tasks can be easier
distributed across multiple machines.
1.3
Vision
Improve user experience by allowing developers to oﬄoad time sensitive tasks to
a queuing system that can guarantee execution of the tasks, within some deﬁned
time intervals. Thus greatly increasing performance, and helping ChurchDesk
provide a better user experience. A plug n play solution will make sure that
developers can continue to use the standard, Drupal functions, while executing
their tasks, making it easy to transition tasks to this solution.
1.4
Problem Deﬁnition
Rewriting the existing code base, as in switching to another more advanced
framework or architecture, is not possible, since ChurchDesk is constrained by

development resources, as it is a bootstrapped startup. Over time, components
can be moved into self-contained micro services. However, right now the problem
at hand is ensuring that the existing application performs. Furthermore, the
existing team has a lot of knowledge in the existing code base and the technology
used. This thesis will analyse how ChurchDesk can overcome these limitations
of the application, and study how a queuing system can be integrated in the
best possible way, lowering the time needed to refactor existing code, taking
scaling, future architectural changes and security into account.
Finally, the queuing prototype will be tested by refactoring some selected API
calls in the applications to get data on the performance improvement, and to
gain an insight to how the queuing system behaves in real life, use scenarios.
The prototype is evaluated by the documentation, ease of use and debugging,
by the backend development team at ChurchDesk.
1.4.1
Delimitations
This report will not try to recommend refactoring of large pieces of the code,
since the ChurchDesk developer team is resource constrained. There are nu-
merous bad design decisions in the application that this report will not go into
depth with.
The prototype developed will only be a proof of concept, and does not neces-
sarily contain all the features needed to implement it fully in the ChurchDesk
application.

Goals

1.4 Problem Deﬁnition
5
G1
Identify a state of the art queuing backend.
The queu-
ing backend should be future proof in terms of supporting
programming languages that ChurchDesk believes will be
integrated into the application in the future. Easily inte-
gratable in the current ChurchDesk infrastructure and have
a thriving community supporting it.
G2
Develop a proof of concept solution integrating with Dru-
pal and a selected state of the art queuing backend. The
solution should be implemented as a Drupal module so it
can be shared back to the Drupal community.
G3
The solution shall follow known design patterns such as
SOLID to ensure easy maintainability and it should be
possible to easily change parts of the solution, in case
ChurchDesk wishes to switch to a diﬀerent queuing back-
end.
Furthermore the patterns used by queueing back-
ends will be analyzed to fully understand their design and
whether they ﬁt into the ChurchDesk ecosystem.
G4
It shall be safe to operate the solution in a multi tenant
hosting environment.
Tasks shall only be allowed to be
started by the site itself and the solution shall include a
way to keep task data secret from other installations or
tenants.
G5
The proof of concept should support the developers at
ChurchDesk to use the queuing system in many diﬀerent
use cases, ensuring consistent code across components.
G6
The proof of concept should analyse and recommend what
infrastructural changes need to be made, to support the

When considering software reuse in software development, two kinds of strategies
can be described: Reference-reuse and fork-reuse [Col].
• Reference-reuse is when an existing library or software project is used
without doing any alterations to the existing code.
• Fork-reuse is when an existing library or software project is forked, which
means that a copy of the existing code is made. The new copy is isolated
from the original project. This means that changes in the original code
will not impact the current projects code in anyway.
Here, the software reuse strategies for some of the components in this project
are described:
The state of the art queuing backend system selected is a reference-reuse which
means that no alterations have been done to the code. The PHP packaging
system composer has been used to ensure easy maintainability of the code.
The proof of concept code itself, which handles the interaction between the
ChurchDesk code and the queuing backend, is packaged as a Drupal module,
which makes it possible for other companies or developers to reuse the project
as a reference-reuse or a fork-reuse.
ChurchDesks current version of Drupal is a fork-reuse since they have forked
the original Drupal project to make way for alterations they needed in the core
code, ﬁtting ChurchDesks needs.
Furthermore, when refactoring the current ChurchDesk use cases to run with
the queuing system, existing code is reused instead of redoing the subroutines
from scratch. A lot of thought has gone into the design of the queuing system,
to ensure that as little as possible should be changed in the original use cases
code. This will be described more in detail in the Design chapter 6.

In this chapter diﬀerent queuing backends will be analyzed and compared. Based

This is also stated in the Goals section G1.
When looking at the state of the art queuing backends, the following is expected:
• Strong integration with PHP which means that most of the Queuing Back-
end operations should be accessible from the PHP client library.
• Ease of installation. The queuing system should be easy to integrate in
the current ChurchDesk infrastructure. The Queueing Backends will be
rated on a 3-star scale (3 is considered best).
3 stars The servers already exist with the required software running to
support the queuing backend.
2 stars One server needs to be set up and the Queuing Backend software
needs to be installed.
1 star Multiple servers need to be set up and the Queuing Backend soft-
ware needs to be installed.

State of the Art of Queuing Backend Systems
• Strong community with many plugins. The Github statistics will be used
to determine the strength of the community.
• Open source code base to allow for modiﬁcations, if needed, both for now
or for the future.
• Future proof. See below paragraph.
• Possibility to prioritize tasks.
• Interface to monitor task status.
As [Woo03] notes Change is inevitable.
Applications change over time.
An
integration solution has to keep pace with changes in the application it con-
nects. To future proof the proposed queuing system, it is important to select
a queuing backend system that supports multiple programming languages, in
case ChurchDesk changes to a second programming language. The client library
needs support for PHP, Python and NodeJS. Python and NodeJS seems to be

3.1.3
Protocol
The protocol is based on the Event Driven design pattern. This means that
consumers subscribe to queues.
When a message arrives in the queue, it is
pushed to one consumer, which is ready. When the consumer receives from the
queue, it enters a busy state, while processing the message. When the consumer
is done processing, it responds with an acknowledgement (ACK), so the queue
knows that it is ready for the next message. Only when RabbitMQ receives an
ACK it will send the next task to the consumer.
3.1.4
Data Storage
RabbitMQ uses its own built-in data store. The data store maintains messages
in RAM to avoid IO penalties. If the system that hosts RabbitMQ runs out of
RAM, RabbitMQ will start moving some tasks to the disk to free up RAM. The
goal of the data storing algorithm is to keep everything in memory, and delete
them as soon as the messages have been processed, freeing up memory again.
In this way it does not matter that the disks are slow, since hopefully there
is enough RAM to buﬀer all the messages. RabbitMQ furthermore supports a
clustering option, where messages are mirrored to a number of slaves, ensuring
a high availability setup.

Resque was initially built by Github, and used internally for tasks [Hof].
The code is opensource and available on Github [Gitb]. The community is large,
and many plugins exist, such as resque-scheduler, resque-retry.

4.2 Asynchronous Queuing
43
4.2
Asynchronous Queuing
The Asynchronous Queuing pattern describes a problem where services interact
with each other synchronously. This can lead to many problems:
• Both services need to be available to communicate.
This can lead to
problems where the initiating services cannot continue because the other
service is not ready to communicate.
• Immediate response is required between the services. Which furthermore
requires the initiating service to wait for a response before it can continue
with its actions. Because the service has to halt the resources, its using is
locked and most often this leads to slow responses.
• Overload of services. Since services are expected to communicate in a syn-
chronous manner they cannot simply wait, handling a message. Instead,
they have to start to handle it right away, so an acknowledgement can be
returned to the waiting service.
Example: In use case UC.SGM the application has to wait for the mail service
responds before it can return to the user and conﬁrm that the message has been
saved. If the mail service is busy or unavailable, then the application would
have to wait until it hits a timeout which then triggers a useless error message
for the user. See ﬁgure 4.1 that shows how the two services are intertwined.

4.3 SOLID Pattern
47
4.3
SOLID Pattern
The SOLID pattern describes ﬁve basic programming principles for object ori-
ented programming. The principles were put together by Robert C. Martin in
the early 2000s. The principles assist the developer, when refactoring code, in
order to make it more legible and extensible. In the following section the ﬁve
principles will be listed together with a brief description [Sol].
SRP. The Single Responsibility Principle. This principle states that each class
should only have one responsibility. This is because classes that only carry out
one thing, are easier to test and debug [SRP].
OCP. The Open Closed Principle. The OCP states that every class should be
closed for modiﬁcation and open for extension. This means that future updates
can extend the classes without needing to modify its source code. Again, this is
very useful, since code changes to a class, result in tests having to be updated
along with other procedures, before code can be pushed back to production
[OCP].
LSP. Liskov substitution principle.
This principle is named after Barbara
Liskov, and describes that objects in a program should be replaceable with
instances of their subtypes, without altering the correctness of that program
[LSP].
ISP. The Interface Segregation Principle. This principle favors many small, ﬁne
grained interfaces instead of one large interface. Interfaces can help systems
become less coupled, but interfaces can also try to cover too many scenarios,
which in turn makes it hard to implement the interface, since most of the time
you would be forced to implement functions that had no relation to your class
[ISP].
DIP. The Dependency Inversion Principle. This principle ensures that high
level classes are not working directly with low level classes, since this would
create a tight coupling. Instead, the high level classes are using an interface
layer instead, when communicating with the lower classes. Since everything is
coupled with interfaces, it gets easier to refactor code, since the code only need
to adhere to the interfaces, and not necessarily everything connected [DIP].
==PAGE:62==
48

4.3.1
Impact
Even though Drupal is not an object-oriented project, it makes sense to develop
the proof of concept as an OOP. This is because, the nature of OOP makes it
easier to refactor parts or classes at a later stage. An example could be that we
wanted to connect to another backend queuing system, instead of the selected
one for the proof of concept. For the proof of concept solution the SRP, LSP,

it is even possible to switch between the implementing classes during runtime.
4.4
Priority Queue Pattern
When the number of tasks added from the applications or microservices becomes
higher than the queuing executors can handle within a reasonable time, you are
left with two options: add more executors or prioritize the tasks, before they are
added to the queuing system. Very often speciﬁc tasks can be identiﬁed as less
time sensitive than other tasks. By using priorities, the queuing system will be
able to handle periods with sudden spikes in the number of tasks being added
to the queuing system, by focusing its power on the important tasks [Micb].
The Priority queue pattern describes a solution where each priority is repre-
sented by its own FIFO queue. Every queue has its own set of consumers.
This solution makes it easy to add more executors or consumers, as visualized in
the ﬁgure 4.3. If one queue fails to uphold its promised turnaround time for the
tasks, adding more machines or more power to existing machines is a possibility.
4.4.1
Impact
The implementation of this design pattern can have various impacts on the
performance of the queuing system.
A simple and somewhat naive solution
would be to keep the queuing as an unordered list of tasks. When the highest
prioritized tasks needs to be fetched, the queuing system would search through
the entire queue. This results in a fast insertion time O(1), but a bad ﬁnd next
task time O(n).
Another implementation, which improves performance, would be to utilize a
specialized heap data structure, which ensures a fast insert and ﬁnd next time.

4.5 Event-Driven Messaging

4.5
Event-Driven Messaging
The Event-Driven messaging describes a pattern to notify subscribers about
new messages. Keep in mind that messages, in regard to this project, could be
the equivalent of a job or a task and the subscriber is a task worker. It has been
chosen to describe this pattern in a way that reﬂects how the queuing backend
RabbitMQ uses this pattern.
This pattern is the central part of RabbitMQ
but it should be noted that other design patterns are also used on top of this
design pattern in RabbitMQ such as Reliable Messaging [Erl09, p. 592] and
Asynchronous Queuing described earlier in section 4.2.
The pattern works by having an Event Manager controlling the queue of mes-
sages as seen in ﬁgure 4.4. When a message gets to the front of the queue,
the Event Manager will contact a waiting subscriber with the message so it can
be processed. For this to work the Event Manager keeps track of all idle and
working subscribers.
This approach requires the workers (subscribers) to subscribe for a given message
so the Event Manager knows, which subscribers it should contact.
4.5.1
Impact
This design pattern avoids the ineﬃcient polling based interaction, where con-
sumers constantly check if a new message or job is available as seen in the
design pattern Polling Consumer [Erl09, p. 599] . Instead the subscribers are
notiﬁed right away and they can commence handling the message. In systems
which have a high load of messages this becomes important since the overhead
of having consumers polling for messages at given interval can be signiﬁcant.
The solution will never be faster than the polling interval itself. A downside
to this pattern is the overhead of the Event Manager having to keep track of
all the subscribers and implementing a solution where newly started consumers
can subscribe for messages.

4.8 Polling Consumer

4.8
Polling Consumer
The design patterns describe how the consumer can ﬁnd out, whether there is
a message ready for processing. This can be seen in ﬁgure 4.6. The pattern
typically works together with the design pattern Competing Consumers. In this
design pattern the consumer continuously polls for messages from the sender.
When the message is processed, it polls for the next message. In this way the
consumer does not have to wait for the next message to arrive from the sender,
instead it can ask the sender, as soon as it is ready, to process the next message.
Relating this design pattern to a queuing system, the consumer and receiver
would be the worker, and the sender would be the queue that contains all the
tasks. Furthermore, this design pattern is used by the queuing backend system
Resque.
4.8.1
Impact
This can be used to ensure that the executor is not overwhelmed with messages.
Instead the consumer only polls, when the executor is ready to process the next
message. The downside is that the consumer blocks, while polling for messages
and consumes process time for messages that are potentially not there [Woo03,
p. Polling consumer]. On the other hand the queue can now be very simple
since it does not have to track its available consumers. This also means that no
discovery protocol needs to be implemented since all consumers ask the same
queue for messages.

processing. See ﬁgure 5.2.
To accomplish this, Drupal is manually bootstrapped, using the drupal_boostrap()[Com]

• Database credentials
• Domain name
• Environment variables
This can be done either by doing a http request to the custom bootstrap ﬁle,
which bootstraps Drupal and executes the task, or it could be implemented as
a PHP cli script that handles the bootstrapping and execution.

The component MQC is responsible for communicating with the rest of the

Every time a task needs to be created, it is done

6.1.3
LocalQueue, LQ
This component stores the payload of the task itself along with the task history.
This component makes sure that the queuing system follows the Messaging
MetaData pattern, described in 4.6, by storing the history of the task. This
means that we can see from the LQ, which consumer executed the task, and
how it went. By utilizing a local queue on top of the central queuing backend
we are able to, easily verify that the task was created by the local installation

can ensure that only the installation itself can access the task payload, which
could contain data that should be kept private from the other installations.
Using the LQ like this, the proof of concept queuing system would even be
deployable in a hosted environment where you do not thrust the other Drupal
installations on the server because of the following: Data is kept private. The
Task data or payload is kept private, stored with the installation itself. Only
the installation can create tasks for itself. Other malicious installations
cannot spawn tasks to run on your installation, since the LQ has the list of tasks
queued. And only the installation itself can write to this system.

system executes a task.
As seen in the diagram 6.6 the interactions start with Resque attempting to
reserve a task inside a loop (1). As soon as a task has been found and reserved
Resque forks a new process and calls the perform() method on the DrupalJob
class (2). The forked process is used to ensure that the main Resque process
gets killed in case of a critical error because of the task. The DrupalJob gets
the site_alias from Resque which is enough to bootstrap the speciﬁc Drupal

the speciﬁed Drupal installation and invokes the DrushQueueRunner (3+4).
Since the speciﬁed Drupal installation has all the data regarding the task, it
only needs the task id (item_id) from the Drush command. Furthermore, a
malicious installation cannot queue a task on behalf of another installation, since
the executing installation would not be able to ﬁnd a record of it. The task is
reserved and the payload fetched, using the MultisiteQueueInterface class (5).
Using PHP function call_user_func() the task is executed with the variables
from the payload (7). When the task has executed, it is marked as processed
together with a small log of the events (8). The task is marked as complete,
to ensure that the task will not be executed again by mistake and to show the
developers the status of the task. In case any error occurs, it will be catched
in the try-catch surrounding the task execution. Two exceptions are handled:

When oﬄoading operations to a central queue there is a risk that a malicious
installation would execute malicious tasks on other Drupal installations. This
risk is not present in the ChurchDesk system, since 3rd party developers are
not allowed to execute code but only communicate via APIs. But since the
proof of concept is made as general as possible, security has been included in
the design. This also fulﬁlls goal G4. This means that the proof of concept

installations belong to diﬀerent organizations and so on.
Security is handled by ensuring that only the Drupal installation itself can create
a task. When a task is created, it is ﬁrst stored locally, after it has been created
and stored within the bounds of the Drupal installation. Then this task is sent to
the queuing system with no sensitive data, other than the task id and sitename
as a reference. When the queue executes the task it gets executed in the context

on other versions of Drupal. Additionally, they allow an easy implementation
with another queuing backend. For the infrastructure recommendations, it was
recommended that ChurchDesk adds two servers running the consumers, and
one extra Redis machine, as a hot standby (Goal G6). In this chapter, it was
also discussed how the proof of concept solution ensures security by keeping
sensitive data locally on the individual installations, instead of relaying all data
and credentials to the queuing system (Goal G4). Finally in chapter seven, the
proof of concept solution was tested using a black box testing method, verifying
that the diﬀerent scenarios work.
==PAGE:105==
Chapter 9
Conclusion
This thesis had the vision to help ChurchDesk improve the user experience,
allowing developers to move time-consuming tasks away from the end user,
and thereby greatly increase the performance. Before, the end user would wait
seconds, when sending a message to his fellow employees in the intranet, because
the application would only return, when all messages had been sent. This led to
frustrating customers, who expected a much more ﬂuent work ﬂow. It is obvious
that long loading times do not belong in a modern web application. Therefore,
the problem was solved with the proof of concept.
The solution allows easy refactoring of the existing code. At ﬁrst, the devel-
opers at ChurchDesk were sceptical, whether this could be done with all the
moving parts included. However, this thesis shows that it is possible, even with
Drupal. The proof of concept solution provides a queuing system that works in
the special multi-site installation environment that ChurchDesk uses. It allows
ChurchDesk to oﬄoad tasks to a central queuing system, and it still has access to
individual Drupal installations, when running the tasks. The proposed solution
is also a community ﬁrst in terms of running a queue with a Drupal multisite
installation environment. At the same time the solution was designed in such a
way that security of the Drupal installation is not compromised, ensuring that
the individual installation has full control over its own data.
Nine goals were listed in the introduction that described a successful implemen-

Conclusion
tation of this thesis, which will now be brieﬂy summarized in terms of how they
were achieved:
G1: Resque was chosen as the state of the art queuing backend, since it met all
the requirements. RabbitMQ was a superior contender, but in the end Resque
won, because of the ease of integrating it into the ChurchDesk infrastructure.
G2: A Drupal module was developed, which can be shared back to the commu-
nity.
G3: The proof of concept solution was developed, following best practices, as
described in the SOLID design pattern.
G4: The proof of concept solution was designed in a way that ensures that only
the individual installation could access the relevant data or queue jobs.
G5: The proof of concept solution was implemented using principles from the
SOLID design pattern, and by extending the DrupalQueue interface, which
ensures that ChurchDesk developers can easily use it for diﬀerent use cases.
G6: The changes suggested were limited, since the selected queuing backend
uses components, which are already present in the infrastructure. The recom-
mended addition was a fail over instance of the Redis machine, and two machines
to host the actual consumers.
G7+G8: Three use cases were found and described in detail. All the identiﬁed
use cases, exhibited the same problem, which was that the user most often had
to wait for an unreasonable amount of time, when performing the use case.The
use case Save event and Save group message were implemented, and both expe-
rienced a much better performance.
G9: The proposed solution allows for a scalable deployment of the architecture,
only requiring the same code base, which the existing web nodes already use.
ChurchDesk now needs to deploy the proof of concept solution to the production
environment, after QA testing, and validate that the ﬁndings of this thesis also
work outside the testing environment. There is no indication that this should
not be the fact.
Later on, the rest of the ChurchDesk application needs to
be refactored, in order to make it apply the proof of concept queuing solution
throughout the application.
For the future, special care needs to be taken, when ChurchDesk grows. The
polling consumer principal might be ﬁne for now and also for the future, since it
can be deployed very fast in a production environment, as the necessary servers

Chapter 10
Future Work
Even though the Drupal 8 version has changed signiﬁcantly in its queuing in-
terfaces, some wrappers could be designed that would allow the use of the same
queuing system for Drupal 8 when it gets released. It was decided not to include
a Drupal 8 version since the ﬁnal Drupal 8 release is still not available.
A lot of thoughts have been put into making this queuing system in a way that
could make it run in a hosted environment. Now, It could be interesting to con-
duct interviews with some of the leading providers in order to fully understand
their needs and their customers needs, regarding a queuing system like the one
this thesis has described. As a result, we could actually get them to integrate
and oﬀer this queuing system solution to their customers. Having a queuing
system for hosted Drupal sites, would certainly be an industry ﬁrst.
The strength of the system lies in the fact that the consumer can be implemented
in any language. Which means that Drupal (PHP) could queue the job while the
job ending up being executed in Symfony (PHP) or even a completely diﬀerent
language such as NodeJS, Python or Erlang, which is gaining traction in web
development. The same applies for the queuing of tasks. A task could be queued
from a diﬀerent service to be executed on a select Drupal installation if data
needed to be inserted into the Drupal database. In this case the security of
the proof of concept solution would have to be relaxed a little bit. Because of
this ﬂexibility in the queuing system, the suggested solution can also support
==PAGE:110==
96
Future Work
the ChurchDesk application when it is time to refactor it into a single database
which has been an ongoing topic of discussion for future developments for quite
a while. This is going to happen eventually because it is an idea and future
plan put forward by the product owners. Centralized data makes it easier to
implement collaboration features between churches. This transition is not going
to happen overnight though, and could take a signiﬁcant amount of time. When
the transition is started, a queuing system becomes even more important in order
to avoid, having the user wait for internal application communication between
the Drupal installations and the central database.
In fact this process has
already somewhat started because of the newly native developed applications
for iOS and Android which need a single API to communicate with.
The proof of concept solution is versatile enough to also be able to integrate with
other queuing backends such as Kafka. This opens up for better integrations
with microservices where communications are done by message queues such as
Kafka or similar. This in term can allow for the decentralization responsibility.
The solution could also support a Reporting Database design pattern to speed
up reads from the database [Fow].
At present, partners would prefer to be
notiﬁed by ChurchDesk, every time an action occurs. This can now be solved
quite easily by simply putting the job in the queue and then letting the job
notify the partner when it gets executed.

The Drupal installation needs to specify in the settings.php which multisite

$conf[’queue_class_multisite_queue’] = ’D7MultisiteQueue’;

ﬁle:
$conf[’queue_connector_class_multisite_queue’] = ’ResqueConnector’;
