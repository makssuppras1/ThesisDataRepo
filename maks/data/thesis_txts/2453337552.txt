Summary
Readers of this thesis is expected to have basic knowledge of program analysis. The
reader should know of abstract interpretation and be able to apply it to simple pro-
grams. The thesis is divided into three major parts, but it is resolved around deter-
mining the value ranges using bitwise representation. The first chapter is based on
records, introducing both independent and relational attributes. The second chapter
is the abstract interpretation, and analysis of value ranges using bits. The ideas of
independent and relational attributes are applied to the second chapter. The final
chapter introduces solutions to interprocedural analysis using abstract interpretation.
The ideas from each chapter are then fused into one solution.

CHAPTER 1
Introduction
Analysing value ranges from the program code can prove rather useful. This can
for example be used to ensure that the code follows the semantics defined for the
program. The static prediction of the value ranges executing a system, using mathe-
matical integers have imprecision. To get rid the imprecision the value ranges using
bits representing the integers is introduced. Bits and records of variable sizes have
common features. When records are defined, they have a finite size. It is a very
similar behavior to bits, which also has a finite size. The intraprocedural analysis
includes abstract interpretation of the records of variable sizes using three different
approaches. Each of these approaches are then applied to the value range analysis us-
ing bits. Intraprocedural analysis is in reality not fulfilling enough to apply to a real
life example. Because most object oriented programming languages have methods
included, such as Java, C# and etc.. Interprocedural analysis is however a complex
task that has to be minimised. The abstract interpretation for interprocedural anal-
ysis is introduced in this thesis to cope with methods of object oriented languages.
These abstract interpretations can then be used to implement a framework that can
do a static prediction of the value ranges.
1.1
Related work
The texts that are cited in this paper are interprocedural analytical, since that is
the topic of most of the related work. Introductory interprocedural analysis can be
found in [SP78], which describes ways to set up a context for an analysis.
Both
a call strings approach and a functional approach is described in [SP78], which in
depth describes the many important factors that are important for interprocedural
analysis. An interesting flow graph abstract interpretation can be found in [VU92],
which also uses call strings to increase the precision of the analysis. Another paper
on the interprocedural analysis can be found in [KS92], which is quite similar to
[VU92], but is more specific on the call stack of the compiler, which is being used to
its advantage. Most of the ideas in this thesis is based on the methods found the book
[NRH05], however they use flow graphs in comparison to the program graphs found
in this thesis. Especially the pages [NRH05, p.82-104] have been used for inspiration
for the interprocedural approach in this thesis, and creating a version that is more
lightweight in the sense of running time. Some of the ideas found in [V A+86], have

1.2
Parsing
In order to develop analyses and to simplify my workload i have set up a simple
programming language derived from C, which have the simple operators, so that
the following chapters can easily refer to, and work with this simple programming
language. The lexer and parser is created using code snippets from [Jon].
1.3
The language
The initial parsing of the language is built of either declarations or statements. I
have decided that in each program the declarations should come first followed by the
statements. Thus a program is a declaration followed by a statement. This is visible
in the table below under programs.
A statement is either an assignment statement, if statement, while loop, read,
write or assignment of a record. An interesting addition to the language is that a
statement also has the option to a pair of statements, resulting in the possibility that
a set of statements is possible.
For the expressions there is a set of labels which is primarily used in assignment
statements and read statements. These labels are either a variable, an address of an
an array or an address of a record. Whereas for an arithmetic expressions we have
exactly the same as for labels, however an extra addition is the bitwise negation and
that an infinite set of operators can be added to the expression using a1opaa2. A
random function has been introduced as it is quite interesting when working with
bits rather than the simple integers.
Currently in the program only integers can be addressed using variables, arrays
and records.
It is also possible not to have any declarations and similarly as for
statements a declaration can be a pair of declarations to create a set of declarations.

CHAPTER 2
Intraprocedural
analysis
The first part of the thesis involves intraprocedural analysis. The section describes
abstract interpretations of a detection of signs analysis. This specific case is also
interesting because the implementation with records. The abstract interpretation for
records leads towards a simple representation of bits, since bits have similarities with
records. The interesting similarity is that both an integer represented via bits and a
record both have a finite size when they are declared.
My implementation of records is built on the semantics of the program code defined
in section 1.3. Everything is defined using program graphs as mentioned in section
1.4. There is designed and implemented a Detection of Signs analysis, to get a simple
overview of how an implementation of records would be in practice. The first intuitive
approach to records is to treat the addresses in the record as one. Thus adding the
new-found information at each edge to the current set of signs for the record. A naive
solution is a good way to cope with the initial struggles of implementing an analysis
of program code.
2.1
Basic solution
As mentioned earlier the basic solution is a naive solution that treats all the addresses
in each record as one, giving a very simple implementation.
For the records i am taking the following assumptions in order to achieve my goal:
• When a record has been declared with a certain size, if that record is accessed
again later with for example a record assignment, i assume that the sizes and
types are matching.
• When a specific ”subvariable” of a record is being accessed, that the record is
already declared. For example if i try to access R.first, then the record R has
to be declared with an address called first.
These assumptions are fair to assume, as this is normal for a typechecker to ensure
that it is been done.

2.2
Independent attributes
From the basic implementation in section 2.1, some problems with imprecision arose
when working with both the arrays and records. However as mentioned earlier, arrays
are quite hard and does not give that much increased precision in comparison to the
increased computations. Records is a more interesting case, since they have a finite
size, meaning it might have a simpler analysis. As it always is in static analyses, we
gain increased precision at the cost of complexity of the analysis. This section will
describe an analysis that is called independent attributes, which is applied to records.

Procedures and functions are a very common feature in programming languages which
is being used in many different ways to simplify duplicative code. However doing
interprocedural analysis is a different story. Each of the programming languages have
wide ranges of how the functions are being compiled and executed. This section is
going to introduce simple syntax of procedures and functions, in order to find simple
ways of analysing them.
4.1
Syntax of the functions
In section 1.3, the language is based on the objective programming language C. The
syntax for function calls and function declarations will also have similarity to C.
The declaration of the functions is int proc(p) {DS} where only a function that
returns an integer is allowed, in order to simplify the analysis.
The name of the
function is proc, and p is a list of parameters where only integers are allowed.
The syntax allows that function declarations F are after the normal declarations
D when defining the program. This gives the feature such that global declarations
are inside the function declarations and can be used.
To call a function and still have restrictions in order to make a very simple analysis
layout, the language only allows x := proc(as) where proc is the procedure name, and
x is a simple variable. The list of arguments for calling the procedure is as, which is
either one or more arithmetic expressions. Thus forcing the programming language
to always have minimum one argument/parameter.
To keep the similarity to C, a return value is required. The return statement
takes an arithmetic expression to ensure that an integer will always be returned.
Function declarations are syntactically very similar to the declaration of a variable.
The main difference is the parenthesis with the set of parameters and the body that
differentiates the syntax of the functions.
For the functions, the following assumptions have been made:
• That functions are uniquely labelled.

allowed which automatically ensures the type is the same).
• That only functions declared in F that are called.
• Recursion is allowed, however i assume that a function call isn’t in the body of
another function call
• A function declaration always has a return in its set of statements S.
• The variables are uniquely labelled, even the variables from the function body
are unique from the global variable names in order to simplify the analysis.

Context insensitive analysis
In the book [V A+86] there is an approach called context insensitive analysis, which
treats each call and return statement as ”goto” operations.
The implementation
that they describe are created so that they create additional edges that captures the
essence of entering and exiting a procedure. The additional edges and nodes that are
created are the following:
• Each call site to the beginning of the procedure it calls and
• the return statements back to the call sites
The approach implemented in this section has many similarities to the context
insensitive analysis described in [V A+86].
The analysis described in section 4.4,
each function call is treated as a ”goto” operation where an edge from the current
part of the program graph is created to the function, and wrap up edge from the
function back the same part of the program graph. As mentioned earlier, it is a very
inaccurate approach to interprocedural analysis, since the more calls to the same
function, the higher is the likelihood of overapproximation. This is all due to taking
the least upper bound. All the knowledge that is passed through the analysis at that
point returns all the values from all the function calls.
A more complex, yet still simple approach
This is a bit more complex ap-
proach to interprocedural analysis, and works well in simple cases. It however still
have many flaws, making the analysis quite imprecise. The reason is the lack of con-
text sensitivity, which is quite obvious since all function calls are to the same function.
As mentioned earlier it is a very inaccurate approach, but it has the opportunity of
being lightweight to implement and quite intuitive. The implementation of this anal-
ysis is comparable of how a developer writes program code. The problem with the
context insensitivity, is the common reason of why the function is made in the first
place. Usually functions are made to avoid duplicative code or to solve the problem

the context c where, x := ret, and ret is removed from that context.

4.5.3
Similarities
Summary-Based Context Sensitive Analysis:
In the book [V A+86], there is
an analysis called Summary-Based Context Sensitive Analysis. It is an interprocedu-
ral analysis which is supposed to encapsulate the behavior of a procedure using what
they call a ”summary”. The main idea of a ”summary” is to avoid reanalysing the
procedures body again and again as in the macro solution. Each procedure is mod-
elled as a region, where there is an entry point for each time the procedure is called.
This analysis also allows nested procedures as they are bound to the entry point of
the region. The analysis are then split into two parts, where the first is a bottom up
phase that has a transfer function to summarize the effect of a procedure. And the
second part is where a top down phase to compute the results of the procedure to the
callees.
This is a very similar approach to what is done in section 4.5. The bottom up
phase with a transfer function to summarize the effects of a procedure is similar to
the approach in section 4.5, and transfer functions are used to encapsulate the effects
of a function. The top down phase to compute the results of the procedure to the
callees is also very similar to the transfer functions that pops and returns.
Flow graphs:
In the book [NRH05], is where motivation and inspiration for the
interprocedural analyses originates. In the book they are dealing with other problems,
since they are using flow graphs rather than program graphs. For example it is simpler
for flow graphs to just add the pop functions to the labels of the nodes. For program
graphs it is a more tedious approach, since at minimum, an extra edge and node has
to be created to encapsulate the same effect.

4.8 Further development

4.8.1
Worklist changes
Currently the worklist only uses the transfer function on the action of an edge, rather
than the complete edge. All the transfer functions only takes an action to apply the
analysis. An example is the analysis domain Action 7→(∆7→L) 7→(∆7→L) for
the last interprocedural analysis. For interprocedural analysis the information of the
nodes is interesting and could be used such that less information are needed on the
actions. This simplifies the analysis and the program graph generation. The resulting
domain for the last interprocedural transfer functions would then be Edge 7→(∆7→
L) 7→(∆7→L), where an edge is (qFrom, Action, qTo).
4.8.2
Increase precision of context sensitive analysis
The context sensitive analysis had problems with the values before a function call and
after. For example if we have the variables x and y before a function call, and y is
changed locally in the function. Only x is overwritten in the call. The imprecision is
that the least upper bound is taken of y from before the function call and the changes
locally within the function. This is an overapproximation, so it is not a problem for
the analysis. However the precision can be increased since in an actual function call,
if the global variable y is being changed locally, it overwrites the global value. An
idea to actually develop this would be to add an extra edge and a node, with a new
action which resets all the values of the variables that are being set locally to the
empty set. When the least upper bound is taken, it is only the values coming from
the function, such that it actually works as an overwrite.
4.8.3
Pass by reference
Another interesting subject to especially functions is pass by reference, or something
similar to pass by reference. Pointers are common in c, and further development of
pointers can possibly solve the problem of pass by reference in other languages. Since
it is common to just pass the pointer of a variable into a function. If that variable
that it points to is being modified in the function, the analysis also has to take in
account of the pointers in order to have a correct analysis.

CHAPTER 5
Conclusion
Implementing the records in the framework, served the purpose of introducing two
ways of analysing records of variable size. The independent attributive analysis is a
good way to analyse records of large lengths. The running time for the analysis is av-
eragely faster both theoretically and practically in comparison to relational attributes.
However the relational attributes have increased precision. The precision gained from
relational attributes in comparison to independent in this framework, is however not
worth it. Especially for the value ranges of the bits. The running time of the frame-
work runs out of proportions very quickly from the tests of relational attributes. This
is of course implementation specific, but also theoretically it is usually a much worse
O factor of the running time. Both abstract interpretations are interesting, and find-
ing new ways of optimising both can have promising ways of optimising the value
range analysis. Using bits for the value range analysis have proven valuable. Having
increased precision, by knowing for example the more significant bits, can be used to
draw more conclusions from the result of the static analyses. The running time of
the bit implementations can be heavy, depending whether implementing independent
attributive analysis or relational attributive analysis is chosen. To upgrade the ab-
stract interpretation to cope with more complex systems interprocedural analysis is
implemented. The context sensitive analysis is the most interesting analysis, which is
more expensive than the other interprocedural analyses. The context sensitive analy-
sis results in a decently quick implementation both theoretically and practically. The
context sensitive is fairly weak when a function is called repeatedly a huge amount
of times, but it is still valuable. Joining both the context sensitive analysis and the
value range analysis in this thesis, is a great start for implementing a framework. The
study can even further be enhanced with multiple opportunities.
