Summary
This thesis investigates the process of gaining access to a Bitlocker encrypted
hard drive partition. The process is discussed in two parts. First, the process
and diﬃculties of acquiring a trusted platform module (TPM) that is com-
patible with a motherboard are analyzed. This was done by comparing TPM
headers on motherboards designed by three diﬀerent manufacturers. The analy-
sis shows that none of the three manufacturers share the same pin deﬁnition on
TPM headers with the same pin layout. Seeing the results of this comparison,
it can be concluded that TPM headers should be standardized. Especially if
manufacturers that are not included in the comparison follow the same pattern.
The latter part of the thesis analyses how tools for gaining access to a Bitlocker
encrypted partition can be used and improved. The analysis is documented so
that each step of the attack is explained. The improved tools manage to gain ac-
cess to the Bitlocker encrypted partition in fewer manual steps than the original
version. The tools displayed in this thesis, along with a soldering iron, provide
an individual with no prior knowledge about TPMs or Bitlocker the means to
gain access to a Bitlocker encrypted partition.
The fact that "just" a student can perform this attack, without any prior techni-
cal knowledge about TPMs or Bitlocker, is something Microsoft should consider.

Chapter 1
Introduction
Conﬁdentiality of personal and corporate data is a great factor in the digital
world that exists today.
Keeping data secret does not indicate that people
are hiding something, they simply want privacy. They don’t want prying eyes
looking around their personal data, possibly ﬁnding something of value. One
of the methods used to maintain conﬁdentiality today is hard drive encryption.
Hard drive encryption software aims to encrypt hard drives resulting in the
drive’s data being unreadable to the prying eyes of an intruder who does not
possess the key to decrypt the data.
This key has to be generated securely
and randomly. Developers have found creative ways to generate such keys and
one such solution is Veracrypt’s [1] solution.
Veracrypt uses random mouse
movements to generate a password and salt which are used to encrypt the drive.
However, having a device which could securely create these keys and even store
them securely would remove this responsibility from the operating system. A
Trusted Platform Module (TPM) [2] is an example of such a device. Using a
TPM in combination with Windows 10’s built-in hard drive encryption feature
Bitlocker [3] provides the desired solution to the conﬁdentiality concerns, but
ﬂaws have been found in it.
Alexander Couzens [4] introduced a method where TPM communication via the
Low Pin Count (LPC) bus could be sniﬀed and used to decrypt the partition
encrypted by Bitlocker. His research has led to multiple versions of the attack
where Bitlocker encrypted partitions are compromised.
This thesis aims to investigate and explain in detail the process of gaining access

to a Bitlocker encrypted drive, improving it and ﬁnally communicate the results.
1.1
Motivation
When hard drive encryption and a TPM is used, a certain level of conﬁdentiality
is expected. Making sure those expectations are met should be challenged by
trying to circumvent the methods used. Encryption solutions only provide con-
ﬁdentiality until proven otherwise. Thus, the motivation of trying to circumvent
the security measures which these solutions provide is only to shed light on their
faults, which should then be used to improve them. With physical access to a
system protected by Bitlocker and a TPM, can they be circumvented in order
to break the level of conﬁdentiality they are supposed to provide?
1.2
Scope
In the beginning, this project’s scope was to replicate the existing attacks and
try to produce new ones.
However, due to diﬃculties acquiring compatible
components the scope had to be revised. Instead, the project was split up in
two parts, where the ﬁrst part addresses the diﬃculties of acquiring compatible
components while the second part aims to utilize existing tools to gain access
to a Bitlocker encrypted drive and improving them.
1.3
Structure
This report is split into ﬁve main parts. Chapter one contains the Introduction
to this thesis. There, the problem at hand is addressed and the scope of the
thesis is explained. In chapter two, the Trusted Platform Module is explained
and its capabilities are generally described. The third chapter lists components
of the computer used for testing, discusses the problem of acquiring compatible
components and analyses the problem.
Chapter four introduces attacks on
Bitlocker encrypted partitions before replicating one of them and discussing in
them detail. Improvements are proposed and implemented with the addition
of supplying ready-to-use tools to use when performing the attack.
Finally,
chapter ﬁve concludes this thesis where the research is discussed along with its
results. In addition, ideas of future work are suggested to further improve and
build upon this research.

Chapter 2
Trusted Platform Module
2.1
What is a Trusted Platform Module?
A Trusted Platform Module, hereafter called TPM, is a dedicated chip that
has the goal of securing systems using its integrated cryptographic keys. The
TPM is standardized by the Trusted Computing Group which releases speciﬁ-
cations that manufacturers must follow [5]. This chip is most often mounted
on a motherboard but can also be in the form of software [6]. The software
TPM is emulated on the device running it and therefore uses its hardware to
function as a TPM while the hardware TPM is a specialized chip designed to
function solely as a TPM. A software TPM is more vulnerable than the hard-
ware TPM because it is easier to tamper with and use existing vulnerabilities
in the operating system. Therefore, a software TPM is ideal for development
and testing of code for a solution that is ﬁnally deployed on a hardware solution
[7],[8]. TPM capabilities include identifying a device, encryption, key storage,
random number generation, Non-Volatile storage (NVRAM), Platform Conﬁg-
uration Registers (PCRs) and Privacy Enablement [6]. These capabilities will
be described in the following subsections.
==PAGE:13==
2.1 What is a Trusted Platform Module?
4
2.1.1
Identiﬁcation
A cryptographic private key, called the primary key, is embedded in the mi-
crocontroller (the TPM). This key is what identiﬁes the chip it is residing on.
Anything possessing the public counterpart of the private key can authenticate
the chip and thereby the device it resides on. The private key is only used by
the TPM and therefore it never leaves the TPM. The key is used together with
an authentication of a person to make "the thing you know" and "the thing
you have". The authentication can be in the form of a PIN or a password. By
combining the two, an IT organization can identify a device. There are multiple
examples of using this feature of the TPM in realistic situations [6]:
• VPN identifying a machine before granting access to a network - Both
personnel and device
• User signing e-mail
• User decrypting e-mail sent to them
• User identifying themselves to their bank
• User authorizing a payment
• User logging in to a system
2.1.2
Encryption
The chip can handle encryption of small amounts of data such as keys. It is
capable of handling symmetric and asymmetric keys. This enables the device
to encrypt and decrypt keys before storing them because by utilizing the conﬁ-
dentiality of the primary key and using it to encrypt other keys before storing
them, it assures conﬁdentiality of those as well. This feature is useful for, but
not limited to [6]:
• File and folder encryption on a device
• Full disk encryption
• Encryption of passwords for a password manager
• Encryption of ﬁles stored remotely
==PAGE:14==
2.1 What is a Trusted Platform Module?
5
2.1.3
Key Storage
The TPM has a relatively small storage space which would be suﬃcient if it was
only meant to store one or two keys. However, it has to be able to store more
than that. To solve this problem the TPM is implemented so it can encrypt keys
using the primary key and store them externally. The TPMs use the computers
hard drive to store the encrypted keys which gives the potential of having almost
unlimited keys. The keys are encrypted with a private key which the TPM itself
generates. Having a feature such as this can be useful for things like [6]:
• Privacy-sensitive solutions that use diﬀerent keys to provide only a mini-
mum of information to a requestor
• Diﬀerent keys for diﬀerent security levels
• Diﬀerent keys for multiple users of the same PC
2.1.4
Random Number Generator
Having a random number generator is important when it comes to generating
keys. A random number generator can produce seeds for other random number
generators such as the one that the OS uses.
Even though there can exist
better random number generators within the system, the TPMs random number
generator can always be trusted. This is because the TPM is always trusted
and other solutions on the system might be predictable or even not secure. It
is also capable of generating nonces or one-time use keys to encrypt ﬁles to give
an example.
2.1.5
NVRAM Storage
TPMs have a small Non-Volatile Random Access Memory (NVRAM) module.
The NVRAMs access can be restricted and therefore it is an ideal location to
store keys when the system is not powered on. It provides a fast method of
accessing data and serves as a intermediary between diﬀerent system compo-
nents [6]. The space is used, for example, for keys that are not accessible when
the hard drive is encrypted for example. When the hard drive is encrypted the
keys stored there, mentioned earlier in subsection 2.1.3, are not accessible until
the drive is decrypted. Therefore this is an ideal location to store a key used
for disk encryption. The NVRAM also stores the state of the machine which is
used during secure boot.
==PAGE:15==
2.2 Types of Hardware TPMs
6
2.1.6
Platform Conﬁguration Registers
Platform Conﬁguration Registers or PCRs, are used to store hashes of measure-
ments. These measurements are provided by software running outside of the
TPM and are produced over the boot process. The measurements are signed
with a special key when the TPM has to report them for authentication pur-
poses. This makes the TPM a root of trust for reporting. These measurements
are useful when access is only allowed under certain circumstances [6]. An ex-
ample of such circumstances could be when a certain feature should only be
accessible if a certain program is running or under speciﬁc circumstances: A
computer should only be allowed to connect to a network if it is running an
approved anti-malware software and the operating system is updated.
The
measurements can be chained together to provide further complexity. PCRs
can only be reset or extended. Thus, data already in the registers cannot be
modiﬁed.
2.2
Types of Hardware TPMs
Hardware TPMs come in three diﬀerent variants. Their diﬀerence lies in how
they are introduced to the system they are or become a part of. These three
variants will be discussed individually in the following subsections.
2.2.1
Modular TPMs
A modular TPM is a chip that can be installed on the motherboard. The system
functions without it and depending on the user’s needs, they can decide to add
it. Not all motherboards support modular TPMs because they require a special
TPM header to interface with the system. An example of the TPM header can
be seen in Figure 2.2. The TPM headers on the motherboards may diﬀer in the
number of pins and their layout. Their layout diﬀers due to the fact that the
TPM communicates through either the SPI bus or the LPC bus. The number of
pins can be 11, 13 or 19 and are called, respectively, 12-1, 14-1 and 20-1. The ’-1’
symbolizes that there is one pin missing from the number before it. The missing
pin is used to indicate which bus TPM communications go through. This can
be seen in Figure 2.1 where the black squares represent pins. Modular TPMs
are discussed further along with their pin deﬁnitions and layouts in section 3.2.

2.2.2
Integrated TPMs
An integrated TPM is presented as a part of a motherboard. The TPM can not
be removed or replaced. This is very common for laptops. Integrated TPMs are
often discreet and hard to notice if one does not know what to look for. Fig-
ure 2.3 shows where an integrated TPM can be found on a Dell Latitude E5470
laptop’s motherboard. These TPMs are harder to access than the modular TPM
and provide therefore slightly more physical security.
2.2.3
fTPM
A ﬁrmware TPM (fTPM) runs in the CPU’s trusted execution environment
and is a ﬁrmware-based TPM 2.0. Even though this solution runs as ﬁrmware,
it can still be considered to be a hardware solution because it is a part of

the CPU. fTPMs rely on the operating system for ﬁle storage and have no
dedicated storage [7]. This is because the fTPM is a part of the CPU, which
is a hardware component. As an example, Intel has implemented this solution
into their solutions under the name of Intel Platform Trust Technology (Intel

2.3
LPC communication
TPMs can be developed to utilize the Low Pin Count (LPC) bus to communi-
cate with the system. Commands and requests can be sent to the TPM and
it can respond appropriately. Just as with regular LPC communication, LRE-
SET resets the device and marks the beginning of new TPM communication,
LFRAME marks the beginning of a TPM transaction, LCLK is a 33.3 MHz
clock and LAD[3:0] is used for carrying multiplexed addresses, data and more.
By utilizing the bus, the TPM manages to exchange data freely with the host
system. For example, this communication is used for taking measurements in
addition to requesting and sending encryption keys, which can be sniﬀed as
demonstrated in this thesis. The TPM can communicate with a variety of en-
tities through this bus. Entities such as other hardware, the operating system
and software can utilize the TPMs capabilities for secure operations.
2.4
Locality
Within the TPM exists a security hierarchy. This hierarchy is called Locality
and applies diﬀerent security privileges for diﬀerent ranges of memory addresses.
Thus, each set memory range has its own locality. The TPM is not aware of
==PAGE:18==

any hierarchy and therefore treats each locality the same. The TPM receives a
command with a locality indication through the LPC bus from the host system.
It does not know who sent the command, it just blindly trusts the locality and
executes the command accordingly. Depending on the TPM’s programming,
only deﬁned sources can change set PCRs. Thus, memory ranges are protected
by the locality indicators supplied with the commands. This provides a chain
of trust where resetting or extending PCRs is restricted to software of a certain
level of privilege [8].

After browsing through the selection of motherboards that support TPMs, pro-

the speciﬁc pin layout required in their speciﬁcations. However, the manual for
each motherboard provides the required information. This means that there
is no eﬃcient way to ﬁlter or search for motherboards with speciﬁc TPM pin
layouts.

The diﬀerences

Chapter 4

a Bitlocker encrypted drive
Encrypting your hard drive gives you the feeling that your data is safe and
should not be available to other people without your consent. This chapter will
demonstrate and document the process of trying to gain unauthorized access to
a Bitlocker encrypted drive.
4.1
What is Bitlocker?
Bitlocker is an operating system feature developed by Microsoft which has been
a part of the Windows operating system since Windows Vista.
It provides
data protection in the form of encryption of the drive running the operating
system. It works best when used with a TPM module which is used for securely
storing the encryption keys [3]. In the modern version of Windows, Windows
10, a recovery key is backed up on a USB drive during drive encryption. This
recovery key is used to decrypt the drive if something goes wrong with the TPM
or the system as a whole.
Bitlocker requires a TPM of version 1.2 or higher, but version 2.0 is always
recommended. It can function without a TPM, but it would require a exter-
==PAGE:29==
4.1 What is Bitlocker?
20
nal device such as a USB drive to save a startup key on. The hard drive to
be encrypted must be partitioned into at least two partitions. The partition
containing the operating system must be formatted with the NTFS ﬁle system.
The partitioning required for Bitlocker happens automatically when installing
Windows [3].
Bitlocker oﬀers a number of possibilities when encrypting a hard drive. These
possibilities diﬀer in level of security due to the number of components used for
the encryption. Below are listed combinations used with TPMs:
• TPM only
• TPM + PIN
• TPM + PIN + USB key
• TPM + USB key
The combination which provides the least security but with the highest comfort
is the one that only uses the TPM. It provides the least security due to the fact
that there is only one component needed for decrypting the Bitlocker encrypted
drive. On the other hand, the combination with the most factors provides the
best security at the expense of less comfort. The lack of comfort comes from
the fact that the user has to insert a USB drive containing a key and provide a
PIN so that the TPM will release the decryption key for the drive.
4.1.1
Bitlocker boot process
When a computer with Bitlocker encryption enabled is started, measurements
are taken of the system which indicate if the system is in the correct state. The
TPM does not produce these measurements itself, but it stores the ones which
are used for comparison. If the measurements are valid, the system is veriﬁed
and the TPM releases the Volume Master Key to the CPU. The CPU utilizes the
Volume Master Key to decrypt the Full Volume Encryption key which resides
in the metadata of the encrypted partition. Once decrypted, the CPU has the
means to decrypt the drive and boot the operating system. Therefore the TPM
does not possess the encryption key required to decrypt the Bitlocker encrypted
partition, but the means to decrypt the Full Volume Encryption Key, which
decrypts the Bitlocker partition.

4.2
Existing attacks on Bitlocker
There have been several successful attempts in intercepting the Bitlocker Vol-
ume Master Key when the TPM sends it during startup.
All the attempts
mentioned below have successfully intercepted the key and used it to mount the
Bitlocker encrypted drive, either by determining the Full Volume Encryption
Key or using the Volume Master Key.
4.2.1
Pulse Security
Denis Andzakovic from Pulse Security [29] managed to sniﬀthe key for both
TPM 1.2 and 2.0 using DSLogic Plus 16 for the TPM 1.2 and a Lattice ICES-
tick for the 2.0. Both attacks attempt to sniﬀthe TPM communication going
through the LPC bus. His method included a python program that received data
from the devices connected to the wires soldered on the TPMs. His program
printed the data received through the device and with the Linux command tee,
he saved the output to a log. Once the log was produced he ran the following
Linux command:
cut -f 2 -d’␣outlog␣|␣grep␣’24..00\$’
|␣perl␣-pe␣’s/.{8}(..)..\n/$1/’
|␣grep␣-Po␣"2c0000000100000003200000(..){32}"
His command trimmed the output contained in the log before searching for
certain strings using Perl and regular expression pattern. This resulted in the
Volume Master Key.
His solution for decrypting the drive was to hardcode
the Volume Master Key along with the data extracted using Dislocker [30] into
another python script. This script used the hardcoded data to determine the
Bitlocker Full Volume Encryption Key.

4.2.2
F-Secure
The research published at the end of December 2020 by F-Secure states that
no public research has been published which attempts to capture TPM commu-
nication from the SPI bus [31]. F-Secure attempts to sniﬀthe communication
going through the SPI bus using a logic analyzer. Another diﬀerence from the
method mentioned in subsection 4.2.1 is that the TPM being sniﬀed in this
case is integrated to a laptop’s motherboard. Logic 2 from Saleae was used in
order to translate the data sniﬀed from the TPM. Logic 2 does not, by default,
support TPM SPI communications and therefore a custom analyzer had to be
written to analyze the communication. After applying ﬁlters for the Volume
Master Key according to the publicly available Bitlocker structures [32], the
key can be read from the software. They skip the step of decrypting the Full
Volume Encryption Key and use the Volume Master Key to successfully mount
the drive.
4.2.3
SySS-Research
This research aims to improve and combine the work done by Alexander Couzens
[4] and Denis Andzakovic [29] (also mentioned in subsection 4.2.1). In their
published documentation [33], they document in detail how to proceed with
sniﬃng the Bitlocker Volume Master Key and using it to decrypt and mount
the Bitlocker encrypted drive. The process is much more ﬂuent than described
in subsection 4.2.1 but can still be improved further. The process consists of
few steps which will enable a successful mount of a Bitlocker encrypted drive.
The ﬁrst step in the process is running a similar python program as mentioned
in subsection 4.2.1, which ﬁnds the Volume Master Key automatically while
sniﬃng the communication on the LPC bus and creates a binary ﬁle to be used
later. The next step is to extract the drive metadata using Dislocker [30] and
using it to decrypt the Full Volume Encryption Key. This is done by supplying
another python program with the extracted metadata and the Volume Master
Key binary. The python program outputs the decrypted key, if everything goes
as planned, which can then be used to mount the Bitlocker encrypted drive.

4.3.4
Programming the FPGA
An unmodiﬁed version of the Verilog code for the FPGA published by SySS-
Research (see subsection 4.2.3) was used for the sniﬃng part of this attack. This
is the third generation of the Verilog code which is based on Alexander Couzens’
work [4]. The second generation included TPM-speciﬁc modiﬁcations by Denis
Andzakovic [29], while the third included a rewritten state machine by Matthias
Deeg [33].
Before programming the FPGA, Fast Opto-Isolated Serial Interface Mode on
channel B needs to be enabled. This allows communication to the FPGA over
four wires with opto-isolators in their path [43]. FT_PROG was used to achieve
this [44]. Now the FPGA is ready to be programmed. By executing the following
commands, the FPGA should be ready to sniﬀthe TPM communications on
the LPC bus.
git clone
https://github.com/SySS-Research/icestick-lpc-tpm-sniffer.git
cd icestick-lpc-tpm-sniffer
make
make prog
The Verilog code used for sniﬃng starts by decoding the LPC data received
through the wires soldered to the TPM. Each LPC frame is decoded and saved
in the internal memory. The internal memory of the FPGA is used as a 48-bit
addressable memory. This is because 48 bits is the exact length of an LPC
frame. The data is ﬁltered to only capture the address range 24 to 27. The
author claims that most TPMs only use address 24, but some use 24 to 27
for FIFO. By applying this ﬁlter, it is made sure that only the correct TPM
messages are being captured.
Since the internal memory is only 48 bits, it
can only hold one LPC frame at once. Because of this, an internal buﬀer is
implemented.
If this buﬀer is full, new messages are discarded and a green
light on the FPGA is lit to notify the user. Once the data has been ﬁltered,
it is converted to serial and sent through the fast-opto connection [43] to the
attacking system.

4.3.7
The sniﬃng script
The two programs published by SySS-Research [33], also mentioned in subsec-
tion 4.2.3, divide responsibilities between them. It is required to run the sniﬃng
program ﬁrst, which sniﬀs the communication to and from the TPM with the
help of the FPGA. It creates a binary ﬁle containing the Volume Master Key if
the sniﬃng was successful. The second program takes the Volume Master Key
as an argument along with the metadata extracted with Dislocker, which has to
be saved to a ﬁle. If successful, the program outputs the decrypted Full Volume
Encryption Key which is then used manually to decrypt the drive and mount it
using the Linux tools bdemount [46] and mount [47]. This could be improved to
save time and increase comfort while performing the attack. The improvements
made include three python programs that aim to increase comfort and reduce
the workload on the user performing the attack.
tpmattacker.py is a program that is a combination of the two programs pub-
lished by SySS-Research [33]. Instead of having to run three commands, exe-
cuting two python programs and one Linux command to be able to access the
Bitlocker encrypted drive, this requires only one. However, this program re-
quires a copy of the Bitlocker encrypted hard drive, or the actual hard drive,
to be connected to the system. The encrypted drive is supplied to the program
as the only argument needed to run it.
The program starts by sniﬃng the
communication with the help of the FPGA and acquiring the Volume Master
Key. The program then proceeds to automatically extract the metadata from
the supplied hard drive. Both the key and the metadata are then supplied to
the decrypt function as arguments. The decrypt function parses the required
information, nonce, Message Authentication Code (MAC) and payload (the en-
crypted Full Volume Encryption Key), from the metadata. Using the Volume
Master Key and the extracted nonce, an AES cipher in CCM mode is instan-
tiated. The cipher is used to decrypt and verify the payload with the help of
the MAC. At this point in the program, all necessary information needed to
mount the encrypted drive is available. The data is written into a ﬁle that can
be used with one of the other programs mentioned earlier. Finally, the program
attempts to decrypt and mount the drive supplied as an argument. This is done
by supplying bdemount with the decrypted key and the encrypted drive. This
mounts the Bitlocker partition. From here, the mounted Bitlocker partition is
mounted to a separate folder with read privileges. Lastly, the program executes
the ls command [48] to show the directory of the mounted drive. Now that
the Bitlocker encrypted drive has been decrypted and mounted, it can be freely
explored via regular operations.

4.4
Behind the scenes
This section will describe what is happening behind the scenes in each step of
the attack.
4.4.1
Sniﬃng
The program ﬁlters data received from the FPGA for bytes received addressed
in the range 0x24 to 0x27. 0x24 is used for TPM DATA FIFO while the other
ones are reserved.
TPM DATA FIFO is a buﬀer used to exchange data for
commands and responses with the host system [50]. The ﬁltered data is kept and
the program attempts to ﬁnd the pattern of the Volume Master Key within it.
Using the found pattern, the start position of the key can be found. Knowing the
length of the key, the end position can be found as well. With this information,
the Volume Master Key can be extracted from the data sniﬀed.
4.4.2
Metadata
The Linux tool called Dislocker [30] is used to extract metadata from the Bit-
locker encrypted partition. This tool extracts vital information which is needed
to gain access to the partition. The information needed is data which Bitlocker
places on the partition in the form of a nonce, mac and payload. Where exactly
this data is placed is neither included in Bitlocker documentation [3], nor Dis-
locker documentation [30]. However, since it can be read with a tool such as
==PAGE:43==
4.4 Behind the scenes
34
Dislocker without having the decryption key, it can be assumed that this data
is not protected by encryption.
However, by reading through the published
code of Dislocker [30], it can be identiﬁed that regions are calculated and used
to identify the header of the data. From the header, the rest of the data is
identiﬁed and categorized. The size of the metadata is measured and used for
extracting. If one was to identify exactly where this data is placed, without
the help of Dislocker, a comparison could be made. This comparison would
involve copying the partition before and after enabling Bitlocker and perform
a bit-by-bit comparison. This would narrow the search area to areas on the
partition that only changed during the encryption. If Microsoft would disclose
this information in their Bitlocker documentation [3], this would of course not
be needed.
4.4.3
Decrypting the key
The program utilizes the parsed metadata to ﬁnd the required components to
get the decrypted Full Volume Encryption Key. The required components being
the nonce, mac and payload. Knowing where each component is located in the
parsed Dislocker results makes it easier to extract them. The program utilizes
the decrypt_and_verify function of pycryptodome [42] to decrypt the payload
resulting in the Full Volume Encryption Key.
4.4.4
Mounting the drive
When the program has decrypted the Full Volume Encryption Key, it uses it
to mount the Bitlocker encrypted drive. Mounting the encrypted drive is done
in two steps. First, the encrypted partition is decrypted using bdemount and
mounted to a folder. In this case the folder is called bitlocker and located in the
mnt folder. The naming of the folder and its location can be changed, but then
it has to be changed accordingly in all python scripts. Next, the mount Linux
command is used to mount the decrypted partition to a folder also located in
mnt called ntfs. The same rules apply with changing this folders name and
location. Once these commands have been executed successfully, the drive can
be browsed at the mounted location. To make sure that these folders exist, they
are created when setupReqs.sh is executed.

4.5
Results & Discussions
Having utilized the tools provided by SySS-Research [33], a Volume Master Key
was successfully sniﬀed during TPM communications. The sniﬀed key was de-
crypted and used to successfully decrypt and mount the Bitlocker encrypted
partition. Once the partition was mounted, all ﬁles could be read freely without
complications. Having successfully used the two programs supplied by SySS-
Research to achieve this, an improvement was accomplished by combining them
and simplifying. The improved version of the combined programs provides the
comfort of not having to run multiple commands in order to perform the attack.
The number of commands needed to perform the attack is reduced to only one.
That is, if the command used to copy the Bitlocker encrypted drive is not in-
cluded.
Setting up requirements was deemed to take a long time. A lot of time was spent
typing install commands and researching requirements for each package/library.
To make the setup process less time-consuming, a script that installs all require-
ments automatically was made. It still takes a long time, but the time saved is
measured in not having to type commands, research requirements and monitor
progress. Thus, the setup process is reduced to only a single command.
Encrypting a drive with Bitlocker without using a PIN and/or USB key, i.e. just
with TPM, is considered to be comfortable. This is because nothing has to be
provided by the user so that the TPM allows decryption. This is compelling for
a user who wants to secure their data while not having to give extra inputs such
as PIN in order to gain access to their computer. However, if the user is willing
to use a PIN and/or a USB key, it would mitigate this attack. That is because
normally, the intruder attempting to gain access to the drive would not possess
these components. Thus, the TPM would not release the Volume Master Key.
This means that ease of use is technically a vulnerability for Bitlocker.
Using the existing tools to perform the attack took some time to ﬁgure out.
However, once all problems had been solved, it was pretty straightforward. No-
table problems encountered involved receiving data from the FPGA and the
installation of libraries and packages. The data from the FPGA was not being
delivered to the attacking system because fast-opto mode had not been enabled.
When enabled, data started streaming to the attacking system. The long list of
requirements required a lot of research. The research included ﬁnding required
dependencies for the libraries and packages so that they could be installed. Some
of them could not be installed easily by using the Linux command apt install

and were only available through Github repositories. This led to the creation of
the bash script which, automatically downloads and installs every package and
library required to run the scripts.
If Bitlocker were to be developed again with this speciﬁc attack in mind, it would
need some drastic measures in order to mitigate it. One possibility would be
to eliminate the possibility of only using a TPM when encrypting and force the
user to use a PIN and/or USB key. Thus, the authentication factors would be
satisﬁed: The TPM (something you are), combined with either a Pin (something
you know) or USB key (something you have), or both. However, if Microsoft
wants to keep the comfort of not having to provide anything to the TPM so
that it releases the key, other solutions are needed. One such solution could
be to utilize the features of TPM 2.0 in combination with the existing solution.
The solution encrypts all data non-essential to booting up the operating system
with a key A and then encrypts the whole partition with key B. This has been
visualized in Figure 4.10. Like in the existing solution, key B is released by
the TPM upon startup. This allows the system to boot the operating system.
Upon a successful login into the operating system by an authorized user, the
TPM releases key A which can be used to decrypt the remaining data. This
is possible by utilizing the Enhanced Authorization which TPM 2.0 provides
[6]. This solution still provides the comfort of the existing solution while also
mitigating the attack presented in this thesis. This mitigates the attack because,
unknowingly, the user is providing a PIN to Bitlocker upon login to the system
in the form of a username and password. Therefore, even though an attacker
manages to perform the attack described in this thesis, he only manages to gain
access to data used to boot the operating system to the login screen. The key
released upon login can still be sniﬀed, but due to the fact that it is only released
when valid credentials, which the attacker does not possess, are supplied, the
attack is mitigated. This could even be combined with Windows Hello [51] for
further development.

Chapter 5
Conclusion
This thesis has mainly focused on two parts. The ﬁrst part analyzed compat-
ibility issues that were identiﬁed while assembling the computer used for the
experiments. The analysis compared pin deﬁnitions of TPM headers for three
diﬀerent motherboard manufacturers. A comparison was made for the three
pin layouts they support. The analysis concluded that none of the manufac-
turers used for comparison used the same pin deﬁnitions for identical layouts.
This results in a modular TPM from one manufacturer and a motherboard from
another not being compatible. If this is a fact for all the motherboard manu-
facturers which were not included in this comparison, it restricts the consumer
buying the product. The consumer is therefore forced to buy a modular TPM
and a motherboard from the same manufacturer to have compatible compo-
nents. However, this could be solved by having standardized headers for TPMs.
Having the headers standardized, the user could choose freely from any moth-
erboard and modular TPM combination.
The latter part of the thesis focused on how it is possible to gain access to a
Bitlocker encrypted partition. Multiple versions of these attacks were discussed
before describing in detail how such an attack works. The research provided
by SySS-Research [33] provided tools that could be improved upon. This thesis
presents an improved version of the tools which reduce user interaction when
setting up requirements and performing the attack. By reducing the commands
required to performing the attack from ﬁve to one and reducing the process of
setting up requirements to only one as well, a user can perform the attack with
==PAGE:49==
5.1 Future work
40
ease. These improvements also reduce the knowledge needed to perform the
attack substantially. A Script Kiddie [53] with access to the script and a solder-
ing iron could perform this attack. Thus, this attack can easily be performed
with limited resources. An image of a bootable USB running Ubunutu Server
20.04 is submitted. The image comes with all tools pre-installed and ready for
performing the attack.
This thesis shows that a Bitlocker encryption, without PIN or USB key, can
be circumvented and the target hard drive’s conﬁdentiality compromised, it is
clear that changes need to be made to Bitlocker. A solution was proposed which
mitigates the attack while also conserving the comfort of not requiring anything
extra from the user. If Microsoft does not change the functionality of Bitlocker,
this attack will remain a threat. Not only Microsoft should consider this attack
though. Developers of other solutions should consider these vulnerabilities for
their own products.

the TPM of the target system and the attacking system was not explored in

fruitful results. Managing to gain write access to the top hierarchy of the TPM
would be the ultimate goal. This is a diﬃcult task, possibly impossible, but
nevertheless an interesting one.
