Abstract 
 
 
This project is an attempt to provide a tool for evaluating soundscape pleasantness in Enscape 
virtual reality rendering engine. It is possible to walk in a 3D architectural model in e.g. Revit, and 
run the Enscape plugin to enable VR devices for navigation. However, in order to communicate 
the position and orientation of the player, it is necessary to access the information of the player’s 
head in the virtual environment. In this project this missing link has been provided using a Dynamo 
script.  
 
Initially, four sound sources have been chosen to compose the soundscape: conversations, water 
feature, drill, and a singing bird. These sound sources have been considered, by consensus of 
literature, eventful, pleasant, unpleasant and pleasant respectively. Having created a grid of 
auralisations of these source signals using Odeon’s 2nd order ambisonic impulse responses and 
Matlab convolution function, 4 scenarios of soundscapes are composed.  
 
Then the location and orientation of the player are extracted from Revit and Enscape using 
Dynamo, and are passed to a previously available Puredata patch as the audio decoding engine.  
 
The tool was tested for stationary and moving player. The stationary condition gives satisfactory 
results in terms of hearing the auralisations, while the moving player condition lacks realism due to 
large latencies. The tool is elaborated in the report and suggestions for using it in subjective 
soundscape evaluation tests are given.

1. Introduction 
 
Acoustic comfort is of high importance in architecture. This comfort is often standardised based 
on Sound Pressure Level (SPL) limits and/or Reverberation Time (RT). However, averaged SPL 
measures like LAeq or Lden are proving to be inadequate metrics especially in explaining sound 
preferences in public built environments such as communal indoor spaces or public urban squares. 
It seems that some soundscape elements are more favourable than the others, even at the same 
SPLs (Li et al., 2018; Liu and Kang, 2015). The question is “How can we know which soundscape 
elements will be statistically more pleasant in a specific architectural design?” 
 
Previous research has been done on evaluation of soundscape pleasantness by comparing test 
subjects’ evaluation of SPLs for fountain sounds and street traffic sound (Alvarez et al, 2006; 
Galbrun and Ali, 2013; Galbrun and Calarco, 2014), footsteps, aeroplanes and dog sounds (Liu and 
Kang, 2015), and many others, but mainly for outdoor environments. Bird songs have been idly 
considered in some research too (Alvarez et al, 2006). Several water features have been compared 
in terms of pleasantness (Galbrun and Calarco, 2014).  
 
But there is a gap in the literature as to how effective it is to include bird songs and water features 
in the soundscape of a communal indoor area, in terms of perceived pleasantness. Besides, most 
studies so far rely on recordings of the actual soundscape in the built environment. However, it 
would be beneficial if soundscape analysis is performed in the design phase and prior to 
construction i.e. using acoustic simulations. This could help prevent unnecessary investment in 
features that are wrongly assumed to increase the pleasantness of the designs, but needs some extra 
computational efforts.  
 
The aim of this thesis has been to study the possibility of improving the soundscape in 
architectural and urban designs. So far many noise control measures like LAeq or Lden have been 
developed to make sure that the average sound pressure level is not exceeding some certain 
amount. But soundscape studies can help understand if sound categories are also important and 
can even outweigh the traditional noise metrics. This could help architects and urban planners to 
make more favourable decisions when it comes to designing for aural perception as well as visual 
perception. 
 
Initially the project was aimed at using an available VR tool package in Unity, in order to evaluate 
the soundscape composed four sound source types: a conversation, a water feature sound, a drill, 
and a singing bird. However there was a change in the path of the project: the rendering VR engine 
was decided to be changed to Enscape which is a plugin for 3D modelling software such as Revit. 
Subsequently, the C# scripts that were written in Unity needed to be translated either into Revit 
SDK or into Dynamo scripts. After performing the steps of running the Odeon simulations and 
obtaining the auralisations, most of the effort was put on finding a way to connect the 
Enscape/Revit combination to the available Puredata patch. Although this process was successful 
in the end but it lead to not being able to carry out the perceptive soundscape evaluation 
experiments. However, it has now become possible to use the Dynamo tool for performing the 
experiments in Enscape.

2. Methodology 
 
2.1. The Initial Methodology 
 
This project will use virtual soundwalking and questions as a method for evaluation of 
soundscape pleasantness. “Soundwalk is usually conducted by a group of people following a pre-
defined walking route and using a structured protocol with a high level of sonic awareness, during 
which soundscape quality may be evaluated” (Kang and Zhang, 2010; Schafer 1977). In this project 
however, each participant performs the pre-defined soundwalk individually.  
 
Virtual reality has been recognised as a valid tool for multisensory evaluation of urban spaces 
(Luigi et al., 2015; To et al., 2017; Sanchez et al., 2017; Signorelli, 2015). This project is suggesting 
that virtual reality can be used to anticipate soundscape preferences in indoor spaces as well. The 
robustness of this suggestion is debatable, but can be checked by means of post-occupancy surveys 
(could be an extension to this study).  
 
The initial project plan was to use the available workflow shown in figure below in order to create 
auralisation of 4 sound sources at different evaluation points in the architectural design (figure 2.1).

architectural rendering industry, it was decided to change the rendering engine. Enscape was

The intention was to put 4 selected sound sources at the four corners of the Uppsala City Hall’s 
design. And create the auralisations and ask participants to walk in the design and stop at 4 defined 
spots in order to evaluate the “pleasantness” of the design on those spots. The sound sources were 
intentionally selected to be sounds of: (1) a conversation, (2) a water feature, (3) a drill, and (4) a 
bird song.

2.4.1. Matlab pre-processing (before Odeon) 
 
In this stage, the semi-anechoic signals for the four sound sources were selected from the 
available libraries. The conversation recordings were not anechoic, therefore the background noise 
was removed as much as possible. This, especially in the case of the bird song, could be a challenge 
as it’s not very easy to take birds into anechoic chambers.  
 
The four sound source signals were mono, 24 bits, and sampled at 4100 Hz. The duration of the 
signals was initially 3 minutes, but after a series of long Matlab post-processing trial and errors, 
they were cut to 30 seconds for avoiding long computation times in convolution process in Matlab.

After this step, the impulse responses were saved. But since Odeon 14 performs a normalisation 
procedure on the IRs, it was necessary to de-normalise these values. For this purpose Odeon 15 
was used which gives out the implemented attenuation factors. These factors were then used in 
post-processing in Matlab which will be explained in section 2.4.3.  
 
 
2.4.3. Matlab post-processing (after Odeon) 
 
Three post-processing Matlab codes were used which are briefly explained here. 
 
1. De-normalising the IRs using the attenuation factors 
Since Odeon 14 normalises the impulse responses to 1 and this creates unrealistic auralisations, 
the impulse responses were de-normalised in Matlab using the attenuation factors obtained from 
Odeon 15.  
 
2. Convolving the IRs with the signal wave files 
In this step the available signal wave files were convolved in Matlab with the obtained impulse 
responses. This was done for the 8 different series of Odeon simulations. This process was one of 
the most time-consuming processes in the project, since initially the length of wave files was about 
3 minutes and this led to a large matrix of samples to perform convolution. Therefore after 
encountering this issue for a few times, the length of the signal wave files was shortened to 30 
seconds and all the process was started anew.

end of this step, three folders were available which included the auralisations and the intended gird

2.4.5. Enscape and HTC Vive 
 
Enscape is a rendering engine which can be run as a plugin on Revit (and some other 3D 
modelling software like Sketch-up). It is a very fast and efficient rendering engine with considerably 
high visual qualities. In addition, it has dedicated libraries of assets like trees, furniture and people 
which are often used in architectural and landscape designs for showing the human an 
environmental aspects.  
 
One of the options provided in Enscape is the easy linkage with VR devices. In the current 
project, the HTC Vive device package at Henning Larsen was used. Enscape in combination with 
HTC Vive, provide two modes of VR navigation: (1) Room scale mode, and (2) Seated mode. 
Room scale mode is the option in which the player moves physically in the room, and rotates their 
head, and the physical position and orientation data is then transferred from the controller and 
headset sensors to the VR engine. The seated mode however, is only based on the controllers 
commands. The player will be stationary in this mode, and the positions and the orientation of the 
player is virtually defined by the controllers.  
 
==PAGE:21==
 
15 
 
In both of these modes, it is possible to select between two movement types: (1) Fly mode 
movement, and (2) Walk mode movement. Obviously, the walk mode was selected throughout the 
project, as the height of the camera was intended to be at the average height of a player’s head, and 
the navigation route was always on the ground. Besides the Puredata patch accounts for the 
horizontal plane only.  
 
Knowing these modes, the process of trial and error started in order to figure out the way to 
access the player position information in Enscape. At the beginning of the trial and error, several 
options were checked including the rendering settings in Enscape, the view definitions in Revit, 
and how it would be possible to connect the Enscape camera to a defined view in Revit (figure 
2.16a).  
 
   
 
Figure 2.16. (a) Connecting Enscape to a custom-defined Revit camera, (b) Moving the Revit/Enscape camera 
manually using a Dynamo slider node 
 
In the next step it was tried to control the synchronised cameras’ movement through sliders in 
Dynamo. This step was successful (figure 2.16b). This option would be useful in the case of using 
the Room scale VR. Because provided that the information from the VIVE headset is obtained, 
then it could be translated to Enscape (and Puredata) so that the audio-visual experiment is 
coherent. However, in order to send the player’s position and head orientation to Puredata, it was 
necessary to obtain these information from Enscape while controlling the location with HTC Vive 
controllers, which was like a vicious circle.  
 
Based on the correspondence with the Enscape technical support team, there was no way to 
access the player position from Enscape. However after a few sessions of trial and error, two 
methods for extracting the position information seemed to be possible: (1) in the Room scale mode, 
by using the physical information obtained from the HTC Vive sensors, and (2) in the Seated mode, 
by extracting the virtual position and orientation of the player via Revit SDK or Dynamo.  
 
Both of these methods were pursued and the tools for extracting the player data were either 
found or created for Dynamo. The next section will be a short introduction to these tools.  
 
 
2.4.6. Dynamo 
 
The main volume of work was done in Dynamo in order to extract the player information. Both 
the room scale and seated modes were examined. The results were as follows: 
  
 
==PAGE:22==

1. Room scale mode 
For this option, it was necessary to obtain the data from the sensors in the HTC Vive headset 
and then scale and map the position to the Enscape camera, so that it moves correctly on the grid 
points of the design model. In other words, if the player was moving 1 metre in the VR room at 
Henning Larsen, it was representative of about 50 metres in the actual design (1:50scale). 
The tool package ViveTrackDynamo created by the Generative Design Group at AutoDesk 
initially gave the option of tracking the position of the Vive controllers. Based on a video illustration 
they have released, the player can create surfaces using the controllers (figure 2.17). This package 
is available as a GitHub repository (ViveTrackDynamo, 2018). 
 
 
Figure 2.17. A snapshot of a video whereby the ViveTrackDynamo package tool is extracting the position data of the 
HTC Vive controllers and converting them to surface drawings (ViveTrackDynamo, 2018).  
An extension of the same package tool was created by the VR centre of AutoDesk in Munich by 
which the illustrator is moving a VR controller on a Revit plan with the scale of 1:100 and the 3D 
model is viewed with the corresponding camera angles with periodic updates of 100ms in Dynamo.  
 
 
Figure 2.18. Moving a VR controller on a Revit plan and translating the data into the Revit camera movement 
(ViveTrackDynamo, 2018)

17 
 
By using a combination of these tools it was possible to use the HTC Vive head mounted device 
in order to obtain the relative position of the player. Then a scaling factor was necessary to map 
the data to the actual Revit/Enscape model in VR.  
 Although this seemed an inspiring solution, but there were several challenges in this approach 
which made the actual experiments almost impossible. These included: 
 
a. The scaling wasn’t precise enough, since a centimetre of movement in the actual room 
corresponded to half a metre in the actual design. Although the grid point distances for 
auralisations were 5m, but still the tolerance for accurate movement in the room was too low for a 
player. Also there would be a confusion in communication with Puredata as well as a confusion in 
the player’s perception.  
 
b. The communication between ViveTrackDynamo package and Enscape VR camera wasn’t 
efficient. In other words, although the camera in Revit was moving along the mapped path (scale 
of 1:50), but Enscape wasn’t following the same motion.  
 
The reasons above made it difficult to use this option as a proper approach. Therefore the effort

2. Seated mode 
After several sessions of examining options in Revit, Enscape and Dynamo it appeared that it is 
possible to show the Revit Live Camera while using Enscape. This is an option in Enscape version 
2.4.0 which provides the live motion of the Enscape VR camera on the Revit plan view. This was 
the promising missing link for extracting the information of the player. Therefore the Dynamo tool 
was developed based on this feature. As it will be explained in detail in chapter 3, the Revit group 
of nodes in dynamo made it possible to get the information from the Enscape camera, process it, 
and prepare the output data to send to Puredata through UDP.  
 
2.4.7. Puredata 
 
As mentioned earlier in this chapter, the audio processing, decoding and playback is done in the 
Puredata which has been developed by Sampedro Llopis (2018). The main input values for 
Puredata were (1) the distances from the 4 nearest grid points in order to calculate the playback 
volumes, and (2) the orientation angles of azimuth, rotation and elevation. These values were 
calculated in Dynamo and were successfully sent to Puredata.  
 
Almost all of the Puredata patch was used without change. The only minor changes were (1) 
creation of a playback loop, so that the auralisations are played continuously, and (2) manipulating 
the UDP port so that it communicates with Dynamo properly. In the next chapter, different parts 
of the Dynamo tool is elaborated.

4. Discussion & Future Work 
 
This report was mainly about the description of possibilities in Enscape, Revit and Dynamo to 
connect the visual experience in VR, to an audio playback engine, which in the case of this project 
was Puredata. The initial scope of the project didn’t include the development of such a tool, and it 
was aiming more at deploying the available tools in Unity in order to perform soundscape 
perception-related tests on subjects. However, through the course of the project valuable 
experience was gained in creating tools in Dynamo and besides, the developed tool, serves the 
initial purpose, i.e. evaluation of soundscape in virtual reality.  
 
Still, evaluation experiments need to be designed in a way that the subjects aren’t moving, and 
they are standing on the evaluation spots instead due to the large latencies. The moving process 
might be done using the “teleport” options in the VR controllers, or with no sounds during the 
movement.  
 
There are several tasks that can be done as the continuation of this project in order to make the 
most out of the tool, or as an improvement to the idea. The following is a review of these 
suggestions: 
 
4.1. Performing psychoacoustic tests for soundscape evaluation 
 
This tool can be used further to evaluate how “pleasant” or “unpleasant” each point (point 1, 2, 
3 and 4) is. This could be done by asking subjects to stand on the corresponding points while 
looking around and listening to the soundscape.  
 
Several soundscape indices are currently in frequent use (Brown et al 2011), but in the present 
project “pleasantness” was intended to be selected as the descriptor. Seven-step or eleven-step 
scales of evaluation could be used to quantify the soundscape index. Tables 4.1 and 4.2 show 
examples of these scales (Kang and Zhang, 2010; Sanchez et al, 2017).

Table 4.2. Eleven step scale 
 
 
 
By performing experiments on subjects and analysing the results it becomes possible to check if 
there is a statistically significant difference between the four points’ pleasantness. Since the relative 
sound pressure level of the sources are changing along the path, each point will be the 
representative of one sound source. Then if the pleasantness is significantly different, e.g. from the 
drill to water feature (points 3 and 2), it might be possible to find a correlation between the sources

4.1.1. Challenges 
 
In order to perform the task that was mentioned in 4.1, it is necessary to make sure of two things: 
(1) the test subjects need to have normal hearing, and (2) the sound pressure level at ear should be 
measured in order to have a correct validation of the simulated auralisations. Although Odeon 
provides the SPL grid, but since there are inaccuracies in the original signal files and because they 
have been pre-processed, it is necessary to make sure that the relative SPLs are correctly measured. 
These might appear as challenges of the suggested task.  
 
4.2. Evaluating the degree of immersion and realism 
 
Using the same scaling system it is possible to also ask the participants questions regarding 
immersion and realism. The same approach has been adopted by Sanchez et al (2017), where they 
played back the sound that was recorded while crossing over a bridge. The sound was combined 
with a graphical simulation in VR, and the immersion and realism was also evaluated though 
questionnaires. Figure immersion shows a typical way of showing the results (Sanchez et al, 2017).  
 
 
Figure immersion. Evaluation of immersion and realism in a VR test (from Sanchez et al, 2017).  
 
4.3. Enhancing the speed of data communication to hear the audio in real-time 
 
As it was explained in this report, there is a big latency between the player movement and the 
audio playback. This might be because of the heavy Revit processes, or because the Enscape camera 
is shown on Revit and Dynamo needs to access that camera, or even because of the if statements 
in the Dynamo script. Since the Puredata patch has been working real-time with Unity through C# 
scripts, the problem doesn’t lie in the Puredata patch.  
 
However, in order to speed up the data communication in the present package, eliminating as 
many middle agents as possible would probably help. For example it might be useful to check if 
the Dynamo script can be written on Revit SDK without needing the Dynamo node groups. Or 
perhaps with collaboration with the Enscape team, this whole audio engine could be translated into 
c++ and implemented directly into Enscape so that no extra Revit activity is necessary.

5. Conclusion 
 
The possibility of creating a tool in Enscape VR to create subjective auralisation tests were 
examined in this project. A tool was developed in Dynamo Seated mode in order to extract the 
position and orientation information of the player. The data was then sent to Puredata through 
UDP. The whole package worked, meaning that as standing on an arbitrary spot, it was possible to 
hear the combined auralisations, i.e. the soundscape. However as the player started to walk (in the 
HTC Walkmode/Seated VR) there was a considerable latency in transferring the position and 
orientation data to Puredata. This made it almost impossible to walk in the design and naturally 
hear the sounds, as after a movement, it was necessary to release the controllers’ buttons, and allow 
the data to be transferred to Puredata. 
 However, in terms of evaluating the soundscape on the 4 defined positions, it is possible to use 
this combined tool, as there can be enough amount of time, like about 20-30 seconds for the player 
to hear the soundscape in that position.

Appendix B. Fixing the relative coordinates for Roomscale VR  
 
If the Roomscale mode is chosen for the experiment, it is needed  to scale the motion of the 
player. Besides it is necessary to create relative coordinates so that the position and orientation of 
the head is constantly measured with respect to the relative origin of coordinates. For this purpose 
it is necessary to pay attention to the fact that Revit has three different origins. The following is a 
quick guide to adjust these origins for the Roomscale mode.  
 
Fixing the coordinate mismatch: 
The origin in Revit (Basepoint) should be moved to the favorable position for the origin. (For 
example the corner of the experiment geometry).  
It should be made sure that the grid sizing in Dynamo is based on mm, as we are working with 
Revit, and Revit is set to mm by default. (The Dynamo background grid will disappear since for 
the present version (2) there is no option available for scaling it.) 
 
In order to move the Origin in Revit it is necessary to check the boxes for “Project Base Point” 
and “Survey Point”, in Visibility/Graphic Overrides menu. 
 
 
 
Then it is possible to pick the Base point, and unpin it from the drawing. Then it has to be moved 
it to the favorable point.
