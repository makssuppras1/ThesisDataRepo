Abstract
With the current prevalence of RESTful APIs, testing becomes increasingly important.
Manually hand­writing tests for RESTful APIs can be costly and inefficient, due to the
large number of possible parameters and scenarios. One approach that addresses this
issue is model­based testing, where tests are automatically derived from models that for­
mally describe the functionality and behavior of the system. SeTTS (Session Type Test
Synthesis) is one tool that has shown great potential in the model­based testing of REST­
ful APIs, as it provides an external domain­specific language (DSL) that can be used to
construct and execute models. However, SeTTS exhibits some issues regarding com­
plexity and expressiveness. This project aimed to address these issues by developing an
internal DSL named MoBaTS (Model­based Test Synthesis). MoBaTS carries the same
core functionality as SeTTS, but has shown to be less complex, and more expressive.
Moreover, it includes additional graphing features for visualizing models. However, Mo­
BaTS lacks extensive testing, especially on real­world systems, which could be a good
starting point for future work.

Introduction
The use of RESTful APIs in modern web development has become widely spread in re­
cent years. According to a survey done by Akamai [1], 83% of all internet traffic belongs
to API­based services. REST (Representational State Transfer) is by far the most used
architectural style for building web APIs, having over 90% adoption rate [2]. Moreover,
given the proliferation of microservices and the trend toward modular systems, APIs have
become a core building block in many modern software systems. With such prevalence,
testing APIs becomes increasingly important to ensure the reliability, efficiency, and se­
curity of web services. The testing of RESTful APIs, however, is notoriously challenging.
This is due to the complexity associated with having multiple endpoints, input parame­
ters, and output formats, which creates a significantly large number of combinations to
be tested. To address this challenge, many have resorted to the automatic generation
of tests. Fully­automatic test generation typically relies on an OpenAPI specification [3],
which is a standard for documenting RESTful APIs. While this approach is highly effec­
tive at producing a large number of tests, it does not take into consideration the possible
implicit dependencies between endpoints. A request in isolation can behave differently
compared to when being a part of a sequence of requests, and it is not possible to infer
such dependencies from the OpenAPI specification. Model­based testing addresses this
issue, as tests are generated from a model that formally describes the behavior of the
application. Using this method, testers can formalize the implicit dependencies between
requests in the constructed models.
One tool that uses such an approach is SeTTS (Session Type Test Synthesis) [4]. SeTTS
provides an external domain­specific language (DSL) that can be used to write models and
automatically generate tests from them. SeTTS has shown great potential in addressing
some of the challenges associated with testing RESTful APIs. However, being an external
DSL, SeTTS carries a number of complexities both in terms of usage and maintenance.
Moreover, the grammar of the DSL is based on session types [5], which significantly re­
stricts the expressiveness of the language. Therefore, the aim of this project is to design
and develop an internal DSL for the model­based testing of RESTful APIs. Internal DSLs
are embedded in a host language, therefore they can utilize the host language’s many fea­
tures. The goal of the internal DSL is to match SeTTS’s functionality, as well as address
its complexity and expressiveness issues.
This paper is structured as follows: Chapter 2 introduces a few necessary theoretical con­
cepts. Chapter 3 expands on the problem that the project is trying to solve. Chapter 4
presents a solution to address the problem outlined in the previous chapter and the rea­
sons behind some of the design choices. Chapter 5 explores the implementation details
and usage of the proposed solution. Chapter 6 attempts to evaluate whether the imple­
mented solution addresses the problem. Chapter 7 discusses some of the limitations of
the solution, and potential future improvements. Finally, Chapter 8 concludes the work
that has been done.

Theoretical background
This chapter explores some of the core theoretical concepts that are relevant to the
project. These concepts include, but are not limited to, domain­specific languages, REST­
ful APIs, model­based testing, and functional programming. Theoretical discussions will
not be exclusive to this chapter, as they will be present in most chapters. However, this
chapter focuses on the foundation necessary to understand the problem that the project
is trying to solve and the proposed solution.
2.1
Domain­specific Languages
A domain­specific language (DSL) is a special­purpose language designed to solve prob­
lems that belong to a particular domain [6]. DSLs are quite common in the software
development world, some of the most common ones include SQL, CSS, regular expres­
sions, and HTML to name a few. The main advantage of DSLs is that they provide a
higher level of abstraction to solve specific domain problems. This has several benefits;
it increases productivity, as solving domain problems becomes less complex. Moreover,
this also increases the readability of the code and reduces the frequency of bugs. Finally,
DSLs are typically easier to learn and are more ”beginner­friendly” to non­developers.
This opens the door for domain experts to interact (understand, modify, and validate) the
domain­specific solutions written in the DSL. There are two types of DSLs:
Internal
(or embedded) DSLs are embedded in a host language. This means that the
DSL uses the same syntax and semantics as the host language, and is developed using
the host’s language constructs (e.g. enumerations in Scala 3). This makes embedded
DSLs (eDSLs) easier to develop and use, as they don’t require an independent compiler.
However, this constrains the syntax and the semantics of the eDSL to be closely related
to the host language.
External
DSLs are independent of any host language and define their own language
processing infrastructure (e.g. compiler). This adds additional complexity in developing
and maintaining the DSL, however, it provides greater control over the parsing and the
grammar of the language.
2.2
RESTful APIs
An API (Application Programming Interface) is a program that is designed to provide an
interface to another program. The design of an API is dependent on its purpose and
usage context. When it comes to web services, a ”REST” design pattern is often used.
The REST (Representational State Transfer) architectural style for software outlines a
variety of restrictions and characteristics for creating web APIs. An API that adheres to
the REST architectural constraints and style is referred to as a RESTful API. There are 5
key constraints that make a web service a RESTful API [7]
1. Uniform interface: There should be a uniform way to interact with the API that is
consistent across all endpoints, irrespective of the client application or device.
2. Client­server: The RESTful APIs are based around a client­server architecture. A
key idea in REST is the concept of storing data as resources on the server. Each
resource is identified by a unique URI, which is used to access and interact with

the resource. A client would typically interact with the resource using the HTTP
methods: GET, POST, PUT, and DELETE.
3. Stateless: All the information necessary to process a request must be contained in
the request itself. In other words, the server should not store anything about the
previous HTTP requests that the client made. An important distinction should be
made here, while the requests and their processing is stateless, the requests are
necessarily changing the state of the server, thus there’s a ”statefulness” aspect to
RESTful APIs which will be further discussed in section 3.1.
4. Cacheable: Resources must declare whether they’re cacheable or not. Cashing is
preferred as it improves performance both for the client as well as the server.
5. Layered system: RESTful APIs should be designed to be used in a layered system,
where each layer has a well­defined interface for communication, and knows nothing
about the other layers that are not in direct communication with itself.

2.3
Model­based testing
Model­based testing refers to the techniques in which test cases are automatically de­
rived from models, rather than being manually hand­written. The models represent and
describe (formally) the functionality and behavior of the system under test (SUT). The
models typically represent a form of a finite state machine [10], as the behavior of many
systems can be represented by changes in state as a response to actions.
The main benefit of Model­based testing (MBT) is the facilitation of test automation that
goes well beyond the mere automatic execution of manually crafted test cases. ”It allows
for the algorithmic generation of large amounts of test cases, including their expected
results, completely automatically and correctly from the model” [11]. This results in several
advantages:
1. Efficiency: The ability to generate large amounts of tests in a short period of time
(compared to manual testing).
2. Accuracy: The ability to generate edge­case scenarios that might be overlooked in
manual testing.
3. Maintenance: Only having to maintain and refactor models, and not individual test
cases.

Model­based testing also has its disadvantages, namely the strong reliance on the model.
The effectiveness and accuracy of the tests are only as good as the model that generates
them, therefore it is important to have well­designed models. Moreover, as systems get
bigger, the complexity of the models increases, making them potentially difficult to rea­
son about. Finally, crafting well­designed models might require special knowledge and
expertise in the domain, which might not be accessible to every tester.
2.4
Functional programming
There are many definitions describing functional programming, a succinct one is given by
Alvin Alexander [12]:
Functional programming is a way of writing software applications using only pure functions
and immutable values.
In essence, a pure function receives some input and computes an output only depending
on the input, and does nothing else. Mathematical functions are pure, for example, a
function for a straight line (e.g. f(x) = 2x) is pure in the sense that it only computes a
value from a given input without any side effects, i.e. it does not modify or depend on
some external state. More formally, a pure function has the following properties [6]:
1. They do not modify parameters, instead, they return a new value.
2. They do not modify any external context/state (database, files, etc.).
3. They do not depend on any external context/state, but only depend on the given
parameters.
While these properties are constraining, they give rise to an important property called
referential transparency, which allows an expression to be replaced with its corresponding
value without changing the meaning or behavior of the program. In terms of the previous
straight­line example, if the program contains the expression 2x, it can be replaced with
f(x). This property (among others), makes programming with pure functions have the
following advantages [13]:
Declarative code
Arguably the essence of functional programming, which makes the
code:
• Easier to write. As code becomes shorter and more succinct.
• Easier to read and reason about. For impure functions, one can think of side effects
as hidden inputs and outputs, therefore their signature might not mean much. The
signature of a pure function however is much more meaningful, as it provides an
accurate overview of the function.
Debugging
As pure functions only depend on their input to produce output, debugging
becomes easier. One could follow the stack trace to find errors, without worrying about
external factors.
Testing
The lack of hidden states and side effects makes testing pure functions signifi­
cantly easier. Moreover, it opens the door for property­based testing, which is a technique
to automatically generate a large number of inputs to test a function, and verify the output
against user­defined properties [14].

Concurrency
Code that doesn’t mutate or rely on any external state, is suitable for writ­
ing concurrent programs. Without resource sharing, threads won’t run into race conditions
or deadlocks, which are notoriously difficult to deal with in imperative applications.
2.4.1
Modularity
Programming with pure functions that are free from side effects has many advantages,
however, in reality, side effects are what makes programs ”do things”. In other words, a
stateless program that is only concerned with converting inputs to outputs does not have
many use cases in terms of modern software development. Most commercial applications
today rely heavily on side effects, whether it’s interacting with databases, external APIs, or
local file systems. Therefore the question becomes how to write ”useful” applications that
take advantage of the benefits of functional programming. The answer to that question is
modularity.
Modularity in this context refers to the design pattern of decomposing an application into
two types of modules: Functional modules containing pure functions, and non­functional
modules containing the impure functions.

Impure code
External environment
Figure 2.1: Illustration of modularity in a functional programming application.
The idea is to have the application consist of as many pure parts as possible, and have an
external impure layer that handles side effects such as communication to the outside world
(Figure 2.1). The declarative pure core of the application would describe what needs to be
done, and the impure modules would be responsible for executing what needs to be done
[15]. This way of organizing applications is referred to as the interpreter design pattern
[16]. The way that the pure functions describe ”what needs to be done” is by creating a
DSL that describes the computation, and then passing that description to an interpreter
(impure code) that executes the computation.
In functional programming, a common way to define computation as a data structure is
by using the free monad [17]. Explaining the theory behind monads is beyond the scope
of this project, however, to put it simply, monads are typically used to chain computation.
The free monad is a special type of monad that is ”free” from interpretation. In other words,
computation is chained but not interpreted, creating an AST (abstract syntax tree). There­
fore, the free monad is a powerful way to represent computation in a pure and declarative
way. Moreover, the decoupling between the description and execution of computation
makes it possible to have different interpreters for the same DSL.

3
Problem setting
This chapter provides context to the problem that the project is trying to solve. As the
problem is related to testing RESTful APIs, the chapter explores some of the challenges
associated with this area of testing. Moreover, the chapter explores some of the state­of­
the­art approaches that are used to address these challenges. One tool that has shown
great potential is SeTTS (Session Type Test Synthesis) [4]. However, SeTTS comes with
its own challenges. Thus, the problem that this project is trying to solve is to develop an
alternative to SeTTS that attempts to address its shortcomings.
3.1
Testing RESTful APIs
There are many different types of testing when it comes to RESTful APIs: functional
testing, performance, and security to mention a few [18]. This project mainly considers
functional testing. For a given request, the response should have the correct headers,
response code, and body. If not, a meaningful error should be returned. The functional
testing of RESTful APIs is notoriously challenging. A few of these challenges are men­
tioned below:
Parameter validation
Parameter validation is the process of ensuring that the parame­
ters passed to the API conform to the expected format, and are valid with regards to type,
range, format, and whether the specific parameter is required or optional. The large num­
ber of parameters, in addition to all the different validation types, results in a significant
number of combinations.
Cost
The vast number of combinations to be tested in a RESTful API makes manual
testing costly. Corradini et al. [19] point this out by mentioning that ”manually writing all
test cases for a REST API can be tedious, time­consuming and expensive.” Moreover,
many of the tests might only slightly differ, potentially leading developers to copy/paste
tests, which is error­prone, and expensive to maintain [18].
Statefulness
While RESTful APIs are famously categorized as being stateless (as men­
tioned in section 2.2), the processing of requests on the server could potentially change
the state of the server. Therefore, a request in isolation could have a different response
compared to when being a part of a sequence of requests. In other words, requests might
have implicit dependencies on each other. For example, looking back at the owner API in
Table 2.1, requesting the second endpoint (GET) could result in different responses based
on the state of the server. If a POST request has been sent beforehand to create the owner,
then the GET response would contain the appropriate resource, with the response code
200 (OK). On the other hand, if the resource wasn’t created, then the GET request would
fail without returning an owner, and the response code would be 404 (not found). More­
over, the resource could have been created by the POST request, and then deleted later
using the DELETE request. Thus, there could be a significantly large number of different
sequences that could potentially result in unexpected behaviors, and writing manual tests
for each scenario might not be realistic.
3.2
SeTTS: Session Type Test Synthesis
To address the aforementioned challenges in testing RESTful APIs, many have resorted
to automated testing.

The automated testing approaches fall into two categories: fully­automatic, or model­
based [4]. Fully­automatic testers typically take an OpenAPI specification as input and
use it to automatically generate random tests that interact with the system. Some of these
tools also try to infer dependencies between the endpoints using the OpenAPI specifi­
cation. For example, RestTestGen [20] creates consumer/producer dependency graphs
between endpoints, which describe how different requests might produce parameters that
are required by other requests. However, this type of dependency does not provide much
information regarding the state of the server and is more tailored toward providing the
correct parameters to the random tests. RESTler [21] is another fully­automatic tester
that, in addition to inferring dependencies between requests, also tries to detect faulty re­
quest sequences using trial­and­error as a way to prune invalid requests from the search
space. Nevertheless, as the OpenAPI specifications do not typically include information
on the effects of request ordering, the fully­automatic tools cannot perform nor infer the
semantics behind complex sequences of independent requests [4].
Model­based testing (MBT) is another approach for the automated testing of RESTful
APIs. Instead of automatically generating random tests based on the API specification,
MBT requires testers to provide a model which is used to automatically generate tests.
Many of the already existing tools either focus on modeling the behavior of single API
operations in isolation [18], or do not consider RESTful web services [22, 23]
SeTTS (Session Type Test Synthesis) [4] is a model­based approach for testing RESTful
APIs implemented in Scala 2. It provides an external DSL that a tester can use to define
models for their API based on session types [5]. SeTTS takes an OpenAPI specification
as input and uses an OpenAPI generator [24] to generate a REST client for the given
specification. The tester can then write a model using the DSL’s session types syntax
and the generated client endpoints. SeTTS addresses all three challenges presented in
section 3.1, being a model­based automatic tester addresses the inefficiencies of writing
manual tests and the large space of different parameter combinations. Moreover, SeTTS
provides a DSL that allows testers to write complex models that capture information about
the state interaction with the SUT (system under test).

3.3
Problem statement
Despite the fact that SeTTS has shown potential in addressing some of the main difficulties
in testing RESTful APIs, it comes with its own challenges.
Complexity
SeTTS exhibits complexity both in terms of development/maintenance and
usage. As it is an external DSL, SeTTS includes a transpiler (source­to­source compiler).
This increases the complexity of maintaining the tool, as well as extending it to include new
features. Moreover, binding variables to response data is also not ideal. For example,
looking at the model in Listing 2, the tester would need to implement helper methods such
as setOwnerId (line 3) to bind the owner ID to a global variable. Having such global state
adds additional complexity to the writing and debugging process of tests.
Regarding usage, the tester needs to get familiar with the augmented session types syn­
tax, and writing models without out­of­the­box syntax highlighting. Moreover, the session
types syntax imposes constraints that could lead to code duplication. For example, con­

Another outcome of constraining the DSL to the augmented session

Finally, it is not possible to create non­regular models using SeTTS. A non­regular lan­
guage is one that cannot be specified by a regular expression [25]. For example, execut­

3.3.1
Problem summary
The problem that this project is attempting to solve is to find a model­based approach
for testing RESTful APIs that is more expressive, and less complex than SeTTS, in both
maintainability and usage. Thus, the solution would be evaluated based on 3 aspects:
whether it is as reliable as SeTTS, less complex, and more expressive.

4
Solution design
This chapter aims to propose a solution to solve the problems presented in Chapter 3. The
aim is to develop a solution that has the core functionality of SeTTS, but also addresses
some of its disadvantages. The chapter proposes developing an internal DSL in Scala 3,
and provides reasons why this solution would address the aforementioned problems.
4.1
Solution proposal
The project aims to solve the aforementioned problems by designing and implementing an
embedded DSL using Scala 3 [26]. The eDSL would be used to create models similar to
the ones proposed by SeTTS (section 3.2), i.e. descriptions consisting of REST request­
s/responses, as well as branching and assertions. It should also be possible to ”run” the
models by executing the requests, parsing responses, and verifying the assertions.
4.1.1
Why eDSL?
Using an internal DSL instead of an external one would significantly reduce some of the
complexity present in SeTTS today. Regarding implementation and maintainability, an
internal DSL would re­use the host language’s compiler and language constructs. This
provides many features out­of­the­box, such as lexing, parsing, type­checking, code gen­
eration, and variable binding. Such features could be complicated to implement and main­
tain, thus relying on the host language would not only significantly simplify the code base,
but also present a higher level of confidence for the users regarding the correctness and
security of the compiler. Regarding usage, if the tester has some experience with pro­
gramming, using an internal DSL would be more familiar and user­friendly, as they would
be able to use an IDE (Integrated development environment) made specifically for the
host language, which provides syntax highlighting, auto­completion, and static analysis.
The expressiveness issue is not directly related to whether the DSL is external or em­
bedded, but rather to how the language is designed. In fact, a developer implementing
an external DSL would have significantly more freedom, as they can decide how the lan­
guage is parsed, and how the code is generated. However, considering the project’s
domain and use cases (testing RESTful APIs), a simple embedded DSL would be more
expressive out­of­the­box. Given that the embedded DSL follows the principles discussed
in section 2.4, especially modularity, the eDSL would have sequencing/free composition,
and would not be limited to regular models. In fact, it would be more costly to restrict the
eDSL to be as expressive as SeTTS’s session types.
4.1.2
Why Scala 3?
Functional languages are often used to implement internal DSLs. This is due to the fact
that the programming style, as well as the features included in most functional languages,
are suitable for building eDSLs. Scala 3 was chosen for this project. While it’s a relatively
new language, its predecessor is one of the most widely used functional languages in
the industry. Nevertheless, any functional language could have been chosen, as most
languages share with Scala 3 the following attributes that make it a good candidate for
this project
Declarative style
As mentioned in section 2.4, the declarative style of functional pro­
gramming makes the DSL easier to develop, more expressive, and modular. Regarding
usage, this makes the language more accessible for some of the domain experts that
might not have experience with programming.

Strong type system
Scala 3 is a strongly statically typed language. Having a strong
type system in the host language allows the DSL to utilize its benefits, which include
catching type errors at compile time, and ensuring that the models are correctly typed.

dynamic languages. This makes it more readable and user­friendly, without sacrificing
type safety.

namely sum and product. Sum types are alterations, e.g. A | B, meaning either A or B,
while product types are a combination of different types (A and B)

Arguably the most essential feature in the DSL, as it facilitates the chaining of computation.

1. Context­dependent: Let m1 and m2 be two models that need to be chained. It should
be possible for m2 to access and use the result of m1.
2. Type agnostic: It should be possible to chain models of different types.
The structure that facilitates such sequencing is shown in Listing 14. The GADT case

type of the sequence (R). This allows models within the chain to have ”intermediate” types,
which are transformed to the ”final” type by the continuation function.

5.3.1
Motivation
Models are the core element in model­based testing. As mentioned in section 2.3, the
automatic­generated tests are as good as the models they’re generated from. Therefore,
in addition to having tools to streamline the development of well­typed models, being able
to understand and reason about models is essential. One approach to facilitate that is to
visually represent models. A visual representation would get the tester a quick overview
of the behavior that the model is trying to imitate, which makes it easier to communicate
ideas between testers and domain experts. The goal is to have a simple labeled state
transition system [35] similar to the one shown in Figure 3.2. Finally, one of the reasons
that Scala 3 was chosen to be the host language of the eDSL is due to its advanced
metaprogramming capabilities, which can be used (among others) to inspect the models
at compile­time.

The general idea behind graphing a sequence operation (>>) is to recursively graph the left

• For a given model graph, an entry node is the first node in the graph. The terms
entry node and source node are relatively interchangeable. Entry is used in the
context of a model graph, while source node is used in the context of an edge.
• Each model graph must only have 1 entry node.

The first level of evaluation is whether MoBaTS has the same functionality as SeTTS, as
it wouldn’t make sense to address SeTTS’s limitations if the core functionality is lacking.
To compare MoBaTS’s functionality, a SeTTS model for testing the PetClinic API [8] is
replicated in MoBaTS (Appendix C). The idea is to test whether MoBaTS catches the
same bugs that SeTTS has reported in [4] by running the same model multiple times, and
collecting all the errors produced. MoBaTS does indeed find the same bugs as SeTTS,
which are reported in Appendix D.
6.2
Q2: Complexity
Regarding development and maintainability, MoBaTS is significantly less complex. This
is due to the fact that it is an embedded DSL. As discussed in section 2.1, embedded
DSLs rely on the host’s compiler, making them lighter­weight. The DSL package in Mo­
BaTS contains approximately 300 lines of code, which is 4 times less than SeTTS’s driver
package (∼1400 as of Jan 2023). This significantly reduces the complexity of maintaining
and extending the language.
Regarding usage, MoBaTS’s free composition of models makes them highly modular.
Users can form small and simple models to test in isolation, and then combine them to
form more complex models. This modularity can be seen when comparing the PetClinic
models in Appendix C, where MoBaTS is able to split the individual API tests as different
functions. This does not only increase the readability of the model, but also gives the
ability to rapidly debug the individual parts. Finally, MoBaTS benefits from Scala’s out­of­
the­box syntax highlighting and static analysis, which further decreases the complexity of
usage.

7
Limitations and future work
Despite the fact that MoBaTS has shown good potential in being a model­based language
for testing RESTful APIs, and addressing SeTTS’s difficulties, it has its own limitations.
This chapter discusses these limitations, as well as potential improvements and future
work.
7.1
Limitations
MoBaTS’s main limitation is the lack of testing on real­world applications. For the purpose
of this project, only the PetClinic API was used for testing. Thus, to confirm the tool’s
potential, more real­world testing is required. The testing should not only be limited to the
scope of applications, but also users. One of the main arguments that this project presents
is that MoBaTS is lightweight and easy to work with compared to SeTTS. However, it is
difficult to prove such a claim without having real­world testers that are not familiar with
either tool.
Another potential limitation is the lack of formal theory behind the communication protocol
that MoBaTS uses. While SeTTS’s augmented session types are highly restrictive, the
protocol is well understood. This can potentially be an advantage in the future if SeTTS
is extended to do formal proofs on its models.
7.2
Future work
The first step regarding future work is to address the aforementioned limitations in Mo­
BaTS. Therefore testing the tool on different environments should be a priority for future
work. Regarding syntax, it is possible to add a restricted mode in MoBaTS that would
make it as restrictive as SeTTS. The general idea would be to implement a prefix opera­
tor similar to SeTTS’s (·) notation, which ensures that the first model in a sequence would
always be a request/response pair. However, this extension should be put on hold until
the benefits of the restriction are identified by SeTTS.
Beyond addressing MoBaTS’s limitations, there’s significant potential for the tool to grow.
Regarding expressiveness, an idea is to add a GADT case for models that can be inter­
preted in parallel, as some models may not need to be interpreted in a sequence. For
example, in the PetClinic API, creating a vet and a pet does not need to happen in se­
quence.
Regarding usability, the processes of starting with an OpenAPI specification and end­
ing with writing models could be streamlined. A user­facing application that takes the
specification as input, and immediately allows the user to write models would significantly
increase the usability of the tool. Finally, adding additional graphing capabilities would
further increase the usability of the tool. One idea is to combine the logging and graphing
features to produce a graph of a certain test run. This could be highlighting a path on the
graph showing the latest test.
Finally, it is worth reflecting on the fact that despite the similar functionality, SeTTS and
MoBaTS are inherently different tools, and could serve different purposes in the future.
It is likely that after more real­world testing is done, some additional requirements might
arise that could be more suitable to implement in an external DSL.

8
Conclusion
The purpose of this project was to address some of the issues in SeTTS regarding com­
plexity and expressiveness. The project managed to address these issues by developing
an internal DSL (MoBaTS) that carries the same functionality as SeTTS, but is less com­
plex and more expressive. Moreover, MoBaTS includes additional graphing capabilities
that can be highly useful when designing and developing models for testing RESTful APIs.
MoBaTS, however, comes with its own limitations, which mainly revolve around the lack of
testing on real­world systems. Finally, there’s a great potential for improvement regarding
future work, both in terms of the language, and visualization.
