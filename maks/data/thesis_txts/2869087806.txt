Abstract
Swarm protocols are a formalism for specifying and verifying the behavior of a set of
distributed peers, who collaborate on a common task with no central coordination. The
participants of a swarm protocol, referred to as machines, communicate by emitting events
and propagating them asynchronously to the rest of the machines participating in the
protocol. Machines may progress at any time without a complete view of the state of
the protocol or a network connection. This means that a system specified by a swarm
protocol is highly available.
It also means that inconsistencies between the machines‚Äô
views of the current execution may arise. A swarm protocol that satisfies a number of
well-formedness conditions is guaranteed to reach eventual consensus. This means that all
machines participating in a well-formed swarm protocol agree on the state of the execution
of the protocol once all events have spread to every machine.
Furthermore, any such
execution is faithful to the specification of the swarm protocol.
We study how to compose different swarm protocols into a single swarm protocol. This
allows us to design complex systems by combining smaller protocols, that are developed
and verified independently. It also allows us to reuse machines implementing some swarm
protocol by automatically adapting them to consume events from other protocols to make
them correct for a composition of their original protocol and these other protocols. We
develop novel methods and tools for verifying the well-formedness of a composition of
swarm protocols, for inferring the conditions under which a number of protocols can be
safely composed and for automatically extending machine implementations to become
correct machines in a composition.

1
Introduction
A distributed system is a collection of independent components located at different com-
puters that collaborate by communicating over a network.
One of the main tasks of
developing distributed systems is to arrange this collaboration.
1.1
Swarm Protocols
Swarm protocols [1] are a formalism for specifying and verifying how a collection of nodes
of a distributed system collaborate in order to accomplish a common task with no central
coordination. We describe this collaboration from a global point of view and obtain local
specifications of the participants, called machines, using a projection operation. Machines
may, depending on the state of the execution of the protocol, invoke commands and
thereby emit events. These events are appended to a log local to the machine invoking
the command and propagated asynchronously to the rest of the machines in the swarm.
Upon receiving an event, a machine updates its log and computes its state. The state
computation depends only on a subset of the events in the log. These events are given by
the subscription of the machine.
Machines operate independently and may progress without accurate information about
the current state of the protocol or an active connection to a network. In spite of this, a
swarm protocol that satisfies a number of well-formedness conditions is guaranteed to reach
eventual consensus [1]. This means that all correctly implemented machines participating
in a run of a well-formed swarm protocol will eventually have agreeing local logs and,
thus, agree on the state of the execution. Furthermore, this execution is consistent with a
behavior specified by the protocol.
Actyx [2] develops middleware that enables the dissemination of events, the machine-runner
tool [3] that allows declaring and running machines, and the machine-check tool [4] for
checking the well-formedness of a swarm protocol and whether a number of machines
correctly implements some swarm protocol.
1.2
Swarm Protocol Compositions
Recent results [5] show that swarm protocols can be safely composed under certain condi-
tions using interfacing roles. When we compose two protocols using an interfacing role the
protocols run concurrently, but synchronize on commands performed by the interfacing
role. Composing protocols enables the modular design of swarm protocols and the reuse of
machine implementations. Instead of developing complex monolithic swarm protocols, we
can develop and verify smaller specialized swarm protocols and infer under which condi-
tions they can be composed and yield a well-formed swarm protocol. In general, to ensure
the well-formedness of a composition, machines implementing some swarm protocol have
to be adapted to subscribe to events from the other protocols of the composition.
In this thesis we study swarm protocol compositions.
We have three main objectives.
These are studying and developing methods and tools for:
‚Ä¢ Verifying the well-formedness of swarm protocols and compositions hereof.
‚Ä¢ Determining the conditions under which a number of well-formed swarm protocols
can be safely composed.

‚Ä¢ Automatically extending implementations of swarm protocols so that they behave
correctly in a composition.
We develop such methods and tools and implement them in the Actyx toolkit [6].
1.3
Contributions and Outline
In this thesis, we build on the work on swarm protocols that has been developed in [1]
and [5]. We advance this work by developing methods and tools for the safe composition
of swarm protocols. A special focus is placed on recent results [5] on the compositional
verification of swarm protocols using interfacing roles.
The main contributions of this thesis are:
A. We develop a set of inference rules for verifying the well-formedness of concurrency-
free swarm protocols (Section 3.1).
B. We develop a set of inference rules for verifying the well-formedness of compositions
of two concurrency-free swarm protocols (Section 5.1).
C. We extend the Actyx machine-check tool to allow the verification of swarm protocol
compositions and the generation of safe subscriptions (Chapters 5 and 6).
D. We extend the Actyx machine-runner tool to automatically transform machine im-
plementations into correct machines in swarm protocol compositions (Section 5 and
6).
E. We evaluate the relative performance and accuracy of four different algorithms for
generating safe subscriptions and two different algorithms for computing projections
(Chapter 7).
The remainder of this thesis is organized as follows.
Chapter 2 introduces the theory
of swarm protocols and Chapter 3 introduces techniques for reasoning about their well-
formedness. Chapter 4 extends this theory to compositions of swarm protocols and ex-
plains the notion of correct implementations of swarm protocols.
Chapter 5 develops
techniques for reasoning about swarm protocol compositions, methods to safely compose
implementations of swarm protocols and presents the design of our extension of the Actyx
toolkit. Chapter 6 presents the implementation of our extension to the Actyx toolkit.
Chapter 7 compares four algorithms for generating safe subscriptions for swarm protocol
compositions, focusing on relative performance and the size of the generated subscriptions,
and two algorithms for computing projections, focusing on relative performance. Finally,
Chapter 8 concludes the thesis and outlines directions for future work.

Background
Swarm protocol verification consists of two tasks. The first is the task of proving that
all participants of a swarm protocol eventually agree on the state of the execution of
the protocol. A well-formed protocol has this property. The second task is to prove the
correctness of a concrete implementation with respect to a swarm protocol.
In this chapter, we present the theory on which the rest of this thesis is based. We begin
with a running example to illustrate the basic mechanisms of swarm protocols and swarm
protocol verification. Then, we introduce formalisms for precise reasoning about swarm
protocols.
2.1
Running Example
Throughout this thesis we will repeatedly consider three swarm protocols describing dif-
ferent procedures carried out at an automated car assembly plant. Each of the protocols
describes how its participants collaborate in order to complete a part of the car assembly
process. The composition of the three describes the complete process of assembling a car.
We introduce the first of these protocols below.

2.1.1
Informal Reasoning
We first informally show that the swarm protocol Gwarehouse is well-formed. Our goal is to
show that the protocol exhibits properties guaranteeing that all participants of the swarm
protocol eventually agree on the state of the execution of the protocol.
The diagram in Figure 2.1 gives us a global view of the protocol. When the protocol is
running the participants operate independently and communicate only through events that
are emitted and propagated asynchronously when commands are executed. This means
that a participants‚Äô notion of the state of the execution is established only through the
events it observes.
We must make sure that roles perform commands only at times when we expect them
to. For instance, we want to avoid the situation where a forklift performs a get command
before a partID has been requested and the situation where a door repeatedly performs
the the close command. A swarm protocol in which these situations never arise is called
causal-consistent.
In our example, we could imagine that a door emits an event of type time right about the
same time as a transporter emits an event of type partID. We assume that events carry
logical timestamps that totally orders events. This is crucial to identify the correct branch
in cases of conflict: the earliest event determines which branch is taken and the events
that correspond to the losing branches are ignored. If, however, the transporter does not
observe the event emitted by the door or vice versa, the role that emitted the event of
the losing branch will be stuck waiting with a false belief about the global state of the
protocol.
If executions of Gwarehouse are to proceed as indicated in figure 2.1, we must make sure that
the participants of the protocol can reconcile and recover from such conflicts. To achieve
this, the participants must observe the events emitted at the branches in the protocol. A
swarm protocol where all roles eventually agree on the branch along which the protocol is
to proceed is called determinate.
Finally, we must ensure that each event type is emitted from only one state in the protocol
and that a state with no outgoing transitions can be reached from every state. We would
not accept, for instance, that partID was emitted on the transition from state 0 to state 1
and on the transition from state 2 to state 0 or that the transition from state 0 to state 3
did not exist. A swarm protocol that satisfies these properties is called confusion-free.
A swarm protocol is well-formed if it is causal-consistent, determinate and confusion-free.

2.2.2
Swarms
A collection of machines is referred to as a swarm. Each machine in a swarm implementing
some swarm protocol participates as some role of the protocol. Depending on its current
state, any machine may decide to invoke a command. The event emitted as a result of a
command invocation is appended to the local log of the machine invoking the command.
This ensures causality preservation since the event is inserted after all the events the
machine previously knew of. The event is then asynchronously propagated to the rest
of the swarm. To model how this propagation takes place and to model how machines
insert events received from the swarm into their local logs while preserving causality we
introduce the notion of a global log. Unlike the local log of a machine, the global log does
not exist in practice. In real implementations events propagate from one machine‚Äôs local
log to another machine‚Äôs local log.

The [Local] rule defines how a machine may invoke a command if the command is enabled
in the state the machine reaches after fully processing its local log.
When a machine
invokes a command, it appends the event type emitted by the command invocation to
its local log. The global log is updated using the merging operator. The resulting global
log is one of the possible logs that contain all events from the original global log and the
updated local log of the machine and preserve the respective orders from the two logs.
The rule allows a machine to have a partial view of the global log and still make progress.
The machine could even have a log containing events of a losing branch, meaning that
in the global log these events are ordered after some event that has lead the execution of
the protocol down some other branch that the machine is not aware of. In such case the
events emitted by the machine would be inserted in the global log in a way that reflects
this.

A swarm protocol specifies the collaboration of a swarm of machines.

2.3
Related Work
Behavioral Types
A behavioral type system can be thought of as a system for reasoning about the interactions
between a set of software entities. We can use behavioral type systems to reason about
concepts such as communication protocols, channels and web services by specifying them
in terms of the sequences of operations that are considered correct interactions between the
Methods and Tools for the Verification of Swarm Protocols
13
==PAGE:22==
involved parties [8][9]. The exact definition of the operations and the notion of correctness
depends on system under consideration.
Session types [9] are a form of behavioral types and come in different varieties. Binary
session types describe interactions between two parties while multiparty session types
extend this to the multiparty case. A session is some structured conversation consisting,
for instance, of the sequencing of messages, choices between different messages and the
repetition of certain patterns of communication through recursion. The structure of a
session is abstracted as a type against which we can check programs implementing the
session [10].
Multiparty Asynchronous Session Types
Multiparty asynchronous session types [11][12] are a formalism for specifying the intended
interactions between a set of distributed processes from a global perspective. A global
specification, referred to as a global type, can be projected onto its individual participants
to obtain local types. A typing system guarantees that processes typed with these local
types interact exactly as prescribed by their corresponding global type. This property is
called session fidelity. Additionally, typable implementations are deadlock-free.
The swarm protocol formalism is similar to the one introduced in [11] and [12] in that
a swarm protocol describes the intended interaction pattern from a global point of view
and can be projected over its participants to obtain local specifications against which we
can check the correctness of implementations. For both formalisms, communication is
asynchronous: sending a message does not block the sender. However, while the processes
in [11] and [12] use channels for communication, participants of swarm protocols exchange
logs.
Correctly implemented participants of a swarm protocol may perform conflicting com-
mands, but are guaranteed to eventually agree on the state of the execution of the pro-
tocol. This means that the formalism of swarm protocols allows correct implementations
to transiently deviate from the specification of the protocol. This is different from session
fidelity, which guarantees that the interactions of typable processes are exactly as specified
by the global type. The swarm protocol formalism abstracts away from the the number
of instances enacting each role. A session type in [11] and [12] is for a certain number of
parties.
Local-first Cooperation
Local-first cooperation [13] refers to the principle that to provide maximum availability, a
networked application should be able to progress independently even without a network
connection. To achieve this independence, all data needed for the application should be
stored on the device running the application.
Furthermore, in local-first cooperation,
devices should be able to communicate with the devices around it using a local network.
This is contrasted with a model where all information goes through a centralized server.
In lack of a central server to decide the current state of the distributed application, each
participant must be able to determine which data are valid and which are not. In [13] they
suggest eventually consistent distributed event logs as a way to achieve this. In distributed
event logs, events are totally ordered, and assuming that all nodes eventually receive all
events they end up with the same notion of the global state. This approach is similar to
the approach of swarm protocols, where events are propagated throughout the swarm and
appended to the local logs of machines in a way such that all machines eventually come
to the same conclusion about the global state of the protocol.

4
Composing Swarm Protocols
The theory presented in Chapter 2 only allows the monolithic design of swarm protocols
and the inference rules presented in Chapter 3 only allows us to reason about such proto-
cols. This means that using only the theory of the previous two chapters, we would have to
redesign and re-verify the protocol introduced in Example 2.1.1, Gwarehouse, if we wanted
a protocol that not only specified the operation of the warehouse, but also described how
the warehouse and the factory shop floor collaborate. Instead, we would like to be able to
create new swarm protocols by combining existing ones. This would allow us to design and
verify small swarm protocols made specifically for one task and combine and reuse these.
It would also allow the reuse of machine implementations and reduce the complexity of
the task of designing a swarm protocol.
This chapter is mostly based on an unpublished technical report by Florian Furbach [5] and
is structured as follows: We first extend the theory introduced in the Chapter 2 to allow the
composition of swarm protocols. We then revise our definition of well-formedness to allow
us to reason about compositions. Finally, we make our notion of correct implementations
precise.
We start by presenting an example that we will later be able to compose with Gwarehouse.

4.2

stemming from different protocols. We say that such events are concurrent. In Definition
2.2.9, we defined a variation of well-formedness that does not consider concurrency. These
definitions are stricter than necessary, when we consider compositions of protocols. We
can relax the well-formedness conditions to reduce the interactions between machines from
the different protocols of the composition while still maintaining eventual consensus and
fidelity to the protocol.

We relax our notion of invariance to account for any concurrency. Confusion about the
order of concurrent commands is permissible, since by our definition of concurrency the
ordering of concurrent commands does not matter. The components of the composition,
however, must have originally only enabled each of their commands in a single subterm.
In our new definition of determinacy, we filter out ‚Äòfalse‚Äô branches. That is, if a subterm
enables multiple commands, but these commands are concurrent, we do not consider them
as branching and require that all involved roles subscribe to the emitted events (item C.1.).
The rule in (item C.2.) describes joining event types. An event type e is joining if two or
more concurrent commands lead directly to a subterm that enables a command emitting
e and if these preceding concurrent event types are not concurrent with e. Only event
types emitted by an interfacing role can be joining. Item C.2. states that if some role
subscribes to some event type that occurs in some branch initiated by a joining event
type, then the role must subscribe to the joining event type and the concurrent event
types immediately preceding it. This rules enables correct branch tracking, a mechanism
that is needed for the safe composition of swarms. We have left out the fact that machines
must implement a branch tracking functionality from our treatment of well-formedness.
With branch tracking, each emitted event points to the last branching or joining event
that preceded it. This mechanism is part of what makes machines able to correctly discard
events from losing branches. We have made this omission to keep our presentation simple
and because we do not implement branch tracking in our automatic extension of machines.
The new definition of causal-consistency is the same as in Definition 2.2.9, except that
we do not require that roles subscribe to the event types emitted by commands that
are syntactically immediately before their own commands, but concurrent with their own
commands.

5
Design
In the beginning of Chapter 2 we described swarm protocol verification as the task of
showing that a swarm protocol is well-formed with respect to a given subscription and
showing that an implementation faithfully implements some protocol. We should be able
to perform these tasks for swarm protocol compositions as well. However, when we reason
about swarm protocol compositions we are more interested in inferring the conditions
under which a number of well-formed protocols can be safely composed and methods for
combining a number of correct implementations to obtain a correct implementation of a
swarm protocol composition.
This chapter presents the design of our tools for swarm protocol verification and automatic
machine extension. Our tools realize the three objectives we set out to achieve in Section
1.2. This means that our tools allow:
‚Ä¢ verifying the œÉ-well-formedness of swarm protocols and their compositions,
‚Ä¢ inferring subscriptions with respect to which a given swarm protocol composition
is well-formed, and
‚Ä¢ automatically extending machine implementations to behave correctly in a
swarm protocol composition.
The chapter is structured as follows: We first describe the methods we use to achieve
each of the three objectives. We then present our implementation from a high level and
describe how it integrates with the rest of the Actyx toolkit. Finally, we show an example
use of our tools.

We present four different algorithms for subscription generation. These are:
‚Ä¢ An exact algorithm generating the smallest subscription with respect to which a

5.2.1
The Exact Approach
We can use the rules given in Definition 5.1.1 for generating subscriptions. To do this,
we traverse the composition, match terms using the rules and add event types to the
subscriptions of roles where the rules assert that they should be present. By doing this
we get the smallest œÉ with respect to which the composition is well-formed. We therefore
call this approach to subscription generation the exact approach.
Using the exact approach, we have to explicitly construct the composition G1||...||Gn.
However, we would like to be able to avoid working with such explicit compositions.
There are two reasons for this.
The first reason is that the composition may become
large due to all the possible interleavings of concurrent events. For k swarm protocols,
each with N states, the composition of the protocols has up to kN states. We would
like to circumvent this exponential growth in complexity when we generate subscriptions.
Second, from a verification point of view, explicitly constructing the composition and
generating a subscription for this is the same as constructing and analyzing a monolithic
swarm protocol from scratch. This goes against our aim of being able to design modular
swarm protocols that can be verified independently and combined.

5.3
Automatic Extension of Implementations
When extending machines, we take as input a machine implementing a role in a swarm
protocol along with additional swarm protocols. We then automatically extend the ma-
chine to consume events from these protocols, making it an implementation of the role in
the composition of its original protocol and these other protocols. In other words, given
an implementation of some role R ‚ààG1, we automatically extend this machine to become
a correct implementation of R in G1 ||G2 || ¬∑ ¬∑ ¬∑ || Gn.
In an implementation, machine states and events may contain data. We call data contained
in states and events payload. The types of payloads are records denoted {x : T1, y : T2, ...},
where x and y are field names and T1 and T2 are types. When a machine receives an event
which is not out of order and to which it subscribes, a user defined function processing the
event is called before the machine transitions to a new state. We call such a function a
reaction. A reaction computes the payload of the target state of the transition and takes
two parameters: the context of the machine and the event itself. The context of a machine
in a given state contains among other things the name of the current state, the available
commands and the state payload. A command in the context of machine implementations
is a user defined function that emits an event. A command may use the state payload to
compute the payload of the event it emits.
The state that enabled some command in the original machine, may become ‚Äòduplicated‚Äô
a number of times in the extended machine due to concurrency. Moreover, due to concur-
rency the events that lead to the states enabling some command may not be the same as
those in the original protocol.
The challenge of extending machine implementations is to preserve the functionality of
the original machine in the extended machine. To achieve this we must correctly transfer
reactions to the extended machine and make sure that the states that enable some com-
mand in the extended machine have the same payload type as the state that enabled the
same command in the original machine.

tocols, subscription generation and implementation checking. The other handles machine
extension. These two libraries are extensions of, respectively, the machine-check [4] and
machine-runner [3] libraries. The original libraries support the verification of of simple
concurrency-free swarm protocols as presented in [1] and the declaration and execution
of machines implementing such protocols. The libraries are part of the Actyx [2] toolkit,
which also includes a middleware system that offers event dissemination based on the

The composition_types module defines various types, the most important of which are

for generating useful error reports that guide the user to the exact point in a protocol
where some well-formedness condition is not met. On successful verifications this list is
empty.

Generating well-formed subscriptions: The exact approach
We outlined the exact algorithm for generating well-formed subscriptions in Section 5.2.
We construct the composition using Definition 4.1.1 and apply the rules from Definition
5.1.1 to derive the smallest subscription with respect to which the composition is well-
formed and which contains the input subscription. When we apply the rules in this way,
we add event types to the subscriptions where it is required by the rules to satisfy well-
formedness.
We take an InterfacingSwarms and a Subscriptions as input and return a Result<Subscriptions
, ErrorReport>, which contains an ErrorReport only if the protocols do not interface or if
any of them is not confusion-free. As mentioned in Chapter 5, the composition need not

7
Evaluation
This section presents our empirical evaluation. We compare the four algorithms for gen-
erating well-formed subscriptions and the two for computing projections. We first discuss
the relevant differences between the algorithms, introduce the benchmarks, and describe
our experimental setup. We then present and interpret our results, in terms of perfor-
mance and accuracy of the different algorithms. Finally, we discuss potential threats to
validity.
7.1
Comparison of Algorithms
7.1.1
Subscription Generation
We compare the algorithm for generating the smallest well-formed subscription for a swarm
protocol and the three algorithms that generate an approximation of this smallest well-
formed subscription.
In the following, let G = G1 || . . . || GN and let Gi have K states and L transitions for
1 ‚â§i ‚â§N. We use the word transitions instead of commands because in a composition,
the same command could appear multiple places due to concurrency. The algorithms are
then summarized as follows:
‚Ä¢ Exact: This algorithm computes smallest well-formed subscription for G by con-
structing the composition and applying the rules of Definition 5.1.1. The time com-
plexity is linear in the number of transitions in the composition, but the number of
transitions in the composition is exponential in the number of protocols. This means
that the time complexity of this algorithm is is O(LN).
‚Ä¢ Fine-grained overapproximation: This algorithm visits all transitions of each proto-
col in isolation, computes the transitive closure of the after relation across all the
protocols using the Floyd-Warshall algorithm [16] and generates an approximation of
the smallest well-formed subscription. Its time complexity is then O(N ¬∑L+(N ¬∑L)3).
‚Ä¢ Medium-grained overapproximation: This algorithm works in a similar way to the
one described above.
The only difference is that all interfacing events (and the
branching events and concurrent events leading interfacing events required by this)
are added to the subscription of every role. The complexity is the same as for the
algorithm above.
‚Ä¢ Coarse-grained overapproximation: This algorithm visits all states of each protocol in
isolation. It then adds the events required by causal-consistency to the subscriptions
of roles. Finally, it adds all branching events, all interfacing events and all concurrent
events directly leading to interfacing events. The time complexity is O(N ¬∑ L).
7.1.2
Computing Projections
The two approaches for computing projections under comparison are:
‚Ä¢ Constructing the composition, then projecting over a role.
‚Ä¢ Projecting each protocol over the role separately before combining the projections.
The algorithm for composing protocols and composing projections is the same, but the
order of its application impacts the runtime of computing projections. Computing and
combining projections should often be faster than first constructing the composition and
Methods and Tools for the Verification of Swarm Protocols
61
==PAGE:70==
then projecting because a role typically subscribes to only a fraction of all the events in
the composition. That is, a role will only subscribe to a (small) number of branching
event types, interfacing event types and event types preceding interfacing events from the
protocols from which it does not ‚Äòoriginate‚Äô. As a result the projections of these protocols
over the role are typically small and combining them requires less work than constructing
the composition of the protocols and then projecting it over the role.

Benchmark Selection
Our benchmark suite consists of 796 example instances of the InterfacingSwarms type
randomly generated using the property testing framework Proptest [20].
Each of the
examples consists of up to 20 protocols, each of which consists of up to 9 roles each
performing up to 9 commands. Some of the protocols were generated such that on average
every tenth command initiates a branch and every tenth command leads to a protocol state
with multiple incoming commands. For some of the protocols this was every fifth command

We partitioned the benchmark suite into three groups. The examples in each group are
generated following a certain ‚Äòpattern‚Äô, by which we mean that the protocols in an example
are generated in such a way that their composition has a useful pattern. The patterns
roughly correspond to different use cases and are described below. Overall, we believe that
the examples in the benchmark suite gives us a good basis for comparing the performance
and exactness of the different approaches.

7.5
Threats to Validity
7.5.1
Benchmark Selection
As of yet, there does not exist any swarm protocol compositions used in industry. There-
fore, we rely on ‚Äòartificial‚Äô protocols that we generate ourselves for benchmarking. Assess-
ing how well the protocols we generated reflect real industry use is hard. We argue that
the refinement pattern is a realistic pattern for protocols used in a factory setting, where
each step of the production line corresponds to a protocol in the composition. We also
argue that the general pattern captures such a wide category of protocols that the results
obtained from protocols in this group are interesting. The pathological pattern is included
to show how the algorithms perform in extreme cases of concurrency. Together we think
that the examples cover a range of scenarios wide enough to allows us to make statements
about the performance and accuracy of the different approaches.
The values determining the probabilities for branches and loops in the algorithms gener-
ating the protocols were set to make the generated protocols appear structurally similar
to the examples shown in [5], [1], and [14]. With more industrial examples these values
could have been set to reflect real world examples of swarm protocols.
One could argue that it is not interesting to compare the accuracy of the results generated
by the medium- and coarse-grained algorithms, where we purposefully add all interfacing
events, with the results generated with the exact- and fine-grained algorithms. Adding
interfacing events is necessary in order to be able to compose machines correctly. Thus,

7.5.2
Maturity of Implementations
The time complexity of the exact approach is inherently greater than those of the approx-
imation approaches since the composition of the input protocols has to be constructed.
However, it is possible that the performance of the implementation of this approach and
the performance approximation functions could be improved with more development time.
The accuracy of the fine- and medium approximation approaches could be improved if we
could determine when behavior is lost in the composition. This is possible and an approach
to achieve it is discussed in Section 8.1. The implementation of machine minimization
could also be improved with more development time.

8
Conclusion
In this thesis, we have presented methods and tools for reasoning about swarm protocols
and compositions hereof. In particular, we have presented
A. a set of inference rules for proving the œÉ-well-formedness of concurrency-free swarm
protocols and an extension of these rules that considers the case of compositions of
two concurrency-free swarm protocols,
B. eÔ¨Äicient methods for generating subscriptions that allow the safe composition of
swarm protocols,
C. methods to automatically transform a correct implementation of a swarm protocol
into a correct implementation of a composition containing the swarm protocol, and
D. an extension of the Actyx toolkit that implements Item A. for any number of pro-
tocols and Items B. and C..
The number of states in a swarm protocol composition may be exponential in the number
of protocols in the composition. To circumvent this exponential growth, we developed
approximate representations that allow reasoning about compositions without explicitly
constructing them. The general observation, witnessed by Chapter 7, is that subscriptions
generated from approximate representations tend to be larger than necessary, but are
significantly faster to compute than those derived from explicit compositions.
To automatically extend a machine implementing some role to make it correct for a com-
position of protocols, we need the projection of the composition over the role. This can
be generated by either composing the protocols explicitly and then projecting or by pro-
jecting each protocol of the composition over the role and combining the results. The
latter approach is significantly faster to compute than the former, but requires that the
role subscribes to all interfacing events, even if this is not required by the well-formedness
conditions. This means that similar to the case for subscription generation, there is a
trade-off between runtime and precision in the projection methods.
8.1
Future Work
We identify three main directions for future work.
The first is making approximated
subscriptions more precise. By detecting when behavior is lost in the composition, the
approximations methods for generating subscriptions could yield the same result as the
subscriptions generated by analyzing the explicit composition (thereby losing their status
as approximation methods). When we compose two protocols, we lose looping behavior
if an interfacing event is involved in a loop in the first protocol, but not in the second.
If we identified these situations, we could compute the exact roles set for all branches,
allowing us to only add those event types required by the well-formedness conditions to
the generated subscriptions.
The second possible direction for future work is to reduce the size of projections after
having constructed them. The idea is to combine projections computed using an over-
approximated subscription that contains all interfacing event types and then remove the
transitions that consume event types added to the subscription to ensure correct synchro-
nization between the combined machines. This might also be realized by ‚Äòprojecting‚Äô the
projection w.r.t. to a more precise subscription.

all input machines. The composed state should be able to emit events from any of the
input states and all input states should perform their respective reaction when receiving
events. Implementing this and branch tracking would bring us closer to realizing the full
potential of swarm protocol compositions.
