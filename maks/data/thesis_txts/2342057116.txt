Abstract 
Each year, a huge amount of cargo is shipped around the world on container vessels operated by liner 
shipping companies. These companies offer a network consisting of a number of scheduled roundtrips, 
rotations, typically operating with a weekly frequency. The network determines a significant part of the 
costs incurred by the liner shipping companies, as e.g. fuel consumption, vessel capital costs and port 
calling costs are given by it. The Liner Shipping Network Design Problem (LSNDP) is thus an obvious 
candidate for application of operations research, as it is very difficult to design a good network manually.  
To be able to evaluate the quality of a solution, it must be decided how the containers should be flown 
through the network – which containers should go on which rotations. Solving this multicommodity 
flow problem (MCF) is a challenge in itself, which is tackled by applying a heuristic method due to the 
running time of exact methods. The capacity constraint of the arcs, representing vessels operating a 
rotation, is Lagrange relaxed. The relaxation implies that the capacity can be exceeded, but as this is not 
desirable, a Lagrange multiplier is added to the cost of using the attractive arcs. The Lagrange multipliers 
are tuned in an iterative process, where a shortest path problem for all cargos is solved in each iteration, 
providing an indication of which arcs are attractive and should have the Lagrange multiplier increased.  
The Lagrange multipliers are not sufficient to obtain a feasible and good solution, however, and a repair 
algorithm is thus implemented as a supplement. This algorithm forces a flow to become feasible by 
moving containers in excess of the capacity away, either by selecting another route or by rejecting the 
cargo. The solutions to the MCF are compared to those obtained by an exact method, which shows that 
an objective value of 5-6 % from the optimal value is achieved for the larger instances. The running time 
is reduced dramatically to only around 1/250 of the exact method. 
To be able to optimize the network, it is necessary to find a starting point with some initial rotations 
that can be improved. As the network is designed from scratch, such an initial network is not available 
and must be generated. This is done through an auxiliary network, where it is assumed that a direct, 
non-stop connection between all ports exists. Such a network is of course not economical to operate, so 
it must be ensured that only a few of the arcs, representing connections, are used to enable them to be 
covered economically by a realistic network of rotations. The cost of an arc is thus set depending on the 
current number of containers using it, such that the cost is highest for the first container and gradually 
decreases with an increasing number of containers. This pricing scheme provides an incentive for using 
arcs that are already active. For the method to function, it is necessary to flow each container 
individually through the network and adjust the arc costs in each iteration.  
The resulting auxiliary flow provides a basis for generating rotations, which is done with a greedy 
heuristic where the rotations are generated serially. For each rotation, the arc with the highest load not 
yet served by any rotation is selected as the first arc. From neighboring arcs to this one, the rotation is 
expanded by selecting the unserved arc with the highest load. This process continues until the rotation 
has reached a preset duration, after which the construction of the next rotation starts. By randomly 
selecting the durations, many different networks to choose from can be generated quickly. Results of 
the method show that the initial network is of relatively poor quality when compared to an optimized

network. As the purpose of the method is to get a starting point and not an optimized solution, this is as 
expected. 
The starting solution is optimized heuristically to get a good solution. Six different neighborhoods are 
defined for the local search procedure, of which four use a delta evaluation procedure. Delta evaluation 
allows a move to be evaluated without actually implementing it, which is a big advantage for the LSNDP, 
where an evaluation of the whole network involves a relatively time-consuming calculation of the 
multicommodity flow (MCF). The delta evaluation is conducted by creating a subgraph, covering only 
the rotation currently being modified. By using such a subgraph, the MCF can be calculated much faster 
than for the whole graph. Much of the cargo is transshipped between two rotations on the way from 
origin to destination, and to include this in the subgraph, where only one rotation is present, feeder arcs 
are introduced as a replacement for the missing rotations. The delta evaluation of moves by using the 
subgraph is not perfectly coherent with the actual effect of carrying out the move in the main graph, but 
is sufficiently good to identify attractive moves.  
The local search procedure is controlled by a metaheuristic inspired by the Variable Neighborhood 
Search (VNS) framework. With this framework, local optima are escaped by doing a random move, a 
shake. With the implementation chosen in this thesis, a shake is carried out only when a local optimum 
with respect to all neighborhoods is reached, as too much randomness is introduced if a shake is 
conducted more often. The algorithm is tested with different active neighborhoods to investigate if all 
neighborhoods contribute positively to the objective value. It is demonstrated that this is the case, and 
all six neighborhoods are thus included in the final algorithm. 
Computational tests of the algorithm are carried out on standardized data sets provided through LINER-
LIB and the results are compared to those obtained by Brouer et al. (2014a). Out of seven instances, 
Brouer et al. finds the best solution for two instances, while this thesis finds the best solution for four 
instances and provides a novel solution to the last and largest instance, all with a running time per 
replication of only a tenth of that applied by Brouer et al. Looking more closely at the results, the main 
problem is the choice of sailing speed, as it is generally very close to the minimum speed allowed. This 
is due to the calculation of optimal speed, which takes only the direct operating costs into account 
without considering e.g. extra revenue, which can be obtained by sailing faster and thus being able to do 
more roundtrips. The low sailing speeds lead to lack of capacity in several of the instances. It also gives 
relatively long transit times, meaning that much cargo is delivered too slowly compared to the time 
limits defined in LINER-LIB. It must be stressed, however, that time limits are not part of the problem 
definition, which can explain the poor results on this parameter. 
Overall, it can be concluded that the applied method successfully finds good solutions to the LSNDP and 
has great potential for further development. The speed calculation procedure obviously has room for 
improvement, but the local search procedure can probably also benefit from more neighborhoods, 
especially a neighborhood which can more effectively promote a hub-and-spoke structure. The most 
pressing extension of the algorithm is the introduction of time constraints to ensure a more customer-
oriented approach which emphasizes not only cost reduction, but also attractive travel times.

1 Introduction 
The liner shipping industry is a vital part of the global economy, facilitating much international trade by 
transporting large amounts of cargo around the world at low cost. A pair of sports shoes, for example, 
often retailing at around 100 Euros, can be transported from Asia to Europe at a cost of merely 0.2 Euros 
(Maersk Line, 2016a). This is only possible by combining a lot of units in a single container, as the 
container allows efficient handling at ports and on the vessels.  
The liner shipping industry has been growing rapidly for many years. From 2005 to 2014, the global 
volume of containerized cargo loaded increased by 68 %, which is almost 6 % per year on average. 
Looking back to 1980, the average, yearly growth has been even stronger at more than 8 % per year. In 
the same period of time, the general growth of international cargo carried at sea has been only 3 % per 
year, which means that the market share of containerized goods has increased considerably (Unctad, 
2015, p.6).  
Even though the environmental impact of container transport per ton-km is very low compared to other 
transport modes, the industry still contributes significantly to the global CO2 emissions due to the large 
volumes transported. In 2012, the international shipping industry was estimated to account for 2.2 % 
of the global CO2 emissions, of which approximately a quarter was caused by container vessels (IMO, 
2014). In 2015, there were around 5,100 container vessels in operation worldwide (Unctad, 2016). 
If liner shipping networks can be improved by applying operations research (OR), there is thus a very 
big potential for both saving money and cutting emissions. Nevertheless, network design has 
traditionally been done manually based on previous experiences, while the research into and 
application of OR within the field is still relatively scarce.  
This thesis describes an implementation of a heuristic to solve the Liner Shipping Network Design 
Problem (LSNDP) from scratch – without an already existing network as a starting point. The objective 
is to maximize the profit for the operator through high revenue and low cost. A liner shipping network 
consists of a number of rotations, which is a scheduled roundtrip between a number of ports. To provide 
flexibility for the shippers, rotations are preferably operated weekly, even though examples of biweekly 
rotations can also be found.  
In larger networks with many ports, it is in practice impossible to connect all ports with each other with 
direct connections. Some containers will thus have to be transshipped to get from origin to destination, 
meaning that the container is switched from one rotation to another at an intermediate port. As the 
container has to be physically moved, this is of course associated with an extra cost compared to a direct 
connection. For the shipper, transshipments are also undesirable, as there is a risk of damaging the 
goods when the container is moved. In addition to this, a missed connection can result in a long delivery 
delay. Although transshipments are necessary for achieving sufficient connectivity in the network, they 
should thus be avoided if possible. 
A rotation is operated by a number of vessels necessary to achieve the desired frequency. If a rotation 
operates weekly, the number of vessels needed is equal to the rotation time in weeks. The capacity of a 
vessel can be measured by the number of containers, it can carry, which can be calculated in TEU or FFE. 
One TEU is a 20-foot container equivalent, while one FFE is a 40-foot container equivalent – meaning 
==PAGE:11==

10 
that the capacity in TEU is double the capacity in FFE. Container vessels exist in many different sizes, 
from less than 500 FFE for the smallest feeder vessels up to almost 10,000 FFE for the largest vessels 
(World Maritime News, 2015). Due to economies of scale, the cost of transporting a container is lowest 
on large vessels. Smaller vessels have other advantages, however, as they can enter more ports and 
canals than the large vessels and can profitably service markets with low demand. Operators typically 
employ vessels of many different sizes to take advantage of these different strengths by using small 
feeder vessels to connect small ports to hubs and large vessels between hubs. Organizationally, the small 
feeder vessels are often operated by a subsidiary of the operator, which has feeder operations as its 
main task. Maersk Line, for example, has its subsidiary Seago Line operating feeder services within 
Europe (Seago Line, 2016a). 
To be able to operate a shipping network, it must be decided how to flow the containers – which 
containers should go on which vessels, where will transshipments occur etc. The containers share the 
capacity of the vessels, but have unique origins and destinations, which must be respected, and it is thus 
necessary to solve a multicommodity flow problem (MCF). Apart from being a prerequisite for operating 
the network, the MCF also determines a significant part of the cost and all of the revenue, in that it can 
be decided to reject cargo. The MCF thus has to be solved when evaluating the network, and the quality 
of the solution to the MCF can easily make the difference between a profitable and an unprofitable 
network. Because of this, the MCF is a vital subproblem to the LSNDP. 
1.1 Literature review 
For a detailed review of the research conducted within liner shipping optimization problems, the survey 
papers Ronen (1983), Ronen (1993), Christiansen et al. (2004) and Christiansen et al. (2013) can be 
recommended.  
Agarwal and Ergun (2008) present three different methods for designing liner ship networks with up 
to 100 vessels and 20 ports . Heuristic, column generation and Benders decomposition methods are 
applied with the most encouraging results obtained by column generation and Benders decomposition. 
Apart from the rather small instance size compared to real world networks, the main drawback of the 
presented model is that transshipment costs are not taken into account. As transshipments are a vital 
part of the liner shipping business, this simplification is unfortunate. 
Transshipment costs are included in the work by Álvarez (2009). As ports are represented by a single 
node, however, the model cannot detect transshipments occurring within the same rotation if a rotation 
visits the same port multiple times, which reduces the accuracy of the cost calculation. In addition to 
this, it is assumed that the cost of transshipping a container is equal to the cost of loading and unloading 
a container, which might often not be the case. It is, for example, easier for an operator to switch 
transshipment port than to change the origin or destination of cargo, making transshipment costs more 
competition prone than load and unload costs.  
In contrast to Agarwal and Ergun, Álvarez imposes no frequency restriction on the rotations, but allows 
any integer number of vessels to be assigned to each rotation. This means that a rotation can be, for 
example, 20 days long, and will be operated every 20 days if one vessels is assigned, every 10 days if 
two vessels are assigned etc. As described by Brouer et al. (2014a), business rules of operators make 
==PAGE:12==
1. Introduction 
11 
this assumption quite unrealistic, as a rotation must meet a basic requirement of either weekly or 
biweekly frequency, which also fixes rotation lengths at whole weeks. If this requirement is not met, the 
schedule will be unpredictable for shippers with changing days of service and low frequency, and thus 
commercially unattractive. The frequency requirement is very important for the generation of rotations, 
as it prevents a solution with many rotations operating infrequently, but together providing a high 
number of direct connections between ports and thus saving transshipment costs.  
Álvarez uses a two-stage approach for solving the problem. In each iteration, the multicommodity flow 
problem (MCF) is solved to determine the cost of the current network. Based on the resulting flow, the 
network is optimized using a tabu search algorithm. The MCF is solved to optimality, which is reported 
to be time-consuming for an instance of only seven ports. 
Brouer et al. (2014a) presents a base integer programming model for use in the LSNDP. This model 
includes the requirement of at least biweekly frequency and also models transshipment costs more 
accurately than Álvarez by keeping this cost separate from the load and unload costs. The rotations are 
restricted to having at most one port called twice, while all other ports can be called only once, which is 
a restriction not included by Álvarez. Although allowing one port to be visited twice, resulting in a so-
called butterfly rotation, is better than prohibiting multiple calls at the same port altogether, it is still a 
restriction not found in real world rotations. Maersk Line, for example, operates the TA3 rotation 
between northern Europe, the southern US, Bahamas and Mexico, which calls the ports at Charleston, 
Savannah and Freeport in both directions (Maersk Line, 2016b). MSC operates the Silk rotation between 
Asia and northern Europe, calling at Shanghai, Tanjung Pelepas and Bremerhaven in both directions 
(MSC, 2016a).  
Brouer et al. solves the problem in two stages like Álvarez. In each iteration, the MCF is solved and new 
rotations are generated based on the resulting flow. The rotations are collected in a candidate list from 
which the most promising are selected to make up the network. This is controlled by a tabu search 
algorithm. Brouer et al. also provides data on instances through LINER-LIB, which is publicly available. 
This data, which is further described in section 2.2, includes distances between ports, demand matrices 
and vessel information for seven different instances, which can be used for benchmarking results. 
Brouer et al. is able to solve six of seven instances, while the largest instance remains unsolved. For the 
instances solved, running times vary from five minutes for the smallest instance to four hours for the 
largest. 
Recently, Karsten et al. has published work on the inclusion of time constraints in the LSNDP, see 
(Karsten et al., 2015a, pt.II). This extension provides a more customer-oriented and commercial 
approach to the LSNDP in the realization that demand is not independent of transit times. If the offered 
product is too slow, shippers will choose another operator. Karsten et al. includes a preset time limit for 
each OD pair such that the demand goes to zero if the transit time limit is exceeded. Although it can be 
argued that such a hard time limit is too simplistic, it is nevertheless an improvement over the LSNDP 
without time constraints, where any transit time is acceptable. 
Other potential extensions to the LSNDP include repositioning of empty containers. Because containers 
are reused many times, demand imbalances make it necessary to move empty containers from typical 
==PAGE:13==
1. Introduction 
12 
import ports to typical export ports. See e.g. (Song & Carter, 2009) and (Di Francesco et al., 2009) for 
work on this subject. 
It can be concluded that many different models on the LSNDP have been presented in previous work, all 
with different strengths and weaknesses. The most troublesome aspect of the LSNDP is the 
representation of transshipments and associated costs, which is either lacking or inaccurate in much of 
the work. When transshipments have been accurately represented, it is at the expense of overly 
constrained rotations that are not allowed to visit more than one port twice. The MCF, which is an 
unavoidable part of the LSNDP, is typically solved to optimality. This is naturally an advantage as all 
evaluations are perfectly accurate, but also leads to long running times for each iteration, which limits 
the possibilities for doing local search.  
1.2 Contribution 
With this thesis we aim to present a new approach to solving the LSNDP by using heuristics for both the 
MCF and local search. With a heuristic approach, the MCF can be solved much faster, allowing many 
more iterations of local search than previously seen. Even though the increased speed is at the expense 
of an optimal solution to the MCF, we believe that it is possible to quickly obtain a solution which is 
sufficiently good to guide the local search procedure in the right direction. With an increased number of 
iterations, more solutions can be evaluated, which will hopefully lead to solutions of higher quality. 
Apart from more iterations, a shorter solution time to the MCF will also allow a more comprehensive 
graph structure, which can both represent transshipments accurately and allow complex rotations 
without restrictions on the number of times a port is visited. It will thus be possible to generate more 
realistic rotations, bringing the solutions closer to those seen in real life.  
1.3 Solution approach 
As described, the LSNDP has previously been solved by using a two stage-approach, solving the MCF and 
doing local search in each iteration. This method will also be applied in this thesis, but in contrast to 
previous work, both stages are solved heuristically. The MCF will be solved by using Lagrange relaxation 
with an iterative tuning of the Lagrange multipliers based on cost-dependent shortest path-calculations. 
To obtain a legal flow, a repair flow-algorithm is devised to supplement the Lagrange relaxation, as it is 
practically impossible to get a feasible solution through the Lagrange multipliers alone.  
The local search procedure will consist of a number of different neighborhoods. To be able to quickly 
evaluate moves, a subgraph containing only the rotation being modified is introduced, enabling a fast 
solution to the MCF for this single rotation. The local search will be guided by a metaheuristic framework 
inspired by Variable Neighborhood Search, which makes it possible to escape local minima encountered 
during the search. 
Before the first iteration, a set of initial rotations is generated, as this is necessary for computing the 
flow and subsequently optimizing the network. To capture the hub structure, which is vital in a liner 
shipping network, the rotations will be generated based on an initial flow, where cargo is flown through 
a complete network with all connections between ports available. The connections are priced such that 
they are expensive at low loads and cheap at high loads to get cargo to group on a low number of 
connections economical to cover by the network.

1.4 Reading guide 
The structure of this report follows the adopted solution approach. Chapter 2 describes the problem in 
more detail, including a general mathematical model and a description of the input data provided in 
LINER-LIB. This is followed by chapter 3, which describes the implemented network representation, 
after which chapter 4 describes, tests and evaluates the solution method to the MCF that includes 
Lagrange relaxation of the capacity constraint. Chapter 5 explains how a flow-first approach is applied 
to get a basis for generating the initial rotations. Chapter 6 describes how the rotations are optimized in 
a local search procedure, primarily by applying a subgraph to each rotation to evaluate the effect of 
inserting and removing ports. This is followed by chapter 7, where computational results to LINER-LIB 
instances are reported, analyzed and compared to those obtained by other authors. Chapter 8 concludes 
this thesis, while chapter 9 provides perspectives on future work. 
We hope that the ideas and methods presented here can inspire others in their work on the exciting and 
challenging LSNDP problem.

2 Problem description 
The objective of the LSNDP is to design a liner shipping network that maximizes profit. Brouer et al. 
(2014a) presents a mathematical model for the problem, which is based on three-index decision 
variables recording both the preceding and next port of each commodity. This is necessary to allow 
accurate representation of transshipments when each port is represented by a single node only. The 
formulation limits each rotation to a single butterfly port, i.e. a port included twice, while all other ports 
can be included only once. 
To allow more flexible generation of rotations with multiple duplicate ports, the graph representation 
used here contains several nodes for each port. Each port is represented by at least two nodes – one 
centroid node for departing commodities and one for arriving commodities. For every call at the port, 
two more nodes are added, one representing the arrival and one representing the departure. A small 
example of this representation can be seen in Figure 2.

2.1 Simplifications 
Even though the presented LSNDP problem is difficult, it still contains some simplifications compared 
to the constraints which have to be applied in real world networks. Most important is the lack of transit 
time considerations for commodities, as the shortest path is computed with regards to costs alone and 
not transit time. This is obviously problematic, as a short transit time is inherently more attractive to 
shippers than a long one. The most basic transit time consideration that can be added to the model is a 
hard constraint for every OD-pair, such that it can be at most 𝑥 days, which has been implemented by 
e.g. Karsten et al. (2015b). This ensures tolerable transit times in all OD-pairs, but gives no incentive to 
offer fast services. A more sophisticated option is transit time dependent revenues and/or demands, 
such that the revenue and/or demand increases with short transit times and decreases with long transit 
times. This is the most realistic option, but also difficult to handle computationally as even more 
variables are added to the problem. To our knowledge, transit time dependent revenue and/or demand 
has not been implemented.  
In addition to transit time, other complex and special constraints have been left out, for example 
cabotage and embargo rules that prevent certain cargo from being transported on certain vessels. The 
capacity at ports is unknown and assumed to be practically unlimited. All port calls are fixed at 24 hours 
regardless of the number of container moves and vessel size, which is found by Brouer et al. (2014a) to 
be a reasonable average even though the duration of each individual port stay can vary a lot. Identical 
port stay duration across vessel classes can be explained by the facilities at the ports called, as the 
facilities at ports called by large vessels will typically be better than facilities at ports called by small 
vessels, leading to port stays of roughly the same duration. 
The time required for piloting at and around ports is not included in the model, and no extra time is 
allocated to bunkering, to cover weather related delays, port congestion etc. This time aspect is 
considered by Guericke and Tierney (2015). For all ports, the handling costs per container is assumed 
==PAGE:18==

17 
to be constant regardless of the number of containers handled at the port. In real world, some bulk 
discount could apply to the rate if many containers were handled at the same port.  
All containers are assumed to be of the same type. Special capacity constraints for e.g. reefer containers 
or high-cube containers are thus not implemented.

3 Network representation 
The choice of network representation is fundamental for any graph algorithm. In the LSNDP, a number 
of rotations 𝑟 must be converted to a graph, which can be used for e.g. computing the multicommodity 
flow. For all rotations, the following information is available: 
 
Vessel class – the type of vessel sailing the rotation. It is assumed that every rotation must be 
served by homogenous vessels, although it in reality is possible to deploy vessels with different 
characteristics on the same rotation. The vessel class determines the capacity of the rotation and 
is also used for calculating the rotation time, as the operating speed depends on the vessel class. 
 
Port call list – a list of ports in the order they are serviced. A port can be serviced more than once 
in the same rotation. This means that each port call 𝑞 is in the set of port calls 𝑄𝑟 for a given 
rotation 𝑟. Based on the port calls in the rotation, the distance is determined. Together with the 
speed information of the vessel used in the rotation, the number of vessels needed to service the 
rotation can be calculated.

4 The multicommodity flow problem 
A vital part of the LSNDP is the multicommodity flow (MCF). The solution to the MCF describes which 
containers are carried on which rotations, where cargo is transshipped and what cargo is rejected. This 
has a significant impact on several cost elements of the network:  
 
The cost of loading and unloading containers at ports 
 
The cost of transshipping containers at ports 
 
The revenue loss for rejecting containers 
 
The penalty cost for rejecting containers 
A profitable solution to the LSNDP is thus dependent on a good solution to the MCF.  
4.1 Arc flow formulation

As the capacity constraint is not present in the formulation anymore, the commodities are no longer 
bundled by a single constraint, and the problem thus decomposes into a shortest path problem for each 
commodity. This shortest path problem is solved by using the Bellman Ford-algorithm, even though the 
running time of this algorithm is asymptotically worse than the running time of the Dijkstra algorithm. 
This algorithm is chosen to allow for the implementation of a time constraint, which, although out of 
scope for this project, is an obvious extension of the problem in the future. When this constraint is 
introduced, the problem turns into a resource constrained shortest path problem with time as the 
constrained resource. This can easily be handled by the Bellman Ford-algorithm while it is more difficult 
to implement with the Dijkstra algorithm. Furthermore, the Bellman Ford-algorithm is able to handle 
graphs with negative cost arcs. Although such arcs are not present with the current graph structure, it 
is not unthinkable that negative arc costs can occur. As an example, a port can be interested in attracting 
traffic and might choose to offer money to customers transshipping in their port, leading to negative arc 
costs. Per the data in LINER-LIB, transshipment costs in ports are fluctuating a lot and some ports, e.g. 
Tunis, Tunisia and Karachi, Pakistan, have a transshipment cost of 0 – close to a negative cost. 
The Lagrange multiplier must be adjusted in an iterative process, which depends on the results of the 
shortest path problem for each commodity. If the capacity of an arc is exceeded, the Lagrange multiplier 
must be increased, while the Lagrange multiplier is decreased if the capacity is not fully utilized (except 
if the multiplier is already at its minimum level). In ideal conditions, this process terminates when the 
multipliers are as low as possible while the capacity of the arcs is still respected. 
For the problem at hand, it is not easy to find the convergence values for the Lagrange multipliers, as an 
equilibrium is almost never reached. This is caused by the composition of the OD matrix, which mostly 
consists of OD pairs with a demand higher than one container per week – some OD pairs have more than 
1000 containers per week. If these containers all yielded a different rate, they would gradually be moved 
away from an overcrowded arc 𝑎 as the Lagrange multipliers were increased. This is not the case, 
however, as all containers are assumed to yield the same rate. There will thus be a cost tipping point, 
where all containers move away from 𝑎 at once. If the cost of 𝑎 is one unit below the tipping point, all 
containers will use 𝑎, while all containers will move away if the cost of 𝑎 is one unit above the tipping 
point.  
As the flow consists of different sets of containers, one per OD pair, where each set has exactly one 
optimal route and most sets consist of many containers, an equilibrium Lagrange multiplier, which 
yields an exact match between the capacity and the load on an arc, is very hard to achieve. The Lagrange 
multiplier will most likely either be a little bit too low, and the capacity exceeded, or it will be a little too 
high, resulting in spare capacity. Of these two, only a flow with spare capacity is feasible, but it is also 
apparent that this flow must be suboptimal, as the capacity is not fully utilized. Even if equilibrium 
Lagrange multipliers are found, the capacity of some arcs can still be exceeded if one or more OD pairs 
have several optimal routes of equal cost. An example of this is shown in Figure 8 below, where two OD 
pairs, 𝐴−𝐺 and 𝐵−𝐻, have two routes of the same cost via either 𝑑 or 𝑒. In the example, both OD-pairs 
are routed via 𝑒, which yields an optimal solution to the Lagrange relaxation, but is infeasible for the 
original MIP-problem. It is thus a problem in itself to find a feasible flow from the correct Lagrange 
multipliers.

4.3 Repair flow algorithm

constraints are violated, into a valid flow through a heuristic method. This is done by first finding all

4.4 Adjusting Lagrange multipliers 
To get the overflow down to the level where it is practical to run the repair flow algorithm, it is important 
that the Lagrange multipliers are set as close to the correct values as possible. This is also vital for 
obtaining a good solution, as the repair flow algorithm is a heuristic method which cannot be expected 
to perform as well as correctly tuned Lagrange multipliers. Ahuja et al. (1993) suggests the subgradient 
method for tuning the multipliers, but this method has turned out not to work very well. Instead, a very 
simple scheme has shown the best results. With this scheme, all Lagrange multipliers start at the value

4.5 Tuning the Lagrange step settings  
The iterative process of finding the best possible Lagrange multipliers is thus subject to three general 
parameters – 𝛿, the step size of the multiplier, 𝜇, the percentage of the step size used when decreasing 
the Lagrange multiplier and 𝜎, how much to decrease the step size in order to narrow in on better 
performing multipliers. 𝜇 is fixed at 1/5 based on preliminary tests, while 𝛿 and 𝜎 will be tuned more 
thoroughly as a wider range of potentially good values can be imagined for these two parameters. As 
mentioned earlier, the point of the Lagrange multipliers is to remove overflow on capacitated arcs, and 
𝛿 should be large enough to price some demands away, but not so large that all demands avoid the arc. 
If 𝛿 is not large enough, the price of favorable arcs will be too low leading to many arcs with load over 
their capacity and a high total overflow for the network. Based on preliminary test runs the following 
step sizes, 𝛿, are decided upon for the tuning process: 𝛿= {5, 10, 20, 40, 80, 160, 320}. The exponential 
increase of the step sizes allows for more thorough testing in the lower end of the range, as it has shown 
to be most promising with respect to the objective values.

5 Generating initial rotations 
When designing a liner shipping network from scratch, it is an interesting problem in itself to find a 
place to start. No existing rotations or cargo flows are available to optimize from, which leaves only the 
OD matrix as a starting point. One method could be to connect the biggest OD pairs with direct services, 
as transshipments are associated with extra cost and should thus be avoided for as many containers as 
possible. This method is not sufficient, however, as it does not solve other basic questions regarding the 
generation of rotations, such as: 
 
Which OD-pairs should be serviced by the same rotation? 
 
What order the ports of a rotation should be called in? 
 
What vessel size is appropriate for a given rotation? 
In addition to this, the method does not take transshipments into account. For larger networks, 
transshipments are essential for achieving a profitable network, as it is of course impossible to link all 
ports in the world to each other with direct services. From a business point of view, it would also be 
unacceptable to service only the OD pairs of large volume, where a direct service is viable, as a lot of 
revenue is gained from the smaller OD pairs. A comprehensive network is a selling point for the shipping 
company as well, so while a service to a low volume port might not make sense in itself, it could be a 
prerequisite for achieving a contract which provides volume across the whole network. 
It is thus necessary to find a method to generate rotations that takes transshipments into account. The 
OD pairs in themselves do not provide any information on ideal transshipment ports, so this information 
must be inferred in another way, for example by looking at the location of transshipment ports in the 
current shipping networks and use this as a guideline for generating rotations. This is done by Meng & 
Wang (2011), who divide the ports into hub and feeder ports. Transshipments are only allowed at hubs. 
For the Asia-North Europe corridor, six hub ports are defined, while all other ports are defined as feeder 
ports and associated with exactly one hub port. The obvious weakness of this method, however, is that 
it is likely to result in a network with a high degree of similarity to the current network and rule out 
solutions with a completely different structure. Although the current network structure is probably of 
high quality, it cannot be known if a different structure is even better when this option is not considered 
due to the method used to generate rotations. 
Both Meng & Wang (2011) and Karsten et al. (2015a, chap.8) suggest a method for generating a network 
from a list of candidate rotations. This does not solve the base problem, however, which is to construct 
the rotations – and a good network can of course only be constructed from a list of candidate rotations 
if the list contains rotations of high quality.  
Reinhardt & Pisinger (2012) use a branch-and-cut method to generate rotations for smaller instances 
of up to 10 ports, which can be solved to optimality. For a larger instance with 15 ports, however, the 
running time reaches the maximum limit of 5½ hours without finding the optimal solution. This method 
can thus not be expected to work for a worldwide instance, where many more ports are required to get 
a realistic network. Plum et al. (2014) introduces the first model which allows multiple butterfly calls in 
a rotation. The model is solved using CPLEX, but optimal solutions are not found within the time limit 
for the smallest LINER-LIB instances Baltic and WAF.

Brouer et al. (2014b) views the construction procedure as a variant of the multiple quadratic knapsack 
problem, with each rotation being a knapsack and the port calls items of the knapsack. The revenue of 
an OD-pair is only obtained if both ports of the pair are selected in the same knapsack. Transshipments 
are handled by dividing the ports into feeder and hub ports, such that demands originating or ending at 
a feeder port are split at a hub port. This only solves the problem of transshipments partially, however, 
as transshipments frequently occur in OD-pairs between two hub ports. The problem is solved 
heuristically, which yields relatively poor results compared to Brouer et al. (2014a) when only the 
construction heuristic is considered. After running the construction heuristic, the rotations are 
improved using other methods, leading to significantly better results. 
5.1 Implementation of auxiliary flow 
The previously implemented methods for generating the initial rotations thus suffer from long 
computational times, result in solutions of poor quality or require a list of candidate rotations. As there 
is room for improvement, a completely different approach is chosen to generate the initial rotations. 
The idea is to reverse the usual work order of first generating rotations and then flowing containers, 
such that containers are flown first and the rotations are then generated based on this auxiliary flow. As 
there are no rotations to flow the containers through, an artificial network must be used instead. This 
network is represented by a complete, directed graph, consisting of a set of nodes 𝑃, with one node for 
each port in the instance, and a set of arcs 𝐴 connecting every node. The underlying assumption of this 
graph is that there is a rotation connecting all ports to each other with direct services. This network will 
of course never be used in practice, as it is far too expensive to run, so the arcs must be priced in a way 
such that the containers are grouped on a few of the arcs.  
In reality, it is associated with a large cost to open an arc, while adding containers to an arc, which is 
already open, is significantly cheaper. This pricing will indeed ensure that the containers are grouped 
on a few arcs, but does so in an extreme way, where the opening of a new arc is avoided at all cost – 
including detours around the world, if this can save opening a new arc. The missing penalty for using an 
open arc will also lead to very high loads on some arcs, higher than can be transported by even the 
largest vessel.

As the observant reader might have noticed, the optimal speed will always be the maximum permissible, 
as the bunker fuel consumption is not included in the objective function. The speed is, however, included 
as a decision variable to reflect the way the heuristic works, as the speed is in fact optimized to get the 
lowest possible operating cost. The optimal speed is a tradeoff between bunker fuel consumption, which 
is lowest at low speeds, and time-dependent vessel operating costs (e.g. crew and capital costs), which 
are lowest at high speeds. As the time-dependent vessel operating costs are directly linked to the 
number of vessels deployed in the rotation, the optimal sailing speed can quite simply be found by 
investigating the cost at the sailing speeds where the number of required vessels changes. As shown in 
Figure 20 below, the candidate points for the optimal sailing speed can be found at the minimum speed 
and at the two intermediate speeds, where the number of required vessels drops. In this case, the 
optimal speed is the lowest of the two intermediate speeds.

The model above can be compared to a multiple knapsack problem (MKP). The rotations are knapsacks 
while the arcs are elements to be placed in the knapsacks. The total load serviced by rotations must be 
maximized while respecting the travel time constraint of each individual rotation and servicing each arc 
at most one time. As a further complication, however, the selected elements of each knapsack must also 
respect the flow conservation constraint (30) and subtour elimination constraint (31). As the MKP is 
strongly NP-hard without the complicating constraints, it is chosen to solve the problem of generating 
initial rotations by using a heuristic.

The solution can be found by implementing a sophisticated algorithm with local search procedures, a 
metaheuristic framework etc. to optimize the objective function (28). An optimal solution does not 
necessarily yield the best network, however, as the auxiliary flow is not perfectly realistic. First of all, 
transshipment costs are not represented, which can skew the flow towards transshipping at other ports 
than what is optimal from a cost point of view. Next, the decreasing costs of using an arc means that flow 
is concentrated heavily on some arcs, with the most intensely used arc having more than 64000 
containers. This is not even close to possible to carry on the largest vessel, which has a capacity of 7500 
containers, and as an arc can only be serviced once, a large part of the flow will be left unserved in the 
optimal solution. 
5.3 Greedy heuristic for generating rotations 
It is thus chosen to adopt a simpler approach for the generation of rotations, as this allows quick 
generation of a lot of candidate solutions. This way, there is a higher probability of finding a good 
solution, even though the objective function (28) is not necessarily maximized very efficiently. The idea 
is to add one arc at a time to a rotation until all rotations have reached their maximum duration, which 
can be done in two principal ways – with serial or with parallel generation of rotations: 
1. Parallel generation of rotations: 
a. In every iteration, a random rotation is chosen, and the best arc for this given rotation 
is added. 
b. In every iteration, the unserved arc with the highest demand is added to the best 
possible rotation. 
2. Serial generation of rotations. Rotations are generated one at a time, and in every iteration, the 
best, unserved arc for the rotation currently being constructed is added. 
The advantage of the parallel generation of rotations is that good arcs can be distributed fairly among 
the rotations, such that not all the good arcs are used in the rotations generated first, while the rotations 
generated last are left with all the unattractive arcs. This behavior is not necessarily as bad as it sounds, 
however, as the rotations can be generated serially in order of vessels size, in which case the arcs with 
heavy load will be assigned to rotations with large vessels – exactly as they should. The restrictions on 
port draft will also ensure that not all good arcs are assigned to the rotations generated first, as many 
ports cannot accommodate the largest vessels. There will thus always be some good arcs left when 
switching to a smaller vessel class. A compromise could be to use parallel generation of rotations within 
the vessel classes, as this will still ensure that the arcs with heavy load are assigned to large vessels, but 
prevent the tendency of decreasing rotation quality within each vessel class. As the gain from this 
approach is estimated to be rather low, however, a pure sequential generation of rotations is chosen for 
reasons of programming simplicity. 
For each rotation, the unserved arc with the largest flow is selected as the first arc in the rotation, and a 
return arc is added to close the rotation. While the rotation is at or below the desired duration, a new 
arc is added to the rotation to expand it, and this arc replaces the return arc. The new arc is the unserved 
arc with the largest demand that either starts at the same port as the return arc, which is to be replaced, 
or ends at the same port as the return arc. A new return arc is added to close the rotation. The selection 
of a new arc is illustrated in Figure 21 below. The selection process continues until it is not possible to

6 Network optimization 
With the initial rotations generated, the network must be optimized to get a high quality solution. This 
is done by a heuristic, which has six different neighborhoods available for optimization. In each iteration, 
a neighborhood is randomly selected and one or more rotations are altered through that neighborhood. 
In order to select the best move, delta evaluation is used to avoid time consuming evaluations of the 
multicommodity flow for the entire graph. Instead, a small graph is constructed, covering only the 
rotation currently being altered. As this graph is much faster to evaluate, more moves can be tested 
before one is selected for implementation. When no objective value improvements can be detected by 
the six neighborhoods, a random move, a shake, is carried out to escape the local minimum. The 
algorithm, which is loosely based on the Variable Neighborhood Search metaheuristic, is described with 
a flow diagram in Figure 22 below.

Other work on the LSNDP uses a candidate list of rotations to choose from for optimizing the network. 
This includes Agarwal and Ergun (2008), who use dual values from the solution to the multicommodity 
flow problem (MCF) to generate new candidate rotations. As the MCF is solved heuristically here, no 
dual values are available and this approach is thus not applicable. Due to the definition of the problem, 
the method used by Álvarez (2009) is not applicable either, as Álvarez has no frequency requirement 
on the rotations and can thus optimize the network by small frequency adjustments. With the frequency 
requirement used here (weekly frequency for all rotations), the use of a rotation is a 0/1-decision, which 
is more difficult to handle. 
Brouer et al. (2014a) also operate with a candidate list of rotations. New rotations are generated in an 
auxiliary problem, which is solved to optimality. The auxiliary problem uses the residual demand 
currently not transported in the network as basis for identifying new rotations along with a number of 
empirical and estimated parameters on e.g. each ports importance as importer and exporter. Acceptable 
running times can only be achieved with up to 20 ports considered simultaneously, so when operating 
with large instances, a subset of the ports must be selected for the auxiliary problem. Tabu search is 
used as a framework for selecting rotations. 
6.1 Search method 
No literature has been found with a classical, heuristic approach to the LSNDP, where rotations are kept 
on throughout the optimization procedure and are altered slightly in each iteration. With the many 
interdependencies between the rotations imposed by transshipments, it can nevertheless be imagined 
that such an optimization process is well suited here, as incremental changes can be carried out without 
destroying the basic structure of the multicommodity flow (MCF) solution. This might not be the case 
when whole rotations are removed from and inserted into the solution, as this can derive highly 
different solutions to the MCF. The reason for using a candidate list of rotations can be the time required 
for evaluating the solution value by solving the MCF, as it can be prohibitive with regards to the number 
of iterations it is possible to run. With the short solution time obtained in this project by solving the MCF 
heuristically, see section 4.6, local search procedures with a lot of iterations become viable. 
Even though the solution time for the MCF is short, it is still too long to allow full scale evaluation of all 
possible alterations of the network. A delta evaluation is thus required to guide the local search 
procedure in the right direction. It is though not immediately obvious how this delta evaluation can be 
conducted – what is, for example, the effect of removing a port from a rotation? The fixed network cost 
effects, i.e. port calling cost, fuel consumption cost etc., can be calculated easily, while the flow cost 
effects are very difficult to estimate. For a port 𝑝 to be removed from rotation 𝑟, these effects depend on: 


6.2 Local search procedure 
The rotation graph-concept is used for four out of the six neighborhoods defined for the local search 
procedure. These neighborhoods will be described in detail below. In order to utilize the neighborhoods 
efficiently, they must be controlled by a metaheuristic framework which can guide the search in the right 
direction and help escape local optima. There are many different frameworks to choose from, and one 
of the best known is Simulated Annealing (SA), first described by Kirkpatrick el al. (1983). The basic 
idea of this framework is to accept degrading moves with a certain probability, which decreases over 
the course of the optimization after a predefined cooling scheme. This often leads to a good solution, but 
requires a high number of iterations, as the search procedure is not guided to the solution very 
efficiently – degrading moves are accepted, requiring evaluation of many suboptimal solutions. 
Although the solution can be evaluated relatively fast with the heuristic MCF applied here, it still takes 
a substantial amount of time compared to many other optimization problems. SA is thus not very well 
suited for the LSNDP. 
==PAGE:58==
6. Network optimization 
57 
Another option is to use Tabu Search, proposed by Glover (1986). This algorithm is based on local search 
to a local optimum, which is escaped by accepting degrading moves. To prevent cycling by returning to 
the local optimum immediately after a degrading move, a tabu list containing solutions or solution traits 
currently not permitted is maintained. Preliminary studies of the local search of larger instances show, 
however, that cycling is very uncommon, probably due to the inherent variability of both the local search 
neighborhood delta evaluations and the MCF evaluation. The expected effect is thus not necessarily 
equal to the actual effect, and the reversal of a move is not necessarily evaluated to exactly the opposite 
effect of introducing the move. As cycling rarely occurs, the effort required for maintaining a tabu list is 
probably larger than the effect justifies. 
Another approach to escaping local optimums is used in the Variable Neighborhood Search (VNS), 
described by Hansen and Mladenovic (2014). The general idea is to apply different neighborhood 
structures throughout the search to exploit the benefits from neighborhood changes. When a local 
optimum is encountered, it is escaped by doing a random move, a shake, from the best known solution 
and do hill climbing from here until a new local optimum is reached. If this solution is better than the 
previously best known, the search is continued from here with a new shake, otherwise the search 
returns to the previously best known solution and searches from here again after a new shake. The 
pseudocode of the metaheuristic is as follows, inspired by the basic VNS described by Hansen and

the search from another point in the solution space. The procedure must change the solution sufficiently

to return to a good solution, and a relatively modest shake procedure is thus implemented. This 
procedure modifies a number of rotations by either inserting or removing a port randomly, without 
considering the effect on objective value. To avoid inserting an obviously irrelevant port, a distance

6.4.3 Service unserved port 
The service unserved port-neighborhood seeks to introduce service to unserved ports. This is not 
captured by the insert port-neighborhood, as it tests insertion of ports into a rotation only if the port is 
connected with a feeder arc. As feeder arcs represent existing rotations and an unserved port obviously 
has no rotations calling at it, an unserved port will not be inserted by the insert port-neighborhood. It 
can, however, be inserted by the service omission-neighborhood, but as this neighborhood considers 
only the five ports generating most omission demand, smaller ports are not likely to be captured by it.  
The service unserved port-neighborhood selects a port between all ports currently not serviced in the 
solution. The probability of selecting a port is equal to its proportion of the demand of all unserved ports, 
such that an unserved port with high demand is more likely to be selected than an unserved port with 
low demand. Apart from the selection of ports, the neighborhood works exactly like the service 
omission-neighborhood described above and can thus be expected to predict improving moves just as 
accurately – although probably with a lower average improvement, as the ports included in this 
neighborhood have less omission demand than the ones included in the service omission-neighborhood.

6.4.4 Remove port 
The remove port-neighborhood tries to remove a port call from rotation 𝑟. The call must either be one 
of more calls at the same port by 𝑟 or have a relatively low exchange of cargo to be taken into 
consideration. If it is one of more calls, it will often be possible to profitably remove it, as the cargo can 
be serviced by the other calls of 𝑟 at the same port. If the exchange of cargo is low, the costs of calling at 
the port can be higher than the revenue of the serviced containers. A low exchange of cargo is defined 
as 30 % of the vessels capacity, while the exchange of cargo is defined as loaded plus unloaded plus 
transshipped containers. A vessel with a capacity of 1200 FFE must thus exchange at most 360 FFE at a 
port call if it is to be a candidate for removal (provided that the call is the only one for the rotation at 
that port).

6.4.6 Create feeder rotation 
Especially for the world and trade lane instances, it is vital for the quality of the network to have a well-
functioning feeder network. The feeder rotations must connect two types of feeder ports to a nearby 
transshipment port: 
 
Ports with low demand, where it is not economical to call with large vessels. 
 
Ports with low draft, where it is not possible to call with large vessels – even though the demand 
might be big enough to justify it. 
==PAGE:72==
6. Network optimization 
71 
It is difficult to “discover” good feeder rotations with the previously described neighborhoods, as they 
rely heavily on the demand matrix. As the demand from the feeder port to the nearest suitable 
transshipment port might be small or non-existent, it is unlikely that the transshipment port will be 
inserted using the insert port-neighborhood. Such an example is seen in the WorldLarge instance for the 
port at Buenos Aires, Argentina. This port allows only the smallest vessel to dock, and cargo must thus 
be transshipped to a larger vessel before continuing to other continents. The only suitable port for this 
in the vicinity is the port at Santos, Brazil, but between these two ports, there is no demand at all.  
A special neighborhood is thus used for creating feeder rotations between a feeder port and a hub port. 
A feeder port is defined as a port allowing only the two smallest vessel classes to enter, while a hub port 
is defined as a port allowing at least a medium-sized vessel of 2400 FFE to enter. Of all feeder ports, the 
one with the highest amount of omission demand is selected. This port must be connected to a hub port, 
which is selected randomly between all hub ports being within double distance of the nearest hub port. 
It is thus not always the nearest hub port, which is selected, as this port might not necessarily be the 
best fit with the network. When a feeder port and a hub port have been selected, a new rotation is 
constructed between the two. The neighborhood is described in pseudocode below. 
 
 
  CREATE FEEDER ROTATION 
 
  
1:  select feeder port 𝑓 with the highest amount of omission demand 
2:  calculate distance 𝑑 from 𝑓 to nearest hub port 
3:  select randomly a hub port ℎ within a distance of two times 𝑑 
4:  construct rotation between 𝑓 and ℎ 
 
 end 
 
This neighborhood does not use rotation graphs and the effect on the objective value is not evaluated in 
advance.

on direct services while the other half is transshipped at least once (Plum, 2015). Compared to this, the

also be related to the scarcity of vessel resources caused by low sailing speeds, as a high number of direct 
connections generally requires many port calls per rotation, which in turn increases the rotation time 
and number of deployed vessels.

7.2 Comparison with real-world networks 
The networks created by the algorithm have not been reviewed by experienced network planners, and 
an overall qualitative evaluation of the networks is thus not available. It is nevertheless possible to point 
out some differences from a real-world network by comparing with publicly available schedules from 
liner shipping companies. The following differences can be observed: 
7.2.1 Intercontinental rotations with more than one headhaul 
Real-world rotations across continents typically operate with one headhaul and one backhaul part, for 
example with the headhaul being the journey from Asia to Europe and the backhaul the journey from 
Europe to Asia. For the rotations generated for the larger instances, it is possible to find examples with 
several head- and backhauls. This is for example seen with a rotation in the solution to WorldSmall, 
which has the following three headhauls: 
 
Yantian, China to Gioia Tauro, Italy 
 
Pusan, South Korea to Jebel Ali, United Arab Emirates 
 
Tanjung Pelepas, Malaysia to Colombo, Sri Lanka 
The many long east-west journeys increase the rotation length to 17 weeks, which is significantly above 
that seen for real world rotations. Other examples of unusual rotations can also be found in the solutions. 
No maximum duration is enforced during the local search optimization procedure, which can explain 
how these unorthodox rotations arise. It should be relatively easy to add such a constraint and it is likely 
that the effect on objective values will be marginal at most. 
7.2.2 Different hub structure 
The data of LINER-LIB originates mainly from Maersk Line. When inspecting the Maersk Line network, 
it is obvious that the primary hub in southeast Asia is the port at Tanjung Pelepas, Malaysia. For the 
WorldLarge instance, however, the algorithm has instead generated a network with a hub at Singapore, 
located a few nm from Tanjung Pelepas. The solution contains seven weekly departures from Tanjung 
Pelepas of which three are with large vessels (at least 1200 FFE), while Singapore has 21 weekly 
departures of which 11 are with large vessels. It is a little bit cheaper to use Tanjung Pelepas than it is 
to use Singapore, so it is probably a coincidence that the algorithm has created a hub at Singapore 
instead of Tanjung Pelepas. It is, however, extremely difficult for a heuristic to escape such a hub 
structure if it has first been created, as many rotations must be moved simultaneously from one port to 
the other if an improvement is to be obtained – moving one rotation at a time will not be a good idea, as 
the rotations are in a symbiosis at a hub. 
It is also worth to note that some hubs are missing completely from the WorldLarge solution. Such two 
examples are Colombo, Sri Lanka and Freeport, Bahamas. Colombo serves as a transshipment point for 
cargo going to and from Indian ports as these ports cannot by serviced by larger vessels due to poor 
facilities. Freeport serves as a transshipment point for cargo going to and from ports in the Mexican Gulf 
and on the American east coast. Several of these ports have poor facilities and can only be called by 
smaller vessels, making a transshipment necessary, while US ports are in general extremely expensive 
to transship cargo in – nearby Miami, for example, is five times more expensive to transship a container 
in than Freeport. Colombo has only weak demand in itself, making it difficult to detect this port as a 
==PAGE:83==

relevant call for larger vessels, while Freeport has no demand at all. The algorithm does not support 
service to ports without demand, so it is not surprising that Freeport is without service. 
7.2.3 Unsuitable vessel deployment 
Generally, it is expected that large vessels operate on long, intercontinental rotations and that small 
vessels operate on short feeder rotations. With such a structure, the long rotations benefit from the 
lower unit costs of the large vessels, while the short rotations benefit from the flexibility of the small 
vessels. It is, however, possible to find examples of suboptimal vessel deployment, for example in the 
solution to the WorldLarge instance – a relatively large vessel of 2400 FFE is operated on a rotation 
calling at only Rotterdam, the Netherlands and Gothenburg, Sweden, while a relatively small vessel of 
800 FFE is operated on a 15-week long rotation with calls in both South Africa, the Arabian Gulf, India 
and China. It should be noted, however, that the vast majority of rotations operate with a suitable vessel 
size. 
7.2.4 Similarities with real world rotations 
Although differences exist when comparing with real world rotations, the “basics” seem to be in order: 
 
There are many direct connections in the most important markets, i.e. in the east-west corridors 
between Asia and Europe, across the Atlantic and across the Pacific. 
 
Large vessels operate in markets of high demand and, generally, on long rotations. 
 
Feeder vessels mainly operate to and from ports of low demand or with poor facilities, where it 
is uneconomical or impossible to use larger vessels. 
 
Transshipment structures are utilized in the network. 
Most problems are found in the solution to the WorldLarge instance, probably because of the very 
fragmented demand matrix, which in itself provides few clues on where to organize hubs.

8. Conclusion

8 Conclusion 
The goal of this thesis has been to develop a new approach to solving the Liner Shipping Network Design 
Problem (LSNDP) by applying heuristic methods. This includes the solution of the multicommodity flow 
problem, as the time required for solving this to optimality is relatively long, especially for larger 
instances. The algorithm developed to solve this problem provides a solution very close to optimality 
for the smaller instances, typically at most 1-2 % from the optimal value, at approximately one third of 
the running time. For the larger instances, the gap is a bit larger at 5-6 %, but the running time reduction 
is also much more significant, as the algorithm is up to 250 times faster than the exact method tested 
here. In addition to the running time benefits, the memory usage is also lower, allowing the algorithm 
to solve larger instances than the exact method. It can thus be concluded that the algorithm is well suited 
for doing local search, as it makes it possible to evaluate a solution fast. Although the solutions are not 
optimal, they are sufficiently good to guide the local search in the right direction.  
A starting solution is generated using an auxiliary flow, where all containers of the instance are flown 
one at a time through a complete graph with load-dependent arc costs. The resulting flow shows what 
arcs are attractive and what are not, which is used for generating the initial rotations. It is reasonably 
fast to compute the auxiliary flow and thus possible to do it several times to get a good starting point. 
The objective values of the obtained networks are far from optimal, but as it is probably not realistic to 
define any algorithm which jumps directly to an attractive solution, the implemented method can be 
concluded to work well for its purpose. The general idea of using an auxiliary flow to compute a starting 
solution thus seems promising. 
The network is optimized using a VNS-inspired heuristic utilizing six different neighborhoods. Of these 
six neighborhoods, four make use of a subgraph to quickly be able to delta evaluate moves before they 
are implemented. The subgraph includes only the rotation currently being modified, which means that 
the delta evaluation of a move in the subgraph cannot be expected to be fully coherent with the actual 
effect of implementing the move in the main graph. With this in mind, the results of the delta evaluations 
are encouraging, as moves, which are expected to improve the solution, most often actually do improve 
the solution. On average, the improvement is larger than expected. From tests of the algorithm, it has 
furthermore been confirmed that all six neighborhoods in combination contribute positively to the 
objective value. It can thus be concluded that the identified neighborhoods are all relevant and that the 
method used for delta evaluation is effective for identifying good moves. 
Computational results of the overall algorithm demonstrate that it provides good results. A comparison 
with the results obtained by Brouer et al. (2014a) shows better solutions in four out of six instances, 
which is very satisfactory. In addition to this, the results have been obtained over a significantly shorter 
running time, as the time per replication has been set to only a tenth of that used by Brouer et al. For the 
seventh and largest instance, WorldLarge, no previous results have been published, and the solution 
presented here is thus pioneering. There is potential for improving the solutions, especially on 
parameters that have been excluded from the problem definition. With an improved solution to several 
of the instances, a novel solution to the largest instance and shorter running times, it must nevertheless 
be concluded that the heuristic approach adopted in this thesis is successful and has great potential with 
regards to the LSNDP.

9. Future work 
84 
9 Future work 
Although good solutions have been identified, there are many ways in which the algorithm can be 
improved. The most obvious is a more intelligent choice of sailing speeds – i.e. by selecting a higher 
average speed. If this is implemented, several problems with the solution can be at least partially solved: 
 
It will be possible to reduce the amount of rejected cargo as less vessels are required to perform 
the same task if the speed is increased. 
 
Higher speeds will allow more port calls with the same number of vessels, which can potentially 
create more direct connections and thus reduce the number of transshipments. 
 
More cargo will be delivered within the time limits specified in LINER-LIB.  
The speeds can be increased in several different ways, some very easy to implement. One option is to 
improve the speed calculating procedure by adding an opportunity cost for each deployed vessel, to 
reflect that an opportunity for carrying extra cargo is deselected when an extra vessel is deployed in 
order to decrease the sailing speed of a rotation. This opportunity cost can be manipulated during the 
search procedure such that it is increased if almost all vessels of a certain type are deployed and 
decreased or removed if a lot of vessels of a certain type are idle.  
It is, however, not necessarily sufficient to add an opportunity cost if the problem of long transit times 
is to be corrected, as the speed required to meet the travel time requirements differs from OD pair to 
OD pair. Solving this is a significant, but interesting extension of the LSNDP, which can involve many 
extra considerations in addition to a time constrained multicommodity flow problem, for example leg-
dependent sailing speed, port call sequencing based on transit time requirements, scheduling of 
connections etc. A further extension can involve transit time-dependent revenues or demands to reflect 
that there are customers for almost any product as long as the price is low enough – and, 
correspondingly, more customers and higher revenue to be gained from an attractive product with short 
transit times. Fortunately, the graph structure implemented in this project is prepared for time 
considerations, and it should thus be relatively straightforward to move on with this aspect.    
The local search heuristic has room for improvement in several areas. First of all, it is unfortunate that 
two of the six neighborhoods are based solely on “rules of thumb” instead of a solid delta evaluation, as 
this can easily lead to degrading moves. These moves are then reversed by other neighborhoods, which 
in some cases causes cycling in the local search procedure. The best solution is to have all neighborhoods 
using a similar delta evaluation to prevent cycling. It is not necessarily possible to rectify the problem 
completely by doing this, however, as a completely accurate match between the effect predicted by delta 
evaluation and the actual effect is impossible to achieve without evaluating the entire solution. It might 
thus be a good idea to use a tabu list in the local search, such that immediate reversal of moves is 
prevented. Another solution is to reverse moves which turn out to degrade the objective value, but it 
will probably require a tabu list to do this too, as a reversed move will otherwise be suggested over and 
over again by the algorithm. 
When inspecting the results, the least successful parts seem to be related to transshipment of cargo. For 
instances with a very fragmented demand matrix, for example WorldLarge, it is difficult for the 
algorithm to detect and properly service hub ports if they do not have a great amount of demand 
==PAGE:86==
9. Future work 
85 
themselves – or perhaps no demand at all. An interesting extension of the algorithm could thus be a new 
neighborhood, where demands are split at relevant hub ports to get a more focused search towards a 
hub-and-spoke structure. It should be possible to do this dynamically through the rotations graphs 
without “hardcoding” a transshipment port for each demand pair.  
The challenge of handling transshipments can probably also be tackled through improvements to the 
auxiliary flow used to generate the initial rotations. This flow is too heavily concentrated on a few arcs 
compared to the actual solutions, which calls for improvements to the cost structure of the arcs or other 
alterations to ensure a more diverse flow. This can also improve the hub-and-spoke structure seen in 
the flow, although there is also a risk of diluting it if the flow is spread across many direct arcs. 
Nevertheless, the method of using an auxiliary flow to generate the initial rotations seems to be a good 
idea, and it should definitely be possible to fine-tune the implementation chosen here or use other 
methods to get an even better auxiliary flow and initial network.
