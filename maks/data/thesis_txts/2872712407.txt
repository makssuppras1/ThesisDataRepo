Abstract
Visual Prolog (VIP)’s official Integrated Development Environment (IDE) is a proprietary
environment hindered by its outdated design and architecture. To address these limita-
tions, this work introduces Vip-Analyzer; a language server implemented in Rust, inspired
by Rust-Analyzer. It leverages the Language Server Protocol (LSP) to deliver modern IDE
services in an editor-agnostic manner via a new incremental query-based compiler. As a
result, all IDE assists work continuously, even on incomplete code, and across multiple
projects concurrently, significantly improving the development experience over existing
solutions.
Through a wide range of evaluations, such as benchmarks, tests, and a developer study,
Vip-Analyzer demonstrates its performance, correctness, and usability, proving to be a
viable alternative to the traditional IDE. Overall, Vip-Analyzer is a complete rework of the
IDE experience for VIP, filling a significant gap in the VIP tooling landscape and paving
the way for future development in this field.

1
Introduction
1.1
Motivation
In modern software development, good tools are essential to maintain high efficiency
and developer productivity. Integrated Development Environments (IDEs), code editors,
compilers, and other development tools can automate repetitive tasks, catch errors early,
and provide features that speed up coding. Without proper tools, developers will spend
more time on actual development than on cumbersome tasks. Therefore, investing in
high-quality development tools directly contributes to faster development cycles, better
outcomes, and ultimately more successful software projects. Investing in such tools has
been a key factor in the success of many programming languages [13].
Visual Prolog (VIP) is a specialized programming language that combines logic program-
ming with other useful paradigms. However, its tooling options are limited. The language
has a long history and many great features, but it has not kept up with the modern devel-
opment ecosystem.
Despite these shortcomings, VIP has the potential to attract more developers if its tooling
is improved. Modernizing the development experience could make the language much
more accessible to newcomers. Better tools would lower the barrier to entry by providing
features that help beginners learn the language and understand code more easily. Im-
proved tooling would also assist existing developers in maintaining and expanding VIP
applications by offering more powerful and reliable editing capabilities.
1.2
Problem Description
The core challenge addressed by this project is to enable IDE-centric analysis for VIP
code. This requires re-implementing the analysis capabilities of a compiler into an inter-
active server that can provide real-time feedback to developers.
The only IDE currently available for VIP is a proprietary application that lacks many fea-
tures of contemporary IDEs. Its user interface has an outdated design and does not
include the extensibility seen in other popular code editors, such as Visual Studio Code
(VSCode). As a result, developing in VIP can feel restrictive compared to working with
other languages that support up-to-date IDEs [17].
A particular limitation of the existing VIP IDE is its heavy reliance on the Visual Prolog
Compiler (VIPc). The IDE cannot offer basic language services, such as error checking
or code completions, without performing a full build of the project. This means even small
changes require recompiling the entire codebase to get updated feedback in the IDE.
Such a workflow is slow and hinders the rapid feedback loop developers require when
using an IDE.
Because the VIP IDE is closed source, external developers cannot improve it or create
plugins that add new capabilities. The IDE does not support standard protocols, so it
cannot be integrated with other editors. There are virtually no third-party tools or resources
available to supplement the official IDE.
Another issue lies in the underlying implementation of the current solution. The parser
used in the VIPc is generated via a proprietary parser generator and tends to produce
poor error messages with minimal context when code issues arise. Developers may only

receive generic and sometimes unrelated error messages, making it challenging to identify
and fix problems. In addition, the IDE itself is written entirely in VIP. Although using the
target language to build its own tools is an interesting approach, it may not be optimal for
performance or for leveraging existing frameworks. As a result, the tool can be slower
and less responsive than alternative approaches.
1.3
Objectives
The primary objective of this project is to develop a Minimum Viable Product (MVP) of
a language server for VIP. By focusing on an MVP, the project will ensure that the core
functionality is in place and works, laying the groundwork for future enhancements.
The main question that this project seeks to answer is:
How can IDE-oriented analysis be done on VIP code?
And further, it also addresses these sub-questions:
• How can analysis be done while gracefully handling incomplete code?
• How can incremental analysis be done efficiently to reduce latency?
• What language representations are good for representing VIP semantics?
• What techniques can be used to handle the unique characteristics of the VIP language?
Another key objective is to prepare the language server for practical use in daily develop-
ment at PDC A/S, the company behind VIP. The intention is to lay the foundation for a tool
to become fully integrated into the development workflow in the future. This means focus-
ing on stability, performance, and ease of integration with existing development practices.
To address this, the project also answers the question of:
How can the language server’s capabilities be assessed quantitatively and qualitatively?
In general, the proposed solution emphasizes the following three values to align with the
objectives:
• Maintainability: Reduce the codebase maintenance burden as much as possible,
for example, by using well-established libraries and frameworks, following best prac-
tices, and writing clear documentation.
• Extensibility: Extensibility implies that new features and improvements can be
added without significant rework. For example, the design should allow for the in-
corporation of additional language features or editor functionalities as needed.
• Open-source community: This project uses open-source libraries and frameworks
where appropriate. The goal is to eventually open-source the language server, mak-
ing it available to a broader community. That means throughout the development,
the code will be structured and documented to allow it to be easily understood and
modified by others.
By emphasizing these values, the project aims to create a language server that improves
the VIP environment functionally and technically.

Background
To understand this project’s context and requirements, it is essential to explore the Visual
Prolog (VIP) language, the concept of Integrated Development Environments (IDEs), the
Language Server Protocol (LSP), and the role of compilers in software development. This
chapter provides an overview of these topics, setting the stage for the later chapters.
It is assumed that the reader has a basic knowledge of computer science and program-
ming languages.
2.1
Visual Prolog language
VIP is a strongly typed, object-oriented dialect of Prolog. A VIP program comprises a
project manifest and compilation units. The manifest contains metadata about the project,
such as its name, packages, and more.
Most of the following sections are based on the official VIP language reference: PDC A/S
[16].

2.1.2
Types and Domains
The type system of VIP is a complex and unique part of the language. The type system
implements a variant of the Hindley-Milner type system augmented with several concepts
such as parametric polymorphism, objects, and inheritance. In VIP, all declarations must
specify types, which can be variable or generic. Further, types are generally inferred
within clause implementations, but explicit typing is allowed via the built-in type assertion
predicate hasDomain.
The main typing building block is the domains. Domains can establish subtype relation-
ships, providing implicit compatibility with base domains. Compound domains define enu-
merations and structured values. Lists have a built-in notation, e.g. integer*.

2.2
Integrated Development Environment (IDE)
In the early days of programming, development was done using simple text editors and
command-line tools. The need for more sophisticated tools became apparent as pro-
gramming languages and software projects grew in complexity. The first IDEs emerged
in the 1970s and 1980s, with notable examples like Turbo Pascal and Microsoft Visual
Basic. These IDEs had a built-in native understanding of the programming language they
supported.
An IDE is generally a GUI application that provides tools to facilitate software develop-
ment. Its primary goal is to ease the development workflow, enabling programmers to
write, modify, build, test, and debug their code in one place. By integrating these com-
ponents with a user interface, IDEs can increase developer productivity, e.g., by continu-
ously providing context-aware feedback, effectively turning a text editor into a full-fledged
environment.

Common IDE features include:
• Code editor with features like syntax highlighting, code folding, and bracket match-

• Syntax checking
• Automatic code formatting
• Linting and other static analysis

2.3
Language Server Protocol (LSP)
The LSP is an open standard for editor-tool communication that connects code editors/IDEs
with language-specific servers providing intelligent programming features. Microsoft orig-
inally introduced it in 2016 and has since become a de facto norm for how development
tools integrate programming language support. It was designed to solve the M × N in-
tegration problem in tool support, turning it into an M + N problem. In traditional terms,
supporting M different languages in N editors required writing M × N separate plugins
or integrations [14].
Figure 2.2: How LSP architecture eases IDE integration. (Image: Microsoft [15])
.
Conceptually, LSP follows a client-server architecture where the editor acts as a client
that sends requests or notifications to a separate language server process. The language
server contains the analysis logic for a particular programming language, often in the form
of a compiler, and responds to the editor’s queries. The client and server communication
is standardized and independent of any specific programming language or editor. LSP
messages are encoded in JSON and exchanged using a lightweight remote procedure
call mechanism (JSON-RPC). The editor and the language server can communicate over
inter-process pipes.

The LSP communication is two-way: the client can call server methods via requests and
notifications, and the server can call the client via responses and messages.

In a typical LSP session, when a developer opens a file in an LSP-aware editor, the client
sends a didOpen notification to the language server, dispatching that file’s content. As the
user makes changes, the editor sends incremental didChange notifications so the server
can update its internal model of the code.
The server typically analyzes the code continuously and sends back diagnostics to the
client, which the editor then displays to the user by underlining problematic code or listing
errors in a problems pane. When the user requests an operation like ”Go-to Definition” on
a symbol, the editor issues a textDocument/definition request to the server with the
file location as payload.
All the features listed above in section 2.2 can be implemented via dedicated LSP meth-
ods. The specific message format can be considered an implementation detail, but it is
worth knowing that this is the LSP’s core.
2.4
Compilers
To provide any meaningful IDE features, the language server must be able to analyze the
source code. The primary method for doing so is via a compiler.
A compiler is generally defined as a program that reads a source program written in one
language and translates it into an equivalent program in another language, typically a
lower-level target language. In practice, a compiler bridges human-readable source code
and other less human-readable formats, e.g., allowing high-level programs to be exe-
cuted by a computer via machine code. This complex translation process is typically
organized into a series of separated phases, each responsible for a particular aspect of
the compilation. Structuring the compiler into phases makes the design more modular
and manageable. Each phase transforms the program representation, passes it to the
next phase, and communicates through well-defined interfaces.
Figure 2.3: Phases of a compiler. (Image: GeeksforGeeks [4])
In modern compiler design, these phases are often grouped into a front-end and a back-
end, separated by Intermediate Representations (IRs). The front-end focuses on under-
Designing and Implementing a Language Server for Visual Prolog
8
==PAGE:16==
standing and verifying the source program, while the back-end optimizes intermediate
code and translates it into efficient machine code.
Since this project is IDE-centric and does not aim to re-implement a complete compiler, it
will not include any aspects of a compiler back-end, such as code generation or optimiza-
tion. Instead, it aims towards a so-called Presentation Compiler design, which is a common
approach in language server implementations, where the focus is on providing features
like syntax checking, code navigation, and other editor-related functionalities without gen-
erating executable code [19].
2.4.1
Lexical Analysis (Tokenization)
The first phase of a compiler is lexical analysis, often called scanning or lexing. In this
phase, the compiler reads the raw source text as a stream of characters and separates it
into tokens. Typical tokens include identifiers, keywords, literals such as numeric or string
constants, operators, and punctuation symbols. For example, the statement x = 10 could
be tokenized as as the following:
1
Token { kind: Identifier , lexeme: "x" }
2
Token { kind: Equals , lexeme: "=" }
3
Token { kind: Int, lexeme: "10" }
Listing 2.6: Example of tokenized VIP code
Each token is represented by a token kind and an associated value called a lexeme.
Most compilers discard whitespace and comment tokens since they do not affect program
semantics.
Acommon lexer design involves using tools like Lex (lexical analyzer generator) in tandem
with a Yacc-style parser generator. Such tools take a specification of token patterns and
automatically generate source code for an executable lexical analyzer. Modern equiva-
lents include Flex (fast Lex) paired with Bison (GNU Yacc).
On the other hand, hand-written lexers are manually coded, often as a big state machine,
and give the implementer more direct control. Historical advice notes that a well-written
hand-coded lexer can be faster than a purely table-driven generated lexer, because it can
avoid certain overheads and leverage more low-level optimizations [1] [23].
A simple high-level algorithm for lexing is the following:
1. Read the next input character.
2. Based on that character, consume the rest of the characters for that token, until a
character is encountered that cannot belong to the current token. At that point, the
token is complete.
3. Finish the token with the kind and value it spans.
4. When this is done for all input characters, the lexer will have produced a stream of
tokens.
Since this algorithm effectively processes the input character by character, the time com-
plexity of lexing is usually linear, O(n), where n is the length of the input text.
2.4.2
Syntax Analysis (Parsing)
The next phase is syntax analysis, commonly known as parsing. In this phase, the com-
piler checks whether the token sequence produced by the lexer conforms to the grammat-
ical structure of the programming language. The parser consumes the stream of tokens
Designing and Implementing a Language Server for Visual Prolog
9
==PAGE:17==
and attempts to build a parse tree, also known as a syntax tree, that represents the syn-
tactic structure of the program. The parse tree represents interior nodes corresponding
to nonterminals (other nodes) and leaves corresponding to terminal symbols (tokens).
If the token stream violates the language’s grammar rules, the parser will emit a syntax
error. Some parsers not only detect such errors but also try to recover and continue
parsing. This can be achieved in many ways, but it is common to see error productions
or simply trying to perform local corrections.
There are various strategies for parsing, divided broadly into top-down and bottom-up
methods. Still, all parsers operate on the same principle of matching the token sequence
against the language’s context-free grammar. In top-down parsing (e.g., recursive de-
scent or LL parsers), the parse tree is constructed from the root down to the leaves, pre-
dicting what productions to use based on the following input tokens. In bottom-up parsing
(e.g., LR parsers), the parse tree is built from the leaves up towards the root, gradually
reducing sequences of tokens to nonterminals.
In many modern compilers, a data structure called the Abstract Syntax Tree (AST) is
produced instead of using the actual parse tree with all grammar productions. An AST
is a condensed form of the parse tree that omits unnecessary syntactic details, such as
trivia nodes, punctuation, etc., and organizes the program into a tree of more relevant
elements.
2.4.3
Semantic Analysis
Once the program’s syntactic structure is established in an AST, the compiler proceeds
to semantic analysis. This phase addresses the program’s meaning rather than its form
by enforcing the semantic rules of the language. While a program with correct syntax is
structurally well-formed, it may still be invalid or nonsensical in ways that syntax alone
cannot express.
One of the primary tasks in semantic analysis is type calculation, also called type-checking.
Other checks could include checking that functions are called with the correct number and
type of arguments, name visibility, assignment rules, scope, and binding rules. In gen-
eral, semantic analysis encompasses a broader range of processing, which is ultimately
determined by the target language’s definition.
To facilitate these checks, the compiler typically builds and maintains an IR of the language
that better suits the specific analysis needs. This is often done by a series of so-called
desugaring passes, which transform the AST into a more canonical form.
2.4.4
Code Generation
Traditional compilers’ overall goal is to translate the source program into an executable
form. The final step is called code generation. In this phase, a compiler will take the IR
produced by the semantic analysis, further transform it into a lower-level representation,
optimize it, and finally generate the code for a specific target architecture.
This project simply treats the code generation, also known as the back-end, parts of com-
pilers as a black box since they are mostly irrelevant for an IDE-centric presentation com-
piler.

Design
The most straightforward approach to building a language server for Visual Prolog (VIP)
would be to use the existing Visual Prolog Compiler (VIPc) as the backend and wrap
that in Language Server Protocol (LSP) bindings. This is more or less the VIP Integrated
Development Environment (IDE) approach, although it does not use LSP but rather a
proprietary protocol where all language analysis information is written to the disk. The
IDE then parses this format to provide IDE features.
Experience has shown that this approach, although simplistic, is not suitable for the life
cycle of a modern IDE. For example, when editing a file, one small change will require
the whole project to be rebuilt before updated IDE information can be provided. I.e, new
language analysis, such as code completions, type errors, and go-to definition, are sig-
nificantly delayed and require full semantic correctness. Furthermore, refactorings are
practically impossible to compute, since the VIP IDE does not have the necessary infor-
mation about scopes, semantics, etc. to perform them. This is a significant hindrance
to the effectiveness of the developer experience and ultimately the effectiveness of VIP
developers.
Vip-Analyzer takes a radically different approach to the IDE landscape for VIP compared
to existing solutions. This fundamentally different approach is what makes Vip-Analyzer
a novel and powerful tool.
3.1
LSP Architecture
Vip-Analyzer’s language server is designed as an LSP-compliant application, which en-
tails that it is editor-agnostic. Upon initialization, the server declares its capabilities, such
as providing diagnostics, code completions, and navigation, and any LSP-compliant client
can discover and use these features. The server also negotiates feature support with the
client during their initial handshake and checks which LSP capabilities the client supports
and adjusts its behavior accordingly. This approach maximizes compatibility and ensures
a robust integration regardless of the chosen editor.
Furthermore, Vip-Analyzer’s design is centered around providing a responsive IDE expe-
rience for developers, where the full power to do in-depth analysis of VIP code is present.
To achieve this, Vip-Analyzer generates all language analysis from scratch, in an IDE-
centric way. This is sometimes called a presentation compiler [19] or simply a compiler
front-end.
This means that the language server can present diagnostics and provide code navi-
gation in the full development cycle, such as while the user is typing and the code is
incomplete. Vip-Analyzer is optimized to provide this functionality on demand with ev-
ery keystroke. Compared to existing solutions, this significantly improves the developer
experience, since it allows for immediate feedback on code changes and errors without
waiting for a complete build cycle to finish.
3.1.1
Workspaces and Projects
The overall design of Vip-Analyzer is centered around being able to handle multiple projects
and workspaces concurrently. This design has affected many areas of Vip-Analyzer, par-
ticularly internal data structures, making them oriented towards being composable and
sharing common data.

3.2
Query-based Incremental Compiler
A traditional compiler typically processes source code linearly, where each phase of the
compilation pipeline is executed sequentially. In the context of a compiler, that approach is
often sufficient, but it works poorly in an IDE, where the user constantly changes the code
and expects immediate feedback. In such a scenario, a more dynamic and responsive
approach is needed.
A query-based incremental compiler offers a flexible architecture by treating compilation
as a network of pure, memorized queries. Each query represents a deterministic compu-
tation, such as parsing a file, resolving names, or computing diagnostics.
The fundamental idea of query-based design is the dynamic construction of a dependency
graph. When a query begins execution, its dependencies are recorded. These depen-
dencies can be other queries, but ultimately root in a set of base facts, such as some
text in the source files. When some dependency changes, the system will identify which
queries are affected and need to be recomputed. This is achieved by maintaining a di-
rected acyclic graph (DAG) of dependencies, where each node represents a query and
edges represent dependencies.
From a usability perspective, incremental queries enhance the performance of IDE fea-
tures. Normal IDE operations, such as error highlighting or semantic navigation, execute
quickly by reusing cached information and only recomputing a minimum set of queries.

This design lies the foundation for the incremental computations constituting the Vip-
Analyzer’s compilation pipeline.
3.3
Lexing
The lexer used in Vip-Analyzer is designed as a so-called hand-written lexer. This means
that the grammar rules are encoded directly in the implementation of the lexer, rather
than an abstract grammar definition. This design choice allows for the following design
requirements to be met:
• Performance
• Full fidelity
• Error resilience
• Maintainability
Each of these requirements is further discussed in this chapter 3 and in chapter 4.
3.3.1
Trivia Tokens
Syntax tokens, such as whitespace and comments, are often grouped into trivia tokens.
In many lexers, trivia tokens are ignored and not included in the token stream. However,
this lexer does include all trivia tokens because they are essential for the overall structure
of the code and relevant for IDE features like formatting, documentation comments, etc.
3.3.2
Error Handling
A fundamental aspect of a lexer is error handling, i.e., when the input stream contains a
character sequence that does not match any pattern of the lexical grammar. For example,
when encountering an illegal character or an unterminated string.
This handwritten lexer is designed to handle such errors gracefully because the user may
be typing incomplete code or making typos. Unless a fallback rule is defined, a generator
like Lex/Flex will typically abort on unexpected input.
The error handling strategy of this lexer is quite simple and has the overall philosophy of
Flag and continue. This means it will emit an Unknown token when something unexpected
is tokenized and then continue tokenizing as usual. This simple strategy is effective and
resilient, and guarantees tokenizing all the input text.
3.4
Parsing
The token stream produced by the lexer is the input to the parser. Here, the trivia nodes
are filtered out since they do not have any syntactic significance in the parsing phase.
However, they are later reattached to the syntax tree so that they can be used in the IDE
features.
Parsing algorithms have been widely discussed in the literature, with various approaches
such as LL, LR, and LALR [21]. However, Vip-Analyzer’s choice of parsing technique
is primarily influenced by earlier successful parser implementations, most notably Rust-
Analyzer. Rust-Analyzer’s parser is, in turn, inspired by the following work: DotNet con-
tributors [3], Swiftlang contributors [20], and JetBrains [5]. The characteristics of Rust-
Analyzer’s parser are resilience and efficiency, which are crucial for an IDE-centric lan-
guage server, and the design is in general driven by both theoretical and practical consid-
erations, aiming to balance complexity with performance and maintainability [8], [6].

3.4.1
Error Handling
The parser’s error handling strategies are designed with a best-fit philosophy, which favors
flexibility. I.e., the parser has a multitude of strategies, but the main ones are the following:
• Fixed-point recovery: When in some known parsing state, the parser can attempt
to recover by skipping tokens until it reaches a point where it can continue parsing.
This is done by skipping to the next comma, dot, or other expected tokens. This is
a heuristic approach.
• Skipping nodes: the parser can arbitrarily skip parsing specific nodes when the
tokens are absent. It will simply move on to parsing the next node.
• Discard and continue: discard the current token and continue parsing the same
node. This is useful for skipping over tokens that might not belong in a given node.
In the above, ”skipping” means marking N amount of tokens as ERROR nodes. Since the
parser is lossless, it will assign a node to everything, even invalid tokens.
Generally, the parser is constructed to be resilient to errors, meaning it will try to continue
parsing even if it encounters errors. It will try to find valid VIP syntax on any arbitrary text
input.
3.4.2
Syntax Trees (CST & AST)
The parser’s output is a syntax tree. In Vip-Analyzer, there is a conceptual distinction
between the Concrete Syntax Tree (CST) and the AST.
The parsing by Vip-Analyzer is lossless, meaning everything will be assigned a node
which can be considered a syntax classification. Accessing these classifications can be done
arbitrarily without any grammar-imposed restrictions. For example, the CTS is a dynamic
container fully decoupled from the grammar rules of the VIP language. This design allows
for a flexible and resilient parsing strategy, where the parser can handle arbitrary input and
still produce a syntax tree.
After the CTS has been built, an API layer is laid on top to provide methods for accessing
relevant nodes according to the VIP grammar. This API layer is what is referred to as the
AST.
3.5
Item Trees
Vip-Analyzer has an internal data structure called the Item Tree, representing the items
defined in a VIP project. This data structure consists of a memory-efficient summary
representation of the VIP items, locations, and other relevant data such as arity, visibility,
etc. In general, this data structure is an implementation detail, but it seeks to aid the
overall design goal of performance and maintainability of the Vip-Analyzer. It will be further
detailed in the implementation section 4.5.
3.6
Name Resolution
The name resolution phase is executed after parsing and maps identifiers to their corre-
sponding definitions within the program. In VIP, names are generally qualified by their
scope name and, optionally, by a namespace, which allows for disambiguation of identi-
fiers across contexts.
VIP is a language developed since the 1980s, but in general, it has been limited in terms
of documentation and resources. The name resolution algorithm suffers from these limita-
tions, and no documentation exists on the name resolution phase in general. The formal

Reverse Engineering the Name Resolution Algorithm
Since copying the algorithm from the VIPc is infeasible due to its complexity, the following
steps were taken to re-implement it.
To reverse engineer the algorithm, a test-driven approach was taken. The algorithm was
implemented step-by-step, gradually increasing the complexity of the test cases, with the
goal of matching the behavior of the VIPc.
A VIPc target program was written that contained a select set of openings, namespaces,
and definitions. The VIPc was used to compile the program. Then, a separate clause
referencing the names was written to inspect where the VIPc resolved the names. The
algorithm was successfully implemented by gradually increasing the complexity of the pro-
gram and making the Vip-Analyzer name resolution algorithm match the VIPc’s behavior.
3.6.2
Name table
The name table, also known as the definition map, is a data structure that contains all the
definitions of a VIP project and is, therefore, a pivotal data structure in the Vip-Analyzer
compiler. This data structure is the direct result of the name resolution phase and is used
to resolve names in the code. For example, when finding the definition of a predicate call,
the name table is used to look up the definition of the function based on its name and arity,
all while considering the actual scope of the usage site.

Implementation
The Vip-Analyzer language server is implemented in the Rust programming language, a
choice motivated by the language’s characteristics of high performance, memory safety,
and mature ecosystem.
Regarding performance, Rust provides low-level control over system resources, enabling
speed comparable to C/C++. This is essential for providing a responsive user experience
during latency-sensitive tasks such as parsing and code analysis. The language also of-
fers a robust model for safe concurrency, allowing the server to process multiple requests
in parallel without risking data races.
The choice of Rust as an implementation language was also influenced by its open-source
ecosystem. Vip-Analyzer leverages many open-source libraries, also known as ”crates”,
some of which will be further introduced in the following sections.
4.1

The Vip-Analyzer language server is packaged as a Visual Studio Code (VSCode) ex-
tension, making it straightforward to install and use in that editor. The extension’s client
component is implemented in TypeScript and is responsible for launching the Rust-based
server and setting up the communication pipes between VSCode and the server.

To maximize performance, the language server’s event handling is highly concurrent.
Most LSP requests are processed by spawning new worker threads. However, certain

4.2
Query-based Incremental Compiler
To implement the incremental query-based compiler, Vip-Analyzer uses the salsa2 Rust
crate. Salsa provides a framework for building a hierarchy of incremental computations
where the results can be cached into an in-memory database. This database functions
as the storage for data throughout the stages in the Vip-Analyzer compiler.
Rather than scatter stateful caches across compilation phases, the system maintains a
central cache mechanism that handles all caching uniformly. This not only simplifies rea-
soning about correctness but also streamlines maintenance and extension.
4.2.1
Cache Priming
With this implementation, the compiler becomes lazy in the sense that it only computes
what is strictly demanded. For that reason, an initial cache priming phase is automatically
started when the language server is initialized. This phase simply queries the name table
for all the projects in the workspace, which then populates the cache. With this simple
trick, the language server can quickly provide code analysis after the initial load.

Performance
Historical advice in compiler literature suggested that hand-written lexers are generally
faster than generated table-driven lexers, because they avoid the overhead of table lookup
and enhance compiler optimizations of the control flow [1] [2].
Since lexing touches every character of the input, even small inefficiencies per character
can add up, and quite a few techniques are used to enhance this aspect of the lexer.
One crucial optimization employed in Vip-Analyzer is reducing unnecessary work in the
lexer’s inner loop. Instead of updating line and column numbers for every character, the
lexer simply tracks the byte offsets of tokens. It only computes line and column numbers
when an error message is needed, rather than for every character processed.
Another optimization was choosing the specific functions to call when advancing the token
stream and peeking for the next characters. The Rust-idiomatic method would be using
the it.nth(1) method on the iterator. However, this idiomatic code proved less efficient
than the unrolled equivalent of calling it.next() multiple times. It is speculated that the
Rust optimizer better optimizes the unrolled code.

4.4
Parsing
The parser in Vip-Analyzer is a hand-written recursive descent parser augmented by Pratt
parsing specifically for handling expressions and statements. It operates according to an
LL(2) grammar structure, using a maximum of two tokens to look ahead.
While the parser implementation is geared to support incremental re-parsing, its necessity
has been non-existent due to the overall runtime efficiency. I.e., a full reparse is completely
feasible in all use cases, and for that reason, the project does not further delve into this
aspect.
In this implementation, the result of a parse is not a syntax tree, but rather a list of syntax
events such as START_NODE x and FINISH_NODE y. This effectively defers the construction
of the syntax tree until later stages and decouples the parsing logic from the syntax tree

Syntax Trees (CST & AST)
Accessing the nodes of the parsed syntax tree can be done arbitrarily without any grammar-
imposed restrictions. For example, the CTS is a simple container of node classifications,
where a classification is a 16-bit unsigned integer. This approach is based on the Rowan3
Rust crate.
The Abstract Syntax Tree (AST) is a higher-level representation of the syntax tree, where
only relevant nodes are exposed. An important aspect of the AST is that it assumes every
node might be missing. For example, when checking the value of some node, it will always
return an Option<AstNode> even though the VIP grammar requires the node to be present.
This implementation reflects the parser’s real-world operating conditions, and the parser
will produce an output even when some syntax is missing.

4.8
Typing
At present, the Vip-Analyzer does not implement the full VIP type system. Instead, the
current design establishes a foundation upon which a comprehensive type system can be
incrementally developed.
The existing implementation can resolve type aliases and perform type checking for value
expressions, such as literals and functors, in trivial cases. The approach is partially in-
spired by this work: Marlow and Wadler [11] and Lindahl and Sagonas [10]. Since the
implementation is still subject to change, it will not be discussed in detail here.
However, the type system’s foundational work ensures that the core mechanisms required
for type analysis are in place, facilitating future extensions to support the more advanced
type system.

4.9.1
Code navigation
Vip-Analyzer utilizes its own compiler implementation to provide code navigation features.
This includes go-to definition and go-to declaration.
The overall algorithm for code navigation is as follows:
1. Request for code navigation at some position in the source file.
2. Parse the source file to obtain the AST.
3. Classify the AST node at the position into a semantic category, such as predicate,
clause, or functor.
4. Find the semantic element’s HIR
5. Use name resolution to find its declaration/definition
6. Dispatch the declaration’s/definition’s source location to the client.

VIPc on the fly as a background process and then parsing its output to provide diagnostics
and build progress information.

detailed information is provided in a hover tool-tip.

User Study
A user study was conducted with professional VIP developers to assess Vip-Analyzer’s
usability and user experience. The goal was to evaluate how well the language server
meets the needs of real-world users and gather feedback on its features and usability.
The user study was conducted on 13-05-2025 with four professional VIP developers. The
participants were asked to use a beta version of the Vip-Analyzer, which was distributed
as a VSCode extension.
They were instructed to do the following:
• Install the Vip-Analyzer extension in VSCode.
• Open their normal VIP projects
• Enable code formatting
• Create new predicates and clauses
• Use code navigation (go-to definition, go-to declaration)
• Otherwise, use the extension as they would normally use the VIP IDE
• Take notes on their experiences
Afterwards, a semi-structured interview was held, which included fixed questions com-
bined with open discussions.

Related Work
Integrated Development Environments (IDEs) and language servers have existed for
decades, and many have been developed for various languages. This chapter will give
an overview of some of the most relevant IDEs and language servers and how they relate
to Vip-Analyzer.
6.1
Official vip IDE
Visual Prolog (VIP) has been developed since the 1980s and has been accompanied
by the proprietary VIP IDE. This IDE is a fully standalone GUI application and is heavily
coupled to the Visual Prolog Compiler (VIPc).
It is the only IDE available for VIP and is used by all developers.
Figure 6.1: VIP IDE
The IDE is written in VIP itself. It is a large application that provides the necessary fea-
tures for developing VIP applications, such as syntax highlighting, code completion, and
debugging, although in a very non-interactive way.
6.2
Language servers
Modern editors and IDEs now rely on an external language server to supply code intelli-
gence. Although architectures differ greatly, some similarities exist, such as keeping an
incremental, in-memory model of the source project. Some of the most relevant language
servers are presented below.
Designing and Implementing a Language Server for Visual Prolog

==PAGE:51==
6.2.1
Rust-Analyzer
Rust-Analyzer is one of the pioneers in incremental, IDE-first design. It splits parsing,
name resolution, and type checking into small queries managed by the Salsa incremental
computation framework. This modular design, together with tight integration with rustc,
has made Rust-Analyzer the de-facto standard for Rust editor support.1 Vip-Analyzer is
based on Rust-Analyzer and follows the same architecture, adapting it to VIP.
6.2.2
Erlang Language Platform (ELP)
ELP was started after the maintainers concluded that the older Erlang LS could not de-
liver the desired low-latency, incremental feedback expected in an IDE. Erlang LS merely
“glued” together existing tools and worked only on syntactically valid files. ELP instead
replaces these batch tools with a new compiler front-end that is fully incremental and
IDE-first, mirroring the architecture of Rust-Analyzer.
Implementing ELP in Rust solved two practical problems. First, Rust already offers the
libraries required for an IDE-grade front-end: Tree-Sitter for error-recovering parsing and
Salsa for demand-driven, incremental computation. Second, complete Language Server
Protocol (LSP) bindings for Rust were available, whereas Erlang’s bindings were only
partial. Using Rust also allowed the authors to follow Rust-Analyzer’s code base during
development closely. The resulting project demonstrates that adopting a query-based
compiler front-end, rather than wrapping batch tools, provides the responsiveness needed
for modern editor integration [22].
The Erlang Language was essentially in the exact same architectural situation as VIP,
where the existing tools did not meet modern IDE requirements.
6.2.3
ESLint
ESLint supplies only diagnostics for JavaScript and TypeScript, delegating parsing and
type information to the TypeScript compiler when available. By running lightweight linters
on every buffer change, ESLint shows that even a single-purpose server can integrate
smoothly with an existing toolchain.
6.2.4
C# (OmniSharp/Roslyn)
OmniSharp sits on top of Microsoft’s Roslyn compiler platform. Using C# features outside
the Visual Studio IDE remains the standard way.
Roslyn exposes an incremental workspace API and has laid the groundwork for many
IDE-centric features such as code actions, refactorings, and semantic highlighting. Its
presence has also affected the design of Vip-Analyzer.
Across these projects, the direction is clear: an IDE-first approach is the way to go. Vip-
Analyzer follows the same line, adapting Rust-Analyzer’s design to VIP.

7
Conclusion
This thesis introduced Vip-Analyzer, the first Language Server Protocol (LSP)-compliant
language server for Visual Prolog (VIP). Vip-Analyzer brings a novel, editor-agnostic archi-
tecture that provides modern code navigation, formatting, and diagnostics to a language
that previously lacked such interactive tooling.
Implemented in Rust, Vip-Analyzer achieves improved performance whilst providing a
better developer experience than previously seen solutions. For example, Vip-Analyzer
achieves an 8-time speed-up in Abstract Syntax Tree (AST) generation and has the tools
to resolve language constructs more accurately than the official Integrated Development
Environment (IDE). Furthermore, it provides a more responsive user experience by giving
real-time feedback on code changes with detailed error messages and allowing for multi-
workspace environments.
Hereby, the work in this thesis directly addresses the central research question of how
IDE-oriented analysis for VIP could be achieved: Vip-Analyzer employs a query-based,
incremental compilation architecture that ensures responsiveness in the whole develop-
ment cycle. To handle incomplete and erroneous code, the server utilizes resilient lexing
and parsing, which produces a lossless syntax tree even from malformed input. The
unique characteristics of VIP are captured through an entirely new set of data structures,
which combined form Vip-Analyzer’s semantic model.
This has been validated through quantitative benchmarks/tests and qualitative user feed-
back, demonstrating that Vip-Analyzer is a viable alternative to the traditional VIP IDE. The
project not only fills a significant gap in the VIP ecosystem but also serves as a case study
in applying incremental analysis, extensive automated and fuzz testing, and performance
benchmarks in the context of a language server.
7.1
Future Work
While Vip-Analyzer has successfully met its primary objectives, several avenues for future
work have been identified that could further enhance its capabilities and impact:
• Extend the type system to support full Hindley-Milner style inference, aligning with
VIP’s capabilities.
• Enhance code navigation to support constructs such as function chaining, lambda
expressions, and implement full LSP features like reference and implementation
lookup.
• Continue iterative development with VIP developers, integrating further feedback.

and edge cases.

This thesis details the creation of a new and modern language server, Visual Prolog, called Vip-
Analyzer. This language server is built in Rust and implements the Language Server Protocol (LSP)

The language server is based on a new incremental query-based compiler, which allows it to analyze

new approach to Visual Prolog analysis compared to existing solutions.
Vip-Analyzer paves the way for a new IDE experience for Visual Prolog and hopes to improve its
