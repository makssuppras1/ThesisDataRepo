Graph generation and route planning in BIM models
by Kia Kafaei Yahyavi
At present the construction industry is looking for ways to leverage the rising evo-
lution in robot technology. One way they set out to do this is by employing the latest
quadruped robot from Boston Dynamics called Spot. This thesis is made in collab-
oration with the construction-software company Dalux and aims to help them use
Spot to walk around construction sites such that it can do point scans of the con-
struction sites. The thesis uses BIM data generated by Dalux to make a pathﬁnding
algorithm for Spot on a given ﬂoorplan. The program uses a grid-based graph struc-
ture and generates the path by using the 2-opt approximation algorithm to solve the
traveling salesman problem for key nodes in the graph.
Although the data used in this thesis included varying levels of precision, the
program still demonstrated the capability of generating Hamiltonian paths that tra-
versed large areas of various ﬂoorplans.

1 Introduction
Robots depicted in Hollywood movies has been a thing of the future for a
very long time, with not much basis in reality. This fact is slowly changing,
and we are now seeing robots looking more and more like stuff from movies.
The spot robot from Boston Dynamics is the newest example of this. The
Spot robot is a yellow quadruped robot that looks like something taken from
a show [This is literally the “Black Mirror” robot dog].
The construction industry follows this development in robot technology
closely, and we are already seeing cases of the construction industry leverag-
ing the Spot robot for different uses [Spot the robot dog adopts role as construc-
tion site inspector].
1.1
Motivation
The project was worked on in collaboration with Dalux, a software company
that makes Building information modeling (BIM) software for construction
entrepreneurs, which helps make the construction process more manageable
for them. One of Dalux’s new missions has been to help entrepreneurs get a
better overview of what is being done on their construction site on a day-to-
day basis; is a new wall being constructed? Are some tubes being assembled?
etc. Dalux essentially wants to give a temporal overview of the construction
process to the entrepreneur while also precisely showing the entrepreneur if
the work being done by his or her employees or contractors is done correctly.
The entrepreneur can use this information to hold his employees or contrac-
tors accountable.
At present, Dalux has to hire an expensive architect to walk around the con-
struction site and check that everything is going according to plan and that
no errors are occurring. Needless to say, this is a costly affair.
This is where the Spot robot enters the picture because if this process
could be automated such that a robot could walk around a construction site
and do point scans instead of a person, and send the data directly to Dalux’s
software, that would be of great beneﬁt for Dalux.
The advantage of using Spot compared to using cheap labor to do this
task is that Spot would be able to perform these actions at all hours of the
day and with a higher frequency. Its route would also be very deterministic,
leaving little room for variation in what will be scanned. Furthermore, doing
pathﬁnding for Spot will allow for potentially many other uses of Spot on the
construction site, and to quote Dalux, "This will be the ﬁrst step in a row of
RD-projects where Spot will be used to interact with the physical world".

1.2
Objective
The objective of this project is to make a 2D pathﬁnding algorithm for the
Spot robot from Boston dynamics on the ﬂoor of a given building. The pro-
gram should be able to get as input BIM data from Dalux for a one-level ﬂoor
plan of a building and should then output the path that the robot should tra-
verse. The path should consider the dimensions of the robot, such that it does
not walk into walls or such that the planned path is not too narrow for the
robot.
The path should be in the form of a loop, such that the start and end
destinations of the path are the same - making it easy for Spot to be put into
the charger the following morning.
The path should furthermore make Spot visit as many rooms as possible
on the ﬂoor.
How will this be done?
The objective will be split in three distinct problem deﬁnitions:
1. Making a graph - which we will denote "grid"- graph - from the in-
put data. From the BIM input, the program will sample nodes with a
given resolution resulting in a discretized ﬂoor plan, denoted "grid"-
graph. The graph is processed in such a way that the robot will always
be able to traverse between two nodes on the ﬂoor plan if the nodes are
connected, either directly or through intermediary nodes.
2. Making a graph - which we will denote "room"-graph - from the "grid"-
graph. The essential nodes that the robot must visit - to get to every
room - will be generated and denoted as "room" nodes.
A graph will be generated where only the "room" nodes are included,
denoted "room"- graph. An optimal graph traversal algorithm will be
used to ﬁnd the shortest distance between each node in this graph. An
important property of this graph will be that it is fully connected; this
means that all room nodes are connected to each other.
3. Making a path that traverses through all "room" nodes in the "grid"
graph. We will ﬁnd an approximate solution to the traveling salesman
problem on the "room" graph. This solution will be used to generate
a path on the "grid"-graph where each "room" node is visited at least
once.
1.3
Assumptions
When doing a project such as this one where there are many alternative ways
to solve sub-problems, each having its own set of pros and cons, it is impor-
tant to work from a set of assumptions such that one knows which features
are important to consider and which are not important.
One question to ask is the following: is it necessary that the robot ﬁnds
the optimal path between the nodes?
The assumption made in this project is that the 90 minutes of walk time that
==PAGE:8==
Chapter 1. Introduction
3
Spot has, will be enough for it to traverse around any ﬂoorplan given. The
scope of this project will not be to take into account the battery lifetime of
the robot when doing the pathﬁnding. It is assumed that 90 minutes should
be enough to traverse all around the ﬂoor plan and back. Therefore ﬁnding
the optimal path between the nodes - the path with the sum minimum dis-
tance - although would be a good thing is not strictly necessary given this
assumption. This knowledge will later allow us to use approximate solu-
tions instead of analytical solutions to the traveling salesman problem, more
on that in 2.12.
Another reason that an optimal path is not wanted is because having Spot
take a longer and more obscure path between nodes will increase the likeli-
hood of Spot covering more area for point scans.
Since the program is not supposed to run in real-time, time performance does
not play an important factor when making the program. For the same rea-
son, this project will not go into detail with analyzing the time performances
of various algorithms or heuristics.
Making the program robust is also an important goal of this project. This
means that we would like to make the program such that it can work on a
wide variety of different ﬂoorplans.

2.3
BIM data
This project works with Building Information Modeling (BIM) data. The dif-
ference between Computer-Aided Design (CAD) and BIM is that there is se-
mantic information associated with each element in BIM data. In CAD, a line
is a line, while in BIM, a line can have the properties of being a wall. Not
only that, you can have all the parameters and properties of the wall asso-
ciated with it [see Understanding the Difference Between BIM and CAD]. BIM
data goes beyond capturing just the geometric information; it also captures
the semantic information associated with the building, which allows it to bet-
ter model reality. There are different formats for BIM data, but the format of
the data used in this thesis is the Industry Foundation Classes (IFC) format,
which is the ofﬁcial international standard format for BIM models.

2.4
Graph Theory
As explained in the objective section 1.2 of the introduction, a big part of this
project relies on the use of graphs, therefore graph theory plays a big role in
this project. The program is essentially representing the building using two
graphs.
For the ﬁrst graph, the building is discretized and sampled in a grid. Each
grid point is in this graph, represented by a node. This will be called the
"grid"-graph.
For the second graph, each room of the building is represented using one or
more nodes. This is called the "room"-graph.
The relevant graph theory will be explained in the following sections, which
will form a basis for the travelling salesman algorithm used to solve the
pathﬁnding problem.
2.4.1
What are graphs?
Graphs are used in a wide variety of different ﬁelds to model networks and,
in general, connections between different objects. This could, for example, be
in electrical networks to model the relationship between resistors and capac-
itors or in social sciences to model connections between human beings.
A graph can be formulated as a function of nodes and edges, G(n, e). A graph
can be directed or undirected. With a directed graph, the edge between two
nodes represents a one-way connection; the edge is thereby "directed" from
one node to another. There is no general direction in an undirected graph,
meaning you can go from node a to node b and from node b to node a. In
this project, the program works with undirected graphs since if there is a con-
nection between two rooms, it is assumed that Spot can walk from room a to
room b the same way it can walk from room b to room a, and the cost of the
walk will also be identical [Chapter 10 in Bondy, Murty, et al., 1976].
FIGURE 2.4: Left: Directed graph, Right: Undirected graph
The edges of the graph can also be weighted or unweighted. The weight
represents the cost of going from one node to another. In this project, the pro-
gram works with weighted edges where the weights of the edges are equal

to the euclidean distance between the two nodes.
2.4.2
Connectivity
The approximate solution to the traveling salesman problem used in this pro-
gram - explained further in 2.12 - has a prerequisite that the graph used is
fully connected - also deﬁned as complete. Connectivity is a measure of how
connected the nodes of the graph are. A minimally connected graph is a
graph that if one of the edges is removed, the graph will become discon-
nected. A fully connected or complete graph is a graph where every node
has an edge to every other node in the graph [Chapter 3 in Bondy, Murty,
et al., 1976].
FIGURE 2.5: Left: Complete graph, Right: Minimal connected graph
2.5
Minimum spanning tree
Another concept needed to solve the Traveling salesman problem is the min-
imum spanning tree (MST). The idea behind the MST is to ﬁnd a minimally
connected graph with the minimum summed edge weights. For a given
graph, there can be multiple MST’s since different trees can have the same
minimum summed edge weights [Chapter 23 in Cormen et al., 2009].
There exist different algorithms to ﬁnd the MST of a graph. In this program,
the MST is found using the built-in function in the NetworkX library. This
function uses Kruskal’s algorithm to ﬁnd the MST [Chapter 23.2 in Cormen
et al., 2009].

2.6
Shortest path problem
Before ﬁnding the minimum spanning tree, another important thing to do is
to ﬁnd the minimum distance between two nodes in a graph. We want to
ﬁnd the minimum distance between all room nodes and use the distance as
the weight of the edges between the nodes. This way, we will have the com-
plete "room" graph and can, from there, start solving the traveling salesman’s
problem.
There exist different shortest path algorithms. The one chosen in this
project is the A-star algorithm, which is an optimal algorithm, meaning it
guarantees to ﬁnd the shortest path [A star].
The A-star algorithm works by using information about how far a node is
from the start node and how far away it is from the end node to assess which
direction to move in. When deciding on which direction to move, it eval-
uates for each neighbor node their: g(n)-cost, h(n)-cost, and f(n)-cost. The
g(n)-cost is the cost from the start node to node n, the h-cost is the cost from
node n to the end node, measured with some measurement heuristic, like the
Euclidean distance, for example. The f(n) cost is the sum of the g(n) cost and
the h(n) cost. The algorithm evaluates the neighbour node with the lowest
f(n) cost when choosing the next node to evaluate.
2.7
Spatial Data structures
Spatial data structures are data structures that store objects and the geomet-
ric information associated with the object. The objects are stored at the leaf
nodes of a tree structure, where the branches of the tree indicate separation
in space [Chapter 1 in Laurini and Thompson, 1992]. The main use of spatial
data structures in this project is to ﬁnd the nearest objects to a given node
==PAGE:15==
Chapter 2. Analysis
10
quickly. This could, for example, be to ﬁnd the closest wall to a given node.
Having a data structure that allows for fast access to nearby objects can dras-
tically speed up the program since the alternative method would be a brute
force approach where each object would have to be checked to ﬁnd the near-
est object.
There exist different spatial data structures, but most of these work only
with point objects. This is, for example, the case with KD-tree, which is a
binary tree that for every branch splits space in two and has point objects at
the leaf nodes [Bentley, 1975].
In this project, we are interested in storing multidimensional objects - such
as walls and rooms - and point objects. We are, therefore, interested in using
a spatial data structure that allows for the storage of such multidimensional
objects.
One spatial data structure that meets these criteria is the R-tree.
2.8
R-tree
R-tree is an abbreviation for rectangle tree, which is a suitable name since it
works by making a bounding box - bounding volume in 3D - around the ob-
jects in a hierarchical structure [Guttman, 1984]. When a query is made, the
R-tree checks which of the bounding boxes at the top layer intersects with
the bounding box of that query, and then goes a level deeper on the branch
of the bounding box it intersected with. It keeps doing this until it hits the
leaf nodes or until the bounding box of the query does not intersect with any
of the rectangles in the given layer.
To illustrate how this hierarchical structure works take, for example, the el-
ements in a house. At the top level, we have the house itself. Inside the
house, we have different rooms, each of which can be bounded by a box. In-
side each of the rooms, we have the furniture which a box can also bound.
When a query is made, the R-tree ﬁrst checks if the query intersects with the
bounding box of the house. If this is the case, it checks which of the rooms it
intersects with and so on. In this way, the R-tree returns the objects that are
closest to the query.

2.9
Distance ﬁeld
The nodes in the building that Spot can traverse and the ones it can not tra-
verse are determined with the use of distance ﬁelds. The distance ﬁeld works
with grid-based graph systems. There are different variations of the distance
ﬁeld [Distance ﬁeld], but the one used in this program is a distance ﬁeld where
each node in the grid is associated with an attribute that indicates the Eu-
clidean distance to the nearest wall. To calculate the shortest distance from
the node to the nearest wall the algorithm described in this article is used
[Minimum distance between a point and a line]. This attribute is then used to
remove nodes that are too close to their nearest wall allowing for a realistic
buffer space between Spot and the walls of the building, resulting in what is
known as a Boolean distance ﬁeld.
2.10
Ray casting algorithm
Figuring out which room each node in the grid belongs to is an important
step before generating the room nodes and is based upon the usage of the ray
casting algorithm. This algorithm checks whether a point is inside a polygon
or outside a polygon. It works by casting a ray in a random direction from
==PAGE:17==
Chapter 2. Analysis
12
the position of the point towards inﬁnity. It then counts the number of times
the ray intersects with a line. If the ray intersects an even number of times
with the lines of the polygon, it means the point is outside the polygon, and
if it intersects an uneven number of times, it means that the point is inside
the polygon [Haines, 1994].
To use the ray casting algorithm, another algorithm is needed to check
for line intersection. The details about the line intersection algorithm used in
this program can be found here [Line segment intersection algorithm].

2.11
K-means
The K-means algorithm is a clustering method and is used to generate the
room nodes that will be used as nodes in the TSP solution. It works by ﬁrst
placing random centroids in the room. The number of centroids is, in this
case, chosen as a percentage of the number of nodes existing in the room.
The algorithm then consists of two recursive parts. The ﬁrst step in this re-
cursion is that all the nodes in the room are associated with one of the cen-
troids depending on which centroid they are closer to. The second step is to
re-centralize the centroids, so they are in the center of all the nodes associated
with them. These two steps are repeated either a given amount of times or
until convergence within a σ of the coordinates of the centroid occurs [Chap-
ter 22.2 in Shalev-Shwartz and Ben-David, 2014].
2.12
Traveling salesman problem
The Traveling salesman problem, abbreviated TSP, is one of the most well-
known optimization problems. The problem was ﬁrst deﬁned in the 1930s by
the mathematician Karl Menger, who presented it as the "messenger" prob-
lem [Page 41 in Schrijver, 2005]. The original idea behind TSP is to solve the
==PAGE:18==
Chapter 2. Analysis
13
challenge of a salesperson having to visit a certain number of cities; the per-
son is only allowed to visit every city once except the starting city, which will
also be the end city. This is also known as a Hamiltonian cycle. The challenge
is now to optimize for the distance traveled, such that the overall distance for
the entire route is minimized [Chapter 35.2 in Cormen et al., 2009].
The TSP optimization problem is an NP-hard problem which means it can
not be solved in polynomial time but only in exponential time. This means
that for non-trivial cases of TSP, approximate solutions are needed.
Since the idea of this program is for it to work on many different buildings
of various sizes, the number of nodes needed to be visited can vary a lot. Us-
ing an approximate algorithm to solve TSP was therefore considered a better
option than trying to ﬁnd an optimal solution to the problem. Furthermore,
the robustness of the program was considered a higher priority than ﬁnding
an optimal route for the robot to traverse, see 1.3.
In this project, we work with the Euclidean-metric and symmetric TSP prob-
lem. In the metric TSP problem, the triangle inequality holds, which states
that the direct path between two nodes a and b is always equal or shorter than
the path that includes an intermediary node c. The Euclidean preﬁx means
that we calculate the euclidean distance as opposed to other distance metrics.
The symmetric property states that the distance between the nodes a and b is
equal to the distance from nodes b and a i.e the graph is undirected. The fact
that the metric property is present is what allows us to approximate the TSP
using the 2-opt approximation algorithm which is used in this project.
2.13
2-opt algorithm
The 2-opt algorithm is an approximation algorithm to solve the traveling
salesman problem. The name of the algorithm stems from the fact that the
cost is at most two times the cost of the optimal path. The algorithm consists
of 4 parts. First, a complete graph is needed. Second, a minimum spanning
tree for the graph is generated. Third, a depth-ﬁrst traversal is performed.
Fourth, duplicate vertices are removed from the tree. [Chapter 35.2.1 in Cor-
men et al., 2009].
Depth-ﬁrst search traversal is a concept in graph theory, where a branch in a
graph tree is searched ﬁrst before searching other branches. The search goes
deeper until it ﬁnds the end node of the branch. This is opposed to a breadth-
ﬁrst search where it searches one level of the graph tree before searching the
next level [Chapter 22.3 in Cormen et al., 2009].

It became clear that some form of discretization would have to take place to
indicate exactly where Spot could be and where it could not be. The question
was whether to discretize the entire map or discretize just the route.
3.1.1
Map discretization
One way to solve this challenge was to discretize the entire map and, for each
node, ﬁnd the distance to the nearest wall. If the distance from the wall to
the node is smaller than the radius of the robot, the node will be deemed not
traversable since this means that Spot will hit the wall. If the distance to the
wall is larger than the radius, the node is traversable. What we get by doing
this is a Boolean distance ﬁeld, where we have traversable nodes and nodes
that are not.
A paper that made indoor pathﬁnding using the grid system was this pa-
per [Xu et al., 2017] also mentioned in 2.1.
The issue with this approach is that the number of nodes is directly pro-
portional to the size of the building. For bigger buildings, this might cause
a problem with the run time of the program. With that said, the distance
ﬁeld allows for certain advantages that makes it a very robust solution. It
is a robust solution because all nodes in the grid indicate whether Spot can
traverse the node or not. This makes the approach useful for all kinds of
obscure buildings. Another advantage of this approach is that it will be ben-
eﬁcial when we later want to ﬁnd optimal node placements in the rooms we
are visiting. This is because the rooms will already be discretized, which
makes generating room nodes a simpler process.
3.1.2
Route discretization
Another way to go about the problem would be to discretize the route. The
implementation of this would be a bit more complicated but would be re-
warded by its scalability to bigger buildings. The idea here is to only dis-
cretize the route that the robot will walk instead of discretizing the entire
map. One heuristic to this could be that along the route, Spot will check if it
is near a wall using a distance function. The interval here can be dynamic,
e.g. if the distance to the nearest wall is ﬁve meters and the radius of the
robot is one meter, it can walk for four meters without hitting a wall. This
means that between two nodes - a start and an end node - a lot more nodes
will be placed. The robot will go to a node, ﬁnd the distance to the nearest
wall, and then walk in a straight line towards the node, the same amount as
the distance to the nearest wall minus the radius. Then it will check again to
see what the distance to the nearest wall is and repeat the process.
In the end, discretization of the entire map was chosen with the thought in
mind that the program is not in real-time and, therefore, fast performance is
not a requirement. The distance ﬁeld would also be a more robust solution
and easier to implement.
With the route discretization, a lot of different worst-case scenarios would
have to be accounted for and considered, while this would not be the case
with the distance ﬁeld since the program tells Spot very precisely which
==PAGE:22==
Chapter 3. Design
17
nodes it can traverse and which it can not. The distance ﬁeld might also be
a better implementation long term since it can allow for certain algorithms
when placing the room nodes, explained further in 4.6).
Representing the building with a graph that would both allow for paths that
would not traverse through walls and also allow for the dimensions of Spot
was only one part of the program. Now we would have to ﬁgure out how to
solve the traveling salesman problem.
Looking at the approximate solutions for the TSP, the minimum spanning

3.3.1
Optimal placement of nodes
The program needs a way to ﬁgure out where to place the "room" nodes.
Ideally, we want to have the "room" nodes placed in such a way that the robot
can do point scans of the entire room. Different approaches can be taken to
solve this challenge.
The simplest idea would be to make a bounding rectangle of the room
and place a node in the middle of that room. This would work on a lot of
rooms that are already square-shaped, but there would be a risk of a node
being placed outside the room for non-convex-shaped rooms.
FIGURE 3.6: Example using room nodes placed at the center of the
room boundaries. Some of the nodes falls outside the rooms.
This realization would lead to another possible solution, namely, doing
convex decomposition of the rooms.
Convex decomposition
Convex decomposition of polygons is the idea of decomposing non-convex
rooms to convex rooms. A convex polygon is a polygon where every node
inside the polygon can be connected to every other node inside it with a
straight line. One solution was, therefore, to do convex decomposition of the
rooms and place a room node at the center of each decomposed and convex
room. The challenge with this solution was to decompose the rooms, which
is the reason it was not chosen as the ﬁnal solution in the end. The data did
not allow for easy decomposition of the rooms. The data was not very "clean"
in the sense that there were a lot of lines and other things that made this ap-
proach difﬁcult; see more in section 4.8.
The idea of placing optimal nodes was also an assumption that would have to
be questioned and looked at from a different angle. Perhaps the best way to
go about the problem would be to place a lot of room nodes and have a high
==PAGE:26==
Chapter 3. Design
21
probability that all corners and areas of the room would be covered. This
idea was implemented using the K-means algorithm. Placing more nodes
than optimal would make the TSP solution slower and require Spot to walk
a bit further. It would make up for it by being a much simpler method from
an implementation standpoint, and depending on how many nodes are be-
ing placed in each room, it would still be a fairly robust solution. This was
the approach that was chosen for this project.
As stated before, generating a complete room graph was needed to solve
the TSP approximate solution; therefore, some kind of shortest path algo-
rithm would have to be used to ﬁnd the shortest distance between each of
the room nodes. This shortest distance would be the weight of the edge be-
tween the nodes. There exist different shortest path algorithms, but the go-to
standard is the A-star algorithm since it is optimal and faster than many of
the alternatives, and it works well with grid-based graph systems.
For one reason or another, all the room nodes may not be connected; this
would be due to some error with the data 4.8. Therefore before ﬁnding the
shortest distance, the program checks if there are room nodes that are not
connected to the rest of the room nodes. How this is done is explained in
4.6.1. If one or more of the room nodes are not connected to the rest of the
room nodes, these nodes are removed from the "room" graph. With a com-
plete room graph ready, the program can now ﬁnd the approximate TSP so-
lution.

The speciﬁc data format
The data worked with in this project is generated by Dalux from the IFC for-
mat. Dalux takes the IFC data of a building and passes it through their parser
that generates ﬁve XML ﬁles: rooms.xml, ﬂoorplaninfo.xml, levels.xml, struc-

Therefore this project does not work with any IFC ﬁle directly but only deriva-

The rooms.xml ﬁle has information about the rooms in the ﬂoor plan. The
ﬂoor plan is generated as a collection of rooms where the coordinates of the
rooms are triangles spanning the room. Each room has associated with it its
minimum elevation level and its maximum elevation level, which represent
the height of the ﬂoor and the height of the ceiling of the room. These ele-

plan. The units of the coordinates are in meters.
The ﬂoorplaninfo.xml ﬁle has information about the doors of the rooms.
For each door, the ﬁle has information about its origin point, i.e., the coordi-
nate placement of the door. Furthermore, it has information about the height
of the door. It also has information about the direction that the door is facing,
given as a unit vector.
In the output.txt ﬁle, information about the translation is given. In some
buildings, this is needed to translate the door coordinates so that they align
with the ﬂoorplan.

know which two rooms this door connect. Having two doors on the other
hand each being in their respective room ensures that the program makes a
connection between the correct rooms.
The correct doors are chosen for the right ﬂoor plan using the height infor-
mation related to the doors. These do not always match in scale, and a trans-
lation will on some buildings have to be performed. The program checks if
this is needed.
4.2.1
Removing redundant lines
The coordinates given in the data forms a mesh of triangles that spans the
entire ﬂoor plan. If just plotted as it is, it does not do a great job of visualizing
the ﬂoor plan appropriately since a lot of unwanted lines will be present.
==PAGE:32==

FIGURE 4.2: Figure showing the ﬂoorplan with the mesh of triangles
Some processing has to be done to remove these unwanted lines and make
the rooms in the ﬂoor plan distinguishable from each other. The idea of how
to go about this problem came from [Nielsen and Co., 2019], where they also
met the same challenge.
The intuition behind their idea was that the lines to be removed are part
of more than one triangle; if e.g., two triangles are used to span a room, the
line that should be removed is the one shared by both triangles.
For this to work, each line had to be hashed. There are some slight mod-
iﬁcations to the way hashing is performed in this project compared to their
project. In this project, each line is hashed, where they hash each point in
their project.
For each triangle the program hash all lines in that triangle in both direc-
tions. This means that we hash 6 lines per triangle. The reason for this is that
a triangle might have the same line as another triangle but in reverse order.
These two lines should be seen as the same. Therefore we have to hash both
directions of each line in every triangle. The hashed keys that have multiple
lines as values are then removed from the plot.
4.3
Spatial data structure framework
As already explained, the spatial data structure is a useful tool to store spatial
elements in a data structure. In this program, the following elements are
stored in a spatial data structure:
• The rooms
• The walls
• The nodes
==PAGE:33==
Chapter 4. Implementation
28
The spatial data structure for the rooms will allow the program to access
quickly which rooms each node belongs to, the usefulness of which will be
discussed further here 4.5. The structure for the walls is used to create a
buffer zone between the walls and the rest of the building. It will be used
to investigate which nodes in the grid graph will have to be removed due to
being too close to a wall. The data structure for the nodes is used so that the
doors are connected to the closest node inside the same room as itself.
The reason these spatial data structures are suitable to use in this project is
that they allow for faster nearest neighbor search. For each of the exam-
ples explained above, the use of these data structures has made accessing the
nearest room, wall, or node faster and, therefore, the overall program faster.
An alternative brute force solution that could be used is to iterate through
every room, wall, and node to ﬁnd the nearest object. The data structure is
therefore not strictly necessary but highly advantageous to the alternative
brute force solution.
There exist different kinds of spatial data structures, though many of them
work only with point objects. Because 2 out of 3 of the objects that the pro-
gram wishes to store are multidimensional, the R-tree data structure - which
works with multidimensional objects - was chosen.
The framework used to make the Rtree data structure was the Rtree mod-
ule from python [Rtree 0.9.7].
Although the Rtree module is designed to store multidimensional objects
like rooms and wall, this data structure was also chosen to store the node
objects which are not multidimensional. This might not be the optimal way
to store point objects. To simplify the program and minimize the number
of different frameworks used in the program, the Rtree was chosen for this
data type. Using another spatial data structure for the point objects is a very
reasonable alternative.

The standard framework used when working with graphs in python is the

reason behind this resolution was that it would allow Spot to approximately
ﬁt within each node of the graph given the dimensions of Spot.
Each node created in the graph would have the following attributes asso-
ciated with it: the point coordinate, the distance to the closest wall and the
node type, i.e., a door node or grid node. Later, the room number would also
be an attribute to the node when doing the room labeling. The distance to
the closest wall could be found by making use of the R-tree generated for the
walls.
The motivation behind including this attribute was to be able to remove
the nodes that were within a certain distance to the closest wall. Including

the coordinate of the node was also necessary for many different sections of
the program, the same with the node type labeling.
Furthermore, an R-tree data structure for the nodes would also be imple-

Since the graph would be as a grid, each node would have an edge to

possible to connect each node to a node diagonal, vertical and horizontal to

When all the nodes would be generated, the nodes within a certain dis-

4.4.1
The door nodes
Originally the doors were also discretized to ﬁt into the grid, which would
move the doors to the closest grid point. This showed itself to be problematic
because the doors would move to another position than they were placed in
reality. There were a few problems with this.
One was that it could cause problems when Spot would want to walk
through the door, and the door would be a maximum of 25 cm away from its
actual position. This was somewhat concerning but might still have been a
durable solution since Spot can detect obstacles and move around them and,
therefore, might have been able to move through the door anyway.
The bigger issue with discretizing the doors was that the doors would
sometimes move just enough that they no longer were connected to the same
room. Although this was a rare occurrence, it could pose a big problem in
the robustness of the program.
With these two things in mind, it was decided not to discretize the doors,
thereby keeping undiscretized doors in a discretized grid at the cost of com-
plicating the program slightly.
The doors would have edges to only two nodes in the graph, one would
be its complementary door node, and the other node would be the closest
node with the same room label as the door node.
4.5
Making the room labels
To generate the room nodes of the room graph, it is necessary ﬁrst to label
each grid node to the room it is inside of. The program iterates through each
node in the grid and ﬁnds the nearest rooms to that node using the R-tree
generated for the rooms. Sometimes due to the way the data is formatted, a
node can be inside multiple rooms at the same time. Therefore, the program
ﬁnds the area of the rooms and sorts them such that the node is tested to be
within the smallest room ﬁrst. This is because smaller rooms are sometimes
inside bigger rooms and we want the node to be labeled to the smallest rooms
ﬁrst.
==PAGE:35==
Chapter 4. Implementation
30
When checking if the node is inside the room, the ray casting algorithm
is used together with the line intersection. In theory, it should be enough to
check one direction to decide whether the node is within the room or not.
This is not the case here, however, mainly since other elements are included
in the ﬂoor plan.
One example are pillars that can be present in the ﬂoor plan and could
therefore interfere with the line intersection. Therefore four directions are
checked, and a majority voting system is used to decide whether the node is
inside or outside the room. If a node is not inside any rooms, it is removed
from the graph since it is in this case outside the building and should not be
a part of the path under any circumstances.
Labeling the rooms serve a few functions. One of the places that it is used
is when generating the room nodes using the K-means algorithm. With-
out knowing how many rooms there are or knowing which nodes belong
to which rooms, we can not make sure that each room has a minimum of one
room node associated with it.
One of the other functions of room labeling is that it removes nodes that are
outside the building. This is because outside nodes will not be part of any
rooms and are, as a result, removed. Doing this ensures that Spot does not
traverse outside the building.
There are some exceptional cases where this algorithm will not remove out-
side nodes. This could, for example, be the case if the building consisted of a
yard inside of it.

4.6
Generating the room nodes
As explained in 3.3, different options exist for generating the room nodes that
made up the room graph. One option and the option that was used in the end
was to use a clustering algorithm to generate the nodes, more speciﬁcally,
the K-means algorithm. The K means algorithm was implemented, and the
number of centroids was chosen by a ratio of the number of nodes in the
room. This way, bigger rooms would have more centroids, i.e., room nodes.
In the case that the centroid falls outside the room, a new centroid is cho-
sen randomly from the nodes in the room.
The K-means stops after ﬁve iterations and after one iteration if there is only
one room node in the room. One could choose more sophisticated conver-
gence criteria, but this was not deemed necessary since ﬁnding the perfect
spot for the centroids was not an essential task of the program.
4.6.1
Connecting the room graph
To solve the TSP approximate solution for this graph, a prerequisite was to
complete the room graph. Therefore edges between all nodes in this graph
were to be added. However, before doing this, as a robustness check, the
program checks if all of the room nodes are connected.
==PAGE:37==
Chapter 4. Implementation
32
The central function used to do this comes from the Networkx library.
This function ﬁnds the minimum number of nodes that need to be removed
before there is no longer connection between two nodes. The function is
based on the the approximate algorithm that originates from this paper [White
and Newman, 2001]. The approximate algorithm ﬁnds the number of node
independent paths between a node pair which according to Menger’s theo-
rem [Böhme, Göring, and Harant, 2001] is equal to the minimum number of
nodes that need to be removed to disconnect two nodes.
If the minimum number of nodes that need to be removed to disconnect two
nodes is zero, it means that the two nodes are not connected. The room nodes
that are not connected to the rest of the graph are removed. After making
sure that the nodes are indeed connected, it is time to create edges between
the nodes. The weight of the edges was found by ﬁnding the shortest path
between each of the nodes. This was found using the A-star algorithm.
4.7
Implementing the TSP solution
As explained before the TSP approximation used in this project is the 2-opt
algorithm, which works as a four-step process. The ﬁrst part is having a com-
plete graph of the room nodes. This is done in the previous section. The next
part is making a minimum spanning tree of this graph, which is done with
one line of code using the Networkx framework. The third step is doing the
Depth-ﬁrst search traversal which is also done using the Networkx frame-
work. The fourth step is removing double vertices from the DFS traversal,
which is done using an algorithm created in the program.
4.8
What the data lacks
A big challenge to the program occurred when there were walls in the ﬂoor
plan of a building that seemed misplaced. In investigating the 3D model
of the building, it seemed that these walls were not supposed to be there.
The distinction between what was illustrated in the 3D model and what was
shown in the plots of the program were because not all information from the
BIM model was parsed to the six ﬁles generated by Dalux.
There was no information in the data which indicated which lines repre-
sented walls and which lines did not represent walls. As a general case, each
line on the ﬂoor plan represented walls, but this was not always the case.
It depended on how the architect decided to design the building. Take, for
example, the ﬂoorplan showed in ﬁgure 4.5.
==PAGE:38==

FIGURE 4.5: Example where the data has erroneously placed lines
where they should not have been placed
In this example the two long narrow rooms and the big main room are, in
reality, one big open space, but the designer deemed it useful to separate this
open space into three rooms, and therefore, lines are separating the rooms.
The issue, as can be seen in the ﬁgure, is that since these lines will not have
doors connected with them, they will result in blocking of the path. Spot can
therefore not traverse through all the rooms in the ﬂoor plan.
A big portion of energy was used to reconcile this distinction between what
is plotted and what is actually the case and ﬁnd ways to tackle this issue.
Since there is no information given in the ﬁles about which lines represent
walls, a heuristic had to be found to solve the issue.
4.8.1
One-wall-algorithm
From investigating the difference between the lines that were supposed to
represent walls and the lines that were not supposed to represent walls, a
realization came.
It seemed that the walls separating the rooms often consisted of two par-
allel lines with a small separation between the lines. This seemed to indicate
the thickness of the wall. The reason for these double lines comes from the
fact that when the rooms are generated, they do not share lines. On the other
hand, it seemed that when there was not supposed to be a line of separation,
the two rooms shared the same wall. Another way to view this was that the
line represented a wall with thickness zero.
With that in mind, a heuristic was implemented that would check for all
rooms if lines were overlapping or shared. This was done by comparing ev-
ery line with every other line in the ﬂoorplan.
==PAGE:39==
Chapter 4. Implementation
34
There are two scenarios in which the wall can be removed. The ﬁrst sce-
nario is if the two line segments are identical. The second scenario is if the
line segments are overlapping, i.e., one line segment is a part of another line
segment.
Finding out if line segments are overlapping is done by checking the slopes
of the points making up the line segments; if the slopes of all the points of
the two-line segments are the same, it means that they are co-linear. This
does not necessarily mean that they are overlapping. To ﬁgure this out, a
projection on the x-axis is performed. In the special case of vertical lines, a
projection on the y-axis is performed. If one of the points of one line segment
is overlapping with the other line segment, this means that the line segments
overlap.
FIGURE 4.6: Left: Co-linear lines not overlapping, Right: Co-linear
lines overlapping
Before removing the line segment, we check if there are doors within a
certain distance of the wall. This is because we do not want to remove lines
that have doors connected to them since this means that the line is a wall or
that the doors are supposed to be used to traverse the line.
The results of using this heuristic were not perfect, as can be seen from the
results section 5.1, they were, however, often an improvement to not using it.

room. This is a special case example and a solution would be for the doors to
check intersection with all lines of the entire building and not just the lines of
the room that the door and the other node share, this would however result
in an increased run time.

6 Discussion
6.1
Challenges
The biggest challenge in this project was to work with the data given from
Dalux. This is because the rooms.xml ﬁles that is used in this program to plot
the rooms are not intended to plot any geometry but is rather a semantic ﬁle
that is used to indicate the placement of the rooms. The reason Dalux gave
me these ﬁles was that they wanted to generate a path in 2D for Spot. The
limitations of using these ﬁles to generate the geometry would show itself in
many different ways.
It would show it self in ﬁgure 5.4 where the doors and the rooms were very
poorly aligned and where the plot showed a lot of superﬂuous doors. This
is most likely due to the designer being a bit imprecise with the semantics of
where the doors are exactly placed. Since this is not used to make the geom-
etry, the consequences of these kinds of errors would otherwise probably be
to a lesser extent.
As explained in section 4.8 another place this data is limited is when walls
were plotted where they should not have been plotted. Although the one-
wall algorithm removed some of these walls not all of them were able to be
removed which led to sub optimal results in the form of limited paths in the
different ﬂoorplans.
Due to the way that the designers would label the minimum and maximum
elevations of the rooms the program would sometimes either include too
many or too few rooms. When generating the plots in the results section
the method used to include rooms was to use the minimum elevation of the
rooms, and only include the rooms that had the same minimum elevation
levels.
This has a limitation of not including rooms that are on the same ﬂoor but
slightly higher or lower on the ﬂoor. An example of this could be two rooms
in the same ﬂoorplan that are seperated by a small stair-step in elevation. In
this case the two rooms would have slightly different minimum elevations
and the program would not be able to include both rooms in the ﬂoorplan.
An alternative way to include rooms would be to use a clipping height where
all rooms in which this clipping height is between minimum elevation and
maximum elevation of the room would be included.
This way the issue mentioned before - with the two slightly different min-
imum elevation rooms - would likely be negated depending on the value of
the clipping height. The downside of doing it this way is that the clipping
==PAGE:53==
Chapter 6. Discussion
48
hide might include rooms that are not on the ﬂoor. And the risk of overlap-
ping rooms would therefore be high.
It is not clear always which method is the better one to use since this partly
depends on the building and also depends on the way that the designer of
the building set the minimum and maximum heights of the rooms of the
building.
Take the example before with the two rooms where there was a small mini-
mum elevation difference between the rooms. It could be that the designer
would neglect this difference and just label both minimum elevations as be-
ing the same. In this case using a minimum elevation as a criteria for which
room to choose would be a good choice.
If the designer decided not to neglect this difference perhaps using a clip-
ping height would be a better alternative. Luckily changing between using
minimum height or clipping height is simple and only requires the user to
change one value in the program.
Another small challenge with the dataset was that sometimes the coordinates
of the doors had to be translated such as to align with the rooms properly.
Again this depended on the design choices of the architect.
6.2
Performance of the program
In general the program did well considering the circumstances surrounding
the data. The program managed to take in different buildings and plot the
different ﬂoor plans of the building.
When the path would be blocked the program managed to take this into con-
sideration and only do a path of the areas that it could. Therefore in terms of
robustness the program did pretty well.
The performance of the program, meaning how many of the rooms the path
visited, varied a bit from building to building and ﬂoor plan to ﬂoor plan.
The reason for this is partly due to the design of the program and partly as a
result of the data.
An example where the program is limited is with regards to narrow corri-
dors or rooms. In these cases, due to the resolution of the grid being too low,
the path did not manage to go into these rooms. This could be solved by
increasing the resolution of the grid but at a cost of increasing the running
time of the program.
Although giving good results, the One-wall heuristic did not always work
perfectly since it sometimes removed walls to the outside as a result of two
rooms sharing that wall. This however did not affect the result of the path
of the program since all the nodes placed outside the building are removed
from the grid.

Other times it did not manage to remove the walls that needed to be re-
moved, which often would result in a limited path.
Placing the room nodes using K means algorithm seemed also to work well.
The nodes seemed most of the time to be placed such that most of the area
of rooms would be covered. The user can experiment with different ratios
of room-nodes-to-grid-nodes to place more or fewer nodes for the path to
traverse.

7 Future Work
7.1
Experimenting with different approximation algorithms
Since the Traveling salesman problem is one of the most famous optimiza-
tion problems many different approximation algorithms has been developed
to try and solve the problem. Future works of this project could be to analyze
and compare the performances and run times of these different approxima-
tion algorithms.
Although this is not a high priority of the project, as mentioned before,
ﬁnding the best approximation algorithm could serve as a neat feature.
7.2
Using a more complex algorithm for placing room nodes
Another focus area for further development of this project is to develop a
better method for placing the room nodes. At the moment the room nodes
are placed using the K-means algorithm which is not an optimal placement
strategy. Experimenting with different placement strategies and analysing
the amount of coverage given for each is an interesting problem for further
work.
7.3
Taking into consideration the battery time for Spot
Taking into consideration the battery time of Spot when developing the route
is also an area worth looking at for future work. In this project it is assumed
that the battery time of Spot is sufﬁcient for it to traverse one entire ﬂoor
plan, but this assumption has not been put under scrutiny. It could be inter-
esting to do some analysis of battery time compared with the route traversed.
Finding the optimal path around the building and taking into consideration
the battery capacity of Spot becomes especially important when it should
traverse multiple ﬂoor plans.
7.4

This project has been part one of a two-part project series where the second
part of the project will be programming Spot to traverse the path generated
in this part of the project.

8 Conclusion
The purpose of this project was to make a pathﬁnding algorithm on a ﬂoor-
plan of a given building that would take into account the dimensions of Spot.
This project was successful in achieving this result. The program showed
the capability of taking as input many ﬂoorplans from different buildings
and managed to make a path that would take into consideration the dimen-
sions of Spot.
The purpose was furthermore that the path should return to its original po-
sition and that it should visit as many rooms as possible.
The project was successful in generating a path that would return to its orig-
inal position.
To visit as many rooms as possible the project was somewhat a success.
Although it on many occasions did not manage to visit all rooms in the
ﬂoorplan this was more often than not due to the nature of the data and rarely
due to the design of the program. In cases where it was due to the design of
the program, solutions were offered to resolve the problem, for example by
increasing the resolution of the grid graph.
Although not delivering perfect results the program has shown robustness
in taking many different ﬂoorplans and buildings as input. It is believed that
this program can be used by Dalux in their future research and development
adventures using Spot, to make Spot traverse if not all then part of a ﬂoor-
plan.
