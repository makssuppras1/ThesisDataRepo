Abstract 
 
In the Home Care Scheduling Problem, a set of tasks must be assigned to a set of employees. An 
assignment should state when the employee starts performing the task. There are several 
constraints for how the tasks can be assigned. There is, for example, a constraint for when a task 
can be performed. For every task there is a specified time window, the task can only start in this 
time window. There are also constraints on which employees a task can be assigned to. For every 
task an employee must have a required set of skills to be assigned the task. The quality of the 
solution is measured on three things. It is not always possible to assign every task to one of the 
employees. A task can either have high priority or low priority. The most important objective is to 
assign as many high priority tasks as possible. After this the most important thing is that the total 
number of assigned tasks is as high as possible. Depending on which employee a task is assigned 
to, we get a score. The schedule should have the highest score possible. The algorithm developed 
for solving the problem should be fast enough to be used in real life operational planning. 
Therefore, the running time should not be more than 1 minute. 
 The project was carried out in cooperation with the Danish company Qampo, which provided 
datasets. Qampo also provided solutions generated with their own algorithm. 
In the project different algorithms were developed for solving the problem. Methods like 
matheuristics and dynamic programming were tried. The methods tried to split the problem into 
smaller sub problems in different ways to make it easier to solve. But these methods were all 
rejected either because the running time was too high or because the quality of the generated 
solution was not good enough. In the end an algorithm, based on adaptive large neighbourhood 
search (ALNS), was developed. The ALNS algorithm is given a 1-minute running time. During this 
time, it performs thousands of iterations. In each iteration the current solution is destroyed and 
repaired to obtain a new solution. The solutions are destroyed and repaired in different ways, so 
the algorithm looks for new solutions in different neighbourhoods of solutions. During the 
iterations the algorithm saves the best obtained solution, which is returned in the end.  
A mathematical model was developed and solved with CPLEX to give an upper bound on the 
quality of the solutions for different datasets. Different tests were performed on the ALNS 
algorithm, to find the best settings. These tests were performed on 10 different datasets and in 
total the program was run several hundred times. The result showed that the algorithm was very 
stable. Changing the parameters did not make a large difference for the quality of generated 
solutions. Overall were the generated solutions very close to the upper bound, so the developed 
algorithm can for many datasets generate solutions near optimal. The algorithm only uses the 
allowed 1-minute running time to generate a solution. The solutions generated with the ALNS 
algorithm were also compared with solutions provided by Qampo. The comparison showed that 
the ALNS algorithm outperformed the provided solutions for every dataset.

1 Introduction 
 
The home care service is of great interest in countries, like Denmark, with a welfare system. If an 
elderly citizen can be helped in their own home instead of being hospitalized, the society saves 
money and the elderly citizen gets higher quality of living. In Denmark the home care system was 
established in 1958.1 Today over 140.000 citizens receive home care services.2 In order to help the 
large number of citizens a lot of employees are needed, which costs a lot of money. An example of 
the size of the home care is Copenhagen municipality, here 1200 employees work in home care.3 
Another example is Odense municipality, here they use 203 million kroner each year on the home 
care services.4 In this project an algorithm for solving the Home Care Scheduling Problem will be 
developed. The objective is to generate schedules for the home care service. A schedule for the 
home care service consists of a set of employees, where each has been assigned to a set of tasks. 
The tasks are performed in the home of the elderly. Examples of tasks can be to give medicine to 
the elderly or help them to the toilet. It could also be to clean their house or do the laundry. The 
quality of the schedule is measured by different parameters and the schedule must comply with 
various restrictions. The project has been carried out in cooperation with the Danish company 
Qampo. Qampo has provided real data from Danish municipalities. Qampo already uses 
algorithms for solving the Home Care Scheduling Problem. For some of the datasets they have 
provided solutions generated with their own algorithms. The goal is therefore to develop an 
algorithm which can either improve their solution or generate a new and better solution. Another 
important goal in the project is that the developed algorithm should be fast enough to be used in 
real life operational planning. There will therefore be a focus on the running time of the 
algorithms through the report. The algorithms are implemented in Java and tested on a computer 
with an Intel(R) Core (TM) i5-7400 CPU 3.00 processor and 8.00 GB RAM. The Java code can be 
found in Appendix A. 
In chapter 2 the problem is described in details and the datasets provided by Qampo are 
presented. A literature overview is given in chapter 3. The results and solution methods for other 
papers that have worked with problems from the home care are presented. It is explained how 
the problem solved in this project is different from the others and therefore unsolved. The basic 
solution algorithm is developed in chapter 4. In this algorithm some of the difficult constraints are 
left out. The chapter is organized in a way that shows the developing process. An algorithm is 
developed and tested. The results from the test give an idea for next algorithm. This way the 
chapter shows the process from the first algorithm to the final algorithm.  In chapter 5 the 
algorithm developed in chapter 4 is improved, in order to handle the difficult constraints, which 
were initially left out. In chapter 6 the final algorithm is tested. In this chapter an algorithm to find 
an upper bound for the quality of a solution is also developed. Finally, an overall conclusion for the 
whole project is made in chapter 7.

2 Problem description 
 
The problem consists of a set of citizens, referred to as service entities. For each service entity on a 
given day, there is a list of tasks which must be performed by an employee. The goal is to assign 
tasks to the employees. For each task assigned to an employee it should be stated, when the 
employee starts performing the tasks. In the end each employee will have a list of tasks with start 
times, this is referred to as the employee’s route. The solution to the problem is the set of the 
generated employee routes. 
Each task has a time window which defines the earliest start and latest start. A task can only be 
started in this time window. In addition, each task has a duration which tells how long it takes to 
perform the task. A task may depend on other tasks. There are three different ways that a task 𝑡ଵ 
can depend on a task 𝑡ଶ:  
 
Same employee Dependency:  
In this case 𝑡ଵ must be assigned to the same employee as 𝑡ଶ. If 𝑡ଶ is not assigned to any employee, 
the constraint does not have any effect. In the report we will sometimes write SE-dependency 
instead of Same employee dependency. If we in the report want to say that a task 𝑡ଵ has a SE-
dependency to another task 𝑡ଶ, it is formulated as: 𝑡ଵ SE-depends on task 𝑡ଶ.   
 
Relying on Dependency:  
In this case 𝑡ଵ can only be assigned to an employee, if 𝑡ଶ is also assigned to an employee. In the 
report we will sometimes write RO-dependency instead of Relying on dependency. If we in the 
report want to say that a task 𝑡ଵ has a RO-dependency to another task 𝑡ଶ, it is formulated as: 𝑡ଵ 
RO-depends on task 𝑡ଶ.   
 
Duration Dependency: 
In this case 𝑡ଵ has two values 𝑚𝑖𝑛 and 𝑚𝑎𝑥. 𝑚𝑖𝑛 is the minimum amount of time there must go 
from the start of 𝑡ଶ to the start of 𝑡ଵ. 𝑚𝑎𝑥 is the maximum amount of time there must go from 
start of 𝑡ଶ to the start of 𝑡ଵ. Figure 1 illustrates an example, 𝑡ଵ must start a time between the two 
red lines. In the illustrated example 𝑚𝑖𝑛 and 𝑚𝑎𝑥 have positive values. They could also have 
negative values, in that case 𝑡ଵ had to start before 𝑡ଶ. In the report we will sometimes write DU-
dependency instead of Duration dependency. If we in the report want to say that a task 𝑡ଵ has a 
DU-dependency to another task 𝑡ଶ, it is formulated as: 𝑡ଵ DU-depends on task 𝑡ଶ.

Each employee has a set of skills and each task has a list of required skills. A task can only be 
assigned to an employee, if the employee has all the required skills. A service entity can have a list 
of blocked employees. If an employee is blocked by a service entity, the employee cannot be 
assigned tasks from that service entity even if he has the required skills. An employee can also 
have a list of must attend tasks. All the tasks on that list must be performed by the employee no 
matter what. Even though the must attend tasks should always be performed, it is sometimes not 
possible, this will be explained in section 6.1. 
 
For each employee it is specified what time of day he works. The employee can only perform task 
in his worktime. The tasks are performed on different locations, mostly in the home of the service 
entities, so the employee must travel between these locations to perform the tasks. There are 
three means of transportation, referred to as transportation modes – walking, cycling, driving. 
Each employee is from the start fixed to one of these transportation modes. If an employee is 
walking or cycling there is an upper limit for how long, he can travel between two tasks. There is 
also an upper limit for the total traveling distance per day. These limits are different for walking 
and cycling employees. In the routes there must always be enough time for the employee to travel 
between tasks.

The quality of a solution is based on three things. Due to lack of time it may not be possible to 
assign all the tasks. Each task has a priority – high or normal. High priority tasks are the most 
important. The solution should always have as few unplanned high priority tasks as possible. Even 
if it means that a lot of normal priority tasks are not planned. For each combination of service 
==PAGE:10==
10 
 
entity and employee there is a score between 1 and 10. The higher the score, the more does the 
service entity prefer to be helped by the employee. Let say that service entity 𝑠 and employee 𝑒 
has a score of 7. We then get a score of 7 each time we assign a task from 𝑠 to 𝑒. In the report we 
will refer to the score we get from assigning a task to an employee as the task score. The score of a 
solution is the sum of all assigned task scores. Quality of the solution is measured by the following 
three things with increasing importance, where the values should be as high as possible:  
 
1. Number of planned high priority tasks  
2. Number of planned tasks 
3. Solution score 
If the value of 1. can be increased it should be done no matter how it influences 2. and 3.. If the 
value of 2. can be increased it should be done no matter how it influences 3.. Finally, the value of 
3. should be as high as possible. 
 
 
2.1 Running Time  
 
The algorithms developed in this project should be able to find a solution, which is feasible with 
respect to the constraints listed in Table 1 and of high quality with respect to the three quality 
measures. Aside from this, it is also a requirement that the algorithm is fast. The purpose is that 
the algorithm should be fast enough to be used in real life operational planning. After meetings 
with Qampo, we know that an ideal running time would not be much more than 1 minute.

3 Literature Review 
 
The two reviews (Fikar and Hirsh, 2017) and (Mascolo, Espinouse and Hajri, 2017) give an overview 
of the work which has been done with the Home Care Scheduling Problem and points out areas 
which could be interesting for future work. The homecare planning can be divided into two 
groups. The first is the single-period problems. Here a single working day is optimized. Most of the 
papers focus on this problem. The other group is the multi-period problems where the planning 
focuses on several days, this can be necessary when service requests spread over more than just 
one day. As explained in chapter 2, our problem belongs to the group of single-period problems, 
this chapter will therefore focus on papers with problems belonging to this group. The problems 
solved in the papers have different objective functions and constraints. Most of the papers 
minimize the traveling distance or traveling cost. (Mascolo, Espinouse and Hajri, 2017) gives an 
overview of the objective functions, which can be seen in Table 4.

As it can be seen in the table, the main focus in the literature is on minimizing the cost in different 
ways. Only 6% of the literature takes patients preferences into account when optimizing, and only 
7% have a focus on minimizing the number of unplanned tasks. Most papers consider a multi 
objective function. To our knowledge, no paper has the same objective function and constraints as 
our problem. (Rasmussen et al., 2012) is the paper with the problem formulation closest to ours. 
They have a multi-criteria objective, where they minimize unplanned tasks and traveling cost, and 
maximize patients’ preferences. They also have different dependencies between the tasks. They 
formulate the problem as a set partitioning problem and develop a branch-and-price solution 
algorithm. Their solution is not so interesting for our problem, since the running time is too high. It 
would not be able to handle the real size problems in under a minute, which is demanded in our 
case. Furthermore, they do not have qualifications for the employees which make the problem 
different.  
Other papers also consider patient’s preferences. (En-Nahli et al., 2015) develops a MIP model to 
solve the planning problem. Their objective is to minimize traveling time and maximize the patient 
preferences and employee fairness. Their model can solve small and medium sized problems. But 
it is too slow for our case. (Braekers et al., 2016) develops both an MIP model to find the optimal 
==PAGE:18==
18 
 
solution and a metaheuristic (MDLS) to find solutions faster. Their objective is to minimize 
traveling cost and overtime and maximize patient’s preferences. Their metaheuristic finds several 
solutions to show the trade-off between cost and patients satisfaction. The paper is interesting; it 
shows that problems can be solved fast using metaheuristics. But the solution cannot be used 
directly to solve our problem, since it has a big focus on traveling cost which ours does not. It also 
allows overtime, which makes it different from our problem where we must fit as many tasks as 
possible into fixed worktimes. (Eveborn et al., 2016) minimizes a cost which is a weighted sum of 
traveling cost, traveling time, scheduled hours, preferences and more. They use repeated 
matching to solve their problem.  Their solution method is embedded a tool called Lap Care. Their 
solution is not interesting for our problem, since they do not consider dependencies between 
tasks. They also have a too high running time.  
 
After reviewing literature, we are sure that our problem is unique and has not been solved before. 
All the papers have a focus on reducing cost, we only have a focus on planning as many tasks as 
possible and maximizing the patient’s preferences. Most of the papers also have a too high 
running time e.g. (Rasmussen et al., 2012) which for some instances with 150 tasks uses around 10 
min, or (En-Nahli et al., 2015) which have a running time of over 2 minutes already with an 
instance of 40 service entities and 8 employees. Our algorithm should be able to solve large 
problem instances, as shown in Table 2 and 3 in section 2.2, and with a running time of only 1 
minute. The other papers do not have the same dependencies between tasks as our problem 
does. Some of the papers do not consider dependencies between tasks at all, and some do 
consider dependencies, but not as flexible dependencies as ours. In fact (Mascolo, Espinouse and 
Hajri, 2017) also points out that future work should focus on providing models which can be used 
in real life on real sized problems and have focus on other things than cost e.g. patient’s 
preferences. Which is exactly what our problem has.

4 Development of Basic Algorithm 
 
In order to make the job of developing an algorithm for solving the Home Care Scheduling 
Problem easier, we will in this chapter develop an algorithm which does not consider the three 
dependencies constraints and the must-attend constraint from Table 1 in chapter 2. The algorithm 
developed in this chapter will then, in chapter 5, be extended to also consider these constraints. 
This chapter is structured so that it shows the flow of the development. The different sections first 
present an idea to an algorithm and then test its performance. Based on the performance new 
ideas are discussed, and these new ideas are then used to develop the new algorithm in the 
following section. In this iteration way an algorithm for solving the Home Care Scheduling Problem 
is established. The tests will be performed on dataset 0-2. These datasets are without 
dependencies and must attend tasks, and they give a good representation of the all the datasets 
with respect to size.

4.2 AlgoTwo – Improve existing solution using dynamic programming 
 
Section 4.1 has shown that it is hard to find the big improvements when we only look at small 
route parts at a time. This section will therefore try to improve whole employee routes at a time. 
Section 4.1 also showed that solving MIP-models was not fast enough when large intervals were 
considered. This section will therefore try to achieve a faster running time by instead using 
dynamic programming. Dynamic programming has been used for route planning before. For 
example, is dynamic programming used in (Larsen J., 2001) to generate routes with minimum cost. 
The dynamic programming in (Larsen J., 2001) uses labels which contain the vertex in the route, 
current time and accumulated demand. A similar approach can be used to generate routes in our 
problem.

The idea of the algorithm, AlgoTwo, is to improve employees’ routes by reordering them to make 
room for more tasks. When we consider a whole route at a time, we may save a few minutes at

minutes.  
The explanation for the poor performance is the weak domination criteria. It is very hard for one 
label to dominate another, since the other labels tasks have to be a subset of the dominating 
labels tasks. Because the tasks have time windows and different durations it is not enough only to 
look at the number of tasks in the label. Because of this the algorithm ends up almost generating 
all the combinations, which of course is very slow. Based on these results dynamic programming 
does not seem to be the way to go. 
 
4.3 AlgoThree – Break route planning for walking and cycling employees into smaller groups 
 
The algorithms developed in sections 4.1 and 4.2 were too slow to solve the problem. In this 
section an algorithm, which can be used to split up the problem when planning routes for walking 
and cycling employees, will be developed. If a fast algorithm is developed to make routes for 
walking and cycling employees, the hope is that the remaining unplanned task will be placed 
easier in routes for driving employees.  
 
4.3.1 Description of algorithm, GenerateTaskGroups 
For walking and cycling employees there is an upper limit for the allowed traveling distance 
between tasks. The idea is to use this limit to divide the tasks into smaller groups like illustrated on 
Figure 4. 
 
 
Figure 4: Tasks divided in groups based on traveling distance 
 
The groups are constructed so that for any combination of two tasks from two different groups 
the traveling distance between them is longer than the allowed traveling distance. If an employee 
is then assigned to e.g. a task in the green group, it will never be possible to also assign him to a 
task in the red or blue group. This can be used in later algorithms to split the problem into smaller 
problems, based on the task groups, which can be solved independent of each other.  
==PAGE:29==
29 
 
Graph theory is used to find the task groups. Each task is seen as vertex in the graph. If the 
traveling distance between two tasks is smaller than the allowed distance, there is an edge 
between the two tasks/vertexes. We must then find a spanning forest in the graph. Each spanning 
tree in the forest is a task group. The algorithm to find the spanning forest is inspired by Kruskal’s 
algorithm for finding a minimum spanning tree or forest. In our case we do not care about the 
weight of the spanning forest, so we do not have to sort the edges. Pseudocode 4 shows the flow 
of the algorithm. 
 
 
Pseudocode 4: GenerateTaskGroups 
 
In the beginning when the algorithm starts each task group consists of only one task. The 
algorithm then goes through every edge. The algorithm finds the task groups for the two tasks 
connected by the edge. If the two tasks belong to two different task groups, it joins the groups. In 
the end it returns the found groups. 
 
4.3.2 Performance of GenerateTaskGroups 
Table 6 and 7 show the results of running GenerateTaskGroups on respectively the small dataset, 
dataset 0, and the large dataset, dataset 1. In both cases the algorithm has been run for all walking 
and cycling employees. The algorithm is fast; the total running time is less than 2 seconds in both 
cases. But the result is not what we hoped for. For dataset 0 every employee only gets 1 or 2 task 
groups and for dataset 1 every employee gets 3 or 4 task groups.  This means that we have not 
been able to efficiently split up the problem. The average working time for an employee in dataset 
0 and 1 are respectively 6.1 and 7.2 hours. Table 6 and 7 show that the total duration of tasks in 
each task group is much longer. We would like to have task groups with a total task duration 
closer to the work time of the employees.

4.3.3 Description of algorithm, AlgoThree 
The idea behind the next algorithm is to reduce the size of the task groups so that the total 
duration of a task group for an employee is no longer than the total work time for that employee. 
We still find it likely that walking or cycling employees are not traveling long distances in the 
optimal solution, so we use the distance to split the task groups. The flow of the algorithm is 
shown in Pseudocode 5. 
 
 
Pseudocode 5: AlgoThree 
 
AlgoThree uses the algorithm, GenerateTaskGroups, developed in the previous section. The 
algorithm is given an employee and a set of tasks. In the beginning the allowed traveling distance 
is set to the upper traveling distance limit for that employee. It then begins a while-loop where it 
finds the task groups. In line 8 the task groups are found based on the current limit for traveling 
distance. In line 10 to 17 every task group is checked. If the total duration of a task group is shorter 
than the worktime, the task group is saved. If the duration is too long the tasks from the task 
group are added to the task set. After that the limit on the traveling distance is decreased. The 
new limit is used to find new task groups for the remaining task. This continues until every task 
group has a total task duration no longer than the work time of the employee.

in the main city, it has been divided into many small task groups, with different colours, which all 
have a total task duration of no longer than the employees work time.

4.4 AlgoFour – Generate routes for walking and cycling employees from empty solution 
 
The algorithm developed in this section are not used for improving an existing solution. Instead, 
the algorithm finds new solutions for employees who are walking or cycling by using a MIP-model 
and AlgoThree developed in the previous section. The idea is that we should first plan routes for 
employees who are walking and cycling since they are less flexible than employees driving. After 
these routes are planned it may be easier to fit the remaining unplanned tasks into routes for 
employees driving. 
 
4.4.1 Description of algorithm, AlgoFour 
AlgoFour starts with an empty solutions and plans routes for the employees walking or cycling. 
Pseudocode 6 shows the flow of the algorithm. 
 
 
Pseudocode 6: AlgoFour 
The algorithm first generates an empty solution. It then plans routes for the walking employees 
and the cycling employees. The walking employees are planned first because they are less flexible 
than the cycling employees. For every employee the remaining unplanned tasks, which can be 
assigned to the employee, are divided into task groups using AlgoThree. For each task group the 
optimal route is found using Model 2. To make the algorithm fast, the running time of Model 2 is 
limited to 10 seconds. For most of the task groups Model 2 is solved in under a second, but a few 
==PAGE:38==
38 
 
task groups are difficult to solve. Since the algorithm is heuristic it does not make sense to wait a 
long time for a few slow task groups, therefore we have the time limit. From the generated set of 
routes, the best is selected. The best route is then added to the employee in the solution. In the 
end the algorithm returns the generated solution.

In the following, Model 2 is presented. Model 2 is very similar to Model 1, but there are two 
differences. Model 2 does only consider one employee at a time. The model does not have a 
constraint on skills, since the model is only given tasks which can be assigned to the employee, the 
task groups. For the same reason it does not have a constraint for blocked tasks. Since the model 
is only given tasks from service entities, who have not blocked the employee.

The objective function maximizes the score of the assigned tasks. Unplanned tasks have a cost. 
High priority tasks have the highest cost, so that the model will always plan a high priority task if 
possible. The cost of normal priority tasks is high enough so that it will always be planned if 
possible and not conflicting with a high priority task. Constraints (1)-(5) ensure the flow. So that 
the employee starts and ends in its start task and end task. And every task is both started and 
ended. Constraint (6) forces 𝑦௧ to 1 if the employee is not assigned task 𝑡. Constraints (7)-(8) 
ensure that the maximum travel distance per trip and per day are not exceeded. Constraints (9)-
(11) ensure that the starting and ending time of a task is correct with respect to task duration and 
employee worktime. Constraints (12)-(13) ensure that a task is only started in the allowed time 
window. Constraint (14) ensures that there is enough time for transportation between the tasks. 
Constraints (15)-(18) defines the decision variables.

4.5 AlgoFive – Insert unplanned task in routes for set of employees 
 
In section 4.4 an algorithm to generate routes for walking and cycling employees was developed. 
For the two large datasets it gave very good results when compared with the solutions provided 
by Qampo. The goal in this section is to develop an algorithm which can plan the remaining 
unplanned tasks in routes for the driving employees. The idea is that it does not have to be a very 
complicated algorithm because AlgoFour has done the hard work. AlgoFour already uses too much 
time, so the algorithm developed in this section should be fast. 
 
==PAGE:45==
45 
 
4.5.1 Description of algorithm, AlgoFive 
The basic idea behind the algorithm is very simple. It goes through every unplanned task and 
places it in the best position in a route for one of the employees in the current solution. 
Pseudocode 7 shows the flow of the algorithm.   
 
 
Pseudocode 7: AlgoFive 
The algorithm is given a solution and a set of employees. In line 2 and 3 it takes the unplanned 
tasks from the solution and sorts them in a specified order. The tasks can be sorted in different 
orders, which will be explained. It then goes through every unplanned task in that order. For every 
task the algorithm goes through every employee in the given set. For each employee it finds the 
best position in the route to fit in the unplanned task. Here the position of a task should be 
understood as which number the task is in the route. E.g. if an unplanned task should be placed in 
a route with five tasks there are six different positions. It can be placed as the first or last tasks or 
it can be placed between two of the planned tasks. This is illustrated in Figure 12 where the five 
black circles are the already planned tasks and the red circle is the unplanned task.  
 
 
Figure 12: Possible position in route for unplanned task 
 
 
==PAGE:46==
46 
 
When going through all the employees the algorithm remembers the global best position. For an 
unplanned task there may be feasible positions in several employee routes. In that case the 
solution is updated by placing the unplanned task in the global best feasible positions. It may not 
be possible to find a feasible position for the unplanned task. If no feasible position is found, the 
unplanned task continues being unplanned.  
 
4.5.1.1 Sorting 
The tasks can be sorted in different ways. They can be sorted based on priority; such that high 
priority tasks are first. As more tasks are planned it gets harder and harder to find room for more 
tasks. So, it can make sense to first plan the high priority tasks. 
Because of required skills and time windows some tasks may only be assigned to a few possible 
employees while others can be assigned to almost every employee. For example, dataset 2 has a 
task, which can only be assigned to one of the employees. In the same dataset another task can be 
assigned to 16 different employees. In this case it would make sense to first plan the task with 
only one possible employee. If the task with many possible employees is planned first, it is 
possible, that it ends up blocking the other task from its only possible employee. The tasks can 
therefore also be sorted after how many possible employees they have.  
 
4.5.1.2 Finding best position 
One of the objectives with the algorithm is to plan as many tasks as possible. To do this we should 
avoid a lot of traveling. We also aim for a high score. How good a position is, is therefore based on 
the following two things:  
 
 Minimum increase in traveling distance 
 Maximum score 
 
The first point is the most important. The algorithm always chooses the position with the smallest 
increase in traveling distance. If two tasks both have the smallest increase in traveling distance, 
the position with the highest score is chosen. If we again look at the example in Figure 12 and 
want to place the unplanned task 𝑡௨ between task 𝑡ଷ and 𝑡ସ, the increase in traveling distance is 
calculated as:

4.6 AlgoSix – Improve existing solution 
 
In the previous section, 4.5, a very fast algorithm which could find a good solution was developed. 
In this section this algorithm will be used in an Adaptive Large Neighbourhood Search algorithm 
(ALNS) to get a much better result. An ALNS algorithm is not a very complicated algorithm. The 
basic idea is to destroy and repair a solution several times in order to improve the solution. In 
order for the algorithm to perform well, we should be able to destroy and repair a solution many 
times. It is therefore ideal to use our fast AlgoFive to repair the solutions. ALNS algorithms have 
previously shown good results e.g. in (Ropke S. and Pisinger D., 2006) where it is used to find a 
good solution for the pickup and delivery problem with time windows in a reasonable amount of 
time. You can decide how long the ALNS algorithm should run and we can therefore obey the 
running time limit of 1 minute.

The algorithm has many repair methods, which will be explained in section 4.6.1.3. Because of the 
great amount of repair methods, the algorithm is very robust for changes in the parameters. So, 
changing the values of the parameters does not have a big effect on the final solution. There will 
therefore not be made further tests of these parameters.  
 
In every iteration a percentage of the solution is destroyed. The percentage of the solution to 
destroy will be referred to as the destroy size. The destroy size is updated while the algorithm is 
running. In the beginning 10% of the solution is destroyed. This means that the algorithm unplans 
10% of the already planned tasks. If the algorithm at some point has not accepted a new solution 
in over 10% of the given running time the destroy size is increased with 10%-points. This can 
happen several times, but it cannot increase to more than 50%. First time a new solution is 
accepted the destroy size is decreased to 10% again. Updating the percentage during the 
iterations, may help the algorithm escape local extremes. In section 6.4 the algorithm will be 
tested with different destroy sizes.

5 Expansion of Algorithm 
 
In chapter 4 we developed an algorithm, AlgoSix, which can generate solutions, which are better 
than the solutions Qampo has provided. The generated solutions have a higher score and fewer 
unplanned tasks. This can be done within a running time of only 1 minute. But the algorithm can 
still not handle the full problem. It is not able to handle must-attend tasks and dependencies 
between tasks. In this chapter the algorithm will be expanded to also handle these constraints. 
The final algorithm will be tested in chapter 6.  
 
5.1 Must-attend tasks 
 
As explained in chapter 2, some employees have a list of must-attend tasks. The employees must 
be assigned all the tasks in the list no matter what. It is not difficult to implement this in the 
developed algorithms. When running AlgoSix, it starts with an initial solution. First, we must be 
sure that the must-attend tasks are assigned correctly in the initial solution.  
 
The initial solution is generated with AlgoFive. A sorting is used, where all the must-attend tasks 
are planned first. If an employee must attend a task, the task is marked as blocked for every other 
employee. In this way all the must-attend tasks are planned correctly in the initial solution given to 
AlgoSix.  
 
To avoid that the must-attend tasks are getting unplanned while running AlgoSix, they are marked 
as fixed. The destroy methods are changed so that they check if a task is fixed before unplanning 
them. If the task is fixed, the destroy method does not unplan it.  
 
In this way AlgoSix can handle must-attend tasks. 
 
5.2 Relying on Dependency 
 
If a task 𝑡ଵ RO-depends on another task 𝑡ଶ, it means that 𝑡ଵ can only be assigned to an employee if 
𝑡ଶ is also assigned to an employee. To make sure that AlgoSix returns a solution which does not 
violate these RO-dependencies we both have to look at the repair methods and the destroy 
methods. 
 
All the repair methods use AlgoFive to generate the new solution. As explained in section 4.5.1, 
AlgoFive first sorts the unplanned tasks and then tries to insert them in the solution. AlgoFive is 
changed to avoid generating an infeasible solution. Pseudocode 9 shows the flow of the changed 
AlgoFive. In the pseudocode the algorithm is called AlgoFiveExpand, but we will still refer to it as 
AlgoFive. The changes from Pseudocode 7 in section 4.5.1 is highlighted with green. If the next

5.4 Duration Dependency 
 
The last dependency to implement is the DU-dependency. If a task 𝑡ଵ DU-depends on another task 
𝑡ଶ, it has two values 𝑚𝑖𝑛 and 𝑚𝑎𝑥. 𝑚𝑖𝑛 is the minimum amount of time that must elapse from 
the start of 𝑡ଶ to the start of 𝑡ଵ, and 𝑚𝑎𝑥 is the maximum amount of time that must elapse from 
the start of 𝑡ଶ to the start of 𝑡ଵ. 
 
It is more difficult to implement this dependency. As explained in section 4.5.1.2, AlgoFive tries to 
find room in routes for unplanned tasks by pushing already planned tasks backward and forward 
in time. In this way it can fit in the unplanned task between the planned tasks. How much a task 
can be pushed forward depends on the next tasks in the route. How much time it can be pushed 
forward could therefore be calculated recursively. But with the DU-dependencies, how much a 
task can be pushed backward and forward does not only depend on the other tasks in the route 
but also on the tasks it DU-depends on. This is illustrated in Figure 19. The illustration shows two 
rows of tasks on a time line, each row is a route. We want to place an unplanned task between 𝑡ଷ

time task 𝑡ଷ DU-depends on 𝑡଻ as illustrated with a red line. To push 𝑡ଷ as much backward as 
possible without violating the DU-dependency, we therefore must also push 𝑡଻ and 𝑡଺ backward.

6 Tests and Results 
 
Through chapters 4 and 5 we have developed an algorithm, AlgoSix, which can generate a good 
solution for the Home Care Scheduling Problem. In this chapter the algorithm will be tested. To 
estimate the quality of the generated solution, a method to calculate an upper bound for the 
score, the number of planned tasks and the number of planned high priority tasks will be 
developed. In this chapter we sometimes write NPT instead of “number of planned tasks” and 
NPHPT instead of “number of planned high priority tasks”. The upper bound will be used when 
testing the algorithm. When AlgoSix runs it keeps generating new solutions. The new solution is 
either a new global best, accepted, or rejected. The algorithm will first be tested with different 
settings for rejecting a new solution. It will then be tested if the algorithm uses all the 48 repair 
methods, or if some of them can be taken out to improve the performance. The algorithm 
destroys a part of the solution in each iteration. Finally, it will be tested how the size of the 
destroyed part influences the performance. During the tests, the approach is to fix the best 
performing setting after each completed test. This fixed setting is then used in the following tests. 
This may not result in the overall best setting. It is possible that a setting does not result in the 
best performance in one test, but combined with settings from the other tests, would result in the 
best overall setting. But since it would be too time consuming to test every interesting 
combination of settings, the suggested approach is used. When all the tests have been performed 
a conclusion is made on the quality of the solutions when compared to both the generated upper 
bound and the provided solutions. 
 
6.1 Upper bound 
 
In this section a method will be developed, which when given a dataset generates an upper bound 
on the score, NPT and NPHPT in a solution for that dataset. A natural upper bound for the number 
of planned tasks is that all tasks are planned, but sometimes it is not possible to plan all the tasks. 
Some tasks are impossible to plan; one reason can be that they require a set of skills which no 
employee has. Another reason can be the tasks time window. For example, dataset 0 has a task 
with the time window [14.00; 17.00] and duration of 1 hour and 25 minutes. The earliest possible 
finish time of this task is therefore 15.25, but the last employee finishes his workday at 15.00. It is 
therefore impossible to plan this task. Besides directly impossible tasks, there can also be groups 
of tasks where not all the tasks can be planned at the same time. An example is the must-attend 
tasks, which should always be planned. In some datasets an employee may have two must-attend 
tasks, but not enough time to travel between them. In that case he can only perform one of them 
and the other one is unplanned, since no other employee is allowed to perform another 
employee’s must-attend tasks. At some points of the day there are simply not enough employees 
to perform all the tasks. The solution generated for dataset 2 in section 4.6.2 had 44 unplanned 
tasks, as shown in Table 18. Graph 7 shows at what time of the day the 44 unplanned tasks can be 
performed based on their time window. We see that every one of the 44 unplanned tasks have a 
time window from 14.00 to 17.15. Graph 8 shows the number of employees working during the 
day. The graph shows that not many employees are working in the time window 14.00 to 17.15. It 
will not be possible for the few working employees to perform all 44 unplanned tasks in addition 
to their already assigned tasks.

Based on all this it is reasonable to think that it, for some datasets, is not possible to plan all the 
tasks. So, to be able to estimate the quality of the generated solution, we would like to generate 
an upper bound on the number of tasks planned in a solution. We would also like an upper bound 
for the number of planned high priority tasks and the score. The running time of the upper bound 
generation is not important since the result is only used to compare with the generated solutions 
and not real-life operational planning. But it will not be possible to solve the complete problem to 
optimality. Even though the computer is given a whole day, it is not able to find a solution to the 
complete problem with all the constraints. We would therefore like to make a relaxation which 
gives the problem a structure which can be solved easier. The relaxation must not make the upper 
bound too loose. Graph 9 to 11 show how the total work time of the employees is spent during a 
workday in the three solutions generated in section 4.6.2. The majority of the time is spent on 
performing tasks, the green part. In all three solutions there is also a lot of time spent on waiting. 
This is when an employee waits for a tasks time window to open. The interesting thing is that the 
employees in total only use between 8% and 9% of their time on traveling. Because the 
percentage of time used on traveling is low, the relaxation used in this section will be not to 
consider traveling time and traveling distance. So, when an employee has performed a task, he 
can immediately start performing the next task, if its time window is open. In addition, the 
relaxation will not consider the DU-dependencies. With these relaxations it is not important to 
know in which order e.g. three tasks 𝑡ଵ, 𝑡ଶ and 𝑡ଷ are performed, as long as they are all performed 
in their time window. This new relaxed problem has a structure which can be solved faster, as it 
will be explained. Since only a small percentage of time is spent on traveling, the optimal solution 
for the relaxed problem is likely not to be much better than the optimal solution to the original 
problem. Two different methods for generating the upper bound will now be presented.

6.1.1 Branch and price 
First, we will try to develop a branch and price algorithm to generate upper bounds for the 
number of planned tasks. Figure 23 illustrates the flow of the algorithm. The first step is to 
develop a master problem, which when solved gives the upper bound. From the master problem 
we get the restricted master problem, which only has a subset of the master problems columns. 
We then start a column generation, where the LP-relaxation of the restricted master problem is 
solved, and new columns are generated by the subproblem using the dual values from the LP-
solution. Our restricted master problem is a maximization, we are therefore looking for columns 
with a positive reduced cost. When there are no more columns with positive reduced cost, the 
algorithm checks if we have an integer solution. If we have an integer solution, we are done. 
Otherwise we branch and a new column generation starts. Besides from the relaxations 
mentioned previously, the column generation will not consider must attend tasks, SE-
dependencies and RO-dependencies.

When this branch and price algorithm was implemented in java, it did not perform very well. The 
subproblem was solved fast, but the algorithm just continues branching with only small changes in 
the object value. The algorithm was only able to finish on small datasets. In order to speed up the 
process a new branching technique was tried. Instead of branching on one employee and one task, 
the algorithm branches on one employee and several of his tasks.  Figure 27 illustrates an example 
where the algorithm branches on two tasks at a time. In each step this gives four branches. In the 
example the first branch is on employee 𝑒ଵ and task 𝑡ଵ and 𝑡ଶ, the four branches are:  
1. Employee 𝑒ଵ must perform task 𝑡ଵ and 𝑡ଶ 
2. Employee 𝑒ଵ must perform task 𝑡ଵ and must not perform task 𝑡ଶ 
3. Employee 𝑒ଵ must perform task 𝑡ଶ and must not perform task 𝑡ଵ 
4. Employee 𝑒ଵ must not perform task 𝑡ଵ and 𝑡ଶ 
The algorithm was run with up to four tasks per branching giving 16 branches. But the algorithm 
still had the same problem where it kept branching. The explanation for this is that for many of 
the datasets, when removing traveling distance from the problem many of the tasks become 
almost identical. They have the same duration and time window and maybe just a few differences 
in required skills. To understand the problem with identical tasks, let 𝑡ଵ and 𝑡ଶ be identical tasks. 
𝑡ଵ is planned in routes for employee 𝑒ଵ and 𝑡ଶ is planned in routes for employee 𝑒ଶ. If the 
algorithm then branches on 𝑒ଵ and 𝑡ଵ, the two tasks can just switch places. After the branch 𝑡ଵ is 
now planned in routes for 𝑒ଶ and 𝑡ଶ is planned in routes for 𝑒ଵ. This can happen a lot of times 
resulting in a lot of branching without change. It may be possible to find a branching technique 
which can handle the problem with tasks switching routes, but instead we will try to solve the 
whole upper bound problem as one MIP-model. The idea will be to expand the subproblem to 
look at all employees at once. The hope is that the model will still be solved fast after this 
expansion.

These three criteria are directly implemented based on the quality criteria of a solution explained 
in chapter 2. It is most important to plan the high priority tasks. The second most important is to 
plan as many tasks as possible. The third most important is that the solution has a high score.  
 
The criteria for a solution to be a new global best solution is fixed. We would now like to test 
different rejection criteria. A solution is accepted if it is neither a new global best solution nor 
rejected. The different ways of rejecting a solution, which will be tested, will be based on 
combinations of the five rejection criteria shown in Table 24. Rejects 1 and 2 are focused on the 
score. When Reject 1 is used, a new solution is rejected if the score is lower than the score of the 
previous solution. Reject 2 is more loose, a new solution is only rejected if the score is lower than 
90% of the score of the previous solution or lower than the score of the global best solution. In 
both cases a solution will never be accepted, if it has a score lower than the score of the global 
best solution. Reject 3 and 4 have a focus on the number of unplanned tasks. Reject 3 rejects a 
new solution if the number of unplanned tasks is at least two higher than the number of 
unplanned tasks in the global best solution. Reject 4 is stricter since a new solution is rejected if it 
has more unplanned tasks than the global best solution.  Finally, Reject 5 rejects a solution if the 
number of unplanned high priority tasks is higher than the number of unplanned high priority 
tasks in the global best solution.

6.3 Set of Repair Methods 
 
AlgoSix has 8 different destroy methods and 48 different repair methods. This is a very high 
number of repair methods. There is only a small difference between the different repair methods, 
so it may be possible to remove a set of repair methods and let the remaining repair methods 
compensate for them. By reducing the number of repair methods, it may be easier for the 
algorithm to reward the good performing repair methods and punish the repair methods with a 
poor performance. There is a chance that the reward and punish systems are drowned in the large 
number of methods.  
 
For each repair method 𝑟 the algorithm has a value 𝑤௥. The probability of the algorithm using 
repair method 𝑟 in an iteration depends on how high the value 𝑤௥ is. If 𝑤௥ is high repair method 𝑟 
has a higher probability of being used. If repair method 𝑟 results in a good solution, 𝑤௥ is 
increased. If a solution generated with repair method 𝑟 is rejected 𝑤௥ is decreased. How 𝑤௥ 
develops during the iterations therefore gives a picture of how repair method 𝑟 performs.  Graph 
20 to 25 show the development of 𝑤௥ for the 48 different repair methods, when AlgoSix is used 
on dataset R6. As explained in section 6.2 most of the improvements happen in the beginning. This 
is also seen in Graph 20 to 25; all the 𝑤௥ values approach zero because all new solutions start 
getting rejected in the last iterations. From the graphs it is not possible to point out specific repair 
methods whose performance is either very good or very poor. Some of the repair methods, like 
Repair 15 in Graph 21 or Repair 18 in Graph 22, do not perform well most of time but then 
suddenly have an increase in 𝑤௥ during some iterations.

6.4 Destroy Size 
 
In each iteration the algorithm destroys a part of the solution. In the first iterations the algorithm 
destroys 10% of the solution. We say that the destroy size is 10%. If at some point no solution has 
been accepted in more than 10% of the given running time, the destroy size is increased by 10% 
point. The destroy size can be increased until it reaches 50%. 
When running the algorithm most of the improvements happen in the beginning as explained in 
section 6.2. We will therefore not see, that the destroy size increases and decreases a lot of times. 
Graph 32 shows a case for dataset R6 where all improvements happen while the destroy size is 
10%. Just before iteration 2000, no new solution has been accepted for 10% of the running time 
and the destroy size is therefore increased to 20%. The algorithm keeps rejecting the solutions and 
the destroy size increases until 50% and keeps this size for the remaining iterations. Graph 33 
shows another case for the same dataset. Before iteration 1000 the destroy size is increased to 
20% and again 30%. While the destroy size is 30% a new solution is accepted and the destroy size 
is therefore decreased to 10% again. The destroy size remains at 10% for the remaining iterations.

6.5 Conclusion on Results 
 
We have now tested different ways of rejecting a new solution.  The algorithm performed best, 
when we used Constant 1 in Table 25 in section 6.2. We have also tested the performance with 
different subsets of repair methods. Here the algorithm performed best when we used Repair 
Subset 7 in Table 27 in section 6.3. Finally, we tested how the algorithm performed when we

showed that the algorithm is very stable when the settings are changed. The difference between 
the best setting and the worst setting was never more than 2% point on the three measures; 
average score, average number of planned tasks, and the average number of planned high priority 
tasks. Some of the results were so close that it may have been a coincidence and not the setting 
that made the difference. The reason for the very close results may be that the results in fact are

7 Conclusion 
 
In this Master’s thesis project, an algorithm for generating solutions for the Home Care Scheduling 
Problem was developed. The project was carried out in cooperation with the Danish company 
Qampo, which provided datasets for the problem. The developed algorithm is based on Adaptive 
Large Neighbourhood Search. The algorithm is provided with a dataset and a feasible solution and 
then starts iterating. In each iteration the solution is destroyed and repaired in order to obtain 
new and better solutions. The solution is destroyed and repaired in different ways in order to look 
for solutions in different neighbourhoods. In the end the best of the generated solutions is 
returned. The algorithm should be fast enough to be used in real life operational planning. It was 
therefore only given a running time of one minute. The algorithm is a metaheuristic and contains 
some randomness. In order to estimate the quality of the generated solutions, a MIP-model for a 
relaxation of the problem was therefore solved with CPELX to find upper bounds for the generated 
solutions. Comparing the generated solutions with the upper bounds showed that the developed 
algorithm is able to generate good solutions, which are often close to optimal. The tests also 
showed that even though the algorithm contains some randomness it is very stable. When given 
the same dataset several times the quality of the returned solutions does not vary much. The 
returned solutions have approximately the same quality. For some of the datasets Qampo had 
provided their own solutions. The developed algorithm was also able to outperform these 
solutions. For some of the datasets it performed much better. 
 
The quality of a solution was measured by three things with increasing importance; number of 
planned high priority tasks, number of planned tasks and score. In future work it could be 
interesting test the trade-off between these three objectives. Would it be possible to get a much 
higher score, if we maybe planned a few less tasks? Would we sometime be able to plan a lot 
more normal priority tasks if we abstained from planning a single high priority tasks? This is 
interesting questions because the real world is not black and white (high priority and normal 
priority). The employees working with the home care service could be presented with a list of 
alternative plans and then use their professional experience to judge which plan to use. 
For some of the datasets it was not possible to plan all the tasks. In future work it could also be 
interesting to test if more tasks could be planned if some of the constraints were relaxed. For 
example, do some of the tasks have to performed by the same employee. Would it maybe be ok 
to let different employees perform the tasks, if this resulted in more tasks being planned? Another 
example could be the time windows of the tasks. Would it be better to start performing a task 15 
minutes after its time window instead of not performing the task at all? The problem has many 
different constraints so many different relaxations can be studied in future work.
