Abstract
This paper aimed to compare Lab Streaming Layer and Apache Kafka frameworks
for creating a data pipeline for mixed reality Varjo XR­1 head­mounted device
(HMD). By measuring the delay and jitter of the data streams: eye­tracking, position,
rotation, and video streams with the Lab Streaming Layer receivers and Apache
Kafka consumers. Pull Chunk receiver from Lab Streaming Layer came out on top
with the least amount of delay and jitter among all the data streams on average.
Due to unforeseen delay in the PsychoPy experiment, another experiment was
conducted without participants, resulting in no issues with delay or jitter.

Introduction
1.1
Motivation
Advancements in human­computer interaction applications where users face a
computer screen and interact with a keyboard and a mouse are moving towards
more complicated devices [1]. Remarkable progress has been achieved in computer
processor speed, memory, and data storage for the last few years, which opens
up innovation in newer input and output devices which includes eye­tracking [2],
[3], [4].
Eye­tracking has been popular in the research community for some time now, and
it focuses mainly on systems used in lab experiments. However, many commercial
and experimental systems are available today for many applications [5], [6].
Head­mounted devices (HMD) have often been studied for desktop applications
because of technological advancements, reduced size and weight, and lowered
costs, allowing researchers to investigate different applications. Valuable data
can be collected with head­mounted devices (HMD), which can be enhanced with
several data streams to understand human behaviour, which includes eye­tracking
data, video, head directions, and hand movements [7].
Head­mounted devices (HMD) can cause motion sickness and discomfort when
using such devices, and a fundamental cause of that problem is the delay. The
delay is the time it takes to go from one endpoint to another endpoint [8].
This paper will be focusing on measuring the delay for various data streams inside
the Varjo XR­1 device between two different frameworks.
1.2
Goal
There is a need for a framework that helps researchers focus on their tasks and
not get lost in technical details when collecting data in their experiments. Several
frameworks exist already to create data pipelines, but none fit our needs exactly.
However, they have different essential points. Therefore, we clarify our requirements,
analyze existing frameworks, and compare two potential frameworks that fit our
needs. We will demonstrate that it works by carrying out several experiments.

Initial Problem Statement
This project aims to research various frameworks suitable for streaming data to the
experiment software PsychoPy, including ioHub, Lab Streaming Layer, Apache

pipeline to stream all the data streams from the Varjo XR­1 device to PsychoPy.

data. The data streams need to be streaming in near real­time into the experiment
software PsychoPy while an experiment is recording.

Lab Streaming Layer
2.3.1
Introduction
Lab streaming layer library allows for time­series data to be transferred between
devices on the local area network and is cross­platform [13]. The core transport
library is built in C/C++ and has programming language support in Python, Java,
C#, and MATLAB. [14].
Figure 2.2: LSL Core Components [15]
Framework for Experimental and Physiological Data Collection in Mixed Reality
7
==PAGE:16==
2.3.2
Sample or Chunk
There are two ways of collecting data with the library, either using a sample that
collects a single sample or chunk that collects multiple samples for improved
throughput or setting the max samples to a single sample to reduce delay [14].
2.3.3
Stream
A stream in the library is a combination of sampled data from a device with its
Metadata. The stream can support a regular sampling rate for audio samples at
44100 Hertz and video samples at either 24 or 60 Hertz. An irregular sampling
rate can support one or more channels, and the data stream must have the same
data type, which can be integers, floats, doubles, or strings [14].
2.3.4
Stream Outlet
A stream outlet makes sure that a data stream is available on the local area
network. Data can be either sent as a sample or as a chunk into the outlet. When
creating an outlet, the stream is made available to the local area network where
other computers can subscribe to it from a Resolver and then connecting a Stream
Inlet to it [14].
2.3.5
Stream Inlet
A stream inlet makes sure that the data can be received from a single connected
outlet on the local area network. It allows retrieval of samples from an outlet in
order and having optional type conversion and failure recovery, and with a reliable
re­transmission [14].
2.3.6
Resolver
A resolver makes sure that it resolves data streams that are available on the
local area network, which can be content­based queries by content­type, name or
queries on metadata [14].
2.3.7
Built­in clock
The library has a built­in clock that allows timestamps to be transmitted samples
that can be mutually synchronized [14].
2.3.8
Reliability
The library data transport protocol uses the TCP (Transmission Control Protocol)
[16] which is message­oriented and type­safe. It provides an automatic failure
recovery to minimize data loss. The data is buffered on the sender and receiver
to allow periodic network failures. The transmission is type­safe and supports type
conversion [14].

2.3.9
Time Synchronization
The library has a built­in time synchronization for all recorded data, designed after
the Network Time Protocol (NTP) [17] to achieve sub­millisecond precision on the
local area network [14].

2.4
ioHub
2.4.1
Introduction
ioHub is an event monitoring framework that provides device event monitoring,
real­time reporting, storage of input device events throughout the system for PsychoPy
experiments [18]. The framework comes with integrated support for eye­trackers
from SMI, SR Research, Tobii LC Technologies, Eye Tribe, and Gazepoint [12].
2.4.2
Device event monitoring
ioHub is responsible for monitoring, bundling, and storing device events that run
on an individual process separate from the PsychoPy process. ioHub DeviceEvents
are monitored continuously throughout the system relative to the PsychoPy window.
There is no graphical window needed to monitor the supported devices. ioHub
Device event monitoring and callback processing happen in parallel despite the
PsychoPy process state [19].
2.4.3
Data storage and retrieval
ioHub Device event data is stored as an ioDataStore HDF5 file to the ioHub Server
[20], [18]. The data can be accessible during an experiment using the Python
NumPy package to retrieve the data in NumPy arrays [19].
2.4.4
Integration with PsychoPy
ioHub is integrated with the PsychoPy experiment software. Debug messages
can be saved to PsychoPy logs and PsychoPy LogEvents to ioDataStore. ioHub
and PsychoPy share the same time base using the PsychoPy global time base
psychopy.core.getTime() [19].
2.4.5
Time synchronization
ioHub process provides a shared timebase that can automatically synchronise
device events from multiple physical and virtual devices [19].
2.5
Apache Kafka
2.5.1
Introduction
Apache Kafka is an event streaming platform that uses a queue to handle high
volumes of data. Transferring data from one endpoint to another and is suitable for
offline or online data consumption. The data is persisted on the disk and replicated
inside a cluster to prevent data loss [21].
10
Framework for Experimental and Physiological Data Collection in Mixed Reality
==PAGE:19==
2.5.2
Brokers
Apache Kafka can run on one or multiple servers, which can also be on multiple
data centres. The multiple servers can form a single storage layer called brokers
[21].
2.5.3
Zookeeper
Apache Kafka uses Zookeeper for controller election, cluster membership, topic
configuration, access control lists and quotas [22].
2.5.4
Kafka Cluster
Apache Kafka cluster consists of one or more brokers running a single instance
of Kafka, which is highly scalable and fault­tolerant. The brokers share workload
amongst each other and ensure that another server can take over the work of the
failed server to continue the data pipeline work without any data loss [21].
2.5.5
Clients
Clients allow developers to write distributed applications and microservers which
read, write and process streams of events simultaneously.
Kafka clients are
available in C/C++, Java, Scala, Python, Go, and other programming languages
[21].
Figure 2.4: Apache Kafka Architecture [23]
2.5.6
Producer
A producer is a client application that makes sure that the messages are published
to a single topic in the Kafka cluster [21].
Framework for Experimental and Physiological Data Collection in Mixed Reality

==PAGE:20==
2.5.7
Consumer
A consumer makes sure that the messages can be consumed by subscribing to
a topic from the Kafka cluster [21].
2.5.8
Topic
Apache Kafka organises messages into categories which are called topics. Each
topic has a unique name across the entire Kafka cluster [21].
2.5.9
Partition
A topic is spread over one or many partitions, enabling producers and consumer
loads to be scaled to handle an arbitrary amount of data [21].
2.5.10
Replication
Apache Kafka replicates messages for each topic partition across several servers.
Moreover, allowing for automatic failover to the replicas when a broker cluster fails.
That ensures that the messages will remain available in the presence of failures
[21].

An open­source framework called Unlock separates the user interface and interaction
logic from the data acquisition. The framework is interesting for researchers in
collecting electroencephalograms (EEG), electromyograms (EMG) or gaze positions.
Functionality includes recording, decoding and working on various signals. The
framework’s focus is on visual BCI through EMG, EMG and eye­tracking record
support. The framework integrates into the Lab Streaming Layer to support the 20
most commonly used EEG recording hardware devices and various eye­trackers.
Experiment sessions were conducted with participants sitting down in a chair
approximately 70cm from the LCD computer monitor with a display resolution of
1920x1080 pixels with a 120 Hz refresh rate. Tobii EyeX tracker was used to track
the participants gaze position during the experiment. Unlock software was running
on a single computer for data collection and task presentation. EEG and gaze data
were streamed into Unlock using Lab Streaming Layer. Event markers such as the
c­VEP presentation cycle were created in Unlock software then synchronized with
the EEG and eye­tracking data with the timestamps generated with Lab Streaming
Layer [28].
Multimodal experiments about human behaviour create a much broader understanding

Apache Kafka
LinkedIn discusses their design and engineering problems when moving from a
data pipeline from batch processing to Kafka’s real­time publish­subscribe system.
LinkedIn’s production pipeline currently handles more than 10 billion message
writes each day and over 172,000 messages per second. Kafka supports various
subscribing systems that can deliver more than 55 billion messages to these
consumers, which process every day [30].
At LinkedIn, Kafka has been used in their production since August 2010. For their
Kafka infrastructure, they have a single Kafka cluster data centre that focuses
on frontend services. The frontend services produce different event data, and
the event data are published to local Kafka brokers in batches. Consumers run
within the same data centre, including processing news postings for each user,
processing that data into a real­time search index, and aggregating discussions
among groups or forms within the same industry. Finally, a security application
limits the account’s usage of accounts with periodically high activity.
Another
Kafka cluster is deployed in a separate data centre for offline analysis, close
to their Hadoop cluster and data warehouse infrastructure. They run reporting
jobs and analytical processing on the data. The end­to­end delay for their entire
pipeline is 10 seconds on average [31].
This study will discuss a replicated logging system with Apache Kafka built. Moreover,
discuss its design and engineering knowledge when replicating Kafka logs for
multiple distributed data­driven systems at LinkedIn, including data storage and
stream processing. There were two main requirements when extending Kafka
to a replicated logging system. A log consensus protocol and resilient stream
processing [32].
The log consensus protocol was used to keep the replicated log consistent between
servers to ensure that the replicas in the log had the same entries in the same
order. Therefore if one server would go down, there was a safety net in place.
Kafka message log replication was designed and implemented to leave the consensus
protocol, such as leader election and membership changes from the log [32].

Introduction
This experiment aims to validate and measure the eye­tracking delay of the receivers
of Lab Streaming and consumers of Apache Kafka. It is essential to set up and
mimic a simple experiment that shows a dot on the screen for the user representing

data pipeline. The endpoint is the PsychoPy experiment, where the data will be
received. The end timestamp minus the start timestamp of the eye­tracking data
is the eye­tracking data delay shown in the figure 6.1.

6.2.1
Introduction
Due to unexpected eye­tracking delay results from Experiment 1, it was decided
to replace the PsychoPy data pipeline endpoint with a simple Python script. The
reason was to investigate if the data pipeline was producing any unforeseen random
delays or not. By removing PsychoPy from the experiment and replacing it with a
simple Python script, we could identify if PsychoPy is the cause of the problem or
the data pipeline. The previous experiment measured only the eye­tracking data
stream.

information, and video data streams delays. All the data streams were tested
individually for Apache Kafka consumers Confluent Kafka and Python Kafka. The
same was done for Lab Streaming Layer receivers Pull Sample and Pull Chunk.
The consumers and receivers were tested ten times for each data stream with no
human involvement.

8.3
Implications
This research shows that Lab Streaming Layer is a better choice for connecting a
data pipeline with Varjo XR­1 than Apache Kafka.
8.4
Limitations
The most important limitation in Experiment 1 was the MilliKey Delux Light Sensor,
an attachment of SV­2 MilliKey Response box responsible for collecting Sensor
Delay in the experiment 6.4.
The plan was to set the sensor inside the head­mounted device (HMD). However,
there was a problem with mounting the sensor inside the HMD because it was
loose and moving a lot.
Due to this limitation, the light sensor was moved from the Varjo XR­1 to the screen
upper left corner, where it was stable. For this limitation, the experiment’s focus
changed to measure the delay in milliseconds it takes for the eye­tracking data to
travel from the start point to the endpoint of the data pipeline.
Another limitation was that the Varjo Base calibration would sometimes decalibrate
during the experiment. The limitation resulted in repeating the experiment.
Another limitation was that the equipment needed for developing and testing with
the head­mounted device was only in a single room in DTU which other students
were using the same equipment, resulting in a limitation of the time I had and
being very dependent on the equipment.
8.5
Future work
Synchronize all the delays from start to finish in the delay timeline 6.4 and optimize
the Lab Streaming Layer data pipeline to be even faster.

Conclusion
The goal was to compare Lab Streaming Layer and Apache Kafka frameworks
for creating a data pipeline for Varjo XR­1 head­mounted device. Pull Sample
and Pull Chunk receivers from Lab Streaming Layer and Confluent Kafka and
Python Kafka were their delay and jitter were compared against each other. This
was done by conducting several experiments where 13 participants took part in
a PsychoPy experiment where users gazes were recorded with a data pipeline.
Due to unforeseen delay in Experiment 1, another experiment was conducted
without participants, resulting in no issues with delay or jitter.

receivers/consumers. Pull Chunk receiver had the least amount of delay and jitter
among all the data streams on average.
