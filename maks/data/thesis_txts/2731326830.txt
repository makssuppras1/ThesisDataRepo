Abstract
Log Analysis is a set of techniques to extract knowledge from computer-generated application logs. In the last
decade, structured logging has emerged and improved the field significantly by allowing Log Analysis tools to
extract the varying parameters of a log message automatically. Its introduction has made significant research
obsolete in automatically detecting and extracting those parameters from unstructured log messages.
Process Mining is a research area which enables the systematic analysis of events, mainly to discover the
underlying processes and enhance or correct them. Object-Centric Process Mining is a new and upcoming
sub-area that allows us to discover processes from various perspectives. Until this point, Process Mining and
Log Analysis can and have been combined but require much preparation when using unstructured logs due
to the aforementioned challenges.
This thesis aims to simplify the combination of Log Analysis and Process Mining by combining structured
logging and object-centric Process Mining in a novel way. This combination allows us to use existing Object-
Centric Process Mining tools directly and without domain-specific preparations. To enable this, we propose a
novel mapping from structured log messages to Object-Centric Event Logs, in which the varying parameters
are treated as dedicated objects. This lets us view the applications’ behaviour visually and from multiple
perspectives, opening the door to new ways of analysing structured application logs, which were previously
not readily possible.
We validate our proposal by implementing a new Log Analysis application capable of handling Object-Centric
Event Logs, combining the strength of Log Analysis and Process Mining techniques. We implement several
components, such as special logging adapters for our novel mapping, object-centric discovery algorithms,
graph layouting algorithms, and more, to enable this. We perform a case study on an actual application
to evaluate existing tools, define the required features, and finally evaluate our new application and its
usefulness.
The result of this thesis is positive in that we could successfully fuse recent advancements of the two
research areas in a meaningful way, opening up new ways of analysing structured application logs. We have
paved the way by implementing many required components and discussing various techniques and their
advantages and disadvantages. We have, however, faced many challenges in creating a viable prototype,
which downgrades the overall quality of it but is unrelated to its fundamental concept. Further research
could resolve these challenges, making our proposed way of analysing structured application logs properly
viable for real-life usage.

1.1 Log Analysis
Logging is the process of recording event information in software systems
and writing them to an output, often text files called log files. Logs are
generated by logging statements within an application’s source code. This
is often complemented by underlying technologies that the application
uses, for example, a web framework. Logging allows for great flexibility
in the detail and format of log files, which in practice, however, can lead
to inadequate and superabundant amounts of events and detail [1].
Log Analysis is a research area focusing on extracting knowledge from
log files. Typically, developers use log files to troubleshoot unexpected
behaviour by manually inspecting the logs where the relevant events
occurred. The issue can then be locally reproduced and fixed. This is
only one use case, though, and there exists a plethora of other sub-
areas, with each having its use case and intended user, such as anomaly
detection, performance analysis, application security, user-behaviour
analysis, pattern detection, etc. [2].
A significant challenge in Log Analysis is that logs are often written to
"flat" text files, where all information, including timestamp, log level,
message, and more, is written to a single line. It is also common that this
information is split into distinct "columns" to allow for improved filtering
and sorting based on a column. However, the log messages themselves
are typically written as-is, with the parameters embedded into the string1.
This makes searching and filtering logs non-trivial if we wish to, e.g. view
all events emitted by a given log statement. As such, another sub-area
of Log Analysis is dedicated to log parsing [2, Category G], the process
of detecting the format of unstructured logs and converting them to
structured logs, improving upon the manual task of analysing logs, as
well as enabling automated analysis.
This problem can, however, be overcome at the source by emitting struc-
tured log files straight away2. For this purpose, a formal grammar called
message templates [4] has been developed, which allows developers to
place named holes into logging statements, which are then filled by values
from the application state when emitting a log event. Logs are stored in
this structured format, making it easy to find all events originating from
the same statement. Furthermore, it allows for searching and filtering
for specific holes. When viewing such logs, they can be rendered to
human-friendly text by filling the holes with actual values, allowing
them to be readable in a manual analysis. This formal grammar has
since been adopted in several logging libraries for various programming
languages [4].

1.2 Process Mining
Process Mining (PM) is a set of techniques capable of analysing event
logs from any system, but often software systems, to gain insight into
the underlying processes and improve them. Often, Process Mining
is applied when little, incomplete, or incorrect information about the
processes exists, with the purpose of discovering the true processes based
on actual data.
Broadly speaking, there are three types of Process Mining: Discovery,
Conformance, and Enhancement [5]. As the name suggests, the first type
focuses on discovering process models from event logs without any prior
information about the process. Many discovery algorithms have been
proposed over the years, with various output formats such as the Directly-
Follows Graph (DFG), Petri Net (PN), and Process Tree. Conformance
checking compares existing process models to other event logs of the
same process, making it possible to detect whether any process rules are
broken. Finally, Enhancement focuses on improving an existing process
model with the new insights gained by analysing the process based on
the actual data.
For a long time, Process Mining research has been working under the
assumption of a single case notion, i.e., the identifier used to distinguish
which trace an event belongs to. In reality, this is not a realistic assumption
since events in many real-life systems can refer to multiple types of
identifiers that could all be considered the case notion3. Attempting to
force such an event log into a single case notion can lead to convergence
and divergence, damaging the process model by potentially replicating
events multiple times and introducing self-loops where they do not exist
in reality [6].
To deal with this issue, a new sub-area called Object-Centric Process
Mining (OC-PM) has gained traction amongst the research community in
the past few years. While most PM tools, especially commercial solutions,
still only have support for classical event logs, OC-PM is well on its way to
replace or at least complement traditional techniques [7]. New event log
formats have been specified [8–10], extensions or entirely new discovery
and conformance checking algorithms have been proposed [11–14], and
new tools have been developed to use the new formats and algorithms
[15–17].
1.3 Log Analysis & Process Mining
Given the fact that application logs are essentially event logs from a
Process Mining perspective4, Process Mining and Log Analysis is a
natural fit to discover an application’s runtime process model [18, 19].
This is nothing new, but the challenges of unstructured log files (see
Section 1.1) have always posed a significant obstacle, as the relevant log
parsing technique has first to be identified and then applied to the specific
log format of the given application5.

1.4 Research Questions
Both research areas have seen a shift to more structured approaches over
the last few years, with the introduction of structured logging in many
logging libraries and the introduction of Object-Centric Process Mining
techniques. We see this as a perfect opportunity to take a fresh look at the
connection between Log Analysis and Process Mining by investigating
whether it is possible to use structured logs with these new techniques
to gain insights that were not possible with classical Log Analysis or
Process Mining.
To approach this topic, this thesis will attempt to answer the following
four research questions:
1
Is it possible to convert structured application logs that use the
message templates grammar to object-centric event logs? Which
object-centric format is best suited for this, and which storage
method is most performant for continually writing large amounts
of logs?
2
Is it possible to discover a runtime behaviour model of an applica-
tion from its structured log files, by using Process Mining discovery
techniques for Object-Centric Event Logs? Can this model be visu-
alised to show this behaviour from various object perspectives?
3
Is it possible to diagnose failures in structured logs by replaying
the failing trace in the discovered model? Can this be visualised
on top of the discovered model, showing the actual trace values
instead of log message templates?
4
What guidelines should applications follow regarding their logging
strategy (e.g. where-, what-, how-, whether-to-log) in order to

2.1 Logging
Definition 2.1.1 (Log, Application Log) A list of events occurring in a

The following definitions formally describe the concept of a structured

their origin, enabling the use of Process Mining techniques.

3.4 Graph Layouting
Discovering process models using Process Mining techniques, including
those from Section 3.3, typically yields some graph. This might be a
Directly-Follows Graph (DFG), Petri Net (PN), Process Tree, or something
else.
Since practitioners directly use these graphs to analyse processes, they
must be laid out pleasantly. As we shall see in Chapter 6, the industry
standard for Process Mining graphs often leaves much to be desired.
Therefore, we also look at related work for Graph Layouting.
There are various approaches to drawing graphs, such as force-based,
circular, orthogonal, layered, and more. The following subsections focus
solely on layered graph methods8, as these are typically most appro-
priate for drawing the types of graphs that Process Mining techniques
produce.
3.4.1 Sugiyama Framework
Sugiyama et al. [40] introduced a method to draw layered graphs in 1981.
The core idea of this approach is to assign each node a rank/layer and
to draw edges downwards whenever possible. To increase readability,
layouts should minimise edge crossings, make edges as short as possible,
and draw lines as straight as possible.
The framework consists of four main steps:
1. Remove cycles and assign a rank to each node
2. Order nodes within a rank to minimise edge crossings
3. Determine actual node positions
4. Draw edges
This four-step framework has become increasingly popular in literature,
and most modern layered graph drawing algorithms are based on this
approach.

The previous tools and methods all produce high-quality drawings of
directed graphs. In the context of Process Mining (PM), however, pro-

to the importance of the underlying nodes and edges, causing the most
important nodes to be spread around the graph11, making it difficult
for users to follow the most common process instances [45]. Another
issue with the previous methods is that the layout can drastically change
with every little change to the model, such as when adding or removing

pre-compute rank assignments and node orderings for all nodes in the
event log and then apply it to each sub-graph we want to render. This
has the effect that nodes stay in the same relative position no matter how
much of the entire event log is filtered out, allowing users to preserve
their mental map. Furthermore, the trace variations in the event log are
added in order of importance, causing the most frequent nodes to appear
in the centre, forming a backbone of the process graph. This makes it much
easier for users to follow the most common paths.

To perform Log Analysis on application logs using Process Mining (PM)
techniques, we must first collect the logs in a proper format compatible
with PM concepts and existing tools. In Section 1.4, we have narrowed
our focus down to only structured logs (see Definition 2.1.5). This chapter
will partially answer Research Question 1 by choosing an appropriate
format, as well as defining a mapping of structured logs to it, and
describing an implementation of the methods. The performance of
logging is evaluated in Section 8.1.2. We also partially answer Research
Question 4 in Section 4.6 by looking at relevant literature that discusses
the topic.
4.1 Choosing a format
Most logging frameworks provide a variety of formats to write log
events to, ranging from simple text files to advanced cloud services.
Since structured logs are technology-agnostic, it is desirable to use a
universal log format that can be understood by all systems, no matter
the programming language. Such compatibility would allow us to collect
logs from a variety of different systems and allow us to use them in other
tools outside of this work as well.
Since this work aims to combine Log Analysis and Process Mining
techniques, it is also desirable to use a format that is known by the
PM community. Whether structured logs are directly constructed in
such a format or converted from an intermediate format later is up to
the practitioner. However, many modern logging frameworks support
writing to different formats simultaneously. This reduces the manual
work required to import log files into a Process Mining tool, as no
conversion is necessary.
As explained in Section 1.2, Process Mining has had the assumption
of a single case notion for a long time. As such, there exists an official
IEEE Standard called eXtensible Event Stream (XES) [48], which defines
a grammar and XML schema for event logs and streams. Practically all
Process Mining tools have built-in support for this standard, and various
extensions make it suitable for a wide range of use cases.
To rectify the limitation of single case notions, an extension to the XES
Standard, eXtensible Object-Centric event logs (XOC), has been proposed [8].
More recently, another format called Object-Centric Behavioral Constraint
model (OCBC) has also been proposed [9]. Both these formats are rather
complex and, unfortunately, face scalability issues [10, 49]. To improve
upon both complexity and scalability, a new standard for Object-Centric
Event Logs (OCEL) has been proposed [10]. This standard has already
gained traction since its proposal by being included in tools like pm4py
[34], amongst others. Furthermore, an extension to the OCEL standard
has recently been proposed to handle changing attributes and object
linkage better [49]. A second version of the standard has been released

4.2 Mapping log events
The OCEL standard [10] formally defines an Object-Centric Event Log to
be the tuple 𝐿= (𝐸, 𝐴𝑁, 𝐴𝑉, 𝐴𝑇, 𝑂𝑇, 𝑂, 𝜋𝑡𝑦𝑝, 𝜋𝑎𝑐𝑡, 𝜋𝑡𝑖𝑚𝑒, 𝜋𝑣𝑚𝑎𝑝, 𝜋𝑜𝑚𝑎𝑝,

events and objects. Events may reference multiple objects at once, and
objects may be referenced by multiple events, allowing for a many-to-
many relationship. This solves the issue of single-case notions, as we now
have multiple viewpoints for different object types, from which we can
conduct an analysis.
To add arbitrary data to such a log, we have the option to attach attributes
to events themselves and to create dedicated objects which themselves
have attributes. The key difference is that event-level attributes are
contained within a single event, while objects may be referenced by
any number of events. An object type is simply a string to identify the
expected attributes of object instances and to allow for the grouping
objects of the same type.

this quickly accumulates to hundreds, if not thousands, of object types
and possibly several versions of the same object types due to inconsistent
naming. In the example above, the duration hole is not a relevant object
type and would be more suitable as an attribute. Unfortunately, there
is no way of identifying such distinctions beforehand without domain
knowledge of the solution where the logs are generated. It is, therefore,
preferable to collect everything as objects and then let users convert some
of them to event-level attributes.

than is contained in the message template. This includes information
like the log level, rendered message template, attached exception, and
more. While this is an implementation detail, we generally want to add
such information as attributes. Furthermore, it should be possible for
practitioners to define beforehand which properties of a log event should
be treated as attributes instead of objects. One way to do this is by
prefixing the property name with a known constant so the logger can
identify such properties.

actual value [12]. Log events do not necessarily have access to previous
events and can, therefore, not know whether an object with the same
type and value has previously been encountered. And if it does have

4.3 Information obtainable by logging
When performing analyses on event logs in Process Mining, it is a routine
task to consider event attributes besides only the activity name and
timestamp. Attributes can be used to filter events that do or do not have
a specific attribute value, allowing practitioners to focus their analysis
on specific areas of a process1.
However, most Process Mining tools treat attributes as complementary
information for advanced filtering and digging deep into traces instead of
as a vital informational tool to visualise process models. This is because
attributes vary enormously between event logs and may have various
meanings even for the same name. It is, therefore, generally not possible
to use such information for any advanced visualisation methods unless
specific expectations for the logs are specified, causing the tool to lose
generality2. In most tools, practitioners can view nodes and edges based
on their frequency or performance, i.e. how often they occur versus how
much time is typically spent on them.
Purely focusing on application logs provides us with some stability that
any arbitrary event log likely won’t offer, requiring Process Mining tools
to be as general as possible to accommodate any event log following a
standard such as XES or OCEL. However, as an example, practically all
logging frameworks distinguish log events by severity like informational,
warning, error, amongst others. Assuming all logs have such a severity
attached to every event, we can change how process models are visualised.
For severity levels, practically any Process Mining tool would allow
practitioners to filter out specific levels. However, this assumption would
also allow us to distinguish severity levels visually by, for example,
changing the nodes’ shape in the graph or assigning specific colours
or icons to each level. As mentioned, this is impossible in regular tools
because they have no concept of a severity level or what it should look
like.
The following lists information that can typically be extracted from
application logs:
1. Log severity/level: how severe a log message is. Typical values are
Verbose/Trace, Debug, Information, Warning, Error, and Fatal. Often,
logging libraries filter log events below a certain severity by default.
2. Exception: an abnormal event that disrupts the normal flow of
execution. Exception handlers are often added where exceptions
are expected to appear occasionally and are typically logged. Many

4.4 OCEL libraries
In Section 4.1, we have decided that the OCEL standard is the most
appropriate Process Mining format to store structured logs. So far, the
official OCEL website [52] lists two libraries and one tool that supports
the standard. All tools and libraries mentioned in Section 3.3 support
it, though some rely on others for the functionality. Since the standard
is defined for both XML and JSON, any programming language that
can handle either of the file formats can use the standard. Dedicated
OCEL libraries make it easy to work with such files, as they can be
used to convert between the two file formats, provide concrete types for
the various elements of a log, and provide utilities to perform common
tasks that are specific to the standard, such as flattening for an object
type. Furthermore, other tools and libraries can reuse such libraries to
implement higher-level functionality, like discovering a process model.

4.4.4 Command-Line Interface
Since the OCEL standard uses universal file formats that various tools
and platforms can understand, it is reasonable to assume that users need
to convert between the different OCEL formats and merge files. This
is easily possible with our .NET library and most tools and libraries
mentioned in Section 3.3. In structured and unstructured logging, it is
common to start a new log file periodically to reduce individual file sizes.
Anticipating this, we have created a command-line interface to our OCEL
library (available at https://github.com/pm4net/OCEL-CLI), which
users can install as a .NET tool8. This should help practitioners with
insufficient programming knowledge to use our OCEL library. It also
simply saves time.
At the moment, it can read all or selected files in a directory and con-
vert them to one of the three available formats (XML, JSON, LiteDB).
Optionally, the files can be merged into one, which is helpful with the
aforementioned periodic splitting. In this case, an entire directory of log
files can be parsed, combined into a single file, and then imported into
the various Process Mining tools that support OCEL files. We have also
added support for removing duplicate objects using Algorithm 1 and
removing non-existing objects.

4.6 Crafting useful log messages
Extracting log messages is only one part of the equation, arguably the most
critical being the actual generation of logs. They must continually record
the application state and contain all relevant details to allow for detailed
post-mortem analyses. If done incorrectly, they can be significantly harder
to analyse than necessary or even prevent it altogether. Gholamian et al.
[2] define four challenges of writing logging code: Where-, what-, how-,
and whether-to-log. Not placing logging statements at crucial parts of the
code can lead to missing context, and too many logging statements at
insignificant places can lead to an overflow of information. Similarly,
the contents of a log message are critical to its usefulness, with too
little information again leading to missing context and too much of
it to informational overflow that is hard to comprehend as a human
observer.
As Gholamian et al. [2] state, the quality of application logs directly
depends on the developers that add them. Creating useful log messages
is an art in itself, and many factors can contribute to a lack of quality
or quantity of such messages, such as the unfamiliarity with structured
logging or the inconsistency between multiple developers on a project.
Additionally, in the world of structured logging, it is easy to accidentally
create unstructured messages by using string interpolation instead of
message templates22 (see Definition 2.1.3). While the messages will still be
logged, their contents will be unstructured and hard to search or filter
systematically.

5.1 Flattening an Object-Centric Event Log
Object-Centric Process Mining has the great benefit of allowing multiple
case notions for an event log, eliminating the issue of convergence and
divergence [6]. However, it can still be useful to view such event logs from
the perspective of a single case notion, whether it is to narrow the focus
or to use object-centric discovery techniques described in subsequent
sections. Aalst et al. [12] describe a technique to flatten an Object-Centric
Event Log for a specific object type, which we implemented in our library
using our OCEL library from Section 4.4.2 as a base.
Essentially, each event is either kept, discarded, or replicated multiple
times, and all remaining events have one and only one reference to an
object, which is of the type for which the event log is flattened. Only
objects of the correct type are considered when looking at a given event.
If it has no such objects, it is discarded. If it has precisely one object, it is
kept as-is. If it has more than one, we create a copy of the event for each
object. We append a counter to the duplicated event’s identifier to avoid
duplicate event IDs.

One of the very first steps in a Process Mining project is to discover a model
that describes the behaviour of the event log. The most basic type of model
is the Directly-Follows Graphs (see Definition 2.2.4), which is often the
default in many tools and sometimes the only option. DFGs are great for
getting an overview of a process, while more advanced models like Petri
Nets or Process Trees are discovered by more advanced algorithms and
allow for better handling of concurrency, enable conformance checking,
and more.
In Section 3.4, we looked at the Sugiyama framework and various
implementations of it. dot [42] is one such implementation and is generally
the standard in Process Mining research, with nearly all libraries and
tools using it to visualise the discovered models. In fact, as far as we can
tell, all non-commercial tools and libraries that we looked at use dot.
While dot is very versatile and generally produces high-quality graph
layouts, it often fails to create layouts that make it easy to understand
the underlying process, as well as to preserve the users mental map of
the model when changes to it occur [45]. The larger a model gets, the
harder it is to understand the process, especially when many edges are
involved. Figure 6.1 shows an example of such an incomprehensible
model involving only five different object types.
Clearly, this is not good enough to properly visualise large and compli-
cated processes, especially when dealing with Directly-Follows Multi-
graphs where there often are multiple edges between the same nodes,
representing different object types. This problem is, however, not exclu-
sive to Object-Centric Process Mining but concerns all of PM.
Mennens [26, 45] addresses this problem with a novel layouting algorithm
that considers the entire event log instead of only the discovered model,
which might have been discovered with filters applied. This additional
information allows graphs to be created so that the most important
variations are centrally aligned, going from top to bottom and avoiding
back edges where possible. The less important a node is to the process,
the farther it is placed from the backbone1. Furthermore, knowing the
event log beforehand allows the algorithm to pre-compute ranks and
horizontal orders for all nodes. This keeps the relative position of nodes
and edges constant, no matter how the event log is filtered for discovery
purposes, preserving the mental map of the user. Figure 6.2 shows a
comparison of the layouts produced by dot (top) and the novel algorithm
(bottom). It shows two layouts each, where the second layout is the same
graph with one edge removed. This highlights the stability of the layout
that this novel algorithm generates.

6.1 Overview of algorithm
We have vaguely described the algorithm to be based on the entire event
log, allowing the pre-computation of ranks and node orderings, prioritis-
ing the most important nodes. This section gives a more detailed overview
of the steps involved, what they achieve, and our approach to implement-

had to find our own solutions to many problems throughout. Following
this overview, we describe notable details of our implementation.
6.1.1 Global Ranking
A Global Ranking of an event log is defined as an ordered list of sets
of nodes, the union of which contains all nodes in the event log [26].

disallowed (except for self-loops). This ranking aims to preserve the

Generating sequences from variations
All that is required to compute a Global Ranking is a set of variations. A
variation is a unique list of events appearing in a full trace. For example,
the traces ⟨A →B →C ⟩and ⟨A →B →C →D ⟩are different variations.
Direct repetitions are, however, removed as they do not contribute to the
layout of the graph [26]. Each variation defines a frequency, i.e. how many
traces with the same sequence of events are in the event log.
Mennens [26, 45] further defines a sequence as a list of nodes and edges,
in turns. Converting a trace to a sequence is straightforward, as it only
involves adding edges between all connections in a variation. E.g., ⟨A →
B →C ⟩becomes ⟨A →(A, B) →B →(B, C) →C ⟩.
These sequences are then ordered by their importance (a combination
of sequence length and frequency of edges, as seen in Figure 6.3). The

Processing sequences
Sequences are inserted into the Global Ranking one by one. While [26,
45] states that only sequences with new nodes and edges are added, it
is not explicitly stated whether this includes an entire sequence or only
parts of it. By experimenting with different approaches, we concluded
that sequences must be evaluated recursively until no elements are left.
We recursively check whether each sequence element is already in the
Global Ranking or has already been seen in the same sequence (for
example, if it loops back to a previous node). If we encounter an element
that has been seen before, the sequence up to that point is added to
the Global Ranking. Afterwards, we process the remaining sequence
elements similarly until none are left.
To illustrate this, consider the sequence ⟨A →(A, B) →B →(B, C) →C ⟩,
and a Global Ranking where the nodes 𝐵and 𝐶are already added, with
an edge (C, B). We check each element until we reach the third element,
node B. Since this node is already in the Global Ranking, we only add the
first two elements of the sequence, ⟨A →(A, B) ⟩. Now, we process the
remaining elements on the updated ranking until no elements are left.
Node B is skipped because it already exists, and we discover that the edge
between B and C is new (only the reverse exists so far). Because node C
already exists, we only add the sequence ⟨(B, C) ⟩. Finally, the sequence
contains no more elements, and the next sequence can be processed.
Adding sequences to the Global Ranking
Mennens [26, 45] defines six types of sequences that may be added via
the previously described process2. Each type is handled differently, and
nodes may be shifted after being added to make space for new nodes.
We do not go into details about this here, as it is explained in detail in
the original work.
Application of Global Ranking
The Global Ranking is used for further pre-computational steps that are
described next. Mennens [26, 45] describes that a discovered model may
contain edges that are not part of the Global Ranking due to filtering of
single activities. This can cause a problem when both nodes of such an
edge are placed on the same rank. In such a case, an additional rank is
added to which the target node is shifted.
We must have the discovered model available to handle this rare edge
case. As the Global Ranking is only the first pre-computational step,
further steps must be computed each time a new layout is computed. For
performance reasons, users might wish to cache all pre-computations
if it is expected that small changes to the model are often submitted
for re-drawing. Therefore, users should be allowed to skip this step to

The first step of computing a Global Order based on a Global Ranking

do not already exist. The index of the currently processed sequence of
the skeleton is used as the discovery index, and we can use it to check
whether a node on the same rank already exists with the same discovery
index. This may be the case when virtual nodes are placed before the
real node. In this case, the virtual node is replaced, and all referenced
edges are updated to point to the newly added real node. This has the
effect that some long edges that are part of the skeleton may end up as
unconstrained due to a real node being placed between the connecting
nodes.

add the actual edge. This is straightforward if the nodes are on adjacent
ranks. If they are further away, we have to add virtual nodes in between
the ranks and connect the two nodes over them. If existing real nodes
have the same discovery index, we connect to them instead of creating
a virtual node. This must be done because all nodes on the same rank
must have distinct discovery indices. It is also important to note that a
virtual node always belongs to one and only one edge. At this point, we
now have an unbalanced Global Order (see Figure 6.5), just like Mennens

Balancing the Global Order
Mennens [26, 45] states three requirements for the final Global Order
that was shown in Figure 6.4: connectedness, minimal crossings, and
balance.
Connectedness describes that nodes that connect each other tightly
should be placed close to each other. Theoretically, this creates shorter
edges and avoids edge crossings [26, 45]. The described approach is
first to find the node sequences that are part of the backbone7, and
then measure how connected each node of the graph is to the backbone
sequences. Nodes on the same rank are sorted by connectedness value,
with backbone nodes fixed in the leftmost position. Mennens [26, 45]
does not specify how to order nodes with the same connectedness value.
We found that, in practice, many nodes will have a value of zero due to
them not being connected to the backbone at all. To still achieve a good
default ordering, we added the discovery index as a secondary sort key,
placing nodes with lower discovery indices closer to the backbone. In
some of our test cases, this avoided several edge crossings where virtual
nodes were connecting nodes on the same two ranks, receiving the same
connectedness value, and being ordered arbitrarily.
After establishing a sort order based on connectedness, connected compo-
nents may now be moved to the left side of the graph to visually balance
it around the backbone. Mennens [26, 45] describes a method where
connected components are found by creating a copy of the Global Order
where the backbone nodes are removed. Isolated clusters of nodes may
then be moved to the left side without creating additional edge crossings
if it improves the overall balance of the graph, i.e. the ratio of nodes on
the left versus on the right is closer to the ideal value of 1 than before.
Once again, Mennens [26, 45] provides no details about possible im-
plementations. In our implementation, we always place the nodes to
be shifted as close to the backbone as possible so as not to leave any
gaps. When nodes are already on the left side and on the same rank,
we compare the original sort orders produced by the connectedness
sort. This allows us to retain the connectedness order when moving
components, as it might well be that components farther away from the
backbone are moved first.

vertical and horizontal ordering constraints for all nodes and all sequence
edges and routes edges that cross multiple ranks through virtual nodes.
Even though minimal crossings are stated as a goal for this graph,
no dedicated crossing minimisation is used in this step. Instead, the
connectedness sort eliminates most but not all of them. Importantly,
this global order only contains sequence edges, i.e. edges that are part
of the skeleton. When the Global Order is applied to an actual model,
non-sequence edges will likely create additional crossings. The next step
includes a dedicated crossing minimisation step to avoid this as much as
possible.
Straightening long edges
In the next section, we describe how to compute actual graph layouts using
these Global Orders. In this process, we found it extremely challenging
to generate layouts where long edges were vertically aligned. Mennens
[26, 45] describe the approach they used for non-sequence edges, i.e.
edges that are not part of the Global Order, but not for constrained ones.
In all their examples, all longer edges always have straight edges. As
the Global Order is constructed without any gaps, our implementation
naively replicates these relative positions for the actual layout, resulting
in zigzag-like edges (see Figure A.1). We initially attempted to straighten
edges as a post-processing step after the layout computation. This,
however, did not work in many cases due to crossing edges repeatedly
pushing each other in an attempt to straighten out.
Finally, we found a solution that straightens constrained and uncon-
strained edges in two separate steps. For constrained edges, we already
do this here for the Global Order. The idea is that nodes with the same
discovery index should be vertically aligned. Because a discovery index
corresponds to a sequence of nodes that appear most commonly in this
order in the event log (see Section 6.1.1), aligning them straightens the
most important edges of the graph. We enforce this by grouping nodes
by their discovery index, finding the X position of the outermost node for
each group, and moving all nodes in the group to this X position. Any
other nodes that are blocking this expansion are moved as well. Typically,
this results in a Global Order that is wider and has empty spaces in
between nodes. A comparison of Global Orders with and without this
step can be seen in the Appendix, Figure A.2 & A.3.

6.1.3 Graph Layout
Everything up until this point has been pre-computations, which apply
to any model discovered from the event log. Now, we want to compute an
actual layout for a model using the Global Order constraints. Generally,
a model may be any directed graph where all nodes are also in the
Global Order. In this work, we are mainly interested in Directly-Follows
Multigraphs (see Definition 2.2.5 and Section 5.2), and the implemen-
tation is geared towards it by distinguishing between start, end, and
event nodes. Edges also have fields to distinguish which object type they
belong to. However, it is easily possible to compute a layout for a regular
Directly-Follows Graph by only adding event nodes and leaving the
object types empty8.
Preliminaries
Besides a novel crossing minimisation algorithm for unconstrained edges,
Mennens [26, 45] provides very little information about the computation
of an actual graph layout. They do describe how they assign values in
the range of [−1, 1] to real and virtual nodes on each rank, which are
subsequently used for crossing minimisation. Furthermore, the last two
steps of the Sugiyama framework (see Section 3.4.1) are not covered, but
instead, reference Gansner’s [42] approach that is also used in dot (see
Section 3.4.2).
While the implementation of the Global Ranking and Global Order (see
Section 6.1.1 and Section 6.1.2) was very strongly based on Mennens
[26, 45], we have developed our own approach to the construction of
an actual graph layout, including node positioning and spline drawing.
We did this primarily due to the lack of details about these steps in [26,
45]9. However, the crossing minimisation discussed later in this section
is based on Mennens [26, 45].
Constructing a graph layout
The first step of going from a Directly-Follows Multigraph (DFM) and
Global Order to a graph layout is adding all DFM nodes and edges.
For nodes, the DFM can not contain any not also in the Global Order.
Therefore, we can directly add the real nodes in the Global Order along
with their relative position and filter those missing from the DFM.
Real nodes must have an equivalent in the DFM. For virtual nodes, we
check whether the associated edge is present in the DFM and discard it
otherwise.
For edges, we can add all edges from the DFM. We have also added
an option to merge edges with identical origin and destination. This
may occur when different object types add the same edge. Edges in our
implementation have a property for information on multiple types. This
allows users of the graph layout to visually highlight that multiple object
types use an edge. This reduces the overall number of edges needed to
display the model, as also discussed in the introduction of this chapter.

Straightening unconstrained edges
Initially, the virtual waypoint nodes of unconstrained edges are added to
the left or right side of the originating node. As discussed in Section 6.1.2,
we also want those edges to be vertically aligned. Just like in Mennens [26,
45], there may be multiple such nodes in the same position due to them
not being constrained. The first challenge is to compute distinct positions
for each while maintaining the alignment, and secondly, to order them
to result in as few edge crossings as possible, which we discuss in the
following subsection.
Like before, naively positioning unconstrained virtual nodes results in
zigzag-like edges (see Figure A.1). Our solution, therefore, considers all
nodes that share the initial X position across all graph ranks. We group
them by their connections so that virtual nodes belonging to the same
edge are considered a group. Those groups are ordered by a heuristic
that Mennens [26, 45] suggests, firstly, the edge length and, secondly, the
total edge weight, i.e. how often it occurs in the model. After establishing
an initial order, the nodes are evenly spread out in the available space.
Importantly, this applies to all ranks. For example, in Figure 6.7, the
unconstrained virtual nodes with X position in the range (0, 1) have three
distinct positions despite there being space to distribute them on only
two positions on each rank. This can result in nodes being unnecessarily
close or far away from the relevant node, but in turn, provides completely
straight edges. Furthermore, the positions in this step are relative, and
proper spacing is added when calculating actual node and edge positions
in Section 6.1.4.
Crossing minimisation
Minimising edge crossings is an important step to improve the readability
of the graph. Mennens [26, 45] introduces a new minimisation algorithm
that is based on Gansner’s approach [42], but with the important distinc-
tion that only unconstrained virtual nodes are considered, as the Global
Order already defines constraints for the ordering of real nodes.
The algorithm’s approach is first to define an initial ordering, which we
have done in the preceding parts of this section. The amount of edge
crossings is then recursively attempted to be decreased by two separate
measures and stopped when there are no more significant improvements
in an iteration.
The first measure is wMedian from Gansner [42]: the ranks of the graph are
iterated top-down and bottom-up in alternating order. For each rank, the
nodes on the rank are sorted by the median 𝑥position of connected nodes
from the rank above or below. The idea is that virtual nodes connecting
across ranks are aligned, reducing edge crossings in the process.
We cannot copy the approach because we previously straightened un-
constrained edges. If we did, edges would break out of the alignment
again. Furthermore, the median value is calculated by the 𝑥positions of
connected nodes on adjacent ranks. The measure has no effect since we
already straightened those that belong together.

Removing unnecessary ranks
Another step that improves the layout quality is to remove unnecessary
ranks. Mennens [26, 45] mentions this while talking about the Global
Ranking, and it is here where it can be applied. There might be unnec-
essary ranks when all nodes on a rank that are normally in the Global
Order are not present in the discovered model due to filtering.
In our implementation, we consider a rank unnecessary when all nodes
on it are virtual and vertically aligned. We process the ranks from top to
bottom, and if all nodes are virtual, we attempt to find the corresponding
virtual node on the following rank for each. We remove the rank if and
only if each corresponding node has the same 𝑥position, making the
graph taller than it needs to be.
6.1.4 Node positions
The final puzzle piece is to draw a graph layout to a screen. The previous
steps have given us graph data structures for various purposes, such as
the reusable Global Order and finally a graph layout for an individual
Directly-Follows Multigraph.
All the Process Mining tools we looked at in Section 3.3 display nodes
as rectangles with the activity name inside the node body. Because we
implemented the algorithm as a general-purpose library, we cannot and
do not want to know at this point how large nodes should be, how to wrap
the containing text, and how much space there should be in between
nodes and in between edges, both horizontally and vertically. We leave it
up to the user of the algorithm to set appropriate parameters for their
purpose. Specifically for node sizing and text wrapping, we allow users
to provide custom functions to calculate them on the fly on their target
platform. This is explained in more detail in Section 6.2.3.
We have developed our own approach to compute actual node positions
based on those parameters. To maintain the vertical alignments that we
discussed in Section 6.1.2 & 6.1.3, we process nodes in the order of their X
position in the Graph Layout, starting with the backbone nodes. We use
the user-provided width of nodes to position nodes on the same vertical
line so that their centre points are still aligned despite the different
widths (see Figure 6.8). Even if there exists only a single node on a given
X position in the entire layout, the remaining nodes on all ranks respect
this space in order to keep the alignments intact12.
After determining the proper X positions for all nodes, we can do the
same for Y positions. The approach for this is very similar in that we
group the nodes by their rank, find the tallest node for each, and align
nodes on the rank on the centre line of this tallest node. As before,
user-specified spacing is added in between each rank (see Figure 6.9).

6.2 Implementation
In the previous section, we gave an overview of the Graph Layouting
algorithm by Mennens [26, 45], our approach to each step, and our own
solutions to the challenges we faced while implementing the algorithm.
In this section, we discuss some notable details, like the calculation of
node sizes and text wrapping.
This part of the project ended up being the most complex and time-
consuming due to the algorithm’s complexity and lack of details in
the original work, requiring us to find our own solution to many of the
described concepts, often through several iterations. The algorithm ended
up being more than 1600 lines of F# code, which is a lot considering the
conciseness of the language. The same implementation would likely be
several thousand lines long in other languages [56].

to any other supported languages. Now that we may not publish the
implementation, this is less relevant. However, it can still be useful to

6.2.3 Calculating node sizes and wrapping text
We previously stated that our implementation allows users to specify
custom functions for node size and line wrapping calculations. This
allows for customisation based on the target platform, character set, font,
font size, and other factors that users may have.
For less sophisticated scenarios, we provide a default implementation
where node text is wrapped by the number of characters on a line, given
some maximum, and the width of a node is determined to be the number
of characters on the longest line. The height is determined to be the
number of lines. This may work fine for line wrapping and node width
when monospaced fonts are used. However, such fonts only define a fixed
width, not height. Since the height is set to the number of lines, this only
works well when all characters are equally tall, and when there is no
spacing between lines.
In the case of this project, we use the algorithm in a web application
(see Chapter 7). Because the algorithm is implemented in .NET, we do
not have the browser’s context available to measure text size and wrap
text, which is needed if we want to properly compute the values for the
environment in which graph layouts will be displayed16.
As for the text wrapping, we use the JavaScript library wrap-ansi17 to
wrap the node text into lines with a soft limit of, by default, 20 characters
per line (long words may extend the limit). Given the multi-line text, we
now want to compute the width and height of its bounding box.
The relevant API for measuring text width and height is included in
CanvasRenderingContext2D18. A canvas is an HTML5 element that allows

Based on this, our use case requires us to pre-compute node sizes in the

graph layout. To ensure that the node size and text wrapping information
are available, the app first requests a list of all nodes in the graph, wraps
the text and measures it in the canvas context, returns the information,
and finally requests the layout to be computed. Now, the .NET server has
a list of node sizes and text wrappings, which it uses to construct the
custom function that is finally passed into the layouting algorithm.

7.1.4 Discussion of results
Both use cases in Section 7.1.2 and 7.1.3 let us conclude that Structured
Log Analysis tools like Datadog and Seq are indispensable for day-to-day
troubleshooting and general monitoring of an application’s state. The
capabilities of the search and filters provide great flexibility to view data
precisely as a case requires, filtering out unnecessary information and
focusing on the relevant parts. They are the first and often last stop when
in need of investigating any issue.
On the contrary, those tools are not made to gain a general overview of
an application’s behaviour. Even with the advanced filters, it is hard to
comprehend the typical application flow by looking at lines of text. And
this is where Process Mining tools are supposed to help. As we have
seen, though, current tools are not yet equipped to visualise large Object-
Centric Event Logs pleasingly and cannot visualise the behaviour of a
single trace in the overall flow. The available filters are often restricted to
removing infrequent behaviour, reducing the graph size but not making
it comprehensible. pm4py offers many filtering functions, but none let

7.2 Definition of required features
Having conducted a case study with real-life data and the existing tools,
we can now define the desired feature set of our own application. Given
the strengths and weaknesses of existing tools, we can fill the gap of the
missing functionality while not replicating too many existing features to
avoid building something redundant.
In the following, we list the relevant features that are required to build a
working application, combining features we have seen in both types of
tools as we see fit.
Log intake
Structured Log Analysis tools generally allow data to flow
continuously, while Process Mining tools require a single event log file.
Given that PM models need to be rediscovered whenever the event log
changes10, we want to allow users to manually sync files when needed but
not read them continuously. This file sync should support the import of
multiple files, as it is common for new log files to be created periodically
and placed in the same directory.
Object type management
We have seen that too many insignificant ob-
ject types can clutter a graph tremendously. These object types primarily
exist in our data due to the direct conversion of message template holes to
objects, as discussed in Section 4.2. Both Structured Log Analysis tools
and Object-Centric Process Mining tools allow for the filtering of such
objects, and we will need it as well. Going beyond that, it should also be
possible to convert such object types to attributes before even discovering
a model, as discussed in Section 4.4.2, greatly reducing the number of
objects needed to filter and visualise.
Filtering options
A Log Analysis tool needs to provide plenty of ways
to filter the log data, allowing us to narrow the focus to the relevant
part of the data. Structured Log Analysis tools mainly provide ways of
restricting the time frame and filtering for specific properties and their
values. Consequently, this allows us to, for example, filter for specific log
levels and namespaces. Object-Centric Process Mining tools, on the other
hand, mainly provide filters for narrowing down the overall number of
insignificant paths. We want to combine the best of both worlds in our
application.

large graphs as images.

Now that we have conducted a case study and defined which features we

We start by discussing the technologies that we chose to build it with and
why they are a good fit, and then go through the feature definitions from

7.3.1 Choosing the right technologies
We implemented everything leading up to this application for .NET14.
Hence, our application also needs to be written for .NET to be able to use
these libraries.
We then had to decide between a Desktop or Web application and
decided to go with the Web due to the ability to host the application as
an externally accessible web service, and the extensive ecosystem of web
technologies and the choices that come with it. If we do wish to use the
application as a Desktop application, that is also possible nowadays with
the advent of Electron15.
While there are also several choices when choosing to build web applica-
tions on .NET, ASP.NET is the go-to .NET web framework for server-side
rendered (SSR) applications. As we will potentially deal with very large
data sets and have to perform computationally expensive tasks on them,
SSR is generally better suited to avoid outsourcing heavy tasks to the
client’s browser16, and allows us to cache frequently needed resources
efficiently.
A middle ground between the performance of a server-side application
and the extensive ecosystem of libraries that can be utilised in client-side
applications is the usage of small JavaScript components that can be
placed on server-side rendered pages that then take care of rendering
the data that the server provided. Any JavaScript framework can be used
for this, and we came across a template application [59] that combines
ASP.NET Razor Pages [60] and Svelte, a JavaScript framework for building
interactive user interfaces. One can choose between several component
frameworks, such as Bootstrap, Material, and Carbon. We decided to use
IBM’s Carbon Design System17 for this project.

Frequency
One of the most important filters in Process Mining tools is
one to reduce the model’s complexity, making so-called spaghetti models
readable. How this is achieved depends on the discovery algorithm used.
OC-PM [15] and ProM’s [35] OCEL plugin, for example, use the Fuzzy
miner [61]. This miner is particularly well-suited to reduce noise and is
often controlled by two sliders, ranging from 0 to 100%: one to remove
infrequent activities and one to remove infrequent edges.
We described our process discovery implementation in Section 5.2, in
which we follow a discovery approach that introduces three filters:
minimum events, occurrences, and successions. Similar to the Fuzzy
Miner’s sliders, these parameters allow us to filter infrequent activities
and edges. They are, however, not relative to the data in terms of a
percentage value but, instead, an absolute threshold that has to be
adjusted for each event log. In our application, we provide numerical
inputs for all three values.
Timeframe
Another important filtering method is to define the time-
frame of the included events. This is crucial in Structured Log Analysis
tools, which are often used to inspect failures at specific points in time.
Many Process Mining tools also support this feature in some form. For
our own implementation, we described in Section 5.2 that we drew
inspiration from a non-object-centric tool called Disco [51]. In addition
to two date-time selectors to define the range, our application also lets
users select one of the five options described in Section 5.2.
Log levels
In some cases, application logs contain excessive amounts
of unimportant information. As they are classified by a severity level (see
Section 4.2), they can easily be filtered for in Structured Log Analysis
tools and are often even configured to be discarded below a certain
severity by the logging framework itself. This is, of course, not available
in Process Mining tools as they have no concept of a log level. Because
we assume in this work that the input files always have such a property,
we can easily filter for it. To still support regular OCEL files without this
assumption, we provide an additional option Unknown that includes all
events that do not have such a property defined.
Object types
Probably the most important filter of Object-Centric
Process Mining is the one to exclude certain object types. We have already
provided users with the ability to convert unnecessary object types to
attributes in Section 7.3.3, but not all analyses require us to consider
all remaining object types. Only including a few relevant ones at a
time greatly reduces the clutter and boosts the comprehensibility of
the resulting graph. To enable this, we include a simple checkbox filter
for each object type. Simultaneously, we generate a unique colour for
each object type that is then used in the graph. The colour is seemingly
random but deterministic by the name of the type20. This ensures that
reloading the application still results in the same colours used.

7.3.6 Graph rendering
The rendering of the discovered process graph is the most important part
of our application. Its quality alone decides whether it is useful for the
purpose that we have set out for or whether it creates an incomprehensible
spaghetti graph as so many other Process Mining tools do when faced
with large models.
DOT
We want to offer as much flexibility as possible and are therefore
offering multiple rendering methods. Firstly, we want to allow for the
models to be viewed with the industry standard tool, dot [41]. In Section
5.2.1, we have already added support for the generation of dot specification
strings to our Process Mining library that are very similar in style to
those generated by pm4py [34]. Graphviz, the program in which dot is
included, is not natively available for .NET or JavaScript. Fortunately, the
WebAssembly [62] instruction format is available in all major browsers
now, enabling us to run a WebAssembly version of Graphviz directly
in the browser. One such version is hpcc-js-wasm22. Using this, we can
pass in our dot string that is then converted to an SVG string, which we
can render directly in the browser. Besides the shortcomings of the dot
layout, as discussed in Section 3.4 and Chapter 6, static SVG files also
mean that we cannot add any user interaction to the graphs besides
panning and zooming. It is, however, useful to have a quick comparison
available directly within the application.
MSAGL
Another layouting algorithm based on Sugiyama framework,
just as dot is, is Microsoft’s Automatic Graph Layout library [43, 44], as
briefly discussed in Section 3.4.3. Just as with dot, we add this algorithm
to our application for comparison’s sake. It is written for .NET, meaning
that we can directly consume it without relying on WebAssembly ports.
As with dot, it produces SVG strings that we can render directly within
the app. An example of a layout discovered by MSAGL can be seen
in Figure 7.10. The library is, unfortunately, not very intuitive to use,
and there seems to be no documentation available. It also has the major
disadvantage that the size and positioning of labels must be defined
manually, as opposed to dot, where the algorithm does this for us. As

Choosing a JavaScript graphing library
We have implemented Men-
nens’ state-of-the-art graph layouting algorithm [26] in Chapter 6. The
resulting layout is completely framework-agnostic, as the given positions
and sizes are of no particular unit. There is a seemingly infinite number
of JavaScript graphing libraries, and we have to choose one for the job.
We have looked at dozens of them and assessed whether they provide
the features we require23. There were finally two options to choose from:
D3.js [63] and Cytoscape.js [64]. D3.js is the de-facto standard in web-based
visualisations due to its sheer flexibility. Cytoscape, on the other hand,
originated in the bio-informatics field [65] and was primarily used for
the visualisation of molecular networks. With the introduction of the
JavaScript library, however, it can be used for practically any purpose. We
have experimented with both D3.js and Cytoscape.js and ultimately chose
the latter due to the higher-level interface24 that allows us to generate
working graphs fairly quickly.
Drawing nodes with Cytoscape.js
A custom graph layout, as discov-
ered by the algorithm discussed in Chapter 6, consists simply of a list of
nodes and edges. A node stores its ID, wrapped display text, node type
(start, end, or event), position, size, and optional arbitrary information
from the event log. An edge stores the IDs of the source and target node,
the waypoints in between (if any), and its total weight and weight for all
object types that share the edge. Each object type on an edge may also
store optional arbitrary information.

custom layout25. We have to first define a CSS-like stylesheet for each
type of node and for edges. When adding the nodes from the layout, we
overwrite some of the CSS properties to match the data. This includes the
position, size, shape, and background colour. Figure 6.10 in Section 6.1.4
shows some of these styles. Start and end nodes are coloured with the

seemingly does not use Gansner’s [41] approach in dot but only uses

able to compute the control points, which the spline itself will not pass
through. Intuitively, it would be much easier to use a type of spline
that does pass through its control points since we already have properly
positioned waypoints. Such a spline is called a Catmull-Rom spline [68]
and is defined by four control points. The first and last points are the
"handles", and the spline is only drawn from the second to the third
point. We could use such splines to draw smooth curves to and from
each waypoint, possibly setting the handles for each segment to be the
previous and next waypoint or node29.

not support Catmull-Rom splines but only quadratic Bézier splines.
Fortunately, simple approximation methods exist to convert Catmull-
Rom splines to cubic Bézier splines [69], and then to quadratic Bézier
splines30 [70]. Given the work required to implement these conversions,
we have opted to leave this for future work and render our edges as
straight-line segments for the time being.
Default Cytoscape.js layout algorithms
Cytoscape.js has several built-in
layouting algorithms and several more community-made algorithms. As
we have discussed in Chapter 6, standard algorithms like dot [41] are not
a good fit for the large process graphs we are trying to render. This is not
changed by the Cytoscape.js layouts, but as with dot before, we decided to
include it for comparison’s sake.
Applicable built-in algorithms are breadthfirst31 and cose32. The former
traverses the graph using breadth-first search and is best suited to top-
down graphs. The latter uses a physics simulation instead. Community
algorithms include, for example, dagre [71]. It is made to lay out Directed
Acyclic Graphs (DAG), which is not applicable to us as our graphs may
contain directed cycles. We have added examples renderings for the cose
and breadthfirst layouts in Appendix Chapter B, Figure B.1 and Figure B.2.
Both algorithms produce layouts that are difficult to follow and overlap
significantly due to the node size not being considered.

Namespace grouping
We have discussed the importance of considering namespaces in Log
Analysis tools in Section 7.2. We have already implemented a names-

namespace grouping.
In short, the idea is to confine nodes to certain areas of the graph
recursively for each part of the namespace. We have experimented with
this idea in Section 5.2.1 by placing nodes in sub-graphs in our dot [41]
renderer. Having experimented with the case study data instead of the
small example in Figure 5.2, it becomes clear that, in most cases, there
is too much interaction between the various parts of an application for
this to make sense. On top of that, the layouting algorithm we have
implemented in Chapter 6 has no concept of sub-graphs, so it would
require major modifications to it in order to allow this. Alternatively,
one could use the algorithm as-is to create a layout for only the high-
level namespaces33 and recursively layout lower namespace levels within
those. This approach would, however, introduce more complications than
we could likely handle. Given that we found that this type of grouping
would likely only overcomplicate the graph layout instead of improving
it, we do not consider this idea further.

in the visualisation without modifying the layout. One such way is

relationships between elements that are not "necessarily adjacent in the

interaction between events and not their namespaces. This approach has
been implemented in a Cytoscape.js plugin34 that we can integrate into our
application. We have done this only for the top-level namespace of each

most cases, the sets are not rendered for no apparent reason. We have

We stated in Research Question 3 that we want
to visually replay a trace, not just view it as a list. Thus, we added a
replay control (at the bottom in Figure 7.8) to the application that can
automatically go through the current trace and highlight the flow visually.

the right so as not to lose orientation of the progress.
Animating the trace proved to be very difficult with Cytoscape.js. In fact,
we could not get the animation system to work how we wanted it at all.
We instead resorted to using CSS transitions39, which was easier to get
working but still lacks flexibility. Essentially, we can define a different
set of styles for nodes and edges that would then be eased in and out
smoothly. For nodes, we darkened the background colour, made the node
border thicker, and coloured it brightly, which can be seen in Figure 7.15.

and manually added and removed with each step, meaning that we had
to define custom styles for each log level due to different background
levels and the opposite to remove the styles again. It seems not to be
possible to compute such properties at runtime, making the system
difficult to work with.
As for edges, we wanted to make it very visible from and to where events
were flowing. We were inspired by Disco [51], which animates transitions
by moving colourful circles along the edges. This is, unfortunately,

8.1.2 Serilog sink and enricher
In Section 4.5.1, we have developed a Serilog sink and enricher that uses
our OCEL library to write structured log events directly to the desired
format. While the results in Figure 8.1 tell us that the LiteDB serialisation
is significantly slower than JSON and XML, this does not automatically
mean that it is the worst format for logging. In Section 4.4.3, we discussed
that both JSON and XML need to be fully parsed before the file can be
meaningfully modified, and any changes typically require a full rewrite
of the entire file to disk. To partially mitigate this obvious bottleneck, we
keep OCEL logs in memory to avoid the need for constant deserialisation
for each new batch of log events. However, we still have to overwrite the
entire file contents with each new batch, which is expensive for large files.
A batch consists of a maximum of 1,000 thousand events by default.
The strength of LiteDB is, instead, the ability to append new events and
objects to the file without the need to parse or rewrite the entire content.
In the following, we benchmark the three Serilog sinks when logging to
real files continuously and also compare the size of the resulting files.

8.1.5 Structured Log Analysis Tool
During the case study in Section 7.1, we tested how our application
handles very large files for the first time. Web applications with lots of
user activity can quickly accumulate hundreds of thousands of events
and millions of objects over the span of just a few days or even less. Due
to this, we discovered several major bottlenecks in our implementations,
but also general issues when dealing with very large event logs.
The bottlenecks were present in several features, such as the merging of
duplicate objects (Algorithm 1), the conversion of objects to attributes
(Algorithm 2), as well as the discovery of Directly-Follows Multigraphs
(Section 5.2 & 8.1.3). Those bottlenecks were mostly due to long lookup
times when we stored very large collections in simple lists. We fixed
those issues by mostly using dictionaries for fast look-ups and reducing
the overall number of required loops. This resulted in the functionality
going from being entirely unusable for our case study data to only taking
a few seconds at most5. As we discussed in Section 8.1.4, our layouting
implementation suffers from several such bottlenecks itself, and fixing
them would require a significant amount of time that we do not have
and potentially a rewrite of several steps.
But even when not using the custom layouting algorithm, we faced
excessive performance problems when rendering our case study data
with dot [41]. The graph specification is easily generated by the server,
but rendering it is much more resource-intensive, especially in the
browser (see Section 7.3.6). Loading our entire case study and applying
no filters, the application froze the browser after several minutes and
ultimately failed to render a graph. Based on this, we conclude that
one should always apply filtering before rendering graphs of such size.
This is reasonable because the resulting graphs are too large to be
comprehensible, even if rendered with the custom layouting algorithm
that is optimised to make processes easy to follow.

8.2 Qualitative evaluation
We implemented plenty of good and well-performing solutions over
the course of this project. Not all of them, however, are of the quality
we initially aimed to reach. Both are evaluated in the following, where
relevant comments can be made.
8.2.1 OCEL library
In Section 4.4.2, we have implemented a new .NET library to support
the OCEL [10] standard. This was successful, and we could read existing
files, write new ones that were correctly validated by the official schemas,
and convert between the two defined formats and one additional that we
introduced.
We later conducted a case study with existing Process Mining tools and
OCEL files generated with the help of this new library in Section 7.1. Over
the course of this case study, three of the four tools we could run failed to
load our files. In one case, we could pinpoint the failure to correctly parse
timezones in ISO 8601 date strings despite the OCEL standard clearly
allowing non-UTC timezones. In the other two cases, we were not given
any error messages, leaving us to guess the cause.
Given that we only received a proper error message in one of the three
tools, which we could blame on the tool and not the files, we believe that
our library is generating OCEL files as intended by the standard.
8.2.2 Graph layouting
We have spent a significant part of this project implementing Mennens’
[45] novel layouting algorithm (see Chapter 6). Besides the performance
issues discussed in Section 8.1.4, we have had to devise solutions to many
of the described parts of the algorithm throughout the implementation.
In this section, we discuss the quality of the resulting layouts.
Conducting a user survey when introducing a new layouting algorithm
is generally advisable. This can determine whether users find it more or
less intuitive to use than an existing standard such as dot [41]. Mennens
[45] has already conducted such a study and has found that the layouts
improve upon dot in nearly all aspects. As we tried to replicate the
algorithm, we refer to this user study to validate the overall quality of
the layouts.
In the context of OC-PM, Mennens’ [26, 45] algorithm may create un-
intentionally bad layouts when many object types are involved. Since
the vertical and horizontal ordering constraints are pre-computed for all
possible Directly-Follows Multigraphs, filtering out most object types
sometimes creates unnecessarily complicated layouts because we have to
respect the constraints. We implemented a feature in our Log Analysis
application that lets users convert unnecessary object types to attributes
before rendering a graph (see Section 7.3.3). We recommend not to skip
this step due to the resulting layout limitations, even when filtering out
nearly all object types.
==PAGE:95==
8.2 Qualitative evaluation
83
In some cases, viewing the event log from the perspective of only one
object type could make sense, as it is done in regular Process Mining.
To achieve this with the custom layouting implementation, one must
first convert all object types except the relevant one to attributes before
rendering the graph. When using dot [41] or MSAGL [43], it is sufficient
to set the object type filter without requiring conversions, as it does not
pre-compute ordering constraints based on the entire event log.
Even then, bad layouts can still appear, as we have seen in Section 7.4.2
where we also applied a namespace filter. Given that the discovered pro-
cess models can be enormously different depending on the selected object
types and namespaces, it could be useful to restrict pre-computations to
this context. Mennens’ [45] algorithm was not made with object-centric
graphs or even namespaces in mind, and it is typically expected that
the graph largely stays the same, no matter which filters are applied.
Restricting this would likely produce much better layouts in many cases
but would also change the mental map [45] when changing fundamental
filters. While we did not implement this in our prototype, it is a trade-off
that could be well worth it in this context.
8.2.3 Graph rendering
Focusing on the rendering of the graph layouts, we are not entirely
satisfied with how the implementation turned out to be. This is mainly
due to our choice of Cytoscape.js [64] as a graphing library. We mentioned
some of these issues in Section 7.3, and they are all are contributing to a
degraded quality of the rendered graphs:
1. Edges are implemented as straight-line segments instead of smooth
Bézier or Catmull-Rom splines.
2. Self-edges are often not rendered for no apparent reason.
3. The sizing of nodes can not be easily adjusted to show actual
rendered messages instead of message templates.
4. Namespaces are not properly highlighted in the graph due to an
issue with the bubble sets extension.
5. The export of graphs to both vector and raster images is missing
the node labels due to the usage of HTML labels.
6. Edge labels are missing and are difficult to add for custom edges.
7. Edge colouring for multi-edges is inconsistent in style.
8. Animations are limited and cannot properly show transitions on
edges.
We decided to use Cytoscape.js in Section 7.3.6 as it seems to be the most
flexible yet easy-to-use graphing library at the moment. Similar libraries
were either inaccessible due to commercial licenses or did not allow us
enough freedom to use custom node positions and edge routing, styling
of nodes and edges, and animations. The alternative was discussed as
D3.js [63], which is without doubt capable of allowing us to implement
all such features. The reason why we chose Cytoscape.js anyway is simply
due to D3.js being too low-level to allow us to put together a prototype
quickly. As such, our choice for this project was not wrong, and we
would likely not have been able to implement many of the features we
did despite some being half-baked. If we were to start over, though, we

8.2.4 Logging best practices
In Section 4.6, we briefly discussed the importance of crafting useful log
messages. With the knowledge of the case study in Section 7.1, we can
now come back to this topic in more detail. We found the following two
topics to be most relevant.
Consistency
Given that we created a mapping that converts holes (see Definition 2.1.2)
in message templates directly to object types (see Section 4.2), it is vitally
important to be consistent about the naming. Even small changes will
classify related log messages as different object types, reducing the data
quality. We saw an example of this in Section 7.4.1, where two versions
of the order number parameter were present.
It is unlikely for many developers to pay special attention to this, but
we believe they should. It is not only relevant for our approach, but
consistent naming also pays off in the Structured Log Analysis tools from
Section 3.2 as one can easily filter for specific parameters.
Of course, only a minority of object types are relevant for the application
flow, as we saw in Section 7.4.1. For such important parameters, we
suggest relieving developers of the burden of getting the naming right
every time and implementing them as enrichers whenever possible. Our
case study application did this, for example, for order numbers. Not
only will this remove any inconsistency, but it will also include more log
messages in general where the given parameter is not the most important
but still contributes to the flow of the application.
Logging all business decisions
It is often the case that only errors are logged. The error message or
exception might be sufficient in some cases to locate the issue, but it is
also important to be able to backtrace how we arrived at the erroneous
state. In complex features with lots of business rules, the error might
very well be caused by a decision further back that is not an error in itself
but implicitly caused it. We, therefore, recommend logging whenever a
business decision is taken in the code, and we have seen the importance
of this in Section 7.1.3, where we could trace an error back to an update
to the customer’s cart. If this action were not logged, identifying this as
the issue would be significantly more difficult.

9.1 Research Question 1
Is it possible to convert structured application logs that use the message
templates grammar to object-centric event logs? Which object-centric
format is best suited for this, and which storage method is most
performant for continually writing large amounts of logs?
Yes. The OCEL standard is best suited, and we recommend our custom
database format for continually writing large amounts of logs.
We have evaluated available object-centric event log formats in Section 4.1
and identified the OCEL standard [10] as most appropriate. We have then
defined a novel mapping from message templates to OCEL in Section 4.2,
preserving the full structure of a message template. The mapping has a
practical limitation, for which we suggest a post-processing solution with
Algorithm 1. To prove the feasibility of this mapping and to collect testing
data, we have implemented a logging adapter for a popular logging
library in Section 4.5, based on our own implementation of the OCEL
standard in Section 4.4.2.
The OCEL standard is defined for two storage methods: JSON and XML.
We have implemented both and added support for a third database-based
method in Section 4.4.3. In order to address the sub-question of which of
the three is most suited for production usage, we created benchmarks in
Section 8.1.2 that measure the time and storage required to write large
numbers of events to each format. The results show that JSON and XML
scale similarly and require less disk space than the database format, and
are also faster at writing a few thousand events or less to a file.
For large amounts of events, however, the database format performs
several orders of magnitude better than JSON and XML, making them
prohibitively expensive to use for large-scale logging without rolling files.
We conclude that JSON and XML should still be used to share OCEL
files due to them being more universally accepted, using less space, and
being faster to write to in one go. For production logging, however, we
conclude that the database format should be used for scalability. It is
easy to merge and convert such files for general usage with our CLI tool
from Section 4.4.4.

9.2 Research Question 2
Is it possible to discover a runtime behaviour model of an application
from its structured log files by using Process Mining discovery tech-
niques for Object-Centric Event Logs? Can this model be visualised
to show this behaviour from various object perspectives?
Yes. We can discover a Directly-Follows Multigraph from the OCEL
files. They can be visualised using the industry standard dot, MSAGL,
or our implementation of a process-specific layouting algorithm. We
have created a novel application to automate all of this.
We have chosen DFMs over other models like Object-Centric Petri Nets
as it is the simplest and is most commonly used by Process Mining
tools as the default way to view process models, both object-centric and
not (see Section 3.3). Other implementations of the discovery algorithm
exist, but we decided to create a custom implementation in Chapter
5, mainly for two reasons: we needed an implementation for the .NET
platform for which we would later implement a Log Analysis tool, and
we wanted to include custom filters that other implementations do not
allow for without modifications. Finally, we have proven the feasibility
of our implementation in Section 8.1.3 by comparing the runtime to the
popular pm4py [34] library, in which our implementation was significantly
faster.
To facilitate the discovery, we developed a user-facing application in
Chapter 7 that is capable of importing multiple OCEL files, converting
irrelevant objects to attributes, filtering an event log in a multitude of
ways that are typical to both Log Analysis and Process Mining, and
finally visualising the discovered model using several techniques. We
have based this feature set on a case study of existing Log Analysis and
Process Mining tools, using a real-life data set, in which we discovered
the strengths and weaknesses of existing software in both categories,
creating a combination of the best of both worlds as a result.
As for the visualisation, we have looked at techniques used in existing
tools in Section 3.3 and some of the underlying graph layouting algorithms
in Section 3.4. Nearly all tools that we looked at used dot [41] as the
underlying algorithm. We quickly realised, however, that dot is not a
good fit for rendering very large graphs comprehensibly. Given that the
Research Questions focus strongly on the visualisation aspect, this was
a dealbreaker for the project’s vision. Fortunately, an alternative graph
layouting algorithm that is dedicated to Process Mining specifically has
recently been proposed [26, 45]. We have subsequently implemented
this algorithm in Chapter 6 and are using it in our custom application in
Chapter 7. As it turned out, though, this algorithm is not the silver bullet
to solve all layouting issues due to the fact that a patent was filed on it, the
implementation being closed-source, and many important details that are
vital to re-implementing it having been omitted or not explained in detail
in [45]. In the end, we have managed to implement a version of it with
many of our own solutions to parts of the algorithm. Upon validating the
performance of our implementation, we have unfortunately realised that
it performs horribly slow compared to industry standards and struggles
with scalability with large models.

9.3 Research Question 3
Is it possible to diagnose failures in structured logs by replaying the
failing trace in the discovered model? Can this be visualised on top of
the discovered model, showing the actual trace values instead of log
message templates?
Yes. We can visualise the flow of a specific trace in relation to the rest
of the discovered model. Actual log messages can be displayed instead
of log message templates to allow for detailed troubleshooting.
We built a replay feature into our custom application in Section 7.3.9. This
feature lets users select a specific object type, which highlights only the
parts of the graph that contribute to it. Furthermore, users can then select
a specific trace and view all of its events in detail. A replay control is
made available where the trace can be visually replayed by highlighting
the relevant nodes and edges in the graph.
Being able to visually identify where a trace deviates from the regular
control flow of an application would be invaluable for many troubleshoot-
ing tasks, as we have discussed in Section 7.4.2. Unfortunately, we faced
severe issues during the implementation, including challenges with the
custom graph layouting algorithm, the graph rendering approach, and
the animations, as we evaluated in Section 8.2.2 and 8.2.3. We have
proposed possible solutions to mitigate the issues in the aforementioned
sections and believe its potential is still substantial.
9.4 Research Question 4
What guidelines should applications follow regarding their logging
strategy (e.g. where-, what-, how-, whether-to-log) in order to pro-
duce structured log files that can be used for the two Log Analysis
techniques?
Logging statements should be placed wherever business decisions are
taken and not only when errors occur. Developers should pay close
attention to the naming of parameters and preferably use automated
enrichers for consistency with domain-critical parameters.

Conclusion 10
10.1 Future work . . . . . . . . . 89
This project set out to fuse recent advancements in both logging and
Process Mining, namely those of structured logging and Object-Centric
Process Mining (OC-PM). Either topic is interesting in itself, but both are
required for our novel idea to work: converting structured log messages
to Object-Centric Event Logs.
We have investigated the topic thoroughly via four Research Questions,
to which we gave detailed answers in Chapter 9. In short, we could
answer all of them positively, showing that the fusion of Structured Log
Analysis and OC-PM is a valuable endeavour. It opens up the possibility
of visually analysing and troubleshooting application logs, which was
previously not easily possible.
Throughout this thesis, we have touched upon several important aspects
to make this fusion possible. We defined a novel mapping, implemented
several general-use libraries, introduced log-specific filters that stan-
dard Process Mining tools do not support, discussed and implemented
graph layouting algorithms and visualisation techniques, and finally
implemented a new Structured Log Analysis tool that adds important
log-specific features which allow application logs to be analysed much
more effectively, as opposed to with existing OC-PM tools.
All of this led to the creation of a full Process Mining pipeline, starting
from the data collection and ending with the analysis of it. While later
contributions are lacking in quality, such as the implementation of the
graph layouting algorithm and visualisation component, everything
leading up to it creates a solid base upon which improvements can be
easily added. The encountered issues are unrelated to the core idea of
this project and should be solvable in future work. As such, the core idea
of this work is still valid although there is much room to improve. We
hope other researchers take up the idea of fusing the two areas in the
future. Given that the visualisation aspects were lacking the most, vastly
better tools can be implemented in the future based on our ideas and
possibly partly our implementations.
10.1 Future work
We are generally satisfied with our contributions and have optimised
them to be versatile, efficient, and easy to re-use for practitioners and
researchers alike. As we concluded, though, we are not satisfied with the
results of our graph layouting and visualisation aspects, and it is mainly
here where we would improve in further work.
Despite this being a Process Mining thesis, we have actually only used
basic techniques that were defined early on in the development of Process
Mining and once again with the invention of Object-Centric Process
Mining, namely the implementation of a Directly-Follows Multigraphs
discovery algorithm in Chapter 5. More recent advancements in the field
introduce significantly more advanced techniques that could possibly
==PAGE:102==

also provide new ways of visualising and analysing our structured
application logs.
Another interesting area of Process Mining that could be relevant here
is that of streaming PM. We briefly mentioned this when defining the
log intake feature of our application in Section 7.2. It could, in theory,
improve the handling of very large amounts of log messages by not
having to reconsider every single event during discovery operations. This
is especially relevant in the context of logging, as it is not uncommon for
there to be a near-prohibitive amount of events to process. This became
clear during our case study in Section 7.1, in which many existing OC-PM
tools could not handle the volume of our data, which was large but not
even close to how large real production logs would be.
A significant part of this thesis has focused on the topic of Graph
Layouting despite it not being strictly relevant to our initial Research
Questions. Industry-standard algorithms such as dot [41] could not
visualise our complex models nicely. Simultaneously, we discovered a
recently proposed layouting algorithm specifically for Process Mining
[26], which perfectly fits our use case. Due to various reasons discussed
previously, our implementation of it did not live up to our expectations.
Therefore, spending more time on improving the implementation for
efficiency and quality would be beneficial. Given that a patent has been
filed on the algorithm (see Chapter 6), there might not be much benefit
since any implementations may not be shared. It could, therefore, be
relevant to conduct further research into Graph Layouting and develop a
new algorithm that is freely available for everyone to use and improve.
Finally, we also faced significant issues with the visualisation aspect of
our application. We evaluated this in Section 8.2.3 and suggested that
D3.js might be a better fit instead of Cytoscape.js if we were to start over.
This would indeed be worthwhile as it could vastly improve the visual
fidelity if done properly. We also introduced an option in our application
to render graphs using dot in Section 7.3.6. Given that they are rendered
as SVG elements, we experimented with styling them interactively in
Section 7.4.2. In the future, this could be much improved by recreating
the replay feature that we implemented for Cytoscape.js (see Section
7.3.9), with proper animations on edges and improved styles. This would
vastly improve upon existing tools without needing a custom layouting
algorithm and visualisation component.
