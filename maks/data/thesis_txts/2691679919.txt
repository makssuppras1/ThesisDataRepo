Abstract
The standard string pattern matching problem is to, given two strings s (the text) and w (the
pattern), ﬁnd whether w occurs in s. This problem can be extended by allowing variables in the
pattern. In the extended version, the aim is to ﬁnd whether we can obtain the string s by replacing
the variables in the pattern by terminal words, using the same word for each instance of any given
variable. This problem is NP-complete, however, if we restrict the considered patterns to speciﬁc
classes, it becomes solvable in polynomial time. Fernau et al.[ACM Trans. Comput. Theory 2020]
designed an algorithm that solves the problem for patterns containing one repeated variable in O(n2)
time. Our algorithms improve the running time, at the cost of restricting the class of supported
patterns even further. The ﬁrst algorithm operates on the patterns in which there is a non-repeating
variable between each instance of the single repeating one. With one additional requirement on the
pattern, we achieve expected O(max(nk, n1+ε)) time, where ε is any positive constant and k is the
number of times a variable occurs in the pattern. It is done by creating pointers in the sufﬁx tree that
we use to move from a node labeled with p to a node labeled with wLpwR (if one exists), where wL
and wR are terminal strings. Given any wL and wR, we generate the pointers in O(n) time for all nodes
in the tree. Additionally, the paper contains a second algorithm that solves the problem in O(nlog2 n)
time for some of the patterns containing exactly two occurrences of the repeating variable.

Introduction
The standard string pattern matching problem is to, given two strings s (the text) and w (the pattern),
ﬁnd whether w occurs in s. This problem can be extended by allowing variables in the pattern. As this
paper is concerned with the extended version, we deﬁne a pattern as a string of terminal symbols and
variables. We will use the ﬁrst few letters of the latin alphabet to denote the terminals, in most cases they
will be a and b. We will focus on patterns with one repeating variable, which will be represented as x.
For the other variables we will use y0, y1, etc.
The pattern matching with variables problem is to, given a pattern α and a terminal string s, check
whether it is possible to replace the variables in α with terminal words and obtain the string s. However,
all the instances of one variable need to be replaced with the same word. In our case, this only applies
to the variable x, as it is the only one that can repeat. For example, if we are given the string bbabbab
and the pattern xaxy0b, we obtain the string by replacing x with bb and y0 with a. However, if we are
given the same pattern with the string abaaaaa, there are no possible replacements for x and y0 that will
give us the string, as they cannot change the last character of the pattern, which is different than the last
character of abaaaaa.
The standard string matching problem of ﬁnding an occurrence of a terminal string w in a text, in our
notation corresponds to matching the pattern y0wy1. If we use the string w by itself as the pattern, it will
only match if the text is exactly w.
The previous results
The pattern matching problem emerges in multiple areas of science, such as
learning theory[6], combinatorics on word[16] and database theory[1]. Dana Angluin has proven that
the general pattern matching problem is NP-complete[8], however, there are restricted classes of patterns
for which there exist polynomial time algorithms[17]. Brenda Baker has researched the injective version
of the problem, where the replacement needs to be different for each variable. Out approach is partially
based on the O(n2) time algorithm for patterns with one repeated variable designed by Fernau et al[7].
1.1
Problem deﬁnition
Formally, we deﬁne a substitution for the pattern α to be a function f : X →Σ∗, where X is the set of
variables occurring in α, Σ is an alphabet of terminals and Σ∗is the set of ﬁnite strings of symbols from
Σ. We deﬁne the replacement for x in f to be the string f(x). In this paper, we allow empty string to
be the replacement for both x and yi. However, it is easy to alter our algorithms to the version of the
problem where only non-empty replacements are allowed. As we only consider patterns with a single
repeated variable, we deﬁne the string f(α) to be the string obtained by replacing all instances of the
variable x in α with f(x) and all variables yi with f(yi). If f(α) = s, than the substitution f is matching
for α and s. A pattern α matches a string s if and only if there exists a matching substitution for α and s.
If α contains more than |s| terminal characters, than it does not match s. If it contains more than
|s| occurrences of x, than the only possibly valid replacement for x is the empty string. As a result, the
problem becomes equivalent to checking whether α′ matches s, where α′ is the pattern obtained after
removing x from α. As α′ is a pattern without repeated variables, this problem can be solved in O(n)
time[7]. If α contains two occurrences of a non-repeated variable in a row, they can be merged into one
variable, as α matches s if and only if α′′ matches s, where α′′ is the pattern obtained after merging
all consecutive non-repeated variables of α. As a result, we only have to consider the patterns where
between any two occurrences of a non-repeated variable there is either a terminal or an instance of x.
Taking all of this into account, we assume that |α| = O(|s|).
We deﬁne the separable patterns with a single repeated variable to be the set of patterns containing
a non-repeating variable between any two instances of the repeating variable x. In other words, the
pattern α is separable if for any two indices j1 and j2 on α such that j2 > j1, α[j1] = x and α[j2] = x,
the following is satisﬁed:
∃i ∈[j1 +1, j2 −1]
α[i] = yt
3
==PAGE:5==
We deﬁne the left adjacent wL of an instance of x at index j to be the maximal terminal substring of
α ending at j −1. If j = 0 or α[j −1] is a variable, the left adjacent of x at j is the empty string. The
right adjacent wR of x at j is the maximal terminal substring of α starting at j + 1, if j = |α| −1 or
α[j +1] is a variable, it is the empty string. For example, in the pattern y1axbby2x, the left adjacent of
the instance of x at index 2 is the string a, the right is bb. Both the left and right adjacents of x at 6 are
empty strings. We deﬁne the set Lα to be the set of all left adjacents of instances of x in α, likewise for
Rα. We say that a pattern α is non-aligned whenever at least one of the following conditions is satisﬁed:
• There exist left adjacents wL1 and wL2 in Lα such that neither of them is a sufﬁx of the other.
• There exist right adjacents wR1 and wR2 in Rα such that neither of them is a preﬁx of the other.
We deﬁne the non-aligned separable patterns with a single repeated variable to be the set of
patterns with a single repeated variable satisfying both the separable and non-aligned requirements.
Non-aligned patterns have a useful property that can be shown using the sufﬁx tree. Generally, to
check whether a pattern matches a string s, the only replacements we have to consider are substrings
of s. If the pattern α contains a right adjacent wR, than for any substitution f, f(α) contains the sub-
string f(x)wR. Therefore for f to be matching, s must also contain f(x)wR. If wR1 and wR2 satisfy
the requirement above, than there must exist an index i such that wR1[i] ̸= wR2[i]. As a result, if s con-
tains both the strings f(x)wR1 and f(x)wR2, than the sufﬁx tree of s must contain a node labeled with
f(x)wR1[0...i −1]. The sufﬁx tree contains O(n) nodes, so for any non-aligned pattern there are only
O(n) possible replacements for x. Additionally, the string s matches the pattern α if and only if the
reversed s matches the reversed α. As a result, if the pattern satisﬁes the condition on left adjacents, we
can reverse both the pattern and s. After reversal the pattern will satisfy the right adjacents requirement.
Now we can state the main result that we will prove in the paper:
Theorem 1. The pattern matching problem for non-aligned separable patterns with a single repeated
variable can be solved in expected O(max(n1+ε, kn)) time and O(n1+ε) space, where ε is any positive
constant and k is the number of times a variable occurs in the pattern.
Afterwards, in section 8 we show our attempt to relax the requirement on separability of the patterns.
However, the algorithm we designed only supports the patterns containing two instances of x. As a result,
we prove the following:
Theorem 2. The pattern matching problem for the patterns of the form α = y0w0xw1xy1, where w0 and
w1 are strings of terminals, can be solved in O(nlog2 n) time and O(n) space.
1.2
Overview of the paper
In this paper we present two algorithms that allow us to prove theorems 1 and 2. We start with outlining
the main ideas behind the algorithm for non-aligned separable patterns in section 3. The algorithm relies
on using auxiliary pointers added to the sufﬁx tree of the string s. Given any terminal string wL, for
each node in the sufﬁx tree with label p, we create a pointer to a node labeled with wLp. This can be
done in O(n) time, and is described in section 4. In sections 5 and 6 we describe how to create the
respective pointers for sufﬁxes, that is pointers from a node labeled with p to a node labeled with pwR.
The pointers allow us to, given any string replacement p for x and a node in the sufﬁx tree of s labeled
with p, check if we can match the pattern substituting x with the string in O(k) time, where k is the
number of occurrences of a variable in the pattern. Sections 4-6 are written to be partially self standing,
they do not refer to patterns at all. However, section 6 is heavily based on section 5. In section 7, we
present the full description of the algorithm for non-aligned separable patterns. Finally, in section 8,
we describe the algorithm that solves the problem for the patterns described in Theorem 2. It utilizes
a completely different approach to that of the main algorithm, instead of using auxiliary pointers in the
sufﬁx tree, it focuses on the starting positions of the possible replacements for x.

Preliminaries
Strings
A string s of length |s| is a sequence s[0]s[1]...s[||s| −1] of elements from a ﬁnite alphabet
Σ. When s will denote the text, the string to which we are matching the pattern, we will also refer
to its length as n. A substring of s is a contiguous subsequence of s. An example of a substring is
s[i... j] = s[i]s[i+1]...s[j]. A preﬁx of s is any substring of s starting at index 0, a sufﬁx is any substring
ending at index |s|−1. In, particular, s is both a preﬁx and a sufﬁx of itself and an empty string is both a
preﬁx and a sufﬁx of any string. The notation p ⪯pre f s is equivalent to saying that p is a preﬁx of s. The
notation p ≺pref is equivalent to p ⪯pref s and |p| < |s|.
The string p[0]p[1]... p[|p|−1]s[0]s[1]...s[|s|−1] = ps is a concatenation of strings p and s. We use
product notation for concatenation:

Range next value
The range next value problem is deﬁned as follows:

Informally, return the smallest value from the subrange of A that is greater or equal to j.
Crochemore et al[5] have shown an algorithm that preprocesses the array A in O(n1+ε) time, where

Range next value is closely related to sufﬁx trees. The descendant leafs of a node v, representing any
sufﬁx of s that starts with the label of v, form a contiguous range in the sufﬁx array. Given any index j
on s and a node v with its descendant range [l, r], we can ﬁnd the ﬁrst sufﬁx of s that starts at or after
j and is preﬁxed with the label of v using RNV(l, r, j). As a result, we can ﬁnd the ﬁrst occurrence of
the label of v starting after j. For example, the range of the sufﬁx array under the node vr labeled with a
(highlighted in red in Figure 1) contains the values:
A[1...4] = [6, 4, 0 2]
We can ﬁnd the start of the ﬁrst occurrence of a at or after the index 3 using the query RNV(1, 4, 3) = 4.
Weighted level ancestor
The weighted level ancestor problem is a generalization of the predecessor
problem. We are given a rooted tree where each node has a non-negative integer weight. Additionally,
the weights are non-decreasing on any path from root to a leaf. We want to support a following query:
Given a node v and an integer q, ﬁnd the shallowest ancestor of v such that its weight is greater or equal
to q.
In our paper, we will use the version of the problem for sufﬁx trees, where the weight of a node is
the length of its label. Gawrychowski et al[11] were the ﬁrst to show and O(n) space data structure that
answers the query in constant time (for this version of the problem). Belazzougui et al[2] have shown an
O(n) time algorithm building such a data structure. In the paper, we will use AncestorAt to denote the
query.
2D range existence query
In our paper, we will be using the following version of the 2D range query
problem:
Given a set U of pairs (xi, yi), where both elements of the pair are integers from the range [0, n] and four
integers xmin, xmax, ymin, ymax, ﬁnd whether U contains a pair (x, y) such that:
x ∈[xmin, xmax]
y ∈[ymin, ymax]
Chan et al[3] have shown an O(nlogn) time algorithm that given a set U, builds a O(nloglogn) space
data structure that allows for a O(loglogn) time query for this version of the problem.

Outline of the main algorithm
Let’s ﬁrst consider a simpler problem, where we want to check whether we can obtain the string s from
the pattern α by substituting x with a given terminal string p. In this simpliﬁed problem, we can separate
the pattern into subpatterns occurring between the instances of variables yi and match the subpatterns
with the shortest preﬁxes of s in a greedy fashion. We will explain this approach on an example, where:
7
==PAGE:9==
• s = aabbaabbaabbab is the text.
• α = y0aaxy1xy2 is the pattern.
• p = bb is the attempted replacement.
The ﬁrst subpattern is y0aax, so we are looking for substrings of s equal to aap = aabb. The ﬁrst one
starts at index 0, so we greedily replace y0aax with s[0...3] (y0 is replaced with the empty string). There
are other preﬁxes of s we could use, for example s[0...7]. However, if there is a substitution for the rest
of the pattern matching with s[8...|s| −1], than we can use it with our greedy start. The only required
change is preﬁxing the replacement for y1 with s[4...7]. Since y1 occurs only once in α, we know that
this gives us a matching substitution. This approach is borrowed from Fernau et al[7].
The separable patterns with a single repeated variable can be factorized in the following way:
α = γ0

maximal terminal substrings ending or starting right next to an instance of x. Examples of subpatterns
satisfying the requirements for γ are y1, y1ay2, y1aby2y3. In particular, the empty string does not satisfy

The factorization stems from the observation above. Given a replacement for x, we will greedily
match the factors wLi xwRi and γi with the shortest preﬁxes of the remaining unmatched part of the string.
Our main idea is as follows:
Given:
• A replacement p for x, together with the corresponding node in the sufﬁx tree.
• The factor wLi xwRi.
• The index j of the end of the mapping for the previous part of the pattern.
Find the mapping to the factor by:
1. Using auxiliary pointers to locate the node in the sufﬁx tree of s labeled with wLi pwRi.
2. Returning the ﬁrst occurrence of wLi pwRi in s starting after the index j.
The ﬁrst step consists of ﬁnding a value in a dictionary and traversing two pointers, so it takes expected
constant time. The second step also takes constant time, assuming we use a fast range next value data
structure. In the subsequent sections we will describe the procedures we use to generate the auxiliary
pointers.

Finding w-preﬁx matching nodes
Let w be a terminal string. A pair of nodes (v1, v2) from the sufﬁx tree of s is w-preﬁx matching if and
only if v2 is the shallowest node in the tree such that:
wpv1 ⪯pre f pv2
where pv1 is the label of v1 and pv2 is the label of v2. In this section, two nodes v1 and v2 are matching if
(v1, v2) is a w-preﬁx matching pair. If the labels of nodes are c, ac, d, ada and w = a, the matching pairs
are (c, ac) and (d, ada), since there is no node labeled with ad. An important observations is that all of
the second elements of matching pairs will be descendants of the node labeled with w, or, if such node
does not exist, the shallowest node with w as a preﬁx of its label.

Time complexity
The execution time of the algorithm is constant per traversed edge. The number of
edges traversed in the for loop in line 9 is exactly the number of edges in the subtree rooted at the node
vw. We never go back to parent while traversing the edges of the main tree, so we never use the same
edge more than twice, once in the search of pv and once while traversing the equivalent edges. As a
result, the execution time of the algorithm is O(n).
Sufﬁx tree implementation
In the analysis we assumed that we have constant time access to the edges
based on their ﬁrst character. However, if that is not the case and the edges are sorted in alphabetical
order, the algorithm can be slightly altered to maintain the O(n) time. We can save the position of the
last used edge of current in line 11. Then, in the next iteration, we can start searching for next edge at
the saved index, as the edges of the parent are in sorted order. Since we never visit the same node twice,
the complexity is O(n).

5
Finding w-sufﬁx-matching nodes
Deﬁnitions
A pair of nodes (v1, v2) is w-sufﬁx-matching if v2 is the closest descendant of v1 such that
w is a preﬁx of the label of the path from v1 to v2. In this section, whenever we say that nodes v1 and
v2 are matching we mean that (v1, v2) is a w-sufﬁx-matching pair. Similarly, when we say that there is
match during processing of node v, we mean that there exists an ancestor va of v such that (va, v) are a
w-sufﬁx-matching pair.
Let j2 be the index in the string s right after the end of the edge of v2. The label of v2 is the substring:
pv2 = s[j2 −|pv2|... j2 −1]
The label of the path from v1 to v2 is the substring:
p−1
v1 pv2 = s[j2 −|pv2|+|pv1|... j2 −1]
Let b = j2 −|pv2|+|pv1|. By deﬁnition of w-sufﬁx-matching:
s[b...b+|w|−1] = w
In other words, there is an occurrence of w in s starting at b and ending at b +|w|−1. We say that this
occurrence represents the match between v1 and v2. It is possible that an occurrence represents more
than one match.
The occurrence of w representing any match must end within the ﬁrst |w| characters of an edge,
otherwise the index before the start of the occurrence would be inside the edge.
Middle occurrences are all but the ﬁrst and last occurrences of w in a label of a node ending within
the ﬁrst |w| characters of its edge. Middle matching ancestors of v are the matching ancestors va such
that the occurrence representing the match is a middle occurrence. Handling middle matches separately
from the ﬁrst and last ones is the key idea we use in our algorithm.

Overview
The aim of the algorithms presented in this section is to, given a string w and a string s
together with its sufﬁx tree, ﬁnd all w-sufﬁx-matching pairs of nodes. In other words, for each node v,
we want to ﬁnd its closest descendant vd (if one exists) such that the label of the path from v to vd starts
with w. For example, let w = cc and the sufﬁx tree contain nodes ab, abcc, abcca, c and ccc. In this case,
we want to output the pairs (ab, abcc) and (c, cc).
We will start with describing a simple solution to the problem. Afterwards, in subsection 5.1, we
present an algorithm constituting the baseline of our main approach. Subsection 5.2 contains description
and proofs of the most important properties we use to improve the baseline, in particular properties relat-
ing to string periodicity. In subsection 5.3 we describe and prove the correctness of the main, optimized
algorithm running in linear time. We show an example execution of the algorithm in subsection 5.4.
Simple solution
We can solve the problem using the standard sufﬁx tree algorithm for checking
whether a string s contains a substring w. In the standard algorithm we, starting from the root, go
down the tree using the edges labeled with characters from w. If we can reach a node with the label of
length at least |w|, we have found that w exists in s. This takes O(|w|) time.
We can alter this algorithm for the current problem. Instead of running the algorithm starting from
the root, we run it starting from all nodes independently. If, starting from a node v with label pv, we
reach node vd with |pvd| ≥|pv|+|w|, we output the pair (v, vd). Since we run the algorithm starting from
all nodes, it takes O(n|w|) time.

The algorithm
Instead of ﬁnding the matches for all nodes independently, we output all the matching
pairs during a single depth-ﬁrst search pass. We keep track of the nodes that are currently in the DFS
stack in a direct access array Stack, with indexing on depth. When we ﬁrst visit an edge of a node v,
we iterate through all the occurrences of w within the edge’s string. The iteration can be sped up using
the Next array. For each occurrence, if there is a node va in the stack corresponding to the start of it, we
output the pair (va, v).
Since the total string length of edges can be up to n(n+1)
2
, there are strings with length n and Ω(n2)
occurrences of w in their sufﬁx trees. An example of such string for w = a is akbak. The algorithm
processes all of the occurrences of w, so this approach by itself is hardly an optimization over the simple
solution. However, if we ignore the occurrences ending after the ﬁrst |w| characters of an edge and w is
not periodic, than we will process at most one occurrence of w per edge, so O(n) occurrences in total.
Therefore the algorithm is optimal for non-periodic strings w. In subsection 5.3 we show how to achieve
linear time for periodic strings.
Example
When the algorithm reaches the node abaaaaa$, highlighted in red in Figure 4, it checks
Next for the ﬁrst occurrence ending at index 9 or later. It ﬁnds an occurrence at 9, so it checks if a
matching ancestor exists in the DFS stack. The current depth is |abaaaaa$| = 8, the last position of
the edge is 11, so the offset is 4. The depth of the end of the occurrence is 9 −4 = 5, so the matching
ancestor would need to have depth 5−|w|+1 = 5−2+1 = 4. There is no such ancestor in the DFS stack
and the algorithm continues to the next occurrence, found at index 10, which corresponds to ancestor at
depth 5. The algorithm ﬁnds the node abaaa (highlighted in blue) in the stack, and outputs the pair
(abaaa, abaaaaa$).

Main ideas behind the algorithm
1. The fundamental structure of the algorithm is depth-ﬁrst search.
2. We process the ﬁrst and the last occurrence of w inside an edge separately from the middle ones.
3. Most of the code deals with the case when |w| ≥2P+1 and there can be middle matches.
4. Unlike in the baseline algorithm, we do not iterate over the occurrences of w. We iterate over
middle matching ancestors instead.
5. We store all the nodes that might be middle matching with current or its descendants in mids.
6. We use Lemma 5.6 and Lemma 5.8 to maintain the invariant that if a node is in mids when line 24
from OutputPairs is executed, it is a middle match for current.
7. Whenever it is evident that mids cannot contain any nodes that will be middle matches for a
descendant of current, instead of clearing it we swap it to a new list, as the contents of mids might
be used in a sibling node.

All of the lines except the loop from line 20 in OutputPairs use constant time per

A note on the data structure used for mids

5. The beginning of mids can be modiﬁed in OutputPairs, so we need to update the start index

Finding w-sufﬁx-matching edge indices
Let v1 and v2 be nodes in a sufﬁx tree. Let pu be the label of the parent of v1, or the empty string if v1 is
the root. Let j be an integer in the range [|pu|+1, |pv1|], so an index on the part of pv1 inside the edge
of v1 or the length of pv1. We deﬁne the triple (v1, j, v2) to be w-sufﬁx-matching whenever v2 is the
shallowest node such that:
pv1[0... j −1]w ⪯pre f pv2
Since the index j is inside the edge of v1, v2 must be a descendant of v1. Additionally, since v2 is the
shallowest node satisfying this requirement, it is uniquely deﬁned. The triples are a generalization of
w-sufﬁx matching pairs from the previous section. A pair (v1, v2) is w-matching if an occurrence of w
starts right after the edge of v1. We relax this requirement for the triples and let the occurrence start after
any index j inside the edge, represented by the second element of the triple.
In this section, whenever we say that nodes v1 and v2 are matching, we mean that there exists at least
one triple of the form (v1, j, v2). Additionally, we say that j is a matching index for v1 if there exists a
node v2 such that (v1, j, v2) is matching.
Properties of matching triples
Lemma 6.1. The string s contains an occurrence of pw if and only if the sufﬁx tree of s contains a
matching triple (v, |p|, vd), such that pv[0...|p|−1] = p, where pv is the label of v.
Proof. First, let’s assume that the triple (v, |p|, vd) is matching and pv[0...|p|−1] = p. By deﬁnition of
matching triple, we have pw ⪯pref pd, where pd is the label of vd, so by deﬁnition of sufﬁx tree pw must
exist in s.
For the converse, let s[t ...t + |p| + |w| −1] be an occurrence of pw in s. Let vl be the leaf labeled
with pl = s[t ...|s|−1]. We have pw ⪯pre f pl, so there must exist a shallowest ancestor of vl, such that
pw is a preﬁx of its label, let’s name it vd and it’s label pd. Since p ⪯pre f pd, there must exist a shallowest
ancestor of vd such that p is a preﬁx of its label, let’s name it v. By deﬁnition, the triple (v, |p|, vd) is
matching.
Lemma 6.2. Let O be the set of pairs of nodes and indices such that (j, vd) is in O if and only if:
• pd[j... j + |w| −1] = w, where pd is the label of vd. In other words, the index j is a start of an
occurrence of w in the label of vd.
• j +|w| ≥|pu|, where pu is the label of the parent of vd. In other words, the occurrence ends inside
the edge of vd.

The aim of this section is the description of an expected O(n) time algorithm that
builds a data structure which can be used to answer the following query in expected O(1) time:
• FindMatchingDescendant(v, j) = vd

We start by examining the number of matching triples. We will give an example of a class of trees that
contain Ω(n2) of them. Afterwards, we show the steps that allow us to avoid having to explicitly store all
of the triples. By grouping the triples, we reduce the required storage to O(n). In 6.2 we will describe an
algorithm that uses the constant time weighted level ancestor query from [11] to ﬁnd the grouped triples.
In the last parts of the section we will show how can we use them to build the ﬁnal data structure and
answer the queries.
Number of w-sufﬁx-matching triples
An important consequence of the matching triple deﬁnition is
that the same pair of nodes can be matching multiple times. Additionally, a node with sufﬁciently long
edge can be matching with itself. Unfortunately, this means that there are sufﬁx trees with quadratic
number of matches for a given w, for example the tree of the string akbak for w = a. For all i ∈[0,k]
it contains a node labeled with ai. Additionally, all of those nodes have a descendant edge labeled with
bak. Each of these edges has k matches within it, so the tree contains Ω(k2) matching triples. As a result,
there cannot exist an O(n) time algorithm that relies on explicitly storing the triples.

into two classes:
• Inner - all the triples (v, j, v), so all the triples where a node is matching with itself.
• Outer - all the other triples.

The number of matching quadruples
Lemma 6.3 allows us to upper bound the number of matching
quadruples by bounding the number of times a node can be a second or later ancestor match. This is the
most important result of this section, so in addition to the formal proof, we will attempt to give a more
intuitive explanation.
In general, whenever we have multiple occurrences of w, they need to be at least P away from each
other. Each match requires an occurrence, so if a node v is a second matching ancestor of vd, than the
start of the occurrence representing the match must be at least P after the ﬁrst one. As a result, the
existence of the ﬁrst match requires a preﬁx of w with length at least P at the end of the edge of the
second matching node.
If v was second match for a different descendant ve, than it would require another preﬁx of length
greater than P ending at the edge. This is only possible if the preﬁxes are aligned modulo P. However,
if they are aligned, than the next character of w will be the same for both preﬁxes. We can repeat this
reasoning for the whole path until either vd or ve. However, both vd and ve have matches both at and
above v. This means that their edges overlap, so they must be the same node.
We deﬁne the matching ancestor sequence of a node v to be the ordered by depth sequence of
distinct matching ancestors of v.

Theorem 6. Any sufﬁx tree of a string s contains less than 5|s| matching quadruples for any string w.
Proof. Let V be the set of nodes in the tree, |V| ≤2|s|−1. We split V into two sets:
• Va - the set of nodes which are in at most two matching triples as the descendants.
• Vb - the set of nodes which are in three or more matching triples as the descendants.
Since all of the nodes are contained in exactly one of Va and Vb:
|Vb| = |V|−|Va|
We have at most 2|Va| matching triples where the nodes from Va are descendants, so at most 2|Va| match-
ing quadruples with them as descendants.
By Lemma 6.3, there can be at most |V| matching quadruples where the ancestor was the second or
later match of a node from Vb. Naturally, a node can have at most one matching quadruple where the
ancestor was its ﬁrst match. Additionally, the leaves cannot be ancestor matches. There are at most |s|
non-leaf nodes in a sufﬁx tree, so there are at most
|s|+|Vb|
matches with nodes from Vb as the descendants.
We sum those two values and get at most:
2|Va|+|s|+|Vb| = |s|+|V|+|Va| < 3|s|+|Va| ≤3|s|+|V| < 5|s|
matching quadruples in the tree.

Main ideas behind the algorithm
1. We check the number of occurrences of w inside the ﬁrst |w| character of the edge before outputting
any quadruple.
2. All of the occurrences of w represent a match, so the sum of counts inside the output quadruples

3. Similarly to OutputPairs from subsection 5.3, whenever there are three or more occurrences the
number of iterations we make is equal to the number of output matches.
4. We iterate over occurrences of w, but after ﬁnding the number of matching triples with an ancestor
we skip to the ﬁrst occurrence that starts in the next matching node.

6.3
The data structure
The query we want to support is, given a node v and an index j on its edge, ﬁnd a matching descendant.
As we have described before, if the index j + |w| −1 is inside the edge, answering the query is trivial.
For the other indices, an obvious idea is to store the quadruples using a dictionary with ancestor nodes
and indices as keys. Unfortunately, this is not enough, as even if the quadruple for index j does not exist,
there might exist one with index j −kP and the count big enough to represent a match at j.
A different approach is to have the ancestor node and start index of the quadruples modulo P as the
key instead. Now, we can store the quadruples for v and j modulo P in a list sorted by depth. Using this
approach we can retrieve any matching descendant by binary searching the list. However, a single list
could contain |w|
P entries, so this results in O(log |w|
P ) query time.
Fortunately, this is unnecessary. All of the occurrences representing matching triples of v must start
either in the last |w|−1 indices of the edge of v, or right after the edge. By Lemma 5.1, if two of them
start at positions equivalent modulo P, than there are occurrences starting for all P equivalent indices in
between the two. As a result, we only need to know the positions of the ﬁrst and the last one.
Taking all of this into account, instead of storing the quadruples, we store a hash array with nodes v
and indices j modulo P as keys and the values:
33
==PAGE:35==
1. The index f of the ﬁrst matching quadruple of v with f ≡P j.
2. The index l of the last matching triple. It can be retrieved from the quadruples using the following:
l = max{ j′ +(C −1)P : (v, j′, vd, C) is matching and j′ ≡P j}
3. The node where the last occurrence ends, in other words the node vl such that the triple
(v, l, vl)
is matching.
We only store the keys (v, j mod P) if there was a quadruple with v as ancestor node and equivalent
index, so the total number of elements in the hash array is at most the number of matching quadruples.
Finding all the quadruples takes O(n) time using the procedure SearchForQuadruples. Retrieving
the values above and building the hash array takes expected O(n) time, as we have to iterate through all
quadruples and, depending on the quadruple, update the values in the array.
6.4
Query
Whenever we answer the query FindMatchingDescendant(v, j), we start by checking if j +|w|−1 is
inside the edge. If this is the case, we check if there is an occurrence ending at j + |w| −1 using the
Match array. Afterwards, we check if the hash array contains the key (v, j mod P). If it doesn’t, than
there is no match and we return. Afterwards, we retrieve the values (f, l, vl) and check if j ∈[f, l]. If it
is not, than j is not matching. Otherwise, we ﬁnd the matching node using the weighted level ancestor
function at the node vl, so we return:
vd = AncestorAt(vl, j +|w|−1)
Finding a key in a hash array takes expected constant time. Finding the matching ancestor takes constant
time, so the query time is expected O(1) as well.

7.2
The main loop
After initialization we move on to the main loop, which consists of processing the consecutive factors of
α in a greedy manner. The actions during an iteration are focused on ﬁnding, for all of the replacements
that we still consider valid, the shortest preﬁx of the remaining string that maps to the current factor.
Importantly, what we consider "the remaining string" will be different for different replacements. After
ﬁnding the shortest matching preﬁx, we remove the found preﬁx from the remaining string by updating
the satellite data in the hash array Valid.
Each iteration of the main loop has its own initialization step. We could add generating the preﬁx
and sufﬁx matching pairs for all left and right adjacents to the steps described in the previous section,
but doing so would increase space requirements, since we would need to store all of the created arrays.
Fortunately, this is unnecessary. After we process the current factor and update Valid for all remaining
substrings, we will not need to access the arrays again. As a result, we can reuse the memory they used
for the pointers from the next factor.

conditions for stopping the loop are described below the steps for the factors.

Otherwise, retrieving the full substitution, that is the replacements for the variables y, can be done in

terminal substrings and non-repeating variables. Using the same greedy approach we described above,
we start with locating the ﬁrst occurrence in s of the ﬁrst terminal substring. Then, we locate the second

The case when wb is empty and the w-sufﬁx-matching nodes
In the description of the algorithm, we
have not used the algorithm for ﬁnding w-sufﬁx-matching nodes from section 5. This is due to the fact
that the sufﬁx matching edge indices algorithm from section 6 is more general and uses the same amount
of time in O notation. However, there are a couple advantages of the matching nodes algorithm. First, it
is a bit more self contained, as it does not use the the weighted level ancestor query. As a result, it can
be implemented quicker. Additionally, while we have not benchmarked them in conditions allowing for
a deﬁnitive speed comparison, we believe that the matching nodes algorithm performs faster in practice.
In general, whenever the pattern contains two right adjacents with different ﬁrst character, the array
Valid simply contains the pairs of nodes with the last indices of their edges. As a result, using a hash
array is unnecessary, as it can be a standard array with indexing on nodes. Whenever this is the case,
we only need sufﬁx matching nodes instead of edge indices and can use the algorithm from section 5,
storing the results as an array. Then, FindMatchingDescendant becomes a simple array access.

7.3
Time and space complexity
Initialization
1. Factorizing the pattern takes O(|α|) time.
2. Building the sufﬁx array takes O(n) time.
3. Building the sufﬁx tree also takes O(n) time.
4. Building the Range array requires only a simple DFS, so it takes O(n) time.
5. Building the range next value data structure using the algorithm by Crochemore et al[5] takes
O(n1+ε) time, where ε is any positive constant.
6. Building the weighted level ancestor data structure takes O(n) time using the algorithm by Belaz-
zougui et al[2].
7. Generating the Match array using the KMP algorithm takes O(n) time.
8. Finding the set V is done by a simple DFS search, so it takes O(n) time.
9. As the AncestorAt query takes O(1) time, ﬁnding the set A takes O(|V|) = O(n) time.
10. Building the perfect hash array Valid takes expected O(|A|) = O(n) time.
All things considered, the most costly initialization step is building the range next value data structure,
so the whole initialization phase takes O(n1+ε) time, where ε is any positive constant.
Factors wLixwRi
The number of iterations of the loop where we process these factors is equal to the
number of times the variable x occurs in the pattern, we will name it kx. We will list the execution time
of the processing steps per iteration:
1. Finding the wL-preﬁx-matching pairs takes O(n) time, as described in section 4.
2. Generating the Match, Next and Previous arrays using the KMP algorithm takes O(n) time.
3. Finding the wRi-sufﬁx-matching quadruples takes O(n) time.
4. Building the data structure supporting the query takes expected O(n) time.
5. Steps a-f take constant time per element of Valid, with the exception of step b, which takes
expected constant time. Since there are O(n) elements, the total time taken by step 5 is expected
O(n).
As a result, each iteration of the loop takes expected O(n) time and the total time taken by processing
these factors is expected O(nkx).
Factors γi
As we further factorize these factors before processing, instead of evaluating the execution
times per iteration, we will consider the total time they take during execution of the algorithm. We deﬁne
ky as the number of occurrences of non-repeating variables in the pattern.
1. Further factorizing γi takes O(γi) time, so O(α) time across all iterations.
2. Finding the node vq in the sufﬁx tree takes O(|wq|) time. As wq are non-overlapping substrings of
α, this step also takes O(α) time in total.

3. Each of the steps (i-iii) of the innermost loop takes constant time. The number of times they are
executed is equal to the number of elements in Valid multiplied by the total number of the factors
wq. Valid contains O(n) elements. All of the factors wq are separated with a variable y, so the
number of factors is O(ky). Taking all of this into account, the total time step 3 takes during

It is clear that step 3 is the most costly in processing of this factors, so the total time taken by this part of

Summary of time complexity
Let k = kx +ky be the total number of occurrences of a variable in the

O(n1+ε +nkx +nky) = O(max(n1+ε, nk))
expected execution time, as we have stated in Theorem 1.
Space complexity
All of the arrays created in the initialization step are of size O(n). The sufﬁx tree,
the weighted level ancestor data structure the Valid hash array take O(n) space and the data structure
for range next value takes O(n1+ε) space. As we do not store the arrays generated during the iterations
of the main loop, the additional space required after initialization is O(n). To summarize, the total space
required by the algorithm is O(n1+ε).

8.1
The algorithm
In this section, we deﬁne a pair of positions i and j to be a matching pair whenever i and j satisfy
conditions 1-4. The aim of the algorithm is to either ﬁnd a matching pair, or determine that no such pair
exists. We start with creating the sufﬁx tree of the string s. Then, we start the main loop of the algorithm,
during which we repeatedly merge two neighbouring leaves of the tree until only one node is left. If the
procedure ﬁnds a match (how it is done is described below), it breaks early. If the main loop completes,
we report that the string does not match the pattern.
Technically, after a merge operation the tree is no longer a sufﬁx tree, since the node resulting from
the merge does not have a well deﬁned string label. This is not a problem, as we do not use labels of the
leafs during the main loop of the algorithm.
Deﬁnitions
• Sv - the set of all sufﬁxes labeling leaves below node v. Sv is a continuous range of the sufﬁx array,
so it can be stored as two indices.
• pv - the string labeling node v.
• si - the sufﬁx of s starting at position i.
• L - the set of all leaves in the tree.
• I - the set of positions i such that the sufﬁx starting at i−|w0| has w0 as a preﬁx. In other words,
all the starts of sufﬁxes satisfying condition 1.
• J - the set of positions j such that the sufﬁx starting at j −|w1| has w1 as a preﬁx. In other words,
all the starts of sufﬁxes satisfying condition 2.
• Iv - the subset of I assigned to node v. For the nodes v that were initially in the sufﬁx tree, the set
of positions i such that the sufﬁx starting at i−|w0| has w0pv as a preﬁx.
• Jv - the subset of J assigned to node v. For the nodes v that were initially in the sufﬁx tree, the set
of positions j such that the sufﬁx starting at j −|w1| has w1pv as a preﬁx.

We take the smaller of the sets Ik and Jl. We check if there is a matching pair Ik and Jl by using the

∀i ∈Sk ∀j ∈Sl

leaf v, Sv = Sk ∪Sl, Iv = Ik ∪Il and Jv = Jk ∪Jl. Since k and l are consecutive children, Sv is a continuous

parent and delete v.

8.2
Implementation of the position sets
The data structures for sets Iv and Jv need to support two operations, range existence query and merging
two consecutive leaves in the tree. We considered three implementations.
2D range query
For all leaves h of the sufﬁx tree we can ﬁnd Ih by ﬁnding vh, the shallowest node
with w0ph ⪯pref pvh. If no such node exists, Ih is empty, otherwise, Ih = {i : si−|w0| ∈Svh}. We use the
w-preﬁx matching algorithm from section 4 to ﬁnd vh for all nodes h. Since Svh is a consecutive range in
the sufﬁx array, our query corresponds to a 2D range existence query, with the index in the sufﬁx array
as one dimension and the position in the string as the other. The nodes k and l are consecutive, so if the
nodes vk and vl exist, they are consecutive as well. Therefore, we merge Ik with Il by simply setting the
newly created Iv to be the range from the start of Ik to the end of Il.
47
==PAGE:49==
Range next value
The values for both of the dimensions of the 2D range query described above are
permutations of the integers from the interval [1, n]. A value in the range [x1,x2] × [y1,y2] exists if and
only if RNV(x1,x2,y1) ≤y2, therefore we can use any range next value data structure. We merge using
the same procedure as 2D range query.
Sorted set
An alternative approach is storing the sets themselves, using any sorted set data structure
such as red-black tree[12]. In the beginning, we use the KMP algorithm [15] to label all positions
satisfying conditions 1 and 2. Afterwards, we create the sets I and J for the leaves. If the sufﬁx sk of a
leaf v satisﬁes condition 1, then Iv = {k}, otherwise it is empty. Similarly for J. The merging of two sets
is done by inserting all the elements of the smaller set into the bigger one. This implementation allows
us to save space compared to 2D range query and range next value.
Time complexity
We deﬁne |Iv|+|Jv| to be the size of the node v and analyse the complexity as if Ik
and Jk were a single set, instead of a pair. Both merging and querying for existence are the essentially
the same for both (Ik, Jl) and (Il, Jk). The only difference this approach makes is that we consider the
number of existence queries as if the algorithm picked the node with smaller total size and then used it
for both pairs. Picking the smaller of the sets separately for each pair, as described in the algorithm, will
always use at most as many queries, so our analysis will show an upper bound.
The tree contains n leaves, so the main iteration and merging will be executed n−1 times. A query
is run for an index if, and only if, the size of the node it will be in after the iteration will be at least twice
the size of the node it was in before. Therefore it can be used for queries at most logn times and the total
number of existence queries is O(nlogn).
The time complexity of a single existence query will depend on the implementation of the I and J
sets. For 2D range query, it will be O(loglogn), for range next value it will be O(1) and for the red-black
tree it will be O(logn).
The preprocessing time is O(nlogn) for the 2D range query. It is O(n1+ε) for the range next value,
what makes it the inferior solution for this type of pattern. It is O(n) for the red-black tree, as we only
have to run the KMP algorithm twice and initialize single element sets.
The total time for merging will be O(n) for both the 2D range query and range next value. For the
red-black tree we can use the same analysis as for the number of existence queries to show that the total
number of times an index is inserted is O(nlogn), so the total time used by merging is O(nlog2 n).
To summarize, the version of the algorithm using red-black trees proves Theorem 2. It is possible
reduce the time complexity to O(nlogn(loglogn)), but at the cost of increasing the space complexity to
O(nloglogn).

Conclusions and ideas for future work
To summarize, we have designed an algorithm that is faster than the existing alternatives for non-aligned
separable patterns with a single repeated variable. Additionally, we have presented an algorithm that
uses a completely different approach, that allows us to relax the requirement on the separability of the
patterns. Unfortunately, it is currently very limited in scope, as it only supports some of the patterns with
exactly two occurrences of the repeated variable. Thus, the obvious question is whether the approach
from the second algorithm can be somehow integrated in the main one.
Another question is whether we can relax the requirement on non-alignment of the patterns. Intu-
itively, the patterns where all of the left adjacents are sufﬁxes of one of them and all of the right adjacents
are preﬁxes of one of them should be easier to solve, so one would imagine that there might exist an
o(n2) time algorithm for these patterns.
In any case, for any replacement for the repeated variable to be valid, the text must contain at least as
50
==PAGE:52==
many non-overlapping occurrences of the replacement for the pattern to match. Designing an algorithm
that can quickly ﬁnd all such replacements in the sufﬁx tree, in the manner of our matching edge-indices
ﬁnding algorithm, would be a good start to ﬁnding even better solutions for the pattern matching with a
single repeated variable problem.
