Abstract
Mental health is a growing issue nowadays among adults. This means that the process
of therapy needs more support from novel technological solutions, so eﬀiciency can
be improved and progress can be tracked. Thus, this thesis aims to address this issue
and propose a clinical personal informatics tool to support therapy by aiding both
patients and clinicians.

1.1
Introduction
Nowadays, mental health continues to be a rising problem, as the number of adults
in the US who undergo mental health treatment has increased from 19.2% to 21.6%
between 2019 and 2021. [EJ22] This showcases the need for incorporating more ro-
bust technological solutions within the mental health treatment area.
Such novel
technological solutions are needed to support both the patient and the psychologist
during the therapy process by clearly displaying what the patient’s progress is. Not
only could this improve the experience during therapy both for patients and clinicians,
but also it could make the process more eﬀicient and cost-effective.
Clinical specialists are interested in comparing and contrasting multiple disease-related
phenomena in a patient’s life. Thus, to support clinicians and patients in the process
of therapy, and to alleviate the rising issue with mental health illness, a personal in-
formatics tool could be designed and developed. However, neither clinicians, nor their
patients, are usually well-versed in working with data visualizations. So, a proposed
solution has to prioritize simplicity.
This thesis describes a project conducted within the last five months, which aims
to address exactly the above-described issues and propose a solution to them. This
is done by designing and developing a clinical personal informatics tool to support
therapy.
The thesis is therefore structured in the following way:
• Chapter 1 introduces the reader to the project and provides background infor-
mation about related works in the field.
• Chapter 2 analyses the problem space of the project.
• Chapter 3 showcases the visual design and iterations of this project’s solution
space.

Related Works
The following subsections present related works, which are key for conducting the
project presented in this thesis.
1.2.1
The One-Button Tracker
The One-Button tracker [Wik23] [Wol21] is a small device that records an observation
via a timestamp upon the press of a button. It is an open-source device that also
records the length of a press, which allows for various ways of using it for self-tracking.
Through different protocols, the data from the One-Button tracker can be trans-
formed into observations of several phenomena.
This can be done by assigning a
timestamp and the length of a press to specific phenomena. For example, two phe-
nomena can be tracked simultaneously with a single press and a double press. Another
option is to distinguish between two phenomena with a short and a long press.
As described in the following chapters, this project uses data from the One-Button
tracker, which utilizes the single-press protocol.
1.2.2
Observable & Vega-Lite
This project also utilizes the Observable framework [Inc23b], alongside the Vega-Lite
visualization framework [Veg23] to implement the proposed solution.
The Observable framework allows for data visualizations to be implemented in a
notebook-like layout, which provides both user friendliness and a clear code structure
for the developer. It also allows for the incorporation of various data visualization
frameworks. The Vega-Lite framework uses either a JSON-syntax, or a JavaScript

Current Solutions
Current solutions, which are widely used, have various limitations, especially when
it comes to tracking multiple phenomena and getting insight from them. Usually,
when visualizing multiple phenomena, current solutions just show two graphs on top
of each other, as seen in Figure 1.1. Even though this can provide some information
about how the visualized phenomena relate to each other, it is still very limited. This
thesis attempts to propose an improvement in this area, as explained in Chapter 3,
where multiple phenomena can be seen plotted on the same graph.

CHAPTER 2
Analysis
The analysis of the problem space of this thesis is a key part of the presented project.
It is thanks to the effort put into this initial stage of the project that the later stages
could result in well-thought-out specific ideas. Thus, the analysis of the project was
done based on a proven design methodology called the Double Diamond. [Cou04]
The Double Diamond is a model, which aids the design process by ensuring all project
stages are thoroughly explored, so a desirable final result is reached. As seen in Figure
2.1, the Double Diamond separates a project into two main spaces - the Problem space
and the Solution space. Each of them is then further separated into a divergent stage
and a convergent stage. This means that in the beginning of a project many problems
are explored, but are then narrowed down to more specific problem statements, which
help the design team solve concrete issues.
Figure 2.1: The Double Diamond design methodology.
The same logic applies to the Solution space: while various solutions are explored
in the beginning, they are later narrowed down or combined to form more concrete
solutions. This gives way to a final result, which solves specific user needs.
The following sections will explain how the Double Diamond was utilized in this
project.

2 Analysis
2.1
The Problem Space
The problem space of this project is very vast to begin with. Discovering novel ways
of visualizing personal data can be done in multiple ways, as current tools do not
provide specific enough solutions to support the therapy process.
A key starting point for this thesis is how to see and compare multiple phenomena
and gather insight about how they interact with each other. Thus, this thesis aims
to provide a proof-of-concept solution to this main problem. As such, this starting
point also represents the initial starting point of the Double Diamond process.
Taking this starting point, it is natural to follow the Double Diamond design method-
ology’s path and find multiple smaller issues, which could potentially be solved in
the project. The goal is to find many such problems, whose solutions would provide
patients, who suffer from mental health issues, with answers to questions about their
therapy progress they have not had before.
These problems are described in the
following section and referred to as use cases.
2.2
Use Cases
Defining many potential use cases for a future solution represents the divergent stage
of the problem space (also called the Discover stage). [Cou04] To discover such use
cases various ideas have been gathered from online sources such as the Personal Sci-
ence book [Wol+22] and show-and-tell sessions from the Quantified Self forum. [Sel23]
Furthermore, a key aspect of any self-tracking project is finding a so-called proxy
for a specific phenomenon one wishes to track. A proxy is ”a noticeable, variable,
and distinct phenomenon that stands in for the main phenomenon, but is easier to
measure”. [Wol+22, p. 32] For instance, instead of tracking tiredness on a subjective
scale, one could track when they yawn to discover patterns in their fatigue. Thus,
the following list represents all defined use cases, along with proxies for various phe-
nomena:
• Medication intake and symptom occurrence.
• Visualizing depression - how many times one lies down tired in bed. [NHS19]
• Visualizing depression - track positive phenomena, instead of negative, so one’s
depression does not worsen. One could track taking a shower, cleaning, doing
exercise, getting out of pyjamas. These positive activities could be compared
to medication intake.
• Anxiety - one could visualize anxiety by tracking nervousness or restlessness.
Another option is recording when one gets ”the sense of impending danger,
==PAGE:23==
2.2 Use Cases

panic or doom”. [Cli18] Finally, a proxy for anxiety could be sweating when
no heat or movement is present. This could again be compared to medication
intake.
• Chronic fatigue and chronic pain. [Wol+22, p. 12]
• Side effects of medications.
• Acne - number of pimples compared to acne treatment use/ medication. This
can also be compared to proxies of depression, as acne is well-known to be the
culprit of depression. [Der23]
• Depression compared to how the person feels at their workplace. [Wol+22]
• Sleep compared to depression. [Wol+22, p. 27]
• Deviation from average bedtime.
• Fatigue (microsleep [Con21] or yawn count with timestamp) compared to sleep

in the eyes, or inability to get to work at the start of the day”. [Wol+22, p. 30]
• Time spent on social media compared to depression or the feeling of ”sadness
or hopelessness”. [Hai23]
• Crying, reason for crying, duration, and intensity. [Wol+22, p. 46]
After defining all of these use cases, the project proceeds with the ones that would
result in the most different visualization prototypes. Thus, this narrows down the
Problem space and starts the exploration of the Solution space.

3.4
Summary & Reflections
On the one hand, the chosen use cases and the design of the visualizations reflect how
to visualize multiple phenomena together to support a person undergoing therapy
==PAGE:47==
3.4 Summary & Reflections
37
or treatment. On the other hand, they reflect a more general and abstract way of
visualizing self-tracking data.
This means that the use cases and their visualizations have also been chosen in a
way that they complement each other based on the type of self-tracking data that is
visualized. For instance, in the first visualization microsleep is used as a proxy for
fatigue, whereas in the second visualization a proxy for depression is the feeling of
”sadness or hopelessness”.
At the end of the day, they just represent instances of data coming from the One-
Button tracker. Thus, their use can be generalized for other instances of One-Button
tracker data, which utilizes the 1-press protocol. As mentioned in Chapter 2, this
could be medication intake, side effects, symptom occurrence, lying down tired in
bed, taking a shower, cleaning, doing exercise, getting out of pyjamas, yawning, or
getting a burning sensation in the eyes. In general, any proxy for a phenomena could
be tracked and visualized by the provided solution, as long as a single timestamp is
enough to register it (the 1-press protocol of the One-Button tracker).
The other type of time series the proposed design supports is a range of time - a
session with a start time and an end time. This can be seen in the first two visual-
izations, where sleep and social media use represent a time frame with specific start
times and end times.
The flexibility this approach provides is that other activities can be tracked instead,
but visualized with the same graphs. For instance, time spent working, socializing,
doing chores, or studying could represent how a person undergoing therapy is doing
when it comes to productivity or overall wellbeing.
Furthermore, the third visualization could also represent such activities where the
duration of the activity over time is of importance, e.g. doing physical exercise.
Finally, such an overall approach - to start with a wide variety of use cases and
then narrow them down to what is key based on user needs and complementing data
types, allows for a focused solution that would give freedom for expansion in the
future for other uses.

CHAPTER 4
Implementation
4.1
Platform
To implement the described design in Chapter 3, the Observable framework [Inc23b]
was used, alongside the Vega-Lite visualization framework. [Veg23]
This solution has many advantages both for the developer and for the user. Case
in point is its easy code maintenance, thanks to code being organized in order-
independent cells. This means that code cells can reference each other no matter
their order in the notebook.
On the other hand, the Observable framework provides a user-friendly interface,
which is easily accessible through web browsers. There are, however, some limita-
tions to the Observable framework. They are further discussed in Chapter 4.4 and
7.4.

CHAPTER 5
Evaluation
The evaluation of the proposed concept and solution was conducted through various
user testing methods. The following sections provide information in detail about the
testing target group selection, along with which testing methods were chosen for the
evaluation and why.
5.1
Methods
5.1.1
Target Group
The target group for the user tests is people who are not very experienced with data
visualizations. This is done because patients and their psychologists are not necessar-
ily well-versed in data visualization. Therefore, the goal for the proposed solution is
to be as simple and clear as possible to people without technical knowledge.
Five people were recruited, whose profile mostly overlaps with the desired target
group profile. Each of them participated in short interview-like test sessions, as ex-
plained in the following subsections.
5.1.2
A/B Testing
A/B testing was used to find out which of the final prototypes in the implementation
stage are the most user-friendly. This method was the preferred one in this stage
of the project, as it was the final stage of the Double Diamond process, so multiple
versions of the same visualization had to be compared, contrasted, and combined.
5.1.3
Think Aloud Testing
The Think Aloud testing methodology was chosen for the final clinical tool prototype,
as it can clearly indicate whether there is confusion when a person is given a task to
complete using the provided prototype. The Think Aloud testing was conducted as
short user interviews.
==PAGE:56==
46

5.2
Procedures
5.2.1
A/B Testing
During the A/B testing, participants were shown all described iterations of the visu-
alizations for the first two use cases - Sleep vs. Microsleep and Social Media Use vs.
Depression, as explained in Chapter 3. They were told what kind of data each visu-
alization represents and asked to explain which version does the best job at clearly
displaying their data. All five participants agreed on which type of sleep visualization
is best, while four out of five agreed on which social media use visualization is best.
The chosen visualizations are shown in Chapter 3.
5.2.2
Think Aloud Testing
Finally, during the Think Aloud testing, participants were shown the final design pro-
totype, as explained in Chapter 3. It was explained to them how the system works,
what the data collection process is, and they were given a task to share thoughts on
observations they see in the provided visualizations.
It was found out that three out of five participants, 60%, are not likely to enjoy
active social media tracking. Thus, they would prefer to passively get data about
their social media usage. This is addressed in Chapter 7.2.
It was also found out that the final visualization requires a bit more technical knowl-
edge to be understood, as two out of the five participants were not sure how the
moving average line reflected on their sleep duration changes over time. As already
mentioned, this visualization is still left in the prototype as a final optional one be-
cause more than half of all participants found it insightful.
Some users also shared that they would enjoy the option of getting their data au-
tomatically loaded in the visualizations. This is also further discussed in Chapter 7.4.
This evaluation is further discussed in Chapter 6.

CHAPTER 6
Results & Discussion
6.1
Results
During this project, a clinical personal informatics tool was designed and developed in
a notebook-like layout, which can support the therapy process of people with mental
health issues in an intuitive way.
The result of this project is proof-of-concept solution, which goes beyond the cur-
rently used mainstream solutions for self-tracking. Thus, the proposed solution has
the potential to be further developed in the future and lead to innovative ways of
supporting therapy.
6.2
Discussion
This section will discuss the evaluation of the proposed solution.
The conducted evaluation gives useful knowledge about the proposed solution and
its user-friendliness. Thus, it ensures that the final prototype, even though it is just
a proof of concept, can provide patients, who struggle with mental health, with an
optimized user experience. However, the evaluation has some limitations.
Given that the project had to be conducted within a limited time period, some parts
of the evaluation process had to be adapted according to these constraints. Thus,
given more time, other types of evaluation would also be conducted.
For instance, surveys can give very useful information about specific features of
a product. However, they are more useful when many participants are recruited to
fill them in. This can be a very time-consuming process, which could also require
other resources, so it was left out during this project.
Moreover, one very precise way to measure the effectiveness of a proposed system
is conducting an experiment. The experiment could compare different versions of
prototypes, similar to an A/B test, however, the results can be precisely measured.
For example, one could measure the time needed to find a specific observation or con-
clude what trends there are in the data. This way various graphs can be compared
==PAGE:58==
48
6 Results & Discussion
and the best one can be found in a less subjective way. As experiments often require
more than one person to be carried out and are very time-consuming to plan, execute,
and analyze, an experiment was not conducted during this project.
Finally, an alternative for a better evaluation could also be a workshop that is
planned with participants who fall into the target user group of the proposed solu-
tion. This can already be done during the initial stages of the project, as it could
gather various useful ideas about use cases and user needs. This means that in the
later stages of the project, some aspects will already be approved by the users and
would not need to be evaluated. However, good quality of the solution would be
ensured and user engagement would be higher, which would could potentially lead to
higher motivation in using the proposed final solution. A workshop is, however, very
time-consuming to plan and usually requires other resources such as payment for the
participants. It was therefore not conducted during this project.

CHAPTER 7
Future Work
Given that the proposed solution is a proof of concept by nature, various aspects of
it can be improved in the future. The following sections present possible future work,
which could improve the proposed solution.
7.1
Introduce Interventions in the Therapy Process
A key aspect of the therapy process - the work a patient does together with a psy-
chologist/ therapist, is at-home exercises and activities. These can vary from mental
to physical ones and are referred to as interventions in the therapy process.
Thus, a future improvement of the proposed solution could be the introduction of
interventions in the therapy process, which can be self-tracked by the patient. This
would represent a new use case in the final clinical tool and it would help gather more
insight into the patient’s progress.
Even though this use case could potentially be very useful, it was not prioritized
in during the period of this project. This was done because the exploration of the
relation and connection between multiple phenomena is considered more important
about understanding the progress of a person with mental health issues who goes to
therapy or receives treatment.
7.2
Passive Screen Time Data Collection
One aspect that could be improved is part of the data collection process, which cur-
rently requires the user to actively track their activities. As mentioned in Chapter
5, most users would prefer to get passively-tracked social media usage data, as it can
be quite cumbersome and error-prone to track time intervals with the One-Button
tracker. Thus, a proposed future improvement to this could be integration with the
Apple Screen Time API for all Apple devices. [Inc22]
The Screen Time API gives developers access to the user’s device usage data, i.e.
when and how much their device is being used. This way usage data for social media

user’s self-tracking discovery journey.
Such an integration also has some risks with regard to inaccuracy, e.g. if the user
leaves their phone on a table while a social media app is open. Even though this could
lead to some inaccuracy, it would still be far more accurate than active self-tracking,
where both a start and an end time have to be registered.

CHAPTER 8
Conclusion
Over the course of this master’s thesis project, a clinical personal informatics tool
to support therapy was designed, developed, and tested. By utilizing the Double
Diamond design methodology [Cou04], the project explores a vast spectrum of use
cases and focuses on specific ones from them.
In the later stages of the project, a wide variety of data visualizations are designed,
narrowed down, and combined into a proof-of-concept solution. The solution pro-
vides data visualizations in a notebook layout, where their current use cases are the
following:
• Microsleep [Con21] compared to sleep data.
• Time spent on social media compared to depression or the feeling of ”sadness
or hopelessness”. [Hai23]
• Moving average of a patient’s sleep data.
These use cases are chosen in a way that the solution can currently provide useful in-
sights into a patient’s therapy by simultaneously visualizing multiple disease-related
phenomena on charts. On the other hand, the use cases are chosen to result in data
visualizations that support various self-tracking data types, and thus can be utilized
for other use cases with similar data types in the future.
Finally, given the results from the project’s evaluation, it can be concluded that
while some future improvements could be done, this project can lead to novel ways of
visualizing self-tracking data and supporting the therapy process in a clinical setting.
