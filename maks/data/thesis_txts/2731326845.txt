Abstract
In the last couple of decades, we have witnessed the emergence of new applica-
tions and services, and the increment of connected devices at a vertiginous pace. As
implementations have grown in complexity, their requirements in terms of latency,
bandwidth, security and robustness have grown, too. And vendors and operators
are challenged to find solutions to cater for these ever-growing demands. In such
scenario, network monitoring has become a crucial element in computer networks.
In-band Network Telemetry (INT), a form of network telemetry, is a framework that
provides a way to generate and export detailed information about the network status
by means of the data plane. Telemetry instructions and/or data are attached to data
packets, reducing overhead in the control plane.
The aim of this thesis is to build a complete monitoring INT system using software
switches and visualization tools. For that, we make use of the SDN technology and
its programmable network profile, which has since the last three or four lustrums
been announcing and introducing a change of paradigm in networks archtecture and
operation.
Keywords: In-band Network Telemetry (INT), Programming Protocol-Independent
Packet Processors (P4), Software-Defined Networking (SDN)

Introduction
In today’s networks, the requirements in terms of latency, packet loss, security, etc.,
are continuously increasing, and being able to identify and fix any hypothetical prob-
lems is sometimes crucial to meet such demands. It becomes very practical to use
tools that allow us to monitor the network in order to be able identify where failures
are bound to happen and detect attacks and other threads quickly enough so that a
solution can be generated dynamically and specifically for that situation.
In Traditional Networking, where devices have dedicated hardware and a fixed func-
tionality, scalability and re-configurability are an issue. In the last ten years, a new
networking technique has emegerged: Software-Defined Networking (SDN). SDN
is a netowrk monitoring and managing approach that means the separation of the
control (routing) plane and the data (forwarding) plane. With SDN, devices in the
network become programmable and unlinked to services and applications, and the
network becomes centralized and orchestrated by one or several network controllers.
1.1
Objective
The aim of this thesis is to build a complete monitoring INT system using digital
switches and visualization tools.

2.1
Network Monitoring
Since some of the services that operators offer require knowing the behaviour and
the status of the applications and devices that are being used, it is essential to have
a mechanism to monitor and be able to modify the behavior of networks. After all,
not being able to tell if things are working as intended would only lead to disaster.
With network monitoring, it is possible to apply management policies to a network,
for instance Quality of Service (QoS) [1] and traffic engineering [2].
Figure 2.1 illustrates the stages and classification of network measurement meth-
ods over time [3]. These are: traditional network measurement, software-defined
measurement that came with the birth of software-defined networks, and network
telemetry that came with the emergence of the Programmable Data Plane (PDP) tech-
nology. In turn, RFC 7799 [4] classifies traditional network measurements in active,
passive, or hybrid. While active measurement triggers events to see the network’s
reaction, which could be something as simple as pinging [5], passive measurements
obtain information about the performance of the network by observing the system,
and looking for the raise certain events to report.
Although traditional network measurement is simple, it does not provide the accu-
racy required by nowadays’ systems, which leads to research on new methodologies.
2.2
SDN
The concept of SDN is widely spreading in many telecommunication fields, even to
areas like the Internet of Things and Smart Cities [6, 7]. According to IHS Markit’s
survey, [8], service providers look to automate and partially virtualize their networks,
where around 75% of networking carriers have already deployed or will deploy SDN,
and the totality of them plan to implement and deploy it at some point in the future.

2.3
In-band Network Telemetry
The term In-band Network Telemetry is widely used to talk about not only the P4
language-based INT [9] but also about other representative schemes[3]. Currently,
research on data plane in-band network telemetry includes In-band Network Teleme-
try, In situ Operations, Administration, and Maintenance (IOAM) [10], Alternate
Marking Performance Measurement (AM-PM) [11] and Active network telemetry
(ANT).
Although in-band network telemetry was initially thought for network monitoring,
research has extended its utility and is now applied to so many aspects of network-
ing that applications are now divided into performance telemetry applications and
functional telemetry applications.
As for performance telemetry applications, the research focuses on several topics:
as examples, CAPEST [12] is designed to provide an offload of network capacity
and measure the available bandwidth in the existing INT solution, having a low
impact in the network and good precision. Pingmesh [13] and EverFlow [14] are
two applications designed to detect irregularities in packet delay in data centers.
In regards to function telemetry applications, some examples of research topics are
NetVision [15] for routing loop detection, CherryPick [16] for path verification,
HPCC [17] for load balancing in data centers, or RCP* [18] for congestion control.

3.1
Software-Defined Networking (SDN)
Different from Traditional Networking, the SDN technology provides a new way to
interact with the devices in the network. In the past, network appliances were hard-
ware elements, typically custom, fixed-function application-specific integrated cir-
cuits (ASICs) and Field-programmable gate arrays (FGPAs) that were ruled by proto-
cols and needed to be (re)configured one by one. In software-defined networks, any
intelligence or network knowledge is now removed from the switches and routers,
and any decision-making capabilities become part of a centralized, software-based
control plane. Figure 3.1 illustrates these two types of networking. In legacy equip-
ment, control and data planes live within the hardware. In contrast, in SDN, the
control plane is stripped from the devices and moved to a controller that dictates
over all the SDN-capable network elements.
When we talk about SDN, we talk of a top-down approach. [20] Figure 3.2 shows
a diagram of the concept of SDN. The architecture consists of three dimensions
or planes: the data, control and application planes. Two interfaces are used for
intercommunication between these planes:
– Southbound interface (SBI): Used for intercommunication between the con-
troller (control plane) and the network agents (data plane). One of the most
well-known protocols used in this interface is OpenFlow, which is open source

and whose first release was in 2009 and was latest updated in 2015 [21], stan-
dardized by the Open Networking Foundation1
– Northbound interface (NBI): It is the way applications and network admin-
istrators have to operate the controller and retrieve data from it. Example of
these applications can be routing algorithms or load balancers. A REST [22]
API is commonly used for this communication.

3.2
P4 programming language
The Programming Protocol-Independent Packet Processors (P4) [23] is a project
originally started by P4.org in 2014, which describes a way to establish the be-
haviour of, formerly, only programmable switches and, currently, a wide variety
1Open Networking Foundation: https://opennetworking.org/
6
==PAGE:18==
of P4-capable devices, called P4 targets, e.g., NICs, routers, FGPAs, and other net-
work appliances, in the data plane. This project provides a programming language,
also called P4 formerly and now called P416 in its latest version. In terms of syntax,
P4 is similar to the C programming language, but, instead of being a general purpose
language, it is domain-specific and exclusively intended for packet-forwarding ap-
plications.
In a traditional switch, the data plane functionalities are fixed and defined by the
manufacturer, and the data plane objects such as management tables are imposed
by the control plane. In contrast, a programmable switch has no built-in knowledge
about how to process packets, the protocols that will be used or about management
tables and similar objects; it fully relies on P4 programs to acquire such information
and capabilities. Figure 3.3 illustrates this.
Figure 3.3: Traditional vs. programmable switches. [23]
The simplest p4 program must at least contain the following sections:
– Headers: The different headers that ban be parsed and processed are defined
here. For each header, the description of its fields (size, position) is also pro-
vided.
– Parsers: Description of how to identify each header in a packet. It imple-
ments a state machine that follows a switch-case logical flow in each parsing
state.
– Tables: Packet-processing instructions depicted in form of match-action ta-
bles. In other words, certain actions will be executed when certain fields in a
header meet certain conditions.
– Actions: They can be seen as functions that are composed of primitive actions
provided by P4 and that are executed when packet-procession is happening.
Some of the input data may be provided by the control plane.
7
==PAGE:19==
– Control Programs: In it is provided the order in which tables are applied to
a packet.
P416 defines P4’s architecture model in the form of programmable blocks, includ-
ing their dataplane interfaces. Any architecture model comprises the description of
the different function blocks of a target and is specified by the target manufacturer.
Some of these functions blocks may be fixed, and the rest will be programmable.
Note that a P4 program is not expected to work on any P4-capable device but on
only those that are compliant with the architecture model it was developed for.
The P4.org Architecture Working Group defines the Portable Switch Architecture
(PSA) in [24], which describes a scheme of 6 fixed programmable blocks and two
fixed-function blocks for programmable switches. Figure 3.4 depicts the Portable
Switch Pipeline as defined by P4.org, where the ones in white are the ones that can
be defined with P4 and the ones in grey are device-dependant.
Figure 3.4: PSA. (P4.org)
BMv2 [25] is the second version of the Behavioral Model (BM) for P4. BMv2
provides a platform for running P4 experiments by implementing a software switch
and without the need of using dedicated hardware. BMv2’s architectural model is
called V1 Model and is defined in [26].
3.2.1
P4Runtime
The P4Runtime [27] specification offers a way for the control plane to operate on
the data plane of devices running P4 programs at runtime. After initialization time,
it is not possible to interact with a programmable P4 device or read/write elements
like tables and counters. But P4Runtime allows us to do it via an API. Such API
is available and provided in the P4Runtime standard [27]. It is required that P4
programs are written in P416 in order to be compatible with P4Runtime.

3.3
In-band Network Telemetry (INT)
In-band Network Telemetry (INT) is a network monitoring framework that provides
a way to generate and export detailed information about the network status by means
of the data plane. Monitoring instructions can be embedded to business data packets
to indicate the network information that must be gathered, which will be sent to a
monitoring system in the form of telemetry reports.
The development of both the specification for INT [9] and for telemetry reports [28]
are a task of the P4 Applications Working group. The initial release for the INT
specification took place in 2015, followed by the release of versions 0.5, 1.0 and 2.0
and 2.1 in 2017, 2018 and 2020, respectively. As for telemetry reports, versions 0.5,
1.0 and 2.0 were released in 2017, 2018 and 2020, respectively.
3.3.1
INT Terms
Following is a list of relevant terminology from INT’s specification:
– INT Source: Based on a configured flow watchlist, this trusted entity cre-
ates and inserts INT Headers selected packets before forwarding it to the next
entity.
– INT Transit Hop: If indicated in the INT Instructions of the packet, this
trusted entity collects network metadata. Depending on the INT operation
mode, the collected information will be directly sent to the monitoring system
or added to the INT Header in the received packet to be forwarded to the next
node.
– INT Sink: Trusted entity in charge of removing the INT Header from packets
so that the telemetry collection process is transparent to other layers and/or
13
==PAGE:25==
domains.
– Monitoring System: Entity that collects telemetry data received from INT-
capable devices.
– INT Header: Packet header where INT information is stored. A more de-
tailed description is given in 3.3.5.
– INT Packet: Simply a packet that contains an INT Header.
– INT Metadata: Telemetry data inserted into an INT Header either by a INT
Source or an INT Transit Hop. A more detailed description is given in 3.3.5.
– INT Instructions: Indicates what type of INT Metadata should be collected
at each INT node. Depending on the INT operation mode, these instructions
are written in an INT Header or set on each INT node’s flow watchlist.
– Flow Watchlist: In it is indicated what INT Instructions should be inserted
or applied for every flow that matches one of its entries.
– INT Domain: A set of INT nodes under the same administration. Interop-
erability must be assured between said nodes even if they’re from different
vendors in a same domain.
3.3.2
INT Modes of Operation
As per the latest version of the standard (2.1), three modes of operation are defined:
– INT-XD (eXport Data), or Postcard
When an event occurs upon packet arrival, telemetry is directly sent to the
monitoring system without modifying the original packet. The Flow Watch-
lists in the INT Nodes contain the INT instructions for exporting the right INT
metadata for each flow. Hence, packets are not modified at any time in this
mode. This is shown in Figure 3.5.
– INT-MX (eMbed instruct(X)ions)
The INT Source firstly embeds the INT Instructions into the packet header.
Then, the INT Source, Transit Hop and Sink send their gathered INT metadata
following these instructions to the monitoring system. Finally, the INT Sin
remove the INT Instructions before delivering the packet to the receiver. This
is shown in Figure 3.6. In this mode, packets are slightly modified; however,
their size does not grow regardless of how many nodes the packets traverse.
– INT-MD (eMbed Data), or Classic INT
Also known as Hop-by-Hop INT, in this mode of operation both INT Instruc-
tions and INT Metadata are inserted into the original packet. Firstly, the INT
Source adds instructions; second, INT Source, all INT Transit Hops and INT
Sink add their own telemetry. At the end, the INT Sink additionally is in
charge of stripping the instructions from the packet and sending the remain-

der INT metadata to the monitoring system. This process is shown in Figure
3.7. In, this mode, packets are modified the most and their size increase with
each hop (up to a point).

Data collection
For the monitoring part of the project, the concept of eBPF is studied.
Extended Berkeley Packet Filter (eBPF) [29], originated after Berkeley Packet Filter
(BPF) is an execution environment inside the Linux kernel that allows to detect and
react to events at the kernel level. eBPF permits the execution of programs in kernel
1https://github.com/ederollora/P4-demo-INT-IDA
21
==PAGE:33==
space with the safety of not having to compromise the overall system due to having
a restricted and isolated execution environment: the use of well-defined APIs being
the only way for eBPF to interact with the kernel minimize any possible security
breach. The eBPF architecture/workflow is depicted in Figure 4.2 and described as
follows:
eBPF programs are mainly written in restricted C. After compilation, an Executable
and Linkable Format (ELF) object containing the translated eBPF instructions is
generated. This object is loaded onto the kernel via a specific systemcall, and the
verifier will study it, in other words, it will analyze all the possible paths the program
might take and decide if it’s safe to execute inside the kernel. A just-in-time (JIT)
compiler then translates the ebpf bytecode into machine instructions to eventually
be attached to a what is called a hook, e.g., XDP or TC [30].
The way to communicate with eBPF programs from user space is through maps,
which are a collection of key-value pairs where data can be read and written from
both user and kernel space.
Figure 4.2: eBPF architecture. [31]
The collector chosen for the collection of our data is a open-source one called INT-
Collector2 [32]. It is a high-performance INT collector that reads telemetry reports
coming from the network. Due to the limitation in length of the eBPF programs, the
collector is divided in two: a fast path and a normal path. The fast path make use of
Linux’s eXpress Data Path (XDP), which allows to process network packets at high
rates by bypassing most of the operating system networking stack since it is at the
lowest layer. The normal path is written in Python instead of C, and is in charge of
processing events and storing metrics.

4.4
Data storage
Prometheus3 is an open-source tool for monitoring from containerized environments
to bare servers running applications in them. It also provides an alerting feature [33].
Initially started by SoundCloud 4, it is now a standalone source project and declared
graduated by the Cloud Native Computing Foundation (CNCF). This means that
Prometheus is a project considered stable and ready to be used in production envi-
ronments.
In this context of Prometheus, we can define metrics, or units of monitoring, as
the different parameters that are to be monitored for a specific target. Metrics are
quantitative data displayed in a human-readable format (text) by Prometheus and
can have either the attribute HELP or TYPE. HELP simply provides a description
of what the metric is, and TYPE can mean the metric is one of the following:
– Counter: How many times something happened, for example, the number of
packets dropped. This value can only increase.
– Gauge: For values that can increase or decrease, for example, the percentage
of CPU used.
– Histogram: This metric is sectioned in buckets for each selected interval and
the count will only increase for that bucket just like in a conventional his-
togram.
The metrics that are relevant to monitor will depend on the target. A scrape target is
any external running job that Prometheus can pull metrics from. Example of targets
could be operating systems (Windows, Linux...), where relevant units of monitoring
could be memory usage and disk space usage, or applications, where relevant units
of monitoring could be number of exceptions or request duration. Prometheus of-
fers plenty of client libraries to expose metrics to it, but it is also possible to build
one’s own exporter to translate a third-party application’s metrics into metrics that
Prometheus will be able to scrape. [34]
In a Prometheus server, which is the main component of Prometheus and does the
actual monitoring work, three parts are identified:
– Data Retrieval Worker: Pulls metrics from targets and pushes them into the
storage database.
– Storage (Time Series Database): Stores metrics data.
– Web server: Receives PromQL queries over HTTP for the data that was
stored and displays them on the Prometheus UI or some other data visual-
