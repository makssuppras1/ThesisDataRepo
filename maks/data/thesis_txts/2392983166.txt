Abstract
This thesis deals with automating the process of nding integral
distinguishers using the division property.
We reduce the problem
to the SAT-problem and then use a SAT-solver to nd distinguishers.
Solvatore is a tool we implemented, which translates a high-level cipher
description to DIMACS and then solves it using the CryptoMiniSAT
5.0.1 SAT-solver.
Using our tool we found improved distinguishers
for DES, LEA, HIGHT, Speck, MANTIS, and PRINCE. The tool
is able to prove that no distinguishers exists and we present security
bounds for several ciphers.

Introduction
Cryptography is the study of communication in the presence of an adversary.
It therefore deals with how to send and receive messages without a third-
party being able to read or change it.
The two goals are referred to as
secrecy and authenticity. In our modern world, it is essential to be able to
communicate privately with your bank on the Internet, or any other website
for which you have to enter a password. Cryptographic algorithms also nd
applications in everything from phones to car keys and medical devices, and
for storing passwords in huge databases.
There are a lot of dierent algorithms that have been optimised for dif-
ferent purposes. Figuring out which ciphers are secure to use is therefore
an important part of cryptography. We do this by trying to nd weaknesses
in the algorithms, a process which is called cryptanalysis. Cryptanalysis is
also important when designing ciphers, as it is much easier to design a good
cipher if you know which properties make it insecure.
Integral attacks are a powerful class of structural attacks on block ci-
phers and other symmetric primitives. It was rst proposed to attack the
block cipher Square [DKR97], and was later generalized by Knudsen and
Wagner [KW02]. The idea of integral attacks is to track the propagation of
four properties for a set of texts. The properties are CONSTANT (C), the
bits have the same value for all texts; ALL (A), the bits take all dierent
combinations; BALANCED (B), the XOR of the bits is 0; and UNKNOWN
(U). If after r rounds of encryption some bits have the B property we say
that the cipher has a an r-round distinguisher, which can be used to attack
the cipher.
Higher-order dierentials, which are the discrete equivalent of the deriva-
tive, were proposed for cryptanalysis by Lai [Lai94].
Knudsen [Knu95]
showed that they could be eectively used to attack ciphers.
Like regu-
lar derivatives, a dierential reduces the degree of the function by at least
one. This can be exploited by the cryptanalyst since the d-order dierential
will give 0 if the degree of the function is at most d. Another way to look at
the integral property is thus to consider it as a higher-order dierential that
evaluates to 0 for all points in the specied bits.
The division property is a generalization of the integral property pro-
posed by Todo [Tod15b]. The generalization makes it easier to nd integral
distinguishers and it was successfully used to nd the rst attack on full
Misty [Tod15a]. Todo and Morii later extended the idea to bit based de-
signs [TM16]. This approach nds better destinguishers but it comes at a
higher computational cost.
To alleviate this problem several approaches have been proposed for au-
tomating the task. The rst one published is to reduce the problem to a
Mixed Integer Linear Programming (MILP) problem and then use an MILP-
solver to solve it [XZBL16].
4
==PAGE:6==
Our approach is to reduce the problem to a SAT-problem and use a SAT-
solver to solve it. The general SAT-problem is known to be NP-complete
and therefore hard to solve. When we model the propagation of the division
property through a cipher we get a structured SAT-problem. A SAT-solver
can use this structure to nd a solution rather fast. We have implemented a
tool which handles the conversion from cipher description to SAT-problem
and it is called Solvatore.
Both the SAT- and MILP-problem are NP-complete, but many industrial
problems require solutions to these problems. Therefore there exists solvers
which can take advantage of the properties of a particular problem and nd
a solution using various heuristics.

Symmetric Cryptography
In Cryptography we usually consider two parties called Alice and Bob, who
communicate, and the adversary Eve, who eavesdrops on the conversation
between Alice and Bob. In symmetric cryptography Alice and Bob share
a key, which is used for both encryption and decryption, as opposed to
public-key cryptography, where dierent keys are used for encryption and
decryption.
In this section we will give a brief introduction to some important con-
cepts in symmetric cryptography. We will look at what a block cipher is,
especially the concept of an ideal block cipher, which will be important when
we discuss distinguishers. We will also discuss adversarial models and generic
attacks.
2.1
Block Ciphers
A block cipher is a specic type of encryption algorithm, which encrypts the
message in blocks of a xed length. Let P be the set of plaintexts, C be
the set of ciphertexts, and K be the set of keys. An encryption algorithm is

Denition 1. Ideal n-bit Cipher
An Ideal n-bit cipher picks an n-bit bijection uniformly at random from the
set of all n-bit bijection for each k ∈K.
In practise an ideal cipher is not possible due to the number of permu-
tations. There are 2n! permutations, and just 2m keys, it is therefore not

guishers, which we will discuss in Section 2.3.
Most of the algorithms we will consider are so called key-alternating
ciphers. These ciphers consists of alternating a key independent function, f,

In a brute-force attack, the attacker tries all possible keys to see which
keys give the correct encryption of a plaintext. The time complexity of this
attack is 2m since this is the number of keys. The data complexity is ⌈m/n⌉.
In a dictionary attack, the attacker collects the encryption of all plain-
texts and store them in a dictionary. When a new ciphertext appears the
corresponding plaintext can simply be looked-up in the dictionary.
This
attack has a negligible time complexity but a data complexity of 2n.
2.3
Distinguishers and Key Recovery Attacks
First we need to introduce the wrong key randomisation hypothesis, which is
the hypothesis that decrypting one round with a wrong key gives us a state
uniformly at random. It is therefore unlikely to get the correct state with a
wrong key.
A distinguisher is any property we can use to distinguish a concrete n-bit
block cipher from an ideal n-bit block cipher. In integral cryptanalysis the
property we consider is whether some bits are balanced, i.e., whether the
sum is zero.
An r−1 round integral distinguisher can be used to mount a key-recovery
attack on an r round key-alternating cipher in a straight forward way. We
guess the bits of the last round key that we need to decrypt the last round
for the balanced bits. We can then check if the bits are balanced. Due to
the wrong key randomisation hypothesis a wrong key guess will be unlikely
to make the bits balanced. If we need to guess less bits than are in the key,
we gain an advantage compared to brute-force. An example of an integral
property and how it can be used to mount a key-recovery attack will be given
in Section 3.1.
In this thesis we will also consider other symmetric primitives like stream
ciphers and hash functions. However for these it is not as easy to go from
distinguisher to key-recovery attack.

Integral and Division Property
Integral attacks are a powerful class of structural attacks on block ciphers.
It was rst proposed to attack the block cipher Square [DKR97], and was
later generalized by Knudsen and Wagner [KW02].
The division property was proposed by Todo [Tod15b] as a generalization
of the integral property. The division property has proved to be very e-
cient for nding distinguishers as Todo showed by attacking the full Misty1
[Tod15a]
This section will start by looking at the integral property and how it can
be used to mount an attack on a cipher. As an example we will consider the
classic 3-round integral property on AES. We will then see how the division
property generalizes this property and why using the division property is
stronger than the integral property.
3.1
Integral Property
We will consider texts as vectors with elements from a eld F2k with XOR
as addition. Integral attacks deal with sums over sets of texts. The idea is
to be able to predict the value of the sum of all elements in the set after a
certain number of rounds of encryption. With XOR as addition this value
is usually 0. To help us predict this value we dene four properties that the
set can have.
 CONSTANT(C)
 ALL(A)
 BALANCED(B)
 UNKNOWN(U)
CONSTANT means all texts have the same value at that position. ALL
means that all possible values for that position are in the set. BALANCED
means the sum of all the texts in the set is 0 for that position. UNKNOWN
means we don't know anything about the sum of the texts.
We will now look at the classic 3-round property of AES, shown in
Fig.
2, as an example to see how it works.
The Advanced Encryption
Standard (AES) is one of the most widely used ciphers today. It is a key-
alternating cipher with a block size of 128 bits arranged in a 4×4 matrix of a
byte each. The round function consist of four sub-functions: SubBytes(SB),
ShiftRows(SR), MixColumns(MC), and KeyAddition(KA). SB applies an 8-
bit bijective SBox to each cell of the state. SR shifts the row to the left by
0, 1, 2, 3 places respectively. MC multiplies each column of the state by a
matrix which is shown in 5.3. KA XORs the round key to the state.

We start with a set where one byte has the A property and the rest have
the C property.
First we need to consider the eect of the KA, but since the same key is
added to each text in the set, they cancel out when we take the sum over all
the texts. We will therefore ignore the key from now on.
The SBox of AES is a bijection, therefore any byte that has the A or
C property before SB will also have the property after SB. For a byte that
has the B property we only know that the sum over all texts is 0, since the
SBox is non-linear. We therefore don't know anything, i.e., the byte has the
U property.
SR does not aect which property each byte has. However it does aect
where the byte is placed. We therefore still need to consider it.
Last we need to consider the propagation through MC. We will consider
three cases for the columns in this example, all bytes are C, one byte is A
and the others are C, and all bytes are A. When all bytes are C, MC will
give the same result for all bytes, hence we still have C for all bytes. When
one byte has A, each byte of the output will be a linear combination of three
C bytes and one A byte. Therefore we get an output where all bytes are A.
When all the bytes are A, all the output bytes will be B due to the linearity
of MC.
By using these simple propagation rules we can get the 3-round integral
distinguisher shown in Fig. 2. The gure also shows that we cannot extend
it to cover more rounds. To mount a key recovery attack on 4-rounds, we
can use that there is no MC in the last round.
We can therefore guess
the last round key, decrypt the last round, and test if the property holds
independently for each byte. The data complexity for this attack is 28, and
the time complexity is 16 · 28 = 212. This can be extended to 6 rounds by
guessing 4 key-bytes in round 1, using the integral in round 2-4, guessing one
byte in round 5 and 4 bytes in round 6. This is a chosen plaintext attack,
because we chose our starting set.

rules are simple. For bit-based designs there are more elements to keep track

4
SAT-model
In this section we will show how to reduce the problem of nding valid
division trails to the SAT-problem which we can then solve using a SAT-
solver.
We will rst introduce the SAT-problem before we turn our attention to
how we go from the propagation rules to a SAT-problem we can solve. On
the way we will introduce CNF, DNF, ANF, and the DIMACS encoding.
4.1
The SAT-problem
The boolean satisability problem, which we will just refer to as the SAT-
problem, is one of the most well known NP-complete problems. In fact it
was the rst problem to be shown to be NP-complete.
A problem is in the complexity class NP if there exists a non-deterministic
polynomial-time algorithm. Informally this means that it is `easy' to check
whether a solution is correct. An NP-problem is NP-complete if it is at least
as hard as any other problem in NP, i.e., a polynomial-time algorithm for
any NP-complete problem would result in a polynomial-time algorithm for
all other NP problems. Informally NP-complete is the class of problems that
are hard to solve but where it is easy to check the solution. So far no one has
been able to nd a polynomial-time algorithm for any NP-complete problem
or prove that it does not exists. This is the famous P versus NP problem.
A consequence of this is that there are no ecient algorithms for solv-
ing the general SAT-problem.
Luckily most SAT-problems contain some
structure, which a SAT-solver can use to nd a solution. Our problem in
particular often contains a number of similar rounds and the same operations
applied on several parts of the state.
We will now give a denition of the SAT-problem.
Denition 6. The SAT-problem
Given a set of n variables X = {x1, x2, . . . , xn} and a set of m clauses
C = {c1, c2, . . . cm}, decide if there exists a truth assignment to all variables
in X such that all clauses in C evaluate to TRUE.
The standard for encoding a SAT-problem is known as DIMACS. Each
line encodes one clause, where the variables are encoded as integers separated
by spaces with −used for negation. The problem must have a special form,
called CNF, when we encode the problem in DIMACS. We will now introduce
CNF.
Conjunctive Normal Form(CNF)
In CNF we have a set of clauses
where each clause contains literals, disjunctions, and negations. The clauses
are connected by conjunctions.
In other words it is the AND of clauses
containing ORs and NOTs.

Solvatore
The idea of Solvatore is to model the propagation of the division property as
a SAT problem. The SAT problem can then be solved using a SAT solver.
Even though the general SAT problem is NP-complete SAT solvers are fairly
ecient for structured problems.
In Section 4 we showed how the propagation rules for copy, and, and
xor, can be translated to CNF, which we will now use to build more complex
functions.
5.1
SBoxes
Solvatore handles SBoxes in two steps, rst we nd the ANF and then we
convert it to CNF as shown in Section 4.3.
To apply an SBox we rst need to add it to the cipher description before
we can apply it. The way to do that is
s e l f . add_sbox (sbox_name ,
sbox )
s e l f . apply_sbox (sbox_name ,
input_bits ,
output_bits )
where sbox is a list of integers, sbox_name is a string that can be used
to specify which SBox to apply later.
The second and third inputs to
apply_sbox, input_bits and output_bits, are lists of either state or tem-
porary variables. They don't have to be the same length, e.g. for a cipher
like DES.
The reason for doing this in two steps is that we only need to calculate
the CNF once. The ANF and DNF are expensive to compute with algo-
rithm having time complexities of n2n−1 and 2n+m respectively. Calculating
the CNF just once can therefore result in a signicant improvement of the
eciency.

presents the number of clauses to describe a selection of SBoxes.

5.7
Provably None
We will now dene what provably none means in our setting. It is a term
that will appear a lot in Section 6, and therefore we will clarify what it means
and does not mean.
We use the term to describe the situation, where we have tried all possible
input division properties, where one bit is constant and the rest are active,
and have not been able to nd any distinguishers.
As a consequence of
Remark 1 we know that there are no distinguishers with less active bits,
thus we know that there cannot be any distinguishers that can be found
using the conventional bit-based division property.
It does not mean that there are no integral distinguishers in general, e.g.
see Todo and Morii [TM16]. There they show that we can also exploit if
we know the parity is 1 instead of just 0. Our tool does not take this into
account.
All this is of course based on the assumption that our model and imple-
mentations are correct.

6.2
Feistel Networks
A Feistel network is an iterated cipher, which in the simple case splits the
message in two parts. The right half is copied, the copy is passed through a
key-dependent function, f, and then XORed to the left half. The halves are
then swapped. A Feistel network with 3 rounds is shown in Figure 7a. The
rst example of such a structure is the Data Encryption Standard (DES),
for which we present our results in Section 6.2.2.
The Feistel network can be generalized to split the message in more than
two parts.
We now have more options for the f-function, since we can
use expanding and contracting functions.This shows an advantage of Feistel
networks over SPNs; that is the f-function is not required to be a bijection.
This increases the possible design space. An example of a Feistel network
with a contracting f-function and where the message is split in four parts is
shown in Figure 7b.
We also present results for the cipher Misty in Section 6.2.1, and the
NSA cipher SKIPJACK in Table 7.
As an example we show the implementation of SKIPJACK in Figure
8. This example also shows how we can handle ciphers with dierent round
functions. The SBox has been left out of the description.

6.3
ARX-based designs
ARX-based ciphers use modular Addition, Rotations, and XORs, hence the
name. The nonlinearity is provided by the modular addition and therefore
there is no S-box. Since the modular addition can be decomposed into a
relatively low number of XORs and ANDs as shown in Section 5.5, our tool
works quite fast for ARX-based designs.
Due to how ecient our tool is for ARX-based designs we show how we
can optimise the design of Speck in Section 6.3.1. We also looked at LEA,
HIGHT, the Belarusian standard BelT, and SPARX.
In Figure 9 we show an example of how we can implement ARX ciphers.
The example is an implementation of Speck. This shows how we can imple-
ment all the version of a cipher with minimal extra work compared to just
one version.

6.7
Evaluation of the Performance of the Tool
The performance of the tool depends on the size of the SAT-model, i.e.,
how many clauses are in the model. The number of clauses in the model
is determined by several factors: the number of state bits, the number of
rounds, and how complicated the round function is.
State Size
The state size impacts the performance in two ways, the size
of the model and how many bits we have to check for balancedness. In order
to prove that there are no distinguishers we also have to check whether we
can nd a distinguisher for all possible constant bits.
Number of Rounds
Assuming the round function is the same for all
rounds we have to add the same number of clauses for each round extra
we want to cover, and therefore it should take longer the more rounds we
want to cover. This is however not always the case. As an example consider
Midori64, where we can easily prove that there are no 8-round distinguish-
ers, but for 7 rounds it takes much longer to just check that there are no
distinguishers with bit 0 constant.
45
==PAGE:47==
Round Function
We will now focus on the round function. We decompose
the round function into XORs, ANDs, and COPYs, and then add the cor-
responding clauses to the model. Therefore lightweight designs, with small
states and round functions that can be decomposed into few logic gates,
should be fairly well suited for our tool to handle.
Non-linear layer
For the non-linear part of the cipher there are two basic
approaches we have considered for our tool, SBoxes and modular addition.
SBoxes are converted to ANF and then converted to CNF. How ecient
our model is, thus depends on the ANF of the SBox. In particular it depends
on the size and the degree of the SBox. The current way SBoxes are handled
is not optimal. The number of clauses used to describe a large SBox, e.g. the
7- and 9-bit SBoxes of Misty, means it is too slow to run in any reasonable
time. As we can see from Table 16 SBoxes with more than 4 input bits need
a lot of clauses to be described. The S7 and S9 of Misty have degree 3 and
2 respectively.
It is worth noting that in the example for the Present SBox in Section
4.3, we need 23 XORs and 23 ANDs to describe the SBox, or an average of
5.75 per output bit for each of the operations.
Another approach to add non-linearity to a cipher is using modular ad-
dition with the modulus 2n, where n is the number of bits of the addends.
Modular addition is decomposed into 4 XORs and 2 ANDs per output bit,
and therefore our tool handles modular addition quite well.
We can also compare the number of clauses per output bit for the SBoxes,
which is also available in Table 16.
Since XOR and AND adds 6 and 5
clauses to the model respectively, we need 34 clauses per output bit to encode
modular addition, however in the actual implementation we do 3n copies,
4(n −1) + 1 XORs, and 2(n −1) + 1 ANDs. The number of clauses is just a
part of what makes a model hard to solve. For modular addition the output
bits are all updated in the same way, where for SBoxes the ANFs can look
very dierent from bit to bit as we saw for the Present SBox.
Linear layer
There are several ways to design linear layers.
Our tool
provides support for permutations and matrix multiplication.
For permutations and rotations the shufflewords function can be used,
which does not add any clauses to the model. This can easily be used for
Feistel constructions and shift registers.
How ecient MixColumns is, depends heavily on the matrix. One would
assume that our tool would perform better with a matrix for a lightweight
cipher, however there are some things to consider. The recursively dened
matrices, used in e.g. LED and PHOTON, have to be applied several times.
Due to the general nature of the apply_MC function, several applications will
provide some overhead. Multiplying out the matrix does not help as the

7
Conclusion
In this thesis we have shown how to automate the process of nding division
properties by reducing the propagation rules to the SAT-problem. We have
implemented a tool that translates a high-level description of a cipher into
DIMACS, which can then be solved with the SAT-solver. To show the versa-
tility of the tool we then implemented a wide range of ciphers. We have seen
how to use the tool for nding good distinguishers and optimising a cipher
design.
The tool performs well for ciphers with simple round functions, which
turns into a SAT-model with few clauses. It is therefore most suited for
lightweight ciphers with relatively small states. The specics of the cipher is
important as we have seen with the MixColumns of LED and PHOTON.
Future Work:
There are several ways the tool could be improved.
Bit-based Division Property with 3 Subsets
There are certain types
of integral distinguishers that cannot be found with the conventional bit-
based division property. By introducing the bit-based division property with
3 subsets this problem can be solved. Our tool could be improved by taking
this into account, however the translation from division property to SAT-
problem would not be as easy.
Handling of SBoxes
We think a dierent way of handling the SBoxes
could lead to a better description of the ciphers. Especially large SBoxes
adds a lot of clauses to our model, which makes it harder for the solver. A
description with fewer clauses could be reached by enumerating the division
trails through the SBox, however this will not be as easy to do generally.
Improvements to MixColumns
The apply_MC function is general and
thus can be used for many ciphers, however this comes at a performance cost.
An example of this are the serial matrices of LED and PHOTON, which is
why we implemented a separate function for this specic type of matrix. An
apply_MC function that could better take advantage of the specic structure
of other types of matrices could improve the performance of the tool.
SAT-solver
The tool currently uses the CryptoMiniSat 5.0.1 SAT-solver
[CRY]. A dierent SAT-solver might provide better performance, however
CryptoMiniSat 5.0.1 is available as free open source software, which means it
is easier for others to get our tool to work. Most of what the tool does is the
translation from high-level cipher description to DIMACS. Since DIMACS
is the standard way to describe a SAT-problem it should be straight forward
to apply a dierent solver.
