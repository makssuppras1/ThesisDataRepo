Abstract
The growth of converter­driven assets, such as motors in industrial systems, presents op­
portunities for enabling distributed flexibility in future energy systems. Conventional con­
trol and modeling strategies are based on deterministic system identification techniques,
tuned offline under stationary assumptions. These methods lack formal uncertainty quan­
tification and do not generalize well in varying operational conditions or asset types. In ad­
dition, deploying these models in embedded settings poses additional challenges, such as
limited data availability, real­time constraints, and the need for interpretability and safety
assurances.
The application of Bayesian Inference in real­time control and system identification of
power converter–asset systems remains largely unexplored, especially in settings where
models must be deployed on edge hardware, operate with minimal supervision, and gen­
eralize across heterogeneous devices.
This thesis explores a Bayesian modeling framework for identifying low­order motor mod­
els using experimental time­series data, with a focus on capturing model uncertainty
through structured priors and probabilistic inference. The plausibility of posteriors de­
rived from one motor may generalizing to others, potentially supporting adaptation across
similar assets, is also explored in this thesis.
An MQTT­based communication layer is implemented to serve as a cyber­secure, lightweight
interface between industrial components—including drives, PLCs, and edge devices. This
protocol enables encrypted data exchange and real­time control, providing a flexible back­
bone for deploying learning­based models in edge environments. The combined frame­
work aims to investigate the feasibility of scalable, uncertainty­aware, and cyber­secure
control in converter­driven motor systems.

Introduction
1.1
Motivation
Converter­driven assets such as pumps, fans, and motors are central to modern industrial
infrastructure. These assets are increasingly seen not just as energy consumers but as
flexible, controllable resources that can support grid stability and reduce energy waste [1].
One sector with high untapped potential is wastewater treatment plants, where motor­
driven compressors and pumps operate around the clock but are rarely optimized for
flexibility or grid interaction.
To extract value from such systems, we need accurate models that describe how these
assets behave under varying conditions. Unfortunately, traditional modeling approaches
fall short [2]. They are typically deterministic, tuned offline under narrow assumptions,
and unable to adapt when system behavior drifts. These models lack mechanisms to ex­
press uncertainty, making them fragile when deployed in real­world settings where noise,
missing data, and changes in operating conditions are the norm.
In contrast, Bayesian modeling offers a more resilient alternative [3]. It allows us to learn
from data while explicitly modeling uncertainty in both the parameters and the predic­
tions. This is particularly useful for deployment in distributed, resource­constrained envi­
ronments such as edge devices in industrial IoT systems where large­scale labeled data
collection is often infeasible due to bandwidth, storage, or operational limitations, and
where transparent, explainable models are essential for system validation, safety, and
diagnostics [4]. Bayesian inference helps in these cases by using prior knowledge and
showing the model uncertainty, which makes it easier to work with small amounts of data
and build models that are reliable and easy to understand.
1.2
Problem Statement
This thesis investigates the Industrial Internet of Things based framework for data acqui­
sition and control of converter­driven motrs and how Bayesian learning can be used to
model the dynamics of such systems, with a focus on edge­deployable, uncertainty­aware
applications. We explore whether models trained on one motor system can generalize to
others, enabling sample­efficient learning across heterogeneous setups. Specifically, we
target wastewater pumping systems, which are ideal candidates for grid­flexible operation
but currently lack intelligent, adaptable control frameworks that can efficiently run on the
edge.
We use Bayesian autoregressive models with exogenous inputs (ARX) to predict motor
electrical power based on variables such as speed and torque. These models are trained
on real­world time­series data from two experimental setups involving Danfoss iC7 and
VLT converters. By using Markov Chain Monte Carlo (MCMC) we test the following hy­
pothesis:
• Can posteriors trained on one machine serve as informative priors for another?
• Can models be updated sequentially as new data becomes available, enabling on­
line learning on edge devices.
• Can uncertainty estimates help detect overconfidence?

1.3
Why Bayesian Methods?
Bayesian inference is a tool which will be used to attempt to solve the hypothesis listed in
the previous section:
• It avoids overfitting by incorporating prior knowledge.
• It handles limited or noisy data.
• It quantifies confidence in predictions.
In contrast, deterministic models offer only point estimates and require manual tuning for
each new system.
1.4
System Context
The experimental platform includes two motor­load setups using Danfoss iC7 and VLT
power converters. In each setup, one motor operates in speed control mode while the
other acts as a load through torque control. These configurations mimic realistic industrial
scenarios, such as pump operation in wastewater plants. Control signals and sensor
data are exchanged over Modbus and PROFINET protocols and streamed via MQTT for
remote monitoring and inference.
The modeling pipeline is deployed on edge devices that act as local data acquisition and
control nodes. These devices run full probabilistic programming workflows (via PyMC) [5],
demonstrating that modern Bayesian inference is now lightweight enough for real­time
industrial applications.
1.5
Contributions and Scope
This thesis presents the following contributions:
1. A complete Bayesian modeling framework for converter­driven motors, imple­
mented on edge computing hardware using real data.
2. Cross­asset posterior transferability, Test if models trained on one motor can
generalize to others.
3. Online model update mechanisms, demonstrating how posteriors can be incre­
mentally refined as new data becomes available.
4. Cyber­secure, edge­based deployment using MQTT and TLS, bridging modern
control with industrial IoT.
While the potential for integrating these models with grid services is substantial, this work
focuses solely on the modeling aspect laying the statistical and computational foundation
for future approaches.

2.1
Motor control in Industrial Systems
The induction motor is the main workhorse of the modern industry. The 3­phase induction
motors are readily available and deployed in a wide variety of industrial applications such
as material handling, pumps and compressors, HVAC systems, machining tools, cranes
and conveyors, printing, textiles, and mining. This wide area of application stems from
the fact that the induction motor has flexible control options allowing it to be controlled ef­
ficiently without compromising on the desired output. A wide variety of speed and torque
control techniques, and the development of power converters, have made these motors
irreplaceable in the modern industry. The idea behind these control techniques was de­
veloped on the basis of the equivalent circuit of the induction motor in Figure 2.1 [6].

appropriate voltage vectors from a switching reference table based on the error
between desired and estimated values.
• Field Oriented Control: Complex control of stator currents through co­ordinate trans­
formation of flux and currents into direct­axis and quadrature­axis components to
effectively decouple the torque and flux components of the motor. This is suitable
for more complex and precise control.

2.2
Power converters and control strategies
Power converter control strategies can be categorized as scalar and vector control. The
strategies are explained briefly.
2.2.1
Scalar Control (U/f Control)
Scalar control methods, also known as voltage­to­frequency (V/f) control, operate by
maintaining a constant ratio between the stator voltage and frequency. This ensures
that the magnetic flux in the motor remains approximately constant, preventing core sat­
uration. However, scalar control does not account for dynamic interactions between the
voltage and current vectors, and therefore offers limited performance in terms of torque
and speed regulation [7].
Advantages:
• Simple implementation.
• Effective for constant­torque, low­dynamic applications (e.g., pumps and fans).
Limitations:
• Poor dynamic response.
• No direct control of torque or flux.
In the experimental setup, the VLT Converter 1 operates in scalar U/f mode with speed
control in an open­loop configuration.
2.2.2
Vector Control (Field­Oriented Control)
Vector control, also known as Field­Oriented Control (FOC), decouples the stator current
into two orthogonal components—one controlling flux and the other torque. This allows
independent regulation of these quantities, closely mimicking the behavior of a separately
excited DC motor [7].
Advantages:
• High dynamic performance.
• Precise torque and speed control.
Variants:
• Field­Oriented Control (FOC): Requires position or speed feedback via encoders for
co­ordinate transformation of
• Direct Torque Control (DTC): Controls torque and flux directly without requiring co­
ordinate transformation.
In the experimental setups:
• The iC7 Converter 1 uses Voltage Vector Control (VVC+) in open­loop speed control
mode [8].
• The iC7 Converter 2 uses Field Vector Control (FVC+) in open­loop torque control
mode [8].
• The VLT Converter 2 operates in closed­loop flux control mode using encoder feed­
back [9].

2.3
Communication protocols and Data collection
One of the most important aspects of the system is the data acquisition and control
pipeline. Internet of Things (IoT) has enabled efficient communication between industrial
devices. The need for a robust and secure pipeline is the foundation for the experiment
to be successful. The main communication protocols used in the 2 setups are MQTT,
PROFINET, and MODBUS.
2.3.1
PROFINET
PROFINET is an industrial ethernet standard developed by Siemens and PROFIBUS &
PROFINET International (PI). It is used for real­time communication between controllers
like PLCs and field devices like power converters (drives) and sensors. In the VLT setup
that is being used, we are using PROFINET for high­speed control and data acquisition
from VLT converters to the PLC and edge computer (Revpi). The Python library used for
handling PROFINET clients is python­snap7 [10].
2.3.2
MODBUS
Modbus TCP, developed by MODICON, is a widely used communication protocol that
operates over TCP/IP networks. It makes use of a simple master­slave configuration for
data acquisition and control of the iC7 power converters. The Python library used for
handling MODBUS TCP clients is pymodbus [11].
2.3.3
MQTT
Message Queuing Telemetry Transport is a lightweight publish­subscribe protocol de­
signed for low­bandwidth, high­latency networks. Originally developed by Andy Stanford­
Clark and Arlen Nipper in 1999 to monitor oil pipilines in a SCADA system, it has become
the go­to standard protocol for IoT systems that are bandwidth­limited and low power. It is
used to send motor/converter parameter data to the external computers and receive the
control signals for setting the speed and torque references from the external computer.
MQTT is a modern, flexible protocol designed for energy­efficient embedded systems and
supports transmission of data over the internet to integrated dashboards, and is scalable
to over 50000 clients connected to a single broker, which in our case is the Revpi for
the VLT setup and Advantech UNO 127 for the iC7 setup. The latest MQTT version is
MQTT 5.0 [12]. MQTT has a bigger advantage over the other protocols in terms of se­
cure communication. It can be secured using Transport Layer Security (TLS), which is a
cryptographic protocol used to encode data between two points in a network. The TLS
certificate makes use of cryptographic algorithms such as RSA (Rivest­Shamir­Adleman)
and ECDSA (Elliptic Curve Digital Signature Algorithm) for encrypting data.

2.4
System identification
System identification is a crucial step in modeling dynamic systems based on observed
input­output data. In this project, system identification techniques were applied to de­
velop mathematical models that describe the behavior of the motor­converter systems
under varying load conditions. These models are then improved for uncertainty aware­
ness through probabilistic frameworks using Bayesian Inference.
The identification process involved collecting time­series data from the converters using
Modbus TCP, PROFINET, and MQTT, capturing key signals such as motor speed, torque,
power, and switch heatsink temperature. The data was then processed and analyzed to
estimate model parameters that characterize the dynamic response of the system.
Both physics­informed models and data­driven approaches, including Bayesian linear re­
gression and autoregressive models with exogenous inputs (ARX), were explored. These
models enable the prediction of system behavior under different operating conditions.
2.4.1
Model types
The basis for choosing the model comes from the understanding of the system. If we
have a good underlying understanding of the physics then we go with a physics­informed
static or quasi­static model (also known as white box model). The other approach involves
data­driven modeling, where the model is fit onto the data collected from a system.
When the in­depth understanding of the underlying relationships between system param­
eters, input and outputs is known, a white box model can be built. This model has very
high accuracy but for time­varying systems when initial conditions and forecasting is de­
sired, data­driven modeling approach is preferred as it allows us to find initial conditions
of a system and forecast future values. Both models will be used for uncertainty quantifi­
cation further in this report.
2.4.2
System identification loop
The system identification loop consists of multiple steps [13].
The first process is model construction, which has three important aspects to it:
• A dataset generation. The dataset is generated to observe the relationship between
inputs and outputs. Ideally, the system identifier should measure the effect of all the
possible system outputs by applying all possible inputs to correctly characterize the
system in the form of measured data which is the baseline for the next steps in
system identification.
• A model structure (white, black, or gray box model). This is chosen based on an
assumption that the model can accurately depict the input and output relationships.
• Assessment rules for fitting the different types of models. This is basically the system
identification process by fitting the best model on the data and observing how it
reproduces the measured data.
The next step is model validation. In this step, we observe the output, which ideally.
The difference between the actual output and the model output is used to calculate the
residuals. These residuals are then evaluated based on their structure to decide if the
model is acceptable. If not, the model training can be revised or a new model can be
chosen. The two basic models used in this report are explained in brief here and detailed
modeling is done in

2.5
Probabilistic modeling
Probability describes how likely it is that an event will occur. It can be ”0” which means
the event is impossible, and ”1” means the event is certain. This is known as probabilistic
inference and is the main idea behind improving basic models to capture uncertainty. We
have frequentist and Bayesian understanding of probability. In Bayesian inference we
compute the posterior distribution. This gives a full distribution of a parameter. From this
distribution we know that the parameter will take a certain value based on the probability
of that parameter from the distribution of values of that parameter.
2.5.1
Bayes’ Theorem and Bayesian regression
Bayes’ rule provides a way to update our beliefs about unknown parameters based on
observed data. It forms the foundation of Bayesian inference and is written as:
p(θ | D) = p(D | θ) · p(θ)

• p(θ | D) is the posterior — the updated belief about the parameter θ after observing
data D,
• p(D | θ) is the likelihood — how likely the data is given the parameter,
• p(θ) is the prior — our belief about the parameter before seeing the data,
• p(D) is the evidence — the total probability of the data, acting as a normalizing
constant.
In practice, the evidence term p(D) is often hard to compute directly, but it does not depend
on θ and can be ignored when performing parameter estimation.
Bayesian inference combines prior knowledge with observed data to produce a full distri­
bution over the model parameters, rather than a single­point estimate. This allows us to
express both prediction and uncertainty in a principled way. If we have to fit a line to a

Aleatoric and Epistemic Uncertainty
In real­world systems, uncertainty can come from two different sources: aleatoric and
epistemic uncertainty [14].
Aleatoric uncertainty is caused by noise or randomness that is part of the system. It
cannot be reduced by collecting more data. For example, sensor noise or unpredictable
load changes in an industrial motor system are aleatoric.
Epistemic uncertainty comes from a lack of knowledge about the model or system. It
can be reduced by collecting more data. It reflects what the model does not know due to
limited observations.
In Bayesian modeling, these two types of uncertainty are often represented as:
• Aleatoric: captured by the likelihood variance σ2(x).
• Epistemic: captured by uncertainty in the model parameters w, or by the posterior
distribution p(w | X, y).

Markov Chain Monte Carlo (MCMC)
Bayesian models often lead to complex posterior distributions that cannot be computed
analytically. In such cases, we use sampling­based methods to approximate the posterior.
One of the most widely used approaches is Markov Chain Monte Carlo (MCMC) [15].

A Markov chain is a sequence of random variables {X0, X1, X2, . . . } with the Markov
property, which states that the future state depends only on the present state and not on
the sequence of events that preceded it [15]. Formally,
P(Xn+1 = x | Xn = xn, Xn−1 = xn−1, . . . , X0 = x0) = P(Xn+1 = x | Xn = xn).
(2.10)
A Markov chain is characterized by its transition kernel K(x, x′), which gives the proba­
bility of transitioning from state x to state x′. Under mild conditions (e.g., irreducibility and
aperiodicity), the chain has a stationary distribution π(x) to which it converges as n →∞.
Monte Carlo methods refer to a broad class of computational algorithms that rely on ran­
dom sampling to obtain numerical results [15]. The goal is often to estimate expectations
of the form:
Ep(x)[f(x)] =

sampling from p(x) is difficult, Monte Carlo techniques approximate the expectation by
averaging over random samples:

MCMC constructs a Markov chain that has the desired posterior distribution p(θ | D) as
its equilibrium distribution. By drawing samples from this chain, we can approximate the
posterior and compute expectations such as means, variances, and credible intervals.
The general procedure involves:
1. Starting with an initial guess θ0,
2. Generating a sequence of samples θ1, θ2, . . . such that:
θ(t+1) ∼q(θ(t+1) | θ(t)),
where q is a proposal distribution that defines how the next sample depends on the
current one.
3. Accepting or rejecting proposed samples based on their posterior probability.
The implementation is done using the PyMC library on the Edge computer, which abstracts
much of the sampler tuning and diagnostics. The resulting MCMC samples allow us to
estimate posterior distributions for model parameters, generate predictive intervals, and
analyze uncertainty. In this thesis, we use the No­U­Turn Sampler (NUTS), a variant of
the Hamiltonian Monte Carlo (HMC) algorithm. NUTS adapts step sizes and path lengths
automatically, leading to faster convergence and efficient sampling in high­dimensional
spaces [16].
Key benefits of MCMC:
• It does not assume a specific form for the posterior.
• It gives consistent estimates of expectations as more samples are drawn.
• It is especially useful for models with few assumptions and nonlinear structures [3].

System Design and Methodology
This chapter introduces the design and working of the experimental setup used in the
thesis. Two different motor­converter systems are used: one based on Danfoss iC7 drives
and the other on Danfoss VLT drives. Both systems use a pair of three­phase induction
motors that are mechanically coupled. One motor is controlled in speed mode and the
other in torque mode. The chapter describes the hardware components, such as motors,
converters, edge computers, and PLCs, and how they are connected. It also explains
the data communication methods used, including Modbus TCP, PROFINET, and MQTT,
as well as how data is collected and stored. This setup forms the base for testing the
modeling and Bayesian inference methods discussed in later chapters. The experimental
setup consists of 2 sets of 2 motors, each driven by their own individual power converters.
The motors in each set are mechanically coupled to each other to emulate industrial motor­
load interaction, e.g., pumps in wastewater treatment. The first experimental setup uses
a pair of iC7 converters and the second experimental setup uses a pair of VLT converters.
3.1
System architecture
The system is divided into two parts: the control/communication side and the power side.
On the power side, A 3­phase AC source provides power to two Danfoss drives (iC7 or
VLT, depending on the setup used). One of the Danfoss drives is configured for speed
control while the other is set for torque control. The rotors are mechanically coupled to
each other at their shafts.
The control side features edge computers connected to both drives via Modbus TCP over
Ethernet in the iC7 setup and a Revpi­PLC combination connected to the VLT drives
through PROFINET. These edge computers and the PLC are powered by 24V 1A DC
power supply. Both edge computers act as the central communication nodes, logging
data to external/lab computers via MQTT. An Ethernet switch is used to connect the Edge
computer to other computers for data visualization and processing.
The power converters are initially configured using Danfoss’ ”Automatic Motor Adaptation”
(AMA) function for easy commissioning. This function measures internal parameters of
the machine, such as stator/rotor resistance and inductance. This function can be static
or dynamic.
Two types of drives are defined based on their configuration and functionality:
• Drive 1 is set for speed control drive and controls the motor that acts as the prime
mover of the pump. The speed references are written to this drive to regulate the
motor speed. The speed, torque, and electrical power of this drive are the main
parameters used for further mathematical modeling of the system.
• Drive 2 is configured for torque control for the second motor in the pair. Torque
references are written to this drive via Modbus , controlling the mechanical torque
delivered to the coupled shafts. This is the load simulator. The two motors form
an electrical power system as any other industrial motor­load configuration, and are
essential to study dynamics for system identification and modeling.
The experimental setups are explained in further detail.

3.7.1
Cyber­securing the control and data signals
The Message Queuing Telemetry Transport (MQTT) protocol default configuration lacks
encryption and authentication, making it vulnerable to attacks such as eavesdropping,
spoofing, and unauthorized access. To enhance security, several best practices can be
adopted. These include enabling TLS/SSL encryption, using client authentication (e.g.,
with usernames/passwords or X.509 client certificates), enforcing access control via topic­
based ACLs, and disabling anonymous access on the broker.
In this project, Transport Layer Security (TLS) was implemented to secure the communi­
cation between Edge computers acting as the MQTT brokers and external MQTT clients.
A private Public Key Infrastructure (PKI) was set up using OpenSSL. First, a self­signed
Certificate Authority (CA) certificate (ca.crt) was created on the broker. This CA was
then used to generate and sign the broker’s server certificate (server.crt) and private
key (server.key). The key is generated using RSA cryptography. The server certificate
is configured with appropriate Subject Alternative Names (SAN) to support IP address
verification.
The Mosquitto broker was configured to listen on port 8883, with TLS parameters defined
in a separate configuration file. Clients were configured to use the paho-mqtt library
in Python, with ca.crt provided to verify the server’s certificate. This ensures that the
connection is encrypted, and that clients can confirm they are communicating with the
legitimate broker. In addition to TLS­secured communication, non­secure communication
over port 1883 was enabled for testing purposes only.
This security setup achieves encrypted communication, mutual trust between client and
broker, and protection against common cyber threats. It serves as a foundation for inte­
grating additional security layers such as client certificate authentication, fine­grained ACL
policies, and intrusion detection systems. This also prohibits other lab/external computers
on the network from publishing/subscribing to the MQTT broker.

4.3
The dynamical model of the motor
The ARX model is being used to verify the dynamic relationship between output (Mo­
tor Electrical Power) and Inputs (Speed and Torque). The ARX model parameters can
be estimated easily using Least Squares regression or, in the case of this experiment,
Bayesian regression. ARX model consists of two main components: an autoregressive
part that uses past output values and an exogenous part that incorporates external input
variables. Using this model the output can be represented as combination of previous
measurements of outputs and inputs. The ARX model is chosen because of the nature of
the data being piecewise linear. This helps us closely approximate the behavior and also
see the effect of inputs such as speed and torque on motor electrical power.

4.4
Bayesian Inference
4.4.1
Sequential posterior update validation
In Bayesian inference, we use Bayes’ theorem to update the posterior distribution with
new data. So the process starts with fitting an ARX model on the data and then update
the posterior with more data and observe the posterior update. To prove that this works,
a test is designed wherein the motor data set is divided into small chunks and then each
chunk is used to sequentially update the posterior. The test has 2 steps. The first step is
pass the entire dataset as a whole and check the parameters. The second step is to run
the sequential posterior update code and compare the parameter distribution of after the
last chunk with those obtained at the end of step 1, wherein the full data set was directly
used. In the sequential loop the posterior of the previous chunk is used as the prior for
the next chunk in the sequence. The posterior is updated as each chunk has different
data points, and the progression of shape of the posteriors is observed at the end of the
update loop. This update loop is the main step of the online posterior update process,
which will be run on the Edge computers in further experiments. This gives rise to 2
models: First model with full data set passed in as a whole, and the second model with
the same data set divided into equal chunks and posteriors updated sequentially. Ideally,
the 2 models should produce the same results, verifying that Bayesian inference can be
used for updating posteriors online on edge devices as more and more data become
available to the model.

When the model was initially trained using the iC7 data (Phase 1), due to high autocorre­
lation between current and previous values of motor electrical power, the model estimates
parameter a close to 1 with b, c, d being much smaller indicating an the new value of output
is very close to the old value of output with minimum effect of inputs.
In Phase 2, we updated the model using VLT data, which comes from a system that
is similar but not identical. The torque profile is different (Figure 4.23. As a result of
this update, the model’s posteriors for the parameters became narrower (as shown in
the marginal posterior plots), indicating reduced epistemic uncertainty, which means the
model is now more certain about the parameter values.
However, the posterior predictive credible interval became wider. This indicates that the
model has increased its estimate of aleatoric uncertainty, i.e., variability in the data that
cannot be explained by the model itself. This is evident in the posterior predictive plot,
where the updated model reflects a broader credible band compared to the initial model.
This means the model has learned that even though it is more confident in its parameters,
the data itself exhibits variability (e.g., system noise, structural mismatch) that cannot be
captured deterministically.
This behavior illustrates a core strength of Bayesian modeling: it allows us to separate un­
certainty in the parameters (epistemic) from irreducible uncertainty in the data (aleatoric).
The widening of the predictive credible interval does not imply model degradation, but
rather a more noise calibrated representation of uncertainty.

5
Conclusion
The main aim of the thesis was to create a robust and cyber­secure Industrial IoT based
data acquisition and control pipeline and create a system model by acquiring data from
the motor­load setup in the lab and updating this model online using Bayesian Inference.
The first part of the thesis was creation of a common interface protocol between the iC7
and VLT experimental setups, and external/lab computers. This was achieved by use
of MQTT and the data/control control pipeline was cyber­secured using Transport Level
Security certificate (TLS).
The next part was data­driven modeling and updating model parameters through Bayesian
Inference. This was attempted using an arx model and the parameters of this model were
updated using online Bayesian inference. Two tests were conducted with two different
data sets.
The first test with the first dataset was the verification of Bayesian Inference through se­
quential posterior updates. The results showed that if we build two models, first model
with full data set used a whole and the second with the same data set used in chunks,
then the final posterior distributions of parameters obtained in both models are exactly
identical.
The second test was testing the possibility of cross­asset transferability by using posteri­
ors obtained from one system as highly effective priors for the other system. By updating
the iC7­trained model using data from VLT system, the model exhibited tighter posteriors
over parameters while simultaneously producing wider posterior predictive intervals. This
reflects a desirable property of Bayesian learning: the ability to distinguish between epis­
temic and aleatoric uncertainty. While the parameter uncertainty (epistemic) decreased
with additional data, the model correctly recognized increased variability in the system
outputs (aleatoric uncertainty) and adjusted its predictions accordingly.
The experiments also showed that the edge computers used in the setup could handle the
Bayesian updates in real­time with low memory usage. This proves that such probabilis­
tic models can be run efficiently on small, low­power devices which are computationally
limited, thus creating the possibility of assessing a wide variety of Bayesian inference
techniques.
