Abstract
We solve the imprecise pareto front problem. We consider imprecise data, where the data
is represented by regions F = {R1, ..., Rn}, where each data region Ri is associated with a
”true” point pi. We calculate the Pareto front when the input is axis-aligned rectangles,
and we work in the preprocessing model. The preprocessing model splits an algorithm
into a preprocessing phase, which only has access to F, and a reconstruction phase that
for each Ri can retrieve the true point pi. A retrieval strategy is a sequence of regions
that needs to be retrieved to calculate the pareto front. Let (F, P) be the regions F and
their true points P, then let r(F, P) be the minimum number of regions that a retrieval
strategy would need to retrieve, to determine the Pareto front of F given P. This thesis
presents two different retrieval strategies that both make O(r(F, P)) retrievals. We also
present an algorithm for the imprecise pareto front problem, that uses O(n) space and has
a O(n log(n)) preprocessing phase and a O(r(F, P) log(n)) reconstruction phase.

Introduction
In this thesis, we will consider a problem from the field of geometric computing. Tradition-
ally, algorithms primarily work with precise data, which can be an unrealistic standard
in the real world, where uncertainty is a regular occurrence. Therefore, in this thesis, we
will consider a problem with imprecise data as input. We will more specifically consider a
model where the input is a set of regions, where each region represents a single imprecise
data point. We say a region can be retrieved to discover exactly where its data point lies.
The goal is then to quickly calculate a precise geometric structure while retrieving as few
regions as possible.
This model can be used to solve real-world problems where it is fast to calculate an outer
boundary for where a data point lies, but expensive to calculate the precise position. The
data could be the position of a moving object or a value that changes over time, like stock
prices. It could also be when data is so precise that it needs to be stored in external
memory, but a less precise data point can be stored in the cache. So finding the exact
value of a data item can be costly, for many different reasons, either because it takes time,
money, energy, etc.
Preprocessing model
We use the preprocessing model introduced by Held and Mitchell
[1] and used by Van der Hoog et al. [2], which is a framework for imprecise input. It is a
model that splits the problem into first a preprocessing and then an reconstruction phase.
In the preprocessing phase, you only have access to the regions, and cannot retrieve any
points. In the reconstruction phase, it is possible to retrieve the regions’ exact points. The
idea is that the preprocessing phase can be used to create a data structure that can be
queried during the reconstruction phase. Thereby, making it possible to solve the problem
faster.
Pareto front
In this thesis, we will consider points in R2 and calculate a Pareto front.
Given a set of points, the Pareto front, also known as the maximal point problem, is the
ordered point set of maximal points, where a point is maximal if no other point dominates
it, that is, no other point has a larger x and y value. The Pareto front can be seen as an
outer boundary of a point set, and is often drawn as a minimum staircase containing all
maximal points.
The Pareto front is known in many different fields today and is most commonly used
for multi-objective optimization, to optimize data based on multiple parameters. There,
the Pareto front is calculated to quickly limit the solution space to only contain optimal
solutions.
Contribution
For a set of regions F = {R1, ..., Rn}, with their associated ”true” points
P, then the goal of this thesis is to calculate the imprecise Pareto front, which is a Pareto
front that contains the exact same ordering of F as the Pareto front of P. We will limit
the input space, F, to axis-aligned rectangles. We will solve the Pareto front problem
while retrieving as few regions as possible. A sub-goal of the thesis is, therefore, to find
an instance optimal retrieval strategy for the imprecise Pareto front. A retrieval strategy
is a strategy to find regions for retrieval, but it does not actually calculate the imprecise
Pareto front. An instance optimal retrieval strategy is a strategy that is optimal for the
instance of (F, P). Let r(F, P) be the minimal number of retrievals a strategy could make

and still determine the Pareto front of F given P. An instance optimal strategy retrieves
O(r(F, P)) regions. We introduce a layer decomposition to help solve the subproblems.
The layer decomposition splits the regions into different layers, based on their position
in the plane, such that all regions in layer i are not dominated by a region in a layer
below. We describe two instance optimal retrieval strategies that are based on the layer
decomposition. The first strategy is simple to implement, but is not trivially instance
optimal. The second strategy is more complicated to implement, but is easier to bound.
The reason we introduce the second strategy is to help bound the first strategy. We also
present an algorithm that uses the first retrieval strategy. This algorithm uses O(n) space,
has a O(n log(n)) preprocessing phase and a O(r(F, P) log(n)) reconstruction phase, the
details are described later in the thesis.
Background
This thesis and our algorithm is built on the ideas described by Bruce et
al. [3] and Van der Hoog et al. [2] that also considers the imprecise Pareto front problem.
Bruce et al. [3] presents a generic strategy to find which regions need to be retrieved
to solve an imprecise geometric problem. [3]’s strategy is based on witness sets, which
are sets where at least one region has to be retrieved to accurately calculate the imprecise
geometric structure. If the retrieval strategy in [3] only retrieves witness sets of size at most
k, then the strategy is a k-optimal retrieval strategy, that is, it retrieves k ·r(F, P)+O(1)
regions. [3] describes how to find witness sets of regions for the imprecise Pareto front
problem. It proves that their generic strategy is a 3-optimal retrieval strategy, and that
this is best possible. The article does not describe how to implement their generic strategy.
Similarly to [3], our second strategy also retrieves witness sets, which makes it simpler to
prove instance optimality. But it uses a different strategy than the one described in [3].
Our strategies are at most 18-optimal and at least 4-optimal.
Van der Hoog et al. [2] use the preprocessing model to solve the imprecise Pareto front
problem for disjoint axis-aligned rectangles. They show that with a O(n log(n)) prepro-
cessing time then the problem can be solved with an instance optimal retrieval strategy.
Their reconstruction phase is universally optimal. [2] proves lower bounds for the imprecise
pareto front problem. It e.g. proves that even with an instance optimal retrieval strategy,
then there does not exist an instance optimal algorithm that can calculate the imprecise
Pareto front problem. It also shows that a O(n log(n)) preprocessing phase is worst-case
optimal, which is the running time of our algorithm. [2]’s reconstruction phase is faster
than our results, but they only consider disjoint rectangles. The main reason that [2]’s
algorithm is faster is because it searches ”locally” for new regions to be retrieved, while
our algorithm searches ”globally”, so our searches always take O(log(n)) time. We search
globally because the extra complexity of overlapping regions led us to use data structures
where local search is not trivial. We will in Future Works (Chapter 11) discuss possible
solutions to solve this issue.
We present both of the articles further in Chapter 3.
Related work
We will next discuss recent articles that solve the imprecise convex hull
problem, and compare their findings to our results. The convex hull problem is the problem
of calculating the minimum convex polygon that contains all points in the plane. It is
an especially interesting problem to compare to the Pareto front problem, because it is
relatively easy to transform between the two problems. This is because the Pareto front
can be seen as a right upper boundary, while the convex hull is the entire boundary.
Löffler and Raichel [4] present a new preprocessing algorithm for finding the convex hull of

imprecise points. They consider two distinct categories of regions: stable and unstable. A
region is stable if the convex hull can be determined without retrieving the region’s point.
With their preprocessing algorithm, the convex hull can be found in time proportional

De Berg et al. [5] present an algorithm for computing the convex hull of imprecise points.
The algorithm seeks to minimize the number of retrievals and the algorithmic runtime.
For constant complexity polygons the algorithm uses Θ(r(F, P)) retrievals, O(n log(n))
preprocessing time and O(r(F, P) log3(n)) reconstruction time. [5] therefore finds a slower
reconstruction time than our algorithm, but it solves a more general problem, as it works
for all constant complexity polygons, where our algorithm only works for axis-aligned
rectangles.

Preliminaries
In this chapter, we will introduce the problem more formally.
Input
Let F = {R1, ..., Rn} be the input to the problem. F is a set of n arbitrary
axis-aligned rectangles. We denote P ∼F as any point set P = {p1, . . . , pn} where for all
i ∈[n], pi ∈Ri. A point p = (x, y) has the coordinates p.x = x and p.y = y. For each
region Ri ∈F, the algorithm can retrieve the exact point, pi, associated with Ri. When
a point is retrieved then it replaces its region, formally:
Definition 2.0.1. Let A ⊆F, then we define F ⊕A as the set of regions obtained by
retrieving for all Ri ∈A the true point pi. This means that F⊕A is the set where all Ri ∈A
is replaced with its true point pi. For A ⊆F and B ⊆F then (F ⊕A)⊕B = F ⊕(A∪B).
We see points as rectangles with a width and height of zero, but we assume that the input
F only contains rectangles, with a width and height that is larger than zero.
Pareto front
Before we can oﬀicially define the Pareto front, we first define domination
and give an ordering to points.
Definition 2.0.2 (Figure 2.1). A point p1 dominates a point p2 if p1.x > p2.x∧p1.y > p2.y.
Figure 2.1: p1 and p2 do not dominate eachother. p2 dominates p3, p4, and p6, p2 is not
dominated by other points. p6 does not dominate any points.
Definition 2.0.3. For points p1 and p2, p1 ≺p2 (precedes) if p1.x < p2.x or if p1.x =
p2.x∧p1.y > p2.y. Similarly, p1 ≻p2 (succeeds) if p1.x > p2.x or if p1.x = p2.x∧p1.y < p2.y.
We can now define the Pareto front for a point set.
Definition 2.0.4. Let P be a set of points. The Pareto front of P is the ordered sequence
of points in P not dominated by any points in P. Let PF(P) be an ordered set representing
the Pareto front of P. PF(P) is ordered according to the notion of precedes. If ∃pi, pj ∈
P : pi = pj then iff i > j then pi comes before pj in PF(P).
PF(P) is an ordered point set, but for ease of exposition, we will sometimes interchange-
ably refer to it as the staircase curve given by the points in PF(P).
General position
We need to allow points to have the same x or y value to get an
instance optimal retrieval strategy. We can therefore not assume that the points lie in
general position, as then identical x and y values would not be allowed.
Assume for
the sake of contradiction that no two points have the same x or y value. Let the input

Because the regions are identical, then there is no way to guarantee that an algorithm

Retrieval strategy
The first problem this thesis will solve is finding an instance-optimal
retrieval strategy. A (recursive) strategy is oﬀicially defined below.
Definition 2.0.5. Let F be the original algorithmic input, a strategy S is then a function
S : {F ⊕A|A ⊆F} →{B|B ⊆F}. The function S outputs B, where B are the regions
that S will retrieve.
A strategy can be recursively called on the algorithmic input until all relevant regions have
been retrieved. Note that B can contain already retrieved regions; nothing will happen
when those regions are retrieved. Formally:
Definition 2.0.6. Let F be the algorithm input, let S be a strategy and let t be a positive
integer then
F(S, 0) = F
F(S, t) = F(S, t −1) ⊕S(F(S, t −1))
t can intuitively be seen as a notion of time, as F(S, t) is the set F after t steps of S. The
retrieval strategy problem for an imprecise Pareto front can now be defined:
Definition 2.0.7. Given the algorithmic input F, the goal of the retrieval strategy prob-
lem for an imprecise Pareto front is to find a strategy, S, where an integer t∗exists s.t.
∀P1, P2 ∼F(S, t∗) : PF(P1) = PF(P2).
The reason a strategy terminates at time t∗, where ∀P1, P2 ∼F(S, t∗) : PF(P1) = PF(P2),
is because then there is no longer any ambiguity about which regions are in the Pareto
front. When ∀P1, P2 ∼F(S, t∗) : PF(P1) = PF(P2), then all regions are either in all
Pareto fronts of F(S, t∗) or in no Pareto fronts of F(S, t∗). Note that a retrieval does
not create any new ambiguity, so for all times t ≥t∗then ∀P1, P2 ∼F(S, t) : PF(P1) =
PF(P2).
Given the algorithmic input F, the points P ∼F, and a strategy S, then we define
retrivals(F, P, S) as the number of retrievals that strategy S makes to solve the retrieval
problem, when the input is F and the regions’ exact points are P. We want to find a
retrieval strategy that is instance optimal in the number of retrievals. Let
r(F, P) = min
S retrivals(F, P, S)
We then want to find a strategy that retrieves O(r(F, P)) regions.

the imprecise Pareto front problem, which is very similar to the retrieval strategy, except
it calculates the outer Pareto front. The outer Pareto front is oﬀicially defined below.
Definition 2.0.8. Let F be a set of regions. The outer Pareto front of F is the Pareto
front calculated using the top right corners of regions in F. We denote OPF(F) as the
outer Pareto front of F.

Background
In this chapter, we will give a more in-depth overview of the two articles [3] and [2]. These
articles were the ones we read in depth before considering the problem ourselves. We will
focus on the parts of the articles that are relevant to our approach to the problem.
3.1
Eﬀicient Update Strategies for Geometric Computing
with Uncertainty
Article [3] presents a generic algorithm, which can be used to solve imprecise problems
with an eﬀicient retrieval strategy. The generic algorithm is based on witness sets, which
we define below, and so is called the witness algorithm. [3] describes how the witness
algorithm can be used to find the Pareto front, when the regions are open connected
regions or trivial regions.
Types of regions
For a set of regions, F, where we want to calculate the geometric
structure, Φ, then [3] defines four categories a region can be in: always φ, partly φ,
dependent φ, and never φ. Where φ is a property that only points in Φ have (Definitions
1 and 2 from [3]). Paraphrased, an always and never φ region will respectively always
and never have the property φ for any point set P ∼F.
Whereas a partly φ region
contains an always φ point, but is not always φ. A dependent φ region is in none of the
other categories. While always or never φ regions will never change category, partly and
dependent φ regions can change category. A partly φ region can become always φ, while
a dependent φ region can change to any of the other categories. Depending on what input
is allowed and the geometric structure you want to calculate, a lot of regions might change
category when a region is retrieved.
Witness set
As mentioned, the generic algorithm is based on the notion of witness sets.
[3] defines a witness set in the following way:
Definition 3.1.1 (Definition 3 from [3]). Let F be a set of regions and let C be a set of
point sets where ∀P ∈C : P ∼F. A set W of regions in F is called a witness set of F
with respect to C if for any P ∈C at least one region in W must be retrieved to verify
the solution. We say W is a witness set of F if W is a witness set of F with respect to
{P|P ∼F}.
Notice that there is a distinction between a witness set with respect to one configuration
and a witness set with respect to all configurations. Per this definition, a witness set can
include regions that do not need to be retrieved, as long as one region in the witness set
needs to be retrieved.
Witness algorithm
Now, for an overview of the witness algorithm, which is [3]’s re-
trieval strategy. The witness algorithm terminates when there are no partly or dependent
φ regions. If there is a partly φ region, the algorithm retrieves all regions in a witness set
that contain a partly φ region. If there are no partly φ regions, the algorithm retrieves
a witness set that contains a dependent φ region. Note that partly φ regions can occur
again later on. When the algorithm terminates, there are only always and never φ regions,
which makes it possible to calculate Φ, without any ambiguity. The algorithm can be seen
as pseudocode in Algorithm 1.

Optimality
[3] defines a c-optimal retrieval strategy to be a strategy that retrieves at
most c·r(F, P)+ O(1) regions. If c is a constant where c ≥1, and a strategy is c-optimal,
then the strategy is instance optimal. [3] argues that if the witness algorithm retrieves
witness set of size at most k, then this retrieval strategy is k-optimal. The argument goes:
let W1, . . . , Wl be the witness sets retrieved by the algorithm, these witness sets must be
disjoint as each is retrieved fully before the next is retrieved. Any optimal algorithm,
therefore, has to retrieve at least l regions, as an optimal algorithm would retrieve at
least one region in every witness set. Since the witness sets have size k, then the witness
algorithm is a k-optimal retrieval strategy. (Lemma 1 from article [3])
Pareto front
Article [3] shows that for the imprecise Pareto front problem then the
witness algorithm can always find a witness set of size max 3. It is therefore a 3-optimal
retrieval strategy for the Pareto front. For the Pareto front, the property φ is maximality,
which means that an always maximal point in F is not dominated by any other points in
F.
First [3] shows that all partly maximal regions are in a witness set of size at most 2. The
argument is that a partly maximal region, A, has an always maximal point, but also a
point that is not always maximal. If A contains a never maximal point, then A is in a
witness set by itself, as A needs to be retrieved to determine if it is on the Pareto front.
If A contains a dependent maximal point, then there is an unretrieved region making the
point dependent maximal. This means that A is in a witness set with that region.
Furthermore, [3] argues (in Lemma 5) that if there are no partly maximal regions, then
all dependent maximal regions are in a witness set of size at most 3. To help prove this,
[3] first proves the following lemma.
Lemma 3.1.1 (Lemma 4 from [3]). Let l be a horizontal or vertical line. Further let l split
two regions A and B. Then the regions A and B cannot both be always maximal regions.
The proof for Lemma 5 then goes: let A be a region with a dependent maximal point p,
such that p is maximal among all dependent maximal points. Then let the line l1 be the
horizontal line that intersects p, and let line l2 be the vertical line that intersects p. In
the top right quadrant Q of l1 and l2, there must be a region B that makes A dependent.
B cannot be dependent maximal, as then p would not be maximal among all dependent
maximal points. B can also not be partly maximal, because we assumed there were no
partly maximal regions. So B must be an always maximal region. B must intersect at
least one of l1 abd l2 and by Lemma 3.1.1 there is at most two always maximal regions B
and C in Q that intersect l1 or l2. So there is a witness set {A, B, C}.

Limitations
Article [3] shows that a 3-optimal retrieval strategy exists for the maximal
point problem with uncertainty and that no better solution is possible. What the article
does not cover is how this can be done. To go from the witness algorithm in the abstract
to a functional algorithm.
There are mainly two problems: determining what regions
are partly and dependent at runtime, and determining witness sets in a reasonable time.
Determining the dependent regions at runtime can be hard to do in proper time, as a
lot of dependent φ regions can become always or never φ when a region is retrieved.
Furthermore, you can not determine the witness set during the preprocessing phase, as a
witness set can change after a retrieval.

Compound regions
Since sinks in G(F) are not in ˜F(P), then a universally optimal
algorithm cannot spend constant time for each sink in G(F). To solve this issue [2] intro-
duces the notion of a compound region. Let [Ri, Rj] be a maximal consecutive sequence
of sinks in G(F). Then R∗is the minimum rectangle that captures all regions in [Ri, Rj].
R∗is the compound region for the regions in [Ri, Rj]. Article [2] proves Lemma 3.2.4:
Lemma 3.2.4 (Lemma 5.1 from [2]). Let [Ri, Rj] be a sequence of sinks in G(F). If
there exists no point pk preceding pi and dominating pi, then all points preceding pi do
not dominate points in [pi, pj]. If such pk dominates pj, then pk dominates all points in
[pi, pj]. Similar statements hold for pk succeeding pj.
The Lemma implies that the regions can be retrieved such that an algorithm only needs
to retrieve a constant number of regions in each compound region. The time spent on
compound regions can thus be charged to regions in ˜F(P). The compounding of regions
is, therefore, a way to lump regions together, such that it is possible to quickly ignore a
large number of irrelevant regions.

Algorithmic preliminaries
As mentioned in previous chapters, there are two problems we want to solve: find an
instance optimal retrieval strategy, and calculate the outer pareto front. To describe our
solutions, we first define several terms.
Relations between regions
Different regions have different characteristics based on
their relations to each other. We will now define some relations that we will use later in
the thesis.
Given a set of regions F, then some regions in F will either always or never be in the
Pareto front of F, independent of which regions in F will later be retrieved. Similarly to
[3] we split regions into different categories.
Definition 4.0.1. Given a set of regions F = {R1, ..., Rn} and a region Rj then
• Rj is always if for all P ∼F then pj ∈PF(P).
• Rj is dominated if for all P ∼F then pj ̸∈PF(P).
The definitions of always and dominated are useful, because they indicate when the prob-
lems have been solved: given a set of regions F, then if all regions are either always or
dominated, then ∀P1, P2 ∼F : PF(P1) = PF(P2). We now define domination between
regions.
Definition 4.0.2. A region Ri dominates another region Rj if ∀p : p ∈Ri and ∀p′ : p′ ∈Rj
p dominates p′.
For axis-aligned rectangles, Definition 4.0.2 implies that the region Ri dominates the region
Rj iff Ri’s bottom left corner dominates Rj’s top right corner.
Similar to the outer Pareto front, we can also consider the inner Pareto front of a set of
regions F. The inner Pareto front of F is the Pareto front of all bottom left corners. Iff
a region is below the inner Pareto front of F, then it is dominated in F. Given a set of
regions F, not all point retrievals affect all regions in F equally. Some regions depend on
each other; so when one region is retrieved, others might become dominated or always.
Definition 4.0.3 (Figure 4.1). For a set of regions F, then a region A ∈F has an outgoing
dependency to a region B ∈F iff
1. A ̸= B and
2. B’s top right corner does not dominate A and
3. there exists a point a ∈A that dominates a point b ∈B, and b is above the inner
Pareto front of F.
Iff A has an outgoing dependency to B, then B has an incoming dependency from A.
We are only interested in whether a region A has an outgoing dependency, and not which
regions it has dependencies to. This is because regions that have no outgoing dependencies
and are always, will never be retrieved by the best retrieval strategy. Intuitively, this is
because they are in the final pareto front, and retrieving them does not affect whether
other regions are in the final pareto front. To sort out some of these regions and inspired
by the compound regions in [2], we define the set D0.

Dynamic Pareto front
We will store the sets:
• ∀i : Li(t) = Li(t)
• ∀i : PF(Pi(t)) = PF(Pi(t))
• Φ(t) = Φ(t) ∪{(−∞, ∞), (∞, −∞)}
• D0(t) = {Rj|j ∈D0 ∧Rj ∈F(t)}
All the sets are dynamic Pareto fronts and initially ∀i : PF(Pi(0)) = ∅, ∀i : Li(0) = Li(0)1,
Φ(0) = {(−∞, ∞), (∞, −∞)}, and D0(0) = {Rj|j ∈D0 ∧Rj ∈F(0)}. We will store each
set in a binary search tree, with a corresponding linked list. Each binary search tree has
the following functions, where B is the search tree and where p is an element denoted by
a point:
• del(p): Iff p ∈B then delete p from B.
• pre(p): Return the element ppre = miny(maxx( {(x, y)|(x, y) ∈B ∧x ≤p.x ∧y ≥
p.y})). Which is p’s previous point in B if p is in B.
• succ(p): Return the element psucc = maxy(minx({(x, y)|(x, y) ∈B ∧x ≥p.x ∧y ≤
p.y})). Which is p’s next point in B if p is in B.
• delete(p): Delete all elements in B dominated by p.
• deleteAndInsert(p): Iff p is not dominated by a region in B, then delete all elements
in B dominated by p and insert p into B.

Layer finding structure
The data structure will also maintain a layer finding structure,
which will help maintain PF(Pi). The layer finding structure, takes a point p and returns
the maximum i such that the staircase OPF({Rj|j ∈Fi}) is not below p. Our layer
finding structure uses the point location algorithm described by Sarnak and Tarjan [7].
Article [7] describes a persistent search tree that can be used for point location. In the
article, the input to the algorithm is a division of the plane into polygons, where you can
query with a point to find which polygon it is contained in. In our case, the polygons are
defined by our layer decomposition, so a polygon i is the polygon with OPF({Rj|j ∈Fi})
as one side and OPF({Rj|j ∈Fi+1}) as another. These staircases will never cross, so
the polygons go to −∞in the x-axis and to −∞in the y-axis, see Figure 7.2. Let m be
the number of line segments dividing the plane, then [7] states that the data structure
uses O(log(m)) query time, O(m) space and O(m log(m)) preprocessing time. In our case,
m = O(n). So it uses O(n) space, O(n log(n)) preprocessing time and O(log(n)) query
time.

10.1
Zero dependency regions
In this section, we will describe how to initialize D0. For each region R ∈F let dep(R) =
true iff R has an outgoing dependency in F. Then D0(0) = {Ri|Ri ∈F ∧¬dep(Ri) ∧
Ri not dominated}. Below, we will prove that for all R ∈F, dep(R) can be found in
O(n log(n)) time and O(n) space.
Definition 10.1.1. For a region, R, let R.left and R.right be the x-values of R’s left and
right borders, respectively. Let R.top and R.bottom be the y-value of R’s top and bottom
borders, respectively.
To help find dep(R), we see an outgoing dependency as a downward or leftward depen-
dency. Formally:
Definition 10.1.2. For a set of regions F, then a region A ∈F has a downward depen-
dency to a region B ∈F iff
1. B ̸= A and
2. B’s top right corner does not dominate A and
3. There exists a point a ∈A that dominates a point b ∈B, where b.x ≥A.left and b
is above the inner Pareto front of F.
And a region A ∈F has a leftward dependency to a region B ∈F iff
1. B ̸= A and
2. B’s top right corner does not dominate A and
3. There exists a point a ∈A that dominates a point b ∈B, where b.y ≥A.bottom,
and b is above the inner Pareto front of F.
Note that a region Ri can have both a downward and leftward dependency to region Rj,
this just means Rj’s top right corner is inside Ri.
Definitions 4.0.3 and 10.1.2 imply
Corollary 10.1.1.
Corollary 10.1.1. A region has an outgoing dependency iff it has a leftward or downward
dependency.
By Corollary 10.1.1, then finding outgoing dependencies can be split into the subproblems
of finding leftward and downward dependencies. We will describe an algorithm to find
downward dependencies, but a similar algorithm can be used to find leftward dependencies.
The problem of finding downward dependencies can be split into two subcases. which is
proven in Lemma 10.1.1. Informally, a region R has a downward dependency iff there

Future Work
We will, in the next chapter, discuss various directions this work can be furthered. We
will first discuss ways we can improve the algorithm and retrieval strategy. We will then
see how our algorithm and strategies work on more generalized input.
11.1
Tighter running time
Our algorithm is not universally optimal. The main reason is that the algorithm currently
uses amortized O(log(n)) time per retrieved region. If we consider the universally optimal
algorithm from [2] then it only uses O(log(Hi(P)) + log(Vi(P))) per region Ri ∈˜F(P).
Recall that ˜F(P) is the subset of F such that each region in ˜F(P) intersects the Pareto
front of P and is not a sink in G(F), so it is not a zero-dependency region.
[2] gets
this faster running time because when their algorithm retrieves a region Ri, then it only
searches locally, so the search grid depends on Ri’s dimensions. Our algorithm will, on the
other hand, search the entire grid. Our algorithm could therefore get a tighter running
time if it searched inside a smaller area.
Dynamic structures
When we update our dynamic Pareto fronts, it currently takes
O(log(n)) time per region. But when you search for a point p, in a dynamic Pareto front,
B, it is only the regions in B.pre(p) to B.succ(p) that we are interested in. A way to
improve the running time is thus to update the Pareto front structure to use local searches
instead of global searches as it does now. One way to achieve this would be to implement
finger search. In finger search, a pointer to an element is used when searching. Given
that d is the distance between the finger and the element you want to find, it only takes
Θ(log(d)) time; this result is also possible on a dynamic data structure [10, 11]. As out-
lined, this requires a finger that is close to the search element. For most searches done in
Algorithm 2, there is no obvious choice of finger, so making this change is not trivial.
Layer finding structure
The second reason our algorithm uses amortized O(log(n))
time per retrieved region is that the layer finding structure has a query time of O(log(n)).
Every query the algorithm does searches for a point pi ∈P. Since we know that pi lies
inside the region Ri, we only really need to search inside Ri, instead of searching in the
entire grid, as we do now. Thus, a way to improve the running time of our layer finding
structure is to further develop the ideas described in [7], which outlines our current layer
finding structure. Given a set of polygons, the algorithm in [7] creates horizontal slabs
through the plane: it is split by a horizontal line through each point. The slabs are then
numbered from left to right. The data structure also consists of a persistent search tree,
which at time t stores the polygons that intersect slab t ordered from top to bottom; the
slab decomposition implies that this order is possible. The query of a point p described by
[7] then first searches over p.x to find the slab t that contains p. The query then uses t to
search correctly in the persistent search tree for p. Both searches currently take O(log(n))
time, where n is the number of points in the polygons. It is relatively simple for each
region to preprocess which slabs they intersect. This will still be O(n) space. It is thus
quite simple to optimize the first search to only search over the relevant slabs. The second
part of the search uses a persistent search tree, and is therefore much more diﬀicult to

Optimality
Both of our retrieval strategies are instance optimal, which means that they are c-optimal
strategies, where c is a constant. In this thesis, we were not interested in optimizing the
strategies beyond instance optimality. Article [3] proves that a 3-optimal retrieval strategy
is the best possible strategy that exists for the Pareto front. As proven in Chapter 5,
neither of our retrieval strategies are 3-optimal. Possible future work is to find tighter
bounds on c for both S1 and S2 and to create a 3-optimal retrieval strategy.
Bounding S1 and S2
We do not know if the bounds found in Lemma 5.3.5 are tight.
We do not see a simple approach to improve the bounds, because Figure 11.1 shows that
there exists a scenario where retrieving region Rj creates exactly 8 new zero-dependency

improve the bounds further.

11.3
Arbitrary polygons
An avenue for future work is generalizing the algorithm. We will discuss what happens
when the input is arbitrary closed 2d shapes.
This problem can clearly be solved by
enclosing each shape in an axis-aligned bounding box. The retrieval strategy will however
no longer be optimal, as regions that did not overlap before the transformation might
overlap after. We will therefore now discuss ways the algorithm could be transformed to
also work for arbitrary closed shapes while staying instance optimal and what some of the
issues are.
Definitions
Our definition of the outer and inner Pareto fronts only work for axis-
aligned rectangles, as it requires that the shape has a top right and bottom left corner,
respectively. We therefore first redefine what the outer and inner Pareto fronts are, so it
works for all shapes. For a region A, let ˆPA be the set of points in A not dominated by any
point in A. Similarly, let ˜PA be all the points that do not dominate any other points in
A. For two regions A and B, A ≺B if there exists a point a ∈ˆPA where ∀b ∈ˆPB : a ≺b.
The outer Pareto front of a set of regions F is the ordered set containing all regions in
PF(∪
R∈F ˆPR). The inner Pareto of F is similarly the ordered set containing all regions
in PF(∪
R∈F ˜PR). In both cases, they are ordered based on the notion of precedes.
Retrieval strategy
With the new definition of the Pareto fronts, both retrieval strate-
gies are correct strategies for arbitrary shapes. Because both retrieval strategies finish
when every region is either a zero-dependency always region, a dominated region, or a
retrieved point.
They will therefore both lead to a correct Pareto front.
We are not
sure whether the retrieval strategies are still instance optimal, because the proof for
Lemma 5.3.1 that bounds the number of new zero-dependency always regions uses a re-
gion’s top right corner. The proof of Lemma 5.2.1, which proves there exist constant-size
witness sets, also technically uses the region’s top right and bottom left corners to simplify
the proof. We believe that the proofs would still work if, for the region R, points in ˆPR
and ˜PR were used instead of R.bl and R.tr, respectively. This, of course, needs a proof to
conclusively prove that the witness sets still exists.
We will next discuss some of the issues of implementing our algorithm for arbitrary shapes.
Preprocessing
The preprocessing does not immediately work for arbitrary shapes. De-
termining D0 is currently very much based on the fact that the regions are axis-aligned
rectangles. It does not seem trivial to extend the indexing, so defining an order might be
a little tricky. It is also not necessarily possible to split into the same subproblems.
Another issue with the preprocessing is the layer decomposition. When making the layer
decomposition, the top right corner is used as a representative of the whole region, as this
can easily be used to determine which regions dominate each other. This is no longer true
for arbitrary shapes.
Layer finding structure
Another issue is the layer finding structure. For arbitrary
shapes, then the layers F1, F2, ... can not be seen as polygons, so the original layer finding

11.4
Pareto fronts in 3d
A Pareto front is used to optimizing over more than one variable, so a natural extension
is to consider the Pareto front of higher dimensions. We will specifically consider what
happens to our algorithm in three dimensions. We will discuss if the algorithm calculates
the correct result and what the new running time is.
Retrieval strategy
Before we can answer these questions then we first need to extent
our definitions to 3d. The input in 3d is F, a set of axis-aligned boxes. A 3d point a
dominates a 3d point b iff a.x > b.x ∧a.y > b.y ∧a.z > b.z. The points are ordered based
on first x, then y, and then z values. We believe that both of our retrieval strategies
will retrieve the correct regions, and that our algorithm will return a correct Pareto front,
because both strategies stop when all regions are retrieved, dominated, or always with no
dependencies. The strategies are no longer instance optimal. The reason our strategies are
instance optimal in 2d is because a retrieval only creates a constant number of new zero-
dependency regions. In 3-dimensions, there is no such guarantee. It is possible, in 3d, to
arrange regions such that they all have outgoing dependencies to Rj, but no dependencies
between them. See Figure 11.3 for an example. The retrieval strategies will, therefore,
depending on the points, retrieve all regions shown in Figure 11.3.
Unit cubes
If the input F is limited to unit cubes, then we believe that the retrieval
strategies are instance optimal. This is because in the unit case, a retrieval only creates

Implementation
We will now discuss the diﬀiculties of implementing the algorithm
in 3d.
There are a few different problems with implementing our algorithm in higher
dimensions. The layer finding structure builds on article [7] that solves point location in
2d, but not in 3d. The algorithm, therefore, needs a new structure to solve the problem in
3d. The algorithm currently stores our Pareto fronts in binary search trees. This is possible
since we know that, in the Pareto front, then iff a ⪯b then a.x ≤a.y ∧a.y ≥b.y. Finding
regions dominated by a point p thus corresponds to a 1d range query from [p.y, p.x].
Similarly, finding points in a 3d Pareto front dominated by a point p corresponds to a 2d
range query; we therefore need to use a different data structure in 3d.
Another issue is the preprocessing. When the preprocessing splits the regions into the
different layers, then it currently sweeps over one dimension and utilizes an RMQ to
quickly search the relevant regions in the other dimension. This clearly no longer works
in 3d. Finding D0, also no longer works, as the subproblems needs to be redefined.

Conclusion
In this paper, we have considered the Pareto front for axis-aligned rectangles. There are
two parts to the results of this thesis: creating an instance optimal retrieval strategy and
solving the imprecise pareto front problem. We have presented two different instance opti-
mal retrieval strategies and discussed how to improve these to create a 3-optimal strategy.
We have furthermore specified an algorithm that solves the imprecise Pareto front prob-
lem. The algorithm uses O(n) space and uses a O(n log(n)) preprocessing phase and a
O(r(F, P) log(n)) reconstruction phase. The preprocessing phase is therefore worst-case
optimal. The reconstruction phase is worst-case optimal, but is not universally optimal,
and it should therefore be possible to create a faster algorithm. We have in this thesis
discussed different ideas to improve the running time. We can lastly conclude that it is
not a simple task to expand the algorithm to work on more general inputs and that there
is a lot of future work that can be done in this area.
