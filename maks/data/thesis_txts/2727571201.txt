Abstract
When optimising the performance of a piece of software, we want to determine if our
changes to the software have improved its performance. To verify this, both versions of
the software are tested using a set of benchmarks. By comparing the results of the original
version with the improved version, we can ideally confirm any improvements.
Modern computers are highly complex machines that introduce noise into the measureยญ
ments. This is because they are simultaneously executing other tasks while performing
the benchmarks. Operating system, number of active processes, or even the current
memory layout can affect the measured results.
To minimise the impact of the noise on the measurements during the benchmarking proยญ
cess, the benchmarks are executed multiple times, generating a distribution of results.
To determine if the software has been improved, the distributions produced by the original
software and the improved software are compared. If there is a statistically significant
difference between the two distributions, it indicates that the changes made to the software
have improved the performance.
The big problem is that most of the time, the baseline noise produced by the computer
differs over time, but also from computer to computer. This means we must execute the
benchmarks on the same machine in an environment with the same average load.
The goal of this project is to improve the benchmarking process, by trying to mitigate the
noise produced by the machine during the benchmarking process. This can help with
gathering and afterwards analysing the data collected during the benchmarking process.
The project will focus on the CPython interpreter because its developers are currently
experiencing difficulties in consistently measuring performance.

Introduction
1.1
Motivation
CPython, the most widely used implementation of the Python programming language, is
often considered relatively slow. However, there is an ongoing project aimed at signifiยญ
cantly improving the performance of CPython. To assess whether the performance of the
programming language is indeed improving, various benchmarking techniques are used.
The objective of this project is to establish proper performance benchmarking techniques
and afterwards apply the acquired knowledge to examine the benchmarking process of
the Python programming language. The goal is to identify potential areas for improvement
in the benchmarking process and determine if it can be enhanced.
1.2
Example
Doing performance benchmarking is challenging. The results obtained from the benchยญ
marking process can be influenced by the noise generated by other programs running on
the same machine. When running the Python Benchmarking Suite on top of CPython, the
output can vary by up to three percent on a dedicated benchmarking rig, which focuses
on running only essential programs to mitigate the noise of the environment. This benchยญ
marking rig provides an environment close to the ideal for benchmarking purposes. If a
developer aims to enhance the performance of CPython, their proposed changes to the
implementation should yield at least a three percent performance improvement. Achievยญ
ing this level of improvement is not an easy task, especially within a single Pull Request.
Moreover, the mentioned variance can be even higher on the regular machines used by
developers. Therefore, it is important to examine a more stable approach to generating
and comparing the results, one that mitigates the impact of noise from other programs
running on the machine.
1.3
Problem we are solving
The primary problem that this project aims to address is:
Does Pyperf enable CPython developers to make accurate decisions about
the performance of their implementation in unstable environments?
1.4
Solution
This project focused on analysing the approach to benchmarking in various programming
languages, with a specific emphasis on Python, particularly the CPython implementation.
It involved examining existing mathematical models of program execution and adapting
them to fit the CPython implementation.

Performance benchmarking
The following chapter focuses on benchmarking from a theoretical perspective. First, it
describes what benchmarks are and discusses types of benchmarks that exist. Following
this, the chapter focuses on performance benchmarking and illustrates how it can be
represented using mathematical notation.
2.1
Benchmarking
Benchmarking is a process, which measures improvement or regression. The purpose of
executing performance benchmarks is to see, whether a change to the system changed
its performance and also by how much.
There are many different types of benchmarks, but the performance benchmarks can be
divided into the following groups, Microbenchmarking and Macrobenchmarking.
2.1.1
Microbenchmark
A microbenchmark, in the world of computer programming and systems analysis, is a
type of test intended to measure the performance characteristics of a small unit, such as
a single component or task.
This type of benchmarking focuses on individual functions or procedures within a given
program or algorithm. Instead of evaluating the performance of an entire system or apยญ
plication, which might involve a lot of moving parts, a microbenchmark narrows down the
field of examination to a specific code snippet, module, or operation. This approach can
offer valuable insights into the behavior of individual components in a system and provide
detailed information about their efficiency and execution time.
However, while microbenchmarking can be highly informative under certain circumstances,
it also has potential limitations. A key pitfall is that these tests can sometimes prove misยญ
leading or inaccurate if they do not adequately simulate realistic scenarios or account for
factors such as compiler optimizations, garbage collection, caching, etc.
2.1.2
Macrobenchmark
A macrobenchmark is a type of a performance test that focuses on evaluating realistic
tasks, often involving several components or entire systems. Unlike microbenchmarks,
which focus on single, isolated units or components, macrobenchmarks account for the
complexity and full scope of realยญworld tasks.
Macrobenchmarks are designed to replicate the regular use cases of an applications or
systems. These are the typical scenarios and functions that a normal user might perform
on a daily basis. Macrobenchmarks look at the performance of a system as a whole.
Macrobenchmarks provide insight into how well an application or system performs under
conditions that reflect its intended realยญworld usecases. The results of these tests are

therefore more interesting to developers and end users alike. They can see how the
application or system performs under realistic conditions, which is usually more relevant
to their needs.
2.1.3
Benchmarking environment
The full control of the whole benchmarking system allows for the best precision when it
comes to performance benchmarking. Using a single machine, with a constant baseยญ
line performance, which only runs the bareยญbone environment for the benchmarks, can
produce statistically reliable benchmarking results.
The environment must remain stable throughout the entire execution time of the benchยญ
marking process. To ensure, that the results, generated during the benchmarking process
are comparable, it is necessary for them to be generated in an environment that remains
consistently stable and under a constant load not only during one execution but throughout
all of the executions of the benchmarking process. Otherwise, the comparison of results
may lack statistical significance due to the systemยญinduced variance.
The environment mentioned above can also be referred to as an โidealโ environment for
executing benchmarking. However, it significantly differs from the environment typically
used by developers of the benchmarked software, as well as from the environment used
by software users for executing the software.
If the benchmarking process tests the performance of a specific version of the software
and it was executed on the same stable environment, then the results produced by the
benchmarking process can be used for comparisons between versions.
This approach provides a significant level of flexibility. For example, if the performance of
each new minor release of a software is consistently tested using the same benchmarking
process in the same environment, the results can be stored in one centralized location.
This enables developers or users of the software to easily compare the performance of
the current version with older versions at a later time. Since the benchmarking process
remains unchanged, there is no need to execute the benchmarking process with both
versions of the software and manually compare them. Instead, individuals can directly
access and download the desired results for comparing the performance differences.
The approach to benchmarking software described earlier may appear promising in theยญ
ory, but it can be significantly challenging to implement in practice, particularly when
benchmarking is meant to be an integral part of the development process. There are
several practical considerations to address.
Firstly, maintaining a dedicated rig solely for benchmarking purposes can be costly and
may not scale well, especially when benchmarking needs to be performed regularly throughยญ
out the development of the software.
Additionally, relying on a single machine for executing benchmarks poses limitations. For
instance, it does not allow for concurrent benchmarking by multiple users. This can be

2.2
A/A testing.
A/B testing is the process of executing tests to compare two different versions in order to
determine which one is better. The definition of โbetterโ varies depending on the aspect
being tested. When it comes to performance benchmarking, the better version is the one
that demonstrates higher performance, resulting in faster execution of the benchmarking
suite.
However, to verify if the tests themselves are producing meaningful results, A/A testing
can be used. In A/A testing, the same version is compared to itself. The expected outcome
of both tests is supposed to be identical. If the compared values generated by the same
version are not the same, it indicates an issue with the way the tests are executed. This
instability can be attributed to incorrect testing methodologies, instability in the testing
environment, or instability caused by the benchmarked version.

2.4
Statistics
A Studentโs tยญtest is a statistical hypothesis test that is used to either accept or reject the
null hypothesis. The null hypothesis states that the difference between the means is zero.
The tยญtest can be either oneยญsample or twoยญsample, depending on whether the mean is
defined in the null hypothesis or whether the difference of means is compared between
samples.
When using the twoยญsample tยญtest, the samples being compared can either be dependent
on each other or not. Based on this, the unpaired tยญtest or paired tยญtest can be used.
The unpaired tยญtest is used to compare two unrelated samples. The paired tยญtest is used
to compare two samples where each observation in one of the samples has a related
observation in the other.
Both types of tยญtests calculate a tยญvalue, which represents the difference between the
means relative to the variability in the data. The tยญvalue is then compared to a critical
value based on the degrees of freedom and desired significance level to determine if the
difference is statistically significant.
If the calculated tยญvalue is higher than the critical value, it suggests that there is a significant
difference between the means of the two samples. In such cases, the null hypothesis is
rejected, which assumes that there is no difference between the previously calculated
means. If the calculated tยญvalue does not exceed the critical value, the rejection of the null
hypothesis fails, which means that there is not enough evidence to conclude a significant
difference. However, the lack of evidence does not translate to evidence of absence.
The tยญtest assumes that the data are approximately normally distributed and that the variยญ
ances of the two samples are equal.

The sources of variability
Because computers are very complex machines, there can be multiple different sources of
variability or instability that contribute to the unpredictable behavior of computer systems.
Understanding these sources is therefore essential for good benchmarking process.
2.5.1
Hardware
Caching is used in computers to store frequently accessed data closer to the processor
in order to reduce the latency of reading the data from further locations like memory or

even worse, the storage itself. However, caching introduces its own source of variability
due to the unpredictable behaviour of cache hits and misses. The efficiency of caching
can vary depending on factors such as cache size and data access patterns.
Thermal throttling is a functionality which prevents overheating. Many modern procesยญ
sors employ thermal throttling mechanisms that dynamically adjust the processorโs clock
frequency based on the temperature of the processor. This variability in processor speed
can impact application performance, especially in resource heavy tasks. The computer
manufacturers, want to create machines, that are thin, light and with very powerful procesยญ
sors. However it is afterwards an issue to cool down the machines properly. Especially if
the machines are supposed to run quietly.
Turbo mode is used in order to quickly help with short, resource intense tasks. Modern
processors, which especially target normal customers, can run in a turbo mode for a very
short periods of time. For instance when opening a new application, the user wants the
application to be in a usable state as soon as possible. Running the Turbo mode for
extended periods of time would generate too much heat and afterwards result in thermal
throttling with a performance degradation.

usually designed either for users or for servers. There are many differences between the
operating systems which run on servers and the ones used by normal users. In servers,

that the system is highly responsive and scalable. In the userยญoriented operating systems,
the scheduler needs to provide a smooth and responsive user experience, while also enยญ

3.3
Interpreted and Compiled programming languages
When it comes to modern programming languages, there are two distinct ways on how
the language interacts with the underlying infrastructure. The Compiled languages and
the Interpreted languages.
In the beginning we have a source code written in a programming language.
โข Compiled languages: When it comes to compiled programming languages, the
source code written in a language is passed to a compiler as an input. The compiler
transforms the input into a binary. The binary is a machine code, which can only
be executed on a computer with the same instruction set. This means that there
are two distinct state transitions of the code of the program. The compilation of the
source code into a binary. And afterwards the execution of the binary. Therefore it
is possible to benchmark the compilation time of the program but also the execution
time of the program separately. This separation allows for performance instability
during the program execution, but also for easier analysis or understanding of the
benchmark afterwards the benchmarking is finished.
โข Interpreted languages: When it comes to interpreted programming languages , the
source code is passed into a program called interpreter, which handles everything.
This means that the interpreter has to take care of a lot of stuff. Usually the code
goes through multiple different stages. These stages verify its syntax but also try to
do various optimizations. From the perspective of state transition there is only one,
and it is between the source code and the running interpretation of the code.

3.4.1
Compilator
The first step in Python compilation is parsing, which involves splitting the input (string
or file) into tokens. Python uses a Parsing Expression Grammar (PEG) parser [17, 18],
which however, does not take as an input a stream of characters but instead tokens.
The parser itself is generated using the Pegen parser generator [19]. This means that the
generator receives a grammar specification as an input, and generates a code for parser,
which is able to parse the predefined grammar. The grammar is defined by the Zephyr
Abstract Syntax Definition Language (ASDL) which provides a structured representation
of the program.
The PEG parser was introduced in Python 3.9 and one of its perks is, that it immediately
allows for generation of Abstract Syntax Tree. Previously CPython used a LL [20] parser
which only parsed the tokens and afterwards the AST (Abstract Syntax Tree) generation
routine, which was tightly coupled to the LL parser itselft, had to create the AST generated
from the tokens outputted by the parser.
After the AST is generated, the compiler translates it into bytecode. The bytecode is
a low level representation of the code, which is not bound to a specific architecture of
processors. This allows for wider variery of supported machines. The whole process
of bytecode generation is fairly complex and because of it, the bytecode is generated in
multiple stages. The first version of bytecode does not contain any jump targets yet.
After the first version of bytecode was constructed, the Control Flow Graph (CFG) is genยญ
erated from the AST. The CFG is a directed graph, which represents the sequences in
which the bytecode instructions are supposed to be executed. Each block in the CFG is
called a basic block and must be executed from start to finish. If there is supposed to be a
jump in a code, like for instance the โgotoโ instruction, the code must be split into multiple
building blocks.
The final bytecode is a flattened version of the CFG obtained using postยญorder depthยญfirst
search. The flattened CFG is stored in a .pyc file, which is the compiled representation of
the Python code. [21]
According to Pep 617 [22], which introduced the PEG parser, the compilation time of the
whole Python standard library takes only 3 seconds. The library at that point of time
consisted of 1,641 files and 749,570 lines. If such a large project compiles in such a short
time, than the more important thing to benchmark is the bytecode interpretation.

3.4.3
Garbage Collector
Garbage collection is an essential feature of Python that helps manage memory efficiently.
The biggest advantage of using a garbage collector is, that users do not have to focus
on cleaning the objects when they are supposed to be deallocated. Of course users still
have the option to delete specific objects manually, but it is not a requirement.
Python employs two different garbage collectors to manage the memory, reference countยญ
ing and cyclic garbage collector.
The reference counting technique keeps track of the number of references to an object,
and when the reference count falls to zero, the object is immediately deallocated.
The cyclic garbage collector is used to identify and collect objects that have circular refยญ
erences, where objects reference each other in a loop, and neither is reachable. Python
contains a lot of objects which are defined in these circular references, for instance exยญ
ceptions are referring to the block where they were caught and wise versa. [9]
The cyclic garbage collector is a generational garbage collector that groups objects into
different generations based on their age. Objects are created in the youngest generation,
and as they survive collection cycles, they move to an older generation. The garbage
collector focuses on the younger generations as most objects in CPython tend to have a
rather short lifespan. The older generations are collected less frequently, which improves
the garbage collectorโs efficiency.
The first generation is collected if the number of allocated objects minus the number of
deallocated objects exceeds a specific threshold. If the number of collections of the first
generations exceeds a specific threshold, the second generation is collected. The final
third generation is collected more sporadically and its not only triggered based on the
number of collections of previous generations.
The previous generations usually iterate over similar amounts of objects and the objects
do not stick around. Whereas the last generation can have a lot of long lived objects,
which could slow down the garbage collection but also the runtime of the whole program.
Because of this, the last generation is collected only if a specific ratio of objects in the last
generation is exceeded. [23, 24]
Python optimizes garbage collection by using various techniques such as reusing fields to
save memory and delayed tracking. Not all objects in Python are tracked by the garbage
collector, for example, integers and strings are immutable objects that are not tracked,
which helps improve performance.
Some more complex objects like tuples or dictionaries which only contain immutable obยญ
jects are not being tracked as well. The option to track or not to track specific objects can

3.4.4
Lifecycle of an object
In CPython, the Python memory manager is responsible for memory management [25].
The raw memory allocator is part of the memory manager, which ensures that the inยญ
terpreter has enough memory to operate. The memory manager accomplishes this by
communicating with the memory manager of the operating system and using the malloc,
calloc, realloc, and free functions defined in the C programming language.
All CPython objects are allocated on a private heap. The allocated heap, also known as
an arena in CPython, is segmented into smaller chunks called pools. These pools contain
objects that can be either fixed or variable in size.
The creation of a new object is handled by one of the several objectยญspecific allocators
that work with the private heap managed by the memory manager. Different types of
objects have different requirements. For instance, when working with integers, it is not
important if they are as small as possible, so they can have a fixed size. Having a fixed
size has some perks. For instance, the allocation process is much faster because the
objectยญspecific allocator does not have to search for empty space of a specific size in the
pools.
The deletion of an object is triggered by one of the garbage collectors. As previously
discussed in 3.4.3, the deletion of an object is triggered either by the reference count of
an object dropping to zero, allowing it to be removed, or by the cyclic garbage collector
deciding that the object is no longer of any use.
The process of deallocating an object involves freeing the memory occupied by the object
and potentially calling the objectโs destructor if it has one. Freeing the memory does not
mean calling the underlying free function because that would return the previously used
space back to the operating system. Therefore, the CPython memory manager only tags
the space as free, and afterwards, it knows that the space can be repurposed for other
objects or merged with adjacent blocks.
3.4.5
Global Interpreter Lock
One of the most significant parts of CPython is the Global Interpreter Lock (GIL), which
has been a topic of discussion and controversy for many years.
The GIL is a mechanism which ensures that only one thread executes Python bytecode at
any given time, preventing race conditions that can arise from multiple threads modifying
shared data simultaneously. It is very similar to mutex (mutual exclusion lock) that is held
by the thread that is currently executing Python code, and any other threads that try to
acquire the lock are blocked until the lock is released. It is however implemented as a
binary semaphore and not a normal mutex. [26, 27]
The GIL has a significant impact on CPythonโs performance, especially in multiยญthreaded

It must be noted that CPython still allows for parallelization, it is possible to use multiple

in Python, including reclaiming unused memory and preventing memory leaks. The GIL
plays a critical role in the GCโs operation, as it ensures that the GC can traverse the object
graph safely and without interference from other threads.

Pyperf handles the execution of benchmarks. To improve the stability of the benchmarking
process, pyperf executes 20 runs 5. Each run is executed by a worker. The workers
are spawned sequentially but as separate subprocesses [40]. Each worker executes
three iterations 6. The output from an iteration is a โvalueโ. Therefore, the benchmark is
represented by a total of 60 values.
The โvalueโ, which is the output from a single iteration, represents the average time taken
to execute the benchmarking function 7.
To obtain this value, the benchmarking function must be executed multiple times within
a single iteration. The number of times the function is executed can be calculated by
multiplying the number of inner loops and the number of outer loops.
Inner loops refer to the number of times the benchmarking function is executed within a
single benchmarking function execution. It can only be defined in the benchmark function
itself. Usually, the desired output of a benchmark function execution is the time taken to
execute all of the inner loop iterations.
The number of inner loop iterations can be passed to pyperf 8. The value, which is an
output of the benchmark iteration, is adjusted to represent the average time it took to
execute a single inner loop. If the number of inner loops is not passed to the benchmark,
pyperf assumes that the number of inner loops is one.
The number of outer loops, on the other hand, is calculated at runtime. By default, each
iteration is supposed to run for at least 100 ms 9.
The number of executions should be large enough to reduce the effect of random factors
like randomized address space layer (ASLR) and the Python randomized hash function
[41].
After executing a benchmark, pyperf generates a JSON representation (8) of the benchยญ
mark execution. This JSON file contains detailed information about each worker run 10.
The information includes CPU temperature, frequency, and duration, as well as the exeยญ
cution time of warmups and each iteration. Detailed output from benchmarks is important
for analyzing the performance of the programming languages.

Experiment
The main research question that this thesis tries to answer is:
Does Pyperf enable CPython developers to make accurate decisions about
the performance of their implementation in unstable environments?
To answer the main question accurately, it needs to be divided into several smaller quesยญ
tions. These questions are important to answer in order to determine whether Pythonโs
performance is tested properly. Additionally, they help identify potential improvements
that can be made to the overall benchmarking process.
RQ1: Can pyperf reliably reject A/A tests as insignificant?
To which the answer is
no. When utilizing pyperf to test the significance of results in a relatively stable environยญ
ment with over 10 independent runs of the benchmark suite, it was observed that 60% of
the results were flagged as statistically significant. Ideally, the goal would be to have 5%
of the results marked as statistically significant. In the later sections 5.2.3 and 5.2.4, we
propose an improved approach for collecting data for future comparisons.
To understand what went wrong, we have investigated the following questions:
โข RQ1.1: Is the warmup process used by pyperf flawed? In order to produce repliยญ
cable results, the warmup phase is vital. The length of the warmup period may vary
depending on the machine used for testing purposes. A flawed warmup process can
significantly impact data collection and, consequently, the comparison of results.
Despite pyperfโs warning that in certain scenarios the warmup process should be
extended 1, the experiments conducted revealed that a significantly longer warmup
process is required (5.2.1). A full execution of the entire benchmarking suite as a
warmup demonstrated that the later measurements showed significantly improved
stability compared to using the warmup approach provided by pyperf alone.
โข RQ1.2: Is it possible to improve the analysis of results using different statisยญ
tical model? There are statistical models that aim to improve the stability of results
after data gathering is completed. Therefore, we investigated whether these means
can be used in the Python benchmarking process. Based on the executed experยญ
iments, we noticed that the number of statistically significant comparisons actually
increased instead of decreasing as initially hoped. As a result, further investigation
was performed to understand the underlying reasons behind these results.

Based on the insights
gained from analyzing the Python benchmarking process, we have deduced that employยญ

produced by the benchmark.

one would expect.

6.1
Robust benchmarking in noisy environments
The paper Robust benchmarking in noise environments [45] presents a model of program
execution runtimes. The authors of the paper assume that a program can be viewed as
a set of executed instructions and delays. The delays are introduced by various factors.
These delay factors can include timer error, OS jitter, and other environmental fluctuations.
I = instruction
In = Nth instruction
D = delay
Dn = Nth delay
program = [I0, D0, ..., In, Dn]
The key contribution of the paper is the proposal of a mathematical model that can account
for these delay factors in a statistically rigorous way. The model provides understanding to
more accurate measurements of program execution times in noisy environments, where
fluctuations can have a significant impact on the results of benchmarks.
Another important contribution of the paper is its recommendation to use the minimum
aggregation function instead of the mean together with standard deviation, which are more
commonly used in benchmarking.
The authors argue that in noisy environments, mean and standard deviation can be heavยญ
ily influenced by outliers, which can result in misleading benchmark results if the benchยญ
mark has not been executed enough times. They showcase how mean, trimmed mean
and median are also capturing bimodality. In contrast, the minimum aggregation function
can be less sensitive to outliers and can provide a more reliable estimate of the programโs
true performance if used in correct way. As the number of benchmark runs increases, the
use of the minimum aggregation function becomes more and more useful and therefore
also stable.
Extreme outliers in noisy environments tend to have values higher then the mean, esยญ
pecially when it comes to performance benchmarking. In such cases, it is often more
relevant to focus on the minimum value when analyzing benchmark results because it is
not influenced by these outliers.

once to download and install everything at once.

whether adjusting the niceness value can be utilized to improve the benchmarking

benchmarking process. This topic would require further inspection and also comparยญ

it may be beneficial to make greater use of statistical methods. By applying rigorous
statistical techniques, we can better account for variability in the data and obtain more
accurate results across different hardware configurations.
