Abstract
One of the more common development projects within Robotics is decision making, and
in order to purely focus on the development of this it would be a huge advantage to
be able to take the physical robot out of the equation during the development process.
Therefore testing the viability of different simulation environments for robots is an im-
portant task to be done, in order to make robot developing more eﬀicient.
The simulation environment and the decision making process that is implemented
and tested in this project is created with ROS using Gazebo as the simulation tool. It
is tested by setting up a world, and then by spawning and navigating a robot in this
environment. With a camera the robot recognises different objects in the world. This is
used to make the robot perceive its surroundings. The perception is used as the basis
for the robot’s comprehension of the situation, and projection of what it might lead to.
With this input the robot decides what to do next.
The testing of the above leads to the conclusion that it is feasible to create a simulated
environment with ROS and Gazebo, which can be used to create and test a decision
making process for a robot.

CHAPTER 1
Introduction
One of the more common development projects within Robotics is decision making, and
in order to purely focus on the development of this it would be a huge advantage to
be able to take the physical robot out of the equation during the development process.
Therefore testing the viability of different simulation environments for robots is an im-
portant task to be done, in order to make robot developing more eﬀicient. This project
will focus on the simulation environment Gazebo with ROS (Robot Operating System)
as the framework for writing the robot software. The software that will be developed
will have the focus on simple decision making.
1.1
Objectives
The objective of this project is to build a modular structure to support situation aware-
ness and cognitive behaviour for a mobile robot. This entails
1. Learn ROS and Gazebo
2. Use ROS and Gazebo to setup a simulation environment
3. Get a robot to make simple decisions in this simulation environment
4. Make the software modular so that it can be used for further development
1.2
Limitations
Because of the global pandemic Covid-19, and the political decision to have people work
from home, the hardware that this project was made on were quite a bit out dated,
and therefore the simulation was simplified by using static objects. Such as using a box
instead of having a humanoid move around in the simulation environment.
The pandemic was also the reason for not testing the outcome of the project on a
physical robot.

2.3
ROS
The Robot Operating System (ROS) is a framework for writing robot software.
It
includes a huge collection of tools, libraries, and conventions that simplify the task of
creating complex and robust behavior for a wide range of robot platforms.
Figure 2.1: ROS Logo.
2.3.1
ROS History
Sometime before 2007, the first parts of what eventually became ROS began to come
together at Stanford University. The background was that it was discovered that many
roboticists were limited by the diverse nature of robotics, which meant that an excellent
software developer might not have the hardware knowledge required. Furthermore, hard-
ware experts might not have suﬀicient software developing knowledge etc. Therefore, a
baseline system was created that provided a starting place for others to build upon. In
2007 Willow Garage, a Company that developed Robot Hardware, began developing the
==PAGE:10==
4
2 Analysis
PR2 and decided to use this as the base for the software to control it. Therefore, they pro-
vided significant resources to extend this concept much further and create well-tested
implementations. In 2012 Willow Garage created Open Source Robotics Foundation
(OSRF). In the years since, OSRF took over primary development of ROS and a new
version has been released every year. In this project the ROS Melodic release, that was
released in 2018 is used.
ROS was designed to be as distributed and modular as possible, so that users can
use as much, or as little of ROS, as they desire. The modularity of ROS allows anyone
to pick and choose which parts are useful for them and add their own self implemented
specialized parts. Any group can start their own ROS code repository and maintain
full ownership of it. They do not need permission. They can choose to make their code
publicly available and receive technical feedback and improvements like all open source
software projects. This means that ROS today is developed at multiple institutions and
for multiple robots. The distributed nature of ROS is also the foundation for the huge
community of user contributed packages that add a lot of value on top of the core ROS
System. At present there are more than 3000 packages in the ROS ecosystem.
2.3.2
Guiding Principle of ROS
The guiding principle for today’s ROS users is “do not reinvent the wheel”. A lot of
very skilled people have been programming robots for many years. Therefore, why not
use the tools they have created and tested, and only create your own code where special
requirements demand it? This principle is also used in this project wherever possible
and feasible.
2.3.3
ROS as the Framework
As mentioned ROS is a framework for writing robot software. ROS is not an actual
operating system since it operates on top of conventional operating systems, of which
Ubuntu Linux is most widely used and has the most comprehensive support and set of
tools. The Ubuntu Linux version that is used in this project is 18.04. ROS is more of a
middleware, a low-level framework based on an existing operating system. It is mainly
composed of two things:
1. A core (middleware) with communications tools
2. ROS packages
2.3.3.1
ROS core and communications tools.
ROS core is a collection of nodes (programs) that are pre-requisites of a ROS-based
system. It is necessary to have a roscore running in order for ROS nodes to communicate.
Roscore mainly consists of three components:
==PAGE:11==
2.3 ROS
5
1. Communication infrastructure
2. Robot specific features
3. Tools
1. Communication infrastructure
The primary mechanism for ROS nodes to ex-
change data is sending and receiving messages. Messages are transmitted between nodes
using one of these methods:
(a) Topics
(b) Services
(c) Actions
a. Topics are mainly used for sending data between nodes. Topics communication
is asynchronous. When using Topics nodes in general are not aware of who they are
communicating with. Nodes that are interested in data subscribe to the relevant topic;
nodes that generate data publish to the relevant topic. There can be multiple publishers
and subscribers to a topic.
Figure 2.2: ROS Subscribers and Publishers communicating through the roscore.
b. Services allows you to create simple client/server communication between nodes.
Services communication is synchronous. A ROS node(server) offers a service using a
string name, and a client calls for the service sending the request message and waits for
the reply.
c. Actions are based on topics. They create an asynchronous client/server archi-
tecture, where the client can send a request that takes a long time to perform, such
as asking the robot to move to a new location. The client can asynchronously monitor
the state of the server, and can cancel the request at any time. Actions are often also
referred to as Preemptable Remote Procedure Calls.
==PAGE:12==
6
2 Analysis
To use any of these methods one either defines one’s own messages or use one of
the many messages already defined in ROS. A message is a simple data structure for
the data that are to be communicated from one node to one or more other nodes. If
predefined ROS messages are to be used, the appropriate libraries containing the format
of the libraries need to be included in the code. ROS also allows the creation of global
settings, named parameters. They are used to give multiple nodes access to the same
settings.
2. Robot specific features
In addition to the core middleware components, ROS
provides common robot-specific libraries and tools, such as these:
(a) Standard Message Definitions for Robots
(b) Robot Geometry Library
(c) Robot Description Language
(d) Diagnostics
a. Standard Message Definitions for Robots.
ROS libraries today have a set of standard message formats that covers most of the
common use cases in robotics. This includes definitions for geometric concepts like poses,
transforms and vectors for sensors like cameras, inertial measurement units (IMU’s) and
lasers.
b. Robot Geometry Library.
In many cases it is diﬀicult to keep track of where different parts of a robot are with
respect to each other. This is necessary if you for instance want to combine data from a
laser with data from a camera. You need to know where both sensors are in a common
reference frame. In ROS this is done with the tf (transform) library, which keeps track
of where everything is in the robot system.
c. Robot Description Language.
It is always necessary to describe the robot in a machine-readable way. ROS provides
a set of tools for helping doing so. This makes it possible to describe the robot in such
a way that it can be understood by the rest of the ROS system. This format is URDF
(Unified Robot Description Format).
An URDF file consists of an XML document,
where the physical properties of the robot are described, including properties such as
sizes and locations of wheels, locations of sensors and visual appearance of each part
of the robot. This also makes it possible to create visualizations of the robot, that can
be used in simulations with tools such as Gazebo and motion planners such as DWA
planner.
d. Diagnostics.
ROS provides a standard way to produce, collect and aggregate diagnostics for the robot.
This makes it easier to see the state of the robot and determine how to address problems
that might arise.
==PAGE:13==
2.3 ROS
7
3. Tools
ROS includes tools for inspecting and visualizing the state of the system
being developed, for plotting and debugging etc.
The underlying publish/subscribe
mechanism allows the introspection of the data flowing through the system and makes
it possible to comprehend and debug issues as they occur. The ROS tools includes both
graphical and command line utilities.
Command-Line tools.
All ROS core functionality and inspection tools are accessible with one of the more than
45 command line tools. There are commands for launching of nodes, inspecting topics,
services and actions etc.
rviz.
rviz provides general purpose, three-dimensional visualization of many sensor data types
and any URDF-described robot. rviz can visualize many of the common message types
provided in ROS, such as laser scans and camera images. With information from the tf
library it shows all the sensor data in a common coordinate frame. rviz can also show a
3D rendering of the robot.
ROS launch files.
ROS launch files facilitates the starting up of several ROS nodes and roscore with just
one roslaunch command. Launch files also contains parameter settings for each of the
nodes. Launch files are written in a special XML format, and are used extensively in
most ROS packages.
2.3.3.2
Packages
A robot application using ROS is broken into many packages and each package contains
one or more independent nodes. The nodes communicate with each other using topics,
services and actions. A simple example from roboticsbackend.com [2] is shown below,
where red arrows symbolize communications using roscore.

A package can for example contain ROS nodes, a ROS-independent library, a dataset,
configuration files, launch files etc. The purpose of these packages is to provide useful
functionality in a simple manner so that software can be easily reused. In general a
ROS package follows a simple principle. That is, it should have enough functionality
to be useful, but not too complex, which would make the package too computationally
demanding and diﬀicult to use for other software. Over the last decade a large amount
of ROS packages have been developed by ROS enthusiasts, institutions and companies.
Many of these packages have been made publicly available so that other ROS users can
use them in their projects. An important attribute for packages is that they work across
programming languages, meaning it is possible to have a mix of for example C++ and
Python nodes. This is possible since all communication is done through the roscore.

2.4
Simulation Program
When Choosing a simulation program for simulating robotics together with ROS, there
are three main options V-REP, ARGoS and Gazebo. The choice of which to use should
be made from a perspective of ”do I need performance or model detail, or something in
the middle”, where:
1. V-REP is a feature rich simulation environment including a large model library,
real time mesh manipulation and many more.
2. ARGoS is the performance focused option that is usually used for projects where
many moving parts are involved.
3. Gazebo is somewhere in between the previous two. It has the capability to use
complex 3D meshes and physics engines.
Even though Gazebo is predetermined to be used, it is still the most ideal of of three
above to be used as the project was made on limited hardware. [1]
2.5
Gazebo
Gazebo is an open-source 3D simulator that has the ability to make accurate, realistic
and eﬀicient simulations of robots in complex indoor and outdoor environments. Gazebo
is widely used in this project.
Figure 2.5: Gazebo Logo.
2.5.1
Gazebo History
Gazebo development began in 2002 at the University of Southern California. Gazebo
was designed to evaluate algorithms for robots. The development continued here until
2009 where Willow Garage integrated ROS and PR2 into Gazebo. This in 2011 led
Willow Garage to provide support for the development of Gazebo. In 2012, Open Source
Robotics Foundation (OSRF) spun out of Willow Garage and became the supervisor of
the Gazebo project. This is still the case today. In this project version 9 of Gazebo has
been used.

2.5.2
What is Gazebo
With Gazebo it is possible to create a 3D scenario on the computer with robots, obstacles
and other objects, and a robot can be tested in diﬀicult or even dangerous environments
without any risk of harming the robot. In many cases it is also faster and easier to
run a simulator than starting the whole scenario with the physical robot. For all robot
developments it is imperative to test the robot application, for instance in respect to
error handling, battery life, localization and navigation.
2.5.2.1
Gazebo Components
Gazebo is built with a distributed architecture and has separate software libraries for
1. Simulation of physical systems, such as rigid bodies
2. Rendering
3. User interface
4. Communication
5. Sensor generation
Additionally, Gazebo provides two programs for running simulations:
1. A server, gzserver that simulates the physics, rendering, and sensors
2. A client, gzclient that is the graphical interface for visualizing the simulation
The following paragraphs describe the main items involved in performing a Gazebo
simulation.
Gazebo server
The server is the focal point of Gazebo. It reads the world file designated on the Gazebo
startup command line, and simulates the world using the designated libraries for physics
and sensor generation.
Gazebo Graphical Client
The Gazebo Graphical Client connects to a running gzserver and visualizes the robot
and other objects in the simulation. It can also be used to add objects to the world and
to move objects.
World files
A world file describes all the elements in the simulation, including robots, lights, sensors
and static objects. It is an XML file formatted in SDF (Simulation Description Format).
Model files
Model files defines objects that are used to populate the world, in which the simulation
==PAGE:17==

11
is to be run. A model file can describe everything from a simple shape to a very complex
robot. They are just like world files XML files in SDF format. Including an object in
the world is simply done by referencing the model file in the world file.
Gazebo Plugins
With Gazebo plugins it is possible to control how Gazebo runs and to control the prop-
erties of robot models, the sensors and the world. Plugins are small software programs
written in C++. The plugins can be classified into 6 groups:
1. World plugins
2. Model plugins
3. Sensor plugins
4. System plugins
5. Visual plugins
6. GUI plugins
Different components of Gazebo control each group of plugins. For instance, the
world plugin is connected to a specific world and makes it possible to control the proper-
ties of this specific world. Likewise, the model plugin allows you to control the properties
of a specific model, the sensor plugin gives you control over a specific sensor (such as a
camera) and its properties, etc. Plugins can be loaded in two ways:
1. From the command line
2. Specified in an SDF file
2.5.3
Gazebo-ROS packages
Gazebo-ROS packages is a set of ROS packages that connect ROS and Gazebo and
makes it possible to simulate a robot controlled by ROS with Gazebo. It uses ROS
services, ROS messages and the dynamic reconfigure feature with which you can update
parameters at runtime without restarting the node.

2.6
Cognitive Behaviour
A major part of cognitive behaviour for a robot concerns being able to act in the world
on the basis of its perception of its surroundings. This part is also called decision making,
==PAGE:18==
12
2 Analysis
and this is what will be dealt with in this project. This will be referred to as decision
making in the rest of the thesis. Decision making is defined in the following.
First we will look at human decision making in order to be able to compare with
decision making for a robot and pinpoint the differences.
Human decision making is a complex process that is often defined as the outcome
of evaluation of alternative options in terms of probability and the value that a person
sets on the outcome of the different options. This theory is known as the Subjective
Expected Utility theory. In general the process that a human goes through when making
a decision can be put into 7 steps [24].
1. Identify the decision to be made
2. Gather relevant information
3. Identify the alternatives
4. Weigh the evidence
5. Choose among the alternatives
6. Take action
7. Review your decision
Robot decision making is simplified version of the human decision process especially
when AI is not part of decision. In this project robot decision making will be defined as
the following process
1. Gather information (sensor inputs)
2. Identify the situation that the robot is in
3. Identify possible actions that the robot can take
4. Perform the action with highest priority
As seen there is no evaluation process since AI is out of the scope of this project.
The focus will be on this simple decision making process. This can also be described in
terms of a finite-state machine (FSM). An FSM is an abstract machine that functions
by being able to change from one state to another in response to the inputs that are
given. In terms of a robot that could for example be when the robot sees an obstacle in
front of it with its sensors. The robot uses the sensor data to figure out which selection
of a finite number of states the robot can change to in order to avoid the obstacle in
front of it. Based on its input, the robot chooses the optimal state that it can transition
to, and then performs the transition [25].

6.1
Introduction
This chapter will describe how the robot will navigate in a Gazebo world while avoiding

6.2
Robot Navigation
This section gives a short description in general terms of how a robot is moved around in
a simulated environment. Details of how this is done with ROS and Gazebo is described
in the next section.
In order to make a robot navigate in an environment, a map of the area is neces-
sary. This map can be created in many different ways. In this project the laser-based
Simultaneous Localization and Mapping (SLAM) method is used. SLAM is a method
that that uses laser sensor data and pose data to create a 2D occupancy map (like a
building floor plan). The numbers that were shown on the map in figure 4.5 can not be
seen since they have not been given any mass in Gazebo.
Figure 6.1: SLAM Generated map.
==PAGE:36==
30
6 Navigation
Once the map is generated, it is used to generate a global cost map using the occu-
pancy information from the map. The cost map contains a set of values, that indicates
how big the chance of hitting an obstacle in the area is. The higher the cost, the higher
the chance of hitting an obstacle.
Figure 6.2: rviz visualization of the cost map.
The cost map is visualised in the figure above. As seen, the color is gray far from
any obstacle, such as a wall. Closer to an obstacle the color turns blue, and even closer
to an obstacle it turns more and more red. This change in color illustrates higher and
higher cost, indicating a higher chance of hitting an obstacle.
Next the global plan that shows how to reach the goal is generated given the size of the
radius of the robot, the robot’s current position and the goal position. The generated
global plan avoids all the obstacles known from the cost plan.
==PAGE:37==
6.2 Robot Navigation
31
Figure 6.3: Map showing the global plan as a curved line.
The figure illustrates that the global plan is a path that avoids all the obstacles.
Next a local cost map for the robots vicinity needs to be generated. It is based on
the current position of the robot, the latest sensor inputs and the global cost map. The
values of the local cost map are interpreted in the same way as for the global cost map.
The difference between the global cost map and the local cost map is that the local cost
map only covers the area that are covered by the laser scanners range. In this area the
local cost map also includes costs for obstacles that might have been introduced after
the 2D occupancy map was generated. The local cost map is illustrated in the following
figure by the blue and red square.
Figure 6.4: Map showing the local cost map as the blue and red square around the
robot.
==PAGE:38==
32
6 Navigation
The more red a position is illustrated, the closer it is to an obstacle. With the global
plan and the local cost map the Dynamic Window Approach (DWA) is used to create
the local plan which produces velocity commands that are to be send to the wheels of the
robot. The DWA planner works by performing simulations to estimate what would hap-
pen if a certain velocity command were applied for a short period of time. It then scores
all the different velocity commands it has simulated based on metrics such as proximity
to obstacles, proximity to the goal and the proximity to the global path. In this process
it discards all the velocities that would make the robot collide with any obstacles. The
DWA planner then picks the highest scoring velocity command from among the possible
velocity commands and sends it to the robot controller that performs it. This is then
repeated until the robot reaches its goal.[9]

The implementation of the navigation procedure described in the previous section is done
using the ROS navigation stack [20] combined with the ROS package gmapping, which
implements the OpenSlam’s gmapping[14]. The map is drawn by gmapping based on
the data published by the robots laser distance sensor. These data are published while
driving the robot around in every corner of the world with a simple teleoperation node,
i. e. controlling the direction and speed of robot with the keyboard.
When the map is available from the gmapping, the navigation stack is launched with
a launch file. In this project the launch file looks like this.

When move_base receives a goal by subscribing to the ROS message
geometry_msgs/PoseStamped and receives the robot’s current position it uses the
global_planner to create a global plan for reaching the goal without hitting any known
obstacles along the planned route. The global_planner uses the information given from
the global cost map that is generated from the 2d occupancy map, which it requests
from the map server.

from the current position of the robot, latest sensor inputs and the global cost map. The
current position of the robot is provided by the AMCL.

avoid all local obstacles and will advance the robot closer to its goal. The local planner
publishes the associated velocity commands. The base controller subscribes to these
velocity commands and applies them to the wheels of the robot for a short period of
time. The local_planner generates a new local plan to repeat the process until it reaches
the goal.
If the robot along this process of getting to the goal perceives itself as stuck it will
attempt to perform recovery. This could for example be the robot rotating 360 degrees
in an attempt to unstuck itself.

7.2
Sensors
The Turtlebot3 burger robot used in this project is equipped with 2 groups of sensors.
The first group are sensors that are used for odometry such as a gyroscope or a ac-
celerometer. The other group is the sensors used for perceiving the world around the
robot, this is a 360 degree laser distance sensor and a Kinect camera. The laser sensor is
used for navigating the robot around in the simulation environment, and for generating
a map with SLAM. This process is described in the ”Navigation” chapter.
The camera sensor is used for object recognition, which is a necessary prerequisite
for identifying the situation in which the robot is at any given point in time. Identifying
the situation is an important step in getting a robot to make a decision.
What object recognition is and how it is applied in this project will be described
further in the next sections.
7.3
Object Recognition
Object recognition is typically defined as a machine learning technology that uses com-
puter vision and image processing to identify objects in images and videos. This is a
technology that is highly used in the development of self driving cars, for example to
recognise a stop sign or to distinguish a pedestrian from its surroundings. This is typ-
ically done by feeding a picture of the object to be recognised to an image processing
program that scans the picture for defining features such as edges, corners and colour
changes. These features are then searched for in a video feed or other pictures in order
==PAGE:44==
38
7 Sensors and Object Recognition
to find the specified object. The object recognition that is done in this project is used
to provide data for the decision making process.
7.4
Object Recognition Implementation
Object recognition has in this project been done using the ROS package find_object_2d
that implements an image processing node. This program detects objects using differ-
ent feature detector algorithms such as SURF (Speeded Up Robust Features), SIFT
(Scale-Invariant Feature Transform), FAST (Features from Accelerated Segment Test),
BRIEF (Binary Robust Independent Elementary Features) and other feature detectors
and descriptors [18].
In order to use the find_object_2d node it needs to be taught, which objects it
should recognise. This is done by placing the object in front of the camera of the robot
in the simulated environment. The image and extracted features of the image are then
saved for future detection. The node can be taught to recognize several different objects.
Figure 7.1: Find_object_2d adding an object.
When the node has been taught which objects to recognize, it is ready for real time
object recognition (detecting objects during operation). The object recognition is done
by scanning every frame of the video feed from the robot’s camera for image features,
==PAGE:45==
7.5 Verification
39
which are compared to the features of the objects taught previously.
If a suﬀicient
number of these features match, an object is recognized.
The output of the find_object_2d node is published on the ROS topic /object with
message type std_msgs/Float32MultiArray. The data published consists of the object
ID, width, height and the homography matrix of the object. The homography matrix is
used to define where the object is relative to the camera. For the purposes of this project
the object ID is the most interesting element of the array. If no objects are recognised
in a frame, an empty element (Null_element) is published on /object.
The find_object_2d node publishes data for every frame in the video feed. This was
discovered to be a problem when trying to recognise an object at a specific position in
the simulation environment. The reason is that the object is not found in every single
frame that actually contained the object, but only in some of them. This seems to
be dependent on the angle of the camera relative to the object. This implies that the
consistency of the object recognition was sub optimal.
To overcome the issue above a ROS subscriber and publisher node was created.
See appendix B. This node subscribes to the ROS topic /object and filters out all the
Null_elements and publishes only to /my_topic when a new object is recognised. This
implies that subscribers to my_topic always get the object ID of the most recently
recognised object. This results in a more consistent object recognition behaviour, which
is considered to be a prerequisite for decision making.

8.1
Introduction
In this project situation awareness is the basis for decision making. This chapter will
describe what situation awareness is, and how the robot achieves situation awareness.
8.2
Situation Awareness
Situation Awareness is awareness of the various objects in the world, and their current
status. For a human, situation awareness can be put into three levels of awareness as
described by Endsley’s theoretical model for situation awareness [12]:
1. Perception. Being aware of ones surrounding
2. Comprehension. What do the surroundings mean
3. Projection. What does the surroundings lead to
Perception could for example be described as a person driving a car noticing a man
crossing the street in front of him. This can for a robot be defined as the measurements,
such as the camera data.
Comprehension could for example be to determine the seriousness of the situation
with the knowledge of what the speed of the car currently is, what the distance to the
man is, and how fast the man is crossing the street. This can for a robot be defined as
the interpretation of the measurement data.
Projection could for example be when the person in the car predicts that the car
is going to hit the man. This can for a robot be to predict that it can only reach the
goal using an alternative route.

CHAPTER 9

9.1
Introduction
In this section it is summarized what a decision is, and the implementation of the decision
making is described. Decision making has already been described in further detail in
the Cognitive behaviour section (2.5) of the analysis.
9.2
Analysis
As mentioned in the Cognitive Behaviour section of the analysis, decision making for a
robot is defined by the following four steps.
1. Gather information (sensor inputs)
2. Identify the situation that the robot is in
3. Identify possible actions that the robot can take
4. Perform the action with highest priority
It is also mentioned that these steps can be described in terms of a finite-state
machine (FSM), which is an abstract machine that can be in exactly one state at any
given time. The amount of states is a limited number. The FSM can transition from
one state to another in response to its inputs. An FSM consists of the following three
items [25].
(a) States
(b) Transitions
(c) Inputs
States describe the status of a system waiting to do a transition.
Transitions. A transition is the set of actions that the system takes in order to go from
state a to state b.
Inputs are the signals/commands a system receives from outer sources, such as a sensor.
==PAGE:54==
48
9 Decision Making
Based on these three items the decision making process for an FSM can be described
with following three steps.
1. Input triggers a necessity to transfer from one state to another
2. Choose between the possible states that the FSM can transition to
3. Transition to the chosen state

10.2
The Solution
The previous chapters combined are used to describe ”Situation Awareness and Mission
Adaptation for Mobile Robots”, the title of the project. The chapters are building blocks
for the final result of the project. The final result is assembled in the following way.
ROS have been used as the framework for writing the robot software. A lot of the
tools and libraries included in ROS have been used to reduce and simplify this software.
This follows the guiding principal of ROS, to not reinvent the wheel.
Gazebo has been used as the 3D simulation tool, and has been used in combination
with ROS as the basis for the simulation environment.
On top of this basis from ROS and Gazebo a world has been build.
The world
describes elements such as the lights and static objects in the world. The world that
is used in this project is the ROBOTIS house world. This world had to be modified
since it had few objects that could be recognised with the robot’s camera. To resolve
this problem boxes with numbers were inserted.
These numbers are relatively easy
recognised by the camera. This world was created so that the chosen robot had an
relevant world in terms of decision making to navigate.
The chosen robot for the project is the Turtlebot3 burger model, which is a com-
mercial robot made for the purpose of exploring ROS. This robot was chosen due to its
relative simplicity to its alternatives and the wast amount of ROS packages that exist
to support it. The existence of these packages results in a considerably less amount of
nodes that needs to be developed from ground up for this project. This includes support
for the ROS navigation packages.
The ROS navigation packages are used to drive the robot along a predetermined
route in the house, while avoiding all obstacles along the way. Elements of the ROS
navigation packages are used to generate a map of the house using laser sensor data.
==PAGE:64==
58
10 Total Solution
They also include ROS nodes that use this map together with the sensor inputs of the
robot to plan a route that get the robot from point a to point b while avoiding all
obstacles. On top of this the ROS navigation packages include elements, that makes the
robot perform the necessary movement to get to point b along the planned route.
Besides the laser sensor the robot uses for navigation purposes, the robot is equipped
with a camera sensor. The camera sensor is in this project used for object recognition.
The object recognition is functioning by teaching the ROS find_object_2d node the
features of the objects to be found, and then searching for these features in the camera
feed. In total the sensors are used to perceive the world around the robot.
The camera is used to make the robot aware of its surroundings. This is the first
step in situation awareness, perception.
The other two steps of situation awareness,
comprehension and projection, are implemented in the ROS node seen in appendix A.
Comprehension is in this project the robot’s translation of the measurement data from
the sensors to a situation. Projection in this project is implemented when the node
predicts that:
• The robot will run out of power
• The robot will hit a humanoid
The previous two paragraphs are the first two steps in the definition of the decision
making process for a robot, see section 2.5. The four steps in the decision making process
are these:
1. Gather information (sensor inputs)
2. Identify the situation that the robot is in
3. Identify possible actions that the robot can take
4. Perform the action with highest priority
In order to facilitate a simpler implementation and a better description of the decision
making process, the third and forth step above is redefined as a finite-state machine with
the first two steps as inputs. This FSM is described with the following three steps:
1. Input triggers a necessity to transfer from one state to another
2. Choose between the possible states the FSM can transition to
3. Transition to the chosen state
For the robot, the FSM decision making process can be described like this: When the
robot receives the sensor inputs they are translated, as described above, to a situation.
The robot uses this situation to decide between the possible next states and executes
the implied transition to get to the chosen state. This decision making process is imple-
mented in the main ROS node that has been developed for this project, see appendix
A.
