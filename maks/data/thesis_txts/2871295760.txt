Abstract
The evolution of Software­Defined Networking (SDN) has brought unprecedented levels
of flexibility, visibility, and control to network architectures. Among its most promising en­
ablers is the Programming Protocol­independent Packet Processors (P4) language, which
allows developers to program the data plane of network devices to enable advanced
packet processing. One critical feature facilitated by P4 is In­band Network Telemetry
(INT), which embeds monitoring metadata within packets to offer fine­grained visibility
into per­hop network conditions such as latency, queue depth, and link utilization. De­
spite these capabilities, several barriers have slowed the adoption of SDN and INT­based
solutions, including outdated tooling and steep learning curves.
This thesis addresses these limitations by developing a modern, efficient and developer­
friendly network monitoring system using INT and extended Berkeley Packet Filter (eBPF)
technologies. The solution is designed according to three core pillars:
• Create a working INT­based monitoring system using P4.
• Prioritize the use of the latest available versions of all relevant software, including
P416 v1.2.5 and Ubuntu 24.04.
• Ensure convenience and ease of development, especially for users working in Win­
dows environments, by supporting the Windows Subsystem for Linux (WSL).
The system includes P4­based switches programmed to collect hop latency data, an eBPF
collector adapted for compatibility with modern Linux kernel versions (tested on Linux
kernel 6.6.36.3), and a visualization layer built with Prometheus and Grafana. Significant
technical challenges were encountered in updating legacy components – such as the
eBPF collector originally developed for much older Linux kernels – and adapting them to
a modern toolchain. In contrast to similar efforts which relied on outdated platforms such
as ONOS and Linux kernel v4.15, this project demonstrates the feasibility of achieving a
similar or improved outcome using contemporary alternatives.
The final implementation, while functional and instructive, also reveals key limitations in
current tooling. For example, the Prometheus pull model is suboptimal for telemetry data
with irregular packet flows and the fact that the collector does not support the entire INT­
MD specification. These insights inform a roadmap for future work, which includes up­
dating to newer telemetry formats (e.g., INT v2.1), testing on more complex topologies,
performance impacts of INT, and supporting alternative databases that better handle irreg­
ular and bursty data streams. Ultimately, this project contributes a practical and forward­
looking prototype while helping to identify structural issues that must be addressed to
accelerate the adoption of SDN and INT in production networks.

Introduction
1.1
Motivation
Software­Defined Networking (SDN) is an emerging network architecture that separates
the control plane (the logic that decides how to handle traffic) from the data plane (packet
forwarding). SDN makes networks more agile and adaptable, which are highly valuable
traits in networking, which is otherwise very hard to obtain. Studies confirm that SDN
dramatically improves network flexibility and programmability, security, and can reduce
operational costs of networks[1]. In practical terms, an SDN controller can dynamically
adjust end­to­end routing, security, and resource allocation without requiring manual re­
configuration of each switch, making the network much easier to manage and evolve.
With modern networks growing in size and complexity, whether for data centers, internet
service providers (ISPs), or local area networks (LANs), the value that SDN brings should
grow equally.
Programmable Data Planes and the P4 Language. A key enabler of modern SDN is
the ability to program the packet­processing behavior of switches and routers at a fine­
grained level. The P4 language (“Programming Protocol­independent Packet Proces­
sors”) was developed with the goal of bringing programmability to the network data plane.
A P4 program defines the packet header formats, parsing logic, match­action tables, and
processing actions that run on a forwarding device. For example, a P4 parser declares
how to recognize packet fields, and tables specify what to do when packets match cer­
tain header values, e.g. forward, drop, or modify the header contents. Importantly, P4
is hardware­independent: The same P4 program can target different Application Specific
Integrated Circuits (ASICs) or software switches with the appropriate compiler[2]. This
flexibility allows developers to create new header types and custom forwarding behav­
iors “on demand.” In the words of the P4 consortium, P4 “turns the traditional model on
its head” by letting network engineers implement custom behavior themselves, enabling
changes “in minutes instead of years”[3].
In­band Network Telemetry (INT). In addition to flexible packet processing, SDN aims to
provide rich network visibility. INT is a core functionality in P4 that embeds monitoring data
directly into the data packets as they traverse the network. In INT, each packet carries
metadata that records aspects of its journey – for example, per­hop latency, queue sizes,
link utilization, and other node statistics[4]. This uniquely enables real­time visibility of
network performance at each switch. The per switch diagnostics can help engineers
better diagnose issues in the network, and maybe more importantly, it can be used in
control plane logic to dynamically optimize network performance.
Obstructions to development and widespread use of SDN. SDN is experiencing sig­
nificant growth in practical usage, and development continues to see advances. The
advancement of SDN features in many modern network does, however, still appear to be
significantly limited when considering the value it can contribute. A 2021 publication[5]
did an exhaustive survey of more than 200 P4 development articles to discuss the chal­
lenges that it faced. A large factor stunting the growth of SDN is the costly transition of
replacing all switches in a network. Gradually incorporating SDN into an existing network
is challenging and a lot of the value from SDN will be lost in these half­implementations.
In most cases, SDN networks will also require a network operator to have knowledge and
experience with SDN. Both the 2021 publication and a more recent one from 2025[6] find

1.2
Project Objective
3 key decisions were made for the objective of this project, pillars both as way­of­working
and as a goal for the project solution. These pillars are set partly to ensure the quality of
the solution proposal but also to avoid some paths of least resistance, so that the potential
problems of SDN development are highlighted rather than avoided.
Create INT­based network monitoring system. P4 appears to be a front runner in SDN.
It sees supported development from tech giants such as Intel and Google and is widely
included in research. INT is one of the main features of P4, enabling per­switch values in
network monitoring. A monitoring system will include a network with P4 switches tracking
INT along with ways to collect and visualize the INT metrics. To be able to handle collection
for a large network, the metric collector needs to be highly efficient, so it is intended to
use eBPF for the implementation of this.
Use newest software versions where possible. Since SDN projects are frequently not
maintained, they may not function on older versions of e.g. Ubuntu or the P4 language.
For these two in particular, Ubuntu 24.04 and P416 1.2.5 should be used. Using new
versions of software creates better compatibility now and in the future. A distinct choice
is also to use only P4 as part of the SDN solution. Particularly ONOS has been used in
combination with P4 frequently, but development on the ONOS project ceased in 2022,
so it should preferably not be included in new research and development.
Ensure convenience for development on the project where possible. The conve­
nience in talk should be considered from the perspective of a third­party developer, using
or further developing the project. Some of it will likely be beneficial during the initial de­
veloping, but will likely also create additional hurdles. It has been decided that the project
will be developed in WSL, which allows Windows users to develop on a Linux virtual ma­
chine while still using the Windows user interface, instead of using a VirtualBox or similar
programs. Using new versions of software also increases the convenience, as new de­
velopers will not need to download and use older versions.

Background
2.1
Linux and its Kernel
Any general purpose computer needs a way to organize and reserve its computational re­
sources, as well as interface with its hardware in order to run applications and processes.
This role befalls the operating system (OS), the most underlying software of the computer.
The operating system may be subdivided into the kernel­software that is always running
on the system, which is responsible for earlier mentioned tasks, and system software,
which is all other software included in the operating system, but which does not need to
run at all times.
Interfacing directly with hardware is often difficult due to the primitive nature of low level
machine languages, as well as the uncertainty of which hardware the application may
need to run on or interface with. Instead of interfacing directly with hardware, most ap­
plications, therefore, interface with the systems kernel through more generalized system
calls instead. The kernel can then interface with the needed hardware through system­
specific drivers on behalf of the application. This also provides a layer of security, since
each individual application is unable to interact with other applications and processes or
interact with the system’s hardware without first going through the kernel which may dis­
allow such actions.
Under such a subdivision, the applications run through interaction with the kernel are
said to operate in user space, while the kernel, device drivers, and any applications which
might forego the application­side interface of the kernel are said to operate in kernel space.

2.1.1
Modifying the Kernel
As mentioned, Linux is an open source operating system, and as such anyone has ac­
cess to the source code and can modify it however they want. This, of course, requires
deep familiarity with the preexisting code base, making the undertaking quite daunting for
inexperienced kernel developers. Should one manage to produce a kernel modification
beneficial for a certain purpose and wish to share this change, further difficulties may be
encountered. For a change to be contributed upstream to the Linux system and included
in later versions, the change first has to be accepted by Linus Torvalds and the respective
subsystem representative. The change will therefore have to align with their vision of the
general purpose Linux OS. Even if the patch is accepted, it will only be included in the
next release, which means a further delay is introduced, and the patch is of no use to
anyone running an older release.
Another way to modify the kernel is through manually configuring and re­building it. This
allows for enabling different kernel functionalities that are not a part of the standard distri­
bution (Ubuntu, Debian, Arch, etc.) or disabling unneeded functionalities that are. This,
of course, does not allow for full customization, but only choosing compiler options.[14]
Lastly, the Linux kernel supports modification through modules. Modules are behavior

2.1.2
Interrupts
While a general purpose computer is as generalized as possible to allow as wide a use
case as possible, different components still exist within the computer in order to distribute
certain tasks to components specialized particular for handling these tasks. Utilizing sep­
arate specialized components means that each component needs to implement strict pro­
tocols for interacting with other components. This is fairly straight forward for components
which can run synchronously either on the same clock or through expected differing clock
pulses. A more complex issue arises when a component needs to interact with the CPU or
other components in an asynchronous and unpredictable timely fashion, most commonly
seen in input/output (I/O) components.
A typical approach to handling such an asynchronous relationship is to store incoming
or outgoing data in a buffer, often a Direct Memory Access (DMA) chip (see fig 2.4), and
then transmit a signal indicating that data is ready to be read in the buffer. The receiving
component can then read the data from the buffer, or compute whatever else the cor­

Tracepoints are functions in code that are used to call other functions provided at runtime.
A function may be attached or ”hooked” onto the tracepoint, in which case the function is
referred to as a probe. In case a function is hooked, it will be called once the connected

Different probing mechanisms exist for different use­cases in the system. Some major
examples are:
• Raw Tracepoint[17][18] refers to tracepoints placed within the source code of the
kernel. These tracepoints cannot be removed or added at runtime and are there­
fore also referred to as static tracepoints. The location of these can generally be
expected to not change between kernel versions, making them useful hook­points
for broad use­case software expected to be widely deployed[19].
Raw trace probes pass arguments as raw values in a u64 array, which means they
are read only[20].
• Kernel probes (Kprobe)[21] can be loaded on demand and attached to instruction
addresses of kernel functions, which means they can be placed very specifically
where they are needed in order to break and examine a kernel function while it is
executing. Kprobes are unique in that they modify the program they are probing
directly in the machine code.
Kprobes access functions and function arguments directly in registers[22][23], which
means they can modify the values in these registers, essentially gaining read/write
access. Arguments may in this way be modified before they are delivered to the
probed function.
• Function entry/exit probes (Fprobe)[24] are also loaded on demand, but unlike
kprobes, they can only attach to entry or exit points of kernel functions. This limits
their applicability but reduces overhead, making them more lightweight. Fprobes
can also be attached to kernel modules such as BPF programs unlike kprobes.
Fbrobes fetch the arguments of the probed function and pass them as typed argu­
ments through BPF Type Format (BTF) [25]. This makes them read only.
These mechanisms can be quite difficult to work with, but fortunately tools exist for stream­
lining this process. This matter is further elaborated upon in sections 2.7.

2.1.6
Context Switching and Mode Switching
A context switch is when the CPU switches from one process or thread to another. A
process in this situation is a currently executing instance of a program. The “context”
being referred to describes the contents of CPU registers and program counters that will
be changed during the execution of a process, so a clean­up is needed before the next
process can be run.[35]
Modern CPUs have process multitasking, making context switching a far more compre­
hensive action than simply cleaning up, and making it required more frequently. With
multitasking CPUs, a process may be paused, while another process is run for some
time, before resuming the initial process. This context switching requires the CPU to tem­
porarily store the state of the process and load in the state of a different one. In modern
CPUs, this involves many registers and resources (general­purpose registers, floating­
point/SIMD state, program counter, processor status, memory mappings, etc.), making
context switches relatively expensive.
Context switching is essential in a multitasking OS because the CPU is a limited resource
and must be shared among many tasks. With a single CPU core, the OS rapidly switches
between runnable processes so that each one makes progress. This enables all pro­
cesses to share a single CPU to finish their execution and allows the system to handle
many tasks in parallel. By cycling through runnable processes, the OS ensures fair use
of the CPU and keeps the system responsive.
Another common cause for context switching is switching modes between kernel and user
space. A mode switch does not always necessitate a context switch, but frequently does,
and in these cases information needs to traverse the two spaces, which also creates
significant computational overhead.

2.3
Mininet
Mininet is a lightweight network emulator that can create a network of virtual hosts, switches,
controllers, and links. A Mininet host is essentially a standard Linux process with its own
network stack, so you can run Linux commands such as ifconfig or ping on the host. The
switches are software­based, supporting OpenFlow and other SDN options,[43] making
Mininet a very popular platform for prototyping in SDN development.
2.3.1
Architecture
Mininet leverages Linux’s lightweight virtualization features to implement its virtual net­
work. Each virtual host in Mininet is a Unix process (or a process group) running in its
own network namespace[44]. A network namespace provides an isolated copy of the
networking stack – complete with separate interfaces, routing tables, and ARP tables.
Namespaces can be connected with pairs of virtual ethernet (veth) interfaces, creating a
link.
Mininet’s core is written almost entirely in Python (except for a short C utility), and it pro­
vides an API for Python programs as well. The Python API provides classes for nodes
(hosts, switches, controllers) and links, as well as a CLI for interactively controlling the net­
work. With the API, a user can define a topology (consisting of hosts, switches, and links)
either as a static pre­made file or programmatically, which makes it possible to create
large and complex networks easily.
Hosts: A Mininet host is a shell process in a distinct network namespace. Any programs
or services started on that host are run on the native Linux kernel and network stack,
as if on a separate machine. By default, all Mininet hosts share the root file system for
efficiency, though isolated file systems can be configured if needed.[45] Because each
host is just a process, tens or hundreds of hosts can coexist on one physical machine.

==PAGE:24==
Switches: Mininet provides software switches, by default using Open vSwitch (OVS),
but can use others, such as BMv2, which is the switch primarily used for P4 development.
The virtual switches are fully controllable via an SDN controller, either via OpenFlow, P4
or other SDN software.
Controllers: An SDN controller in Mininet is a process that communicates with the switches
to centralize the control­plane. This can happen via OpenFlow, ONOS, P4 or other SDN
software solutions.
Links: Each link in a Mininet topology is implemented by a pair of veth interfaces. For
example, if host h1 and switch s1 are to be connected, Mininet creates a veth pair and
places one end in h1’s namespace and the other end in s1’s namespace. These vir­

standard Linux traffic­control and monitoring tools.
2.3.2
Use Cases
Compared to traditional virtualization, Mininet has major benefits:[43]
• Boots faster: seconds instead of minutes
• Scales larger: hundreds of hosts and switches vs. single digits
• Provides more bandwidth: typically 2Gbps total bandwidth on modest hardware
• Installs easily: a prepackaged VM is available that runs on VMware or VirtualBox
for Mac/Win/Linux with OpenFlow v1.0 tools already installed.
Mininet is especially prevalent in SDN research and development. Its ability to instanti­
ate realistic networks with various SDN switch types makes it ideal for prototyping and
testing SDN controllers and data­plane programs. Developers can write a new routing or
forwarding application, run it on a Mininet controller, and test it on an arbitrary topology of
emulated switches and hosts within minutes. Since the traffic is real packets on a real IP
stack, metrics such as latency and throughput are meaningful. Many SDN­related papers
and projects use Mininet for experiments, and it has been credited with helping “usher in
the SDN revolution” by making SDN experimentation accessible.[46]
Beyond SDN, Mininet can emulate traditional network scenarios. Since it runs the full
Linux network stack, researchers can experiment with unmodified protocols (e.g. TCP
congestion control) on virtual topologies. Large­scale topologies are feasible: Because
Mininet hosts are lightweight, one can study scalability or performance of distributed al­
gorithms on hundreds of nodes.
Mininet has also been extended for emerging domains. For example, Mininet­WiFi adds
virtualized 802.11 radios to simulate wireless and mobile networks. Other efforts integrate
Mininet with Docker containers or virtual machines, allowing complex network function
virtualization (NFV) experiments where each virtual host may run multiple services.

2.4.1
P4
This section is based on the P4 v1.2.5 specification [47]. The P4 language is developed
explicitly for managing packet forwarding planes, distinguishing it from general­purpose
languages such as Python, Java, or C++, as it is a domain­specific language. P4 uses the
term ”targets” for the devices it is designed to run on. These targets can be hardware or
software switches, NICs, routers, or ASICs. While many targets have both a control plane
and a data plane implemented, P4 is only designed to specify the data plane functionality.

P4 can, to some extent, define the interface by which the control plane and the data plane
communicate.
Figure 2.8: Traditional switch VS P4 switch[47]
Figure 2.8 shows how a P4 switch differs from a traditional one. In a traditional switch,
the manufacturer will have pre­defined the data plane functionality, and the control plane
would control the data plane by managing entries in routing tables. However, in a P4
switch, the data plane is defined by a P4 program. This program is configured when the
device starts up (red arrow). The control plane will still communicate with the data plane
as it would in a traditional switch, but the tables it uses are no longer fixed and can be
defined in the P4 program. By P4 being protocol­independent, it can have custom or
future protocols not yet described.
• Headers: Define what the header looks like and the fields it may contain, along with
the sizes of set fields.
• Parsers: Define the rules for legal header sequences, how to identify headers, and
which fields to extract from them.
• Tables: Tables in a P4 switch can be used similarly to those in a traditional switch
for purposes such as routing tables, flow lookup tables and access control lists.
Importantly, they also support user­defined tables.
• Actions: In P4, actions can be viewed as function­like blocks of code that define
how packet headers and metadata are manipulated. However, unlike functions in
conventional programming languages, the program logic does not invoke actions
directly. Instead, they are dynamically selected by match­action units based on the
results of a table lookup during packet processing.
• Match­Action Units: Match­action units operate in the following sequence:
1. Build a lookup key using packet fields or metadata.
2. Perform a table lookup based on the constructed key and select the corre­
sponding action.
3. Execute the selected action.

• Control Flow: Describes the order in which operations occur during packet pro­
cessing on the device. Deparsing (packet reassembly) can also be managed through
control flow.
• Extern Objects: These can be used with well­defined APIs by a P4 program, how­
ever, they cannot be programmed in P4 itself. An example would be checksum
units.
• User­Defined Metadata: Metadata structures that are defined by the programmer
and associated with each packet.
• Intrinsic Metadata: Metadata provided automatically by the target architecture,

The three fixed­function blocks are:
• Arbiter
The arbiter block receives packets from one of the connected inputs (Ethernet, CPU,
or input recirculation). If a packet is received on one of the Ethernet ports, the Eth­

sum, the packet is discarded. When the checksum does match it is removed from
the packet’s payload. An arbitration algorithm decides which packet to pick next if
multiple are available. If the queue is full and a packet is being processed, the input
ports can drop the newly arriving packets without any indication or reporting. When
receiving a packet, the arbiter block also sets the inControl value (green arrow be­

packet was received on.

• Demux/Queue

2.5
In­band Network Telemetry
INT, which is managed by P4, is a way to collect real­time data from a network by embed­
ding telemetry metadata directly into the network packets already found in the network.
Contrary to conventional methods such as ICMP echo requests/replies, which require
separate packets, INT allows each device it traverses in the network to append metadata
such as switch ID, ingress and egress ports and timestamps, queue occupancy, hop la­
tency, and Egress port TX Link utilization, directly into the packet. This provides valuable
information about the state and health of the network and gives a better insight into the
behavior of the data plane hop by hop. The INT data collected can then be extracted
either at the destination of the packet or at a collector designed to extract the INT data.
This extracted metadata can then be analyzed for a range of use cases, including network
diagnostics, performance assurance, traffic engineering, and security monitoring.[48] [49]
2.5.1
INT v1.0
This section is based on the INT v1.0 specification [49]. The INT architecture consists of
multiple elements that enable several interesting high­level applications, such as network
troubleshooting and performance monitoring, advanced congestion control, advanced
routing, and network data plane verification. The INT architecture consists of the following
components:
• INT Header: This is a packet header which contains the INT information, and as of
version 1.0, there are two versions of the INT header.
– Hop­by­hop.
– Destination.
• INT Packet: This is a packet which contains the INT header.
• INT Instruction: The instructions are placed within the INT header and are then
read by the INT capable switch to communicate which metadata to collect. The
collected metadata is then written into the INT header.
• INT Source: This entity is in charge of creating and inserting the INT header into
the packets.
• INT Sink: This entity is in charge of extracting the INT headers along with the path
data from the INT header. This is done at the edges of the network to ensure trans­
parency for the upper layers.

• INT Metadata: This is the information that is inserted into the INT header.
• INT Domain: A collection of INT capable devices under the same control. It exists
to ensure that there is interoperability between INT nodes from different vendors
and ensures that there are INT sinks placed at the edges of the domain to prevent
INT data from leaking outside of the domain.

any encapsulation type either as an option or as payload. This is possible as long as the
encapsulation header provides enough space for the INT information and that all the INT
capable devices in the network can agree on the location of the INT headers. For the
purpose of this thesis the focus will be on INT in UDP.

6936 [51]. In the case of an INT source, transit, or sink receiving UDP traffic with a
zero checksum, they must not update the checksum. When the UDP header does not
contain a zero checksum, the checksum field must be updated as the INT switches modify
the payload either by inserting or removing INT headers and/or metadata. This can be
updated in one of two ways by the INT source or an intermediary INT switch:
• The UDP checksum field can be updated so that it matches the new checksum for
the packet after the INT changes, such as adding headers or removing them.
• The INT source can indicate that checksum neutral updates are allowed by setting
the flag for checksum complement in the instruction set. This way the INT capable
devices (apart from the sink) in the network will know to assign a value to the check­
sum complement metadata field so that the checksum in the UDP header remains
the same. The idea behind the checksum complement is that some hardware reads
packets serially and therefore cannot go back and change the checksum field in
the UDP header. Since the complement checksum is placed as the last metadata
field in the INT metadata stack, the limited hardware can use this instead. How­
20

==PAGE:31==
ever, should the switch be able to go back and change the UDP checksum field, it
can choose to do so. If this is the case, then the metadata field for the checksum
complement is assigned 4 bytes of the reserved value (0xFFFFFFFF).
In either case, the host that verifies the UDP checksum will remain unaffected since the
checksum of the UDP header should match the payload. The INT sink was excluded from
performing checksum neutral updates and this is due to the fact that its job is to remove
all the INT headers from the packet and therefore it should update the UDP header check­
sum field.
Regardless of whether the UDP checksum field was changed directly or whether the INT
checksum complement metadata was used, the checksum calculations should always be
done incrementally. This way, corruption of packets is still detectable. If the new check­
sum was just calculated without checking the old one, corruption would not be possible to
spot. Therefore, a checksum validation should precede the new checksum calculation.
INT Over UDP
INT over UDP is a versatile encapsulation format since it does not require any tunneling
or virtualization mechanisms. A shim header is placed after the UDP header and then
the INT headers are placed between the shim header and the UDP payload. This can be
seen below in Figure 2.11.
Figure 2.11: INT over UDP [4]
INT over UDP works under the assumption that the intermediary devices that are not INT
capable can either not read past the UDP header or can use the INT shim header to
read the length and skip over the INT stack that way. INT v1.0 proposes two options for
indicating that INT headers are present between the UDP header and payload.
• With IPv4 Differentiated Service Code Points (DSCP) or IPv6 traffic Class field, there
is the option to allocate a bit or use a value to indicate that there is INT data after the
UDP header. This bit or value is then set by the INT source. The option of storing
the original value from DSCP and then having the INT sink restore it exists, but it is
not necessary.
– When allocating a bit in DSCP to indicate the presence of INT, it allows for the
rest of the DSCP bit field to be used for Quality of Service (QoS). The DSCP
bit field is 6 bits and with 1 reserved to indicate the presence of INT there are
5 bits left and it is important that the QoS within the network is programmed to
ignore the reserved bit. This is to ensure that INT­enabled packets behave like
the original ones with the same QoS level as intended.
– When it comes to brownfield scenarios, e.g. when introducing INT into already
existing networks, the network operator might not have a DSCP bit it can re­
serve for INT but they could have 32 fragmented values in the DSCP field. This
allows the network operator to allocate these fragmented values to indicate INT

2.6
Telemetry Report
This section is based on the Telemetry report specifications for version 1.0 [48]. A network
device generates a telemetry report to relay monitoring information to an external system.
Each report provides a snapshot of the triggering packet, typically capturing both inner
and outer headers and the metadata that the reporting device or metadata collected by
INT. The reports are then encapsulated in IP and UDP to be forwarded to the monitoring
system. Telemetry reports could be used in a wide range of scenarios. Three current
examples include: Tracked Flows, Dropped Packets, and Congested Queues.
• Tracked flows: Telemetry reports are created from a specific flow definition called
a flow watchlist, which determines which packets to monitor from the header fields.
The reports would contain information about the packet’s path and other metadata
collected along its way.
• Dropped packets: Here, a drop watchlist is used to generate the telemetry report
for dropped packets that match the criteria in the drop watchlist. The purpose of
generating these reports is network visibility.
• Congested Queues: When a queue becomes congested, telemetry reports are
created for packets arriving in the already congested queue. This information helps
to understand what creates the congestion and which flows are affected by the con­
gestion.
To create these telemetry reports, the devices on the network need to be set up to know
when to generate telemetry reports. When set up, not all packets that match a watchlist
have to result in a telemetry report. Each device in the network can employ filters to
watch for events that should warrant a telemetry report. For instance, a network device
can initiate telemetry report creation whenever it receives a packet corresponding to a
monitored application flow on a new path, or if there is a notable change in latency at a
specific hop. This is called event detection.

2.7
Extended Berkeley Packet Filter (eBPF)
eBPF is a kernel extension framework fully available from the Linux kernel version 4.4 with
origins as a packet filtering tool[53], but has later expanded to encompass much more.
It is able to load and run programs directly in the kernel of a system, directly accessing
kernel resources and thereby bypassing the need to request these[15].
At its base, eBPF is a full implementation of a 64­bit register­based virtual machine which
is run entirely within the kernel of the host system. This means that any eBPF program
can run on any architecture that supports the eBPF virtual machine without the need for
recompilation. Furthermore, it circumvents the need to make direct kernel changes which
can cause critical kernel errors[54].
2.7.1
Development and deployment of eBPF programs
Since eBPF runs in kernel space and has access to vulnerable resources, great care
should be taken when developing such programs. In order not to overburden the devel­
oper, some restrictions have been placed upon the allowed structures of such programs.

developed. While eBPF programs themselves are written in C, BCC provides support

eBPF hookpoints
eBPF programs are hook­based, and therefore need to be attached to hookpoints in order
to run. A wide array of options are available and can be seen neatly summarized in
various different documentations available on the web[59]. As an example, kprobes and
fentry/fexit are available for very specific probing purposes, but for network probing the
most notable hook is the XDP hookpoint.
XDP[60] is a framework that places a hook in the driver of network cards. The hook is
placed as soon as an entire packet has been received, but before any memory has been
allocated for storage of the packet in the sk_buffer, preparing the packet for the kernels
network stack. This means that the hook activates at the first possible moment, the entire
packet is available in the system, before the system has had a chance to dedicate any
computational resources for the packet[61][62].
An eBPF program utilizing the XDP hook is often referred to as an XDP program but will
in this scope still be referred to as an eBPF program or XDP enabled eBPF program.
An eBPF program can be attached to an XDP hook in a number of ways[63][64](3.2.4).
• SKB mode/Socket buffer mode (Generic mode) – This mode attaches the eBPF
program further up the network stack and thus does not gain the full advantage of
what XDP is meant for. It attempts to imitate proper XDP for the purpose of making
XDP eBPF available on systems which do not have XDP enabled NICs. This mode
also has to forego some functionality (XDP_REDIRECTED, XDP_TX, discussed later,
and the ability to operate on TCP trafic), leaving this mode very undesirable. If this
is the only mode available, the eBPF documentation recommends hooking onto the
Trafic Control module in the network stack using a CLS enabled eBPF program
instead[65].
• DRV mode/Driver mode (Standard mode) – The XDP enabled eBPF program is run
in kernel space within the NIC driver.
• HW mode/Hardware offload mode – In this mode the eBPF program is completely
offloaded to the NIC itself. This completely removes the execution of the program
from the CPU, but at the cost of removing access to kernel space, thereby limiting
the possible functionality of the program[66]. This mode is only supported on the
Netronome nfp NIC[61].

it expects a response to its hook call.

packet that triggered the hook call. The return codes are[63]:

Writing eBPF
EBPF is written in limited C[15]. Since causing stalling or crashes in the kernel is critically
disallowed, the eBPF program must undergo rigorous verification to ensure that it will
do neither of these things. The eBPF documentations and community summarizes the
verification process into the following 5 overarching points[70][71]:
1. The process loading the eBPF must have the required privileges to do so.
2. The program does not crash or cause harm to the system.
3. The program always runs to completion.
4. The program is not allowed to deadlock.
5. The program is not allowed to read arbitrary memory.
Point 1 is rather self­explanatory, but must be done. An exception is made if unprivileged
eBPF is enabled, allowing the loading of functionally restricted program by unprivileged
users. This mode is not of interest to the scope of this project, and will not be further
outlined.
The other points are more complex. In order to make sure that the program never crashes,
the verifier first runs a depth­first search of the program’s control flow graph. During this
process, each instruction reached is also marked. If an unreachable instruction exists
within the program, it is immediately rejected.
Next, the verifier simulates the execution of the program. Each branching statement is
fully simulated through both branches. After each instruction the verifier checks if the
program has performed an out­of­range accessing of data, jumped out of bounds, or
dereferenced an uninitialized pointer which could contain a null value. These checks are
done to insure that no unauthorized information is accessed and that the program does
not crash.
During the simulation the verifier also keeps track of how many instructions it executes
in all permutations of the program, which is referred to as the complexity of the program.
Before Linux kernel 5.2 the instruction length limit of the program was 4,000 instructions,
and the complexity limit was 128,000. In 5.2 both limits have been raised to 1 million.
32
Advanced Network Monitoring with SDN and In­band Network Telemetry
==PAGE:43==
This ensures that no infinite loops exist in the program, and that the program executes in
a timely manner.
To avoid race conditions, eBPF provides spinlock capabilities, the ability to lock other pro­
cesses out of manipulating specific data. The verifier protects the system against dead­
locks by ensuring that the program releases any spin locks it may hold in all permutations,
and the program never takes more than one spinlock at any given moment[72].
The size and complexity limit of eBPF programs, of course, has the consequence of lim­
iting the functionality of programs. In order to address this concern, the concept of tailing
has been introduced. Tail calls allow for one eBPF program to call another. In order to
do this, a reference to the called program must be stored in a BPF_MAP_TYPE_PROG_ARRAY
map, and the calling program can then call the tail program using bpf_tail_call using
the map and and index to the desired program. Unlike function calls, tail calls do not re­
turn control flow to the calling program after terminating. Each eBPF program in a tailing
chain is verified in isolation, which means that the size and complexity of the chain is 1
million per program. To prevent infinite loops and unacceptably long execution times a
tailing chain will not continue after 32 tail calls, this results in a maximum chain length of
33 eBPF programs[73].
The concept of function calls with an argument list of up to 5 arguments was also in­
troduced with the Linux kernel v4.16. Function calling behaves much as expected: A
function defined in the program scope may be called and will return after termination, al­
lowing for code reuse. Until kernel v5.6 the verifier would however verify each function
on a call­by­call basis, resulting in the complexity of the program being exactly the same
as if the function had simply been written directly into the top level function. In version 5.6
this was changed to only apply to static functions. Non­static functions are now verified
on a function­by­function basis, resulting in code re­use reducing the complexity of the
program. Functions may, however, only return scalar values, and may only be passed
pointers to program context and scalars. Furthermore, function calling has a maximum
call depth of 8[74].
As stated earlier, BPF maps are used to allow the eBPF program to communicate with a
Python script in user space.
Maps act as both persistent storage outside of program run­time scope and as a bridge
for communication between different eBPF programs and user space applications.
Maps come in many different forms for different use­cases, though they all use a key/value
structure for access. Map entries may be spin­locked if multiple programs are able to
access them concurrently. For this project only BPF_TABLE maps are used, which act as
any regular table[75]. A full list of map types can be found in the eBPF docs[58](BPF C,
Maps).
Further limitations compared to regular C includes[76](LLVM):
• Global variables are not allowed. A workaround is to use a BPF_PERCPU_ARRAY map,
which can have the size of 1 entry but of arbitrary value size.
• Strings and arrays are not allowed.
• Library code is not allowed, but common library code can be implemented as inline
functions.
• Loops are only allowed when unrolled up to Linux kernel v5.3. Past this version
bounded loops are allowed, but should be used with care, since the verifier will
check every possible permutation of the loop. It is recommended to enforce strict
range bounds on loop conditions in a way the verifier can understand. See [77]

33
==PAGE:44==
bound enforcement and other loop­like helper functions.
• Each program has a maximum stack space of 512 bytes. When both tail calls and
function calls are utilized in the same call stack, the maximum stack space space is
reduced to 256 bytes.
• Inline assembly is possible since Linux kernel v6.0.
• Increasing the size of a network packet is only possible if done before the packet is
dereferenced. This is due to the verifier deeming the dereference invalid since it is

Compiling eBPF
eBPF is built on the compile once, run everywhere (CO­RE) approach[78]. This means
that any eBPF, when compiled, should be able to run on any machine with the eBPF VM
installed. This is of course not possible with programs utilizing functionality introduced
in a later version than that of the kernel in which it resides. CO­RE instead refers to the
ability to run within two different kernels which support all functionalities of the program,
but provides this underlying support in sometimes very different ways. Data structures
may vary drastically between kernel releases[79].
In order to support CO­RE, the type format BTF has been developed and remains con­
stant within the linux kernel across releases.
When limited C code is compiled to an eBPF program it happens in 2 steps. First, the C
code is compiled into bytecode, compact numeric code optimized for compactness and
swift executability. During this step, BTF relocators are used to capture high­level de­
scriptions of what the program is trying to access, which is preserved in the byte code.
For example ctx->data would not be optimized to directly refer to the location in the ctx
structure where the structure member data would be found, but would instead preserve
the structure full navigation in the bytecode[79].
This byte is then passed to the eBPF VM which performs the eBPF verification process.
If the bytecode passes verification it is then loaded and just­in­time compiled (JiT­compiled)
into native code that the host systems kernel can run. The BTF relocators become rele­
vant here in the case that the ctx struct has changed but still contains the data member.
The native code can then be attached to the appropriate hook in the host kernel[54].

This chapter documents the implementation and development of the network monitoring

The P4 program, incorporating INT to the monitoring system, is the centerpiece, and what

Routing
Ordinary packet routing is performed via what is called table0. It is present in the P4 file
table0.p4, and in the control plane P4 files sX-runtime.json. The table is defined in the
P4 file to be able to route based on data from the metadata, Ethernet header, and ipv4
header. It has the possible actions to mark the packet to be dropped (_drop), which is
the default action, to send it to the CPU via a defined port (send_to_cpu), or to forward it
with a given destination mac­address and port number (set_egress_port). Each of the
runtime files has five entries for table0, one for each of the four hosts and an additional
entry for the monitor server where INT reports are sent. These table entries all use only
the ipv4 destination address as a match key, and all give the action, set_egress_port.

This is clever design, as these two options are viable for the two very relevant ways of
monitoring a network. Using ICMP pings can give an image of variance in network per­
formance over a long period of time, as ICMP pings do not create much traffic. The UDP
monitoring can be used for a stress test of many packets and high bandwidth load.
The INT shim header and INT metadata stack is inserted in the packet after the ICMP
header. It is not specified by the INT Dataplane Specification[49] where the INT header
should be placed on ICMP packets, but this matches where it should be on UPD/TCP
packets.
This INT­MD implementation is only set up to monitor hop latency and does have the cor­
rect INT instruction value in the header for this as well.
The second part of the INT implementation sends a P4 Telemetry Report[48]. This is an
alternative to having an INT processing unit connected at every endpoint in the network,
so instead when a packet with an INT shim header and metadata reaches an INT sink,
it is cloned and encapsulated in a UDP packet to be sent to the processing unit. This is
done first by the ingress table tb_set_last_hop, with the same table entry on all runtime
json files, where if the egress port is set to be 1, then a metadata value for the packet
marks it as the last hop. This works because hosts 1, 2, and 3 are all on their respec­
tive switches’ port 1, so this would in fact not trigger when a packet is sent to host 4. In
the P4 file switch-int.p4, the IngressImpl control has an if­statement that checks for
this last hop value along with the validity of the INT shim header and metadata, and if

all these are set, the packet is cloned with the mirroring values from the command file,
mentioned is section 3.1.2. On egress of a switch packets are checked with the table
tb_generate_report, and if the metadata instance type of the packets is set to 1, the
action create_int_report will be applied to the packet. The instance type is specifically
set to 1 on cloned packets, so these are the ones that are sent as a telemetry report. The
telemetry report header has an empty sequence number and ingress timestamp, but oth­
erwise fully complies with the version 1.0 specification. The timestamp can, however, be
very relevant for consistent network monitoring, as the report could be delayed in transit,
which would give an unclear image of the timing of network performance inconsistencies
without the ingress timestamp.

eBPF Program
The eBPF program is meant to be able to read INT data from packets and send or ex­
pose the formatted data to a database, such as the included Python collector script from
Inband_Network_Telemetry, mentioned in section 3.1.2. While investigating possible op­
tions for the P4 baseline, an eBPF program with some of this functionality was found
through the ONOS INT Control Application tutorial[81]. This has the ONOS application
create a pipeline to forward copies of the packets with INT shim header and metadata
from the INT sink switch to the eBPF client. The eBPF program is INTCollector[86][87],
which is a high­performance collector supporting INT version 1.0 and Telemetry Report
version 1.0 and able to send data to either Prometheus or InfluxDB.
3.2.1
INTCollector Implementation Design
The INTCollector includes client and collector scripts for InfluxDB and Prometheus as two
separate database option. These are written primarily in Python, aside from the InfuxDB
collector script written in the Python­like language Pyrex. These act as the interface,
invoking the eBPF XDP program written in C, connecting to the databases and formatting
the data between them.
Parsing a Packet
The collector C­program parses all packets fed to it. Parsing the packets happens by mov­
ing a cursor, which points to a specific point in the packet. With the function CURSOR_ADVANCE,
the pointer moves a given amount, and the values between the old and new cursor posi­
tion are stored in a variable. The collector first filters the packets down to those that have
the headers structure of Ethernet, IPv4, UDP as well as the correct UDP destination port,
which should indicate that it is a telemetry report. It will then parse the report data, which
should include another header stack as Ethernet, IP, UDP/TCP, and INT shim header
and metadata. From the INT shim header only the length is used to calculate the number
of hops there is data for, by dividing it with the value from the metadata header ’hop ML’
which is the Per­hop Metadata Length.
The INT instruction set is then read and a variable is set for each of the seven possible
instructions. The metadata is read in a loop running an iteration for each hop. For each
iteration it checks all seven instruction variables for whether it should read and store data,
moving the cursor.
46
Advanced Network Monitoring with SDN and In­band Network Telemetry
==PAGE:57==
The collector also does some data processing on what was saved from the metadata, with
the intent of limiting the INT data that the client program receives to only relevant data.
This includes many checks deciding whether to submit a new report at all. This ends with
checking variables for flow latency, hop latency, queue occupancy, and TX utilization, so
if none of these are changed, the collector does not send a report to the client program.
Exposing Data to Prometheus
The client program, consisting of both files PTClient.py and PTCollector.py, receives
data from the collector by processing events pushed by the collector. A Python library
called prometheus_client is used to set the correct format for the exposed data. All INT
values are created as Prometheus gauges. The client program starts an HTTP server,
which has the values formatted, so Prometheus can read them, which can be seen in Fig­
ure 3.5. The values on the HTTP server are updated as soon as the client has processed
the event from the collector.

Modifications and Challenges
BPF Compiler Collection on WSL
Installing the BCC and running XDP application of WSL did cause significant issues. XDP
application requires modules to be loaded on to the kernel of the Linux machine, and the
default setup of the WSL kernel does not enable this. The BCC installation guide [88]
does include instructions for installing on WSL, which include downloading a WSL ker­
nel configuration from https://github.com/microsoft/WSL2-Linux-Kernel.git, and recompil­
ing the WSL Linux kernel. This did, however, not solve the issue at hand. A blog post
found[89], details more thoroughly the steps to be able to run an eBPF application on
WSL, with the significant difference from the BCC instructions that some of the kernel
config settings should be changed. A total of 13 settings are listed, but only two of them
do not already have the given value set in the downloaded kernel configuration from
Microsoft’s GitHub. The two changed are CONFIG_IKHEADERS being changed from ’m’
(loadable module) to ’y’ (yes, always loaded), and CONFIG_VXLAN from ’y’ to ’m’. After
this linux-headers-generic, linux-tools-generic, linux-tools-common should be
installed, which is only possible due to enabling the IKHEADERS configuration. This blog
post also included a simple test application to see if eBPF XDP applications can load into
the kernel correctly, which made troubleshooting significantly easier than using the test
script from the INTCollector, since then it is necessary to discern errors with BCC and with
the INTCollector. The compiled version of this test program is saved in p4-project/test
xdp/. The change to config setting CONFIG_VXLAN is likely not necessary, as the default
setting was ’y’, meaning it would always be loaded anyway.
Telemetry Report Format
Per version 1.0 of the Telemetry Report Format Specification, there are no specifications
for how the telemetry report data should be formatted, only the telemetry report header.
The typical format of the data will be a full packet with headers, including Ethernet, IP,
INT shim and INT metadata. This also holds true for what both the INTCollector and
the P4 program expect. The miscommunication between the two happens because the
INTCollector expects the encapsulated packet to be either UDP or TCP, as mentioned
in section 3.2.1, but the P4 program is implemented for ICMP packets. Since the UDP
header is 8 bytes long, and the ICMP header is 4 bytes long, this results in the INTCollector
looking for the INT shim header and all the following data 4 bytes offset. The packet was
ultimately dropped because off this, because the INT instruction from the shim header
was read as having a 0 in the first bit, which is invalid. If the packet had not been dropped
other issues would have occurred instead.
The INTCollector had an additional small error in format used for the INT shim header. It
has two different definitions for structs for the header, INT_shim_t and INT_shim_v10_t,
which appeared to be for INT versions 0.5 and 1.0 respectively. The INT_shim_v10_t was
being used for parsing the header on the packet, but this didn’t include the last 8 bits of
the header. Since the struct for the v0.5 header has the correct length as well, and the last
8 bits are not read in the program, it was changed to instead use the v0.5 header struct.
A different change to how the headers were parsed was which IP header data was read.
The INTCollector would read the destination and source IP addresses from the header
in the telemetry report, but it was decided that it was more relevant to see the addresses
of the outer IP header. This was changed by moving the cursor back to the outer IP
header before it would parse the inner IP header from the report. Then after reading the
IP header, the cursor was moved forward to the correct position.
Program Too Complex for BPF Verifier
The INTCollector was created on Ubuntu 18, kernel v4.15. For this project, since running
new software versions was a high priority, Ubuntu 24 with kernel 6.6 was used. Between
48
Advanced Network Monitoring with SDN and In­band Network Telemetry
==PAGE:59==
the versions a significant change was to limit how large programs that are loaded into the
kernel can be. For BCC programs, the BPF verifier is used before loading the program,
to check the program size and other possible issue. The size is determined by instruc­
tions, and these instructions are counted by every possible path that can be taken in the
program, which means that an if­statement will double the number of possible paths, and
a loop will multiply the number of paths with the maximum number of loop iteration – this
also means that all loops must be bounded. This problem is also reported in the issue
page for the INTCollector[86], where the author has commented in 2020, that he had not
had the time to solve this issue on newer kernel versions.
The solution made for this project implementation was to remove some functionality of the
eBPF program to make it simpler. A very large factor for complexity was found were the
INT metadata is iterated for each hop and checking all seven possible INT instruction vari­
ables. The loop could iterate up to 6 times, which was a maximum number of hops set in
the program, and because the if­statements were individual and sequential, the number of
paths is multiplied by seven factorial (5040). Since the P4 program always uses the same
INT metadata structure, the INTCollector was modified to only read that same structure.
The modification includes not using the INT instructions from the metadata header and
deleting the if­statements and code for all types of INT metadata other than hop latency.
This made the BPF verifier accept the program and load it into the kernel. Additionally, the
loop iteration hops, which previously had a maximum of 6, were adjusted to a maximum
of 2.
Updating Values on HTTP Server
The INTCollector could be run with debug mode enabled, which causes it to write log
messages with all the INT information attained from every packet parsed. This was a
very useful tool to see if packets were being parsed, and if the correct bytes from the
packet were addressed to the correct values. Whether this was even shown, however,
was as mentioned in section 3.2.1 decided by a set of variables for whether any values
had changed or been set. The expected behavior was that since every packet parsed
should have hop latency INT values, the variable for whether hop latency was set or
changed should be set as ’1’, and the if­statement should succeed, but this was not the
case. There were four different places in the code that could set the value of the hop
latency variable, but instead of figuring out which should or should not set it to one, the
simple solution of always setting it as one, and hence passing the if­statement, was made.
With this change, values would be posted in the log messages and on the HTTP server
for Prometheus. There was, however, an issue of the values on the HTTP server not
updating for every packet parsed. In these cases, the new values would still be posted in
the logs. This issue turned out to be somewhat similar to the problem with log information,
as there was a variable, is_update that was used in if­statements, deciding whether to
post updated data. An example of code using this was the following:
1
if (is_update)
2
tb_flow.update(&flow_id , &flow_info);
Again, similar to the previous issue, there were several places where is_update could be
set as ’1’, but since the wanted behavior of the program was to always update the values
for every packet parsed, the if­statement was simply removed, so the update function was
always used.

3.3.1
Resolution of Time Series and Pull Databases
Prometheus reads data off the HTTP server from the INTCollector, which, as described
in section 3.2.1, updated with every telemetry report with INT data parsed. Prometheus
works on scraping, which is a pull system with a set interval between each scrape. A push
data­base setup would be able to simply store a new value for every packet by pushing
to the database, but with the pull system, considerations must be made with regards to
how often new values should be pulled.
Considering a simple monitoring system with ICMP pings used to collect INT hop latency
data, as the P4 program used for this project would do, the optimal scrape interval would
be the same as the ping interval. If the ICMP ping interval is one second, the INTCollector
should receive a report and update the HTTP server every second, so Prometheus scrap­
ing every second would see a new value. This is, however, only when assuming there is
no flutter in network latency and packet processing. Flutter in the update interval could
cause Prometheus to read the same value twice or miss new values, with higher chances
of this happening, when the flutter is greater. This is especially problematic because it
is a network monitoring system, and the most relevant monitoring will typically be when
the network does not function optimally. By lowering the Prometheus scrape interval,
the chance of missing new INT values can be lowered significantly. For instance, if the
scrape interval is 0.5 seconds, and the ping interval is 1 second, the flutter would have
to be 0.5 seconds for the scrape to miss a new value. Increasing the scrape interval will
naturally cause Prometheus to store the same values multiple times. Figure 3.6 shows
the difference in scraping every second and every 100 ms while host 1 is pinging every
1 second. Having multiples of the same value can be misrepresenting and can interfere
with calculations or statistics that may be needed for the metrics. So far it has only been
covered how the scrape interval impacts monitoring of mostly consistent INT reports data
intervals, but it becomes even more of an issue when reports are at an inconsistent inter­
val. This could be for instance if the P4 program is set up to monitor common traffic or if
testing with very high packet rates. The P4 system is designed to monitor UDP packets
with destination port 4444, so a stress test of the network can be run. To test this an iPerf
server can be initiated on host 2, and host 1 can send UPD traffic as a client. This was
tested with the command iPerf -u -c 10.0.2.2 -p 4444 -i 1 -b 1M -l 1000, send­
ing 128 packets of 1 kB per second. For Prometheus to read the report for 128 packets
every second, the scrape interval would be 1000ms

Discussion
The preceding chapter documents both the development and the results of the initial so­
lution proposal. The specific goals were to develop a network monitoring system using
P4 INT and eBPF while prioritizing the use of the latest software version and developing
the solution for convenience.
The implementation uses pure P4 with the newest version at P416 v1.2.5 to create per­hop
latency monitoring with INT. It has been connected to visualization tools via a very fast col­
lector eBPF program. This collector was otherwise made to only function on older Linux
kernels but has been updated to work on modern kernel versions, tested on 6.6.36.3,
release for WSL on July 2, 2024, and as of writing, 6.6 is still the latest available minor
release for WSL. Similar implementations of an INT collector with eBPF have previously
only been seen using ONOS on kernel version 4.15 (released in 2018), and not in WSL.
The visualization system uses Prometheus and Grafana, which are both simple tools to
set up and widely used. With specific configuration for Prometheus, it has proven to be
usable for INT network monitoring but has shown issues and limitations.
As part of making the solution a convenient product, an installation guide for the project
has been written through development and is present as the README file in the project
code in appendix A. The installation includes necessary modifications for WSL to be able
to run eBPF programs, configurations for Prometheus and Grafana, as well as some
changes made to enable interfacing between parts of the system.
4.1
Project Objective Pillars
This part of the chapter will be a strict evaluation of whether the implemented solution
adheres to the three pillars defined in the project objective.
4.1.1
Create INT­based Network Monitoring System
This was fully fulfilled, as results proved satisfactory monitoring of the limited test net­
work. There are significant limitations to the system, as it is only able to monitor one
specific form of INT, namely hop latency in INT­MD. Improvements can also be made on
the collector, where the Python part of the program may become a limiting factor for pro­
cessing speed, in part because of modifications made to ensure values for every report
are posted. Prometheus’ pull­method also sets significant requirements and limitations for
how telemetry reports can be read and works best with reports sent in regular frequency
such as ICMP pings at 1 second intervals.
4.1.2
Use Newest Software Versions Where Possible
A significant amount of work has been put into making the P4 implementation and the INT
Collector work on Ubuntu 24.
The P4 implementation has been updated to work on the newest versions of P4Language
and P4Runtime (respectively v1.2.5 and v1.4.1). However, the newest versions for INT
and for telemetry report format are not used in the implementations for the P4 program
or the INT collector. They currently use v1.0 (2018) for INT rather than the newest, v.2.1
(2020), and telemetry report header format uses v1.0 (2018) rather than v2.0 (2020).
4.1.3
Ensure Convenience for Development on the Project Where
Possible
A direct way this was targeted was by developing the project and making it usable on WSL.
Having the project runnable on WSL and documenting the necessary kernel changes
Advanced Network Monitoring with SDN and In­band Network Telemetry

==PAGE:72==
should facilitate easier use and development for users who primarily use Windows. The
choice of using the Grafana and Prometheus combination was also heavily weighted by
the popularity of the two, making it more likely to be a familiar experience to developers.
Requiring modifications to the kernel configuration is not a very user­friendly experience,
however, this was necessary, as the use of WSL and eBPF was a goal as well.
4.2
Challenges and Implications
Working with the eBPF implementation proved to be a great challenge and continues to
hamper the value of the system with difficult installation and limited functionality. Installa­
tion of the BCC, requiring custom kernel configuration and recompilation, is a steep task
to ask of new developers. It is, however, completely necessary to be able to run it on
WSL. Changes in modern Linux kernel versions have also made it so that older projects,
such as the original INTCollector[86] are too large and complex and must be simplified.
Despite this, eBPF has immense value as it is the collector option able to process teleme­
try reports the fastest. Depending on how much traffic the monitoring system is expected
to monitor, a slower collector program could be favorable, if the implementation is easier.
The current eBPF implementation has been simplified to only work with hop latency INT
values, which matches what P4 implementation outputs in telemetry reports. This, how­
ever, means that changes to the INT monitoring in the P4 implementation would break
the functionality of the collector. It would be preferable if modification could be in one part
of the system without requiring the work of similar changes in other parts. The current
collector is built this way in part because of the size and complexity limitations of eBPF
programs, but it is likely possible to still make some improvement with regards to rendering
it more versatile.
The database scrape­method of Prometheus appears to be a suboptimal solution for net­
work monitoring of packets. While it can be configured to provide sufficient monitoring
for most cases of network monitoring, it is not viable for monitoring inconsistent teleme­
try reports, e.g. if real traffic were to be monitored, where there are no regular intervals
between packets, and data can be bursty.
4.3
Future Work
4.3.1
Update to New INT and Telemetry Report Format Versions
To apply the goal of using new versions of software, updating the monitoring system to
use INT v2.1 and telemetry report v2.0 should have high priority. This would require
the P4 program to format the headers differently, and the eBPF collector program would
correspondingly need to parse the new headers correctly.
4.3.2
Test on Other Network Topologies
The system has only been used on the network topology described in section 3, Figure 3.1,
and only on a virtual Mininet network. There is significant value in seeing how different
parts of the system perform with a larger topology, possibly using several different INT
sinks and even several collectors. Currently the network is implemented with hard­coded
switching through the runtime json files, which means it would require significant work to
expand it, likely adding routing to the P4 controller.
4.3.3
Test and Measure Negative Impact of INT on Network
SDN­based networks generally create overhead traffic. Packet parsing can cause higher
latency, and extra traffic can be created either as communication to the CPU, as additional
headers, or in the case of INT with telemetry reports, the reports are transmitted over the

network. To get a realistic assessment of this impact – particularly regarding the packet
processing time – this should be done on hardware as opposed to a virtual network.
4.3.4
Enable Other Database Options
Since Prometheus has certain limitations, the system should have other database options
available. A different database would need to work with data pushing from the collector
rather than with scraping. The database can either run in tandem with Prometheus and
possibly be used to monitor different kinds of traffic, if the collector can be modified for this
– e.g. if the original packet uses ICMP, post the data to the HTTP server for Prometheus,
and in case of UDP push the data to the other database.

Conclusion
This thesis set out to explore and overcome several common challenges in the devel­
opment of SDN­based monitoring systems, particularly those built on the P4 language
and incorporating In­band Network Telemetry. By grounding the work in three clearly
defined pillars – building a fully functional INT monitoring system, utilizing the latest soft­
ware versions, and enhancing developer convenience – it provides a realistic and modern
implementation that aligns with ongoing trends and obstacles in the SDN community.
The implementation achieved its primary goal: The development of a network moni­
toring system capable of tracking per­hop latency using INT metadata inserted by P4­
programmed switches. A key factor was the use of a high­performance telemetry data
collector using eBPF, adapted to run on the latest WSL­enabled Linux kernel. This is a
significant improvement over many existing solutions, which rely on outdated components
and software stacks, making them increasingly difficult to maintain or extend.
Efforts to modernize the toolchain were successful to varying degrees. The core P4 im­
plementation and runtime were updated to their latest versions, and the entire system
was built to operate on Ubuntu 24.04. However, the INT and telemetry report formats
used are still from 2018 (v1.0) rather than standards introduced in INT v2.1 and telemetry
report v2.0. Similarly, although the eBPF collector operates reliably in its current scope,
it lacks generality: Any changes in the telemetry format would require parallel changes in
the collector, limiting the system’s adaptability and scalability.
Convenience for future developers was also a primary design objective, and running the
entire project within WSL contributes significantly toward that goal. Developers using Win­
dows systems can now engage with SDN tooling in a more natural environment, avoiding
virtual machines and other cumbersome solutions. Documentation – including an installa­
tion guide – further lowers the entry barrier for newcomers. However, the need to modify
configurations and recompile the Linux kernel remains a considerable challenge for ease
of deployment.
Looking ahead, several avenues for further work have been identified. These include:
• Updating the system to support the latest INT and telemetry header formats.
• Extending the system to support larger and more dynamic network topologies.
• Evaluating system performance under real­world traffic conditions on hardware plat­
forms.
• Integrating more adaptive database solutions.
Collectively, these improvements would greatly enhance the practical viability of the sys­
tem.
In summary, this thesis delivers a modern, functional, and developer­aware SDN monitor­
ing system while contributing valuable insight into both the technical and structural chal­
lenges facing INT­based development today. It serves both as a tool and a case study
that reflects the current maturity of SDN ecosystems and helps shape the path ahead.
