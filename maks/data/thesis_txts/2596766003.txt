Abstract
This master thesis investigates whether it is possible to perform autonomous assembly
of a complex object with high precision, when the poses of the object’s parts are initially
unknown. The research is conducted using a single robot arm equipped with a two-finger
gripper and a camera.
A three-step model consisting of localising the object parts, categorising the object
parts, and, finally, assembling the object parts is proposed as the final solution. The
localisation is utilising deep learning object detection and the categorisation is utilising
deep learning image classification.
The final solution provides good results in the context of completing the whole assem-
bly process, where it manages to succeed 96.24% of attempts with an average running
time of 379 seconds.
Though it is endorsed to further expand on the solution and
optimise the current implementation if the system is to be applied in industrial manu-
facturing.

CHAPTER 1
Introduction
Robotics has been one of the main factors in the increased industrial productivity. The
ever-expanding use of robotics has helped workers with monotonous and simple tasks,
such as pick and place movement, and difficult heavy-duty tasks, such as automobile
assembly. Applying robotic automatisation has reduced the cost of manufacturing, and
resulted in completing tasks faster and more efficiently. However, robots in manufactur-
ing and assembly lines often lack in perception and vision of the workspace, which has
lead to the need of placing parts in a structured environment, so the robots know the
exact position of the parts without having to sense this themselves. And setting up this
structured environment is a cost in the manufacturing process.
In order for manufacturing in unstructured environments it is necessary to equip
the robotic manipulators or their environment with perception sensors, such as cameras.
This allow the robots to handle flexible tasks autonomously, where the flexibility grant
an ability to work with arbitrary initial conditions and make impromptu adjustments
during processes. Even though progress in computer vision is taking place at a rapid
pace it is difficult to achieve high precision rivalling that of structured environments and
fine assembly has not yet reached industrial processes in a large capacity.
1.1
Problem Formulation
The aim of this project is to explore a solution for automated assembly using a single
robot arm and a camera for perception. The assembly to be handled will be a rather
complex object that requires intricate path-planning on top of detecting and classifying
the different parts that make up the object. As the parts are close-fitting in the final
configuration of the object, it is necessary to have high precision when handling the
parts. The whole assembly process is to be conducted with no feedback except for the
visual perception.
The main research focus will be the following question:
Is it possible to achieve high-precision robot assembly in an un-
structured environment with eye-in-hand camera perception
and a single robot arm equipped with a gripper?

CHAPTER 2
State of the Art
Robotic assembly of small and intricate parts in an unstructured environment is desired
yet still out of reach of commonplace off the shelf robot solutions. While structured
environments can appreciate very automated processes, such as automobile manufactur-
ing, multiple problems arises when the environment in the workspace is unknown. ”The
main difficulties arise in the precise localization of the parts in an unstructured environ-
ment and the control of contact interactions.” Suarez-Ruiz and Pham (2015) said [SP15].
Their study will be detailed in Section 2.2.
2.1
Part Localisation
Recent trends in computer vision have been showing a resurgence of feature-based meth-
ods and machine learning, especially with the advancement of deep learning algorithms
surpassing prior methods in tasks like classification and segmentation. [Mah+19]
Lin et al. (2018) used a deep learning approach to recognise objects and estimate the pose
in 6D of these objects. In their study they had images including multiple overlapping
objects with varying positions and rotations in all dimensions. The camera that was
used in this study was an RGB-D camera, which added depth as a fourth channel that
had the distance from the camera to each pixel in addition to full colour images. They
used convolutional neural networks (CNNs), the customary type of network to use when
analysing visual imagery, to perform semantic segmentation.
With these networks each pixel was linked to the respective object, if such an object
was present. The output of the CNNs were refined by conditional random fields (CRFs)
that were applied for pattern recognition. As such, they managed to link pixels directly
to object and with this point cloud of information, along with the distance to each point
in the point cloud, an iterative closest point (ICP) algorithm was used to compare this
point cloud with the point cloud model of the 3D object to determine the object’s rota-
tion along all axes. Thereby they managed to find the pose of the object in 6D. [Lin+18]
In a study from 2017, Kehl et al.
showed that it was not necessary to rely on the
depth from RGB-D cameras to estimate 6D poses of objects in images and that it could
be handled using only the colour image. The study concluded that it was possible to
match and surpass the state of the art methods that used RGB-D data with colour-based
detectors. The localisation of the objects in the image was conducted using the Single
==PAGE:16==

Shot Multibox Detector (SSD) [Liu+15] that outputs bounding boxes for the possibly
multiple objects in an image. In addition, a classifier network was trained for a discrete
6D pose space to determine the view from an image.
The 3D models of the objects were known in advance, and a 6D pose was to be
inferred from the 2D bounding box that was outputted from the SSD network. Parsing
these bounding boxes and the view determined from the view classifier, a pool of 6D
pose hypotheses was created for each object, from which the best 6D pose was chosen.
The object pose was then refined using an edge-based approach to further determine the
actual pose of the object. [Keh+17]
2.2
Contact Interactions
Suarez-Ruiz and Pham (2015) used a setup with two robot arms, both equipped with
parallel-jaw grippers, and optical motion capture for perception in an eye-to-hand ap-
proach. The robot arms were further equipped with force torque (FT) sensors. The
goal of their project was to assemble an IKEA chair autonomously in an unstructured
environment and they were in particular focusing on bimanual pin insertion, where the
two robots had to work together to insert a small wooden pin into a pre-drilled hole.
The objects were all rigid and nondeformable.
Due to the uncertainty in perception ranging ± 3 mm it was not possible to simply
pick up objects. To combat this error they introduced compliant grasping that allowed
the gripper to follow the table exerting a low force within the mechanical compliance of
the grippers. The gripper was then closed while the force controller maintained contact
with the table. This way of grasping the object allowed for less precision in perception
without detriment to picking up the objects in the same way each time. [SP15]
An eye-in-hand approach to the full assembly process was studied in 2018 by Litvak et al.
Using a single robot arm equipped with a gripper and a depth camera, they managed
high-precision assembly of a gear-like mechanism as part of the Siemens Innovation
Challenge. The objects were all rigid and nondeformable.
The perception scheme to find the parts in the workspace was managed in two stages:
first determining a rough pose estimate of all the parts from a position far away from
the parts and second moving the end effector, and therefore camera, close to the each
part to boost the pose estimation accuracy. They managed to obtain an average pose
estimation error of 2.16 mm and 0.64°.
Even with this improved accuracy when picking up the parts, it still resulted in
errors, and they added a safety using force-based feedback during assembly of each part.
If the force sensor registered a significant force in the opposite direction it would indicate
unexpected contact with either the base plate or another part. [LBB18]

3.1.1
Changes to the Object
Block A, confer Figure 4.4, seemed to be completely square seen from the end of the
block and had to fit in an also seemingly square hole, which is best represented in Figure
7.5 in Section 7.2.2.3 where the actions taken to fit the block into the hole are described.
However, it turned out that the width changed slightly, giving Block A a width 20 mm
on one side and 20.3 mm on the other side. This small difference made it so the block
could only fit in the hole when it was turned correctly.
As there was no way for a person to discern which side was which without trying
both in the hole, it was deemed a fault of intolerance during production, and it was
concluded that it was not part of the puzzle to figure out how Block A should be turned
to assemble the puzzle. Therefore, one side of the block was ground, such that the block
could fit in the hole no matter how it was turned.

CHAPTER 4
Solution Concept
To go from having all the blocks scattered in an unstructured environment to a fully
assembled wooden puzzle, the solution was based on three separate steps:
Step 1: Localising the blocks
It was known that there would be six blocks randomly
placed in a set area of the workspace and the robot was equipped with a camera. This
camera was used to capture the whole area the blocks could occupy. With computer
vision it would be possible to detect where in the image these blocks were lying, and
from there localise the blocks in workspace coordinates.
Step 2: Categorising the blocks
The blocks were all fairly similar in shape and
appearance, which made it difficult to determine from a single glance which was which.
To distinguish them, each of them would be observed from different angles. Once again,
the camera would be used and computer vision applied.
Step 3: Assembling the wooden puzzle
After the blocks were ascertained, the
assembly process would be managed with simple open-loop processes, placing the blocks
according to pre-planned paths.
4.1
Computer Vision
The only perception that was used in the setup was an eye-in-hand camera, which re-
sulted in computer vision being the chosen solution to work with this feedback. The
captured images were used in different ways, depending on the situation, for both local-
ising and categorising the blocks.
4.1.1
Block Localisation
Detecting the position and angle of the blocks was managed using object detection,
which will be expanded further in Chapter 5. According to Mahoney et al. (2020) a
deep learning approach can outperform traditional computer vision in an object detection
paradigm: ”Compared to traditional CV techniques, DL enables CV engineers to achieve
greater accuracy in tasks such as image classification, semantic segmentation, object
detection and Simultaneous Localisation and Mapping (SLAM)” [Mah+19].
==PAGE:28==
16
4 Solution Concept
Multiple different types of models are available for detection networks, most com-
monly split in two different categories:
Two-stage detectors
This architecture is split in two parts, where the first part is
to find regions of interest and the second part is to use these region proposals for object
classification and bounding box regression.
An example of this type of detector is the Faster R-CNN (Faster Region Convolu-
tional Neural Network), which is described in Section 4.1.1.1.
Single-stage detectors
This architecture is simpler than the two-stage detectors’ as
the whole object detection is treated as a regression problem that takes care of the
classes and bounding box coordinates.
Examples of single-stage detectors are the SSD (Single Shot Multibox Detector) and
YOLO (You Only Look Once).
Due to the simpler architecture, the single-stage detectors have a considerably reduced
detection time compared to two-stage detectors and with single-stage detectors it is
much easier to obtain real-time detections in camera feeds and the like. The trade-off is
that this simpler architecture typically reach lower accuracy rates [SI18].
The number of times object detection was used to find the blocks was only once per
assembly, and therefore it would not be used for high frame-rate detection, but instead
for a single image. Since the highest accuracy was desired, a two-stage detector would
be the best solution, and the Faster R-CNN model was chosen.
4.1.1.1
Faster R-CNN Model
The Faster R-CNN’s [Ren+15] prediction model can be seen in Figure 4.1. The process
can be split in three parts:
Part 1: CNN (Convolutional Neural Network)
The first part of the model will
extract features using convolutional layers and output feature maps. These features can
be shapes, colours, etc.
Part 2: RPN (Region Proposal Network)
The second part is to slide a network
over the feature maps from the CNN and determine whether or not there are objects
and in that case predict bounding boxes.
Part 3: RoI (Region of Interest) Pooling
The third and last part of the model
is a network that takes the feature maps from the CNN and the proposed regions from
the RPN as inputs and determines what object would be in the region of interest using
classification and determines the bounding box of the object using regression.

CHAPTER 5
Block Detection
The implementation of the Faster R-CNN model was conducted using the TensorFlow
Object Detection API, a popular object detection framework. This plug and play ap-
proach to open-source framework build in TensorFlow allowed for easily constructing,
training and using new object detection models.
In order for the object detection framework’s neural network to be applied, it was
necessary to train them for the task at hand. While the pre-trained network allowed
for detection of people, planes and chairs, this was not trained for detecting blocks.
The neural network required training data that would make the network generalise well
to the actual images that would appear during use [Ng18]. These were images with
multiple blocks in them, scattered randomly throughout the whole image. The images
were labelled with the positions and classes of the blocks in the images and were used
as inputs for the object detection model.
The outputs from the object detection network gave positions in pixels and therefore
in the image’s coordinate system. The positions in the image coordinate system were
translated to workspace coordinates, which would be possible to use for move commands
to place the robot in the right position to pick up the blocks.
5.1
Pre-trained Model
The chosen pre-trained architecture and data set was given by the
faster_rcnn_inception_v2_coco model from TensorFlow’s model zoo. The architecture
of the convolutional layers was the Inception-v2 neural network [Sze+15] and the data
set that the network had been pre-trained on was the COCO data set [Lin+14].
5.2
Data Acquisition
When generating the training sets, it was important to note what kind of data that
could show up in the online use of the setup. While it was possible to narrow the error
sources from the environment due to the fact that the setup was to be established as a
possible solution in an industrial setting, where parameters such as the substratum and
lighting could be controlled, there was still the need for some robustness in other places.
The robot arm would maybe not place the block at the exact same pixels every time,
leading the position to be offset a few pixels in any direction, or maybe with a slight

Therefore, it was necessary to incorporate these minute changes in the data set so it

• Position: Changes in the position in the image, placing the block in every possible

• Angle: Changes in the angle of the block compared to the image frame.

While the training set was acquired according to the methods described in this section,

data.

especially lighting, were more perceptible and more representative of how it would pos-

Block Categorisation
The block categorisation relied on sequential classifications of the block sides, turning
the block 90° between each classification. Some of the blocks looked the same from a top-
down view depending on their rotation, making it difficult to categorise the block only
from a single image. As such, the blocks themselves weren’t categorised immediately
from a single image. Instead, the block sides were classified from the images and a block
categorisation constructed from multiple block side classifications.
6.1
Pre-trained Model
The VGG-16 neural network was pre-trained from to PyTorch’s model zoo. The database
used for pre-training was ImageNet [Den+09].
6.2
Data Acquisition
Generating data sets was conducted much like the data sets for the object detection
paradigm described in Section 5.2. However, due to the fact that the robot itself is
placing the blocks when handling the block categorisation, the block placement in the
images for classification is not totally random. As such, the position and angle of the
block in the data set only varied slightly throughout the data, allowing a more focused
training that would generalise well to the actual use case, where the blocks will not differ
much when placed and picked up by the robot. While the range of the angle of the block
was very limited in this data set generation, there was still the fact that the block could
be rotated 180°, and for each class the block was also rotated 180°.
The lighting was handled the same way as for the object detection, described in
Section 5.2.
The background was chosen to be black to remain consistent with the
substratum colour.
The varying positions, angles, lighting and rotation in the data set can be seen in
the two examples from the same class in Figure 6.1.

table.

6.4
Determining Block and Direction
The block and the direction the block is rotated can be determined from two sequential
classifications. Taking for example Block B; it can be be seen in Figure 4.5 that by
turning the block the class sequence would be 1 →2 →3 →4, and the sequence would
of course be a circular sequence, because it would go from class 4 to 1 if the block
was continuously turned. If the block was rotated 180° the class sequence would be
4 →3 →2 →1. Seeing as each of these blocks have distinct sequences of sides, it was
possible to gather both which block it was and how it was turned with consecutive block
side classifications.
In case the sequence of classifications was 4 →3 it could be concluded that it was
Block B and that the block was turned 180°.
Conferring Figure 4.9, it can be seen that if the sequence was 9 →g the block would
have to be Block F and that the block wasn’t rotated.
6.5
Fail-Safe Operation
In case the two classifications resulted in a sequence that did not exist for any of the
blocks, it could be deducted that at least one of the classifications were wrong. In this
case, both classifications were discarded, and a new attempt was made to determine the
block, by continuing to turn the block and classify the two next sides. If the block was
not categorised at this point, a failure code was returned, and the assembly process was
stopped. This allowed for a higher chance of categorising the block even if there should
be a misclassification.
However, this procedure did not resolve the cases where a misclassification resulted
in a valid sequence and a miscategorisation.

CHAPTER 7
Assembly
The assembly process was handled with pre-planned paths for each block. There was
no feedback, so each movement of the robot was manually programmed. This included
movement of the robot in both workspace coordinates and tool coordinates for a process
that would only work in this specific environment.
A rather large part of the path-planning included reducing position uncertainties
of both the assembled and to-be-assembled blocks, such that the blocks fitted when
interlaced during the assembly.
7.1
Assembly Order
The sequence in which the blocks were put together was dependent on the fact that only
a single robot arm was used. It was not possible to hold anything in place or keep two
blocks together after assembly of a block, since the robot had to be used to assemble
the next block afterwards.
The solution included the assembly unit, confer Section 3.2.4.3, which would cradle
the first block, and the subsequent blocks were then stacked on top of this, such that
they could either lie on top of or hang from already assembled blocks. The first five
blocks all followed this procedure, however the last block had to be handled differently,
as it was necessary to push it in.
To allow for easier entry of the last block, the hitherto assembled blocks were picked
up and rotated in the assembly mount. After releasing the blocks from the gripper, only
gravity acted on them, and they flattened naturally into a bigger opening for the last
block. The last block was then slightly lodged into the hole, before it was released by
the gripper, and then pushed the rest of the way in.
The assembly process can be seen in Figure 7.1 which shows how each block managed
to stay assembled even after the robot arm released it.

During the assembly small uncertainties were introduced when placing the blocks. These
uncertainties came from the block sticking to the gripper when opening, dropping the
blocks a few tenths of a millimetre when placing them, the robot’s own repeatability

to incorporate a method to reduce the accumulated uncertainty during the assembly,
which is described in Section 7.2.2.
7.2.1
Rectifying Grasp
Using the gripper it was possible to adjust the position of a block after picking it up, such
that the object detection uncertainty did not persist throughout the assembly process.
As the gripper position uncertainty was only reliant on the repeatability of the robot,
block position uncertainty was reduced to this same uncertainty.
The block was placed and the gripper moved down next to the block. The gripper
was closed slightly, such that the width of the gripper fingers was the same as the width
of a block. The gripper then pushed the block laterally, making sure the side of the
block was parallel to the gripper, thereby removing the lateral uncertainty. The block
was picked up from this position, and any longitudinal deviation was removed due to
the fact that the gripper picked up the block from the ends. This process can be seen
in Figure 7.2.
7.2.2
Simultaneous Assembly Adjustment
The raised edges on the assembly unit described in Section 3.2.4.3 allowed for pushing the
partially assembled object into a fixed position and reducing the uncertainty pertaining
to its position considerably.
Just like the method described in Section 7.2.1, these adjustments resulted in posi-
tions uncertainties in the same order of magnitude as the repeatability of the robot.
The method was used in key points during the assembly process, where the blocks
had to be heavily interlaced, which was the case for the assembly of the last three blocks.

CHAPTER 9
Discussion
The results have shown that this concept for assembling a wooden puzzle is a viable
solution and can manage to run unattended for a long time and with minimal mainte-
nance. There are still some improvements to be made, which will be considered in this
chapter.
9.1
Additional Feedback
The assembly process rarely failed, though it was still a possibility, and if it were to fail
there was a good chance of the robot arm destroying the assembly unit. Even if the
robustness of the assembly unit was improved to an extend that the robot arm would not
have enough force to break it, either by using plastic moulding or making the assembly in
another material than 3D-print, a failed assembly could then result in damaged wooden
blocks.
This problem could rather easily be circumvented by improving the feedback scheme,
specifically the force and torque feedback sensors in the setup. The robot used in this
project has a simple Force Stop functionality where it only stops if a threshold is ex-
ceeded, and this stop results in a complete stop of the robot and needs an operator
to re-enable the robot. With a dedicated FT sensor these issues could be avoided by
stopping before a Force Stop is reached.
If the FT sensor reaches a threshold it can be inferred the assembly has failed, and
the robot can disassemble the hitherto assembled blocks before starting a new assembly.
It might put the misassembled block, the assembled blocks, and the still to be assembled
blocks into a container and the blocks could then be presented for the robot again as
a new object to be assembled. Since little to no force should be exerted by the blocks
during assembly this threshold can be kept much lower than the robot’s Force Stop.
9.2
Improving Vision Performance
While the computer vision paradigms could be improved by simply increasing the amount
of training data and training time, this is a time-consuming process that results in
diminishing returns. There are other methods that could be implemented without much
extra work.
==PAGE:68==
56
9 Discussion
Controlled environment
By keeping the variables in the environment at a minimum,
the computer vision has less trouble generalising the features. These variables include
light and using a substratum that does not deteriorate or get dirty during use.
Using a diffuse light would result in few and indistinct shadows, reducing the distur-
bances for the networks to train on.
A hard surface with little friction, such as plastic, would not be damaged easily by
the blocks and allow for easy sliding during the grasp rectification process. An important
feature of the substratum would be the reflection, where a matt finish would reduce the
shine from light sources.
Capturing multiple images
In this project only a single image has been captured
by the camera each time either block detection or block side classification was desired.
However, as the neural networks’ CNNs are especially dependent on spatial information,
a small change in the image, such as objects being moved mere pixels, could result in a
completely different result.
In the case where the object detection network falls short and doesn’t detect all six
blocks or detects a seventh false positive block, it is known that something is wrong,
however, the only solution is to return a failure code and not run the assembly process.
Instead, it would be a possibility to move the camera slightly up or down and try again,
as the blocks would have new positions in the image. This requires that not the full
workspace is utilised when the blocks are randomly scattered, since moving the camera
would mean capturing a new area of the blocks’ surroundings, which should not include
new disturbances.
When the block sides are classified it would be possible to capture multiple images
by moving the camera slightly, keeping the top-down view, but moving either to the
sides or changing distance to the block. This could either be conducted if the network
did not return an answer with a high confidence or as part of majority voting using
multiple images for the classification: i.e. five images could be captured by moving the
camera slightly, each of them classified using the neural network, and then a majority
voting decided which class was the most probable one. The trade-off for this increased
robustness would be additional time in each assembly, however, a misclassification could
be disastrous if the robot tried to assemble a block that simply did not fit.
9.3
Initial Block Scrambling
In this project, the focus has been to find and assemble the complex object from an un-
structured environment. However, the environment has been artificially unstructured,
as the blocks have always been scattered in an unconnected fashion, which would al-
low the gripper enough space to pick up the blocks. Depending on the setup of the
block feeding system that is to operate prior to this system’s setup, this assumption of
completely separated blocks might be too optimistic or expensive to implement. For
the setup to be able to work in an unstructured environment where the blocks can be
==PAGE:69==
9.4 Implementation in Production Line
57
lying flush against each other or perhaps piled up, it would require preprocessing of the
environment to a point where the blocks are separate.
This preprocessing would need a way to move the blocks, and either re-scramble them
completely or purposefully move them and try to separate connected blocks. Common
for these processing methods would be that it would be necessary to determine whether or
not the blocks need any separation or not, which could be conducted using many different
types of computer vision or be based directly on results from the object detection network
designed in this project.
Re-scrambling the unstructured environment
This process could be introduced
in many different ways, such as a movable substratum that could vibrate and reposition
the blocks at random, a scraper that would push all the blocks into a container and
would then pour the all the blocks back on the substratum, or moving the gripper back
and forth over the substratum hitting blocks on its way indiscriminately.
This type of process would be akin to a Las Vegas method, where it would keep
going until it found a solution of scattered blocks. This could be a lengthy process and
possibly infinite.
Focused rearrangement
Based on prior perception this method would need to know
where the regions of connected blocks were, before trying to untangle the blocks. Then
a focused effort could be applied in this region, which could be comprised of i.e. using
the gripper to move blocks in the area.
This could possibly be faster than re-scrambling the entire batch of blocks, since fewer
blocks will need to be rearranged on average. But there also exists the possibility of the
setup painting itself into a corner, constantly trying to fix the same difficult situation.
9.4
Implementation in Production Line
It is a definite advantage to be able to process objects in an unstructured environment, as
more leeway is allowed between tasks, which commonly means cheaper implementations.
Additionally, many changes can be made to improve the setup at little to no extra
costs, as described earlier in this chapter, which would reduce the need for maintenance,
making the solution even more viable.
However, there are still multiple factors to take into account for whether or not it
would make sense to implement an automated setup in a production line.
Time
The whole assembly process took 379 seconds on average to complete, which is
much more time than a person trained in assembling it would need. A person would
easily manage to assemble the whole block from an unstructured environment in less
than a minute, which is almost an order of magnitude difference.
Using robots in this use case would lead to a longer work time, however, it would
be possible to have the robots work non-stop. But it comes at an added initial cost too.
==PAGE:70==
58
9 Discussion
An initial cost in organising the automated process in both time and money, as it takes
time to train neural networks and determine assembly path-planning.
Cost
The cost of this kind of setup is a sum of the initial cost and maintenance. The
initial cost is very high, as it requires many man-hours by skilled engineers to setup
and the instruments are also a significant investment. It would definitely be cheaper to
employ people for the assembly process if a moderate amount of products were desired
in a short time frame.
This initial cost would have to be reapplied at small changes in the assembly process,
while retraining people could be accomplished faster.
Assembly path-planning
A way to reduce a portion of the initial setup costs would
be to generalise the assembly path-planning to happen around a point. In this project the
assembly process has been based on the assembly unit placed in the robot’s workspace
and will only ever work with a setup exactly like this. However, since this has been
accomplished by calling movement commands to move the robot to specific points, these
points could all be considered in the objects coordinate system.
With this kind of setup it would only be necessary to give the assembly unit’s pose in
the robot’s workspace coordinates to allow it to assemble the object. If multiple robots
were to be assembling the object, the process could be applied easily to new setups
allowing for scalability of the operation.
Production uncertainties
This project has been conducted using a single complex
object and solutions have been conducted with this element in mind. This includes
the colouring of the wood for computer vision and the way the object has been cut for
assembly. However, as the object is made of wood, there is a good chance that other
similar objects might not have the exact same dimensions or appearance, both of which
could result in misassemblies.
Differing appearance is an issue that can be solved by changing the training data to
incorporate many different objects.
Differing dimensions, however, are difficult to resolve, if the uncertainties are too
large for the assembly process. Since there is no feedback during assembly, it would
possibly lead to not just failures in the assembly but actual ruination of the assembly
unit. Even if this problem is circumvented, an autonomous solution would not make
sense if a part of the objects were discarded because they didn’t fit this setup, even
though they might still have been perfectly usable. To combat this uncertainty would
require a more sophisticated solution that incorporated force and torque feedback and
worked with compliant assembly as described by Suarez-Ruiz and Pham (2015), confer
Section 2.2.

9.5
Attempting the Assembly Process on
Similar Object
After having designed the localisation, categorisation and assembly processes using a
single wooden puzzle, a second similar puzzle was introduced. With no changes to the
setup, the new object was used to test the flexibility of the system. Only a single attempt
was made.
The assembly of the new object was almost completed when a fault happened. It
had been localised, categorised and the first five blocks had been assembled, however,
with the last block the system failed in the step in Figure 7.1(h).
In Section 3.1.1 it was described how Block A had been ground down to become
square and fit into the hitherto five assembled blocks no matter how it was turned, as
this was deemed the intended use. The new object had the exact same issue and in the
single attempt made, it was turned the wrong way, which meant it did not interlace
correctly with the already assembled blocks. If this did not happen, it is believed that
the assembly would have been successful.
This, however, proves to be quite the issue for the setup, since this kind of problem
cannot be overcome with the current design of block categorisation. The only way to
distinguish this small difference in sides would be to look at the grain of the wood, if
using computer vision, however, this changes drastically from object to object, which
means it would not be possible to generalise to. Instead, it would be necessary to take
a mechanical approach to determine the width of the block side and whether it should
be turned to fit.
In Section 3.1.1 it was found out that the difference between the two sides of Block
A is 0.3 mm and in Table 3.3 the specifications show that the gripper has a resolution
of 0.4 mm. With the gripper used in this project it might just be possible to determine
which side is which from the position of the gripper fingers when the block is grasped,
however the resolution is a bit too low to be certain. If a gripper was introduced with a
resolution safely below this difference of 0.3 mm, then it would be possible to infer from
the gripper fingers’ positions the width of a grasped block and whether or not the width
of this block could fit in the hole.

CHAPTER 10
Conclusion
The aim of this thesis was to determine whether high-precision assembly from an un-
structured environment, in which the individual object parts were randomly scattered
yet separated from each other, was possible. To answer this, the thesis followed the pro-
cess of the implementation of such an assembly scheme, which was conducted in three
steps: localisation, categorisation and assembly.
The localisation was designed using object detection based on the Faster R-CNN
neural network, which achieved an mAP of 98.66% and managed to correctly detect all
six blocks in 94.78% of images.
The categorisation was designed using image classification with consecutive classifi-
cations to categorise each block. The classification was implemented with the VGG-16
neural network, which achieved an accuracy of 99.95% and needing two consecutive
correct classifications, the categorisation achieved an accuracy of 99.90%.
The assembly was conducted using pre-planned paths in a setup without any feed-
back. The paths were found manually and the same assembly paths were used for each
assembly, resulting in an assembly accuracy of 99.48%.
With this setup, a full process run had a success rate of 96.24% taking 379 seconds to
finish on average. It is concluded that it is possible to achieve high-precision autonomous
assembly of a complex object in an unstructured environment, using a single robot arm
equipped with a gripper and an eye-in-hand camera.
10.1
Future Work
Reducing running time
One of the biggest hurdles for moving this kind of assembly
setup to an actual production line is the fact that it is very slow, compared to a human
operator, and with the steep setup cost it is not necessarily a viable alternative. If the
process was conducted at the same speed a human would be capable of, it would be an
attractive alternative, and the solution would still be much more viable if the running
time was reduced even though it might not compete with a human still.
Reducing the running time would mainly be from designing a more efficient workspace,
where the robot doesn’t have to move as far to pickup and place the blocks; planning
shorter assembly paths, simply reducing the distance the robot has to travel; and in-
creasing the speed of the robot movements, as this speed has not been optimised and
can be further increased with little to no detriment to the assembly process.
==PAGE:74==
62
10 Conclusion
Re-scrambling solution
An earlier discussed issue with the current setup is that the
blocks have to be completely separated in their initial poses. This is a limited approach
that does not allow much flexibility.
Solutions to this issue have already been examined, and it is deemed an important
implementation before it could be rolled out on a larger scale.
Compliant assembly
While the repeatability of the setup has proved that the same
object can be assembled and that it almost managed a never before seen object, there
were still times where the assembly process failed.
Introducing compliant assembly,
where the robot can find the gaps it needs to fit a block in with feedback, would allow
for quicker manual path-planning or might even allow a totally autonomous assembly.
Any feedback scheme would have the potential to further reduce the number of
failures in assembly.
