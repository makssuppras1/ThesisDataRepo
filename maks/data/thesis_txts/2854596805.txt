Abstract
This thesis explores the application of machine learning (ML) techniques to enhance fraud de-
tection in the financial sector by leveraging secure multiparty computation (MPC) to ensure data
privacy. Simulated and synthetic datasets, primarily IBM’s credit card fraud dataset, are utilized
for training and testing the ML models. These datasets contain credit card transactions and per-
sonal information, enabling the identification of anomalous patterns associated with fraudulent
activities.
Feature engineering techniques were used to improve data quality and assist the ML models in
capturing patterns while minimizing complexity and reducing training time. During training and
testing, a wide range of ML models were tested, which included feedforward neural network (FNN),
logistic regression (LR), decision tree classifier (DTC), isolation forest (IF), and random forest clas-
sifier (RFC). All the models were evaluated using their Precision-Recall Area Under the Curve
(PR-AUC) metric score to measure the performance of the models using imbalanced datasets. The
feedforward neural network was specifically selected to be trained and tested in a cryptographic
environment using the framework MP-SPDZ. By employing MPC protocols and a semi-honest secu-
rity model, this thesis enables financial institutions to train an ML model collaboratively without
revealing proprietary data.
The findings highlight the feasibility and potential of feedforward neural network-based models in
MPC for credit card fraud detection, providing a robust privacy-preserving solution. Although scal-
ability challenges remain, this thesis offers insights into balancing data privacy and collaborative
machine learning. The analysis includes performance trade-offs between cryptographic operations
and model efficiency, showcasing the promise of MPC for secure, collaborative applications. This
thesis lays the groundwork for deploying MPC protocols in real-world systems by bridging the gap
between data privacy and effective fraud detection. It offers insight for future research to enhance
the scalability and performance of machine learning models.

1
Introduction
In today’s digital economy, financial fraud has become and always has been a major concern, with
fraudulent techniques like phishing, money laundering, and credential theft affecting organizations
and individuals alike. According to [34], fraud causes a large financial loss globally, with an esti-
mated impact of $102.6 billion in 2023. As organizations and individuals increasingly rely on credit
card transactions, with cash being phased out, the risk of fraud continues to grow, creating an
urgent need for effective fraud detection methods within the banking industry.
Traditional fraud detection systems for online transactions rely on a rule-based system, which uses
predefined rules, such as exceeding a spending limit, to identify potential fraud. Having only a
rule-based system will limit these fraud detection methods by their reliance on predefined patterns
and inability to adapt to new fraudulent schemes [14]. Machine learning techniques have begun
to be implemented, with real-time models trained to recognize patterns and anomalies based on
transaction history, location, and user behavior. In today’s digital economy, it is almost impossible
for financial institutions to cooperate with sharing or using each other’s data to improve fraud
detection due to GDPR, amongst other things.
This thesis addresses this challenge by exploring various machine learning models and training
the best-performing ones in a cryptographic environment with multiparty computation (MPC) to
enable parties, such as banks, in the financial sector to collaborate on proprietary data without
compromising privacy. This exploration would allow banks to combine their data with other banks
into one large dataset from which a machine learning model can be trained without revealing the
actual data, as it is kept private through secret sharing.
Through a proof of concept, exploratory data analysis, and feature engineering, this research seeks
to understand how cryptographic operations and methods impact the accuracy and performance
of machine learning models. Additionally, it explores the feasibility of training machine learning
models within a cryptographic framework for collaboration between parties in the financial indus-
try.
1.1
Problem definition
The project aims to integrate multiparty computation (MPC) with machine learning (ML) models
within the financial industry to detect fraudulent credit card transactions. This integration initially
involves choosing suitable datasets and creating and evaluating machine learning models without
using multiparty computation protocols.
Subsequently, the project will assess the impact of integrating MPC on the model’s performance
and accuracy based on the metrics defined in the evaluation section 9.3 and evaluate the security
benefits of these protocols.
The objective is to build efficient, accurate, and privacy-preserving fraud detection systems by
utilizing machine learning and MPC protocols and ensuring a comprehensive understanding of
their standalone and combined impacts.

3.1
Multiparty computation
Multiparty computation (MPC) is a subfield within cryptography that allows multiple parties to
jointly compute a function over their inputs while not revealing the input and, in some cases, the
outputs [53]. The mechanism of MPC creates new opportunities for collaboration among banks and
entities, where it is paramount that data, such as personal information, credit card transactions,
and proprietary data, remain secure and private. This allows, for example, the banks to collabo-
ratively train and run a machine learning model, such as a neural network, without compromising
privacy, as parties must be involved in the training and inference process.
This can be accom-
plished by using the various protocols that allow for MPC, where privacy is preserved through a
concept known as secret sharing, which is explained in section 3.1.3 for the replicated-ring-party
protocol.
The focus is to utilize an MPC framework and examine its impact on the machine learning models,
as per research question 5 in section 1.2. This entails which machine learning operations exist
and how they provide security. Implementing MPC can be carried out using various frameworks
and protocols within these. MP-SPDZ is the chosen cryptographic framework for this project. This
framework was chosen because of its high number of supported protocols, integration with Ten-
sorFlow and PyTorch, and high flexibility.
These reasons are instrumental in keeping the same
architecture for the feedforward neural network and finding the best privacy-preserving protocol
for this use case. It is important to mention that the various supported protocols can be suitable for
different use cases, and some are even good for specific operations conducted in the cryptographic
environment. Note that MP-SPDZ will be used as a tool for privacy-preserving machine learning
training and inference rather than a deep dive into the complex technical aspects of cryptographic
schemes.
Different security models of MPC protocols exist, where the level of security the protocol provides
depends on this. The focus of this project has been on the "semi-honest" version, as there is less
computational overhead and, consequently, fewer security constraints involved than with another
model called "malicious." These are explained in section 3.1.1 and 3.1.2.
3.1.1
Semi-honest
The semi-honest model is a security model used in cryptography. In this model, the participants
follow the protocol exactly as described without cheating or altering any steps. Furthermore, the
participants are curious, meaning they will try to learn as much as possible from the data they
observe during the protocol and still follow the protocol instructions [11].
In this model, specifically when examining the replicated-ring-party protocol, there are no mecha-
4
==PAGE:12==
nisms to detect deviations or verify correctness, such as message authentication codes (MAC) or
zero-knowledge proofs (ZKP), which are present in the malicious security model. A MAC ensures
that data, in this case, secret shares, remain consistent and unaltered during computation [49].
In contrast, a ZKP is a technique used to prove that computations on secret data are carried out
correctly without disclosing the actual secrets [22]. Note that MAC and ZKP techniques will not
be discussed in depth but are only mentioned to provide an overview of the difference between the
malicious and semi-honest security models. The absence of these mechanisms means that devia-
tions, such as halting network communication or sharing incorrect shares, could cause the protocol
to fail to execute properly or produce incorrect outputs.
3.1.2
Malicious
The malicious model is a security model used in cryptography. In this model, a participant tries to
corrupt the system by deviating from the protocol description in any way [11]. The malicious model
does not protect against multiple malicious adversaries. This includes altering data, injecting false
information, or manipulating protocol steps to gain additional information or disrupt the overall
process. In this model, the cryptographic techniques ensure that if a party acts maliciously, they
still cannot acquire information beyond what is allowed by the protocol, or it will cause incorrect
outcomes. As mentioned in section 3.1.1, the malicious security model employs MAC and ZKP to
detect deviations and verify correctness. In MP-SPDZ, all malicious protocols follow the malicious-
with-abort model, which means that if a deviation is detected, the protocol cannot recover from
that point onward [23].
3.1.3
Replicated-ring-party protocol
This project uses the replicated-ring-party protocol through the MP-SPDZ framework [24].
The
protocol requires exactly three parties to run and allows for multiparty computation. Parties can
collaboratively compute the output of a given function over a network where each party provides
their data without revealing the actual value of the data (input). This sounds contradicting, but by
using a technique called replicated secret sharing, the privacy of the data provided by each party
can be preserved.
Depending on the protocol in mind, multiple types of secret sharing methods exist, such as Shamir
secret sharing, Additive secret sharing, and replicated secret sharing [24].
Additive and replicated secret sharing are very similar. The difference is that in additive, the party
only holds one secret share, which means that all parties must be present to reconstruct the secret
(input). For replicated secret sharing, the default fault tolerance is that two of the three parties are
needed to reconstruct the secret - meaning that each party holds more than one secret share for
each data point.
During protocol execution, parties must work together to do arithmetic operations, such as addi-
tion, multiplication, division, etc. To preserve privacy, these simple arithmetic operations are done
differently in MPC. First, an example with addition will be explained; then, the example will be
expanded with multiplication. The other high-level arithmetic operations, such as division, expo-
nential, and subtraction, can be expressed using addition and multiplication, as these are complete
operations.

3.1.4
Addition
Suppose the parties would like to calculate the sum of three numbers, where each party wants to
provide a number Ni and i ∈(0, 1, 2) in a way that the other parties cannot directly read it. Each
party locally computes what is referred to as the secret shares of the number.
Since there are
three parties, three secret shares will be created for each number, and these can be denoted as
(Si,0, Si,1, Si,2) where Ni = Si,0+Si,1+Si,2 mod 2n. The secret shares are computed in such a way that
they are not uniformly generated, meaning that if N1 = 9 then the shares (S1,0, S1,1, S1,2) ̸= (3, 3, 3)
as this would potentially allow each party to calculate the original input from one share. The shares
are usually chosen uniformly at random from the ring Z2n (the highest number possible), except
for the last share, designated to ensure that Ni can be reconstructed again from the shares. So, if
N0 = 8 and (S0,0, S0,1) = (4, 3) then the last share must be S0,2 = 1, satisfying N0 = 4 + 3 + 1 = 8.
Additionally, correlated randomness is added to the results shared to provide additional security
so that the parties do not reveal partial information during intermediate results that need to be
shared for reconstruction [6]. The randomness is added so that, in the end, when the output needs
to be reconstructed, it cancels out and equals 0, resulting in the real output not being affected. The
same procedure is used for multiplication in section 3.1.5.
The modulo part mod 2n of each arithmetic operation forms the ring, also denoted as Z2n, hence
the replicated-ring-party protocol. This is a necessary operation to ensure that numbers stay within
range and that the number wraps around in the event of an overflow [6] [23]. The relevant shares
are now distributed securely over a network to each party.
MP-SPDZ uses TCP (Transmission
Control Protocol) with TLS (Transport Layer Security) to encrypt the network packages, including
SSL certificates, to provide authenticity.
After distribution, each party should contain a subset of shares from the other parties. An example
of the distribution could be as follows:
Party0 = (S0,0, S1,0, S2,0)
Party1 = (S0,1, S1,1, S2,1)
Party2 = (S0,2, S1,2, S2,2)
Note that since this is replicated secret sharing, each party receives two secret shares from each
other, which allows the protocol to continue functioning with two of the three parties present (see
section 9.2.1 for a contextual example). However, in these calculations, the additional secret shares
are omitted.
Each party has partial information (secret shares) about each number Ni, but not enough to recon-
struct the original number without additional information from the other parties.
In the next step, each party locally computes the sum of the shares received.
Party0,sum = S0,0 + S1,0 + S2,0
Party1,sum = S0,1 + S1,1 + S2,1
Party2,sum = S0,2 + S1,2 + S2,2
Finally, the local sums are sent to all parties to be combined to reveal the final sum, hence the
output.

3.1.5
Multiplication
The addition operation can be expanded with multiplication using secret sharing; suppose Party0
and Party1 want to multiply their two private inputs N0 · N1; this can also be done for three inputs,
but the third is omitted to keep the example simple. First, the secret shares of the two numbers
must be chosen and distributed to the parties so that:
N0 = S0,0 + S0,1 + S0,2 mod 2n
N1 = S1,0 + S1,1 + S1,2 mod 2n
These shares are then distributed among the three parties over a network. This means that each
party gets two shares from each input. For multiplication, it follows that:
N0 · N1 = (S0,0 + S0,1 + S0,2) · (S1,0 + S1,1 + S1,2)
Each party receives two shares of each input (note that the specific combinations of shares received
depend on the specific protocol and implementation; this is one example):
Party0 : (S0,0, S0,1) and (S1,0, S1,1)
Party1 : (S0,1, S0,2) and (S1,1, S1,2)
Party2 : (S0,0, S0,2) and (S1,0, S1,2)
The next computations occur locally for each party, where they compute the product of their shares
for each input.
Party0 : (S0,0 · S1,0) + (S0,0 · S1,1) + (S0,1 · S1,0) = X
Party1 : (S0,1 · S1,1) + (S0,1 · S1,2) + (S0,2 · S1,1) = Y
Party2 : (S0,0 · S1,2) + (S0,2 · S1,0) + (S0,2 · S1,2) = Z
The intermediate results X, Y , and Z can be shared for reconstruction. Each party only receives
the results, not the underlying shares.
Now that all parties Party0, Party1, and Party2 have received the intermediate output, the re-
construction can be done. The output is now equal to the addition of all the shares: N0 · N1 =
X + Y + Z
Lastly, because addition and multiplication are complete operations, they form the basis for other
arithmetic types, such as exponentiation, subtraction, division, etc., for secret sharing computa-
tions. So, matrix multiplication and other more complex arithmetic operations used during neural
network training are derived from addition and multiplication. Still, the implementation in MP-
SDPZ can differ; these are theoretical examples showing the high-level core principles and under-
standing [9].
3.2
Feedforward neural network
A neural network is a computationally directed graph of nodes (neurons) and edges [32]. Each node
handles an input from the flowed edges, a weighted sum, applies an activation function, and passes
it to the next edge until it reaches the output layer. The nodes are organized in layers, meaning
nodes only receive inputs from the previous layers and only pass outputs to the next layer through
the edges.
A feedforward neural network (FNN) does not form loops, meaning it is one-directional, going from
input to output layers through hidden layers. As a standard, feedforward neural networks have
7
==PAGE:15==
fully connected layers, meaning all nodes in the current layer are connected with all nodes in the
next layer.
3.2.1
Batch methods
The first method is Batch Gradient Descent, which is when the batch size equals the size of the
dataset, so every sample (the entire dataset) is processed before the model parameters are updated.
Stochastic Gradient Descent is when only one sample is processed, which results in a batch size
of precisely 1. The last method is Mini-Batch Gradient Descent, which is somewhere in the middle
where the batch size BS is 1 < BS < N and N is the length of the dataset [10].
Batch methods dictate how many samples the network should process before updating its weights
after a forward and backward pass. Using a larger batch size is generally faster because weights
are less frequently updated, saving computation.
However, different values of the batch size
can yield various results in terms of how well the model will generalize and perform on unseen
data.

Binary cross entropy loss function
The loss function used in this project is binary cross entropy (BCE), commonly used in classification
tasks with a binary class and is also referred to as log loss. The binary class in this project is whether
or not a credit card transaction is classified as fraudulent or non-fraudulent - hence, binary (0 or 1).
However, the output of the machine learning model ranges from 0 to 1. This output is what the BCE
function uses. Still, in the end, it is classified based on a specific threshold, typically 0.5, so if the
model prediction is y ≤0.5, it is non-fraudulent, and y > 0.5, it is a fraudulent transaction.
The loss function measures how well the model’s prediction matches the specific sample’s truth
values (labels) [47] - for example, a model prediction of 0.99 for a sample where the correct label
is 0 would result in a high loss value. Since the loss function is of logarithmic nature, the loss will
logarithmically increase for predictions far from the sample’s truth value (label). A flawless model
would have a training loss and test loss equal to 0 [47].

Isolation forest
Isolation forest (IF) is an anomaly detection algorithm that isolates anomalies within a dataset
to identify outliers [30]. The algorithm builds decision trees by randomly selecting and splitting
values. Since anomalies are easier to isolate because they are few and far from the majority of data
points, they tend to have shorter paths in the isolation trees than normal data points. Ultimately, the
algorithm gives the data point an anomaly score, which describes the path length used to isolate
the data point. If the data point has a shorter path, it will have a higher likelihood of being an
anomaly.

Datasets
The choice of dataset has been crucial for this project as it determines the quality and realism
of the models used for predicting fraud. Dealing with fraud in the financial industry, especially
non-fraudulent and fraudulent transactions, involves sensitive, private, and confidential data. Laws
protect this kind of information and cannot be made public by banks and other relevant parties.
As a result, getting real and authentic datasets is not feasible as they are not publicly available for
good reasons, as this would compromise privacy and security for customers/individuals affected.
However, this section will discuss alternative options such as synthetic, transformed, and generated
data. Furthermore, it will thoroughly discuss the reasons for selecting a specific dataset for this
project and its significance.
4.1
Source and research
Before delving into why a specific dataset was chosen, it will be covered how the datasets were dis-
covered and their contents, source, and dataset type will be examined. The primary source used in
this project to collect datasets was through Kaggle.com, which is a popular site for finding datasets
for machine learning-related purposes. Other sources did not provide reliable datasets; they refer-
enced Kaggle.com or were found through Kaggle first. For example, this source [15], which is not
Kaggle, provided a dataset containing credit card transactions, both fraudulent and non-fraudulent,
but when determining the original source of the dataset, it links back to Kaggle.com.
Most datasets in this project are available from Kaggle, but their actual sources might be differ-

As the simulators are
open-source, this ultimately means that it is possible to generate more data if necessary. Other

is not feasible when the simulation is closed-source, as it is provided "as is" without access to the

Choice of dataset
Multiple datasets were examined and assessed during the project’s research phase, but only two
were analyzed in depth. The first dataset, Sparkov [44], was used initially. Later in the process,

are explained in more depth in this section.
The Sparkov dataset [44] from Kaggle (source code on GitHub [21]) was initially chosen for training
and testing the models designed and created in this project, as mentioned above.
This dataset consists of synthetically generated data. It was chosen because it met all the criteria:
it resembles actual data, the creation process is known, and it contains sufficient data points to

These criteria were chosen to facilitate the development of the

Another advantage of the Sparkov dataset is its ability to generate additional data. This means that
if a larger dataset or more unseen data is needed, it can be generated under the same criteria as
the original data generation process.
The Sparkov dataset includes user profiles designed to match specific spending patterns based on
age, gender, and location. Each profile contains various elements influencing credit card transac-

It also includes information on vacation days, daily transaction volume, and the most common days
for spending money, making it a more realistic dataset.
After examining the variables in the Sparkov dataset, it was evident that feature engineering was

based on existing variables, such as calculating the distance between the card being used and the

4.3.1
Transactional data
The transactional data file consists of the variables in the third column of table 3.
Many vari-
ables are standard information about credit card transactions, such as year, month, day, time, and
amount. The merchant information, merchant name, merchant city, and merchant state are also
part of the variables, which describe who the merchant is and where the transaction occurred.
Furthermore, the zip variable is part of the merchant information since it describes the zip code of
the merchant’s location.
In addition to the basic information, a user may have multiple cards linked to their IDs, known
as "User". This is why the card variable can also be considered an additional ID. The "Use Chip"
variable indicates the transaction method, which can be swipe, online, or chip.
The last three variables, "MCC", "Errors?" and "Is Fraud?", are not self-explanatory. "MCC" stands
for Merchant Category Code, which describes the category a specific merchant falls into, ranging
17
==PAGE:25==
from 0 to 9999 according to the IBM paper [3].
Many merchants, from multinational ones like
McDonald’s to local ones like cleaners and jewelers, have been assigned to these categories.
The second variable, "Errors?" indicates whether an error has occurred during the credit card
transaction. There are five possible errors: Bad PIN, Bad CVV, Bad Expiration, Bad Card Number,
Insufficient Balance, and Technical Glitch. If no error has occurred, the value will be "No" instead
of an error. Errors with "Bad" in front indicate that something is wrong, such as an incorrectly
entered PIN, CVV, card number, or an expired card. It’s important to note that multiple errors can
occur simultaneously, for example, "Bad PIN" and "Insufficient Balance," meaning that there is not
enough money on the card and the user entered the wrong PIN.
The last variable, "Is Fraud?" serves as the dataset’s label, indicating whether the given transaction
is fraudulent or not.
4.3.2
Card data
The card data file consists of the variables specified in the second column of table 3. This includes
standard information about credit cards for each user, such as their brand, type, card number, CVV
code, when it expires, when it was issued, and if there are any spending limits, etc. Most of these
variables are, as mentioned, self-explanatory. The variable "Has Chip" indicates if the credit card
is of a modern type.
In addition, users can have multiple cards from different brands linked to their IDs.
The ID is
assigned by the "User" variable in the dataset.
The variable "Card on Dark Web" is of boolean type, which dictates whether or not credit card
information has been leaked on the dark web. Briefly explained the dark web is the unindexed part
of the internet where both illicit and legit websites require special web browsers to be accessed
[50]. These are typical places for data to be leaked, bought, and shared.
4.3.3
Personal data
The personal data file contains personal information about each user, denoted by the "Person"
variable in the dataset seen in table 3.
This variable stores the name of the person as seen in
table 4, and the row enumeration functions as the user ID. For example, row 1 corresponds to user
ID 0 and row 2 to user ID 1, etc., which is relevant when merging the data with the other data
files. In addition, typical information about each person is given. This includes location details
such as address, apartment, city, state, and coordinates. Also, financial information, such as total
debt, income, FICO score, and number of credit cards. Lastly, personal information includes age,
retirement age, gender, birth date, etc. The FICO score, also known as a credit score, is a metric
used to determine financial stability. It is typically used to assess whether a person can borrow
money and how much. A higher FICO score indicates a better credit score and is associated with
more financial stability than one with a lower score.

5.1
Data overview
During the data overview, the first step was to check for any gaps in the dataset, specifically missing
values. It is important to investigate and handle null or missing values because they can impact the
training of various models.
In the IBM dataset, three variables had some of their values being null: "Merchant State," "Zip," and
"Errors?" However, after analyzing the data, it can be inferred that each null value in the dataset
has a meaning. Starting with "Merchant State", when a purchase is made online, the merchant’s
name is set to "Online", and both "Merchant State" and "Zip" have null values because no address
is associated with an online purchase. This specific type of null value appears 2.7 million times in
the dataset.
The "Zip" variable contains null values even when "Merchant State" is not null. This only occurs
when a purchase is made outside of the US. When this kind of purchase is made, the dataset
only includes the city for international transactions (e.g., Zürich) and considers the state as the
country (Switzerland); the country has no zip code. This null value appears 157314 times in the
dataset.
The final variable with null values is "Errors?". As explained earlier, the variable indicates the type
of error that occurred during the transaction. When null is present in this variable, it means that
no errors occurred. So, essentially, null is equivalent to "No" in this section of the dataset. This is
why the null value occurs almost 24 million times in the dataset.

5.1.1
Imbalanced dataset
When using the data to train machine learning models, such as a neural network, it’s essential to
review whether or not the target class consists of an uneven distribution of observations in the
dataset [33].
This is crucial because it can affect the accuracy and performance of the model,
potentially bias it towards the majority class, and generalize poorly to new unseen data. This is
especially true for the minority class.
It also gives an insight into which models potentially fit the task, such as the isolation forest al-
gorithm, as mentioned earlier. However, there exist techniques like oversampling and synthetic
minority over-sampling technique (SMOTE), to mention a few, that can address the dataset class
imbalance issue [33]. This can make models of other types normally failing at such tasks perform
exceptionally well. The distribution of the target classes can be seen in figure 1.
Figure 1: Shows the distribution of fraudulent and non-fraudulent transactions
Examining figure 1, it can be observed that there are about 24.3 million non-fraudulent transac-

Here, the "Number of FT in Interval" is the total number of fraudulent transactions for that specific

5.2
Choice of machine learning methods
Much information was gained about the IBM dataset from the EDA explained above. From this
analysis, it was decided to treat fraud detection as a binary classification problem since the label,
’Is Fraud?’, is binary. The investigation of which machine learning method would be most suitable
for the classification problem was conducted simultaneously with the EDA as more knowledge was
gained about the dataset. Here, four models were chosen, also described in Sarker’s book [42].
The models are listed below:
• Feedforward neural network (FNN)
• Logistic regression (LR)
• Decision tree classifier (DTC)
• Random forest classifier (RFC)
Further research on anomaly detection, the last method, isolation forest (IF), was discovered and
selected, which this paper describes [30]. These five methods were chosen to best fit the fraud
detection use case. Existing work has already been done on this dataset, where techniques such
as recurrent neural networks of (LSTM) [45] and language models (BERT) [4] have been used for
prediction.
Notably, not all five models will be used continuously throughout the project. Instead, they will
be tested to identify the initial best-performing models.
This approach will save time and help
focus on enhancing the performance of the most effective models rather than spending effort on
underperforming ones.
Feedforward neural network: A feedforward neural network (see section 3.2) is a model that was
in focus from the beginning and has become our primary approach regarding models. This interest
is because of the model’s universal function approximation and transfer learning capabilities, as
mentioned in section 5. By performing transfer learning, it would be possible to tailor a model to
each user segment to know the exact spending pattern of said group and predict from this instead
of only having a generalized model that can predict from the general patterns of the dataset. The
users have been divided into segments from percentile ranges, PL, PM, and PU, which is described
further in section 7.1.
Logistic regression: Logistic regression is a simpler method than the other four and was chosen
as a good baseline for comparison with the other models. More complex models can be compared
to a minimum standard by choosing the simplest model as a baseline. If the other models perform
worse, it can be deduced that something is wrong with them.
Decision tree classifier: The decision tree classifier (see section 3.4.1) was chosen because it
has more flexibility and capabilities for capturing non-linear patterns between features, compared
to logistic regression, which is highly likely to be present in the IBM dataset after conducting
exploratory data analysis (EDA). Furthermore, the decision tree classifier has high interpretability,
making it easier to understand and visualize how the model has reached its outcome, which is
the main reason for using this method. Understanding how the model reaches its decision can be
essential for further improving and debugging the model.
Random forest classifier: Random forest classifier is described in section 3.4.2 and was chosen
because it is an "extension" of the decision tree classifier, which means that it will be less prone
25
==PAGE:33==
to overfitting and, in many but not all cases, perform better than a singular decision tree classi-
fier.
Isolation forest: The IBM dataset is imbalanced, as mentioned in section 5.1.1, which is one of the
reasons for choosing the isolation forest (see section 3.4.3). The isolation forest algorithm is used
because of the anomalies in the dataset, where this method will help detect fraudulent transactions
effectively.

Feature Engineering
Feature engineering is the process of selecting, manipulating, and transforming raw data into fea-
tures usable in machine learning methods.
This section explains the techniques used to create
these features and why these processes are essential for achieving the best potential for the cho-
sen methods.
The feature engineering techniques used on the dataset, such as different types of encoding vari-
ables and scaling techniques, min-max and standard scaler, are well-studied and are inspired from
these sources [38] [40] [36], which explains them in depth.
As explained in section 5.2, multiple machine learning methods were chosen. However, a decision
has been made to apply the same feature engineering techniques to all models, resulting in a single
feature engineered dataset.
Each of the five models could perform better if they had features
engineered specifically for their use. However, by using the same techniques for all methods, a
foundation is established to compare the performance of the models in the best possible way.
It is crucial to note that each model has not been trained after each technique has been used. With
numerous techniques and five models, the possible combinations have increased exponentially. As
a result, testing all the combinations of techniques and methods used for feature engineering in

After thoroughly analyzing the dataset through exploratory data analysis, the next step is to evalu-

sary to predict fraudulent transactions accurately.

6.1
Data preprocessing
The data must be preprocessed several times before a machine learning model can be trained. This
includes handling missing values and categorical variables, sampling, and removing unnecessary
columns that are likely not useful for the model to train on.
A feedforward neural network cannot handle null values, and the models’ performance can worsen
and even fail to train on the dataset. In the IBM dataset, there were three variables with null values,
"Merchant State," "Zip," and "Errors?" which needed to be handled so as not to be negatively
affected performance-wise. There are multiple ways to handle missing values, such as removing
the row or column completely, replacing the missing value, or flagging it. It was decided to replace
26
==PAGE:34==
the missing value since removing it can lead to losing valuable information, as explained in section
5.1.
In the preprocessing stage, the non-numeric data must be transformed. This is because models like
the feedforward neural network can only handle numerical data. Most of the removals or changes
to non-numeric data will be done on the categorical variables because they have non-numeric data
in the IBM dataset.
Section 6.2 will explain it in more detail.
However, two instances of non-
numeric data are not part of the categorical variables. These are the "Amount" and "Is Fraud?"
variables. It was chosen not to remove these, as doing so would result in losing information.
The first variable is "Amount", and it is given in dollars ($). When converting this to a numeric
value, the $ sign can be removed, and then the variable can be converted to a float to handle
decimals.
The second variable, "Is Fraud?", is the target class and was first represented in string format as a
"Yes" or "No," also making this a binary classification problem. These values can then be mapped
to 0 when the transaction is non-fraudulent ("No") and 1 when it is fraudulent ("Yes"). This label
serves as a reference for the model’s prediction of the correct output.
6.2
Categorical variables
Categorical variables can provide valuable information and meaningful relationships among the
data, allowing models to learn patterns effectively. Regardless, for a model to learn from these
types of variables, the data needs to be transformed so that the model can either 1) differenti-
ate between these categories or 2) use a new feature created from categorical data. There are
multiple techniques for accomplishing this, such as one hot encoding, which is the most direct ap-
proach, frequency and target encoding, and dimensionality reduction, such as PCA, and numerous
others.
However, with one hot encoding, a typical drawback when there are many distinct categories is the
introduction of new variables (columns). This increases the dimensionality of the data, which can
negatively affect the model’s performance, and phenomena such as the "Curse of dimensionality"
can become a real issue. Neural networks tend to perform best when less complexity is introduced.
This is because as the number of features increases, the model becomes subject to overfitting and
poor generalization when there is insufficient training data. The amount of data grows exponen-
tially as the dimensionality of the dataset increases [8].
As a result, other methods have been used, such as frequency encoding, creating new features from
the categorical data, and reducing the number of categories. This relates to the fact that numerous
variables in the dataset contain an excessive number of different categories. For instance, there
are 10, 000 shopping categories, denoted by the nominal column "MCC" (see section 4.3). Using one
hot encoding would create 10,0000 new features (columns), which is impractical. Here, frequency
encoding has been used instead, which will be explained in section 6.2.2.

most effective when the data have a high cardinality and diversity across the categories, meaning
that in an ideal scenario, the frequency of each category differs.

it challenging for the machine learning model to tell them apart. This is not always disadvanta-
geous since it depends on how the data is clustered around the target variable. Figure 8 illustrates

6.3
Feature extraction from time series
The dataset provides time series data, such as the year, month, day, and the specific time (time col-
umn) a transaction occurred. Based on the exploratory data analysis in section 5.1.2, it is extrapo-
lated that time plays an essential role in identifying fraudulent activity and transactions. Therefore,
time data will be part of the final dataset, but first, it must be determined which type of time data
is essential and which is not.
Deciding whether to drop or keep the year variable can be challenging. The concern with including
the year is that the model learns specific patterns and associates those with that given year. Sup-
pose a neural network makes predictions for future transactions in another year. In that case, its
performance may become affected due to potential overfitting and bias towards the given years it
was trained on.
However, a potential issue with dropping the year is that the transaction count for a particular
date, month, and day without the year associated becomes misleading. For instance, if there are
transactions for the same month and day but different years, the model might learn that one partic-
ular date has a high number of transactions or other patterns. But this is because the data is from
two different years. So, it is crucial to differentiate between years, which is why all time-related
information is kept.
In addition, the time column is converted to the following format: the decimal part represents the
minutes, and the whole number is the hours. Furthermore, the day of the week can be calculated
from the entire date. The day of the week can give valuable insights into which weekday has an
31
==PAGE:39==
increased risk of fraudulent activity.
Finally, the time between each transaction per credit card can be calculated. Users may develop
specific spending patterns and become accustomed to certain time gaps between purchases. For
example, users may consistently buy something in the morning and later in the evening. An ir-
regular time interval between transactions can signify fraudulent activity. A new feature called
"TimeBtwTransPerCard" has been created to capture this. Notably, it is deliberate that the feature
is based on the time interval per credit card and not between each user transaction. This is because
a user might use multiple credit cards for different purposes; one could be work-related and an-
other for grocery shopping. This approach treats each card as a distinct user, potentially allowing
the model to learn more subtle patterns.

6.4
General feature extraction from existing variables
When attempting to extract features from existing variables, the general thought process has been
to explore whether more information can be derived from them.
By adding new features, this
additional information aims to provide the models with a means to learn more sophisticated pat-
terns, and they become more apparent, potentially predicting fraudulent transactions more accu-
rately.
The first feature, the average expenditure per merchant, can be calculated using a combination of
the "Amount" and "Merchant Name" variables. This feature was further explored to assess whether
it could help detect fraudulent activity in the final dataset.
The average expenditure per merchant describes the average amount of money spent at each mer-
chant in the entire dataset. Each transaction in the dataset has been grouped according to the
merchant, where the mean is computed. By adding this new feature, it can be determined how
much money the merchant expects each customer to spend in their establishment.
The feature "AvgExpenditurePerMerchant" includes all fraudulent and non-fraudulent transactions
without any distinction between the two. Making this distinction could lead to a risk of data leakage.
Data leakage would occur if the model used information from the target value (the label) to make
predictions. An extreme example of this would be training the model on the actual label rather
than a feature inferred by the label, such as the average expenditure per merchant for fraudulent
transactions.
Another combination of variables that were explored was the "Card" and "User" variables. It was
observed that a user can have multiple cards, as explained in section 4.3.2. It was therefore ex-
plored if a pattern could be seen between the probability of fraudulent transactions and the number
32
==PAGE:40==
of credit cards a user possesses. A barplot was created to illustrate the probability of a user with a
specific number of cards experiencing a fraudulent transaction (see figure 11).
Figure 11: The probability of fraud by how many cards a user has.
In figure 11, the barplot shows a clear pattern: as the number of cards a user has increases, so
does the risk of exposure to fraudulent activity. One potential reason for this pattern is that having
more credit cards increases the likelihood of losing one or having the card information stolen, as it
becomes more challenging to keep track of multiple cards compared to just one.
The last feature added, "TransOrderPerCard," combines "User" and "Card" to add transaction or-
ders for each card a user has and to identify patterns related to fraudulent transactions. Each card
behaves as a separate user where the transactions are ordered after when they have occurred. So,
the first transaction being number 1 and the second number 2, etc. It was found that the average
transaction number where fraudulent activity is most likely to happen is after 3316.26 transactions.
This suggests that cards’ likelihood of encountering fraudulent activity is very low during their first
transactions. To provide a more accurate representation, the median transaction number was also
calculated for fraudulent activities, which is 2076. This still indicates that the risk of encountering
fraud increases with the number of transactions conducted by a card.

Data Scaling
Data scaling is in machine learning, a term for transforming all the values of the features in the
dataset to be within the same range [40]. The reason for performing this technique is to ensure
fair feature contribution.
For instance, the feature "Card" in the dataset has low values, while
another feature, "Amount", can have much higher values, disproportionally influencing the model,
especially the feedforward neural network, training and leading to biased predictions.
By one
feature having that much larger value, the model can think that the specific feature weighs more
35
==PAGE:43==
than the one with lower values.
Two prevalent scaling techniques, standard and min-max scaling, were considered to address this
issue. The standard scaler is most optimal on normally distributed data (see section 3.3.1 for more
information). Therefore, the preferred option is the min-max scaling technique (see section 3.3.2.
This is because when examining the data in section 5, most of the data was not normally distributed,
for instance, the yearly transaction volume, which is pictured in figure 4. However, not all aspects
of the data have been analyzed regarding the distribution, and an assumption has been made that
most of the data is not normally distributed.
Even though min-max scaling is likely to be the best scaling technique for this dataset, as described
above and based on the assumptions made, both methods will be tested (see section 8). Testing
both scaling techniques will determine if the assumption was correct and which method will be
best for fraud detection.

Data Sampling
The dataset should be sampled in a way that is as fair as possible, and the sampling should be
representative of the whole dataset in an even distributed fashion, both for training and test sets
36
==PAGE:44==
used for the models created in this project. This is to avoid the pitfalls of only randomly sampling
for a specific period, for example, from 2000 to 2010, and possibly missing out on trends and
patterns of the remaining period from 2010 to 2020.
In addition, fraudulent transactions only
account for 0.12% of all transactions, which is extremely low, so when sampling, it is desired to
include all fraudulent transactions for the whole dataset and then downsample the non-fraudulent
transactions randomly to match a number that is realistic for both training and testing in terms of
computational cost, training time and performance.
It may be argued that this gives a false evaluation of the models created and trained for future
predictions because our test and training set are from shared mixed periods.
However, when
evaluating the models, it is desired to see if they have learned the patterns from the given periods
they are trained on. This means that both the training and test data need to have data from the
same periods.
Another disadvantage is that if the models are evaluated on a test set from a future year, there is a
possibility that the behavioral patterns of fraudsters and their methods have completely changed.
This could result in the models performing poorly because they cannot recognize these new pat-
terns. As a result, it may seem like the model has learned little to nothing, even though it might
have.
Apart from those mentioned above, a generalized model is to be preferred. By having data from all
the different periods of the dataset, it will follow the principle of generalization. By the definition
of generalization, the data used to test the model is drawn from the same distribution as the one
used to create the model [51], which is the methodology applied in this project.
Furthermore, a more generalized model, such as a neural network, will make it a better candidate
for transfer learning. Transfer learning will be constructed to tailor the model for a specific group
of people and their spending patterns. In theory, this will make the model more individualized and
more effective in identifying fraudulent behavior, as it would have a more targeted and precise
understanding of the users’ spending patterns. Not all types of models are applicable for transfer
learning, such as decision tree classifiers or isolation forests, which can be a disadvantage of these
types of models.
The evaluation section (section 8) of the models presented in this project explains these other
models more in-depth.

7.2
Training and test set
To effectively train and evaluate machine learning models, it is essential to have both a training
and a test set. When examining figure 14, the original dataset has 24 million rows without any
row deletions, data preprocessing, or feature engineering. Training and validating a feedforward
neural network with this many rows and multiple configurations would be very time-consuming. To
address this, downsampling (or undersampling) will be used to reduce the amount of data used for
training and testing. In this case, downsampling involves randomly selecting a set number of non-
fraudulent transactions from all the non-fraudulent data samples. It was decided to downsample to
500,000 before splitting the dataset, as depicted in figure 14. As mentioned in section 7.1, some
users have been removed from the dataset. This explains the decrease in fraudulent transactions
after downsampling from the initial dataset, as shown in figure 14.
Downsampling not only reduces the computational power and time required but also serves an
important purpose: to evaluate the data quality based on the model’s performance.
Generally,
a machine learning model benefits from a larger amount of input data as it aids in recognizing
patterns across the entire dataset and enhances generalization. Therefore, if the machine learning
model yields favorable results after downsampling (using less data), it indicates good data quality
and a successful feature engineering process.
39
==PAGE:47==
Figure 14: The steps from the initial dataset to the final test and training sets.
After downsampling, the dataset is divided into training and test sets. The training set contains 80%
of the data, while the test set contains the remaining 20%. The new dataset is randomly selected
to maintain the same ratio of fraudulent to non-fraudulent transactions. Keeping this ratio intact is
crucial for validating the model, as a skewed ratio in the test set could lead to inaccurate validation
results, with a worst-case scenario of only non-fraudulent transactions in the test set.
By downsampling to 500,000 samples, the training set will consist of 400,000 data samples for
both fraudulent and non-fraudulent transactions after applying upsampling (duplicating instances
of the minority class). This approach makes training more manageable within the hardware con-
straints.
If the dataset had not been downsampled and thus maintained the initial number of 24 million
non-fraudulent transactions and 29757 fraudulent transactions, the training set would include 19.2
million non-fraudulent transactions and 23805 fraudulent transactions when split in an 80-20 ratio.
If the fraudulent transactions were then upsampled, the result would be 19.2 million transactions
for both categories, leading to a training set of 38.4 million rows. This would exceed the size of the
original dataset, making model training non-feasible due to the high computational power and time
required.
Moving forward, all models have been trained both with and without upsampling. The hypothesis
was that upsampling would yield the best results, as elaborated in section 8.
This assumption
was based on the idea that upsampling would improve the models’ ability to detect fraudulent
40
==PAGE:48==
transactions by balancing the distribution of the two categories, providing more instances of the
minority class (fraudulent transactions) during training.
The reason for employing upsampling is due to the imbalance in the dataset.
In cases where
the dataset is skewed towards one class (non-fraudulent transactions), training and testing the
model using all available fraudulent transactions could lead to a model biased towards the majority
class. This bias may cause the model to not learn from the minority class (fraudulent transactions),
resulting in poor predictive performance.
In fraud detection, it is crucial to identify instances
from this minority class, and having a small sample of them could significantly reduce the model’s
effectiveness.
In general, classification models should avoid bias towards the majority class, as this would nega-
tively impact the overall accuracy and performance. Upsampling can potentially mitigate this issue
by adjusting the dataset to achieve a more balanced ratio, which in this case is a 1:1 ratio between
fraudulent and non-fraudulent transactions. This ensures that the model learns from both classes
and is not biased toward the majority class.

8.1
Model training
The dataset has undergone various processes, such as exploratory data analysis, data preprocess-
ing, and feature engineering, which means it is now ready for training. The data will be fitted to
a logistic regression as a baseline for evaluating the complexity of the problem. The results are
expected to perform worse based on the assessed metrics than the other models because the clas-

will be fitted to a random forest classifier, decision tree classifier, and isolation forest. These models
do not have as many parameters to tweak as a neural network, as their capabilities are considered
more fixed. The test results of the models can be seen in tables 8, 9, 10, and 11, and are evaluated

8.1.1
Feedforward neural network
The neural network’s architecture is kept simplistic, reducing training time and computational
cost. As shown in this paper [32], the universal approximation theorem states that any continuous
function can be approximated given one extra hidden layer with a non-linear activation function
such as Rectified Linear Unit (ReLU). This means that, in most cases, having many hidden layers
in the model is unnecessary. So, as a starting point, choosing a few layers is the best approach,
as more layers can always be added when needed. This is highly dependent on the nature of the
data; having more layers or nodes in one layer can be beneficial, especially if the data have out-
liers, which can be the case with fraudulent transactions. Outliers can require a deeper connected
network, potentially allowing the model to capture more intricate patterns and better distinguish
anomalies.
The model architecture used is shown below in figure 15. The input layer consists of 14 nodes, as
there are 14 features. This is followed by three hidden layers, with 15, 7, and 3 nodes, respectively,
using the ReLU activation function. Lastly, the output layer consists of one node using the sigmoid
activation function to give a probability ranging from 0 to 1, explaining the likelihood a transaction
is fraudulent.

Metrics
However, all models are still compared and evaluated, and the best ones are selected for final

• F1
• F2
• PR-AUC
• Recall
• Precision
• ACC 0
• ACC 1
These metrics evaluate the model’s performance and detection reliability. Each metric measures a
model’s shortcomings and strengths in different aspects. The most widely used and standard metric
is overall accuracy (see section 3.5.1), which accounts for all classes. However, it is also the most
misleading one in our case. The problem arises with an imbalanced dataset where a large portion
of the dataset constitutes the majority class, giving a skewed distribution. In our case, the non-
fraudulent class makes up for 94% after downsampling, and in the original dataset, it is over 99%.
This means that if the model detects none of the fraudulent samples and all of the non-fraudulent
samples, the accuracy will still be 99%.
A model that can only correctly classify one of the classes in a binary classification problem is use-
less. This is why the metric must be "split up" to measure the accuracy for each class independently,
giving a more accurate benchmarking measurement. This can be observed in table 7 where two
columns, "ACC 0" and "ACC 1," denote the accuracy for non-fraudulent transactions (the number
0 for negative) and fraudulent transactions (the number 1 for positive), respectively. The formulas
for "ACC 0" and "ACC 1" are the same as the accuracy, except it is class-specific.
The standard metrics such as precision and recall have been used (see section 3.5.2. These metrics
are essential to monitor and evaluate as they assess the model performance in isolated areas and
play a crucial part in other metrics, which will be introduced shortly. This is because these metrics
combine precision and recall into one overall metric.
45
==PAGE:53==
The chosen metrics are F1 and F2, commonly used in machine learning when dealing with imbal-
anced datasets. This makes it easier to compare with other models from related research, and
the metric itself is valuable for measuring the model’s performance; see section 3.5.3 for more
information.
The last metric used is the "Precision Recall-Area Under Curve," also known as PR-AUC (visit sec-
tion 3.5.4 for more information). This metric gives the most balanced view of the model’s perfor-
mance by considering precision-recall under various thresholds [13].
The F2 and PR-AUC metrics differ because PR-AUC offers a more comprehensive assessment of the
balance between precision and recall at various thresholds, meaning the balance may vary across
different models. In contrast, the F2 metric maintains a fixed balance between recall and precision
due to its set formula. This means recall will always be more important than precision, as indicated
in equation 9.
The F2 metric was chosen to select the initial models that would be tweaked and used in a mul-
tiparty computation environment. This decision was based on the significance of recall in fraud
detection. From a business perspective, false negatives are more costly than false positives, which
is explained further in section 8.5.
It was later determined that the PR-AUC should be used instead for selecting models because it
offers a more detailed view of the model’s performance (see section 8.2.3). This is because the
trade-off between precision and recall, as explained earlier, is an important factor. Using the PR-
AUC, the model is evaluated on various thresholds, as mentioned, giving a more accurate estimate
of the model’s performance. So, essentially, when comparing to F2, PR-AUC would test a range
of different F1 values on various thresholds, providing more detailed information about the actual
metric score of the model (see section 3.5.4).

8.2.3
Selection process
Initially, the metric used for the selection process of models for further feature engineering was
the F2 metric.
The reason for using the F2 metric for the selection process is because recall
gets weighted higher than the precision of the model, as seen in equation 9. This is an important
quality when it is desired to detect all instances of fraud, even at the expense of precision (false
positives).
After evaluating the F2 metric, the upsampled configuration, "FE_UP_MM," for the feedforward
neural network, was selected for further feature engineering. However, it was decided to do this
for the configuration without upsampling, "FE_MM." It was done because the sum of recall and
precision metrics is better without upsampling than with, as seen in table 7. When upsampling, a
trade-off between recall and precision happens, meaning the recall will be higher when upsampling,
but the precision will be lower.
Using the sum of the two metrics will give an overall sense of
how balanced the model is.
In this case, upsampling improves recall but worsens precision so
much that the overall balance is better without upsampling.
This means that the configuration
without upsampling, "FE_MM," is more likely to have better generalization due to its better balance
between recall and precision.
However, after testing the feedforward neural network model without upsampling but with further
feature engineering (see section 8.3.8), the model showed a significantly increased precision and
acceptable high recall compared to the model with upsampling and extra feature engineering. In
this case, the model without upsampling still has a better balance (sum of recall and precision).
This led to a reevaluation of the metric used for the selection process, resulting in a decision
to switch to PR-AUC because it better reflects the balance between precision and recall.
The
PR-AUC metric provides an effective way to evaluate the overall performance of the models on
imbalanced data, as it focuses on recall and precision rather than overall accuracy. Additionally, PR-
AUC helps highlight how well the model identifies positive instances (fraud), especially in datasets
with few positive instances, such as the IBM dataset. Furthermore, PR-AUC is commonly used for
imbalanced datasets where the positive class (fraud) is small. However, whether it is better to use
PR-AUC as the selection metric depends on the business model and use case, which will be further
discussed in section 10. This scenario illustrates the value of evaluating multiple metrics, as it can
provide a more accurate understanding of the model’s benchmarks.

8.3.1
Refined/extra feature engineering
There are many methods and approaches to a classification problem, and finding the best configu-
ration, solution, and hyperparameter tuning for each approach can quickly become time-consuming
and complex. The first feature engineering phase (as described in section 6) is intended to give the
chosen machine learning methods the best potential for learning and finding meaningful patterns
and features from the dataset, improving the dataset’s quality. The wide range of selected mod-
els, as explained in section 5.2, was to identify the initial best models and use the best-performing
ones for further improvement, narrowing down the choices of how to approach the classification
problem. Feature engineering phase two, the final one, was performed on the dataset to improve
the two selected models and focus on them. The chosen ones are the feedforward neural network
and decision tree classifier models with the "FE_MM" configuration. Note that the final feature
engineering phase is called "extra feature engineering".
To further improve the existing models and the quality of the dataset on which they are trained,
it was decided to examine the IBM datasets that did not include transactions, meaning the user
dataset (sd254_user.csv) and cards dataset (sd254_cards.csv).
8.3.2
Card dataset
No immediate valuable features could be extracted from the card dataset, except for the number
of cards a user possesses, as this can be associated with a higher risk of losing a card. However,
this feature was already calculated and derived from the dataset containing transactions, which
means it is a redundant feature from the card dataset. The other features from the dataset, such as
card issue date, expiration date, card brand, etc., do not provide any direct information related to
fraudulent behavior. Therefore, those features were not included in the extra feature engineering
process.
One feature that initially looked promising was the "Card on Dark Web" (true or false). However,
upon further analysis, it was determined that none of the cards in the dataset had been leaked to the
Dark Web, making this column unanimous with all values being false. As a result, this feature did
not contribute any new information regarding fraudulent activity and was ultimately disregarded
for being included in the final process. Consequently, the sd254_cards.csv was not used.

8.3.3
User dataset
On the contrary, multiple features can be derived from the user dataset (sd254_user.csv), and many
of the variables from the user dataset could be extracted as features directly without any further
data manipulation. These include the user’s retirement age, yearly income, FICO score, total debt,
and gender. All these variables can be associated with the user being more exposed to fraudulent
activity.
Retired individuals can have different spending patterns than non-retired.
Total debt,
FICO score, and yearly income can be related to where the user lives and their spending patterns,
including where they shop and how often they travel. These associations can increase the risk of
fradulent activity if the individual gets their credit card stolen physically or by using a malicious
website. So, these variables are added directly to the final dataset as observed in table 12.
However, some variables, such as the user’s name and apartment, are not directly associated with

8.3.4
User’s age
The user’s current age can be used to calculate their age at each transaction they have engaged
in, as it is known when a transaction has occurred. Here, the timestamp of the transaction can be
subtracted from the current age. This feature can give more context to the transaction by relating
to a specific age.
8.3.5
Income per capita ratio
The "per capita income" by zip code variable can be combined with the yearly income of the user
to calculate the "Income to Per Capita Ratio" called "IPCR", using this simple formula:
IPCR =
Yearly Income
max(Per Capita Income, 1)
Similar to the FICO score, this gives an insight into the individual’s financial situation by measuring
how well the person is doing financially compared with the income standard in that specific state/zip
code, as these can vary a lot. A similar technique is used in this project [27]. The max(Per Capita
Income, 1) is to ensure that the denominator is never zero. A high IPCR value means the individual
is doing well financially in terms of income. Debt and other factors could paint a different picture
of the overall financial well-being of the individual. A decision was made to keep both the "Yearly
Income" and the "IPCR" features, even though "IPCR" incorporates the yearly income in the formula
because the "yearly income" can still be relevant in terms of the user’s spending patterns that are
not related to the zip code or living area, such as online shopping or shopping in other regions and
countries.
8.3.6
Distance from home
The "DistanceFromHome" feature calculates the distance between the user’s home address and
shopping location using the merchant’s zip code and the user’s longitude and latitude of their
52
==PAGE:60==
home address. The dataset containing all the transactions only provides each merchant’s zip code,
so the merchant’s longitude and latitude must be derived. Using a "geocode" library, any US zip
code can be converted to latitude and longitude. The caveat is that it does present issues regarding
the accuracy of the location, as it is not the precise geolocation of the merchant but the center
coordinates of each zip code.
In addition, there are special conditions, such as when the purchase is made online or overseas.
Here, the distance from home is set to 0 in case of an online purchase. If the transaction happened
in another country, the longitude and latitude of the country’s center are used. This is not optimal
as it is inaccurate, but it still provides the necessary information to establish that the transaction oc-
curred in another country because of the long distance. Transactions occurring in another country
are subject to fraudulent activity because it could indicate that the user’s credit card information
has been stolen.

8.4
Transfer Learning
The final and primary pre-trained model has been used for transfer learning. Here, the model con-
tinued training on data from specific user groups. The groups are drawn from different percentile
ranges based on the distribution of fraudulent activity per user. These percentile ranges are de-
fined as PL (1% - 25%), PM (median) and PU (75% - 99%) (revisit section 7.1 for more information).
Transfer learning has been applied to the model by freezing the weights of the existing layers so
they cannot be trained or adjusted further. Then, new fully connected layers are added, which are
trainable, so the model can adjust and become more customed and tailored towards the data from
the specific user groups on which it’s trained. This way, it can learn new patterns and specific
features unique to these users, potentially allowing the model to perform even better than a more
generalized model that might only have learned the prevalent fraud patterns.

8.5
Performance & Accuracy
One of the main goals is to achieve the highest possible accuracy for both classes, which are mea-
sured separately (metric ACC 1 and ACC 0), fraudulent and non-fraudulent transactions, combined
with the highest PR-AUC score. This would represent the ideal outcome for the machine learning
models trained and created in this project. However, in a real-world scenario, another objective can
take precedence. In this case, the most crucial goal could be for the machine learning model to ac-
curately identify fraudulent transactions when they occur, even if it allows for some false positives
in non-fraudulent transactions.
This reflects how real-life machine learning models are designed to flag suspicious transactions for
further human review. Therefore, it is more acceptable to have false positives in non-fraudulent
transactions because the individual can investigate them.
In summary, the focus is to avoid scenarios where fraudulent transactions are not flagged correctly,
even if it means occasionally flagging a non-fraudulent transaction. However, the acceptable level
of false positives can vary based on the specific use case. In some scenarios, it might be preferable
to have a model that reduces the number of false positives, even if it leads to a lower detection
rate of true positives (fraudulent transactions).
Too many false positives can make the manual
review process too long or result in unsatisfied customers with blocked or flagged credit cards.
Ideally, there has to be a fine balance between the precision and recall of the model.
Yet, it is
difficult to quantify when a model has reached a level that is satisfactory, as it is highly dependent
on the specific business case, customer needs, KYC (Know Your Customer) process, and company
requirements.
9
MP-SPDZ (Multiparty computation)
Multiparty computation allows multiple parties to jointly compute a function without revealing the
inputs to a mathematical function, allowing parties to see only the output if they reveal it. However,
in some cases, the output of a function must also be private as this can be a temporary result, which
is inputted into another function. MPC also allows for this type of behavior; see section 3.1 for
more information. As discussed in section 3.1, the theoretical understanding of MP-SPDZ has been
presented in a broader sense rather than delving into all the intricate details, as it has served as a
tool throughout this project.
9.1
Use case
The approach used in this project with MPC is to simulate three individual banks, each with its
own training and testing dataset containing credit card transactions and user meta information.
The data must be joined securely so that each bank’s data (inputs) are not revealed to other
banks.
The advantage of MPC is that it allows banks to combine their data and train a model with as much
data as possible, potentially resulting in a more sophisticated model that captures patterns, trends,
57
==PAGE:65==
and features from transactions across different bank accounts. This allows the model to learn the
most recently discovered fraudulent activity the banks have available.
The concatenated dataset will be used as input to train a feedforward neural network model and a
decision tree classifier securely using a semi-honest and malicious protocol in the MP-SPDZ frame-
work. After training, the model will be used for inference to test the model’s performance using
the same metrics described in section 8.2.1. Furthermore, to make the comparison as fair as pos-
sible, the same machine learning architecture, configuration, and parameters as the final models
selected in section 8.3 are re-created and defined in MP-SPDZ to evaluate the framework and pro-
tocol’s impact on the model’s performance.
Besides a performance evaluation, the models’ training time will also be compared. MPC relies
heavily on exchanging information over a network combined with cryptographic operations, which
introduces a significant overhead that can slow down the model’s training phase. This can affect
the framework’s usability in real-world applications in the financial industry.
9.2
MP-SPDZ
The MP-SPDZ framework provides various protocols for secure computation in multiple applica-
tions. The project focuses on machine learning, specifically a feedforward neural network and a
decision tree classifier for fraud detection. These are, as mentioned, the final chosen models. This
means the protocols must adhere to the requirements of running such a model, as the protocol must
be able to support training and inference of the model using the security provided by the protocol.
This is also called secure training and inference. Not all protocols are compatible or feasible for
accomplishing this. This will be further elaborated in detail later in this section, as the first step is
preparing the final dataset for MP-SPDZ before a protocol is chosen.
9.2.1
Data preperation
To simulate the use case with three individual banks, as mentioned in section 9.1, where the dataset
from each bank is combined into one extensive dataset for training, the same dataset used to train
the final models in section 8.3 is applied here and split into three parts. This is also necessary to
make the most accurate comparison between the models trained using MPC and those without, as
they are trained on the same data. The data split is carried out as depicted in figure 18 where
the dataset is randomly split into three parts containing 1/3 of the training and test dataset. At
this stage, each part is created in plaintext as Input-P0-0, Input-P1-0, and Input-P2-0 files in the
Player-Data folder so that MP-SPDZ can recognize them as separate parties when executing one
of the protocols; in this case, banks A, B, and C. This is because the protocols are executed on the
same machine, and a network is simulated; however, each bank would have the file stored locally
on its machine in a real scenario.
58
==PAGE:66==
Figure 18: High-level overview of how the dataset is loaded into the MP-SPDZ environment.
After that, the files from each bank must be shared with the other parties and merged into one
dataset without revealing the actual data. To accomplish this, a concept known as secret sharing is
used (see section 3.1.3). This project will use the replicated-ring-party protocol, which uses repli-
cated secret sharing for secure computation and ensuring the privacy of the inputs. The reasoning
behind choosing this protocol is explained in section 9.2.4.
The secret shares for each data point in the three datasets are computed locally for each bank: A,
B, and C. This results in three shares for each data point, denoting the shares as S = (s0, s1, s2)
where S is the input and S1, S2 and S3 the secret shares in the Z2n domain (MP-SPDZ uses n = 64
when it is running on a 64-bit processor, which in this case it is). The secrets are calculated so that
all shares must be present to reconstruct the input, but not all parties, so that S = S1 + S2 + S3
mod 2n [6] (see section 3.1 for information). The shares are generated when the features of the
plaintext data are loaded into a sfix tensor and the labels in the sint tensor datatype in MP-SPDZ.
This is relevant because the sfix data type converts the 64-bit floating point numbers to fix-point
representation using a precision of 31 bits, where 16 bits are used after the decimal point. This
does not necessarily mean a loss of precision for all numbers, as some can have fewer decimal
places than others. Still, there can be a discrepancy between the original plaintext number and the
number used for calculations in MP-SPDZ, as fewer bits represent the numbers. This can influence
the final result of the model training as round-off errors can accumulate. The labels are binary,
either a 0 or a 1, so these are converted with no loss.
Once each bank has independently created the secret shares for each data point in the dataset,
the shares are distributed to the relevant party over a network. This means that from bank A’s
59
==PAGE:67==
perspective, bank B (P1) receives the secret share (S1, S2) and bank C (P2) the share (S2, S0), while
bank A (P0) performs the calculation and keeps (S0, S1). This process is done for each bank. The
distribution is done through a simulated local area network (LAN) using the TCP (Transmission
Control Protocol) protocol with TLS (Transport layer security), which provides confidentiality. Note
that the protocol is executed on the same machine. Each party ensures authenticity through an
SSL certificate where the public key is shared among peers to provide authentication.
Now, the banks have their respective secret shares for the datasets, which essentially is a concate-
nation of all three datasets into one, but in a partial state, as each party does not hold all the secret
shares. The concatenation is seen in figure 18. Most importantly, the dataset is no longer plaintext
data (original input) but represented by secret shares, preserving privacy and securing the original
input. The next step involves defining the neural network’s architecture and training on the tensors
containing secret shares and labels.
9.2.2
Neural network
As explained previously, the same architecture is used as the final model (without extra layers).
MP-SPDZ does support the Keras library from the TensorFlow framework, but the training resulted
in poor performance due to a subtle issue. The problem is that the loss function that MP-SPDZ
utilizes for neural networks is MSE (Mean Square Error), which is perfect for regression problems
(predicting continuous values, for example, the temperature as a function x).
However, in this
case, credit card transactions are classification tasks requiring a cross-entropy loss function or
something similar.
After discovering this, the architecture and settings of the final model are replicated in Pytorch
instead, as MP-SPDZ also supports this framework.
Here, it was confirmed that binary cross-
entropy is supported in the MP-SPDZ implementation of PyTorch.
Now, the replicated-ring-party protocol is ready to be executed to start the secure training and
testing. After training, the model is prepared for inference; here, the model’s prediction (output)
on a specific sample is revealed to all parties to evaluate the model’s performance with the specified
metrics. A high-level overview of the training process is shown in figure 19 and how the output is
revealed, emphasizing that this is a very simplified overview.

==PAGE:68==
Figure 19: A high-level overview of how training of the neural network and inference is done in

Each bank runs its part of the neural network with its secret shares and exchanges newly computed
secret shares during training when necessary; for example, during a forward pass, the output
from one layer becomes the input to another layer, where the output must be re-shared among
the parties. This ensures that the necessary secret shares are propagated through each neural
network layer. The primary source of the overhead in MPC arises from the network packages being
sent between parties whenever new shares are distributed. This exchange is essential due to the
involvement of multiple parties in the protocol. If there are delays introduced in the network, the
training time could increase significantly, as millions of rounds of communication occur during the
training process.

Decision tree classifiers
MP-SPDZ supports decision tree classifiers; however, it was only possible to use around 30% of

the implementation of the decision tree classifier in MP-SPDZ involving recursion, as the excessive

However,

it was decided not to include the decision tree classifier in MP-SPDZ, as the comparison with the
final model would be too unreliable with only using 30% of the original dataset.
9.2.4
Protocol selection
The MP-SPDZ framework supports various protocols tailored for different use cases. These proto-
cols differ in terms of the type of adversaries they protect against, their computational and commu-
nicational overhead, and their compatibility with binary circuits. Among the protocols supported in
MP-SPDZ, the replicated-ring-party protocol was selected primarily due to its shorter training time
compared to others. An initial training was completed for a single epoch, taking approximately 686
seconds. This suggests that training for 100 epochs would require approximately 19 hours; see
section 9.3. All the training times in table 16 are approximated since they were initially run on one
epoch.
The protocols listed in table 16 were evaluated for the specific use case. A description of its training
process represents each tested protocol, although many were found incompatible or took too long
to complete.
One reason for their incompatibility is that they are designed for a maximum of
two participants. In a real-world scenario, there is no limit to the number of banks collaborating
to combine their data and improve their fraud detection systems.
Because there is no limit on
participants, it was desired to have a system that could handle more than two parties since more
data could provide a model that captures patterns, trends, and features across different banks and
learn the most recent known fraudulent techniques.
For this reason, it was decided to simulate a scenario with three banks and to use a protocol
specifically suited for this situation, as it aligns more closely with realistic conditions while still
allowing for a manageable training time. The selected protocol, known as the replicated-ring-party,
requires exactly three parties.
Alternatively, the rep4-ring-party protocol could be used, which
accommodates four participants. However, opting for this protocol would lead to a longer training
time since it involves four parties instead of three, as indicated in table 16.
It was chosen not
to consider protocols involving more than four participants due to the extended training times
associated with them.

machine learning model’s performance would improve marginally beyond this point. To compare,
the feedforward neural network in the normal environment would take ∼6 minutes to train 100
epochs.
At the end of each training, the number of rounds and the total data sent are logged for party
0, which resulted in 587.541.237 rounds and 748.589 MB sent in the semi-honest version of the
replicated-ring-party protocol.
Here, it is assumed that the rounds are the number of network
packages sent to the other parties. The network delay is negligible as it is run locally over LAN
on the same machine. However, introducing an artificial delay of 10 ms would have an enormous
impact. This would result in an additional training time of ∼68 days. A network delay would make
it a more realistic setup because the banks would not have the model and all data stored in the
same data center or servers, a significant increase from 19 hours to 68 days. Note that this is only
for party 0.
The malicious security model of the replicated-ring-party protocol was trained next. This initial
training showed that a single batch out of 3309 would take approximately 4.25 minutes to run. It
would, therefore, take approximately 9.7 days to complete one epoch. Given that it needs to be
trained for 100 epochs used, the malicious-ring-party protocol would take approximately 2.6 years
to complete, which meant the protocol would be disregarded.
The malicious-rep-ring protocol has a much longer training time because it allows one malicious
adversary to execute it. MP-SPDZ will generally be more expensive due to more intricate measure-
ments needed to keep the protocol secure [23].
The semi-honest version is much more feasible, so it was chosen as the model for comparison with
the feedforward neural network without MPC and evaluation.

9.3.3
Conclusion
The models presented in table 17 are similar, with some differences in the metric scores between
the two models. These differences may stem from round-off errors and the inherent randomness in
neural networks, as discussed in section 9.3.2. Using multiparty computation (MPC) for machine
learning, particularly the semi-honest version of the replicated-ring-party protocol, leads to an
excessive increase in training time, and this is considering that it is the fastest protocol tested of
those listed in table 16.
On top of that, implementing a realistic network delay would render the training process unfea-
sible. Especially considering that this is a downsampled and small dataset where the number of
transactions one bank would have can easily reach tens of millions. The model trained through MPC
can be improved further when trained with more epochs, and the training time might be improved
with better hardware than what has been employed in this project (see hardware specifications in
section 1.3). However, a network delay will remain a bottleneck, irrespective of the hardware used.
The only practical scenario for mitigating such latency would involve running everything over a
local area network (LAN) in a high-performance data center to minimize the latency. It is possible
to achieve a latency of microseconds or even less [35] in such data centers, but this would defeat
the purpose of using MPC.

Discussion
Throughout the project, multiple considerations have been discussed, and this section is about the
elements that have not been discussed before or are vital parts of the project and justify further
discussion.
10.1
Dataset
The first decision was what dataset should be used for training and testing the various machine
learning models. The ideal dataset type would be genuine, authentic data given by a bank, but this
is impossible due to a breach of GDPR and possibly security measurements implemented by the
bank. The most significant difference is whether a wholly fictional or simulated/generated dataset
should be used. The closer the data is to being genuine and accurate, the more impact it can have if
good results are achieved. The difficulty is that real data (or close to it) will be highly imbalanced,
meaning fraudulent transactions will be a tiny subset of all the transactions.
Furthermore, there will likely be labeling issues in a real dataset because not all fraudulent trans-
actions are detected and, therefore, not guaranteed to be labeled correctly. This can have con-
sequences for a machine learning model because the tendencies and patterns it will find might
not be accurate. Yet, with an entirely fictional dataset, the machine learning model will become
almost useless because it is not a true reflection of the real world and, therefore, is less likely to
detect fraud in the world correctly. Due to the low likeliness of correctly identifying fraud in the
real world (labeling), it was decided to use a simulated dataset for training and testing instead
of a generated one, which has the same imbalance (few fraudulent transactions) as a real dataset
because it is based on statistics extracted from the American population to make the data as close
to the real thing. Another positive feature of simulated datasets is that more data can be generated
if necessary, as explained in section 4 and 4.2.
10.2
Model selection
Many methods and approaches could be used to select the models trained for fraud detection for
such a classification problem. This could get out of hand if it was not decided to narrow down
the choices of models as explained in section 5.2. Not all the initially chosen models trained were
selected for further feature engineering and tuning.
Only the initial configurations and models
that indicated the best preliminary results were decided and, in the end, used in the MP-SPDZ
framework.
These results were based on the PR-AUC metric score, as described in section 8.2.3. However, all
metrics are relevant when trying to reach a good generalization of a machine learning model, which
was also discussed in the previously mentioned section. Using metrics other than the PR-AUC as
selection criteria might have resulted in another model being chosen. Still, PR-AUC was used due
to an imbalanced dataset, which accounts for precision and recall at various thresholds.
10.3
Training and MPC
The goal of training the model on the dataset is to have the shortest training time with adequate
results for each metric specified in section 8.2.1, specifically, high recall and precision.
This is
important because of the overhead, which MP-SPDZ introduces. With a complex architecture of the
machine learning models, the training time will increase, meaning the architecture of the machine
66
==PAGE:74==
learning models aims to be simple. One way to reach a simple architecture and low training time
is by improving the data quality used for training and testing. By focusing on feature engineering,
many patterns can be extracted from the dataset, potentially making it more effortless and faster
for the machine learning model to learn and improve. This allows the machine learning model to
learn features more efficiently, which is why a simple architecture can be used. It is essential to
mention that even though the data quality can be improved by feature engineering, this does not
correlate to a simple architecture. Some challenges still require a more complex architecture, even
with high-quality data.
An example of this could be by creating an anti-money laundering model.
Such a model would
require more than one transaction for predictions but rather a set of transactions to learn money-
laundering patterns. Even when this is the case, the model will most likely still learn the features
faster than without feature engineering, but a simple model might not be the best option.
Besides improving training time in a machine learning environment, techniques like one hot en-
coding should be avoided to keep the dataset simple.
Thus, the model will also become more
straightforward, significantly impacting the cryptographic framework MP-SPDZ. Since the frame-
work has a substantial overhead in terms of training time and general performance, by having a
simple model, the configurations between the two environments can be as close to each other as
possible not only by avoiding techniques but also by applying various techniques, such as a correla-
tion matrix, to remove redundant features removed to simplify the dataset, as described in section
6. This would provide a better foundation for the comparisons in section 9.3.
These improvements that can be made to speed up the training process are crucial when used
for multiparty computation and in the MP-SPDZ framework.
When considering the security vs.
performance trade-off for MPC, prioritizing performance is necessary. This is because the security
measures inherent in MPC significantly slow the training process, making it unfeasible to train
models within a realistic timeframe, especially when the protocol allows for a malicious adversary.
This means sacrificing security, e.g., using the semi-honest version of the protocol, to train the
models.
Adding to the fact that the dataset that was trained in this project was downsampled,
whereas more realistic datasets would be significantly larger, containing millions of rows.
An example of the slowdown in MP-SPDZ occurs when the parties need to distribute secure shares,
especially during intermediate results during a forward pass in the neural network. This leads to an
increase in network communication rounds, significantly extending the training time. If a realistic
artificial delay of 10 milliseconds were introduced, the training time for the current models in
MPC would increase from 19 hours to approximately 68 days. This poses a considerable challenge
in real-world scenarios, especially since delays can exceed 10 milliseconds when participants are
located in different areas.
Besides the cryptographic environment and the impact of the protocol on machine learning models,
MP-SPDZ cannot run on GPUs. If it could, machine learning training and testing time could be de-
creased when comparing CPU and GPU in a standard environment. Since many mathematical oper-
ations, such as matrix multiplication, in machine learning can take advantage of the GPU[20].
Another method alternative to MPC is federated learning. This method works by locally training
the model and sending it to a central aggregator to update the weights. Each party then receives
the weights and trains some of the models again. However, this method comes with an issue called
inference attacks [17]. An attacker can reverse engineer the output by sharing the weights to get
the input data, which is not possible with MPC.

Environmental impact
A long training time will decrease technical efficiency and increase the power consumption of

Future works
Model improvements: It is possible to improve the machine learning models further through
hyperparameter tuning, combining multiple configurations, adjusting the learning rate, or training
the models for longer epochs. Also, the dataset selected in this project contained much more data

general improvements in the metrics.
Transfer learning: Apply transfer learning again for the newly trained model, as more data might

results of inference of the machine learning model, such as a list of the highest probability of fraud
being shown.
Extension of the model: To create a comprehensive model for the financial sector, the machine
learning framework could be expanded to incorporate anti-money laundering (AML) measures. It
would be interesting to explore whether a single model can effectively address anti-money launder-
ing and credit card fraud and the techniques required, such as Long Short-Term Memory (LSTM)
networks. If successful, this enhanced model would allow for the simultaneous identification and
flagging of both types of fraud.

Conclusion
This thesis has explored and analyzed the integration of multiparty computation (MPC) with ma-
chine learning (ML) models to test, evaluate, and improve fraud detection within the financial
industry while maintaining data privacy.
Simulated datasets containing credit card transactions and personal information generated based
on statistics from the US population have been used. This makes the dataset closely mimic real-
world data, giving the basis for training and testing machine learning models to be effective at
fraud detection in a simulated environment and possibly in the real world. A comparative evalu-
ation and analysis of the models were conducted using the metrics defined in 8.2.1 on the range
of different classifiers suitable for such a task. This includes random forest (RFC), isolation forest
(IF), Decision Tree classifiers (DTC), logistic regression (LR), and a feedforward neural network
(FNN). The analysis shows that the decision tree classifier, random forest classifier, and feedfor-
ward neural network are the best candidates for effectively performing fraud detection classifica-
tion models based on the Precision-Recall Area Under Curve (PR-AUC) metric. The specific results
are showcased in tables 17, 14, 11, 10, 9, 8. However, the feedforward neural network, based on
the hardware specifications used in this project, is the only one suitable for use in the MP-SPDZ
framework.
While combining the two, ML and MPC, allows collaborative fraud detection among multiple banks
or parties, critical considerations and shortcomings must be addressed to ensure the practicality
and effectiveness of running such a system. This is because running such a system where machine
learning is used together with MPC introduces significant limitations both in terms of security and
performance. Network latency among the parties using an MPC protocol leads to increased training
times to impractical levels, making the feasibility of such systems inoperable in most use cases.
This can be mitigated by operating the MPC protocols within a local environment, e.g., a local area
network (LAN). Consequently, it limits the scope of secure multiparty computation, defeating the
purpose of collaboration with complete privacy and security. The training time and execution of
MPC protocols can be sped up using a semi-honest model but compromises the security benefits of
the malicious version that allows for one malicious adversary.
To combine machine learning and MPC, a strong emphasis on the dataset’s data quality is vital.
This can further reduce the training time in the cryptographic environment. The model can con-
verge and learn more effectively by extracting patterns and features from the dataset using feature
engineering techniques. Simplifying the architecture of the feedforward neural network models in
combination with clever feature engineering would further help lower the computational overhead,
which aligns with the challenges of the MP-SPDZ framework.
This thesis addresses existing fraud detection challenges while laying the groundwork for future
developments. Potential improvements for machine learning in multiparty computation (MPC) in-
clude optimizing MPC protocols for better scalability, using additional techniques to enhance the
dataset’s quality, and optimizing the machine learning models to reduce training time and computa-
tional cost. These enhancements could help bridge the gap between data security and collaborative
machine learning, ultimately contributing to developing secure machine learning systems capable
of detecting fraudulent activities.
