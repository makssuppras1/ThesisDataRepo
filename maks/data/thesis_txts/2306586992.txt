Industrial adoption of Model Based Software Development did not live up to initial
expectations in the scientiﬁc community. One of the factors for this is the tooling. We
believe that, in order to make progress along this dimension, the availability of high
quality sample model data is an indispensable prerequisite for this to happen. However,
we observe a shortage, if not to say almost complete lack of such sample model data.
The goal of this thesis is to provide the Software Engineering Conceptual Model Reposi-
tory (SECOMORE), a platform for collecting, curating, and sharing of conceptual mod-
els in software engineering as such.
Apart from the operational system as such, our
approach also comprises a concept for incentivizing future contributions, by publishing
models which allow for citations.
We pursued a highly iterative, Persona-driven approach and have achieved very high
levels of quality, as witnessed by test coverage in excess of 90%, build automation, and
frequent code reviews. We have validated our work by deploying SECOMOREto pro-
duction usage, seeding it with almost 3GB worth of conceptual models, thus creating the
by far largest model repository available today.

Chapter 1
Introduction
Model Based Software Development (MBSD, [1]) has been proposed as a method to
improve productivity in software development. Substantial academic research has been
conducted over many years, yet industry adoption of MBSD has been far below expec-
tations. Researchers have studied this phenomenon [2], and have subsequently identiﬁed
MBSD-tooling as a key factor [3–7]. If this is indeed true, we ﬁnd it obvious to ask
how the state of MBSD tooling might be advanced: What can be done to support and
expedite the evolution of tools for MBSD?
This is somewhat similar to the situation of modeling tools in the pre-UML era: There
were many insigniﬁcantly diﬀerent methods and languages for object-oriented modeling,
each supported by its own limited tool. These tools existed in their respective niche
markets and did not compete with each other. With the advent of the Uniﬁed Modeling
Language (UML, [8][1]) in the late 1990s, the market for OO modeling languages con-
solidated rapidly. All of a sudden, many tools competed in a much larger marketplace.
New players appeared on the market, eventually leading to much better modeling tools.
Applying this lesson to MBSD, we ask: What factor is missing to instill more competition
between existing MBSD tools? How can MBSD-tools be made comparable, like modeling
tools were made comparable by a widely accepted modeling language such as UML? We
believe the answer to this question lies in the availability of model benchmarks, or more
generally, sample models of high quality and with rich meta data. Such data sets would
allow tool manufacturers and research groups to compare their respective approaches, or
develop new approaches in order to better address market needs. There are a few online

repositories of models available today (see [9] for a recent survey), but they are scarce,
do not enjoy generous licensing conditions, lack meta data, and are of unrealistic origin,

In this thesis, we introduce SECOMORE, the Software Engineering Conceptual Model
Repository, referred to as SECOMORE from here on out.
It is based on a software
platform that we have built in our MSc-thesis, and ﬁlled with model data that our
supervisor has collected over the years. We have created SECOMORE in a way that
should support and encourage a self-supporting positive feedback loop of contributions
and beneﬁts for MBSD researchers and practitioners. We aspire to promote the sharing
of data within the community, and hope to have a lasting eﬀect on software development
as such.
1.1
Background
1.1.1
Models
A model, in our context, is a description of the problem domain. Models make it easier
for us to comprehend the problems we face and put them into terms we can better under-
stand. By clarifying ambiguous entities, models try to eliminate diﬀerent interpretations
between diﬀerent individuals.
A paper published in 2013, Universal Modeling Language in Practice [10] [11], investi-
gates how the Universal Modeling Language, or UML, is being used in the industry by
performing interviews with professionals. In the ﬁrst section, called Background, the
author investigates previous research done on the subject. Many software engineering
professionals there proclaim that they see no inherit beneﬁt of using UML and more
believe the cost of using it is too high.
In the same paper the author performs her own study, the consensus being that most
software developers either don’t use UML at all or only use it selectively. Most of the
developers asked, argue that they cannot ﬁnd a reason to switch from their already
established work ﬂow to something like UML.

We conclude from these researches that there exists no empirical evidence on what tools
leads to success and high quality software. There does not appear to be a suﬃcient
amount of quality data to perform those researches.
We aspire to change this situation. To create a solution that provides quality data to
researchers within our small niche, model driven development.
This project might not be born out of necessity, but the beneﬁts are compelling.
1.1.2
Model Based Software Development
Model Based Software Development, or MBSD, is the method of developing software
using models. There exist a few diﬀerent levels of this method.
Model Driven Architecture, or UML, is one end of the spectrum. Followers of that design
approach only create models and then generate code from that.
In the middle of the spectrum we have those that create models, generate some code and
then write the remaining code themselves to connect it together.
Finally, on the other end of the spectrum we have those that only sketch up models, either
using software or whiteboards. All coding is performed by these individuals themselves,
with the models serving as a point of reference.
This entire spectrum is what we deﬁne as modeling. SECOMORE is aimed for the entire
spectrum, as all of these diﬀerent models can become quality data sets.
1.1.3
Importance of data sets in scientiﬁc research
This lack of data has also been identiﬁed by other researchers around the world, as
there have been prior attempts to provide that data. However, these previous attempts
have had varying degrees of success. They are all either too restrictive, providing only
a particular type of model, or provide low quality data. These other projects are gone
over, in detail, in section 1.2.
This data has to come from somewhere, it has to be published by researchers. As pointed
out by Christian Collberg and Todd A. Proebsting in Repeatability in Computer Systems

Research [12], "...we must acknowledge the root of the scientiﬁc-repeatability problem is
sociological, not technological" and "The real solution to the problem of researchers not
sharing their research artifacts lies in ﬁnding new reward structures". This means that
for our solution to work, we have to adequately reward researchers when they provide
us with data. Researchers can be very territorial with their data, but provided the right
incentive, we believe they can be convinced.
1.2
Other Model Repositories
There are a few platforms out there that are trying to provide access to more data, but
they all fail in some way. These platforms are discussed in the following sections. Other
platforms were also identiﬁed, but are not worth discussing in our opinion. These can
be found in the Free Models Initiate poster from 2014[13].
1.2.1
ReMoDD
The Repository for Model Driven Development, ReMoDD 1, is a repository led by re-
searchers at Colorado State University in the United States of America.
It aims to
support the work of researchers and educators in the model driven development commu-
nity.
Their goal is to support sharing exemplar models and be a source of data for model
driven research. They however, in our opinion, fail to accomplish their goals.
The ﬁrst problem we identiﬁed with ReMoDD is that when you create an account, you
cannot immediately start using the platform. You have to be allowed permission by an
administrator, which is not encouraging visitors to share their models or partake in the
community. As a user, I want to start working immediately. We could not identify any
meaningful reason for this restriction.
There does not exist a lot of meta-data on their models, making it hard to assess the
quality of the model at ﬁrst glance. This implies it is very hard to categorize and compare
the models stored there, making it impossible for researchers to know exactly what to
expect from the models.

type, which makes it very hard for researchers to ﬁnd the correct data. We believe this to
be a result of the poor categorization and meta-data collection performed on the models.
If for example a researcher would want to ﬁnd all Class Diagrams, with more than 10
classes, created in the last year, it would be impossible.
They do not allow users, who don’t own the model, to add to the overall quality of the
model. That eﬀectively puts all of the workload on the shoulders of the contributor,
to improve the quality of the models. This will lead to poor quality of models from
researchers that do not have a lot of time, which we estimate to be most of them.
Researchers usually don’t have a lot of free time on their hands, which makes this platform
unusable for time constrained people. The only way to get some sort of idea what other
people think of the model, is to read the forum or groups on the platform. That again
will be a very time consuming task and meaningful contributions will be lost.
The statistics section of the platform only includes data from 2012 and older, which
suggests that the platform is not being actively developed or used. As of June 12th,
there are 81 results returned when viewing all artifacts on ReMoDD. That is a very
small collection of models and one would wonder how much help can be in such a small
collection.
All in all, ReMoDD is not an ideal implementation of a model sharing platform. They fail
immensely in extensibility of their search and meta-data gathering functionality, making
it close to impossible to use if your time is limited.

UML Repository (Gothenburg)
The UML Repository 2 is a platform solely focused on storing UML Class Diagrams led
by researchers at Chalmers University in Sweden.
Their goal is similar to ReMoDD, to improve the research community by providing a
platform around the sharing of models and data. However, they only accept UML Class
Diagrams. In our opinion the UML Repository is an interesting attempt at a sharing
platform, but ultimately fails in many areas.

The ﬁrst obvious problem with this platform is that it is solely focused on UML Class
Diagrams, which is a very narrow ﬁeld in the world of model driven development. If a
researcher wants any other kind of model, this platform is useless.
Their upload functionality only allows for image uploads, which is a bit restricting con-

uploading, it appears to perform some analysis on the model, counting the number of
classes and elements which is a decent metadata gathering feature but far from being
extensive.
Their search functionality is solely based on the analysis performed on the images. It
only supports text search and ﬁltering based on the numbers the analysis generates, as
in number of classes, number of elements and so on. This is, much like ReMoDD, a
very restrictive search. A search of class diagrams created in the past year would be
impossible.
Contributors and viewers cannot add additional metadata to the models to improve the
quality, which is another hindrance. That means that the quality is entirely dependant
on the analysis done by the platform, which is very limited as models can be subject to
diﬀerent assessments performed by diﬀerent individuals.
All in all, the UML Repository is a decent platform for UML Class Diagrams, albeit not
very extensive. The downfall, like ReMoDD, is its extensibility. The search functionality
is very limited and the overall quality of the models is entirely dependant on the very
minuscule analysis performed by the platform.

1.4
Approach
At the start of this project we did not have a clear idea of what the best solution to
this problem might be. We had some ideas of what direction we wanted to take, but it
became apparent that an iterative approach would suit best.
As we are on the forefront of science, there were a large amount of unknowns. With an
iterative approach, we would quickly develop a feature and test it out. That way, we
could look for the optimal solution via trial and error.

With the diverse requirements of our persona’s, the decision on what kind of database
to use is imperative.
3.5.1
SQL vs NoSQL
In the following bulletpoints, we will go over some factors we researched in an eﬀort to
pick the right choice for our platform.
1. Performance for small amounts of data, between SQL and NoSQL is minimal.
For large amounts of data, NoSQL beats SQL in most operations. A single GET is
required to fetch the data needed. In contrast to that, SQL databases might have
to perform multiple JOIN operations to get the same data from diﬀerent tables.
The diﬀerence would be calculated in milliseconds.
Milliseconds of wait does not negatively impact any of our persona’s, as the speed
diﬀerence is not humanly noticeable.
2. Data integrity is very important to Thomas and Ranjit. SQL beats NoSQL in
data integrity at all levels. There have even been examples of NoSQL databases
losing data 2, which is completely unacceptable in the platform we are designing.
3. From our initial brainstorming of the data structure of the platform, we have
concluded that it is relational in nature.
Our main data models are models,
reviews and users. Normalizing our data into tables by those data models makes
much more sense to us rather than storing them all together in one large document
or object.
4. Searching for contributions is one of the core features. Véronique wants to be
able to input parameters of the contributions she wants to acquire and get back a
list of contributions that best ﬁt that criteria.
NoSQL does not provide us with a standardized query language to get data based
on a query from our users. NoSQL does not provide us with standardized objects
to work with, which means there has to exist a search engine on top of the NoSQL
2https://aphyr.com/posts/284-call-me-maybe-mongodb
==PAGE:39==

31
database. Some objects might also include attributes others do not have. We could
force the users to include all important data in the model form in the user interface,
but then we are moving away from the beneﬁts of a dynamic schema in NoSQL.
5. Reliability: SQL databases have been in heavy use for over 30 years now. NoSQL
has just become very popular in recent years. We want to ensure that the database
we pick has the best reliability there is, as our persona’s want the platform to
survive for eternity.
Conclusion
Having gone over some important factors of our platform, the choice seems fairly obvious.
SQL databases provide us with better data integrity, are more reliable and give us a very
robust and standardized query language. The only factor NoSQL wins is performance
and that is not a very important one for most features. The fact that NoSQL have
been reported to lose data made the decision all the easier as that would be completely
unacceptable for this type of system.
Going with a SQL database ﬁts better with the needs of our persona’s. Within SQL
databases, we chose to go with PostgreSQL, as we have great experiences with it and it
is considered to be the most widely used and reliable database system out there.
However, as mentioned later, our search engine B.1.8 uses a NoSQL database for creating
an extensive search. This has no eﬀect on our actual data storage, as that database is
purely for searching.

3.7
Safe storage
Our persona’s require SECOMORE to have a high level of data integrity. We, as the
developers, cannot aspire to create the needed ecosystem to provide a high level of data
integrity all by ourselves.
That requires a lot of hardware components to maintain
redundancy and backups of the data. This would both cost us a lot of time and a lot
of money to do ourselves. With our limited resources, this is impossible. That led us to
look for a service providers that could help us meet the needs of our persona’s.
Many companies today are promising high levels of data integrity by storing ﬁles in the
cloud. Companies like Google, Dropbox, Microsoft and Apple all have their own versions
of cloud ﬁle storage.
Our supervisor, Harald Störrle, mentioned to us that he knew of a very secure cloud
storage service provider called Zenodo3. Zenodo’s main focus is sharing research, which
ﬁts perfectly with our vision. Zenodo is also responsible for storing data generated by
the Large Hadron Collider built CERN, located near Geneva in Switzerland.
Zenodo also has an added beneﬁt compared to the other choices of cloud storage. As
they focus on research, they provide their own Digital Object Identiﬁers, DOI, at no
additional work or cost. After uploading a ﬁle to Zenodo and publishing it there, a DOI
is automatically generated. This is extremely useful to Thomas, as the bibtex generated
for his models will contain its DOI, to allow for easier citations. Therefore, Zenodo is
the obvious choice here.
However, we cannot place all of our trust on Zenodo being operational to the end of
times. Therefore, we have planned for a functionality that will allow an administrator
to download everything from Zenodo to his local computer, in the event of Zenodo ever
ceasing their operation.
The technical implementation is discussed in greater detail in section 4.6.11.

4.6.1
Create contribution
The create contribution process was one of the ﬁrst things we implemented. Being one
of the cornerstones of the platform, we had to get it right both from a users perspective
and from a technological perspective.
Thomas and Ranjit are both very busy individuals. They want to spend as little time
possible when contributing to the platform.
Therefore, the creation process has to be as simple and painless as possible for them.
We do not want to discourage Thomas and Ranjit from contributing to the platform by
having a lot of diﬀerent ﬁelds to ﬁll out, which takes a lot of time.
We decided that to create a contribution, Thomas and Ranjit only have to input the
bare minimum needed. The bare minimum is a name, a type, collection and under what
license the user wants to store the contribution. The form can be seen in Figure 4.7.
The available types are:
1. Diagram is an image of a model
2. Speciﬁcation is a structured description of a model
3. Prose text is any free ﬂowing text
4. Model Item is a speciﬁc ﬁle containing a model
5. Model Family is a collection of Model Items, Diagrams, Speciﬁcations, Prose
Text and other Model Families
6. Model Repository is a collection of Model Items, Diagrams, Speciﬁcations, Prose
Text and Model Families
Model families and repositories do not require a ﬁle to be uploaded with the contribution,
but all others do.
The collection option allows Thomas and Ranjit to create a hierarchy of repositories,
families and items to give their collections of contributions a more logical structure
within the platform. The available options in this ﬁeld are the model families and model
repositories currently owned by the user that is logged in.

4.6.2
Create new version of contribution
In the contribution detail page, Thomas or Ranjit can decide to upload a new version
of their contribution. That only requires them to upload a new ﬁle and choose a new
version number. We do not set any restrictions to the version number, other than format,
and put it at the discretion of Thomas and Ranjit to choose a version number that makes
sense in the context of their contribution.
Behind the scenes the new contribution is then related to the old contribution using a
many-to-many connection. We decided to go with a many-to-many connection to allow
for branching of versions. To allow one version to lead to two other versions and those
two versions then later merged into another one. That would provide us with a rich and
ﬂexible version history.
However, this functionality does not exist in SECOMORE yet, but is a suggested add
on. We did not implement this feature as creating a new version was introduced rather
late in the development process. With the complexity of having branching versions and
the limited time we had, we decided that the simpler implementation of linear versions
would suﬃce for the time being.

There are a few background processes, or cron jobs, needed to maintain and communi-

4.6.11.1
Upload to Zenodo
Figure 4.21: Upload to Zenodo activity diagram
The background process maintaining the uploading to Zenodo is crucial, as that is solely
responsible for maintaining the high level of data integrity we require. Each contribution
has a boolean attribute that maintains whether the contribution has been uploaded to
Zenodo or not.
==PAGE:66==
Implementation
58
The process queries the database every night, checks if this variable is false for every
contribution, uploads the ﬁle to Zenodo if it is, updates the metadata Zenodo stores,
retrieves the Digital Object Identiﬁer that Zenodo generates for each upload and then
ﬁnally changes the aforementioned boolean variable to true, to make sure the contribution
is not uploaded to Zenodo again. This is visually depicted in Figure 4.21.
Zenodo requires a type, title, creator names, description, access right and a license.
The access rights can be any of ’open’, ’closed’, ’restricted’ and ’embargoed’, which we
partially mimic in SECOMORE by providing ’open’ and ’closed’ as our access rights.
As Zenodo requires a license with their upload, our users must pick a license from a
predeﬁned list. SECOMORE uses the same list as Zenodo, to provide a more tightly
coupled integration.
Zenodo has been providing this service for quite some time, so
we believe their license list is adequate, therefore SECOMORE does not provide any
additional ones.
As discussed in Section 3.7, Zenodo fulﬁlls all of our requirements as our safe storage
service provider. However we are a bit sceptical in putting such an important task outside
of our reach. What if they decide to shut down their operation in a few years? We do
not want to lose our data in that case. We decided that implementing a fail safe would
allow us to have all of our bases covered, discussed in the next section.
4.6.11.2
Download all from Zenodo
We implemented a feature, executable by administrators only, to batch download ev-
erything we have stored on Zenodo and put it into a single zip ﬁle. The administrator
merely has to click a button, which creates a process request in the database.
==PAGE:67==
Implementation
59
Figure 4.22: Download all from Zenodo activity diagram
We have a background task, that runs every hour, which checks if any new process
requests have been created in the database. If a download all from Zenodo task has been
created, the process will start downloading everything from Zenodo and put it into a
folder.
When everything has been downloaded, that folder is zipped. The zip ﬁle is attached to
the process request. This is visually depicted in Figure 4.22.
To access the zip ﬁle, SECOMORE allows the admin to download it to his personal
computer. How this is displayed in the view can be seen in Figure 4.23.
==PAGE:68==
Implementation
60
Figure 4.23: Download all from Zenodo layout
There are a few reasons for executing the business process in this manner. This process
takes a lot of time and we do not want to force the admin’s browser to wait for a
response from the server while it downloads everything. When our storage on Zenodo
starts getting really large, this could take hours.
Any interruption to the browser’s connection to the server would result in the admin
having to restart the process all over again. This is all avoided by using a background
process that is not connected to the users computer.
4.6.11.3
Challenges
An issue we discovered, while implementing these features, is that ZenodoâĂŹs supposed
Restful API, is not Restful at all.
We found that after receiving a 200 status code from Zenodo, for example after an update
on a resource, the update had not been persisted or performed when we sent our next
request to their API.
They sent back a 200, an OK response, while not having performed the action at all.
This goes completely against the Restful architecture.
To perform an edit on Zenodo, you must ﬁrst send them the information you want to
edit in one request, the access right in our scenario, and then perform a publish action
in another request for the edit to be saved.
We had to send the edit request ﬁrst, wait for a response and then send the publish
request after receiving a 200 status code. The weird behavior was that sometimes it
appeared that we were trying to publish something that had not been edited, which is
impossible, even though we had clearly sent the edit before the publish.
==PAGE:69==
Implementation
61
This forced us to implement a rather cheap solution, something we are not entirely happy
with, but are forced to do. After receiving a response from Zenodo, we tell our process
to wait for 15 seconds.
Zenodo is in our mind still the best choice, even after realizing their Restful API is not
optimal. We decided to stick with Zenodo.
We contacted the Zenodo support about this issue and they said they were aware of it
and were working on optimizing their REST API. Sadly, they do not expect to ﬁnish
their optimization until the middle of June 2016, around the time we are handing in this
thesis. That makes it impossible for us to fully optimize the process in time for the hand
in. We suggest that the optimization be implemented at a later stage.

6.1
Iterative approach
As mentioned earlier, with the large unknown factors of this project, we decided to take
an iterative approach to accomplish our goals.
We aimed to deliver small branches at a time, using Git to the fullest, to enable numerous
iterations on each functionality. By delivering small branches, it made reviewing the code
easier.
That allowed us to keep better track of what each of us was doing and made it easier to
catch bugs.
6.2
Code Reviews
Bitbucket has a code review feature. Every time a contributor creates a merge request
on a feature branch, the diﬀerence between the master branch and the feature branch
is color coded. Bitbucket then allows you to write comments, creating a way for us to
communicate directly regarding speciﬁc lines of code.
Therefore, we decided that it would be best in our scenario to have a small code review
process every time a new merge request was created. The creator of the merge request,
would assign it over to the other contributor, who would then read the code and write
comments on lines of code noting any questions or possible defects.
71
==PAGE:80==

unit tests locally on his machine to make sure they all pass. If there are any failures, the
creator is requested to ﬁx them. Finally the reviewer performs manual testing on the
new functionality introduced on this branch. If everything is working as intended, the
reviewer is responsible for merging the feature branch onto the master branch, which is
mostly performed by Bitbucket.

6.4
Continuous Integration
Continuous Integration, or CI, is a practice within software engineering where a central-
ized system fetches your code, builds it, runs tests and integrations and ﬁnally reports to
you if any problems arise. Having a CI makes sure we are not building a system speciﬁc
to our own operating systems.
Our choice for CI was Codeship. Codeship is a web based tool and has built-in support for
GitHub and Bitbucket. Codeship can listen to code being pushed to the Git repository
and react to it. How it reacts is conﬁgured by the developer.
==PAGE:87==
Development process
79
SECOMORE’s Codeship conﬁguration is that every time a push is made to the reposi-
tory, regardless of the branch, Codeship pulls the code, installs the project, builds it and
runs test. If any of these steps fail then an email is sent to the developers.
The next step is to actually ship the code itself. This was only done if there were made
any changes to the Master branch. While there are no restrictions on the tests with
regards to code coverage, the code shipping process will halt if any of the tests fail.
The code shipping invokes a script that overwrites the ﬁles on the development machine,
with the ﬁles that Codeship has built. Then it will install any dependencies and run the
project.
The reason for using CI is to reduce the workload on the remote development machine.
Shipping a compiled project means that the remote development machine does not need
to build the project, and therefore does not need to install the packages that are required
for building the project. It only needs to install the system dependencies and run the
project.
This automated shipping of code was important for our project as it gave us quick
feedback if something was wrong with the deployment.
Having it connected to the
master branch meant that the whole deployment process was tested every time a new
feature was implemented.

support each other.

Angular-charts is currently in version 0.10.2, but when

integration, all graphs in the same view render correctly.
We learned that the newest version of a library can still include some major bugs,
despite being stable versions. Haystack and Angular-charts are both leaders in
their respective domains, but still exhibit these bugs. We have learned to proceed
with caution when using libraries.

B.1.5
Django-cors-headers
Cross origin resource sharing1, CORS, has mostly become a standard practice these days
with the advent of nearly every website being a distributed system.
CORS makes sure, on a browser level, that only resources (images or scripts) from trusted
sources are used in the rendering of a webpage. By default, content from the same origin
is allowed. So a user directing his browser to http://www.example.com will load any
content that the frontend will make a HTTP call for, if the path of that content is also
1https://developer.mozilla.org/en-US/docs/Web/HTTP/Access_control_CORS
==PAGE:103==
Appendix B
95
http://www.example.com If however the frontend of http://www.example.com makes a
HTTP request to http://facebook.com for a user proﬁle picture, then Facebook needs
to set the CORS header allow the display of their content (usually done with an asterix,
signaling a wildcard) and the developers of http://www.example.com need to state that
content is allowed to be displayed.
In short, if the frontend and backend are not of the same origin, they need to be in
agreement that the content can be used.
Django-cors-headers allows us to set this CORS header on for every HTTP request in
the app settings, instead of having to explicitly write it into every HTTP response. This
is done with a whitelist of domains:
CORS_ORIGIN_WHITELIST = (
’localhost:9000’,
’obara.azurewebsites.net’
)

B.1.8
Elasticsearch
Elasticsearch is our chosen search engine. The python library used to communicate with
the Elasticsearch search engine is also called Elasticsearch. The Elasticsearch library
then depends on urllib3 for creating the HTTP requests, as Elasticsearch uses a restful
API. Elasticsearch is built on top of Apache Lucene, an open source information retrieval
library.
Elasticsearch appeared to be the easiest to set up of the four available search engine
options that Haystack supports (Elasticsearch, Solr, Xapian and Whoosh). It was only
recommended to use Whoosh for testing or very small scale project, which made us
exclude that as an option. Xapian is written in C++, requires compilation and is not
fully included in Haystack due to licensing, which made us exclude that as an option. Solr
appeared to be harder to work with and not fully restful, like Elasticsearch. Elasticsearch
is fully open source, which made it an even more enticing option. Haystack only supports
Elasticsearch 1.X, and not 2.0 or higher, but the maintainers are currently working on
upgrading.
==PAGE:107==
Appendix B
99
To search, Elasticsearch uses a few well known tricks to ﬁnd the relevant records. Firstly
it uses the Boolean Model, which simply utilizes the AND, OR and NOT boolean oper-
ators to ﬁnd all records matching the query. The search query ’John’ AND ’Smith’ will
only return results containing both words.
Secondly it uses Term Frequency, the more often the search term appears in the record,
the higher the weight of that record becomes. A record containing the search query ﬁve
times, is more likely to be relevant than a record that only contains it once.
Thirdly it applies Inverse Document Frequency, as in how often does the search term
appear in the whole pool of records. The more common the search term, the less weight
it will have. Common terms like ’and’ will therefore not contribute a whole lot to the
relevance of the record. However, an uncommon word like ’bison’ would narrow down
the search to relevant documents easily.
Fourthly it employs Field-length normalization, where the longer the value of an at-
tribute, the less weight it has. Which means that if the search term occurs in an at-
tribute with small values, like the ’name’ attribute, it is more likely the record is about
that search term.
While if it appears in attribute with normally long values, like a
’description’ attribute, it is unlikelier that the record is actually about the search term.
These three methods are calculated at the time of insertion into the index for every
record. Jointly, they are utilized to calculate the weight of a single term in a speciﬁc
document. This means that every word in a search, gets turned into a single number.
This can then be used to compare the search term to the documents in the index.
Finally it makes use of the popular Vector Space Model[2], presented by Salton, Wong
and Yang in 1975. The Vector Space Model gives us a method for comparing search
terms, containing a lot of words, with the documents stored in the index. Each word’s
weight in the search term is calculated and put into a vector. A vector is then also
created for each document, if the document contains the word the weight is put into the
vector, else a zero is put in there instead. This is best explained in an example.
Let us assume we have the search term ’blue curtains’. The index contains three doc-
uments, document 1 has the text ’The sky is blue’, document 2 has the text ’Those
curtains are beautiful’ and ﬁnally document 3 has the text ’I own blue curtains’. Blue
is a relatively common term and is therefore assigned the weight of 3. Curtains are a
==PAGE:108==
Appendix B
100
Figure B.2: Cosine similarity between vectors
relatively uncommon term and is therefore assigned the weight of 7. The search term
vector therefore looks like [3,7].
Document 1 only contains the word blue and not curtain, therefore the vector for doc-
ument 1 looks like [3,0]. Document 2 only contains the word curtains and not blue,
therefore the vector for document 2 looks like [0,7]. Document 3 however contains both
words, giving it the same vector as the search term, [3,7].
From that we can map those vectors onto a two dimensional ﬁeld and then compare the
angle between them, to determine their relevance. We can clearly see in ﬁgure B.2 that
document 3, D3, is the most relevant one based on the search query as there is no angle
between that document and the query. Document 1 and 2, D1 and D2, are less relevant
as the angle between them and the search query is larger.

Jasmine
Jasmine is a testing framework for Javascript. Jasmine utilizes a common syntax, making
it easy to read and write tests. Jasmine is best explained using examples, the example
shown in listing B.21 is taken from their website.
describe("A spy", function() {
var foo, bar = null;
beforeEach(function() {
foo = {
setBar: function(value) {
bar = value;
}
};
spyOn(foo, ’setBar’);
foo.setBar(123);
foo.setBar(456, ’another param’);
});
it("tracks that the spy was called", function() {
expect(foo.setBar).toHaveBeenCalled();
});
it("tracks that the spy was called x times", function() {
expect(foo.setBar).toHaveBeenCalledTimes(2);
});
it("tracks all the arguments of its calls", function() {
expect(foo.setBar).toHaveBeenCalledWith(123);
expect(foo.setBar).toHaveBeenCalledWith(456, ’another param’);
});
it("stops all execution on a function", function() {
expect(bar).toBeNull();
});
});

To create a collection of tests, you begin by calling the global Jasmine function describe,
which takes two arguments, a string explaining what is being tested and a function which
contains the test cases.
To create a test case, you call the Jasmine global function it. It, like describe, takes in
a string explaining what is being tested and a function which contains the test. Within
each it function, there exist one of more expect functions. The expect function takes in
two values, the actual which is usually a variable and the expected, which is usually a
static concrete value.
Between the actual and expected values is something Jasmine refers to as Matchers.
Jasmine includes an abundance of matchers, some examples are toBe, toEqual and toBe-
Deﬁned. Adding a .not matcher in front of any of the other ones inverses the comparison.
Some matchers, like toBeDeﬁned, take no expected value, as it is expected that the actual
is deﬁned, not a speciﬁc value.
The expect functions then take these matchers and either evaluate to true, if the actual
and expected are the same according to the matcher, or false, if they are not. A test
case including only expect functions that evaluate to true, is deﬁned as a passing test.
If one of these expect functions evaluates to false, the test has failed.
To minimize duplicate code, Jasmine provides a few setup and teardown functions, These
are beforeEach, afterEach, beforeAll and afterAll. BeforeEach and afterEach are exe-
cuted after each test case respectively, while beforeAll and afterAll are executed before
and after a test collection respectively. This allows for creating variables which will be
used by all of the tests, instead of deﬁning them repeatedly in each test case.
Jasmine also utilizes a thing called spies. A spy can replace any function in the code block
the test case is testing, track calls to that function and the arguments it receives. This
is commonly referred to as Mocking. The usefulness in that is that it allows us to have
completely decoupled tests that only test a speciﬁc component, and not the components
it depends upon. This is accomplished by using the spyOn(<class>, <function>), which
will replace any call to that function in that class with a spy.
In the context of those tests, we do not care what logic the other dependant components
implements, we only care that the correct function is being used and invoked with the
correct parameters. The logic of the dependant component is tested in that components
==PAGE:123==
Appendix B
115
speciﬁc test collection, which means it is already covered somewhere else. There is no
reason to have double coverage.
Double coverage also makes changing and refactoring the code much harder, as tests
in the component and the related components will all break when the logic is changed.
That creates more work for the developer, as he must now repair a larger amount of tests,
compared to when using mocks he would only have to repair tests in the components he
changes.
The spies have speciﬁc markers, like toHaveBeenCalled, and toHaveBeenCalledWith that
can be used to verify that the function was called and was called with the correct ar-
guments. It is then also possible to have the spies return a speciﬁc value, through the
and.returnValue marker. This allows us to completely control what data is returned and
if the function in the tested component is processing the received data correctly.
Jasmine is a great and easy to use testing framework that helped us greatly in improving
the overall quality of the platform.

B.2.15
Angular-Local-Storage
Angular-Local-Storage is an Angular wrapper for manipulating the local storage of
browsers. All modern browsers have local storage that allows for information storage
between tabs and sessions. The Angular-Local-Storage library allows us to connect to
==PAGE:126==
Appendix B
118
that local storage in an angular way, through a service. This makes the code both more
consistent and readable.
Local storage arrived to the scene with HTML5 in 2014, greatly simplifying frontend
storage. Before local storage, developers were forced to use cookies to store information,
which were sent with every request. Local storage is both safer and has a far greater
storage limit (at least 5MB, compare to 4000 bytes in cookies). Local storage is also
considered to be easier to use and to have a friendlier API. The Local storage is a key-
value store, meaning you provide a unique key and associate it with a value. When you
want to retrieve the value later, you simply provide the key.
In our platform we are using the local storage to store authorization tokens, user proﬁle
information and the expiration date of the token. This allows us to automatically log in
users, if their token has not yet expired. Each token is valid for 24 hours, which means
that we allow for automatic log in within that time. The user information stored is used
to present the email in the navbar alongside other information without having to contact
the server. A code example for this can be seen in listing B.22.
localStorageService.set(’Auth-Token’, currentAuthorizationToken);
localStorageService.set(’User’, currentUser);
localStorageService.remove(’Auth-Token’, ’Auth-Token-Expire’, ’User’);
localStorageService.get(’User’),
Listing B.22: Angular-Local-Storage example
The Angular-local-storage wrapper exposes three functions, Set, Remove and Get. Set
takes in two arguments, a key-value pair, with the ﬁrst argument being a key in String
format and the second one being a value in the form of a Javascript variable. The Remove
function takes in any number of String arguments, the String’s being keys. The Remove
function then removes the key-value pairs, uniquely identiﬁed with those keys, from the
local storage. The Get function takes in a single String argument, a key, and returns the
value associated with that key.
All of this allows us to give our users a better experience and limit the hassle of having
to authenticate with the platform on multiple occasions throughout the day. The library
is also extremely easy to use and allows us to be consistent throughout the code base by
Angularizing local storage.
