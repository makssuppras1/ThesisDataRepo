Summary
In the field of autonomous robots localization is always of utmost importance so finding
new and useful ways of localizing with different sensors is a fast moving area. In partic-
ular, when navigating around an indoor environment, GPS positing is very unreliable or
impossible. Therefore, other means of localization is required and must be investigated.
In this project indoor navigation af a hexacopter will be investigated when the area
is partially known. Using a camera, detection of visual markers with known location
will be used as global references for positioning. Having the markers distributed around
the area seeing such a marker with the camera and combining it with the knowledge of
its location the copter can calculate its precise position.
Having markers distributed all around the area would be impractical so the problem of
constantly relying on visible markers needs to be overcome. This is done using optical
flow to estimate the motion of the copter so that even when no known markers are visible
it is still possible to track the movement of the copter. Tracking Shi-Tomasi features in
the images using the Lucas-Kanade feature tracker with pyramids it is possible to find
the physical motion of the hexacopter and thus calculate how the position evolves as it
moves.
As reliable as optical flow might be, once in a while the position should be corrected
by means of global references e.g. the visual markers. This is done by making sure it
visits a known visual markers. To ensure this a path planning scheme is necessary that
can balance flying blind and using visual markers. Therefore the knowledge of the mark-
ers are arranged in a graph representing a map. This graph is then used for planning
paths by searching for the shortest path between markers and using this to generate
waypoints to fly between.
Combining all this together this project creates a method for flying around a partially
known environment while following safe paths using a camera for localization.

CHAPTER 1
Introduction
This project explores a solution for handling localization and navigation of an au-
tonomous hexacopter that must work in an indoor environment. The indoor environ-
ment is assumed partially known such that this knowledge can be used in the localization
scheme. The solution is to be self contained so that everything will be processed onboard.
1.1
Motivation
Multicopters or drones have in the last couple of years increased massively in popularity
among hobbyists and as toys. But not only are they fun to fly with, they can also be
very practical. Because they can maneuver all directions in space and even stand still
if need be they can easily be used in tasks where flying and accessing various locations,
otherwise inaccessible, is needed. Their versatility in maneuverability makes them very
useful in a wide variety of fields. Multicopters can be equipped with a wide array of
sensors and used for many kinds of surveying tasks such as geological, agricultural, or
archaeological. This could be measurements of every thing from magnetic fields to live-
stock locations to mapping ancient ruins to locating forest fires.
All of these examples are of course outdoor tasks and can thus make use of global
positioning systems (GPS) for localization. But if we move the attention to other areas
were multicopters can be used it is clear that we cannot always rely on GPS. For example
if we were to use a drone for building inspections the GPS signal could potentially be
blocked. Many inspection tasks requires reaching high locations or small spaces were hu-
mans could only reach with great difficulty if at all. Here drones could prove very useful.
Another very compelling quality of drones is that they can be made very cheap and
are thus highly expendable. This could be used in dangerous situations such as fires or
explosion in factories where affected areas need to be reached and examined or devices
operated. Sending humans would be too dangerous and ground vehicles might not be
maneuverable enough as paths might be blocked. All these obstacles can be overcome
using drones.
Also, in search and rescue they can be used to find people in collapsed buildings or
caves and bring food, water, or something else. But for any of this to be possible the
location of the drones, also in an indoor environment, needs to be known and for this

1.2
Problem formulation
The goal of this project is to, based on the work done by Hjertstedt[Hje18], to develop
a localization scheme that uses a bottom mounted camera instead of the laser scanner
used by Hjertstedt. Incorporating the use of visual markers as known global positions
the hexacopter should safely be able to fly indoors without losing its position. Besides
being able to localize, safely navigating from marker to marker is also desired. There-
fore, a scheme for representing the known space as a map and being able to plan paths
in this area should also be implemented. A big restriction when flying with markers
as apposed to with a laser scanner is that while there will always be walls around to
detect, there might not always be markers to see so re-localizing once in a while is needed.
From this several project objectives can be formulated:
• Determine how well the hexacopter can currently fly without any laser scanner or
camera data (free-flight capabilities)
• Explore if the free-flight capabilities can be enhanced purely from IMU data
• Develop a vision based approach for enhancing free-flight
• Use visual markers for global localization and use this with the current controller
• Develop a way of representing the known information about the markers
• Autonomous navigation in the known marker space with safe path planning
1.3
Thesis outline
The following 9 chapters of this report will detail how the problems formulated above
will be handled and the solutions that are chosen. In Chapter 2 the problem will be
analyzed and the thoughts behind the solutions and why they are chosen are explained.
This chapter will also grant an overview of the available hardware and its limitation.
Chapter 3 is a description of the visual marker detection and position estimation from
these markers. Chapter 4 is an analysis of how well the drone can fly without visual aid.
This will give even more reason to why an alternate localization scheme is required.
With the motivation for an enhancement of the free-flight capabilities chapters 5, 6
and 7 will describe the new free-flight localization scheme based on visual aids. Chap-
ter 5 describes the feature tracking and optical flow algorithms. Chapter 6 develops a

2.1
Related work into drone localization
Localization in indoor environments can be handled using many different approaches.
One such way is using a 2D laser scanner and matching the located lines (walls) with
known wall configurations as done by Hjertstedt, 2018[Hje18]. This approach uses the
laser scanner as the global reference and fuses this with data from an inertial measure-
ment unit(IMU), between the global updates using a Kalman filter. Similar to this Sani
& Karimian, 2017[SK17] fused positioning estimates from ArUco visual markers with
IMU data in a similair fashion though this was to be used for precise landing situations.
Purely using the IMU to base flights upon have been done by Hussein et al. in 2015[Hus+15]
with the data processing done on an off-board computer. Only using IMU is always prone
to bias integration error but can potentially be used in smaller intervals without problem.
This problem can be attempted fixed using a jerk based approach as done by Magdy &
Osama in 2015[IM15] to remove the bias.
Relying only on IMU is not to be preferred and some way of global localization is of
interest. Focusing on the visual approached such as the ArUco marker position other
good approaches have been tried. One such approach is for indoor flying proposed by
Franscisco et al. in 2016[BTN16] and Garcia, Mattison, & Ghose in 2015[GMG15] was
the detect of doors in hallways and localize from the door placements. Also simultan-
ious localization and mapping (SLAM) approaches have been taken such as Haddadi &
Castelan in 2018[HB18].
Another approach that moves velocity estimation from the IMU to camera techniques
is the velocity estimation from optical flow proposed by Grabe, Bülthoff, & Giordano in
2012[GBG12] which seems promising.

2.2.1
Notes on the position controller on the PixHawk
The position controller on the PixHawk being used in this project is an optimal con-
troller with a Kalman filter(linear quadratic Gaussian, LQG) developed by Hjertstedt.
In that project a Kalman filter is used to keep track of the copter’s position. The op-
timal controller can then send the correct pitch, roll and yaw commands to the flight
controllers low level regulators to get to the required position. The position estimate
fed to the Kalman filter is a laser scanner, but any global position estimate will work
with this controller. The LQG runs at 200Hz but the laser scanner is not that fast (only
30-40Hz) so in between laser scanner updates the LQR runs on a simple copter model
that relates the roll, pitch and yaw angles to the thrust generated and thus how it will
move based on these angles. This is combined with the data supplied by the IMU so
that accelerations and angular velocities are also used when updating the position.
This controller works very good when a high speed global positioning system such
as a laser scanner is running, but if no such data is available the LQG will be running

2.3
Motivation for solutions
As this problem is about using visual aids for localization it is convenient to start here.
Prior to this problem the ArUco markers have been investigated and showed good per-
formance and was easily integrateable with the LQG on the copters. The work by Sani
& Karimian[SK17] also showed that using ArUco in connection with autonomous drone
flight was very viable so the ArUco markers remained the basis for global localization.
Now the markers had only been used as a relative position and not in a global sense
so it is necessary to find a way of using multiple markers for localization over a large
area. Using ArUco markers are also very good as the copter has a down facing camera
so relying on features around the copter like some of the other methods used would not
be very applicable.
The next problem relates to the copters flying capabilities when no markers are visi-
ble. In section 2.1 we saw that based on IMU measurements only some groups have had
luck in only using this. They must either have had very good calibration on their IMU
and very small model uncertainty, or as one group, the estimated jerk from IMU was
good before using it. As will be seen later just using IMU, as the copter currently does,

is not good enough for reliable flight.
So using a jerk estimation approach is the next possibility. This was briefly investigated
but discarded. Based on the work by Nakazawa et al. in 2003[NII03] accelerometer
data from previously recorded flights were used to estimate jerk using Savitzky and
Golay smoothing filters(Savitzky & Golay, 1964)[SG64]. Using the offline filtering did
show some improvement compared to the Kalman filter on the copter, though it was
not reliable enough. The work by Nakazawa et al. showed that the Savitzky and Golay
filters were better at estimating jerk on offline data than a causal online filter would
be so implementing this on the copter would worsen the performance even more so this
approach was benched.
With IMU based solutions out of the question the next approach was to use optical flow
like shown by Grabe et al., 2012[GBG12] as their work seemed to fit the hardware setup
that of this project well. This also fit very well with using the camera as a complete
substitution for the laser scanner. Implementing an ArUco marker combined optical
flow for free flight localization was thus chosen as the basis for this project.
With a complete localization scheme only the navigation part of the problem needs
to be solved. Here it is chosen to use a graph based approach as the information that
is assumed available e.g. the positions of the visual markers is very similar to what is
contained in a graph. Using graphs also gives access to good and tried search algorithms
very suitable for autonomous robots such as the A* shortest path algorithm.
Getting all of this working should yield a good solution for solving the problems
stated in the problem formulation.
2.3.1
Final note on coordinate frames
In this report many different coordinate frames will be mentioned. The first frame is
the global frame or world frame(WF) which is the overall frame where everything is tied
together. The next coordinate system is that of the visual markers(MF). Around each
marker is a local coordinate system. This system will have a position in the world frame.
The last set of frames is related to the hexacopter. The center of the copter where the
IMU is located is the copter’s own coordinate frame called the body frame(BF). Lastly
the camera has a frame of its own. This frame is the camera frame(CF). This is the
frame used when relating real world objects or motions relative to the camera. Also there
is the image frame(IF) which is the image taken by the camera. This is for example
pixel positions of objects in the image. The image frame is coinciding with the camera
frame which is rigidly attached to the bodyframe.
All the frames and there relation is visualized in Figure 2.3.

ArUco visual markers
An important part of autonomous navigation is localization with respect to a global
reference. Without a global localization we will basically be running on odometry so we
only know how we have moved based on the perceived motions over time. This can be
good to an extend but will not suffice in the long run. Therefore, it is necessary to be able
to determine the global position based on some metric. In this project visual markers,
and in particular the ArUco(Augmented reality University of Cordoba) by Rafael Muños
Salinas[Sal], are used. An example of an ArUco marker can be seen on Figure 3.1.
Figure 3.1: An ArUco marker used to estimate the position of the drone.
The idea of using visual markers is that they are a unique feature so that we can
use beforehand knowledge about the marker’s position to estimate the copter’s position.
This is similar to using knowledge about a room’s walls to estimate the position with
a laser scanner. Here it is instead the markers that are identified and used to find the
position.
The markers are each unique and the pattern represents a number or ID. A marker

3.1
Position estimation from ArUco markers
As said it is possible to find the position of the drone using ArUco makers. Here it will
be explained how this is done.
3.1.1
ArUco marker detection
When detecting ArUco markers in an image the following steps are calculated:
• The image is converted to grayscale as the markers are black and white
• A filter is applied to detect all edges in the image
• All enclosing contours are detected
• Now it is necessary to remove all contours that are not markers
• Small contours are removed
• The contours are fitted to approximate a polygon with 4 corners and all that do
not fit are discarded
• Contours that are too close to each other are then removed. This is done to remove
the internal contours of the markers.
• Now for each of the remaining contours:
– A homography is fitted to get a frontal view of the potential marker
– Otsu thresholding is applied to the contents to get a binary image
– The binary plane is divided into a n×n grid where n is the size of the marker
matrix plus two
– If the contour was a marker each grid point will be either 1’s or 0’s and if so
a (n −2) × (n −2) matrix containing the marker ID is calculated
– If the marker was valid the matrix is compared with a marker dictionary to
find the correct integer ID. The matrix may need to be rotated to match
– If the marker exist in the dictionary the corners pixel position is refined to
sub pixel accuracy

CHAPTER 4
Current ’free-flight’
capabilities
To successfully be able to navigate around we need to know how well we can traverse
distances when no markers are visible. Especially how accurate a distance can be tra-
versed is very important. The error in position when flying is what we are interested
in finding. As we have seen when ArUco markers are always visible the copter has no
problem in flying to a specific position. But when markers are not visible we rely only
on the internal state estimator which uses IMU data and a model of copter behavior to
calculate positions. How well this works in practice, where we have a lot of uncertainties
on IMU data and calibration and physical configuration of the copter is not necessarily
good enough to be useful flying with. Therefore, a series of experiments were carried
out to determine these free-flight capabilities.
4.1
Loiter capabilities
The first thing tested was the drone’s abilities to hold its current position in space
(loitering). This was done by having the drone take-off and hold a position above an
ArUco marker until it was stable. Then the position estimated from the ArUco marker
was withheld from the controller. That is, no global position update was sent, so that
only the IMU measurements on the PixHawk and the copter model was used.
The position found from the ArUco marker was stored so that the true position could
be compared with the perceived position of the state estimator. To ensure that the true
position could be tracked for a long time a grid of ArUco markers were laid out as can
be seen in Figure 3.3. Having a grid like this ensured that the true position could be
tracked for a longer time compared to just having a single marker.
4.1.1
Results
Looking at Figure 4.1 we can see that very soon after the ArUco position estimation is
not send to the Kalman filter it goes wrong. It is clear that the actual position of the
copter moves very fast away from the target position. After only a couple of seconds
the copter has moved 1 meter in each direction so clearly the copter is not able to hold

its position.
Moreover if we look at how the Kalman filter position estimate evolves we see that the
position starts to drift slowly and then starts to correct. The reason for the errors in
this can be accredited to the bias errors as the ArUco will keep the drone at the needed
roll and pitch angle to stay the same but when this is no longer the case these angles
will theoretically move the drone so that is what the copter estimates.
4.2
Flying uncertainty
The next thing to test was the uncertainty on the position when flying. This is needed
to know how far between global ArUco markers the drone can fly. The uncertainty
when flying between markers is important to know for when planning routes on a map
of ArUco markers. To test this a similar approach as for the loiter test is conducted.
The copter takes off and hovers above a marker. Then it is commanded to fly towards
a line of markers with no markers between the start marker and the finish line so that
it will be flying blind between, as seen on Figure 4.2. When the copter then reaches the
line of markers the position estimated by the ArUco markers can then be compared to
how the position estimate of the Kalman filter has evolved.
4.2.1
Results
After running multiple instances of the test with various distances combining all the
data in one figure it is visualized in Figure 4.3. Here it is very clear that the uncertainty
of the position estimate grows with the distance so that the longer the copter has to fly
the worse the position gets. From the figure it is also seen that after only 0.5 meter the
position is only correct with an uncertainty of ±0.25 meters which is very bad. From
these test it is learned that flying free is basically unusable. On the figure is depicted red

An important note on the result is that here is only depicted the flights that actually
made it all the way and as the distance increased the number of times when the drone
would simply not reach the target line, and start drifting in the completely opposite
direction, increased.

5.2.2
Lucas-Kanade with pyramids
One problem of this method is that if the movement is not small enough there will be
no corresponding pixels in the window in question. Therefore we make use of image
pyramids so that we can overcome this problem with scale and large movements.
When moving up a level in the pyramid the size of the image is reduced by 4 as illustrated
in Figure 5.1. Level 0 is the original image. This reduction means that the window of
our Lucas-Kanade is effectively also increased by four. Therefore motions that in the
original image would results in features going out of their window could be detected in
the higher levels of the pyramid.
This principle is used so that the features motion is first determined at the top level
of the pyramid. Then this motion is applied to the windows on lower levels so the feature
is hopefully within the window. Calculating It is then done in the new window position.
This is done so that the window is moved with the motion as we go down the pyramid.
This means that we for each level of the pyramid, as we move down, calculate the Lucas-
Kanade flow and then use this flow when calculating the lower layer as illustrated in
Figure 5.2. This is done until the lowest layer of the pyramid is reached, meaning we
have reached the full resolution image.
It is important to note the flow we calculate at each level needs to be scaled according
to the level in the pyramid so that the scales are not mixed.
By using the pyramid we can overcome, as explained, large motion. But we can also

Combining Shi-Tomasi features with

With the theory behind the feature extractor and optical flow calculator explained it is
time to combine the two to get a full feature tracker. This is a very simple process.
First, a set of features is calculated using Shi-Tomasi features. This will yield the
location of the features in the first(old) frame. Next, instead of using the Lucas-Kanade
on all pixels in the image it is only calculate at the locations of the Shi-Tomasi features.
This will then give the movement of the features which is what we are interested in.
This process is repeated over and over for continuous flow.
Of course one needs to handle the features correctly and make sure not to ”run out
of features”. Often the new location of the features as computed by the optical flow is
then used as the new starting position of the features in the next iteration. If doing so
the features will eventually leave the image and are thus lost, so once in a while it is
necessary to estimate a new set of features if say the number of features go below 20
features.
Alternatively, as is done in this project, as we do not need to track the same feature
over a long distance one can calculate a new set of features after each flow calculation.
By doing so we always have a good set of new fresh features, given that there are any.
This can be summarized to:
• Given an old image from t −1 and a set of good features, f1, in this image
- Take the image from time t
- Calculate the flow of f1 between the two images
- Calculate a new set of features into f1 from the new image from time t
• Repeat

CHAPTER 7
Camera localizer
Now that all we need for localizing with camera is established this all needs to be put
together to work in practice. This is where the camera localizer comes in. The camera
localizer is supposed to run continuously and give an estimate of the position whenever
possible.
Therefore it needs to make use of the ArUco, Shi-Tomasi features, Lucas-
Kanade, and planar homography estimation to give the best possible global position to
the flight-controllers Kalman filter.
7.1
The localizer algorithm
The ArUco position estimation is the best and gives the true position without any drift
over time and if available at all time perfect flight would be possible. Therefore the
homography estimation is only used as a backup. The camera localizer can then be
described as follows.
The localizer starts as soon as it is activated and starts by opening the camera.
Hereafter it takes one initial image and saves it as the old image to get an initial set of
features for optical flow. After this is done the localization loop starts.
First an image is saved as the new image. This image is tested for ArUco position es-
timate. If an ArUco marker is present in the image the optical flow is skipped. So if a
known marker is detected the position is calculated. If no marker is detected a flag is
raised to calculate optical flow.
Now if the flag is raised we need to calculate the optical flow. To do this the old set of
features is tracked in the new image. From the two sets of feature positions we estimate
the planar homography and then decompose into the translation. This is then rotated to
the world coordinate system and added to the previous position to get the new position.
After the position is calculated a new set of features is extracted from the latest image
to be used on the next iteration.
A last feature of the camera localizer is that if an ArUco marker is detected after using
optical flow the new position given by ArUco is not used directly as the new current
position, but is smoothed towards this position over a couple of iterations. This is ex-
plained in more detail later, but it is done to prevent big jumps in the position estimate
given to the flight controller so no sudden movements are introduced.

7.1.1
Transitioning from optical flow estimation
In order to not have big jumps/discontinuities on the position estimations sent to the
flight-controller a smoothing scheme is introduced. The idea is that when the copter has
been flying with optical flow the position estimate will degrade in accuracy over time.
Therefore we will expect to have a difference between the true position (ArUco) when
that is found and the estimate until now. To solve this we only move the estimate a
fraction of the difference at a time.
∆p = pArUco(true) −pestimated(current)
(7.1)
pestimated(next) = pestimated(current) + 1
n∆p
(7.2)
Here ∆p is the difference between true position from the ArUco marker and the current
position as calculated from optical flow. This current position is now updated to the
next position by adding a fraction of ∆p based on n. n determines how many iterations
it takes before the true position from ArUco is used alone. So if n = 10 it will take 10
ArUco estimates to approach the position. The ∆p is updated between each frame by
the new location from the ArUco so that the movement the copter has while smoothing
is also accounted for.
A second part is that n is decreased by one between each update. This is done so that
the step size comes close the full step as we approach it so that in the end the true
position is reached when the smoothing is done. An example of how this looks can be
seen below in Figure 7.1. The smoothing is basically a ramping function but with a
changing endpoint and changing step size.

Graph construction
As mentioned a couple of things need to be considered when the graph is constructed.
First we need to consider how the graph is to be stored. Here either an adjacency matrix
or adjacency list could be used. Because we will be searching the graph for a shortest
path the adjacency list is convenient as the search will require looking at all the connec-
tions of each node when visited.
With the representation of the graph in place it needs to be filled out. To do this it
will be checking all the markers one by one and then checking if they are close enough
to fly between given the current height.
The first marker is selected and the visibility radius, rvis,1, is calculated. rvis,1 is the
radius around the center of the marker that it is safe to assume that given the copter
is within this radius at the current altitude, h, the marker will be visible in the image
frame. It is calculated according to

8.2
Finding the shortest path
With the graph constructed it now needs to be searched for the path from copter to
goal position. To do this the A*(A-star)-algorithm is used. A* is a search algorithm
that is used for finding paths between two points in a graph. It is similar to Dijkstra’s
shortest path algorithm but instead of searching the whole graph and give the shortest
path from the start to all nodes as Dijkstra’s does, A* just finds the shortest path to
the goal node. In order to improve performance over Dijkstra’s A* uses a heuristic to
determine the direction in which to search in the graph. As we are searching for short
paths in a 2D environment and it is possible to move in all directions Euler distance is
very suitable as the heuristic. The Euler distance is calculated as
d =
√
(x1 −x2)2 + (y1 −y2)2
(8.4)
with xi and yi being the nodes x and y coordinates. Using this heuristic A* will prioritize
searching adjacent notes that are close to the goal instead of just searching them all in
random order.
8.2.1
A*-algorithm
Here the basics of the A* will be described as it has been implemented (Swift)[Swi17].
The A* uses 3 parameters for each node to describe their quality. The parameters are:
• G which is the total distance from the starting node to the current node. This
distance is the traversed distance from going from node to node until the current
node is reached and is not the direct Euler distance.
• H is the heuristic described above so it is an estimate of the distance from the
current node to the goal.
• F is the total cost of the current node: F = G + H
Given a starting node and a goal node we start by creating two lists. One for nodes
we need to visit, openList, and one for nodes that we have visited, closedList. It
starts by setting the starting node as currentNode and then adding it to closedList
and the search begins.
Letting the children being all nodes connected to currentNode, it is then checked
for each of the children:
• If they have been visited by seeing if they are in closedList. If the child has
been visited it is ignored.
• If the child is not in openList it is added to openList and G, H, and F are
calculated for the child. Also the parent node of the child is set to the currentNode.
==PAGE:70==
56
8 Path planning
• If the child is already in openList it is checked if the G of the child from openList
is smaller than the G that is found by using the currentNode as the parent. If an
improvement of G is found the G and F are updated and the parent of the child
is set to currentNode.
Once all children of the currentNode has been checked the node in openList with the
lowest F-score is set to the currentNode and the above points are checked for all the
children.
The search stops once the goal node is reached by adding it to closedList or alter-
natively if openList is empty so that no path could be found from start to the goal.
Once the algorithm is finished and the goal is reached the path can be found by
starting in the goal position and then going to the parents and continue moving to the
parents’ parents’ and so on until the starting position is reached. This list of nodes will
then be the path of waypoints that we need to follow to reach the goal safely. The whole
algorithm is written out in Algorithm 1 showing the pseudo code.
8.3
Following a path
Once the graph is constructed and a path has been found using A* we need to also follow
the path. The controller on the copter should not have a problem moving from waypoint
to waypoint because we have designed the path to be within the copters uncertainty
range.
To follow the path we send the position of the next waypoint as the target
position. This position will always be over an ArUco marker so once it is reached the
correct position of the copter can be found. So once the waypoint is reached we need to
stabilize the copter after possible position errors. Therefore we require that the drone is
within 10 cm of the waypoint position for 1.5 seconds. This will ensure that the copter
and the position estimated by the copter is correct and stable. After being close within
the waypoint for 1.5 the next waypoint will be set as the next target. This is repeated
until the goal is reached. Figure 8.4 shows a flow chart describing how this is handled
in the software.

CHAPTER 10
Conclusion
The objective of this project was to develop a solution for using a camera and visual
markers for localization and navigation of a hexacopter in an indoor environment. The
solution should function in connection with the current hardware setup and the low
level position controlling system that had previously been developed. Besides solving
the problem it was also required to get an idea of how well the copters flying capabilities
was prior to implementing this projects solution.
From investigating the flying capabilities of copter without the use of a global posi-
tion estimator it was evident that flying from pure model and IMU data was in no way
viable. There was simply too much uncertainty in the model that affected the perceived
motion of the copter heavily. Even short distances such as half a meter were simply not
possible to safely fly by the copter. Even staying in the same position was impossible.
From this it was very clear that to use the copter the global position scheme, be it laser
scanner or camera needed to send position estimates all the time.
After the investigation of the copters flying abilities the camera localizer needed to
be enhanced. To do this the ArUco marker position estimation was extended to work
with multiple markers distributed in an area each with a known position. Based on such
a map of markers global localization was no longer a problem and as long as markers
were visible flight in this space worked flawlessly.
The next problem to be tackled was the estimation of position when no markers were
present. To solve this estimation of the copters movement was done with a combination
of optical flow using Lucas-Kanade and estimation of a translation homography. This
was successfully implemented but it had problems with the matching of features and
in some cases bad feature tracking resulted in very big translation errors that would
effectively make the copter lost and manual override was needed. Ideas to improve this
will be discussed in the next section.
The localization problem has thus in theory been solved but in practice some work is
still needed for it to be robust enough to rely on.
Navigation of the drone however has been successfully implemented using a graph
based approach for the creation of maps based on the information given about the vi-
sual markers in the area. The graph can be created and updated based on the markers
visibility. Also searching the graph to get a route to a desired marker is working and the

copter can follow and complete such a route with no problem given that the localization
works. As the localization is a separate unit we can conclude that the path planning
and following part of the problem works satisfactory.
Looking at the overall problem a solution has been found but unfortunately it is not
robust enough to be used in its current state. The code that has been developed in this
project is attached but an overview of the different files is given in Appendix C. In the

10.1
Future work
From above it is clear that the future work of this project lies in the localization part.
Here the robustness of the optical flow and translation estimation need to be improved
even further. To improve this a two step approach is suggested. The first part consists of
making sure that the feature correspondence between the two images is correct. Another
part is making sure that the estimated homography is good and realistic.
10.1.1
Improving feature tracking
If the features are not tracked correctly the estimated translation is also wrong. One
approach to improving this is doing a backwards tracking. This is done by first tracking
normally e.g. forward in frames from frame[i] to frame[i+1]. After doing this tracking
the positions of the features in frame[i+1] they are now tracked backwards to frame[i].
If the original feature position in frame[i] corresponds to the position estimated back-
wards we can assume that the feature was tracked correctly. This of course increases
the computational load so this also needs to be taken into consideration.
Another approach could be to use feature matching instead such as a SIFT(scale-invariant
feature transform) features this might provide a more robust matching but again this
would come at a computational cost.
Both of these solutions should be tested but as the camera localizer now only runs at
around 10Hz it is not desirable to make it even slower.
10.1.2
Improving homography
For the homography part of the localizer we can also improve on the robustness. Firstly
we could consider if the estimated movements were likely. Say if the homography says
that the copter has moved 0.5m between two frames this is very unlikely as we would not
want the drone flying at these speeds. So if such an estimate is given it could be skipped
or handled in some way so that we do not use bad estimates. How unrealistic estimates
==PAGE:81==
10.1 Future work
67
should be handled is not obvious but looking here could help solve this problem.
Another approach often used for robustness is the use of random sample consensus
(RANSAC) estimation. Using RANSAC we will most likely get a better estimate and
only use the feature correspondences that are relevant so that wrongly tracked features
will be ignored. This will add another layer on the uncertainty in the quality of the
feature tracker. Using RANSAC it is of course needed to have a way of testing if a
feature is an inlier or an outlier but as the homography relates the feature in the two
frames it should be possible to find a measurement of the validity of the homography
for the given features.
10.1.3
Improving IMU based flight
This last section is a method for better IMU performance I was made aware of late
in the project, but it is basically to use the Kalman filter to estimate the bias of the
accelerometer and roll and pitch angles so that the quality of position updates improve
when accounting for the bias. Getting this running at all time on the PixHawk could
potentially make the need for the global position estimate smaller so that reliable flight
might be possible even with only updates at 1-5Hz. This would make room for heavier
computations on the OBC when performing camera localization or other tasks.

tant variables are declared. This is also where all the threads needed are run from, this
is the camera localizer thread and the thread that handles the logging of data. This is
also where the mission is chosen based on the users choice when the program is run. The
user can specify which mission to run by adding an integer when calling the program.
Calling './autnav.c 03' will for example call mission 3.
C.2
Mission handling
aruco_mission_list.h
• This file has a list containing an enumerate with all missions and their correspond-
ing integer.
aruco_missions.cpp & .hpp & .h
• The aruco_missions.cpp files contains all the mission functions that are called
from the main program. There are functions for many different missions such as fly-
ing in a square or testing the camera or path planning. Also the scheme for follow-
ing a path as described in section 8.3. aruco_missions.h and aruco_missions.hpp
are the header files where the .h file is used when it is necessary to use the missions
in a C program.

C Code overview
C.3
Utility
aruco_map.cpp & .h
• The aruco_map files contain the functionality of saving ArUco markers positions.
The functionality for reading the .xml file and decoding the information and stor-
ing it in a map object from where the markers’ positions can be extracted. The
map is the implementation of the ArUco maps from chapter 3.
aruco_pose.cpp & .h
• These files are the ones for calculating the position of the copter from an image
based on the ArUco markers in the image. To find the positions in the images
and solving the n point perspective it makes use of the computer vision software
OpenCV. In particular OpenCV’s ArUco library is used. The things implemented
here are the position estimations from chaper 3.
camera_localizer.cpp & .hpp & .h
• The camera localizer implements the camera localizer algorithm. Besides the al-
gorithm these files also contain the implementation of the optical flow and mo-
tion estimation. The camera localizer makes use of the computer vision software
OpenCV. Besides all the matrix and vector calculation tools here the Shi-Tomasi
feature detector and Lucas-Kanade feature tracker is also found in the OpenCV
software. The camera localizer contains the implementation of the subjects from
chapter 5, 6, and 7.
path_planner.cpp & .h
• These files have the functionality for doing the path planning. The path planning
is handled by having a path planning object where the graph is stored. The graph
can be updated by sending setting a new flying height to account for the new
visibility radius. The graph is created based on the map aruco_map. The path
planner also has the functions for doing a shortest path searching and returning
the path. These files are he implementation of the subjects of chapter 8.
rotations.cpp & .h
• The rotations files contains functions that can create a rotation matrix from
three Euler angles. It can also give the Euler angles if given a rotation matrix.
