Summary
Time-predictability is an essential property of software running on real-time systems.
The single-path approach to code generation produces programs that have a single
execution path which ensures execution-time is independent of input data.
This
greatly simplifies worst case execution-time analysis.
The Patmos architecture is specifically designed for real-time systems. It is time-
predictable, with a fully predicated instruction set and a dual-issue pipeline. Previous
work has implemented single-path code generation for the Patmos compiler. However,
the produced code does not make use of the second issue slot available in the Patmos
pipeline.
In this report we detail the steps taken to ready the compiler for future research
into how to best make single-path code use the second issue slot. We first detail
how single-path code can be tested automatically, then how refactoring can be used
to get familiar with a new codebase—in this case, the Patmos compiler. We finish
by describing how a bundling pass can be added to the single-path transformation,
and how the compiler is updated to support this. A simple technique is designed
and implemented that produces bundled instruction, showing how it can be done and
that the compiler is ready for experimentation into different techniques.

CHAPTER 1
Introduction
Real-time systems are used in domains where the running time of a device or appli-
cation is of critical importance. A system is real-time if it:
Controls an environment by receiving data, processing them, and returning
the results sufficiently quickly to affect the environment at that time [1].
(James Martin, #4 most influential person in comp. tech. (1992)[2])
As such, the total correctness of a real-time system is not only dependent on the
logical correctness of its response, but also on the timing of said response. The time,
before which a response is expected, is called a deadline. We can classify real-time
systems by the consequences of a response missing the deadline.
A soft real-time system can handle missed deadlines without critically failing.
Though the quality of the service provided may decrease, the system will be able to
either recover from it or continue working regardless.
Most people interact with such devices on a daily basis. An easy example would
be any video or audio playback device like a television or laptop. In these cases each
frame of a video must be displayed for a specific interval after which it is swapped
for a new one. Ideally, the device will be able to load the frames and have them
ready before they are needed.
However, sometimes the device is unable to meet
this periodic deadline of 1 frame every ~42 milliseconds (for a 24 frames-per-second
viewing experience). E.g. this could be caused by sudden network issues prohibiting
the device from downloading the frames quickly enough.
Luckily, there are many ways to resolve such issues: Playback could be paused
until the underlying issue resolves itself, after which it is automatically resumed; if a
live event is being displayed, playback can resume at the most recent frame, ignoring
any missed frames; or, in the worst case, playback could be abandoned completely
and the user asked to resolve the network issue before trying again. Regardless of
the underlying issue and how it’s handling of it, no severe or permanent damage has
been done. The user will surely be frustrated with the quality of the video, but will
either choose to wait it out or go do something else.
Hard real-time systems cannot afford for a deadline to be missed, the result of
which could be significant commercial losses and/or unacceptable bodily harm. Tra-
ditionally, airplanes are used as an example for real-time systems. Many modern
airplanes use a system called fly-by-wire, which replaces the physical cables between

on the flight controls is sent to a computer, which determines how to move the control

might lose control of the plane. At worst, this could cause the deaths of passengers;

Real-time systems are increasingly moving into our daily lives.
This is easily
noticeable in cars, where an increasing number of automated systems are taking over
different driving tasks. These systems are often meant to increase safety, and must
therefore be incredibly reliable. Take the Autonomous Emergency Braking system,
which will trigger the brakes if it senses an obstacle in front of the car. A program must
analyze the inputs from the sensors, identify any obstacle, and determine whether
emergency breaking is required. This must be done quickly, with enough time to
spare, such that the brakes are able to bring the car to a standstill—before hitting
the obstacle. If the analysis is not done it time, the resulting crash could harm the
passengers of the car and/or bystanders outside of it.
In this project we focus on the software components of real-time systems. As with
the examples given so far, the execution time of the software running on a system
is of paramount importance to the overall system. We will also be focusing mainly
on hard real-time systems, where we want to be able to guarantee that a program
always meets its deadlines.

1.1.1
WCET analysis
For real-time systems, we want to be able to make a guarantee about the execution
time of a program. In particular, we are interested in the worst-case execution time
(WCET), i.e. the maximum amount of time the program will need to finish executing.
WCET analysis looks at a program and tries to produce an estimate of the WCET.
To do that, it will have to take into account all possible inputs and environments and
how they affect the execution time. It will then produce an estimate that puts an
upper bound on the WCET, which we can use as our guarantee.
Because of the high level of variability in inputs and environment, WCET analysis
can be difficult to do accurately (i.e. having the estimated WCET be close to the
actual WCET) and computationally hard to run.
The first part of accurately estimating WCET is to analyze the program statically,
i.e. without running it, and finding the longest running path through the code. Differ-
ent analysis techniques are used, to increase the accuracy by minimizing the number
of possible paths through the code. Thereby we hope to eliminate the longest paths
and reduce the WCET. E.g, by analyzing the possible value of variables, the analysis
might be able to determine that a given path can’t be taken.
Accuracy is also achieved by modeling the environment in which the program will
run. This includes modeling the memory hierarchy, caches, and other aspects of the
hardware running the program. When caches are used, the default assumption the
analysis must make is that any memory access will result in a cache miss. By modeling
the cache and tracking what might be in it at a given point in the program, the
analysis can sometimes deduce that a given memory access will not miss. Therefore,
the estimate does not have to account for a cache miss, and the overall WCET estimate
can be reduced [3].
However, not all things can be modeled. E.g. unconstrained preemptive multi-
tasking is nigh on impossible to predict, and therefore is often not taken into account
when doing the analysis. Instead, specialized scheduling techniques are used, which
might depend of the WCET. Earliest Deadline First (EDF) scheduling can be used
to ensure that all programs finish before their deadline [4]. This is done by always
scheduling the task that has the earliest deadline, using preemption to stop the exe-
cution of tasks with later deadlines. Given enough computing power, this scheduling
technique guarantees all tasks get to finish before their deadlines. This is possible by

using a WCET estimate that doesn’t account for preemption and theoretically check-
ing that all deadlines can be met. If the check fails, the WCET of the tasks must
be reduced; either by changing the code to be faster or by increasing the computing

As program size and analysis accuracy increase the difficulty of running the anal-
ysis increases exponentially. This is especially caused by the fact that the number
of paths through the code increase exponentially with the number of control flow
alternatives [5]. In practice, this means that running the WCET analysis might take
hours or days for industry systems [6]. This make the usage of the analysis intractable
and seldom used in industry [7][8].
The topic of this report surround the Patmos instruction set architecture and
single-path code. Patmos is specifically designed for use in real-time systems, with
a design that makes it highly predictable to ensure WCET analysis is both easier
and more accurate. Additionally, the architecture enables the use of single-path code,
which eliminates input-data-dependent control flow, making running analyses much
easier and even trivial in some situations.
1.2
Patmos
Traditional hardware architectures, like x86 or ARM, are designed for general-purpose
computing: Personal computers, servers, or other applications without real-time re-
quirements. To maximize performance, these architectures try to minimize Average-
Case Execution Time (ACET) such that, in general, performance is good. This works
well for the intended purpose but makes these architectures difficult to use in real-time
systems.
Many of the techniques used to reduce ACET make the hardware less predictable,
making WCET harder and less accurate. The Patmos Instruction Set Architecture
(ISA) is specifically designed for real-time systems [9]. Therefore, the design tries
to reduce the WCET instead of ACET and makes the processor as predictable as
possible.
Patmos has a 32-bit, Reduced Instruction Set Computer (RISC), statically sched-
uled, Very Long Instruction-Word architecture with an exposed, dual-issue pipeline.
It uses split caches and a fully predicated instruction set. We will go through each of
these features and see how they make WCET analysis easier and more accurate.
1.2.1
RISC
Contrary to Complex Instruction Set Computers (CISC), RISC architectures try to
minimize the number of available instructions and makes them as simple as possible.
Though not specifically intended for it, this makes such architectures perform well for
real-time applications [10]. The simplicity makes WCET analysis easier since each
instruction is simpler to predict, with fewer behaviors and interactions with the rest
of the system.

First, it loads from memory the source and destination addresses of the data byte to

depending on a flag, it either increments or decrements the source and destination
addresses [11]. In a RISC architecture, such an instruction would not be available.
Instead, a sequence of simple instructions would provide the same functionality. In

1.2.2
Statically scheduled, exposed pipeline
Pipelining is a method of increasing the throughput of a processor by splitting the
execution of each instruction into stages, each of which takes one cycle. Since less
work is being done in each cycle, the clock frequency can be increased.
In each cycle, instructions move to the next stage, and a new one is issued in the
first stage. Therefore, the number of instructions being executed at the same time
can be as high as the number of stages. The same number of instructions are issued,
and finish executing, each cycle, so the increased throughput only comes from the
higher clock frequency.
The Patmos processor has a 5-stage pipeline:
1. Fetch: Gets the next instruction(s) to be executed.
2. Decode: Analyzes the instruction to figure out what the next stages need to
do. Also reads the values of the registers that the instructions might need.
3. Execute: Perform calculations. Includes arithmetic operations, like addition,
but also calculations of effective addresses, e.g. for loading of values from mem-
ory.

4. Memory: Performs memory accesses to addresses calculated in the previous

This setup isn’t uncommon for RISC architectures and doesn’t provide any real-
time benefit in and of itself. However, Patmos exposes various delays caused by using
such a pipeline, allowing WCET to accurately predict its behavior. These delays are
the results of data- and control-hazards any pipeline must tackle to be efficient.
Data-hazards materialize when one instruction needs the result of a previous in-
struction for its execution. Say we have the instruction add $r3 = $r1, $r2 imme-
diately followed by add $r5 = $r3, $r4. The result of the first add is calculated
in the Execute stage and only written back to the $r3 register after the last stage.
When the second add reaches the decode stage, where it reads the value of its two
operand registers, the value of $r3 would not have been updated yet, as the first add
is still in the Memory stage. The second add would therefore have the wrong value
to sum up. The solution to this problem is easy enough: we send the results of the
Execute stage back to the beginning of the same stage, such that the next instruction
can use it if need be. This is called Pipeline Forwarding.
Other data hazard aren’t as easy to solve. Say we have a load instead of the first
add: lwc $r3 = [$r1]. The result of the load will only be ready after the Memory
stage. However, at that point the add would have already finished its Execute stage
without the loaded value. Pipeline forwarding cannot solve this problem. The only
solution is to stall the add for one cycle, such that the result of the load can be
forwarded from the Memory to the Execute stage.
For traditional processors this can be optimized by only stalling an instruction if
it actually needs the loaded value. However, since the worst case will stall, Patmos
would not benefit from this. Instead, it exposes this pipeline problem in the ISA as
an instruction delay. The ISA specifically states that the result of a load cannot be
used in the instruction immediately following it. Instead, either a different instruction
must be inserted (that doesn’t depend of the loaded value) or a no-operation (nop)
must be used.
Returning to the code in Listing 1.2, we can see how the store instruction on line 5
depends on the result of the load on line 4. Therefore, a nop has been added between
them, such that the delay is respected. Alternatively, we could have taken the add on
line 6 and moved it up to the nop’s position to make better use of that cycle. We can
do this because the add in not dependent on either of the previous two instructions.
Control-hazards are similar to data-hazards, except that they are caused by control-
flow-affecting instructions: The effective address for the instruction to execute after
a branch is only ready after the Execute stage. Patmos’ solution is again to expose
this as a delay. For a simple conditional branch, br, the delay is 2 cycles; meaning
the two instruction following the br are also executed before the branch takes effect.
The longer delay is caused by the address of the branch being needed in the Fetch
stage, but is only ready two stages latter.

By using delays in the ISA, Patmos solves the hazards introduced by pipelining
in a way that improves the accuracy of WCET analysis, without introducing much
complexity. Traditional ISAs like x86 and ARM don’t do this, opting for solutions
that decrease the number of stalls needed on average, but either don’t affect the worst
case or make the analysis harder.
An often-used technique for managing control-hazards is branch prediction. To
avoid having to stall the pipeline, the address of the branch is predicted before it
reaches the Execute stage. The prediction is then used to speculatively execute the
next instruction based on the prediction. However, if it is ultimately proven wrong,
the instructions that are in the pipeline are thrown out (flushed), and the correct
ones executed instead.
With a good predictor most branches are predicted correctly. For example, using
a Branch Target Buffer, which remembers both the result and the target address of
a branch instruction that has been executed before, more than 95% of all branches
are predicted correctly [12]. However, it is very difficult to perform accurate model-
ing of branch prediction. Therefore, WCET analysis would often have to assume a
prediction is wrong, which makes the over-all estimated execution time much worse
that the actual.
Another heavily used technique for managing data-hazards is Out-of-Order Execu-
tion, also called Dynamic Scheduling. Looking again at Listing 1.2, we have already
established that the add on line 6 can safely be moved up to replace the nop on
line 4. Modern processors, without exposed pipelines, can analyze the instructions
before they are issued, and detect where there are data dependencies. They can then
reorder the execution of the instructions to maximize the usage of the pipeline. E.g.
they could automatically detect that the add is independent of the sbc on line 5 and
execute it immediately after the lbc, thereby eliminating a stall in the pipeline.
The complexity introduced by out-of-order execution greatly increases the diffi-
culty of doing WCET analysis [13]. Instead, Patmos uses static scheduling: Instruc-
tions are executed in the exact order they are issued by the compiler. The burden
of identifying data dependencies, and reordering instructions optimally, is therefore
put on the compiler instead of the hardware. However, this makes WCET analysis
simple, since the order of execution is known and unchanging.
1.2.3
VLIW
Regardless of how many pipeline stages are added, how well branch predication per-
forms, or how perfectly instructions are scheduled (statically or dynamically), at most
one instruction is issued and executed per cycle. This theoretical limit puts a ceiling
on throughput that cannot be exceeded without specialized techniques.
superscalar processors can issue multiple instructions each cycle from a sequen-
tial instruction stream. Additional functional units are present, like the Arithmetic
Logic Unit (ALU), such that multiple instructions can be executed simultaneously. A
scheduler chooses which instruction get to use each functional unit in an optimal or-
==PAGE:19==
1.2 Patmos
9
der. While out-of-order superscalar processor are not useful for real-time applications,
for the reasons stated before, in-order superscalars can be [14].
Patmos takes a slightly different approach. Instead of issuing from a stream of
single instructions, it issues from a stream of bundles, which can contain up to two
instructions. This is caled a Very Long Instruction-Word (VLIW) architecture: A
good way to achieve high performance as well as time-predictability [15].
The dual-issue pipeline issues one bundle each cycle and executes both instructions
simultaneously. Like superscalars, it has duplicate functional units to allow this. But
unlike superscalars, it doesn’t need to schedule the instructions, since use of the
functional units is decided by their position in the bundle, called slots. Conceptually,
the compiler schedules which instructions run on which functional units.
The two slots in a bundle are not equal, however. Only the first slot can issue
memory or control-flow instructions. This means lwc and br cannot be scheduled
by the compiler in the second slot. Otherwise, the most-used instructions can be
scheduled in both slots.
1.2.4
Caches
Caches complicate WCET analysis since ti can be difficult to predict what memory
regions are stored in the cache at a given point in the program. Accesses to caches
can also happen multiple times each cycle; once to fetch the next instruction and
potentially another as part of a load or store instruction. As such, accurate modeling
of the caches in a processor is critical to accurate WCET analysis.
Traditional processors have two caches: an Instruction Cache (I-cache) and a
Data cache (D-cache). Instructions are generally loaded sequentially from memory,
which is good for a cache, since a single cache-miss will load a group of instructions
simultaneously.
This predictability in memory access can also enable prefetching,
where the cache get instructions from main memory before they are actually needed.
Some prefetching techniques can achieve an accuracy of 99.5%; meaning only around
0.5% of instructions will miss in the I-cache [16]. Having a D-cache is essential for
instruction prefetching accuracy, since the cache used for instructions is not tainted
whenever some non-instruction data is accessed. Still, prefetching is difficult to model
for WCET analysis, and may even be detrimental, since it could increase WCET in
some cases [17].
Splitting a processor’s cache into I-cache and D-cache improves performance, be-
cause techniques can be used to reduce the miss-rate of each cache. Patmos takes this
idea to the extreme. It specializes the I-cache into a Method Cache (M-cache), and
splits the D-cache in two: a Stack Cache (S-cache) and D-cache for data that doesn’t
reside on the program stack.
Instead of a I-cache figuring out which instructions to fetch, a M-cache is meant
to cache whole methods (or functions) at once. The compiler is then tasked with
issuing instructions that prefetch each method before it is executed. E.g. the call
instruction implicitly will make the M-cache load the called method in its entirety.
==PAGE:20==
10

Of course, some methods are too large for the cache to contain. The compiler
must then split the method into blocks, and add cache-fetching instructions between
each block. The branch instruction brcf (branch with cache fill)can be used for this.
It will fill the cache with the block at its destination before doing the actual branch.
Using an M-cache therefore limits cache-misses to specific fetching instructions.
All other instructions can’t miss in the M-cache.
This simplifies WCET analysis
greatly, as caching can be ignored for all instructions except a select few. And, as an
added bonus, prefetching can be utilized in a way that doesn’t increase WCET [18].
To ease the modeling of which data is in the D-cache, Patmos uses the S-cache to
split data which resides on the program stack from all other data. Stack data is very
predictable:
• Liveliness: A method-local variable is only live when the method is called, and
is no longer used when the method returns.
• Location:
All method-local the variables reside in the stack frame of the
method.
If the method is executing, then its stack frame is on top of the
stack and directly accessible through the stack- and base-pointers.
• Size: Local variables must have a known size at compile time. By extension,
they will have the same size throughout a programs execution.
Because of this, Patmos allows the compiler to directly control the S-cache. It
can issue instructions to fill and spill the S-cache, such that a method’s stack frame
is always in the cache.
Access to the S-cache uses dedicated memory access instructions. The lws load
instruction is equivalent to the lwc instruction, except that it loads its value through
the S-cache (lwc always loads through the D-cache). Similarly to the M-cache, WCET
can be trivialized: All loads through the S-cache can be assumed to hit. The S-cache
therfore only misses when executing S-cache-control instructions like sens, which
ensures a stack frame is in the cache (potentially fetching it if not).

CHAPTER 2
Single-Path
As we have already covered in Chapter 1, as programs become larger, the WCET anal-
ysis becomes exponentially harder to run. The single-path code generation technique
is specifically designed to counter this [23]. It avoids complex control-flow structure
and produces code that allows for simple and accurate WCET analysis. This is done
by generating code whose behavior is independent of the input data by always follow-
ing the same—and only—execution path.
Having programs always follow a single execution paths greatly simplifies WCET
analysis since it doesn’t have to do path analysis—the program always takes the same
path. This means complex analysis of the possible control flow can be omitted. In
some cases, WCET analysis can be as trivial as simply measuring the execution time
of the program with any arbitrary input. This also means complex modeling of the
hardware can be omitted. Only with the use of caches is measurement-based analysis
not sufficient as the caches need to be modeled.
In this chapter we will present te single-path approach to code generation. We will
cover how a WCET analyzable program can be converted to use single-path code and
how an automatic conversion can be used. We will then present an algorithm that
can do the conversion and how it is implemented for Patmos’ compiler. All of this
will ignore Patmos’ second issue slot, since that is how the compiler was implemented
before this project started. In Chapter 5 we will then describe how the algorithms
and implementations can be updated to use the second issue slot.
2.1
Basic Idea
To achieve a single execution path throughout the program, the single-path approach
needs to remove any variability to execution time. There are three structural elements
that cause this variability and we will go through each of them and show how they
can be transformed to no longer be variable.
2.1.1
Conditionals
As we have already covered in Chapter 1, if/else statements introduce variable ex-
ecution times because each path can have a different length.
The solution is, as
covered before, using predication to conditionally run both paths. This also called

of if-statements and then if-converting them all.
2.1.2
Loops
Loops of any kind may vary in the number of iterations they execute. The variation
is input data dependent and must be eliminated when converting to single-path code.
We mentioned before that only WCET analyzable code can be converted to single-
path. To be WCET analyzable, a program must have an upper bound on each loop,
that specifies the maximum number of times the loop will iterate. This requirement
is needed, because otherwise the analysis cannot always determine an upper iteration
bound. In turn, an upper bound for its execution time cannot be achieved. This is
a theoretically impossible problem to solve, as it would require the analysis to solve
the halting problem [23]. It states that it’s impossible to determine, for any arbitrary
program and input, whether that program ever finishes executing (i.e. halts) [24].
The halting problem also makes recursive functions intractable in real-time systems.
In essence, recursion can be thought of as equivalent to looping and therefore can’t
always be shown to terminate. As such, the single-path conversion cannot deal with
recursion.1
To remove the variation in iteration count between different runs of the same loop,
we make the loops always iterate the maximum number of times. On its own, this
would change the behavior of the program. So, we also predicate all the instruction
in the body of the loop. By having the predicate hold the value of the condition of
the loop, any superfluous iterations will make the instructions in the body have a
false predicate—and therefore have no effect.
In Listing 2.1 we can see the effect of converting a traditional loop into single-path.
In (a) we see a loop that iterates according to the value of $r1. The loop begins by
incrementing the loop counter $r2 and then evaluating whether it has become greater
than $r1. If so, the required number of iterations of the body (replaced here by ”...”)
has been achieved and the loop is exited using a branch. If additional iterations are
needed, the branch isn’t taken, and at the end of the loop we simply start over. This
loop will iterate the exact number of times needed by any given run of the program.
In (b) we see the same loop, except that it uses the single-path approach. The
maximum possible iteration count is given in $r3 and is 10. Initially, the loop starts
out equivalent to the traditional approach. However, instead of using a branch to
skip the body of the loop, the instructions are predicated with the condition. If $r2
isn’t greater than $r1, then the loop body is execution, otherwise predication ensures
it does nothing. At the end, we compare $r2 to $r3. If they are equal, the maximum
number of iterations has been reached and the loop is exited (by skipping the branch).
Otherwise, we still need more iterations and we branch to the beginning. Therefore,

Formalized Transformation
We can define a set of rules that formally describe the transformation to single-path
code [25]. We give these rules using a high-level language containing: single or se-
quences of statements, conditionals, loops, and function definition and calls. In Ta-
ble 2.1 we see the rules; each row has a language construct on the left and how it
should be transformed on the right. For each transformation, the variable σ defines
the precondition under which the resulting single-path code should be enabled. δ is
a variable that enables the transformation to create unique identifiers and is a simple
counter.
It is easiest to start with function definitions. Whenever the original code has
defines a function (f), we need to create a single-path version of the function. To do
so we define a new function (fsp) with an additional argument (cond) which specifies
whether the function should be enabled or not when called. We then transform its
body to be predicated on that argument.2
For every conditional we start by inserting code that calculates the condition and
puts the result in a guard (guardδ). We then transform each alternative path such
that they are predicated on the guard (or its negation) and on whether the scope
enclosing the conditional was enabled.3
This ensures that if it is disabled—as a
whole—then so are both alternatives.
We transform all types of loops (here given as a while-loops) into for-loops that
iterate the maximum number of times (N). The body is then transformed such that
it is predicated on the condition of the loop and on whether the loop is enable or not.
Notice that the resulting loop is not predicated at all. This ensured that it always
iterates N times. If scope enclosing the loop was disabled, then only the loops body
is disabled, not the iteration.
For the simpler constructions, a simple statement and a sequence of statements
are simply predicated on whether their enclosing scope is enabled. For function calls,
the call itself is not predicated, but the called function is passed whether the calling
scope is enabled, and will enable or disable its body accordingly.

Algorithm
To implement the single-path transformation, we need an algorithm that applies
the rules on a control-flow graph (CFG), which is often the data structure used to
represent programs within a compiler. The algorithm is given by [5]. However, we
will describe it informally as done in [22].
2.3.1
Definitions
Basic Block (BB): A straight-line sequence of instructions, where execution always
start with the first instruction and only branches on the last.
==PAGE:30==

CFG: A directed graph where the nodes are BBs and the edges model how the
control flows from one block to the next. A BB that ends with a branch will
have multiple out-edges in the CFG; one for each possible target.4
Entry Node: The single—and only—node in a (sub-) CFG where execution starts.
Exit Node: The single—and only—node in a (sub-) CFG where execution ends.5
Dominates: Node x dominates node y if all paths to y from the entry node go
through x.
Postdominates: Node x postdominates node y if all paths from y to the exit node
go through x.
Loop: A strongly connected set of nodes (every node in the loop is reachable from
every other node in the loop).6
Header: Every loop has a header, which is its entry node.7
We also associate a
header with every node. A node’s header (header(x)) is the header of the in-
nermost loop the node is contained in. We also treat a whole function as a
pseudo-loop with its own pseudo-header, which is the function’s entry. There-
fore, all nodes in a function have a header, except the function’s entry node.
Back Edge: A edge who’s source node is in the loop and destination node is the
loop’s header.
Exit Edge: Has a source node in the loop but destination node outside it. I.e. the
edge exits the loop.
Dual Edge: Given a node with two outgoing edges, the dual edge of one of the edges
is the other edge.
Latch: The source node of a back edge.
Natural Loop: Has a header and a single back edge. The natural loop of a back
edge (l, h) is h (the header) plus all the nodes that can reach l (the latch)
without first going through h. If two natural loops have the same header, we
treat them as one loop that is identified by the header node.
If they don’t
share a header, they are either disjoint (don’t have any common nodes) or one
is nested in the other.

Reducible CFG: A CFG is reducible if all its cycles have a back edge that is asso-

Forward CFG (FCFG): An acyclic CFG that is the result of removing all back

Control Dependent: In a CFG, given the nodes x, y, and z, x is control dependent

going through either. We also say that x is control dependent on the edge (y,z).

Equivalence Class: Two nodes are in the same equivalence class if they are control
dependent on the same set of edges. All BBs in the same equivalence class will
have the same predicate condition and therefore can share a predicate register.
This is because even if there are branches between two nodes in the same class,
control is guaranteed to merge back at a node in the same equivalence class.
2.3.2
Transformation
We start by analyzing the whole CFG and identifying all the loops in it to construct
a loop tree, which shows which nodes are part of which loops. The root of the tree is
always the entry node of the function. The root’s children are either the headers of
any loops in the function, or nodes that aren’t in any loop. Likewise, for each loop in
the function, its header has children which are either the headers of nested loops, or
nodes that are not part of another loop. A loop header and its children are a scope.
The root node and its children are the root scope.

2.4.1

The LLVM Intermediate Representation (LLVM-IR) is the heart and soul of the

this setup we call clang a frontend and llc a backend.
Using llc, any language that can compile to LLVM-IR can be further compiled
to all ISAs supported by llc.

2.4.2
Single-Path Passes
Facilitating the generation of single-path code is done as a set of passes over the
LLVM-IR and MSIR. The actual single-path transformation is done as the last pass
on the MSIR. All previous passes prepare the code for the transformation, ensuring
that all needed properties are enforced.
The first set of passes are done on the LLVM-IR, where we ensure that our re-
quirements for its CFG are met: that there are only a single entry and exit node and
that all switch-statement are converted to conditionals. LLVM has built in passes
that can tranform the LLVM-IR to adhere to those requirements.
Next comes the two passes SPClone and SPMark. Together, these passes ensure
that every function that needs a single-path version are cloned such that the final
binary will have a single-path and a non-single-path version of the function. SPClone
works on the LLVM-IR where it looks for all functions that might be called from a
single-path context and clones them. SPMark then works on the MSIR, taking all
the clones and updating all their function calls such that they call each other instead
of the non-single-path versions.
The heavy lifting for the single-path transformation happens in the SPReduce pass.
Here the CFG is analyzed, loop trees and FCFG are created, equivalence classes are
define and allocated to the Patmos predicated registers (using a specialized predicate
register allocation), and the final transformation is done.
At this point the single-path code has been created, but the Patmos-specific laten-
cies for each instruction are have not been accounted for. Therefore, a scheduling pass
is run, which reorders instructions and adds nops where needed such that the code,
e.g., doesn’t use a loaded value in the bundle immediately following the load.9 Note
that the scheduling pass is not unique to the single-path approach, but a standard
pass used in all compilers. The specific pass used here is implemented for producing
traditional code for Patmos and reused for the single-path approach.

immediately following it.

CHAPTER 3
Testing
For any sufficiently complex program, significant effort must be put in ensuring that
it works as intended.
Compilers in general quickly become complex, and testing
must therefore be part of any non-trivial development effort. Having the tests being
automated is also a must, as the sheer number of tests required make manually
running them impractical.
As stated before Patmos’ compiler is built on the LLVM compiler, and has an
automated testing setup for the generation of traditional code. However, there is
no automated setup for testing of single-path code. Therefore, before initiating any
development effort towards bundling instructions, we design and implement a testing
setup, such that we can ensure that the existing features aren’t lost, and that the
new features we want to implement work as expected.
In this chapter we will describe the testing setup that was designed and imple-
mented for the single-path code generation of the compiler. We first describe the
requirements the setup must adhere to and then present the three types of tests that
will be used. The setup is inspired by LLVM’s own setup for testing code generation
for other architectures [28] and uses its existing infrastructure where possible.
3.1
Requirements
We first specify three functional requirements, which are features different tests must
have to enable us to efficiently develop on and verify the implementation of the
compiler:
Correctness: We need to be able to validate that correct programs are successfully
compiled and, when executed, produce the correct result. This is the goal of
any compiler and ours is no different. If a compiled program doesn’t behave as
the semantics of the programming language it was written in specify, nothing
else matters, as the program becomes useless. Therefore, our testing setup must
be able to verify that, for a given correct program, the compiler can compile
it, and that running the resulting binary produces a result equivalent to some
reference value.
Unlike other compilers, our compiler also needs to produce single-path code.
Not all the code in the binary should be single-path, only the functions that are
==PAGE:38==
28
3 Testing
declared as single-path and any others they call. Therefore, our testing setup
must be able to distinguish between binaries with traditional and single-path
code, and verify that single-path code is correctly used in the needed places.
Maintenance: As users exercise the compiler, they are sure to uncover bugs that
have not been caught by the testing setup. Alternatively, a developer might
also uncover problematic cases that he thinks should be added to the testing
setup—even though they are not currently manifesting a bug. Therefore, the
setup should enable targeted tests that isolate a given bug or feature as much
as possible, such that finding errors is as easy as possible and bugs aren’t rein-
troduced by accident.
Productivity: We want our testing setup to be a development tool and aide in the
implementation of correct functionality. Be it test-driven development or other
techniques, we want to the setup to be a boon to the developer and make him
more productive for both maintenance and the implementation of new features.
Additionally, finding the cause of errors should also be as easy as possible and
aided by the test setup.
We also have two general requirements for the setup that the design will strive
towards, but can ignore where necessary:
Low Coupling: The setup should be as decoupled from the implementation of the
compiler as possible.
This will minimize the need for revising tests as the
compiler is updated, which could compromise the tests by deviating from their
original intent.
Usability: Running the tests and interpreting the result should be as easy as possible
and preferably foolproof. We want to ensure that failing tests aren’t overlooked
either because the tests weren’t run correctly or the test report was misinter-
preted as a success.
Guided by these requirements, we have designed a testing setup that divides tests
into three categories: Unit Tests, Regression Tests, and Program Tests. Each category
of tests has its own features and addresses a subset of our requirements. They are
also designed to integrate easily into the existing LLVM testing setup, which they are
heavily inspired by—we even use the same names for the categories as LLVM does
[28].

3.3
Regression Tests
The second category of tests in our setup consists of small programs that are custom-
made to exercise a specific feature or bug.
When running these tests, the setup
ensures the programs compile, run, and use single-path code.
To ensure we can minimize any variables that aren’t relevant to each test case,
we will need to simplify the tests as much as possible. We do this by writing each
regression test in LLVM-IR instead of C. We then use llc directly to compile the
program, eliminating the use of clang, which allows us to better control how the code
looks right before llc begins its work. If we had used clang, we wouldn’t be able to
control how it converts a C program to LLVM-IR, which could make it very difficult
for us to create tests that are specific to how the LLVM-IR is structured.
After a successful compilation, we use the pasim simulator—which simulates how
an executable will run on the Patmos processor—to ensure that running the program
produces the expected result. For each test cases, we provide a set of input/output
pairs: We then simply test that for a given input, its expected output is returned
when the program is run through pasim.
3.3.1
Testing for single-path
It can be difficult to tell whether a program is using single-path code. Simply running
it and inspecting its output doesn’t say anything about how the code is working. We
therefore need a way to ensure that the program is using single-path code. But how
do we do this?
A naive approach would be to simply inspect the program’s assembly code, and
ensure its single-path by comparing it to some assembly we create by hand (and
therefore know is single-path). While this could work, it is not very flexible. First,
this is highly implementation dependent, as we are forcing the compiler to produce
==PAGE:42==
32
3 Testing
code that exactly matches ours. We would like to give the compiler some leeway, e.g.
by giving it freedom to choose which registers should be use for a given instruction
sequence. This is useful if we, for example, want to optimize the produced code. This
assembly inspection, we would need to update the test to reflect this optimization.
What we do instead is test for what makes single-path code unique: it always
runs the same sequence of instructions, regardless of input. An ideal solution would
be for pasim to provide us with the instruction sequence of the program it runs. We
could then run the program with the different inputs, and ensure that all instruction
sequences are identical.
Sadly, pasim cannot produce such an instruction sequence. Instead, it can provide
us with the number of cycles the program used, a table of how many times each
instruction type was executed, and the number of times each function is called. We
use these as a surrogate for the instruction sequence; We validate that each run of
the programs uses the same number of cycles, instructions, and function calls. Since
pasim accurately models caches. However, we configure it to use an ideal data cache,
which never misses. This eliminates variance in cycle count caused by cache-misses
for different input data.
This solution isn’t ideal. We cannot guarantee that no non-single-path program
fails this validation. It could be possible that such a program uses the same cycles and
instructions for the all the specific inputs. However, it should be extremely unlikely
that a non-single-path program uses the same number of cycles, instructions, and
functions calls when given different inputs. In exchange for the uncertainty, we get a
setup that is flexible. It allows the compiler to produce any single-path code it wants
with only a negligible chance of false positive test results.

3.3.3
Debugging
The traditional meaning of debugging covers the general act of resolving errors and
bugs in programs. When we use it, we more specifically talk about the act of in-
specting the internal workings of a program—our compiler—while it is working. We
could be looking at the contents of an internal or following the execution path to see
whether there are any deviations from our expectation.
One way to do debugging is through execution-time inspection by using a debugger
tool. The debugger will insert itself into a program, and manage its execution. The
developer can then step through the program’s execution, and can inspect the internal
memory of it at each relevant point. This can be very powerful for investigating errors,
as it allows us to see exactly what the compiler is doing, as it is doing it.
However, a bit deficiency of debuggers, is their lack of support for non-standard
data structures.
This means we cannot have a fine-grained look at many part of
==PAGE:45==

35
the compiler. As an example, LLVM’s classes that comprise the LLVM-IR can’t be
inspected to see how exactly the code evolves within the compiler.
As different method we can used is print debugging. Instead of inspecting the
compiler as it runs, we add debugging statements into the compiler’s code, such that
it prints any relevant information we want to a file. When execution ends, the file
will contain all the output of the debugging statements, which can then be analyzed
to find errors.
This type of debugging can be a better alternative to using debuggers, as it al-
lows us full control of what information to output and how it should look. E.g. to
inspect the LLVM-IR, we simply create a debug statement at the relevant point in
the compiler that prints the IR in a human-readable format.
Our test setup supports print debugging for regression tests. By default, when the
tests are run, debugging output is automatically saved in a file that can be inspected
immediately. This should make it seamless for the developer to run the tests, see
the error report, and inspect the corresponding debug file for the failure source. Of
course, the debug file will only include information that has been specifically output
by a debug statement. If additional information is needed, the developer will need to
add a debug statement to the compiler and run the tests again.
The regression tests focus on our maintenance requirement, allowing us to create
fine-grained tests that ensure any fixed bug will not be reintroduced and isolated
features can be tested directly. Being fully automated, the tests increase developer’s
productivity, by providing clear and plentiful information on errors. They are almost
trivial to run, practically impossible to run incorrectly, and clearly highlight failures.
Being written in LLVM-IR and don’t inspect assembly directly, they are also fully
decoupled from the compiler implementation.
Lastly, the nature of the tests also means they validate correctness; any regression
test failure must means there is an error in the compiler. However, since each test
should be as minimal as possible, they don’t validate that each feature works well
when used in conjunction with others. They also don’t provide confidence in the
compiler being able to compile real-world programs, as they all are doctored for the
specific feature or bug they test. Therefore, we need the last test category to validate
that the compiler can correctly compile real-world programs.
3.4
Program Tests
Similar in idea to the regression tests, this category of tests validate the correctness
of compiler. However, to ensure real-world conditions, the tests should compile self-
contained programs that solve some real task. Additionally, the programs should be
written in a high-level language supported by clang, like C or C++. Otherwise, the
validation process should be similar to the regression tests: compile, run, and ensure
correct output and use of single-path code is used. A broad set of test programs will
ensure that the compiler is exercised in a way similar to the real-world and thereby
giving us the confidence that it works as intended.

3 Testing
This type of tests are usually called a test suite and is a staple of any mature
compiler. LLVM is no exception, with its test suite contained in a different code
base [31]. The Patmos project also has a test suite for when the compiler produces
traditional code but doesn’t test single-path code generation. Because of time con-
straints, implementation of an automated setup for single-path code generation is left
as future work. The existing infrastructure of either LLVM’s test suite or Patmos’
own suite can be leveraged in the future, with reuse of the setup for the regression
tests also possible.
3.5
Usage
Together, the three categories of tests cover all our requirement. However, adherence
to a few rules will ensure the most efficient use of developer time and confidence in
the testing.
Whenever a change to the compiler is implemented, we want to be confident that
no bugs are reintroduced and all features still work. To this end, the regression and
program tests should be our source of truth. If all their test cases pass, then we can
be confident everything still works as it did before.
When a a bug is found, tests should be added that catch the error and ensures
we don’t reintroduce it moving forward. Ideally, we would be able to add a program
test that does this but it might not always be possible. E.g. the bug could be found
by a user that isn’t being able to provide a real-world program that suffers from it.
Regardless of whether we have a program test or not, we should also have a regression
test for it, since their simplicity makes it easier to find the source of errors. We should
also be careful not to only use unit tests when fixing bugs, as they may be deleted or
modified substantially in the future.
Preferably, we should introduce tests in the reverse category order. First we a
program tests is added for a feature or bug. We can run this tests before beginning
the implementation work and it should fail.
Using the failed program test, a set
of regression tests should be made that cover all the isolated reason for the failure.
Lastly, as part of the implementation work, unit tests are added where needed. This
should ensure we make it easy for ourselves to maintain the compiler moving forward.
Of course we can be flexible, switching between the categories as needed or omitting
program tests where required.
The division of tests into three categories is meant to make efficient use of the
developers’ time. The program tests give us the greatest confidence in the correctness
of the compiler, followed by the regression and then unit tests.
With the higher
confidence come an increase in the running time needed for the executing the tests.
In the day to day work, having long-running tests can severely reduce productivity,
which is not optimal.
With proper test management, we can create a setup where if a program test
fails, a regression test should also fail. Likewise, for every regression test, some unit
test should also fail. We can achieve this if we create tests as described above. If

becomes more difficult, as analyzing one class requires knowledge about other classes
first. Second, it make the code less flexible. As coupling increases, it become more

One straightforward way of decoupling is to go through the different classes, and

PIMPL Idiom
Since we are using C++, we must have any private class members declared in the
header file. We would like to work around this, such that the header only contains
public declarations of each class. This will give us a single point to look at when
analyzing class interactions.
The Pointer to Implementation (PIMPL) idiom is a technique for hiding the
private members of a class behind a pointer to an implementation class. The main
benefit of using this idiom is reduced compilation times, as less code resides in header
files that are compiled multiple times [33]. However, in our refactoring effort we will
use this idiom for an additional benefit: It allows us to separate the implementation
of classes from their public interface, which gives us a better overview of how classes
interact and how to reduce their coupling.

scopes have been initialized through their constructors, they are queried for various
information, like finding the header blocks of scopes or the topological order of its
blocks. Therefore, the SPScope objects don’t change after initialization, and their
internal representation is of no importance to other classes. However, as can be seen
in the listing, all private members of the class are present in the header file and, if
changed, require a recompilation of all .cpp files that depend of that header.

file, as seen in Listing 4.3. The original will then have a pointer to its implementation

private members is then always done through SPScope::Impl.

4.4
The Right Tool for the Job
There are almost always multiple ways of implementing the same functionality. E.g.
different data structures can be used for the same purpose, like std::vector can
be used as a container for non-repeating integers, while std::set can do the same.
Choosing one implementation option instead of another might better reflect the in-
tent of the code. std::vector doesn’t convey to the reader that it only contains
unique integers, std::set does. By using the later, we automatically give additional
information to the next reader, without having to change the design or write explain
anything in comments.
Using the construct that most precisely encapsulates the intent of the code can also
minimize developer errors. It is impossible for an ignorant or attentive developer to
add two identical integers to a std::set; it is very much possible with std::vector.
When refactoring, changing the code to use more strict constructs can help you
the code. If you can exchange the use of std::vector for a std::vector, it must
mean there are no duplicate values, which can be very useful to know.
The following is a list of additional constructs that were used in the compiler, but
could be exchanged for better alternatives:
Integer values to distinguish categories of values: When RAInfo has performed
register allocation, at each point in the program the value of a predicate could
be in one of two location: in a predicate register or spilled to a stack slot. The
original distinction between these two location was encoded in an integer: If
it was in a predicate register, the value was between 0 and 6, signifying the
registers $p1 to $p7; and if it was above 6, it signified the predicate value being
spilled to the stack, at the stack slot (n −6).
A better alternative would be to use an enumeration, where define predicates
and stack slots. We then use an unsigned integer to specify which register or
slot, without a need for value conversion. When both the location and the value
is need simultaneously (which is the case for RAInfo), std::pair can be used
to return them from methods together.
Negative integers to signify the lack of values: This is very similar to the pre-
vious example, but is prominent enough that we will risk repeating ourselves.
RAInfo tracks, for each predicate, whether it should be loaded from stack be-
fore a given blocks is executed and whether it should be spilled again afterwards.
Originally, it did so by using two signed integers, one for the loading and one

4.5
Fields
Excessive use of fields, private as well as public, can complicate a class. Since they are
accessible from any class method, they are a type of internal dependency of one part
of the class on another. Similarly to decoupling classes from each other, we also want
to minimize the use of fields to the absolute minimum, to decouple a class’ methods
from each other—as much as possible.
We will highlight three types of fields that should be avoided and removed when
refactoring. Fields can exhibit the properties of more than one of these types:
Transient Fields: Fields live as long as the object their in. Therefore, they must
be used of we want to have data that lives as long as the class instance is alive.
Transient fields contain objects that are not needed for the whole lifetime of
the class instance. They are instantiated when a certain method is called, and
live for the lifetime of that called, potentially being used by multiple additional
methods. When the original call terminates, the field is not used again. These
fields can often be eliminated completely and using method-local variables in
their stead.
RAInfo had a transient field called FreeLocs which was used by the register
allocation algorithm. It tracked the set of predicate registers and spill slots
that were free at that point in register allocation.
Register allocation is only done once, after which the result is stored in other
fields. FreeLocs could therefore be demoted to being a local variable of the
main register allocation method and completely ignore by any part of the class
that doesn’t involve the algorithm.
Cache fields Unlike transient fields, cache fields can be used by multiple methods
and can be useful for the whole life of the class instance. However, they contain

data that is an aggregate of other field’s data and can be obtained through
other means.

Cache fields can make the class’ logic more difficult to understand, since it’s not

also makes the class less flexible, as any change to the fields the cache is based

needed.

CHAPTER 5
Bundling
Now that we are familiar with the implementation of the compiler, and have a test
setup that can guide our work, we can begin introducing support for bundled single-
path code.
There exists many ways to capitalize on ILP for traditional code. One method is
to group basic blocks into what is called hyperblocks [35]. By grouping the right basic
blocks into a hyperblock, analysis and optimizations can be applied such that their
instructions can be bundled across basic block boundaries.
We want to leverage our knowledge of single-path code to find pairs of basic blocks
to merge. By merging, we mean schedule one block’s instructions in the first issue slot,
and the other’s in the second issue slot. For each block pair, we want to be certain
that—under all circumstances—at most one of them is enabled, and the merging will
not change the semantics of the program. To do this, four properties must hold for a
given block pair to be mergeable:
Non-equivalence: They must be in different equivalence classes. Otherwise, they
would always be enabled simultaneously.
Non-domination: Neither block can dominate or postdominate the other. If either
does, execution requires one block to finish before the other starts, which means
they can’t be executed simultaneously or merged.
Scoping: They must be in the same loop scope. This is to maintain the program’s
semantics and the blocks are iterated over the correct number of times.
There is a workaround to this property: if we can merge two loops into one
shared loop with both bodies of the original loops in it, we can now merge
blocks across loops. For two loops to be mergeable, similar properties must
hold for them as for blocks.
Ordering: There must exist a topological ordering of the CFG where the two blocks
immediately follow each other. The effect of merging two blocks is that they will
occupy the same place in the new ordering, which can be thought of as equivalent
to following each other in the old. If they can’t be ordered in sequence before
the merge, then merging them will result in a non-topological ordering which is
not compatible with the single-path transformation.

The act of merging two blocks involves more than bundling their instructions. To
allow the single-path transformation to finish post-merge, we need to preserve the
semantics of the CFG. This means all edges must also be preserved such that control
flow remains the same. If one block has an exit edge, then the merge block must also
have an exit edge to the same target node.
5.1.1
SPScope
SPScope originally managed all data related to the CFG, and would have needed to be
updated to implement the merging correctly. This meant going through all its relevant
fields and updating them accordingly.
This would not have been easy, since the
data was spread across the different subscopes in scope tree. Instead, we introduced
PredicatedBlock. Besides wrapping a block, it also manages all information that is
related to the block; its predicates, edges to other blocks, etc. We then implement
all the merging logic inside that class. This is much easier, as we only need to merge
the information from the two original blocks in the new block. As the information is
readily available in each PredicatedBlock, the merging of predicates, edges and so
on, becomes almost trivial.
5.1.2
RAInfo & PatmosSPReduce
Register allocation must also be updated to support blocks with multiple predicated.
Luckily, the algorithm used is well suited for this and only required minor tweaks.
It walks the CFG in the final topological ordering and assigns each predicate
it encounters to a free predicate register. When no register is available, it spills a
register-occupying predicate, choosing the one that is furthest from being used again.
It can then assign the freed register to the required predicate. Additionally, when a
loop is encountered, all active predicate are spilled, such that the loop has all seven
registers available when it begins.
The algorithm doesn’t need a design change to support bundled blocks. We simply
update the implementation to be able to assign multiple predicates to registers at the
same time, and spill more than one register at the same time.
Likewise, the final SPReduce pass—that performs the reordering of blocks and
insertion of predicate definition, load, or spill instructions—needs only tweaks such
that multiple definitions, loads, or spills can be performed for a given point in the
CFG.
5.1.3
Scheduling
As described in , after the single-path transformation is performed a scheduling pass
does a final tweaking of the code to make it adhere to Patmos’ latency requirements.

5 Bundling
The existing scheduler that performs this pass was designed to traditional code
generation, which also doesn’t use bundled code. This means the scheduler doesn’t
work for single-path code that has been bundled.
It main issue is that it does not have enough information about the code to be
able to do its work. As an example, when it encounters a merged block, it does not
know that its two predicates are disjoint and might therefore reorder the instruction
incorrectly.
As a result, we had to introduce a custom scheduler that is only used for single-
path code. Because of time constraints, we had to use the minimal viable implemen-
tation we could. It therefore only performs nop-insertions, and does so whenever it
encounters a latent instruction, regardless of whether a delay is actually required in
the given situation.
5.2
Proof of Concept
So far we have only talked about what is needed for supporting merging of block,
but haven’t touched on how to find blocks to merge. The design presented is meant
to be general purpose and support different merging strategies. This will allow the
experimentation, without having to redesign the rest of the single-path transformation
and implementation.
To show how a strategy could be implemented, we design and implement a very
simple merging strategy: For each loop scope, we go through all the basic blocks,
looking for one that has exactly two immediate successors. If we find one, we ensure
the following about its successors:
• The successor edges are neither back nor exit edges.
• Neither of them is a header of a nested loop.
• Neither of them postdominates the other.
We repeat until we cannot find a block with successors to merge. We do this for
all scopes, starting at the root and following a depth-first traversal.
We show that any pair found with this strategy is mergeable:
Non-equivalence: If the block who’s successors we are merging—the parent block—
isn’t itself a merged block, then its trivial to see that the successors must be in
two different equivalence classes. If the parent is a merged block, each of the
block’s to merge is the successor of one parent block. We know that the two
parent blocks are of different equivalence classes, which means their successors
can’t be of the same equivalence class.
Non-domination: Since execution branches at a shared parent block, the two blocks
can’t dominate each other, and since we also disregard pairs that postdominate
==PAGE:65==
5.2 Proof of Concept

each other, the non-domination requirements must hold for all pairs that are
chosen.
Scoping: Since we disallow block pairs that have exit edges from the parent, we
know they can’t be in a superscope of the parent block’s scope. Since we also
disallow them being a header of a subscope, the only possibility left is that they
are both part of the parent block’s scope.
Ordering: Since the two blocks being merged are successors the same parent block,
it is trivial to see that there must exist a topological ordering where the two
blocks follow each other.
Using this strategy on the CFG in Figure 2.1 will result in a merger of the blocks
c and d as we have discussed in Figure 5.2.
This strategy is not meant to produce efficiently bundled code. We will simply use
to show that it is possible to implement a strategy that produces bundled instruction.
Being very restricted in which block pairs it considers for merging, we expect this
strategy to merge only a few blocks in a function. As such, a program that is compiled
using this bundling strategy would probably not run faster in any practical sense,
however, we should be able to measure the difference in cycle count when run with
pasim with caching disabled.

CHAPTER 7
Conclusion
In this project, we have successfully readied the Patmos compiler for future research
into efficient bundling of single-path code.
We first designed and implemented an automated test setup that ensures we can
update the compiler while maintaining functionality. The setup also increases pro-
ductivity, by having a dedicated category of tests—unit tests—that are easy to create
and fast to run. Two other test categories then ensure we maintain the compiler’s
correctness, of which only regression tests were implemented.
Before adding support for bundled code, we used refactoring to get familiar with
the compiler and make it easier for newcomers to work with it. Different techniques
that are optimal for learning a new codebase were presented, and also techniques to
make code more readable. Even though our refactorings had potential to reduce the
compiler’s performance, measurements showed they had little effect in that regard.
Lastly, we updated the compiler with infrastructure that allows for the implemen-
tation of block merging strategies without the need to update the compiler further.
We showed how such a strategy could be implemented and that it improved the
performance of a subset of the TACLE benchmark—slightly.
Additional Work
Because of time constraints, not all aspects of this project’s work can be regarded as
finished.
First of all, even though the testing setup works as intended, it is woefully in-
adequate in its number of test cases. Attention should be given to increasing the
coverage of regression tests, and possibly unit tests, such that they better cover the
needed functionality.
Second, the refactoring work performed as part of this project should not be seen
as either the minimum amount needed nor the maximum. The initiatives described
were simply the ones undertaken as part of this project and could easily be continued
in the rest of the codebase when required.
Lastly, the updates to the compiler have likely introduced flaws into the single-
path code generation. This is apparent in that only 10 of the TACLE benchmarks
manage to compile and run. Therefore, some work needs to be put into fixing errors,
such that the compiler can handle most programs given to it.
==PAGE:76==
66
7 Conclusion
Future Work
To begin with, future work should focus on implementing an automated setup for pro-
gram tests. Our experiences in this project have shown that, without real programs,
it can be difficult to asses how correct the compiler is. Afterwards, we should contem-
plate whether additional testing categories can be introduced to cover non-functional
requirements.
Our experiences have also given us insight into the quality of the single-path code
generated. Even though the code follows the single-path transformation rules, there
are many small details that make the code inefficient. As an example, when predicates
are spilled, a general purpose register is used to store its value until it is needed again.
However, that register is immediately spilled to the stack whenever a predicate is
saved in it. This is usually not needed, since the general purpose register can hold up
to 32 predicates at one time. Therefore, some effort can also be put into improving
the single-path code generated by the compiler. This is crucial if we want to show
that single-path code can be used in real-world scenarios.
Lastly, research into efficient block merging strategies is the main avenue for future
work. The magnitude of that work was one of the reasons why this project did not
undertake it seriously—the others being the lack of a testing setup and unfamiliarity
with the code. This project has hopefully made it much easier to experiment with
different strategies with minimal implementation work needed.
