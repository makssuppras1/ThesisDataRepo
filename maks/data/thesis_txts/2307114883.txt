Summary (English)
The goal of this thesis is to implement a tool capable of analyzing the static
costs of a hardware model described in a GEZEL-based language, as well as the
run-time costs of an execution based on a given input. The purpose of these
analyses is to make the users of the tool able to make reasoned choices when
designing hardware solutions.
This has been accomplished by designing a modeling language based on the
GEZEL hardware description language and a tool that translates models in this
language to an abstract syntax, which is then analyzed and simulated. The
tool consists of a lexer and a parser that does the translation, a compiler that
converts the abstract syntax into machine instructions, a virtual machine that
simulates an execution of these instructions and a cost analyzer that calculates
the static costs and the run-time cost based on the abstract syntax and the
execution, respectively.
The thesis describes the choices of which types of costs to analyze, the design of
the modeling language using concrete and abstract syntax along with context-
free grammar and the implementation of the tool components, as well as shows
some uses of the tool during some experiments.
In conclusion, the goal of this thesis has been achieved, although the tool does
not have the full modeling and simulation capabilities of GEZEL.

Chapter 1
Introduction
When designing hardware, it is rather expensive to alter a solution after it
has been implemented. This is why you would often describe and simulate the
solution beforehand, using one of the numerous hardware description languages.
One of the more simple-to-use is the cycle-based GEZEL[1]. It is based on the
Finite State Machine with Datapath model, meaning that you describe the
available instructions in the datapath and the control operations in the state
machine.
It is often necessary to meet some cost requirements for the hardware, e.g.
the maximum time for an execution and the maximum physical area available.
You would therefore have to choose between multiple implementations of the
solution, e.g. one that favors the time over the area or one that has a higher
peak energy than another while using less energy in the long run. This could
sometimes be dicult to compare by yourself and as it is not possible to nd
these costs using GEZEL, a tool capable of this and just as simple would be
useful.
The goal of this thesis is therefore to develop such a tool, as well as designing
the associated modeling language. The language will be based on GEZEL and
have almost the same syntax.

Chapter 2
Examples
When designing applications that should work with both hardware and software,
one of the rst things to consider is the balance between those two. Moving
towards a more hardware-oriented implementation could increase performance
and energy eciency, while moving towards a more software-oriented one could
make the application both easier and cheaper to develop.
One of the advantages of implementing in hardware is the ability to run things in
parallel, but even though there are multiple operations able to run independently
of each other as separate components, it might not be possible to run these in
parallel due to area and energy constraints. Therefore a decision must be made
on how much of the work should actually be done sequentially instead of in
parallel.
To make this decision easier, it could be practical to analyze the costs for the
specic design, e.g. time, energy and physical area. This could help the designer
to choose a dierent approach, if needed, and thereby optimize the costs to favor
the most important one of them, or maybe even nd an acceptable compromise.
To give an example of this, consider the formula

Time and area are generally opposites, i.e. if the area is reduced, components
may need to be reused thus eliminating the use of parallelism and causing the
time to increase. This is at least the case in more extreme conditions as those
described below. Energy, on the other hand, can be calculated as a function of
time and area. There are two types of energy which are relevant here: Peak
energy and total energy consumption.
Peak energy is the maximum needed energy output from a power source at any
given time while the system is running.
This data is useful to determine if
the power source is capable of providing enough energy to the system at peak
moments.
Total energy consumption is the amount of energy needed for a full run of the
system. Knowledge of this can be helpful in making the system more energy
ecient, as well as in determining if a power source with limited energy, e.g. a
battery, has enough capacity to run the system at all.
2.1
Example 1 - Time optimization
In order to optimize for the time cost, as much as possible must be calculated
simultaneously. One way to do that, if the hardware allows it, is to run all
addition and subtraction operations in one clock cycle. To achieve that there
must be an operator for each of the operations. First of all there need to be
n subtraction operators. Then each of the n results must be added in pairs,
which means that n/2 addition operators are needed in this step. Following this
method, each additional step needs half the number of addition operators as the
previous one. The connected hardware components can therefore be visualized
as a binary tree with the n subtraction operators as the leaves and n−1 addition
operators down to the root. Figure 2.1 shows an example where n = 8.
2.2
Example 2 - Area optimization
To optimize for the area cost, the best start would be to have one addition
operator and one subtraction operator and then reuse these two for the entire
calculation. Also, a register is needed to function as an accumulator for the
temporary values. If the input is seen as two vectors (or two streams), two
multiplexers are needed to divide these into chunks of the right size.

3.1
GEZEL
GEZEL is a hardware description language made to model hardware. By using
it, one is able to implement a nite state machine with datapath, also called
FSMD.
In GEZEL a datapath consists of registers and a number of instructions in the
form of signal ow graphs.
The wires connecting the hardware components
are represented by signals.
The signal ow graphs contain expressions that
assign values to registers and signals through various operations. The state ma-
chine dictates how the instructions in the datapath should be executed through
various transitions, each representing a clock cycle. To give an example of a
GEZEL implementation, Figure 3.1 shows one of Euclid's algorithm for nding

After having passed the semantic checks the compiler can begin to generate the

As described in Section 4.1.3, the datatypes are constructed as a hierarchy, e.g.
a state machine contains transitions, transitions contain instructions, instruc-

that each part of this hierarchy is handled by its own function, e.g. one function
handles all instructions in a transition and another handles all assignments in an
instruction. With the exception of the function that handles expressions, each of
these functions delegates the compiling of the level below them in the hierarchy
to another function while they are responsible for putting the result together
correctly. This goes for state machines all the way down to assignments.
Expressions are a bit dierent. They are the lowest level in the hierarchy and
are constructed as recursive types. The function responsible for compiling ex-
pressions is therefore also the only recursive one.
Before any of the compiling functions are used, a variable environment is cre-
ated, mapping all input, output, register and signal variables to an index. This
environment is used to identify the position of each variable on the stack in the
virtual machine. Further explanation follows later.

The cost analyzer is the tool component that dierentiates this tool from GEZEL
and the code for it can be found in Appendix C.2. As the name suggests, it is
what enables this tool to nd the relevant costs for a given hardware model. It
is designed to nd the four types of costs mentioned in Chapter 2, i.e. the area
cost of the hardware components and the time, peak energy and total energy
costs of the operations.
The area cost is calculated for all instructions and registers. The time, peak
energy and total energy costs are calculated for all instructions and transitions.
This is due to the area not being relevant for transitions, as no matter how
many instructions are used in a transition, the area does not change. The cost
analyzer is also capable of calculating run-time time and energy costs, i.e. the
time and energy needed for each clock cycle, as well as the total cost for both.
The total energy cost is simply the sum of all clock cycles, but, as all clock

When initializing a label in the compiler, it is given a reference ID, but in order

5.2
Virtual machine
Before the virtual machine is going to execute the program using the execcode
function, the program list and value stack are both initialized as arrays, which
gives quick access to their values.
The system then goes through the program list one by one. Each time it meets
an integer, this integer is stored on top of the value stack. When meeting a
machine instruction, the associated values are then retrieved from the top of the
stack, used in the computation and then replaced by the result. The following
example shows how the machine instruction ADD is executed.
| ADD
-> s.[!sp -1] <- (s.[!sp -1] + s.[!sp]); sp := !sp -1; pc := !pc+1;
Prior to reaching this instruction in the program list, two integers have been
stored on top of the value stack. The integers are then referred to by the code
s .[! sp−1] and s .[! sp], corresponding to the former and current position of the
stack pointer, which represents the top of the stack. The result of adding these
two integers together is then stored on the value stack in place of the rst integer
and the stack pointer is moved one position back. This is done such that only
the relevant temporary values are kept on the stack and thereby keeping it from
growing rapidly.
One of the auxiliary functions that stands out in the virtual machine, is the
getSubBit function.
It is responsible for handling the BITRANGE machine
instruction. In order to get a subrange of bits from an integer, this integer is
rst converted to a string representing the bits, before a substring corresponding
to the subrange is extracted and converted back to an integer.

5.3
Cost analyzer
This section will describe the parts of the cost analyzer implementation that need
elaboration beyond the description in Section 4.2.5 and is not self-explanatory
in the code.
The cost libraries for operations are, as mentioned in Section 4.2.5, implemented
as maps.
For simplicity, all the operation costs are static, meaning that no
matter the sizes of their operands, their costs would stay the same. The same
goes for registers that keep the same cost values for all sizes. The multiplexer
component has only one cost value as well, although it is prepared for basing
the cost on the size of the input and output.
Expressions can be nested and should therefore be handled as such, e.g. when
calculating the cost for a dyadic expression the two operands are therefore also
treated as expressions. When the cost for both of the two operands is calculated,
it is then added together with the cost of the operator, which is found by making
a lookup operation on the relevant map. All dyadic expressions are treated the
same way, as are all monadic expressions. The rest of the expression types have
unique structures and are therefore all treated dierently, e.g.
the selection
operation is nding the minimum and maximum cost for the two branches and
the nary operation need to nd the cost of a whole list of operands.
The ve main cost functions use a type called costCalculationType to let the
sub-functions know which type of cost they are to calculate. This way there do
not need to be a dedicated set of sub-functions for each cost type, but they can
use the same ones by just making them behave dierently depending on which
of the main functions that called them.
To account for the dependency between signals across instructions, the ndIn-
strDependence function starts by nding which instructions the various signals
are depending on and then it orders the instructions according to these depen-
dencies. The instructions in the implementation of Example 2 from Chapter 2
showed in Section 6.2 will therefore get ordered exactly as in Figure 3.3.

5.4
System
The system code found in Appendix C.1 is what binds all the tool components
together. It is the system that handles the input from the user, i.e. command
line input specifying the input le, which state machine to run and the values
for the datapath input. The command line input needs to be on the following
form:
filename state_machine input1 value1 input2 value2 input3 value3...
The system then delegates the work to the various tool components and display
the output in the console.
==PAGE:49==
5.4 System
37
It starts this process by passing the input le to the lexer and parser to extract
the sets of datapaths and state machines. Then it runs the semantic analysis
on the chosen state machine and the associated datapath. The next step is to
create the variable environment, as well as to extract the pairs of input variables
and their values from the command line input. All this, along with the state
machine and datapath, is then handed to the code generator, that returns the
program as machine instructions. The area, time, peak energy and total energy
costs are then calculated by the cost analyzer and printed to the console by the
system. Finally, the program is run by the virtual machine, followed by the cost
analysis of the run.
Some examples of console output from the system can be seen in Chapter 6.

Chapter 7
Conclusion
In the last chapter of this thesis, there will be given a summery of the work, an
evaluation of the result and some suggested improvements to it.
7.1
Summary
First in this thesis you were introduced to some motivating examples that
showed why there is a need for such a cost analysis tool and what kind of
cost analyses it should be able to do. Then you were given a short introduction
to GEZEL, the reason for choosing F# as the programming language and a
requirement specication for both the modeling language and the tool based
on some analyses of them. This was followed by the design description of the
language, e.g. what types it consists of and how it is structured, as well as of
the tool, e.g. what components it is made of and how they are working together.
The implementation came next, which showed some more in-depth descriptions
of the tool components and their functions. Lastly, the experiments chapter
showed what the nished tool was capable of by implementing the examples
from the beginning of this thesis.
The work in this thesis resulted in a fully functioning modeling language and
a tool capable of analyzing both the static and the run-time costs for a given
==PAGE:64==
52
Conclusion
model, as well as simulating an execution of it. They both have some shortcom-
ings, though, as will be described later.
7.2
Evaluation
The main goal of this thesis was to develop a tool that makes the user of it
able to make well founded choices when designing a cost-restricted hardware
solution, as well as designing a modeling language capable of describing such
solutions.
The nished language is capable of describing some simple hardware models,
consisting of a single datapath and a single state machine. It meets all require-
ments from Chapter 3, but due to time constraints of the design, some GEZEL
features have unfortunately been discarded, e.g. lookup tables and structural
hierarchy. Therefore it is not possible to model nested datapaths or parallel-
running state machines in this version.
In the nished tool it is possible to use other cost libraries, but it is done by
exchanging the values in the relevant maps and functions in the source code.
The tool is capable of checking a model for errors, as well as running a type
check on it. As shown in Chapter 6, it is capable of calculating the correct area,
time, peak energy and total energy costs based on some given cost values. The
tool does also meet all requirements, although it is a bit troublesome to change
cost libraries.
7.3
Improvements and further work
As mentioned earlier, both the modeling language and the tool have some short-
comings. A number of them will be mentioned here, some even accompanied by
a possible solution.
The most obvious functionality to implement in an updated version would be
the use of nested datapaths and multiple state machines. The current lack of
this functionality greatly inhibits the usefulness of the tool. GEZEL has the use
construct for the purpose of implementing structural hierarchy, so something
similar would make sense.
Lookup tables was also one of the functionalities that was discarded. Although
they were deemed unimportant for reaching the goal due to the time constraints,
==PAGE:65==
7.3 Improvements and further work
53
they would probably be easy to implement as a list of registers in an updated
version.
The tool does not take type overow into account, meaning that the simulation
will not be done correctly if such an overow should occur. It would be nice if
the tool was able to simulate that behavior or at least detect it, which might be
done by checking every value assigned to a variable against the type of it during
the simulation phase.
If a model is not implemented as having only one register load or store operation
in each instruction, the tool is not capable of ordering the instructions correctly
with parallelism. It would therefore be a very nice feature if the tool could split
these instructions up by itself, generating the optimal code for the user to see.
When the instructions are ordered due to their signal dependency, they are put
in the earliest step possible. This does not necessarily mean the most optimal
ordering. The tool would therefore be much more powerful if it could optimize
this ordering based on which type of cost that is chosen to have the highest
priority, or maybe even generate a solution for all of them.
