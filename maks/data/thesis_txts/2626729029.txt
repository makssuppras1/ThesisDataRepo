Abstract
In this masters thesis, a microservice architecture has been designed and implemented
for the purpose of collecting, processing, accessing and visualizing geospatial data. The
architecture has been developed for The Live Road Assessment project currently in devel-
opment through collaboration between The Technical University of Denmark, The Danish
Road Directorate and Green Mobility. Through reviewing publications and discussions
with members of the LiRA team, a thorough analysis of the domain has been conducted.
This domain knowledge was used to design and later implement the microservice archi-
tecture with the key goal of allowing users to dynamically deﬁne data processing pipelines
consisting of processing steps to process road data. The project also suggests a CI/CD
development workﬂow to be used in the continuation of the project. Finally, the archi-
tecture has been evaluated based on an interview with an end-user and a performance
evaluation.

Introduction
Modern cars are produced with an ever-growing number of sensors, such as GPS sensors,
accelerometers and cameras. Each car produces large amounts of data when it moves, and
with data from many cars in a road network, data about the pavement can be extracted.
This data can potentially be used to determine the damage on the pavement.
The Danish Road Directorate (DRD) is in charge of maintaining the greater Danish
road network such as motorways and highways going through the country. Over time,
the health of the road deteriorates steadily until a point where the health decreases
exponentially and therefore, the cost of repair increases. The cost-eﬃcient point to repair
the road is just before the exponential decrease. The Danish Road Directorate uses models
and pavement management systems (PMS), such as RoSy and VejMan, to estimate this
point in time.
The current road condition assessment method is to collect the data
from the roads annually using trucks ﬁtted with sensors for road measurement, known
as Automatic Road Analysers (ARAN) and by manual inspection. Manual inspection
and ARAN devices produce reliable results, but unfortunately, it is expensive, and The
Danish Road Directorate can only get new data a few times a year.
The Live Road Assessment Project (LiRA) aims to improve pavement management by
collecting and processing the data produced by the sensors in modern cars. By using
the data from modern cars, LiRA is expected to improve the prediction and detection
of damage to the roads and save The Danish Road Directorate around 20 million DKK
every year. LiRA is a collaboration between The Technical University of Denmark, The
Danish Road Directorate, Green Mobility and Sweco to create a system that can utilize
the data from the modern cars and integrate the data into the systems and processes of
The Danish Road Directorate. Green Mobility, a car renting company operating almost a
thousand cars[15] in Denmark, has agreed to supply data collected from their cars to The
Danish Road directorate for this purpose. The LiRA Project covers the entire process of
collecting IoT data from the cars, processing and structuring big data and interpreting
the data with machine learning to ﬁnd the road conditions. This project will be focused
on the processing and structuring of data.
The cars generate a large amount of data on each of their rides, and in the future, more
sources of data with diﬀerent sensors may also be used. This requires a system that can
handle large amounts of data and data with diﬀerent processing requirements at diﬀerent
times. The system needed to process the data must be modular to accommodate diﬀerent
data processing processes depending on the incoming data.
Data may be processed in diﬀerent ways for diﬀerent purposes, in a sequence of processing
steps.
This is where processing pipelines are introduced.
A processing pipeline is a
sequence of processing steps for processing a set of data. All incoming data needs to be
cleaned and reﬁned, a pipeline should be responsible for preparing new data entered in
the system. Similarly, when enough data has been collected for a stretch of road to gain
meaningful results, a processing pipeline is to be used to collect the relevant data and
1
==PAGE:7==
extract new information through machine learning.
This project sets out to design and implement a microservice architecture for processing of
large amounts and heterogeneous geospatial data. A microservice architecture has, among
many others, the advantage that it can be dynamically scaled to meet the high demand
of the services. New services encapsulating new functionality can also be integrated into
the system with little or no disruption of the rest of the system. Finally, a microservice
architecture can isolate failures to allow the system to keep running, even if parts of
the system fail. These three advantages over monolithic systems are signiﬁcant for this
domain as data is collected continually.
Research Question
The goal of this masters thesis is to produce an architectural design of a microservice
architecture that can handle large and varied amounts of IoT data. The LiRA Project
domain will be used as a vessel to demonstrate the architecture’s utility and feasibility
as a design science artefact. As a result, the architecture may be applied to problems in
similar domains. On this background, the research questions for the thesis is stated as
follows:
How should a conﬁgurable microservice-based architecture be designed, intended for col-
lection, processing, access and visualization of geospatial data?
1. How should an architecture be designed to process large and continuously generated
amounts of IoT data?
2. How should an architecture be designed to be extensible by new processing pipelines?
3. How should a stable architecture be designed to handle failures in its processing
pipelines?
To answer these research questions, the Design Science Research Methodology[2] will
be used. A thorough analysis of the LiRA domain will be conducted to understand the
aspects of the problem. A deep understanding of the problem is required to make a design
to solve it. A design will iteratively be developed along with the implementation of the
architecture of the LiRA Project. The feasibility and utility of the architecture will be
proven through demonstration of its use in the LiRA Project. Finally, the architecture
will be evaluated through interviews with key stakeholders of the LiRA Project and a
performance evaluation of the architecture.

Related Work
Pavement Maintenance Systems (PMS) such as RoSy and VejMan have been around for
many years and pavement management systems are used in many countries for monitor-
ing the state of their road networks. New developments in the scene of smart devices
and IoT have started new developments of integrating new sources of data into pave-
ment maintenance systems. Government agencies and private companies like the Finnish
company RoadCloud, that uses IoT sensors to detect snow and ice on the road[1], have
started using the data to determine the state of the road. The use of IoT data in pave-
ment maintenance management has also been attempted at the Tampere University of
Technology in Finland, where a study has been made on the viability of road data col-
lection via smartphones in vehicles, in Challenges in the Interpretation of Crowdsourced
Road Condition Data[27]. Their preliminary ﬁndings were that smartphones could, in
principle, be used to ﬁnd road surface damage but were more useful to determine the
overall state of the road.
Researchers at The Technical University of Denmark are in collaboration with The Dan-
ish Road Directory, Sweco and Green Mobility developing the Live Road Assessment
Project (LiRA)[25]. LiRA uses state of the art technologies such as machine learning,
big data and microservices to detect damage on the road network based on data from
modern rental cars in and around Copenhagen. LiRA has recently submitted an article
on their work so far. Live Road Condition Assessment With Internal Vehicle Sensors[16]
explains the LiRA project at a high level and the areas of focus such as IoT data col-
lection, data pre-processing/structuring and ﬁnally data analysis/interpretation through
machine learning. The LiRA project has also produced three master theses so far. Live
Road Assessment based on modern cars: A prototypical geographic information system
for road maintenance planning[4] produced a prototype of the LiRA Map for visualizing
road condition data on an underlying map for a better overview of the existing road
surface problems. Development of the LiRA Map was later continued in LiRA Map: A
Cloud-based Geo-information System for Road Maintenance[3] along with a REST API
for accessing the data. A Flexible Framework for Generic Data Collection from Vehi-
cles[6] dealt with the structuring of data and data transformation tools that will also be
used in this project.
The use of Microservice architectures provides unique beneﬁts as demonstrated and rec-
ommended by A Progressive Web Application Based on Microservices Combining Geospa-
tial Data and the Internet of Things[20]. This paper explains the use of microservice
architecture and web applications to collect, process and visualizing geospatial IoT data.
Microservice architecture is also used in IoT and Microservices Based Testbed for Con-
nected Car Services[7] to develop a testbed framework to integrate cars as data sources.
They argue that microservices can solve scalability challenges, integrate diﬀerent tech-
nologies and protocols, and integrate heterogeneous IoT devices.

3
Analysis
The goal of the analysis is to understand and connect the concepts of road maintenance
and LiRA. The domain will be explained in detail as to what data exists, how it can be
processed and transformed, and ﬁnally be interrelated to the real world to be useful for
the Danish Road Directorate. The analysis will also examine the role of each user and
their interactions with the system through use cases.
The analysis has been based in large part on interviews with the LiRA team and publi-
cations about LiRA[16].
3.1
Domain Concepts
In this section the LiRA domain will be explained. The domain can be described in three
parts: Data Concepts, Map Referencing and Processing. Like any software system, it
is crucial to understand the domain to design a solution that solves the problem. By
analyzing each domain concept in detail, a domain model has been produced that shows
in detail the attributes and relationships between objects in the LiRA domain.
The heterogeneous data collected by the car as it is being driven is collectively categorized
as sensor data.
Sensor data is simply raw measurements without much value in its
unprocessed state. As mentioned in the introduction, the data is interpreted by machine
learning models to determine the state of the pavement. When sensor data is collected,
it goes through a process of reﬁnement known as pre-processing before it can be used by
the machine learning models. The condition for the data to be interpreted is that the
data can be linked geographically to the real world. The data can be cleaned, structured,
and referenced to a road in the real world through processing. The machine learning
models consume sensor data, and produce data that represent damage on the pavement.
The damage on the pavement is represented as condition data, which can be used by The
Danish Road Directorate to prioritize their work.
In the following subsections, each area of the domain will be explained in greater de-
tail.
3.1.1
Data Concepts
There exist two main categories of data in the domain: condition data, and sensor data.
Common to all types of data in the LiRA domain is that they can all be related to a
road, speciﬁcally to the road section where the data is relevant.

values recorded at the measurement, one for each axis. Some sensor data also contains a

Condition Data
describe the road conditions of the road. A pothole, as seen in Figure
1, is an example of a road condition. A pothole has a size and a location on the road. The
condition data would describe this as a width, a depth and a timestamp for when the road
condition was discovered. Some types of condition data, for example, cracks in the road,
cannot be described as a single point on the road but rather as an area covered by the
road condition. An example of condition data can be seen in Figure 2. The condition data
contains a description of the road condition through the Defect, Amount and Unit and
a description of when and where the road condition is. Condition data can be collected
in diﬀerent ways. It can be collected as a result of a report by a road inspector, it can
be reported by an organization such as Sweco or it can be reported through machine
learning models on sensor data. With enough sensor data collected on the same stretch
of road, machine learning models can transform sensor data into condition data.

Sensor data and condition data are conceptually diﬀerent and have diﬀerent purposes,
but they still share some similarities. A general form of data is introduced that contains
a timestamp, a position and a map reference as seen in Figure 3. Data must have a
timestamp to tell when the data was created; this is important in a domain like this
where data can become too old or irrelevant due to road repairs. The data also has a
position value that describes where the data refers to in the real world. Not all sensor
data has a GPS position as seen in Listing 1, but the position can be found through
processing, which is why the position in Data is optional. Condition data always has a
GPS position. The data with natural and computed positions should be distinguishable,
so the sensor data has a computed ﬂag. This way, there are three valid states of the
sensor data. If the data has no position and is not computed, it means that it has not
been computed by interpolated yet. If the data has a position but is not computed, it
means that the data originally came with a position and in the third state if the data
has a position and has been computed it means the position is a result of interpolation.
There is also the fourth invalid case where the data has no position and is computed, but
this would mean the data has been corrupted.
Besides the GPS position, the data has to be connected to the road where it was collected.
Such a connection is called a map reference and will be covered in detail in Section 3.1.2.
Data has a number of data values, typically one, but may have more as in the case of
Listing 1.
All data has a data type to give context, for example, ’acceleration’ from
an accelerometer. The data types have data properties that describe the data values
associated with that data type. In the example of the gyroscope, there would be one
Data object with a data type of ’gyroscope reading’.
The gyroscope data type has
three data properties: x-axis, y-axis and z-axis. The three data values produced by the
gyroscope then refer to the data property that they represent. To give an example of this
with condition data, it could be a pothole with three descriptive values such as width,
length and depth of the hole. Also to note on the data property, it has a maximum and

a minimum. These are used for visualization to provide a scale of the data value.
The raw sensor data does not hold much value on its own, but with other measurements
collected around the same time, it can be reﬁned. Sensor data is grouped by the trip it

A device is a common term for data producers, for example ARAN Devices and Green

some condition data, such as the data collected from ARAN Devices that produce both
sensor data and condition data, can be related to the ARAN Device’s trip. The Device
type describes the type of device, for example ARAN Device or Green Mobility Car, and
the Organization that operates the device.
3.1.2
Map Referencing
Referencing data to the real world has two purposes. First, the data must be relatable
to the road network for the machine learning models to interpret the data and ﬁnd out
where conditions exist. Secondly, the users of LiRA need an overview to see where con-
ditions exist and how much/what kind of data has been collected on the roads. Previous
master’s and bachelor’s projects have designed and implemented a map to visualize the
condition data to the users, but eventually, the users would also like to have sensor data
visualized.
7
==PAGE:13==
A map is a representation of the real world with geographic data. The map used in LiRA
builds on the Open Street Map (OSM) map structure. OSM has many geographic features
such as railway lines, property boundaries and nature reserves, but for the purpose of this
project, only the map features related to the road network will be used. At its core, the
map is built up of points, which represent a point on the map with coordinates. Lines
between points are called sections. LiRA uses these sections to relate data to the real
world, in that way the machine learning models can interpret all the data on a section to
determine if any conditions present. A relation consists of one or more sections to make
up what is commonly understood as a road. Together, the points, sections and relations
make up the road network and map that is used in LiRA.
Figure 4: A sample of the Open Street Map showing points and sections of a residential
area
Figure 4 is a sample of the Open Street Map (OSM). The LiRA domain point, section and
relation correspond respectively to the OSM node-, way- and relation- map elements[17,
19, 18].
The stretch of road highlighted in red colour is a section and the circles inside are the
points that deﬁne the section’s shape.
As LiRA uses Open Street Map’s map data, there is no control over the map elements. To
relate the data in the LiRA domain to the data in the OSM domain, map references and
Coverages are used. A map reference creates a relationship between a point of condition
data or sensor data to a section on the map. The map reference approach makes good
sense for data such as sensor data that can be connected to a single GPS point or condition
data such as a pothole, whose position can also be expressed as a GPS position. For some
condition data, such as the conditions alligator cracks and upheaval that can cover a wide
area of the road, a map reference to a single point on the section does not make sense.
Instead, a coverage is introduced, which makes the reference between the condition data
and the section, but instead of having a single point, the coverage has a start point and
end point of where the condition exists. In Figure 5 the LiRA Data domain objects can
be seen from Figure 3. On the right-hand side the OSM domain objects exist and in the
center are the references between LiRA and OSM.

The center of Figure 5 is Data Aggregate, which will be used by visualization tools to
visualize aggregates of sensor data. Users of LiRA have requested that they should be
able to see information about the collected sensor data on the map. For example, a user
should be able to see how many points of a certain kind of measurement has been gathered
on a section before applying machine learning models to the data. The Data Aggregate
is then used as an overlay on the map to display aggregates such as sums of data or
minimum, maximum or average of data values depending on the user’s needs.
3.1.3
Processing
Until now, the types of data and referencing to the real world has been discussed. This
section will look at the processing of sensor data to transform it into condition data. The
transformation raw sensor data is performed by a sequence of processing steps: import,
interpolation and map matching, before it can be used in machine learning to ﬁnally
compute the condition data. This process of preparing the data for machine learning is
known as the Pre-Processing Pipeline.
Import
The sensor data collected by Green Mobility cars is structured as JSON ﬁles
containing a header with metadata, followed by a series of measurements. The import
process parses the JSON ﬁle and structures the data in a database.
Interpolation
Only some of the measurements taken during a trip have an associated
GPS position as seen in Listing 1. To reference each measurement to a point on the map,
the positions must be computed for the measurements that do not have one already.
Interpolation constructs new data points between a set of already known data points.
Common to all the measurements is that they have a timestamp of when they were
9
==PAGE:15==
created.
The missing GPS positions can then be interpolated using the known GPS
positions and the timestamps. Figure 6 shows a small sample of measurements, marked
with blue colour, that have been interpolated between the original GPS measurements
marked in green colour. In reality, there can be hundreds of measurements between GPS
measurements, but for visibility, only a few of them have been drawn on the map.
Figure 6: Map showing interpolated GPS measurements (blue) created from existing GPS
measurements (green)
Map Matching
When the measurements are taken, it is not known which road the
vehicle was driving on. Map matching is the process of comparing GPS positions to a map
to produce map references. A map matching algorithm uses a sequence of GPS positions
to estimate which sections the positions were most likely taken on while driving. Due to
calibration errors and general inaccuracy, some map matching algorithms will also align
the GPS positions on the road where the algorithm believes the position was actually
taken. This can be seen in Figure 7, where the red dots are the original GPS positions,
and the green dots are where the map matching algorithm has aligned the positions with
the road. Sometimes positions cannot be matched either due to using outdated maps or
the error on the data being too high for the algorithm.
Machine Learning
Machine learning algorithms are used in LiRA to interpret sensor
data and produce condition data. While the actual machine learning is outside the scope
of this project, the resulting architecture must accommodate for such processing steps.

The sequence of processing steps is collectively called a processing pipeline. Figure 8
shows that there are two sides to the pipelines and processing steps, one is the execution
of the pipeline and the other is the pipeline deﬁnition. Execution of a pipeline, or pipeline
run, is simply to execute each processing step in order. The pipeline deﬁnition is used by
users of LiRA to create and modify the process at which data is processed, for example
the Pre-Processing pipeline.
A pipeline deﬁnition contains the name of the pipeline and the sequence of processing
steps to execute. Each processing step has its own processing step deﬁnition that deﬁnes
what processing step it is, a list of arguments required by the processing step and a list of
return values resulting from successful execution of the step. The pipeline deﬁnition uses
the Composite Pattern, which means that a pipeline deﬁnition can nest other pipeline
deﬁnitions as part of its execution.
Just like the processing step deﬁnitions, the pipeline deﬁnition has required arguments
that will be used in the execution. Arguments could for example be connection strings
or descriptions of the data to process. Some arguments will always be the same for a
pipeline, such arguments are Fixed Arguments and are written in the pipeline deﬁnition.
Arguments in the pipeline are available to all processing steps and nested pipelines during
execution. Return values from processing steps are also considered as arguments, which
means that the pipeline can use the output from one step as input in a later step.
Pipeline runs are long-running processes and can experience failures from any number of
reasons such as invalid input, invalid or missing data and network errors just to name a

4
Design
In this section, the design of the LiRA pipeline processing architecture will be discussed
and how it may improve over the initial LiRA architecture. The design has gone through
several iterations, but only the LiRA processing architecture’s initial state and the ﬁnal
design will be discussed here.
4.1
Architecture
The LiRA architecture must provide storage, processing and later access to the sensor
data, condition data and any metadata related to the processing. A capable architecture
must be designed that is also easy to maintain and implement new functionality, such as
new processing steps.
There are three main requirements that the architecture must fulﬁl regarding the pipelines
runs:
1. The architecture must be conﬁgurable to run diﬀerent pipeline deﬁnitions on dif-
ferent data.
2. The architecture must be scalable to run multiple pipelines in parallel.
3. The architecture must be able to recover from simple errors.
As explained in the analysis, the data collected is heterogeneous, and each type of data
may have its own set of processing steps needed to transform it. The architecture must
fulﬁl the ﬁrst requirement so that the architecture can handle these processing steps and
data.
Secondly, the architecture must be scalable since data comes in continuously and possibly
at high rates as more devices are added to the LiRA project. The architecture must be
able to keep up with the rate of incoming data to prevent a loss of data and secondly
avoid a backlog of data building up internally.
Recovery from simple errors is required to reduce the need for manual monitoring of the
system. When more and more pipelines will be running in parallel and around the clock,
errors become more frequent. Manual recovery from errors will become tiresome and
infeasible at scale. If a user expects a pipeline to run overnight, they will be dissatisﬁed
if the pipeline run fails on a simple network error halfway through. A lot of simple errors
can be ﬁxed by the system itself without the need for manual recovery.

4.1.2
Envisioned Architecture
The envisioned architecture seeks to improve upon the initial architecture on three areas:
Using pipeline deﬁnitions to simply conﬁgure data processing, orchestration of individual
steps to process multiple trips simultaneously and to recover from failures during data
processing. The envisioned architecture for the LiRA project remains the same externally,
but the new internal architecture can be seen in 12.
In the envisioned architecture, the processing steps have been split up so that each step
has become its own microservice. In the case of the Pre-Processing pipeline, it means that
each of the steps import, interpolate and map match has become its own microservice.
Splitting up the processing steps into microservices has many beneﬁts over a monolith
with a ﬁxed, sequence of steps, for instance, adding new steps, modifying the imple-
mentation of existing steps and rearranging the steps for a diﬀerent processing pipeline.
Being microservices, the steps are independent of the pipeline runs and can be shared.
It is good practice to make a loosely coupled service, which is achieved.
The independent Processing Step Services need to know when and what data to process
when a pipeline run is executed. The Processing Step Services should only be concerned
with performing the processing, so an Orchestrator microservice has been introduced.
The Orchestrator is responsible for executing the pipeline runs in the correct order of
steps by invoking and waiting for the result of each Processing Step Service. The Or-
chestrator uses pipeline deﬁnitions to dictate the order of processing steps, the use of-
and structure of pipeline deﬁnitions will be detailed later. With the Orchestrator driving
the execution of the pipeline runs, it can also generate high-level logs to monitor the
progress.
Processing step services may crash or fail to execute the processing step. Failure can
be caused by simple problems such as resources being unavailable or errors in the data.
Most of these transient faults can be resolved by simply retrying the step at a later point.
Some failures can also be resolved by redoing an earlier step in the pipeline. A Supervisor
microservice has been introduced to detect and resolve failing pipeline runs. The Super-
visor can detect failing pipelines explicitly by a Processing Step Service reporting error
or by a Processing Step Service running for an unexpectedly long time. Determining if a
Processing Step Service has timed out on a step is often relative to the size of the data
set. The Supervisor should be able to determine an upper and lower bound of processing
time for each step relative to the data set. Once a failure is detected, the Supervisor
should attempt to resolve the failure or report the failure for manual intervention.
To avoid conﬂicting or duplicate pipeline runs, the competing consumers pattern[11] is
used. The competing consumers pattern uses message queues to ensure that when two
18
==PAGE:24==
consumers are awaiting a message, that only one of the consumers receive it instead of
both or none of them. A message queue is a queue of messages managed by a message
broker such as RabbitMQ. Users and services can publish and subscribe for messages
in the message queues to send and receive messages. The message queue ensures that
messages are kept in the order that they were published and that no messages are lost.
Message queues are especially useful in microservice architectures because the publisher
does not need to know who receives the message, which decouples the system. In addition,
if any part of the system crashes, messages are still maintained in the queue as long as
permanent data storage remains intact.
In the situation where the system experiences a high number of requests, the system
should be able to scale to handle the requests by instantiating more service instances.
With the competing consumers pattern[11], the message queue ensures that each request
is only handled once. Using message queues makes communication in the system more
resilient and scalable while keeping the services decoupled. To meet high application
demand, all components of the architecture are scalable by instantiating new instances of
the microservices. Plugging in new instances of services is straightforward; they simply
subscribe to the relevant message queues to become part of the application. In the current
state of the project, scaling the microservices is a manual task, but a load balancer may
monitor pipelines to determine if the rate of incoming messages corresponds to the rate
of consumption and start or stop microservices to meet the demand. This topic will be
discussed further in the Future Work section.
The LiRA Core architecture is very similar to the Scheduler Agent Supervisor pattern[13].
The Scheduler Agent Supervisor pattern is used for executing multiple steps as a whole
process where the steps may be using remote resources. The Scheduler Agent Supervisor
pattern has three roles. The agents are each responsible for executing a step and also
have a compensating action to undo the step in case the step should fail. The Scheduler
is responsible for arranging the agents to execute the steps in the correct order of the
process. The Supervisor monitors the processes to ﬁnd processes that may have failed and
in that event tell the relevant agents to undo their steps. The Scheduler Agent Supervisor
pattern also suggests that the Supervisor should consider a process failed if it takes an
abnormal amount of time to ﬁnish. The actors in the Scheduler Agent Supervisor pattern
use asynchronous communication like message queues.
It is evident that the Scheduler Agent Supervisor pattern has implicitly been implemented
in this architecture with some diﬀerent names. The agent corresponds to the Processing
Step Service, the scheduler corresponds to the Orchestrator and the Supervisor is, of
course, the Supervisor. There are some points where the architecture diﬀerentiates itself
from the Scheduler Agent Supervisor pattern. Timing out steps may not be practical or
necessary in the LiRA project as most of the time is spent on processing which depends
on the amount of data being worked on. A step in LiRA may take anywhere between
ten minutes to several hours, so setting a ﬁxed complete-by may cause more problems.
It may be feasible to have a dynamic timeout depending on the size of the data-set. This
19
==PAGE:25==
will be discussed later in the Future Work section.
The Supervisor Agent Scheduler pattern also states that any remedial actions should be
done solely by the agents. This is a good solution for problems that can be resolved by
undoing the step and trying again, and it keeps a loose coupling between the services.
Some problems cannot be solved by simply undoing the work and trying again, so for
these scenarios the Supervisor may modify the step or insert an intermediate step to
resolve the problem. This requires the Supervisor to have some knowledge of the steps and
resolutions, but the responsibility could also be delegated out to a new microservice.

4.2
Communication
The LiRA Core system uses both synchronous and asynchronous communication between
services depending on circumstance. Synchronous communication is used for common
tasks such as remote database operations and internal and external communication to
APIs where the result of the operation is needed to continue.
The communication to the Orchestrator, and in between the Processing Step Services,
should be asynchronous as the processing steps can take a long time and the Orches-
trator will be processing other messages while waiting for the result. Communication of
messages between services is facilitated by a message broker such as RabbitMQ. Message
brokering adds complexity to the system, but also simpliﬁes many aspects of scalable and
decoupled event-driven architectures[21]. The main beneﬁts of the message queue for the
architecture are that services are easily plugged into the application without disturbing
existing services, messages are not lost and services can easily be scaled because of the
competing consumers pattern that ensures that messages are only processed once.
All internal communication between microservices in the LiRA architecture is through
message queues. With a message broker like RabbitMQ, message queues can be deﬁned
between services as shown in Figure 12. Each type of microservice in the LiRA Core
architecture has a message queue, and all instances of the services will be subscribed to
the same message queue. For example, if a message should be sent to an Orchestrator,
it is simply published to the PipelineControl queue, then the ﬁrst available Orchestrator
service instance will take the message.
When a pipeline run is being executed, the Orchestrators have to invoke the Processing
Step Services, but the Orchestrators should not know which Processing Step Services
exist, to keep a loose coupling, and with the same reasoning, they do not know which
message queues exist.
In AMQP 0-9-1 Model[26], the model used by RabbitMQ for
message queues, there is the concept of message exchanges. A message exchange does not
store messages but distributes messages to message queues based on a routing key. The
routing key is an identiﬁer attached to the message that tells the exchange to which queue
the message belongs. Each type of Processing Step Service has a message queue, and when
an Orchestrator wants to invoke them, it simply publishes the message with a routing key
and arguments based on the pipeline deﬁnition to the ProcessingSteps message exchange
which puts the message in the correct message queue based on the step. Figure 13 is
a visualization of the message exchange and message queues between the Orchestrators
and Processing Step Services. The message exchange keeps the Orchestrator decoupled
from the Processing Step Services, and adding a new type of Processing Step Service
is simply done by creating a new message queue and linking it to the exchange with a
routing key.

Step Processing
The Processing Step Services execute the business logic in a step. A Processing Step
service is a long-living data processing service, that can be used by multiple pipelines. A
Processing Step Service is invoked by sending a processing request by the Orchestrator
as part of a pipeline run. The processing request contains arguments, often describing
the data or the algorithm to modify the behavior of the service. During the execution of
the step, return values can be generated. When the Processing Step Service is ﬁnished
with a step, it will report back to the Orchestrator with the ProcessingResponse message,
28
==PAGE:34==
containing a status indicator and return values if any.
4.5
Orchestration
The Orchestrator services coordinate the pipeline run on a set of data with a given
pipeline deﬁnition. To keep track of each running pipeline’s status and progress, the
Orchestrators have a Pipeline State database where it stores the pipeline runs and pro-
cessing steps seen in Figure 8. The pipeline runs represent the overall state of the run,
while the Processing Steps represent the state of each step in the run. With the pipeline
run and processing step, the Orchestrators can see how far the pipeline run has come and
if the pipeline run has completed or failed. When a PipelineRequest message is received,
the Orchestrator creates a new pipeline run in the database and starts the ﬁrst step in the
pipeline deﬁnition by sending out a ProcessingRequest to the ProcessingSteps exchange.
When the processing step service is ﬁnished, and a ProcessingResponse message is re-
ceived by an Orchestrator, it updates the processing step in the database as ﬁnished. If
the ProcessingResult reports success, the next step of the pipeline is started, otherwise,
the pipeline run is marked as failed and further execution is halted for that particular
pipeline run. The failed pipeline run may later be ﬁxed and restarted by the Supervi-
sor. The Orchestrator services monitor the database continuously to see if any ongoing
pipelines are ready to execute their next step. A pipeline is ready to be progressed if it
has no running steps and none of the previous steps have failed.
As there can be multiple Orchestrators monitoring the Pipeline State database at the
same time, there is the risk of two Orchestrators starting the same processing step con-
currently. To avoid this, a SQL skip-lock[24] or similar should be used to ensure that
if two processes attempt to lock the same row in the database, one of them will skip
the row. The process that has to skip will simply not see the row and therefore gets an
inconsistent view of the data. For the Orchestrators, an inconsistent view of the data is
not a problem.
When coordinating a process with microservices there are two architectural schools of
thought that are interchangeable in most scenarios. In orchestration, a central service is
in charge of distributing the execution of the business process to specialized microservices
and choreography where each of the specialized microservices are aware of their place in
the business process and simply subscribes for events. Choreography is usually preferable
because it creates a more decoupled system and there is no danger of an Orchestrator
becoming a ’god’ service doing all the actual processing but it has the downside, that it
is harder to monitor the process[21].
For LiRA Core, the orchestration architecture was chosen because we are dealing with
a process where each step must be executed in a ﬁxed sequence. This is diﬃcult as all
the services receive the event at the same time and will therefore not run the pipeline in
order.
Though choreography could be used for coordination while maintaining the order of steps,
29
==PAGE:35==
Figure 14: Visualization of the choreography pattern
it would result in higher coupling than orchestration. With choreography, each processing
service would have to know the pipeline deﬁnition so it can invoke the next step of the
pipeline deﬁnition. With orchestration, only the Orchestrators have to know the pipeline
deﬁnitions and the processing services can simply do the work they are supposed to.

4.6
Supervision
The Supervisor monitors and attempts to correct problems in pipelines so they can run
as expected. The Supervisor continually observes the pipelines in progress by the Orches-
trator by also observing the Pipeline State database. The failures that the Supervisors
monitor occur when a Processing Step Service is unable to execute a step successfully.
For example, the data could be invalid, or the Processing Step Service depends on an
unavailable external service. The Supervisors have two ways of detecting failures in a
pipeline; they can either be explicitly failed after a Processing Step Service has failed the
step or the Processing Step Service can be stuck in an endless loop trying to execute the
30
==PAGE:36==
step. The problem resolution of the Supervisor requires that the steps are idempotent.
In the scenario where some steps are not idempotent, the Supervisor should simply leave
the problem to be resolved manually. When the Supervisor attempts to resolve a failure,
it sends ProcessControl messages to the Orchestrator who will then act accordingly to
the control message.
Transient faults are common in distributed systems; often they can be resolved by simply
waiting and trying again. For example, if a network connection cannot be established or
a required service is busy. If a pipeline is failing the Supervisor can restart part of or the
whole pipeline, with the expectation that the fault will be resolved. Complex resolution
of problems is also an option, such as modifying a step, but requires deeper knowledge
about the step for the Supervisor. Some steps such as Interpolation, Map Matching or
Machine Learning can fail if data is not available. Such a scenario could happen if the
data has been deleted during the pipeline run. The problem could be solved by redoing
the Import step, but this would either require that the Supervisor has an understanding
of the relationship between the steps or a description to resolve potential errors in the
pipeline deﬁnition.
Determining if a Processing Step Service has timed out on a step is often relative to
the size of the data set. Some steps have an easily determinable upper and lower time-
bound, but for others, it may require some complex heuristics using the input to the
step to determine the time-bounds. For some services, it may not even be possible to
predetermine the time bounds.

Implementation
In this chapter, the implementation of the architecture will be documented and the
technologies used will be covered shortly. The following sections will cover the details

5.1
Technologies Used
All microservices in the system have been implemented with the Microsoft Asp.Net Core
framework due to its versatility for developing distributed systems, and because the
LiRA Project had already been implemented with this framework at the beginning of
this project. The architecture is not dependent on the Asp.Net Core Framework, and
future components can be implemented in any language as long as it can conform with
the messages exchanged between services.
Asp.Net Core supports the dependency injection pattern for achieving low coupling be-
tween classes and their dependencies through dependency inversion. Dependency inver-
sion makes the application more modular and easier to test and maintain[9] because
classes can easily be replaced
The sensor- and condition-data databases and the PipelineStates databases have all been
implemented in PostGreSQL databases. Asp.Net Core encourages code-ﬁrst database
design through the Entity Framework Core framework that can both be used for con-
struction and later query of the databases, making interfacing between the databases
and services easy. In the code-ﬁrst approach to database design, the application’s do-
main objects are created and ’coded’ ﬁrst and the database is then generated from the
domain model as implemented in the application. The approach is the opposite of the
more traditional approach of designing the database ﬁrst and then coding the application
using the data after.
Each component of the system, be it microservices, databases or message queues, is hosted
in Docker containers to simplify conﬁguration and isolate the component from the rest.
Docker containers simplify the hosting of services as they can easily be created or taken
down.

C#/.NET. For utility, the RabbitMQ Management Plugin has been installed that pro-

any requests. Asp.Net Core has Worker Services[10] just for this purpose. In contrast to
regular Asp.Net Core MVC Services[12] where the controller is recreated for each request,

When a Processing Step Service is started as a new docker container, it subscribes to

5.3.3
Batch Data Processing
Processing Step Services such as the ImporterService, InterpolatorService and MapMatch-
erService perform data processing on large quantities of measurements at a time. Special
behavior has been implemented to increase both time and memory performance of these
steps. Platoon ﬁles refer to a method of collecting sensor data by driving multiple cars
in convoy to collect large amounts of data eﬃciently. Platoon ﬁles can reach sizes of 15
million measurements, depending on the amount of vehicles and time they move.
The naive solution would be to send each measurement to the database, but this would
signiﬁcantly slow down the process due to network delays and errors. Conversely, loading
the entire trip into memory would also signiﬁcantly slow down the process and assumes
that the Processing Step Service has the capacity to ﬁt the entire ﬁle in memory.
36
==PAGE:42==
The solution chosen for this implementation is to fetch and insert the sensor data in
batches. With batch processing, the service will work on a ﬁxed, manageable, amount of
measurements at a time and then insert them without putting stress on the database. The
current conﬁguration fetches and inserts 50.000 measurements at a time, but the optimal
batch size has not been found yet. The batch size has to strike a balance between reducing
the number of network calls to the database while not allocating all the memory on the
host system. Entity Framework Core tracks all changes that have been made to the data
by default. To work in batches; Entity Framework has to be explicitly told not to track
the changes of previous batches, otherwise, there will be no diﬀerence.

5.6
DevOps
DevOps is a set of practices for software development to unify the processes of develop-
ment teams and operations teams to provide continuous delivery and increased quality.
As more microservices are created and integrated into the LiRA Core, each with its own
40
==PAGE:46==
development cycle and requirements, it becomes increasingly necessary to have a uni-
form and automated process of continuously testing and deploying each release of the
microservices. For this reason, part of this project has been to develop a continuous
integration/continuous delivery (CI/CD) strategy for the LiRA Core microservices. For
the LiRA Core project’s ongoing development, it is recommended that the development
team adopts the Gitﬂow workﬂow for strict control of changes and releases. The Gitﬂow
workﬂow is a model of managing git branches for large branches to ensure high standards
of quality in the release of the software. CI/CD is highly recommended for the Gitﬂow
workﬂow. The following sections will discuss a development model using Gitﬂow and the
CI/CD strategy that has been chosen and implemented.
5.6.1
Gitﬂow Workﬂow
Figure 17: Visualization of the GitFlow Workﬂow. The circles represent commits to the
git repositories, and the arrows represent the succession of commits.
The Gitﬂow workﬂow, ﬁrst published by Vincent Driessen[14], enforces strict control over
changes and releases of a large project, with the goal of maintaining a high quality of
releases. As the name suggests, the model builds on the Git version control system. LiRA
Core uses GitLab for Git version control, and the following discussion of the workﬂow is
therefore in the context of GitLab.
The simplest form of Gitﬂow has two main branches, development-branch and master-
branch. The example in Figure 17 will be used to explain the GitFlow workﬂow. The
master branch is exclusively for complete, working versions of the software being devel-
oped. Development-branch branches oﬀthe master-branch and is where the new changes
are made to the software. The development branch does not have a strict guarantee that
the software is working as intended, but once the development branch reaches a stable
point, the changes can be pulled into the master-branch to make a new release.
Besides the two main branches, Gitﬂow uses a number of supporting branches, special-
ized development branches. When a new feature is decided to be part of the project, a
new feature-branch is created oﬀthe development branch where that particular feature
can be implemented and tested. Once the feature is deemed ready, the feature branch
is merged into the development branch. This approach isolates feature changes to the
41
==PAGE:47==
program, resulting in less disruption across parallel development of features and gives a
simple overview of which features have been added in a release. Feature branches also
have the beneﬁt that a feature can be developed without having to be included in the
next release; this means that long-term changes do not aﬀect short-term releases.
Once new features have been implemented in the project, the development team may
decide to create a release-branch acting as a cut-oﬀfor new changes. The release-branch
is created oﬀthe development-branch and has the sole focus of quality assurance before
being merged into the master-branch as a release.
The last supporting type of supporting branch is the hotﬁx-branch. In the event that a
major bug has made its way into a release, an emergency hotﬁx-branch can be created
oﬀthe master-branch to correct the bug.
5.6.2
Continuous- Integration, Delivery & Deployment
When developing microservices, each microservice should live its own development life,
independent of other microservices in the system[21]. Each microservice has a git reposi-
tory for the code relevant to the service to keep the services decoupled. Since the services
share the same domain, a LiRACore repository of shared code between the services has
been created.
Due to the growing amount of microservices in the system, automatic testing and deploy-
ment is required to save time, and because of the simplicity of individual microservices,
continuous- integration, delivery and deployment should be easy to automate.
Continuous Integration
is a software development practice to ensure that a system
never regresses to a worse state by making sure that new changes to the code always
integrate with the existing code. For example, when a new feature is developed, inter-
mediate changes may be made in the codebase. When the feature is ready, it should
integrate correctly with the intermediate changes. In practice, it means that whenever a
new change is made to the code, the code is built and tested at diﬀerent levels.
Continuous Delivery
is a software development practice to easily and reliably release
new versions of the software. If the change to the code passes continuous integration,
continuous delivery is usually followed. Continuous delivery compiles and packages the
code and publishes it as a new release that is ready to use.
Continuous Deployment
is the practice of automatically deploying new releases of
the software to the production environment. In contrast to continuous delivery that only
releases the software, continuous deployment deploys the software. The process usually
consists of dismantling the running version of the software and replacing it with the new
and updating any related dependencies or conﬁgurations.
At the implementation of continuous integration and deployment in LiRA, the choice
of technology was between GitLabs built-in GitLab CI and Jenkins CI. Jenkins CI is a
42
==PAGE:48==
mature and widely used automation tool, but in the end, GitLab CI was chosen as it
promised to provide the same functionality time and eﬀort could be saved by not having
to do the extra conﬁguration of Jenkins. Reducing the number of dependencies on third-
party technologies also decreases the complexity and requirements for conﬁguration of
the LiRA project as a whole.
GitLab also oﬀers built-in ﬁle hosting of NuGet packages and Docker images which would
not be possible with Jenkins without setting up a hosting service.
GitLab uses CI pipelines, not to be confused with the LiRA processing pipelines. Any
discussion of pipelines in this section will refer to GitLab CI pipelines. A CI pipeline is a
script containing a sequence of stages with the intent to execute continuous- integration,
delivery and deployment. In most CI management systems, a pipeline is usually triggered
by a change to the source code, but can also be triggered by other events such as a timed
event. At the LiRA project’s current state, pipelines are only triggered by changes to the
source code or manually.
Each microservice in LiRA Core has a CI pipeline and follows the same pattern of stages:
Build, Test, Release, Deploy. In the Build stage, the microservice source code is compiled,
and if successful, the Test stage begins. In the test stage, the microservice is tested at
diﬀerent levels in a bottom-up approach, starting with unit testing and later integration
testing.
If all the tests are successful the service will be built into a Docker image
that is pushed to GitLab docker image hosting. Now the service can be deployed by
simply downloading and running the Docker image; this makes instantiating services
very quick as there is no need for building the image or compiling code when the service
is needed.

To verify the quality of the system, diﬀerent types of automated tests have been imple-
mented. In agile development, there should always be a working version of the product
available. With automated tests, the chances of releasing and deploying a faulty system
are reduced.
The system is automatically tested at three diﬀerent test levels: Unit testing, integration
45
==PAGE:51==
testing and end-to-end testing. The automated tests are implemented with the MSTest
testing framework.
5.7.1
Unit Testing
Unit testing tests the functionality of individual methods and classes. Unit tests are used
in regression testing to ensure that previously working software does not regress into a
non-working state during later development. In practice, it means that whenever new
code is added, there should be a corresponding unit test that tests that the code works.
Through iterations of updates to the code, the tests should prove whether or not the code
still works. Regression tests works well with continuous integration, as the code will be
tested with each update to the repository.
Unit tests can quickly test classes and methods on diﬀerent input; ideally, the entire code
should be covered by unit testing. Due to time constraints, this project has focused on
unit testing classes responsible for the ’business’ logic of the applications. That is, the
unit tests the Processing Step Services on their ability to process the data correctly, and
for services like the Orchestrator they test that it correctly executes the pipelines and
send out messages.
The unit tests test a single class, where all classes and method that it relies on are mocked.
Mocking means that the class or method is replaced by a fake that can be controlled in
the test for better and easier control. Dependency injection makes this operation very
easy, as the mocked objects are simply injected instead. To mock the objects, the Moq
Framework is used. Mocking the database has made tests a lot easier because the class
that is being tested can simply be given a fake database provider class. When the tested
class uses the mock database provider, the test can verify how many times and what data
the tested class is storing.
Listing 14 shows one of the unit tests for the ImporterService, a test consisting of three
stages. An ’Arrange’ stage where preconditions for the test is setup, in this case, a trip
must exist before the test is run. The ’Act’ stage, where the class method is called.
Finally, an ’Assert’ stage where the method is investigated whether it performed as ex-
pected.
The Moq framework gives the ability to verify, through their mock classes,
whether or not a method was called and how many times. In this unit test, this utility
is used to verify that the trip provider class gets called as expected. The stages have no
real functionality and is simply for the developer to easily distinguish the diﬀerent parts
of the test.

Integration testing tests that diﬀerent components work together. Through integration
testing, each microservice in the project is tested with the components it interacts with.
The test should be isolated from the production environment so that any message queues
or databases required for the test are duplicated and used solely for the test to protect the
production environment. As a result of the integration tests using external components,
they require more time and resources to run as network calls must be awaited and test
data must be created and cleaned up before and after the test. The integration tests
are fewer and more coarse-grained as compared to the unit tests. The integration test’s
primary concern is to test that a request to the service can be handled and that network
messages are formed correctly, while more ﬁne-grained unit tests, that are quicker to
execute and develop, cover internal service logic.

6.3.1
DRY Principle In Microservices
In his book Building Microservices[21], Sam Newman argues that applying the DRY
(Don’t Repeat Yourself) principle across microservices opens up a dangerous avenue for
tight coupling between microservices. The DRY principle is a software engineering prin-
ciple that code should, wherever possible, only be stated once so as to avoid having to
change the same functionality in multiple places and eventually introduce bugs to the
system. Sam Newman argues that the DRY principle should be applied within each mi-
croservice boundary, but shared libraries can cause tight coupling. As each microservice,
in theory, has its own development life cycle, a change in one microservice could cause
unforeseen changes in another when they are tightly coupled.
Through the implementation of the LiRA Core architecture, this advice has been used, as
57
==PAGE:63==
it makes good sense that the Orchestrator should not share any code with the processing
step services, for example. The processing step services, though, all have to implement the
same messaging protocol and message handling classes. These classes were implemented
in the shared library LiRACore, because having the message handling code duplicated
across all the processing step services causes more problems compared to applying the
DRY principle across the services. Of course, the processing step services can always
decouple themselves from the shared library if they need to implement speciﬁc function-
ality.
The key takeaway is that Sam Newman is correct that microservices that are of diﬀerent
nature should not have shared code as it can quickly cause a lot of problems, but the
DRY principles still holds for some scenarios where microservices are very similar.

Future Work
During the writing of this thesis, a number of potential improvements have been discov-
ered, but due to time and scope constraints, they have not been investigated further. In
this section, each improvement will be discussed in regards to their beneﬁts to LiRA.
Hopefully, some of these improvements can serve as an inspiration to later Bachelor and

7.1
Compensating Actions
The current implementation of microservices uses transactions to ensure atomicity when
they modify data in the databases. According to the ACID principles, when one trans-
action is in progress on a set of data, any other transactions wanting to modify the data
will have to wait.
An alternative to transactions is compensating actions. With compensating actions, ev-
ery action that changes the data should have a compensating action that can undo the
initial action. A sequence of actions can thereby be undone be applying the compensat-
ing actions in the reverse order. By using compensating actions, multiple threads can
modify the data concurrently, which means that the compensating must only undo the
work of the initial action and not the work performed by the other threads. Just like the
initial action can fail, the compensating action can also fail. If the system cannot recover,
manual resolution may be required.
The downside to using compensating actions is that they can become very complex or
may not be able to undo certain actions. Given that multiple threads can work on the
same data, it can be diﬃcult to determine in what state the data should be.
7.2
Database Bottlenecks
In large distributed systems, a centralized database can easily become a bottleneck for
the whole system. A centralized database, is easier to design and manage compared to
a distributed database but has the disadvantage that it can become a bottleneck during
high traﬃc. The alternative is to distribute the database onto diﬀerent machines. There
are two types of distributed databases, homogeneous and heterogeneous.
The homogeneous distributed database is simply a perfect clone of the initial database.
Migrating to a homogeneous distributed database is simple, a copy is simply made of
the ﬁrst database, and any subsequent changes are applied to both databases. In a dis-
tributed environment, changes may not be applied at the same time in both databases,
leading to data inconsistency. The databases will, however, become consistent eventually.
The heterogeneous distributed database distributes the data by tables. There is no repli-
cation of data, but the data is in diﬀerent locations. Heterogeneous distributed databases
are often used in microservice applications. A microservice should have a clearly deﬁned
area of responsibility and in the same line of thought, the microservice should also ’own’
the data related to it. Letting the microservice ’own’ its data means that the data will
59
==PAGE:65==
only be changed by the microservice. Making changes to the database structure is also
easier as the changes will only aﬀect the one microservice. In comparison, when a central-
ized database is used in a microservice environment, changes to the database structure
may cause problems in multiple microservices due to tight coupling.
The LiRA project is still in development, and using central databases has been the quick-
est and easiest solution at the time. LiRA Core has three databases at the moment, the
PipelineState database, the sensor data database and the condition data database. High
loads may be put on the system to keep up with the incoming data, it is recommended
that future projects look into distributing the database to give the processing step ser-
vices ownership over the data they use and create.
Sensor data is collected in large quantities, and additional distribution may be required.
Data partition such as ’sharding’ distributes tables by rows on diﬀerent machines while
keeping related data together. An example in LiRA could be to store trips of the same
geographic area on the same machine or ’shard’. Sharding the data by geography may
be useful in LiRA as data from the same area is usually compared. The data could also
be partitioned by the time it was collected.
7.3
Dynamic Microservice Scaling
The microservices in LiRA Core are currently scaled manually by deploying more Docker
containers to compensate for high loads on the system. For better management of re-
sources and to eliminate the manual task, it is recommended that the microservices be
scaled automatically based on the load of the system. RabbitMQ has live diagnostics
on the number of messages in each queue, how long each message has been waiting and
many more diagnostic points that can be utilized to determine the strain on the system.
A microservice could be implemented to monitor the pipelines and automatically deploy
more or less Docker containers if the messages are not processed quickly enough.
7.4
Parallel Trip Processing
Processing multiple trips in parallel is possible with the new architecture. It is believed
that individual trips can be segmented and each segment processed in parallel. Some
data sets, like the Platoon Data ﬁles, are very big and take a long time to process. If
the trip is partitioned and the parts are distributed over multiple threads, the trip will
be processed quicker. Interpolation uses multiple points to generate the missing GPS
positions, if some of the points needed for the algorithm is not in the partition, they can
be ’borrowed’ from the other partitions. As long as they are guaranteed not to change,
race conditions should not be a concern.
60
==PAGE:66==
7.5
Processing Step And Batch Optimization
The processing steps have been implemented to process the data correctly, but there may
be better implementations or algorithms to give the same functionality at a slightly better
performance. The steps are also retrieving, processing and storing data in the database
in batches because it sped up the processing time signiﬁcantly. Finding the optimal batch
sized for each step could potentially give a signiﬁcant performance increase.
7.6
Complex Supervision and Recovery
The Supervisor is very simple at the moment, but it has the potential to ﬁx a wide range of
problems that can arise during a pipeline run. Further work in this area should look into
ﬁnding a way to describe failure states and resolutions that the supervisor can interpret.
Some resolutions may be to simply restart the failed step with diﬀerent arguments, but
other more complex resolutions may be to execute a previous step again. Failures in one
step can also indicate a failure in a previous step, for example, a failed interpretation step
could follow from a failed import step if no measurements were stored in the database.
The Supervisor may not be able to solve every problem, but it could assist in manual
recovery by generating a report of the error that may help the user to ﬁnd the error and
resolve it.

the architecture has to solve. Using the knowledge from the domain analysis along with
the short analysis of use case scenarios, an improved architecture was designed through
many iterations. The designed architecture satisﬁes the research questions put forth in the
introduction by being capable of handling large amounts of data, being easily extensible
with processing pipelines and being able to handle simple failures during processing.
The architecture achieves scalability through microservices and message queues, which
combined allows each of the processing steps to be duplicated to process the data at a
higher rate. A messaging protocol was also designed for inter-service communication.
Making the system extensible with new processing pipelines was achieved by pipeline
deﬁnitions, letting the user easily deﬁne the order of steps and supply with the necessary
arguments to run the pipeline. The Orchestrator is responsible for running the pipeline
as deﬁned in the pipeline deﬁnition. The last question of handling simple failures is to
some part solved by the Supervisor, but it is expected that improvements can be made
to make the Supervisor more capable.
The system has been implemented and deployed, namely with Asp.Net Core, Docker and
RabbitMQ. A development strategy has been suggested for further development, based on
the Gitﬂow Workﬂow and automatic Continuous -Integration, -Deployment and -Delivery
pipelines. Continuous integration is realized with unit- and integration testing at every
commit to the source code repositories.
Finally, the architecture has been evaluated based on an interview with a data scientist
working with LiRA as an end-user of the system, and a performance stress-test to evaluate
how the system performs under diﬀerent loads.
This project presents the foundation for a highly modular microservice architecture. As
covered in Section 7, there are still things to expand and improve on the architecture, but
as it is, the architecture successfully performs its purpose of processing large amounts of
road maintenance data.
