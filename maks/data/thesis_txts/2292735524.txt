Summary (English)
The goal of this thesis is to explore alternatives to the now unsupported
product Microsoft InfoPath for companies where it is used as a electronic
forms platform. This will make it possible for companies to understand the
role InfoPath plays in their solution and what their alternatives are. This
will make it possible for companies to schedule a replacement of InfoPath,
to a different product.

1.2
TAP
TAP[A/Sb] is a well-structured, modern, ﬂexible system for business process
management based on service oriented architecture (SOA). TAP is conﬁg-
urable and built upon a standard platform, which allows for a fast and se-
cure implementation process. TAP has a large number of standard functions
and components, which is set up to support an organisation’s individual re-
quirements, work processes and business rules. It is also possible to add
custom developed components to the solution, which can be used in con-
junction with existing systems. TAP is scalable and can be implemented in
both large and small organisations. The system uses and builds upon the lat-
est technology from Microsoft and is an open system that can be integrated
with other solutions.
One of the technologies used in the TAP system from Microsoft, is InfoPath[otb].
Microsoft InfoPath is used to design and ﬁll out electronic forms, such as
expense reports, time cards, surveys, and insurance forms. The design is
done by using standard form controls, such as text boxes, list boxes or cus-
tom controls. When ﬁlling out a form in InfoPath, users can use familiar,
document-like features, like check spelling in their form or insert formatted
text and graphics into certain ﬁelds. If InfoPath is combined with Microsoft
SharePoint Server running InfoPath Forms Services, it enables browser-
compatible form templates in InfoPath and enable them for use on internal
and external Web sites.
Users aren’t required to have InfoPath installed
on their computers to ﬁll out a form, nor are they required to download
anything extra from the Web, all users need is access to a browser.
1.3
Motivation
Today the TAP system has a hard dependency on InfoPath and InfoPath
Forms Services, by using them as electronic forms, for users of the TAP
system to ﬁll out, in their external facing web component. The motivation
for ﬁnding an alternative to Microsoft InfoPath is, that Microsoft have dis-
continued InfoPath as of the 31-01-2014[otc], and have not provided an al-
ternative. An alternative would not only move the TAP system away from a
discontinued product, but would also beneﬁt TAP by:
Reduced license cost
==PAGE:17==
1.4 Vision
3
TAPs web component uses InfoPath Form Services for SharePoint, to
publish and show the InfoPath electronic forms to the users of the web
component. This is the only scenario for SharePoint in TAP. With the
removal of InfoPath there is no need for SharePoint, and the license
cost will be reduced.
Reduce complexity
The web component needs to integrate with a SharePoint component
to display the InfoPath forms, also the InfoPath forms needs to be pub-
lished before they can be displayed on the web component. With the
removal of InfoPath the need for publishing and integration with an
external component will also be removed, and reduce the complexity.
Update the software stack
An introduction of a new component into the TAP system, will provide
an opportunity to update the software stack, this could beneﬁt the
system by handling new use cases found and removing annoyances by
the existing solution.
1.4
Vision
The vision for this project is to ﬁnd a replacement for Microsoft InfoPath and
create a migration strategy for the replacement in TAP. This should enable
TAP to deviate from competition, and have a clear plan on the future with a
new component to handle electronic forms.

2.1
XML and XSD
XML is a simpler and easier-to-use subset of the Standard Generalized Markup
Language (SGML), which is the standard to create a document structure
created by IBM based on their proprietary GML format[sta].
The basic
building block of an XML document is an element, deﬁned by tags. Tags
are deﬁned by an opening "<" and a closing ">" with the tag name in be-
tween e.g. "<FirstName>" xml elements has a beginning and an ending tag
where the ending tag is deﬁned by "/", e.g: "</FirstName>" or "<FirstName
/>". All elements in an XML document are contained in an outermost ele-
ment known as the root element. XML can also support nested elements, or
elements within elements. This ability allows XML to support hierarchical
structures.
Element names describe the content of the element, and the
structure describes the relationship between the elements.
XSDs stands for XML Schema Deﬁnition, and are a way to describe the data
types that an XML document contains. Since XML is a portable data format,
so is XSDs a portal type deﬁnition, unlike some programming language,
like C# and Java, where types deﬁned in the speciﬁc language is limited to
that language only. XSDs do not support behavioral polymorphism like ob-
ject oriented programming language and XSDs do not support the concept
of behavior, but are designed to represent representational polymorphism,
where you can have a abstract format for a data type that then can be de-
rived into specialized types[sta].
2.2
What is Microsoft InfoPath
Microsoft InfoPath was designed, at its core, as a powerful XML editing en-
gine that enables end users to interface easily with data, this enables users
to design and ﬁll out electronic forms containing structured XML data. This
is done by using standard form controls, such as text boxes or list boxes,
or insert controls that offer users the ﬂexibility to add, remove, replace, or
hide sections of a form to make a richer user experience. If InfoPath and Mi-
crosoft SharePoint Server running InfoPath Forms Services are combined,
it is possible to design browser-compatible forms in InfoPath and enable
them for use on web sites. Users aren’t required to have InfoPath installed
on their computers to ﬁll out a form, nor are they required to download
anything extra from the Web. All users need is access to a browser. When

TAP and InfoPath
2.2.1
InfoPath Forms Services
InfoPath Forms Services is a server technology that makes it possible for
people to ﬁll out forms in a browser, it also enables you to store and man-
age forms and form templates in a centralized location. When accessing a
form in a browser, the InfoPath menus and toolbars are replaced by a tool-
bar across the top of the form, which displays options for saving, closing,
printing, and updating the form.
This bar also appears at the bottom of
a form, so that users don’t have to scroll back to the top of a long form
to complete their form.
If the submit options for your form template is
enabled, users also see a Submit button on the toolbar, when the submit
button is clicked, the form data is submitted directly to the data source that
you speciﬁed when designing the form template, such as a Web service. If
the form template has more than one view, users will also see a View list on
the toolbar.[ota]
2.2.2
InfoPath Designer
InfoPath Designer is used to create and publish InfoPath form templates.
When designing a form template ﬁle, you create a single ﬁle that contains
the supporting ﬁles that implement the layout, views, and logic upon which
a particular InfoPath form solution depends.
When users ﬁll out a form,
they are actually ﬁlling out a form instance, which is based on the associ-
ated form template. InfoPath Designer lets users quickly create forms that

3.2
Goals
When developing a new product, a fault is to specify solutions rather than
requirements; this phenomenon is called premature commitment. Prema-
ture commitment to a particular solution often rules out other, potentially
better solutions. Better may be referring either to quality attributes (e.g.
faster, more reliable,and modular) or to project objectives such as develop-
ment cost and duration. People also have a tendency to be carried away
by system-in-the-making, specifying more and more features that have little
or no beneﬁt to the client. One way to reduce premature commitment is
using goal analysis and linking all subsequently requirements to the deﬁned
goals.
A goal is a prescriptive statement of intent the system should satisfy[Lam09],
deﬁned as a description of what a stakeholder wants to change or improve
in his business environment. It is important that a goal refer to the system
as a whole, not to individual features and/or parts of the system and their
properties. Examples of well-deﬁned goal is: "Increase cost efﬁciency" or
"Increase more sale to customers". In contrast a invalid goal is more spe-

to customers before they can pay".[Sch06] Goals will ensure that every fea-

3.3.4.2
Interviews
The interview method avoids many of the problems that the surveys have,
there exist two types of interview approaches:
• Structured - Have a set of predeﬁned, but rather open questions
• Qualitative - Set the top but don’t predeﬁne the questions
In the Structured interview type the interviewer asks a question, and the
subject can then answer as they wish. If there is a need for more insight
the interviewer may probe for more detail, but does not set the terms of the
interview. The issues of whether the question asked can be answered at all,
and whether the question asked is valid and understandable. The issue is
that people know how to do many things that they cannot describe. It is
a commonplace in ethnography that people’s descriptions of how they do
perform certain tasks, bear a complex and opaque relation to how they can
be seen to do the tasks. In social science this problem is called the say-do
problem[Gog93]. When interviewing people it is then important not to ask
people to describe activities that they don’t normally describe, or if done,
don’t believe the answers.
Another problem that an interviewer can face when using the structured
interview, is that people answer the question to what they think the inter-
viewer wants to hear, or that their description don’t match the way they
perform the task.
The structured interview also demands that the inter-
viewer have some insight into the work ﬂow of the subjects and have made
a decision on what to focus on. The subject could very much disagree with
the selected focus and not feel that it is the most important, or that it don’t
capture what they need. The qualitative interview type helps migrate some
of these problems by shifting some of the work unto the subject, by making
them dictate the questions. The interviewer can then steer the subject in
the direction by selecting topics and asking more for more details. The risk
with this interview type is the lack of coherence between different subjects
and sharp focus on speciﬁc areas.

5.1
SOLID
Solid is a state that matter can have, like ﬂuid and plasma, and when think-
ing about a solid, a normal thought would be unbreakable, a solid matter.
But solids can be very brittle like glass, and can break easily, and that is
not a state that is desirable for the code base. So the aim, from a develop-
ment point of view, is not to make the code solid, but to create supple code
where it is possible to shape it into different forms and change the form if
it is discovered that the original shape is wrong or not optimal. The rea-
son for having this desired state is that you never have all the requirements
op front, even if it is thought that all the requirements have been stated,
they will almost always change as the development is making progress. The

SOLID principles make it so that the code is supple and is easy to change
when new requirements are found.[Bob]
5.1.1
Single responsibility principle
Every class should have a single responsibility
Another way to state what SRP is to say that a class should only have one
reason to change. This will create a separation of concerns and reduce cou-
pling and complexity, by handling each of the concerns independently. This
will also make refactoring of the code easy, because there are no crossover
of requirements.
5.1.2
Open/Closed principle
The class should be open for extensibility but closed for modiﬁcation
This principle has two primary attributes, a class should be open for extensi-
bility, meaning that the behavior can be extended, to make it behave in new
and different ways to conﬁrm to changes in requirements. At the same time
a class should be closed for modiﬁcation, meaning that the code of the class
is inviolate, so no changes to the code is allowed. The reason for having this
strict rules is to reduce ensure that clients that now depend on the class
can have some predictability, since changes in the class could have rippling
effect to clients that are dependent on your client.
The principle has a very hard dependency on inheritance, where the prin-
ciple could be achieved by using base classes and inheriting from them. In
the mean time it has been discovered that inheritance ain’t a good way to
model object oriented design, and composition should be favored over in-
heritance. One way to follow the open/close principle and use composition
instead inheritance is to use a design pattern called strategy, see 5.2.1.
5.1.3
Liskov substitution principle
States how polymorphism should work
==PAGE:93==
5.1 SOLID

The Liskov principle was described by Barbara Liskov as:
What is wanted here is something like the following substitution property:
If for each object o1 of type S there is an object o2 of type T such that for
all programs P deﬁned in terms of T, the behavior of P is unchanged when
o1 is substituted for o2 then S is a subtype of T.
This is a mathematical way of saying that subtypes must be substitutable
for their base types. The easiest way to understand this principle is when
the consequences of violating it is shown. As stated above, if S is a sub-
type of T, the client that consumes T should be able to substitute T with S
without altering the clients correctness. Correctness is used instead of be-
havior since correct behavior is application speciﬁc where correctness is a
more abstract term. In this example if the client switches T with S the client
should not crash, if it crashes T and S violates the principle. Another viola-
tion is if S removes a feature that is present in T and the client depended on
this feature.
5.1.4
Interface segregation principle
States how interfaces should be designed
When designing interfaces it is important to think about who owns the in-
terface, is it the concrete implementation of the interface or is it the client?
With this principle it is the client that owns the interface and it states that:
"Clients should not be forced to depend on methods they do not use" When
thinking about this, this will reduce the number of methods and responsi-
bilities a interface have (Think SRP). This is also called "thin-interfaces". It
is important to understand that using thin-interfaces, will normally increase
the number of interfaces, but it will also make them simple and more fo-
cused.

DIP states that: "high-level modules should not depend on low-level mod-

Both should depend on abstractions" And "Abstractions should not
depend upon details. Details should depend upon abstractions"

5.2
Design patterns
Design patterns is not a software concept, but has it origins from construc-
tion they have been deﬁned by Christopher Alexander as:
"Each pattern describes a problem which occurs over and over again in our
environment, and then describes the core of the solution to that problem, in
such a way that you can use this solution a million times over, without ever
doing it the same way twice."[Fow02]
The gang of four book [Gam94] then deﬁned patterns from a software stand-
point as:
"A design pattern systematically names, motivates, and explains a general
design that addresses a recurring design problem in object-oriented sys-
tems. It 60 Analysis of Patterns describes the problem, the solution, when
to apply the solution, and its consequences. It also gives implementation
hints and examples. The solution is a general arrangement of objects and
classes that solve the problem. The solution is customized and implemented
to solve the problem in a particular context."
A pattern does not provide a concrete implementation, but provides an ab-
stract description of how a general arrangement of classes and objects can
solve a design problem.
5.2.1
Strategy Pattern
The strategy pattern allows for the client to chose between behavior from
the same contract. It could be an interface that had an action called "Sort()".
Then different sorting algorithms would implement the interface, allowing
for the client to chose between the different algorithms by always knowing
that it should just call "Sort()".

5.2.2
MVC
The Model-View-Controller pattern separates the domain, the view, and the
actions based on user input into three separate modules, each having its
own responsibility.
• Model manages the behavior and data of the application
• Controller Captures inputs from the user, informing the model and/or
the view to change
• View Manages the display of information to the user
Both the view and the controller depend on the model, however the model
do not depend on the view nor the controller. This is one the key beneﬁts
of the separation. The separation allows the view and controller modules
to be tested independent by mocking the model, and the model to be tested
independent of any other logic.
5.2.3
MVVM
MVVM is a variation of the MVC pattern(5.2.2) where the view is further
abstracted away from the underlying code. It was developed at Microsoft
for their new graphical stack WPF and Silverlight, where the view wasn’t
meant to be handle by a developer, but by a designer creating the view in a
declarative language like XAML or HTML. The model is the same as in MVC;
it is the data or business logic, completely independent from the UI, which
represents the state. As stated above the view is abstracted away consists
of graphical elements. Now the view and the model needs to communicate,
and that is handled by the ViewModel, this is done by binding. It is possible
to bind the View directly to the Model and parts of the Model are simply
displayed in the view by one-way data binding. Other parts of the model can
be edited by directly binding controls two-way to the data, so that changes
in the UI propagate down to the model. [Joh]

AngularJS
AngularJS is a JavaScript MV* framework, where the * stands for more than
one design pattern.
AngularJS mixes the two design patterns, MVC and
MVVM, so AngularJS have the notions of models, controllers and views,
from the MVC pattern, but it also have view-models from the MVVM pat-
tern. This gives AngularJS two-way data binding so that changes in a view
is automatically propagated down to the controller. This decoupled design
also enables AngularJS to make heavy use of dependency injection.
This
makes apps written in AngularJS more maintainable and testable since it
decouples code and uses dependency injection to build a coherent applica-
tion. AngularJS differs from other JavaScript MV* frameworks by:
• Introducing markup in the HTML
– This is positive since it integrates easy with existing applications,
because the markup is normal text that are evaluated once the
DOM is done loading
• Uses plain old JavaScript objects as the model
– Usage of JavaScript objects for the models reduces the need for
new knowledge and makes integration with existing data sources
easy
• Heavily leverage of dependency injection
– Increases the testability by decoupling the code
AngularJS do not depend on third party scripts like jQuery, but AngularJS
have support for any jQuery plug-ins.
All of this makes AngularJS, easy
to maintain, ﬂexible and testable. It also uses the client-side architecture
pattern and have a large community tougher with a strong company backing
in the form of Google.

8.3
Conclusion
It is concluded based on the outcome of this thesis, that it is possible for TAP
to replace Microsoft InfoPath with an in-house developed solution. This is
accomplished by developing a web application using AngularJS. The entire
solution is made up of three different components, all build following the
SOLID principles to allow for future changes in the code base. Two of the
components are build in AngularJS, one application to display the electronic
forms and the other to create the electronic forms. The FormDesigner will
guide the creator in designing the form by allowing for drag and drop if
different form components, all combined into a single electronic form. The
data of the form will be based on JSON Schema, and the FormDisplay will
generate the HTML and CSS based on the JSON Schema. The last compo-
nent is created in .NET and allow for interoperability between the different
components in TAP, by serializing and serializing to POCO and XML.
This is demonstrated by case studies where the proof of concept created
FormProject, showed that an in-house developed solution would provide the
best solution for TAP. The solution was tested in both existing setups and
thought of future setups, allowing for TAP to use the solution today and
know that it is future proof.
The choice of platform and technology allow TAP to display electronic forms
cross-platform without compromising with the possibilities, that its devel-
opers are accustomed to the Microsoft stack, especially ASP.NET and WPF.
It will also allow TAP to invest more in the Microsoft stack by adapting
Microsoft’s Typescript language.
Since Angular and web development is
moving so fast the developers involved with carrying this project forward
will not be accustomed to the new ways of creating apps, that will affect the
development time, until they get more familiar with the technologies, but
it would not have been possible to meet the requirements and vision of the
project without these.
The solution is not at a stage where it is production ready, and there are still
uncertainties with the workﬂow of the form creators. Future work to make
the solution production ready is discussed on section 8.4.
Overall it can be concluded that the outcome of this project has shown
TAP that they should develop an electronic form solution themselves and
it should be based on the web as a platform and technology.

Development
The starting point of the development will be based on the two opensource

faster development since some of the concepts can be seen and tested, but
since it will be a very complex solution, the language used should be Type-
script. This will force TAP to develop the features from scratch and increase
the initial cost of the project, but it should reduce the cost and complexity
in the long run, allowing for quicker changes to the code base and imple-
mentation of new features.
