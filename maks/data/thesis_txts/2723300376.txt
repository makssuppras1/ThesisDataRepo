Master Thesis
Abstract
Reliability is one of the biggest concerns about communications nowadays; communications
which are not restricted between humans but also expand between devices. This Thesis
focuses on the communication between Terminals and Drones using 5G and the evaluation
of its’ reliability through simulation scenarios. To achieve that, the Simu5G simulation tool
is used, and the user modifies the "SinglePair-UDP-Infra" example for D2D communication.
Before reaching the simulation part of this Thesis, the reader is introduced to Drones tech-
nology, their different types and usages, and the different ways they can communicate with
the infrastructure but also with each other in Drone Swarms. The communication technolo-
gies that can serve the Drone to Terminal communication are analyzed and compared so
that a user can decide which one suits best in any use case. In the last part of the The-
sis, the different simulated scenarios are presented through generated graphs. Nevertheless,
although the results can lead to the inference that the communication is reliable, this can-
not be considered realistic. After further analysis, many inadequacies and omissions of the
simulation tool were revealed, rendering the results less valid.
On the whole, this Thesis serves two purposes; Firstly gives an introduction to Drones
and communication technologies, and secondly can be considered the first approach to the
reliability of Drones-Terminal communications using 5G.

Introduction
Throughout human history, communication has always played a crucial role. Communication
helped people to achieve important things such as organizing in societies, exchanging ideas,
work together on common goals, which contributed to the evolution of humankind. Different
techniques and means of communication were developed through the centuries, but things
became more complicated during the second half of the 20th century. The technological
evolution that took place brought the need for people to communicate not only with each
other but with machines. In addition to that, this evolution brought IoT to the spotlight. IoT
is the communication between different devices, exchanging information and data through
an Internet connection. More and more devices started to become "smart", being equipped
with specific sensors and software, allowing them to communicate with each other through
direct communication links or through infrastructure.
The development of new devices like smartphones, sensors, UAVs, etc., introduced new fea-
tures and, as a result, new communication requirements. The rapidly increased number of
users, devices, and exchanged information implied the increased need for coverage and ca-
pacity. In many cases, the devices themselves are used to cover these needs. For instance,
UAVs can be used as relays, flying individually or organized in Drone swarms to extend the
coverage and provide capacity when required. In addition, IoT made many very complicated
and expensive applications that demanded professionals operate them more accessible and
affordable to the mass market. Applications like these could be surveillance, aerial photog-
raphy, weather monitoring, and even delivery services.
Last but not least, the kind and importance of the information and data transferred brought
the need for Ultra Reliable Low Latency Communication. Different communication technolo-
gies are used per case to ensure connectivity between the devices and meet the reliability
requirements. Some of these technologies will be presented in the following chapters of this
Project.

1.1
Objectives
The main goal of this Thesis project is to simulate and evaluate D2D communication through
a base station using 5G communication technology. More specifically, the 5G communication
between a UE and a drone through a base station will be analyzed. To achieve that, the
Simu5G [1],[2] is used. Simu5G is a network simulator built on OMNet++ tool libraries.
Since Simu5G was not initially created for UAV simulation, it needed to be modified before
being used, and the user had to enter the appropriate values according to the investigated
scenarios.
As a result of time limitations, available resources, and lack of previous experience in sim-
ulation, the created model cannot be considered an absolutely valid model which can be
used widely, providing a final solution to the concept of "Reliable Communication Between
Terminals and Drones using 5G". This Project is a first approach to the problem, and future
work should take place to make this research more valuable and reliable. Other objectives of
this Thesis Project are to introduce the readers to several technologies that are widely used,
such as Drones, communication technologies, and drone swarms, by providing an overview
of them.

Reliability
Before trying to focus on and analyze Reliability and its’ importance for a network, network
Resiliency needs to be introduced. In general, creating a system or network that never fails
is impossible. Everything fails, especially these days when structures are far more complex.
What needs to be done is to keep the time duration that a network is out of service as short
as possible. This is where the term Resiliency is introduced.
Resilient is a network or a system in general that can provide continuous operation, and it
is defined by resistance to disruptions and rapid recovery in case of failure. The two most
important aspects of Resiliency are Availability and Reliability. There are many different
given definitions of Reliability, and there are many more people that are mixing up Reliability
and Availability.
Availability is the probability of a system to operate at a certain point in time. It can be
defined as A= (System Uptime/ System Operational time). To calculate the average avail-
ability of a network, variables like MTBF and MTTR need to be taken into consideration,
which according to [3]:
• MTBF: "It is the average time between repairable failures".
• MTTR: "It is the average time it takes to repair a system".
Using the above metrics, Availability can be calculated as: A=1-(MTTR/MTBF) and Aav-
erage= MTBF/(MTTR+MTBF).
*In this Thesis Project the main focus is on Reliability and there will be minor
references to Availability.
Reliability as it is defined on [4] is "the probability of a system or a sub component functioning
correctly under certain conditions over a specified interval of time".
Reliability is of significant importance for a network, so it is crucial to ensure it. Nevertheless,
as a network consists of different components, it becomes complicated to provide Reliability.

A way to assure the networks’ Reliability is to ensure the Reliability of each entity of it. As
a result, the Reliability of the components will bring the Reliability of the whole network.
A way to refer to the availability of a network is by referring to the "nines" that come with
it. The more "nines", the more available the network is. However, it is not meant that
Availability and Reliability are always side by side. A network can be available but not
reliable.

There are many ways to predict and design a network’s reliability and many available tools.
Some of them are Protection and Restoration, Reliability block diagrams, Fault tree, Markov
chain, and Bayesian network models. Simulation is also playing a vital role in predicting
and testing the reliability of a system and is a handy tool for engineers [4].

Reliability is essential in communication as well. Providers and customers are always looking
for reliability during a phone call, an online meeting, when controlling drones, or when using
health care applications, i.e., applications for people with heart disease which log the patient’s
pulses and notify emergencies if something is out of the average level that the doctors have
set. It is imperative not to experience delays or interruptions during operations like the ones
referred to before, which might have minor to major consequences, even death, in the case

Cellular
This division includes radio communication technologies such as 2G, 3G, 4G,5G, which make
D2D communication feasible. All these technologies are part of the same family, and one is
the evolution of the other. However, this does not mean that the arrival of the new version
cancels or replaces completely the previous one. There are many applications where the old
generations of cellular technologies are implemented.
2G, stands for second-generation cellular networks. It was deployed in the early 90s and
brought a revolution to the way of communication. Set on GSM (Global System for Mobile
Communication), it introduced the digital signal for making calls and replaced the analog one
used until then. It also developed new features like SMS, MMS, and data packet transmission.
On its first version, it could support data rates up to 64Kbps, but in the subsequent enhanced
versions of it, GPRS (2.5G) and EDGE (2.75G), it could offer data rates of up to 384Kbps
[19], [20]. 2G is still in use and will probably be for many years, as it is used in IoT setups,
is exceptionally cheap, and includes all kinds of devices.
Following 2G, third generation 3G took over. 3G is an enhancement of 2.5G. Its set of
standards meets the requirements and is compatible with the ITU-2000 specifications package
from the International Telecommunication Unit (ITU). It introduced several new applications
and services, the most important of them being: Internet access on mobile devices, VoIP
telephony, and video calls. It offers higher data rates of up to 2Mbps and operates on a
broader spectrum of frequencies. Improved authentication and security of communications
was another significant change [19], [20].
The increasing need for capacity from the applications brought the need for an evolution
of the network technology in use. Because of that, LTE or 3.95G was introduced. Among
others, LTE provided higher data rates of up to 30Mbps for the users, lower latency, and
improved the user’s experience. However, in contrast to the marketing used by Telecom
providers, promoting LTE as a 4G network, this is not the case. LTE or 4G LTE, as it can
be referred to sometimes, does not meet the requirements set by ITU for the 4G network.
4G network is an IP-based cellular technology implemented on the existing LTE, but it is

5.1
Connectivity types
The Direct connectivity is the simplest way of establishing a communication connection
between two entities, in the investigated case, between a drone and a terminal.
In this
scenario, the two devices communicate directly through an established link without any
external elements involved. Bluetooth and/or Zigbee are technologies that can be used in
this type of communication to provide connectivity.
Nevertheless, things are more complicated when cellular technologies are applied in the
communication between the drone and the controlling terminal.
In this case, as shown
in figure 18, in design number 1, a base station is included in the communication. Both
segments are linked wirelessly to the base station. At least two communication channels
(also relying on the duplexing mode) are established. The signal is first transmitted from
the terminal to the base station, then from the base station, is sent to the drone. When the
drone responds to the terminal, the reply follows the opposite direction.
These two types of connection are mainly used when the drone operates in the "Line of
Sight" (LOS). LOS is the definition used to describe the kind of communication where the
remote pilot can clearly see the drone while the latter is flying. This is a distance of some
hundred meters away from where the controller is located.
On the other hand, there is also communication "Beyond Visual Line of Sight" (BVLOS).
This is the case where the flight operation is initialized at the LOS, but the distance from
the remote pilot is increased so that, at some particular moment, the sight of the drone is
lost. How can communication still exist after that? That moment is where the connection
through multiple base stations takes over.
Initially, the procedure is the same as in the single base station case; the signal is first
transmitted from the terminal to the base station. Then in the single base station scenario,
the base station forwards the signal to the drone. However, in the multiple base stations
instance, after the first base station, the signal is transmitted to another base station (and
then to another one if needed) before finally reaching the drone.
Page 27 of 67
==PAGE:46==
Master Thesis
The drone is connected to one base station as long as it flies in its coverage area and then
connects to the next most immediate one. It could be said that there is a signal "hoping".
This kind of connection increases the flying capabilities of the drone as it increases the
distance it can reach. In this way, drones can serve purposes other than entertainment, like
delivery of goods, rescuing, etc., as it was referred to on 3.3.
5.2
Drone Swarms
Apart from the capabilities and beneficial uses of drones discussed in Section 3.3, there are
also other ways that drones can be used in order to serve needs. One of the most known
operations where drones can be very helpful, provide solutions to crucial problems and have
a positive impact, is the Swarms.
Swarms, as it was presented in [27] by Mitch Campion, Prakash Ranganathan, and Saleh
Faruque "is generally defined as a group of behaving entities that together coordinate to pro-
duce a significant or desired result". However, Swarms are not only met in drone operations.
There are many domains in everyday life where groups work together to achieve a common
purpose. Humans are the best example as throughout the history of human beings, groups of
people with the same vision worked side by side to solve a problem, get out of a problematic
situation, evolve, and meet their goals. Ants are another representative example of group
coordination in favor of a common target; i.e., they analyze the data from the environment
and split up into teams to carry their food or build up and expand their colony. In this way,
the common goal of surviving is achieved. Similarly to that, Drones can exist in Swarms,
communicating and cooperating with each other, not for their survival but for executing
what they were programmed to. In Drone Swarms, the are two kinds of connection and
communication; the U-T-I (UAV to Infrastructure) and the U-T-U (UAV to UAV).
UAV to Infrastructure communication applies to the connection between the drones of
the Swarm and the Ground Control System (GCS), which actually controls the flight of the
drone and provides the "orders" to the drones.

Centralized is the architecture in which each drone is connected to the GCS directly. U-T-I

subjected to the GCS. In this architecture, the drones of the Swarm are not directly connected
to the GCS. Only one UAV connects to the GCS and acts like a Gateway to which all the
other UAVs are connected. Both U-T-U and U-T-I are applicable in this design. In such an
implementation, the drones can access the needed information regarding their flight operation
and send data back to the GCS. The distance between the drones of the Swarm is smaller
than the distance met in the centralized architecture mentioned above [28].
One requirement for the Decentralized architecture is that drones must operate at the
same speed levels and follow the predefined direction to maintain connectivity among them.
In the case of low-demanding applications, small drones with low capabilities can operate
adequately to provide connectivity and meet the application’s requirements [28]. In that
case, where all operating drones in the Swarm have the same characteristics, it is said that
a "Single-Group Swarm Ad-hoc Network" is implemented. An example of this is presented
in figure 22, taken from [28].

• Star Topology: In this topology, the UAV nodes are connected between them in a
star design. One UAV is in the middle, and the rest are directly connected to this
one exclusively. The UAV in the middle is the Gateway one and hence is connected to
the GCS (U-T-I communication) [28]. The communication between two or more UAVs
or the Swarm can only occur through the Gateway UAV. This topology is not very
reliable, however. Since there is only one UAV Gateway connected to the GCS, if it
fails, then the whole Swarm will fail as well, and in this example, there is a SPOF [28].
• Ring Topology:In the Ring design, all UAVs are connected with each other, so even
if a communication link fails, information and data can be transmitted through an
alternate one. There is no specific UAV that acts as a Gateway; any UAV node can
operate like this. Thus, in this case, there is no SPOF, and the system’s reliability is
increased [28].
• Mesh Topology: In any system, but also in UAV Swarms, Mesh topology is consid-
ered the most reliable because it combines both Star’s and Ring’s characteristics. All
UAV nodes or the Swarm are connected with each other by at least two communication
links, and any UAV can act as a Gateway [28].
Contrary to what was discussed in 5.2 about the "Single-Group Swarm Ad-hoc Network",
some applications require different kinds of drones with different characteristics to operate
in parallel. These different types of UAVs cannot be part of the same Swarm, as according
to the theory, all drones of the Swarm need to have the same characteristics and operate at
the same parameters [28].
In "Multi-Group Swarm Ad-hoc Network", drones are separated into Swarms (groups) ac-
cording to their functionalities. All the groups are connected to the GCS through their
Gateways. The communication between the different Swarms (groups) is called Group to
Group (G-T-G) and can be achieved through the GCS [28]. Multi-Group Swarm Ad-hoc
Network is a semi-centralized architecture as it combines the U-T-I communication between
the Groups and the GCS, if each Group is considered an entity, and also U-T-U inside each
Swarm, figure 26, [28].

6.1
OMNet++ and Simu5G
The OMNet++ framework and Simu5G were used for the simulations of this Thesis Project.
According to the definition given in [33], "OMNeT++ is an extensible, modular, component-
based C++ simulation library and framework, primarily for building network simulators.". It
contains libraries, like INET, providing a bunch of functionalities and built examples. These
examples can be either modified or used as a guideline for engineers who want to build their
simulation scenarios from scratch. Scripts are written in C++, and when executed, a graphic
representation of the simulated network is provided. Being open-source and free to use, it is
prevalent among network engineers, students, and people interested in network simulation.
Simu5G is a network simulator built on OMNet++. It was developed at the University of
Pisa, Italy, by the Computer Networking Group and, more specifically, by Giovanni Nardini
(Assistant Professor), Giovanni Stea (PhD - Full Professor), and Antonio Virdis (Assistant
Professor). Simu5G is the advancement of SimuLTE, which was developed by the same
Team; Simu5G uses the characteristics of SimuLTE and introduces the 5G NR features [2].
Simu5G contains many capabilities which are impossible to be explained under the time
limitations of a master Thesis. However, some of the most important and relevant to the
selected use cases and investigated scenarios will be presented below. The simulator gives a
chance to the user to choose, after modifying the parameters, either FDD or TDD for the
different selected frequencies. Different gNBs can communicate through X2 interfaces as well.
In a simulation scenario, it is also possible to implement both eNBs and gNBs and provide
communication between them. Carrier aggregation is also an embedded feature that allows
resource allocation according to the needs. Mobility is supported by importing the mobility
models provided by the INET library. The feature of device-to-device communication, which
Page 35 of 67
==PAGE:54==
Master Thesis
is also the one used for the simulations of this Thesis, is also provided. Simu5G allows users
to interact with real devices in real-time through real-time emulations [1].
5G main components are the Core Network (CN) and the Radio Access Network (RAN),
and as it is shown in Figure 28 provided by [1], RAN consists of UEs and gNBs, which
communicate at layer 2 (data link layer), and the Core contains the User Plane Functions
(UPF) which link the Data Network and the RAN.
Figure 28: 5G Components. [1]
According to that, Simu5G is simulating the Core and Radio Access Network as depicted in
figure 29. UE contains the TCP, UDP and IP protocols and the gNB the IP protocol and
the Point to Point Protocol which needed for the wired communication of the base station
with the Core network. The 5G NR characteristics are given to both UEs and gNBs by the
Network Interface Cards (NIC) which are called nrNicUe and NrNicGnb respectively. Their
components are presented in figure 30.

Following a short explanation of the NrNic entities is provided.
• Packet Data Convergence Protocol: PDCP is used to compress the headers of
the arriving IP packets [1].
• Radio Link Control: RLC is located between the PDCP and MAC layer and can op-
erate in three modes: Transparent, Un-acknowledge, and Acknowledge. These modes
have different characteristics, but they are all used for transmitting and receiving data
from and to the upper and lower layers by storing the data in Protocol Data Units
(PDU), which are stored in transmission or reception buffers respectively [1].

from the RLC, adding the MAC header, and providing the channel for transmitting
them to and from the physical (PHY) layer [1].

is a module responsible for storing network information that can be accessed by either UEs
or gNBs. Carrier Aggregation is a module that stores information like carrier frequency and
resource block availability, entities related to the carrier components [1].

6.3
Reasons to choose Simu5G
Since none of the Simu5G and the LARISSA profile were ready to be used in the investigated
use case of "Communication between Terminals and Drones using 5G", and both needed to
be modified, it is time to have a closer look at the reason behind choosing the Simu5G
instead of the LARISSA.
LARISSA, as mentioned in 6.2, was originally deployed for UAVs intercommunication and
UAV to GCS communication. Hence, at its initial state is more suitable for a Drone Swarm
use case. On the other hand, Simu5G was initially created for D2D communication providing
many use cases, i.e., communication between terminals and vehicles. This is the first point
that made Simu5G more suitable to use.
Nevertheless, since both are configurable, both could still be used in the inspected use
case.
Here is where the second point of difference is introduced.
Simu5G provides an
already deployed example of 5G D2D communication, consisted of gNBs and UEs with NR
specifications and characteristics, based on the INET library and by adding the NR features.
This fact saves a lot of modification time. LARISSA profile is also using the INET library.
However, as it is not created to meet 5G communication standards, the user has to modify
all the components and add new ones that do not exist (i.e., base station), to be compatible
with the 5G NR features.
LARISSA uses the communication technologies depicted in figure 32. Neither 5G nor any
cellular communication technology is part of them. After communicating with Mr. Emerson
A. Marconato from Centro UniversitĂĄrio EurĂpides de MarĂlia - UNIVEM, who was
part of the Team that developed the profile, he asserted that the user could add 5G NR
specifications; however, it could be complicated.
Page 39 of 67
==PAGE:58==
Master Thesis
Figure 32: LARISSA Profile Supported Technologies. [34]
The fact that made the significant difference, apart from the time limitations of a master
Thesis, and therefore made it plaint which simulator was more suitable for this specific
Project, was that even if the LARISSA profile gets modified and the needed changes take
place, time will still be needed to test it, to evaluate its performance and calibrate it in order
for the model to be valid to use.
Conversely, Simu5G is a tested and published simulation tool, which increases its’ validity
and trustworthiness. The 5G communication technology is already embedded, and users do
not have to create it from scratch. Considering all these, it was decided to proceed with
Simu5G for the executed simulations.

8.2
Discussion
After running 54 different simulations (27 for 1Mbps throughput and 27 for 50Kbps through-
put), for all the different combinations of "X" axis distance and altitudes, as these were
presented in 8.1, the writer came up with some valuable points. Although there are some
slight differences in the above results, these are so minor to be considered. From the results,
one could claim that the communication is reliable since the latency, in any case, is in the
range of ± 20ms.
However, this is unrealistic. It is more likely that this is the classical propagation delay of the
signal, and it looks like it is a prefixed delay. The delay was expected to change according to
the changes in the distance. This leads to the conclusion that the simulation model focuses
more on the protocol layers. Thus it is not so much affected by the changes in the physical
layer, i.e., distance, interference, building heights, obstacles, and reflections.

9.1
Conclusions
As the technology field is evolving rapidly and the number of connected devices on a network
is increasing significantly, the need for coverage and capacity is also increasing. For this
reason, new technologies have been developed to cover these needs and serve the customers,
as well as new ways to use the devices to achieve these goals. This Thesis Project tried
to cover all the above sections.
Firstly the need for Reliability in communications was
pointed out. Following, there was a detailed presentation of Drones and how their usage can
be, apart from entertaining purposes, also beneficial for the technology itself, the society,
and the environment.
The different wireless communication technologies were explained
and compared. The last part of the Thesis was an experimental part based on simulation
scenarios, trying to analyze and evaluate the Reliability of Drone-Terminal communication
using 5G communication technology.
After simulating the scenarios referred to in 7.2.1, it is clear that this Thesis Project is
a first step in approaching the investigated use case of "Reliable Communication between
Terminals and Drones using 5G", but it is not a solution to that. After discussing the results,
it becomes apparent that the simulation tool is not detailed enough and not fully developed.
There are some "TO DO" parts in the code of the different components and others that need
to be changed, i.e., the antenna type and the buildings’ height of the depicted background,
in order for the simulator to be fully functional and thus the results to be more realistic
and precise. Lastly, the fact that the tool is more focused on the protocol layers makes it
better to be used for testing conformance to the standards or conformance to the higher
layer protocols but not for predicting the performance in the physical layer. The Author
believes it can be considered a good first approach to the problem, but there is room for
improvement in the future.

height and obstacles, must also be included in the upcoming simulator versions. Additionally,
the type of antenna needs to be changed or provide the users with different options to choose
between antennas according to their needs. It would also be helpful to provide the option of
adjusting the bit-rate as the user wishes, and not only by changing the packet size and the
sampling time. Focus on the Physical layer should also be considered. If these changes are
made, the tool is expected to be more reliable and the simulation results more precise and
realistic.
Regarding the examined use case, there is a sequence of actions that the user can take to
improve this study. At first, running the simulations on a more powerful machine will be
very useful to avoid the impact of limited system resources and get the advantage of higher
simulation speed. Another challenge will be for the user to simulate other applications like
video transmission and analyze the exported results. In applications like this, the simulator
offers additional parameters for analysis, such as end-to-end latency. Transferring the exe-
cuted scenarios in real life using real equipment could provide valuable data, which the user
could crosscheck and compare with the simulator outputs.
The user also considers using other simulation tools and recreating the simulation scenarios
for future work. This way, there would be many available data for evaluation and comparison.
Tools like these are MATLAB with 5G waveform generator and Vienna 5G simulator. If all
the above is done, the user would be able to provide more valid and reliable conclusions,
which could lead to a solution that could be used widely.
