Abstract
Today there is no way for a doctor to remotely monitor their heart patients in real-time. The
existing solutions require patients to use oﬄine devices. Therefore ﬁnding a way to monitor
heart patients remotely in real-time can be valuable for both patients and doctors.
This report introduces the project, being the creation of an end-to-end system allowing re-
mote monitoring of heart patients in real-time with the help of IoT. Firstly the fundamentals
of ECG and the technologies are presented. This is followed by the system which consists
of a Maxim ECG sensor, that sends ECG and heart rate via BLE to a custom built device.
The device processes the data and transmits it via LTE-M to a backend provided by DTU
Photonics, where the data is stored. There is also created a mobile application, which uses
an API provided by DTU Photonics to fetch the data and visualise it in real time. Further-
more, the implementation of the device and mobile application is explained. The device is
ﬁeld tested for diﬀerent use cases. These tests compare two diﬀerent ISPs with the intention
to test the device in diﬀerent walking and driving scenarios. The objective is to test the
network performance of the two ISPs, looking at coverage, signal strength and packet loss.
The tests are conducted in Copenhagen, Denmark and Northern Zealand, Denmark. The
results show that both ISPs provide good coverage and strong signal strength in this part of
Denmark. The walking tests gave 0% packet loss whereas the driving tests showed packet
loss between 1-5 % depending on the speed and area. This proves that the device and overall
system are capable of diﬀerent use cases.

Introduction
An increasing amount of devices are being connected to the internet. A considerable amount
of these devices are Internet-of-Things (IoT) devices. Out of these IoT devices, Ericsson
predicts that 5 billion of them will be connected via cellular access technologies[8]. IoT
devices do not consume as much data as a mobile phone. They tend to use less bandwidth,
memory, processing power etc. An example could be an ECG sensor, sending real-time data.
The data that is being sent is sent as small packets throughout the day. Therefore there is
a need for networks that are more tailored to IoT use cases such as the before mentioned,
for eﬃcient utilisation of power and range. This is where LPWANs (Low Power Wide Area
Network) play a vital role. As the name states these types of networks are used for wide area
and low power applications. Figure 1 shows diﬀerent radio access technologies on a graph
concerning performance and coverage.
Figure 1: Access technologies addressing diﬀerent IoT use case areas [8].
For healthcare use cases this excludes some of the access technologies, especially with remote
monitoring of heart patients which is the goal of this project. In order to remotely monitor
heart patients, deep coverage is needed. Therefore group one from Figure 1 is excluded. In
order to send ECG data in real-time no duty cycles should limit the data rate, resulting in
group two being excluded as well. This leaves the cellular networks to choose from and given
a patient should be able to move around, mobility is needed, excluding also NB-IoT. Having
not to send large data packets, it leaves LTE-M as the perfect network to use for such use
case.
This project intends to create an end-to-end system that can help remotely monitor heart
patients’ ECG with the help of IoT. This is done by having an ECG sensor, sending data
via BLE to a device that processes the data and transmits it via LTE-M to a backend. The
backend stores the data for later usage. An API will be used by a mobile application to
fetch the data in real-time and visualise it as well as visualising historic data collected over

Motivation
Today’s problem is that doctors have no way of remotely monitoring their heart patients
in real-time. Often patients go to regular consultations for the doctor to keep track of the
patients’ health. In other cases, a heart patient can get an online ECG monitor, which mea-

constantly in real-time could prove valuable, both to doctors and patients. The idea is to
achieve this with the help of IoT and a small device receiving ECG data from a sensor. This
should be as small as possible due to the patient having to wear it all day. There will also

application via BLE and will be in production later this year in the US. On the market,
there is no system sending ECG via LTE-M, especially not on the Danish market. Therefore
this project intends to help the research in this ﬁeld to potentially lead to a system that the
Danish healthcare sector can use or fuse with their current systems.

2.1
Electrocardiogram
Electrocardiogram (ECG) is a test of the heart that measures the activity of the heart from
the electrical signals it releases as it beats. A heartbeat is triggered and carried out by
electrical signals. The ﬁrst signal starts at the top of the heart and travels down through
the heart until it reaches the bottom. A heart problem can often be discovered by looking
closely at the electrical signals of the heart. This is possible because this electrical activity
results in a very distinct electrical wave. Issues that might indicate a heart problem and give
reason for an ECG test are some of the following:
• Chest pain
• Tiredness
• Pounding, racing or ﬂuttering heart beat
• Uneven heartbeats
There are diﬀerent use cases for detecting diﬀerent heart problems, some problems only oc-
cur during stress while others rarely occur. Therefore there are diﬀerent use cases for using
an ECG tester i.e. under stress or for longer periods of time.
The ECG test is performed using electrodes which can be done at diﬀerent levels of precision.
An ECG test can be performed with anything from 1 to 12 leads. The most common ECG
test used by medical staﬀis a 12 lead tester which utilises 10 electrodes to create these 12
leads i.e. a pair of electrodes can create a lead. These electrodes are placed around the
body and on the arms or legs to measure the amplitude and direction of the electrical signal.

2.2
Bluetooth Low Energy
Bluetooth Low Energy, also referred to as BLE, is a low-power version of Bluetooth Classic
developed by the Bluetooth Special Interest Group(Bluetooth SIG). BLE was ﬁrst introduced
in Bluetooth version 4.0 and has been further developed together with Bluetooth Classic
until the current version 5.2. The new low energy radio type of Bluetooth, was as the name
suggests developed for low power operations, and supports a large variety of solution areas
i.e. use cases, depicted in Figure 5.
Figure 5: Bluetooth Low Energy Solution Areas.[10]
BLE also supports diﬀerent types of communications between devices, including point-to-
point communication which is also supported in Bluetooth Classic. A special feature of BLE
as oppose to Bluetooth Classic is that it can also be used for device positioning.
The BLE radio operates in the unlicensed 2.4GHz ISM band as Bluetooth Classic, and also
utilises the same frequency band from 2.402-2.480 GHz. The BLE radio divides the utilised
frequency band into 40 channels with a 2 MHz spacing between each channel. Each of these
physical channels is dedicated to a speciﬁc type of traﬃc, whereas 3 are advertising channels,
and the remaining 37 channels are general-purpose channels. BLE uses a frequency-hopping
spread spectrum(FHSS) algorithm to avoid interference and jamming. This also helps to
make a more robust communication in the open spectrum.[10]
BLE supports 4 diﬀerent data rates, two uncoded and two coded. The default and mandatory
data rate of BLE are 1 Mbit/s which is referred to as the LE 1M PHY, because this data rate
physically sends 1 million symbols per second, with each symbol representing a single bit.
7 of 108
==PAGE:20==
MSc Thesis
DTU Photonics
June 30, 2021
Furthermore, this physical signal can be used as a coded signal with s=2 or s=8 meaning
that 2 or 8 symbols respectively represent a single bit, resulting in data rates of 500 Kbit/s
and 125 Kbit/s respectively. The fourth supported data rate in BLE is 2 Mbit/s which is
achieved by doubling the symbol rate of the LE 1M PHY signal. By supporting diﬀerent
types of data rates, BLE increases its usage possibilities, e.g. The coded PHYs can help
increase the range of BLE and lower bit error rate(BER) in cases of exchange of important
information. Figure 6 depicts the mentioned types of supported physical data rates in BLE
and the combination of modulation and coding schemes.[11]
Figure 6: Four mentioned supported data rates in BLE.[12]
BLE supports two diﬀerent multiple access schemes, with the ﬁrst being the Frequency
Division Multiple Access(FDMA) scheme, which utilises the 40 diﬀerent channels which make
up the spectrum of which BLE operates. The second supported scheme is TDMA, which
is equivalent of FDMA with the diﬀerence of the division type being in time as opposed
to in frequency. The channels are therefore divided into time units known as events. All
communication in BLE is done in these events. There are four diﬀerent types of events in
BLE, which is:
• Advertising event: Used for advertising devices and small data packets.
• Extended Advertising event: Used for advertising packets too large for the regular
advertising event.
• Periodic Advertising event: Used for ﬁxed interval advertising.
• Isochronous event: Used for communication events.
The advertising events occur in the physical advertising channels and can be used for ad-
vertising device information such as information needed for another device the initiate a
connection event. The Isochronous event, also referred to as the connection event is commu-
nicated on the physical general-purpose channels, also referred to as the data channels.[11]

The second way of exchanging data between devices in BLE is quite the opposite, where
bi-directional communication is established between two devices. Such a connection can be
referred to as an Asynchronous Connection-Less(ACL) connection. This connection is es-

channel as the advertisement was received.

must implement. This proﬁle, depicted as the vertical line in Figure 8, deﬁnes the basic
requirements that a Bluetooth device must implement. The horizontal lines in Figure 8
are the requirements i.e. layers, deﬁned by the GAP. The proﬁles deﬁnes interconnection
between layers as well as peer connectivity between speciﬁc layers of diﬀerent devices.

More speciﬁcally GAP deﬁnes for BLE the physical layer, link layer, L2CAP, security man-

the behaviours of BLE actions such as connection establishment and service discovery. In
BLE the GAP proﬁle deﬁnes four types of roles which a device can implement one or more

• Peripheral: This role can only hold a connection to a single device, and can therefore

• Central: This role supports multiple simultaneous connections and can therefore en-

attribute protocol is designed to allow devices to read or write small amounts of data de-
pending on the attribute permission. Figure 9 depicts the ﬁelds included in the attribute

4G: Long Term Evolution
Before presenting the IoT networks, it is necessary to understand the fundamentals of how
the present 4G mobile network’s architecture is constructed. The fourth generation of mobile
networks, known as Long Term Evolution Advanced (LTE-A), was standardised in 3GPP’s
(3rd Generation Partnership Project) release 10. This was the ﬁrst actual true 4G, which
fulﬁlled the requirements set by ITU-T (International Telecommunication Union). It is an
improvement of the ﬁrst 4G, LTE, which was standardised in release 8. LTE did not fulﬁl
the requirements for 4G, which led to it only being commercially named 4G, where the true
4G network was standardised to releases later as mentioned.
For every network generation, there is an expectation of improvement. LTE oﬀers higher
bit rates, lower latency and more capacity than the previous mobile network. It is the ﬁrst
fully packet-switched network released by 3GPP. More of the functionality is moved to the
node-b (cellular network node that connects a mobile phone to the wider network). LTE
also introduces new features such as MIMO antenna systems and a new radio access scheme
based on OFDMA (Orthogonal Frequency Division Multiple Access) and SC-FDMA (Single
Carrier-FDMA).
Now that LTE has brieﬂy been introduced, it is important to understand how the IoT
networks are a part of the existing network architecture.
The network consists of three
major components: UE, E-UTRAN and EPC.
• UE (User Equipment):
UE is as it stands for the user equipment e.g. a mobile phone for LTE. It has a UICC
(Universal Integrated Circuit Card), also known as a SIM card(Subscriber Identity
Module). The SIM card is used to access the LTE network and contains information
about the user, unique identiﬁers for a speciﬁc mobile network and keys for authen-
tication, ciphering and security. The UE communicates with the eNode-B to handle
signalling.
• E-UTRAN (Evolved UMTS Terrestrial Radio Access Network):
E-URAN is the radio access part of the network, which contains the eNode-B. In LTE
12 of 108
==PAGE:25==
MSc Thesis
DTU Photonics
June 30, 2021
all radio related functionality is controlled by this entity. As mentioned it communi-
cates with the UE and controls them in one or more cells. It also manages the radio
resources, mobility, scheduling and retransmission.
• EPC (Evolved Packet Core):
The core has three main entities: MME (Mobility Management Entity), PGW(Packet
Data Network Gateway) and SGW(Serving Gateway). The MME manages mobility,
authentication and security in the network.
The SGW forwards data between the
eNode-B and PGW. At last, the PGW acts as the gateway to the PDN (Packet Data
Network), which it communicates with.

NB-IoT
NB-IoT (Narrow-band Internet of Things) is a LPWAN (Low Powered Wide Area Network)
standardised by 3GPP in release 13. As mentioned the IoT network is based on LTE and
its integration is regarded as a new air interface. It is also not backwards compatible with
LTE. NB-IoT oﬀers minimised signal overhead, low power consumption, longer range, data
support for IP and non IP. In order for NB-IoT to achieve this, most LTE features are not
supported e.g. handovers meaning it does not support mobility.

LTE-M
Like NB-IoT, LTE-M (LTE Machine Type Communication) is a LTE based LPWAN net-

to coexist with the existing LTE infrastructure like NB-IoT. Opposite to NB-IoT, LTE-M
is not designed just for IoT, but for machine type communication, hence the name. This
also means that LTE-M focuses on similar network parameters as NB-IoT, such as coverage,
cost and power consumption. LTE-M oﬀers a 1Mbps data rate and deep coverage. Unlike
NB-IoT, it inherits handovers from LTE, which allows for mobility use cases. It also supports
voice via VoLTE (Voice over LTE).

brings signiﬁcantly higher peak data rates, lower latency and more capacity among other key

frequencies, but results in lower coverage per cell due to higher path loss. This has no beneﬁt
for IoT, where low bit rates, low latency and good coverage is more critical. The lower latency
with 5G will especially be an advantage for the IoT networks, which will be mentioned in
the next section. Further 5G characteristics are heterogeneous networks, network slicing,
Device-to-Device communication, Massive MIMO, Beamforming and mmWaves, which will
not be covered here.

There are 2 main deployment options for 5G which are deﬁned as ”Non-Stand Alone” (NSA)
and ”Stand alone” (SA). The NSA option connects the 5G access network to the 4G core
network, while the SA option access network 5G core network.
Figure 17 depicts a simpliﬁed network architecture of the 5G NSA option. As mentioned
this option utilises the 4G network core. The new addition lies in the access network, where
5G nodes have been added. This deployment option makes the 5G technology available,
giving higher bit rates, lower bit rates etc. but without the 5G services. By choosing this
option there is also cost savings due to not having the need to replace the network elements
in the core.
This deployment option is also known as E-UTRA-NR Dual Connectivity (EN-DC). The
dual connectivity is seen in ﬁgure 17, where the 5G enables UE to connect to an eNode-
B and an eNode-gNB. The logical 5G node (eNode-gNB) connects to the 4G logical node
(eNode-B) via the X2 interface, which is used to connect two nodes. The eNode-B acts as
the master node, which handles the radio communication with the UE. The eNode-gNB on
the other hand acts as a secondary node. Overall this option is seen as a step towards ”true
5G deployment” and is the one that is currently rolled out in Denmark. Note that diﬀerent
variants of the SA and NSA deployment options can be found in release 15[29].

2.4.1
LTE-M & NB-IoT In 5G
Now that LTE-M and NB-IoT have been mentioned in 4G context, their role in 5G will be
presented. As mentioned one of the three major use cases for 5G is mIoT. The applications
for this use case naturally ﬁts Low Power Wide Area Networks (LPWAN). Given 5G is a
cellular technology standardised by 3GPP, it makes sense to use the existing cellular IoT
networks, LTE-M and NB-IoT, for the 5G IoT use cases. In fact these two technologies
already fulﬁl the requirements for mIoT and can therefore be seen as 5G technologies, even
though they utilise 4G technology. They will coexist with 5G, while smoothly migrating to
a 5G core ensuring the preservation of the two LPWAN’s deployment.
Since the ﬁrst release of LTE-M and NB-IoT, there have been some enhancements in release
15. Looking at NB-IoT there are new functionalities and enhancements of the existing ones.
They primarily focus on reducing UE power consumption. The enhancements are on the
air interface and on the protocol layers. The new functions include support for small cells,
extensions to the NB-IoT standalone mode and time division duplexing[29].
LTE-M also has improvements in its features with general improvements on latency, power
consumption, spectral eﬃciency and access control[29]. Furthermore, these enhancements
have introduced support for new use cases. The potential use cases can utilise the support
for higher UE velocity, lower UE power class or new gaps for a dense position reference signal.
The existing cellular networks for IoT fulﬁl the requirements for 5G as mentioned, but that is
achieved without any new 5G capabilities. For the next generation of UE, they will support

System Design
This section will describe the design phase of the product. It will determine the requirements
for the hardware and mobile application. Based on the requirements, an appropriate proto-
type for the hardware and application is created. At last, it will present the ﬁnal design for
the hardware prototype, application prototype and an overview of the end-to-end system.
3.1
Hardware Prototype Design
For the hardware prototype design phase, the ﬁrst step is to identify which hardware compo-
nents and technology can be used to solve the problem statement. Therefore market research
has been done, in order to ensure that the hardware can live up to the desired speciﬁcations.
For this project, an ECG sensor is needed. This sensor must be able to send ECG data
to a custom-built board via a low powered communication technology such as BLE. The
custom-built board must be compatible with the same technology as the sensor to receive
data from it, therefore it should contain a BLE module. Furthermore, it must be able to
send the data to a backend. In addition to the ECG data, location data is also desired,
meaning the need for a GNSS (Global Navigation Satellite System) module, which utilises
the American GPS. Based on the theory, the best suited communication technology for this
use case is LTE-M. This is based on the estimated payload having to be sent in real-time,
mobility and the bandwidth supported for this technology compared to NB-IoT. Compared
to other LPWAN’s LTE-M has the advantage of already utilising an existing network infras-
tructure, giving it a coverage advantage. Also it is not limited by any duty cycle given it
operates in licensed frequency bands. The backend should be able to handle the data from
the custom-board and store it in a database. The stored data in the database will be fetched
by a mobile application, which will be able to visualise the data in real-time and show his-
tory data. The visualisation will be further addressed in subsection 3.2 Mobile Application
Prototype Design.
In order to achieve the desired functionality of being able to receive ECG data, send it to a
backend and visualise it in real-time, some hardware components are needed. The hardware
criteria for this project is:
• Microcontroller
• LTE-M module
• BLE module
• GPS module
• 1x Antenna for GPS
• 1x Antenna for LTE-M
• ECG sensor

With Maxim as the chosen ECG sensor, a microcontroller is needed to receive the data from

it must support the bands that are utilised in a user’s region. The SARA chip supports

initial design phase, this has not been included.

4.1
Device
This section describes how the device establishes a BLE connection with the Maxim sensor,
the main functionality and the LTE-M connection. The device is as mentioned a custom
design, which can be programmed using Arduino IDE. The bootloader for this board is
the Arduino MKRZERO. Every command and function is veriﬁed using the Arduino serial
monitor. Based on the criteria in the design section, this device must be able to connect
to the Maxim sensor via BLE, process the data and send it via UDP to the backend in a
reliable way.
4.1.1
BLE Connectivity
As mentioned the device can only connect to the Maxim sensor via BLE. Given that the
device is custom, no existing Arduino library can be used to establish this connection. This
connection is therefore coded with AT-commands, that is sent via Serial to the BLE module.
All the AT-commands can be found in the documentation[41]. In order to establish the BLE
connection with the device, the following AT-commands are being sent to the BLE module:
• Read BLE role: AT+UBTLE=?
• Setup new service: AT+UBTGSER=<uuid>
• Setup new characteristic:
AT+UBTGCHA=<uuid>,<properties>,<security read>,
<security write>,<value>
• Discovery: AT+UBTD=<discovery type>, <mode>,<discovery length>
• Establish ACL connecetion: AT+UBTACLC=<bd addr>
• Get services: AT+UBTGDP=<conn handle>
• Discover all characteristics of service: AT+UBTGDCS=<conn handle>, <start>,<end>
• Write Characteristic value: AT+UBTGW=<conn handle>, <value handle>,<hex data>
• Discover all characteristic descriptors: AT+UBTGDCD=<conn handle>,
<value handle>,<service end handle>
• Subscribe to Notify: AT+UBTGWC=<conn handle>, <desc handle>,<conﬁg>
Every AT-command has been tested manually with a hardware ﬂow script, that allows for
manual input via the serial monitor to be written to the BLE module. The response from
the module is also shown in the serial monitor to verify the response. Every AT-command
comes with some standard Ublox response, which can be found in the documentation. There-
fore each response from the BLE module is processed, so the desired String value is retrieved.
When the device is powered on, it enters a bonding phase. Here its BLE role is changed to
peripheral. The role is changed, so the E-Med application can bond with the device. When
the role is changed, the device sets up a service and a new characteristic. The characteristic

Before running the method, the application checks if the phone has any connection to the
internet, otherwise there is no reason to run the method. If the phone has a viable internet
connection, the method is run. First, it checks if the ﬁle that is used to store the data
exists. If the ﬁle exists, the application already stores some data from the database and it
is, therefore, necessary to check if the stored data is older than today’s date. If the latest
stored data is older than today, the method calculates how much older i.e. how many days
between today and the latest stored date. This is done for each data type stored in the
application, which is currently the four following:
• Steps: Stores the daily number of steps taken by the patient.
• Deep sleep: Stores the amount of hours of deep sleep the patient has slept.
• Light sleep: Stores the amount of light sleep hours.
• Rem sleep: Stores the amount of REM sleep hours.
The download method is then run in a separate thread because network connectivity is an
asynchronous task, which is not allowed to be run in the main thread of the application.
The main thread is also referred to as the UI thread because it handles the UI and must
always react to user inputs[47]. Therefore time-consuming tasks, such as internet connec-
tivity, must always be run in separate threads to avoid the UI thread being blocked, which
would result in the application shutting down.
The four diﬀerent data types are sequentially downloaded in the same thread. The reason
for doing a sequential download instead of doing a parallel download of all four data types
simultaneously is because the amount of data is very small, resulting in low download times.
Doing four parallel downloads is deﬁnitely possible, but would require four separate threads
and would increase the complexity of the download because the data is also written to a ﬁle,
which can not be done in parallel. Therefore the trade oﬀin time vs. resources proved that
saving resources were the best and simplest solution.
The download from the database is done through HTTP GET requests. The database API
has a few diﬀerent types of endpoints, but the endpoint used for downloading the latest data
for each of the four data types has the following structure:
https://heartrater.live/api/measurements/x/patient_id/datatype
With x as the number of data points from latest to oldest, which is equal to the earlier
calculated days since the last download. This API call returns the speciﬁed number of data
points of the speciﬁed data type. The data is downloaded for speciﬁc patient identiﬁed by
the patient id.

ﬁle. Another reason for loading the stored data from the ﬁle while downloading new data is
also to append the new data to the old data before writing to the ﬁle. This is because data
cannot be written directly to an already existing ﬁle without overwriting the existing data.
Duplicates can emerge if for some reason there has not been uploaded any new data to the
database for a given number of days. This results in the patient opening the application, e.g.
after three days without having uploaded new data. Then three days of data is downloaded
from the database, although this data is not new because it is already stored in the ﬁle, and
should therefore not be saved. After ﬁnishing the necessary downloads the data is written
to the ﬁle.
If the patient were to begin navigating the application while the download is not yet ﬁn-
ished, this could lead to issues and maybe even crash the application. To avoid this, a loading
screen is implemented when booting the application. This is what is referred to as a splash
screen in Android and can be seen in Figure 32. The splash screen is displayed for the ﬁrst
two seconds after creating the application to give it time to load all the necessary resources
and synchronise the data. When two seconds have passed, the application navigates to the
home screen.

The two other important methods that are located in the MainActivity is the two methods

manage associated devices. When a device has been connected to the application through the
CompanionDeviceManager, it becomes associated with the application until it is uninstalled
or the device is disassociated manually.
The reason for using the CompanionDeviceManager for the BLE connectivity is due to
the location capabilities of BLE devices which results in applications connecting to BLE
devices must have location permissions. The permission must be actively requested by the
application and given by the user to the application even though the application never uses
any location data.
This might make some users question why the application needs to
know their location. Therefore to avoid making users insecure, all connectivity between the
application and the BLE devices are handled by the CompanionDeviceManager, excluding
the need for location permissions, because the application is no longer able to access location
data.

File and storage
Storage
Because the data that this application downloads and stores is labelled as personal data, it
is very important how the data is handled and stored. Personal data must not be stored in
any way where it can be accessed by other applications as this would not be GDPR compli-
ant. Without going into any details about GDPR, the data is simply stored without being
accessible by others to avoid any issues in relation to the storage of personal data.
Android recommends storing persistent sensible app-speciﬁc data in the internal storage
where it can not be accessed by other applications.[50] This ﬁle storage is accessed using
getFilesDir(), which does not require the application to have any speciﬁc permissions as this
storage is related to the given application. Furthermore the ﬁles stored here are removed
when the application is uninstalled which also ensures that there is no personal data ﬁles left
behind by mistake when e.g. the application is uninstalled. Another advantage of this stor-
age is that from Android 10 this is also encrypted to protect the stored data even better.[51]
File
The ﬁle is stored under the patient id and can only be accessed from the application, with
the use of this id. This is static in the application, such that for future prototypes where a
login and proﬁling is added, the patient id would be taken from the proﬁle that is currently
logged in. This ensures that even though there might be several personal data ﬁles stored
in the internal ﬁles directory, the ﬁles are not accessible by other patients using this phone.
Most modern phones has plenty of storage available in the internal directory and the stored
data is measured to approximately 1MB of data for an entire year of steps and sleep data.
Model

as java objects and stored in the ﬁle as JSON, some processing of the data is necessary to
change between the diﬀerent formats. The Gson Java library from GitHub[52] is used to
simplify this going back and forth between data formats. All that is needed for Gson to be
able to parse the data is a model. This model is a java class that speciﬁes the setup of the
data objects.

• onServicesDiscovered: Called on service discovery.

where the subscribed data becomes available. As long as the application notify for this data,
onCharacteristicChanged is called each time new ECG or heart rate data becomes available.
The incoming data is diﬀerentiated by the length of the received packet, and is processed so
that it can be displayed correctly.

Plotting
In the sensor fragment, data is simply plotted in the graph as it is received from the sensor,
this is possible because the sensor sends data every 125ms compared to the server which only
sends data each second. If the server fragment were to plot the data each second the plotting
would become laggy and diﬃcult to look at and especially analyse. Therefore, to smooth it
out, the plotting is done in a separate thread, which can be paused for a short period of time
between each plotted data point, more precisely the thread is paused for 7.8125ms between
each plotting so that it takes exactly 1 second to plot the 128 data points that are received
each second.
The graph is created through the use of a library called MPAndroidChart[48] which is im-
plemented from GitHub. The graph is a line graph designed such that the x-axis displays
two seconds of ECG data; Two seconds of displayed data gave the best visual results. The
graph is implemented to simulate a regular ECG plotter i.e. plotting from left to right until
the right border is reached and then clearing the graph only to start over with plotting from
left to right. A pause button is implemented in the lower right corner to oﬀer the ability to
pause the plotting for deeper analysis of the currently displayed ECG data.

6
Conclusion
This project has been carried out to give a solution to the current problem of remotely mon-
itoring heart patients. As mentioned in the report, there is ongoing research about this, but
there is no available system as of now. The solution can help both patients and doctors as
it might help give patients a peace of mind, and help doctors gather information about their
patients; This can lead to better care. The solution provided was to design and implement a
fully functioning end-to-end system consisting of a sensor, device and application setup. The
solution uses one of the available licensed LPWAN networks for connecting the patient to
the internet. The requirements for such a system is established in collaborating with DTU
Photonics who has been in dialogue with medical personal to help shape the best possible
solution. The implemented solution, even though it is fully functioning, is implemented as
a proof of concept prototype which can be further developed to reach a ﬁnal product. The
idea of this solution is to prove that the included concepts are possible and necessary to have
in such a system.
Research of the available LPWANs was conducted to discover which technology to use for
this solution. The LPWANs operating in the unlicensed spectrum was quickly ruled out
because of the duty cycle. NB-IoT was discarded because of the lack of support for mo-
bility. The winner was LTE-M which proved to support all the needed requirements for
this project, with fairly high performance and fairly deep coverage giving a great average
technology. 5G-NR might theoretically prove as a better solution for this system, but it was
not possible to investigate this because there are no currently available networks with a 5G
core.
The prototype was tested through ﬁeld tests with the goal of proving not only that the device
was functioning correctly, but also to test the coverage of LTE-M to ensure that the patient
would always be monitored no matter the use case. The coverage of LTE-M was tested for
two diﬀerent ISP’s referred to as ISP1 and ISP2 to investigate if one had better coverage
than the other. The result proved that LTE-M was the perfect choice for this solution.
Coverage was investigated by measuring the RSSI of the network for each second while con-
ducting the tests. Coverage was great and ﬁnding an area with no signal proved a bigger
challenge than expected which was positive. With both walking and driving tests conducted,
neither ISP ever got below a signal power of 20 for a longer period of time; Signal power is
measured on a scale from 0-31 with 31 being the best. Although both ISP’s was very equal
in most of the tests, ISP1 has a slightly lower packet loss percentage in each of the driving
tests and a little higher or the same in most of the walking tests.
The diﬀerence in packet loss between the ISPs is so low that it can not be concluded if one
is better than the other or if the equipment is to blame. Although ISP1 often has a slightly
higher signal power in the tested areas this might not show the same results in the entire
country. The conclusion on the choice of network connectivity is therefore that LTE-M is
the best possible solution and either ISP can handle the task.
The application is implemented with all the design parts and each part functions as expected.
The application can display real-time ECG from the server or by directly connecting to the
sensor. Historic data can be downloaded from the database by the use of the received API

and display the data to the patient. The historic data can be displayed for a day, week, month
or an entire year at a time. The Me page shows personal information and doctors information
as well as implement buttons that can easily navigate the patient to the web or their default

application and even by encryption on Android 10 and newer versions. A fallback method

6.1
Future works
When doing time constrained projects such as this, it is not always possible to create a ﬁnal
product which implements all the ideas and goals for the project. Instead, in this case, a
functioning prototype that proves the overall concept has been implemented. Therefore this
section will shortly explain some of the better ideas which should have been implemented if
time allowed it.
6.1.1
Application
Further development on the application should include adding a proﬁle in the application
with belonging login functionality. As mentioned, this proﬁle would help import the correct
doctor and patient information needed for the me page as well as it would increase the
protection of the stored personal information.
Another improvement would be to further develop the fallback functionality such that it was
able to either store or upload the data received from the sensor. The functionality of the
communication between phone and device should also be improved such that the device was
also able to alert the phone after a connection was reacquired.
The application could also be improved to implement a diary functionality, which could be
used by both doctor and patient to help remember and understand certain periods of time
where collected ECG data could be compared to the patient’s diary information.
6.1.2
Device
Further development of the device would include the obvious parts such as adding a case
and battery capacity such that the device becomes mobile. From the experience gathered
in this project, it might also be necessary to investigate changing the current SARA module
and GPS setup out with a SARA-R422M8S module that has built-in GPS functionality.
The software of the device could be improved to also be compatible with other ECG sensors
than just the Maxim sensor used for this project.
In coherence with the mentioned diary function in the application, a real-time algorithm for
analysing the ECG data could also be implemented to locate issues in the ECG data and
store events like a skipped heartbeat or others. Later this data could be added to the diary
in the application, with a timestamp, event information and a section of ECG data around
the event which could then be further analysed by a doctor. The algorithm might instead
be added at a server level to conserve CPU usage on the device.
