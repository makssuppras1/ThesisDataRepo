Summary (English)
The goal of the thesis is to illustrate the usefulness of partial reconﬁguration in
FPGA technology, by implementing a system that can conﬁgure reconﬁgurable
modules on demand. The system runs a Linux operating system and provides
a Linux kernel module for triggering the reconﬁguration.
The reconﬁgurable modules are used to illustrate the usefulness of a software
application that can be accelerated by accelerators that are implemented in
hardware. The implemented accelerators are based on an HLS version of the
Sobel edge detection algorithm, and the histogram equalization.

Chapter 1
Introduction
The computing industry is in a rapid expansion, with a greater requirement for
faster processing, lower latency, while often requiring high eﬃciency. Depending
on the speciﬁc requirements, several solutions come to mind. If faster processing
is the main area of interest, the modern solution is to oﬄoad the processing to a
dedicated circuit, that is fast at processing that speciﬁc application. A historic
example of this is seen, when dedicated sound microchips and dedicated graph-
ics microchips were introduced into the personal computer. These microchips
oﬄoaded the computation of sound or graphics from the central processing unit
(CPU). The dedicated microchips came with a huge performance beneﬁt, but
also some disadvantages. They were expensive to develop, and lacked the ability
to be upgraded, as they were implemented as an application-speciﬁc integrated
circuit (ASIC). ASICs are considered the fastest implementation, and when pro-
duced on large scale, often quite cheap per device. A big disadvantage is the
lack of upgradability since once the circuit is printed, it cannot be changed. Al-
though this can be mitigated, with the support for ﬁrmware inside the circuit,
this strategy will also make the whole circuit much more complex.
In newer times other technologies have become a good alternative to ASICs.
The use of FPGA technology, which historically was mostly used for prototyp-
ing hardware circuits, because of its ability to be reconﬁgured to another circuit
design. But today FPGA technology has become cheaper, faster, and larger
than ever, making the technology much more widespread and available. This
==PAGE:12==

means that FPGA is used for much more than prototyping. If a business has
a speciﬁc application it wants to be accelerated, the application can be devel-
oped in a hardware description language (HDL) and implemented in FPGA
technology, instead of using a much longer time on a less versatile ASIC imple-
mentation. This makes FPGA chips very versatile but is not as fast, nor eﬃcient
as ASICs. But for small scale implementation use, an FPGA implementation is
much cheaper than an ASIC implementation.
Another solution to accelerate a system is to add more dedicated chips, FPGA
or ASIC, to do the desired applications.
This has the clear disadvantage of
requiring more physical space and require more power. Sometimes the chips are
not even being used, because there is no need for the accelerator at that time,
hence resources being wasted. To tackle this, several chips may be combined
into one FPGA, and the accelerator conﬁgured on demand. This technique is
part of a new paradigm that has formed. A paradigm that divides the FPGA
into smaller sections, or partitions, that can be conﬁgured on-demand. When
the demand for a particular accelerator disappears, the resources it occupied is
put back to the available resource pool. This paradigm provides even more ver-
satility to the FPGA technology, allowing an FPGA chip to accelerate multiple
applications on-demand at the same time.
When splitting an FPGA into multiple partitions, some resources outside the
partitions are used to connect the partitions to other logic. This part of the
design is the static part and is persistent across all accelerators. The partitions
contain the dynamic part of the design, this is where the accelerator is im-
plemented and can be dynamically conﬁgured into the FPGA. When a partial
conﬁguration is done, only the dynamic part is changed, and the static part is
unchanged. The amount of partitions that ﬁt in an FPGA is limited by the chip
resources, size of each partition, and the size of the static design.
Several managing schemes can be used, ranging from using a static schedule, to a
more complex implementation using a scheduler that in real-time determines the
demand. When using a static schedule, the ﬂexibility of the system is lost, but
this kind of schedule is very easy to implement. A demand-driven conﬁguration
scheme requires a system-wide low-level implementation of a manager, that
determines the demand in real-time, and conﬁgures the partitions depending on
this information. Depending on how deep the manager implementation is, it
requires a deeper knowledge of the operating system.
In works related to partial reconﬁguration, [Pan15] showed that when accel-
erating convolution tasks with FPGA technology, the speedup is about 33x,
when compared to software computation. And 10x when compared to software
with NEON acceleration. The thesis [Pan15], also showed the diﬀerent software
code optimizations and their speedup. And also show that NEON acceleration
==PAGE:13==
3
provides about 2-10x speedup, above pure software computation. Although it
researched it, the thesis did not implement any partial reconﬁguration, because
of missing procedures at the time.
The thesis [Lom16] dived into partial reconﬁguration using the ICAP interface,
and implemented their own ICAP conﬁguration controller, that transfers the
bitstream from system memory to the ICAP interface. Their implementation
has a theoretical throughput of 400MB/s (at 100MHz), and even overclocked it
to 180MHz, reaching a theoretical throughput of 720MB/s. The thesis imple-
mented a couple of accelerators, and the speedup was shown to be from 1.3x
to 271x. depending on the speciﬁc accelerator. Using ICAP to conﬁgure an
F4 module ( [Lom16] speciﬁc sizing), the latency was about 430-450us when
the ICAP is operated at 100MHz. Although not directly comparable, they also
found that using PCAP to conﬁgure a larger F1 module (Again [Lom16] spe-
ciﬁc sizing), the latency was 18ms, and the throughput of PCAP was 110MB/s,
limited by the theoretical PCAP throughput of 145MB/s.
In [Esp16] a partial reconﬁguration design was implemented running with petal-
inux, but they needed to use PCAP to load the module conﬁguration, because
of the lack of a Linux driver for their custom ICAP controller. They show no
measurement of conﬁguration latency.
The Thesis [Edo18] used partial reconﬁguration to implement a cryptographic
engine, supporting the RSA and Blowﬁsh algorithms. The accelerators showed
a speedup ranging from 10-63x compared to CPU only computation.
This thesis will evaluate the possibilities and limitations of partial reconﬁgu-
ration and the Xilinx speciﬁc methodology Dynamic Function eXchange when
working from a Linux environment. The work will involve developing a couple
of sample reconﬁgurable modules, that will illustrate the use of this technology.
The modules will be provided as illustration only, and not as a showcase of their
respective performance or abilities.
The structure of this thesis is divided into chapters, where chapter 2 will in-
troduce the basic background knowledge to follow this thesis. chapter 3 will
introduce the design methodology when working with partial reconﬁguration
and Petalinux, and all of its parts. In chapter 4 the implementation is discussed
in detail. The results found will be presented in chapter 5, and a conclusion
formed in chapter 6.

ZYNQ processor
The evaluation board includes an SoC from the Zynq 7000 series by Xilinx. The
part number of the speciﬁc chip used on the Xilinx ZC702 evaluation board is
XC7Z020-CLG484-1, more details of this SoC are discussed later. The Zynq

and PL are connected with several interconnects between the two. The intercon-

are also available. Figure 2.2 shows a detailed block diagram of the Zynq 7000

2.2.1
Processing System
The processing system includes two ARM Cortex-A9 cores running at 667MHz.
The processing system includes 32KB L1 instruction and 32KB L1 data caches,
a 512KB L2 cache, and supports external memory. Connected to the processing
system is 1GB of DDR3 RAM operating at 533MHz. The processing system
is able to run code inside an operating system, or on a standalone platform by
using Xilinx provided tools, e.g. Xilinx Vitis, to run code directly. When using
an operating system, the system consists of many more layers, which create a
greater distance from the applications to the hardware. But the beneﬁt of an
operating system is that it greatly improves the ﬂexibility of the system, al-
lowing for multitasking, and for services and applications to run in parallel. A
standalone platform provides less abstraction, thus fewer layers, and it is, there-
fore, easier to manipulate the hardware components directly. But a standalone
system might also lack basic functionality, that the user will have to implement,
if not already implemented by Xilinx.
2.2.2
Programmable Logic
The programmable logic is the generic name for the integrated Artix-7 series
FPGA, which for this Zynq processor part, includes 85K logic cells, 53200 lookup
tables (LUT), 106400 ﬂipﬂops (FF), 140x 36Kb Block ram (Bram) at a total
of 4.9 Megabits and 220 Digital Signal Processors (DSP). In addition to the
above primitives, the PL also includes other types of primitives that can be
instantiated in a design. Primitives include clock managers, one ICAPE2, and
others. For partial reconﬁguration, the ICAPE2 primitive is an important part
of the PL, since this is the component that allows the design to use the ICAP
interface for partial conﬁguration of the system.
2.3
Partial Reconﬁguration
A partial conﬁguration stands in contrast to a full conﬁguration, which is the
traditional method of conﬁguring the programming logic.
As the name im-
plies, a partial conﬁguration can be used to conﬁgure a smaller partition of the
programming logic, thereby greatly reducing the conﬁguration latency, and in-
creasing the versatility of the implementation. When we expand conﬁguration
to the term reconﬁguration, it means that, after an initial full conﬁguration, we
can reconﬁgure a smaller partition of the logic, without changing or interrupting
the rest of the design.
==PAGE:19==
2.3 Partial Reconﬁguration
9
 Interfaces
Figure 2.3: Static and dynamic logic of the system
In a design that utilizes partial reconﬁguration, the programmable logic is split
into two, the static logic and the dynamic logic. The static logic of the design
is only changed when a full conﬁguration occurs. However, the dynamic logic
is the partition that can be reconﬁgured on demand, each implementation of
logic inside the partition is referred to as a reconﬁgurable module (RM), and
the partition as a reconﬁgurable partition. Figure 2.3 shows an example of the
static logic, with the dynamic logic inside it. If desired multiple RPs can be
included in parallel inside the static logic.
When an interface crosses the RP barrier, it is a good idea to isolate the signals
before reconﬁguring, because the interface may ﬂoat, thus putting unwanted
signals into the rest of the system. More on this topic later.
The typical conﬁguration interface used is the Processor Conﬁguration Ac-
cess Port (PCAP), which can only be used from the processing system side.
An alternative conﬁguration port is the Internal Conﬁguration Access Port
(ICAP), which uses the ICAPE2 primitive component that resides inside the
programmable logic.

2.3.1
Conﬁguration interface
The PCAP interface port is only available on Zynq processors and is allowing
for the processing system to conﬁgure the FPGA resources either by doing a full
or partial conﬁguration. The ICAP interface is only available from inside the
programmable logic and only allows for partial conﬁguration because the ICAP
interface has to be driven from logic inside the PL. Both [Lom16] and [Esp16]
used a custom made ICAP controller because the oﬃcial ICAP controllers (Xil-
inx AXI_HWICAP) available at the time, were too slow, and faster IPs were
either not available or they required special licensing for use. However, Xilinx
has since ﬁxed this problem.
The PCAP interface has a maximum theoretical bandwidth of 145MB/s but
is further limited by the PS AXI interconnect, thus the real throughput is
about 110MB/s. [Xil18, Lom16] The ICAP interface can accept 32Bits every
clock cycle.
The ICAPE2 component is rated at an operating frequency of
100MHz. This means that the interface has a theoretical maximum throughput
of 3.2Gbit/s or 400MB/s. Please be aware of the diﬀerence between MB and
MiB, where the latter uses 1024 bytes pr kilobyte, and the former uses 1000.
The numbers given above are all in MB/s or Gb/s.
2.3.2
Dynamic Function eXchange
A couple of years ago, Xilinx improved its design methodology for partial re-
conﬁguration. They chose to call it Dynamic Function eXchange (DFX). This
action simpliﬁed and automated the methodology when implementing a par-
tial reconﬁguration system. Along with the DFX methodology, Xilinx chose
to include the licensing of DFX technology and IPs in their standard licens-
ing, making the technology readable available to everyone. The DFX design
methodology is described below and in more detail in [Xil20e].
Xilinx has included four DFX IPs in the license, but the DFX methodology
does not strictly require any of the IPs.
The IPs are provided to ease the
implementation of a DFX system for the user.
A central DFX IP component is the DFX controller, which once triggered, col-
lects the bitstream data via an AXI channel, and pushes it to the ICAP inter-
face. Additionally, the controller can optionally handle the decouple, startup,
and shutdown sequences of the reconﬁgurable module. To isolate the interfaces
of the RP, a DFX Decoupler can be used, to decouple the interfaces when the
DFX Controller is reconﬁguring a new RM. More detail about the IPs may be

2.4
Petalinux
Petalinux is an operating system that uses the Linux kernel and is made by
Xilinx. Petalinux is made to run natively on the processing system and is made
to ease the steps required to support custom hardware in the programmable
logic. The latest Petalinux versions are typically running a long-term stable
(LTS) Linux kernel and are well developed.
2.4.1
Linux Kernel Modules
A Linux Kernel Module (LKM), is an extension of the functionality of the kernel
and is a module that is generally used to communicate with hardware, it can
also be called a device driver. As the name implies, the module is running in
kernel space, which means it has escalated permissions. By running in kernel
space, the driver is allowing more control but may also have catastrophic results.
If implemented incorrectly, the system may experience instabilities, crashes, and
corrupted data. Thus caution has to be given when designing an LKM, especially
when manipulating system and device memory.
2.4.2
User applications
Compared to LKM, user applications run in user space, which is less privileged
and is safer to use in regards to system stability. In general, user applications are
common applications, ran by the user. The user application can communicate
with hardware through the LKM using system calls, either by reading/writing
directly to a device driver, or using a system call.

Chapter 3
Design methodology
In this chapter, the design methodology for implementing a partial reconﬁgu-
ration system that uses the Petalinux operating system will be presented. In
basic terms, Vivado is needed to create the hardware platform, and on top of
it, Petalinux is used as the operating system. Device drivers control the custom
hardware, and the user application sends instructions to the hardware through
the device driver.
Because these types of projects include the use of many diﬀerent tools, only
the tools that are the most important to partial reconﬁguration and its reliance
will be described. This means that tools like Vitis HLS and the design method-
ology of it, will not be described, as the reader is expected to have a general
understanding of the tools used for FPGA projects. The basic Xilinx Vivado
workﬂow will only be touched upon lightly because it sets the premise for further
methodology explanations.

3.1.1
Dynamic Function eXchange
The DFX workﬂow is largely the same as a non-DFX workﬂow, but with some
additional steps and some limitations. A simpliﬁed workﬂow when creating a
Vivado project using the DFX methodology is shown below. More details than
described in this section can be found in [Xil20e,Xil20f].
1. Add sources (only one RM)
2. Enable Dynamic Function eXchange
3. Create Partition Deﬁnition
4. Run Synthesis
5. Assign a pblock to the partition and assign properties
6. Add more reconﬁgurable modules with the DFX Wizzard
7. Run Implementation
8. Generate bitstream
9. Export hardware
10. Generate binary ﬁles of the partial bitstreams
During project creation, add only one of the reconﬁgurable modules to the design
together with the rest of the sources.
After project creation, DFX will have to be enabled by selecting
Tools -> Enable Dynamic Function eXchange

tition. This is done by right-clicking on the RM in the Sources hierarchy and
selecting Create Partition Deﬁnition and give it a name. Now synthesis can be
run.
After the end of the synthesis run, open the synthesized design, and right-click
the reconﬁgurable partition in the netlist tab, and select

The DFX methodology comes with some limitations and severe unsupported
features, that have to be considered when implementing the reconﬁgurable mod-

project. Support for these features may be added in future releases of Vivado.
• IP sources cannot be top level in the RM, they have to be at a lower level.
• Block diagrams cannot be included in RMs at all.
• Modules within a block diagram cannot be set as RMs.
• Simulation is not supported within the project.

4.2.1
DFX Controller LKM
The DFX Controller Linux kernel module is a fully custom developed device
driver, that has basic functionality used to control the Xilinx DFX Controller
IP. When the LKM is loaded into the running kernel, it will register the device
driver, then map the DFX controller’s AXI slave address into local memory.
The last thing it will do is enable the ICAP interface, by resetting bit 27 of
the devcfg_CRTL register (0xf8007000), as the default conﬁguration interface
is PCAP, and the two conﬁguration interfaces are mutual exclusive.
Once the module is loaded, a node has be created for it, to let applications talk
to the driver, and this is done by issuing the command:
mknod /dev/dfx_control c ### 0
Where ### is the major number of the driver when loaded. For this Petalinux
implementation, the major number has always been 244, but be aware, that the
number can change if other modules are loaded beforehand.
Now the driver is ready to accept commands via the ioctl system call from
an application. The accepted ioctl commands of the DFX controller are listed
in Table 4.3. Some of the commands take an argument, of either the address
of the bitstream or the size of the bitstream. The address argument must be
the physical address of the bitstream in memory, because the DFX Controller
component, only handles physical memory. The size argument is the size of the
bitstream for that reconﬁgurable module and is measured in bytes.
There are two triggers, that are used to instruct the DFX Controller to begin
the conﬁguration of an RM. One command will pull the running status of the
DFX Controller, and another command will collect the information stored in
the DFX Controller of the reconﬁgurable modules. The latter two commands
are mostly used for debugging.

The OpenCV library is used in both hardware and software implementations.
The hardware uses OpenCV for reading and decoding the images, and for writ-
ing and encoding images back to the persistent storage, and uses the hardware
for the computation. The software implementation makes use of OpenCV for
all of the operations, including the computation.
The test application dataﬂow for the hardware test is shown below.
1. Use OpenCV to read and decode an image from persistent storage.
2. Use OpenCV to allocate local memory for the output of the IP core.
3. Copy image to physical memory address.
4. Conﬁgure the IP core with the size of the image, and location in memory.
5. Start the IP core computation, and wait until completion.
6. Copy the output image from physical memory to the previously allocated
memory.
7. Use OpenCV to encode and write the output image to persistent storage.
Because the hardware accelerators fetch the data with the AXI bus, which only
operates with physical memory addresses, the image data has to be loaded in
a known memory location.
But since user applications in Linux use virtual
address space, a physical address of the data cannot be obtained from local
data. Therefore a workaround has to be added, where the applications have to
ﬁrst load and decode the data into local memory space with the use of OpenCV,
then copy the data to a known physical memory address, where the accelerator
can fetch the data, and push the output to another known physical address.
Once the accelerator is done with the computation, the application has to copy
the data from physical memory back to the virtual address space of the OpenCV
matrix, before it can be saved in persistent storage. This workaround means
that the hardware implementation wastes some time on a lot of copy operations.
The resource managing scheme implemented in the test applications is a simple
static implementation. When the application wants to use hardware accelera-
tion, it sends the trigger request to the DFX Controller via the ioctl system call
and awaits its success. Once done it can use the accelerator, until a new trigger
request is issued to reconﬁgure to the other RM.

Chapter 6
Conclusion
This thesis work has dealt with partial reconﬁguration and the methodologies
of the paradigm, and the Xilinx proprietary Dynamic Function eXchange has
been explored in depth. The basics of partial reconﬁguration are covered, and
the methodologies and their limitations are detailed.
The work also integrated the paradigm into a Linux driven operating system,
Petalinux, creating a Linux kernel module for controlling the DFX controller, as
well as several test applications, that uses the UIO framework to communicate
with the reconﬁgurable modules.
Two reconﬁgurable modules have been created, that provide a use-case in the
real world. The Sobel RM provided a high speedup over the OpenCV imple-
mentation, and the histogram equalization RM saw a moderate speedup. We
learned that having fast storage is important when computing.
For the partial reconﬁguration side of things, the conﬁguration latency was
shown to be larger than expected, but we also saw that the full bandwidth of
the ICAP interface was utilized. Which means that the high latency is caused
by the large size of the Pblock and the corresponding partial bitstream.
In conclusion, an interesting subject, that has many real-world applications and
use-cases. It’s certainly a subject that deserves more research and development.

6.1
Future Developments
Because of many issues with the implementation in other parts of the system,
the more advanced managing schemes have not been implemented, thus only a
simple managing scheme was included.
For faster RM conﬁguration with the DFX Controller, compression can be used
to minimize the size of the binary bitstreams. The size reduction is estimated by
Xilinx [Xil20a] to be between 30% and 70%. Alternatives to using compression
could also be to try and minimize the Pblock, since that will make smaller binary
ﬁles. It may not be possible to make the Pblock much smaller, since the RMs
already ﬁlls much of it, and timing and routing could likely be a problem.
A performance drop is caused because the PL is not able to address virtual
memory locations, and the workaround was implemented. Maybe either, sup-
port for virtual addresses can be implemented in the PL, or more likely, a way
to load the data directly into the physical memory from user applications can
be implemented.
