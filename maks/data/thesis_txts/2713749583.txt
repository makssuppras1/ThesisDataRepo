Abstract
Improvements of quantum computers performance will make in a near future current testing using
the randomized method with simulators obsolete which means there needs to be a better alternative.
Mirror Circuit and Cycle Benchmarking are the new state of the art performance testing methods which
should allow to more precisely measure the error rate of the system. These methods would not require
anymore to run the circuit on a simulator to get correct results. Currently, it is unknown which method
predicts the error rate more precisely and both are being tested on quantum computers. To execute
the algorithm it needs to be implemented in a provider specific language which means that it must be
re-implemented for each quantum computer provider independently.
This thesis proposes a solution in a form of a highly configurable, extensible yet easy to use quantum
performance benchmarking tool independent of a quantum computer provider architecture. The tool
includes the improved performance testing methods out-of-the-box. It can be used as a standalone
application or as an external library for a different project. The solution should be used as a foundation
on which greater and more complicated algorithms are developed for future quantum computers with
a higher number of qubits and lower error rates.
The state of the tool is that without any additional extensions it can perform quantum system
performance measurements and compute the Quantum Volume of the system. There is also a possibility
to define a custom circuit as a base template from which the testing circuits are generated. The circuits
and provider jobs can be highly customized to specific needs.
We demonstrate the tool functionality using IBMQ Provider, where we measure the Quantum
Volume of all available quantum computers. Then we execute all current and proposed performance
testing methods and the Deutsch-Jozsa algorithm to measure the error rate of each run. Using the
measurements we compare how precisely each method shows the performance of the system for a real
quantum algorithm execution.

CHAPTER 1
Introduction
Quantum computers is an interesting field of study. Despite it being in an early stage of development it
has already shown a great potential to solve NP problems. These contain scheduling problems, search
optimization problems and more. As an example, Shor’s algorithm might break the encryption codes
such as RSA with enough qubits for computation. Even though it sounds promising the computation
power of quantum computers is not great enough for any practical algorithm to be executed. Also, the
error rate of the machine is much higher than on a classical computer and causes a lot of development
issues. In a few words, this could be described as a Noisy Intermediate Scale Quantum Era (NISQ). To
improve performance we need to increase the number of qubits which is being done at a fast pace. But
it is also important to focus on the other big issue of quantum computers which is the error rate of the
system.
There are some error mitigation processes that can be applied. The current state does not allow to
use of error correction circuits, so we have to rely on calibration runs of quantum computers or mapping
physical qubits to logical ones using techniques such as Toric codes. But using any of these processes
could lead to a possible performance regression with a small or possibly no gain on the error correction.
Because of this, it is important to measure both the performance and error rate of the system and not
only focus on a single field. Another important property of the quantum computer is that the error rate
seems to randomly change over time. Therefore we cannot compare only different machines, but also
the same machine in different time periods. This could help to know what algorithm should be run on
the system at a specific time frame.
For different algorithms, different requirements need to be met.
By doing random performance
testing we will not get useful information about the performance of the specific group of algorithms.
To solve this we can use some type of structured benchmarking that could give us more information
on whether the quantum computer in that exact time fits the best to specific requirements. Quantum
machine is a spare resource and it is important to utilize them eﬀiciently.
This thesis will focus on the ways how we can compare quantum computers or their configurations
with each other and if there is a consistent way to say whether one computer is objectively better
performing with respect to speed and correctness than the other.
There are already some existing
structured testing methods, for example the Mirror Circuits or Cycle Benchmarking. The work will be
based on these techniques and will try to apply them to our specific use case. It is needed that the
technique is independent of specific computer architecture. This way any future quantum computer can
be assessed. The used technique should also easily scale with the number of qubits used for the given
circuit. This number is steadily increasing and a lot of testing algorithms depend on computing correct
results on a classical computer, which can take up to O(2n) in time and space, where n is a number of
qubits. Those will not be usable in the near future anymore.
For this purpose highly configurable, extensible yet easy to use quantum performance benchmarking
tool is developed. The tool implements the most common known benchmark techniques but also allows
users to define their own circuits for testing. In this way, users can customize benchmarks to their
requirements without any need for complicated programming. The tool can communicate with any
implemented provider independent of a used language. Therefore a test suite can be run on multiple
quantum computers without any modifications. Having this tool available could potentially reduce the
complexity for a user to test out different combinations of quantum circuit algorithms on a quantum
computer and optimize its run to get more precise results.

1.1 Performance Benchmarking Problem

1.1
Performance Benchmarking Problem
As mentioned above, currently we are in the early phase of the development on quantum computers.
A lot of research needs to be done before we are able to have a practical quantum system that can be
used for day to day problems. The first step is to provide the start of working on the issues of the
quantum computers as easy as possible without the need for long term studying and understanding of
all small details of the system Today anyone who wants to start working on an issue needs to understand
programming, choose a specific provider and implement solution only for them. For instance to start
working on IBMQ devices a user needs to implement the solution in Qiskit and communicate only with
IBMQ computers. There is an initiative to define an open source quantum language, but it is in the
far distant future before it gets implemented by all providers. And an open question is whether it will
be implemented at all.
This makes it harder for any new user to start working on quantum computers and test their solution
on multiple different providers. The implementation needs to be provider specific without any possibility
to use a library from a different provider SDK that user can already know. Another issue is when users
want to test some specific method of the benchmark they need to implement it by themselves. There is
no common implementation that would work on any provider. Each algorithm needs to be implemented
many times using a specific language and libraries defined by the provider.
The issue can be even more prominent when more quantum computers from different providers get
developed. If the trend will continue then more custom proprietary solutions will be implemented and
users would need to duplicate their work without any benefit from their side. Additionally, when more
complicated benchmark methods are developed then implementing each from the ground up will again
increase the complexity of starting to work on quantum computers which should be avoided. This
complexity should not increase when the ecosystem is more developed than is now.
Tool Usability Requirements
Before we can start solving some of the problems we first need to identify what are the strict requirements
for the tool that need to be handled. With these rules defined, we are able to keep the tool focused on
a single set of issues. This means that it will not become a toolset of random features that try to solve
many completely different use cases. Next, we will introduce all of these restrictions.
Firstly, the tool needs to be provider agnostic which means thof at there is a restriction on func-
tionality. If some feature is not implemented by a supported provider then this functionality cannot
be part of the tool. The subset of features needs to be well defined and can be extended only when
supported by all providers. This also needs to hold the other way around. Before any provider can be
added they need to support all already implemented functionality.
Secondly, for the tool to be as simple to use as possible, there cannot be required dependencies for it
to work. The tool should be self contained and only optional features can use additional resources when
needed. This means that the tool should be a single executable or library file which can be executed or
included and will just work.
The last requirement is it needs to be easily configurable and extensible. The idea is that the current
feature set will not be in the future suﬀicient and there will be a need to extend the functionality with
additional performance benchmarks or other algorithms. The tool should contain configuration to not
restrict users in a single use case, but rather allow tweaking the tool execution to specific needs. This
should be done using some type of extensions systems which allows replacing any part of the tool when
possible. As an example, if there is an already existing solution for a specific provider then it should be
easy to integrate rather than trying to re-implement the solution again. An important note is that it
should not create a performance overhead for any other use case. When the tool can be extended it also
needs to be possible to integrate it into an existing ecosystem of tools used in quantum programming.
Therefore it needs to have a public interface for communication with other applications. This way we
are capable to automatize the tool execution and using the result in other stages of development.

Use Case Scenarios
Next, we will define use cases that should be supported by the tool. Each defines a group of individual
scenarios which were identified as an important functionality that can improve the current state of
working with quantum computers.
As mentioned before the main use case is when we need to compare the performance of two or more
quantum computers or a single computer at different time points. Currently, we cannot output a single
value that will define which quantum system has better performance for all algorithms. Each algorithm
can be more sensitive to a different set of errors. Therefore multiple different measurements need to be
done to find the most suited quantum computer for a specific algorithm.
The error rate changes significantly over time and even when calibration measurement will show the
algorithm should be successful it does not need to be true when executed at a later time. Therefore
there should be an option to do quick measurements before the algorithm run to measure the error rate
of the system. This should give an insight into whether the execution will be successful or not.
The last use case is when some algorithm should be implemented in a form independent of a specific
provider platform. The idea is to allow a single circuit definition that can be used on many different
quantum computers. Each provider needs to be supported and fulfil the requirements defined before.
1.2
Proposed Tool Solution
The issue that we are trying to solve needs to have as simple use case as possible. The idea is to develop
a set of tools that will allow a user to start working on quantum computers straight away without any
complexity of setting up the project. Also when the user gets more experienced with the ecosystem and
decides to change provider or extend their solution it should be as simple as possible.
With this in mind, we develop a tool that can be used as a standalone application or integrated into
a bigger project as a library with already implemented all parts that are needed for simple use cases
described before. Therefore, if a user wants to do a simple custom test then it should not be needed
to implement anything more than a template circuit from which all of the necessary more complicated
circuits will get generated. There are many more algorithms, such as Quantum Volumes, supported out
of the box that should make testing of quantum computers much easier.
If needed the tool has a public interface which allows it to connect to other applications and output
serialized results that can be easily interpreted by the next stages of the execution pipeline. This allows
users to extend the tool with any algorithms they need and still use all other features such as automatic
circuit generating of it. The extension is not restricted only to an algorithm definition. Any part of the
tool is interchangeable and simple extensible using a custom developed scripts or shared libraries.
The tool is developed in a way that it is provider independent and can be easily extended with future
providers. This means that any solution developed by a user can be used at any supported quantum
computer and if the support is not implemented it needs to be done only once in the tool. Then any
user solution can be run on the newly supported quantum computer.
To fulfil the requirements we need to first define a subset of features that are implemented by
supported providers. This set will define what other use cases can be supported and if the solution with
a custom set of functionality is maintainable for a long time. The tool needs to expose configuration
that can be used for application options, but also as library arguments. This interface should be unified
over both execution models.
When all requirements are fulfilled we need to implement the use cases described before. To compare
quantum system performance we need to implement algorithms that can do these measurements. This
needs to be supported by the tool without any extension required. When this is supported and execution
speed is eﬀicient enough it can also be used as a measurement before the algorithm is run to find out if
the execution will be successful. All these implementations need to be provider agnostic and therefore
will allow the execution of custom algorithm defined by the user.

CHAPTER 2
Basic Principles of
Quantum Computers
Quantum computing holds the promise to solve some of the biggest challenges of this era. These contain
for example molecule simulations, optimization problems and cryptography.
First, we will introduce some basic principles on which quantum computers are built on and com-
pare them with how classical computers interact. Then we describe what is the usability of quantum
computers in the current era and how to improve this. Next, we will focus on a specific theoretical
sub-part that this thesis will work with and how it can be used. The main topic will be the error
correction of quantum computers and how we can measure the performance of the system.
Before any more complicated theory can be introduced we need to cover all basic principles such
as quantum state representation and changing the quantum state for single and multiple qubits using
quantum gates. We will show how these states can be visualized and what are the physical properties of
a quantum system on which the theory is built on. There are a lot of similarities with classical computers
so in some cases the new concept will be introduced with reference to the classical counterpart. This
should help to understand how quantum gates interact with qubits and why it is possible for quantum
computers to behave the way they do.
Next, we will introduce the most common quantum gates and what is their functionality. There are
single and double qubit gates which are then later used in the next chapter. Each gate has a special
property and we try to create an overview of each gate and its use cases. The list of gates is extensive
but follows a repeating structure so it should be easier to find all relevant information.
2.1
Introduction to Quantum Computing
Classical computers are reaching their limit with the number of transistors increasing yearly and Moore’s
law [SS15] cannot be applied anymore.
With respect to this, there are some problems that these
computers cannot solve even when we could still continue to double the number of transistors. These
are called the NP-complete problems [Ull75], which are problems that only have polynomial solutions.
As an example, we can shortly introduce the travelling salesman problem [JRR95] which is defined as
follows. Given a list of cities and the distances between each pair of them, what is the shortest possible
route that visits each city exactly once and returns to the starting city? For simple sets, the solution can
still be found using classical computers, but with the higher number of nodes the complexity grows and
after some limit, it becomes impossible to compute. The design of the classical computers do not allow
us to overcome this issue. That is why there is a new group of computers, called quantum computers,
that has shown the potential to solve these issues in a non-polynomial time or at least their subset.
There is a complexity group called BQP problems [Bin], which stands for bounded-error quantum
polynomial time problems. These are the decision problems (yes or no problems) that quantum com-
puters can eﬀiciently solve. It has been proven that the P group is a subset of the BQP group. This
means that quantum computers can solve all problems that a classical computer can solve.
It has been shown that BQP problems are not a strict subset of the NP-class [RT19]. Therefore in
1993, Ethan Bernstein and Umesh Vazirani [BV97] defined another complexity class called the PH class

or Polynomial Hierarchy. It is a generalization of NP class. Problems in the PH class are NP problems
extended with expressions whether it is true for all or does it exist for a particular x.
It is not proven that quantum computers can solve all NP problems, but there are some examples
where it has been shown that there exists a quantum algorithm which can solve some specific NP
problems.
For example, we can mention the two most important problems that can be solved by
quantum computers. First, there is Shor’s algorithm [Mon+16]. It is able to solve the NP-hard problem
of factoring large numbers, which means that any encryption depending on this (such as RSA) is easily
solvable. Next, researchers at the Chalmers University of Technology have been able to solve a small
part of a logistics problem in the aviation industry called the Tail Assignment Problem [Vik+20], which
is assigning airplanes to flights with the goal of minimizing connection times between flights and keeping
in mind maintenance constraints.
Quantum computers use different mechanical principles to describe the state of the system and their
interaction. The classical system follows rules that we are used to in classical mechanics. But using
quantum mechanical principles we can remove these limitations and do some types of computations
that were not possible. We can show that quantum computers can do some additional operations that
were not previously possible, but also important note is that they are Turing complete [Sho98] and
therefore can do the same work as a classical one.
The quantum mechanical principles used in quantum computers are mainly the principle of super-
position, qubits entanglements and quantum interference. We will introduce all of these principles later.
To get a better understanding of quantum computers we need to introduce some principles that are
counter-intuitive, but important for the quantum system. These are:
1. A physical system in a definite state can still behave randomly.
2. Two systems that are too far apart to influence each other can nevertheless behave in ways that,
though individually random, are somehow strongly correlated.
Throughout the theoretical part, we will compare classical systems with quantum for better under-
standing and to not need to describe all principles from the ground up. This will be mainly used for
basic computer principles and naming conventions. Most of the quantum elements will have a classical
counterpart that is used as a base and then extended or altered with some additional behaviour.
With classical computers, we are used to not having bit flips or any other types of errors on a
hardware level. This is not the case for quantum computers. These systems are using highly unstable
components which can randomly change state throughout the computation. The more operations or
longer the execution last the higher error rate can occur. This rate can increase to a level where instead
of an error state we get a false positive state, which is a state where a single error occurs more often
than any other state and then this state can be wrongly used as a correct result.
2.1.1
The Qubit
The same way as for a classical computer there is a bit as a building block of the whole system for
quantum computers this is the quantum bit, or in short qubit [Tsa10]. The basic difference is that
a classical bit can have only one of the two states 1, 0, but the state of a qubit can be any linear
combination of the two states.
This could lead to the thinking that quantum computers are analog computers because of the real
value of the qubit, but it should be noted that this is not the case. Even though qubit can have any
value out of this throughout the computation. The qubit can output again only one of the values 1, 0.
Therefore these computers are not analog nor digital. They are a combination of both worlds.
There exist many ways how we can build qubits on real hardware, but currently, two methods are
mainly used. One method is using trapped ions [Oze11] and the other uses superconductors [Kja+20].
Both have their advantages and disadvantages but currently, superconductors seem to be a better fit
for a qubit. We will not focus on the physical side of the system. There are some errors that can occur
because of the underlying architecture, but our focus will be on issues that rather occur because of the
structure of the algorithm.

where each number represents with what probability the system is in this state. If the system cannot
be in this state we write 0 and if it is for sure in another state we write 1. Sum of these numbers must
be equal to 1.

2.3.1
Normalization
We can see that amplitudes are closely related to the probabilities. The probability of the state must
be 1. This means that also the magnitude of the statevector must be equal to 1.
⟨ϕ|ϕ⟩= 1
Thus if:
|ϕ⟩= α |0⟩+ β |1⟩
Then:
|α|2 + |β|2 = 1
From the equation we can see that adding powers of magnitudes must be equal to 1 and each define
the probability of the state.
2.3.2
Alternative Measurement
The measurement rule defines only with what probability the state |ϕ⟩is measured as |x⟩. It does not
define that the qubit can be only in one of the states |0⟩or |1⟩.
There is no rule that defines we need to measure only this way. There is an infinite number of ways
how qubit can be measured. Measuring qubit differently gives us states with different magnitudes. This
means that if qubit is in state |1⟩we do a measurement we expect to measure this value for sure. What
we can do is choose different eigenstates as a norm for measurement and then we will not get this state
with certainty, but only with some probability based on the way we chose to measure.
2.3.3
Global Phase
From the previous chapter about the measurement rules 2.3 we know that measuring state |1⟩gives the
output 1 with certainty. If we choose a different state, for example i |1⟩. After applying the measurement
rule we can see that factor of i disappears.
|⟨x|(i |1⟩)⟩|2 = |i ⟨x|1⟩|2 = |⟨x|1⟩|2
This effect is completely independent of the state |x⟩. This means that the probability of the state
i |1⟩is identical as for the state |1⟩. Because the only way how we can get any information from the
system is by measuring, it implies that these two states are equivalent in all ways that are physically
relevant.
Note that this global phase is distinct from the phase differences between terms in a superposition.
This effect is known as a relative phase and cannot be ignored.
2.3.4
The Observer Effect
Before measurement state can be in a superposition, but after the measurement it collapses to some
specific state and now we know for sure in what state the system is. Any additional measurement will
always output the same result. This effect is referred to as a wave collapse [Ber98] and changes the
state of the qubit. To achieve the full potential of the quantum system we must restrict measurements
to the end of the algorithm. Doing a measurement after each operation would lead to a system that is
not different from a classical computer.
There are many theories on why measurement collapses the quantum state. We describe it as an
observation which is never fully explained and it is hard to define on what scale quantum mechanics
change to classical mechanics. There are two most common explanations for this phenomenon, but many

the qubits. This gate can be selected in both directions. Or if we remove the other gate we get a
different circuit with the same effect. There will be no functional differences. That is why there are two
implementations of the swap gate which are the same. In figure 2.14 there is a visual representation of

2.8.4
The Gate Identities
Implementing all important gates on a physical layer is almost impossible, therefore we use a gate
identity [LCJ13] to implement other gates using already existing ones.
As an example, we can take a look at how to create a controlled-Z gate from a CNOT gate. This
gate is doing the same operation as a CX gate, except instead of rotating around X-axis it rotates
around the Z-axis. There are two qubits because the gate is controlled. The operation is applied only
if the control qubit is in |1⟩.
Now we will describe how to create a CZ gate from already defined ones. First, we define that qubit
starts in the |1⟩or |0⟩state. Now we need to put a qubit into a superposition. To do this we can use
the Hadamard gate. This will transform qubit to |+⟩or |−⟩respectively. Now we can apply the Z gate
to rotate the qubit around the Z axis. This has the same effect as the X gate in the controlled-X gate.
H · X · H = Z
H · Z · H = X
Here we can see that we are able to do any controlled rotation gate using the same principle as
described above. This applies to almost all gates. There is only a subset of gates that needs to be
implemented on a physical layer and then the rest can be simulated using the base ones.

Methodology
Having a single qubit in a superposition is nowadays not a complicated thing to do [Web+10]. The
issue is to have this superposition stable for a long period of time and apply quantum gates without
any interference or errors on the qubit. The system needs to be tested as a whole. Trying all qubit
separately does not give us the overall error rate of the system. For example, if the probability of a
state flip of a qubit is 0.05% and we have a system with 5 qubits it could seem that the error rate of
the system is 0.25%. This is not correct. We need to take into account also entanglement of qubits and
their interference. The realistic error rate will be much higher than the computed.
With this in mind, we need to develop more complicated error rate metrics that show the realistic
error rate of the system. Currently, only randomized unstructured test suites were developed [CH99].
There were proposed more structured algorithms which do not require the quantum simulator to com-
pute correct outputs.
Another issue with quantum system testing is that each algorithm has different requirements. Some
require greater depth of the algorithm but do not require many qubits. Other algorithms work the
other way around. This is the reason why only testing algorithms with the same depth and width is
not suitable. With this in mind there is also a possibility to change the algorithm so it does not require
as many qubits but instead increases the depth of the algorithm [Ara+21].
In this chapter, we will describe how the basics of error testing and correction works on the quantum
computer. Also, we will try to use a more complicated approach to test the system more precisely. It
is important to know how algorithms are built to not test something that has no value for the user.
Then how do quantum computers interact with a classical computer. This communication can be a
bottleneck and needs to also be tested when possible.
When all basic information for more complicated algorithms is described we will focus on different
testing techniques that are implemented in the benchmark tool. All do have some advantages and
disadvantages. Each can be used to test different aspects of the quantum computer. This will focus
on the user customizability of the algorithm and does not implement some single specific technique the
way the other do.

3.1.2
Stages of Quantum Algorithms
To be able to know what errors can occur when executing an algorithm we first need to understand
how these algorithms are built off. We could categorize them based on the problem they are trying to
solve. For example, some used techniques for developing quantum algorithms include phase kick-back
or estimation, the quantum Fourier transformation, quantum walk and amplitude amplification. But
there is a structure that most algorithms follow. We will focus on this structure and try to describe
stages that can be used in a test suite to improve error rate accuracy.
First, there is always an initialization stage. This puts all qubits to a specific state that the algorithm
requires to output the correct result. This can be any combination of a single qubit Clifford gates but
needs to be well defined.
The next step is to apply a sequence of gates that build up the quantum algorithm. This could
consist of the quantum oracle circuit. Quantum oracle is a black box operation that is used as an input
to another algorithm [BB94]. Generally, we can describe this as a function that applies an unknown
operation on an input and based on the output we need to find out what type of operation was used.
We can represent it as U : (x, y) −→(x, y ⊕f(x)), where ∀x, y ∈0, 1.
The last step is always to do some measurements. This operation is destructive and therefore last
in the circuit. There will never be measurements and after some other gates as part of the algorithm.
Generally, errors can be also split to three groups as with the algorithm stages.
To get a better
understanding of how the algorithms can be implemented, there is a simple algorithm as an example.
3.1.3
Deutsch-Jozsa Algorithm
Next, we will introduce one of the examples of how quantum algorithms can be used to solve a problem
much faster than a classical computer. This problem is named as Deutsch-Jozsa problem. The problem
starts that we have an unknown Boolean function f, which takes a string of bits as an input and returns
either 0 or 1.
f(x0, x1, x2, ...) −→{0, 1},
where xn ∈{0, 1}
Property of the function f is that it is guaranteed to be either balanced or constant. The constant
function always returns the same value independent of the input. This means that for each input the
output will be always 0. On the other hand, the balanced function returns 0 for exactly half of the
inputs and for the other half it returns 1. The problem is to determine whether the function is balanced
or constant.
Classical solution
Using a classical computer there is no better way how to find a solution than just try to run the Boolean
function many times with random input. We could try to find some heuristic to select some sets of
==PAGE:30==

inputs and improve mean runtime, but the worst case will always be the same. In the best scenario,
we need only two runs to find out whether the function is balanced or not. For example, if we get two
solutions as below then we know the function is balanced.
f(0, 0, 0, 0, ...) −→0
f(1, 0, 0, 0, ...) −→1
In the worst case scenario we would need to test half the times plus one to be sure that the function
is constant. The reason for it is that even with half of the tests we cannot be sure that the next output
will be again the same, but could be different and the function is balanced instead. The probability of
this happening is really low, but not zero. This means that if the length of the input is n, we need to
have at least 2n−1 + 1 runs to output whether the function is constant or not.

3.2
Classical-Quantum Systems Interaction
In the current NISQ era having the whole algorithm running on the quantum computer is almost
impossible.
The error rate would be too great to get some reasonable result.
Therefore, quantum
algorithms use an idea to run critical parts for the algorithm to work on the quantum computer, but
anything else possible run on a classical computer [SS78]. This uses the advantage of both systems and
enables to run of algorithms that would not be possible in the current state.
These algorithms are mostly designed in the same way as other quantum algorithms. The difference
is that they are using an iterative approach to get some reasonable result before any critical error could
occur.
In figure 3.3 we can see that each iteration consists of a classical part which sets up an environment
for the quantum state with updated parameters. Then the quantum algorithm is run which outputs the
best possible solution for it. If the result is correct then the algorithm terminates. Otherwise, it runs
another iteration. This approach is called Hybrid Quantum Computing [Llo03]. We can see that the
algorithm is not the most optimal, because we need to keep algorithm depth within some limit and then
switch from superposition to classical state. But currently, it allows us to compute more complicated
circuits without any critical issue. This type of communication between classical and quantum systems
can be done in multiple ways. There are two main ones that are described below. Both do need different
testing techniques and therefore it is important to know what type of architecture quantum computer
uses.

Each quantum computer provider implements its own programming language and the SDK just to
interact with the computer.
For example, the IBMQ provider has the Qiskit SDK for developing
quantum circuits for their computers. Instead of using a proprietary language for circuit development,
there is an initiative to have a common open-source language that can be used on any quantum computer.
This language is called OpenQasm [Cro+21] and has currently the third edition with extended classical
algorithms support.
The language has some similarities with the asm language but is more complicated with extended
features such as if statements, loops and much more. The language focuses on how to describe quantum
algorithm without knowledge of the underlying physical layer, but if needed there is support to describe
more in detail how the quantum circuit should be implemented on the hardware. There is also support
for classical algorithms. The idea is that most of the quantum algorithms will be used in combination
with classical algorithms. There will be part of the process executed on a quantum computer. Then the
classical computer will use this result to compute new initial conditions and again quantum algorithm
will be executed to get a more precise result.
The language has some built-in gates and instructions that are common for all quantum computers.
These gates are some of the introduced above and instructions are for initialization and measurement.
Other than the built-in gates user can define any other they want and then use this gate throughout
the computations. The lowest level of implementation is called Pulse-level description and is needed
only when some specific operations for some hardware needs to be done.

3.4
Quantum Computer Providers
In the current state it is not possible to have access to a quantum computer whenever needed for
development, but also for testing. Therefore there is a need for quantum computer simulators that can
run quantum algorithms on classical computers. This task is done using virtualization where a classical
computer does all the computations that quantum would normally do. We have shown that quantum
computers can compute a superset of problems and therefore simulations can never be optimal and
there always be a limit for a simulation size.
The complexity grows exponentially with the number of qubits and with the depth of the cir-
cuit [ZSW20]. At some level, the simulations will be so complicated that it will be impossible to do any
simulations even on supercomputers. If this would not be the case then there is no need for quantum
computers in the first place. Here we will introduce the most commonly used quantum simulators and
compare the advantages and disadvantages of each of them.

3.4.1
Qiskit
The most used simulator is developed by IBM company as part of Qiskit SDK [Cro18]. It does not
contain only simulators, but also allows connecting to their public quantum computers when an account
is created. It uses Python as a programming language but is implemented in C++ for performance
purposes.
This SDK has a list of supported simulators. Each is an ideal simulator by default, which means that
they are trying to simulate a computer without any errors. We can define what type of errors can occur
in the system The most important error types are phase flip, bit flip and decoherence errors. These
can be simulated using phase_damping_error, amplitude_damping_error and coherent_unitary_error.
We can also combine errors together to create a more realistic system.
The existing quantum simulators are described later. Each has an additional configuration that can
change the behaviour. For example, if simulation should run on a graphical unit or what computation
precision should be used.
The most common is AerSimulator, which is the most generic backend
available in the Qiskit SDK. This backend has a lot of simulation methods that can be chosen from.
Results from each are really similar therefore we can keep the default one. Next are the deprecated ones
StatevectorSimulator, UnitarySimulator and PulseSimulator. All have been replaced by AerSimulator
and are not needed anymore.
The last one is QasmSimulator which we use in the tool for all simulated runs. The reason is that
all generated circuits are in OpenQasm format and are not using the qiskit SDK definition to make the
circuit be usable on as many computers as possible.
3.4.2
cuQuantum
NVIDIA cuQuantum [SF] is an SDK of optimized libraries and tools for accelerating quantum com-
puting workflows. Using NVIDIA GPU Tensor Core GPUs and cuQuantum SDK, we can speed up
the quantum circuit simulations based on a state vector and tensor network methods. This is a low
level implementation and requires specific hardware for it to be run, but results are a great magnitude
faster than classical simulations. This framework is integrated with the most common providers such
as Qiskit and Cirq. It uses C++ language for implementation. Next, we will introduce two simulation
methods used by this framework. Both have advantages in different scenarios.
First, there is the state vector simulation [Jon+19]. it keeps track of the entire state of the system
in time through each gate operation. The advantage of this approach is that it can simulate circuits
with great depth and highly entangled noisy qubits. The disadvantage is that the complexity grows
with a number of qubits. For circuits with a great number of qubits, but no circuit depth the second
approach is more suitable. To show the simulation performance. An NVIDIA DGX A100 with eight
NVIDIA A100 80GB Tensor Core GPUs can simulate up to 36 qubits.
The second approach is to use tensor networks [Lev21]. The system scales better with the number
of qubits rather than the number of quantum gates. This makes it possible to simulate very large
qubit counts with smaller gate counts on large supercomputers. Using this algorithm they were able
to simulate a variational quantum algorithm for solving the MaxCut optimization problem using 1688
qubits to encode 3375 vertices on an NVIDIA DGX SuperPOD.

simulators or connect to public computers. The quantum simulator can use cuQuantum when GPU is
supported and uses wave functions or density matrices. The Cirq framework also works with state of
the art simulator qsim.

3.5
Error Simulation Models
Simulating a system without any error does not give us any additional information that could be used
to improve the error rate of the real computer. Some errors are simple which means that they apply
only on a single qubit with some probability and are easily computable. Some are more complicated in a
way that we need to work with the system as a whole and cannot isolate individual qubits. These errors
are the ones that cause that we just cannot count error probability from the error rate of individual
qubits and their errors.
Therefore we will try to simulate the most common errors that can happen in the system. All
errors can be divided into specific categories whether the error is applied to a single qubit or multiple
qubits. Both groups do not contribute to the error rate approximately the same. Creating an isolated
high-quality qubit without a great error is nowadays a fairly simple task. The diﬀiculty is to make a
device that contains many qubits that all work within an acceptable error rate.
Another issue is that even when we decrease the error rate of the system, it still does not mean that
the system will be more powerful or faster in computations. In this specifically, IBM recognized [Pro+19]
some rules in which this holds:
1. Adding more qubits does not increase the computational power of the device if an error occurs
before they can all interact in a circuit.
2. Decreasing the error rate per operation does not increase the computational power if you can
already reliably access any state on all available qubits.
3. IBM defines the effective size of the device as the largest number of qubits on which the processor
can reliably produce a random state.
Also, knowing the greatest depth of the circuit does not answer any questions about what algorithm
can successfully be run on this system. This means that classical Quantum Volumes where we increase

depth and width always by one in each iteration as described in section 3.6.6 does not cover some use
cases of the quantum algorithms. Therefore we want to generalize Quantum Volumes to iterate over all
combinations of depth and width until the circuit increased by one in both directions will fail.
If we look at the most common algorithms we can see that they do not have a square complexity
with respect to depth and width:
1. Scrambling algorithm: width n qubits, but O(n) in depth
2. Shor’s algorithm: width O(n), but O(n3) in depth
3. Groover’s algorithm: width O(n) qubits, but 0(2n12) repetitions of a O(poly(n)) oracle subroutine
For some algorithms there is an option to choose whether the circuit should be more in depth or
width. As an example polylog(n) algorithm is possible if we use more qubits than is the minimum for
the algorithm to be run [CW00].
3.5.1
Simulated Error Types
First, we will describe basic errors that can happen on single qubits. We can further divide these errors
to a group which happens on some operations and then which are introduced in the system randomly
even when nothing is executed. The first group covers all three stages of the system. Later, we will
talk about each part separately. The multi qubit errors are more of error models that are applied on
multiple qubits together. As mentioned above, we cannot just multiply the error of each qubit and use
this as a system error.
3.5.2
Single Qubit Errors
Generic algorithm consists of three stages. Each of them can introduce some error to the system. First,
there is a reset state which sets the qubit to an exact state. Here are two steps, first we use reset which
can introduce a reset error. Next, we use a combination of gates to set the state of a qubit. In this
step, it can have the same errors as the second stage of the algorithm.
The second stage contains gates which are applied to the qubits. Here we can get errors that are
introduced when some gate is applied. As an example of this type of an error, we can mention the
bit flip error which changes the state for example from the state |1⟩to |0⟩. Or phase flip error which
changes state from |+⟩to |−⟩.
The last stage is when we measure the system state and get the result of the algorithm. This is a
destructive operation and therefore there are no other steps ever needed. In this step, we can introduce
the measurement error, which means that even when there is a correct result the measurement will
output an incorrect one.
The measurement error is not the same for both measurable states of a
qubit [TQ19]. There is a noticeably higher error rate when measuring all one string of qubit states than
to measure error for all zero string. This can be improved by introducing an X gate that will flip the
qubit state before the measurement.
The next group are errors which occur randomly without any operation being executed. It is the
main reason why quantum computations are unstable and we cannot run algorithms for a long time.
Therefore some iterative approaches have been developed to overcome this restriction.
One of the
errors is for example thermal relaxation [Ros+20]. This model, in short, uses tweakable variables which
are a thermal relaxation time constant T1 and a dephasing time constant T2 which simulates qubit
decoherence and randomized state change respectively.
We can define quantum error in Qiskit error model using a simple algorithm. The thermal model is
predefined. The only restriction is that T2 ≤2 × T1.

3.5 Error Simulation Models

3.5.3
Multi Qubit Errors
To simulate multi qubit errors we need to use some models that can simulate their behaviour. There
is a great number of methods which differentiate in complexity and precision of error simulations. We
will describe the basic models that are used in the tool to get a more precise simulation without too
great complexity.
First, there is the Pauli error channel [Rus00] which is defined as a list of Pauli gates and their error
probabilities [(P0, p0), ...]. The sum of probabilities must be equal to 1. This error is fairly simple to
use and by testing different combinations of errors, we found out that it simulates system errors on a
usable level.
Next is the Unitary error which can have two types. The error model can be either a Coherent unitary
error that is defined by a single Unitary matrix U or it can be a Mixed unitary error that is defined
as a list of unitary matrices and their probabilities [(U0, p0), ..]. The usability is more complicated but
has overall better precision with error simulation.
Last is an n-qubit depolarizing error channel [Vov+21] parameterized by a depolarization probability
p. The main idea of an error simulation is to whenever an error with a given probability occurs, we
replace the state of any qubit with a completely random state.
For two qubit gates, it is applied
independently to each qubit. An important note to keep in mind is that combining all errors together
does not improve the error simulation precision of the system as described in the next subsection.
3.5.4
Error Composition
All errors described above can be combined together. Each error has some advantages and by combining
two or more we can make the error simulator as precise as possible. This does not mean that adding all
errors in all combinations is the best option. All models do have some disadvantages and inaccuracies in
simulations and whenever we do combine two models we also combine their errors. There are different
options for how to combine errors that we will now describe.
• Composition: E(p) = E2(E1(p))
• Tensor product: E(p) = (E1 ⊗E2)(p)
• Expand product (reversed order tensor product): E(p) = (E2 ⊗E1)(p)
3.5.5
Two Layer System: Virtual Qubits
As already seen in the OpenQasm section 3.3 there can be two layers of qubits defined. One is a physical
layer which are the actual qubits on a physical hardware. These are imperfect and in the current state,
there is a lot of unwanted behaviour. It is impossible for them to work without any error and have a
perfectly defined quantum state. The issue is that quantum algorithms expect qubits to be perfect and
error-less. Because we cannot do this on a physical layer we define another layer that contains logical
(or also called virtual) qubits [KE01]. These qubits are strongly entangled physical qubits with some
error mitigation algorithm to make them more resistant to errors.
The downside of this is that many physical qubits map to a single logical qubit. This reduces the
number of usable qubits in an algorithm but greatly increases their precision. There are a lot of different
error mitigation algorithms using virtual qubits. As an example, we can mention the simplest which is
the Repetition codes [Gün+21] and currently the most common one is called Toric code [Sta+11]. The
first method instead of using a single qubit as an information unit it encodes the same information
to multiple qubits next to each other.
Then if we apply the same algorithm on each of them we
should get the same result from all of them. The second method is an example of topological quantum
code [Bom13]. These codes use principles that only qubits in close proximity can interact and try to
use it in combination with strong entanglement to create stable logical qubits.
The idea of virtual qubits is in the current state of quantum computers not feasible. It requires
many qubits to be entangled together and interact without error rate over some threshold. In the future,

this will need to be implemented, but currently, it is more important to focus on physical qubits and
some error correction of them. Before any virtual qubits are implemented we need to know whether
they improve an error rate or not. Therefore the performance tool can be used for this scenario. There
needs to be seamless integration with future virtual qubits and needs to be thought of in the design.
3.6
Performance Benchmarking Algorithms
In the previous section about Virtual Qubits 3.5.5 we have shown some methods that can decrease the
error rate of the system. But it is also important to be able to measure whether some method really
decreased the error rate or just added some computation time with no benefit. Therefore next we will
describe some existing methods to measure the error rate of the system.
3.6.1
Randomized vs Structured Quantum Benchmarks
In the beginning the easiest way how to test quantum computers was using unstructured randomized
benchmarks. The idea was to run some random circuits on a simulator as well as on the quantum
computer and then compare the results of the systems. This way of testing the quantum computer was
not complicated and the performance was not great enough for the simulator to not be able to output
the result. Nowadays the quantum systems are getting in the state that simulators run even on the
biggest supercomputers cannot output the correct result [Pro+22]. That is why more fitting test suites
must be developed for the quantum system to be tested.
One idea is to make the benchmarks suites more structured and easily predict the output of the
system, therefore not requiring the simulator run to get the correct result.
This can be done still
with the randomized approach or we can also define a structure of the circuit with gates in specific
positions. As always, there is no clear winner and both types of testing should be used for different use
cases. It is possible that using a randomized benchmark does not show what is the performance of the
quantum computer for some specific quantum algorithm. Some groups of algorithms will run better on
a quantum computer with a worse score than the randomized benchmark. Therefore there needs to be
a special suite of tests for each group of quantum algorithms to correctly measure the quantum system
performance.
There are two proposals for the quantum test suite structure that do not require the quantum
simulator to compute test results. One is Mirror Circuit Benchmarking [Pro+21] and the other is Cycle
Benchmarking [Erh+19]. We will introduce both approaches and compare them with each other in
the next section. Both methods can be used in the randomized or structured form. There is no strict
requirements on gates included in the circuit. Therefore we will introduce both methods and compare
what should be more precise in the error measurement.
3.6.2
Randomized Mirror Circuit Benchmarking
Mirror Circuit Benchmarking uses the idea that any quantum circuit can be reversible. We can always
create an exact copy of the original circuit and then reverse it. This will mean that the result should
be always the same as it was at the start of the algorithm. The second approach is to not reverse some
gates that are easy to compute. Then the result will be different, but the computation will be much
more simpler than without any reversed circuit.
For simplicity, we will now describe only the version where the output is the same as the input. The
idea is that first, we apply a circuit layer which puts each qubit to some well defined, but random state.
After that, there are n different layers containing each one line of Clifford gates and one line of Pauli
gates. These layers are not strictly defined and can be changed for different implementations. Then
the gates are selected randomly with a normal distribution. Pauli gates are added to each layer as a
second gate to remove coherent effects, similar to the randomized compiling procedure [WE16] which
solves this issue in different scenarios that can also be applied to our use case.
==PAGE:41==
3.6 Performance Benchmarking Algorithms
35
When this is all built we take all layers and create an inverse version of them. This means that if the
layer consists of two gates for each qubit then these gates are swapped with each other but still applied
on the same qubit. These changed layers get applied in reverse order from the original ones. Then at
the end, we take the first initial layer and inverse it as well. The length of the half of the circuit is n/2,
where n is the depth of a circuit.
As expected, the length of the circuit cannot be odd if we want to keep the result the same as the
input. One way how to solve this is using some operation on which we know the result and then expect
the input string will change only by this operation. Including this allows us to have circuits of odd
length and we still do not need to simulate the circuit. This can be for example applying the X gate
on all qubits. Then the result of the run will be inverted from the original string. The downside of
this solution is that we need to change the result state of the run. This would not be an issue if both
states have the same measure error, but from the theory described above, we know that |1⟩state has a
greater error rate than the |0⟩state.
In figure 3.6 we can see a visual representation of the generated circuit. We define a circuit with 5
qubits and 3 layers. First, all qubits are reset and then set to a random state using Rn gates. This can
be skipped to get always all zeros as an output. Next, there is an original circuit in a single color. The
structure is that for each layer there are two gates for a single qubit. The first gate is the Pi, n, where i
is iteration and n is an index of the qubit. These are always single qubit gates. The second gate is the
two-qubit gate when possible. If there is an odd number of qubits then one will have only a one-qubit
gate. It is shown as Li, n in the schema. When all layers are defined then we add an inverse version of
the circuit. This is shown in different color and contains inverse gates to the original circuit.
Figure 3.6: Visual representation of a randomized mirror circuit.
The randomized approach can use different probabilistic methods for choosing the correct gates.
The circuit gates are chosen uniformly from the Clifford group on n-qubits. This means that each gate
has the same probability of occurring in the circuit. Currently, there is no better distribution for the
gates with an added benefit for the randomized benchmark approach.
The described circuit is only one of the options of how to structure the circuit. The main idea is
that we need to be able to know the result of the circuit before execution. This is done using a mirrored
part of the circuit. Each layer can be and is defined in different research papers differently. The more
complicated layer becomes the less granular the circuit is. This means that for example if we want
to test the quantum volume of the system we need to be able to increase the circuit size by one gate.
When the layer is structured of multiple gates then the increase needs to be done by this number of
gates.
This issue can be solved by introducing a partial layer or layers that apply only a subset of gates.
The downside is that then these partial layers cannot have all functionality of the main layer. Which
has been the reason why we introduced these gates in the first place. Therefore the complexity of the
layer is kept as simple as possible.

Structured Mirror Circuit Benchmarking
In this approach instead of doing a randomized selection of gates all gates are predefined on all layers
and then we create their inverse versions for the second half of the algorithm. At the end of the run,
the output string will be still the same as an input. The difference is that there is more control over
what sequence of gates is used for each layer of the circuit.
The idea is that we can try out some combinations of gates that are commonly used in algorithms
and therefore test the specific set of errors whether they occur or not. This also has a drawback in that
we test a specific set of errors in all runs. This means that some errors that are not part of our designed
test suite are never tested.
In figure 3.7 we can see a structured mirror approach. Again, each object is a single gate. The first

Fixed gates structure allows us to deterministically test each configuration of the same quantum
computer without any randomization.
Using the same circuit for each run should see even small
improvements if there are any changes in the quantum computer. The downside is that we always test
for the same issues. In quantum computing, it also depends on the positions of gates in the circuit.
Therefore if there are two gates in the system their location can change the error rate of the system.
Some compilers are trying to mitigate this issue, but it cannot be done for all use cases.
To mitigate the issue described above we can generate different combinations for multiple runs. Each
iteration will define how the circuit generating should change and a new circuit is generated. This allows
us to test different configurations, but still it keeps the circuit deterministic. How complex the test suite
of a single size should be depends on how many combinations we generate. Each iteration should be
as different as possible. This way we can cover many more errors than just generating circuits that are
closely related.
3.6.4
Randomized Quantum Cycle Benchmarking
Alternative approach is described by Joseph Emerson and called Cycle Benchmarking. Here, instead of
mirroring the circuit we rather use an iterative system.
The idea is to split the algorithm to layers consisting of some combinations of gates. These gates
can be randomly picked or can be predefined using some specific rule. This step is the same as in the
Mirror Benchmarking. Again, we can have layers consisting of multiple gates. We can also introduce
the two gate layer approach that is the same as in structure benchmarking. This allows us to reduce
coherence errors as described before. Also using this approach we can test the communication between
the classical and quantum computers when an iterative algorithm is implemented. In figure 3.8 we can
see a visual representation of the circuit.
When layers are created we then need to generate inverse layers for each layer of the original circuit.
Using this we can again remove the need for a quantum computer simulator that would need to compute

3.6.6
Quantum Volumes
In the same way as on classical computers we need some metric to compare the performance of quantum
computers we need something similar for quantum computers. In the first stages of computer devel-
opment, the number of transistors was a metric to compare performance of the computer. Later this
needed to be replaced with some other metric to show a more realistic comparison between computers.
For quantum computers, the situation is the same. Previously, the number of qubits was a capable
metric to compare quantum systems. Nowadays, it does not show what quantum algorithm can be run
on the system and therefore it is not a realistic comparison. There is a new metric that is used instead
of the number of qubits which is a Quantum volume [tea]. This volume is a combination of a width of
the circuit that is represented as a number of qubits, the depth of the circuit which is how many steps
of the circuit are executed and errorness of the result of the algorithm.
The idea is to start with the smallest circuit possible. Then increase the width and depth both by
one and execute the circuit. Repeat this until the success rate of the system drops below some threshold.
When this happens we know the greatest cubic circuit that the system can run correctly.
The logarithmic quantum volume log2(QV ) of an n-qubit system is defined as [Cro+19]:
log2(QV ) = argmaxmmin(m, d(m))
where m ≤n is the number of qubits and d(m) is the number of qubits in the largest square circuit
for which we can reliably get the correct result with probability greater than 2/3.
We define the quantum volume as a log function because now we can see how many qubits and gates
are introduced in each circuit. Also, to use non-log values we will need to use a really high number.
As an example, if some processor has a quantum volume of 222 > 4000000 then our log2(QV ) value is
simply 22. Except for seeing the grow of performance of the system there is no specific reason to use
these great numbers.
The proposed Quantum Volume metric uses only two qubit gates when generating a circuit. When
there is an odd width of the circuit then there is no gate applied on the last qubit. This metric is useful
for systems with a great number of qubits which will be the case in the future. Currently, the technique
would be not sensitive enough to measure small improvements on quantum computers. Therefore the
tool also implements a solution where only single qubit gates are used for circuit generating. Then both
algorithms should be run on quantum computers to compare their quantum volume until the second
algorithm will show no benefits to the measurements.
3.6.7
Visualize Quantum Volumes
Instead of using a single value we can also visually represent the results of the system. The quantum
volume algorithm works in a way that we first start with the smallest circuit and then we increase it
one in width and in depth. We can also increase only in a single dimension and cover a wider field of
results. Each circuit size should be run more than once to remove any performance exceptions. Then
we can graphically visualize what circuit sizes the quantum computer is capable to run with an error
probability smaller than some threshold.
In figure 3.10 we can see a result of a Structured Mirror Circuit run on a quantum simulator. Each
width and depth have been run for 1024 shots and increased in each direction until all combinations
were run. Then each result is compared with a correct result and the most common result is shown.
Any value above 2/3 is counted as a correct one. Anything below is incorrect.
We can see that increasing circuit depth or width decreases the precision of the system. When an
error is too high there is a possibility that the system will output the wrong solution with a higher
count than 2/3. This is called a false positive and it is the main reason why we need to check the result
and do not expect that the most common output is the correct result.

3.6.8
Quantum Performance - CLOPS
In the previous section about Quantum Volumes 3.6.6 we describe how quantum performance can be
measured. This does not contain measurements of how fast the circuit can be executed, but rather
what is the biggest size of a circuit that can be correctly executed. Here we will describe how we can
measure the quantum performance with respect to the speed of execution.
We can as always compare the new system for quantum measurement with the classical one. To
measure the performance of the supercomputer we use the FLOPS metric (floating point operations
per second). We can do something similar for the quantum system. Here we can see how this metric is
then computed.
M × K × S × D
time_taken
where
M = number of templates
K = number of parameter updates
S = number of shots
D = number of quantum volume layers = log2 QV
Quantum volumes can be used as a base performance metric instead of floating points, but the idea
is still the same. The metric is called CLOPS (Circuit Layer Operations Per Second) [Wac+21] and
defines how many layers of the circuit can be executed per second. The idea is to have a parameterized
circuit that is executed on a quantum computer multiple times and then measure the time of execution.
In the current state quantum computers needs to interact with classical computers throughout the
computation and sometimes this communication is the bottleneck of the performance of the system.
This is why the metric measures also how long it takes to load the circuit on a quantum computer
and also how long it takes to read the result. What this metric does not measure is the quality of
the quantum circuit the system is capable of running correctly. For this, there is a Quantum Volumes
metric. Also, it does not measure the scale of the quantum computer with respect to the number of
qubits. This is another metric which just simply counts the number of working qubits.

Application Design
In this chapter we will describe how all of the principles and problems described in previous chapters
can be used to develop a generic tool that can measure quantum computer performance and can be
used as an input to some algorithms to do further optimizations such as automatic error mitigation
and more. The tool is implemented as customizable as possible without any extra diﬀiculty of use.
This means that each part has multiple implementations which can be chosen to fit the best to wanted
output. The whole tool is implemented in Rust, but some dependencies require Python to be used.
Next, we will describe the tool structure and how each part of it solves some issues described in the
previous chapters. The main focus was to create a foundation on which all parts can be extended and
further improved for better quantum computer performance benchmarking.
4.0.1
Quantum Provider Specific SDKs
The tool does not try to re-implement already existing solutions for communication with some specific
quantum computer providers. Working on that would be unnecessarily complicated and there will be
no added value to the solution. As mentioned before, the tool tries to make the whole quantum circuit
development independent of a specific provider. This means that it tries to create a custom abstraction
above the provider specific interfaces. For IBMQ there is a Qiskit SDK which is used when this provider
is specified. Each provider uses a different SDK, but the tool uses an intermediate format to be provider
independent. Most of the providers use Python as a programming language. Therefore there needs to
be a Python binding for the provider implementation.
There is implemented an OpenQasm and Qiskit language support, but any other can be easily added.
For providers, there is an IBMQ support and statevector simulator with or without a noise model. The
tool uses a Python interface for provider communication, but native extensions can be implemented.
As an output, there are two options. One is to have a text based output that is easier for users to read.
This format uses native terminal operations to draw a table with results. Therefore it is not possible
to have this format as an output value when used as a library. Another format is using a serialized
output. This is suited more for machine parsing after the run. This type does not need to use standard
output, but can directly forward value in memory. More information about each extension is in the
next chapters.

4.1.1
Comparing Performance of Quantum Computers
The main use case that is supported by the tool is to measure a quantum computer performance. It
is important to have some metrics to compare how each computer performs with respect to another.
There are already existing algorithms that try to create a metric for quantum computer performance.
This is not the performance with respect to the speed of execution, but rather the error rate of the
system.
The measurement can be done using Quantum volumes. These volumes are a single value that
shows the biggest circuit that can run on a quantum computer with respect to the number of qubits
and the number of gates. The tool implements two versions of Quantum Volumes. One uses only two
qubit gates (Volume2) and the other uses only single qubit gates (Volume). The first approach follows
the algorithm described in the research paper. But throughout the measurements, we have noticed
that these circuits can be too complicated for a quantum computer to output any interesting value.
Therefore there is also a second approach implemented. This outputs a different value that is not the
correct quantum volume but gives some insight for the system performance. The recommended type
of measurement is to run first the Volume2 and then Volume algorithms to get the best performance
measurement for Quantum volume possible. This approach can find the greatest square circuit. But
we do not have any information about the performance of irregular circuits.
To measure the performance of non squared circuits we need to introduce a different approach. The
idea is to create a lattice of all combinations of depths and widths of different circuits. Width is defined
as a number of qubits and depth as the number of gates for a single qubit. This type of measurement can
use two different algorithms with two configurations. So there are four options in total. The algorithms
are either Mirror or Cycle benchmarks. Each is described more in the previous Performance benchmark
section 3.6 with all the advantages and disadvantages of each approach. The tool allows to choose any
of these algorithms as a quantum circuit generator and then execute using the lattice orchestrator.
The measurement does not need to be done only with two different quantum computers. Currently,
there are big differences in the error rate of a single computer at different times from calibration. As an
example, the tool could be run periodically at some specific times to measure how the error rate changes
on a single system. Another use case is when there is some error correction mechanism deployed to
measure whether the error rate has been improved or not. Also, it could be used when one computer
should be replaced with a new one to measure how much the performance will improve. There are many
more situations in which this measurement could add information and be useful.
4.1.2
Measurements Before the Circuit Execution
When some algorithm is executed it is important to know what is the error rate of the system. If the
error rate is too high then we can get false positive results. Generally, this changes quite often and we
can never use an old measurement as an expected error rate. For this use case, there are implemented
performance testing algorithms described above. The execution should be as fast as possible and should
contain only circuits that do add some value to the expected error rate. Users can define a set of circuits
that will be executed using some specific algorithms and then use results to compute a system error
rate. This can be later used to see whether the correct value should be expected.
The first approach gives us a generic error rate, but it has been shown that different algorithms will
have different error rates when run. Therefore we can run a set of circuits that can be more suited
for some specific algorithm we want to execute. The idea is to run a small set of circuits consisting
of critical parts of a bigger circuit. This set will then show whether each part can be executed on a
quantum computer. The important note is that this does not predict whether the circuit run will be
correct or not. It shows if some parts should be optimized or implemented differently for higher change
of a correct run.
The circuits can be either generated using the Base template or the FileSystem circuit generators.
First takes some bigger circuit as an input and then generates smaller out of it. The second iterates
over predefined circuits and executes all as a set. These should be a great tool to run a simple set of
circuits to find out if the algorithm will be successful.

4.1.3
Multi-platform Circuit Implementation
Last use case that the tool focus on is the possibility to execute a quantum circuit on any quantum
provider independent of in what programming language the circuit is described. Ideally, we want to
create the circuit only once and be able to run it on any provider that is supported. This is possible
using the developed tool. The idea is to use an intermediate format that is transformed into the provider
specific language and executed on the machine. The tool is capable to transform some already known
formats into an intermediate format.
The most supported is a subset of the OpenQasm language. Currently, there cannot be any classical
computation operations. This feature is not supported by all providers. For the intermediate format to
work it needs to contain only a subset of all features of all providers. This could be restrictive in some
scenarios, but generally, the feature set is enough for the performance benchmarking.
When the circuit is implemented in the supported format it can be run using the Single orchestrator
and FileSystem circuit generator. This will execute only the defined circuit. The tool is capable of
creating a Mirror or Cycle Benchmark for a defined circuit. Or execute the circuit as is. This way
we are able to create a provider agnostic implementation of an algorithm that can do performance
benchmarking or any other operation if needed.
4.2
Internal Components Structure
The tool is mostly built by extensions which are highly configurable by a user. As described in previous
chapters there are some communication channels with the quantum computer. These are split into
sub-parts and for each, there is defined an interface. This should be a common interface for all QC
providers. More about this is in the Extensions section 4.2.3. Then there is a main core of the tool
that implements inter-communication between extensions. This is just a bare minimum that is needed
for the extensions to work. Most of the tool functionality is done via extensions so it is really simple to
change its behaviour. This main core property allows to be used in multiple ways which are described
in the next section.
4.2.1
Standalone Execution Model
The tool has two execution models. One is as a standalone application. This way there is no need for
any programming skill from a user. When executed the application will perform a predefined algorithm
and output the result. But there will always be requirements from the user to customize the behaviour.
Therefore there is an option to configure the application using the configuration file. The configuration
can be done in multiple ways. All are described in separate Configuration section 4.3.3.
The benefits of this model are that it is really simple for a user to start working with the tool. There
are no dependencies required for the application to work correctly and no configuration is disallowed
when run in this mode.
4.2.2
Library Execution Model
Another execution model is as a Python library. The whole tool can be wrapped as a Pip package and
distributed to any project as a dependency. The tool uses wheel approach to distribute the binary file
as a package. It is supported out of the box by Pip package manager and there is no need for user
configuration. The tool then defines a constructor with all configuration options that in the first model
were defined in a configuration file. There is no restriction with options in respect to the first model.
The benefit of this approach is that the tool can behave as a core library for quantum computer
communication and can create an abstraction from QC providers. This way single solution can work
on any supported provider without the need for a modification. This approach is preferred and should
be used when more complicated applications are developed.

The method is independent from the constructor because generally, these two steps should be inde-
pendent of each other. In other words, object construction should not fail if there is no memory error.
One option is to use a builder pattern that can check valid arguments. Another option is to define a
check method that needs to be called before any use of the object. The second approach makes imple-

Below we describe what configurations are restricted and why it is the reason.
IBMQ
IBM provider requires the account ID and machine name to be defined. The first is an ID assigned by
the IBMQ account. It is a single string which can connect to the IBMQ account and interact with the
public API. It needs to be able to access the IBMQ providers to execute jobs on the quantum computers.
The second defines the name of a quantum computer. A single account can access multiple computers
and each can be used for circuit execution. The name of a quantum computer is a unique string.
Quantum Volume
The Volume orchestrator needs Volume circuit generator to work properly. Any other combination is
not reasonable for the orchestrator. The main task of it is to compute the Quantum Volume of the
system and this can be done only by some specific circuit generator.
FileSystem
The FileSystem circuit generator needs Linear or Single orchestrators. The filesystem of a system is
linear and there is no reason for defining another dimension. Therefore it only has the linearized output
or it can output only a single circuit from the directory.
Another constraint is that the FS generator supports only the OpenQasm schema language. This is
the open source format that is also type checked and therefore should be preferred to be used.
Custom Python Extension
Each Python extension can define custom constraints if needed. This step can be skipped and not
implemented which means that no constraints are required. As input, there is the tool configuration
and as an output it can be either an Ok value or an Error with a reason what constraint was violated.
The tool configuration cannot be changed in this step.

Noisy simulator uses a combination of error simulation models to make the behaviour closer to the
current quantum computers [GEZ21]. These models try to simulate the quantum errors as close as
possible without introducing errors that do not occur in quantum computers. The most important
models are bit and phase flip error models. These models do random value flips with some probability
after every operation on a single qubit. These are the single qubit errors. We can see the output of this
simulator in figure 4.5.
Then there are used some multi qubit errors. Mainly, one is called the thermal relaxation model and
the second is depolarizing error channel. Thermal relaxation is an irreversible process that describes the
thermalization of the qubit spins towards an equilibrium state at the temperature of their environment.
This process involves the exchange of energy between the qubits and their environment, which drives
the qubits either towards the ground state or the excited state. The second model with some probability
will flip any qubit state to random a state. More about all methods are in the Error Models chapter 3.5.
When the circuit is tested and outputs are in a form how expected then a noisy simulator can be
used to see what error rate can be expected from a quantum computer. This model allows to test some
error rates without executing code on a quantum computer and reducing the need to use real hardware.

4.6.2
Cycle Benchmark Method
Cycle benchmark technique is an alternative approach to the Mirror benchmark technique. Both use the
same basic principles from section 3.6 which is that all circuits are reversible and on an ideal machine,
the output will be the same as the input. For each gate, we can introduce to the circuit a reverse gate
which will negate the effect of the original one. The main difference between these two approaches is
that the cycle technique splits the circuit by layers and then for each layer it generates and inserts the
inverse layer right after the original one.
The advantage of this approach is that there is an option for a QC Provider to remove some types
of errors. For example, the coherence error when each pair of layers is used as s independent unit
which does not require other layers to exist. Generally, both approaches do have some advantages and
disadvantages. Each approach can make some of the errors more dominant, but there is still no clear
idea which of these two techniques is better. Therefore the tool implements both and compares the
performance of them.

5.1
Provided Quantum Error Model
For better user experience the tool implements an error model. This model is for development purposes
only. It does not mirror any existing quantum computer error rate. The idea is that the circuit can be
developed by a user using the noise model and after it is well tested they can run the circuit on a real
quantum computer. This way the usage of real hardware is as minimal as possible.
The error model implements the most common errors that can happen on a quantum computer.
For each stage, there is a Pauli error implemented to simulate an error that can happen with some
probability. For the model to be simple and easy to use there are no cross communication or stagnating
errors. The model is more optimistic about correct results that are measured on real hardware. It is
expected for quantum computers to increase performance in the short future, therefore it is better for
a simulator to overestimate the performance that underestimates it.

5.3
Performance Optimization Techniques
Implementation techniques that have been used were focused on the tool execution speed and config-
urability. The tool will be used before any algorithm execution and therefore circuit generating should
be as fast as possible. There could be used some other techniques to make the tool even faster, but
then we would lose the possibility to easily configure it. Therefore some type of dynamic dispatch has
been used. This allows to simply swap some parts, called extensions, with alternative implementations
without any diﬀiculties. It is also important to correctly predict the execution time. For this scenario,
no dynamic exceptions were used. If needed any execution path can be computed and predicted before
the run.
5.3.1
Enum Dispatch
The tool should be easily extendable and therefore some dynamic dispatch needs to be used for parts of
it. The idea is that all parts have some public interface which can be implemented and then dynamically
used. Normally this would be done using virtual methods which are mostly implemented using virtual
jump tables. This approach slows down the execution when there are accesses to implementations in
every iteration which in our case is true. Therefore we decided to use an optimization method called
Enum Dispatch [Laz]. It is a common practise in Rust programming language and support is directly
in the language.
The idea is to use a procedural macro that implements a trait for a fixed set of types in the form of
an enum. This is faster than using dynamic dispatch because type information will be built-in to each
enum, avoiding a costly virtual table lookup. First, we define what interface (or trait in rust) needs
to be implemented by a module. Then we register this module to the enum variant that contains all
implementations. Necessary code is generated automatically using procedural macro. This code does
not contain any virtual table and all information is directly next to the data.
If we compare the performance of dynamic dispatch we can see that with no special optimizations
the speed up can be up to 10x. This is because there is no pointer indirection and all data are tightly
together. The speed up has been tested on a test in listing 5.1.
First, we create a vector of 1024 structs of arbitrary type at random. For each test scenario, all of
these structs implement the tested dispatch method. Now we iterate over the vector and call dispatched
method on each element. We measure time for the whole iteration with no vector construction and
destruction. The return value of the method call is obfuscated so there is no compiler optimization
possible.

We can see that the Enum Dispatch method is much faster than any other dynamic dispatch method.
The main reason for it is that objects in a vector are enums with data rather than addresses which means
that accessing the element takes half of the indirection as the other techniques. Another optimization
is that no virtual table is needed which removed another indirection in some languages.
We can have similar optimization applied automatically using the GCC compiler. It is the AOT
speculative devirtualization optimization. The downside is that it can be applied only in some scenarios
and we cannot expect this to be always applied. Therefore even when some codebases can be optimized
to the same performance level, the worst case scenario is always worse than in our case.

5.3.2
Result Error Type (Static Exceptions)
Using dynamic exceptions can create a big difference between expected execution time and measured
when some error occurs.
Therefore the tool uses a static exceptions model supported by the Rust
programming language to keep the execution time more stable. Another advantage is that this type
of exception is also used on extensions interface. This way we can use languages that do not support
exceptions and connect it with the rest of the tool. The interface is C-like specifically to allow simple
integration with other languages.
The type for expressing exception is called Result or Option as can be seen in listing 5.3. It is a
template type which can take an Ok value and the first type can also define an Error value. Each

5.8
Application Test Suites
Each application needs to have a complex set of tests in order to be stable and in bigger projects to
not introduce any functionality errors. There is a wide range of test types from which each is better
for different scenarios. As an example, the web based application needs to correctly communicate with
a server and it is a critical part which means that API testing needs to be done, but for desktop
application, there is no server communication and therefore no API testing is needed.
Another issue is to show that test suites do cover the whole application and that there is no applica-
tion part that is not tested. Therefore code coverage is also implemented with the test suites described
below. We will focus on a basic suite of test types to cover the most common errors in applications.
There are two types. One is a low level test that tries to isolate functions as much as possible. These
are called unit tests. Another type of test is the whole application testing. This uses a public API to
communicate with the application. We will describe both principles in the next sections.
5.8.1
Unit Testing
First, we will describe the base testing system that is often used in the agile development process.
It is called unit testing [Ola03]. This concept tries to isolate each piece of the program as much as
possible. The idea is to expect that each method can be wrongly implemented and for example, some
computations can be incorrect. This type of testing is important for critical parts of the application.
For example, if there are some computation models.
The downside of it is that we will never test the application as a whole. This means that even when
all parts work correctly, together it can misbehave. Therefore we will later introduce another testing
process that does not cover each small piece, but rather the communication between them.
Basic unit test support is embedded in the Rust language which for our use case is usable because
there is no need for any complicated mock framework. The unit tests are written in the same file as
the implementation and they have access to private methods of the file.
==PAGE:75==
5.8 Application Test Suites
69
5.8.2
Application Testing
When all parts of the application are tested that all is working correctly. Now we need to find out
whether the whole application works correctly. To do this we will introduce application testing. This
can be often interchanged with integration testing. The idea of both types of testing is almost the same.
It is to test all parts of the application working together. The difference is that integration testing
takes a subset of components and tests them together. Application testing tries all components of the
application together. We have chosen to use application testing because there is not a great number of
components that need to be tested separately. It would also add a lot of mock testing with no benefit.
The tests are written in a separate folder from the source files. There is no need to have access to
the private methods, therefore any visibility modulators are not needed. It is important to first have
the component tested to whether the output is correct or not. If this would not be done, then it would
be hard to detect where the error happened and fixing would be complicated.
5.8.3
Code Coverage
As mentioned above doing any type of testing does not guarantee that the application will run correctly.
If we do not test some part of the application that doing any type of testing is no use. Therefore we need
to somehow know that the application as a whole has been tested. Because of this, we will introduce
the code coverage [TH02]. It is a principle which helps us to know what parts of the application have
been tested.
The basic idea is to build the application with special debug information that can map executed
code back to the source files. In Rust language supported is LLVM IR or Rust MIR backend which
maps back to the source files. Then additional instrumentation is added to count how many times each
line or logical expression has been executed. This way we know what parts of the application have been
executed.
The newly added type of code coverage is the source based coverage. The difference is that the
instrumentation set is added by the compiler itself as soon in the compilation as possible. That means
some constructs do have better coverage.
For example, short-circuited conditionals, closures, and
match guards are all precisely counted. And since instrumentation counters are injected as regular MIR
statements, the compiler can further optimize the program without affecting coverage results.
To test the application correctly we use grcov library developed by Mozilla. It uses the prof data
generated by executing the library and generates graphical output that shows statistics about the source
code as can be seen in figure 5.2.

These results correspond to the complexity of each algorithm and the trend of each runtime speed of
the algorithm is as expected. There is still a possibility to improve the algorithm implementation using
parallel computations or optimizing some computation parts, Overall, there will always be a steadily
increasing complexity with a higher number of qubits or gates, but the tool is usable for circuits of
smaller sizes for current quantum computers and for much bigger circuits used by quantum computers
in a future.

CHAPTER 7
Future Work
The tool currently should be in a state that it can help to solve some of the issues with performance
testing, such as Quantum Volumes or Base template Mirror Benchmarking but there are still a lot of
areas that could be improved. Generally, the tool tries to create a foundation on which other solutions
should be built on. Using the provider agnostic tools it should improve interoperability and reduce
repeated work that currently needs to be done for each algorithm. Below we will shortly describe a few
examples of how the tool could be extended to support some additional or improve the already existing
use cases.
7.1
Extend Supported Subset of Circuit Operations
The tool uses a subset of a circuit description from other providers to be able to work on any supported
quantum computer. This subset has enough options for performance benchmarking, but there is still
a possibility to extend this set with another useful options.
Here we will introduce some optional
extensions for the circuit definition, but any added functionality can be also included in the tool directly
when it is supported by all providers.
Support for Classical-Quantum Systems Interaction: The current set of circuit operations focuses
only on quantum state and how to work with qubits. This is helpful for simple use cases, but with a
hybrid quantum computer, there should be also an option to define classical operations when supported.
OpenQasm format does allow to combine these two types of operations together and the tool uses this
language for circuit description. Therefore it should be straightforward to add support for it.
Allow to Define Custom Registers in the Language Schema: The current intermediate format support
only to define the size of quantum and classical register, but the name of registers and their count cannot
be changed. The performance benchmark algorithms do not require these options, but for future use,
they could improve usability. The idea is instead of applying all gates to a single register to allow a
user defined custom registers and then use those names for gates definitions. This could allow to define
some more complicated algorithms which require more than a single quantum register.
OpenQasm 3.0 Support: Additionally, the tool uses currently supported version 2.0 of an OpenQasm
language. There is already a proposal for an improved language version OpenQasm 3.0. This language
supports an extended set of classical operations and improves the syntax for some operations. When
supported by other providers this should become a standard circuit description used by a tool.
7.2
Improvement of Communication Interfaces
Next there is a list of small but helpful extensions that could be done to improve the general user
experience and usability of the tool. Even though the tool should be feature full some implementations
are sub-optimal or hard to use. Instead of adding new features, we would prefer to look at already
existing parts and show how they could be improved.
Extend Application and Extensions API: Another useful improvement could be to add other pub-
lic interfaces binding than the already existing ones. Currently, there is a Python interface for the
application with extensions, the configuration file for the application and a Rust interface for native
extensions implementation. This could be extended with the C API for low level communication with

7.3 Implementation of Additional Extensions
80
other performance focus languages. But more interesting would be to implement a GUI with configu-
ration options for a standalone execution model. This would more reduce the barrier for a new user to
start using the tool. When needed they could still use a different interface if the UI will not support
some functionality.
Custom Arguments for Extensions: The last improvement could be to allow an extension to define
custom arguments. Now only predefined arguments can be used by an extension, but if allowed there
is a great potential to improve the tool’s usability. Currently, each extension can define restrictions to
disallow some combinations of arguments. As shown in section 4.3.4 these restrictions can be really
useful. Therefore also allowing to extend the configuration should also show some benefits.
Improve Time Estimates for Collected Jobs: As a first example, the time measurements are done
only when the orchestrator is not set to collect all circuits before execution. The tool measures how long
it takes to create a circuit and then execute it on real hardware. Currently, there is no way of doing this
measurement when all circuits are run in a single job. Therefore, there needs to be a provider access to
the time measurements done for each circuit and then the tool needs to access these data. This needs
to be implemented by the provider and then accessed by the tool
7.3
Implementation of Additional Extensions
Performance benchmark algorithms are already part of the tool, but there are many more which can
be implemented and the algorithms do not need to be only about performance measurement, such as
speed testing using the CLOPS measurements. Next, we will introduce some additional extensions that
can be implemented to improve the usability of the tool. This is not limited to shown algorithms, there
are many more that can be implemented. This should be used only for demonstration purposes.
CLOPS Measurement: The first and most interesting extension that could be implemented and does
belong to the performance measurement group is the CLOPS measurement algorithm. The idea is to
also measure the speed of execution of an algorithm instead of only the error rate. In the NISQ era this
measurement is still not so important and therefore is not part of the tool, but when the error rate is low
enough for quantum computers to run for a longer period of time then some type of speed measurement
will be required. Some providers, such as Qiskit, already implement this algorithm, but as is it with
all other implementations these are provider specific algorithms. Using this tool in combination with a
generic implementation will allow running the same algorithm on any supported provider.
Non-Python Simulator (cuQuantum): The next improvement that could be done is a better quantum
simulator. Currently, the tool requires a Python interpreter to be used when some simulation is executed.
This can be for example when no benchmark model is used and then the circuit must be simulated to
get a correct result. As an optimal implementation, the cuQuantum could be used. It uses a computer
graphics card for simulations and is implemented in C++ programming language therefore should have
higher performance. This language does not require an interpreter and can be directly statically or
dynamically linked to the tool. Even when this library would be preferred there are still many other
alternatives, such as Intel-QS, Cirq or Bluecat mentioned in section 3.4, that could be used to improve
the simulation speed.
Add Virtual Qubits Support: Error correction does not need to be a software solution only, but as
shown in section 3.5.5 there are already hardware error correction algorithms. The tool needs to work
with them to not become obsolete once it is used on quantum computers. The tool design expects to
work with virtual qubits instead of physical. But currently, there is not an option to choose which layer
it should use for quantum gates operations. The OpenQasm format allows to distinguish between the
physical and virtual qubits and this functionality should also be supported by the tool.

CHAPTER 8
Conclusion
In this thesis we have focused to create a provider agnostic quantum performance benchmarking tool
and find out what would be the user experience of working with it. The quantum provider’s SDKs do
contain a subset of functionality that can be used for this purpose. Therefore we propose an intermediate
format which can be translated to provider specific code and then executed. The format contains all
necessary functionality to describe a circuit with all common stages. With some restrictions, we were
able to design a tool that can be used on any supported quantum provider. The restrictions are a
provider specific functionality which cannot be implemented in this format. But generally, this should
not limit the tool functionality.
The tool has been implemented with state of the art benchmark methods which are the Mirror
Circuit and Cycle Benchmark in a way that it is simple to extend with additional functionality when
needed.
The extensions can be implemented either as part of the tool using the Rust language or
as a Python script file. There are two execution models available for different use cases. One is as a
standalone application and the second is a Python library. Both contain the same configuration options,
but each should prefer a different output format. The application should prefer the user focus text based
output formats such as Text or Colorized and library should use the serialized format for ease to parse
the output.
We have shown different use case scenarios for the tool where the implemented functionality im-
proves the user experience. The tool is capable of measuring Quantum Volumes of supported quantum
computers. Any other provider can be easily added using the provider extension. User can execute the
performance benchmark algorithms to measure the error rate of the system and if needed implement a
custom algorithm and then execute on any supported provider. We have found that the stability of the
quantum system is not on a level of classical computers where the performance measurement can be
done only once and is correct at any time point. The measurements show big differences in error rate
over time. The tool can be used to improve the error correction algorithms to measure improvements
when some new method is applied.
The important focus for the tool was to not be limited to the current state of quantum computers
but to be future proof when simulators will not have enough performance to get correct results of the
algorithms due to the much higher number of qubits in the system. The tool is capable enough to
generate circuits with thousands of qubits and gates without any speed limitations. The implemented
performance benchmark methods do not require a simulation run to get the correct results.
Overall the tool can help with error correction and remove the need to create a provider specific
implementations. Any additional functionality that is needed in the future can be easily implemented
using the extensions system, but all the latest performance benchmark methods are available out of the
box.
