Abstract  
This full stack project is made in collaboration with the ReHyb project, which is in the process of 
developing a fully functional rehabilitation system for stroke patients. The system is made for three 
different user groups such as patients, caregivers, and therapists. Thereby each user group has its 
dashboard, which provides distinguishable different services and functions to the users of the ReHyb 
system. The role assigned on this project is as a full stack developer and will thereby have the 
objectives of developing a new backend server, that will have API endpoints that will handle data 
from the client to the database and vice versa. Users will be able to read, create, update, and delete 
data in brackets of their rights on the dashboards, thereby providing them with a seamlessly 
functioning system, and not just standalone prototypes. To determine the bottlenecks, challenges, 
and whether the backend server was production-ready. Procedures were taken to test the 
performance, concurrency, scalability, and security of the backend server. Based on the results it was 
discovered that the backend server, was not ready yet, for a bigger audience, as it could only handle 
a minority number of concurrent clients. Additionally, the systems security needs to be further 
improved, to be able to live up to GDPR rules and handle sensitive personal data accordingly, since 
the data is easily attainable by malicious individuals.

1. Introduction 
A Stroke is a challenging health complication occurring in multiple age groups, which causes long 
term harm to the body that leads to losing motoric skills in the limbs and nerves. The second leading 
cause of death and cause of disability worldwide is stroke. It was estimated in 2010 that around 17 
million stroke incidents have occurred, and approximately 30 million worldwide have survived stroke 
attacks. Unfortunately having so many patients with this disability induce high usage of the health 
sectors resources, personal and time. In the US alone the direct cost was estimated to be around at 
22.8 billion dollars, and it is expected to increase to 95.6 billion dollars in 2030. Although stroke is a 
dangerous disease to contract, it is possible to take early precautions, such as multidisciplinary 
rehabilitations, which will help prevent further complications to the brain and body, regardless of the 
severity and placement of the stroke induced part of your body12 in efficient terms that speeds up 
the recovery process, reduces the resource usages and cost of recovery for both the hospitals and 
the patients.  
With today's technological advancements made in multiple sectors, including the health sector, 
solutions with current hardware and software products such as virtual reality, human digital twins, 
augmented reality, machine learning and artificial intelligence can be used to enhance, assist, 
discover, simplify and make the rehabilitation process more entertaining for the whole health sector. 
All of this can be done over an internet connection from the comfort of the stroke patients home 
environment. Helping in reducing the health sectors resources by allowing the patients to train from 
anywhere and still have access to the provided assistance from the therapist assigned to the stroke 
patient. Thereby providing the users of the system a seamless, simple and reliable system, which will 
help with reducing the complexity of the rehabilitation procedures. Additionally having a similar 
rehabilitation system as mentioned could potentially reduce the strains on the therapists, since the 
rehabilitation progress of the patients is being uploaded online to able to track the progress, 
communicate with the patient, plan exercises and much more. 
The ReHyb rehabilitation project is one of those technological advancements towards simplification 
of the rehabilitation procedure for stroke patients. ReHyb is effectively working on developing an 
online solution for three user groups which are stroke patients, caregivers, and therapists. With each 
user group having their own distinctive custom dashboard, that serves different services and 
functionalities depending on which group the user is from.

2.1.1 Frontend development: 
Frontend development consists of developing the visible components of the graphical user interface 
(GUI) such as webpages, apps, games and etc. A shared characteristic of these elements is their 
interactive capabilities. User actions, including clicks, scrolls, text input, and other interactions, can 
trigger various responses throughout the application3. The people that handle the design of these 
GUI’s are called frontend developers. Frontend developers use for web development primarily 
HyperText Markup Language (HTML) and cascading style sheets (CSS). HTML is being used to 
develop the skeleton of the webpage, whereas CSS is used to design the webpages themselves. 
Additionally, to add functionality to webpages programming languages such as JavaScript (JS) and 
Typescript (TS) are being applied, to determine what will happen when a button is pressed on a 
webpage.  
 
2.1.2 Backend development  
While it is feasible to develop an application solely through frontend development, resulting in a 
static application, it is commonly practiced incorporating a backend service. This backend serves as 
the communication hub for the frontend, transforming the application into a dynamic entity. In a 
dynamic application, the data and visual components undergoes alterations based on responses 
received from the backend4. The backend serves multiple purposes behind the scenes which the 
customers/clients of the webpages don’t see. When a user performs an action, such as clicking "buy" 
on a webshop, a request is sent to the backend. The backend processes the request, querying the 
database to ensure the availability of requested items. Concurrently, the server verifies the financial 
balance of the user. If the conditions for purchasing the items are met, the purchase proceeds to 
fulfilment and is later shipped to the customer. Additionally, the backend can be implemented with 
multiple other programming languages such as Java, Python, Node.js, C and etc. These are a just a

2.1.3 Middleware 
In API development, middleware can be described as the software that acts as the mediator between 
the applications. In web API’s, middleware is the layer that stands in between the incoming HTTP 
requests and the application’s final response to the client. The middleware will be standing between 
these two ends to handle the request, modify, generate HTTP responses, and thereby send it to 
other middleware or applications. Some of the use case scenarios that middleware is being applied 
in are5: 
• 
Authentication – by implementing the authentication with the help of for instance JWT 
tokens and cookies. 
• 
Authorization – controlling and giving access to the users of the system based on their user 
type. 
• 
Routing – gets the data found at the specified existing URI.

A full stack system is what anticipated as a whole computer system. An online computer system 
must have these two components if it must be considered a full stack system: a frontend and a

2.2.1 React 
React6 is famous and widely used JavaScript frontend library that is used to build modern 
webpages/web applications, which is founded by Facebook and have been used to develop multiple 
big webpages such as Instagram, Facebook, Netflix, and other known content delivery webpages. 
React lets the developers easily create UI’s (user interfaces) out of pieces called components, which is 
a way to build and containerize webpage modules logic and design together with. This could be 
anything from a button or even an entire page, and the components are reusable in any place seen 
fit. 
React gives the opportunity to either create your own custom components for the webpage or use 
multiple other prebuild components from other developer libraries on your own webpage to save 
development time. It is important to mention that it isn’t possible to build fully working webpages 
using only react, since React alone doesn’t provide or use any logic for routing, data fetching/sending 
and generating html elements7. Additional packages are needed to be installed to fulfil the missing 
functionalities for the webpage development.  
2.2.2 Next.js 
Next.js8 is a React based framework, which is used to build full stack web applications. To build the 
components of the web applications, React components will be applied and for the additional 
features such as routing, data fetching/sending, generating HTML and much more will be needed 
from a framework to build a fully functional web application.

2.2.3 Vite.js 
Vite.js11 is another frontend framework which is made for creating web applications fast and solves 
some common problems regarding the developer experiences of frontend frameworks/tools. Vite 
can be used to setup the development environment for libraries such as Vue and React and are well 
supported in the TS and JS stacks. 
Two of the main features that vite.js provides is the development server and bundling of the code. 
The development server provides support for HMR (Hot module replacement) which is being used to 
update modules under the execution of the web application. When changes are being made to the 
source code on the project, only the places of the affected code changes will be updated, meanwhile 
the other parts remain the same, so it doesn’t have the necessity to update the whole page each 
time a change is made. This can help with development speed and decrease development time12. 
Additionally, Vite.js has implemented an innovative approach to quickly compile and execute 
projects, independent of the size of the project. This is achieved by splitting up dependencies 
andinto modules. It is known that dependencies usually don’t get updated as often, so Vite pre-
bundles the dependencies to improve subsequent page load times. Meanwhile, the source code only 
gets updated on demand requests13.

2.3.4 JWT token 
JSON Web Token (JWT) is an open standard which allows to transmit information securely between 
two parties as JSON objects. The information received, can be verified and trusted, since it digitally 
signed. There is two ways you can sign your information based on the official information from JWT, 
which is either using a HMAC algorithm as a secret or using a public/private key pair which uses RSA 
or ECDSA as the encryption protocol25. 
According to RFC 751926 and JWT27, JWT’s can be encrypted to also provide secrecy between parties 
with signed tokens. If you are for instance unsure of the integrity of the information received by the 
response from the other party, signed tokens can here be used to check and verify the authenticity 
and integrity of the information.  
Another way to ensure the information is signed by the correct party, is by signing the information 
through public/private key pairs. So, by signing the information with the private key and sending it to 
the other party, you can here ensure and verify that you are the one sending the information and no 
one else. Because there can only be one person that has access to the same private key as you have, 
in a private/public key encryption28. This is why JWT’s can also be good for information exchange 
between the parties of the system.  
JWT is a quite popular tool to use when users must be authorized once logged into the system. When 
logging in, the users login will be tied with a token. The token is useful, because it can store 
information on it, which can be used to authorize the user on the specific web application with their 
username and password credentials for a certain time span (which is also called SSO/single-sign-on). 
Thereby It is possible to now access all the resources and services the web application provides, 
inside the frames of what access rights delegated. If a user tries to access the web application

2.3.5 MongoDB 
MongoDB is an online document-based database, which provides a scalable and flexible schema 
database solutions for development projects of both big and small sizes29. MongoDB is following a 
NoSQL structure, so it allows the data to be stored outside the usual strict structures of an SQL 
database. Being a NoSQL database also provides the opportunity to store both structured and 
unstructured data, and it won’t cause any errors regarding having both the data structures at the 
same time in the same collection, compared to a traditional SQL database, which won’t simply 
accept any other structure than what is defined to store in it.  
 
The way the data is being stored in the document inside MongoDB is like JSON, which makes it easy 
to map objects and their values into the database, since normalization of the data is not a concern 
for the developer. Another good thing to mention about MongoDB, is that it supports multiple

performance, concurrency, and scalability will be conducted on the backend server, given the 
significant modifications done to it.  
Following the tests of the functionality, the attention will be on the potential security vulnerabilities 
within the backend. The potential issues will be identified, and solutions will be discussed. 
 
API- and Functional testing 
When creating an endpoint in the backend server for the ReHyb-system, as mentioned in section 5.2 
a must do action every time after having written the endpoint, were to test it. To ensure that 
everything regarding the endpoint were working accordingly to the requirements.    
Additionally mentioned in the same section, all 6000-7000 lines of code have been tested with the 
help of functional testing (To see if the HTTP endpoints works as anticipated with the database) and 
API testing (to confirm the endpoints are returning the valid results). 
 
To verify the integrity of each endpoint, a series of comprehensive tests were undertaken. Initially, 
the retrieval of the requested data from the endpoint was examined to confirm its successful 
transmission. Secondly, update requests were evaluated to ensure the data was modified correctly 
and the correct data was returned, confirming the successful update of the database. Furthermore, 
the create requests were checked for any missing fields. Lastly, delete requests were executed and 
examined to confirm the removal of the data. 
 
Concurrency, performance, and scalability test 
To test whether the new Node.js server can handle multiple users on the ReHyb system, it is 
important to conduct tests that evaluate its ability to handle user traffic prior to the deployment to a 
production environment. Therefore, it is important to test the capabilities of the node.js server when 
it comes to concurrency, performance and scalability.  
==PAGE:61==

Concurrency is the ability to handle multiple users and their request at the same time, while 
performance describes how fast it takes for the user to send the request to receive a response, and 
scalability is the ability of the system can grow in a way that it can handle a growing traffic without 
dropping data and functionalities of the system.  
If the Node.js server can fulfil all these requirements, then that would apply that the server is 
production ready.  
The testing library chosen and that can fulfil the earlier mentioned three requirements is called 
LoadTest, which is running on a CLI terminal. The command simply needs a URL to test on, run time, 
and the number of concurrent tasks running at the same in the specified timespan.

better, worse, or equally in comparison with the other system. The LoadTest library is handling

6.1 Security 
The security section’s focus will be outlining some of the security concernes that can be found in the 
ReHyb system. While developing, improving and fixing the systems multiple bugs can arise. And 
these errors can arise by human factors by the developers themselves or from the users of the 
system unknowingly. Overtime the system will gain maturity in terms of performance, functionality 
and security for each day that goes by, which induces that it would impossible to achieve a perfectly 
secure system, since hardware, software, security protocol gets outdated (because of newer or 
better alternatives) and security flaws starts appearing over time. 
Security of HTTP 
As widely known, HTTP is the backbone protocol used to be able to visit all the webpages found on 
the internet and locally on your own network or device itself. HTTP is a useful protocol since it can 
transmit HTML elements and data based on whatever information is needed by either using TCP or 
UDP to fulfill the requests between the clients and servers41. There is however a caveat of using http. 
This is easier explained with an example. 
Imagine the scenario you are sitting at a café, are connected to the free WI-FI access point the café 
provides. You have to login into a webpage found on the world wide web (WWW) with your 
credentials (password and email typically) to access the content and service being provided here and 
the webpage uses HTTP as the transmission protocol. The issue here lies with that the data must be 
sent to the server over the internet while being completely unprotected. You may wonder how you 
would be able to get the contents of the request, that should be hard right?

HTTP is an old protocol that is unfortunately still being used today on some pages for malicious 
reasons such as getting your email, credentials or even credit card info sometimes. This is why HTTP 
is being replaced with the newer, better, and more secure descendant which is HTTPS. What is the 
difference between HTTPS and HTTP? 
HTTPS is HTTP making use of SSL/TLS (secure socket layer/transport layer security), to ensure that 
the clients and servers of the system can securely connect, communicate, and ensure privacy of the 
data in the transmit between the client and server, and no one can interfere the connection

can see the contents of the endpoint42. 
Therefore, applying SSL to all the HTTP endpoints found on the new Node.js server in the future 
would increase the security substantially. However, by applying encryption with the help of SSL, can 
create some overhead which results in latency and can prevent some network functions43. 
But in the end, it is better to have a secure connection, than having none. Taking precautions for 
better security would thereby be recommended.  
 
Encryption, access rights and measurements taken in case of data leak 
In the data department of the project, some actions have been taken to increase the security of the 
ReHyb-system. The first security management that has been taken is the encryption of the password 
of the users of the system. This is done because in case of data gets sniffed or leaked somehow to a 
malicious person, they won’t be able to easily access the user’s password and login to the system. 
Every time a new user wants to be registered to the ReHyb-system, their password immediately gets 
encrypted with Bcrypt added with a salt of 10 characters, to ensure that the password doesn’t get 
that easily cracked.  
Currently it is only the password that is being encrypted, for the convenience of testing purposes of 
the system. Preferable in the future before the system gets released that the unique identifiers such 
as email, phone number, full name and other personal data gets encrypted, to ensure that even if a 
malicious person gets their hands on the information, they can’t access the data without having to 
crack or brute force this information easily.  
Unauthorized access has also been taken into consideration as mentioned earlier in thesis (in the 
Implementation section and shortly in the JWT sub section). On the patient dashboard and therapist 
dashboard it is not possible to access the pages by updating the URI in the browser, to access the 
dashboards services and its content. This is done so only users of these dashboards that has an 
account and has logged in with that account, can access the services provided on each of these 
dashboards and not outsiders of the system. This helps with prevention of unauthorized access and 
robbery of confidential information such as patients’ data.

7. Discussion, technical challenges, and limitations 
RQ1: How could a Full-Stack System be implemented that seamlessly that allows communication 
between a HDT database and multiple user interfaces in a stroke rehabilitation setting? 
In the theoretical background section, it was mentioned that a full stack system, is a system 
structure consisting of multiple components that works as one and functions together seamlessly. A 
full stack system consists of a backend, frontend and database. 
To enable communication between the components, thereby allowing clients (such as the 
dashboards) to send requests to the backend and receive responses, it is essential to implement 
communication protocols among these components. HTTP protocol is used to facilitate the 
transmission of requests and responses, whether it is for retrieving images, profile information, or 
phone numbers, serving as the foundation for the communication in the ReHyb system. Meanwhile 
backend stacks such as Node.js is providing the systems with functionalities that allows the system 
to have routing, authentication, and authorization. 
For the ReHyb project, the implementation of a new backend API was necessary because of an 
overhaul in the database architecture. This update rendered the existing Rust backend incompatible, 
as it could no longer fulfil requests for documents from the new database structure as mentioned in 
section 2.3. For instance, a client requesting a patient's name from the new database would 
encounter issues if the request were made through the old Rust backend. This was due to a change 
in the database schema where the patient's name identifier was altered from "name" to "fullName". 
Consequently, the Rust backend, designed to query "name", would fail to retrieve the required data 
since the "name" field no longer existed in the updated schema. To address this challenge and speed 
up the creation of new endpoints aligned with the updated database architecture, the decision was 
made to use Express.js, a streamlined and widely-used library for Node.js. This choice significantly 
reduced the time and effort required to develop each API endpoint necessary for interfacing with 
the new database structure. Express.js enabled a more efficient setup of API endpoints, allowing for 
rapid development that matched the project's needs. Additionally, as the original Rust server was 
not fully completed, so adopting Express.js offered substantial time savings, compared to having to 
fully complete and fix the issues on the Rust backend. This made it possible to focus on further 
==PAGE:68==
67 
 
enhancements to the dashboards and other project components within the allocated timeframe, 
thereby improving the overall project. 
Although the backend could have been implemented in Rust, there was significant challenges in 
recreating the endpoints in a timely manner due to the complex setup of the Rust backend. While a 
Rust backend might have offered a better foundation and error prone solution in the long run, the 
decision to transition to a Node.js presented its own set of advantages. One major benefit of 
switching to Node.js is the reduction in overall system complexity. Given that the technologies used 
across the ReHyb system are JS-based, this alignment simplifies the architecture, making it more 
accessible for future developers. They can more easily grasp the logic and flow of code across both 
the frontend and backend, facilitating a more cohesive development environment. Additionally, 
Node.js has a lot of libraries and packages, along with a significantly larger community compared to 
Rust. While Rust offers library support and has a growing community, the scale of Node.js resources 
enhance the development process. This does not only accelerate development with readily available 
solutions but also provides a wealth of knowledge and assistance from the broader JavaScript 
community, making it a pragmatic choice for the project's immediate needs and future scalability. 
With the backend server implemented, the dashboards could then proceed to call the respective 
needed endpoints and thereby display data found in the database. As realized in the Frontend-
Development section, with the endpoints provided in the backend the users of the therapist- and 
patient dashboard can utilize multiple services, that will allow the users to create, read, update, and 
delete data on demand from the database. Thereby inducing that the dashboard won’t necessarily 
need to further utilize the hardcoded and mocked data, but only use the online data from the 
database. 
The components required in full stack development were realized through the implementation of a 
frontend user interface for the therapists and patients as described in the Frontend-Development 
section, which communicates to a backend server via HTTP as described in the REST API subsection, 
and by storing all the relevant data in a structured way in a corresponding database such as 
MongoDB as mentioned in MongoDB subsection. 
RQ2: What are the key challenges and bottlenecks that can arise in such a system, particularly 
concerning the performance and security? 
In the tests and security section, through the loadTest library tests the new ReHyb backend were 
tested in terms of scalability, performance and concurrency. The tests were made on a small scale

running locally on personal devices acting as the servers, and based on these results, multiple things

Setup b got the best results out of all the setups in the tests. Having two extra cores in the CPU, it 
was manageable to handle more requests than the other two setups in the tests, which have a fewer 
amount of CPU cores to handle the request. This also indicated that with more and potentially 
powerful CPU cores it would be possible to scale up and handle more request concurrently, without 
compromising on performance on the overall handling of the requests. It was clear in all the setups 
that after a certain number of concurrent users, the setups were bottlenecking and possibly 
dropping the requests. More than 500 concurrent clients were showcasing an iteratively increasing 
number of errors (which could indicate the request packages being dropped, because the request 
couldn’t be handled in the assigned timespan), an increasing amount of latency and less request 
were being successfully handled by the setups. Setup C with the M1 CPU crashed halfway through 
the test after multiple runs in 1000, 2000 and 5000 concurrent clients, but a successful test were 
achieved with the 1000 concurrent client. The ReHyb system will potentially in the future have a 
number of concurrent clients surpassing 500 concurrent users. Therefore, the system is only 
recommended to be deployed on a smaller scale for testing purposes in this phase. Since having an 
increasing amount latency with more concurrent users would give a bad user experience and slow 
feeling dashboard experience, whereas a fast and reactive experience is much preferred. Overall, the 
results were positive, and the tests gave insightful results.  
Meanwhile in the security section, a wide-known issue regarding the HTTP protocol has been 
discussed. In transmit, the packages between the client and the server are not being protected in 
any way and can easily be sniffed with the help of Wireshark. Since the personal data is easily 
accessible through a packet sniffer, this will count as a violation against the GDPR rules and will lead 
to a big sum of fines. Therefore, more precautions will be needed to be taken in the future of the 
security and anonymity of personal data to live up to GDPR standards, before the ReHyb system can 
be applied to handle personal data both in and outside European countries. 44

8. Conclusion 
On the given time span of this full stack project, a comprehensive number of contributions to the 
ReHyb system were delivered. As noted in the theoretical section a new backend was implemented 
for the ReHyb system, that would seamlessly allow communication between the frontend 
components and the database itself. The new backend implemented in this project provides the 
users of the system services that would allow to retrieve, update, delete and create data from the 
database accordingly. Thereby the dashboards that have been updated and will no longer display 
hardcoded or mocked data which will give the users of the system an experience of a functioning live 
system. Inducing a solid foundation for the ReHyb system, leading the ReHyb system towards an 
online rehabilitation solution for future patients, therapist, and caregivers. 
Additionally on the frontend dashboards, patient- and therapist dashboard, the user interface and 
code logic have been updated to further improve the dashboards with newer functionalities and 
updated code structure. This was done to display data as initially intended from earlier students 
work with functioning API calls on the dashboards. Thereby the system can fully take advantage of 
updated database architecture. 
With the tests and experiments done in the tests and security section earlier, some key findings 
related to the performance, scalability, concurrency, and security were discovered. The findings 
displayed that the backend server was not production ready for a bigger audience of users, since the 
backend server is only able to run locally on the personal laptop devices. Here it would be suggested 
that the backend server would be uploaded to a cloud provider, which would benefit the ReHyb 
system in the future with the increasing number of users and data in the system. Without this 
recommendation it would be hard for personal laptops to be handle the increasing number of 
concurrent user’s overtime, while not compromising on the performance of the system itself. 
Although, the backend server isn’t secure enough to handle personal data yet, and it wouldn’t be 
advised to deploy this server for production. For the communication between the frontend, backend 
and database, the HTTP protocol is applied. Since HTTP is not protecting the communication in 
transmit it is vulnerable to being sniffed and rewritten by anyone. HTTPS is thereby recommended to 
protect the communication, but additional security measurements need to be taken to protect 
against the weaknesses and challenges that can rise from only using HTTPS in the future. By not 
taking these security challenges into consideration, it could lead to leaks of personal information on 
the internet for malicious individuals. To prevent this challenge it is thereby strongly advised to take 
these precautions into consideration mentioned in the security and test section.

9. Future work 
The ReHyb system has reached a stage, where all necessary endpoints have been implemented. As 
mentioned in Tests and security, there remains some security concerns, which should be resolved 
before releasing it to production. Given that the system contains personal data about the patients, it 
is crucial that such information is not accessible to malicious individuals. 
As noted in Tests and security section, a more robust solution would be to implement SSL encryption, 
so the communication between the client and server is encrypted and conducted via the secure 
HTTPS protocol, rather than the vulnerable HTTP. However, it is important to recognize that while 
HTTPS enhances a secure communication, additional measures must be implemented at the 
endpoints themselves to secure the system even more. Addressing this aspect remains a priority for 
future development, requiring careful consideration by the assigned developer. 
Another improvement of the ReHyb system involves moving the server to a cloud provider. This 
move will assist in scaling up the platform, improving performance and concurrency, which will result 
in a better user experience on the dashboards. 
Lastly, the patient dashboard needs to be optimized with either a different library or with different 
techniques next.js provide to improve the overall performance of the dashboard itself. Compared to 
the therapist dashboard, the time it takes to switch between some of the pages is a bit slower than it 
should be. It seems like the pages are not being cached to improve the performance and user 
experience, this need to be considered in the future to give future patients of the patient dashboard 
a better overall user experience.
