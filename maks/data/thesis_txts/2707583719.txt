Abstract
Building performance simulation tools are powerful when used to aid the designer during the decision-
making process. However, while the use of such tools is widespread in the later design phases, they are not
often used successfully in the early design phase. Some hindrances of this are a lack of knowledge about the
input parameters and time limitations. Additionally, because simulation results are rarely efﬁciently stored
they cannot be easily recycled, meaning that this knowledge goes to waste after a project is ﬁnalised.
This is an attempt to enable efﬁcient use of simulation tools in the early design phase and increase data
reusability. Therefore, an investigation of the use of graph databases for storing and utilising previously
run simulations was carried out. With input from domain experts, the scope and functionalities for a
novel tool were deﬁned. Following this, new ontologies were created to represent simulation data and
room geometries. These ontologies were developed to comply with standards from the World Wide Web
consortium, thereby building on previous work carried out in this ﬁeld. Three ontologies were developed
to connect room geometries with indoor climate simulation results. One main ontology (RSOT: Room
Simulation Ontology Topology) and two sub-ontologies (DSO: Daylight Simulation Ontology and NVO:
Natural Ventilation Ontology) were developed for this purpose. Utilising the developed ontologies, an open
database was developed on a server at the Technical University of Denmark, containing the ontologies
and simulation results. Finally, a ﬂexible and easily expandable tool was developed to interact with the
database, allowing users to quickly obtain and visualise simulation results but also to add new data to the
database.
The tool was evaluated together with domain experts, where it was found that this type of tool can enable
designers to explore new designs and optimise performance parameters by storing previously simulated
data. As the tool was developed as a proof-of-concept, some limitations were found. If the data was to
be fully reproducible, the ontologies should be expanded to contain new classes and properties. However,
if further development was carried out, this could be rectiﬁed. Developing the tool further could create a
sustainable workﬂow, where designers enable each other to populate the database. The study showed that
there is great potential for the use of linked data and semantic modelling to utilise previously simulated
results and optimise building design tools of the future.

Introduction
The use of building performance simulation tools in building design is widespread. The tools are often used
to aid the decision-making process in the later design phases. Commonly used tools in the industry are:
BSim, Radiance, IDA-ICE, EnergyPlus. However, these tools are often limited to later in the design phase.
This is due to the modelling being time-consuming, the designs rapidly changing, conﬂicting requirements
and large uncertainties (Østergård, Jensen, and Maagaard, 2016). In the paper by Østergård, Jensen, and
Maagaard, 2016, they investigate why building performance software is not commonly used in the early
design phase. The most important factors that hinder the use are: the users’ knowledge, the interoperability,
the complexity of the algorithms, independent objectives and whether the software is parametric. Another
important hindrance for applying building simulation tools in the early design phase is lack of time and
knowledge about input parameters. Over the past years, the industry has come a long way to ease the
implementation and use of simulation tools, but some designers and architects still struggle to apply the
tools (Attia et al., 2009). There are a number of reasons for this. In some cases it is because the simulation
software is not an integrated part of the tools which the designer or architect already uses. In other cases,
it is because the geometrical designs cannot be imported from the modelling software to the simulation
software (Attia et al., 2009). Another paper investigating the hindrances for the use of building energy
simulation and optimisation in the early design phase concluded that the three main problems that need to
be solved are: long calculation times, lack of adequate advertisements and a lack of well-deﬁned standards
(Tian et al., 2018).
Building performance software can be combined with optimisation tools to explore and generate new
interesting designs in the early design phase. For evolutionary design optimisation, different tools are
available. Often, the user speciﬁes a numerical performance criteria that the geometry will be optimised
towards (Harding and Brandt-Olsen, 2018). Galapagos and Octopus are software tools which can be used
for this purpose. In the article by El Daly, 2014 a design process which uses evolutionary algorithms (via.
Galapagos) in combination with simulation tools in Grasshopper is described. When using an optimisation
tool, the user must specify a performance metric, which the generated designs are evaluated and optimised
against (El Daly, 2014). This means that the user must deﬁne the most important performance metric very
early on in the design process. Biomorpher is another optimisation tool, which takes a different approach
when optimising and exploring designs. Here, the intention has been to create an evolutionary design
process which should be adaptable and guide the designer. Instead of having the user deﬁne the weights of
a performance criteria, this tool allows the user to select designs from design proposals. Thereby, the user
chooses which designs are reviewed and included in the next generation when performing an evolutionary
optimisation (Harding and Brandt-Olsen, 2018). A disadvantage of this is that the user is "forced" to select
designs early in the design phase, limiting the designer and design exploration. Also, when the simulation
software and optimisation tools are combined, it can be difﬁcult to keep track of the large number of inputs.

To help users keep track of large amounts of data input and output, it was proposed by Long, Fleming,
and Brackney, 2011 to create a database to help the user keep track of the inputs. This was proposed
as the integrity of the simulation data is important to the engineers and designers (Long, Fleming, and
Brackney, 2011). Also, a study was carried out at the University of Michigan where a tool named ParaGen
was developed. ParaGen was a design exploration and optimisation software, where the user could select
designs, much like Biomorpher. However, all the generated data from the Genetic Algorithms was stored
in a database. This tool used an SQL-database stored on a web-server and was applied to a case study for
structural optimisation of a roof geometry (Turrin, Von Buelow, and Stouffs, 2011). The concept for the
tool was developed in 2011, and no further work has been done on it since. In another paper by Negendahl,
Perkov, and Heller, 2014 the idea to create a result database was investigated. Here, the idea was to create
a sentient tool, that observes user activity and can simulate anticipated design situations (by predicting the
space of interest). The tool was meant to improve the interpolation and result prediction, the larger the
database becomes. This database was designed to deliver fast results.
During the past decade linked data and graph databases have become more popular in the building industry
(Bonduel et al., 2018). Different types of graph databases exist, and in the article by Roith, Langenhan,
and Petzold, 2017 an investigation of labelled property graphs was performed. They present a novel design
tool, which retrieves ﬂoor plans from a labelled property graph based on input of ﬂoor plan sketches. The
Neo4j graph database was used to describe the topological relationships between the spaces. First, the hand
drawings are analysed and translated into a graph. The different rooms are represented by nodes whereas
the walls and doors are represented by edges. By translating the ﬂoor plans into a graph, similar ﬂoor plans
and building models can be retrieved from the database. This sketch-based image retrieval method was
also applied in the paper by Zhao et al., 2021 for facade design, allowing the user to decide whether he/she
likes/dislikes the aesthetic of the design proposal.
Another type of graph database is the Resource Description Framework (RDF) triplestore. To use these,
ontologies need to be developed and implemented to describe the relationships and enable inference be-
tween the data points. The Building Ontology Topology was developed to deliver a main ontology for
description of a buildings topology. As stated in the article by Rasmussen, 2019 "(...) each practitioner
should be able to extend the generic terminology with domain-speciﬁc terminology". The BOT ontology
established a foundation for users to describe building topologies. Many have since created new ontologies,
building upon the classes and properties from BOT. The paper by Djuedja et al., 2021 aimed to create an
approach and tool that could make the necessary data available to carry out life cycle assessments (LCA)
in the early phase of the building life cycle. This tool and ontology was developed in conjunction with
BOT and the aim was to integrate LCA data into the Building Information Modelling (BIM) environment.
The approach was built upon linked-data principles and semantic web technology, while also utilising the
newest BIM techniques (Djuedja et al., 2021). In the paper by Donkers, Yang, and Vries, 2021 the ’Build-
ing Performance Ontology (BOP)’ was developed to describe sensor data along with a python tool for

monitoring building performance. The goal was to compute the thermal comfort of a space in real time,
creating a digital twin, to improve the building performance. The ontology was applied to IoT data and to
a case study. The main ontology that was built upon was the BOT ontology (Donkers, Yang, and Vries,
2021).
1.1
Motivation and Objectives
When performing building simulations, data often gets deleted or forgotten about afterwards. Simulation-
ists spend a lot of time setting up and performing simulations, so the possibility of saving and reusing the
data is valuable. Creating a database containing simulation results would solve this problem and make the
data valuable after the simulation is complete. Thereby, the user can quickly obtain results, along with the
simulation parameters. Another advantage of creating a simulation database is that the data can be used in
an interdisciplinary way. If multiple users add simulation results from varying simulation types, the users
can map the relationship between design consequences and their parameters.
The motivation of this work was to create a database and tool for design exploration and optimisation
for the early design phase. The aim was to create a tool that can connect to an open database placed
on a DTU server using the plug-in Grasshopper for Rhino. The goal of the tool was to aid and inform
the user by quickly retrieving previously simulated geometries and results, which can then be used to
determine the direction of the design. Because of the speed and inference capabilities this was done by
utilising RDF triplestores. To do this, ontology development was key. Therefore, the main goal of this
thesis was to connect a room geometry with different types of simulation results and boundary conditions.
The tool should enable the user to interact with an open-source database to retrieve data for visualisation,
design exploration and optimisation. The tool should have ﬁltering functions, that allow the user to search
for geometries based on demands (e.g. Danish Building Regulations). The user should be able to add
and delete data to the database via the tool. Adding more data to the database will improve the tool
functionality.

Theory
This section will cover the relevant theory applied throughout the work. The theory will cover the most
relevant database types, the semantic web and ontology development. Afterwards, different room represen-
tation strategies are explained and the relevant indoor climate theories and simulation types are described.
2.1
Databases and Linked Data
There are many ways to model data and the main difference lies in how the data is stored. A database is
a collection of data or information, that is organised speciﬁcally for querying and retrieval. The data can
be managed by a database management system (DBMS). When the user makes a query, it is the DBMS
that extracts data from the database in response to the query (Database, Computer Science). Thereby, the
DBMS is the software that interacts with the database. Database technology is rapidly evolving and various
types of databases have been developed over the past decade. As the amount of database types increases,
it becomes important to choose the right type of database for the given application.
2.1.1
Different Types of Databases
This thesis focuses on No-SQL databases, more speciﬁcally graph databases. The more traditional type
of database is the relational database. Relational databases store data in predeﬁned structures (rows and
columns), while No-SQL databases store data in a non-tabular way (DBMS, Types of Databases). The
difference between the two is illustrated in ﬁgure 2.1.
Figure 2.1: Illustration of a relational database vs. a graph database.
Relational databases can be used for storing tabular data, which is the simplest type of data set. Most
people are familiar with tabular data which can be created in Excel spreadsheets. Tabular data is stored in
tables, which is advantageous because it is simply structured and thus easy to read and modify. However,
tabular data has its limitations. When the data is stored in multiple spreadsheets, there is no easy way to
search across the documents (Segaran, Evans, and Taylor, 2009).
Page 5 of 92
==PAGE:19==
Relational databases allow multiple tables to be joined. The data must be presented in rows with the
properties added as columns, which are then grouped into tables. In relational databases it is possible to
query across the joined tables. Some of the software that exists for relational databases are: Oracle DB,
MySQL, Postgres and SQLite. They are all well-tested and mature systems. Relational databases work
well when the use of the data is predictable and the data model is well understood. The main limitation of
this type of database is the predeﬁned structure, which makes it more rigid and not as extendable (Segaran,
Evans, and Taylor, 2009). Relational databases are not very ﬂexible and it can be difﬁcult to add new
relationships or information. To add a new relationship, the schema needs to be restructured or new tables
need to be created and linked together with the existing table (Batra and Tyagi, 2012). Therefore, relational
databases are best suited for a ﬁxed type of data and predictable use (Segaran, Evans, and Taylor, 2009).
No-SQL databases are used for storing large datasets and are often applied in real-time web applications.
No-SQL databases can be further divided into four categories, where one of the categories is the graph-
database, which is the type of database investigated in this work. Graph-databases are good for storing large
datasets in graph-structures. Many websites for social-networking use a database of this type. Another
example of this is Google, which uses a semantic search engine (Johnsen, 2019). The graph structure is
very ﬂexible. This is due to the the relationships being described by the data itself (Segaran, Evans, and
Taylor, 2009). An resource description framework (RDF) triplestore refers to a semantically rich graph
database, where the data can be described using web-ontologies. The database can become very ﬂexible
and valuable, by enabling the users to create their own logical models and apply the data, without having
to change it (What is a NoSQL Graph Database?).
2.1.2
Labelled Property Graphs and Resource Description Framework Triplestores
There are different types of graph databases which provide different possibilities of exploring and graphi-
cally depicting data. Labelled property graphs is one type and RDF triplestores is another.
Neo4j is a popular software developed to manage labelled property graphs (LPG) in Java. It is a scaleable
and robust graph database. Neo4j has its own query language named Cypher, which is a declarative lan-
guage (Batra and Tyagi, 2012). The labelled property graph has an internal structure. The data is stored
in nodes, relationships and properties. The relationships that link the nodes together are called edges. The
internal structure allows both the nodes and the edges to have properties. An example of this is shown in
ﬁgure 2.2, where it can be seen that the two nodes both have properties and that the edge has no property.
Page 6 of 92
==PAGE:20==
Figure 2.2: Illustration of a labelled property graph.
The nodes have IDs in order to identify them (’Room’, ’Result’), whereas the edge has a type, which in this
example is ’hasResult’. An advantage of the labelled property graph is its simplicity, where the properties
are "on" the nodes and edges, resulting in shorter and simpler queries. It is more intuitive as it represents
drawing on a whiteboard (RDF Triplestore Vs. Labeled Property Graph). A disadvantage of the labelled
property graph is that inference is not enabled.
Resource Description Framework (RDF) triple-stores are the standard by the World Wide Web consortium
(W3C) for data-exchange on the web. A single triple is composed of a subject, a predicate and an object.
The subject is a resource. It is the same as the node in the labelled property graph. The predicate is the
relationship, which is similar to the edge in a labelled property graph. The object can either be another
node, a value or a literal. When there are multiple triples in a database, the object of one triple can be
the subject of another. Thereby, large graph structures can be constructed. If the predicate connects to
another resource, it is an "ObjectPropery". Whereas, if it connects to a value or a string (literals), it is a
"DatatypeProperty" (W3C: OWL 2 Web Ontology Language Direct Semantics (Second Edition)).
Figure 2.3: Illustration of a triple.
One of the main differences between the labelled property graph and RDF triplestore is that the subject, the
predicate and the object can be identiﬁed by a URI (Uniform Resource Identiﬁer) in the RDF triplestore.
A URI is a unique sequence of characters that identiﬁes a resource (RDF Triplestore Vs. Labeled Property
Graph). Using URIs is a part of the semantic web, where the URI is the link between the RDF data and the
web. IRIs can also be used to represent linked data. They act as an expansion of a URI, as they can include
characters from non-Latin alphabets.

2.1.3
The Semantic Web
The semantic web was designed to bring structure and meaning to all the content on the internet. Before
the idea of the semantic web was developed, the content on the internet was only made for humans to read
and understand. By expanding this, and making all data understandable by computers, the users would be
capable of ﬁnding all the data related to a single resource (Berners-Lee, Hendler, and Lassila, 2001). The
implementation of the semantic web should be able to save the user large amounts of time, and even be
able to assist the evolution of human knowledge as a whole (Berners-Lee, Hendler, and Lassila, 2001).
In 2006, Tim Berners Lee created four rules, or expectations of behavior for the semantic web expansion
(Linked Data). His four points were:
• Use URIs as names for things.
• Use HTTP URIs so that people can look up those names.
• When someone looks up a URI, provide useful information, using the standards (RDF, SPARQL).
• Include links to other URIs, so that they can discover more things.
The semantic web is built upon the idea that all data needs to be structured the same way, for everything
to be interconnected correctly. If the four rules are not complied with, it does not break the system, but is
viewed as a missed opportunity for linking the data (Linked Data).
Without this clear structure, the semantic web could not be successful. The semantic web layer stack shows
which rules and schema that must be applied by the structuralization of the data for the semantic web to
understand the data. An illustration of the stack is shown in ﬁgure 2.4. It shows the architecture of the
semantic web. Each layer uses the capabilities of the layer below. The top layers are still not fully deﬁned,
but will have to be in order for the full vision of the semantic web to be realised.
Figure 2.4: Illustration of semantic web layer cake.
Page 8 of 92
==PAGE:22==
The standard data model of the semantic web is based on the use of RDF data. What makes RDF data
valuable is that it allows for assertations. An example of this could be that a person always has two parents
(Alesso and Smith, 2004). The RDF schema language enables inference and is expandable. The use of
inference is very valuable and can be used to draw conclusions based on the data. An example could be that
a person always has two biological parents, and the same counts for their parents. Inference ensures that
the computer understands the relationships between them (grandparents, parent and child, etc.) (Alesso
and Smith, 2004).
The semantic web is based on an open world assumption. This means that if an answer to a query is not
deﬁned, it will return that the answer is unknown. If this happened in a closed world assumption, the
computer would always return an answer. An example of this is the statement "Ben is a citizen of the
USA". If the question "Is Ben a citizen of Denmark?" was asked, the closed world assumption would
return the answer "No", whereas in the open world assumption, the answer would be unknown. The open
world assumption is applied when the database or system does not have complete information (Open World
Assumption vs. Closed World Assumption). This project uses RDF data, so the open world assumption was
applied. Therefore, no incorrect conclusions to the queries could be drawn. The slogan ’Anyone can say
Anything about Any topic’ (AAA) also applies to the semantic web. This means that the computer allows
for any statements to be written. This means the user should be careful of what is being said. The database
can end up implying statements that are illogical, drawing conclusions different to what was intended
(Allemang and Hendler, 2011).
2.1.4
Ontologies
An ontology is a formal vocabulary, which contains classes and properties designed to describe a certain
’part of the world’. To represent ontologies, the World Wide Web consortium recommends the use of
OWL (Web Ontology Language) which has been regarded as the standard since 2004 (Hitzler, Krotzsch,
and Rudolph, 2009). Here, it is essential to be aware of the level of detail, as the ontology should be
scalable, but also be able to describe the data sufﬁciently enough to enable inference (Hitzler, Krotzsch,
and Rudolph, 2009). The ontologies are used to turn data into RDF data instances (Abox). Working with
the ontologies makes it necessary to separate the Tbox (assertions on concepts) and Abox (assertions on
individuals) (De Giacomo and Lenzerini, 1996). TBox is a shared data structure or an ontology. The actual
data is in the ABox, which stores the data as an RDF graph (Bonduel et al., 2018).
The work done in this thesis is based on the principles from W3C and Tim Berners-Lee, using web ontolo-
gies (Linked Data). OWL was the main ontology used to develop the new ontologies for the tool. Besides
OWL, the new ontologies build on the Building Ontology Topology (BOT). BOT was developed for the
AEC industry as an ontology to allow the user to deﬁne a building’s topology. This is done by using BOT
classes, bot:Zone, bot:Interface, bot:Element and bot:Space (Bonduel et al., 2018). To further develop the
Page 9 of 92
==PAGE:23==
ontology, new classes can be made to be owl:subClasses of the classes from the BOT ontology.
2.1.5
Querying Languages
Graph databases (RDF) usually utilise the same querying language. SPARQL is the standard querying
language by W3C. The SPARQL querying language allows the user to create, edit and retrieve data from
the database (Kumar, Kumar, and Kumar, 2011).
SQL queries, that are used for relational databases, support a null-value which is used to represent missing,
unknown or inapplicable data. Even though the null-values represent "no value (yet)" they take up server
space. This is different to SPARQL queries and RDF-triplestores where null-values are not expressed.
Thereby, it does not complicate the queries or take up server space (Kumar, Kumar, and Kumar, 2011).
SPARQL uses keywords for querying the data, e.g. SELECT, FROM, WHERE, UNION. SPARQL queries
work by matching graph patterns, and showing the data selected by the query. This is shown in ﬁgure 2.5.
Figure 2.5: Illustration of the SELECT and WHERE SPARQL queries.
Figure 2.6 shows a query which returns all triples that have a subject, a predicate and an object indicated
by the ’?s ?p ?o’. Question marks are used when deﬁning variable names. Thereby, they can match any
subject, object or predicate. This query will return all triples in the database. The SELECT clause returns a
table of variables and values that match the query. The WHERE clause speciﬁes the conditions (GraphDB
What is SPARQL?).
Figure 2.6: Example of a SPARQL query.

The preﬁxes in the beginning of the query can be used as abbreviations instead of writing the entire URL.
The SELECT * (asterisk) selects all variables mentioned in the query. This could be replaced with ’?s’

Another important SPARQL clause is CONSTRUCT, which returns a subset of the graph. The INSERT
and DELETE clauses can be used to add or remove triples (GraphDB What is SPARQL?).
2.2
FAIR Principles
The four FAIR principles: Findability, Accessibility, Interoperability and Reusability, enable good data
management and help users of data and data generators to extract maximum beneﬁt from research data. The
principles state that data should be well-curated, indexed and searchable, integrated and special-purpose
repositories should provide support (Wilkinson et al., 2016).
The FAIR Guiding Principles from the paper by Wilkinson et al., 2016 are listed in the same way here:
To be Findable:
• (meta)data is assigned a globally unique and persistent identiﬁer
• data is described with rich metadata
• metadata clearly and explicitly includes the identiﬁer of the data it describes
• (meta)data is registered or indexed in a searchable resource.
To be Accessible:
• (meta)data is retrievable by its identiﬁer using a standardized communications protocol
• the protocol is open, free and universally implementable
• the protocol allows for an authentication and authorization procedure, where necessary
• metadata is accessible, even when the data is no longer available.
To be Interoperable:
• (meta)data uses a formal, accessible, shared, and broadly applicable language for knowledge repre-
sentation.
• (meta)data uses vocabularies that follow FAIR principles
• (meta)data includes qualiﬁed references to other (meta)data.
Page 11 of 92
==PAGE:25==
To be Reusable:
• meta(data) is richly described with a plurality of accurate and relevant attributes
• (meta)data is released with a clear and accessible data usage license
• (meta)data is associated with detailed provenance
• (meta)data meets domain-relevant community standards.

Indoor Climate Simulations
It is important to ensure a good indoor climate inside buildings, as almost 90% of our time is spent indoors
(Ventilation, Velux). Many designers opt to use building simulation tools to achieve and document that the
building complies with the regulations (Østergård, Jensen, and Maagaard, 2016). This section contains a
short deﬁnition of each of the simulation result types used. The result types applied in this study relate
to daylight and natural ventilation simulations. The natural ventilation simulations were carried out using
Computational Fluid Dynamics (CFD).
2.4.1
Daylight
In the past, studies have proved that light affects health, well-being, alertness and sleep quality (Shishegar
and Boubekri, 2016). This makes the simulation and implementation of sufﬁcient daylight important (1.7
Daylight calculations and measurements). When performing daylight analyses, different simulation types
and analysis methods exist. This section solely covers the simulation types applied in this thesis.
Page 13 of 92
==PAGE:27==
When performing a daylight simulation, the measurement height and grid size are important factors. Ac-
cording to the Danish Building Regulations, the measurement height for dwellings must be 0.5m above the
ﬂoor and for work spaces it must be 0.85m above the ﬂoor. This is illustrated in ﬁgure 2.9 (Lys og Udsyn,
Bygningsreglementet.dk).
Figure 2.9: Illustration of the standardised measurement heights for daylight simulations.
Daylight analyses are performed 0.5m from the walls. The mesh of the calculation area must be equally
distributed, where the ratio between the grid size must not exceed 0.7. To ensure comparable analyses,
the weather data from a Design Reference Year, DRY2001-2021 should be applied (Lys og Udsyn, Bygn-
ingsreglementet.dk). The daylight simulation types and their metrics are described below.
Daylight autonomy (DA) is described as the percentage of the occupied hours when the target illuminance
at a point in the space is met by daylight (1.7 Daylight calculations and measurements).
Spatial daylight autonomy (sDA) is the percentage of analysis points across the analysis plane that meet
the target (set to 300 lux for the Danish Building Regulations) for at least 50% of the occupied hours (1.7
Daylight calculations and measurements).
The daylight factor (DF) is deﬁned as a ratio of the illuminance available inside a room at a point (on a
work plane) compared to the simultaneous external illuminance on a horizontal plane from an unobstructed
sky (1.7 Daylight calculations and measurements).
The useful daylight illuminance (UDI) is deﬁned as the percentage of time across the occupied hours
where the illuminance at a point is between the selected minimum and maximum levels (1.7 Daylight
calculations and measurements). The UDI daylight thresholds used in this study were chosen to be the
same as the Honeybee thresholds used in Grasshopper (Ladybug tool: Honeybee). The three thresholds
used are:
• Are within the range deﬁned as useful (i.e. 100–2000 lx);
• Fall short of the useful range (i.e. less than 100 lx);
Page 14 of 92
==PAGE:28==
• Exceed the useful range (i.e. greater than 2000 lx).
2.4.2
Natural Ventilation
Natural ventilation is a cheap and sustainable ventilation strategy. However, natural ventilation can lead
to issues with the thermal comfort inside the space, if it is not designed correctly. This is due to the
fact that natural ventilation can lead to low temperatures and high air velocities. High air velocities and
low temperatures can lead to an increased risk of discomfort, and an indoor climate that lies outside of
the acceptable range deﬁned by ASHRAE (Olesen and Brager, 2004). The deﬁnition of draught is an
unwanted local convective cooling of a person (Fanger et al., 1970). The draught rate is the percentage
of people predicted to be dissatisﬁed due to draught (Innova AirTech Instrument: Thermal Comfort). The
draught rating depends on the activity level, the air temperature and the turbulence intensity of the air. The
measurements of the temperature and air velocity are simulated at the standardised measurement heights
(Standard, 2007). These values can then be averaged, to ﬁnd the spatial average air temperature and
velocity. A ﬁgure of the four standardised measurement heights is shown in ﬁgure 2.10. There are different
measurement heights for seated and standing occupants.

Discussion with Domain Experts
In order to develop a relevant and usable tool, it was important to facilitate a dialogue with domain experts

represent different disciplines from the building industry, thus mapping different areas of interest. The
discussions were carried out to uncover what would make a database and tool with indoor climate results
valuable for each of the experts.
The domain experts selected for the discussions was an architect from BIG (Bjarke Ingels Group) and a
professor from the Technical University of Denmark (Institute of Civil Engineering and Indoor Climate).
They were chosen due to their different demands and wishes for design tool capabilities.
When discussing the tool and its potential with the architect, he stated that a database would be useful. He
thought it would be useful to store simulation data in a database, as he often ﬁnds that the simulation data is
used only once. He believed that architects and simulationists often became experts in their own ﬁeld, but
the knowledge was often not passed on. After the completion of a project, all the accumulated knowledge
was not utilised further in other projects.
The architect also stated that he would prefer a tool that can deliver a live response to how a geometric
change will affect the other parameters, based on simulation results. By doing this, the tool would give the
architect insight into the relationship between multiple parameters. For varying indoor climate parameters,
he would ﬁnd a tool of this nature very useful. However, he would ﬁnd it even more valuable if it could
deliver interdisciplinary results. Cost and sustainability as additional parameters to indoor climate could
add even more value to someone like him, as he would then be able to examine the relationship between
multiple parameters and see how they affect each other.
Also, he stated that a fast response is more important than the accuracy of the tool, stating that, “All models
are wrong, but some are more useful than others". As regards tool capabilities, his wish was for a visual
room geometry that the user can change while simultaneously receiving results. He was also interested
in a "map" of the solution space, where the relationship between the variables could be shown. The map

4.1.1
Validity of Data
In order to make a sustainable database it was important to allow the users to import data to the database.
Increasing the amount of data would improve the all tool functionalities, due to a greater variety of available
designs. When allowing the users to add data, it is important to consider how this data is validated.
There are different ways to ensure that the data is trustworthy. For example, the data can be veriﬁed/peer-
viewed, it can be added by a trustworthy simulationist or all the simulation parameters can be included
(allowing for reproduction). The architect from the interview mentioned that they would not be interested
in using an open shared database. Due to most companies keeping their simulation setups in-house, it
is assumed that not many would be willing to share all their data in an open database. However, some
companies might be willing to share simulation results, without sharing the simulation parameters (not
allowing for exact reproducibility).

Development of Ontology
In this section the development of the ontologies will be described. The development was an iterative
process, based on trial and error. Therefore, the ontology was fully developed after some iterations. The
three most signiﬁcant versions will be explained further in this section by using examples. All the changes
that were carried out were documented and tested using different SPARQL queries. The full ontologies are
shown in appendix 9.5, or on GitHub (GitHub, Simulation Ontologies).
The aim when developing an ontology is to make it detailed enough that it can be applied to a speciﬁc
situation, but also to make it expandable. Therefore, it is important to ensure that it is understandable,
well documented and can allow for future users to develop it further. The ontologies developed during
this work are an extension of the Building Ontology Topology (BOT: Building Topology Ontology). The
BOT ontology was developed to describe building topologies by describing spaces, zones and interfaces.
No ontologies have been developed to describe room geometries and simulated results. Therefore, it made
sense to connect a new ontology to the BOT ontology. This was the Room Simulation Ontology Topology
(RSOT) made to describe the simulated room geometry and simulation results. The ontologies developed
during this work were centred around this main ontology, with two sub-ontologies that were developed to
contain results from different simulation types. In ﬁgure 5.1, the relationship between the ontologies are
shown. The RSOT ontology was developed to be expandable and to contain different types of simulation
results and boundary conditions. During this work the natural ventilation ontology (NVO) and the daylight
simulation ontology (DSO) was developed. In ﬁgure 5.1, the possibility of expansion is illustrated by the
addition of an indoor acoustics ontology and an energy simulation ontology. However, these were not
developed.
Figure 5.1: Illustration of the overall ontologies.
The sub-ontology ’Natural Ventilation Ontology’ (NVO) was developed to contain simulation results and
boundary conditions related to natural ventilation simulations (CFD-simulations). The sub-ontology ’Day-
light Simulation Ontology’ was developed to contain boundary conditions and results from various types
of daylight simulations.

Reproducability
The ontology developed in this thesis was made to contain simulation results and the most important
boundary conditions. However, during the ontology development the focus was to ensure that the database
and queries could work for a design purpose. The ontology was delimited to this, as the scope of the project
work was to create a design tool. Therefore, the ontology has not yet been developed to represent the full

to describe the necessary inputs.

Development of Tool
In this section, the tool and database setup are developed and explained. In order to create a database for the
users to input and retrieve data, a DTU server was used, allowing for both input and output. Afterwards, the
tool was developed to interact with the database based on the scope of the tool. Following the explanation
of the tool and database setup, the tool and its functionalities will be displayed via examples. Finally, the
domain experts’ feedback on the tool functionalities will be explained.
6.1
Database Setup
The tool was developed as a plug-in to Grasshopper, where Grasshopper serves as the canvas to interact
with the database via the developed components. The tool was developed by creating components, where
the functionalities were written in Python code. Figure 6.1 illustrates the actions that can be performed by
using the tool. The tool makes it possible to obtain and visualise simulation results from the database, as
the tool outputs simulation data. The obtained simulation data and results can be further processed using
Grasshopper. The ﬁgure also shows that data can be added or deleted from the database, directly from the
Grasshopper interface.
Figure 6.1: Illustration of the software architecture of the tool.
When developing the tool and database, it was ensured that the data fulﬁls the FAIR principles (FAIR
Principles). To make the data ﬁndable, identiﬁers were used to name the instances within the database.
The data was made accessible by using an open DTU server. Queries can both be executed by using the
developed components, or by writing new SPARQL queries. The interoperability of the data was ensured
by using a web ontology, building on the foundation created by the W3C community. The reusability of
the data was ensured by allowing users to query the existing results in the database. The current version
of the ontologies allows the user to add his/her credentials and whether the simulation has been veriﬁed or
benchmarked. The database fulﬁls most of the FAIR principles.
Page 37 of 92
==PAGE:51==
The tool was developed so that multiple users can access and use the database simultaneously. This was
done by installing the GraphDB software on a DTU server. A remote repository was set up in GraphDB.
This was done to enable access by all users on a DTU network. GraphDB can also be run locally, but by
using the remote repository, it is not necessary for the users to install the GraphDB software locally. One of
the focuses of the tool was to develop a simple and user-friendly interface. This was done by "hiding" the
SPARQL queries and HTTP request methods in the developed Grasshopper components. The setup of the
tool was done so that the user does not have to write any SPARQL queries. This was essential, as the tool
would otherwise not be usable to anyone without prior knowledge of the SPARQL language. Therefore,
the capabilities shown in ﬁgure 6.1 are performed solely by using the Grasshopper components.
The HTTP POST method was used to fetch statements (results, boundary conditions and geometries) from
the remote repository. It retrieves the data from the repository that matches the ﬁlter (from the SPARQL
query). The HTTP request was set up, so that the SPARQL query is URL-encoded within the Grasshopper
components. Thereby, if the user has prior knowledge of SPARQL, it is possible to write and execute
his/her own SPARQL queries. The ’query’ endpoint is used to query the existing data in the database.
However, to delete or insert new data, the ’update’ endpoint was applied.
6.2
Performance
In this section, the test performed on the database is explained. Before the deployment of the database, the
functionality was tested, to ensure that it was capable of returning fast results. This was a priority as the
speed was a signiﬁcant concern to one of the domain experts.
Over the years, many different software tools have been developed to work with graph databases (Best
Graph Databases in 2021). Many of these work in different ways and have different price tags. Initially,
the choice to work with GraphDB was made, as they have a free version of the software, and many of the
other tools available are expensive. Some of the tools, for instance RDFox, have other tool capabilities
that can be very valuable to the users. An example of this is the way the reasoning inside the database
works. Where GraphDB will calculate the inferred results each time they are queried, RDFox will only
calculate the values once, and store them in the database afterwards, making for much faster queries and
results. RDFox has a speciﬁc rule language, which enables the user to input and retract rules, removing the
cumbersome and error-prone manual work of maintaining the database when changes are made (RDFox: 6.
Reasoning in RDFox). In spite of this, GraphDB was chosen, as it has a free version available. Therefore,
it was important to test the capabilities before implementation.
GraphDB does not have the same reasoning capabilities as RDFox. Therefore, a workaround was made and
tested. SPARQL queries were used to input the average, minimum and maximum values of the simulated
results to the GraphDB database. This was done to obtain faster results, as the queries would be consider-

6.4
Add and Delete Data
Increasing the amount of data in the database can greatly improve the possibilities for exploration and
optimisation of room designs. To fully utilise the capabilities of the database, it is essential that more
simulation data can be added. Therefore, it was important to develop components allowing for this. This
was done by creating components to generate RDF triples. Each of the components was made to generate
data for the room geometry, results and boundary conditions (both natural ventilation and daylight). The
room geometry and result components were made to be general, so that they could be used with different
result types. The simulation-speciﬁc components were made to contain the most important boundary
conditions. However, at this stage, the tool is not capable of delivering reproducible results. For these
components to contain reproducible data, the tool and ontology would need to be expanded to contain all
the required values, as only some are represented at this stage. The speciﬁed values are listed in appendix
9.3. All components used for adding and deleting data can be seen in appendix 9.4.
To partially validate the data, it is helpful to know which users have simulated it. To accommodate this,
the ’User Settings Turtle Generator’ component was made, where the user can enter name, email, title and
notes. In the notes, the user can write if the data is benchmarked or veriﬁed. All the generated triples should
be merged in the ’Merge Turtle Files’ component, which creates the ﬁnished SPARQL query. To execute
the SPARQL query, thereby adding the data to the database, the component ’Send Data to Database’ was
made. This component uses the SPARQL query "INSERT DATA" method.

For the user to delete data, the ’Delete Data’ component can be used. This component uses the "DELETE
DATA" SPARQL method. As the data always includes randomly generated UUID’s, the users cannot
accidentally delete large amounts of data. This ensures that the deleted data was simulated by the same
user, who wishes to delete it.
6.5
Exploration
During a design exploration process, the designer is not meant to be restricted. Any tools the designer
uses should enable creativity and design freedom. For design exploration, two approaches were previously
deﬁned in section 4.2. These two approaches will form the basis of the design exploration functionality
of the tool. The approaches were deﬁned as a performance-based approach and a keyword/geometric
approach.

The component requires the user to specify some inputs in order to ﬁlter. One of these is the ’analysisType’,

By many of the components, the same input values were applied. These standard input values are: ’oper-

==PAGE:55==
to deﬁne the constraints of the query, by choosing between larger than, smaller than or equal to. In ﬁgure

boundary conditions. However, if it is set to false, which is the default, it will only contain the geometric
values (i.e. window ﬂoor ratio, width, depth etc.) and the selected result(s). The ’limit’ variable can be
used to limit the number of results being extracted from the database. The ’query’ is also a standard output,
which returns the resulting SPARQL query, ready for execution.
The ’Single Condition Query’ component is set to query the natural ventilation results after they reached
steady-state. A more detailed description of the simulation setup, grid independence study and benchmark
simulation can be found in appendix 9.1.
The ’Room Query’ component was developed to be used for a keyword-based design exploration approach.

The ’Execute Dynamic Query’ component was developed to execute queries and return results from
the database. The component is shown in ﬁgure 6.6. It takes the ’url’ input, which is the link to the
database repository. In this case, it is the link to the GraphDB respository on the DTU server ’http://byg-

Figure 6.6: Illustration of the ’Execute Dynamic Query’, which sends and executes the query.
The ’Execute Dynamic Query’ component shown in ﬁgure 6.6 was created to output the room geometries
and boundary conditions. The component will always output: result, roomType, roomWidth, roomDepth,

Besides the ﬁxed outputs, there will be varying outputs from the different simulation types. These will

simulation, for example glazing transmittance is associated with daylight simulations and outdoor veloc-

always disconnect the outputs when it is updated. Therefore, two other components were developed to
be used if the user wants to set up a script and keep the components connected when recomputing (these
components are shown in appendix 9.4).

6.6
Optimisation
After the initial design exploration the user can choose to further optimise and visualise the simulation
results. By retrieving previously generated results the user can visualise the conditions throughout the
room, delivering understandable results quickly. This stands in contrast to running new simulations for
visualisation and documentation purposes, which can be very time-consuming. Therefore, using the tool
can save the designer a lot of time. Furthermore, performing simulations requires a lot of knowledge to set
up correctly. Whereas using the tool requires less simulation-speciﬁc knowledge, as evaluating simulation
results is easier than performing simulations. To use the optimisation components it is still required that
the user has an idea of what is a good and bad simulation result.

6.7
Tool Flexibility
The tool was developed on the basis of the ﬁnished ontologies from section 5. The developed ontologies
were made to ensure future tool ﬂexibility. It is possible for users to develop and apply new ontologies for
other simulation types. For users to add new types of data to the database, the basic components can be
used, but simulation-speciﬁc components would have to be added. If the user understands basic Python,
this should be possible. New ontologies can be sent to the database via a SPARQL query. The ’Single
Condition Query’ component can also easily be expanded to work with new parameters and simulation
results. If the data is entered to the database, the user can tweak the component (using Python) to function
with the new variables. Furthermore, if the user knows how to write their own SPARQL queries, these can
also be written in Grasshopper panels and executed by the tool.

Feedback from Domain Experts
After the development of the tool, discussions with new domain experts were carried out. Due to the
pandemic, it was not possible to meet up with the same experts again.
The ﬁrst domain expert worked as an engineer with the Danish architectural ﬁrm C.F. Møller. He said that
especially the ’Single Condition Query’ component seemed very useful. For a non-technical designer this
component would be able to deliver useful results without having extensive knowledge about simulation
set-ups and building regulations. He also thought the geometric output was easy to understand. He found
that the ’Natural Ventilation Query’ component and ’Daylight Query’ component used for optimisation
might be too technical for users without extensive simulation knowledge. However, if the user has the

Also, the expert stated, that in order for this tool to become "the perfect setup", the database should be
low maintenance. As most industry professionals are busy people, it should be easy to handle and that
adding or deleting data from the database should not take more than ﬁve minutes after the simulation is
complete. He stated that the tool should be aimed at a speciﬁc target audience. This statement was aimed
at the future development of the tool, as it is now a proof-of-concept. However, he stated that the tool was
"very impressive".
The second domain expert was an architect from the Danish architectural ﬁrm Dorte Mandrup. He stated

when querying and could deliver fast results. He said that it is not uncommon, even for architects, to
be familiar with the Grasshopper interface. Therefore, he thought the decision to develop the tool in
Grasshopper was a good choice. However, scripts can be powerful tools to the designer, but passing on a
script to a new user can be challenging.

7
Discussion
The aim was to develop a tool for exploration and optimisation in regard to indoor climate simulations,
using a graph database and semantic modelling. The developed database and server makes it possible to
store preexisting and new simulation results, which can be retrieved by the tool. The developed tool was
a proof-of-concept and the database and ontologies functioned as intended. However, the tool had some
limitations meaning some functionalities could be developed further.
7.1
Limitations
The necessary validity of the data is dependent on what the data is to be used for. If the use is for academic
purposes, the simulation setup and benchmarking of the results can be important. To understand and val-
idate the simulations, documentation of the entire simulation set-up is necessary. For academic purposes,
the tool could be further developed and the database could be populated further. In the developed tool, it is
only possible to add contact information, credentials and a note on the result veriﬁcation. For the tool to be
even more useful and trustworthy, the entire simulation setup could be included and documented further.
For a designer working in a commercial setting, this type of tool can be valuable, even if the simulation
set-up is not fully documented, as mentioned by one of the domain experts. He stated that retrieving fast
simulation results can help guide the designer in the early design phase. However, despite the lack of full
documentation, the tool could be very useful in a real life setting. This is based on the statement from one
of the domain experts, as he said he would value speed above accuracy.
Another limitation of the tool is that it can only query and return existing results from the database. This is
a considerable limitation as the solution space will be very limited. The design exploration experience will
be inhibited by having a limited number of input options. This could be rectiﬁed by creating interpolation
and extrapolation options. If this was implemented, along with error estimation, the tool could become
even more valuable.
As many simulationists work with parametric scripts in Grasshopper, it is impractical that the ’Room Turtle
Generate’ component inputs must be changed manually. The components were made for adding data to
the database, but in the future this functionality could be optimised. The component could be made to
only accept one input, the room geometry, and automatically separate it into the necessary data points.
If more time was spent on database and ontology development, the automatic deduction of room types
could be implemented. This would mean that if a user had designed an ’L-shaped’ room, the tool could
automatically comprehend the geometrical shape, and label accordingly.
Some aspects of the tool have not been fully developed. An example of this is the context geometry and
shading factor, used when performing daylight simulations. At this stage, the context geometry can only be
represented by a speckle object. This means that the user cannot query for the size of context or shading,
Page 53 of 92
==PAGE:67==
and will have to receive a speckle object, and visualise it in order to investigate it. This is impractical,
and could be optimised by adding new tool functionalities and expanding the DSO ontology. Another
limitation is that only the ’and’ boolean operator was developed and implemented in the ’Join Query’
component. The ’and’ boolean operator will return the overlapping solution space. Other highly useful
boolean operators exist and it could be very valuable to add the ’or’ and ’not’ operators. The ’or’ operator
would return both solutions spaces, and ’not’ operator would exclude one. In future work, these boolean
operators could be implemented.
The tool and ontologies are interconnected. This means that when a change is made to an ontology the
tool needs to be updated and vice versa. Any of the proposed changes would result in additions to each
of the ontologies, and including other disciples would require the addition of new ones. Any user could
create an ontology further building on the RSOT, NVO or DSO ontologies. However, the user would
have to understand the descriptions of the classes and properties in the developed ontologies. If the user
misunderstands the application of the ontology and tries to develop it further, the resulting inference can
become illogical. This means that prior knowledge of semantic modelling is necessary. However, the
design tool can still be used without knowledge of linked data and ontology development. This was a
conscious decision, to allow for users that are new to semantic technology.
7.2
Future Possibilities
Previously, it has been shown that graph databases are capable of handling large amounts of data. In the fu-
ture, large amounts of data could be added to the database. Thereby, powerful predictive capabilities could
be implemented, naturally improving the tool the more it is used. This would enable the tool of delivering
diverse results for different geometries and simulation types. Therefore, more data in the database would
mean a more applicable tool. However, this tool has not yet been tested with large amounts of data. It is
possible that if the database was expanded, it could increase the response time.
Another possible expansion of the tool is that it could be made to include simulation results from other
disciplines. As one of the domain experts stated, the relationship between the parameters is valuable when
designing a new project. If the consequences of a simple alteration resulted in a chain reaction, the tool
would be able to show the resulting changes. Not many tools are capable of showing these relationships.
Therefore, it would be an incredibly powerful tool if more disciplines were added.
The tool was developed for design of room geometries. In the future, it would be possible to expand the
capabilities to whole buildings. This could be done by further inclusion of BOT (BOT: Building Topology
Ontology). Using the tool as it is now, it would be possible to model each room, and create a picture of
a full building design. However, to document compliance with the energy requirements from the Danish
Building Regulations, it would be necessary to simulate whole buildings and add the data to the database.
Page 54 of 92
==PAGE:68==
Linked data and semantic reasoning tools have great potential. In this project, only a small part of the
possibilities have been explored. In the future, it could be possible to use other tools in conjunction with
what has already been done. In the early development of the tool, it was planned to incorporate statistical
functionalities. For example, the plan was to add a function to ﬁnd the nearest simulation result to what
was being queried and to implement linear interpolation. This was not possible within the time frame of
the project, but in the future it could be implemented using powerful inference engines, e.g. RDFox. If
RDFox had been the graph database used, it would have been easier to implement more complex statistical
functionalities.
Genetic Algorithms work by combining a chosen solution with different solutions from the entire space,
generating completely new ’mutated’ designs. The implementation of Genetic Algorithms (GA) could also
be very valuable. As one of the domain experts said, "It can be hard to know where you are going, if you
don’t know where you are", referring to the solution space. This means that the application of GA could
allow for a much more diverse design exploration function.
The more users the database and tool have, the more data would be added, and the tool could potentially
form the basis of further studies and development of new building design and optimisation tools. In the
paper by Negendahl, Perkov, and Heller, 2014, it is discussed how sentient building simulation tools could
be developed. A step in this direction could be implementation of automatic simulation, if the result or
design is not yet in the database but is being queried. The tool is capable of capturing the data when it is
simulated. However, adding automatic simulation would make it capable of data generation. Another way
to generate large amounts of data, populating the database, could be to implement crowd-computing. If all
employees’ computers were running simulations when they are out of ofﬁce, they would continuously be
generating and adding data to the database. Assuming all employees work an average of 8 hours a day, the
computers could run simulations 16 hours a day, which, when multiplied with the amount of computers,
would over time result in a massive solution space. Using crowd-computing instead of cloud computing
could be an easy way to populate the database without sacriﬁcing data protection (as many companies wish
to keep their data in-house). Applying crowd computing would be time consuming, but the implementation
of this and interpolation algorithms could result in a building performance simulation tool of the future.

8
Conclusion
The aim of this thesis was to develop an ontology and tool to store simulation results and utilise the data
for design exploration and optimisation. The tool should aid the designer during the early design phase. By
carrying out discussions with domain experts, it was decided that the tool should be capable of delivering
fast and understandable results, which can guide the designer to make informed decisions and map the
relationship between performance and design.
The main goal of the ontology development was to store building simulation data in a meaningful way.
This meant that the ontologies had to be capable of describing room geometries and simulation results, but
also connect these in a way that would enable future users to add new types of simulation results. The
main ontology was the Room Simulation Ontology Topology (RSOT), which was designed to contain an
adequate description of the room geometry for simulation purposes. Two sub-ontologies were developed
for the indoor climate parameters: natural ventilation and daylight. The ontologies: Natural Ventilation
Ontology (NVO) and the Daylight Simulation Ontology (DSO) were made to contain boundary conditions
and simulation results for each subject. This hierarchical ontology structure was made to ensure future
ﬂexibility and to ensure that any potential user can add a new ontology and simulation results to the same
database. The validity of the data and full simulation set-up cannot yet be fully described by the ontologies.
However, if future work was to be carried out, new classes and properties could be added to better describe
this.
Based on the developed ontology, the ’Turtle’ tool was developed to utilise an open database on a DTU
server via Grasshopper. The tool allows any user (on a DTU network) to add results to the database, query
for results and visualise simulations previously carried out. It was shown that the database was capable of
handling large amounts of data and returning results quickly. The tool was made so that queries can be
created without knowledge of the SPARQL querying language. The tool was applied to multiple examples
of queries for indoor climate conditions, presenting the capabilities of the tool. These capabilities were
designed to be used during the early design phase, enabling the user to query for multiple performance
criteria, geometric features and visualisation of the simulation results throughout the room. The tool could
only query existing data. This was a considerable limitation, as the database was not fully populated. If fu-
ture work was to be carried out, the tool functionalities could be expanded to include prediction algorithms.
As the domain experts pointed out, this could result in a highly useful and valuable tool. By applying a tool
of this nature, future users could save a lot of effort, performing complicated and time-consuming simula-
tions by utilising and thus reusing simulation results previously generated. This would create a sustainable
workﬂow, whereby designers help each other by sharing data, exploring new designs and optimising per-
formance parameters. This study showed that linked data has a lot of potential for efﬁcient storage and
utilisation of building simulation results.

9.1
Natural Ventilation Study
All the natural ventilation data used in this thesis was carried out by the authors for their bachelor thesis
(Thermal Comfort in Naturally Ventilated Buildings). In total, 108 CFD simulations for different boundary
conditions, room geometries and orientations were run, generating a large data set. The CFD simulations
that were run in conjunction with the project were benchmarked and a grid independence study was carried
out. Besides this, the quality and convergence was ensured by monitoring the residuals. This section will
brieﬂy cover the simulation setup, benchmark simulation results and grid independence study.
9.1.1
Benchmark and Grid Independence Study
The benchmark study was carried out using the paper about single-sided ventilation by Papakonstantinou,
Kiranoudis, and Markatos, 2000. The paper was chosen as they made a comparison to experimental results,
as well as CFD simulations. The simulations run in the paper were performed using the steady-state
SIMPLE algorithm. Using OpenFOAM, the SIMPLE algorithm turned out to be too unstable. Therefore,
the PIMPLE algorithm was used. The geometry and probe points are shown in ﬁgure 9.1 and the results of
the simulation, along with the results from the paper are shown in ﬁgure 9.2.
Figure 9.1: An illustration of the geometry and probe points.
Page 61 of 92
==PAGE:75==
Figure 9.2: Results from benchmark study, showing the simulated and experimental results from the paper along with
the simulated results from the thesis.
The results of the benchmark showed that the simulations led to results closer to the experimental values
than those found by the authors of the paper. The details of the simulation set-up and benchmark study are
shown in the thesis.
Besides the benchmark study, a grid independence study was carried out. This was done to ensure that the
mesh was ﬁne enough to describe the airﬂow correctly, but not so ﬁne that the simulations would require
too much computational power.
Figure 9.3: Illustration showing the grid independence study. Velocity ﬁeld of the middle plane at a height of 1.1m
from the ﬂoor for decreasing cell sizes.
A grid independence study was carried out by reducing the cell size. In ﬁgure 9.3, the velocities of the
middle plane at a height of 1.1m for decreasing cell sizes are shown. The acceptable cell size was found to
be 0.1m.
The assumption of steady state was investigated. This was done by running the simulations to 3,500
iterations, which is equal to 350 seconds. The average accuracy was set to 0.01 m/s. It can be seen in ﬁgure

Figure 9.4: Result of the steady state investigation, with the threshold 0.01 m/s illustrated.

9.2
Daylight Study
9.2.1
Daylight Factor Benchmark Simulation
In order to validate the simulated daylight data, a benchmark study was carried out. This was done by using
the paper by Iversen et al. and comparing their results to the simulated results from Grasshopper, using
RADIANCE. The results showed that the tendency was identical. The results are shown in ﬁgure 9.6.
Figure 9.6: Benchmark study of the simulated daylight factor throughout the room.
9.2.2
Simulation Setup
The daylight simulations performed in this study were done in Grasshopper, using the Ladybug/Honeybee
plug-ins. The plugin connects to a Radiance engine which then performs the simulation based on the inputs.
The simulations were performed using a ray tracing technique, which supports reﬂection, transmission and
refraction properties of the surfaces (Iversen et al., 2013).
All daylight simulations were run with some ﬁxed parameters:
• The analysis surface was set to 0.5 metres inside the room (from the walls).
Page 64 of 92
==PAGE:78==
• No external obstacles were added to any of the simulations (no context geometries were added).
• The grid size was set to 0.25 metres.
• The analysis height was set to 0.85 metres.
• The simulations were run in the middle of a circular ground terrain with a radius of 100m.
• For the daylight factor simulations, the applied sky type was an overcast sky.
• For daylight autonomy simulations (annual), a weather data ﬁle (.epw) from 2011 for Copenhagen
was used.
9.3
Possible Extensions
In order for the data to be reproducible, new classes and properties should be added to the ontologies.
The values necessary for the data to be fully reproducible are listed below. For the DSO ontology only
the type of sky parameter is missing. However, for the NVO ontology only the boundary conditions were
included. The reproducability of the natural ventilation data could be ensured by adding the CFD technical
parameters to the ontology.
DSO:
• sky condition (e.g. uniform sky)
NVO:
• solver
• grid size
• cell count
• mesh type
• skewness
• residuals
