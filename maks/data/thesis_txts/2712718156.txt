Abstract
The thesis aims to explore the state-of-the-art range filters, compare them, and
propose new solutions focused on improving the false positive rate and reducing
space usage.
I focus on two problems: string range filtering for ranges of different sizes and
prefix filtering, which can be considered a subset of the string range filtering
problem. I conduct a series of experiments with various datasets and query sets
of varying distributions, which allow me to pinpoint weaknesses of the analysed
data structures and propose improvements.

Chapter 1
Introduction
Given a set of strings S, a range filter is a space-efficient probabilistic data
structure used to test whether there are any elements in S within the queried
range. The data structure guarantees one-sided error, i.e. false positive matches
are possible but false negatives are not. In other words, a range query can return
one out of two possible answers: "the range is empty" or "there may be elements
in the range". A range filter should consume less space than a non-probabilistic
(exact) data structure for range queries.
Given a set of strings S, a prefix filter is a space-efficient probabilistic data
structure used to test whether there are any strings in S starting with a provided
prefix p. Same as in the case of the range filters, prefix filters also guarantee
one-sided error. A prefix query returns either "there are no strings in S starting
with p" or "there may be strings in S starting with p". A prefix filter should
consume less space than a non-probabilistic (exact) data structure for prefix
queries.
Every prefix filtering query can be expressed as a range filtering query.
==PAGE:12==
2
Introduction
1.1
Range filters in LSM trees
Range filters have received increased attention in the research community be-
cause of their practical application as an auxiliary data structure in key-value
stores based on log-structured merge trees. [1]
Log-structured merge (LSM) trees [2] have been used as a core building block of
write-optimized storage engines, used widely in the industry ([3], [4], [5]). LSM
tree achieves fast ingestion time by maintaining multiple tree levels of increasing
sizes. The first level is usually stored in the main memory, and the rest of the
levels are on disk. [6] Each level contains sorted elements. When an element is
inserted, it is inserted only to the first level of the tree. Elements are flushed
together to the next level only once the level is filled in, limiting the number of
I/Os between main and external memory.
The downside of this design is a relatively expensive entry retrieval time, as an
element may reside in any tree level, depending on when it was inserted. In order
to limit the number of required I/Os, LSM-tree implementations commonly
use probabilistic, space-efficient data structures for approximate membership
queries to prune the search by filtering out some tree levels. A space-efficient
range filter could speed up the range query time in LSM trees in the same way
AMQ data structures speed up the point query time.
1.2
Related work
1.2.1
Approximate Membership Query data structures
An Approximate Membership Query (AMQ) data structure is a space-efficient
probabilistic data structure supporting membership tests that are allowed to
make mistakes in exchange for requiring smaller space than an exact member-
ship tester would require. [7] A range filter can be considered a data structure
analogous to an AMQ filter but supports range queries instead of point queries.
There exist multiple AMQ data structures; the thesis focuses on two of them:
the Bloom filter [8] and the quotient filter[9]. Chapter 3 describes both data
structures in detail.
==PAGE:13==
1.2 Related work
3
1.2.2
Prefix Bloom Filters
A common approach for speeding up range queries in LSM trees is using prefix
Bloom filters ([10], [11]). Prefix Bloom filter is a regular Bloom filter that
stores all the prefixes of the entries in the dataset or a subset of the prefixes of a
predefined length. The limitation of this approach is that it supports a limited
number of range queries, namely only such range queries that can be expressed
as a fixed-prefix query.
I evaluate the prefix Bloom filter in chapter 3.
1.2.3
Adaptive Range Filters
K. Alexiou et al. introduced Adaptive Range Filters (ARF) [12] - a binary tree-
based data structure for range filtering that requires training before it can be
used effectively; It adapts to the query and data distribution.
The leaves of ARF represent ranges and use an occupied bit to indicate whether
there is any key in the dataset that belongs to the range. The intermediate
nodes help to navigate the ARF to find the right leaves. ARF uses a space-
efficient breadth-first encoding which also supports efficient navigation through
the tree. ARF adapts dynamically to the data and queries; it will assign more
bits to the regions queried more frequently. The authors describe the operations
of splitting and merging nodes applied based on the adaptation policy that tries
to balance the space usage and the time required to adapt the tree.
1.2.4
Lower bound for the approximate range emptiness
problem
Goswami et al. [13] prove the lower bound for the approximate range filtering
problem. They show that any structure answering the problem with false pos-
itive rate ϵ will use asymptotically the same space as L Bloom filters with the
same FPR ϵ (for range of length L).
==PAGE:14==
4
Introduction
1.2.5
Succinct Range Filter (SuRF)
Succinct Range Filter (SuRF) is a static data structure supporting approxi-
mate point membership queries, approximate range queries, and approximate
range counts. [1]. It is based on Fast Succinct Trie (FST) data structure - a
static space-efficient trie that uses bitmap-based encoding schemes. The per-
formance of SuRF is workload-dependent. Authors evaluate SuRF in empirical
experiments. They show that SuRF improved the range queries performance
of RocksDB - an in-memory key-value store based on log-structured merge tree
data structure - while causing only a small increase in FPR rate on point queries
compared with vanilla RocksDB, which uses Bloom filters.
Chapter 4 is fully dedicated to the SuRF and FST data structures. I use FST
as a baseline for benchmarking range filters. I run extensive experiments on
multiple input and query workloads to assess different variants of SuRF proposed
in the original paper. I also propose and benchmark new data structures based
on SuRF.
1.2.6
Rosetta
Rosetta [14] is a probabilistic range filter consisting of a set of Bloom filters,
one per binary prefixes of some length. The range queries are broken down into
dyadic intervals; this way, maximum log2 L Bloom filters need to be queried for
a range of length L.
Chapter 5 focuses on Rosetta and provides a more detailed description of the
data structure. It includes the description of my implementation of Rosetta and
the experimental evaluation of its performance. I also propose modifications to
the original data structure’s design.
1.3
Aim and contributions of the thesis
The goal of the project was to (1) benchmark the state-of-the-art data structures
for range query filtering and prefix filtering and (2) propose new and improved
range filter and prefix filter data structures and verify them empirically.
The thesis makes the following contributions:

1.4
Implementations and external libraries
As Goswami et al. [13] showed, the theoretical worst-case space bound on any
range filter is close to the space required by the original data. The authors of
the two most popular state-of-the-art range filters, SuRF and Rosetta resort to
empirical experiments to show that their data structures achieve good practical
results, despite the rather unimpressive theoretical space complexity. I follow
their approach in this thesis and rely strongly on comprehensive empirical ex-
periments to evaluate the discussed data structures.
Because of that, I needed an implementation of every evaluated data structure.
Some of the code is of my authorship, but I also rely on the following open-source
libraries and projects:
• SuRF source code [15] - official implementation of SuRF published by the
authors of the data structure,
• Quotient Filter [16] - a quotient filter implementation written in C by
Vedant Kumar,
• MurmurHash3 [17] - a non-cryptographic hash function implemented by
Austin Appleby.

Chapter 2
Testing framework for
string range filtering
This chapter details the testing method I use in prefix and range filtering ex-
periments to benchmark the state-of-the-art filters and their modifications.
All the existing approaches to the range filtering problem depend on the dataset
and query set distribution. The Adaptive Range Filter [12], as the name it-
self suggests, adapts over time to the query set distribution.
SuRF [1] and
Rosetta [14], in turn, focus on concrete types of queries. SuRF is designed to
effectively filter out long empty ranges and to disregard entirely short ranges.
Rosetta is supposed to be more effective in filtering short ranges, but by design,
it is not suitable for datasets of strings with lengths with big variance. A uni-
versal range filter does not exist, but we can consider the tradeoffs and select
the best filter for a given task.
The papers on two data structures I focus on most in this thesis - SuRF and
Rosetta - base their results on empirical evaluations, and the datasets that the
authors chose have some particular properties.
Both use the Yahoo!
Cloud
Serving Benchmark (YCBS) [18] to generate the workloads consisting of 64-bit
integers (both use uniform and normal distribution, SuRF additionally runs tests
against the zipfian distribution).
Notably, these workloads consist of entries
of equal lengths, and if we treat them as strings (as SuRF does), the size of

the alphabet is relatively small. SuRF additionally runs experiments on one
more workload generated with YCBS: host-reversed email addresses of varying
lengths (22 bytes on average, max 129 bytes). Rosetta uses a different dataset
for testing strings range filtering capabilities: a randomly selected subset from
the Wikipedia Extraction (WEX) [19] dataset (consisting of a processed dump
of English language in Wikipedia); the word length variance is not provided in
the paper.
According to the SURF’s code source [15], the queried ranges were all of the
same length: 128 for the email dataset and 92233697311 for the integer dataset.
However, in the case of Rosetta, the queried range lengths vary from 1 to 64 for
the integer dataset and are of a fixed length (128) for the WEX-based dataset.
This shows that the results discussed in the papers are based on experiments of
a limited extent; there may be some types of data and query distributions for
which none of the two data structures will give a satisfactory result. It cannot
be excluded that the results of the comparative experiments between Rosetta
and SuRF would be different if the workloads had different properties.
One of the project’s objectives was to design a comprehensive testing framework
to systematically evaluate the performance of range filters in different setups and
underpin their strengths and weaknesses. I use these findings to design different
modifications to the data structures.
In this chapter, I describe how I generated workloads and queries used in the
prefix and range filtering experiments and describe their properties.
2.1
Types of datasets
The datasets and queries used in all the experiments are randomly generated
according to the following parameters:
• number of words to be generated,
• size of the alphabet,
• the average length µ and the standard deviation σ2 in lengths of the
generated words,
• letter distribution which takes one of the values: uniform, normal, power-
law, and different last letter.
==PAGE:19==
2.1 Types of datasets
9
Dataset generation method
The generation of words can be seen as build-
ing a trie in a depth-first order starting from the root. At each node, a recursive
procedure is called. For letters distributions uniform, normal, and powerlaw,
given the number of children that the node will have n, the alphabet, and the
letter distribution, n letters are chosen out of the shuffled alphabet according
to the letter distribution. If a letter has been chosen k times, k words have the
letter appended to the current prefix, and the recursive method is called with
parameter n = k and a new prefix. Additionally, each node corresponding to a
prefix of length p is marked as the end of the word with probability cdf(p), where
cdf is the cumulative distribution function of the normal distribution N(µ, σ2);
in such a case, n −1 letters are chosen out of shuffled alphabet as next letters.
At the root of the tree, the procedure is called with n being the number of words
to be generated.
In the case of the letter distribution different last letter, n/2 words are generated
according to the normal distribution. Then each word is copied between 1 and
4 times (chosen uniformly at random), and a random letter from the alphabet
is appended to each copy.
Dataset intuition
The selection of parameters helps generate datasets with
different characteristics such as tree shape, the amount and lengths of common
prefixes, and location of chains of nodes with only one child. Both uniform and
normal letter distributions will result in rather symmetrical trees with chains
of nodes with only one child (I will be referring to them as one-child-nodes for
brevity) only at the bottom of the tree and containing leaves. Normal will have
more common prefixes than uniform.
The powerlaw letter distribution type
will result in asymmetrical trees with a higher probability of one-child-node
chains appearing higher up the tree than in the case of normal and uniform
distributions. However, most of the one-child-node chains will be at the bottom
of the tree with leaves. In the last letter distribution, the chains of one-child-
nodes will not include leaves in most cases.
These characteristics can be observed in figures 2.1, 2.2, 2.3 and 2.4 which show
trees of generated datasets with different letter distributions consisting of 15
words of average length 10 and standard deviation of length 2, built from the
alphabet of size 26.

based on AMQs
A common industry approach to range queries is to maintain a prefix Bloom
filter - a Bloom filter with all (or a subset of) prefixes of the strings in the
dataset. [1] This chapter investigates the performance of prefix filters based
on Bloom filters and another approximate membership query data structure -
quotient filter. I also explore the possibilities to improve the filters’ performance
by taking advantage of some of the properties of string prefixes.
3.1
Baseline: prefix and range Bloom filter
3.1.1
Bloom filter
The Bloom filter is a space-efficient AMQ data structure that does allow false
positives but not false negatives. [8]
We are given a set with n elements from the universe U.
A Bloom filter is
a bit vector B of m bits; all bits are initially unset.
It uses k independent
==PAGE:26==
16

hash functions h1, h2, . . . , hk which hash every key in U to R = 0, 1, . . . , m −1
uniformly at random.
When an element x is inserted to the Bloom filter, we set the bits B[h1(x)],
B[h2(x)], . . ., B[hk(x)].
When an element x is queried, we check the values of bits B[h1(x)], B[h2(x)], . . .,
B[hk(x)]. If all of them are set, the answer to the query is positive; otherwise,
the answer is negative.
The probability of a false positive in a Bloom filter is (1 −e−kn
m )k. The optimal
number of hash functions minimizing the probability of a false positive is k =
(ln 2) · m
n .
3.1.2
Prefix and range Bloom filter
The data structure I use as a baseline in prefix filtering experiments is the prefix
Bloom filter: a Bloom filter accepting parameters m and words, where m is the
size of the bitvector, and words is a list of strings whose prefixes are to be
stored in the Bloom filter. A set prefixes is generated from words. Given m
and n = |prefixes|, the number of hash functions k is calculated: k = m
n ln 2.
Querying a prefix is straightforward; it requires only a lookup in the Bloom
filter.
A prefix Bloom filter can be easily extended to a range Bloom filter; the data
structure and the insertion operation remain unchanged. The only difference is
the range query operation which, given a query [leftKey, rightKey], will query
all the elements in the range and return positive if at least one of these queries
is positive.
3.1.3
Implementation
I use my implementation of a Bloom filter in experiments. It uses MurmurHash [17]
for hashing the elements from the dataset and queries.

3.2
Replacing Bloom filter with quotient filter
This section investigates a range filter based on another AMQ data structure
- the quotient filter (QF). I start with the description of the data structure
and its base operations. Next, I describe a small experiment I run to decide
on the values of the parameters of the filter. Then, I provide the results of the
experiments comparing the performance of the prefix Bloom filter and the prefix
quotient filter. Finally, to make understanding the obtained results easier, I run
another experiment focused on demonstrating the root cause of the observed
behaviour.
3.2.1
Quotient filter
A quotient filter [9] is an AMQ data structure based on a compact hash table
that allows elements to overflow to the neighbouring slots instead of requiring
an auxiliary data structure to store the conflicting entries. Each slot stores a
part of the key and 3 metadata bits that allow identifying for each element its
canonical slot - a slot to which the element would be inserted if a conflicting
entry did not already occupy it.
Overview
A quotient filter is initialized with parameters q and r. It creates
a hash table with 2q slots, each slot of size r + 3. Initially, all bits are unset.
When an element x is inserted into the quotient filter, a hash function generates
a fingerprint of length q + r. First q bits (quotient) denote the number of the
slot in the hash table where the element should be stored, and the remaining
r bits (remainder) is the part of the key that should be stored in the slot.
If there is a soft collision, that is, another element with a different remainder
already occupies a slot, then the remainder will be stored in some slot to the
right. In such a situation, we say that a run was created: run is a sequence of
neighbouring slots in the compact hash table that store elements with the same
quotient, so belonging to the same canonical slot.
Two or more runs may be contiguous; we talk about a cluster of runs in such a
situation. It is also important to notice that if a cluster with at least two runs
exists and we insert a new element that should become part of a non-last run of
the cluster, the remaining runs will have to be shifted to the right. This implies
that there is no guarantee that the first element of the run is in its canonical
slot.
==PAGE:28==
18
Prefix and range filters based on AMQs
In order to be able to both insert and query elements in the quotient filter, we
need to determine which bit belongs to which run and cluster. This is what the
3 metadata bits per slot serve for:
• occupied bit is set when there exists an entry in the quotient filter for
which this bit is its canonical slot (even if its remainder was shifted to the
right),
• continuation bit is set when the remainder in the slot belongs to a run
but is not the first element of the run,
• shifted bit is set when the remainder in the slot has been shifted from its
canonical slot.
Finding the element’s run
Both insert and query operations will often
require finding the element’s run. In order to do so, first, we need to perform a
scan to the left until we find an element whose shifted bit is unset - this way,
we will find the beginning of the cluster. Now, we need to scan to the right to
find the correct run. We can do so by gradually increasing two pointers, both
initially pointing to the beginning of the cluster. First, we increase the first
pointer until we find a slot whose continuation bit is unset - this effectively
means skipping all the elements from the current run. Then, we increase the
second pointer until we find a slot whose occupied bit is set. We repeat these
two operations until the second pointer points to the slot with the same quotient
as the inserted/queried element. When we reach this point, the first pointer will
point to the beginning of the correct run.
Inserting an element
When we insert an element x into the quotient filter,
we generate its fingerprint and find the slot denoted by the first q bits of the
fingerprint. We set the occupied bit of the slot. If the slot is empty, we can
insert the remainder in the slot and finish the operation.
Otherwise, we need to find the x’s run according to the algorithm described
above. Then, we continue to scan the run right in order to find the correct slot
to insert the element so that the elements in the run remain sorted. We scan
right as long as the value in the slot is smaller than the remainder of x. If the
value in the current slot equals the remainder of x, the element must have been
already inserted. Otherwise, we need to insert x at the current slot and shift
the remaining contiguous values to the right. When shifting, the value of the
occupied bit remains as it was before the shift, the continuation bit is shifted
together with the value, and the shifted bit is always set.
==PAGE:29==

Querying an element
The algorithm to query an element x is similar to the
insertion algorithm. First, we check the slot pointed by the x’s quotient: if its
occupied bit is unset, the element certainly is not in the filter.
Otherwise, we find the x’s run according to the algorithm described above. Then
we scan the slots in the run until we encounter the slot with a value equal to
the remainder of x (we return positive in this case) or with a value greater than
the remainder (in this case, the answer is negative). If we reach the end of the
run, we also return negative.

generate a set of all the prefixes from words, I hash them using MurmurHash [17],
and I insert the hashes into the filter.
3.2.3
Choosing saturation: cluster size and query time
The Bloom filter’s bit array size m can be any natural number, and m achieving
a false positive rate δ is proportional to the size of dataset n. However, the
size of the quotient filter’s array bit is a function of positive integers q and r:
2q(r + 3) and, as such, cannot be as easily adjusted to n.
It would make sense to try to keep the coefficient q as low as possible to keep the
space usage low. Nevertheless, it is also important to note that the percentage
of the data structure being filled in, which I refer to as saturation (α =
n

strongly influences the insertion and query time. This is due to the lookup and
insertion times being dominated by a scan through a cluster - a sequence of
consecutive not empty slots. The original paper’s authors [9] suggest that a
quotient filter should not be saturated in more than 75% to keep the insertion
and query time reasonable. They provide empirical experiments to demonstrate
that after crossing this threshold, the quotient filter’s performance deteriorates
rapidly. However, this means that with n = (3/4 + ϵ)m, the filter size must
double. I run an experiment with q = 20 and n ∈[26214, 996148] for different
values of α and r to investigate if this threshold could be increased in order to
keep space usage as low as possible while maintaining reasonable query time.

3.3
Extensions of prefix filters
In this section, I propose and evaluate various extensions and modifications of
the AMQ-based prefix and range filters, intending to take advantage of some of
the properties of the string prefixes.
3.3.1
Doubting
Doubting is a process of making additional queries in the filter in order to exclude
false positives. The term was introduced in the Rosetta paper [14].
A prefix filter can be extended with a process of doubting by taking advantage of
a very simple property: if a prefix s = s1s2s3 . . . sn is found in the Bloom filter,
all the prefixes of it (s1, s1s2, . . . , s1s2 . . . sn−1) also must be found in the Bloom
filter. Otherwise, it is a false positive. In order to keep the query time low, we
may introduce a coefficient maximum doubting level d: if d provided, maximum d
prefixes of a prefix s will be queries: s1s2 . . . sn−d, s1s2 . . . sn−(d−1), . . . , s1s2 . . . sn−1.
Expectedly, this extension will not help in the case of queries such as those from
query sets common prefix and last letter - those for which all their prefixes are
prefixes of some of the elements in the dataset, but they are not.

3.3.2
Multi-Prefix Bloom filter
The prefix Bloom filter is simply a single Bloom filter with all the prefixes of the
strings in the dataset. An alternative approach is to have a set of n Bloom filters,
where n is the length of the longest string in the dataset. The i-th Bloom filter
contains all the prefixes of size i. The size of a single Bloom filter is proportional
to the number of elements it contains. The performance of a Multi-Prefix Bloom
filter was not immediately apparent to me: on the one hand, by decreasing the
size of a Bloom filter, I increase the probability of false positives within a filter,
but on the other, two prefixes of different lengths that would collide in PrefixBF,
will not collide in the Multi-PrefixBF because they are in separate filters.

3.3.3
Multi-Prefix filter with level-dependant penalty
The advantage of the Multi-PrefixBF data structure is greater flexibility in
adjusting the filters’ sizes. SuRF is based on the idea that the beginnings of
words are more important than their endings for range filtering, and this is
why the authors decided to cull the bottom parts of a trie. The same idea can
be applied to the MultiPrefixBF data structure by assigning a bigger memory
budget to filters with shorter prefixes. I introduce a coefficient p, which is a
maximum penalty that can be imposed on a filter. Given that the Multi-Prefix
BF will comprise n filters in total and the regular memory budget m split across
filters proportionally to the number of elements in the filter, the size of a single
filter for prefixes of length i is (1 −p(i/n))m.

gain in using the filter with penalties for the random query type. These kinds
of queries share very short or no prefixes with elements in the dataset, so a
decrease in the sizes of filters corresponding to the longer prefixes does not
influence the filter’s performance since they are rarely queried. On the other
hand, expectedly, the FPR/space tradeoff for queries of type last letter is better
if the penalty is not applied - since for these kinds of queries the filters which
correspond to the longer prefixes are queried more often.

3.3.4
Range Bloom filter with Karp-Rabin fingerprint
As described above, I use the Murmur hash [17] for hashing the prefixes in the
prefix and range Bloom filter. That means that for querying a range of size x,
kx prefixes need to be hashed and the corresponding kx bits are queried in the
Bloom filter.
In this section, I consider replacing the Murmur hash with the Karp-Rabin
fingerprint.
One of the properties of the Karp-Rabin fingerprint is that the
hashes of two neighboring strings (that is, strings of equal length that differ by
only the last letters, and these letters differ by 1 in ASCII representation) will
differ by 1. This means that for querying elements in a range, we would query
k neighboring bits, potentially resulting in better data locality.
Another property of the Karp-Rabin fingerprint is that given the hash of a sub-
string s[a..b], calculating a hash of a substring s[a..b+1] is a single constant-time
operation. This property could be used when calculating hashes of neighboring
strings belonging to a range.
Overall, the idea was that if the Karp-Rabin fingerprint proved to be a good
hashing function, the query time could be improved while maintaining the same
FPR.
Unfortunately, the experiments showed that FPR has increased when using the
Karp-Rabin fingerprint. Figure 3.7 shows results of the experiment compar-
ing the prefix Bloom filter with MurmurHash3 (RangeBF) and the Karp-Rabin
fingerprint (RangeBF-KR). Following the results, I will continue using Mur-
murHash3 as the hashing function in the subsequent experiments.

SuRF and modifications
In this chapter, I am taking a closer look at Succinct Range Filter (SuRF) data
structure [1]. I start by describing the Fast Succinct Trie (FST) - a data struc-
ture underlying SuRF, which I am using as a baseline for comparing other filter’s
FPR/space usage tradeoff. Next, I benchmark different variants of SuRF pro-
posed in the original paper to discover their advantages and shortcomings. Then,
I propose new data structures - Splash and Bloomed FST - which are based on
SuRF but include modifications addressing some of the discovered shortcom-
ings. Finally, I run experiments investigating comprehensively the performance
of all prefix and range filters introduced until now.
4.1
Fast Succinct Trie
Fast Succinct Trie (FST) is a space-efficient trie representation underlying SuRF.
I use it as a baseline data structure for benchmarking prefix and range filters
because of its small size. In particular, it comes in handy in assessing the filter’s
size. It provides a practical upper size limit for the static range and prefix filters
since the ones with a size greater than FST could be replaced by FST, which
always yields 100% accurate results.

4.1.1
Description of the data structure
Fast Succinct Trie (FST) uses two encoding schemes to encode the upper and
bottom levels of a trie, both of them being variations of the Level-Ordered Unary
Degree Sequence (LOUDS) scheme introduced by Jacobson [20].
LOUDS
LOUDS represents an ordinal trie by reporting the node’s degrees in
breadth-first order. Such a sequence of integers uniquely identifies the trie. [20]
Each integer is encoded using the unary code (the integer d is represented by
the string 1d0), and then all integers in the sequence are concatenated.
A significant advantage of the LOUDS representation is the ease with which
the tree can be traversed (including operations such as accessing the sibling,
children, and parent nodes and counting the children). All the operations are
based on atomic operations rank and select:
• rank0/1(m) operation counts the number of unset/set bits in the first m
bits of the bitmap,
• select0/1(m) operation finds the position of the mth unset/set bit.
For example, the position of the first child of the node with starting position p
can be found by running operations select0(rank1(p)) + 1. Efficient select and
rank operations have been a research subject ([21], [22], [23]), and there are
implementations using look-up tables with precomputed results that achieve
constant time.
LOUDS example
Figure 4.1 shows an example trie with its LOUDS encod-
ing. In order to find a position of a child node of node a (node a starts at
position 8), we need to calculate select0(rank1(8)) + 1. The number of set bits
until position 8 is 6. The position of the 6-th unset bit in the encoding is 21.
Therefore, the position of the child node of c should be 22; it is correct, as at
position 22, the representation of the node a starts, and a is indeed the first
child node of c.
LOUDS-DS
FST uses a LOUDS-DS encoding scheme consisting of a query
time-optimized LOUDS-Dense scheme and a space-optimized LOUDS-Sparse
scheme. The design follows an observation that upper levels consist of only a
few nodes that are accessed very often. In contrast, the bottom levels contain
most nodes in the trie, but they are accessed less often.

LOUDS-Dense
LOUDS-Dense allows for a very fast child node search (one
array lookup) at the price of increased space consumption.
Each node in
LOUDS-Dense is encoded using three 256-bit vectors. The first one sets the
bits corresponding to the ASCII codes of the labels of the edges outcoming from
the node. The second indicates if the child node is a leaf, and the third one
indicates if the prefix leading to the node is also a valid key.
LOUDS-Sparse
LOUDS-Sparse is very space-efficient but requires a longer
query time. It consists of a byte sequence comprising the labels of the nodes in
the breadth-first order, a bit-sequence indicating whether a node is a leaf, and
finally, the classical LOUDS bit-sequence.
4.1.2
Implementation
In order to be able to use the FST data structure as a baseline in benchmarks,
I modified the code source of SuRF [1] to obtain an implementation of FST.
In particular, I reused the LOUDS-Sparse and LOUDS-Dense implementation
and modified the parts of the code responsible for selecting which parts of the
keys to include in the data structure.
4.2
SuRF
SuRF is an FST without the lowest levels of the trie. The idea is to reduce the
size of the trie by removing the parts corresponding only to single keys, so the
ones which can be considered less significant for keeping the FPR rate of the
range filter low. The original paper [1] introduces four variations of SuRF:
==PAGE:42==
32
SuRF and modifications
• SuRF-Base: the base variant is simply an FST without the one-child-node
branches,
• SuRF-Hash: This variant of SuRF additionally stores a few bits of a hash
of the entire key per each key leaf. The goal is to improve FPR for point
queries,
• SuRF-Real: This variant stores extra bits per leaf - thr first bits of the
suffix corresponding to the culled chain of nodes. This variant’s objective
is to improve FPR for range queries,
• SuRF-Mixed: This variant includes a combination of hashed and real bits
suffixes. Its objective is to improve both point and range query perfor-
mance. The amount of bits assigned to each type of suffix is configurable.
4.3
Benchmark of SuRF variants
SuRF, as described in the original paper, comes with two types of parameters:
• the type of suffix and its length,
• the number of levels represented by LOUDS Sparse and LOUDS Dense
bitmaps.
The purpose of the following experiments was to evaluate different variants of
SuRF in the context of range filtering.

4.4
Splash: parameterized SuRF
While SuRF has been proved to have practical use cases, I believe some of its
properties can be considered suboptimal.
First of all, SuRF’s performance depends on the data and query set distribu-
tion, as the SuRF’s authors point out in paper [1] and as I confirmed in the
experiments described above (section 4.2). One of the implications is that for a
data distribution with very short branches of nodes with a single child, the size
of SuRF will be very close to the size of the exact data structure. While the
opposite situation (a trie with very long branches of single-child nodes which
will result in SuRF of small size but with considerable FPR) can be remediated
by increasing the size of the real suffix, in this case SuRF does not provide any
variants or parameters that would decrease the filter’s size.
Second of all, the parameterized lengths of suffixes provide limited control over
the memory/FPR tradeoff. As I already pointed out, there is a certain lower
limit of the filter’s size determined by the data distribution. Moreover, increas-
ing the real suffix length does not take into consideration the lenghts of the
branches.
Such a situation could occur if the variance of the lengths of the
strings in the dataset is considerable. In such a case, some of the suffixes could
end up being empty, occupying extra space while not contributing to reducing
the FPR. Arguably, a more straightforward way to control the memory/FPR
tradeoff independent from the dataset and query distribution would be an asset.
Considering the above points, I designed a new data structure based on SuRF:
Splash.
4.4.1
Data structure description
A Splash is a SuRF with stronger parameterization allowing for greater flexibility
in adjusting the FPR/memory usage tradeoff. The parameters are as follows:
• Cut-off coefficient: ratio of node’s height to the size of the subtree the

node is root of. The nodes are culled if cut-off coefficient is greater or
equal to γ. Effectively, SuRF is Splash with γ = 1,
• Restraint type and parameters:
– Absolute restraint type: given parameter i, a suffix is removed if the
height of the node is no longer than i,
– Relative restraint type: given a parameter d, a suffix is removed if the
ratio of the length of the suffix to the length of the longest root-leaf
path passing through the node does not exceed d.
Figure 4.4 visualises differences between different configurations of Splash. The
nodes in the edges represented in grey are the removed parts of the tries. Each
node is annotated with its height, the size of its subtree, and the ratio of these
two.
Figure 4.4a shows which parts of the trie would be removed in the case of SuRF
Base. It is noteworthy that only the three branches on the right are removed;
the whole subtree on the left remains in SuRF because of the forks at the bottom
of the trie. This is not the case for Splash with the cut-off coefficient 0.8 without
any restraints. Figure 4.4b shows that the left subtree is removed regardless of
the forks in the bottom. Finally, figures 4.4c and 4.4d show how the absolute
and relative restraints of different values allow adjusting the number of removed
nodes and finding the middle ground between the two extremes.
Such a set of parameters allows controlling the FPR/memory tradeoff easily,
regardless of the dataset distribution. However, Splash’s construction time is
longer than SuRF’s (even though it remains O(n)): it requires an extra pass
through the dataset to gather the dataset statistics - the height of every node
and the size of the subtree rooted in it.
4.4.2
Implementation
Same as in the case of FST, to obtain an implementation of Splash, I modified
the code source of SuRF [1]. In particular, I reused the LOUDS-Sparse and
LOUDS-Dense implementation and modified the parts of the code responsible
for selecting which parts of the keys should be included in the data structure.

4.4.3
Experiment: practical influence of the cut-off coef-
ficient
The purpose of the following experiment was to evaluate the influence of the cut-
off coefficient on the FPR/memory usage tradeoff of Splash. Figure 4.5 shows
the results of the experiments for the relative and absolute types of restraints.
Cut-off coefficient
In both cases, the choice of the cut-off coefficient for
letter distributions uniform, normal, and powerlaw proved to have a marginal
significance. The difference is significant in the case of the last letter different
distribution where the size of SuRF is very close to the size of FST due to SuRF
not removing branches ending with a fork at the very bottom. In the edge case,
filter size may be same as the size of the exact data structure, missing the point
of approximate data structure. The choice of the cut-off coefficient smaller than
1.0 helps avoid such a situation. In the case of "tricky" queries last letter and
similar, the FPR increases, but for the random queries, the FPR stays very low
while saving a much space.
In the following experiments, I will be using Splash with a cut-off coefficient
fixed to the value of 0.7.
Types of restraints
The experiments show that the shape of the curve rep-
resenting the FPR/memory usage tradeoff depends mainly on the query type,
and different types of restraints proved to yield better results for different query
types.
For the random query type (the types of queries which will have rather shorter
common prefixes with the entries in the input dataset), the absolute type of
restraint is a clear winner - the area under the curve is very small. In turn,
in the case of the last letter query type, it is the relative type of restraint that
yields a much better FPR/memory tradeoff.
4.5
Bloomed FST
I propose one more data structure derived from SuRF: Bloomed FST. Bloomed
FST combines FST with the prefix Bloom filter. The top part of a trie repre-
senting a dataset is stored in FST (with the number of FST levels being param-
eterized). The remaining levels are stored in the prefix Bloom filter.

Data structure description
Bloomed Splash takes an integer parameter h. It is composed of a Fast Succinct
Trie of height h and a prefix Bloom filter for all the prefixes that did not fit
into the succinct trie. Prefix Bloom filter can be replaced with another variant
(e.g. Multi-Level Bloom Filter), but the experiments were run only on the base
variant.

Chapter 5
Rosetta-based filter
Rosetta [14] (Robust Space-Time Optimized Range Filter for Key-Value Stores)
is a probabilistic range filter comprising a set of Bloom filters that store all the
binary prefixes of strings in the dataset. In this chapter, I describe the data
structure, present my implementation of a filter based on the base ideas behind
Rosetta, and run experiments comparing its performance with other filters. I
also describe a modification to the data structure from the original paper, which
aims to improve the filter’s performance for small alphabets.
5.1
Description of the data structure
Rosetta is a static data structure specifically designed for LSM-tree-based key-
value stores. Given a dataset, it will create l Bloom filters of different sizes,
where l is the length of the longest string in the dataset. An i-th Bloom filter
stores all unique binary prefixes of length i of the strings in the dataset, making
Rosetta form an implicit segment tree.
Rosetta answers the range queries by breaking them down into dyadic intervals.
Each dyadic interval corresponds to a subtree of the segment tree which com-
prises all the keys within this interval. A dyadic interval of size x will contain
==PAGE:58==
48
Rosetta-based filter
entries with a common prefix of length 2x, so it is enough to query Rosetta for
this prefix to verify if Rosetta does not contain any entry within the interval.
This way, maximum log2 L Bloom filters need to be queried for a range of length
L.
Suppose at least one of the queries corresponding to the dyadic intervals returned
yes. In that case, Rosetta starts the process of doubting this interval, which
means querying all the entries with the prefix x at the lower levels of the segment
tree. If no entry was found at a level, the positive at the higher level must have
been false.
The key to Rosetta’s small space consumption is a number of optimizations
based on the workload and past queries. Several techniques are used, including
adjusting the size of a Bloom filter at a level depending on the number of
probes to this level in the past queries. This implies that Rosetta also maintains
statistics about the previous filter usage.
5.2
Implementation
The authors of Rosetta did not publish the source code. Therefore, I imple-
mented a basic version of Rosetta, which follows the core design, but differs in
some aspects, e.g. excludes the space consumption optimizations.
Building the data structure
I start by making a single traversal through
all the strings in the dataset to determine the length of the longest string l and
count the number of unique prefixes of length i for 1 ≤i ≤l. I then create l
Bloom filters, with the size of the i-th Bloom filter proportional to the number
of prefixes of length i.
I introduce an additional parameter top level penalty which decreases the size of
all the layers except the bottom one. The number of bits for each level except
the last one is calculated as described above and additionally multiplicated by
1−top level penalty.
I use MurmurHash3 [17] for hashing the prefixes, and I store the hashes in the
corresponding Bloom filters.
Range query
The range query algorithm closely follows algorithm 2 in the
paper [14].

Memory allocation across levels
While my implementation follows the
main idea behind Rosetta, it varies in how the memory is allocated across levels,
which may have an impact on the data structure’s performance. The discrepan-
cies between my implementation of Rosetta and what is described in the paper
are as follows:
• The size of non-bottom levels is parameterized. The authors allo-
cate less memory to the non-bottom levels, following the reasoning that
the most accessed levels should be assigned more memory. However, they
achieve it by setting the non-terminal node’s FPR to a fixed value rather
than by parametrization, as I do it. As explained above, the size of the
non-bottom levels in my version of Rosetta can be adjusted by a coefficient
top level penalty.
• I do not adjust the levels’ sizes based on histograms and counts
of past queries. The original Rosetta distributes the bits across levels
based on workload information and past query statistics. This optimiza-
tion makes sense for Rosetta as a data structure designed specifically for
LSM trees: new Rosetta is built for every new run of the tree, so the past
query statistics can be used at each rebuild. The thesis focuses on entirely
static data structures that are not supposed to be adaptive; hence I did
not consider this optimisation relevant.

datasets may be using considerably less than 255 characters. For example, a

digits, and a few extra characters such as [@] or [.]. In this case, some levels
would be simply redundant since the highest bits will always be 0.

5.4.1
Rosetta performance for strings of varying and equal

Rosetta by design should work better for strings of equal lengths
When we query an element x in Rosetta, first we intent to query the levels
as high as possible (the ones corresponding to the dyadic intervals) to be able
to prune the search; however, once the search in high levels returns positive,
we proceed to query lower and lower levels in the process of doubting, until we
reach the lowest one. A consequence of such a design is that all the strings
must be right padded to have equal lengths before being inserted into Rosetta.
Otherwise, doubting on level length(x)+1 would return negative and incorrectly
determine that the positive on the higher level was false.
Right padding the strings means that we increase the number of the prefixes
to be stored in Rosetta. In some cases the prefixes generated by padding may
make up a very significant fraction of all the prefixes. It is easy to imagine an
extreme example of a dataset where only one string is significantly longer than
the others; all the others would have to be padded to have their lengths aligned.
Notably, all the experiments in the Rosetta paper [14] are run for strings with
equal lengths.
Experiment motivation

datasets of strings of equal lengths. I selected a dataset with 100000 words of
the normal dataset distribution type. To test the first case, I ran the experiment
on this dataset as is; to test the second case, I right-padded all the strings in
the dataset so that they all have equal lengths.

Chapter 6
CHaREQ
In this chapter, I propose a new range filter - CHaREQ (Compact Hash Table
for Range Emptiness Queries). CHaREQ is built on top of a similar type of
hash table as a quotient filter and uses the rank primitive to speed up the query
time.
The experiments show that CHaREQ may be considered an alternative to SuRF
for some types of data and query workloads.
6.1

such that sl ≤c2,j ≤sr. In such a case the answer of the exact range query
[p1·sl, p1·sr] will be no, but the answer of the approximate range query with the
same parameters will be yes. However, if the answer to an approximate range
query is no, then we have 100% certainty that the answer to the exact range
query with the same parameters also is no. This behaviour is consistent with
the base principle of approximate data structures - allowing for only one-sided
errors.

• a sorted array of all the first characters in the dataset,

Compact hash table
The compact hash table is very similar to the hash
table underlying the quotient filter. Conceptually, it consists of n slots, each
slot comprising r + 3 bits, including metadata: is occupied bit, is continuation
bit and is shifted bit. The significance of the metadata bits is identical as in the
quotient filter; the difference is that CHaREQ does not rely on quotients and
remainders of hash values. Given a string s = p · c where c is a single character:
• the number of the slot in the hash table where s is to be stored is calculated
as hash(p) % n,
• the r bits of the slot are set to value encode(c) where function encode
assigns a unique code to the character according to the scheme described
in the chapter 5.
A collision (hash(p1)%n = hash(p2)%n for p1 ̸= p2) can be understood as
merging two nodes n1 and n2 such that prefix(n1) = p1 and prefix(n2) = p2,
just as described above.
Figure 6.2 shows an example of a trie and how it would be encoded in a compact
hash table given the hash values for the prefixes in the table next to the trie.
There is a collision for prefixes ca and d, and as such we can see that the union
of their children (n, o, p, r, t) is a single run starting at index 7.
Querying the compact hash table
The compact hash table supports point
queries and range queries for keys of the same length differing only by the last
character. A key q is split into p · c where p is prefix of length length(q) −1
and c is the last character of q. The number of the slot is found by calculating
the hash of p, and the last character(s) are encoded. Both queries first find the
beginning of the run to which the key belongs and then traverse it linearly in
search for the match by comparing the r-bit values stored in each slot to encoded
s. In case of the point query, we look for the exact match with encode(s); in
case of the range query, we look for a value x such that c1 ≤x ≤c2 where c1
and c2 are the last characters of the left and right keys correspondingly.
The algorithm finding the beginning of the run is identical as in the quotient
filter, and it has been described in chapter 3.

Querying CHaREQ
In the range querying algorithm for simplicity I assume
that the length of both keys is at least 2 and that the left key is smaller than
the right. Strings are 0-indexed.
Given a range query [ql, qr] I first do a binary search in the sorted array of all
the first characters of the words in the dataset to find either the first characters
of q1 and q2 or their predecessors. If the difference between the indices of the
found characters is greater than 1, I can immediately return the answer yes.
Otherwise, I continue to query the compact hash table.
I find the common prefix p of ql and qr. If length(p) > 1, I make length(p) point
queries for each prefix of p. If at least one point query returns no, the answer
to the range query in CHaREQ is also no.
Then, if length(ql) = length(p), I return yes because we have already checked
that p exists in the compact hash table.
At this point we know that both ql and qr have lengths at least length(p) + 1. I
range query the compact hash table for range [ql[0..length(p)], qr[0..length(p)].
If the answer is yes, I return yes.
If length(qr) > length(p) + 1, we can continue with range queries [qr[0..i] ·
0, qr[0..(i + 1)] for i = length(p) + 1, length(p) + 2, . . . , length(qr) −1. If answer
to any of the queries is yes, I return yes.
In all the remaining cases, the answer is no.

Making compact hash table compacter
A compact hash table as de-
scribed above would take up a considerable amount of space. I argued in chap-
ter 3 that a quotient filter should not be fully filled in, as after a certain thresh-
old, the query time starts to grow exponentially due to very long clusters. It
is reasonable to expect that CHaREQ will have the same property since the
underlying query mechanism also relies on traversing the clusters of slots. The
number of slots correlates negatively with the number of collisions, impacting
the false positive rate.
The r-bit values are stored together with 3-bit metadata in the quotient filter to
allow insertions. CHaREQ is a static data structure (due to the encoding schema
that relies on a single traversal through all the keys to assign codewords to the
characters) and, as such, does not support insertions. In order to recognize a
slot as empty, looking up the metadata bits is enough, which renders the r-bit
parts of empty slots redundant. Another observation is that the i-th slot holds
a value in the r-bit part of the slot iff is_occupied[i] OR is_shifted[i].
Therefore, if we ordered each non-empty slot starting from 0, the number of the
i-th slot would correspond to the number of set bits in is_occupied[0..i] OR
is_shifted[0..i] - an operation which is often referred to as rank1.
Therefore, in order to reduce the space usage, we can get rid of the empty r-bit
values, store all the remaining r-bit values in a separate array, and rely on the
rank operation to access the right value corresponding to the i-th slot in the
compact hash table. The challenge now is to keep the query time low.
Time and space-efficient rank operation
Many succinct data structures
rely on fast and space-efficient primitive operations rank and select [25] - among
them SuRF, as mentioned in chapter 4. There exist multiple implementations
of the constant-time rank primitive.
I decided to use the same approach as
SuRF [1] which requires o(n) additional space for a lookup table that stores
some precomputed information.
The n bits are split into n/x chunks of size x. I selected x = 512, the same as
SuRF in the LOUDS-Sparse scheme. The lookup table (LUT) of 32-bit integers
has size n/x, with an entry per every chunk. An entry i stores the number of
the set bits in the vector is_occupied[0..x*i] OR is_shifted[0..x*i]. At
a query rank(j), we can query LUT[floor(j / x)]. We can count the remaining
set bits using the GCC’s popcount operation [24]. A single popcount operation
takes O(1) time, so the query time will take O(x/64) - a constant time.

Building CHaREQ
CHaREQ accepts as input parameters a vector of strings
representing the keys in the dataset and a float value saturation. Saturation is
a parameter used to adjust the compact hash table size.
Building CHaREQ requires 2 dataset traversals. During the first traversal, I
count the number of all unique prefixes m, and I build a 256-bit vector rep-
resenting used characters which I then feed to the encoding algorithm. Given
the number of unique prefixes count and saturation I calculate the number of
slots in the compact hash table n = count/saturation. The total number of bits
required to encode the used character is the size r of the value part of a slot in
the compact hash table.
At the second traversal, I build the sorted array for storing the first characters
of all the strings in the dataset, and the compact hash table, at first consisting of
3 nb-bit vectors (occupied, continuation, shifted) and a rn-bit vector for storing
values of the slots. In the end, I build the lookup table LUT with n/x 32-bit
slots (so occupying O(4 ∗n/512 = n/128) bytes), the compact version of the
value vector (occupying O(rm)bits), and I remove the old value vector.

Implementation
In the majority, the implementation of CHaREQ is of my authorship. I recycled
the logic for finding the beginning of the run in the compact hash table from
the quotient filter implementation [16], and I used the source code of SuRF [1]
as a reference when implementing the logic related to the rank operation.

I describe them in this section.
I also describe a dynamic version of CHaREQ, which may be an interesting

Chapter 7
Conclusions
Throughout the thesis, I explored the existing state-of-the-art range filters
(SuRF and Rosetta), proposed new data structures for range filtering (Splash
and CHaREQ), and benchmarked all of them in the context of prefix filtering
and range filtering. This chapter aim to give an overview of all the results.
7.1
Results overview
Figure 7.1 shows a decision diagram which may help estimate which range filter
is the best for the given use case. I created the diagram based on the results of
the experiments presented in this thesis.
Bloom filters for small ranges
The experiments showed that in multiple
cases, the prefix/range Bloom filter achieves the best results in all its simplicity.
Prefix Bloom filter proved to be a very good prefix filter for all the investigated
datasets and query types. It may also be a good solution for very small ranges,
up to 4-5 bits. For wider ranges, it still performs very well for types of queries
that share only a short common prefix with elements in the dataset (similar,
random). However, it is better to select another filter for other types of queries.

performs better for small alphabets, whereas CHaREQ is a better option if the
size of the alphabet is big.

7.2 Future work

7.2
Future work
A common approach to range filtering is to make the data structure adapt
dynamically to the dataset and queries distribution - examples of data structures
which do that are Rosetta and Adaptive Range Filters.
In the future this idea could be adapted to e.g. Splash and Layered CHaREQ.
Splash could adapt its type of restraint, the restraint parameters and the cut-off
coefficient dynamically depending on the historic queries. Layered CHaREQ
could work in an analogous manner to Rosetta, allocating the bits across its
layers according to the past query data.
