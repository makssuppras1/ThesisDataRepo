Abstract
The primary project objectives is the development of a stereo image acquisition platform to be used on
UAVs. This platform integrates two high resolution Matrix Vision cameras with an Odroid-XU4 ARM
development board and a u-blox NEO-M8 GNSS receiver. A selection of stereo image datasets has
been acquired using this platform and each dataset includes a ground truth for comparison. A maximum
frame rate of 5 Hz has been achieved due to CPU throttling issues.
Additionally, a stereo visual odometry application has been developed to be used with stereo im-
age datasets acquired from UAVs. This application supports the SURF feature type, stereo matching
algorithms and RANSAC based outlier rejection schemes. The rigid motion estimate is computed using
singular value decomposition. The algorithm has been tested using self-acquired stereo image dataset.

Chapter 1
Introduction
This chapter gives an overview of the background and the motivation leading to this project. The overall
project requirements and the project strategy will be deﬁned in the following sections. The project has
been done in cooperation with DTU Space, and the primary objectives has been to analyse, design,
implement and test an image processing system used for Stereo Visual Odometry (SVO).
In this project, the system is treated as a supplement to the existing UAV sensor package. It can there-
fore be developed (and tested) separately, independent from the development of the UAV platform(s).
However, as the image system will likely be integrated with this platform in the future, it is necessary to
consider a number of design requirements, to ensure that this will be possible.
1.1
Background
Localisation is a capability required by all autonomous vehicles. These capabilities are often provided
by a combination of inertial sensing (by using gyroscopes and accelerometers), wheel odometry, and
GPS positioning. There are several disadvantages with this approach. All inertial sensors will suffer
from integration drift, as small measurement errors of the acceleration and velocity will accumulate
over time. The computed position is based on the previous position and the measured acceleration and
velocity. Integration drift will therefore cause the position error to increase over time. Depending on
the drift, it may therefore be necessary to periodically correct the position using the data provided from
other navigation systems[1, 2]. Wheel odometry has a different but just as severe issue, as the wheels
tend to slip and sink when used in uneven terrain, resulting in unreliable measurements.
Visual Odometry (VO) covers the process of estimating the movement and orientation of an object
(be it a human, robot or vehicle), by using one or more cameras attached to the system[3, 4]. VO is
comparable to wheel odometry, where the motion of the vehicle is estimated from the number of turns
of its wheels. Unlike wheel odometry, VO is not sensitive to factors such as uneven terrain, and has
the advantage of a more accurate trajectory estimate. The VO system used on the Mars Exploration
Rovers (MER) illustrates this very well, as it enabled the the rovers to drive safely and more effect-
ively in sloped and sandy terrains. The VO system was often used to detect and compensate for wheel
slip, correcting for errors in the wheel odometry-based trajectory estimate. This enabled the rovers to
investigate environments, where wheel slip was inevitable[5].
The navigation system used for the Unmanned Areal Vehicle (UAV) at DTU Space relies on accurate
data from global positioning system (GPS) receivers in addition to inertial measurement units (IMUs)
and other systems. GPS positioning with a sufﬁcient (and continuous) accuracy can be hard to obtain in
situations, where it moves close to the ground, where tall structures degrade the GPS signal quality[2].
In situations, such as urban canyons1 or on other planets, where the GPS based positioning is unreliable
or unavailable, the VO system can be an alternative.
Vision sensing has always been vital for autonomous navigation of vehicles and robots. UAVs of

1.2
Project Description
The UAVs used at DTU Space do not rely on any vision sensors for navigation at this point. A Stereo
Visual Odometry (SVO) system must therefore be developed to aid the existing navigation system, and
to investigate whether this type of motion estimation can be used as a viable solution to sensor drift and
GPS signal uncertainties. Visual Odometry has already been proven possible by existing research[5, 8].
It is therefore not a question of whether the method will work but how well it will work when operated
on the UAV, as opposed to a vehicle on the ground.
The primary goal of the project is to develop an SVO system that can be used to evaluate the perform-
ance of existing algorithms and identify performance issues and bottlenecks. The system performance
will be highly dependent on, not only the operating conditions, but also the available processing power.
For the initial development, all image processing will be performed off-line, using a computer. However,
since off-line processing will not be a solution to the navigation issues introduced earlier, it is expected
that an ARM or FPGA based processing platform will be used for a real-time implementation. When us-
ing the ARM platform to do the computations, the real performance of the algorithms will be revealed.
Due to the payload limitations, the development of a fast and efﬁcient SVO payload would require a
thesis on its own and is therefore not selected as the focus in this project.
Since the ARM processor performance is relatively limited, it will therefore not be used to perform
the computations in real-time. The SVO data is not required by the navigation system at this point
and a fast computation is therefore not strictly required. However, to identify bottlenecks and to assess
the overall system performance in a more realistic scenario, the system should be designed with high
performance in mind. The end goal is a UAV payload that can provide reliable position and orientation
estimates - independent of any existing sensors.
Project strategy
During the initial analysis and development phase, a hardware bring-up phase is needed to get a software
development environment up and running and to get the ARM based platform operating together with
the cameras2. In this phase, software/hardware hurdles should be identiﬁed (if any) and solved before
the software development begins. A set of image acquisition applications will be developed as well.
This will make it possible to acquire a number of stereo image datasets, in various, realistic scenarios,
using only the two cameras and the ARM platform. The cameras will not be available all the time, as
there are other projects relying on them. Furthermore, test ﬂights with UAVs will be limited due to
the vast amount of planning and time involved. Therefore, it is essential to have reliable stereo image
datasets early, so the SVO system can be developed independent of the camera availability. Naturally,
it is important that the datasets themselves represent a realistic scenario and navigation environment for
the UAV, so they can be used to evaluate the algorithm performance. However, stereo image datasets
such as KITTI[8] will be considered during the development of the SVO algorithm.
To create the foundation needed to develop the Stereo Visual Odometry system, the second phase
of the project will cover a number of existing methods and algorithms that has previously been used for

1.3
Project Requirements
The SVO system must be designed with modularity in mind, using well deﬁned data structures. This will
make it easy to adapt and (re)use in future designs, for other datasets and for other UAVs. Furthermore,
system tests will be easier to perform when the design is modular and when test data can be fed to the
system in stages. To ensure the system is suitable for the UAVs, and to ensure the system provides useful
data for other projects at DTU Space, a number of requirements has been deﬁned.
Physical Speciﬁcations
The camera stereo baseline should be selected based on a UAV working altitude of 2 to 30 m or better.
The SVO system must operate primarily when the UAV is within this range, as the low altitude will
cause the GPS based position to be affected by tall structures. On the other hand, the SVO system will
not be used at higher altitudes, since the GNSS receiver is expected to be accurate enough.
All test ﬂights will use a ﬁxed stereo baseline due to mass limitations and to ensure that the cameras
are on a very rigid platform. On the other hand, ground tests will not have this limitation and it is
therefore possible to use a range of (realistic) baselines. In this way, the algorithm performance can be
veriﬁed using different stereo baselines and depth resolutions.
In the same way, the camera pointing angles should be adjustable, making it possible to test the
system on the ground, with varying degrees of image overlap and in this way evaluate how much the
degree of overlap affects the system performance. The pointing direction should also be adjustable, as
it affects how accurately the features can be tracked between images. During forward ﬂight, using a
nadir view camera angle4, the features will move in and then out of the image frame relatively quickly,
limiting the time that the landmark can be tracked. If an acute pointing angle is selected instead, the
features will move slower but will also lie further from the camera, increasing the uncertainty in their
estimated 3D position[9]. A suitable pointing direction will be determined based on tests and previous
research. Due to mass limitations, an adjustable pointing direction will most likely not be possible for
the test ﬂights but will be limited to the ground tests alone.
Aerial Operation
The system will be designed for a UAV. Aerial vehicles move differently and relatively unconstrained,
when compared to vehicles on the ground. Since the movement is different, algorithms meant for

Hardware Speciﬁcations
The SVO system will operate using two MatrixVision mvBlueFox3-2051a Greyscale USB 3.0 cameras
paired together with a Odroid-XU4 ARM development platform. No other sensor data will be available
for the image system. The power consumption for each camera is speciﬁed as 5V/0.9A (peak) while

distribution board must therefore be developed for the payload. Naturally, the board must be compatible
with the UAV supply rail voltage(s).
System Accuracy & timestamps
Each image pair must have an accurate timestamp that can be directly coupled to the timestamps used
by the navigation system on the UAV and/or different payloads such as high accuracy GNSS receivers.
The payload will not be provided with data from any other sensors. However, as the two cameras must
acquire their images simultaneously, and with accurate timestamps, it will most likely be necessary to
use a very accurate time source, such as a GNSS receiver to synchronise the image capture sequence.
The cameras themselves can be externally triggered using the digital IO Interface, completely in-
dependent of the USB bus. As the USB bus latency can be relatively high due to drivers and ﬁrmware
layers, a direct connection is an advantage. In addition to simple sync pulse inputs, the mvBlueFox3-2
camera series also support a Hardware Real-Time Controller (HRTC) that can be used to synchronise
image acquisitions.
The estimated position and attitude should be as accurate as possible, only limited by the camera
sensors, frame-rate, available processing power and the algorithms used throughout the project. There-
fore, no upper system accuracy limit has been deﬁned. The maximum possible accuracy should be
calculated for all test scenarios and for all algorithms. In this way, the overall system performance will
be evaluated to create a foundation for further research.
System Output
The system will compute the estimated change in position (δx, δy, δz), change in attitude (pitch, roll
and yaw). Each estimate will have an associated GNSS/GPSS timestamp, so it can be compared with the
expected ground truth. Additionally, a covariance matrix must be computed, to provide a metric of the
estimated accuracy, as this is a requirement when the system is used together with eg. a Kalman ﬁlter.

Chapter 2
Theory
2.1
Navigation techniques
Navigation is a ﬁeld of study that covers the process of monitoring and controlling the movement of
a craft or vehicle. This normally includes determining the position and direction of the vehicle and
planning the most suitable route to the destination.
Most modern navigation techniques rely on a signal transmitted from reference stations or from
satellites. Therefore, they do not operate unless an appropriate signal is received. This type of system
usually provides the user with a three-dimensional (3-D) position, a velocity and a time. The remaining
navigation techniques either rely on knowing the position beforehand, using visual input to determine
position and orientation or a combination thereof. The various navigation techniques can be grouped
into the following categories:
Pilotage
”Piloting involves navigating a vessel in restricted waters and ﬁxing its position as precisely as
possible at frequent intervals”[10, ch. 8]. This type of navigation is one of the oldest and relies on
”ﬁxes”; using ﬁxed points of reference (landmarks), to determine the current position or course of
the navigator. Usually, a nautical or aeronautical chart or a topographic map is used for reference.
In this way, the position and/or orientation can be estimated using a known reference.
Horizontal ﬁxes to known points can be determined by using bearings1, where the bearings to
multiple charted objects or points of land are used to obtain a position ﬁx(2.1a). Techniques
such as radar ranging can be used as an alternative and as radar systems(2.1b). The radar can
provide simultaneous bearing and range information, making it possible to obtain a hybrid position
ﬁx(2.1c)[10]. Vertical position ﬁxes may be obtained using a depth sounder (a sonar) or by using
an altimeter.
Dead reckoning
Dead reckoning (DR) is the process of determining one’s present position by projecting course(s)
and speed(s) from a known past position and predicting a future position by projecting course(s)
and speed(s) from a known present position[10, ch. 7]. As this method relies on previous res-
ults, it is sensitive to cumulative errors. The position obtained from DR is therefore only an
approximation. Satellite navigation systems2 has made this type of navigation obsolete for most
uses. However, inertial navigation systems use the dead reckoning principle to calculate accurate
directional information.

2.1.1
Satellite Navigation
Satellite navigation commonly refers to a system that relies on satellites to provide autonomous geo-
spatial positioning, a so-called global navigation satellite system (GNSS). In this way, it is, it is possible
to use relatively small electronic satellite receivers to determine the three dimensional position with a
very high precision, usually down to a few meters. At the time of writing, there are several operational
GNSS systems; the global positioning system (GPS), GLONASS4, BeiDou, and Galileo[2].
Each satellite transmits a timing signal and a navigation message. The navigation message contains
ephemeris data, that can be used to compute the position of each satellite and also contains an almanac
for all the satellites in the constellation. The navigational messages are used to determine the satellite
positions x, y, z and the time of transmission s, the satellite clock bias[11, p. 22]. These values are
designated as [xi, yi, zi, si], where the subscript i denotes the satellite number with the value 1,2,..., n
and n ≥4. The receiver compares the local receiver time of reception ˜ti with the satellite time si
to determine the time-of-ﬂight for the satellite signal. Due to receiver clock inaccuracies, the receiver
clock time ti,rec includes the true receive time ti and a bias, terr,rec for both the receiver and the satellite
clock(2.1) [11, p. 21]. Assuming all satellites are perfectly synchronized (with no bias), the clock bias
will be the same for all received signals and can therefore be compensated for.

2.1.2
Inertial Navigation
An inertial navigation system (INS) usually uses a small computer and a sensor package containing
motion sensors (accelerometers) and rotation sensors (gyroscopes). The measurements are then used to
calculate the position, orientation and velocity by dead reckoning. No external references are needed,
apart from the initial position provided by a GNSS satellite receiver or a human operator and altitude.
==PAGE:16==
8
Chapter 2. Theory
This makes the system immune to jamming and deception but also makes the system sensitive to integ-
ration drift, as small measurement errors will accumulate over time.
As an alternative sensor, a GNSS (typically GPS) receiver is often used, if the vehicle is operating
with direct sky visibility. Combining the INS and GNSS sensors is an advantage, as they have very
complementary error characteristics. The short-term position errors for the INS system are small, but
degrade over time. On the other hand, GNSS position accuracy is poor over the short term but the
accuracy does not degrade over time[2].
Kalman ﬁltering is often used to fusion the INS and GNSS measurements. By combining a system
with tens of meters position uncertainty (GNSS) and a system, with a position uncertainty, that degrades
with several kilometres per hour (INS), it is possible to obtain a position uncertainty ranging from
centimetres to metres[2], if the GNSS augmentation techniques are used. Combining INS and GNSS
therefore makes it possible to obtain a better performance than if the systems were used individually.

Image Acquisition Platform
In this chapter, different approaches for the image acquisition platform (IAP) will be presented. All
the proposed system architectures will use a common camera and processing platform, where a high
performance Odroid-XU4 ARM development board is used in conjunction with a pair of MatrixVision
BlueFOX3 USB3.0 cameras. The architecture will be selected based on how well it fulﬁls a number of
design criteria.
3.1
Design Criteria
Before the selection of system architecture takes place, a number of criteria has been deﬁned, based on
the project strategy and requirements deﬁned in section (1.2). The acquisition platform will be shared
between multiple DTU student projects and the design tasks have therefore been distributed between the
participants. The power distribution board (PDB), the camera synchronisation circuitry, wiring harness
and payload software will be developed by the author of this report. On the other hand, the camera
mounts and the payload enclosure will be designed by two bachelor students from DTU Space. Their
part of the design will not be elaborated further in this report.
Synchronised image acquisition
When developing a stereo camera, it is crucial that the cameras capture images at exactly the
same time. Otherwise, even the slightest movement of the camera platform will result in a stereo
image where the image perspective between left and right image has changed signiﬁcantly, making
stereo-matching impossible.
In addition to the USB3.0 interface, the BlueFOX3 camera series provides multiple digital IOs
and timer functionality that can be used to generate very accurate trigger pulses at a wide range
of frequencies and lengths. Image acquisition can be initiated using an external trigger signal. It
is preferred to use these existing features to avoid additional components and to avoid using the
USB bus to request an image due to latency issues.
Internal, external triggering
The image acquisition is triggered by either an internal trigger signal or an external PPS signal
generated by a GNSS receiver or a similar device. By using the receiver to generate a trigger
pulse, it is possible to acquire stereo images at an interval directly related to the receiver time,
thus making it possible to synchronise the image acquisition with a GNSS time-stamp or position.
The acquisition platform can be operated independent of the GNSS receiver by utilising a trigger
pulse generated by one of the cameras. This is particularly useful in situations where no accurate
time and/or position ﬁx can be obtained (eg. inside buildings) and in situations, where an accurate
timestamp has no relevance (eg. when performing camera calibration).

To provide an accurate timestamp that can relate the stereo image to the navigation system on

Power Regulation & Distribution Board
The acquisition platform will require a reliable power source for the two cameras and for the
Odroid. A separate power regulation- and distribution board (PDB) will therefore be required.
According to the speciﬁcations [13], a peak power consumption of 5V/0.9A (4.5W) should be
expected from each camera when powering from the USB bus. If the cameras are supplied by the
external power connector, a power consumption of less than 12V/0.375A is therefore expected.
The cameras can be powered using an external supply with a (typical) input voltage of +12V
to +24V (with absolute min/max being 10V/30V). It is expected that the peak power will be
approximately 4.5W at any of these voltages. Based on rough estimates and documentation from
the manufacturer, the Odroid will have a power consumption of 5V/2.5A (peak) if no additional
bus powered USB devices are in use. The GNSS receiver from u-blox has a stated peak power
consumption of 3V/67mA (0.2W). When powered by the USB bus, the power consumption will
most likely be higher due to a 5V to 3V voltage regulator on the board, but it is expected that
the added power usage will be negligible and it has not been considered in this design. A power
consumption of 5V/40mA (0.2W) is therefore assumed.

3.4
Conclusion
When the different DC/DC converter options were evaluated, it was concluded that a single 5V/6A
DC/DC converter would be the simplest solution and it was expected to be cheaper than the second
proposal. However, as this DC/DC converter was not available from RS electronics at the time, separate
5V and 12V DC/DC converters were selected instead. Both of them support a wide input voltage and
they have been selected with a slightly higher power rating than necessary. It is expected that they will
provide enough headroom if additional systems are added to the image acquisition platform.
It is clear that the second camera synchronisation approach is the simplest to design, as both cameras
will share the same trigger conﬁguration. Therefore, the synchronisation board layout will be simpler to
design. This architecture has the added advantage of an overall simpler camera conﬁguration - especially
when the GNSS receiver is used to generate the trigger pulse, as no extra timer conﬁguration will be
needed in this case. It is also possible to any type of external trigger signal for image acquisition,
making the stereo camera system easy to adapt to other payloads or projects.

Colour or greyscale imaging
Most computer vision and stereo algorithms apply to greyscale images. A better performance is
therefore not expected, if colour images are used instead[19]. Unlike greyscale images with only
a single channel, colour images will require three channels and therefore three times the storage,
assuming that 8 bits are used per pixel, in each channel. This results in 24 bits per pixel for
RGB images as opposed to 8 bits per pixel for a greyscale image. The BlueFOX3-2 series can be
ordered in both a colour and a greyscale version but the latter has already been ordered.
Baseline
The required baseline1 depends on the scene that will be viewed by the camera pair. The baseline
effectively deﬁnes the accuracy of the depth estimate computed from the disparity and should
therefore be selected according to the scenario. With a larger baseline, the overlap will be reduced
and this will in turn make it more difﬁcult to perform stereo matching between the two cameras,
unless the ﬁeld of view is changed as well. By using two separate cameras for the acquisition
platform, any baseline distance can be selected. However, the image acquisition system will be
mounted inside a UAV payload enclosure with a ﬁxed size. In this conﬁguration, it is expected
that only a relatively small baseline of 6-8 cm will be possible.
Overlap
All stereo matching algorithms require overlapping frames. Therefore, a suitable combination of
lens focal length (i.e. an appropriate ﬁeld of view) and baseline must be selected to ensure that
enough overlap will be possible. For navigation and pose estimation, a large baseline and longer
focal length is preferred, as this will result in a better depth estimate and a longer range.
Focal length
For obstacle avoidance, it is preferred to use a short focal length, as this will make it possible to
focus on objects close to the camera. The camera lenses have a focal length of 12.5 mm and will
therefore be suitable for situations where the image acquisition is operated close to the ground.
However, when operating the stereo setup on a UAV, the limited range due to the focal length and
baseline may become an issue for the accuracy of the estimated motion.
Shutter
It is crucial to have a camera sensor with a global shutter as this will ensure all pixels in the scene
are captured at the same time. If a rolling shutter is used instead, the pixels will be captured
sequentially. If this type of sensor is used on a moving platform, effects such as wobble, skew or
smear may occur, causing the images to be unusable for feature matching purposes. All cameras
in the mvBlueFOX3-2 series use a global shutter and can therefore be used without any issues.
Image resolution
Computer vision algorithms are generally computationally demanding. For this reason, the highest
possible resolution is not always preferred, unless it greatly improves the feature detection and
matching.
All images must be captured in a lossless format2 to ensure they can be reliably used for feature
matching. Therefore, it is not possible to use compression algorithms such as JPEG. A higher

Frame rate
The choice of frame rate mostly depends on the movement speed of the image platform. Based

computations. Therefore, the maximum frame rate is not limited by the computation speed of an
algorithm but instead limited by the speed of the image compression and image storage. The goal
is to design an acquisition platform that can capture images continuously at the highest possible
frame rate, as long as the storage device is not full. The maximum frame rate will be determined
in the following sections.

must be suitable for an environment with many vibration sources, as it is the case on an UAV.
From this initial evaluation, it can be concluded that the choice of cameras and optics is suitable, as long
as the majority of tracked features are relatively close to the cameras. The main issue is the combination

The Odroid-XU4 development board provides several features that are suitable for the acquisition plat-

The Odroid-XU4 uses a Samsung Exynos-5 5422 ARM system on chip (SoC), identical to the SoC

Heterogeneous Multi-Processing (HMP) is used to automatically move heavy tasks to the high perform-
ance Cortex-A15 cores while light tasks will be moved to one of the lower performance Cortex-A7
cores. Unlike clustered switching, all eight cores can be used at the same time. The Linux Scheduler
will distribute the tasks according to their performance requirements. This concept is illustrated in ﬁgure

Power Distribution
The power distribution board (PDB) has been designed around a 5V/3A THN 15-2411WI DC/DC con-

already available in the DTU Space Drone lab.
Both DC/DC converters share a common DC input with an input voltage ranging from 9 to 36V.
Either a standalone battery or the UAV supply rail will be connected here. Three molex connectors has
been added to both DC/DC output, making it possible to power other systems in addition to the Odroid
and the cameras. Pads for additional ﬁlter capacitors (C1-C8) has been added across the input rail and

4.3.1
Image Queue
The multi-threaded image acquisition is a good example of a bounded-buffer problem. The producer
(the image acquisition thread) and the consumer share a common, ﬁxed-size buffer as a queue. The
producer acquires images continuously, stores them in the buffer and repeats. At the same time, the
consumer will consume the data when they become available. The problem occurs, if the producer is
adding data to a full buffer or if the consumer tries to remove data from an empty buffer. One way to
solve this problem is to make the consumer wait, until there is data or make the producer wait, until the
buffer is not full.

If multiple producers or multiple consumers are sharing the same buffer as illustrated by ﬁgure (4.7),
this simple waiting scheme is not adequate. If more than one producer is present, it is possible for both
of them to attempt adding new data at the same time - concurrently. A similar problem exists when
multiple consumers are present. One way to solve the problem is to ensure the critical section is only
executed by one producer at a time, while making sure the consumers will wait, if the buffer is still
empty. This is achieved by using monitors to ensure both mutual exclusion4 and the ability to make a
thread wait for a for a condition variable (i.e. wait for data to be added to the buffer). A monitor consists
of a mutex (lock) and condition variables. The mutex is used to protect shared data from being accessed
by multiple threads while the condition variable is used to block or freeze a number of threads, until
new data has been added to the queue. If the buffer is empty, the consumer will wait until the condition
variable is set by one of the producers. Each time new data is added to the buffer, the condition variable
is set to notify one of the waiting customers.
The image queue used in this design is a thread-safe implementation of a common FIFO queue
(std::queue), present in C++11 revision and onward. To ensure a thread-safe queue, a combination
of mutex and condition variables has been used. The implementation of the image acquisition thread
can be found in appendix (C) in listing (C.4). Each frame in the queue contains the image data, the
camera ID, the frame ID and the decimation rate.
4.3.2
Logging functionality
Mutual Exclusion (std::Mutex) can also be used to provide thread-safe logging functionality for all
application threads. This is achieved by using mutex to protect access to the console output stream
(std::cout) or to the ﬁle output stream (std::ofstream). In this way, it is ensured that the
character stream will not be interrupted due to multiple threads writing the same stream. By using the
class std::lock_guard as a mutex wrapper, it is possible for a thread to own the mutex for the
duration needed to write a complete stream of characters. Afterwards, when the lock guard goes out of
scope, a different thread will gain access to the stream.
Two classes are used to provide thread-safe stream access. By streaming characters to CoutThread{},
the cout stream can be accessed from multiple threads. In a similar way, multi-threaded ﬁle access is
achieved for the log stream. However, due to the nature of std::ofstream, it must be initialised be-
forehand. This is illustrated in listing (3). The complete implementations of the thread-safe ﬁle stream
and console output stream can be found in appendix (C) in listing (C.5).

4.3.3
Image Storage
When an image has been acquired from the camera, it must be moved to the internal image queue stored
in system memory. This queue must be emptied continuously to avoid running out of system memory.
Therefore, multiple image storage threads are usually allocated by the application. Each acquisition
thread has access to the queue and can request a frame from it. Each frame contains the cameraID
(master/slave), the image ID and a decimation factor. This information is necessary to ensure that the
images will be stored in the correct order. The image storage process is based around the two following
functions supplied by the OpenCV library[21].
imwrite
This function provides functionality to store images using various formats. The image format is
controlled by the ﬁlename extension and ﬁle formats such as *.bmp and *.png are supported.
Only 8-bit (or 16-bit) single or 3-channel unsigned images can be stored using this function, when
using the PNG format. By setting the params, it is possible to use a range of PNG compression
rates ranging from 0 to 9. Additionally, a number of compression strategies can be used, depending
on the preferred performance and image size.
resize
This function provides functionality to resize an input image and then return an image with a
reduced size. The output image size is set by dsize, and should be selected according to the
scale factors used. The scale factors can be set in the x and y direction using fx and fy. The
interpolation method can be set by the parameter interpolation, depending on the use case.
A range of interpolation methods are available. For a decimation or image size reduction, the
INTER_AREA resampling is usually preferred.
The following steps will be performed by each image storage thread. It is assumed that an image queue
is available for all image acquisition threads. It is expected that the images in the queue have already
been resized by the image acquisition process, if image decimation/resizing is enabled.
1. Setup the imwrite parameters according to the compression rates set by the user.
2. Wait until an image frame is added to the image queue and then continue to the next step.
(a) If the queue is empty but the acquisition is ongoing, keep waiting (run=1).
(b) If the acquisition has ended, exit the loop and end the image acquisition thread (run=0).
3. The following steps will be performed for each new frame in the queue:
==PAGE:37==

(a) Acquire the latest frame from the shared image queue (dequeue). The image has already
been resized at this point, as the image acquisition process will resize the image if necessary.
(b) If a dummy frame is detected (i.e. a camera ID of 255), the image storage thread will end
gracefully (go to step 4). In this way, the main thread can still end the image storage threads,
if they are currently waiting for a new image frame.
(c) Set ﬁle output string according to the camera ID, image ID and the image size (i.e. the
decimation rate). The output ﬁle will be named m_000000 (or s_000000), depending on
the camera ID. If the image in the queue has been resized (decimated), the ﬁle name will
have an _2 or _4 appended to it, depending on the decimation rate. At this point, the string
contains both the ﬁle path and the ﬁle name, eg. ”.\outfolder\m_000000” but no ﬁle
type is appended.
(d) Select the image format, depending on user preferences. Only BMP and PNG formats are
supported. Depending on the selection, .png or .bmp will be appended to the output string,
as this will deﬁne the image format used by imwrite.
(e) Call imwrite with the appropriate parameters. If the image could not be successfully
written, return an error. Keep track of the number of failed attempts for logging purposes.
4. Print error statistics and end the image storage thread.

The mvPairCalibration application is used for acquiring a number of stereo images to be used for camera

Live display
Naturally, it is necessary to provide the user with a live preview of the scene currently viewed by
the camera. In this way, the user can easily select a camera position that ensures the chessboard is
within the view of both cameras. Visual feedback is also crucial to ensure that the camera focus,
aperture and shutter speed is suitable for the scene in question. While the Odroid has an HDMI
port to be used with an external display, an SSH client can be used instead to access the xwindow
window instance running on the Odroid. This makes it possible to view a live image feed running
on the Odroid, without any additional software running on the host PC.

Internal triggering
The cameras will both be triggered by the same signal and one of the cameras will generate the
trigger signal using the method introduced in the earlier chapters. It is expected that a very low
trigger frequency will be used due to the limited frame rate that can be used when displaying

Manual acquisition
When the camera is placed correctly, the user will acquire the image pair manually. To make it
possible for the user to store a speciﬁc image pair on command, the existing SSH connection can

Camera parameters
The calibration quality depends on the quality of the images captured by the cameras. Therefore, it
should be possible to set the camera parameters according to the speciﬁc scenario. Both automatic
and manual exposure should be available, as a ﬁxed exposure is not always suitable for a dynamic
outdoor environment.
Image resizing
When calibrating the stereo camera setup, it is preferred to use the highest possible resolution but
when the stereo cameras are used to acquire images for other purposes, it may be necessary to
acquire images in a different resolution. Therefore, the images captured by this application will
always be stored in both the full resolution and a quarter of the original resolution. It is then up to
the user to select the most suitable image resolution.

A number of parameters must be conﬁgured each time the application is started. All parameters has
been listed in table (4.9). It will be necessary to determine the camera IDs for the current conﬁguration,
if they are not already known. A list of valid devices can be generated by launching the application with
the parameter -list_devices enabled. The master and slave camera ID can then be set according
to the cameras listed. The IDs will usually not change, unless a camera is added or removed.
As a default conﬁguration, the cameras will use line5 as the trigger input and the master camera will
use line1 for the trigger output. Depending on the speciﬁc scene, it may be necessary to set the camera
exposure time and/or enable automatic exposure. An output directory will automatically be selected, if
no speciﬁc directory is provided by the user.
Image sharpness
While the camera focus can be adjusted by eye, this method is very prone to human error. To ensure
that both cameras are focused at the same point, a mathematical approach will be used to determine if
the image is out of focus. First, it is assumed that a well-focused image will have sharper edges. This
assumption only holds true, if textures are present in the image. By applying the Laplacian operator to
the image, the edges can be highlighted, resulting in a gradient image as illustrated by ﬁgure (4.8)5. The
variance of this image can be used as an image sharpness metric. A high variance indicates less blur - a
sharper image, since there is a wide spread of both edge-like and non edge-like responses in the image.
If there is a low variance, there will be very few edges in the image, indicating that the image is blurry.
For this method to work well, it will be necessary to photograph a textured area, preferably with texture
in both the foreground and the background. This should ensure that most of the scene is in focus.

4.4.1
Application ﬂow
The application ﬂow is described by the following steps:
1. List devices: The available devices will be listed at startup to ensure that all cameras are present
and no camera connection issues are present.
2. Parameter validation: During the ﬁrst step, the command line parameters are parsed and valid-
ated. If invalid parameters has been set, they will be replaced with default values.
3. Folder validation: The output directory will always be validated. If the folder does not exist, it
will be created. If this is not possible, the application will exit immediately. If the output folder
has not been set by the user, the default folder will be used instead. If the default directory already
exists, a number will be appended to the folder name and no existing data will be overwritten.
4. Log ﬁle initialisation: Before it is possible to use ﬁle log functionality, it is necessary to initialise
the thread-safe ﬁle stream according to listing (C.5). If the ﬁle cannot be created, the log ﬁle
functionality will be disabled.
5. Camera selection: In this step, the camera selection will be validated further. This ensures that the
user has selected two different IDs for the master and slave camera. If less than two valid cameras
are available, the application will exit. If both selected camera IDs are valid, the device pointers
pMaster and pSlave will be initialised by calling getDeviceFromUserInputId.
6. System conﬁguration: A vector of device pointers is created, as this makes it possible to access
the individual cameras using indexing. Afterwards, the camera parameters are validated. The
second validation is used to ensure that the user has used valid camera IDs for the master/slave
camera. The selected digital IO lines are also validated here. If a parameter is invalid, it will be
ignored and the default value will be used instead. For some parameters, there exists no default
values. In this case, the user will be requested to enter a value manually.

conﬁgured. First, the exposure is conﬁgured using the function setupExposure. Afterwards,
the timers are conﬁgured using setupFrameTimer.
8. Thread initialisation: At this point, the threads can be launched. An image acquisition thread is
needed for each camera but only one thread is needed for image storage, due to the low number
of frames that will be stored.
9. Trigger initialisation: All threads are now ready and setupFrameTriggerInput can be
used to enable the trigger inputs for both devices. The master trigger is then activated using
enableFrameTriggerOutput.

(a) Image validation: With both image acquisition threads in operation, a pair of images will be
added to the display queue every second. The main loop will continuously dequeue images
from the display queue and store them as temporary frames (frameA and frameB). It is
expected that the images are received in left-right pairs, unless one image acquisition thread
becomes out of sync. If the IDs are mismatching, the application will exit immediately.

Continuous Acquisition - mvPairCapture
The mvPairCapture application is used to acquire image pairs continuously, either using external or
internal triggering. To achieve this, a number of features are needed:
Internal & external triggering
For the continuous image acquisition, the GNSS receiver will be used to generate the trigger pulse,
making it possible to link each image pair with a timestamp and a position. If the external trigger
functionality is disabled by the user, or if a GNSS ﬁx can not be obtained, one of the cameras will
be used to generate the trigger signal instead. The same trigger signal is used for both cameras and
trigger frequencies ranging from 1 to 30 Hz must be supported by the application. The functions
used for the camera trigger conﬁguration in mvPairCalibration are also used for this application.
GNSS receiver conﬁguration
The GNSS receiver will be used to generate a time pulse with a speciﬁc frequency and it will be
necessary to initialise the receiver before the image acquisition can begin. If the default conﬁgur-
ation is used, time pulses will only be generated with an interval of 1Hz.
Therefore, it will be necessary to add u-blox speciﬁc functions to the current IAP application
framework. It may also be necessary to detect and ﬁlter speciﬁc messages transmitted by the
GNSS receiver, as not all of the transmitted messages will be relevant.
The GNSS receiver supports periodic transmission of the current position and timestamp on the
virtual COM port. However, only simple NMEA messages will be transmitted when the default
conﬁguration is used. The default message rate matches the default time pulse frequency of 1
Hz. The UBX message6 UBX-NAV-PVT contains all the relevant information, including the
current position, time and ﬁx status, in one single message. Therefore, this UBX message should
be enabled as a supplement to the NMEA messages. The message rate should be changed to
match the time pulse frequency. This will ensure that each image pair will have a matching
UBX-NAV-PVT message containing the position and an accurate timestamp. To enable this mode,
it will be necessary to send a number of UBX conﬁguration messages to the GNSS receiver[14].
Due to the increased message rate, it will be necessary to increase baud rate of the virtual COM
port. The default baud rate of 38400 is not adequate for very high message rates. Therefore, it
will be necessary to increase the baud rate to eg. 115200 baud to avoid message corruption. This
is also achieved using UBX conﬁguration messages.
GNSS message parsing and logging
All GNSS messages are transmitted on the virtual COM port associated with the USB interface.
These messages must be parsed and if they contain relevant information, they must be logged
to a ﬁle. It is expected that only the UBX-NAV-PVT messages will be necessary to log, as this
message contains all the needed information, including the position, timestamp and the GNSS ﬁx
status.
Camera parameters
Naturally, the quality of the dataset and the estimated position depends on the quality of the images
captured by the cameras. Therefore, it should be possible to set the camera parameters according
to the speciﬁc scenario. Both automatic and manual exposure should be available, as a ﬁxed
exposure is not suitable for a dynamic outdoor environment.
Image resizing
At high frame rates, the Odroid storage performance will most likely be inadequate. Therefore,
it will be necessary to resize the images in order to achieve continuous image acquisition at high
frame rates.

4.5.1
U-blox GNSS receiver conﬁguration
To enable conﬁguration of the u-blox receiver using the UBX protocol, a set of functions have been
developed and can be found in the ubx.h library. Only the features essential for this project has been
implemented. All UBX messages begin with a 2 byte header, 2 bytes describing the class and ID of the
message, a 2 byte payload length (little endian) followed by the payload with a length corresponding
to the length speciﬁed by the previous two bytes. Finally, the message ends with a two byte checksum
that can be used to identify and reject corrupt packages. Knowing the length of the payload is essential
to know how many bytes to read, as no end-of-message character will be transmitted. Most of these
functions will require the fd index to perform read/write operations.
It is always necessary to validate the navigation solution provided by the GNSS receiver. It is
therefore expected that the NMEA valid ﬂag or the gnssFixOK ﬂag in UBX-NAV-PVT messages will be
used for this purpose[15].
bool ubx_message_wait(fd)
The GNSS receiver transmits messages using multiple protocols (UBX, NMEA etc) and message
ﬁltering will be needed. UBX message header is deﬁned as two bytes; ”0xB5” followed by a
”0x62”, and is transmitted at the beginning of every UBX message. This function will compare
all incoming bytes to this header and will return true, when the header is detected. The function
will automatically timeout and return false, if no header was received. The class and the ID byte
is received immediately after the header and can be used to identify the received message.
ubx_ack_wait(fd,classByte,idByte)
When a conﬁguration command (UBX-CFG) is transmitted to the GNSS receiver, it will reply
with an acknowledge (UBX-ACK, class:0x05 and an ID:0x01) message to conﬁrm that the com-
mand was received correctly. Alternatively, a not-acknowledged (UBX-NAK, class:0x05 and an
ID:0x00) message will be transmitted. The payload is two bytes long and it will always contain
the class and ID identical to the command originally transmitted by the host. The last two bytes
contain the checksum and can be used to validate the (UBX-ACK) message by comparing it to the
expected checksum. The function performs the following tasks:
1. Wait until an UBX header is received. The function ubx_message_wait is used for this
task. The function will time out and return -1, if no header is detected in the incoming bytes.
2. If a header is detected, the class and ID bytes will be received and compared to the expected
values for an ACK.
3. When an ACK message is detected, the payload length (2 bytes), payload (2 bytes) and
checksum (2 bytes) will be read by the host.
4. To verify if a valid UBX-ACK or UBX-NAK message is received, it is necessary to compute
the expected checksum for these two messages using ubx_compute_checksum.
5. By comparing the received checksum with the computed responses, it is possible to determ-
ine whether the response was a valid UBX-ACK, UBX-NAK or neither.
ubx_compute_checksum(& CHK_A,& CHK_B,* Buffer,byteOffset,length)
Each time a UBX message is transmitted to the receiver, it is necessary to compute and transmit
two checksum bytes CHK_A and CHK_B as well, as they are required by the UBX protocol. It is
also possible to detect corrupt messages, by comparing the computed checksum to the received
checksum. The checksum is calculated from the class and ID bytes, the length bytes and the
payload stored in the Buffer with a length given by the length parameter.
The bytes are calculated using a 8-bit Fletcher Algorithm[14]. When calculating the checksum
bytes, it is important to mask the resulting values, unless they are stored as unsigned 8-bit integers.
ubx_set_port_rate(fd,baudRate, nmeaEnable)
If the message rate is increased, it will be necessary to change the USB UART baud rate used

ID:0x00) is used for this purpose. The command is used to conﬁgure all the supported ports,
including the SPI, UART and USB interfaces. Even if it is only necessary to change a single

Each interface can be conﬁgured to receive and/or transmit using protocols such as UBX and
NMEA. By setting the in and out parameters in the UBX-CFG-PRT message, it is possible to

ubx_set_meas_rate(fd,freq_Hz)

speciﬁc position and timestamp, even if a trigger pulse frequency higher than 1Hz is used.
ubx_enable_msg(fd,msgClass,msgID,port,rate)

receiver is powered on. This includes messages such as UBX-NAV-PVT. By using the UBX-CFG

UBX messages (selected by a class and an ID) on speciﬁc receiver port(s) with a speciﬁc rate. The
rate is set relative to the rate selected by UBX-CFG-RATE. By using a rate of ”1”, the frequency
of the measurement and message rate will therefore be identical. When enabling broadcast of the
UBX-NAV-PVT message, its’ associated class and ID (class:0x01 and ID:0x07) must be used.

Some GNSS receiver revisions support using a different time pulse conﬁguration, when no GNSS
ﬁx is available. This functionality was not supported by the receiver used in this design. Therefore,
the time pulse conﬁguration will not change if/when the ﬁx is lost.

2. Receive all bytes, until CR and LF are received.
3. Parse the NMEA message. An NMEA-ZDA message will contain the characters ”ZDA” as
the fourth to sixth character and can easily be identiﬁed.
4. Filter all NMEA messages matching the above criteria and display them.
The UBX-NAV-PVT message was considered more versatile than NMEA-ZDA and was therefore
selected instead.

4.5.2
GNSS Logging
A separate thread has been dedicated to the GNSS logging process. The following steps will be per-
formed, when the image acquisition has started. The implementation of the GNSS logging thread has
been listed in (C.6) in appendix (C). Ideally, the GNSS logging should be running in parallel with the
image acquisition threads, to ensure that the number of logged messages is identical to the number of
acquired images. This is not possible, as each thread must be launched one after the other. Therefore,
the number of logged messages may be larger or smaller than the number of acquired images, depending
on what thread that started ﬁrst.
1. Initialize the GNSS logﬁle stream. This stream is only accessed by one thread and can therefore
be initialised as a simple ofstream type.
2. Verify if the log ﬁle is open. If the log ﬁle could not be accessed, disable GNSS logging.
3. Print the header using the function ubx_write_nav_pvt_header.
4. Request a single UBX-NAV-PVT message and print the parameters returned by this message.
5. Perform continuous logging, until thread is stopped by setting threadRun=0.
(a) Log all received UBX-NAV-PVT messages and store them in a pvtNav payload.
(b) Store the received payload using the function ubx_write_nav_pvt
(c) Keep track of the number of logged payloads.
6. Print the number of logged payloads to the terminal, close the log ﬁle and exit the thread.

4.6
Conclusion
In this chapter, the image acquisition applications were developed and suitable PNG compression para-
meters were selected. At this point, the image acquisition platform is operating without issues, as long
as very low frame-rates of 4 Hz are used. This limitation is caused by a combination of poor storage
performance and thermal throttling.
At higher frame rates, all CPU cores will usually be fully loaded. Due to insufﬁcient cooling, the
CPU cores will reach a certain threshold (95◦) and CPU throttling will be enabled, slowing down the
image compression speed by more than half. This bottleneck causes the the image storage threads to
be unable to keep up with the image acquisition threads. If the CPU does not cool down, the thermal
throttling will continue. Eventually, the image queue will outgrow the available RAM, causing the ap-
plication to crash. Thermal throttling is apparently a known problem (feature?), according to the [Odroid
XU4 benchmark webpage]. To reduce the operating temperature of the Odroid, it will be necessary to
improve the cooling solution. At this point, no other solutions have been investigated.
The eMMC storage used on the Odroid is insufﬁcient for higher frame-rates, unless image compres-
sion is used. On the other hand, image compression causes the Odroid to overheat and slow down. It
will therefore be necessary to improve the storage solution using an external USB 3.0 storage device or
improve the cooling solution with a larger heatsink and/or fan.

Stereo Visual Odometry
In this chapter, different approaches for the stereo visual odometry (SVO) algorithm have been evalu-
ated. Most existing visual odometry (VO) research are related to autonomous vehicles, driver assistance
systems or other systems used to aid human drivers[8, 24, 25]. The same principles have also been
applied to autonomous robot navigation [26] and on interplanetary missions[5, 27, 28] such as the MER
rovers used during the Mars exploration program by NASA.
Visual Odometry (VO) covers the process of estimating the movement and orientation of an object
or vehicle, by using one or more cameras attached to the system[3, 4]. VO is somewhat similar to wheel
odometry but unlike wheel odometry, VO is not sensitive to factors such as uneven terrain, and has
the advantage of a more accurate trajectory estimate, with relative position errors ranging from 0.1%
to 2%[4]. Although VO is also sensitive to drift, the drift rates are often lower than all but the most
expensive inertial measurement units (IMUs). It can therefore be an advantage to combine a VO system
with the existing sensor package to reduce inaccuracies.
Wheel odometry works by integrating the number of turns for each wheel. In a similar way, visual
odometry operates by incrementally estimating the relative pose of the vehicle, using images acquired
form an onboard camera. By examining the image changes between image pairs, the 3-D motion can
be estimated[4]. The end goal is to estimate the 6-degree-of-freedom (DoF) motion parameters (the
egomotion) to aid existing sensors used on the vehicle. The six parameters are deﬁned as the translations
along x, y and z in addition to the three rotation angles about these axes (yaw, pitch and roll).
Stereo visual odometry is often investigated as an alternative pose estimation method to be used in
situations where the GNSS positioning is unavailable[9, 29]. In this particular study, a stereo camera
rig was mounted on a small remote controlled (RC) helicopter and a number of datasets were collected,
but no real-time computation took place. A range of camera pointing angles were investigated and
a pointing angle of 66.5◦down from horizontal was selected as a suitable angle. In [30], a slightly
different approach was used to estimate the pose and motion of an aircraft using computer vision. Here,
a catadioptic system was used to estimate the attitude of the UAV while a stereo camera system was
used for odometry estimation. In [6, 7], a number of high performance on-board processing systems
were evaluated for use on UAVs. This research creates the base for the SVO system in this project.
The concept of monocular visual odometry was ﬁrst introduced by Moravec[31] in 1980. The re-
search was related to the development of planetary rovers, motivated by the NASA Mars exploration
program[4]. Although the ﬁrst research was done using a single camera only, today’s research is primar-
ily based on stereo visual odometry (SVO). The research done by Moravec is still relevant, however, as
it deﬁnes the the motion-estimation pipeline used by most visual odometry algorithms.
A Monocular Visual Odometry system can be used for measuring the angular movement of a feature
or point but the motion scale cannot be observed using this method as the scale factor is unknown. The
absolute scale must be computed in other ways, eg. by measuring the size of an object in the scene.

The monocular VO algorithm can be described by the following steps[4]:
1. Image acquisition: Capture a new frame denoted Ik
2. Pair matching: Extract and match features between Ik−1 and Ik
3. Essential matrix: Compute the essential matrix for image pair Ik−1 and Ik
4. Motion: Decompose essential matrix into rotation Rk and translation tk and form the camera
motion; the transformation Tk.
5. Scale: Compute the relative scale and scale the translation tk.
6. Pose: Obtain the current camera pose by concatenation: Ck = Ck−1 · Tk.
Stereo Visual Odometry solves the scale problem by using two cameras to provide feature depth inform-
ation. Assuming the projective parameters of the cameras are known, it is possible to triangulate features
for each stereo pair and then obtain measurements at a known scale. Naturally, even a stereo camera rig
will degenerate to monocular vision at far distances, so it is crucial to consider the working distance of
the system.
Most of the visual odometry systems are used for driver assistance and autonomous vehicle navig-
ation and therefore require a real-time, high-performance implementation. The algorithm performance
is also limited by the on-board processing power. A stereo visual odometry system for UAVs is not a
small task. It will require both algorithm optimisation to ensure a real-time computation is possible and
will also require a high performance computation platform that is within the power and mass limits set
by the UAV designer. A number of on-board processing methods have been evaluated by[6, 7] and an
implementation based on an FPGA or ASIC and/or a high performance ARM processor could be a good
way to achieve real-time performance in a very small and light package. While this topic is very relevant
for this project, it will require a signiﬁcant amount of research and development. The hardware architec-
ture of the processing system is therefore out of scope for this project and will not be evaluated further.
Instead, the algorithm will be an ofﬂine implementation, where all acquired datasets are post-processed
using a relatively powerful desktop PC. While a fast computation is preferred, it will not be a design
driver. All focus will be on the current situation: stereo visual odometry on datasets acquired from an
UAV.

Stereo Visual Odometry Overview
Most visual odometry algorithms consist of several subsystems and each of these subsystems can be
implemented in a number of ways. To limit the scope of this project, only a small number of methods
will be evaluated and used in the ﬁnal system. It is likely that the ﬁnal SVO algorithm will be a combin-
ation of several existing methods. Most stereo odometry algorithms[31, 32, 5, 4, 33] follow a structure
consisting of the following steps.
1. Image acquisition: Capture an image pair Il,k−1,Ir,k−1 and perform image rectiﬁcation.
2. Stereo matching: Identify features in Il,k−1,Ir,k−1 and perform stereo matching. Triangulate the
matched features for each stereo pair.
3. Image acquisition: Capture another image pair Il,k,Ir,k and perform image rectiﬁcation.
4. Stereo matching: Identify features in Il,k,Ir,k and perform stereo matching. Triangulate the
matched features for each stereo pair.
5. Pair matching: Extract and match features between Il,k−1 Il,k

5.1.1
Image acquisition & rectiﬁcation
The image acquisition platform is used to acquire and store stereo images, linking each image pair
with an accurate timestamp and a position computed by an on-board GNSS receiver. An image will be
captured simultaneously from both cameras at a ﬁxed interval set by the user. All captured images will
then be stored on the on-board storage device. It is expected that both ground and ﬂight datasets will
be collected using this platform. Datasets such as KITTI[8] may also be considered for the initial part
of the algorithm development. Each stereo image dataset will consist of two sets of n images denoted
by Il,0:n = {Il,0, ..., Il,n} for the left (master) camera and Ir,0:n = {Ir,0, ..., Ir,n} for the right (slave)
camera. A log ﬁle from the GNSS receiver provides the ground truth and will also be included with each
dataset. In this way the performance of the algorithm can be evaluated.
Each camera observes the same scene from a slightly different view and it is therefore possible
to extract 3-D information from the image pair by observing the relative position of objects found in
both images. This requires object matching between the left and the right image, a process known as
stereo matching. Since the cameras are not using the same viewpoint, it may be difﬁcult to ﬁnd a match
between an object or point found in both images. For this reason, the matching process may require a
two-dimensional search.
If the cameras are aligned to be co-planar, the feature matching can be reduced to a single dimension;
a search along the epipolar line in the right image. While it may be possible to physically align the two
cameras with a very high precision, image rectiﬁcation is often used as a more practical alternative.
Image rectiﬁcation relies on an accurate calibration of the stereo camera to obtain the intrinsic and
extrinsic stereo camera parameters. These parameters are then used to transform the original images
and bring them onto a common image plane, aligning the two images. The image rectiﬁcation greatly
simpliﬁes the stereo matching process and reduces the computation time, and is therefore included in
most visual odometry algorithms. This is illustrated in ﬁgure (5.1).

5.1.2
Stereo matching
When the acquired image pair has been prepared (rectiﬁed and/or pre-ﬁltered)[25], stereo matching is
performed using either sparse or dense algorithms. With a rectiﬁed image pair, the images are row
aligned[19] and the matching is narrowed down to a single line, starting from the original point position
and ranging from right to the left. This is illustrated in ﬁgure (5.2).
Two common approaches are used to ﬁnd stereo correspondences: The pixel-based methods are
used to ﬁnd the best match between a block of pixels extracted from the left image and a search window
deﬁned in the right image. Usually, a small block of pixels is deﬁned around every pixel or feature in
the left image. This block is then compared to a window of pixels in the right image. By comparing the
block with all pixels in the right image, it is possible to ﬁnd the best match. The corresponding block is
selected to ensure the similarity between the all pixels is the highest. This process is repeated for all the
relevant features or for all blocks in the image and will result in a disparity value for most points in the
image.
Common pixel-based matching algorithms include the sum of absolute or squared differences (SAD,
SSD) and normalised cross-correlation (NCC)[34, 35]. The SAD algorithm is computed by taking
the absolute difference between each pixel in the left block and each pixel in the right block used for
comparison. All differences are then summed, resulting in a metric deﬁning the block similarity. In a
similar way, the SSD is computed by calculating the sum of the squared difference for each pixel. In the
NCC algorithm, the two pixel blocks are compared using normalised cross correlation(5.1).

As the search window or block size grows, so will the computation time.
The disparity map[19] illustrates the position difference of a pixel (or object) when viewed by the
left and right camera[36]. The disparity here is deﬁned as d = xl −xr. When the images are correctly
rectiﬁed, there should be no disparity in the y-axis. While the stereo matching can be done by comparing
a single pixel to all pixels in the window, this approach will result in a very noisy disparity map. For this
reason, a block of pixels is usually preferred for pixel-based stereo matching algorithms.
The second approach relies on features to determine the disparity. Features are identiﬁed in both
images and then matched using various feature matching algorithms. For this reason, the disparity
values can only be computed for the features matched in both images (sparse stereo) as opposed to the
previous method where a disparity value exists for most points (dense stereo). This will limit the number
of features that can be tracked between each image pair, as only features with a deﬁnitive stereo match
can be used. However, a dense disparity map is not always preferred, as some degree of averaging will
be needed to reduce noise. A dense disparity map may therefore introduce errors in the calculated object
distance, as some disparity values will be computed from averaging and not a deﬁnitive match between
two points.
The semi-global matching[37] method has also been considered as a promising combination of the
global and local stereo methods as it provides accurate, pixel-wise matching at low computation time.
However, due to the limited time in this project, this method was never implemented in the ﬁnal al-
gorithm.

5.1.5
Outliers and Inliers
After feature matching has been performed, it is expected that the resulting dataset is ’contaminated’ by
outliers[40], i.e. incorrect matches. The number of outliers in a dataset can be signiﬁcant and will affect
the solution[41].
One of the common methods to detect outliers is the RANSAC (Random sample consensus) al-
gorithm. This algorithm is used to determine the inliers by randomly selecting a number of feature
correspondences from the dataset and then estimating the egomotion based on the samples[8]. The
sample size S should be as small as possible, while large enough to estimate the egomotion. The fea-
ture correspondences are all deﬁned as 3-D coordinate pairs and therefore at least S = 3 points will
be needed for the estimate[40]. Because of the low number of data points, the number of samples will
be low, even if a large number of outliers exist in the dataset[8]. The estimated egomotion can then
be applied to the ﬁrst set of features, reprojecting them onto the second set. The error between the es-
timated and the actual 3-D position for each feature deﬁnes the Euclidean reprojection error1 and can
be used as a metric to detect inliers. For instance, a feature can be considered as an inlier, if the Eu-
clidean reprojection error is lower than a speciﬁc threshold, as it is proposed in [8]. In a similar way,
the MATLAB function estimateFundamentalMatrix can be used to estimate the fundamental
matrix from corresponding points in stereo images and detect outliers in the process.

5.1.7
Weighed data
To improve the motion estimate, each point pair should be individually weighed. The stereo disparity is
inversely proportional to the depth and can be used as a weighing metric. When observing objects far
from the stereo camera, the disparity will be very small and the objects will be difﬁcult to resolve. This
will result in a larger depth error. By weighing the matches using the disparity, the result will be less
sensitive to depth errors.
Weighing can also be used to improve the result in situations where either the rotation or translation is
dominant. The computation of the translation is dependent on the accuracy of the 3-D point positions and
close features are therefore preferred. On the other hand, far objects will generally be more suitable when
computing the rotation parameters. While the rotation angle will remain the same, a greater distance to

5.1.8
Alternative solution
The non-linear least-squares minimisation problem listed in 5.22 can be solved in several ways. As an
alternative to the method using singular value decomposition, iterative methods such as the standard
Levenberg-Marquardt (LM) least-squares algorithm can be used. However, the solution based on SVD
gives a closed-form solution and provides the best possible solution in one step and does not require an
initial guess. While LM can be used for more general non-linear least-squares problems, it is expected
that the SVD solution will be faster and just as accurate.
5.2
Conclusion
In this chapter, a number of SVO approaches were introduced and a number of algorithms were selected
for the ﬁnal application. A range of algorithms have been selected for each computation step, to make
it possible to test the performance of each method, when used together with a UAV dataset. In this way,
the most suitable methods can be selected for the next version of the SVO application. Either a pixel-
based or feature-based stereo matching approach can be used. A sparse disparity map will be calculated
in both cases. While the pixel-based matching is much slower than feature-based matching, it can be a
good alternative when the same features cannot be identiﬁed in both the left and the right image. The
pixel-based matching can be used as a secondary approach, if the feature-based method fails.
Harris-Stephens and SURF feature types have been selected for the ﬁnal algorithm. SURF has
been selected due to its rotation and scale invariance while Harris-Stephens has been included as a
comparison. Both RANSAC and the Constraint Satisfaction Method (CSM) will be used for the outlier
detection step. The reliability of CSM is not as well documented so the RANSAC algorithm has been
included as a benchmark. The motion parameters are estimated using SVD and both a weighed and
unweighed computation can be used.

6.1
Algorithm Overview
The following methods have been selected for each step, based on the evaluation in the previous chapter.
1. Acquisition & rectiﬁcation
A speciﬁc algorithm has not been selected for the ﬁrst step. Instead, it is assumed that a set of
stereo calibration images have been acquired and the stereoCameraCalibrator application
included with MATLAB has been used to compute the intrinsic camera parameters. These para-
meters are then used to rectify all the captured images to ensure they are row aligned and ready for
stereo matching. Image rectiﬁcation is computationally expensive and will only be done once for
each dataset. The resulting images will be stored on the PC for later review and/or computation.
2. Stereo matching
The stereo matching process will be implemented using a combination of both the pixel-based and
feature based algorithms. To avoid inaccuracies due to disparity approximation (and to reduce the
computation time), the stereo matching will only be computed for the features identiﬁed in the
left image, resulting in a sparse disparity map. In contrast, a dense disparity map is computed by
matching all pixels in the images.
The stereo matching itself will be done using either feature matching or a pixel-based comparison,
depending on the user selection. When feature matching is selected, it will be necessary to identify
features in both the left and the right image and then perform feature matching using built-in
MATLAB libraries. Instead of attempting to match one feature in the left image to all features in
the right image (and risk introducing incorrect matches due to unexpected similarities), a region
of interest (ROI) will be deﬁned around the expected location of the feature. While there may
be a similar (but incorrect) feature outside this region, only the features within the ROI will be
considered for the matching. This is illustrated in ﬁgure (6.2), where the ROI is indicated by the
blue border and incorrect matches are illustrated by the red lines. The ROI size can be estimated
based on the maximum expected disparity. The ROI location can be set based on the fact that
features matched in the right image will be constricted by a right to left movement i.e. a positive
difference between the left and the right x-coordinate. Features that does not fulﬁl this requirement
should therefore be avoided. This is illustrated by ﬁgure (5.2) in the previous chapter. A pixel-

4. Outlier removal
When a set of potential matches have been computed, it will be necessary to check the dataset for
outliers or incorrect matches. These outliers can skew the result and must therefore be removed
to ensure a correct motion estimate. Normally, a robust statistical method such as weighted least
squares or a random sampling approach (eg. RANSAC) is used[41].
Features are matched based on their similarity. Repetitive structures and textures can therefore
be problematic, if each feature is not sufﬁciently unique. Incorrect matches are also caused by
effects such as reﬂections, transparency, moving objects. Most of these effects are difﬁcult to
avoid by constraining the pair matching process. Instead, outliers must be detected from a set of
contaminated data. By applying the RANSAC algorithm introduced in the previous chapter, the
inliers can be detected. As an alternative to the iterative method, a point distance constraint can
be used to determine outliers, as described in section (5.1.5). If the point distances are not equal,
one of the corresponding points must be incorrect. This method was proposed in [41] and will
be referred to as the Constraint Satisfaction Method (CSM) throughout this chapter. It will be
possible to select either RANSAC or CSM as the outlier detection algorithm.
5. Motion estimation
After the outliers has been removed, the motion estimate will be computed using SVD with the
matched 3-D points as the input, according to section (5.1.6). The solution is computed as the
rigid transformation (R,t) that aligns the two dataset, with the smallest possible error (5.22). A
weighted solution is also included for comparison. In this case, the disparity for each point will
be used as the weight. Dynamic weighing, as introduced in section (5.1.7) could be a good way
to improve the results further, but it will not be implemented at this time.

6.2.6
Outlier removal
When both feature sets has been matched, it will be necessary to remove outliers. Currently, it is possible
to select either a RANSAC or CSM based outlier removal algorithm. While it was considered to im-
plement the RANSAC algorithm by scratch, the existing function estimateFundamentalMatrix
was used instead. This function is used to compute the fundamental matrix, using a two matched sets
as an input. In the process of computing this matrix, the function can detect outliers using a number of
algorithms including RANSAC. Therefore, this function has been used to detect outliers.
When CSM is used to remove outliers, a consistency matrix will ﬁrst be computed according to
(5.11). Then, the clique with the largest number of consistent matches is detected. The maximum clique
algorithm has not been developed in this project, as the author does not have a very strong backgorund
in graph theory. Instead, a solution by[Avi Singh] has been used.
6.3
Conclusion
In this chapter the MATLAB implementation of the SVO algorithm has been described. The application
supports a range of feature types, stereo matching methods and outlier detection algorithms. While this
makes it possible to evaluate the performance of many different algorithms, it is expected that only a few
of the available options will be relevant for datasets acquired on the UAV. A scale and rotation invariant
feature type is preferred in this case, making the SURF feature detector a suitable choice. The SAD, SSD
or NCC pixel-based stereo matching algorithms are very slow and do not provide sub-pixel accuracy,
unless interpolation is used. The feature based stereo matching is both fast and gives good results,
as long as features can be reliably identiﬁed in both the left and the right image. To make sure the
feature-based stereo matching will not be problematic, each camera must use identical focus parameters
and preferably identical shutter time. Setting the camera focus to be almost identical should not be a
problem but the cameras can only use an identical shutter time, if they are not using auto-exposure. In
most test scenarios with daylight as the primary light source, ﬁxed-exposure is not a good choice. It is
expected that the cameras will use a similar shutter speed, as long as the scene captured by both cameras
have similar lighting conditions.

In this chapter, the SVO algorithm will be tested using datasets acquired using the image acquisition

7.1
Camera calibration
Before any images can be acquired, the stereo camera rig must be calibrated. Camera calibration covers
the process of estimating camera parameters using images of a calibration pattern. The camera paramet-
ers typically include the camera intrinsics, distortion coefﬁcients and camera extrinsics. Camera calibra-
tion is essential, as the camera parameters will be needed to perform image rectiﬁcation and compute the
disparity map for each image pair. The camera intrinsic parameters are used to correct for imperfections
in the camera construction while the distortion parameters are used to correct for imperfections in the
lens construction.
When calibrating the cameras, a known reference is needed. Typically, a ﬂat chessboard is used, as
it is easy to create and easy to store. By photographing a chessboard, where the grid size is known, it is
possible to obtain a mapping from the object plane to the image plane. Both the relative location of the
two objects and the camera projection matrix will be obtained in this way. This is illustrated in ﬁgure
(7.1). It is also possible to calibrate the cameras using a photograph of a 3D object such as the grid
pictured in (7.2). This method was considered, as such a grid is available at DTU, but as the calibration
grid is located off campus, it was never used during this project. Instead, a chessboard was printed and
glued to a whiteboard available in the DTU Space drone lab. This setup is seen in ﬁgure (7.3a). The
main disadvantage with this method is the large number of calibration images needed to obtain enough
corners to solve for the camera intrinsics and distortion parameters. As a rule-of-thumb, 20-30 stereo
images should be captured and the calibration chessboard should be photographed from many different
angles to obtain a ”rich” set of chessboard views. The workings of the camera calibration algorithm is
out of scope in this project but sources such as [19, 40, 36] provide a good introduction to this topic.
When acquiring the calibration images, it is crucial to have proper illumination. As the drone lab is
generally too dark, most calibration datasets have been acquired outside in direct sunlight.
To ensure the cameras have not lost their calibration since last image acquisition, they will always
be calibrated before starting a new acquisition. When a calibration dataset has been acquired, it will be
veriﬁed immediately using the MATLAB Stereo Camera Calibrator application. This application can
be used to estimate the intrinsic and extrinsic parameters of each camera in a stereo pair and can also
be used to ﬁlter poor calibration images, based on the reprojection error. The mean reprojection error
can be used to determine whether the calibration set is of good or poor quality. The calibration datasets
used in this project all have a mean reprojection error of less than 0.5 pixels. Under perfect conditions,
a reprojection error of less than 0.1 pixels has been obtained. This good result required ideal lighting
conditions combined with a manual exposure and almost identical focus used on both cameras.

Figure 7.7: Pair matches, ﬁrst image pair.

7.7
Conclusion
Based on these initial tests of the SVO application, it is clear that there are severe issues with the depth
estimation. It is expected that this issue will cause problems with incorrect translation distances and
rotations. At this point, it is assumed that the poor performance is caused by the dataset and not the SVO
application itself. It is therefore a good idea to test the complete SVO algorithm with an existing dataset
such as the [New College Dataset] to ensure that there are no functional issues with the SVO application.
Using RANSAC for outlier detection is a reliable but slow choice. There are other alternatives,
including the constraint based method but this method has not been evaluated at this point. The outlier
ﬁltering proposed in [26] is a very reliable addition to the outlier removal sequence. If outliers are
ignored by the ﬁrst step, they will most likely be caught by the second step - assuming that they deviate
signiﬁcantly from the median.
Many of the features supported by the SVO application have not been tested at this point. The estim-
ated motion has not been compared to a known ground truth, mainly due to the fact that the estimated
motion is so far off the expected path.

Chapter 8
Discussion & Conclusion
Image Acquisition Platform
During the ﬁrst part of the project, the image acquisition platform was developed. This required the
development of the camera triggering system and the power distribution board. When the different
DC/DC converter options were evaluated, it was concluded that a single 5V/6A DC/DC converter would
be the simplest solution and it was expected to be cheaper than the second proposal. However, as
this DC/DC converter was not available from RS electronics at the time, separate 5V and 12V DC/DC
converters were selected instead. Both of them were selected to support a wide input voltage and a power
rating slightly higher than necessary. This was done to ensure that enough headroom was available, if
additional systems were added to the image acquisition platform.
A simpliﬁed camera synchronisation architecture has been used, as it makes it possible for both
cameras to share the same trigger conﬁguration. This also made the camera synchronisation board
easier to design. The camera conﬁguration is also simpler - especially when the GNSS receiver is
used to generate the trigger pulse as no extra timer conﬁguration will be needed. With the simpliﬁed
camera synchronisation architecture, it is possible to use any type of external trigger signal for the image
acquisition, making the stereo camera system easy to adapt to other payloads or projects.
A set of applications have developed for the image acquisition platform and they have been de-
veloped using a multi-threaded architecture, to ensure that the images can be acquired at a high enough
frame rate. At higher frame rates, all CPU cores will usually be fully loaded. Due to insufﬁcient cooling,
the CPU cores will reach a certain threshold (95◦) and CPU throttling will be enabled, slowing down the
image compression speed by more than half. This bottleneck causes the image storage threads to be un-
able to keep up with the image acquisition threads. If the CPU does not cool down, the thermal throttling
will continue. Eventually, the image queue will outgrow the available RAM, causing the application to
crash. Thermal throttling is apparently a known problem (feature?), according to the [manufacturer]. To
reduce the operating temperature of the Odroid, it will be necessary to improve the cooling solution. At
this point, no other solutions have been investigated.
At this point, the image acquisition platform is operating without issues, as long as very low frame-
rates of 4 Hz are used. This limitation is caused by a combination of poor storage performance and
thermal throttling. The eMMC storage used on the Odroid is insufﬁcient for higher frame-rates, unless
image compression is used. On the other hand, image compression causes the Odroid to overheat and
slow down. It will therefore be necessary to improve the storage solution using an external USB 3.0
storage device or improve the cooling solution with a larger heatsink and/or fan.
It has been concluded that a frame rate above 5 Hz is not possible, unless a different processing
platform is selected. An Intel based four-by-four inch mini PC, the so-called NUC is a very promising
and alternative with much better performance. This system would also be suitable for a real-time stereo
visual odometry implementation, in case this is required in the future.

Chapter 8. Discussion & Conclusion
Stereo Visual Odometry
During the second part of the project, a versatile SVO algorithm has been described and implemented
using MATLAB. The ﬁnal application supports a range of feature types, stereo matching methods and
outlier detection algorithms. While this makes it possible to evaluate the performance of many different
algorithms, it is expected that only a few of the available options will be relevant for datasets acquired
on the UAV.
The SAD, SSD or NCC pixel-based stereo matching algorithms are very slow and do not provide
sub-pixel accuracy, unless interpolation is used. The feature based stereo matching is both fast and gives
good results, as long as features can be reliably identiﬁed in both the left and the right image. To make
sure the feature-based stereo matching will not be problematic, each camera must use identical focus
parameters and preferably identical shutter time. Setting the camera focus to be almost identical should
not be a problem but the cameras can only use an identical shutter time, if they are not using auto-
exposure. In most test scenarios with daylight as the primary light source, ﬁxed-exposure is not a good
choice. It is expected that the cameras will use a similar shutter speed, as long as the scene captured by
both cameras have similar lighting conditions.
A scale and rotation invariant feature type is preferred in this case. Harris-Stephens and SURF
feature types have therefore been selected for the ﬁnal algorithm. SURF has been selected due to its
rotation and scale invariance while Harris-Stephens has been included as a comparison. Both RANSAC
and the Constraint Satisfaction Method (CSM) will be used for the outlier detection step. The reliability
of CSM is not as well documented so the RANSAC algorithm has been included as a benchmark. Using
RANSAC for outlier detection is a reliable but slow choice. There are other alternatives, including
the constraint based method but this method has not been evaluated at this point. The outlier ﬁltering
proposed in [26] is a very reliable addition to the outlier removal sequence. If outliers are ignored by the
ﬁrst step, they will most likely be caught by the second step - assuming that they deviate signiﬁcantly
from the median. The motion parameters are estimated using SVD and both a weighed and unweighed
computation can be used.
Tests and results
Based on the initial tests of the SVO application, it is clear that there are severe issues with the depth
estimation. It is expected that this is the cause of the incorrect distance and depth estimates and incorrect
rotations. At this point, it is assumed that the poor performance is caused primarily by the stereo camera
setup and not the SVO algorithm itself. It is likely, that a poor combination of baseline and focal
length has been used, resulting in very poor depth resolution. The next step will therefore be to test
the SVO algorithm with an existing dataset such as the [New College Dataset] to ensure that there are
no functional issues with the SVO application. Many of the features supported by the SVO application
have not been tested at this point. The estimated motion has not been compared to a known ground truth,
mainly due to the fact that the estimated motion is so far off the expected path.
