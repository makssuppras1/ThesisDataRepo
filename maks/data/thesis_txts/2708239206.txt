Abstract
This thesis proposes an Intrusion Detection System (IDS) that can detect ran-
somware. The system uses honeytokens to detect ransomware activity.
To create the system efficient ransomware first had to be analysed to under-
stand how ransomware encrypt.
For this two test environments were setup.
One using virtual machines and one using real computer and devices. 30 differ-
ent ransomware were analysed. It was analysed how ransomware traverse the
directories in a system and in which order the files are encrypted. Findings were,
that most ransomware use a sort of depth first traversal and most ransomware
follow Windows 1252-ordering. Two of the ransomware changed behaviour de-
pending on the file types, with xlsx files observed as being given the highest
priority. No evidence of any ransomware prioritising based on file size or last
modified/accessed/changed times.
From the analysis an efficient way for creating and placing honeytokens was
found. Honeytokens need to be at the front and back of the root directory and
front and back of the left most and right most leafs in the directory structure.
Finally, an IDS was developed.
With the IDS efficient honeytokens can be
created in a directory. When a honeytoken is encrypted by ransomware, the
IDS will kill the process that performed the encryption. The IDS was able to
stop 90% of the ransomware the first time a honeytoken was encrypted. 2 of
the ransomware was stopped the seconds time a honeytoken was encrypted. A
single ransomware could not be stopped. A mean of 21.49 files were encrypted
with a median of 14 files.

Chapter 1
Introduction
Ransomware is currently a big problem worldwide. Ransomware attacks are on
an upwards trend. With Ransomware as a Service (RaaS) which is a subscription-
based model, where an affiliate can use expertly coded ransomware to execute
attacks and get a percentage of the pay-out, the upwards trend is not looking
to slow down. Some victims chose to pay the ransom demanded from a ran-
somware attack, as it outweighs cost of downtime and consequences of this, for
example loss of revenue or decreased brand value. This makes ransomware very
lucrative for cybercriminals. This project will design an Intrusion Detection
System (IDS) using honeytokens to detect and stop selected ransomware.
First, this project will analyse ransomware in a test setup. The test setup will
use a victim machine, where a file directory can be monitored, and an analy-
sis machine for results. The project investigates in which order a ransomware
traverses a file directory and encrypts the files in the system. It will analyse if
the ransomware has specific priorities when encrypting. As the ransomware will
encrypt most files on the system, the results of the analysis needs to be sent out.
For this an analysis machine on the same network is used. The victim machine
will send the results in real time to the analysis machine.
Secondly, using the analysis of how the ransomware encrypts files, a prototype
of a Host-Based Intrusion Detection System is designed using honeytokens. A
honeytoken is a file with no actual value. Any interaction with the file is consid-
==PAGE:9==
2
ered an attack. The analysis clarifies which directories the ransomware traverses
and in what order, and this is used to place the honeytokens, for them to be as
efficient as possible.
The design of the Intrustion Detection System (IDS) ensures, that once a hon-
eytoken is triggered, the IDS will try to isolate the process that encrypted the
honeytoken. The honeytoken will trigger if a process tries to modify or delete
the honeytoken, thus creating an interaction with the honeytoken which is con-
sidered an attack. The IDS will find all child processes that could have been
created by this process. The IDS will attempt to kill all these processes includ-
ing the main process. This happens each time a honeytoken is triggered. By
killing the processes, the goal is to stop the ransomware from encrypting more
files.
Finally, the IDS is evaluated on ransomware.
A new test setup is created.
The IDS is installed on a victim machine. The ransomware is then executed.
It is then monitored if the IDS is able to stop the encryption process of the
ransomware. If the IDS manages to stop the ransomware a log can be created,
noting what honeytokens were triggered. The log together with a copy of the file
system can be used to see how many of the files in the system were encrypted.

Chapter 2
State of the art / Related
works
Venkatesh et al [26] tries to identify and isolate ransomware in the intrusion
phase. A lot of malware and ransomware attacks come from phishing attacks.
A honeypot is used to detect whether a website is phishing or not, using rule
based analysis. In the honeypot system, the system can download, install and
execute the malicious files present on the website with a direct link to the host
system.
Voter Choice by Keong Ng et al.
[27] also tries to stop ransomware at the
gateway of the network.
They use machine learning and honeypots.
Their
approach is divided into different components. The first component is block and
redirect all downloads to the gateway. The second is using static detection. This
static detection is used to detect whether it is ransomware. The third component
is using dynamic detection. This is where a honeypot is used. A honeynet, which
is a system containing one or money honeypots, is used to capture all system
activities in a log. The log is analysed to detect any encryption or other file
features. This is used to classify the ransomware family.
Paxcariu et all [28] also uses honeypot to detect ransomware activity. A Rasp-
berry Pi is setup as the honeypot. The Raspberry Pi is setup as a network
share in the system. All actions on the files on this network share are logged.
A periodic check on the files is performed. If any files have been modified, then
==PAGE:11==
4
the file data is collected together with the log data. An alert is then sent out.
Lee et al [29] performs an analysis of ransomware. This is done by analysing
the source code of found ransomware. They analyse the encryption method of
the ransomware. It is investigated if ransomware uses write-in-place, rename-
and-encrypt or create-encrypt-and-delete for the encryption method. They also
analyse in which order the ransomware encrypts. Following Windows-1252 or-
dering or reverse. Additionally, it is studied if the ransomware traverses in the
order of file or folder size or specific attribute values like last accessed time of
the file. In the end they give a suggestion from the analysis of the source code
on how to create efficient decoy files for detecting ransomware.
The paper by Moore [30] proposes a honeypot method to detect ransomware
activity. Microsoft File Server Resource Manage and EventSentry are used to
monitor Windows security logs. A honeypot folder is monitored with this. A
staged response is setup to respond to the attacks with thresholds. For normal
activity there is no action. The first event is logged, and an email is sent. On
the second event the account is disabled. On the third event the networking is
disabled and on the fourth the server is disabled. The system was not tested
with live ransomware, but with a simulated one encrypting files.
El-Kosairy and Azer [31] propose an IDS using honeytokens. First, they identify
the important properties for creating efficient honeytokens. These properties are
believability, veriablity, entichingness, non-interference, conspicuousness, differ-
entiability, detectability and shelf-life. Then an IDS is created using File System
Event Handler watcher. Both high and low-interaction honeytokens are spread
out with a positioning technique. This technique is based on ASCII code values.
Once a honeytokens has been triggered, an email is sent, and it is logged with
a message. The IDS was tested with six different ransomware families and was
let run for 15 minutes. They noted the time it took for the system to detect
the ransomware. They also tested two different methods for detection. One
is comparing the hash values of the files to see if they changed. The other is
watching the entropy of the files. These methods were found to have a slower
detection time than using the Event Handler Watcher.

Chapter 3
Theory
3.1
Malware
Ransomware is a type of malware. Malware is a term that is used to describe all
sorts of different malicious programs and code. These malicious programs try
to bring damage or steal data from the device that is has infected [1]. Malware
can infect a system in multiple ways. This could be a person inserting a USB-
drive with the malware that will automatically run the malware or if a person
clicks a malicious link in an email. Several malware types exist, below the most
common:
• Virus - A more general word used for malicious code or program. The
intent of the virus is to alter the ways the computer operates, reducing
the computers processing power so it will run slower. It could also be
altering/deleting files, causing unexpected behaviour and damaging effects
[2]. The virus will try to spread to other hosts.
• Worm - A computer work is a type of malware that can replicate it-
self onto other computers. A worm will usually spread across a network
through the internet connection. A worm will typically exploit vulnerabil-
ities in the software to infect and spread. The difference between a worm
==PAGE:13==
3.2 Ransomware
6
and a virus is that a worm can operate on its own where a virus needs a
host computer to run. [3]
• Adware - Adware is software that will display unwanted ads and popups
on the infected computer. Developers of the adware will gain revenue by
displaying ads. [4]
• Spyware - Spyware is designed to infect a computer and gather personal
information. This personal information is then forwarded to a third party
without consent. The data compromised by spyware could be login cre-
dentials, credit card numbers and browsing habits. [5]
• Trojan horse - A trojan horse is malicious software where the user is
tricked into downloading what seems to be a non-malicious program, while
not realising that there is a trojan horse bundled with the program. A
trojan horse can give full control over the infected computer and can install
additional malware. [1].
• Backdoor - A backdoor is any method where an authorised or unautho-
rised user can get around the normal security measures that exist. The
backdoor can then be used to steal data or install new malware. Not all
backdoors are malicious. A backdoor can be installed into a program in
purpose. For example, if a customer is locked out of a system, and need a
way to get around the normal security measures [6].
• Rootkit - A rootkit is designed to provide continued privileged access
(root access) to a computer while staying hidden. A rootkit is not the
same as a backdoor, as backdoor is a broader term. A rootkit can however
be used to open a backdoor, which will allow a malicious actor into the
system [7].
• Keylogger - Keylogger is a program that records all typed strokes and
commands on the system. This could be passwords, account numbers and
more that can be recorded and used.
• Ransomware - Ransomware will lock or encrypt the victim’s system and
demand a ransom for the system to be unlocked again. Ransomware is
the focus of this thesis and will be described further in section 3.2.
3.2
Ransomware
Ransomware is a type of malware 3.1. The goal of ransomware is to extort its
victims to pay a ransom. Ransomware has evolved a lot since it first appeared,
==PAGE:14==
3.2 Ransomware
7
and it continues to do so every year. The following gives a brief overview on its
evolving and the known features of different ransomware.
The first known ransomware is known as the AIDS Trojan or PS Cyborg back
in 1989 [32]. A Harvard-trained biologist sent out 20,000 infected floppy disks.
They were labelled "Aids Information - introductury Diskettes". These diskettes
however, had a trojan in them. After 90 reboots of the infected computer the
directories on the computer would be hidden and the file names were encrypted.
The actual files were not altered. However, since knowledge of these types of
attacks was limited, this was not clear. If a user then wanted to regain access
they would have to pay 189$ to a post office box in Panama.
3.2.1
Widespread ransomware
Even with the long history that ransomware has, it did not become widespread
until around 2011. Some ransomware existed in the beginning of the internet
age, one is the GPcode ransomware that spread via en email attachment in
2006 [33].
The reason why ransomware did not become more widespread is
likely due to the difficulties with payment collection. The money had to be
collected and somehow kept hidden from authorities. The emergence of Bitcoin
and other cryptocurrencies in 2010 opened an easy and untraceable way to
receive a ransom payment, and ransomware became very lucrative.
Around 2010 locker-ransomware started appearing. Locker-ransomware works
by locking the infected machine so basic computer functions do not work. A
screen will be shown to the infected user demanding a ransom for the machine
to be unlocked again. The locker-ransomware does not aim at destroying files.
The files will not be encrypted, the user is just locked out. It is therefore usually
possible to recover files or remove the lock without any destruction of files.
Around 2013 crypo-ransomware became more widespread. Crypto-ransomware
encrypts files instead of locking the infected user out. The encrypted files cannot
be opened or used. The goal of crypto-ransomware is not to interfere with basic
computer functions.
The goal is to force the user pay a ransom to get the
encrypted files opened. As the files are encrypted using an encryption algorithm,
a key is needed for decryption. This makes crypto-ransomware more destructive
than the locker ransomware.
When referring to ransomware in the following, it refers to crypto-ransomware.
In 2013 the CryptoLocker ransomware appeared being a crypto-ransomware.
When infected, the ransomware would contact and command a control server
==PAGE:15==
3.2 Ransomware
8
(C&C server) to generate a 2048-bit RSA key pair. A C&C server is a computer
controlled by the attacker. It is used to send commands to the malware, that has
infected the system. It can also be used to receive data from the malware. Once
the key-pair is generated, the public key would be sent to the infected machine.
The infected machine would use this public-key to encrypt files across local
hard-drives and mapped network shared. Only data files with certain extensions
were encrypted. This were files like Microsoft Word-documents, pictures and
more. When the files have been encrypted the payload would display a message
demanding the ransom. There would often also be a time limit within which
the ransom had to be paid or else the private key would be destroyed. If the
ransom was paid, a decryption tool could be downloaded which had the private
key to decrypt all the files.
Cryptolocker would spread through email attachments. An executable attach-
ment was made to look like a pdf file. The victims would then click the attach-
ment and the ransomware would be executed. Quickly after the appearance of
CryptoLocker a lot of clones started to appear.
The following years CryptoLocker ransomware exploded. In 2017 ransomware
like WannaCry appeared. These ransomware have worm-like techniques that
can make them spread worldwide. This commonly happens using system vul-
nerabilities that the ransomware can abuse. WannaCry worked by exploiting a
vulnerability in the Windows Server Message Block also known as SMB. This
exploit is known as EternalBlue. This gave the ransomware unrestricted access
to any computer running an unpatched version of Windows. This also gave
WannaCry worm capabilities. It used the vulnerability to automatically propa-
gate between computers on the same local area network [18]. It used port 445
to install the ransomware and spread to other computers. It also communicated
with a C&C server on the TOR network.
3.2.2
Double ransom and RaaS
Instead of trying to spread as much and possible, focus changed to target big
companies. This is also known as "big game hunting" [37]. The attackers try to
target a big company in the ransomware attack. A machine might get infected
with malware but not run ransomware initially. The actors will try to perform
some lateral movement to get more access so more files and documents can get
encrypted. The idea of "big game hunting" is to focus on few targets with large
financial payoff.
After ransomware became very big, more emphasis amongst companies was
placed on backups and how to restore the system, if it was hit with ransomware.
==PAGE:16==
3.2 Ransomware
9
This made the attackers change their approach to use double ransom.
The
ransomware encrypts all files and demands ransom.
An extra layer is then
added on top of this. If the ransom if not paid, then all the encrypted documents
are published online [34]. It can become very critical if a victim has secret or
sensitive information. The victim might end up paying anyway although having
proper backup of data.
The most recent trend in ransomware is Ransomware as a service (RaaS) [35].
This works as a subscription model. First, expertly coded ransomware is devel-
oped. The developers then distribute this ransomware to affiliates. These affili-
ates then spread the ransomware and try to infect victims with the ransomware.
The affiliates then earn a percentage of the successful ransom payment.
Even though ransomware is very widespread, it is difficult to get precise infor-
mation about how ransomware behaves. Reason being that most ransomware
have anti-analysis behaviour [46]. If software used to analyse malware is de-
tected, the ransomware might not run or in some cases give false information
about how it runs. The same is the case if it detects, that it is running in a
sandbox environment. These features make ransomware more complicated to
analyse than traditional malware.
3.2.3
Encryption methods
There are 3 methods commonly used to encrypt files. These methods are Write-
in-place, Rename-and-encrypt and Create-encrypt-and-delete [15].
Write-in-place
works by first creating a new temporary file. The data in
the original file is then read. This data is then encrypted and written to the
temporary file. The data in the temporary file is then read and written back to
the original file and the temporary file is deleted [16]. In some cases, memory
can be used instead of a temporary file, but it will still be in the write-in-place
category.
Rename-and-encrypt
works by changing the file name to be encrypted to
the different name, encrypt the file the same way as Write-in-place and change
the name back to its original name [15].
==PAGE:17==
3.2 Ransomware
10
Create-encrypt-and-delete
works by first creating a new file. The data is
then encrypted and written to the new file. The original file is deleted.
3.2.3.1
Encryption order
The ransomware will also have to find the files before encryption. The directory
structure has to be traversed. For this the ransomware can use common searches
to find the files. The directory structure is similar to a tree, where each file or
subdirectory is a child. These are then ordered alphabetically from left to right.
To find all files, the tree can be traversed in a specific way.
Depth-first search (DFS)
In depth first search the algorithm look as deep
as possible in a child before going to its sibling. The root node is first visited.
Then the left most nodes tree is recursively traversed, then traverses from left
to right for all the children. There are then three different ways in which order
each node in the tree is visited.
Pre-order
1. The current node is visited
2. The left most subtree is recursively traversed
3. The next subtree is recursively traversed until all subtrees from left to
right have been traversed
In this order each parent is always visited before any of its child nodes.
Post-order
1. The current nodes subtrees are recursively traversed from left to right
2. The current node is visited
Here each subtree is traversed first before the current node is visited.
In-order traversal
1. Recursively traverse the current nodes left subtree
==PAGE:18==
3.2 Ransomware
11
2. Visit the current node
3. Recursively traverse the current nodes right subtree
If looking at the directory structure where both files and directories can be nodes
then ordering will not matter. It will not change the order in which the files are
visited as only directories can have children.
If instead only the directories are nodes and the files in a directory are encrypted
when that directory is visited, then the ordering type does change the order in
which files are encrypted.

and directories are nodes. The number indicates in which order
the files would be visited in a depth first search.

on which point they are accessed if using depth-first search. It is seen that the

3.3
Intrusion detection system
An intrusion detection system (IDS) is a device or software application that
monitors a network for malicious activity or policy violations [24]. If something
malicious is spotted it will be collected in a central system. Even though IDS are
used to spot malicious activity and policy breaches, they are all prone to false
alarms. This means the IDS needs to be setup and tuned to fit the network it is
running in. IDS can be classified into several different types, described below.
3.3.1
Types of Intrusion Detection Systems
Network Intrusion Detection System
works by inspecting traffic on the
network. The system will be set up a specific place on the network. All traffic
from all devices is collected. All the traffic will be matched and compared to
known attacks or anomalies [25]. If an attack or anomaly is spotted an alert
will be sent. This could be to a system administrator or a monitoring system.
A classic example of an intrusion system is one that is placed at firewalls. All
traffic from the firewall is monitored and if someone is trying to brute force or
scan the firewall, an alarm will be sent out.
Host-based Intrusion Detection System
inspects all locally available in-
formation. This can be application log-files, system log-files or running activities
and processes. Instead of running on the network this intrusion system runs in-
dependently on the hosts or devices on the network.
Protocol-based Intrusion Detection System
is usually a system or an
agent in the front end of a server. This system will control and interpret the
protocol used between a user and the server it is protecting. The system will
monitor the behaviour and state of the protocol. A common use of the system is
to monitor the HTTP stream of a web server. It will monitor requests with its
corresponding response; every request should have a predictable response and
those responses that fall outside of expected results will be flagged and analysed
further [38].
Application protocol-based Intrusion Detection System
will be within
a group of servers. It will analyse and monitor application specific protocols.
This could be the SQL protocol used for interacting with databases in the web
server.
==PAGE:21==
3.3 Intrusion detection system
14
Hybrid Intrusion Detection System
is a combination of two or more of
the system described above.
3.3.2
Methods used in IDS
IDS have two main methods used for detection of attacks or anomalies.
A
signature-based method and anomaly-based method.
Signature-based method
will try to match specific known patterns. This
could be the hash value of the malware. This type of detection methods requires
the malware to already be known. It can therefore not be used to find new
attacks.
Anomaly-based method
will establish a base line for normal communica-
tion. It will then try to detect abnormal traffic patterns. Machine learning,
pattern matching and big data is commonly used to help detect. This can be
used to detect new attacks that do not have a specific signature, that can be
used in the signature-based method.
An IDS differs from a firewall. IDS focuses on looking outward for intrusion and
stops them from happening. An IDS will raise an alarm if suspected behaviour
is spotted. A firewall on the other hand restricts access between networks to
prevent intrusion. If there is an attack on the inside of the network, the firewall
will not catch this.
3.3.3
Logs
A company can have many different IDS. These IDS can send logs so the com-
pany can know if something critical is happening. Some IDS’ also use logs to
determine if there should be an alert. These logs could be audit logs.
An audit log is a record of events and changes. Devices will create logs based on
these events. When an activity is performed it usually logs who did the activity,
what was done and how the system responded. This could be a user modifying
a file and the system responds with success. Not all audit logs operate in the
same way. This depends on the devices, application and operating system [39].

3.4
Honeypot
A honeypot is a security mechanism used to detect and deflect attacks and to
study new techniques used by attackers. This is needed as attacks continuously
evolve, and a static detection of new attacks is not feasible. Automated attacks
based on machine learning are increasing. This means that adjusted defence
methods are required [41].
There are multiple types of honeypots all depending on how they are used and
placed. They all serve the same purpose. To detect intruders and analyse their
behaviour. All interactions with the honeypot are considered an attack. No
legitimate user would ever access a honeypot system.
A classic honeypot is a single host or network device that will try to lure poten-
tial attackers. This is done by making the honeypot look like a real target, that
could be attacked. This could be a server attached to a company’s network.
This server would look like it is part of the company network, it is however
designed to log and alert if someone tries to do something malicious with the
server.
3.4.1
Honeytokens
A honeytoken is a type of honeypot. However, instead of being a computer
or a physical resource, a honeytoken is a resource that the malicious actor can
interact with. A honeytoken can be any file on a system. It can be an excel
file or pdf. A honeytoken does not only need to be a file type. It can also be
a credit card number or login credentials. These resources are monitored. As
with honeypots all interactions with the honeytoken is considered an attack.
The honeytokens do not have any authorized use [42].
[43] show a concept of perfectly believable decoys.
They proposed a set of
properties to help design the honeytokens.
==PAGE:24==
3.4 Honeypot
17
1. Believability The attacker needs to believe that the honeytoken is au-
thentic. The attacker will not use the honeytoken if it does not appear
authentic.
2. Enticingness The honeytokens need to be look attractive enough for the
attacker to interact with it. If the honeytoken is not attractive, then it
will not get accesses and no attack can get detected.
3. Conspicuousness It should be easy to locate and retrieve the honeyto-
ken. This is to maximize the likelihood that a malicious actor will take
the bait and access the honeytoken
4. Detectability The honeytoken needs to be detectable. If it is not then
it is useless
5. Variability It should not be possible for an attacker to identify all hon-
eytokens due to some shared value or invariant. That way the attacker
would be able to steer clear of all the honeytokens and they would bring
no value.
6. Non-interference The honeytokens should not interfere with normal
users’ activity. This is the likelihood that a legitimate user would interact
with the honeytoken. All interaction with a honeytoken is considered an
attack it is therefore important to reduce the likelihood that these false
alarms happen.
7. Differentiability It should be easy for users to distinguish between real
documents and honeytokens.
8. Shelf-life The honeytokens may have a limited time period in which they
are effective. The malicious actors will with time learn about them.
It can be difficult to maximize all the properties. There has to be trade-offs
between the properties to create the most efficient decoys for the system they
are to used in.
The honeytokens can be divided into two overall file types, high interaction
honeytokens and low-interaction honeytokens.
High-Interaction
These honeytokens contain fake critical data to confuse
the attacker. This could be some login credential or similar to another server
which could be a honeypot.
The attacker can then be lured into this other
deception system to get more information about the attacker.
==PAGE:25==
3.4 Honeypot
18
Low-Interaction
These honeytokens contain random data. This is to detect
any automated attack or attacks that just traverse everything. The data does
not have any value for the attacker.

4.4
Creating efficient honeytokens
The honeytokens need to created so detection can be as fast as possible without
creating honeytokens in each directory. Figure 4.17 shows how the honeytokens
should be placed.
Figure 4.17: The figure shows efficient placements of the honeytokens in a
directory structure
The placements in 4.17 is derived from the analysis above. A honeytoken needs
to be one of the first files that are encrypted by the ransomware to be as efficient
as possible. This way the honeytoken will be encrypted before the real files, and
the system will have time to react.
If the ransomware encrypts using depth first search where both files and direc-
tories are nodes, then the honeytoken A should be the first file encrypted if the
ransomware encrypts the files ordered, or D if the ransomware encrypts the files
reverse. If the ransomware uses a reverse depth first search, then E and F are
the first tokens to be encrypted.
If the ransomware encrypts using depth first search where the directories are
nodes, then there are different scenarios depending on the ordering the depth
first search uses.
• Pre-order-traversal: Will encrypt A first as the home directory is visited
first
• In-order-traversal: Will encrypt C first
==PAGE:61==
4.4 Creating efficient honeytokens
54
• Post-order-traversal: Will encrypt C first
In a breadth first search the ransomware will encrypt the honeytoken A first.
If it encrypts the files in reverse, then the honeytoken B should be the one
encrypted first.
These placements should therefore cover all the found traversal types from 4.3.3.
To get a honeytoken like honeytoken A, the honeytoken should be placed in
front of all other files in the directory according to the ordering used. This can
be achieved by using the first symbol in the ordering which is most often !.
The opposite is to get the B honeytoken. The letter ÿ is the last in the ordering
and can be used to get this placement.
For the honeytoken file types, file types such as xlsx, docx, pdf should be used.
They are not identified to be excluded by any ransomware.
Xlsx might be
to prefer since WannaCry encrypt these files first and other ransomware also
encrypt xlsx as one of the first files.
To detect ransomware that use the create-encrypt-and-delete encryption method,
if a file is created with the same name as the honeytoken but with a different file
extension, the honeytoken should also trigger. Only one set of efficient honey-
tokens are created. The proposed honeytoken placements cover all of the found
traversal types that ransomware use. In this placement strategy the honeyto-
kens are the first files to be encrypted. Any other strategy would encrypt more
files, when ransomware that traverse random is not considered.

5.3.2
Creating honeytokens
The honeytokens need to be created at the most optimal file locations as anal-
ysed in 4.4. The honeytokens are created when the button to create honeytokens
in the application is pressed.
The honeytokens are created in each subdirectory of the chosen directory. Ini-
tially, all the subdirectories in the directory are found. This is done using a
python function called os.scandir(path). This function checks if the found
entries are directories and saves them in a list. All subdirectories where honey-
tokens are needed are then identified.
Trees for each of these subdirectories are initialized so they can be traversed.
For each subdirectory it is necessary to find where to place the honeytokens.
The first location is found by performing a postorder traversal of the directory
tree. The first element is saved. The second location is found by performing a
preorder traversal and taking the last element met in the traversal. This gives
the front and back locations required from the analysis. Last the given path is
added to the placements. By this, there are decoys as high up as possible in the
directory tree.
After all the locations are found, duplicates are removed. If a directory only has
one subdirectory it will get registered twice. There is however no need to have
extra of the same honeytokens at the same location.
The honeytokens are now created.
The application has the required files to
create the honeytokens together with the application executable. These are the

All the files are called honey.* where * is their respective file extension.
More than one honeytoken is generated in each placement folder. There needs

5.3.3
Audit log
Now that the honeytoken placements are found and the files are created, they
need to be monitored because they are honeytokens. In this application this is
done using Windows audit logging.
By using the python package winevt [45] it is possible to subscribe to new events
within a specific query. This means that if a new event is logged within that
query the subscription will run a callback function.
The application is required to give a notification and kill processes if a honey-
token is deleted or modified. The application therefore subscribes to the event
with EventID 4663 as that event is the event that happens if a file is deleted or
a file is modified, as described in 3.3.3. These events are in the Security event
log. It is in Event/System from the xml of the event log that the EventID can
be found. The query for the 4663 event is "Event/System[EventID=4663]".
When a new event happens the subscription will run the callback function.
Algorithm 2 Honeytoken Modified
if event.filename == honeytoken then
if event in access_mask then
for child in event.process do
child.kill
end for
process.kill
end if
end if
Algorithm 2 describes what happens when a honeytoken is modified. This is
triggered when a new event with EventID 4663 is created. First the algorithm
checks if the event is for a honeytoken. If it is not then the algorithm ends. If it
is a honeytoken it checks if the access_mask of the event is one of the ones that
are critical. These are the ones described in 3.3.3. If that is the case, then the
process that triggers the event is found. All the child process that event might
have is found are killed and afterwards the process itself is also killed.

Discussion
7.1
Priorities
The testing of ransomware used six different file types. These are considered
commonly used file types. Only a few exclusions for the selected file types were
found, e.g. txt, db and png. Several ransomware did also prioritise xlsx files
types as one of the first file types to be encrypted. However, many file type
exists globally. Testing all different file types would create a large number of
tests to be run. Unless access to source code is possible, it is impossible to
identify how ransomware behave with all the different file types. There could be
a file type not part of this analysis, that the ransomware always will encrypts
first. Creating a honeytoken with exactly this file type would then be optimal
for the IDS.
Some ransomware change behaviour if run in a specific region like Russia. The
tests performed with the analysis and IDS were all done from Denmark, and
all tests were from the same location and IP. It is therefore unknown if the
ransomware would change behaviour if the location was changed. It is a general
trend that all the specific priorities the ransomware has cannot be found unless
there is access to the source code. The ransomware could change encryption
method depending on the day, region or anything. This cannot be known for
sure due to how obfuscated most ransomware are and how complicated it is to
==PAGE:82==
7.2 Sample size and access to ransomware
75
test and analyse. The conclusions on ransomware priorities are therefor limited
to scenarios selected.
7.2
Sample size and access to ransomware
The analysis and the evaluation of the IDS used 30 different ransomware. This is
not a large sample size. Several other ransomware that do not follow the found
encryption methods can exists. The nature of ransomware however makes it dif-
ficult to get hold of many ransomware to analyse and test. There is no official
dataset or an official place to download ransomware from and be sure that the
claimed ransomware is actually the ransomware that is downloaded. The ran-
somware used in this test are downloaded from the website app.any.run. These
ransomware are user submitted. Any person could upload a ransomware to this
website. It is therefor not certain that ransomware marked as "WannaCry" is
the original WannaCry.
7.3
Continuous evolution
Ransomware has evolved fast the past years. It will likely continue to do so.
For any IDS to efficient, it is important to continuously conduct analysis of
ransomware. Subsequently, honeytoken location strategies can be updated and
adapted to the newest ransomware trends.
Once developers of the ransomware understand methods for intrusion detection,
they will undoubtedly adapt ransomware to avoid the honeytokens. The created
honeytokens in this IDS have very high detectability and low variability as
described i 3.4.1. The created honeytokens all start with ! or ÿ. This makes
them very easy to detect and avoid. The reason why an IDS is successful, is
because the developers of the ransomware have not adapted the ransomware
to avoid these tokens. Once a single honeytoken is found, it will be easy to
detect the rest. The honeytokens of the same file type are all identical except
2 different types of docx. The ransomware could be checking the hash value of
the file and compare if a file is the same as the found honeytoken. With this
the ransomware can avoid the honeytokens.

7.4 Speed of ransomware
76
7.4
Speed of ransomware
The analysed ransomware encrypt very fast. For some of the test with smaller
directory structures the ransomware finished encrypted within 2 seconds. The
IDS has a small delay before the ransomware process is killed and completely
stops.
If this delay is minimised by just a few milliseconds there could be
multiple files are not encrypted. Small improvements in the IDS speed can give
a large reduction in amount of files encrypted. This IDS was not build with
speed in mind. It uses the Windows built-in event log function to get the event
that tells a file has been modified. There could be a faster way to get the events
that the files were modified. The IDS is also written in Python which is not
known for its speed compared to other programming languages like C and C++.
7.5
Tested on single machine
The analysis is run on a single machine. The same is the case when the IDS is
evaluated. If the IDS is to be used just by a single user, it would work. However,
as more ransomware perform Big Game Hunting to get more profit, it would
be ideal if it also worked in a company. Companies have many of their files on
file servers. It should be tested if IDS can run only on the file server or if it
should run on every user in the network. The analysis and IDS are evaluated on
a directory structure. A ransomware might start at specific common Windows
directory, for example Documents or Downloads. This situation has not been
analysed. This is also less relevant when looking at company settings, where
most of the files would be on a file server. Consequently, the analysis should be
precise in how the selected ransomware traverse a directory structure.
7.6
False alarms
The implemented IDS does not trigger on every event where a file has been
modified. It looks in the access mask values to check, if it should trigger. This
eliminates a lot of false alarms. Windows services perform different indexing. If
the IDS would trigger on every event and on every access mask value, then it
would trigger frequently and kill non malicious Windows services.
However, if a regular user modifies a honeytoken then the IDS will trigger, as
every action performed on a honeytoken is considered an attack. This would
make the program the user used to edit the honeytoken crash, which is not
==PAGE:84==
7.6 False alarms
77
critical. If the IDS is running on the machine an alert will also be shown in the
notifications, which will show the user that they modified a honeytoken. If the
IDS is to be used in a company setting, it is not setup to send a notification to
any system administrators. It would be ideal to do so, such that false alarms
can be differentiated from real ransomware.

should be used to analyse location of further honeytokens. This way, even if a
ransomware is encrypting using a random traversal or does an unknown traver-
sal, there will still then be higher probability that the ransomware encounters a
honeytoken early in the traversal.
8.2
Handling false alarms
The current program handles false alarms by only trigger on specific access mask
values in the event log, that it is subscribed to. Further false alarm handling
can be implemented. This can be done by waiting for a second honeytoken to be
==PAGE:86==
8.3 Robustness
79
triggered in a small timeframe. This requires all honeytokens to be paired up,
so there is a honeytoken with another honeytoken right next to it. Otherwise
too many files would get encrypted before a second honeytoken is encrypted.
The program should also send a notification to system administrators or other
roles in charge of monitoring. This way false alarms can be handled and looked
at, making sure it is not ransomware that has triggered the honeytoken.
Employees in a company should also be educated so they do not delete or modify
honeytokens.
8.3
Robustness
The designed IDS runs as an executable. One ransomware was able to kill the
created process of the program so the monitoring could not continue. A future
work could be to make the program more robust, so it is difficult to kill the
process running the system. Further robustness can be reached by hiding that
the IDS is running, ensuring that the ransomware cannot identify what to watch
out for.
8.4
Optimise
As described in 7.4, the speed of the detection system from detection to killing
the process can be critical.
Few milliseconds can be the difference between
many files getting encrypted or not. An important further work is to optimise
the system so the detection will be faster and the time from detection to killing
all the processes is minimised.
8.5
Testing Environment
To further develop the system, the testing environment should be optimized.
This system was tested on real machines on real network. To further develop
the environment more devices should be added. It should be monitored how
to detect the ransomware, when it is not just spread from an executable, but
spread using worm-like capabilities, that some ransomware have.
==PAGE:87==
8.6 Honeytoken Properties
80
The testing environment used system images on a separate ssd to restore the
machine back to original state. This was very time consuming between each
test. The machine had to be turned off and the ssd manually plugged back in
the machine. The system would get restored and the ssd had to be plugged out
again. Automating these tasks would make it faster to test the ransomware in
the testing environment. This would help both further analysis of ransomware
and the IDS.
8.6
Honeytoken Properties
Another further work is to improve the honeytoken properties that are described
in 3.4.1. A method for creating the honeytokens dynamicly with different con-
tent and different names should be found. This would obscure the honeytokens,
hereby it will be more difficult for ransomware to identify the honeytokens. If
the ransomware is not able to properly identify the honeytokens then the IDS
will work better.
8.7
Useability of the IDS
The IDS currently only has simple functions like stop, pause, create honeytokens,
create a log and exit. Further work should allow a user to monitor the current
created honeytokens to get an overview of where there exists honeytokens. A
user should be able to update honeytokens or delete honeytokens. This can be
done in a type of dashboard.

Chapter 9
Conclusion
The purpose of this project was to design an Intrusion Detection System (IDS)
using honeytokens to detect and stop selected ransomware. To achieve this,
ransomware behaviour was first analysed. The analysis investigated the order
and the priorities in which selected ransomware encrypt files. The output of the
analysis was used to discover where the best locations to place the honeytokens
were. Based on this, an IDS was created with the honeytokens in the found
locations. When ransomware encrypts a honeytoken, the IDS was designed to
kill the process, that performed the encryption.
Two test environments were created for the analysis. The setup used a victim
machine, on which the ransomware was executed and an analysis machine where
the logs were sent to.
First, a test environment using virtual machines was
setup. However, due to the nature of ransomware, not all ransomware ran in
the virtual environment. To counteract this, a testing environment using real
machines and devices were setup. This setup granted all devices full access to
the internet and did not use any virtualisation. All ransomware would run on
the testing environment using real devices.
The analysis of ransomware encryption method was not limited to which order
the ransomware traverse, but also looked at how ransomware prioritise specific
file types and if the ransomware prioritised file size or the last change/modi-
fied/accessed time of the file. Six significant ransomware were analysed in a
==PAGE:89==
82
baseline test, to give an overview of the different methods ransomware used. A
further quantitative analysis included 24 additional ransomware. In total the
analysis included 30 different ransomware.
It was found that 23 (77.66%) of the ransomware encrypted following a sort of
1252-ordering. 1 ransomware (3.33%) had a random ordering. 6 of the ran-
somware (20%) was not tested for ordering, but they did all encrypt alphabeti-
cally when only looking at letters. 27 of the ransomware encrypted forward, 2
encrypted in a random order and 1 backwards when looking at the directories.
When encrypting files in a directory, 25 of the ransomware encrypted the files
following the ordering forward, 1 backward and 4 could not be concluded to
encrypt the files in a directory neither forwards not backwards.
It was found that 26 of the ransomware used a type of depth first traversal. Of
these 18 used a depth first traversal where both files and directories are usedm
as nodes. 8 of them used a depth first traversal where only the directories were
used as nodes. One of these performed the traversal in reverse. 2 ransomware
traversed using an unknown traversal and 2 traversed using breadth first search.
The most common type of traversal appears to be depth first traversal.
2 of the ransomware encrypted specific file types before other file types. 5 of the
ransomware excluded specific file types, with txt file being the most excluded
type. File prioritisation and exclusion can be observed in ransomware.
No ransomware changed behaviour depending on the size of the files or when
the last changed/modified/accessed time was. Thus, size and last modification
are not found as significant input to behaviour of ransomware.
Using the results, a method for creating efficient honeytokens was created. Hon-
eytokens should be placed in the front and back of the root directory. There
should also be honeytokens at the front and back of the most left and most right
leaf in the directory structure.
An prototype for a Host-Based IDS was created. The IDS would create hon-
eytokens using the results from the analysis, ensuring the honeytokens were
placed efficiently. When a honeytoken was modified, the process that modified
the honeytoken should be killed.
The testing environment using real devices was used to evaluate the IDS. The
IDS stopped 27 (90%) of the ransomware the first time a honeytoken was mod-
ified.
It stopped 2 ransomware (6.66%) the second time a honeytoken was
modified. One ransomware (3.33%) was not stopped. It can be concluded that
the IDS was successful in detecting ransomware for the majority of ransomware
in scope.
==PAGE:90==
83
When using honeytokens to detect ransomware, there will be real files encrypted
as well. The mean of files encrypted was 21.49 files. There was however two
ransomware that had extreme values. The median of files encrypted was 14
files. One ransomware only had a single file encrypted. The maximum of files
encrypted of not looking at the ransomware that was stopped was 131 files.
Overall, the designed IDS was successful in killing the processes that were en-
crypting files for a significant number of the ransomware. The analysis was thus
effective in giving design input for the IDS.
