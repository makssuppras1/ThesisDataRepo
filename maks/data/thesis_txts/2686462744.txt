Abstract
Although frequently used, passwords have proven to be a vulnerable approach in authentication pro-
tocols.
This has motivated the widespread use of multi-factor authentication protocols, providing
additional security and robustness in authentication schemas. In order to easily describe and auto-
matically analyze these protocols, this thesis introduces the high-level protocol specification language
SPX, which can be used to describe multi-factor authentication protocols, and translate these descrip-
tions to a lower-level language used for automated security protocol analysis. SPX includes a succinct
syntax to describe protocol related exploits in a general manner, and defines a semantics for this ex-
ploit model, allowing it to analyze weaknesses in multi-factor authentication protocols. Based on the
thorough definition of SPX syntax and semantics, this thesis provides a compiler implementation for
the language which is able to automatically translate from SPX to a lower-level specification language
for protocol analysis. The translation is proven useful based on results from testing the compiler on
different protocols with known vulnerabilities.

CHAPTER 1
Introduction
The use of online services grows continuously every day.
Many of these services use some kind of
authentication protocol in order to interact with the user and keep the user’s data secure. The most
common form of digital authentication is the use of a password or a PIN. Used alone this single-factor
authentication method is considered weak, as sharing the password can compromise the user’s account
without further action. Several different techniques exist in order to obtain passwords for unauthorized
use, including phishing attacks and password cracking. Where phishing involves attempts to trick users
to give up their password, and password cracking is a procedure used to recover passwords from stolen
credentials data. This calls for more sophisticated authentication methods and protocols [1].
Authentication can be performed using other factors than a password. These factors can be catego-
rized in three groups:
• Knowledge: Something the user knows. Commonly a password or a PIN.
• Ownership: Something the user has. A smartphone, a key card, or other physical token.
• Biometric: Something the user is. Biometric data such as voice, fingerprint, or face.
Putting together multiple factors from the different groups increases the robustness of authentication
protocols in many cases, introducing the concept of multi-factor authentication [1].
A well known example in Denmark is the integration of NemID[2] in many danish web services
dealing with sensitive personal data. NemID couples username and password login with either a key
card, a hardware authenticator, a USB token, or a smartphone app [2]. But with all these different
approaches how can one verify that such a protocol is secure and how much it takes to break the
security?
Different tools exist which can be used to analyze and verify security protocols. Traditional security
protocol analysis focuses mainly on establishment of secure channels, whereas the analysis of multi-
factor authentication protocols assumes that the communication happens on already secured channels.
The interesting part of the analysis thus becomes figuring out under which threat scenarios the protocols
are still secure and what it takes in order for them to break [3]. It is possible to express these types
of protocols with the given assumptions in many of the existing analysis tools, however most of them
depend on low-level protocol specification languages that in some ways are more diﬀicult to work with.
To deal with the problem of complex languages, some analysis tools offer a high-level language which
can be used instead. The trade-off here is that these languages are not as expressive as the low-level
alternatives. Since most of them are designed with traditional protocol analysis in mind, they are not
suited for analysis concerning multi-factor authentication protocols [4, 5].
The goal of this thesis is to provide a high-level protocol specification language, which follows
the design of existing similar languages, and on top of that includes a way to specify different threat
scenarios, which will provide the means to analyze and test the robustness of multi-factor authentication
protocols.
The structure of this thesis will start out with a presentation of relevant background material and
related work. The chapter after that will dive into the design process of the language proposed by this
thesis covering syntax, semantics, and translation of the language. The next part of the thesis describes
some of the details revolving implementation of a compiler which translates the proposed language into
one that can be used for analysis. Finally, the solution is tested and a presentation of the results is
given followed by thoughts on further work of the project and a conclusion.

CHAPTER 2
Background
The goal of this thesis is to create a general and automated solution for protocol security verification,
where protocol interfaces might be compromised due to malware or exploits. Jacomme et al. [3] looks
at protocol verification of multi-factor authentication protocols and introduces a malware model in
order to formally describe compromised devices in a protocol.
One of the examples they work on
is the Google 2-step with verification code (g2V) protocol shown in Figure 2.1 which is a two-factor
authentication protocol. In this case the computer might be affected by some kind of malware, which
allows an intruder to read the keyboard input. Given the nature of multi-factor authentication, this
might not be enough in order for an intruder to attack the protocol. This and many other problems
like it are solved by Jacomme et al. [3] using automated protocol analysis. However, the solutions are
implemented in Applied Pi Calculus, which is a lower-level language and therefore not trivial to use.
This thesis proposes a high-level language, which is able to provide concise protocol specifications and
imply potential exploits in given protocol.
Figure 2.1: An illustration of the g2V protocol. The user is required to send the usual login informa-
tion and a generated code in order to get authenticated. Unmodified [3].
This chapter will introduce some of the topics required in order to work towards the goal of this
thesis. These topics include an introduction to languages, and how these can be described in a formal
manner. Furthermore this section covers how computers work with languages, and how programs are
made in order to translate from one language to another. The focus will then move towards how a
protocol specification language might be defined, and how such a language can be used in automatic
analysis and verification of protocols.
2.1
Compiler Theory
Aho et al. [6] states that “a compiler is a program that can read a program in one language - the source
language - and translate it into an equivalent program in another language - the target language”.
Compilers are commonly used to translate from high-level programming languages such as C, C++,
C#, Fortran, and Java to executable machine code. Other compilers translate from one high-level lan-
guage into another high-level language. Overall compilers exist for thousands of programming languages
with different purposes [6].
==PAGE:13==
2.1 Compiler Theory
3
2.1.1
Compiler Structure
A compiler consists of two main parts. The first part is analysis also known as the front end of the
compiler, and the second part is synthesis also known as the back end of the compiler [6].
The analysis breaks the source program into pieces based on a defined language structure.
An
intermediate representation of the source program is then created from these pieces. This part also
deals with error handling, meaning that the compiler should inform the user of syntactically ill-typed
source code and of any semantic errors. As well, a symbol table is created in this part, which keeps track
of the semantic data [6].
The synthesis uses the intermediate representation from the analysis together with the symbol table
to create the target program. This part also deals with different kinds of code optimization, however
these steps are optional [6].
Figure 2.2 shows how a compiler can be divided further into phases. Each phase outputs a new
representation of the program, which is used as input by the next phase. It should be noted that this
is not a strict structure in that some phases are optional and others may be grouped together [6].
Figure 2.2: Visualization of the compiler phases. Unmodified [6].
The lexical analyzer, or lexer takes as input a stream of characters representing the initial source
program. These characters are grouped together using rules to form lexemes. For instance the characters
can be grouped into words, numbers, and symbols. The output of the lexical analyzer is a stream of
tokens containing information about each lexeme. An example of this could be the string “9-5+2”,
which the lexical analyzer might then break into the tokens 9,-,5,+, and 2 in that order [6].
The syntax analyzer, or parser uses the token stream and a set of grammatical rules to create a
syntax tree, which describes the structure of the given program. Figure 2.3 shows an example of such
a tree, where each node is a token. For this given example the operators “+” and “-” are turned into
binary nodes in the tree, with each child being the operator arguments [6].
==PAGE:14==
2.1 Compiler Theory
4
Figure 2.3: The syntax tree produced for the input 9-5+2. Unmodified [7].
The semantic analyzer analyzes the nodes in the syntax tree and verifies that each node and
related information in the symbol table, match the semantics of the defined language. A big part of
this step is type checking which involves checking that types of function and operator arguments match
their definition, so the instructions are valid and describe a meaningful program [6].
The intermediate code generator generates an intermediate representation of the program from
the syntax tree. Intermediate representations can be another syntax tree, a machine-like instruction
code, or something else entirely. The most important attribute of the intermediate representation is
that it should be easy to create and easy to further translate into the target language [6].
The machine-independent code optimizer optimizes the intermediate representation on an
abstract level to improve execution speed and minimize code size. Examples of such optimizations
are removal of unnecessary instructions and simplification of operations that can be precalculated at
compile-time [6].
The code generator translates the intermediate representation into the target language. For many
compilers this involves mapping variables to registers on the target machine in order to create target-
machine code. For source-to-source compilers the intermediate representation is mapped into target
high-level language [6].
The machine-dependent code optimizer is the final compile phase. As the name suggests this
phase involves code optimization on a low level specific to the target machine [6]. This phase is not
within the scope of this thesis.
2.1.2
Context-Free Grammars
A lot of the work carried out by the compiler front end involves creating a syntax tree which describes
the structure of the given program. The tree structure is determined by the syntax of the compiler
language. Context-free grammars can be used in order to formally describe the syntactical structure of
such a language. Sometimes a context-free grammar is referred to as just grammar [7].
A context-free grammar consists of terminals, nonterminals, and production rules that describe the
relation between the first two. A terminal is a fixed sequence of characters, in terms of compilers this
might be thought of as a token. A nonterminal is a variable which can take different forms, this being
a terminal, a sequence of terminals, or empty. A production rule has a head and a body, where the
head is a nonterminal and the body is a sequence of terminals and nonterminals. The purpose of the
production rule is to describe how the head may take form in the given language [7].
An example of a context-free grammar is shown below. The grammar describes a language which
contains strings such as “{a,b,cd}” and “{cd,a,a,b,b}”.
==PAGE:15==
2.2 Alice and Bob Notation
5
array →{ elements }
(2.1)
elements →elements , element
(2.2)
elements →element
(2.3)
element →a | b | cd
(2.4)
The names in italic font are nonterminals, which in this case are array, elements, and element. The
characters in bold font are terminals, thus the terminals of this particular grammar are:
{ } , a b cd
Each line is a production rule, thus array is the head and “{ elements }” the body of the first production
(2.1). Production (2.2) specifies that elements can consist of multiple comma-separated element non-
terminals, while production (2.3) specifies that elements can also be just a single element nonterminal.
The vertical bars in (2.4) are not terminals but define the different forms element can take, so for this
example it can be either “a”, “b”, or “cd”. Compiler languages are more complex than this, however
they are still built from the same type of rules [7].
2.1.3
Parser Generators
Different algorithms exist for parsing languages described by a context-free grammar. Details about
parsing algorithms will not be discussed in this thesis, however Aho et. al [8] covers the various problems
and approaches related to these algorithms.
Parser generators also known as compiler-compilers are programs which takes a language as input
and then uses a parsing algorithm in order to output the source code for a parsing program in a given
programming language. Using a parser generator it is possible to cover most of the compiler front end,
provided that the language is well defined, for instance using a context-free grammar [8].
The input to a parser generator is written in a language defined by the specific generator. For
context-free parser generators the input language should be able to describe all context-free languages.
An overview of different parser generators is given by [9], which shows that most of them use Backus–
Naur form (BNF) or extended Backus–Naur form (EBNF). These are both context-free grammars which
are able to describe any context-free language [7, 10].
The simple language defined in Section 2.1.2 can now be written in EBNF:
array
= "{" elements "}" ;
elements
= element ("," element)* ;
element
= "a" | "b" | "cd" ;
This definition could then be used as input to a parser generator supporting EBNF in order to create a
parser for the specified language. Compared to the original BNF, EBNF introduces some new syntactical
elements such as the “*” used above, which means repetition of the element before it. In the example
above “("," element)” can appear between zero and any number of times. Other symbols of EBNF
are “+” which means repeat at least once and “?” is used to declare that an element appears once or
not at all. The ISO standard for EBNF [10] goes into further detail, describing the entire language.
It should be noted that different variations or dialects of EBNF exist, meaning that the languages are
similar, but might use slightly different syntax elements.
2.2
Alice and Bob Notation
The Alice and Bob notation provides a simple way to describe security protocols. The notation describes
a protocol by providing information on the agents and the messages that are sent between agents in an
==PAGE:16==
2.2 Alice and Bob Notation
6
honest protocol run. An honest run specifies that agents only perform actions according to the protocol
definition and that there is no interference from outsiders. Several languages exist based on this notation.
The language Alice and Bob (AnB) [5] will be used in Section 4.1 as basis for defining the syntax of the
new protocol language proposed by this thesis, which will focus on multi-factor authentication protocols.
2.2.1
AnB Syntax
To introduce the syntax of AnB, the Needham–Schroeder Public-Key Protocol (NSPK) given as an
AnB specification is shown in Figure 2.4. The specification comprises five different sections, each with
its own purpose. These sections will be covered below, however for further details [11] gives a complete
definition of the AnB language.
Protocol : NSPK
Types :
Agent A, B;
Number NA, NB;
Function pk;
Knowledge :
A : A, pk, inv(pk(A)), B;
B : B, pk, inv(pk(B));
Actions :
# A creates NA
A →B : {NA, A}pk(B)
# B creates NB
B →A : {NA, NB}pk(A)
A →B : {NB}pk(B)
Goals :
B authenticates A on NA
A authenticates B on NB
NA secret between A, B
NB secret between A, B
Figure 2.4: AnB specification of the Needham–Schroeder Public-Key Protocol (NSPK) [12].
The Protocol section is used to state the name of the protocol. This section does not have any
actual impact on the semantics of the defined protocol [5].
The Types section declares the identifiers of the protocol. All identifiers used in the protocol and
their respective types must be declared in this section. Identifiers that start with a lower-case letter
are constants and functions, thus they do not change between protocol executions. On the other hand
identifiers that start with an upper-case letter are called protocol variables, which are instantiated
during each protocol execution. Additionally, the protocol variables of type Agent are called roles, as
they can be “played” by anyone during an execution of the protocol [5].
The NSPK example declares two roles, these are A and B. NA and NB are protocol variables
created freshly by the the two agents respectively during protocol execution, which is determined by
the specification given in the sections below. A value being fresh means that it was created independently
==PAGE:17==
2.2 Alice and Bob Notation
7
during a protocol run. The function pk is used to map public keys to agents, thus pk(A) is used to
declare the public key of A.
The Knowledge section specifies the initial knowledge tied to each role in the protocol. The initial
knowledge is the knowledge needed in order to carry out the specified role, thus the first element of
each line must be a role. Protocol variables that are not present in this section, will be created with
a fresh value by the first role to use the variable [5]. In the example A and B know the public key of
all agents as pk is in their knowledge without any given arguments. As well each agent know their own
private key from inv(pk(A)) and inv(pk(B)) respectively, as inv is a predefined function which returns
the inverse of an asymmetric key. Since NA and NB are not mentioned in this section, these will have
to be freshly created during protocol execution.
The Actions section is used to specify the exchange of messages between roles in an honest protocol
execution. The lines of this section are all of the form A →B : M, which specifies an exchange of
message M between role A and role B [5].
The example consists of three steps. In the first step A sends NA and A to B. The message is
encrypted using the public key of B. This is the first mention of NA since it was declared. The semantics
of AnB specify that for this reason, A will freshly create NA during protocol execution during this step.
Following this rule NB will be freshly created by B in the next step of the protocol [5].
The Goals section defines the protocol goals, this can be seen as what is to be achieved by the
protocol. If it is not possible to violate the goals, the protocol is considered to be safe [5].
In the case of the example, one of the goals is that B should authenticate A on NA. This means
that when B receives NA, it should be equal to what was freshly created by A, in other words the goal
here is integrity of the message. The other goal type states that NA is a secret between A and B, this
goal is violated if the value of NA is known by any other party at the end of a protocol run, this goal
is used to ensure confidentiality of NA.
A lot of the AnB syntax is based on messages. The knowledge section uses a list of messages, the
actions section use messages, and the goals are also based on integrity and confidentiality of messages.
A message can take various forms, Figure 2.5 shows the AnB syntax for messages as a grammar [5].
The first rule msg specifies the different ways a message can take form, whereas the second rule msgs
is simply a comma-separated list of messages. The right-hand side elements const and var are constant
and variable identifiers using a lower-case and upper-case starting letter respectively. A function func
is a lower-case identifier followed by one or more arguments in parentheses. The final rules are used for
asymmetric and symmetric encryption respectively, where asymmetric encryption uses a pair of keys for
encryption and decryption, and symmetric encryption uses a single key for both. The rules show that
the encryption is applied on a list of messages, where the argument in parentheses is the encryption
key, thus a single message [5].
msg →const
| var
| func ( msgs )
| { msgs } ( msg )
| { | msgs | } ( msg )
msgs →msg (, msg)∗
Figure 2.5: AnB message syntax as a context-free grammar [12].

2.2.2
AnB Semantics
While Section 2.2.1 covers the basics of the different sections of AnB, this section will give an overview
of how a protocol specification given in AnB can be interpreted for verification analysis.
A protocol specification such as the one for NSPK shown in Figure 2.4, defines a system in which
infinite protocol runs can take place, with an intruder and various honest agents taking on the different
roles provided by the specification. A protocol run can be thought of as a state transition system, where
the run starts in an initial state and in the ideal run reaches a final state, where none of the specified
goals are violated. The knowledge section is then used to determine the contents of the initial state,
and the actions section is used to define transition rules from one state to another [5].
The semantics of AnB can be directly translated to a lower-level language called Intermediate Format
(IF). IF is more direct than AnB in the way it specifies states and the transitions between these [13].
Section 2.3.1 covers IF in more depths and gives insights as to how one might translate from AnB.
2.3
Open-source Fixed-point Model Checker
An example of a robust program developed for security protocol analysis is the Open-source Fixed-point
Model Checker (OFMC) [13]. OFMC is based on the On-the-fly Model-Checker, which is a part of the
back end tools provided by the Automated Validation of Internet Security Protocols and Applications
(AVISPA) project [14]. While the AVISPA project is based on a language called High-Level Protocol
Specification Language (HLPSL) which is another language derived from Alice and Bob notation [15],
the newer version of OFMC uses AnB [13] covered in Section 2.2.
OFMC is used for symbolic security protocol analysis, which is achieved by protocol falsification
and bounded session verification techniques. As mentioned AnB is used as input language, which is
then translated to another language called IF. It should be noted that IF is actually the main language
of OFMC, meaning it by default is used directly as input, however as it is a lower level language, AnB
is to be preferred in many cases for human readability [13].
2.3.1
Intermediate Format (IF)
A specification written in IF comprises three parts. The first part of the specification is the initial state
of a protocol. The second part is a set of transition rules, which determines how the protocol transitions
between states. The final part is a set of goals that is used to classify whether a state is an attack state
or not. A goal could be that an intruder should not be able to know the content of some message within
the protocol. The analysis of OFMC is based on whether or not it is possible to reach an attack state
from the initial state using the transition rules. A protocol is considered to be safe if it is not possible
to reach an attack state [13].
In IF, a state is defined as a finite set of facts. A fact is a term in IF, which can be seen as a function
with one or more message terms as arguments. Message terms in IF are similar to messages in AnB,
as constants start with a lower-case letter and variables start with an upper-case letter. An example
of a fact could be stateR(m1, ..., mk), which specifies the local state of role R. Another example of a
fact is iknows(m) which indicates that message m is part of the intruder knowledge, this being a set of
messages known by the intruder [5, 13].

CHAPTER 3
Related Work
Various tools exist for security protocol verification, on the list of these tools are Scyther[16], AVISPA[14],
ProVerif [17], Athena [18], DeepSec[19], and Tamarin[20]. This goes to show that this is by no means
a new topic within the field of computer science. The tools use different input languages and vary
in verification techniques and approaches. As well results and potential attack traces are presented
differently [21].
The work of Basin et al. [22] uses Tamarin in order to provide and analyze a formal model of the
5G AKA protocol. One of the reasons they chose to use Tamarin, was the support for XOR operations,
which is a hard operation to deal with symbolically, due to the algebraic properties. Tamarin however
uses a very expressive language, that can be hard to use and understand by humans.
Section 2.3 covers the protocol analysis tool OFMC, which uses IF as its native language, although
it provides support for the high-level language AnB, which can be translated to the more expressive
and lower-level language. In the same manner Almousa et al. [4] introduces the Security Protocol
Specification language (SPS) which is another high-level language based on the Alice and Bob notation.
SPS can be translated to Applied Pi Calculus, which is the input language of ProVerif.
Jacomme et al. [3] uses ProVerif to analyze various multi-factor authentication protocols. They
use the lower-level Applied Pi Calculus language to model the protocols and the malware models. A
high-level language such as SPS would probably have been easier to use, however SPS does not support
the necessary semantics for a malware model.
The contribution of this thesis is to provide a high-level language supporting semantics regarding
malware and exploits in protocol specifications, which to the best of the author’s knowledge is a novelty
within the field of automatic protocol analysis.

CHAPTER 4
Design
The goal of this thesis is to provide a high-level language to describe protocol specifications that may
contain compromised interfaces. The language shall have a precise formal semantics. The semantics
is defined by translation to IF, which will allow for automated protocol analysis using OFMC. The
proposed solution to the given problem is the Security Protocol with Exploits Specification language
(SPX), which is the product of this thesis.
This chapter will cover the development of the language, including the design choices behind the
language syntax and semantics. SPX is closely related to both AnB[5] and SPS[4], while simplifying
some of the concepts from these. However, based on the malware model from Jacomme et al. [3], SPX
introduces new aspects to the semantics regarding interface exploits, that the original languages do not
cover, this being the novelty of the project.
Throughout this chapter the g2V protocol illustrated in Figure 2.1 will be used as the main example,
and will sometimes just be referred to as the example.
4.1
SPX Syntax
This section covers the syntax of SPX and the design choices behind it. The language syntax follows
the principles of Alice and Bob notation and is primarily inspired by the SPS language [4], which is
also the reasoning for the similar name. The syntax is first introduced by example, and for that Listing
4.1 shows a specification of the g2V protocol given in SPX. It should be noted that the browser in the
specification corresponds to the computer in Figure 2.1.
The g2V protocol allows a server to authenticate a user using two authentication factors.
The
protocol uses regular username and password authentication through the user’s browser, after which
the server will generate a code and send it to the user’s mobile by SMS. The user can then read the
code and send it back to the server through the browser. A break down of the different specification
sections can be found below.
The Types section declares the different identifiers used throughout the protocol.
The section
closely resembles the types section of both SPS and AnB, although it introduces a concept of ownership,
meaning that an agent can be the owner of another agent.
The g2V example declares four roles, these are User, Browser, Mobile, and Server. Code is a protocol
variable created freshly by the Server during protocol execution. It is specified that User is the owner
of Mobile and Browser. This means that each instance of User produces a corresponding instance of
Mobile and Browser. If a protocol has more than one user role, the specification will need additional
mobiles and browsers to match each role.
The Mappings section declares functions that can be used to map a list of messages to another
message. The functions specified in the mappings section are not cryptographic operators, thus they
can not be computed. However, they make it easy to write predefined constants such as public keys,
where pk(X) for instance can be used to specify the public key of agent X[4]. The example uses this
section to declare a password between two agents and the public key of a single agent.
The Formats section is used to specify real world message formatting such as JSON or XML.
A format is another type of function which allows to declare custom non-cryptographic operators for
structuring information. This can be used to reason about the structure of a message in real applications.

The Channels section is a new section not seen in either SPS or AnB. This section is used to

The example declares three channels, ch1 is a secure channel between User and Browser, while

confidential, meaning that messages can not be changed or read over this channel, while the insecure

For this reason SPX uses manual session control inside the protocol

The first session in the g2V example is a regular session between the two honest agents a and s.
This will instantiate Mobile and Browser to mobile_a and browser_a respectively as these are owned
by User according to the types section. The other sessions are used to give the intruder insight on each
role in order to get a more thorough analysis by OFMC.
The Actions section specify the exchange of messages between roles in the protocol. Here channels
are used to determine how the messages are exchanged specifically. This section has taken more inspi-
ration from AnB than SPS, by supporting multiple messages and automatic handling of fresh messages,
whereas SPS rely on formats and explicit specification of fresh message creation [4, 5].
The actions in the example follow the exchange of messages shown in Figure 2.1. The first appearance
of Code since it was declared in the types section is the exchange between Server and Mobile. Just as for
AnB, the semantics of SPX specify that for this reason, Server will freshly create Code right before this
exchange during a protocol run. The exchanges between Browser and Server use the built-in function
crypt to encrypt messages since the exchange takes place on an insecure channel and could be read by
an intruder otherwise. Inside the encrypted messages, square brackets are used to indicate the main
matter or payload of the message. In this case the payload is readable by the intruder despite encryption,
since the intruder has read-only access on Browser.
The Goals section defines the goals of the protocol. These are used during analysis to determine
whether the protocol is safe or not. The syntax of this section is almost identical to the one of SPS,
although SPX also supports weak authentication, which originates from the AnB language. In short
weak authentication is authentication that ignores the session identifier (SID) of messages, however a
more thorough description of the semantics is covered in Section 4.2.3.
In the g2V example the different goals cover that Server and User authenticate each other on Code,
and that Code should be a secret between Server and User. All these goals are achieved under normal
circumstances, however due to the exploit on Browser the intruder will be able to read Code during
execution, thus compromising the security of the protocol. The analysis results from this example and
others can be found in Chapter 6.

work on SPS by Almousa et al. [4]. Elements in bold font are terminals, while normal font is used

4.2
SPX Semantics
The semantics of SPX are similar to those of SPS and AnB. This section will mainly focus on how the
semantics differ and how SPX handles parts of the semantics differently, although a complete overview
of the semantics is still provided throughout the section.
Overall SPX takes the message exchanges from the actions section and converts them into operational
strands for each role. An operational strand is a sequence of actions performed by a single role in a
protocol specification. The semantics of operational strands are formally described by Almousa et al.
[4], although SPX uses a less sophisticated model and only a subset of the covered operations. SPS
uses a message model which is able to handle protocols where part of a message may not be visible to a
recipient, and where construction of messages may depend on algebraic properties. The message model
can then be used to verify the executability of a protocol. This is not within the scope of this thesis,
although Chapter 7 describes the possibilities of transferring the semantics to SPX.

4.2.1
Channels
SPX supports two types of security on channels.
The insecure channel (denoted →) is completely
exposed so that messages sent on this channel can be read directly by the intruder, as well the intruder
can send messages on this channel given that the recipient can actually accept the message according
to protocol.
For the g2V example in Listing 4.1 Browser sends an encrypted message containing pwd(User,
Server) to Server. The intruder can see the encrypted message but only read the payload if the private
key is known in order to decrypt the message. However, by knowing the public key, the intruder can
construct messages containing the same fields and send to the Server during a protocol run. The secure
channel (denoted ⇒) is both authentic and confidential, and the intruder can neither change nor read
the messages sent on this channel. These channel properties are encoded using encryption with a fictive
symmetric key between the endpoints. This means that the intruder can still block or replay such
messages, even if they cannot modify or decrypt them [23].
The two channel types can be both simplex and duplex, thus for the g2V example it would not
be possible to have a message exchange from User to Server on channel ch3, since it is specified as a
one-way communication channel going from Server to User.
The g2V protocol is supposed to use a channel with Transport Layer Security (TLS) between Browser
and Server [3]. TLS provides a pseudonymous secure channel, which is a confidential channel with
authenticity based on a pseudonym [23]. As a simple way to model this, and avoiding to model the
==PAGE:27==
4.2 SPX Semantics
17
entire TLS protocol, crypt is used on an insecure channel capturing the essential properties of the TLS
channel. A better approach would have been to let SPX support this type of channel directly, Chapter 7
takes a look at how this might have been done for pseudonymous secure channels and other channels not
supported by SPX at the moment. Additionally, the connection between User and Browser is specified
using a secure channel. This allows the intruder to block messages on the channel, due to the intruder
model of OFMC[13]. In reality the intruder is not able to block messages on a closed network, at least
not without some kind of exploit or malware in place. The result is a potential overestimation in the
OFMC analysis, however in that case the user is assigned with the responsibility of verifying whether
the attack trace presented by OFMC is valid or not.
4.2.2
Exploits
As mentioned earlier, SPX distinguishes between two types of exploits. The first type is the knowledge
exploit, which is interpreted as initial intruder knowledge. In Listing 4.1 the exploit pwd(User,Server)
specifies that the intruder knows the user password before the protocol executes. There are many ways
this might happen, for instance the intruder might have installed a keylogger on the user’s computer
and then gathered the password, or the intruder might have carried out a successful phishing attack
which involves tricking the user into giving up the password. Another way to steal passwords is through
data leaks, since many people reuse their passwords for different web services, a leak of passwords from
one service might give access to another [24]. It should be noted that the knowledge exploit does not
cover just passwords, but rather any information that the intruder might be able to gather beforehand.
The other type of exploit provides the intruder with control over a role in the protocol. Jacomme
et al. [3] models malware by abstracting the system to a set of interfaces with inputs and outputs.
The model can then formalize a compromised system by specifying that the intruder has read or write
access on a specific interface in that system. A succinct notation is introduced Minterface
in:acc1,out:acc2, which
specifies that the intruder has input access acc1 and output access acc2 on interface, where acc1 and
acc2 can be either RO or RW, as the model assumes that write access implies read access, leaving out
WO access from the notation. Given this notation a keylogger can then be denoted Musb
in:RO given that
usb is the computer interface that connects the keyboard [3].
SPX carries over this model by treating each agent as an interface with one input and one output.
The line “Browser(io : RO)” in Listing 4.1 is then equivalent to the denotation Mbrowser
io:RO . Continuing with
this example the exploit specification determines that all messages sent to Browser and all messages
sent from Browser can be read by the intruder. When the intruder can read a message due to an
access exploit, the payload of that message is directly exposed, meaning that the exploit bypasses any
encryption there might be on that message.
The RW access type provides write access on top of the read access given by RO. For instance
Mbrowser
in:RW denotes that the intruder is able to send message directly to Browser. Semantically this is
handled by allowing the intruder to send just the payload of a message, thus bypassing any encryption
or channel security that would normally ensure the integrity of a message. RW access on the output
of an interface allows the intruder to directly send payloads to agents that receive messages from the
compromised interface, for instance Mbrowser
out:RW allows the intruder to send direct payloads to Server, but
only for the messages that Server expect to receive from Browser.
4.2.3
Goals
Goals are used to determine whether a state in a protocol run is an attack state. An attack state is a
state that violates one or more of the goals given in a protocol specification. A protocol specification
in SPX can consist of three different types of goals.
An authentication goal consists of a verifier, a claimant, and a message, where the purpose of the goal
is to check that the verifier can trust the content of the message sent from the claimant, meaning that
the message has not been modified since it was initially constructed. In other words, the authentication
goal is a one-way promise of integrity between two agents. The default type of authentication is strong
==PAGE:28==

18
authentication, which also makes sure that the SID of a message is correct, thus making sure that the
protocol is not vulnerable to replay attacks. On the other hand weak authentication only verifies that
the message has not been changed and ignores checking the SID [13].
Finally, the secret goal is used to verify that a message is kept secret between a set of agents. During a
protocol run this goal is violated if the secret message becomes part of the intruder knowledge, recalling
from Section 2.3.1 that the intruder knowledge is a set of messages known by the intruder [13]. This
can happen if the secret message is sent on an insecure channel with compromised encryption, or if the
intruder is able to read the message through a compromised interface.
In comparison SPS does not support weak authentication, however AnB implements the goals cov-
ered above as well as a model that allows the use of channels as goals [4, 13]. Chapter 7 briefly covers
the extended use of channels.
4.3
Translating to IF
In order to use SPX for protocol analysis, it must first be translated to a language which can be
interpreted by an analysis tool. This thesis uses OFMC to perform both protocol falsification and
bounded session verification. To work with OFMC it is required to use one of the supported languages
AnB or IF. OFMC does not work directly with AnB but translates it to IF first. IF on the other hand
is the native input language of OFMC[13]. It might be possible to translate from SPX to AnB and let
OFMC do the rest from there, however dealing with the low-level IF language directly potentially gives
more freedom to express the semantics of exploits in SPX, which are not directly supported by AnB.
This section deals with the description of how an SPX specification is translated to IF while still
preserving the protocol semantics. The first of the following subsections describes how SPX deals with
the intruder deduction rules of IF, the rest of the subsections are structured based on the four sections
comprising an IF specification. Appendix B shows the resulting IF specification translated from the
g2V example found in Listing 4.1.
4.3.1
Intruder Deduction
IF is designed to allow a wide variety of intruder models, since it is able to work with custom sets of
intruder deduction rules. This means that the algebraic rules of functions in IF can be defined manually
to fit the desired intruder model. For instance the default intruder model of OFMC includes intruder
deduction rules for an exp function, with algebraic properties of exponentiation providing the relation
exp(exp(g, X), Y ) ≈exp(exp(g, Y ), X), which allows OFMC to correctly analyze protocols based on
the Diﬀie–Hellman key exchange[13].
SPX uses a subset of these functions and does not allow user specified deduction rules. The pair
function is completely transparent as the intruder can freely construct messages with the function and
read the arguments. As mentioned in Section 4.2, the pair function is used to pack multiple messages
into a single message. Another function supported by SPX is apply, which takes two arguments as well.
The apply function is used on mappings and formats, so that the first argument is a function identifier
and the second argument contains the mapping or format arguments packed using pair. For instance
pwd(User,Server) in SPX translates to apply(pwd,pair(User,Server)) in IF. The rules of apply follow
that mappings are not cryptographic operators but predefined constants, thus the arguments of apply
are not readable. Since formats should be readable, the format identifiers specified in SPX are added
to the intruder knowledge, making the formats transparent to the intruder. On top of that, SPX uses
crypt and scrypt for asymmetric and symmetric encryption respectively. To provide a relation between
public and private keys, the inv function can be used. For instance if pk(X) is the public key of X, then
inv(pk(X)) denotes the matching private key. Chapter 7 discusses the possibilities of applying custom
intruder deduction rules in SPX.

5.1
Analysis Phase
The parser generator ANTLR[26] was used in order to take care of both lexical analysis and syntax
analysis. The SPX compiler specifically uses the C# implementation of ANTLR 4 (4.6.6). The input
language used by ANTLR is a dialect of EBNF, adding a few features which makes it easier to work
with the generated parser[27]. The grammar that was used to generate lexer and parser for the SPX
compiler can be found in Appendix C.1. The lexer rules match the symbols, keywords, and identifiers
of SPX in order to create tokens for the parser. The parser rules are the same as those found in Figure
4.1, rewritten to match the EBNF dialect of ANTLR.
A visitor was implemented to traverse the syntax tree generated by the parser. The purpose of the
visitor is to turn abstract ANTLR tokens into custom class objects which are more flexible to work
with. The output of the visitor is stored in an IntermediateRepresentation class, which has a structure
that closely resembles the SPX sections covered in Section 4.1. The types, mappings, formats, and
channels are stored in a SymbolTable class, since these sections all declare identifiers that are used in
the remaining sections. The rest of the sections are stored as lists of corresponding elements, thus the
knowledge section becomes a list of objects from a Knowledge class.
During the semantic analysis, the compiler checks that the various constant and variable names
are defined in the protocol specification. This is done by calling Resolve on all the elements in the
intermediate representation, most implementations of Resolve will verify the element and recursively
call Resolve on nested elements. For instance, the Resolve method of Knowledge calls Resolve on the
specified role and checks that the declaration is an agent, the method also calls Resolve on all the
messages specified in the knowledge element.
Each constant and variable in a protocol specification is stored as an object of the abstract Message
class.
The Message class has different sub classes for the various types of messages.
For instance
Code from the g2V example in Listing 4.1 is declared using the number type, then in order to resolve
occurrences of Code, the name is simply found in the symbol table to verify that it has been declared.
On the other hand, occurrences of pwd will return the mappings declaration from the symbol table, and
in order to resolve the pwd message, the arguments will be resolved as well to verify that the argument
count and types match the declaration.

CHAPTER 6
Tests and Results
The following chapter will use OFMC to analyze two different protocols, given various exploit scenarios.
The first test is performed on the g2V protocol, which has been used as example throughout this thesis.
The protocol is illustrated in Figure 2.1, and the SPX specification used for analysis can be found in
Appendix C.2.1. The second test investigates the Google 2-step with One-Tap (g2OT) protocol, which
is illustrated in Figure 6.1 and specified using SPX in Appendix C.2.2. The two protocols both rely on
a mobile as second authentication factor, however g2V uses SMS to deliver a verification code to the
user, whereas g2OT uses TLS to transfer a token from server to mobile and back again from mobile to
server[3].
Figure 6.1: An illustration of the g2OT protocol. Unmodified [3].
In order to analyze the two protocols, the SPX compiler provided in Chapter 5 is used on the
specifications mentioned earlier. The IF translations are then given as input to OFMC, which produces
the results found in Table 6.1. The protocol column states the analyzed protocol, the exploits column
determines the exploit scenario for the test, and the security column gives the result of the analysis.
The “-” is used for the base protocol without any exploits and serves the purpose of verifying that the
protocol specification is secure on its own. The OFMC output on which the table is created can be
found in Appendix D.
Each protocol is tested on a minimum amount of exploit scenarios, meaning that the analysis stops
once the protocol is deemed insecure, since adding more vulnerabilities at that point is sure to return
the same result.
As an example the g2V protocol with an Browser(in:RO) exploit is insecure and
OFMC returns an attack trace. This exploit is not used again, since the same attack would be possible
given additional exploits. On the other hand, the g2V protocol is deemed secure given the exploits
pwd(User,Server) or Mobile(in:RO) on their own, thus they are combined which returns an attack.
Listing 6.1 shows the output of OFMC given the g2OT protocol with pwd(User,Server) exploit.
The output summary states “ATTACK_FOUND”, and further below the actual attack found by OFMC
can be seen under “ATTACK TRACE”. The attack corresponds to the findings of Jacomme et al. [3] who
describe how the intruder can abuse a lack of association between the user login request and the tap
request from the server to break the security. In short, the attack trace shows that the intruder can
block the login request of an honest user, after which the intruder can perform their own request using
the honest user’s credentials. The server then sends a token to the user’s mobile, and the user will
accept it since they think the request is related to their own login attempt, which finally results in
authentication of the intruder instead.

CHAPTER 7
Future Work
This chapter looks at how SPX can be further developed given more time. The starting point of this
work is based on what has already been achieved by the developers of AnB and SPS.
With the current foundation, it is possible to expand SPX with additional channel types.
The
current language supports insecure and secure channels, while SPS also covers authentic and confidential
channels [4]. On top of that, AnB supports pseudonymous secure channels and can use all these channel
types as both assumptions on message exchanges as well as goal specifications [13]. This is based on the
work of Mödersheim et al. [23] which provides a formal definition of the different channel types. They
present a model for channels as assumptions, meaning that messages sent through a particular channel
is perceived differently by an intruder based on the channel type. The model also covers channels as
goals, which can be used to check that a protocol provides a desired channel type, for instance the goal
of TLS is to provide a pseudonymous secure channel. Future work on SPX might include integration
of these models.
The intruder deduction model of SPX is very minimal at this point in time and lacks a general
handling of algebraic properties in the operational semantics. Both AnB and SPS use a more sophisti-
cated model allowing full message composition and analysis. Additionally, this allows tools like OFMC
to verify the executability of a protocol specified in AnB, since the model ensures that agents in an
honest protocol run hold the required knowledge during message composition. A future goal of SPX is
to integrate this model in the language.
SPX specifications include a format section, where one can define message structures with the intend
of using these for real implementations of the specified protocol. The section is inspired by SPS which
uses formats when generating JavaScript code. As of now SPX does not support code generation for
other languages than IF, thus formats have not been utilized in this thesis where the main concern has
been on security analysis. Further work on SPX might either include support for translation to real
world implementations or removal of the format section in order to let SPX solely focus on the analysis
aspect.
Due to the nature of multi-factor authentication, the protocols quickly end up comprising many
agents and message exchanges. This results in large search spaces, which makes the OFMC analysis
take longer compared to small protocols.
One idea is to include an optimization step in the SPX
compiler, which simplifies the problem while still preserving the semantics. The optimization could
include a step which takes related agents such as devices owned by a user and combines them without
changing the protocol. However, for the moment not much work has been put into a formal definition
of how this might be achieved.
Finally, in order to ease the use of the implemented compiler, the code project could include OFMC
and make it part of the build. By integrating OFMC, the program would be able to take a protocol
specified in SPX as input and then output the OFMC analysis directly. This resembles how OFMC
already handles input specified in AnB.

CHAPTER 8
Conclusion
This thesis has dealt with the design of SPX, a high-level protocol specification language, which as
able to describe multi-factor authentication protocols and include information on exploits that might
affect the specified protocol. SPX is syntactically based on the Alice-and-Bob notation, which describes
protocols by the list of actions executed by agents in an honest protocol run. The thesis gives a thorough
description of the language semantics and how it is closely related to the low-level protocol specification
language, IF. More precisely both syntax and semantics of SPX is based on the languages AnB and
SPS, although SPX separates itself from the two by providing a formally described exploit model, which
is essential to the analysis of multi-factor authentication protocols. The contribution of this thesis to
the field of security protocol analysis is thus the integration of an exploit model in a high-level protocol
specification language, and the work related to specifying the semantics of such model.
A compiler was implemented in order to automatically translate from SPX to IF based on the
SPX language design, which ensures that the protocol semantics are preserved across translation. In
order to test the capabilities of the language, the g2V and g2OT protocols were specified in SPX and
translated to IF using the implemented compiler. The IF specifications were then used as input to
OFMC, which is a security protocol analysis tool. The results showed an interesting vulnerability in
the g2OT protocol, which had already been found using another analysis tool and a low-level protocol
specification. However, the result verifies that even though SPX is a high-level language, it can still be
used for deep security analysis of multi-factor authentication protocols.
