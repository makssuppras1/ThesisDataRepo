Abstract (English)
This thesis presents a programming-language-based method to operate the Dig-
ital Microfluidic Biochip (DMF) platform. A new language, ‘BioGo’, is designed
for this system. The system is mainly based on C# and divided into two sub-
systems: compiler and executor. Both compiler and executor have their own
exceptions handler to convey more detailed information.
The compiler analyzes the input source file and generates an internal file for the
executor. And a syntax checker is developed for the compiler to help the user
identify the possible errors in their program.
The executor reads and executes the operations deserialized from the internal
file. The executor applies the A* algorithm to find a suitable path when execut-
ing operations that need to move droplets. Users can choose to run this execu-
tor in synchronous mode or asynchronous mode. The synchronous mode must
run with a real DMF platform because it needs an external system’s real-time
droplets’ positions. The purpose of synchronous mode is to support real-time
droplet position calibration.
The asynchronous mode can generate complete
instructions to operate the DMF platform directly. The generated file could be
executed by both simulator and DMF platform.
There are unit tests for both the compiler and the executor, consisting of func-
tionality and robustness tests. Moreover, a real-world case of the polymerase
chain reaction (PCR) test is programmed and executed on the simulator, which
is widely applied to test COVID-19. The system can bring convenience to op-
erating the DMF platform, especially for medicine or biotechnology researchers
who are not familiar with the DMF platform.

DMF Simulator
The DMF simulator is a web application under development at DTU Com-
pute [CJM22]. The same as the real DMF platform, this simulator supports
BioAssembly. The basic user interface is shown in Figure 1.2. Users should
upload the configure file and the ’.basm’ file that contains the full instructions

simulator, users can execute them on the physical platform.

1.5
Motivation
DTU Compute currently works on software, firmware, and hardware develop-
ment related to digital microfluidic biochips. When programming and testing
on the DMF platform, every user felt it inconvenient to operate the DMF plat-
form with BioAssembly directly. Hence, developing a high-level programming
language is necessary to make programming on the DMF platform automated
and effortless.
1.6
Presentation
This thesis presents the design and implementation of a classic software toolchain
consisting of a programming language, a compiler, and an executor. The pro-
gramming language should be clear and easy to learn because the purpose is to
bring convenience to users. It consists of basic operations and repeated opera-
tions to support more experiment protocols. The compiler should compile the
source code and detect syntax errors and illegal usage in the source file. The
executor should be able to generate the BioAssembly instructions, interact with
an external system, detect the error, and report necessary information to users.
When generating the BioAssembly instructions, the executor should ensure these
instructions are correct. Hence, the executor should support the representation
of the DMF platform and path searching for the droplets’ movement. Moreover,
it should support connecting to a DMF platform and exchanging messages in
real-time to support interaction with the external system. Meanwhile, it should
also support generating full instructions first. Then the users upload the gener-
ated file containing BioAssembly to the DMF platform or the DMF simulator to
execute. At last, the functionalities and robustness of the compiler and executor
should be evaluated with test cases executor.

2.1.2
DMF Operations
Microliter-sized droplets are independently addressed on an open array of elec-
trodes coated with a hydrophobic insulation layer on the DMF platform which is
shown in Figure 2.1a. After the voltage is applied between conducting droplet
and electrode, the surface wettability of the electrode coated with a thin hy-
drophobic dielectric layer changes. In chemical heterogeneity, with a wettability
gradient, the droplet moves spontaneously and directionally on the flat surface.
In addition, droplets are immersed in silicone oil, which reduces the voltage
necessary for droplet movement.
By applying a series of potentials to these
electrodes, droplets can be individually driven to merge, mix, split, and dis-
pense from reservoirs.
Figure 2.2 presents the DMF operations supported by Bioware developed by
DTU Compute.
It supports moving, mixing, merging, splitting, dispensing,
disposing, heating/cooling, and temperature sensing. The background of the
droplets in Figure 2.2 looks like a metallic ‘grid’. This ‘grid’ consists of some
electrodes.
Applying an electric potential to the electrodes can control the
movement of corresponding droplets. These operations mentioned in Figure 2.2
are realized by applying BioAssembly instructions to control the electrodes and
related sensors on the platform.

2.2
Compiler
The core part of a computer program is the source code written by a program-
mer with a text editor. It is human-readable but not directly understandable
by the machine.
Thus, it is necessary to convert a high-level programming
language (i.e., source language) into a machine language (i.e., target language)
that the computer can understand. This conversion process is called compila-
tion that executed by a compiler. Any compilation consists of two major tasks:
(1) Analysis: Discover the structure and primitives of the source program and
determine its meaning. (2) Synthesis: Create a target program equivalent to the
source program. The analysis considers only the properties of the source lan-
guage. It converts the essential properties of the program text into an abstract
representation, usually conceptualized as an abstract syntax tree (AST). Based
on the abstraction, the synthesis arguments the tree by attaching additional
information that reflects the source-to-target mapping [Wai13].
2.3
Routing and Schedule
To move droplets automatically, the system should find the path and schedule
them smartly. All the droplets can be reckoned as agents on the DMF platform.
Hence, the problem of dorplet’s path finding is the problem of multi-agent path
planning (MAPP).
Traditional multi-agent path planning takes two main approaches: centralised
and decentralised [BL91]. A centralised method incorporates a single decision
maker. For example, planning for all units simultaneously, as in a centralised A*
search, or using Conflict-based search [SSFS15] to detect and resolve collisions
as a whole.
Although theoretically optimal, in practice this approach has a
prohibitive complexity and cannot scale up to many units [WB+09]. On the
==PAGE:24==
12
Background
other hand, a decentralised method can significantly reduce computation by
decomposing the problem into several subproblems.
In this project, my method is the decentralised method. The MAPP problem
is divided into several subproblems of finding path for single droplet. In this
way, all the other droplets will be set as blocks when searching the path for one
droplet.
2.4
Available Languages
The exploration of programming languages for biochips is becoming increas-
ingly popular, and the available languages are mainly instruction-based. Several
microfluidic high-level programming languages and instruction set architecture
(ISA) have been proposed.
The [TUTA08] presents BioStream, a portable language for describing biology
protocols, and the Fluidic ISA, a stable interface for microfluidic chip designers.
In later work, Amin et al. proposed a comprehensive fluidic instruction set,
AquaCore Instruction Set (AIS), and a fluidic microarchitecture, AquaCore, to
implement AIS [ATV+07]. Afterward, Ananthanarayanan and William devel-
oped BioCoder, a practical programming system for standardizing and automat-
ing biology protocols [AT10]. Recently, the [OLC+18] introduced BioScript, a
domain-specific language (DSL) for programmable biochemistry that executes
on emerging microfluidic platforms and relevant portions of its compiler. Pri-
marily when potentially hazardous chemicals are used, its type system, which
defines the operational semantics of BioScript, can provide safety guarantees.
This thesis introduced a new language based on BioAssembly. Compared with
the languages mentioned above, its executing platform could be both a simulator
and a real DMF platform, and the grammar is more straightforward and easy
to use. It provides a complete solution from inputting source files to running
with the DMF platform.

3.1
Objectives and Specifications
This compiler is designed to offer a portable and convenient method for be-
ginners and biotechnology researchers to operate the DMF platform. In this
way, users need not directly operate the DMF platform with BioAssembly men-
tioned in Section 2.1.3. They can program with a higher-level language, and the
compiler can generate the BioAssembly to operate the platform automatically.
However, considering the extensibility of the whole system, the system is divided
into one compiler and one executor. They are both implemented in C#. The
general workflow of the whole system is shown in Figure 3.1. First, the compiler

3.5.1
ANTLR and Grammar
ANTLR (ANother Tool for Language Recognition) is a parser generator for
reading, processing, executing, or translating structured text or binary files.
It is widely used to build languages, tools, and frameworks. From a defined
grammar in a g4 file, ANTLR can generate a parser to build and traverse
parsed trees.[Par13]
In this compiler, the grammar is defined following ANTLR’s rule. The abstract
syntax tree (AST) and the parent class of the syntax node visitor are generated
by ANTLR.
First, there are five basic elements defined:
• ID starts with ’a-z’, ’A-Z’, and ’_’, and consists of several ’a-z’, ’A-Z’,
’0-9’, and ’_’.
• INT consists of multiple numbers.
• FLOAT consists of two INT and one dot ’.’ between them.
• COMMENT starts with ’#’ and ends with ’\r’ or ’\n’ and the content
after that should be ignored.
• SPACE includes the ‘\t’, ‘\r’ and ‘\n’. All of them will be ignored when
compiling.
Then, all the basic operations declaration, input, move, merge, split, mix, out-
put and store consist of one keyword and a couple of brackets that contain one
or more arguments. Figure 3.12 conveys the structure of the AST of the ba-
sic operations. The rectangles in Figure 3.12 contain the keywords defined in
the ANTLR grammar file, and the circles in Figure 3.12 contain the matching
pattern or regex expressions.

Droplet Lifetime
For every droplet, the lifetime is defined to declare if it is operable.
If one droplet needs to be operable, it must be assigned to a declared variable.
Hence, a droplet’s lifetime begins when it is assigned to a declared variable
and ends when it is merged with another droplet, divided into two droplets,
or outputted(i.e., removed from the platform). In other words, the operations
input, split, merge can assign a droplet to a variable to start the lifetime of a
droplet. Moreover, the operations output, split, merge can end the lifetime of a
droplet. The left operations do not affect the lifetime of droplets.
Droplet Lifetime Checker
While compiling, the DMFOperationVisitor will go through the whole file to
check if there is any syntax error, as mentioned in Section 3.5.3. Although all
the statements satisfy syntax, some errors can not be detected by DMFOper-
ationVisitor.
For example, a variable needs to be declared before it can be
assigned a value. Moreover, one variable can only be assigned with one droplet
before it is released. These errors related to droplet lifetime can not be detected
by DMFOperationVisitor directly, but they will result in the executing failure
when the executor executes the compiled file. Hence, it is necessary to add a
Droplet Lifetime Checker to the compiler.
Two lists are used to help the compiler check the droplet lifetime. One list,
called declared droplets, is used to store the variables declared but not used, and
the other, which is called occupied droplets, is used to store the variables that a
droplet has occupied.

4.6.1
Path Planning Techniques
When a droplet needs to move because of some operations like move, it must find
a path from its current position to its destination. Many algorithms could be
used for path searching. Because the area of the programmable area is small and
the synchronized mode requires real-time searching, the A* algorithm [HNR68]
is chosen.
==PAGE:67==
4.6 Movement Manager
55
A* Algorithm
A* algorithm is a graph traversal and path search algorithm used in many fields
of computer science due to its completeness, optimality, and efficiency. It can
only find the least cost path. In this executor, the cost of moving from one cell
to another is always 1. For A*, one practical drawback is its space complexity,
as it stores all generated nodes in memory. Thus, in practical travel-routing
systems, it is generally outperformed by algorithms that can pre-process the
graph to attain better performance [HNR68].
Preparation
As mentioned above, A* can find the shortest path from a droplet to its destina-
tion position, and proper preparation on the graph can improve the performance
of A*. In this system, the A* algorithm is based on cells, which is introduced in
Section 3.3. It can only calculate a path from one cell to another. Hence, some
preparation before path planning is necessary.
For every time path planning, it only aims at one droplet. The executor can
not plan the paths of two droplets simultaneously. All the other droplets on the
platform will be reckoned as blocks to avoid other droplets’ disturbance. For
example, in Figure 4.5, the red droplet wants to find a path to the blank area.
The area of the green droplet will be reckoned as blocks before path planning
for the red droplet.
Furthermore, because this A* is based on cells, directly planning a droplet
consisting of multiple cells is challenging.
The executor uses the position of
the droplet’s left-top cell to represent this droplet. And then, all the areas on
the platform which are inaccessible to the droplet will be reckoned as blocks.
The inaccessible areas include the bottom and right border, and the left-top
areas next to the other droplets. These areas are deep gray areas in Figure 4.5c
and Figure 4.5d. The width of blocks depends on the radius of the droplet.
The approach to calculating the radius according to the droplet’s volume is
mentioned in Section 3.3. The equation is width = ⌈radius⌉. For example, in
Figure 4.5c, the red droplet is 2 × 2. The width of block areas is one and shown
in a deep gray. In Figure 4.5d, the red droplet is 3 × 3, so the width in deep
gray is two.
In the real world, two droplets must be separated by at least one cell. Otherwise,
they will merge automatically. To avoid the unexpected merging, the width of
the block of all the other droplets should be widened with one other cell, shown

researchers. However, for different medical tests, different methods should be
considered. A* would not always be an excellent path-finding algorithm. For
example, there is only one droplet on the platform. The droplet only needs to
move to one position, heat for a specified time, and move to another position
to sensor its color. A simple greedy algorithm is better for such a corner case
because A* requires significant memory and computation.
Hence, users can customize a new path-searching algorithm according to their
requirements. There is an interface RouterInterface which only contains one

4.6.3
Deadlocks and Corner Cases
Some special cases need to be handled to avoid deadlock and unnecessary com-
putation. In this section, all the cases mentioned occur when the executor is
implemented with the A* algorithm to search for a path.
Merge and Split
When two droplets merge, the last steps are to move two droplets to the same po-
sition. As mentioned in Section 4.6.1, all the other droplets and their surround-
ings are reckoned as blocks during the preparation process. The two droplets
cannot find a path to move to the same position. For example, as shown in
Figure 4.6c, two droplets, A and B, are trying to merge. Droplet A is closer
to the destination (in red), so it arrives at the destination first in Figure 4.6d.
However, droplet B finds the destination in the block area of A, which it can not
enter. B will wait for A to move, but A has arrived and will not move anymore.
Finally, a deadlock occurs.
When one droplet splits into two droplets, there is also one moment when these
two droplets are next to each other without any space. In this way, both droplets
are placed on a block. They can not move anymore, and a deadlock occurs.
For example, in Figure 4.6a, one droplet splits into droplet A and droplet B.
Nevertheless, for the next moment, shown in Figure 4.6b, droplet B found itself
in A’s block area, and so does A. Finally, they are both waiting for each other’s
movement. This results in a deadlock.
In conclusion, for Merge and Split operations, there is one moment when two
droplets affect each other’s movement. At this moment, the other droplet should
not be reckoned as a block. The solution is to add a property exemptDroplet
in droplet class. The textitexemptDroplet will not be reckoned as a block in

Not every time could a path be found. For example, shown in Figure 4.8, droplet

found path.

4.7
Interaction Manager
The interaction manager generates and exchanges data with an external system.
It has two different modes - Sync Mode and Async Mode.
4.7.1
Sync Mode
Because of the unreliability of the hardware, it is impossible to 100% ensure the
droplets move successfully and thoroughly follow the instruction. A computer-
vision-based real-time detection system has been implemented in another project
to avoid such a problem. This real-time detector can detect the positions of
droplets based on computer-vision technology and share the results. Sync Mode
aims to support real-time feedback systems such as the computer-vision-based
real-time detection system.
First of all, it is necessary to define the ’timestamp’ of Sync Mode. The ’times-
tamp’ is not an index of time in the executor. Because the droplets should move
parallel, the executor collects all the movable droplets’ next step positions for
next step. The timestamp is the index of movement steps from the point of the
platform.
==PAGE:73==
4.7 Interaction Manager
61
The workflow of the sync-mode executor is shown in Figure 4.9. For every step,
the executor will send a message containing BioAssmebly instructions to the
DMF platform to control the movement of droplets. After the DMF platform
replies with a message which means execution is completed, the executor will
fetch the latest detection result from the computer-vision-based real-time de-
tector. The detection results contain all the current droplets’ positions in the
physical system. Then the interaction manager will analyze the results. If every
droplet’s position is correct, move on next step. If some positions are incorrect,
update them in the activeDroplets and executingDroplets.
Because the hardware is unavailable now, this function is not thoroughly tested
with the DMF platform. Now this project supports a subscription-based inter-
face to support the message exchange with the external system. The user can
implement an adaptor class to specify the concrete communicating method like
MQTT or ROS if necessary.
Figure 4.9: Workflow of sync mode.
4.7.2
Async Mode
If a user chooses to run in Async Mode, the user will get full instructions about
platform controlling instructions. It is not real-time because this executor will
collect and store all the information on how to move every droplet from the first
step to the last step. After all, the steps are collected, it will generate a file
containing all the instructions, which can be executed by the DMF platform or
simulator directly. The user only needs to upload it to the destination platform.

Chapter 6
Conclusion
This chapter presents the summary and future works of this thesis.
6.1
Summary
This thesis conveys a programming-language-based method to operate the DMF
platform.
It is implemented by C# and divided into two sub-systems: the
compiler and the executor.
The compiler analyzes the input source file and outputs a JSON file for the
executor. The executor reads and executes the operations deserialized from the
JSON file. Users can choose to run this executor in different modes. Under
synchronous mode, the executor connects to a real DMF platform and corrects
the droplets’ position information according to the external messages from a
computer-vision-based droplet detector.
Under the asynchronous mode, this
executor generates the complete instructions to operate the DMF platform di-
rectly. The generated file could be executed by both simulator and DMF plat-
form.
Moreover, the compiler and the executor both have exception and error handlers.
There are functionality and robustness tests for the compiler, the executor, and
==PAGE:86==
74
Conclusion
their exception handler. Finally, a real-world case of a PCR test is programmed
and executed on the simulator. It generates more than 8000 lines BioAssembly
instructions, but users only need to write 12 lines BioGo code. According to
the tests and the case, the programming-language-based method is correct and
possible to make the operations on the DMF platform more straightforward and
more convenient.
6.2
Advantages and Disadvantages
The advantages:
1. This language is straightforward, easy to learn, and flexible for designing
complex tasks.
2. Users can implement their path-finding algorithms according to their re-
quirements.
The disadvantages:
1. The grammar of BioGo is challenging to be extended.
2. Now BioGo does not support all the operations the DMF platform sup-
ports, such as setting the biochip’s temperature, sensing the color of one
droplet, and sensing the temperature of one droplet.
3. The tests are not enough and only cover some situations.
4. For the Sync mode of the executor, network security is not considered.
6.3
Future Work and Extension
For the compiler, it should cover more functions of BioAssembly, such as sensing.
Moreover, the compiler should implement the conditional statement, i.e., the
keywords of while and if.
For the executor, it should be tested with the real-world platform. Because
the platform is unavailable, only unit tests and the simulator are used to test
the functionalities. More tests should be added to ensure its robustness. More
corner cases should be considered. Moreover, the computer-vision-based real-
time detection system should also be included in this system.
