INTRODUCTION 
Up until recently there were only 2 leading graphical API’s. One was Khronos’s (used to be ARB’s) OpenGL 
and Microsoft’s proprietary Direct 3D (A part of DirectX suit). Those API’s have a long history touching the 
beginnings of GPU’s, namely 1990’s. With  Nowadays along with newer and faster GPU also new API try to 
exist In the market. One of them is Mantle by AMD for PC’s and the Metal API by Apple for mobile 
devices. Those new API’s claim to give users a performance boost up to 40% by reducing so called driver 
CPU overhead. Mantle API in addition to being developed by one of 2 biggest GPU manufacturers was also 
supported by DICE game studio which is renown as one of the best game studios worldwide. Does those 
new API’s stand a chance against strongly established competition ?   
In this thesis I will analyse API’s based on their history, architecture, ownership and I will answer the 
question if they are really benefit the industry or they are only a so called “turf war” between GPU 
suppliers.  
This thesis will be devided into 3 parts. Introduction , Research and Conclusions. Introduction will 
introduce the topic and make some assumptions. In the next chapter called research I will analyse the 
history, architecture and ownership dependencies of API’s. Finally in Conclusions I will make conclusions 
based on analysis in research, answering the question if the new API’s do really benefit the market.  
What is API 
Wikipedia definition of API is: 
In computer programming, an application programming interface (API) is a set of 
routines, protocols, and tools for building software applications. An API expresses a 
software component in terms of its operations, inputs, outputs, and underlying types. 
(Wikipedia, 2015) 
More or less it means that API is a common language of developer and software. It translates what 
developer wants into a set of routines a underlying software has to process and execute to deliver result 
of what developer asked for. It also narrows what developer can do. Developer using API is limited to set 
of instructions API  provides.  
The specific group of API’s this thesis will concern are graphics API’s. Those are API’s allowing developer to 
‘talk’ with graphics driver which is actually just another piece of software. The most known API’s until now 
are OpenGL and DirectX yet there are also some more niche API’s like RenderWare, Glide API or 
RenderMan used mostly by companies that they were developed by.  
Old API’s 
By old API’s in this thesis I don’t mean the old historical API’s that are not used anymore but the API’s with 
long history that are still widely used. In this thesis term ‘old API’s’ will concern mainly OpenGL and

DirectX. OpenGL was created in 1992 and DirectX soon afterwards in 1995. Since then both evolved along 
the graphics hardware up to stage we see today.  
OpenGL is based on SGI’s Iris GL and is an open standard. It was at first managed by ARB( Architecture 
Review Board ) which included such companies as 3Dlabs, Apple, ATI, Dell, IBM, Intel, NVIDIA, SGI, Sun 
Microsystems and Microsoft. In March of 2003 Microsoft left the group to concentrate on their own 
proprietary API (DirectX) and finally on 31 July 2006 at SIGGRAPH control of OpenGL was passed 
to current owner of API – Khronos Group. Current version of OpenGL is OpenGL 4.5. OpenGL is 
a state machine based API which is very useful yet confused programmers for a long time. Lately 
OpenGL finally added so called Direct State Access into standard which allows to directly 
manipulate the OpenGL states without explicitly binding them. 
DirectX is a proprietary API of Microsoft. It was created in 1995 based on Reality Lab API of 
RenderMorphics company. The before mentioned company was founded in 1992 and later 
bought by Microsoft in February 1995. Differently from OpenGL DirectX expects the application to 
manage hardware resources and OpenGL makes the implementation do it. This gives DirectX a 
better control over resources allocated. Current version of DirectX is DirectX 11.x shipped with 
Windows 8.x and Xbox consoles.  
New API’s 
New API’s in this thesis are literally newly developed APIs. Lately the speed of existing APIs became too 
slow for developers. The cause was announced to be a large CPU based driver overhead. As there was no 
immediately response from existing API manufacturers ATI decided to develop, together with DICE studio, 
a new API which will lower the driver overhead to minimum. This API is called Mantle. Mantle allows 
developers to directly control GPU by creating so called command buffer queues. It also simplified what is 
sent to GPU and grouped the data into vertices and images. 
At the same time Apple wanting to cut off from dependency on OpenGL ES for graphics decided also to 
make a new API. The Apple API is strongly adapted to their GPU and system and is called Metal.  
Direction of changes 
The new API’s are now racing to make a driver overhead marginal. Mantle already is in beta stage. Metal 
is released with iOS 8. DirectX 12 and OpenGL Next (name pending) are also declaring to join this race and 
lower the driver overhead to minimum. Let us see than if what manufacturers say and towards what they 
stride is really worth the work.

RESEARCH 
Since 1992 when OpenGL was first released a lot changed in case of graphics API's. The API’s like OpenGL 
and DirectX evolved to suit new hardware capabilities. Yet still there seems to be space for fresh ideas in 
API development for graphics. Just recently new API's like Mantle and Metal emerged.  
Taking into account historical fall-back of OpenGL and DirectX I would like to start this research with the 
brief historical analysis of both HW capabilities and software API's architecture.  
OpenGL represents the way of open-source based on Iris GL by SGI. On the other hand DirectX is a 
proprietary software of Microsoft developed from scratch in 1994 as response to OpenGL and changing 
operating systems. Finally there showed Mantle and Metal API’s which were newly created from scratch 
and introduced not so long ago. We will compare ‘experienced’ OpenGL and DirectX with new blood in 
graphics API’s. Will those new API’s introduce some freshness into graphics industry or would they 
become some niche code in shadow of their older competitors. 
History lesson 
HISTORY OF GPU 
The simplest definition of GPU by Wikipedia is: 
A graphics processing unit (GPU), also occasionally called visual processing unit (VPU), 
is a specialized electronic circuit designed to rapidly manipulate and alter memory to 
accelerate the creation of images in a frame buffer intended for output to a display. 
(Wikipedia, 2015) 
It means basically that GPU is a hardware component specialized in creation of images in a framebuffer. 
While being specially prepared for this task GPU is a magnitude faster at processing matrices and vertices 
as well as rasterizing than CPU.  
The first GPU (Graphics Processing Unit) was introduced in 1970’s arcade 
system boards. Fujitsu created MB14241 video shifter which was capable 
of accelerating drawing of sprite graphics. It was used to display arcade 
games like Sea Wolf (1976) or Space Invaders (1978).  Afterwards the 
Namco Galaxian arcade system board was developed in 1979. It was an 8-
bit arcade game system board which was first used for Galaxian game as 
name suggests. It used specialized graphics hardware supporting RGB 
colour, multi-coloured sprites and tile map backgrounds what allowed it to 
display colourful images.  
The power of those GPU’s was just a fraction of current graphic cards, yet for people at that time they had 
huge capabilities and were revolutionary.

1980’s 
During 1980’s the variety of graphics processing units increased. From arcade system boards they 
expended their use towards home computers and consoles. The set of features was not so wide as today. 
Early GPU’s mostly only supported drawing of simple geometrical primitives and character graphics to a 
bitmapped display. Most graphics hardware back than implemented so called pseudo-3D 2D graphics 
meaning they used sprites and bitmaps to create an illusion of 3D rendering.  
In the late 80s graphical units started to be integrated into CPU hardware. In 1986, Texas Instruments 
released the TMS34010. It was the first microprocessor with on-chip graphics capabilities. Later cards with 
fixed-function 2d primitives support were created (example: IBM 8514) and finally in 1988 the first 
dedicated polygonal 3D graphics boards were introduced in arcades. For example: Namco System 21 
"Polygonizer" or Taito Air System. 
 
1990’s 
In 1991 the first of many to come graphics processing unit with 2D acceleration support was created by S3 
Graphics. Introduction of HW acceleration made new cards surpass the general purpose graphics 
coprocessors created in last decade in case of performance and made them fade away from PC market. 
Throughout the 1990s, 2D GUI acceleration continued to evolve. In addition graphical API’s like DirectX 
and OpenGL were created to make graphics development easier and to allow and manage access to 
steadily growing in power hardware. 
In early and mid-1990’s 3D acceleration was becoming common in consoles and arcade machines. This 
made manufacturers look for a way to supply the market with cheap 3D-accelerated graphics hardware 
instead of only 2D accelerated.  
The first low costs chipsets to support 3D-acceleration were essentially previous-generation 2D 
accelerators with 3D features bolted on. 
The first performance 3D-accelarated GPU’s were a discreet boards lacking the 2D-acceleration in favour 
of 3D acceleration. In this way they were able to boost 3D performance much higher than multifunction 
chipsets yet they required another dedicated 2D accelerated card. The most known chips of that time 
were PowerVR Series 1 and 2 and 3Dfx Voodoo 1 and 2.   
PowerVR differently from all available z-buffer based chipsets used a different approach called tile-based 
deferred rendering. First the polygon data is stored in memory as triangle strip or indexed format. The 
rendering is not performed until all polygon data is collected for current frame. Also expensive operations 
as texturing and shading is delayed until visible surface of pixels is determined. Next the screen is divided 
into rectangular tile grid and each tile has associated the list of triangle that are visible on this tile. Finally 
it is determined in a way similar to ray-casting which triangle is the closest one for a specific position and 
that pixel is rendered. This method does have an advantage that no calculations need to be made to 
determine what a polygon looks like in an area where it is obscured by other geometry. Unfortunately this 
approach did not survive on the PC market and is only used in some of integrated GPU’s now, but on the 
other hand newest PowerVR cards power our mobile devices nowadays.   
==PAGE:8==

However, with progress, 2D GUI acceleration and 3D functionality were all integrated into one chip 
without performance loss. 
During 1990’s a big step was implementing so called T&L (transform, clipping, and lighting) solution. This 
solution consisted of 3 parts namely transformation, clipping and lightning. Transformation is the task of 
producing a two-dimensional view of a three-dimensional scene. Clipping means only drawing the parts of 
the scene that will be present in the picture after rendering is completed and finally lighting is the task of 
altering the colour of the various surfaces of the scene on the basis of lighting information. Until late 90s 
T&L was software implemented yet CPU’s did not cope with steadily rising demand for more complex 
scenes.  First to implement HW T&L were arcade systems such as the Sega Model 2 and Namco Magic 
Edge Hornet Simulator. Later on Rendition in collaboration with Hercules and Fujitsu tried to create a PC 
graphics card with full T&L yet failed to market it. 
Hardware accelerated T&L was finally implemented in consumer class cards by NVidia GeForce 256 (also 
known as NV10). With this T&L set the path for future vertex and pixel shaders.  
During 1990’s also a lot happened regarding API’s. First OpenGL was released in 1992 yet in the beginning 
it lost a performance battle against already existing Glide API. However the performance issues of OGL 
were quickly resolved and it won the war against its competitor. During that time also parity emerged 
between OpenGL API and hardware feature set. The other popular API which was created at that time 
was DirectX (1994). 
2000-2005 
In early 2000’s OpenGL and DirectX API’s evolved introducing shaders, what led to implementation of 
special hardware units to process those. Each pixel and each vertex now could be defined by a short c-like 
program and processed before being put to framebuffer.  NVidia was first to produce a chip capable of 
programmable shading (GeForce 3 - code named NV20). Later on ATI also caught up with the introduction 
of the ATI Radeon 9700 (also known as R300).  
After 2006 
With introduction of NVidia’s GeForce 8 series GPU became a more generalized computing device. With 
power of CUDA or OpenCL API’s for GPU parallel computing GPU’s started to compete against CPU’s in 
computing speed. General Purpose Computing on GPU, has found its way into variety of fields from 
machine learning, scientific image processing, and 3D reconstruction to stock options pricing 
determination.

Conclusion 
 
HISTORY OF OPENGL 
OpenGL was first created as an open and reproducible alternative to Iris GL which had 
been the proprietary graphics API on Silicon Graphics workstations. Although OpenGL 
was initially similar in some respects to IrisGL the lack of a formal specification and 
conformance tests made Iris GL unsuitable for broader adoption. Mark Segal and Kurt 
Akeley authored the OpenGL 1.0 specification which tried to formalize the definition 
of a useful graphics API and made cross platform non-SGI 3rd party implementation 
and support viable. (Khronos Group, 2014) 
OpenGL is the oldest of 3 API’s I will write about in this thesis. It was based on SGI’s Iris GL and was turned 
into open standard due to weakening SGI’s market share. At that time the leading programming language 
was C so first OpenGL API was made C style which stayed until today. It is built on the concept of a state 
machine, though more recent OpenGL versions have transformed it into much more of an object-based 
system.  
OpenGL was first released in 1992. During that time each GPU supplier provided his own API what made 
cross-platform development really hard. OpenGL was made by SGI based on their proprietary API called 
Iris GL. The idea of OpenGL was to create a useful, cross-platform graphics API for everyone to use. The 
first version did disappoint a bit. It had performance problems and did not support texture objects. Lack 
of texture objects made dynamic change of texture impossible. The feature was added soon in OpenGL 
1.1 patch and made the API much easier and dynamic. Now developers could use texture objects to 
define textures they use by name.  
When OpenGL started to really have an impact on the market SGI decided to transfer control of this open-
source standard to so called ARB (Architecture Review Board). The ARB included such big companies as: 
3Dlabs, Apple, ATI, Dell, IBM, Intel, NVIDIA, SGI and Sun Microsystems. Also Microsoft was once a 
member but left in March of 2003.

Under control of ARB OpenGL was developed promoting some extensions by each version. Each GPU 
manufacturer was able to define extensions that they supported. Examples of such extensions are in early 
days: EXT_vertex_array defining vertex array feature in OpenGL 1.1 or EXT_texture defining additional 
texture object capabilities also in OpenGL 1.1.  In 1998 version 1.2.1 of OpenGL defined ARB extension 
concept.  
ARB extensions are not required to be supported by a conformant OpenGL 
implementation, but are expected to be widely available; they define functionality 
that is likely to move into the required feature set in a future revision of the 
specification. (Khronos Group, 2014) 
This concept helped to manage expanding market of OpenGL extensions. Without it OpenGL forks would 
become proprietary APIs of many different manufacturers and it would stop being cross-platform.  
As mentioned above each consecutive version of OpenGL promoted some extensions to become a 
standard. The full list of promoted extensions can be found in Appendix A. Here I will explain the most 
important and interesting ones.  
As mentioned in the beginning OpenGL 1.0 was missing Texture Objects. Those were introduced in 
OpenGL 1.1 by promotion of EXT_texture_object. This extension introduces named texture objects what 
allowed to create dynamic texture objects not bound to static display list as in pre-extension OpenGL. 
Together with texture objects extensions came also EXT_texture, EXT_copy_texture and EXT_subtexture. 
First did formalize texture formats and behaviours while others allowed to change image data hold by 
texture. Last important and widely used improvement of OpenGL 1.1 was EXT_vertex_array which added 
the ability to specify multiple geometric primitives with very few subroutine calls. This made OpenGL 
much faster when needed. OpenGL 1.1 is the last OpenGL supported by default drivers in Windows 
systems. 
In 1998 OpenGL 1.2 was released. It contained mostly extensions concerning texture and pixel format. 
Only non-colour/texture extensions were EXT_rescale_normal and EXT_draw_range_elements. I will start 
with explaining those 2. Rescale normal when enabled made it possible to rescale normal vector after it is 
multiplied by the inverse modelview matrix and before it is normalized. With this it was possible to avoid 
normalizing normal vectors in special cases. EXT_draw_range_elements on the other hand was an API 
extension allowing usage of glDrawRangeElementsEXT command which allowed user to draw a range of 
elements instead of full contents of vertex array. As for texture/colour extensions probably most 
important were addition of new blending equations, 3D textures and texture edge clamping.  
During the same year concept of ARB extensions was developed and introduced. With this ARB extensions 
were created which are defined as: 
ARB extensions are not required to be supported by a conformant OpenGL 
implementation, but are expected to be widely available; they define functionality 
==PAGE:11==
 
10 
 
 
that is likely to move into the required feature set in a future revision of the 
specification. 
GL implementations of such later revisions should continue to export the name strings 
of promoted extensions in the EXTENSIONS string, and continue to support the ARB-
affixed versions of functions and enumerants as a transition aid. (Khronos Group, 
2014) 
With this OpenGL development was moved under control of ARB board which was now expecting that if 
an extension is introduced as ARB it will be widely available. This helped to make OpenGL even more cross 
platform because until than there were a lot of vendor specific extensions. 
Three years later in 2001 OpenGL 1.3 was released. It was 90% texture based. It introduced multitexturing 
so widely used now as well as cube maps. Multitexturing allows developer to apply more than one texture 
to an object. With this new possibilities opened like easier decal rendering as well as normal or bump 
mappings. Cube maps also were a nice additions allowing for creation of environment maps as well as 
skyboxes in much simpler way. Together with multitexture came texture environment extensions which 
defined dependencies between applied textures. Also the first antialiasing was included in this release 
with ARB_multisample extension. 
OpenGL 1.4 (2002) added a few interesting API based features. Two of them are depth texture and API 
shadow support. Depth texture is was a very important extension. It defined new depth texture format 
which is widely used not only in shadow casting but also in displacement mapping or image based 
rendering. Shadow extensions uses the before mentioned depth texture to compare texture R coordinate 
to depth value to produce shadow maps. First version of extensions supported only 2 comparison 
equations:  GL_LEQUAL and GL_GEQUAL. Another important extension from the point of view of this 
thesis was EXT_multi_draw_arrays. This extension addressed so called small batch problem which was hit 
much later. It allowed to specify a list of vertex arrays to be drawn by one API call. This allowed for 
batching of separate primitives. The call to MultiDraw* was equal to calling Draw* function in a loop. 
Apart from before mentioned extensions update 1.4 of OpenGL specification added a lot of other 
extensions like new blending equations, new texture modes including texture mirrored repeat, fog 
enhancements and some more. 
The next year OpenGL was moved to version 1.5. It only promoted 3 extensions yet those had a huge 
impact. The first and probably the most important of those was definition of VBO (Vertex Buffer Object 
[ARB_vertex_buffer_object]). This extension allowed large portions of data to be cached inside high-speed 
GPU memory instead of each time being transferred from RAM. With this extension the cost of drawing 
static primitives was cut by the value of speed loss at memory transfer. Another extension is also 
performance based. It is occlusion query. Occlusion query allows program to check how many samples of 
primitive are visible and gives the user the result. Based on this result programmer can either draw or not 
a primitive. Very useful when we have many small primitives occluded in the scene. Finally the last 
==PAGE:12==
 
11 
 
 
extension included in version 1.5 is EXT_shadow_funcs which expands the previous shadow extension to 
support all 8 comparison equations.  
Next version of specification was OpenGL 2.0 (2004). It finally featured more or less programmable 
pipeline with introduction of shaders. Shaders are simply speaking computer programs that are used to 
do shading. Shading is a process of producing appropriate colour within an image and special effects. 
Shading is also used in video post-processing. The OpenGL shading language is called GLSL (GL Shading 
Language). It’s written as C style short programs with its own constants and methods defined. GLSL 
shaders are compiled at runtime of OpenGL program. Shaders can be used to define vertices, geometry, 
pixel values as well as nowadays to compute data. In OpenGL 2.0 though GLSL was still in early 
development (v1.1) and it featured only vertex and pixel shaders. OpenGL 2.0 also already featured 
multiple rendering targets (MRT). MRT is all about drawing the fragment shader data to multiple colour 
buffers. This opened a way to technique called deferred rendering, which later was improved by 
introduction of framebuffers. This version of specification also added non power of two textures which 
made it easier to create and store textures as well as some lesser stencil improvements.  
Afterwards users had to wait a long time for big changes. In the meantime in 2006 ARB introduced a small 
patch called OpenGL 2.1 which included pixel buffer objects and sRGB texture as well as updated shading 
language to version 1.20. The most important change in GLSL was introduction of non-square matrices. 
Finally in 2008 already under control of Khronos Group OpenGL in revolutionary (evolutionary) version 3.0 
was released.  
OpenGL 3.0 most important addition was not any of extensions, but a new concept. This concept was 
called deprecation model. Deprecation model was all about cleaning up the API. Features marked as 
deprecated from now on are allowed to be removed in future versions of API. The list of features marked 
as deprecated in each new API version is listed in APPENDIX B.  
Since Khronos takeover OpenGL development sped up and many new extensions were added. To make 
the history lesson more concise I will skip the lesser ones and focus on really important extensions 
introduced and major events. 
The first of very important extensions introduced in OpenGL 3.0 is mentioned before as improvement to 
MRT framebuffer object. Framebuffer object allowed users to create a virtual render target with set 
parameters and attached targets hidden from end-user view. So called offscreen renderer. With this it 
became possible to pre-render scene behind the scenes and later use it as texture or textures.  
Framebuffer objects are fully user defined so not dependant on hardware framebuffer. This extension 
opened way for offscreen rendering with cheap deferred rendering, high resolution reflections and 
refractions as well as shadow maps, screen-space effects and many more.  
Next extension introduced which improved performance were Vertex Array Objects.  Vertex Array Objects 
group Vertex Buffer Objects to allow quick binding. Instead of binding each VBO separately we bind VAO 
which contains all needed VBO’s. It has to be noted that we also can enable or disable Vertex Attribute 
Arrays for different purposes.  
==PAGE:13==
 
12 
 
 
OpenGL 3.0 also expanded Occlusion Queries by introducing Conditional Rendering extension. Now it was 
possible from the API level to render complex objects based on result of occlusion query of cheap dummy 
object. 
Finally Transform Feedback was added. It improved performance by allowing program to register 
transformation state after vertex shader pass. With this transformations can be done only once for first 
object and further same objects can be drawn with transformation already applied in vertex data. 
OpenGL 3.0 also implemented a competitive way of storing textures (Texture Arrays against Texture 
atlases) and some new formats and colour types. 
During next year OpenGL 3.1 and 3.2 were developed. Features marked as deprecated in version 3.0 were 
removed except wide lines. OpenGL 3.1 introduced Shading Language 1.40 and OpenGL 3.2 upgraded it to 
1.50. Also texture extensions were included namely buffer texture, texture rectangle and multisampling 
for textures. Multiple primitive drawing was boosted by primitive restart extension and instanced 
rendering was also introduced. Primitive restart was better than MultiDraw* because it allowed to restart 
a primitive at special index instead of after a set number if indexes drawn. With this it was possible to 
draw at once lots of different primitives. Also to improve shaders Uniform Buffer objects for storing 
uniforms were added. OpenGL 3.2 was focused mainly on improving GLSL. Probably the biggest addition 
to OpenGL 3.2 and GLSL 1.50 were geometry shaders. Geometry shaders govern the processing of 
primitives. Now it was possible to input a primitive into a shader and output 0 or more primitives from it.  
Year 2010 was a very important year in history of OpenGL. 2 new versions were released. OpenGL 3.3 for 
older GPU hardware and newly implemented 4.0 for newest GPU’s. 
OpenGL 3.3 included small update to GLSL to version 3.30, explicit shader locations for vertex attributes 
and fragment shader outputs as well as improvement to occlusion query. Occlusion query now was able 
to output simple Boolean answer instead of amount of occluded fragments. Also Timer queries were 
introduced for measuring GPU time spent on command.  
OpenGL 4.0 on the other hand added a large update of GLSL to version 4.40. Also added tessellation what 
allowed users to use the new GPUs feature. What exactly is tessellation? OpenGL defines tessellation as: 
Tessellation is the Vertex Processing stage in the OpenGL rendering pipeline where 
patches of vertex data are subdivided into smaller Primitives. This process is governed 
by two shader stages and a fixed-function stage. (Khronos, 2015) 
Tessellation allows for dynamic level of detail control not only based on texture detail but also on polygon 
count. With it when more details are needed to be displayed tessellation shader outputs more vertices to 
achieve bigger geometry detail.  
OpenGL 4.0 also expanded texture array support to include cube-maps and added some new texture 
formats. It additionally improved transform feedback mechanism.  
==PAGE:14==
 
13 
 
 
In the same year OpenGL 4.1 was released. It added support of pre-compiled shader programs what 
allowed users to ship their own precompiled shaders with program to avoid runtime compilation which 
slowed down loading of program. Also it added ability to use shaders in mix-and-match fashion. Now it 
was possible to bind specific shaders to specific pipeline parts instead of binding all shaders at once. 
In 2011 OpenGL 4.2 was mostly usability expansion. It updated shader language to version 4.20, added 
some floating point operation improvements and some shader and buffer operations. 
OpenGL 4.3 apart from major usability improvement based on improving already existing mechanisms 
and extensions added finally support for Arbitrary Compute Shaders. Compute shaders are shaders that 
do not calculate GPU specific data like vertices or fragments but allow user to make arbitrary 
computations on sometimes much faster GPU unit rather than CPU.  
OpenGL 4.4 improved shader layouts and added texture mirror clamp to edge extension. Texture mirror 
clamp to edge adds a new special clamping mode that doubles the size of texture in each dimension 
mirroring it once in negative direction. 
Finally the newest OpenGL standard was released in 2014 marking version 4.5. Apart from some minor 
improvements OpenGL 4.5 added long awaited state machine override called Direct State Access. This 
functionality basically allows querying and modification of object state without binding. So if we want to 
change texture parameter we don’t need to as we used to save previous binding, bind new texture, 
change parameter and rebind. It can be done in one simple direct state access command. 
As we can see from above history OpenGL instead of revolutionizing the Graphics programming merely 
adopted what was created. The idea of 3rd party extensions made it follow trends instead of making new 
ones. But on the other hand each OpenGL feature was already widely supported when being introduced 
differently from DirectX that I will analyse next.  
 
HISTORY OF DIRECTX 
In 1994 Microsoft was working on a new operating system, Windows 95. The 
Microsoft game programming platform of that time was DOS. Most programmers 
found DOS a better platform to build on, then the new Windows 95. 
(...)  Microsoft needed a way to give the programmers the same access to (these) 
devices on the new operating system Windows 95 (as in DOS) 
(…)Three Microsoft employees, Craig Eisler, Alex St.John and Eric Engstrum, came 
together to fix this problem. (…). They build the first version of DirectX upon the 
concepts of another development system “Exodus”. “Exodus” was developed by 
Kinesoft Development. Microsoft and Kinesoft worked closely that year (1994) to 
develop DirectX 1.0. (CodingUnit.com, 2014) 
==PAGE:15==
 
14 
 
 
In 1992 RenderMorphics was started which developed an API called “Reality Lab” for medical and CAD 
software. In February 1995 this company was bought by Microsoft and “Reality Lab” became the 
foundation for new API called Direct3D. First version of Direct3D was shipped with DirectX 2.0 and 3.0. It 
included so called "retained mode" and "immediate mode" 3D APIs. The first mode was a COM-based 
strict scene graph API. This mode only survived till DirectX 3.0 due to it strictness and little robustness. 
The second mode also had issues in the beginning. Due to its extremely awkward to program “execute 
buffers” approach it actually raised calls for Microsoft to adopt OpenGL as the official 3D rendering API for 
games. But Microsoft did not yield and continued to improve Direct3D.  
Next versions didn’t show significant improvements till not even released Direct3D 4. This version of 
Direct3D contained changes designed to support Microsoft Talisman project. It was a novel 3D hardware 
spec which used sprite-based image composition as a way to boost 3D rendering performance. It was the 
predecessor for nowadays MRT (multiple render targets) but did not succeed due to low hardware 
support. Interesting fact is that dominant back than vendors who decided not so support new idea 
eventually went out of business. Even if Talisman did not succeed the API for SetRenderTarget was 
included in this and future versions. And here we can see a significant difference between OpenGL and 
DirectX. As OpenGL merely adopted the ideas of HW vendors DirectX went against them developing its 
own approach. To be able to proceed against the HW tide Microsoft decided to make certain Hardware 
DirectX certified what when DirectX became popular made a good marketing point for vendors. 
Finally next released version after close work of hardware vendors and Microsoft was released. Due to 
this cooperation vendor-specific extensions had to be added to GPU drivers. Also Direct3D 5 as it was 
called contained all improvement of Direct3D 4 and also introduced the DrawPrimitive API that eliminated 
the need for applications to construct execute buffers. 
Next release was just an evolution incorporating numerous features covering contemporary hardware. 
Those were multitexturing and stencil buffers. Also it optimized geometry pipelines and optional texture 
management to simplify programming. Starting with this Direct3D release API also supported features 
that had been licensed by Microsoft from specific hardware vendors for inclusion in the API, in exchange 
for the time-to-market advantage to the licensing vendor. Those were for example: S3’s texture 
compression, or TriTech's proprietary bump mapping technique. Those licenced features were added into 
DirectX and were required to be present for GPU to get Windows Logo. This encouraged broader adoption 
of features in other vendors’ hardware. 
Direct3D 7.0 further expanded feature base. It added .dds textures and support for T&L (transform, 
clipping, and lighting) API. Also it allowed for allocation of vertex buffers in hardware memory. This vertex 
buffer allocation was first substantial win of DirectX over OpenGL at that time. 
Next Direct3D 8.0 introduced finally shaders which were a much simpler way to achieve things than the 
lightning pipeline from previous release. It was a major departure from OpenGL-style fixed –function 
pipeline architecture. Also DirectDraw became a separate API back then. Actually even with shaders 
Direct3D was not considered user-friendly at the beginning yet update to 8.1 resolved many usability 
==PAGE:16==
 
15 
 
 
problems. This DirectX contained already a lot of powerful 3D graphics features like: vertex and pixel 
shaders, and API based bump and texture mapping.  
Direct3D 9.0 came with further improvements. It contained new version of the High Level Shader 
Language (HLSL), Multiple Render Targets (MRT), support for floating-point texture formats, Multiple-
Element Textures, texture lookups in the vertex shader and stencil buffer techniques. 
There was an extension for Direct3D called 9.0Ex (9.0L) exclusively for Windows Vista. It used the 
advantages offered by Windows Display Driver Model (WDDM) introduced in above mentioned release of 
Windows OS. It for example allowed graphics memory to be virtualized and paged out to system memory, 
what allowed graphics operations to be interrupted and scheduled and allowed DirectX surfaces to be 
shared across processes.  
During Windows Vista time on market in 2007 came a large update to Direct3D called Direct3D 10. It 
featured shader model 4.0 and optional interruptability of the shader programs. In this shader model all 
stages support a nearly unified interface, as well as a unified access paradigm for resources such as 
textures and shader constants. The HLSL language was also extended. The newest pipeline also includes a 
geometry shader stage which allow geometry to actually be generated from within a shader, allowing for 
complex geometry to be generated entirely on the graphics hardware. DirectX 10 is supported only by 
Windows Vista and above. Direct3D 10 also resigned from so called “capability bits”. Instead, it defines a 
minimum standard of hardware capabilities which must be supported for a display system to be "Direct3D 
10 compatible". The most significant changes introduced by Direct3D 10 were change to fully 
programmable pipelines instead of fixed ones, new state object to enable (mostly) the CPU to change 
states efficiently, shader model 4.0 which added instructions for integer and bitwise calculations, texture 
arrays enable swapping of textures in GPU without CPU intervention, predicated Rendering that allowed 
drawing calls to be ignored based on some other conditions and better instancing support, allowing 
multiple instances of similar meshes, to be rendered in a single draw call, reducing the processing time 
needed for multiple similar objects to be rendered. 
Direct3D 10.1 was announced shortly after Direct3D 10.0 was released as a minor update. It contained 
mostly minor updates for formats and conversions as well as Shader model 4.1 update. 
DirectD3 10.1 also introduced concept of "feature levels". New features are supported exclusively by new 
hardware which expose needed feature level. 
In 2008 with release of Windows 7, Direct3D 11 was presented. It had a long feature list .Shader Model 
got updated to version 5.0. Dynamic shader linking was added. Also a few texture updates and new types 
and function were introduced. Resource management was improved with addressable resources and 
additional resource types. DirectX 11 also came with improved multithreading and finally support for 
Tessellation and Tessellation Shaders as well as compute shaders. Tessellation and compute shaders were 
already described in previous chapter.

HISTORY OF MANTLE 
In 2013 AMD and DICE decided to create their own low-level high-performance graphics API for Radeon 
cards independent from OpenGL and DirectX. That’s how Mantle’s history started. Mantle is meant to be 
a console style API which allows developers to code “closer to metal” which means that it allows more 
direct access to hardware. This results in lower CPU overhead what in instance is said to give about 9x 
better performance. Mantle is said to be similar to NVidia CUDA API yet the latter one is meant for GPGPU 
and Mantle is the typical 3D targeted API. First Game that included Mantle backend was Battlefield 4 from

OpenGL is a C-style API. To make it object oriented user needs to implement his own wrappers. OpenGL is 
also a state based machine. That means its execution is based on bind states. User needs to bind the 
element he wants to edit or read before accessing it. In OpenGL 4.5 API Direct State Access was 
introduced for a few features which greatly improved the experience of programming in OpenGL. 
DirectX 
DirectX is a C/C++ library. It used to be shipped independently from Windows systems but since 9 for 
Windows Vista it started to be shipped inside new systems. DirectX is made to resemble graphics card 
interfaces.  
Graphics pipeline of DirectX looks quite similar to OpenGL’s. It first assembles vertex and index buffers 
and passes them to vertex shader. Vertex shader processes vertex data by use of shader resources and 
constants and outputs data to tessellation stage starting with hull shader.  Hull shader takes patch control 
points as input and outputs to tessellation shader how much to tessellate patch and tessellator mode 
declarations and to domain shader the patch control points after basic conversion. Next tessellator 
computes the input data and outputs U V domain points to domain shader and topology to primitive 
assembly. Finally for each tessellator output point domain shader outputs one vertex. Those vertices go 
through geometry shader stage and pipeline moves to rasterizer. Finally it reaches pixel shader and 
output merger which outputs data to render targets. Parallelly Compute shaders are invoked yet it is 
impossible to read/write at the same time from shader resources together by rendering pipeline and 
compute shader. Graphics pipeline of DirectX can be seen below.

NEW API’S 
Mantle API 
My Mantle API analysis I based on prerelease presentation at GDC 14 by AMD. From code snippets shown 
on this presentation Mantle syntax is a kind of hybrid of OpenGL and DirectX syntaxes pointing a bit more 
towards OpenGL. Syntax consists of commands, constants and data types as in OpenGL but also has 
complex macro structures passed to methods as DirectX. Each command starts with gr prefix. Next come 
words each starting with a capital letters.  
Next on list are constants. Those start with GR_ prefix and have all capital letters. Words are separated by 
underscore. The example can be GL_COLOR_BUFFER_BIT. 
Data types in Mantle API are defined by macros. For example GR_UINT. 
Mantle API introduces new look on graphics pipeline. It resigns from driver set context pipeline and allows 
user to define pipelines through the API. The monolithic pipelines can contain resources and shaders and 
define states connected to latter. They do not include render targets, blending, index buffers nor dynamic 
states which go to default fixed pipeline.

Mantle introduces some interesting new features. It has a new execution model based on queues, takes a 
fresh new look at memory management and has a new hierarchical binding model.  
The execution model of mantle is based on access to command buffer of GPU. It allows developer to push 
commands to queues themselves and not make them rely of GPU driver context. Mantle devides GPU 
logically into Graphics, compute and other DMA parts. Than it creates queues for those parts to put 
commands into. This way developer has full control over performance. Also queues can be accessed and 
filled by many applications what enables multithreaded rendering.  The execution model diagram can be 
seen in image below.

Those are memory and images.

any other computable resources.  
Image resource type is used to 
store textures and shader output so 
basically speaking all image based 
data.  
Mantle also introduces new resource binding model.  
The Mantle binding model takes what is good in both traditional resource binding model as well as in new 
bindless models. It introduces a concept of descriptor sets. Descriptor set is an array of slots prepared for 
different resources. This way for static data descriptor sets can be prebaked in the beginning what 
improved reading and loading performance drastically. Also for dynamic data it helps to have a 
hierarchical binding model because when we need to update dynamic resource we do not have to reload 
everything in the descriptor. We just update and redraw the part that gets changed.

many lost blocks due to this kind of allocation. In mantle API object is just an array describing object. 
Memory has to be allocated by developer. This approach allows developer to manage memory more 
efficiently and as well to reuse already allocated memory while it is not needed by previous object.

Ownership and compatibility 
Each of above mentioned APIs has its own ownership status and compatible devices. Let us have a quick 
look at ownership and compatibility. 
Let us start from OpenGL. OpenGL is not an open-source library. OpenGL is an open specification led and 
governed by Khronos Group. With this OpenGL as itself does not have any implementation. OpenGL 
specification is being implemented by hardware vendors in their drivers. Only one OpenGL 
implementation existing away from HW vendors is called Mesa3D. With this way of spreading OpenGL in 
higher or lower version is compatible with almost every device available.   
==PAGE:24==
 
23 
 
 
DirectX on the other hand has a different approach. It is a set of proprietary libraries owned by Microsoft. 
New versions of DirectX are mostly delivered with new Windows OS iterations. DirectX differently from 
OpenGL is only running on hardware which is marked compatible by Microsoft in the process of 
certification so that it behaves the same on each.  
Finally Mantle API is also proprietary API owned by AMD. It is also closed source and is unfortunately only 
compatible with newest top ATI graphics cards using up to date drivers. In this case it’s somewhat similar 
to OpenGL by being driver implemented yet on the other hand it’s being reserved only for ATI. 
How they differ 
All above APIs differ a lot. It can be said that they support mostly the same set of features and allow 
similar effects to be achieved, but they differ in both performance, architecture and even history.  
OpenGL and DirectX are old APIs with a long and successful history. Both OpenGL and DirectX were 
started in early 90s and were developed ever since. On the other hand Mantle is a new player. It’s still in 
beta and still have many child-time flaws. But is it really that much worse without long record? OpenGL 
and DirectX contain even now some APIs that were developed still in early 90s based on concepts from 
those times. Sometimes fresh look like Mantles can become revolutionary. When one uses the concept 
for a long time it’s sometimes hard to tell if the concept is still valid. Mantle does not have this burden. It 
can start anew not influenced by its history.  
Another point that differs the APIs is their availability. Here OpenGL and DirectX win clearly. Both older 
APIs have a huge base of developers, users and hardware to run on. On the other hand Mantle is 
narrowed to a few ATI cards. That’s a big drawback because ATI market share is not as good as for 
example NVidia’s.  
Finally we have a difference in performance and complexity. That’s where Mantle exceeds its older 
competitors. Both OpenGL and DirectX through the years grew slow and build a large Driver overhead. 
Also in case of complexity additions made in each version made those APIs have a lot of methods and 
features which nowadays are not used or rarely used and should be replaced or removed. Mantle started 
clean and managed to create fast, small and concise API which addresses modern graphics developer’s 
needs.

and later move to my own driver overhead test.

Measurement of current driver overhead 
To prove that Mantle creators were not lying about driver overhead in current API implementations and 
to prove above results cause I decided to measure naively the driver CPU overhead of OpenGL API.  
My CPU setup is like follows:

8 GB

I wrote a simple OpenGL Application. A rotating textured cube. This application included VAO/VBO 
creation , texture load, state bindings and drawing. Then I used OpenGL Timer Query to measure the time 
needed to run one random API call as proof of concept. The API call was: 
glVertexAttribPointer(0,3,GL_FLOAT,GL_FALSE,0,(void*)0); 
This API method binds already created VBO to layout position 0. It should be straightforward and easy on 
CPU.  
I decided to measure time this command spent on CPU and GPU. CPU measurement was straightforward 
because I could use glfw built in method called glfwGetTime(). This method returns double type number 
indicating how many seconds passed since the start of glfw. The measurement of GPU time was a bit 
more tricky but during research I stumbled upon a nice hidden functionality of OpenGL called Timer 
Query. It allows to measure the time spent on GPU between glBeginQuery and glEndQuery calls. To make 
it clean I used macros :  
#define START_QUERY(name) \ 
 functions.push_back(name); \ 
 queries.push_back(0); \ 
 glGenQueries(1, &queries.back()); \ 
 glBeginQuery(GL_TIME_ELAPSED, queries.back()); \ 
 timeEnd = 0; \ 
 timeStart = glfwGetTime(); 
 
#define END_QUERY \ 
 timeEnd = glfwGetTime(); \ 
 glEndQuery(GL_TIME_ELAPSED); \ 
 cpuTimes.push_back((timeEnd - timeStart)*1000000000.0);

START_QUERY pushes a function name passed as parameter and creates an empty query. Than it 
generates new query on created item and starts to query GPU also it sets timeEnd to 0 and time start to 
current CPU time.  
END_QUERY sets the end time for CPU query and stops the gl query as well as pushes cpu time to list of 
cpuTImes.  
At the end of progam execution the following code reads glQueries and cpu queries and writes the to file 
called Results.txt.

of validation was given to user what made this kind of situations not that stressful to CPU. 
CONCLUSIONS 
Based on all above research let us now answer the question we posted in the beginning. Are new APIs 
really a step forward or a marketing turf war? Are they adding some real new value to industry or are they 
just showing off their skill? 
As we can see from above research the performance increase Mantle offers is significant. The closer-to-
metal approach reducing driver overhead which got proved to be huge does show a point. Older APIs like 
OpenGL and DirectX for years collected parts of useless code and were not cleaned up. Specifications 
written in early 90s even updated and extended are reaching their limits. The basic principles on which 
they work are becoming less and less valid confronted with modern GPU architecture as well as modern
