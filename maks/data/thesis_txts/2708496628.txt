Abstract
The cyber physical interaction becomes an important field with a tendency using Ethernet
technology replace the traditional field bus. Hard real time system is required for time
critical applications, in automation, autonomous system, communication for IoT devices
in factories etc. This work focuses on building up a real time TCP/IP network stack, which
provides deterministic time bounds for network transmission. The stack avoids blocking
methods in the socket programming, and runs on the time predictable processor Patmos.
The original ethlib is modified to suit the time predictable requirement. This stack extends
the Patmos processor with a WCET analyzable network software implementation.

Introduction
TCP/IP protocol stack is widely used in network connections. In [1] and [2], the authors
give an introduction about the Berkeley TCP/IP stack and how this classical stack works in
industry as Ethernet/IP. Socket programming is the way to build up networks, using block­
ing API functions. Although it works for the normal internet applications, the trend that
replacing the fieldbus with Internet technology in the industry expects time­constrained
solutions. Most of the applications belong to soft real time system, which means they
don’t have a strict deadline for each task. The demand of strict time deadline in hard real
time system becomes more important in network communication domain [3].
In order to build up a real­time network stack, the traditional socket method should be
switched to a non­blocking way, with bounded iterations and polling for periodic tasks.
Here TCP/IP 5 layers model is used as shown in table1.1.
tpIP [4] is a time predictable TCP/IP stack for real time IoT devices based on Patmos [5],
a time­predictable excution platform. The Patmos platform supports worst case execution
time (WCET) analysis, also an ethernet controller in both hardware and software, while
it lacks a time perdictable network stack for deterministic transmission through Internet
cables. tpIP provides a basic structure of a real time network stack.
The purpose for this master thesis aims to continue with tpIP work and complete it as a
fully time predictable network stack, focus on the local area network (LAN), make the rest
layers of TCP/IP network, e.g. application, data link layer, time predictable and WCET
analyzable.
The thesis is organized as follows. Chapter 2 provides recent research outputs about the
real time TCP/IP stack for embedded devices. Chapter 3 gives a introduction of back­
ground knowledge, such as protocols used in this thesis. Chapter 4 lists the hardware
and software tools that this thesis relied on, as well as the way to configure the system.
Chapter 5 illustrates the implementation of the TCP/IP stack. Results is shown in Chapter
6. The conclusion and future work are presented in the last chapter. The code is pushed
to tpip repository [6].

Related Work
2.1
Real Time Network Stack
Nowadays, most of the systems developed based on 4.4BSD Lite from UC Berkeley sup­
port a famous open source TCP/IP stack. It provides a set of the BSD socket API for net­
work connection [7]. This socket programming approach has to block the current thread
when the program is waiting for a valid signal, so that it is time unpredictable. Another
drawback is this stack is too heavy to be carried by embedded system [8]. Therefore,
there are two directions to adjust current TCP stack for embedded things: smaller size
and faster response time.
Some TCP/IP stacks focusing on size optimization are built up. The author of [8] makes
two light weight TCP/IP stack for embedded systems called lwIP and uIP, short for light
weight IP and micro IP respectively. lwIP supports TCP retransmission while uIP does
not. Their target architecture ranges from 8 bits to 32 bits RISC­V microcontrollers. The
code size numbers compiled for each platform are given, which only needs a few tens
Kilobytes.
LyraNet [9] applies a zero copy policy to shrink the code size by 25% compared with the
original network stack. The data from user applications are kept inside a piece of user
memory space. It uses¨an array buffer to store the data address and length in the user
memory instead of copying the whole data between different buffers. The sending and
receiving time is optimized significantly at the same time. Memory copy should also be
avoided in this project.
TINET [10] makes the network stack in a component based shape which enables the
scalability of the network structure. The protocols and functions inside different layers are
organized as different modules, which makes it much easy to modify the structure, add or
delete some components.
2.2
Real Time Protocols
Except building up a whole stack, there are also research outcomes which try to improve
the TCP stack in a specific layer.
CoAP [11] is developed for resource constrained devices in the application layer, in order
to replace the general used HTTP. It is based on UDP and provides the same services as
HTTP does, but runs with much less resource.
In [12] and table V in [13], the authors analyze several popular application layer proto­
cols in terms of latency, such as MQTT, CoAP, RESTful(representational state transfer),
WebSocket etc. They conclude that AMQP and MQTT are not suitable for real time ap­
plication. MQTT, CoAP, and XMPP are possible options for web applications that require
real­time communication, but with different latency characteristic. Most of the influential
application protocols for IoT are listed in [14].
[15] shows the trend of the field level network development. It shows a clear map that
the field networking is growing up with merger between traditional field bus technology
and modern network software stack. And now several wireless standards are issued,
like IEEE 802.15 series. [16] lists some well known current development products and
gives their delivery time. Time Triggered Ethernet (TTEthernet) in [17] provides a time

Background
This chapter explains the hardware structure, network protocols and software concepts
used in this project for a time predictable network stack.
3.1
Ethernet­MAC Controller
A Ethernet controller based on EthMac IP [18][19] from OpenCore is integrated inside Pat­
mos. The open source IP EthMac, which connects with Patmos between an Open Core
Protocol (OCP) interface, is used as the main controller accepting transactions from PHY
chips. It only supports for media­independent interface (MII) with 100 MBit/s Ethernet.
The EthMac IP can be divided into EthMac RX/TX buffer and EthMac descriptor, which
has a OCP and WishBone (WB) interface respectively. The signal from Patmos passes
the Demux first. Configuration signal is guided to EthMac controller through a OCP/WB
transfering bridge, converting OCP signal into WB format. Otherwise it will store in the
buffer.
This controller has a valid address offsets from 0x0000 to 0xFFFF, where the last 16KB
(0xF000­0xFFFF) is used as the EthMac descriptor. The descriptor monitors the status
of the buffer. Each descriptor controls one cell in the buffer, has the information about
configuration, status flags, and pointed buffer address. After the current cell is filled, one
descriptor should pass the control to the next one, always pointing at the next empty buffer
element.
During the receiving sessions, empty places in the buffer should be guaranteed for com­
ing data, otherwise the packets are discarded automatically. Data are fired as soon as
possible after setting relative descriptors as ready. Then the descriptors are reset to not
ready automatically.

3.2.1
Ethernet II
Ethernet type II applies as the datalink layer protocol, defined in RFC894 [20]. This project
supposes the transition delay is time predictable here, as some works like Time triggered
Ethernet [17] provide possible solutions. The frame is defined in figure3.2.
The destination source MAC addresses are deterministic. There is a list of code numbers
for Ethertype listed in [21]. Some common used ones are 0x0800 (IPv4) and 0x0806
(ARP). Upper layers’ data attached after these 14 bytes Ethernet header. Minimum length
of an Ethernet frame is 64 Bytes with a upper bound 1518 Bytes.
Figure 3.2: Ethernet type II frame
3.2.2
Address Resolution Protocol (ARP)
ARP defined in RFC826 [22] is mainly used to solve MAC address request. Before two
terminals building up a connection, they may do not know each other’s MAC address,
it means the message cannot be received by anyone in its network domain. ARP table
keeps MAC addresses matched with related IP addresses. When Host A wants to send a
message to Host B, it will first search for Host B’s Mac address using Host B’s IP address
in the ARP table. An ARP request is sent if unmatched in the table. Update the ARP table
with an ARP reply which carries the required MAC address. Then the channel between
A and B is built up, messages get ready to transfer.
Figure 3.3: ARP message diagram
3.2.3
IPv4 and UDP
UDP [23] can transmit packets directly, without handshaking. It is a light weight protocol
for embedded systems. IPv4 [24] is the most common used protocol. A lot of transport
layer’s protocols are based on it. The UDP diagram is attached after the IP header in
figure 3.4. A list of protocol numbers maintained by IANA is the reference for the Protocol
parameter [25]. For example, UDP is 17 (0x11) and TCP is 6 (0x06).

Checksum
The checksum is used to detect if there is packet loss during the transmission, and is
set as 0x0000 in origin. The 16­bits one’s complement is then applied for calculating the
checksum. If there is a carry, it is added back in the result. The checksum is generated
until no carry bit shows up.
IP checksum only needs the IP header, while the UDP checksum calculation needs to use
the whole data part as well as a pseudo UDP header. As the one’s complement should
be done in a 16­bits format, one byte of 0 is added at the end of the UDP diagram when
the length is odd.
Checksum verification also uses the 16­bits one’s complement, but including the check­
sum added by the sender. The correct received packet should produce a sum as 0x0000.
Otherwise, the diagram will be discarded.
Figure 3.5: UDP pseudo header for checksum calculation

mention here, which have type 8 and 0. The well­known Ping command is based on this
message, which is few instants the user can interact with the ICMP. The sender’s local

3.2.5
TFTP
TFTP in RFC1350 [27] is used to load images into diskless devices with UDP in the early
age. It can also act as a program loader through network, updating the firmware in a
embedded system.
Using two hosts system as an simple example. Host A sends a write or read request
(WRQ/RRQ) to Host B first, with the target filename and transfer mode (netASCII/Octet).
Then Host B will sent the first acknowledgment with the block number as 0 for a write
request, followed by DATA messages transmitting until the last message which less than
512 Bytes. Data transfer starts immediately after receiving read request. Each transmis­
sion is followed by an acknowledge signal with the same block number. The 5 different
diagrams are listed below.
Figure 3.7: 5 TFTP messages without appending options
There is an extension for TFTP defined in RFC2347 [28], which indicates an additional
operation called OACK. The difference with the original one is that it allows the request
to contain more options, like frame size. An OACK repeat is returned after both write and
read request.

3.3
Time Deterministic Method
To build up a behavior deterministic network stack, the program should at least obey the
following rules [4]:
1. unblocking methods
2. bounded loops
3. limited recursions
4. pre­allocated memory
5. avoid functional pointers
WCET (worst case execution time) analysis is commonly used in time constrained system.
An overview survey is provided in [29].A WCET tool called platin associated with T­CREST
platform [30] can be used within Patmos platform.
Functions such as read and write functions should avoid blocking. The stack is designed
in a polling way, each task is called periodically. Unlimited iteration is impossible to gen­
erate a valid WCET number. All the loops should have limited iteration times. Recursion
is not used in network stack, so no effect on the WCET method. The dynamic mem­
ory allocation, such as malloc and new, can also spend unpredictable machine cycles.
Therefore, all of the buffers and variables are defined during compilation time. No memory
operations during the running time. Functional pointer sometimes cannot be recognised
and extracted address properly from binary code by the WCET tool.
Polling method is an ideal way to control the time intervals. Each device can only send or
receive frames when it is their turns. The other messages are abandoned out of this time
range.
In a nutshell, all the functions and loops used in this stack should executed within limited
machine cycles. This guarantees a fixed maximum transition delay in the software part.
In terms of jitters and transmission latency through Ethernet cables, it can be bounded
within 10 us by using TTEthernet. As this project focuses mainly on the software part,
assume Ethernet is bounded. There is a tool called platin in Patmos. The program is
translated to pml database firstly, then platin can analysis and provide a report according
to it.

System Design
This chapter illustrates the software implementation details for tpIP network stack. The
library is based on the original Patmos C library Ethlib [33]. Ethlib supports a set of pro­
tocols. But there is no segmentation between each network layers. It applies a manner
that accesses the ETHMAC memory directly to interpret the carried information for the
target protocol. Buffer memory address is provided to each protocol for accessing the
data it needs. One apparent drawback is that this structure only matches itself with ETH­
MAC memory map. A lot of efforts have to be done if the address space gets modified or
updated to another IP core with a new structure in the future. Another point is on mainte­
nance and scalability. Each layer should be separated, even the protocols and functions
should be coded in a modularization way if possible.
The program will be constructed in a polling way to receive messages, as the Ethernet
controller does.
5.1
Network Stack Overview
The network stack is shown in figure 5.1. SLIP is used as the datalink layer protocol at
the first version. It is extended to Ethernet in this project. The 3 layers above the datalink
are decoupled from the datalink layer. This method separates the software domain and
hardware domain. There can be more decouple queues between different layers if neces­
sary. The middle two layers provide several functions, such as pack/unpack, checksum
verification. Then the data is passed to application layer. TFTP as an application protocol
is a light weight choice for resource constraint devices.
In terms of time predictability, the software stack should be WCET analyzable locally,
provides a solid number about the maximum executing time. The delay in the physical
layer can be guaranteed by TTEthernet mentioned in chapter 3.3.

5.2
Ethernet Implementation
The Ethernet controller hardware is introduced at section 3.1. The description of ETHMAC
buffer descriptor is in eth_mac_driver.h. It provides the address map and control bits,
masters the sending and receiving. The relative marked buffer descriptor would trigger
shooting or receiving data. The controller status can be detected by registers accessing.
eth_patmos_io.h is for buffer reading or writing, one byte or one word (32 bits) each time.
mac.h reads the Ethernet header and can check the MAC address. It also tells its upper
layer the received protocol type by mac_packet_type.
In order to decouple different layers, TX/RX queues are introduced between data link layer
and network layer. tpip_load_bufin and tpip_load_bufout operate the software buffer
behavior, load the RX buffer and free the TX buffer with ETHMAC memory.

5.3
Transport and Network Layer
In these two layers, the dominant protocols are IP and UDP with their functionalities. ICMP
and ARP perform to reply specific requests.
The protocol type defines which way the message should go to in the network layer. If it
is a ARP request, the ARP functions will handle it, send local MAC address as a reply.
The checksum verification is executed first for a normal IP packet in case some data loss
during the transmission. Then the buffer data are unpacked to a struct type called ip_t.
IMCP functions will process the Ping request, otherwise the message should be passed
to UDP, which has everything in struct udp_t. The application should receive its message
if checksum is correct and this place is its destination.
The sending process is much similar. UDP and IP headers are added in front of the data
block. The source IP and destination IP can be swapped if it sends to the opposite way,
most of the information in the headers is the same as the received one. The IP and UDP
checksum are calculated after the packed IP loaded into the TX buffer, as it is an efficient
and easy way to visit continuous memory address and add the values together.
The Patmos processor is a big endian system. It means an integer e.g. 0xAABBCCDD
stores in the memory in the same order as it shows. So is the network byte order. There­
fore, no need to convert byte order before transmission.
For reading, Ethernet frames is put into the buffer and IP network will de­queue it. If the

5.4
Application Layer Design
For this layer, TFTP is used in this project. As mentioned in section 3.2.5, it is a trivial
file transfer protocol used for boot system based on UDP. Another buffer is introduced in
between this layer and transport layer. A set of APIs about sending and receiving things
are given. Because a massive applications can be used above the application layer, they
can focus on their own works and simply transfer the data into the buffer to send them.
The rest things are hidden by the APIs.
Defined in RFC1350, the client sends read or write request to the host port number 69
(TID) firstly, then the host should allocate a free port as the temporary connection port.
There are slight differences between read and write request. When a read request is
issued, host will transfer the desired data as the acknowledge signal directly. An ac­
knowledge response is required for a writing request. The transmission ends when the
size of the last data packet is smaller than 512 Bytes. It will also be terminated when it
reaches its timeout. The process is shown in figure 5.4.
The timeout approach is an important part. If an ACK or a DATA frame is missing in a
certain time range, it should send back an error message, asking for retransmission. The
connection is closed after several failures. In the polling method, the signal missing cycles
are counted instead of using the real local clock.

7
Conclusion
A time predictable network stack extened from tpIP is built up. The original ethlib C
library is modified in a segmentation manner between different layers. The datalink layer
supports both SLIP and Ethernet. ARP and ICMP are integrated into the network layer.
And more functionalities, such as checksum calculation, are applied to support it. Then,
TFTP performs interaction between users and this network stack. A TFTP client is loaded
into the Patmos development board. The stack’s execution time is bounded running on a
time predictable platform, like Patmos.
7.1
Future Work
The ETHMAC IP can be updated to support higher speed. The structure is designed 20
years ago. The TFTP protocol can be used to download images into the FPGA board,
which is not tested yet. It can dramatically accelerate debug process with the hardware if
people can download huge images through Ethernet.
Until now, the layers uppon datalink layer are WCET analyzed. Datalink also needs to be
deterministic. TTEthernet [17] can be a choice. And TFTP is just one application protocol
within its family. More interesting protocols for real time systems can be applied according
to different requests.
