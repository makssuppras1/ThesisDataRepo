Abstract (English)
Personal informatics systems, is getting more accessible through smartphones. The
applications that currently exists, mostly focuses on one type of phenomena that can
be tracked with a limited, to non, option to add contextual data, which can add value
for self knowledge gain.
This is why this project investigates if it is possible, to make a system that allows a free
form registration, of any type of phenomena. Furthermore the project investigates
the possibility of making visualisation tools, that help the user reflect on the data
collected by actively using the contextual data.
Trough an iterative approach, a MVP has been developed through multiple versions,
each being tested before iterating. The MVP that have been developed and validated,
was used to verify if the system can be used for the previously mentioned purpose.
The results showed that the developed MVP, had potential for working as a personal
informatics system, allowing free form phenomena tracking, with the option to add
contextual data. Furthermore the results showed that the developed visualisation
tools, help the user to reflect on the data registered through the application, gaining
deeper self knowledge.

at de udviklede visualiseringsværktøjer kunne hjælpe en bruger til at reflektere over
den registrerede data og derved give en dybere viden om sig selv.

CHAPTER 1
Introduction
A person can learn a whole lot by themselves, with self tracking. In this day an age
it is getting even easier, as the technology that can help with self tracking is getting
more accessible. It is getting more common to use technology for keeping track of
various aspects of their lives. Common topics of interest for self tracking is produc-
tivity[Kim+16], finances[Kay+14] and learning[Riv+13].
It can also be seen, that since it is getting easier for the average individual to get
hands on technology for self tracking, the use of it is growing significantly. One topic
in particular is very popular in the self tracking field, namely self tracking regarding
health. Surveys showed that 41% of Canadians in 2018 [PLB18] did some sort of
health related self tracking.
As the world currently is in a global pandemic (COVID-19), studies have showed
that there are long-term health consequences of patients that have been discharged
of the hospital[Hua+21; Jen]. Here personal informatics systems can come to use. A
system where phenomena, symptoms and any other occurrences can be tracked, may
help to get a better understanding in the late sequelae that an individual is experienc-
ing, resulting in better and deeper self knowledge. Since not much research have been
made in this field, a system that can help as much as possible would be preferred. If
the system is given as much context as possible, it may help an individual understand
themselves better.
This is one of the reason, why this project was made. The project will try to develop
a system, which allows free form registration of any type of phenomena, with the abil-
ity to add contextual data. Further more the system will help the user, to interpret
these registrations by providing visualisation tools, that with the use of contextual
data, can help the user gain a deeper self knowledge.
The system developed in this project, will not be limited only for COVID related
phenomena. It will be as generic as possible, to support any type of phenomena. So
anyone, even an individual without a technical background, can use the system and
gain value form it.

1.1
Thesis Goal
There will be developed an Android application for a smartphone which can help a
person track any given phenomena.
Further more it is intended that the application can give insight in the phenomena
that is being tracked, even if you aren’t someone with an analytical, mathematical or
computer science background.
One of the main focus points in the development of the application, is that it should be
really fast and easy to register a given phenomena. Since the application uses active
tracking, it should be as seamless as possible. Another important part about this
application, is that it strives to be as generic as possible, resulting in no limitation
for what types of phenomena can be tracked. Further more it should be possible
to attach contextual data when registering a given phenomena, since that can give
greater insight and improve understanding why the registered phenomena looks as it
does. At last, as a goal for the application, it should be easy for any individual to
get a visualization and interpret of the data that have been registered, and thereby
get greater insight in the tracked phenomena.
1.2
Thesis Structure
The thesis is structured in the following way. Chapter 2 looks at other studies within
the same field, that have looked at similar topics, furthermore examples of mobile
applications within the field are discussed. Chapter 3 discusses the theory behind
personal informatics, together with what issues that can arise and how they can be
addressed. Chapter 4 describes how the design process of the application have been,
while describing how and why each component ended up looking like it did. Chapter
5 talks about the implementation decisions that have been made, how the application
is working and diﬀiculties that was discovered during the implementation. In Chapter
6 the varies test that have been conducted during the development process will be
evaluated, together with what kind of results that were produced. Chapter 7 will
discuss the overall result of the project as a whole, where Chapter 8 will discuss
possible future work. At last Chapter 9 will make a final conclusion look at the goals
that have been set for the beginning at the project and see to what degree they have
been reached.

2.1
Related studies
Experience Sampling Method (ESM), which is a research method that is used to
study how individuals feel and think during their every day life, is used a lot in stud-
ies in the past years. The method consists of making individuals doing systematic
self reporting, where these self reports is used to make a diary-like archive of their
daily experiences[LC14].
A similar approach, which focuses on eliminating the memory recall bias in retrospec-
tive self-reporting, also known as Ecological Momentary Assessment (EMA), also re-
sulted in interesting studies. Joshua M. Smyth et al. [SS03] made a study where they
questioned the way of collecting data in a laboratory setting or medical setting, com-
pared to a natural environment. It was here found that the EMA approach(reading
data in a natural environment) were much better suited. The results showed a signif-
icant difference between the ESM group compared to the natural environment group,
which showed how important the EMA approach was, to make a evaluation of physi-
ological and psycho social processes.
In health care and clinical research, a very popular patient assessment tool, is the use
of pen and paper, working as a diary. An interesting study made by Arthur A. Stone
et al.[Sto+03], who raised a concern, that patients did not comply withe the agreed
diary protocol, which could result in completely invalidating the diary data. Here an
experiment was made where patients with chronic pain, were assigned to either use a
paper diary or an electronic diary with enhanced compliance features. Further more

the patients were instructed to register three pain entries per day, at specific times
for a period of 21 days. The results of the studies showed, that the patients were not
compliant with the paper diary, but were compliant with the electronic diary with en-
hanced compliance features. The discoveries made, therefore raise questions whither
or not the paper diaries where the best approach for collecting diary information, sug-

A big focus of interest in this thesis, is to see how context can add greater insight in
whatever phenomena that an individual would like to track. In that relation Li et
al.[LDF12] identified systems, that gives physical activity awareness are emerging in
the market, systems that show information like step count, energy expenditure, heart
rate etc. which for many is enough for achieving self knowledge. But something that
was missing, is information about the factors that affect the physical activities, which
can lead to even deeper self reflections and more self knowledge. Li et al.[LDF12]
therefore explored the use of contextual information, such as events, places and peo-
ple, to enable further reflection regarding factors, that affected the physical activities.
The study led to three interesting findings.
1. Individuals make associations between physical activities and contextual infor-
mation that increased awareness of the factors that affect the physical activities.
2. When individuals reflect on the contextual information, combined with the
physical activity, they can gain greater awareness of the possibilities for physi-
cal activity.
3. At last the study showed that automated tracking of physical activity and con-
textual information, was great for long term reflection, but could potentially
have a negative effect on in-the-moment awareness.

2.3
This projects contributions
As seen in the previous sections, a lot of studies have been made in the self tracking
field. It is important to, in some way, take another approach so the project adds
value to the research field, instead of repeating what has already been done.
In this project an Android application will be developed that tries to make registration
as easy and quick as possible, while also giving the opportunity to make as free and
customizable registrations as possible, adding context as see fit by the user. The
project will try to make visualization tools, so the users can gain deeper self reflection
and self knowledge.
Many of the existing apps have an outdated look, which is why the app developed
in this project, will try to take a new and fresh approach to design through UX and
UI best practices and guidelines. The hope is to make an application that is ready
to be published to the Google Play store, so it can help as many as possible with self
tracking, gaining deep self reflection and self knowledge.

3.1
Personal Informatics
Personal informatics is systems that have the goals to help individuals improving
self knowledge. By improving self knowledge an individual can gain greater self in-
sight and can help promote positive behaviour resulting in a healthier lifestyle and
in general better well being. Since the technology in human-computing is growing
rapidly, new challenges and opportunities arises in different domains, regarding of
improvement of life quality. Domains such as fitness, nutrition, mental health and
sustainability.[Li+13].
Within the personal informatics field Li et al.[LDF10] derived a stage based model of
personal informatics, consisting of five stages. The model explains how such personal
informatics systems work, and what stages an individual goes through, when using
such a system. The visualisation of the model can be seen in Figure 3.1.
The five stages the user goes through when using such a personal informatics sys-
tems and its barriers, is as follows:
1. Preparation This stage happens before an individual event stats collecting
data. This stage is about the individual determining what data to track, how
to do it and what the motivation is. Barriers that can be noted at this stage, is
that an individual can potentially pick a tool that does not fulfill the information
needs. This can result in the person, needing to change tool, and if migration
of data is not supported, which it in many cases isn’t, the date can potentially
be lost.
==PAGE:22==

2. Collection Here it is when people are collecting the personal information of rel-
evance for the system. Any type of information can be used, it can be thoughts,
behaviour, interactions and other informations about their environment. It can
be informations that happens multiple times a day, a few times a week or even
more rare than that. Multiple problems that can arise during this stage. It can
be that the individual does not have time to input data, therefor not prioritiz-
ing it or simply just forgets it. It can also be that the data that is collected is
subjective, which therefore is not always constant.
3. Integration The integration stage is between the Collection and the Reflection
stage and is where the data is prepared, combined and transformed into some-
thing, that the user can reflect on. This stage can be long and time consuming
if the user have to do a lot of work themselves to prepare the date for reflection,
if the system does not do this automatically. The barrier that can occur in this
stage, is if it is not trivial to prepare the collected data for reflection, thereby
getting stuck and not continuing to the reflection stage.
4. Reflection In the Reflections stage, the user have to reflect on their data. Re-
flection can be looking at lists of the personal data or interpreting or interacting
with some kind of visualization of the data. The reflation can be done in a short
time after recording, which can be great since it leads to an awareness of a cur-
rent state. The reflection can also be done over a longer time span, which can
allow the user to reveal possible patterns that can be found. Here a barrier can
be that it is really time consuming or diﬀicult to make sense of the data, which
therefore makes it diﬀicult to gain any insight and self knowledge.
5. Action The final stage, the Action stage, is where the individual have to deter-
mine, what to do with the new gained self knowledge. It could be the case that
the reflection was about following a progress towards a goal. An action could
then be to change behaviour to accomplish this goal. Some systems even give
specific action suggestions, while other motivates the user to take some sort of
==PAGE:23==
3.1 Personal Informatics
9
action. Most systems do not give specific actions, which leads to the barrier in
this stage. It can be a problem, if the user do not know what to do, with the
new gained self knowledge. This problem can be eliminated in multiple ways.
As mentioned, if the system gives specific suggestions, that can help. Some try
to involve others, such as doctors, friends, family members or other kinds of
social forums, in the reflection stage to get help interpreting the data.
Ian Li et al. further presented four holistic properties, that was coupled with the
stages of the model. They will here be discussed together with their implications of
designing a personal informatics system.
• Cascading barriers A key property that is discovered in the stage based model,
is that problems that might arise in prior stages, can have an negative effect on
the following stages.
• Iteration Another property of personal informatics systems, is the fact the
stages are iterative. This means that the system needs to be flexible and support
a users changing data collection needs.
• User-driven vs. system-driven Each stage can be either user-driven, system-
driven or a combination. This tells wither the user or the system is responsible
for taking action. If the stage is user driven, it can be time consuming and
overwhelming, where as a system-driven stage is all handled by the system
itself.
• Uni-faceted vs. multi-faceted information An individual have many facets
in their lives, such as home life, work life, social life, physical and mental health
and so forth. Many systems only focus on one facet, uni-faceted systems, making
the collection and integration stage easier, but limits the awareness potential
in comparing data from different facets. On the other hand if systems enables
the tracking for multiple facets, multi-faceted systems, it can get diﬀicult in the
reflection stage if the system only provides, which is in most cases, separate
visualization.
The tool being develop in this project, tries to focus on flexibility in the collection
stage, so the users wont get restricted and can basically track whatever sees fit, al-
lowing the user to change data collection needs. The systems is not locked only to
one faced, allowing multi-faceted information to be collected. The collection stage is
combined user-driven and system-driven, but user-driven dominant, giving the user
the possibility to add as much context as possible. The integration stage is purely
system-driven, so the user should not run into any barriers related to transforming
the data and preparing it for reflection. This also makes sure, that iteration wont be
affected negatively in this stage. The reflection stage is somewhat combined user-
driven and system-driven, giving the user interactive visualisation tools that allows
reflection of short- & long-term reflection, making it easier to make sense of the data.

what should be visualized, and what should not. If the visualization is cluttered
with data, the user could have a diﬀicult time interpreting it, therefore it is of
great importance to ask, why is this data is present for visualisation.
3. How is it presented?
- Visual representation: At last it needs to be
determined how the data should be presented, which can be greatly determined
by the first two questions.

3.3
Prototyping
Prototyping allows to make draft versions of a system, extremely quick and cheap.
It allows testing and exploration of ideas, showing design concepts with minimal
recourse investment in development. It is therefore recommended to make these pro-
totypes as early as possible, in the whole development process, since a lot of feedback
can bet found by doing so. [Usa].
Nielsen[Nie03] also found that some of the biggest improvements where discovered,
user experience wise, by gathering usability data as early as possible with the use of
paper prototypes.
A common way of doing it, is by making the prototype as simple as possible, only
showing the core features of the system. Thereby possible wasted work will be min-
imized. The first step will then be, to make a low-fidelity prototype, which often
means it is paper-based consisting of hand drawn sketches. This way of prototyping
is the quickest way of testing early visualisation of alternative design solutions which
can help provoke innovation and improvement. Since this approach is so quick and
low on resources, test persons will often feel more comfortable suggesting change.
Thereafter the next step usually is making a high-fidelity prototype, which is computer-
based and usually looks very realistic and will allow the user to interact. This ap-
proach tries to come as close as possible, to the final product and can therefore be
effective for collecting human performance data, which could be how much time a
user spends to complete a certain task.
Prototyping can be really effective when done iteratively, by starting as simple as
possible (paper-based) and incrementally adding features as collecting feedback and
usability data see Figure 3.6. As you iterate, make a high-fidelity prototype that looks
as close as possible to the final design(computer-based). Another reason to start with
the low-fidelity prototype is, that if a person started making a realistic prototype and
spent a lot of time on it, they would be less likely to listen to feedback, if it would
require changes, since it would mean scraping the work that had been done. This will
not happen to the same extend, if the prototype would have been low-fidelity which
means it would have required much lower effort and resources.

5.3
Application architecture
To build a robust IT system, it is a must to use software architecture, which de-
fines how the system should be structured, to achieve a high quality. When build-
ing applications for android, Google recommended architecture is called Model-View-
ViewModel(MVVM)[And]. Further more Google have made a diagram, to illustrate
how the components of the application, should interact with each other, which can
be seen in Figure 5.2. The first layer activity/fragment, can be translated to the view
of the architecture, which is the layer that the user is interacting with. One layer
deeper is the View Model which is responsible for updating the view layer and handle
the input that the user give trough the view layer. Since the developer does not have
control of activities and fragments, e.g. if the are destroyed or recreated, the view
model is responsible to hold the data, that the view needs to display. This mean that
if an activity/fragment is recreated, the data can still be found by the view in the
view model.
The next layer in the diagram is the repository, which is a controller which is
responsible, for being a facilitate between the view model layer and the model layer,
while making sure that the data en the model layer is synchronized with the remote
data source(if there is one). In this project there is no remote data source, which
means that the responsibility of the repository, is to manage the local model layer.
Looking at Figure 5.3 it can bee seen, how the architecture have been applied for
this system. The diagram is not a complete reflection of all the files and classes the
system contains. Further more it can be seen, that the repository pattern applies, as

bar chart. In the bar chart, each bar height was then calculated, by first finding the
actual height of the y-axis in the skeleton view, thereafter using the max value of the
y-axis to calculate how tall the bar should be. This height was then calculated for
each bar, and displayed with an animation using an ValueAnimator16 animating its
height smooth from 0 to the calculated target height.

CHAPTER 7
Discussion
Many tools exists which can help and individual tracking a phenomena. These tools
are usually very specific for one type of phenomena, that should be tracked in a spe-
cific way. Tools that allow tracking of any type of imaginable event, with a complete
free way of constructing the registrations is not prevalent, which is why this project
tries developing such a system.
Trough multiple iterations of prototypes followed by tests, the registration process
have been validated to be user friendly and intuitive. Even when the feature was
tested by actually users in an MVP, where the response showed positive feedback
of the registration process and it capabilities. Even tho the results showed positive
results, room for improvement were still found. Smaller nice-to-have features could
be added, to improve the user experience, these will be discussed on the following
chapter.
A big focus of the developed system, was its visualisation tools. This is a big topic
and it should be mentioned that the topic of visualising data, could be a different
project on its own. The problems on that topic, is still faced in this project, but
to a lesser extend with only selected use cases. Through various validations of the
developed visualisations tools, it showed that using contextual data for visualisation,
is a challenge. The developed visualisation tools was only made with a few use cases
in mind. The developed calendar page showed great potential for detecting patterns,
and using the provided context as a possible explanation. The results was based on
dummy data, which could affect the outcome of the test, since the dummy data is
made with an agenda in mind. It might be that the tools will give a different result,
in a natural environment, since the users may not know, what contextual data would
be relevant. The developed visualisation tools is purely depending on, what the user
manually have been registering, which means that the visualisation tools have limited
knowledge.
Since this project had a deadline, planning have been crucial to reach a finished
product. In the beginning of the project, a Gantt chart was made(see Appendix A
Figure A.1). This plan was used as a guideline, during the project, to make sure that
there was enough time for everything that was planed, while the major milestones
was reached. To be able to evaluate on the plan that was made in the beginning
==PAGE:68==

of the project, a Gantt chart was made during the process, see Appendix A Figure
A.2. When comparing the pre-plan with the actual process, it can be seen, that they
align decently. It can even be seen that the actual process, went much more iterative
than the initial plan suggests. The iterative approach helped a great deal, focusing
the work on one specific feature at a time, which created a structured flow of the
project. Further more a daily journal was kept, which was functioning as a simplified
Kanban board1. This helped giving an overview of what was made and what was

discovered during the process, could be written down here so the would not be for-

continued. Each time some thing new was created, it was validated. This secured
that problems was addressed as quickly as possible. If the test and validations would
be addressed at the end of the project instead, discoveries that would be found could
show that the foundation was flawed, which could result in wasted resources.
Even tho the test and validations have been made iteratively, in hindsight the compar-

CHAPTER 8
Future Work
Since the application that is produced in this project, is a MVP and is mainly a proof
of concept, there is room for improvement. The project was on a deadline which
cause ideas and features to be prioritised differently. The improvement of this MVP
could go in multiple ways.
8.1
Release of a MVP
Since an MVP got implemented, it would be possible to release a production version
of the application to the Google Play store. This can give greater insight, in what
works and what does not, since it in this way will get out to more people in a natural
environment. No amount of testing can predict what type of behavior every type of
user will have, which is why releasing it to the public, will on one hand, see if the
same results will show as was discovered through the validations, and on the other
hand it is a way to discover what kind of changes would be liked by the mass.
8.2
Additional features
Multiple features was considered during the implementation phase, but since time
was a limited resource for this project, some features could not be implemented.
8.2.1
Parameters
The current MVP only allows 3 types of parameters, it could be considered to add
more types of parameters for different use cases. Furthermore it could be considered
to extend the current parameters. For instance the slider parameter always have a
scale form 1 to 5, but maybe 1 to 10 would be preferred, so adding the possibility of
customizing this would make the context more precise. For the number parameter
it could also be considered to add a unit to specify, if the number is related to e.g.
distance, time or weight. In Chapter 4 it was discussed how behaviour should be,
for adding parameters. It was attempted design a suggestion function, for quicker
parameter addition. This was not solved and the behaviour used, ended up being in
the simplest form. It could be a considered in future work, to take this parameter
==PAGE:70==
56
8 Future Work
suggestion to consideration again, since it will improve the speed in which the user
can register events when also adding parameters.
8.2.2
Data Visualisation
A feature that was requested during the data visualisation validations and also was
considered but not priorities in development, was the possibility of filtering what
events should be showed in the different visualisation tools. If an individual is track-
ing many different types of events, at the same time, the visualisation tools can get
very cluttered with data, which can lead to more confusion than clarification. The
filtering of data would be relevant for the calendar view and the statistics view.
Further more it could be considered to add more interesting values, than the cur-
rent showed in the statistics page in the stat average section. The current values does
not add a whole lot, that is not already accessible through the other tools.
As mentioned before the topic of data visualisation could be a project on its own.
It could therefore be considered, to add more sophisticated data visualisation with
the use of machine learning. This will help the user even further, to understand corre-
lations and maybe even suggestions to what actions to take to lower a phenomena(if
lowering it is an improvement).
8.2.3
Passive tracking
As mentioned previously, this application mainly uses active tracking, with a limited
amount of passive tracking. It could be considered, to add passive tracking as param-
eters in the event registration. This can lead to gaining deeper self knowledge, since
data now will be registered, which the user would not have thought could have any
influence, but might add valuable context to a given phenomena. Since the average
smartphone have multiple sensors build in, it would be a start to utilize that. For
instance utilize that smartphones have a GPS and can thereby save the location with
a given registration.
Since people spend a lot of time on their smartphone, time
spend on the phone or in different apps could be relevant for different phenomena
occurrences.
8.2.4
Smart watch
To continue on the topic of passive tracking, it could be considered to make an
integration with smart watches.
Most smart watches these days contain multiple
sensors which can passively tracking many different types of data, which can give
further context. Examples of types of passive tracking that could be integrated could
be; physical activity, sleep, blood pressure. It can here be interesting to see if the
actively tracked events, have some sort of correlation with any passively tracked data.

8.2.5
Widgets
To further increase the speed and ease of registering events, it could be considered to
make widgets, that can be displayed in the home screen of the phone. This feature
is seen in TapLog, which is discovered in Chapter 2, where the registration of events,
can be made from the home screen without ever needing to open the app it self.
Furthermore widgets that show data visualisation is used, which can give the user a
better overview, of the registered data, since it in this way will be presented to the
user much more often.
8.2.6
Import/export data
The possibility of importing or exporting data from the application, is in the current
state of the MVP not possible. It could be a possibility, that a user wants to take
data from one system, into this system or the other way around. By enabling import/-
export of data it will make sure, that the user does not lose data, if moving from one
system to another. It could also be that further analysis wants to be made, outside
of the system, where exporting the data would be required, otherwise it would had
to be done manually which most people would not botherdoing.

CHAPTER 9
Conclusion
This thesis have two main questions to investigate. Is it possible to make a free form
data collation application for self tracking, which allows to add context to the reg-
istrations. Furthermore is it possible to make visualisation tools, that can help the
user reflect on the collected data, by using the context that have been added. To
answer these questions, it have been investigated what types of solution existed al-
ready, within the same field. It was here discovered that there was hole in the market
for such system. Thereafter the field of personal informatics and data visualisation
techniques was analysed, to understand what would be required, to create such a
system.
An application have been designed, which have gone trough a multitude of itera-
tions and validations. Hereby arriving with an application, focusing on achieving
the specified goals with a user friendly and intuitive user interface, by applying the
analysed methods. Furthermore it have been discussed, that certain features had to
be omitted hence the project had a deadline. Even tho the project have a working
application, it is an MVP after all, which is meant as a proof of concept. The product
still have possibility to be expanded upon, for achieving an application that solves
the problems addressed to an even greater extent.
As the tests showed, users of the system were possible to make registrations with
the free form approach, adding contextual data to registrations. Furthermore it was
discovered that the designed visualisation tools could be used for achieving self knowl-
edge, by using the context, which concludes that the project turned out as a success
since, the goals set in the beginning of the project has been reached.

experiment was conducted. Two test users where found and they got the following
instructions:
Goal of test:
• Test if the registration progress is user friendly
• Test if registration process is intuitive
• Detect bugs
• Discover ideas for improvement of user friendliness
Test user instructions:
I am in the process of developing an app that is supposed to help you track a
given phenomena/pattern/habit that you are interested in learning more about. An
example could be that you are smoking, and would like to know more about your
smoking pattern, when, where and how much you are smoking in your everyday life,
but it could just as well be anything else.
The goal of the app is to give greater insight in the given phenomena/pattern/habit
and thereby improve well being.
The current state of the app only allows the user to input data.
The purpose
of this test is to see if the process of inputting data in the app is user friendly and
intuitive.
The test will consist of you selecting one or two phenomena/patterns/habits that
happen on average at least once a day and track it with the app.
Monday 19/4 the test will end. You will be given a questionnaire that will ask about
your experience using the app.

APPENDIX R
Epic 2 User Experiment
- Final MVP product
At the end of the development phase, an user experiment was conducted. Two test
users where found and they got the following instructions:
Goal of test:
• Test usability of visualisation components, if they can be used to gain insight
in the registered events.
• Detect bugs.
• Discover ideas for improvement of user friendliness.
Test user instructions: I am in the process of developing an app that is supposed
to help you track a given phenomena/pattern/habit that you are interested in
learning more about. An example could be that you are smoking, and would like
to know more about your smoking pattern, when, where and how much you are
smoking in your everyday life, but it could just as well be anything else.
The
goal of the app is to give greater insight in the given phenomena/pattern/habit
and thereby improve well being. The test will consist of you selecting at least two
phenomena/patterns/habits that happen on average at once every other day and
track it with the app. It would be optimal if these events that you are tracking had
the possibility of having some sort of coherency. Tuesday 8/6 the test will end. You
will be given a questionnaire that will ask about your experience using the app.
Disclaimer:
This is a MVP(Minimal viable product) which means that errors may occur during
usage. If you encounter things that don’t make sense to you or if something just
doesn’t work, write it down somewhere so you won’t forget it. It can be used in the
questionnaire you will receive at the end of the test and that kind of information will
be really valuable.
Security:

all kept locally on your phone. The only thing the app will use internet connection

I can see what happened and fix the error so it won’t happen for the next version.
