ABSTRACT 
This master thesis aims to propose solve the problem of searching for library through a hybrid 
recommendation approach by content-based and knowledge-based recommenders. As such the system 
takes a textual description of a project as input and generates a set of libraries that cover all 
requirements specified. The idea behind creating one recommendation comes from keyword 
extraction methods and the set cover approximation problem. 
In this thesis, literature about techniques for building recommender systems, keyword extraction 
algorithms and set cover approximation algorithms will be surveyed. Based on these, a recommender 
system will be designed and implemented, and the process of this will be documented in the following 
thesis.

1. Introduction 
Nowadays the complexity of software engineering projects has reached a larger scale than ever. As 
such, the effort and tasks necessary to design and implement such a project have also increased in 
number and diversity. Completing such a project is by no means as simple as knowing a programming 
language and writing code that fulfils the given task.  
Developers must undertake a variety of activities in order to make a software project come to fruition. 
Even a small software project now involves the use of multiple programming languages, libraries and 
frameworks, but making the decision on which of these to use is a task that involves a lot of effort. 
For that a developer should resort to searching for the possible resources to use, choosing which are 
most appropriate for the task at hand, deciding if they are compatible, learning about the methods of 
use and installation for each etc.  
Professional software development has long evolved from just deciding on a project and its 
requirements and starting to program. Different software development methodologies have been 
created, built upon and evolved, (some of them have died out) all with the purpose of making 
professional software projects more efficient and manageable (1). What most of these methodologies 
have in common is that they divide the lifecycle of a software project into different phases, and as one 
might expect the actual implementation phase is not the first phase in the cycle.  
Before the actual development stage can begin there is often a need for planning, both for specifying 
requirements or functionality and for making technical decisions that will affect the whole project. 
This planning stage, or more specifically the system design stage, is the most relevant to the contents 
of this master thesis. This is when preliminary work that development will be based on happens:  in 
this phase, decisions about the architecture, design as well as technical details, should be taken. At the 
end of the design process, a developer should have a feasible plan for the software, including what it 
does, how it does it and how it looks (2).  
This is also the point in this process, where we should be considering which languages, APIs and 
frameworks will be used in the development stage. The recommender system I am proposing aims to 
help with making this decision.  The system will take as input a description of the project that is to be 
developed. Based on this description and a knowledge base containing information about different 
libraries/ APIs, the system will output a list of resources covering (most of) the requirements extracted 
from the given project description.  
The goal of this master thesis is to design and implement such a recommender system for software 
engineering that should fit into the pre-development stage of a software project. This system will take 
as input the description of a software project and give as output a list of libraries / APIs that can be 
used to develop the project.

2. State of the art 
The goal of this master thesis is to design and implement a targeted recommendation algorithm that 
can generate sets of libraries, frameworks and other coding resources to help increase efficiency of the 
design and implementation phases of software projects. As such, we will first present some general 
information on types of recommender systems and the logic behind them. 
2.1 Recommender Systems 
Recommender systems are at their core information filtering systems that can be used to lessen the 
damage of the information overload problem we face nowadays. Given a large set of items, their 
purpose is to predict which items the user would be interested in. (3)   
Recommender systems are especially relevant nowadays, as the amount of information available to 
users keeps increasing. As mentioned before, this is also a significant problem in the field of software 
engineering because the information overload problem can bring about a large overhead in the design 
and implementation part of a software project. 
The paper (4) present an overview of different types of recommendation algorithms, the techniques 
used, as well as methods to evaluate such systems. 
 
Figure 1 Recommendation phases taken from (4) 
The above figure presents the three main phases involved in recommending an item to a user 
according to the paper mentioned before. 
As observed, the first phase is the Information collection phase, during which a recommender system 
gathers information on the user in order to build a profile. It is possible to gather information in two 
main ways: explicitly or implicitly, however a combination of the two methods can be used for best 
results. Gathering information explicitly is done by requesting the user to provide information through 
the UI, using a variety of input methods, while the other method relies on monitoring and analysing 
the user’s activity and interaction with the system, through actions such as web searches, button clicks 
etc. 
In the Learning phase, the system processes and structures the data gathered previously, in order to 
use it as efficiently as possible. 
The third phase, and the most interesting one for this paper, is the Recommendation Phase, where a 
recommendation algorithm is used to predict which items the user will be interested in. The different 
types of algorithms that can be used in this phase will be presented in the next sub-section. 
==PAGE:10==
9 
Jannach et al. divide recommendation algorithms into three main types: collaborative 
recommendation, content-based recommendation and knowledge-based recommendation (5). These 
will be further explained and explored in the next subsections. 
2.1.1 Collaborative Recommendation 
Collaborative recommender algorithms leverage information on the community of the user, and 
determine similarities between users, based on their purchase history for example, in order to 
recommend items. 
There are two main approaches used in implementing collaborative recommendation algorithms: user-
based filtering and item-based filtering.  
In user-based filtering, the system finds users which are similar to the target user, thus building what 
is called a neighbourhood of users. Similarity between two users is computed based on the users’ 
previous ratings through two main computation methods: cosine similarity and Pearson Correlation. 
After a neighbourhood is obtained, the system is ready to make recommendations by computing a 
predicted rating for those items that have good ratings in the target user’s neighbourhood, but the 
target has not rated (or purchased) yet. 
Item-based approaches work similarly to the previous case, but the neighbourhood built consists of 
items instead of users. 
Collaborative recommendation systems have been researched extensively, and put into application in 
many domains. This does not mean however that they are without disadvantages. On the one hand, a 
major issue appears when a new user or item is added to the system: accurate recommendations 
depend entirely on previous history, so they cannot be made until the user (or item) has interacted 
with the system. On the other hand, when the user and item database increase greatly, as is the case 
with most e-commerce systems, the runtime and efficiency of a traditional collaborative filtering 
algorithm is greatly impacted. 
2.1.2 Content-based Recommendation 
Content-based recommender systems have been explored, described and evaluated extensively in 
literature such as (6) (7). Based on these, this section will present some basic knowledge about how 
they operate and their advantages. 
Content-based recommenders take advantage of the features or attributes of the items being 
recommended, in order to compute similarity between items. In this type of system, the user will be 
given recommendations based on whether the candidate item is similar to other items the user has 
interacted positively with before.  
Content-based recommendations usually rely on two sources of data. First of all, a description, which 
can be textual or include other attributes, of various items is needed. Second, the system can depend 
on a user profile, obtained by explicit or implicit user feedback. 
It is possible to divide most content-based recommenders into components based on their function, 
giving a better picture of their operation, as described by Aggarwal in (6). 
The first component is responsible for pre-processing and feature extraction. Because content-based 
systems are applied to a variety of domains and item-types, the item’s description can be highly 
unstructured and varied, sometimes including different sources of information. As such, a content-
==PAGE:11==
10 
based system first needs to process this input and extract relevant features or attributes in order to 
function properly. 
The second component can involve learning of user profiles, where a user’s interest can be linked to 
different item features. Most times, this is done by depending on the user’s feedback in the form of 
ratings or purchases. This user’s history is used along with the item features extracted previously, to 
serve as training data for the construction of a learning model. Depending on the type of feedback 
used, this phase can be likened to a task of classification or regression modelling.   
The last component is responsible for making the actual recommendation by using the model obtained 
from the previous step. Here items that match the user’s profile are found and recommended. 
Complementary to this, the authors from (7) describe some methods of representing the content of an 
item, computing similarity between items and retrieving recommendations based on these. 
As mentioned before, oftentimes the items in this type of recommenders have some sort of textual 
description which can be processed and leveraged when computing similarity between items. To this 
purpose, the TF-IDF statistic measure along with the vector space model has been used quite 
extensively.  
Term Frequency Inverse Document Frequency is a measure used to compute the relevancy of a term 
in a document, relative to a corpus (set of documents) it is included in. More details on this measure 
and its computation methods will be presented in section 3.1.1. It is important to note that, by using 
TF-IDF, a document can be represented in the vector space model as a vector of weights, of size N, 
where N is the number of terms identified in the whole corpus. It follows that in order to compute the 
similarity between two documents, one can apply a method such as cosine similarity on the TF-IDF 
vectors of the two documents. 
2.1.3 Knowledge Based Recommendation 
Knowledge-based recommender systems are usually applied to cases where neither collaborative nor 
content filtering perform well. This can happen in domains where the items recommended are 
particularly complex, or where item acquisition happens at large time intervals (such as buying a car), 
such that building a user profile based on ratings is difficult.  
This type of system relies heavily on explicit information about the items, specifications provided by 
the user or even explicitly defined recommendation rules.   
Jannach et al. present two basic types of knowledge-based recommendation systems in  (5):constraint-
based systems (8) and case-based systems (9), depending on the input that is provided by the user. 
2.1.4 Hybrid Recommendation 
Hybrid recommender systems aim to combine different types of recommendations in order to resolve 
the disadvantages brought by a single approach, and bring about better results.  
There can be multiple ways of creating a hybrid recommendation system depending on the goals a 
developer aspires to achieve. One such way is to implement two different recommendation systems

3. SolRec: a hybrid recommender system for software engineering 
As mentioned before, through this master thesis, a recommender system for software engineering will 
be proposed, based on the principles of content-based and knowledge-based systems.  
The item-domain of this recommender system will consist of programming libraries, APIs or other 
programming tools, and will be referred to as resources from now on.  
One recommendation will be referred to as a solution and will consist of a set of resources, which can 
be used together to fulfil the requirements specified by the user. 
From now on the term resource will be used to describe libraries, modules, APIs and any other 
programming tools that are included in the output of the system. 
Goal: make suggestions of resources to be used by a developer when planning a project. 
Input: project description, either textual or as list of keywords. 
Steps: 
1. Preliminary: create knowledge base about existing resources, apply a keyword extraction 
algorithm on these. 
2. Create user profile: preferred programming languages, level of experience in software 
development etc. 
3. Input project description and extract keywords.  
4. Generate multiple solutions and display only x solutions that have the best score. 
5. Gather feedback from user.

The first step described in the previous algorithm corresponds to the information gathering phase of a

The second step, involves creating a user profile, in order to better understand the user, his 
preferences and interests as well as experience level. 
The third step requires the developer to input a project description, either as textual information or as

4. Building the knowledge-base 
This section will focus on the first component needed for SolRec presented in section 3. We will start 
by giving an introduction on the task of keyword extraction, then present some existing methods for 
implementation and evaluation of this task. Finally, the whole process for building the knowledge 
base will be described in detail. 
4.1 Introduction to keyword extraction 
A ‘key’ element of a document is an element which is a good descriptor of the document’s content. 
The difference between a key-phrase and a key-word comes from whether the key element contains 
one or multiple lexical units. 
 A keyword extraction algorithm takes as input a document and processes it, in order to retrieve these 
‘key’ elements from it. The purpose of such an algorithm is to obtain a list of lexical units which can 
describe the content of the document. This category of algorithms has been studied extensively 
throughout recent years, as the internet is flooded with more and more information and documents. 
Multiple methods for keyword extraction, both supervised and unsupervised have been developed and 
tested on a variety of input data, such as scientific articles, abstracts, web pages or event short texts 
(twitter posts for example). Each of them has their own advantages and disadvantages, and it can be 
said that no particular approach has turned out dominant so far []. Generally, it is best to decide on the 
approach used, on a case-by-case basis depending on the form of input data, the purpose of the 
algorithm and other factors. 
Many natural language applications involve the task of extracting “key” elements (words or phrases) 
from unstructured textual data. These extracted key elements can be used in different ways in order to 
gain value from this task. Keywords can have value as metadata that enriches documents, giving an 
explicit and computer-processable description of the text content, be used in content management 
tasks (browsing, indexing, topic detection, classification) and semantic-aware applications 
(exploratory search, recommendation, reputation analysis, or contextual advertising). In another case 
they can be used in order to automatically summarize the contents of a document. 
There are a few different ways of categorizing keyword extraction algorithms: 
The first one classifies the methods as either supervised or unsupervised.  
The idea behind supervised keyword extraction algorithms is that this task can be thought of as a 
classification task: lexical units from the text are classified as either keywords or non-keywords. Such 
algorithms require training data and rely on training features for training a classifier. The lack of 
training data is precisely what can become a disadvantage for supervised algorithms. 
Unsupervised algorithms on the other hand mostly see the keyword extraction algorithm as a ranking 
problem. Different methods can be used to compute scores for lexical units and rank them according 
to those scores. 
 Some algorithms used basic statistical methods for calculating the scores and extracting keywords. 
An advantage of these is that they are mainly language and domain independent. Such a method is 
TFIDF, which has been used and adapted by many since the Theory of Term Importance was 
proposed by x in [15]. Another method is based on N-Grams hypothesis which proposes that any 
sequence of units of any length could be a ‘key’ element. Another unsupervised algorithm is YAKE, 
which is based on statistical features extracted from a single document. YAKE and TF-IDF will both 
be explored in greater detail in the upcoming sections. 
==PAGE:17==
16 
Other unsupervised algorithms represent the document as a graph and then compute a ranking for the 
nodes based on the graph information. Such an algorithm is TextRank which will be tackled more in 
the next section. 
4.2 Existing Methods 
In this section multiple existing algorithms for keyword extraction will be surveyed and tested in 
order to choose the best option for our system.  
4.2.1 TF-IDF 
TF-IDF (term frequency-inverse document frequency) is a weighting statistic used to determine 
whether a word is relevant to a document within a larger corpus. For example, the corpus could 
include a set of articles in computer science, while the document is a specific article from that set. 
This calculation method is based on the idea that a term’s relevance is not only based on its frequency 
in a document, but also on whether the term is occurring often within the whole corpus or is specific 
to the document [5].  
As such, the TF-IDF value is calculated based on two statistics: the term frequency and the inverse 
document frequency.   
The term frequency (TF) is the number of times a term occurs in a specific document. The size of a 
document within the corpus might vary greatly, which is why the numbers obtained should also be 
divided by the length (number of terms) of the document.

4.2.2 YAKE! 
The authors of paper (15) proposed YAKE! as an algorithm for unsupervised keyword extraction 
based on statistical features extracted from a single document. This method does not depend on an 
external corpus of documents, nor does it need training data as do supervised approaches.  
YAKE! performs a series of actions in order to extract keywords. 
First the input text is cleaned and pre-processed resulting in the output of a list of candidate terms. 
Next, this list of words is represented using on a number of statistical features based on the casing of 
words, position within the document and relevance to context. These features are also part of the 
calculation of scores reflecting the importance of each term. Furthermore, keyword candidates with 
multiple words are generated, and scores for each of these candidates are calculated. Finally, similar 
keyword candidates are found and compared, in an attempt to see whether removing some of them 
could improve the results.

4.2.3 TextRank 
TextRank proposed in (16) is an unsupervised graph-based ranking algorithm used in natural language 
applications such as keyword and sentence extraction. The idea proposed in the paper relies on 
representing input text as a graph where each node is a word and the edges are constructed by 
observing the co-occurrence of words inside a moving window of predefined size (16). Important 
nodes of the graph, computed with an algorithm similar to Google’s PageRank, represent keywords in 
the text. 
It is up to the developer to decide how to represent the text as a graph in order to extract keywords. As 
such there are some basic steps that need to be met when designing and implementing such an 
algorithm: 
• 
Decide the text units that should be added as vertices in the graph for the task. 
• 
Find the relations between them and add them to the graph. 
• 
Calculate the scores for each vertex using the graph-based ranking algorithm. 
• 
Sort the vertices based on their final scores. 
Graph-based ranking algorithms decide the importance of a node, based on information about the 
entire graph. Such an algorithm works based on the idea of “voting” or “recommendation” as 
described in the paper by Mihalcea. The connections between nodes represent these so-called “votes”, 
so that the more votes or connections a node has the more important it is considered. 
 Furthermore, the weight of a vote is also influenced by the node casting it. Let’s say there are two 
nodes N1, N2 which are both connected to node N3 and N1 has a higher score than N2. When 
calculating the score of N3, the node N1 will have a greater impact than the node N2. 
Another factor taken into consideration by the TextRank algorithm is the “strength” of the connection 
between two nodes, which can be based on how many times the two words occur together in a 
sentence multiple times. 
For the purpose of this paper, the python library spacy was used to implement the TextRank algorithm 
and use it in the following stages for comparison to the chosen algorithm.

5.  Solution generation 
The basic idea of the solution generation algorithm is derived from the set cover approximation 
problem. In the following section, I will provide a brief definition of this problem and the greedy 
algorithm for solving it, as well as the way this has influenced the algorithm proposed and its design. 
5.1 The Set-Cover Problem 
Given a collection S of sets over a universe U, a set cover C is a sub-collection of the sets whose 
union is U. The set-cover problem is to find the “best” set-cover, given S.  
Depending on which sub-type of set-cover problem we would like to solve, the definition of the best 
set-cover can vary: it can be the set-cover with the minimum-cardinality in the basic case, or with the 
minimum total weight in the case of the weighted set cover problem. 
Greedy algorithms follow the problem-solving heuristic of making the locally optimal choice 
available at each stage, without considering the future consequences of this. Applied to set cover 
problem, the Greedy algorithm works by making a series of decisions, choosing sets one at a time, 
until valid solution is reached (21). At each step, the algorithm chooses the set with the best 
(minimum) cost-benefit ratio at that point in time. This ratio would be the value of the cost of a set (its 
weight) divided by the value of the profit brought by choosing that set (the number of elements it 
covers). 
In the textbox above, we can see a simple algorithm for solving the approximation set-cover problem 
using the Greedy heuristic.

5.2.6 GRASP metaheuristic 
The GRASP metaheuristic is a random iterative optimization procedure. This metaheuristic has been 
used to solve diverse problems of optimization, including scheduling, route design, graphs, transport, 
and telecommunications problems, among others. In paper (22), the authors propose a method for 
implementing this method to solve the uni-cost set cover problem.  
GRASP metaheuristic presents two basic phases: a constructive phase and a local search phase. In the 
first phase, a modified randomized greedy heuristic is used with the purpose of obtaining an initial 
solution.   
In the next phase, local search is used to explore the neighbourhood of the generated solution with the 
purpose of finding higher-quality solutions. After a local search iteration, newly constructed solution 
is compared to the best solution obtained previously, to decide which one to move forward with. A 
stopping criteria must be defined to decide when the local search stops and the algorithm returns the 
best found solution. 
In order to use the GRASP procedure to solve our set-cover approximation problem, we need to 
define the following elements integrated in the heuristics: 
1. the randomized constructive method used during this procedure 
2. the neighbourhood of the solution and the procedure to investigate it 
3. the stopping criterion

5.3 Alternative algorithm - MetaRaps 
 
MetaRaps is a meta heuristic for solving approximation problems which has been applied to multiple 
types of problems. In paper (23), the authors try implementing this heuristic to weighted and non-
weighted set covering problems. 
The basic idea behind this concept is introducing randomization in the solution construction phase of 
SCP in order to avoid getting stuck in a local optimum, as happens with GRASP. The MetaRaps 
algorithm constructs a solution by adding a feasible element at each step. As with greedy, each 
element defines a greedy score or cost-effectiveness. At each step of the construction phase, the 
element with the highest greedy score is either chosen as the best feasible element or the worst. 
 This method introduces two parameters: priority and restriction, in order to add randomness to the 
algorithm. As such with a probability of %priority, we either choose the best element or we choose 
some random element from the list of feasible elements. The other parameter %restriction defines the 
interval in which the score of an element should be, in order for it to be considered feasible. The 
smaller the %priority parameter is and the higher the %restriction is, the more randomness is 
introduced.  
 
Besides the construction phase the MetaRaps also defines an improvement phase usually based on 
neighbourhood search. 
In this case a solution X is considered to be in the neighbourhood of another solution Y if their 
intersection contains at least one element.  
The MetaRaps improvement heuristic will use neighbourhood search by removing a number K of 
elements and then solving the partial set cover problem thus obtained. It also defines an 
%improvement parameter to decide whether a given unimproved solution moves into the

As explained in section 5.2.1, the basic building blocks of our solution generation algorithm are as

1. Construction phase

The method RunAlgIteratively is what ties all these steps together by constructing an initial

There are three different child classes that extend the class SetCoverBase. They correspond to 
the different algorithm variations that were described in sections 5.2.3, 5.2.6 and 5.2.7. 
Each of these classes defines a method FindMatch for finding the next element at a certain step 
during the construction phase. This method is implemented differently depending on the strategy 
adopted by the class.  
GreedySolGen is a class that defines a method for generating solutions based on the classic greedy 
algorithm for set cover problem. For this case, construction phase uses a greedy algorithm that aims to 
maximize the score of the resource chosen at any step. 
The method FindMatch mentioned thus first calculates the score for every candidate resource and then 
chooses the resource with the highest score. The algorithm terminates when all target keywords have 
been covered.  
A difference from the traditional greedy implementation is that if there are still keywords left but none 
of the remaining resources covers them, the algorithm backtracks by removing one of the resources 
chosen so far and then choosing the next-best option.  
GRASPSolGen defines a method for constructing solutions based on the metaheuristic described in 
section 5.2.6. The method FindMatchGRASP mentioned first calculates the score for every candidate 
resource, creates a restricted candidate list with the top K best resources and then randomly chooses 
the resource from the restricted candidate list. 
MRSolGen defines a method for constructing solutions based on the metaheuristic described in 
section 5.2.7. This class has two parameters (priority, restriction) that are used when constructing 
solutions in order to introduce randomization in the construction process. 
The method FindMatchMR mentioned first calculates the score for every candidate resource, then 
decides with a probability of priority whether to choose the best resource or not. If not, it goes on to 
creates a restricted candidate list with the resources that have scores over an accepted threshold 
(computed based on the restriction parameter) and then randomly chooses the resource from the 
restricted candidate list.

7. Solution evaluation 
In order to make it possible to evaluate and improve the solutions obtained from the software resource 
recommendation algorithm, a quality score will be defined and calculated. As designing and 
developing a recommendation system for software engineering is a complex task, in a novel and 
rapidly changing domain, defining an objective, generally accepted quality score would be difficult. 
This is why the quality score defined will be based on accepted software engineering principles, 
literature found on the subject of software library and APIs quality, as well as intuition formed 
through my own experience as a software developer. 
In the paper (32), the authors mined information about hundreds of APIs throughout a period of a 
year, in order to observe changes in the popularity of APIs and deduce the causes behind these 
changes. The authors, processed the source code of these APIs, but also the usage of certain API 
elements (classes or objects for example) by analysing their presence in the import statements of java 
projects. The premise behind this is that an APIs popularity can be indicative of its quality.  
In another paper (33), the authors propose a set of metrics that can be used for comparing software 
libraries, and carry out a survey in order to verify which of these metrics are considered important by 
developers. Among the metrics presented and investigated here the ones with the highest influence on 
most developers according to the survey were: 
• 
Popularity: the number of projects using a given library.  
• 
Performance: score indicating how efficient the code is.  
• 
Security: score indicating whether the library has any vulnerabilities. 
The following subsections will be structured as described. First of all, the quality score and all its 
components will be defined. Next, certain aspects and variants on the solution generation algorithm 
will be investigated according to the quality score. 
7.1 Defining the quality score 
The following measures will take part in the final quality score and they will be explained and defined 
in detail in the following subsection. 
• 
The covering score based on TF-IDF scores of keywords from the target and the found 
resources. 
• 
Popularity score based on the number of downloads that a resource has.  
• 
The total size in KB of the found solution. 
• 
The number of resources that are included in the solution.

8. Conclusion 
As noted through the literature and development done in recommender systems for software 
engineering, nowadays there is a great need for a more efficient way of choosing programming 
libraries to use along with software projects. 
The problem of overabundance of information has become more and more severe with the growth of 
the internet, and recommender systems have strived to lessen the burden in many affected domains, 
especially in domains like e-commerce, entertainment (for example Netflix and Spotify), learning 
analytics, but also in IT.  
In the field of IT, advancements are made at a very rapid rate and new APIs, frameworks and other 
tools appear constantly. The growth of the use of open source software, can be seen from articles such 
as (34), which documents how the percentage of companies that use OSS has nearly doubled from 
2010 to 2015. Another interesting view can be obtained by looking at statistics on Github, a well-
known hosting server for code repositories. Looking at (35) and (36), we can see that the number of 
repositories on Github has grown from approximately 6 millions in 2013, to 21 millions in 2015 and 
reaching the number of over 128 millions in 2020. A lot of those are open-source (or not) projects and 
APIs, that a developer can use to facilitate writing a new software projects. Filtering through this 
enormous amount of resources however is no easy feat, which is what my proposed system aims to 
solve. For the case of IT and software development the problem of finding appropriate libraries can 
take up a lot of time and effort, for any developer whether experienced or beginner. 
The system proposed aims to solve the problem of searching for library, through a recommendation 
approach inspired by content-based and knowledge-based recommenders. As such the system takes a 
textual description of a project as input and generates a set of libraries that cover all requirements 
specified. The idea behind creating one recommendation comes from keyword extraction methods and 
the set cover approximation problem. 
Various methods based on heuristics for the set cover problem have been tried for generating 
solutions covering all requirements specified by the user. A quality score has also been defined to help 
evaluate the solutions, based on findings in literature on the topic, as well as on software development 
principles and intuition.  
Each of the approaches has been evaluated accordingly and found to have their own advantages and 
disadvantages. Combining greedy heuristic with local search has been found to be a good option in 
term of quality score and runtime, however its disadvantage is that the solutions obtained through it 
are relatively similar, which can be detrimental to the output of a recommender system. Adding 
randomization by using GRASP to explore a wider range of solutions is thus a good idea, and 
provides some much needed variety. The alternative algorithm explored based on the MetaRaps 
heuristic is also a good option from this point of view, however it would be best used for more 
complex input projects as it performs better when the solution generated has a larger number of 
resources.  
The domain of recommendation systems for software engineering is one that has seen a lot of work in 
recent years and will see even more to come, however it is not one where it is an easy task to define a 
objective way of evaluating quality of solutions. The item domain itself, specifically that of software 
libraries, frameworks and tools, is particularly complex on its own, and even more so when combined 
with the subjectivity and variables that come from the wide range of users and their needs. Because of 
these reasons, it would be best to use solutions generated by multiple approaches as 
recommendations, in order to give the user a variety of options. 
After the work done on the thesis and the methods implemented and evaluated, I believe it is feasible 
and desirable to create such a recommender system to aid with the development of a specific software 
==PAGE:44==
43 
project, based on the principles of content and knowledge-based recommender systems, keyword 
extraction algorithms, set-cover approximation problem.  
However, the current version of the system defined is far from perfect which is why there is much 
future work that could be done to improve it. For example, the preliminary phase of building the 
knowledge base could be improved by adapting the keyword extraction algorithm better to the 
software domain and filtering the input data further to obtain more relevant descriptions of the 
libraries. Furthermore, the recommender system itself could be improved by taking more clearly 
defined user input, such as the programming languages preferred or the prioritized attributes of the 
resource (such as size, performance etc) alongside the textual description.

Appendix 
A Original Project Description  
 
Problem definition: The goal of a recommender system is to generate meaningful recommendations 
to a collection of users for items or products that might interest them. There are multiple major types 
of recommender systems: content-based, collaborative, knowledge-based, all with their advantages 
and disadvantages. A hybrid recommender system aims to combine two or more types of 
recommender systems to obtain a better solution. 
The recommender system proposed is targeted at software developers and will take as input a 
description of the project that is to be developed. Based on this description and a knowledge base 
containing information about different libraries/ APIs, the system will output a list of resources 
covering (most of) the requirements extracted from the given project description.  
In order to achieve this goal two major steps are identified: a requirement (in the form of keywords) 
extraction step and a solution generation step, in which set cover heuristics will be used to cover the 
extracted keywords. 
Previous work: 
In [1], we can see an overview of what recommendation systems for software engineering are, how 
they work, what they recommend and some examples of such systems. CodeBroker [2] is such a 
recommender system that analyses the text written in comments by developers and aims to 
recommend class library elements to help the developer achieve their goal. 
In [3], we see a description of CrossRec, a recommender system that aims to help developers by 
recommending third party libraries. CrossRec uses a collaborative filtering technique to recommend 
libraries to developers by relying on the set of dependencies, which are currently included in the 
project being developed. 
Aim of project: 
The aim of this thesis is to design and develop a new recommender algorithm, based on existing 
algorithms for keyword extraction and set cover problem. Multiple approaches will be reviewed, 
implemented and compared, starting with the TFIDF algorithm [4] and the Greedy algorithm for set 
cover problem [5].
