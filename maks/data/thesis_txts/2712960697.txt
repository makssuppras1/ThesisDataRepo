Abstract
Designing secure cryptographic protocols is hard and error-prone. Therefore it is use-
ful to check the security properties the protocol should have using an automated model
checker, as these can find attacks on the protocol that could otherwise have been over-
looked.
Ultramarin is a new transactions based protocol specification language with no formal
semantics. This thesis designs and implements a translator enabling Ultramarin specifica-
tions to be checked using the model checker ProVerif. At this point the translations can
only be checked for secrecy goal attacks. The translation has the theoretical value, that
it assigns to Ultramarin a semantics in terms of ProVerif Applied π-Calculus.
The thesis also implements an Ultramarin specification of the Needham–Schroeder
Public Key protocol which is known to be vulnerable to a man-in-the-middle attack.
When running ProVerif on the translation produced, ProVerif finds the known attack.
The translator is also tested on the fix of the protocol known as the Needham–Schroeder-
Lowe protocol. When running on the translation of this protocol, ProVerif does not find
an attack, which is the expected result.

Introduction
A protocol may be described as a rule based system dictating how parties should inter-
act in a given setting or situation. In digital systems protocols are ubiquitous. To prevent
digital systems from being misused they need to be secure. An example of protocols in use
is online banking. In order to check your account balance or make a transaction through
online banking, your computer needs to communicate with a web server. That web server
may need to talk to another system in order to get your account details or ask for a trans-
action on your behalf and finally send something back your computer can display. A lot
of protocols will be invoked in order to archieve this, and a lot of thought should go into
the design of them in order for it to be secure. The main ingredient in protocol security
is the use of cryptograhic primitives, as they, in theory, allow us to send messages that
can only be read by the intended receiver (confidentiality), authenticate the sender of the
message (authentication) and prove someone composed a message (non-repudiation). The
“in theory” part is due to the fact, that designing a truly secure protocol, even using cryp-
tographic primitives, has turned out to be hard and error-prone. This is the case even for
simple protocols, and many real life protocols are not simple. There are many examples of
attacks on cryptographic protocols [Panti et al.]. One example is the Needham–Schroeder
Public Key (NSPK) protocol [Needham and Schroeder, 1978] for which Gavin Lowe found
an attack 17 years after the protocol was published [Lowe, 1996]. A way to decrease the
risk of design flaws in a protocol is to use formal methods to check for possible attacks.
This works by modelling the protocol within a model in which an adversary (or attacker)
is given some capabilities and some goals. One example is the Dolev-Yao model [Dolev
and Yao, 1983], in which the adversary has full control of the communication medium.
However, the adversary does not have the ability to read an encrypted message if it can-
not produce the appropriate key.
Or in other words, the model assumes an unflawed
implementation of cryptographic primitives.
Various tools exists in which security properties of a model can be defined and tested.
For instance, in the tool Open-source Fixed-point Model Checker (OFMC) one can define
some goals for the protocol (which are attack states - states the adversary should try to
reach) [Mödersheim and Vigano, 2009]. OFMC can check a model but is limited to a
bounded number of sessions. Another tool is ProVerif which can check an unbounded
number of sessions. This ability does not come for free as this is only possible because the
ProVerif model uses over-approximation. This may lead to false positives (here positive
means an alleged attack). However if the model does not find an attack, that is a strong
indication that the model is actually safe. These tools use different input languages in
which the model and goals (attack states) can be defined. Examples are various Alice and
Bob style notations, such as AnB [Mödersheim, 2009], AVISPA Intermediate Format IF
[Armando et al., 2005] and Applied π-Calculus [Abadi et al., 2017]. Alice and Bob nota-
tion is simple, but also limited. Applied π-Calculus on the other hand is very powerful, as
it allows modelling several concurrent processes, but it is more diﬀicult to use. Another
language is Ultramarin [Mödersheim, 2022]. Ultramarin introduces atomic transactions
and Object-oriented Programming (OOP) style local state, where an instance of a role
may declare local variables which transactions of the role can access and manipulate. This
makes it simpler to model for instance the Double Ratchet algorithm, as a transaction
may include spinning the ratchet, and the result can be stored in accessible local state.
Ultramarin is a new language which does not have its own tool for automatic verification
of security properties. It has been connected to OFMC through a translation to IF.
This thesis connects Ultramarin with ProVerif by implementing a translator from Ultra-

2.2
Ultramarin
This section is based on a draft of a manual for the Ultramarin language by Sebastian
Alexander Mödersheim [Mödersheim, 2022]. Ultramarin has support for protocol sets.
Protocol sets act as a state that may be shared between roles and different sessions.
Unfortunately a key element of using protocol sets in modelling, is the ability to not just
add something to the set, but also remove it. This is not supported in ProVerif due to its
over-approximating approach. Therefore the rest of this section will describe Ultramarin
without the bits only related to sets.

2.2.1
Syntax
The structure of an Ultramarin specification is expressed in pseudocode in listing 1 on
page 5. All line numbers in this section refers to this listing unless another is explicitly
referenced. Any text in listing 1 in italics, is a name of a function, event, role or transaction.
One exception is where arity is written. There it should be noted what the arity of the
function or event is. One can declare zero or more events and public, private or transparent
functions respectively.
As mentioned, in section 2.1, the parser for Ultramarin have been modified to allow
omitting the Public, Private and/or Transparent sections if no functions need to be declared
in one of these categories. Public functions can be used by all agents, Private functions
are not available to the attacker and Transparent functions can both be composed and
decomposed by all agents.
The bracket [, more params] should show that a role can have more parameters than
the one, which is the agent running the role. The bracket [rolei initiating steps], in lines
19 and 30, can contain any steps, but is typically where the roles declare the variables that
will serve as shared state for the transactions of the role. The brackets [roleitransactionj
steps], in lines 21, 25, 32 and 36, is where the steps of a transaction is declared. Usually a
transaction begins with an “if” step, that checks whether the transaction should run given
the current shared state. This will be elaborated in subsection 2.2.3.
What is not shown in listing 1 is steps. Steps use messages. Messages are either a
varible x or a function f, if it is a constant with arity 0, or f(t1, . . . , tn), if it has arity n,
where ti is also a message. When m and t are messages and r(t1, . . . , tn) is a role taking
n parameters, then a simplified list of the possible steps is
• send(m)
• receive(m)
• new x
• x := m
• try m = t
• if m = t
• if m != t
• event: m
• spawn r(t1, . . . , tn)
(only supported in end of main role)
Further restrictions apply on the messages for a given step. For instance, variables used
in new, receive and the left hand side of try steps should be new (not yet declared).
The step new can only be used inside transactions. When receive is a function, it may use
only public or transparent functions, excluding the decryption operators. In ProVerif only
transparent functions (known as data functions in ProVerif) may be used. Both a receive
and try step uses pattern matching, but may fail if what is received, or given, does not
match the expected pattern. This stops the process and rolls back any changes if the step
is in a transaction. For formal details one may refer to Mödersheim [2022]. The paper
states that the variable x assigned to with x := m, can be new in the initiating steps
of a role, but should be introduced first, when used in a transaction (using new). This
limitation did not seem to add value in the translation to ProVerif and has therefore been
left out.

2.2.2
Roles
While the first 10 lines in listing 1 on page 5 sets the scene, by declaring protocol sets,
functions and events, the rest of the specification is roles with transactions inside. Besides
the main role, which should not declare transactions, Ultramarin requires that all roles
have at least one transaction. Transactions are declared after any initial steps in which the
role may setup a local state, which is in scope and shared by all transactions. This part of
the Ultramarin design is comparable to OOP, if the role is seen as a class and transactions
as member methods of the class. The initial steps are the constructor setting up local
state (parameters of the role are considered part of the state), and the transactions are
member methods that may access and modify it.
The main role, which is seen in lines 11–17, has no parameters and is different from
the other roles. The other roles have at least one parameter with the first one interpreted
as the agent running the role. The main role is most commonly used to setup the model
environment and send a private key to the attacker if asymmetric encryption is used.
The attacker may also be allowed to pick the agents running roles, such that different
combinations of agents can be run in parallel sessions (assuming that the model checker
used, models more than one session or possibly an unbonded number of sessions). Finally
the main role should spawn the roles with the agents picked for the session.
2.2.3
Transactions
Ultramarin models transactions (which consists of a sequence of steps) as atomic units.
That means it is easier to reason about changes to shared state. Either a transaction
succeeds which may include modifications to state, or it doesn’t and nothing changes. This
also means that there are no race-conditions between steps in transactions. A transaction
may be applied any numer of times. Usually a step counter or similar measures is used to
control when a given transaction should be able to succeed. Remember that a transaction
may “run” at any time (however only one at a time), but if it cannot succeed, then nothing
changes.
A small example is given in listing 2 on page 7, in which Step is initialised to 0 in line
2. That means the transaction role_transaction2 will initially fail in line 9 and rollback
(meaning no changes committed). Transaction role_transaction1 would however succeed
having altered the value of the shared variable Step to 1. Now role_transaction1 cannot
run again, but role_transaction2 would now succeed. role_transaction2 could also have
altered Step back to 0, if the two transactions should take turns.
This illustrates how the notion of transactions is powerful in allowing one to model
advanced systems in a simple way using small units of actions. For instance an API could
be modelled by each transaction initially receiving a message, and then checking whether
that message has the correct format which prompts that given transaction to respond
(with a sequence of steps being taken) or otherwise rollback (do not act).

2.2.4
Semantics
There is no formal semantics defined for Ultramarin, but cryptograhic primitives are
defined using a “term algebraic model where terms are equal modulo a set of algebraic
properties” (Mödersheim [2022]). Modelling the algebraic properties needs to be handled
by the model checker. For instance they are fixed in OFMC. How this is handled in the
translation to ProVerif is described in subsection 3.1.1 on page 15.
The algebraic properties cover the functions:
• exp
• scrypt and dscrypt
• crypt and dcrypt
• vsign

AST
The implementation of the translator operates on an AST produced by the parser for
Ultramarin. The type definitions describing the AST can be seen in listing 3. The main
type which contains an entire Ultramarin specification is Spec which is seen in lines 8–17.
Most of the fields in the type are function and event declarations (of type Symdec), but it
also contains fields for the main role and the other roles. The type definition for a role is

2.3
ProVerif Applied π-Calculus input language
This section is based on the user manual for ProVerif [Blanchet et al., 2021].
2.3.1
Basics
The basic building blocks of Applied π-Calculus are processes and terms. The process
and term grammar will not be replicated here. An overview of the ProVerif term and
process gramar can be seen in Blanchet et al. [2021, p. 16] with the full term grammar in
Blanchet et al. [2021, p. 40]. Much of the term grammar is not relevant for translation
from Ultramarin as it has to do with conjunction, disjunction, negation and natural num-
ber comparison operators which are not included in Ultramarin. The process grammar
for conditionals says “if M then P else Q” where M is a condition and P and Q are
processes. When omitting “else Q” the interpretation is simply that Q is the null process.
This shorter version fits better with Ultramarin, as each transaction is modelled as a sub-
process and any failures in an if statement should abort the transaction, and is therefore
used. The same applies to the term evalutation process “let T = M in P”, which not
only serves as a name restriction but also as a pattern matching test. So process P is only
run if the pattern in T can be matched with M. Otherwise the process terminates as the
null process is then run. ProVerif Applied π-Calculus supports the use of macros to define
sub-processes which can then more conveniently be combined using the process grammar.
The translator in this project uses this feature widely. The syntax for macros is given in
subsection 2.3.6.
2.3.2
Types
The Applied π-Calculus used by ProVerif is strongly typed. When doing term evalua-
tion (binding something to a term if the patterns match), the type may be omitted. But
only if it can be infered from the context. The same applies for binding using let syntax.
User defined types can be declared by writing “type t.” where t is a name for the type.
However the translation uses no user defined typed. Instead it uses just three built-in
types, which are bitstring, channel and nat (natural number). Terms of type channel are
used as the medium on which messages can be sent between processes. nat can take part
of aritmetic expressions (as seen in the full term grammar in Blanchet et al. [2021, p. 40]).
The type bitstring is sort of a base type, for instance it is the type of tuples.
2.3.3
Functions and events
User defined functions and events are easy to declare. It may be either a constant or a
function of arity n. Assume name is the name of the function, ti is the type of its arguments
for i ∈{1, . . . , n}, treturn is the type of the function application and attributes are either
none (in which case the brackets should be omitted) or one of private or transparent. Then
a function declaration can be made as shown below.
const name : treturn [attributes].
fun name(t1, . . . , tn) : treturn [attributes].
If a constant or function has no attributes, it is public, so all agents including the
attacker has access to it. If the data attribute is used, then it is public and implicitly
functions exist that allows the arguments of a function application to be retrieved as if
it was a tuple.
In other words given a data function f(t1, . . . , tn), t1 through tn can
be recovered. This is equivalent to a transparent function in Ultramarin. If the private
attribute is used, then it is not accessible to the attacker.

2.3.5
Declaring cryptographic primitives
In ProVerif there is no built-in cryptographic primitives. Instead they must be declared.
In many cases this is however relatively simple. If symmetric encryption is needed we may
decide to use the function name scrypt for encryption and the function name dscrypt for
decryption. One may then define this cryptographic primitive with the lines:
1
fun scrypt(bitstring, bitstring) : bitstring.
2
reduc forall m: bitstring, k : bitstring;

The main challenges in translating from Ultramarin to ProVerif are i) rearranging

cryptographic primitives (see subsection 3.1.1), iii) modelling the secrecy goal (subsec-

Before diving into the translation implementation, this chapter will begin with a de-

Listing 1 on page 5 showed in pseudocode how a Ultramarin specification is structured.

The functions may have attributes as described in
subsection 2.3.3 on page 10. Following function and event declarations, all of the roles

In ProVerif a process is defined in terms of other processes. As mentioned in subsec-

must be defined before being used. This means that the translation to ProVerif Applied
π-Calculus will be reversing the order of the Ultramarin specification, such that before
main, the roles are declared, and before the roles, the transactions used in the roles are
declared. The transactions need to be declared at the top level, as it is not possible to

The entry point is what follows the process keyword. At the end of the listing, in lines 44–

One of the main reasons for using ProVerif to check a model is the ability to model an
unbounded number of sessions. That is why the main role is started as a replication. This
bit is the same in any translation to ProVerif.
The main sub-process in lines 41–43 will contain some initialising steps and then the
process starts the role sub-processes as a parallel composition. These will be passed the
same parameter values as in the Ultramarin specification, which means they will all be
passed at least one parameter, which is the agent running the role.
There may be several roles but they will have the same structure. First they will have
some initialising steps, and then they create a new channel. There is no need to encrypt
messages sent on this channel, as only the processes that have a reference to it may use
it. On this channel they share the local state, which will consist of the parameters of the
role and any new variables introduced in the initiating steps of the role. Finally the role
process starts the transaction sub-processes that belong to the role. They are started as a
parallel composition with each of them as started as a replication, meaning they can run
any number of times. It passes the newly created channel to the transactions, so they can
use it to receive the shared state.

ProVerif base
In Ultramarin all communication is assumed to be sent on a public channel which the
attacker controls. In ProVerif no channels exists before being declared, and therefore the
prelude introduces a public channel c on which communication is sent by default.
In
ProVerif the attacker has the same control, over channels it knows, as the attacker in
Ultramarin has of the one channel used there.
The only communication not sent on this channel is the internal sharing of state between
transactions of a role, which is something that is added in the translation and not part of
Ultramarin specifications, where sharing of state between transactions is an implicit part
of the Ultramarin logic.
The prelude also declares that the attacker exists under the name i and is of type
bitstring. This is important as the logic of the secrecy modelling depends on the attacker
being refered to as i. If a model requires the attacker to have its own private key, it needs
to be sent to the attacker as is done in the main role of the NSPK Ultramarin specification
in listing 11 on page 24.
The last thing declared in the prelude is a type conversion function nat_to_bitstring
which is used when it is neccesary to convert a nat (natural number) to a bitstring for
instance when comparing terms. In subsection 3.1.3 it will be described when this type

The NSPK protocol is an interesting case as it has a bug allowing a man-in-the-middle
(MITM) attack and as a simple fix exists as well, known as the Needham–Schroeder-
Lowe (NSL) protocol [Lowe, 1996]. This will allow us to test that the ProVerif translation
is able to catch the secrecy violation in the original NSPK protocol and verify that the
secrecy goals hold for NSL (see chapter 4).
The full Ultramarin specification used in this example is shown in listing 11 on page 24
and a walkthrough will be given in subsection 3.2.1. Then section 3.3 on page 24 will show
relevant excerpts from the corresponding translation to ProVerif. The full translation can
be seen in Appendix B on page 63.
3.2.1
An Ultramarin NSPK specification
The NSPK protocol model used for this project is a simplified version, in which the
trusted server S that is used to distribute public keys has been left out. This is because
the acquisitioning of public keys from the server S has no significance for the attack on the
protocol. The goal of the protocol is for two agents to establish a shared key for ongoing
communication, which should only be known by the two agents when the protocol finishes.
The Ultramarin specification of the NSPK protocol can be seen in 11 on page 24 and a
walkthrough will be given now.
Instead of distributing public keys using a trusted server, a public function pk is de-
clared in line 4. This means that given an agent name, a, the model allows agents to
compose the public key of agent a. Also declared in line 4 is a public constant Nothing. It
is simply used as in initial value for shared state variables of a role, that will be updated
with another value in one of the transactions.
The transparent functions msg1, msg2 and msg3 are used to encode messages consisting
of several values. This also prevents similar messages (same number of values with same
types) from being misinterpreted. As this protocol is simple and the content of the mes-
sages are checked to be what is expected at a given point, this is not strictly neccesary,
but it means that the use of transparent functions is also tested when the specification
gets translated to ProVerif. The secret event is declared in line 8 and will be used to
check that the nonce NB is a shared secret between agents A and B when the protocol has
finished.
The main role is declared in lines 9–16. The intruder receives an identity in lines 10
and 11, consisting of the name i and a private key for i, by transmission of these on the
channel he controls. The intruder may choose the names of agents A and B who should
run the protocol by sending some names which are received in lines 12 and 13, and then
used to spawn the two roles that will run the protocol. The role ClientA is the initiator
of contact and therefore needs the name of both the agent running it and the agent who
should be contacted. The role ClientB receives, in a message, the name of the agent with
whom it should run the protocol. Therefore role ClientB only needs to know its own
identity.
Lines 17–35 declares the role ClientA. As all roles except the main role needs to use
transactions in Ultramarin, ClientA initialises a Step variable in line 18 to control when
each transaction should run. ClientA sends a nonce NA to agent B, in the first transaction
and checks that it is part of the message it receives from agent B in the second transaction.
Therefore the nonce NA needs to be shared between transactions and is therefore declared
in the initialising steps of role ClientA in line 19. At this point the Nothing constant is
used as the step new is not allowed outside transactions. In transaction ClientA_step1,

3.3
Translating the Needham–Schroeder Public Key
protocol
This section will show how different parts of an Ultramarin specification, specifically
the specification of the NSPK protocol, that was introduced in subsection 3.2.1, can be
translated to ProVerif.
The first part of any ProVerif translation is a prelude which declares cryptographic
primitives and other pieces used in ProVerif modelling, such as the public channel c,
reserving i as reference to the attacker and the mechanics of querying for breaches of
secrecy goals. The content of the prelude will be elaborated on in subsection 3.1.1.
The next sections will elaborate on how the ProVerif translator implements checking of
the secrecy goal (subsection 3.3.1), models both roles and transactions as sub-processes,
and shares state between transaction sub-processes through a channel only known to the
transaction processes (subsection 3.3.2) and finally, the main role, which is where other
roles are spawned, is described (subsection 3.3.3).
3.3.1
Secret goal
Translating the secrecy check is non-trivial as the the interpretation of a secret event
is determined by the context. For instance, refering to listing 11, the event in line 58 is
encountered in a transaction, which is executed by Agent B (the first parameter of the
role, as seen in line 36). The first parameter passed to the event is NB and the second is
(Agent) A. This should be interpreted as, NA is a secret between agent B and agent A.
In ProVerif it is easy to test whether an attacker can produce some value X. You simply
put in the line
query attacker (X).

It is possible to query whether an attacker can produce some locally declared varible,
such as a nonce for instance [Blanchet et al., 2021, p. 100]. It is however quite a bit
more complicated than referring to a globally declared variable.
So instead a simple
implementation is used, which declares a secret global value, symbolic_secret. This value
is then sent by Agent A to Agent B on a public channel (for which a channel named “c” is
declared) encrypted by the value X, which we wish to test whether is secret between the
two. If the attacker can produce the secret global value, which is only sent encrypted by
X, which itself should be secret, then that encryption key X must have been produced by
the attacker and used to decrypt the message containing symbolic_secret. The fact that
the attacker has produced the secret key X will also be evident from the output ProVerif
produces when it shows that symbolic_secret cannot be proven to be out of reach of the
attacker (see section 4.1). Of course we need to prevent false positives in the case where
the attacker is simply participating as an honest agent and therefore have acquired X. To
ensure this, the symbolic_secret is only sent, encrypted by X, when the attacker is neither
agent A or B. The secrecy test can be implemented as a sub-process macro in ProVerif as

3.4
Translation implementation
This section will describe the Haskell implementation that translates an AST of an
Ultramarin specification to ProVerif Applied π-Calculus specifications. The full code for
the translator can be found in Appendix A on page 47.
As seen in the pseudocode structure of a ProVerif translation shown in 4 on page 15,
a translation to ProVerif has the following structure:
• Prelude
• Functions and events declared in Ultramarin specification
• Main part with translation of roles and transactions
– Transactions for a given role need to be defined before the role which spawns
them
• The main role
• ProVerif entry point
The main role is handled as a case of its own as it i) has no agent running it, and
hence no parameters, ii) has no transactions and hence do not need to share it’s state on
a private channel, iii) needs to spawn the other roles as a parallel composition.
When doing the translation most parts can be translated independently, with the ex-
ception of transactions which need to be translated with the role to which they belong.
Once the initial steps of a role have been translated, the transactions of the role can how-
ever be translated independently as the only dependency for the translation is the type
map produced when translating the initial steps of the role. As the prelude, functions and
events, the main role and remaining roles can be translated independently, the indiviual
translation tasks will be described in a section of their own.
The combination of the results into the final translation is done in the function trans-
latePVspecAux which is shown in listing 19 on page 30. The function formatSeveral
is used throughout the translation and simply takes a delimiter and a list of Strings and
concatenates the Strings with the delimiter in between. As many of the functions used
in the translation may return an empty string, formatSeveral removes those from the
list before doing the concatenation with the delimiter. The prelude is a hardcoded String
combining the definitions mentioned in subsection 3.1.1 on page 15. Each of the functions
translateFunctions, translateEvents, translateMain and translateRole translates
a specific part of the specification. The last bit added is the ProVerif entry point that
simply starts a replication of the main role.

Haskell type constructor Either a b is used throughout the translation implementation2.

3.4.3
Varible scope and type determination
The strongly typed Applied π-Calculus used by ProVerif requires the translation to be
explicit on types. Types can be used actively to protect against modelling mistakes when
writing a protocol by hand. However when doing translations from a non-typed language
such as Ultramarin it is neccesary to assign a type signature to functions and variables.
ProVerif has a built-in type called bitstring. The translations to ProVerif use this as the
main type.
So by default, everything in the translation gets set to type bitstring, functions take
type bitstring and return type bitstring. ProVerif will not allow us to assign a number to
a bitstring variable. So the step counter used to coordinate transactions within a role, will
have to be of type nat (natural number). Currently natural numbers is the only special
case that needs to be handled. ProVerif also requires that terms are of the same type
when using the comparison operators for equality or disequality. To ensure this, the types
of terms are checked using a type map of the variables that are in scope. If they have the
same type, they can be compared without type conversion. If they are not the same type
each term is converted to the bitstring type before comparison.
The type map is created and maintained during translation and needs to contain the
type information of any varible that is in scope. Globally declared functions (of public,
private or transparent availability) are not in the type map as these are all assigned the
type bitstring. Constants are however included in the initial type map.
The determination of which variables are in scope is not to complicated. For a role just
the parameters and global constants are initially in scope. The initiating steps of a role
may introduce more variables, which are all in scope in the transactions of the role along
with the parameters of the role. Inside a transaction new variables may be introduced,
but these are only in scope within the transaction and not shared between transactions.
Whenever a step is processed, it is determined whether the step introduces a new variable
and if so which type it has. If a new variable is introduced then the variable and type is
added to the type map for subsequent steps.
In order to pass the correct type map to the translation function for a given step, the
translator needs to process the initial steps of a role first and then each transaction can
be processed independently of each other.
The type map is kept in a translation environment using a reader monad4. The envi-
ronment also contains the variable name of the agent running the role, as this is needed
to model the secrecy goal correct (both the sender and receiver is passed to testSecret).
Listing 23 on page 35 shows how the type map and environment for the reader monad is
declared. Using the reader monad allows the translation functions, that implement it, to
ask for the environment if it is needed and otherwise ignore it. When within the reader
monad environment, it is possible to call another function using the reader monad with an
altered environment (using the function local). This is used to update the environment
whenever needed.

3.4.5
Steps
The translation of steps is close to the description given in subsection 3.1.3 on page 17
which included listing 10 on page 20 with a semantics function for the translation of steps.
The translation of steps does however include several checks that were not part of the
listing. If the checks fail, the translation will abort and return an error message. What
is checked is that the steps do not i) use ProVerif keywords or names reserved for the
translation, ii) assign a value of a new type to a variable that has already been given a
type, iii) use new x if x is not new, iv) reference a variable that is not in scope.
The function for translating a series of steps is run in the reader monad environment,
which means that at any point it is possible to fetch the current type map (and agent
name). In some cases, the calling function may want the updated type map arising from
having taken all of the steps being translated. For instance when translating the intial steps
of a role, the updated type map with new bindings needs to be passed to the function
that translates the transactions, as these bindings are in scope in the transactions and
is therefore needed to translate the transaction steps. Therefore translateStepsNoEnd
returns both the translation of the steps given as parameter and the final type map.
The basic design of the function is to look at the first step in the list of steps. Translate
that step using the function translateStep that takes only a single step and returns the
String translation. Then figure out, if the current step is binding something, and if so, add
it to the environment used to translate the remaining steps. If it is not binding anything,
then the step can just be translated and the current environment can be passed on, without
modification, to translate the remaining steps as it is seen in listing 10 on page 20. The
remaining steps are translated using a recursive call to translateStepsNoEnd, so this call
will return the updated type map and the translation of the remaining steps. Finally,
combine the translation of the current step with the translation of the remaining steps,
and return the updated type map to the calling function. If there are no steps left, then
simply return an empty string and the current environment.

specification which will be put to the test in section 4.1 on page 41. The protocol is also
interesting, as the fix of it is very simple and easy to implement. Therefore the model

4.3
Future work
The translator is currently shown to work on one protocol. Future work should include
implementing and testing it on more Ultramarin protocol specifications. The translations
have logic for checking secrecy goals but should be extended with logic for checking au-
thentication goals with witness and request events. Logic could also be implemented that
simply supports checking that an event can be reached. That would be useful for gener-
ating a test suite of protocols to test and ensure that the translated programs have and
keep having the expected semantics. The translation can also be extended to support the
protocol sets of Ultramarin with the limitation that any protocols that remove elements
from the sets, should be rejected, as ProVerif does not support this.

5
Conclusion
The thesis connected the transaction based protocol specification language Ultramarin
[Mödersheim, 2022] to the powerful ProVerif [Blanchet et al., 2021] Applied π-Calculus
[Abadi et al., 2017] language. This allows Ultramarin protocol specifications to be checked
using ProVerif, in which an unbounded number of sessions can be modelled and checked
for attacks.
The translation was done by i) fixing a prelude introducing the cryptographic primitives
needed and introducing ProVerif specific global variables for the rest of the translation
to rely upon, ii) designing a mapping from the structure of Ultramarin specifications to
a structure fitting the ProVerif Applied π-Calculus syntax and iii) developing semantics
functions to fill in the gaps. The thesis has not only provided a practical tool for transla-
tion, but in doing so, also given the new Ultramarin language a semantics in terms of the
translation to the ProVerif Applied π-Calculus language.
Ultramarin specifications of the Needham–Schroeder Public Key (NSPK) protocol
[Needham and Schroeder, 1978] and the Needham–Schroeder-Lowe (NSL) protocol Lowe
[1996] has been implemented, and used to test the translator.
The NSPK protocol is
known to have an attack [Lowe, 1996], and the results were promising as ProVerif found
the expected attack when running on the translation of the NSPK protocol in ProVerif.
When the translator was tested on the fixed version of the protocol, the NSL protocol, it
could not find any attacks, which was the expected result.
Translations from Ultramarin to ProVerif Applied π-Calculus can be checked for viola-
tions of the secrecy goal. Future work should extend the translator such that authentica-
tion goals can also be checked when running the translations with ProVerif. The semantics
of Ultramarin should be formalized and in doing so a test suite of Ultramarin specifications
could be made to ensure that translations to ProVerif Applied π-Calculus have, and keep
having, the expected semantics as work on the translator continues.
