Introduction 
Vision-based robot control is applied in many factories around the world for pick-and-place 
tasks. A robot arm sits next to a conveyor belt as shown below. There is a camera attached to the 
end-effector and one overlooking the conveyor belt. This system is called a hybrid eye-in-hand 
and eye-to-hand camera system. The robot first picks up an object from the belt; and then it 
tracks a colored marker on the same belt. Once it is close enough to the marker, it drops the 
object onto it. This marker shows the location of a container or compartment into which the 
object must be dropped. These tasks are accomplished using P and PID controllers and image 
processing.  
 
Figure 1: Workspace; the red circle shows the location of the eye-to-hand camera;                                                                    
the blue circle shows the location of the eye-in-hand camera. 
There are two types of visual feedback control: direct visual servo control and dynamic look-
and-move control. The system diagram for the former is shown below. 
==PAGE:5==
5 
 
 
Figure 2: Direct visual servo control 
The robot joints are directly controlled by a controller that uses information from the image. In 
dynamic look-and-move, the robot joints are controlled independently of the camera. This 
system is much more widely used due to its simplicity. There are two subcategories of the 
system: image based control and position based control. Their corresponding system diagrams 
are shown below. 
 
Figure 3: Image based control 
In image based control the features such as corners, points, or lines are extracted from the image.  
This information is then subtracted from a set value, and the resulting error is passed to a feature 
space control law. The output from that goes to the robot joint control system. This method of 
control is based on the following vector equation: 
𝑓̇ = 𝐽(𝑟)𝑟̇ 
==PAGE:6==
6 
 
where 𝑓̇ contains the rates of change of image feature parameters, and 𝑟̇ represents the velocity 
of the end-effector in Cartesian coordinates. They are related by 𝐽 which is called the image 
Jacobian. 
 
Figure 4: Position based control 
In position based control image features are first extracted again. Using this information the pose 
of the end-effector is estimated. This is subtracted from a set value for the pose. The resulting 
error is sent to a Cartesian control law, and then to the robot joint control system. The final result 
is that the pose of the end-effector is brought closer to the desired value. In this project I utilize 
position based control for both picking up and placing objects. The feedback loops I use are 
slight modifications of the one shown above. They are presented further below. 
I use an eye-to-hand as well as an eye-in-hand camera attached to the end-effector. I use a similar 
methodology to that described in [1]. The eye-to-hand camera sits on top of the conveyor belt 
with a wide field of view. It determines the velocity of the items on the belt as well their x and y 
coordinates. The eye-in-hand camera is used to track the moving container. If the velocity is low 
a second mode of operation is activated. Multiple objects are seen by the eye-to-hand camera. 
The robot arm picks them up and places them onto a stationary surface. 
Some recent papers have used this system [1] [3], whereas others have only used an eye-in-hand 
camera [2]. A combination of position based control and image based control is used for pick-
and-place tasks in [3]. A PID visual feedback controller is used as well. This paper suggests that 
a single camera is not sufficient for large, complex environments. However, in some situations a 
single eye-in-hand camera is sufficient. Point error and area error are needed to obtain depth 
==PAGE:7==
7 
 
estimation [2]. This paper makes use of image based control. Determining camera velocity from 
the image features is one of the primary objectives. 
Image processing is another important part of this project. It is needed to isolate the moving 
objects from the conveyor belt and other surroundings; it is then used to place objects onto the 
moving table; and lastly it is used to form a pile on the stationary surface. The image processing 
is done using the Saturation Hue Value color scheme which is described as follows: Hue is 
simply the color ranging from red to violet. Saturation is the intensity of a given color. Its value 
ranges from gray or colorless to highly colorful. Value is the brightness of a given color ranging 
from black to very bright (almost white) [4]. A cylinder is used to represent this system. Hue 
varies going around the cylinder. Saturation or color intensity varies radially outward; and value 
or brightness varies up or down for a given hue. 
 
Figure 5: HSV diagram [5] 
The color of the objects is different from that of the conveyor belt. This makes it possible to 
distinguish them from the conveyor with the image processing. On the second conveyor, a small, 
colored rectangle is used as a marker. This marker shows the location of the container. The end-
effector is able to track it at various speeds.

System Overview 
This project simulates a factory environment with two conveyor belts. The robot picks up objects 
from the first conveyor and places them onto a second conveyor. One belt is used to simulate 
both. The control algorithm uses the eye-to-hand-camera to calculate the velocity of the object 
when it moves into view. If the velocity is between 0.17 m/s and 0.35 m/s the time-to-grab is 
then calculated. Afterwards the robot arm extends to a specified location on the conveyor. It does 
this with two P controllers: one for x-control and one for y-control of the end-effector. A 
coordinate transform is used in the algorithm to obtain coordinates in the end-effector frame. 
Finally, the end-effector goes down by approximately 20 cm and the gripper grabs the object. 
The end-effector then moves to a position overlooking the edge of the conveyor. This motion 
simulates moving to the second conveyor. Now the eye-in-hand camera is used to track the 
colored marker once it comes within view. I use a PID visual feedback controller for x-control 
and a P controller for y-control. Once the end-effector encounters a second colored marker it 
drops the item. Also a time limit is used for this purpose. The robot moves back to its original 
position, and the cycle repeats. 
If the object velocity is less than 0.17 m/s a second mode of operation is activated. In this mode, 
the spacing between objects can be small because of the low velocity; therefore multiple objects 
can enter the view of the eye-to-hand camera. A bubble sort is used in the control algorithm to 
obtain the x and y coordinates of the object that is furthest to the right. The robot arm extends to 
various positions along the conveyor to grab each object. Once the robot picks up an object it 
drops it onto a stationary surface, and the cycle repeats. 
Before the control algorithm was created, velocity measurements were taken with the eye-to-
hand camera. The purpose of this was to see how well the camera performs as a velocity sensor. 
The results of these measurements are shown in the next section.

Velocity Measurements with Camera 
An algorithm for determining object velocity with the camera was created. Most of the code of 
this algorithm is incorporated into the control algorithm. There is an encoder attached to the 
conveyor which measures its velocity. A BeagleBone board reads the velocity values and sends 
them to the computer. Velocity values obtained using the camera are compared to those provided 
by the encoder. Each plot shows an oscillation of velocity. 
 
Figure 6: Velocity oscillation plot 1 
The velocity is kept constant for each time interval. There is a delay on the camera velocity 
which grows to approximately one second. However, in the time intervals where the velocity is 
held constant, the camera values are almost the same as the encoder values. This shows that 
when the velocity is held constant, the camera provides very accurate results. The noise 
amplitude is between 0.02 and 0.03 m/s. This noise is caused by the image processing. 
==PAGE:10==
10 
 
 
Figure 7: Velocity oscillation plot 2 
Now the velocity reaches 0.5 m/s, and the time delay is more clearly visible. However, despite a 
few spikes, the camera still provides very accurate velocity values with a time delay. 
 
Figure 8: Velocity oscillation plot 3 
In each time segment the velocity is gradually either raised or lowered. The camera gives very 
good results once again, with a time delay. 
==PAGE:11==

Figure 9: Velocity oscillation plot 4

a time delay. 
For the rest of this thesis, the velocity is held constant. So, the time delay will not be relevant. 
However, the noise will be relevant. This is discussed in the Experimental Validation section.

Conclusion 
The robot was not always able to grab the objects. This is because there is some noise in the 
velocity values provided by the camera; this noise affects the time_to_grab value, and the robot 
is sometimes not able to grab the object at the right moment. However, the control loop for 
picking up objects gave good results. The robot is always able to reach the position marked by 
vc1. The visual feedback control loop gave good results as well. The plots show that the robot 
can track movements up to 0.6 m/s. The end-effector stayed near the marker, and the objects 
were placed close to it as well. 
 
I obtained good results in low-velocity mode. The robot always reached its designated position, 
vc1. It was able to pick up most of the objects on the conveyor.
