Abstract
Security protocols are distributed algorithms for digital communication on networks. They
are designed to ensure that certain security properties are met even on insecure networks,
but verifying that protocols actually satisfy the intended properties is no simple task.
There exists various automated model checkers, but none of these tools are perfect, as they
usually cannot model and test all types of protocols for all types of security properties.
Ultramarin is a new protocol specification language with no testing tools of its own. This
paper designs and implements a transcompiler, that can translate Ultramarin specification
into Tamarin, with the intention of being able to test the protocol with the Tamarin Prover.
This paper also translates Ultramarin specification of the Needham-Schroeder Public Key
(NSPK) and Needham-Schroeder-Lowe protocol (NSL) into Tamarin, and verifies with the
Tamarin Prover that NSPK does not satisfy the secrecy property, while NSL does.

Introduction
Protocols define behaviour through rules, and security protocol specifically are distributed
algorithms defining communication between parties on networks. They are designed to
ensure a number of security properties are met, such as secrecy and authentication, even
on an insecure network, where an adversary can pick up as well as send messages unin-
hibited.[2]
Protocols are essential to safe communication, and it is crucial that a protocol can ensure
that the required security properties are met. Sometimes it is simply a matter of being
able to exchange confidential information while being sure of who the parties involved are,
and sometimes it should be possible to send a message anonymously, while still being able
to verify that the message was received. Regardless of the purpose, it is essential to be
able to verify that a protocol can deliver what is required, and many protocol specification
languages also have associated, automatic testing tools. The verification and falsification
of a protocol can be a very complex job, and so there is no tool that is able to check all
types of properties, of all types of protocols.
Ultramarin is a new formal specification language for security protocols. It is a transaction-
style specification language, and supports long term states.[3] It does not have a testing tool
of its own, but there have been made two translators, although both of these translators
have their limitations. One translator translates to AVISPA IF and OFMC, which require
that the number of sessions are bound. The other translator translates to ProVerif, which
have trouble handling protocols with a mutable, long-term state.
The Tamarin Prover does not have these limits. It is a tool for symbolic modelling and
analysis of security protocols, and its language is based on multiset rewriting rules. It
can construct proofs even with an unbounded number of protocol sessions. It has a fully
automated mode, but as there is no guarantee that the Tamarin Prover can finish its
verification of any given protocol, it also offers an interactive mode.[2, 5]
This paper will present both the design and the implementation of a transcompiler, that
aims to translate Ultramarin to Tamarin, in order to further expand the practical use
of Ultramarin to a wider range of protocol types, as the Tamarin Prover offers different
protocol support and testing abilities. It will also propose an extension to the syntax
of Ultramarin, in order to support the translation and testing of en even wider range of
security properties.
The background information required to understand the presented design is included in
chapter 2 where the basic semantics and syntax of both Ultramarin and Tamarin are
described. The design, defined with semantic functions, is given in chapter 3, along with
an example of translation. The paper uses the Needham-Schroeder Public Key protocol
(NSPK) as an example, and when evaluating the transcompiler in chapter 4, the Tamarin
translation of NSPK is tested by falsifying a secrecy property, which is known not to
be satisfied, and verifying that same property in the Needham-Schroeder-Lowe protocol
(NSL), which is an updated version of NSPK, having fixed the aforementioned flaw. This
will be followed by a conclusion in chapter 5. All protocols, both the Ultramarin versions
as well as the Tamarin versions, are given in Appendix A, and the main code of the project
is in Appendix B.

Preliminaries
2.1
Ultramarin
Ultramarin is a transaction-style protocol specification language. It also features sets to
represent long-term states, that can be shared between sessions and agents. The semantics
of Ultramarin are informally described as a state transition system in [3], and it has been
formally explained via translation to OFMC.
This paper generally does not make use of the OFMC translator definitions, which includes
function semantics and a number pf pre-defined security properties. Instead it makes use
of the functions semantics defined by Tamarin. Security property-wise, this paper will
present an extension to Ultramarin in section 3.1, so that there can be more options when
translating an Ultramarin specification into another language, allowing for a wider range
of security properties to be translated and tested, than the ones that have been defined
with the OFMC translator.
2.1.1

A protocol has a name, and consists of collections of functions (divided in groups of
public, private and transparent functions) as well as families of protocol sets,
event predicates, roles and a single main process.

Public functions and transparent function are available to everyone, whereas private func-
tions are not available to an adversary. Transparent functions are used to model message
formats, as these can be inverted without any extra knowledge, such as keys, unlike public
functions.
The main process is a single transaction which has the primary purpose of spawning new
instances of roles - this process can be executed any number of times.
Roles have names, parameters (set of variables), an initialisation transaction which ini-
tialises the role with its local variables, as well as a number of regular transactions.
Transactions are atomic - and all steps in the transaction represent one unit. If, for some
reason, a step cannot be completed, any previous steps of the transaction is rolled back.
Possible steps are:
• Send send(x)
• Receive receive(x)
• New new x
• Assign x := t
• Try try cond where cond is either p = m, p in s, p < −s
• If if cond where cond is either m1 = m2, m1! = m2, m in s, m notin s
• Insert m−> s
• Let let x = m
• Event event : e(t1, .., tn)
• Spawn spawn r(t1, .., tn)
All variables and messages usually have to be defined as local variables, with the exception
of x in New, Let and Receive, as well as p in Try. x is a new variable, known only by the
current transaction, and p could either be a new variable, or a format f(x1, .., xn) where
(x1, .., xn) are new variables. When Try-step happens on the condition p = m, then it
tries to match p with either a variable or decryption m and in this way assign p to m for
the remainder of the transaction. p < −s means it is attempted to remove p from the
set s. If a try-step does not succeed, or if an if-step evaluates to false, the transaction
will not be executed. Send and Receive happens on public channels. Let function as a
substitution so let x = m can be read as ”substitute x with m in the remainder of this
transaction”. In order to understand what these steps mean in practice, en example is
presented in subsection 2.1.2.

2.2
Tamarin
The Tamarin prover (or Tamarin for short) is used for symbolic modelling and analysis of
specification protocols. It uses multi-set rewriting rules to model both the protocol and
adversaries. The rules define a transition system, that symbolically represents both the
adversary knowledge, state of the protocol and information on generated values.[5]
The security properties of Tamarin are trace properties - these are checked against the
various traces of the transition system defined by the rules. Tamarin can also check for
observational equivalence.
A Tamarin specification is called a theory and is introduced by declaring which functions
are used, followed by the rules that model the steps that available in the protocol. This
Ultramarin to Tamarin

is followed by restrictions that impose restrictions on which traces that can be considered
when checking the security properties and lemmas which are the trace properties used to
model the security properties.
The general format of specifications in Tamarin is given in Listing 2.3 below:
theory [protocolName]
begin
builtins: [builtin names]
functions: func1/a,..,funcX/a [private],..,funcN/a
[rules]
[restrictions]
[lemmas]
end

It should be mentioned that there are more options available, that can be included in the
protocol, such as predicates and equations, but they are not relevant to this paper.
2.2.1
Functions and built-ins
Functions can be used in Tamarin - but they must be declared. This can be done by
declaring the function name, its arity and optionally whether it is a private function,
meaning it is not known by an adversary. For example, a function defined as f/2 could
be called on as f(x, y) as it has an arity of 2. If it was a private function, it would have
been defined as: f/2 [private].
Tamarin also offers a number of predefined functions in the form of built-in message theo-
ries. For example, asymmetric-encryption offers a basic public encryption scheme with the
functions aenc/2, adec/, pk/1 which are related by the equation: adec(aenc(m, pk(sk)), sk) =
m.
2.2.2
Rules
Rules cannot be defined without first defining facts. Facts can be seen as parts of state
information and are of the form Factname(x1, .., xn), the name always beginning with
an uppercase letter, and a fixed arity, meaning that having the two facts with similar
names, but different arities, is not allowed. There are a number of special (or predefined)
facts, Out, In and Fr, where In and Out model the receiving and sending of messages on
the public channel, and Fr denotes the generation of fresh variables. A rule is used to
represent an atomic action, and is a transition from one state to another in the transition
system. A rule is essentially consuming and producing facts.
The following rule is a conventional way for Tamarin protocol specifications to generate
public and private keys, and it is also the first rule presented in the Tamarin manual. It
is likely one of the most used rules across different protocol specifications:
rule Register_pk:
[ Fr(~ltk) ]
-->
[ !Ltk($A, ~ltk), !Pk($A, pk(~ltk)) ]

The first part before the arrow is called the premise (also referred to as the left hand
side of the rule). The premise dictates which facts should be present in the current state
in order to transition to the next - in other words, which facts are to be consumed. In
Register_pk, a fresh value is generated in the premise via the special fact Fr - this can
only appear in the premise and allows for the use of fresh variable ltk in the conclusion.
Fr is special in the sense that it is not a requirement to the current state, meaning that a
rule with only the Fr-fact in the premise can be executed at any time. It is used to bind
new variables, as variables always have to be bound. The essentially means that variables
have to appear in the premise in order to be used either in the label or conclusion of a rule
(like ltk). Public values and constants are an exception, however, and are denoted as $A
and ’A’ respectively. Another special fact that only appears in the premise the In-fact,
but it requires that there is an equivalent Out-fact in the current state in order to execute
the rule.
The part after the arrow is called the conclusion, but is often also referred to as the right-
hand side of the rule. The conclusion dictates which facts will be in the next state - as
in which facts are produced. Generally, if a state appears in the premise, but not in the
conclusion, that fact will be consumed and no longer be available in state, unless it is a
persistent fact. Persistent facts are denoted with an exclamation point (!). A persistent
fact can in other words be consumed any number of times, but will still remain available
to future states. In the example of Register_pk, two persistent facts are produced. The
persistent fact Ltk which models the long-term key (more commonly known as the private
key) of some agent $A, while the persistent fact Pk represents the public key.
A fact that can only appear in the conclusion is the Out-fact. If a message m is sent on
the public channel, it is modelled as Out(m) in the conclusion.
rule Step1:
[Fr(~n), Fr(~b)]-->[Out(<n,b>)]
rule Step2:
[In(<x,y>)]-->[ Step2(x,y)]
rule Step3:

2.2.3
Trace properties: Restrictions and Lemmas
Restrictions are trace properties, which limit which traces of the transition system that are
taken under consideration when checking if the security properties, defined by lemmas,
holds for the transition system.
Both lemmas and restriction are defined in the same
manner, using first-order logic over action facts and time points. The part of the syntax
which is used in this paper, from the Tamarin Prover manual[5], includes:
• All for universal quantification, temporal variables are prefixed with #
• Ex for existential quantification, temporal variables are prefixed with #
• ==> for implication
• & for conjunction
• | for disjunction
• not for negation
• f@i for action constraints, i is temporal
• x = y for an equality between message variables ‘x’ and ‘y’
A common restriction would be that of the equality between two messages, and a common
goal would be secrecy, as represented in the label of Listing 2.6 :
restriction Equality:
"All x y #i. Eq(x,y) @#i ==> x = y"
lemma Secret:
"All B #i. (Secret(B) @ #i ==>
not( Ex #j. K(B) @ #j ))"

Expanding Ultramarin with Goals
Both OFMC, ProVerif and Tamarin require that the security properties are defined in the
protocol itself - as these languages are not only used to modelling protocols, they are also
tools for running automated tests which require instructions for what to test. Ultramarin
does not have its own notation or semantics defining the intended security properties of a
protocol specification. The OFMC translator solved this by defining a number of states
called standard attack states, using a predetermined set of event predicates that can be
used in Ultramarin.

satisfied. An example of such a state is secrecy:

Goals are listed in the beginning, also called the prelude, of an Ultramarin specification.

subset of first-order logic, using ==> for implication, ! for negation, || for disjunction and

as it is convention when using public keys to have a rule modelling the intruder revealing
long term keys - exclusively to be used in the security properties that are lemmas. There
is no type in Tamarin that models identities which could be compared with the one of the
intruder.

Translating Functions
Functions are declared in almost the same way in both Ultramarin and Tamarin, but the
main difference being that Ultramarin has a third type of functions, transparent functions,
that Tamarin does not. Tamarin only divides its function in public and private, and the
difference between the two is whether an intruder can make use of the function or not.
Transparent functions in Ultramarin are exclusively used for formatting messages, meaning
that it is possible for anyone (including the intruder) to invert transparent functions to
the original message on which they were applied. In order to support this, the use of
transparent function will be of the form < functionname, x1, .., xn > for a transparent
function functionname/n. This is essentially a tuple, and including the function name
means we still know what format the message is supposed to have.
This also means
transparent function in Ultramarin are not considered functions in Tamarin, and will
therefore not be declared as such.

3.2.2
Translating Roles
Roles in Ultramarin each contain a number of transactions, which are atomic, not unlike
rules in Tamarin.
The roles have parameters and local variables that do not have an
obvious translation in Tamarin. The values of the various parameters and local variables of
a role represent a state of the role. The state can be changed repeatedly by the execution of
transactions. Transactions, by changing the the state of the role, can be seen as consuming
one state, and producing another - not unlike how rules in Tamarin can consume and
produce facts. As the local variables (and parameters) of a role are fixed - as in, new ones
cannot be added, and facts have a fixed number of messages, the state of a role can be
expressed as a fact. Each transaction of a role will be translated into a rule, that has a
fact in both the premise and the conclusion, that represents the state of the role.

3.2.5
Generating restrictions
Restrictions are trace properties, using the same semantics as lemmas. However, they can
only be generated by the translator after all roles and transactions have been translated,
at the restrictions may not reference action facts that are not used by any rule.
Given a set actionNames = a1, ., , an which contains the unique names of all action facts
of a protocol, then restrictions will be generated as show below. NotElemOf<setname>
would be a restriction that is added if a check has been performed, on a given message not
occuring in the set setname. If there is no activity in the protocol, adding or removing
elements from that set, the restriction will always evaluate to true, as there cannot possibly
by any elements. It is assumed that protocols in Ultramarin is using well-formed sets in
the sense that there will not be multiple of the same element, in the same set (multisets).
Without this assumption, it would not be possible in a translation to Tamarin to verify
whether there truly was none of the given element left. Action facts can be compared
time-wise - specifying which action fact must have occurred before another, but their
occurrences cannot be counted. So if there are prior occurrences where the given element
could have been both added and removed, the restriction (the first one below) will require
that either the element has not been added before the NotIn check, or if it has been added,
then it must not have happened after the element has been removed.
restriction NotElemOf <setname >:

3.4
Implementation
The implementation was built upon the base of the OFMC translator, specifically by re-
using and altering the lexer, parser, main and makefile. The main contributions are the
new translator, but also the updates made in connection with extending the Ultramarin
semantics.
3.4.1
Extension With Goals and AST.hs
Extending the Ultramarin specification with goals only contained minor updates to the
lexer and parser, but it all amounts to the updates to the abstract syntax tree, with the
addition of the new types. The goals are optional, meaning that an Ultramarin file can
be parsed to an AST even without goals, but the Tamarin translator will not accept it
and print out an error message, since the Tamarin Prover cannot run without lemmas.
The abstract syntax tree is in the AST.hs file in the appendix (). The idea with making
the goals optional when parsing an Ultramarin specification, is so that the preexisting
translators (like the OFMC translator) should be able to use the same AST. After all,
one of the main points of abstract syntax trees, is that it can be used as an intermediate
representation, used to facilitate transcompilations from Ultramarin to any other relevant
language.
3.4.2
Translator
As for the translator, it is contained in the same file.
It needs to be given an Ultra-
marin specification that has been parsed to an AST, as we as a boolean, indicating
whether or not the Register_pk-rule should be included in the final translation. This
part is handled by Main.hs, which accepts a file name and optionally the following tag
--autoPKdistribution. If the tag is added, then the boolean given to the translator will
be True (and False otherwise).
The translator and its functions follows the semantics functions as described in this chap-

by what it is translating. There are several functions for translating the same aspect of

parts.

Evaluation
The transcompiler presented in this paper will first and foremost be evaluated by having
it translate a protocol with known issue as well as a protocol in which the issue in question
has been fixed, and then verify that it has been translated appropriately by checking if the
Tamarin Prover can parse and respectively find the issue, or verify that the protocol does
not have the issue that is tested for. In both cases the Tamarin prover is run by asking it
automatically to solve any ambiguities with sources and types, to improve running time on
proofs. The command used is tamarin-prover --auto-source --prove <filename>.
All of the protocols referred to in this chapter, both the Ultramarin and Tamarin versions,
can be found in Appendix A.
The protocol with the known issue has already been presented and translated in this
paper, and it is the NSPK protocol. Gavin Lowe found that it was vulnerable to a Man-
in-the-middle attack. Basically, the attack is that the client encrypts a message with the
intruder’s public key. The intruder can decrypt, and encrypt it the Server’s public key.
The server will think the message was from the client, as client is identified in the message,
and encrypt the received nonce along with a new nonce, with the client’s public key. The
intruder cannot read this message, and simply passes it on to the client, decrypts it, and,
because they think the intruder generated the message, will decrypt it and send back the
nonce that the server generated, encrypted with the intruder’s public key. This way the
intruder knows both nonces, that were supposed to be a secret between the client and the
server. Here is the attack in Alice-Bob notation, where pk(A) is the public key of A, and
IntruderClient is simply to say ”Intruder, but pretending to be Client:

4.1
Future Work
The Tamarin Prover offers no guarantees in terms of actually being able to complete its
proofs. In some cases, it may not even be able to complete the precomputation phase,
before starting on the proofs themselves. This is, in most ways, due to how the protocol
has been specified, and therefore both the size and complexity of the transition system the
protocol models. In this paper a few of the design choices of the translator were made in
an attempt to improve the likelihood of completing the proof analysis, but it has not been
a focal point. In the Tamarin manual, there are descriptions of multiple ways of further
specifying a protocol, such as to increase the likelihood of getting a useful analysis. It
could be interesting to investigate possible options to automatically further optimise and
enrich translations, in order to make the most of this translation tool.

5
Conclusion
This paper has connected the transaction-style security protocol specification Ultramarin
to the rule-based specification language of Tamarin, in turn enabling Ultramarin specifi-
cations to eventually be tested by the Tamarin Prover, once they have been translated to
Tamarin. This further expands the practical use of Ultramarin. This was done by design-
ing and implementing a transcompiler, that can translate Ultramarin protocol specifica-
tions into Tamarin. For this purpose, this paper also designed an extension to Ultramarin
in form of goals in section 3.1, which allows for translators to translate an even wider
range of security properties, which in turn allows for the associated tools to test this wider
range of properties. This can be especially useful when a protocol either has a very specific
security property it needs to satisfy, or if the pre-defined options for security properties
cannot be directly applied to the protocol specification in question.
The translator was designed in section 3.2, defining semantics functions for converting
the structure of Ultramarin into the rule-based Tamarin structure, whilst still modelling
the same behaviour and information. Ultramarin has indirectly been enhanced with the
semantics of Tamarin, as the translator accepts the use of the builtin functions of Tamarin,
used in Ultramarin specifications.
Ultramarin specifications of the Needham-Shcroeder (NSPK) and Needham-Scroeder-
Lowe (NSL) has been written and used to test the translator, by running the Tamarin
Prover on the protocols, after they were translated to Tamarin. It was expected that
Tamarin could find a certain attack on the NSPK protocol, which it did, while it could not
find that on NSL, as anticipated. This means translated protocols can not only be parsed
by the Tamarin Prover, but also be properly verified. Future work with this transcompiler
should focus on optimisation, increasing the likelihood of the Tamarin Prover being able
to automatically complete its proofs.
