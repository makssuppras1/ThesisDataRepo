Abstract
The project is motivated by Danmarks Radio’s (DR) need for journalistic oversight to en-
sure compliance with public service obligations such as diversity, impartiality, and geo-
graphical reach. To address this need, an automated system for Named Entity Recogni-
tion (NER) and Relation Extraction (RE) was developed to provide a clear overview of how
web articles align with these standards. By using both open-source and closed-source
Large Language Models (LLMs), specifically an LLM version based on Mistral-7B and
OpenAI’s models, different annotation methods are explored.
The project formulates NER and RE tasks as Machine Reading Comprehension (MRC)
problems, which improves flexibility and efficiency. A recent technique from 2023, Direct
Preference Optimization (DPO), was applied to fine-tune the LLM based on Mistral-7B’s
architecture for the specific NER and RE annotation tasks, with the purpose to enhance
its performance. Additionally, the project studies the ability of the open-source models to
generate valid JSON responses using structured output generation techniques.
While OpenAI’s models show better performance, the open-source alternative, with an
estimated size below 1% of GPT-4, shows promising results, highlighting the potential for
further improvements and making it a viable solution if full control over the toolchain is
desired.

Introduction
1.1
Background and Motivation
As a public service broadcaster, Danmarks Radio (DR) is obliged to deliver high jour-
nalistic quality across multiple platforms while complying with certain requirements such
as diversity, impartiality, and broad geographical reach [Kulturministeriet, 2022]. One of
these platforms is DR’s website. In order to get an overview of how the web articles align
with these requirements, a largely manual task to classify the articles into the different
categories is performed. Needless to say it is a tedious and time-consuming task, prone
to errors.
1.2
Project’s Aim
Motivated by the above mentioned need of a journalistic overview, the aim of the project
is to develop an automated analysis system for categorizing articles, providing DR with
a journalistic overview of their work. The system shall specifically be able to identify
the different entities in the articles, a task referred to as NER. In this context, named
entities are not only restricted to names of persons or organizations but include other
expressions representing meaningful elements within the text. To provide deeper insights,
a RE task shall also be included in the project to identify the relationships between entities
and quotes in the articles, determining their significance within the context.

1.4
Rationale for Selection of Methods
In this section, the rationale behind the selection of the methods in the project is presented,
serving as an introduction to the subsequent chapters.
Earlier NER methods
Perhaps the simplest approach for NER tasks is the dictionary-based method which relies
on a predefined list of words. Because the list is static, the method is not able to adapt
to unseen entities. Rule-based methods suffer from a similar problem, as they require
updating the set of rules to handle new patterns. More ”classical” machine learning algo-
rithms, such as those based on Hidden Markov Models (HMMs), treat the NER task as a
sequence labeling task, where each input word in the sequence is assigned a label [Ju-
rafsky and Martin, 2024]. These HMMs-based algorithms have shown to be as accurate
as more complex methods, and comparable to human performance.
Transfomer-based LLMs
A natural question arises: why use a more complex method if a HMM method already
achieves human-like performance? The complex methods, such as those based on trans-
formers (described in Chapter 3) like OpenAI’s GPT-4 and Google’s Gemini, offer several
advantages. They are in fact LLMs but they are so capable that they can be used for NER
tasks. Without going into details here, transformers-based methods are more flexible and
are able to handle sequential data and relations more effectively. They can be pre-trained
to understand a language and can then be fine-tuned for a specific downstream task. In
contrast, HMM methods are typically task-specific and need to be trained from scratch
for different tasks. The ever-evolving LLM frameworks allow to leverage the benefits of
transformer-based methods with much less effort compared to a few years ago.
Formulation of NER+RE tasks as a MRC task
The capability of transformer-based LLMs allows to formulate the NER task as a MRC
problem, more specifically as an extractive QA, where a prompt is issued to the LLM and
in turn, a response is received containing the requested NER entities. A prompt example
could be: ”Find and extract all the persons (PER) in this Danish article: ’Det var ikke en
tur i parken, men Caroline Wozniacki...’” and the response would be in this case ”Caroline
Wozniacki”. Thanks to the capability of the LLMs the complexity is encapsulated within the
LLM and the code for the extraction logic is reduced to an agent containing the descriptive
instructions. In the same vein, the RE task can be reformulated within the MRC framework
by issuing a specific prompt to extract the relationships between the entities. A prompt
could for example be: ”Find and extract the source of this quote: ’Jeg følte, at hun holdt
et...’ from this article: ’Det var ikke en tur i parken, men Caroline Wozniacki...’”. The
model is then expected to return ”Caroline Wozniacki”.
Open-source vs. Closed-sourceLLMs: (DK-Mistral-7B vs. GPT-4o)
LLMs can be broadly categorized into open-source and closed-source models based on
the availability of their source code. Each type comes with its own set of advantages and
disadvantages. Open-source LLMs can provide full ownership of the entire toolchain, re-
sulting in complete control over privacy during training and during inference. If the frame-
work used with the LLM relies solely on open-source packages, then there is also full code
transparency allowing the responsible to decide when and whether to update certain parts
of the codebase. However ownership and freedom come with a price. Time and technical

skills are needed to set up the entire pipeline. A scalable infrastructure is also needed,
which should adapt to the required throughput.
Without compromising most of the benefits provided by an open-source LLM, larger cor-
porations like Google, Amazon and Microsoft, offer cloud solutions that require minimal
setup and simplify the deployment.
On the other hand, if one is interested in current state-of-the-art LLMs, there is no other

model.

Structured Output Generation
A topic which has received large attention in the applications of LLMs is the ability to guide
them to generate structured outputs. Some of the most known projects include Outlines
[dottxt.co, 2024], Guidance [guidance, 2024] and SGLang [LMSYS ORG, 2024]. The justifi-
cation of these projects arises because, simply put, the LLMs primarily learn to predict the
next ”word” in a sequence and consequently are not inherently able to follow e.g. a JSON

output format. These tools guide the LLMs to comply with specific grammatical structures.
The applications are numerous since a structured output enables LLMs to communicate
with software programs in a simple and efficient way. Recently, OpenAI launched a so-
called JSON mode, [OpenAI (JSON), 2024], which enables the GPT models to comply
with JSON grammar.
Besides performing the NER and RE tasks to achieve the project aim, enforcing JSON

2.1
Annotation based on OpenAI’s LLMs
Figure 2.1 illustrates the architecture designed to annotate the articles using OpenAI’s
models. Starting from the left, DR’s dataset can be conceptually understood as divided
in two parts: the articles to be annotated, and the (target) JSON structures containing the
annotated information. Only the articles are processed by the Annotator Agent, which
also fetches the annotation instructions.
Figure 2.1: Architecture to annotate articles based on OpenAI’s LLMs.
The annotator agent constructs a prompt for each article, incorporating the annotation
instructions and the article itself. Each prompt is then sent to the LLM through OpenAI’s
API, which returns a response for each article. If successful, the response contains a
valid (predicted) JSON structure with the requested annotation information.
After annotating the articles, the Evaluator compares the target and predicted JSON
structures and generates a report. Since the information to be evaluated not only involves
multiclass classification but also extracting strings of varying lengths (such as names and
quotes), heuristics are applied where the degree of exactitude can be specified. For

Thanks to OpenAI’s API and the capabilities of the GPT models, along with the prior

2.2
Annotation based on DK-Mistral-7B
The overall purpose of the annotation task is the same as described in the previous section
and hence they share common elements. The architecture begins similarly with DR’s
dataset, divided into articles and target JSON structures, as shown in Figure 2.2.
Figure 2.2: Architecture using open-source LLM for annotation.
The articles are retrieved by the Annotator Agent, but here the process differs. GPT-
4 is estimated to have nearly 2 trillion parameters [Wikipedia (GPT-4), 2024] while DK-
Mistral-7B, with its 7 billion parameters, is below 1% of GPT-4’s size. Given the rela-
tively small size of DK-Mistral-7B, it is more efficient to divide the annotation queries into
smaller parts. Hence the relevant set of instructions needs to be fetched depending on
the labels to be annotated. Moreover, the appropriate regex (regular expression) string
also needs to be generated based on the actual label class structure. Recall that the LLM
does not have inherent capabilities of returning JSON-formatted responses, and for this
reason, the model needs to be guided based on the regex strings.
The query is passed to SGLang’s API and is sent to SGLang’s backend. The SGLang’s
backend receives the query and passes it further to the LLM.
Once the query is processed by the LLM, a response is received through SGLang’s API
back into the Annotator agent. In there, the multiple JSON-formatted fragments are
validated and then joined to form a complete valid (predicted) JSON structure with the
requested annotation information for the article in question.

2.3
Fine-tuning DK-Mistral-7B
The third and final architecture component is related to DK-Mistral-7B’s fine-tuning.
While the LLM is already versatile and capable of generating text, to enhance its perfor-
mance on the specific annotation task, it is advantageous to fine-tune it for said purpose.
Recall that for clarity the not fine-tuned model is referred to as base-DK-Mistral-7B and
the fine-tuned model is referred to as FT-DK-Mistral-7B.
Figure 2.3: Architecture to fine-tune FT-DK-Mistral-7B using Direct Preference Opti-
mization (DPO).
There are several ways to fine-tune an LLM, one of which is Reinforcement Learning
from Human Feedback (RLHF), where an intelligent agent (the LLM to be fine-tuned in
this context) is aligned to human preferences. However having a human in the loop under-
mines the purpose of the project which is precisely to develop an automated system that
can perform annotation tasks without human intervention. A recent paradigm to fine-tune
LLMs from preferences without reinforcement learning is using DPO, which solves the
standard RLHF problem with only a simple classification loss [Rafailov et al., 2023]. The
optimization problem is converted to a binary classification task, where the model aims
to maximize the likelihood of the chosen responses over the rejected ones. In practice
DPO works by comparing two versions of the model’s output and adjusting the model to
prefer the chosen answer and to move away from the rejected answer. With virtually no
tuning of hyperparameters, DPO is reported to perform similarly or better than existing
RLHF algorithms.
To construct a DPO dataset, the chosen (correct) answers are retrieved from the ex-
isting manually annotated DR dataset, see Figure 2.3. The rejected (wrong) answers
are fetched from the outputs of base-Mistral-7B, as a result of the annotation process
described in the previous section. Since not all the answers are wrong, to ensure a com-
plete set of rejected answers, the missing rejected answers are automatically generated
through string manipulation techniques. This automated process ensures that the project
remains fully automated, aligning with the goal of eliminating manual intervention. During
training, the weights of FT-DK-Mistral-7B are adjusted such that the likelihood of chosen
responses is maximized, while the likelihood of the rejected responses is minimized. To
prevent FT-DK-Mistral-7B to forget the originally learned language capabilities, base-
DK-Mistral-7B is included in the training loop as reference.

Methodology
The objective of this chapter is to describe the methods used in the project. However,
it is not the intention to give a detailed historic overview of the methods (for a detailed
background, the reader is referred to available literature on the topics, see e.g. [Mur-
phy, 2022] and [Jurafsky and Martin, 2024]). Instead, this chapter focuses on how these
general methods apply specifically to achieve the goals described in Section 1.2. Note
that in this chapter, the terms words and tokens are used interchangeably for ease of
understanding.
3.1
Natural Language Processing
For a language model to perform NER and RE tasks successfully, it needs to handle the
complexity of human language and understand subtle nuances. Given the sentence: ”I
took a siesta because I was very...”, the model would assign a probability to every possible
next word, with the most likely next word being <tired> rather than <fantastic>. Such a
model, which assigns probabilities to the next words, is referred to as a language model.
n-gram
The most basic language model is the n-gram language model [Shannon, 1948], which
estimates the probability of the nth word given the previous n-1 words. Its simplest version
is the bigram language model, which only relies on the preceding word to estimate the
next one. Using the former sentence as an example, the model would then calculate the
following conditional probability:
P(<tired>|<very>)
(3.1)
In a very simplistic case where the corpus is only formed by one additional sentence:
”Deep learning is very interesting”, the probability could be estimated with the Maximum
Likelihood Estimation (MLE). For the bigram model, MLE calculates the probability of a
word following another word based on their count or frequency in the training data (the
corpus of two sentences previously introduced):
P(<tired>|<very>) = C([<very> <tired>])
C(<very>)
= 1

Architecture of Mistral-7B
As already mentioned in Section 1.4, the selected open-source LLM is DK-Mistral-7B,
therefore it is appropriate to review Mistral-7B’s architecture, [Jiang et al., 2023], and
describe the functionality of the each component.
Figure 3.1: Architecture of Mistral-7B.
Mistral-7B LLM consists on 32 successive encoder blocks, which resembles the encoder
of the original autoregressive transformer architecture. Additionally, a linear layer and a
softmax layer are added on top, as depicted in Figure 3.1.
Tokenization
First, it should be mentioned that words are not the internal unit of representation. Instead
the sequence of words are first tokenized into manageable indexed units for processing.
In practice many tokens are words but the tokenizer can handle unseen words by break-
ing them down into subword units. For example, the word ”colder”, could be tokenized
into <cold> and the morpheme <##er>. The morpheme <##er> can then be used for
other unseen comparative words. For the sake of clarity, assume in the following that the

sequence of tokens is [<the> <cat> <sat> <on> <a> mat>].
Input embeddings
The input to the input embeddings block is in the example a sequence of indexes of
length seq = 6, where each index corresponds to a specific token in the vocabulary. The
embeddings translate the tokens into vectors in a continuous vector space of dimension
dm = 4096, where similar tokens are closer together, helping the models to understand
and capture the semantic meaning of tokens. Each token is converted to a static em-
bedding vector, xi ∈R1×dm, using a pre-trained embedding matrix, that is, there is a
one-to-one mapping between the tokens and the input embeddings.
Normalization
As the name indicates, the normalization block normalizes the embeddings. It is a com-
mon technique employed in deep neural networks that helps to stabilize and accelerate
the training process. For further details about the normalization in Mistral-7B, refer to
[Zhang and Sennrich, 2019]. For simplicity, let the normalized matrix containing the input
embeddings be X ∈Rseq×dm.

Rotary Positional Encodings
As shown in Figure 3.1, the matrices Q and K include positional information. Language
models, before transformer-based models, were sequential and therefore the models had
a notion of how distant the words were from each other. Because the transformer-based
models are not inherently sequential, a method is needed so the language model is aware
of the position of the words. In Mistral-7B, the rotary positional encoding is used, which
in essence measures the distance between the tokens [Su et al., 2023]. The consequence
and purpose of the positional encoding is the reduction of the intensity of the relationship
between the words, the further apart they are in the input sequence.
Feed Forward with SiLU
The main purpose of the feed forward layer, together with the Sigmoid Linear Unit (SiLU)
activation function, is to introduce non-linearities in the architecture to allow the model to
capture more complex relations in the text.
Linear + Softmax Layers: Generation of Next Token
After normalizing the output of the 32nd encoder block, the resulting embedding matrix of
size Rseq×dm, contains dense representations of the input tokens. These output embed-
dings have the same dimensions as the input embeddings. While the input embeddings
provide a static representation of each token, capturing its basic meaning, the output em-
beddings also include positional information and interactions with all the other tokens in
the sequence. This representation is then passed through a linear layer that maps the

output embeddings to the size of the vocabulary, associating a logit value with each token
in the vocabulary. Finally, a softmax layer is applied to predict the next token in the se-
quence by converting the logits into a probability distribution over the entire vocabulary.
One simple method for selecting the next token is greedy sampling, which chooses the
token with the highest probability.
3.3
Operational Aspects of Mistral-7B
In this section, key operational aspects of Mistral-7B are presented to provide a practical
understanding of its functionality which serve to better understand how Mistral-7B can
be guided to comply with a specific grammar.
Training Mistral-7B
A way to train the model is with the next token prediction task, where the model is given
a prompt and the goal of the model is to predict the next token. For example (again
assuming that words are tokens for simplicity), given the input prompt:
<SOS> <The> <quick> <brown> <fox> <jumps> <over> <the> <lazy> <dog>
the target sequence would be:
<The> <quick> <brown> <fox> <jumps> <over> <the> <lazy> <EOS>
The model is trained to learn a mapping between input and output, so when it sees <SOS>
it is expected to generate <The>. When it sees <The>, it is expected to generate <quick>
and so forth. The special token <SOS> is prepended to the sentence to indicate the start of
sentence and when it sees <EOS>, the model is expected to stop the generation because it
has reached the end of sentence. Once the predicted tokens are obtained, the cross en-
tropy loss is calculated and by running back-propagation the weights are updated. Thanks
to the self-attention mechanism, the training process can be parallelized.
Inference with Mistral-7B
For inference, the process runs for as many steps as tokens are to be generated/predicted.
The input sequence would be <SOS> and the model is expected to predict <The> because
it has learned the mapping during training. <The> is appended to the input sequence so
now the input sequence is [<SOS> <The>] and the model is expected to predict <quick>.
The process is repeated until the <EOS> is reached.
KV cache
At every step in the inference, the interesting token is only the last one because the previ-
ous tokens have been generated and are therefore already known. If the model is given
the task of predicting the next token of the sentence
<The> <quick> <brown> <fox> <jumps> <over> <the> <lazy>
it is not sufficient to give the token <lazy> as the prompt. The same would apply for an
n-gram model. The conditional probabilities are not the same:
P(<next token>|<lazy>) ̸=
P(<next token>|[<The> <quick> <brown> <fox> <jumps> <over> <the> <lazy>])
For a transformer-based model, the conditional probabilities of the next token are affected,
among other factors, by the matrices Q, K and V in the self-attention mechanism, which
form the context for the next token. The calculations cannot be skipped but are repeated.

Rolling Buffer Cache
Note that not all the dot products need to be calculated and stored due to the use of
the causal mask, otherwise the requirements of the KV-cache would in principle increase
quadratically with respect to the length of the input sequence.
As explained and illustrated in Figure 3.3, Mistral-7B makes use of the sliding window
method which can be further exploited by the KV cache strategy to only store the com-
putations for the period covered by the sliding window. Older computations, which are
outside the sliding window, can be deleted/overwritten.
Prefill and chunking
For the KV-cache to become functional, it first needs to be built up. Because the input
prompt is known in advance, its tokens are fetched to prefill the KV-cache. Adding one
token at a time would be a slow operation (sequential execution). On the other hand, a
fully parallel execution, where all the tokens from the prompt are fetched, is not usually
possible due to limited memory resources in the GPU. The solution is to divide the prompt
in chunks corresponding to the length of the sliding window, as shown in the upper part
of Figure 3.5.
Initially the KV-cache is empty and the QKT matrix uses the tokens in chunk 1. After-
wards, the next chunk is fetched, the QKT matrix is calculated for the current chunk but
is expanded to also include the results from the previous chunk. This expansion is nec-
essary to ensure building the self-attention between chunks, otherwise the first token in
chunk 2, <a>, would have no context and the self-attention mechanism would fail to find
a relationship to the previous tokens [<cat> <sat> <on>], even if they are in the sliding
window.

3.5
Guiding LLMs: JSON grammar
Some concepts have been described in the first sections on how LLMs work, especially
applied to Mistral-7B. The model can be trained to learn a language and can be used
afterwards to generate text based on the next token prediction task. There are different
methods to generate the next token but normally, they rely on conditional probabilities.
The question is how to enforce a grammar, more specifically, how to guide the model to
follow a grammar such that its response is in a valid JSON format, at the same time it
performs the requested NER+RE annotations tasks.
Logit bias (token masking)
As earlier described, the generation of the next token involves processing the output of the
32nd encoder block with a linear layer and softmax layer. The linear layer transforms the
encoder output into logits, which are real-valued scores for each token in the vocabulary,
logit ∈R. The softmax then maps these logit scores R to probabilities: [0, 1], so as an
example, a logit with a −∞score, is mapped into a probability of 0.
During inference, if the input prompt is:
<The> <quick> <brown> <fox> <jumps> <over> <the> <lazy>
all the 32000 logits but one (the one corresponding to <dog>) could be biased to a lower
value or set to −∞(effectively masking them). Consequently, the softmax output would
assign a low or zero probability to all other tokens and <dog> would end up having the
highest probability. This is the main principle used to guide/force LLMs to generate spe-
cific tokens.

the quote (only forename and in lower case), and FUN , the function of the quoted entity
(either ”Amateur” or ”Professional”). The prompt could be worded as shown here:

The jump-forward algorithm examines the FSM of the specified regex string to identify all
the singular transition edges, that is, the places where the model does not have to make
a choice because the task is bounded. Then all the consecutive edges are compressed
together into a singular path. Instead of decoding the singular paths token by token,
they are prefilled, jumping forward until the next branching/decision point. Recall that the
tokens in the singular paths can be processed in parallel by the model, improving the
performance of the algorithm compared to the first presented where the FSMs are not
compressed.

Tokenization boundaries
So far, the included examples have served to present the methods in an accessible way.
However, when it comes to guiding a model with a specific grammar (like JSON-formatting
in this project), all the methods face a common challenge, which is dealing with the tok-
enization boundaries.
Suppose now that the model is allowed to add multiple entries to SRC and that the regex
instructions are modified accordingly. After decoding a name, the model has two choices:
to close the list or to add a new entry to SRC . Assuming that the token <",> is more
frequent/probable than <">, after returning <",> there is no other option in the FSM than
guiding the model to generate a space followed by a quotation mark, < ">, to add a new
entry. After the new entry is added, the model selects <",> again over the less probable
token <">, leading to an endless loop where new entries are added to the SRC list without
being able to stop.
To address this issue, SGLang makes use of a retokenization mechanism to better align
with the desired output format. For instance, the token <",> can be split into two separate
tokens, <"> and <,>, which would allow the FSM to handle each token independently,
preventing the model from entering an endless loop. In practice, when the SRC class is
defined using pydantic, [Pydantic, 2024], a constrained list can be defined to limit the
maximum number of entries in SRC .
In conclusion, using SGLang can in principle guarantee that the response from the model
is JSON compliant. What SGLang cannot guarantee is that the decoded content (the an-
notated information) is correct. The next step is therefore to ensure that the content gen-
erated by the model, is not only JSON compliant but also accurate. This is where training
the model comes into play.

To better understand the DPO technique in the next section, it is helpful to introduce the
basic concepts of how Reinforcement Learning from Human Feedback (RLHF) works.
RLHF involves humans providing direct feedback on the outputs generated by the lan-
guage model. This feedback loop allows the model to adjust its behavior to align more
closely with human preferences and expectations.
Figure 3.8 summarizes how RLHF works. The dataset is composed by multiple prompts
which both the reference LLM (pre-trained model) and the policy LLM (model being fine-
tuned) are asked to respond to. The reward model evaluates the answer of the policy LLM
and together with the DKL(·) term (which ensures that the policy LLM does not deviate
significantly from the reference LLM) they constitute the loss. Using backpropagation,
the gradients of the loss are computed, which are needed to update and optimize the
parameters of the policy LLM.

Discussion
The discussion focuses on selected main points of interest from the outcome of the project.
These points are chosen to provide a clear understanding of the most important aspects
encountered in the NER+RE annotation tasks. Although not every detail is examined, the
selected subjects should give a nuanced understanding of the key findings and challenges
addressed throughout the project.
Quality of DR’s dataset
The statistical analysis of DR’s dataset in Section 3.4 was helpful in crafting suitable
prompts and setting sensible parameter values for the different components, such as
the annotator agents and the DPO algorithm. However, the process of crafting prompts,
analyzing the behaviour of different LLMs, and comparing predicted responses against
the target responses, provided deeper insights into DR’s dataset. In that activity, certain
anomalies in the target responses were discovered. Some examples from the training
dataset include:
• ARTICLE[019], QUOTE[12]:
SRC is missing. Should have been ”Andrea Chris-
tensen”.
• ARTICLE[079]: ”Eurovision” is not labeled as ORG .
• ARTICLE[100]: The word ”skal” (shall in English) is wrongly tagged as an organiza-
tion.
• ARTICLE[111]: The same person is labeled with her family name and with her full
name. When she is quoted with her family name, SRC is empty.
The above list is not exhaustive. Only a few examples were included but it illustrates very
well the motivation of the project, described in Section 1.1 where manual classification
and labeling of the articles is a task prone to errors. Furthermore while it cannot serve
as the explanation of why FT-DK-Mistral-7B does not achieve better performance, the
correctness of the training dataset is a factor to consider and stresses the importance to
have a training dataset cleaned for errors.
Class imbalance and annotation categories
A topic not specifically addressed during the fine-tuning of the model with the DPO al-
gorithm is the class imbalance. While an article has one list of PER and ORG , it may
have up to 17 quotes (as seen in the training dataset). This means that the nested cate-
gories within the quotes, such as GEN and FUN , are overrepresented compared to PER
and ORG when fine-tuning with DPO. Although no direct evidence was found that class
imbalance was a problem, it is something to consider.
Another issue is FT-DK-Mistral-7B’s relatively low ability (with any β value) to identify and
extract quotes. More effort is needed to improve this because the quality of the nested
categories’ information depends on correct quote extraction. The logic developed allows
to select specific categories to be fine-tuning with DPO, making it easier to observe the
effects of targeting the fine-tuning process towards a specific category type. Addition-
ally, once the model is fine-tuned, extra flexibility is introduced during the annotation of
the articles. Specific categories can be bypassed, with the annotator fetching the target
responses instead of the model’s responses. It is an efficient method that enables the

33
==PAGE:44==
assessment of the model’s performance in annotating nested categories, assuming e.g.
an ideal quote extractor.
Guiding LLMs
A subject not treated in the project but mentioned in [dottxt.co, 2024], is how guiding an
LLM may affect the content of the response or the conditional probabilities of the sub-
sequent tokens to be generated. Depending on how the regex strings are constructed,
together with the combination of state transitions, and how words are split into tokens,
there are different paths to the same answer. Each path has different conditional proba-
bilities, which end up affecting the probabilities of subsequent tokens.
In theory, it is possible to access the logit values and probability distributions after the
hidden states in the last decoder layer to analyze how they may be affected by different
state transitions. The guiding frameworks work because they affect the probabilities by
applying a logit bias. However, the more restrictions are applied, the less freedom the
LLM has to generate responses, which may not always be beneficial.
It is an interesting topic but out of scope for the current project.
Optimization Challenges in DPO
In RLHF’s objective function, the parameter β allows to find a balance between obtaining
rewards rϕ(x, y) from the reward model, and staying close to the reference policy πref,
where staying close is measured using the Kullback – Leibler divergence. DPO refor-
mulates RLHF’s objective function while retaining the parameter β. Although Kullback –
Leibler divergence is not directly used, a similar regularization effect is achieved with the
log ratio of probabilities. The effect of β was demonstrated when its value was too low.
The DPO algorithm would allow the policy πθ to deviate significantly from the reference
policy πref, causing πθ to become greedy searching for rewards. Numerous hallucinations
were observed as a consequence, rendering the trained policy unusable. By increasing
β, πθ was forced to stay closer to πref, resulting in a more reliable policy.
The results obtained from the fine-tuned Mistral-7B-based LLM showed that the open-
source approach is viable and that the concept works as intended, even though the current
set of parameters is not yet optimal. As seen from the logged traces with W&B, the training
process was unstable, with large spikes in the gradient norm. Reducing the learning
rate and increasing the gradient accumulation helped but did not eliminating the issue.
Therefore, some effort should be dedicated to find a better set of parameters to ensure
stable training and improved performance.
Prompt Crafting
There are generally two different but non-exclusive approaches to improve the perfor-
mance of NER+RE annotation tasks with LLMs. One approach is to fine-tune the LLM
for the specific purpose, where the model weights are adjusted. The other approach is
to manually craft better prompts to guide the model in a desired direction, relying more
on the quality of the prompts rather than the internal mechanism of the model. While it
is tempting to address very specific issues, such as ”also find children” to identify ”Elliot”
as a person (see the example in Section 4.2), prompt crafting alone may not always be
reliable. However, neither approach can stand alone effectively and better results are
achieved by combining prompt crafting and fine-tuning.
Learning a new behavior with DPO
One of the key features and advantages of the DPO algorithm is that the model can
learn from both the accepted and rejected responses. This is especially useful because
the rejected responses can be used to correct undesirable behaviors. As shown in the

associate SRC with a person and never with a URL.
Technical Insights and Reflections
As explained in Section 1.4, one of the main benefits of open-source LLMs is the full
ownership of the entire toolchain, which is particularly relevant for maintaining complete
control over privacy. However, this approach requires significant technical expertise. Ex-
ploring both open-source and closed-source solutions provided valuable insights into their
benefits and challenges. The project was intentionally kept free from implementation de-
tails to focus on its main objectives and avoid diving into numerous practical details. Con-
sequently, it may give the impression that the setup was straightforward. In reality, most
of the time was spent on the open-source LLM, including setting up a virtual machines,
ensuring that the correct combination of software libraries and drivers were installed, fit-
ting Mistral-7B in the GPU, etc. It is important to note that choosing the open-source
approach requires significantly more effort.

Conclusion
In the project an automated Named Entity Recognition (NER)+Relation Extraction (RE)
solution was developed to help Danmarks Radio (DR) gain an overview of journalistic
parameters, enabling them to better understand their compliance with public service obli-
gations such as diversity, impartiality, and geographical reach. Two routes were explored
for this solution. One approach was based on closed-source LLMs. Here, OpenAI’s mod-
els were used, with GPT-4 omni serving as a baseline. The functionality was accessed via
OpenAI’s API, removing the programming complexity. The other approach involved using
an open-source LLM, giving the possibility to have full ownership of the entire toolchain.
In this case, an LLM based on Mistral-7B’s architecture was used, more specifically,
munin-neuralbeagle-7b, which is a Danish capable LLM.
Three main components were developed to achieve the aims of the project. First an anno-
tator agent was built to enable OpenAI’s models to perform the NER+RE annotation tasks,
returning valid JSON responses. It turned out to be a relatively straightforward undertak-
ing because the programming complexity was hidden behind OpenAI’s API. An evaluator
was also built to compare the generated responses from OpenAI’s models against the
manually annotated articles by DR.
The second component involved developing the corresponding annotator agent to anno-
tate the articles with the Mistral-7B-based LLM. This was a more complex task. Firstly,
additional logic was needed to craft specific prompts where the instructions had to be
adapted depending on the identified entities. Secondly, the LLM had to be guided to
generate valid JSON responses. The SGLang framework was utilized for this purpose,
enabling the LLM to reliably return structured JSON outputs.
Finally, once having an annotation framework where the Mistral-7B-based LLM was
able to return valid JSON responses, it was deemed necessary to increase its annotation
performance. The LLM was able to generate valid JSON responses but the annotated
information was not of sufficient quality.
A recent technique to fine-tune LLMs, called Direct Preference Optimization (DPO) was
successfully applied. It reformulates RLHF’s objective into a simpler objective where train-
ing a reward model (to convert the qualitative human feedback into quantitative preference
scores) is not necessary. A training dataset was constructed, which besides containing
the chosen/correct answers, also contained the rejected/wrong answers generated by
Mistral-7B-based LLM during the annotation process. Hence, it was possible to divert
the preference of the pre-trained LLM toward unwanted behaviour and instead, during the
fine-tuning/training process, give rewards when the correct answer was generated. It
proved to be an effective algorithm, being β the main hyper-parameter to tune. It allowed
to balance the search to get rewards but without deviating significantly from the initial
model (reference policy).
The results obtained from the fine-tuned Mistral-7B-based LLM showed that the open-
source approach is viable and that the concept works as intended, even though the current
set of parameters is not yet optimal. Additional optimization, such as finding a better set
of parameters, could help reduce the gradient norm during training and further improve
the quality of the annotation results.
