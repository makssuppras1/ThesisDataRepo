Abstract (English)
The goal of the thesis is to develop guidelines for systematically selecting a cross-
platform mobile application development framework based on the features and
characteristics of selected frameworks. To establish these guidelines, the thesis
selects four frameworks for testing. An example application, with features from
already existing popular applications in selected categories, is deﬁned and de-
veloped in each of the four frameworks. These example applications, and the
experience gathered when developing them, are used for evaluating the frame-
works based on a set of common evaluation criteria. Lastly, the results from
the evaluations, and the characteristics of the frameworks, are used to estab-
lish the guidelines for systematically selecting cross-platform mobile application
development frameworks based on requirements from the developer and the ap-
plication that is to be developed.

Chapter 1
Introduction
Since Apple revolutionized the phone industry, releasing the ﬁrst iPhone back
in 2007, a lot of diﬀerent operating systems have been trying to establish them-
selves on the market such as iOS, Android, Symbian OS, Windows Phone, and
BlackBerry OS. Today, the two dominating operating systems are Apple’s iOS
and Google’s Android OS with a combined worldwide market share of 97.30 %
of mobile operating systems [W1] and 99.69 % of tablet operating systems [W2]
as of January 2019.
When developing mobile applications, it is often beneﬁcial to make the appli-
cation available on both iOS and Android OS. Previously, this either meant
developing two separate mobile applications, which is increasing the develop-
ment and maintenance cost, or developing mobile web applications to minimize
native code, trading oﬀperformance, user-experience, and access to platform
speciﬁc features such as sensors [1].
Because iOS apps are developed in Objective-C or Swift [W3] and Android
apps are developed in Java or Kotlin [W4], developing two applications is not
only time consuming, it also requires diﬀerent skill sets [2].
Cross-platform
development frameworks for developing mobile applications started to emerge
as products developed by startup companies in the early 2009. The ﬁrst ones
to gain widespread attention were PhoneGap [W5, W6] and RhoMobile [W7,
==PAGE:12==
2
Introduction
W8]. The promising ones were later bought by major technology companies, as
for the above mentioned Adobe and Motorola respectively.
As of now, many major technology companies have a cross-platform development
framework of their own. Some of the notable ones today are: Xamarin [W9] (Mi-
crosoft), React Native [W10] (Facebook), Flutter [W11] (Google), PhoneGap
(Adobe), and NativeScript [W12] (Progress). All these frameworks are devel-
oped and maintained by multi-million dollar software companies [W13]. The
main beneﬁt, is that by choosing a cross-platform development framework, it is
possible to achieve a high amount of code sharing between the applications for
the platforms, which leads to a shorter time to market and a lower maintenance
cost [3].
The approach used to achieve cross-platform compatibility varies for each frame-
works. Some of the frameworks build webview-based applications with access
to native API’s, also called hybrid applications.
This is the approach taken
by PhoneGap.
This type of frameworks promote code sharing to the point
where the code can not only be shared between mobile platforms, but also with
web applications. Frameworks such as Xamarin and React Native, build native
component-based applications, making them practically impossible to distin-
guish from natively developed applications.
This increases the performance
compared to hybrid applications, but in some cases requires the developer to
implement platform speciﬁc logic.
Even within these classiﬁcations every framework has its own methods for
achieving cross-platform compatibility and therefore each framework has dif-
ferent strengths and weaknesses. Which framework to choose depends on the
speciﬁc application.
1.1
Problem Statement
The increasing number of options available for building cross-platform mobile
applications makes it diﬃcult for developers to keep up with the strengths
and weaknesses of each framework. Furthermore, developing successful cross-
platform mobile applications with shared codebases for iOS and Android re-
quires experience in the particular framework used. As a result of this, develop-
ers tend to choose a framework they are experienced in, which might not be ideal
for the particular application. This can lead to poor user experience, prolonged
time to market, and a high maintenance cost. Currently, there are no guidelines
for systematically selecting the best suited framework, which covers the newest

and most promising frameworks. This is a problem because developers are not
always aware of the frameworks that exist and that might lead to developing an
application in a framework that does not support, or is not optimal for the type
of application that is to be developed. As mentioned this can lead to poor user
experience, prolonged time to market, and a high maintenance cost.
1.2
Objective
The aim of this research is to develop guidelines for systematically selecting
cross-platform mobile application frameworks for iOS and Android, based on
their features and characteristics. These guidelines will help developers with
limited prior experience in mobile application development, to systematically
select the best suitable cross-platform mobile framework for their application
and the developer’s individual experience and preferences. It will also guide
experienced application developers who wish to expand their knowledge in dif-
ferent frameworks. By assisting developers in systematically selecting a cross-
platform development framework for an application, the developed applications
will be easier to maintain, and developers will have the best starting point for
achieving the speciﬁc requirements of that application. Research that compares
cross-platform development frameworks does exists, but does not include the
most promising frameworks at the time of writing (see Chapter 2), which this
study will focus on.
1.3
Method
This study explores some of the diﬀerent frameworks available for developing
cross-platform mobile applications for iOS and Android. To determine which
frameworks to study, the frameworks will be categorized and selected from this
categorization. A mobile application will be developed in each of the selected
frameworks, and a cloud server will be providing the applications with data
to process and display. To determine which application will serve as the ex-
ample application, this study will categorize diﬀerent types of applications and
systematically select the requirements and features from this categorization.
The applications and the development processes will be evaluated based on
commonly established evaluation criteria. The research will consider functional
characteristics of the applications such as performance, code sharing, eﬃciency,
and reliability, but will also cover aspects such as ease of development, and
required platform knowledge. In order to gain domain knowledge, this study

1.4
Results
The results will be presented in the form of two score-sheets. One score-sheet

presents the guidelines for the respective developer and application proﬁle.

Chapter 2
Related Work
This chapter presents some of the relevant related work. The ﬁndings from the
work are presented.
A paper by Andreas Sommer and Stephan Krusche from Technische Univer-
sität München was published in 2013 [6]. The paper compares cross-platform
frameworks by evaluating and weighing them against native SDKs. The pa-
per compares the frameworks: Titanium SDK, Rhodes, PhoneGap, and Sencha
Touch.
The paper deﬁnes a sample business application which is developed
in each of the frameworks. The paper then scores the frameworks based on
a common set of criteria. The paper ﬁnds the performance of the application
developed with cross-platform frameworks to be very poor compared to natively
developed applications. It recommends native applications for applications with
a focus on user experience, and for performance-critical applications. The paper
states that if these problem areas can be considered minor, it recommends to
employ either Titanium SDK or PhoneGap for production use, as these oﬀer
the best performance.
In 2013, a study was published at Department of Information Systems, Univer-
sity of Münster. The study was done by Hennig Heitkötter, Sebastian Hanschke,
and Tim A. Majchrzak [7]. The study aims to compare cross-platform develop-
ment approaches and compiles a set of criteria to assess these. The criteria are
used to compare Mobile Web apps, PhoneGap, Titanium Mobile, and natively
==PAGE:18==
8
Related Work
developed applications. The ﬁndings of the study are that Mobile Web Apps
has advantages in maturity and platform accessibility while the disadvantages
are the access to platform-speciﬁc features, the look and feel, and the lack of
distribution. For PhoneGap, the study ﬁnds it oﬀers access to platform-speciﬁc
features and development is easy while the visual appearance and the behavior
reﬂects more that of a website, than a native application. For Titanium Mobile,
it ﬁnds the look and feel is that of a native application, but the performance has
limitations. It also ﬁnds that a paid subscription is required for many advanced
features, and the required knowledge slows the development. For native appli-
cations, it ﬁnds they inherently look and feel native, but requires very speciﬁc
knowledge and applications has to be developed for each platform.
In 2015, another study was done at University of Münster by Tim A. Ma-
jchrzak, Jan Ernsting, and Herbert Kuchen [8]. This study focuses on devel-
oping model-driven cross-platform applications with the MD2 framework. The
study presents an adapted version of a categorization of cross-platform develop-
ment approaches from [9] which divides approaches into two diﬀerent categories
namely Runtime Environment and Generative Approaches, and divides these
two subcategories into more categories. The paper motivates the approach of
MDSD (Model-Driven Sofware-Development) as a cross-platform development
approach for developing business applications. The paper concludes that there
are merits for MDSD in application development, but it is not yet a dominant
technology. It is a ﬁeld that requires future work.
In 2017, a Master’s thesis was published at DTU, authored by Patrick Schøller
Larsen [10]. The thesis aims to establish the necessary knowledge about the
application being developed, and the developer, before choosing a development
tool. To carry out the study, three frameworks are chosen for test. The the-
sis explains and recommends frameworks suited for diﬀerent applications and
developers. The thesis categorizes applications and performs a comparison of
the diﬀerent frameworks in question. It focuses on the frameworks: Xamarin,
Appcelerator Titanium, and PhoneGap. At the time of this thesis these were
found to be the most beneﬁcial frameworks to compare. The thesis presents ex-
amples for developer proﬁles for each of the frameworks. One of the topics this
thesis mentions as future works is creating a tool that recommends development
tools based on application and developer information.

Chapter 4
Cross-Platform
Development Frameworks
This chapter categorizes and outlines the diﬀerent approaches within cross-
platform development for mobile applications. Furthermore, the cross-platform
development frameworks that are examined in this study are selected, based on
comparison criteria such as the approach, popularity, and the support of sensors.
4.1
Approaches
Within mobile application development there are several approaches for devel-
oping applications. A classiﬁcation of the diﬀerent approaches from [2], can be
seen in Figure 4.1.
The categorization divides application development into two main categories
namely Cross-Platform and Native, Platform-Speciﬁc. This study does not in-
clude native, platform-speciﬁc development and will not discuss this approach
any further, as this approach does not achieve cross-platform compatibility. This
study will focus on the Cross-Platform category, that allows code to be written
once and deployed across diﬀerent platforms.
==PAGE:32==
22
Cross-Platform Development Frameworks
Figure 4.1: Categorization of application development approaches from [2]
(adapted from [8], originally from [9])
The cross-platform category is divided into two subcategories, Runtime Envi-
ronment and Generative Approaches which are explained in more detail below.
Runtime Environment
The runtime environment category covers frameworks that abstract from the
native platform SDK by using a runtime environment to run the platform inde-
pendent code. This is typically done by either shipping a runtime environment
with the application package or utilizing a built-in runtime environment such
as a JavaScript engine, a browser, or Webviews. If the application has access to
native device components, the communication between the native context and
the platform independent environment is typically done via exchanging com-
mands over so-called bridges, or via injecting the native API’s into the runtime
environment.
Generative Approaches
Generative approaches cover frameworks that generate the platform dependent
code to run on the target platform. Frameworks in this category should techni-
cally perform just as well as native, platform-speciﬁc applications since they do
not make use of any runtime environment or abstraction from the native SDK.
Frameworks in this category rely more on it’s maintainers since every feature
==PAGE:33==
4.1 Approaches
23
from the native SDK needs to be rewritten in the transpiler or interpreter be-
fore it is usable in a platform independent context, and the performance of the
applications are heavily dependent on this implementation.
These two main categories are further divided into four and two subcategories
respectively. These categories are described more in detail below.
Mobile Web Apps
Mobile web apps are applications that solely rely on web development. They
run on any device with a web browser installed but are also limited to the
restricted capabilities of the browser, meaning no access to sensors or native UI
components. Applications in this category are not deployable to app stores.
This study will not cover mobile web apps, as this approach only uses traditional
web development techniques and does not create applications that are deployable
to app stores. These traits make mobile web application development similar to
web application development, which is outside the scope of this study.
Progressive Web Apps
Progressive web apps (PWA’s) are similar to mobile web apps because they run
inside a browser and rely on web development. However, PWA’s have access
to a limited set of native components as implemented by the browser.
The
recent announcement of progressive web apps by Google [W23] shows interesting
progress for this type of cross platform development, especially since Apple
added support for PWA’s in iOS 11.3 [W24]. The developments in this area
means that PWA’s now have access to sensors, geolocation, notiﬁcations, and
other features from the native API, and are even deployable to some app stores.
PWA’s are inherently small in size since they do not need to wrap a lot of
packages into the application package, hence organizations such as Twitter are
using PWA’s for their "Lite" editions [W25].
This study will not cover this type of cross-platform development, as it is inher-
ently diﬀerent from traditional cross-platform mobile application development
but similar to web application development, and it still remains to be seen if
this approach will be adopted broadly. For further reading on this approach see
[2].
==PAGE:34==
24
Cross-Platform Development Frameworks
Hybrids
Hybrids are web applications that come with a native packaging making them
deployable to app stores. The applications are in reality still web applications
as they run inside a Webview provided by the native SDK. The native pack-
aging means that hybrid applications can leverage a subset of the native API’s
depending on what has been made available by the speciﬁc framework used.
These applications are developed using HTML, CSS and JavaScript making
them quick and easy to develop. However, for the same reason they look and
feel like web applications making them easier to distinguish from natively de-
veloped applications.
Self-contained runtime
Applications in the category of self-contained runtimes ship with their own run-
time making them able to run platform independent code. This approach gains
access to the native API’s through bridges between the native API’s and the
runtime. Because of this, these applications are able to use native GUI com-
ponents, making them impossible to distinguish from natively developed ap-
plications. However, the bridges add a performance bottleneck and a level of
indirection which needs to be implemented by library maintainers.
Model Driven Software Development
Model driven software development or MDSD is an approach where the devel-
oper deﬁnes the application in a platform independent modelling language and
the platform speciﬁc code is generated from this model. It is a fairly uncommon
approach and currently exists mostly in research. The most notable framework
in this category is MD2 [W26] which was developed at the University of Münster
and serves as the basis for several research papers. Since this approach is not
used for application development in the industry, this study will not cover any
frameworks in this category.
Transpiling
The last approach in this categorization is transpiling. Transpiling is the last of
the two generative approaches. Frameworks in this category transpile platform
independent code into platform dependent code that is able to run directly on
==PAGE:35==
4.2 Frameworks
25
the target platforms. This approach results in fully native applications that
do not rely on bridges for communication. This inherently results in access to
all native features such as sensors and other API’s. With this approach it is
crucial that the framework is well maintained and frequently updated, since
native API additions or changes need to be implemented by the framework
maintainers before they can be used in development.

Flutter [W11]
Flutter is a new and promising free open-source framework developed and main-
tained by Google. Flutter supports iOS and Android. The ﬁrst stable version
was released in 2018 and has rapidly gained popularity and attention by the
application development community [W29]. Applications are developed in the
Dart language which supports ahead-of-time (AOT) compilation, which means
the code is compiled to native code and run on the target platforms. For de-
velopment builds it uses a VM to support just-in-time (JIT) compilation which
results in support of hot reloading. For this reason applications developed in
Flutter are ultimately in the transpiling category, but for development it is in
the self-contained runtime category.
Flutter works similarly to game frameworks since it relies on Google’s own Skia
engine to render graphics, and not the platform’s native components.
This
means that applications will look identical on both platforms, but it also means
that changes in the UI of the native platforms will not aﬀect the UI of the
application. This forces maintainers to implement the UI changes within the
framework instead of taking advantage of the native implementations.
Ionic [W30]
Ionic is a free open-source framework maintained by the company of the same
name Ionic. Ionic supports both iOS and Android. The languages HTML, CSS,
and JavaScript are used for development.
Ionic is built on top of Apache Cordova [W31] and thus builds apps in the hybrid
category. Ionic is a library that allows applications to look and feel native and
allows for development in popular web development frameworks such as Angular
[W32], React.js [W10], or Vue.js [W33].
Apache Cordova is the engine that
allows the applications to access native device functionality with JavaScript.

Microsoft PowerApps [W34]
Microsoft PowerApps is a framework for building business applications, devel-
oped and maintained by Microsoft. It requires a subscription but has a free
trial. Microsoft PowerApps supports both iOS and Android. It builds apps in
the hybrid category, as it uses a container application to display the content
built with the Microsoft PowerApps editor. Microsoft PowerApps uses a drag-
and-drop editor to build applications and requires no coding, which classiﬁes
Microsoft PowerApps as end-user programming [15]. Because of this, applica-
tions built with Microsoft PowerApps are restricted to the functionality provided
by Microsoft which limits the usability considerably to data-driven business ap-
plications.
NativeScript [W12]
NativeScript is a free open-source framework maintained by Progress Software
Foundation. NativeScript supports both iOS and Android. NativeScript allows
for development in NativeScript XML, JavaScript, or in Typescript, using the
popular web development frameworks Angular or Vue.js. It builds applications
in the self-contained runtime category by utilizing the built in JavaScript engines
JavaScriptCore [W35] on iOS and V8 on Android [W36]. NativeScript relies on
the native UI components in the diﬀerent platforms it supports. This means
the application will look exactly as expected on that respective platform. The
framework relies on a bridge to communicate between the JavaScript and the na-
tive runtime. This bridge requires components to be supported by NativeScript
before they can be used in the JavaScript runtime called plugins. NativeScript
relies heavily on its open-source community to provide these plugins.
PhoneGap [W5]
PhoneGap is a free open-source framework maintained by Adobe Systems. Phone-
Gap allows applications to be developed using HTML, CSS, and JavaScript.
PhoneGap supports iOS and Android, and builds applications in the hybrid
category.
==PAGE:39==
4.2 Frameworks
29
In 2012 the PhoneGap codebase was donated to the Apache Software Foundation
and dubbed Apache Cordova.
Apache Cordova is now the engine that runs
multiple frameworks including PhoneGap. PhoneGap contains Adobe speciﬁc
features on top of Apache Cordova such as a cloud build server and a custom
deployment tool.
React Native [W10]
React Native is a free open-source framework maintained by Facebook. It sup-
ports iOS and Android. Development of applications in React Native employs
JSX for the UI and JavaScript for the application code. It builds applications
in the self-contained runtime category as it utilizes the built-in JavaScriptCore
engine on iOS and ships the JavaScriptCore engine with its Android applica-
tions. It uses native UI components for the GUI and communicates via a bridge
from the native context to the JavaScript context. For development it leverages
the same technologies as the popular React.js framework [W37] also developed
by Facebook. Developers also have the possibility to add static typing via Flow
[W38] to their JavaScript code. As with NativeScript, React Native relies on
it’s vast community to implement component libraries that bridge between the
native runtime and the JavaScript runtime allowing developers to use the com-
ponents in the JavaScript runtime.
The ﬁle size of React Native’s Android
package is inherently large because of the shipped JavaScriptCore runtime.
Rhodes [W39]
Rhodes is a free open-source framework maintained by Tau Technologies. It
supports iOS and Android.
Rhodes supports building applications with the
typical web development languages HTML, CSS, JavaScript, and also supports
the less common Ruby language for development. Rhodes build applications in
the hybrid category as it relies on the built-in Webview to display the applica-

that seek to monetize, to pay Tau Technologies a percentage of the earnings.
Titanium SDK [W40]
Titanium SDK is a free open-source framework maintained by Axway. It sup-
ports iOS, Android, and Blackberry.
The GUI is written in XML and the
application code in JavaScript. Titanium SDK builds applications in the self-
contained runtime category because they use native components but run the

application code in the respective platforms built-in JavaScript engine, which
on iOS is JavaScriptCore and on Android is V8. Titanium SDK requires appli-
cations to be developed in its own IDE.
Xamarin [W9]
Xamarin is a free open-source framework owned and maintained by Microsoft.
Xamarin provides two diﬀerent strategies for development namely Xamarin Na-
tive or Xamarin Forms. The diﬀerence is that Xamarin Native does not share
the UI code between the diﬀerent platforms, but only shares the business logic.
Xamarin Forms additionally shares the UI code. Both Xamarin Native and Xa-
marin Forms are based on the open-source Mono framework and are developed
in C#. Xamarin Forms allows for the UI to be developed with XAML. Xamarin
Native and Xamarin Forms support iOS and Android natively. Xamarin Native
and Xamarin Forms build applications in the transpiling category since the C#
code is transpiled to byte-code that is run on the target platforms.

5.1
Flutter
Flutter is a relatively new cross-platform framework released in 2017 by Google
[W11]. Development in Flutter is done with the Dart programming language.
Dart is a programming language invented by Google and originally intended to
be the new standard for web development, intended to take over JavaScript,
bringing object orientation and garbage collection to the web. Dart is primarily
==PAGE:46==
36
Example Application Development
used to create mobile and web applications, but can also be used to create
desktop and backend applications [W45]. Dart can be compiled both to machine
code and to JavaScript code and Flutter takes advantage of this ﬂexibility.
Flutter compiles the Dart code to machine code when building applications
for production. This eliminates the need to run the code inside a JavaScript
runtime and increases the performance of the applications. Development builds
however, are compiled to JavaScript and run inside a JavaScript runtime. By
doing this, hot reloading becomes available when developing in Flutter, which
makes mobile application development a far better experience. Hot reloading is
a feature that allows the framework to inject modules in the running application
without changing the state of the application. This means that changes in the
code, are instantly reﬂected in the application without the need of restarting.
One characteristic of Flutter is that the applications look the same on Android
an iOS, since Flutter uses the Skia graphics engine to render the GUI.

does not require any prerequisites to be installed.
To be able to run a Flutter application on an iOS device or simulator for devel-
opment, Xcode 9.0 [W47] or newer needs to be installed on a Mac computer.
To run the application on an Android device or simulator for development, An-

5.2
PhoneGap
PhoneGap is a framework released in 2009 by Nitobi. Nitobi was purchased by
Adobe in 2011 and the PhoneGap framework was open-sourced and contributed
to the Apache Software Foundation. PhoneGap continued at Adobe as a product
providing extra services on top of the Apache Cordova engine. PhoneGap builds
applications in the hybrid category, meaning the application is run inside a
WebView. Accessing platform speciﬁc features is done through a bridge which
is provided by the Cordova engine. Applications in PhoneGap are developed as
web applications with HTML, CSS, and JavaScript. [W49]
5.2.1
Installation & Set-Up
The setup and installation of the PhoneGap framework can be done in two ways.
One way is to use the PhoneGap build service which is a cloud service provided
by Adobe, that lets developers set up a project in the cloud that compiles, builds
and distributes the applications. The distribution works in the way that the
required certiﬁcates are added to PhoneGap build, and then the service takes
care of building and uploading the applications to its distribution channels.
This liberates developers from the platform requirements of speciﬁcally iOS
that requires a Mac computer to develop and distribute applications locally.
The other way is to set up a project locally with the use of the PhoneGap
Command Line Interface (CLI). The latter approach was taken in this case.
Setting up the PhoneGap project with the PhoneGap CLI was easily done by
following the PhoneGap CLI installation instructions [W50]. The PhoneGap
CLI is supported on Mac and Windows and has node.js and git as prerequisites.
To run the application on an iOS simulator Xcode 8.0 or newer is required. To
run the newest PhoneGap version on Android, Android version 4.4 (Android
==PAGE:56==
46
Example Application Development
API 19) or higher is required.
It is also possible to run the application on a physical device, either by using the
PhoneGap application which, when connected to the same network as the devel-
opment machine, can display the application, or by connecting a physical device
to the computer and deploying the application through the PhoneGap CLI. Fur-
thermore, since applications in PhoneGap are built using web technologies, it
is possible to test applications using a browser. This makes debugging with
browser development tools possible and also required, since debugging within

5.3
Xamarin
Xamarin is a framework released in 2011 by an independent company founded
by the creators of the Mono framework. Xamarin was purchased by Microsoft
in 2016 and the Xamarin framework was open-sourced and bundled as a free
tool inside Microsoft Visual studio.
Xamarin builds applications in the transpiling category, meaning the application
code is transpiled into platform speciﬁc code.
In Xamarin applications are
==PAGE:63==
5.3 Xamarin
53
developed using C#. Access to platform speciﬁc features is also done through
C#. This can be done since the platform API’s are all ported to C#.
Xamarin comes with two approaches for developing cross platform mobile ap-
plications. The ﬁrst one is called Xamarin Native. With this approach, it is
possible to share only the business logic of the application. The UI is imple-
mented separately for each platform. The second approach is called Xamarin
Forms which also allows for sharing of the UI by allowing developers to de-
velop the UI in either C# or XAML by using Xamarin’s own cross-platform UI
components. This allows for faster developing and prototyping. For developing
the Chatter example application, Xamarin Forms was used thus the rest of this
section will be about Xamarin Forms.

The setup and installation of the Xamarin framework was done by installing
Visual Studio for Mac. This IDE comes bundled with Xamarin and is the IDE
that is recommended by Microsoft to develop applications with Xamarin [W52].
It is also the IDE that was used to develop the example application.
To run the application on an iOS simulator, Xcode 10.0 or newer is required.
To run Xamarin on Android, Android Platform Tools and Android SDK Tools
are required. It is possible to conﬁgure the Android version in Xamarin.
It is also possible to run the application on a physical device, by connecting
the device to the computer and selecting it as the build target in Visual Studio.
Xamarin Forms leverages the capabilities of Visual Studio which includes a
debugger and a WYSIWYG editor for the XAML UI. The WYSIWYG editor
can emulate diﬀerent devices running both Android and iOS. It is also possible
to support hot reloading of the XAML code by installing a plugin.

React Native
React Native is a framework developed and maintained by Facebook. React
Native builds applications in the self-contained runtime category. The applica-
tions are shipped with a JavaScript runtime which is used to run the application
code. All the application code is written in JavaScript. Development in React
Native is similar to the web development framework React.js, also developed
and maintained by Facebook, using regular JavaScript for business logic, JSX
to develop the UI, and the option to utilize Flow for static type-checking. Ac-
cess to platform speciﬁc features is achieved either by using React Native’s own
components or by installing plugins developed by the community and using the
components from these.
5.4.1
Installation & Set-Up
Installation of React Native is easily done by following the installation guide
[W59]. The installation oﬀers two diﬀerent approaches either using the Expo
CLI or the React Native CLI. Expo [W60] is a free and open-source tool-chain
built around React Native. With Expo it is possible to build, deploy, and release
applications for both iOS and Android without the need of Android Studio and
Xcode as it normally requires. The drawback is that if the application requires
use of native code it needs to be "ejected" to the React Native CLI since Expo
does not support this. This is also the case if a plugin that contains native
code is installed.
For this reason, Expo is a tool which is mostly suited for
small applications or prototypes.
For the Chatter example application, the
React Native CLI was chosen. The React Native CLI is supported by MacOS,
Windows, and Linux.
To be able to run a React Native application on an iOS device or simulator for
development, Xcode 9.4 or newer needs to be installed on a Mac computer.
To run the application on an Android device or simulator for development,
Android Studio needs to be installed with the Android 9.0 (Pie) SDK installed.
It is also possible to run the application on physical devices by connecting with
a cable. Cable connection is only required the ﬁrst time. When React Native
runs it starts a development server, which can be connected to with a speciﬁc
ip-address and port number.
React Native allows both live-reload and hot-
reloading. Live-reload refreshes the application when a ﬁle changes, whereas hot
reloading only refreshes the speciﬁc ﬁles that were changed without altering the
state of the application. This is possible because of the self-contained JavaScript

PhoneGap and Cordova has been around since 2009, making PhoneGap the
framework that has been around the longest of the ones included in this com-
parison. Because of the maturity of the framework, many plugins have been
developed. However, most of these plugins are not maintained by a large or-
ganization such as Apache or Adobe that maintains Cordova and PhoneGap
respectively, but maintained by the community. When developing applications
in PhoneGap it became clear that many of the plugins are not maintained or
do not support newer versions of the framework.
==PAGE:93==
6.2 Evaluation Results
83
Long-term feasibility
PhoneGap does not release new versions very often, with only three releases in
2018 [W70]. PhoneGap supports the newest android and iOS versions and has
a large community surrounding it. One of the beneﬁts of using the Cordova
engine is the shared community between all the frameworks that relies on this
engine, since issues and solutions are often shared between these frameworks.
Apache is the maintainer of the Cordova engine, and Adobe is the maintainer of
the PhoneGap framework which are both large organizations. PhoneGap has a
large market share with AppBrain reporting it to be 7.97 % of all applications
in the Google Play Store at the time of writing [W71].
Look and feel
PhoneGap builds applications in the hybrid category which makes the applica-
tions run inside a Webview. There are drawbacks of this approach and they
become visible in this criterion, since the applications practically behave as a
website. What this means is that all UI components are rendered in a Webview,
making no use of the native UI components and the look and feel of those. In
regards to the look and feel of the components, everything such as animations
or reactions to touch input, has to be implemented. One concrete example of
this is the response when pressing a button. In PhoneGap, the button has no
behavior and all behavior has to be implemented to achieve a similar look and
feel to that of a native application. There are approaches in the development
that can be taken to improve the responsiveness such as building the application
with the single-page architectural approach. However, the start up time suﬀers
from this approach since the WebView needs to initialize and afterwards load
all scripts and stylesheets before being able to display the application.
Application speed
The application speed of PhoneGap is not optimal. The hybrid approach some-
what limits the applications performance as it adds an extra layer between the
application code and the platform, namely the WebView. This can be felt in
the responsiveness of the navigation as well as when starting up the application.

Framework Evaluation
Distribution
PhoneGap has two approaches for developing an application with the framework.
One is to develop the application locally and distribute it manually. PhoneGap
sets up native iOS and Android projects that build the applications which can
be distributed just as natively developed applications. The second approach
is to utilize PhoneGap Build which is a cloud service provided by Adobe that
allows developers to compile and test the application in the cloud, as well as
distribute release builds, prototypes and tests builds from this platform. It does
this by allowing the developer to upload the required certiﬁcates, and PhoneGap
Build takes care of the actual distribution process.
6.2.2.2
Development
Development environment
PhoneGap is independent of development environments meaning any preferred
development environment or text editor can be used to develop applications
with PhoneGap. Any development environment used for web development suits
PhoneGap. The PhoneGap CLI sets up a starting project which includes the
boilerplate environment. IntelliJ was used to develop the Chatter example ap-
plication in PhoneGap. IntelliJ has a plugin supporting code completion for
Cordova and PhoneGap, and a toolbar to easily run the PhoneGap application
from inside the IDE.
PhoneGap build connects to a Git repository making it easy to develop Phone-
Gap build applications locally using any desired development environment.
GUI design
PhoneGap does not support hot reload or live reload out of the box. However,
live reloading can be added to PhoneGap by installing a plugin. Neither does
it come with a WYSIWYG editor to make it easy for developers to design
the UI. Since it is possible to debug PhoneGap applications in the browser, it
is possible to design and test the GUI using the browser’s development tools
which eliminates the need of constantly redeploying the application to see the
eﬀects of changes in the UI code, and it allows web developers to use the browser
environment they are accustomed to.

Ease of development
The documentation of PhoneGap installation and setup is suﬃcient and easy
to follow, but beyond this the documentation is lacking. PhoneGap has three
tutorials on three diﬀerent topics but does not provide more documentation on
how to develop applications in PhoneGap. Developing the example application
proved that getting started on PhoneGap is diﬃcult because of the lack of
documentation.
The PhoneGap community is large and most problems are
covered in community forums such as GitHub and Stackoverﬂow. It is easy to
ﬁnd solutions for the problems that might arise, especially in the initial phase
of learning the framework.
As mentioned, applications with PhoneGap are developed using the common
web languages HTML, CSS, and JavaScript. This makes development in Phone-

Maintainability
The PhoneGap application consists of about 2000 lines of code. Since Phone-
Gap does not enforce any structure or technique upon the application code,
any approach can be taken and thus any desired JavaScript framework can be
used. The decision of JavaScript framework used for development has a higher
impact on the maintainability than PhoneGap itself. Thus highly maintainable
JavaScript libraries can be used. However, Cordova plugins become obsolete as
the surrounding platforms gets updated, if the plugins are not updated accord-
ingly. This is an issue when the community that develops the plugins are not
maintaining them.
Scalability
Since the framework allows for web development, modularization is possible by
including third-party libraries that allow this. Unfortunately, PhoneGap does
not support the newer ECMAScript 6 which includes modularization natively
in the language, so third-party libraries need to be included.
PhoneGap does not enforce any structure on the application code which can
be developed as the developers see ﬁt with any desired JavaScript library, so
additional policies are needed in order to ensure scalability.

The aim of this research was to develop guidelines for systematically selecting
cross-platform mobile application frameworks for iOS and Android, based on
their features and characteristics.
To accomplish this goal, it was decided that an example application was to be
developed using diﬀerent cross-platform mobile application frameworks. The

the categories that were to be covered by the example application in Section 3.1.

Games. Secondly, in Section 3.2 the features included in the example application
were established by examining popular applications in the covered categories.
It was determined that the messaging applications on the list covers most of
the features, hence the example application was to be built as a messaging
application. In Section 3.3 the UI and functionality of the example application
(Chatter) was determined.
Chapter 4 categorized the diﬀerent approaches for mobile application develop-
ment in Section 4.1. It was decided that the study would not cover the categories
of Mobile Web Apps, Progressive Web Apps, and Model Driven Software Devel-
opment. Then, a list of the most prominent cross-platform mobile application
development frameworks in these categories was made by combining informa-
tion from recent studies and other relevant factors. The factors and the list can

frameworks were selected to be the ones that were to be covered in the study.

5 shows how to install and set up each of the frameworks. This chapter also

Each of the four cross-platform mobile application development frameworks were
evaluated in Chapter 6 by establishing common evaluation criteria in Section
6.1. The frameworks received a score from zero to ﬁve in each criterion based
on the example applications and the experiences from the development thereof.
The guidelines for systematically selecting cross-platform mobile application
development frameworks were established in Chapter 7. The guidelines require a
developer proﬁle form and an application proﬁle form to be ﬁlled out. From these
proﬁles, a score for each framework is calculated based on values established
from ﬁndings in Chapter 5 and the evaluations in Chapter 6. The framework
scoring the highest is the recommended framework for that speciﬁc developer
and application proﬁle.
As such, the guidelines for systematically selecting cross-platform mobile appli-
cation development frameworks were realized using an algorithm which is based
on both the developer’s proﬁle and a proﬁle for the application to be developed.
The guidelines cover selected frameworks that support both iOS and Android.
The guidelines were made based on criteria that cover the features and charac-
teristics of the selected frameworks. The evaluations were done by developing an
identical example application with each framework and scoring the applications
as well as the development experiences. The features of the example application
were decided by examining popular applications in selected categories.
Some examples of how these guidelines could be improved are discussed in the
following section.
8.1
Future Work
This study covers four diﬀerent cross-platform mobile application development
frameworks. For this reason, the guidelines only include these four frameworks.
The guidelines could be extended by developing the Chatter application in dif-
==PAGE:115==
8.1 Future Work
105
ferent cross-platform development frameworks and scoring the frameworks ac-
cording to the criteria mentioned in Section 6.1. As the frameworks each have
their own strength and weaknesses, adding more frameworks to the guidelines,
will result in a more accurate recommendation. Ideally, the guidelines would
cover all the major cross-platform mobile application development frameworks
mentioned in Section 4.2.
The calculation of the guidelines are based on scores from the evaluations es-
tablished in Chapter 6. These scores could be reﬁned by a machine learning
model based on developer experiences with the guidelines and the frameworks.
This would strengthen the accuracy of the guidelines and the probability rec-
ommending the correct frameworks.
The guidelines could be made publicly available on a website where developers
could create their own developer proﬁle. It could be open-sourced to allow de-
velopers to add new cross-platform mobile application development frameworks
as well as provide feedback about their experiences with the guidelines for fur-
ther improvement. Based on this information, the algorithm that recommends
the frameworks could be improved.
