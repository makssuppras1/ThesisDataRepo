Abstract
Infrastructure as code has become the industry standard for provisioning cloud
infrastructure for all sizes of systems. It allows for easily creating many diﬀerent
kinds of computing resources the tools available are getting more advanced over
time.
Unfortunately, companies and other organisations are experiencing a shortage of
cloud infrastructure experts, whilst the discipline is becoming and increasingly
complex aﬀair. This phenomenon is associated with an overspending on cloud
resources.
This project makes a contribution to solving this problem by designing and
implementing a software tool, Corsa, which can generate infrastructural archi-
tecture designs based on speciﬁcations of the requirements of a software system,
with the aid of our other contribution, a new speciﬁcation language called CRS.
Corsa relies on open standards and existing state-of-the-art research into open
standards for the infrastructure as code approach, and leverages tools such as
TOSCA and EDMM to provision the generated infrastructural architectures.
We evaluate Corsa and ﬁnd that while it does make a contribution towards
solving the problem, it still requires further development to fully prove its worth
in addressing the problem.

Chapter 1
Introduction
Acquiring access to computational resources has never been easier, due to the
proliferation of online platforms which oﬀer an instant allocation of a wide
range of computational and data storage resources, collectively known as cloud
computing. Broadly, the mass adoption of cloud computing has vastly decreased
the time needed to manage computational infrastructure . Furthermore, as
the adoption of cloud computing has matured, an automated approach to
infrastructure management, known as infrastructure as code (IaC), has been
adopted widely in the industry. The IaC-approach consists of specifying the
infrastructural architecture in a format which allows software tools to interpret
the speciﬁcations and automatically construct new infrastructure or modify
existing infrastructure, depending on the provided speciﬁcation.
The IaC-approach provides numerous beneﬁts: The process of creating or modi-
fying cloud infrastructure has been reduced to modifying a text ﬁle, and then
letting the automated tools take care of eﬀectuating the necessary operations nec-
essary to create, modify, or destroy the infrastructure. Furthermore, a particular
infrastructural architecture can be readily reproduced, since the speciﬁcation is
independent of the state of the infrastructure. Also, the text-based nature of the
speciﬁcations allow them to be version-controlled with existing version control
systems. As such, an update with breaking changes can be reverted back to a
==PAGE:13==
2
Introduction
previously functioning version.
However, the relative ease of making modiﬁcations to a cloud-based infrastructure
also means that small changes to a speciﬁcation can lead to drastic modiﬁcations
of an infrastructure that is already in use. For example, a typo in an IaC
conﬁguration incurred a 150 million USD loss for Amazon Web Services [4].
Furthermore, the IaC-approach does not improve the process of designing an
infrastructural architecture; it only makes it easier to instantiate the design,
once it is complete. As mentioned, the changes can be rolled back, but the
damage, in the sense that crucial infrastructure might have been removed or
unnecessarily expensive infrastructure might have been created. The process of
designing correct infrastructural architecture is itself a substantial challenge, as
it demands a deep understanding of the requirements of the system that utilizes
the infrastructure, and which architecture meets those requirements. When this
is combined with the observation that there is a shortage of professionals with a
competence in cloud infrastructure design, resulting in non-experts writing IaC
conﬁgurations, it becomes clear that designing correct IaC conﬁgurations is a
present and relevant challenge.
The project described in this report addresses this problem by providing a tool
which can automatically generate IaC conﬁgurations based on a speciﬁcation of
the requirements of a software system. The tool interprets the requirements of
the input systems and ensures that the generated IaC conﬁgurations contains the
declaration of an infrastructural architecture which satisﬁes those requirements.
This report documents the problem analysis and solution design, and is structured
as follows:
• In Chapter 2, we explore the problem further and present a narrowly scoped
problem statement. Furthermore, we provide a review of existing literature
regarding the problem and a description how the project described in this
report contributes to the state of the art.
• In Chapter 3, we introduce a running example of how the tool is used and
provide an overview of the solution concept. Furthermore, this chapter
provides an overview of the concepts and tools which the solution depends
on.
• Chapter 4 provides a detailed description of the design of the solution,
including descriptions of concepts from the external tools, where they are
relevant.

Chapter 2
Problem
In this chapter, we ﬁrst document the presence and relevance of the problem
presented in the introduction of this report, namely the problem of designing
correct IaC conﬁgurations. Afterwards, we discuss the root causes and state a
problem deﬁnition based on that discussion. Also, we present a hypothesis based
on the problem deﬁnition, along with relevant research questions. Lastly, we
review the identiﬁed existing attempts at addressing the problem.
2.1
Problem Analysis and Deﬁnition
In this section, we qualify the problem with ﬁndings from both academic literature
and surveys of IT professionals. Then, we identify the root causes of the problem,
which then lead to a problem deﬁnition for the project described in this report.
2.1.1
Existing Research on Challenges
Here, we present literature which provide an insight into the relevance of the
problem as well as indications of why the problem exists. The literature and
other material was found through a explorative approach, as we do not aim
to provide a comprehensive review of the literature on the problem. Rather,
we focus on identifying a suﬃcient amount of documentation to claim that at
least a signiﬁcant amount of organisations experience challenges regarding cloud

In conference paper titled “Co-evolution of Infrastructure and Source Code - An
Empirical Study” from 2015, Jiang et al. [1, page 54] document their survey of
256 software projects, and they ﬁnd that
• IaC conﬁgurations are often large and change very frequently, which “could
indicate a potential of introducing bugs”
• “infrastructure ﬁles are usually changed by regular developers instead of
infrastructure experts”
• “IaC ﬁles should be considered as source code ﬁles not just because of the
use of a programming language, but also because their characteristics and
maintenance needs show the same symptoms as build and source code
ﬁles”
Thus, IaC development and maintenance is costly because of the expertise
required to correctly design IaC conﬁgurations, but also because of the signiﬁcant
amount of damage caused when IaC conﬁgurations are not designed correctly.
In the report titled Cloud Maturity Drives Business Success [13, page 11],
a report commissioned by IaC technology provider HashiCorp, the following
ﬁndings are made based on an online survey “of 1,194 global, cross-industry
enterprise technology practitioners and decision-makers”:
• “91% of respondents reported diﬃculties with cloud waste at their organi-
zation”
– Here, “cloud waste” refers to expenditure on cloud infrastructure
resources that was not necessary
• “A Lack Of Skills And Overprovisioned/Idle Resources Are Biggest Cloud
Waste Factors”
• “Most Organizations Face Cloud Infrastructure Expertise Shortage”
By putting those three ﬁndings together, we get an indication that organisations
are allocating, and thus paying for, more cloud resources than they really need,
and that this might be associated with a lack of skills in managaing cloud
infrastructure.

2.1.2
Discussion of Findings
From the above literature, we determine that software organisations experience
problems with infrastructural architecture design, namely that the design is often
more costly than neccesary and that the IaC conﬁgurations can contain bugs.
We assess that the problem stems from the following root cause: Designing cloud
infrastructure architecture is a diﬃcult task, as it requires an understanding
of the system which utilizes the infrastructure as well as an understanding of
the capabilities and cost of the infrastructural resources available. Furthermore,
the expertise required to conduct infrastructural architecture design is in short
supply for many organisations, which means that the available cloud infrastruc-
ture experts are overburdened and that non-experts often make decisions on
infrastructural architecture design of a system. As a result, the professionals
designing the infrastructural architecture make errors in the form of allocating

2.2
Problem Deﬁnition
Based on the root causes identiﬁed in the previous section, we chose to address a
particular sub-problem: The challenge of designing infrastructural architectures
which allocate only the required resources and which don’t contain bugs, while
being constrained by the lack of expertise in cloud infrastructure architectural
design. By "lack of expertise", we mean that cloud infrastructural architectures
are often not designed by domain experts, and the experts that organisations do
employ are often overburdened.
2.3
Hypothesis and Research Questions
In the project described in this report, we solve the problem deﬁned above by
designing a software tool which is used as an aid in the infrastructural architecture
design process. More specically, the designed tool can generate an infrastructural
architecture based on provided information about the software system which is
to be deployed on the infrastructure. Thus, we examine the following hypothesis:
H1: It is possible to design a software tool which can generate infrastructural
architecture, such that the infrastructural architecture design process is made
easier and less error-prone.
The following research questions provide a basis for examining the validity of
the hypothesis:
• RQ1: How can a language be designed to allow the speciﬁcation of which
infrastructural resources a software system requires?
• RQ2: How can such speciﬁcations be used to construct an infrastructural
architecture model?
• RQ3: How can the infrastructure in the constructed model be automati-
cally provisioned?

2.4
Review of Existing Solutions
This section presents a brief review of the research-based and industrial eﬀorts
into addressing the problem statement of the project described in this report.
The research reviewed here was discovered through an explorative process, as we
discovered that searching literature databases for keywords such as "infrastructure
as code" or "DevOps" yielded very few relevant results. Instead, our research
discovery was based on exploring the citation graph connected with the few
relevant research papers we identiﬁed.
The ﬁeld of research on addressing the challenges of the IaC approach is active.
Most central to the project described in this report is the research conducted
at the University of Stuttgart, more speciﬁcally the Institute of Architecture of
Application Systems, which we will review in this section. The research is focused
on creating an open-source alternative to proprietary IaC-technologies, by using
an open standard for infrastructure modelling and designing and implementing a
toolchain around that standard. The open modelling standard is named Topology
and Orchestration Speciﬁcation for Cloud Applications (TOSCA), which was
created by the Organization for the Advancement of Structured Information
Standards (OASIS) [3]. Of most relevance to the project described in this report
is the research headed by Michael Wurster, which is focused on providing a
deployment modelling approach that is agnostic to the choice of deployment
technology. Their research has yielded the following papers:
• In a paper from 2019, titled The essential deployment metamodel: a
systematic review of deployment automation technologies [6], Wurster et
al propose the Essential Deployment Metamodel (EDMM), a modelling
framework which contains the essential components shared by most IaC
technologies. The framework is based on analysis and catogorisation of
the most popular deployment technologies, and a mapping from EDMM
to the model contained in each of these technologies is provided.
• In another paper from 2019, titled The EDMM Modeling and Transforma-
tion System [5], Wurster et al. present their implementation of a graphical
tool for Eclipse to model and export EDMM-models, as well as an API (the
EDMM Transformation Network) which can transform EDMM-models
into any of the 13 diﬀerent technology-speciﬁc deployment models analysed
in ﬁrst paper we presented in this list [6], which includes both the cloud
platforms such as AWS and Azure, IaC-tools such as Terraform, but also
purely orchestration tools such as Kubernetes.
• In a paper from 2020, titled TOSCA Light: Bridging the gap between
==PAGE:21==
10
Problem
the TOSCA speciﬁcation and production-ready deployment technologies
[7], Wurster et al. propose a subset of TOSCA, TOSCA Light, which
is compliant with EDMM. As such, a TOSCA Light model can thus be
transformed into an EDMM model, and from there into any of the 13
aforementioned deployment technologies.
• In another paper from 2020, titledTOSCA Lightning:
An Integrated
Toolchain for Transforming TOSCA Light into Production-Ready Deploy-
ment Technologies [8], Wurster et al. propose and implement the TOSCA
Lightning toolchain, which contains a TOSCA Light GUI modelling tool,
a TOSCA Light validator, a TOSCA Light to EDMM transformation tool
and the EDMM Transformation Network. With this toolchain, one can
import TOSCA-models, edit them to achieve compliance with TOSCA
Light (which is validated by the toolchain), and ultimately transform that
model into a model for a speciﬁc deployment technology.
• In a paper from 2021, titled Automating the Deployment of Distributed
Applications by Combining Multiple Deployment Technologies [9], Wurster
et al. provide an approach to automatically deploying distributed sys-
tems across multiple and diﬀerent deployment technologies, by extending
the EDMM syntax with "Technology-Speciﬁc Regions", which map each
component to a deployment technology.
Somewhat in parallel, the same institute has also produced the OpenTOSCA-
toolchain, which is another toolchain that focuses on a full version of TOSCA
and is lead by researcher Miles Stötzner. Furthermore, it focuses less on the
infrastructure needed and more on the application modelling. Also, the toolchain
depends on a TOSCA "orchestrator" to deploy the produced models.
The
toolchain contains such an orchestrator (TOSCA Container), but the toolchain
is compatible with other TOSCA orchestrators [12, 17]. However, this branch of
the literature is less interesting to the project described in this report, as the
focus of this branch is on enhancing TOSCA-modelling tools and achieving a
fully TOSCA-based deployment pipeline. The project described in this report is
more focused on automating existing infrastructure provisioning workﬂows, and
the research by Michael Wurster is thus more relevant. An earlier attempt at
addressing the issues of the project statement of the project described in this
report is found in a paper from 2018 by Bhattacharjee et al., titled A model-
driven approach to automate the deployment and management of cloud services
[2], which proposes a domain-speciﬁc modelling language (DSML) that provides
high-level models of system components in order to reduce the domain knowledge
required to create deployment models. However, the research of Michael Wurster
appears signiﬁcantly more developed, with numerous concrete implementations

provided for many deployment technologies. The project described in this report

There also exists research in IaC challenges that does not consider TOSCA. A
paper from 2024 by Qiao et al. [15], titled Statically Inferring Usage Bounds for
Infrastructure as Code, describes the challenge regarding the unpredictable cost
of IaC-based operations and proposes and implements a tool for static analysis
of IaC deployments that can validate resource usage estimates.
Existing research on automatic generation of IaC conﬁgurations focuses on using
large language models to generate deployment models from natural language. In a
paper from 2024, titled A Survey of using Large Language Models for Generating
Infrastructure as Code [16], Srivatsa et al. review numerous research attempts
and tools using this approach, and they present their own proposal which uses the
language-model GTP3.5 to generate Terraform conﬁgurations. It is noteworthy
that all the existing research mentioned in the paper generate deployment models
for speciﬁc deployment technologies. In the paper, Srivatsa et al. also present
their own approach, which outperforms a commercial tool, in terms of the rate
of achieving an exact match between a reference IaC conﬁguration and the IaC
conﬁgurations generated by their approach, although more than 40% of their
generated conﬁgurations were not exact matches. In the context of the project
described in this report, LLM-based approaches contain a weakness: The person
prompting the model still needs to be an on the system to be deployed, as they
need to know what infrastructural resources the system needs.
The approach that the project described in this report is inspired by "Infras-
tructure from Code" as proposed in the article by Aviv et al. [11], seems to
only be adapted by industry-led eﬀorts, as of the time when this explorative
literature was conducted, which was in October of 2024. In an AWS conference
talk the same year [10], Jeremy Daly from the company Ampt also introduces
the term. Ampt also hosts a website of the topic [21], where they list a number
of commercial approaches, as well as one open-source approach, called Nitric.
On closer inspection, it appears that Nitric provides its’ own abstraction of
common cloud resources, that can be accessed from a software-development kit
that is provided for Node.js, Python, Go, Dart, C#, and Java. The developer
can then instantiate the resources in the source code of the system, for example
to connect to a database or to run a web application. Also, Nitric provides a
graphical user interface for visualising the architecture of the system. However,
the list of possible deployment targets is much smaller than what is provided by

only be deployed to AWS, Google Cloud, or Azure. Another concern is the fact
that there is very little control over the environment that each component will

written in the same language in order for Nitric to understand the relationships

by Nitric is not based on an open standard, which is a disadvantage in the

Thus, the project described in this report will focus on leveraging the capabilities

Normally, in order to add new components to an infrastructural architecture,
a central IaC conﬁguration is modiﬁed with the addition of references to the
executable forms of the new components, most likely in the form of container
images.
Moreover, the central IaC conﬁguration must also be modiﬁed to
assign adequate infrastructural resources to the new components. In eﬀect, the
infrastructural architectural design must be modiﬁed to include and accommodate
the new components. As described in Chapter 2, this design process is both
time-costly and introduces the risk of errors, especially if the existing design
is complex. To reduce the time consumption and risk of the design process,
the designer might decide to allocate brand new infrastructural resources to
each of the new components. However, this approach introduces the risk of
overspending, as it might be cheaper to host some of the new components on
the same infrastructural resources as the components that already exist in the
system.
Our approach is to let the system designer focus on the requirements of the
new components, and not the rest of the system: By using Corsa, the designer
merely has to create a speciﬁcation of the requirements of the new components,
and Corsa will integrate the new components into the existing infrastructural
architecture. Furthermore, Corsa optimises the infrastructural architecture with
respect to cost, such that the generated architectural design minimizes the risk
of overspending.

3.2.1
The Essential Deployment Metamodel (EDMM)
The Essential Deployment Metamodel (EDMM) is a metamodel which “provides
a common denominator of the features of the most important deployment
technologies” [6, P. 2] and was proposed in 2019 by Michael Wurster et al. at the
Institute of Architecture of Application Systems at the University of Stuttgart,
in a paper titled "The essential deployment metamodel: a systematic review of
deployment automation technologies" [6]. Furthermore, software tools have been
implemented for EDMM, which allow EDMM-based models to be transformed
to multiple diﬀerent IaC-tool formats [5]. Since EDMM is a crucial component
of Corsa, this section provides an overview of the core EDMM concepts.
3.2.1.1
Relevance for Corsa
We assess that EDMM provides a simple, yet powerful, framework for describing
software system topologies, which is the term we will use to describe any-graph
based model of a system where the nodes are are software or hardware compo-
nents, and the edges are the relations that the components have with each other.
Furthermore, initial analysis of the implemented EDMM software indicates that
it does indeed make it feasible to use the same model speciﬁcation to generate
viable speciﬁcations for multiple diﬀerent IaC tools. The metamodel is designed
to capture the essential concepts of a wide range of IaC tools, speciﬁcally those
tools for which speciﬁcations are written declaratively. As such, it contains
the same problem that Corsa solves: The designer is still tasked with creating
the design, which, as we have described in Chapter 2, is a laboursome and
error-prone process. However, EDMM also forms a part of our solution to the
problem: Corsa leverages EDMM and its related software tools in two diﬀerent
ways:
1. The metamodel is used by Corsa as a target modelling language, such
that systems speciﬁed with CRS ﬁles are transformed into EDMM-based
models. Thus, Any software tool which accepts EDMM-based models can
be integrated as a ﬁnal phase of the Corsa pipeline.
2. The EDMM software is utilized to transform systems speciﬁed with CRS
into an IaC conﬁguration that can be readily deployed.
3.2.1.2
TOSCA
The Topology and Orchestration Speciﬁcation for Cloud Applications "is a stan-
dardized modeling language allowing to declaratively specify portable multi-
service application deployments" [8].
The standard is created, owned, and

3.2.1.3

During our research on the problem (see Section 2.4), the TOSCA concept
was encountered frequently, and we initially assessed that it could provide an
essential modelling language for Corsa. However, we did not identify implemented
software tools for TOSCA-based models that were simple enough to implement
an integration with Corsa. In section 2.4, we identiﬁed two papers, also by
Michael Wurster et al. [7, 8], which proposed TOSCA Light, a TOSCA subset
which is directly transformable to EDMM, and a software toolchain, named
TOSCA Lightning, which allows TOSCA models to be validated for EDMM
compliance, and then transformed to EDMM-based models. However, we found
that the implementation of TOSCA Lightning was only accessible through a
graphical interface, which signiﬁcantly hinders the possibility of integrating
TOSCA Lightning with Corsa. When this discovery was made by us, signiﬁcant
eﬀort had already gone into creating a TOSCA Light-based solution. However,
we chose to keep using TOSCA Light as a modelling language internally in the
design and implementation of Corsa. This decision has multiple advantages.
Firstly, we found that TOSCA provides a comprehensive model of software
systems, in the sense that we found it contained all the concepts which we
wanted to capture about a software system. We speculate that is due to TOSCA
being a widely used standard, and thus, the speciﬁcation has evolved with the
feedback of many diﬀerent kinds of users. The second advantage is that modelling
the input software system with TOSCA allows for Corsa to be integrated with
TOSCA-based tools in the future, as the internal state of the model can be
output as TOSCA ﬁle at any point of the Corsa pipeline.

4.1
The Component Requirements speciﬁcation
Language
The ﬁrst necessary design concept of Corsa is to deﬁne how Corsa users can
specify the requirements of their software components in such a way that it is
feasible to automatically generate a viable infrastructural architecture.
In the running example, a software developer is tasked with adding new compo-
nents to an architecture. In Chapter 2, we identiﬁed the challenge of undertaking
the design process required to integrate new components into an existing in-
frastructural architecture. As mentioned, the challenge is the mental load of
analysing and modifying an IaC conﬁguration, without spending too much time,
without introducing errors, and without allocating resources that are more costly
than necessary, all at the same time. Corsa addresses these problems by reducing
the design work to specifying the requirements of each software component. We
argue that this reduces the mental load required, because the dependencies of a
software component are best known to the designer of that component, since
they naturally possess a deep understanding of the behaviour of that component.
In Figure 4.1, we show the example as we last saw it on page 17 in Chapter 3, but
with the following modiﬁcation that each component now relates to a vaguely
deﬁned service. The point of this ﬁgure is to illustrate the knowledge gained
from inspecting the source code of each component: For example, the source
code of the ResidentFrontendServer component would naturally have references
to an external backend server. If we assume the source code is written after best
practices, the connection strings to the backend server would not be hardcoded,
but instead be represented as variables, with the values of those being supplied at
runtime. Thus, the ResidentFrontendServer does not directly depend on another
component, but rather a particular service oﬀered by some component. Since the
example shown in Figure 4.1 is small, it is obvious which components provide the
required services. However, to modify the infrastructural architecture for much
larger systems, the designer will have to identify the relevant components, ﬁgure
out how to connect the new components to the relevant ones, and then make
sure this connection is possible through the architecture. Thus, Corsa provides
a contribution by reducing the knowledge needed to add new components to the
knowledge of the requirements of that new component.

scope of CRS service relations to HTTP-based connections, as we deem that most
cloud-based infrastructures rely exclusively on this protocol for communication
between components. However, the design does not prevent future work from
introducing support for other protocols.
A second CRS concept is the component, which contains services and contains
a reference to a container image which can be used to execute the services of
the component. Each component should declare the hardware requirements it
makes of the hosting environment, such as requirements concerning computation,
memory, and data storage. To solve the problem of cloud overspending, a CRS
component must also specify the requirements it makes of the virtual environment
that is hosting the container referenced by the CRS component.

contains one or more components, as well as references to other CRS ﬁles. The
choice of YAML was made based on the commonality of YAML as the primary
declarative language in tools concerning IaC and software deployment in general,
such as Docker Compose and Terraform. The conceptual model of CRS is
illustrated in Figure 4.2.

an internal, TOSCA-based model of the system.
The transformation to TOSCA is based on the observation that the TOSCA
modelling framework supports declarations of software component requirements
that are semantically equivalent to the CRS language, through the concept which
the TOSCA speciﬁcation refers as a “dangling requirement” [14, P. 23], which
is essentially any dependency on a component, where the concrete component
==PAGE:43==
32
Solution Design
is not yet identiﬁed, which closely mirrors the CRS concept of service depen-
dencies. Furthermore, CRS hardware requirements can be expressed in TOSCA
through so-called capability allocations, which allows, for example, to model a
virtual machine component and total date storage capacity it oﬀers to software
components, as well as modelling software components and their usage of that
storage capacity.
As mentioned in Section 3.2.1.3 on page 21, TOSCA is only used by Corsa as
an intermediate format between CRS and EDMM. However, since EDMM can
be expressed as a subset of TOSCA, transforming CRS to TOSCA ﬁrst means
that the later transformation to EDMM is very simple. Furthermore, as also
mentioned in Section 3.2.1.3, having a TOSCA-based internal model of the input
system allows for future iterations of Corsa to integrate with other TOSCA-based
tools.
4.2.1
TOSCA Concepts
In this section, we brieﬂy describe the TOSCA concepts which are essential for
Corsa.
TOSCA is a model-driven approach which aims to provide a standard langauge
for managing the entire lifecycle of large software systems [14, P. 14-15].
TOSCA was expressly designed to address the complexity associated
with managing large systems by providing a language for specifying
an information model and automating the lifecycle management of
large complex systems.
(OASIS TOSCA Version 2.0: Committee Speciﬁcation Draft 06, page 14.)TOSCA2-
6 TOSCA models software systems as graphs, such that components (both
software and hardware) are nodes, and the connections between the nodes are
the relations between the nodes, for example dependendcies [14, P. 15-16]. Each
node is expressed through a TOSCA template, which represents a component in a
system architecture. A collection of templates is referred to as a service template,
which is analogous to a graph by containing node templates which are connected
using relationships, which can be speciﬁed as relationship templates or within
the node templates [14, P. 16 & 79-80]. In the context of LaundryBook, the
running example, the entire system would be modelled in TOSCA as a service
template, and each component would be a node template.

Using TOSCA, we want to

laundry_service, without specifying a concrete component which oﬀers this
service. We accomplish this by creating a TOSCA requirement assignment
which targets a node type. In most cases, a requirement assignment will target
another node template, which fulﬁlls the requirement. However, in this case,
by deliberately referencing a node type instead, we render the requirement as
unfulﬁlled, or in the words of the TOSCA speciﬁcation, dangling [14, P. 23].
As described in Section 4.1, each CRS service is contained within a CRS com-

4.6 shows the node template for the resident backend server component. As a

speciﬁes that it must have a single "HostedOn" relationship with another node

is used here to represent a CRS component. We add an allocation block to the
requirement, which contains the CRS hardware requirements of the service, and
forces the node in the other end of the relationship to provide enough hosting
capacity.

allocate a part of a capability of another node template. For example, if the
LaundryServer of the running example needs 12 GB of storage, it will have a
requirement assignment targeting a Compute node template where the allocation
block of the requirement assignment means that the Compute node now has 12
GB less to oﬀer to other node templates that might be hosted on it.

backend server, with the name “host”. This allows other node templates, which

4.3
Assigning Infrastructural Resouces
Once a set of input CRS ﬁles have been parsed and transformed to TOSCA,
Corsa resolves the unfulﬁlled requirements, namely the “host” requirements
contained in the node templates that represent the CRS components (see Figure
4.6 on section 4.2). We resolve the requirements by searching for compatible
node templates in a repository of node templates, which represent the types of
infrastructural resources which are available to the user of Corsa. In the current
state of development for Corsa, the repository of node templates only consists of
types of virtual machines oﬀered by the cloud infrastructure resource provider
Amazon Web Services (AWS), and Corsa currently only supports generating
infrastructural resouces consisting of virtual machines. Therefore, it was not
necessary to implement any advanced logic for resolving the infrastructural
requirements.
Future iterations of Corsa should support diﬀerent types of
infrastructural resources, for which a more sophisticated approach might be
required.
However, to optimise the selection of virtual machines in terms of cost, the
strategy employed by Corsa is to generate a list of all possible ways the unfulﬁlled
requirement assignments to virtual machines could be resolved. To do this, Corsa
treats the optimisation as a problem of ﬁnding all the possible partions of a set of
labeled elements, where each block of a partition represents a set of requirement
==PAGE:49==
38
Solution Design
assignments that will be resolved by the same component.
Then, for each
block, the cheapest virtual machine type that can resolve all the requirement
assignments in a block is selected. It is possible to make this selection, since each
requirement assignment contains the aforementioned TOSCA allocation block to
represent the hardware requirements it makes, and each node template in the
repository contains speciﬁcations on its hardware capabilities and its cost. For
each partition, Corsa then computes the sum of the cost of the virtual machines
required to fulﬁll each block, and selects the partition with the lowest total cost.
However, for a set n elements, the number of possible unique partitions is given
by the Bell number, according to The On-Line Encyclopedia of Integer Sequences
(OEIS) [22]. The Bell number grows exponentially with n, and already at n = 15
there are more than a billion diﬀerent partitions4. Therefore, future work should
address this problem to allow for optimisations of the virtual machine selection
of larger systems.
One last detail to note is that Corsa only permits optimisations where all the
components hosted on each machine have a relation to another component
hosted on that same machine. This condition is added to avoid generating
infrastructural architectures where components that have dependencies on each
other are put close together, in terms of the communication links between them.

Chapter 5
Evaluation
This chapter provides an evaluation of how Corsa addresses the problems identi-
ﬁed in Chapter 2. The problem investigated in this report is based on the mental
load experienced while performing a cognitive task, and thus, it is very diﬃcult
to quantify, without performing a large scale experiment with human subjects,
which is outside the scope of the project documented in this report. However, to
provide a basis for validating the stated hypothesis and answering the research
questions (see Section 2.3 on page 8), we chose an approach where we evaluate
the solution on a mix of formal and informal criteria.
To evaluate to which extent Corsa solves the problems, we expand the running
example to provide more realistic scenario with more components. Then, we

Framework (see Section 2.4 on page 9) The second way is by using CRS to
specify the requirements of each component, and then deploying the system
using Corsa, which internally uses EDMM and its Terraform plugin to deploy
the example system to AWS.
Then, we will compare the two methods based on the following criteria:

5.6.1
Review of Research Questions and Hypothesis
Based on the above results and discussion, we review the research questions and
hypothesis and evaluate to what extent the project has provided an answer to
the questions and hypothesis.
==PAGE:64==
5.6 Discussion
53
We start with the ﬁrst research question:
RQ1: How can a language be designed to allow the speciﬁcation of which
infrastructural resources a software system requires?
To answer this question, we provide in Section 4.1 the design of a declarative
language, CRS, which is used to provide the following speciﬁcations for a software
component:
• The capabilities it provides for other components, which we named services
• The dependencies the component has on other services
• The hardware requirements necessary to host the component
• A reference to a Dockerﬁle which builds the component from its source
code
However, we only support the speciﬁcation of one type of infrastructural resource,
namely the virtual computational and storage resources provided by a virtual
machine. While this is a serious constraint, we argue that we have still provided
a partial answer to RQ1, as we have shown how one type of infrastructural
resource requirement can be speciﬁed. Furthermore, since we base our solution
on open modelling standards (TOSCA and EDMM), we argue that our approach
can be expanded to support additional types of infrastructural resources, since
it has been demonstrated in research that these modeling standards do support
diﬀerent types of infrastructural resources. Therefore, future iterations of Corsa
could extend the CRS language to support additional types of resources.
The second research question is:
RQ2: How can such speciﬁcations be used to construct an infrastructural
architecture model?
In Section 4.2 we review how we can utilize concepts from the TOSCA speciﬁ-
cation and the TOSCA Simple Proﬁle to construct a rich model of the input
system that was speciﬁed with CRS.
==PAGE:65==
54
Evaluation
The last research question is:
RQ3: How can the infrastructure in the constructed model be automatically
provisioned?
In Section 4.4 we show how speciﬁcations made in CRS can be transformed in
order to allow the speciﬁed components to be deployed on automatically provi-
sioned infrastructure that complies with the speciﬁed hardware requirements.
Our method is to transform the modelled system into an EDMM deployment
model, and then use that model as input to the EDMM Transformation Frame-
work, which supports transformation of models to Terraform conﬁgurations
targeting AWS.
Based on these ﬁndings, we evaluate how we have answered the hypothesis,
which is stated below:
H1: It is possible to design a software tool which can generate infrastructural
architecture, such that the infrastructural architecture design process is made
easier and less error-prone.
We assess that we have partially validated the hypothesis, as CRS and Corsa
together form a tool which can generate an infrastructural architecture. Thus, the
process of achieving an infrastructural architecture is made indeed made easier,
because the approach of specifying the hardware requirements automates the work
of ﬁnding and selecting an adequate virtual machine type from the cloud resource
provider. Furthermore, the approach is less error-prone because Corsa ensures
that a particular virtual machine satisﬁes the hardware requirements before it
assigns a component to be hosted on it. However, the generated infrastructure
only consists of virtual machines and the containerised software that runs on it.
Additional research into the possibility of generating infrastructural architectures
with more types of infrastructural resources is needed to fully validate the
hypothesis.
5.7
Future Work
The project described in this report aimed to developed a tool which can
generate infrastructural architectures, thereby reducing the workload and risk of
designing infrastructural architectures. The developed tool, Corsa, can generate
architectures consisting of virtual machines and their connections to each other,
based on a repository of concrete types of virtual machines available to use. As
shown in 4, Corsa generates this by parsing component requirements speciﬁcations
==PAGE:66==
5.7 Future Work
55
(CRS), a format which currently only permits the speciﬁcation of the following
types of requirements for a software component:
1. Dependencies on the services oﬀered by other components of the same
system
2. A dependency on the virtual hardware which is hosting the component.
This dependency can be said to be abstract, because no speciﬁc resource
is speciﬁed, only the requirements that the assigned must fulﬁll.
Thus, Corsa only addresses as small part of infrastructural architecture, as there
are many other infrastructural elements than virtual machines, which could
also be included in the generated architecture. For that reason, we argue that
future work on the problem of reducing the workload of designing infrastructural
architectures should focus on allowing the designer to specify requirements on
more types of abstract infrastructure. We argue that the best path to achieving
this is by using the existing TOSCA Simple Proﬁle to provide a library of abstract
types of infrastructural services, which can be then speciﬁed as targets of the
dependencies of a CRS component. Examples include databases, message queues,
and load balancers. Lastly, a repository of concrete types of cloud resources for
all the abstract types in the library should be created for each cloud resource
provider that Corsa should support. Since, Corsa is currently only supporting
AWS, future work could provide a proof-of-concept by extending Corsa with
AWS infrastructural resources other than virtual machines.
Another aspect that could be addressed by future work is the virtual machine
assignment strategy used by Corsa. Currently, the strategy is to generate all
possible partitions of the system components, and then assigning the cheapest
viable single virtual machine to each block of each partition, such that all the
components in a block will be hosted by that virtual machine. Then, the total
cost of a each partition is computed as the sum of the cost of the virtual machines
assigned to each block of that partition. This approach is problematic, as the
number of all possible partitions that can be made with n elements is equal to the
Bell number, which for n = 15 is 1,382,958,545. Therefore, future work should
address this problem by adapting the virtual machine assignement strategy for
larger system topologies. We suggest an approach where a system topology
graph is subdivided into groups such that each group is small enough for all
possible partitions to be computed for that group.
Lastly, future work should address the problems experienced with integrating
Corsa with the EDMM Transformation Framework. Particularly, the “multi
==PAGE:67==
56
Evaluation
deployment mode” of the EDMM Transformation Framework2, proposed in a
paper from 2021 titled Automating the Deployment of Distributed Applications by
Combining Multiple Deployment Technologies by Wurster et al. [9]. This mode
of the EDMM Transformation Framework allows for diﬀerent parts of the system
topology to be deployed with diﬀerent technologies in order to circumvent the
limitations of each technology. For example, the Terraform plugin of the EDMM
Transformation Framework is limited in the sense that can only create virtual
machines and must ﬂatten the topology when transformation a deployment model
into the Terraform format, as discussed in Section 4.4.2.1 on page 42. According
to the aforementioned paper, by using the multi deployment mode, Terraform
can be used to deploy the virtual machines whilst Ansible and Kubernetes can
be used to deploy the components which are hosted on the virtual machine.
If this can be feature can be integrated with Corsa, it might also remove the
restriction that the software components speciﬁed with CRS must be referenced
via a Dockerﬁle.

Chapter 6
Conclusion
The project described in this report aimed to provide a solution to the problem
posed by the diﬃculty of designing infrastructural architectures.
To that end, a software tool, Corsa, and a speciﬁcation language, CRS, was
developed. In CRS, the requirements of each software component of a system can
be speciﬁed. Each component requirements speciﬁcation is then used as input to
Corsa, which generates and deploys an infrastructural architecture that satisﬁes
the requirements speciﬁed in the input. We show that Corsa can reduce the
mental load of designing infrastructural architecture by automating the selection
of virtual machines based on the speciﬁed hardware requirements of the input
system. Furthermore, Corsa optimises the cost of the infrastructural architecture
generated by analysing if it is less expensive to host multiple components on the
same virtual machine. Thus, this project has made the contribution of showing
that it is possible to reduce the problem of the diﬃculty designing infrastructural
architecture by developing new tools for infrastructural architecture design. We
also demonstrated how a new solution can leverage the capabilities of software
tools that are developed in conjunction with existing research on the topic of
cloud infrastructural architecture.
However, the developed tool only addreses a small part of infrastructural ar-
==PAGE:69==
58
Conclusion
chitecture design, namely the selection and assignment of virtual machines.
Furthermore, for Corsa to generate an architecture for a software system, the
components of that system must each be containerized in a Docker container.
Lastly, Corsa is limited by the limitations of the technology it depends on, namely
the EDMM Transformation Framework, although we ﬁnd that these limitations
might be overcome by using a particular feature of this tool. We provide multiple
suggestions for how future work can addess these shortcomings.
Beyond this, we provided a brief overview of the existing literature and industrial
eﬀorts to adress the problem of designing infrastructural architecture.

CRS Language Speciﬁcation
The pseudo-value <dockerfile_path> represents a string that is a path in
the ﬁle system of the codebase to a Dockerﬁle that can be used to instan-
tiate the component.
Alternatively, if the component uses a image from
Docker Hub, a string representing the tag of the image can be given, which is
represented by the pseudo-value <dockerhub_image_tag>. The pseudo-value
<service_declarations> represents a list of CRS service declarations. Lastly,
the boolean value <standalone_value> is true if this component must not be
hosted on the same virtual machine as other components.
The grammar of a CRS service declaration is informally listed below:
1
<service_name>:
2
service_ports:
3
- <service_port_list>
4
service_dependencies:
5
- <name_of_dependency_target_service_list>
The pseudo-values of the above grammar are deﬁned as follows:
• <service_name>: A string representing the unique name of the service
being declared.
• <service_port_list>: A list of integers representing the network port
that this service accepts requests to.
• <service_dependency_target_service_list>: A list strings which matches
the name of a service oﬀered by another component.
• <minimum_cpu_core_value>: An integer representing the minimum num-
ber of CPU cores that must be available exclusively to this service.
• <minimum_cpu_frequency_value>: A real number representing the mini-
mum clock frequency required of the CPU cores assigned to this service.
• <minimum_memory_value>: An integer that represents the minimum num-
ber of megabytes of random access memory this service requires.
• <minimum_storage_value>: An integer value that represents the minimum
number of gigabytes of persistent data storage that must be available to
