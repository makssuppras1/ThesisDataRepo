Abstract
Number of residential proxies as a service has started to grow as an emerging
business, in which a provider utilises the hosts inside residential networks in or-
der to handle network traﬃc problems such as server-side blocking and detection.
However, residential IP proxies are observed to exhibit diﬀerent characteristics
that indicate new challenges for detection. In this thesis, diﬀerent detection
techniques that are trying to detect VPN and proxy services will be examined.
Based on these approaches, similar methods to detect residential IP proxies will
be explored.
Most of the existing analyses and algorithms for proxy detection are based on
the data that either collected on the target server node(i.e., web-server user
trying to reach) or collected on the client node(i.e., proxy user), but not focused
on the middle node(i.e., proxy node forwarding the traﬃc). These approaches
work on the incoming traﬃc of the target server node, along with the outgoing
traﬃc from the client that is created after connection requests . A fair amount
of these approaches make classiﬁcations according to the source IP addresses of
the generated network ﬂow. Aim of this thesis is to comprehensively investigate
residential hosts that are abused by proxy services. End goal of this thesis is to
provide a technique that raises a suspicion if a device is involving in any kind
of network traﬃc forwarding activity in a proxy service as an intermediary.

Chapter 1
Introduction
A Virtual Private Network (VPN) is a technology that creates a secure com-
munication channel over a less secure network [63]. History of VPN dates back
to 1996, when a Microsoft worker developed a peer to peer tunneling proto-
col(PPTP). PPTP supports multi-protocol to be encrypted and encapsulates
traﬃc within IP packets [52]. PTTP can be seen as the predecessor of modern
VPNs since it enables to create more secure and private connection between the
Internet and a computer. As the Internet became more popular, the need for
more advanced security systems was transpired. That is when VPNs kicked in.
Even though VPN technology was initially developed to provide access to cor-
porate applications and resources to remote or mobile users, nowadays, almost
everyone is using a VPN service on their device in order to ensure their pri-
vacy and security. Frequently, VPN services are used to hide the real traﬃc in
the network which may be otherwise not allowed or may be monitored [41]. Al-
though encryption is not a inherent part of a VPN connection, a secure VPN can
be created by using cryptography with no deﬁnition of the data path through
the public network. Source packets are encrypted and encapsulated into new
==PAGE:18==
2
Introduction
packets that pass through the public network. Nobody can read or change the
content of original packets [37].
To sum up, Internet users may secure their connections with a VPN to circum-
vent geo-restrictions and censorship, as well as they can use this technology to
protect personal identity and location to stay anonymous on the Internet [62].
After deﬁning VPN, the concept of "Proxy" should also be explained. A proxy is
a software that resides on a server or a node, and has the purpose of mediating
access between the client’s machine and the destination server [48]. Immedi-
ately upon a client application create a request to get a particular resource, the
request is transferred through the medium of a demarcated proxy server. After
the proxy server obtains the request, the desired resource accompanied by its
machine or appointed server is decided by the proxy server, along with the any
supplementary information that needs to be relayed. Once everything settled
on the proxy server, the request is forwarded to the target server by using the
established connection and the proxy server starts to wait for a response. Upon
fetching the response, it forwards the response back to its end client [55].
As another functionality, a proxy server is a kind of buﬀer between the user’s
computer and the Internet resources user is accessing. Proxy servers accumulate
and save ﬁles that are most often requested by thousands of Internet users in a
special database, called "cache". Therefore, proxy servers are able to increase
the speed of user’s connection to the Internet. The cache of a proxy server may
already contain information user needs by the time of user’s request, making
it possible for the proxy to deliver it immediately. The overall increase in per-
formance may be very high. In addition, proxy servers can help in cases when
some owners of the Internet resources impose some restrictions on users from
certain countries or geographical regions. Among proxy servers, there are so
called anonymous proxy servers that hide user’s IP address thereby saving user
from vulnerabilities concerned with it [29].
As it can be understood from the aforementioned VPN and proxy server def-
initions, proxy servers and VPNs have a mutually shared purpose with some
key diﬀerences. Both of these services allow a user to masquerade his or her
internet traﬃc to seem as it is coming from a diﬀerent IP address. If a user
==PAGE:19==
3
is utilizing one of these services, the source IP address of the incoming traﬃc
on the end node will belong to the intermediate node(i.e., proxy server or VPN
server). Since these intermediate nodes were consisting of data centers, some
web services started to block the network traﬃc coming from them. Moreover,
as the IP ranges of data-centers usually start at the same few integers, they are
easier to block as a whole range.
Consequently, proxy providers started to use residential hosts which have resi-
dential IP addresses. A residential IP address is an IP address tied to a physical
device, for example, a mobile phone or a desktop computer. In contrast to non-
residential IP addresses, residential ones are directly controlled and appointed
by an Internet Service Provider(ISP) [38]. Although having a high level of pri-
vacy over the Internet is a privilege, directing one’s network traﬃc through
another person’s device without the knowledge of the device holder generates a
lot of problems. RESIP proxy services utilizing these residential hosts as their
server, which are recruited with skeptical ways and it comes as a new challenge
in the literature to detect such malware-like applications. The ultimate goal of
this thesis is to detect if a device is involving in such forwarding activity in a
proxy service. Detection will be made by making use of the network traﬃc logs
on the device.
Detailed information on the state of the art is given in Chapter 2. It presents
an overview of the diﬀerent studies regarding the web proxy detection, injected
site detection, anomaly detection in network traﬃc, VPN detection and RESIP
proxies.
Following the state of the art section, investigation on proxy services provided
in the Chapter 3. This chapter comes as a novel work that is revealing how
RESIP proxies are the most sneaky and resilient ones when it comes to detection.
Particularly, their recruitment process of the residential hosts and threats posed
on these users are interesting ﬁndings of this study. In addition, this chapter
explains how existing preventing mechanisms are vulnerable to these services
since they operate on the target server side, as well as why coping with these
proxy networks can only be done by eliminating residential hosts from their exit
node pool.

State of the Art
In order to proceed to the further steps of the thesis, one should investigate
what kind of studies have been conducted so far. This chapter contains informa-
tion regarding the literature studies about web proxies, network traﬃc anomaly
detection, injected web-site detection, VPN detection and RESIP proxies. In
addition, it will answer questions like: What kind of techniques have been de-

have been used to develop these algorithms? What kind of data has been used

useful for detecting suspicious traﬃc on a middle node(i.e., residential host for-
warding the traﬃc) of a residential proxy network? How are these residential
hosts might be infected to be used as proxy servers?

State of the Art
2.1
Web Proxy Detection
Weaver et al. [60] conducted a measurement study to understand the purpose
of free proxy services based on how they modify traﬃc. The study used a proxy
detection approach that was employing a Hypertext Transfer Protocol(HTTP)
client and server under control to exchange precisely known HTTP messages
and then looked for deviations from the expected. The study applied several
tests on the data, such as Non-responsive Server Test, Proxy Traceroute, HTTP
404 Fetches, Customized HTTP Fetch, Non-HTTP Fetch, Invalid Host Field,
Caching and Transcoding, Filetype Filtering, and EICAR Test Virus Filtering.
According to the test results, the study managed to identify eight categories of
web proxies(Antivirus, Caches, Security and Censor Proxies, Transcoding, 404
Rewriters, Login Gateways, Content Injectors, Spyware Proxies, Dark Proxies).
O’Neill et al. [47] investigated the existence of TLS proxies and identiﬁed thou-
sands of malicious TLS proxies intercepting TLS communications. The study
implemented a tool that works by sending a ClientHello message to a TLS-
enabled server, and then recording the ServerHello and Certiﬁcate messages
received in the response. The retrieved certiﬁcate is then compared with the
valid certiﬁcate presented by the server that the client is contacting. If there
is a mismatch, it indicates that some kind of proxy has been intercepted the
client’s traﬃc to that particular server.
Carnavalet et al. [34] proposed a framework for the evaluation of client-end TLS
proxies, by addressing limitations of regular TLS test suites, and adding more
tests speciﬁcally relevant to such proxies. The study found that not a single
TLS proxy implementation is secure with respect to the test results and these
weaknesses might lead to an impersonation under an active man-in-the-middle
attack.
Perino et al. [50] created a distributed measurement platform called "Proxy-
Torrent" in order to monitor the free proxy ecosystem. This platform is used to
make an analysis of 10 months of data spanning up to 180,000 free proxies and
1,500 users. The research revealed that only half of the free proxies provide good
enough performance to be used. Nevertheless, it was also found out that these

best working proxies are free for a reason and they tend to display malicious
behaviors such as ads injection and TLS interception.
Tsirantonakis et al.
[57] have designed a methodology for detecting proxies
which, instead of passively relaying traﬃc, actively modiﬁed the relayed con-
tent. The study has performed a content modiﬁcation detection technique that
compares the Document Object Model(DOM) tree of the web page’s static tem-
plate with the DOM tree of the web page fetched by a proxy. 38% of HTTP
proxies used in the study have shown some form of content modiﬁcation. Addi-
tionally, among tested proxies, 5.15% of them performed content modiﬁcation
behaviours that can be considered as an unwanted or malicious. These malicious
proxies were found to carry out activities such as ads injection and code injec-
tion for collecting user information that can be used for tracking, ﬁngerprinting,
attempting to redirect the user to the pages that contain malware.
2.2
Injected Site Detection
Borgolte et al. [30] established a system to identify changes in the DOM trees
that are linked with malicious and benign behaviours of web sites. Essentially,
the study was helpful to point out why a website is infected by a speciﬁc cam-
paign and it provided a way to deal with such infections caused by malicious
code.
Liao et al. [43] developed a semantic-based technique that is called "Semantic In-
consistency Search" for eﬃcient and accurate detection of the promotional injec-
tions on the sponsored top-level domains with explicit semantic meanings. The
study utilized Natural Language Processing to identify the bad terms related to
the illicit activities(for instance: fake drug selling) that are most irrelevant to
the sponsored top-level domains’ semantics. It was reported that these promo-
tional infections introduce a large semantic gap between the infected sponsored
top-level domains and the illicit promotional content injected.
Li et al. [42] made a study for detecting compromised websites that serve as
malicious redirectors.
In the study, a tool called "JSRED" implemented to
==PAGE:24==
8
State of the Art
automatically identify malicious script hosts that are trying to redirect web
traﬃc. JSRED compared a JavaScript library ﬁle with its clean reference to
extract their diﬀerence, which was considered to be malicious if it was a re-
director. This technique managed to detect most of the compromised websites
with almost no false positive.
Stringhini et al. [54] analyzed how a large and diverse set of web browsers reach
malicious web pages, instead of looking at particular features of a malicious web
page. In this study, the redirection chains from a collection of diﬀerent users
were aggregated and redirection graphs were built, where each redirection graph
showed how a number of users have reached a speciﬁc target web page. When
the properties of the set of visitors of a page and its corresponding redirection
graph were analyzed, distinction between legitimate and malicious pages could
be done accurately.
Nappa et al. [46] studied malicious servers and compromised hosts that listen
for incoming network requests. In the study, probes were sent to a remote host
and responses were examined to decide whether the remote host is displaying
malicious behavior or not. Furthermore, the study built ﬁngerprints for several
malicious families and used them to identify malicious servers and P2P bots
through Internet-wide scans.
Xu et al. [61] made a host-level dynamic binary analysis to automatically extract
built-in remotely-accessible mechanisms in P2P malware, followed by network-
level informed probing for detection. With the help of this two-phase detection
framework, the study managed to detect P2P malware and beyond, as long as

3.2
Dark Side of Proxy Server Usage
As the pool of proxy server users grow each day, there are various reasons
for organizations and individuals to make use of these services. Some of the
companies use proxy servers as a gateway to the World Wide Web in order
to monitor internet usage of their employees.
Forcing the network traﬃc to
ﬂow only through the company’s proxy server implies that company would have
comprehensive network logs which contain information about all of the websites
and contents that are accessed by an employee. This information can be quickly
used to reveal an employee’s misbehavior. In addition to that, there are other
valid reasons to use proxy as company’s gateway to the web, for example, to
improve speed with some network band-with saving(proxy servers can cache the
frequently visited web-sites), to enhance the security of network with the help
of encryption, to enhance privacy of the connection, and to reach out the web
content that is not available in their region.
Even though most of the users of proxy services utilizing these functionalities for
benign purposes, there exists a fair amount of malicious parties that is utilizing
proxy services for evil actions. Examples of these several malicious approaches
will be analyzed for the remainder of this section.
To begin with, anonymous IP addresses are frequently utilized in order to boost
income that is gained from online advertising. Malicious programmers can de-
velop programs or make use of bots to create fabricated online clicks, actions
and data. With the help of anonymous IP addresses, these scammers can change
the source IP address of incoming traﬃc to any desired geographical location
in order to have the correct location for getting payouts from advertisement.
Furthermore, anonymous IP addresses open up a way to dodge network traﬃc
control tools that check repeated views and clicks from one or few IP addresses.
Similar approaches are also utilized to promote products, for example, to en-
courage application installs by increasing number of downloads [25].
Another evil activity that is carried out with the help of proxy servers is to
hide identity when making a money transaction with a stolen bank account in-
formation. Whenever a consumer tries to buy products and services through

web, transactions are being checked by the control mechanisms of the busi-
ness organizations. Suspicious activities regarding the orders are detected by
checking information provided by the customer such as the billing address, the
shipping address and IP address location of the customer. Since these checks
are required by almost all of the online selling services in present times, mali-
cious users with stolen credit card information utilize anonymous IP addresses
to spoof geographical locations near to the billing or home addresses of the

As last example of misuses, by using anonymous IP addresses, users of media

3.3.3
Recruitment Process of Residential Hosts
After seeing such a huge residential host pool from all around the world, consid-
erable amount of suspicions are aroused inevitably. How is that possible to have
an IT device almost in every city in the world to use as a proxy server? It is not
viable to set up enormous amount of IT devices and maintain them everywhere,
even for business leader companies. Therefore, only reasonable explanation be-
hind this scheme is that residential hosts are devices belonging to the organic
users, and yet somehow these users got their devices fall into the proxy server
pool of proxy providers. As a consequence, recruiting ways of these residential
hosts need to be analyzed.
In order to enlighten recruitment process of RESIP hosts, web-sites of all of the
RESIP services mentioned in the table 3.2, have been checked with an attentive
manner. In addition to that, query results obtained from search engines, that
have been revealed after entering keywords such as residential host recruitment,
residential host volunteer, residential proxy exit nodes were examined. Amongst
all of the proxy services provided in the table 3.2, only Luminati was found to
provide an explanation for the recruitment process of RESIP hosts [7].
Luminati has a software development kit(SDK) which is integrated into vari-
ous types of applications. Luminati is making agreements with the application
owners to place this SDK in their application. Users of applications can choose
to opt-in to become Luminati exit node, in return for advertisement free appli-
cation. For every user that opts-in to the Luminati network, Luminati pays a
monthly fee to the application vendor. HolaVPN application, which is consid-
ered as the biggest contributor to Luminati host pool, can be given as one of
the examples. Once user’s machine installed with the free HolaVPN, it will be
recruited as one of the Luminati’s exit nodes [22].
Apart from Luminati, recruitment techniques of other RESIP services remain
unknown. Yet, it is reasonable to assume that they have taken similar approach
but with a more skeptical manner. There were some rumours about Oxylabs,
which later followed by a court case [26], asserting that Oxylabs installs some
malware on to a user’s device, unknown to the user, by bundling it with other

software that the user downloads [6]. This malware makes it possible for Oxylabs
to sell of computing power, bandwith and IP address of the infected device.
3.3.4
RESIP Host Involvement as an Attack Intermediary
As it already mentioned in the section 3.2, proxies are being utilized heavily as
a way of disguise by the malicious users of the Internet. Even with a simple
proxy service usage, attackers can wreak havoc on the Internet. Yet, it poses
less of a threat for the Internet users when it is compared with the usage of
organic user’s device as a network traﬃc forwarding tool, which is exactly what
RESIP proxy services are performing. Now, consider two scenarios: in the ﬁrst
scenario, a culprit commits a crime and leaves a fake id in the crime scene. In
the second, a culprit commits a crime and leaves a stolen id, which belongs to
an authentic person, in the crime scene. Clearly, it is not hard to grasp that
second scenario would lead to more dire consequences. Similar to these scenarios
given above, using someone else’s IP address to hide the real IP address is way
more dangerous and evil than IP spooﬁng, which was already identiﬁed as a
signiﬁcant threat by the several studies in the literature [59] [35] [39].
After revealing the danger of residential host involvement in attacking scenarios,
it needs to be checked that if RESIP providers protect their exit nodes against
this hazard. The study [45] that has been made on the RESIPs, revealed that
RESIPs are abused as attack intermediaries to be part of illegal activities such
as fast ﬂuxing, phishing, malware hosting, blackhat SEO and botnet campaigns.
Surprisingly, even in the RESIP pool of Luminati, which is claiming to have com-
pliance oﬃcers to handle every report of abuse including investigating, warning
and blocking suspicious clients, 2.32% of the RESIPs were found as blacklisted
in the study.
In conclusion, residential hosts are not protected by proxy providers, instead
they are being abused by them, similar to how a botmaster control botnets. On
that account, these IP addresses are the primary choice for evil actors to use
for bypassing geographic ﬁlters, submitting fake data through lead generation
channels, taking advantage of free trials, creating duplicate accounts, engaging

3.4
Proxy Prevention Mechanisms
As already mentioned previously in the report, since the concept of using cap-
tured personal device to forward proxy traﬃc is an unaccustomed one, the exit
node component of the proxy services have not been investigated before in the
literature. Therefore, one needs to check if there is an existing mechanism to
detect suspicious activity on the exit node, provided by some services available
online.
In order to ﬁnd such proxy prevention mechanisms, comprehensive search has
been made through search engines, then results were examined accordingly. It
has been found out that there exists several business services which have been
focusing on VPN and proxy detection such as IPQualityScore [12], IPinfo.io [11]
and IPHub [10]. Detailed information about these services are provided in the
below:
• IPHub: Provides an API which is focusing on proxy providers that use
data-centers for their connections. They have collected a vast array of
non-residential Autonomous Systems and mapped out the large portion of
the web. Data regarding the proxies, is updated regularly based on data
reported from public lists, game server brute-force attempts and website
logs.
• IPQualityScore: Provides a tool having an IP lookup table in order to
analyze an IP address and other forensic factors of the connection. For
each IP address occurring in the network ﬂow, the tool provides a detailed
risk assessment of the click, user, or transaction.
• IPinfo.io: Provides an API that performs full internet wide customary
scans to detect active IP addresses of VPNs. This is combined with data
on public SOCKS, HTTP proxies and TOR exit nodes. According to the
==PAGE:42==
26

obtained data, the tool makes IP usage type classiﬁcation to determine
which IP ranges belong to proxy hosting providers.
In addition to the aforementioned ones, several diﬀerent APIs are available
online, but they do not oﬀer diﬀerent approaches. None of these techniques
focus on residential proxy exit node recruitment process. As a matter of fact,
they do not have suﬃcient mechanisms to detect incoming network traﬃc from
a residential host, unless the host is registered in a blacklist by having involved
some amount of malicious activity beforehand.
3.5
Summary
After seeing the power of the proxies, it can be understood that why they
are preferred amongst organizations and individuals. As the study [44] claims,
VPNs are on the path to becoming ubiquitous in the context of professional en-
vironments and business. Considering that these proxy approaches will be more
prevalent in the future of internet, they have to comply with certain security
measurements for the sake of everyone.
For the free proxy services, as it is identiﬁed in the Section 2.1, most of them
involve in evil ways such as stealing private data from the clients(password,
credit card information, locations, citizenship number). For that reason, they
will die out as users become more aware. Yet, the other proxy approaches will
still be in high demand for the forthcoming years.
All of these proxy techniques, except the RESIP proxies, have certain features
that reveal their connection to the detection tools as a proxy activity. However,
for the RESIP proxies, there is not much can be done on the target web-site
they are trying reach. Even if their connection detected as a suspicious one, it
does not make sense to block their IP address for two main reasons. Firstly,
ISP providers are assigning the same IP address to multiple clients behind a
large Network Address Translation(NAT) pool since the IPv4 pools have been
depleted in many regions. Therefore, blocking an IP address from a target web-

in an instant, traﬃc of the client of the RESIP service will be forwarded through
an another residential host. In that case, blocking the IP address of the RESIP
connection will make the target web-site unavailable for the residential host, but
not for the client which is, in fact, the one that is supposed to be blocked.
Even though some of the providers may claim that the user is consented to be
part of residential proxy network, in reality, the user is unaware of dangerous
consequences of being part of such traﬃc forwarding activity. Therefore, such
privileges should not be given to the applications. As a solution, this study oﬀers
a new approach to handle RESIP proxies. The solution suggests to eliminate
residential hosts from the pool of proxy exit nodes by detecting anomaly on
the device. The study [56] deﬁnes an anomaly as any circumstance that makes
network traﬃc deviate from normal behaviour. This study will aim to ﬁnd such
behaviours on the device to detect residential proxy involvement as a server.

Chapter 5
Residential Host Detector:
Design and Implementation
After wearing the attacker’s cape, a detection mechanism to reveal involvement
of a device in a residential proxy network as an exit node should be created.
Firstly, an algorithm needs to be built in order to disclose network traﬃc for-
warding activity on the device. Secondly, implementation should be made to
generate the detection tool. This chapter provides the information regarding
the algorithm, implementation of the algorithm and integration of the detection
mechanism into the existing system.
5.1
Algorithm
In order to generate an algorithm, simple test cases were observed for the pur-
pose of deﬁning proxy traﬃc characteristics of a residential host in a proxy net-
work. In addition to that, the previous studies [32] [31] [45] that were made on
the residential proxy networks were examined. When deciding a characteristic
==PAGE:64==
48
Residential Host Detector: Design and Implementation
of a proxy network ﬂow on the forwarding component, it is important to choose
the ones that will be existent in every residential proxy service. Therefore, it
is important not to focus on the aspects that will be in the system because of
the implementation choices made in the applications, which were developed to
create test data.
As a rule of thumb of the proxy network traﬃc forwarding characteristic, it is
expected to have a speciﬁc ordering among the packets in the network traﬃc
ﬂow. This type of ﬂow will be referred as "suspicious ﬂow" which is expected to
happen on the residential hosts that are acting as a server in a proxy network.
The suspicious ﬂow is described as follows:
1. From client to compromised host: The client needs to send the in-
formation regarding the resource on the web that the client is trying to
get.
2. Connection between compromised host and target web-server:
The resource from the target web-server of the client needs to be fetched. It
is expected to have some ﬂow of network packets between the compromised
host and the target web-server.
3. From compromised host to client: The compromised host needs to

5.1.1
Suspicious Flow Check
Suspicious Flow Check algorithm requires three parameters as follows:
• device_ip: IP address of the device. It will be used to classify packets as
an incoming or outgoing.
• network_ﬂow: Network logs of the device that contains a list of packets
ordered by their time of arrival. Information about packets such as desti-
nation IP address of a packet, source IP address of a packet, position of a
packet in the network logs will be utilized in the suspicious ﬂow check.
• threshold: Minimum number of packets that need to ﬂow between receiving
the last packet of the request from the client and sending the ﬁrst packet
of the fetched response from the target web-server of the client.
The algorithm starts out by iterating over packets in the network_ﬂow. Firstly,
the algorithm checks if it is an incoming packet. If it is an incoming packet, the
algorithm checks if the last incoming packet had the same source IP address.
Unless a packet is coming from a diﬀerent IP address, the algorithm proceeds
to the next packet iteration. Otherwise, the algorithm checks if it is the ﬁrst
packet that is sent from the source IP address of this packet. If it is, the source
IP address of the packet will be added to the list of IP addresses that sent a
packet to this device before. At the end, the source IP address of the current
packet is assigned into the previous incoming packet IP.
If the packet is an outgoing one, distance to the last occurrence position of the
destination IP of the current packet as a source IP in the network logs will be
calculated. This reveals the information about how many packets were ﬂowed
after this device receives a packet from an IP address and send a packet back to
it. Afterwards, the algorithm jumps to the next packet iteration if it is the same
==PAGE:66==
50

destination IP address with the previous one. Otherwise, it will check if a packet
was sent to this destination IP before. If it is the ﬁrst time, the destination IP
address of the packet will be inserted into the list of IP addresses that received
a packet from this device before.
If the destination IP address of the packet has sent a packet to this device be-
fore and distance to this packet is bigger than the deﬁned threshold, it will be
considered as a violation of the ﬁrst check and this packet will be inserted into
the suspicious connections list with the destination IP, position of the packet,
number of packets ﬂowed during suspicious connection(i.e., distance) and vio-
lated rule. At the end, the destination IP of the current packet will be assigned
into the previous outgoing packet IP.

Data Check

order to calculate the amount of total data sent to the suspicious IP address.

the total data amount is returned.

• time_threshold: Approximate time that is required to make a DNS lookup
after a request is received from a suspicious IP address. It is calculated by

is received from the client regarding the target web-server that the client

The algorithm starts out by iterating over the connections from the suspicious

DNSCheck method is called for calculating the time by passing parameters

The method begins iterating over packets,
starting from the position of the packet that is received from the suspicious IP

or a packet found with a DNS protocol. When there is no DNS lookup found,
the method returns the time diﬀerence value as the maximum possible integer
value. If there is a DNS protocol found during iterations, the time diﬀerence is
calculated by subtracting the time of the packet that belongs to the ﬁrst iteration
from the time of the packet that belongs to the current iteration. After the time
diﬀerence is calculated to make the ﬁrst DNS lookup, the iteration halts and
the method returns with this value.
After the time diﬀerence is calculated, if the diﬀerence is bigger than or equal to
the deﬁned time threshold, the algorithm continues with the next iteration of the
connections from the suspicious_connections. Otherwise, it means that there
is a DNS lookup after receiving the request from the suspicious IP address, and
the connection of the current iteration violated the DNS check. The connection
in the suspicious_connections is updated accordingly.

6.2.1
Discussion of the Test Results
As the aforementioned results revealed, the detection system has shown promis-
ing results under diﬀerent conditions when 3 algorithms considered together,
except for the case with real computers. Comments regarding the checks indi-
vidually, provided below:
• Flow : It was found out as the least accurate one among the others.
However, considering that it is the prerequisite condition for the other
checks, it is sensible to have a large amount of false positives. If this check
was designed to be too picky with suspicious connections, then it means
that there would be considerable amount of extra misses for the further
checks because suspicious connections would not be passed to these checks
at all.
• Data : It was found out to be the one that provides the most of the
accuracy to the detection. Even before conducting the DNS check, the
accuracy and precision of the results increased signiﬁcantly for all of the
ﬁve test data. Also, the data check was the one that resulted with the
least false positives which can be seen as the reason of the increase in the
accuracy and precision.
• DNS : It was found out that the DNS check helped with the accuracy and
precision to the some extent, but not as much as the data check. Still, it
was proven to be more resilient to the false negatives than the data check,
and it rarely missed a correct suspicious connection. For the case with the
real computers, the DNS check was useless because no DNS lookup made
for the web-sites that were already in the DNS cache of the computer
representing the compromised host.

6.2.1.1
Real Computers in Comparison to Simulation Environment
After obtaining such successful results on the simulation environment with the
virtual machines, further testing on a new setup with real computers is decided
to be conducted in order to challenge the detection mechanism. Even though
connection and proxy setup between the two computers are established with the
same approach of the cases with virtual machines, the computer that is acting
as the compromised host has diﬀerent network traﬃc ﬂow characteristics than
a virtual machine. Unlike the simulation environment, in the case with the real
computers, there exists a lot of network traﬃc ﬂow generated by the machine,
beyond our control. Yet, the results for the suspicious ﬂow check and data check
are observed to be similar to the cases with virtual machines, which show that
these algorithms are solid enough and not aﬀected by the uncontrolled network
traﬃc ﬂow.
However, for the DNS check, as it is stated before, the performance decreased
signiﬁcantly because of the DNS caching of the real computer. Since almost all
of the web-sites had visited by the computer beforehand, DNS lookups did not
happen. In reality, apart from the most popular web-sites, some of them would
not expected to be in the DNS cache of a computer. If the DNS cache of the
computer was chosen to be cleared before the tests, then the results would also
not be realistic. Hence, in the real life scenario, DNS lookups are expected to
happen more than the case with real computers, but not for every request as it
was happened in the tests with the simulation environment.
6.2.1.2
Findings Regarding the Data Check Failures
After obtaining the test results, data check failures are investigated to uncover
reasons behind it. Two reasons for the failures are identiﬁed as follows:
• Noise: Whenever the compromised host is fetching the data from the
target web-server for a suspicious connection, if legitimate requests happen
to diﬀerent web resources at the same time, then the data check algorithm
might select the total data amount that is received by one of these diﬀerent
==PAGE:87==
6.2 Evaluation of the Existing System
71
connections. It means that if there is an another connection that makes
the compromised host to receive more data in total than the suspicious
one, this amount is going to be used in the comparison. Therefore, the
data that is forwarded back to the client might not be same data with the
compared data, as a result, suspicious connection will fail to violate the
data check.
• Surplus amount of data from target web-server: For some of the
web-sites such as www.blogger.com, telegram.org and www.aliexpress.com,
it is found out that the total data that is forwarded back to the client
is less than the total data received by the compromised host during the
connection with the target web-server. For such web-sites, it is predicted
that they send big certiﬁcates for the connection establishment. Hence,
the encryption between the compromised host and the target web-site
might increase the data size, as a result, amount of the total data received
from the target web-server becomes bigger than the total data sent back
to the client. Nevertheless, it should be considered as an assumption, not
a certainty, therefore, encryption implementations of these websites could
be investigated more in the future to obtain more scientiﬁc results.
6.2.1.3
Summary
To sum up, the detection mechanism performed well even under the extreme
normal and extreme suspicious network traﬃc in the simulation environment,
which can be considered as a really challenging case. In addition, the detection
mechanism is proven to be consistent when it is used in the real environment,
with the exception of the DNS check. Therefore, it can be concluded that the
detection mechanism will provide a solid groundwork for the further studies to
detect suspicious network traﬃc activity on the device, which is acting as a
proxy server in a RESIP service.

6.2.2
Discussion of the Design Choices
After seeing the results of the tests, which are generated by using the simulation
of RESIP structure, one needs to discuss about the design choices of the sim-
ulation system. In addition, it needs to be discussed that how would diﬀerent
choices might aﬀect the performance of the detection mechanism.
Firstly, in the simulation system, the connection between the client and the
compromised host is decided to be kept open until the response from the target
web-server is forwarded back to the client. As an another option, the connection
between the client and the compromised host could be closed when the compro-
mised host got the request from the client. After the compromised host received
the resources from the target web-server, another connection could be opened
by the compromised host in order to send the response to the client. One can
say that second scenario would be more deterministic than keeping the connec-
tion open all the time throughout the proxy activity, and as a result, it will be
easier to characterize. In addition, since the suspicious ﬂow check evaluates if
there is enough distance between the last packet received from the IP address
and the packet sent back to it, the algorithm would also work as intended for
the second scenario.
Secondly, it was assumed that there will not be network activity between the
client and the compromised host after the request is received from the client and
until the response is sent back to it. But, in reality, RESIP services might be
probing the compromised host during this time. This will result in a suspicious
connection to be seen as a legal ﬂow by the detection system because there
will not be distance between the last packet received from the IP address and
the packet sent back to it. However, this probing approach will provide another
pattern for the detection and will reveal RESIP activity that can be traced back
to the IP address of RESIP gateway. After these IP addresses are revealed, the
connections with these IP addresses can be used for the other checks later on.
In addition, these probes mean that the network band-with of the compromised
host is used for no reason, which instead can be used to forward network traﬃc
to serve clients. Therefore, this approach will not provide any good value to
RESIP services, other than making the network traﬃc confusing for detection.
==PAGE:89==
6.2 Evaluation of the Existing System
73
Another possible diﬀerence between the real life scenario and the simulation,
which is also revealed in the test case with real computers, is that DNS lookups
are made on the compromised host for every request when the request is received
from the client. However, in reality, some of the clients’ target web-sites might
be in the DNS cache of the compromised host and there might not be DNS
lookup for some requests. Thus, this means that the DNS check is not useful
for some connections, but considering that it was the least contributing check to
the accuracy and precision of the detection system, it will not cripple the system
to the great extent. In addition to that, when the client is trying to conduct
malicious activity, since it is unlikely for the compromised host to have this
web-site in the DNS cache, there will be DNS lookups for the most dangerous
activities with a high possibility.
As another design choice, HTTP request objects are sent from the client to
the compromised host. One might argue that RESIP services choose not to
send request objects because it might be unusual to get a HTTP request object
for a host and it could make RESIP service involvement more vulnerable for
the detection. Therefore, it might be preferable to send an url address which
is integrated into a more probabilistic type and then extract the url from it,
on the compromised host. However, even though the simulation system sends
HTTP requests from the client, in the detection mechanism, it has no impact
in ﬁnding out the suspicious connections. Hence, both approaches would work
same in the system.
In conclusion, it is possible that RESIP proxy providers could be engineered
to circumvent our measurements. Yet, in order to keep network traﬃc organic
and undetectable, options are limited for them. For every eﬀort that is made
to escape from detection, it would also mean that network traﬃc will be less
organic. For example, some delays during connections can be introduced, but it
will make their system slower and less preferable among other choices. Hence,
the checks generated with the help of the identiﬁed characteristics of the RESIP
proxies in this thesis, could be tailored for diﬀerent versions of RESIP systems
in order to be utilized in the detection.

Chapter 7
Conclusions
Proxy services are rising online businesses, whose security aspects are being an-
alyzed in the literature for some time. In the study, RESIP proxy structure,
which is the latest approach among the proxy techniques, is researched system-
atically. Since it has not been analysed as much as the other proxy techniques,
study shed light on these unregulated services in detail.
Chapter 2 revealed the information regarding the several proxy techniques along
with the established detection methods.
It demonstrated how several proxy
services were handled with diﬀerent approaches, as well as revealing the fact
that how the Internet can be dangerous if one does not have control over its
device.
Chapter 3 unveiled that when residential proxy services are abused, they can
outmatch traditional public proxies or even anonymity networks to assist their
clients masquerade as clean and benevolent sources to communicate with the
targets. Therefore, their security implications were investigated in this chapter
as a novel work. It has been concluded that these RESIP services are almost
==PAGE:92==
76
Conclusions
unstoppable on the target web-server side. For that reason, eﬀective solutions
should be developed to mitigate their security risks.
Chapter 4 explained how the system architecture is set up in order to mimic
RESIP proxy infrastructure. It showed that how data is generated from scratch
for testing purposes since there was no existing data from a residential host side
of a proxy network in the literature. For that reason, data generation itself can
be seen as a technical contribution.
Chapter 5 and Chapter 6 demonstrated the algorithms for the detection, as well
as the experimental results of the detection mechanism. Results have revealed
the strengths of the algorithms by comparing them.
In conclusion, even though the detection mechanism was able to ﬁnd suspicious
connections with a decent accuracy and precision, further research on RESIP
services is needed to get more comprehensive and complete system in order to
detect a residential host involvement in a RESIP service. Moving forward, these
uncontrolled RESIP services indeed endanger the Internet users and certain
regulations are necessary.
==PAGE:93==
Chapter 8
Future Work
As an improvement to the existing detection system, suspicious network traﬃc
activity can be traced back to the application. With the help of the operat-
ing system command netstat1, network traﬃc activity can be recorded for each
application id. Therefore, by utilizing the information such as source port num-
ber, destination port number and time from the network traﬃc logs recorded
in the compromised host, application id that sends the suspicious packet can
be detected. After the application id is found, it can be tracked back to the
application. As a result, the application that makes the device to be recruited
by RESIP services can be identiﬁed and deleted.
As an another future work, in order to obtain a real RESIP network traﬃc, a
device can be compromised on purpose by installing one of the applications that
RESIP services integrated their SDK. But, this might be dangerous to work on
because these services are prime targets for users with evil intentions, as it is
revealed in the study. Hence, all of the necessary security mechanisms should

For the further characterizations of proxy applications of the RESIP services,

malwares. As a result, correlation can be established between malwares and
residential host application of RESIP proxy providers.
Lastly, the detection mechanism implemented in this project requires long run-

consuming manner considering that recordings would have much more packets
for longer time durations.
