Abstract
The growing use of open-source libraries in applications coupled with the size
of these presents a problem of manually verifying the nonexistence of hidden
functionality. The injection of hidden functionality into libraries is specially a
problem in the popular package manager for JavaScript npm, because of the
many long dependency chains between the contained libraries. There is thus
a need for automatic tools to verify the nonexistence of hidden functionality
in npm libraries. The aim of this thesis has been to investigate methods to
perform such a veriﬁcation of JavaScript code. The focus has been on detecting
authentication bypass backdoors. A path sensitive dataﬂow analysis using an
automata to model the states of the authentication ﬂow has been designed and
implemented. The tool is able to detect real-world backdoors rewritten to a
subset of JavaScript, which is general enough to also apply to other programming
languages. The tool however needs further development as it is limited by the
time complexity and manual input regarding authentication features.

Chapter 1
Introduction
It has become increasingly common that modern software applications use open-
source libraries. There are many advantages of this; they are free; developers
may save hours on implementing the library features; they are ﬂexible, partly
because the source code is public, so developers can read the code and perform
modiﬁcations, but also since the maintainers of the library can be requested to
add features or ﬁx bugs. However, the lack of control with the published code
opens up to exploitation. This coupled with the size of the libraries presents
a problem of how to verify that the code does as expected and not anything else.
This is particularly a problem in npm, which is the most popular package man-
ager for JavaScript. This is due to multiple reasons; the popularity of JavaScript,
the large amount of libraries on npm and the many transitive dependencies be-
tween these libraries The transitive dependencies enables an attacker to spread
hidden functionality from a library with few standalone downloads by having it
integrated in a dependency chain for a more popular library. We present some
numbers in order to support theses claims; JavaScript is currently the most used
language in Github repositories1, it has 1.015.213 libraries with 11.421.572.742
downloads last week2 and in 2016, half of the npm packages had more than
22 transitive dependencies and yet half of those had more than 95 transitive

2Source: https://www.npmjs.com/, 24/06-19

dependencies[DMC18].
Hidden functionality in npm libraries is already a problem in the real world. In
recent years there has been a wave of attacks originating from malicious code
being introduced in libraries that are part of dependency chains for popular
npm libraries [Hun18], [Cla17]. The malicious code in the attacks aims to steal
sensitive data such as cookies, credentials and private keys.
The introduction of Node.js an engine for executing JavaScript backend ap-
plications means that hidden functionality becomes an even larger problem,
as JavaScript has diﬀerent levels of privilege based on where it is executed.
JavaScript has limited privilege when executed in the browser, which means
that it will not be able to access the ﬁle system directly amongst other things.
However, when running in Node.js applications per default have free reign over
the OS resources. This means that if anyone is able to inject malicious code into
a Node.js application it can have dire consequences. Thus, it is extra important
for backend applications to ensure benign code.
Since JavaScript originated as a scripting language for creating frontend applica-
tions, most research within malicious code detection has been focused on client-
side attacks such as XSS and drive-by-download attacks [FASKC12], [WCW16],
[XZZ13]. To the best of our knowledge there is no research within malicious code
detection in JavaScript libraries. Thus the focus of this thesis was to attempt to
solve the problem of hidden functionality in open source software by analyzing
how to detect it and implementing it in a tool. We considered many approaches
before we decided to focus on how the more classical backdoor of authentica-
tion bypass can be detected in JavaScript. We designed and implemented a
path sensitive dataﬂow analysis to verify that for each possible execution path,
authentication is only granted if a valid password is supplied. We found that
many of the features which are speciﬁc to JavaScript are quite complicated to
analyze, thus we ended up supporting a subset, which is not necessarily speciﬁc
to JavaScript. This means that the analysis is more general, and will apply to
many programming languages.
The thesis will be structured in the following way: At ﬁrst we present the rele-
vant background material. This will be a short introduction to static analysis,
JavaScript and a selection of backdoors discovered in the real world. We will
brieﬂy mention the relevant previous work within malicious code detection in
JavaScript and backdoor detection in general, since as far as we are concerned,
we are the ﬁrst to explore backdoor detection in JavaScript. We will then present
the design of our analysis of a subset of JavaScript. The design of the analysis
has formed the basis of an implementation in Scala, from which we will present
the results. Next, we will discuss the applicability of our tool along with general
ideas on how to detect backdoors and how to ensure the security of open-source
software in general. Finally, we will conclude on our analysis and its usability
in terms of ensuring the safety of open-source libraries.

2.1
Injecting malicious code
There are multiple ways of injecting malicious code into npm.
One way is
to create a useful package and embed malicious code within it. The package
would have to gain popularity within the npm community or become a transitive
dependency of a popular package in order to have impact. Other ways exist,
which do not require the development of a useful library, this will be the topic for
this section. Thus in this section we will motivate the need for better control of
the published libraries by describing various techniques to inject malicious code
into npm.
==PAGE:14==
4
Background
A method to inject a malicious package into a dependency chain is through
typosquatting. Typosquatting is when an attacker publishes a package with a
name similar to the name of a valid package, in order to create confusion between
the two packages. An example of this happened with the npm package cross-
env, where an attacker uploaded a package with the name crossenv[nB17a]. This
meant that any project referencing the wrong package, would send environment
variables containing passwords etc. to a server controlled by the attacker. This
package accumulated 700 downloads before being recognized as an attack and
removed from the registry. npm responded to this by introducing a naming pol-
icy which forbade names that could be hard to distinguish[nB17b], thus making
typosquatting harder. The problem is that the rules are not exhaustive.
Another mean to inject malicious code into a package is by taking over the ac-
count of the person owning the repository. This can be done by either stealing
the credentials due to weak credentials, reuse of credentials from compromised
websites, phishing or spyware or simply by social engineering. We will support
this claim by presenting examples of this from the real world.
In 2017 it was found that 52% of all the npm accounts either deployed very weak
credentials or leaked the account password in the source code [Cim17]. It was
estimated that if an attacker had compromised all those accounts, she would
control 13% of the entire npm ecosystem!
In 2018 a case of credential stealing was seen with the npm repository eslint.
The attacker uploaded malicious versions of some packages, which on installa-
tion would execute code stealing the npm account credentials of the installer
[Blo18]. Which caused further credentials to be stolen.
An example of obtaining the credentials of the maintainer willingly was seen
with the npm package event-stream[Hun18]. Here the maintainer gave owner-
ship of his library event-stream to another person, because he did not use the
package anymore and as he said, he did not get paid to maintain it, so he had no
motivation to continue maintaining it. The other person responded to obtaining
the repository by uploading a new version with unwanted functionality in. We
will describe this speciﬁc backdoor in more details in Section 2.4.
In conclusion there exists many ways to compromise an npm package. This
leads to the question of which eﬀorts npm are currently employing to ensure the
safety of the uploaded code. In May 2018 npm audit was added to npm3. This
tool gives a vulnerability report over known vulnerabilities in all the libraries
which a project depends upon and suggests to update, but it will not discover
any new vulnerabilities. The vulnerabilities are added manually by the npm se-
curity team, which uses various sources such as user reports to get insights into

2.2
Static analysis
The purpose of static analysis is to reason about program-behavior at compile
time as opposed to dynamic analysis which reasons about program-behavior
by observing an execution of the program.
The obvious advantage of static
analysis is that it can consider all paths through a program, not only the ones
that are executed during a dynamic analysis. While the immediate drawback
is that it can be computationally intensive and it is proven that any non-trivial
question about a program is undecidable, thus a static analysis needs to over-
approximate the behavior of a program and sometimes output a larger set of
solutions [NNH99, p. 2]. This is performed in order to avoid the problem, which
other methods e.g. model checking suﬀers from, the inﬁnite set of states which
a program can be in. A static analysis is typically evaluated in terms of sound-
ness, precision and time/memory complexity [Fel15].
If a static analysis tries to establish whether a property holds then;
Soundness means that if this property holds then it will be detected, but it
can possibly return false positives.
Precision means if the analysis says that the property holds, then it also holds,
but it can possibly return false negatives.
Time and memory complexity describes how well the analysis scales with
larger programs.
2.2.1
Data ﬂow analysis
We will provide a short introduction to how data ﬂow analysis is usually con-
ducted. The classical data ﬂow analysis approach requires a control ﬂow graph
(CFG) and a lattice of ﬁnite height. The lattice describes the abstract infor-
mation the analysis associates with each program point [MS18, p. 41], which
can also be referred to as the abstract state. The usual approach to data ﬂow
analysis is constraint based. A set of constraints describe how the statements
in the program change the lattice, these are typically called transfer functions.
In order to combine information from diﬀerent execution paths, a combination
operator for the lattice needs to be deﬁned. This is either ∩or ∪depending on
whether the data ﬂow information must be true for all execution paths or just
4Source: https://www.npmjs.com/policies/security, 27/06-19
==PAGE:16==
6
Background
for at least one. In a forward analysis the data ﬂow facts from the predecessors
in the CFG are combined using the combination operator and then the transfer
function is applied to the resulting lattice. Where a forward analysis means that
the analysis processes the CFG in the same direction as the execution would,
as opposed to a backward analysis which processes the CFG in the opposite
direction. This means that information about which lattices came from which
paths is not kept [NNH99, p. 66-72].
The transfer functions or constraints are derived from the CFG, a solution to
these needs to be computed.
If the transfer functions are monotone then a ﬁxpoint algorithm can be used
to ﬁnd a solution to the constraints. This is an algorithm that keeps applying
the transfer functions until a ﬁxpoint has been reached. It is guaranteed that
a ﬁxpoint will be reached, because the lattice has ﬁnite height and the transfer
functions are monotone [NNH99, p. 74-78]. The intuition behind this is that
one can only ascend a ﬁnite number of times in the lattice.
A problem with the above solution is that it loses precision when data ﬂow
information is combined across the diﬀerent paths. Such an analysis is referred
to as a path insensitive analysis. A diﬀerent way of computing the solution
to the transfer functions, that keeps the information of which execution paths
are actually valid, is referred to as a path sensitive analysis. It does not use a
combination operator to combine lattices. Instead, at each program point a set
of lattices corresponding to the information from diﬀerent paths are saved. The
obvious problem with this approach is that termination cannot be guaranteed.
If there are an inﬁnite number of paths through the program, the analysis will
not terminate [NNH99, p. 78-79].
2.3
JavaScript
JavaScript is a programming language that ﬁrst appeared in 1995 for use in
web programming in the Netscape browser. A standard was created called EC-
MAScript to enable other vendors to implement JavaScript for their browsers.
Today, JavaScript is the standard language for browser programming.
ECMAScript does not have a universal implementation. It is implemented dif-
ferently in diﬀerent engines. The most famous engines are V8 and Spidermon-
key. V8 is created by Google and used in both Node.js and Chromium-based
browsers such as Google Chrome. Spidermonkey is created by Mozilla and is
primarily used in their browser Firefox.

2.3.1
JavaScript basics
JavaScript has evolved substantially over time. By adding new features with
almost every major release the language has grown considerably in size. To
give an idea of how much the language has grown, the original ECMAScript
speciﬁcation from 1997 was 110 pages [ECM97], while the newest speciﬁcation
from 2018 is 805 pages [ECM18].
We will now give a brief overview of some selected features in JavaScript. We
will start by presenting the more basic features, where some diﬀers from more
classical programming languages. Then we will present features, which are rel-
evant to an analysis of npm libraries as they are commonly used.
2.3.1.1
Scoping
Scoping in JavaScript happens via functions. Each JavaScript program has a
global scope, which all subscopes have access to. Each function has their own
scope, which is a superset of the global scope. If functions are deﬁned inside
other functions, they will be the superset of the child function. This creates a
"scope chain" between functions.
The var-keyword is used in order to declare that a variable inside a scope should
not be visible in the encapsulating scope. If a variable is declared without this
keyword, it will be declared in the global scope rather than the current scopes
[ECM18, s. 13.3.2].
Functions declared as f = function(args){...} will be deﬁned in the global
scope, but can still access variables from the containing scope. This is called a
function closure.
Usually programming languages have block scope, thus variables declared in-
side (if/else/while) will only be visible in their own block. This was not the
case in JavaScript until ECMAScript v6 [ECM15] where the let keyword was
introduced, which is a counterpart to var, that deﬁnes functions only available
to the current scope.
2.3.1.2
Boolean conversion
JavaScript performs implicit type conversions as needed [ECM18, p. 95]. This
means that if variables are used in places where the type is incompatible, an
implicit type conversion will be made.
In the case of the boolean type, all
non-boolean variables will be converted into either true or false. EcmaScript
==PAGE:18==
8
Background
[ECM18, p. 97] provides tables describing these type conversions. For example
for numbers, if the number is 0 or NaN the type conversion will return false,
otherwise true.
2.3.1.3
Undeﬁned
The undefined keyword is used for variables that has not yet been assigned a
value[ECM18, s. 6.1.1]. When one would expect an error in a classical program-
ming language, undefined might instead be returned. This is not an error in
the sense that it does not cause program termination. As an example, a variable
will be assigned to undefined, if it is assigned to the return value of a function
call, which does not return anything. undefined can be cast into various types
such as string or boolean, where it would get the values "undeﬁned"[ECM18,
p. 102] and false[ECM18, p. 97]. This means that operations on undefined will
not throw an error, but the result of the operation might be unexpected by the
programmer.
JavaScript engines will, however, throw exceptions in some cases, so exception
handling is part of the language. For example a ReferenceError exception
will be thrown if a variable is read on an object which is uninitialized [ECM18,
s. 6.2.4.8].
2.3.1.4
Variable assignments
Assignments in JavaScript have side eﬀects in the sense that when an expression
is assigned to a variable, the expression will also be returned. This means that
assignments can be used in other expressions, for example as conditions [ECM11,
s. 11.13.1]. This is quite common in programming languages. The diﬀerence in
JavaScript is that variables do not have to be explicitly declared before values
can be assigned to it.
2.3.1.5
Advanced features
Objects in JavaScript are collections of properties or key/value pairs. Objects
can be dynamically extended with properties and methods. Thus even though
ECMAScript v6 [ECM15] introduced a class notation, JavaScript is not a class
based language such as Java or C++. Inheritance and shared properties can
be modelled using a prototype architecture. When an object is created using
the new keyword it will automatically get a prototype. If no prototypes are
explicitly stated it will get the default prototype. An object created using a

constructor will have an implicit reference to the prototype of its constructor.
If this prototype also has a prototype, they form what is called the prototype
chain. Thus when a property on an object is accessed, the value will be the
property found in the ﬁrst object in the prototype chain [ECM18, p. 51]. It is
possible to overwrite prototype functions, hereby changing the functionality of
a method for all objects created with this prototype.
In JavaScript, functions are ﬁrst class objects so they can be assigned to vari-
ables, used as arguments to functions and be returned from functions. Gallaba
et al. [GMB15] investigated the usage of callbacks and found that every 10th
function takes a callback argument. It can be a challenge to static analysis,
because the callback may be called at any time during the function execution.
Thus the call graphs become non-linear. In Node.js, callbacks can be used to
serve multiple clients simultaneously as they allow asynchronous execution. In
ECMAScript v6 [ECM15] promises were introduced, which enables chaining of
callback functions.
Finally, for completeness we shall mention dynamic code evaluation as this is
a central feature in JavaScript.
Code can be dynamically executed through
the eval construct. Thus any string representation of JavaScript code can be
evaluated with eval. A pattern often used is loading code via Ajax calls and
dynamically evaluating it using eval [Jen13].
2.4
Backdoors
Schuster et al. [SH13] presents the following deﬁnition of a backdoor:
"A backdoor is a hidden, undocumented, and unwanted program or a program
modiﬁcation/manipulation that on certain triggers bypasses security mechanisms
or performs unwanted/undocumented malicious actions."
The deﬁnition does not state whether the backdoor was intentional or uninten-
tional. A backdoor might arise as the result of a bug in the code unintentionally
left by the developer. An intentional backdoor will be left by someone with
access to the code in order to perform malicious activities.
We will in this thesis mainly consider intentional backdoors. These are typically
characterised by the fact that they will not trigger for everyone. The person
placing the backdoor is not interesting in anyone ﬁnding it, since this will prob-
ably lead to the backdoor being removed. Thus an intentional backdoor will

This section will provide a short introduction to our chosen approach to detect
backdoors. We considered multiple approaches to backdoor detection, which we
will elaborate on in Section 6.3.
In order to design our tool we mainly considered the backdoors in the linux kernel
and in PHP presented in Section 2.4 and the artiﬁcial backdoors presented in
Section 3. These backdoors have in common that they are diﬃcult to discover,
through for example white box testing, as speciﬁc properties must be present
for the backdoor to trigger.
This lead us to focus on a more general approach, where we check whether
an execution ﬂow through the program exists where authentication is granted
without a prior successful password check.
This is opposed to searching for
odd properties in authentication code, like the comparison of a variable with a
hardcoded IP address. This will allow the ﬁnal solution to be general enough
to work for all considered backdoors.
In order to detect a backdoor, we basically want to perform a form of program
veriﬁcation. We want to verify that a certain property holds for every possi-
ble execution path, namely that authentication is only granted if the password
check succeeds. We got inspiration from [DLS02] on how to to conduct such
an analysis. They analyse the GNU C compiler to verify a safety property con-
cerning the use of fopen and fclose. They conduct a path sensitive analysis in
order to determine whether all calls to fopen happen when the ﬁle has not been

opened yet, and all calls to fclose happen after the ﬁle has been opened. In order
to trace the state of the ﬁle, they introduce a Finite State Machine (FSM) and
add which state the FSM is in to the abstract state they use in their analysis.
They conduct a complicated path sensitive analysis in order to make it scalable
enough to run on the full GNU C compiler, which is 10.000 lines of code. Thus
they use many optimizations etc, which are infeasible for us due to the time lim-

Analyzing objects is complex. The primary issue with objects in static analysis
is that, when a function is called on an object it is not directly apparent, which
function is called. Consider line 20 in Listing 4.1, where the function ﬁndOne
is called on the User object. The analysis needs to discern which object User
is an instance of in order to ﬁnd the implementation of ﬁndOne. This involves
traversing the prototype chain in order to ﬁnd the ﬁrst object, which contains
an implementation of ﬁndOne. As JavaScript has function pointers, ﬁndOne
might be an alias for another function, which then needs to be known by the
static analysis in order to determine the implementation of the function. An
analysis that is able to ﬁgure this out is called a points-to analysis or pointer
analysis. Due to the intricate workings of the prototype based notion of objects
in JavaScript, this analysis is quite complicated. Feldthaus [Fel15] dedicated his
entire Ph.D thesis to the topic of Pointer analysis for JavaScript.
Finally, callback functions adds complexity to a static analysis.
The second
argument to ﬁndOne is an anonymous callback function, which is called back
somewhere in the implementation of the ﬁndOne function. In static analysis this
causes two problems; 1) It would require a points-to analysis in order to correctly
analyze this and 2) correctly handling when a function callback is executed is
quite complicated, as function callbacks are commonly used for asynchronous
programming. If we consider ﬁndOne, it looks up the user email address in
the database in order to ﬁnd the corresponding user. The code for this is not
included in the listing. When the database query has been executed, the func-
tion calls back the callback function. Moreover the entire call User.ﬁndOne is
inside a callback function, which is supplied as the argument to the function
call router.post. The analysis would have to correctly model the execution order
of the callbacks in order to perform a sound analysis of this code. This is not
trivial.

supported features are general and applies to many programming languages.

4.2.2
Transforming Boolean Expressions
In Section 4.3 we will explain that knowing whether a path is feasible is impor-
tant to our analysis. This means that when our analysis encounters a conditional
node, it needs to be able to conclude whether the condition evaluates to true or
to false.
This means we need some form of constraint reasoning. One way to do this
would be to have the path sensitive analysis handle this, which it could do us-
ing binary decision diagrams (BDD). Where one uses a binary tree to represent
composite boolean expressions in order to evaluate it given the truth values

4.3.2
Flow functions
The path sensitive analysis needs to know, which execution paths are feasible
through the program by determining for each conditional node, what the condi-
tion will evaluate to. This requires that control ﬂow and data ﬂow information
is combined. Das et al. [DLS02] presents a way where they specify a ﬂow func-
tion for branch nodes and supply this with an extra parameter in the form of
a boolean. If the boolean is equal to "true" the ﬂow function will evaluate the
condition for the true successor, and return an empty set, if the condition can
not evaluate to true in the given abstract state and likewise for the false succes-
sor. Thus they need to apply this ﬂow function twice for each conditional once
with the boolean equal to true and the other with the boolean equal to false.
Møller et.al. [MS18] presents another way, that does not require the transfer
functions to consider control ﬂow information. They perform a path sensitive
dataﬂow analysis where they enumerate all possible paths in the program. This
is done by considering all combinations of the variables in the program, thus it
will result in 2#variables paths. They then introduce assert statements inside the
branches of conditionals. If the conditional node contains the expression x, then
the true successor of the node will be assert(x) and the false successor will
be assert(!x). The transfer function for the assert node will then return ∅if
the values of the variables in the abstract environments contradicts the asserted
values. In this way the transfer functions do not need to know about the control
ﬂow, while at the same time paths that are not feasible inside a branch will not
be analyzed in it.
The obvious problem with this approach are the number of paths that are con-
sidered, it is unlikely that all combinations of variables will result in diﬀerent
==PAGE:44==
34
Analysis
execution paths.
Thus there will be much redundant computation.
We de-
cided against this approach for this reason as well as the fact that one needs
to introduce two new nodes for each conditional. This seemed like unnecessary
overhead.
Thus we preferred the approach presented by Das et al. Where the analysis start
with only one abstract state, and whenever a conditional node is encountered,
the abstract state is split into two and the condition is mapped to true in the
abstract state which is propagated to the true successor and to false for the false
successor.
We deﬁned a set of functions using the notation deﬁned in [NNH99]. The for-
mal speciﬁcation of the functions can be seen in Figure 4.2. We shall also refer
to these as ﬂow functions, they are akin to transfer functions, because they
describe how nodes are transforming abstract states. As they take this extra
parameter in the form of the boolean value, we do not refer to them as transfer
functions. A ﬂow function f takes a label and a boolean, lookups the command
and applies the corresponding function to the input abstract state. It returns a
set of abstract states. Labels are the nodes in the CFG.
The ﬂow functions use an evaluation function B, which takes an expression and
evaluates it in the abstract environment. We will further explain this function
later on.

Abstract evaluation
The abstract evaluation function B takes as input an expression and an abstract
environment and maps the expression to a value in the set of boolean values. B
becomes quite simple as we have dealt with constraint solving, when we created
the CFG. Otherwise the evaluation function would have been more complex. It
would have needed to evaluate all supported operations in the abstract domain.
Which required deﬁnitions of the operators on the set of abstract values to be
speciﬁed. The consequence of dealing with this on CFG level is that B only
contains three rules:
(1) If the expression is a boolean constant, it just returns this.
(2) If the expression is a variable and it already exists in the abstract environ-
ment, the abstract value it is mapped to is returned.
(3) Any other expression maps to {true, false}, as our analysis cannot handle
composite boolean expression where the operator is not AND or OR.

we will not consider programs, which cannot execute in Node.js without errors.

undeclared variables, but this is not the aim of our tool and other analysis tools
exists, which can ﬁnd and report the use of undeclared variables.

4.3.3
Algorithm
This section will cover how to compute the analysis information. The general
idea is to employ a ﬁxpoint algorithm. Nielson et.al. [NNH99, p. 75] presents
a generic worklist algorithm. It exploits the fact that when a ﬂow function has
been applied to a node, and a ﬁxpoint has not been reached, this change can
only aﬀect the successors to that node. So it is only the ﬂow functions for the
successor nodes that needs to be applied.
The pseudo code for our implementation of the worklist algorithm can be seen
in Algorithm 1. The analysis takes as input a CFG constructed as described in
Section 4.2. The algorithm generally works by maintaining two data structures;
a worklist, W, deﬁned as W : List[AbstractState × V ∪Vbranch], and a map
Analysis deﬁned as Analysis: V ∪Vbranch →P(AbstractState).
Initially in Step 1, the ﬁrst node in the CFG along with an abstract state with
an empty environment and the initial state in the automata is added to the
worklist. Analysis is instantiated with the empty set for each node in the CFG.
In Step 2, the main loop keeps dequeuing nodes until the worklist is empty.
When a node n is dequeued from the worklist, it is checked whether n has one
successor or two successors. This determines whether the ﬂow functions need
to be applied once or twice.
If the node only has one successor, we apply the ﬂow function once for the
abstract state dequeued from the worklist. Then the analysis tries to propa-
gate the resulting abstract state to the successor node by calling the function
AddToWorklist. AddToWorklist checks whether Analysis already contains the
abstract state s for the node. If not, s is added to the Analysis information for
the node and s along with the node is added to the worklist. It should be noted
that s, if its not ∅will be a singleton, as the ﬂow functions either return a single
state or ∅. Thus we are only enqueuing a single state.
If n is a node with two successors, we need to apply the ﬂow functions twice
for the abstract state from the worklist. AddToWorklist is called once for each
successor in order to propagate the output state to that node. If one of the ﬂow
functions returned ∅, no information is added to the node and it is not added
to the worklist. This is to ensure that infeasible branches are not taken.
Finally in Step 3, we present the analysis information. Here we check the ab-
stract states for each node in the CFG. We use snd to extract the second element
from the abstract state, ie the monitor. If it has reached an accepting state,
then this means that a violation has occurred and we add the abstract state to
the output.
Our algorithm diﬀers slightly from the one presented by Nielson et al. [NNH99,
p. 75]. Their worklist consists of edges, where ours is pairs of an abstract state
and a node. It makes sense to use edges, when the analysis is not path sensitive,

4.3.6
Complexity analysis
We will now look at the time and memory complexity of the algorithm note that
these are the same. The worst case for our algorithm would be if all nodes were
analyzed for all abstract states. The number of abstract states are bounded by
how many variables there are in the program, as each variable can have up to
3 diﬀerent values, we have a total of 3X variable combinations where X is the
number of variables. This gives us a complexity
O(|3X| ∗|V ∪Vbranch|)
It should however be noted that the worst case complexity is typically far from
the average case complexity. This is because it is unlikely that all variables in
a program occur in conditionals.

analyzed in the same context. This is the approach used by Jensen et al. in
their analysis of JavaScript[Jen13].

have to analyze the function again, we can just take the abstract states in the
summary table and propagate to the after-call node. The summary table is
created by associating each abstract state obtained while analysing a function
with the abstract state at the entry to the function. When the function exit
is reached, a mapping from the entry to the exit abstract state is added. This
process will be described in more details in Section 4.4.7.

variable x, in the context given in the argument to {false}. This is because
JavaScript maps undeﬁned to false as mentioned in Section 2.3.1.3.
4.4.6.3
Call
One might specify a ﬂow function for function call and another for function
entry, but since any ﬂow function needs information from both nodes in order
to perform the required steps, we only use one ﬂow function, which handles both
nodes.
At function call the analysis needs to create the abstract state to propagate
to the function entry. This includes two steps: First it must determine which
variables are visible in the scope of the function. This is simple as all functions
are deﬁned in the global scope, thus it will just be the global variables that
are accessible. If we allowed nested function deﬁnitions we would have to keep
a mapping from a function deﬁnition to the context it was deﬁned in. This
would require that we performed the control ﬂow analysis simultaneous with the
data ﬂow analysis, as we would have to have a context, when we processed the
function deﬁnition. This required more comprehensive changes to our analysis,
which time did not allow. We avoided this by requiring that every function is
deﬁned in the global scope, as then we know that the context a function was
deﬁned in, is the global context. So instead of having a mapping from each
function to the contexts it is deﬁned in, we just save the initial context and use
this as the deﬁnition context for each function.
The second step, the ﬂow function for function calls need to perform, is to map
the formal parameters to the function to the given arguments at the function
call.
The ﬂow function for call nodes as seen in Table 4.5 has the label [p(a1, ..., ak)].
As described above the ﬂow function creates a new abstract state, Sfunc, where
the monitor is equal to the monitor at the abstract state at the call-site. Then
all the parameters to the function are mapped to their value in the function

the call-site, a variable mapping is added to the abstract environment δfunc.

If n belongs to V ∗, the case is almost as in Algorithm 1. We apply the corre-

the function in this context. Thus sfe is added to the worklist with itself as

AddToWorklist.
In either case we need to add the abstract state at the call-site to the analysis
information of the corresponding after-call node. This is why we needed the
special edge from the call node to the after call node. There are two reasons for

be propagated to the after-call node, so they are not lost at the function call.
2) The after-call node needs to tie the call-site context together with the func-

The output from the after-call ﬂow function is propagated to the successor node
of the after-call node by calling AddToWorklist.
This might seem like a lot of work in order to ﬁgure out in which context the

new abstract states are created at function exit.
The ﬁnal case matching concerns function exit nodes, if the node is a func-

straightforward, a mapping for the function identiﬁer from the context, from
the worklist, to the abstract state, from the worklist, is created. A function
exit node may have multiple successor nodes, if the function is called multiple
times, thus for each of those successors we call AddToWorklist with the function
context and the unmodiﬁed abstract state.

Discussion
We will begin this section with an evaluation of the path sensitive analysis in
terms of soundness, precision and time/memory complexity and suggestions on
how to improve the precision and the complexity. This will be followed by a
discussion of how usable the tool is, especially considering the input and the
output. Next we will look at a broader scope and discuss how backdoors can be
detected statically. Finally we will discuss other measures to ensure the security
of open-source libraries.
6.1
Evaluation
6.1.1
Soundness and Precision
Nielson et.al. [NNH99, p. 60] proves the correctness of an analysis by proving
that the analysis safely approximates the program behavior in each step of the
analysis. This is proved using a proof by induction on the shape of the inference
tree, which is used to describe the semantics deﬁned for each program construct.
JavaScript does not have a formal semantics, thus a formal proof of correctness
can not be given. We however have such a small subset of JavaScript, when just
considering the intraprocedural analysis, that it would be possible to specify
==PAGE:86==
76
Discussion
a formal semantics. It would be very similar to the semantics of the WHILE
language [NNH99, p. 56]. It is not trivial to conduct such a proof of correctness,
thus time did not permit it. We however strove to obtain a sound analysis, as
we want to make sure to ﬁnd backdoors if they exist.
The precision of our analysis rises with path sensitivity, as we do not suﬀer from
inconclusive results that a path insensitive analysis might do. We will however
still get false positives in some cases, this is due to the overapproximation of
converting a variable of any type other than boolean to boolean. Consider the
following code:
1 if(input=="secretpassword"){
2
passwordValid = 1
3 }else{
4
passwordValid = 0
5 }
6 if(passwordValid == 1){
7
authenticated = true
8 }
Our analysis would evaluate passwordValid to {true,false} in both branches,
which would cause it to output that the code has a backdoor, even though
authenticated = true will clearly never get executed unless the password
check succeeded. This imprecision can be solved by expanding the set of ab-
stract values to also keep track of numbers. This would require us to expand
the abstract evaluation function to be able to handle operators such as >, <, +
, - etc. We would also need to change how we construct composite assignments
in order to avoid over-approximation to booleans already on CFG construction.
Thus it would require some changes, but the main issue with adding numbers
to the set of abstract values would be the many possible execution paths. The
number type in JavaScript has a ﬁnite number of values, but there are so many
that the execution time will be appalling. One could perhaps adopt a kind of
interval analysis to try and limit the number of paths created, but it is likely
that this still correspond to very many paths.
Moreover, adding support for numbers will allow us to gain precision on exam-
ples such as the one described above, but the analysis will still loose precision
when considering strings. If strings were considered, we would no longer have
a ﬁnite number of abstract values, thus we would not be able to guarantee
termination of our analysis. Jensen et al. [JMT09] creates a type analysis of
JavaScript, which has a complicated lattice that can hold all possible data types
in JavaScript, but their analysis is not path sensitive.
==PAGE:87==
6.1 Evaluation

Thus expanding our set of abstract values to contain other types than booleans
would introduce precision, but also complexity and would require additional op-
timization techniques that are not needed on booleans.
6.1.2
Complexity
We did not apply any measures to try to optimize the speed nor the memory
complexity of our analysis to become more practically realizable. This section
will contain some ideas on how to optimize this.
6.1.2.1
Selective static analysis
Authentication logic in a program are usually not that many lines of code, thus if
we could limit our tool to only analyze this, we could create a very exact analysis
with a very high complexity, while still being practically realizable. We already
input the password check succeeded event and the authentication granted event,
thus the authentication logic are roughly going to be between those two events,
so it easy is to ﬁgure out which part of the program to analyze. One could
perhaps adopt a lazy path sensitive approach, such that the analysis starts at
the password check succeeded event. Whenever it encounters a variable which
it has not seen before, it needs to backtrace through the CFG in order to ﬁnd
the deﬁnition, just like creating a use deﬁnition chain for that variable[NNH99,
s. 2.1.5]. In this way we could limit the analysis to only analyse the relevant
code for the authentication logic.
6.1.2.2
Copy on write
The examples presented in Section 4.3.4 and 4.4.9 illustrates that the abstract
states are very similar across nodes. Thus one might adopt a copy on write ap-
proach to decrease the memory complexity. The way this would work would be
to make all nodes with the same abstract state reference one abstract state in-
stead of creating a copy for each entry in the analysis information table. When-
ever a ﬂow function would result in changes to the abstract state, we would
create a copy of the abstract state to make the changes on, so they would not
apply to the other entries referencing this abstract state.
==PAGE:88==
78
Discussion
In this way we could greatly decrease the memory usage, while not impacting
the time complexity.
6.1.2.3
Reanalyzing Functions
When we determine whether a function should be reanalyzed, that is whether
it is called in a new calling context, we use all the global variables to decide
this. This means that an update to a global variable might cause a function
to be reanalyzed. If this global variable is not even read in the function, then
the function is reanalyzed for nothing.
This results in a big overhead when
analyzing code where there are many global variables and the functions are in
general pure. Jensen et al. [JMT10] observed the same problem, when they
created their static analysis tool TAJS. They solved it by adopting what they
refer to as lazy propagation. It works by initializing all non local variables to ⊥in
the abstract state at function entry, and then only propagating the values of the
variables into the function, if they were needed. Then when checking whether
a function should be reanalyzed, they just need to compare the variables not
equal to ⊥. Our interprocedural analysis could greatly beneﬁt from adopting
the same approach.
6.2
Applicability
When considering the practical applicability of the tool, it is limited, even when
the time/memory complexity is not considered. This is because we only support
a small subset of the JavaScript speciﬁcation. Especially since most JavaScript
libraries use objects and asynchronous programming, as illustrated in Listing
4.1.
Thus if the tool should have practical appliance, it should be able to
handle these constructs. In general, the analysis needed to soundly handle all
JavaScript constructs, otherwise the tool could be fooled by a backdoor using
unsupported constructs.
Another issue that makes the tool less applicable in the real world is that back-
door code will typically be obfuscated. Our tool is not able to handle most cases
of obfuscation. This might partly be solved by using it in combination with a
deobfuscation tool, but in general the performance of those are not great, so it
might not be good enough for our need.
A limit to the applicability of the tool also comes from the fact that the pro-
gram output is not that helpful and the fact that the input has to be supplied
manually. The next two sections will discuss these two issues.
Thus the practical applicability of the tool is limited, we however ﬁnd that as
==PAGE:89==

was able to ﬁnd the rewritten real backdoors. Thus this can be used as a partial

of unwanted authentication ﬂows.
In Section 4 we mentioned, that we just chose the automata with three states as

6.2.1
Program output
Currently the output consists of the abstract state(s) where the automata transi-
tioned to an error state and the lines of code which corresponded to the traversed
nodes in the CFG. It can be hard for a security researcher to determine exactly
what went wrong and where, as variables in the program that are not responsi-
ble for the authentication ﬂaw are in the abstract state. This output could be
improved by comparing the erroneous abstract state with other abstract states
corresponding to the same CFG node. Any variables occurring with both true
and false in non-error states are probably not the cause of the backdoor. Thus
the program might suggest that variables where this is not the case are likely
to be the cause of the backdoor. There will be exceptions to this rule, so the
variables should just be suggested by the program.
The main problem with the program output was hidden by the fact that we used
a variable as the placeholder for a function returning true or false depending on
the time of the day. If we redeﬁned this to be just a placeholder for a function
returning the time of the day, such that we would have a backdoor trigger equal
to: timeOfDay == 3, to symbolize that at three of clock this condition becomes
true. This would mean that this condition would not appear in the abstract
state, thus requiring more by the user in terms of locating what the cause of the
backdoor is.
A graphical approach might also be considered, where we instead of giving a
textural representation of the path gives a graphical output. This might be on
the form of a graphical representation of the CFG, where the visited nodes are
highlighted. This can be extended by allowing the user to view the abstract
state(s) at each program point.
This approach could also be integrated into developer tools, where the path can
be highlighted in terms of the code rather than the control ﬂow graph. We
suspect this might be easier for programmers who do not concern themselves
with control ﬂow graphs.

6.2.2
Automating the input
It greatly limits the applicability of the tool, that we need to input the password
check event and the authentication granted event. Thus we will in this section
discuss how the manual input can be automated.
Much research has gone into the topic of feature location, which explores how
features can be mapped to the source code.
Dit et a.
[DRGP13] presents
a survey of methods in feature location. They are in general divided into dy-
namic, static and textural approaches. The dynamic feature location techniques
involves collecting information during program runtime, and are as such not rel-
evant for us.
Static feature location considers how diﬀerent aspects of the code is related.
Rather than using a control ﬂow graph, a dependence graph will be used, de-
scribing how functions and variables depend on one another. These methods
can greatly help developers discover how code is related. As we are not con-
cerned with the relations between program components, but the identiﬁcation
of speciﬁc components, static feature location is not useful for our problem.
Textural feature location is based on locating functionality through text. Tech-
niques such as information retrieval and natural language processing is used for
this. This is the method that seems most promising to us.
One of the big motivators behind feature location has been to develop tools that
can assist developers read, understand and ﬁnd features in existing code bases.
Because of this, some of the techniques relies on developer input to specify what
they want to ﬁnd, and output the most relevant results which might beneﬁt
them. In our case, we have to rely on less input, and the output has to be very
precise.
The current design of our analysis is a variation of textural feature location,
where both the query and requested source code is parsed before being located.
This ensures that diﬀerences in terms of spaces does not impact the result. It
is however very inﬂexible. Consider the following code:

6.3
Backdoor Detection
In this section we will present a more general discussion of how backdoors can be
detected through code analysis. We will present the ideas we had on how a tool
might detect the malicious behavior of the event-stream backdoor as presented
in Section 2.4. It is almost impossible to design a tool which could automatically
go through all the steps of deminifying, decrypting and deobfuscating in order
to detect this backdoor, thus we shall just present the ideas we had to detect
malicious behavior in the resulting source code.

6.3.1
String pattern analysis
We initially thought that the use of hexadecimal encoded or base64 encoded
hardcoded strings, which are later decoded and used as addresses looked like
suspicious behavior.
Our intuition was that this means the developer made
eﬀorts to hide either where data is send to or where it should receive data from.
This however presents some challenges:
First, how do you actually determine whether the strings are encoded? The
obfuscation detection techniques using machine learning, presented in Section
3, does most likely not perform well on such short strings. Even if the string
pattern analysis used by Choi et al. [CKCL09] could be trusted to work, averting
the metrics they use might be straightforward for an attacker. The attacker
could deﬁne their own encoding function, which uses a more even distribution
of characters, no special characters and uses whitespaces once in a while in
order for it not to be detected by their analysis. Additionally, if an attacker
implements their own encoding function, checking whether library functions for
encoding/decoding have been used would yield no result.
Second, what if the strings are not encoded, will the tool then not ﬂag the
code as malicious? If the tool should still ﬂag the code, then this means that
hardcoded IP addresses are considered malicious. We however argue that there
are many legitimate uses as well. It is not unlikely that the IP address of a
logging server, which logs errors occurring during execution, is hardcoded into
the code.
One might also consider hardcoded public keys as indicators of malicious code,
as these can be used to encrypt the sensitive data before exﬁltration.
This
ensures that it is only the attacker possessing the private key, that is able to
use the exﬁltrated data. Plus it will be more diﬃcult for anyone monitoring the
network at runtime to detect that data is being exﬁltrated. We concluded that
even though hardcoding the public key is extremely bad practise, since it means
the code needs to be redeployed in order to revoke the public/private key pair,
it is probably not exclusive to malicious code. One could however argue that
a tool to detect hardcoded cryptographic keys, might still be useful to ensure
secure and eﬃcient code.
6.3.2
Prototype redeﬁnition
We considered whether redeﬁning prototype functions far away from where the
prototype is declared was abnormal code. We thought that this was probably
analogous to overriding super class functions in a subclass in a regular object
oriented language like Java, where this is standard code. But we anyhow de-
cided to investigate it by manually inspecting the source code of the currently
==PAGE:95==

most popular ﬁve libraries on npm; lodash, request, chalk, react and express. We
found that lodash redeﬁned prototype functions inside the same ﬁle and that
express contained a dependency that redeﬁned the prototype function of a built
in object. Thus redeﬁnition of prototype functions, whether occurring in the
same ﬁle as the deﬁnition or not, is not abnormal JavaScript code. This how-
ever does not mean that detecting redeﬁnitions to prototype functions might
not be valuable, if it is used in combination with other properties, that might
be indicative of malicious behavior. The output from such a tool will in any
case be a suspicion, which has to be veriﬁed manually.
6.3.3
Silent errors
Another feature that might be indicative of a malicious behavior in the event-
stream backdoor is the amount of silent try/catch statements. By silent we mean
that the catch block contains no statements. In the full code there are ﬁve silent
try/catch statements. To a human reader this may seem like the programmer
was very eager to have as much as the execution of the code to fail locally, thus
not interrupting the entire execution. This is not good practice in production
code, because it makes it impossible to identify whether an error occurs, one
should at least log it. On the other hand, since the programmers maintain-
ing the libraries do in most cases not get paid to do so and maybe do not even
personally use it, they might not concern themselves with proper error-handling.
We could have made a tool which combined the above suspicious traits in order
to give a score indicating how "malicious" a piece of code is.
It is however
doubtful whether this tool would be precise enough to provide any value. It
is probably more useful to use a form of taint analysis to detect whether data
might be exﬁltrated. In the case of the event-stream backdoor, one would mark
the bitcoin wallet as tainted, taint any variable that touches this and ﬁnally
output an error if the bitcoin wallet is supplied as argument to the request
function.
This could even be done statically, though this would mean that
the analysis had all the same challenges concerning JavaScript as described in
Section 2.3. Otherwise dynamic techniques might be more suited to discover
exﬁltration backdoors. Either in the form of dynamic taint analysis or network
traﬃc monitoring.

6.4
Ensuring the security of public repositories
As one of the motivators for this project has been the emerging backdoors
published through npm, we want to brieﬂy discuss how this could be further
prevented. This thesis and the previous section has only considered source code
analysis as a way to detect backdoors. Another way to prevent malicious code
is to focus on the humans who are introducing malicious code. Zimmermann
et al. [ZSTP19] proposed that a selected group of developers maintaining the
more popular repositories should have their real life identity tied up with their
account, and that they should receive security training in order to prevent their
account from being misused. The obvious problem with this approach is that
hidden functionality in any dependency is a problem, so it is not enough to
only consider the popular repositories. If the idea is expanded to include all
repositories such that all contributors need to be veriﬁed, then the approach
becomes unrealistic, as it would require a huge amount of developers to be
veriﬁed. Some developers might view it as a violation of their privacy, if they
have to tie their identity to their work, which challenges this idea further.
Even when not considering this ﬂaw, it is also a culture issue, it would require
a substantial change to the workings of npm to implement this. As one of the
beneﬁts of npm currently is the usability, we do not think that npm community
would embrace this idea. If npm becomes too cumbersome, then the community
will ﬁnd another solution. We will now propose some improvements to npm that
could be implemented without impacting developer habits:
• npm already links to the github page of a library, so they should conﬁrm
that the contents of the github project reﬂects the package.
• npm should perform their own miniﬁcation, then it would not be possible
for developers to upload source code, which is diﬀerent from the mini-
ﬁed code, and thus making it easier to hide functionality in the miniﬁed
code. In this way developers would not have to read the miniﬁed code and
the source code in order to be sure the library does not contain hidden
functionality.
• npm should not allow contributors to use weak credentials and should do
a better job of informing contributors of the dangers of including their
npm account credentials in the source code.
• npm should enforce better rules against typosquatting, as the ones cur-
rently in place has some weaknesses. They could probably also issue a
warning if an application is using a library with a name very close to a
library, which is more popular.
==PAGE:97==
6.4 Ensuring the security of public repositories
87
These are suggestions, that are simple to implement, but would make it more
diﬃcult to hide functionality in libraries. But ultimately it is up to the devel-
opers to use libraries responsibly.

Chapter 7
Conclusion
The focus of this thesis has been on investigating methods to automate the pro-
cess of manually inspecting source code in order to detect hidden functionality.
We have limited the scope to a subset of JavaScript, and chosen to focus on
hidden functionality in the form of authentication bypass backdoors.
In order to design an analysis that can automatically detect backdoor code, we
have investigated which properties are indicators of this. This formed the basis
of the design and implementation of a tool based on dataﬂow analysis, which is
able to detect whether the authentication can be granted prior to a successful
password check in an authentication ﬂow. A simple automata was used to rep-
resent the diﬀerent states in an authentication ﬂow. The dataﬂow analysis was
made path sensitive in order to increase precision.
The resulting proof of concept tool is quite promising. We are able to detect
real-world backdoors rewritten into our supported subset of JavaScript. More-
over the subset we can analyze does not use constructs speciﬁc to JavaScript,
thus it easily extends to other programming languages.
The practical applicability of our tool is limited by the exponential running time
of the analysis; the fact that code corresponding to the password check and au-
thentication granted event must be manually input; and that we only support
a subset of JavaScript.
Thus further work is required if the tool should provide value to a package
manager such as npm. Most importantly, this includes optimising the time and
memory complexity and implementing a more automated input speciﬁcation.
