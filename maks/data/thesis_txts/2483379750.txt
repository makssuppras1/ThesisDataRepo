Abstract
BADGraph is a tool that is utilised to model attack scenarios through
attack-defense trees, providing a detailed and quantitative analysis of the
attacks, such as the execution time and the rate of success. It uses a highly
customisable language which enables the use of various parameters such as
the cost, number and type of attacks as well as the type and eﬀectiveness of
defences. In this report, I will ﬁrst discuss the modelling and analysis of a
real-life use case, based on previous research of major attacks, revealing the
attack’s eﬀectiveness and impact. The report will then present a scalability
analysis of the tool, where the performance and maximum model size of the
tool were tested. The use of the tool in modelling real use cases combined
with the performance analysis aims to provide an overview of the real-world
usability of attack tree modelling software.

1
Introduction
Security breaches and cyber-attacks targeting the infrastructure of both minor and major
companies pose a real threat to our way of life. In the past decade, obtaining the means
to execute an attack and conducting it have become increasingly simpler, as new tools
and methods have been discovered and the price has reached a very aﬀordable level,
where any person can utilise tools such as Booter portals [7] [12] [9] to conduct attacks,
disabling services which can aﬀect millions of users [4]. An increase in attacks has led
to a demand for eﬃcient modelling tools, required in order to analyse both previous and
possible future attacks, the impact they have on the targeted information or services and
the eﬀectiveness of the current defence system. By modelling various use cases, the overall
security of a service can be analysed and improved. BADGraph is a tool which utilises
attack trees [1] [2] [8] to model various attack types and scenarios, combining an easy to
learn and highly customisable language to create various parameters for both attack and
defence nodes, such as cost, type and eﬀectiveness, creating an overview of the attack
ﬂow through a graph [2]. In Figure 1, seen below, the BADGraph user interface can be
observed. This is explained in detail during subsection 3.1.
Figure 1: BADGraph User Interface
The models can then be analysed to reveal the impact of the attacks on the protected
data or service, the vulnerabilities of the system and possible improvements in defence
without a signiﬁcant increase in cost. In this project the tool has been used to model
real attack scenarios, based on previous research of signiﬁcant past attacks and security
breaches, such as the Xbox Live and PSN Christmas 2014 DDOS Attacks [4], which had
a massive impact on a large number of users[7]. A scalability analysis was performed by
using several parametric synthetic models, where several parameters were modiﬁed and a
high number of analysis runs were performed, displaying the performance of the tool and
the maximum supported load, as well as the applicability of the approach. This aims to
create a study into the real world use of the attack tree modelling tool.

to the Use Case analysis, thus postponing the scalability testing.

Getting familiar with BADGraph
The BADGraph tool was ﬁrst shown to me during an introductory course, where the goal
was to gain a good understanding of the syntax and the methodologies used to create and
analyse models. We created several simple models which we then improved in order to
better exploit the syntax of BADGraph, we discussed the models in detail and analysed
them in order to see the various options provided by the tool.
The tool provides an easy to use interface, based on the Eclipse IDE environment, with a
simple syntax that is used to specify the parameters for various types of attack scenarios,
such as Cost, types of defences for the target, eﬀectiveness of the attack and the chance of
detection, as well as the logic and transitions that the attacker needs to follow, contained
in the attacker behaviour section [2] [14]. BADGraph can be used to model real life
scenarios, providing an accurate way of analysing the attacks by observing key data such
as execution time, chance of success and defence eﬀectiveness, potentially leading to an
improvement in security for the company that chooses to use it to analyse potential
weaknesses to various attacks.

Listing 1: Attack and Defense nodes
In order to create the ﬁrst model, the user has to right click the project folder on the left
side and select the option to create a new BADGraph ﬁle. Here, the user can choose to
use a Simple Model as the initial template of the model. This model is a general scenario
where an attacker tries to hack a login system. In the ﬁrst part of the model, the attack
nodes are speciﬁed. These nodes represent some sub-attacks to be completed. These
nodes are AccountHacked, denoting that the attacker has managed to hack the account,
PhishingAttack, denoting that the attacker has successfully carried out a phishing attack
and guessed the user’s credentials and BruteForceAttack, denoting that the attacker has
successfully carried out a brute force attack in order to guess the user’s credentials. These
are the diﬀerent sub-attacks that the attacker will have to complete in order to complete
the attack. Next, the defence nodes have to be speciﬁed. The defence node in the Simple
Model is named ChangeCredentials, a security measure that attempts to change the user’s
credentials if it detects that they might have been altered or obtained by a 3rd party.
This node tries to prevent attack nodes from completing their transition and has a pre-
determined chance of success. In addition, countermeasure nodes are also implemented
as a line of defence against the hacking attempt.
9
==PAGE:11==
The countermeasure node in this model is named BlockAttacker, a node that has a chance
to block the brute force attacks launched by the BruteForceAttack node. The diﬀerence
between defence nodes and countermeasure nodes is that while the attack nodes have no
choice but to go through the defence nodes, the countermeasure nodes are reactive and
have to be activated by an attack detected event. Thus, the combination of defence and
countermeasure nodes provides a realistic modelling of a potential defence system, as no
system has complete security and these display the protection of the system in the form
of a chance to prevent the attack from happening.

Modelling and analysis of a real-life threat scenario
Can I use BADGraph to model real world scenarios? In order to answer this important
question, I searched through various online sources, articles and well-known scientiﬁc
conferences in order to ﬁnd information about past attacks which aﬀected millions of
users and created huge losses for numerous small and large businesses. Here I present
some of the identiﬁed models.
4.1
Modelling DDOS attacks with BADGraph
4.1.1
Identifying a use case
In the past decade, Distributed Denial of Service attacks, also known as DDOS attacks,
have become a serious threat to our society [7] [13][12][9][3]. Attacks such as the 2014
Christmas Attacks which targeted and successfully took down the Microsoft Xbox Live
and Sony PlayStation Network servers for close to 2 days [4] have proven that no system is
safe in the face of large scale attacks, with even the most secure systems being vulnerable
to the ﬂood of requests. In this report, the focus was on modelling real DDOS attacks
by using attack tree modelling of BADGraph. The analysis was based on a previously
released paper, titled ”Stress Testing the Booters: Understanding and Undermining the
Business of DDoS Services” by Mohammad Karami, Youngsam Park and Damon McCoy,
which was presented during the 2016 Edition of the WWW Conference [7].
4.1.2
Model details
The ﬁrst step to encoding the attacks inside BADGraph was to create an attack tree
diagram. The ﬁrst version of the diagram was designed to illustrate two types of attacks
presented in the paper, DNS and NTP attacks [7]. Domain Name Server (DNS) Am-
pliﬁcation Attacks are easily executed attacks that have a huge impact on the targeted
domains. The attacker sends multiple DNS Queries to the targeted WebServer’s DNS
Server, requesting all the available IP records for the targeted page. The attacker utilises
a spoofed IP address, an IP address that has a modiﬁed source in order to hide the iden-
tity of the sender, identical to the one used by the target. This results in a very large
number of DNS Queries, in some cases over 206 times higher than the initial attack being
sent to the targeted WebServer [7]. The other type of attack that was implemented, a
Network Time Protocol Reﬂection Attack or NTP, utilises the weakness posed by the
UDP transfer protocol, which allows for data to be received by a WebServer without the
completion of a handshake protocol, combined with the ampliﬁcation potential of the
MONLIST command, which requests the last 600 timestamp records from the targeted
server [5]. The attacker uses a spoofed IP address to redirect the massive amount of time
requests towards the targeted WebServer through a network of bots, regular computers
that were previously infected and are in an idle state until the requests are made. In order
to correctly display the defence method that is described within the paper [6], Cloudﬂare,
a defence node was implemented for the DNS attack branch. Cloudﬂare is a cyber se-
curity service that private users and companies can purchase on a monthly subscription
basis, which provides security against DDOS attacks up to a certain extent.

The model in Figure 6 above is based on a real Distributed Denial of Service attack
scenario of minor size, which utilises booter services [9]. The attacker has to initially
identify a booter service, a page that can launch DDOS attacks against one or multiple
targets, temporarily disabling their service, which can lead to massive ﬁnancial losses.
The attacker purchases a subscription, inserts the information of the target and launches
the attack. In this model, two types of DDOS scenarios were analysed, more speciﬁcally
DNS and NTP attacks, also known as Ampliﬁcation Reﬂection attacks [7] [5].
After the initial DNS Resolver Query has been completed, 3 requests are sent in the
form of DNS Server Requests to ﬂood the target server with IP Requests. In this DDOS
attack scenario, the targeted server is protected by a Cloudﬂare subscription, displayed
in the form of a defence node. Similarly, NTP Requests access 3 nodes from the BotNet,
a network of infected systems designed to amplify attacks by sending a ﬂood of time
requests, targeting the victim’s server. In most cases, the ﬁrst line of defence against
DDOS attacks is a IP blacklist checker, which has been previously proven to be ineﬃcient
against large scale DDOS attacks due to the IP spooﬁng capabilities of most of the attack
types. Thus, a blacklist countermeasure node has been added to illustrate the eﬀectiveness
in a possible scenario. If the forwarded IP addresses are not part of the list, then the
time requests are forwarded to the server, which sends them to all the connected users.
Both of these scenarios lead to a successful DDOS attack, where the target’s service is
temporarily disrupted.

Listing 12 contains the attackers section, where the main attacker is speciﬁed, an identity
that will be used to describe the behaviour and path of the attack. The main section of the
model is the attacker behaviour, describing the sequence of events that the attacker has
to follow and the conditions that need to be met in order to either pass or fail the attack
scenario. After the attack is initialised, the main attacker that was previously created
is speciﬁed, followed by all the states that will be part of the model. The ”transitions”
section contains the logic for the attacker path in the scenario, as well as the algorithm
for choosing the next step. For this model, I decided to use a ”Roll-of-the-dice” type of
algorithm, where the ”tryAction” can be executed in transitions with diﬀerent probability.
With the value 1, it choses the NTP DDOS attack and with the value 2 choses the DNS
DDOS attack.
Once the initial path has been chosen, the algorithm runs through each transition state,
choosing probabilistically the transition to be executed, using the same tryAction variable
to choose the next step, with increasingly lower chances for the main attacks. Once the
main attacks circumvent the defence or countermeasure, the DDOS attack is considered
successful and the analysis is completed. The states are structured into a start, ﬁnish
and sub-attack states, where for each path of the model, DNS or NTP respectively, we
have a state for each sub-attack starting from the beginning of the model.

4.1.4
Analysis results
The analysis graph displays a multitude of interesting facts. We ﬁnd out that the DNS
type attack is the one with the lowest chance of success, completing the FloodServer-
WithTimeRequests node in approximately 11 time units with a 64 percent chance. By
comparison, the BotNetTimeRequest attack node of the NTP type attack completed in
11 time units with a 82 percent chance. The DDOS attack was successfully completed
after approximately 50 time units, where the chance of success increased to 98 percent.
Figure 7: Distributed Denial of Service - Model Analysis
In addition, we can observe that the IdentifyBooterService had an approximately 40
percent chance of success and completed after the 10 time units mark. The DNS Request
and BotNet attack nodes had similar completion times, with the notable observation that
the DNS Server Request completed instantly, displaying an advantage for the DNS type
attack in the early stages. From this analysis, we can conclude that the NTP type of
attack had the higher chance of success due to the inferior Blacklist type of defence, much
inferior to a service such as Cloudﬂare. While faster in the initial stages, the DNS type
DDOS attack had an approximate chance of success of 60 percent, translating into a 20
percent diﬀerence between a system utilising Cloudﬂare versus the traditional Blacklist
defence, which was proven to be ineﬀective against attacks that can create a potentially
unlimited amount of spoofed IP addresses [7].
The analysis was completed after 8.48 seconds, completing an amount of 255 queries and
1520 simulations of the model, with a parallelism degree of 1. The delta list returned a
value of 0.05, proving that the degree of accuracy was high for this analysis. Thus, the
results of this parametric expression have displayed that the BADGraph simulation and
modelling tool for attack trees can be used in a real life scenario to identify the eﬀec-
tiveness of attacks and defences as well as identify the best method of protection against
large scale attacks such as Distributed Denial of Service attacks, with a simple interface
and syntax, providing a user-friendly environment for accurate analysis. The analysis
proves that dedicated protection services such as Cloudﬂare can dramatically improve
the protection of companies and potentially prevent attacks from being successful.

5
Scalability analysis of BADGraph
Can BADGraph’s performance be suitable for modelling real world scenarios? In order to
answer this important question, I performed a scalability analysis, testing the capabilities
of the tool, the maximum size of the models it supports, the performance depending on
the number of attack nodes and analyzed the results to provide an overview of the tool’s
real-word capabilities.
5.1
Designing the scalability model
The scalability model was created by using a simple design, which has the following
components: We have a group of attack nodes which can be given the desired size through
the execution of an external script, a main attack node which groups the attack requests
and disables three defence nodes, which serve as a line of defence for the ﬁnal state node,
where the attack is successful.
I decided to evaluate the performance and scalability of the tool by utilising an analysis
where the model would be executed for a pre-determined number of steps. During the
ﬁrst test runs, an error in the analysis results became clear. The implemented logic in
the attacker behaviour would verify if the attacks were successfully added and skip the
rest of the simulation steps if the value was ”True” (for example, if the root attack was
previously completed and the attack was successful, the simulation would stop). Thus,
I modiﬁed the model so that it doesn’t check if the currently processed node has been
previously successfully added. This way, the analysis is run for the given amount of steps
without any interruptions, providing a correct result for the scalability tests.

The arrow pointing left across the connections between the attack nodes and the main
attack in the above model represent the AND parameter, where all of the inputted attack
nodes need to be added in order to proceed to the main attack. Thus, increasing the
number of attack nodes can have a signiﬁcant eﬀect on the overall execution time of the
analysis. I decided to use the ordered AND (OAND) parameter, where the tool executed
all of the sub-attacks in order before proceeding to the main attack, as the OR parameter
proved to be ineﬀective in testing performance, as the tool picks one attack node to
execute and continues with the rest of the nodes in the model. By using AND, the model
is forced to execute all of the attack nodes that have been speciﬁed by the user until the
number of selected simulation steps is reached.
5.2

Java script that takes a previously created base model as input and adds a user-deﬁned
number of attack nodes to it. The data that needs to be added is the attack nodes deﬁni-
tion, the attributes in the form of a Cost, where I chose a default cost value of 0.5 as the
baseline, the detection rate of the attack nodes, where I set a value of 0.1 or 10 percent

the attack diagram is created, where the relationship between the main attack and the

a eval type query that runs the model for the inputted number of steps. For the sake of

the form of inputStream and outputStream to write to the ﬁle, favouring reliability over
performance (the performance of the script is good, as creating a model with hundreds
of attack nodes can be executed in under 10 seconds). The following part handles an
input box where the user can write the desired number of attack nodes to be added to
the model, along with the respective attacker behaviour and logic. This script could later
be added to the tool in the form of a menu option when creating a new BADGraph ﬁle,
where in addition to the Simple Model template the user could also select a Scalability
Model template, followed by an input box where the number of nodes could be speciﬁed.
5.3
Scalability analysis
The tests were run on a MacBook Pro 2017 system utilising macOS 10.15 Catalina, with
an i5 dual-core processor running at 2.3Ghz, an Intel Iris Plus 640 graphics processor with
a total of 1536 VRAM, 16GB of LPDDR3 RAM running at 2133Mhz and a 256GB SSD
storage unit. As the tool is designed to run on macOS and Linux, the test environment
was compatible with the requirements of the tool. The system hardware is not high-end,
so the assumption that some of the test results could be inﬂuenced by this factor was
made. In addition, within the BADGraph.ini conﬁguration ﬁle, a limit of 2048mb of
RAM was set as the maximum amount of memory that can be used by the tool. Thus,
the system RAM was considered appropriate for evaluating the tool under high load
scenarios, which could lead to instability due to memory limitations.
5.4
Analysis results
5.4.1
AND Parameter
The initial scalability test was based around the AND parameter, so that all of the added
attack nodes would have to be executed in order to allow for MainAttack attacks. The
number of attack nodes was set to 50 at the start of the test, with 10 diﬀerent simulations
being recorded, from 100.000 steps being evaluated up to 10.000.000 steps. The value of
10 million steps was an arbitrary value that was chosen due to the extensive time required
to run the tests, where each run would take several hours to complete. A time limit of
10.500 seconds was set for each test, where exceeding the value would be considered a
timeout. Each analysis was repeated for a number of 10 times, in order to provide the
most accurate results in the form of the average run-times. After completing the 10 test
runs, the number of attack nodes was increased by 50 and the tests were rerun.
The tests revealed various factors in regards to the scalability of the tool. While at a
lower amount of steps, such as 100.000, doubling the number of nodes would return a
100 percent increase in run-time execution (for example, increasing the number of attack
nodes from 50 to 100 resulted in run-times of 14.38 and 28.44 seconds respectively) , when
inserting a higher amount of steps, such as 5.000.000, the run-time results would become
signiﬁcantly slower (for example, going from 50 to 100 nodes increased the run-time from
605.94 to 1664.87 seconds), in some cases by as much as 175 percent.
27
==PAGE:29==
These results revealed that while a high amount of attack nodes would not negatively
impact the run-times in a scenario where the lowered amount of steps led to shorter
simulations, increasing the accuracy as well as the scope would signiﬁcantly impact the
run-times of the test. I decided to investigate this factor in the following tests, in order to
observe the eﬀectiveness of the tool in handling large scale scenarios. This increase in run-
times in connection with a higher number of simulation steps became less apparent when
the number of attack nodes reached 200, where comparing the results between 100 and 200
attack nodes at the 5.000.000 steps setting displayed a insigniﬁcant 50 percent increase
in run-time execution. By comparison, at the shorter simulation setting of 100.000 steps
for example, the run-time increase amounted to 167 percent. Thus, while a lower number
of nodes can observe a signiﬁcant increase in run-times with higher accuracy settings in
the analysis, a high number of nodes decreases this margin, proving that BADGraph can
be very eﬀective when modelling and analysing very large scenarios where hundreds or
even thousands of attack nodes could be implemented.
Increasing the number of attack nodes to 100 put the tool under stress, leading to the
ﬁrst observed crash at the 750.000 steps mark. Running the test at 150 nodes/10 mil
steps resulted in an extremely long run-time for one of the tests, where the execution
of the test took over 2 and a half hours to ﬁnish. The following 9 runs returned more
reasonable values, reducing the run-time to one hour. The ﬁnal value of the 150/10 mil
test was set to 3840.4 seconds. Upon increasing the number of attack nodes to 200, the
tool became extremely unstable. Running the test at 200 nodes/10 mil steps resulted in
3 crashes where the tool appeared to run out of memory. After several more attempts,
the tool managed to complete the tests, revealing a stability issue when the scope of the
model is larger than 200 nodes. In a scenario like this, the tool would need to process over
400 actions and transitions in attacker behaviour and logic for each run of the model.
The tests ran without issues with 250 attack nodes, displaying a 66 percent increase in
run-time execution at 10.000.000 steps, from 5378 to 8934 seconds. At this point, the
number of processed transitions exceeded 700. After increasing the number of nodes by
50 to the new baseline of 300 attack nodes, the tool started crashing every time a new
simulation was started. I decreased the number of nodes to 290, with the same results.
After several more tests, I concluded that the tool would crash at any value over 280
attack nodes. At this point, the model would have to go through 285 states and 844
transitions for each run.
I attempted the test with 280 attack nodes on a secondary system, an ASUS X550VX
computer with i5 6300HQ quad core processor clocked at 2.30Ghz, 8GB DDR4 RAM at
2133 Mhz, a GTX 950M graphics processor and a Samsung 860 EVO SSD, with similar
results. The secondary system began crashing when the value of attack nodes exceeded
275, a slightly lower limit than the macOS system, but on a currently unsupported oper-
ating system (Windows). BADGraph was run on Windows by using a virtual machine,
emulating Linux. Thus, while utilising the AND parameter, the current model size limit
of the tool is between 275 and 280 attack nodes.
Another factor to consider is that
utilising a more complex model would decrease the maximum model size that can be
analysed.

5.4.2
OR and K:N Parameters
Analyzing the model by replacing the AND parameter with OR within the connection
between the MainAttack and the user inserted attack nodes revealed that while in the
early stages of the analysis, where the number of nodes was set 100, the run-time execution
times where substantially lower, as for each run of the model the logic only selects one
attack node before continuing to the Main Attack, when increasing the model size the
results were almost identical, with the tool starting to crash at the same maximum
amount of nodes, 281. Similarly, K:N performed similar to OR at the lower model sizes
but became identical in results for the large size types. In conclusion, the AND parameter
is the desirable parameter for a scalability test, as the model has to be executed in its
full size and with the completion requirement for all the nodes before being successful.

Discussion
6.1
Conclusion
In conclusion, the combination of a real-world threat scenario analysis through attack tree
modelling and a scalability and performance analysis of the tool displays the real-world
usability of BADGraph and how it could be applied to research major past attacks which
caused major ﬁnancial losses and view the eﬀectiveness of various defensive tools in order
to improve security for users and businesses all over the world.
The BADGraph attack tree modelling tool is a simple, eﬀective way of designing models of
real, documented attack scenarios, analyse various components that lead to their success
or failure and scale the models to the capabilities oﬀered by the tool, only limited by the
user’s system memory. The scalability model could later be added as one of the selectable
templates that a user can choose from when creating a new model, where the user could
select the desired number of attack nodes, so that the performance of the model could be
veriﬁed before the full design is created.
In the future, additional models and attack types could be analysed through BADGraph
to reveal the attack’s eﬀectiveness and possibly create a database where users could see
the BADGraph model counterpart for each type of real-world attack. In addition, other
types of scalability tests could be performed, where the attacker behaviour is implemented
in alternate ways, combined with the veriﬁcation of the several available parameters. This
thesis project proves the real-world usability of the tool through an in-depth analysis of
past attack scenarios and a scalability analysis which provides detailed information about
the limitations and performance of the tool.

Final thoughts
To conclude this project, I would like to thank my supervising teachers, Andrea Vandin
and Alberto Lluch Lafuente, whose dedication and continuous help with the project tasks,
discussion of objectives and observation of my progress were invaluable in the deﬁnition,
writing and completion of this thesis project. Our regular meetings and discussions, the
analysis of the documents and papers that we found relevant, the attention to detail to
guarantee the accuracy of the performance and scalability analysis of the tool were key
in obtaining these results.
The courses I attended at the Technical University of Denmark provided the background
knowledge necessary for this project, through the understanding of distributed systems,
network protocols, security algorithms and encryption types, attack methods and system
vulnerabilities, designing secure systems and completing performance tests in order to
obtain the capabilities of the system and see how they scale with the scenario that is
being analysed.
This master thesis project provides the complete picture of the usability of BADGraph
when used to model real-life threat scenarios, analyse attacks to improve security, view the
eﬀectiveness of defence tools and scale models based on the available system resources, in
order to obtain the best performance and results. Attack tree modelling has the potential
of being a key factor in the improvement of security methods for users and businesses
worldwide, severely lowering the eﬀect hackers have on our society.
