Abstract
The IoT market has been and continues to grow worldwide, but surveys still show that
these devices often forgo security analysis. The main reason is the high cost associated
with cybersecurity. This thesis aims to continue previous work on automated threat model-
ing software that can provide companies with a cheap cybersecurity analysis of their new
devices. This software should not need any security knowledge to operate, and the model
result must be something a developer could use to implement changes in the device in
question. This is done by using information the developers are expected to know about
their device and presenting them with the threats related to the entities in that device and
how to mitigate them.
This project does not present a full program but rather implements most key features and
allows future software developers to continue work with clear documentation of the code
base, including UML diagrams.

Introduction
IT is still growing even when global markets are pressured by uncertainty both by the war
in Europe and important trade routes targeted in the Red Sea. The global IT services
market grew from revenue of 1.12 trillion USD in 2022 to 1.24 trillion USD in 2023, an
increase of about 10%, and is expected to grow to 1.36 trillion USD in 2024, about 9%[1].
Growth in the IT market is, as always, marred by malicious actors that exploit or misuse
systems for illicit gains. According to IBM, the average data breach cost has increased
to 4.45 million USD, a 15% increase over the last three years[2]. With an estimated 11.5
attacks per minute[3], there is plenty of reason to invest in attack mitigation. This is re-
flected in the 14% expected cybersecurity spending growth for 2024[4].
Unfortunately, the increased spending is not spread out universally, and most small de-
vices, systems, and apps are still not seeing the benefits. In a study on IoT-related ap-
plications, researchers found that almost 95% of mobile apps with more than one million
downloads had severe cryptographic violations[5]. They also found that out of a sample
of 500 libraries used, 481 (65 unique) were subject to 79 CVEs.
This paper aims to help developers of such apps and other smaller devices and systems
where proper security analysis is not considered important enough by lowering the cost
of such analysis. By making threat modeling available to developers with a requirement
for security experts, the hope is that these developers can easily analyze and mitigate the
worst problems in their systems.
1.1
Purpose
This project aims to provide an OS-independent tool for automatic threat modeling. This
tool is to be designed such that it does not require extensive experience or education in
cyber security. Thus, the resultant threat model must accept input that any developer of
a system or device should know. The model output should be a clear list of threats and
vulnerabilities prioritized by how significant the problem is. This is best accomplished by
giving the developer severity and likelihood information. The threats and vulnerabilities
should come with recommendations for mitigations and how attack patterns look. This is
intended to help developers make decisions on applicability. Based on this, the following
project requirement list has been created:
1. No required prior security knowledge
2. Model provides actionable results
• List has an initial priority ranking
• Provides mitigation suggestions
• Provides attack patterns
3. Model must use both threats and vulnerabilities for the results
4. The tool works on Linux, MacOS, and Windows
1.2
Scope
The threat modeling tool created by this project is not expected to be a full replacement
for threat models created by security experts. The tool will depend on a database for its

2
State of the art
Threat modeling is a well-understood concept; many good frameworks have been de-
fined, and they have helped cybersecurity experts worldwide develop new, more secure
devices.
A threat model is a way to describe the potential threats a device is facing. They aid de-
velopers in finding security problems that may not have been clear from the start. It then
presents this in a structured manner to enable the mitigation of the threats.
People who know risk assessments will see that threat modeling is closely related. How-
ever, there are some significant differences between the purpose and the use of the two. A
risk assessment is typically about an organization’s objectives and managing these risks.
Often, it is more holistic and grand strategy-related. The risk assessment can start after
the organization is established and continues to evolve for the duration of the organization.
Threat modeling is more concerned with the direct threats that a specific system or device
faces. The threat model should be completed or close to completion before the system or
device is completed for it to be fully exploited. This is because the model gives actionable
information about how the development of that specific system should be done. Some
changes could fundamentally change the structure of the device or system.
CISCO defines five steps involved in the threat modeling process[6]:
1. Identify assets. An asset could be account data, intellectual property, or simply the
reliable functioning of a system.
2. Diagram the system. Data Flow Diagram (DFD) provide a high-level, asset-centric
view of systems and the data flows of attacks. An attack tree, or graphic represen-
tation of an attack path, illustrates the possible origins and paths of attacks.
3. Analyze threats. Use threat modeling methods to further analyze specific threat
types, identify potential threats, map data flows, and quantify risk.
4. Perform risk management and prioritization. Many threat modeling tools produce
threat scores and data for calculating risk. Stakeholder input is essential to this
step.
5. Identify fixes. Once you identify the areas, assets, or threats that matter most to
the organization, the next steps may be apparent. Changing firewall, encryption, or
multi-factor authentication settings are examples of steps to address a threat.
This specific threat model must fulfill an additional capability; that is, it must be possible for
a developer who is not a security expert to gain valuable information and guidance without
having the typically required cyber security foundation during step 3, analyze threats.
2.1
Existing threat models
When creating a new threat model framework, reviewing why the current models are in-
sufficient for the outlined purposes is important.
Many models work based on the initial values defined in the mid-1980s when the Confi-
dentiality Integrity Availability (CIA) triad was defined. This sets up the three critical com-
ponents a security expert has to consider. The Confidentiality of information, the Integrity
of information, and the Availability of information. From this, models that expanded on
these parts were designed to help make thorough threat models.

While many frameworks exist, this paper will only cover the most common one and one
other that has interesting aspects.
2.1.1
STRIDE
The most popular model is, by far, the STRIDE model created by Microsoft. This model
splits the properties defined in the CIA triad into more direct threat vectors: Spoofing, the
act of pretending to be someone or something you are not; Tampering, the act of changing
information; Repudiation, the act of claiming inaction when this is not the case; Informa-
tion disclosure, the act of giving out information without permission; Denial of service, the
act of preventing someone from using a service often done by exhausting the resources
of the service; Elevation of privilege, the act of getting more authorization than permitted.
When applying STRIDE, the simplest method is to go over the system methodically by
checking the connections and relations the system has. The related threats are noted and
categorized according to each STRIDE letter. Each connection and relation is expected
to have multiple threats, and a threat can belong to more than one letter of STRIDE at a
time.
The splitting of the threat types helps a user think critically about all the threats that could
impact the system, and it helps with evaluating the threats later according to specific or-
ganizational priorities.
This model relies directly on the implementors’ understanding of potential threats and their
knowledge about how a connection could be vulnerable to unexpected threats. The expe-
rience of the people working on it will directly impact the derived value from the process.
This is the exact problem this model attempts to remove.
2.1.2
PASTA
This model differs from the STRIDE model mainly in one significant aspect. Process for
Attack Simulation and Threat Analysis (PASTA) employs the business context directly in
the threat modeling. This can lead to a more focused threat model process that better
accounts for the critical threats to the business.
PASTA uses a seven-step process where each step feeds into the next step. The first
step is to identify the objectives for the threat model; this includes the business objectives
related to the system and the threat model itself. The second step defines the technical
scope of the threat model; this means defining what aspect of the system the model does
not cover. The third step ’disassembles’ the system into Data Flow Diagrams, sequences
diagrams, and other system-specific information; this is done to get a full understanding
of how the system works. In the third step, the model identifies the threats based on the
’disassembled’ system models; here, information like previous security incidents and logs
will give a better output. From step five onwards, the PASTA model evaluates informa-
tion not involved in the base STRIDE model; the PASTA model now does a vulnerability
scan; users should use publicly available libraries such as CWE[7], CVE[8], CVSS[9],
and CWSS[10]. For the sixth step, the model does an attack analysis, checking how the
threats and vulnerabilities that have been identified could be exploited, which will be im-
portant information when attempting to prevent them. The last step evaluates the risks
and impacts of the identified threats and vulnerabilities.
This model contains several exciting elements that make threat modeling more useful.
By implicitly including risk analysis of the found issues, the users are better equipped to
prioritize the critical issues and mitigate the impact of the threats and vulnerabilities. How-
ever, as seen in STRIDE, the model relies on the security qualifications of the people
implementing it.

2.2
Existing threat modeling software
This knowledge requirement is common with threat models and is not only present in the
two models discussed here. Tools have been created to alleviate the issue somewhat,
but these tools mainly aid with making the modeling more structured and presentable.
This is an important aspect of the threat models but does not solve the user knowledge
requirements.
Microsoft has a threat model tool designed for the STRIDE model called Microsoft Threat
Modeling Tool (MTMT) [11]. To use the model, users design a DFD of the system, and
the tool automatically provides a list of threats based on the data flows.
This tool gives a good automated method of creating a base threat model, but it does not
help the users prioritize these threats; all threats are default high priority, and it is up to
the users to define this priority list themselves. The tool also does not make a vulnerability
scan, as included in PASTA. This is a big downside, as common vulnerabilities repeatedly
cause attacks. This is caused by the MTMT glossing over what version of WiFi or Blue-
Tooth a device uses. By doing that, it does not discover the vulnerabilities associated with
these methods of communication.
Another problem with the MTMT product is its Windows 10 or later OS requirement. This
makes it more complicated to use if a user prefers a different OS.
OWASP has created a tool called ”Threat Dragon”[12]. This tool works in much the same
way that MTMT did. It uses a DFD as the basis for the threat model. It differs because
it does not provide an automated threat generation feature but assists the threat modeler
with a structured model and reporting. Threat Dragon does support other threat modeling
frameworks like PASTA.

3
Decisions
Implementing an automated or partially automated program for threat modeling carries
several problems. Mads Astrup Andresen has already covered some of these in ”Threat
modeling made simple” [13]. Andresen laid the groundwork for the framework of the threat
model and thus handled problems directly related to the model but not problems related
to automation. The decisions related directly to automation build upon the challenges of
designing a threat model suited to this type of automation.
The main goal of creating a threat modeling tool that someone without security education
can use means that several challenges are nontrivial. The model must be simple enough
that the result is understandable and suitable for development. This requirement is par-
tially covered by Andresen[13], but some new challenges and decisions are introduced
when implementing the model in a computer program.
The threat database is the base that defines what information is available to the program
and, therefore, the information it can provide to the user. This problem is discussed in
Andresen, this author will not cover further database development as having the tool do
that would require some advanced AI.
Once that is decided, the model outcome must be defined. What is valuable information,
and what is redundant information that does not help a developer with understanding and
countering only the relevant threats and vulnerabilities.
These are the main challenges related directly to the model that the implementation has
to consider.
Concerning the actual implementation, several challenges present themselves.
The first problem is what framework the program should be implemented in or whether
a new framework should be created. This last option is easily discarded as there is no
need to ’reinvent the wheel’; several suitable frameworks are available to create a fully
functional UI.
The framework naturally defines parts of the design, but several choices must still be made.
How will the user navigate the program? What is a meaningful path from creating a new
model to the final product? What needs to be persistent, and what does not? These ques-
tions help define how the program should be designed, and a clear and concise design
helps create readable and maintainable software.
3.1
Modeling
The model proposed in Threat modeling made simple [13] is very simple and easy to grasp.
Based on the box model, the threat model presents an abstraction that developers should
readily understand regardless of formal security education. They should be able to define
the inputs, outputs, controls, and the state of the device or entity they are developing.
These inputs, etc., may themselves be called entities and will be called that from now on.
While initially confusing, consider that a WiFi modem could be an input to a final product,
but it is also an entity.

Based on the Box Model, Andresen defines the Layer Model that structures these lists of
entities into what type of entity it is. This was done as the capacity an entity is running
in is relevant to the threats it is exposed to. This model can model any given device in
a simplified manner so that a software program can work with the data in a structured
manner.
In the original model proposed by Andresen, there is a strict requirement for version
numbers when they are relevant. This is due to the vast difference in relevant threats
to an entity. An example could be the difference between using TLS 1.0 and TLS 1.3. In
TLS 1.0 ”[...] makes TLS_DHE_DSS_WITH_3DES_EDE_CBC_SHA mandatory to imple-
ment.”[14], which is now considered a weak protocol[15]. Additionally, TLS 1.0 and 1.1 do
not support Authenticated Encryption with Associated Data, the currently recommended
encryption scheme. Indeed, the Internet Engineering Task Force states: ”This document
updates the following RFCs that normatively reference TLS 1.0, TLS 1.1, or DTLS 1.0.
The update is to obsolete usage of these older versions. Fallback to these versions is
prohibited through this update.”[14].
This poses an issue in two locations now. Namely, a decision must be made regarding
how to decide the relevance of explaining it to a user or remove the requirement and de-
cide how to represent the issue from the lack of information.
The author considers the relevance problem quite significant as the purpose of the model
should be accessible to users without this understanding. The ability to judge when such
things are relevant runs counter to this purpose, and thus, the problem now becomes
whether to require version numbers for all entities or to waive the requirement for all enti-
ties.
Requiring full version numbers for all inputs makes using the threat model more cumber-
some and prohibits fast exploration of potential development paths. Instead, the threat
model should be easy and quick to use and encourage changing things without fully defin-
ing the used entities. The downside is that the model’s final output will contain more
threats than is strictly necessary, as it will have to include threats to potential versions
of some entities. The author prefers this solution as it gives users better information to
decide on the version they should implement. In a sense, it can guide the developer to a
more secure device even before security measures are actively implemented.
This outcome aligns with the goal of the final product. Therefore, the requirement of ver-
sion numbers is removed but still recommended for simplifying the final output of the threat
model. The entities that would benefit from a more precise version number are flagged
in the result, and a simple text should be able to explain why it is beneficial to the threat
model that a version number is included.

3.2
Threat database
To enable a program to do threat modeling, we need a good database with information
about the relation between the entities and what threats they are exposed to. Andresen
created an initial database with these relations for some entities; this paper does not
expand on that database. However, this paper considers that the database created in
Andresen does not contain all the information that would help a developer remove or
mitigate potential security problems.
As stated in the goals for the threat model presented in section 1.1, the model should be
able to give an idea about what can be done or how such attacks could be carried out. For
this reason, the CAPEC list is used extensively. This list is not a list of threats but rather
a list of attack patterns. While a CAPEC entry can be close to being worded like a threat,
it is not always. Due to this, there needs to be a balance between when one database is
used and when another is. Considering that the database from Andresen is limited, the
CAPEC list will be heavily used for now.
3.3
Threat model outcomes
Finally, a decision must be made about how the threat model should be presented to the
user. What is the most important information, and how does the threat model software
help the user to implement mitigations?
The threat model should present a simple and easy-to-read structure. This is achieved
with the final output proposed by Andresen, which contains a list of threats sorted by the
entities they affect and a simple way to check off threats as they are mitigated. This allows
a developer to delegate the threats to the development teams in charge of the different
entities, with an easy way to keep track of progress made.
This author differs slightly from the proposed solution by using the CAPEC list as the
primary database for this part. This is done to include recommendations on how these
attacks are mitigated, which is an excellent way to ensure the vulnerabilities are mitigated
correctly. Focusing heavily on guiding and assisting the users in how they should use the
threat model directly leads to beneficial changes.
3.4
Programming framework
There are many different GUI frameworks out there. We can choose the preferred pro-
gramming languages, remove the web development tools, and exclude all game develop-
ment frameworks to narrow the list. The main languages considered for the project were
C, C++, C#, and Java. These languages were chosen as they are familiar to the author
and most other students who would have to work with the resulting code base.
Java was removed from the list due to how simple SWT and Swing (the two main GUI
frameworks in Java) look and feel. The author wanted to provide the option of creating a
more in-depth UI that does not look like it was made with frameworks over 20 years old.
This leaves two main frameworks for consideration: Avalonia and GTK.
3.4.1
Avalonia
Avalonia is a .NET cross-platform framework that supports Windows, Linux, and MacOS.
The framework was inspired by Windows Presentation Foundation (WPF), a Windows-
specific framework designed to provide a consistent programming model, and both Aval-
onia and WPF use Extensible Application Markup Language (XAML) with Avalonia using
an extended version called Avalonia Extensible Application Markup Language (AXAML)
to define the UI elements. These AXAML files then connect data parts of the UI to the

underlying C# code through data binding. This allows strict control over what data a user
can change on the backend from the UI, as these relations are strictly defined.
The framework has good options for making reusable templates for UI elements or ’con-
trols’ as they are called in Avalonia. They are used multiple times over many parts of the
design and provide several easy-to-use base controls like data grids, panels, and a menu.
When used in Visual Studios, the only IDE it officially supports, it is possible to see how

3.4.2
GTK
GTK is a framework designed for systems based on X11 and Wayland. It also works on
other platforms like Windows and MacOS. This framework also uses XML as the basis
for the UI elements. GTK has many UI designers created; some implement a ”you get
what you see” design style with dragging and dropping elements. This is similar to what
Avalonia provides but is more involved as it allows direct implementation in this live view.
GTK is often criticized for its lack of backward compatibility when new releases are made
public. This problem should be fixed by the latest release of GTK 4.0, which implements
many features that are now expected of these frameworks.
Both examined frameworks have the essential features this project requires, but Avalonia
has an excellent tutorial that helps new people get started with projects quickly. The
extensive library of ready-made controls with good explanations on how they are used
and the limitations of what they are not good at helps facilitate accessible developments
without the source code getting too cluttered from inexperience.
3.5
Design
In deciding on the overall design philosophy of the program, one point was essential:
the clear and concise design layout that makes it easy to change parts of the program
without fully understanding the entire program. This would allow developers to improve
or change some aspects more easily in the future. This is important as no software is ever
fully ’complete,’ and this program especially was expected to require further development
after this master’s program was concluded.
For this reason, the author seeks to separate the actual threat model from the UI elements
in a different program. This allows updates to the threat model system without going
through the more complex UI elements.

Current results
Several diagrams were designed to guide the progress of creating the program. This

4.1
UML Diagrams
UML covers many different diagrams, and they all have their uses for software develop-
ment. In this project, the author has chosen to use only three types of diagrams as these
are sufficient for this project. While further diagrams could provide more in-depth details
of how the final product should look, it is not necessary to decide this fully yet as different
master’s students will have different opinions on the final structure. This author thinks
that leaving room to shift the design allows future authors to take the parts that work and
replace the ones that do not. For this reason, only use case diagrams, activity diagrams,
and class diagrams were created for the project.
4.1.1
Behavior diagrams
These diagrams describe how the program is expected to be perceived by a user. It does
not go into how something is implemented but describes the flow of decisions and behav-
ior based on the actions taken by an actor. Using behavior diagrams, at least in part, to
define when and where you create a new method or class helps ensure these decisions
are made from the logical flow of information, which creates readable and maintainable
code.

What should the program do, and what is the rough internal structure in charge of these

delete the files directly if needed.
The last activity diagram to consider is updating the database. This action is important as
the threat landscape is constantly shifting and changing; entities are updated and have
threats mitigated, and old entities have new threats discovered. Therefore, the ability to
update the database the threat model uses to create its models is critical. However, this
problem is left mostly for the future authors to work out. For now, a simplified version of
the created data with entities to threats (”Tech-Threat-map.csv”) has been manually cre-
ated and put in the Database folder for the project. Additionally, some of the CAPEC lists
have been manually downloaded and extracted for use when looking for severity, likeli-
hood, and possible mitigations.
Ideally, there should be some automated method to update these databases so the cre-
ated threat models remain relevant. The most difficult one will be the entity to threat map,
as this is not publicly available and will have to be updated by future authors as needed.

4.3
Implemented threat model
The current implementation uses a simplified method of creating the threat model. This
implementation does not directly use the security properties when going over the mitiga-
tions, as it was seen as more helpful for a developer to select the most critical threats and
then go over them directly. This also permits them to group and mitigate multiple threats
should that be applicable. No method currently displays the communication paths and
the related threats; this problem is left for future work on the program. It was also decided
to use the probability or likelihood and severity provided by CAPEC rather than the more
straightforward version provided in Andresen[13], which determines the probability simply
from how exposed an entity is. While that certainly influences the likelihood of an attack,
this has already been considered in the CAPEC values, and these values will help differ-
entiate the threats that relate to the same type of exposure.
With these caveats, the implemented threat model functions by taking the provided enti-
ties in their respective layers and checking the premade threat map. Currently, there is
no method of updating or adding to the threat database, but the implementation uses a
more straightforward database than what was used in [13]. The database is only a map

explained in [13], but the resulting database would be simpler.

Minor issues
The menu bar has functionality as of yet but is expected to be one of the main methods

The initial view should include more aspects that are expected of this type of software as
seen in MTMT:
Figure 7.5: Microsoft Threat Modeling Tool initial view

Depending on how future developers choose to change the initial view, the included short
introduction should be changed to describe the use of the program more correctly.
The database needs to be simplified a bit more. Currently, the database contains two
ID systems, the one proposed in Andresen[13] and the one used in CAPEC. This is not
ideal as some threats have the same ID in both, but for others, they have been changed,
and this causes overlap where two distinct threats will have the same ID numbers. This
problem is temporarily solved by not using the new ID system made by Andresen[13].
Additionally, the database is quite lacking as of writing. Further expansion would be re-
quired; perhaps this can partly be solved by allowing product users to add new entities to
the database.
The program is currently using pixel amounts for size definitions. This is not ideal, as
things do not work as expected when resizing the window. Especially on larger screens,
there is a lot of space that is not used due to this. It is not trivial to implement automatic
resizing, but some effort should be put into allowing this.

8
Conclusion
This thesis has created and presented the framework and key features of an automated
threat modeling tool. This tool can create a simple threat model for small devices, recom-
mend mitigations to the found threats, and explain how such a threat could be exploited.
The tool uses well-established threat databases to ensure the threat list is up-to-date and
relevant. Further work is required on the tool before it contains all aspects of the model
proposed in Andresen[13], but the design was created to allow this expansion by future
developers.
UML design diagrams were created to ease this future expansion by showing the internal
design of the program clearly. The chosen diagrams illustrate the current implementation
in structure diagrams and the program’s intended behavior in the behavior diagrams.
The UI received significant attention in the thesis as this is what users will interact with,
and it was important to establish a simple and easy-to-use initial layout. The software was
designed to allow cross-platform use for the same accessibility reasons inherently.
Finally, an extensive future work guide was provided to show what needs to be completed
before the program is finished. This also includes potential additional features not initially
included in the requirements but considered worthwhile. UI examples from contemporary
tools are also provided to encourage borrowing from their layouts and ideas.
