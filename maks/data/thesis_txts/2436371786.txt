Summary
AIF is an approach for verifying stateful distributed systems such as web servers
and crypto APIs. It is based on an abstract interpretation approach that represents
an over-approximation of all facts that can ever become true in a system. A new
extension of AIF, called AIF-omega, allows for the declaration of infinite sets of
constants and corresponding infinite families of sets. An existing approach for AIF
for computing the fixedpoint of reachable facts is inefficient for large sets of constants
and does not terminitate on infinite sets in AIF-omega, in general. This project is to
investigate new strategies to overcome this problem and to compute a efficient finite
representation of the infinite fixedpoint. In particular, we use facts with variables
to represent an infinite set of concrete facts. This shall be combined with existing
techniques of representing the fixedpoint, in particular the factoring out the so-called
term implications to yield a compact and efficient representation. We shall, with
practical experiments assess the feasibility of the techniques.

CHAPTER 1
Introduction
A very successful abstraction approach over-approximates a protocol by a set of first-
order Horn clauses [3][10]. It assumes that all information participating in a single
session and sharing across different sessions is immutable. However, it has limitations
when protocols are able to change some information across multiple sessions. For
example, a certification authority might keep a list of keys that has been registered
and certified by it. Each key might be updated or revoked due to the client request
or the deadline has been reached. This non-monotonic behavior, where a fact derived
at some points become unavailable later, cannot be handled by Horn clause approach
directly. It requires what is true cannot become false by learning more information
[7].
AIF[5] was the first tool to tackle this problem while conserving the property
of over-approximating protocols by Horn clauses.
It is an approach for verifying
stateful protocols or distributed systems such as web services[9], based on an abstract
interpretation approach that represents an over-approximation of all facts that can
ever become true in a system. AIF allows users define a fixed number of sets and
constants which can be added to and removed from sets in a transition system. If
in the same sets the value of members are equal, they are abstracted as the same
equivalence class. It is the abstraction that encodes the set memberships as terms,
e.g. it identifies all keys that are known to the server as valid and at the same time in
the key ring of a honest agent. In this way, it is possible to model an attack such as
the intruder knows a private key belongs to an honest agent and the key is registered
as valid in a server. A set of Horn clauses describe a set of facts and if the attack fact
is reachable, the security of a protocol is violated.
The limitation of AIF is that one needs to specify a fixed number of constants to
reach a concrete number of N sets. A new extension of AIF, called AIF-omega [7],
allows for the declaration of infinite sets of constants and corresponding with infinite
families of sets. For instance, we can declare User is an infinite set and specify a set
ring(User) using one family to represent each user a ∈User maintains a set of keys
in its own key ring ring(a). One restriction defined in AIF-omega is that the sets of
a family must be pariwise disjoint [7]. If this restriction is violated, for instance, for
any two users a and b such that ring(a) ∩ring(b) ̸= ∅, it is regarded as an attack.
Hence, it is very important that AIF-omega requires all family parameters of infinite
type must have the uniqueness invariant [7].
An existing approach for AIF to compute the fixedpoint of reachable facts is

CHAPTER 2
Preliminaries
AIF is an over-approximation approach verifying protocols and web services. The
limitation is one needs to declare a fixed number of N sets. AIF-omega, the extension
of AIF, overcomes this limitation by allowing to specify N families of sets. Each family
contains a countably infinite sets. For instance, we can declare a Agent is an infinite
set and a family ring(Agent) represents for every a ∈Agent it maintains a set of key
ring(a).
The problem is the existing approach of AIF computing fixedpoint does not stop
on infinite sets. The main idea of this thesis is using facts with variables to represent
infinite set of concrete facts. Each variable has a type which ranges over a set of
infinite constants.
In this chapter, we will introduce the most general unification algorithm. Then
we will demonstrate how we construct a type tree based on user-defined types. The
type tree will be further used to compare if one type is more general or equal general
than the other one. Furthermore, we will introduce a simple example modelling a
very simple security system and translate the example to Horn clauses.

2.2
Type Tree
The AIF-ω specification can declare user-defined types either by giving a finite sets,
or using ”...” to represent a countable number of elements. Additionally, a type also
can be defined by the union of other types. A type of all kinds of users can be the
union of Honest and Dishon as User = Honest ∪Dishon. An extensional semantics
[[T]] is given to each type to make the ”...” notion more precisely[7]. For the running
example, we have
Honest = {...}, [[Honest]] = {honestn |n ∈N}
Dishon = {...}, [[Dishon]] = {dishonn |n ∈N}
Agent = {...}, [[Agent]] = {agentn |n ∈N}
Variables declared in this way are called enumeration variables[5]. All sets or database
defined in the AIF specification include enumeration variables. For expample:
Sets : ring(Honest), db(Server, Agent, Status)
means that every honest agent has his keyring ring(Honest) and every server has a
database for every Honest A and the key’s status includes either valid or revoked. The
problem is if a type contains infinite elements, it will lead to the expansion of infinite
sets.
The idea is we use a variable and a typing Γ to denote an infinite set of a type.
Typing Γ contains variables type and represents all values of each type can be used to
substitute variables. We can write variable A is type of Agent as A : Agent, variable
H is type of Honest as H : Honest and so on. Each family includes several variables
and a typing Γ. In order to maintain the uniqueness invariant requirement, different
types should be represented by different variables. It is not allow to define variable
A : Agent and A : Honest in the same family. Hence, families can be reconstructed
as:
Sets : ring(H), Γ1 = (H : Honest)
db(S, A, revoked), Γ2 = (S : Server, A : Agent)
For comparing one variable is more general than the other one, we need type
information of two variables. We build a tree structure to illustrate the relation of
all user-defined types. In the AIF-omega specification, we can define the following
types:
Honest = {...}
Dishon = {...}
User = Honest ∪Dishon
Server = {...}
Agent = Honest ∪Server
Status = {valid,revoked}
From the example, we can observe that the Agent and User contain an overlap
type Honest. This causes a problem, for example, if we define variable A ∈[[Agent]]
and variable U ∈[[User]], it is impossible to say either A is more general than U,
or U is more general than A. In order to avoid this situation, we make two extra
restrictions for user defined types.

2.3
Running example
We use a model of key-server rules as an example in this thesis. There are three actors:
an honest user, an honest server, and a dishonest user (the intruder). Each user is
considered to have a keyring and a key pair which consists of a public key k and a
private key inv(k). The keyring is denoted as k ∈ring(A) and A represents infinite
constants of type User. The server S keeps two sets for each user: a set of valid keys
and a set of revoked keys. A valid key of a user A stored in the server database is
denoted as k ∈db(S, A, valid). We express a revoked key as k ∈db(S, A, revoked).
For each dishonest user, iknows(M) means the intruder knows M.
keyReg(A : User, S : Server, PK : value)
= [PK] => iknows(PK) · PK in ring(A) · PK in db(S, A, valid)
The first keyReg rule describes every user can register a freshly generated public
key to a server’s database. The left-hand side of the arrow is empty means that the
rule can be applied in any state. A fresh value PK is created by the user A and the
intruder can learn it as it is a public key. The key is added to the keyring of A as
well as the server S keeps the PK as a valid key. Although the keyReg rule does not
mention whether PK is in db(S, A, revoked) or not, the fresh value should not be a
member.
userUpdateKey(A : Honest, S : Server, PK : value, NPK : value)
iknows(PK) · PK in ring(A)
= [NPK] => NPK in ring(A) · iknows(sign(inv(PK), pair(A, NPK)))
The second userUpdateKey rule describes that an honest user can update any of its
valid keys contained in keyring A by creating a new public key NPK. The NPK
is further added to keyring A and PK is removed from A’s key ring. Since all facts
are persistent, PK ∈ring(A) is not repeated on the right-hand side. User A sends a
update message sign(inv(PK), pair(A, NPK)) to the server and it can be obtained
by the intruder.
serverUpdateKey(A : User, S : Server, PK : value, NPK : value)
iknows(sign(inv(PK), pair(A, NPK))) · PK in db(S, A, valid) · NPK notin db(_)
=> PK in db(S, A, revoked) · NPK in db(S, A, valid) · iknows(inv(PK))
The third rule serverUpdateKey illustrates the server receiving the update message,
adding new key in valid key set and putting the old key to revoked key set. When the
server receives a update message, it will check if the signing key PK currently exists

in its keyring. The attackDef rule defines what is an attack: if the intruder gets
a private key that is registered as valid key in some servers’ database as an honest
agent, it will be considered as an attack.
In the original AIF-omega specification, every variable has a certain type such
that they are under control of what they can be. Except for user-defined types, it
has variables of type value and type untyped. A variable of type untyped represents
the range of arbitrary message, which means the intruder can apply any message
to untyped variables contained in a rule.
Then the intruder can produce infinite
messages. In the abstraction, variables of type value are always replaced by terms of
the form val of 0, or a set expression, or a variable that ranges over set expressions,
then we call it type of membership. This is not the ordinary terms. They are just
terms for denoting set memberships. The type membership should not be the range
of arbitrary messages which lead to an infinity. They can only range over the sets
that the user has defined.

2.4 Translation to Horn clauses

2.4
Translation to Horn clauses
In AIF, facts are persistent. A fact that holds in one state also holds in all other
reachable states [5]. Persistent facts can be easily translated to Horn clauses as they
have similar semantics.
The non-persistent facts in our running example are set
memberships that they might belong to some set in one state but not in other states.
For example, we may have k ∈ring(A) on the left-hand side of a rule and it removed
from the right-hand side because key k is no longer kept in A’s keyring.
In AIF-omega, an infinite set can be declared as, for instance, Agent and family
ring(A) is introduced to represent a countably infinite number of sets. AIF defines a
new function symbol val/N where N is the number of families. In this way, we can
combine memberships we wish to encode as a single fact. For each user A, we have
the set ring(A) to holds the keys of this user and valid(A) and revoked(A) are the sets
where the server stores the valid and revoked keys of A respectively. We can define the
function val/3 and encode these three sets as val(k ∈ring(A), k ∈db(S, A, valid), k ∈
db(S, A, revoked)) for a key k generated by user A. A valid, non-revoked key, and
also kept in the keyring of a user can be described as val(ring(A), db(S, A, valid), 0).
The val function can be used as a sub term. Hence, we can model the intruder
knowledge of keys. For example, we can express an intruder learns a key k that is
registered as a valid and non-revoked, and not keeping in the keyring of a user as
iknows(val(0, db(S, A, valid), 0)). If key k changes its membership from valid key set
to revoked key set, the intruder can also derive iknows(val(0, 0, db(S, A, revoked)))
as the key does not change. This intuition has been defined by [5] as term implication.
Take the userUpdateKey rule as an example, we translate it to Horn clauses as:
iknows(val(ring(A), db_valid, db_revoked)) ·
occurs(val(ring(A), db_valid, db_revoked))
⇒
iknows(sign(inv(val(0, db_valid, db_revoked)), pair(A, val(ring(A), 0, 0)))) ·
occurs(val(ring(A), 0, 0)) ·
timplies(val(ring(A), db_valid, db_revoked), val(0, db_valid, db_revoked))
The variables of db_valid and db_revoked are type of membership that can be sub-
stituted by either 0 or the user defined set of db.
A new fact symbol occurs/1 is introduced to represent every variable in the rule.
In the example, a valid key PK occurs on the left-hand side of the rule and is
removed on the right-hand side. A freshly generated key NPK only occurs on the
right-hand side. Hence, we add occurs(X) on both left-hand and right-hand side of
the rule. The variable db_valid and db_revoked are placeholder for what status the
key has. Actually, this rule removed a key from the keyring but the key keeps the
same status in the server sets as on the left-hand side. The timplies fact reflects that
for any fact that contains the abstract value of val(ring(A), db_valid, db_revoked)
then val(0, db_valid, db_revoked) is true also.

Defining infinite constants in AIF-omega leads to infinite sets such that the fixedpoint
computation cannot terminate. Our strategy is using variables and its types to rep-
resent infinite or a large set of constants and introduce them into facts and sets. We
require every variable must have a type. It can be either user-defined types which
range over infinite sets of constants, or a type of membership. The membership type
variables can only be substituted by terms rather than arbitrary messages. It avoids
the messages explosion such that the fixedpoint computation will not stop. In this
chapter, we are going to introduce some concepts and demonstrate algorithms we
used to compute fixedpoint.
Applying variables in facts requires an additional typing information to describe
which types those variables are. For user-defined types like Agent, [[Agent]] denotes the
infinite set of constants. A fact f(A) represents a set of ground facts {f(a), f(b), ...}
if [[Agent]] = {a, b, ...}. The semantics of a symbolic fact is defined as:
[[F, Γ]] = {δ(f)|δ(x) ∈Γ(x), for all variables x that occur in f }
Γ is for all variables introduced by substitution δ and may remain from original typing.
A symbolic fact contains a set of symbolic terms and a symbolic term can be
a variable, a constant or a composed term that contains a set of symbolic terms.
A typing Γ is then needed to show the types of those variables contained in terms.
Hence, the symbolic term is defined in the same way as symbolic fact as:
[[s, Γ]] = {δ(s) | δ(x) ∈Γ(x), for every x ∈s }
The definition of ground fixedpoint has been defined as
Definition 1. Given a set of Horn clauses R and r = (p1, ..., pn →q) for each
r ∈R, let fact F be the least set such that for every grounding substitution δ if
{p1δ, ..., pnδ} ∈F then qδ ⊆F. We call such F is fixedpoint[2].
Based on that, we can define what a symbolic fixedpoint is:
Definition 2. A symbolic fixedpoint ¯F is a set of symbolic facts together with typing
such that [[¯F]] = F.

side symbolic facts are derivable. We can obtain a set of substitutions with a variable
typing by unifying every left-hand side fact with every fact in C. The unification may
fail and the result of Nothing is returned representing right-hand side facts cannot be
inferred.
For each left-hand side symbolic fact, its variables are firstly substituted by a given
unifier. The given substitution may be from other left-hand side facts’ unification or
initially empty. Then unifying the substituted symbolic fact with every symbolic fact
in C, we may obtain a list of substitution as it is possible to be unified by several
facts. Each unifier is merged into the given substitution and it will further be used
to substitute the rest left-hand side symbolic facts.

Fixedpoint computation
The process of computing fixedpoint stops when no new consequences can be inferred.
The function Ωtakes a set of Horn clauses and a set of symbolic facts C contained in
fixedpoint as arguments and it derives a set of symbolic facts C′. Since C′ contains all
symbolic facts Ωderived, we need to compute whether there are some new symbolic
facts are inferred or not. If there are some symbolic facts in C′ are covered by C,
it means those symbolic facts has been derived by the earlier computation or those
symbolic facts are less general. We will not add those symbolic facts in C. On the
other hand, there are some symbolic facts in C may also be covered by C′ because
Ωmay derive some symbolic facts that are more general. Since fixedpoint only keeps
the most general symbolic facts, those less general facts in C will be further replaced
by more general ones.
In order to compare whether one symbolic fact is more general or equal general
than another one, we apply the variant unification defined in section 3.2. It returns
a Boolean value to illustrate ≥relation based on the given symbolic facts.
The
variant unification returns True only if f ≥g or f and g are disjoint facts.
We
define a function isCoverd takes a symbolic fact (f, Γ) and C as arguments and it will
compare whether existing a symbolic fact in C that is more general or equal general
than f. If it does, isCoverd returns true and we can define (f, Γ) is covered by C.
Otherwise, the result is false.
Algorithm 17 isCoverd(f, Γ), C
Input: A symbolic fact (f, Γ), a set of symbolic facts C
Output: Boolean
• if existing symbolic fact (g, Γ′) ∈C such that isGEQ g f (Γ′ ∪Γ) is True, then
return True.
• if existing symbolic fact (g, Γ′) ∈C such that isGEQ g f (Γ′ ∪Γ) is False, and
the result of isGEQ g f (Γ′ ∪Γ) is False, then return True.
• Otherwise, False.

and decrypt, such that the intruder is able to reveal other sensitive keys. The iik1
and iik2 rules show that the intruder initially can obtain key K1’s handle and key

intruder can decrypt a cipher if he gains the key that is used to encrypt the message
by rule decrypt2. The security goal is represented by the attack rule attack1 that it

CHAPTER 5
Conclusion
In this thesis, we have explored a new strategy to tackle the problem that the original
approach of AIF computing fixedpoint does not terminate on infinite sets. We use
symbolic facts to represent infinite sets of concrete facts. Each symbolic fact contains
variables and types which range over infinite sets of constants. Since each variable
has a concrete type, it cannot be substituted by arbitrary messages. Therefore, it
avoids the issue of infinite messages explosion.
The benefit is the fixedpoint computation can terminate and it contains a set
of derived symbolic facts. It is also verified that if an attack fact can be reachable
by concrete facts, it is also reachable by symbolic facts. We have shown how our
algorithms compute fixedpoint. With the deeper understanding of this project, we
realized some algorithms we implemented previously cannot handle all cases and we
proposed more precise algorithms in this thesis. We will transfer them into code as
the future work.
During writing the thesis, in the process of making it more formal, we understand
the problems better. Now we have precise mathematical models but the implementa-
tion is lagging a little bit behind. It still has few bugs and we get into the wrong result.
But we have cleared how they should be. We will complete the implementation in
the future and make more experiments, for example HSM[8], with different protocols
to find more problems we may not take into account.
