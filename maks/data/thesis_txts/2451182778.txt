Summary (English)
Most people have devices that connect to the internet. These form the Internet
of Things(IoT). They have various functions. There is the motion sensor that
tells the lights to turn on, the TV that streams internet content, or the door
lock that can be unlocked remotely with a phone. Many IoT systems lack in
security, and can be exploited. An attacker might learn when a person is home
by looking at the previously mentioned devices, they might even unlock the
door. Less direct attacks are also possible. Sensitive information can be pulled
from the device, or the attacker can add the device to a botnet.
In this thesis, we propose a model for performing a penetration test on a generic
IoT system. It establishes the attack surface through an exploratory mapping
of communication patterns and API. Common attack vectors are explored to
ﬁnd vulnerabilities, and show how they can be exploited. We use the model to
examine the security of the three home security systems bought over the counter
at DYI stores in Denmark. By using the model to ﬁnd vulnerabilities, we hope
to demonstrate it’s value.

Chapter 1
Introduction
We are currently experiencing explosive growth of devices connected to the
internet. There are smart fridges that let you see the contents of the fridge,
reminiscent of the original webcam which was made to see if there was coﬀee in
the break room. There are lights connected to the internet, so that they could
be controlled from virtually anywhere with internet access. There is the baby
monitor that lets you monitor your sleeping baby, giving you information like
heart rate, position, temperature, audio and so on.
The big question is, are these devices safe? Research has shown that many IoT
devices have vulnerabilities that can lead to malicious attackers compromising
them. A hacker could watch you through your camera. They could listen in
through your smart speakers.[Pap18] Even if the attackers do not intend to twist
the intended use of the device, they could be added to a botnet, using them to
perform DDOS attacks, or mine cryptocurrencies.
In the paper Internet of Hackable Things[DGM17], Dragoni performs a survey
where he ﬁnds that the security of many IoT devices is lacking. From hospital
equipment to Smarthome devices to Cars. The security of the surveyed devices
have been hacked because their security was either weak, or it was non-existent.
How do we evaluate the security of IoT devices? There are guidelines on how
to develop secure IoT systems, and there are many ways to verify speciﬁc vul-

nerabilities. There are however not as many comprehensive guidelines on how
to evaluate the security of IoT systems as a third party. Through this work we
will design and test a generic model for analysing the security of IoT systems.
The model is a four step plan as outlined bellow.
• Map Attack Surfaces
• Analyse for possible vulnerabilities
• Test whether we can exploit the vulnerabilities
• Suggest mitigation techniques to protect against found attacks
The attack surface mapping consists of noting the functionality, protocols, phys-
ical features, security policies and so on of the components of the system. This
is used to perform an analysis of what kinds of attacks the system is vulnerable
to. This is then directly translated to testing of of the attacks.
We test the model by using it to analyse the security of three home security
systems. By using the model, we where able to show that there are critical
ﬂaws inn examined systems, and that they are vulnerable to, amongst other
things, a malicious actor gaining full remote control of the systems without
physical access.
The model and the evaluation of the model have varying levels of details. This
is reﬂective of the amount of time available to commit to a project, and the
knowledge required of diﬀerent technologies to perform a comprehensive anal-
ysis.
There are several ﬁelds we have only been able to perform a cursory
analysis of because of this, and it means that there is room for further work.
As an example, we don’t have deep knowledge of how to perform a side-channel
attack.

In order to develop and test a model for examining the security of IoT systems,

theory behind IoT. We take a look at the structure of IoT systems, common

Furthermore, we look at several practical examples of attacks on IoT, and how

Because of the eventual goal of analyzing the security of home security alarms

2.1
IoT Fundamentals
What is the Internet of Things? The traditionally the internet was made up
of primarily computers and servers communicating with each other. Then we
started adding internet functionality to every day objects, the earliest known
example being connecting a camera to the internet in order to see weather there
was coﬀee in the pot or not. Later, internet functionality have been added to
many diﬀerent types of devices. Fridges, baby monitors, lamps, even cars. It is
estimated that there will be ten billion IoT devices by 2020. [Lue]
Generally the systems consist of sensors, actuators, and a dashboard for control.
It can make use of backend systems for business logic and authentication.
IoT devices are commonly controlled or accessed through either a web interface
or an app.
2.2
OWASP Mobile Top 10
The Open Web Application Security Project (OWASP) is a community project
worked on by various security minded people, and has done signiﬁcant work in
the ﬁeld of mobile security. They have listed the ten biggest security vulnera-
bilities that might aﬀect mobile applications[Pro], some of which we explore in
the following subsections. They are ranked by a combination of exploitability,
prevalence, detectability, and technical impacts. They explain for each vulner-
ability what examples of attack vectors would be, and give suggestions for best
practices to detect, mitigate and prevent exploits.
2.2.1
M2 Insecure Data Storage
Data needs to be secured stored. This ﬂaw is can be primarily called a lack
of cryptographic security when handling data. The scenario runs as follows, if
the adversary has achieved physical possession of the mobile device, or if they
have managed to install malware on the mobile. They might be able to access
any information stored on the device. This could include passwords, usernames
and other less obvious private information. Internet history or images are other
example. If the mobile is connected to a open or compromised access point,
then it anything sent over the internet could be accessed by an adversary.
==PAGE:21==
2.2 OWASP Mobile Top 10
7
In order to mitigate the vulnerability, one ought to consider how the app stores
information. If the app stores information it should ideally be in such a manner
that even if it is accessed by an adversary, it is not useful to the adversary.
Encryption of data, hashing and salting of passwords are the two immediate
goto’s. The developer should consider how libraries and API’s handle their own
storage, such as caching, internet history and other logging.
2.2.2
M4 Insecure Authentication
Only giving access to those who ought to is one of the core aspects of security.
Authentication fails when an adversary is able to exploit vulnerabilities in an
authentication scheme, or to bypass it. This would result in an adversary being
able to access information and services they should not be able to, with the app
failing to recognise who it is dealing with. Several examples are given of how
insecure authentication might occur, such as storing passwords or shared secrets
locally, having a weak password policy or not using an access token when calling
backend services.
OWASP mentions several ways to preventing insecure authentication.
Some
key examples include not lowering authentication standards when porting a
service from web to mobile, authenticating server-side, not relying on spoofable
information for authentication, and not relying on four digit pins or TouchID.
2.2.3
M5 Insuﬃcient Cryptography
Using cryptography to secure your data as described in subsection 2.2.1 is well
and good, but it is necessary to use cryptography that is robust to attacks. If
the process(es) responsible for encryption and decryption are ﬂawed, or if the
encryption itself is weak, then any information hidden behind it is vulnerable.
Other causes of this problem is developers using in-house implemented cryptog-
raphy, or cryptographic algorithms with known vulnerabilities, such as MD5,
SHA1, WEP. The proposed solution is to avoid storing data on the mobile de-
vice, to employ cryptographic algorithms that are expected to last for at least
ten years, and to follow NIST guidelines on recommended algorithms.

2.2.4
M9 Reverse Engineering
Reverse engineering is the process of taking code that has been somehow pro-
cessed (compiled, obfuscated) and gaining an understanding of what the original
source code was, or to understand what the . By downloading an app used by
a mobile device, source code, libraries and other resources can be extracted by
using tools such as jadx, hopper and so on. The primary information an ad-
versary would be interested in is information related to backend servers, what
protocols are used for communication, how data is stored and so on.
The only way to counter this is security through obfuscation. It is thus merely
a layer of security that exists to hide other security ﬂaws, or to otherwise incon-
venience an adversary.
2.2.5
M10 Extraneous Functionality
Extraneous functionality includes any functionality that is included in an app
that is not intended for the end user to use. This can lead to an attacker gaining
access to logging functionality that gives information about the backend, or
API’s only intended for testing being open.
It is primarily prevented by having secure code reviews where it is veriﬁed
that there are no debugging switches in the released software, undocumented or
unintended API endpoints, and so on.
2.3
Authentication
National Institute of Standards and Technology (NIST) has created the docu-
ment "National Identity Guidelines". It outlines requirements for three diﬀerent
Authenticator Assurance Levels (AALs). These range from assuring some con-
ﬁdence that they are authenticating the correct person, to assuring very high
conﬁdence of the same.
Some of the noteworthy requirements for the lowest level of AAL is that PINs
are required to be at least six digits, if chosen randomly, and eight if user chosen.
Hints as to what the password is, is never to be provided to an unauthenticated
entity. Passwords are to be at least eight characters long, with the option to
be at least ﬁfty characters long, and are to be stored in a hashed + salted
format with their salts. Further requirements are made for how to establish

2.4
RF Security
What is the state of security in RF system?
Home alarm systems are one
example of systems that sometimes use RF for communication. Dragoni says
that there are critical ﬂaws in several home alarm systems. Some of these ﬂaws
allow an attacker to take control of the system: triggering alerts, turning them
oﬀ, jamming sensors or just making the system ignore them.[DGM17] . This is
conﬁrmed by security consultant Cédric Bassem’s security experiments where he
examines two systems. He ﬁnds a couple of vulnerabilities, such as vulnerability
to jamming attacks, replay attacks, and other vulnerabilities covered in the
following sections.[Bas18]
2.4.1
Rolling Keys
One way of mitigating replay attacks is to use rolling keys. The principle of
rolling keys is that both receiver and transmitter have the same pseudo-random
keygenerator. The receiver pre-generates N keys (K0, K1 .. KN). When the
transmitter sends a message it is encrypted with a newly generated key Kn.
The receiver then tries to decrypt the message with key K0, and goes on until it
either successfully decrypts the message or it runs out of keys. It then deletes all
keys up until Kn, and pre-generates as many new keys as was deleted. There is
usually some mechanism for synchronizing the keys of a receiver and transmitter
if they no longer have an overlapping keyspace.[Bra01]
While this technique ensures that a receiver cannot receive the same key twice, it
is still vulnerable to replay attacks in scenarios where the transmitter transmits
a message with a key, but the receiver does not receive it.
One example of
this is the "rolljam" vulnerability that has been successfully been used to hack
cars. It works by having an adversary jams the ﬁrst two keys transmitted, then
replays the ﬁrst one, thereby giving them the opportunity to replay the second
key, until the transmitter transmits a third key. [Kam15]

2.4.2
Relay Attack and Distance Bounding
Even with rolling keys, there are examples of attacks that can be made. The
"rolljam" attack is one example, another is a relay attack. It is done by simply
relaying the authentication sequence, such as between a car and a car key. It
is mitigated by employing a technique called distance bounding, which ensures
that the authenticating party and that the verifying party are both within a set
distance. It is done by measuring the time it takes to make a challenge and get
a response, as round time and the speed of lights sets a upper bound on how
far the two parties can be from each other. It can be further narrowed down by
knowledge of the speciﬁc devices and protocols involved.[MSTPTR18]
2.5
Attacks on IoT
In this section, we discuss attacks made against various IoT devices and proto-
cols. We take a look at attacks that disconect devices from the network, that
compromise communication over IP, and cover some ﬂaws found in IoT devices.
2.5.1
Deauthentication
Network protocol IEEE 801.11 is a protocol used by many AP’s. While most of
the communication done between a connecting Client and the AP is encrypted,
management frames are not. Management frames are used for various purposes.
One of them is to tell a connected device that it is no longer connected to the
access point. It requires only knowledge of the protocol used, the MAC address
of AP and Client. Both of which can be gathered by passively listening to any
management frames sent between the two.
This means that an adversary can potentially disconnect a client from a network,
and keep them disconnected by repeatedly sending spoofed deauthentication
frames.
This vulnerability is solved in the IEE 802.11ax protocol released in January
2019.
It has encryption on management frames, making it more diﬃcult to
perform deauthentication attacks. It is therefore preferable to use routers and
other AP’s which have implemented this more recent protocol.[BT09]

2.5.2
KRACK Attack
Further vulnerabilities have been found in the 802.11 WPA and WAP2 protocol.
WPA and WPA2 uses a fourway handshake to generate a fresh session key. The
KRACK attack forces a victim to install an already-in-use key by editing and
replaying handshake messages. This resets nonces and replay counters, making
them vulnerable to replay attacks.
The attack aﬀects diﬀerent implementations of WPA2 diﬀerently. Used against
AES-CCMPT it allows for replaying and decrypting packets, this is also true
for WPA-TKIP, but here it allows forgery of packets as well. The authors have
veriﬁed that every WiFi device is vulnerable to some variant of the attack, but
notes that the most extreme example is Android version 6.0, where they are
able to reliably set the encryption key to all-zeros.
As it is a ﬂaw in the standard, the only way to prevent this attack is to update
vulnerable devices.[VP17]

Chapter 3
A Generalized Approach to
Examining the Security of
IoT Systems (Method)
In this chapter, we discuss a generalised model that describes how to examine
the security of a IoT system. We look at how we map out the attack surface.
We then look at how we identify attack vectors, how to select techniques that
implement the attack vectors, and how we verify that the techniques work.
We want to know what the components of the system are made up of and how
they interact with each other. This is done in three parts, physical mapping,
software mapping, and communication mapping. We then examine the mapped
attack surfaces. We then start asking questions designed to identify vulnerabil-
ities in the mapped system. When we have identiﬁed potential vulnerabilities,
we attempt to verify that they are a problem. This is either done trough desktop
analysis or empirical testing.

(Method)
3.1
Analysis
In section 2.6, we looked at a couple of diﬀerent approaches to testing the se-
curity of IoT systems and web applications. We want to draw something from
each of them, while adding something new. We want have the thorough mapping
done in the IoT hackers handbook, we want to expand upon the communication
mapping done by Bjørneset, and select vulnerabitlies which can then be anal-
ysed and tested. Making something more manageable than OWASPs testing
framework.
We’re starting with the question, how can we test the security of a system?
While it is impossible to prove that a system is completely secure, it is possible
to prove that it is resilient, or vulnerable to speciﬁc attacks.
It only makes
sense test for vulnerabilities that are relevant. The ones that have potential for
dealing damage when exploited.
We need to perform an analysis that takes the whole system into consideration.
This will include the physical components, the software running on them, and
their communication.
We then start asking questions about the security aspects of the mapped attack
surfaces. We question how communication is secured, how diﬀerent components
authenticate each other, how data is processed and where it is stored.
3.2
Attack Surface Mapping
In this section we discuss mapping of the systems, physical and digital and net-
work. By rigorously examining the systems, we hope to uncover possible attack
surfaces. We perform three diﬀerent layers of mapping: Hardware, Software and
Communication.
Physical mapping will give us information regarding what the devices are, and
what their capabilities are. It will give us some information regarding how to
physically interact with their inputs and outputs. Furthermore it will give us
information regarding how they interact with each other on a user level. The
network mapping will give us information regarding who the entities are and
how they communicate, including backend systems. Software mapping will give
us information about how information is processed, how it is stored and how
authentication is done.
It might also reveal other things, for example poor
implementations that exposes the system to attacks.

3.2.1
Physical Mapping
The very ﬁrst point of order when examining a IoT system should be to take a
look at every component of that system. We want to start with documenting
every interesting part of each component of each system.
There are several
diﬀerent things that might be considered as interesting depending on the nature
of the investigation.
Sensors are the primary way that the system perceives the world around them.
They primarily consist of analogue readings of things like temperature sensors,
light sensors, motion sensors, radio receivers, cameras, microphones etc.
Data Ports includes any port from which the system reads digital data, such
as Ethernet ports, SD card ports, USB ports, SIM card ports, UART, etc.
System to user communication includes any part that communicates infor-
mation to the user. Examples are display screens, lights, speakers, etc.
Physical access barriers are screws, access ports, etc. These are primarily
documented to show how easily other things can be accessed.
Chips are amongst other things where the ﬁrmware is installed. It might be
possible to retrieve the ﬁrmware for reverse engineering purposes.
3.2.2
Software Mapping
The general idea of software mapping is that accessing and decompiling the
software, we can gain information about how the system works.
The three
kinds of software we focus on are:
• Apps: Run on client devices such as PC’s or Phones.
• Webpages: Runs on servers, accessible through IP.
• Firmware: Runs on hardware, semi-permanent and low-level code.
Generally when we decompile source code, we are interested in information
about how authentication is done, how information is stored, processed and
transmitted, what other components the software communicates with and how
it does this.

Furthermore, we are interested in spotting poor coding practises which may be
security issues. If it is possible to identify code libraries, then they might have
known security issues.
App source code accessible through decompilation tools such as jadx. We want
to know what and how information is stored on the device.
Webpage code semi-accessible on client (javascript typically). We want to know
whether XSS and CSRF are prevented. We want to know what information is
stored in cookies.
Firmware can either be acquired from the producer, downloaded from update
location, or copied of the ﬂashed device. Depending on the state of acquired
code, it might be required to use reverse engineering tools.
3.2.3
Communication Mapping
How do the diﬀerent components communicate? Do they communicate over
radio? Bluetooth? Are they only able to communicate over telnet on an open
WiFi network? Depending on the communication medium, there might be built
in security, or there might be known vulnerabilities. We are therefore interested
in knowing what communication protocols are used by the components.
This might be found in the manual, product speciﬁcations, or supplied by the
retailer/producer. If the system is FCC certiﬁed, then there should be an FCC
ID on it. FCC ID’s can be looked up on the FCC website, but we like using the
website www.fcc.io. Sometimes the precise protocols are not speciﬁed. Should
this be the case, it might be necessary to analyse communication in order to
identify the protocols.

IP Communication
If any of the components communicate over IP, it would be appropriate to set
up an AP capable of recording datapackets, fx. PineApple Nano, and connect
the internet capable parts of the system to said AP. Now perform regular user
activities. Install any required app or access the any other interface. Create an
account. Log in. Send some commands. Change some settings. Do whatever
regular activities are appropriate. Then analyse the packets, see what protocols
where used. If possible, see who the diﬀerent devices communicate with. De-
pending on the level of security, this will yield information regarding protocols,

Now that we know what MAC and IP addresses are used by the components in

3.3
Vulnerability Identiﬁcation
So we know what components are included in each system. We know how they
communicate and how they work. But what can a malicious actor do? In order
to identify vulnerabilities in a system, we need to start looking for the things
a malicious actor looks for. Generally speaking, there are two reasons for an
attacker to start focusing on an IoT system. Either they want to leverage some
king of control of the system, or they want to access information stored within
the system. But how can one ﬁgure out what techniques to use? Just ﬁnding
one vulnerability can be enough for an attacker, as it might be a big enough
vulnerability to accomplish their goal.

3.4
Validating vulnerabilities
In this section we discuss various ways to verify vulnerabilities. We use both
empirical testing and desktop analysis. Empirical testing is done by performing
attacks on the attack surfaces. Desktop analysis is done by looking at attack
surfaces and It is generally preferable to test any vulnerabilities found through
desktop analysis, but it might not be practical.
3.4.1
Desktop Analysis
Desktop analysis is done by looking at any given attack surface, comparing it
to a technique, and proving on paper, that the attack surface is vulnerable to
the technique
==PAGE:36==
22
A Generalized Approach to Examining the Security of IoT Systems
(Method)
This would include things like:
• Proving that a used encryption method can be broken, and communication
using it can thus be compromised.
• Showing that there is no freshness in a message, which can then be re-
played.
• Finding that a password is stored in plaintext, shows that it is not stored
properly.
Furthermore, any technique found in a vulnerability database that matches one
of the attack surfaces can be assumed to be proven through either desktop
analysis or empirical testing, and is thereby a valid proof that the technique
works.
3.4.2
Empirical Testing
Having mapped the attack surfaces, decided on a attack vector and technique, we
might lack the resources to prove that the technique would work when applied.
This is when we want to just go ahead and perform the attack.
If we can
conﬁrm that it works, then we have identiﬁed a vulnerability and a technique
that exploits it. An example could be that you’re seeing several instances of the
same message. If you can re-transmit the message, and the system reacts as if
it where a legitimate message, you’ve proven that the system is vulnerable to
replay attacks.

5.2.2.4
Password Policy
The password policy of App B is that the user inputs a four digit PIN twice
(once for conﬁrmation), and a hint. There is no input validation on the hint
or the PIN, other than a length check on the pin, and a conﬁrmation that the
two PINs are the same. When accessing the login screen, it is possible to either
login with pin, see password hint, or press forgot password. If forgot password
is chosen, the user needs conﬁrm that they want to delete all local data.
According to NIST, a four digit secret is not acceptable, nor is it acceptable
to share information about a password to an unauthenticated user, in the form
of a hint. There is no input validation on the password hint, meaning that a
user can compromise the security of the app by entering the PIN as the hint, or
otherwise reveal information about the password.
5.2.2.5
Bypassing Login Activity
The only way the app authenticates the user, is to open a login screen whenever
the app is opened. This is the case whether the app has been shut down, paused,
or stopped so other apps can run. This is done adding the login activity to the
activity stack, thus setting it as the current activity. The back button usually
pops current activity, and sets the next activity on the activity stack as the
current activity. If there is no activities left on the stack, the app closes We are
therefore interested in seeing what the back button does.
By reading through the code for the login screen, we notice that there is no code
written to handle the back button. This implies that we can bypass the login
activity and warrants further testing.

identiﬁed vulnerabilities, any exploits tested, and suggest further possible ex-

B comes out of the box with the same password, so if an attacker is able to
access the same network as one of the cameras they will be able to connect to
the camera too, assuming the user has not changed the password. This also
makes it worse that Camera B accepts open AP connections, beyond the usual
security concerns with such networks.
We have shown other vulnerabilities in the apps, such as insecure password
storage, authentication bypass and poor password/authentication policy. These
mostly require either physical access to the devices they’re installed on to exploit,
or root access, rendering them more situational than the other exploits.
The physical examination of each component is the one part of our attack surface
mapping that we feel gave us the least useful information, compared to the time
required to properly document examining it. We believe that we could have
gotten more out of it, if we had spent more time focusing on reverse engineering
ﬁrmware, UART communication and so on. It also did not really help that
we chose to primarily focus on attacks that did not require physical access, as
those are the attacks that do would likely have beneﬁted from a close physical
examination.

6.2
Reﬂection
Adding alarm systems to the internet of things gives additional functionalities,
some of which are hugely useful, such as remote control and remote alarm no-
tiﬁcations. It also adds further attack surfaces, creating new opportunities of a
malicious actor to compromise the system, and new reasons for them to do so.
This is reﬂected in table 6.1, we have found half as many exploitable vulnera-
bilities for system C as we have for the equivalent system B. Not only are there
more opportunities to compromise the system, there are also more reasons to
do so. As an example, system B has a webcamera, which if compromised is not
only a source of video footage, but a device located on the inside of a homes
network. It also communicates directly with a phone. While we have not found
any immediate vulnerabilities related to this, the potential is still there, and
thus the point stands. When you add internet functionality to devices, security
needs to be a core part of the design.

Chapter 7
Conclusion
We have designed a process for evaluating the security of IoT systems. The
model outlines a method for mapping an IoT system, for analysing it for possible
vulnerabilities, testing whether we can exploit those vulnerabilities, and then
suggesting mitigation techniques.
Using the model, we have been able to ﬁnd several vulnerabilities in three home
security systems bought of the shelf in DYI stores in Denmark. The vulnerabil-
ities range from giving an attacker control of the system remotely, to making it
easier for an attacker that gains physical access to a phone that is tied to the
system to gain access to that system.
7.1
Further Work
There are multiple sections of the model and test of the model that could be
ﬂeshed out. As an example, hardware security, ﬁrmware, and reverse engineering
have only been given cursory attention. The software mapping has not been very
thorough either.
We can reﬁne our model for analysing vulnerabilities.
