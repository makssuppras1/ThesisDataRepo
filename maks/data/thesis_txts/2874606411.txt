Abstract
Large Language Models (LLMs) such as GPT have demonstrated strong capabilities in
invoking external tools based on user requests. However, they often face accuracy chal­
lenges, such as selecting the wrong tool, omitting necessary ones, or generating incom­
plete arguments, particularly in multi­step tasks or conversations that rely on prior context.
A key reason for these difficulties lies in a limitation of the current tool calling API, which
forces the model to output either a text response or a tool call in a given turn, but not both
simultaneously.
This restriction prevents the inclusion of explicit reasoning alongside tool calls, reducing
interpret­ability and hindering error diagnosis. As a result, developers gain little insight
into why a model chose a particular tool or produced specific arguments, complicating
debugging and limiting trust in LLM­powered agents.
This thesis addresses these challenges by introducing a reasoning­augmented structured
output. The proposed schema extends the tool calling response to include both a tool call
and an embedded, Chain­of­Thought (CoT) style reasoning trace in a single response.
Evaluation is conducted using the Xlam and Gorilla benchmark, both widely used re­
sources for assessing LLM­based tool invocation. Its diverse and standardized tasks
allow systematic testing across single­step, multi­step, and multi­turn scenarios.
Our evaluation shows that both our agent and the default tool calling agent achieve strong
accuracy across general­purpose models, with only a small margin between them. Com­
pared to expensive reasoning models, however, our approach delivers stronger perfor­
mance while enabling non­reasoning models such as GPT­4o to generate explicit reason­
ing during tool calling.
In summary, our approach overcomes the limitations of current tool calling, while also
delivering lower latency, reduced cost, and transparent reasoning.

Introduction
This chapter provides an overview of the research context and objectives. It begins by
introducing the background of Large Language Models (LLMs) and their role in tool calling,
followed by a discussion of the key challenges that motivate this work. The research
objectives and questions are then presented to clearly define the scope of the thesis.
Finally, the chapter concludes with an outline of the overall thesis structure.
1.1
Background
The Transformer architecture introduced by Vaswani et al. [1] played a central role in the
rise of large language models (LLMs). It allowed models to be trained more efficiently on
massive amounts of text and opened the way for scaling to much larger systems. Models
such as GPT­3 showed that increasing size and training data enable LLMs to perform well
across a wide variety of language tasks [2, 3, 4]. However, these models remain limited
when tasks require external information or the reliable execution of specific actions [5].
This limitation has motivated the shift from text­only LLMs to tool­augmented systems,
where the model is empowered to interact with external functions, databases, or APIs.
Instead of producing answers purely from its internal parameters, an LLM can now trigger
external computations or retrieve up­to­date information, a process commonly referred to
as tool calling or function calling [6]. Throughout this thesis, the term tool calling will be
used for consistency.
Tool calling enables LLM to move beyond plain text generation by recognizing when an
external action is required and producing a structured request to invoke the appropriate
tools [7].
A tool definition typically consists of three parts: a meaningful function name that clearly
reflects the operation to be performed, a description that outlines the intended purpose
and appropriate usage of the tool, and a JSON schema that specifies the parameters
required for execution.
In the case of the get_weather example in listing 1.1, the schema illustrates how different
types of parameters shape the behavior of the tool. The field location is marked as re­
quired, meaning that a valid tool call must always include this input in order for the function
to identify the target city. The field unit, by contrast, is defined as optional, it enriches the
call with additional detail when provided, but the tool will default to Celsius if it is omitted.
Finally, the schema is configured with additionalProperties: true, which permits arguments
not explicitly listed in the schema, such as a hypothetical field for forecast_days. This de­
sign choice highlights the trade­off between strict validation and flexibility, as it enables
future extensions without modifying the core schema.

By supplying tool specifications, developers can extend the model’s capabilities with data
retrieval or complex computations. Users can then interact with the system in natural
language, while the model generates structured tool calls and the backend executes the
corresponding functions seamlessly.
To illustrate this process in practice, Figure 1.1 summarizes the overall workflow, which
can be described in five main steps:
1. Tool Definition and User Message: The developer provides a tool definition, for ex­
ample get_weather(location), and the user asks “What is the weather in Paris?”.
2. Tool Call Generation: The model interprets the query and produces a tool call, e.g.,
get_weather("paris").
3. Function Execution: The backend executes the function with the given arguments
and returns the result, e.g., {"temperature": 14}.
4. Result Submission: The output is passed back to the model together with the prior
conversation context.
5. Final Response Generation: The model integrates the result and replies in natural
language, e.g., “It is currently 14°C in Paris.”.

Second, tool calling performance depends not only on correct tool selection but also on
accurate argument construction that adheres to predefined schemas. Prior work, such as
Gorilla [12], has shown that LLMs often produce malformed or incomplete arguments,
especially when the user input is ambiguous or requires transformation.
Such errors
can cause execution failures or lead to incorrect results, reducing the reliability of LLM­
powered agents.
Third, in multi­turn and multi­step scenarios, the absence of explicit reasoning traces limits
the model’s ability to iteratively refine its tool calls. Without an intermediate representation
of its decision process, the model must either re­interpret the original user request from
scratch or rely solely on past outputs, both of which increase the likelihood of compounding
errors.
Consequently, there is a clear need for a mechanism that allows LLMs to simultane­
ously produce (1) a structured, machine­readable tool call and (2) an embedded, human­
readable reasoning trace. This combined output would improve transparency, enable
better error diagnosis, and facilitate iterative correction in complex workflows.
1.3
Research Objective
This project aims to address the above challenges by integrating Chain­of­Thought (CoT)
reasoning with OpenAI’s structured output feature. The goal is to enforce schema adher­
ence while enhancing the model’s tool selection accuracy and argument completeness,
particularly when using lightweight models such as GPT­4o that lack native reasoning
capabilities.
The specific research questions addressed in this project are as follows:
• RQ1: How can we design a tool calling approach that returns both the tool call and
the supporting reasoning behind it?
• RQ2: What is the effect of embedding chain­of­thought reasoning on tool selection
and argument invocation?
• RQ3:How can we design and enforce a structured output schema that conforms to
a specific tool call format while embedding reasoning without breaking parsing?
• RQ4: Can structured output with embedded reasoning enable non­reasoning mod­
els (e.g., GPT­4o) to approach the tool call accuracy of reasoning­augmented mod­
els (e.g., GPT­o4)?

2
Related Work
This chapter reviews prior work most relevant to our approach. We organize the discus­
sion into three topics. First, we survey findings on reasoning in large language models,
including the emergence of step­by­step problem solving and methods that aggregate
multiple reasoning paths. Second, we cover prompt engineering techniques, such as
Chain­of­Thought (CoT) and ReAct.Third, we review structured output mechanisms that
constrain a model’s responses to predefined schema.
2.1
Large Language Models and Reasoning
Research on large language models (LLMs) has shown that scaling model size and train­
ing data can lead to reasoning abilities beyond simple text generation. Jason Wei [13]
demonstrated that LLMs can exhibit step­by­step problem­solving behaviors, which in­
spired prompting strategies such as CoT [14]. CoT encourages models to generate in­
termediate reasoning steps, improving performance on arithmetic, logical, and multi­step
tasks. Building on this idea, Xuezhi Wang [15] introduced self­consistency, where multiple
reasoning paths are sampled and aggregated for more reliable results.
Other work has extended reasoning further. Kojima [16] showed that even without fine­
tuning, zero­shot prompts can elicit reasoning traces. Ofir Press [17] explored composi­
tional reasoning, which underpins tasks such as tool use and retrieval­augmented gener­
ation.
Reasoning­augmented models, such as GPT­o4, directly integrate these ideas by produc­
ing rationales before their answers. While effective, these methods come with trade­offs.
They increase computation and latency, since generating reasoning traces requires more
tokens. They are also typically tied to specialized model variants, leaving lightweight
models such as GPT­4o without transparent reasoning support. This limitation motivates
research into methods that can provide reasoning in a structured, accessible way without
relying on expensive reasoning­augmented models.
2.2
Prompt Engineering
Prompt engineering refers to the practice of designing input instructions that guide large
LLM to perform specific tasks without additional fine­tuning [18]. Since LLMs are typically
trained on general­purpose data, prompt engineering allows developers to control the
model’s behavior by crafting effective input prompts. It has become a widely used strategy
for enabling zero­shot or few­shot learning in large models [19].
Early prompting strategies included zero­shot prompting, where the model is given only
a task description, and few­shot prompting, where a handful of examples are provided in
the prompt.
To improve reasoning performance on complex tasks, CoT prompting was introduced [14].
CoT guides the model to break down a problem into intermediate reasoning steps before
producing a final answer, significantly improving accuracy on arithmetic and logic­based
tasks.

natural language text or a tool call object. While developers have little control over the
structure of the output.

put [21], supported in newer models such as GPT­4o. This feature allows developers to
specify the response format when making API calls. Instead of generating plain text or tool
call object, the model is instructed to return a machine­readable JSON object that con­
forms to a developer­defined structure. This makes the output deterministic, parseable,
and easier to integrate into downstream applications.

Methodology
This chapter outlines how the study was conducted. We first describe the research design,
an iterative design–build–evaluate cycle used to develop and test the agent. We then
detail the data preparation pipeline, moving from a controlled synthetic suite to public
benchmarks. Finally, we summarize the technology stack and tooling that underpin the
experiments.
3.1
Research Design
Following the principles of Design Science Research [22], this work adopted an iterative
design–develop–evaluate methodology structured around six key stages: explicating the
problem, defining requirements, designing and developing the artefact, demonstrating the
artefact, evaluating the artefact, and refining the solution through iterative feedback.
Figure 3.1: Research design pipeline [22]
Explicate Problem
The initial challenge addressed in this research was the low ac­
curacy and limited interpretability of large language models (LLMs) in tool calling tasks.
Prior work had demonstrated that models often produce valid tool calls but fail to include
reasoning that justifies their choices. This gap motivated the need for a methodology
that integrates explicit reasoning with tool invocation to improve both transparency and
reliability.
Define Requirements
From this problem analysis, we derived key requirements for
the artefact: (1) the system must generate structured output defined by developer. (2)
structured output must include reasoning content that can be verified. (3) the methodol­
ogy must support diverse evaluation benchmarks and API formats. and (4) the iterative
process should reduce recurring errors such as incomplete arguments or inconsistent
reasoning.

Design and Develop Artefact
The core artefact developed was an orchestrator pipeline
capable of generating tool calls with embedded reasoning. Multiple prompt variants and
output schemas were designed to align with the orchestrator’s parsing logic and bench­
mark definitions. Implementation followed principles from Test­Driven Development (TDD)
and iterative prototyping, enabling rapid feedback and incremental improvements.
Demonstrate Artefact
The artefact was tested through controlled experiments across
multiple benchmark datasets, with test categories divided into single­turn and multi­turn.
Failure and exception cases were logged, and the responses were manually analyzed.
Evaluate Artefact
After each demonstration, a systematic evaluation was conducted.
Performance was assessed using quantitative metrics such as tool selection accuracy
and argument completeness, as well as qualitative analysis of reasoning consistency.
Special attention was given to recurring error patterns, for example, when the correct tool
was selected but arguments were incomplete, or when reasoning content conflicted with
tool calls.
3.2
Data preparation
Two complementary data preparation strategies were adopted to support the evaluation
of the proposed tool­calling framework. In the initial phase, a synthetic dataset was gen­
erated using the large language model to create diverse domains, tool definitions, and
queries. This approach offered full control over scenario design, allowing systematic test­
ing of specific reasoning challenges such as multi­step dependencies, missing arguments,
and partial tool coverage. However, its artificial nature meant that tool definitions lacked
grounding in real­world APIs, and the scenarios, while varied, did not always reflect au­
thentic user requests.
To address these limitations, the second phase employed established public benchmarks,
including the xLAM­Function­Calling dataset [23] and Gorilla­LLM­Leaderboard [24]. These
resources provided large­scale, domain­diverse, and API­grounded examples, enabling
evaluation under more realistic and externally validated conditions. Before integration into
our pipeline, both datasets underwent a series of refactoring and preprocessing steps to
ensure compatibility with our structured output schema and experimental setup.
3.3
Technology Stack and Tools
The implementation of the proposed framework was carried out entirely in Python (ver­
sion 3.11), chosen for its mature ecosystem of libraries supporting natural language pro­
cessing, data transformation, and evaluation workflows. The orchestration pipeline was

Conceptual Solution
This chapter introduces our conceptual solution for making LLM tool calling both accurate
and interpretable. Section 4.1 addresses RQ2 by presenting a reasoning­aware prompt
that guides tool selection and surfaces missing inputs. Section 4.2 addresses RQ3 by

4.1
Prompt Design
This section aims to answer RQ2: What is the effect of embedding chain­of­thought rea­
soning on tool selection and argument invocation?
The core of this solution lies in a prompt design that emphasizes reasoning as a prerequi­
site for action. In contrast to default prompting strategies that ask the model to immediately
output a tool call, we structure the prompt to guide the model through an explicit thought
process. The primary objective is to ensure that the LLM not only selects the correct
tool but also justifies its choice and clearly identifies any missing information. To achieve
this, we adopt a CoT approach, placing reasoning at the center of the prompt design. This
technique has proven especially effective in reducing hallucinated arguments and guiding
the model toward verifiable, structured decisions.
The prompt itself is composed of four conceptual components, each introduced as a result
of iterative refinement during experimentation. The full version of the prompt is in the
Appendix.

4.2
Structured Output Schema Design
This section aims to answer RQ3: How can we design and enforce a structured output
schema that conforms to a specific format while embedding reasoning without breaking
parsing?
One of the core contributions and innovations of this research is the integration of the exact
OpenAI tool call object structure into the structured output schema, requiring that every
element of the model’s response conform precisely to the OpenAI tool calling specification.
This strict adherence ensures that the returned output is immediately executable by the
target system without post processing or structural conversion. This schema formalizes
the structure and constraints of the model’s responses, enabling deterministic parsing,
enforcement of tool calling validity, and transparent reasoning.
The schema defines a top­level object with two required fields: reasoning and tool_calls.
The reasoning field is a string that captures a detailed natural language explanation of
the model’s decision process. This explanation must include justification for tool selec­
tion, identification of any missing information in the user query, and a final summary in
the form of a numbered list of tool calls (e.g., “tools should be called: 1­ ..., 2­ ..., etc.”).
The inclusion of these requirements ensures that the reasoning is both comprehensive
and actionable. Moreover, the field is expected to be between 50 and 500 words, which
encourages the model to produce substantial but bounded rationales.

derived from the reasoning process. Each item in the array represents a single tool call
and adheres to OpenAI’s tool calling format. Every tool call object is required to include
three fields: an id (a UUID that uniquely identifies the call), a type (which must always
be “function”), and a function object that specifies the name and arguments of the tool

5.1
Runtime Orchestration Workflow
The runtime orchestration of the system follows a sequential, dataset­driven workflow in
which each test case is processed from input to evaluation in a structured manner. The
process begins with the retrieval of a test case, including the user query, the available tool
definitions, and the corresponding ground truth.
In the generation stage, the system constructs a structured prompt that incorporates the
system role, reasoning guidelines, and the exact OpenAI tool calling schema, ensuring
that every generated tool invocation strictly matches the expected format. The language
model then receives this prompt together with the user query and produces a structured
response consisting of two elements: a reasoning field, providing a natural language ex­
planation of the decision making process, and a set of tool invocation objects conforming
to the schema.
Following generation, the workflow transitions to the evaluation stage. Here, the predicted
tool calls are compared against the ground truth reference, with two complementary as­
sessments. The first measures whether the correct tools were selected, and the second
verifies whether the provided arguments are both complete and accurate. These evalua­
tions produce quantitative metrics, such as confusion matrices, and also generate detailed
records of successes and errors for later analysis.
The final layer of the orchestration process is dedicated to logging. At this stage, the
system persistently records two complementary forms of information. First, the complete
message history of each interaction is stored, capturing user queries, model reasoning

5.2.1
Exploratory Dataset Prototyping with Synthetic Queries
The project initially began with the design of a synthetic dataset to support controlled ex­
perimentation with function calling under diverse conditions. We constructed a generation
pipeline that defined over 100 hypothetical industry domains, each paired with five syn­
thetic tools. For each domain, a set of queries was constructed to simulate common and
challenging tool invocation scenarios observed in practical use cases.
The motivation behind defining these scenarios was to systematically test the model’s
ability to handle typical user requests in tool­augmented environments. Users often sub­
mit composite queries that require multi­step reasoning, parallel tool calls, or handling
of incomplete or ambiguous input. Our dataset aimed to cover this diversity through the
following representative scenarios:
• Single­tool execution: The query can be resolved by invoking a single tool with a
complete set of arguments. This serves as a baseline scenario.
• Parallel multi­tool execution: The query requires invoking the same tool multiple
times in parallel with different arguments. This scenario evaluates the model’s ability
to recognize and decompose queries into multiple tool call objects.
• Nested or dependent multi­tool execution: The output of one tool is needed to
form the input for another. This tests the model’s ability to reason over sequential
dependencies.
• Missing argument handling: The query omits required fields, either implicitly or
explicitly, to assess how well the model can infer or request missing data.
• Tool unavailability: The query asks for an action without a corresponding tool de­
fined, simulating cases where fallback behavior or user feedback is expected.
• Partial tool coverage: The query involves multiple tasks, but tools are only avail­
able for some of them. This scenario tests the model’s ability to selectively generate
tool call only for solvable sub­tasks. .
Although the synthetic dataset supported controlled experimentation, it exhibited two crit­
ical limitations that rendered it unsuitable for robust end­to­end evaluation. First, the tool
definitions lacked any connection to actual backend functionality, meaning that once tool
call objects were generated, their execution could only be simulated through mock func­
tions. This prevented a realistic assessment of tool usage within a complete pipeline.
Second, while the dataset was designed to simulate real­world domains, the industry
definitions and associated tools were themselves generated by a language model, result­
ing in scenarios that were often disconnected from actual industrial practices. For these
reasons, the dataset served well as an exploratory sandbox, but proved inadequate for
rigorous evaluation of structured reasoning and tool alignment.

5.5
Tool Call Evaluator
Once the agents generate tool call responses, an evaluation mechanism is required to as­
sess their correctness against the annotated ground truth. To this end, we implemented a
tool call evaluator that relies on a confusion matrix formulation, allowing us to capture pre­
cision, recall, and F1 score as the primary accuracy metrics. The evaluator was designed
in parallel to the two agent pipelines introduced earlier: one for the reasoning­enabled
structured output agent, and one for the baseline agent without reasoning. This dual de­
sign ensures that both approaches are tested under identical conditions while preserving
their distinct output formats.
The evaluation process unfolds in two stages. First, the model’s predicted tool calls are re­
trieved, either as a structured JSON object containing both reasoning and tool calls in the
case of the So­agent, or as a direct list of tool invocations in the case of the normal agent.
In both cases, the predictions are parsed and aligned against the reference annotations.
Second, the evaluator compares predicted tool names with the ground truth, computing
the quantities that populate the confusion matrix. Beyond tool names, the evaluator also

examines the correctness of arguments supplied to each tool. Here, the same matching
logic is extended to parameter values: correctly predicted arguments are counted as true
positives, missing parameters as false negatives, and incorrect or extraneous values as
false positives. By aggregating these results, the evaluator computes precision, recall,
and F1 not only at the tool level but also at the argument level. Evaluating both levels is
crucial, since a model can select the correct tool yet fail to provide valid arguments.
Throughout the evaluation, all case results are recorded in structured logs. Each test
query produces a JSONL entry that captures the ground­truth tool calls, the predicted
outputs, and the evaluation outcome, including the reasoning trace when available. Sep­
arate log files are maintained for the reasoning agent and the normal agent, ensuring that
downstream analysis can be carried out independently or compared side by side. Once
evaluation across the dataset is complete, the logs are summarized to yield aggregate
statistics, providing an overall view of model performance.

reasoning traces and tool calls, many realistic scenarios unfold over multiple user turns.
To address this, we implemented a multi­turn pipeline that maintains conversational mem­
ory and allows the agent to reason across sequential interactions. This design enables the
evaluation of more complex settings where tool calls and final answers are not determined
in isolation but depend on the accumulated history of the session.
The pipeline builds on the same principles of structured output and schema enforcement
introduced earlier, but extends them with a session­level memory system. To simulate
this interaction in a controlled manner, we define a list of user queries that are processed
sequentially. Each query in the list can represent either a single­step request, resolvable
through one tool call, or a multi­step request requiring several invocations. By iterating
through this list, the pipeline effectively reproduces a multi­turn dialogue between a user
and the LLM, in which the model must not only generate reasoning traces and tool calls
but also reuse and integrate outputs from prior turns.
The multi­turn interaction unfolds in three interconnected stages. First, the current user
query is combined with the accumulated conversation history, which is flattened into con­
text and passed to the model together with a reasoning­oriented system prompt. This
prompt instructs the model to either continue tool usage or, if sufficient evidence already
exists in the history, to terminate with a call to finish_session(). Second, the model’s
outputs are parsed and normalized into a uniform schema. Tool calls are executed lo­
cally, and their results are appended back into the log under a dedicated tool role entry,
ensuring every invocation is explicitly recorded with arguments and return values. Third,
if the model signals the end of the session, the pipeline triggers a routine that produces
a natural language summary integrating reasoning traces with accumulated tool results.
This response is appended to the log and returned as the final answer.
The history log lies at the center of this process. Its structure is deliberately strict, forming
a repeating block for each user query: a user message containing the raw query, followed
by an assistant message issuing tool call specifications, then a tool message recording
the execution results, and finally a second assistant message providing the natural lan­
guage reply. This fixed sequence governs every conversational cycle because multiple
queries are processed in sequence, the log grows turn by turn. This keeps the order of
events clear, so the agent can look back at earlier tool calls, avoid repeating them, and
know when the session should end. Crucially, the invocation of finish_session() serves
as an explicit control flag for termination, signaling that the dialogue has converged on a
conclusive answer.
Importantly, this design mirrors the message flow enforced by the OpenAI toolcalling API,
where user input, assistant tool specifications, tool responses, and final replies follow a
specific order.
Finally, the pipeline integrates mechanisms to sanitize and normalize tool calls before
they are logged or replayed. Arguments are consistently stringified into JSON to ensure
schema validity, and missing or malformed inputs are corrected through fallback rules.
This guarantees that the conversation history remains both machine­readable and se­
mantically coherent, even in the presence of imperfect model outputs. Since the memory
persists across turns, the system supports cumulative reasoning: the agent can refer­
ence earlier results, build on prior decisions, and dynamically determine when enough
information has been collected to end the session.

producing final responses conditioned on the complete interaction history.

7
Conclusion and Future Work
7.1
Conclusion
This thesis investigated whether structured output with embedded reasoning can improve
the accuracy, reliability, and interpretability of tool calling with large language models. Our
work was guided by four research questions (RQ1–RQ4) introduced in Chapter 1. In this
section, we revisit each research question and summarize the main findings, followed by
a general conclusion.
RQ1: How can we design a tool calling approach that returns both the tool
call and the supporting reasoning behind it?
Our results show that the key to answering this question is to use a structured output
format that includes both the reasoning steps and the tool call. This format makes sure that
the model explains its thinking and gives the tool call together. To make this work, we use
chain­of­thought (CoT) prompting, which helps the model first show its reasoning before
producing the tool call. With this schema and CoT support, we reach a high level of format
accuracy, which makes downstream parsing and evaluation more reliable. Compared
with the baseline agent, which only outputs raw function calls without explanation, the
SO­Agent provides greater transparency, better control, and more consistent accuracy
across tasks.
RQ2: What is the effect of embedding chain­of­thought reasoning on tool
selection and argument invocation?
Embedding reasoning traces substantially affected both tool selection and argument ac­
curacy. At the tool level, the SO­Agent reduced false negatives in complex queries that
required multiple tool calls, particularly on reasoning­optimized models such as o3­mini
and o4­mini. This improvement was achieved because the reasoning trace guided the
model to generate full tool chains instead of prematurely terminating after a single call,
which was a frequent behavior of the baseline agent.
At the argument level, the SO­Agent generally increased precision, producing more schema­
consistent and interpretable arguments. However, this came at the cost of recall in some
cases. Stricter reasoning occasionally led to more conservative predictions or absten­
tions, resulting in higher numbers of unparseable or incomplete outputs. Thus, the chain­
of­thought mechanism improved reliability but also introduced new forms of errors that
need to be managed.
RQ3: How can we design and enforce a structured output schema that
conforms to a specific tool call format while embedding reasoning without
breaking parsing?
For RQ3, we find that strict schema enforcement is the critical factor that allows reasoning
and tool calls to be reliably integrated into one format. By adopting OpenAI’s native tool
call object and enforcing constraints such as strict: true and additionalProperties:
false, the schema ensures that outputs are both machine executable and transparent.
This design enabled deterministic parsing and reliable evaluation while keeping reason­
ing tightly aligned with tool invocation. However, the stricter enforcement also reduces
flexibility, requiring models to adapt to tighter format boundaries.

RQ4:Can structured output with embedded reasoning enable
non­reasoning models (e.g., GPT­4o) to approach the tool call accuracy of
reasoning­augmented models (e.g., GPT­o4)?
Our experiments show that structured output with embedded reasoning enables non­
reasoning models such as GPT­4o to achieve tool call accuracy that is close to, and in
some cases even better than, reasoning­optimized models like o4­mini and o3­mini when
used with the baseline agent. The SO­Agent delivers strong and stable performance on
general­purpose models, narrowing the gap without requiring specialized reasoning ar­
chitectures. This means that comparable accuracy can be achieved while saving both
computation time and financial cost.
General Conclusion
Overall, this thesis shows that structured output with embedded reasoning is a promising
framework for improving tool calling performance. The SO­Agent consistently delivered
more reliable and interpretable outputs across diverse model families, especially in sce­
narios where the baseline struggled with incomplete or hallucinated tool calls. While the
approach introduced additional complexity and higher rates of unparseable responses on
reasoning models, these trade­offs were outweighed by the gains in robustness, preci­
sion, and interpretability.
The central contribution of this work is the demonstration that structured reasoning not
only enhances tool­call accuracy but also provides a clearer window into why failures
occur. This transparency enables more effective debugging and downstream correction
than baseline tool calling.
7.2
Future Work
While this thesis provides evidence that structured output with embedded reasoning can
improve the robustness and interpretability of tool calling, several limitations remain. These
open avenues for future research:
Real­world multi­turn evaluation
The current evaluation relied on synthetic and pre­designed multi­turn scenarios derived
from benchmark datasets. Although these cases capture many challenging aspects of tool
invocation, they do not fully reflect the dynamics of real­world usage, where conversations
unfold in unpredictable ways and context may shift across turns. A natural next step is
to embed the SO­Agent into industrial workflows and log its performance in live multi­turn
interactions.
Intelligent memory management
Our implementation of multi­turn reasoning employed a simple memory strategy: all prior
turns were concatenated into the conversation history until a finish_session() call sig­
naled termination. While effective for controlled experiments, this strategy is not directly
transferable to real­world systems, where no explicit termination signal exists. A promis­
ing direction is to design an adaptive memory module that can automatically distinguish
between relevant and irrelevant prior turns.
Expanding to new function­calling APIs
This thesis focused primarily on the Chat Completion API for structured function call­
ing. However, the landscape of model APIs is evolving rapidly. The recently released
Response API, for instance, includes built­in reasoning mechanisms and different affor­
dances for tool integration. A systematic comparison across APIs­examining not only raw
accuracy but also reliability, latency, and interpretability, would deepen our understanding
of how structured reasoning interacts with underlying model infrastructure.
