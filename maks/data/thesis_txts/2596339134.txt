Abstract
Artificial neural networks, a subcategory of artificial intelligence, are experiencing a
rapid development. An ever-increasing number of problems, such as cognitive task
where speech recognition and object detection are common examples, can be solved
using artificial neural networks. The complexity of the networks is proportional to
the growing complexity of a problem that is being solved. As a consequence, both
execution time and power consumption are increased. This renders the use of offline
artificial intelligence in highly power-constrained devices, such as hearing aids, un-
feasible. The desire to reduce the power consumption of Artificial neural networks
has, in part, given rise the development of 3rd generation artificial neural networks,
also known as spiking neural networks. This Master’s thesis presents the design and
implementation of a neuromorphic accelerator capable of running inference in spik-
ing neural networks. The design focus is on obtaining a low-power implementation
to assess whether spiking neural networks could enable offline artificial intelligence
capabilities in highly power-constrained devices. As a proof of concept, the presented
neuromorphic accelerator has successfully been implemented on the Genesys 2 FPGA
board, running a network capable of classifying handwritten digits from the MNIST
database.

Terminology List
1. Spiking Neural Network Terminology
• Spike - Represents a binary event. Neurons of a spiking neural network
communicate with each other only using spikes. Furthermore, including
timing of the spike, a spike carries more information than the binary value
it is represented by.
• Neuron - Describes a core component of all spiking neural networks. Spe-
cific neuron models vary greatly. Although specific neuron models vary
greatly, common to all models is that they model a membrane potential.
The membrane potential is a value that represents how excited a neuron
is. When a neurons membrane potential exceeds some threshold, it will
generate a spike.
• Synapse - Refers to a connection between two neurons in a spiking neural
network. Specific synapse models vary, but all models can propagate a
spike generated by a neuron in one end of the synapse to the neuron in the
other end.
• Time-step - Artificial spiking neural networks are evaluated in discrete
time-steps. There is a set period between time-steps, because spiking neu-
ral networks can evaluate non-finite input spike trains. It must, therefore,
be possible to fully evaluate the state of a network within a time-step
period if no inputs should be ignored.
• Spike Train - Refers to a series of spikes (most often the input of a spiking
neural network). Inputs to the network, such as images, sensory data etc.
are encoded into spikes before it can be processed by the network. A spike
encoded input is referred to as an (input) spike train.

CHAPTER 1
Introduction
Second generation artificial neural networks (ANNs) is the computing structure that
drives virtually all machine learning applications in use today.
This introduction
chapter starts with a brief high-level description of such ANNs and the benefits they
have. Following, the potential disadvantages of these ANNs are highlighted. This
will lead to a motivation for employing third generation ANN, called spiking neu-
ral networks (SNNs) as an alternative. SNNs is an effort to imitate the dynamics
of biological brains more closely. Biological neurons communicate through electric
impulses, which is referred to as spikes.

1.1.1
Benefits of Second Generation ANNs
Research on how to utilize ANNs has grown steadily for the last 50 years. Machine
learning might currently be the most active research field within computer science
[Sta] [Mic].
As a result, well-defined strategies for network model structures and
training ANNs exist for most applications. There are numerous tools for ANN devel-
opment, such as Google’s TensorFlow [Goo] and Facebook’s PyTorch [Fac], making
the development very accessible.
The accuracy of cognitive tasks, such as recognition, is unmatched by other strategies.
Examples are MNIST classification with accuracy of 99.84% [BKD20] and fast object
detection that with great accuracy can localize and classify objects in an image at
high frame rates [RF18], suited for aiding autonomous driving. High accuracy and
great accessibility make the second generation ANNs the computing structure for the
vast majority of artificial intelligence applications.
1.1.2
Disadvantages of Second Generation ANNs
The size of ANN models tends to grow with the complexity of the problem the network
tries to solve. One of the best performing detection networks, YOLOv3 [RF18] that
localize and classify 80 objects from persons to hairdryers, consists of 83 layers and has
240MB of 32-bit weights and biases. Running inference on a single input, YOLOv3
performs 141·109 floating-point operations to produce a result. As it can be observed
in 1.1, the vast majority of operations in running network inference is multiplications
and additions. This contributes to three major issues.
1. Execution time is an issue for applications where the networks are part of a
real-time system, such as autonomous driving. Multiplications is a complicated
operation that usually takes multiple processor clock cycles even for efficient
hardware algorithms such as the Tomasulo algorithm [Tom67]. In these appli-
cations, network evaluation must be parallelized sufficiently to achieve real-time
execution. This is typically obtained by running the networks on GPUs or spe-
cialized accelerators [Nur+17].
2. The large quantity of parameters needed can be an issue. Some devices are
heavily memory constrained, leaving these large networks unfeasible to imple-
ment.
3. The last major issue is power-consumption, which for most applications is the
largest limiting factor. This is an issue for power-constrained devices like em-
bedded battery-driven systems, such as mobile phones and hearing aids. Some

1.2
Introduction of Thesis Subject
Many have argued that the key to developing low power ANNs should be found in how
biological brains process data [Ako+15] [UH20] [Tan+15]. Biological brains are the
most advanced and power-efficient computers we know of. The human brain consists
of 86 billion neurons that are collectively and, all in parallel, are processing sensory
data, while consuming only 20W of power [SS15]. This has given rise to research in
3rd generation ANNs, better known as Spiking Neural Networks (SNN). SNNs imitate
a biological brain’s compute structure more closely than second generation ANNs.
SNNs have more complex neuron models, wherein neurons propagate information
through the network using binary events, called spikes, instead of real-valued data.
This means that information can be encoded in the timing of spikes rather than
real values being propagated through a network. It is this feature that makes SNNs
promising in terms of low power evaluation of ANNs. The two major reasons than
power can be reduced using SNN are:
1. Arithmetic operations are lighter for SNNs, than what are required to second
generation ANNs (this is explained in Section 2.2 and 3.1.1). It is possible to
run inference of SNNs without using a multiplication unit.
2. SNN neural network should ideally, experience low spiking activity. The vast
majority of computing needed when running inference of SNNs are performed
evaluating spikes. This can potentially make for less switching for the circuit
running the network
In this thesis, a neuromorphic accelerator design capable of running SNNs is pre-
sented. The focus of the thesis is to design the accelerator for low-power, but most
importantly the accelerator should correctly run a SNN with minimal loss in accuracy.
These systems are generally referred to as neuromorphic processors but they are es-
sentially hardware accelerators as the only function that they carry out are running
SNN inference. A dedicated accelerator is more suitable for low power SNN inference.
A dedicated accelerator can be tailored to take advantage of the SNN’s structure and
sparse activity. By building a compute structure that closely resembles the structure

2 Spiking Neural Network Theory
2.1
An Overview of SNN
SNN model architectures vary vary immensely in the literature. This large variety is
present for two main reasons:
1. There are two main lanes of research that are working with SNN. One being
neuroscience research. Research in this lane wants to gain insights into how
the biological brain operates.
In this research detailed and complex models
are often found, where the objective is to mirror the complex dynamics of a
biological neuron and the widely varying connectivity between neurons of the
different part of a human brain. The second lane of research is using SNN for
machine-learning (ML). In ML research, the neuron models are often modeled as
simple as possible to keep computational requirements low, as this often makes
for higher performance and lower power consumption. In this field of research
the network connectivity is usually inspired by or identical to the connectivity
we see in second generation neural networks, e.g. FCNNs or RNNs.
2. The second main reason for the high variety in SNN models is that SNN research
has been explored less than second generation neural networks. This means that
the methods and architectures have not been standardized as heavily, partly as
there are more parameters and models to experiment with and partly because
superior architectures have not been determined for different applications. This
could be seen in CNNs that are superior for image classification.

2.2 Neuron Model
9
A SNN contain of three major components: i) The encoding model, ii) neuron model
and iii) synapses model. These models are explained in more detail in the coming
sections. SNN processes information in the form of binary events called spikes. There-
fore, the input has to be converted to a spike train used as SNN input over time. This
makes SNN especially suitable for real-time or continues input applications, such as
sound processing. Biological brains are analog computers that process continuous
input. This requires infinite temporal resolution, which is not feasible with a digital
circuits. Therefore, we decide on a time resolution. This time resolution is most
often set to 1 ms per time-step. A 1ms time resolution is reasonable. 1 ms is for
most circuits enough time to evaluate the state of the SNN. 1 ms, Furthermore, allow
SNNs to match spiking activity a biological brain, where a neuron typically spike 5-50
times per second [The19]. This means that states of all components/models of the
SNN are updated every 1 ms.
2.2
Neuron Model
Common to all SNN neurons is that they model what is known as a neurons mem-
brane potential. The membrane potential is a value representing how excited this
particular neuron is. When a neuron receives a spike, the spike is consumed by the
neuron and the membrane potential is increased. This process is callled ”integrating
input current”. When a neurons membrane potential exceeds a potential threshold,
unique for this particular neuron, the neuron fires. When a neuron fires, it generates
a spike on all its outgoing connections/synapses. After a spike generations, the neu-
ron’s membrane potential is reset to a value with this neuron. These are the minimum
dynamics of a spiking neuron.
Most used synaptic models are more complex and are extended by modelling a poten-
tial decay. Potential decay refers to a neurons membrane potential decreasing over
time. When potential decay is modelled, a generated spike carries more information.
With potential decay, the occurrence of a spike indicates that the neuron that gener-
ated the spike has experienced high activity on its input synapses. If potential decay
is not part of the neuron model, a generated spike will only indicate that a single
spike occurred at the input of a source neuron, as its membrane potential could have
been just below the threshold for an infinite time before generating the spike.
Another commonly included dynamic is the refractory period. A refractory period
is a duration of time after generating a spike where neuron cannot ”integrate input
current”. After a neuron has fired, its membrane potential will remain at its reset
value for the duration of the refractory period. Refractory period limits the maximum
spike activity that a neuron can produce. This can be used as a way of lowering a
neuron’s sensitivity to input stimuli. Reducing spike activity is good for neuromor-

The three most commonly used neuron models are the Hodgkin-Huxley model, the
Izhikevich model and the Leaky Integrate and Fire (LIF) model. These three models
have a decreasing level of complexity, respectively. The Hodgkin-Huxley model re-
sembles a biological neuron very closely. Naturally, this model is heavily used in the
neuroscience lane of SNN research. The LIF model is the neuron model supported by
the neuromorphic accelerator presented in this thesis. The LIF neuron is modelled
as a resistor in parallel with a capacitor. The membrane potential is the voltage over
the resistor and capacitor, and spikes are modelled as current pulses [Kas19]. The
behaviour of a LIF neuron seen in Figure 2.2. The LIF models used in the presented
system is further extended with a bias. A bias is a unique value for every neuron that
is added to its membrane potential in every time-step.
In a given time-step a LIF neuron’s state is updated through the following operations.
A received spike will increase the neurons membrane potential based on the strength
of the spike. This requires an addition for every received spike. The strength of a
spike is determined by the synapses model, covered in the a upcoming section. In
every time-step, the membrane potential must be decreased based on the potential
decay, which in some interpretations of the model is a percentage of the neuron’s cur-
rent membrane potential. This requires one multiplication and one subtraction. In
simpler interpretations, the membrane is decreased by a fixed amount which reduces
the operation to only a subtraction. When the membrane potential has been updated

2.3
Spike Encoding
Spike encoding refers to the process of converting real-value or continuous input data
into a spike train that conveys the same information.
In other words, some pre-
processing is needed before input can be injected into the network in the form of
spikes. This section presents the two most used methods of spike encoding: rate-
based and time-based encoding. The showcase network in this work is trained with
Poisson encoding. Poisson encoding is an extended version of rate based encoding.
Input to a second generation ANN is most often a collection of values. If the input
is an image, these values could be the pixel colour values. In this case, there would
be an input channel for each pixel. In SNNs, a spike train consisting of the encoded
input channel is injected to the network for each pixel. With rate-based encoding,
these values are converted to a spike train with rates of spikes proportional to the
pixel colour values. The encoding is illustrated in Figure 2.3. Poisson encoding is
very similar to rate-based encoding. Instead of spikes having the same period between
each other (pure rate-based), the period between spikes of a spike train is a Poisson
distribution. This means that the average period of spikes in a Poisson encoded spike
train is the same as for rate-based encoding. This stochastic noise makes for better
generalization when training networks.

2.4
Synaptic Models
The simplest model of a synapse is merely a connection from one neuron to an-
other, capable of carrying a spike. The vast majority of SNNs for machine learning
model synaptic weights. A synaptic weight determines the strength of spike travelling
across this synapses. In most implementations, the weight is directly the membrane
potential increase of the receiving neuron. The higher the weight, the stronger the
connection. Synaptic weights are the only synapses parameter that the presented
neuromorphic accelerator support.
Another commonly modelled synapses parameter is a synaptic delay. Here delivery of
a spike is delayed by the duration specified by the synaptic delay that can be tuned
during training. The reason that synaptic delay can contribute to a better performing
network is that a neuron might have a set of in-going connections/synapses from
other neurons that fires sequentially. Because of this time interval between firing, the
receiving neuron might not get to spike as potential decay prevents it. With synaptic
delays, these spikes can be received at the same time-step causing the receiving neuron
to fire, which might be beneficial for performance.
2.5
Network Evaluation Order
The state of a SNN is evaluated every time-step. SNN are typically evaluated such
that spike propagates one layer per time-step, rather than propagating the full net-
work in a single time-step.
Using the network in Figure 2.1 as an example.
In
time-step n spikes of the spike train are used to evaluate the state of neurons in the
first layer. The first layer is special as these neurons do not model a membrane po-
tential. Input neurons have the single purpose of injecting a frame of the spike trains
from the input onto the network. This means that the input layer either generates
a spike or not based on the spike trains. In time-step n + 1, neurons in the second
layer are updated based on injected spike trains frame from time-step n (e.g. updating
membrane potentials, possible generate spikes and going in refractory mode). Spikes
generated by the second layer in time-step n + 1 are used for neuron evaluation of the
third layer in time-step n + 2. Therefore, it takes 3 time-steps for a spike presented
at the input to propagate to the output, for the example SNN in figure 2.1.

The Design Objective
In the exploratory phase of this project, it became apparent that SNN research has
investigated an overwhelmingly wide range of models and structures, as stated in

interest of neuroscientist as well as computer scientist. Neuroscience is interested in

machine learning engineer rather than a digital system engineer.
The intended target device group for the presented neuromorphic accelerator com-

relevance and functionality.

High-level Overview of the Accelerator
This section introduces the design from a high abstraction perspective. At this ab-
straction level, the design is significantly inspired by well-established neuromorphic

3.2.1
Top View - Neuromorphic Multi-core Systems
Most neuromorphic accelerators share the same overall structure. They consist of
several neuromorphic cores that communicate spikes to each other through some
communication fabric. This is illustrated in Figure 3.1. Each core is responsible for
evaluating a set of neurons from the neural network that the accelerator is running.
For the evaluation of a neuron, a lot of parameters must be considered: membrane po-
tential, synaptic weights, neuron bias, potential decay, refractory periods, etc. These
parameters must be written to the core local memories when configuring neuromor-
phic accelerators. As we will see, the cores are either reading or writing a parameter
to memory nearly every clock cycle. For this reason, neuromorphic accelerators can-
not be designed following the classic Von Neumann architecture [Bou+19], which has
been the backbone of the majority of processors designs since the birth of digital
processors. Not complying with the Von Neumann architecture, makes neuromorphic
accelerators very interesting and special from a computer architecture perspective.
Even with only a few neuromorphic cores, a shared main memory would become

on-chip memories that hold all parameters needed for evaluation of a neuron, which

3.2.2
Neuromorphic Crossbar
Many exciting neuromorphic accelerators share the same high abstraction view of the
neuromorphic core design, namely the neuromorphic crossbar [Ako+15] [Bou+19]
[Son+20] [Dav+18]. The neuromorphic crossbar abstraction view (illustrated in Fig-
ure 3.2) is adopted for the neuromorphic core design developed for this thesis. The
neuromorphic crossbar is an abstract compute model, suitable for evaluating a fully
connected layer of a SNN. It is a simple model that reflects the types of components,
the overall structure, and data-flow of the specific neuromorphic core design presented.
The crossbar consists of a set of neuron evaluation units (triangles in Figure 3.2), a
set of axons (rectangles in the figure), a network interface, and a structure (the cross-
bar) that connects axons to neuron evaluation units. The neuron evaluation units
are represents the neurons of the layer mapped to a corresponding core (enclosed by
dashed lines in the network model of Figure 3.2). Each axon delivers spikes generated
by one neuron in the previous layer of the network. Neurons of the previous layer are
mapped to another core. The intersections of the crossbar represents synapses/con-
nections, and therefore the weight, of the network model. When a spike generated in
the previous layer are received by this core through the network interface. A spike,
received from the network interface, is put in the specific axon that hold spikes gen-
erated by a specific neuron in the previous layer. Thus, an axon can be seen as an
incoming spike buffer. Furthermore, it also means that the core must have at least
as many axons as there are neurons in the previous layer. Through the crossbar, one
axon is connected to all neuron evaluation units. The core only need to receive a
generated spike, rather than the indications of all synapeses that the spike will travel
across. Consequently, all neurons in a core must share inputs from the same source
neurons, which is a commonality of fully connected networks (shown in Figure 3.2
and 2.1). The individual synapses are embedded in the crossbar by the intersections.

3.3
Power Reduction Approach
When designing systems for low power, the objective is often to reduce switching
activity to the bare minimum. This strategy comes to fruition by utilizing a sparse
activity in SNN. Sparse activity means that a neuron does not spike in every time-
step. This way, some of the information that is being processed by the network lies
in the timing or rate of spikes being generated. In the worst-case scenario all in-going
connections to a neuron carries a spike, in a given time-step. The hardware should be
able to support this, to accurately run a trained SNN on a neuromorphic accelerator.
To the benefit of the design, it is rarely the case that all in-going connection are
carrying a spike in the same time-step. Measurements on the showcase network used
in this thesis show that, on average, an input spike train delivers a spike in 1.75% of
all time-steps, while the output neuron generates a spike in only 0.08% of all time-
steps. This signifies that if a neuromorphic core can process all received spikes as
fast as possible, the neuron evaluation units will idle for around 99% of the time.
The strategy for achieving low power consumption is to process all incoming spikes
as fast as possible whereafter the neuron evaluation units are clock gated to ensure
no switching while there is no spikes to process. Thus, only leakage current should
consume power in these units while idling.

required by the network. For an implementation with a lot more cores, a NoC might
be implemented instead of a bus to support the spiking traffic. Additionally, the
bus interface is responsible for converting an AER package from the communication
fabric to an axon ID.
The valid signal of the valid-ready interface that the core uses to transmit spikes
directly interacts with the bus arbitration protocol. The valid signal serves as the
request signal for the arbiter. The grant signals from the arbiter fulfill multiple pur-
poses in this context: they (i) are used as the ready signal to the core, (ii) ensure
request is only held high until the core is served by the arbiter, and (iii) select signals
for the multiplexer to let data from the valid-ready protocol into the tx channel of
the bus. The valid-ready interface for spikes to the core is also facilitated by the bus
interface component. When a spike subscribed to by the core is present on the bus,
a valid signal is generated by the look up memory along with MSBs of the axon ID
the spike are encoded into (axon IDs are explained shortly). The ready signal for
this valid-ready interface is not connected at the core as the core is always ready to
consume an incoming spike.

bus, such that only spikes that the core of the interface subscribes to are delivered
to the core. This filtering is done using the LUT memory illustrated in 3.6. Using
the AER messages from the bus, a look up is performed. As only fully connected
networks are supported and a core must hold neurons from the same layer of the
evaluated SNN. A given core will subscribe to all or none of the spikes produced by
neurons in another core. This, in turn, means that in order to determine if an AER
message should be consumed from the bus, only the core ID of the AER message needs

3.7.1
The Axon System
The Axon system has the function of storing spikes received in one time-step such
that they can be used for the evaluation of neurons in the next time-step. The Axon
system plays a critical role in the strategy for achieving low power consumption. Re-
call that the strategy is to clock-gate neuron logic as early as possible in a given
time-step. To implement that the core must be able to consider all incoming spikes
==PAGE:44==
30
3 Design
as fast as possible. This is the idea behind the axon systems design.
From the neuron evaluation equation 3.1, it is known that a spike is used only to
signify whether a synaptic weight should be added to the membrane potential of a
neuron or not. This is why a conversion to axon ID is performed by the bus interface.
As explained in Section 3.7.2 the evaluation memory holding weights for neuron eval-
uation, is organized such that an axon ID can be used as the LSBs of the addresses for
the synaptic weights of the synapses the corresponding received spike travels across.
This follows the neuromorphic crossbar design methodology.
The Axon system schematic can be seen in Figure 3.8. Spikes encoded to an axon
ID are received from the Core-Bus-Interface along with a valid signal. Axon IDs are
stored in a memory in one time-step and read by the Neurons block during neuron
evaluation in the next time-step. For this reason, the Axon system has two memories,
one for writing and one for reading. These two memories change their function in
every time-step. As mentioned in Section 3.3, it is extremely rare that all neurons
spike in a given time-step. It means we can assume that all spikes have been received
before the time-step is over. This assumption led to the dual memory approach.
The Axon system should ensure that the core can evaluate the incoming spikes as
fast as possible. This is achieved by counting incoming spikes in a time-step (this
counter is referred to as the spike counter from here on). The circuit responsible for
spike counting is in the lower left of Figure 3.8. When a time-step starts, the spike
cnt register is reset, which is indicated by the input inOut changing value. The signal
inOut will alternate between 1 and 0 every time-step. When a change of inOut the
selector signal for the multiplexer driven by the XOR gate is high for one clock cycle.
When this selector is high the spike counter register is reset to zero. When the Axon
system receives an axon ID, the spike cnt value is used as an address for storing the
axon ID. The valid signal is used as the other selector for the multiplexer causing
the spike counter register increment in the following clock cycle. The spike counter is
connected to an output of the Axon system. This output is sampled by the Neurons
block in the first clock cycle of the next time-step. This is a critical design choice for
the fast execution strategy, as the neuron Control circuit thereby receives information
about how many spikes to process within a given time-step. From the Neurons block’s
point of view, the Axon system can be treated as a read only memory (ROM). This
is elaborated on in Section 3.7.2. The ROM interface between the Neuron block’s
Control unit and the Axon system can be inspected in Figure 3.9 The input inOut
ensures that one memory is used for writing the axon IDs while the other is used for
reading them. The interface which the Axon system and the Neuron block’s Control
unit, exchange control information over is illustrated in Figure 3.10.

communication fabric.
When the NEU has generated a spike, the Transmission
system is informed. The Transmission system should assemble an AER message con-
taining the global address of the neuron that spiked. The Transmission system should
send a request to the communication fabric to transmit the AER package.
As illustrated in Figure 3.7 and explained in Section 3.7.2, the Neurons block contains
multiple NEUs. The NEUs work in parallel, and could, in theory, all generate a spike
in the same cycle. The communication fabric only allows for a single transmission
per cycle.
To serve all NEUs, some spike buffering is needed.
The Transmission
system can buffer one spike per NEU (P). In Figure 3.13, the Transmission system
schematic is illustrated. The subset of the transmission circuit enclosed by a dashed
line contains the P spike buffers. The buffers share the input n, which is directly
connected to the counter ni of the Control circuit. Register n will, therefore, hold
the MSB of a buffered spike’s local neuron ID. The buffers have their own dedicated
spike signal. The spike’s bus contains a wire for every NEU in the core. When a
NEU generates a spike, the Transmission system will receive a pulse on this dedicated
spike’s wire. This interface is illustrated in Figure 3.14. The pulse will update the
spikei buffer register to hold 1, if this buffer is not currently occupied by a spike that
has not yet been transmitted. As there can be multiple buffered spikes waiting for
transmission, the system needs a fair way of serving spikes held in the spike buffers.
This is facilitated by the Encoder Mask Reset unit. This unit takes the spike buffer
values as input and generates four signals:
• A priority encoding, which is used for assembling the AER message.

a network. For the implementation to be used in an application, there needs to be a
way for an external system to deliver the network spike trains as input, and a way of
replying to this external system with output spikes. In this section, the design of the
three components facilitating this communication with an external system are pre-
sented. These are: the Off-chip communication, Input and Output core components

3.8.1
Input Core
The function of the Input core is injecting the spike train input to the network. To
keep the system uniform and simple, Input cores should interface with the communi-
cation fabric identically to Neuron cores. This means that a single Input core can only
supply as many input channels as the number of neurons that Neuron cores evaluate
(N). This ensures conformity with the global address organization of neurons. For
this reason, the system needs two Input cores as the input size is 22*22 = 484 for the
showcase network. The Input core interfaces communication fabric using the same
bus interface component as Neuron cores (introduced in Section 3.6.2). For this rea-
son, it only makes sense to use the spike transmission system component to facilitate
the transmission of spike trains onto the network. As it can be seen in Figure 3.15,
this is exactly the case for the Input core design developed for this implementation.
The rest of the design is non-generic and has been designed for the showcase network.
The Input core is build for the network being evaluated. The Input core design can be
seen in Figure 3.15. The network has been trained to classify handwritten digits us-
ing the MNIST dataset [LCB]. The MNIST image has been scaled down from 28x28
to 22x22 pixels. The images were scaled down to fit the show case implementation
which is explained in section 4.2. The scaled image has been converted to spike trains
using rate-based encoding, at spike train length/duration of 500 time-steps, equiva-
lent to 0.5s. With rate-based encoding, a pixel value is converted to a rate of spikes
that should be injected into the network. To decrease the off-chip communication
throughput requirements, a spiking period for each pixel is calculated off-chip and
transmitted in the Input core through the Off-chip communication module, which is
introduced shortly. The alternative, and the most generic implementation is to have
an external system to supply spike trains directly. This, however, requires 500 times
higher off-chip bandwidth when a spike for every pixel is received in every time-step.
This cannot be handled by the simple UART protocol that was chosen. Furthermore,
timing is involved, as spikes must be delivered in the correct time-step. The Input
core uses these periods to inject spikes onto the network at the appropriate time-step.
The periods have the range [1:500]. If a spike train has no spikes at all, it will have
the value 0.

to store spike train periods for the input to be evaluated after the current input has
been evaluated. The other memory is read by an FSMD that injects spike trains onto
the network for the input that is currently being evaluated. As is the case for the
axon system, these two memories alternate between their roles for every input being

3.8.2
Output Core
The function of the Output core is to forward spikes generated by output neurons
to the external system utilizing the accelerator. These output spikes are forwarded
==PAGE:59==
3.9 The Off-chip Communication System
45
through the Off-chip communication module. The most generic way of doing this is to
transmit the global neuron ID from the output neuron that has spiked. However, the
current Output core design forwards only the local neuron ID of output neurons, as
the showcase network fits the full output layer in a single Neuron core. The schematic
for the Output core module is depicted in Figure 3.17. It consists of the same bus
interface as all other cores, that are configured to consume spikes from output neu-
rons. Spikes from the communication fabric are put in a queue for transmission to the
external system using the neuromorphic accelerator. The queue is implemented as a
memory with 16 entries. There are three counters associated with memory: One in-
cremented for every spike received used for addressing writes, one incremented when
delivering a spike to the Off-chip communication module used for addressing reads,
and a counter holding the number of elements in the queue. If a spike is received
when the queue is full, the spike is dropped. This will very rarely happen as spike
activity is sparse. Dropped output spikes has not been observed for the showcase
implementation. Spikes are delivered to the Off-chip communication module through
a valid-ready interface. This queue is necessary due to the UART protocol being
a slow communication protocol. Furthermore, spikes will usually be generated very
early in the time-step. Again, the design has enough overhead to support a spike on
all inputs synapses of a neuron in a given time-step, but this is an extremely rare case.
The most generic design would have a queue memory with a word size equal to the
axon id width, as this is the size of data received from the communication fabric
interface. In the design used in this project, a word size of 8 bits has been used the
queue memory as there are 200 output neurons in the showcase network (8 bits can
represent IDs from 0 to 255). Furthermore, 8 bits fit with the data wordlength of the
standard UART protocol.

The Off-chip communication (OCC) module is the bridge between the neuromorphic
accelerator and the external system that utilizes the accelerator (see Figure 3.3). The
OCC has been designed specifically for the showcase network. A block diagram of

The OCC module is connected to all Input and Output cores with a valid-ready in-

has a valid-ready interface for transmitting and receiving data. The UART is built
for one stop bit and no parity bit, and can be configured to a desired baud-rate the

4.1
BindsNET
With artificial SNN being a hot topic in computer-science, as well as neuroscience, the
number and variety of development tools is astonishing. Accuracy achieved by SNN
for machine learning purposes has not yet caught up to the accuracy achieved with
second generation ANN. This results in less standardization for training methodolo-
gies compared to second generation ANN development tools. With second generation
network training, gradient descent and back propagation have become the standard
for supervised training of ANNs. This standardization has led to Tensorflow [Goo]
and PyTorch [Fac] becoming the default ANN development tools. With a standard
methods, refinement becomes more obvious, which, in contrast, is not yet the case for
SNNs. None of the available training methods shows a consistent on par performance
with gradient descent and backpropagation, which is why the development of several
training methods for SNNs is ongoing. Furthermore, the wide range of neuron, synap-
tic, and encoding models are being explored and employed. This, in turn, led to a
variety of development tools. Each of these tools has its strong suit on the one hand,
and some disadvantages on the other. An overview and discussion on the investigated
development tools can be found in chapter 7
BindNET sums up their tool as follows: ”A Python package used for simulating spik-
ing neural networks (SNNs) on CPUs or GPUs using PyTorch Tensor functionality.
BindsNET is a spiking neural network simulation library geared towards the develop-
ment of biologically inspired algorithms for machine learning.”, leaving BindsNET as
a prime candidate for the development tool of choice. The workflow of BindNET is
quite familiar for users who previously worked with PyTorch or Tensorflow. The work-
flow can be roughly summarized with the following steps for building and training an
SNN using supervised learning:
1. Define a network
a) Define layers, and the neuron model for each layer. BindsNET has support
for several neuron models, which are referred to as nodes. The node/neuron
models also include training rules.
b) Define connections between layers. Connections are, effectively, synapses
connecting layers. Therefore, the model for connections is specified along
with learning rules.
c) Define monitors for the network. Monitors can probe different attributes
of the network such as spike activity on connections or membrane potential
of neurons, throughout the evaluation of a spike train.
2. Build a dataloader
a) Dataloader functionalities from PyTorch are used supplying the SNN with
input stimuli. This includes receiving data from the database. Further-

4.2.2
Training the Network
The showcase network is trained using supervised learning. The dataloader and train-
ing loop have been taken directly from the supervised MNIST example provided by
BindsNET. The dataloader resizes the handwritten digit images down to 22x22 pix-
els, that are thereafter encoded into spike trains using Poisson encoding. Poisson
encoding, as explained in Section 2.3, is very similar to rate-based encoding, but
adds stochastic noise that ensures better generalization.
Before the training loop can be comprehended, the output layer and the learning rule
of the connection In To Ae must be understood. Layer Ae act as both, the input and
the output layer. Traditionally, the output layer has as many neurons as there are
classes, which would be 10 for the MNIST dataset. This layer has 200 neurons, with
the first 20 neurons responsible for recognizing the digit 0, the next 20 recognize the
digit 1 and so on.
The details of how the learning rule, used for the connection In To Ae is imple-
mented are outside the scope of this thesis. The basic idea behind the learning rules
stems from the neuroscientist Donald Hebb’s hypothesis: ”Neurons that fire together,
wire together”. Synaptic weights in artificial SNN reassemble how strong the con-
nection/wire between two neurons is. The purpose of this learning rule is to update
==PAGE:68==
54
4 Network Development
a weight for generating pre- and post-synaptic (neurons in both ends of a synapse)
spiking activity equivalent.
The training loop is built utilizing this learning rule,
which is done by injecting a spike train to the network, obtained form a MNIST
image. This was implemented across 500 time-steps. As we use supervised learn-
ing, we know which digit the spike train represent. The training loop forces one of
the 20 neurons responsible for recognizing this digit to spike in all 500 time-steps
regardless of its membrane potential and threshold. This will utilize the learning
rule, that will ensures the pixel pattern get a strong connection to an output neuron
that should recognize the digit present input image. The learning rule will increase
synaptic weights proportionally to the spiking activity of the channel/neurons from
the In layer and the neuron that is forced to spike in layer Ae - thus, making these con-
nections/wires stronger. This ensures that we are following the philosophy ”Neurons
that fire together, wire together”.
4.2.3
Fitting the Network
The trained network cannot be run directly on the neuromorphic accelerator for
multiple reasons. The major reason is that the trained network is evaluated using
floating-point numerical representation. The membrane potential decay factor is not
supported by the hardware. This means that we must format the network parameters
prior to mapping them the to accelerator. This is done by rounding all parameters,
except for the decay factor, to the nearest integer.
The decay factors is rounded
to the nearest supported decay factor. These can be inspected in table 3.2. The
fitted network can be evaluated in software and should achieve similar accuracy when
running on the presented neuromorphic accelerator.
4.2.4
Accuracy of the Network
The accuracy is measured based on inference of the same 1000 MNIST images for
both, a fitted and a unfitted network. The fitted network is tested with spike trains
being encoded with pure rate-based encoding, as for the neuromorphic accelerator.
The best trained network achieved ccuracy of 73.8% and 73.2% for unfitted and fitted,
respectively. Unfortunately, this network file was overwritten by mistake. Parameters
for this lost network were extracted for mapping before overwritten. It is this network
that is used for the show-case implementation. The network results in table 4.1 were
retrieved from another network trained with exactly the same parameters, which
unfortunately do not reach the same accuracy. This accuracy is far from the state of

Chisel
The neuromorphic system is described using Chisel [Unia] [Bac+].
Chisel is an
open-source project from the University of California, Berkeley, described as follows:
”Chisel is a new open-source hardware construction language developed at UC Berke-
ley that supports advanced hardware design using highly parameterized generators and
layered domain-specific hardware languages. Chisel is embedded in the Scala program-
ming language, which raises the level of hardware design abstraction by providing
concepts including object orientation, functional programming, parameterized types,
and type inference.” [Bac+]. Chisel generates a synthesizable Verilog code for im-
plementation of digital circuits in silicon or FPGAs. It enables the user to describe
complex digital circuits in very few lines compared to VHDL. A big contribute to
line count reduction compared to VHDL and Verilog is due to the fact that Chisel

counting from 0 to 42, is implemented on only two lines in listing 5.1. When a regis-

In all other cases the register value refers to its output, such as RHS of assignment,
comparisons etc.

5.2
Parameterized Design
This section contains a summary of what and how parameters are set, and the con-
straints of these parameters, and how they are set. The neuromorphic accelerator
is implemented in a parameterizable fashion. This style of implementation eases to
accommodate the specific need of a given network.
The system parameters are all defined in the Constants Scala object, which can
be inspected in appendix B.1.
The Constants can be imported to all Scala files.
Thereby, they are accessible for them. The following list presents a summary of the
parameters:
• The clock frequency is set by the Scala constant FREQ. The clock frequency
is constrained by the fact that a Neuron core should be able to evaluate the
maximum number of incoming spikes in a time-step. This is limited by the
number of axons in a core A. If equation 5.1 is fulfilled the constraint is not
violated.
==PAGE:73==
5.2 Parameterized Design
59
• Baud rate of the UART component is set with the constant BAUD. The
baud rate does not carry any constrains.
• The duration of a time-step is set by the constant TIMESTEP, which should
match the time-step duration that the network has been trained with. For the
showcase network, the input is discrete data (image), encoded to a spike train.
For this reason, it does not matter if the evaluation of a time-step exceeds the
1ms duration that the time-step of the network has been specified to. However,
this is a special case for the showcase network. If the accelerator was running
a keyword spotting network, the network would receive new input spikes every
time-step with no finite number of time-steps. This differs from the input images
of the showcase network that are represented as spike trains of 500 time-steps.
In the keyword spotting example, a time-step must be processed within the
given duration of a time-step. The time-step duration has the same constrain
as the clock frequency (see Equation 5.1). Based on the frequency and time-step
duration, the number of clock cycles per time-step is derived.
• The number of cores that the system should be implemented with is set. The
top module needs to be written accordingly. The top file connects cores to the
communication fabric. The top Verilog module can be inspected in appendix
B.4
• Wordlength of network parameters is set. The design implementation only
supports integers, but the width of these has no constrains. All arithmetic units
in the NEUs will follow this wordlength.
• Number of NEUs per core (referred to with the symbol P) must be to
powers of 2 (2,4,8,...).
• Number of neurons that one NEU should evaluate (n) through time-
multiplexed execution is set. n must also be in powers of 2. The number of
neurons that can be mapped to a single core is N = n · P. Therefore , N is also
in powers of 2.
• The number of axons in a core (A) is set. A has to be equal to N or a
power of 2 larger than N.
In equation 5.1, N is the number of neurons that can be mapped to a core and A is
the number of axons of a core. For every time-step a core spends at least one cycle in
the idle state. For every N neurons, 9 of the FSM states are not dedicated to adding
weights to the membrane potential. The highest number of weights that should be
added to a neuron is equal to A. Thus, the LHS of equation 5.1 is the maximum
number of clock cycles it takes to evaluate all neurons mapped to a core.

5.2.1
Configuration of Implemented Accelerator
The neuromorphic accelerator is implemented on Digilent’s FPGA development board
Genesys 2 [Dig]. The Genesys 2 board is equipped with Xilinx’s Kintex-7 325T FPGA.
This is a fairly large FPGA, containing 51k logic slices with a 6-input LUT and 8
flip-flops per slice. More importantly, this FPGA contains 16 Mbits of fast single-
cycle block RAM. The presented design breaks with the Von Neumann architecture
since it has core local memories rather than one shared main memory. As a result,
the design is very on-chip memory heavy, which it is not well suited FPGAs. The
vast majority of the footprint is occupied by on-chip memory. The foot-print ratio
between logic and on-chip memory is not equivalent to the ratio generally seen in
FPGAs. Therefpre, only small configurations of the neuromorphic accelerator can be
implemented on the Kintex-7.
The implementation is configured with 5 cores. Core 0 and 1 are Input cores, 2 and 3
are Neuron cores, while core 4 is an Output core. The network is mapped onto these
cores according to Figure 4.1. The implementation is configured with a frequency of
80MHz, time-step of 1ms, UART baud rate of 115200, P = 8, n = 32 (thus N = 256),
A = 1024, and neuron evaluation wordlength of 17-bit. With this configuration, the
Neuron evaluation memories need to store 9Mbits (see Equation 5.2), 41Kbits in the
Axon systems memories (see Equation 5.3), and 9Kbits for Input cores memories (see
Equation 5.4). In total 9.02Mbits of data needs to be stored in block RAM for the
accelerator to be implemented. If these on-chip memories can be perfectly mapped
to the available block RAM, the Kintex-7 should have 7Mbits excess capacity. Un-
fortunately, this is not the case. Vivado reports insufficient available block RAM for
the implementation. An explanation for this insufficiency is given in Section 6.1 of
the result chapter. Hence, the accelerator can not be implemented on the Kintex-7
FPGA without any modifications.

and rWeight2 of the ASM chart (see Figure 3.12) the specific part of the weight

for the unmodified design.
As the effective number of axons is not in powers of
2 larger than N, concatenation cannot be used to obtain the weight address for the
modified implementation. Instead, the specific part of the address is obtained through
following arithmetic addrspecific = ni · 768 + axonID. The trade-off associated with
this modification is turning a free concatenation operation into a multiplication and
an addition, in favour of reduced memory requirements.

Configuring the System
The presented neuromorphic accelerator is configured to run a specific network by
populating various memories. One type of memories that must be populated as part
of configuring the accelerator is the look-up memories situated in the bus interface
components. The content of these memories governs which AER packages are con-
sumed by a given core. This process is network specific. The other type of memories
to be populated is the Neuron evaluation memories which contains all network param-
eters. In the current state of the system, there is no mechanism for populating these
memories, post implementation. Such a mechanism is discussed in Section 7.3. In
this section, it is described how these memories were populated at compile/synthesis
time rather than through a dedicated circuitry for configuring the accelerator.

The Neuron evaluation memories are populated using the Verilog construct (readmemb/readmemh).

is given the path to the memory to be initialized. Vivado can interpret a readmemb

function call from a Verilog file and compile memory initialization content into the
bit-file for programming a FPGA. To successfully incorporate the memory content in
the bit-file, the readmemb must be written in the same Verilog module as the decla-
ration of the RAM, and the files containing the initialization data must be correctly
imported to the Vivado project.
Several modification were necessary to successfully initialize the block RAMs with
network parameters on the FPGA. Before explaining these, the implementation of
the Evaluation memories must be inspected (see code listing 5.4). In Chisel, a Class
extending a textttModule results in a Verilog module with the same name in the gen-

content. As the readmemb needs to target each of these memories individually, they
cannot be instantiations of the same Verilog module. Thus, Chisel must be tricked
to generate 16 Verilog modules with unique memory names. It is possible to suggest
a name for net/variable declarations in the generated Verilog code with Chisel. This
is done in line 18 of listing 5.4. However, this does not make for 16 copies of the
memory module. Instead, we need to write some ”point-less” code that cannot be
realized with multiple instantiations of the same Verilog module. This is obtained by
declaring an output of the Chisele Module, nothing, and connecting it to a constant
value that is unique for this instance of the Chisel Module. This output is, in line
12, connected to the constant that is a concatenation of the core ID and the neuron
ID. This output is not connected to anything and, thus, it is optimized away by the
synthesis tool. As a result of these actions, the generated Verilog code contains 16
copies of the EvaluationMemory2 module with unique memory names.
Chisel has a function for generating readmemh function calls for memory initialization
(see line 21 of listing 5.4). This will generate a separate Verilog file with a memory

only work for initialising memory in simulation. In order to the initialization into

declares the memory. As a workaround, the python script in appendix B.2 needs to

initialized, the script inserts there a readmemb function. After the script is run, the
Verilog file complies with Vivados guidelines. This results in memory initialization
being compiled into the bit-file.

5.4
Inferred Clock Gating
The (naive) strategy for clock gating the NEUs when the core is idle was to let the
synthesis tool infer clock gating by having all registers in a module enabled using the
same enable signal. This strategy was suggested by the Chisel community that had
discovered that the synthesis tool of Synopsis Design Vision did a good job at inferring
clock gating. Unfortunately, the same cannot be said about Vivado’s synthesis tool
for FPGAs. Consequently, clock gating has not been implemented for the current
version of the system. Clocks distribution for synchronous systems must be handled
==PAGE:79==
5.5 Debugging and Testing
65
with care in order to prevent clock skew. Therefore, FPGAs typically have dedicated
clock management blocks to ensure good clock distribution. FPGAs contain fully
implemented flip-flops that are driven by a somewhat ridged clock tree. Clock gating
for the Kintex-7 should be performed through the clock manager circuit blocks that
the FPGA contains. These programmable clock manager tiles are the only way of
manipulating the clock signals. As these require special setup, clock gating is not
inferred by the synthesis tool. In Chisel, clock and reset signals are implicit. In order
to implement clock gating, several advanced Chisel functionalities must be employed.
To instantiate a module not described in Chisel (a clock manager), the black box
functionality is needed. Secondly, another clock signal must be used for the clock
gated modules. It is possible to specify individual clock signals on a per module basis
with Chisel. However, this combination of using black boxes from where a clock signal
is obtained breaks Chisel’s PeekPokeTester. PeekPokeTester is the test and debug
environment that is supplied by Chisel, and is covered in the next section. Although
true clock gating was not pursued further, it is still considered for future work.
5.5
Debugging and Testing
This section presents the tools used debugging and testing this implementation. All
functional testing and debugging of the system has been performed using Chisel’s
PeekPokeTester [Unib]. Waveform inspection is done with GTKWave. PeekPokeTester
is a functional test environment, for debugging and testing of digital circuits described
in Chisel. Much like Verilator [Sny], PeekPokeTester enables the user to write a test
harness for testing a Module described in Chisel. The PeekPokeTester gives the user
four operations that interact with the device under test (DUT):
1. Set inputs of the DUT with the poke function.
2. Record the value of an output of the DUT in a given clock cycle with the peek
function.
3. Test, a given clock cycle, an output of the DUT against an expected value with
the expect function. If an output does not have the expected value, the test
will fail and the failing expect statement is reported.
4. Let X clock cycles progress with the step(X) function.
With known interface behaviour of the individual component of the design, unit-tests
have been written for most components. An example of a unit test testing the bus
arbiter can be inspected in listing 5.5. The Class BusArbiterTest is the test har-
ness itself, while the Class BusArbiterSpec sets up and calls the PeekPokeTester
machinery. This call allows for a lot of paramters for the Chisel backend. In list-
ing 5.5, the arguments result in a vcd file containing waveforms for inspection, that
==PAGE:80==

Having everything embedded in Scala makes test-
driven development easily accessible, as the developer can have a reference software
model and the hardware description of a circuit in the same environment. With this
co-simulation, the hardware simulation and reference model can be run in parallel.
Therefore, the output values from the reference model can easily be compared to the
hardware simulation. However, a reference model, however, was not built for this

The last debugging method used in the development process is Vivado’s Integrated
Logic Analyser (ILA). ILA is part of Vivado’s IP catalogue and allows the user to in-
corporate a logic analyser in the design. With ILA, the user can specify which signals
of the implemented design should be probed by the logic analyser. These ILAs are
implemented on the FPGAs, which enables the users to probe signals on the FPGAs
in realtime. Therefore, bugs that cannot otherwise be found through functional or
post-implementation timing simulations, can be discovered. For this project, ILAs
were used to find bugs in the UART component caused by timing errors, which soft-
ware simulations with perfect timing were not able to catch. ILA debugging can be
cumbersome for several reasons. If a user does not know where a bug is occurring, it
is not possible to know if the placed probe is able to detect the bug. Also for every
reconfiguration of the ILA, the full implementation flow has to be run, i.e.
from
synthesis to bit-stream generation, which is a time-consuming process.
Although
time-consuming, ILAs can record all bugs in a system and, therefore, represents a
powerful debugging tool.

Timing Analysis
A timing analysis has been performed on the neuromorphic accelerator implemented
with a clock frequency of 80MHz. The timing constraints were met for this frequency
without any tweaking or pipelining of the original design. To test the limits of the
system, higher clock frequencies have been tried out. These tests show that the max-
imum frequency that can be obtained implemented on the Kintex-7 is approximately
115MHz.
==PAGE:90==

The critical path was found to be from the output of the period memory in the Input
cores to the spike buffer register of the Transmission system in the same core. This
critical path is sensible, as the signal has to pass the RAM logic followed by the
modulus operator and a comparison unit before reaching a register. As previously
stated, the modulus operation requires a large circuit containing a divider and a
subtractor. This path has a total delay of 11.9 ns, resulting in a slack of 0.6 ns. This
suggests that little effort has been put into meeting timing constraints. Designs that
are close to violating the timing constrains will usually report a slack very close to
0ns. This is positive as it indicates that the optimization tool has great overhead
for power optimization. The critical path delay is composed of 5.2 ns in logic delay
and the remaining 6.7 ns in net delay. Having net delay greater than logic delay is
very typical for FPGA implementations, and results from the architecture of FPGAs.
Logic is mapped to CLBs that are connected through a mesh of interconnect wires
and their switch metricises. This gives FPGAs a regular structure, rendering them
capable of implementing all standard sequential circuits. However, this also causes
long wires between CLBs.
Not only are the wires long, but signals also have to
propagate the logic of the switch metricises at every intersection.

The work done in this Master’s thesis is considered as an initial investigation of
using SNN to integrate offline machine learning capabilities into power-constrained

Although the current version of the neuromorphic accelerator consumes
more power than required by a sub 5mW device, like a hearing aid, it is the first

7 Discussion
7.1
Reducing Power
Power optimizations are a big field in digital design, and methods for reducing power
are plentiful. IMB’s TRUENORTH has a power density of only 20mW per cm2 [Mod].
A full TRUENORTH chip consumes 70mW of power while being capable of perform-
ing 46 billion synaptic operations per second, equivalent to 657 million synaptic op-
erations per 1mW. In comparison, the current state of the presented neuromorphic
accelerator is capable of 263 million synaptic operations per second, which is equiv-
alent to 266 thousand synaptic operations per 1mW. In other words, TRUENORTH
is approximately 2500 times more power efficient per synaptic operation. This is a
motivating observation as a factor of 2500 reduction in power would make it feasible
to implement a neuromorphic accelerator in an ultra low-power device. This section
discusses which steps must be taken to approach this state-of-the-art power reduction.
7.1.1
Power Estimations of Silicon Implementation
FPGAs are not the the most suitable platform when it comes to low power consump-
tion. The focus of FPGA architecture, is to allow an arbitrary digital design to be
implemented on them. They cater digital system development prototyping and func-
tional verification of designs before production of a chip. FPGAs are therefore rarely
used for a final product, unless only a small quantity is expected to be sold, making
the silicon production price too high to gain a profit, or where reconfigurability is an
integrated part of the product. Making an exact estimation of power reduction gained
by moving from FPGA to silicon is extremely challenging as power consumption is de-
pendant on a lot of variables: Fabrication process, cell library, design methodologies,
power optimization techniques, RAM compiler etc. [KR07] tries to measure the gap
between FPGA and ASIC with respect to logic density, speed and power. This paper
is from 2007 and compares different benchmark circuits for 90nm CMOS technology
for both FPGA and ASIC/silicon. The interesting observations for this discussion are
that [KR07] reports that dynamic power is reduced 12x on average when considering
logic only, while dynamic power is reduced 9.2x on average when considering logic and
memories. This suggests that more reduction can be gained for logic than memory.

7.1.2
Reducing Power Consumption of Memories
The strategy for obtaining low power in neuromorphic systems is by exploiting the
sparse activity. Through coordination and timing of arithmetic operations, the system
or regions of the system can idle for the majority of a network input evaluation. This
gives the option to power off specific regions completely during idle phases. Conven-
tional RAM implementations are volatile, thus, data are not retained when powered
off. The high consumption of static power from a large amount of on-chip memories
can be reduced by voltage scaling regions contain RAMs to the minimum voltage that
that still allows data to be retained.
In order to power off memory regions completely when they are idle, non-volatile mem-
ories must be employed. Research in resistive-switching memory (RRAM) has seen a
great progress in recent years [Iel18] [Zhu+19]. This memory technology stores data
by changing the resistance across a dielectric solid-state material. This is possible as
the RRAM contains the memresistor components. Memresistors’ resistance can be
changed by applying strong voltage differences over them [Sin+16]. As such a logical
==PAGE:98==
84
7 Discussion
0 is stored as low resistance while a logical 1 is stored as high resistance. Details of
the technology are outside of the scope for this discussion. Research shows RRAM as
a promising technology for neuromorphic system storage. Also RRAM proves good
scalability, high speeds, low power, and CMOS compatibility [Iel18]. Development of
RRAM has not quite taken them to commercial products, but the literature suggests
RRAM being the memory of choice for future neuromorphic systems. As mentioned,
the presented network is idle 99% of the time. The possibility of power gating these
memories and the NEUs could yield significant reduction of power.
7.1.3
Asynchronous Implementation
The literature suggest that neuromorphic systems are prime candidates to asyn-
chronous implementations [You+19] [Ako+15] [Bou+19] [Ind+11].
Asynchronous
circuits generally require a higher transistor count than their synchronous counter-
part. This is because asynchronous circuits needs handshake circuitry that propagates
token from one handshake latch to the next (In the case of asynchronous circuits built
with handshake methodology). In turn, clock signals are not used, which effectively
means that switching only occurs when there is an event. Asynchronous design can
make for power reduction, even with the handshake transistor overhead. Power re-
duction is possible due to the activity being sparse in the neuromorphic accelerator.
Figure 6.3 illustrated that the dynamic power consumed by the clock tree is 18 times
greater than what logic consumes. With an asynchronous implementation, no clock
tree is needed, resulting in reduction of power.
State-of-the-art projects is employing an asynchronous design methodology, such as
IBMs TRUENORTH [Saw+16] with a hybrid design combining synchronous and
asynchronous circuits, and Intel’s Loihi [Dav+18] that is fully asynchronous. How-
ever, FPGAs are not suited for asynchronous implementations. Libraries exist that
makes the implementation of asynchronous circuits easier to implement on FPGAs
[MJS19]. An asynchronous implementation of the presented neuromorphic accelera-
tor was considered. However, it was decided not to pursue this possibility as would
expand the size of the project beyond reasonably for a Master’s project.

7.3.2
Implement Clock Gating
In the implementation chapter, it was explained that clock gating was not imple-
mented fully. Implementing clock gating correctly is fairly straight forward for in
FPGA. Each core should contain a clock manager with an enable signal. The out-
put of the clock manager should drive the clock signal for NEUs and the evaluation
memories. The enable for the clock manager is part of the current design. This is
the enable signal for registers in the NEUs.
The implementation chapter describes how this should be implemented with Chisel
specific functionalities.
Implementing clock gating should make for a substantial
power reduction. The memories consume 66% of the the clock trees dynamic power.
They also consume 73% of the overall dynamic power consumption, which is expected
to be reduced as well.
7.3.3
Better Arbitration
The current arbitration method for both the Transmission system serving NEUs and
the bus arbiter serving cores has a disadvantage. The disadvantage stems from the
use of a mask that should ensures more fair service. The disadvantage is that higher
==PAGE:102==
88

priority request than the one served in a given cycle is masked in the subsequent cycle.
If only higher priority requests are present, no requests are served in the subsequent
cycle. This can under some request patterns lead to a lot of wasted cycles. The
number of wasted cycles can approach 50%.
The showcase network has very sparse spiking activity. Therefore, arbitration band-
width has not degraded performance. This might not be the case for all networks,
thus another arbitration method should be used. This can not be solved by simple
solutions. Pure priority service can block service from low priority request completely.
Round robin service can make for more than 50% wasted cycles under low activity.
A simple queue can not be used as multiple new requests can occur in the same cycle.
A possible solution is to extend the current design with a multiplexer and an OR
gate. This solution can be inspected in Figure 7.1. When the masked requests are all

7.3.4
Design for Sparse Activity
It has been stated that the system should support the worst-case spiking activity. This
means the system must be able to evaluate all neuron under the maximum number
of possible spikes. This constrains the number of neurons that can be mapped to a
core, as only X clocks cycles are available in a time-step. Ultra low-power devices
typically run on very low clock frequencies, which in turn means that cores with the
current design can only evaluate a very low number of neurons. Therefore, only small
networks can be run. For the current implementation running at 5MHz, each NEU
==PAGE:103==
7.3 Future Work
89
has to be reduced to evaluate only 2 each, making a single Neuron core able to have
only 16 neurons mapped to it. The showcase implementation is idle for more than
99% of the time. Therefore, the design should be altered to not require worst-case
service. This only requires a small change to the Control circuit of the Neuron cores,
where the FSMD should go to the idle state at the last clock cycle of a time-step
regardless of the current state. This can degrade accuracy as not all neurons are
guaranteed to be evaluated in a given time-step. If an analysis of a network shows
that spike activity stays below 2% for 99% of time steps, the system can be configured
to support 5% activity which might result in neglectable accuracy loss.
7.3.5
Generic Input and Output Cores
Input and Output cores have not been designed in a generic manner. For the acceler-
ator to run an arbitrary network, all aspects of the design should be generic. For the
Output core, this is just a matter of transmitting the global address of a spike rather
than the local neuron id of the output layer.
The Input core needs a more thorough redesign. The simplest generic approach is to
have the Off-chip communication system deliver input neuron IDs in every time-step
that corresponding input neurons should inject a spike onto the network. This means
that the system is not constrained to a specific spike encoding or a maximum spike
train duration. However, this approach requires much higher bandwidth between the
system delivering spikes to the accelerator and the accelerator itself.
7.3.6
Silicon Implementation
The design should ideally be implemented as ASICs in silicon rather than on FPGA
to obtain minimal power consumption. Silicon implementation gives more control
as it is not constrained to be mapped on the resources on an FPGA. This includes
power optimization options that were covered in Section 7.1.
Implementing and
simulating the design with Synopsis Design Vision was considered for this project. It
was ultimately moved to future work in favour of time spent elsewhere. The workload
of this process was hard to estimate, as the author has no previous experience with
RAM compilers. Furthermore, the level of modification to the source code that would
allow integration of compiled RAM was not clear. This is, however, an important
point in future work for estimating the viability of the design, as a silicon simulation
would give realistic power estimates for practical implementation.
7.3.7
Microcode Cores
The current design is constrained to fully-connected SNNs using the LIF neuron model.
The LIF seems to be the model of choice for the vast majority of SNNs, mainly due to
the simplicity of the model that leads to a minimal amount of computation. The fact
==PAGE:104==
90
7 Discussion
that the Neuron cores are controlled by an FSMD makes the design simpler and easier
to optimize. However, this also constrains the possibility of networks that can be run.
Implementing cores as microcode accelerators can broaden the diversity of networks
that can be executed. It is possible to implement a core tailored for neuron evaluation,
while still running on microcode. With such an implementation, different micropro-
grams could be written to a dedicated memory, allowing for other network structures
such as convolutional SNNs and other neuron models. This implementation would
most likely, reduce throughput, increase power consumption and area, enable more
diverse system. Although this is a future step in a slightly different direction, it would
be interesting to investigate such an alternative.
7.4
Evaluating the Project
In this final section, the success of the presented neuromorphic accelerator is discussed.
This discussion starts with a functional evaluation of the presented accelerator.
7.4.1
Functional Evaluation
The result chapter shows that the network accuracy was degraded by only 2.4% be-
ing mapped from a 32-bit floating point model to 17-bit integer model running on
the neuromorphic accelerator. This proves that the presented system, is capable of
running a SNN successfully.
Had the baseline accuracy been larger, one could expect a greater degradation of
accuracy than 2.4%. When a network has high accuracy, its parameters are very
finely tuned. To avoid accuracy degradation in the mapping process, the network
should be trained using the data formats that are available on the target processor.
Popular second generation ANN development tools, such as TensorFlow [Goo] or
PyTorch [Fac] have such support. TensorFlow has a library (TensorFlow Lite) for
running ANNs on embedded devices. This library has support for clever quantization
from floating-point to 8-bit integer format. When SNN development tools mature,
similar functionalities can be expected available for SNN, which will allow for better
accuracy on constrained devices.
7.4.2
Implementation Evaluation
The current state of the accelerator still requires additional optimizations in order to
be deployed to a power-constrained device. For example, 1W of power consumption
would drain a button call battery, comparable to a hearing aid battery, in less than
15 minutes. Even with the generous estimation of 66mW power consumption of a
silicon implementation, a button cell would drain in under 3 hours.

CHAPTER 8
Conclusion
This Master’s thesis report presented the design, implementation and evaluation of
a neuromorphic accelerator for the execution of spiking neural networks. The design
objective was to build a low-power system suitable for highly power-constrained de-
vices, such as hearing aids.
The largest contributor to the high power consumption of second generation neural
networks execution is the high number of multiplications that are needed. It was
found that third generation artificial neural networks, known as spiking neural net-
works, require only few multiplication when executed. Furthermore, it was found
that spiking neural networks experience sparse activity. These two characteristics of
spiking neural networks make them good candidates for enabling low-power machine
learning capabilities in power constrained devices such as hearing aids. By exploiting
these characteristics, a low power accelerator was designed and implemented on an
FPGA, through minimizing switching.
Through approximation, a multiplication free design was developed, still achieving a
neglectable accuracy loss. The accelerator was written in a parameterized manner.
This enables an implementation of the accelerator to be easily sized to a specific use-
case. Furthermore, the system can be configured to run an arbitrary spiking neural
network, that utilizes fully-connected topology and the leaky integrate and fire neu-
ron model. To demonstrate the neuromorphic accelerator, a showcase network was
trained and run on the accelerator implemented on a FPGA. Here, it was observed
that the accuracy of the network degraded by only 2.4% when running it in software
using 32-bit floating point in comparison to running it on the accelerator using 17-bit
integers.
Resource consumption, timing analysis, and power consumption was measured, with
the focus being on power consumption. The most noteworthy result is that the sys-
tem consumes 989mW of power, where 73% of the dynamic power was found to be
consumed by the high count of Block RAM that the accelerator uses.
several aspects of the project were discussed. Most interesting to the focus of the
thesis are discussions on power consumption. Here multiple optimizations were pro-
posed, such as employing employing an asynchronous design that can take further
==PAGE:108==

advantage of the sparse activity. Optimizations possible with an ASIC implementa-
tion is considered. Most noteworthy is the use of an alternative non-volatile RAM
technology, RRAM, that allows idling regions of the accelerator to be powered of
completely. An estimation of power consumption with an ASIC implementation is
found to be around 70mW.
This Master’s project is a successful initial step towards a neuromorphic accelerator,
that can enable machine learning capabilities in ultra low-power devices.
