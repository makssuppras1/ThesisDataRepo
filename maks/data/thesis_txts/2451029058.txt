Summary 
For students studying in a technological university it is very positive to get hands-on 
experience with a real project. For this reason, a group of students created the 
Vermilion Racing Team in 2017. The purpose of this team is to build a formula car able 
to compete in the formula student and reach the top positions. Many different 
subsystems must work together to make the car able to run, and that is the reason it 
is a very good opportunity for students to learn teamwork.  
One of the critical components in a car is the master controller, an element that 
coordinates the electrical subsystems using the CAN communication protocol. The 
team already developed a master controller’s code during the 17-18 season, 
however the code was not optimized and a number of changes in the hardware 
made necessary to develop a new code. For this reason a new team of members 
should be dedicated to create a new master controller’s code and make it better, 
faster, reliable and scalable. 
Along the development of the master controller code, new needs arise in the team. 
Some of these problems are fixed in a few hours or a few days, but some of them 
constitute projects by themselves due to the amount of workhours required for their 
development and testing. It is the case of the BSPD and the optimization algorithm for 
a Simulink model. Therefore these elements have also been included in the thesis as 
their development took an important fraction of the available time. 
The BSPD is a hardware board that reads a brake pressure transducer and a current 
transducer and disconnects the high voltage system if the motors receive enough 
current to deliver 5 kW and at the same time the driver actuates over the mechanical 
brakes for more than 500 ms. The circuit may reset itself if the opening conditions are 
no longer present for 10 seconds. This apparently simple behaviour is greatly 
complicated by the fact that no software control is allowed. Hardware circuits are 
more reliable but also more complicated and take much longer to develop for the 
same functionalities.  
Most teams who develop high performance vehicles have a mathematical model of 
their vehicle. A mathematical model allows the team to evaluate the system 
performance. The model combined with an optimization algorithm gives the vehicles 
optimum parameters for an optimum performance. These apparently small 
differences in performance make a huge difference in competition environments.  
This report documents the development of the three of these projects and evaluates 
the results obtained. As it can be seen in along the thesis, all objectives are met 
satisfactorily. Even though not projects have been fully developed, it has been mostly 
due to external factors. The final tests will be done in the pit during the FSUK, but the 
results cannot be included in this thesis due to calendar incompatibilities.

1 Introduction  
1.1 Historical background of the electric car 
The first electric car is dated from the 1832, made by Robert Anderson shortly after the 
invention of the electric motor, but it is not until the 1870s that it becomes practical. 
Electric cars had a golden age in 1920s, but because of their short range and the 
battery recharging time, they started to loose popularity against the combustion cars, 
much faster to refuel. However, the importance of electric cars has raised 
dramatically in the last years after proved evidence that humanity is running out of 
petroleum reserves at the same time that petroleum combustion is causing a global 
warming effect, which may lead us to an environmental disaster. 
The first attempt of the automotive industry to implement an electric motor was made 
by Toyota in 1997 when they launched the Prius, the world's first mass-produced hybrid 
car. The market accepted the Prius and it was the only hybrid car available until 2010, 
when other manufacturers launched their own hybrid versions of their models.  
This trend can be observed nowadays, with over 2 
million fully electric cars sold around the world. It also 
appears in competition environments like the 
Formula-E, or for instance the FSUK (Formula Student

challenge for mechanical engineers, however in 2008 the class 1A was introduced 
which made especial emphasis on environmental impact, making place for electric 
cars. After 4 years of development, the gap between electric and internal 
combustion cars was so narrow that the organizers decided that cars should compete 
in the same category. [1]

2 State of the art 
2.1 CAN Protocol 
The Controller Area Network (abbreviated as CAN) protocol was developed by Bosch 
and officially released in 1986. Since its release, Bosch published several versions of 
CAN and the latest is CAN 2.0 published in 1991. In 1993, the International Organization 
for Standardization (ISO) released the CAN standard ISO 11898, which eventually 
made CAN become the standard protocol in the automotive industry. [4] 
Since 1993 several CAN extensions have been published, like CAN 2.0B and CAN FD. 
CAN 2.0B adds eleven more bits for the identifier, which allows the bus to handle a 
wider variety of frames. CAN with Flexible Data-Rate or CAN FD was published in 2012 
and added many features. One of the main features is a faster transmission speed 
that reduces the time delay and makes the system closer to a real-time machine. In 
addition, the bandwidth was increased, the identifier increased, and the maximum 
data size increased from 8 bytes to 64 bytes.  
2.1.1 Frame types 
The CAN standard has three different frame types, the data frame, remote frame and 
error frame. The most used frame is the data frame. It contains 12 fields, of which the 
designer defines only five; identifier, RTR, extended identifier bit, data length, data.

Remote frames are a special kind of frame used to request data from a node. This 
frame’s RTR is set to high and data field is empty. Once the listener receives the remote 
frame it generates a data frame with the same identifier and sends it through the bus. 
This function is useful to request some specific information from a node in the network. 
Nonetheless, it is not frequently used as the most common way is program the node 
to send the data periodically.  
Nodes automatically create error frames when they notice a problem with the bus. 
These error frames have an active/passive flag that communicates if the error is still 
present or not, and an error counter. [5] 
2.2 LabVIEW 
Laboratory Virtual Instrument Engineering Workbench (LabVIEW) is a development 
environment for a visual programming language from NI (National Instruments). The 
graphical language is named "G", not to be confused with G-code. Originally 
released in 1986, LabVIEW is commonly used for data acquisition, instrument control, 
and industrial automation. [6] 
The main characteristic of LabVIEW is that it is a graphical 
programming language. This allows people to drag and drop a 
function or VI (Virtual Instrument), as they are called in LabVIEW, 
and define the function arguments through wires connected to 
the VI. Therefore, the programming paradigm in LabVIEW is based 
Figure 5: LabVIEW 
==PAGE:16==
 
 
 
LabVIEW

data availability. If a VI has the data available in all its input wires, the VI runs and the 
values returned by the function flow through the output wires. With this paradigm it is 
easy to define in a graphical way which VIs must run first. This way to define the 
execution order through wires has another important advantage. It is very easy to 
create VIs that run in parallel. For any VI to be executed it must have all its input 
arguments or wires, if two or more VIs have all the input arguments at the same time 
they can run in parallel. LabVIEW lets the user easily define priority levels to each VI so 
the code knows where to allocate more resources in run-time.  
As a programming environment developed by National Instruments, it has dedicated 
libraries to control National Instruments hardware. This creates an ecosystem where 
LabVIEW advances at the same pace as their products do. Still, LabVIEW also has tools 
to control third party instruments and some users even created libraries specific for an 
instrument.  
2.2.1 X-NET library 
The X-NET library is a library created by National Instruments 
to control CAN 2.0, CAN  FD and LIN modules. This is a high-
level library which has functions that manage the module’s 
hardware and the FPGA and gives the user simple VIs to 
control the CAN bus. The main characteristic of this library is 
the CAN dictionary. This dictionary has a list of frames that the 
device should read, and defines how the data in these 
frames must be translated into engineering values. 
Each dictionary has clusters. Each cluster defines the frames 
for every protocol. Thus, when the system has more than one 
communication protocol, CAN 2.0 and LIN for example, the 
developer must create a cluster for each protocol. Inside 
each cluster there are different frames. Frames have an 
identifier, a data length and a timing type. Timing type can be periodically or event 
based. A periodical frame will be transmitted at a fixed time interval regardless how 
many times the data field was updated. An event-based frame will be transmitted 
only when the data field changes. Each frame has signals. Signals are elements that 
transform the frame data into engineering units, something the user will understand. 
Many frames contain several parameters, so it is common to find frames with many 
signals, one for each parameter.  
2.2.1.1 Session types 
There are three session types for signals and three session types for frames. In addition, 
each one of these session types can be periodic or event-based. The following 
descriptions are focused on reading operations, but write operations have the exact 
same behaviour. [7] 
 
Signal single-point: This session type returns only the last signals received by 
the NI 9862. For example, if one node in the CAN network sends five frames 
Figure 6: NI 9862 module 
==PAGE:17==
 
 
 
 State of the art 
7 | P a g e  
 
with the same identifier and after that we perform a read operation, we will 
only read the last frame’s data.  
 
Signal XY: This session type returns all values received since the last read 
operation. In this mode, the NI 9862 creates a buffer in the FPGA that stores all 
signals received together with a timestamp, and when a read operation is 
performed it returns all elements in the buffer and flushes it afterwards.  
 
Signal Waveform: In this session the NI 9862 creates a FIFO buffer in the FPGA 
of any size. Then every time the transceiver receives a frame that contain that 
signal, adds it to the FIFO buffer together with a timestamp. When a read 
operation is performed, it returns all elements in the FIFO, but without flushing 
it afterwards. This method ensures that a read operation will always return the 
same amount of data points, even though it is possible to lose some data if 
the period between read operations it too high.    
 
Frame single-point: Same as the signal single-point returns the last element 
received but instead of returning signal values returns a frame cluster. A frame 
cluster contains; identifier, an extended and echo Boolean flags, type (CAN, 
LIN, CAN FD …), timestamp and payload (as an array of bytes).  
 
Frame queued: Very similar to signal XY, this session type stores all frames with 
the same identifier in a FIFO and returns all elements when a read operation is 
performed. Afterwards it flushes all elements in the FIFO. The developer must 
create one different session for each frame he desires. 
 
Frame stream: This session type is similar to frame queued with the difference 
that there is no frame filtering. All frames present in a cluster and received will 
be stored in a FIFO and returned every time a read operation is performed. 
This session type is optimized for data logging. Thus, when a frame arrives it is 
stored in the session it belong to and the stream FIFO. When a queued 
operation flushes their FIFO for example, it will not affect frame stream session 
as the data is stored in different places.   
2.2.2 TDMS file format 
Usually developers focus in higher-visibility decisions such as software architecture or 
hardware design, overlooking data file formats. In the long run, this creates 
inconsistent file formats, low readability, and scattered information around different 
locations. In the end developers end up storing lots of unusable information. At this 
point the need for a new file extension arises.  
NI developed the TDMS file format to overcome all these problems. It was designed 
to be compact, searchable, and that data could be written and read at high speed. 
Another key point is its easy conversion to other file formats or software other than NI´s.  
TDMS files have a hierarchic structure. Every file contains groups, every group contains 
channels, and every channel contains several measurement points. An easy way to 
understand it is to make an analogy with Excel. The file would be a workbook, a group 
a worksheet, channels would be columns, and rows would contain every 
measurement point. To ease search and improve readability every element can 
contain properties. Properties can be inherent of a file, group or channel.

available online for the most common extensions, but developers can also create 
their own. [8]

3 Vermilion Racing car’s description 
In order to produce a working car, many components must work together and in 
complete synchronisation. These components may have very different functions and 
may be less or more critical. To better understand how these subsystems work together 
all vehicles classify their components as mechanical components or electrical 
components. Classic examples of mechanical components are the chassis, shock 
absorbers or the brakes. For electric vehicles, the motor is the clearest example of 
electrical component, but other classical examples are the dashboard or the ABS 
system. As this thesis’ objective is to develop a code for a master controller, 
explanation about the mechanical components will not be further expanded.   
In the electrical systems, components are often classified by if they are placed at the 
car’s front or at the back, being the firewall1 the elements that separates the two 
areas. At the car’s front there is the dashboard, which is the interface between the 
driver and the primary electrical subsystems. In the dashboard the driver has the RTD 
button to turn on the car, but also a shutdown button to turn it off in case of 
emergency.  
At the driver’s feet there is the pedalbox board, which reads the accelerator and 
brake pedals position and send it to the MCU through CAN. At the same time the 
pedalbox board reads the steering wheel position with a potentiometer and the front 
wheels’ speed with a rotary encoder, and sends the information through CAN to the 
MCU. Very close to the pedals too is the BOTS (Brake Over Travel Switch) and the BSPD 
(Brake System Plausibility Device). The BOTS is a switch placed at the end of the pedal 
travel. If the brake lines work properly, it should not be possible to touch the BOTS with 
the pedal, but if there is a brake fluid leakage the pedal should reach the BOTS and 
flip it. When the BOTS is flipped, the shutdown circuit disconnects the HV (High 
Voltage) system.  
Most electrical components are placed at the rear of the car. Doing a sweep from 
the cockpit to the rear wheels, the first component is the LV (Low Voltage) battery. 
The LV battery outputs 24 V for all low voltage subsystems and is based on LiFePO4 
chemistry. Next to it there is the cooling pump. This pump makes the coolant flow 
through the vehicle, and the coolant absorbs the excessive heat from the motor and 
the motor controller and releases it to the atmosphere using the radiators.

Close to the LV battery there is the electronics box. Inside this box there is the MCU, 
the shutdown circuit, the inertia switch and the BSPD. The MCU is the brain of the car. 
It implements CAN communication and uses it to receive information from critical 
subsystems, processes it, and send commands to other subsystems. The shutdown 
circuit is a very important safety feature of the car. When any circuit connected to it 
sends an error signal, the shutdown circuit opens the AIRs and disconnects the HV 
system. Before opening the AIRs, the shutdown board sends a frame to the MCU and 
gives it a 250 ms time window to decrease the current flowing through the HV system. 
Doing this the relays’ activation is not so aggressive and increases the life expectancy 
of the components. The inertia switch is a mechanical device that detects hard 
accelerations. When there is an accident and the car crashes, the inertia switch is 
triggered and sends a signal to the shutdown board to disconnect the HV system. Last 
circuit in the electronics box is the BSPD. This circuit uses a current transducer and a 
pressure transducer to check if the motor has more than 5 kW of power and the driver 
is braking at the same time. When this happens, it is considered a panic situation and 
the BSPD sends a signal to the shutdown board to disconnect the HV system.

Moving further to the back there is the motor and the motor controller. These two 
elements are the muscles of the car, what makes it run. As there are no batteries able 
to store AC power and most high power motors require an AC line, the motor 
controller is the element that takes the DC input from the accumulator and transforms 
it into AC. The motor controller also implements many safety features such as 
overcurrent or overheating protection. It also controls the power delivered to the 
motors by modulating the AC lines frequency. Since the motor is a synchronous motor, 
the motor controller controls it by modifying the output frequency and voltage.

Right at the back of the car there is the accumulator box. Inside this box there are the 
cells that store the electrical energy but also other components; the AIRs, pre-charge 
circuit, TSAL, IMD and AMS. The AIR (Accumulator Isolation Relay) is a relay that isolates 
the accumulator from the outside. There are two AIRs, one for each pole of the 
accumulator. The shutdown circuit controls the AIRs. The pre-charge circuit ensures 
that the intermediate circuit is pre-charged to at least 90% of the accumulator 
voltage. The second AIR will close once the pre-charge circuit detects that the 
voltage level is correct within 2 seconds. Otherwise it will send a signal to the shutdown 
board to open the first AIR.  The TSAL (Tractive System Active Light) is an indicator that 
says if the HV lines are connected or not. The TSAL will be red and blink continuously 
when either one AIR is closed , the pre-charge relay is closed, or the voltage outside 
the accumulator box is over 60 VDC. The TSAL will light green when all the conditions 
are not present. The IMD (Insulation Monitoring Device) is responsible to check that 
the HV and LV circuits are galvanically isolated. Otherwise, it sends a signal to the 
shutdown circuit to open the AIRs. The last subsystem to describe is the AMS. This 
element basically protects and controls the accumulator cells. It has overcurrent and 
overtemperature protection to avoid potential problems with the cells. Nonetheless 
its main function is to control all cells are equally charged when they charge or 
discharge. Even if all cells are part of the same batch, they will have small differences 
between them that make ones charge or discharge faster than the others becoming 
a limiting element, and that reduces the accumulator capacity as well as becomes 
a potential safety problem in case cells are being overcharged. The AMS purpose is 
to balance all the cells. All the systems in the accumulator box are powered directly 
from the HV system. As most electronics do not withstand voltages of 600 V, a DC/DC 
switch mode power supply reduces the voltage from 600 V to 24 V.

3.1 Motor Controller 
The subsystem including the motor drive and the electric motor is a system containing 
external connections to the other subsystems such as the cooling of the car, the 
master controller and the accumulator. The system operates through CAN 
communication from the master controller to the motor controller, a Rinehart 
PM100DZ, which controls the parameters of the motor Emrax 228. The motor controller 
draws 600 VDC from the accumulator, and converts it into AC for the motor to use. 
The Emrax 228 delivers a continuous motor power of 28-42 kW depending on the 
speed and is chosen on the basis of earlier experience with this motor, and the 
compatibility with the chosen motor controller, with the difference from last year 
being the motor now running at high voltage >250V. The motor controller was chosen 
based on other subsystems on the car communicating in CAN, while also being 
reliable and efficient.  
 
Figure 12: Motor test setup with Emrax 228 (bottom left) and Rinehart PM100DZ (right) 
The Rinehart PM100DZ comes with an already pre-built CAN database. Hence, 
developers cannot define the frames identifiers one by one. Nonetheless, it is possible 
to disable specific frames or add an identifier offset, which changes the identifier of 
all frames. This is a limitation in our design stage, nonetheless the CAN database 
available is very complete and has all the data the master controller must know.  
3.2 Car’s front sensors 
There are some sensors placed in the car’s frontal part, and one Arduino manipulates 
all these sensors. These sensors involve the pedals, steering wheel and encoders.  
The technique to measure the accelerator pedal position and the brake pedal 
position is the same for both. Each pedal has attached two Bourns 53RAA-R25-A15L 
potentiometers on its base. These potentiometers change position as the pedal 
moves, and the resistance value changes linearly with the angle movement. Then the

Arduino reads the voltage in the potentiometer and uses transfer function to transform 
the voltage into pedal travel in percentage. Every potentiometer has a different 
transfer function to avoid calculation errors. The Arduino can know the voltage at 
resting position and at flooring position because it has been previously calibrated.

The steering wheel angle position sensor uses the same technique as the pedals. A 
Novotechnik SP2841 rotary sensor potentiometer is attached to the steering wheel, 
and it turns as the wheel turns. The Arduino reads the output voltage of the 
potentiometer and determines the steering wheel angle through a transfer function. 
An important difference with the pedals is that it only has one encoder, so there is no 
redundancy check. Finally the position of all elements is packed in a CAN frame and 
sent through the bus to the Master Controller. The Arduino generates a new package 
with the latest data every 50ms.  
To measure the car’s speed a Bourns Optical Encoder is used. There are two encoders 
on the front wheels and they are connected to the Arduino. To calculate the speed, 
the Arduino counts the number of ticks for 20 ms and from it can obtain the average 
speed at every moment. Nonetheless, in case the car is cornering, the speed of the 
inner wheel and the outer wheel will be different. Then what the Arduino does is the 
average of both speeds to calculate the forward speed. It also calculates the angular

3.3 Shutdown circuit 
The main objective of the shutdown circuit is to open the AIRs (Accumulator Isolation 
Relays) in case of failure. The AIRs are capable of disconnect the accumulator from 
the rest of the car, entering this way a safe state where the driver can safely stop the 
vehicle regardless which element fails.  
There are some rules on how the shutdown circuit must behave. First, the shutdown 
circuit must open the AIRs if any of the control-inputs is low. If the AMS or IMD control 
==PAGE:26==
 
 
 
Accumulator Management System 
16 | P a g e  
 
signal is low, it must be latched in such that the only way to close the AIRs again is to 
manually reset the LVMS. The shutdown board may implement a delay circuit that 
delays the opening of the AIRs 250 ms at most since the signal is received. The motor 
controller may use this time window to turn off the engine and avoid huge currents 
flowing through the high voltage wiring when the AIRs open. Finally, the shutdown 
circuit must have CAN communication and send the fault cause to the MCU. 
 
3.4 Accumulator Management System 
The accumulator management system used in the car is the n-BMS developed by 
Lithium Balance. This system contains one master board, the MCU (Main Control Unit), 
and 12 slave boards, the CMUs (Cell Monitoring Units). There is one CMU per 
accumulator segment, which can measure the voltage and temperature of each of 
the 12 sub-segments within each accumulator segment with an accuracy of <±1.5mV 
in the range between 1-4.5V for the voltage measurements, and an accuracy of ±1°C 
in the range between -40-80°C for the temperature measurements. The typical 
sample rate is 10Hz. The system also measures the total current throughput of the 
accumulator pack, using an Ohmite SHT1-200C075DE shunt resistor. This gives a range 
of 0-200A with a resolution of 13mA. 
The system has various ways of communicating with other systems. Sixteen digital 
GPIO ports can be used for contactor control between the accumulator pack and 
the load or charge. This includes contactor control for a pre-charge circuit. In case of 
any critical error, all contactors will open. The AMS also has CAN bus functionality, and 
can be customized to send any relevant data out on the CAN network to be 
monitored externally from the AMS. 
The n-BMS MCU draws power from the High Voltage System, while the CMUs draw 
power directly from their monitored accumulator segments.

3.5 Dashboard 
The dashboard offers an interface with the driver and gives information about the 
status of the car. Some of its indicators are hardware controlled, like the shutdown 
circuit indicators or the low voltage master switch, but others admit software control. 
The RTD (Ready to Drive) button for example is controlled with an ATMega. Every time 
the driver pushes the button a frame is sent to the CAN network. The master controller 
reads this frame and decides to either turn on the car, turn it off, or do nothing. Two 4-

4 Master Control Unit 
The MCU (Master Control Unit) is the main computational unit of a car, often called 
the car’s “brain”. The objective of this unit is to coordinate all different ECUs in the car.  
4.1 Functions 
The MCU must coordinate all systems connected to the CAN network. In addition, it 
must implement certain safety functionalities required by the rules. Here is a list with all 
the functions it must implement: [9] 
 
The motor must receive a torque request proportional to the accelerator 
position in such a way that the accelerator response is linear and predictable. 
 
The motor must receive a negative torque request and act as a generator 
when the brake pedal is pressed. The brake response must also be linear. 
 
When both the accelerator and brake pedals are pressed, the accelerator 
position must be ignored. 
 
The response time of the pedals should be under 80 ms. 
 
Regenerative braking must be disconnected for speeds under 5 km/h. 
 
Even when hard braking, the regenerative braking must remain active. 
 
The commanded motor torque must be 0 Nm if hard braking and the 
accelerator pedal is at more than 25% travel. The commanded motor torque 
must remain at 0Nm until the accelerator is at less than 5% pedal travel, 
regardless of whether the brakes are still actuated or not. 
 
In case of a deviation of more than 10% for more than 100 ms between both 
values of the APPS, the motor must be shut down completely. It is not 
necessary to deactivate the complete tractive system. 
 
There must be a defined sequence to turn on the car. The car will indicate it is 
in a ready to race condition with a characteristic sound, between 80-90 dB.  
 
After the TS has been activated, additional actions must be required by the 
driver to set the vehicle to ready-to-drive mode (e.g. pressing a dedicated 
start button). One of these actions must include the actuation of the 
mechanical brakes while ready-to-drive mode is entered. 
 
The ready-to-drive mode must be left immediately when the shutdown circuit 
is opened. 
 
A delay of up to 250 ms is allowed before opening the AIRs in order to reduce 
the current in the TS. 
 
The shutdown circuit will send a frame to the master controller before 
opening the shutdown circuit. When the signal is received by the MCU it must 
let the motor spin freely so no current flows through it. An indicator must show 
the origin of the error that triggered the shutdown circuit. 
 
The master controller must make sure all devices in the CAN bus work properly 
and enter in a safe state in case it detects a misbehaviour.  
 
The MCU must log all the frames sent and received in the CAN bus. 
 
Functions that are not mandatory but that should be implemented in the 
MCU are ABS and traction control.

Apart from its specifications, the cRIO’s most relevant feature to become a MCU is its

characterized by its ability to always run a script in a specific time. Regular computing 
systems have jitter, which means there is a certain variation in the time it takes to run 
the same script. Nonetheless, real-time computing systems guarantee that the code 
can be executed always under a certain time. This feature is very interesting for real-
time applications because it guarantees the maximum time since the data is received 
until the new value is sent.

4.3 Previous code 
The 2017-2018 season was the first season for the Vermilion Racing team, and with it 
came the first version of the MCU code. A major difference from this code to this 
season’s code is that it used different modules. Instead of using the NI 9862 for CAN 
communication, it used the NI 9853. The NI 9853 provides low-level control of the CAN 
interface and has no compatibility with the X-NET library. As this module is connected 
to the FPGA, all its functions must be compiled in an FPGA VI and the Main VI can 
read the output with an FPGA communication like such as; DMA FIFOs, user defined 
variables or simple FPGA read/write controls. Another key difference is that the motor 
controller worked with CAN Open instead of CAN. Therefore, another communication 
network was present, and the MCU became the link between both networks. To 
provide the cRIO with CAN Open the NI 9881 module was added to it.   
This first code has three stages; Initialization, Operation and Close. The Initialization 
stage starts the CAN and CAN Open communication, and creates a new log file.  The 
operation stage is divided into two sections, Operational and Vehicle control – drive. 
The difference between these two sections is diffuse. The Operational stage takes 
care of the entire car’s subsystems: motor, shutdown system, AMS...  and the data 
logging. Finally in the Close stage the cRIO is shut down.  
One of the main defects of last year’s code is that it does not have a particular 
architecture. It is a combination of while loops and sequence structures that 
communicate data between themselves using different queues and notifiers. This 
worsens the code readability and makes almost impossible to scale. The lack of a 
definite software architecture and the different modules used made the master 
controller team decide to scrap last year’s code and create a new one from the 
beginning. [10]

Most of these frames are self-explanatory, but some require a description, such as the 
cRIO_Info. This frame does not belong to any ECU. This frame carries information about 
which state is the master controller in. then as the frame is echoed to the receiver, it 
can be logged in a TDMS file along all other received frames.  
Other frames work as a pair of information input and output, such as the 
Shutdown_message and the Shutdown_trigger, or are designed to create a 
handshake synchronization, like the Calibration_TX and Calibration_RX. These frames 
are always contiguous and usually carry equivalent data.  
For more information about the motor controller frames, see the RMS CAN Protocol 
document. [11] 
The CAN database also gives a quite accurate estimation of the CAN bus utilization. 
Cyclic frames are the ones that take most of the bandwidth. As event frames are 
usually sent due to human interaction, they can be left out of the calculation as they 
have a very low contribution to the bus utilization. To estimate the CAN utilization of 
every cyclic frame first is to know its DLC, and use it to calculate the number of data 
bytes. All CAN frames have a 64 bit overhead that must be added to the data bytes. 
As CAN uses NRZ3 codification, every 5 bits with the same value a new bit is stuffed 
with the opposite value. This means that for a frame with eight data bytes, the total 
minimum amount of bits is 128 and the maximum 153. The amount of bits per frame 
times the frame transmit frequency divided by the bus speed gives the time it takes to 
send that frame at the given frequency. Doing this for every frame and then adding 
the transmit times, the result is that in the worst case scenario the frame is in use 272 
ms out of every second, and in the best case scenario 228 ms out of every second. 
4.5 Code implementation 
The first key decision when making the code was to decide a software architecture, 
as this was one of the main reasons last year’s code was scraped. LabVIEW has a few 
design patterns that ensure the code will be scalable, modular, and simple. The basic 
design patterns are; Producer/Consumer, Event Handler, Master/Slave, and State 
Machine. Out of these, the fittest one for the MCU is the State Machine design as can 
describe the different states the MCU has. Producer/Consumer and Master/Slave 
designs are focused in multiple loops running in parallel performing multiple actions, 
and the Event Handler design is suitable for applications that require direct user input

In the Run state the motors respond to the pedals position, therefore the driver can 
move the car. In the run state there are some safety features implemented. In case 
the driver pushes the brake and accelerator pedal at the same time it goes into the 
No torque state. In this state the master controller sends a zero torque command to 
the motor regardless the pedals position. Only when the driver releases the 
accelerator it moves back to the Run state and the driver gets the control of the car 
back. This state is implemented as a safety feature to avoid the driver to push both 
pedals. Without this feature the driver could accelerate and brake at the same time 
inducing the wheels to skid and might lose control of the car.

In case that while the machine is in run state a critical error is received, the machine 
goes into safe state. A critical error is defined as such error that impedes the correct 
and safe operation of a device. For example, if the accelerator pedal is disconnected 
from the pedal box this will trigger a critical error because the pedal box is not able to 
work anymore. Other errors, for example the motor over temperature error, indicate 
that it is not safe to operate under these circumstances. When a device crashes it is 
also considered a critical error. In the safe state a zero torque command is sent to the 
motor, and at the same time an error frame is sent to the shutdown circuit. When the 
shutdown circuit receives the error frame it will wait for 250 ms and then open the AIRs. 
The 250 ms time window is necessary to make sure that no current is flowing to the 
motors. Without it huge currents might flow the moment the AIRs are open, and it 
would be an aggressive operation for both the AIRs and the motor controller. After 
some time the critical error may fix itself and then the machine is allowed to go into 
the idle state again. Once the driver has the car under control, can push the brake 
and RTD button again to move the machine into Run state again and keep driving. 
Finally, if the driver pushes the RTD button in any state other than the Idle state the 
machine moves into turn off state, shuts down the communication and opens the HV 
circuit.  
When a device detects an error the node sends an error frame to the CAN network 
that the MCU reads, however there is no direct way to detect when a node has 
crashed. To detect if any node has crashed all critical ECUs have a counter signal. 
This signal is used to check the devices’ status as well as diagnose the network state. 
For example, if any ECU crashes the counter will not be updated anymore, and the 
cRIO will send a Shutdown_trigger frame to open the shutdown circuit. At the same 
time the shutdown circuit expects a counter update from the cRIO, so even if the cRIO 
crashes the car will go into safe state.  Same procedure is used to diagnose the CAN 
network. If the network has any connectivity problems, open wires, corrupted data or 
a short circuit to ground or supply voltage, the shutdown circuit will not be able to 
receive a counter update and will automatically set the care in safe state.  
Other methods have been evaluated to carry out this same purpose. The first method 
was comparing timestamps. One of the features of the NI 9862 is that every time it 
receives a frame, it does not only interpret and store it but also saves the receiving 
==PAGE:37==

27 | P a g e  
 
timestamp. This timestamp can be read when in frame sessions, but not in signals 
sessions. As signal sessions are the preferred ones in the master controller code this 
idea was discarded. Another method was to read the :trigger: property. This property 
is exclusive from single-point sessions and tells whether a new frame has been 
received since the last time the code performed a read action.  The strongest 
advantage of this method is that it consumes very few resources as it is a Boolean 
variable. Nonetheless its main disadvantage is that it is not able to detect if a system 
has crashed. In the particular case when one ECU crashes and sends the same frame 
regardless the input, this method is unable to detect that the ECU has crashed, unlike 
the counter method. It is a very particular case, but taking into account that the cRIO 
hardware is not a bottleneck the resource consumption becomes less important.  
Parallel to the finite state machine there is the data logging loop. The data logging 
loop has two versions. The first and default version is implemented using Brian 
Hoover’s4 CAN I/O library. The second version is implemented using some NI examples 
as reference. Even though Hoover’s library is faster and has more features, it is a third 
party library and must be revised every time there is a new LabVIEW version. Both 
versions do the same tasks; create a TDMS file with the timestamp as name, log all 
frames received by the NI 9862 in this TDMS file, and save the file when the driver 
decides to turn off the vehicle. An extra feature is that every 30 minutes a new TDMS 
file is created. This feature pretends to avoid having massive files that take very long 
to process and are likely to crash.

4.6 Results 
After testing the code under several circumstances, all functions work as expected. 
All safety features have been tested and they work properly. At the beginning the 
cRIO run the execution loop every 20 ms. Yet it works properly, the pedal box’s frames 
do not have a constant message period. On average they have a 20 ms period, but 
every cycle is slightly over or under this value. This maladjustment increases the 
response time under worst circumstances from 40 ms (20 ms for pedal position update 
and 20 ms to process the frame and send it to the motor controller) to 60 ms (20 ms 
for the pedals, 20 ms for a lost frame and 20 ms to process the new frame).  
In evidence of the results, the cRIO execution frequency was split in half. Now it runs 
every 10 ms, and the maximum response time is 30 ms. Event though, the motor 
response will not be immediate, as the motor controller has an integrated PID 
controller. This PID controller will delay and smoothen the request signal. In case the 
team considers the response time insufficient after the road tests, the PID parameters 
must be tweaked rather than the cRIO response, as 30 ms is already unperceivable 
for humans.

5.2 Model description 
The car model in Simulink made by Hyundai is created from empirical data. As an 
input it takes several powertrain parameters like the accumulator capacity and 
power, the gear ratio, the inlet temperature of the cooling system, among others. At 
the same time, it takes car’s external parameters like the drive profile, which describes 
the required speed as a function of time in order to obtain the best lap time. The lap 
time is the objective function to minimize. 
One of Hyundai’s main concerns when designing an electric competition car is motor 
derating. This phenomenon occurs when the stator surpasses a certain temperature 
and the motor performance drastically drops. The motor maximum available torque 
linearly decreases until it reaches the final derating temperature, where it stabilizes 
again. Figure 23 shows this behavior in the model. Main factors that affect motor 
temperature are motor torque delivery, cooling inlet temperature and regenerative 
power. Using the motor as a generator increases the temperature as much as using it 
as a motor, because in both cases current flows through it and have the same power

losses. 
 
Figure 23: Motor derating region 
Another element that can limit the motor performance is the accumulator. The 
accumulator itself has a maximum output power. The accumulator SoC and the cell 
voltage also affect the maximum power. An accumulator with almost no charge left 
can deliver less power. At the same time if the cells voltage is low, the accumulator 
voltage is low and it requires more current to deliver the same power, and the 
accumulator may not be able to deliver such amount of current. At the same time 
higher currents produce higher energy losses and induce higher temperatures in the 
powertrain.  
Finally yet importantly, the motor itself must be modeled. Despite electric motors are 
known for their linear torque delivery, it is not completely linear. Maximum torque and 
power delivery have been tested in all the speed range and then created a model 
with the results.

5.3 Method 
The optimization focuses in three areas. First, is to find the optimum accumulator size 
to complete the drive cycle without accumulator derating. Second, is to find the 
optimum regenerative profile to complete the drive cycle without thermal derating. 
Third, is to study how the car’s specifications change if the drive profile is filtered to 
achieve a smother waveform. The third area is not technically an optimization but a 
comparison, but its results will help when doing key decisions.  
The first task of the optimization is to find the smallest accumulator size that guarantees 
that the car will finish the race without derating. As input of the optimization 
application the driving profile and the car model are required, as well as the desired 
maximum error of the solution. Once all the required parameters are set, the 
application runs the simulation at several points to check if the accumulator suffers 
derating starting with a minimum size accumulator. Once it finds two consecutive 
points where one point derates and the other does not, the app starts a binary search. 
It checks the middle point and if it derates, it moves the interval from the middle point 
==PAGE:45==
 
 
 
 Model Optimization 
35 | P a g e  
 
to the non-derating point, and if it does not the new interval is from the derating point 
to the middle point. This process is repeated until the interval between a derating and 
non-derating point is smaller than the maximum error asked by the user. As an output, 
the app gives the smallest possible accumulator size that does not derate. It also 
multiplies the voltage times the current of the final drive profile and outputs the 
maximum as the maximum power required from the accumulator. 
 
Figure 25: Binary optimization script 
Second task of the optimization algorithm is to find the best regenerative profile that 
allows to run the drive cycle without either temperature or accumulator derating, with 
the minimum accumulator size. The idea behind this is that as regenerative braking 
can easily be connected and disconnected, the car could use its GPS coordinates to 
know in which corners regenerative braking should be activated and in which ones 
not. Therefore the first task is to identify all corners or braking events. A breaking event 
is defined as an interval of time in which the drive cycle has an interrupted negative 
torque. Once all braking events are listed, the script grades their quality. First it 
calculates the potential amount of regenerated power in that turn by multiplying 
speed times torque, times an efficiency factor. Afterwards it calculates the 
temperature increase in that area as a function of the torque input. Finally creates a 
ratio with both values and ranks each corner by its suitability score. Finally it runs the 
simulation, activating the regenerative braking for the most ranked corner. If it is able 
to finish the drive profile without derating it stays, otherwise that corner is discarded. 
The same process is repeated for all corners. This is necessary because less ranked 
corners may be shorter and induce less temperature increase than better ranked 
ones. With this process all corners are checked, but giving preference to the best 
ranked ones. At the end of the script, the accumulator optimization script described 
in the section above runs to find the optimum accumulator size under these new 
conditions.

components. The main idea is to filter the accelerator and brake inputs to get a softer

misleading results. Using the first optimization application, we obtain the accumulator

6 Brake system plausibility device 
The Brake system plausibility device (abbreviated as BSPD) is a safety element 
required by the competition rules. The main functionality of the BSPD is to open the 
shutdown circuit when the driver is hard breaking5 and power is delivered to the 
motor(s) at the same time. Therefore it is a safety system that actuates in case of panic 
or a software bug that drives the motor regardless the pedal position.   
Rules state that the BSPD must open the shutdown circuit when more than 5kW power 
is delivered to the motor(s) and hard braking occurs for more than 500ms. Once the 
shutdown circuit is open, it can close the shutdown circuit in case the opening 
conditions are no longer present for more than 10s. All signals involved are considered 
SCS (System Critical Signals). 
6.1 Electronics design requirements 
The BSPD has three main circuits to fulfil all its requirements. Two of these circuits are 
designed to read a signal from a sensor, compare it with a threshold value an output 
high or low voltage if the signal is inside a certain window. One of the design criteria 
is to keep the BSPD as small as possible in order to make it easy to fit in small spaces. 
Small boards are also easier to place close to the sensor, which is very important 
because short wires reduce the signal vulnerability in front of EMC. Noise reduction is 
a key element. Both sensors use analogic signals as output, and EMC can lead to false 
readings.  
A pressure transducer BENXU BX-P201 is used to detect hard breaking. This sensor 
outputs an analogic voltage that is directly proportional to the pressure in the brakes 
hydraulic signal. The transfer function from the output voltage to the sensed pressure 
is: 
𝑃(𝑉) = 750 ∗𝑉−375 
 Where the voltage is expressed in Volts and the pressure in PSI. This sensor can 
measure pressures from 0 to 3000 PSI, and the output voltage ranges from 0,5 to 4,5 V. 
This small offset of 0,5 V is very useful to detect if the signal has been shorted to ground 
and the low end of the window must be placed at some value between 0 and 0,5 V. 
The higher end of the window is set experimentally using a potentiometer. This 
approach is convenient considering the lack of documentation of the pressure sensor 
and the low reliability of its manufacturer.

A Hall Effect current transducer is used to measure the current delivered to the motors. 
The sensor model is LEM HTFS 200-P. This sensor has an offset of 2,5V and the output 
voltage increases or decreases directly proportionally to the current flowing through 
it. According to its datasheet, the output voltage increases 6,25 mV/A. The reference

above 200A, however for the BSPD the 2,5 V default offset is fine for this application. 
Same as the pressure sensing circuit, the current sensing circuit reads the transducer 
signal and outputs 0 or 24 V if the signal is inside or outside a voltage window. A signal 
amplifier is the only difference with the pressure sensing circuit. This signal amplifier is 
necessary due to the hall sensor sensitivity. It is too low to measure and calibrate the 
same way as the first circuit. This low sensitivity also affects the quality of the 
measurement. Since it is that low, small EMC noise in the signal can create huge 
reading differences and the noise immunity has been one of the main concerns when 
designing this circuit.  
Both the pressure and current sensing circuit’s outputs are connected to the third 
circuit. The output of both circuits is either 0 or 24 V, so it has a high immunity against 
noise and is suitable to travel big distances in the car. This feature is very convenient 
when placing the pressure sensing circuit, because it can be placed very close to 
brake pedal at the front of the car without worries.  
The third circuit is the delay circuit, and its main function is to check if both signals are 
present and if they are for more than 500 ms open the shutdown circuit. Once the 
shutdown circuit is opened, the BSPD may reset itself if the opening condition is no 
longer present for 10 s or more. The BSPD is a mandatory part of the car, and must be 
designed according to section T11.6 of the rules book. The circuitry to implement these 
functionalities is rather simple. First an NOR gate outputs a high signal only when both 
threshold signals are low. The output of the NOR gate goes to an RC circuit that delays 
the signal. The output of the capacitor is connected to a Schmitt trigger, which 
implements hysteresis to the signal output. In this configuration, the RC circuit is

39 | P a g e  
 
charged through a small resistance and discharged through the big resistance to 
ensure the charge and discharge time difference.  
Alternative hardware was evaluated to perform the time delay. The main alternative 
was to use an IC 555 for the delay. The main advantage of this circuit is that it is 
implemented with digital logic, and that eases the test process a lot as well as it 
improves its tolerance to EMC.  The idea was rejected because it did not fulfil all the 
requirements. For example, the IC 555 in monostable configuration adds a delay to 
the output signal, but does not make a difference for how long the input condition is 
true. Then every time the trigger condition is true it would trigger the shutdown circuit, 
regardless if the input condition has been present for 500 ms or not. Another 
alternative was to use the IC 555 in astable configuration to create a square wave, 
connect the square wave to a counter and the output of the counter to a set of 
Boolean gates that change their state whenever a number of square waves has 
passed. This configuration was finally rejected because it contradicted one of the 
main advantages of digital circuits, their simplicity. The RC circuit is a well-known 
configuration with few components, which makes it harder to fail and simpler.  
Furthermore, all signals in the BSPD are SCS. This means that the hardware must be 
able to detect any signal failure and go into safe state. In this context the safe state is 
defined as output low, and the signal failures for analogic signals are; short circuit to 
ground, short circuit to supply voltage, or open wire.

6.3 PCB design 
Once it is sure that the schematic works as desired is time to design a PCB. A PCB can 
be done with either a protoboard, a custom PCB with SMD (surface-mounted device) 
components or THT (through-hole technology) components. Each of them has their 
own benefits and defects. Protoboards are good for fast prototyping, and allow a lot 
of flexibility to fix the circuit. On the other hand, they use a lot of space, and most 
connections are not properly protected, with pins and wires in the air. Custom PCBs 
with SMD components are the best in terms of size and protection. The many different 
footprints available and small pads make them become very compact boards and 
with no air wires. Their biggest downside are their very low flexibility, hard to find 
components due to the high amount of footprints available and require a heat gun 
for soldering. Custom PCBs with THT components are the intermediate option between 
protoboards and SMD boards. They have the same footprints as protoboards, so most 
components are considerably bigger than their SMD counterparts are, but they are 
easier to find and can be soldered without a heat gun. At the same time these are 
custom PCBs, which means that there are no air wires but nets that give extra 
protections and better aesthetics, but on the other hand takes out flexibility when 
modifying and fixing the circuitry. At the beginning Vermilion Racing aimed for SMD 
PCBs, but after some tests and discussions the final choice was to make THT PCBs.

7 Conclusion  
As commented in the respective results chapter of each project, the implementation 
can be considered successful as all the initial objectives are met. Only the model 
optimization has no results because it has not been possible to perform any tests. Major 
problems in the motor set-up out of the scope of this thesis delayed the motor testing

required data, analyse it, build a model, and then perform an optimization of such 
model. It is true that it can be considered as a failure; nonetheless it is not a big 
inconvenience. It is not a big inconvenience because right now the car is too unstable 
to create a model. Vermilion Racing is currently building their second car, because it 
is a very young team. Every year its member learn more and more and every year the 
car incorporates major design changes that improve its performance. With such big 
modifications going on in the car, it does not make much sense to build a model out 
of it as the model will become rapidly outdated. Once the car design becomes stable 
a mathematical model will become a very powerful tool to improve the car, but now 
this is far from reality.  
The reader must also note that all the tests have been conducted under a laboratory 
environment. The laboratory environment makes many external factors disappear, 
such as the EMC noise or atmospheric conditions (rain, fog…), and that these external 
elements might induce a malfunction in any of the devices. Unfortunately, the car is 
not ready yet to be tested under real conditions at the time of writing this thesis. Once 
the car is ready, all systems will be tested under real conditions and any device that 
fails will go into another design iteration to fins the root cause and fix it in time for the 
competition.

8 Future work 
As the Vermilion Racing team is relatively new, many aspects of their car need 
improvement. Many team members have worked very hard to achieve a working 
car, but this car is far from ideal and there is still a lot of work ahead to compete for 
the first positions in formula student competitions. Many teams have participated in 
FSUK for years, and every year the car went through another design iteration to find 
weak points and turn them into strong points. Vermilion Racing must follow this path. 
In the next lines there are some of ideas the author has gathered along his work to 
improve the car performance. 
One of the easiest problems to solve is the lack of precision of the pedals. To measure 
the position of the pedals, these are attached to a potentiometer. When the driver 
pushes the pedal, it moves along with the potentiometer. Then an ATMega uses its 
ADC to read the voltage value in that potentiometer, and transforms that voltage into 
a value from 0 to 100 using the calibration parameters. The problem resides in that the 
potentiometer does not use its full travel distance, just a small fraction of it. This in 
combination with the 10 bit ADC of the ATMega makes that the pedals have a 
resolution of 1%. Considering the pedals are actuated with the driver’s feet this is not 
critical as humans do not have as good position perception of their feet as their hands 
but even then, this resolution should be improved. The proposed solutions for this 
problem are either designing a new pedal box with some kind of transmission ratio for 
the potentiometer, where one degree angle in the pedals becomes 3 degrees in the 
potentiometer, for example. Alternatively, in case the team does not want to redesign 
the pedal box, using an audio potentiometer. Audio potentiometers have two slopes 
trying to mimic a logarithmic curve. Replacing the actual potentiometer for an audio 
potentiometer could give more resolution as in the second slope one degree of 
movement represents a bigger resistance change. This bigger resistance change 
would create a bigger voltage difference and in the end, a better resolution. 
Another subsystem that could be easily improved is the pump control. Right now, the 
pump has an open loop control so there is no feedback from the system. The open 
loop works, but it is not as efficient as it could be. Having a more efficient pump control 
would allow for a smaller LV battery, as the pump is the main power consumer in the 
LV system. To make a proper pump control a thermal model of the car is required. This 
model would take into account the radiators and motor thermal properties, and 
obtain the required coolant flow in every situation to keep the motor temperature 
always under the derating temperature. This kind of control requires many work hours 
into it, but any improvement can make a big difference as right now there is no control 
at all. 
A feature that must be implemented in the short term is regenerative braking. This 
season’s car has no regenerative braking because the mechanical components do 
not allow it, but it must be implemented for next season as it is one of the most 
attractive features of electric cars. A mixed storage is already under development to 
improve the regenerative braking efficiency. The mixed storage consists of some 
super-capacitors that store the energy generated during braking and then deliver it 
==PAGE:63==
 
 
 
 Future work 
53 | P a g e  
 
the next acceleration event. Super-capacitors are more suited for this task than 
regular accumulator cells because they have very small charge and discharge times 
and are rated for more cycles.  [13] 
The dashboard also requires a design update. This system may not be critical for the 
car’s operation but it is very useful for debugging and can give the pilot a lot of 
information about the subsystem’s status. A touchscreen would be a perfect fit for this 
task as it gives information to the pilot and the pilot can send commands at the same 
time. The cRIO supports touchscreens and it is able to send the GUI to the 
touchscreen, so designing an interface is very easy with LabVIEW and have 
contextual menus depending on the car’s state.  
Other features should be planned as medium/long term objectives. The most 
notorious one is to replace the current EMRAX motor for four in-hub motors. Having an 
individual motor for each wheel has many advantages. Since each motor can be 
controlled individually, it is possible to implement features like ABS, ESP or a more 
advanced traction control. Otherwise, ABS and ESP are not possible because brake-
by-wire is not allowed by the rules. On the other hand, their main disadvantage is that 
they add another layer of complexity for the master controller, and at the same time 
they may saturate the CAN bus.  
While the in-hub motors are not implemented, a feature that deserves to be explored 
is a CVT (Constant Variable Transmission). This kind of transmission changes the gear 
ratio automatically and continuously. This means it can be used to increase the motor 
torque at low speeds, and later decrease the gear ratio to allow a higher top speed. 
Nonetheless, this kind of systems are usually heavy and complicated so the team 
should first conduct a study to determine if this system advantages overpass the 
disadvantages. 
Once the car design is stable, that means there are no big changes from year to year 
like motors or accumulator, a very good project is to create a car’s mathematical 
model using Simulink. This model should represent both the mechanical and electrical 
main systems. With a model like this, it should be possible to calculate the best drive 
cycle with the mechanical model and if the electrical model can perform it. It will give 
an overview of what subsystems are a bottleneck in the car performance and should 
be improved in the next design iteration. This model is also very good for optimization. 
Optimizing the car model gives the best parameters values within a certain feasible 
range. These parameters could be for example: accumulator capacity, vehicle 
center of mass, or shock absorbers stiffness. An optimization would also give 
information about parameter sensibility, and how many consequences each 
parameters’ deviation from the optimum value has in the results. 
All teams in the FSUK top positions have a DRS (Drag Reduction System), and Vermilion 
Racing must have one too. The implementation of a DRS may seem easy as it only 
consists of a few servomotors coupled to the aerodynamics package and controlled 
through the master controller. However this is not true. To successfully implement a DRS 
it is necessary to have an accurate model of the tires and aerodynamics package. 
==PAGE:64==
 
 
 
Results 
54 | P a g e  
 
Otherwise there is no accurate way to find the optimum spoiler angle. That is the 
reason why, even this subsystem should appear in the Team’s design timeline, it should 
be planned as a long-term objective.  
Finally, one of the most ambitious objectives of Vermilion Racing is to create a 
driverless car. Moving to a driverless car requires a lot of effort and adding many 
subsystems, but one of the most notorious changes is the MCU. As stated in Hardware, 
right now a NI cRIO-9033 is being used. This hardware suits properly today’s 
requirements, but it is not powerful enough to control a driverless vehicle, neither 
contains the appropriate libraries to do so. Therefore a new MCU hardware is 
necessary. Along the semester some alternatives have been considered, but all of 
them have their own pros and cons. One of hardware considered is an AutoBox from 
dSPACE. This hardware can be coded using Simulink blocks and maintains the Plug-
and-Play characteristic of the cRIO. Coding the MCU with Simulink blocks will be very 
positive when the car’s model is finished, as the master controller could be added to 
the block diagram and simulate the car’s response. The main downside of the dSPACE 
hardware is its price. Other FSAE teams use boards with an ARM Cortex processor 
combined with a FPGA, a configuration very similar to the cRIO. Nonetheless this 
boards are open source and do not need to be coded with LabVIEW. There is a library 
in Simulink devoted to code this kind of boards, so they also have the advantage of 
being coded with Simulink blocks. Their main disadvantage is that they are not plug-
and-play devices and require some set-up before start to coding. The last option is to 
use a Patmos processor. This processor is developed by DTU compute, which means 
that using it would generate some nice synergies between different DTU departments. 
This processor is specially designed for Real-Time applications, which suits the MCU 
purpose perfectly. Its main downside is that it must be compiled with C, and no 
Simulink compatibility is available. As mentioned, all options have their strong and 
weak points, and the team must do a consensual and premeditated decision.

Master controller code description 
The code is divided into four main areas: start-up, FSM (Finite State Machine), data 
logging and shutdown.  
Start-up sequence 
The start-up sequence runs every time the 
cRIO powers up. The objective of this 
sequence is to start the NI 9862 module 
and put all subsystems into safe state. The 
first task is to start the FPGA as the CAN 
modules makes intensive use of the FPGA. 
This module always requires to open an 
FPGA VI even if it is empty, as there are 
many functions that are transparent to the 
developer 
but 
allow 
communication 
between the module and the cRIO. Next it starts a notifier and sets it to false. This 
notifier is used along the code to stop the master controller’s execution when there is 
an error or the driver sends a command to turn off the car. As the first notifier is false, 
all notifier read operations will return false until any loop writes a true. This only will 
happen when there is a run time error or the driver pushes the RTD button twice, 
meaning he is turning off the car. 
The second part of the start-up sequence consists on creating the XNET sessions. All 
the sessions are signal single-point but one of frame stream type, which is used in the 
data logging. Signal single-point is a session type that returns only the last value 
received for that signal. In this context, this is very useful as most of the time what 
matters the most is the last value and intermediate data points are not relevant. In 
addition, signals are more convenient than frames as they are already parsed 
according to the information in the CAN dictionary. There are six sessions in the code, 
three signal output single-point, two signal input single-point, and one frame input 
stream. The criteria to group these signals has not been the frame they belong to, but 
how are they used in the code. This practice lets the code be cleaner and more 
understandable. Furthermore, all output sessions have their echo property enabled. 
This makes the NI 9862 to listen all messages sent by itself. As it listens to its own frames, 
these can be read in a stream input session.  
The first node contain signals related to the 
output of the state machine. These signals 
are related to the motor and the shutdown 
circuit. This node is the most important one 
as it sends commands to the motor and 
triggers the shutdown circuit. It tells the 
motor which is the requested torque and 
which direction it should spin. It also enables

Finite State Machine 
The FSM is the most important loop of the MCU. It is in this loop where the state 
machine is implemented. This is loop also has most safety features. For example, it is in 
this loop when the software checks if any ECU has sent an error frame and safe state 
must be triggered, or where the rollover counters are checked making sure any 
device has not crashed.  
The state machine has four states: Idle, RTD, No torque and safe state. In the Idle state 
the master controller does practically nothing and waits for the driver to send 
commands. This is the only loop when the pedals can be calibrated. This feature is 
meant to fasten the union between the mechanical pedal box and the sensing ECU. 
It is also positive from the ergonomics point of view as any driver can easily set their 
preferred pedal travel in a few seconds.

Detecting error frames is easy. As all ECUs normal state is 0, the software must check if 
the error signal is different than 0. If true, it puts the car into safe state. To check the 
rollover counter is slightly harder but it does not consume many resources either. To 
ease comprehension this functionality has been enclosed in a sub-VI called “Has 
crashed”. This sub-VI takes an array of counters and compares them to their previous 
values stored in memory. If the new value is different from the previous one, the 
function resets a particular timer for that counter. If it is the same, the timer keeps 
going. If the timer reaches 500 ms the master controller interprets it as one of the 
critical ECUs has crashed and puts the car in safe state.   
Data logging 
The data logging loop has two versions. The first and default version is implemented 
using Brian Hoover’s6 CAN I/O library. The second version is implemented using some 
NI examples as reference. Even though Hoover’s library is faster and has more 
features, it is a third party library and must be revised every time there is a new 
LabVIEW version. Nonetheless, both libraries work in a very similar way when it comes 
to log frames, so the following description can be applied to both versions. 
 
Figure 52: Data logging loop with Hoover's libraries 
                                               
6 Brian Hoover, commonly known in NI forums as Hoovahhh, is a highly recognized LabVIEW 
developer who has done many talks in NI congresses. Even though he is not a NI employee, 
he is known to create and share library more efficient than the ones provided by NI.  
==PAGE:73==
 
 
 
 Appendix 
7 | P a g e  
 
 
The data logging has two nested while loops. The internal loop contains the VIs 
required for logging the CAN frames and the outer loop is implemented to avoid the 
TDMS file of becoming too big. The external loops starts by creating a new TDMS file. 
The name of the file is the date and time the file is created with the format yy-mm-
dd_HHMM. A VI configures the new created file. The configuration VI adds information 
about the database used, protocol and conversion settings to the TDMS file. Once 
the TDMS file is configured, the internal loop starts. Inside this loop a read VI outputs 
an array with all the CAN frames that arrived since the last read action. The NI version 
reads the values as raw frames, an array of 24 bytes for each frame. Hoover’s version 
on the other hand reads the values as a CAN cluster which contains all the data 
properly labelled. This is the main difference between both versions. Then another VI 
writes the data in the TDMS file. The simplicity of this process makes data logging highly 
efficient, yet the data is written as bytes, which means that is very impractical to 
analyse and that it requires to be parsed later on if we want to obtain any information 
out of it. At the end of the internal loop there is an elapsed time VI that ensures that 
any file will not have more than 30 minutes of data. Finally, the TDMS file data is stored 
and closed. As the master controller is deployed in the cRIO, the TDMS file is stored in 
the cRIO internal memory. The file by default is stored in /home/lvuser/natinst/LabVIEW 
Data/. 
Shutdown sequence 
The shutdown sequence is triggered before the cRIO powers off. This state can be 
reached if the driver pushes the RTD button a second time, or if the cRIO encounters 
a run-time error. The first task in the shutdown sequence is to turn the motor off and 
disconnect the HV system. To disconnect the motor the master controller sends a 
frame to disable the inverter and request zero torque. To disconnect the HV system it 
sends a trigger error to the shutdown circuit with message 255. Even though the 
shutdown trigger and the inverter disconnect are sent at the same time, the shutdown 
circuit has a 250 ms delay implemented by hardware that ensures the motor to be 
disconnected when the relays are open. This feature greatly improves the work cycles 
of the relays.

Next in the shutdown sequence, it switches the brake light off and closes the FPGA 
reference, which in its turn disables the NI 9862 and NI 9485 modules.  Finally, an error 
case structure logs the errors that happened in run time in the TDMS file, creating a 
new group for it called “Run time error”.  
Log file parsing VIs description 
As mentioned in Data logging section the master controller logs all frames in a TDMS 
file, but it logs the frames as bytes and not as signals to improve performance, which 
makes it very hard to directly read it after. That is the reason a parsing file is necessary. 
To transform the TDMS file into something more easily understandable and adapted 
to other software more appropriate for performance analysis.  
Log files created with Hoover’s library have more flexibility and are processed around 
10 times faster. This is the reason most parsing VIs are only available to log files created 
using this library. Even so, one VI has been especially developed to parse files created 
with the NI library.  
Parse TDMS for Matlab 
The purpose of this script is to read all frames in a log file, parse the bytes into 
engineering units, and display them in an adequate format. An adequate format is 
defined as a format that a Matlab script can read and easily convert into any of the 
Matlab typical extensions. As Matlab is a very good software to treat huge amounts 
of data, it is very suitable to analyze the CAN frame history of a formula student car.   
The VI’s GUI contains a file selector, an XNET database as inputs, and a CAN table 
and a number of read frames indicator as output. The only mandatory field is the file 
selector. The VI reads this file path field and uses it to open the selected TDMS file. 
Once the file is open, the load configuration VI reads all parameters of the CAN 
channel, such as the database or protocol the data is in. With the database name,

Once the new file is ready, the data parsing starts. The objective is to make one group 
for every frame, and one channel for every signal. To do so a for loop is implemented. 
Every iteration of the for loop represents one frame. Thanks to the auto-indexing 
property of LabVIEW, the number of iterations is adapted to every database 
automatically. Inside the for loop there are two areas. The first one sets the frames and 
channels properties. First every frame has its identifier written.  Then it lists all signals the 
frame has and adds their unit and comment. The comment property is useful to 
understand error frames as the comment contains the meaning of each error bit. 
Once the frame and signals have their properties set the second part of the code 
starts; the data parsing. First a conversion session is created. Even though all frames 
are fed to the conversion VI, only signals of one frame will be converted, thanks to the 
fact that the Create Session VI has an input that allows signal filtering. The conversion 
mode is XY. This conversion mode returns the data as an array of data points with 
value and timestamp. The final step is to separate them and write in the appropriate 
channel. Once the for loop ends, the TDMS file is saved and a TDMS viewer opens the 
file for the user to check all data is written appropriately.

This VI is available as an example in Hoover’s library. As the author of this thesis just 
made some changes to make it compatible with LabVIEW 18 and did not develop it, 
this explanation focuses in how the GUI works rather than the VI itself. The first 
difference is that this VI works with events, instead of being streamlined as all the 
previous ones. The users must first run the VI and later manipulate the controls, vice 
versa of streamlined VIs. Then the users select the file path, the VI opens the file and 
updates the time window and list box fields. The VI automatically loads the database 
in the TDMS file properties and throws an error if fails to do so. The time window is able 
to filter frames by their timestamp, very useful in long files. The listbox displays all signals 
present in the database, and lets the users filter the ones they are interested. Once all 
parameters are set, the users must only push the read button and all the requested 
signals will appear in the graph within the desired time range. The graph itself also lets
