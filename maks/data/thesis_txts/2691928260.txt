Abstract
In recent years e-commerce retailers have experienced an increased growth in the number of received
orders. This has led to increased pressure on the fulﬁlment houses and as a result, new innovative
systems and methods have been applied. In order to keep up with the pressure, the Robotic Mobile
Fulﬁlment System (RMFS) has become more and more popular. Consequently, this study will simulate
an RMFS and analyse its performance. The main idea of an RMFS is to achieve a high picking rate
by constantly having mobile robots feeding the pickers with the items demanded by the customers.
To investigate how the performance of the RMFS is aﬀected by implementing diﬀerent tactical and
operational rules of decision the discrete-event simulation is used.
Here one of the decision rules
focuses on the tactical level by looking at the allocation of the pickers throughout the week, and the
remaining decision rules targets policies related to the operational level. This includes a new rule for
order handling, implementations of new dwell points for the mobile robots and modiﬁcations to how
a mobile robot will be assigned to an order. The result shows that each of the policies concerning
the operational level all can lead to improvement in the performance of the system. This study can
therefore be used to get an understanding of how an RMFS can be simulated using the discrete-event
simulation and provide the reader with basic knowledge about diﬀerent aspects to consider when
setting up an RMFS.

Introduction
In recent years, online shopping has become more and more popular. In 2017 e-commerce sales were
accountable for $2.3 trillion in sales worldwide and are expected to be nearly doubled to $4.5 trillion in
2021. Furthermore, in the U.S. alone 10% of all shopping already happens online and it is predicted that
in 2040, 95% of all purchases will be made online.[1] This growth in e-commerce sales has and will cause
additional pressure for the retailers’ fulﬁlment houses due to an increased amount of customer orders
which need to be fulﬁlled fast and eﬃciently. For example, if an order is placed within a certain time of
the day, a large number of retailers oﬀer day-to-day delivery. To satisfy this requirement, it is crucial
for the fulﬁlment houses to minimise the handling time. So far, most e-commerce order fulﬁlment
houses have preferred the picker-to-part method because of the associated ﬂexibility. However, since
this requires a picker to handle each order manually by walking to diﬀerent storage locations this
method is very ineﬃcient and time-consuming.
In recent years, the part-to-picker method, which
can be achieved by the implementation of a Robotic Mobile Fulﬁlment System (RMFS), has been
implemented as a better alternative to the traditional picker-to-part method, e.g. Amazon Robotics
and Swisslog. Here, mobile robots carry inventory pods, i.e. shelves containing Stock Keeping Units
(SKU), to a pick station whereafter it can be handled by the picker and the demanded items can be put
into the customer’s basket. Thus, the system can eliminate the waste of walking time for the pickers
who instead can focus on inserting the items into the customer’s baskets. So far, an implementation
of this type of system has indicated that the picking rates may be doubled compared to the previous
method. Thus, the RMFS can become a vital part for all e-commerce retailers if they want to remain
competitive in the future.

When implementing an RMFS many diﬀerent challenges and questions arise about how this should be
designed. The system oﬀers a lot of ﬂexibility since it is not necessary to place the inventory pods at
ﬁxed positions and thus, having the ability to be placed at new storage locations throughout the day,
over and over again. Also, these pods can be found in diﬀerent shapes and have a variety of accessible
faces. In addition, the storage of the SKUs also needs to be considered since these may be placed across
diﬀerent pods to increase the likelihood of having a pod containing the correct SKU located closer to a
pick station leading to a lower travelling time. However, the inventory spread across multiple pods also
results in more replenishment for each pod which will increase the order throughput time. Another of
the main components, the pick stations, can also vary in numbers and location which can be adjusted
in the attempt to optimise the performance of the system. The last main component is the mobile
robots, which will serve the pickers. These can be very costly and thus, it will be beneﬁcial to ﬁnd the
optimal number of these in order to satisfy the customer’s demand within the desired service time.
To investigate how some of these challenges in the design of the RMFS can be dealt with the method
discrete-event simulation (DES) will be employed. This has already proven to be a useful and powerful
tool to improve the performance of complex inventory systems. These systems will often be too com-
plex to be evaluated analytically and will be very costly to implement in real life. Here, simulation can
facilitate the decision-making for companies contemplating investing in such systems and it provides
them with qualiﬁed knowledge. Thus, simulation oﬀers the best opportunity to justify the implemen-
tation or extension of a system when a company contemplating building this. Furthermore, in a world
where cyber physical systems become more prominent, integrating DES can be excellent preparation
for a future deﬁned by Industry 4.0 since the obtained data can provide crucial information such as
where the bottlenecks might occur or give exact information about the processing time for the diﬀerent
steps in the system.
1.1
Aim and Objectives
The focus of this study will be on how diﬀerent operating policies for the main components in an RMFS
can improve the eﬃciency in a fulﬁlment house. Diﬀerent scenarios will be developed and assessed
to investigate the inﬂuence on the order throughput time of a system, which includes the processes
from pods being replenished to orders being handled by a picker at a pick station. Based on this the
following research question is going to be examined:
How can a discrete-event simulation model be used to analyse a Robotic Mobile Fulﬁlment System
within e-commerce to improve the performance in terms of the order throughput time?
This problem formulation is followed by a number of sub-questions:
• How can the layout of the RMFS be conﬁgured in a simulation model?
• Which "what-if" scenarios could be relevant to investigate in order to analyse the overall perfor-
mance of the RMFS?
• How will diﬀerent policies for the mobile robots aﬀects the order throughput time in the RMFS?
In the study, we thus aim to optimise the eﬃciency of the RMFS by minimising the total order
throughput time. The problem will be handled using a DES to monitor and predict the behaviour of

1.2
Scope and Limitations
The scope and limitations for the study are elaborated in order to address the extent to which the
research area will be explored and, similarly, deﬁne what will be neglected in this study. The scope
of the study is to investigate the processes and their interconnection within the RMFS and examine
diﬀerent setups and policies to see how these aﬀect the overall performance of the system. The exact
processes are described in Chapter 3. The limitation of the study is stated below.
• This study will only focus on phase 1 (problem formulation) to phase 4 (experimental design) of
the general simulation methodology. [3]
• There will not be looked into what happens behind the replenishment station and how the SKUs
are delivered to the fulﬁlment house.
• It will not be taking into consideration how the order is further processed after it has been
completed by a picker.
• The study has been conducted without any external collaboration. Therefore, the developed
system will not be corresponding to an existing system but instead representing a system in-
cluding some of the vital processes and behaviours for an RMFS. Consequently, no real data has
been provided and thus, the data will be generated based on diﬀerent sources leading to some
limitation in the quality of the output for the developed model.
• Knowledge on fulﬁlment houses is derived from the literature study. Thus, decisions and as-
sumptions are based on this acquired knowledge and intuition rather than previous experiences.
• The exact cost of resources has been neglected and will only brieﬂy be considered in relation to
the tactical planning of an RMFS.

2.1
Simulation
With high initial costs, as well as many changes in processes and allocation of resources to consider
when implementing or extending an RMFS, it can be beneﬁcial to analyse and evaluate this in a risk-
free virtual environment to support the decision-making ahead of any investment. This can be achieved
by using simulation which often is the next best thing to observing the real system.[3] Simulation both
oﬀers the opportunity to carry out a descriptive analysis of existing or proposed systems but it can also
be used in the context of a prescriptive analysis. This means simulation can include both determining
potential bottlenecks in a current system but also a prediction of future scenarios. The disadvantage
with simulation is that the output of the developed model is only as good as the data provided to the
model. Therefore, it is required to acquire as detailed data as possible to give the best representation
of a real-life system. Furthermore, it is important to conduct a thorough validation of the model to
ensure that the simulation is representative of the real system in order to increase the justiﬁcation for
the decision-makers.
Within simulation, there are two predominant types of dynamic simulation: DES and continuous sim-
ulation. In recent years, a new technique is becoming more and more recognised, namely agent-based
simulation (ABS), which is a variant of the DES.
A DES models the behaviour of a system as a sequence of distinct events in time where each event
marks a change of state in the system. This technique is often used on the operational level for complex
logistic systems when materials, people, etc. ﬂows through a system and complex interrelationships
occur between these. In the ABS the model is based on a collection of autonomous decision-making
entities termed agents. Each agent will have the ability to assess its situation based on some deﬁned
rules and make a decision upon them. These decisions will occur at a countable number of points
in time and thus, it can be seen as a variant of DES. However, the focus of this technique is to a
larger extent on the behaviour of each individual agent instead of being on the overall operation of
the system. In continuous simulation, the state of variables will change continuously in time. A main
technique inside this type is system dynamics (SD) which is often used for designing and improving
strategies in business when studying systems with continuous ﬂows. A downside for SD is that it has
some diﬃculties in dealing with queues in a system.

As already stated in Chapter 1, this study will use the simulation type DES in the attempt to develop
the simulation model. This is chosen since the focus is on the tactical and operational level rather
than the strategic level. At the operational level, the entities move throughout the RMFS and events
will be triggered by the entities and the resources at a distinct point in time. Furthermore, the system
will consist of diﬀerent queues which are best handled by the DES. The model development will also
include a minor part from the ABS technique since each agent will have diﬀerent attributes and assess
its situation when interacting with other agents. However, it should be noticed that the focus will not
be on evaluating each agent and its behaviour.
2.1.1
Simulation Methodology
This study will follow the modiﬁed version of the DEGREE methodology for problem-solving in the
development of a simulation model. A general methodology for problem-solving consists of the following
steps:
1. Deﬁne the problem.
2. Establish measures of performance for evaluation.
3. Generate alternative solutions
4. Rank alternative solutions.
5. Evaluate and iterate during process
6. Execute and evaluate the solution.
The DEGREE methodology provides good guidance in order to solve problems, however, some mod-
iﬁcations are needed to encapsulate the unique steps that must be performed in simulation when
interfering with the overall problem-solving process. Thus, the reﬁned general methodology for apply-
ing simulation to problem-solving is divided into six new steps deﬁned by Rossetti[3] and is illustrated
in the below ﬁgure.

future changes to the system, it is important to have some tangible measurements when evaluating the

2.2
Simulation Languages
When seeking to imitate or simulate the behaviour of a real-world system it quickly becomes too
complex to obtain an analytic solution through the use of mathematical methods. Thus, the use of
computer simulation becomes a vital factor when real-world systems are evaluated analytically. Using
computers to simulate allows for evaluating systems numerically, and in the attempt to estimate the
true characteristics of the system genuine or realistic data are gathered. To translate a system into a
computer model involves a complex logical structure which, roughly speaking, can be obtained through
two approaches. Either through implementing the model from scratch in a general-purpose language
such as C/C++, Java and Julia or by using one of the diﬀerent simulation modelling software tools,
such as ArenaTM and AnyLogicTM.[3]

towards programming while others are drag and drop oriented, and it usually applies that languages
that are more ﬂexible in representing complex situations also requires more awareness and work to
account for the logical setup of the model.[3]

2.2.2
Simulation in general-purpose programming language
One of the greater advantages of simulating in a general-purpose programming language is the greater
ﬂexibility compared with certain simulation software, and the required execution time is, in general,
lower in the general-purpose programming languages. For this study, Julia has been chosen as the
general-purpose programming languages for the development of the DES. Julia is fast and suitable
for performing intensive tasks and includes various mathematical libraries, data manipulation tools,
and packages for general-purpose computing. Furthermore, Julia is a known language from previous
work and thus, preferred above languages such as C/C++ and Java. Julia has the package SimJulia[6]
which is a discrete-event process-oriented simulation framework but due to its limited ﬂexibility, this
package is not utilised nor further clariﬁed.
As aforementioned DES relates to the modelling of a system that evolves over time and where the
state of the system changes at given points in time. These changes in the state of the system occur
when a given event takes place. Because of the dynamic nature of DES models, it is necessary to
keep track of the current simulated time so it is known when these diﬀerent events occur, and it is
possible to advance the simulated time from one event to another. The current simulated time is known
as the simulation clock and the advancing of this can be accomplished by two diﬀerent approaches;
ﬁxed-increment time advance and next-event time advance.[7]
2.2.2.1
Fixed-increment time advance
With the ﬁxed-increment time advance approach, the simulation clock is advanced in increments of
precisely ∆t time units. Each time the simulation clock is advanced with ∆t the model checks if an
event should have occurred in the previous interval. If this is the case, then this event is set to occur
at the end of the time interval and the state of the system is updated. This approach is illustrated in
the ﬁgure below:
Figure 2.2: Fixed-increment time advance [7]
Here the arrows represent the advancing of the simulation clock and ei represents the time the event
actually occurs. In this example, the ﬁrst event occurs in the interval between time 0 and ∆t, which
then means the ﬁrst event is considered to occur at time ∆t. No events are considered to occur in
9
==PAGE:17==
Emil Rendbæk Nygaard s164524
Theis Schäfer Gregersen s160691
time 2∆t, nonetheless, the simulation clock is advanced to this time and the model needs to check for
potential events, before advancing the time further. Event 2 and 3 are considered to occur at time 3∆t
and this approach continues until the end of the simulation end is reached. One of the disadvantages of
this approach is the postponement of the events which occurs due to the events ﬁrst being considered
when the simulation clock has been incremented by ∆t. Furthermore, an appropriate choice of value
for ∆t is necessary, so neither the simulation clock is advanced by too large increments in time and nor
too small which would lead to an increase in the execution time. Because the actual times between
the occurrence of events can vary to a great extend this ﬁxed-increment time advance is rarely used
for DES. It should be noted that it is possible to implement the ﬁxed-increment time advance when
using the next-event time advance by creating events and set them to occur at every ∆t time.
2.2.2.2
Next-event time advance
By using the next-event time advance approach the simulation clock is advanced to the time of the
occurrence of the most imminent event of the future events. Then the state of the system is updated
and the times of the occurrence of the future events is updated at this point as well. The simulation
clock is then again advanced to the occurrence of the most imminent event and an update of the system
is again conducted and this is repeated until the stopping condition is reached.
In contrary to the ﬁxed-increment time advance inactive periods are skipped since all increments of
times is dependant on an event’s occurrence, which potentially can reduce the overall execution time.
Figure 2.3: Next-event time advance [7]
The ﬁgure above illustrates how the next-event time advance approach is carried out for a single-

of occurrence ti. Ai is the interarrival time between two orders arrival and Si is the serving time

therefore occurring when the time of A1 has pasted. Since the server is idle the serving of order 1 is
initiated immediately at this point in time. After A2 has past order 2 is received at time t2 which
represents the occurrence of event e2 and t2 is computed as t1 + A1. Since the server still servers order
1 the serving of order 2 can not be initiated at this point, but instead needs to enter a queue. Order
1 is then completed and is represented by event e3 and this point in time can be calculated as t1 +
S1. This also means the server is idle again and therefore can initiate the serving of order 2. The next
event e4 is the arrival of order 3 and because the server is busy completing order 2, order 3 must wait
being served. The last event e4 in this short example is the completion of order 2, which occurs at the

of t1, t2, c1, t3 and c2. The next-event time advance approach is the most widely used approach for

2.2.2.3
Components of a DES model
When applying the next-event time advance approach to a DES model it is necessary to identify the
characteristics of the model’s diﬀerent events and then establish a set of event routines for each of
these and how each changes the state of the system diﬀerently. Therefore, DES models usually have
common components and when using the next-event time advance approach when programming in a
general-purpose language the following components are usually found:
• System state: The state variables needed to describe the overall system.
Each entity of the
system is characterised by its own attributes, which brought together compose the entire state
of the system.
• Simulation clock: The variable representing the current value of time simulated
• Event list: A list with the time of occurrence for each event
• Statistical counters: The variables storing statistical information about the system e.g. utilisation
of the diﬀerent resources
• Initialisation routine: Subprogram initialising the system at time 0
• Timing routine: Subprogram used for determining which event to occur next
• Event routines: Subprograms used for updating the system state when the diﬀerent types of
events occur. There will be one event routine/subprogram per event type
• Library routines: Subprograms generating random observations from predetermined probability
distributions.
• Report generator: Subprogram producing a report at the end of the simulation which uses the
statistical counters to present the desired performance measurements of the system.
• Main program: The main program used to invoke the timing routine and depending on the
occurrence of the next event invoke the appertaining event routine. The main program should
also check for termination of the simulation and at the end invoke the report generator.
Law [7] has tried to illustrate how the logical relationships (ﬂow of control) between these components
can be set up in the ﬁgure below:

Figure 2.4: Controlling the next-event time advance in a general-purpose language[7]
The simulation is initiated by the main program invoking the initialisation routine which then sets the
simulation clock to 0 and initialises the system state, statistical counters and the event list. After the
initialisation routine has been conducted the main program invokes the timing routine to determine
which event is next to occur and the simulation clock is advanced to the time of occurrence of the
given event. When the type of the next event has been determined the main program invokes the event
routine according to the type of that event. Law argues that the event routine typically consists of
three activities: (1) An update of the state of the system corresponding to the occurrence of the given
event; (2) an update of the statistical counters; (3) an update of the times of occurrence of future
events. After the event routine has been invoked the main program should check if the simulation
has reached a predetermined stopping condition. If this is not the case the main program repeats the
invoking of the timing routine and the event routine until the simulation end is reached. The main
program should then invoke the report generator and the simulation is then terminated.
This ﬂow of control illustrated in Fig. 2.4 is typically applied when programming DES in general-
purpose programming language and is also known as the event-scheduling approach since the times of
occurrence of future events are explicitly coded into the model and thus scheduled in the simulated
future.

2.3
Literature review
A review of previous studies regarding an RMFS and similar systems will now be conducted. Due to
the novelty of the RMFS, the literature in this speciﬁc research is very limited. However, the intro-
duction of the RMFS has led to a large number of new research question inside the warehouse theory
and thus, this area has increasingly come to the attention of the academic community in recent years.
This literature review will focus on the high variety of challenging problems which is discussed in the
literature to illuminate the complexity in the system and the importance of narrowing the scope for
this study.
Lamballais et al. is one of the ﬁrst to model the RMFS[8]. In 2016 they published a paper whose
main contributing was to gain insight into the system which includes accurate driving behaviour of the
mobile robots and multi-line orders based on some developed semi-open queuing networks (SOQN).
These were used to estimate and evaluate the performance of the system and the utilisation of the
mobile robots under various system parameters, fulﬁlment house layouts, and control policies. The
performance was measured using three metrics; order throughput, average order cycle time, and mo-
bile robot utilisation, and it was shown that their analytical method using queuing models accurately
estimated the performance metrics. Furthermore, it was shown that the combination of the setup
of the fulﬁlment house layout together with the use of storage zones had an inﬂuence on the order
throughput time and order cycle time. Some notable limitations were also mentioned in the paper,
which may be relevant for further work. The models do not include the congestion between the mobile
robots, which will have an impact on the driving time. However, the authors argue that this will
only have a minor impact on the result since the system is designed with the aim to increase the pick
rates. Thus, the picker will always be busy and there will always be mobile robots in a queue at
the work station and the additional seconds, which congestion will cause, will be neutralised by these
waiting times. Furthermore, congestion is very unlikely to appear since the number of mobile robots
often is small relative to the size of the fulﬁlment house. The second limitation concerns the mobile
robot switching between pick stations which is not explored in this paper since the focus was on the
design area whereas this aspect could be relevant when looking at some operational decisions inside
the system. Tessensohn et al. have in 2020 extended their model in a new article[9] which reveals how
three key decision variables in the RMFS; the number of pods per SKU, the ratio of the number of pick
stations to replenishment stations and the replenishment level per pod, can be optimised and how this
impact the throughput performance of the system. The article concludes that the order throughput
time decreases as the SKUs are scattered among diﬀerent pods, when an optimal ratio level between
the number of replenishment and pick stations is found, and when the replenishment will be carried
out before it is empty.
While Tessensohn et al. tries to optimise decisions variables on the tactical level Gharehgozli and
Zaerpour, in a paper from 2020, studies the operational problem of scheduling for pod retrieving in
order to reduce the total travelling time of the mobile robots.[10] The paper models the problem as
a generalised asymmetric travelling salesman problem (GATSP) and is using an adaptive large neigh-
bourhood search (ALNS) heuristic to solve this. It can be seen that this metaheuristic can be used
in the decision making of selecting a pod to decrease the throughput time of customer orders. Fur-
thermore, it is also concluded that an increasing number of storage locations in a fulﬁlment house
lowers the travel time of the mobile robots while adding general precedence constraints in order to
13
==PAGE:21==
Emil Rendbæk Nygaard s164524
Theis Schäfer Gregersen s160691
prioritise the customer orders increases the travel. A similar problem is tackled by Boysen et al., in a
paper from 2017, [11], which treats the order processing in a mobile robot based order picking problem
(MROP), i.e. the sequence of picking orders and the pods brought to a pick station in a part-to-picker
system using Kiva Systems (now know as Amazon Robotics). This study has high similarities to the
problems which arise in an RMFS and the previously know Kiva Systems is also often mentioned as
an example of this in other literature studies. The paper reveals that an optimised order processing by
using a decomposition procedure leads to a much more eﬃcient system where more than half amount

The more spread the SKUs are among the pods and the higher the variety they have, the fewer pod
movements are needed in the order processing.

Deﬁne Simulation Project
This chapter will focus on the ﬁrst phase of the general methodology for applying simulation to problem
solving. This includes a deﬁning of the problem and the system to gain a common understanding of
the scope of the study. In order to be able to evaluate the problem in the future decision-making,

3.1
Deﬁne the problem
As an e-commerce retailer, it is of general interest to oﬀer a high service level to their customers at
the lowest operational cost. This means that the retailers must provide an eﬀective delivery policy but
with the lowest cost of resources as possible. Thus, it is desired to ﬁnd a good trade-oﬀbetween the
resources supplied and the service level oﬀered and the objectives of this simulation study, therefore,
becomes as follows:
• Investigate how a change in the order handling policy can aﬀect the systems order throughput
time.
• Examine the eﬀect of applying diﬀerent policies for the operation of the mobile robots.
• Comparing the performance measurements for the diﬀerent scenarios.
• Investigate the inﬂuence of the allocation of pickers in relation to the systems’ performance.
These learning objectives can be related to either the tactical level or the operational level.
The
sequence of which the orders are handled belongs to the operational planning and thus can vary from
day to day or even hour for hour. The deﬁned system will use the FIFO method to decide which
order will be served next, however, this can potentially weaken the overall performance since a pod
may be used for multiple orders already in the system. Furthermore, the eﬀect of applying diﬀerent
policies for the mobile robots will be investigated. This also belongs to the operational planning and
it will be examined how a new dwell point policy, seeking to minimise the distance to a pod at any
given time, aﬀects the performance measurements. The allocation of pickers relates to the tactical
planning and since the salary varies throughout the week and there usually are weekend allowance, it
will be interesting to investigate if it would be possible to allocate the weekend shifts without having
a negative eﬀect on the performance measurements.

3.2.2
Processes
The RMFS consist of many diﬀerent processes where diﬀerent resources are required to perform various
activities which change the state of the system. These activities have been divided into eight main
processes and will be elaborated below.
Order receival
The process ﬂow is initiated when an order enters the system. An order can contain one or more items,
and it is possible that the order demands more items of the same SKU, which means that the model
will be treated as a multi-line order handling. The time of the order receival corresponds to when the
customer places the order and the order information enters the system.
Mobile Robot to Pod
After an order has entered the system a pod containing the demanded items are located together with
an idle mobile robot. In this setup, the system will locate the pod which can satisfy the most items
of the given order and in case of more pods being equally suitable, the pod with the lowest ID is
17
==PAGE:25==
Emil Rendbæk Nygaard s164524
Theis Schäfer Gregersen s160691
chosen. Subsequently, an idle mobile robot will be selected, and the model will here choose from all
idle mobile robots and pick the one that has been idle for the longest time. The selected mobile robot
will now be assigned to this order and will ﬁrst be released when the order is completed. Hence, if
more than one pod are required to satisfy the order the same mobile robot will be used to pick and
collect all the pods. However, there is an exception to this rule, which will be elaborated in Section 4.2.
Pod to Picker
When a mobile robot and pod have been selected to serve an order, it is checked if any of the pick sta-
tions are idle which the mobile robot can transport the pod to. Each pick station has a predetermined
queue capacity where pods can wait before being handled by the pickers. The pickers can only operate
on the pod right in front of them, leading to minimising the walking distance and thus, heighten the
eﬃciency of the picking rate.
Figure 3.3: Top view of a pick station
When a picker is assigned to an order this picker has to complete the order before the queue of the
pick station can be updated and the next order’s item can be picked from the pod waiting in line.
When choosing which pick station should be used for serving the given order, it is looked if any of the
pickers are idle. If more than one of them are idle, then the picker, who has been idle for the longest
time, will be chosen. If all of the pickers are occupied, the pod will be delivered to the ﬁrst idle pick
station with the lowest picker ID.
Retrieve SKU
When a pod is delivered to the pick station located just right in front of the picker, the picker will
retrieve all the demanded items from the pod. If the pod contains all the demanded items, the order
will be completed after this activity and it will therefore leave the system. The pods have a capacity for
each of their containing SKUs and each time an item is retrieved from the pod, this SKU’s inventory
level will decrease by one and if this gets below a certain amount the pods need to be reﬁlled, which
will take place at a replenishment station.
Replenishment
After having retrieved all the demanded items from the pod, it will be checked if the pod is required
to go to the replenishment station to be reﬁlled. This process will be triggered if the level of items for
one of the SKUs in the pod is lower than a predetermined minimum replenishment level.
This can be set individually for each SKU in each pod which will allow fast and slow movers of SKUs
to be replenished at diﬀerent rates. However, this replenishment will not allow for new lines of SKUs
18
==PAGE:26==
Emil Rendbæk Nygaard s164524
Theis Schäfer Gregersen s160691
to be stored in the pod, and the pod is, therefore, reﬁlled with the exact same SKUs as it was initiated
with. When the pod is delivered to the replenishment station all of its SKUs will be reﬁlled to their
maximum capacity, even though they still are above the minimum replenishment level. The handling
time for reﬁlling the pods will correspond to the handling time for the retrieving time of the picker.
Return Pod
When the pod is done with either serving the picker or has been reﬁlled at the replenishment station,
it is returned to its ﬁxed storage location. If the mobile robot is not triggered by a new activity, then
it will wait here until needed for a new order, meaning that this is the dwell point of the mobile robots.
Charging of Mobile Robot
The last process in the system, which may occur in the process ﬂow, is the charging of the mobile
robots. According to Wulfraat[13] mobile robots are charged at a frequent interval throughout the day.
The Mobile robots receive a ﬁve minutes battery recharging for every couple of hours at a designated
charging station and then returns to the production line again. This is implemented in the model by
checking if a mobile robots’ last recharging time exceeds two hours and if so, then the mobile robot
will drive to its charging station and recharge there for ﬁve minutes before becoming idle again. In this
setup, it is decided that the system only will check for recharging if the mobile robot has completed
an order, which is intended to reduce the later deﬁned active part of order throughput time. Wulfraat
[13] mentions that mobile robots, which are included in a consistent daily usages, typically requires
battery change-out within 1.5 - 2 years, and due to a much smaller simulation time this change-out is
not taken into consideration when building the model.
Scheduling of Pickers
The data analysis, which is conducted in chapter 4, shows that the amount of placed orders ﬂuctuates
throughout the day and that the least amount of orders are placed within the night hours. Conse-
quently, it is decided to incorporate a schedule of the pickers, where the entire day is divided into three
working shifts of eight hours each, leading to the following division:
• Shift 1: 00-08
• Shift 2: 08-16
• Shift 3: 16-24
Therefore, a number of pick stations have to be closed for the night, and the handling of the orders
currently located at a pick station need to be considered. It has been decided that the order being
served by the picker needs to be completed before the station can shut down and the orders located in
the pick station queues are returned in the system. This is implemented such that the orders, which
are returned, are getting their information restored, meaning that no mobile robot or pod are located
to them and their time is restored to the time of their receival. Furthermore, the pods are returned
to their storage location. To ensure these shifts take place at the desired point in time the method
ﬁxed-increment time advance is used, where events of type Shift have been created for every eight
hours.

3.2.3
Entities & Resources
In the system, interactions between the entities and the diﬀerent resources continuously takes place
and it is, therefore, necessary to have attributes for each of these which makes it possible to record
their current state at any given time of the simulation.
The entities in the system are the orders, which are placed by the customers and the resources, which
are necessary for serving the orders, are the pickers with belonging pick station, mobile robots, pods
and a replenishment station. It applies to all resources that they have an ID, current status, time
and location, and the orders have an ID and current status. For the orders it is necessary to know
exactly which items are demanded and also when the order was placed by the customer, to ensure
the correct items are packed and shipped within an appropriate amount of time. Furthermore, it is
necessary to record when the order is completed but also when a mobile robot starts driving the ﬁrst
suitable pod for the order down to the picker. This will be used for measuring one of the later deﬁned
KPIs in Section 3.3. As aforementioned each picker has its own pick station with a given number of
queue stations to it. It is, therefore, necessary that the pickers’ attributes also capture the information
for these, and some of the main attributes are, therefore, the status of the pick stations and their
locations. When updating the queue at the pick station, it is found convenient to record the order
ID of the current orders located at each station. The attributes for the mobile robots, which diﬀers
from the other resources, are the recording of the time since the mobile robot last visited its charging
station and the coordinates of this station. It goes without saying, these are used to determine when
and where the mobile robot needs to be recharged. For the pods, it is necessary to know which SKUs
they contain and the current inventory level of these. In addition to this, it is also necessary to know
the maximum capacity of the pod’s SKUs, which is used for the purpose of reﬁlling. Since the pods
have a ﬁxed storage location, it is required to know the start location of the pod. The replenishment
station’s attributes do not diﬀer from the other resources and will not be elaborated further. A more
detailed overview of the attributes for the entities and the diﬀerent resources are given in Appendix B.
3.2.4
Travel times
A central aspect of the model is how to calculate the travel distance for the processes, which involve
re-positioning of the mobile robots and pods. For the processes Mobile Robot to Pod and Charging
of Mobile Robot the travel of the mobile robot can be categorised as unloaded since it does not carry
any pod. The travel distance for unloaded travel can be calculated using the Manhattan distance,
reasoned by the fact that the mobile robots can drive beneath the pods. However, the calculation of
travel distance for the processes including loaded travel becomes more complicated. The unit distance
corresponds to the width of a storage location in the horizontal direction and length in the vertical
direction. Once the travel distance is known, this must be divided by the speed of the mobile robot
to obtain the time for the distance travelled. Here it will not be considered how the mobile robot
will have to accelerate or decelerate for taking a turn, and thus, it is assumed that it will drive at a
constant speed.
3.2.4.1

that deadlock will not occur and reduce the congestion between the mobile robots which carry a pod.[8]

The KPIs is found to identify potential bottlenecks and assess the performance of the system. The
average resource utilisation will contribute to a better understanding of the eﬀectiveness of the chosen
distribution of resources and can indicate if these are the reason for bottlenecks. The average utilisation

system is. The order throughput time is the total time it takes to pass through the various processes
needed to complete an order. The lead time for each of these individual processes is measured and

the system performs and works as a good benchmark when evaluating the diﬀerent scenarios. This
performance measurement is further divided into two categories; passive and active. Where the passive
order throughput time represents the time from an order enters the system and starts being handled,
and the active part represents the time from an order starts being handled to it is completed. The
passive part, therefore, constitutes the time from the order enters the system until a mobile robot
initiates the delivery of the pod to a pick station, whereas the active parts are measured as the time
from a pod is being delivered to the pick station and until the order has been completed and leaves the
system. This implementation makes it possible to detect more accurately where and to what degree
further improvements impact the system.

Figure 3.7: Flowchart of the deﬁned RMFS
As illustrated in the ﬂowchart the entities of the system are the orders and the ﬂow is thereby initiated
when an order enters the system. When an order has been received an idle mobile robot drives to
the pod which is best suited for the given order as described in Section 3.2.2, and the mobile robot
then elevates the pod so the transportation can start. The mobile robot delivers the pod to an idle
pick station and depending on the pickers status, the retrieval of items from the pod can begin. If the
picker is busy serving another order the pod is delivered to the ﬁrst idle pick station and moves up the
line each time the picker completes an order. When the demanded items have been retrieved from the
pod the ﬁrst of three checkpoints are met. If the order is satisﬁed with all its demanded items then
the order is completed, and it will leave the system. No matter if the order is completed or not, the
following part of the ﬂow is executed. The system then needs to check if the pod needs replenishment
of items and if this is not the case, the mobile robot will return the pod to its storage location. If
replenishment is necessary the mobile robot will instead deliver the pod to the replenishment station
where it will be reﬁlled. When the pod has been replenished the mobile robot returns it to its storage
location. After the pod has been returned, it is checked if the mobile robot needs charging, which only
can occur if the order is completed. If the mobile robot needs charging it will drive to its charging
station and recharge for ﬁve minutes. If the order is not completed after the picker has retrieved the
necessary items these steps are repeated until the order is completed and leaves the system.

3.5
Document Model Assumptions
The ﬁnal step of the ﬁrst phase of the general simulation methodology is the documentation of the
model assumptions. This ensures all assumptions made during the modelling process is captured and
well understood by all potential parties. Furthermore, it facilitates the process of late examination
of the eﬀect of these on the model by modifying some of them during the veriﬁcation and validation
phases.
• Robot never breaks down, meaning that maintenance of these are neglected.
• The dwell point policy for the mobile robots is when the pod has been returned and the Point Of
Service Completion (POSC) has occurred, which means the mobile robots do not need to travel
to a predetermined dwell point after an order has been completed.
• No priority of orders. Follows FIFO methods, meaning that the model do not consider Day-to-day
delivery, standard delivery, etc.
• Retrieving time and time of replenishment is equal. It take the same time to remove one item
from the pod as it takes to put one item into the pod.
• The time of preparing a box for a new order is not taken into consideration, nor the time of
wrapping up the box.
• No distinction is made between the physical size of the order, which means the pods capacity of
each SKU do not vary.
• Each mobile robot has its own charging station which is located at the robot’s starting location.
• When the system is initialised all pods are fully loaded and the mobile robots are fully charged.
• When an order ﬁrst has started being served by the picker, the picker must complete this order
before continuing on to the next order.
• When a mobile robot has been chosen to satisfy a given order this order must be completed
before the mobile robot can be used for another order.
• The mobile robots can travel anywhere in the fulﬁlment house when not carrying a pod.
• Mobile robot velocity is constant. This also includes when the mobile robot will have to accelerate
or decelerate for taking a turn.
• The time of pod lifting and storage are constant.
• Mobile robot congestion or blocking in aisles will never happen.
• The shift of pickers do not require any additional time and the new picker will just continue from
where the previous picker left.
• The model only have knowledge about the orders which already have been completed or are
currently in the system.
• Failures within the system do not occur. E.g. items will not fall down from pods.
25

Theis Schäfer Gregersen s160691
• The retrieving time is per SKU and not per item, due to the assumption that the time of retrieving
a single item and the time of retrieving two identical items are more or less equal.
• No small small breaks inside a shift will be incorporated for the pickers.

Simulation Model Building
The following chapter will follow the second phase of the general simulation methodology. This involves
collecting input data, building and coding a simulation model in a general programming language which
is followed by verifying and validating the behaviour of the model. First, a description of the collected
data will be carried out, which should clarify the quality of the input data. Next, the ﬂowchart will

4.1.2
Input data modelling
The foundation of the data generation was presented in the previous section. However, in the presented
data only the average is revealed, and thus, the expected stochasticity related to it is omitted. This will
not correspond to a real-life system where it must be assumed that there will be stochasticity in some
of the inputs of the system. To handle these, input modelling will be carried out where probability
distributions will be selected to represent this randomness in the input random variables. It should be
noticed that there is no "true" input model for any stochastic input, however, the aim is to obtain the
best approximation that encapsulates the key characteristics of the underlying input process. Since no
historical data is provided, it will not be possible to ﬁt probability distributions to the data. All the
gathered information will be used for constructing an input model which is converted into a readable
form in Julia.
SKUs in pods
The top view of the physical setup of the fulﬁlment house has already been presented in section
3.2. This includes a total of 900 pods, each containing four shelves, resulting in 3600 unique storage
locations. To ﬁnd a realistic trade-oﬀbetween storage locations and the number of SKUs in the system,
there has been taken inspiration from a real-life fulﬁlment house using Kiva Robots. Here a fulﬁlment
house in Penn Township has been found, which has 12,000 SKUs distributed among 3,500 inventory
pods giving a ratio of 3.4 between the number of SKUs compared to the numbers of pods.[13] Thus,
it is decided that the input data for the simulation model will consist of the same ratio between these,
leading to a total of 3060 SKUs. Furthermore, it needs to be speciﬁed how these should be distributed
in the pods. No data was found concerning this area, and thus, some decisions about how to storage
the SKUs in the pods has to be taken. It is assumed, that some SKUs are more popular than others,
and thus, these will be placed into more pods to minimise the driving distance and also minimise the
risk of waiting for an SKU to become available if a pod, containing the speciﬁc SKU, is kept busy.
Since it has been chosen to not implement storage zones in the deﬁned layout of the storage area, all
SKUs are randomly placed into a pod. This also ensures that the SKUs which has a higher frequency
in pods are more likely to be placed near a pick station or to be near an idle mobile robot at any
given time. For the distribution of how many pods a unique SKU occurs in, the PERT distribution
is used. This is deﬁned by three parameters; the minimum, most likely and maximum. These are the
same parameters as in a triangular distribution, however, the PERT oﬀers a smoother shape, which is
desired since only a few SKUs is assumed to be sold at a much higher rate than the remaining orders.
Due to limitations of distributions available in Julia, the programming language RStudio has been
used to generate a PERT distribution for the probability of SKUs being inserted into a pod.

4.3
Veriﬁcation
Before the experimental design can be initiated, it is of most importance to determine whether the
developed model accurately represents the actual system. The ﬁrst step, in accordance to the general
simulation methodology, is to verify whether the model performs as intended. Rossetti [3] emphasises
the importance of performing model debugging, whereas Law [7] introduces eight diﬀerent techniques
which supplemented by each other forms the veriﬁcation of a simulation model. Technique 1-4 have
been utilised in the process of verifying the developed model.
Technique 1
The ﬁrst technique recommends starting with a moderately detailed model and then gradually increase
the complexity, instead of developing an immediately complex model, which could lead to a tedious
and extremely diﬃcult process of determining the location of potential error in the program. Thus,
the model was initially written only containing the main program and a couple of the key subprograms
with simpliﬁed details. Additionally, a continuous process of implementing further subprograms and
gradually enhancing the level of detail in the programs was carried out, where each further addition
was followed by a debugging if required.
Technique 2
Technique 2 advises having more than one person reviewing the computer program, preventing the
writer of a particular subprogram from getting stuck when debugging or wrongly being convinced that
a statement is correct. Each subprogram has therefore been reviewed by more than just the main
writer of the particular program to ensure both agree on the diﬀerent subprograms and their eﬀect on
the simulation model. In this study, it applies that almost none of the subprograms have one main
writer since many of the programs, in some manner, depends on each other, and it has therefore been
necessary to get familiar with each of them.
Technique 3
This technique states that the simulation model should be tested under a variety of input parameters
and then ensure that the diﬀerent outputs are reasonable and behave as expected. The model has,
37
==PAGE:45==
Emil Rendbæk Nygaard s164524
Theis Schäfer Gregersen s160691
therefore, been run with various resource settings seeking to both stressing the system but also ensur-
ing that an increase of resources resulted in an improved model output. The system was furthermore
stress-tested by increasing the demanded orders. The demand has been increased by 150% in order
to test the system to it limits, and thus, ensure that it will not reach a deadlock. The demand has
been increased by a reasonable percentage, where the output acted as intended. Furthermore, by also
testing the system to its limits by increasing the demand by 150% it was ensured that the system
would not reach a deadlock.
Technique 4
It is here dealt with the problematics of debugging a DES program, and Law [7] states that one of the
most powerful techniques for this is the so-called trace. By incorporating a trace the programmer is
meant to display the state of the system after each event has occurred, and the exact content to display
can vary from case to case. The general purpose for this is to enable the programmer to compare the
state variables or certain statistical counter with hand calculations to see if the system behaves as
intended. This has been implemented for each subprogram, so the changes in the system’s state for
the diﬀerent calls of event could be controlled and veriﬁed. This technique has, amongst other things,
resulted in the clariﬁcations of the need of statement four in the process of determining, which event
to be executed next. By displaying the state of the system after the execution of each event, it was
made clear that the system could reach a deadlock if an order was being served by a picker and the
next pod, which the order needed before becoming completed, was busy serving another order. This
has been circumvented by incorporating that an order, which is initiated and currently handled by
a picker, always can take over a pod serving another order not being handled by a picker yet. Even
though this technique has its clear advantages for debugging codes, it also has some shortcomings. It
is not always clear which information to display to highlight the error when the simulation is run, and
some particular errors might not occur in the short simulation run. This can entail that the simulation
has to be run multiple times and therefore, can this technique become quite time-consuming.

5.1
Scenario 1
The deﬁned system in Chapter 3 serves as the As-Is scenario and represents a simple version of the
RMFS containing basic decision-makings. The As-Is scenario is usually corresponding to the current
setup of an existing system. However, since it is not the case for this study, scenario 1 is based on the
most vital ﬁndings from the literature study and allows for further improvement. Before conducting
the simulation experiments some factors related to the simulation output needs to be considered; the
types of statistical variables, starting condition and the period of time over which the decision are made.
The simulation experiment is conducted by setting the input parameters and hereafter running the
model. This results in continuous changes in the state of the system where diﬀerent statistical counters
are generated, and these will be summarised in an output report, when the simulation terminates. This
process is equivalent to a one sample path, where the system starts from its initial condition and termi-
nates when it reaches its ending condition. This can also be referred to as a replication. A simulation
can be either non-terminating or terminating, depending on whether there is a well-deﬁned ending
time or not. In this study, the system operates continuously, and thus, acts as an inﬁnite horizon with
no natural termination point. However, since it is possible to determine a period of time over which the
order arrival repeats itself, a clear ending point can be applied. From the ﬁndings in Section 4.1 this
cycle is identiﬁed as a week. The termination condition is, therefore, set to if either all orders, received
within that week, has been completed or if the simulation time exceeds the time corresponding to a
week. Furthermore, some initial conditions used to start the simulations will be considered. To avoid
the problem of the initialisation bias problem the concept of a warm-up period will be incorporated
into the simulation. This should avoid any of the ﬁrst orders having a negative bias on the performance
measures and it will be more likely that the collected simulation output is more representative over the
long term for the system. Thus, it is decided that an extra Sunday, will be used as the warm-up period.
A simulation experiment can be considered as a single sample path or it can be based on multiple
replications, from where the statistical counters can be computed. Since the underlying randomness
of the simulation can be controlled, it is possible to make the repetitions of the simulation indepen-
dent, and thus, the statistical measurements sampled across multiple replications can be considered as
independent and identically distributed. The statistics collected across multiple replications are called
across replication statistics and these are applied to analyse the simulation output for the diﬀerent ex-
periments. To determine the number of replications needed, before the distribution of the performance
measurements reaches the point where it is suﬃciently similar to the desired steady-state distribution,
diﬀerent methods can be applied.
5.1.1
Determining the Number of Replications
Since each replication is independent the statistical counters for a single sample path can diﬀer greatly
from the true characteristics for the model, and thus, relying on a single replication can lead to
erroneous conclusions and potentially resulting in the basis for the decision making to be insuﬃcient.
To prevent this, the performance measurements should be computed on the basis of unbiased estimates,
and in this section, a practical approach to obtain this will be presented and subsequently supported
by a more analytical approach.

Scenario 4
In this scenario, the eﬀect of a new order handling policy will be investigated. As shown by the sce-

In this system, it is assumed that the model only has knowledge about the orders which have either
already been completed or currently are in the system. This scenario is, therefore, trying to gain some
beneﬁts from knowing which orders are currently in the system. This is done by letting the pickers
initiate a new order if they just have completed an order and the new order demands the same items
as is in the pod located at the picker. This means that orders taking advantages of the new policy,
skips the processes Mobile Robot to Pod and Pod to Picker. Consequently, the time it would have
taken to completed these steps will now be removed from the throughput time for the given order.

order and completed it in the same step, without having the mobile robot deliver a new pod to the

contains at least one of the items demanded by the new order.

5.5
Scenario 5
In the previously presented scenarios, only one robot has been assigned to an order, which seems like a
very ineﬀective approach when many of the orders contain two or more items in it. This is also reﬂected
in the eﬀective workload for the pickers, which must be fair to assume are far below a satisfying level
and reveals the weaknesses of the current system. Therefore, the eﬀect of assigning two mobile robots
to an order, if needed, will be investigated. This should eliminate much of the wasted time for orders
which require more than one pod. Furthermore, the scenario will allow investigating if the mobile
robots can be a bottleneck in the system.
The implementation of this scenario will require changes for the event structure in the coding where
two new attributes, representing the second pod and mobile robot, will be added. Furthermore, many
of the subprograms need adjustments to deal with the new policy for assigning two mobile robots
which had led to large modiﬁcations in the code. These modiﬁcations will be elaborated by giving
a description of the overall behaviour of the system and the decision-making for assigning an extra
mobile robot to an order. The system will now be able to assign two mobile robots to one order if one
of the two following situations is fulﬁlled: (1) if the queue at the pick station is empty and the pod
for the order, therefore, can be driven directly down to the picker without having to enter the queue
or; (2) if the order moves up the queue and now are being served by the picker. Furthermore, it will
only be necessary to assign two mobile robots, if the order requires more than one pod to be fully
satisﬁed. For the orders not handled by a picker, no extra robot mobile can be assigned since these
will be held back by the orders being served in front of them. For orders received where there will
be no queue in the system two robots and two diﬀerent pods satisfying diﬀerent SKUs in the order
will be assigned immediately. The mobile robot with the lowest time will deliver the pod to the pick
station, while the second mobile robot waits at the second pod’s storage location. When the picker
has retrieved the items from the ﬁrst pod, this will be returned, and the next pod will be delivered.
This is implemented so that when the mobile robot initiates the returning of the pod from the pick
station, the second mobile robot will initiate the delivery of the next pod. If more pods are needed
to satisfy the order, the ﬁrst mobile robot will then be assigned to a new pod or otherwise, it will be

6
Discussion & Outlook
In this chapter, a discussion of the results from the experimental study in the previous chapter will be
presented. This includes the choice of methods and the intentions behind the setup of the system and
how these have contributed to the overall understanding of an RMFS. Furthermore, the collection of
input data will be reﬂected and relevant aspects to look into for future work will be put into perspective.
6.1
Method & Simulation Languages
A simulation model is a representation of a real-life system and is based on a collection of diﬀerent sets
of rules. This is used to investigated what would happen in real-life situations and how changes will
impact the system being studied. The simulation model is only as good as the rules used to develop
it, and it can therefore be complex to create an entirely realistic system since the set of rules are based
on knowledge about the given system or research. One of the main disadvantages of simulation is that
even though the model is made as realistic as time and technology allow it, the simulation model will
never become the real thing. Law[7] emphasise the importance of including so-called subject-matter
experts as guidance to determine the level of detail incorporated into the system. These experts are to
be included when the most important components of the system are to be determined and thus, neces-
sary to be more careful about when building the model. As stated earlier, knowledge about fulﬁlment
houses is derived from the literature study, meaning that determining the most important components
of the system are based solely on the acquired knowledge, which has led to some important factors
might have been omitted. For example, the decision of how to determine when and how a mobile robot
should recharge were initially implemented diﬀerently but reasoned by later ﬁndings this was changes
and incorporated in a new manner. Thus, preknowledge or experiences with the system will facilitate
and lead to great beneﬁts in determining the level of detail but also eases the overall validation of the
model. It goes without saying, that when using simulation there will always be a natural time limita-
tion on the development of the model, either if it has to do with the cost of developing a simulation or
if there is a ﬁxed end time for when the results from the model need to be accessible. This is of course
an important factor that also needs to be considered when the level of detail is determined which also
has applied for this study.
One of the more important decisions to make, before beginning the development of the simulation
model, is to determine which simulation language to conduct the model in. Here it is necessary to
ﬁnd a suitable balance between the ﬂexibility of the program and the diﬃculties of applying it. If
this is not obtained, it is possible that the simulation model produces erroneous results or that the
model might not even will be completed. Law [7] states that one of the most important features of
any simulation language is the modelling ﬂexibility, meaning the ability to tailor the complexity of
the model so it matches the complexity of the system being simulated. The ﬂexibility enclosed in a
general-purpose programming language was exactly one of the reasons for the choice of using Julia
64
==PAGE:72==
Emil Rendbæk Nygaard s164524
Theis Schäfer Gregersen s160691
as the program used to develop the simulation model. However, with the greater programming ﬂexi-
bility, some disadvantages also follow. When building the model with the use of simulation software,
the diﬀerent features, required to develop a simulation model, is automatically provided and this will
lead to a signiﬁcant decrease in time used on programming. E.g. the drag and drop feature provided
in AnyLogicTM facilitates the process of setting up the overall process ﬂow of the system, and their
interconnection is visually illustrated by the model itself. Exactly the visualisation of the process ﬂow
is one of the beneﬁts of using a simulation software. A simulation software with built-in animation,
which displays the dynamical change of position of entities and resources while the simulation model
evolves over time, is one of the reasons for the increased interest in using such packages when building
simulation models. The opportunity of applying animation is more than just a cool feature and can
indeed ease some of the steps in phase 2 of the general simulation methodology. By displaying how
the entities and resources interact and change position, it can potentially be captured if one of these
does not act as intended and might unintentionally get caught in the system. Thus, it can assist in the
process of debugging the model, and it can even highlight exactly where to look at in the coding of the
model since the animation is connected to speciﬁc places in the model. In this study, the debugging
revealed to be a recurring hurdle and more time consuming than initially expected. Since the use of
animation has not been an opportunity with the use of a general-purpose language, the trace method
quickly proved to be the preferred approach when debugging the code. But as mentioned in Section
4.3 this method requires that correct information are being asked to be displayed, and this was not
always a known factor. Thus, the initial information displayed was often very vague and it required
many repetitions of reruns, which each time led to more precise details of where the problem might
have occurred. The inability of using animation also led to a more complicated process of validating
the model since details of the process ﬂow inside the model were "visualised" by text output which not
has the same level of explanatory force.
The ﬁfth phase of the general simulation, which is not applied in this study, deals with the document
programs and report results, and these are very important if the simulation model is to be used in a
corporate coherence. The documentation consists of a technical and a user manual, and these con-
tribute to the reusability and maintenance of the model. It applies, that a model build in a simulation
software is, in general, easier to maintain and modify than a model developed in a general-purpose
programming language. For an external, it would be quite tedious and time-consuming to familiarise
themselves with each step of the code in this study, and it would therefore be required to produce a
well documented technical manual.
These reﬂections are all necessary to take into consideration when determining which simulation lan-
guage to use, and it is important to counterbalance the pros against cons for the speciﬁc situation and
its circumstance.
6.2
Data Collection
The collection and generation of data have been a vital area for this study in order to obtain good
output data that represent a real system to the fullest extent possible. As Rossetti[3] states, the output
of the developed model is only as good as the data provided to the model, and therefore, a thorough
collection of information and data for an RMFS has been conducted.
In Chapter 4 it was elaborated that the input data modelling was constructed from diﬀerent infor-
65
==PAGE:73==

Theis Schäfer Gregersen s160691
mation gained from various sources since no historical data could be found and the study has been
conducted without any external collaboration. This has obliviously led to some drawbacks for the
developed model which needs to be considered when evaluating the results from this study. The col-
lected information about the orders is mainly found from the source XMLShop where data for over 3,5
million orders are included in their statistic. However, these data are collected from all their users of
the system, and thus, the average statistic is generated from e-commerce shops which can vary greatly
in size, product types and product range. The size will have a high inﬂuence on the order arrivals since
it must be expected that larger e-commerce retailers will receive a larger number of orders. Therefore,
a fulﬁlment house with a similar arrival ﬂow of orders and size was found, and thus, the order arrival
rate seems realistic in relation to the size of the deﬁned system. However, some larger limitations
for the data can be found in the input data for the number of items per order. It must be assumed
that the product types and range for the e-commerce retailers have a high inﬂuence on the number of
items a customer buys on average per order. For example, e-commerce shops that are selling expensive
watches will be expected to have fewer items in an order per customer, compared to shops selling
clothing. Therefore, the type of e-commerce shop will have a high impact on this random variable in
the deﬁned system. Since this information has been omitted in this study, it is important to take this
into consideration when evaluating the results.
Another aspect of the input data is the parameters set for the layout. These are, as earlier described,
based on an existing system similar in size. However, when analysing the output analysis, it is indi-
cated that the optimal trade-oﬀbetween the resources and the service level oﬀered has not been found.
In the As-Is scenario, it can be seen from Figure 5.13(a) that the mobile robots have a utilisation close
to 90% during the busiest periods from Monday to Friday. However, in these intervals, many of the
mobile robots will be placed at the pods’ storage locations waiting for one of the queues at a pick
station to become idle. By taking into consideration that the average order throughput time will be
higher than the actual travelling time down to the queue, it could be argued that the ﬂeet of mobile
robots may seem a bit too high. To investigate if the same service level could have been obtained
with fewer mobile robots a grids search, similar to the one performed in Section 5.1.3, could have been
conducted. However, it was found more relevant to investigate some of the operational levels, leading
to this not being examined further. When optimising the As-Is scenario, it appeared that the system
performed way better when it was allowed to assign two mobile robots to one order. Here the utilisation
of mobile robots decreased relatively much, revealing that in an eﬀective system fewer mobile robots
can be used to for serving the current order arrival rate. The number of mobile robots, therefore,
relies heavily on the eﬀectiveness of the use of mobile robots, which indicating that the optimal num-
ber of these needs to be considered when implementing an RMFS or making changes to the current one.
Regarding the distributions applied on the processes to describe their behaviour over time, no historical
data or information was found. This aﬀects how accurate the simulation represents a real system since
the distributions will be based on the best guess, rather than actual data obtained through observing
a real system. To achieve the best guess, diﬀerent modelling situations has been investigated and
based on this, a distribution for the process has been selected. For example, the Poisson distribution
is often ﬁtted on situations where it is desired to model the counts of occurrence in an interval while
the triangular is used when a minimum, maximum and most likely value can be found. Thus, these
were used to the distribution of order arrival rate and pickers retrieval time, respectively. However, if
historical data was been given or found, it would have been better to ﬁt a distributional model to the
data, which can be performed through various tools oﬀered in diﬀerent program languages.

6.3
Model Results
By a further examination of the initially developed model, it was deduced that the system and the
setup hereof had its apparent limitations, and the overall eﬃciency did not seem rather impressive.
One of the main reasons for this is the level of complexity that has been integrated into the system.
It will be unrealistic to believe that an actual RMFS would have the same simplicity to it, which also
is illustrated by the extend of improvement the alternative scenarios induced. However, the developed
scenarios investigated how diﬀerent implementations and policies aﬀected the As-Is scenario and in
situations where such implementations have not been incorporated, the model results indicate it could
be proﬁtable to do so.
From the results, it was deduced that each of the presented scenarios, except scenario 3, induced im-
provements to the As-Is scenario where the implementation of having two mobile robots operating on
a single order, yielded the greatest improvements for the average order throughput time. Furthermore,
it could be seen that combination of these scenarios proved to have the highest performance. How
these scenarios complement each other has not been fully detected by the ﬁnal results, due to the
extensive impact of scenario 5. Since the system in scenario 7 handles the orders almost as quickly as
they enter the system, the implementation from scenario 4 will no longer contribute as much as before.
To utilise the feature of allowing the picker to initiate a new order, if it contains the same items as the
order just completed, it is necessary that the system has at least a minimum of uncompleted orders in
it and the more the merrier. Thus, to see how these diﬀerent implementations supplement each other,
it could have been relevant to increase the pressure on the system to expose their full potential.
By only looking at the utilisation of the resources in the As-Is scenario, it would have been believed the
pickers were the bottleneck in the system. However, from further examination of the eﬀective workload
and the process ﬂow, it was deduced that the actual bottleneck was the use of the mobile robots and
the policy of how to assign them to an order. This discovery did indeed highlight the importance
of establishing relevant and informative KPIs. The eﬀective workload of the pickers and the mobile
robots showed that the model was not as eﬀective as ﬁrst interpreted. This is due to some of the
rules and assumptions made at the beginning of the model development. It was decided that a picker
only can serve one order at a time and the queue at the pick station should be assigned to diﬀerent
orders. This has led to much wasted time in the system because the picker was required to wait on
the mobile robot delivering a new pod. This problem was sought to be solved by incorporating the use
of two mobile robots for orders needing more than one pod. As already mentioned, this improved the
system performance to a large extent, but the eﬀective workload still proved to be relatively low. The
core purpose of the RMFS is to obtain as high pick rates as possible, and thus, in theory, the mobile

robots should be able to feed the pickers with a new pod as quickly as the pickers retrieve the items.
However, to obtain this it is important to create a system where the use of mobile robots together
with the order handling policies never will become a bottleneck in the system.
6.4
Future Work
This study has focused on several tactical and operational decisions inside an RMFS but as described
in the literature review, see section 2.3, there is a high variety of challenging decisions problems, and it
will not be possible to cover them all in one study. For future work, diﬀerent interesting problems could
be relevant to investigate, where some of them will be related to how the model could be improved
and some will relate to relevant research areas.
Due to a natural time limit for this study, some features have been left out in the developed model.
It was decided that a picker only could handle one order at a time, however, this will not always be
the case in an RMFS. In some of the studies from the literature review, a picker is able to handle
more than one order at a time which will increase the eﬀective workload since the picker does not has
to wait on a mobile robot delivering and returning pods. This would have been another approach to
optimise the As-Is scenario signiﬁcantly instead of allowing two robots per order. However, this will
require major changes in the code and structural adjustments, but it could be a relevant extension
to the model. Another aspect that is not tested is the eﬀect of applying a non-ﬁxed storage location
policy where pods can be placed more favourable depending on how regularly it is used. This is one of
the advantages the RMFS oﬀers, and hence, it could be relevant to implement this into the model to
investigate how a non-ﬁxed storage location can improve the performance of the system. This could
be implemented by continuously register the popularity of the pods and then placing the most popular
pods closest to the pick stations. Similar, an incorporating of storage zones will also contribute to a
better understanding of how a more favourable placement of pods will aﬀect the system’s performance.
Here various numbers of storage zones, the height-width ratio and storage policies inside these can
be applied. Aside from the ﬂexibility in the location of pods, the RMFS can also be very ﬂexible in
the use of the other resources. Here the system can dynamically increase and decrease the number of
mobile robots and pickers used depending on the pressure on the system. For example, mobile robots
can be added to the system quickly and picker stations can be opened and closed as needed. It should
be noticed that there will be regulation for a minimum working time for the pickers which needs to
be considered if a dynamical closing and opening of pick stations are applied to the system. Lastly,
the ﬂexibility can also relate to the placement of SKUs in pods. Since the pod can be driven to a
replenishment station, SKUs can easily be replaced with other SKUs. Therefore, it can be beneﬁcial
to conduct regularly forecasts so SKUs often ordered together will be placed in the same pods. This
could be relevant if campaigns or high discounts were oﬀered for some items, resulting in higher sales
for these particular items in a short period of time. Also, some products must be assumed to have a
higher correlation for being ordered together, and here diﬀerent approached as for example clustering
methods could be applied to identify these. However, this will require real data to capture these pat-
terns properly.
More general decision problems which go beyond the RMFS can also be examined. At the opera-
tional level, it involves determining scheduling the pick order sequence and the arrival sequence of the
pods to pick stations. By optimising these, it will be possible to reduce the overall travel time in the
68
==PAGE:76==
Emil Rendbæk Nygaard s164524
Theis Schäfer Gregersen s160691
system. Since the information about an order is unknown until the order has been received, it will
only be possible to schedule orders already in the system and thus, these decisions problems will have
the largest eﬀect when multiple orders are in the system. For the scheduling of the order sequence
and their sequence of arriving at the pick stations, theory from the Asymmetric Travelling Salesman
Problem (ATSP) can be applied. By attempting to ﬁnd the shortest route between the pods which
are required to satisfy the orders and their items within, a large reduction in the travel times can be
achieved.
At a tactical level, the number of storage locations per SKU could be an area to investigate. The liter-
ature review has already revealed that a higher diversion of SKUs in pods will lead to lower travelling
times, since this will increase the probability for a pod to be located close to a picker. In relation to
this, it could be interesting to explore how this would inﬂuence the replenishment of the pods since a
higher diversion of SKUs in pods would increase how often the pods will require replenishment.

7
Conclusion
The growth in e-commerce sales has led to increased pressure on the fulﬁlment houses and as a result,
new innovative systems and methods have been applied, seeking to cope with the raised amount of
customer orders. The retailers have experienced an augmented demand for fast delivery, and this can
become a pivotal factor when the customers decide where and whether to place their order. Thus, the
fulﬁlment houses are required to increase their eﬃciency and improve the order throughput time, with
the aim of separating themselves from the masses or might even just stay in business. One method
to obtain this, and which has proven to be eﬃcient, is the part-to-picker method and this can be
achieved by the implementation of a Robotic Mobile Fulﬁlment System (RMFS). When implementing
an RMFS many diﬀerent challenges and questions arise, which can not be answered by solely intuition
or mathematical models. It has, therefore, been decided that discrete-event simulation will be applied
in order to investigate the eﬀect of implementing diﬀerent tactical and operational decisions variables
in an RMFS. Simulation oﬀers the opportunity to analyse and evaluate diﬀerent alternatives to a
system in a risk-free virtual environment, and thus, this framework is well suited for exploring the
eﬀect of various settings and rules when seeking to improve the service level of an RMFS. To explore
these eﬀects, it is necessary to develop a model representing the system, and here the reﬁned general
methodology for applying simulation to problem solving by Rossetti[3] has been used. The model
has been translated into a computer simulation program where the general-purpose programming
languages, Julia, has been employed.
This formed the basis for the investigation of the aim and
objectives for this study, and here six main "what-if" scenarios have been established to evaluate the
RMFS’ performance. Whereas one of the scenarios focused on the tactical level by looking at the
allocation of the pickers throughout the week, the remaining scenarios targeted policies related to the
operational level, which included a new rule for order handling, implementations of dwell points for
the mobile robots and modiﬁcations to how a mobile robot will be assigned to an order. The scenarios
coping with the operational level all led to improvements in the performance of the system. It could
be deduced that the implementation of allowing multiple mobile robots serving a single order at one
time induced the largest improvement with an increase of 97% in the total average order throughput
time, corresponding to a reduction from 1 hour and 13 minutes to 2 minutes and 12 seconds. This
huge increase in the performance of the system has revealed the high level of simplicity for the initially
developed model which the alternative scenarios have been compared with. The new rules and policies
have been combined into one scenario to investigate how these complemented each other and whether
their interaction would improve the performance further. Here the greatest improvement has been
achieved with an average order throughput time of 1 minute and 59 seconds. Despite the increased
eﬃciency of the system, the eﬀective workload of the mobile robots and pickers indicated that further
model improvements can be obtained. This study has thus, illustrated how a Robotic Mobile Fulﬁlment
System can be simulated by use of discrete-event simulation and how diﬀerent "what-if" scenarios can
be applied to analyse and evaluate the established performance measurements.
For further work, the extension to the model which is found to be most interesting to investigate is

Theis Schäfer Gregersen s160691
the implementation of allowing a picker to handle more than one order at a time since this will be
expected to have a signiﬁcant eﬀect on the system’s performance.

Structure of Entities and Resources
A detailed overview of the attributes for the entities and the diﬀerent resources and entities are
presented here.
Orders
The attributes for the orders are:
• ID: The ID of the order.
• SKUs: An array containing the items which the order are missing before becoming completed.
• Current SKUs: An array containing the items which is currently satisﬁed in the order.
• Time of Receival: The time the order enters the system i.e. the time the order was placed by
the customer.
• Time of Completionn: The time where the order is fully satisﬁed and completed by the picker.
Corresponds to when the order leaves the system.
• Time of Start Activity: Records the time when an mobile robots start driving the ﬁrst suitable
pod for the order to a picker. This is used for one of the deﬁned KPIs in section 3.3
72
==PAGE:80==
Emil Rendbæk Nygaard s164524
Theis Schäfer Gregersen s160691
• Status: The current status of the order. Starts as Received and becomes In Progress when a
mobile robot and a pod is located to it, and when the order is ready to leave the system it
becomes Completed.
Pickers
As aforementioned each picker has its own pick station with a given number of queue stations to it. It
is therefore necessary that the pickers attributes also capture the information for these, and thus, the
attributes becomes as following. The pickers attributes
• ID: The ID of the picker.
• Status: The current status of the picker.
• Location: An array containing the coordinates of the picker’s location.
• Time: The current time of the picker.
• Station Location: An multidimensional array containing the coordinates for each of the stations
locations.
• Station Status: An array containing the current status of each of the stations
• Station Order ID: An array containing the order ID of the current order located at the given
station, which is used when the queue at the pick station is updated.
• Station Time: An array containing the current time for each of the stations.
Mobile Robots
The mobile robots has the following attributes:
• ID: The ID of the mobile.
• Location: An array containing the coordinates for the current location of the mobile robot.
• Status: The status of the mobile robot
• Time: The current time of the mobile robot
• Charge Station: An array containing the coordinates of the mobile robot’s charging station.
• Time Since Last Charging: A recording of the time since the mobile robot last has visited the
charging station.
Pods
The attributes for the Pods are:
• ID: The ID of the pod
• Start Location: Since the pods have a ﬁxed storage location this array contains the start location
of the pod, which is the location it must be returned to after serving the picker.
73
==PAGE:81==
Emil Rendbæk Nygaard s164524
Theis Schäfer Gregersen s160691
• Current Location: An array containing the coordinates of the pod’s current location.
• SKUs: An array containing information about which SKUs the pods are carrying
• Status: The current status of the pod.
• Time: The current time of the pod.
• Capacity SKUs: An array with the maximum capacity of its containing SKUs.
• Current Capacity SKUs: An array recording the current inventory level of the SKUs in the pod.
Replenishment Station
The replenishment station has the following attributes:
• ID: The ID of the replenishment station. Since this study only examines setups with one replen-
ishment station this attributes becomes redundant, but if the fulﬁlment house becomes much
larger and it suddenly becomes necessary to implement an extra station for replenishment it will
become important to know its ID.
• Status: The current status of the replenishment station.
• Location: An array with the coordinates of the replenishment station’s location
• Time: The current time of the station.
Event
The event has the following attributes:
• ID: Correspond to the order ID
• Type: Current type for the event, e.g PodToPicker, ReturnPod, etc.
• Time: The current time of the event.
• PodID: The pod ID for the pod assigned to the event.
• MobileRobotID: The mobile robot ID for the mobile robot assigned to the event.
• PickerIDID: The picker ID for the picker assigned to the event.
• PickStationIDID: The pick station ID for the pick station assigned to the event.
