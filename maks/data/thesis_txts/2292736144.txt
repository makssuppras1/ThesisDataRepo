Summary (English)
The aim of the thesis is proposing a Model-Centric API description language,
referred to as Bluem, that can be used for synthesis of client code, server code
and documentation. Bluem is a specialized version of the API Blueprint.
The API Blueprint is a documentation-oriented description language for web
APIs.
The API Blueprint can be used with a wide range of tools, includ-
ing tools for creating server mock-ups and generating client SDKs. The API
Blueprint have a well speciﬁed syntax that allows the description of arbitrary
APIs. However, the API Blueprint does not provide help for creating RESTful
APIs. Bluem implicitly provides help for creating RESTful APIs, by providing
an even higher-level of abstraction than the API Blueprint.
A prototype has been developed, which is capable of parsing and analyzing a
Bluem description. The prototype has support for plugins that can be used for
synthesis of code. Three plugins have been made as a proof of concept. The
three plugins are a client plugin, server plugin and a documentation plugin.

Chapter 1
Introduction
APIs are widely used for easy communication between servers and clients.
Google, Facebook, Netﬂix, Spotify, Twitter, Amazon Web Services and the list
goes on, they all rely heavily on their APIs. APIs are developed to be con-
sumed by other applications, either internally in the organization or externally
by third-parties. Regardless of the intent, the API should provide a way for a
consuming developer to get information on how to make use of the API. Often
the APIs have a whole website dedicated to documenting how to consume it.
The API can also be discoverable by the use of hypermedia links consisting of
a relation and an URI.
When an API is updated, new features might be introduced and in some cases
it might also break compatibility with existing clients. The documentation for
the API will also be in need of an update to reﬂect the changes made. Clients
created for consuming the API also need to be updated and depending on the
popularity of the API, the amount of clients needed to be updated might be of a
substantial size. The amount of tedious maintenance can be greatly reduced, if
the documentation and clients could be generated automatically. This is doable
by using an API description language.
The purpose of this thesis is to research the possibility of modifying the API
Blueprint description language, such that it gains support for strongly typed
models and data validation. Based on the modiﬁed speciﬁcation it should also

be possible to automatically generate documentation and documented server
and client code for various programming languages and frameworks.
1.1
API Description Languages
An API description language, is used for writing a formal description of the API.
A bonus for using a well deﬁned description language, is the often accompanying
tool support, which in most cases include automatic code generation of client
code and documentation. The formal description deﬁned using a API descrip-
tion language also works as a contract between the server and client. The server
must implement the interface outlined in the formal description.
The client
know, based on the formal description, how to communicate with the server.
Two of the ﬁrst commonly used description languages, were the Web Services
Description Language (WSDL) and the Web Application Description Language
(WADL). Both descriptions are written using the Extensible Markup Language
(XML). Originally the Web Services Description Language was developed for use
with SOAP based system [CCMW01], but was updated to support REST based
system.
The Web Application Description Language had support for REST
based systems initially. XML is a machine readable document format written in
plain text.
When the JavaScript Object Notation (JSON) and YAML Ain’t Markup Language
(YAML) started to increase in popularity, new description languages using these
formats were proposed. The two that got the most attention was Swagger and
RAML. Both have support for describing RESTful APIs. Due to the increasing
popularity of these two description languages, tools to generate client SDKs
and server mock-ups were developed.
Businesses, such as Mashape [Mas15]
and Mashery [Inc15], that provide services based on the description languages,
started to appear. The services include automatic generation of client SDKs,
interactive documentation and facilities to conduct automated tests. Both JSON
and YAML are data-oriented machine-readable formats that are written in plain
text.
The newest kid on the block, is the API Blueprint. The API Blueprint uses
Markdown as the format for the description.
The API Blueprint originated
from Apiary [Api15e], a company focusing on providing interactive documen-
tation, coding samples, server mock-ups and facilities to perform automated
tests. Apiary created the API Blueprint as an alternative to machine-oriented
description languages to allow developers and non-developers to contribute with
the description of an API. The API Blueprint is starting to become a popular
==PAGE:19==

3
alternative to Swagger and RAML, due to, not only the the tools provided by
Apiary, but also the community contributed tools.
The API description languages have some things in common. They can be used
for describing APIs of an arbitrary architectural style, even if the language is
intended for describing a speciﬁc architectural style, e.g. REST. They have sup-
port for specifying the payloads used by the endpoints, some can even include
validation rules or an entire validation schema, that dictates type and value
constraints.
All but the API Blueprint, have a description format intended
for machine consumption and does thereby not invite non-developers to con-
tribute. The description languages all support some form of reuse of previously
deﬁned entities, however their support for reuse of validation rules and valida-
tion schemas could be improved, e.g. by the use of models that incorporate the
validation schema as part of its description.
1.2
Model-Centric Description Language
Writing an API description is not the same as programming an API, hence
well-known programming paradigms may not be applicable.
The proposed
Model-Centric description language, henceforth referred to as Bluem1, is in-
tended to incorporate the concept of Model Driven Architecture [Poo01] into
the API Blueprint.
With Model Driven Architecture, platform-independent
models are deﬁned prior to any coding. With the Model-Centric description
language, the focus is to be on deﬁning strongly typed models, which can be
validated and used for the communication between the server and the clients.
1.3
Project Goals
The aim for this project is to research the possibility of modifying the API
Blueprint, such that it becomes Model-Centric and support:
• Describing an API in a human-readable language agnostic format.
• Describing resources, their URI templates and their supported actions.
• Describing URI template variables and query parameters, including vali-
dation rules based on types and constraints.

• Describing models and their properties, including validation rules based
on types and constraints.
• Documenting of all the entities included in the speciﬁcation.
Further, the project also aims at creating a prototype capable of performing the
following actions based on a Model-Centric API description:
• Parsing and analyzing, to detect syntax errors, circular dependencies and

• Synthesis of documented client SDKs, that includes all speciﬁed entities.
• Synthesis of documented server code, that includes all speciﬁed entities.
• Synthesis of documentation, describing all the speciﬁed entities.
1.4
Contribution
This thesis analyzes and identiﬁes problems with the API Blueprint speciﬁcation
pertaining to creating RESTful APIs. The thesis proposes a new Model-Centric
description format, Bluem, that attempts to solve the identiﬁed problems. A
parsing and analysis library is created for the Bluem description format, along
with a tool that, based on a parsed and analyzed description, can generate client
code, server code and documentation.
1.5
Structure
The rest of this thesis is organized as follows. Chapter 2 gives basic conceptual
information of web APIs.
Chapter 3 introduces the API Blueprint descrip-
tion language and describes the parts that are related to this thesis, including
identifying problems and a discussion of alternative description languages and
tool support. Chapter 4 describes Bluem and the changes made from the API
Blueprint. Chapter 5 compares the original API Blueprint with Bluem. Chap-
ter 6 describes the steps involved in the synthesis of code based on a Bluem
description. Chapter 7 details the prototype created as a proof of concept for
the thesis. Finally, Chapter 8 will summarize and conclude on what have been
achieved with directions for future work.

Chapter 2
Web APIs
This section will give a brief introduction to the concept and purpose of a web
API. Two architectural styles for APIs will be described. A way to classify the
maturity of web based systems will also be explained.
2.1
Basics
The Application Programming Interface, or API, deﬁnes a set of entry points
into the underlying piece of software, that can be used to exchange data. The
interface can also deﬁne the data formats which it expects. There is a distinction
between using APIs deﬁned by software locally on the computer and when using
web APIs. The local APIs are often invoked by calling a speciﬁc method deﬁned
by the interface using parameters of speciﬁc types which are also deﬁned by the
interface. When using web APIs, the method and data must be communicated
in some format and interpreted by the Web API. The communication between
the client consuming the web API and the web API is often carried out using
the HTTP protocol. The client will issue a request to the Web API and expects
a response in return.
==PAGE:24==
8
Web APIs
2.2
Architecture
The way to architect a web API can depend on the intended purpose for the
API. Two of the more common architectural styles are Remote Procedure Calls
(RPC) and REST.
2.2.1
Remote Procedure Calls
RPC can be built on top of a variety of technologies, more commonly it is
build on top of the HTTP protocol [WPR10]. When built on top of the HTTP
protocol, the communication is carried out using just a single HTTP verb, the
POST verb. RPC uses the HTTP protocol merely for transportation of data. The
content of a RPC request describes the intended purpose of the request. The
server receiving the request will have to handle and understand the contents of
the request. RPC systems are considered to be level 0 system in the Richardson
Maturity Model [Fow10], see section 2.3.
2.2.1.1
RPC Protocols
The focus of Remote Procedure Calls is to provide a way for applications run-
ning on diﬀerent operating systems to communicate via the internet. Three of
the most prominent protocols used for performing Remote Procedure Calls are
Plain Old XML (POX), XML-RPC or Simple Object Access Protocol (SOAP)
[WPR10, Lan10]. All three protocols uses HTTP as their underlying transport
protocol. The HTTP POST verb is used when sending requests, as the data for
the RPC is encoded as XML and included as the body of the request. The for-
mat of the XML varies depending on the chosen approach. POX and SOAP uses
data models deﬁned in namespaces to convey how data should be structured.
With XML-RPC, a predeﬁned data model is not included in the body. Instead
XML-RPC deﬁnes an interoperable lowest-common-denominator type system
that can easily be mapped into a data structure [WPR10].
2.2.2
REST
REST, REpresentational State Transfer, is an architectural style inspired by
the web, with focus on the concept of resources. With resources, a client is not
only asking a web server for something, the client is also identifying the speciﬁc
==PAGE:25==
2.2 Architecture
9
resource it is requesting. The web server can then choose to honor the request
or return an error message.
REST is most commonly built on top of the HTTP protocol, even though this
is not a requirement [New15]. Using the HTTP protocol is an obvious choice, as
the capabilities of HTTP ﬁt well with the REST style, due to the HTTP verbs
already having a well deﬁned meaning when working with resources.
2.2.2.1
REST Constraints
Roy Fielding deﬁned six constraints for the REST architectural style in his PhD
dissertation [Fie00].
1. Client-Server
Clients and servers communicate via an uniform interface, thus allowing
for separation of concerns.
Clients are not concerned how the data is
maintained at the server. Servers are not concerned about the state of
a client nor which interface the client uses. The separation of concerns
allows for the server and client code to be developed independently, as
long as the uniform interface is not modiﬁed.
2. Stateless
The server and client does not share any state. The client has to send all
required information, including session state, in each request. If desired,
the server may transfer the session state to another service capable of
persisting the state. When a client is ready to transition to a new state, it
will send a request to the server. Clients are considered to be in transition
while a request is pending.
3. Cache
The server must explicitly inform the client whether or not a resource is
cacheable. When a resource is cacheable, it should be cached to avoid
unnecessary requests to the server. Having a well-implemented caching
strategy improves the scalability and performance of the service.
4. Uniform Interface
The most fundamental constraint is the uniform interface, which ensures
the simpliﬁcation and decoupling of the architecture, thereby allowing the
involved parts to evolve independently. The uniform interface is split into
four speciﬁc constraints:
• Identiﬁcation of Resources
A request must explicitly identify the resource it intends to access,
==PAGE:26==
10
Web APIs
e.g. URIs for web-based systems. The way a resource is represented
does not need to match how it is maintained internally. The server
may format the resource in accordance with the requested represen-
tation format, e.g. JSON, XML or HTML.
• Manipulation of Resources Through Representations
A client can create, modify or delete a resource by manipulating the
representation of a resource, including any attached metadata.
• Self-Descriptive Messages
The messages sent between the client and server must be self-descriptive.
That is, the message and any attached metadata, must contain enough
information for the receiving part such that it knows how to process
it.
• Hypermedia as the Engine of Application State
Clients should be able to make state transitions based on actions
identiﬁed by the resource representation or the attached metadata.
A few ﬁxed entry points must be available for the client to start
discovering the system.
5. Layered System
The system may consist of multiple layers, however the client is unaware
of this. The client may connect directly to the end server or be serviced
by multiple intermediaries, such as load balancers, reverse proxies and
security enforcing components.
6. Code-on-Demand (Optional)
Clients can be served scripts or applets to execute. The serving of Code-
on-Demand may be a temporal extension to the functionality of the client.
Code-on-Demand is an optional constraint.
Only when an application is conforming to all the REST constraints, can it be
considered truly RESTful [RR07]. A truly RESTful system is a level 3 system
in the Richardson Maturity Model [Fow10].
2.3
Richardson Maturity Model
The maturity of a web-style system can be explained using the Richardson
Maturity Model developed by Leonard Richardson [Ric08].
The Richardson
Maturity Model consists of four deﬁned levels, where each describes a further
reﬁnement of the web-style system. The model was created to help determine
the quality of the web service. According to Richardson, the three technologies,
==PAGE:27==
2.3 Richardson Maturity Model
11
HTML (hypermedia), HTTP and URI are the real-world implementations of
the RESTful constraints [Ric08]. Richardson also notes, that web services can
make use of the three technologies, but not necessarily use them correctly and
thereby violating the RESTful constraints.
Level 0
Level 1: Resources

2.4
Data Formats
When a client and a server is exchanging data, the data can be of an arbitrary
type and format. The type of the data should be speciﬁed in the Content-
Type header. The Content-Type header deﬁnes the MIME-type of the data,
==PAGE:30==
14
Web APIs
which consists of a top-level type name and a subtype name [FB96]. Further
reﬁnements to the Content-Type can be included.
With web APIs, the more common top-level type is application. This indicates
that the exchanged data is intended for application use. The subtype diﬀers
slightly, but most common is xml, json and yaml [New15]. The subtype deﬁnes
the data format, i.e.
when the subtype is xml, the data must be formatted
as XML. Each of these data formats have their pros and cons, both in terms of
ﬂexibility, size and support.
With RPC the primary data format is XML (Extensible Markup Language), as it
natively supports namespaces and attributes [WPR10]. The oﬃcial MIME-type
for XML is application/xml [FKBH15].
With the more modern web APIs, the preferred data format is JSON (JavaScript
Object Notation) [New15].
JSON is lightweight and has implicit support for
types. The basic types of JSON are number, string, boolean, array, object
and null. The oﬃcial MIME-type for JSON is application/json [FKBH15].
2.4.1
HAL
Extensions to both XML and JSON have been done to provide uniform sup-
port for hypermedia. The extension is called Hypertext Application Language
or just HAL. Originally HAL was designed speciﬁcally for JSON, but a vari-
ant for XML has also been made. The MIME-type used should indicate if the
HAL extension is in use, this is done by using hal as a preﬁx to the subtype
(application/hal+json, application/hal+xml).
The purpose of using HAL, is that the system will support HATEOAS, thereby
becoming a level 3 system in the Richardson Maturity Model (see 2.3.4). The
HAL model focuses on representing two simple concepts, Resources and Links.
Resources contain links, embedded resources (i.e.
other resources contained
within the resource) and a state (standard JSON or XML data) [Kel13a, Kel13b].
Links have a target (a URI), a relation (the name of the link) and other optional
properties [Kel13a, Kel13b]. Figure 2.6 depicts the HAL structure.

API Blueprint
This chapter will describe the API Blueprint while identifying problems within

discussed. At last, alternatives to using the API Blueprint will be introduced.
3.1
Speciﬁcation
The API Blueprint is a documentation-oriented web API description language,
created by Apiary. It is open source and maintained not only by the people at
Apiary but also by the open source community. The API Blueprint is essentially
a couple of semantic assumptions over the text formatting syntax Markdown
[Gru14]. API Blueprint has been designed to be readable for humans, but is
also understood by machines. The main purpose of the API Blueprint, is to aid
in the design of web APIs, by easily allowing for comprehensive documentation
of the web API. With diﬀerent available tools, the API Blueprint can also be
used for quick prototyping, interactive documentation and automatic testing.
The core of the API Blueprint is the document in which the web API gets
speciﬁed. There exist some web interfaces with API Blueprint speciﬁc syntax
==PAGE:34==
18
API Blueprint
highlighting to aid in the writing process, though any text editor with support
for Markdown syntax can be used. Due to everything being written in text,
it is trivial to use any revision control system for collaboratively working on a
document.
The API Blueprint document can be parsed into an Abstract Syntax Tree (AST)
using the parser released by the API Blueprint team called Snow Crash. The
resulting AST is formatted as a JSON document, thus it is easily interpreted
by machines and have no language speciﬁc restrictions.

It was possible to identify 3 problems with the Payload section.
3.2.2.1
Repetitive and Redundant Headers
Specifying the headers for each request can be tedious, especially considering
the same headers being speciﬁed over and over again. Further, some headers
should be derivable based on the speciﬁcation, such as the Allow header.
3.2.2.2
Individual Body Sections for Actions
Similarly to the Payload section, the Body section can also be deﬁned indi-
vidually for each action under the same Resource section. Having multiple
responses for a resource, allows the resource to reply with varying content. The
resource should not diﬀer in content, but it can reply with diﬀerent representa-
tions of the content, e.g. formatted as JSON or XML. Responses should be reliable
and conform to the principles of REST as much as possible.
3.2.2.3
Bloated Schema Validation Syntax
Schema validation is great, but the JSON-Schema syntax is bloated and does not
support describing the properties using pure Markdown notation.

diﬀerent sections, but based on the speciﬁcation, all of the attributes are op-
tional. The speciﬁcation does not state, that the attributes must be included. It
only states that they may or should be included. With the current speciﬁcation,
the Attribtues section can be included yet overwritten entirely.
3.12.2.2
Action Speciﬁc Attributes
Each action should not require its own speciﬁc model, as this is the same as
varying content. Having varying content indicates a RPC like functionality.
3.12.2.3
Excessive Way to Describe Validation Schema
The Attributes section can work with the Schema section, but its an even
more excessive way to describe a validation schema.

There already exists tools that make use of the API Blueprint. The tools range

Apiary
The biggest tool yet, is made by the people whom also created the API Blueprint.

downloadable server code. The generated mock-up works when performing tests
that verify the format and models of the data being transferred, it does not
test the semantics of the request. Further, the mock-up relies on the speciﬁed
example requests and responses, it does not support dynamic content.

by Apiary. The tools only support generating the mock-up servers to run in

geared more towards humans than machines. Both Swagger and RAML use a

In the following section, I will outline and explain my suggested modiﬁcations

4.1
General
With the current API Blueprint description language, the described web APIs
are not guaranteed to be REST based nor more mature than a Level 0 system
according to the Richardson Maturity Model. With the changes suggested in
this chapter, the described web APIs will be given a helping hand in becoming
REST based systems and are guaranteed to be at least a level 2 system according
to the Richardson Maturity Model.
The changes suggested in this chapter, are not 100% compatible with the cur-
rent API Blueprint description language. The current parser used by the API
Blueprint, will not be compatible with the suggested changes either, hence a
new parser is also required.

All the properties can have constraints added to them. The constraints are used
to explicitly state how a value of a property is validated. The constraints are
type speciﬁc. Multiple constraints can be used for the same property, please
be aware that constraints can be mutually exclusive.
When using multiple
constraints, they all have to be satisﬁed.
The constraints are used by appending the type with a colon : and then the
constraint. Some constraints have parameters. If the constraint have param-
eters, these must be wrapped in parentheses and be typed immediately after
the constraint. If no parameters are required, the parentheses can be omitted.
Parameters must be separated by commas.

Chapter 5
Comparitive Analysis
In this chapter, I will compare the API Blueprint version 1A revision 8 from
chapter 3 with Bluem, which have been outlined in chapter 4. The comparison
will be based on an analysis of the features provided by the description languages
while also discussing the pros and cons when synthesis of mature REST based
APIs are desired.
5.1
Payloads
A core part of a web API are the payloads being sent and received.
Both
description languages support modeling of the payload, though the modeling is
implicit for Bluem.
With the API Blueprint, the writer is able to specify headers, attributes, the
body and a validation schema. The reason for being able to specify all this, is
that the API Blueprint is used for documentation, and pre-specifying this allows
for static payload examples. Examples of the headers returned can be included,
which is a great feature, though the headers are static hence dynamic headers
such as rate-limiting cannot be properly imitated. Bluem does currently not
have support for custom headers, such as rate-limiting or authorization schemes,
==PAGE:84==
68
Comparitive Analysis
but these are features that could be implemented and speciﬁed once, thus being
implicitly included automatically as part of the response.
Whether the API Blueprint or Bluem is the best approach, greatly depends
on what the API is trying to achieve. If the API is to be REST based, then
using Bluem will be a great help to ensure many of the RESTful principles are
implemented. On the other hand, if the API is made to support a very speciﬁc
set of actions or act as a RPC service, then Bluem will not be compatible.
5.2
Actions
It is possible to specify action speciﬁc URI parameters in both versions. The API
Blueprint supports action speciﬁc payloads and even conditional requests and
responses. Bluem restricts the possibilities within the Action section, but also
removes the concern of using proper responses for the diﬀerent HTTP methods.
The actions specify how to interact with the API. Both the API Blueprint and
Bluem allow for multiple ways of specifying the actions. The API Blueprint
does not limit the amount of times an action can be speciﬁed, thus making it
possible to create a RPC like system. Bluem ensures that an action can only
be deﬁned once. For a mature REST based API, Bluem could improve on the
amount of time and work needed for writing up the description.
5.3
URI Parameters
When specifying resources, an endpoint also have to be speciﬁed. The endpoint
is speciﬁed using an URI, which is based on an URI template. The URI template
have support for variables. In the API Blueprint there is no requirement to
specify these variables. The variables can be self-explanatory, but in some cases
a description would be beneﬁcial. With Bluem the variables must be described.
Both versions have support for specifying which type the variable have, an
example value, a default value and documentation for the variable. Only Bluem
have support for further constraints on the variable. The API Blueprint does
not have support for deﬁning a validation schema for the URI parameters, hence
only type validation is possible, where as Bluem also supports value validation.
The URI templates can also specify query parameters. In the API Blueprint,
the query parameters can be included in the URI template, though it is not a
==PAGE:85==
5.4 Models
69
requirement. Bluem does not support specifying query parameters as part of
the URI template, instead they should be speciﬁed as part of the Parameters
section.
Both versions support specifying URI parameters for both the resource and the
individual actions.
The Parameters section from Bluem provides more capabilities for properly
documenting and handling URI parameters, though it could be an annoyance
having to document all query parameters.
5.4
Models
The core of Bluem are the models used throughout the description. In the API
Blueprint, models are actually predeﬁned payloads, but in version 1A revision
8, they introduced an Attributes section.
The Attributes section of the
API Blueprint, is used for specifying certain attributes that may be a part of
the payload. The attributes are speciﬁed using MSON [Api15c], which supports
types but not constraints, hence the attributes must be combined with a schema
section to support both type and value validation. With Bluem the speciﬁed
properties must be a part of the payload. When creating the models, both type
and constraints can be speciﬁed, thus both type validation and value validation
is supported, without having to write two very similar descriptions.
The attributes of the API Blueprint supports mixins as it is based on MSON.
Mixins is used to include a previously deﬁned set of attributes within another
part of the description. Bluem supports model inheritance, meaning a previ-
ously deﬁned model can be inherited by a new model which will then inherit
the properties of the previously deﬁned model. Both mixins and inheritance
make it easier to re-use already deﬁned attributes/models, though they diﬀer
in the capabilities. With mixins the attributes are appended to the new list of
attributes. With inheritance the properties are added to the set of properties.
The important thing to notice is that the API Blueprint is using a list struc-
ture and Bluem is using a set structure. When using mixins, several attributes
could have the same name but possible diﬀerent type and meaning. When using
inheritance, the properties of the previously deﬁned model is used as the base
set of properties, when new properties are speciﬁed they will override previously
deﬁned properties if the names match. Mixins can be used multiple times within
a model, inheritance can not as it is based on inheriting from a single model. If
models based on key-value pairs are desired, then mixins can cause undesired
and undocumented eﬀects, whereas inheritance ensures that keys are unique.

77
The API Blueprint have a customized parser called Snow Crash [Api15d]. Snow
Crash is built on top of the Sundown Markdown parser [Mar12], and is built only
for parsing API Blueprints. Snow Crash uses regular expressions to parse the
API Blueprint speciﬁc sections and Sundown to parse the Markdown sections.
Everything parsed is put into an abstract syntax tree, which is then used to
generate HTML output.
Due to the amount of changes made, a new parser had to be developed for
Bluem. Snow Crash could be adapted, though it would require a lot of work,
hence the decision to write a new parser as part of the proof of concept was
made. The new parser will be built on top of a Markdown parser, just like
Snow Crash is. The chosen parser is MarkdownSharp [Atw14] that originated
from StackExchange, it was chosen based on availability and simplicity.
6.1.2
Parsing Using Regular Expressions
Parsing the description using regular expressions, can be done by reading one
line at a time and performing a set of regular expressions on each line, checking
for a match. When a match occurs, assumptions about the line can be made
and based on those assumptions, the line can be further analyzed.
The regular expressions required to parse a description can be hard to read and
debug. Furthermore they must be context aware, which means that the expres-
sions performed for a line could depend on the previous lines. When a Model
section is found, the following line should be indented with a tab or 4 spaces. If
one of the properties of the Model is an enum then it must specify enumeration
values that are further indented by a tab or 4 spaces. The enumeration values
shall not be seen as properties of the model, but only as enumeration values of
the property, hence the context is important when performing the expressions.
When using regular expressions, capturing groups can be used to easily extract
the required data.
When a line is not matched by an expression, it can be
assumed to be erroneous. The line can easily be reported as erroneous, but
the exact error might be harder to determine, when it has to be more speciﬁc
than "Syntax error". Regular expressions are suitable when considering per-
formance, though they have to be constructed properly, avoiding backtracking
with exponential complexity.

a Model deﬁnition:

6.1.3
Parsing Character by Character
Parsing the description character by character, also known as scanning, requires
more manual work than using regular expression as it is entirely hand coded.
The whole description should be parseable by a ﬁnite state machine, each char-
acter corresponding to a state. The ﬁnite state machine would begin at the ﬁrst
character and only have two ending states, one when an error have been found
or when reaching the End Of File (EOF).
Due to the nature of the syntax for the description, the amount of scenarios to
check for can be limited simply by looking at the ﬁrst character of each line.
If none of the scenarios match, then there is a syntactic error which can be
==PAGE:95==

described and located very precisely.
Using regular expressions, capturing groups can be used to extract speciﬁc
chunks of text. Reading character by character, the same can be achieved due
to the way the syntax of the sections have been composed. It should not be
necessary to backtrack at any point when parsing the description, hence the
parsing can be done in linear time.
6.1.4
Parsing by Querying the HTML
The description is written using Markdown, which means it can be translated
into a HTML document. The HTML document will have the same structure as the
description. The Document Object Model (DOM) of the HTML document, can
then be queried to extract the required information. To extend the querying
capabilities, a tool supporting CSS queries can be used instead.
Parsing time is increased and if a syntactical error is found, then it is not pos-
sible to pinpoint exactly where it is. When querying for resources, the query
would be h1, h2, as a resource can be speciﬁed by # (h1) and ## (h2). After
getting the h1 and h2 nodes, their content would have to be checked to make
sure they match a speciﬁc pattern. This pattern would most likely be a regu-
lar expression pattern, hence much of the same expressions used when parsing
using only regular expressions will also be used when using this method. The
beneﬁt of using this method, is that it allows for extraction of a node and all its
descendants. In case of a model deﬁnition, the node containing the model and
its optional name, would have a list of descendants. These descendants would
be the properties of the model, thus the owner model of the properties is easily
identiﬁed.
6.1.5
Choosing a Parsing Method
The purpose of the parser is not only to understand the description, but also be
able to correctly identify syntactical errors and report where they are located.
The only method that could easily support this is the character by character,
hence that was chosen to be used for the proof of concept.
A parser using the character by character method was created and parsing a
rather large API description (Listing B.1) took approximately 20 milliseconds.
To be certain that the character by character method was the way to go, a small
benchmarking test was done. Two incomplete parsers were made:

6.3
Synthesis of Code
Synthesis is the act of taking multiple entities and artiﬁcially creating some new
based on these entities. In the scope of this thesis, synthesis will be the act
of taking a human-readable API description, the parser, the analysis library
and the concept of automatic programming, to generate code that works and
behaves as described in the API description.
The steps involved during the automatic generation of code, depends on the
desired output. For server code, the focus will be on constructing the endpoints
and being able to handle and validate requests. For client code, the focus will be
on sending valid requests and being able to parse and understand the responses.
For documentation, the focus will be on compiling all the included information
into a format such that it allows for easy navigation and interaction.
Depending on the target programming language and framework, the generated
code may be very diﬀerent. Therefore requirements have been made that must
be satisﬁed by the generated code, regardless of programming language or frame-
work.

The communication between the server and client, is being transmitted using

be interpreted.
The status codes are to be used as described in the HTTP/1.1 standard [FR14]:
• “The 200 (OK) status code indicates that the request has succeeded.”

• “The 202 (Accepted) status code indicates that the request has been ac-

• “The 204 (No Content) status code indicates that the server has success-

• “The 400 (Bad Request) status code indicates that the server cannot or

7.2.1
Resources

The URI templates speciﬁed for the resources, are checked to ensure that there

References to models are resolved when the whole abstract syntax tree have
been processed, this ensures that the order the models are deﬁned in is insignif-

referencing.
A directed graph of the model inheritance is being created while the models are
being analyzed. 2-cycles are detected upon resolving the model references. To
further determine if there is a circular dependency between two or more models,
the directed graph of model inheritance is checked for cycles using Algorithm 1.
The predicate P is deﬁned as:
is model.parent defined AND is dependency = model.parent
The predicate ensures that only the inherited model will be considered during
the check.
A directed graph of the property dependencies for the models is also being
created while the models are being analyzed. To determine if there is a circular
dependency in the property chain, the directed graph of property dependencies
is checked for cycles using Algorithm 1. The predicate P is deﬁned as:
is dependency a property AND dependency.defaultValue ̸= null
The predicate ensures that only property dependencies that do not explicitly
state null as their default value will be considered.

Conclusion
In this thesis I have researched the possibility of creating a Model-Centric API
description language by modifying the API Blueprint. I have introduced the
concept and purpose of a web API and outlined two architectural styles, Re-
mote Procedure Calls (RPC) and REpresentational State Transfer (REST). The
Richardson Maturity Model, which is a model used to determine the quality of
a web service, was explained and used to highlight the diﬀerences between RPC
and REST. REST is able to achieve the highest level of maturity. The research
of the API Blueprint was therefore focused at conforming to the principles of
REST.
The API Blueprint is a documentation-oriented web API description language.
I have described and analyzed the API Blueprint. Problems within the API
Blueprint speciﬁcation have been identiﬁed and described. A discussion of the
tool support for the API Blueprint, showed that it was possible to create mock-
up servers and generate client SDKs. Alternative API description languages
were described, while highlighting the reason for choosing the API Blueprint as
the basis for this thesis.
I have proposed a new Model-Centric API description language, referred to as
Bluem.
The purpose of Bluem was to introduce Model Driven Architecture
and also solving the problems identiﬁed within the API Blueprint speciﬁca-
tion. Bluem is focused on creating models that support type and constraints,
==PAGE:130==
114
Conclusion
such that data validation can be performed. Bluem is also capable of aiding in
the creation of REST based systems and the resulting system described using
Bluem is guaranteed to be at least a level 2 system according to the Richardson
Maturity Model.
An analytical comparison of the API Blueprint and Bluem was performed. The
analysis showed that the API Blueprint provides better granularity when writ-
ing the description. However, the granularity also made it possible to create
RPC like systems. The analysis also showed that Bluem aids in the creation of
REST based systems and abstracts some of the more ﬁne grained details that
can be diﬃcult to implement correctly. The analysis showed, that the two de-
scription languages have an almost identical feature set, except when comparing
inheritance. The API Blueprint allows duplicates to exists, where as Bluem does
not and thereby oﬀers deterministic inheritance. A small example description
was written using both the API Blueprint and Bluem. Comparing the example
descriptions showed that, in some cases, the amount of lines written and total
size of the description can be reduced by almost 50% when using Bluem.
A parser capable of reading and interpreting a Bluem description was created.
The method used for parsing was described and it was explained why it was cho-
sen. It was explained that the parsed speciﬁcation must be analyzed to further
validate and optimize the description. The analysis must be able to detect mu-
tually exclusive constraints and circular dependencies. The optimization must
merge constraints when possible and apply compile time constraints. The re-
quirements which the generated code must conform to have been detailed. The
requirements will ensure that the resulting system is mature and REST based.
I created a prototype as a proof on concept. The overall system architecture of
the prototype was explained. It was stated that the system consists of a parser,
an analysis library, a synthesis tool with plugin support and three plugins. I
have described how the analysis library determines if there are circular depen-
dencies and how it further validates and optimizes the parsed description. The
synthesis tool was presented along with the reasoning behind the support for
plugins. Three plugins were made, a server synthesis plugin based on the Nancy
web framework, a client synthesis plugin based on the RestSharp library and
a documentation synthesis plugin using Bootstrap as a web framework. A test
was performed to verify the synthesis process and the validity of the generated
code.
I have completed all the goals for the project. The thesis have showed that
it is possible to modify the API Blueprint such that it becomes Model-Centric
and support describing and documenting an API, while also including support
for validation rules based on types and constraints. At last a prototype was
created. The prototype showed that it was capable of parsing and analyzing

8.1
Future Work
Bluem is a work in progress and can be extended and improved in many ways.
Improvements can range from changes in the syntax to the implementation of
more advanced artiﬁcial intelligence. The following is a list of suggested ideas
for future work:
• The analysis library could be improved, such that it is able to identify
how resources are related without imposing further restrictions on how to
write the Bluem description. The identiﬁed relations could then be used to
encapsulate the responses returned by the API in a format like HAL. This
would make the API become a level 3 system according to the Richardson
Maturity Model and satisfy the Hypermedia as the Engine of Application
State constraint in REST.
• Create more synthesis plugins and determine the best method for remotely
storing and retrieving them. Considered methods could include the use
of the NuGet package management or the creation of an online service
exposing the storing and retrieval functionality through an API.
• Research ways to optimize the Bluem speciﬁcation, more speciﬁcally the
syntax of the various sections, when considering readability and ease of
writing.

# Group Categories
Categories are the way articles are classified on the shop. The categories are
organized in a tree structure where each category has another category as
parent, except for the top level category.
One thing to notice about categories is that articles can relate to several
categories at the same time. A certain shoe, for example, may be related to "
Women" and "Shoes" at the same time. Another fact about categories and articles
is that if an article relates to a certain category, then it is automatically
related to all parent categories as well. If the hierarchy is ’Catalog’ -> ’
Women’ -> ’Women shoes’, an article related to ’Women shoes’ is also related to
’Women’ and ’Catalog’ by default. A direct result of this policy is that the
top level category relates to all articles available.

Appendix C
Formatting
C.1
Names
The URI parameter names and the model property names are not obliged to
conform to any speciﬁc casing format. The synthesis tool can properly under-
stand most casing formats, including PascalCasing, camelCasing and under-
score_casing.
Which casing format to use is often based on which programming language the
content is to be written in or personal preferences. A study have been made,
analyzing the code-readability of camelCase and underscore_case [SM10]. The
study was carried out using eye-tracking technology, measuring correctness, ﬁnd
time and visual eﬀort. The study showed that on average it took approximately
20% longer time to read camelCasing identiﬁers than underscore_casing iden-
tiﬁers. Further, the study also showed that camelCasing requires a higher av-
erage amount of ﬁxations to read identiﬁers of 3 words or more.
==PAGE:150==
134
Formatting
C.2
String Escaping
The strings used in Bluem can contain characters that need escaping. To solve
problems when the character used for starting the string must occur inside the
string, escapes can be used.
A string is a sequence of characters wrapped between either single quotes (’text’)
or double quotes ("text"). The character used for escaping is backslash (\).
When a character need to be escaped, it is preﬁxed with the escape character.
The following characters can be escaped:
• Single quote ’ becomes \’
• Double quote " becomes \"
• Backslash \ becomes \\
• Newline becomes \n
This way of escaping is commonly used [Net15, Man01, Ora15]. Carriage returns
(\r) are stripped away.
C.2.1
Examples
The following examples demonstrate how to use string escaping.
’this is a string in single quotes’
’this is a string in \’single quotes\’, cool huh?’
"this is a string in double quotes"
"this is a string in \"double quotes\", awesome right?
’This is a string with a backslash \\ wrapped with single quotes’
"This is a string with a backslash \\ wrapped with double quotes"
The example below demonstrates how a string with newlines can be escaped.
"This is a string with
multiple lines
without newlines escaped"
"This is a string with\nmultiple lines\nwith newlines escaped"
