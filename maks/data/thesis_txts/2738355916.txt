Abstract
Process mining is a technology that analyzes and models business processes by extract­
ing knowledge from event logs to understand the process in more depth. Traditional pro­
cess mining approaches often require technical expertise, which can be a complex task for
researchers with basic programming knowledge. Additionally with process mining reach­
ing a more mature state, different organizations, e.g. universities and companies, have
the desire to use various datasets and algorithms from another organizational entity which
raises privacy issues and requires manual steps. This thesis acts as a proof of concept in­
troducing a conceptual model and visual editor for simplifying process model workflows by
proposing the idea of process mining pipelines. The solution is aimed at automating data
management and algorithm execution in the setting of various organizational boundaries.
We present the conceptual design and proof of concept implementation of a pipeline editor
that allows for the creation, validation, and execution of process mining pipelines across
multiple organizations. We show a clear conceptual model for process mining pipelines
that incorporates different organizations, resources and algorithms into it. Our visual edi­
tor, using the previously mentioned model, includes a user interface for non­programmers
to build and manage process mining pipelines using drag and drop components.
This thesis contributes to the field of process mining by providing a framework for dis­
tributed process mining operations, creating an easier collaboration between organiza­
tions, and expanding the accessibility of process mining tools for a non­technical audi­
ence.

Introduction
Process mining is a data­driven discipline that extracts insights from process execution
data stored in event logs, combined with process models or business rules. Numerous
tools, including open source options like ProM and commercial products, offer diverse
algorithms for process analysts. Choosing specific tools involves navigating various fac­
tors, such as context­specific requirements, data constraints, and algorithm sharing pref­
erences.
As process mining has matured as a field, more complex operations have become avail­
able and using these in connection with each other has become more normal. An example
of this could be to discover a process model using an event log, followed by a conformance
check using another event log all the while saving the different results. Handling all these
steps one at a time in person gets tedious quickly, therefore, the prevalence of process
mining pipelines or workflows is emerging.
Creating process mining pipelines across organizations remains complex, requires pro­
gramming, manual data adjustments, and has to address privacy concerns. Most event
logs owned by organizations contain some sort of private data e.g. personal data, medi­
cal records, which should not be shared with others. These logs have to be anonymized
to use in public domains, or researchers have to have access to algorithms by source
code or by developing their own algorithms to run the algorithm locally. Handling these
complexities are challenging and should be simplified to make it more accessible to non­
programmers, which is one of our goals with this thesis.
As a solution to these issues, we are trying to introduce a platform based on a distributed
architecture that makes it easier to share process mining resources among researchers
and run process mining pipelines across different organizations. Distributed Architecture
refers to a system design in which the components of the application could be shared
across organizations or different nodes. This approach could make collaboration with
other organizations easier while still respecting the boundaries and sensitive information
of the organization.
The plan of a Distributed Architecture for Process Mining Pipelines (DAPM) is to create
a platform where process mining pipelines can be created easily, by a user without prior
programming experience, using plug and play components. DAPM should make it possi­
ble to define organizational borders that ensure that sensitive data does not cross without
it being explicit, while also making it obvious, for the user, what the necessary inputs for
a given algorithm are and what the expected result will be.
This will take form as a workflow management system that lets the user build pipelines
represented by directed graphs in a graphical user interface and execute them directly in
the same window.
To ensure that the pipeline being built is correct, a language must be put in place. This
language should contain all necessary information required to describe a valid pipeline,
including a type system for resources. This will make it possible to validate and find errors
before the pipeline actually runs.
Prior to this thesis another group has made a proof of concept to show the general idea
of DAPM. This code base can act as a starting point for our project. [1]

Background
In this chapter, we begin by introducing the evolution of process mining and its applications
in the literature. Here after, we will introduce the term ”pipeline” and how it has historically
been used in data science. Following this, we will introduce the state­of­the­art in process
mining. And finally, we will review related work and highlight the significance of this paper.
2.1
Literature Background
In this section, we will describe some of the historical context for topics mentioned and
used in this project. It will start out with some background context of process mining. This
will be followed by the context for pipelines in computer science.
2.1.1
Process Mining Background
Process mining is a relatively new discipline used to extract knowledge from event logs
and improve a process. Early work by van der Aalst introduced the concept of describing
workflows as Petri­nets[4]. This foundational work laid the groundwork for automated
techniques for discovering, monitoring, and enhancing process models directly from event
logs. [5] In the next few sections, we will go over the three types of process mining,
discovery, conformance and improvement.
Discovery
The goal of Discovery is to take an event log and produce a process model which captures
that event log. These models can then be assessed over four different quality dimensions:
fitness, simplicity, precision and generalization. Different families of algorithms exists to
discover these models such as the alpha­algorithm, the heuristic­miner and the inductive
miner. Each has its own benefits and disadvantages. [6]
The alpha­algorithm advantages lies in the simplicity, formal basis and its ability to di­
rectly convert an event log to a Petri net. Its limitations include being sensitive to noise
and being unable to handle complex process structures. However it continues to be used
as an educational tool to improve the understanding of process mining techniques. [7]
The advantages of the heuristic­miner are its resistance to noisy and incomplete data,
its ability to capture frequent behavior, and it flexibility due to its parameter tuning. It is
widely used in industry because of its robustness. However, it still has its disadvantages.
These include its complexity of parameters and its that it is still sensitive to noisy data. [8]
Finally the advantages of the Inductive­miner is its guarantee of soundness, its ability to
produce scale­able models and its flexibility and customization. Its disadvantages include
having a large computational overhead, being prone to produce overly complex models
and its sensitivity to log­quality. [9]
Conformance
Conformance is used to explore how well a model fits in the real world. This requires an
event log and model. This model may either be created by hand or can be discovered.
Conformance checking relates events in the event log to activities in the model and com­
pares both, with the goal of finding commonalities and discrepancies between the model
and the observed behavior.
This can be valuable for business alignment and auditing. This could be replaying an
event log on a model to find undesirable deviations that could suggest fraud or ineffi­

Improvement
When a model is discovered and conformance has been run on it, process mining can be
used as a way to enhance the processes. Some of these enhancements could include
repairing a model by identifying deviations between the model and the observed process.
Since event logs often contain a lot of additional information such as timestamps, process
models can be enriched by adding these. By doing this, timestamps can be used to find
bottlenecks and frequently used paths.[11]
2.1.2
Ingredients for Process Mining
When working with process mining, there are some standard resource types that will be
used. these include event logs, Petri nets and BPMN models. Here we briefly introduce
these.
Event log
An event log is the primary data source for process mining. They contain records of events
that describe the executions of activities within a business process. The components of
an event log is a case id, an activity and a timestamp further more there can be many
more components to further flesh out the event. Event logs come in many file types, such
as CSV, XML and XES.
XES is a standardized format used especially in process mining. It helps organisations
ensure that event logs adhere to a compatible format for most process mining tools. [12]
Petri net
Petri nets are a powerful tool for modeling concurrent systems. They are used in many
applications ranging from workflow management to distributed systems. The foundational
work for Petri nets was made by Carl Adam Petri, and they have become a crucial com­
ponent in the toolkit of researchers. [13]
BPMN
Business Process Model and Notation (BPMN) is a standardized graphical notation de­
veloped by the Object Management Group (OMG) used to specify business processes in
a workflow. BPMN provides a comprehensive set of symbols and elements to depict com­
plex business process workflows clearly and intuitively. It helps communicate between
business process design and implementation, making it accessible to both business an­
alysts and technical developers. [14]
2.1.3
Challenges in Process Mining
As process mining is a relatively young discipline, there are a lot of challenges to overcome
still. Some of these are related to how process models are created, such as represen­
tational bias and concept drift [5]. But some others are more about how process mining
is used. These include cross­organizational mining, combining process mining with other
types of analysis, improving usability for non­experts and improving understandability for
non­experts.[11]
2.1.4
Pipeline Background
These challenges line up well with what other data science branches has faced in the past.
This, in part, has been made easier with the use of pipelines. In computer science the
concept of pipelines has been widely used in branches ranging from software engineering
and machine learning to computer architecture. A pipeline is made up of a series of steps,
each performing a specific function.
The concept of a pipeline can be traced back to the early days of computer architecture,
where pipelining was introduced to improve the throughput of CPUs by allowing multiple

sequential pieces has been adopted in many fields of computer science.
In software engineering, building and deploying pipelines, or continuous integration, have

In data science and machine learning, pipelines have been used to streamline the pro­

been widely used to automate scientific workflows with a complex sequence of tasks.
2.2
Related Work
Process mining is receiving more and more attention, leading to many tools being re­
leased both by private companies and open­source. The two main applications to run,
compare and visualize different process mining algorithms are ProM and Disco. There
are other existing technologies worth mentioning which is the library PM4PY and the web
application Smyrida which is somewhat similar to our project, although it is lacking many
features which are essential to us. Some of these has seen some of the same challenges
as mentioned earlier and have started to act on them, such as the RapidProM extension
to RapidMiner. And institutions such as DTU prompting projects like DAPM.
2.2.1
ProM/RapidProM
ProM is an extensible framework that supports numerous different algorithms, file formats,
data visualizations, and conformance checking techniques. Although the disadvantages
of ProM is the user experience, that could be further improved. It still uses a UI design
that is considered obsolete and the user is not properly informed about the errors that
have occurred in the system, which unnecessarily frustrates the user. [16]
The people behind ProM has teamed up with the people behind RapidMiner to make the
RapidProM extension. This is meant to make it easier for non­experts to use process
mining tools and to create pipelines with process mining components. [17]
2.2.2
Disco
Disco serves as a desktop application that is capable of executing process mining al­
gorithms, performing conformance checks, and applying filtering alongside model and
statistics visualizations. However, Disco lacks built­in functionality for resource sharing,
necessitating external exportation for collaboration, a process that is comparatively more
intricate and time­consuming when compared to alternative solutions. Despite this limita­
tion, Disco’s user­friendly interface and extensive features make it a favorable choice for
specific scenarios. [18]
2.2.3
PM4PY
PM4PY stands out as a widely­used open­source process mining tool designed as a
Python library featuring a console interface, enabling users to seamlessly integrate pro­
cess mining directly into their codebase. PM4PY has capabilities such as running process
mining algorithms, performing conformance checks, and executing various other process
mining operations. [19]
2.2.4
Smyrida
The Smyrida web application features a shared open­source modular software system
as its backend. Published recently to the initiation of our project Smyrida incorporates an

2.2.5
A Distributed Architecture for Process Mining: Controlled Sharing
of Data and Algorithms
This paper (DAPM CAiSE) [21] investigates the issues and needs when setting up com­
plex process mining pipelines across organisations. It proposes a distributed architecture
for process mining (DAPM) that should allow for easy creation, deployment, and execu­
tion of process mining pipelines. This also includes aiding analysts in sharing of both
data and algorithms while at the same time ensuring the privacy of these resources. In
the paper, the problem is presented and a set of scenarios (Figure 2.1) is shown. These
are then verified through feedback to express the relevance of the issues. Finally some
requirements are given for a novel process mining platform geared towards facilitating
analysts in the controlled sharing of data and algorithms. This paper is the initiating factor
for all of DAPM and therefore all the projects underlying it.

2.2.6
Process Dry Lab
Previous to our thesis, another group has made a prototype for DAPM which was able
to deploy a simple process mining pipeline. This was called Process Dry Lab [1]. This
project was started because of the increasing attention to the process mining field. It
suggests a new way to solve the problem of sharing implemented artifacts for validation.
2.2.7
DAPM Systematic Literature Review
Along side our project, a systematic literature review (Appendix A) was conducted to in­
vestigate How the current state of process mining pipelines looked, if there was a common
language used regarding pipelines between different fields of computer science and to see
how sharing data across organisations were handled in other fields. The findings in the
SLR were that the language we use in DAPM was fairly in line with what others use.
2.3
Significance of Study
Following the related works and as part of DAPM, we aim to further ease the process
of sharing process mining workflows between researchers. This includes easily sharing
data and algorithms between organizations in a safe and secure manner. It can be seen
from the challenges in process mining mentioned previously and the issues presented
and validated in DAPM CAiSE that these goals are very relevant.

3.1
Problem Statement
Process mining researchers working in different organizations face major challenges when
integrating data sources with different levels of privacy and confidentiality, orchestrating
algorithms with different levels of visibility of source code, and allowing the interoperability
of data repositories and techniques between organizations. An important part of address­

process flows. By integrating resources, repositories, data sources, and algorithms, the
model creates a structured approach to process analysis and optimization. The following
sections provide an in­depth look at the components and interactions within this system,
highlighting the mechanisms for resource management, data integration, and pipeline
execution.

5.2.1
Organization nodes
Organization nodes act as a container for the flow nodes so it is easy to distinguish be­

ferred from one organization to another they have to cross the boundary of the organiza­
tion nodes highlighting the change in organizations.
5.2.2
Flow nodes
Flow nodes are fundamental units in the pipeline, representing different processing steps
or data elements. They are connected through handles, which define the flow of data, be­
tween the different handles. Different type of flow nodes have different amount of handles,
both input and output handles. Each flow node should be connected to an organization
node to be able select the instantiation data for the node.
There are different types of flow nodes:
• Data Source: Using a data source node we can select a specific resource from the
organization that the node is contained in. A data source can be categorized into
three types, which are Event logs, Petri nets or BPMN models. A data source node
has one single output handle, which is the selected resource.
• Data Sink: Using a data sink node we can select a repository and save the con­
nected resources in the repository. A data sink has one single input handle, which
refers to the connected resources.
• Operator: Using an operator node we can select an algorithm from the organization

Handles
Handles refer to the input and output endpoint on nodes and edges, facilitating the flow
of data and configuration between components. Input ports receive, while output ports
send them to the next component in the pipeline. To ensure proper connectivity and data
transfer across various nodes, ports have a specific type that guarantees no mismatches
can happen in the pipeline and ensure proper execution by the platform.
Configurations for different nodes
To ensure that the pipeline can be executed by the platform, it is necessary for the nodes
to be connected to existing resources stored within the platform. These specific details
are found within the configuration of each node and vary for each individual node. Exact

5.4
Life cycles of a pipeline
This section explores how a process mining pipeline can be broken down into differ­
ent stages ensuring the optimal process analysis and addressing privacy concerns when
sharing a pipeline. The life­cycle of a pipeline consists of three different stages the tem­
plate, instantiation and execution stage. The process of creation of the pipeline goes
through these three stages, first the user creates the nodes and connect them, this is the
template data. Then the user selects the configuration for the different nodes, which uses
existing resources from the platform, this is the instantiation data. Finally the user deploys
the pipeline which is the execution stage in the back­end. These stages are important to
differentiate, since they refer to different levels of complexity in terms of detail. In the
following sections, we will go over the three stages in detail.
5.4.1
Template
The template stage is where the template of the pipeline is designed. Here, the main thing
to be defined are the nodes, their handles, and how they are connected. It has no direct
association with an organization or the back­end.
In the template stage the
data source node only have an expected resource type, the
operator node has a hint to give an idea of which algorithm are to be used and the
data sink node has no additional information.
A point about the template stage is that this is the stage which is supposed to be stored
and shared between users. This is duo to the users being more interested in the struc­
ture/template of a pipeline and not necessarily all the details. The exact details might not
be available to everybody and it can potentially contain sensitive data in the organization.
Data associated with the template:
• Data Source: Handles and expected resource type.
• Data Sink: Handles.
• Operator: Handles, expected resource types and hint.

5.4.2
Instantiation
Instantiating the pipeline means connecting the nodes to specific resources and reposi­
tories. For the data source node some data are loaded in the form of a resource and for
the operator node an algorithm is loaded as a resource. For the data sink node a repos­
itory where data can be stored will be chosen. You can notice that organizations are not
part of the instantiation data, even though it was mentioned in the conceptual mode. The
reason is that it is only in the pipeline for a visual representation in the editor for a clearer
distinction between different organizations. The organization is referenced in resources
and repositories as an organizationId. Repositories are identified by an organizationId
and a repositoryId and resources are identified by an organizationId, repositoryId and
resourceId.
Data associated with instantiation:
• Data Source: Organization and specific resource.
• Data Sink: Organization and specific repository.
• Operator: Organization and specific algorithm.

5.4.3
Execution
In this stage the pipeline will be assigned to actual running components and executed.
The stage mainly consist of parsing the pipeline, breaking it down to different steps and
dependencies, and executing them by transferring data and running the specified algo­
rithms. When it comes to the execution there are two main approaches, orchestration
and execution. Orchestration is when the different steps of the pipeline e.g. data transfer
or algorithm execution is handled by one central entity, the orchestrator in this case. The
orchestrator breaks down the pipeline to steps and instructs the other microservices to
execute these different steps. Choreography is a more decentralized approach where
each microservice ’knows’ its own role and does not need to be instructed by a central
entity. The platform uses the orchestration approach for simplicity and scalability.

Evaluation
Chapter 5 of this thesis outlines the conceptual model for defining a process mining
pipeline, including its essential components and their interactions. In this chapter, we

7.1
Creation
Requirements about the creation of a pipeline mostly included having the correct type of
nodes, being able to connect them and link them to actual resources. Our implementation
fulfills the requirements about the creation of a pipeline. The editor allows users to create
all type of nodes by dragging them onto the drawing board. All nodes contain the correct
number of handles and they can be connected by dragging the handles together. The
organization node can serve as a container node and flow nodes can be placed inside
to indicate that the flow node uses data from the selected organization. When a node is

node, categorized into template and instantiation data. Additionally, when a file is being
saved to the data sink the user can select the edge and name the file in the configuration
sidebar. Overall the editor allows the user the create the necessary nodes, connect them
together and link them to existing resources in the system.
7.2
Validation
Requirements about the validation include type mismatches, missing filenames when an
edge is connected to a data sink, allowing no cycles in the pipeline and the successful
deployment of the pipeline. When an edge has a type mismatch, meaning that the two
connected handles have different resource types, the edge will change color to red. This
is to indicate the error and its location in the pipeline. In the same way when an edge
is connected to a data sink without a specified file name the edge will also change color
to red to indicate the error. Additionally, no cycles can be connected in the pipeline, the
last edge would not connect if it would be complete a cycle. Finally, when we deploy the
pipeline all the resources show up in the sidebar in the overview page indicating that they
have been created by the pipeline. To further improve validation the user should not be
able to deploy a pipeline if there are any errors, this has not been implemented yet and
is included in future work. Overall all the requirements about the pipeline have been met,
although they could be improved further.

Now that the conceptual model has been proposed and the product has been explained.
We will discuss the results and explain some of the choices made throughout the process,
how it affected the project, and other ways it could have been done. The first step will be
to go through how we think we solved the research questions, then we will talk about how

8.1
Solving the research questions
Here we will go through each research question from Chapter 3 and assess if we have
made progress with them. First we will go through the conceptual questions, then the
product questions and finally the overall questions for DAPM.
For the conceptual questions, we have made a solid conceptual model. This both the de­
scribes what a process mining pipeline is and it details the stages in a pipelines life cycle.
When talking about what requirements it takes for a pipeline to be shareable, we have
found that this greatly depends on how a user wants to share it. If the user wants to share
the general procedure, only the template of the pipeline needs to be shared. However, if
the pipeline needs to be reproducible, then the user needs to share the instantiation step.
Sadly, we did not get to implement this in our product, but the groundwork was laid for
future groups to save and retrieve the different steps in the pipeline life cycle.
To answer the product research questions. We first formulated the requirements for it to
encapsulate the conceptual model we had previously build. The evaluation of how that
went can be found in Chapter 7. Further more, it helps solving the research questions by
giving the user a good overview over what different resources, repositories and algorithms
they have available. It lets the user intuitively build a pipeline and deploy it. Currently,
there is some form of validation, but it is not very well enforced and only validates on the
template stage.
Our contribution to the overall research questions has been to help users without technical
expertise build process mining pipelines in what we think is an easy and intuitive way. In
addition, we have made a tool which provides a way for users to upload and manage their
organisations, resources, repositories and algorithms.

Through the process of making the product, we have had different ideas of how the flow
of creating a process mining pipeline would be. They differentiate on when a user will
use the organisation nodes and on when a user will fill out template and instantiation

Would they start out by creating the organisation nodes or would they build a pipeline

minor detail, the way we thought it to be shows in the conceptual model. In Figure 8.1 it

A thing that was discussed to some degree in the early conceptions of our product was

should each deployed front­end belong to one organisation.
We ended up going with each organisation having their own front­end. This was due to

belong to a single user, an independent user database would have to be made, keeping
control of all users and what organisations they are connected to. Instead an organisation
can now freely manage their users. The same reasoning can be used to argue against
having one front­end that all organisations and users would have to use. Someone would
need to be in charge of it.

Conclusion
This thesis presented a Distributed Architecture for Process Mining (DAPM), focusing on
the design and the proof of concept implementation of the system. The primary objective
was to address the mentioned issues with traditional process mining approaches, the
need for technical expertise, manual data management, and the challenges of cross­
organizational collaboration.
Our solution introduced a conceptual model and a visual editor for process mining pipelines.
The conceptual model provided a structured way to include various organizations, re­
sources, and algorithms into a unified framework. The visual editor, designed for user­
friendliness, makes it easier for users to select various resources from organizations and
execute algorithms on it.
Key areas of this thesis include:
• Architecture for Distributed Process Mining: Together with the other groups we
developed the design for the system, which consists of the visual editor on the front­
end side, the platform on the back­end side. The back­end handles the communica­
tion with other organizations and provides the necessary data for the front­end. The
front­end contains a visual editor that lets the user drag­and­drop nodes to create
process mining pipelines.
• Conceptual Model for Process Mining Pipelines: The conceptual model defined
a solid structure for how a process mining pipeline should look like and introduced
different life cycles of these pipelines. We separated the template, instantiation and
execution to make sharing the pipeline easier with others. This model serves as
a foundation for creating, validating, and executing complex process mining work­
flows.
• User­Friendly Visual Editor: The visual editor was designed as a proof of concept
implementation of the conceptual model. It lets users drag­and­drop nodes, connect
them together, link the nodes to actual resources in the back­end and execute the
pipeline in one editor. The editor also supports the validation of pipelines to ensure
correctness before execution.
• Future Work and Improvements: While the thesis have a solid foundation, several
areas for future work were identified, including further validation, the addition of
custom resource types, monitoring and management of deployed pipelines, and the
development of additional pipeline components.
In conclusion, this thesis created a proof of concept implementation in the area of pro­
cess mining by letting different organizations utilize each other’s data. The editor allows
users to easily create pipelines by drag­and­dropping different nodes, connecting them
and linking the nodes to actual resources, algorithms and repositories. Future research
and development can build on this foundation to further enhance the capabilities and ap­
plications of distributed process mining.

Future work
Here we will talk about what could have been added and improved to the product, had
we had more time. Since the DAPM project is in its early stages, this could also serve as
suggestions on how future groups could continue.
10.1
Further validation
Currently the validation of the pipelines only happens on the front­end and on a somewhat
low level. This is convenient because it is fast and easy. However, to fully check whether
a pipeline would be deployable, a lot of other criteria could be checked.
One such thing could be to check whether resources and operators were deployable and
configured right, before deploying a complete pipeline. This would require some work
from the back­end, as it would require some new API calls to check the status of these
resources. It could also be taken a step further out, to check the connection to the repos­
itories and organisations. This would both make it easier and faster for the user to find
faulty components in their pipelines.
Another thing that currently is not validated, is the format of the chosen resources. An
example of this could be to check if an event log had the needed XES attributes, to run a
given operator. This would further ease the creation of the pipeline for the user.
10.2
Custom Resource Types
In our product, the available resource types are hard set to what we found were the most
common and simplest types. However, since the user can choose what algorithms they
upload, the use can be expanded to not only focus on process mining, but to include a vast
array of data science tools. This would require a looser control of what kinds of resource
types exist in the system.
A way to fix this could be to let the user manage their resource types. This would pose
the problem that different organisations would have different sets of resource types and it
would be hard for them to share resources and algorithms.
Another solution could be to have extensions set with different standard resource types
that an organisation or a user could choose to use. This would give the problem that
someone would have to create these extensions and determine what the different stan­
dards should be.
10.3
Monitoring deployed pipelines
When a pipeline is deployed the rest of the responsibility lays in the back­end. This leaves
the user with kind of a black box, where they will not know what progress has been made
or if something has gone wrong. This could be remedied by having a page on which the
user could follow the progress of a pipeline. It could let the user see which operators were
currently running and their status. It could even let the user see intermediate results in
the pipeline. This would again require some more API calls from the back­end.
10.4
Managing pipelines
The pipelines shown on the overview page is currently only stored in the users browser.
We wanted to make it so that the overview page showed all the pipelines stored in the

it would be an integral part of how users would share and contribute pipelines in and
between organisations. Ideally the user should have to option to open a pipeline, edit it

The biggest hurdle for making this a reality is that recreating a pipeline from the JSON

Introduction
Process mining is a data-driven discipline that extracts insights from process
execution data stored in event logs, combined with process models or business
rules. Numerous tools, including open source options like ProM and commercial
products, offer diverse algorithms for process analysts. Choosing specific tools
involves navigating various factors, such as context-specific requirements, data
constraints, and algorithm sharing preferences.
Creating process mining pipelines across organizations remains complex, requir-
ing programming, manual data adjustments, and addressing privacy concerns.
In response, we, from the DAPM project, are trying to tackle this problem by
defining a Distributed Architecture for Process Mining (DAPM) [4] to facili-
tate the controlled sharing of data and algorithms among analysts. The central
question is ’How can we assist process analysts in sharing data and algorithms
for process mining in a controlled manner?’
In this paper a systematic literature review will be done to explore terminology,
state of the art, and how sharing of data and algorithms is handled by others.
This will be done by having an exploratory phase in which a set of keywords
relevant to the topic will be found. This will be done through interviews with
experts and stakeholders in the domain. Hereafter a phase of structuring will
be used to create a search strategy to distinguish what literature is relevant.
Finally, the literature will be analyzed and discussed and put into perspective
to the DAPM project.
2
Literature search methods
In this section, we will describe how the literature search was conducted. First
research questions were made and an exploratory phase was initiated to gain
a more in depth understanding of the pipeline domain. Secondly, a structur-
ing phase was started, where a search query was found and the results were
structured, and the relevant articles were found. Finally, the included articles
1
==PAGE:55==
were read through and the data necessary to answer the research questions were
extracted.
2.1
Research questions
Here the research questions for this paper will be introduced with an explanation
of why its relevance. These questions were made with the help of stakeholders
and all concern the current needs for DAPM.
What is the current state of data processing pipelines in the context
of process mining? (Q1) This will help to understand what directions are
currently being researched and what shortcomings already have been identified
in published material.
what is the definition of process mining pipeline and, in particular
”distributed” ones? (Q2) Here, we will explore what terminology is being
used by others and if there are perhaps commonalities between both different
research groups, but also between the different branches of research being done
in this space.
How is data sharing handled between organizations? (Q3) This will be
explored by looking at how sensitive data and algorithms are being shared in
the literature.
2.2
Exploratory Phase
The search started with an exploratory phase in which the goal was to find a
basic idea of what keywords were used by others. This included using Google

2.4
Search string
Using the keywords found in the exploratory phase, the search string was con-
structed using logical operators to include various ways to address pipelines in
different contexts. The keywords were stemmed and an asterisk was added to
include all words starting with the same core word. Finally, exclusion keywords
are added to scope in the search to a manageable result size. The final search
string settled on was
”(Creating OR enacting OR execution) AND (”data process* pipeline*” OR
”distributed data process* pipeline*” OR ”data pipeline*” OR ”unified data-
to-outcome platform”) AND NOT image AND NOT GPU AND NOT manu-
facture* AND NOT biology AND NOT ”case study””
The keyword ”processing pipeline” was removed because it gave too many
false hits with regard to processors. For similar reasons the NOT statements
”image”, ”GPU”, ”manufacture*”, ”biology” and ”case study” were added to
avoid papers that focused on very specific industries.
To validate the search strings, the number of search results was taken into
account. If the result was larger than a single person could realistically sort
through within the writing period, more restrictions would be added, and vise
versa, if the result was too small, the restrictions would be loosened. Here after,
the titles of the results would be quickly be looked through to find out if a large
amount of the results seemed irrelevant. If this were the case, exclusion criteria
would be added to further scope in the result.

3
Analysis of findings
In this section we will further dig into the included articles and start to answer
the research questions.
3.1
Current state of process mining pipelines
Even though pipelines are widely used in many branches of data science, such
at sensor data processing and machine learning, it is still have not caught on in
the process mining sphere, where it is still requires a lot of hands on program-
ming and tedious work. This seems true even though the field has more and
more techniques available, leading to a more tedious and error-prone analysis
process. This makes it harder to ensure repeatability and provenance, because
the pipelines have to be executed manually. [5]
4
==PAGE:58==
3.1.1
Research lines
When broadening the scope a bit from process mining pipelines to general
pipelines, it seems that most research is done in a few specific areas. These
areas include scheduling tasks as in [19, 15], data streaming as in [10, 3], and
behavior between organizations as in [18, 8].
3.2
Emerging terminology
When researching pipelines, an emerging terminology seems to appear.
Al-
though there are a myriad of ways to address them, the two most prominent
are certainly workflows or pipelines. The components that a pipeline is made
of are called nodes, blocks or tasks.
And these can come in three different
types sources, operators/transformer/flows (when talking streaming), and sinks.
[10, 18, 16]
When talking about the life-cycle of a pipeline, it seems to boil down to four
phases Template/composition, mapping/instance, execution and metadata and
provenance.
Templates/Composition is where the pipeline is told which
types of components are to be used and what kind of data is needed. It is also
at this step the pipeline that can be shared the easiest. The representation of
a pipeline in this step can be expressed in many ways such as directed graphs,
petri nets, UML and more. Mapping/Instance is where the components are
mapped to the specific component it is to be run on. this can either be done
by a user or it can be done by the system and leaves room for optimizations in
the pipeline or making sure is is run on the most resource optimal components.
Execution is where the execution of the pipeline happens. The actual data is
chosen and the components run it. Metadata and provenance is where the
history of pipelines will be stored. This step is important because it makes it
easier to reproduce the results. [7, 9]
3.3
Sharing of data
Sharing data between organisations needs different kind of considerations when
it comes to data security. One thing is data being sensitive, like patient data
from a hospital. Another being how resources are being shared.
Some research has been done on how sensitive data should be used when work-
ing in a cross-organization setting. In [8] an alpha algorithm is proposed that
takes into consideration how sensitive data from one organisation can be in-
cluded in larger executions. This works by running part of the algorithm locally
and sending results that have obscured the sensitive data to the larger model.
In [18], it is discussed how to optimize the sharing of resources between organ-
isation with regards to who gets to use what resources and how many. They
propose a framework for negotiating both by monetary means, but also with
how many resources can be used.

Discussion
Here the the findings of Section 2 will be used to answer the research questions.
First we will take a look at the current state of the art and where DAPM fits
into it. Then the leading terminology will be put up against how we have c to
do it in DAPM. Finally, we will discuss how the findings about sharing between
organisations fit into our project.
4.1
The State of Process Mining Pipelines
Even though the adaptation of pipelines in data analytics in general is wide
spread, it is still only in its infancy for the process mining domain. This seems
to be partly because of the overall maturity of the domain it self. As from [11]
However the demand for it is slowly picking up and tools that cater to demands
starting to be made. An example of this is RapidProm which is an extension
to RapidMiner, with this pipelines can be made utilising process mining tools.
RapidMiner can be considered a competitor to DAPM, but a need RapidMiner
does not fulfill is the need for different organisations to share both tools and
and data. The reason for this is that it is only works locally on a computer.
Here DAPM seeks to be build on a distributed architecture which allows for
easy collaboration between organisations and sharing of data.

4.2
Pipeline Terminology
From section 3.2 it seems like there is a terminology emerging and it is being

When comparing this terminology to what is used in DAPM it seems like we

nodes. These nodes can be a data source, a data sink or an operator.
A new thing we have learned from this is the terminology for how to describe
a pipelines life cycle.
Where we previous have had a mix of template and

The template and instance will be created on the front-end and the execution
will happen on the back-end. What is not yet included in our model is the
provenance, which will be important in the future, but is out of scope for our
project.
5
Sharing Data Between Organisations
When working with distributed architectures that span multiple organisations, it
is always a big task to maintain the privacy of both data from each organisation,
but also the tools being used.
One of the solutions found in section 3.3 was that the algorithms themselves
can be designed in a manner that made sure the data stayed confidential. [8]
In DAPM we have chosen to take a different route by instead making sure that
the data does not leave the organisation it belongs to and that the algorithms
are instead run on their system. This removed the need for designing specific
security aware algorithms.
6
Threats to validity of SLR
Some treats to the validity of this paper exists.
One being that documents
included during the exploratory phase could have been picked with some bias.
To combat this clear inclusion and exclusion criteria were made to make sure
only relevant articles were included. Another treat could be the source of the
main literature search is that it was only done on ACM and that is only included
articles labeled ”research article”.
It was later found out, that part of the
research articles from other publishers did not have this label. This means less
articles from other publishers were included that first assumed.
7
Conclusion
In this SLR we have explored existing literature to find a deeper understanding
of where process mining pipelines currently are at and where they are headed.
This showed us that the demand for pipelines in this domain are growing and
7
==PAGE:61==
that competitors are slowly being made.
It is also explored what the existing terminology for general pipelines are and if
they correspond to the DAPM project. Here we found that DAPM pretty much
follows the leading terminology, however some new terms were found, that will
be adopted.
Finally we looked at how cross organisation interactions are handled in the
literature.
We found that sharing of sensitive data can happen on different
levels, one being on an algorithm level and an other being where the algorithm
are being run.
