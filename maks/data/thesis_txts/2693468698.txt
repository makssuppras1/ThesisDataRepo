Abstract
The main purpose of this study is to identify validation approaches for data within big data
and build a functioning validation framework that can handle dynamic data, intended to
notify the user about erroneous or unwanted data. In line with the increasing demand by
stakeholders, as farmers, for taking advantage of IoT devices serving their businesses,
which means that data is increasing, and the possibility of erroneous data may occur, the
necessity for tools that can validate acquired data is obvious.
The success of a company or a product mostly depends upon choosing the highest qual­
ity data to be utilized in production. However, working with increasing data, can lead to
many difficulties along the way, as the data can appear in multiple formats and different
variations. Many companies need a quality check on data generated from different sen­
sors or other sources. As we have a collaboration with DTU and nitrogen sensor, our test
data will be focused on agriculture. Soil temperature is an important parameter when it
comes to sowing and irrigation of crops, therefore it is highly essential that it is validated
in relation to the weather and other parameters. To validate this data we will make use of
some validation strategies, which are in the form of adding predefined rules and custom
rules to validate desired data. The functionality of comparing parameters from one source
with parameters in an external source will also be included. The results of the different
validations approaches will also be displayed in a graphical manner.

Introduction
Applications are only as good as their provided data is. That has become the mantra
in the industry. Data is collected everywhere from weather stations, space to internet of
things like TV’s, appliances, or soil sensors etc. Data can be used to make us wiser in
several fields of research, data validation is the bedrock of data quality, thus validation
is a necessary step in data preprocessing. Businesses simply cannot afford to deal with
wrong data, whether the purpose is for legal compliance, for displaying it, collected from
Internet of Things sensors, or with the purpose of sales campaigns and business reports
[20].
Data is therefore a very important source in the modern world. If data is not validated from
the start, it will affect the value of the collected data and the output will not be accurate.
Therefore it is important to have some data validation done before analyzing, preparing
and presenting the data to the stakeholders. Although data validation is a meaningful step
in the workflow, it is often seen as an additional step that will slow down the pace of work.
Therefore the data validation is often skipped over even though it may provide value and
good results in the end.
The goal of this project is to research, how to make data validation available across var­
ious data seamlessly as well as software development of a framework that can validate
different kinds of data. These data can be sensor data or any other kind of entry for a data
base containing attributes. As a starting point the test data for our framework will come
from sensors which measures the soil’s moisture and temperature. Further in the project
process, we will use multiple data sources to compare values from one data source with
values from other data sources.
As already mentioned, this project is about developing a framework. The word framework
is defined in the English dictionary as ”a supporting structure around which something can
be built” [80] and in technical terms, the definition of a framework is a platform for devel­
oping software system and software applications. A framework can include predefined
classes and functions to help the developer on his or her journey for faster development.
The developers do not need to reinvent the wheel every time they develop a new appli­
cation. This definition of a framework can be transferred to a general idea of having the
user forming his content, according to a given purpose. For this project the purpose is
limited to allowing the framework to offer the user data validation opportunities.
Our project will address common data validation tasks, as well as relating them to larger
data sets (big data) by applying big data quality parameters, and statistical data validation
principals to the desired framework.

1.3
Scope
This study covers the aspects of data validation, the different methods and approaches
to it. In this thesis we are focusing on exploring the area of data validation, as well as
designing and developing a framework which is as dynamic as possible. Meaning that it
should be able to validate multivariant data from different origins. Available sensor data
with different measurements from agriculture in Europe is provided for this project, and
the initial utilization of the framework will involve the mentioned data.
1.4
Delimitations
This study is limited to developing a validation framework that will not consider various
existing data storage formats (like for example, XML, JSON, CSV, etc.), only focusing on
one as a proof of concept. Neither focusing on various data source types, different data
will be accessed from a single data base type. This does not mean that multiple data
bases are not taken into account, but the type of data storage application is of same kind.
Also, dealing with data validation often involves a prior step in order to make the validation
possible, which is the data collection part. This part is not within the scope of this project,
due to the main focus being on applying validation techniques on already collected data.
Machine learning will be addressed in this project too and the advantages of utilizing it for
data validation. That said, it will only be theoretically, without an implementation of the
concepts in the framework.

Big Data
Validating data within Big Data is crucial, to get the best quality out of it. The definition of
Big data would be the collection of large amounts of data from entities like web­browsing
data trails, social network communications, sensor and surveillance data that is stored in
computer clouds. Big data and analytics is significantly receiving attention in industries,
retails, banking, government and education sector. Plenty of companies and researchers
are recognizing the importance of data and are beginning to utilize big data for making
better business decisions. As an example for clarifying the importance of big data, the
retail industry is increasingly analyzing the buying behavior of its customers and the in­
dustry is using some predictive analytics to look for future consumers by utilizing external
consumer data from some social network websites. Plenty of people today has realized
the importance of big data in our lives. It has also been reported that more than half of
the time spent in big data projects goes towards data cleansing and preparation. There­
fore, with the data being so important, poor data quality would have significant effects on
company/business decisions [10]. Which makes big data quality an important and critical
issue.
2.1
Why Big Data quality
Poor data quality can cause expensive tangible and intangible losses for businesses. An
estimation made of revenues losses for a typical organisation, could reach as high as
8­12 % generating up to 40­60 % of the expenses of the service organization [77].
Other than the fact that poor data quality can cause an obstacle to revenue goals for busi­
nesses. It can also be a main cause for dissatisfied customers [24]
Poor data quality can also consume a lot of unnecessary resources. Even though, or­
ganizations usually would not know the importance of ensuring good data quality before
problems arises, and not before that seeking for solutions, and wasting large amounts of
labour as well as time [24].
Another motivation for ensuring good data quality is faulty data leading to the making of
poor decisions and by that low quality products. This would result in having products not
living up to expected quality standards, leaving hard work, invested time, and labor with
low value.

2.2
Big Data quality process
The process of data quality assurance consists of different stages. In this section, we will
explore the integration of data validation in that process.
The main stages of big data, in the process of ensuring data quality are, Data collection,
Data cleansing, Data transformation, Data Loading, Data analysis and reporting.
Dealing with big data quality assurance can be divided into five different stages [24]:
• Data collection
Collection is done from desired sources like storages and warehouses, communi­
cation messages, sensor data, mobile devices, or many others, under strict security
in order to ensure the integrity and availability of the data.
Data needs to be collected in order to be processed.
• Data cleaning
At this stage the purpose is to find corrupt or inaccurate or incomplete data parts
and correct or remove the unwanted data, from a data set, table or database. It is
notable that the removed data is corrupt data with no value. Unuseful data might be
useful in the future as new knowledge arises, therefore it is not part of the intent of
this stage.
• Data transformation
Data transformation converts the data formats from the source to the format of the
destination system. Big data formats can come as various types, structured or un­
structured, from different geographic places, real­time medias, different languages,
time series and networks. Therefore converting the data values to the format of
the destination is necessary, but original formats can be kept along in order to ac­
commodate future changes and needs concerning formats. This stage can also be
characterized as normalization.
• Data Loading
Data loading is the stage where data is loaded into a large data repository. Like for
example a Hadoop environment or another NoSQL big data repository.
• Data Analysis and Report
The Data analysis and report stage refers to the process of documenting and writing
reports about the data quality assurance process discussed. This stage will produce
reports regarding the quality degree of the processed data. Answering questions
like,how many valid or invalid records are there?, to what degree is the data valid?
Data consistency and completeness, and other quality dimensions are analyzed.

2.2.2
Big data quality parameters
Insuring big data quality can consist of multiple parameters, where validity is one of them
[49].
• Completeness
Defining the availability of required values in data sets. Measuring how the amount
of available data is compared to the desired amount for the intended purpose. This
measurement will verify if deficiencies in the data will impact the usability of it. Com­
pleteness is therefor the proportion of stored data hold up against the potential com­
plete data.
• Accuracy
Defines how accurate data are describing events or objects. This is done by com­
paring the data with known third party references data of same kind, which is con­
sidered true and accurate. Accuracy answers questions about the data regarding if
it is causing ambiguity or how well it is reflecting the the right data.
• Timeliness
Defines the arrival of data at anticipated time. This quality standard is important
within big data, due to the constantly emerging of new data. Measuring timeliness,
is done by marking the difference between the times for occurrence of an event and
when it is recorded. The purpose is therefore to detect the difference of what time
data are expected and what time they are readable, insuring the data has arrived.
• Uniqueness
Defines how redundant data sets are. Measurement of uniqueness is done by de­
tecting duplicates of data items within itself or amongst other data sets. This quality
check for big data sets, helps to eliminate redundancies.
• Validity
Defines how data sets follow defined rules. Validity measures if data conform to
specified rules for, for example syntax of formats, ranges or types. This insures that
the data is valid for their intended purpose.
• Consistency
Defines contradiction occurrences in data sets. This form of quality check, looks
for logical meaning within the data. This is done by measuring a data item with
itself or with other data sets. Focusing on how well the data is logically correct and
complete it is, for example for comparing multiple sources of same data item, that
should consist of same values each.
• Reliability of sensors
Defines the reliability of devices’ gathering of data. This area focuses on how reliable
networks for data gathering are, in a continuously changing network structure. It is
done by measuring how well a system is working according to its specifications given
for a specific time, checking if sensors are determined to be reliable.
10
==PAGE:18==
• Usability
Defines the data’s correspondence to specific needs. This area checks how well
the data meets the users’ needs, and is considered useful. This measurement is
done by correlating the other quality measure fore usability, timeliness, accuracy
and completeness, all calculated to a holistic quality parameter.

Data validation Concepts and Methods
Knowing that validation is a main branch for ensuring data quality for big data, we will in
this section dig deeper into general data validation and also narrow it into actual models
and typologies.
3.1
What is Data Validation?
At first it is important to define what data validation is and what it would mean to perform
data validation, in order to stick to a definition that we can carry throughout the research.
Data validation is a discipline that has been practiced before by various data scientist,
engineers and researchers. Therefore there is a large number of existing opinions, for
defining data validation. We have in this thesis studied the most relevant ones according
to our problem definition, and narrowed them down to a common definition that fits our
scope. One of the definitions given by ’Unece glossary on statistical data editing’ states
that data validation is:
”An activity aimed at verifying whether the value of a data item comes from the given (fi­
nite or infinite) set of acceptable values.”[66]
This definition covers the aspect of validation and defines it in terms of holding data up
against a set of acceptable values. This definition is appropriate as long as given set
of acceptable values are known to the validation system without considering unknown
parameters. Additionally, this act of validation is referred to as concerning a single data
item that needs to be validated, without explicitly embracing consistency among different
data items for verification. Besides digging deeper into the definition given by Unece, we
also considered other definitions. The following definition approaches the topic from bit
different angle, but still in between the same common understanding.
”Data validation is the act of falsifying data against explicit assumptions for some down­
stream purpose, like analysis, modeling, and visualization.” [4]
The given definition explains the process from the data’s perspective as being not suitable
for a given assumption. Which is relevant for some kind of purpose.
In addition another definition could be considered, given by the European Statistical Sys­
tem.
”Data Validation is an activity verifying whether or not a combination of values is a

3.2
Topologies and classifications of data validation
When it comes to abstracting the term data validation we can furthermore define a clas­
sification which can characterize, to what extent a given validation combination is in, due
to the variety of validation steps and procedures, compared to other combinations, ex­
pressed as different levels. And by that be able to tell, to what extent a given data set has
been validated by the validation procedures applied to it.
To do so, we will examine different classification typologies mainly adopted from statistical
points of view, due to the similarities for validation application on data.
In order to develop such a classification system, we need to consider which parameters
that can separate the different classes of validation. Such principles could be [83]:
– automated or manual validation.
– objective or subjective/expert knowledge based validation.
– structural or content based validation.
– the set of data being validated, whether it is: in­field, in­record, cross­record,
cross­data set, single domain, cross domain, etc.
– placement in the statistical process: input, throughput, output.
– logical type of validation rule: equality / inequality operations, more complex
logical rule.
In various literature data validation classification is defined from a practical perspective
like a specific segment or business, and also from a more abstract perspective, defining
the topic by generalizing validation acts for any purposes.
In the following sections we will explore some of those definitions, to be considered in
conjunction with supporting the aim of this project.
3.2.1
Practical Perspective
Data Validation levels by Sonoma Technology, Inc.
In a presentation given by the consulting firm Sonoma Technology, Inc. a clarification
regarding dividing data validation into three different levels, was explained. This presen­
tation was based on chemical measurements that needed validation. But the concept can
be generalized to any kind of data, therefore their validation taxonomy could be relevant
to include for a development of a data validation framework.
14
==PAGE:22==
The validation levels are split into three levels:
Level 0 – This level defines the lowest validation level as being, Routine checks, op­
erations from Fields and laboratories, data processing, standardized reports conducted,
review of unusual events, lookups in field data sheets, instrument performance checks
Level I – The second level looks for internal consistency. Checks are done for iden­
tification of values appearing atypically when compared to other values from the entire
associated data set.
Level II ­ At the highest level, checks are also done for consistency, but with external data
sets. Identification of values in the data that appear atypical when compared to other data
sets. [28]
Even though the purpose of the given for data validation levels classification originates
from a different yet specific segment or domain, i.e. the chemistry industry, an abstraction
could be derived out in order for it to be applied in more general purposes.
The above mentioned classification of validation levels shows a clear involvement of mul­
tiple data documents and them being from other sources going from the lowest to the
highest validation level. The lowest level mentions only individual events, for example a
document itself, the second level mentions other data from the associated data set, and
the last level compares the data with external data from outside the data set.
In order for validation checks to rise in level and by that complexity, a broader insight
using external reference data is demanded.
Data Validation levels from a business perspective by Sarah Geissing Katrin
Walsdorfer
Due to the plenty of options when it comes to data validation, it is preferable to have a
hierarchical definition of ’to what extent’ a data set has been validated, which we can call
a validation level, according to the validation procedures applied to it [83].
From a business perspective, which is a topology produced on the basis of procedures
used in national statistical institutes, the focus is on the different activities that data vali­
dation is performed with, as well as the needed amount of information and the phases for
the validation process.
For this approach it is assumed that there are two overall levels defined as:
A. Technical integrity, i.e., consistency with the expected IT structural requirements
B. Logical and statistical consistency of the data
[26]
Out of these two general definitions of validation levels, the following six validation levels
can be derived out.

The logical and statistical consistency of the data is split into these different levels:
• Level 0 : consistency with the expected IT structural requirements
• Level 1: consistency within the data set
• Level 2: consistency with other data sets within the same domain and within the
same data source
• Level 3: consistency within the same domain between different data sources
• Level 4: consistency between separate domains in the same data provider
• Level 5: consistency with data of other data providers.

The validation levels from figure 3.1 are explained in more detail in the following sections.
Validation level 0
The first and lowest validation level ’0’, is the most simple form of validating data docu­
ments. At this level individual documents can be validated without the consideration of
other documents, by for example checking only the document structure or the format of
the variables. This could for example be a naming convention following a specific format
with exactly x number of characters.
More examples are:
• Checking if a document’s column separator, is the right one.
• Checking if the document’s has the right number of columns.
• Checking if the document’s first column is of numeric type.
• Checking if the document’s second column meets a particular date­format.
• Checking if every required attributes and values are present in the file.
This validation level can be simplified as being ”file structure and format checks”.
Validation level 1
For this level, checks are also done solely using the individual file or document. The
difference is that the validation can include multiple variables from the same document.
Examples are:
• Checking whether the year from a column is in accordance with the year from the
data set’s title.
• checking whether the content of a column is consistent with the sender of the data,
data for a country should not be sent from another country.
• checking the consistency of two or more variables: by looking for a certain combi­
nation of codes that is considered illogical.
• checking the consistency of two or more variables: That the total inhabitants equals
the summation of male inhabitants and female inhabitants, or that Female inhabi­
tants equals (the total inhabitants / 2) +/­ a deviation of 10%
17
==PAGE:25==
Validation level 2
This level concerns checking consistency of the data based on comparing content of a
document with the content of ”other documents” within the same system or domain and
data source.
Examples are:
• ”Other documents” could be older versions of exactly the same document. In this
case the check is meant to detect ”revisions” or ”time series checks” by comparing
with previously sent data.
• Checking the consistency between three documents sent at the same time from
the same source: where each of the documents includes data from three different
categories.
• Annual data sets can be compared with quarterly data sets.
Validation level 3
Level three concerns checking consistency of the data based on comparing content of a
document with the content of ”other documents” within the same system or domain but
different data source compared to level two.
• Checks can be performed on same data sets but from different data providers.
• Mirror checks which verifies consistency of declarations between different sources
that refers to the same phenomenon, like, data showing the export from country A
to country B should correspond to data showing the import declared by country B
from country A.
Validation level 4
The definition of validation level 4 is that it concerns plausibility or consistency checks
among separate domains within the same data provider. Checks at this level could in­
volve the plausibility of results that are describing the same phenomenon from different
statistical domains.
Examples are:
• Checking unemployment from registers versus from Labour Force Survey
• Checking inhabitation of a dwelling from survey of owners versus from population
register
• Comparing external trade with international transport activity at ports.
Validation level 5
Validation level 5 could be defined as being checks of plausibility or consistency between
data from a data provider and reference data / information available from outside the data
provider.
18
==PAGE:26==
An example could be:
• Comparing soil moisture data with weather forecast data.
Summarizing, the different classifications of validation levels that where described above
indicates a growing degree of complexity from lower levels to the higher ones. Without
necessarily implying a growing technical complexity of the individual validation checks.
A rise in complexity might involve more parties and stake­holders, due to the diverse
parameters. Defining validation checks on the higher levels of classification might also
require more conceptual knowledge, potentially with different needs and requirements
regarding data validity.
3.2.2
Formal perspective
Decomposing metadata
In order to define a validation function based on metadata the following technical definition
is used.
v : S→{0, 1}
This function expresses a validation by a Boolean v, on a data set S. Out of the data
set S, single elements s ∈S an be data records or data fields. The Boolean function
outputs either a 0 (FALSE / invalid) or 1 (TRUE / not invalid) interpretations. This way
of expressing a validation by a function may seem different from the other practical def­
initions addressed above. Where definitions may for example be based on equality or
inequality constraints. Operators such as <, >, ≤≥or = can also be seen as functions.
When a comparison operator for instance ’<’ is used in a rule x < y it can be interpreted
as a function that yields a Boolean, 0 if x ≥y or 1 if x < y.[39]
This form of validation distinguishes between solely testing whether a given relation holds
or not on a set of data points. And interpretation of failures, digging deeper into what
individual data point caused the violation of the rule, which is called error localization and
which these expressions of validation functions do not take into account.
For distinguishing between different types of validation functions, a model can be made to
decompose metadata of individual data points which are elements of the data set s ∈S.
A data point is defines as a recorded value composed of a universe U which is roughly
the type of object that the data originates from, the time the measurement took place τ,
the selected element u and the observed variable X, ending with (U,τ,u,X).

3.2.3
Comparison of classification concepts
In this section we will discuss the similarities and differences between the previously de­
fined data validation concepts. The discussion is seen from a theoretical perspective of
both viewpoints, and the validation levels will be correlated to each other by examples
for illustration. The discussed level classifications are the ones seen from the practical
perspective and the ones seen from the formal perspective. We would like to explore how
these two perspectives relate to each other and what parallels can be drawn out of these
diverse views.
Theoretically, the differences between the addressed rule classifications are related to
the principals they each include in order to distinguish between data validation rules. As
an example, if we address the validation levels defined from the practical perspective
(Section 3.2.1) it is clearly that they are inspired by some form of domain, statistics as an
example. For statisticians, data are stored in batches of files (data sets) arrived from a
data source that regards a specific domain. That data can origin from a single provider
(an institute), or it can origin from multiple providers. Therefore if we address the principle
of validation rules that are labeled as being validation rules for data from single or multiple
sources, it is clearly that the interpretation of the meaning ’source’ strongly depends on
the procedure of the data management and handling of data.
It could occur that one data scientist that wants to perform data validation for two given
data sets, receives two files and apply a validation rule that correlates the two files, and
in that sense there was performed a cross­file data validation. Then the data scientist
merges the two files into one and sends it to another scientist. The other scientist perform
the exact same validation, but in this case the validation have concerned an in­file valida­
tion, due to the two files being merged into one and the validation rule has all the needed
information in one file. This possibility of turning definitions by to some extent subjectivity,
is a problem especially when a comparison is performed for validation levels.
On the other hand, the classification for data validation defined on the basis of a formal
perspective, is based on the metadata of the data for validation, by decomposing the in­
formation into the domain, the measurement time, the observed object and a variable. By
taking into account that all the indices for the data, such as the ’domain, the measurement
time, the observed object and a variable’, cannot be selected completely independent of
each other. The characterization of rules gets classified according to if they regard single
or multiple domains, objects etc. The formal classification of validation rules can be seen
as a supplement, due to the fact that the advantages and disadvantages of the formal per­
spective are similar to the advantages and disadvantages that can be point out regarding
the practical perspective of validation rules. The main advantage of classification of vali­
dation rules according to the formal topology, is its objective view of data, by considering
metadata. The disadvantage would be that it requires greater effort analysing the data’s
conditions.

the explanations are based on assumptions due to the abstractions of some of the rules.
Example rule 1
An attribute ’location’ should only be of entries from a specific list of field locations.
The mentioned rule checks whether the value of a variable called ’field’ is part of a static
list containing allowed values. Matching this rule with the defined rule classifications, it

If a leaf area index (LAI) of a reference period is higher than the current period, the at­

Assuming that the LAI of the reference period is measured at a different time than the

LAI and the attribute of harvest. According to the formal classification this type of rule
would be labeled as being level 2 (smsm), and according to the practical classification the
same rule would go under level 2 also, due to it validating ’previously sent data’.

If the the number of precipitation types for an area is two (rain and snow), the two types
must be in the file of the precipitation types record.
This type of rule considers in a statistical manner two universes, the area and the precipi­

classification would correspond to level 3 (msmm).

As mentioned before, data validation is a general practice, performed by businesses and

Before explaining what automated data validation is, it is important to know what a data

in one place using the extract, transform and load method (ETL). The data within a data
warehouse can for example be historical sales data or employee data etc. Data ware­
house is basically used for decision support, reporting and analytical purposes. So basi­
cally the purpose is to check and compare the quality of the finding data up against data
contained in the data warehouse using queries. The data warehouse testing comprises
a large data volume, which significantly affects productivity and performance.
Data warehouse has two types of validation functional and non functional. The functional
validation contains standard validations steps and business validations steps: Functional
Validations [56]
• Standard Validations: Date validation, Count validation, End to End testing.
• Business Validations: Straight/Direct move, Data transformation,Look up validation,
Defaulting, Signage, Translation, Filtering, Average balance, calculation, Data in­
tegrity validation, External field validation.
Non Functional Validations
• Load testing
• Volume testing
Because of the huge data within the data warehouse, the manual data validation method
is time­consuming and not perfect due to the large volume of data and complex transfor­
mation rules. Therefore another approach called automated data validation is introduced.
This method contains two phases. The first is the test preparation phase, here various
things are done like splitting the requirements for test conditions. Make SQL query for
each test condition for each rule in requirements. These queries must show pass or fail,
depending on the outputs obtained when testing source data with target data. Store all
the SQL queries along with test conditions in MS Access database in one table. Then a
module is created in MS Acess database which will run all queries on the test data and
collect the test result in a separate table. This result table is then used for identifying the
conditions where the data is incorrect in the target source or target database.
The second phase is called test execution. This includes importing the source and the
target data into the MS Access tables and then run the module to validate the target data
using the written SQL queries in previous phase.
The automation strategy provides some advantages like the correctness of data and data
consistency. The test process is less time­consuming than the manual test approach.
This strategy does not depend on the client database technology [56].

Machine Learning
The idea of having machine learning for data validation is a posibility. Data validation is to
this far primarily performed either by manually ’eye­balling’ through data, or in more effi­
cient approaches performed automatically through logical checks with thresholds. Using
automatization without machine learning, can perform sufficient data validation that can
lead to correction of data errors. Similar results can be achieved by applying machine
learning to validation of data, without replacing the existing automatic mechanisms but as
a complement, automating mechanisms that can go beyong human defined thresholds
and logical checks.
Machine learning can be used for forecasting of values which can the basis of rejecting
data values with the purpose of validation. Used algorithms can be dependent on already
validated data, in order to prevent erroneous learning by accepting wrong data, making
it impossible to distinguish between right or wrong. Which is why automatic validation
techniques can go hand in hand with machine learning.
Machine learning is basically statistical learning which has been practiced soon in the
course of a century. The algorithms are a collection of regression and classification tech­
niques, respectively if the aim is to predict quantitative or categorical data. What differs
the past two decades is the rise of use of machine learning algorithms in applications, due
to the increase and availability of stronger computing power, larger memory for computing
devices and the availability of larger quantities of data [33]. There are three forms of ma­
chine learning algorithms: Unsupervised learning, Reinforced learning and Supervised
learning.
Unsupervised learning
Is primarily not used for prediction of data, due to the lack of having an associated re­
sponse variable. This type of algorithm would rather be applied for data analysis and for
finding patterns in data.
This machine learning technique do not need the users supervision prior to discovery of
patterns. Instead the model works on its own discovering patterns and previously unde­
tected information, mainly dealing with unlabeled data.
An important aspect regarding unsupervised learning is clustering, which basically means,
grouping and structuring data according to information it receives. By processing the data,
the algorithm will find natural categorizations for the data, putting them into clusters or
groups [67]. Some of the types of clustering for unsupervised machine learning are:
• Hierarchical clustering
This algorithm builds clusters in a hierarchical structure. Starting with data that is
assigned to a cluster of its own. Then every two close clusters are included in the
cluster, ending when there is one cluster left.
• K­means
Is an iterative clustering algorithm, providing the highest value of each iteration. This
32
==PAGE:40==
algorithm clusters data into a desired k number of groups. It assigns the data points
to one of the k groups, and eventually outputs a group of ’labels’.
• K­NN (k nearest neighbors)
The K­nearest neighbour algorithm differs a bit from the others, due to it doesn’t
produce a model. Instead the machine learning classifier stores the available cases
and classifies new instances according to similarity measures.
• Principal Component Analysis The principal component analysis machine learning
algorithm finds subsets of a larger base, which is smaller in size compared to the
original data points. It clusters data points together which are highly correlated to
each other.
Reinforced learning
Reinforcement learning is an algorithm learning itself by trial and error in order to obtain
a solution to a given problem. The algorithm learns to make a sequence of decisions,
achieving a desired goal in an uncertain and complex environment. In order for an ap­
plication utilizing a reinforcement learning algorithm, the machine learning algorithm gets
either rewards or penalties for its performed actions. The algorithm’s goal is to obtain
as many rewards as possible. Even though it is the developer that defines the policies,
which corresponds to the rules of the game, the model will get no hints or suggestions for
solving the given problem. This would be up to the algorithm to find a solution to how a
task should be performed in order to maximize the rewards [53].
One of the downsides of using reinforced learning is their chance of optimizing their prize
without performing the actual task it was designed to solve. Another challenge when
using reinforced learning is the importance of preparation of very realistic simulation envi­
ronments, which the algorithm is highly dependant on. Developing a reinforced learning
system out of a training environment to real life scenarios can be a challenge.
This type of algorithm can be useful for specific applications, not necessarily relevant in
every context. What differs reinforcement learning from unsupervised learning is related
to their goals. For unsupervised learning the goal is to find similarities and differences in
data. While reinforcement learning aims to find the most suitable action model that maxi­
mizes the cumulative sum of rewards for the algorithm.
While reinforcement learning is an other type of learning algorithm compared to unsuper­
vised learning, by learning from rewards and punishments as signals for right and wrong
behaviours. It also makes it different from supervised learning, where provided knowledge
is key for prediction [54].

Supervised learning
Various algorithms exists for supervised machine learning, each working with regressions
as well as classification. These learning algorithms are categorized into three subtypes,
linear models, nonlinear models and tree­based models. The process of the algorithm is
split in two phases, a learning (training) phase, and a production of a predictive model.
The phases work by introducing a large amount of data to the algorithm so which it can
detect patterns out of and eventually learn to produce a predictive model. Commonly
amongst all the supervised machine learning algorithms, is their ability of predicting vari­
ables using independent variables in data sets. How the actual algorithm would run, is
what differs the algorithms from each other [19].
An example of could be made for staff employees. Each staff could either work as one
of the four classes: Professor, lecturer, research assistant and administrative employee.
Hence the staff category would be our dependent variable, and the independent variables
would be for example: gender, full_time_employee, age, university_field and nationality.
The proposed function, showing the independent input parameters and prediction result
output would be:
Staff_category = f(gender, full_time_employee, field, age, nationality, university) [19]
The algorithm would start training the prediction using the available historic data which can
be considered as delivered knowledge data. After the training has been done, new data
can be introduced to the already trained algorithm in order to test the new data. The al­
gorithm would compare its predicted variable with the ones observed from before. The
algorithm not only predicts the most likely value of the variable to be considered, it also
finds the most unlikely value for the variable. This can be useful for finding most unlikely
cases, where the values could be proved wrong by having the lowest probability of being
correct.
It is important to emphasize that the prediction of the algorithm not necessarily imply that
a particular dependent variable is wrong. The error could be traced to another of the inde­
pendent variables. The actual benefit of algorithm’s output could be formulated as being
pointing out that something is wrong with the data record, not necessarily what is wrong.
Feedback limitations
One limitation that will be faced when applying machine leaning algorithms in conjunction
with data validation, is its ability to present the cause of violation. Unlike statically defined
validation rules, the machine learning algorithm does not have a specific defined Boolean
pattern that can explain a specific data record’s violation, and which can be forwarded
to stakeholders directly. Define static rules could be: variable’s value not being from a
specific list, or a variable containing an illegal value due to the measurement time being
between a specific time range.
Instead the machine learning algorithm predicts an error, which cannot be forwarded to a
stakeholder stating that something is wrong, not knowing what and why. This combination
of prediction performance and interpretability is called the ‘Ockham’s dilemma’ [19].
Supervised machine learning algorithms could be very useful applying it for data valida­

confirm whether it is correct or not, within a probability.

could be incorrect, or the observation could be incorrect, or also there is a possibility that
the specific case could be very unlikely but yet true. In any of the cases there will always
be a good reason why divergences arises. But the possibility of false positives will always

Cerberus
Cerberus, a library made for Python programming language, it is customized for data
validation. This library provides functionality designed for simple validation tasks yet ex­
tensible for supporting custom validation use cases.
The plug­in works by initially setting desired schemas, before using the defined schemas
for matching future introducet data records up against. A schema is basically a copy
of a data record including constraints instead of values for each defined variables. The
schema is after initialization passed to a validate() function which takes it as an input and
the desired input data for validation as well. Cerberus offers a set of already defined check
functions such as if a field is required, minimum or maximum possible value for a variable,
and more, which can be utilized for setting a desired schema.
After completion of validation, in case any of the input data fails the validation an error
message of the violated rule can be extracted, explaining the cause of violation [9].
Cerberus is a very flexible library allowing various checks. Besides many obviously rel­
evant checks for variables, Cerberus allows for extension of the existing implementation
by subclassing the Cerberus Validator class and adding a custom method, the custom
defined method can then be referenced in the schema and it would work as an integrated
part of the validation library [21].
The Cerberus library is not version specific. This is one of the advantages of using Cer­
berus, which makes it is not dependent of specific Python versions, allowing a strong
flexibility when developing a comprehensive application. The library is also easily acces­
sible through pip (Python programming repository) [51]. Cerberus supports various types
of data input formats, making it applicable for Dictionary, JSON, YAML, CSV and other
formats.
One of the downsides of the library, is that the validation function within the library does not

with single validation schemes each. This can lead to limitaions of correlation algorithms,

Design
Our goal with this chapter is to apply the knowledge we have gained from the previous
chapters to design our data validation framework. A validation framework is a system or
environment where validation of data is possible. This validation framework or system
can be in multiple types for example mobile application, website, or an API (Application
programming interface). The choice of system type depends on the use cases and the
target audience. In our case the target audience are data owners who already have some
knowledge about data analysis. We could choose to develop the framework as a mobile
application, but it would not suit the specified use cases, having the user to add rules and
display results on a small screen when dealing with big data. It would simply not fit users
who work with data analysis using mobile applications to do their work.
Another option could be to develop the framework as an application programming inter­
face (API). But this would also not be an optimal solution, because it would not be the
user interacting with system but the users program. Our target group must be able to
interact directly with our framework without a middle party. Therefore, we intend to use a
web­interface out of other options for system types.
The system we are designing will be split into two areas, basic validation and custom
validation fulfilling use case 1 to 4. In order to provide basic validation checks we need
to design a framework that is able to apply basic rules for any kind of data. Therefore the
basic validation checks from 2.4, are considered for the basic validation part. In regards
to the second part, which is the custom created validation part, requires a design that is
capable of amending continuously.

characteristics, we chose the Python programming languages for our main developing
language, over the other programming languages, because of its many advantages. Be­
fore programming in python, we also had to consider the right editor for our development
environment. There are several editors for python. After some research on the internet,
we found out that pycharm editor was the most obvious choice. Pycharm contains a built­
in terminal, this makes it easy to run code in the editor instead of having the terminal in a

Correlation algorithm
When dealing with multiple data sources, one of them has to be interpreted as a refer­
ence data source. For our implementation, the rules for the first data source would be
interpreted as the main data, and the second as the reference data.
In order for proper correlation of variables, an aggregation has to be done on the basis
of context that the multiple data records have in common, which could be the timestamp.
Multiple data can be stored in different resolutions, which means that a comparison would
not be viable, if the main and reference data sets are imbalanced.
For this issue, we have implemented the functionality of comparing only the data records
having the hour of the day in common, per day, according to the defined time range limit.

from the main data. For example for sensor data, measurements of temperature can vari­

7.3
Security considerations
Considerations can be made according to the security aspects of the framework. Security
is a very important part of systems dealing with private and sensitive data. This data
can for example be owned by private companies. These private companies would not
allow if their competitors were able to access their private data through a framework /
website. Or data can contain personal sensitive data, which should not be accessed by
unauthorized people. According to the CIA triad (Confidentiality, Integrity and Availability)
[11], especially two of the three topics from the triad, can be mapped to security concerns
tied to our framework.
Confidentiality
One thing to consider regarding the security of the framework is the visibility of the data,
to other than the authorized user.
Retrieving data can expose the credentials for a data base, therefore connection to the
data base should be established without transferring the credentials in clear text.
By using the pymongo library to connect to the MongoDB, SCRAM­SHA­256 (Salted Chal­
lenge Response Authentication Mechanism) is used by default for authentication [46]. The
actual authentication requires nothing but a username, a password and a name for the
database. The password will never be transmitted through the network in clear text, so
an adversary would never get the password itself. There is a possibility that the hash
could be exposed, but because the hash is salted with a challenge, a man­in­the­middle
adversary will only be able to use it only for a single authentication process.
Integrity
Regarding the integrity part, imitation of the authorized user could occur. Therefore a
security mitigation of a CSRF (Cross site request forgery) attack is addressed. This form
of attack allows a malicious user to execute actions using the credentials of an authorized
without the authorized user’s knowledge or consent [18].
The used Django library has a built­in protection against most types of CSRF attacks,
providing which we have enabled and used where appropriate. The CSRF protection
works by including a secret in each POST request, which will be checked for validity. This
will ensure that no adversary will be able to replay a request of POST form, to the web
page of the framework. Due to the adversary needs to know the secret, which is a user
specific cookie.

Conclusion and future work
9.0.1
Conclusion
During this project, we have managed to answer of our research questions regarding how
a data validation framework could be built in order to support validation of various multi­
variant data. The knowledge we have gained from our research and analysis process has
given us the properties to be able to understand the quantity of data validation and to build
a validation framework that can handle multi­variant data of different types. We gained
a more solid understanding of different validation libraries and programming languages
to build a flexible validation framework. On the basis of our research about the defini­
tions and approaches to data validation, gained us insight to the characterization of data
validation. And based on our analysis we encountered the different possibilities for data
validation and their advantages and disadvantages, for applications or libraries, related to
the programming languages for the utilization.This has resulted in the development of a
framework that accommodates the defined user requirements for a data validation frame­
work allowing for validation of multi­variant data that can be fetched from Mongodb.
During the project some challenges have been faced, which have paved the path for the
development. A decision, based on our analysis, was made to utilize the R package and
Cerberus libraries, but tests made us exclude the R package from the framework. Our
intention for the framework was to benefit from the validation results, allowing the user to
take further actions, which the R package did not support sufficiently. These advantages
were found in the other library, that our implementation was based on (Cerberus), yet
introducing general validation challenges the higher classification of validation levels we
achieved.
The implementation have proved that forcing dynamicity can be cumbersome, due to the
lack of normalization of diverse variable conventions leading to issues when validating
across multiple data sources. Without centralizing and normalizing the data.
Disregarded the faced challenges we managed to implement a framework for data vali­
dation to the most flexible extent as possible.
9.0.2
Future work
A Cerberus scheme cannot contain diverse variables in one single scheme, which means
multiple schemes are generated, and that means weak performance. This is a needed
improvement for the future development of the framework, which is also motivated by the
idea of increasing the capacity for more incoming data to be handled by the framework
along with its possibility of validating as dynamically as possible. Besides optimizing the
validation software for better performance and the dynamicity of it, machine learning is
also an essential area for the future development. By introducing machine learning, like
supervised learning, a much more sophisticated validation of the data can take place,
allowing anomalies to be detected without requiring the knowledge of the user.
