1. Introduction 
1.1 Advanced Driver Assistance Systems overview 
The unhappy consequences produced by car accidents, resulting in material damages and even 
lives losses, are well known worldwide. The technological evolution aimed to improve vehicle 
safety and ease of driving by designing systems that automate, adapt and enhance the control 
functions of the vehicles. These type of systems, which may individually control different driving 
functions, are generically known in the specialty literature and the automotive industry under the 
name ADAS (Advanced Driver Assistance Systems).  
 
A classification of ADAS altogether with their performance evolution in time for auto vehicles 
according to the provided automated functionality can be found in article (Shaout, A., Colella, D. 
et al., 2011). This classification contains: 
 
a. Cruise Control Systems - systems that automatically controls the speed of the car. 
 
 
 
Fig. 1.1 Cruise Control System Block Diagram 
 (Shaout, A., Colella, D. et al., 2011) 
 
b. Advanced Cruise Control Systems - enhanced cruise control systems that relies on a 
sensor that scans the area in front of the vehicle. 
 
 
 
Fig. 1.2 Advanced Cruise Control System Block Diagram 
 (Shaout, A., Colella, D. et al., 2011)

c. Pre-crash Systems - systems that are able to detect imminent accidents and take safety 
measures to limit the potential damages, such as: precharging the brakes, adjusting the 
seats for added support, tensioning the seatbelts.   
 
 
 
Fig. 1.3 Precrash System Block Diagram (Shaout, A., Colella, D. et al., 2011) 
 
 
d. Blind Spot Information Systems - systems that are meant to improve the ease of 
driving, by giving the driver visibility to areas that are very difficult to be seen (also known 
as blind spots).

The classification mentioned above is definitely not exhaustive, but it clarifies at least the 
background of the ADAS purpose and functionality. It can be easily observed that the functions 
accomplished by ADAS are directly related to human safety. This aspect implies that ADAS is a 
system with safety-critical functionalities. By definition, safety-critical systems are “those systems 
whose failure could result in loss of life, significant property damage, or damage to the 
environment.” (Knight, J. C., 2002). That makes the ADAS hardware and software development, 
a process of high complexity and responsibility, that should follow some clear specific rules and 
stages specified by the corresponding functional-safety standards. For ADAS used in the 
automotive industry the safety guidelines are specified by standard ISO26262 (ISO 26262, 
2011). 
 
Another particularity that must be considered for ADAS consists of the time of reaction that these 
systems must have when handling different situations in which specific actions are supposed to 
be performed. These actions have to be executed within a specific time limit, such that the result 
will be optimal for human safety and the risk minimized. This aspect confers to ADAS real-time 
systems characteristics. 
 
There are more definitions for a real-time system. A real time system may be viewed as a 
“system whose logical correctness is based on both the correctness of the output and their 
timeliness.” (Laplante, P. A. and Ovaska, S. J., 2012). A definition that better matches the ADAS 
context is: “A real-time system is a computer system that must satisfy bounded response-time 
constraints or risk severe consequences, including failure” (Laplante, P. A. and Ovaska, S. J., 
2012). 
 
Next aspect that should be mentioned about ADAS refers to their system architecture. It can be 
observed from fig. 1.1 to fig.1.6 above, that these systems are made of different components, 
depending on the functionalities that have to be accomplished. Even though their components 
are heterogeneous, a general architecture pattern can be deduced. Therefore, an ADAS system 
architecture is composed of three building functional blocks: 
 
a. Input block - made out of different sensors and input devices 
 
b. Processing block - contains the platforms that implement the algorithms used  for 
identifying and reporting the situations, when specific actions should be taken 
 
c. Output block - made of different output devices/actuators that, based on the information 
received from the processing block, will take the corresponding (safety) measures (either 
by signaling directly to the driver or controlling a functional unit that is supposed to adjust 
any parameters in the vehicle behavior). 
 
Diving a little deeper in the system architecture and discussing at component level, it can be 
observed that most of the ADAS components are from the embedded systems category. 
Because in most of the cases, the information to be processed come from multiple inputs and the 
actions must be taken under timing constraints, these embedded systems must rely on 
multitasking real-time hardware platforms. In some cases, the system must also be able to emit 
multiple outputs. In the real time systems world, the capability of dealing with multiple inputs and 
outputs is achieved by implementing a preemptive priority scheduling policy. This policy involves 
that the system offers support for executing different tasks, that process input data from different 
sources, which are assigned different levels of priorities depending on their safety importance. 
These tasks will gain their execution slot depending on their priorities, i.e. if two tasks of different 
priorities need to be scheduled for execution, the one with the higher priority will be chosen, if 
they compete for accessing the same resource. At the same time, if a high priority tasks needs to 
be scheduled, a lower priority task currently executed will be suspended (pre-empted) and the 
high priority task will gain execution access. This policy guarantees that the systems will be able 
to perform the crucial safety functionalities within the fixed deadlines.

1.2 ADAS in agriculture 
 
ADAS and their functionality have been described so far in the context of automotive industry. 
However, this type of systems has been also designed for other industries and activity areas, in 
order to enhance specific processes. Agriculture can also be found among these areas. 
 
ADAS have been designed and introduced in agriculture as a technical solution for a sensitive 
problem of present days, that being the gap between the continuous population growth and the 
quantity of food being produced. The usual production processes are becoming gradually 
inefficient in producing the required amount of food needed. For example, an agricultural 
machine equipped within an ADAS, can exploit in a more efficient way, a difficult accessible area 
than using basic human labor methods. Moreover, introducing autonomous farming mechanisms 
solves a demographic problem: the lack of qualified personnel for handling the agricultural 
vehicles since there is a continuous trend of population movement from rural area to 
industrialized urban areas. 
 
Last but not least, some of the ADAS designed for agriculture are environmental friendly, since 
they provide the agricultural machines with safety functionalities that permit agricultural machines 
to autonomously handle significant physical hazards in the field, such as the presence of animals 
or human beings in the machine’s activity area. Once animals or humans are detected, the 
ADAS will issue some specific commands to the agricultural machine in order to avoid harming 
the living obstacle.

1.3 General ADAS system architecture considerations in 
agriculture 
 
In the first section of this chapter, there have been discussed some general aspects about 
ADAS system architecture used in automotive industry. However, the functionalities that 
must be accomplished by an ADAS used in agriculture differ (as it can be noticed from the 
previous section), altogether with the environmental conditions. This will be reflected in 
different challenges that must be taken into account when designing an ADAS system 
architecture for agriculture. 
 
The agricultural activities take place in general in large, open fields not on limited areas 
which are well delimited and marked in lanes, such as highways, where auto vehicles are 
driven. This involves, at first sight, that a larger area has to be observed and monitored. The 
agricultural terrain does not have a flat surface like a highway. It presents lots of uneven 
surfaces that should be detected as obstacles and avoided sometimes, in order to prevent 
damage being done to agricultural machines. Fields are not as well delimited as highways, 
therefore, the agricultural ADAS must be able to distinguish more accurately which are the 
bounds of the area where the agricultural machine is allowed to perform its work.  
 
The possible set of obstacles is much larger in the case of an agricultural field. The 
obstacles may be represented by terrain irregularities, trees, animals, and even human 
beings that can appear in the working area of the machine. Things get even more 
complicated in the case of animals, because their colour might be very close to the field 
colour, making them very hard to be differentiated from the ground and, implicitly, very 
difficult to be detected. The weather and the luminance conditions are also additional factors, 
that might make the obstacle detection for an ADAS used in agriculture a very difficult task. 
 
All these challenges must be considered when designing the system architecture for an 
ADAS used in agriculture. This will be reflected at many levels of the systems, starting from 
the needed sensors (high performance imaging sensors/cameras, LIDAR sensor for 
accurately detecting the inclination field irregularities or thermal sensors for detecting living 
obstacles when the detection based on images is not reliable enough etc.) and continuing 
with higher performance object detection algorithms, as the variety of possible obstacle is 
very large. Parts of these aspects will be considered and explained in the section 2.1, that 
will present the design decisions for the agricultural ADAS system presented in this thesis.

a trivial task since the system must be able to face many different types of challenges. These 
challenges will be handled by different components of the system, each of them with its specific 
functionality.

implement a specific functionality within the system. The considered benchmarked algorithm

analyzed by the system in order to take an accurate detection decision.        
 
The obtained results propose to answer the question: Does the new considered component and 
the algorithm bring a performance improvement in comparison with the system that

2. System architecture and requirements 
2.1 System architecture block diagram 
 
A proposed system architecture for an ADAS system used in agriculture contains the following 
functional blocks: 
 
a. Input block  
b. Non safety critical block  
c. Safety critical  block  
 
This section describes the overall functionality of each of these blocks.  
 
The input block contains a collection of input data gathering elements. This block acquisitions 
all the data needed to be processed, so that the system will be able to fulfill its functionality. 
Practically, this block is composed of a set of different sensors, that gather different types of 
information. Besides the data acquisition, this block is also able to forward it to the next block in 
the system architecture hierarchy that is in charge with the data processing steps, i.e. the non 
safety critical module. A detailed description of each sensor type can be found in sub-section 
2.1.1.  
 
The non safety critical block receives the different types of raw input data from the input block 
module and performs the corresponding processing on this data. The overall purpose of this 
process is to convert the raw data in a format that is understandable and useful for the safety 
critical block. Once the processing is done, this module is able to send the required data to the 
safety critical module, using the communication infrastructure that interconnects them. The 
processing elements that are part of this block will be detailed in subsection 2.1.2. This block is 
considered to be non-safety critical, as it is not supposed to implement functionalities that must 
be compliant with the safety critical requirements. 
 
The safety critical block is in charge with providing the control and decision logic of the system. 
Based on the processed data received from the non-safety critical block, it will perform as an 
actuator of the whole system, by taking a set of decisions that will be converted in specific 
commands for the controlled agricultural machine. Due to the importance and the required 
accuracy of these decisions, this block must be compliant with the requirements specific to a 
safety critical architecture.

2.1.1 System sensors 
 
The required sensors for the considered systems architecture and their role within the system are 
mentioned in (Christiansen, P., Hansen, M. K. et al., 2015), that proposes a similar architecture 
for an ADAS system used in agriculture. This section will enumerate them and will briefly 
describe their functionality. Overall, the considered system architecture contains the following 
sensor types: 
 
a. imaging sensors  
b. position estimation sensors 
 
The imaging sensors consist of two different types of cameras: a thermal camera and a stereo 
camera. 
 
A thermal camera sensor represents an imaging sensor that captures the heat radiation in the 
form of different intensities, depending on the temperature of the environment. The functionality 
of a thermal camera is to identify living objects from the variation in temperature, in different 
climates and weather conditions. These type of sensors present some certain strengths, but also 
an obvious weakness at the same time. The strengths are given by the good object detection in 
difficult environment conditions, such as night or foggy weather or an environment where animals 
may suddenly appear or be camouflaged. The weakness is due to the fact that the detection 
loses its accuracy once the ambient temperature is very close to the ambient temperature. On 
the other hand, the detection is limited by the physical characteristics of the thermal camera. The 
living object detection remains low for a cheap thermal camera with limited resolution and limited 
visual characteristics (Christiansen, P., Hansen, M. K. et al., 2015). 
 
A stereo camera is an imaging sensor that is able to capture 3D images. Both the depth and the 
colour information may be exploited from the captured images, enabling the possibility to perform 
the detection of an object based on visual characteristics (shape, colour, texture) and to 
determine also the relative positioning of that object to the vehicle. The main drawback of the 
object detection, based on the information captured by a stereo camera, is that the gathered data 
loses its accuracy in unfavorable weather conditions and during the night (Christiansen, P., 
Hansen, M. K. et al., 2015). 
 
The position estimation sensors consist of a LIDAR sensor and a RADAR sensor. 
 
A LIDAR sensor measures the distance to a target object based on laser light that is used to 
illuminate the object. Like a stereo camera, it can also be used  to acquire depth information, 
even of a higher performance, in terms of distance range and object orientation. The information 
captured from this type of sensor may be useful for the object detection, only in particular 
environmental conditions. It still presents major inconveniences, because it cannot make any 
distinction between some environmental conditions such as: weather change, illumination, 
protruding of other objects in the environment (Christiansen, P., Hansen, M. K. et al., 2015). 
 
A RADAR sensor is a position estimation sensor, that may be used to measure both the 
distance to an object and its velocity. Its functionality is based on sending radio waves towards 
the object direction and measuring the received reflection. The main advantage of RADAR 
sensors, in comparison to the LIDAR, is their insensitivity to the different environmental 
conditions mentioned above (Christiansen, P., Hansen, M. K. et al., 2015). 
 
 
16 
 
==PAGE:17==
Overall, all the sensors that will be part of the system gather useful information for performing 
living object detection in an outdoor environment. However, the specific information acquired by 
each of them, might be more or less accurate depending on the environmental characteristics 
such as climate, colour, illumination or the objects` physical characteristics (colour and size, that 
might make them very difficult to be identified in that environment). A combination of these type 
of sensors seems to well balance their particular strengths and weaknesses, making the living 
objects detection more accurate in any of the environmental conditions mentioned above.

2.1.2.1 Nvidia Jetson TX1 board general presentation 
 
The non-safety critical block contains the NVidia Jetson TX1 board (NVidia Jetson TX1 board 
overview, 2016), that is part of the Jetson board family, which was specially designed by NVidia 
for providing a powerful hardware support for the computationally intensive algorithms, from the 
deep learning and the computer vision areas. This family contains two different boards: NVidia 
Jetson TK1 that supports an ARM 32-bits Instruction Set Architecture, and the Nvidia Jetson 
TX1, which represents the newest version of Jetson boards designed to support an ARM 64-bits 
Instruction Set Architecture. 
 
The most important particularity of Jetson TX1 resides, in the fact that it presents a very complex 
hybrid CPU - GPU hardware architecture, that supports a large amount of parallel computations. 
The CPU chipset is part of the Tegra family which represents a series of SoCs (System on Chip) 
designed by Nvidia for mobile devices. The Tegra X1 chip is composed of a multi-core CPU 
containing four Cortex-A57 cores, four Cortex-A53 cores and a GPU made of 256 Maxwell 
architectures CUDA cores. A more detailed technical overview of the capabilities of this chip can 
be found in (Tegra X1 chip review, 2015) and (Whitepaper NVidia, 2015).

This system architecture has been considered in such a manner, that the system should be able 
to perform all the functionalities that are expected from it. The purpose of this system is to be a 
fully functional ADAS, which can be mounted on agricultural machines. This implies that the 
following main goals must be fulfilled: 
 
a.  the system will be able to detect a large set of obstacles on the field, i.e. all the objects 
needed to be detected in order to make it compliant with the existing performance and 
functional safety standards in agriculture (see more details in section 2.3) 
  
b.  the system will be able to detect also the eventual protruding obstacles on the field 
 
c. the system will be able to make an accurate obstacle detection no matter the terrain 
texture type 
 
d. the system will be able to make an accurate obstacle detection, no matter the weather 
conditions, climate or the illumination of the outdoor environment 
 
e. once all these criteria above satisfied, the system will be able to take the decision to go in 
a safe state in case an obstacle is detected on the field 
 
f. 
the switching to the safe state must be performed in a suitable time limit, such that the 
potential hazards and risks can be avoided.   
 
The system goals have been formulated based on (Christiansen, P., Hansen, M. K. et al., 2015), 
which describes the design challenges for an autonomous farming system and also on some 
considerations from (Mogensen, J.T.I. et al. , 2015), which tries to propose a new approach for 
structuring the safety requirements for perception systems, such that they will more accessible to 
be  interpreted and implemented by the developers.

3.1 Parallel computing principles 
 
The evolution of computer systems has been influenced in time by a generic reason: solve 
problems of higher complexity in a shorter time. That’s why more complex parallel computer 
systems have been designed. Their architecture relied on the idea that the system should 
provide the hardware/software computational infrastructure, in order to divide a complex problem 
(problems with large input data size or with a large number of computational steps) in smaller 
chunks, which can be executed in parallel. A parallel execution reduces the total amount of time 
needed for solving a problem, obtaining the aimed speed up. 
 
These aspects are a good foundation for a  general definition of a parallel computer system: “A 
parallel computer is a collection of processing elements that communicate and cooperate to 
solve large problems fast.” (Culler, D. E., Singh, J. P., et al., 1999). 
 
The idea of parallel processing is pretty old. One classification of computers that includes also 
parallel computers and is still valid nowadays has been formulated in 1966 and refined in 1972 
by Flynn (FLYNN, M., 1972). It differentiates the computer systems according the concurrency 
that exists when handling the data, respectively the instruction streams. The distinguished 
categories are contained in the following table:

final correct results. Only the program sections that are not affected by this data dependency 
may be executed in a parallel manner.

The Fraction time affected (by parallel execution) is directly proportional with the program section 
size that can be executed in parallel. Therefore, this section can be concluded with a very 
important theoretical observation: the speed-up obtained when parallelizing a program is 
dependent on the fraction of program that can be executed in parallel. Besides this, if the number 
of parallel processing increases while the parallel section size remains fixed, the speed-up, in 
relation with the number of parallel processors used, decreases. On the other hand, if the parallel 
section of the problem is increased altogether with the number of processors, the speed-up, in 
relation with the number of parallel processors used, increases.  
 
These aspects designate the strong, respectively weak scaling concepts, which are limiting 
factors, that should be considered in order to obtain the best possible speed-up, when executing 
a program in a parallel environment.

3.2 GPU computing 
In the early days, the computers were designed in such a way that the processors ought to 
perform also the graphical computation. This aspect slowed in a big measure the overall 
execution time of the programs. In order to improve this, special processors, which are in charge 
with handling, specifically, the graphical computations, have been designed. These graphical 
processors have been named GPUs (Graphical Processing Units), in order to be distinguished 
by CPUs (Central Processing Units), that remained in charge with performing the general 
purpose computations (Patterson, D. A., & Hennessy, J. L., 2009).  
 
This idea is illustrated by the following figure that shows the evolution of the computer 
architecture.

The graphical operations consist of different vector and matrices operations, which influenced the 
GPUs hardware architecture to be designed in a SIMD fashion. The most important features of 
the GPUs are listed below:  
 
● 
GPUs have been designed to be highly multi-threaded, which helps in hiding the memory 
latency. A thread is a section of a program that is executed in parallel, which may 
process, both its own data and shared data with other threads. The memory latency 
designates the time needed for the processor to get the necessary data from memory, in 
order to execute an instruction. Since hundreds and even thousands of threads can be 
executed in parallel, the memory latency that occur in one thread is hidden by the other 
threads, that continue their processing which is not dependent on that specific data 
request. 
  
● 
The GPUs hardware architectures facilitate the improvement of the instruction 
throughput, i.e. the amount of instructions that can be executed per clock cycle, when 
there is not any dependence among the computed data.*) This aspect is illustrated by the 
following figure, that shows the comparative benchmarking made for the instruction 
throughput between CPU and GPU based platforms.

● The GPUs memory chips present a hardware design, that facilitates the obtaining of

This features made the GPUs very attractive for solving the problems with a high degree of 
parallelism. That’s why in time, specific GPU programming languages have been created and 
offered the software developers, the possibility to implement also general computing algorithms, 
which could be executed on GPU platforms. The most well known GPU programming language 
is CUDA (Compute Unified Device Architecture). CUDA offers a high level API, that abstracts the 
GPU hardware, and simplifies in a significant measure the software development. It basically 
permits the developers to write programs in C, that can be executed on GPUs. CUDA can be 
also seen as a GPU parallel programming model, that has its own terminology.

which are grouped in one, two or three dimensional blocks. The thread blocks are grouped, at

3.3 Stereo block matching algorithm  
 
Designing and implementing an ADAS system, that performs the obstacle detection, requires 
knowledge and skills from two different main areas: embedded systems (hardware design and 
software development) and computer vision. The platforms that runs the computer vision 
algorithms are part of the embedded systems category. 
 
To keep it simple, computer vision tries to emulate the human eye perception on different types 
of computer systems. This is achieved by specific algorithms, that are in charge with processing 
inputs represented by digital images and provide output(s) in form of a transformed image. The 
can be further used to reach a decision such as: there is a person in the scene or there is a 
particular object in the scene etc. A list of the most popular computer vision algorithms is 
presented in (C.V. topics, 2010), and covers the most important topics in computer vision. This 
list is illustrated in the following figure: 
 
 
 
Fig. 3.15 Computer Vision main topics 
(C.V. topics, 2010) 
 
 
The discussion will focus on the stereo matching, which represents a technique of creating a 
model representation, from two input images. The images are captured using a stereo camera, 
which is an image capturing device, that incorporates two cameras. Those are situated 
symmetrically to the left, respectively to the right of an imaginary vertical axis, which intersects 
the camera front side. This type of image capturing emulates the way how the image is perceived 
by the human eyes (see fig. 3.16). 
 
The model obtained after the execution of the stereo matching algorithm reproduces a three-
dimensional representation of the captured scene, based on the luminance intensity of the pixels. 
It can be further used to identify some specific objects within that scene. This model may take the 
form of a depth map. The procedure of identifying an object from the obtained model consists of 
solving another computational stereo problem, that is generally known as reconstruction. 
  
 
36 
 
==PAGE:37==

(C.V. intro pres., 2016) 
 
 
 The following picture will be used for explaining  the processing done by a stereo matching 
algorithm:

(S.M. overview pres, 2016) 
 
 
Basically, a stereo matching algorithm goal is to find the correspondence of two points (one from 
each input image), that are the projections of a 3D point from the real immortalized scene. In the 
case of figure 3.17, there should be found the points ml, respectively mr, which are the 
projections from the two captured images of the 3D point M. 
 
The stereo matching algorithm accuracy is influenced by many factors that will be mentioned 
below. This is the reason to why additional transformations have to be performed on the input 
images, before proceeding to execute effectively a stereo matching algorithm. These 
transformations are known as calibration, undistortion and rectification. 
 
37 
 
==PAGE:38==
During the calibration, the external geometry (the relative positions and orientations of each 
camera) and the internal geometry (focal lengths, optical centers, and lens distortions) of the 
camera are estimated, such that the image data (expressed in pixels) relates as accurate as 
possible to an external world coordinate system. (Brown, M. Z., Burschka, D. et al., 2003) 
 
The undistortion process is needed because the image pixels are initially radial distorted, 
because of the circular shape of the camera lenses. The transformed obtained image contains 
the undistorted pixels that appear now to only be part of a straight plane, not a radial one. 
 
The ideal geometry for the camera pair of the stereo camera is represented by the so-called non-
verged geometry, which is illustrated in the following figure: 
 
 
Fig. 3.18 The non-verged stereo geometry 
(Brown, M. Z., Burschka, D. et al., 2003) 
 
 
In figure 3.18, the points OL and OR represent the optical centers of the left, respectively the right 
camera. These two points are aligned on a line T, named baseline, which is parallel to x 
coordinate axis of the cameras. In the non-verged geometry, the two image planes are perfectly 
parallel to the scene plane. The real stereo systems are impossible to be built according the non-
verged geometry.  This situation is illustrated by the following figure:

along an epipolar line (solid) (Brown, M. Z., Burschka, D. et al., 2003)

The rectification procedure will perform the specific computations, such that the images are 
transformed to respect the non-verged geometry principles. This is represented in the figure 3.19 
by the two dashed rectangles representing the rectified images.  
 
The geometry of the figure 3.19 can be described like this: “A point P in the scene is imaged by 
the left and right cameras respectively as points p and p`. The baseline T and optical rays OL to 
P and OR to P define the plane of projection for the point P, called the epipolar plane. This 
epipolar plane intersects the image planes in lines called epipolar lines.” (Brown, M. Z., 
Burschka, D. et al., 2003).

Once the images have been rectified, the effective stereo matching algorithm may be executed in 
order to match the pixels of the left image with corresponding pixels from the right picture, 
situated on the same epipolar line. The comparison across the epipolar line reduces significantly 
the computational complexity and the number of possible false matches (Brown, M. Z., Burschka, 
D. et al., 2003). 
 
As it can be observed from figure 3.18, that the projection of a point from the real space is 
located in two different places in the captured images. This displacement in the obtained pictures 
is called disparity and it makes not feasible to attempt to match individual pixels. Instead, 
different regions surrounding pixels, which are candidate for matching, are considered. 
Depending on the size of these considered regions, the stereo matching algorithms can be 
divided in two categories: local matching and global matching algorithms.  
 
The local matching considers only small blocks of surrounding pixels that may be matched with 
small regions of pixels from the same epipolar line while the global matching algorithms consider 
much larger amounts of pixels, for example all the pixels across a scan epipolar line.  
39 
 
==PAGE:40==
The global methods are more accurate in obtaining better disparity maps than the local ones, 
however, the computational complexity is higher. More information about the accuracy of this two 
different categories of stereo matching algorithms can be found in (Sabater, N., Morel, J.-M. et 
al., 2011). The matching criteria is given by different metrics.

The considered algorithm for computing the stereo matching is a block matching algorithm. It has 
been chosen because of its simplicity and the degree of parallelization that it presents. At the 
same time, it is fast and can guarantee the satisfaction of real time constraints (Nalpantidis, L.,

4.3 General guidelines about the benchmarking process  
 
All the underlying logic needed for solving the stereo correspondence problem in a GPU 
environment, using the block matching algorithm, is implemented in OpenCV inside a class 
named StereoBM, defined within the gpu namespace. The method that specifically computes the 
disparity map of two stereo images consists of StereoBM_GPU::operator() and represents the 
function whose computational performance will be evaluated. 
 
The main advantage of using the functionality that is already provided by a powerful computer 
vision library, like OpenCV, for solving our problem, looks obvious at first sight. The stereo 
matching is far from a trivial problem when looked at from three perspectives: the algorithm 
difficulty, the large amount of computations involved and the complexity of parallelizing the 
problem in a proper manner. In this context, the OpenCV library provides a very high level of 
abstraction, by performing all of these by the means of the routine mentioned above.  
 
The algorithm is executed in our case, in a GPU environment. The overall obtained performance 
will depend on the hardware characteristics of the parallel computing environment, the 
parallelism rate of the problem (see Amdahl law’s discussion in section 3.1) and the way how the 
computation is assigned to different parallel threads, either in a transparent way by the execution 
environment or explicitly by the programmer. Moreover, the OpenCV library already provides a 
set of parallel operations, in the form of an API (Application Program Interface). The way how 
different computations that can be executed in parallel are distributed among the processing 
elements of the functional units is implemented internally within the library.  
 
There can be found some benchmarking procedures performed for evaluating performance of 
proposed block matching algorithms executed on different platforms. Nevertheless, the authors 
do not use the same set of metrics in their evaluation and the effective benchmarking procedures 
are not detailed. As following, a few articles regarding this topic will be mentioned.

4.4 Description of the inputs and the measurement procedures 
4.4.1 Linux timing measurement concepts  
 
The benchmarking process consisted of performing timing measurements (CPU time, and the 
CPU/GPU Wall clock time) for three different implementation of the matrix multiplication: iterative 
version, OpenCV GEMM version for CPU and OpenCV GEMM version for GPU. 
 
Defining these two terms in a general manner for computer systems, the CPU time refers to the 
actual time within the processor, executing strictly the instructions of a program, leaving aside the 
time needed for the I/O operations or scheduling, while the Wall clock time takes into account 
all these aspects and designates the total amount needed for a program instance to be executed. 
As follows, there will be presented a brief overview of the time management operations, provided 
by a Linux operating system, a justify for the ones chosen, in order to perform the proposed 
measurements altogether with the description of the effective measurement procedure. 
 
A Linux operating system distribution provides different mechanisms of time measurements, for 
both kernel and user mode. In the Linux context, the kernel timers are named dynamic timers, 
while the timers used by the user mode processes are named interval timers. A summary of the 
purposes of kernel timers may contain: 
 
● 
Update the time since the system startup 
● 
Update the system  time and date 
● 
Update some resource usage statistics 
● 
Determine if the allocated time by the scheduler, for a process to run, has expired 
● 
Check if the software timers have expired  
 
 
 
47 
 
==PAGE:48==
On the other hand, the software timers may be defined and used by the user mode processes in 
order to: 
 
● 
Get/modify the current time and date of the system  
● 
Implement notification mechanisms that a time interval has lapsed     
 
The way how this timekeeping architecture is implemented inside the operating system's kernel 
(i.e. which are the clock sources used by the timers and which are the corresponding data 
structures and methods for performing the timing measurements), depends on the internal 
hardware architecture of the considered system. Mostly, the specialty literature provides many 
details about this topic from the perspective of the x86 architecture, but the things differ when it 
comes to the ARM architecture. There is information about the ARM timing circuits in the 
processors` reference manuals from the ARM website (ARM arch. ref. manual, 2016), but not that 
specifically in regards how the Linux kernel is designed to interact with these circuits. 
 
Anyways, for the current benchmarking process the focus is on the software timers used 
by the user mode processes. The POSIX 1003.1b introduced a new type of timers for user mode 
processes, which rely on a few virtual clock sources that provide a good abstraction of the

The provided resolution is expressed in milliseconds (as it can also be observed in resolution

measurements invoked by gettimeofday() are performed using CLOCK_REALTIME as source 
clock. 
 
As for the CPU clock time, another timer, specific POSIX system call has been used:  
int clock_gettime(clockid_t clk_id, struct timespec *tp); (Clock_gettime() manual page, 2003). 
This function gets the current value of a POSIX timer and stores it in timespec structure. The 
POSIX timer type is specified by the first parameter. The chosen clock type has been 
CLOCK_PROCESS_CPUTIME_ID (High-resolution per-process timer from the CPU), as it was 
desired to measure the CPU time for a specific process. The provided resolution is expressed in 
nanoseconds. 
 
According the Linux timer classification, gettimeofday() is implemented by the kernel by means 
of interval timers, while the second used POSIX timer makes use of a dynamic timer. 
Theoretically, dynamic timers are more flexible and reliable than interval timers (Bovet, D. P. & 
Cesati, M., 2006), but in this situation both of them have been used in the appropriate measuring 
context.

4.4.2 GEMM benchmarking procedure 
 
For the benchmarking, three different routines that perform the matrix product computation have 
been considered: 
 
a. the first function computes the matrix product using a common method: the matrices are 
stored and accessed as arrays and the computation is done iteratively inside three 
corresponding for loops. First loop iterates among the rows of the first matrix, the second 
one the columns of the second and the most inner one performs the traversal of the 
elements of the rows, of the first matrix, respectively the elements of the columns of the 
second matrix, altogether with the products and the sums that are needed to be 
computed, in order to fill in the resulted product matrix.

OpenCV in order to handle the matrix multiplication. There have considered three 
OpenCV Mat objects named A, B, C of type CV_32FC1. In OpenCV an expression such

is stored in matrix C

One of the formulas (eq. 4.7) or (eq. 4.8) must be replaced in (eq. 4.2), in order to get the

4.4.3 Block matching benchmarking procedure 
 
The entire stereo correspondence procedure consists of several steps (see algorithm details in 
section 3.3): 
 
a. Camera calibration 
b. Captured pair of images undistortion and rectification 
c. Disparity map computation based on the rectified images 
d. Depth estimation, if needed 
 
The target of the benchmarking is focused on evaluating the performance of the step c. regarding 
the procedure mentioned above. This step is accomplished by the OpenCV operator() method of 
the StereoBM_GPU class (StereoBM_GPU class, 2016). This method has the following 
prototype: 
 
void cv::gpu::StereoBM_GPU::operator() ( const GpuMat &  left, const GpuMat &  right, GpuMat 
&  disparity, Stream &  stream = Stream::Null()) (StereoBM operator, 2016).  
 
It basically takes a pair of images as input parameters and will compute as output the disparity 
map. For this method the parameters left and right designate the pictures taken by the left 
camera, respectively right camera while the disparity parameter represents the computed 
disparity map.    
 
Therefore, both the CPU time and the Wall Clock time have been measured for this method in a 
similar way how it has been performed in the case of the general matrix multiplication evaluation. 
 
The obtained value for the disparity map throughput is measured in frames per second and has 
been computed using the following formula:

5.3 Benchmarking results analysis 
There been have presented the obtained measurements for the benchmarking performed for the 
GEMM, respectively the block matching algorithm. This section will contain a set of 
considerations that intend to present a qualitative analysis of the obtained results. 
 
As a first general consideration, the obtained measurements are correct. The Wall clock time is 
always higher than the CPU time for any of the benchmarked algorithm. This is a normal result, 
since the first time involves measuring the overall processing, involving all the I/O operations 
consumed time and the context switching between different operating systems tasks executed at 
the same time with the benchmarked applications. The CPU time implies measuring only the 
actual processing time, needed for executing the benchmarked application. 
 
First, there will be discussed the benchmarking results of the general matrix multiplication 
algorithm. 
 
Overall, the performed measurements offered some information about the contexts, when 
choosing a GPU parallel implementation of a function, it is a better option than the corresponding 
CPU version. Both the CPU time and the Wall clock time are really high in the case of using the 
GPU version of the algorithm for small input data size. This reveals that it is not indicated to use 
a parallelized version of the algorithm for small input matrices size. No matter the number of 
iterations considered for executing the general matrix multiplication for the performed tests, a real 
advantage for executing the GPU version of the algorithm is obtained, when the input matrices 
size is equal or higher to a 200x200 size. 
 
It can be stated that the standard version of algorithm has a very poor performance. Basically, it 
proves to be a good choice only in case of a very small size of the input matrices (10x10) and not 
for a very big number of iterations (this should be smaller than 250). Otherwise, in the case of the 
timing measurements done for the iterative version and the CPU OpenCV version of GEMM, it 
can be observed a linear relation. This concludes, from the fact that, if we increase the iterations 
execution number with a factor of 1.5 or 2, this increase will be reflected in a similar manner in 
the obtained measurement results (see tables 5.1, 5.2 and 5.4, 5.5 from section 5.1). 
 
This fact is not valid for the GPU version of the algorithm as more factors influence the 
measuring results. At first sight, there can be mentioned among these factors: the time sets 
needed for performing the data transfer among the processing elements involved in the 
computation (roughly speaking the time needed for transferring the input and output data 
between the GPU and the CPU), how the caching is performed by the involved processing 
elements (it can vary depending on the size of the computed matrices), the cache coherency 
policy of the system and how the input matrices can be distributed in an efficient manner to the 
computation warps. Even if the linearity is not preserved and even if the memory accesses can 
be a bottleneck for the overall system performance, there can be deduced from the performed 
measurements that the GPU version of the algorithm is far more efficient in case of large size 
input matrices. This can be easily observed from the plots depicting the CPU time and Wall clock 
time measurements for 10, respectively 500 iterations of GEMM executions shown in figures 5.1, 
5.2, 5.3 and 5.4 from section 5.1.

b. only once one of these two conditions is achieved, the guaranteed operation throughput

FPS once the number of the computed disparity maps is high enough. A balanced value for this 
average has been obtained for a number larger than 100 computed disparity maps for the 
performed tests.

in different points of the executed iterations. For the lowest obtained frames rate per seconds, 
the average computed disparity maps rate is around 43 frames per second. This variation is also 
illustrated by the histograms shown in figures 5.6 and 5.8 from section 5.2. The variation of this 
measurement is due to localization of the computed data in different memory layers of the 
system memory hierarchy. Practically, the memory access time cause this pretty high variation

minimum rate values occur when no needed input data is stored somewhere in the cache 
hierarchy of the system, and this should be considered as an overall measure of the disparity 
map throughput performance. In comparison to the current used system, this value is around 
twice better, certifying that the NVidia Jetson TX1 board is a serious candidate for replacing the 
current system.

6. Conclusions 
 
The thesis topic has been interesting and challenging at the same time, as it offered the 
possibility to get insight on two relatively new directions, where both research and application 
design are very generous: GPU computing and computer vision. More than this, the 
benchmarking, as an area of computer science, requires a good understanding of the potential 
strength and weaknesses of the system hardware and software architectures in order to choose  
relevant parameters, to be measured and obtain a clear view of the system performance. There 
is not a unitary predefined procedure that can be used to benchmark the system performance in 
a general manner.  It is up to the person who performs the benchmarking to decide, in a proper 
way, which are the proper parameters to be measured and the proper metrics, such that the 
obtained system performance measurements will achieve the purpose of the benchmarking, 
which may consist of: confirm the efficiency of design decisions, observe if the system 
performance satisfies some constraints, figure out if the designed systems obtains better 
performance than a system which provides a similar functionality etc. 
 
For this thesis, the goal was to benchmark one functionality of a system implemented, based on 
a new hardware platform, i.e. stereo matching performance against the performance of this 
feature provided by the current system used by Class. The thesis performed evaluations for 
determining the performance related to the execution time and the amount of useful information 
(in this case disparity maps) that can be achieved for the stereo matching, a very important 
feature for the overall system purpose: the obstacle detection. The goal of the thesis has been 
accomplished because the obtained results confirmed undoubtedly that the new system provides 
higher performance than the current system for the considered metrics (see the results and their 
analysis corresponding sections). More than this the new system offer better hardware 
capabilities related to the extensive parallelization that offers a very good speed up and also the 
much higher amount of memory. At the same time, the current system is based on hardware 
implementation that restricts the flexibility of implementing a different algorithm in the case that 
one with higher performances is considered at some point. The new one removes this 
inconvenience since the chosen algorithm has a software implementation. It only implies a robust 
and efficient software design to replace the current algorithm with a better one in reasonable time 
and with a decent amount of effort. Additionally, the very good performance of the board, when it 
comes to general computation capabilities, has been confirmed by the measurements of the 
throughput of the evaluated general matrix multiplication algorithm.  
 
This thesis does not cover all the possible benchmarking that could have been done. A more 
detailed analysis may also contain some power performance analysis and also some cost 
evaluation for effectively, implementing the whole system. These are two very important aspects 
that remains at the moment as topics for a future benchmarking work, that will complete the 
results of the efforts made so far. 
Moreover the decision of effectively implementing the system will be fully justified as soon as all 
these measurements will prove, if the tradeoff between the implementation  of the system and 
the higher performance achieved is totally convenient.  For the moment, the opinion is that the 
benchmarked board, is definitely a very good choice for implementing the considered 
functionality, but the final decision to implement the system should be taken once the results of 
the suggested future work will be available.  
 
As a personal final thought, I hope that my work and the obtained results will be useful in the 
future and will bring a small contribution to the decision mentioned above, and maybe also to the 
process of standardizing the final system, in case the system will be effectively implemented.

3. Instruction and Memory Latency
Instruction and memory latency limit the performance of a kernel when the GPU does not have enough work to keep busy. The 
performance of latency-limited kernels can often be improved by increasing occupancy.  Occupancy is a measure of how many 
warps the kernel has active on the GPU, relative to the maximum number of warps supported by the GPU. Theoretical occupancy 
provides an upper bound while achieved occupancy indicates the kernel's actual occupancy. The results below indicate that 
occupancy can be improved by reducing the number of registers used by the kernel.
3.1. GPU Utilization May Be Limited By Register Usage
Theoretical occupancy is less than 100% but is large enough that increasing occupancy may not improve performance. You can 
attempt the following optimization to increase the number of warps on each SM but it may not lead to increased performance. 
The kernel uses 40 registers for each thread (5120 registers for each block). This register usage is likely preventing the kernel from 
fully utilizing the GPU. Device "GM20B" provides up to 32768 registers for each block. Because the kernel uses 5120 registers for 
each block each SM is limited to simultaneously executing 12 blocks (48 warps). Chart "Varying Register Count" below shows 
how changing register usage will change the number of blocks that can execute on each SM.
Optimization: Use the -maxrregcount flag or the __launch_bounds__ qualifier to decrease the number of registers used by each 
thread. This will increase the number of blocks that can execute on each SM. On devices with Compute Capability 5.2 turning 
global cache off can increase the occupancy limited by register usage.

4. Compute Resources
GPU compute resources limit the performance of a kernel when those resources are insufficient or poorly utilized. Compute 
resources are used most efficiently when all threads in a warp have the same branching and predication behavior. The results below 
indicate that a significant fraction of the available compute performance is being wasted because branch and predication behavior is 
differing for threads within a warp. 
4.1. Divergent Branches
Compute resource are used most efficiently when all threads in a warp have the same branching behavior. When this does not 
occur the branch is said to be divergent. Divergent branches lower warp execution efficiency which leads to inefficient use of the 
GPU's compute resources. 
Optimization: Each entry below points to a divergent branch within the kernel. For each branch reduce the amount of intra-warp 
divergence.
4.2. Function Unit Utilization
