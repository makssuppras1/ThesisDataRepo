Abstract
This Master Thesis will look into different IoT technologies for mobile and stationary sensor
collection solutions. A system architecture will be described and implemented from pro-
curement, modular sensing platform, transmission, data storage, visualization and data
integration in a third party software. The prototype aims to be a solution fitting the need
for easy and fast deployment for monitoring buildings and logistics. The modularity of
sensors is the focus of this thesis.
The Thesis is based on research into different wireless communication methods for IoT.
The method rapid prototyping was used and multiple interviews were conducted. In con-
nection with the rapid prototyping method the V-model was used to create a list of system
requirements. Here multiple iterations of a prototype were conducted and verified with
testing and validated from the V-model method. Beside unit test, two real world tests
were conducted; at a shop floor and during car ride through the northern part of Zealand,
Denmark.
The unit test showed that a modular sensing platform with LoRa is working. However,
the real world test showed that coverage and power consumption was unacceptable, as
per the system requirements. Testing in user experience showed positive feedback to the
developed prototype, with an emphasis on the modular design.
Data accessibility during transport was insufficient due to the coverage of the used net-
work. Further roll out of the LoRa network could resolve this issue. The high power con-
sumption is due to the prototype board, and can be rectified with a custom PCB. It was
possible to make a modular sensing platform with custom resolutions in order to save
transmission time and thereby battery life.
The project was a collaboration between DTU and Mickael Pohti LTech A/S formerly of
SysTec A/S.
Keywords— LoRaWAN, Internet of Things, Modular Sensor Implementation, Rapid Development,
IoT Architecture

Resumé
Denne kandidatafhandling undersøger forskellige IoT teknologier til brug i mobile og stationære
sensor indsamlingsløsninger. Der vil blive beskrevet og implementeret en systemarkitektur fra
indkøb, opbygning af en modulær sensor platform, sending af data, datahåndtering og datavi-
sualisering. Ydermere, vil data blive sendt til tredje-parts software. Målet med den designede
løsning er, at opfylde ønsker om nem og hurtig monitorering af bygninger og logitiske systemer.
Hovedfokusset er de modulære sensorer.
Kandidatafhandlingen undersøger forskellige trådløse IoT kommunikationsmetoder. I den forbindelse
er ”rapid prototyping” brugt, samt udførelse af interviews med interessenter. Ved brug af ”rapid
prototyping” blev V-modelens valideringstrin brugt, til at lave en liste af systemkrav, samt valider-
ing af prototypen. Flere iterationer af prototypen blev udviklet, og verificeret igennem test, samt
valideret ud fra V-modellen. Udover ”unit test”, blev to felttest udført. Felttestene blev udført i en
butik, samt under kørsel på den nordlige del af Sjælland.
De individuelle komponenttest viste, at den udviklede modulære sensor platform med LoRa virkede.
Dog viste felttesten, at dækningen og strømforbruget var uacceptabelt ud fra de specificerede sys-
temkrav. I de udførte test af brugervenligheden, kom der positive tilbagemeldinger til den udviklede
prototype og dens modulære design.
Data tilgængeligheden under transport var utilstrækkelig grundet manglende dækning hos den
brugte netværksudbyder. Dette kan løses ved yderligere udrulning af LoRa netværket i Danmark.
Det høje strømforbrug for prototypen skyldes udviklingsbrættet, dette kan forberedes ved udvikling
af et PCB-bræt. Kandidatafhandlingen beviser, at det er muligt at lave en modulær sensor plat-
form med optimerede sendingsopløsninger. Herved kan sendingstid spares og batterilevetiden
forlænges.
Afhandlingen er lavet i samarbejde med DTU Fotonik og Mickael Pohti LTech A/S tidligere SysTec
A/S.

Introduction
The scope of the project is to create a modular sensor package that can easily be implemented in
different environments without the need to directly install the system with wires for communication
and power. The prototype is being developed in collaboration with SysTec aps located in Værløse.
SysTec is a company that works with automated building control systems, and are interested in
using IoT networks to increase their competitive edge. The prototype needs to collect and transmit
different sensor inputs and to be integrated with the existing BMS system used by SysTec.
SysTec will be used as a test case to ensure that the prototype can be integrated with an existing
automated building control system. The main focus for the project is to develop a modular prod-
uct that allows for sensors to be switched according to needs, by simply changing a USB cable,
allowing for modularity.
Such a solution does not appear to be currently available on the market, and could bring great
value to several industries as well as small businesses.
The end goal is to have a configurable product that can be tailored to costumer needs, with a
simple Tacton model[1] as the baseline for a product configurator.

3.1
Internet of Things
The Internet of Things is an implementation of technology that allows the user to connect with
an object without having to constantly interact with it physically. A device is connected with the
internet and is then able to send information and receive commands and then act upon them
when the required prerequisites have been met. It was originally developed at Carnegie Hall,
where programmers had written a server program that allowed to assess whether or not a Coca
Cola was available at the vending machine three floors below. However the term Internet of Things
wasn’t conceived until 1999 by Kevin Auston of MIT, and became a popular topic within the industry
in 2003[5].
In order to have a successful IoT implementation certain criteria have to be met. The criteria for
a successful implementation can be found in Appendix A.3.1. For a seamless integration three
criteria have to be met, which can be seen in Appendix A.3.1.

3.2.1
LoRaWAN
Long Range Wide Area Network(LoRaWAN) is used for the IoT devices and are characterised for
its long-range and low battery consumption [9]. The range of the LoRaWAN technology greatly out-
performs that of WiFi, which has a rate of 46m indoors and 92m outdoors, compared to the range of
LoRaWAN which has a range of 2-5km in urban areas and more than 10km in suburban areas[9].
However this great increase in range comes at a cost of data rate, the LoRaWAN technology only
offers a data rate of 27 kbps which is a great deal smaller than that of WiFi with a data rate of 1.3
Gbps and Bluetooth Low Energy(BLE) with 2 Mbps [9].
In regards to battery life LoRa has been developed to use minimal power. The goal from the
developers’ side is to allow for a battery life ranging from months to years [9].
The message length of LoRa is limited to 100 bytes which is greater than that of SigFox, while
also being cheaper to build a gateway than that of SigFox as LoRa does not require an expensive
chipset [9]. However LoRa also has a limitation on how long time the end device can transmit data.
This limitation is set to 1% [10], This means that in order to transmit data more frequently the data
packages have to be smaller, or the data needs to be sent at a lower spreading factor seen in
section 3.2.1. It uses different regional spectra due to regulatory requirements. [11]. Below is a

Spreading Factor
LoRa uses a frequency modulation in order to improve the range of the
system. This modulation can be based on a negotiation between the end device and gateways, or
hard coded on the end device to optimize transmission time but a a risk of loss of data. If the end
device has been able to transmit data stably for an extended period of time the Spreading Factor
(SF) will be reduced by the network. The two following parameters are used in the negotiation:
Received Signal Strength Indication (RSSI) & Signal-to-Noise Ratio (SNR) [10].
RSSI
The RSSI value indicates the received signal based on the power of the signal in mW and
is represented in dBm [10]. The dBm value of the RSSI value is measured as a negative value,
increasing to 0 as the signal improves. The minimum value in order to transmit data is -120 dBm.
and a signal of -30 dBm is considered a strong signal.
The environment that the end device and gateway are in has a great impact on the RSSI value,
as an increase the amount of obstacles the signal has to traverse results in a decrease in signal
strength. A table of different materials impact on the RSSI value can be seen in section A.3.2

SNR
The SNR value is an indication of the ratio between the received power of a signal to the
noise floor power [10]. The value is measured in dBM in the range of [-20, +10], where a signal
value above 0 indicates that the signal operates above the noise floor. If the SNR value is below 0
it can cause corruption in the transmitted data which can lead to re-transmissions [10]. However
LoRa is capable of operating below the noise floor level by means of demodulation of the signals.
It is possible to demodulate signals in the range from -7.5 to -20 dBm, allowing LoRa to receive
messages below the noise floor level.[10]

Device Class
LoRaWAN devices come in three different variances which all have the same
basic abilities. The device types are as follows:
• Class A - All End-device¨
• Class B - Beacon
• Class C - Continuous
Class A devices
are the baseline for LoRaWAN device types, whose functionalities are car-
ried over to Class B & C.
Devices of class A support bi-directional communication. This is done by having a uplink trans-
mission followed by two windows that can be used for downlink transmissions to the end-device
from the network server. The network server can transmit a package in one of these windows but
not in both[13]. The time slot for the transmission is based on the end-device’s own needs allow-
ing for small variances based on a random time basis (ALOHA-type protocol)[13]. End-devices
of class A type have the least power consumption of the three types, as it only needs downlink
communication in a short moment after transmitting. However it is not possible to transmit any
changes to the device at any other time period meaning that the server has to wait for the next
transmission before being able to send corrective commands to the end device[13].

3.2.2
Gateways
Gateways act as the physical layer of the LoRaWAN network server (LNS), by listening and broad-
casting messages to and from end-devices[14] as LoRa modulated signals. Once the aforemen-
tioned security checks have been performed the transmission will be sent along with meta data
listed below
• ”Radio Signal Strength (RSSI)
• Signal-to-Noise Ratio (SNR)
• Receive Channel Frequency
• Reception Time
• Data Rate (DR)”[14]
The gateway is also an important factor in regards to compliance and regulations, as the LNS
schedules downlink messages on the gateway, that are sent before the transmissions have been
initialized [14].
Another benefit of the gateways is that they do not need to know device specific information, such
as certificates or other cryptography information. This attribute is beneficial in regards to security
and scalability, as the gateways cannot be used to gather information on specific devices nor is
the connect-ability limited to approved devices by the entity installing the gateway. It also allows
the gateway to focus on its primary function which is to relay information, without being slowed
down by security protocols and checks [14].
LoRa gateways are often powered by a main power line but are also available with battery power.
A high-bandwidth connection acts as the backhaul to the system using either an Ethernet cable
or cellular connection. The gateway is often Linux based, but in the recent time a more cost-
effective but resource-constrained protocol based on real-time operating systems has been im-
plemented[14].
Security
As the security of the system is paramount, security measures are taken in order to
secure the LoRa network. This is often done by using Transport Layer Security (TLS) or Internet
Protocol Security (IPSec). These methods work by having mutual authentication in order to link
up. Using IPSec allows the user to have secure application sessions inside the network. How-
ever using IPSec can be resource demanding, meaning that if the gateway has a limited volatile
memory, TLS might be more beneficial[14].
On top of these join processes it is also important to keep the following aspects of security in mind

3.2.3
MQTT
The use of Message Queuing Telemetry Transport began in 1999 when it was invented by Dr
Andy Stanford-Clark and Arlen Nipper[15]. It is a lightweight messaging protocol that is designed
for Machine-to-Machine (M2M) communication with low bandwidth, high latency and/or unreliable
network connection[16].
It works by using a publisher/subscriber setup, using a server (Broker) that receives and sends
the data from the correct publisher/subscriber. The end result yields the possibility of creating a
lightweight measuring system that can be used in a wide variety of industries.
A report conducted at the National Institute of Technology Warangal found that the use of a mobile
air quality sensor fleet based on MQTT would be beneficial to the monitoring of air quality as
the implementation would be cheaper than building fixed monitoring stations[17]. The reasoning
behind the statement is that in order to create and maintain fixed monitoring stations is expensive
as the station has to be constructed costing land resources.
In the mobile solution a mobile sensor platform can be replaced by a new one from the fleet.
Hereby it becomes possible to replace or update sensor equipment without losing data.[17] Kodali
and Sarjerao also gave another benefit to a mobile sensor fleet that the data could be acquired at
different places with GPS localization, which could grant better data.
In regard to security MQTT utilizes Transport Layer Security (TSL) and Secure Sockets Layer
(SSL) to encrypt data and ensure that data cannot be misused and acquired by inappropriate per-
sonnel. However this security comes at the cost of a significant network overhead cost [18].

Modular Components - I2C Compatible Sensors
I2C is a serial communication protocol that allows multiple devices to be connected via only two
pins. The implementations of I2C is wide spread and is used for many devices ranging from RAM
to OLED screens [4]. The capability of being able to connect multiple devices using only two
wires, in addition to the common power and ground pins, is an advantage when comparing to SPI
communication, combined with the fact that all devices can act as a master of slave [4]. The two
wires used for I2C is a SDA wire used for data transfer and a SCL for a clock. This is important as
only 1 master can operate at a time[4]. The master controls the clock signal and thereby the flow
of data, at a standard of 100 kHz or faster at 400 kHz although arbitrary values can be used [4].
The protocol works by the master changing the SDA line to low and the SCL line to high. This is
the start bit of the transmission indicating to the other connected devices that it is now the master
and the other devices should not attempt to claim the role of master [4]. An illustration of the data
transfers can be seen in figure 3.8.

Figure 3.8: Data is loaded onto the SDA with every falling edge of the SCL and is read
on the SDA at every rising edge of the SCL. The transmission stops when the master
releases the SDA[4]
In order to send or receive data to/from the slave the master sends a start bit indicating that it is
the master along with an address of the slave. Following the address is a bit that indicates if it
wants to write or read. If the master wants to write it sends a bit with the value of 0 and 1 if it
wants to read [23]. By being able to use the addresses it can communicate with a specific slave.

via the same I2C connection [24].
I2C also supports a useful feature in regards to quality assurance for collecting sensor data. This
feature allows the master to acknowledge(ACK) or not acknowledge(NACK) the data transmission,
assisting with error handling of the system [25]

Document databases store the data as either a JSON, BSON or XML document [31]. By doing

in such formats is that there is less translation between data formats in order to use the data on

Column based databases reads data a single row at a time, making them fast when running

Graph databases are centered around the relationship between data. These connections are
called links and each element of the data is stored as a node [31]. A graph database is optimized
to search for connections between elements and is often used as a strategy to avoid joining multiple
tables from a SQL database[31]. The use cases for graph databases are social networking, fraud
detection and knowledge graphs. However due to the structure of the graph database it is often
used in conjunction with another database scheme [31].

service had to be able to support the industrial market’s demands as well. For this reason it was
deemed necessary that the service provider also had support available, both for the MQTT aspect
but also support for Windows.
Based on these criteria HiveMQ would be the best option as it has a self-hosted broker or a cloud-
hosted service. However as HiveMQ is not supported on a Raspberry Pi which was used as the
broker server for the implementation. The broker selection opted for Mosquitto. An open-source
MQTT broker, with many of the same functionalities as HiveMQ seen in Appendix A.3.8 but without
the same level of support.

It is difficult to measure power consumption on a microcontroller due to the fast shifting of the
current draw. This is a discipline in itself [35]. And due to the time limitations seen in section 1.4
this will not be a part of the testing. The project group has used a PicoScope 2000.
A few tests were however done in order to gain insights into the behavior of power drainage on
the battery. The tests were performed using a PicoScope 2000A oscilloscope, combined with a
breadboard and a Ω1 blunt resistor with a tolerance of 5%. The test setup can be seen in figure
A.33. Based on the article by Tektronix [35] a visual representation of the measurement was used,
using the built in software of the PicoScope (PicoScope 6).
In order to test the power consumption the oscilloscope used the following setting:
Milliseconds
per division = 500 & sampling rate = 100k samples/second. The attenuation ratios was set to
X10 as it has a better voltage range and help with noise reduction[36]. An initial test with no
microcontroller connected showed almost cyclic noise in the signal, see figure A.28. This test
shows that it is difficult to detect power consumption that lies within the operational noise spectrum.
The power consumption test was conducted under normal running conditions. This means that
there will not be an analysis regarding how much power the downlink uses. In order to determine
the average value of the current draw, a center line was placed based on manual interpretation.

Expected Battery life
The battery life of the UModIt sensor implementation can be calculated
using the information gathered. As the battery used to power the UModIt is a 3000 mAh battery
the life cycle of the battery can be calculated using the following equation:
BatteryLifeTime =
BatteryCapacity
PwrMeasurement + PwrTransmission + PwrSleepmode
Where
PwrMeasurement = PowerConsumption[mA] · MeasurementTime[s]
3600s/h
PwrTransmission = PowerConsumption[mA] · TransmissionTime[s]
3600s/h
PwrSleepmode = PowerConsumption[mA] · SleepmodeTime[s]
3600s/h
SleepmodeTime = 3600s −(MeasurementTime + TransmissionTime)

As proven by the test performed in subsection 6.1.4 the spreading factor has a large impact on
the power consumption of the sensor implementation. For this reason a test was conducted in
order to determine the different RSSI values on Zealand. Here the UModIt was placed in a wind
shield of a car(see Appendix A.34). Due to differences in LoRa coverage for a moving sensor
implementation the negotiation to reach a lower SF could be long, or data loss could occur due to
the wrong SF settings.
The test was conducted by having a UModIt placed in a car, driven over the course of a day to
different locations on Zealand, Denmark. Additionally the downlink signal strength at the different
locations were measured using a Adeunis LoRaWAN field tester 868MHz at a time interval of 10
minutes. The data was then evaluated in order to determine the optimal solution for a fixed SF.

requirements set out in table 1.1 can be considered met.

It is possible to store critical data using an SD-card,

User experience & User data readability (Accepted): Through Qualitative testing it was shown
that the user could easily retrieve information out of the visual data representation in the form of
graphs. Qualitative testing also showed that the time needed to set up the prototype within the
time limit of one minute. Based on these tests the requirement is considered met.
System feedback (Partially Accepted):
The system gives feedback to the user indicating that
it has been re-initialized. However the system does not give direct feedback to the user indicating

monitors the messages sent via either MQTT or the Dashboard in Node-RED. For this reason the
requirement has only been partially met.
Alert system (Not Implemented):
No alert system has been implemented for sending push
messages if the system detects exceeding values. Therefore the requirement has not been met.

The user experience in regards to acquiring the product
and setting up the product, is within the time limit. Further improvements can be made in order to
make it more user friendly. However the requirement can be considered met.
Overall evaluation (Not Accepted):
As the requirements for data accessibility and battery life
are not met at the current state the acceptance test fails. Furthermore improvements on System
Feedback have to be performed in order to fully meet the requirements, and an alert system has
to be implemented.

7.2
Problem Statement and Subproblems
Which IoT technologies can be used to provide live data during transport 80 % of
the time at a low power consumption rate?:
LoRaWAN was tested as a possible solution
to the problem of transmitting data at long range and at a low power consumption rate. However
due to the lack of coverage it cannot at the current state of deployment be used to provide live
data 80 % of the time. Different technologies could be investigated as well as their deployment in
logistics carriers to find a better solution.
How can data packages be decoded?:
The sensor sends encoded data where each sen-
sor has an one byte ID followed by the data encoded into hexadecimals. Some sensors have
to send negative number therefor two’s compliments is used. In order to decode the message
an Application server(Node-RED) is used and is connected with a WebSocket to the Networks
server(Cibicom). In Node-RED the encoded data is decoded with a dynamic decoder that has the
ID stored and uses a switch statement to get the end of the encoded messages. The output is a
JSON object with key value pairs representing the sensors.
How will the customer’s wishes for data resolution impact the designed solution
and which effect does it have on power consumption?:
The designed solution took
the customer’s wishes into consideration and was created with a dashboard that could easily
be used to change the data resolution. This brought an additional benefit of saving the power
consumption of the sensor implementation, as the reduced payload size can lead to a reduction
in time spent on air by 9% thereby increasing the time that the implementation can spend in sleep
mode. However having this implementation will also lead to a more complex system that has to
be maintained when adding new sensor modules to the procurement aspect.
Which IoT platforms can be used for storing & visualize the end nodes collected
data?:
It was proven that Node-RED could be used as an IoT platform in conjunction with a
Cloudant database from IBM to visualize and store data from the end node in a user friendly
manner.
How can communication with the end node and the customer be established on a
web interface?:
By using the Node-RED dashboard it becomes possible for the customer to
establish a downlink communication with the end node for configuring the resolution of the sensor
implementation. Furthermore by using the dashboards created in Node-RED the customer is able
to easily read historical data collected by the end node.
How can a quality assurance procedure be implemented that ensures that no data
is lost from the designed sensor solution?:
It was possible to add a SD-card to the
sensor implementation that allowed for secure data storing in an off-line document that allows
the customer to open the file and quickly analyse the data, in case of loss of LoRa coverage.
The format of the file allowed sensor disconnections to be stored as values that can be read in
Excel and be illustrated on a graph as a missing point. The added SD-card showed little to no
power consumption as it was not possible to distinguish the additional power consumption from
the voltage drain’s signal noise floor.

The system architecture could be designed for a modular sensor implementation by having four
main components: An end node that collects and transmits the data via LoRaWAN to a network
server that can be used to distribute the data to the correct WebSockets. From this WebSocket it
was possible to create a user interface in the form of a dashboard. By adding a Cloudant database
it is also possible to store the collected data increasing the QA of the modular sensing platform.
The data could then be further distributed by an MQTT broker, supplying the data to either a BMS
or a costumer. Hereby, it was possible to create a modular sensing platform from acquisition to
further delivery of treated data via MQTT. Although some system requirements were not met, and
future work needs to be performed, the project is considered successful.

8.1
Discussion of findings
Based on the conclusion of the project the following aspects can be further discussed or developed.
8.1.1
Sources of Error
Battery test
The measurement tools for getting the current was found by using an 1 Ωresistor
that has 5% tolerance and a oscilloscope and converting the voltage. Furthermore, the voltage
supply to the test was conducted using a iPhone charger which could cause further noise in the
testing.
8.1.2
Battery Life
The testing of the power consumption with a product configuration using a 3,000 mAh battery would
be insufficient for the customer requirement of 2 years. A theoretical calculation using changed
parameters was performed in order to determine what parameters would be most beneficial to
improve in order to attain the goal of a battery life of 2 years.
The first calculation was conducted by changing battery from a 3,000 mAh capacity battery to a
40,000 mAh capacity battery. This calculation showed that the change would still be insufficient
as seen in Appendix A.37 attaining a battery life with a transmission every 15 minutes using all
I2C and GPS of 184 days.
A different calculation using a reduced power consumption in sleep mode was made. This cal-
culation also indicated that the change of a single parameter would be insufficient as seen in
Appendix A.38, attaining results of 165 days using the same sensor and transmission configura-
tion as above. The reduced sleep mode power consumption is based on a theoretical value of 0.5
mA which should be achievable according to sslupsky [37]. However Heltec.com sells a product
that can achieve a power consumption in sleep mode of 3.5 µA [38].
However when combining both of the factors it becomes possible to reach an expected battery
life of 2,193 days as seen in Appendix A.39. Further improvements to the battery life could be
achieved if the transmission time could be reduced by improving the SF.
8.1.3
LoRa
LoRa was determined to be the best solution for communication based on a theoretical analysis
comparing range and power consumption. Further theoretical analysis of coverage maps also
indicated that a LoRa implementation would be plausible in Denmark. However the testing indi-
cated that the coverage in Denmark on the Cibicom network is insufficient to transmit gathered
sensor data. Different solutions to this problem could be that as LoRa becomes a more wide
spread technology the coverage will improve. Another possibility for logistics purposes could be
that gateways are installed in transporters. If this solution is implemented by logistics companies
the coverage could improve significantly resulting in sufficient data coverage. Even if it is not
implemented across all carriers.

8.2.7
MQTT - Added security
The data transmitted by the UModIt sensor implementation is not sensitive data hereby a simple
username and password was used to protect the data. However when dealing with sensitive
corporate data added security becomes something that has to be considered. By using TLS or
SSL to add an additional layer of security certificates has to be created for both the transmitting
device and possible subscriber. This can be done using open-source software such as OpenSSL.
The service generates certificates and keys that can then be saved and shared with the appropriate
personnel, thereby greatly increasing the security as long as general security on the personnel side
is in place. However this feature might be redundant if the setup of the Mosquitto broker has been
done correctly.
On a M2M basis, these usernames and passwords could be stored on a secure data-site, allowing
for easy management, as well as a high level of security if the right precautions are in place. Having
this capability to easily update access is also beneficial from a business point of view as users can
be added and removed using a database and ACL.
MQTT - Broker Selection
Due to the limitations of the project Mosquitto was chosen as the
broker for the MQTT implementation of the UModIt. Different brokers might be more suited if the
product should be launched ensuring as little downtime of the broker. This could be implemented
using more servers for the broker should the service be self-hosted, or using a MQTT cloud plat-
form such as HiveMQ Cloud. A cost benefit analysis should be conducted on the different possible
avenues of platform development.
8.2.8
Investigation Into Different Communication Methods
As it was determined by the analysis of the LoRa network the coverage does not allow for a 80%
access to live data. For this reason an investigation into different means of communication has
to be performed analyzing the pros and cons of various technologies. Should another technology
prove to be useful for the logistics scenario a new model of the UModIt has to be developed and
integrated into the configurator.
8.2.9
Climate Control with UModIt
A possibility for further use cases of the UModIt is for it to be a means for controlling systems
using the MQTT protocol. This feature would require that the UModIt is integrated with the control
system. The UModIt should then have topics for the specific variables that it should control allowing
users with write privileges to send new settings commands to the UModIt. These commands
should then be relayed to the control system.
8.2.10
I3C
I3C is a new technology that could replace I2C. The data rate on I2C is 1 Mbps compared to
12.5 Mbps on I3C. This allows for faster data transmission and also helps save battery life. This
makes it an ideal solution for measurement cycles. Another benefit is that I3C is also backwards
compatible, meaning that an I3C chip will be able to collaborate with I2C sensors[39].

shorter sprints it becomes possible to prevent dissatisfaction with the end user as expectations or
criteria might change throughout the development.

The second part is to analyse the problem and determine what problems need to be solved in

A.4.4
Tacton walk-through
A configuration model was created using the Tacton program. The model uses a list of different
classes for each component type such as sensors, box, battery, and component sensors. Each of
the classes then consist of objects. The object is the specific part such as the Honeywell BME280
or 10.000mAh battery. The classes are then realized, and constraints are set ensuring that not
configurable products can’t be created by the consumer.
Once the constraints has been set steps are created leading the consumer from one step to the
other. The steps are visible depending on what prior choices the consumer has made, meaning
that the consumer only sees the relevant choices.
Figure A.18: Tacton - Creating a new order
The first step is to select that a new order will be placed. After selecting to make a new order
the product has to be selected as seen in fig A.19. Here the consumer will have the possibility to
choose between buying a mountain bike and a UModIt.
