Abstract
The number of complex tasks solvable using artificial intelligence continues to increase due
to the vast amount of available data and yearly upgrades in computing power. However, the
mantra of doing more with less remains prominent in the field of artificial intelligence.
To
learn effectively, a common strategy is to use a precise and often complicated reward system,
resulting in countless hours spent tweaking and updating these systems. Inverse reinforcement
learning seeks to turn this paradigm around by enabling a reward function to be extracted from
observed behavior. This project surveys the field of inverse reinforcement learning for literary
works and implements the method known as maximum likelihood inverse reinforcement learning
to approximate the reward function of a grid navigation task. This is done by using the value
iteration technique to create a optimal agent that can generate data on which the maximum
likelihood estimation process can take place. Additionally, the project builds an active learning
framework as a way of optimizing the performance of this process. Both methods are tested in
three different experiments in which their ability to replicate the behavior of an observed expert
is measured and evaluated.
Keywords: Inverse Reinforcement Learning, Maximum likelihood estimation, Active learning,
Value iteration, Navigation in grids.

Introduction
Artificial intelligence (AI) has been one of the most researched, speculative, and revolutionary
fields in all of computer science and even in science in general since 1956[2]. Everything from
movies, books, articles, and scientific papers has theorised about what a created intelligence
might be like. However, there is no doubt that the ability to learn is present in all versions of
such an intelligence, which has resulted in many approaches to give a machine this capability.
What started as algorithms that used smart and efficient problem-solving techniques has grown
into a huge field with multiple different methodologies that all try to solve the question of how
a machine can learn in general.
This master’s thesis takes a closer look at one of these sub fields called Inverse Reinforcement
Learning (IRL), which has grown increasingly more popular the past two decades[6]. As artificial
intelligence solves progressively more complex problems, the amount resources needed to train
these AI’s seem to remain as high and costly as when the field was founded. At its essence,
Inverse Reinforcement Learning takes a core concept from biology, the idea and ability to learn
from those who are better than oneself, and makes it the foundation of its methodology in the
hopes that it might be a way to mitigate the arms race between complexity and resources. Of
course, this has led to many theories about how learning from others might be formalized, with
emerging solutions like maximum likelihood estimation, explored in this project, being seen as
potential means to replicate the human ability to learn from others.
However, as it often is the case in nature, just being able to learn is not always enough. The
learning has to be efficient and fast in order to overcome complex and challenging problems.
Therefore, new methods like active learning has emerged as a possible way to expedite the
teaching of these algorithms. Unfortunately, there is no guarantee that combining methods will
be more effective. Therefore, the main topic of this project will be to explore the question of
whether active learning and inverse reinforcement learning can be effectively combined.

Literary review of Inverse Reinforcement Learning
Before addressing the main question, it is necessary to provide some background understanding of
what the IRL field actually is. This section was created to provide a broad understanding of the
field of IRL and explore the literary works that have contributed to its creation and advancement.
By covering aspects such as IRL’s inception and the fundamental building blocks of a IRL
problem definition, this section will provide the essentials that later sections will elaborate upon.
While exploring the literature, many influential papers was found. This section will conclude
with the summary of the influential papers, as they guided this project in defining its scope and
objectives.
2.1
The story of IRL
A interesting question to answer early is ”why” a field even exists. This can provide a intuitive
feeling of the field itself, as knowing the why something is, reveals a lot about the core problem
that is being addressed.
So why is IRL a thing?
To answer this it is imperative to look
at the broader field to which IRL belongs, namely apprenticeship learning[15]. The main
idea of apprenticeship learning is to study the process in which a observer learns to behave
correctly given demonstrations of expert behaviour. The field is built on the idea of how humans
learn by watching others and thereby learn quicker than what would have been possible if
everything needed to figured from scratch each time.
There are in two main categories of
apprenticeship learning, namely inverse reinforcement learning and imitation learning, which
utilize the same approach of learning from observing but differ in the goal.
Where IRL is
concerned with understanding why the expert behaves like it does, imitation learning is more

2.2
IRL as a setting
Knowing where the field of apprenticeship learning and its sub category IRL comes from, the
next important thing to explain is what different elements that make up an IRL problem. To
ensure that these concepts are clearly understood and well defined a breakdown of IRL setting
has been made.
2.2.1
Agents and Environments
If one is familiar with IRL’s sibling, Reinforcement Learning (RL), then the following concepts
should not come as a surprise. Like RL, one of the most central pieces is the agent, e.g., a football
player. An agent is responsible for taking actions and thereby learn to solve the problem at hand,
e.g. getting the ball into the goal. Therefore, an agent is sometimes also referred to as the learner
or apprentice. The agent is placed in an environment e.g. a football pitch, where it acts. It
is common to use the Markov decision process (MDP) definition[14] to describe a environment
as a set of variables (S,A,r,T,γ). These variables are: state S, actions A, rewards r, transition
probabilities T and discount factor γ.
States are bunches of information that describe how the environment currently look. A state
is therefore also a big part of transitions and rewards, as these depend on states. The football
example has been modelled as MDP on figure 1 and here a state is where the ball is on the field,
but could also hold additional information like how big the goals are and what type of surface
the field is. All this information has an effect on how easy or difficult it might be to kick the
ball into the goal. All of the different configurations of state is then expressed as the set S. In
the figure 1 states are visualised as light green circles.
The set of actions A covers what actions an agent can perform in the environment.
In the
football example this could be to move further away from the goal or kick the ball. On figure
1 action a0 is representing moving the ball further away and a1 is kicking the ball which are
visualised as orange circles. Every time an agent performs action the environment reacts and
either a new state appears or the action results in the same state as earlier. This is also visible
on figure 1 as each action has directed arrows showing to which states a certain action leads.
Rewards, or r in the context of MDP, are what the agent receives when it’s actions changes
the environment and a new state is created. This new state could be more preferable than the
previous and would therefore receive a reward. However, the new state could alternatively leave
the agent in a worse position, which would be penalized, meaning rewards can be both positive
and negative. Negative rewards are sometimes referred to as costs.
Reward functions are used to specify how these rewards are given, and this function is defined as
R(s, a, s′). Reward functions can be linear, non-linear and discontinues. An example of an non
linear reward function could be that, given a state-features function Φ = {ϕi : S →R} which
could assign values to every state, R(s, a, s′) = ϕi(s, a) ∗ϕi+1(s, a) where ϕ(s, a) denotes some
feature value, with features being some additional information about the state e.g., the agents
distance from its goal. However, in many cases when working with IRL, the reward function is
assumed to be a linear combination of weighted features. This form of rewards can be expressed
with the following notation:
R(s, a, s′) = θ1 ∗ϕ1(s, a) + θ2 ∗ϕ2(s, a) + · · · + θn ∗ϕn(s, a)
(1)
Where θ denotes the weights that is associated with some feature. Expressing rewards function
with the above mentioned form is also a fundamental assumption of this project.

The transition probability matrix T or a transitions model, which have the following notation:
P(s′|s, a), is used to describe how changes in the environment function. Transitions occur when
an agent takes an action in the environment and classified as either deterministic or stochastic.
In the case where transitions are deterministic, then actions taken by the agent will always lead
to desired state. In the football scenario, let’s say the agent was 7 meters away from the goal and
the conditions were perfect, this is represented by the agent being in state s0. If the agent then
kicks the ball, it would always end up in the goal, that is a deterministic outcome. A stochastic
outcome is, by contrast, ruled by probability. In environments with stochastic transitions, actions
are not guaranteed to only have one outcome but instead have a probability distribution over a
set of new states. If this was applied to the football example from above, a change from perfect
conditions to a rainy day would make the surface slippery and unpredictable. When the agent
now kicks the ball from 7 meters it now scores most of the time, but sometimes it misses. The
kick action now leads to two different states with some probability for both of them. Whether
transitions are deterministic or stochastic, they are still in the context of a MDP encoded in the
transition probability matrix. Each entry in this S × S matrix is a probability distribution and
is indexed by and a specific state s and wanted state s′ : T(s, s′). On figure 1 the transition
probabilities is shown as small numbers next to the actions. Additionally, both deterministic
and stochastic transitions is present the illustration.
γ is known as the discount factor and is used to weight future actions outcome compared to the
actions taken in the present. As humans we tend to prefer being rewarded sooner rather than
later. A very used example of this is the question “would you rather get $10 right now, or get
$10 next week?”. Most would prefer to take the $10 straight way, but if every condition where
guaranteed to be exactly the same but just a week later, then the choice actually would not
matter. However, in a RL environment γ is used to guide this choice. When computing an RL
example, γ takes the a value ∈[0, 1]. The lower γ is, the lower future rewards are weighted, and
if future rewards are worth as much as present, then γ = 1
2.2.2
Solutions and Policies

solve a MDP. As the agent could theoretically start in any state no single fixed action sequence
is a solution for a MDP. The solution must therefore tell the agent what to do in any reachable
state it can encounter. It could be expressed as the solution recommends what action the agent
should take when it is in a particular state. This sort of solution are called policies and it is
normal to denote them with π. Because the action the policy recommends depends on the state a
query or solution to a particular state is expressed with π(s). If there is some reward to be gained
in the environment some policies are better than other. The way policies are evaluated against
each other is based on their expected utility/reward.

2.2.3
IRL problem definition: MDP’s without rewards
Compared to the standard MDP as described above, IRL distinguishes itself from it’s sibling
RL by working with a altered version of a MDP’s, specifically MDP\r: (S,A,T,γ). These types
of MDP’s are also sometimes named a Markov decision problem. In order to solve such a MDP,
the rewards most be firstly be obtained, and this is IRL algorithms comes in play. Some IRL
algorithm use a MDP\r and observed behaviour from other agent as the inputs. The observed
behaviour is known as Trajectories and is the referred to as τ. In most literature the set of
trajectories is denoted the following way D = {τ1, . . . , τN} were each trajectory is made up of a
sequence of state-action pairs likewise, τi = {(s1, a1), . . . , (sn, an)}.
The agent that created these trajectories is referred to as the expert since it is assumed that
this agent is successfully maximizing some reward. With these inputs, the MDPr and expert
trajectories, IRL algorithms try to obtain an approximation of the expert’s reward function. As
mentioned in section 2.2.1, the rewards function in a IRL setting is expressed in the form of
equation (1), which describes that some reward weights, are applied to a feature vector. The
observing agent, which in this assignment is also referred to as the ”IRL agent”, must therefore
learn these reward weights so that it can maximize the sum of expected rewards in relation to
θE, where θE represents the expert’s weights
2.2.4
Known problems in IRL and AL
Throughout most of the time spent researching AL and its sub-area IRL, a few recurring problems
have needed to be addressed each time. These problems are the assumption of rationality in the
expert and the ambiguous problem definition of IRL.
To begin with, the assumption of rationality. To assume rationality means that the expert is
optimal in every instance of the problem, i.e., the expert cannot make a wrong move. This is a
Page 4 of 40
26/2/23
==PAGE:11==
2

strong and problematic assumption due to a few different factors. The first factor is related to
the performance of the observing agent. If the expert was really rational in all scenarios, then
no suboptimal moves would be made, and in the case of most experts, that is simply not true.
A good example of this would be the GO game between Lee Sedol and Google’s AI AlphaGo1.
“Another problem with this approach is that the assumption of a rational expert is unrealistic.
It means, for example, that a robot observing Lee Sedol making what eventually turns out to
be a losing move against ALPHAGO would have to assume that Lee Sedol was trying to lose
the game”[15]. This is also why IRL is, in some cases, more appealing than its sibling, Imitation
Learning. As stated in 2.1, the goal of imitation learning is to replicate the expert’s behavior,
and in some problem instances, that is more than valid.
However, the observing agent can
never exceed the expert, which is why some IRL approaches get around the problem of assuming
rationality. This allows some IRL approaches to not only replicate but also improve on the
expert’s behavior, as the expert is now also allowed to be somewhat suboptimal, leaving room
for improvement.
The next problem is that of IRL’s problem definition. As the goal of IRL is to approximate
an expert’s reward function, it seems that finding some optimal reward function R∗would be
enough. Recall to section 2.2.2 that the reward function was a key component of quantifying
policies over each other. Therefore, if R∗is to be optimal, it must make observed expert policy
optimal as well. R∗must, therefore, be defined as R∗≥R where R represents all other reward
functions. This R∗would then maximize the total expected discounted reward under the expert’s
policy so that it is ≥any other policy. However, there are infinitely many reward functions that
could make the expert’s observed behavior optimal. This has been illustrated in figure 2. An
example of a degenerate solution to this is R∗= 0, as when there are no rewards at all, then
any action is optimal. Any IRL approach must, therefore, address the problem of sampling a
reward function out of the infinite set that actually makes the observing agent learn the desired
behavior.

2.3
Research findings and IRL algorithms
Apprenticeship learning, as mentioned in 2.1, is concerned with learning meaningful behavior
from an expert’s behavior. Like neural networks were inspired by the anatomy of the human
brain, AL is inspired by our ability to infer meaning through demonstrations. As humans, we
have an innate ability to understand what a person is trying to achieve simply by observing
them. This skill has played a vital role in how we developed as a species, as a certain task
could be learned from scratch, and afterward, it could be learned somewhat effectively through
observation. In an effort to effectively solve the problem of ”hard-to-specify reward functions”,
which is present in RL problems[18], the AL approach has become fairly popular within the
machine learning field. With this popularity, a plethora of different papers on both imitation
learning, which formulates the problem as supervised learning of a policy from the expert’s
state-action pairs, and IRL, which tries to infer reward information from the expert’s behavior,
have been written.
The following section, therefore, seeks to delve deeper into relevant research and findings in
the AL field.
This will provide an overview of the solutions that exist to the IRL problem
and how they were achieved. Firstly, a table of different studies will be provided, along with
the category/approach of IRL the paper belongs to. Afterwards, a short description of each
paper will be provided to give an understanding of the problems and methods used in each
study. Lastly, a short summation is made to wrap up why this project chose MLIRL to infer an

2.3.1
Amit and Mataric, 2002: Learning Movement Sequences from Demonstration
The first paper introduced focuses on imitation learning instead of IRL. The reason for this is
to establish how imitation learning might be used and its ability to learn behaviors like IRL.
The paper named Learning Movement Sequences from Demonstration [1] showcases an imitation
learning’s ability to copy and identify complex behavior. In this paper, the authors use a robot
that is tasked with learning to identify and imitate movements from a human demonstrator. The
authors use a hierarchical reward structure to try and extract a policy from the demonstrated
behavior, feeding information from the lower levels that involve base primitives, which includes
movement of joints, to the higher layers that try to learn specific movements that are demonstrated
often.
The highest layer in this reward structure is then able to represent these complex
movements.
The method works by alternating the parameters in the lowest layer to be as
close to the demonstrated movements. The next layer then learns the specific motion, which
is built up from primitive joint movements from the first layer. The highest layer uses hidden
Markov models to treat each motion that is passed on from the lower level as a symbol in a set of
symbols that is used for inferring which motion is being performed. The accuracy of the method

This paper showcases some of the strengths in the imitation learning field. Although learning
a policy, instead of a reward function, makes the generalization of the learned behavior much
smaller, complex behavior is still able to be learned, and sometimes the approach might be

2.3.2
Ho and Ermon, 2016: Generative Adversarial Imitation Learning
In a 2016 paper, a new method called Generative Adversarial Imitation Learning (GAIL)[9]
showed an impressive ability to copy expert behavior even in large and high-dimensional environments.
This paper, which highlighted this method for the first time, is included in this project as a
showcase of what the newer methods within the imitation learning field are able to achieve.
The authors approached the problem of mimicking expert behavior by using a technique known
as a Generative Adversarial Network (GAN). When using GANs, two networks are needed: the
generator, which is tasked with generating new data from input data by learning its distribution,
and the discriminator, which is tasked with classifying whether a data point was generated from
the generator or is an actual data point from the sampled dataset. Applying GANs with imitation
learning, the authors of this paper created a generator network that tries to mimic the behavior
from the expert, but the network is never exposed to the actual trajectories of the expert.
Instead, only the discriminator gets to be trained on the expert’s generated data, which it then
uses to try and classify whether a given set of behavior actually comes from the expert or the
generator.
The paper showcases that this approach is able to achieve impressive results even with very
limited expert data available. Additionally, the method even breaks some of the limitations
of normal imitation learning, as it, in some instances, performs better than the expert and
generalizes enough to even handle unseen behavior. The performance of GAIL showed promising
results and has since been a benchmark against which the imitation learning and IRL fields have
been trying to beat.
2.3.3
Hadfield-Menell et al 2016: Cooperative Inverse Reinforcement Learning
Another paper that has formulated interesting thoughts within the IRL field is that of Hadfield-
Menell, Dragan, Abeel, and Russell called Cooperative Inverse Reinforcement Learning[8] (CIRL).
This paper is reviewed due to its proposed solution to the problem of value alignment, which
has long been a central problem within the field of artificial intelligence (AI). As the paper itself
points out, if a vacuum cleaner robot was told to clean up, it just dumps the dirt it just cleaned
up to maximize its reward.
The paper starts off by addressing two known problems within the IRL field. The first is that
of value alignment. How does one specify that a learning agent should not adopt a witnessed
reward function as its own, i.e., if a human wants a coffee in the morning, the agent should learn
to brew a cup of coffee for the human, not drink a cup of coffee itself. The next problem is that
of learning from a sub-optimal expert. These problems get cast into what the authors call a
CIRL problem in which two players, a human and a robot, have partial information. The human
knows the reward function, and the robot only gets the reward the human gets. Therefore, the
optimal solution becomes to maximize the human’s reward function. To solve this problem, the
authors suggest an approximation algorithm that computes the best response the human can
give to the robot while the robot is running IRL. By giving the robot a better understanding of
the true reward function instead of just being optimal, the robot is able to better maximize the
human’s reward and thereby its own.
By defining the CIRL problem as a two-player game, where only one party knows the reward
function but both parties try to maximize it, the authors have defined an approach to tackle one

of the central problems with AI. This, coupled with the solution of how to better communicate
intentions in a shared goal environment, is an inspiration for what this project tries to explore.
2.3.4
Russell and N.G, 2000: Algorithms for inverse reinforcement learning
In the paper named Algorithms for Inverse Reinforcement Learning[13], Authors, Russell and
Andrew Y. Ng provide some of the fundamental work within the IRL field.
In this article,
they highlight three different settings within the IRL field. In the first setting, they are given
an MDP and the expert’s policy is known, not just observable through trajectories, and the
number of states in the MDP is finite. When proposing a solution to find the reward function
for the given parameters, the authors found that an infinite number of reward functions were
yielded as a result. In order to choose a useful reward function, they suggest a solution where
they constrain the problem by making single-step deviations from the expert’s policy as costly
as possible but also try to make the reward function as simple as possible by favoring solutions
with small rewards. Using constraints in a linear programming formulation allows them to find
a solution that assigns non-zero rewards to very few states.
The second setting is MDPs with very large state spaces. Here Ng and Russell propose casting
the reward function as a linear combination of known state features, the same way mentioned
in section 2.2.1, by using function approximation. The problem then becomes to estimate these
feature parameters instead. With this definition of the rewards, the solution is again obtainable
by using linear programming. Using a similar constraint as the first approach, a reward function
can be found such that the expert’s policy is valued higher than any other policy.
The last setting is one in which the expert policy is no longer given and is therefore reminiscent
of later formulations of the IRL problem as the expert’s policy is only observable through
demonstrations. Again, the authors use the same formulation that the reward function is a
linear combination of known state features. They then propose an algorithm that starts off with
a random policy and iteratively improves the estimate of the feature parameters by solving a
linear program that uses estimations of the expert policy derived from the expert’s trajectories.
They finish the paper by testing all three algorithms on three different environments. These
are a discrete 5 x 5 grid world, where transitions are stochastic, and there is only one goal
state. The next environment is a well-known RL domain known as the mountain car, where the
goal is for a car to climb a hill. And lastly, an experiment of a continuous grid world. All the
experiments were considered successful, leading the authors to claim that the IRL problem was
indeed solvable.
2.3.5
Vroman 2014: Maximum Likelihood Inverse reinforcement learning (MLIRL)
The arguably most impactful and most influential work among these listed papers/articles/dissertations
is by Monica C. Vroman from 2014[20]. This dissertation is a write-up of all the author’s previous
publications and lays the foundation on which this paper is built.
In the paper, the authors present three maximum likelihood approaches to three different
domains within the IRL setting.
The first method is a maximum likelihood estimation for
single-agent environments where the reward function is assumed to be a linear combination
of known state features. Using a probability model that uses a set of approximated weights
to create a value function, the method is able to find the optimal behavior by using gradient
ascent. As this method is used in this project and is directly adopted from this paper, much
more information will be given on how this method exactly works.
The second setting in which a maximum likelihood approach is present is one where the reward
function is nonlinear. The algorithm proposed here uses elements from supervised learning with
standard inverse reinforcement learning setups to fit the learned rewards to a given hypothesis.
This supervised learning component is then integrated into the MLIRL solution given for linear

2.3.6
Ho et al 2016: Showing versus Doing: Teaching by Demonstration
The paper Showing versus Doing: Teaching by Demonstration[10] offers an interesting perspective
on the MLIRL method. As it approaches IRL from a behavioral science outlook, it raises some
intriguing points regarding how humans learn and what defines a good teacher or expert. These
ideas greatly influenced the direction of this project, and the paper is therefore represented in
this thesis due to its approach highlighting some of the dynamic aspects of the IRL field.
In this paper, Ho et al. pose the question of how teachers go about teaching a particular task to
a learner. Is there a difference between showing how to do a task versus actually doing it? To
explore this question, the authors use a pedagogical model that combines ideas from IRL and
Bayesian models. The model aims to find the best way to convey the expert’s objective, such as
doing a task sub-optimally and over-expressing objectives to showcase the goal clearly.
The authors then test this model against an optimal MDP solver, as well as humans who were
instructed to both do and show the same task. Two experiments are conducted, which suggest
that the model can predict the solutions derived by the group of human participants who ”show”
the tasks. As a result, the authors train an agent using the MLIRL method on the ”do” and
”show” datasets, and find that the agent performs better on the ”show” datasets.
Overall, the paper indicates that an expert could be a better teacher if they provide disambiguation
and sometimes sub-optimal solutions. This led to the idea of incorporating the concept of active
learning into this project, with the thought process of whether a student/learner can also optimize
the training process.

3.3
Active learning
The last concept that will be introduced is active learning. Active learning is a method where
the goal is to obtain good or equal performance to other machine learning approaches by having
a smaller and more qualitative data set to train on. This quote is a good summation of the
core concept: ”“The key idea behind active learning is that a machine learning algorithm can
perform better with less training if it is allowed to choose the data from which it learns””[17].
In order to understand how active learning was used in this project, a breakdown of the key
characteristics will be provided in this section. This will later be used as the active learning
concepts will be implemented into the code base in section 4.5.
As the quote formulates, the method involves some level of cooperation between the learner,
typically the machine running some form of learning algorithm, and an oracle, typically a human,
who is able to label data and thereby guide the learning of the machine. The process then starts
with a very small amount of labelled data compared with the rest of the training data, e.g. 5% of
the data is labeled. When training the algorithm, it first forms a hypothesis and starts assigning
probabilities to how confident it is about a specific task e.g., whether a picture contains a cat
or not. In cases where it is uncertain, it is able to query the oracle/expert and ask whether a
certain condition is met or not met e.g., was there a cat in that picture, or was it another animal
with roughly the same size?. This should, in theory, allow the oracle to only spend time labelling
data in which the learner is uncertain about something. This should, in theory, help the learner
better detect these edge cases and thereby learn more effectively. The newly labelled data is
then added to the training set, from which the process starts a new iteration. This flow has been
depicted in the figure, where the different components described above should be recognizable.

4.2.4
Features
In the context of machine learning, features are individual properties that can be measured or
are characteristics of some phenomenon [5]. Features add information to states based on the
characteristics of the environment. For example, if an agent moves into a goal state, humans
intuitively know that this is a good thing, but how should the agent know? With features, it
becomes possible to differentiate states based on more information than just their enumeration.
A goal state could be measured based on a binary goal feature, simply indicating whether the
agent is standing in the goal state. Alternatively, a continuous distance to goal feature could be
calculated as follows: 1/N, where N denotes the number of tiles the current tile is away from
the goal tile. In this project, features are binary, as this seemed to be the simplest approach,
which aligned better with the assignment’s goals.
Another aspect of working with features is the question of which ones to select. Feature selection
is a problem-dependent process, and selecting the most relevant features can help with overfitting
and achieve better generalization [16].
There are several techniques for choosing the most
significant features, including statistical approaches that rank features based on a statistical
evaluation, wrapper methods that evaluate features based on the performance of the machine
learning algorithm, and embedded methods that evaluate and select features under the training
process using methods such as decision trees. As this project will not be working with high-
dimensional and complex datasets, these methods will not be used here. However, they are
important to keep in mind for more complex environments or when a greater insight into the
feature space is needed.
In this project, features are distinguished by their visual cues. This approach is based on the
human notion that if something looks different, it might have different properties than something
else. An example of how visual cues might impact a learning scenario is a professor’s favorite
cup. Suppose a learner tried to make the professor as happy as possible by brewing them a cup
of coffee. If the professor had a favorite cup, the learner would hopefully learn that brewing
coffee in that specific cup made the professor extra happy. Now, if the learner switched the cup
with an identical one, the professor would probably never notice. The same logic applies to this
assignment’s navigation task. The expert may have had a certain tile they preferred over a lot
of identical tiles, but it would be very hard for the learner to infer this.
Therefore, features in this assignment are simply distinguished by their appearance. For example,
a tile with the letter ’A’ on it will have a different value associated with it than a tile with the
letter ’B’ on it. An example of how these tiles and subsequent features are visualized in MAvis
can be seen in Figure 6.

4.3
The Expert/Demonstrator
A main part in the paradigm of IRL is to have a expert to observe and learn from. As mentioned
in 2.2.4 the expert needs to be proficient at solving the task at hand for the learner to infer
meaningful behaviour, but the expert does not necessarily need to be optimal depending on
which IRL technique is used. For this project the expert is optimal and the following section
will explain how value iteration was utilised for the experts implementation. The section will
then explain how the expert generates data for the learning agent and showcase how this data
is structured. Lastly the section will cover a small example of the expert generating data using
the implemented components.
4.3.1
Value Iteration
Introduced in section 3.1 this section will explain the implementation of the value iteration
algorithm. In the code-base the implementation can be found in the class called ValueIteration.
This class is upon its instantiation populated with a set of all states for the given level. This
allows the class to iterate over all states, which is of course needed to compute the utility function
V (s). On figure 7 a high level abstraction of value iteration algorithm is presented, where the
theory from section 3.1 is clearly visible. Using the transition function, reward function and
discount factor as values in the bellman update, the utility of the next iteration gets calculated
in the inner loop by taking the action with the highest utility score is. Once all states have
been iterated through, then the utility function is updated with the values newest value and the
loop is executed again. This will repeat over and over until the updates get so small the loop is
broken.
Figure 7: Value Iteration expressed as high level code[12]
The reason why the value iteration algorithm was chosen for this project was due to a couple
of different factors. Firstly, the value iteration algorithm is, as the equations also show, a very
simple algorithm to implement. This was seen as a strength as the project revolves more about
the IRL agent than the expert. Having a simple expert allowed more time to IRL functionality.
Secondly, the intention was never to run the expert in large/infinite state spaces in which the
value iteration algorithm fails as it runs in polynomial time[11] depend on the amount of states.
This it normally not a problem but if the client was run on very large environment the algorithm
would take to long to converge.

4.3.2
Data Generation
By employing value iteration, the expert can solve a Markov Decision Process optimally, becoming
an expert in the specific instance. The subsequent problem was to make this data available to the
learning agent. As described in Section 2.2.3, this data should be in the form of a list consisting
of one or more trajectories that contain a sequence of state-action pairs. Listing 1 provides an
example of the trajectories over a simple environment. The interpretation of the elements of this
textual representation will be explained shortly. Having these trajectories allows the IRL agent
to infer an approximated policy and measure how likely this policy is to be the expert’s policy.
The MAvis tool does not support data extraction functionality, so it was necessary to build
it into the client tool. This section will explain how this functionality was implemented and
showcase an example of the expert generating five trajectories.
In order to create this data extraction process, several different requirements needed to be met.
These are as follows:
1. Extract a list of actions taken to complete a level.
2. Extract a list of states visited to complete a level.
3. Packaging these lists of states and actions together in state-action pair sequences.
4. Generation more than one trajectory on level.
5. Deliver the generated data in a readable format.
The first two requirements for the data extraction process are quite similar and both are solved
using the same technique. As introduced in section 4.1, the method generateTrajectories() is
used to invoke the Generator class. The Generator then uses the value iteration algorithm to
first solve the given level. Once the utility function is created and has converged for the given
MDP, the Generator queries the ValueIteration class with its current state. The returned action
is selected based on which one leads to a state with the highest utility score. Both the state and
the optimal action are then appended to their own list. These lists are indexed based on the
sequential order that the state and action appear. So, the initial state S0 is placed as the first
element, and so on. Once a sequence of actions leading to the goal state has been found, then
the lists are sent to a Writer class. This flow is also showcased in figure 8

added to the JSON file. This is expressed in the file as amountOfTrajectories and specifies how
many times the sequence in Figure 8 should be executed. However, since it would not make sense
to have the expert execute the same solution over and over, it is instead moved to a random tile
on the same level, checking that this tile is, of course, not the goal tile. The expert then queries
the utility function and solves the MDP again. This solution is then added to the trajectory file,
and the agent is moved to a new random tile again. This process is continued until the specified
number of trajectories has been created.
With these five requirements implemented, the expert meets all the needs for the IRL agent to
estimate the reward function of the expert with varying amounts of data with random starting
positions available.

4.4.2
Maximum Likelihood Inverse Reinforcement Learning
The main functionality of the entire client is of course the maximum likelihood estimation
method. This functionality is as stated in section 3.2 what allows the agent to approximate
which rewards weight was used to generate the expert trajectories. The following section seeks
to explain how this estimation process works and provide clarification on how is functions as to
give a better view into how implementation of the method took place. The maximum likelihood
inverse reinforcement learning (MLIRL) algorithm in this project is based on the implementation
described by Monica v Babes[20]. The full implementation details from the paper can be found
in appendix C as well as the actual code used in this project, which can also be found on github5.
To showcase how the algorithm works, it is natural to start with the beginning where a hypothesised
θA is used to induce a probability distribution over the action space available over all states.
Recall that reward function is assumed to be a linear combination of features which means θA is
just some randomly assigned weights, at least in the very beginning, that is used to calculate the

and additionally since the action space is a probability distribution this allows the algorithm to
assign likelihood to the trajectories in D. Having calculated the rewards of each state enables the
state-action utility function ”V (s)” to be computed. The reason why θA induces a probability
distribution over the action space is due to how it is incorporated in the state-action utility
function. Normally the bellman equation uses the ”max” operation as in section 3.1, but in this
case instead the Boltzmann exploration method is used, also known as the softmax operation,
because MLIRL works by gradient ascent. The V (s) function can be defined as:

To further clarify how the algorithm described above operates, a small example will be showcased.

The example is the same as the data-generation one, so Figure 9 can be used for reference. The
variables γ and β will both be set to 1, as this example is by design kept very simplistic as the
coming sections will explore larger and more feature-rich MDPs. In order to run the algorithm,
expert trajectories are of course needed. In this example, there will be two trajectories: τ1 and
τ2, where τ1 = {(s1, right), (s2, right)} and τ2 = {(s2, right)}.
The algorithm is run for one
iteration, and the inner value iteration loop is run twice.
The initial θA = 0, 0, which means the probability induced for the action space at each state will
be 1/|A|, where |A| denotes the number of available actions in a specific state. This inherently
makes sense as the rewards of each state at this specific time will be 0, and thereby all actions
in a specific state will be equally likely. Using the value iteration approach, both the utility and
state-utility function will be 0 at all states. Since both these functions depend on the rewards
and therefore depend on θA, these will stay at 0 until θA is updated after the first value iteration
cycle is finished. As stated, the Boltzmann exploration policy will lead to a uniform action space
as all actions are equally likely. For example, in the first state s1, both π(s1, stay) and π(s1, right)
will be 0.5. This equals a log likelihood of the two trajectories of approximately −1.4 calculated
with the following method: Li = 0.5 × (log(0.5) + log(0.334)) + 0.5 × (log(0.334)) ≈−1.44319.
Where 0.5 comes from the likelihood of τ1 or τ2. Finding the derivative of this log likelihood
with respect to the reward weights leads to a decrease of the normal tile feature. However, as
the agent moves towards the goal, the goal tile feature increases. If value iteration was only
run for this one step before updating θA, it would be the following: θA = {−0.6666, 0.6666}. In
the second value iteration cycle, the same process is run, calculating over each value again. At
this point, θA has not yet been updated, but taking the derivative of the same log likelihood as
before reveals that more emphasis should be put on θ2 and less on θ1. The second iteration ends
with θA = {−1.05, 1.05}, which, as typical when working with value iteration, is an update a
little smaller than the first. This would continue until the algorithm converges, and the updates
become insignificantly small.

4.5
Active learning
In this section the concepts introduced in 3.3 will be transferred into a specific implementation
of active leaning. More specifically, the aim of the section is to elaborate on how the active
learning framework functions and how it utilises the MLIRL method to gain a better estimation
of the experts reward function. The combination of MLIRL and active learning was one of the
projects main objectives and how this objective was achieved will be demonstrated by answering
some key questions.
When looking a the concept of active learning, explained in 3.1, a few components must be
present with respect to how the flow of active learning functions. Firstly, the data used in this
project is not labeled and neither will it be, instead the approach became to make the data
as simple as possible. That way, even without a label, the intention of the expert would be
very clear. A natural question would then be something like ”what does the learner ask the
expert?”. However, before answering that a more basic question of how the IRL agent even
should communicate with the expert will have to be acknowledged. Lastly, a natural follow
up questions would be ”what does the IRL agent do with the answers from the expert?”. The
following section seeks to answer these questions which should allowed the reader to gain a good
insight into how active learning functions in this project.
The first question to answer is that of communication. As the IRL agent needs to ask the expert
for more information in the form of more trajectories, a clear need for a channel between the
IRL agent and expert arises. The solution has been depicted on figure 10. The solution takes
advantage of the decoupling of the client application through the modes the client supports.
Recall that the client can be invoked using a shell environment, such as Bash6, to call the
entry-point of the client. This promoted the creation of the class called Bash.java.
The Bash class uses the built in java object called ProcessBuilder which allows java to start
other processes on the machine executing the client. The working directory of the bash script
is the same as where the Bash.java file is located. Using some simple bash commands the
ProcessBuilder navigates out of the current working directory and into the src folder. From
here it starts a new run of the expert but now the expert is run on a different level than the
original. This process can be executed multiple times which allows the IRL agent to make as
many queries as wanted the expert. As mentioned in section 4.3.2, the expert simply just writes
the needed trajectories to file, so the IRL agent can simply just fetch these before querying the
expert again.

presented with two types of features. If the expert is drawn towards one feature over the other,
the MLIRL method should be able to clearly catch and replicate similar behaviour. This is then
repeated for each permutation of features that was present in the original problem.

Experiments: Using IRL with active learning framework
Having talked about the theory surrounding IRL and active learning and implemented these in
the code base the coming section is all about testing these theories and methods. The section
aims to explain the context in which these experiments are placed and why they are able to
quantify the difference between the methods chosen. To do this the report will firstly provide
the some contextual information about the experiments like what is the domain, what is the
expert trying to achieve and what the expected outcome is. Afterwards the section will line
up three experiments that will be exploring different aspects of the MLIRL method, with and
without active learning. The first experiment is a small scale MDP with few features that was
designed to test whether approximation capabilities of the active learning framework were better
than just having the expert create a bunch of trajectories. The second experiment has the same
goals as the first, but is tested in a larger MDP to see how the size of the problem affected the
approximation process. Lastly, the third experiment is designed to checked whether the IRL
agent is able to duplicate the experts behaviour even when the environment change.

5.2
Experiment one: Small Room
The first experiment were created to test the hypothesis that active learning could be combined
with MLIRL to approximate the value function of an expert. The IRL agent will have three
different trajectory set to approximate the experts reward function from. The first dataset will
contain 100 trajectories in which the expert is placed randomly in the environment and will
navigate to its goal. The second dataset will be made up of 10 hand picked trajectories from the
expert. The reason why this is done is because the random dataset will also have trajectories in
which the expert starts on ”forbidden” tiles, this will be elaborated in following descriptions of
the environment. However, these trajectories will add some noise to the dataset as the expert
normally would avoid standing on these tile, but when its starting position is picked randomly
this cannot be avoid, which explains the existence of the second dataset. The last dataset, is
the one where the IRL agent uses active learning, it will therefore query the expert in order to
learn which features the agent prefer. This experiment was used as the foundation that the two
coming experiments were build upon.
5.2.1
Domain
The environment is visualised on figure 13 and is a 3x3 grid that was created with the intend
to simulate a room. To keep things familiar imagine the agent as a robot vacuum cleaner. The
yellow tile ”0” is the charging station which is located in the bottom right corner and the doors
leading into the room is represented with the ”A” tiles on the map. These door tiles is considered
dangerous to the agent and should therefore be avoided in order for the agent not to take damage
from a door being opened.

5.2.2
Goals
The goals of this environment is to navigate safely on to the charging station. For the IRL agent
to do so it must learn from the expert to avoid the A tiles whenever possible, and additionally
learn that the tile ”0” is an attractive state.
5.2.3
Expectations
The expectations of running this experiment is that all three different dataset will lead the IRL
agent to approximate the expert reward function to a degree where it is able to recreate the
optimal behaviour showcased from the expert. However, there is a clear assumption that the
ratio between the reward weight between feature 0 (the normal grey tile) and feature 1 (the
”A” tile) will be much smaller, as the IRL will have a harder time to distinguish whether the
expert prefers to navigate away or onto the A tiles due to the random starting position in the
first dataset.
5.3
Experiment two: Self storage center
Having test the approximation method in a small scale setting, the second experiments aim
to test whether the MLIRL method itself and the method coupled with active learning is able
to scale. In this experiment the expert will navigate through the larger environment with a
increasing number of features. The experiment itself is therefore also divided into three parts.
The first part where there are four features presented. The second with six features. And the
last part with 8 features that will have to be approximated.
5.3.1
Domain
The environment in this experiment is suppose to represent self storage center. These types of
environment often consists of a a lot of small corridors as this is the most space efficient solution.

5.3.2
Goals
The goal of this environment is to navigate through the hallways with the least traffic. Imagine

constant which is only turned of once the goal is reached.

degrees of data. This will of course be done for the parts of the experiment. More clearly, all
environments no matter whether it is the four,six or eighth features will be approximated using
the 50,100 or 150 random trajectories.

Experiment three: Self storage center under maintenance
Now that both a small prof of concept and larger more behavioural complex example has been

can carried over into a environment that changes over time. This experiment will use the results
from the previous test but will however alter the MDP and see whether the IRL agent can
showcase the same behaviour as the expert in a environment it has not seen before.

Domain
This domain will be the same self storage center as before. However, as seen on the figure 15

5.4.2
Goals
The objective of the IRL agent in this environment is replicate the same behaviour in a environment
that is different from the one that was used to approximate the expert rewards.
If a good
approximation has been made then the value iteration algorithm should return the same route
for both the expert and the IRL agent.
5.4.3
Expectations
With the random trajectory data set the IRL agent is expected not to recreate the same behaviour
as the expert. This is based on the intuition that the random trajectory data will simply not
have enough data point for the IRL agent to effectively approximate all the feature in this
environment. This should however be the case with the active learning framework.

6.1
Results of experiments 1
The first experiment, called ”small room”, the IRL agent was tasked with navigation of a small
3x3 grid. The results have been depicted on 16, which as expected show that MLIRL method
without active learning struggles to recreate the experts behaviour.
The reason for why it
struggles is due to its approximation of the features ϕ2, ϕ3. These two features encodes the
normal tile and A tile and when |θ2| < |θ3| then the behaviour of the IRL agent changes to
favour walking on the A tiles rather than, as the expert, avoiding them.

clear to see that the randomly generated actually has a few trajectories in which ϕ3 get expresses

tile helps to get a better approximation. A possible reason why this is the case could lie in the
fact the algorithm applies a positive value towards a certain features when the expert moves
from a state were i.e feature ϕ3 was active, to one where ϕ2 was active instead.

6.2
Results of experiment 2
Where experiment 1 was about understanding which dynamics impact the approximation process
and testing the basic functionality of the MLIRL method with and without active learning, the
second experiment is much more focused on scale. The added complexity of a larger and more
feature rich environment definitely impacted the IRL agents ability to approximate the experts
reward function. Figure ?? showcases how well the MLIRL method performed in the domain
with 4 features. Again, the MLIRL method seemed to be struggling with approximating θ2 > θ3
which results in behaviour that deviates from the experts as shown on figure 20. Adding more
trajectories to the estimation process does, as illustrated on the same figure, not increase the
precision of the approximation. This could indicate some possible pitfalls like local optimum in
which the process gets stuck.
In general the MLIRL method in its current state is a bit fragile.
Often the process seems
to diverge leaving the approximated numbers in a unusable state, as they either grow to large
or incredibly small. The active learning framework does however capture the experts reward
function and acts optimally when solving the MDP no matter the starting position.
This
suggests that the active learning framework does help the MLIRL method scale better and
possible explanation for this could be that instead of approximation everything at once, the active
learning framework breakdowns the approximation to smaller and contained instances.
The
added complexity of the larger environment, should therefore not influenced the approximation
to the same degree, which is supported by the generated results.

Discussion
As with any project there have been some failures and some achievements throughout the span
of this project. This section aims to take a step back and discuss what went right, but certainly
also what went wrong. In order to have of a structure to how these point delivered, a list have
been made. On this list each of the main point of this discussion can be found. Each paragraph
of this section will start of with a emphasized rephrasing of the same point found in the list.
The points the following discussion will cover is:
• Experiments and results
• Code
Experiments and Results: There are quite a few things that could have been done more
better or prioritised higher when it came to the phase of producing results. The MLIRL method
took a longer time than anticipated to have functioning which did not leave a great amount
of time for brainstorming or executing different experiments that would show the capability or
dynamics of the active learning framework nor the normal MLIRL method. One thing that in
particular that seemed like a obvious flaw was not to incorporated the recreation of the original
results from the paper by Vroman[20]. At the time of implementation the example provided
in the paper seemed to work with a environment and testing setup that would be too time
consuming to recreate. However upon reaching the project conclusion this reason was invalided
as a rereading of the paper, showed that this would indeed have been possible within a reasonable
time frame. If the papers results had been recreated a much better foundation of this project
experiments would have been ensured. The current foundation the experiments build upon is the
hand written example from the appendix and the fact that it worked in the initial environment
that was used in experiments 1.
Another shortcoming in regards to the results is that the variety of dynamics the experiments
cover is rather low.
Two problems there was also mentioned in the results section are, one
the problem of divergences in which the algorithm answer becomes invalid as the approximated
values start fluctuating to either extremely negative of positive numbers, second the problem
of not reaching converges on in some experiments indicate that either the algorithm does not
work as intended tying in to the point above, or the algorithm reaches a local maximum. As
the algorithm is prone to getting stuck in local maximum an experiment that explored which
dynamics could help mitigate could also be argued to be more valuable than experiment 3.

CONCLUSION
Lastly is the active learning framework that was build on top of the MLIRL method. Even though
the method proved better at approximating the expert reward function in some environment
like in experiment 2. However this is only a very slight indication that the method will actually
perform better in alternative environments. A case could be made that the project should have
allocated more time to the testing this framework and maybe even have reiterated on how it
functions. There is likely be many other approaches to implementing a active learning framework,
a in the scope of this project only one of these were tested, and furthermore only tested in a
very limited capacity.
Code: A point should be made about how important the code aspect is to a project about IRL.
To code the method is of course necessary in order for the the project to be completed, but how
much emphasize should be put in making more than just functioning prototype. The approach
chosen in this project was more on build a reliable tool that made it easy to run test and use the
MLIRL method in general. This more modular approach were meant to make the code easier to
maintain and pass on if there was a wish to do so. Doing this has however also taken time from
the project other aspects such as experimentation with the active learning framework. This is
therefore a short term vs long term choice about whether a coding a tool that allows for easier
testing and extension in the future was worth the time, if it might never be used again. And
valid argument can definitely be made about whether the IRL client might be over engineered
and a more crude prototype could might have done the same job but left more time for the other
parts of the project.
Lastly, a point should be made about the choice to implement the MLIRL method from scratch.
Choosing to code something, especially something as complex as the MLIRL method, comes with
a lot of risks. First, there is a large risk of misunderstanding some implementation details from a
scientific paper, or alternatively, that the paper might contain a flaw. Secondly, designing efficient
code is a time-consuming and difficult process. If a Java implementation of the MLIRL method
had been available, it would have probably been used instead of the current implementation. As
mentioned in the results section, the code’s current state is usable but, at times, very fragile. As
problems sometimes occur hundreds of iterations into an estimation process, finding the cause
can be a tiresome endeavor.

Conclusion
This paper has explored the field of inverse reinforcement learning. By surveying the literature
and publications this project has provided an entry point into the field of IRL from which
further study can take place. The research highlighted introduced a solution to the IRL problem,
which is the method of maximum likelihood inverse reinforcement learning, and this has been
further investigated.
Clarifications of how this algorithm functions has been made and an
implementation of the method was created in client-side tool, in the client-server architecture,
for the visualisation tool MAvis. This IRL tool provides all capabilities to run an IRL process.
From solving a given MDP optimally by using the value iteration algorithm and showcasing
the optimal behaviour via a expert, to extracting these expert trajectories and making them
available for the MLIRL process is present in the tool.
On top of the functionality to run the maximum likelihood inverse reinforcement learning this
thesis has described and implemented an active learning framework. This framework, which
is based on the theory of the T-maze from behavioural science and has indicated an ability to
approximate better than the regular form of MLIRL. Both versions of the MLIRL method was
tested in three different experiments where the amount of feature was increased and thereby
making the expert reward function progressively harder to estimate.
However, none of the
methods tested in this project managed to solve all instances of the problems tested. Do to nature
of estimation method is was difficult to determine if this was simply due to the environments
tested, or problems within the implementation of the estimation method. The active learning

Future work
A fun part of any project is to plan a bit further than what scope assignment allows. Working
on a project for months at the time always yields in a bunch of ideas that might just be a little to
ambitious to actually implement or simply does not fit within the original scope of the project.
This section will therefore explore these thought that might could be whole new projects in their
own right or simply just use want have been built in this project as a foundation to extend upon.
Extension of the Active Learning Framework: An natural continuation of this project
would be to further develop and test the active learning framework from proposed in this thesis.
As the approach, as it currently implemented, could be extended by taking a page from the
divide and conquer methodology like the merge sort algorithm. This could achieved by adding
another layer on the current active learning framework in which the four lowest approximated
feature value were added to a simple environment, like the choice.lvl introduced in this project.
As the results of this assignment revealed that even though the active learning process was able,
in some instances, to guess the right ordering of features but get the right approximation of what
numeric distance there should be between these features. Coupling more features together than
just two at the time might help the algorithm to get a better approximation in this regard.
Multi-agent capability: Another natural continuation of this project would be to add the
multi intention algorithm which is also specified in the paper by Vroman[20]. This would enable
the IRL agent to approximation which of the multiple agent in environment created a certain
trajectory. With this capability it would be interesting if the IRL agent able to approximate
other agents reward function and thereby help the specified agent. This could maybe be achieve
with the approach specified in the paper by Hadfield-Menell, Russell, Abbeel, Dragan[8] where
the problem is cast a two player markov game.
