Abstract
This report presents real-time Visual Odometry pipeline implementation details for embedded plat-
forms. The expected outcome of this project is to present eﬃciency vs accuracy recommendation for
diﬀerent VO algorithms. The project aims to use OpenCV library as much as possible to reduce devel-
opment time and rapid prototyping. OpenCV oﬀers CUDA implementation of some of the Computer
Vision (CV) functions which can take advantage of GPU acceleration. In short, this project is aiming
to compare the improvement in runtime by taking advantage of a multicore CPU using OpenMP and
oﬀ-loading time-consuming tasks on GPU.
Work related to Visual Odometry has been carried out in the last couple of years at DTU Space.
That work is ported and optimized using C++ as the ﬁrst thing in this project so that a baseline results
are available. Depending on the results, the author aimed to improve speciﬁc stages in the VO pipeline.
The most time consuming stages in VO pipeline were identiﬁed and they were modiﬁed to use OpenMP
and/or GPU. The author introduced ORB features from ORB-SLAM2 (This is referred as ORB2 in this
report) and proposed a new pipeline with a mix of descriptor based stereo match and optical ﬂow-based
feature tracking. An exhaustive benchmarking is performed by sweeping parameters in feature extraction
and feature matching. Benchmarking results contain runtime results of each VO pipeline stage and Root
Mean Square Error (RMSE) of pose estimation. The results help to conclude the parameters that the
user should choose when making trade-oﬀbetween runtime and accuracy while implementing VO for a
speciﬁc application.
KITTI dataset is used in the beginning for benchmarking and once the VO pipelines are running,
the author integrated Intel RealSense D435i stereo camera into VO Algorithms. A program is developed
that takes images from D435i, performs pose estimation at 15FPS, and sends the debug image and the
estimated pose over socket - all of this happens real-time.

Chapter 1
Introduction
This section describes the motivation behind this project, its scope, and a general idea about Visual
Odometry (VO) systems.
1.1
Motivation
Finding an accurate location of the autonomous vehicle is one of the trending research topics in the
ﬁeld of robotics. In simple words, the localization technique answers - “Where am I?”. There are numerous
methods to localize the autonomous vehicles. A wide range of sensors are available which can aid in the
perception of the surrounding environment ex. LIDAR, RADAR, Camera, GPS, etc.
GPS cannot sense the surrounding of the autonomous vehicle. Moreover, GPS position is accurate only
in outdoors and dependent on the environment conditions. The GPS performance degrades signiﬁcantly
inside the building, tunnel, forest, underwater, etc. Active sensors ex. LIDAR, RADAR etc. cannot
discern between objects of similar shape. For example, it is diﬃcult to determine if the object in the
middle of the road is a plastic bag or rock.
Visual sensors can overcome such limitations and hence it is getting attention among researchers.
Vision based localization is getting popularity for its bio-inspired working principles, simplicity, and the
ability to discern objects. We humans, rely mainly on vision to perceive the surrounding, guidance and
navigation. We estimate the depth or distance by using pair of eyes. Visual Odometry (VO) is the method
of estimating the position of autonomous vehicles using a single or multiple camera.
1.2
Project scope
This project aims to carry forward the work performed by Runar, Johannes, Thor, and Jakob at
DTU Space[3]. These authors explored diﬀerent blocks in VO pipeline and modiﬁed them to increase
the accuracy in pose estimates. However, none of the work is carried out to execute these algorithms
in real-time and speciﬁcally for embedded platforms. The author aims to implement VO pipeline using
these author’s work and then benchmark it for eﬃciency vs accuracy. Depending on the results from these
benchmarkings, the author will try to optimize and reduce the runtime of a speciﬁc stage in the pipeline.
1.3
VO Systems
The term VO reﬂects similar operation to wheel odometry. Wheel odometry estimates the position
of a vehicle by incrementally integrating the number of turns. Likewise, VO estimates the the pose of
the vehicle by incrementally estimating pose of the vehicle using cameras[4]. VO is essential in the ﬁeld
of autonomous vehicles, Virtual reality (VR) and Augmented reality (AR). In GPS denied environments
like tunnels or space, VO systems are very useful to estimate the position of the vehicle.

VO vs SLAM
Simultaneous Localization And Mapping (SLAM) is a method of constructing a map of an unknown
environment while localizing the vehicle in that environment[5]. SLAM estimates the path of the robot
globally while VO motion estimate is locally i.e. only relative motion estimates are calculated. SLAM
builds the map of the environment even though a map is not explicitly required. It needs a map to detect
previously visited areas, which is called loop closure. Look closure helps to reduce the drift in motion
estimate. On the other hand, VO estimates the path incrementally between two frames. The poses are
then concatenated to calculate the pose with respect to global coordinates. VO may be used as a building
block for SLAM. If the user is not interested in building the map of the environment, then VO should be
used. SLAM is more precise than VO because it uses more constraints in estimating the pose. However,
it also requires more computation power as it needs to keep the history of all previously visited places[4].
User needs to select trade-oﬀbetween performance and accuracy.
Mono Vs Stereo VO
The main diﬀerence between monocular and stereo VO is that stereo cameras can calculate 3D features
at every single frame. Monocular VO requires three views to solve for the scale. Since stereo camera
estimates the depth in a single frame rather than from adjacent frames as in the monocular case, the
stereo camera exhibits less drift in VO pipeline[4]. Stereo camera has uncertainty in depth estimation
and this uncertainty grows larger when the distance to the object is very far than the baseline distance.
In such scenarios, stereo VO are not useful. Monocular VO can perform well in such cases.
Summary
This chapter glanced through motivation behind the project and scope of the project. A very basic
VO operation is mentioned and diﬀerence between VO vs SLAM and Mono vs Stereo VO is presented.
The next chapter will explain about camera models, camera calibration and basic building blocks of VO
systems. .

Chapter 2
VO Theory
This section describes the pinhole camera model, intrinsic and extrinsic camera parameters, and basic
building blocks of a VO pipeline.
2.1
Camera Modeling and Calibration
This section describes perspective camera model, camera parameters, and calibration of camera.
2.1.1
Perspective Camera Model

2.1.2
Camera Calibration
Camera calibration is the process of estimating the intrinsic and extrinsic parameters of a camera.
The camera coordinate axis and the world coordinate axis do not align always. Sometimes, it is necessary
to transform between the camera coordinate system and world coordinate system. The extrinsic camera
matrix transforms world coordinate system into a camera coordinate system[1]. MATLAB has toolkit
to calibrate perspective and ﬁsh-eye mono as well as stereo camera [1]. OpenCV provides C++ API to
calibrate ﬁsh-eye and perspective mono as well as stereo camera[6].
2.1.3
Distortion and Rectiﬁcation
Distortion
Images acquired by a camera may get distorted due to imperfections in the optical system. In simple
words, the distortion causes a straight line to appear curved in an image. Mostly, distortion follows a
pattern. Depending on the shape of the pattern, the type of distortion is categorized as either radial
distortion, pincushion distortion or barrel distortion.
Distortion causes a point to appear on diﬀerent location in an Image than it would have been without
distortion. If the distortion parameters are not incorporated into camera model, then the calculations for
depth estimation or pose estimation will be erroneous.
Figure 2.2: Distorted Image (left) and Undistorted Image (right) from EuROC dataset
Rectiﬁcation
Image rectiﬁcation is the process of making stereo camera image planes parallel to each other. In
other words, image rectiﬁcation makes the epipolar lines of the stereo camera parallel.
Rectiﬁcation makes it easier to perform epipolar ﬁltering which helps to remove outliers. After recti-
ﬁcation, the epipolar lines become parallel and hence the search window for stereo matches narrow down
to a single row in an image. Note in Fig.(2.3) that corners of the chess board lies on the same line on the
rectiﬁed image.

2.2.3
Feature Matching/Tracking
Feature matching is the process of ﬁnding the best match between keypoints of one image from set
of keypoints of another image. The similarity between two keypoints are calculated based on keypoint
descriptors. To compare the similarity, a cost function is calculated. Cost functions can be hamming
distance, L2 or L1 distance, etc.
In VO pipeline, stereo matching is immediate step after feature extraction. Stereo matching is the
process of matching keypoints from image Il,T to keypoints in image Ir,T . Feature matching is also used
in descriptor based feature tracking methods, i.e., circle matching.
Some VO algorithms uses optical ﬂow-based methods instead of a descriptor to ﬁnd or match the
keypoints.
2.2.4
Motion Estimation
Motion estimation is one of the most important part in VO pipeline. In this step, the relative rotation
and translation change is estimated between the current frame and previous frame.
By adding the
subsequent estimated poses, the relative pose of the camera is calculated with respect to the global
coordinates.
The estimated rotation and translation values are with respect to the camera center. It is assumed
that the camera is rigidly mounted on the vehicle. When the vehicle pose is mentioned in this report,
it will be assumed that the pose is with respect to camera’s center. It is also assumed that the vehicle
is rested at (0, 0, 0) which is aligned with the global coordinate system. All subsequent relative pose
estimations are transformed back to the global coordinate system (which is the position of vehicle before
VO pipeline starts).
Since the estimated poses are concatenated, the error variance in pose estimate grows over time. This
is one of the limitation of VO system. VO system always exhibits the drift in pose estimates. Optimization
is often used to reduce this drift.

Optimization
The last step in VO pipeline is to optimize the predicted poses. The optimization can be local or
global. Local optimization means the pose prediction is reﬁned by taking into account the pose estimation
of the last n poses. Global optimization means the pose prediction is reﬁned by taking into account all
pose estimations till now.
Pose-graph and Bundle Adjustment are methods for pose optimization.
This step is optional although. It has been reported that optimization improves the position estimation
by a factor of 2 to 5[4]. This part of VO pipeline is not explored in this thesis. All VO pipelines mentioned
in this report does not perform optimization on pose prediction.

3.1
Embedded platform selection process
This project aimed to execute VO pipeline in real-time. A naive way of achieving this is to use a more
powerful CPU. However, the author wanted to look into a solution that can be mounted on UGVs or
drones and compute everything online, i.e., without passing data to other computer for further processing.
Vijay Kumar et. uses Intel NUC based computer to run VIO in real-time at around 30 FPS[7]. The
authors regarded the solution as bulky in another paper[8]. Moreover, most NUC based kits does not have
dedicated graphics card. This narrows down the search from x86 computers to ARM based embedded
computers.
The author preferred to have NVIDA GPU on the embedded platform. As explained before, the project
targets to use OpenCV as much as possible. OpenCV oﬀers APIs for NVIDIA’s CUDA compatible GPUs.
This reduces the development time because OpenCV oﬀers a lot of API and one does not need to write
programs speciﬁc to any accelerator chips. For example, FPGA also oﬀers concurrent execution. However,
the author did not want to look into VHDL or any other hardware description languages. Hence, the idea
of using FPGA is also dropped.
Apart from these, the board should have PCI-e or SATA interface for SSD to store captured/processed
images. It should have WiFi or Ethernet connectivity for transferring data or programming and USB 3.0
ports to connect stereo cameras.
In terms of software, the embedded platform should be capable of running the latest Ubuntu or any
Unix based distribution. Ubuntu is preferred because Debian packages are easily available and easy to
install.
This reduced the search to NVIDIA’s Jetson Platforms. The author selected Jetson TX2 and Jetson
Nano because it matches with aforementioned criteria. Several researchers have used TX2 to test VO
in real-time[8]. Since, TX2 is already powerful enough, it is not required to select more powerful Jetson
Xavior platfor. Hence, other product lines in Jetson family are not investigated.
Jetson Nano is selected because of its low power and small form factor. This gives end users more
freedom to choose from. There are certain applications where small form factor and low power consump-
tion is necessary (Rovers in space explorations). And this is one of the main goals of this project - list
trade-oﬀbetween performance vs accuracy and let the user decide the system conﬁguration depending on
the application.

OpenCV APIs to extract features
OpenCV library has API to extract ORB, SIFT, and SURF features. All API related to feature

The descriptor based feature matching requires to have a descriptor associated with feature points.

ORB2: ORB from ORB-SLAM2
Note that ORB2 is not an actual term associated with feature type. It is used throughout this report

4.1.4
Stereo matching
Methods for Feature Matching
OpenCV oﬀers two types of descriptor based matching : Brute Force (BF) and Fast Library for
Approximate Nearest Neighbors (FLANN).
Brute force is very naive algorithm. It matches the descriptor of a keypoint from one image with all
descriptors in the other image and returns the best match. The best match is calculated in terms of a
cost function. Examples of cost functions are L1-norm, L2-norm, hamming distance, etc.
If the left image has n keypoints and the right image has m keypoints, the time complexity of BF is
O(mn). For the same number of features in left and right image, the execution time for BF increases
exponentially if the number of features increases.
FLANN builds K-D Tree of keypoint descriptors and uses the fastest near neighbour method to cluster
the data and approximately ﬁnd the nearest neighbour. D.Lowe’s ratio test is applied on the matches
found by FLANN to reject outliers[14]. It is observed that FLANN reports more false matches than Brute
Force. Hence, the ratio test is very necessary.

4.1.6
Bucketing
Bucketing is the process of dividing an image into grids and ﬁlter out features in each grid based on
criteria like age, keypoint response, etc. Each grid is called bucket. Bucketing helps to select features
that are spread across the images. It has been noted by Scaramuzza, etc. that features spread across the
whole image produce better pose estimation as it takes into account the features near to the camera and
features far from the camera[16].
Bucketing also limits the number of keypoints to process and hence keeps in check the computation
requirement to process the feature points.
The author implemented quick sort algorithm to select keypoints in the bucket based on keypoint
response value. Once the image is divided into buckets, the keypoints are ﬁltered into buckets based on
their x-coordinates and y-coordinates. For a given bucket, if the number of keypoints exceeds maximum
number of keypoints per bucket, then bucketing is performed for that bucket, i.e., sorting the keypoints
based on its response value using the quick sort algorithm and selecting the keypoints with the strongest
response value. Rest of the features are discarded in that bucket.
OpenCV class cv::KeyPoint stores the keypoint data. It contains the public member response which
is used by quick sort algorithm for sorting.
Quick sort algorithm implemented speciﬁcally for cv::KeyPoint class is outlined in Algorithm Listing
5. Quick sort algorithm implemented uses the last element in the keypoint array as a pivot point. The
worst case time complexity of quick sort is θ

4.1.9
RANSAC
Features tracked with LK Optical ﬂow has more outliers than circle-based matching.
It is very
important to ﬁlter out those outliers in order to get more accurate pose estimation.
OpenCV API cv::ﬁndFundamentalMat() is used to estimate fundamental matrix between Il,T−1 and
Il,T based on tracked keypoints returned by optical ﬂow. FM RANSAC with 99% conﬁdence is passed
as arguments in this function. The function returns outliers and estimated fundamental matrix. The
purpose of ﬁnding fundamental matrix is to perform epipolar ﬁlter between Il,T−1 and Il,T . To ﬁlter
outliers between frames Il,T−1 and Il,T , fundamental matrix is required between those two view points.
Since the motion is not known, the fundamental matrix is estimated ﬁrst. Once the fundamental matrix
is estimated, the outliers are ﬁltered out based on the epipolar constraint.
Scaramuzza proposed a better way to estimate fundamental matrix using IMU which uses only 2-point
RANSAC[4]. Due to the time limit on the project, the author did not implement this.
Fig.(4.5) shows outlier ﬁltering using RANSAC on one of the image from KITTI dataset. The red
and green lines shows the motion estimated by LK optical ﬂow algorithm. Red lines show the outliers
ﬁltered after RANSAC. Green lines show the inliers.
4.1.10
Triangulation
Triangulation is the process of estimating the world coordinate of a point in an image. Since a stereo
camera is used, the triangulation process requires ﬁnding a match between the left and right image.
OpenCV function cv::triangulatePoints() is used to triangulate points after the stereo match. Even
after ratio test and epipolar ﬁlter, stereo match has outliers. If the outliers are used for triangulation, it
produces coordinates which are either too far from the camera or it has negative Z coordinates. (Negative
Z coordinates means the point is behind the camera which is not possible.) Hence, after triangulation the
points are ﬁltered out based on their X, Y, and Z coordinates.
The VO Algorithm in this project ﬁlters out 3D points for which either X > 2,000, Y > 2,000, Z >
2,000 or Z < 0. The 3D point units are in meters.

4.1.11
Pose Estimation
Only 3D-2D method is used to estimate the motion. Scaramuzza has pointed out that 2-D to 2-D
and 3-D to 2-D methods are much better than 3-D to 3-D methods[4]. For monocular VO, 2-D to 2-D is
preferred as it does not require point triangulation to calculate depth. For the stereo camera setup 3-D
to 2-D works best as the depth calculation can be performed without worrying about scale ambiguity.
Moreover, 3-D to 2-D requires only three point correspondence to remove outliers during RANSAC while
2-D to 2-D requires at least ﬁve points. Hence, 3-D to 2-D motion estimation is faster than 2-D to 2-D
motion estimation.
The problem of estimating the pose using 3D-to-2D correspondence is called perspective from n points
(PnP) algorithm. OpenCV function cv::solvePnPRansac() is used to solve PnP algorithm along with
RANSAC based outlier rejection[6]. SOLVEPNP EPNP method is used to solve PnP problem in OpenCV
API because Thor and Jakob reported that EPNP is the best in estimating position values[3].
VO Algorithms in the project uses 3D points at time t=T and 2D coordinates of these 3D points at
time t=T −1. Hence, cv::solvePnPRansac() estimates the rigid transformation from time t=T to time
t=T −1.
4.1.12
Pose update
Pose estimation step estimates relative rotation and translation change in subsequent frames from
time t=T to time t=T −1. The estimated pose is then concatenated at each frame to estimate the pose
of the vehicle with respect to the world coordinate system.

Chapter 5
Benchmarking Results
This section presents the benchmarking results of this project. The ﬁrst part contains a preliminary
benchmarking which compares the feature extraction timing, feature matching timing, etc. on Jetson
platforms. Second part of this section contains a detailed description of the timing and accuracy results
of all four VO Algorithms. KITTI Sequence 09 is used to generate data in this section unless stated
otherwise. A detailed description of KITTI dataset is listed in Appendix C.2. For all results in this
section, the word real-time is considered as 100 ms since KITTI dataset has 10 FPS.
All timing data presented in this chapter is calculated using C++ routine std::chrono::high resolution clock.
std::chrono::high resolution clock uses a high resolution system clock which has the shortest tick period,
i.e., it has the highest resolution. The accuracy and precision of the clock are not measured systematically.
It is assumed that the system clock is stable enough to report accurate timing up to milliseconds. The
project does not aim to benchmark the execution time very precisely, i.e., in microseconds or nanoseconds.
Execution time in this report is measured in micro seconds [µs] unless speciﬁed otherwise.

Preliminary benchmarking was the ﬁrst thing performed to check whether OpenCV libraries and

In all VO Algorithms, image acquiring, feature extraction and feature matching are the ﬁrst steps.
Literature survey indicated that these steps take a signiﬁcant amount of time to execute. An exhaustive
qualitative summary was not available for Jetson platforms. Hence, these three steps were benchmarked

Location of keypoints CPU vs GPU
CPU and GPU handles ﬂoating point arithmetic diﬀerently. Hence, results generated by CPU and
GPU may be a little diﬀerent for the same input data.
Since both Jetson Nano and Jetson TX2 SoC have ARM Cortex-A57 cores, ﬂoating point data handling
and ﬂoating point precision are the same on both boards. It has been observed that same code generates
exactly the same output on both Jetson Nano and Jetson TX2. The output could be location of feature
points or the estimation of pose in VO pipeline.
Jetson Nano GPU has Maxwell architecture and Jetson TX2 GPU has Pascal architecture. Pascal
architecture is predecessor of Maxwell architecture. Maxwell architecture lacks double precision ﬂoating
point arithmetic support[18]. Hence, the results will diﬀer a little between Jetson Nano and Jetson TX2
for the same input. It has been observed that the location of a couple of keypoints diﬀers a little from
the results generated by CPU, Jetson Nano GPU, and Jetson TX2 GPU. This diﬀerence is not much for
a single image, but when the VO pose are concatenated, the diﬀerence in end poses becomes noticeable.

gives the best results in terms of computation eﬃciency and accuracy.

Chapter 6
Integration of Stereo Camera
KITTI dataset was used so far to ﬁne tune the settings of the diﬀerent stages of VO pipeline. After
getting satisfactory results, the algorithms were tested with Intel RealSense D435i.
6.1
RealSense D435i information
D435i has an infrared stereo camera, one RGB color camera, one Infrared pattern projector, and IMU
(Fig.(6.1)).
Figure 6.1: Intel RealSense D435i camera
It uses active infrared projector to estimate the dense depth. It emits infrared dots on the scene.
Infrared stereo camera picks up this pattern and computes correspondence based on pattern matching
between left and right camera view. All of these processing is oﬀ-loaded to Intel’s custom chip - RealSense
D4 Stereo-vision ASIC. User program does not have access to the background calculations for depth
estimation. Depth estimation is available in real-time along with rectiﬁed and undistorted infrared and
RGB image streams.
6.2
Integration with VO pipeline
SDK for controlling Intel RealSense cameras is available for free use. It is called librealsense2 and is
available in C++11. D435i is conﬁgured and initialized through librealsense2. Under the hood, RS SDK
controls the D435i through USB Video Class (UVC)[19]. D435i is conﬁgured for image resolution, sensor
streams, frames per second and bits per pixel before VO pipeline start. Only the left and right infrared
sensor streams are tested in this project.
To integrate RealSense D435i device with VO Algorithms, only the image reading block is modiﬁed.
The image reading block is changed to use librealsense2 image polling API instead of cv::imread(). Based
on conﬁgured frames per second, Image is captured and passed down to librealsense2 polling handler. The
raw data from librealsense2 class is then copied into cv::Mat object for the left and right image. Once

the data is stored in cv::Mat object, VO Algorithms treats images from D435i similar to KITTI dataset.
The VO Algorithm works as expected.
6.3
Streaming debug image and pose using sockets
This feature is added in VO pipeline only for D435i camera. The basic idea is to stream debug images
(Images showing either circle matches or optical ﬂow) and the estimated pose over network - either LAN
or WiFi. Fig.(6.2) shows a simple diagram of socket programming.
Figure 6.2: Block diagram showing data streaming over sockets
C++ standard library on Unix delivers a very simple interface to send data over a socket. The unix
socket library is used in this project to stream images and pose information over the socket.
C++ program running on Jetson Nano or Jetson TX2 acts as a socket client. Python socket server is
conﬁgured on PC or a laptop. Python is chosen on sever side for its simplicity and ease of use. Socket
client waits till it hears something from socket server on conﬁgured IP address and port address. Once
it receives the echo from the socket server (PC or laptop), socket client (Jetson Nano or TX2) starts
executing VO pipeline.
On the client side, threads are used to send data over socket. Threads are used to make sure that the
latency in the network does not disturb the execution of main thread and the VO algorithm maintains
its original FPS. Once the pose is estimated, the program on the client starts a thread and detaches it
from the main thread. Detaching thread from the main execution allows it to run it independently from
the main thread. The detached thread draws tracked keypoints on the image, reduces its resolution by
half, convert it into raw stream, and sends it over the socket. The resolution is reduced so that the local
network has free bandwidth for other devices in the network.
The client receives the raw stream. This raw data is then converted into image and pose data and
displayed on the screen.

Chapter 7
Conclusion and Future Work
This chapter summarizes some important results from this project and then suggests a few methods
as a future work.
7.1
Discussion
Use of OpenCV
OpenCV provides a lot of functions related to VO. This project used OpenCV extensively from feature
extraction to pose estimation. OpenCV’s cv::Mat was very helpful to handle data and perform matrix
operations in C++.
OpenCV also ported few functions to NVIDIA’s CUDA compatible GPUs.
This
project used those APIs in developing VO Algorithm 2 and VO Algorithm 4. Since, OpenCV’s CUDA
APIs are fairly new, it lacks good documentation and has a few bugs. The current limitation of OpenCV’s
CUDA API restricts the same function to be executed in parallel on GPU with two diﬀerent data sets.
This is the bottleneck for VO Algorithms developed in this project. Feature extraction is the most time
consuming operation in the whole VO pipeline for a reasonable number of features per image. If OpenCV’s
feature extraction CUDA API can execute in parallel for left and right image, VO Pipeline 2 and VO
pipeline 4 runtime would have been reduce even further.
Use of OpenMP
OpenMP is used to reduce the execution time of image reading/writing and feature extraction on
CPU. Without using OpenMP, Any of the VO Algorithm cannot run in real-time on CPU. Again, this is
because feature extraction takes up the most time in VO pipeline and OpenMP cuts the execution time
for feature extraction almost by half.
On Selection of Jetson boards
Jetson Nano and Jetson TX2 are very powerful embedded platforms.
The benchmarking results
suggest that Jetson Nano can execute VO Algorithm 2 and VO Algorithm 4 with a reasonable number
of features per image. GPU on Jetson TX2 is very powerful than GPU on Jetson Nano. Hence, Jetson
TX2 can run Algorithm 2 and Algorithm 4 with a very high number of features.
NVIDIA’s oﬃcial
carrier board for Jetson Nano does not come with PCI-e or SATA storage expansion options. Jetson
Nano is booted using USB SSD drive. It has been observed that sometimes the benchmarking results are
more inconsistent with Jetson Nano than Jetson TX2, i.e., the standard deviation for some stages in VO
Algorithm are high. Some of the third party vendors sell Jetson Nano carrier boards with PCI-e or SATA
for SSD. The author suggests benchmarking on Jetson Nano with diﬀerent carrier board as a future work.

Thermal throttling on Jetson Nano
It is observed that prolonged use of Jetson Nano reduces its performance a little. Jetson TX2 has a
fan which is always turned on before executing benchmarking scripts. A carrier board with a fan should
be selected for Jetson Nano.
High error in Z estimation for ORB
ORB feature type is a good candidate for real-time VO algorithms. However, OpenCV’s implementa-
tion does not distribute ORB features evenly across the entire image. This produces a high deviation in
Z estimates. To improve the quality of pose estimation, the author introduced ORB2 at the later stage
in the project.
This problem is not observed in SIFT and SURF. SIFT and SURF features are nicely distributed over
an entire image. This was one of the reasons to introduce ORB2 into this project.
ORB2 feature type
ORB2 extractor code is ported from ORB-SLAM2 [12]. GPU implementation of ORB2 is ported from
freely available github repository by Author T.Ngyuyen[13].
ORB2 distributes features evenly across the image and ORB2 can be executed feature extraction of
left and right image simultaneously on GPU using OpenMP. The results suggest that ORB2 requires less
number of features than ORB for the similar value of RMSE. Also, ORB2 outperforms other feature types
in all comparisons.
Circle matching vs Optical ﬂow
The binary descriptor used by ORB and ORB2 does not match quite well during circle matching
stage. The number of feature points tracked by VO Algorithm 1 and 2 are less than the number of feature
points tracked by VO Algorithm 3 and 4. The accuracy of optical ﬂow-based VO Algorithms 3 and 4
is much higher than circle match based VO Algorithms 1 and 2. Generally, descriptor based matching
is more robust and accurate than optical ﬂow-based tracking. However, there are no sudden movements
or angular rotations in KITTI dataset. Hence, the optical ﬂow-based method seems to work better than
feature-based circle matching.
7.2
Future Work
This section discusses about the ideas that can be used in VO Algorithms described in this report.
Mostly the ideas are about ’How to run something faster on CPU or GPU?’. At the end, a diﬀerent VO
pipeline based on sensor fusion is discussed.
Software improvement for existing VO Algorithms
It has been established that feature extraction, feature matching and circle matching consumes a sig-
niﬁcant amount of time. To use more features per frame, code for feature extraction should be optimized.
Just like ORB2, OpenCV’s API for ORB should be re-written such that feature extraction for left and
right can be executed in parallel.
Other stages of VO Algorithms, ex. bucketing, epipolar ﬁlter, pose estimation, RANSAC and pose
update neither uses OpenMP nor GPU. These stages should be rewritten to use threads, OpenMP, ARM
Neon or CUDA.
Bucketing quick-sort keypoints locally for each bucket. There is no dependency of data from one
bucket to another one. Hence, quick sort execution per bucket can be run in parallel either using threads,
57
==PAGE:60==
Mitul Vekariya
s181344
Real-time implementation of a Visual Odometry system
on embedded platform
March 28, 2021
OpenMP or CUDA. Similarly, the epipolar ﬁlter for the rectiﬁed image compares y-coordinates of the
matched keypoints. This comparison for each keypoint is independent of other keypoints. Hence, the
epipolar ﬁlter can be executed in parallel using threading, OpenMP, ARM Neon or CUDA.
A complete optimizing using GPU requires rewriting all functions into CUDA. Starting from image
acquisition to pose update. Once the image data is uploaded to GPU, all operations may occur inside the
GPU without transferring data to and from the host memory and GPU. This requires signiﬁcant amount
of time and eﬀort. Hence, the author leaves this as a future work.
Real-Time Linux patch for Jetson Platforms
Jetson platform uses Ubuntu based linux distribution. Ubuntu is not Real Time Operating System
(RTOS). It has been observed during benchmarking that sometimes operations like image reading or
writing take longer time than usual. This is due to the fact that OS on Jetson is not RTOS. Function
call deadlines are not guaranteed. Linux Foundation releases a PREEMPT-RT patch that can be used
to build linux based RTOS for Jetson boards[20].
Add BA into VO Algorithms
BA can reduce the drift in pose estimation in VO. Open source library like ICE-BA can be integrated
into existing VO Algorithms[21].
Hardware improvement
Open Vision Computer uses FPGA along with Jetson TX2 module for real-time VO application[8].
The FPGA of OVC extracts the features and pass it to Jetson TX2. This is one of the methods to oﬀ-load
feature extraction from Jetson platform if the end-user does not want to spend much time on porting and
optimizing code for GPU. However, the GPU on Jetson TX2 is very powerful and it is able to extract
features in real-time[22].
Relasene D435i test with UGV
The author implemented the RealSense D435i interface to execute VO Algorithm 2 and VO Algorithm
4 in real-time. Due to the limited time, this could not be tested with the UGV against the GPS data.
The author leaves this as a future work.
Modiﬁed VO Algorithm speciﬁcally for Relasene D435i
RealSense D435i stereo camera uses active IR dot projector to calculate depth. The depth calculation
is performed by the camera ASIC and the end user can have live stream of left and right infrared images
along with depth information. Since the depth is precalculated by the camera, VO pipeline does not
require to perform stereo matching and triangulation. Moreover, the optical ﬂow-based VO Algorithm
needs only keypoint location. It does not require the descriptors. Hence, some time is saved during feature
extraction stage and the feature matching stage is completely eliminated. During PnP pose estimation,
3D points are parsed directly through D435i depth stream and 2D points are parsed through optical ﬂow
calculations. The author believes that this algorithm has a good potential of running at high frame rate
with more than 5000 features per image.
Integration of IMU - VIO
Vijay Kumar, et.
proposed to use optical ﬂow along with 2-Point RANSAC method for stereo
matching[7]. It has been observed in VO Algorithms 3 and 4 that optical ﬂow tracking is much faster than
circle matching. 2-Point RANSAC method uses IMU to remove outliers while tracking the features[23].
58
==PAGE:61==
Mitul Vekariya
s181344

on embedded platform
March 28, 2021
Using IMU reduces standard 5-point or 7-point RANSC to only 2-point RANSAC and hence speeds up
the execution of RANSAC stage. Since, optical ﬂow-based stereo matching does not require to calculate
descriptors, the feature extraction and matching time should be reduced signiﬁcantly. msckf vio[7] uses
this approach with FAST corner and optical ﬂow along with a kalman ﬁlter (to fuse IMU data) on Intel’s
NUC6i7KYK to execute VO at 30 FPS.
If optical ﬂow is used for matching features in stereo images, then FAST corner detection can be used
instead of ORB, ORB2, SIFT, or SURF. FAST detector is faster than the rest of the feature types. In
this project FAST is not used because FAST does not have a descriptor associated with it and the author
only used descriptor based stereo matching.
Optimizing CUDA code for TX2
Scarammuzza et. optimized Jetson TX2’s CUDA program such that FAST feature detection and
tracking could be performed at 1000 FPS on Jetson TX2[22]. Apart from this, they also integrated VIO
pipeline with Bundle Adjustment on EuROC dataset which resulted into 200 FPS.
FAST feature detector along with optical ﬂow tracking and 2-Point RANSAC based outlier methods
looks very promising to implement VO pipeline in real-time while incorporating kalman ﬁlter to fuse data
from IMU, GPS, etc. The author recommends to explore in this direction for future improvements.

C.1
Quick Sort for Bucketing
Algorithm 5: Quick sort algorithm used in Bucketing
def partition(keypoint, descriptor, low, high)
pivot keypoint ←keypoint[high]
i ←low - 1
for j ←low and j < high -1
if keypoint[j] > pivot keypoint
swap keypoint[i] and keypoint[j]
swap descriptor[i] and descriptor[j]
end if
end for
swap keypoint[i+1] and keypoint[high]
swap descriptor[i+1] and descriptor[i+1]
return (i + 1)
end def
def quicksort(keypoint, descriptor, low, high)
if low < high
pi = partition(keypoint, descriptor, low, high)
quicksort(keypoint, descriptor, low, pi-1)
quicksort(keypoint, descriptor, pi+1, high)
end if
end def

The recorded sequences are 8-bit grayscale images and recorded at 10 Hz. Although KITTI dataset
oﬀers raw data, the author used an undistorted and rectiﬁed image to keep the visual odometry pipeline
simple.
KITTI dataset provides accurate ground truth estimation for sequence 00 to sequence 10. The ground
truth estimation is synchronized with the stereo camera. It is useful to calculate RMSE to evaluate the
