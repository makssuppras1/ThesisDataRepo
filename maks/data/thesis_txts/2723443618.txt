Abstract
This thesis proposes a novel approach for implementing join-patterns in Scala
3 using meta-programming, without any modiﬁcations to the compiler or new
language constructs. The formal syntax and semantics of join-patterns are pre-
sented, along with two matching algorithms: a brute-force (stateless) algorithm
and a tree-based (stateful) algorithm. The implementation is demonstrated to
support the Actor Model, allowing for modeling of real-life systems with multi-
message communication requirements. Formal semantics are used to verify the
correctness of the proposed solution. A performance evaluation is conducted to
compare the two matching algorithm implementations and identify their limita-
tions. Finally, we lay the groundwork for future work by proposing potential
optimisations.

Introduction
Concurrent systems refer to computer systems that can perform multiple tasks
simultaneously. These systems are designed to take advantage of multi-core pro-
cessors and distributed computing environments to enhance performance and
scalability. Concurrent systems rely on the principles of concurrency, parallelism,
and synchronization to ensure that different parts of a program execute concur-
rently without interfering with each other. Such systems are often implemented
as distributed systems, where different components of the system run on sepa-
rate machines connected over a network.
One common approach to building such systems is to use message-passing,
where independent processes communicate with each other through message
passing. In such systems, each process operates independently, with no knowl-
edge of the state of other processes.
In the context of distributed systems,
message-passing plays a critical role in enabling communication between com-
ponents running on different machines. The message-passing paradigm allows
components to communicate by exchanging messages that are sent and received
asynchronously over the network, enabling loosely coupled and fault-tolerant
systems. A popular implementation of the message-passing paradigm is the
Actor Model [1].
1.1
The Actor Model
The Actor Model [1] is a concurrency model of computation, where actors are
independent, encapsulated units of computation that communicate through mes-
sage passing. In the context of distributed systems, the actor model provides a
programming paradigm for building scalable and fault-tolerant distributed ap-
plications, as it allows for seamless distribution and concurrency of actors across
multiple nodes in a network. Each actor can process messages independently,
and failures in one actor do not affect the rest of the system. However, the actor
model is not without its limitations. In particular, the single message matching
model of the model is not expressive enough to model complex interactions be-
tween actors. This is because the single message matching model only allows
for a single message to be matched at a time. In this thesis, we will explore how
to overcome the limitations of the single message matching model, by using
join-patterns.
1.2
Join Patterns
Join Patterns were invented as part of the Join Calculus [2, 3] process calculus1.
They were introduced as part of a thread-based concurrency model that was
implemented as a language extension to the C# programming language, called

Polyphonic C# [3]. The join-patterns were used to model asynchronous concur-
rency constructs in C#. However, the technique of using join-patterns to model
concurrency was popularised in the implementation of the Scala Joins library [4],
which demonstrated how join-patterns could be used to model asynchronous
communication using the actor model. In Chapter 2, we will present some of
the implementations of join-patterns in other programming languages.
Furthermore, we present algorithms that can be used to match join-patterns.
This is utilised by the Actor framework such that we are able to support both
single message and composite message matching. The matching algorithms pre-
sented in this thesis are a simple brute-force algorithm and a tree-based algo-
rithm. The brute-force algorithm matches join-patterns against a sequence of
messages. However, it does not maintain any state between updates to the mes-
sage sequence, which means the algorithm will keep matching against a pattern
even though the messages we have so far are irrelevant for a given join-pattern.
The tree-based algorithm, aims to improve upon the brute-force algorithm by
maintaining state between updates to the message sequence, such that we main-
tain a tree structure that contains the information about where the messages
we have seen so far match a given join-pattern. The tree-based algorithm we
present is based on intuition and does not yet have a formal proof of correctness.
Formalisation and proof of correctness of the tree-based algorithm is a work in
progress in [5], and will be addressed in future work. State maintaining match-
ing algorithms have existed in other domains, and have been used by other
implementations of join-patterns to improve the performance of the matching
algorithms. There are two notable algorithms from the domain of expert sys-
tems: the RETE and TREAT algorithms [6, 7]. However, we have only informal
description of these algorithms, and no formal foundations.
1.3
Objectives
We aim to fulﬁll the following objectives for our implementation of join-patterns
and the two matching algorithms that we are considering:
• Correctness: The implementation of the join-patterns and the two algo-
rithms should be correct, according to the formal semantics of the join-
patterns introduced in [5].
• Expressiveness: The implementation of the join-patterns should be expres-
sive enough to model complex interactions between actors.
• Relative Performance: We expect the tree-based algorithm to be more efﬁ-
cient in terms of computation compared to the brute-force algorithm, but
less efﬁcient in terms of memory consumption.

Related Work
In this section we present a brief overview of notable related works that imple-
ment the join-patterns concurrency model.
• JoCaml [9] is a dialect of the OCaml programming language that im-
plements the join-calculus concurrency model.
JoCaml extends OCaml
with new language constructs to express the concepts of the join-calculus,
such as join patterns, which specify the synchronization constraints be-
tween processes. The implementation runs using threads, however, it uses
message-passing mechanisms for synchronisation instead of shared mem-
ory.
• JErlang [10] is an extension of the Erlang programming language. Erlang is
a functional language that uses the Actor model for concurrency and sup-
ports the message-passing paradigm through pattern-matching. However,
coding synchronization between multiple processes is not simple in Erlang.
To address this issue, the authors have designed and implemented JErlang,
an extension to Erlang inspired by the Join-Calculus concurrency model.
JErlang offers a rich set of language features with joins, and two alterna-
tive solutions are presented - a library and an altered VM. The authors in
[10] also discuss their optimisation efforts, in which they have adapted the
RETE [6] algorithm and it have resulted in good performance for JErlang.
• Sparrow [11] is a join-pattern language that aims to extend the traditional
single-message match interface of actors to support multiple-message match,
heavily inspired by Complex Event Processing (CEP) languages and frame-
works. Sparrow is implemented as a domain-speciﬁc language (DSL) using
Elixir’s macro facilities and beneﬁts from its integration with the extensive
set of Elixir/Erlang libraries.
• Scala Joins [4] is a library for that implements the join-patterns in Scala
2. It presents the Actor model as a potential use case for the library. The
library is implemented using Extensible Pattern Matching. Our work is
inspired by this library, but we use metaprogramming facilities in Scala 3
to implement the join-patterns.
• Scala 3 Join-patterns [8], this library is the basis of our work. It implements
the join-patterns in Scala 3 using metaprogramming. The library also im-
plements a matching algorithm for join-patterns matching. However, the
algorithm does not comply with the join-pattern semantics that we present
in this thesis. Our work is updates the library, where we demonstrate a
correct algorithm for join-patterns matching and we also present a new
algorithm that aim to improve the performance of the matching algorithm.

Background
In this section, we provide an overview of the necessary background knowledge
needed to understand the subsequent chapters of this thesis. We explain the
manner in which meta-programming is supported in the Scala 3 programming
language and its signiﬁcance in the implementation of join-patterns, which are
implemented using the macros and reﬂection which are meta-programming con-
structs provided by the programming language. Furthermore, we give a brief
introduction to the code, which is divided into code generation and matching
algorithms.
3.1
Meta-Programming in Scala 31
Metaprogramming is a programming approach where a program operates on it-
self or takes another program as input and generates an output program. It has
the potential to manipulate a program at compile-time or run-time. There exists
several mechanisms for metaprogramming, such as template meta-programming,
macros and reﬂection [12]. In our context, we will be focusing on macros and
reﬂection, which are the two main mechanisms for metaprogramming in Scala
3.
3.1.1
Macros
Macros are program fragments which are used to specify an input sequence pro-
vided by a programmer, which is then mapped to the desired output sequence.
This is called macro expansion [13]. Macros allow us to analyze, modify and gen-
erate programs at compile time. In Scala 3, macros operate on expressions of
type Expr[T], where T is any valid type in concrete Scala. Furthermore, the lan-
guage provides two central constructs that ease the use macros; Quoting with
concrete syntax '{ ... } and Splicing with concrete syntax ${ ... } [14]. The
API is deﬁned in the package scala.quoted, which provides the Expr type.
Regular Scala code written within quotes will be translated into an expression
of type Expr[T] and this is the Abstract Syntax Tree (AST) representation of the
code and its execution is delayed. On the contrary, spliced code i.e. code written
within a splice is evaluated and inserted within the surrounding code [12, 14].
For example, consider the following Scala code:

4.2
Matching Algorithms
In the previous section, we presented the syntax and semantics of the join-
patterns. In this section, we present a comprehensive overview of the design
of the matching algorithms used for pattern matching. We have implemented
two matching algorithms: the brute force matching algorithm and a matching
tree algorithm that maintains state across the matching cycles. The brute-force
matching algorithm does not maintain any state between matching cycles, thus
it matches all messages in the queue against the pattern vector, every time a new
message is added to the queue. The matching tree algorithm maintains a state

==PAGE:19==
between matching cycles, this is if the current conﬁguration of the messages in
the queue partially matches a pattern, then we record the partial match and use
it in the next matching by only inspecting the newest message.
All of the algorithms are implemented as blocking algorithms, which halt pro-
gram execution until the matching is completed. They work on a thread-safequeue
object for receiving messages and a pattern-vector.
4.2.1
Brute Force Matching Algorithm
The brute force matching algorithm is a simple matching algorithm. It maintains
a list of messages taken from the queue and iterates over the pattern vector to
ﬁnd a match. A high-level description of the algorithm is as follows:
• A simple matching algorithm that maintains a list of messages taken from
the queue
• Iterates over the pattern vector to ﬁnd a match
• If a candidate match is found, based on the Match and Candidate rules
from Section 4.1.2, then we add the index of the messages that matched
the pattern i.e. I and the pattern index i.e. the ith position of the pattern
vector A into an internal record that maintains all the patterns that can be
matched or activated by the current conﬁguration (Q | A). Furthermore,
we also record the substitutions σ that may be used to match the pattern
in a map data-structure, that maps the variables that occur in the pattern
to the concrete values of the matched messages.
• Once the algorithm has recorded all the patterns that can be matched per
our semantics, the following steps are performed:
– If the record is non empty, then we lexicographically sort the record
of patterns that can be matched on the on Is.
– We then select the ﬁrst match from the sorted list and execute the
right-hand side of the pattern and return its value. This is following
the Selected rule from Section 4.1.2.
– Finally, we remove the messages that matched the pattern from the
queue as per the Successful Match reduction rule from Section 4.1.2.
• If no match is found i.e. the internal record is empty, then algorithm will
pull a new message from the queue and repeat the process.
4.2.2
Matching Tree Algorithm
In this section, we present the design of the tree-based algorithm. The objective
of this algorithm is to maintain partial matches across matching cycles i.e. the
algorithm will only inspect the newest messages in the queue and use the partial
matches from the previous matching cycles to ﬁnd a match. This is opposed to
the brute force matching algorithm that matches all messages in the queue every
match cycle. In the following subsections, we will provide the intuition behind
the algorithm and then present a high-level description of it.

The Algorithm
In this section, we will provide a high-level description of the algorithm. The
algorithm performs the following steps:
• First, it creates an empty matching tree for all patterns in A. So that initially
there are no partial matches. Furthermore, the queue Q is empty initially.
• Now, a new message m is received.
• Then, it iterates over the pattern vector for each pattern and performs the
following steps:
– We check if m matches any pattern in the join-pattern π. If it does,
we update the associated matching tree of the pattern with the data
of the message m.
– Then, we check if the matching tree of the pattern has any candidate
matches. That is if we have received enough messages to match the
pattern. If we do, then we check if the candidate matches satisfy the
predicate of the pattern using the Candidate rule, then we add the
candidate matches to an internal map of candidate matches.
– Otherwise, if we do not have enough messages to match the current
pattern, then we continue to the next pattern.
• Finally, we check if the matching tree of the pattern has any candidate
matches. Using the Selected rule from Section 4.1.2 we select the ﬁrst can-
didate match and execute the right-hand side of the pattern. Additionally,
we delete the matches messages from the list of seen messages and prune
the matching tree of the pattern by removing the nodes that contain the in-
dexes of the matched messages. Otherwise, we wait for the next message
to arrive, and repeat the steps above.
We note that this algorithm is based on intuition and is not formally proven, yet.
The formalization of the algorithm is early work in progress in the [5], and will
be addressed in future work. However, the idea behind the algorithm is that it
will exhibit some mathematically provable properties that will allow us to prove
the correctness of the algorithm.

gorithm. It is implemented as a class that extends the Matcher[M, T] trait. It
exhibits the same blocking behaviour as the brute-force matching algorithm de-

state management to the algorithm. As described in Section 4.2.2, the algorithm
uses a tree data-structure to store partial matches that are maintained accross
matching cycles. We have implemented two versions of the algorithm that dif-
fer in the way they manage the state of the matching tree.
We encountered
some issues with the ﬁrst version, and we have implemented a second version
that addresses these issues. We will desceribe the algorithms in the following
subsections.

Conclusion
Now we will evaluate the results of our tests and benchmarks with regards to
the objectives we set out to achieve in Section 1.3.
• Correctness: We have demonstrated through empirical evaluation and ex-
tensive unit tests as seen in Section 6.1 that our join-patterns implementa-
tion with two matching algorithms are correct based on the semantics that
we presented in Section 4.1.2. Thus, we can conclude that our objective of
correctness has been achieved.
• Expressiveness: Scenarios such as the Smart House and the Santa Claus
exhibit complex interactions and coordination between actors. However,
we are able to model these using our join-patterns implementation in a
simple and declarative manner, by only using the standard Scala pattern-
matching syntax.
We have achieved this by leveraging the Scala meta-
programming capabilities, while still maintaining a simple and declarative
syntax. Thus, we can conclude that our objective of expressiveness has been
achieved.
• Relative Performance: This objective has not been totally met, but we will
discuss several optimisations that can substantially improve the current
implementation with limited effort in Section 7.2. Based on the results
of the benchmarks, we conclude that the our current implementation of
the tree-based matching algorithm is not memory efﬁcient. However, the
brute-force matching algorithm is more efﬁcient than the tree-based match-
ing algorithm, when matching singleton patterns. However, we were not
able to produce a benchmark for the tree-based matching algorithm, that
matches composite patterns, as the memory usage of the tree-based match-
ing algorithm was too high to produce results with comparable benchmark
conﬁguration as the brute-force matching algorithm.

7.1
Contribution
In this thesis, we have presented the design and implementation of join-patterns
in the Scala 3 programming language. We have proposed a novel approach to
implementing join-patterns that leverages the existing pattern matching infras-
tructure using meta-programming, without any changes to the Scala 3 compiler
or the introduction of new language constructs.
To support our implementation, we have presented the formal syntax and se-
mantics for join-patterns, and demonstrated the correctness of our implementa-
tion through extensive testing. We have also demonstrated the expressiveness of
our implementation by showing how it can be used to express complex patterns
using standard Scala 3 language constructs.
In addition to these contributions, we have presented two matching algorithms
for join-patterns: a brute-force algorithm without state management, and a tree-
based algorithm with state management. Although our benchmarks showed that
the tree-based algorithm did not provide the performance gains we expected, we
have laid the groundwork for future optimizations.
We have also shown how our implementation can be utilized by the Actor Model
to model real-life systems that require multi-message communication between
actors, which is not supported by mainstream Actor Model implementations
such as Akka.
7.2
Future Work
There are several avenues for future work that could build on our ﬁndings.
The ﬁrst avenue is to explore way to lessen the memory-consumption of the
tree-based algorithm.
We have already identiﬁed the source of the memory-
consumption, which is the generation of permutations of possible matches, which
now for composite patterns of size n, computes n! permutations of the possible
ﬁts. We could explore ways to compute the permutations lazily and in a lexico-
graphically sorted order, which would then allow us to only compute the ﬁrst
element of the permutation, and obtain the optimal match.
Another avenue for future work is to explore ways to improve the expressiveness
of declaring the join-patterns. Currently, we are using tuple syntax to express
composite patterns, however, this is not close to the formal syntax we presented,
which uses & to express composite patterns. This would be similar to the syntax
of the Scala Joins implementation [4].
Finally, it is worth noting that the tree-base algorithm described in this work
is currently based on intuition and has not yet been formally proven. Formal-
ization of the algorithm is currently in its early stages in [5]. Despite this, the
