Abstract
In this thesis an investigation of creating a didactic compiler targeting RISC-V in Go is
made. First the background of the host language, RISC-V, and the simulation tool for
RISC-V is presented. This also shows the reasons for choosing RISC-V and Go. Then
the design phases of a compiler and the source language for it is presented.
The implementation of the compiler has been done in incremental steps which is further
described in the implementation chapter. The features implemented include basic arith-
metic operators, single argument functions, composite data types etc.
While the compiler does not support closures, the rest of the project has proven to be a
success.

Introduction
Compilers are often looked upon as mystical beast created and maintained by wizards.
This view is understandable if one tries to look upon one of the industrial grade compiler
such as GCC which have more than 14.5 million lines of code[1]. Although, one will
be suprised how little code is actual needed to create a working compiler for a minimal
language.
In this project, a minimal working compiler for a made up source language that support
modern concepts such as first-class function and closures that compiles down to RISC-V
assembly. The compiler’s size1 is a total of 6107 lines of code, where 3278 of the lines
are test code. That means a working compiler can be written in 2829 lines of code which
vastly different from 14.5 million lines of code of GCC. Although its a working compiler at
2829 lines of code, there is no optimising phase. Therefore, the code generate performs
poorly. One of the reasons that GCC and other compilers of the like are so big. They
have many optimising phases to produce the most optimal output.
This thesis will guide the reader towards this compiler where the implementation will be
in steps. Each step implements a language feature. Further, each step will always keep
the compiler in a working state, which means all tests will pass in each step.
In chapter 2 the reader can get the sense of the host language of the compiler, what is
RISC-V, and how is the compiled code executed.
In chapter 3 the reader can read the overall design of a compiler and source language
that the compiler compiles.
In chapter 4 the implementation of the compiler is described in steps. Each step imple-
ments a language feature.
In chapter 5 future work is desribed. The chapter denote what can be improved with the
compiler and missing features that was not implemented due to time limitations.
1.1
Target Audience
Before reading it is wise to set the stage of the knowledge the reader must have, as this
thesis goes into subjects that might be uncommon for a software engineer.
The assumptions for the reader is to have some basic knowledge about the front end of
compilers: Lexical analysis, Syntactic analysis, and Semantic analysis as well as limited
knowledge about assembly. Although it is assumed that the reader knows what the stack
and heap is.

Background
In this chapter the host language of the compiler is described, what RISC-V is, and how
the emitted RISC-V assembly has been tested.
2.1
Host language
The host or implementation language of the compiler is Go[2]. Go is a minimalist lan-
guage, which makes it easier to grasp and thus easier to follow the steps of implementing
a compiler. The language’s syntax is modern but programmers that know the C syntax
will understand it quite easily.
The interesting note is that the source language, which is to be implemented, have a
similar syntax of Go. An advantage of this is that one does not have to think about two
completely different languages at the same time.
2.2
RISC-V
RISC-V [3] is a free and open Instruction Set Architecture(ISA), that thrives on open stan-
dard collaboration, which drives its innovation.
The RISC-V is designed to be customised and specialised[4] making it possible for com-
panies to extend the ISA to suit their needs. The standard ISA is in itself, made up of ex-
tensions provided by the RISC-V organisation. These are called the standard extensions.
RISC-V uses a naming convention to refer to these standard extensions. An example is
I which refers to the base integer ISA extension. This is prefixed to RV32|64 which de-
scribes the user-level address spaces of 32-bit or 64-bit. The only mandatory extension
for all RISC-V implementations is I.
2.2.1
Why RISC-V
RISC-V have gained a lot of attention in the recent years and is said to be the next revo-
lution in the Open Hardware Movement[5]. It provides open or free hardware designers
with an open CPU architecture. Thus, the future would probably have many more devices
with an RISC-V processor within. This makes it interesting to target that specific archi-
tecture. Furthermore, the RISC-V is a RISC architecture, later described in section 2.2.2,
there is only a handful of instructions to grasp, making it easier comprehend the assembly
language.
2.2.2
RISC vs CISC
Complex Instruction Set Computer (CISC) is an architecture that emphasis on making the
hardware doing as much of of the work instead of the programmer writing in assembly.
The architecture uses variable length instruction format, where a simple instruction might
use 2 bytes, and a complex instruction could use 5 or more bytes. There is no limit on
how many bytes a instruction might use, therefore a instruction can be very complex.
The benefit of complex instructions is that they can perform several actions and from
the programmer’s view operate directly with memory; the instruction hides the loading of
values into registers. These complex instructions might take up several clock cycles to

==PAGE:12==
perform but the size of the code is smaller, therefore less requirement for memory storage.
To allow these complex instructions the CPU must use more transistors to store complex
instructions leaving less transistors for general purpose registers.
Reduced Instruction Set Computer (RISC) is an architecture that emphasis making the
software doing more of the work instead of the hardware. Where CISC can work with
memory RISC only work with registers. Therefore, doing any computation would require
more lines of instruction to perform the same work in a CISC architecture as loading and
storing values need their own instructions. This is also why RISC is known as a load/store
architecture. Although there are more instructions, RISC instructions only require one
clock cycle to execute due to instructions being fixed in length - 4 bytes each. So even
there are more instructions they both approximately execute in the same time. And the
benefit that each instruction only takes one clock cycle, is that pipelining is much easier
to perform which results in faster programs. As the instructions are more simple, they use
less transistors, so there are more general purpose registers to work with.

RISC-V Assembler and Runtime Simulator - RARS
RISC-V processors are not widely spread yet as a consumer processor and still a more
niche processor with some development boards[8]. One could buy one of these develop-
ment boards but that is an investment and might require one to know how to work with an
embedded board. Therefore, a simple approach is to simulate RISC-V processor with the
RARS tool[9] which was built on MIPS Assembler and Runtime Simulator (MARS)[10].
RARS is a light weight IDE for developing RISC-V assembly which can be seen in fig. 2.1
and can assemble and simulate a RISC-V processor seen in fig. 2.2. In simulation one
can run the program or debug by execute instructions step-by-step. The usefulness of
debugging is that one can see registers and memory locations change values as the pro-
gram progresses.

Design
In this chapter the overall design of the source language for the compiler and the ab-
stract design of compiler is described. Many smaller design details of the compiler will be
mentioned in chapter 4 as they occurred while implementing the compiler.
3.1
Source language
In this section, the source language for the compiler is described.
Designing a new language is not a trivial task, as there are many details that need to
be outlined and thought through to ensure that the language does not introduces am-
biguous behavior. Details such as what is the behavior of declared variables without
assigned value? How are variables resolved, lexical (static) or dynamic? If the language
is typed, how many bits makes up each type? Is a type mutable or immutable? What
types are comparable? And so forth. As the language grows in features so does the lan-
guage specification, and an example of this is Java SE 17, that has a 848 page language
specification[11]. The Go language specification is a around 50 pages[12][13] and that is
considered a minimal language.
As this thesis is about creating a didactic compiler and not about language design, much
of the design and behaviour is taken from Go Specification. The language could be con-
sidered as Go--.
3.1.1
Notable differences from Go Specification
While there are a lot of similarities with Go, there are following differences:
• There is no need for a main function - the file is executed from top to bottom like in
a scripting language.
• Print is a statement and not a builtin function.
• There is no type inference.
• No variable overshadowing.
• Functions must be declared before usage. Function prototypes are allowed like in
C language.
• No string concatenation with the ”+” operator.
• Arithmetic on number literals is only allowed if they are the same type, e.g. 2.0 (float)
+ 1 (int) is illegal.
• Numeric types are only in 64 bit.
• Boolean type is 64 bit.
• String type is a pointer to the string that represent it. The pointer is 64 bit, while the
string can have variable length.
• The source file can only be in ASCII.

3.1.2.1
Lexical Grammar
The lexical grammar defines the syntax of tokens. These tokens fall into these categories:
• Keywords - also known as reserved words - these are tokens that can not be used
for naming variables, structures, and functions.
• Identifiers - these are names for variables, structures, and functions that the pro-
grammer provide.
• Numbers - as the name suggest hold numbers such as integers, floating point num-
bers.
• Strings - a set of characters that is neither a identifier or keyword. These are used
for literal text within the program.

3.2
Compiler
A compiler translates a source language to a target language. Normally, the source lan-
guage is a high level language and the target language is a low level language. If the
source and target language are of same abstraction level (both considered high level lan-
guages), one would call it a source-to-source compiler also known as a trans-compiler.
A compiler is a collection of components working together to accomplish the task[14].
The basic components are Lexical analysis, Syntax analysis, Semantic analysis, and
Code generation, also known as machine code generation. Other components could be
Intermediate code generation, Register allocation (typically, used for a target language
that is machine language), and Assembly and Linking. The Intermediate code genera-
tion normally also includes optimisations to improve the Intermediate Representation (IR).
These components can then be further grouped into what is called the Front-, Middle-,
and Back-end of the compiler. The Front-end is Lexical, Syntax, and Semantic analysis.
Middle-end is Intermediate code generation. Back-end is Register allocation, Machine
code generation, and Assembly and Linking.
To keep the scope of this compiler more simple and approachable the Intermediate code
generation and the Assembly and Linking is not included. Therefore, the Abstract Syntax
Tree (AST) produced in the Syntax analysis, will be the IR that the compiler works with.
The AST will be annotated by the Semantic analysis and Code generation phase with
information needed to complete the compilation. The Register allocation is combined into
the Code generation so it is done in one setting.
The target language of the compiler is RISC-V assembly.

3.2.1
Lexical analysis
Lexical analysis also known as lexing or tokenisation is the process of breaking a se-
quence of characters, the source file, into a sequence of tokens[15][16]. A token is an
object that have an assigned meaning that makes it identifiable. The meaning of a token
can be its type, such as identifier, semicolon, and etc. Furthermore, tokens could record
there position (row and column) in the source file to enable better error reporting to point
where did it fail in the source file.
The design of the lexer, follows that described in the book Writing an Interpreter in Go[17],
and by looking at other lexers such as the one for Go[18] and for Wren[19][20] they all
follow a similar pattern. In particular, this pattern is to read a character at a time and take
actions upon the character and produce a token.
3.2.2
Syntax analysis
Syntax analysis also known as parsing is to take the tokens reform them into a data
structure either a Syntax Tree or Abstract Syntax Tree (AST). The AST is normally the
16
A Didactic Compiler Targeting RISC-V
==PAGE:25==
data structure to use, as it removes unnecessary informations such as semicolons and
brackets, which are not needed for the later phases in the compilation. The form that the
tokens are put together is the order that is defined in the syntax grammar.
There are two ways which the AST can be built, either top-down or bottom-up. In this
thesis we only consider the top-down approach and in particular a parser that is based
on the recursive decent parser: Top Down Operator Precedence parser[21]. This is also
known as a Pratt parser named after its inventor Vaughan Pratt. In a traditional LL(k)
parser each grammar rule is associated with a parsing function. In Pratt parsing the
parsing functions, which is described in his paper as semantic code, are associated with
single token types. A token type can then have multiple parsing functions attached to it
depending on the position of the token; prefix or infix. In the paper the parsing functions
attached to tokens in prefix position are called null denotations (NUD)s as there is no
expression left of the token. And those in infix are called left denotations (LED)s since
there is a expression left of the token.
The algorithm for the Pratt parser is as follow:

the precedence of operators comes into play. One can interpret the right_binding_power
as how many tokens right of the current expression is going to be bound to it and
left_binding_power how many tokens to the left of current expression will be bound to
it. These in conjunction determines determines if an expression 2 + 2 * 3 should be
evaluated as (2 + (2 * 3)) or ((2 + 2) * 3) depending on whether the precedence
of the ”+” is greater than ”*” or not.
In [17] shows how such a parser is implemented and therefore, the parser for this thesis
is modelled after it. This will be shown in chapter 4.
3.2.3
Semantic analysis
As the Syntax analysis is about the structure of a program, the Semantic analysis is about
the meaning of a program. The meaning of a program is what operations that are valid
and not valid. These can not be expressed in the syntax grammar so these are defined
outside the grammar in the language specification.
The semantic analysis consists of several components such as Type Checking, Name
Resolution, and Flow Control Check. The Type checking is about checking whether an
operation on a type is valid and also assigning proper types to expressions. The name
resolution checks that variables are declared before usage and populate the symbol table
with these variables and their scope. The Flow Control Check checks that statements
that can alter the flow is valid, such as return statement is disallow out of function scope.
The design follows much the suggestion from the book Introduction to Compilers and
Language Design[15] in the Semantic Analysis section.

Symbol Table
The symbol table is designed as an imperative symbol table, and imple-
mented as a stack. The stack behaviour suits the scoping mechanism since entering a
scope is to push an new symbol table on the stack.

4.1
Process
The process of writing a compiler is like writing any other piece of software. Start with a
small feature and incrementally adding more features. For compilers it is even better to try
making the process follow a Test Driven Development (TDD) practice. The TDD practice
makes one think about the shape and behaviour of the feature by writing the test first and
then the implementation. These test will serve as a form for documentation of a feature
as they describe what is expected for a feature. So if a test break then some part of the
code changed the behaviour of a feature either by mistake or intentionally. If intentionally,
then the test need to reflect the new design of the feature and again the documentation
of the feature is correct. Otherwise, one finds the error in the code that introduced the
failure.
For the brevity sake, this chapter does not include the tests for a feature. As mentioned
in the chapter 1, there is actual more test than implementation code. Furthermore, com-
ments and variables only needed for testing is also removed. This should not effect the
understanding of the code.

The AST, as any tree data structure, defines the what a node represent. Node is defined as
an interface with the methods TokenLiteral and String. TokenLiteral is mainly used
for debugging and testing purposes, to easily get the token literal of a node. String con-
structs a textual representation of a node which helps a lot when debugging. Statement
interface inherits the properties of a Node and have a dummy method statementNode
which purpose is only to make the Go compiler error when a statement is used as an ex-
pression. Expression interface also inherits from Node and as Statement also have the
dummy method expressionNode to disallow an expression used in a statement context.
Furthermore, it defines two methods Register and Type, where the former returns the
register name assigned to expression and the latter the type of the expression.
These interfaces are then implemented by the concrete types: Program, PrintStatement,
IntegerLiteral, FloatLiteral, StringLiteral, and BoolLiteral. As can be seen they
hold the same syntax structure as defined in the grammar, with the exceptions that those
being expressions have additional fields for a register and type information. The Program
node serves as the root of the AST that parser builts.

parser/parser.go
As mentioned in section 3.2.2 the parser is modeled after the Pratt parser. We define the
types prefixParseFunc and infixParseFunc to describe the signature for functions that
need to parse prefix expressions and infix expressions. The prefixParseFunc simply
matches functions that returns an expression. The InfixExpression matches functions
that takes an expression as argument and returns an expression. The argument expres-
sion is the left hand side of an infix expression, so taking this infix expression: 2 + 3,
the 2 is the left expression that would be the argument. The New function is where the
token type is associated with a parsing function. For now it only the register the tokens
token.Int, token.Float, token.String, token.True, and token.False as prefix tokens.
The function returns a new Parser that is primed and ready to use. The ParseProgram
method starts the parsing and constructs the AST until the end of the file is reached. It
calls the parseStatement method that will further call the correct parsing method for the
statements we defined in the grammar. The parsePrintStatement construct the AST
node for a print statement and here we see the parseExpression method that starts the
heart of the Pratt parser. It takes a precedence which can be thought of as the prece-
34
A Didactic Compiler Targeting RISC-V
==PAGE:43==
dence level of the current token type and returns the AST node for the expression. The
body of parseExpression first grabs the prefix parsing function for the current token type,
such as parseIntegerLiteral, and if that does not exist then it will fail. Otherwise we
call that function and grab what it produces. If we had any infix expressions this value we
just grabbed would be the left expression of the infix expression.
Why did we pass the Lowest precedence level to parseExpression in parsePrintStatement?
The simple reason is that from that point, we have not parsed any expression beforehand
and therefore, nothing to compare the precedence with. In fact, every statement that
parses an expression will always pass the Lowest. When does the parseExpression get
a higher precedence level? This will be shown when we implement infix expression in
section 4.3 where we also implement more precedence levels.

The Compiler holds 2 slices of strings, one for the constant values and one for code (as-
sembly instructions). We have the constant and code separated since they need to go into
different segments in the assembly output; constants in data segment and code in the text
segment. Furthermore, it has a table of registers that is available when requesting a regis-
ter, and label which will be used to generate unique assembly labels. The code generation
starts when the method Compile is called. And like the type checker it uses recursion.
When we encounter a *ast.PrintStatement we compile the value expression of the print
statement to generate the assembly instructions. We then check the type of the value
expression. We do this as the assembly code generated for a print is type dependent.
Thereafter, we fetch the register from the compiled value expression and with the type
information we supply as argument to the print method. This method emit instructions
to setup the a0 and a7 registers with the information needed for the ecall. So if we con-
sider print 2, a0 would hold the value 2 and a7 would hold the value 1, where 1 is value
for instructing ecall to print integer values. The values in the registers are determined by
RARS[22]. After the print instructions are emitted, we deallocate the register that the com-
piled value expression had been given. The literal expressions *ast.IntegerLiteral,
*ast.FloatLiteral, *ast.StringLiteral, and *ast.BoolLiteral all follow the proto-
col of allocating a new register and emit instructions to load the value that the literal rep-
resent into the register. The *ast.FloatLiteral and *ast.StringLiteral have a bit
work as there is no RISC-V assembly instruction that takes a literal float or string value.
These need to be created as assembly labels and stored into the data segment. Then we
can load the value of these labels from their address point. The createASMLabelLiteral
method constructs the assembly labels with the correct data type for the literal.
The method Asm combines all the generated assembly instructions in the correct format.
The special labels __start and __end are used to mark the start and the end of the code.
These will be important when we are going to construct functions in section 4.7 and will
be further discussed there. Right after __end label, we create a exit environment call, so
our program terminate with a exit code of 0.

Code generation for var statements is to get the symbol of the variable and create the
assembly label in the data segment. We create the assembly label with the

for an assign statement to handle the value.

in the data segment as labels and a local variables will live on the stack. The latter will

value into that address. That is the reason we have the loadGlobalOrPtrValue method

In createASMLabelIdentifier method we create a string as a number with the assembly
directive .dword and not as a string with .string directive. We do this so we treat vari-
ables of string type as pointers to an actual string that never changes. The reason is that
data in the data segment is laid out continuously, which means that if we e.g. declare a
string and then a int variable, they would be side by side. So if we modified a string to be
longer than it was before we might overwrite any data that was stored after the unmodi-
fied string. Therefore, we treat string variables as pointers they can simply point to a new
string in memory. This means that a lot of strings will be created in the data segment but
for this project it is fine.

symbol/table.go
The new scope is LocalScope. This scope is used whenever we are defining a variable
inside a block. That is shown in Define method where it gives local scope when the sym-
bol table has an outer symbol table. Remember that the symbol table was implemented
as a stack, so if the outer table is not nil then we entered a new local scope. We add a
new scope with the NewEnclosedTable function.
Besides adding a new scope, we also include new fields to the Symbol and Table structs.
These fields are going to help us when we are generating assembly code. As assembly
does not have scopes we are simulating scopes by using the stack. Therefore, we add
fields in the Symbol struct that can describe where the variable is on the stack and an
offset if the variable is from a previous scope. And the Table struct have a field that is
the amount of stack space needed for that block/scope. The stack space is calculated
by the ComputeStack method. As per the RISC-V calling convention[7] the stack space
allocated always need to be 16 byte aligned so we make sure of that at the end of the
method. Besides calculating the stack space the method also sets the stack point of
each variable in the symbol table. As Go maps have random iteration order the keys are
sorted by the order the variables were put into the table so we can assign the stack points
accordingly. Technically, there is no need to do that except that it is much easier to debug
if the order of variable definitions in the source language matches the stack points in the
target language.
There is two resolve methods: Resolve and resolve, note the lowercase name of the
second. The Resolve method is changed to call the resolve method which tries to resolve
a variable by first looking at the current scope and then walks down the stack of scope
until it is resolved. Furthermore, it sets the stack offset to a symbol if the symbol was not
from the current scope.

Now the question is where we want to store a variable of struct type in memory. We could

into the global scope or vice versa it would require many instructions; first to load the
addresses of the structs, load one of the struct’s fields by offsets, and store into the the
other struct’s fields by offsets. This would work, but it would be problematic when we
implement functions in section 4.7 that return structs. As we will see when a function
returns it clears up its stack, which will mean that the struct that was on that stack would

global struct. For that reason we will use the heap to hold a variable of struct type both
for global and local scope. Both variables will then be pointers into the heap space. Then
storing a local struct into a global and vice versa is simply to move pointers around.

allocation instructions.

is used to load a value struct or store a value into the struct. Therefore, we let the code
generation for *ast.AssignStatement case handle the offsetting into the address of the

compiler/compiler.go
For generating code for functions we need to take some consideration into the calling
convention. How are we going to pass in arguments and how results are returned back
to caller. Furthermore, we need to consider how we are going to represent a function
in assembly as assembly does not have a notion of functions as we know it from higher
level languages. RISC-V define their calling convention in [7] and we partially uses this
definition for our calling convention. We do not save the temporaries registers, which
means that if we use any temporary registers in the global scope they might be overridden
by a function. Therefore, any global scope register are using the saved registers so they
preserve their value. The way we will create functions in assembly is to make a label that
refers to the instructions that make up the function like so:
func-name:
func-prologue
func-body
func-epilogue
The func-prologue is instructions that will setup the function such as allocate enough
stack space to save the function argument and the return address. The func-epilogue
is then to load the return address back into a register, clean up the allocated stack space,
and return to caller. The func-body is body of the function we defined in our source
language and that we generate code for like any block of code.
In the *ast.FuncStatement case we can see all the above take place. In our prologue we
compute the stack space needed for the argument and the return address and store them
on the stack. Then we generate code for the body of the function. Lastly, we generate
code for the epilogue, where we load the return address into a register, clean up the
stack and return to caller. If the epilogue returns back to caller, why is a return statement
needed? The reason is that a return statement is used to return early from a function
or to return a value to caller. So generating code for a return statement, as seen in the
*ast.ReturnStatement case, is to load, if any, the return value into the return registers
(a0), and make a jump to a functions epilogue. Furthermore, it also generates code for
cleaning up any stack space that has not been deallocated as the jump instruction might
jump over instructions that deallocate space on the stack. We keep track of potential stack
space that needs deallocation with in the field stackSpace in the Compiler struct.
The code generation for a call expression is to emit instruction that load the argument into
a register and then the instruction to make the call which is to make a jump to the function
label. The call instructions not only makes the jump but it also saves the return address
into the ra register for us.

Future work
In this chapter missing features, improvements, and optimisation of the compiler will be
addressed.
5.1
Closures
As stated in the introduction, the source language should have modern language features
such as closures but this was unfortunate not implemented. Although the building blocks
to enable closures are in the language. These building blocks are structs and function
types. So if we consider how a closure could look like:
func adder() func(int) {
var x int = 0
func t(y int) {
x = x + y
print x
}
return t
}
var add func(int) = adder()
add(1)
This could be rewritten by the compiler to:
type closure struct { x int; f func(closure, int) }
func call(c closure, x int) {
c.f(c, x)
}
func adder(env closure, x int) {
env.x = env.x + x
print env.x
}
var c closure
c.x = 0
c.f = adder
call(c, 1)
Then we can use the code generation for struct and functions to make closures. Although
functions have been implemented to only take one argument, so we would need to change
that. Furthermore, we would need to change that a struct’s fields can allow for function
types.

5.2
Control flow features
In section 4.5 we implemented for-loops with the for-statement construct. Although, in
our grammar we defined that a for-statement may have optional init and next steps but
these are implemented as required steps. The changes we would need to do is change
the parser to allow skipping parsing the init and next steps. Then in the type checker and
the code generation needs some guards not to consider these steps.
Furthermore, we are missing the break and continue statements to have greater control
over a for-loop. Lets recap the template for a for-loop:
for init-expr; expr; next-expr {
init-expr
body-statements
top-label:
}
expr
beqz t0, done-label
body-statements
next-expr
b top-label
done-label:
Implementing a break statement should be straightforward as we do not need to change
the template for a for loop as the break statement should be a simple jump instruction
to the done-label of the for-loop. For continue statement we would need to change the
template as a continue should jump to the next-expr. So we would change our template
to this instead:
for init-expr; expr; next-expr {
init-expr
body-statements
top-label:
}
expr
beqz t0, done-label
body-statements
continue-label:
next-expr
b top-label
done-label:
Then generating code for a continue-statement would emit a jump instruction to the continue-label.
5.3
Relational and boolean operators
While we have implemented the relational operators: <, ==, and !=, we are still missing the
others: <=, >, and >=. The code generation for <= would require use of a new assembly
instruction. The other two on the other hand we do not as we could rewrite the >= to <=
and the > to < simply by switching the position of the operands.
We could also implement boolean && (AND) and || (OR) operators which would allow
more complex conditions in if-statements and for-statements.
5.4
Recursive Structures
The structs implemented can only be with fields of the types: int, float, bool, and string. It
would be interesting to allow structs to also have fields of struct type. That could then lead

5.5
Aggregated data types
Lists and maps were not implemented, but would be useful data structures for a user of
the source language.
5.6
Reuse string labels for string literals
An small optimisation to the generated code is to reuse string labels that have the same
content. So if multiple string literals in the source language have the exact same string,
then they all could refer to the same assembly label in the generated code instead.

6
Conclusion
In this thesis, an investigation in creating a didactic compiler targeting RISC-V from scratch
in Go was made. While designing and implementing a compiler, there are a lot of things
to consider. First, the host language and the target architecture had to be chosen; Go is a
minimal language and therefore easy to comprehend and RISC-V is an architecture with
a few instructions. Therefore, both of these provided a good background for implementing
a compiler. To keep the project simple, the design of the source language is based on
a subset of Go, with some notable differences e.g., in the source language there is no
need for a main function where Go requires this. The source language is executed from
top to bottom like a scripting language. It would have been too large of a task to create a
compiler for the full language of Go which is why they differ.
During the implementation, each language feature was implemented in incremental steps
where each step is built upon the previous. Each step can be compiled independently
so each feature can be tested in isolation. The tests were written beforehand, making
the process take after TDD. This ensured that each new feature did not break an already
existing feature.
The result of the implementation is a working compiler. The compiler supports opera-
tions such as printing, basic arithmetic operators, storing of program state in variables
and functions. Functions support a single argument to simplify the compiler. Further-
more, the compiler supports control flow with if- and for-statements. Lastly, the compiler
supports composite data types. While these provide a working compiler for a minimal
language, there are things that could be improved in future works. The one thing that was
not implemented was closures.
While there are some improvements that could be made to the compiler, this project has
generally been a success.
