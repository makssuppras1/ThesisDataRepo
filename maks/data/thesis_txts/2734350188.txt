Abstract
A build system is a model that describes how source code is transformed into executables.
A forward build system is one that automatically traces the dependencies of its targets,
eliminating the need to pre­specify them, while also providing the benefits of non­forward
build systems. However, keeping track of these dependencies is a complex task. Recent
forward build systems use native system calls or library interposition to track dependen­
cies, rendering them tied to specific operating systems and frameworks.
Nevertheless, there exist tools that, when executed within a self­contained virtual envi­
ronment, could address these limitations; however, the price of virtualisation is usually a
drop in performance. This leads to the fundamental question: Is it possible to isolate and
trace builds in a platform­independent and efficient manner?
This thesis investigates the potential of using different tools to trace the execution of build
commands. To ensure cross­compatibility across various operating systems, we create
a framework, which relies on common virtualisation methods. Lastly, this work demon­
strates the practicality of this approach by integrating it into a contemporary forward build
system.

Introduction
Build systems are tools that are responsible for transforming source code into executables
[1]; they help developers repeatedly compile source code into machine­readable content
without fail. The main benefit of build systems is that they reduce the cognitive load that
programmers have to bear when working on their code: they know they can rely on this
tool to always correctly and quickly run the commands necessary to put the executable
together. The larger and more complex the code base and environment, the more useful
they will be. However, build scripts are not easy to write, and build systems differ from
platform to platform. Initially, build systems were declarative; they required the user to
specify targets to build, their dependencies, and how to build those targets (what com­
mands to run). Some examples are Make [2], Bazel [3], Gradle [4] and Maven [5].
The declaration of steps depend on one another. Since it is necessary to specify the
inputs and outputs of each task, developers are also incurring the risk that the actual and
declared inputs/outputs go out of sync. For example, in Figure 1.1 one would need to
update the input of the task with output ”main.exe”, if they were to change the output of
either of the other two commands.
1.1
Forward build systems
To reiterate, the maintenance of build systems is proven to be a hard task [7], as such
any improvement that can be done to alleviate the pressure of software developers to
maintain build­related information is seen as a plus. One way to do this is to use forward
build systems [8]: users only need to provide the script with the commands to build their
targets, without the need to specify the inputs and outputs; see Figure 1.2a. The build
system will then try to figure out what dependencies the different build commands have
and will try not to rerun tasks which have not had their dependencies modified, just like
declarative build systems. However, there is less information the system can work with,
and more that is left to be inferred. Examples of such a build system are Memoize [9],
Fabricate [10], Rattle[11] and Riker [12].
To provide the same benefit as declarative ones, forward build systems rely on programs
that allow them to trace which files were read from or written to by build commands.
These are then recorded and used to map out a similar dependency structure to that of a
backward (declarative) build system’s, see Figure 1.2b.
As mentioned before, dependencies of declarative build systems can go out of sync. In
forward build systems, however, the ease of writing build scripts comes at the cost of
performance, as the build system under the hood has to do more work than its declarative
counterparts.

1.2
Platform dependency
The notion of dependency tracing is the backbone of forward build systems and often­
times the sole reason for hard platform dependency. For example, Memoize relies on a
program called strace [13], which helps to track system calls on Linux. In the case of Rat­
tle, this program is called fsatrace [14], which accomplishes this task through the means
of library interposition [15]; it injects code between dynamically linked libraries to trace
calls between them. While the former is an operating system­specific implementation,
the latter, by definition, will not work with statically linked libraries. Fabricate proposed a
solution to this problem by tracking file access times on Windows (similarly to what Make
is doing under the hood when it is deciding which commands to rerun) and trying to deter­
mine which files were written and read this way. While traditional build systems are ported
to most used operating systems, that is not the case for forward ones. In fact, Fabricate
is the only contemporary forward build system with some sort of platform independence
by design. One could argue that Rattle can technically be ported to Windows, as kernel
hooking is available in fsatrace with the help of which we are able to trace 64bit processes,
but that also comes with its own set of limitations [16].
Just this fact alone renders contemporary forward build systems undesirable for wide­
spread use. However a system, that were to introduce dependency tracing, isolated from
the underlying platform, that could also reproduce a build process regardless of what kind
of operating system it is being run on, whilst still having the benefits that forward build
systems have, would be desirable. As of today, there are no such tools, which raises the
question:
RQ: Is it possible to isolate and trace builds in a platform­independent and
efficient manner?
To which our answer is: it depends. Since our proposed solution is not the only way to
achieve the expected mechanism, we cannot safely say that it is not possible, however
as we will outline in this thesis there is a mild to significant performance drop when using
our approach to answer the question. If this slowdown is acceptable to the user, then
our answer is yes, if it is not then we argue that we have failed to answer the research
question. Our approach to solving this problem is as follows:
i To achieve platform independence, the solution builds upon common virtualization
techniques, such as Docker[17], which is available for the most common of operating
systems.
ii To be able to isolate and trace builds, it traces independent build commands, starts
the virtualisation procedure, transfers the files necessary to run the command and with
the help of the Fuse 3.0 API[18] it produces a trace log.
iii To show what the cost of using this approach is, we set up detailed benchmarks and
break down the performance implications granularly.

1.3
Contributions
In this Master’s Thesis, we propose a novel approach to tracing build dependencies. A
virtualised dependency tracing tool, Cairn, was built using the Fuse 3.0 API on the Docker
engine. After this introduction and a chapter on the theoretical background (chapter 2),
the structure of the thesis follows the following main contributions:
• In chapter 2, we discuss current forward build systems and what type of tracing
mechanism they employ and we investigate the possibility of combining existing
tools to arrive at a cross­platfrom solution.
• We have implemented a novel tracing tool ”Cairn”, which can isolate and trace the
building of any Linux program on any platform. Details in chapter 3.
• We have evaluated the isolated tracing mechanism. There is a natural performance
drop, due to the use of the Docker engine and an unexpected slowdown due to file
system optimization problems using chroot as an isolation method. It runs slower
than fsatrace, which was expected, however the magnitude of this was not. This
comparison was made possible by using the benchmarks of Rattle.
A detailed
breakdown of these can be seen in chapter 4.
• In chapter 5, we discuss our approach to the development of this tool along with the
integration process to a modern forward build system, Rattle.

Theoretical background
We will now introduce most of the necessary background knowledge and related work
required to understand this thesis. The outline of this chapter is as follows:
• In section 2.1, we discuss build systems, which is necessary for the reader to un­
derstand subsequent contents of this thesis.
• We discuss related work in section 2.2, where we go over tracers, dynamic depen­
dencies and potential existing solutions to our problem.
• Lastly, we discuss technologies that were used, such as Docker or FUSE in sec­
tion 2.3.
2.1
Preliminary knowledge
To understand the benefit of forward build systems, we also need to understand what
build systems in general bring to the table.
2.1.1
Build systems
Up until the introduction of Make[2], ad hoc build scripts were used to automate anything
related to delivering software products from source code[7]. Make makes use of make­
files to describe tasks and dependencies in textual form, an example of such can be seen
in Figure 1.1. The main point of make is to keep track of the relationship between dif­
ferent parts of the program, and execute just the right amount of commands to bring the
executable into an up­to­date state after changes were made to the source code. The
makefile describes the relationship between these parts to conclude which exact parts
of the source need to be recompiled. So, the textual representation is the same as the
theoretical task dependency graph in Figure 1.1. Based on the information in the graph,
Make schedules tasks in topological order. To be able to see which files changed, Make
keeps track of file modification times as persistent build information.[19]
The combination of file modification times and the task dependency graph leads Make to
being minimal, which is the ability of a build system to
• execute tasks at most once per build
• only execute tasks if they transitively depend on inputs that were changed
To give an example on how Make works, let us take the precondition that the user has
already built the program contained in Figure 1.1. If one makes changes to main.c af­
terwards, util.o does not need to be rebuilt as util.o does not depend on main.c. If the
programmer then changes both util.c and main.c, the build system needs to acquire both
util.o and main.o, however these tasks do not depend on one another, as such they can
be done in parallel.

With the example above we have described two of the main features of build systems:
incrementality and parallelism. Incrementality is the ability to build only the parts of a
project that have changed since the last build, rather than rebuilding the entire project
from scratch. Parallelism is the ability to perform build steps, whose output does not
depend on one another in a parallel fashion. [8]
Minimality, incrementality and parallelism are just few of the many properties that build
systems can be compared on. Make lacks a property that is essential to continue the
discussion on this work: dynamic dependencies.
2.2
Related work
Although the topic of command tracing is not well researched, through research on forward
build systems we can gain insight to how these tools work and what properties they should
have.
2.2.1
How to achieve dynamic dependencies
With regards to make, one needs to, with each step, declare not only the scripts to run, but
the inputs and outputs of those scripts as well. It is not able to discover dependencies of
build steps automatically. There are numerous ways to circumvent the use of such static
dependencies.
Scheduler
Each build system has a part that is responsible for determining the dependency order,
which is called a scheduler. There are three main types of schedulers when it comes to
build systems:
1. Topological: pre­computes the order of tasks in a way that makes sure that the build
result will be correct.
2. Restarting: builds tasks in arbitrary order, and when there is a task which requests
a dependency, which is out of date, it switches to that dependency as the next task.
3. Suspending: a reiteration of the restarting scheduler; it will only build dependencies
when they are requested. It requires tasks to be suspendable.
Type 1 lacks dynamic dependencies as the order of tasks cannot be changed on the
fly; while type 2 lacks minimality, as it might be the case that a task starts, does some
meaningful work, realises that there is a dependency missing and halts again. Type 3
achieves both minimality and dynamic dependencies.[19]
Tracing
A build system may choose to trace dependencies of tasks based on what files are being
written to or read from. There are numerous tools out there that look interesting; as such,
we chose to try them out to see if one could potentially arrive at a cross­platform tracing
mechanism which is capable of meeting the requirements set by our research question.
In most cases, we are interested in the possibility of running the tracer in some kind of
virtual environment, preferably Docker, as discussed in subsection 2.3.4.

strace
is a tool that is used to monitor and manipulate interactions between processes
and the Linux kernel.[13] It is able to track system calls, signal deliveries and changes of
the process state. strace is made possible by the kernel feature known as ptrace. The
earliest Forward Build Systems, such as Memoize[9] and Fabricate[10] rely on strace to
monitor relevant system calls to determine the inputs and outputs of commands.
ptrace
, as mentioned above, is a system call that provides a process with the ability to
observe and control another, while also being able to examine and change the memory
and registers of the target.[20] It is primarily used to implement breakpoint debugging and
system call tracing. The most notable current Forward Build System implementation that
relies on ptrace is Riker [12]. Riker notes down a set of system calls depending on the
processor architecture and determines file modifications and inputs and outputs of the
command based on that. Both strace and ptrace work within the confines of a Docker
container if the user provides the container with the SYS_PTRACE capability, discussed in
subsubsection 2.3.1.
fsatrace
is a utility that injects code into other applications to trace accesses to files[14].
It uses library interposition to do so, which is code injection between dynamically linked
libraries. The tool is employed by later versions of Shake [21] and is also the basis of
Rattle. There are several issues when it comes to using fsatrace, which the authors of
Rattle have also discussed in greater detail in [11]:
• It is not able to trace reads and writes between statically linked libraries.
• On MacOS, it is only able to trace invoked binaries of system binaries if we disable
system integrity protection [22].
• On Windows, it is not able to trace 32bit processes that spawn 64bit processes.
It is not worth discussing whether fsatrace can or cannot be applied as a cross­platform
tracing tool with the help of Docker, as it can already be compiled for most larger platforms.
libbigbro
is a library that provides functionality to run a command and track what files
it reads or modifies.[23] As strace, it is also made possible by the kernel feature known
as ptrace. It seems promising, and even the authors of Rattle plan to integrate it in fu­
ture versions.[16] There already exists one build system which employs libbigbro, called
Fac[24]. However, during our experiments we realized that it was not able to keep track
of everything that was happening under the hood. For example, if we delete a file, it will
not show up in the trace, as can be seen from the trace in Figure Figure 2.1: when we
create a file there is a trace for writing the file, however, the same cannot be said for the
case when we remove it.

2.2.3
Speculation and Hazards
The limitation of the previously mentioned approach lies in the fact that it is single­threaded.
To address this limitation, the concept of speculation was introduced. Speculation in­
volves predicting commands that are unlikely to conflict with each other but are antici­
pated to be necessary in the future. Then, these commands are executed before they
are actually required. The relationship between speculation and the concept of hazards
is significant [16]. Speculation revolves around assessing whether a command may lead
to hazards, allowing the system to execute only those commands in parallel that do not
pose a hazard.
There are three types of hazards when, for example, dealing with Rattle:
8
Cairn: virtualised build isolation and dependency tracing for forward build systems
==PAGE:17==
• Read­before­write hazard: Occurs if a file is written to after it has been read.
• Write­before­write hazard: Occurs if a file is written to after it has already been
written.
• Speculative­write­before­read hazard: Occurs when a file is written to by a spec­
ulated command and then read from by a command in the build script before the
build script requires the speculated command.
Addressing hazards involves reexecuting a command with reduced or no speculation in
most scenarios. While speculation is introduced in this discussion due to its relevance
to the evaluation process, it is noteworthy that it has minimal association with our tracing
implementation. We emphasise that tracing operates in a single­threaded manner, and
forward build systems leverage speculation to imitate parallelism. However, this intro­
duces the risk of hazards, necessitating the re­execution of commands, leading to longer
runtimes.
2.2.4
Existing solutions
To see if we actually need to come up with our own solution to the problem, we first
investigate if there are tools that already have the properties that we are looking for.
Outrun
Outrun is a utility tool that allows its users to execute local programs on a remote machine
without having to install the program to be executed on that remote machine.[25] This
opens up the possibility of exploiting resources that would otherwise not be available.
It does so by using a FUSE file system to mount the users’ local file system remotely
using an RPC[26] file system[27]. In theory, one could construct a local Docker container
and trace dependencies by executing the command within the container, and Outrun is
already capable of finding out what dependencies are missing. However, Outrun has
strict constraints to run. The host and the remote machine must be binary compatible;
it is not possible to set up a session between machines that have processors from the
x86 or AMD architecture family[28]; the two machines must be compatible. The tool also
is only available on different Linux distributions. Even in a hypothetical scenario, where
the host machine is the one running the build system, and the remote machine is just a
Docker container running Ubuntu [29], one can imagine that the use of Outrun will not be
possible, if, for example, the host is running Windows [30].
Try
Try is a tool with which you can run any command and observe which files were af­
fected[31]. It heavily relies on overlay files system implementation being present on the
host machine; with the help of it, it will run the command in the upper layer and presents
the effect that the command would have on the system. One can even delay the effect of
a command and apply it later on the system this way. As is the case with Outrun[25], Try
is only available on Linux distributions. However, given the scenario in which we were
to run Try inside a container on the host machine, it would work. When we tried running
Try in a Docker container, however, it would not work, no matter which base file system

2.2.5
Summary
In Table 2.1, we have collected the most notable build systems discussed in recent re­
search papers. As we can see, there are only three build systems, which are both forward
and achieve some levels of OS­agnosticity, namely Fabricate, Shake and Rattle. How­
ever, as already discussed in subsubsection 2.2.1, Shake and Rattle come with their own
set of issues due to their reliance on fsatrace, and Fabricate has no clear support for
MacOS.
There also exist potential tools to achieve dependency tracing without us having to create
our own solution to the problem; however, these are lacking in aspects, such as virtuali­
sation support or not being complete.
2.3
Technologies
To introduce platform­independency, we rely on common virtualisation techniques. In this
section we go over such technologies, along with FUSE, a technology that has already
been used related works (as we saw in subsection 2.2.4), and will also be employed by

2.3.1
Docker
Docker is an application that was designed to simplify the management of the lifecycles
of applications through the means of containers. [33] The ”container” is the fundamental
10

==PAGE:19==
building block of Docker: it is a process with a virtual environment that has all the nec­
essary files, libraries, etc. for the application bundled into it. To obtain a container, one
has to build it from an image. An image is a set of instructions that describes what com­
mands to run during the build, what the entry point of the container should be and what
other images it can rely on. Choosing to use Docker over standard means of running an
application has a few benefits to it:
• The application becomes portable across operating systems supporting the Docker
Engine
• Operating system agnosticism incurs significantly less overhead as virtual machines
• Easy distribution and management (using docker registries), which also helps with
continuous integration and delivery. [34]
• Applications run in an isolated environment
• Being lightweight and isolated, they allow for efficient scalability, reducing infras­
tructure clutter
The main drawbacks of Docker are as follows:
• Running docker incurs a performance drop compared to running the application
natively.
• Docker containers are not completely isolated from the host system, posing the risk
of compromising it.
• Requires more expertise to manage
The remainder of this section is a discussion that goes over several key features when it
comes to Docker and how they are relevant to this project.
Docker layers
At the core of any dockerized project lies the Dockerfile; it is a sequence of instructions
that tells the Docker engine how an image should look like and what steps it should go
through to create a container out of an image. Once a container is built and we try to
rebuild it, it looks up caches for each instruction and skips the ones for which a cache
exists. [35]
We can choose to create a Dockerfile from an existing image and build on top of that or
we can use the instruction FROM scratch to build a very minimal image.
Docker capabilities
When it comes to running containers, in general, the greatest emphasis is placed on
security and potential vulnerabilities. A barebones container on startup is going have as
few privileges as possible. This means that by default it will not be possible to mount a
FUSE­based file system, as it will not have the correct rights to do so. However, granting
root privileges to a Docker container is usually a bad idea. [36]

==PAGE:20==
This is where a feature called docker capabilities comes into place. The Linux kernel is ca­
pable of breaking down the privileges of the root user into units called system capabilities.
[37] This breaking down of root privileges into granular capabilities allows us to:
1. Remove individual capabilities from containers run as root
2. Add privileges to containers run as non­root one at a time
For example if one were to run the ptrace/strace[20][13] commands (as will be discussed
in subsubsection 2.2.1) one could add the SYS_PTRACE system capability to allow the con­
tainer to trace arbitrary processes using ptrace. If one needs to run a filesystem over
FUSE within a Docker container, however, among other things, they are required to add
the SYS\_ADMIN capabilities, which is essentially granting it root permissions. One may
layer such capabilities in any amount they wish, thus granting just the right amount of
rights to a container for the task at hand.
To allow, among other things, every capability and root access, the --privileged flag
can also be used when running a docker container.
Devices
There exists a /dev directory under Linux, which contains special or device files. [37] It
contains files, such as hda1 or hda2, which represent partitions of the main drive of the
system. The device that allows the spawning of user space file systems is called fuse.
FUSE file systems cannot be mounted without the use of this device.
By default, a container is not allowed to access any devices. If, however, the afore­
mentioned flag to make the container privileged is used, Docker will enable access to all
devices on the host.
Since the use of the privileged flag is not recommended (and most applications run with
root permissions), the --device=[...] flag can be used.
The design of Linux file systems allows for another way to make the FUSE device ac­
cessible within a container. Devices are just special files; as such they can be created
as needed. To create the fuse device one can run the mknod /dev/fuse c 10 299 or
MAKEDEV * commands. While the former creates the character­oriented (the input c), mis­
cellaneous (the major number 10) fuse device (the minor number 299), the latter will create
all known devices (MAKEDEV is only available when using an Ubuntu­based distribution).
Storage
A purely isolated container is only useful, if there is a way of giving it data and getting data
back from it. There are, in general, two ways of persisting data generated by and used
by Docker containers, if we don’t take networking into consideration.
Volumes
are the preferred mechanism to store data, since they are completely man­
aged by docker. They have a number of advantages over bind mounts (paragraph 2.3.1):
• Easier migration and backups
12

==PAGE:21==
• Manageable through Docker CLI or the Docker API
• They work on Windows containers as well
• Can be integrated with the cloud, encrypted, or be extended with other functionality
• It is more safe to share among containers
• Have higher performance on Mac/Windows hosts
To use volumes one can use the -v flag or the --mount flag with type volume.[38]
Bind mounts
are useful in scenarios where it is necessary that:
• the bind propagation be customized and rprivate is not enough or
• the container needs to continuously get access to new artifacts from the host ma­
chine.
Bind propagation is a concept that tells us whether or not a bind mount can be propa­
gated to replicas of that mount. There are a number of options when it comes to bind
propagation:
• shared: Sub­mounts of the original mount are exposed to replica mounts and vica­
versa.
• slave: Similar to shared but only in one direction.
• private: Sub­mounts are not exposed in either direction.
• rshared: The recursive counterpart of shared.
• rslave: The recursive counterpart of slave
• rprivate: The same as private
The recursive counterpoints are the same as the original ones, with the addition that the
propagation also extends to and from mount points nested within any of the original or
replica mount points.
To use bind mounts, one can use the -v flag or the --mount flag with type bind. Bind
propagation can be specified as bind-propagation=.... [39]

Virtual machines
A virtual machine (VM) provides users with a complete operating system with the illusion
that there exists a machine dedicated to them. They are controlled by virtual machine
managers (VMM) or hypervisors, which provide an abstraction layer between the VM and
the underlying hardware. These hypervisors are usually divided into two categories: Type
1 and Type 2, with the former ones being able to access the hardware directly (they are
run in kernel mode and have dedicated CPUs), while the latter ones run on a piece of
software that is being run on top of the OS with virtual hardware. [40]

Advantages of virtual machines
Virtual machine­based virtualisation has the benefit of enabling programmers to use mul­

2.3.5
File system in Userspace
Filesystem in Userspace (FUSE) is a user space file system framework, available for most
Unix­like operating systems along with OS X and Windows. A user­space file system is a
file system in which data and metadata are provided by an ordinary user­space process.
The file system can be accessed normally through the kernel interface[37]. What is the
benefit of writing file systems in the user space?
Most computers have two modes of operation: kernel mode and user mode. The operat­
ing system runs in kernel mode, a mode in which it has complete access to all hardware
and can issue any instruction that said hardware is capable of executing. The rest of the
software runs in user mode, where only a subset of the instructions are available. I/O op­
erations, for example, are not allowed in this mode.[42]. The terms kernel space and user
space refer to the two areas that system memory gets divided into; kernel space being
only permitted for use by processes running in kernel mode, while processes running in
user mode only have access to the user space. User space processes can only access
a small part of the kernel through an interface that is using system calls.[43].
This is an important detail, as when a user space program needs to communicate with
the file system, it issues a system call, as file systems are part of the operating sys­
tem. The FUSE interface allows users to create their own file system implementations
that are different from the kernel implementation. FUSE file systems are quite flexible:
they can be mounted/unmounted, programmed and deployed by an ordinary user rather
quickly.[44][37]
The FUSE framework comprises of three different parts:
• a kernel module, fuse.ko, the purpose of which is to facilitate communication be­
tween kernel­space and user­space,
• a user­space library, libfuse.*, which exposes the necessary functions to handle file
operations and communicate with the kernel from user­space
• and a mount utility tool, fusermount, the purpose of which is to enable a non­
privileged mount without compromising underlying system, as the standard mount()
system call is a privileged operation.
The libfuse user­space library contains many operations that can be implemented to cre­
ate a functioning file system, all of which are optional, although there are some which
are essential, as all of them are required to create the simplest of file systems. These
operations are:
• getattr: Retrieves attributes (metadata) of a file or directory.
• read: Reads data from a file.
• readdir: Reads the content of a directory.

Platform­specific implementations
The libfuse user­space library is not cross­platform, as it is only available on Linux dis­

tools that grant users the ability to develop their own file systems on different operating
systems
• WinFsp (Windows File System Proxy) is a FUSE implementation for Windows, which
provides a runtime and development support for custom file systems on windows[45].
ne may choose to use Dokan instead of FUSE to write file systems for Windows.
Recent versions of Dokan also come with a FUSE wrapper to provide compatibility
with the FUSE API.
• macFUSE (formerly known as OSXFUSE) is a software package for macOS, which
provides multiple APIs for running and developing file systems on macOS. One of
the APIs is a superset of the original libfuse API. [46]
Developing file systems in different frameworks and languages
FUSE file systems can be developed in multiple programming languages. Below is a
collection of the most popular frameworks to do so.
• C: The official FUSE library[18] is primarily designed for C.
• Python: The library fusepy[47] provides us with bindings with which we can cre­
ate FUSE file systems in Python. It is especially useful for rapid prototyping and
development. It is compatible with libfuse, osxfuse and WinFsp.
• Rust: The fuser[48] library is a complete rewrite of the official FUSE library in rust,
providing the same level of compatibility.
• Go: Like fuser, bazil/fuse[49] is a from­scratch implementation of the kernel­userspace
communication protocol in Go.
Example applications
FUSE can be applied in a number of scenarios.
• An overlay file system[37] is a type of file system that enables the stacking or
layering of multiple file systems on top of each other, presenting a unified view of
the files and directories. Each layer of the overlay represents a distinct file system,
and the contents of these layers are combined to create a composite view. A typical
use case for overlay file systems is Docker: it overlays a local directory with the
contents of a container. As another example, one can overlay his local file system
with a cloud storage to create a unified view of all files.
• A passthrough (proxy) file system[18] is a type of file system that does not intro­
duce any additional layers or modifications to the underlying file system. Instead
of creating a composite view like the overlay file system, a passthrough file system
simply forwards file system requests directly to the underlying or host file system
without any modifications.

Cairn: the tracing tool
Cairn is a Docker­based tracing tool that creates an isolated environment where it traces
file system commands using the FUSE 3.0 API. It comprises of two major parts: a locally
hosted FUSE file system running inside a Docker container (Cairn FUSE) and a client,
running on the host (Cairn Client).
3.1
Overview
To introduce Cairn we first go through an example with the help of a sequence diagram
that can be seen in Figure 3.1. Let us take the simplest example and say that the build
script that we are trying to run consists of a single command gcc donut.c -lm. We use
gcc to compile a C file, the contents of which can be found in section A.1. We break down
the sequence diagram into the following three parts: initialisation, tracing, and cleanup.
3.1.1
Initialisation
After the build process has been initialised (1) and the build script has been parsed (1.1),
the tracing environment needs to be started. The logic of instantiating the docker contain­
ers is not something that is implemented in Cairn Client, so this has to be done either by
the build system or manually, before running the build. Although the implementation is
trivial, we provide an example script on how to instantiate the containers, along with the
corresponding Dockerfile in section A.2(1.2). The instantiated container then starts the
file system (2).
3.1.2
Tracing
To begin the tracing, the forward build system calls Cairn Client (1.3). An example com­
mand invocation is as follows: cairn rwmdt output.log -- ./gcc donut.c -lm, where
we specify the necessary operations we would like to trace (we discuss this in detail in
subsection 3.2.2), the target log file and the command. Cairn Client then issues a call
to the container to execute this command with a script (1.3.1) that essentially issues the
command to be run with a different root, using chroot (1.3.1.1).
While the command is being executed, the file system logs each operation that was issued
on the bind mount folder (1.3.1.2). This intermediate trace produced by the file system
can be seen in Figure 3.3a.
After the command has been executed, the client reads the log and filters it (1.3.3). With
the help of bind mounts, it is able to directly access the tracer log, as this is the location at
which the file system is producing its log. It then produces the log at the location specified
by the build system. An example of a final log can be seen in Figure 3.3b.
3.1.3
Cleanup
Although not necessary, if the build system decides to free up allocated resources, it can
destroy the container, thus stopping the file system.

Structure
As discussed above, our solution comprises two major parts: a client and a file system.
In Figure 3.2, we describe how the different components of a system using Cairn are

3.2.1
Cairn FUSE
Cairn FUSE is a passthrough file system running inside a Docker container. It is written
in Rust[51] and is built upon the fuser library [48]. A detailed explanation on the choice
of framework and programming language can be found in chapter 5. Cairn FUSE is es­
sentially a proxy file system, which acts as an intermediary between Cairn Client and the
underlying file system of the Docker container. The primary purpose of this implementa­
tion is to add tracing as an additional side­effect of file system operations.
Note that the file system is not complete in the sense that there are a number of operations,
which are not implemented, as they are not strictly necessary for the demonstration of this
approach. Such omitted messages of the protocol include:
• Messages that deal with extended attributes such as SETXATTR GETXATTR, LISTXATTR,
REMOVEXATTR

• Synchronization messages such as FSYNC and FSYNCDIR
• Messages that can be replaced by a combination of other messages, such as CREATE
• MacOS related messages, such as SETVOLNAME, EXCHANGE, GETXTIMES
• Messages that only make sense with mount options that are not used by us or only
useful for different types of file systems, such as BMAP, SETLK, IOCTL
• Miscellaneous messages, such as FALLOCATE, LSEEK, COPY_FILE_RANGE and many
more.
To simplify things we decided to use bind­mounts (discussed in paragraph 2.3.1), so that
the build directory can be easily made available to the container and the FUSE file system.
There are also two important steps to be taken to be able to run a FUSE system in a Docker
environment:
• As shown in subsubsection 2.3.1, it is recommended by Docker to grant the SYS_ADMIN
capability to the container
• It is necessary to mount a FUSE device OR run the container with the --privileged
flag and create the device file inside the container, as seen in subsubsection 2.3.1.
Although conceptually it would not be necessary, applying a chroot on the mount point of
the FUSE file system whenever we are issuing commands allows us to manage what files
can be accessed/seen, as such we can be more explicit on what libraries and binaries are
used. It also helps us mitigate issues that may arise when dealing with paths in general.
In an ideally developed and structured system, calls to these paths could be dealt with on
a file system level.
3.2.2
Tracing
We already saw that our choice of tracing method is based on a custom user space file
system implementation. Other methods of tracing were either
• incomplete, such as fsatrace, or bigbro,
• were not compatible with a Docker­based environment, which is the case when it
comes to try,
• required significant modification, to the extent where it is easier to make an imple­
mentation from scratch, such as the case with Outrun.
A ptrace­based tracer, such as strace, would have been a valid choice for the imple­
mentation of a tracing system with a custom wrapper around it to make it work with the
environment. There are mainly two reasons as for why it did not end up being the tool of
choice:
1. The forward build system Rattle[11] provides decent benchmarks, with which we
can provide more meaningful insights into how reasonable it is to choose our work
is as a tracing tool for forward build systems. Since Rattle uses fsatrace, the tracing
22
Cairn: virtualised build isolation and dependency tracing for forward build systems
==PAGE:31==
format of which is much easier to produce than that of what strace provides, FUSE
seemed to be the correct choice.
2. A FUSE­based tracer has much more potential for future extensions, such as not
necessarily needing a perfect system replication in the chroot environment in the
form of file system mounts. In an ideally developed fuse layer, these accesses
could be dealt with to increase robustness and reduce computational needs.
The result of a tracing format is a newline­separated sequence in the following format:
<operation-type>|<path-to-the-file>[|<path-to-the-origin>]. <operation-type>
can be one of the following:
• r: for read (such as open(2) with read flags),
• w: for write (open(2) with write flags),
• m: for move (rename(2)),
• d: for delete (unlink(2)),
• q: for file stat (stat(2)),
• t: for touch operations (utime(2)).
Refer to the Linux manual pages for a precise definition of the respective system calls[37].
<path-to-the-origin> is only present whenever we are dealing with move operations.
Otherwise <path-to-the-file> is the path with which the respective operation is asso­
ciated.
The example in Figure 3.3 was produced by running the same command that we dis­
cussed in section 3.1: cairn rwmdt output.log -- ./gcc donut.c -lm.
There are
some things to be noticed when looking at the resulting log:
1. This command accesses gcc from the working directory. As we will discuss it in
section 3.5, system binaries do not get traced when using the current version of
Cairn. The most important system­directories are mounted and accessed not by
the FUSE file system but by the file system of the container and so there is no trace
in relation to them. In fact, using gcc instead of ./gcc would also work, however the
first line of the trace would not be there.
2. r or w operations are not traces of atomic reads and writes, but rather open calls
with access mode read and write flags, respectively. If both flags are present in the
open call, it is traced as a write. This implementation detail is the same as it is in
fsatrace.
3. The q option was omitted to make the log more readable.
To achieve the output seen in Figure 3.3a, there exists a mapping between messages of
the FUSE protocol to operation types:

==PAGE:32==
• SETATTR maps to w if SETATTR can be matched to one of (chmod, chown, truncate).
Otherwise, it maps to t, since the remaining cases only deal with modification or ac­
cess times.
• UNLINK maps to d as it essentially just removes files.
• RENAME maps to m, which is self explanatory.
• OPEN maps to either r or w depending on what access mode was used to open the
file. If both read and write are present, it defaults to w.
• GETATTR maps to q, as GETATTR is analogous to stat(2) or fstat(2) according to [37].
The above mapping results in more verbose traces compared to fsatrace. As can be read
from the discussions on Rattle1, the above mapping seems to be an improvement, as it
results in traces that contain modifications to file attribute information. From a perspective,
it is also too verbose, as GETATTR is a message that is sent numerous times during a build
process and spams the log file quite a bit.
3.2.3
Cairn Client
Cairn client is a thin binary the purpose of which is twofold:
• It wraps the issued commands, making sure they run on within the container, on top
of the file system.
• Filters the log produced by the file system (Figure 3.3a) based on filtering criteria
and produces a log for the build system (Figure 3.3b).
The file system does not have any notion of commands being run, as such, it logs every
operation that occurs on its level onto a tracer.log. As we can see from Figure 3.3a,
Cairn FUSE also logs the IDs of the processes and their parents, which invoked the op­
erations, as well as the timestamp at which the trace was produced.
Cairn Client notes down the root process ID and the timestamp at which the command
being traced spawns, and post­processes the log produced by Cairn FUSE to filter in
traces of which:
1. the process ID matches the root process ID,
2. the process ID is a child of root process,
3. the command occurred at or later then the spawn time.
Keeping track of spawn time is necessary as it might be the case that the container decides
to reuse process ids for which there are already traces in the logs.

Provisioning
As seen in subsubsection 2.3.1, there are multiple ways of storing data coming from a

the tracer running inside the container access to the necessary files, which are usually
just repository files. For this purpose, we are using the bind mount feature of containers,
allowing us to bind specific directories to the container. This allows us to
• not duplicate build directories on the system,
• provide the container with the necessary binaries,
• keep the produced trace by Cairn FUSE locally, so that post­processing the log will
not incur another overhead of connecting to the Docker container.
3.5
Limitations
This version of Cairn comes with a number of limitations.
• When constructing the container and the working directory into which the client will
send its chroot command into, the system­related directories, bin, lib, usr ...,
are mounted (see Figure 3.2). As such, the FUSE file system is not responsible for
handling these files and is also unable to trace any related file system operations.
The solution to this not conceptually difficult, however it was chosen to be omitted
from this implementation as it is cumbersome to develop and test thoroughly. One
may imagine that an easy solution to this problem (since this FUSE layer is just a
passthrough implementation) is to just delegate the appropriate file system opera­
tion to the root of the container. Another solution is to keep an up­to­date copy of
the underlying system directories within the FUSE mount directory and to have it
managed by the file system.
• Another limitation or assumption is that the user will be able to provide the necessary
binaries for the project to build. Hypothetically, this could be done by provisioning
the container with the necessary binaries at the mount point (see section 3.4) or by
the use of preconstructed base tracer images, as we talked about in subsubsec­
tion 2.3.1. The latter would most likely be preferred, as then the user of the build

rustc[51] etc. However, from our point of view, this is just a software distribution

3.6
Security
While creating a secure tracing tool is not one of the main purposes of this thesis, it is
worth going over some of the security aspects, when it comes to Cairn and using it locally
or in the cloud.
3.6.1
Docker and FUSE
For a Docker container to be able to run a FUSE­based file system, it needs to be granted
the CAP_SYS_ADMIN kernel capability, as discussed in subsubsection 2.3.1. The use of
this capability is highly advised against, as it may grant the attacker access to the host
machine; a container breakout can occur.[54]
3.6.2
Docker and virtual machines
In general a virtual machine can be more secure than using a container for virtualisation.
There are mainly two reasons for that:
1. Most of the time, containers share their kernel with the host2.
2. Virtual machines run their own operating systems with their own security features
and they are also run on a hypervisor, which comes with another layer of security.
These provide more security zones, which may be less vulnerable. [40]
3.6.3
Cairn CLI command wrapper
For the sake of simplicity, Cairn CLI is just a wrapper, which executes commands in a
docker container using the docker exec [OPTIONS] CONTAINER COMMAND [ARG...] com­
mand. And since inputs are not sanitized, one may easily fabricate a command, such that
it adds a --privileged flag or the likes, making Cairn be vulnerable to command injection
attacks[55].
3.6.4
Conclusion
With all of the above mentioned points considered, we do not recommend the use of this
tracing tool, as it most certainly poses as a security risk.

Evaluation
To evaluate our findings in this MSc. thesis, we go over our research question again
(introduced in chapter 1) and break it down into smaller parts. We go over each and
answer them and point to the relevant sections that form the basis of our argument. Our
research question goes as follows:
RQ: Is it possible to isolate and trace builds in a platform­independent and
efficient manner?
To investigate this question we break it down into the following parts:
RQ1: Is our approach capable of isolating commands in a platform­independent
manner?
To which our answer is yes! This argument has already been covered in section 3.3 where
we discussed isolation and platform independence as properties of Cairn, which emerge
due to its dependence on Docker and the FUSE file system running within that.
RQ2: Is our approach to tracing commands efficient?
No, our approach is definitely not efficient. When dealing with long­running tasks it achieves
on average a 5.6 times increase in runtime, and with file system heavy calls it can be 22.6
times slower. This raises another question:
RQ2.5: Why is this approach slow?
When measuring the performance of short­lived commands we can observe a major per­
formance drop in two parts of the application:
• Most of the performance drop stems from opening the connection to the Docker
container.
• A portion of the slowdown stems from the fact that we rely on chroot to make sure
that the command uses the libraries and files on the file system level. It is even more
apparent on commands, which need to look up more external binaries, making the
approach scale with the complexity of number of lookups.
When measuring the performance of long­running tasks, we can observe that, while the
performance drop is just as significant, the reasoning behind it is a little different:
• Connecting to the docker container incurs a fix flat increase in time, which is as
expected, as we only open the connection once per command.
• A more significant portion of the slowdown is explained via the use of chroot.

more desirable than plain virtual machines for virtualisation in our use case. In section 3.3

Development
In this chapter, we go over the development process of Cairn, and how we arrived at the
solution that we are presenting.
5.1
Methodology
Throughout the course of this MSc. project, we followed the iterative design process[61].
This process involves cyclic phases of prototyping, testing, analysis, and refinement ap­
plied to the designated product or process. The goal is to enhance the overall functionality
and quality of the product by incorporating the insights gained from testing and feedback.
Although actual user testing was not performed, we embraced the methodology by con­
sistently employing benchmarks.
Over the course of the project, we successfully completed four major cycles, each con­
tributing significantly to the evolution of our tracing tool. The subsequent sections delve
into the details of each iteration. To see the benchmark results of each iteration, refer to
chapter 4.
5.2
Initial Planning
In the initial planning phase, our primary objective was to ascertain the feasibility of our
approach and emphasize the creation of a tracing tool with reasonable performance. This
phase laid the groundwork for subsequent iterations by establishing the initial design prin­
ciples and performance expectations.
5.3
First Iteration
• Requirements: Our focus in the first iteration was to prototype a Dockerized tracer,
choosing from the tools Outrun, Try, or FUSE.
• Analysis & Design: Despite initial attempts with Outrun and Try, we determined
their insufficiency and opted for a quick prototype file system in Python using the
fusepy library [47].
• Testing: Manual testing was conducted to verify the capabilities of the Python­
based tracer.
• Evaluation: To establish a foundation for performance comparisons, we developed
a set of benchmarks that could be reused throughout the project.
5.3.1
Discussion
The first iteration resulted in a tracer implemented in Python. Initially we compared the
runtime of Cairn to that of running commands locally and running commands on top of
a Docker container, which are environments used by software developers in their daily
work. The data revealed a significant slowdown of a factor of 7 when utilizing the tracer

Second Iteration
• Requirements: Building on insights from the first iteration, our goal in the second

• Analysis & Design: Recognising Python’s inherent slowness, we rewrote the file

• Testing: A set of unit tests and manual checks was implemented to ensure the

• Evaluation: The benchmarking suite was rerun to assess performance improve­

5.4.1
Discussion
In the second iteration we have added more test cases. We added test cases, which were
more complex compilation­wise, and opted for ones, which had to link other libraries, such
as the mathematical library, thus increasing average compilation times. Compared to our
initial expectations, the performance improvement was not as significant as we initially
expected it to be. However, there was an improvement in terms of speed, as the tracer
shows a slowdown of around 5 instead of 7. Therefore we introduce another passthrough
implementation, written by the authors of the libfuse library. The purpose of the new con­
text was to determine whether the implementation is at fault, or it is this costly to run a

FUSE file system within a Docker container. Judging from the results, we determined
that doing so should not incur such performance slowdowns. We hypothesized that the
continuous opening and closing of the shell to the Docker container might be a factor
contributing to the observed performance overhead. Consequently, we opted to dive into
understanding the performance difference more deeply by introducing additional bench­
mark layers.
5.5
Third Iteration
• Requirements: The third iteration aimed to further enhance performance by iden­
tifying and resolving issues that persisted from the previous cycle.
• Analysis & Design: To pinpoint performance bottlenecks, we systematically broke
down benchmarks into different steps, carefully assessing the implementation and
the overall approach.
• Testing: Additional benchmarking steps were explored to identify potential areas
for performance improvement.
• Evaluation: The benchmarks were executed to uncover the root causes of the
observed slowdowns, providing valuable insights for further refinement.
5.5.1
Discussion
We have introduced 3 new layers from which we came to the following conclusions:
• Cairn FUSE, our file system implementation, is not the bottleneck in our approach,
as with its added layer of tracing and logging it barely falls behind in terms of speed
from the worse performing libfuse passthrough implementation. Our file system
implementation is around on average 1,25 times worse from that.
• Our hypothesis that the continuous opening and closing of the shell with docker exec
calls only explain some part of the slowdown when it comes to tests on gcc, and a
large part of it when dealing with tests on ls. Therefore, it was not correct to assume
that this was the most important slowdown factor. We put aside the large difference
in performance between ls and gcc as a problem that can be attributed to our choice
of virtualisation method, and we hypothesize that the major difference between the
two is due to the latter needing to pipe its results back to the host machine.
Our goal for the next iteration was to fix major issues with Cairn Client, as this seemed to
be the place where the main issue lied.
5.6
Fourth Iteration
• Requirements: In light of the findings from the third iteration, the fourth iteration
focused on the integration of the tracing tool into a forward build system, as further
improvements seemed unattainable.
• Analysis & Design: We explored various forward build systems, eventually select­
ing Rattle for its simple integration and existing benchmarks despite the unfamiliarity

of the programming language used, as discussed in subsection 3.2.2.
• Testing: The rattle benchmarking suite was integrated into our system and the
results were systematically evaluated for consistency and precision.
• Evaluation: The results of the benchmarks were examined to assess the perfor­
mance implications of the tool’s integration into the chosen forward build system.
The objective of the fourth iteration was twofold:
• The first one was to optimise the performance of the client as previously we had
hypothesised that this is where the main performance bottleneck lies.
• The second one was to integrate Cairn into Rattle and run its benchmarks as detailed

Throughout this development cycle, we had come to the conclusion that it was actually
not the client, which was lacking in performance. As it turned out, the majority of the
performance issues lied in the fact that we had to introduce a chroot call to the directory
structure which is running Cairn FUSE. The chroot call was introduced as a means to run
commands in the isolated environment of the Docker conatiner on top of Cairn FUSE. We
initially did not speculate that it would lead to bad performance, however as it turns out it
certainly is one of the factors which lead to this approach being slower than anticipated.
5.7
Integration
To integrate Cairn into any forward build system the following approach can be used:
1. Replace the forward build system’s command invocation to that of Cairn’s and deal
with its output format.
2. Initialise the Docker containers, where the specified mount point is the root directory
of the project, which is being built. In the Cairn repository there exist scripts that deal
with this. An example initialization script can be found in section A.2.
To evaluate Cairn, we have chosen to integrate the tracer into Rattle[11]. To see the
results of the evaluation, refer to chapter 4. During the evaluation of Cairn in terms of
forward build systems we have chosen to value speed of integration, instead of reliability.
As discussed in Table 2.1, Rattle uses fsatrace under the hood to trace dependencies.
Our simple approach to make sure that it uses our binary was for us to develop Cairn
in way that it matches the inputs, output formats and binary name to that of fsatrace’s.
This way, we were not required to make modifications to any of the libraries of Rattle
and its dependants, such as Shake. The next step was to make sure that the folders,
which the benchmarking suite relied on, were used. Each Rattle build command was
configured to be run in a temporary folder, to which the current commit was checked out
into. Therefore we extended the process by instantiating and killing Docker containers
with each checkout to make sure that the correct directories were being mounted and all
the paths were accounted for. The drawbacks of this integration approach is that currently

6
Discussion
This MSc. thesis discussed the possibility of the enhancement of tracing for forward build
systems by introducing a proof­of­concept of a cross­platform tracer, Cairn. Nowadays,
there is a wide­spread use of containerised tools as they seemingly work out­of­the­box,
with just the help of a Docker engine, and minor performance implications. It is therefore
common sense to try and see if our favourite tools could be used inside of a container.
6.1
Future work
One may argue that it is not necessary for a FUSE­based tracing tool to be constrained
to the confines of a virtual system, and since, as discussed in chapter 2, the FUSE API is
available for most modern operating systems, it is entirely possible to create a tracing tool
that provides native support for most of these APIs. The engineering might be difficult,
however there are file system implementations with cross­platform support [62][63] to
serve as examples. Cairn is a proof­of­concept for a cross­platform tracing tool, and as
such it comes with its compromises, which were discussed in greater detail in chapter
chapter 3.
The initial draft of this thesis work framed its research question in a way that discusses
the possibility of whether it is also possible to suspend the execution of commands in
the isolated tracing environment. Due to time limitations, this aspect was not researched.
However a tracer that could potentially halt its execution until the file dependency require­
ments are met would be desirable.
Another feature that could see potential is the capability of tracing web dependencies of
commands; the caching of the results of HTTP calls could allow a forward build system
to outsource this work (which is the responsibility of a regular build system) to the tracer.
In chapter chapter 4, we tried explaining the implications on the performance of build
systems when using this variation of a tracer, however one clear way to optimize any such
Docker­based approach is to keep the shell, which one uses to access the hosted system,
alive. Cairn was designed with the intention of tracing single commands, however in a real
world scenario it is rarely the case that only a single command gets executed throughout
the course of a build. Therefore, if one were to design a production­ready system such
as this one, it is recommended to keep command piping in mind.
The paper on Rattle[11] discusses what their ideal tracing tool would be capable of:
• has a cross­platform API
• has precise detection times
• capable of detection as access happens
• capable of executing code at interception points
Cairn: virtualised build isolation and dependency tracing for forward build systems

Of these four points, Cairn checks the first three ones out. It has a cross­platform support
with the help of Docker and is able to log precise times of file system operations. Due to the
nature of FUSE file systems, it is also able to detect as soon as access occurs. However
our implementation does not address the issue of tracing system binaries, unlike fsatrace,
which Rattle relies on, which we discussed in section 3.5.
6.2
Conclusion
In this work we have investigated if it is at all possible to help out the advancement of
research on forward build systems, by asking the question whether it is possible to create
a tracing tool, such that it is platform­independent, through the means of virtualisation,
but is also efficient. We have investigated this issue by introducing Cairn a cross­platform
tracer, integrated it into a forward build system and employed performance benchmarks
to analyze whether we were able to accomplish such a feat. In the end, the approach
turned out to be rather slow, so we cannot safely say that we are able to create such a
tool, however, as there are other means of arriving at a similar solution, we also cannot
argue that it is not possible.

“A cairn is a human­made pile (or stack) of stones raised for a purpose, usually as a marker or as
a burial mound. Cairns have been and are used for a broad variety of purposes. In prehistory, they
were raised as markers, as memorials and as burial monuments (some of which contained cham­
bers). In the modern era, cairns are often raised as landmarks, especially to mark the summits of
mountains. Cairns are also used as trail markers. They vary in size from small stone markers to
entire artificial hills, and in complexity from loose conical rock piles to elaborate megalithic struc­
tures. Cairns may be painted or otherwise decorated, whether for increased visibility or for religious
reasons.”
