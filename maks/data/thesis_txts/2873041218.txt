Abstract
This project deals with implementing Rust support for the RISC-V based Wildcat project.
In addition to this, the project also aims to facilitate support for performing static worst-
case execution time analysis for Rust-based target towards the Wildcat architecture. As
a result, this project has ensured that Rust can be used in development of applications for
the Wildcat project, and furthermore, has managed to facilitate initial support for WCET
analysis for these applications. For Rust support on Wildcat, this project uses RISC-V
tools developed by the Rust community, allowing for effortless development of reliable
embedded software. For WCET analysis, this project utilizes Rusts LLVM compatibly,
and builds upon the approach of the T-CREST project, leveraging their development of
extended LLVM tools, which, among others, facilitates support for WCET analysis using
the Platin Multi-Target WCET tool also used in this project. While acknowledging pos-
sibilities for improvement and considerations for future work, in conclusion, this project
predominantly completed the objectives originally set out to achieve.

Introduction
This chapter will provide an introduction by presenting the motivation, objective, report
structure, relevant resources, and delimitation for the project.
1.1
Motivation
Embedded systems are becoming increasingly integrated into vast areas of our society,
whether it is personal assistive devices like medical devices, autonomous vehicles or
large-scale IoT systems (e.g. monitoring critical infrastructure). Many of these are safety-
critical systems, where both faulty software and failing to fulfil time-restricted tasks may
have fatal consequences. Writing software for embedded systems using Rust, developers
can guarantee memory-safe software while maintaining performance, unlike languages
used embedded systems like C/C++. Furthermore, being able to perform WCET analysis
for Rust programs for embedded systems, developers can guarantee that time-restricted
tasks will be fulfilled.
1.2
Objective
This project has of two major objectives:
1. To compile and execute Rust-based programs for the ISA simulator and hardware
designs of the Wildcat project.
2. To perform worst-case execution time analysis of Rust-based programs using the
hardware designs of the Wildcat project.
1.3
Report structure
Including this first introductory chapter, this report consists of ten chapters. The second
chapter contains related work for this project. Afterwards, the third chapter gives a brief
introduction to the background concepts, which readers need to understand the project.
Following this, the fourth and fifth chapters describe the process of implementing Rust
support and WCET analysis for the Wildcat project. After this, the sixth chapter presents
the results of the practical section, which is followed up by the seventh chapter discussing
these results. The eight chapter contains the conclusion of the project. Finally, the ninth
chapter present considerations for future work.

2.1
Worst-case execution time analysis
Our work for this project heavily depends on the concepts and tools developed as either
a part of, or originating from, the T-CREST project1, including their approach to compilers
and WCET integration[1]. In practice, we use the extended version of LLVM and Clang
as an integral part of our custom tool-chain, facilitating the possibility for WCET analysis
of Rust-based programs. Moreover, we use the Platin tool[2] to perform WCET analysis.
Other comparable approaches for WCET analysis include academic, open-source tools
such as SWEET[3], developed at Mälardalen University, Sweden, and Chronos[4], de-
veloped at National University of Singapore. However, integrating support for Rust into
these tools requires a vastly different approach than what has been done for this thesis,
as none of these tools leverages LLVM and therefore not natively support Rust.
Although sources on WCET analysis for Rust-based programs are limited, Marcus Lind-
ner et al.[5] explore WCET analysis for the Rust-based RTFM (abbr.
Real-Time For
the Masses) framework, now known as RTIC (abbr. Real-Time Interrupt-driven Concur-
rency)2, using the KLEE tool3[6]. The KLEE tool operates directly on LLVM-IR, thereby
allowing integration with Rust-based programs. This approach of leveraging an LLVM-
based tool like KLEE for WCET analysis is similar to our approach of using Platin. The
main difference is that Lindner et al. rely on symbol execution and HiL (abbr. Hardware-
in-the-Loop) to dynamically determining WCET, while our approach with Platin uses IPET
(abbr. Implicit Path Enumeration Technique) and hardware architecture models to formu-
late ILPs (abbr. Integer Linear Program) to determine WCET statically.

3
Background concepts
This chapter outlines the background concepts, which provides the foundation for this
project.
3.1
RISC-V
RISC-V (abbr. Reduced Instruction Set Computing V) was introduced in 2010 as a part
of Andrew Watermann’s PhD thesis[10]. It is an ISA (abbr. Instruction-Set Architecture)
originally designed to support computer architecture research and education, but with a
wish to become a standard free and open architecture for industry implementation[11,
p. 1]. As wished, RISC-V has become widely adopted in both industry and academia,
and have managed to gain a large community, which is actively improving support for
RISC-V across a variety of fields, both in the form of hardware implements, but also as
compilation targets in projects like LLVM and GNU.
The term RISC was first introduced by David Patterson around the early 1980s, as a
response questioning and challenging the concepts of CISC (abbr. Complex Instruction
Set Computer). Here, RISC is provided as an cost-effective alternative focusing on a
small instruction set, consisting of simple, single-responsibility instructions, executed in
a single cycle. The purpose of this was to simplify the software and hardware design
processes.[12][13].
3.1.1
RISC-V and Wildcat
RISC-V is essential for this thesis, as it is foundation for the Wildcat project[14]. The Wild-
cat project was created by Martin Schoeberl, and consists of a collection of RISC-V cores
designed for education and real-time systems, containing an ISA simulator along with
three different hardware implementations for the RV32I base instruction set[11, p. 2]. Even
though multiple hardware implementations are mentioned, the main focus is converging
towards the three-stage pipeline implementation, which therefore also will be the main
focus for this thesis. The three-stage pipeline implementation is interesting, as it deviates
from traditional RISC five stage pipeline. By departing from previous norms, the three-
stage pipeline simplifies hardware requirements and eliminates common pipeline hazards
- all while performing better than the project’s four- and five-stage counterparts[14, p. 4].
3.2

The LLVM (abbr. Low Level Virtual Machine) project1 was originally began as a research
project at University of Illinois, IL, US by Chris Lattner and Vikram Adve. Their philosophy
was to create a compiler framework designed to support transparent, lifelong program
analysis and transformation for arbitrary programs[15]. In modern times, LLVM has be-
come the backbone in various different (high-level) programming languages, providing

3.3
Rust
Rust is a systems language, whose design philosophy to help developer write faster,
more reliable code. Rust does this by providing high-level features and performance.
Unlike other languages like C/C++, Rust allows for low-level control without the usual
memory safety challenges, which is another key feature of Rust. It was originally started
by Graydon Hoare at Mozilla in 2006, and has since established a thriving community,
being one of the most beloved languages amongst developers in recent times.
3.3.1
Embedded Rust
As the Wildcat project is still in its early stages of development, higher-level features such
as operating systems and dynamic memory allocation are not available. This requires
programmers to write code that interacts directly with the hardware. This concept is com-
monly known as embedded or bare-metal programming.
When writing for embedded or bare-metal systems, Rust is more limited, as standard
library (std crate2) is dependant on an operating system in order to be loaded and used[7,
p. 1.2][7, p. 1]. Instead, embedded Rust relies on the core crate3, which provides basic
building blocks of all Rust code without linking any upstream or system libraries. In order

3.4
WCET
At its core, WCET analysis is used for determining an upper-bound of a given task using a
specific hardware system. This is crucial for real-time systems, where failing to complete
a given task within a certain time limit will lead to incorrect results, and, in worst-case
scenarios, have fatal consequences.
For this thesis, we use Platin to statically determine WCET on Rust-based programs using
a architecture model based of the three-stage pipeline implementation from the Wilcat
project. In order for Platin to be able to determine WCET for a given program, it relies on
a meta-information file with information derived from a given source program, along with
a corresponding binary file. For this thesis, we generate both the meta-information file
and corresponding binary file for a given Rust source file using emitted LLVM-IR, which
is then compiled used the extended version of LLVM and Clang (see section 1.4.3).
The meta-information file includes a control-flow graph for a collection of bit-code func-
tions from LLVM-IR along with potential flow fact annotations, a collection of machine
code functions, and a relation graph, which describes how the two representations are
connected. Using this information, Platin is able to derive a control-flow–relation graph,
which can be used to lower the potential flow fact annotations to machine code used in the
analysis. From control-flow graph and machine code transformed annotations, Platin can
derive a IPET (abbr. Implicit Path Enumeration Technique) formulation, which is a tech-
nique for finding the longest path by exploring every possible path. When combining this
with a given architecture model describing the resource cost executing a given instruction
(clock cycles, in our case), this gives raise to an ILP (abbr. Integer Linear Problem), an
optimization problem aimed at finding the maximum or minimum value, which is solved
by an external ILP solver[2].

Rust support for Wildcat
This chapter will outline the process of implementing support for Rust-based programs
within the Wildcat project, which can be directly linked to the first objective mentioned in
section 1.2. For this chapter, we will be focusing on using the latest stable version of Rust
(1.87.01 at the time of writing).
4.1
Rust to RISC-V
As mentioned in section 3.3.1, Rust supports cross-compilation for a wide variety of built-in
targets. Fortunately for this project, Rust also provides various targets for RISC-V archi-
tectures. In Rust version 1.38.0, the target riscv32i-unknown-none-elf was included
into the built-in targets2, which should be compatible with Wildcat. In section 4.1.1, we
describe how we set up a Rust project targeting RISC-V architecture capable of running
on Wildcat.
4.1.1
Project setup
For this part of the project, we are using the latest stable version of Rust. Therefore,
we can use Rusts built-in system and package manager cargo to set up and manage
our projects as described in the Rust Programming Language book [17, p. 1.3]. Using
this, we set up a new project using cargo new upstream --vcs none ( --vcs none, as
version control is already handled at the root of the Wildcat project). Furthermore, as we
are targeting RISC-V, we configured the project (using .cargo/config.toml3) to build for
riscv32i-unknown-none-elf by default. Moreover, we also provided an argument for
the linker, ensuring the text section of the binary starts at address 0 (as described in the
README of the Wildcat project4).
It should be noted that, while what we have described in this section was used to set up
the upstream project found in the Wildcat fork4, it can be used as a general approach for
setting up new Rust projects targeting RISC-V architecture with the Wildcat architecture
in mind.

4.4
Further Wilcat modifications
As mentioned in the end of section 4.3, we discovered that there were a few fundamen-
tal issues with the three-stage pipeline implementation that would impact the rest of the
project if not addressed. Overall, there were three major issues, which will be discussed
in the following sections:
• Premature updates to the program counter on the initial clock cycle.
• Lack of proper initialization of pipeline stages.
• Lack of proper handling of control hazards.
4.4.1
Premature program counter increments
The first issue is related to unwanted updates of the program counter in the first clock
cycle. This was a known problem, and something that has been addressed both through
comments in the code and the to-do list in the official Wildcat project7. In the code, there al-
ready existed a comment-disabled line, which sets the initial value of the program counter
set to the -4, such that the program counter and next program counter will be set up cor-
rectly from the initial clock cycle. However, this has been changed to 0 due to a few failed
tests.
As we have introduced a dynamic starting address, simply setting the program counter
to 0 will not work. Instead, we have implemented the advice of the comment and set the
initial value of the program counter to the starting address specified in the given ELF file
minus four. In our experience, this works as expected. However, it does not explain why
some tests would fail. Although, we assume that the issue might stem from a lack of
proper initialization of the pipeline stages, which have caused failed tests for us as well.
Therefore, we will address this next.

Control hazards
Finally, the last issue we discovered was lack of proper handling of control hazards. As-
suming we have already fixed the program counter and pipeline initialization issues, the
output still deviated from what we expected. When examining the ThreeCats.scala, we
discovered that the decoded instruction register was not properly flushed, leading to ex-
ecution of instruction.
As an example, we will examine the program of listing 4.4. Once the branch instruction
at address 0x68 is evaluated by the execution stage, the branch instruction of address
at 0x6c would already be loaded into the decode register. On the next clock cycle, the
decode register would be properly flushed. However, the decoded instruction of the last
clock cycle would already be written into the execution register. Therefore, executed on
the next clock cycle, which would lead to undesired behaviour.
In order to fix this, we implemented a extra multiplexer, which would flush the execution
register once the doBranch signal was invoked. The code for this can be seen in listing
4.5. With this update, we could now ensure that the pipeline would behave as expected,
hereby making it possible to execute program such as listing 4.4.

the Wildcat paper[14, p. 2.4]. For now, the branch decision is still included as part of the

Third attempt at ’Hello World!’
After resolving the issues present in section 4.4, we were finally ready for the third at-

program in listing 4.4, it returned an error: java.lang.ArrayIndexOutOfBoundsException:

It is mentioned in the RISC-V ABI that the stack grows downwards[18, p. 2.1], and there-
fore, we assume that there must be an implicit expectation that the stack pointer has
been initialized before executing a program. However, in our bare-metal setup, where we
execute programs directly without any former setup, the stack pointer will most likely be
initialized to 0. This explains the out-of-bounds error, as 0−10 would resolve to 1073741823
for an unsigned 32-bit integer.
For the sake of simplicity, we chose to initialize the stack pointer by rewriting the entry
point function _start8. For this, we used the naked_function crate9, which allowed us
to define assembly directly within our program. Furthermore, we needed a method for
checking successful execution, both to ensure that our program behaves as expected,
and to terminate the hardware simulation. For this, we chose the common approach of
returning 0 from main to indicate success, and use an ecall instruction directly after.
Combining all of this, we wrote a modified ’Hello World!’ program, which can be seen in
listing 4.6.
After building the source code of listing 4.6, and disassembling the corresponding binary,
we are left with the content of listing 4.7. Here, it is most important to note that the initial
instruction at address 0 of the entry point _start initializes the stack pointer as expected.
Furthermore, we see that the second to last instruction at address 0x6c in main sets the
a0 register to 0 as expected, which, after returning to _start, will result in a successful
termination by the ecall instruction at address 0x0c.

It should be noted that the extended clang/LLVM from FAU Erlangen-Nürnberg are forked
from version 7.0.13, which introduces a major issue. Cross-compilation for RISC-V targets

LLVM was also upgraded to version 8 in the same update. As cross-compilation for RISC-
V was essential for Wildcat compatibility, we chose to use the Rust version 1.37.05, as
this was the newest version of Rust with RISC-V target support still using LLVM 8, which
was the closest we could get to the extended clang/LLVM version 7.0.1.
For Platin, we use a fork6 of the main project7 containing Wildcat architecture model for
the three-stage pipeline.

Wildcat architecture model for Platin
Before we can perform WCET analysis using Platin, we need to create a model of the
Wildcat architecture (as seen in figure 5.1). The Platin project already contains models
for other RISC-V based architectures such as Flexpret11 and ESP32C312, which we have
used for the baseline of our Wildcat model13. We have based our design on the three-
stage pipeline design ThreeCats.scala14, and adjusted the estimated cycles cost of each

The most notable difference of the Wildcat model compared to the other RISC-V models
is how branching is estimated. As mentioned in the Wildcat paper[14, p. 2.4] and section
4.4.3, a taken branch will have a cycle cost of three. Therefore, any branch instructions
have been set to a worst-case cost of three, including any pseudo-instructions expanding
to branching instructions (e.g. 'PseudoRET', which expands to 'jalr x0, x1, 0'15).
Furthermore, load and store instructions[11, p. 2.6] have been simplified to a cost of one
cycle, as we will be using a simulation of the Wildcat hardware, where these instructions
are counted as being resolved in a single cycle.
Finally, we have disabled the use of instructions from the RISC-V M extension[11, p. 12],
as these instructions are not implemented for Wildcat yet.

To summarize, the results of this chapter showcase that it is possible to find a fitting WCET
estimation for some Rust programs. However, the introduction of external libraries in-
creases complexity of the WCET analysis, which makes it highly unpredictable. The main
takeaway from this is that when performing WCET analysis for Rust using the Wildcat
architecture, developers should ensure that all dependencies are included in the Rust
source code if possible. This will give the best and most realistic estimates, even coming
across as a promising alternative for, or companion to, C.

Discussion
This chapter discusses challenges which were discovered throughout the working process
of this thesis, which have not been fully resolved or initiated.
7.1
WCET limitations
One of the primary limitations of this project is lacking support for user-annotated control-
flow bounds. Unlike C, Rust does not support pragmas, or any other Rust-native alter-
native for achieving this. This has limited this project to only focus on naturally bound
problems, or problems where bounds would be inferred by the compiler. This has greatly
weakened our ability to compare WCET analyses of Rust programs to similar programs
written in already-supported languages like C.
This of course begs the question if this project has managed to successfully fulfil the ob-
jective of facilitating support for WCET analysis of Rust-based programs using the Wild-
cat architecture. Here, we argue that while lack of user-annotated control-flow bounds
imposes significant drawbacks on the amount WCET analysable programs, this project
demonstrates how is it possible to perform WCET analysis for Rust-based programs. Fi-
nally, as this is of most importance, it will be further discussed in chapter 9.
7.2
Analysis of (nested) library dependencies
In addition to the limitations for WCET analysis on Rust, we have also noticed programs
that rely on built-in library functions usually provided at link-time poses a rather tricky
challenge when using Platin for WCET analysis. Throughout this project, we have been
comparing WCET estimates to hardware simulations for certain programs, and here, we
have found that the WCET estimates in some cases are lower than the measured cy-
cle count. The problem stems from Platin’s approximate cost of library functions like
__mulsi3, __umodsi3 and __memset. As these functions are (in all likelihood) not defined
within the original source file, Platin is unable to access them in their original forms. As
these library functions are quite common, e.g. used in multiplication or arrays in Rust, the
architecture models instead implement an estimated cycle cost of the entire function call.
For this project, this becomes especially tricky, since the library functions for Rust provided
by compiler_builtins.o1 for the RISC-V targets tend to be quite complex and suffer
from nesting. This makes it hard to estimate a realistic cycle cost for these functions,
which can lead to both severely underestimating or heavily overestimating WCET bounds.
Furthermore, as the three-stage pipeline implementation of Wildcat is a brand new and
norm-breaking design, we cannot safely rely on existing estimates.
To make matters worse, the library function for riscv32imc-unknown-none-elf in Rust
version 1.37.0 include instructions from both the M- and C-extensions, neither of which are

supported by Wildcat. To accommodate for the latter problem, we have tried extracting the
compiler_builtins.o from riscv32i-unknown-none-elf of Rust version 1.38.0. While
this appears to work as expected, it adds an extra layer of complexity and vulnerability
to the project. To summarize this section, we consider handling and support for external
libraries as being out of scope for this project.
7.3
Compatibility
Another undesirable feature of this project is the fragile foundation of relying on potentially
non-compatible versions of LLVM, as mentioned in section 5.1.2. While this fortunately
has not been an issue during this project, we assume that it is highly probable that issues
will arise from this incompatibility. We have experimented with upgrading to Rust version
1.38.0 to resolve the problem mentioned in section 7.2, but we found that this version of
Rust, built upon LLVM 9, is incompatible with the extended versions of LLVM and Clang.
As the current setup of Rust version 1.37.0 has not caused any significant problems, it
would definitely be preferable to upgrade to a more stable build in future.
Overall, this project would have benefitted from focusing on upgrading the extended ver-
sion of LLVM and Clang to a newer version, which is more compatible with both RISC-V
and Rust. However, this would have significantly changed the course and objectives,
considering the original objectives of the project, as upgrading to a new LLVM version
would be a cumbersome process.

Conclusion
In section 1.2, we presented the original objective for this project, which consisted of
providing Rust support for Wildcat project, and facilitating support for WCET analysis of
Rust-based applications using the Wildcat architecture. Throughout the chapters of this
report, we review the processes of how these objectives were achieved, by showcasing
small examples of progress throughout the entire project.
In chapters 2 and 3, we assessed the current state of the fields, drawing parallels to related
projects, and giving the reader a brief introduction to underlying concepts of the project.
This acts as a primer, preparing the reader for what is to come in the following chapters.
In chapter 4, we addressed the first objective by exploring how Rust support could be
implemented for Wildcat using the latest stable version of Rust, relying purely on the
Rust ecosystem for integration. Here, we showcased how to set up required tools for
cross-compiling, developing and executing bare-metal Rust on the Wildcat architecture.
In addition to this, this chapter also addressed a few architectural and technical issues
in the Wildcat project, which were handled in order to ensure full support for the Rust
language.

project, how the Platin WCET tool would be used to perform WCET analysis for program,

tool-chain capable of both facilitating support for WCET analysis for Rust programs, while
maintaining the ability to execute these programs using Wildcat. Through a delicate pro-
cess picking and tinkering with various versions of LLVM, RISC-V GNU, and Rust, we
were able to fine-tune a model, which worked for most applications, while simultaneously
developing a Docker setup, to spare others from doing it by hand in future.
In chapter 6, we demonstrated what the most recent version is capable of, showcasing
both the positive and less positive results for this project. Following up on this in chapter

9
Future work
This chapter presents the hopes and dreams for features that was not realised as part of
this project, but hopefully can become a reality in the future.
9.1
Control-flow information in Rust
The first and most important feature which should be realised is implementing support for
user-annotated flow fact annotations in Rust programs. The value that this would bring
would elevate this project, as it allow us to analyse programs which are not naturally
bounded or for which the compiler cannot infer the required bounds.
Furthermore, this would also make it possible to better compare WCET estimates for
programs written in Rust with equivalent programs written in C/C++. In addition to this,
this would possibly alloow for development of a set WCET benchmark tests like The
Mälardalen WCET Benchmarks[19] or TACLeBench[20] written in Rust (assuming Rust
proves to match C/C++).
Our initial considerations were to explore the possibility of using attributes like #[inline]
and #[no_mangle] to target loop-specific keywords like for and while using an attribute
like #[loopbound min 0 max 10] for facilitation user-annotated loop bounds. However,
thi is only a pre-assumption that this could work.
9.2
Tool-chain improvements
Secondly, we would like to improve upon our custom tool-chain by making it more robust.
As mentioned in section 7.3, relying on two different versions of LLVM is may cause issues.
In addition to this, as mentioned in section 7.2, Rust 1.37.0 only has stable support for a
RISC-V target which includes the M- and C-extensions of the base RV32I instruction set,
which are not currently support by the Wildcat project. However, in order to upgrade to
newer version, the underlying tools should also be upgraded, which will likely be a tedious
and complex task. Nevertheless, it would be worthwhile to upgrade, both for the stabilized
of the project and the extended Rust support.
9.3
Further support for Wildcat
Over time, it would also be relevant to start developing more Rust infrastructure specific for
the Wildcat project. A suitable place to start would be implementing a HAL (abbr. Hard-
ware Abstraction Layer) crate with reusable components, which would greatly improve
the development process for Rust software targeting the Wildcat processor. However, for
now, we consider this to be a quality-of-life improvement, which is not as important as the
features presented in sections 9.1 and 9.2.
