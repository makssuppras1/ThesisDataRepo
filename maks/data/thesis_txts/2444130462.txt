Abstract
The main focus of this project is to propose a solution enabling a robot to robustly
navigate a chosen forest path utilizing computer vision. The robot should be able to
navigate the entire path, excluding public roads. While autonomously navigating the
path, the solution should enable the robot to avoid obstacles such as pedestrians while
also providing means to stop the robot when approaching public roads.
A deep learning approach is used, providing the solution with a semantic segmenta-
tion of the acquired images, splitting these into path and background. The segmented
images are processed using classical image analysis in order to provide the robot with
a driving direction, making it follow the path. A GPS based point of interest system
has also been added, enabling the robot to stop when approaching traffic, while also
providing a way of e.g changing parameters when driving a particularly difficult path
segment.
Through testing, the developed system has been successfully evaluated and the sug-
gested solution was found to fulfill the project goals.

CHAPTER 1
Introduction
Research in the field of autonomous robotics is becoming increasingly important, as the
demand for solutions such as self operating vehicles and manufacturing equipment is
rising.
In the area of self operating or autonomous vehicles, an important topic is the navi-
gation of the environment around the vehicle, especially if the environment is outdoors.
In an outdoor environment there are several factors which complicate a navigation task,
such as changing weather and lighting conditions.
Another important topic for autonomous vehicles is collision avoidance, as an au-
tonomous vehicle has to be able to avoid collisions with objects in the environment.
In an outdoor setting, this is difficult due to non static objects such as other moving
vehicles and pedestrians.
First of all, an important task when considering autonomous driving is the task of
finding the road on which the vehicle should drive. Finding the right path can be done
using various sensors, such as RGB or RGB-D cameras, GPS, radar or laser range finders.
Most self-driving vehicles utilize a combination of the above sensors in order to construct
an adequate perception of the surroundings.
The main focus of this project is to enable a robot to drive a chosen path au-
tonomously using vision based navigation. By using computer vision, the robot will
be able to find and navigate the road while avoiding static obstacles.
1.1
Objectives
In this section, the objectives of the project are presented. The objectives have been
chosen such that, if they are fulfilled, the chosen vehicle should be capable of robustly
navigating the environment around a predefined path.
The robot should be able to:
• Detect the road on which it drives on.
• Select a path to drive from the detected road.
• Avoid static obstacles such as humans standing in the way.
• Stop driving when approaching a traffic heavy road.

2
1 Introduction
• Autonomously drive the entire path chosen, excluding public roads.
1.2
Chapters overview
In this section, an overview of the chapters found in this project is presented. In the
chapters, the analysis of the problem is described, together with a theoretical description
of the different components of the solution. Finally a system test is performed to ensure
operation within the objectives.
• In chapter 2, an analysis of the problem and the objectives is conducted. Fur-
thermore, a choice of vehicle and general system structure is made. Related work
in the field is included in the problem analysis.
• In chapter 3, the acquisition of images using a camera is presented. This includes
choice of resolution, brightness correction and timely capturing of frames from the
camera.
• Chapter 4 discusses the problem of finding the path in the images taken with
the camera. A neural network is trained to robustly segment images of the path
chosen to drive on. The chapter includes the structure of the network, together
with optimization for a better segmentation.
• In chapter 5, the line extraction module is introduced. This module handles the
extraction of a driving direction for the robot, from the images segmented by the
neural network. This includes image transformation and information condensation
using relevant image analysis methods.
• Chapter 6 describes the GPS Module designed for handling the interrupt-like
event needed when e.g. stopping at a road. This includes translation of the raw
GPS data from the GPS antenna, along with a description of the different GPS
events used.
• In chapter 7, a thorough testing of the entire system is conducted. The testing
is split up into several sub tests, each testing a relevant part of the system, along
with an overall test of the robots performance when driving autonomously.
• Chapter 8 and 9 contains a proposal for future work which could be added to
the system, along with a conclusion of the performance of the designed solution as
a whole.

CHAPTER 2
Analysis
To be able to achieve the objectives presented in chapter 1, several subsystems will need
to be combined. On the hardware side, a suitable platform is to be selected among
the available vehicles at the automation and control department at DTU Lyngby, along
with suitable sensors for the achievement of the chosen objectives. On the software side,
an analysis conducted in this chapter will use information gained from recent articles
published in the field of autonomous vehicle navigation to decide on an approach to
solving the task of finding and navigating roads and/or paths.
2.1
Route
A fitting stretch of road has to be determined in order to be able to fulfill the objectives of
the project. At DTU Lyngby, there is a running path which encircles the entire campus.
The path is approximately 5 km long, and consists of gravel path with varying amounts
of vegetation and trees around it. There have been other attempts at navigating this
road autonomously[3], which could serve as benchmarks for this project. Furthermore,
this route is a mostly controlled environment with the only obstacles being pedestrians.
Therefore, the running path is chosen as the route to be navigated in this project.
2.2
Vehicles

Related work
Autonomously navigating vehicles are becoming an increasingly large area of research,
with a lot of effort going into self-driving cars. It is therefore relevant to analyze recent

research in this area before deciding on an architecture for the software system in this

One of the more interesting recent developments in the field, is using a camera as
the primary sensor for the problem of determining the position of a given road or path
in relation to the autonomous vehicle. The use of cameras has been made possible by
the emergence of powerful commercially available graphical processing units which are
able to process all the information in the images in such a way, that it is possible to
be used in applications running in real time on a self driving vehicle. For self driving
vehicles, the task of finding the road on which the vehicle should drive is known as a
segmentation problem in the field of image analysis.
The problem of segmenting a road from the background in an image can be solved
using semantic segmentation, where each pixel in an image is given a specific label, from
which information can be extracted. An example of an advanced semantic segmentation
can be seen in figure 2.4. In the figure, several images of roadside scenes are segmented
into different classes. In this case, red pixels in the output image corresponds to roads,
yellow to sidewalks, purple to cars, cyan to pedestrians, blue to buildings, green to trees
and white to the sky.
Figure 2.4: Example of semantic segmentation of an image. From Deep residual coalesced
convolutional network efficient semantic road segmentation[4]
In the simplest case of road segmentation, only two classes are needed, namely road
and non-road. An example of a solution to this segmentation problem is proposed in
the paper Efficient Deep Models for Monocular Road Segmentation[9]. An example of
their results is shown in figure 2.5.

Much of the recent research in road segmentation revolves around using convolutional
neural networks for segmenting the road from the background, see [4], [20] and [9],
although other methods such as Markov random fields and Bayesian methods have been
tried as well, as in [13] and [11].
The approach in Deep residual coalesced convolutional network for efficient semantic
road segmentation[4], relies on coalescing convolutional layers from different types of
network into a so-called residual coalesced convolutional network (RCC-net), which is
more efficient computation-wise than previous attempts at semantic road side image
segmentation.
In this article, they achieve good results on a very efficient network
layout, and are able to do a forward pass in 25.5 ms on a GTX 1080 graphical processing
unit.
In Distantly Supervised Road Segmentation[20], Tsutsui et al. describe a segmentation
problem much like the one required for this project. Their approach to the solution is
to reduce the work of doing pixel-wise annotation of images, as it can be a very time
consuming process, especially when the training images need to be used in a setting with
other road users. Tsutsui et al. therefore propose a method called weakly supervised
segmentation, which uses training sets with image-level labels from publicly available
data sets, reducing the amount of work needed for annotating images. They achieve
comparable results to traditionally annotated images using this method.
Another efficient implementation of convolutional neural network for monocular road
segmentation is proposed in Efficient Deep Models for Monocular Road Segmentation[9].
Here, Oliveira et al. puts forward a network architecture based on fully convolutional
layers, as these are very efficient computation-wise. The network includes layers from
the VGG-network[25], which is a widely used classification convolutional neural net-
work. Using only convolutional layers, they achieve a very efficient implementation of
a monocular road segmentation network, getting forward pass times usable in real time
applications.
Due to the results of the research presented here, an approach for segmenting the
running path at the DTU Lyngby campus using a single camera and a neural network
seems feasible. For this approach to be successful, a suitable architecture for the network
needs to be determined. The architecture used in Efficient Deep Models for Monocular
Road Segmentation[9], looks very promising, but to be thorough in the analysis a more
general semantic segmentation architecture should be considered.

In order to select a driving direction for the robot on a given path, a computer

Such a

3.1
Camera setup on the iRobot ATRV-Jr
During the initial testing of this camera after mounting it on the iRobot ATRV-Jr, it
was found that the built-in auto focus functionality of the camera was activated when
driving over small bumps on the path or other obstacles such as twigs. This led to the
images being unfocused until the auto focus self-corrected. Therefore, it was decided to
manually set the focus point at infinity using the V4L drive API in Linux, as focusing
on objects close to the lens is not needed in this project.
Furthermore, it became apparent that the brightness correction built into the camera
was insufficient for the purpose of imaging the path in varying outdoor lighting conditions.
In some cases, the brightness correction increased the exposure so much that parts of
path became overexposed. An example of this is presented in figure 3.2.
Figure 3.2: Overexposed path images using the built-in brightness correction.
Therefore, a solution for controlling the brightness of the images was designed. This
solution utilizes functionality of the V4L camera API for Linux, enabling direct control
over the exposure of the camera, which in turn controls the brightness of the images
taken. A small Python thread was written for the purpose of controlling the exposure
of the camera, based on the grayscale mean intensity of the images taken. The thread
simply in- or decrements the exposure of the camera directly, based on the grayscale
mean intensity of the previous frame taken by the camera. This gives a high degree of
control over the brightness in the image compared to the built-in auto exposure setting
on the camera.
A module test of this system with comparisons to the original lighting correction
system can be found in section 3.2. The Source code can be found in appendix B.5.
To minimize the computation time of the forward pass on the neural network and
the following image analysis, the resolution of the images taken needs to be as small as
possible, as the main contributor to computation time is the number of pixels. When
testing, 128 × 128 px was found to be a good resolution, as this provides a good trade-off
in terms of image size and details retained in the images.

CHAPTER 4
Segmentation
Image segmentation is the process of dividing an image into several regions, which can
then each be assigned a label. This makes further analysis easier. One segmentation
task could be to segment an image of a house into two segments: house and background.
Further segmentation can then be done, segmenting the house into facade, door, windows
and so on. Segmenting an image into different regions does not necessarily assign any
meaning to the individual regions. An algorithm might be able to segment a house from
the background without assigning the meaning that the foreground is a house. Semantic
segmentation is a type of segmentation problem in which, instead of each region being
separated, each pixel is assigned to a class. Using semantic segmentation an algorithm
will be able to tell which pixels make up the house and which make up the background.
The process of semantic segmentation is vital for this project, as it enables the robot
to determine which parts of an image makes up the path that should be followed.

4.1
The U-Net Architecture
As stated in chapter 2, it has been decided to use a neural network based on the U-
Net architecture for segmenting the images. The U-Net architecture is comprised of
several layers, which are connected using operations known as convolutions. In order to
understand this architecture, it is beneficial to introduce the basic layers from which it
is built, this will be done in the following sections.
4.1.1
Network Layer Types
4.1.1.1
Convolutional Layer
A convolutional layer is able to extract features from an image [10]. It works by con-
volving a filter kernel of a chosen size with an image, generating an output feature. The
network will be able to learn the values contained in the filter kernel based on training
images in the training set. Usually, many kernels are trained yielding features which can
be sent further into the network. Normally, multiple convolutional layers are arranged
in series, generating more specific features the deeper into the network the layers are.
A convolutional layer has some relevant parameters known as stride, kernel size,
padding and number of filters. Stride is the amount the kernel should move between
each step in the convolution. A stride of 2 would mean that the center of the kernel
will move two pixels per step. Kernel size indicates the size of the filter which is moved
across the input. A kernel size of 3 would give a 3 × 3 px filter.
Padding pixels can be generated around the edges of the input, such that the filter
kernel is able to start the convolution at the edge of the input. The value of the padding
pixels can be chosen to be any value.
The number of filters determines the amount of filters that should be trained for a
given convolutional layer.
An illustration of a convolution can be seen in figure 4.2, where a 3 × 3 kernel using
padding and a stride of 2 is convoluted over a input, yielding an output feature. Here, the
blue square represents the input with the white pixels around it being the padding. The
gray square laying on the input is the kernel being used and the green square represents
the output.

Max Pooling Layer
Max pooling is often used together with convolutional layers [10]. In the max pooling
layer, a kernel is shifted over the input layer. For each step during shifting, the maximum
value under the kernel is chosen and added to the output of the layer. Doing so reduces
the dimensionality of the input layer while retaining the maximum response from the
input feature, as can be seen in figure 4.3.

4.1.1.3
Deconvolutional Layer
A deconvolutional layer, also called a transposed convolutional layer, upsamples a con-
voluted input to its original size [26]. The operation is not the same as the pure mathe-
matical deconvolution, which would restore the input to its original form. A transposed
convolution upsamples the input while maintaining the information gained from the
convolutional layer. Transposed convolution works by zero-padding around each data
point in the input followed by a standard convolution using an appropriate kernel size.
An example of this can be seen in figure 4.5.

An activation layer simply takes an input and applies a non-linear function to that input.

4.1.2
Architecture
Now that the basic layer types have been presented, the U-Net architecture can be
properly introduced.
The network consists of two parts, a contracting part and an expanding part. An
illustration of the U-Net architecture is shown in figure 4.7, where the contracting part
is on the left side of the ”U” shape and the expanding part is on the right.
==PAGE:41==
26
4 Segmentation
The contracting part follows a typical convolutional network structure, having blocks
of two 3 × 3 convolutional layers with ReLU activation followed by a 2 × 2 max pooling
layer with stride 2 for downsampling. The outputs of the blocks are saved for later use
in the expanding part. After each downsampling block, the number of filters in the
convolutional layers is doubled.
The steps in the expanding part consist of a transposed convolutional layer, upsam-
pling the features from the input. The upsampled feature map is then concatenated
with the corresponding saved output from the contracting part. This is followed by two
convolutional layers with ReLU activation. For each step in the expanding part, the
number of filters in the convolutions is halved.
Finally, a 1×1 convolutional layer is used to map the resulting features to one of two
classes. The classes, in our case, being 1 for path and 0 for background. The final layer
uses a sigmoid activation, as a large feature response will be activated to 1, while a large
negative feature response will be activated to 0, making binary segmentation possible.
Figure 4.7: The U-Net architecture[17]
Note that in figure 4.7 the output size of the network is smaller than that of the
input. This is because the convolutions in the original U-Net are done without padding.
In our case, padding is added so that the output size matches that of the input. The
padding used in this project has a value equal to that of the nearest pixel in the image.

4.2.3
Measures of performance
When training a neural network, the computer measures the performance of the current
iteration of the network by looking at a given error function, also called the loss function.
Optimal performance is then obtained by minimizing this loss function. The loss function
is defined by the user and can vary from task to task.

the loss function.

4.3
Experimentation
A big part of building a neural network is experimentation. When training a network,
there is no general optimal parameters, as these vary a lot between different data sets.
In this section experiments are carried out in order to try and optimize the network.
As there are many parameters to test, we will not be able to find an optimal network,
though it might be possible to improve on the base U-Net suggested in the article ”U-Net:
Convolutional Networks for Biomedical Image Segmentation” [17].
The base U-Net has been implemented in Python 3.5 using Keras. Keras is a high-
level neural networking API, which makes implementation and experimentation easier.
The network is trained using a NVIDIA GeForce GTX 1080 GPU. As outlined in chapter
3, the data images are resized to 128 × 128 px before being fed into the network, as this
was the lowest possible resolution in which enough details were retained in the images.
Keras is able to use different backends. A backend is the library from which the API
builds the network. Different backends have different optimizations for training and
inference speeds. Inference being the process of feeding an input image to the network
and obtaining an output. Some commonly used backends are Tensorflow, Theano, CNTK
and MXNet. As a pretest, the Tensorflow and MXNet backends were tested against each
other in order to investigate which one to use. During this test, MXNet was found to
do inference around 40% faster than Tensorflow while having identical performance.
Therefore MXNet is used as backend for further use.
When conducting experiments, it is important to evaluate the network on a validation
set. A validation set is an annotated data set consisting of data points that are not used
during training. The trained network is evaluated on a validation set in order to estimate
the amount of overfitting, which is when a model is exceptionally good at predicting the
outcome of the data on which it has been trained, but performs significantly worse when
predicting the outcome of new data. As the network will be used to segment images
taken during operation, it has to perform well on new data. In this case, the validation
set has been obtained by randomly selecting 10% of the annotated images in the data
set. This results in a training set with 7811 images and a validation set with 868 images.
In order to enable comparison of networks, the same training and validation set is used
throughout the experimentations.

Dropout
One way to decrease overfitting, and therefore hopefully decreasing the validation loss,
is to add dropout layers to the network [2]. A dropout layer works by randomly setting
a fraction of the inputs to 0. Doing this turns off parts of the network for each image.
This results in a network where the weights are all trained on an unique random subset
of the data. Thereby reducing overfitting.
As an experiment, a model with dropout layers was trained for 50 epochs.
The
dropout layers were added in between each set of convolutional layers throughout the
network. The dropout rate was set between 0.1 and 0.3, depending on where the layer
was placed in the network. A dropout layer near the output of the network had a low
dropout rate, as it is not desired to remove a lot of information right before the output.
A plot of the resulting losses and scores can be seen in figure 4.15. It is seen that
the validation loss seems more stable and less prone to start increasing down the line,
suggesting that dropout indeed reduces overfitting. Furthermore, if looking at the results
presented in table 4.5, it is seen that the lowest validation error achieved using dropout
is significantly lower than without. The inference time was not affected much, which is

If the true test error of a network trained using all data was to be known, K-fold cross
validation could be used to validate the networks performance. K-fold cross validation
is a process used to estimate the generalization error of a model.
The difference between the validation error and the generalization error is that the
validation error is found when evaluating a network on a validation set. The problem
with validation sets is, that it is hard to know if a given validation set is a particular
easy or a particular difficult set, due to the fact that it is randomly chosen from the
entire data set. The validation error can therefore be misleading.
The generalization error, on the other hand, is the true error rate of a network. Unlike
the validation error, the generalization error is the error rate of a model when evaluated
on all available data. The problem here is, that the available data must be used to train
the model, therefore defeating its use for evaluation. K-fold cross validation seeks to
go around this problem by splitting the data into a number of subsets called folds, for
example 10. After splitting the data, a model is trained using 9 of the 10 folds and
validated on the remaining fold. The resulting validation error is kept and the model is
trained on another selection of 9 new folds, and so on. In this way, the whole data set is
eventually used for validation, but based on several different trained network using the
same architecture. Finally, the mean of the, in this case, 10 saved validation errors is
found, and this is then used as an estimate of the generalization error. It is important to
note that K-fold cross validation does not yield a trained network but only the estimated
generalization error of a given network structure.
As a model using 200,000 augmented data points was found to take ∼2500 s/epoch
to train, it would take roughly two weeks to estimate the generalization error of the final
model using 10-fold cross validation. Therefore, the usage of K-fold cross validation for
estimating the generalization error of the network designed was decided against, as real
life performance would be a more relevant evaluation. For the purpose of this project, a
field test using the network on new images taken during run time was used instead to
validate the networks performance.

CHAPTER 5
Line Extraction
The process of segmenting the images as described in chapter 4 condenses all the in-
formation in the input image into the relevant information to be able to navigate the
path, which is the pixel-wise location of the path in the image. This however, is not
enough for the robot to correctly follow the path, as the information contained in the
image so far is in the coordinate space of the camera, and not the robot. Furthermore,
the information about the pixel-wise placement of the path in the image needs to be
transformed into something understandable by the robot. The iRobot ATRV-Jr uses the
mobotware platform in a Linux environment. To make the robot drive using mobotware,
a line needs to be passed to the robot, which corresponds to a desired course for the
robot.
This chapter will describe how the path information in the images will be transformed
to conform to the commands used by the iRobot ATRV-Jr. Furthermore, tests will be
conducted to determine the optimal solution in terms of robustness and computation
time.
The general strategy of the line extraction is outlined below:
• Transform the segmented image to the robot’s coordinate system using a homog-
raphy.
• Find contours in the transformed image.
• Clean up the contours found.
• Estimate the right and left edges of the cleaned contour.
• Estimate line based on the edges found.
• Send either the right or left line found to the robot.

driving the robot along a path in a smooth manner.

commands from a program running in a loop. This makes driveon the most compatible
command of the options in SMR-CL.
The program implementing the SMR-CL commands, and thereby controlling the
robot, is called MRC. MRC can be started as a socket server on a chosen local port. A
Python script can thereby connect to the socket server in order to both issue commands

5.2
Edge detection
Once a segmented image in the proper coordinate system is obtained, the next step in
the computer vision algorithm is the condensing of information in the image into what
is relevant for the operation of the iRobot ATRV-Jr.
As the commands for the robot are lines it has to follow, a method for extracting
lines from the segmented path is needed. One method for finding edges in an image is
by convoluting the image with a laplacian kernel, thus differentiating the image.
Convolving with the laplacian is often used for finding edges in non binary images
where the edges are not as pronounced as in binary images, and yields a so called edge
response image, where the pixels containing edges have a high response.
The edge
response image can then be thresholded, giving a binary image containing the edges.
The kernels used for this method are very sensitive to noise and by thresholding, the
resulting edges are often split into several pieces even though they belong to the same
edge in the original image.
As the images are already binary and has a small amount of noise at this point
in the computer vision algorithm, a better suited method for finding the edges of the
segmented image is the contour search algorithm.
5.2.1
Contour search
The contour search algorithm is a simple method for obtaining the edge pixels of objects
in a binary image. I works by simply iterating through all pixels in a binary image,
starting from e.g. the top left pixel. The iteration is done in a line by line manner
until a pixel with a value of 1 is found. Then another search is conducted by looking
at neighbouring pixels in e.g. an eight-connected manner, starting from the neighbour
pixel 90 degrees to the left of the direction from the previous pixel to the found pixel,
then 45 degrees, then 0 and so on. If another object pixel is found, this gets added to
the current contour and the neighbour search starts over from this pixel. This continues
==PAGE:73==
58
5 Line Extraction
until the starting pixel in the contour is reached again, or until a maximum number of
iterations have passed.
Then, the line by line search continues from where the neighbour search started,
but does not take pixels into account that has already been assigned to a contour. An
overview of this search is shown in figure 5.5.

5.2.2
Contour cleanup and approximation
Although the segmented image used as input already has a low amount of noise due
to the being trained very well, steps have to be taken to handle cases where noise is
present in the images. Also, in order to decrease the computation time of the following
steps in the algorithm, a method of reducing the dimensionality of the found contours
is relevant.
The path in the segmented image is expected to be large in relation to the total size
of the image, so removing very small contours from the output of the contour search
algorithm is a straightforward way of reducing noise. Other sources of noise, such as the
shape of the segmentation differing from the true shape of the path is expected to have
minimal influence, or be reduced by other steps in the algorithm, so noise removal other
than removing small contours is not done at this point.
To lighten the computational load of the following steps in the algorithm, the number
of points in the found contours needs to be reduced as the raw contours can be comprised
of several hundreds of points. By usage of the function approxPolyDP(), the number
of points in the contours can be reduced by at least an order of magnitude. It works by
usage of the Ramer-Douglas-Peucker algorithm[18][6].

Another situation often encountered on the running path at the DTU Lyngby campus

segment the running path itself, places where e.g. a sidewalk is crossing the running
path will lead to the neural network segmenting the image into to two separate segments

This will lead to the contour search finding two separate contours.
Defining the
left and right edge for each contour separately will lead to the parts of the contours
corresponding to the edge between the running path and the crossing path to also be
defined as either a right or left edge, leading to ill defined edges, and therefore erratic
behaviour of the robot. Due to the specificity of this particular situation, a general
method for handling is situation does not exist.
Therefore, an ad hoc algorithm is
proposed for the situation where two contours are found, using the following steps:
• For each point q1i in the first contour, find the point q2i,min in the second contour
which has the shortest euclidean distance to q1i.
• For each point q2j in the second contour, find the point q1j,min in the first contour
which has the shortest euclidean distance to q2j.
• Check if more than one pair exists, where q2i,min = q2j and q1j,min = q1i, i.e. a pair
of points from each contour where the points have the shortest distance to each
other.
• Find the right- and leftmost of these pairs by sorting them based on their x-
coordinates in the image space.
• Concatenate the two contours into a single contour by removing all points between
the right- and leftmost pairs and concatenating the remaining points into a single
list of points.

tions found for the test line, as this can be used to deduce which intersection belongs to
the right and left edge of the contour. The script can handle cases where two, three and
four intersections are found, but theoretically cases where even more intersections are
found could be handled. However, more than four intersections are rarely found when
driving the iRobot ATRV-Jr on the running path at DTU Lyngby and handling of these
rare occurrences is therefore not implemented.
The intersections are handled as follows
• If two intersections are found for the test line, the intersections are appended to
the lists contained the left and right edge estimates based on their x-coordinates
in the image space, see figure 5.14.
• If three intersections are found, the pixels on the segmented image between the first
and second, and second and third intersections are checked. If the checked pixel
belongs to the background, the opposite pair of intersections is used for the edges,
with right and left being assigned to the intersections based on their x-coordinate
in the image space.
The two situations are shown in figure 5.15 and 5.16.
If
none of the pixels checked belongs to the background, the pair of intersections
corresponding to the widest path is used.
• If four intersections are found, the first and second intersection in the sorted list
of intersections are used, if the difference in x-coordinates is greater than the
difference in x-coordinates of the third and fourth intersection, i.e. the widest

5.3
Line estimation
Once a good estimate of the left and right edges of the contour corresponding to the
segmented path is obtained, estimation of a line for the robot to follow can commence.
There already exists several robust line estimation techniques, such as regression, random
sampling consensus (RANSAC) and least squares estimation, so there is no need to
reinvent the wheel for this task.
In the Numpy[16] library for Python, a polynomial fitting function is implemented in
the function numpy.polyfit(). This is a very general fitting tool, able to fit any order
polynomial to a set of data points, however it also works well when used to estimate
lines. It simply takes the data points and a polynomial order as input, and outputs the
estimated coefficients. The estimation is done by using a least squares approach.
In order to make sure that there is room for the robot when driving on the path, the
x-coordinates of the edge estimates are moved a set distance, d, towards the center of
the path before sending them to the polyfit()-function, see figure 5.19.
==PAGE:85==

Figure 5.19: Graphical overview of the line estimation.
Figure 5.19 presents an overview of how the line estimation is done. The estimate
of the right edge is shown as a solid green line, while the estimate of the left edge is
shown as a solid red line. These lines are moved, as shown by the dotted lines, before
line estimation takes place. The resulting estimated lines are shown in dark green and
dark red for the right and left edges, respectively.
To ensure stability of the estimated lines when running the script on the iRobot
ATRV-Jr, certain measures have to be taken. To avoid erratic movement of the line
estimated based on e.g. noise in the segmentation, a running average of the five latest
estimates is used instead of the estimates one by one. Furthermore, line estimates with
a large change in slope compared to the previous estimate are rejected, and not included
in the running average.
The behaviour of the driveon command in mobotware is to drive in a curve towards
the line received, eventually becoming coincident with the line. In tight corners, this
can become a problem as the lines estimated here will become almost horizontal. As the
robot tries to get close to the line as well as following it when using driveon, this leads
to the robot turning in the opposite direction of the corner, which in turn leads to the
robot losing sight of the path.
Therefore, the line sent to the robot is a line going through the center of the robot,
which intersects with either the right or left line estimate at a set look ahead distance
from the robot. Which of the right or left lines is used as the end point, can be chosen
before running the script, but if the chosen line estimate does not exist the other line
estimate will be used instead. An overview of this can be seen in figure 5.20. Here, the
actual line sent to the robot is shown in blue. The look ahead distance, dl, is denoted
with a double arrow. For the robot to be able to follow a line, the line needs to be
explained by a coordinate and a direction. Here, the coordinate used is the current
robot odometry position, as this coincides with the center of the robot. The direction
for the line is the angle of the desired direction relative to the current robot yaw.
==PAGE:86==

71
Figure 5.20: The actual line sent to the robot.
To further ensure robust operation of the robot, a couple of functionalities are added
before generating the line which is sent to the robot. These are:
• If the line which the robot has been set to follow in the script is found by the
algorithm, a point at a distance controlled by the lookfwd variable is found on this
line, and a line from the center of the robot to this point is sent to the robot.
• If the line selected in the script is not found, but the opposite one is, the robot is
set to follow this instead.
• If the angle between the two estimated lines is too great, or the angle between the
image y-axis and the estimated line is too great, the line sent to the robot is a line
pointing straight ahead instead.
• If no lines are found, the line sent to the robot in the previous loop is reused.
• To increase maneuverability, the velocity of the robot is decreased proportionally
to the steering angle, making the robot go slower in sharp turns.
The behaviour of the line estimation module is tested in section 5.4.

CHAPTER 6
GPS module
In this chapter, the GPS module of the proposed solution is described. GPS is used in
order to track the global position of the robot. In this project, GPS is not used for path
following, but rather for keeping track of points of interest (POI).
The GPS antenna used is a Navilock NL-8004U USB[15]. The antenna uses the
UBlox 8 UBX-M8030-KT chipset and has a claimed accuracy of 2.5 m. GPS data is
obtained through USB utilizing the NMEA 0183 protocol.
Figure 6.1: The Navilock NL-8004U antenna.
The NMEA 0183 protocol broadcasts various information over the USB connection.
The information comes as lines of ASCII encoded data, where each line contains different
types of information. Each line has a key which tells what the upcoming information
is about. Two types of information is used in the GPS module of this project, fix and
position. Fix has the key ”GNGSA” and is a measure of the quality of the satellite fix.
A fix of 0 is bad, and no position information can be determined, while a maximum fix
of 3 ensures a good position reading. Position has the key ”GNGGA” and contains the
current measured latitude and longitude of the GPS antenna. As the antenna broadcasts
single lines at the time, the GPS module runs in a different thread, constantly checking
for the used keywords. When a relevant keyword is found, the information is stored in
a variable that the main program can access when needed.
As mentioned, the GPS module is not used for path following but rather for knowing
if a point of interest (POI) is near. A point of interest could for example be a civilian
road where the robot should stop before crossing, and be driven manually. Another point
==PAGE:95==
80
6 GPS module
of interest could be a difficult path crossing, where a simple odometry based SMR-CL
operation should be carried out instead of using the neural network based algorithm. The
different points of interest are stored in a text file which is read and stored i memory
during start up of the main program.
Each point of interest has a number of flags
associated, which explains the action needed at this particular POI. The flags used are:
• Vel is used for changing the velocity of the robot from the standard of 1 m/s.
• Edge is used if the robot should follow the left side of a path instead of the
standard right side.
• Odo is used if the robot should perform a driveon command based on odometry
alone. The value of this flag determines the distance driven in meters.
• Stop is used for stopping the robot e.g before a road.
• EdgeOffset determines the distance the robot should drive from the edge of the
path. Standard is 0.8 m
• LookForward changes the look forward distance used to calculate the angle be-
tween the robot and the approximated offset edge line. Standard is 1.5 m
• SaveYaw is used to save the current yaw of the robot. This operation is used to
calibrate the direction driven when driving odometry based.
• Turn is used if the robot should turn a certain amount relative to the saved yaw
when driving odometry based.
• Hairpin switches of a sanity check for the found lines that otherwise would cause
the robot to drive straight forward when encountering extremely sharp turns.
The use of these flags enables the robot to perform a predetermined operation in
places where the standard procedure would otherwise fail.
A such place could be a
crossing sidewalk which is too wide for the robot to see the path on the other side.
Furthermore it enables the robot to stop before a road, such that it can be driven across
manually as a safety measure.

7.11
Loop time
In this test, the computational time of the solution was to be found. During testing,
the main loop of the script was timed using the Python function time.time(). The
timing of the loop was done with everything running as to get a true estimate. 1000
loops were timed and a mean value was calculated. The test resulted in a loop time of
110 ms which is a loop frequency of 9.1 Hz. If driving the standard 1 m/s, this yields a
solution of 11 cm driven per line update. This should be a sufficient update frequency
when driving 1 m/s.

7.13
Summary
In this chapter, a series of test results have been presented. The overall solution has
been tested in various scenarios which could occur while autonomously navigating the
DTU running path. While testing, it was found that the network sometimes yields a
somewhat spotty segmentation, but the edge detection algorithm usually manages to
find a usable line that the robot can drive according to. Furthermore, it was found
that the GPS-based point of interest system successfully enables the robot to navigate
path segments that would otherwise fail using the standard implementation of the path
following procedure.
It was also investigated if the robot would perform as desired in a foreign environment
with new paths. Here it was found that the solution is robust enough to function in a
foreign test environment. Finally the system was tested with a full autonomous drive
of the whole DTU running path. Here the robot performed as desired and was able
to navigate the path autonomously for both clockwise and counter clockwise driving
direction, with few unintended manual interruptions due to e.g road or construction
work.
Finally it was determined that the solution can run with a loop frequency of 9.1 Hz,
which should be sufficient for autonomous driving at the velocities used in the project.
Furthermore, it was found that the path following script does not affect the sample time
of the MRC significantly.

CHAPTER 8
Future Work
In this chapter, a selection of suggestions for future work, potentially improving upon
the solution, will be presented.
• Improving the segmentation: During testing it was found that the U-Net
occasionally yielded spotty segmentation. This behaviour could be improved upon
by obtaining more training data, thereby making the network more robust.
• Adding a network for snowy conditions: The solution presented was found
to be unable to segment a path with snow.
Therefore it could be relevant to
implement a version of the network that is trained on paths with snow.
• Making the system less dependent on GPS: The implementation for driving
around the DTU running path includes multiple points of interest where to robot
crosses a sidewalk based on odometry. In order to make the system less dependent
on GPS, the sidewalk crossing procedure could be improved in the future.
• Added logic for edge detection algorithm: As of now, the edge detection
algorithm is able to handle situations with up to four intersections found in the
segmentation. It might be an advantage to increase the number of intersections
handled.
• Navigating towards a goal: As of now, the robot is only able to simply follow a
path, unless a number of points of interest are given, enabling the robot to navigate
a particular path segment. It could therefore be a good addition to develop an
algorithm enabling the robot to drive towards a given GPS position using the paths
of an environment. This functionality should not be pre-programmed. The robot
should therefore be able to make decisions on whether to i.e turn left or right at a
path split in order to guide it closer to the goal.
• Laser scanner emergency stop: Though not being used, the iRobot is equipped
with a laser scanner. This laser scanner could be utilized for an emergency stop,
braking the robot in case an object comes too near in front of the robot.

CHAPTER 9
Conclusion
In this project, a solution for driving a robot autonomously along the running path at
DTU Lyngby was designed. The iRobot ATRV-Jr was used for the project due to its
size, range and sensors.
Current research in the field of path segmentation was examined and from this, an
approach using a neural network was decided upon. A camera was mounted on the robot,
enabling the design of a neural network based on images from this camera. Furthermore,
the work was split up into several modules to get a better overview of the project.
A neural network based on the U-Net architecture was designed for the purpose of
doing semantic segmentation of the running path, which determined the robot’s position
in relation to said path. Using these segmented images, an image analysis algorithm was
implemented, which extracted lines from the segmentation for the robot to follow.
Measures to ensure robustness and speed were taken to get the best possible solution
for the objectives in the project. A GPS module was also included in order to achieve a
sufficient level of safety against driving into traffic when driving autonomously.
The system was thoroughly tested, both in terms of the different submodules and
the entire system as a whole. In terms of the objectives, the tests gave the following
results:
• Objective 1: Detect the road on which the robot drives on. This objective
was achieved using the neural network designed in chapter 4. The designed network
performs well, and is able to segment the path from the background in a wide
variety of situations, as depicted in the test conducted in section 4.4 and chapter
7.
• Objective 2: Select a path to drive from the detected road. This objective
was achieved using a classical image analysis algorithm on the binary segmentations
of the path generated by the neural network. It works on all the shapes of paths
encountered of the running path at DTU Lyngby, and is also able to handle the
path being split into two parts by a crossing sidewalk. The tests done in section
5.4 and chapter 7 show that this objective is achieved when using the algorithm
designed.
• Objective 3:
Avoid static obstacles such as humans standing in the
way. This objective is inherently achieved due to the design of the segmentation
==PAGE:119==
104

and line extraction modules working in conjunction. The segmentation module
segments static obstacles as belonging to the background, which in turn causes
the line extraction module to estimate a line going around the objectives, making
the robot avoid it. The achievement of this objective is documented in tests in
sections 5.4.5 and 7.2.
• Objective 4: Stop driving when approaching a traffic heavy road. To
achieve this objective, a separate module was designed using a GPS antenna to
track the robot’s position in relation to several points of interest. This enabled

• Objective 5: Autonomously drive the entire path chosen, excluding pub-

robustness.

APPENDIX A
User Manual
Here, a brief user manual will be presented for anyone who wishes to test the system.
The python script used for running the robot is positioned in the folder ”path naviga-
tion” on the iRobot. A user should open a terminal inside this folder and run the script
”joypadScriptStart.py”. The script has a series of arguments for different funcionallity,
hereunder:
• -log: If using this flag, the GPS position of the robot will be logged until the script
is terminated.
• -debug: Using this flag enables the debug functionality. When debugging, the
network will segment images from a video file instead of live images.
• -CW: This flag makes the robot use the point of interest list for driving clockwise
around the DTU running path, standard is counter clockwise.
When the script is running, the joypad is used for giving commands to the system.
The different commands that a user can give is:
• Pressing X starts the main script.
• Pressing B stops the main script (press again to exit the joypad script entirely).
• Pressing A makes the robot drive at a velocity of 1.5 m/s instead of 1 m/s
• Pressing Y Opens a map of DTU with the route driven plotted on top.
• Pressing LB starts logging of GPS position, odometry and images until LB is
pressed again.
• Pressing RB takes a single image and names it the current GPS position of the
robot.
• Pressing Logitech enables manual driving.
The robot is able to navigate the DTU running path when started at any location on
the path. Though a autonomous stop is implemented as a safety before all civil roads,
the user should always be ready to press the emergency stop, should the GPS module
fail.
