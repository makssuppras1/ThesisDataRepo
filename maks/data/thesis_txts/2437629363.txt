Summary
The use of unmanned aerial vehicles (UAVs) in the industry has grown over the past few
years. With the increased demand on the market, several autonomous solutions have
been developed for many outdoor applications, where the position of the UAV are known
through a global positioning system (GPS). Using a special kind of UAVs called multi-
rotors which are particular suitable for indoor flying, an untouched field of autonomous
solutions using UAVs indoor is now feasible.
The aim of this project is to develop such an autonomous solution, where a multirotor
is supposed to navigate trough a known environment by using a two dimensional metric
map. To do this, Kalman localization is used, which combines a model describing the
motion of the multirotor with measurements from a laser scanner. By extracting lines
from the laser data and comparing them to the lines from the two dimensional map,
the algorithm is capable of determining the multirotors position and orientation, also
known as the pose. In this project the model of the multirotor is implemented on the
autopilot as an optimal state estimator, where the estimated states are used as input to
the Kalman localization. The resulting estimated pose from the localizer is then used
as a measurement in the innovation of the state estimator on the autopilot. Having this
cascade configuration will allow high frequency dynamics to be estimated by the state
estimator on the autopilot, which then can be used to enhance the estimation of the pose
in the Kalman localization. The control objective of the project is to have offset-free
tracking of the position in the x-,y- and z-direction, while also being able to control the
heading of the multirotor. Part of this objective is realized by using an optimal full state
feedback controller, where the output of the controller is used as input for the existing
attitude controller. The control of the position in the z-direction is realized by using the
existing altitude controller and replacing the existing altitude estimates with the ones
from the new state estimator. Both the optimal state estimator and the controller are
tuned using a simulation of the system. Results from the real system showed that no
additional tuning was necessary.
By implementing an interface between the on-board companion computer and the
autopilot, the autonomous solution is fully capable of navigating through missions of
varying complexity and duration. As all of the design specifications of the project were
achieved some suggestion are proposed on how to take the autonomous solution closer
towards a real life application.

CHAPTER 1
Introduction
This project concerns the development of an autonomous multirotor for indoor flying.
The multirotor will be equipped with a laser scanner and a small on-board computer,
which will be used in combination with the autopilot to create a fully autonomous
solution, such that the multirotor will be able to navigate on its own in a known indoor
environment.
1.1
Motivation
Over the recent years, the use of unmanned aerial vehicles (UAVs) has grown rapidly.
The development of the UAVs has in many years been driven by the military indus-
try, where the UAVs has been used for surveillance and as weapons. However today
the industry has expanded to the commercial and consumer market, where they have
been primarily used for surveillance, aerial photography, product delivery and as enter-
tainment for drone enthusiasts. A new category of UAVs called multirotors has recently
been introduced to the industry, and are already one of the most preferred type of UAVs,
mainly due to their ability of doing vertical take-offs and modifiable structure. Accord-
ing to [Com], the future of the UAV industry will be driven primarily by the commercial
market. This is due to a huge potential in the many processes that can be automated
by the use of UAVs. One example is how the parcel company Amazon invests millions
of dollars into Amazon Prime Air [Ama], a business case where UAVs autonomously will
be used to deliver packages to customers. Another example is how large UAVs are used
in the agricultural industry for inspection of crops. In this way a huge of amount of area
can be inspected in a much more efficient and fast way than when using ground vehicles.
In both of the presented examples, the UAVs are flying autonomously using a Global
Position System (GPS) for the localization of their absolute position. However in many
other commercial application, the UAV will be flying indoor and thus GPS is not the
best choice of sensor mainly due to the lack of accuracy and the loss of signal when
indoor. An example of such an application could be an inspection of a product silo,
where the purpose could be to look for cracks in the inner walls of the silo. Normally
this would require constructing a scaffold, from where an operator would be able to
inspect the walls of the silo at one floor level at a time. But by mounting a device for
the inspection, e.g a camera on the UAV, one would be able to inspect the whole silo
efficiently and without the need to construct any expensive and time consuming scaffold.
==PAGE:20==

1 Introduction
Another example could be a search and rescue scenario, where the UAV could be used
to enter buildings which were considered too dangerous or not possible for humans to
enter e.g a building on fire or at risk of collapsing. In this case a UAV could, without
any risk, enter the area and for example provide an oxygen mask for the people trapped
inside the building. A third example could be if a new construction with multiple floors
were to be mapped. One could then use an UAV running a SLAM1 algorithm to map
the area efficiently, precise and without having to modify the environment.
As it can be seen, the list of indoor applications using UAVs is widely spread over
many industries. Common for all them is that they need a localization system that can
not rely on the use of GPS. Thus the motivation of this work comes from developing
such a system, that would be able to take the industry to the next level and thereby use
some of the huge potentials in autonomous solutions using UAVs.
1.2
Problem statement
The main objective of this project is to develop a fully autonomous multirotor which is
suitable for indoor flying in any known environment. The autonomous solution should
be able to localize the position and orientation of the multirotor using only on-board
sensors and a given map of the environment. A control strategy should be implemented
that will make the autonomous solution able to track any given position reference in the
three dimensional space as well as the orientation.
The main objective can be divided into the following subtasks.
• Design and assemble a multirotor suitable for indoor flying.
• Develop a linear model describing the motion of the multirotor.
• Use the model to design a state estimator and controller for the autonomous solu-
tion.
• Make a simulation environment of the control problem, which can be used to test
the solution and tune the estimator and controller.
• Modify the firmware on the autopilot of the multirotor, such that the autonomous
solution can be implemented.
• Establish communication between the autopilot and the on-board companion com-
puter.
• Implement a localization algorithm on the companion computer.
1SLAM: Simultaneous Localization and Mapping, is an algorithm used for localizing in an unknown
environment, while simultaneously creating a map of the environment.

1.2.1
Design specifications
The following design specifications for the autonomous solution should be fulfilled.
• The set-point control of the position in the x, y and z-direction should be within
±6 cm relative to the given reference, which corresponds to approximate 10 % of
the multirotors width and length in each direction.
• The control of the heading should be within ±2 deg.
• The actuation of the pitch and roll angles should not exceed ±10 deg in order to
avoid rapid accelerations.
• The solution should be fully autonomous, i.e the take-off, flying and landing should
all be without interference of the operator.
• The solution should be robust in the sense, that the complexity of the mission
should not affect the solution.
The specifications related to the precision of the autonomous solution are chosen such
that the multirotor will be able to fly in narrow indoor spaces, e.g through a standard
doorway.
1.3
Thesis outline
The thesis is structured in the following way. Chapter 2 is a presentation of the assembled
multirotor and the choice of hardware. Chapter 3 is an analysis of the basic principles
used in the solution. These principles includes; A description of the used coordinates
systems and the transformations between them.
An analysis of the basic dynamics
of the multirotor and how it is able to move. An analysis of the error generated by
the data of the laser scanner when the multirotor is pitching or rolling. In Chapter
4 the linear model describing the motion of the multirotor is derived, and discretized
for further use. Some properties of the linear model is analysed, here among stability,
controllability and observability.
In Chapter 5 and 6 the model is used to design a
linear state estimator and controller, respectively. In Chapter 7 the model of the system
is simulated and some properties of the resulting closed-loop system are analysed. In
Chapter 8 the implementation of the autonomous solution is described, here among
how the solution is structured between the autopilot and the companion computer. In
Chapter 9 the results of the experiments when testing the autonomous solution in real

CHAPTER 2
Assemble of the multirotor
2.1
The frame
As explained in the introduction, part of the project is to construct a multirotor suitable
for indoor flying. Several options have to be considered. First one needs to decide how
powerful the multirotor should be. This is of course a compromise with how much flight
time is required, as more power consuming rotors tends to draw more current from the
battery, thus decreasing the flight time. If a powerful multirotor is a high priority, one
could also consider to increase the number of rotors, which gives a more stable and
powerful vehicle, but on the cost of manoeuvrability and power consumption. The most
common number of rotors are 3,4,6 and 8, from which the name convention yields a
trirotor, quadrotor, hexarotor and octorotor, respectively. When choosing the number
of rotors, one also needs to choose the rotor configuration, which basically describes how
the rotors are mounted with respect to the axes of rotation. Some examples can be seen
in Figure 2.1.

Obviously the number of rotors and the configuration needs to be supported by the
choice of autopilot, such that the autopilot is capable of distributing the control signal
between the rotors for a desired manoeuvre. When a multirotor is to be used in an in-
door environment, the control space can be narrow and thus the size of the vehicle must
be prioritized, such that the multirotor has as much space for manoeuvring as possible.
In most pre-build multirotor frames, the size of the vehicle tends to increase with the
number of rotors, unless a contra-rotating configuration is used, where for example the

2.2.5
Motors, propellers and ESCs
The spinning of the rotors creates the thrust which essentially lifts the quadrotor. When
an angular velocity is desired by one of the motors a pulse-width modulation (PWM)
signal is send from the autopilot into the ESC which interprets the PWM signal into a
corresponding power which is then sent to the motor. The motors are typically 3-phase
brushless DC motors. One of the tasks of the ESC is therefore to switch the power
between the 3 phases such that the electromagnet of the stator gets magnetized in a
pattern that will drag the permanent magnet of the rotor towards the magnetic field.
By switching the magnetic field in a circular pattern will make the rotor spin in the
corresponding direction. When the rotor is the outer part of the motor, it is said to
be an outrunner motor. The opposite is when the rotor is the inner part of the motor,
which is called an inrunner motor. The outrunner motor is most common in multirotor
applications. Besides powering each of the phases, some feedback in the form of back-
emf is send to the ESC, which helps it to determine the next position of the stator to
be magnetized. When choosing an ESC, one should also consider the firmware. Today
many suppliers has specialized ESC firmware for different kind of multirotor applications.
One is SimonK7, which is a firmware specialized in general multirotor applications. One
of the modifications compared to a traditional ESC firmware, is that with the SimonK
firmware, the ESCs will not cut off the power when the voltage of the battery drops
below a certain level. This will of course risk damaging the battery, but the alterna-
tive is to cut off the motors while the multirotor is still airborne. Another important
advantage is increased response time of the angular velocity of the motors which in the
end affects the response time of the pitch, roll and yaw dynamics. For this project, 4

When deciding which propellers to use, the primary things to consider are size and
material. The stiffer the material is, the less likely the blade is to flap when rotating fast.
Blade flapping is an undesired dynamic of the blade of the propellers and thus damping
this dynamic will increase the thrust and hence make the propeller more efficient. The
most stiff material for hobby propellers is carbon fibre, other materials are glass fibre,
nylon and polycarbonate plastic. Because the quadrotor is to fly indoor in a narrow
environment, carbon fibre blades are not the best choice due to safety precautions, as
they tend not to brake when in contact with the environment. This would mean that in
a situation where a person should get in the path of the quadrotor, the outcome would
be much more critical with carbon fibre blades than when using other alternatives. The
size of the propellers determines how much air they will be able to move and hence
how much thrust is generated. The maximum size however is limited by the physical
constraints of the frame and how much torque the motors are able to generate. Having
these things considered, 4 × 11 inch glass fibre propellers10 are chosen for the quadrotor.
2.2.6
Battery
When choosing the battery, one needs to ensure that the voltage level and discharge
rate are compatible with the ESCs and motors. When an applicable battery is found,
one can consider the type, weight and capacity. The most common battery type for
multirotors is lithium polymer batteries (LiPo). This is due to that this type of battery
has some desired properties, such as; small dimensions, light weight, applicable voltage
levels and a relatively flat energy discharge curve. The capacity of the battery is a com-
prise between flight time and the size and weight of the battery. This is due to that the
weight and size increases with the capacity, and thus one could risk that the multirotor
would not be able to take-off if a too large battery is chosen. For this project a 3 cell
(12.2 V) LiPo 5000 mAh battery is chosen. The battery has a discharge rate of 50 C
which is fully applicable with the chosen ESCs and motors.

3.2
Movement of a multirotor
In order to control a multirotor autonomously, one first needs to understand how the
vehicle is able to move, its manoeuvrability and the constraints associated with the
movement. For the sake of simplicity, some of the dynamics contributing to the non-
linearities of the multirotor are ignored, more specifically these dynamics are:
1. Global drag force (air friction) - Force that applies to the vehicle as a function of
the vehicles speed.
2. Friction force - The friction force the rotors are experiencing when spinning.
3. Blade flapping - The deformation of the flexible propellers when spinning a high
rates, thus dynamically changing the parameters of the propeller at different oper-
ating points.
4. The gyroscopic effect - The coupling effect when a body rotates around a given
axis and that axis changes. The gyroscopic effect will depend on the inertia of the
body and the angular velocity of the rotation.
A true rigid body has the ability to move freely in a three-dimensional space. This
means the body is free to change its position in the x-, y- and z-plane while also being
able to change its orientation through rotations about the three axes, more specifically
changing its roll, pitch and yaw angle. However a multirotor do not has the freedom to
control all of these movements independently of the others. In fact a multirotor only
has 4 degrees of freedom (DoF), which means that even though it is able to move in the
three-dimensional space, this can only be done by controlling the thrust, pitch, roll and
yaw. This makes the multirotor a nonholonomic vehicle and thus the movement in the
x and y-direction needs to be controlled through the other states.
In general the movement of any multirotor is produced by the rotors. A rotor consists
of a motor and a propeller, as illustrated in Figure 3.3.

It should be remembered that the laser readings are used in a localization algorithm
to match predefined lines. If the error is small, it will just propagate as a minor error
in the pose of the multirotor. However if the error is large, there is a chance that the
localization algorithm will not be able to match with the predefined lines, and thus no
pose can be computed. However, as this would require large pitch and roll angles or
walls far away, it is assessed that the error will not be significant enough and therefore
compensation of the error is chosen not to be implemented.
A much more significant error to be considered in this context, is if the perceived
environment changes when the multirotor is pitching or rolling, for example if the laser
looks above a wall or down into the floor. Such type of error could easily make the
multirotor lose its pose, potentially resulting in a crash. The ideal way to avoid this
would be to have a map and localization algorithm working in all three dimensions.
However if this is not possible, one can make sure the conditions of the environment
are optimized to avoid this problem, while also choosing the flying altitude to be in

CHAPTER 4
Model
In this chapter the model used for the controller design, the state estimator and the
simulation is derived and described. When choosing a model describing any kind of
system, it should always be considered what the objective of the model is, and how
precise the model needs to be in order to reach the objective. In general the model
of lowest order that reaches the objective is always preferred.
The objective of the
model used in this project is to describe the linear motions in the three dimensional
space as well as the heading. The model is then to be used to design a controller that
eventually will be able to control the multirotor in any of those directions with a given
heading. It should be noted that for simplicity in the notation the time argument of the
continuous-time equations is omitted.

the phase, resulting in a phase margin of zero.
Therefore in order to control the position in the x- and y-direction one would need
to either differentiate the position as in a classical PD-controller or by estimating the
derivative of the position i.e the velocity and using the state in the feedback loop, e.g
by having a full state feedback controller.
As the presented model already has the
velocity as a state, the last solution is chosen. In the model, the heading is estimated by
integrating the yaw rate one time, and thus is should be enough with only a proportional
gain to control this state.

4.7
Controllability analysis
Before designing any controller, it should always be considered if the system is control-
lable.
When discussing controllability, one should first consider if the system is controllable,
i.e if a series of inputs can drive the system from any initial state to the zero state.
Secondly it can be considered if the system is reachable, i.e if a series of inputs can
drive the system from any initial state to any final state. In continuous-time these two
properties are equivalent, however for discrete-time systems, having controllability does
not imply reachability, but having a system that is reachable does also imply that it
is controllable. Thus when analysing the discrete-time system for these properties, one
begins by checking if the system is reachable, hence also controllable. This can be done
by checking that the controllability matrix M has full rank [EH08b], where the matrix

One of the risks of using this configuration, is that it creates a feedback loop, where
the pose of the localizer will be based on a corrected version of the previous pose, and
hence if the localizer should estimate a false pose, this would also be amplified in the
loop. However it is considered that the benefits of the configuration outweighs the risk.
If one should consider the risk as being too large, some way of detecting the false poses
could be implemented as a solution to this problem.
5.1
Local estimator
Due to process noise, measurement noise and missing information about some of the
states, an estimator is to be designed. In this case an optimal estimator i.e the Kalman
filter is chosen.
Whether the filter should be dynamical or stationary both have some pros and cons.
In a dynamical filter the estimates converges much faster than in the stationary filter,
due to the possibility of high value initialization of the covariance matrix. However it
is also known that normally, within a small number of samples, the dynamically covari-
ance matrix will converge towards the stationary, and thus the difference between the
two filters will be small under stationarity. With this in mind one of the cons of having
a dynamical filter is that in each iteration of the estimation procedure, one will need
to recalculate the covariance matrix and also the Kalman gain, besides also doing the
time and data update of the states. As the filter is supposed to be implemented inside a
400 Hz loop, it is very important that the number of calculations does not violate the run
time of the loop. Hence a discrete-time ordinary stationary Kalman filter is chosen. An
alternative to the ordinary Kalman filter is the predictive, where the latest observation
is not included in the calculation of the estimate. This can be useful in situations where
hardware limitations are an issue.
As explained previously the model used for the estimator and the controller is defined
in the world frame. For the sake of simplicity in the notation it should be implicit that
the model used in this chapter is defined in the world frame, unless stated otherwise.
Also all measurements and inputs are assumed rotated into the world frame.

broken. If not taken care of, the integrator will accumulate the error and in worst case
make the system unstable when the feedback is restored. This is known as integrator
windup. One example is when the control signal saturates. In this case the system
runs in open loop and the controller cannot do anything before the control signal gets
below or above the saturation limit. Having integral action this will cause a significant
accumulation of the error and therefore increase the time the system will run in open

6.5
Reference generator
In order to keep the control signal at a reasonable level, the magnitude of the error as
seen by the controller needs to be bounded. However when giving the reference as a step
this can be difficult as the error will depend on the size of the step. A solution to this
problem is to shape the reference signal as a ramp, such that the size of the error will
depend on the slope of the ramp instead. However in order to track the reference to a
desired set-point, one would also need a way to stop the ramp. A proposed algorithm
to do so can be seen below
Algorithm 1 Reference generator
1: loop:
2: if abs(xtarget −xref) < tolerance then
3:
xtarget = xref
4: else if xtarget < xref then
5:
xtarget = xtarget + xvel ∗Ts
6: else if xtarget > xref then
7:
xtarget = xtarget −xvel ∗Ts
8: end if
In the algorithm xtarget is the ramp signal, xref is the set-point and the tolerance
is a measure of how close the multirotor should be to the set-point before holding the

value of the ramp signal. As the controlled states are all positions, the slope of the
ramp corresponds to the velocity of the state, which is indicated by the name xvel. Thus
when using the proposed algorithm, the multirotor will be able to move with a constant
velocity towards any desired set-point. The algorithm will be used for generating the
reference signal rk from the control law in Equation 6.24. The slope of the ramp is set

8.1
Autonomous flight mode
In order to separate the autonomous solution from the existing firmware on the autopilot,
a new flight mode is to be implemented. The ArduPilot firmware has 20 preprogrammed
flight modes, which can be switched between in order to manually operate the multiro-
tor to fly in a unique way specific to the chosen flight mode. One of the most common
flight modes is stabilize, where the pilot’s throttle-stick on the radio controller is directly
interpreted as thrust to the motors. Another common mode is altitude-hold, where the
multirotor is able to hold its altitude using the existing altitude controller. In this mode,
the vertical movement of the pilot’s throttle-stick is interpreted as a climb rate command,
where the rate will be zero when the stick is in the middle position. When the stick is
above the middle position, the climb rate will be positive and the multirotor will ascend.
Similarly when the stick is below the middle position, the climb rate will be negative
and the multirotor will descend. The commanded climb rate is then transformed into a
altitude reference for the altitude controller. Similarly the horizontal movement of the
throttle stick is interpreted as a yaw rate command, which is then transformed into a
desired angle in the existing attitude controller. The other pilot stick is used to control
the pitch and roll angle, where the stick position is interpreted as a desired angle for the
attitude controller.
Instead of designing a complete new flight mode, the structure of the altitude-hold
flight mode is used as a base for the autonomous mode. Thus in order to fuse the new
controller with the existing structure, some modifications needs to be made. In order to
give the existing altitude controller a climb rate as a reference without using the sticks
on the radio controller, the algorithm shown below is to be implemented.

The payload of the MAVlink packet is called the message. Each message has an
unique 8 bit identification code on the system, thus it is possible to declare 256 different
message. Some of the basic messages are already defined on the native firmware of the
autopilot, e.g a message with the IMU data or a message to change the current flight
mode. Hence when declaring new message for the autonomous mode, one should be
aware of not using reserved message id’s.
On the ArduPilot firmware, new MAVlink messages are declared in a .xml file. When
the firmware is compiled, each of the messages defined in the xml. file will be generated
into a separate header file, which will contain the message definition, as well as the
functions used for encoding and decoding the message. For this project several new mes-
sages are needed. The most important custom messages are: The message carrying the
pose estimates from the autopilot to the companion computer and the message carrying
the pose estimate from the companion computer to the autopilot. Additional custom
messages are implemented in order to collect the required data for logging, however the
autonomous solution will work with or without these. In Appendix F is a guide of how to
create new MAVlink messages and how to use the variables in the ArduPilot firmware.
8.2.2
Network socket
On the companion computer two separate programs are running. Those are the main
program and the ulmsserver. In order for them to exchange data with each other, a
socket connection is to be programmed. In the connection, the ulmsserver will act as a
client and the companion computer will be the server and thus responsible of initializing
the connecting. When the connection is made, both are capable of transmitting and
receiving data to each other. Another socket connection is between the ulmsserver and
the laser scanner, however in this connection, the ulmsserver is the server and the laser
scanner is the client.

CHAPTER 10
Conclusion
The primary objective of the project was to develop an autonomous multirotor suited
for indoor flying. Based on the results shown, it can be concluded that the objective has
successfully been achieved. The proposed solution is able to perform fully autonomous
missions in any known environment described by a two dimensional metric map. In
the rest of the chapter it will be discussed how the individual subtasks of the problem
statement have been solved.
The assembled multirotor of the proposed solution, is a manoeuvrable, light and rel-
atively small quadrotor having a pixhawk flight controller using the ArduPilot firmware.
Additional to the standard hardware, a laser scanner and a small on-board computer
was mounted on the platform. As no pre-bought frames had the desired mounting pos-
sibilities, some custom parts were made on a milling machine and a 3D-printer.
A model describing the motion of the multirotor was successfully derived. The model
is based on a motion analysis of the multirotor. The analysis showed that by varying
the thrust on each of the rotors, a torque is generated which will make the multirotor
able to rotate around the pitch, roll and yaw axis. A simple first order model of the
pitch and roll dynamics were then found experimentally and used in the model to de-
scribe the motion in the horizontal plane of the multirotor. The derived model was
linearised by using a small angle approximation, as one of the design specification was
to keep both the pitch and roll angle small. The linear model was then expressed as a
state space system and later discretized using a zero-order-hold parametrization of the
input. An analyse of the discretized linear model showed that the open-loop system was
controllable and observable, but not stable. Thus a control strategy was implemented
based on an optimal linear state estimator, i.e the Kalman filter and an optimal full
state feedback controller, i.e a linear quadratic regulator. The estimator and controller
were successfully simulated in the simulation environment, from where it can be con-
cluded that the proposed control strategy stabilizes the system and makes it possible
to track the reference for all of the controlled variables. As it was seen in the results,
the responses of the real system was very similar to the ones of the simulated system
and thus it can be concluded that even though the proposed model is very simple, it
is sufficient to describe the dynamics of the problem. This conclusion is supported by
the fact that the parameters found in the simulation for both the Kalman filter and the
controller, did not need to be adjusted on the real system.
==PAGE:90==
72
10 Conclusion
The implementation of the Kalman filter and the controller were made on the autopi-
lot by modifying the existing firmware with an additional flight mode. The framework
of the autonomous flight mode was made from another flight mode, called altitude hold.
As the output of the new controller is a desired pitch and roll angle as well as a desired
yaw rate, they were directly realizable by using them as input to the existing attitude
controller. By using the existing structure of the firmware, the new functionalities were
efficiently implemented by expanding the linear system into single line expressions. From
an analysis of the existing control architecture of the autopilot, it was seen that the al-
titude was already part of the control scope. However as the state estimates used in the
controller were correlated with the barometer, they were concluded not to be suitable
for indoor flying. Thus the estimates from the Kalman filter of this project were used
instead in the existing controller.
For the localization, a Mobotware plug-in algorithm called aulocalize was used. The
algorithm was successfully modified to be used on a multirotor instead of a mobile robot,
where the initial guess of the pose came from the Kalman filter implemented on the au-
topilot. Using the data from the laser scanner, the algorithm was able to estimate the
global position of the multirotor, which was then used as a measurement update in the
Kalman filter. Results showed that when using the two estimators in cascade, the mul-
tirotor was able to estimate the states with a bandwidth and robustness that made the
control of the position and orientation feasible. In order to share data between the two
filters and in general between the autopilot and the companion computer, the MAVlink
protocol were used with some custom defined messages.
From the results it could be seen that the implemented solution was able to control
the position in the x, y and z-direction with a precision of approximate ±5 cm. However
in the z-direction a stationary error of almost 5 cm was seen.
This was assumed to
be caused by some undesired property of the existing controller.
For the control of
the heading, a precision of approximate ±1 deg was achieved. Moreover it was seen
through the three presented studies, that the implemented solution was not affected by
the complexity of the mission and therefore it can be concluded that the solution is both
flexible and robust. When hovering, the actuation of the pitch and roll angle was able to
stay within ±1 deg around a bias value that would depend on how skewed the multirotor
was initialized. Consequently, by having the described properties in the implemented
solution, it can be concluded that all of the design specifications have been achieved
successfully.

10.1 Future work
73
10.1
Future work
The implemented solution presented in this thesis, is a great foundation for future work
using multirotors for autonomous applications.
It has been shown that autonomous navigation and localization with a multirotor in
a known environment is feasible. The next step could be to expand the solution to also
work in an unknown environment. In order to do this, the multirotor should be able to do
simultaneous localization and mapping, also known as SLAM. This would require some
more processing power, and it should therefore be considered if the choice of companion
computer in this project is sufficient. Another option could be to change the sensor and
modify the localization algorithm to be working in three dimensions. This would make
it possible to navigate and operate in much more complex environments. However as 3D
laser scanner are very expensive, a camera would possibly be a more natural choice for
this kind of modification. Yet another possibility for future work, is to test the proposed
solution on a real life application, which could be one of the applications proposed in
the motivation section of the introduction. One could for example simulate a search
and rescue scenario in a known building to see if the autonomous solution is able to
find a person inside the building. This would however require an extra sensor for the
detection of the person, which in this case could be a camera. In the results it was seen
that the control of altitude had a small stationary error. Moreover there was an issue
with the existing altitude controller that caused the multirotor to not always make a
clean vertical take-off. Therefore another possibility for future work is to remove the
exiting altitude controller on the autopilot and replace it with a new one, where the
properties of the implemented controller are better known, for example the one used in
the simulation of the system.

APPENDIX E
Getting started
This appendix is a guide of to compile and start the autonomous solution. In the guide it
is assumed that the user is connected to the network of the Odroid, called quadcopter01
and a terminal is opened on the Odroid, e.g through a ssh connection.
E.1
The structure of the repository
The structure of the repository of the autonomous multirotor solution can be divided
into three groups, as seen below
• The first group is the main file, called autnav.c. The file is located at local/
mobotware/mrc/trunk/autcopter.
• In the second group are all the missions used in the autonomous solution. They
are located at local/mobotware/mrc/trunk/autcopter/missions.
• The third group is the multirotor library, which is a collection of all the func-
tion used in the autonomous solution of the multirotor. It is located at local/
mobotware/mrc/trunk/autcopter/lcopter. It is also within this library that the
MAVlink headers are located.
E.2
Compiling the source code
When compiling the source code of the project, one should use the provided Makefiles.
The Makefile located in the main directory local/mobotware/mrc/trunk/autcopter
will compile both the main program and also the missions. For compiling the multirotor
library, one should use the Makefile located inside the library directory.
E.3
Starting the program
The following section describes how to start the main program for the autonomous
navigation and location of the multirotor.
==PAGE:108==
90
E Getting started
1. Start the ulmsserver in the /local directory. The init file is in the same directory.
2. Make sure all the switches on the radio controller are flipped to the upper position.
3. Arm the multirotor and switch the flight mode to altitude hold. (Tip: Set the
throttle around the middle position, such that one can switch to stabilize without
cutting the thrust.)
4. Start the main program called autnav located at local/mobotware/mrc/trunk/
autcopter.
5. The autonomous mission will start after a few second. If the mission is programmed
properly, the program will end after the multirotor has landed and disarmed itself.
6. Quit the ulmsserver by typing ’q’ in the terminal window of the server.

APPENDIX F
Creating a MAVlink
message
This a guide of how to create custom MAVLink messages. The guide is a modified
version from previous work [Hje17a].
Step 1: First of all one needs to navigate to ardupilot/modules/mavlink/message_
definitions/v1.0 where all the message definitions are located. It is recom-
mended to create the new messages in /ardupilotmega.xml, but in principle
they can be placed in any of the xml-files, as long as they are linked properly.
Step 2: Next thing is to make sure that the new MAVlink messages has a unique message
ID between 1 and 255, if no space are available take one of the ID’s of some
messages that are not relevant to the project.
Then insert the custom message with the desired attributes, an example could
be the one shown below
1
<message id="196" name="CUSTOM_CAMERA">
2
<description>Sends the custom camera cooordinates</description
>
3
<field type="int16_t" name="num">Number of pictures send</
field>
4
<field type="float" name="xposcam">Y position from camera</
field>
5
<field type="float" name="yposcam">Y position from camera</
field>
6
</message>
Step 3: When compiling the ArduPilot source code using the ./waf copter command,
the compile script automatically generates the header files needed for using the
protocol. These header files are also referred to as dialects. In order to have
the same dialect on both system that are suppose to communicate one needs to
copy the dialects from the Pixhawk in ardupilot/build/px4-v2/libraries/
GCS_MAVLink/include/mavlink/v2.0 onto the second system (e.g an Odroid
==PAGE:110==

XU4). The dialects which are per default generated in C++, should then be
linked in the compile of the program on e.g the Odroid XU4.
If one needs the dialects in another program language, running the script mav-
generate.py in /ardupilot/modules/mavlink lets the user specify the language
of the resulting headers (choose 2.0 as protocol).
Step 4: For the custom messages send from the Pixhawk one needs to include the mes-
sages in the enum list called ap_message which are located at ardupilot/
libraries/GCS_MAVLink/GCS.h. The messages should then be included as a
case inside the switch statement in the function try_send_message and put
into a stream in the function data_stream_send, both functions are found at
ardupilot/ArduPilot/GCS_MAVLink.
Step 5: For custom messages received on the Pixhawk, one needs to include cases for
these inside the switch statement in the function handleMessage, found at
ardupilot/ArduCopter/GCS_MAVLink. The easiest from here is then to copy
the content of the messages into variables inside the Copter class, which can
be accessed in all of ArduPilot directory. The variables of the class are defined
ardupilot/ArduCopter/Copter.h. Once this is done the variables can be used
for various purpose in whatever flight modes needed.

APPENDIX G
Creating a new flight mode
This a guide of how to create custom flight mode. The guide is a modified version from
previous work [Hje17b].
Step 1: First thing is to find a exiting flight mode with some properties that are useful
for the new mode. The flight modes are located at ardupilot/ArduCopter and
have ”control_” in front of the filename.
Step 2: Copy the chosen flight mode (cpp-file) and rename the file the name of the new
mode, e.g control_newmode.cpp.
Step 3: Inside control_newmode.cpp the initialization function and run function should
be renamed to the name of the new mode e.g newmode_init.cpp and newmode_run.cpp.
Step 4: The prototypes of the two functions needs to be included in ardupilot/ArduCopter/
Copter.h. A reasonable place for this is under the existing declarations.
Step 5: The new mode should then be included in the enum list control_mode_t which
are located at ardupilot/ArduCopter/defines.h. The given number will be
the one used when switching to the mode from the companion computer.
Step 6: The new flight mode also needs to be included in the parameter list in ardupilot/
ArduCopter/parameters.cpp under ”FLTMODE1”, ”FLTMODE2”, ”FLTMODE3”,
FLTMODE4”, ”FLTMODE5” and ”FLTMODE6”.
Step 7: Next is to include the new flight mode in the handling of all flight modes. This
is done in ardupilot/ArduCopter/flight_mode.cpp. Firstly the initialization
function should be included in a case inside set_mode. Next the run function
should be included in a case inside update_flight_mode. Lastly the new flight
mode should be included in a case inside print_flight_mode.
Step 8: When all of the above steps are done, the flight mode can be modified as desired.
It is recommended to define variables inside the Copter class if they also should
be used outside the flight mode.
