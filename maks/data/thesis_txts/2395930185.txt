Abstract
To describe the trust that developers and users put into software projects, Trustworthi-
ness is analyzed and a large set of attributes are described in the literature, of which the
most important factor is security. Besides security, software engineering provides more
metrics that can be used to evaluate trustworthiness of software.
The National Vulnerability Database (NVD) stores information about Common Vul-
nerability and Exposures and their criticality, in the Common Vulnerability Scoring
System, which can be used to quantify the security aspect of software products that
have registered vulnerabilities. Data to describe other aspects of software engineering is
more difficult to come by, but by focusing on Open-Source Software (OSS) a lot of data
is available through repositories, and also Open Hub collects and analyzes these kinds
of data.
The tool TOSS - Trustworthiness of Open Source Software [15] uses these data to
create a combined Trustworthiness score, from the Security, Maintainability, and Team
sub scores. This project analyses the data as well as the scoring system and provides
suggestions for improvements and additions to the system.

CHAPTER 1
Introduction
Software is used everywhere from dish-washing machines to spaceships and we are con-
tinuously finding more applications where processors and software can be added. As
these devices are being attached to networks, security on computers is still a field where
we are trying to catch up with remediation. At the same time more and more faith is put
into computers, they are trusted with credit card information, social security numbers,
patient journals, and other sensitive information.
The trust that is put into these systems is often based on nothing but a belief, a
comment from another person, or just the fact that the system is needed and it probably
works. It does not get easier, when trying to define what makes the system trustworthy
and what trustworthiness of software is.
To define Trustworthiness in software, theory from Software Engineering can be
applied to find attributes that each describes a small part of what makes a trustworthy
software product. Some of these attributes can be quantified individually, to get a score
for each aspect of the software project, and those scores must be aggregated to give a
combined score for the complete trustworthiness of the project.
Open Source Software (OSS) is used widely and it is almost impossible to use a
computer without using OSS to some extend, if not directly then through some project
that depends on it. This has the advantage that a lot of data is readily available about
OSS, and these data can be analyzed and used for grading attributes of a given OSS
project.
TOSS - Trustworthiness of Open Source Software, is a tool developed by Nielsen
in [15] which is based on this theory and uses freely available data from, e.g., Open
Hub, and based on these gives a score representing the trustworthiness of any given OSS
project. With security being the main aspect of trustworthiness in software, it is the
largest part of the score, but the score also takes into consideration a maintainability
and a team score, grading the maintainability of the project and the developers behind
the project. However, as identified by [15] the scoring system has some flaws.
This project will analyze TOSS and the attributes it uses and attempt to improve
the scoring system to give a more fair rating of trustworthiness of OSS.

CHAPTER 2
State Of The Art
The prevalence of Open-Source Software (OSS) is larger than most people imagine, but
little effort has been made to rate the trustworthiness of open-source projects. This
chapter provides an analysis of trustworthiness in software, to discover attributes thereof.
It is then analyzed how the attributes have been used in a tool made with the purpose
of rating trustworthiness.
2.1
Trustworthiness
Trustworthy means worth of being trusted, reliable, honest, and dependable. For soft-
ware this is not easy to asses, but the literature provides definitions and tools that can
be applied to rate software trustworthiness.
Trusted Software Methodology (TSM) is a joint effort between governmental and
commercial organizations such as the National Security Agency (NSA), Ballistic Missile
Defense Organization (now Missile Defense Agency), and AT&T Bell Laboratories (now
Nokia Bell Labs). The work on TSM was started in 1989 and was in 1991 described
by Amoroso et al.
in [1].
Previously trustworthiness were by some defined only as
security, for instance by the NSA department National Computer Security Center in
their ”Orange Book” Trusted Computer System Evaluation Criteria (TCSEC).
Because other definitions used different criteria, such as software engineering tech-
niques used, a collectively agreed upon definition of trustworthiness was needed and it
came up as:
”The trustworthiness of software is defined as the degree of confidence that
exists that it meets a set of requirements” [1].
To have confidence in a project, a user must, besides believing in the security of the
project, also has to believe in the quality. This idea was used to derive the 44 Trust
Principles, that can be used as factors to rate the trustworthiness of a project, through
the model shown in Figure 2.1. In Software Engineering there are methodologies and
tools that can be used to enhance the process. From Security there are safeguards and
countermeasures to handle risk. Software Engineering and Security are the essence of
trustworthiness, through the combination of these two factors, the Trust Principles were
generated.
==PAGE:18==
4

Figure 2.1: The model used to derive Trust Principles in TSM from [1] from Software
Engineering methodology, Security principles and the relationship thereof.
The definition of trustworthiness specifically says degree of confidence, specifying that
trustworthiness is not binary value. A software project that is compliant with most of
the Trust Principles should also be rated more trustworthy than a project that is only
compliant with a few. To rate projects, a scale from T0 - T5 was developed, where T0
is an unknown number of Trust Principles met, and T5 all of them.
Very few projects are intended to reach T5, as it in the same way as developing
secure software is very expensive. It requires extreme amounts of resources and quality
assurance and is typically not needed. There are of course purposes, such as military
and space travel, where the highest trust class can be justified. For deciding a required
trust class the stakeholders should, depending on the project type and requirements,
asses the required investment for the needed trustworthiness. Threats, costs, and the
critical-ness must be investigated. Cost is the only reason to choose a lower trust class,
as it does not make sense to make a big investment to mitigate a small risk that a not
critical error will happen. However, if there are severe threats, or it is a critical point
of failure it might require a higher trust class. It must also be included in any libraries
used, as a lower trust class in this would constitute a weak link that lowers the standard
of the entire project.
The Trust Principles consist of some simpler components to verify such as access con-
trol and documentation of design rationale, and very complex things like formal methods.
Most of these require a lot of insight into the whole development and life cycle process.
A lot of literature focuses on qualitative methods for rating software quality, where the
architecture must be analyzed, components identified, and requirements analyzed for
these. This is not feasible for analyzing large quantities of projects retroactively, but
Mohammadi et al. provides an analysis study of Software Quality Attributes in [13]
where the important attributes that were identified are shown in Figure 2.2

2.2
Open-Source Software
This project focuses on scoring Free and Open-Source Software (FOSS), because of the
availability of data about the developers, the evolution in size of the code base over time,
vulnerabilities and more as is described in Section 3.1. FOSS also has advantages in the
independence it provides, security in that every one can see the source code and licenses
which are fair and easy to read.
Open-Source Software (OSS) is defined in the Open-Source Definition (OSD), by the
Open Source Initiative (OSI) by the following 10 points, which were originally based on
The Debian Free Software Guidelines [8]:
1. Free Redistribution
2. Source Code
3. Derived Works
4. Integrity of The Author’s Source Code
5. No Discrimination Against Persons or Groups
6. No Discrimination Against Fields of Endeavor
7. Distribution of License
8. License Must Not Be Specific to a Product
==PAGE:20==

2 State Of The Art
9. License Must Not Restrict Other Software
10. License Must Be Technology-Neutral
For a software project to qualify as open-source, it must be published under a license
that allows anyone to redistribute the code as part of their product. The license must
ensure that the source code is freely available to use and modify and further distribute
under the same terms as the original license, with the exception that a license can require
the integrity of the original source code to be kept when distributed, if it allows patch
files to be distributed along with the original code.
While the OSS term is often used interchangeably with Free Software (not to be
confused with Freeware), as defined by the Free Software Foundations (FSF), there are
small but significant differences. Most OSS licenses such as MIT License, GNU General
Public License (GPL), and Apache License are accepted by both FSF and OSI, but for
instance the Open Watcom license is accepted by FSF and not OSI, as it does not allow
one to modify the code and use it for private use. The FSF defines free software by the
four points of freedom1:
0. The freedom to run the program as you wish, for any purpose (freedom 0).
1. The freedom to study how the program works, and change it so it does your
computing as you wish (freedom 1). Access to the source code is a precondition
for this.
2. The freedom to redistribute copies so you can help your neighbor (freedom 2).
3. The freedom to distribute copies of your modified versions to others (freedom 3).
By doing this you can give the whole community a chance to benefit from your
changes. Access to the source code is a precondition for this.
While the description is less explicit than the OSD, such as not specifically mentioning no
discrimination, the accepted licenses are almost the same as for OSS. As free is sometimes
mistaken for ’free of charge’ instead of as in ’liberty’, the term libre is sometimes used.
To avoid upsetting neither the OSI nor the FSF the terms FOSS or FLOSS are widely
used, where the ’L’ in FLOSS adds libre to the free- and open-source software [22]. An
extensive table comparing which licenses are approved by FSF and OSI can be found on
Wikipedia 2.
Releasing OSS can be done by an individual or an organization, it only requires
that the license is included in the project. Typically larger projects are not made by
individuals but by one of the four categories [17]:

Single Vendor Open Source projects aims to sell proprietary software licenses by hav-
ing dual-licensing. External contributors are required to are required to sign agreements
to transfer the rights of their contributions to the vendor, who can then sell enterprise
editions. The vendor has full control of the project, but the business model is strange
because all competitors can see the full source code and they can fork it to create their
own versions at no cost. MySQL and Magnolia CMS are examples that it does work
for some organizations, but it is up to the owners to create incentive for people outside
their own company to contribute.
Developer Communities is a more typical kind of OSS projects, well known from the
Mozilla Foundation, the Linux Foundations and many others. Projects can be started
by individuals, universities or an organizations, but with their contributions, the group
of stakeholders and owners expand. Usually meritocracy is used to distribute control,
meaning the control is shared between the main contributors although it might differ
from to foundation/association to which the project belongs. Likewise the copyright
is shared among multiple stakeholders. This kind of community do not need to create
incentive for contributors to join the project in the same way as single vendor open
source projects. If the project is attractive or interesting, contributors will join for many
reasons, such as wanting to learn, wanting to use the project for their own purpose or
because of ideology.
User Communities are similar to Developer Communities, although the users them-
selves will hold the proprietary rights of the project. This is a way to reduce costs,
because multiple organizations might need the same or a similar project, so they can
pay a company to develop it under an open source license, while the user community
keeps control and can also let others contribute. This is for instance done in the GENIVI
Alliance, which is a group of car manufactures and suppliers, reducing their expenses by
sharing software.
Open Source Competence Centers does not in the same way contribute to develop
OSS projects, but they facilitate the use for instance in public organization and can con-
sult or and provide research and information about open source. They also do lobbying,
organize conferences, information platforms and things that can promote open source.
A competence center should stay platform- and vendor-neutral.
2.3
Vulnerabilities
According to RFC 4949 [20] a vulnerability is:
A flaw or weakness in a system’s design, implementation, or operation and
management that could be exploited to violate the system’s security policy.
==PAGE:22==
8
2 State Of The Art
There are many definitions, such as the ones described by ISO27005, NIST, ENISA, The
Open Group, etc., but they have the same intent. If a vulnerability exists, there is a
risk that it will be exploited by an adversary, resulting in a threat consequence.
Vulnerabilities can be grouped into memory safety violations, input validation errors,
race conditions, privilege confusion, privilege escalation, and arguably a couple others.
Examples of these are widespread from buffer overflows to SQL or other type of injec-
tions, cross-site scripting, or cross-site request forgeries.
An unknown vulnerability is called a zero-day vulnerability and it remains a zero-day
vulnerability until someone interested in mitigating the vulnerability becomes aware of
it. That moment becomes day zero, and from there on patches and workarounds should
be applied and rolled out. In general, the further after day zero the more strength the
vulnerability loses, as it becomes better known patches are likely to be developed and
reach more and more of the vulnerable machines.
To track and share information about vulnerabilities there are different entities around
the world that register and maintain information about the vulnerabilities. This informa-
tion is collected in a central register known as the Common Vulnerability and Exposure
(CVE) as described in Section 2.3.1.
2.3.1
Common Vulnerabilities and Exposure
The Common Vulnerability and Exposure (CVE) is a list of identifiers for publicly
known cyber security vulnerabilities, that ensures a single identifier for each individual
vulnerability. Previously there were multiple systems for registering vulnerabilities. The
CVE list is synchronized with the National Vulnerability Database (NVD) [19], which is
a U.S. Government repository that also stores further information on the vulnerabilities
and scores them with respect the Common Vulnerability Scoring System (CVSS) system
as explained in Section 2.3.2.
The system is maintained by The MITRE Corporation, which is an American, feder-
ally funded, not for profit organization. MITRE also maintains The Common Weakness
Enumeration (CWE). CVE IDs are submitted by one of currently 79 CVE Numbering
Authorities (CNAs), when a CNA discovers a vulnerability, or a third party (vendor,
security researcher, or private person) reports a vulnerability to a CNA. MITRE works
as the primary CNA where other CNAs belong to one of the four groups: Bug Bounty
programs, National and Industry CERTs, Vendors, and Projects or Vulnerability Re-
searchers. Vendors and Projects comprise the vast majority of the CNAs, maintaining
their own products and perhaps something related, while vulnerability researchers are
responsible for either their own products or specific vulnerabilities they are analyzing.
The bug bounty are currently only two CNAs, who each has a bug bounty service for
which they register CVEs. The CERT group has responsibility for vulnerability disclo-
sures for nations or industries and as part of this duty they can register CVEs if it is
within the scope.[4]

The CVE system is made as a dictionary using the CVE identifier as the key. CVE
identifiers were up until 2015 15 characters long, in the format ”CVE-YYYY-XXXX”,
with YYYY corresponding to the year and XXXX being a unique identifier. As the num-
ber of CVEs grew, the CVE10k problem would have been inevitable with the system, so
it was decided in 2015, that when needed an extra digit would be added. That way the
minimum number of characters in the identifier has remained 15, but it can now also be
16 or 17 characters long. All previous CVEs have kept their 15 character identifier.
As a vulnerability might accidentally be registered twice, or something might initially
be registered as a vulnerability and later discovered not to be, and because of the CNA
system, the CVE identifiers are not strictly sequential. This also means, that even if a
vulnerability with the identifier ”CVE-2020-9999” existed, it would not be certain that
there were 9999 vulnerabilities registered in 2020.
Originally the CVE system was simply an identifier that could reduce confusion.
According to Poul-Henning Kamp [11], main developer of Varnish-Cache, there has been
issues with CVEs being used as ”trophies”, so some security researchers would attempt
to register anything because it would look good on their CV. Furthermore the CNA
system was questioned due to overlap of responsibilities and CNAs like Airbus being
responsible for their own products. Lastly Kamp claimed that the process of registering
a CVE was more difficult for vendors who do not do it often, as the CNAs are suspicious
of people they are not used to submitting CVEs. This could be discouraging for a vendor,
if they know a vulnerability in their project but it is tough to get it registered, and they
might not care to use the CVE system.
2.3.2
Common Vulnerability Scoring System
The Common Vulnerability Scoring System (CVSS) is a standardized system for rating
the severity of vulnerabilities. The project originated from the National Infrastructure
Advisory Council (NIAC) in 2005, and the responsibility of further development was
delegated to the Forum of Incident Response and Security Teams (FIRST) the same
year.
The outcome from CVSS is a score between 0 and 10, with 10 being the most severe.
The score is made from up to three metric groups: Base Metrics, Temporal Metrics,
and Environmental Metrics. Only the Base Metric group is required to calculate the
severity as the others are not necessarily static. The temporal metrics will change to a
worse score as exploits are developed or become better when patches and fixes become
available. The environmental metrics depends on the other two metrics and the party
rating it, as the severity of a vulnerability is subjective depending on the assets it can
affect [5]. The three metric groups are shown in Figure 2.3 and described in more detail
in the following sections.

impact metrics deal with properties of the vulnerable component. As a vulnerability
might affect multiple components, the metrics should be graded according to the worst
affected component.
The Exploitability Metrics describe the properties of the vulnerable component
through the Attack Vector (AV), Attack Complexity (AC), Privileges Required (PR),
and User Interaction (UI). The attack vector metric describes the proximity required
to the vulnerable component, to exploit the vulnerability which could be remote from
a network, an adjacent network, or physical access. The attack complexity describes
whether an attacker is fully able to perform the attack as desired or if some special
circumstances are required. Privileges required explains the preconditions of the attack,
if the attacker must have some sort of access (be authorized) before exploiting the

2.4
Dependencies
Even when writing relatively simple programs, most programmers would happily import
a library or package to make it easier. Code is being reused to save time and resources,
and it is efficient. As projects are developed with this method, they will be dependent
on other projects. Furthermore, the project that the new project depends on, might
have even more dependencies. Luckily for the user, installing all dependencies is easy
with a package manager. Unfortunately, this creates a whole lot of new projects that
needs to be trusted for using the original project. By the principle of the weakest link, a
project is no safer than the weakest dependency, so it is important to keep track off all
packages that will be installed. On UNIX based systems package managers are a norm,
Windows also added one under Windows Vista but while many Unix users would know
about a package manager, barely any Windows user would, at least before the Windows
Store.
In Debian based systems such as Ubuntu, the Debian Package (dpkg), gives the
ability to manage packages. A package may depend on other packages or they can have
a conflict, these are relationships which are different from build dependencies. There are
7 types of relations as described below [21]:
• Depends The package is required for the package to work
• Pre-Depends Similar to Depends, but must be installed before installing the
package
• Recommends packages will usually be required but the package might work
without
• Suggests indicates that another package might be more useful together with one
or more other packages
==PAGE:28==
14
2 State Of The Art
• Enhances is similar to suggests, but it means that the functionality of the package
might be enhanced
• Breaks will not allow a package to be unpacked before the other packet is dis-
configured.
• Conflicts is stronger than breaks and will not allow the two packages to be un-
packed at the same time, even with one being dis-configured.
These relationship types are used by the package manager to ensure the correct
packages are installed for a package to work and that no existing package will be in
conflict with the new one. It is important to include this when considering the security
of one package.
2.5
Maintainability
A lot of research goes into Software Engineering and development of new software, while
Software Maintenance has been overlooked. Maintenance is important because it has
the potential to reduce the need for new development. Maintenance is defined as ”Modi-
fication of a software product after delivery to correct faults, to improve performance or
other attributes, or to adapt the product to a changed environment” by IEEE [12] and
ISO has adopted a similar definition. Maintainability is related to the complexity at-
tribute of trustworthiness, and there are many methods for measuring complexity. The
Master Thesis of Hassan Bhatti [2] describes several metrics and they are classified as:
• Process Metrics
• Product Metrics
• Resource Metrics
The process metrics are related to the development process. Examples of attributes
are duration, cost, and quality which are a not exact numbers. Lines of Code can also
fall within Process Metrics and is a precise attribute that is easy to measure but less
descriptive. Product metrics can be both external and internal, and can also include
things from the development process, such as documentation. The examples of internal
attributes are complexity, correctness, and size of the software, while external could be
usability and efficiency. The resource metrics are either physical resources like servers
or resources as in man-power.
2.5.1
Source Lines of Code
Lines of Code is one of the easiest measures to compute for an Open-Source Software
project, as it only requires counting lines and can be done with no manual intervention.

It can tell the size of the
project, but alone even this might not be comparable between implementation languages.
Source Lines of Code (SLOC) was originally intended to be used as a measurement of
productivity, but it is very ineffective because some code can take up many lines for
simple methods that are fast to write, while other more complex methods might be
compact and take more time [2]. Likewise coding style, or coding experience, can be
different from programmer to programmer, or it could even be auto-generated code
taking up a different number of lines. Experienced programmers will be able to write
very compact compared to less experienced.
Lines of Code is counted per line, but depending on the definition this count can
include blank lines and comments or only code lines. Table 3.8 shows that some projects

former removes blanks, comments, and standalone brackets and the latter only counts
semicolons. Counting semicolons makes it unusable by languages like Python.
A third approach is Comment to Code Ratio, which calculates the ratio of comments
compared to lines of code. This removes the description of project size, but is relevant for
the project complexity as it can give an understanding of how well the code is understood
and described.

Team
In most software development companies, there exists a structure where managers have

2.10
Summary
When analyzing trustworthiness as a concept of software, security is the aspect reoc-
curring everywhere. By the TSM and Figure 2.2 several other attributes from Software
Engineering have been identified, which can help provide a more nuanced representation
of the trustworthiness concept.
Security, as an attribute for trustworthiness, has the advantage that there is a well
defined database listing all known vulnerabilities, in the CVE register. This data can
easily be fetched from the NVD, and that also contains CVSS ratings, that classifies the
severity of each vulnerability.
Organizations within open-source are dependent on the individuals and the commu-
nity. Contributors work individually and have different motives, and the team behind
a project can therefore be seen as a team of individuals instead of one organization.
Similar knowledge was used by Nielsen in [15] to develop the tool TOSS - Trustwor-
thiness of Open Source Software. The tool combines the use of attributes from security
and Software Engineering to give a score on a 0-10 scale for the trustworthiness of a
given open source project. The data used for the score still needs analyzing, to show the
meaning of the score.

CHAPTER 3
Analysis
In Chapter 2 TOSS - Trustworthiness of Open Source Software was introduced with a
description of the equations comprising the trustworthiness score. This Chapter provides
a deeper analysis of this tool, the data sources used, and the outcome of the sub scores
in Section 2.9. These results are then used to suggest improvements and additions to
the system.
3.1
Data
The data used to rate the projects are collected from three sources: Open Hub, apt-
rdepends, and NVD. Each data source is described in more detail in the following three
subsections.
The differences between the data sources often lead to confusion, as some projects
are named differently for each service and there might end up being three or even four
different names for one project. An example is Chrome, where the open-source browser
that is the base for the Windows version of Google Chrome is called Chromium. Open
Hub calls it Chromium (Google Chrome), but with Chrome being its shortname. To
search for Chrome with apt-rdepends, a repository first has to be added to APT as there
is no chrome by default. When this is done, the apt name is google-chrome-stable. If
we used the apt-rdepends name for searching with CVEsearch, only one CVE would
appear, which is wrong. To get the correct search results the nvd name google-chrome
will be used.
3.1.1
Black Duck Open Hub
Open Hub originates from what used to be Ohloh.net, which was bought by Black Duck
Software in 2010. Black Duck decided to change the name to Open Hub in 2014. Their
own definition of their service is a directory of free and open-source software (FOSS),
which as mentioned in Section 2.2 is similar to OSS but a more neutral term than OSS
or Free Software.
For each project, Open Hub has data on the source code (lines of code, comments,
and blank lines), activity (commits), and community (contributors) per month. For
the contributors, it can be distinguished how many commits they have made each and
whether they are registered on Open Hub. There is also some limited information about
==PAGE:40==
26
3 Analysis
vulnerabilities per version, but more detailed vulnerability information is found with
cve-search as described in Section 3.1.3 below.
Open Hub provides an API1, which does not seem to be maintained anymore, and
is still called Ohloh API. It requires registration and only allows 1,000 requests per day
and it does not provide access to all the data used by TOSS. Therefore the information
used for rating projects is gathered by web scraping with the RoboBrowser library for
Python as done in [15].
Working with Open Hub proved to be increasingly challenging. The search function
often returns wrong projects if there are no exact matches, and sometimes the site
is taken down for maintenance which can last up to a day. Another issue was that
arbitrarily, the connection would be reset, i.e., Open Hub would reply to a request with
a RST packet crashing the program, or just time out. Most of the time this could be
solved by allowing a couple retries, but sometimes even 10 retries failed.
Open Hub obtain their data by analyzing code from version control systems, like
GIT from GitHub or other repositories. They rely on users to update information, add
projects, and provide links to the repositories. Adding a link to a repository worked
fine as the process was automated and after a couple of hours the project was updated.
However things that required manual action worked less well, such as reporting duplicate
projects which took more than a month to merge, and no confirmation was ever given.
While the data obtained from Open Hub for the most part was of good quality there
were a couple hick ups, such as one project that reported the number of source code
lines in the project to be negative.
At the time of writing, there are currently about 230,000 registered users on Open
Hub, where many of these never committed anything to an open source project, or at
least they have not claimed it.
Open Hub registers the name of committers on the
version control systems they have access to such as git, but unless a user claims the
commits to be theirs they will remain unclaimed committer ids. Similarly, Open Hub
has a feature for users to claim that they use a project, but as the number of active
users on Open Hub is very limited, it is limited what these numbers can tell.
3.1.2
Advanced Package Tool
As part of the Debian Package Manager (dpkg), the Advanced Package Tool (APT)
allows for easy installations and configuration of packages and their dependencies. This
is available by default on all Debian based Linux systems. When releasing a software
product, information about the dependencies should be included, so it they will be
installed automatically.
Included in APT is the rdepends package, which uses the APT cache to find depen-
dencies of a package, recursively so that dependencies of dependencies are also found. It
can only find the dependencies for the current version of a package.
For APT to work, names have to be an exact match and it can sometimes be a
challenge to find the right project, as projects might be released as different packages,

Naturally, if a package is not in the default repository used by APT, the repository

National Vulnerability Database
The National Vulnerability Database (NVD) [19] is a database storing all registered

While the

The tool, called cve-search [14], stores all the information in a MongoDB database,

package was used. As described in Section 3.1 it is important to use the correct names
when searching the database. As TOSS was searching NVD with shortnames, it gave
some problems.
The worst was a project called Community Scripts, which had the
shortname scripting. Searching NVD for scripting returned so many results that it took
a couple minutes to print it to the console and it crashed the search for CVSS scores.
The CVSS ratings by NVD only contains the base metrics as discussed in Section
2.3.2, meaning they should not change. However there are both CVSS v3 and CVSS
v2 ratings. The Temporal and Environmental metrics can be manually inputted on the
NVD website and it will provide the adjusted score, but this is not necessary for TOSS.

3.2.4
Uncertainty Factors
During the analysis, many factors adding uncertainty to the score were identified, in-
cluding the quality of the data used and the uncertainty about the results of different
search queries. The list below tries to summarize these and give a short explanation of
each.
• Projects not found NVD. If no vulnerabilities are found for a project, it could
of course be a good sign, but it could also mean that the name was wrong and
the project could not be found. There is no way to check if the correct project is
found.
• Correctness of results from CVE-search. Some search results could poten-
tially yield CVEs for multiple projects, or just a wrong project if the names are
too similar.
• Projects not found Open Hub. When retrieving a project from Open Hub,
None will be returned if it does not exist, so it is known that it is missing on Open
Hub. However the project will just be scored as secure, on no other basis than
benefit of doubt.
• User numbers from Open Hub. Open Hub has a feature for users to declare
that they use an OSS project. This data is used to for the user evaluation, for
projects with at least one vulnerability but ncve less than 5. The user numbers
are not representative and the usage is limited.
• Same contributor with multiple aliases. Open Hub has more than 230.000
users, and of those it seems that only few are active. Open Hub tries to put com-
mits by the same contributor under the same ’unclaimed committer ID’, however
there would often be multiple unclaimed committer IDs with the same name, just
typed differently or some as part of an email. This could have an effect on the
user evaluation if the same contributor has counted as multiple contributors and
otherwise it at least has an impact on the team score when not every commit is
counted.
• Missing data on Open Hub. Some projects had long periods where OpenHub
had registered the code base to remain the same size, most likely because of a
missing repository for that period. Likewise, this can have impacted the commits.
There were some other issues with the data such as negative lines of code for one
project.
==PAGE:53==
3.2 TOSS
39
• Versions. Registration of CVEs, code base, and more is done differently from
project to project and some projects are split into smaller projects depending on
the purpose. For instance looking for dependencies for MySQL requires specifying
mysql-server, mysql-client, or another MySQL project. For Apache, APT needs
’apache2’ as input, where this would return a wrong project on OpenHub.
3.2.5
Scoring Projects Historically
When scoring projects historically it is interesting to look at what has happened during
previous events, and to see the development of the score through out time. However, it
posed some challenges as most of the data retrieved only reflected the current state of
the project.
For the security score of each project, the task was simple for those with ncve above
5, as it was only dependent on CVEs and their respective CVSS scores. If the user
evaluation had to be done, it was a more complex problem as the list of contributors
had to be filtered for contributors that had not contributed before the desired point in
history to score the project. For the number of users of a project, Open Hub provides a
list of everyone who declared that they use the project. Unfortunately there is no way
of seeing when a user started using a project, so it is not possible to accurately do this.
For the maintainability score, finding the data was challenging because it was previ-
ously done by HTML scraping where only the current SLOC and TCL was available in
text format. Other methods for finding data about lines of codes were explored, such as
looking at repositories, mostly at GitHub, where plenty of tools are available to analyze
the code base. Unfortunately this requires cloning of the repository and of each version,
which is not very scalable.
Eventually it was discovered that the data used for the
graph showing code base evolution over time on Open Hub can be downloaded as a json
element. This gives both lines of code and comment lines. For some projects the data
quality was good, but for other projects it either had a couple of years with no changes,
possibly because of a missing repository for that period, or some strange large decrease
in code base, which did not coincide with major new releases. Some errors have been
removed during the span of this project.
Calculating the Team score historically is practically impossible with the current
ways of retrieving data. First, it requires getting the information on contributors for
this project, who had already contributed before the desired point in history and it
must be known how many commits they had made at that point. Next, information on
the projects they had contributed to at this point in time is required, along with their
security scores, which might also require further analysis for the contributors of that
project, if it is rated by user evaluation.
These calculations could be interesting as they could potentially also show which
contributors had contributed before vulnerabilities were introduced to a project, but
they would require analysis of practically every single commit to the project.

3.4
Suggestions for Changes
Based on the analysis above, this section proposes some changes to improve the scoring
system.
3.4.1
User Evaluation for all Projects
As shown in Table 3.7, for a large majority of the projects with ncve below 5 are rated
as secure, because no CVEs are registered, and they receive the benefit of the doubt.
While this is intentional, it does rate a lot of projects secure, even though they might
not be, but are just not used enough for the vulnerabilities to have been discovered and
reported. The disadvantage is that the scoring would be moved from one model with a
lot of uncertainty, to another model that is also very uncertain, with the user data on
Open Hub being far from exact.
3.4.2
Trending on Moving Averages
As discovered in Section 3.2.1.1, the trend line used for calculating the vulnerability score
is prone to big deviations and some extreme values. Moving averages could be used to
resolve this, and make the graph more smooth. Furthermore, reducing the number of
years used for the trend calculation could also be considered as currently it is the lifespan
of the project or as long as data is available.
The moving average would for this purpose be implemented with a size 3 or 5. If
the size is 3 and a list of data points were available, it would convert data point 1-3
into their average value for a new first point, then the average of data point 2-4 for the
second point and so on.
3.4.3
Forgetting
As discussed in Section 2.8.2, older data points should not influence a score as much as
newer data points. The way the security score is build, mainly on trends and averages, is
not compatible with a discounting function. The forgetting feature could either replace
or be a new part of the security score.
Equation 3.1 shows how the forgetting could be implemented, although it would need
to be adjusted to the 0-10 scale. As explained in Section 2.8.2, a forgetting factor of 1
means no scores are forgotten and 0 would mean that only the last element was kept. It
would require testing to find a good value.

3.4.4
Discount at New Releases
When new versions of software products are released, they often introduce new errors
and vulnerabilities. These are usually fixed through minor updates or in larger packs
such as the Service Packs known from Windows. The many vulnerabilities a new version
introduces could create extreme data points, which to some degree is expected after a
new version. Therefore it could be argued that these vulnerabilities might affect the
Trustworthiness score more than intended. To lower this effect, a discount could be
implemented, so the vulnerabilities would count less just after a new version, or it
could be done depending on the code base, either looking for changes larger than a
certain percentage in the code base, or anomalies in code base expansion, compared to
the normal trend. The disadvantage of this approach is that the security sub score is
supposed to reflect the security of the package, and by implementing a discount it would
not be reflected correctly.
3.4.5
Trustworthiness Tuple
Because of the way the Aggregated Security Score is designed, only the worst Security
Score of a main project or its dependencies is used for the Trustworthiness Score. This
scoring system is fair because an attacker would find the easiest way to attack the system,
which would be the weakest component.
It could also be argued that dependencies
could be replaced, either by other packages or possibly by the required features being
implemented. With this logic, it would be fair to also give some weight to the security
score of the main project.
Keeping the security score could be done by either making a percentage of the Ag-
gregated Security Score be the Security score of the main project, or alternatively by
turning the Trustworthiness score into a two-tuple. The tuple could have the Trustwor-
thiness score as one part, as it currently is and the other part being the security score
of the main project.
The advantage of the tuple is that it still reflects the state of security as it really is.
Furthermore it is a flexible method that could be easily changed or expanded to include
extra scores. The disadvantage is that it ruins the user friendly and easily comparable
0-10 scale.

Possible New Parameters
In Section 2.1 attributes of trustworthiness were defined. Although it is not the focus
of this thesis, this section briefly describes some attributes and ideas for how they could
have been scored.

3.5.1
Usability
Usability is identified as a group of attributes and especially Satisfaction could be an
easy attribute to rate using user feedback or ratings. However, user feedback will always
be subjective and prone to disgruntled users or users with a personal belief for or against
something. This would decrease the objectivity and thereby the trustworthiness of the
trustworthiness score.
Another attribute in the usability group was Learnability which could be related to
the documentation of a project. A user should of course have the desire and ability
to learn, but it also requires the right resources, which for a software project could
be the documentation. The documentation of a program should contain the necessary
information, but also not be too long for efficient learning. Besides the size, the content
and structure could be analyzed to find criteria for good documentation, but discussing
this in detail could perhaps be a thesis on its own.
3.5.2
Comprehensiveness of Tests
The quality and correctness of a program could be related to the tests of the program.
There are many different kinds of tests, and no single testing system is sufficient by
itself. Testing can be either white-, gray-, or black-box, where the first has full access
to the design and code of the system and tests both the structure and the internal
methods of the system. On the other hand black-box testing is done with no knowledge
of the design or inside of the system, and instead aims to test the functionality. Gray-
box testing combines the other two and tests the functionality of the system, but with
the knowledge of how it was designed.
White-box testing includes unit testing and
integration testing, where especially the first is easy to perform and check, although the
thoroughness of the test might be difficult to define, and a large test suite won’t help if
the tests are tautologies. System tests and acceptance tests are traditionally black-box
tests, but at least system tests can with reason be performed as gray-box tests. The
system tests run through complete test scenarios and the acceptance test is the final test
before a product is accepted. This would usually be a customer, but in OSS projects it
could also be someone within the open source community.
It is not trivial to judge how well a test suite tests all possible scenarios in a project,
and it is even more difficult to judge the completeness of the manual tests, but it could
be a valuable resource for rating the quality of software.

CHAPTER 4
Design & Implementation
The program TOSS - Trustworthiness of Open Source Software was designed by Nielsen
in [15] and the code is available on GitHub [16]. This Chapter introduces the design
created by Nielsen and explains the modifications that has been made to improve the
program.
4.1
TOSS Design
The program was designed to perform the calculations of trustworthiness described in
Section 2.9 from the open source data described in Section 3.1. Python was chosen as
programming language, because it is a high level programming language, that is object
oriented and great for data manipulation. Within academia, Python is widely used and
liked and there are many libraries that make data manipulation easier.
Figure 4.1 illustrates the design of the score system with the relations between the
sub scores and the imported data.
Figure 4.1: Design by Nielsen in [15].
The Aggregated Security score is using the scale opposite to the Trustworthiness,
Team, and Maintainability scores, because a severity rated 10 on a scale 0-10 is consid-
ered critical and bad, while a Trustworthiness score of 10 is the most trustworthy. To
==PAGE:62==
48
4 Design & Implementation
make up for this, the Aggregated Security score is subtracted from 10 as shown in Equa-
tion 2.2, before being calculated into the Trustworthiness score as the trustworthiness
security score. The Aggregated Security score is the highest security score of a project,
or any of dependency of that project. To find the dependencies, apt-rdepends is used.
The Security score is calculated in one of two ways, depending on the average num-
ber of vulnerabilities registered per year. If more than 5 vulnerabilities were registered
on average per year for the project, it is done based on CVE and CVSS data gathered
from the NVD database, through the cve-search tool. Alternatively, data from Open
Hub is used to give a score depending on the number of contributors and the number of
users.
Data from Open Hub is also used for the Team score, which is a weighted average
of the Contributor score, calculated for all contributors to the project. For each of the
contributors, the Security score for all projects they have contributed to is calculated.
Those security scores are combined to a weighted average to find the Contributor score.
The Maintainability score is the simplest part, only using data from Open Hub
about Source Lines of Code (SLOC) and Total Comment Lines (TCL).
4.2
Modifications
Running TOSS only gives meaningful results on Debian based linux distributions due
the use of apt-rdepends, which must be installed manually. There are some other re-
quirements that must be installed as well, including mongodb for the use of cve-search.
While this setup takes a couple hours, TOSS did not work as intended. The first major
error was the output from cve-search was not like a json element, which it was intended
to be.
Searching change logs lead to no explanations and it would have been quite
a cumbersome job to convert it into json with regular expressions. Eventually it was
found that it could be specified that cve-search should output it as json. This output
was different from what had previously been used in TOSS, but the code was modified
to accept it.
Another error was found when creating the team score, some projects could have a de-
pendency score set to None, which raised an exception in the ProjectContributorMetric
class. These projects were discarded.
There were also less critical errors like creating a trend on a dataset that was not or-
dered by year (making a trend over a year range 2015,2014,2016,2013 instead of correctly
ordered), and using the wrong name when searching for CVEs.
4.2.1
Cache
Because of the difficulties working with Open Hub, and because making 1000s of HTTP
requests is slow, creating a cache to store the data fetched from Open Hub and possibly
NVD was necessary. Dependencies and WebSearch are implemented as scripts, which
made the design for the cache not trivial, as there was no place to store the data persis-
==PAGE:63==
4.3 Summary
49
tently, without having to reload a file every time the script was called. A cache could
be implemented in the OSSProject class, which would be simple to implement and fast
as there were few recalculations. However, it would be desirable to store the bits of
data about the projects and dependencies as well as contributors, such that it could be
used both for analyzing it and so that changes to the calculations could be done without
reloading everything.
Another idea that was briefly explored at the end was the use of in-memory databases,
such as Redis1. Redis can store strings, hashes, lists, sets and bitmaps, but more com-
plicated structures such as dictionaries and nested objects would need to be converted
and it could cause some problems and require extra computation.
Each project has a list of CVEs, a list of CVSS scores for these, and a list of de-
pendencies. The dependencies also have their own CVEs, which also have CVSS scores.
The cache should work for a number of projects, and it would be optimal if CVEs and
CVSS scores did not need to be looked up again if two or more projects had common
dependencies, making the data structure even more complex.
All the data mentioned above could be stored in the OSSProject class, but sharing
it with the ProjectContributorMetric class would then be a challenge. In the end the
solution was make caches in the scripts, even though it requires a lot of reopening files.
While it is not super efficient, it at least ensured that the same project would not have
to be fetched twice from Open Hub, and it could be used for analyzing the data as well.
4.2.2
Moving Average
The moving average was based on CVEs per kLOC rather than just CVEs so firstly
the historic code development needed to be retrieved. As described in Section 3.1.1 this
could be fetched as a json element and urllib3 client was used for this purpose. The data
exists with a granularity of months but only years were need. It was decided to simply
use the datapoint from November, as it seemed unnecessary to calculate an average or
in other ways spend extra processing time on this.
The average was done over 5 values, to get a smooth graph while still having enough
data points to make the trend over. It was only implemented for the vulnerability score,
but it could easily be transferred to the security score as well.
4.3
Summary
The existing design of TOSS was in general very good, although there were some errors in
the implementation. The data sources, especially Open Hub, provided some challenges
and to cope with these caching was introduced. Because of the structure using scripts
rather than classes, the implemented cache is not ideal as it has to read the files many
times. This could be improved by changing to classes for some of the helping scripts,
passing more data back to the project class or by using an in-memory database.

CHAPTER 6
Conclusion
Trustworthiness in software affects both end users and developers or vendors. As an end
user, ones personal data, system, or possibly things the user did not know a software
project stored about them is at risk. For a vendor the reputation and thereby possibly
the entire business is at stake, if their security is breached, customer information is
leaked, and customers loose their trust in the system or vendor.
To be able to quantify trustworthiness some metrics had to be designed. Security
was found to be the most important metric, but aspects from software engineering
also contribute to trustworthiness. Besides security, two attributes were found to be
important and quantifiable in OSS projects with a reasonable effort: maintainability and
the experience of the team. This kind of system was already implemented by Nielsen in
[15], but required further work.
Maintainability is important for both further development of the project and also to
be able to maintain and update the level of software quality, when faults or vulnerabilities
are discovered. Rating the team behind the project gives an idea of their capabilities,
which should be reflected in the given project.
The Aggregated Security Score is weighted the highest in the Trustworthiness score,
and the Team score is also based on the Security score. All projects without known
vulnerabilities are counted as secure, which is very uncertain. The user evaluation, used
for projects with few vulnerabilities, favoured larger projects, but the aggregation of the
Security scores means that smaller projects have an advantage as only one bad project
would drag the entire score down.
The vulnerability score based on trends was found to be improved by implementing
a rolling average of vulnerabilities per year per kLOC.
The Aggregated Security Score still needs a redesign so that it reflects the uncertainty
in the ratings. The analysis of this project focused mainly on the non-user evaluation
part of the security score. A solution was proposed, by changing the the trustworthiness
score into a tuple that would still reflect the score as it is, and then in addition show the
security score of the project to indicate that the project might be good, but it depends
on insecure packages. This idea was abandoned as it ruined the simple scoring system
on a scale from 0-10 and would make it more difficult to compare packages.
The trustworthiness score provides an objective rating and it reflects that there
are problems with the security of large OSS projects, but it should be noted that the
uncertainty of the parameters used to derive the score is not reflected in the score and
it does rate these projects very harshly. The system could be used by a developer to see
which packages would be preferable to use or invest effort in maintaining or updating
to increase their own project’s trustworthiness score overall.

6 Conclusion
6.1
Future works
While there was put effort in to the Aggregated Security Score it would have been better
to look at the user evaluation rather than the vulnerability and severity score. This is
an area that needs focus. One possible direction for this is to look into belief theory or
subjective logic as briefly mentioned in Section 2.8
Although the focus is on OSS, all the vulnerability data is available for any project,
so it could be interesting to look into projects that have a reputation for being insecure
and see if any trends could be found in these data.
The user evaluation should be updated prior to this, but new attributes could be
added to the project, such as suggested in Section 3.5.
CVSS has the temporary metric, which is not reflected in the data retrieved from
NVD. The temporary metric shows the current state of remediation and exploits, so if
implemented it could give a more current score.
While the methods and time frames used, such as vulnerabilities or size of code base
per year, give a nice overview, it would have some advantages to work with vulnerabilities
per release/version instead.
The data received should be analyzed more carefully to make sure a CVE is relevant
for this specific project and also ensure that all the relevant CVEs are returned, as
they could be missed if the search criteria were in a different format, for instance if
apt-rdepends return the dependencies with with a name containing a version number,
in a way that NVD does not recognize.
