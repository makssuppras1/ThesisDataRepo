Abstract
The project presented in this paper deals with design, implementation and evaluation of object searching
system with mobile robot. The general idea is to obtain a system that provides high level of autonomy
and versatility and is able to complete the task given minimal input knowledge. First, the development is
focused on two crucial parts of such system - object detection and autonomous exploration of an unknown
environment. Then, it is presented how the complete object searching system is set up using these two blocks.
The evaluation of object detection is done in terms of ability to spot and correctly calculate the position
of the object present in ﬁeld of view of the robot. Autonomous exploration is tested based on checking its
ability to limit visiting parts of unknown environment that have already been explored. Final result of the
project is the system running on real robot that is able to ﬁnd planar or textured objects in sandbox laboratory
environments.

1.1
Motivation
Object search in an unknown environment with mobile robot can be considered a difﬁcult task as its both
parts, object detection and autonomous exploration, are known to be demanding in machine domain. The
ultimate goal of these two tasks is to achieve a performance that is close to the one of humans realizing
them. Certainly, getting towards that goal is worth any effort as number of possible applications of systems
involving object detection and autonomous exploration fused together to perform object search is signiﬁcant.
There has been many cases in recent history that after an unexpected catastrophic event, affected
area became inaccessible or contaminated. In such situations, limiting direct human involvement in post
catastrophic activities (such as searching for objects) is crucial. First of all, if the area is inaccessible for
humans, involving a robot that can get to it quickly and perform searching for source of failure or casualties
could vastly increase efﬁciency of the mission. Moreover, it could be observed many times that rescuers
paid with their lives or long term health issues after performing tasks in contaminated environments. Being
able to replace them with machines is necessary in the future.
Robots have been already applied in search and rescue missions. The unfortunate opportunity to use
them happened during the events following the collapse of World Trade Center in New York in September
2001. One of the issues addressed in an extensive analysis of performance of robots provided in Casper and
Murphy (2003) is the lack of autonomy. Therefore, the robots required constant human supervision which
affected the performance of the mission as people after long hours of cognitive effort and lack of sleep are
prone to making mistakes. According to Casper and Murphy (2003), the crucial ability of the robots that
would improve their autonomy is object recognition. That feature would enable live victim recognition with
limited human supervision. Autonomous exploration of affected terrain would further increase autonomy
of the robots in search and rescue missions. The overall increase of this property would be also beneﬁcial
if human resources for the mission are limited and there are multiple tasks to handle. In general, efﬁcient
cooperation between humans and autonomous (or semi autonomous) robots can safe many lives in the future.
Another possible area for applying object searching in an unknown environment with a robot is household
environment. Such functionality could be included in robot system used for assisting elderlies whose motor
skills are limited. Namely, object search could be an important part of the systems that are used for ﬁnding
and bringing objects to the users. As households are in general dynamic environments, object search together
with autonomous exploration is crucial for completing that kind of tasks. To give an example, the robot
could easily assume that a mug with tea is very likely to be found in the kitchen. However, after getting
there, knowing where exactly it is placed is impossible and that is where object search would be necessary.
Another important possible feature of object search for assisting elderlies in households would be ability to

1.2
Scope of the project
Project work includes choosing solutions and implementing two main parts of object searching system
which are object detection and autonomous exploration. The solutions are implemented as additional
functionalities of existing robot software. They are evaluated separately before building complete standalone
object searching system that is supposed to communicate with previously implemented blocks.
Project goal is to obtain an autonomous object searching system that is able to operate in unknown
sandbox environments of various characteristics and is capable of ﬁnding multitude of objects no matter
where they are placed in the environment. The solution also aims to limit amount of input data that has to be
provided to the system before starting the task.
Object detection is evaluated in terms of ability of the robot to notice the object if it is present in ﬁeld
of view of the camera. Moreover, the robot should be able to correctly calculate position of the object in
order to approach it. It is also checked if object detection part of the system is prone to false detections. The
overall performance is evaluated with respect to objects to be detected of growing complexity.
Autonomous exploration is evaluated by inspecting if the robot keeps visiting new areas of unknown
environments during exploration with limited comebacks to already explored places. It is assumed that
exploration takes place in an environment where it is possible to obtain reliable distance measurements to
objects around the robot.
The whole system fuses object detection and autonomous exploration and it is tested in terms of being
able to perform object search providing high level of robustness and efﬁciency. Robustness means ensuring
safe (without hitting obstacles) autonomous exploration, ability to spot the object with camera no matter
where it is placed in the environment and lack of false detections. Efﬁciency of the whole system can be
understood as lack of coming back to already explored areas in unknown environment and detecting the
object as soon as it appears in ﬁeld of view of the camera for the ﬁrst time.
As it is ﬁrst instance of object searching system, it should be built in a way that provides high level of
modularity and is easily reusable and possibly extended in the future.

CHAPTER 2
Object detection
2.1
Introduction
Object detection in computer vision can be deﬁned as the process of ﬁnding and identifying real world
objects in images or videos. In other words, the essence of detecting objects is obtaining meaningful
descriptions of the image scene. It is one of fundamental problems of computer vision which aims to imitate
the accuracy of object detection with human eyes and brain. Humans can recognize multitude of objects
very fast and with little effort, even if they are scaled, rotated or partially occluded.
Figure 2.1: General goal of object detection.
Object detection, which is very easy for humans, is much more challenging for machines. Problems
that are irrelevant for humans grow signiﬁcantly in computer vision. Main challenges for machines are
changes in scale and rotation of objects, lighting conditions and partial occlusion of object to be detected.
Another difﬁculty is that the same object can have various forms – cat is the perfect example of such object.
Furthermore, if one wants to detect objects of given class, variation of their appearance within that class has
to be taken into account.
In general, number of possible applications of object detection grows with growing performance of
detection algorithms. They have been successfully applied to numerous tasks. Most obvious one is face
detection in digital cameras. Another example is vehicle detection which can be used for automated parking
systems or estimating the speed of a car. Object detection is also widely used in manufacturing industry
in order to enable machines to detect some particular types of objects. There are many other applications
where recognizing objects is crucial.
What should be noted is that, for now, object detection in computer vision involves systems that are
meant to work for some speciﬁc object or class of objects. Increasing the generality of such systems is
nowadays main challenge of this task.

2.1.1
Methods of solving object detection task
First step of object detection is extracting a set of features of an image which should ideally be compressed
and descriptive. Here, one can distinguish two types of features – global and local.
Global features are gathered from all image pixels. Typical examples of such features are RGB or HSV
colour histograms. Especially the latter one is widely used as it is illumination invariant. One step further is
building feature vector based on colour moments of each colour channel – mean, standard deviation and
skewness. That way, 3x3 global feature matrix is obtained for the image. Another, more advanced example
of image features calculated globally is histogram of oriented gradients (HoG) introduced ﬁrst in Dalal
and Triggs (2005). Generally speaking, this method aims to get the shape of structures present in an image
by calculating the information about gradients. For each image block (usually 8 x 8 pixels), histogram of
gradients is built to obtain dominant gradient orientation that corresponds to shape of the structure present in
the block.
Local features (interest points, keypoints) are parts of an image that somehow differ from their immediate
neighbourhood (e.g. in terms of colour or texture). Such features can be small image patches, edges or
points. The basic task here is to get the set of local features that are repeatable and distinctive. Repeatability
means that the same keypoints of object are extracted independently from its rotation, scale or illumination
conditions. Distinctiveness is achieved when the information carried by the patch around keypoint is unique
so that different image structures can be distinguished. For example, the keypoint of part of soccer ball
should signiﬁcantly differ from the keypoint that is found in soccer shoe. Other important properties of
good set of local features are sufﬁcient quantity of features, computational efﬁciency in ﬁnding them and
robustness to image noise, discretization effects or blur. Some commonly used detectors are Harris Detector,
Laplacian of Gaussian Detector and Difference of Gaussian Detector. Once the set of local features has
been obtained, its content has to be described in a way that is highly distinctive and possibly invariant to
viewpoint and illumination changes. This has to be ensured since a descriptor should facilitate matching
between local features of the patches of the same parts of the object. Most popular methods of obtaining
keypoints descriptors are SIFT and SURF.
After extracting the set of either global or local features of an image, it is possible to obtain a description
of an image which is the crucial part of object detection task. Two different approaches can be distinguished
in that stage.
First involves learning phase which means that before being able to detect desired object, sufﬁcient
number of positive and negative examples of it has to be provided to the system in order to train the classiﬁer
(see Figure 2.2). Such classiﬁer can be understood as a border placed in feature space that divides positive
and negative examples as accurately as possible. After it has been obtained, detection of object in given
image boils down to getting its set of features and checking whether it is placed in positive or negative side
of the classiﬁer (border). This kind of approach is used in Kachouane et al. (2012) for real time human
detection, where classiﬁcation is based on support vector machine (SVM) classiﬁer trained in HoG space.
Another approach is performing direct matching of features between reference image of object to be
detected and another query image (as shown in Figure 2.3). Such matching is based on some dissimilarity
measure (e.g. Euclidean distance). In general, if dissimilarity between two keypoints is small enough, it may
imply (after fulﬁlling other requirements) that those features have been obtained from image patches of the
same object. The detection of object can be further determined by the number of matched features. This
approach is particularly feasible if used together with local image features and for detection of same instance
of the object. Its main advantage is that it does not require learning phase, the only input that is needed to
start detection is the reference image or set of images of the object to be detected.

2.2
Scale invariant feature transform (SIFT) method
One of the methods that has been particularly successful in numerous applications involving object
detection is Scale Invariant Feature Transform (SIFT) method proposed in Lowe (2004). SIFT is one of the
algorithms that extracts and describes local image features. The uniqueness of this method is determined by
the fact that SIFT features are extremely distinctive and highly repeatable. This enables successful matching
between images with different illumination, rotation, viewpoint and scale changes.
The advantages of SIFT method are relevant for object search with mobile robot problem. First of
all, the method enables direct matching of features which means that detection is possible after providing
just reference image of object to be found. Furthermore, the robot can recognize the object from different
viewpoints and in different scales (which depends on the distance from it). Finding sufﬁcient number of
matchings has to be ensured in such situations and it is claimed that SIFT method is capable of that.
The crucial parts of SIFT are detecting local image keypoints and obtaining the descriptor for each
of them. After that matching between reference image of object to be detected and query image can be
performed. All those steps are described in next sections of this chapter.
2.2.1
Detecting SIFT keypoints
First step of identifying SIFT keypoints is generation of Difference of Gaussian images. This is achieved
by ﬁrst applying Gaussian ﬁlter with given σ to the original image and then repeating that procedure with
growing value of sigma. After that the set of Gaussian-blurred images (called octave) is obtained. Then,

blurred images. The whole process is repeated after downsampling the original image by the factor of 2.
Both number of octaves and number of DoG images per octave are parameters that can be adjusted. Also
the initial value of sigma and multiplying factor across scales can be freely chosen. Lowe (2004) has shown
that optimal trade off between performance and computational cost is achieved while sampling 3 scales per
octave. Optimal number of octaves depends on image resolution. The higher it is, the number of octaves
should grow as well. As for initial value of σ, Lowe (2004) shows that growing sigma increases repeatability
of keypoints but also decreases efﬁciency. Taking this into account, the value is chosen to be σ0 = 1.6.

The orientation of keypoint is obtained from orientation histogram of pixels around this keypoint.
Namely, for each pixel in the region, orientation is calculated and added to histogram with contribution that
is weighted by the magnitude. Then, ﬁnal orientation is the bin of histogram with maximum value. Lowe
(2004) suggests to use 36 bins histogram covering 360 degrees. If there are some other bins of histogram that
have values higher or equal to 80% of maximal one, they are used to create another keypoints with different
orientations. Obtaining location, scale and orientation of keypoints enables to describe them relatively to
these parameters which results in invariance to them.
Next step is to calculate a descriptor of a region around the keypoint which aims to be highly distinctive
and in the same time invariant to changes in illumination and 3D viewpoint. Again, the starting point is
Gaussian-blurred image with the level of blur determined by scale of the keypoint. First, gradient magnitudes
and orientations of pixels around the keypoint (Lowe (2004) suggests 16x16 pixel samples) are calculated.
Then all gradient orientations are rotated relative to keypoint orientation. This allows to achieve orientation
invariance of a descriptor. Also, all magnitudes of the region are scaled with Gaussian weighting function
with σ equal to one half the width of the descriptor window.
Now, the 16x16 pixel sample is transformed into 4x4 subregions (such size has given best performance).
Each subregion contains the histogram of pixels that belongs to it which is build in the same way as described
previously. The only difference is that here, there are 8 bins instead of 36. To wrap up, each subregion
contains 8 amplitudes of bins and there are 16 subregions, therefore the descriptor has 128 elements all
together. Finally, the vector of descriptor’s values is normalized to unit length in order to reduce illumination
change effects. Last step involves clamping all vector elements to 0.2 and renormalizing it to unit length.
This is done to put more emphasis on distribution of orientations than on magnitudes.

To sum up the whole procedure of obtaining SIFT image features, its main steps are:
1. Getting octaves of DoG images
2. Obtaining the keypoints that are maxima and minima of DoG images across the scales
3. Discarding keypoints that have low contrast or correspond to an edge
4. Calculating orientation and scale for each keypoint
5. Calculating a 128-elements descriptor from pixel sample around the keypoint
6. Normalizing the descriptor to unit length with maximum value of elements equal to 0.2
Essentially, applying SIFT method to an image means transforming it from pixel representation to set of
local features each of which is described by 128-elements descriptor.
2.2.3
Applying SIFT to object detection
As mentioned before, one of the methods of object detection involves direct matching of features between
reference image of the object to be found and some query image. That is the approach that gives best results
when used together with SIFT features. It works particularly well for detection of the same instances of the
object.
One of the methods for matching SIFT keypoints between a pair of images can be found in Lowe
(2004). It involves ﬁnding two closest neighbours of each keypoint from image 1 in image 2 in terms of
Euclidean distance between descriptors of keypoints. Now the fact that correct match of the keypoint should
be obtained if the distance between its descriptor and descriptor of closest neighbour is signiﬁcantly smaller
than for descriptors of other keypoints is exploited. The ratio of distances between closest and second
closest neighbours that gives best results in terms of correct matching is, according to Lowe (2004), 0.45.

2.4
Attempt to improve SIFT - RootSIFT method
The tests discussed previously in this chapter show that while SIFT method performs well for planar
objects, there is still some room for improvement when it comes to detect 3-dimensional ones. As the
detection is based on matching between features, the aim is to increase the number of correct matches and
prevent from obtaining false ones.
One of the methods that is claimed to improve SIFT based detection, RootSIFT method, is described in
Arandjelovic and Zisserman (2012). The authors of paper point out that Euclidean distance dissimilarity
measure between SIFT descriptors which is used in SIFT method may not be the optimal one for object
detection. It is proposed to replace it with Hellinger distance.
The method boils down to two modiﬁcations of SIFT descriptor. First step is to L1 normalize the
descriptor:
xi =

(2.7)
where xi is the i-th element of SIFT descriptor and ∥X∥1 is L1 norm of that descriptor.
Then, each element of normalized vector has to be square-rooted. Now, after modifying SIFT descriptors in
that way, calculating Euclidean distance between them is equivalent to getting Hellinger distance of original
SIFT descriptors. All in all, RootSIFT method can be summarized as applying Hellinger kernel to original
SIFT vector. Converting SIFT features of an image to RootSIFT involves just modifying descriptor of each
keypoint according to steps described above.

2.5
Explanation of test results
The tests of both SIFT and RootSIFT based object detection show that the difﬁculty of the task of correct
feature matching (and hence object detection) grows from rather easy for planar and textured barcode object
to very difﬁcult for three dimensional SMR robot with most of surfaces being plain.
As for growing difﬁculty of detecting 3D objects, it has to be understood that SIFT keypoints are,
generally speaking, image blobs. A blob is an image patch that has some particular properties. In case of
SIFT keypoints, blobs are image patches around local minima or maxima of DoG images across scales
==PAGE:45==
2.6. PLUGIN IMPLEMENTATION
25
(see subsection 2.2.1). It can happen that some of the extracted blobs (keypoints) of 3D object correspond
to three-dimensional structures2. Such structures can look completely different if the viewpoint angle is
changed (which is not the case for planar objects). Therefore, a part of 3D object that is extracted as a
keypoint in one viewpoint can be often missed by the algorithm in other viewpoint. This shows that difﬁculty
of correct object detection grows in very basic level of blob detection. Even if the same three-dimensional
part of an object is extracted as a keypoint in different viewpoints, the fact that its aspect can be drastically
altered may result in obtaining descriptors that are too far from each other in terms of Euclidean distance to
be correctly matched.
Regarding the detection of textured and non-textured objects, textures are helpful in increasing the
distinctiveness of keypoint’s descriptors. Multiple patches of the same plain surface are difﬁcult to distinct
as essentially, they look the same and their descriptors are very similar to each other. Therefore, correct
matching between them is difﬁcult and less robust to illumination and shadow changes. This problem has
less signiﬁcance if the patches are textured.
2.6
Plugin implementation
After performing numerous off line tests of SIFT based object detection in Matlab, it is necessary to
enable the method to run directly on SMR robot so that in ﬁnal version of object searching system it is
executed in (or at least close to) real time. In order to achieve that, the functions of SIFT based detection are
implemented within a plugin which can be treated as the additional functionality of Mobotware 3.801 C++
software.
Regarding C++ implementation of SIFT method functions, VLFeat library is used again (as for off
line tests done in Matlab). To be more speciﬁc, C++ code of functions extracting SIFT features of an
image and performing SIFT matching is harvested from .c ﬁles of the library that are used to generate .mex
functions used in Matlab (following the suggestions from VLFeat SIFT with OpenCV (2012)). They have to
be modiﬁed such that mex utilities are replaced by standard C/C++ functions. Another major modiﬁcation
is changing input argument of function extracting SIFT features. In case of the plugin, the input image is
represented as vector of ﬂoat grayscale pixel values.
2.6.1
Functionality of SIFT plugin
Loading single or multiple reference images
Working with SIFT plugin should be started with loading single or multiple reference images of the
object to be detected later. In case of multiple references, the keypoints and descriptors of each image are
merged to form two single sets of them that have the same form as for single reference image. Therefore, all
procedures that are run after are executed in the same way no matter if single or multiple references are used.

computationally demanding. Possible solution to that would be limiting number of octaves or downsampling
the input image in the initial step. Unfortunately, both of the solutions would decrease the precision and
overall performance of SIFT based detection.
As for increasing the efﬁciency of matching, Lowe (2004) suggests to replace the procedure shown in
Algorithm 1 with an approximate nearest neighbour search called Best Bin First. The method provides
signiﬁcant speed up of matching but again the precision can be affected. This is not desired especially when
it comes to detection of 3D objects. As has been shown before, the margin for decreasing precision and still
being able to detect an object is rather small.
Multi-threaded detection
The solution that avoids the issues mentioned above is introducing multi-threaded detection. Unfortu-
nately, the delay between taking a picture by the camera and getting the result of detection is impossible to
avoid. However, what can be done is running the detection more often, each time on the different thread.
Thanks to that, it is not necessary to wait till the end of detection procedure and then take another picture with
camera, this can be done more often and therefore the information loss about what robot sees is decreased.
In order to include multi-threading within SIFT plugin class, std::thread (introduced in C++11 standard)
library is used. Namely, four threads that run SIFT based detection procedure described in previous
subsection are included in the class. Additionally, the thread that schedules the execution of SIFT threads is
implemented.
Scheduling ensures that four SIFT threads are run one after another between predeﬁned sampling time.
In that case, sampling time of SIFT threads is chosen to be 2 seconds, therefore the detection procedure is

Regarding the delay that occurs between start and end of SIFT based detection, its effect can be
particularly negative when it happens while the robot is moving. Namely, the robot can take the picture that
contains the object to be detected and then, while SIFT procedure is being run with that picture as input,
perform a manoeuvre which results in loosing the object from the ﬁeld of view. In such case, the object
would be detected but calculating desired rotation towards it would be impossible for the current position of
the robot. Therefore, in order to deal with such situation, the time stamp of taking a picture in the beginning
of each SIFT detection procedure is being saved. Having that information, the robot can backtrack to the
position that corresponds to the one that occurred in the moment of taking a picture containing the object to
be detected.
It should be noted that backtracking is not always necessary. In majority of cases, the delay between taking
a picture and obtaining the end result of detection will not result in loosing it from ﬁeld of view. In such
situations, the robot can just stop and perform the detection procedure again.
Other functionalities
Apart from core parts of the plugin, which are handling the reference images and performing single or
multi-threaded detection, there are some other functionalities implemented. The user can enable so called
RootSIFT mode which converts SIFT descriptors to RootSIFT form and performs all the operations in
RootSIFT feature space. Moreover, saving the images obtained from the camera together with calculated
matches to .png ﬁle is available. Also, Kinect camera of SMR robot can be tested by saving the picture taken

2.7
Summary
The tests of SIFT based object detection discussed in this chapter showed both advantages and weaknesses
of this method.
As for advantages, SIFT based detection can be executed right after providing single or multiple reference
images of the object to the system. Therefore, the amount of input information that is necessary before
starting the detection is limited to minimum. The method does not require any learning phase that usually
uses big portion of data. It has been shown that the method works well for planar and textured objects
(barcode) and objects that are essentially sets of such surfaces (cubic robot). Such objects contain many
distinct image patches and matching process gives satisfactory results robust to viewpoint angle, scale and
illumination changes.
SIFT based detection performed worse for objects that lack planar surfaces and distinct patches, as has
been shown in attempts to detect SMR robot. In general, there is a type of objects for which SIFT based
detection will perform rather poorly.
It should be also kept in mind that SIFT based detection is a method that works well when it comes
to detection of same instance of an object. That particular object has to be provided as the reference and
sometimes high quality images of it might not be available. In such cases one should use other methods of
detection that are suited better for more general class to which object to be detected belongs.
Last but not least, if one wants to use SIFT based object detection in real time (as for robotic applications),
it should be investigated if the hardware can ensure time of detection that is short enough. This is an issue
that occurs on SMR robot which has been partially solved by introducing multi-threaded SIFT detection.

CHAPTER 3
Autonomous exploration
3.1
Introduction
Autonomous exploration by the robot is another (together with object detection) building block of
the system that aims to solve the task of object search in an unknown environment. The general goal of
exploration problem can be deﬁned as maximizing knowledge over the particular area in possibly short
time period. Another deﬁnition, taken from opposite perspective, is minimizing the overall time needed for
covering an unknown environment.
As for object detection, the task that is rather intuitive for humans is considered challenging in machine
domain. Main problem is that getting an information about an unknown area is somewhat limited by the trade
off between the amount of information about it, gathered from sensors, and computational cost of processing
it and actually obtaining some meaningful knowledge (as it has to be done in real or close to real time).
Regarding sensors, their limitations have to be also taken into account. As an example, most frequently used
in exploration tasks laser scanners have the limited range (despite high cost). On the contrary, human eyes
can estimate distances effortlessly within much higher ranges than most laser scanners. Moreover, laser
scanners give information only about distances in some range of angles, the rest, such as presence of open
spaces or obstacles, has to be deduced by the robot.
The most intuitive application of autonomous robot exploration is reaching the area that is inaccessible
or dangerous for humans. So far, robots have successfully explored volcanoes, abandoned mines, the sea bed
or even other planets. Autonomous exploration is also a crucial element of robot systems used in search and
rescue missions.
3.1.1
Autonomous exploration - approaches
There are many exploration strategies that have been proposed so far. The simplest ones involve choosing
random direction of exploration from a set of detected open spaces. Such strategy performs surprisingly well
when less than 50% of area is explored.
More complex approach, called frontier-based (introduced in Yamauchi (1997)), searches for the borders
(frontiers) where known space ends and unknown starts. Then all found frontiers are prioritized depending
on their size, presence of obstacles and distance from other frontiers. In the end, robot chooses to move
towards a frontier that has highest priority. Main advantage of frontier-based method is that knowing the
structure of the environment to be explored is not necessary, the method has been shown to perform well in
both cluttered and open environments.
Another approach called multi-objective or multi-criteria exploration introduces multiple utility functions
in order to allocate desired locations (in terms of exploration) for a robot. The utility functions depend on
31
==PAGE:52==
32
CHAPTER 3. AUTONOMOUS EXPLORATION
several factors such as presence of open areas, the distance from the current position to desired locations and
presence of obstacles. For each potential desired location, the value of utility function is calculated and the
robot chooses the one that gives the highest utility. The examples of this kind of exploration approach have
been shown in Calisi et al. (2007) and Marjovi (2009).
Next interesting group of methods is behavioural-based exploration. Such methods combine basic
reactive behaviours in order to come up with efﬁcient ﬁnal exploring action. They are implemented as
layered control systems where each layer is a single behaviour. Such approach has been thoroughly depicted
in Brooks (1985).
Figure 3.1: Layered behavioural-based control system.
Some examples of predeﬁned behaviours in such strategies are wall following, avoiding obstacles or
previous locations, looking for open spaces or building maps. It has been shown that, while behavioural-based
methods rely on local knowledge and behaviours reactive to such knowledge, complex global exploration
can still be achieved. Another advantage of this kind of approach is modularity. As there is clear distinction
between individual behaviours, it is simple to add new ones or remove some that occur to be obsolete.
A particular solution using behavioural-based exploration that has given promising results is presented in
Cepeda et al. (2012). It has been decided to base the solution of autonomous exploration of object search
system on that method. The main motivation is that the authors of the paper claim that the method resulted
in similar performance as more sophisticated solutions. Furthermore, the method is rather computationally
cheap. This factor is especially emphasized after experiencing slow execution of SIFT based detection run
on SMR robot. What is more, the solution (and behavioural-based methods in general) gives a lot of freedom
in terms of possible modiﬁcations. Lastly, the method uses laser sensor as a tool for measuring distances
around the robot which makes it well suited for applying to SMR equipped in such sensor.
3.2
Outline of behavioural-based exploration
The solution proposed in Cepeda et al. (2012) deﬁnes four different behaviours, out of which three
are relevant to single robot systems (fourth one is concerned on multi robot exploration). Namely, these
behaviours are avoiding obstacles, searching for open spaces and avoiding past locations. The former two of
them involve purely reactive control as the robot takes into account only its current state while choosing the
action. Avoiding past locations requires having a spatial memory that stores previous locations. Therefore,
the overall solution puts together the simplicity of purely reactive control with deliberative approaches.
After obtaining sensor data, the individual behaviours are fused into ﬁnal action by either subsumption
of the output of one behaviour over others or weighted summation of behaviours’ outputs. To be more
speciﬁc, weighted summation takes place when ﬁnal action is executed according to avoid obstacles and
avoid past locations behaviours. In such case, looking for open spaces is disabled. Then, every given number
of iterations (speciﬁed by wandering rate) of executing the exploration action, looking for open spaces

Avoiding obstacles
The most basic behaviour regarding autonomous exploration is obstacle avoidance. The algorithm has
to ensure that exploration is executed safely and the robot does not hit the objects nor gets blocked in the
corners. The execution steps of this behaviour are presented in Figure 3.3.
It can be seen that there are two parameters that have to be set for this behaviour - ObstacleAwar-
nessThreshold and TooCloseThreshold. The former deﬁnes a range of distances between obstacles and the
robot for which the robot has to undertake an avoidance action. The latter deﬁnes a range for which obstacles
are too close. Being too close to obstacle means that the robot has to move back to ensure safe exploration.
The actual values of these thresholds are mentioned in section describing tests of the whole exploration
strategy.
Locating open areas
Next behaviour that is the part of exploration strategy aims to ﬁnd the largest open space in which robot
ﬁts. This action is triggered every set number of iterations which is deﬁned by so called wandering rate. It is
claimed in Cepeda et al. (2012) that this behaviour is very important for overall performance of exploration.
The procedure is graphically presented in Figure 3.4.
The parameter that is present in this behaviours is OpenSpaceThreshold. It deﬁnes the minimum distance
measurement that can be treated as open space.
Avoiding past locations
Avoiding past locations behaviour has been introduced ﬁrst in Balch and Arkin (1993) as a solution to
box canyon and other navigational problems. As it aims for gathering newest, unvisited locations, it is also
useful in autonomous exploration. First of all, the method introduces spatial memory which means that the
information about visited locations has to be stored. The value of desired heading angle is calculated based
on visit counts in three rectangular regions in front of current position of the robot, see Figure 3.5. This kind
of approach can be understood as leaving the chemical trail in visited locations. Such trail is later used for
deciding where to move.

Exploring
This part of the algorithm fuses the outputs of the behaviours described above in a way that has been
also shown in Figure 3.2. Two states can be distinguished here - ﬁrst one is responsible for looking for
open spaces and another fuses the output of avoiding obstacles and avoiding past locations behaviours.
An important thing to mention is that once looking for open spaces behaviour is triggered, it subsumes
(overrides) the output of other behaviours. That is why, an important parameter of the algorithm is how
often such triggering takes place. Cepeda et al. (2012) suggest wandering rate of 10% which means that
triggering is done every 10 iterations of the whole exploration algorithm. As for fusion between avoiding
obstacles and avoiding past locations, there are two important parameters that determine the output angle -
AvoidPastGain and AvoidObstaclesGain. Typically, avoiding obstacles has higher priority than avoiding
past locations and therefore AvoidObstaclesGain > AvoidPastGain. The execution of exploration strategy is
graphically presented in Figure 3.7.

3.4
Plugin implementation
After all off line tests described in previous section, the functions handling the behaviours of exploration
strategy (together with some other auxiliary functions) are implemented as ’explorer’ plugin of Mobotware
3.648 C++ software. The general idea is that all the processing of laser range scan that has to be done in
order to obtain some meaningful information is executed within the functions of the plugin. In other words,
all the procedures that require laser range measurement as an input are implemented in the plugin.
The starting point is getting distance measurement from the laser mounted in SMR robot. This is achieved
using functions implemented within the class ULaserData which is part of Mobotware software. Using that,
it is possible to obtain the distances as an array of double type values together with arrays of angle values
in degrees and radians. As may be expected, i-th element of array with laser distances corresponds to an
angle from i-th element of array with laser angles. Initial step of processing laser data involves discarding
the scans that correspond to the angles lower than -90 degrees and higher than +90 degrees. It has to be done
because scans from such angles are limited by the wheels of SMR robot and therefore they are meaningless.
After doing that, the laser scan from angular range of -90 to +90 degrees is ready to be applied as input to
plugin functions. Regarding the functions, they are implemented such that their outputs have compact form
which is easy and fast to send to the other parts of the whole object searching system.
What is also important here is that some of the parameters mentioned in the description of explo-
ration strategy are deﬁned in the plugin. This includes ObstacleAwarenessThreshold, TooCloseThreshold,
OpenSpaceThreshold and also angular range of minimum size of open space and sizes of rectangles used in
avoiding past locations procedure.

Getting distance between the laser and detected object
That procedure is necessary to be used after the robot detects the object and turns such that it is directed
towards it. After that, it is desired that the robot approaches detected object. In order to do that, the
distance between the robot and the object has to be found. It is achieved by getting the minimum distance
measurement from the subset of measurements that corresponds to angle range in front. That range is deﬁned
as -10 to +10 degrees and the minimum distance from that range is taken to ensure that the robot will not
crash with detected object and approach it safely. The output distance obtained in this procedure will be
further used in MRC’s commands.
Getting information about occupancy of rectangles in front of the robot
The function iterates over all distance measurements obtained from the laser and checks if any of them
indicates that an obstacle lies inside one of the rectangles. As described before, this is done by checking if
3.10 holds. The function outputs three bool values corresponding to occupancy of each rectangle. Logical
’1’ means the rectangle is free and ’0’ - rectangle is occupied.
Detecting if the robot is in narrow corridor
Implementing the procedure that detects if the robot goes through narrow corridor is necessary to prevent
it from oscillatory behaviour caused by potential situation in which robot ends up in previously visited
corridor and tries to avoid past locations but has no space to do that. The robot would constantly try to
deviate from past route but on the contrary avoiding obstacle would direct it back there and so the robot
would keep turning in one direction and then the opposite. Therefore, avoiding past locations should be
disabled in corridors.
As most of the time the robot travels parallel or close to parallel to the walls of narrow corridor, its width can
be checked by summing distance measurements corresponding to +90 and -90 degrees. Obtaining the value
below speciﬁed threshold CorridorThreshold implies that the robot goes through narrow corridor.
Other functionalities
Apart from main parts of the plugin, there are numerous other functions that are used by the main ones.
Moreover, logging laser measurements together with corresponding angles has been implemented. It is called
every time any of the functions using laser data as an input is called. The logs with input laser measurements
can be used in diagnosis and debugging.

environment. Obviously, such knowledge is non existent in real world unknown environment however it can
be assumed that there are other ways of obtaining precise information about robot’s position.
3.5.1
Methodology of tests
As said before, the tests take place in two types of sandbox environment. For each set of tested parameters,
there are two runs of exploration starting from two different positions. As for evaluation, the grid of 0.5 x
0.5 m squares is applied to sandboxes (as shown in right subﬁgures of 3.14 and 3.15). Using such grid, the
performance measure of exploration rate can be used. It is deﬁned by the percent of visited squares over
time. It can be observed here that in order to detect an object there is no need for full exploration in terms of
odometry, it is rather the ﬁeld of view of the camera that should cover the whole environment. However,
exploration rate gives a valuable information about exploration redundancy, whose presence means that
robot keeps visiting already explored places. This corresponds to ﬂat regions of exploration rate curve over
time.
Another way of evaluating exploration algorithm is plotting the path of each run and visually inspecting
if all crucial places of the sandboxes have been visited.
As advised in Cepeda et al. (2012), initial tests involve exploration using only avoiding obstacles and
looking for open spaces behaviours. This kind of strategy is so called ’safe wanderer’ and can be used to
determine the optimal value of wandering rate. This parameter is expressed in per cents - it determines

3.7
Summary
Cepeda et al. (2012) provides a high level description of exploration strategy that fuses purely reactive
actions (avoiding obstacles, looking for open spaces) with deliberative approach that uses spatial memory.
The low level implementation to actually achieve the behaviours is not speciﬁed and therefore it has had to
be carefully tested off line with laser measurements as input and in simulator.
One of the advantages of the method is short path between sensing and acting (see Figure 3.1). This
results in consuming less computational power comparing to other exploration methods brieﬂy described in
the beginning of the chapter. Another signiﬁcant advantage is that using subsumption architecture, as in that
algorithm, provides high level of modularity. It is easy to include new behaviours in exploration strategy
and fuse their outputs with previously implemented ones to get ﬁnal action. Moreover the method does not
require any sophisticated mapping technique.
It should be noted that using spatial memory for avoiding past locations requires reliable localization. In
case of SMR robot extended Kalman ﬁlter has been used which utilizes the knowledge about predeﬁned
walls. Obviously, when it comes to real life autonomous exploration of unknown environments, other
methods of localization would have to be used.
Cepeda et al. (2012) claims that no knowledge about the environment to be explored is required for the
algorithm, however the tests in sandboxes presented in this chapter prove that it may be beneﬁcial to have
some overview of them. This is helpful in setting the parameters of the method shown in Table 3.1.
As for test results, it should be noted that in the task of object search in an unknown environment it is
not necessary to cover the whole environment to get exploration rate of ≈100%. The aim is rather to cover
all the environment with ﬁeld of view of the camera for which the object can be detected. Therefore, it is
important to ensure that all crucial places of the environment are visited and that is why exploration route has
been shown in the tests. It can be concluded that behavioural-based exploration strategy should be capable
of enabling the robot to ’see’ all (or close to all) the environment. This will be evaluated further in the thesis
where all the system of object searching is tested.
Enhancing the performance of presented behavioural-based exploration might involve introducing spatial
memory also for choosing open spaces. Namely, the robot could save its position in the moment of choosing
open space. Then in case of looking for open space again from approximately the same position as before,
the robot could avoid choosing the one that has been already selected before. Another enhancement would

to it in another. For example, if object is detected in detection thread, this information has to be sent to
exploration thread in order to stop exploration procedure. Detailed description of both state machines is
provided in following subsections.

that the object has been detected is received, it stops the robot and sets ﬂag that conﬁrms ﬁnishing exploration.

5.5. IMPLEMENTATION ISSUES
79
The tests emphasize very important feature of object searching system that its two main blocks facilitate
and complement each other. If the object can be easily spotted by object detection block, there is more of
a margin for the performance of autonomous exploration. On the other hand, if object detection performs
poorly, exploring area well still often does not cause successful ﬁnding of the object. Generally speaking,
both blocks of the system have to ensure a certain level of performance, otherwise ﬁnding the object in
unknown environment gets very difﬁcult.
It should be noted that in order to test the system fully, more runs should be done and also more locations
of object to be found could be introduced. However, test runs presented in this chapter can give some
overview of performance of the system and its limitations.
5.5
Implementation issues
Setting up the whole system and performed tests has revealed some issues that are caused by building
a system in a way that plugins and standalone application are separate entities and hence they have to
communicate with each other using TCP/IP sockets. This is particularly problematic in executing exploration
because it introduces delays between sending a command to the laser, getting a response in standalone
application and sending driving command to the motors. Each delay takes ≈0.1 second and therefore it
makes it impossible to close the loop between desired heading angle and current heading. Rather than that, it
is necessary to execute the manoeuvres in sort of stepwise manner. Namely, procedure calculates heading
angle, sends it to motors and makes sure to wait long enough such that the whole manoeuvre has been
executed before sending the new command.
This makes it necessary to run the exploration with low speed such that the negative effect of the delays
is decreased.
In defence of using the plugins, such solution is beneﬁcial in terms of the ease of reusing the function-
alities that are provided by them. They can be used in the future as building blocks for other systems. In
general, the solution with plugins and standalone client application provides a high level of modularity of the
system. Also, as mentioned in previous chapter, using plugins enables taking advantage of pre implemented
functionalities of Mobotware software.
The way to speed up the system would involve implementing an application that is a single block and
has a direct connections to all actuators and sensors of the system. That would enable exchanging the data
at much higher rate. This would be especially beneﬁcial for autonomous exploration in which data ﬂow
between the laser and the motors could be more efﬁcient.

CHAPTER 6
Conclusions
The goal of the project was to obtain object searching system implemented on mobile robot that is able to
operate in various types of sandbox environments and can detect multitude of objects. Two required features
speciﬁed in Chapter 1 are robustness and efﬁciency. Another objective was to provide a modular solution
that can be easily reused or extended in the future.
It can be concluded that robustness in terms of safe exploration has been achieved. The robot has not hit
any obstacle in all the test runs in both sandboxes. Another factor determining robustness of the system is
the ability to see (not necessarily detect) the object no matter of its location in the environment. This feature
has been almost fully achieved. There are only two test runs (see Figures 5.11 and 5.26) in which object
search is not obtained within the deadline because the robot has not had a chance to see the object in camera
images. When it comes to lack of false detections, this has been achieved in searching for barcode and cubic
robot. However, false detections of SMR robot have been registered. This is caused by the fact that SMR
lacks planar surfaces and textures which facilitate object detection with SIFT method.
Efﬁciency of the system is determined by lack of exploration redundancy and detecting the object right
after it is spotted for the ﬁrst time. Inspecting the routes of the robot in test runs presented in Chapter 5
reveals that in most cases exploration is efﬁcient. However, sometimes exploration redundancy is very
difﬁcult to avoid due to characteristic of the environment. When it comes to detecting the object after it is
spotted by the camera for the ﬁrst time, this has been achieved for barcode and cubic robot. If the robot
manages to reach the position where it sees them, the detection takes place. This is not the case in searching
for SMR robot. Not detecting it when the robot is able to see it with the camera results in exceeding the
deadline of the search.
The system provides high level of modularity with two separate plugins and standalone application
communicating with them via TCP/IP sockets. It is easy to implement handling new commands within
existing plugins or create a new plugin and connect it to standalone application. However, such solution
is not free of disadvantages. The main one is rather long time needed for exchanging the data through
sockets. This was especially harmful for exploration part of the system where it was impossible to have fast
connection between laser and motors of the robot.
6.1
Future work
As shown in the tests of object detection in Chapter 2 and in the evaluation of the whole system depicted
in Chapter 5, SIFT based detection method has its limitations. This was particularly visible when it comes to
detection of SMR. Therefore, in order to be able to detect objects of such type, it is necessary to introduce
another detection algorithm to the system. This could be one of the methods that is classiﬁcation based and

requires learning phase before being able to detect an object. However it should be noted that these methods
require big amount of labelled data in order to obtain the classiﬁer. Ideally, object searching system could
have Internet connection to databases of labelled images and that way it would be able to train itself in terms
of detecting objects of given class. Having two available detection methods (SIFT based and classiﬁcation
based), the user would be able to choose the one that is optimal for particular object to be found. Another
possibility would be running two different object detection methods in two separate threads in parallel and
obtaining ﬁnal detection result as a fusion of these two detection methods results.
Another possibility to improve object searching system would be introducing user interface. Such
interface would be useful for human supervision of the system. For instance, the user would be able to
choose the methods of object detection and autonomous exploration and adjust their parameters within the
interface before the run. Then, during the run it would be possible to observe the state of the system and
provide a supervision if necessary. That could include manual driving of the robot to the places that have
been missed during autonomous exploration.
Improving the performance of the system also requires hardware enhancement comparing to what is
offered by SMR platform. That would include increasing ﬁeld of view of the robot which can be achieved by
adding more cameras. In particular, having a camera directed towards back of the robot could be beneﬁcial.
That way, less exploration effort would be necessary as the robot would not have to approach objects only
from the front in order to spot them. Also introducing stereovision could be helpful in estimating the distance
towards detected object. Moreover, it could be seen that execution of sophisticated methods such as SIFT
algorithm takes to much time on SMR’s computer. Replacing it with a faster one could enable running the
whole system with higher rate.
