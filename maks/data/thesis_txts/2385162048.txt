Summary
Virtualization is progressively getting popular with the wide adoption of Cloud Com-
puting and the promising future of Network Function Virtualization (NFV) in Infor-
mation and Communication Industry. Flexible and prompt resource provisioning is
one of the key features that are available in a virtual environment. Resource manage-
ment has been being one of the challenges for stakeholders to cut down on the waste
such as cooling cost and boost the profit.
The waste in cloud is underestimated by the cloud users. Respondents estimate
30 percent waste, while RightScale has measured actual waste between 30 and 45
percent [1]. Efficient resource scheduling generates more profits with the same amount
of resource for cloud providers. Similarly, operators are also facing the challenge of
efficient resource management when they are shifting their services to a cloud-based
deployment to avoid vendor lock-in and reduce cost. Enterprises that are adopting
cloud strategy encounter the same issue of resource management to reduce cost as
well.
The goal of this thesis is to explore the ways of improving the resource efficiency
in a virtualization environment.
This thesis project investigates different aspects
including virtualized technologies, resource metering, time-series data storage, data
manipulation, etc.
A priority-based preemptive solution is proposed and verified
in a cloud environment deployed by OpenStack, which is one of the most popular
open-source cloud deployment software. In the implementation, both resource and
resource consumers are tagged with different priorities.
The algorithm for cloud
management is devised to meet the resource demand of as many consumers as possible,
but also guarantee consumers with higher priority when the resource is in shortage.
The source code is available in GitHub at URL: https://github.com/DTUbigdata/
Thesis_Source_Code.git

Introduction
1.1
Background
Back to the mainframes days in the early 1970s, virtualization technology emerged
and achieved initial commercial success in IBM 370 series [2] under the key driver of
server consolidation. Server consolidation essentially utilizes the capability of running
virtual servers simultaneously on the host server to improve the utilization of hardware
resources. Contemporary servers are much more powerful and more virtualizable at
much lower cost than it was before. Specifically, virtualization has also been extended
to all types of hardware resources including CPU, memory, NIC, disk, GPU and so
forth. For instance, Intel and AMD both developed and integrated hardware-assisted
virtualization features into their CPUs named as Intel-VT and AMD-V respectively;
Extended Page Table (EPT) addresses memory virtualization bottleneck; Single Root
I/O Virtualization (SR-IOV) helps network performance. Meanwhile, both commer-
cial and open-source hypervisors are evolving to utilize those hardware features, in-
cluding VMWARE, KVM, Xen, etc.
Some high level orchestration platform are
emerging aiming at resources coordination and orchestration including OpenStack
and CloudStack, etc.
Just at the right moment, emerging paradigms such as Cloud Computing, Net-
work Function Virtualization (NFV) are proposed and further developed with the
support of those virtualization technologies and open-source communities. The pay-
as-you-go model of cloud computing put requirements on cloud provider to utilize
finite resources to achieve the potential maximum profits by serving and satisfying
their customers according to their Service Level Agreement (SLA), which requres
cloud providers to exploit available resources as efficient as possible. In NFV, net-
work service providers are shifting their services from proprietary platform to the
commercial off-the-shelf (COTS) where optimized resource usage of the COTS helps
significantly reduce their the Capital Expenditure (CAPEX) of purchase and Oper-
ating Expenses (OPEX) of space and power. For example, 5G technology is paving
the way for enoumous bandwith intensive services and new design challenges such
as radio access network’s (RAN) functional, architectural and commercial viability
comes up [3], where integration with NFV can be alluring.
Not only lowering the cost of providers and raising revenue, resources usage opti-
mization can also cut down the carbon footprint of resource pools from the perspec-
tive of environment-friendly consideration. The carbon emission released by the data

1.3
Solution Proposal and Related Work
This thesis proposes a general solution to tackle the challenges faced by stake-holders
regarding cloud resources management. In the following subsections, the solution is
introduced at a high-level and related work will be reviewed.
1.3.1
Methodology and Solution Proposal
Generally speaking, a research can be conducted using simulation method or in a real-
world scenario. In this thesis project, both of them plays important roles in different
steps. Firstly, an cloud environment with an amount of of resources needs to be set
up for the later testing and verification. Instead of using simulation tool, OpenStack,
one of the most popular cloud computing framework, is chose to deployment a real
cloud environment with physical hosts, which makes the solution more realistic by
considering more critical factors such as response latency, status transition time, etc.
Secondly, the testing workload that imposed on the cloud environment is created by
simulation instead of real application workload traces, because of the difficulties to
reproduce the real application workload.
==PAGE:17==
1.3 Solution Proposal and Related Work
5
With this methodology, the project is dedicated to provide a state machine and
preemption based cloud resource scheduling using virtual machine (VM) live migra-
tion. It is assumed that there are services with various priorities running in the cloud
and the cloud resources can be categorized into multiple clusters or multi-site clus-
ters, where each cluster is reserved for different priority class of VMs. The idea is to
achieve the following goal: any VM can be scheduled on different cluster when there
is no resource contention; high-priority VMs must be guaranteed when resource con-
tention exists and the reaction should be prompt; low-priority VM can be preempted
if necessary; VM should be consolidated into as few physical server as possible when
the overall resources utilization is low enough in order to switch off physical servers
for energy saving.
1.3.2
Related Work
Researches on energy savings and resources usage optimization in cloud environment
and data centers are hot and popular in recent years.
Bichler et al. [12] proposed static placement solution by using VM’s predefined
resource demand to optimize the overall resource allocation. Similarly, Jin et al. [13]
proposed solution of virtual machine (VM) placement under multiple and dynamic
resource demand when the VM creation requests arrive. By treating VM placement as
bin-packing problem, the near-optimal VM placement algorithm is executed to choose
the physical server. However, the workload might vary over time in real life and the
resource planning might become not optimal later. With resource over-provision, the
problem can be alleviated, but it can also cause resource shortage at some point,
which hurts the VMs’ performance.
Instead of a static VM placement solution, a dynamic and continuous resource
optimization can be more powerful to consider time-varied workload and avoid also
workload overload. Many researches have been done by using VM live migration,
which can re-locate VM from one physical server to another with a limited downtime
(depends on migration strategy, as well as available bandwidth and VM memory us-
age characteristics). Anton et al. [14] design a framework in parallel with OpenStack
to deal with VM consolidation in OpenStack cloud, but current OpenStack has incor-
porated necessary metering function as well as necessary Application Programming
Interfaces (API). Fetahi et al. [15] proposed dynamic resource allocation also in an
OpenStack Cloud, it differs from this thesis in several aspects: it is implemented in
a distributed way that might have potential problem of convergence; it uses external
metering tools while OpenStack built-in tool is adopted in this thesis project; it does
not consider different priorities of VMs and clusters. Similarly, [16][17][18] do not
consider priorities either, which is desired by stake-holders to differentiate services.
Takeda et al. [19] considered the physical server ranking by categorizing server and
use First Fit Decrease to pick a physical server to accommodate VM, but it only
employed two threshold to define the states of physical server, which might not be
enough considering there might be bursting workload that can appears in short pe-

1.4
Diversions from the Initial Planning
Since this project is launched, some diversions from the initial planning were adopted
during the implementations. The reasons are some unpredicted problem appeared,
hardware limitation as well as new ideas that have not been considered initially.
1.4.1
Not Implemented
Although various metering stats is accessible in this project, such as memory, network
traffic, I/O operations and so on, a single objective optimization model, i.e. CPU
resources, is designed and implemented due to the time limitation and the complexity
of algorithm design of considering multiple factors. However, it can be argued that
the algorithm can be extended to multi-objectives model by simply going though
multiple resources inspection and employing policies.
1.4.2
Hardware Limitation
In the initial project planning, this project was expected to utilize the advanced
Intel Xeon CPU features such as CPU cache, TLB monitoring to achieve a highly
optimized resources management, however, it was found that those fancy features is
not available on current platform, therefore this idea is abandoned. Secondly, multiple
physical servers are supposed to be deployed to accommodate VMs. It turned out
that there are not enough compute resource available in DTU during the period of
this project. Instead, the workaround is to work on a single powerful platform with
32 CPU cores (hyperthreading) and 256G RAM, and nested virtualization technology
of KVM is used to create first-level virtual machines as the ”physical machines” and
nested virtual machine as the user VM instances as shown in figure 1.3.
1.4.3
New Considerations
During the implementation, it was realized that the CPU workload transition might
not be detected promptly and also there are some workload spikes because of inaccu-
racy that comes from sampling interval. To solve this, a state machine based physical
server status transition is implemented to accelerate transition recognition and also
avoid false positive and negative.

2.1
Virtualization
Traditionally, a single operating system controls all the underlying hardware resources,
which may lead to under-utilization of hardware resources. With Virtualization, mul-
tiple operating systems are able to share the same hardware platform with the help
of hypervisor, which arbitrates the accesses to the underlying hardware resources.
Figure 2.1 illustrates non-virtualization and virtualization. The advantages of vir-
tualization includes server consolidation, service isolation, faster server provisioning,
disaster recovery, dynamic load balancing, faster development and test environment,
improved system reliability and security, OS independence (reduced hardware vendor
lock-in) [21].
Figure 2.1:
Non-Virtualization vs Virtualization .

2 Virtualization and OpenStack
2.1.1
Virtualization Evolvement
Virtualization can be typically categorized in to three groups: full virtualization,
para-virtualization and hardware-assisted virtualization.
In computer science, the hierarchical security is implemented as protection ring
as shown in figure 2.2. In native environment, OS is running in the most sensitive
ring 0 to gain the privilege of accessing hardware directly.
Figure 2.2:
Protection Rings [source: wiki] .
In full virtualization, guest or VM OS is running in protection ring 1. All the
privileged instruction such as directly accessing hardware resources are trapped and
emulated by hypervisors, therefore there is no need to modify guest OS. However this
mechanism incurs significant overhead compared with OS in native environment.
To mitigate the overhead of full virtualization, para-virtualization is introduced.
Para-virtualized OS is actually modified to run in ring 0, but the critical instructions
are replaced with the so-called hyper-calls, whereby the guest OS is aware of that it is
running in a virtual environment. Para-virtualization provides greater performances.
While the hardware-assisted virtualization such as Intel-VT and AMD-V relies
on hardware support to achieve the optimal performance , it introduces two CPU
operation modes. The first one is called root mode which is basically similar to the
4 protection rings mode in native environment, while a second mode called non-root
mode is designed for VM, which has also ring 0 but with less privilege than ring 0
in root mode. Guest OS will be running in this special ring 0 with CPU support in
non-root mode. Hardware-assisted virtualization make it less complicated to design
a hypervisor and gives near native performance, therefore widely adopted in modern
virtualization techniques. Figure 2.3 illustrates the the differences between the three
virtualization technologies.

2.1.2
Hypervisors
Hypervisor is the layer between guest operating systems and hardware and its main
role is to present abstract hardware to VM and arbitrates the accesses to hardware.
There are numerous hypervisors available including Kernel-based Virtual Machine
(KVM), VirtualBox, Xen, VMWARE, Hyper-v. These hypervisors differs in imple-
mentation details, underlying platforms, performance, features and so on. Instead of
going through all these projects, only KVM will be touched here since it is adopted
in this thesis projects.
KVM is the latest virtualization framework of open-source virtualization. The
motivation behind is to build a powerful hypervisor by learning from the experience
of previous virtualization generations, making use of CPU hardware-assisted feature
and reusing the kernel code. Essentially, KVM turns Linux kernel into a hypervisor
and it has been made into Linux mainline, used for most cloud today, such as Google

12

cloud. Also, OpenStack, one of the most popular cloud deployment framework has
full support of KVM.
KVM consists of a loadable Linux kernel module, i.e. kvm.ko and a processor
specific module namely kvm-intel.ko or kvm-amd.ko.
KVM has to be used with
Quick Emulator (QEMU) to fulfill the goal of virtualization, where QEMU emulates
the peripheral devices and also plays the role of control for virtualization. From the
perspective of Linux host machine, each VM is actually an QEMU process when it is
launched and vCPUs are represented by threads, which are scheduled by Linux kernel
scheduler. Some tuning methods can be done to improve VM performance, such as
CPU pinning, Hugepages, NUMA awareness, etc. Apart from CPU pinning in the
subsection of ”Neseted Virtualization”, others are not covered in this thesis.
2.1.3
Libvirt
Although using only hypervisors are capable enough to start VMs, it is inconvenient
and inefficient to master different operations of various hypervisors and it also lacks
advanced features such as performance monitoring. Libvirt is such a tool to solve
this problem by creating a API layer above hypervisors as shown in figure 2.4. It
supports multiple underlying hypervisors such as KVM, Xen, VMWare ESX, etc.
and it presents APIs to the upper management tools such as virsh, virt-manager,
OpenStack and so on. Libvirt also enables VM related monitoring through ”/proc”
such as CPU, memory, disk I/O and network interfaces, all of which provides feasible

2.2
VM Live Migration
VM migration is the process of re-mapping VM to physical servers. Migration can be
beneficial in various scenarios, such as resource re-scheduling, maintenance, recovery
from host failure.
Typically, VM migration can be cold or live. A cold VM migration requires a VM
to be shutoff before migrated to another location, thus the application or service are
suspended, but on the other hand cold migration is good for VMs that requires a
huge disk to be migrated.
Compared with cold migration, live migration causes only a very short period of
downtime, sometimes even can not be sensed. A live migration is preferred when the
service can not afford a long downtime. A shared storage makes live migration more
==PAGE:26==
14
2 Virtualization and OpenStack
reliable and faster since the whole disk does not need to be transferred, which oth-
erwise consumes both significant bandwidth and compute resource. In this project,
shared-storage based VM migration specifically live migration is used to achieve effi-
cient resources management by migrating virtual machine when necessary.
2.2.1
Pre-copy and Post-Copy
Two live migration strategies are available including pre-copy migration and post-copy
migration.
In the pre-copy migration, VM is continuing running on source physical server
while the memory is transferred to destination physical server. Until the untransferred
memory is small enough and can be done in a pre-defined limited time, the VM is
frozen and start again in the destination machine after all the rest memory has be
successfully moved to destination. During the memory transferring, memory pages
might be re-transmitted if the transmitted pages are dirtied at the source side, which
means memory intensive VM, such as video streaming, might be in trouble and takes
extremely long time or even never to finish migration. However, pre-copy can ensure
VM performance during the migration and pre-copy is recoverable if migration fails,
then VM continues running in source physical server.
In the post-copy mode, the VM will start to run in the destination machine as
soon as possible. In the beginning of post-copy migration, the VM will be suspended
for a very short time to transfer a minimum subset of state of target VM and then
VM will be running in the destination machine, meanwhile the memory is transferred
to destination.
If the running VM access the memory pages that have not been
ready yet at destination, a network page-fault is generated to invoke a transfer from
source machine. Obviously, pages are transferred only once in post-copy migration,
which saves resources. The disadvantages are VM performance might be hurt due
to network page-fault and VM is not recoverable in case migration failure since the
state of VM is distributed at both source and destination side.
It can be concluded that pre-copy is the best choice for VMs that are not memory
intensive and otherwise post-copy is preferred. In the implementation of this project,
pre-copy is used to live-migrate VM from one server to another because only one
objective, i.e. CPU load, is taken into consideration and memory utilization is light.

5.2
Expected Results
The expected results can be categorized into two types.
Firstly, if a host is underutilized, this resource manager should try to move the
inactive VMs on the host to other hosts in order to switch off this underused host for
energy saving. When necessary, a switched-off host might be re-activated again later.
Secondly, low-priority VMs can be migrated to the reserved cluster if the high-
priority VMs are not utilizing the full resource in normal cluster. In contrast, if a
high-priority can not get desired resource because low-priority is eating the resource
in reserved cluster, the low-priority VMs should be preempted and migrated back to
normal cluster even if the resource there is in shortage.

In this chapter, the resource manager is tested against different scenarios with dif-

5.4.2
Future Work
This project demonstrates that a centralized resource manager makes the a cloud
more efficiently utilized and environment-friendly. Based on the result of the tests,
there are several directions to further extended this project.
==PAGE:61==
5.4 Chapter Conclusion and Future Work
49
Firstly, more factors should be taken into considerations. This project is limited
to only one objective, i.e. CPU. However, considering other aspects such as RAM,
disk I/O, network performance and even cache can be incorporated and beneficial.
With multiple factors are considered, services with different feature can be scheduled
more accurately, for example, MapReduce task is both CPU and Disk Intensive, thus
both CPU and disk should be taken into account. The implementation of adding
these additional factors is similar to the currently implementation for CPU.
Secondly, finer optimization at each status. In the current implementation, seven
statuses are defined according to the workload, but only three extreme statuses (two
high load statuses and one light load status) out of them have corresponding actions to
adjust the cloud environment. The algorithm can be further developed to have actions
on all seven statuses to achieve a more accurate and precise management and control.
For example, low_efficiency state should also be considered to be consolidated but
less proactive than underused status.
Thirdly, a global overall resource utilization is recommended to get the current
resource usage in the cloud. With the global resource usage, aggressive consolidation
can be avoided in the situation where some hosts are underused but should not be
switched off since the overall usage is high and the cloud is in transition.
Moreover, the limitation described in 4.4 needs to be tackled. The issues is that
the meter obtained from monitor might not reflect the true value that is caused by
the resource in badly shortage and the Completely Fair Scheduler (CFS) in Linux.
To achieve this without invading VMs because of security, machine learning can play
a role to accurately estimate how bad the resource shortage is. If this issue is solved,
the recover time of hosts will be shortened.

CHAPTER 6
Conclusion
In conclusion, this thesis focuses on the resource management in cloud environment
using virtualization technologies. The challenges and use cases are identified, and
different technologies are compared.
To achieve the goal, an experimental cloud
environment was deployed using OpenStack and KVM hypervisor. The computing,
networking, storage and metering services in the OpenStack environment are covered.
Time-series database is adopted to persist the meters for further analysis. Pre-copy
VM live migration is used to dynamically optimizing the cloud resource. An preemp-
tive and multi-statuses algorithm is devised to scheduling the VMs in the cloud to
improve the utilization, guarantee performance and save energy at the same time.
An resource manager application was programmed to implement the algorithm.
The performance of the resource manager was evaluated under different scenarios
from light workload to heavy workload. The results shows that the resource man-
ager successfully consolidated VMs and combined the fragmented time into a single
longer period, which created the opportunities to switch off host servers and further
save energy. In the test scenario of heavy workload, resource manager significantly
improved the resource utilization, reduced the number of times that a host suffers
resource shortage and shortened the hosts’ recovery time spent transiting from an
unhealthy status to a healthy status. Shortly, the algorithm was verified working
as expected in an cloud environment. Although the environment deployed is rela-
tively small, the preemptive and multi-statuses idea behind the algorithm and the
architecture of resource manager theoretically can be scaled to a large and complex
environment.
Moreover, there is chance to further improve the algorithm and perfect resource
manager. The potential directions of extending this project are spotted and sugges-
tion on future work is presented.
