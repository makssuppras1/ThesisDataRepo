Introduction
The history of Mobile Networks goes back further than what is possible for any engi-
neer to remember. Mobile Networks have changed tremendously through time, but
even with a long history that includes signiﬁcant changes, recapping the history of
Mobile Networks is not one of the major concerns in today’s digital world. There-
fore, this is a task left for the tech-savvy storyteller to solve, considering that the
development of mobile networks is an integral part of a Telecommunication Engineers
job description. Not only is it vital for engineers to constantly improve the existing
deployed systems, but even more so to ﬁnd new and sustainable solutions for the
world of tomorrow – or even better; ten years ahead. To some, this might sound
obvious, but it has been proven to be rather diﬃcult. Constantly seeking new and
improved solutions to existing telecommunication systems would not be so challeng-
ing if the obstacles and requirements of the future were predetermined. Due to a
rapidly changing nature of technology, even more abstract questions are yet to be
answered; what will future mobile networks look like, and what are the future trends.

2. Network Elements
Alexander Gregaard Brandi (s103208)
2
Network Elements
The theory of mobile networks is complex and contain various elements. The up-
coming sections will attempt to analyze and discuss the most essential and relevant
parts of the theory. The main focus of the project is to investigate the use of mea-
surements from the physical layer only to describe the performance in another layer
of the stack. Hence, information from communication taking place in the physical
layer will be exchanged using standards from 3GPP. As the sections follows, diﬀerent
components will be explained using ﬁgures to enable a simpliﬁed, fast interpretation
and understanding of more complex elements.
The project takes its starting point in Long Term Evolution (LTE), moreover
from 3GPP Release 9 and onwards.
This network was selected since it is one of
the newer standards, available for the genuine consumer. More importantly, it is
supported by the devices selected for testing and subsequently collecting data for
the project. Release 9 is also interesting as it was standardized using a new access
solution based on OFDM. By combining higher order modulation (up to 64QAM),
larger bandwidth (up to 20MHz), and spatial multiplexing (4 × 4 MIMO) in the
downlink direction, much higher data rates than previously seen can be achieved.
The standard introduces OFDMA, which makes it possible to achieve high radio
spectral eﬃciency and better scheduling in both frequency and time domain.
A
later improved version, combines results in theoretical peak datarate of 300Mbps in
downlink (using 4×4 MIMO) and 75Mbps in the uplink direction. Another important
adjustment that was made to make this possible, was decreasing the Transmission
Time Interval (TTI) to only 1ms. During this short time period three things happen:
• Radio Quality Reporting
• Prioritize QoS
• Allocate Resources
To achieve this, special signalling mechanisms were implemented into the trans-
mission located at the physical layer. Measurements performed and taking place in
the physical layer are also known as Reference Signal (RS). The reference signals are
an essential part of this project as they represent the physical parameters used for
prediction of the performance of a layer located higher in the communication stack.[3]
Two methods are available for data collection. As mentioned, the standard is
deployed by ISPs and is available to the public. This allows for data collection by
using diﬀerent artifacts. The other possibility is the creation our own network in a
laboratory. In this way, initial testing can be performed prior to data collection in
the real world. The following section takes a closer look at the reference signals, their
origins and what they represent.

Network Architecture
The theoretical starting point of this project takes toll in the general architecture
of the Long Term Evolution (LTE) network.
Figure 2 depicts a simpliﬁed illus-
tration of the LTE network architecture. If the user wishes to have his/hers daily
Netﬂix-and-chill-ﬁx or Facebook-cravings fulﬁlled, they use a UE to access the land
of opportunities, also known as the Internet. Meanwhile, the user is mostly unknown
to the fact that an extensive amount of components are inter-connecting to perform
the requested operation. In the access network, or Evolved UMTS Terrestrial Radio
Access (E-UTRA), a UE is sending and receiving wireless data to an eNB. The eNB
then communicates with the Evolved Packet Core (EPC), which empowers a commu-
nication link between the access network and Internet. Whenever the user is moving
closer to another eNB the Mobility Management Entity (MME) is used to determine
if there is a need for a handover to another cell.

carrier technology that enables the bandwidth to be divided into a multitude of
mutual, orthogonal, narrowband subcarriers. The individual subcarrier is deﬁned by

Figure 4 depicts a spectrum of multiple OFDM subcarriers of constant amplitude
without any interference. The subcarriers are spread across the entire bandwidth
with ∆t interval. LTE has a low symbol rate of only 15ksps. In comparison, other
technologies such as Universal Mobile Telecommunications System (UMTS), which
has a much higher symbol rate at 3.84Msps. Nevertheless, in the same 5MHz channel
bandwidth, LTE is capable of transmitting 300 subcarriers simultaneously providing
an aggregated symbol rate of 4.5Msps.
The hawk-eyed reader might already have notice that 5MHz divided by 1.5kHz

2.4.1
Reference Signal Received Power (RSRP)
In LTE, the network requires the UE to detect and monitor the presence of multiple
cells. This ensures that it is idling or connected to the most suitable cell. Reference
Signal Received Power (RSRP) is one of the reference signals acquired by the UE to
determine if it needs to perform cell reselection. As implied by the name, RSRP is
a measure for indication of the quality of the power, not to be confused with signal
quality. RSRP is deﬁned as the linear average over the power contributions of re-
source elements within the considered measurement frequency bandwidth, measured
in Watt (commonly converted to dBm). However, RSRP is only measuring power
contributions of the resource elements that carry CRS. Therefore, it was decided to
direct attention to the CRS as described in Section 2.4.
To determine the value of RSRP, the UE uses the reference signals in R0 that
are carried in the CRS. This means that it will use the reference signals that are
transmitted from antenna port 0. In case the UE is capable of reliably detecting that
R1 is available, it may use this in addition to determine R1. The mapping of reference

3.1.3
Crowdsourcing
Crowdsourcing is a sourcing model in which the process of getting work, funding
or data is sourced from a crowd of multiple sources. The deﬁnition Crowdsourcing
derives from a combination of two words; crowd and outsourcing. The idea behind
crowdsourcing is to decentralize hard labour and get the tedious tasks done faster
so that we are able to focus on the more important things in life. A methodology
that brings nothing but tranquility to the process of collecting vast amount of data
ready to be processed. Accordingly, as an experimental way of collecting data from
multiple sources, we developed an Android application for crowdsourcing. Like the
SwissQuall QualiPoc, described in Section 3.1.2, this to collect information about
network conditions and geographical coordinates etc., but instead of saving it locally
and export it manually, the data is sent directly to a centralized server, ready to be
used by others. In this way we are able to collect a massive amount of data, from
diﬀerent places at once. By installing our Android application into several devices we
are able, not only to collect data a whole lot faster, but getting information from real-
life scenarios. By means of patterns that enable us to understand network conditions
on a human-level, and maybe getting closer to a deﬁnition of QoE.

In Figure 21 we try to use Principal Component Analysis (PCA) analysis to
project our data from high dimensional data onto a lower dimension. The purpose
is to perform a orthogonal transformation to convert a set of observations of pos-
sible correlated variables into a set of values of linearly uncorrelated variables, also
known as PCA. Using PCA naturally includes Single Value Decomposition (SVD)
which provides an easy way to compute n eigenvectors corresponding to the n largest
eigenvalues. Essentially, SVD tries to reduce a rank R matrix to a rank K matrix. In
other words, it takes a list of R unique vectors and use them to approximate a linear
combination of K unique vectors. Even simpler said, SVD may result in suitable
basis for a given space, where the matrix can be expressed in a simple diagonal form.
For any N × M matrix X, from which entries derives from the ﬁeld K with either a
ﬁeld of real or complex numbers. Then there exists a factorization described as the
Single Value Decomposition of X:
X = UΣVT

Introduction
The aim of this course is to gain better understanding of where to use Machine Learning
within Mobile Networks. There are various ways of looking at Machine Learning, and
even tedious discussion on what we deﬁne as Machine Learning or Artiﬁcial Intelligence.
The amount of techniques that can be applied to this ﬁeld of study are beyond what
one can comprehend. [1] Due to the massive amount of techniques that can be applied
to this ﬁeld of study, this paper will dive deeper into the investigation on how Machine
Learning can be used in Mobile Networks. [2]
The purpose of this document is to outline the overall process of the master thesis
to limit the scope with focus on to reach the end-result.
This document includes a
timeplan, milestones and main goal of the thesis.
In short, research shows Machine
Learning withholds great potential to be used in Mobile Networks, but researchers are
still unsure how to implement these complex methods. Instead of creating a complex
Turing machine, this project will limit the scope to collect, test and explicate the use
of machine learning in LTE networks by trying to ﬁnd correlations in between Quality
Identiﬁers and HTTP response time.

Project description
The main goal of this project is to ﬁnd use for Machine Learning algorithms in mobile
networking. Furthermore, collect, map and analyze data to test and verify the practical
use of machine learning algorithms. Machine learning itself is rather comprehensive and
in which the project is limited to the scope of status reports and quality indicators
in LTE networks.
At the moment, researchers are trying to solve complex problems
and implement machine learning without knowing which parameters to use. Therefore,
this project will try make a simple proof-of-concept to investigate if RSRQ, RSRP and
other quality indicators are correlating or can be used as features to train a model using
machine learning in mobile networks to improve the QoE. To clarify, the project can be
described as:
Collect network status reports and analyze the data for improving QoE by using
machine learning.
Figure 1: Overview of the LTE network illustrating the general architecture and components.
Figure 1 illustrates the general architecture of a LTE network which includes several
components that could be optimized, but this project will look closer on the control
sequences in between the UE and eNB, i.e. RSPQ and RSRP. The project is then to
compare the status reports with the HTTP response time. Figure 2 illustrates that the
status reports are transmitted in between the UE and eNB, where the HTTP request are
sent through the network.

Achievements
The project can be divided into sets of achievements that needs to be accomplished to be
realized. Fetching data is the ﬁrst step and will be done through SwissQual’s Qualipoc
that is a tablet with a custom app.

• Prediction Model used as learning model using machine learning

aforementioned achievements will be developed and tested during the project period in
which the goal is to verify whether or not quality indicators can be used to optimize QoE
