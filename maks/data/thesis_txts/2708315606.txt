Abstract
As the infrastructure of a company grows larger, so does the complexity, main-
tainability and transparency of ﬁrewall policies. Maintaining a transparent and
manageable ﬁrewall policy throughout rapid scaling of a company proves almost
impossible. This is complicated further by having multiple people or teams de-
ploy and maintain these ﬁrewall policies. Diﬀerent people might tackle a prob-
lem diﬀerently and come to diﬀerent solutions which therefore in turn leads to a
diﬀerent ﬁrewall policies. Adding tasks such as updating, patching and testing
these policies on top of that, complicates things further.
The solution involves designing and building a web application that tackles the
problem of detecting anomalies in ﬁrewall policies using machine learning and
then explaining it to the end-users in human-friendly format.
The machine
learning models will use ﬁrewall logs from internal ﬁrewalls and therefore learn
the intricate structure of the network traﬃc. The machine learning models will
then learn to predict the expected output from the logs and if this expected
value diﬀers from the predicted value an anomaly has been found. This web
application most be built using the newest best practices, technologies and follow
the recommended design principles.

Chapter 1
Introduction
Computer security is becoming more and more important as time progresses.
In this modern age of digitalization it is almost unavoidable to include software
in every venture and solution. But as the computational power increases and
our software knowledge expands, so does the adversary. Bad actors constantly
ﬁnd new ways to exploit systems and the methods of which they do so become
more eﬃcient and automated, which makes more system vulnerable. One of the
most vital component in keeping a network secure is Firewalls. Firewalls either
hardware- or software-based, are vital in the protection of both internal and
external attacks on a network. A ﬁrewall typically acts as a barrier between
two network zones, from here it can monitor and control incoming and outgoing
network traﬃc.
Essentially this type of ﬁrewall can either allow or deny a
network packet to pass through the barrier. To control this functionality ﬁrewall
policies are created to determine how these network packets should be handled.
As the cooperations of today expands their infrastructure rapidly in order to
keep up with the increasing demand for products and services the likelihood
of making just one mistake in the ﬁrewall conﬁguration drastically grow. The
same story is told again and again by the security responsible in these large
cooperations, the developers genuinely just wants their service to work and is
often uninvolved in ensuring the bare minimum of security gabs in the network.
The so called "any" policy is often seen as the easy solution developers take to
ensure all their services function correctly, meaning a complete opening of either

Furthermore as the infrastructure of a company grows larger, so does the com-
plexity, maintainability and transparency of these ﬁrewall policies. Maintaining
a transparent and manageable ﬁrewall policy throughout this rapid scaling of
a company proves almost impossible. This is complicated further by having
multiple people or teams deploy and maintain these ﬁrewall policies. Diﬀerent

Such a mistake can be extremely costly. A simple typo could cause the entire

4.1
General Design Principles
While developing any software it is important to have some core design principles
in the back of your head. And even through the solution is only developed by
one person, the maintenance and updating will be passed on to other colleagues
within the company in the future. Therefore it is essential to have clean, eﬃcient,
cohesive and transparent code.
Another developer must be able to quickly
understand the overall structure and how the application works.
4.1.1
Cohesive Design
Cohesion within software projects, especially of this size is one of the core con-
cepts that must be used. Cohesion can be viewed of the process of grouping
==PAGE:18==
10
Design Principles and Technologies
components together in sections and keeping them relatively isolated from each
other. This enables maintainability and further development to be much more
painless, since each component and its’ responsibility will be clear.
Leading into this principles is the principle of reducing coupling.
Coupling
essentially refers to the dependency of other classes within classes and functions.
Having a high coupling in a project makes it diﬃcult to trace where any task is
executed, since it may happen in multiple diﬀerent sections of the project. It is
a ﬁne line however, since some coupling is beneﬁcial to keep the code organized
and of course having global helper functions, extensions etc is a must.
4.1.2
Flexible Design
Keeping a ﬂexible design for a software solution is required to make the codebase
able to withstand the test of time. Keeping the solution ﬂexible is anticipating
changes and complexity in the future. Patching to new frameworks, porting old
code bases to new etc. is much more straightforward with this approach.
4.1.3
Abstracted Design
In developing terms the level of abstraction refers to how simpliﬁed the solution
is. How general and reusable the code is. One of the core beneﬁts to keep-
ing a high abstraction level is that major changes within the project is much
easier since the code is more general.
The use of abstract C# base classes
with inheritance and polymorphism is used throughout the project to apply to
this principle. Furthermore C# interfaces is heavenly used, this ensures that
when creating new speciﬁc classes a certain structure must be upheld and some
methods must exist within the class.
4.2
User Interface
Since the ﬁnished web application will be used as an internal portal that is used
for production environment related task, it is essential to keep the user interface
optimal and responsive. Although it is important for applications to have a GUI
with high usability the time restraint has limited the GUI design to be simple
yet eﬀective and not include any User Centered Design principles.

Chapter 7
Dataset
This chapter is all about the data collection, data processing and data storage.
The overall goal for the chosen architecture is to separate the logic for data-
collection, processing and ML training. Each of these steps should be able to
be computed asynchronous and each step should be independent of the others.
This allows for a more scalable and less error-prone design, since each tasks
could be computed on separate hardware and at diﬀerent time intervals.
7.1
Data collection
The data collection will be split into two separate sections due to the diﬀerent
use-cases they each address.
Netcompany processes around 300 GB of ﬁrewall log ﬁles on a daily basis. These
log ﬁles are distributed using a load balancer and is then stored on several
diﬀerent log servers. These log servers will then publish the data to a central
log system called Splunk.
Splunk then enables the end-users to create searches, monitoring and analysis
of this data.
==PAGE:30==
22

7.1.1
Stored Logs
The application uses some of these stored logs directly for the initial training of
the ML Models. These logs has been specially pre-selected to ensure consistency.
Using this approach is mainly to ensure that the test data for the ML Models is
consistent and is correlating with the real environments. Imagine a temporary
network issue caused multiple network log entries to give a diﬀerent result than
than usual. Ensuring the initial training session for the ML models is safe and
controlled enables the model to perform better in diﬀerent environments and
conditions.
7.1.2
Live Data
Once the initial training has been done with the pre-selected logs, the dynamic
and versatile training can commence.
Using the before-mentioned software
called Splunk, a saved search can be created that will automatically create

location based on the saved search in a JSON format. This data can then be
consumed in the application at runtime and used to train the ML models.

when having multiple models for multiple diﬀerent tasks.

order to avoid having to waste space on empty database columns.
