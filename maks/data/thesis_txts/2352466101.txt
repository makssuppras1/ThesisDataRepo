Abstract
This thesis deals with the Belarusian cipher family called Bel-T from 2011. It
consists of a block cipher, a MAC function, a hash function and an Authen-
ticated Encryption with Associated Data function.
All 4 algorithms will be
presented, followed by an analysis focussed on the hash function. The presenta-
tion is crucial since there, to my knowledge, is not any English source explaining
the 4 algorithms. The hash function is picked because the MAC function and
the AEAD function, share a lot of properties with existing designs that have
been studied extensively in the past. The block cipher was researched last year
by a fellow DTU master student. The cryptographic results in this thesis are, to
my knowledge, the ﬁrst of its kind, and will violate 3 diﬀerent security properties
• A collision is found for the two inner states of an accumulated bit size of
384 in 2134 calls to the compression function for a message of at least 26
blocks.
• Two second preimage attacks is found on the hash function in 2n−log2(t)
calls to the hash function for a message of at least t blocks where t can be
28 . . . 2110.
• The Chosen Target Forced Preﬁx preimage resistance is violated in 2172.94
calls to the hash function for a message of at least 28 blocks.
All the attacks are below the generic attacks, meaning that the given property
for the hash function is considered broken. But all three complexities are still
so high that they cannot be done in practice meaning that the security of Bel-T
is not at risk.

Chapter 1
Introduction
When an algorithm is created for security purposes, it has to meet some condi-
tions in order to be considered "safe". It does not mean that the algorithm is
useless if one of them is compromised, but it is good way of comparing functions,
whether or not the fulﬁl the properties in the way they are supposed to.
In 2011 the article Data encryption and integrity algorithms was published as
a "Preliminary State Standard of Belarus" [Bel11]. It included four algorithms
a block cipher, a MAC function, an AEAD function and a hash function. In
this thesis, the biggest focus will be on the hash function and whether or not it
fulﬁls some of the most important properties.
1.1
Motivation
Looking at the four algorithms it is clear that the selected designs used in the
MAC function and the AEAD function, is very closely related to algorithms
that has been researched for years. This leaves the block cipher and the hash
function. The block cipher was researched by a fellow DTU student last year so
the obvious choice for this thesis, is to have a look at the hash function which
posses quite a unique design.

1.2
Hash functions
A hash function is a function that takes an input of arbitrary size and maps an
output of ﬁxed size. Hash functions come in diﬀerent versions. The most simple
ones are used to accelerate indexing of data, while more sophisticated ones are
used for cryptography purposes. Common for the two is that they should map
their outputs as equally distributed as possible. It does not help the indexing
much if half its input values are sent to the same container. And it would be a
big blow for the security of a cryptographic hash function if it only used half its
sample space.
1.3
Cryptographic hash function
A hash function is a function that maps an input of arbitrary size to an output
of ﬁxed size. The cryptographic hash function is a mapping that, as opposed
to an ordinary hash function, also fulﬁls some critical conditions. First of all
it should be diﬃcult to reverse, this property will be mentioned later on as the
preimage resistance.
This means that if an attacker obtains a hash value it
should be very diﬃcult for her to ﬁgure out which messages that hashes to that
hash value.
Cryptographic hash functions are used in several diﬀerent applications where
hashing of passwords, documents and even programs are the most common
examples. Passwords are hashed to serve the security of the user, so even if the
server gets compromised and the attacker obtains a hash value, she still has a

long way to ﬁnd the users password. Documents and programs are hashed by
the publisher which then publishes the hash values along with their products.
When the user then downloads the item, he has the chance of checking it himself.
If the downloaded item hashes to the same as published by the publisher it is

1.3.1
Attacks
There are three diﬀerent properties that always get the most attention with a
hash function. Finding a preimage, a second preimage and a collision. Besides
this there are some additional properties that do not gain as much attention,
like the CTFP preimage resistance. For a property to be fulﬁlled it should be
harder than the generic attack to violate them.
• Preimage: The hash function should be diﬃcult to reverse. This means
that if an attacker was supplied with a hash value, y, it should be dif-
ﬁcult to ﬁnd a message, x, such that Hash(x) = y. The generic attack
complexity for a preimage attack is 2n.
• 2nd Preimage: If an attacker receives a speciﬁc x it should be diﬃcult
to ﬁnd an x′ ̸= x such that Hash(x′) = Hash(x). The generic attack
complexity for a 2nd preimage attack is 2n.
• Collision: It should be diﬃcult for an attacker to ﬁnd x and x′ such
that x′ ̸= x and Hash(x) = Hash(x′). The generic attack complexity for
collision attacks is 2
n
2 .
• CTFP preimage resistance: If the function has a good CTFP preimage
resistance, it is diﬃcult for the attacker to ﬁnd an arbitrary amount of
blocks, denoted suﬃx, which when appended to the message, makes the
overall hash end on a predeﬁned value. The generic attack complexity for
violating the CTFP preimage resistance is 2n.

2.1
The block cipher
As mentioned earlier, the block cipher is the most crucial part of this family. It
is used in all the other three, and can also be used alone as a standard block
cipher encryption/decryption scheme. The cipher takes a plaintext, X, of 128
bits and produces a 128 bit ciphertext. It can take keys, θ, of size 128, 196 and
256 bits. The key expansion scheme is extremely simple, and can be found in
appendix.
The 128 bit input is divided into four parts of 32 bits called a, b, c and d.
These parts are sent through the round function, Figure 2.1, and returned,
concatenated, in the order b, d, a, c.
The round function consists of simple operations and some combined calls in-
cluding a rotation and and sbox call. An example is the ﬁrst line in the round
function where the G call is: G5(a ⊞K7i−6). This means that the 32 bit a is
added with the K7i−6 key, and then divided into four 8 bit parts. The four
parts are sent through the sbox, then concatenated back together and cyclically
shifted left 5 positions. In every iteration seven keys are used. The use of the
keys follows the scheme provided in appendix, but can also be calculated from
the the loop iterator, i.

2.2
The MAC
The MAC cipher in the Bel-T family looks very familiar. It is very close to the
CBC-MAC algorithms explained in [Knu13]. The cipher takes the text, X, of
128 bits and combines it with a key, θ, and creates the tag,T with a size of 64
bits.
The algorithm is build from the commonly known CBC design [Knu13], where
the current state, Si, is found by xoring Si−1 with Xi and sending it through
some kind of function, in this case the Bel-T block cipher with θ as key input. So
the basic construction of the Bel-T MAC is identical to the known CBC-MAC
algorithm. The signiﬁcant diﬀerences can be found in the ﬁnalisation part.
The ﬁnalisation depends solely on the size of the last block of the message. If
the block is 128 bits long it is treated as usual and xored with ρ1(Fθ(0)), if not
it is padded to ﬁt a size of 128 bits before it is xored with ρ2(Fθ(0)). At the
end, the function returns the ﬁrst 64 bits of the block cipher call of the current
S as the Tag T.

The AEAD
The AEAD is the Bel-T version of an "Authenticated Enryption Associated
Data"-algorithm. The AEAD works in very much the same way as the block

key, θ, using the block cipher. Next up is generating the polynomial hash. This
is done trough the associated data and the r, which is the double encryption of S.
Lastly the tag is updated to also depend on the ciphertext. All multiplications

3.1
Building a hash function from a Block Cipher
Many hash functions are build from block ciphers.
Since block ciphers are
designed to be unpredictable they serve as great randomisers inside a hash
function. Block ciphers are reversible, which is necessary for encrypting and
decrypting but a disaster for a hash function. Therefore a principle like the
Davies-Meyer construction [Sch96] is often applied. The Davies-Meyer construc-
tion uses the message block as key input and the recent hash value as plaintext

The second of the three properties is the ability to resist a second preimage
attack. It should be diﬃcult (more expensive than 2n calls to the hash function)
to ﬁnd x′ ̸= x such that H(x′) = H(x), for a given x.
This chapter presents two second preimage attacks on the Bel-T hash function.
Both the attacks will have a complexity around 2n−log2(t) for diﬀerent intervals
of t, where t denotes the amount of blocks in the message. The ﬁrst attack reuses
parts of the an attack on Streebog-512 presented in [GJL+14]. The similarities
between the two attacks will be discussed. Secondly an Expandable Message
attack on Bel-T will be presented, where the connections to the Streebog-512
attack will be discussed as well.
Streebog is picked for the same reason as MDC-2, because the architecture
has some signiﬁcant similarities with Bel-T. For MDC-2 the most important
architecture is the two parallel states that both went through some kind of
function, in this case for Streebog the most important thing is how the Σ works
as a checksum in much the same way as the S in Bel-T. All this will be further
explained in the following chapter.
The whole Streebog cipher consists of two functions, Streebog-256 and Streebog-
512. But just as in [GJL+14], this thesis only focuses on Streebog-512 which for
simplicity will be denoted just as Streebog moving forward.

5.1.1
Introduction to the cryptanalysis of Streebog-512
Looking at Streebog-512 seems to be a good place to start, in order to gain some
knowledge regarding the possibilities for a second preimage on the Bel-T. The
Streebog-512 is a good choice because the checksum, Σ, behaves a lot like the
way that σ1 is stored in Bel-T. Besides that, a lot of the function is very similar
to the Bel-T architecture.
There are three parameters in Streebog that need to be matched: Σ, N and
h. In Bel-T there are the two internal states: S and h that need to match,
and then the amount of blocks used should also be the same as the attacked
message. Looking at the 3 variables from Streebog compared to the Bel-T
• h and h:
h is the output of the compression function in both Streebog and Bel-T.
The h state is inﬂuenced by almost everything else in every iteration.
• Σ and S:
In Streebog Σ is the sum of all the message blocks that have been sent
through the algorithm. This is very close to the role that S has in Bel-T,
being the XOR sum of all the keys calculated by the σ1 function.
• N and |X|:
N is a counter in Streebog, at every iteration the counter is incremented.
Besides this, Streebog also uses the amount of blocks as input to the last
call. In Bel-T there is not a counter in the same way, but the amount of
blocks, |X|, is used as input at the end.
The h’s go very well for a direct application since they play the exact same role
in both ciphers. The same goes for the Σ and S variables. Leaving the N and
|X| left. Those ones should not be a problem either, because the way that the
attack deals with the length should work for the Bel-T function as well.

6.1
The Herding attack
Kelsey and Tadayoshi present the Herding Attack in [KK06]. It is an attack
on the CTFP preimage resistance of hash functions that are build around the
Merkle-Damgård construction, Section 3.2.
The Bel-T is not quite Merkle-
Damgård, but it is close. The most important thing is that it is iterative, which
is the property that Kelsey and Kohno takes advantage of in [KK06]. So it
makes perfect sense to have a look at their Herding attack.
P denotes the prior mentioned challenge.
This is what the attacker claims
to know before it has happened, which for obvious reasons cannot be true.
The attacker then receives the challenge at the time that it becomes public to
everyone, and then has to ﬁnd an S matching the challenge, P, that leads to
Hash(P||S) = H as it can be seen in Figure 6.1. P has to be rather complex,
this means that it should be able to be a signiﬁcant amount of diﬀerent values.
6.2
Finding a suﬃx by the herding attack
The attack proposed by Kelsey and Kohno builds on the creation of a diamond
structure, Section 3.6, that consists of intermediate hash state collisions. The
diamond structure in Figure 6.2 consists of nodes that represent the intermediate
hash state of the message block that is denoted by the edge leading to the node.
The k factor determines the size of the diamond. 2k is the amount of nodes at
the widest leaf level, and there will be 2k+1 −1 intermediate hash states in total.
The clever thing is now, that when the tree has been build, the attacker will
have 2k diﬀerent possibilities of entry at the leftmost leaf level of the diamond.
The suﬃx, S, is now divided into two parts: S = S′||Q. Where S′ is all that
has to be appended to the challenge P in order to hit one of the intermediate
hash values inside the diamond. All the blocks, edges, that are needed from the
diamond in order to get from hash(P||S′) to H = hash(P||S) are then denoted
as Q.
So the attacker ﬁrst of all builds the diamond as described in Section 3.6, and
then publishes H from the diamond.
When the attacker then gets to know
P, she hashes it until all blocks are used and then tries to ﬁnd a message
block that connects hash(Pn−1) with one of the leaf nodes in the diamond.

Chapter 7
Discussion
Finding collisions
Through this thesis it has been shown that there can be found collisions for
the compression function in 2134 calls to the compression function. This can be
achieved for messages of at least 26 blocks. The principals behind the successful
attack is mainly dividing the function into 2 parts, attacking the bigger one with
a Joux Multicollision and afterwards ﬁnding a path trough the multicollision
that would lead to a match for two of the message combinations.
This attack does not extend to the whole function due to the obvious reason
that the complexity budget on the inner state is much bigger than the one on
the whole thing because of the diﬀerence in n. This can also be seen through
the fact that ﬁnding a generic attack for the big state would cost the same as
the budget of 2128 for the hash function.
In order to achieve the collision result, diﬀerent approaches was taken. The
biggest one beside the successful one, was an angle used on the MDC-2 cipher.
The main reason that the MDC-2 attack could not be directly applied is that
the inﬂuence the two states have on each other is so much smaller than the
inﬂuence the two states have on each other in Bel-T.
==PAGE:70==
62
Discussion
Finding second preimages
It has also been shown that it is possible to ﬁnd a second preimage for the hash
function. This can be done in 2 diﬀerent ways, where the best complexities for
ﬁnding a second preimage is around 2n−log2(t) calls to the hash function for a
message of at least t blocks for t = 28 . . . 2110. The ﬁrst approach presented uses
a diamond to commit to an intermediate hash value that should be linked to the
original message. The second one make use of an Expandable Message, again
to commit to an intermediate hash value, that should be linked to the original
message.
The inspiration to achieve the second preimage attack mainly comes from an
attack on Streebog-512. The attack referenced in this thesis is created to exploit
a repeating pattern of the counter used in Streebog-512. As mentioned in the
analysis, this does not aﬀect an attack on the Bel-T. So a lot of variables are
replaced and some of the steps are replaced with similar techniques that ﬁt an
attack on the Bel-T better than the Streebog-512 case.
To defend yourself against this kind of attack it would make sense to have Si
inﬂuence the value of hi. Instead of S just being the xored sum of keys, it would
make sense to use it much more. This would make it very diﬃcult to operate
with the checksum issue, since picking a new path in the multicollision in Figure
5.3 would lead to a new intermediate hash value in the next state. This would
apply for the diamond as well, making it very diﬃcult to create.
Violating the CTFP preimage resistance
At last it was also showed that the CTFP preimage resistance for the hash func-
tion can be violated. This is done with a complexity of 2172.94 for messages of
at least 28 blocks. It is done with an invented combined Herding-multicollision
attack, where the Herding attack take care of the lower state, while a multicol-
lision make sure that here will be at least one usable output matching the top
state
The original Herding attack was designed to work on all hash functions designed
using the Merkle-Damgård construction. As mentioned in the presentation of
the hash function, Bel-T has a lot of the properties that are a part of a Merkle-
Damgård construction but is not entirely Merkle-Damgård.
In order to avoid attacks like this it would make sense to have some kind of
extra input to the compression function. This would work like a salt, making
the whole part about creating the diamond ﬁrst and then publish the random
H from that much more diﬃcult, because the diamond now cannot be created
before hand.

Chapter 8
Conclusion
All four algorithms from the Bel-T publication have been presented and ex-
plained brieﬂy and the focus on the hash function has resulted in a violation of
three diﬀerent properties, all 3 the ﬁrst of their kind for the Bel-T. First of all
a collision is found in 2134 calls to the compression function for a message of at
least 26 blocks. Which is below the generic birthday attack complexity of 2192.
Two diﬀerent second preimage attack has been presented. They both have a
complexity of 2n−log2(t) where t denotes the amount of blocks. The interval for
t is diﬀerent for the two attacks but together they cover a range of t = 28 . . . 2110.
The generic attack for a second preimage attack on Bel-T is 2256 which makes
both attacks below this as well.
Lastly it has been shown that the CTFP preimage resistance can be violated in
2172.94 calls to the hash function for a message of at least 28 blocks. Also this
result is below the generic attack of 2256
Since the results exclusively are obtained trough a black box approach, they
would work in the exact same way if one would instantiate the same architecture
but with any secure block cipher as the randomiser. All three complexities are
to big to be performed practically and therefore do not jeopardise the practical
security of the Bel-T hash function.
