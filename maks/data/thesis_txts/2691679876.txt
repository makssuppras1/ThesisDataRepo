Abstract
As database sizes increase, the importance of fast and eﬃcient compression and search algorithms
is more signiﬁcant than ever. In this thesis, we present solutions to a challenge presented by FOSS,
a Danish company providing high-tech analytical solutions for food and agriculture. The challenge
consists of eﬃciently compressing and running nearest neighbour searches in a database of electro-
magnetic spectra.
We present several compression solutions using prediction based compression and entropy encoding.
The presented solutions achieve signiﬁcantly better compression rates than the benchmark provided
by FOSS, while maintaining fast decompression speeds. The best solution compresses a spectrum
to just 5.82 bits per ﬂoat, 1.88 bits per ﬂoat better than the benchmark, and decompresses in 0.41
ms.
To solve the problem of nearest neighbour search on a database of spectra, we present three dis-
tinct locality sensitive hashing-based solutions, including a novel approach. The solutions perform
extremely fast searches and scale well with the database size. On a 50 000 spectrum database, the
methods achieve search times that are over 2 000 times faster than a well-implemented brute force
approach with query results that are less than 0.1% worse than the brute force results. Each solution
has a version that can run on a compressed database with no decompression necessary to run the
search.
As the solutions to both elements of the challenge run fast and the search works on compressed
spectra, they are well suited for working in combination to solve the overall challenge.

Introduction
As methods of measuring the physical world improve, the resolution and frequency of measure-
ments increase, leading to larger and larger databases. Therefore, methods of storing and using
the databases must also continuously improve.
This thesis will cover such methods, speciﬁcally
compression and nearest neighbour search.
This thesis is written in collaboration with FOSS, a Danish company providing high-tech analytical
solutions for food and agriculture. Their analyses are based on electromagnetic spectra that describe
the composition of a material. These spectra consist of a sequence of numerical values. The methods
presented in this thesis are designed to work well on this data, but would likely work on other types
of similar data as well.
The challenge presented by FOSS, which we attempt to solve here, is that the growing number of
spectra needs to be stored eﬃciently and in a way that allows for nearest neighbour search. We
propose a two-part solution to this challenge.
First, we present diﬀerent compression methods that are able to achieve signiﬁcantly better com-
pression rates than the benchmark provided by FOSS while remaining competitive on decompression
speed. These methods build on prediction-based compression, where we attempt to predict values
based on preceding values, and only save the prediction error instead of the actual value.
The
compression methods and required background knowledge is presented in Section 2.
Second, we present several solutions for nearest neighbour search including a, to our knowledge, novel
locality sensitive hashing-based solution. Most of the presented solutions are based on approximation
techniques, but tests show that the quality of the results is close to optimal. The search times are
far better than any achieved by simple approaches. Furthermore, the solutions all include a version
that can run on a compressed database. These solutions and the theory required to understand
them is presented in Section 3.

Compression of spectra
This section presents our solutions to the problem of compressing a spectrum. We will analyse the
structure and characteristics of a spectrum to improve our understanding of the data. We then use
these ﬁndings to design solutions that can compress a spectrum with a high compression rate, fast
decompression, and without loss of information. We test the solutions and present the results before
discussing the design choices and trade-oﬀs we had to make.
The compression problem Given a spectrum, compress it losslessly with a high compression rate
such that it can be decompressed fast.
2.1
Background
2.1.1
Number representation
In a computer, diﬀerent types of numbers are represented diﬀerently. This section will brieﬂy describe
the number representations relevant to this paper.
Unsigned integer
Unsigned integers can be used to represent positive numbers. When interpreting a binary string B
of length n as an unsigned integer, the decimal (base-10) value can be calculated as:
decimal value = Bn−1 · 2n−1 + Bn−2 · 2n−2 + ... + B0 · 20
This means that a 32-bit unsigned integer can hold values in the range [0, 232−1] = [0, 4 294 967 295].
Signed integer
Signed integers are very similar to unsigned integers but they allow for negative values as well as
positive ones. This is done by reserving the most signiﬁcant bit (MSB) as a sign bit. There are a
few ways to interpret the sign bit, but the perhaps most commonly used is a method called two’s
complement where the sign bit represents −(2n−1). Therefore, the decimal value can be calculated
as [8]:
decimal value = Bn−1 · −(2n−1) + Bn−2 · 2n−2 + ... + B0 · 20
This allows 32-bit signed integers to hold values in the range [−231, 231−1] = [−2 147 483 648, 2 147 483 647].
Floating point
When fractional numbers are needed, ﬂoating point values (ﬂoats) can be used. The IEEE 754
technical standard deﬁnes a single precision ﬂoat using 32 bits [15]. The ﬁrst bit is the sign bit, the
next eight are exponent bits, and the remaining 23 are mantissa/fraction bits as shown in Figure
2.1. These bits are interpreted as a decimal number as follows:

2.1.3
Run-length encoding
Run-length encoding compresses a string of symbols by ﬁnding runs of recurring, identical symbols
[17, p. 179]. A run is then encoded as its length and the recurring symbol. For instance, a run-length
encoding of aaaaaaaaaabbbbb is (10,a)(5,b).
Run-length encoding performs well on structured data where symbols often occur in repetitions. On
more chaotic data, run-length encoding performs poorly and might even increase the ﬁle size.
2.1.4
LZ77
LZ77 is one of the most popular lossless compression algorithms and has been widely used since its
creation in 1977. It is a universal compression algorithm meaning it can work on any type of data.
The algorithm works by moving a sliding window across the data and replacing repeated occurrences
with a reference to where it happened previously in the window. The repeated occurrence is encoded
as a distance to the previous occurrence and its length [22].
2.1.5
Prediction-based compression
Generally speaking, prediction-based compression works by predicting a value based on its context,
such as the preceding values, and storing the diﬀerence between the actual value and the predicted
value. Ideally, the prediction is accurate which leads to a small diﬀerence, also referred to as the
residual, which can be compressed eﬃciently with entropy coding [9].
6
==PAGE:8==
It can be beneﬁcial to choose from a set of predictors. This is especially true when the compression
scheme will be used on diﬀerent types of data, or if there are sections within a ﬁle that are suﬃciently
diﬀerent to require distinct prediction methods. This is called a switched prediction scheme [9]. This
improvement in prediction ability comes at the cost of having to store the choice of predictor. If we
have two predictors, we could use one bit for each predicted value to represent the choice of predictor.
Thus, the gain in compression due to the improvement in prediction quality should exceed one bit.
2.1.6
Entropy
The entropy of a random variable is the amount of uncertainty in the variable’s possible outcomes.
Let X denote a discrete random variable with possible outcomes x1, x2, ..., xn. The entropy of X is
[17]:
H(X) = −
n
X
i=1
Pr[X = xi] log Pr[X = xi]
(2.3)
where Pr[X = xi] is the probability of outcome xi occurring. When using the binary logarithm (log
base-2), we can interpret the resulting entropy as the number of bits of entropy.
Consider ﬁrst a fair coin. It has two possible outcomes each with probability 0.5, so its entropy is
H(fair coin) = −(0.5 log2(0.5) + 0.5 log2(0.5)) = 1, so the fair coin has one bit of entropy. This is
the maximum entropy that a coin can have since all outcomes are equally likely which maximises
the uncertainty. Now consider a biased coin that has probability 0.75 of heads and 0.25 of tails.
It has entropy H(biased coin) = −(0.75 log2(0.75) + 0.25 log2(0.25)) = 0.82, so it has 0.82 bits of
entropy. The lower entropy value signiﬁes that there is less uncertainty in the outcome. If we guess
that the outcome is heads, we will be correct 75% of the time.
Calculating the entropy gives a theoretical bound on the compression we can hope to achieve. The
entropy corresponds to the minimal number of bits needed to encode each symbol [19]. A sequence
of tosses of the fair coin cannot be encoded by less than one bit per outcome. The biased coin, on
the other hand, is lower bounded by just 0.82 bits per outcome.
Unfortunately, this lower bound is not valid when the data is structured.
The deﬁnition from
Equation 2.3 states that X is a random variable. However, if we know that our data often has many
streaks of zeros, then the probability of a zero occurring is dependent on the symbol that came before
it. For instance, the sequence 001100110011 has Pr[X = 0] = 0.5 and Pr[X = 1] = 0.5 which gives
an entropy of 1, but if we know that digits always come in pairs then we can encode 00 as 0 and 11
as 1 which uses just 0.5 bits per symbol instead of the 1 bit per symbol lower bound given by the
entropy. To sum up, if there is a non-structured distribution over the possible outcomes, then the
entropy will give a lower bound on the bits per symbol needed to encode the sequence of outcomes.
2.1.7
Entropy coding
Entropy coding is a method of lossless compression where symbols are assigned to codes with lengths
that are inversely proportional to the frequency of the given symbol. This means that a symbol that
occurs often will get a short code, while symbols that occur less frequently get longer codes. Huﬀman
coding and Arithmetic coding are two examples of entropy coding algorithms.
Huﬀman coding
Huﬀman coding is an algorithm that ﬁnds an optimal codebook for a probability distribution of
symbols [11, 17]. A codebook is a function F : Σ →{0, 1}+ that maps symbols from an alphabet
7
==PAGE:9==
Figure 2.2: Huﬀman tree construction for the symbols D,C,B,A with frequencies 12, 5, 2, 1. A gets
code 111, B gets code 110, C gets code 10, and D gets code 0
Σ to preﬁx-free codes. Huﬀman’s algorithm builds optimal, variable-length codes such that no code
is a preﬁx of another code, and no auxiliary information is needed to delimiter between codes. A
tree-structured codebook is built using a full binary tree, a binary tree in which all nodes except
leaves have two children, such that symbols are stored in leaves and the root-to-leaf path is the code
for the symbol. Putting symbols in leaves ensures that the code for a symbol can never be a preﬁx
of another code.
The algorithm initially gives each symbol a node with priority corresponding to the frequency of
the symbol. The nodes are put in a priority queue. Then, the algorithm iteratively extracts the two
nodes a and b with minimum priorities. A new node c is created that has priority equal to the sums
of priorities of a and b. The new node has a and b as its children. The new node is placed back into
the priority queue. When there is only one node left in the priority queue, it must be the root of
the tree. Given a Huﬀman tree, the Huﬀman code for a symbol can be found by following the path
from the root to the leaf containing the symbol. When a left path is taken, add a 1 to the code,
when a right path is taken, add a 0 to the code. Figure 2.2 shows an example of the construction
of a Huﬀman tree for the symbols D,C,B,A with frequencies 12, 5, 2, 1, respectively. The ﬁnal tree
gives codes of length 3 to A and B, a code of length 2 to C, and a code of length 1 to D.

Arithmetic coding
We saw that Huﬀman coding deﬁnes a codebook such that each input symbol is mapped to an output
code. This can cause problems when the distribution of the input symbols requires a fractional
number of bits. For instance, a string baaaa consisting of 80% a’s and 20% b’s will get Huﬀman
codes a:0 and b:1 which means the encoded string is as long as the input string. Arithmetic coding
ﬁxes this by giving a code with length less than one bit to the symbol a and a code with more than
one bit to the symbol b.
Concretely, each input symbol is represented by a probability equal to its relative frequency in the
input string. The input string is then represented as a range by which it is uniquely identiﬁed as
shown to the left in Figure 2.3. To represent this range as a binary sequence, we iteratively halve
an interval starting at [0, 1[ until the interval is fully contained in the range that identiﬁes the input
string, as shown to the right in Figure 2.3. The steps taken to halve the interval give the binary
sequence that encodes the input string. In Figure 2.3, the input string baaaa gives the interval
[0.80000, 0.88192[ which fully contains the interval [0.8125, 0.8750[ so the ﬁnal code is 0010.
Comparison of Huﬀman and Arithmetic coding
Arithmetic coding can get arbitrarily close to entropy when the input is independently and identically
distributed. Huﬀman coding, on the other hand, only reaches entropy when all symbol probabilities
are powers of two [17]. This means that arithmetic coding tends to perform better than Huﬀman
coding.
Consider an input sequence from an alphabet with four symbols with probabilities 0.75, 0.15, 0.05, 0.05.
Naively, we can choose to use two bits per symbol to encode the sequence. If we Huﬀman encode
9
==PAGE:11==
the sequence, the symbol with probability 0.75 will get a code consisting of one bit, the symbol
with probability 0.15 will get two bits, and the remaining symbols will get codes consisting of three
bits. This gives an average of 0.75 · 1 + 0.15 · 2 + 0.05 · 3 + 0.05 · 3 = 1.35 bits per symbol. Finally,
arithmetic coding encodes the input string arbitrarily close to entropy, so we can calculate its en-
tropy to get the number of bits per symbol in the arithmetic encoding of the string. The entropy
is H(input string) = −(0.75log2(0.75) + 0.15log2(0.15) + 0.05log2(0.05) + 0.05log2(0.05)) = 1.15
which means we can encode the string using just 1.15 bits per symbol.
Despite the relatively simple theoretical foundations of arithmetic coding, implementing the arbitrary
precision arithmetic needed to calculate the range is far from trivial. Therefore, one may choose to
use Huﬀman codes if there is not much to gain from arithmetic coding.
Canonical Huﬀman codes
When a string has been encoded using Huﬀman codes, we need to save the Huﬀman codes in the
compressed ﬁle so it can be decompressed later on.
One way of representing Huﬀman codes is
by using Canonical Huﬀman codes. Canonical Huﬀman codes are very similar to regular Huﬀman
codes yet they follow a few basic rules that allow them to be represented compactly [18]. Canonical
Huﬀman codes have the same lengths as regular Huﬀman codes so the compression is unaﬀected.
Canonical Huﬀman codes are made to follow two rules:
1. When aligning codes at the leftmost bit by appending zeros to the right of shorter codes, the
numerical value of the originally shorter codes must be higher than that of the originally longer
codes
2. For codes that have equal lengths, the numerical value of codes increases in lexicographic order
Knowing that the codes follow these two rules, we can reconstruct the codes only from the code-
lengths of each symbol. Thus, with symbols from an alphabet of size a and codelengths from an
alphabet of size b, the codes can be represented using just a⌈log2 b⌉bits.

Characterisation of data
The data consists of a set of spectra. The spectra are created by measuring the absorbance of light
at diﬀerent wavelengths. This is done by shining light through a substance and detecting it on the
other side.
The spectra in the data set used for this paper consist of 4200 ﬂoats. These spectra have a high
resolution and have gone through processing to remove noise and sudden spikes which means they
are generally very smooth from point to point. Figure 2.4 shows an example of such a spectrum.
The close-ups in the middle and on the right show the smoothness of the spectrum.
Due to how the instruments work, measurements before and after point 1400 are measured diﬀerently
and are later stitched together.
2.3
Compression framework
Our compression framework consists of three components: prediction, residual calculation, and
encoding. Prediction serves the purpose of creating a value based on previous values, a prediction,
such that we only need to save the diﬀerence between the prediction and the actual value. Residual

calculation is the calculation of this diﬀerence between the prediction and the actual value. The

2.3.1
Prediction
If we look closely at a spectrum like in Figure 2.4, we quickly see that the points appear to be
following a clear pattern. This suggests that points can be predicted based on preceding points. A
natural method of predicting a point is using polynomials based on previous points, and extrapolating
to the point that we are trying to predict. Figure 2.6 shows an example of polynomial prediction
where the 5th point (black dot) is predicted using polynomials of degree 0 to 4. In this case, we see
that the 2nd, 3rd, and 4th degree polynomials are able to predict the next point accurately. These
polynomials were created by ﬁtting an nth degree polynomial to the n + 1 preceding points.
When a value has been predicted, we choose the closest prediction and calculate the residual which
is saved and compressed. When decompressing, we can calculate the original value using the residual
and the prediction.
11
==PAGE:13==
Figure 2.6: Predicting the black point using ﬁve polynomial predictors
Fitting polynomials is a computationally demanding task and doing so for each point in our spectrum
would take a long time. Furthermore, it would require that we somehow save the resulting polynomial
coeﬃcients. To avoid this, we use an alternative method as presented in [9] which allows us to directly
compute the value of a point as it would have been predicted by an nth degree polynomial ﬁtted to
the n + 1 preceding points. This can be computed for an arbitrary n value without much additional
computation. The ﬁrst four of these polynomial predictors are shown below, where Pn(i) is the nth
order prediction of point i.
P0(i) = s[i −1]
P1(i) = 2 · s[i −1] −s[i −2]
P2(i) = 3 · s[i −1] −3 · s[i −2] + s[i −3]
P3(i) = 4 · s[i −1] −6 · s[i −2] + 4 · s[i −3] −s[i −4]
(2.4)
As seen in Figure 2.6 we often have a situation where multiple predictors are very close to the actual
value. Therefore, having more predictors to choose from should give a higher chance of ﬁnding one
that accurately predicts the point. We tested this theory by using a varying number of polynomial
predictors ﬁtted to various numbers of preceding points, and seeing how it aﬀected the prediction
error. These results are shown in Figure 2.7 where we see that the more predictors we use, the
smaller the prediction error. However, the improvement in prediction has to be held up against the
time taken to calculate additional predictions and potentially having to represent the polynomial
choice in the compressed ﬁle.
Framed polynomial prediction
The decompression requires knowledge about which predictor was used to generate the residual so
that the process can be reversed. To avoid having to say which polynomial was used at each point,
they are grouped into frames and the polynomial that performs best across the frame is used to
predict all the points in the frame [9]. A frame is then simply a collection of residuals and a choice
of polynomial.
This method has the drawback of forcing all the points in a frame to use the same predictor.
Furthermore, the bits that are used to represent the choice of polynomial decrease the compression
rate.
12
==PAGE:14==
Figure 2.7: Prediction error decreases with the number of polynomial predictors
Implicit polynomial prediction
Alternatively, the choice of polynomial can be represented implicitly. This is done by deciding on
a predictor based on its ability to predict preceding points. To do so, the performance of diﬀerent
polynomials on previous points is compared such that s[i] uses the polynomial that best predicts
s[i −k, ..., i −1] for some constant k which we call the lookback.
This method does not use any space on representing its choice of polynomial. However, the choice
of polynomial is based only on previous values which can make the choice of polynomial suboptimal.

of leading zeros in the binary representation of the residual is proportional to the accuracy of the
prediction. Naturally, the residual computation must be reversible meaning that we can calculate
the actual value given the predicted value and the residual such that the compression is lossless.

Iterative delta residuals
The iterative delta residuals method of calculating residuals is diﬀerent since it does not use a
prediction value. This means that it skips the prediction step described in Section 2.3. It works by
iteratively replacing each value with the diﬀerence between itself and the preceding value. Figure

Doing this on ﬂoats generally does not work well for future encoding since a smaller ﬂoat does not
necessarily equate to more leading zeros in its binary representation. Therefore, we interpret the
values as signed integers when calculating delta values. Finally, we move the sign bit from the most
signiﬁcant bit (MSB) to a position after the least signiﬁcant bit (LSB). In other words, we left shift
and move the previous MSB to the new LSB position. This is done to prevent the case where a sign
bit results in zero leading zeros.
As described in Section 2.2, the values of a spectrum in our data generally change very little from
point to point. It is for this reason that this method is useful on this data.

very little structure once the leading zeros end. However, when the actual and predicted values fall
on either side of a power-of-two, we get some structure in the bits following the leading zeros.
As an example, we had a actual value 0.49191263 and a predicted value 0.49191290. This is an error
of 0.00000027, much smaller than the average error of 0.00017. Despite the accurate prediction, we
get a low leading zero count. The explanation can be found in the binary representations of the

actual value

= 00000000000000000000011111111011

zero count of just 21. However, the exponent boundary creates a streak of 8 leading ones. Thus, we
must use both the leading zeros and the leading ones to capture as much redundancy as possible.
The average residual has little to gain from encoding leading ones. We found an average leading

if we only look at the residuals with leading zero counts below 19, we ﬁnd an average leading one
count of 7.2. To get a better understanding of this interaction, see Figure 2.10. The ﬁgure shows
the average leading one count for the residuals with varying leading zero counts. The leading one

will thus have 18 + 8.4 = 26.4 bits of redundancy that we can encode eﬃciently.

2.5
Discussion
2.5.1
Huﬀman versus arithmetic coding
As described in Section 2.1.7, arithmetic encoding can get arbitrarily close to entropy when the input
is independent and identically distributed which means that it usually encodes using fewer bits than
Huﬀman encoding. Therefore, one might ask why we did not use arithmetic encoding on our leading
zeros and ones instead of Huﬀman coding. The reason is that arithmetic coding is diﬃcult and
time consuming to implement. Additionally, since arithmetic coding encodes using a number of
bits corresponding to the entropy, we can calculate how arithmetic coding would perform if we did
implement it. Our calculations show that using arithmetic coding could save us at most 0.04 bpf
compared to the 5.82 bpf achieved by the scheme using framed polynomials, integer residuals, and
Huﬀman coding of leading zeros given in Table 2.1.
2.5.2
Segmentation
The spectra in our data set have a signiﬁcant diﬀerence in predictability on either side of point 1400.
This makes it easy to decide to segment at 1400, however, there might be a better segmentation
that is not so obvious.
Furthermore, the number of segments and their placements might vary
between spectra. Therefore, future work includes designing an algorithm for determining the best
segmentation of a spectrum.
2.5.3
Reference compression
Since we generally have large groups of spectra that are very similar, our initial idea was to compress
them using a reference spectrum. The idea was that if a spectrum at a given index i more closely
24
==PAGE:26==
resembles other spectra at index i than preceding points in the same spectrum, we could predict
values based on the reference spectrum. It turned out that since the spectra vary so little from
point to point and do so in a predictable way, it was possible to more accurately predict values
based on preceding values rather than a reference spectrum. Furthermore, using reference spectra
for prediction would require one for each group of spectra which would cause problems in a small
database with high variation. Therefore, we decided not to pursue this idea further.
2.5.4
Generalisation to other types of input
The methods and schemes presented in this paper have all been developed speciﬁcally for the spectra
described in Section 2.2. However, this does not mean that the schemes could not work on other
inputs, but the new inputs must have certain characteristics to work well. First of all, the new input
must be a sequence of numerical values. For the prediction-based schemes to work, the new inputs
must resemble values that can be described using low degree polynomials. For the iterative delta
scheme to be competitive, the delta values, i.e. the diﬀerence from one value to the next, must be
small and consistent.
Using the presented methods and schemes on data which does not have the above-mentioned char-
acteristics would most likely require signiﬁcant modiﬁcations.

Nearest neighbours search on spectra
This section presents solutions to the problem of ﬁnding k approximate nearest neighbours in a
database of spectra where the distance between spectra is measured by their correlation coeﬃcient.
First, the background knowledge required to understand the subsequent sections is presented. Sec-
ond, the solutions are presented, and ﬁnally, results from tests on the solutions are presented and
analysed.
The searching problem Given a spectrum s and a database db of spectra, preprocess db such that
we can quickly ﬁnd k spectra from db that are approximately the most similar to s with respect to
their correlation coeﬃcient.
3.1
Background
3.1.1
Nearest neighbour search
Nearest neighbour search is the problem of ﬁnding the most similar point in a set of points to a given
target point. The search space can consist of many dimensions where each dimension represents an
attribute of the data points. These attributes could be coordinates in a Euclidean space or something
completely diﬀerent.
For low dimensional spaces this can be solved with methods that closely resemble those used in a one
dimensional space. However, for higher dimensional spaces the problem becomes harder and these
traditional techniques become less eﬀective. In fact, all solutions to the nearest neighbour problem
degrade to linear search when the dimensionality becomes suﬃciently high [21].
For this reason, focus has shifted from trying to solve the nearest neighbour problem to solving the
approximate nearest neighbour problem where we allow answers with a distance that is within a
constant factor of the actual nearest neighbour.
A variant of nearest neighbour search is k nearest neighbour search where the k closest neighbours
are found instead of just the closest.
3.1.2
K-d tree
The k-d tree allows for associative searching, meaning it can search for items in its structure based on
a combination of their attributes. The k represents the dimensionality of the data and search space
such that it reads as k-dimensional tree. K-d trees are essentially the multidimensional equivalent
of binary search trees, and in fact a k-d tree with a dimensionality of one is exactly a binary search
tree. Recall that a binary search tree works such that all internal nodes have higher or equal values
to those in their left subtree, and lower or equal values to those in their right subtree. In a k-d tree
with k > 1, the levels of the tree alternate between evaluating the diﬀerent attributes of the data
points. Consider the example of a k-d tree built on data with two attributes, lets call them x, y. In
this case the ﬁrst level, the root, would evaluate based on x such that all nodes in the left subtree of
the root have x values lower than or equal to that of the root while all nodes in the right subtree of
26
==PAGE:28==
the root have x values higher than or equal to the root. The next level would then evaluate based
on y following the same logic, then x again on the next level and so on [3].
Given a target point, k-d trees can be used to solve the nearest neighbour problem with respect to
the target point by running the following algorithm: traverse down the tree as when traversing in a
binary search tree. That is, at each level evaluate the attribute associated with the level and move
left if the target point is smaller, right if it is greater. On the way down, calculate the distance from
the target point to each node visited and keep track of the smallest found. After reaching a leaf,
traverse back up (backtrack) and at every node, evaluate if it is possible for the subtree not visited
to contain a point closer than the currently closest found. Traverse down the new subtree as before
if that is the case. The subtree not visited can contain a closer point if the distance between the
target and the cutoﬀmade by the current node is less than the smallest distance found so far. Keep
doing so until we are back at the root. To search for k nearest neighbours, simply keep track of
the k smallest distances found so far instead of just the one. This can be done eﬃciently using a
min-heap.
The analysis for k nearest neighbours search in a k-d tree is tricky as it depends both on the number
of items in the structure as well as the dimensionality of the search space. The initial traversal is
equivalent to that of a balanced binary search tree traversal which runs in O(log n) time where n
is the number of items in the structure. It is shown in [10] that the number of nodes visited in the
backtracking is dependant on the number of neighbours being searched for, k, and the dimensionality
of the search space, d, but not on the ﬁle size, n. Therefore, we can say that the expected search time
is O(log n) [10]. This expected search time only holds when d = O(log n) and k = O(1), otherwise
the number of extra nodes visited in backtracking becomes signiﬁcant. In the worst case we have
to visit all nodes. This means that k-d trees do not perform well on high-dimensional data where
d = Ω(log n).
3.1.3
Locality sensitive hashing
The idea behind locality sensitive hashing (LSH) is to have a hash function that hashes inputs such
that the more similar the inputs are, the higher the probability that they hash to the same value.
Such a hash function can then be used to calculate the hash codes of a set of items. This creates a
sort of grouping where similar items have the same hash codes which allows for approximate nearest
neighbour searching by calculating the hash code of the target item and looking in the resulting
bucket. The items found in this bucket can then be considered candidates for the nearest neighbour
and further evaluation can be done to ﬁnd the best candidate.
The similarity of two items x, y is evaluated through a similarity function sim(x, y) ∈[0, 1] where
0 means no similarity at all and 1 means full similarity. Alternatively, a distance function dist(x, y)
is sometimes used which is simply deﬁned as dist(x, y) = 1 −sim(x).
Formally, a locality sensitive hashing scheme on a family F of hash functions and similarity function
sim(x, y) ∈[0, 1] is deﬁned in Equation 3.1 [5]:
Prh∈F[h(x) = h(y)] = sim(x, y)
(3.1)

The deﬁnition given in Equation 3.1 requires that hash functions are designed such that the proba-
bility of two items hashing to the same value is exactly equal to their similarity. The deﬁnition given
in Equation 3.2 is less strict in that it does not require the probability to be exactly the similarity.
Instead, it requires that two points within a distance R of each other must hash to the same bucket
with some probability greater than or equal to p1 while two points farther apart than CR must hash
to the same bucket with a probability less than or equal to p2 where p1 > p2. By considering what
happens when R is varied, we can interpret this as meaning that the probability of x and y hashing
to the same value must be strictly decreasing as the distance between them increases.
To get an intuitive understanding of LSH, consider the case where our data exists in a one dimensional
space with values between 0 and 1 and the similarity function sim(x, y) = 1−dist(x, y) is used. The
hashing scheme samples a hash function by picking a random number between 0 and 1 and giving
every point that is less than the random number a hash code of 0 and everything else a 1. Since
the randomly sampled number has an equally high chance of being anywhere in the range [0, 1], the
probability of the sample landing between the two points and thereby giving them diﬀerent hash
codes is dist(x, y). Therefore, we have that Pr[h(x) ̸= h(y)] = dist(x, y) =⇒Pr[h(x) = h(y)] =
1 −dist(x, y) = sim(x, y).
Sampling just one hash function from a family of hash functions will separate the items into two
buckets. It is often useful to make more than just two buckets. This can be done by sampling
additional hash functions from the family of hash functions and concatenating their hash codes.
Then, the probability of two items hashing to the same concatenated hash code is the probability
of every hash function giving the same value to the two items. Building on the one dimensional
example from above, if l hash functions are sampled, the probability of x and y hashing to the same
concatenated code is sim(x, y)l.

In Section 3.1.3, we introduced a locality sensitive hashing scheme on one dimensional data. A

3.3.6
LSH common methods
This section presents common methods for the three LSH based schemes: PCC, RHP, and RAUC. In
the previous sections, we have described how a hash code is found for each spectrum in the database,
and how the hash code is found for a target spectrum. However, we have not described the methods
involved in ﬁnding the k candidates that are actually returned by a query.
Improving the stability of results
The three schemes all sample random cuts. A common issue for LSH based solutions is that the
buckets made by these cuts have hard cutoﬀs; a pair of points are either in the same bucket or they
are not. This can result in unfortunate cases where two points that are very close end up in diﬀerent
buckets. We solve this problem by keeping several instances of the same LSH scheme where each
scheme samples its cuts independently of the others. Queries are then run by querying each instance
of the LSH scheme and using the union of the candidates as the new set of candidates. If a pair of
points that are very close happen to be separated in one scheme, then one of the other schemes will
make up for it with very high likelihood. This greatly reduces the probability of nearest neighbours
being separated.
The number of instances needed to reach a desired stability is data dependent. We will show an
empirical analysis of the required number of instances in Section 3.4.2.
35

Ensuring that we ﬁnd k neighbours
It is diﬃcult to make any claims regarding the number of candidates in a bucket, and even with the
multiple instances described above, there is no guarantee that the set of candidates contains the k
candidates needed to answer a query. To ensure that we are always able to return k approximate
nearest neighbours, we allow the schemes to search using a subset of their hash functions. Using
fewer hash functions means larger buckets. This works such that when a query runs and the resulting
set of candidates has size less than k, we search using half of the hash codes, chosen at random. If
the candidate set is still too small, we keep halving the number of hash functions used until we have
at least k candidates.
To enable searching using a subset of the hash functions, we must store the corresponding hash
tables. Since the hash codes start with length l, they can be halved O(log l) times. Additionally, the
lengths are halved each time, so the hash tables will use space l n, l n

query times given in Table 3.1.
Finding the best candidates
When at least k candidates have been found, we have the task of returning the k best candidates. We
have developed two methods of doing so. The ﬁrst method, which we consider the normal method,
ﬁnds the correlation between the target spectrum and each candidate and returns the k most corre-
lated spectra. This means that the k spectra returned are the best among the candidates. However,
calculating the correlation between the target spectrum and each candidate is time consuming and
approaches the Bruteforce solution if we have many candidates. If c is the number of candidates
and m the length of a spectrum, then it takes O(c m log k) time to ﬁnd the correlation for each
candidate and return the k best.
The second method, which we call compressed compatible (CC), uses the many instances of the LSH
scheme as described above to ﬁnd the best candidates. The candidates are ranked according to the
number of times that they were hashed to the same bucket as the target spectrum in the I diﬀerent
instances of the scheme. The intuition is that a spectrum that was hashed to the same bucket as the
target in all I instances of the scheme is probably very close to the target, whereas a spectrum that
was only hashed to the same bucket once is probably further away. The ranks of the candidates can
be found in O(c I) time. Then we can use counting sort since all candidates have a rank between
1 and I in O(c + I) time [7]. Then, the top k ranked candidates are returned. In total, this takes
O(c I +c+I +k) = O(c I +k) time. Generally, c will not be much larger than k due to the methods
described above, and I will be constant in most realistic cases. Therefore, this method does not add
much to the query time. The schemes using this compressed compatible method have the suﬃx CC
in the results in Section 3.4.

3.4.3
Code length
The length of a hash code corresponds to the number of hash functions that were used to create the
hash code. Each additional hash code should split the spectra into twice as many buckets. With
smaller buckets we expect the spectra in a bucket to be more similar. Thus, the longer the hash
code, the better an average correlation we expect to see.
Figure 3.8 shows the average correlation and query time by code length for the three LSH schemes.
The test was run on a database consisting of 20 000 spectra. The ﬁgure shows that the average
correlation improves as the code length increases. However, beyond code lengths of about 10, we
see very small improvements. The average query time decreases as the code length increases.
These results indicate that we should use the longest code possible to achieve the highest correlation
and shortest query time. However, these improvements come at the cost of having to store the
additional hash functions and longer codes for all spectra.
3.4.4
Optimal frame size for RAUC
The frame size in RAUC determines how the spectrum is split up and summarised. Setting this
parameter low means that the curves are cut into many small pieces. If the frame size is set too low,
it could decrease the quality of the results as the summary data now only describes a few values.
On the other hand, if the frame size is too large, the frames contain too much information to be
captured in the simple summary data. This could also result in decreased quality of results. The
optimal frame size will vary for diﬀerent data types, where smoother curves will generally favour
larger frame sizes since the summary data can properly describe a larger section here.
We tested the eﬀect of diﬀerent frame sizes on our data, the results of which are shown in Figure
3.9. The scheme was run with a maximum code length of 32 using ﬁve instances. The entire thing
was then run ﬁve times and the results were averaged across these ﬁve. We see that for frame sizes
smaller than 1000 points, the results are generally good and quite stable.

3.5
Discussion
3.5.1
Validity of artiﬁcial spectra
We did not have enough spectra to properly test how the schemes perform on large databases.
Therefore, we artiﬁcially created more spectra as described in Section 3.2 by averaging pairs of
randomly chosen spectra. There are 10 types of food in the original data, so when creating new
spectra by averaging two random spectra, we will either average two spectra of the same food type
40
==PAGE:42==
or two spectra of diﬀerent food types. Averaging two spectra of the same food type will create a
new spectrum that closely resembles the two spectra from which it was created. However, when a
new spectrum is created by averaging two spectra from diﬀerent food types, we get a spectrum that
looks nothing like either of the spectra from which it was created. Thus, if the original data had 10
food types, the new data will have 10 · 10 = 100 food types.
If we assume that a larger database would have contained the same 10 food types as the one we
received, then the results from our artiﬁcial data are not completely representative. Having spectra
from 100 food types could potentially make it easier to bucket similar spectra since we have many
small clusters between which we can cut instead of large clusters that must be split up.
While the results are not representative of a larger database consisting of the same 10 food types,
it shows us that the schemes also perform well on a new type of data.
3.5.2
Methods of ﬁnding the best candidates
As described in Section 3.3.6, we have two diﬀerent ways of evaluating which of the candidates found
are best. We have the normal version which actually calculates the correlation between the target
and each candidate, and one we call compressed compatible, CC, which only evaluates candidates
based on their hash codes in the diﬀerent instances. Section 3.4.1 showed us that the normal version
gives slightly better results when measured on correlation but runs slower than the CC version.
However, the CC versions have some distinct advantages over the normal version when it comes
to usability. The CC versions never have to actually look at any spectra to reach a result to a
k approximate nearest neighbours query. This means that users can store the scheme structure
and not have to look in a database before knowing exactly which spectra to look up. This could
potentially save time and network traﬃc depending on the setup.

Conclusion
The objective of this thesis is to solve the challenge presented by FOSS of storing electromagnetic
spectra eﬃciently and in a way that allows for nearest neighbour searching. We split this into two
separate problems with the following deﬁnitions:
The compression problem Given a spectrum, compress it losslessly with a high compression rate
such that it can be decompressed fast.
The searching problem Given a spectrum s and a database db of spectra, preprocess db such that
we can quickly ﬁnd k spectra from db that are approximately the most similar to s with respect to
their correlation coeﬃcient.
In Section 2, we present several solutions to the compression problem, all achieving better com-
pression rates than the benchmark provided by FOSS. Most notable is the solution using framed
polynomial prediction, integer residuals, and Huﬀman coding of leading zero counts which achieves a
compression rate of 5.82 bits per ﬂoat where the benchmark uses 7.7 bits per ﬂoat. The decompres-
sion time for this solution is 0.41 ms. The compression rates, compression times, and decompression
times for all solutions can be found in Table 2.1.
Section 3 presents solutions to the searching problem. The solutions achieve much faster and better
scaling search times than a well-implemented brute force method. On a database consisting of 50 000
spectra, we achieve search times that are over 2 000 times faster than the brute force approach. These
solutions come with the trade-oﬀof giving approximate results, but as shown in Table 3.2, the results
are close to optimal. The results also show that PCC CC and RAUC CC are the fastest solutions,
using roughly 0.15 ms to search for 50 approximate nearest neighbours in a database of 50 000
spectra.
By combining our solutions to the two problems, we get an overall solution to the challenge presented
by FOSS. We then have a highly compressed database of spectra in which we can quickly search for
and decompress nearest neighbours of a target spectrum.
