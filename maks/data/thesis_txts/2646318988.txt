Abstract
Attacks which subvert the control-flow of legitimate computer programs are a large con-
cern in cybersecurity. Control-Flow Attestation (CFA) is the practice of validating the ex-
ecution path a program takes at runtime. Such validation needs to take place in real-time
in order to be an effective countermeasure to such attacks. This is a difficult challenge,
as it requires the extraction and analysis of information about the behavior of a program
during runtime. Many solutions have been proposed to address this issue. However,
they tend to make compromises between performance, security and usability. Clear and
precise data on the effectiveness and performance of available tools is required in order
to produce more effective and efficient real-time CFA solutions.
Intel Processor Trace (IPT) is an execution tracing feature of modern Intel processors
which provides detailed execution information in real-time with minimal performance im-
pact. As such, it has shown great promise as a tool for real-time CFA. The goal of this
thesis was to examine the viability of IPT for this purpose. In order to simulate the per-
formance characteristics of IPT in this context, a proof-of-concept solution was created,
which extracts the control-flow of a program in real-time using IPT data. This solution was
tested in an effort to reveal the relevant capabilities and drawbacks of IPT. The results
showed that IPT scales unfavorably with the complexity of the target program’s control-
flow. They also revealed a potential use case in closely scrutinizing critical or vulnerable
functions of programs. By limiting the trace context of IPT, it could likely be used to great
effect in applications requiring only partial control-flow reconstruction, or in combination
with more efficient, albeit less thorough approaches to real-time CFA.

Introduction
There are many methods of compromising a computer system. Many of these methods
aim to subvert the flow of execution of a running program, making it behave in ways it
was not designed to. This type of exploit is known as a control-flow attack. Programs
are made up of machine instructions which are stored in virtually contiguous regions of
memory and executed sequentially from a given starting point. Some instructions can alter
the order of execution by directing the instruction pointer (IP) to a different location, from
which execution will then continue. These instructions are known as branches. The flow
of execution between contiguous blocks of code is referred to as a program’s control-flow.
Several software vulnerabilities enable control-flow exploits, attacks which compromise
the Control-Flow Integrity (CFI) of the target program. Buffer overflow vulnerabilities - for
example - can often be utilized for this purpose. The most basic form of this vulnerability
arises when user input is read, without limiting its length, into a fixed-length buffer. Sim-
ply put, this allows a malicious user to overwrite different parts of the program’s memory,
including other variables and return addresses. When a function is called, the IP imme-
diately following the call instruction is stored as a return pointer for the called function.
When that function returns, control-flow is redirected to that pointer. Control over this
data can allow an attacker to hijack another program’s process to execute arbitrary code.
There are ways to mitigate the risk of control-flow attacks. They include address space
layout randomization, non-executable stacks and stack canaries. These can be over-
come, however, and more robust protection requires some way of verifying that a pro-
gram’s control-flow is as expected. This is the domain of Control-Flow Attestation (CFA),
an essential practice in the defense against control-flow attacks. There are many different
ways to approach this, but they all include tracing a program’s execution, interpreting the
branches it takes and validating them.
Ideally, this verification should occur in real-time. If a control-flow attack can be detected
as soon as the first illegitimate branch executes, the process can be terminated to prevent
further compromise. To accomplish this, execution tracing and attestation must run in par-
allel. There are many challenges to this approach, most of which have to do with perfor-
mance. Of course, a program takes longer to execute if each branch requires validation.
There are also performance issues with many execution tracing methods. Traditionally,
they operate via system interrupts, or some compile-time instrumentation of the target
program, both of which can significantly slow down the execution of the target program.
The literature review for this thesis includes research into control-flow attack vectors and
CFA as a protection mechanism. In particular, it covers existing real-time CFA solutions,
current challenges and unanswered questions in that active and progressive area of re-
search. This thesis is intended to provide additional groundwork for future research in
this area by examining the potential of one promising execution tracing mechanism to
overcome some of the current challenges.
Many existing solutions have taken clever steps to mitigate these challenges, but they
tend to involve compromise between security and performance. The most secure solu-
tions tend to be slow and those that are fast and efficient tend to be less secure. There
are also those that further mitigate their limitations but require specialized hardware or
modifications to the target program. The ultimate goal for this line of research is to find a

reliable and widely compatible method of tracing and attesting program execution quickly
and efficiently in real-time, while providing protection against any control-flow corruption.
One of the most difficult performance challenges facing such solutions is scalability. For
every branch in the target program, the attestor must verify the source and target of that
branch against some model. This operation will always be more costly in terms of re-
sources than the branch itself. This means that the more a program’s control-flow com-
plexity increases, so does the performance overhead incurred by real-time CFA.
Intel Processor Trace (IPT) is an execution tracing feature included in Intel’s recent Core
and Atom line processors. Being implemented in hardware, it is able to perform its role
with very little performance impact, enabling the complete reconstruction of the traced
execution at the machine instruction level. Typically, it is used to generate execution
trace files for post-mortem analysis, though it has been used for the purpose of real-time
CFA with certain limitations.
Given the current status of research in this area, it is clear that some creative innovation is
necessary to create a solution which solves all the current challenges. Further research
will reveal more information about the practicality of available tools for such innovation.
In order to make this possible, more data is required. The goal of this thesis is therefore
to provide useful data on the performance characteristics of IPT when used in real-time
CFA. This will hopefully encourage and facilitate further research into this technology as a
powerful utility in the pursuit of widespread, fast and thorough real-time CFA. In particular,
this thesis will shed light on the scaling characteristics of IPT with regard to the target
program’s control-flow complexity and overall runtime.
This thesis includes the development of a minimal proof-of-concept solution which traces
a program’s execution and reconstructs its control-flow in real-time using IPT, providing a
realistic estimate of the performance characteristics of a full CFA solution. This solution
is then tested thoroughly against controlled test cases to give a full profile of its scaling
characteristics.

Related Work
This chapter covers the main work on which this thesis is based, the theory behind the
relevant concepts introduced therein and existing solutions in the field.
2.1
Control-Flow Attestation
Control-flow attestation (CFA) is an important security measure. It is intended to accu-
rately detect deviations from the intended control-flow of a program. The monitoring and
enforcement of correct control-flow is also referred to as Control-Flow Integrity (CFI) en-
forcement [Abadi et al. 2009]. The process of CFA can be divided into two parts: the
definition of a correct control-flow profile, in the form of a control-flow graph (CFG) or
constraint-based policy, and the validation of a program against that profile.
Such profiles can be generated in a number of ways, including manual definition and
binary analysis. The correctness and completeness of these profiles directly impact the
utility and security of CFA performed against it [Xu, Sun, and Su 2010].
2.1.1
Static vs. Dynamic CFA
There are different methods for attesting a program’s CFI. The static approach relies on
source code to generate a CFG containing all possible execution paths a program could
legitimately take. It has been used to great effect as a way to determine whether an ex-
ecutable has been tampered with, or detect potential vulnerabilities to control-flow-based
attacks [Cheng et al. 2019]. This method is a useful preventative measure, but it has
some drawbacks: it scales quickly with increased control-flow complexity and has trouble
identifying legitimate target addresses for some indirect branches.
The dynamic approach, however, involves tracing the execution of a program, mapping
the execution of - and branching between - contiguous blocks of code. In essence, it
generates a partial CFG, or a single traversal of the program’s CFG. This can be used
to verify that particular execution - or set of executions of the program against the estab-
lished profile. This has the advantage of only tracing a single traversal of the CFG each
time before validation. It scales much more favorably with control-flow complexity since
it does not require the computation of all potential traversals of the CFG. It can also be
beneficial in narrowing down potential target addresses for indirect branches [Kinder and
Kravchenko 2012].
2.1.2
Real-Time CFA
Real-time CFA is dynamic CFA performed during execution of the traced program. Often,
some initial static or hybrid analysis is used to generate a CFG for a given program. Then
its execution can be monitored in real-time and verified against that CFG during execution.
This is computationally expensive and is mostly applied on simple, embedded platforms
and isolated OS kernel functionality in practice. Several challenges remain unsolved in
establishing ubiquitous solutions for real-time CFA of user-space applications on more
complex systems.
Each approach has its own advantages and drawbacks. Some tracing systems operate
by interrupting the CPU during execution to gather information about its state. These tend
to add significant performance overheads to the execution times of the traced programs.
Others are more efficient, but require compile-time instrumentation of the target program
in order for the CPU to emit the required data. There are also systems which require

2.2
Extended Berkeley Packet Filters
Berkeley Packet filters (BPF), also known as BSD packet filters, were initially proposed
as a network filtering tool [McCanne and Jacobson 1993]. It was extremely efficient com-
pared to existing tools at the time and allowed for fine-grained, modular filtering with lit-
tle performance overhead. The Linux kernel later added support for extended Berkeley
Packet Filters (eBPF) [Corbet 2014]. It allows BPF to be applied to things other than
network packets, such as tracepoints and sockets. This enables real-time monitoring of
system calls and communication between processes.
2.3
Existing Solutions
Many solutions exist that approach real-time CFA in different ways, each with their own
capabilities and limitations. This section lists and describes a notable selection of these.
2.3.1
C-FLAT
C-FLAT is a remote runtime CFA tool for embedded systems. It requires modifications to
the target binaries, adding so-called trampolines at each branch instruction, so that they
inform the tracer each time a branch is encountered. An external, trusted device handles
the actual attestation from there. The attestation is stateful, i.e., it is able to match an entire
execution path to its CFG. C-FLAT unloads a large portion of the performance overhead
to external hardware. What remains on the target machine is linear overhead correlated
with the number of branch instructions in the target program [Abera, Asokan, et al. 2016].
2.3.2
LiteHAX
LiteHAX is a remote runtime CFA tool for low-end embedded systems. It attests not only
the control-flow of a program, but also its data flow, allowing for the detection of a very
wide variety of attacks. Like C-FLAT, it does require specialized hardware to function. In
their case study, the researchers measured a ≈15% performance overhead on the prover
device [Dessouky et al. 2018].
2.3.3
Microsoft Control-Flow Guard
Microsoft Control-Flow Guard is a local real-time CFA tool first introduced in an update to
Windows 8.1 in November 2014 and included in all versions of Windows 10. It requires
compile-time software instrumentation which adds runtime target checks at each indirect
call instruction. Though this is an efficient protection mechanism against indirect call cor-
ruption, it does not check other indirect branches, resulting in limited protection [Control
Flow Guard - Win32 apps 2018].
2.3.4
CFIMon
CFIMon is a tool that leverages Intel Branch Trace Store (BTS) functionality - in combi-
nation with performance counters - to validate branches locally at runtime with very little
overhead. It operates without special modifications to program binaries and requires only
a compatible Intel processor to function. However, it is limited in that it only validates
branch targets without context, and cannot detect high level semantic control-flow viola-
tions [Yubin Xia et al. 2012].
2.3.5
GRIFFIN
GRIFFIN uses Intel hardware tracing functionality in execution tracing and CFI enforce-
ment with a hybrid approach. Requiring a compatible Intel processor to function, it uses
simple security policies for validation. These policies can be course-grained, providing a

lookup table of legitimate branch destination addresses, allowing for very fast validation
of branch destinations. To balance security and performance, GRIFFIN also supports
fine-grained policies. These denote legitimate branches in terms of both source and des-
tination addresses. Validating against this is much more secure, due to the fact that ille-
gitimate control-flow may still branch to legitimate destination addresses. It is also much
slower since in order to extract the source address of the branch the control-flow must be
reconstructed from the trace. GRIFFIN is capable of switching between course- and fine-
grained policies at runtime to minimize the performance overhead and delay between the
occurrence and detection of control-flow violations during execution [Ge, Cui, and Jaeger
2017].
2.4
Intel Processor Tracing
IPT is an architecture extension introduced and implemented in Intel processor microar-
chitectures from Broadwell and Apollo Lake onward. IPT provides hardware support for
tracing code execution with minimal CPU overhead. It defines a set of model-specific reg-
isters (MSRs) the operating system can use to enable and configure tracing, and outputs
a stream of packets (see section 2.4.1) containing compressed execution information.
This packet stream can be written to a memory buffer. The most common use case is
recording the IPT packet stream to disk for post-mortem decoding [R. 2013], though it has
been used for real-time CFA in GRIFFIN (see section 2.3.5).
To supplement the IPT trace, the operating system can also provide sideband information,
such as linked binaries and context-switching when tracing multiple threads on the same
CPU. This information can be used to correlate the IPT trace with the linked binaries
to reconstruct the traced execution. IPT generates very large amounts of data when
tracing, so some filtering is necessary in order to isolate the relevant information. For
this reason, IPT supports multiple ways of narrowing down the context of a trace. These
include filtering by user- or kernel-space, process and IP. Thus, it is possible to narrow
the trace context down to a single function, or even part of a function [Intel® 64 and IA-32
Architectures Software Developer Manual 2020].
2.4.1
Packets
The trace stream generated by IPT consists of packets of various types, encoding in-
formation about the processor’s behavior. In this section, the relevant packet types are
listed and briefly explained. Some packet types and sub-types which provide no infor-
mation about control-flow have been omitted [Intel® 64 and IA-32 Architectures Software
Developer Manual 2020].
• PSB: Packet Stream Boundary
These packets are generated at regular intervals
and act as markers for the decoder to synchronize with the packet stream.
• OVF: Overflow
When the processor drops packets due to an internal buffer over-
flow, it generates an OVF packet to inform the decoder of this loss of data.
• TNT: Taken Not-Taken
These packets contain information on whether conditional
branches were taken during execution, each branch represented as a single bit.
They are instrumental in reconstructing the control-flow of a program execution.
• TIP: Target Instruction Pointer
For various branching events - including (but not
limited to) returns, interrupts and exceptions - a TIP packet is generated, containing
the target instruction pointer of that branch.
• FUP: Flow Update Packet
In cases of interrupts, exceptions and other events

for which the source instruction pointer cannot be reliably determined, a FUP packet
is generated, containing the source address.
• MODE
These packets encode information about the current execution context of
the trace, including execution mode (16-, 32- or 64-bit) and TSX transaction events.
2.4.2
Intel libipt
Intel provides an open-source reference IPT decoder library implementation called libipt1.
The library includes sample implementations of several utilities: a packet dumper, trace
disassembler, trace test generator and several utility scripts for working with recorded
traces.
Layers
When it comes to decoding IPT traces, libipt offers several different layers of abstraction
for interpreting the IPT packet stream:
• Packet
On this layer, the output of the decoder will consist of raw IPT packets.
This will expose all of the information gathered by IPT and thus enables very detailed
analysis of the recorded execution.
• Event
Also referred to as the query layer, the event layer encodes combinations
of packets into higher level CPU events for a slightly broader and more abstract
overview of CPU activity.
• Instruction
This layer combines a trace with the source binaries - the execution
of which was traced - to reconstruct the entire execution at the machine instruction
level. The instruction layer is more abstract than the event and packet layers and
provides a comparatively legible view of the recorded execution.
• Block
The block layer further abstracts the instruction layer into code blocks.
These blocks encode contiguous regions of executed instructions, making this layer
ideal for CFA. It operates faster than the instruction layer, though additional post-
processing is required in order to analyze individual instructions within the blocks.
2.4.3
Linux perf
IPT support was added to the Linux kernel in version 4.2 via perf, a set of tools for per-
formance monitoring and execution tracing. This allows Linux user-space applications
to gather and inspect IPT trace data. More importantly for this research, the underlying
kernel module allows a process with sufficient privileges to interact with IPT via a set
of system calls and IOCTL requests. This enables the development of a simple tracing
program which traces directly to system memory [Perf tools support for Intel® Processor
Trace - Perf Wiki 2020].

3
Problem Statement
Current solutions are mostly limited to embedded systems due to the significant perfor-
mance overhead invoked by applying execution tracing techniques to real-time CFA. The
main issue is scalability. Existing local real-time CFA solutions can reach a performance
overhead upwards of 35% in real-world applications [Dessouky et al. 2018, Yubin Xia
et al. 2012, Ge, Cui, and Jaeger 2017], which may be acceptable for simple embedded
systems, but can be crippling for larger machines running more complex software.
The tracing overhead of IPT is extremely low - around 5% [Bakhvalov 2019] - due in part to
the fact that it is implemented in hardware and does not operate via CPU interrupts. How-
ever, decoding the trace is several orders of magnitude slower, meaning that it will almost
certainly be the bottleneck for real-time CFA using IPT. As discussed in section 2.3.5,
GRIFFIN, which uses IPT for CFA, mitigates this issue by switching between course- and
fine-grained policies. This allows it to speed up by avoiding the reconstruction of control-
flow when needed, balancing security and performance [Ge, Cui, and Jaeger 2017].
This poses interesting research questions: What are the performance characteristics of
IPT when reconstructing the control-flow of a program in real-time, how does the control-
flow structure and complexity of the target program affect the performance overhead of
such an application, and are there ways to mitigate this performance overhead without
sacrificing the security of fine-grained analysis? It is the goal of this thesis to explore and
answer these questions.
3.1
System Requirements
IPT does not require any special instrumentation of the target binary to be traced. Nor
does it require specialized, external hardware. The only fixed requirement is a working
system equipped with a compatible Intel processor (see section 2.4). Memory require-
ments depend primarily on the selected IPT buffer size, the sizes of binaries loaded by
the target program and the attestation model used (see section 6.3.1).
3.2
Performance Concerns
Many factors affect the performance overhead of a real-time execution tracing solution.
It is likely, due to the decoding bottleneck of IPT, that the largest performance factor will
be the branch density of the target program. I.e., the number of branching instructions
per total number of instructions. Additional factors to take into account are the scope of
the trace context (see section 2.4) and the relative portion of time spent within it during
execution. These are all explored in section 6.3.
3.3
Formal Definition
Given a program P, defined as a sequence of N code blocks B0, ..., BN−1, such that
each block is a memory address offset at the beginning of a sequence of 0 or more non-
branching instructions, followed by 1 branching instruction in the source binary of P. Let
B0 →B1 denote a control-flow transition from B0 to B1, more specifically, from the last
instruction of B0 to the first instruction of B1. Unless otherwise specified, it is assumed
that a block identifier on the left-hand side of →refers to the last instruction of the identified
block, and an identifier on the right-hand side refers to the first instruction of that block.
To specify a different instruction within a block, the following notation can be used: Given

branches as transitions between them. S denotes the start state of an execution and E

4
Design
Given the amount of active research in this area, many considerations need to be made
for an evaluation such as this, from tool selection and design to test plans. This chap-
ter documents the design process for a minimalistic and reliable way of assessing the
performance characteristics of IPT as the basis for a real-time CFA solution.
4.1
Goals
The overarching goal of this project is to create and evaluate a proof-of-concept imple-
mentation of such a solution using IPT. This solution will be similar to GRIFFIN operating
on a fine-grained policy (see section 2.3.5), with a few important exceptions:
1. The solution will not perform any attestation on the target program, but focus on
extracting enough control-flow information to enable it. The efficacy of this approach
to CFA has already been established along with the negligible performance impact
of the attestation itself [Ge, Cui, and Jaeger 2017], and this abstraction allows for a
better focus on the performance of the control-flow reconstruction.
2. The solution will not handle multi-process or multi-thread target programs. Nor will
it handle programs that make use of Just-In-Time (JIT) compilation. These features
are not necessary to run the test plan (see section 6.1) and would unduly complicate
development.
3. The solution will run in user-space and require no changes to the kernel. This sim-
plifies development but restricts functionality. The tracing program will handle the
execution of the target program to ensure the required permissions to manage its
process at runtime.
4. The solution will not employ parallel processing in decoding the trace output. Since it
will only be targeting a single process, and context-sensitive CFA requires sequential
decoding, a single decoder will suffice.
5. The solution will filter the trace down to a single function, and not trace the entire
process. This will make it easier to control the control-flow complexity of test cases.
The solution shall be able to launch, trace and profile the control-flow of the various test
cases introduced in chapter 6.
4.1.1
Performance Analysis
The evaluation of this solution is intended to provide an objective view on the practical
viability of using IPT for real-time CFA. Multiple sample programs with varying control-
flow complexity will be written to test the effectiveness and efficiency of the solution. The
sample programs will be used to show whether it accurately detects different types of
branches and measure its scaling characteristics. The performance testing process will
focus on two primary metrics:
• Performance overhead
Measure the running times of the target applications with
and without the tracing solution. This will enable the computation of the performance
overhead incurred by tracing and decoding.
• Scalability
Determine how the control-flow complexity of the target program im-
pacts the performance overhead of the analysis. This shall be done by correlating
the performance overhead with a scaling factor for the target’s complexity.

4.2
Technical Building Blocks
This section covers the technical foundation on which this solution is built, including the
relevant IPT components, as well as key concept definitions.
4.2.1
IPT Components
As stated in section 2.4, the trace output of IPT is highly compressed. To achieve this,
IPT relies on a set of principles and design elements intended to minimize its output while
guaranteeing full reproducibility of the traced execution. One core assumption is that all
binary code run during the trace is available when decoding, along with information on how
it was loaded. Based on this assumption, IPT can focus on encoding only the information
required to reconstruct the execution flow from the source binaries [Intel® 64 and IA-32
Architectures Software Developer Manual 2020].
Trace Output
IPT supports several output schemes, including a contiguous region of physical memory,
a table of physical address ranges and various platform-specific trace transport subsys-
tems. Whichever scheme is used, IPT bypasses the CPU cache and writes directly to
memory instead, which is a key factor in its low performance overhead.
The output of IPT is organized into packets, as introduced in section 2.4.1. These packets
encode information about the traced execution. Some are more relevant to this endeavor
than others, since only a few provide information about control-flow directly. For example,
PIP packets are of little interest, since the solution is only meant to trace a single process.
TSC and CBR packets are similarly irrelevant for the operation of this solution, since TSC
is intended for correlation with timestamped data and the solution only deals with static
binaries, and CBR is primarily an indicator of system performance and does not yield any
information about control-flow.

Change of Flow Instructions
IPT makes use of the concept of basic code blocks, contiguous regions of code executed
in order. Code within a block does not need to be traced. Certain instructions and events
may change the control-flow of a program. Collectively, these are known as Change of
Flow Instructions (COFI), and form the main focus of attention for IPT’s trace output. This
term encompasses branch instructions and events such as exceptions and interrupts.
Different COFI require different handling when generating the trace output. The three
basic categories of COFI govern how they are handled when encountered.
• Direct transfer COFI
These include conditional branches and unconditional di-
rect jumps. The relative target addresses of direct transfers are embedded in the
instructions themselves, and can therefore be obtained directly from the source bi-
nary. For conditional branches, IPT only encodes a single bit in a TNT packet, indi-
cating whether the branch was taken. Unconditional direct jumps are not encoded
in the trace unless they cause a transfer into - or out of the trace IP filter region.
• Indirect transfer COFI
These include indirect jumps, calls and returns; branches
which resolve their target addresses at runtime from memory or a register. Since the
target addresses cannot be obtained from the source binary, IPT encodes them in
TIP packets, with one exception: In most cases, the target IP of a return instruction
can be derived from the call stack and source binary. In those cases, given that a

15
==PAGE:23==
corresponding call instruction has been recorded since the previous PSB packet,
return instructions are compressed to a single 1-bit in a TNT packet.
• Far transfer COFI
These include exceptions, interrupts, traps, TSX aborts, sys-
tem calls and other instructions which are not covered by the other categories. IPT
generates a TIP packet for each far transfer encountered. Some far transfers are
asynchronous events, and thus their source IPs cannot be accurately derived from
the source binary. For those events, IPT generates FUP packets containing the IP
at which the event occurred.
Trace Filtering
IPT supports three different methods of trace filtering. These filtering mechanisms control
the circumstances under which tracing is enabled. This can be used to limit the scope of
the trace, which greatly reduces the performance overhead of both the tracing itself and
the decoding. The following is a short description of each filtering capability and how they
are utilized in this solution.
• CPL filtering
Current Privilege Level (CPL) filtering allows IPT to enable and
disable tracing based on the privilege level of the traced execution. This filter has
three settings: CPL > 0, CPL = 0 and regardless of CPL. In practice, this means
that IPT can be configured to trace only user-space code (CPL > 0), kernel-space
code (CPL = 0) or all code. For the purposes of this research, the focus shall be
on user-space code (CPL > 0).
• CR3 filtering
Control Register 3 (CR3) points to the page tables for the currently
executing process. It also contains a process-context identifier, which can be used
to uniquely identify a running process. Filtering by CR3 value therefore allows IPT
to enable and disable tracing based on which process is currently being executed.
Effectively, this means that the trace can be limited to a single process.
• IP filtering
As mentioned in section 2.4, IPT also enables filtering by the instruc-
tion pointer address. This allows fine-grained control over the scope of the trace,
down to the instruction level. This enables the isolation of a single function, or even
part of a function for tracing.

Key Concepts
This section lists and defines useful key concepts used when discussing this solution. All
of the following definitions apply in the context of this solution.
• Target
The application being traced.
• Tracer
In addition to IPT itself, this refers to the part of the application which
manages the execution of the target process, loading of target binaries, initialization
of IPT and decoder setup.
• Decoder
The part of the application that runs in parallel with IPT and the target
application. It reads the buffer, decodes as necessary and outputs a profile of the
observed control-flow.

The following set of broad design principles were determined before implementation in
order to help guide the development of the desired solution.
• Focus on control-flow
Practically, this means that the decoder should operate
on the block layer. This seems to be the best choice for CFA because it allows omit-

tance of most of the irrelevant trace data and puts focus more clearly on branching

• Reuse open-source code where possible
Much of the hard work in developing
the tools required to interact with IPT and managing the packet stream has already
been done as part of libipt and perf tools, both of which are open-source. The

• Minimize post-processing
As stated in chapter 3, decoding the trace packet

Therefore, it is important to limit decoding as much as possible to reduce the per-

• Limit tracing context
As discussed in section 4.2.1, IPT supports IP filtering,
meaning that the context for which the trace will be captured exclusively can be
manually defined. This will not only minimize the already low performance overhead
of recording the trace, but also simplify the testing process and allow the tracer to
ignore the relatively massive amount of extraneous code included when executing
a simple program.

by the tracer and traced by IPT. The decoder is started once tracing has begun. It reads
from the trace buffer and reconstructs the control-flow of the target execution. A sequence

4.6
Notable Design Choices
This section aims to detail and justify certain choices that were made in the design of this
solution.
4.6.1
IPT Setup
Few design challenges were posed by the setup of IPT itself. The notable ones had to do
with the allocation of the trace output buffer itself.
Trace Buffer Type
When allocating the trace output buffer in memory, IPT provides two options: a ring buffer
and a linear buffer. These are distinguished during allocation by their protection flags. A
read-only buffer is treated as a ring buffer, so that when the buffer is full, IPT will keep
writing from the beginning of the buffer. A read-write buffer, however, is treated as a
linear buffer. When a linear buffer is full, all subsequent packets are dropped. The ring
buffer can be useful as it effectively allows IPT to trace indefinitely. Special considerations
need to be made when implementing a decoder operating on a ring buffer to ensure that
it keeps reading from the start of the ring buffer where the trace continues. Though this
functionality was deemed unnecessary for the achievement of the test plan, the ring buffer
was selected, though without the necessary decoder customizations required to make use
of its benefits.
Trace Buffer Size
The size of the trace buffer is a very important factor and the required size depends on the
amount of tracing data generated. Initially, when assessing the efficacy of this solution,
the buffer size was set to 4KB, the lowest possible value, which proved to be more than
large enough to accommodate each of the effectiveness test cases (see section 6.2). The
performance test cases, however, presented a more difficult decision. Due to difficulties
in estimating the required buffer size for a given program execution, it was decided to
leave this up to trial and error during performance testing. Details on this process can be
found in section 6.3.1.
4.6.2
Compatibility
Some compatibility limitations became apparent during the design and early implementa-
tion phases of this project. The decision was made, in accordance with the stated design
principles (see section 4.3), not to address such issues unless necessary to run the test
plan as laid out in section 6.1.
32-bit Support
Both IPT and libipt have built-in support for 32-bit binaries. However, when customizing
the binary loading functionality of libipt, several issues with address extraction arose when

attempting to apply the changes made to the 64-bit ELF loader to the 32-bit loader. Since
all tests would be run on 64-bit binaries on a 64-bit processor, this effort was abandoned
as unnecessary.
Multi-Thread Support
In order to support multi-thread and/or multi-process targets, special considerations need
to be taken. Since all tests would be run on single-thread, single-process targets, and the
amount of additional effort and complexity such support would add to both development
and performance evaluation, the decision was made not to implement this.
JIT Support
Support for JIT is a non-trivial problem in execution tracing, as noted by Ge et al., who
also describe their solution to this problem in the development of GRIFFIN [Ge, Cui, and
Jaeger 2017]. Although JIT compilation is a widely used tool nowadays, support for it was
deemed irrelevant to the goal of this thesis.

5
Implementation
This chapter covers the implementation of the design specified in chapter 4. The solution
is a single program consisting of two main components: the tracer and the decoder. The
tracer encapsulates the execution of the target program and setup of IPT so that it outputs
to a user-space memory buffer.
The decoder reads the IPT buffer and decodes it in parallel with IPT and the target pro-
gram. It inspects the execution block-by-block in order to minimize the decoding required
and identifies entries into, exits out of, and branching operations inside the user-specified
trace context. The source code is available on GitHub1.
5.1
Tracer
The tracer uses perf tools to interact with IPT. The program accepts the full path of a target
executable file, the trace buffer size in 4KB pages and an optional context filter string that
is either a function name or a range of address offsets as parameters. This solution only
supports filtering on code contained in the executed binary file itself.
A child process is spawned for the target program. The tracer uses the perf_event_open
system call to instantiate a file descriptor for the IPT packet stream and maps it to a
read-only memory buffer. It then proceeds to initialize IPT, filtering the trace by the target
process ID.
No sideband information is captured. The tracer instead relies on loading the executable
files involved into the decoder’s image section cache. Once all linked executables have
been loaded into the decoder’s image section cache, the tracer applies the user-specified
IP filter and enables IPT packet generation.
5.2
Decoder
The decoder is a modified version of libipt’s ptxed block decoder. The decoder was set up
to read the IPT packet stream directly from the active memory buffer rather than a trace
file. This was done by removing the standard functionality which loads a trace file into
memory and passes the mapped address range to the decoder configuration. Instead,
the tracer passes the active recording buffer to the configuration directly. Furthermore, the
block decoder was streamlined so that it only decodes the last instruction of each block
in order to identify and analyze branches taken during execution.
5.3
Challenges
Several challenges were encountered in the creation of this solution. This section aims to
detail these challenges, their implications and decisions made to mitigate or solve them.
5.3.1
Concurrency Issues
The target program’s process must be spawned in order to obtain its process ID. This is
a prerequisite to the instantiation of the IPT file descriptor with process filtering. Further
initialization is required after the initial IPT setup. This, and the fact that the tracer and
target process run in parallel, posed a number of challenges in the development of this
solution.

Target Process Exit During Setup
The most immediate issue encountered as a result of this was that the target process
tended to exit before the tracer finished reading the mapping information required to pop-
ulate the decoder’s image cache. This caused the tracer to crash or hang before starting
the decoder.
The solution of choice was to use ptrace to set the target process up such that it hit a
breakpoint immediately at the start of execution. This allowed the tracer to finish the
required setup before instructing the target process to continue.
Empty Buffer at Decoder Start
Another concurrency issue was encountered only when the IP filter was added. When
filtering on a function, for example, the IPT buffer remained empty until the process IP en-
tered the filter context. This caused the decoder to discover an empty buffer, immediately
report an end of stream status and quit.
Due to the way ptxed is set up, this was easily addressed. Rather than synchronizing
once at the start of decoding, the decoder was made to synchronize repeatedly until the
status is not end of stream. This means that the decoder will wait for input to appear in
the buffer before attempting to decode.
Premature End-of-Stream
One issue arose when experimenting with the runtime scaling of a function outside the
IP filter context. The traced function was set up to call an external function from inside
a loop. The called function took much longer to execute than the in-context instructions
between calls to it. The result was that the decoder progressed through the trace buffer
faster than it could be populated. Eventually, the decoder reached the end of the packet
stream and exited, though the process was still executing.
This issue was mitigated to some extent by detecting whether the target process is still
executing before processing pending events on the buffer. If an end of stream is encoun-
tered and the process is still running, the decoder goes through the following steps to
re-synchronize:
1. Determine the current decoder offset
2. Determine the current block count within that offset
3. Determine the previous synchronization point (PSB packet)
4. Set the decoder’s position to the previous synchronization point
5. Resume decoding
6. Skip processing blocks until the previous offset and block count have been reached
This is expected to produce some additional overhead, since events on the trace buffer still
need to be re-processed in order for the decoder to work its way back from the preceding
synchronization point to its previous position, but only in cases when it would otherwise
have exited prematurely. An edge case remains: When the target program terminates just
before the decoder checks its status but IPT is still writing to the buffer when the decoder
receives the end of stream status, causing packets to be dropped and the decoder to exit.
This appears to be rare enough, however, that testing may still proceed if the test script
is made to detect this edge case and retry the trace if necessary.
IPT Overwrite of Decoder’s Position
The final issue of this type arose during performance analysis, when scaling up the tracing
workload and measuring the performance overhead. At some point, IPT is going to reach

==PAGE:30==
the end of the ring buffer and continue writing from the beginning, overwriting the existing
trace. Eventually, it may catch up with the decoder and overwrite its current position. This
causes the decoder to stop before reaching the end of the buffer. This is a symptom of
the buffer being too small, so the solution was to increase the trace buffer size to support
the workloads being tested.
5.3.2
Populating the Image Cache
Though little modification was required to read the trace buffer from memory, it appeared
that reading the sideband information from memory - in order to match packets to code -
would require much more extensive changes to ptxed. The alternative was to load image
sections directly into the decoder from the binary files. Since ptxed already supported this,
the only required addition was the automatic detection of linked binaries and extraction of
executable section offsets.
The first step was to determine what to load. This was achieved using the ldd command in
Linux to determine the expected number of linked binaries during runtime. The tracer then
allows the target process to step through its initialization, block by block, monitoring the
target’s maps file until the expected number of files has been linked. Once all the required
files have been identified, the offsets of their executable sections must be determined. For
this, the tracer uses a customized version of the load_elf utility included with libipt, which
reads this information directly from ELF files. It then uses ptxed’s built-in functionality to
load the raw executable sections into the cache.
5.3.3
Inconsistent Block Boundaries
Two variations of this issue were observed during development:
1. Blocks ended on non-branching instructions, regardless of IP filter config.
2. Blocks failed to end on branching instructions, depending on IP filter config.
The former was easy to handle, as the decoder only needed to determine whether the
last instruction of the block was a branch, and whether that branch was taken.
The latter, however, posed a bigger challenge. It seemed to be an issue with IPT’s han-
dling of IP filters, as it arose only when using certain filter configurations. Branches leading
outside the filter range were ignored in some cases, leading to blocks extending outside
the trace context, containing branching instructions that should have marked the end of
that block and caused IPT to temporarily disable tracing.
The first approach to this issue was to pass the IP filter range to the decoder and check
whether each block ended within the range. If it did not, the block was truncated at the
last instruction within the filter range. This was an expensive operation, since it required
the decoding of every instruction up to and including the branching instruction, in addition
to the final instruction of the block. While this was an effective workaround, it violated the
design principle of minimizing post-processing, as described in section 4.3.
Further investigation revealed that this issue resulted from a known bug in IPT itself. When
tracing is disabled via a direct, unconditional branch, the resulting TIP packet may not con-
tain the target IP. This bug was originally documented in erratum SKL014 [6th Generation
Intel® Processor Family - Specification Update 2015]. A workaround for this has already
been implemented in libipt. If the IP filter configuration of IPT and the correct CPU identifier
are explicitly included in the decoder configuration, the decoder can use this information
to bind the TIP packet to the correct instruction and determine the end of the block.

5.4
Output
Since this is a proof-of-concept solution, it needs only show that it is able to reconstruct
the control-flow of the target program. For real-time CFA, the output should be processed
further and validated against a previously generated CFG, but that is considered out of
scope for this solution. Instead, the output shall print each taken branch in a human-
readable format for simplicity. The following information is printed:
1. Entry point into the trace context
Since the filtering happens at the IPT level during tracing, the decoder has no infor-
mation about the previous instruction when entering the context from outside. This
shall therefore be classified as a generic >ENTER event.
2. Whether the branch target is inside or outside the trace context
For a branching instruction inside the trace context, the decoder indicates whether
the branch targets an address inside or outside the context with the symbols | (in-
side) or < (outside).
3. Which class of branching instruction was executed (jump, call, return, ...)
The decoder can determine the class of the branching instruction at the end of a
block. This is explicitly indicated in the output.
4. The address of the branching instruction
The instruction pointer address of a given branch instruction is provided to enable
validation of source addresses. The decoder outputs this for every branch instruc-
tion.
5. The target address of the branching instruction
The decoder fetches the target instruction pointer of each branch instruction either
from source code or the corresponding TIP packet, where available, and outputs it.

6
Evaluation
This chapter details an evaluation of the effectiveness and performance of the imple-
mented solution.
6.1
Evaluation Plan
The tests shall be conducted according to the following plan. Test cases are written as
simple C programs to provide minimal test cases covering each test’s requirements. All
test cases shall be compiled using GCC with optimization level set to 0. This allows fine-
grained control over the control-flow of the test case.
The effectiveness shall be evaluated based on the solution’s ability to output the correct
control-flow profile for the target test case. The performance shall be measured against
two scaling profiles: high-complexity and low-complexity. Complexity refers here to the
branch density of a test case. The maximum complexity by this definition would be a
program consisting exclusively of branching instructions, and the minimum would be a
linear program containing no branching instructions aside from the final return.
Each performance metric will relate to two scaling factors: the number of executed in-
structions in the target program and a secondary scaling factor which will reduce the ef-
fective complexity of the trace in different ways for the high-complexity and low-complexity
tests. This secondary scaling factor shall affect the number of instructions executed out-
side the trace context for high-complexity cases, leaving a larger portion of the target’s
runtime untraced, and the number of non-branching instructions between branches for
low-complexity cases, effectively lowering the target’s branch density. Each test, given
a pair of scaling values, shall time the execution of the target program. It will then time
the tracing and decoding of the target with the same parameters. It shall also extract the
time taken for tracing and decoding individually in order to determine the performance
overhead.

6.3
Performance
This section presents the results of the performance tests, run in accordance with the plan
laid out in section 6.1, evaluating the performance overhead of each test case. For both
high- and low-complexity testing, the secondary scaling factor was given five set values
values at exponential increments: 1, 10, 100, 1000 and 10000. This is enough to give a
clear picture of the overall effect they have on the scaling characteristics of this approach.
The primary scaling factor will range from 1000 to 1000000 in increments of 1000. This
strikes a balance between fine-grained test results and speed of testing. The full results
of these tests are available on GitHub1.

Discussion
”What are the performance characteristics of IPT when reconstructing the
control-flow of a program in real-time, how does the control-flow structure and
complexity of the target program affect the performance overhead of such an
application, and are there ways to mitigate this performance overhead without
sacrificing the security of fine-grained analysis?”
These questions were posed in chapter 3, and it is the goal of this chapter to interpret
and discuss the results of the research presented in this thesis in an effort to answer
them. The tracing performance of IPT itself is extremely good. However, this performance
comes at a cost: IPT saves bandwidth by heavily compressing its output. Decoding this
output requires relatively large amounts of computing recourses. As anticipated, this was
identified as the bottleneck for real-time control-flow reconstruction with IPT.
7.1
Capabilities and Limitations
The tests indicate very unfavorable performance characteristics for branch-dense target
programs. This is characterized by the decoder falling continuously farther behind the
current end of the trace stream. There is, however, a threshold of complexity at which the
decoder begins to catch up with the trace. Up to this threshold, the performance overhead
decreases steadily. After it is passed, however, the variance in performance overhead
increases quickly, resulting in less predictable performance. This relates to the basic be-
havior of libipt’s decoder, which is the basis for the solution used here, in that it jumps
to the end of the buffer when it encounters the current end of the trace stream. When
this happens, the decoder needs to be re-synchronized before continuing. This is ex-
pected behavior from a solution intended to read from pre-recorded traces, but becomes
problematic when attempting to read from the buffer in real-time. The re-synchronization
process used here is described in section 5.3.1.
The most promising results showed in the 10000 Iterations Out-of-Context test case of
the high-complexity tests in section 6.3.2, where best-case performance overheads as
low as 1.09% were observed. However, the variance was great and the same dataset
also showed worst-case performance overheads as high as 35.54%, despite the limited
trace context. With some modifications to the decoder’s re-synchronization functionality,
however, the upper boundary of this variance could potentially be lowered. This indicates
that IPT may provide the basis for a powerful real-time CFA tool if applied to a limited
scope. It could be used to monitor security- and/or safety-critical code within a larger ap-
plication without a significant performance impact. It could also be applied to applications
with known control-flow vulnerabilities, in cases when it may not be possible to patch the
vulnerability itself, providing additional security by monitoring the vulnerable code. Fur-
ther research is needed to establish the practical applications and security assurances
provided by such a solution.
This thesis is written in the hope that the data presented herein may help to inform future
development of real-time CFA solutions. This in-depth assessment of the performance
characteristics of real-time control-flow reconstruction using IPT may also help to inform
further research into other processor tracing technologies for the purpose of real-time
CFA.

7.2
Current Challenges
Real-time CFA is a very busy area of research nowadays. In practice, it is mostly applied
to lightweight and security-critical software on embedded systems. The technology still
has ways to go in order to efficiently provide the same level of protection on larger, more
complex systems, such as personal computers. With the rapid emergence of new real-
time CFA solutions, possibilities abound. The current challenges mostly revolve around
the performance of such solutions, calling for more research into the performance char-
acteristics of various methods of execution tracing.
Then there are challenges regarding security assurances. Different implementations pro-
vide different levels of security in different areas. For example, remote CFA has an in-
herent advantage over local CFA in that it ensures a degree of separation between the
attestation software and the potentially-at-risk target program. A CFA solution is as vulner-
able as the platform on which it runs, so a trusted, external attestor is ideal for security.
However, remote CFA tends to be less accessible due to requirements for specialized
hardware. There are many trade-offs and compromises to be made in the design of
widely applicable real-time CFA solutions. It is clear from the results that IPT alone is
not the definitive solution to all of these challenges, but it is a powerful tool in the quest
for more scalable real-time CFA.
There is a real need for efficient execution tracing mechanisms that work on heteroge-
neous devices and provide complete and easy-to-parse information on control-flow tran-
sitions during execution. IPT is an effective and efficient execution tracing tool, and a
valuable addition to the arsenals of security researchers and engineers. Its main draw-
backs are that it (a) only works on specific models of Intel processors and (b) requires
considerable computing resources to fully reconstruct control-flow from its highly com-
pressed output.
7.3
Future Work
The tracing capabilities of IPT remain promising. If the decoding process can be stream-
lined to provide both the security assurances of fine-grained control-flow reconstruction
and the speed of a more course-grained approach, it would provide an ideal basis for a
real-time full-coverage CFA solution. This seems unlikely for IPT alone, however, and
alternatives must be considered.
7.3.1
Efficient Fault Handling
In cases where the target program has extremely low control-flow complexity or the trace
context is very limited, the only obstacle to a negligible performance overhead is the pro-
cess of re-synchronizing the decoder. Theoretically, a more efficient approach should be
possible. For example, the generation frequency of PSB packets is software-configurable,
and could be set much higher, reducing the total number of bytes the decoder must read
when running repeatedly into the end of the trace stream. It is unclear, however, what
effect frequent PSB generation might have on IPT’s tracing performance. The decoder’s
need to re-synchronize might also potentially be avoided altogether. It might be feasible

to maintain a copy of the decoder’s previous state at all times, to be restored immediately
when the decoder reaches the end of the trace stream.
7.3.2
Hybrid Approach
With more efficient fault handling, more diverse applications for this technology would
become available. The performance tests indicate that a hybrid approach may be the most
efficient, balancing performance and security. GRIFFIN already does this to an extent, but
the approach is very performance-oriented, focusing on keeping the decoder close to the
current end of the trace without reaching it [Ge, Cui, and Jaeger 2017]. While this results
in good performance overall, a different approach might provide a more consistent level
of security. For example, fine-grained CFA could be applied to vulnerable or security-
critical functions, while course-grained CFA is applied elsewhere. Since this method offers
control over which code is covered by the more thorough approach, it may provide better
security, depending on which code is covered by the fine-grained approach. This selection
might even be automated, using known static vulnerability detection methods [Cheng et
al. 2019].
Another possibility would be to use a different approach for course-grained CFA, employ-
ing IPT only where fine-grained attestation is required. It is possible that such a hybrid
approach may benefit from using different technologies for different parts of the trace,
such as eBPF for system-call tracing and Intel BTS for course-grained user-space CFA.
There are many ways to approach this problem. Further research is required to make real-
time CFA as ubiquitous in computing systems as other, more common runtime protection
mechanisms, such as ASLR and stack-smashing protection.
7.3.3
Multi-Level Tracing
The concept of multi-level tracing refers to the simultaneous tracing of a program’s exe-
cution at different levels of detail. This is primarily used in performance analysis, enabling
a good overview which can be drilled down to minute details when needed. This execu-
tion tracing strategy could provide an ideal basis for a hybrid approach to real-time CFA.
To this effect, IPT may be utilized as a low-level trace option for fine-grained control-flow
analysis, alongside higher-level execution tracing options. While this will increase the
performance overhead incurred by tracing, it can also reduce the amount of decoding
required and may result in a positive effect on overall performance.
7.3.4
Virtualization
IPT supports tracing on virtualized hardware. This has vast implications for real-time CFA
case since running public-facing servers on virtual machines and containers has become
the industry standard. This also applies to the cloud, where most of the processing hap-
pens in a virtual environment. Assuming a host machine is not running at full capacity, it
is possible that IPT could be used for monitoring control-flow on its managed virtual ma-
chines in real-time with negligible performance overhead. This could also provide greater
security by adding a degree of separation between the CFA mechanism and the target
application as in remote CFA.
7.3.5
Data-Flow Tracing
Data Integrity Attestation (DIA) is another method of runtime security enforcement. The
process is similar to CFA, but revolves around Data-Flow-Diagrams (DFD) rather than
CFGs for attestation. DFDs describe the flow of data between components such as vari-
ables during execution. DIA is usually performed statically, relying only on source code
analysis. For a program that has been verified by static DIA, it can therefore be assumed
that the data inherit the integrity of the code which accesses it. This indicates that CFA

could be useful in attesting the continued runtime data-integrity of statically attested pro-
grams. The best-case high-complexity scenario identified in the test results shows that
IPT performs well under those conditions. One example of such an application is DIAT, a
DIA solution for autonomous, collaborative distributed systems. It leverages control-flow
tracing to monitor execution within and interactions between specific isolated software
modules on each device. The trace targets are exclusively those modules which have
direct access to the data. This effectively detects control-flow-based attacks on data in-

IPT based on the performance characteristics presented in this thesis [Abera, Bahmani,

8
Conclusion
This thesis seeks to assess the viability of IPT as a tool for real-time CFA. The results of
a series of performance tests, using a minimal IPT-based control-flow extraction solution,
indicate that it scales very poorly with the control-flow complexity of the target program.
IPT produces very little performance overhead from tracing alone, but its output is so
compressed that reconstructing control-flow from the trace is orders of magnitude more
resource-intensive. No mitigation strategy was identified which did not compromise se-
curity in some way. However, the results do show promise in cases where IPT is applied
to a limited trace context, despite high control-flow complexity within that context. This
indicates that IPT may be used to great effect in combination with other execution tracing
technologies in real-time CFA applications.
Based on the current state of research into real-time CFA, it is clear that further research
would benefit from more data on the performance characteristics of the available tools.
One of the most promising tools available is IPT, a relatively recent hardware-based ex-
ecution tracing feature of Intel processors. Due to its extremely low tracing performance
overhead and the level of detail provided in its output, it seems an ideal subject for such
analysis.
To achieve the results presented in this thesis, a proof-of-concept solution was created,
based on IPT via Linux perf tools and Intel’s open-source decoder library libipt. This solu-
tion executes a specified program with IPT enabled and parses the trace buffer in parallel,
extracting the control-flow of the target program during runtime. This was the subject of
the evaluation, in which the performance overhead was measured based on scaling fac-
tors affecting the overall runtime and control-flow complexity of the target program, as
well as the relative scope of the trace context. The results were presented, analyzed and
interpreted in an effort to identify the capabilities and limitations of IPT in the context of
real-time monitoring.
It is clear from the results that IPT alone, while powerful, does not lend itself to full control-
flow reconstruction in real-time without incurring a substantial performance overhead.
However, as shown by previous work, this overhead can be lowered significantly with
a hybrid approach, whereby only selected parts of the code are subjected to fine-grained
attestation. This entails a compromise between security and performance which can be
achieved in many different ways. This thesis recommends identifying potentially vulner-
able regions of code and applying fine-grained IPT-based CFA to them, covering the rest
with a faster and less thorough attestation model. This can be done entirely with IPT,
changing only how the decoder interprets the trace stream. However, it might prove more
beneficial to use other tools, such as eBPF and Intel BTS, for the course-grained attesta-
tion. In addition to these avenues for future research, questions remain about the effect of
virtualization on the effectiveness and performance of IPT-based real-time CFA, as well
as its utility in other security applications such as data integrity attestation.
This thesis will hopefully help to inspire and inform further research into real-time CFA
by providing data on the relevant performance characteristics of the powerful execution
tracing tool that is IPT. More data is a prerequisite for better knowledge and more per-
formance analyses of available tools will help to pave the way towards faster and more
secure real-time CFA solutions.
