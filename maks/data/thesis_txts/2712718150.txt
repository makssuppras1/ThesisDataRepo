Abstract
Filters are a core component for any modern database that exists nowadays. In this the-
sis we will focus on the Succinct Range Filter (SuRF) and its underlying Fast Succinct
Trie (FST). We analyze the information-theoretic and complexity bounds of the FST and
compare it with the currently being used solutions that exists out there. Furthermore, we
implemented SuRF and FST in C++ focusing on high performance on both the memory
layout and compute. Finally, our implementation is tested against state of the art indexes
and filters, additionally we benchmark the improvements of our solution [1] with the im-
plementation given by the authors of SuRF [2].

Introduction
The Succinct Range Filter (SuRF), found in 2018 by Zhang, Lim, Leis, Andersen, Kamin-
sky and Keeton [3], is a fast and compact datastructure that supports approximate mem-
bership tests and approximate range emptiness. Our goal in this thesis is to overview
the work that has been done in both SuRF and FST and extend it by providing a C++
implementation of both datastructures, implementation details on both construction and
queries on SuRF and the underlying FST, performance improvement extensions, and
benchmarks to empirically prove such improvements.
1.1
Problem
Approximate Problem
Maintain a data structure that for a given set S ⊆Σ∗and an ϵ
that depends on S, where Σ = [1, σ], that solves the approximate test:
• approximateMember(x): if x ∈S return ”yes” otherwise if x ̸∈S return ”no” with
probability ≥ϵ
• approximateRangeEmptiness(a, b): if [a, b] ∩S = ∅return ”yes” otherwise if [a, b] ∩S ̸=
∅return no with probability ≥ϵ
Non Approximate Problem
Maintain a data structure that for a given set S ⊆Σ∗, solves
the non approximate tests:
• Member(x): return ”yes” if x ∈S and ”no” otherwise
• RangeEmptiness(a, b): return ”yes” if [a, b] ∩S = ∅, ”no” otherwise
1.2
Why succinct datastructures?
We live in an era where technology is advancing at an unprecedented rate, and so does
data. Facebook had 2.936 billion monthly active users in April 2022 [4], users generat-
ing data directly such as text, pictures, videos, music, voice or indirectly such as, likes,
friendships, clicks, navigation patterns, searches. If only we assumed each user gener-
ated around 1 megabyte of data per day then Facebook would be generating around 3
petabytes of daily data, and we are probably underestimating by a big margin.
Over the years, there have been major technological improvements, disk sizes are in-
creasing while latency is decreasing, same can be said for main memory, but these tech-
nological advancements cannot keep up with the steep growth of data that is getting
collected nowadays, this concern is raising importance on memory layouts and access
patterns. This is extremely important, knowing that an HDD seek takes 10 ms compared
to an L1 cache reference which takes about 0.5 ns [5], we are talking about a 2×107 times
difference, this is why it is so important to operate on data that stays in main memory and
avoid random access patterns that lead to unpredictable behaviour, it gets worse when
working with IoT devices as these work on memory and compute constraints.
It’s not all about how we access data, we also have to think about how efficiently we
store the data. It’s not the same to store a tree using pointers, O(n log n) space, than
a balanced paranthesis representation, 2n space. Unfortunately, having more compact
data does come with trade-offs, traversal requires more steps compared to the pointer

1.3
Succinctness and DBMS
Popular embedded databases like RocksDB [6], LevelDB [7] or CassandraDB [8] use the
log-structured merge tree (LSMT) [9], a tree based data structure characterized by it’s
high insertion throughput, widely used in main-memory database management systems
(DBMSs) 1, data stored resides in immutable SSTables across all levels of the tree. When
accessing data, multiple SSTables might be loaded, this would incur multiple costly I/Os,
for this reason, popular databases use mechanisms to avoid expensive I/Os, the most
prominent ones are bloom filters.
Bloom filters are small and compact data structures that solve the approximate mem-
bership test, this tests guarantees that if a key is contained in the SSTable then the filter is
always going to return true, but might return true for a key that is not in the SSTable, this is
what is called ”false positive”, on practice, bloom filters greatly reduce the number of I/Os
done while maintaining a low memory foot print (small enough to be kept in main-memory).
Bloom filters solve the approximate membership test, a test that is really handy for point-
queries, but once we get to more complex queries like range-queries bloom filters have
no use, main issue with bloom filters is their lack of flexibility, they do not preserve order-
ing of keys, prefix bloom filters are used to give more flexibility, but still, it is not general
purpose, as they solve queries on fixed length prefixes defined at construction time. The
alternative is to rely on fast pointer based indexing structures like B-trees that can both
solve range and point queries, however pointers take a large chunk main-memory [10]
which is crucial as we keep adding more data in memory, for this reason modern DBMSs
are opting for compact trie data structures, the idea behind this is that a well designed trie
outperforms indexed based data-structures such as B+trees [11].

Notation

computation models, strings, sequences and intervals. We will be using a similar notation

2.1
Asymptotic notation
When analyzing an algorithms time and space usage, we often estimate time and space
as a function of the input size. The idea behind O-notation is to define the growth of such
functions, considering sufficiently large values and ignoring constant values. Asymptotic
notation is fundamental when talking about time and space of algorithms.
Consider f(n) to be the cost function of an algorithm, n being the size of the input, it is
said that f(n) belongs to O(g(n)) (f(n) ∈O(g(n))), if there exist constants c > 0 and n0
such that f(n) ≤c · g(n). O(g(n)) is thus the set of functions where f(n) ≤c · g(n) for
any n ≥n0. The cost function of an algorithm f(n) ∈O(1) is used to denote that the
cost is independent of the input size, in other words, such algorithm is solved in constant-
time. We will also consider f(n) ∈Ω(g(n)), Ω(g(n)) being the set of functions where
f(n) ≥c · g(n) for any n ≥n0, and f(n) ∈Θ(g(n)), where Θ(g(n)) = Ω(g(n)) ∩O(g(n)).
It is common writing = instead of ∈or ⊆, thus expression such as f(n) = O(g(n)) denote
f(n) ∈O(g(n)) and Θ(f(n)) = O(g(n)) denote Θ(f(n)) ⊆O(g(n)).
2.2
Word RAM model
We adopt the model of a random access machine, in such model memory is divided into
w = log(n + 1) (or so) bit words, n being the size of the input. Basic arithmetic and logic
operations on words are done in O(1) time, and input addresses fit into a single word. This
model is more specifically called the trans-dichotomous model as stated by Fredman and
Willard [13], the dichotomy between the machine model and the problem size has to be
reasonable, thus not considering possible hidden parallelism by doing operations on ”long
words”, resulting in a more fair model as computers have a fixed word length.
2.3
Strings, arrays and intervals
We define the interval [a, b] to be the set {a, a + 1, . . . , b}, the array A[1, n] represents the
sequence of n elements, A[1], A[2], ..., A[n]. Strings are simply arrays whose elements
are drawn from a finite alphabet Σ, for example Σ = [1, σ] would be the alphabet defined
by the set {1, 2, . . . , σ} assuming σ is an integer, for particular cases some strings will need
a terminator $, we will define alphabets with such terminator as Σ$ = Σ ∪{$} . Given a
string S[1, n] we know it is a string of size n but we will also reference it’s size as |S|. We
consider S[i, j] to be the sub-string S[i], S[i + 1], . . . , S[j] , in the case where i > j, S[i, j]
denotes the empty string ε.

Bitvector
4.1
Introduction
The bitvector (a.k.a. bitstring, bitmap, bitarray) is a bit array B[1, n] that supports the
following operations
• read(B, i) : returns the bit B[i], for all 1 ≤i ≤n
• Rankv(B, i) : returns the number of occurrences of the bit v ∈{0, 1} in B[1, i], for all
1 ≤i ≤n, in case i = 0 then Rankv(B, i) = 0.
• selectv(B, i) : returns the position of the ith occurring bit with value v for any 0 <=
i < n.
We will be using static bitvectors heavily throughout the report, for this exact reason it
is crucial coming up with a solution that is both fast and compact when computing the
previously mentioned operations. Bitvectors are going to be of utmost importance for
achieving succinctness.
In some use cases, bitvectors end up being heavily sparse, either having more ones than
zeros, more zeros than ones or having clusters of ones and zeros. For this cases it is
beneficial to compress the bitvector, reducing it’s size. Main drawbacks of compression is
that when needing access, data has to be decompressed and this affects the performance,
for this reason we will avoid compression and instead add additional space to improve
performance of the previously mentioned operations.
4.2
Structure
One could simply store a bit array as a vector of booleans, boolean primitives normally
use 1 byte (8 bits) of space, meaning the this array would take 8n + o(n) bits of space. It
is possible to do better by using less bits per boolean.
A more efficient approach is to store the bit array B[1, n] as a contiguous array of words
W[1, ⌈n/w⌉], a compact layout compared to the standard 8 bits boolean using only n bits.
Main drawbacks compared with previous solution is that now to read a specific bit we have
to first find the respective word and then do bit manipulation to find the corresponding bit.
As n grows this solution becomes faster as this packed fits better in main-memory.
4.3
Read
read(B, i) using only W can be done by accessing the corresponding word to the bit i
and then bit shifting and masking the corresponding bit.

4.5
Rank
For simplicity’s sake, we define Rank(B, i) to be Rank1(B, i), this is due to the fact that
Rank1(B, i) = i −Rank0(B, i), thus Rank1 solution is analogous to the one for Rank0.
4.5.1
O(1) time and O(n log n) additional bits
A trivial solution for Rank would be to construct an look-up table R[1, n] such that R[i] =
Rank(B, i), for all 1 ≤i ≤n. With this solution we compute Rank in O(1) time at the
expense of using n · w bits, which is way more than the n bits needed for the bit array B.

4.5.3

5
Trees
Trees are known to be one of the pillars in Computer Science when it comes to data
structures. They are commonly used in search trees, heaps, abstract syntax trees, file
systems, space partitioning and many more. Frequently we are used to seeing their stor-
age represented with child pointers in addition to other information.
However such pointer-based structures are very costly. Suffix trees are one of those
cases, they are really handy and known to be really efficient for analysing sequences
in the genome, but also use a lot of space when using a pointer-based tree data struc-
ture. To store human DNA that is around 3.3 billion bases, it requires 800 megabytes
using 2 bits per base {A, C, G, T}, but it’s suffix tree requires 30 gigabytes as each base
takes 10 bytes or more. Here is were succinct data structures come in play, the suffix tree
of a human genome can be stored in less than 4 gigabytes using succinct data structures.
As we have previously seen, we need at least 2n −Θ(log n) bits of space to represent an
ordinal tree, one could easily represent a ordinal trees in 2n bits of space by representing
each node with open and closing parenthesis open being one and closing being zero.
Succinct binary trees are well studied and best results achieved this far come from the
work of Munro and Raman [27] [28], they achieved constant time operations on parent,
left child, right child while using only 2n + o(n) bits of space. They have event shown
same space on succinct dynamic binary trees.
The same cannot be said for higher degree trees. There are two forms of trees, ordi-
nal trees and cardinal trees. Ordinal trees have nodes of arbitrary degree while cardinal
trees have a fixed degree k. A clear example of ordinal tree would be an XML file where
every node has a non fixed amount of children, on the other hand a trie would be a case
of cardinal tree, where each node has children ranging from 1 to σ. There has been plenty
of work on succinct ordinal and cardinal trees but when it comes to performance, they are
magnitudes slower than their pointer-based competitors.
In this chapter we will go over LOUDS, a succinct tree representation that plays a big
role as a the foundation of SuRF.
5.1
LOUDS
From the work of Jacobson [29], the Level-Ordered Unary Degree Sequence (LOUDS) is
a tree encoding where each node of degree d is encoded as a string of d 1 s followed by
a 0. Assuming we have a tree T with n nodes, such encoding can be represented using
a bitvector B[0, 2n−1] as there are n 0 s and n−1 1 s, each node encoding being written
to B in a level-ordered traversal. For example, in 5.2 the encoding of the node 0 being
1110 and the encoding of the node 2 being 0.
LOUDS is considered a succinct representation as its space representation is close to
the theoretic lower bound of 2n −Θ(log n) bits, as seen in 3.2
LOUDS ends up being a bitvector representation B, supporting Rank and Select opera-
tions and B is uncompressed as it contains as many 1 s as 0 s. We are able to traverse
the LOUDS tree using the following operations:

6
Fast Succinct Trie
In this chapter we will go over the Fast Succinct Trie (FST) [3], a trie-based index structure
and the core datasctructure of the Succinct Range Filter (SuRF). FST is based on LOUDS,
one of the fastests succinct tree representations, and adds preformance improvements
achievent results comparable to state of the art pointer-based indexes (B+Tree [31], ART
[32], . . . ). FST is designed to be static, thus not supporting insertion or deletion.
When traversing a trie multiple times, upper levels tend to be accessed more often than
lower levels, FST is designed having this in mind, it combines LOUDS-Dense, an encoding
scheme for cardinal trees using fixed size bitmaps for quick look-ups at the expense of
using more space. And LOUDS-Sparse is an encoding scheme for ordinal trees that is
extremely space efficient at the expense of slower look-ups.
By merging both LOUDS-Dense and LOUDS-Sparse we get FST, a trie data structure that,
while staying succinct, is an order of magnitude faster than current succinct data structures
and its performance is comparable to current state of the art pointer-based indexes.
We will be assuming that the node of the trie has a fixed fanout of 256 (radix 8) and keys
are of fixed length.
6.1
LOUDS-Dense
LOUDS-Dense core purpose is to use bitmaps for quicker access to the branching nodes,
using bitmaps of 256 size we know in one access if a label does branch from a node
without the need to traverse all branching labels of that node. LOUDS-Dense Structure is
as following.
• D-Labels: a bitmap that stores the branching labels for each node, if a node n has a
branching label l, where 0 ≤l ≤255, then D-Labels[l + n · 256] = 1, example.
• D-HasChild: a bitmap that stores the branches that do not terminate, if a node n
does not terminate in label l,where 0 ≤l ≤255, then D-HasChild[l + n · 256] = 1,
(example)
• D-IsPrefixKey: a bitvector that stores if a node’s prefix is a valid key, if a node n
contains a valid prefix key then D-IsPrefixKey[n] = 1, (example)
• D-Values: an array of fixed size elements, mapping leafs to values, stored in a level
order, normally used to store pointers to the keys.
The numbering of the nodes must follow a level ordering for it to work.
6.1.1
Tree Traversal
Similar to LOUDS, tree navigation is done with Rank and Select operations, we denote
Rank and Select operations for a given bitarray B as Rank(B, i) and Select(B, j). For a
given bit position p we can do the following.
D-ChildNodePos(p) :
Assuming D-HasChild[p] is set, p has a child and we can retrieve
the child node position. By computing the rank r = Rank1(D-HasChild, p) we get the num-
ber of nodes (r+1) that are before position p, as each node has fixed size 256, the starting
bit position of the child node is r · 256.

S-Traverse(key, pos)
Starting from position pos to S-LastNodePos(pos), we search
for the matching label in S-Labels, if a matching label is found in position p then
we check if such position has a child in S-HasChild, if it does then we traverse
downwards by computing the first child position S-ChildNodePos(p) and the last one
S-LastNodePos(S-ChildNodePos(p)), we repeat the search for these positions this un-
til the label is not found or there is no child node. For the case where the label could not
be found we return null. For the case where the label was found but it has no children,
i.e. is a leaf node, we return the position of such leaf and the level it was found.
S-LowerBound(key, pos, flag)
Starting from pos, we can retrieve the node position that
maps to the closest key k in lexicographic order greater than key by traversing the tree
nodes that share label with key, once we have a non matching label we traverse to the
leftmost leaf from the current node that holds a key k greater than key , if no such key
exists then we jump back to the last node with non matching label and traverse to the
leftmost leaf from there, similarly returning the position, level and flag set to true. If such
key does not exists, position null is returned.
S-KeyFromPos(pos)
retrieving the key from the leaf position pos is done by traversing up-
wards the tree and each time storing the label of the current node position until S-RootNodePos
has been reached.

6.3
FST
The Fast Succinct Trie (FST), merges both the speed of LOUDS-Dense and the space
efficiency of LOUDS-Sparse by merging both tries, LOUDS-Dense taking care of the upper
levels of the trie and LOUDS-Sparse the lower levels.
It is crucial to maintain a balance between both levels, otherwise FST is going to be
fast but not space efficient or the other way around.
For this reason a ratio R be-
tween LOUDS-Dense and LOUDS-Sparse is defined, assuming that the trie has H levels,
LOUDS-Dense-Size(l) and LOUDS-Sparse-Size(l), for 0 ≤l ≤H, represent the size of
LOUDS-Dense and LOUDS-Sparse encoding respectively, l being the largets cutoff level
such that LOUDS-Dense-Size(l) · R ≤LOUDS-Sparse-Size(l). Reducing R leads to more
dense levels and faster queries at the cost of more space, by default R = 64.

6.3.2
Space
For a given trie T of n nodes:
• LOUDS-Dense is built from D-Labels+D-HasChild+D-IsPrefixKey using 256n+256n+
ni ≤513n bits of space, where ni is the number of internal nodes.
• LOUDS-Sparse is built from S-Labels+S-HasChild+S-LOUDS using 8n+n+n = 10n
bits of space.
FST ends up using 513n/R + 10n = 10.16n bits of space, where R = 64 is the ratio
between LOUDS-Dense and LOUDS-Sparse, if we support Rank and Select operations in
O(1), we need an additional o(n) bits of space, assuming uniformity and approximately
same number of 1-s and 0-s.

In this subsection we talk about the added extensions that make FST differ from the orig-

Removal of D-IsPrefixKey
when a key is a prefix of another one in LOUDS-Dense we set
a bit in D-IsPrefixKey in the prefix key position, on the other hand when a key is a prefix of
another one in LOUDS-Sparse we have a $ label with value 0xff instead, it is possible to
apply the same idea to LOUDS-Dense, instead of setting a bit in D-IsPrefixKey, we set a bit
in D-Labels mapping to $. By doing this we remove the need of having D-IsPrefixKey and
makes the implementation more intuitive as it uses the same idea as in LOUDS-Sparse to
represent prefix keys.
Null terminated strings
null terminated strings or C strings are strings of n characters
and a null terminator 0x00. By having null terminated strings we remove the need to have
the special character $ for cases were a key is a prefix of another one and makes it more
reasonable as as the null terminator has value 0x00 instead of 0xff as it was for the case
of $, which made linear search in a node from LOUDS-Sparse more complex as first label
could have value 0xff breaking the lexicographic order of the labels in that node. With
the null terminator, the ordering is not broken and it makes it more simple linearly search
for a label and traversing nodes from top to bottom, as it follows the lexicographic order
of the set of keys stored.

7.1
Introduction
In this section we are going to go over the Succinct Range Filter (SuRF), a trie based
filter datastructure from the work of [3] that supports both ApproximateMember and
ApproximateRangeEmptiness queries built on top of FST.
7.2
SuRF-Base
Filters are extremelly space efficient datastructures, FST are also space efficient but not
as efficient as prevailing filters, this is expected as we are storing a trie-based index that
has no false positives, by prunning the trie the filter loses accuracy and gains memory
efficiency, a minimal solution is SuRF-Base, an FST storing the minimum length prefix
such that we can uniquely identify each key.
Filters accuracy heavily relies on both the distribution of the stored keys and the distribu-
tion of the queried keys, depending on the keys distribution, there can be more or less
pruning, so worst case we could get little to no pruning. Accuracy of a filter is measured by
the False Positive Rate (FPR), defined as
FP
FP+TN , FP being the number of false positives
and TN being the number of true negatives. Figure 7.1 shows an example of a pruned trie
from a set of keys S, both keys 'faster' and 'fasten' share the prefix 'fast' in the trie,
meaning that in SuRF-Base, 'faster' would be a false positive while 'fasten' would
be a true positive. Assuming that the distribution of the inserted keys is independent of
the queried ones, SuRF-Base FPR is bounded by N · 256−Hmin, N being the number of
keys and Hmin being the minimum length height leaf, moreover for a given leaf node L at
height H such query has 256−H probability of reaching L. In practice inserted keys and
queried keys are highly correlated, for this reason it is necessary to analyse empirically
the performance of this filter.
7.3
Tunning False Positive Rate
FST contains an array of values V alues split between D-Values and S-Values, arrays of
fixed size values that keys map to, normally pointers pointing to keys are stored as values
but such array can hold other purposes. Reducing the False Positive Rate (FPR) is one
of them, by storing additional information related to each key it is possible to reduce the
FPR:
• SuRF-Hash: for each pruned suffix K, store in V alues the n least significant bits of
H(K), being H(·) a hash function. Having n bits from H(K), the collision probability
is 2−n, so for the case where n = 6, we get a 1.5% FPR.
• SuRF-Real: for each pruned suffix K, store in V alues the n most significant bits of
K.
• SuRF-Mixed: for each pruned suffix K, store in V alues the ns most significant bits of
K concatenated with the nh least significant bits of H(K), being H a hash function.

8.1
Introduction
In this chapter we will cover the implementations made for the FST and SuRF, additionally

8.2
C++
When dealing with succinct datastructures we want perfect control of memory, knowing
when value is stored, moved or deleted. For this reason we discarded any garbage col-
lected language, it is known that most garbage collected languages have phases like
compaction that avoid programs having memory fragmentation leading to more efficient
and less memory usage but as we will be dealing with continuous bitarrays whose lifetime
corresponds to the one of the program, we have major benefits by having the freedom to
manage our resources.
Another key factor is having access to hardware specific instructions, instructions such
as PopCount, TZCNT and PDEP are necessary to provide fast Rank and Select operations.
Has available build automation, packaging and installation software, as we will benchmark
with other state-of-the-art index structures and filters, it is mandatory that we can build and
link our program with other libraries in an automated manner.
Lastly, the language has to have a reputation for producing high performance code as one
of the main resons of this thesis is to provide a fast implementation of both FST and SuRF
with a low memory footprint. Language has to provide basic optimizations (loop unrolling,
loop vectorization , . . . ) to produce efficient executable code while staying readable.
For this reason we have decided to go with C++, the build system CMake, and the bench-
mar and testing framewords from Google.

its consequences.

Rank’s implementation uses superblocks R[0, ⌊n/s⌋] of size s = 512, where R[i] =

a single cacheline is fast, we found unnecessary to add the blocks array R′ in Rank. We
end up needing an additional 0.125n bits of space.

sequentially and every ⌈s/w⌉words insert the accumulated rank to R, as following:
auto construct_R(span<uint64_t const> W, span<uint64_t> R){

for (size_t j = i*s; j < (i+1)*s; ++j){

}
}

construct
S is built by traversing W from the start and storing the locations of the set
bits that are multiple of s. S is constructed as following.
auto nthset(uint64_t w, size_t i){
// returns the position of the ith set bit in w
return tzcnt(pdep(1ULL << i, w));
}
auto construct_S(span<uint64_t const> W, span<uint64_t> S) -> uint64_t {
assert (W.size() == n/64 && S.size() == m/512);
uint64_t rank = 0;
uint64_t next_rank = 1;
size_t j = 0;
for(size_t i = 0; i < W.size(); ++i){
uint64_t pop = popcount(W[i]);
if (rank + pop >= next_rank){
S[j++] = i*64 + nthset(W[i], next_rank - rank);
next_rank += 512;
}
rank += pop;
}
}
select
Select’s main purpose is computing child node’s position S-ChildNodePos in
LOUDS-Sparse and worst case the distance between two consecutive ones in S-LOUDS
is going to be σ, as σ is small, our implementation traverses linearly W thus Select can
take up to O(sσ/w) = O(σ log n) time considering PopCount takes O(1) time. The imple-
mentation looks as following:
auto select(span<uint64_t const> W,
span<uint64_t const> S,
uint64_t rank) -> uint64_t {
assert (W.size() == n/64 && S.size() == m/512);
rank -= 1;
size_t pos = S[rank/512];
rank = rank % 512;
if (rank == 0) return pos;
size_t offset = (pos&63) + 1;
size_t i = pos/64;
// remove already counted bits
uint64_t w = (W[i] >> offset) << offset;
uint64_t pop;
while((pop = popcount(w)) < rank){
rank -= pop;
w = W[++i];
}
return i*64 + nthset(w, rank-1);
}

8.4
SuRF & FST
Considering the pruning extension on FST, the difference between FST and SuRF only
relies on what is stored in V alues, for the case of FST, pointers to the pruned suffixes are
stored and for the case of SuRF fixed size hashes from those pruned suffixes are stored.
Implementation whise, there is not much difference to the point where core construction
of both data structures is shared. For this reason in this secition we will be deliberately
covering both data structures at the same time.
The strings are considered to be null terminated but as they are pruned the null terminator
is only going to be stored in the case there is a prefix key, a key that is the prefix of another
one, additionally LOUDS-Dense does not contain D-IsPrefixKey and instead null terminator
is stored in D-Labels, we will assume that it is not possible to branch with a null terminator,
in other words , the null terminator cannot be in D-HasChild. Thus we end up using all the
extensions related to tree structure.
When constructing both SuRF and FST, the input is considered to be an array of strings
Ssort[0, N] sorted in lexicographic order, this comes from the assumption that we are work-
ing in a Log Structured Merge Tree (LSMT) [9] and the keys stored come from a Sorted
Strings Tabel (SSTable), a core component of the LSMT. In ideal circumstances we would
have a tree that we could traverse in a level ordered manner to construct FST, unfortu-
natelly this is not the case.
One could in fact traverse the strings in a level ordered manner, by first traversing the first
character on all strings, then the second, and so on while keeping track of which node
they are at. This implementation makes it trivial to construct both tries as we are inserting
the elements in a level ordered manner. Unfortunatelly simple does not mean fast, as we
are accessing each string as many times as the maximum depth of the trie.
A more efficient approach is to access every string only once, and keep track of the con-
structed trie per level, for example when building LOUDS-Sparse, instead of keeping track
of a single bit vector S-LOUDS you would have multiple bitvectors, one per each level
and once all strings have been traversing traversed, bitvectors are merged in a level or-
der manner into a single bitvector, leading to the resulting LOUDS-Sparse. Such approach
hash drawbacks, as it has to keep track of all bitvectors per level and not knowing the size
of each level leaves to vector reallocation which might have a considerate cost. A way
to avoid reallocation cost would be to traverse K twice and first time compute the size of
each of the bitvectors per level and allocate their full extent before constructing them in
the second sweep. As K becomes larger, traversing K twice becomes more costly, for
this reason we decided going for a single sweep approach as K tends to be large. We
will be calling this supporting data structure Builder.
8.4.1
Builder
The Builder is a supporting data structure that keeps track of a trie and is constructed from
a sorted set of strings, the necessary information to keep track is:
• B-Labels is an array of byte-arrays each containing the branching labels for each
level of the trie.
• B-HasChild is an array of bitvectors containing a bit set for each label in B-Labels if
it’s respective label has a child node.
• B-Louds is an array of bitvectors containing a bit set if, for each label in B-Labels, the
label is first in a node.

• B-Suffixes is an array of arrays, each nested array stored in a level order corresponds

Our benchmarks have to measure extremely fast datastructures, for this reason precise
timings are really important, we only want to measure what we are really interested in.
When implementing a microbenchmark one must assure that the language does not add
overhead that influences the results. For this reason it is usually recommended to use
existing tools which make it easier to produce meaningful results. For this experiment
we will be using Google Benchmark [34], a benchmarking suite in C++ that provides all
the tools to make statistically correct benchmarks by tuning the benchmark iterations and
provide tools to avoid compiler optimizations that could lead to wrong results.

Conclusion
This report builds up from the work of [3] and their respective SuRF and underlying FST
data structures. Going through their work their work, analysing their decisions and ex-
tending from them. We have been abled to provide a C++ implementation [1] from scatch
for both SuRF and FST that outperforms their original implementation [2]. This imple-
mentation has been benchmark with state of the art pointer-based indexes , filters and
succinct indexes. It has been shown that FST is superior to current succinct indexes both
in performance and space, additionally SuRF operates in the same order of magnitude as
pointer-based indexes while using orders of magnutide less space than their respective
pointer-based alternatives. We have been abled to see that some of the claims from the
original empirically tested in YCSB also extend to other workloads that differ consider-
ably, but also found workloads where this claims are not satisfied. From the experiments,
we believe that SuRF and FST are great candidates as a replacement for filters and in-
dexes by using one order of magnitude less space at the extent of having slightly less
performance.

Problem Description
Approximate Problem
Maintain a data structure that for a given set S ⊆Σ∗and an ϵ
that depends on S, where Σ = [1, σ], that solves the approximate test:
• approximateMember(x): if x ∈S return ”yes” otherwise if x ̸∈S return ”no” with
probability ≥ϵ
• approximateRangeEmptiness(a, b):
if [a, b] ∩S
=
∅return ”yes” otherwise if
[a, b] ∩S ̸= ∅return no with probability ≥ϵ
Non Approximate Problem
Maintain a data structure that for a given set S ⊆Σ∗, solves
the non approximate tests:
• Member(x): return ”yes” if x ∈S and ”no” otherwise
• RangeEmptiness(a, b): return ”yes” if [a, b] ∩S = ∅, ”no” otherwise
Two problems are targeted as the intent of this report is to analyze the Suc-
cinct
Range
Filter
(SuRF),
a
filter
that
solves
both
approximateMember
and
approximateRangeEmptiness, and it’s underlying index, the Fast Succinct Trie (FST), an
index that solves both Member and RangeEmptiness.
Previous work
Indexes being data structures that solve both point queries (Member) and
range queries (RangeEmptiness), are widely used in the world of database management
systems (DBMS), similarly filters are widely used in main-memory DBMS, more specifi-
cally they are used in log-structured merge trees (LSMT), a tree-based data structure that
relies on the speed given by main-memory, the core idea is to eliminate disk latency by
loading the data that we want to query into main-memory, thus eliminating any possible
I/O when solving the query. This data stored resides in immutable SSTables across all
Succinct Range Filters
55
==PAGE:62==
levels of the tree. Problem is, that accessing data might incur multiple costly I/Os, for this
reason filters are used to avoid expensive I/Os.
There is a wide variety of succinct filters out there, such as Cuckoo filters [40], Gaussian
filters [41], Quotient filters [42], Morton filters [43] but for data-stores the most widely used
are the Bloom filters [44] and their extensions [45] [46] [47], Bloom filters are known to be
both compact and fast but they are not made for range based queries.
A possible way to deal with range based queries is to use pointer-based uncompressed
trees like B+Trees [31], ART[48] or C-ART [49] but still these are not as fast nor as compact
as bloom filters, mainly because of their pointer based nature as they use O(n log n) bits
of space. Data stores like RocksDB solves this problem by using prefix bloom filters but
they lack flexibility, have to be tuned by the user and have an additional space cost.
Another alternative is the use of succicnt trees, there has been a lot of research during
the last 20 years, such as the work from Munro and Raman [27][28], where they showed
binary trees using 2n + o(n) bits of space while having constant time operations, unfor-
tunately many of the existing implementations are still one order of magnitude slower
than their relatives pointer-based uncompressed trees. Jacobson came up with Level-
Ordered Unary Degree Sequence (LOUDS) [29], a succinct tree representation that uses
rank and select operations for tree traversal, the operations were originally shown to be
run in O(log n) but with modern rank and select implementations it achieves constant time
by using additional space on a look-up table. From this work came SuRF [3] a succinct
range filter that is on pair with the speed and space of nowadays bloom filters with the
additional flexibility of supporting fast range based queries.
Aim of project
The goal of this thesis is to further analyse, implement and extend SuRF
and the underlying FST. From here we will empirically test our implementation and the
possible extensions, additionally it will be compared with state of the art pointer-based
indexes, succinct indexes, filters and the original implementation of SuRF [2].
