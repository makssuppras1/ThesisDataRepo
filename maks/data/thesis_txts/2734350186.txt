Abstract
Evolutionary algorithms is a class of algorithms inspired by biological evolution. As
with life, these algorithms generally have a stochastic element making it diﬃcult to
prove tight complexity bounds compared with deterministic algorithms. Most complex-
ity research has therefore historically been focused on simpliﬁed evolutionary algorithms
where the population is 1. This creates an inherent discrepancy between our theoretical
understanding of these algorithms and the algorithms that are used in practice, as most
practical applications of these algorithms uses a population of more than 1.
This project introduces a software framework that will allow researchers to empirically
study population based evolutionary algorithms to help them prove the computational
complexities theoretically.
We will also use the framework to give empirically based
estimates of tight complexity bounds for 2 speciﬁc algorithms on 3 di↵erent benchmark
problems.

Introduction
Evolutionary algorithms (EAs) are a class of algorithms inspired by the process of bio-
logical evolution. At their core, EAs progressively improves a ﬁtness function by evolving
individuals randomly and only letting the ﬁttest survive, much akin survival of the ﬁttest
in nature. This ﬁtness function is often treated as a black box, which means there is no
information about the speciﬁc problem that is being solved. The algorithm only knows
whether it is moving closer to the goal or not.
The individuals being evolved can be thought of as points in a search space, where the
goal is to ﬁnd a point/individual with the highest ﬁtness. One might be inclined to think
that since these points are chosen somewhat randomly, evolutionary algorithms might
perform poorly compared to other algorithms. Surprisingly however, it has been shown
that even very simple evolutionary algorithms can optimize all linear functions in expected
⇥(n log n) time [7]. This is in stark contrast to the worst case optimization time of ⇥(2n)
for an exhaustive search.
While the power of evolutionary algorithms is no doubt impressive, analyzing the expected
optimization time of these algorithms is by no means a trivial task. This is especially true
if the algorithm or problem is complex. Thus, in order to gain insights into how more
advanced evolutionary algorithms perform, researchers usually analyze simpler algorithms
on various benchmark problems in the hope that these results can be extrapolated to more
complex scenarios.
As the individuals can be viewed as points in the search space, one can see that there is an
inherent exploration and exploitation trade-o↵in choosing which points to pursue. How
much should the evolutionary algorithm focus on individuals that have shown good ﬁtness
at the risk of getting stuck at a local optimum, compared to exploring new individuals
which might never lead to an acceptable solution. One of the ways to control this trade-
o↵is to introduce a population. At any point in time the EA has µ individuals in the
population, and chooses a subset to evolve. Unless the subset always consists of the single
ﬁttest individual, it is intuitively clear that having a population forces the EA to explore
more of the search space compared to only having a single individual.
To simplify the analysis, most complexity research for evolutionary algorithms, with few
exceptions, has focused on the case where the population size is µ = 1. While this is a
good starting point, it is not necessarily representative of how evolutionary algorithms
are used in practice.
To close this gap, this project introduces a versatile framework
for the empirical study of the expected computational complexity of population based
evolutionary algorithms. The goal is to develop a framework that can guide researchers in
their theoretical analysis of population based evolutionary algorithms, in order for them
to prove tight bounds of the computational complexity of the algorithms.
We will develop various visualizations and other tools, and use these to empirically study
the expected optimization time of some population based evolutionary algorithms on var-
ious benchmark problems. The rest of this report is structured as follows. In chapter 2
we introduce the theoretical background for evolutionary algorithms and the benchmark
problems we will be studying. In chapter 3 we describe the design of our framework, and
in chapter 4 we describe how it is implemented. In chapter 5 we use the framework to give
empirical complexity bounds to guide future theoretical research. Finally, in chapter 6 we
conclude and discuss future work.

2.2
Evolutionary Algorithm
One of the algorithms we will be studying throughout this project is the (1 + 1) EA
with a population.
The (1 + 1) EA is a simple evolutionary algorithm that has been
studied extensively in the literature. It has a proven complexity of ⇥(n log n) for any
linear pseudo-boolean function [7].
The algorithm starts by initializing a random bit string x 2 {0, 1}n. At each iteration,
the algorithm ﬂips each bit of x uniformly at random with probability pm. pm is usually
set to 1
n, which means that 1 bit is ﬂipped in expectation at each iteration. If the ﬁtness
value of the new bit string f(x0) is better than the ﬁtness of f(x), the algorithm replaces
x with x0. This process is repeated until a stopping condition is met. In our case we know
the maximum ﬁtness for all our benchmark problems, so we stop the algorithm once this
ﬁtness is reached.
The pseudocode for the (1 + 1) EA is shown in Algorithm 1 [7].

Notable Complexity Results
We will here try to summarize some of the notable complexity results for the (µ + 1) EA
on the benchmark problems we will be studying.

OneMax
A tight bound of ⇥(µn + n log n) for OneMax was proven by Witt in 2006 [24]. The proof
for the upper bound, O(µn+n log n), is based on an analysis of the potential L deﬁned as
the maximum ﬁtness value of the current population. The proof calculates the probability
that L increases at each iteration as the probability that an individual with ﬁtness L is
selected, multiplied by the probability that the selected individual is improved. A waiting
time argument gives the ﬁnal expected runtime.
The proof for the lower bound, ⌦(µn + n log n), is divided into two parts. The ⌦(n log n)
term is based on a proof by Droste et al. [7] and generalized by Witt to µ individuals [24]
whereas the ⌦(µn) term is based on an analysis of the family trees introduced in this
context also by Witt [24]. The idea behind family trees is that each individual in the
population can be traced back to one of the µ individuals in the initial population by
considering the sequence of mutations that produced the individual. If this path is small
at a large time t, it means that few mutations have occurred and the individual is still
relatively similar to the initial population, making it unlikely that it is the global optimum
for the ﬁtness function. The proof thus upper-bounds the length of the path, thereby
lower-bounding the time it takes to reach the optimum [24].
For the ⌦(n log n) term, Droste et al. argue that all bits that are initialized as 0 must
ﬂip at least once before an optimum is reached. The expected time before all these bits
have ﬂipped must therefore be a lower bound on the expected running time on linear
functions [7] [24]. This argument is also known as the coupons collector problem. The
proof is valid for any ﬁtness function that has a unique global optimum, and is thus also
valid for OneMax.
LeadingOnes
A tight bound is known of ⇥(µn log n + n2) for LeadingOnes, which was also proven by
Witt in 2006 [24]. The proof for the upper bound follows the same approach as the upper
bound proof for OneMax, but the lower bound is based on a slightly di↵erent argument.
The proof of ⌦(n2) comes from analysing the stochastic process behind the bit ﬂips (Droste
2002). The analysis shows that with high probability, n2/6 steps are not suﬃcient to reach
the optimum [7], thus proving a bound of ⌦(n2). The proof of ⌦(µn log n) by Witt [24]
is based on a similar idea as the ⌦(µn) part for OneMax. By analysing the path in the
family trees, Witt upper-bounds the length of the path from one of the root nodes to the
leafs which translates to a lower-bound on the time it takes to reach the optimum [24].
BinVal
As BinVal also has a unique optimum, we know from Witt that we can bound the expected
runtime to ⌦(µn + n log n).
In the unpublished work (µ + 1) EA on Linear Functions by D. Sudholt, F. Neumann
and C. Witt [25] they prove an upper bound of O(µn log n) for (µ+1) EA on an arbitrary
linear function, therefore also BinVal. The proof ﬁrst deﬁnes the ﬁtness layers as the sum of
the weights in the linear function up to position i, i.e. Li = {x | wn+wn−1+· · ·+wn−i+1 
f(x) < wn + wn−1 + · · · + wn−i}.
They argue that it takes O(µ log n) expected time
until the current population contains at least n individuals in level Li, afterwards the
expected time to leave the level is O(µ) as the probability of leaving the level is at least
⌦((n/µ)(1/en)) = ⌦(1/µ). Summing over all the n levels gives the bound O(µn log n) [25].

2.3.1
Notable Complexity Results
We will here try to summarize some notable complexity results for the (µ + 1) RLS. Most
complexity research has been focused on the (1+1) EA and its population based variants,
so we only have rough bounds for RLS.
The resemblance between (1 + 1) RLS and the coupons collector problem has been noted
previously in the literature [10] to obtain an O(n log n) bound on OneMax [15]. We here
extend the argument for population based RLS to obtain an O(µn log n) bound for (µ+1)
RLS on OneMax and BinVal. In fact, this bound applies to (µ + 1) RLS for all strictly
monotone functions, i.e. where f(x) < f(y) for all x < y.
At each iteration of the algorithm, exactly one individual will be chosen for mutation.
Due to elitism, we know that the child is only added to the population if it has at least
the ﬁtness of the current worst individual in the population. As RLS ﬂips exactly one
bit and the ﬁtness of OneMax and BinVal will increase if and only if a bit is ﬂipped
from 0 to 1, we know from the coupons collector problem that this individual, if always
chosen for mutation, will reach the optimum in O(n log n) iterations in expectation. In the
worst case scenario, all individuals are progressed uniformly meaning that all individuals
will eventually be located at the ﬁtness layer just before the optimum. We assume that
all individuals stay independent throughout the execution, as this is the worst case. As
there are µ individuals to progress, this will have taken O(µn log n) iterations, thereby
concluding the argument.
As proven by Doerr et al. [5], one cannot make a similar bound for (µ + 1) EA as the
one we just made for (µ + 1) RLS for all monotone functions, when considering mutation
probabilities pm = c
n with an arbitrarily large constant. The pm increases the likelihood
that the (µ + 1) EA will make detrimental steps as determined by the distance to the
optimum.
In the coupons collector problem, the collector is not allowed to uncollect the previously
collected coupons. Thus, the coupons collector argument above does not apply for Leadin-
gOnes, as multiple individuals can have di↵erent chromosomes but same ﬁtness value. We

note that a trivial lower bound of ⌦(µ + n) can be obtained by considering the initializa-
tion of µ individuals and that RLS can only progress one ﬁtness step at a time. As each
individual is expected to have n
2 1’s at initialization, an individual will with very high

3
Software Design
The overall goal of this project is to develop a software framework that helps researchers
estimate the complexity of evolutionary algorithms, which will help them prove the com-
plexity bounds theoretically. To accomplish this we will develop a set of visualizations
that allows the researchers to dive into the characteristics of the speciﬁc algorithms, and
a program that itself uses genetic algorithms to automatically estimate the complexity
of the evolutionary algorithms under study. All of this uses the same generic foundation
which we will outline in the architecture section below. This allows us to both model the
evolutionary algorithms that uses bitstrings as chromosomes together with the algorithms
used for the automatic complexity estimation, where the chromosomes are more complex
expression trees.
While this project focuses on algorithms that only has mutation as a search operator, the
generic foundation is designed to also support algorithms that has a crossover operator to
expand the class of algorithms that can be analyzed in the future.
3.1
Software Architecture
The project is written in Rust [22], so to properly understand the architecture it is ben-
eﬁcial to ﬁrst introduce some of the basic Rust concepts. Speciﬁcally traits and generics
are used to obtain a very modular architecture.
Traits are very similar to typeclasses in Haskell, and somewhat similar to Java inter-
faces. The idea is to deﬁne the ”object” and ”what-the-object-can-do” separately. If one
for instance wants to model a dog they would have a struct Dog class which might im-
plement a trait Bark, trait Walk and trait Eat. Other classes can then be generic
over a type and use these traits as constraints for what that type must be able to do.
Generics in Rust are types known at compile time that an object is parametrized over.
For instance continuing from the previous struct Dog example we could also have a
struct FoodBowl<A> object that is generic over the type of animal A. It doesn’t make a lot
of sense to have food bowl for an animal that doesn’t eat, so one can specify the constraint
that the generic type should implement the Eat trait as struct FoodBowl<A: Eat>. Mul-
tiple traits and constraints can be speciﬁed this way, building a constraint system that the
compiler must solve to be able to compile the program. If one has designed this constraint
system suﬃciently, the compiler will reject a lot of invalid programs during compilation,
increasing the conﬁdence that when the program does compile it’s mostly also correct.
With this background information out of the way, we can now discuss how we have used
these concepts in our software architecture. A simpliﬁed class diagram can be found in
Figure 3.1.

The GeneticAlgorithm class is the main class in the program, responsible for executing

fn run(&self) {
while !self.stop_condition.should_stop(&self.population) {
let (mut parent_a, mut parent_b) = self.selection.select(&self.population);
let mut offspring = self.crossover.crossover(parent_a, parent_b);
self.mutation.mutate(&mut offspring);
offspring.set_fitness(self.fitness.score(&offspring));
self.tracker.add_node(&[parent_a, parent_b], offspring);
let mut worst = self.population.peek_worst();
if offspring.fitness >= worst.fitness {
*worst = offspring;
}
}
}

3.2
Data Visualization and Gathering
in order to be able to analyze the performance of the evolutionary algorithms, we have
made the genetic algorithm generic over a Tracker type. The Tracker does not inﬂuence
the execution of the algorithm, but simply records data about the execution which can
later be analyzed. This allows us to implement di↵erent trackers to analyze di↵erent as-
pects of the evolutionary algorithm. We have implemented various di↵erent trackers used
to visualize di↵erent behaviors of the algorithms as outlined below.
Family Graphs
We want to study how the population evolves over time and how speciﬁc individuals get
mutated in order to produce the optimal solution. For this we will visualize the family
trees introduced by Witt [24] and explained in subsection 2.2.1 as the paths in these trees
shows the mutation a particular node has undergone. As each node can have multiple
parents when using crossover as a search operator, we will have to generalize the family
trees into family graphs where each node can have multiple parent nodes. Note that if
we maintain a direction from each parent node to its children, we get a directed acyclic
graph.
The family graph is created with the FullGraph tracker which constructs the FamilyGraph.
The graph is then visualized in a GUI showing how each node was evolved from its parents.
It turns out that visualizing trees, and by extension planar graphs, is by no means a trivial
task. In fact, if one wants to visualize the tree with a minimal width without preserving
the relative order of the children, the problem is NP-complete [21]. As this project is not
really focussed on the graph layout problem, we adopt a simple visualization strategy: To
position the nodes in the graph, we use the depth value as the y-coordinate and position
the node on the smallest possible x-value that has not yet been occupied for that depth.
This is done recursively from each node to its children. We then add some random noise
to each coordinate to give the graph a more natural look.

3.3
Complexity Estimation using Genetic Programming
In this section we will outline how we use concepts from genetic programming to em-
pirically estimate the running time of evolutionary algorithms.
Our work is a speciﬁc
application for the more general symbolic regression problem [23].
To our knowledge,
this is the ﬁrst project trying to use genetic programming to estimate the complexity of
evolutionary algorithms.
The main idea is to treat the asymptotic complexity function of the expected running time
of an evolutionary algorithm on a particular problem as an expression tree and evolve the
best expression possible based on empirical data, which we will gather by running the
algorithm under study numerous times on di↵erent values of µ and n while recording the
number of ﬁtness evaluations. While this might seem fairly straight forward, if one were to
do this complexity estimation naively they would end up with a very complicated expres-
sion which would ﬁt the training data very well but would not yield the true asymptotic
complexity. As such, the main problem to tackle is to reduce bloat of the expression tree.
Some of our bloat reduction techniques come from the genetic programming and symbolic
regression literature and as such are generally applicable, whereas others are speciﬁcally
designed for our asymptotic complexity estimation task.
3.3.1

the leafs refer to either a constant or a variable. In our expression trees we also have the
notion of an empty leaf which is useful when evolving the tree. A simple expression tree

3.3.2

is typically very diﬃcult to tune. Because of these results, we use Double Tournament
combined with depth limiting in this project.

3.3.3
Mutation and Crossover
To mutate the tree we ﬁrst generate a new random node. If the node is an internal node,
we generate random leafs corresponding to the arity of the internal node. As stated earlier,
the generated leafs can either be an empty node, a variable or the constant 1.
The generated expression then gets merged into the mutating individual at a random part
of the tree, which we will call the mutation node. We try to preserve the subtrees of the
mutation node as much as possible, and only substitute the node itself with the internal
node of the generated expression. This experimentally seems to provide a good trade-o↵
between exploration and exploitation of the search space.
Suppose the mutation node is a and the internal node in the generated expression is b.
There are now 5 cases to consider in ascending order:
1. a = b: substitute the nodes and keep subtrees from a.
2. a or b is a leaf: substitute the nodes and keep subtrees from b.
3. arity(a) = arity(b): same as case 1
4. arity(a) < arity(b): substitute a with b, replacing one of the subtrees of b with
the one from a. This should keep one subttree from b and one from a.
5. arity(a) > arity(b): remove a random subtree from a and substitute a with b

==PAGE:20==
This mutation procedure gives us a new individual where a large part of the new tree
should be equivalent to how it was before mutation.
The crossover operation we use is very simple compared to the mutation operator. We
ﬁrst select two trees from the population to be used as parents for the crossover operation.
We choose a random node in each of the two trees and swap their subtrees, producing two
new trees. One of the new trees is returned as the o↵spring. Note that the replacement
strategy in crossover is not the same as the one during mutation. The intuition is that
each of the chosen subtrees have survived more than a single iteration of the algorithm,
and as such will likely ﬁt the data better than a randomly generated tree. Performing the
simple replacement in crossover thus forces the algorithm to explore more of the search
space, while not completely obliterating good individuals.
3.3.4
Fitness Function
As stated earlier, our goal is to evolve the asymptotic expression that corresponds to
our empirical data. We would expect that if we sample from the same ranges we have
allowed n and µ to take when generating the empirical data and insert these samples
into the generated expression, the sampled datapoints should largely come from the same
population as our empirical data. The Mann-Whitney U-test [19] is a statistical test that
can be used to test whether two samples are likely to come from the same population.
Our ﬁrst approach therefore used the p-value from the Mann-Whitney U-test as a ﬁtness
function for the genetic algorithm, as a larger p-value would likely correspond to a better
ﬁt. What we found was, that using the p-value as a ﬁtness function introduced a large
amount of bloat in the generated expressions, and the generated expressions would quickly
reach the maximum p-value of 1.0. We therefore ended up using the simpler mean squared
error as the ﬁtness function. Since evolutionary algorithms generally maximize the ﬁtness
function and whereas we want to minimize the mean squared errors, we multiply the
function with −1 to obtain our ﬁnal ﬁtness function

3.3.5
Ensemble Method
We found that a single run of the genetic algorithm may converge to di↵erent asymptotic
expressions due to the inherent randomness of genetic algorithms. To address this issue
and increase the reliability of the discovered expressions, we adopt an ensemble method.
We run the genetic algorithm for a ﬁxed number of iterations, in our case 10000 iterations.
This process is repeated 100 times, each generating a potentially di↵erent asymptotic
solution. Each execution uses a population of 1000 trees, with a crossover and mutation
probability of 1.0.
After the runs, we group the results based on the identiﬁed asymptotic solutions. For
instance, if two runs result in the expression ⇥(n2), they are grouped together.
For each unique asymptotic solution, we assign a score. To calculate the score, we ﬁrst
select an individual (full expression with constants) from each unique asymptotic solution
that got the best mean squared error for that particular asymptotic solution. We then
calculate the p-value from the Mann-Whitney U-test for each of these individuals.
19
==PAGE:21==
The score for the asymptotic solution is then calculated by multiplying this p-value by the
number of times the asymptotic solution was the best across all runs. Formally, if Pbest is
the p-value of the individual with the best mean squared error for an asymptotic solution
and C is the number of times the asymptotic solution was the best overall, the score S is
given by:
S = PbestC
(3.2)
This scoring mechanism penalizes over-complicated expressions that might achieve good
ﬁtness as they are less likely to be consistently discovered across multiple runs. On the
other hand, the solutions are rewarded from having a better p-value. The intuition is
that simpler, more general solutions will have a higher probability of emerging in multiple
independent runs, suggesting their robustness.
After scoring all the unique asymptotic solutions, we select the solution with the highest
score as the ﬁnal ensemble output. This ensures that we favor expressions that are both
ﬁt and consistently discovered, thus likely leading to the correct complexity estimate.

Implementation
The project is written in Rust [22] due to performance requirements and previous experi-
ence by the authors. One of the primary features (and sources of frustration) in Rust is
the borrow checker and the concept of ownership.
In short, the borrow checker ensures that there is only one mutable reference to a value at
any given time. This also entails that each value has a single owner, and that the owner is
responsible for freeing the value when it goes out of scope. This is a very powerful concept
as it completely sidesteps the need for a garbage collector, while still not requiring the
programmer to manually free memory. It does, however, make it diﬃcult to model concepts
such as doubly linked lists and graphs, as each node in the list or graph would have to
be owned by both its predecessor and successor. This is not possible in Rust, as it would
require two mutable references to the node at the same time.
Family Graphs
The usual way to solve implement circular referenced data structures, and how we have
implemented our family graphs, is to store the nodes in a Vec<Node> and introduce a
NodeId type which is an index into the vector. This way, the nodes are owned by the
vector, and the vector can be mutated even though multiple NodeIds exists for the same
node.
The graph can then be built by storing the parents in each node struct as a
Vec<NodeId> and the children likewise as a Vec<NodeId>.
Chromosomes
We have implemented two di↵erent chromosome types. One is a tree chromosome which
is used for the asymptotic complexity task (section 3.3), while the other is a bitstring
chromosome used to evaluate the evolutionary algorithms. The tree is implemented as a
recursive algebraic data type, while the bitstring is implemented as a wrapper around a
vector of 8-bit unsigned integers and a length. We then bitshift the integers to get the
correct bit when indexing the bitstring. This is done to avoid the overhead of allocating
a new byte for each bit which would be the case if we used a vector of booleans. This
not only reduces the memory consumption by a factor of 8, but it also increases the cache
locality for the CPU which increases the performance.
For instance, suppose we have a bitstring consisting of 128 bits which would be stored
in 16 8-bit unsigned integers.
To set bit 42 to 1, we would ﬁrst calculate the correct
byte as b 42
8 c and the bit index in the byte as 42 mod 8. The bit is then updated by
shifting 0b00000001 left 7 −(42 mod 8) times and performing the bit-or operation with
the existing byte. If on the other hand we wanted to set the bit to 0, we would invert the
shifted byte and use the bit-and operation with the existing byte. This would e↵ectively
use the shifted byte as a mask of which bits to keep in the stored byte.
Note that this bit assignment can be implemented without any branching assembly in-
structions by ﬁrst performing the false case after which we perform the true case. The
modulo operator can also be implemented as a logical and since 8 is a power of 2. Thus,
updating a bit only requires very simple instructions without any branch predictions,
making the entire process highly eﬃcient.

GUI
All graphical user interfaces are implemented using the egui library [8] which renders the
GUI natively using OpenGL. This is done to avoid the overhead of a web browser based
application, and to avoid the need for a web server which greatly simpliﬁes the project.
Egui is an immediate mode GUI library, which means that the GUI is rendered every frame
instead of being retained in memory. This is in contrast to retained mode GUIs such as
Qt [12] and GTK [11] where the elements are retained in memory and only updated when
necessary. The advantage of immediate mode GUIs is that they are much simpler to use,
while the disadvantage is that they are less eﬃcient as the GUI has to be re-rendered every
frame. This is not a problem for our application as the GUI is not very complex and can
thus easily be rendered every frame.
Parallelization
All our evolutionary and genetic algorithms only use a single thread when executed. This
leaves a lot of performance on the table, as modern CPUs have multiple cores which can
be utilized to speed up the execution. However, it is not trivial to parallelize an ordinary
evolutionary algorithm as the population at step t is dependent on the population at step
t −1.
During data gathering and the ensemble method in the asymptotic complexity task, we run
multiple instances of the same algorithm, sometimes with di↵erent conﬁgurations. Each
instance of the algorithm has their own population and is thus completely independent of
the other runs. This means we can trivially execute each instance in parallel which greatly
speeds up the overall execution time, as we can utilize all the cores in the CPU with almost
no communication overhead between the threads. This is done using the rayon library [4]
which provides a parallel iterator over a collection. The library automatically distributes
the work to the available threads in its threadpool and collects the results.
As the ownership model in Rust ensures that there is only one mutable reference to a value
at any given time, the compiler statically prevents data race conditions. This increases
the conﬁdence in the correctness of the parallel program.
Big Numbers
Some ﬁtness functions might be diﬃcult to compute if we restrict ourself to 64 bit ﬂoating
points. Consider the binval function, where the chromosome is interpreted as an unsigned
integer. If we represented the ﬁtness value as an f64 (64-bit ﬂoating point) or u64 (64-bit
unsigned integer), we would restrict ourselves to a problem size, n, of at most 64 (actually
less for ﬂoating points due to how they are represented in memory). Thus, we need a
better way to represent very large numbers in memory. For this we use the bigdecimal
library [1]. We will here brieﬂy outline how this library works internally. The library
includes a number of convenience functions (addition, multiplication, conversions etc.)
which is why we use the library instead of implementing the logic ourself.
The bigdecimal library stores the value as a BigInt struct and a 64-bit unsigned integer
to denote where the decimal point is. The BigInt is essentially a list of 64-bit unsigned
integers. This allows us to represent decimal numbers with up to 264 digits, or arbitrarily
large numbers without decimals. In practice this limit is non-existent as we rarely need
to represent numbers with more than 264 −1 digits.

5.7
(µ + 1) EA on 3-CNF With Planted Solution
To investigate how the algorithms perform on more real life problems, we will now conduct
an analysis of (µ + 1) EA on the 3-CNF satisﬁability problem with a planted solution.
The model that we will study here has also been studied by Doerr et al. on (1+1) EA [6].
The problem and model are as follows.
A k-CNF formula is a logical formula, F, over n boolean variables that consists of m
clauses. Each clause consist of exactly k literals. When the formula is a conjunction of
clauses, i.e.:
F = C1 ^ C2 ^ · · · ^ Cm
Where each clause is a disjunction of literals, i.e.:
Ci = li1 _ li2 _ · · · _ lik
The formula is said to be on conjunctive normal form, CNF. Each literal consists of either
a boolean variable, v, or its negation, ¬v. In the k-CNF satisﬁability problem our goal is to
ﬁnd a boolean value for each variable such that F is satisﬁed. As all propositional formulas
can be written in CNF, the satisﬁability of k-CNF formulas has a strong resemblance with
the classical k-SAT problem.
In this model we will treat the bitstring as a boolean assignment of the variables. In other
words, bit bi will determine the value of variable vi. We want the algorithm to evolve the
solution that satisﬁes all clauses in the formula, thereby satisfying the formula. As such,
the ﬁtness function we use will simply count the number of clauses satisﬁed by a given
chromosome, x:
f(x) = |{C 2 F : C is satisﬁed by x}|
Without loss of generality, we will randomly generate the formula F in such a way that
the all one bitstring x⇤= (1, 1, . . . , 1) is a planted solution that satisﬁes F. There might
be other boolean assignments that also satisﬁes F.
Doerr et al. [6] theoretically and experimentally analyzed how the (1 + 1) EA algorithm
performed on this model. They found that the algorithm has a computational complexity
of O(n log n) when the clause density is m
n = ⌦(n). They argue that the behavior of the
(1 + 1) EA on k-CNF has a high resemblance to the OneMax problem as they prove there
is a high ﬁtness-distance correlation in both problems. We will here focus on instances
with a clause density of ⌦(n) and k = 3.
Note that we cannot guarantee that the (µ + 1) RLS algorithm terminates in this model.
As Doerr et al. did, we also allow duplicate clauses in F. One can therefore imagine
a situation where the algorithm has to perform a search step that reduces the distance
to the optimum by more than 1 in order to ﬁnd an individual with a better ﬁtness. As
(µ + 1) RLS always performs exactly 1 bit ﬂip at each step, it will not be possible for the
algorithm to make progress on the problem. Suppose we have the following set of clauses
in the formula:
F = (¬v1 _ v2) ^ (¬v1 _ v2) ^ (v1 _ ¬v2) ^ (v1 _ ¬v2) ^ (v1 _ v2)
Here C1 is a duplicate of C2 and C3 is a duplicate of C4. Suppose the algorithm has a
single individual in its population with the bitstring x = (0, 0). This would satisfy all
clauses except the last one, giving it a ﬁtness of f(x) = 4. As (µ + 1) RLS only ﬂips
exactly on bit at each step, it’s only possible for it to generate these two new individuals:
46
==PAGE:48==
x1 = (1, 0) and x2 = (0, 1). Each of these new individuals would successfully cause the
last clause to be satisﬁed, but would either make C1 and C2 or C3 and C4 unsatisﬁed.
The ﬁtness of these new individuals are thus f(x1) = f(x2) = 3. As this is lower than the
ﬁtness of the original individual, neither of them would be chosen due to elitism selection.
As such, it would not be possible for the algorithm to move from a ﬁtness of f(x) = 4 to
the optimal individual f(x⇤) = 5 and would therefore never terminate. We will therefore
only study the (µ + 1) EA algorithm here.

In this project, we have designed and implemented a software framework that gives re-
searchers insights about the complexity of population based evolutionary algorithms. We
have successfully utilized the tools to validate the complexity of (µ+1) EA on 2 problems
while giving empirical estimates of the tight complexity bounds for (µ+1) EA and (µ+1)
RLS on 2 and 3 open problems respectively. An overview of our results can be seen in
Table 6.1 with p-values from the Mann-Whitney U-test for our estimates.
In this framework, we have also implemented a system that treats the asymptotic complex-
ity estimation problem as a symbolic regression problem, and used genetic programming
to solve the complexity estimation task. While our results seemed very promising initially,
we found that the system produces estimates with unsatisfactory p-values for some of our
studied problems. The main diﬃculty seems to come from estimating the importance of
constants. As can be seen in appendix 7.5, some problems seems to su↵er from having
solutions where the constants are severely overﬁtted, resulting in solutions that are not
complex enough in terms of the variables whereas other estimates su↵er from the reverse
problem. Further research is needed to study how to dynamically perform the trade-o↵
of the importance of constants for each speciﬁc problem, so that the solutions we ﬁnd
are neither overﬁtted nor underﬁtted. A list of unsuccessful experiments we have tried
can be found in appendix 7.5. We here provide some approaches that future research can
potentially investigate:
1. Try di↵erent mutation and crossover operators.
2. Introduce a parsimony coeﬃcient and penalize bloated trees during selection. In
[14], Kronberger et al. introduced an algorithm that would dynamically adapt the
parsimony pressure to reduce overﬁtting for symbolic regression problems.
3. Liu et al. [17] found that a speciﬁc genetic programming algorithm used for symbolic
regression, tends to over replicate low-complexity solutions early in the generation.
These solutions take over most of the population, causing the ﬁnal solution to be
suboptimal. To combat this, they introduce some diversity maintaining operators.
4. Change the genetic programming algorithm to be based on the works in [26], which
introduces a di↵erentiable genetic algorithm for symbolic regression.
While they
49
==PAGE:51==
introduced this approach for problems with very high dimensionality, di↵erentiability
might also improve the individuals generated for lower dimensional problems like
ours.
These are very interesting, and in our opinion, promising avenues of research which might
improve the generated solutions to such a degree so that researchers can automatically
estimate tight complexity bounds of evolutionary algorithms based on empirical data.
As our framework is designed to be easily extendable, it will hopefully be relatively easy
for future researchers to add and study more complex population based evolutionary al-
gorithms or benchmark problems in order to theoretically prove their complexity bounds.

close to their theoretical bounds, whereas for the remaining combinations the solution is

from varying importance of the constants and other lower-order terms for each problem
which adds signiﬁcant noise to the empirical data, making it harder to ﬁnd the true
asymptotic complexity.
Looking at the estimate for (µ+1) RLS on OneMax, we see that the estimated complexity
of ⇥(nµ) actually manages to get a relatively high p-value, even though the ﬁnal solution
seems to be underﬁtted in the sense that it’s missing some terms from the complexity
function. This situation happens, when the constants in the solution matches the sampled
data too well, causing the algorithm to prioritize solutions with better constants instead
of focussing on the variable terms n and µ. The direct opposite problem is seen in (µ + 1)
RLS on LeadingOnes, where the complexity is clearly too complex. Here, the algorithm
should actually focus even more on the constants in the solution which would allow it to
remove some complex terms.
As such, we see that the importance of constants is highly dynamic based on which algo-
rithm we are studying and on which problem. To alleviate this problem, we tried di↵erent
approaches that unfortunately didn’t work.
Unsuccessful Experiments
We ﬁrst tried to use non-linear solver based on the Levenberg-Marquardt algorithm [16]
[20] to ﬁnd the constants that best ﬁt the sampled data. This caused all the solutions to
be severely overﬁtted (most of the estimates were ⇥(nµ)) as the constants were simply
too good.
We then tried to introduce a train/test split where we would use x fraction lowest param-
eters as a train split to ﬁt the constants, and evaluate the ﬁtness function on the upper
(1 −x) fraction. Unfortunately, this did not seem to meaningfully improve the solutions.
We then abandoned the non-linear solver and tried to generate constants randomly in
an interval the user could control from the conﬁg ﬁles. The user could also control the
probability that a newly generated leaf would be a constant or a variable. This improved
the solutions compared to using the solver, but was inferior to using only the constant 1
and letting the algorithm evolve more complex constants. The approach was also more
complex as it had more parameters to tune.
Finally, we tried using a train/test split again during ensemble selection, such that the
algorithm would evolve the solutions based on the train split but would select the best
asymptotic solution based on the test split. This did not meaningfully improve the solu-
tions.
