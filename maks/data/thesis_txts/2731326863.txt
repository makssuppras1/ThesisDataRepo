Abstract
We consider the random access in persistent strings problem [3]. A persistent string is a
collection of similar strings that can be modeled as a version tree where the nodes represent
the strings and the edges are labeled by edit operations (insertion or deletion of a character).
We consider the problem of supporting efficient access(v,j) queries on such a collection,
which corresponds to retrieving the jth character of version v. To do this, we consider the
analogous problem of segment selection, which is the problem of finding the jth segment
crossing at time i given a set of horizontal line segments. We present the reduction from a
persistent string into an instance that supports segment selection queries. We establish a
baseline solution to segment selection using the segment wavelet tree [15]. We then generalize
the segment wavelet tree to an arbitrary degree segment wavelet tree supporting access(v,j)
in O(log(n)) time using O(n) space like the segment wavelet tree, but being faster in practice.
We present five solutions to the random access problem in persistent strings, implemented in
C++. We achieve the fastest solution at scale with our generalization of the segment wavelet
tree with degree 16 using a cache-efficient memory layout.

Chapter 1
Introduction
The random access problem is to preprocess a data set such that we can retrieve a data point
efficiently from anywhere in the original data. In this paper, we will look at the random access
in persistent strings (RAPS) problem. A persistent string is a collection of strings where
each string in the collection results from applying an edit operation on some other string in
the collection. The only exception is the empty string, which we denote as ϵ, which is always
included. An edit operation is either an insertion or a deletion of a character. Therefore, the
RAPS problem is preprocessing a persistent string so we can retrieve a character from any
string in the collection efficiently.
Intuitively, such a collection represents a single string’s history of changes over time. The
problem is to be able to look up a specific string in the history and return a particular
character at a certain position in that string [3, 15].
This has several interesting use cases. A very natural application is that of saving a document’s
complete edit history, allowing fast retrieval of any version. Instead of explicitly saving the
entire history, we could keep it as a version tree with edit operations on the edges and then
use that representation to retrieve a node quickly. This can also be expanded to version
control systems working in memory [3].
Furthermore, we can think of a set of highly related strings as different versions of the same
string and thus represent them in a compressed way that still allows fast retrieval of any
string. An example could be storing and accessing related genome sequences in memory.

1.2
Persistence
Ordinary data structures are ephemeral in that the old version is destroyed once modified.
On the contrary, persistent data structures are data structures that preserve all previous
versions of themselves when they are altered and provide access to the earlier versions [8].
There are four levels of persistence: Partial persistence, full persistence, confluent persistence,
and functional persistence.
Partially persistent data structures provide access to any version of the structure but
allow only updates to the newest version. Looking at the version tree in fig. 1.2a, we can see
that it forms a path with modifications allowed on the newest version.
Fully persistent data structures provide all the same functionality as partially persistent
data structures but also allow updates to any version. Looking at the version tree in fig. 1.2b,
we can see that it forms a tree with modifications allowed on any version, thus creating a
new branch.
Confluently persistent data structures provide all the same functionality as fully persistent
data structures but allow the merging of two versions. Looking at the version tree in fig. 1.2c,
we can see that it forms a directed acyclic graph.
Lastly, functional data structures are immutable and thus do not allow any modifications.
Instead, an entirely new data structure is created from an old one. Functional data structures
have the same properties as all the others. The difference between the other types of persistent
data structures is that they are allowed not to have to create a complete copy as long as
they represent the same data. However, functional persistence requires an entirely new copy
to be made [5].
Furthermore, we can divide persistence into static and dynamic persistence. Static means
that when creating our data structure, we know ahead of time all the versions we want to be
able to access, whereas dynamic means that we do not.
This paper will be concerned with fully persistent static data structures. Hence, we can
represent a sequence of events (edits) that have already happened and where changes were
made to any version.

1.3
Previous Work
Persistent data structures are a well-studied problem [3, 5, 8, 11]. It was first formally studied
by Driscoll et al. [3, 7]. In [8], Driscoll et al. present a general approach for making linked
data structures both partially and fully persistent with an amortized constant overhead per
modification. This implies that RAPS can be solved by representing a dynamic string with a
fully persistent balanced binary search tree (BBST) [3].
Such a representation of the string can be maintained implicitly with weights in the tree.
Since the BBST is a linked data structure that can be represented in a pointer machine, we
can make it fully persistent by applying the node splitting method proposed in [8]. This
implies that we can get a O(log(n)) query-bound solution using O(n · log(n)) space on
persistent strings.
There exists an approach to reduce a partially persistent BBST from O(n · log(n)) space
to O(n) space presented in [14]. However, we need the string indices represented with the

the worst case, we must rebalance the weights from leaf to root. Therefore, we still need

string with O(n) space and optimal random access in O(

Chapter 2
Algorithms and Data Structures
This chapter presents the relevant algorithms and data structures from a theoretical per-
spective. We explain two simple solutions establishing an upper and lower bound on query
time and space. We then present the analogous problem of segment selection and the
transformation from an instance of RAPS into a set of segments [3]. Then, we show how to
solve it using the segment wavelet tree of [15]. Lastly, we propose a generalization of the
segment wavelet tree to support arbitrary degrees.
2.1
Establishing Bounds
We start by looking at establishing some bounds for the RAPS problem by solving it in two
simple ways. These solutions will give us an upper and lower bound on both the time and
space complexity of the problem. The naive solutions are to store all versions explicitly or to
store only the changes.
2.1.1
Saving All Versions Explicitly
A simple solution to the problem is to store all the strings S1, . . . , Sn in a data structure
such as an array, A[1, n]. Construction is done for each string by copying a previous string
==PAGE:14==
2.1 Establishing Bounds
7
according to the version tree and then applying the edit operation. Here, we can perform
an access(v,j) by doing a lookup for Sv ∈A and then simply retrieving the jth character
from Sv.
Since we can do the lookup in the array and the string in constant time, this leads to a query
complexity of O(1). However, when it comes to space, we save all the strings of which there
are n. Given that all edit operations are insertions, the strings will have length 1, 2, . . . , (n−1).
Summing this together yields a worst-case space complexity of O(n2). To build the data
structure, we must, for each of the n −1 edit operations, copy the corresponding previous
string and apply the operation to it. Copying each string takes O(n) time as the length of a
string can be n −1 long in the worst case, as explained earlier. Since we copy n strings, this
leads to a worst-case preprocessing complexity of O(n2). This solution is also proposed in [3]
as a naive way to solve the RAPS problem.
2.1.2
Saving All Edit Operations
Another simple solution to the problem is representing a persistent string using a version
tree with n nodes. In each node (except for the root), we save the edit operation applied to
the node’s parent to make the node.
We store a node and three pointers for each version: a pointer from child to parent, parent
to child, and an access pointer to the node. Furthermore, we store n −1 edit operations.
Therefore, we obtain a space-bound of
n + 3n + (n −1) = O(n)
To perform an access(v,j), we start with an empty stack. We first access node v in
constant time due to the access pointer. After this, we traverse upwards until we reach the
root, pushing the edit operation onto a stack. In the worst case, a version tree represents a
partially persistent string, which is just a path. Hence, the tree’s height is at most n. After
reaching the root, we reconstruct by continuously popping edit operations off the stack and
applying the change to the initially empty string. Lastly, when the string is reconstructed,
we can access the jth character in constant time.
In the worst case, if we only insert at position 0 in the string we are constructing, we must
continuously shift all characters to the right. This results in 1 + 2 + · · · + (n −1) = n·(n−1)
2
such shifts. Therefore, we have a worst-case query bound of O(n2).

2.2
Segment Selection
In this section, we introduce the geometric problem of segment selection (SS). The problem
and transformation are first introduced by Bille and Gørtz. In Bille and Gørtz, they show
SS is analogous to RAPS. Hence, we can take our instance of RAPS, a version tree, reduce it
into an instance of SS, solve it, and thereby solve RAPS. We use the transformation of Bille
and Gørtz, with a slightly different definition of when a segment crosses a vertical line [15].
We then introduce our own general transformation to rank space in order to have a solution
that works on any space and our own solution to solving RAPS by solving SS in any space.
We denote a line segment in the plane l = (x1, x2, y) and a set of segments L. We denote L
sorted increasingly by x1 as L, and similarly, we denote L sorted increasingly by x2 as R. SS
is defined as follows:
Definition 2.1 segment-select(i,j): return the segment with the jth smallest y-
coordinate among the segments crossing the vertical line through x-coordinate i [3].
For definition 2.1, we define a segment to cross the vertical line through x-coordinate i if
x1 ≤i < x2. I.e., the left endpoint of a segment is crossing the vertical line, but the right
endpoint is not [15].
The x-axis of the spaces in the following can be seen as a time axis, so we will, therefore,
sometimes refer to values on this axis as time i.
2.2.1
Reducing RAPS to SS
To convert the version tree into a set of segments, we perform an Euler tour starting at
the root. During the traversal, we construct an auxiliary sequence of strings ¯S0, . . . , ¯S2n−2,
which we denote as marked strings. Each marked string in the sequence corresponds to a
step in the traversal. For each marked string, we distinguish between marked and unmarked
characters. A character in ¯St is marked if it does not exist in the string at step t during the
traversal. Therefore, conceptually marked strings represent all characters that have existed
at some point across versions. Figure 2.1 shows the marked strings visualized next to the
nodes at which we constructed them during the traversal of V . Suppose we are at step t −1
in the traversal and traverse edge e. Based on the edit operation performed by traversing e,
we do the following:
Insertion:

Now that we have reduced an instance of RAPS into a set of segments, we can solve the
segment selection problem. To construct the set of segments, we perform O(n) amount
of work in each node in the Euler tour since we continuously must find the ith unmarked
character. Hence, we can make the reduction in O(n2) time.
We will now introduce some terms for the spaces we are considering. We will always have at
most n segments on the y-axis. If all edit operations are insertions, we will have n segments,
each corresponding to a unique y-coordinate. Given that there are deletions, then several
segments share the same y-coordinate. We have different spaces regarding the x-axis as the
segments can be more or less "stretched." We will use the term interval space to refer to
the space the segments of eq. (2.1) are in, which is also the space the segments of [3] are in.
Interval space has redundancy because we might have multiple x-coordinates representing the
same information. In particular, we know from [3] that during the Euler tour, we construct
a sequence of ¯Si that has length 2n −1, and we multiply each ending interval by 2, thus
leading to interval space having a width of ≈4n.
However, each segment’s endpoint contains precisely the information of a single change of
the version of the persistent string. This is because it signifies either the beginning or end
of the existence of a character in time. Therefore, we can reduce the width of the space to
2|L| = 2n. This is what Tarnow calls rank space. Tarnow also has a transformation from
any space to rank space; however, we introduce our own as we expect it to perform better
in practice when querying in one space and representing the segments in another. We can
transform the following way:
Let ˆL represent segments in interval space, and L′ denote the corresponding segments in
rank space. Each segment li = (x1, x2, y) ∈L is divided into two tuples: ti,1 = (x1, y)
and ti,2 = (x2, y). We construct the set T by ordering these tuples based on ascending
x-coordinates, denoted as Ti. We define L′ as follows:
L′ = {(i, j, Tiy) | Tiy = Tjy ∧i < j, ∀i, j ∈[1, n]}
We construct the new segments by finding the tuples with the same y-coordinate and using
their index in the ordered set as the new x-coordinates.

Solving the RAPS Problem by Solving SS Problem
We now show how we can solve the RAPS problem by solving the SS problem. In [3], the
authors show that the x coordinate corresponding to a version in the version tree can be
found as 2i where i is the first time we meet the node in the Euler tour. This is, however, no
longer the case since we use the definition of when a segment crosses a vertical line from [15].
Tarnow proposes a solution to this problem; however, we introduce our own event-based
solution as we expect it to perform better in practice using slightly more space.
The SS instance of the problem contains all the information we need to solve the RAPS

problem. However, we have that segment-select(i,j) ̸= access(i,j), which can be
seen that the span of the segments is 2n while we can only query each of the n versions.
We notice that each version corresponds to an event, either the insertion or deletion of a
character in the persistent string. The insertion event corresponds to the beginning of a
segment. Since we always make a segment with a new y coordinate when we handle insertion,
the first x coordinate corresponds to an event for each y coordinate. Furthermore, we know
that each segment’s endpoint corresponds to the deliberate deletion of a character except
for the last one, as that is naturally the character fading out of existence as we go up in
the tree when we do the Euler tour. The order of the version events in the SS problem will
correspond to when we meet them in the Euler tour.

We can save an auxiliary structure to allow for fast ranking. We do this by keeping the result
of the rank operation as a checkpoint at regular intervals. We will then rank by going to
the checkpoint closest to the index we want to rank until and then scan the rest of the way.
Using this auxiliary structure leads to the following lemma:
Lemma 2.4 rank(B,b,i) can be done on a bit array in O(1) time and O(1) extra space.
The lemma holds because we only save a constant amount extra per bit as the checkpoints
are at regular intervals. Using these checkpoints, we are always a constant away from the
index we want to rank until (at most the distance between two checkpoints) so the scan
becomes constant, and we can do the lookup of the checkpoint in constant time.
We can represent both interval and rank space using a bit array. We do this by keeping a bit
for each x-coordinate in the space. We need 4n bits for interval space, and for rank space,
we need 2n bits. We scan through the x-coordinates to construct this bit array and set the
corresponding bit in the bit array given some condition. We will utilize this to encode the
left and right endpoints of segments in section 2.3.3. Doing this is linear in the number of
x-coordinates, which we know to be O(n).
Lemma 2.5 We can construct a bit array that represents interval and rank space using
O(n) bits in O(n) time [12].
2.3.2
Wavelet Tree
A wavelet tree is a succinct1 data structure that supports rank queries on a sequence from
an ordered alphabet. The root of the tree represents the whole sequence. Then, for each
layer of the tree, the alphabet is split evenly into two subranges based on the alphabet’s
order. We keep a bit array in each node that encodes each symbol in the sequence, whether
it becomes part of the subsequence for the left or right child.
For the wavelet tree, we expand the definition of rank from bit arrays to any sequence S[1, n]
of some orderable alphabet [1, σ] as in [15]:
1A succinct data structure is a type of data structure that uses space quantifiably close to the information-
theoretic lower bound [2, 6]. To classify a data structure as succinct, it must use OPT + o(OPT) bits of
space[6]. Here, OPT denotes the information-theoretic lower bound for the structure.

Definition 2.6 rank(S, α, i) : return the number of occurrences of symbol α ∈[1, σ] ∈
S[1, i], for any 1 ≤i ≤|S| [15].
We construct the tree from the root and partition the symbols into two. We encode all the
symbols of the first partition as 0 and all the symbols of the second partition as 1. These
bits are what we save in the root. We then concatenate all the symbols encoded as 0 and
recurse with them on the left child and concatenate all symbols encoded as 1 and recurse on
the right child. We continue until we no longer have any ambiguity.

We first consider what is encoded at the top of the tree to give the intuition as to why this
works. In the case where σ ≤2, we have the answer simply by performing the rank query for
the symbol. This is the case since we, by definition, when σ = 2, will encode one symbol as 0
and the other as 1. Hence, we can perform a rank query for the symbol and get the answer.
When σ > 2, we have ambiguity since the 0 or 1-bits we are counting at the root represents
both the symbol for which we are ranking and other symbols of the subrange. Therefore,
when we rank over these bits, the rank is an upper bound on the number of occurrences of
the specified symbol in the sequence. Hence, when we go down the proper subtree and the
symbols are re-encoded, we know that the rank is still an upper bound on the number of
occurrences of the symbol in the sequence - however, the bound becomes tighter since the
number of symbols that are encoded is half that of the parent node. Therefore, in the end,
when there is no ambiguity, we have the exact number of occurrences of the symbol in the
sequence up until the rank position.
We partition σ evenly at each node, which we at most can do log(σ) times. We perform a
rank query for each log(σ) level. Because we can compute rank queries in constant time
by lemma 2.4, a wavelet tree can be queried in O(log(σ)) time. To analyze space, we
note that we store n bits at each level of the tree plus some constant amount according to
lemma 2.4. Since we have log(σ) levels, we get a space complexity of O(n) · log(σ) bits. We
have that log(σ) ≤log(n). We can represent a symbol with log(w) bits, and therefore, we
have log(σ) ≤log(n) ≤log(w) (where w is the word size of the machine). Hence, we get the
following:
O(n · log(n)
log(n) ) = O(n)
To construct a wavelet tree over a sequence S[1, n] on an alphabet [1, σ], we have to partition
S at each level. We do so by scanning through S and determining the coding, which is linear
in n. Since we do this at each level, we get a preprocessing time of O(n · log(σ)).
We sum these results up as the following lemma:
Lemma 2.7 We can construct a wavelet tree in O(n · log(σ)) time to support rank queries
in O(log(σ)) time using O(n) space.

Segment Wavelet Tree
In this section, we describe the Segment Wavelet Tree of [15]. Then, we explain how it solves
segment-select(i,j) queries. Lastly, we analyze the time and space complexity of the

In order to use the segments of section 2.2, we will have to make sure that the y-coordinates
are unique so that we always split the segments according to y-coordinate evenly. This is
done for each y-coordinate from the bottom by pushing up all segments above when there
are multiple segments with the same y-coordinate except for one chosen arbitrarily. Thus,
we achieve segments with unique y-coordinates in the same order as before. See the top node
in fig. 2.6 for an example of this transformation on fig. 2.3.
We let YL denote a wavelet tree over the y-coordinates of L. Similarly, we denote the wavelet
tree over the y-coordinates R as YR (L and R are as defined in section 2.2). Since the trees
are constructed over the same set, they will have the same topology [15]. The segment wavelet
tree (SWT/BSWT)2 consists of YL and YR as well as a bit array BLE[1, 4n] and BRE[1, 4n]
in which bit i is set if there is an endpoint (i, y) ∈R where x = i. We specifically define two
bit arrays, one for holding the left endpoints and one for holding the right endpoints. This is
because we thereby create a structure that works in general, which includes interval space.
Tarnow defines BE and R(BE) which is just the inverse of BE; however, that only works
for rank space, as we have the guarantee that there is an endpoint in each x-coordinate.
Furthermore, we denote the bit arrays within the wavelet trees as BL(v) for a given node in
YL and BR(v) for the equivalent node in YR and vice versa.
The segment wavelet tree supports segment-select(i,j) queries [15]. For a query, we
rank on the bit array of the correct node in both YL and YR. The reasoning is as follows.
Rank queries on YL only provide information on segments starting until i (because the left
endpoint is to the left of i). With YR, we have information on how many segments are ending
before i (because the right endpoint is to the left of i).
We navigate each node using locally computed variables lv, rv, and jv. lv denotes the number
of segments in v with a left endpoint up to i. Similarly, rv is the number of segments in v
with a right endpoint up to i. The difference between lv and rv is the number of segments
with a left endpoint to the left of i but not having a matching right endpoint to the left of
i. Intuitively, this is the number of segments that start but do not end at time i, meaning
segments that cross time i. It should now be obvious why we cannot reduce to a space
smaller than rank space as mentioned in section 2.2.2. If we do that, we would end up with
segments being points, and then they would count for both lv and rv any time they are
counted at all, meaning that for the difference between them, they would just disappear.

Arbitrary Degree Segment Wavelet Tree
We now generalize the segment wavelet tree to support an arbitrary degree. By degree of a
tree, we mean the maximum out-degree of any of its nodes. We denote this data structure
arbitrary degree segment wavelet tree (DSWT). First, we generalize wavelet trees to support
arbitrary degree. We introduce the X-bit array to do this. Afterward, we expand the segment
wavelet tree segment-select(i,j) query to the new structure.
The number of rank(B,b,i) queries on the wavelet tree is bound by the height of the tree.
Since the tree’s height is logarithmic in the number of versions, we can increase the base of
the logarithm. Therefore, this will not yield a better asymptotic query bound, but we expect
it to perform better in practice.

2.4.2
Arbitrary Degree Wavelet Tree
Now that we have a way to represent more than two symbols, we can generalize the wavelet
tree to support arbitrary degree. We call this data structure an arbitrary degree wavelet
tree. The structure is similar to that of the wavelet tree. The arbitrary degree wavelet tree
supports rank queries on a sequence S[1, n] as defined in definition 2.6.
We denote the degree of the tree by D. Each node in the arbitrary degree wavelet tree has
an X-bit array. Since we need to be able to represent the chosen degree D, we have that
X = ⌈log2(D)⌉so that each X-bit in the X-bit array is able to hold the D values.
We construct the tree from the root and partition the symbols into D subranges based on
the alphabet’s order. The X-bit array in each node encodes each symbol in the sequence by
0, . . . , (D −1). We then proceed as in section 2.3.2. Given that σ > D, there is ambiguity at
the top of the tree, and we can not infer how many of each symbol there are.

3.3
Version Tree
We need to create a more explicit VT for the following models than what we saw in the
section 3.2.2. The reason for this is that we need to traverse it in a more intricate way with
an Euler tour than the simple node-to-root path we saw earlier.
We do this by reading the file version_tree.txt line by line. We have a stack of nodes along
with their ID and their parent’s ID on which we push the nodes as we create them. The
nodes consist of a vector of children, a parent, and an edit operation. We start by creating
the root node and pushing it onto the stack. We then read through version_tree.txt and
create a node for each line. We connect the new node in both directions to the node on top
of the stack and then push it onto it. When the parent of the line we are reading is not equal
to the top of the stack, it must be because we no longer follow a path and jump to the next
node in the preorder tour. This node must be found somewhere above in the tree as we are
at a leaf. We will then pop the stack until we find their lowest common ancestor, which will
be the parent of the newest node we are looking at, and we will then start a new path from
that one. We continue this until we eventually try to pop the root off the stack, and then we
know we are done as we have been through all of the nodes in the tree.
3.4
Segment Selection Instance
Part of preprocessing for all variants of the segment wavelet tree is the reduction from a
version tree into a set of segments. We call this the segment selection instance (SSI) as in [3].
The main output of this class is the final version of ¯S and a vector of segments. We use an
iterative approach to transform the VT into the segment selection instance. The reason for
this is that it allows us to create the instance from much bigger trees as we are not limited
by the size of the call stack like we would be if we were to do it recursively.
To do the tour iteratively, we keep track of where we are by using an offset array O of the
offsets for the levels and an index k into the array. The k represents where we currently are
height-wise, and O[k] is the child we are exploring at the kth level. We navigate as follows:
We start at the root node and then perform the Euler tour. We go down as far as we can
using only the nodes’ zero children, incrementing the k variable. When we go up we set the
current O[k] = 0, decrement the k and increase the new offset value. In this way, it means
that coming up from the child of a node, we will then be looking at the next child of the node
if we were to go down again; if it does not exist, we go up again, and if it does, we go down.
By doing this, we perform the Euler tour iteratively. See fig. 3.5 for an example of this.
We are either going down or up for each iteration of the traversal. For each of these, we
are either performing an insertion or deletion or coming back from performing an insertion

We will be using four bit-wise operations on the ints in BCpp, left and right bitshift, as well as
the bit-wise OR and AND operations. The left bitshift operation, denoted ≪, shifts the bits

index of these ints in BCpp as first_int and second_int. To get an X-bit at index b_index

much faster operations on a computer. The compiler might do these optimizations for us,
but we do them explicitly to ensure they are done.

is contained within a single int. Otherwise, the X-bit spans two ints. For sizes of X where

3.6.1
Block Rank Cumulative
The rank query is a vital operation that plays an important part in the analysis, as we will
perform one rank query per layer in the tree for an access(v,j) operation. First, we first
present a very simple approach. After that, we present an approach that is both fast and
space-efficient.
==PAGE:45==
3.6 Model 3: BSWT
38
One simple way to rank on a bit array is to scan over the bits and count the number of bits
set to 1. This could is however not very efficient as it is O(p) query time. We have made an
improved version of the linear scan which can be found in appendix E however it is only a
constant factor better.
The block rank cumulative (BRC) rank structure uses a block rank scheme. We use the ideas
from [12] with blocks to decrease the number of steps. The idea is to partition the bit array
into partitions covered by super-blocks and then these partitions further into subpartitions
covered by blocks. For every 4096 symbols, we save the result of the rank query for each
symbol in a super-block. In the binary case where the symbols are 0 and 1, we could settle
with only precomputing the rank for one of the symbols, as the other can be calculated by
subtracting the result from the total number of symbols. However, we will use this scheme
for arbitrary alphabets in section 3.7 to precompute the rank for all symbols.
For each super-block, we save some smaller blocks for every 512 symbol with a value relative
to the super-block, which means there will be 7 blocks for each super-block. This is because
4096
512 = 8, but we do not need to save a small checkpoint on top of a large checkpoint as we have
that checkpoint already. Each block must be able to represent up to 4096 −512 + 1 = 3585
symbols, as that is the maximum number of symbols before another super-block would have
an updated absolute result. The plus one is there because we also have to save the zero value
in case there are none. This can be represented with ⌈log2(3585)⌉= 12 bits. Since we have a
total of 7 blocks for a super-block, we will use 7 · 12 = 84 bits to store these. We want to be
cache efficient, so we would like the space we are using to line up with the cache line which
according to appendix D is 128 bits. Using 128 bits, this leaves 44 bits for the super-block.

3.6.2
Query Robustness
Suppose that we do a segment-select(i,j) where j is either greater than the maximum
y value in L, which we will denote max_y or there does not exist j crossing segments at
==PAGE:48==
3.7 Model 4: DSWT
41
time i. Due to the behavior of the underlying wavelet trees, if we query out of bounds with
regards to j the model returns the segment with max_y value.
Consequently when max_y is returned from a query it is ambiguous and it is not clear if the
corresponding segment is a legitimate answer or a result overshooting the number of segments
at this time. To remedy this behavior, we apply an artificial ceiling in the form of an additional
segment lc = (maxx +1, maxx +2, max_y +1), where max_x denotes the maximum x value
in L. By using this extra segment we ensure that when max_y is returned from a query it
is the legitimate result. Furthermore if we get segment-select(i,j) = max_y + 1 then
we know it is not a correct answer.

Model 4 is based on the findings in section 2.4. The SSI transformation is the same as in
section 3.6. We expand on the construction presented in section 3.6 to construct wavelet
trees with arbitrary degrees. We now use the X-bit array instead of a regular bit array since
we represent more than two symbols. The main differences with the construction to obtain
arbitrary degree are twofold. We now encode D different values in the bit arrays of the nodes
that we use to route in the tree. Additionally, we change how the slices are partitioned to
maintain balance in the tree. The balance property of the tree is crucial since we depend
on cache efficiency within the nodes to off-load some of the extra computation trade-offs
there are with arbitrary degree. Regarding balance, we stay consistent with section 3.6 by
prioritizing the right skewing of segments in each node where the segments can not be split
evenly. If a node has three segments, the right child will contain two segments and the left
child a single. Let us suppose we have degree D for all nodes in the arbitrary degree wavelet

3.8
Model 5: FDSWT
Model 5 is also based on section 2.4. We merely optimize the representation of model 4. In
the previous section, we saw that the wavelet trees consisted mainly of nodes consisting of an
X-bit array. However, there is a more efficient way to store the data. We are almost certain
to incur a cache miss every time we traverse from one node to another in the trees, as the
nodes are most likely located far from each other in memory. Furthermore, we incur a lot of
==PAGE:55==
3.8 Model 5: FDSWT
48
Figure 3.15: Model 4 with degrees 16 and 32 using BRC and BRCA.
memory overhead as we have to store an object as each node and the pointers to the children,
the X-bit array, the rank structure of the X-bit array, and all the associated "startup costs."
Instead, we can store each wavelet tree as a single X-bit array. We will refer to this as the
flat segment wavelet tree with arbitrary degree (FDSWT). The idea is that since we know
each tree’s topology, even if we store it in a single X-bit array, we can compute the indices
of the part of the X-bit array that corresponds to a node in the tree. It is, however, vital
that we can rank in constant time as we now get an X-bit array that is much longer than
previously. If the ranking were in any way related to the size of the X-bit array, we would
not be able to achieve the same query time as before. However, since it is not, we can, when
we have found the range of a node, do the rank queries as usual and then subtract a rank
query at the beginning of the range to get the relative rank, which would be equal to a rank
query in a node of model 4.
For trees where we can fit more than one level in the cache, we can reduce the number of
cache misses when we rank. The test machine does not have an L3 cache; however, the size
of its L2 cache is 24MB [1], meaning if we were only processing the X-bit arrays, we expect
a potential speed-up for data sets of up to slightly below 96 million versions. This is because
24MB equates to 192 million bits, and each layer of each of the trees has one bit for each
version of the data. When going left in the tree and ranking, we would potentially access
bits just over one layer in the tree apart. We expect the speed-up to be more significant for
data sets smaller than 96 million versions as more of the tree would fit in the L2 cache and
lower-level caches.
According to [13], typical L3 sizes, which are more commonly the largest cache on a computer,
==PAGE:56==
3.8 Model 5: FDSWT
49
are up to 8MB, so the speed-up would also be very machine-dependent. However, we expect
a speed-up for data sets up to slightly below 64 million versions, even on these machines.
Furthermore, as we store all the nodes for each tree in a single X-bit array, we also reduce
the space overhead incurred by the large number of objects with pointers and other auxiliary
data in the DSWT when using the FDSWT.
We could build the FDSWT from the SSI and VT as we do with the DSWT, but now that
we already have the DSWT, we will use that as a starting point. The main complication
of transforming the DSWT into an FDSWT is moving the X-bits from the X-bit arrays in
the tree nodes to the single X-bit array in each of the wavelet trees of the FDSWT - and
modifying segment-select(i,j) to work accordingly. We could calculate the indices of
the nodes in the X-bit array as we perform segment-select and traverse the nodes. However,
it is only an easy calculation if the tree is complete. Therefore, to save query time, we instead
build an index to look up the positions of the nodes in the wavelet trees of the FDSWT.
Even though some cache misses might still occur when we look up the positions in the index,
they should still be minimized due to the cache size of the test machine and typical machines,
as stated earlier. Since the index would share the cache with the large X-bit arrays of the
wavelet trees of the FDSWT, the speed-up mentioned earlier would occur for trees of smaller
sizes than mentioned. But in general, it seems there would be a speed-up on most machines
for a decent-sized dataset.
To build the index and transfer the X-bits, we want to do a level order traversal to move the
X-bits of the nodes to the single corresponding X-bit array. To build the FDSWT, we do
the following for each wavelet tree. We use a queue where we start by pushing the root of the
tree. We then repeatedly pop a node from the queue, push its children onto the queue along
with their position in the index, and save the current length of the new X-bit array in the
index. Lastly, we move the X-bit array from the node to the end of the X-bit array of the
new 1D representation, which means that the length we saved before in the index now refers
to the position where the X-bits of the node we just copied start in the single X-bit array.
We can calculate the access indices using row-major order, which means that for any index
node with position p in the index, we can calculate the position of its children using the
following formula:
ci = p · d + (i + 1)
(3.8)
where ci is the ith child of p and d is the maximum out degree of the tree.
The only thing left is how to do segment-select(i,j) now that we have all the necessary
structures. Most of it is comparable to section 3.7; however, there are a few key differences.
Firstly, we use an iterative approach instead of a recursive one, as the recursive method is
==PAGE:57==

less efficient and does not benefit from the natural data orientation of a pointer-based tree.
Secondly, for any rank inside a node in the array, we are technically ranking from the start
of the whole array, so to simulate ranking only from the beginning of the node, we also rank
to the beginning of the node and then subtract that rank from the rank we get from the
rank query we would typically do. Then, to find the next node, we do not follow a pointer;
instead, we use eq. (3.8) to calculate the position of the bucket in the index that holds the
position of the child node in the X-bit array that we are proceeding to.
The way we determine the y coordinate is slightly different compared to the DSWT. Because
the bottom layer of the tree is not necessarily full, as we do not restrict it to complete trees,
we stop at the second to last layer. When we are at the second to last layer, we have that

would have taken to the last child. We do this by adding b + 1 where b is the child we would
go to and 1 as segment select is 1-indexed. In fact, we do not need to store the last layer as

Chapter 5
Conclusion
In this paper, we have presented the problem of random access in persistent strings (RAPS).
We have presented the relevant theory to reduce a version tree over a persistent string to
a segment selection instance [3]. Furthermore, we have introduced the theory pertinent to
solving the segment selection problem and, thus, solving the RAPS problem. The primary
data structure of our solutions to segment select builds on the segment wavelet tree introduced
in [15]. We have also generalized the segment wavelet tree and relevant operations to an
arbitrary degree using the X-bit array, which allows "bits" of variable size.
We have implemented five solutions in C++ to solve the problem. Models 1 and 2 provide
loose bounds on the problem. They perform access(v,j) in O(1) time and O(n2) space,
and O(n2) time and O(n) space respectively. The models 3, 4, and 5 solve access(v,j) in
O(log n) time and O(n) space. Model 3 uses the binary segment wavelet tree [15] (BSWT).
Model 4 uses our generalization to the arbitrary degree segment wavelet tree (DSWT). Lastly,
we changed the structure of the DSWT to optimize memory usage in the form of the flat
arbitrary degree segment wavelet tree (FDSWT).
We have benchmarked all the models and found that model 1, although the fastest, is not
scalable past 210K versions on the test machine. Model 2 is the most scalable memory-wise;
however, depending on the use case, it may be unable to query sufficiently fast at large inputs.
Using model 3 provides a good scalable baseline solution to the problem. Model 4 improves
on model 3 and is the fastest model for smaller inputs; however, it is not scalable past 1.1M
versions on the test machine. Lastly, model 5 provides the best compromise between query
speed and space usage, making it much more scalable than model 4 while being almost as

fast.
5.1
Further Work
In this section, we look at some of the possible optimizations for our models and some natural
areas into which our work could be expanded.
5.1.1
Optimizations
There are a few optimizations to look into to optimize our codebase. An easy optimization
is not explicitly saving all the nodes at the bottom of the tree where we no longer have
ambiguity. This will reduce the space usage quite a bit, particularly for the DSWT, as the
bottom layers have the most nodes, and each node has overhead in terms of pointers and
other auxiliary structures.
Another thing to look at is the preprocessing. In this paper, we have focused primarily on
optimizing the query time and space usage as much as possible; however, for real-world
applications, it could make sense to look at optimizing the preprocessing, depending on the
use case.
For model 5 in particular, it could make sense to look at the index we are currently building
to find the nodes in the structure. An interesting thought is to calculate it along the way
instead and compare whether or not that would be faster. It might be faster because the
index may incur some cache misses.
Lastly, another thing to look at is for the SWTs to be left-skewed instead of right-skewed
(as showcased in fig. 3.7b). It would allow us not to have to allocate the entirety of the last
layer (which is the biggest) for the FDSWT. Some more computation could also do this
with our current model; however, that would slow down querying and make the code more
complicated. Furthermore, left skewing is a much more natural way to think of the skewing
and would make the code more intuitive.
5.1.2
Expansions
Our models have focused much on being input-independent. No matter how the version tree
looks, they perform roughly the same. Therefore, an exciting area to explore is how to do
something similar but with a focus on the structure of the input. Specifically, one could
tailor the solution to the use case of the problem. For example, if we were trying to save the
==PAGE:72==
5.1 Further Work
65
history of someone writing a text document, we expect that most edit operations would be a
character added to the end of the string. For as long as that is the case, the position of a
character in a string is also equal to its version. I.e., the first character inserted is version 1,
the second character inserted is version 2, and so on. In this way, we could achieve constant
time look-up and linear space. This solution could be implemented for the parts of the string
where it is the case, and then a version tree with a segment wavelet tree of some type could
be used for the rest of the string.
Another interesting problem is how one might efficiently return a substring of the string
at version v instead of just the jth character. Right now, if one were to do that with our
solution, one would have to do a segment-select(i,j) for each character, leading to a
running time of O(log n · m) where m is the length of the substring. Notably, it could be
interesting to see if it is possible to achieve O(log n + m) time so that once the first character
is found, there is a link to the rest of the characters at that time.
Lastly, the wavelet matrix of [4] appears exciting to explore. According to the paper the
wavelet matrix is significantly faster for large alphabets compared to the input size. For our
use case, the alphabets are the y coordinates that are all unique, which means we have that
σ = n, which is the largest possible alphabet. Therefore, it would be interesting to see if
we could use the wavelet matrix instead of the segment wavelet tree and if it is significantly
faster for our use case. Furthermore, it is also interesting to see if it is possible to generalize
the wavelet matrix to simulate an arbitrary degree tree.

Appendix A
Succint Data Structure Library
Models
We have also tested models 3, 4, and 5 using the cutting-edge Succinct Data Structure
Library (SDSL) [10] for out-of-the-box bit arrays and ranking. We use the bit_vector class
with a rank_support_v rank for all the models.
For model 3, replacing our bit arrays with SDSL one-to-one is pretty straightforward. We
need another abstraction for models 4 and 5 to rank more symbols. While one bit_vector
can be used to rank for two symbols, we cannot simply use two to rank for four symbols and
so on. This is because when ranking for specifically either 0 or 1, we only know the zeroes
based on them not being ones. When we have two bit_vectors to represent the four symbols,
we can only specifically rank for two, and the others would be absent.
Instead, we use the very simple abstraction, which we call an SDSLXBitArray, where we
have an SDSL bit_vector and rank_support_v for each symbol. Therefore, ranking the
SDSLXBitArray for a specific symbol is done by ranking the bit_vector and rank_support_v
corresponding to the symbol. We also use some simple getter and setter functions since we
only need those during preprocessing, not querying. The getter works by looping through all
the bit_vectors and returning the index of the one set to 1. We set a symbol by setting all
the bit_vectors to 0 and then setting the one corresponding to the symbol to 1.
The SDSLXBitArray is not very space efficient as we save a bit_vector for each of the
symbols, nor is it very fast for getting/setting as we straightforwardly implement those

behavior of the rank for SDSL.

Appendix B
Data Generation
We generate the data using an iterative approach to generate the version tree, where we
write the data to a file such that reading the file would correspond to a preorder tour of the
version tree. We write only the changes to the file as writing the complete file would take a
(relatively) very long time and take up a lot of space for the large data sets. The data itself
is generated randomly and emphasizes creating trees that, in general, are very tall, i.e., that
have a long path and then relatively long paths branching out from the long path. This is
because we thought it relevant to simulate the kind of data that comes from entering a lot
of text, backtracking, and going in another direction. We randomly decide where to apply
the edit based on the previous string, ensuring we cannot delete a character from an empty
string or insert a character several places after the last character on a string.

Appendix E
Block Rank
The block rank (BR) scheme as a rank structure improves on the simple scanning scheme
by precomputing rank queries at predetermined block sizes. We divide the bit array into
blocks and then precompute the rank spanning these blocks. To rank, we then sum the
precomputed rank of the blocks that span the interval we want to rank over. If the interval
does not line up with the blocks, we scan the bits themselves the rest of the way.
Furthermore, we can have several levels of precomputed ranks where each is more fine-grained
than the next. We can then sum the precomputed ranks on the coarsest level as far as we
can, move a level down, and sum the precomputed ranks as far as we can, and so on. Doing
this decreases the number of blocks we need to add up. The more levels we add, the more
space we use, but the fewer blocks we need to add up.
This is still O(p) query time; however, it decreases the constant as we are not scanning the
whole bit array. The scheme might still be fast on relatively small bit arrays since it is a
cache-efficient way to rank while using a relatively small amount of memory.
From fig. E.1, we can see that the cache-optimized block rank cumulative (BRC) approach
both with (BRCA) and without all symbol rank performs better than the block rank approach
as expected.
