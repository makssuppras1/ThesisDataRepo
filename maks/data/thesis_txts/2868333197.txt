CHAPTER 1
Introduction
In complex, distributed systems, understanding and ensuring the correctness of software be-
haviour is a complex challenge.
Formal Specification-Driven Development (FSDD) [1] combines the method of Test-Driven
Development (TDD) with formal methods. In FSDD, a formal specification is needed to generate
unit tests and some parts of the implementation.
However, writing formal specifications is associated with a steep learning curve that discour-
age programmers from using formal methods in practice. Developers working with Elixir lack
modern, automated tools that can provide insights into their systems without requiring deep
expertise in model checking.
To support programmers in using an approach similar to FSDD, Steenman, E. [2] defined a
high-level language, which is automatically transformed into both a formal model and an imple-
mentation in Java. The model is implemented in the formal specification language mCRL2 that
is designed for specifying, analysing, and verifying concurrent systems and protocols. Steenman
demonstrated that using this high-level language to create formally verifiable code is feasible for
simpler scenarios, such as a system controlling traﬀic lights.
In this project, we developed Steenman’s approach further by focusing on distributed pro-
tocols. Concretely, our research can be summarized in the following question: To what extent
can a high-level language be developed to specify distributed protocols that enable the generation
of both an mCRL2 formal model and an Elixir implementation with identical behaviour, while
addressing the challenges of language design, formal model simulation and implementation fi-
delity? This question has multiple aspects: How can the high-level language be designed to
facilitate the generation of both the formal model and the implementation? How can the formal
model be designed to simulate real distributed systems while also making it easy to analyse
the system behaviour? How can the implementation be designed in such a way that the pro-
gram’s behaviour follows the rules of the formal model? And how complex can the protocols
this approach supports be?
During the project, we developed a high-level domain-specific-language (DSL) embedded in
Elixir to specify distributed protocols with special Elixir syntax. The specified protocols are
based on different processes, each containing multiple states. In each state, the processes may
receive certain messages that trigger behaviours specified by using different commands. These
commands are accessing and modifying data in the processes (like send message command,
if-then-else command, etc.).
As the behaviour of Elixir programs and mCRL2 specifications are fundamentally different,
we made it easier to compare their semantics by separating the control flow (processes, states,
receiving messages) and the data flow (other commands) in the system. In this semi-formal
approach, we mapped the behaviour of the protocol states and their changes to a nondeterminis-
tic automaton, making the foundation for specifying the same protocol behaviour in Elixir and
in mCRL2. After that, we compared the behaviour of the different commands in both output
languages, showing that they modify the data the same way.
After the input and output languages are defined, we implemented the input DSL using a
library called Spark and the logic to generate the output Elixir and mCRL2 files. The tool
currently supports protocols that contain basic instructions like sending/receiving messages,
broadcasting messages, and structures like if-then-else or internal choices. The choices are made
==PAGE:6==

by the mCRL2 model in-deterministically or made by the user running the Elixir implementation.
To communicate with the user, an interface is designed for Elixir.
The generated formal model can be analysed by configuring different environments. The
underlying network layer can be specified to enable the possibility of losing messages or swapping
the message ordering. Processes can also be configured to enable the possibility of crashing.
With these options, a protocol specification can be tested against these different scenarios.
To evaluate the features of the tool and justify the approach, we specified and analysed
three different protocols in increasing complexity: addition, two-phased commit, and simplified
RAFT protocol. We demonstrated how the mCRL2 toolset might be used on the generated
formal specifications to analyse how the systems behave with different configurations. As an
example, we showed that our RAFT specification is able to elect leaders even if some nodes
crash. We also show how nodes can be generated in Elixir and how they can be configured to
run in a distributed environment.
This approach advances our overarching goal of helping developers produce industry code
supported by formal methods. By generating formal models together with the executable code,
the developers may gain a deeper understanding and ensure the reliability and correctness of
their systems, leading to more robust and dependable software.

CHAPTER 2
Background
mCRL2
mCRL2 is a formal specification language with an associated toolset. The language is based
on process algebra that combines actions to describe different behaviours.
To describe the
concepts used in this project, we refer to the descriptions and examples in the mCRL2 language
reference [3].
The main operators are sequential composition, the dot ”.”, which puts two behaviours in
sequence, and the alternative composition, the plus ”+”, which represents the choice between
two behaviours.
1 act a, b, c, d, e;
2
3 proc P = a . b . c;
4
Q = d + e;
5
R = (a + b) . c . d . e . R;
6
7 init P || Q || R;
In the above example, the actions ’a’, ’b’, ’c’, ’d’, ’e’ are defined together with processed ’P’, ’Q’
and ’R’. In process ’P’ actions ’a’, ’b’ and ’c’ can happen in sequence. In the process ’Q’ there
is a choice between actions ’d’ and ’e’. Finally, the process ’R’ consists of first doing a choice
between ’a’ and ’b’, followed by a sequence of ’c’, ’d’ and ’e’ and a recursive application of the
same process. The keyword ’init’ indicates that the process starts with a parallel composition
of the three processes.
A multi-action is a sequence of actions that happen at the same time. They are separated
by bars. E.g. ’a|b’. The empty multi-action is written as ’tau’. It is conventionally called hidden
or internal action that can not be observed.
Conditions can also be defined in the form of ’c -> p <> q’ to achieve the behaviour of “if
c then do process p else do process q”.
The sum operator allows to formulate the choice between a possibly infinite number of
processes. The process ’sum n: Nat . p(n)’ is equivalent to the process ’p(0) + p(1) + p(2) +
....’.
Each process and actions can carry data, for example, ’proc Clock(n: Nat) = tick(n) .
Clock(n+1)’.
Communication is also supported with the mCRL2 syntax. That is described in detail in
the project in figure 5.2.
Elixir
Elixir is a dynamic, functional language for building scalable and maintainable applications.

structures, and even function clauses. We can provide multiple function clauses with different
guard expressions (after ”when”) for the same receiveMessage function. When the function is
called, the first function clause matching the guard condition will be executed. Similarly, the
function arguments are also matched. A behaviour similar to type checking can be achieved
when using type structs on the arguments.

GenServer (Generic Server) is a behaviour module in Elixir used to implement the common
client-server interactions between processes.
A GenServer would typically consist of a client part, which can be used by external processes;
and the GenServer callbacks, which handles incoming messages. A GenServer keeps track of a
state, which can be updated by the return value (the last expression in the function block) of
the message handler functions.

List of contributions
Here, we summarise what has been done in the project.
1. We designed a domain-specific language to define high-level specifications of distributed
protocols.
a) A protocol may consist of multiple processes, each with multiple instances.
b) Each process has multiple states that define what messages can the process receive
at certain points during the protocol execution.
c) Each process may consist of the basic instructions of sending/receiving or broadcast-
ing messages, if-then-else statements and indeterministic choices.
d) The specification supports some basic data types, such as Natural numbers, Booleans,
lists and structs, as well as the basic operations on these data.
2. Based on the protocol specification, we designed corresponding Elixir code in the following
way.
a) Each process follows the protocol specification and can communicate with other pro-
cesses.
b) To handle interaction with the user during start-up and indeterministic choices, a
separate API module is also defined for each process.
c) The generated code can be executed in a distributed environment, without modifying
the generated code.
3. Based on the protocol specification, we designed a corresponding formal specification in
the following way.
a) Each process follows the protocol specification.
b) The processes may communicate with each other through a network layer.
c) The network layer can be configured to enable the possibility of losing messages or
swapping the message ordering.
d) The processes can be configured to enable the possibility of crashing.
e) Additional state labels (mCRL2 actions) can be added in the protocol specification
to be used in mu-calculus formulas.
f) The generated formal specification can be used for model checking, without modifi-
cation.
4. We implemented a tool to generate the aforementioned Elixir code and mCRL2 specifica-
tion based on the high-level protocol specification.
5. We defined and analysed 3 different protocols to showcase the strengths and weaknesses
of this tool and the approaches chosen.

CHAPTER 4
Designing the DSL
As an input, a way to specify protocols has to be designed. Based on the assumption that
the same users will specify the protocols and execute their implementations (referenced as ‘user‘
hereafter), a Domain-Specific Language (DSL) embedded in Elixir has been selected as the input
language. This choice ensures that the syntax remains familiar to Elixir users while providing
built-in parsing capabilities, making the implementation easier.
The DSL is made with Spark [4], an Elixir library that helps to build simple DSLs that
are extensible and well documented. Syntax can be easily defined with schemas and they also
offer features for validating and transforming the input.
From the DSL, Spark generates a
struct-based data model in Elixir automatically upon compilation.
Despite Spark being the only available Elixir library to write DSLs, alternatively, one could
utilise Elixir’s robust macro system directly to achieve similar functionality. The implementation
of the DSL using Spark proved to be a straightforward process. Modifying and expanding the
language is simple and does not require a comprehensive understanding of the entire system,
which might be required when extending custom macros. It also provides autocomplete and
inline documentation in the supported text editors ”out of the box” (see figure 4.1). However,
the rigidity of certain Spark concepts led to some constraints in the DSL, as elaborated in section
7.2.2.1. Additionally, unfortunately the library is only partly documented and as such, some
functionalities are hard to understand at first sight.
The DSL structure relies on Spark’s Sections and Entities.

4.2.5
Behaviour
The specified protocol does not have any semantics on their own, as the purpose of the lan-
guage is purely to provide a descriptive structure of what the processes consists of, using the
commands listed before. Each of these commands follows the conventional behaviour of popular
programming languages.
However, the behaviour of the states, data and other details are discussed in this section to
give the users an overview on how to define protocols and what the expected behaviour looks
like after the structure gets transformed into the two different models with different semantics.
4.2.5.1
Combining entities
Entities can be written one after the other. For processes, that denotes a parallel, for receive
and timeout commands an alternative, and for the other commands a sequential composition of
their actions. As a general rule for commands, the sequential execution follows the commands
in their children block before the next command placed afterwards.
4.2.5.2
States and transitions
The DSL’s structure is based on the intuition that the protocol behaviour can be simplified as
a nondeterministic automaton (Q, ∑, δ, q0, F).
==PAGE:20==
16
4 Designing the DSL
• Q = the states of the process;
• ∑= the messageType;
• δ = the values of the state! commands (resulting states) mapped from the product of
the messages satisfy the parent receive command’s condition (transitions) and the parent
state (current state);
• q0 = the state set in the initial block;
• F = ∅.
This automaton essentially defines for each state:
• The set of possible messages that can occur in that state.
• The resulting states reached after each of these messages.
The simplification relies on abstracting the state parameters away. We can show that this
way, the automaton remains unaffected by the commands’ behaviour, except the receive and
change state commands.
Indeterministic choices and if-then-else structures does not interfere with the automata:
1. The if-then-else can be treated as an indeterministic choice, as its condition depends solely
on variables.
2. Their behaviour only affects which case is executed, containing different commands. In
the children, receive commands are not allowed, so we only have to consider change state
commands.
3. The structures can only appear as children of receive commands. Consequently, all change
state commands appear in δ, regardless of the different cases.
It can be trivially shown that the other commands, namely send!, broadcast! and label! do not
interfere with the automata.
Although this abstraction might seem excessive, this state machine provided the fundamental
basics of how the Elixir and the mCRL2 output can behave the same way. As the core semantics
of communication differ in the two output languages, this automaton provides clear instructions
to what messages are expected to be received in the system. On the contrary, the behaviour
of the commands and data operations can be more easily mapped between the two output
languages.
4.2.5.3
Parameters
Each process has parameters where parameter names are mapped to values. This state may
only change upon changing protocol states.
All parameters are constant and can be accessed based on where they are defined:
1. The arguments of the process can be used in the whole process.
2. The arguments of the states can only be used within the state’s children commands.
3. The receive commands may add additional constants for the received message and the
sender’s identifier. These can only be used within the receive’ children commands. Similar
rule applies for the indeterministic choice’s value.

Designing the generated
output
The generated output, namely the mCRL2 formal model and the Elixir implementation has to
be designed so that the Elixir code’s behaviour follows the behaviour of the generated formal
model (i.e. there exists an execution for all possible paths in the formal model). This way,
whatever property is proven with the formal model can apply to the implementation, leading to
more robust and reliable industry code.
This requirement is hard to prove formally, as Elixir does not have formal semantics defined.
However, as we separate the control flow (processes, states, receives) and the data flow (other
commands) in the system, it is easier to compare the different models.
In both languages, the code’s structure is based on an automaton defined before in section
4.2.5.2, which acts as a fundamental base of how the runtime states change according to different
incoming messages. We define the output file’s structure for the two output models separately
in sections 5.1 and 5.2 and show how it maps to the automaton in question. As a result, we can
expect that the states in regards to incoming messages are working the same way.
Afterwards, we show how the data flows the same way, which means we can also expect that
the messages with the same data are sent in the different states in both output models. We
compare the commands’ mCRL2 and Elixir translations one by one in section 5.3. We show
that the commands behave the same way, given that an additional rule, described below, holds.
As we combine the rules for the control flow and the data flow, we can expect that, by design,
the incoming messages systematically modify the runtime state while triggering appropriate
outgoing messages.
However, these rules do not guarantee the successful reception of sent
messages.
An additional safety property must be introduced (see figure 5.1) to check that
all incoming messages can be received after they are sent. This property however can not be
defined in the general case and it is slower to verify than most other properties we used during
evaluation.
Alternatively, a simpler property can be formulated by modifying the model to
enforce this behaviour directly. This option is investigated and described in 5.1.9.
5.1
The structure of the mCRL2 specification
Before we get to the mCRL2 structure, we take a look at why mCRL2 is an ideal candidate for
modelling the protocols.
mCRL2 is an open-source formal specification tool that has a focus on concurrent systems.
Its process algebra is based on sequential processes that can carry data parameters, but it also
supports alternative composition, which we are using for indeterministic behaviour. mCRL2
has an easy to use concept to model communication between parallel processes, which was
necessary to model distributed protocols. A linear process can be easily generated from the
models, which makes the representation easy even for systems with infinite state space. mCRL2’s
system analysing features comes with an extensive toolset to generate and work with different
representations of the system. For modal checking, it uses µ-calculus, the modal µ-calculus [5]

5.1.1
Actions
An mCRL2 action represents an event of any kind.
In our case, these might be predefined
actions, such as sending or receiving messages, losing messages in the network, crashing and
resuming processes or timeouting. There might also be custom state labels that are referenced
in the label! command (see section 4.2.4.7). An example for a custom label is the ’exposeLeader’
action in the Raft protocol (see section 7.2.1), which is used to examine and detect when the
leader is elected.

referenced as a state label to express a final state of a process, after the protocol executed as ex-

state of the whole system. The ’emptyNetwork’ action is added automatically to the network
process to denote a state when there are no messages in the network.

single ’done’ action. This behaviour is elaborated in listing 5.2.

colDone’, together with the ’emptyNetwork’ action are blocked as single isolated actions, whereas

they are specified as a ’doneRequirement’.

5.1.4
Communication
As we are focusing on distributed systems, a crucial part of the specification is how communica-

Similar actions for broadcasting are also defined, as an alternative to sending single messages.
Sending messages through broadcasting uses a list of recipients. The network layer divides this
into separate messages for each receiver.

5.1.7
Crashing and resuming processes
Crashing nodes are also simulated by another mCRL2 process. The mCRL2 execution might
choose to select the ’crash’ action any time instead of receiving messages. In that case, the
protocol process will ignore all messages until it is recovered. In practice, the messages are still
received, clearing the messages from the network layer, but there are no actions associated to
the received messages, the process just stays in the crashed state. Just as crashing, recovering
from crashes is encoded as an indeterministic behaviour. The related action is called ’resume’,
after that the protocol state is restored. The code shown in listing 5.7 is generated for each
protocol state.
This implementation of recovery is a very simple abstraction of real behaviours. In real
systems, recovering from a crash might trigger special recovery algorithms, as well as certain
data might get lost after crash. These scenarios are currently not supported in the system,
however, crashing seemed to be valuable to analyse certain system properties.

• Q = the processes generated for each state;
• ∑= equivalent mCRL2 type or struct of the ’messageType’;
• δ = calls to new processes (resulting processes) mapped from the product of the messages
satisfy the conditions before the ’receiveMessage’ action defined sequentially before the
calls (transitions) and the parent process (current state);
• q0 = the process called in the initial process;
• F = ∅.

To check that any message coming from the network can be received at any point after they

This approach adds additional actions to the model, slightly increasing its complexity. On
the contrary, the message receiving actions will by design enforce the model to receive any
message in the network or otherwise result in a deadlock. The occurrence of that deadlock

5.2
The structure of the generated Elixir code
Elixir is chosen as the language for the implementation of the protocol due to its robust fea-
tures for distributed programming and its functional nature, which aligns well with the mCRL2
specification.
A requirement for the Elixir files is that each process has to be able to compile and start
independently. In order to communicate between each other, the node addresses have to be
passed in their arguments when configuring the running environment. They accept any address
that is supported by ‘GenServer.cast/2‘.
All nodes utilize the GenServer behaviour, which is typically associated with servers. In
our case, this uniform approach has its benefits. The DSL and the Generator do not have to
differentiate between servers and clients, which also simplifies the case in some protocols where
the nodes are equal or the server node can change (like leader-election protocols). Furthermore,
the GenServer behaviour is a well-established concept in Elixir, which enhances the debugging
capabilities. However, the GenServers imply the usage of a message-driven system: after the
initial setup, all system actions are triggered by an incoming message. This also aligns well with
the automaton’s behaviour (see section 4.2.5.2).
In process algebra, a common practice is the alternative composition, which represents the
choice between different behaviours. We wanted to support this alternative choice even though
Elixir has no concept for indeterministic behaviour like that. Instead, we propagate the choice
to the users.

5.2 The structure of the generated Elixir code

To separate the communication with the user and the communication with other processes,
each process is split up to two different Elixir modules. An API module communicating with
the user and a protocol module, implementing the protocol communication.
The Elixir generation also adds some logging statements in the code to show what the
program is doing. These statements are omitted from the code snippets in this document.
5.2.1
API module
The trivial approach to handle communication between the protocol and the user is to use direct
function calls from the protocol to an interface, where the user can define the return value. This
seems straightforward from the protocol’s point of view: whenever it needs a choice from the
user, it asks for it, and the returned value can be used. However, from the user’s perspective,
the protocol implementation is just a tool, a library to use, where the behaviour is flipped: the
user initiates the function calls.
Inspired by fluent interfaces [6], we designed an API that supports method chaining with
Elixir’s pipe operator. With the respective function calls, the user can start the protocol (’start’),
request the next choice from the protocol (’wait’), and return the chosen value (’chooseChoiceLabel’).
To enforce some constraints on the chaining, an API state is introduced that is returned by
each function. It acts as a type to enforce some type constraints while also containing the values
of certain related data (e.g. current variables). Starting the protocol process returns ’IdleState’,
which is accepted by the ’wait’ function, which then returns the choice’s state, etc.
The usage of the API functions is shown in figure 5.3.
The main goal of the API module is to handle the user’s requests of the next choice, while
receiving those choices from the protocol module. However, the two actions are not synchronised.
The module might have to store any incoming user requests (’waiting_answer_from’) until it can
reply, or it might have to store new choice states (’state_to_return’), until the user requests it.
As such, the state of the API module GenServer is ‘{state_to_return, waiting_answer_from}‘.
The state changes in relation to incoming user requests or choices are demonstrated in figure
5.4.

5.2.2
Protocol module
The protocol module primarily implements the specified protocol behaviour of a process. In the
Elixir implementation of the protocol, the protocol’s states and parameters are both encoded in
a map, stored in the GenServer’s state. Each time a parameter is accessed, the value is returned
==PAGE:33==
5.2 The structure of the generated Elixir code
29
from the state. Each time new parameters are added or modified, the values are changed in the
state. Ultimately, each command accesses the state and returns a new state.
A difference here from the mCRL2 code’s behaviour is that the values in the module state
are not deleted when changing states. The values are kept in the state until they are overwrit-
ten or the program is stopped. However, if the mCRL2 model is type-checked correctly, the
corresponding Elixir code will not access these ”extra” values.
The protocol state is defined by the ’:state’ value in the GenServer state. To restrict the
execution of the receive cases based on the underlying protocol states, an additional guard is
placed to each receive clause, checking the ’state.state’ (i.e.
the protocol state value in the
module state map).
With this approach, messages are received by defining many clauses of the ’handle_cast’
GenServer callback. The execution follows the first clause that matches both the state and the
receive command’s condition.
5.2.2.1
Timeout
A special case in the Elixir code is how we model the timeout behaviour. Even though GenServers
support timeouts, those are used for specifying a maximum waiting time before a caller receives
a response. The protocols here have a different behaviour, to specify a maximum waiting time
between two incoming messages. For that, a custom logic is used based on Elixir’s timer object,
stored in the module state.
The timer is initialised by a random number between 0 and 10 seconds1. Whenever the time
passes without resetting the timer, a function clause gets called, triggering the specified timeout
actions. Depending on the protocol specification, each state might have a separate function
clause, but a general case is defined to capture timeouts in states without a timeout command.
The timer resets at every message with another random number. For simplicity, the resetting
logic (see listing 5.8) is placed in the ’updateState’ function, which gets called upon every received
message.

CHAPTER 7
Evaluation
In this section, we answer the question stated in the introduction. Namely, to what extent can a
high-level language be developed to specify distributed protocols that enable the generation of both
an mCRL2 formal model and an Elixir implementation with identical behaviour, while addressing
the challenges of language design, formal model simulation and implementation fidelity?
In the previous chapters, we defined a candidate high-level language as well as a design
for the formal model and for the implementation. We showed that an implementation can be
designed to follow the formal model’s behaviour. The remaining questions are: How complex
protocols can this approach support? And To what extent and how easily can we simulate and
verify different scenarios in the formal model?
We specify three different protocols in increasing complexity. We show some characteristics
of the formal models and verify some properties in each case. The generated mCRL2 and Elixir
files for these protocols can be found in the end of the appendix.
Afterwards, we investigate the performance and readability of the generated code.

Exposing the leaders is inspired by an existing RAFT mCRL2 specification [10].
To limit the size of the state space, we can restrict the model to reach higher terms by
enclosing the broadcast incrementing the term in a condition. If the model reaches a state when
the only option would be to timeout on the highest allowed term, the system ends in deadlock.
The properties below are verified on this limited space by allowing maximum 2 terms and by
specifying the network model to use ’fifo’ messages.
RAFT is supposedly tolerant of failures of up to (N-1)/2 nodes, where N is the total number
of nodes. As long as the majority of nodes are alive, it can work normally.

(7.2)

We can see that we can define a variety of different protocols, that are using sending/receiv-

7.2.2.1
DSL constraints
Using Spark resulted in some limitations when it comes to defining protocols in the DSL:
1. The arguments of the entities are always required and cannot be omitted.
2. Recursive structure of the entities (e.g. nested if! commands) is ignored without warning.
A workaround is to move the nested entities to a different state.
These issues might be annoying in certain cases, and debugging them might be challenging
without a familiarity of how Spark works.
Furthermore, there are a couple of cases where a protocol specification can be expressed in
the DSL but it might not result in correct generated code. As these issues are not a result of
the chosen approach, rather than just missing features and bugs, they are listed under future
work in section 9.1.
7.2.3
Debugging
In terms of the code quality of the resulting generated code, the Elixir code has a lot of un-
necessary ”noise” (like empty blocks, multiplied function calls) and it might be hard to debug.
However, the code is structured nicely around the callbacks of the GenServer behaviour, which
is a well-known concept in Elixir.
On the other hand, the mCRL2 is not necessarily harder to read than human-generated code.
One can get used to the generated structures, and identifying bugs is easy by using the mCRL2
toolset. This however assumes that the programmer has a basic understanding of mCRL2’s
concepts, which might not be the case for the average Elixir developers.

In general, code duplication might easily happen between states, which is due to how the
protocols can be specified in the DSL.
Once a bug has been identified, it is assumed to be easy to identify what needs to be changed
in the protocol specification, thanks to the DSL commands that follow the conventional names
and behaviour of different constructs.
7.2.4
Performance
The generated Elixir code is not affected by any major bottlenecks related to its performance.
The GenServers are commonly used in the industry, and their usual bottlenecks related to paral-
lel message handling do not apply in our case. However, for trivial programs, using GenServers
is usually unnecessary and their initialisation costs might become comparable to the program’s
runtime.
The mCRL2 model however suffers from the usual challenge in model checking: the state
space explosion. The state space is especially affected by the network layer. If the network is
configured with message swapping, it can multiply the amount of transitions available in each
state compared to ’fifo’ networks. In the case of the RAFT protocol, using set as the message
storage lead to more than a 100 times more states. As a result, verifying properties in such huge
models can easily take several hours on an Apple M2 Macbook Pro computer.

CHAPTER 8
Related work
In this section, we try to highlight some interesting research made within the area of formal
methods that are either relevant to mCRL2 or when the formal methods are combined with
some implementations.
8.1
Combining formal methods with software development
Many project tried some sort of conversion between formal specifications and industry code. One
way is to generate the formal specification from the implementation. There exists a tool that
generated mCRL2 from a small subset of Java [11] or the etomcrl tool, which translates Erlang
into mCRL2 [12]. The other way is to generate the implementation from the specification (eg.
RTPA to Java [13]). McErlang [14], however, uses an inner formal representation of the Erlang
code, and the model checking algorithms are directly implemented in Erlang.
Both etomcrl and McErlang provides model checking tools for Erlang, a language which is
based on the same foundation as Elixir. These tools however are very complex as they support
almost arbitrary code. As such, verifying properties are cumbersome and the issues are hard to
trace back in the source code. Additionally, none of these tools have been updated to keep track
of modern model checking approaches. etomcrl for example only supports the less expressive
LTL formulas instead of model µ-calculus, and do not utilise modern mCRL2 features such as
counter-example generation.
Other than converting from/to formal models, there are also other ways to use formal models.
A semi-formal approach is to use dynamic techniques (analysing event traces) to create an
abstract state transition diagram that can be analysed with formal methods [15].
In Formal Specification-Driven Development (FSDD) [1], which combines the methodology
of Test-Driven Development (TDD) with formal methods, a formal specification is needed to
generate unit tests and some parts of the implementation.
8.2
Formally specified systems
There are also numerous examples of systems that are defined in mCRL2. These examples vary
from famous logical problems or games [16] [17] to concurrent or probablistic systems.
When it comes to distributed protocols, our research was inspired by the two- and three-
phased commit [18] and the RAFT consensus algorithm [10].
It is worth mentioning other
distributed systems defined in some other tool, such as failure detectors in the model checker
tool UP-PAAL [19] or the formal definition of a symmetric protocol to establish service level
agreements [20].
There are also some books focusing on designing mCRL2 systems in general, such as ”Tu-
torial: designing distributed software in mcrl2” by J. F. Groote and J. J. Keiren [21] or ”Un-
derstanding behaviour of distributed systems using mCRL2” by M. Atif and J. F. Groote [22],
which also includes a couple of example applications.

CHAPTER 9
Conclusion
In this project, a high-level domain-specific-language (DSL) has been developed to specify sim-
ple distributed protocols consisting of processes sending/receiving/broadcasting messages. The
processes support if-then-else blocks and indeterministic choices to structure the data flow. The
processes are based on a nondeterministic automaton by specifying different states and state
transitions triggered by incoming messages. By separating this control flow from the data flow,
we facilitated the generation of a formal specification in mCRL2 and an implementation in Elixir
from the protocol specification. In this semi-formal approach, we showed how the Elixir code
follows the behaviour of the mCRL2 model by mapping them to the automaton and sequentially
comparing their data flow. For the Elixir implementation, an additional interface module has
been designed for the user to provide values to the protocol process upon indeterministic choices.
The generated Elixir code can be configured to run in a distributed environment, while the
generated mCRL2 model can be used without modification to analyse system behaviour and
verify different model checking properties. The generated formal model can also be configured
in different ways to simulate how the system behaves in different environments (e.g. unreliable
network or nodes).
We evaluated the usability, expressivity, and performance of the approach by specifying and
analysing three example protocols in increasing complexity. We demonstrated that both general
and protocol specific model checking formulas can easily be formulated and verified against the
generated formal models.
Overall, while acknowledging the challenges of the semi-formal methodology, the state space
explosion in model checking, and the tool’s currently missing features, we believe that the tool
implemented acts as a good proof-of-concept of this approach and it provides a good base for
future improvements and extensions.
9.1
Future work
There are numerous ways the tool can be further improved and extended. The tool has some
currently known bugs and missing some much needed features to make its usage easier. There
are many possibilities to add new features to extend the protocols that can be specified.
Making most of these features are straightforward tasks with the knowledge of the tool’s
implementation.
However, there is an option to strengthen the reasoning behind the tool’s correctness, which
might result in rewriting the generation logic.
9.1.1
Alternative formal model
Currently, the tool uses a nondeterministic automaton to model the system’s states, while ig-
noring its parameters and the data flow in general.
Alternatively, both the state transitions and the semantics of the data flow could be en-
capsulated in an automaton that extends nondeterministic state machines with data. A good
starting point could be the Extended Finite State Machine (EFSM) [23], which combines both
==PAGE:60==
56
9 Conclusion
the control and data flow. That machine has to be defined in an indeterministic and non finite
way.
Since the mCRL2 code can be generated automatically from state machines, a similar trans-
lator has to be written for this new automata. The new translation would strengthen the formal
model that is used to prove the equivalence between the DSL and the mCRL2 semantics.
9.1.2
Missing features and known issues
Here we list the known limitations, missing features and extension possibilities of the current
version of the tool.
1. Checking for name conflicts:
a) The parameter names are not checked for naming conflicts. Conflicts may result in
undefined behaviour or type checking errors in the generated languages.
b) The parameter names used in arguments are not checked if they exists in the state.
Undefined names result in type checking errors in the generated languages.
c) The parameter names might conflict with names like ’state’, ’timer’, ’pid’, any re-
served keyword in Elixir on in mCRL2, as well as with random generated names of
form ’pidN’ and ’valN’ where ’textitN is a natural number.
2. Adding type checking: The type of the values are not checked and mistyped expressions
might result in type checking or runtime errors.
3. Enforce valid protocol structure:
a) Empty children (e.g. ’then!’ without children) might result in syntax error in the
generated mCRL2 code. A workaround is to add a ”label! :tau, []” command, which
adds the special hidden ”tau” action.
b) In every receive block, execution should reach to a change state command as the last
executed command, otherwise, the mCRL2 process will reach a deadlock.
c) The ’self’ value can not be used in the receive commands’ condition. This is a limita-
tion of using functions in Elixir guards.
4. Missing configuration options:
a) Limiting the maximum amount of messages stored in the network (mCRL2): The
network model is currently limited to hold maximum 20 messages at a time, which
should be customizable.
b) Customising the time before timeout actions (Elixir): The timeout currently is defined
as a random number
5. Issues with the generation:
a) In Elixir, any command placed after indeterministic choices are executed before the
children commands of the choice. As such, this results in undefined behaviour.
6. Adding new features:
a) Implementing support for more data types (strings, sets, nested data structures, etc.)
b) Implementing support to specify different recovery algorithms after the nodes crash.
c) Providing more insight on the protocol’s state to the user through the API module
(Elixir): the user might want to access or modify the states of the processes.
