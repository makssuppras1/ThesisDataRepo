Abstract
Requirements engineering is a process that involves a lot of iterative steps to extract and re-
ﬁne system requirements in order to establish the right speciﬁcation that matches the client’s
actual needs. However, in a large scale project, the requirements can be immense and there
can be a lot of people working on the same project together. It often becomes a challenge for
a system analyst to translate the requirements gathered by the requirements engineer to the
corresponding model due to their immense size. Moreover, validation becomes terribly diﬃcult
and traceability is often lost in the process. This thesis tries to explore a novel approach to
represent each individual requirement with many small sketches of model fragments. When the
model fragments are small, they will be easier for the stakeholders to understand and validate
and they can be traced back to the original requirement. These various model fragments can
then be weaved (merged) together to form a larger model that can potentially be used to es-
tablish the formal model, with the additional beneﬁt of automated creation of the traceability
links within every weaving steps. This thesis aim to create the tool that can be used for the
weaving process, within the context of Requirements Engineering eDitor (RED) tool.
The goals of this thesis are: (1) to extend the functionalities in RED for creating visual editors
for UML diagrams, (2) to implement model weaving functionality in RED, (3) to ensure trace-
ability within the weaving process, (4) to perform and evaluate the result of model weaving on
an existing Library Management System case study

Chapter 1
Introduction
Model weaving is an ongoing research topic which has gained some popularity, mostly within
research area of model driven software engineering. In this thesis, the author also attempts to
apply model weaving techniques, but limited in the context of software requirements. Creation
of model fragments and weaving of model fragments during requirements engineering phase may
potentially add value by improving clarity of the requirements to all the stakeholders involved
in a software project, and shall be discussed in more detail within this thesis.
This thesis is a continuation of research done by previous MSc. students, Ms. Sanam Ali and
Mr. Jakob Kragelund, who have worked on subjects related to software requirements and model
weaving under Prof. Dr. Harald St¨orrle. Ms Sanam Ali, whose thesis titled: ”Weaving Re-
quirements Fragments: An empirical investigation” sought to establish various rules that can
be used to transform textual requirements into model fragments and at the same time con-
ducted empirical investigation on the potential use of model fragment within the case study
of a Library Management System Ali [2014]. Mr Jakob Kragelund, whose thesis titled: ”Ad-
vanced Tool Support for Requirements Engineering” had come up with a basic tool developed
within the context of RED (Requirements Engineering Editor) tool, that can be used to create
model fragments and perform basic weaving Kragelund [2012]. However, the tool have a lot of
weaknesses and require a major revamp.
In subsection 1.1, the author will try to explain the issues related to software requirements that
will need to be solved in this thesis. There will also be some pictures to illustrate the issues,
and there will be some proposition that the author think are quite important, which form the
basic motivation of this thesis.
1.1
Motivation
Requirements engineering is important because it is the ﬁrst phase in development of software
and it functions as a foundation to be used for all the next phases of software engineering.

==PAGE:12==
1.1. Motivation
If requirements are not speciﬁed correctly, all the next phases of system design, analysis and
development can be at risk of deviating from the actual customer needs. A few research study
shows that often, the failure in software industry is attributed to errors in requirements. For
example Lutz [1993] shows that 60% of errors in critical systems were the results of requirements
errors.
Another study by Bey [1996] shows that 60% of the software companies in Europe
consider requirements engineering problem as very signiﬁcant. Even so, lots of software projects
still go on in quite unstructured and chaotic requirements engineering process.
Proposition 1.1. Software failures - signiﬁcant number of errors in software are mostly at-
tributed to errors during requirements engineering
Based on Mich et al. [2004], most of software requirements in industry are speciﬁed in natural
language. This happens despite the fact that natural language can cause ambiguities. Some
classic examples of the ambiguities are as follows (taken fromWasow et al. [2005] ):
1. Small dogs and cats
2. We saw the man with the telescope
3. No student solved exactly two problems
Specifying requirements in natural language therefore can often raise a lot of ambiguities. Some
things are better to be speciﬁed in pictures/diagrams/movies.
Proposition 1.2. Ambiguity in natural language - Requirements can be improved if they are
accompanied with pictures/diagrams
This proposition is based on the study conducted by previous MSc. student Sanam Ali in Ali
[2014]. Sanam performed a case study on LMS (Library Management System) which is a good
case study to be used as a reference because it has been used for a few years in the course 02264
Requirements Engineering in DTU. Sanam’s ﬁnding concluded that most of software require-
ments can be translated into sketches of model fragments based on some transformation rules
(which can be found in Appendix B) and when users are forced to follow the transformation
rules, the requirements often need to be re-formulated according to the transformation guide-
line(for example passive sentence needs to be reformulated as active sentence to be considered
as a good requirement). Overall this practice often brings higher quality of requirements and
the generated model fragments also provides better clarity to the users.
Figure 1.1 shows one example of software development life cycle model (SDLC) based on rapid
prototyping model from Martin [1991]. Although there are a lot of diﬀerent software life cycle
model, most of them always include the requirements phase and the design phase. A lot of
times, software project failure happens because there is an issue during the transition from
requirements to design (highlighted as red arrow in the ﬁgure).

In a large scale software projects, those various software phases often are taken care by diﬀer-
ent roles. For example, there can be a requirements engineer who is in charge of elicitation of
requirements from the customers, system analyst/software architect who is in charge of trans-
lating the requirements to the models, and the developers who develop the model into actual
code. (In this thesis, the author assumes that the term ”model” refers to UML model, which is
the most commonly used standard in the industry).
A lot of time, communication does not happen very well between those people with diﬀerent
roles. For example, the requirements engineer probably provide the requirements to the analyst
simply by throwing the documents and expecting the analyst to understand everything based
on the text. Lots of the errors are then encountered during this step, because this translation
process is a manual process and depends largely on the knowledge and comprehension of the
text by the analyst.

1.1. Motivation
This manual translation process from requirements speciﬁcation to software model is often
very error-prone.
This issue therefore raises the following question:
”Can the process of
translating software requirements to model be automated to reduce the errors caused by manual
translation?”
There are actually a few tools that has been developed by academic researchers to solve this
issue. Some of them are listed in section 2.6. Most of the tool mentioned usually use Natural
Language Processing (NLP) engine, accompanied with some manual annotations to work, and
they can perform some amazing functionalities (for example it can generate class diagram and
sequence diagram from a piece of textual requirements). However, a review done by Yue et al.
[2011] shows that most of the tools are not perfect and the generated models are sometimes
incorrect.
Proposition 1.3. Natural Language Processing - With the current state of technology, perfor-
mance of NLP tool is usually not perfect, and using the tools can be worse than not using them.
If automated requirements to model translation tool is not perfect, then the most feasible solu-
tion would be to run the process manually, but certainly some other things can be automated. As
discussed earlier in the ﬁndings by Sanam Ali, it seems that requirements can be translated into
model fragments by transformation rules (in most cases they are a one to one mapping). If there
are many model fragments generated from each requirements, they can then be weaved/merged
together. If this process is implemented as a technical solution (with a tool support), this pro-
cess can provide better result than an analyst trying to interpret many textual documents and
formulate them into model manually.
Proposition 1.4. Weaving - Transformation of requirements to model fragments and automated
weaving of those model fragments can help to reduce software model errors.
The rationale of such proposition is because when a customer is presented with large model,
it can be extremely challenging for him/her to comprehend and verify it (due to its immense
size). Furthermore, every individual stakeholders may have diﬀerent knowledge and experience
in software engineering. Some stakeholders may have an academic or professional background
related to software engineering and are familiar with all the terms and various visual models
used in software engineering.
However, it may not be the case for some others.
With the
proposed solution (translating manually to model fragments followed by model weaving), the
model fragments will be very small and it should be very easy for any user (even those without
IT background) to verify that the model fragments corresponds to the requirement.
As an
example, Figure 1.3 shows a model fragment that is translated from LMS case study requirement
CSG3=”A search yields a list of overview results that may each be expanded to detail results.”
It will be easier to verify this small fragment against the requirement because the fragment is
small and easy to understand and the user already knows that the source of this fragment is
requirement CSG3.

can be veriﬁed and is trustable?”

1.1. Motivation
In this case, the author thinks one important factor to consider is that the weaving tool should
be able to provide suﬃcient proof of the source requirement of a model fragment. The tool
should not surprise the user with some ”magic” procedure that can generate diagrams from
text, instead they should be allowed to visualize the model weaving performed step-by-step,
therefore they can understand the input and output of each step. The tool should also provide
suﬃcient debugging capabilities for the user to troubleshoot issues.
Proposition 1.6. Trust - Trust can be achieved when model translation is done by a tool that
can provide step-by-step visualisation of the output and is supported with suﬃcient debugging
capabilities
One thing that aﬀects how the user can trust the output of a requirements to model translation
tool is traceability. When large textual requirements are translated into large models directly,
they often lost all traceability. The user will often have to dig through all the documents to
understand which requirements is the source of the model. However, if individual requirements
are translated to model fragments as a one-to-one mapping, we can deﬁnitely still keep the
traceability. When the model fragments are weaved, new traces will need to be created so that
the model fragment can be traced to the weaved diagram. If this process of trace link creation
is repeated for every weaving steps, then it is possible to follow the traces from the initial model
fragments to the weaved diagram that are created in every intermediate steps and then to the
ﬁnal output.
Figure 1.5: Traceability from requirement to the model fragment and to the weaved diagrams
(shown as blue arrows)
Proposition 1.7. Traceability - If traceability link is created for every weaving step, it will be
possible for the user to trace from requirements to the model and vice versa from the model to
the requirements.

Chapter 1. Introduction
Another key challenges in running a software project is that a lot of changes may occur in the
middle of the project. Customers may modify requirements at anytime and the requirements
engineer and the analyst will then be expected to update their artifacts. Therefore it is important
to be able to synchronize changes from the requirement to the model. In this thesis, the tool
developed should provide suﬃcient method to synchronize changes of the requirement to the
model.
Proposition 1.8. Synchronization - Synchronizing the updates from one diagram to all the
weave source and output can help to keep the requirements and the model in sync.
Interestingly there is not many commercial tools that can be used for this process of requirements
to model translation yet, therefore this thesis is a novel approach towards ﬁlling in the gap
between a requirements speciﬁcation and the formal model by creating and weaving various
model fragments that corresponds to the individual requirement.
1.2
Goal
The primary goal of this thesis is to build a model fragment editor, implement a weaving tool,
and then apply model weaving on an existing case study of Library Management System, as a
proof of concept of the beneﬁt of model fragment creation and model fragment weaving.
Although there is already an existing model fragment editor within RED tool that was developed
by previous MSc. student Jakob Kragelundt, it is then realized that there are a lot of limitations
on the tool. The existing tool has dependencies on various software libraries such as Swi Prolog,
JPL (Java to Prolog connectors) and various other libraries, which makes the tool setup more
complicated and diﬃcult to conﬁgure. Besides, many feature gaps (due to its non-traceability
and the low maintanability of the code), justify as the reason to discard it and replace it with
another one which provide better feature and tighter integration. Meanwhile, another MSc.
student, Jeppe Bloch Kristensen who started his thesis 3 months earlier than the author, has
also created a tool for creating visual diagram within RED tool. Jeppe’s visual editor tool
initially was developed for the purpose of drawing goal diagrams for goal analysis, but it has
been designed to be extendable for other types of diagrams as well. Because of its generality, it
was decided to extend it for creating model fragments as well.
In addition to that, the thesis work from the previous MSc. student Sanam Ali has covered
the empirical study on various rules in transformation of requirements into model fragments,
the beneﬁts of adding these model fragments by conducting some examples and an experiment
from a Library Management System case study, and additionally outline some basic information
about model fragments weaving, and can be used as a baseline to establish the suitable model
weaving tool.
Overall, the focus will be to ensure that creation and weaving of model fragments can be done
as simple as possible, with automatically generated traceability, and in parallel also provide a

Approach
The approach for this thesis is to ﬁrst ﬁnd out the existing issues with model design from
requirements in general, which does not have traceability, very diﬃcult to verify and largely
depends on the skills of the analyst who perform the translation. The author tries to formulate

The author tries to answer the propositions by proposing weaving of model fragments as a
solution. The author then tries to develop a tool that can address this issue, and tries to use
the tool on an existing case study of Library Management System as a proof of concept. At the
end of this thesis, the author then tries to evaluate again if the propositions presented earlier
have been suﬃciently addressed by the tool created in this project.
Overall the focus of this thesis will be to show that the practice of translation of small units
of requirements to model works better as compared to the existing practice of ”design all and
verify all”. There can be more than one approaches to solve this issue, but in this thesis, the
author attempts to develop the tool that can suﬃciently support the functionalities required for
drawing model fragments and weaving them together within the context of RED tool, which
is the tool that has been used as a teaching tool in 02264 Requirements Engineering course in

2.1
Requirements
In [Glinz, 2011], requirement according to the classic deﬁnition from IEEE standard 610.12 can
be deﬁned as:
1. A condition or capability needed by a user to solve a problem or achieve an objective
2. A condition or capability that must be met or possessed by a system or system component
to satisfy a contract, standard, speciﬁcation, or other formally imposed documents
3. A documented representation of a condition or capability as in (1)or (2)
The term requirement has been used within many contexts, however it is important to limit the
context within this thesis, that it only refers to the following two types of requirements:
• Functional requirements: A requirement concerning a result of behavior that shall be
provided by a function of a system (or of a component or service), otherwise known also
as features
• Non functional requirements: A requirement that pertains to a quality concern that is not
covered by functional requirements, otherwise known also as quality attributes

2.2. Requirements Traceability
A small survey conducted by [Nikula et al., 2000] on twelve Small- and Medium-sized enterprises
shows that there has been a very slow adoption of best practices in requirements engineering from
the academia to the industry. Even though a lot of literatures about requirements engineering
are available, most of the standard practices in academia has very limited knowledge of the
proper requirement engineering process. The usage of Requirements Management tool shown
in the survey was quite low, with many of them making requirement documents with general-
purpose tools such as word processor and spreadsheet in their daily practice. [Friis, 2012] in his
Msc Thesis did a comparison of various commercial tools such as RequisitePro and DOORS,
and open source tools such as OSRMT. Among their disadvantages are their high license cost,
high learning curve, and diﬃculties in customization, which becomes the motivation for the
development of Requirements Engineering eDitor (RED) tool.
Most of software requirements and domain descriptions are speciﬁed in the form of natural
language [Mich et al., 2004]. Some research has been done to show how the use of natural
language is closely related to software design. One such example is the classic example from
Abbot, which shows the technique for the design of an ADA program given an informal English
description. It is shown that data types, operators, and control structures can usually derived
from informal English by the following informal strategy [Abbott, 1983]:
1. A common noun suggests a data type.
2. A proper noun or direct reference suggests an object.
3. A verb, attribute, predicate, or descriptive expression suggests an operator.
4. The control structures are implied in a straight-forward way by English language.
[Cheng and Atlee, 2007] stated that natural language was and will remain the main form of
requirements documentation. Usually an analyst will transform this requirements documenta-
tion into a formal model manually, therefore the details and the correctness of the model highly
depends on the skills of the analyst. Besides that, the analyst may also create another informal
model which is easier to understand for the client in form of natural language. Keeping both the
informal model and the formal model in sync can be an exhaustive process and can be costly,
therefore there should be a mapping procedure to ﬁll the gap between textual requirements to
the formal model [Dawson and Swatman, 1999].
2.2
Requirements Traceability
Requirements traceability is one of a concern in requirements engineering which has been gaining
some research attention. Although traceability is deﬁned diﬀerently by diﬀerent practitioners
and in diﬀerent literature, [Gotel and Finkelstein, 1994] deﬁnes traceability as ”the ability to
describe and follow the life of a requirement, in both a forwards and backwards direction (i.e.,
from its origins through its development and speciﬁcation to its subsequent deployment and
use, and through all periods of ongoing reﬁnement and iteration in any of these phases).”
12
==PAGE:23==
Chapter 2. Background
Requirements traceability is important, especially during requirements validation process, where
every software requirements need to justiﬁable and traceable back to the source of the require-
ments, the goals and the stakeholders involved. In addition to that, traces are valuable because
they can help to identify pairs of relating artifacts which can then be validated and veriﬁed
against each other[Winkler and Pilgrim, 2010]. Traceability has been identiﬁed as one of soft-
ware quality attributes, and a system with good traceability can deﬁnitely beneﬁt in a better
quality, in terms of the completeness and understandability.
[Jarke, 1998] lists four kinds of traceability links within the whole system’s life cycle, which
distinguishes how the trace relates to the requirements:
• Forward from requirements: assigns responsibility for implementing a given requirement,
and makes it possible to analyse the impact of changing a requirement
• Backward to requirements: shows the compliance of systems back to the requirements and
prevent gold-plating (design without requirements).
• Forward to requirements: shows the link between the change in the stakeholder needs to
the change in the requirements
• Backward from requirements: are used to validate the requirements based on the under-
lying factors
The ﬁrst two traceability links are quite commonly referred to as Post-Requirement Speciﬁ-
cations traceability (Post-RS traceability), while the latter two are commonly referred to as
Pre-Requirement Speciﬁcations traceability (Pre-RS traceability)
In addition to these 4 various traceability which shows the vertical link from one process (re-
quirements) to the next process (design), [Ramesh and Edwards, 1993] also identiﬁes the concept
of horizontal traceability link which exist within the same level of process. This type of trace-
ability are used to trace from one artifact to another artifact within the same process, for
example tracing of dependencies or conﬂicts between each requirements during a requirement
speciﬁcation process. These various traceability links concepts can be illustrated in Figure 2.1
[Winkler and Pilgrim, 2010] observed that the main barrier to creating and maintaining trace-
ability is mainly due to the economics behind maintaining the traces. However, if the economics
are ignored, and all requirement traces are recorded as much as possible, it will beneﬁt the stake-
holders in answering the following questions about the artifacts:
• What information is recorded in the artifact?
• Who has created or updated the artifact and documented the information?
• Where has the information come from?
• How is the information represented?

• Why has the artifact been created, modiﬁed, or evolved?

2.5. Model Weaving
2.5
Model Weaving
2.5.1
Deﬁnition and example of model weaving
Model weaving can have various meanings in diﬀerent context, however in this thesis project,
model weaving is be deﬁned as the process of merging various small model fragments to form a
larger piece of model that can potentially be used to create a formal model.
Figure 2.5: Model Weaving Process (image courtesy of Harald St¨orrle)
As can be seen in Figure 2.5, model weaving process begins with creation of model fragments
to accompany every speciﬁed requirements. The beneﬁt of creating a model fragment for every
requirement is that it allows for easier veriﬁcation and validation process with the stakeholders.
It is really diﬃcult for a stakeholder to look in a large piece of model and verify that it matches
with all textual requirements that have been speciﬁed. However, by talking with the stakeholders
in the smallest possible unit of artifact, which is in this case requirement and its corresponding
model fragment, it becomes more manageable for the stakeholder to verify that the model
fragment really matches with the requirement.
Each requirements (labeled as Rx in the ﬁgure) may have one, more than one or no model
fragments (labeled as Fx,y) attached to it. Each of these fragments will need to be derived and
clariﬁed against the textual requirements. [Ali, 2014] has done some research into various rules
that can be used to decide which type of model fragment should be created from a requirement
and various rules to derive them. A summary of transformation rules used for this process can
be seen in the appendix B. The model fragments can then be weaved together step-by-step to
form the intermediate models and there may be overlapping visual elements within the various
model fragments which will get merged. After all the weaving steps have been completed, it
can then be restructured to form a complete model. This complete model can potentially be
exported to a ﬁle format which is suitable for import into commercial model drawing tools such
as Magic Draw.
Figure 2.6 shows an example of one of the step in the weaving process. Every weave step is
assumed to take two inputs visual diagram and produce one output diagram. From the ﬁrst

2.5.3
Existing model weaving within RED
RED was also developed with an experimental model fragments weaving feature [Kragelund,
2012]. However, over the time it was discovered that the existing model weaving feature has a
lot of technical diﬃculties in its setup. The weaknesses of the existing tools are as follows:
1. Dependencies on external program.
The existing model weaving feature is a java program that calls a prolog program, therefore
implying that the user need to install SWI prolog and Java to Prolog (JPL) connector.
It was also discovered that the prolog software and the JPL connector needs to be in the
right version before it can work, therefore making this setup not working well for any
users.
20
==PAGE:31==
Chapter 2. Background
2. Feature gap
The existing weaving feature also suﬀers a feature gap due to unavailability of tracing
method within the tool. After a model fragment has been merged with another model
fragment, it is therefore not possible to trace back on what are the sources of each of the
visual elements.
3. Unnecessary feature
The existing weaving feature comes with its own weaving editor in addition to visual
fragment editor used to create the model fragments. The weaving editor idea comes from
the idea of UML package merge, where the two input diagrams are ﬁrst imported into the
weave editor, and each visual elements are then linked manually to indicate elements to
be merged. However, this feature turns out to be impractical because it requires every
model fragments to be manually linked. There is not much beneﬁt achieved from having
this weave editor, because although the users can manually override matching of elements
despite their naming inconsistencies, it is actually not very likely to be used in a real
scenario. If there are naming inconsistencies between two elements from the two input
diagrams, the inconsistencies should ideally be ﬁxed in the input diagrams directly, and
not manually overridden during the weaving process.
4. Low maintanability
The existing visual editor relies on an automatic code generation framework for visual
editor, called Eclipse GMF (Graphical Modeling Framework) which makes it possible to
speed up development of a graphical editor at the expense of lower ﬂexibility.Because the
code is auto-generated, it actually creates a huge amount of code that is diﬃcult to modify
because they are machine generated.
Another Msc thesis student Jeppe Bloch Kristensen has then decided to develop another
visual editor within RED which uses a diﬀerent framework: Eclipse GEF (Graphical
Editing Framework) which provides more ﬂexibility when a developer would like to add
new type of diagram or new type of visual elements. Jeppe had an overlapping thesis
period with this thesis because his thesis ends in March 2015, therefore the ﬁnal version of
the visual editor that he has developed can only be extended for this project after March.
Given all of these disadvantages with the existing weaving tool, it is therefore decided to do a
major revamp of the existing weaving tool to use a program that is purely developed in Java and
is based on the foundation that has been laid out by Jeppe by using Eclipse GEF framework.

2.6. Other Related Works
[Yue et al., 2011] performed a review on existing approaches to transform software requirements
into analysis models. In the paper, it is mentioned that requirements mostly come in textual
model, which is not a model formal enough to be understood by computers, therefore require-
ments are not very suitable for automated transformation, and instead only manual heuristic
method such as Abbot’s heuristics is generally followed. The paper also explains about a con-
ceptual metamodel that can be used to manage model transformation and review process for
the transformation, as shown in Figure 2.9. A requirement may go to a direct transformation
into an analysis model, or it may go through various transformation where it is temporarily
stored in multiple intermediate models.
Within every step of transformation, a traceability
link need to be stored to keep track of all the changes from the original requirement to all
the intermediate model up to the ﬁnal analysis model. It is also mentioned that although the
existing tools reviewed in the literature implemented some kind of model weaving techniques,
traceability models within each of these tools are either not supported or not very well deﬁned.
For model transformations which require one or more intermediate models, traceability links are
very important because the users should be allowed to trace the requirements to the analysis
model without going into the details of each of the intermediate models. A traceability link
should at least indicate the link between the source and target elements and should preferably
indicate which transformation steps are applied to create the link.

Problem and Domain Analysis
3.1
Users
The main users of RED tool are typically students and teachers, especially those who are
participating in the course 02264 Requirements Engineering in DTU. Because RED is planned
to be released as an open source application, it is expected to be introduced to wider range of
users as well. As a start, it is planned to be introduced to students from other universities. In a
broader perspective, it should also be possible to provide this tool in a real software project for
software analysts, requirements engineers, architects, or their clients. Therefore, it is expected
that the current features in RED may change in the future, depending on what additional
features are required by these future users.
Therefore, the following criteria will need to be considered:
• Usability is one of the most important quality attributes that need to be achieved. A
novice user should be able to ﬁgure out how to create model fragment and how to weave
the model fragments without special knowledge. However, because most of the users are
students who are proﬁcient with computers, a moderate learning curve should be still
tolerable.
• Operability, which in this case relates to the time and eﬀort for the weaving process. It
is undesirable for the users to spend a lot of time in the weaving process. Previously
with the existing RED weaving tool, the user needs to manually link each visual elements
to indicate that they are similar and therefore should be merged. This makes the whole
process tedious, impractical, and requires a lot of time.
• Performance. It is expected that a lot of the requirements can be derived into model
fragment. In a large scale software project, it means that there will be a lot of model
fragments to be processed, therefore the overall performance will be one issue to watch
out for.

• Extendability is also one of the concern for the tool. Although the primary focus is to be

3.2
Usage Scenarios
A typical usage scenario for a model weaving process will be as follows:
1. The user goes through each requirement and derive a model fragment based on the trans-
formation guidelines.
2. The user then selects two or more model fragments to be weaved.
3. RED will attempt to weave two model fragments at a time, and create an intermediate
model fragment to store the result of every step.
4. If any weaving step results in an error, the user is notiﬁed with the error cause and have
to ﬁx the model fragment before he/she continues with the merge.
5. At any time, the user can perform forward tracing from any of the elements in the input
model fragments to ﬁnd out the corresponding elements in the weaved diagram up to the
ﬁnal diagram.
6. The user can also perform backward tracing from the ﬁnal diagram to ﬁnd out the elements
from the input model fragments.
7. The user can then restructure the ﬁnal diagram with a layout that is better for reading.
8. After the ﬁnal diagram is complete, the user can decide to export it to the format that is
supported by some other commercial tools.

3.3.3
Sketchiness aspect
After requirements have been collected, model fragments can then be derived according to some
transformation guidelines. This happens in quite an early phase, and is still part of the overall
requirements process. Therefore, it is important not to make an impression to the stakeholders
that any design decision has been made. It should be highlighted to the stakeholders that the
27
==PAGE:38==
3.3. Creating Model Fragments
main goal of creating these model fragments is to verify the list of requirements that needs to
be done and not how to implement those requirements.
UML diagrams is usually considered as a formal model, therefore it may not be suitable for
the purpose of creating model fragments, because using UML may give the impression to the
stakeholders about the formality of the design. Instead, a sketchy version of UML model could
be used as an alternative, because it provides the beneﬁt of implying design uncertainty and at
the same time it will simplify the process of transforming it into the formal model.
[Buxton, 2010] highlights some properties of a good sketch in user experience engineering in his
book, they are:
1. Quick to make, cheap and easy to dispose of
2. Use appropriate degree of reﬁnement, where it should not depict the concept that exceeds
the actual state of thinking or concept
3. Intentionally ambiguous, for the purpose of allowing it to be interpreted in diﬀerent ways
4. Suggest and explore, rather than conﬁrm (to support ideation rather than conﬁrming
decision)
In addition to that, [Boukhelifa et al., 2012] also researched on the eﬀectiveness of using sketch-
iness in depicting uncertainties. Techniques to depict uncertainties can usually be grouped to 3
diﬀerent categories:
1. Techniques that involve color/hue/saturation, where weak colors typically represents de-
grading data quality or conﬁdence
2. Techniques that involve focus, such as blurring, increasing transparency and lowering
resolution to depict uncertainties
3. Techniques that involve geometry of lines, such as using dashed lines or sketchy eﬀect to
depict uncertainties
The research concluded that sketchiness is only slightly less perceived than other techniques to
be showing uncertainties, and in fact not any single technique is much better perceived than the
others. For drawing diagrams, sketchiness and blurring receive almost similar eﬀect, and in fact
sketchiness gives better impression of informality. While dashed lines are the most preferred
way to depict uncertainties, it may not be suitable for diagrams because dashed lines may carry
its own meaning in certain diagrams.

3.4.1
Desirable properties from model weaving
Model fragments may appear in various type of diagrams and may contain many types of
visual elements.
Since model fragments are closely related to requirements, we will expect
that the number of model fragments in a requirement speciﬁcation will increase as the number
of requirements increase. Merging all of these model fragments can be done by merging one
diagram to another diagram iteratively until all the diagrams have been merged. This approach
however raises a question on whether the sequence in weaving the diagrams may impact the
actual output of the diagram. Therefore, the following properties of model weaving will be put
into consideration for designing the weaving algorithm.
3.4.1.1
Commutativity
Assuming that we label a merge operation as an operation ⊗. Therefore an operation A merged
with B can then be labelled as A ⊗B. With commutativity property, the operation A ⊗B is
expected to yield the same result as B ⊗A.
3.4.1.2
Associativity
With associativity property, it is expected that a merge operation (A ⊗B) ⊗C should yield the
same result as a merge operation A ⊗(B ⊗C)
To further simplify the assumption, commutativity and associativity property of the merge
operation should only consider the type of visual element and the textual data that it contains
as relevant factors. The visual data for example the location, size and color of the visual element
which does not carry any meaningful information other than to improve readability, should not
be considered at this point.
The commutativity and associativity properties become one of the basic consideration of the
weaving operation that will be designed for this project. In ﬁgure 3.6 it is shown that in the
existing RED weaving implementation done in Kragelund [2012], there is a weave editor that is

4.4.1
Weaving algorithm
During the design of the weaving algorithm, a lot of prototyping steps were performed by the
author. During the prototyping steps, a few design ideas are considered such as:
1. Every elements should be matched against each other only based on its own values and
the immediate elements around it.
Some visual elements can have nesting containment, for example packages, classes and
attributes can generally be considered as a tree structure where a package may contain
another package or a class. However, algorithm based on subtree matching have been
known to have slow performance. Therefore, in thesis all the elements are ﬂattened and
compared one-to-one based on a naive-based matching. This is supported by Ali [2014],
38
==PAGE:49==
Chapter 4. Design
which concludes that usually model fragments do not have deep nesting of element con-
tainments.
2. There are two diﬀerent ways of matching elements: matching by values and/or matching
by structures.
For matching by values, it is possible to use some of the attributes such as names, multi-
plicity, return type, visibility, etc). The list of attributes for every elements are shown in
the appendix in table C.1 to C.4 and they impact the design decision that the author took
on how they should be compared (whether by value matching or both value+structural
matching).
3. Another abstraction level is needed to contain the visual element/visual connection.
Instead of running the weave process directly on the visual elements and the visual connec-
tions, an object called ”MergeElement” is created based on the type of visual element and
the visual connection (refer to section 4.4.3 for the mapping between the visual element to
MergeElement). Each MergeElement may use its own matching criteria that can be dif-
ferent from other MergeElement. Another reason to add this abstraction layer is because
the previous Msc student Jeppe is only planned to ﬁnish his thesis about visual elements
editor in RED in March 2015. Therefore, initial prototyping of the weaving algorithm can
only be done without the visual editor.
4. There is a need to keep track of the knowledge about list of matching elements so that
the knowledge can be used for matching by structures.
In this thesis, this knowledge is termed as Match Knowledge Base (MatchKB). For example
two connections are considered a match if both the connection ends have been matched
before based on the knowledge in the MatchKB). The MatchKB can store the knowledge
of matching elements and answer a query of whether two MergeElements are a match with
the result of TRUE (it is a match), FALSE (not a match) or UNKNOWN (does not exist
in the match KB yet or cannot be determined at this point).
5. When matching elements have been identiﬁed, the matching elements will need to be
merged and all references updated.
During the merging, there can be a merge source and a merge target. The values from
the merge source will be copied over to the merge target before it is removed from the set.
All the elements with existing references to the merge source will also need to be updated
to refer to the merge target. References can either be parent, child, source and target of
connections, incoming and outgoing arrows of control nodes.
6. It is possible to encounter a match conﬂict when the merging of two matching MergeEle-
ments will result in conﬂicting containment structure.
As shown in the example in Figure 4.3, Medium is contained within package library in
the ﬁrst diagram, but is contained within package catalog in the second diagram. It is
however impossible to know if library contains catalog, or catalog contains library or if
they are in the same level.

The ﬂattening phase is performed by a java class called MergeInputReader.
All vi-

which is partitioned and ordered by the type of element it contains, e.g. ﬁrst partition
only contain all classes, second partition only contain all associations etc. The ordering
is done so that those which are container elements and those elements which naturally
have names will be processed ﬁrst so that the ﬁnd-match will be able to ﬁnd the matching

faster. The ordering of the partition are conﬁgured in PartitionTypeComparator class and
are conﬁgured in the following sequence from higher priority to lower priority: Package,
System, State, Class, Enum, Action, State, Decision/Merge, Fork/Join and then followed
by all other type of elements set with same priority.
2. Find-Match phase (line 3-10):
The ﬁnd-match and merge phase are performed by a java class called MergeProcessor.
Because they are lengthy, they are placed under the Appendix D. In ﬁnd-match phase,
the program goes through every unmatched elements and compare them until eventually
no more match are found and the algorithm converges. Diﬀerent MergeElement may have
diﬀerent matching criteria, which will be explained in the section 4.4.4.
The ﬁnd-match phase always try to ﬁnd new match based on the newly gained knowledge
in the MatchKB until it eventually stops (when there is no more new matching pair found
in that particular phase). The ﬁnd-match phase and merge phase are illustrated in Figure
4.4 - 4.7. Initially there are no matching pair in the matchKB and all the MergeElements
are still unmatched and are partitioned based on the type. Then every MergeElement
goes through comparison and every new matching pair found (marked as red line) will
be put into the matchKB. This process repeats until no more new matching pair can be
found. This concept is similar to the concept of ﬁxed point algorithm in mathematics,
which iterates to ﬁnd a value of a function until the point of convergence. In the next
step, merging will occur, and MergeElement that has any reference to the merge source
(marked with id = 53, 55, 56) will need to be updated to use the merge target instead
(marked with id= 3, 5 and 6).
3. Merge phase (line 11):
For every list of matching sets of MergeElements, merge the MergeElements so that every
matching set contain only one MergeElement. Meanwhile, all the MergeElements which
contain references to the merge source will be updated. For example, if the matching sets
are : {{A, B}, {C, D, E}{F, G}}, the result of the merge will be only one element from
each set, e.g. {A, C, F}. One important thing to note is that if between the two matching
element that needs to be merged, if one of them does not have parent containment, it
will be merged to the one which has parent containment, otherwise if both have parent
containment, the one with the smaller hash values will be merged to the one with the
larger hash values to ensure that the operation is commutative.
4. Output phase (line 12-13):
The output phase is performed by a java class called MergeOutputWriter. Its role is to
combine the merged set and leftover unmatched elements into the output set, then write
them back into the visual elements and visual connections and save it into the diagram.
Regarding the three desirable properties (commutativity, associativity and idempotency) of
weaving operation described in section 3.4, it should be noted that: The ﬂattening operation
and writing back to the output does not aﬀect any of the properties above because it is just
performing one to one mapping from visual element to merge element and vice versa.

1. Commutativity (A ⊗B = B ⊗A)
The ﬁnd-match operation is commutative because the operation only performs equality
matching of two elements based on its attributes and structures (m1.matchValues(m2)
will give the same result as m2.matchValues(m1) and m1.matchStructures(m2) will give
the same result as m2.matchStructures(m1)
The merge operation is commutative because the algorithm always picks the element
without parent and merge it to the one with parent, and in the case when both elements
have parent, the one with smaller hash values will be merged to the one with the larger
hash values such that the one with largest hash values will be the one to remain.
Because the two operations are commutative, the result will be commutative.
2. Associativity ((A ⊗B) ⊗C = A ⊗(B ⊗C))
The result of the ﬁnd-match and merge operation of (A⊗B) is essentially just a set union
of A and B’s elements. ((A ⊗B) ⊗C therefore is just a set union of A and B and C’s
element. Because set union is associative, the output of (A ⊗B) ⊗C will be the same as
A ⊗(B ⊗C)).
3. Idempotency A ⊗A = A
If a diagram A is merged with diagram A, the ﬁnd-match operation will ﬁnd that all the
elements will have its own matching pair of element from the other diagram. The merge
operation will then merge each of those and reduce the output to the same set as the input
(removing duplicates).

4.4.4.2
ConnectionME
For connection MergeElement, the match() and matchValue() operations are designed to have
the same logic as generic MergeElement. However if both names are empty, matchValue will
consider it as a match. This is because it is possible for two connections to be a match if they
both don’t have names, as long as their structure matches (both are connected to elements that
have been matched). Therefore, The matchStructure() implementation only checks that both
connection-ends have been matched.
4.4.4.3
ArrowME
Notice in Figure 4.9 that for Arrow and Control Node, the matching implementation is using
R∧operation. For arrow and control node MergeElement, the matching algorithm is slightly
more complex than the previous two MergeElement type. It is because in addition to having
a match result of TRUE and FALSE, the algorithm will also make use of UNKNOWN match
result. The reason is because for matching of arrow (for example control ﬂow or state transition)
it is possible that the arrow will points to a control node. A control node (for example decision
or merge node) sometimes can have names but typically they don’t. Therefore to match control
node, it will need to rely on the knowledge of the incoming/outgoing arrows have been matched.
However, matching of arrows will require matching of src and tgt which will be the control node
itself, so it is possible to have a cyclical dependency of matching to be satisﬁed.
Consider an example of the two diagrams from Figure. 4.12 where an action goes into a decision
node without any name. For decision node 2 to be matched with decision node 52 it will require
that the incoming or outgoing arrow to be matched, for example arrow 5 and 55 need to be
matched ﬁrst. However, for arrow 5 and 55 to be matched, it will require that action node
1 is matched with 51 and decision node 2 to be matched with 52. In another word, for the
decision node to be matched, it requires the arrow to be matched, and for the arrow to be
matched, it requires the decision node to be matched (cyclic dependency). To resolve this cyclic
48
==PAGE:59==
Chapter 4. Design
dependency, the matching criteria needs to be relaxed so that the matching of src and target
does not necessarily needs to be both TRUE. Instead if matching of src returns TRUE and
matching of tgt returns UNKNOWN, the algorithm will consider it as a match (vice versa when
src is UNKNOWN and tgt is TRUE). In this example, matching of 1 and 51 returns TRUE,
and matching of 2 and 52 returns UNKNOWN, so arrow 5 and 55 will be able to be matched.

but the names are diﬀerent. In this case, during the ﬁrst ﬁnd-match phase, it would have been
registered in the MatchKB based on the result of matchValue() that 2 and 52 are FALSE,
therefore 5 and 55 cannot be matched.

5.1.3
Auto Layout
GEF also comes pre-bundled with a framework for layouting purposes, called ZEST. It contains
various layout algorithm such as spring layout, vertical tree layout, horizontal tree layout etc.
However historically ZEST had been developed as a standalone framework independent of GEF,
and it has a loose-coupling with GEF. Any layouting to be done for GEF has to be done manually
by the developer, by converting the ﬁgures into tree-based structure of layout-entity objects and
passed to the layout algorithm.
The layout implementation implemented in the visual editors are for example: directed graph
layout, grid layout, radial layout, spring layout, tree layout and vertical layout. However, upon
testing, it was found out that auto-layout in ZEST have various weaknesses such as follows:
1. For composite ﬁgures (ﬁgures which contain other ﬁgures as children), the layout algorithm
does not consider all the children, i.e. all the children are just defaulted to be placed in the
location (0,0) relative to its parent.
2. For composite ﬁgures, the bounds of the parent does not get resized automatically even
though the size of the children may extend beyond its parent’s size.
3. Amount of crossing lines does not get optimized by adding connection bendpoints.

5.2.3
Preference conﬁguration page for weaving
A conﬁguration page for weaving is also added under the preference options in RED, which
allow the user to conﬁgure some settings, as shown in Figure 5.7.
The conﬁguration page provides the following functionalities:
1. Edit distance threshold: controls the case-insensitive similarity measure used to compare two
labels of visual elements. Setting to 0 means that only two elements with the same labels will
be matched. If the user would like to relax the comparison, the edit distance can be increased.
2. Auto layout: if ticked, the diagram elements will be put next to the input diagram when it
is merged. When it is not ticked, the diagram elements will maintain their original location,
which may cause the elements to be overlayed on top of each other.
3. Cascade updates: when enabled, any updates to an element in a diagram will also be reﬂected
to the other elements of which it has been weaved from/to.
4.
Allow conﬂicting of sub-element names: when enabled, it will ignore merge conﬂict for
59
==PAGE:70==
5.2. Implementation Areas - Weaving Tool
some sub-elements, for example if class ”Corpus” has attribute ”name” and class ”User” also
has an attribute ”name”, the merge conﬂict will be ignored and the ”name” attribute will not
be matched. Conﬂicts that will be ignored are for example Operations/properties of diﬀerent
classes, use cases of diﬀerent systems and subsystems of diﬀerent systems.
5. Separate multi weave: determine how the weaving wizard should group the elements, whether
grouping them by their type (eg. class diagram, use case diagram, etc) or by their respective
containment folder.
6. Weaving sequence: this option will decide how does the tool picks the next input to be weaved.
For example, when there are 4 diagrams: A, B, C, D to be weaved. Choosing sequential option
will weave the diagram as (A ⊗B), followed by ((A ⊗B) ⊗C) and then (((A ⊗B) ⊗C) ⊗D).
Choosing the tree option will weave the diagrams as (A ⊗B), followed by (C ⊗D) and then
((A ⊗B) ⊗(C ⊗D)).
Both the sequential mode and tree mode have the same number of weaving steps because every
weaving step always consumes two inputs and produces one output. The sequential weaving
sequence however, is more suitable to be used when the user would like to run weaving step-
by-step. The reason is that when the diagrams are weaved sequentially, only the intermediate
result will grow considerably while the other input will always be diagram in its simplest form
which is derived from the requirements. This makes the user easier to track and understand
his/her own actions about which elements will be merged.
The tree option on the other hand is more suitable to be used for re-weaving in batch. The user
most likely has done the weaving previously and would have known that all the input diagrams
can be weaved without any issues. In addition to that, when the tree option is used, merge
executions for multiple diagrams can be done in parallel in several threads to improve the speed,
although that feature has not been implemented yet in this thesis.
Technically, the output from sequential and tree sequence should be the same because the
elements and the connections are compared using equality matching based on their values and
structures.

6.4
Performance
The new functionality developed in this thesis does not cause any noticable performance degra-
dation within the tool. The weave algorithm performs quite fast even with large number of
elements in the input diagram when executed using Eclipse command line. After integration
to RED however, it was found out that weaving a lot of models actually perform quite slow.
After some investigation, it was found that the root cause is the slowness in EMF during data
serialization. Especially when running with a lot of input diagrams, EMF will perform the
serialization in every iteration, therefore slowing the entire weaving process. This weakness
has been improved by changing the serialization to be called only after all iterations have been
completed.

diagrams are weaved together regardless of the diagram type. Figure 6.7 shows the performance

Chapter 7
Conclusion
7.1
Summary
The goal of this thesis is to design and implement a model fragments weaving tool in the
requirements engineering editor tool (RED). This tool is a novel approach which allows the user
to create various type of model fragments which can represent software requirements, and then
allow weaving of those fragments into one large model which can potentially be used to create
the formal model.
In this thesis, the author has discussed about various propositions that the author think is
important and become the motivation to this tool, such as:
1. Software failures - signiﬁcant number of errors in software are mostly attributed to errors
during requirements engineering
2. Ambiguity in natural language - Requirements can be improved if they are accompanied
with pictures/diagrams
3. Natural Language Processing - With the current state of technology, performance of
NLP tool is usually not perfect, and using the tools can be worse than not using them
4. Weaving - Transformation of requirements to model fragments and automated weaving of
those model fragments can help to reduce software model errors.
5. Veriﬁability - Verifying many small model fragments against its corresponding requirements
are easier than verifying large model against large textual documents
6. Trust - Trust can be achieved when model translation is done by a tool that can provide
step-by-step visualisation of the output and is supported with suﬃcient debugging capabilities
7. Traceability - If traceability link is created for every weaving step, it will be possible for the
user to trace from requirements to the model and vice versa from the model to the requirements
8. Synchronization - Synchronizing the updates from one diagram to all the weave source
and output can help to keep the requirements and the model in sync

7.2. Discussion
This tool is created because an existing tool in RED which is based on Eclipse GMF framework
and Prolog program is very diﬃcult to setup and has some feature gaps. The existing tool
created using GMF is very diﬃcult to be customized or extended, and does not provide enough
traceability support. The dependencies on Prolog program also makes it more diﬃcult to setup
and prone to failure whenever there is any version upgrade between the libraries used such as
the Java Prolog connector and the SWI Prolog library.
Chapter 2 in this thesis provides the user with the background information related to require-
ments, requirements traceability and model weaving so that the reader will be more familiar
with those concepts.
In chapter 3, the usage scenarios, and various types of diagrams and their elements are outlined,
which deﬁne the scope of this thesis. It also discuss further about desirable properties that needs
to be addressed from a good model weaving tool.
In chapter 4 and chapter 5, the design of the software and its implementation are then presented
to the reader.
Finally, in chapter 6 the author attempts to evaluate various aspects of the
software quality.
7.2
Discussion
The key challenges with this project are the diﬃculties in designing an algorithm that is simple
to understand, yet still performs quite well for comparison of various types of visual elements
and merging them.
In addition to that, the author has to learn about how to develop visual editor application using
GEF, and using the foundation code that has been developed by the previous Msc. student
(Jeppe) within a short period of time, while dealing with challenges such as integrating changes
with other RED developers. Overall, although the author overcome many challenges, the result
has been quite rewarding and the tool will deﬁnitely bring a nice feature that is probably not
yet available in other requirements engineering tool available in the market.
It still needs to be evaluated further though, on how eﬀective is this tool for being used in
development of large software.
Although the result on LMS case study seems to be quite
satisfactory, there may be cases where further improvements need to be made so that this tool
can be adapted by wider audiences.

Appendix D
FindMatch and Merge algorithm
Algorithm 2 Find-Match procedure
1: procedure findMatch(unmatchedElements, matchKB)
2:
arr = toArray(unmatchedElements)
3:
for i ←0, arr.length do
4:
for j ←i + 1, arr.length do
5:
matchResult ←match(arr[i], arr[j], matchKB)
6:
if matchResult == TRUE then
7:
addMatchingToMatchKB(arr[i], arr[j], matchKB)
8:
listOfMatchTuples ←listOfMatchTuples ∪{arr[i], arr[j]}
9:
else if matchResult == FALSE then
10:
addNonMatchingToMatchKB(arr[i], arr[j], matchKB)
11:
end if
12:
end for
13:
end for
return listOfMatchTuples
14: end procedure
The program just convert the unmatched elements into an array and compare it against itself
(they are split into half diagonally so that comparison does not occur twice). If any matching
pair is found, they will be added into the MatchKB as positive-match list. If non matching pair
is found, they will also be added into the MatchKB as negative-match list.

The algorithm ﬁrst get the list of matching sets from MatchKB. This method automatically
convert the positive matches in MatchKB into the form {{A,B},{C,D,E},{F,G}} and then
proceed to merge them as {A,C,F}. The elements which have parents will take higher priority,
therefore those without parents will be merged into them. If both elements have parents, then
the one with larger hash values will take priority so that eventually only the one with the largest
hash value remains.
