Abstract
The increased global consumption of power and its eects on CO2 emissions has
led to more focus on reducing global warming and ensure future energy security.
The European Union has set ambitious energy targets for 2020. These targets
are also known as "20-20-20". The targets aims to reduce EU's gas emissions
by 20%, raise the share of EU's energy consumption produced from renewable
resources to 20% and to improve EU's energy eciency by 20%.
Smart Grids are considered to be a part of the goals set by the EU commission.
A Smart Grid can ensure that electricity is supplied in an eciently, sustain-
able and secure way. This entails the need to ensure Smart Grid security in
order to facilitate higher eciency, reduce failures and subsequent power out-
ages through fault detection, alerts and mitigation strategies.
This thesis proposes a Smart Grid monitoring and diagnostics aggregator using
visualization. The proof of concept prototype is developed using recent web
application technology interfacing dierent platforms and data sources. A secu-
rity evaluation method is suggested for evaluating Smart Grid units (Distributed
Energy Resources). This method has been based on several components includ-
ing, monitoring of the DER's, its applications, interfaces and communication.
Next a vulnerability assessment of the DER is performed, contributing to the
proposed method with known vulnerabilities and threats. Then an intrusion de-
tection system, monitoring network activity, and based on dened rules, alerts
are red. Finally an anomaly detection method for inspecting the physical units
behaviour including the power production and meteorological station have been
proposed. These components operate as individual security evaluations but are
also used in the total security evaluation of the DERs.

Denmark is also part of these energy targets for 2020 where Denmark's goal
is to reduce gas emissions by 40% which is signicantly more ambitious than
EU and in 2050 Denmark has a vision of becoming completely fossil fuel free
and instead use renewable and sustainable energy [DKEC][p. 27]. In order to

A Smart Grid can ensure that electricity is supplied in an eciently, sustainable,
and secure way. An example could be that a washing machine waits until there
is a surplus of electricity in the Smart Grid before washing, which will make the
electricity usage much more ecient.

1.4
Project Description
The development process of the Smart Grid is still ongoing and therefore many
essential parts are missing. Monitoring and diagnostics are two important as-
pects of a Smart Grid which will help utilities maximize DER performance and
reduce failures and subsequent power outages through fault detection, alerts
and mitigation strategies.
The main focus of this project is to implement an aggregator for the Smart
Grid.The aggregator will compose of a monitoring and diagnosing part which
consists of three main steps: data acquisition, data processing and maintenance
decision making.
A visualization is created in order to present the outcome from the monitor-
ing and diagnostics on a dynamic visualization platform using web technologies.
The visualization is made on an network view 1.1 which results in a dynamic
map containing nodes representing each computer connected to other nodes in
a grid. The second view is based on the electric units mapped onto a grid. The
nodes contains information such produce/consume rate, the state of the unit,
which other node it is communicating with, list of software running on that node.
The monitoring part of the aggregator makes use of techniques, methods and
algorithms to determine how the DER's inside the Smart Grid are communi-
cating. This results in a graphical visualization of the Smart Grid making it
easier for system operators to see how the communication between the DER's is.
Furthermore a datastore is implemented, to ensure that historical data related
to each unit is stored. This datastore ensures that the operator has the ability
to look back in the history of each unit and its components. This entails that
each DER is being observed and its actions are being compared to the other
DERs in the Smart Grid. An example could be that the aggregator monitors
how the DERs in the Smart Grid are reacting to requests from other DERs.
Monitoring is additionally useful when controlling DERs with distributed algo-
rithms. The data from the monitoring can be used to improve the performance
of the distributed control algorithm and thereby maximize the DER operation.
The diagnostics part of the aggregator detects problems with the DER's in
the Smart Grid and helps mitigating the problem by automatically deploying a
repair crew, or notifying the system operators. An example could be that an
alert is shown because a solar panel is damaged and not working properly.
The project contains a scientic study in the area of cyber-physical security.
The scientic study investigates security aspects so that the system itself can
discover attacks. Furthermore the scientic study will contain mechanisms to

of the DERs and its relevant information.
The monitoring must be incorporated into the overall security of the DERs.

The system must be able to detect intrusion caused by persons outside or inside

DERs.

The Smart Grid consists of several units where each unit it connected to a
computer.
In an ICT visualization each computer will act as a node in the

2.1
Smart Grid
A Smart Grid is an intelligent electrical grid that consists of a network of trans-
mission lines, substations and transformers. The Grid is given the name smart
because of the fact that it allows two-way communication between the utility
and the end users. The Smart Grid contains computers, automation, controls
and other technology equipment which will all operate together. The Smart
Grid is also made so that it can handle the quickly changing electric demand
[SMA].
The Smart Grid plays a major role in the eld of future energy resources which
can be split into:
Distribution Intelligence(outage detection and response),
Renewable energy(sustainable energy resources), Smart Homes(appliances and
software to control/schedule), Consumer Engagement(energy-saving programs),

Operation Centers(controlling the transmission system) and Plug-In Electric Ve-
hicles(infrastructure for vehicles) [SMA]
The Smart Grid projects are launched in order to move the energy industry
into a new era of reliability, availability and eciency that will help improve the
environmental and economic circumstances. The advantages related to Smart
Grid include:
• Ecient electricity transmission
• Shorter power outages
• More integration with renewable energy systems
• Better integration with customer-owner power generation systems
• Less peak demand
• Cheaper management and operations costs for utilities
• Lower power costs for consumers
• Security improvements
2.1.1
DER: Distributed Energy Resources
The DERs are small units connected in a grid where the energy is decentralized
meaning that the energy resources are distributed in the grid.
The DERs can dier between either being energy producing unit (Prosumer),
energy consuming unit (Consumer) or energy storing unit. There are many dif-
ferent DERs such as Cogeneration(steam turbines, fuel cells, microturbines etc.),
Solar Power(photovoltaics), Wind Power(wind turbines), Hydro Power(hydroelectricity),
Waste-to-energy(food waste, natural waste etc.), Energy Storage(vehicle-to-
grid, PV storage, Flywheels etc.)

2.5
Intrusion detection
A network based intrusion detection system can be applied on a Smart Grid
to provide protection against existing and future sophisticated security attacks.
The paper [SGI] proposes a model for intrusion detection distributed across a
three layer network (see gure 2.2):
1. Home area network
2. Neighbourhoud area network
3. Wide area network
The Home area network (HAN) is the lowest layer in the Smart Grid, where
all devices are connected in the home consisting of smart meters and metering
modules.
The neighbourhood area network (NAN) is the middle layer in the Smart Grid,
connecting a range of bottom layer networks to each other, called community
networks.
The wide area network (WAN) is the top layer in the Smart Grid, connecting grid
devices like power plants, substation etc. This form of networks are connecting
Smart Grid networks.

Hardware
The backend will communicate with devices in order to collect data and mea-
sured data through web services, storing of data through a database and passing
data through http protocols, the underlying hardware is not considered to re-
quire a heavy computational power.
The visualization application is intended to do the rendering tasks which will
require a browser supporting HTML5 and CSS3 meaning that the hardware
platform must have enough CPU and GPU power to achieve its requirements.
The choice is between X86 machine [INT] and ARM single-board [ARM]. The
X86 is mainly known to operate in todays PCs meaning that it comes with
heavy computational power and features compared to the ARM.
Regular X86 machine The regular machine is bigger, more power consuming
and more expensive compared to single-board computers. The regular machines
are physically bigger and the dierent hardware units are typically placed on
dierent circuit boards. The regular machines do often support the X86 ar-
chitecture which allows most of the popular operating systems like Windows,
Linux etc.
Single-board computer The single-board computer is a small, power ecient
with a single circuit board, meaning that micro-controllers, I/O, mememory
etc. are all placed in a single board. The single-board computers are typically
built with an ARM processor which have a operating systems designed for this
type of hardware architectures. The operating systems designed for this type of
architecture are typically some linux distributions.
The general performance of a single-board computer is much more low compared
to a regular computer but the computational power of a single-board computer
will be enough to meet the requirements related to the solution proposed in this
project.
The single-board computer is the best choice in terms of hardware platform.
The features and hardware requirements here are enough to meet the overall
requirements for this solution. The selection has been made based on the fact
that single-board computers provide enough computational power to solve the
tasks related to this solution. It does not exclude the fact that it is still possible
to use a regular machine.

4.6
Backend Platform
The Backend platform must consist of one application. The platform is going to
communicate with the existing system, SYSLAB located at DTU Risø campus.
SYSLAB oers its data through several communication standards XMLRPC,
OPCUA, RMI and Web servies. As mentioned in section 4.2, the restful web
service communication standard has been selected. The choice has been made
upon major cross-platform programming languages based on the requirements.
By using an API it will provide a more structured and cleaner architecture, it
will be easier to x bugs, add new features and make changes to the existing
code base. The REST paradigm uses HTTP requests and responses, verbs and
status codes in order to communicate. Restful web services use HTTP which
can be consumed by almost all devices makes it much easier to communicate
with and it does not require additional information about the API.
Java The existing software running on SYSLAB is written in Java. Java is
a major cross-platform programming language that has all the necessary fea-
tures and functionalities. Java itself does not support restful web services but
there are other java API's which do. JAX-RS [JAX] is a java api which oers
the functionality in order to create restful web services. The Jersey framework
[JER] extends the JAX-RS API with additional features and functionalities to
simplify the restful implementation.
ASP.NET Web API is a framework that supports its restful services on top of
the .Net framework [ASP]. The .Net framework itself oers a lot of features and
functionalities which can be utilized. The HTTP services can be combined with
the Newtonsoft JSON framework for .Net which enables the use of JSON and
XML formats. The JSON framework enables serialization of objects, deserial-
ization of objects, format switch between XML and JSON. The use of ASP.NET
Web API and Newtonsoft JSON framework ensures that the most fundamental
programming tasks in order to achieve restful web services at a high abstraction
level.
ASP.NET Web API is a Microsoft technology which is only available on top of
a Windows platform therefore it requires an operating system running windows.
The existing system, SYSLAB is operating in a linux environment which makes
it unable to run any microsoft related technology. In April 2014 microsoft an-
nounced that a lot of their technologies including ASP.NET were going to be
open-source [NET]. This will allow ASP.NET Web API projects to be deployed
on a linux platform. The current version is only a "preview" which does not
cover all the necessary functionalities and features .Net provides with.
The proper solution here is to use an open source implementation of ASP.NET
Web Api, Mono [MON]. This will allow a solution based on Mono to be hosted
on the current environment deployed at SYSLAB.

Intrusion detection
This subsection covers the intrusion detection mechanism. The intrusion detec-
tion will provide essential information about the network activity of the mon-
itored DERs.
The IDS will be an independent component connected to the
network 4.20 and monitor ingoing/outgoing network activity.
The fact that
network activity is monitored will be an essential part in keeping a high security
level of the system. The monitoring will provide alerts whenever unusual or
unexpected activity is detected.
Intrusion detection systems are often split in two groups; anomaly detection and
"known" attacks detection. The "known" attacks detection operates by com-
paring collected information with a large database containing already "known"
vulnerabilities and attacks.
The anomaly detection has to be set up specic for a given network structure.
The system administrator has to know exactly the expected behaviour of the
units that is going to be protected. It will be necessary to dene the dierent
things related to anomaly detection like protocol, packet size, communication
direction etc.
The IDS can be further split into two types, host based (HIDS) or network
based (NIDS). The dierence between these two, are that NIDS are placed at a
point to detect intrusions in the system meaning that the travelling packets will
be matched with intruder signatures. Whenever suspicious activity is monitored
and alarm will be triggered. Whereas the HIDS is only targeted hosts and will
monitor the trac going in and out from these hosts. The hosts will be provided
with an agent running on it and whenever suspicious activity is monitored an
alarm will triggered.
The IDS can be further categorized into passive and reactive systems.
The
passive IDS will only monitor the trac and only trigger an alert when suspi-
cious behaviour is seen. The reactive systems extends the functionality from the
passive systems with the prevention aspect meaning that it will try to prevent
the suspicious activity.
The intrusion detection systems can be based on two types of terminologies; sig-
natures and anomalies. The signatures is dened by pattern matching whereas
anomalies is dened by learning from normal behaviour and then look after
changes from the normal behaviour.

it easier to order the vulnerabilities.
The OpenVAS vulnerability manager primarily consists of a scanner and a man-
ager. The scanner uses vulnerabilities from the online NVT against the target.
The manager has the job to keep track of the vulnerabilities by using a database
where results are stored.

The overall solution architecture is an interaction between several parts; Nagios

The sys-

client-server system provides a service from a central point, the server, meaning
that several clients can make requests to the server which the server responds
to. This architecture ts to the Smart Grid organisation and its devices. The
Smart Grid is build up of several devices, each providing services like a server.

also acting as server and client. The server role is assigned whenever a browser
makes a request and as a client when requesting data from the data aggregator.
The client-server architecture places the workload on the server where it is
responsible for the application management, data management and database
hosting. The use of a two-tier architecture versus a three-tier architecture will
make it more simple and easy to modify. The two-tier architecture can be fur-
ther split into a thin-client model and a fat-client model, see gure 5.1. The use
of a thin-client will simply place the application management part on the server
side making it more "lean". The model used in this project is the thin-client
which will ensure easier maintainability meaning that application on the server
will change whenever an update/change is needed.
The architecture selected for this project is the two-tier architecture. The gure
in 5.2 represents the two-tier architecture used with a thin-client model. The
client is only only based on javascript and html, which requests data from the
server. The server is split into two platforms, the visualization application and
the data aggregator. This ensures that they can be placed on dierent machines

6.3
Summary
This chapter has dealt with the proof of concept implementation of the solu-
tion containing the Visualization Application frontend and the Data Aggregator
backend. The implementation has been derived from the design choices pre-
sented in chapter 5.
The implementation of the Visualization Application has ensured that it is pos-
sible to give a graphical representation of the DERs including their status, revi-
sion, applications, communication interfaces and communication history. Fur-
thermore a presentation of the security parameters. The security parameters
have also been included in the multi-line charts which gives an overview of the
historical security evaluation.
The Data Aggregator has been implemented containing several parts. The na-
gios system for retrieving information about the DERs from a decentralized
point. The web services of the DERs have been used in order to retrieve in-
formation about the DERs, the revisions, the applications, the communication
interfaces and the communication history. The intrusion detection system has
been integrated into the solution contributing with detected intrusions. The
intrusion detection system has been setup with a ruleset in order to detect pos-
sible intrusions. The vulnerability assessment system has been integrated into
the solution contributing with the current detected vulnerabilities and threats
for each DER. The anomaly detection system has been implemented using Gaus-
sian distribution in order to predict possible anomalies in the physical security
of the DERs.

Implementation
The Data Aggregator backend and the Visualization Application have been
implemented using a MVC architecture. This will ensure low coupling high co-
herence. It will be easy to replace components and modify the existing solution.
The Data Aggregator backend has been implemented using ASP.NET Web API
supporting the MVC design pattern. The frontend Visualization Application
has been implemented using AngularJS which also follows a MVC design pat-
tern. The Visualization Application has been implemented using a thin-client
approach making it totally independent from the backend. The Visualization
Application will only deal with visualizing the data.

Visualizing DER Security
This case study deals with the demonstration of the security of some selected
units inside the SYSLAB network. The demonstration will look into monitor-
ing of the DERs, monitoring of applications, monitoring of the communication
interfaces, intrusion detection, vulnerability assessment and physical security of
the unit. Each part will be contributing to an overall assessment of the unit
resulting in the security level. The security assessment can be used to determine
whether a unit is behaving as expected or not which could indicate that a unit
is being controlled by third party actors, is malfunctioned or none of them.

units are located at DTU Risø Campus, producing active power measured in
kWh (kilowatt-hour).

read looking in the generated vulnerability assessment report (see C.1). The
