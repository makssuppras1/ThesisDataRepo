Abstract
In large, mission-critical applications with requirements on high availability, it is critical to detect and
ﬁx the most signiﬁcant amount of code errors before applications are deployed. The tool Dialyzer can
be used to detect discrepancies statically in applications written in Erlang and Elixir. However, some
errors are not caught and may cause a server to crash at runtime. Uncaught type errors especially
occur in projects using the gen server behavior, where Dialyzer cannot detect discrepancies between
the server implementations and the exposed client APIs. To this e↵ect, this paper describes the
design and implementation of an extension to Dialyzer that signiﬁcantly improves the number of
concurrency-related gen server discrepancies detected.
Like Dialyzer, our tool guarantees a fast
analysis with sound warnings, without relying on users to annotate their code. We have applied our
tool to several real-world applications, including the production codebase of Motorola Solutions. The
results show that our tool can detect multiple discrepancies even in seemingly bug-free, high-quality
codebases.

Introduction
Concurrent and distributed systems are a known standard within large industrial applications. Er-
lang and Elixir are popular open-source, functional programming languages designed to build dis-
tributed and fault-tolerant systems. They are often used for mission-critical applications, and it is,
therefore, crucial to detect and ﬁx the largest number of code bugs before applications are deployed.
Erlang and Elixir do not, however, have a classic type system but instead, use a tool called Dialyzer
to provide static type checking. It uses the concepts of success typings (described in section 2.4.2)
to catch discrepancies in the programs without reporting any false positives. Consequently, only
real code errors are reported, but some errors can go undetected and occur at runtime, resulting in
a server crashing. E.g., a common error that can occur is a type discrepancy between a server’s im-
plementation and the public APIs that are exposed. In Erlang and Elixir, the gen_server behavior
is commonly used to write these client-server applications.
In this project, we aim to improve the detection of concurrency-related errors in Erlang and Elixir
programs by providing a tool that extends upon Dialyzer to catch more discrepancies when using
the gen_server behavior.
The motivation for the project comes from examples of discrepancies provided by Motorola Solutions
that Dialyzer does not detect. Therefore, some of the choices we made in this project were made
with Motorola Solutions in mind, as we want them to be able to use the produced tool. Furthermore,
they allowed us to test the improvement in detecting discrepancies by our tool, by letting us analyze
parts of their codebase that run in production applications. The programs provided by Motorola
Solutions are written in Elixir, and though supporting Elixir is part of our scope, we will almost
exclusively focus on Erlang in the examples throughout the paper. Elixir merely builds on top of
Erlang, providing more productive tooling and extensible design. However, it is still run on the
BEAM virtual machine, which is also used to run the Erlang language. Therefore, the concepts we
explain using Erlang generally carry over to Elixir, and Dialyzer itself is written in Erlang.
Dialyzer has some notable characteristics that were, and still are, important when it was integrated
into the Erlang/OTP, mainly[1]: 1) It should report no false positives (soundness). 2) It should
be fast. 3) It should require little to no annotations or modiﬁcations from the user. Since Dialyzer
is well established in the Erlang community, it is crucial that our produced tool still satisfy these
requirements. Otherwise, our tool would have a low chance of being accepted by the community in
the future.
We now enumerate the main objective and requirements of this project. We will further elaborate
upon them in the analysis, and we will refer to them throughout the paper:
Main objective:
• Improve the detection of discrepancies between client APIs and server implementations in
Erlang and Elixir, when using the gen_server behavior.
1
==PAGE:6==
Requirements:
1. The tool should remain sound, i.e., no false positives.
2. The tool should remain fast.
3. The tool should require minimal, preferably no, annotations or modiﬁcations from the user.
To achieve the main objective, we will introduce the concept of MFAAA (Module, Function, Arity,
Atom, Arity) - an extension to the MFA (Module, Function, Arity) used in Dialyzer when performing
the type analysis. The MFAAA allows us to analyze the server implementations of the gen_server
behavior with more precise type information, to catch the largest amount of discrepancies.
The ﬁrst part of the report covers the background of this thesis. It is essential to understand the
project’s domain and describe the ﬁndings of researching Erlang, Elixir, and Dialyzer, as well as
other related work.
We will analyze the problem and deﬁne the project’s scope based on these
ﬁndings. Then, we will discuss the choices we made during the design and implementation of the
tool, as well as analyze and present measurements of the improvement to justify that we have met
the main objective and requirements. Lastly, we will evaluate the entire project and address how
the tool and Dialyzer, in general, could be improved with future work.

Background & Related work
This section will introduce the background and theory that make up the domain of the project.
Section 2.1 introduces Erlang and the gen_server behavior. In section 2.1.1 we describe the concepts
of concurrency in Erlang, and section 2.1.3 explains how and why the gen_server behavior is so
useful. We will introduce the analysis tool Dialyzer in section 2.4, as well as the theoretical concepts
on which Dialyzer bases its analysis.
At the start of this project, the overall goal was to improve the static detection of concurrency errors
in Erlang and Elixir programs. To understand the current possibilities and limitations in Erlang
and Elixir applications, it was necessary to thoroughly research the concepts and theories that make
up the project’s domain.
Domain. The domain of this project mainly consists of Erlang and Elixir, type systems in general
and in Erlang, concurrency in general and in Erlang, the gen_server behavior, and Dialyzer.
We started by studying Erlang and Elixir, including how they relate, how their type systems work,
and why they excel at concurrent and distributed programming. We then studied Erlang’s included
behavior modules, particularly the gen_server module for implementing generic servers. The most
signiﬁcant part of our domain is Dialyzer. We researched all the components of Dialyzer: the theory
behind its success typings, the source code, which discrepancies it detects and which it does not.
This knowledge helped us decide which approach to take, and the sections below document our
theoretical ﬁndings.
2.1
Erlang and gen server
Erlang is a general-purpose programming language used to build massively scalable soft real-time
systems with requirements on high availability[2].
Erlang is usually used interchangeably with
Erlang/OTP or Open Telecom Platform (OTP), which contains the Erlang runtime system itself
and multiple components and design principles that developers can use to build powerful Erlang
programs.
Erlang is a dynamic language and does not have a classic type system (see section 2.3). Instead, it
uses static analysis tools like Dialyzer to provide type checking, which we will describe in section
2.4.
Erlang is similar to other popular functional programming languages in that it has immutable data
and powerful pattern matching. What sets Erlang apart is the fact that it has built-in support for
concurrency, distribution, and fault tolerance. These principles are incredibly powerful, and while
most modern programming languages have adopted them, the entire core of Erlang is built around
these concepts. Therefore, they are especially well integrated, which is one of the main reasons
developers still use Erlang.
2.1.1
Concurrency in Erlang
Concurrent and parallel computing refers to programs where multiple threads can execute during
the same time period. They are often confused with each other, the di↵erence being that in parallel
3
==PAGE:8==
computing, the execution of the threads occurs at the same physical instant. Parallel computing
can, therefore, only be achieved using multiple processors, cores, or computers and not on a single
processor (a processor with only one core). In concurrent computing, the execution merely occurs
during overlapping time periods. Each processor usually creates an illusion of true parallelism by
rapidly switching between threads running di↵erent tasks. However, at a single clock cycle, only one
task is executed. Despite this, concurrent programs may be executed in parallel, for example, by
assigning each process to its own processor or core of a multicore processor, or by distributing the
program across a network.
While concurrent and distributed programming is possible in most modern programming languages,
it is not always a simple job. Usually, one needs to consider things such as threading, shared memory,
locking, and semaphores to avoid race conditions. In Erlang, concurrent programming is made much
simpler by its usage of processes.
Processes. Erlang excels at concurrent and distributed programming because Erlang applications
consist of very lightweight processes that run in the Erlang runtime system. Each Erlang process is
a single thread of execution, and they share no data with other processes. This approach enables
distribution and fault tolerance since multiple instances of the process can exist on multiple machines.
Every time the state of an instance is updated, the other instances of the process are notiﬁed and
updated. If an instance’s server crashes, the other instances can continue handling incoming requests,
which is why Erlang systems are fault-tolerant. Also, more instances of the processes can easily be
created, making the system massively scalable.
Spawning a new process in Erlang is as simple as calling spawn:
Pid = spawn(Module, Name, Args)
This creates a new process with the process identiﬁer (pid), Pid, and starts executing the code in
Module:Name(Arg1, ..., ArgN)
where N = length(Args). The pid of the process can now be used to pass messages to the function.
Message passing. The processes of an Erlang application are entirely independent and can only
communicate using message passing. Sending a message to another process occurs almost instantly,
and the execution of the sender is not blocked since the send operation is asynchronous and non-
blocking. Each process has a queue where incoming messages are stored until the process can handle
them, sort of like a mailbox.
To send a message to a process, the send operator ! is used:
Pid ! Message
This sends the message Message to the process with (pid), Pid. A message can be sent to multiple
processes using the send operator multiple times:
Pid1 ! Pid2 ! Pid3 ! ... ! Message
A process can listen for messages that has been sent to it, using the receive keyword followed by
pattern matching on the message received. Below is an example of a function listening for messages
that are either ok or error:

2.1.2
The Erlang Virtual Machine (BEAM)
What makes all of the features that make Erlang excel possible, is the Erlang Virtual Machine
(BEAM). It is the virtual machine that is used to run Erlang code in the Erlang/OTP and is part
of the Erlang Run-Time System (ERTS). The ERTS is what compiles the Erlang source code to
bytecode, which is then run on the BEAM. Files compiled to bytecode to be run on the BEAM
have the .BEAM ﬁle extension. The Elixir programming language (see section 2.2) is also compiled
to BEAM ﬁles, and can therefore leverage the Erlang Virtual Machine to run massive, concurrent
systems.
2.1.3
Client-server applications: The gen server behavior
Now that we have introduced the idea of concurrent and distributed programming in Erlang, we
want to introduce the gen_server behavior, which is also a signiﬁcant part of our domain. Listing
5 shows an example of a server managing a dictionary resource, implemented using the gen_server
behavior.
A common structure used in distributed computing is the client-server model, which clearly distin-
guishes the providers of a resource or service, the servers, and the service requesters, known as clients.
The clients want to share a common resource, with the server being responsible for managing the
resource. This architecture is often associated with the client and server running on di↵erent com-
puters separated by a network. In the case of the client-server model in Erlang, the words ”client”
and ”server” are only used to identify their roles: the client wants to access a resource that the
server manages. The client and server are simply two separate Erlang processes, using the message
passing explained previously to communicate. They can therefore run on the same computer or two
di↵erent computers. Client-server applications can be programmed in Erlang and Elixir using the
gen_server behavior. This behavior contains callbacks that help the programmer implement the
generic parts of the server.
In the following, we will explain why the gen_server behavior is so useful and how developers can
use it to create the server of a client-server application. We will provide an example that separates
the server into a generic part common to all servers and a part containing business logic. First, we
will create the generic part of the server ourselves. Second, we will let the gen_server behavior
handle the generic part. The example is heavily inspired by the example in chapter 4 of the Erlang
documentation[3].
5
==PAGE:10==
Creating the generic part of the server without gen server. The example problem consists
of a resource, which is a simple dictionary of key-value pairs. The server is responsible for managing
the dictionary and provides ways for the clients to interact with the dictionary. The goal now is to
divide the functionality of the server into two di↵erent modules:
• The server module: The generic part of the server that contains all the code that deals with
spawning processes and sending and receiving messages.
• The dictionary module: Contains all the code that has to do with the implementation of the
dictionary, e.g., adding and getting values from the dictionary.
The server module is the generic part of the server, which is common to all servers. It can therefore
be reused to build other client-server applications. It contains no business logic of the dictionary
resource, as this is all contained in the module dictionary. For example, to create a new server to
manage a queue resource, only the module containing the business logic of the queue needs to be
added.
The dictionary in our example, will be implemented as a list of tuples of the form {Key, Value}.
The dictionary module will contain the following functions:
• dictionary:start() -> true, starts the server.
• dictionary:stop() -> true, stops the server.
• dictionary:add_value(Key, Value) -> ok,
adds the given value to the dictionary with the key, Key.
• dictionary:get_value(Key) -> {value, Value} | not_found,
gives a tuple containing the value with the given key, or not_found if the key is not present
in the dictionary.
Naturally, the server must contain ways to start, stop and handle errors. However, let us only look at
the most generic part of the server. A single function, f, can describe how the server should update
its state in response to events. The server’s internal state, in this case, the state of the dictionary,
can be described by the state variable S. In order to add and get values from the dictionary, the
server must be able to receive a query Q, update its state accordingly to S’, and ﬁnally send a

2.3
Type systems
A crucial part of reducing the possibility of bugs in computer programs is the type systems that
usually are part of programming languages. A type system consists of rules that assign a type to
the various components of a computer program. For example a variable of type string or a function
that returns an array of integer. This approach helps prevent type errors, which are discrepancies
between the data types of the program’s functions, constants, and variables, e.g., adding a string
to a boolean, since there is no deﬁned way to handle this case. Checking that the components
11
==PAGE:16==
of a program are used consistently regarding the data types can happen either at compile-time
(statically), at runtime (dynamically), or as a combination of both.
2.3.1
Static type checking
Languages that use static type checking are known as statically typed languages, and the type safety
of these programs is veriﬁed by analyzing their source code. This is made possible by having the
programmer explicitly state the data types of the constructs they create. A program is guaranteed to
satisfy some properties of type safety for all inputs if it passes a static type checker, thus minimizing
the number of type errors that can happen when the program is executing. Such a program is called
a well-typed program, and to quote Robin Milner:
”Well-typed programs cannot ’go wrong’”[8].
Of course, this famous quote must be understood in the context of type safety. Plenty of things can
go wrong in well-typed programs, e.g., null pointer exceptions and division by zero. However, type
errors will never occur at runtime and cause an error.
Popular statically typed languages include programming languages like Java and C#.
Most static type systems are very conservative by nature, in the sense that they will reject programs
that will never encounter a runtime error. This can be demonstrated with the following example of
a program containing the statement:

However, this assignment occurs in the else
branch of the if-else-statement, with the condition being true, which is by deﬁnition always true.
The else-branch will therefore never be reached when executing the program, and the assignment
will not give a runtime error. Despite this, the compiler will give the following error:
Cannot implicitly convert type ’int’ to ’string’.
Most type checkers will call the statement ill-typed because it can be diﬃcult or even impossible to
determine statically if the else branch will ever be executed. The if-statement condition might be
very complex, making it nearly impossible to detect if it will always be true. Therefore, the program
is rejected, ensuring that the statement never gives a runtime error. This example introduces the
concepts of soundness (the type system will never reject a program that will not encounter runtime
errors) and completeness (the type system will reject all programs that will encounter runtime
errors)1. The concepts of soundness and completeness in type checking, will be explained more
thoroughly in section 2.4.1.

2.3.2
Dynamic type checking
In contrast to statically typed languages, dynamically typed languages verify the type safety of their
programs at runtime. This concept is generally implemented by associating a runtime object with
a data type during the program’s execution. Therefore, a program that has not passed a static
type checker is not guaranteed to satisfy a set of properties for the data types of the constructs in
the program. That is, the program may, by deﬁnition, fail at runtime. Most dynamic languages
accommodate this fact by providing ways to anticipate and recover from these errors when they are
encountered.
Erlang is a dynamically typed language, and type errors may therefore occur at runtime. To reduce
the number of these errors, however, Erlang developers can use the tool Dialyzer (see section 2.4)
to catch as many discrepancies as possible before the code is deployed. This is exactly why im-
provements to Dialyzer, including those we introduce in this project, are crucial in helping Erlang
programs become safer and more reliable.
A common question might be when one would ever choose a dynamically typed language over a
statically typed language, and this question can lead to intense discussion in the programming
community. However, there are some cases where programmers may prefer not to explicitly state
the data types of the constructs in their programs. Some examples include, but are not limited to:
• The codebase is small and encapsulated, with few people working on it; explicitly stating types
is time-consuming and not worth the e↵ort.
• The core of the problem is in logic and not in the data types themselves; stating types explicitly
takes focus away from the actual problem.
Since type declarations do not need to be stated by the programmer, development can progress more
rapidly in a dynamically typed language. There is also never a need to convince the type system
that what one is doing is correct, which can sometimes be frustrating. However, the freedom of
not having to state the purpose of one’s programs through type annotations has some drawbacks:
fewer trivial errors like typos are caught by the compiler, and the code can become more diﬃcult
to read and maintain. Not having explicit type declarations can make it diﬃcult to make sense of
a piece of code written years ago, often by another developer. Comments may aid in this situation,
but relying on developers to write fulﬁlling comments to all their code and maintain them as the
codebase changes, is a lot to ask. Again, this is where a tool like Dialyzer can help bring clarity to
the code by automatically inferring and annotating the types in the program.
2.4
Dialyzer
We now introduce the static analysis tool that is Dialyzer. Section 2.4.1 deﬁnes how we interpret the
concepts of soundness and completeness in type checking. We then introduce the concept of success
typings in section 2.4.2 and describe how Dialyzer’s analysis works in practice in section 2.4.4.
Erlang and Elixir are dynamically typed languages that determine variable and function types at
runtime. Therefore, introducing a static analysis tool becomes signiﬁcantly more challenging than in
statically typed languages. To introduce the static analysis concepts into Erlang, Tobias Lindahl and
Konstantinos Sagonas developed the initial version of Dialyzer in 2004 (DIscrepancy AnaLYZer for
13
==PAGE:18==
ERlang programs)[1], along with the concept of success typings[9]. Today Dialyzer is an integrated
part of the Erlang/OTP, which is available with Erlang installed.
Lindahl and Sagonas developed Dialyzer with these important criteria in mind[1]:
• Dialyzer should provide sound warnings rather than being complete.
• Dialyzer should be fast.
• Dialyzer should work without guidance or modiﬁcations from the user, i.e., the code should
require no function annotation or modiﬁcation for Dialyzer to work.
When they introduced Dialyzer, Erlang was already a well-established programming language used
by large telecommunication companies. The developers might be discouraged from using the tool if
it reported warnings in parts of the code, they had seen run successfully in production for a long
time. Dialyzer should therefore provide sound warnings. Also, they might be less inclined to use
the tool actively, if the analysis was very slow and required too much e↵ort and code changes for
the tool to work.
Soundness and completeness are relevant when discussing type systems. In the next section, we
will formally deﬁne how we interpret the concepts of soundness and completeness in type checking
throughout this paper. With that established, we will dive into the algorithm and principles used in
Dialyzer. Finally, we will discuss how this is put together and carried out in practice. The description
of success typings, and the examples used to explain this, are heavily inspired by Lindahl’s and
Sagonas’ paper on success typings[9].
2.4.1
Soundness and Completeness of Type Checking
Important properties of any type system are the notions of soundness and completeness. We will
refer to these properties throughout the paper and explain the interpretation of the two terms we
will be using.
First, we will establish what false positives and negatives mean. A false positive warning is a warning
reported that is not a real code error and will never occur during runtime. On the other hand, a
false negative warning is a real code error, that the type checker does not catch and will cause a
runtime error.
Typically, when discussing type systems, compilers, and type checkers, these terms will be used to
describe the properties of the tool. Di↵erent pieces of literature have di↵erent interpretations for
soundness and completeness, but one thing is always persistent. The relation between soundness
and completeness is always the same.
Two common interpretations of soundness and completeness are the following.
Notice how the
interpretations merely reverse their respective deﬁnitions of sound and complete:
14
==PAGE:19==
Interpretation 1:
• Soundness: A sound type system will reject all programs that will encounter a runtime error
(no false negatives).
• Completeness: A complete type system will never reject a program that will not encounter a
runtime error (no false positives).
Interpretation 2:
• Soundness: A sound type system will never reject a program that will not encounter a runtime
error (no false positives).
• Completeness: A complete type system will reject all programs that will encounter a runtime
error (no false negatives).
Interpretation 1 is most used when reading about compilers and type checkers in general. However,
we have seen both interpretations used in multiple academic articles and believe both are valid.
This paper will refer to soundness and completeness based on interpretation 2. Even though this is
not the most commonly used in the literature, we saw interpretation 2 ﬁrst. Interpretation 2 is the
interpretation that Dialyzer uses in its documentation. The following is a direct quote from Erlang’s
webpage regarding Dialyzer: ”Dialyzer bases its analysis on the concept of success typings, which
allows for sound warnings (no false positives).”[10]. As they deﬁne sound as no false positives, it
implies that completeness should read no false negatives.
2.4.2
Success typings
Dialyzer uses success typings to infer types. Tobias Lindahl and Konstantinos Sagonas introduced
success typings[9] as a new way to infer types in dynamically typed languages, without having to type
annotate programs. They based success typings on the idea of an unrestricted sub-typed system.
It was necessary because a uniﬁcation-based type system in the tradition of Hindley and Milner
would require code rewrites to work - which was not a realistic approach as an introduction to a
type system for Erlang.
Uniﬁcation based type systems. In the tradition of Hindley and Milner, uniﬁcation-based type
systems try to solve type constraints in the form of ↵= β. Here ↵and β are arbitrary types.
Subtyping systems. Contrary to Hindley and Milner-type systems, subtyping systems try to solve
constraints in the form of ↵⊆β. It makes subtyping systems more general than Hindley and Milner
type systems since all types expressed in Hindley Milner can also be expressed in a subtyping system.
The following example shows a limitation of a uniﬁcation-based type system:
1
send(Pg , Mess) when
is_atom(Pg) ->
2
global:send(Pg , {send , self (), Mess});
3
send(Pg , Mess) when
is_pid(Pg) ->
4
Pg ! {send , self (), Mess}.
Listing 7: Showing an example of a subtyping system implementation.
In the example above, Pg can be either an atom or a pid (process identiﬁer). Because of this, a
constructor-based type system like Hindley and Milner would not be able to infer types for the
15
==PAGE:20==
function send/2. It would be necessary to create a declarative super-type to match on Pg. This
declarative type should have constructors for both pid and atom to properly match the expression,
which alone requires much rewriting in the code, which again is not acceptable for large Erlang
applications.
Instead, subtyping systems come to the rescue in this given situation. A subtyping system that
allows for the disjunction of types could infer the type of Pg as atom ￿pid.
The general idea of success typings. Unlike statically-typed systems, success typings are not
about guaranteeing complete type safety. The concept of success typings tries to capture the largest
set of terms for which it can guarantee that type errors will occur at runtime. It does so by using
an algorithm that infers the compliment, namely an over-approximation of all the possible types
a function can evaluate to. The domain of the function signature includes all possible values the
function can accept as parameters, and its range includes all possible return values for this domain.
The largest type of success typings is the type any while the smallest type is the type none. Fun-
damentally, the concept of success typings di↵ers from statically typed systems, as success typings
capture all intended usages of a function, even those not intended by the user. It means that a
function invoked with types not allowed by its success typings is guaranteed to fail. Furthermore,
this makes success typings great as a documentation tool, as it will capture all possible uses of a
function - no matter how unintended.
We now compare the types inferred by success typings and a Hindley-Milner based type system with
the example below:
1
and(true , true) -> true;
2
and(false , _) -> false;
3
and(_, false) -> false.
Listing 8: Implementation of and/2.
A static type checker based on Hindley-Milner would have to infer the type: (bool(),bool()) →bool().
This is a general problem with static type checkers, namely that it limits the intended usage of the
function. E.g., the function invocation and(false, foo) will succeed and return false - however,
this will not be allowed in a statically typed language.
On the contrary, success typings will optimistically infer the type: (any(),any()) →bool(). Notice
how this will not catch the runtime exception thrown when invoking and(foo, bar). It will, how-
ever, catch any wrong pattern matches on the return type, e.g., 5 = and(false, true) will not be
allowed by success typings.
The formal deﬁnition for success typings can be seen below:
Deﬁnition 1 A success typing of a function f is a type signature, ¯↵→β, such that whenever
an application f(¯p) reduces to a value v, then v ∈β and ¯p ∈¯↵. [9]
In other words, a success typing of a function f is a type signature for the function that expresses
all possible domains and values for that function.
Success typings work by ﬁrst generating a set of constraints.
The solution to these constraints
16
==PAGE:21==
eventually ends up being the success typings. If no solution is possible for the set of constraints,
then there is a type discrepancy, and we infer the type none for the function.
The algorithm to infer types.
The algorithm that success typings use to infer the types of
constructs in a program consists of:
1. Generating a call graph based on the modules to be analyzed.
2. Generating type constraints that the program must satisfy
3. Solving the generated type constraints. The solution to the generated constraints is the success
typings.
4. Reﬁning the inferred types by a forward-ﬂowing analysis.
Success typings work by generating the constraints that the constructs of a program must satisfy
and then solving the constraints to end up with an inferred type of that speciﬁc construct. The
modules to be analyzed are transformed into a directed graph before the constraints are generated.
This directed graph is called the call graph. Each function is a node in the call graph, and there
is an edge e(f,g) if function f depends on function g. The call graph is condensed to a graph of
strongly connected components (SCC) (e.g., if function f and g are mutually dependent, they are
condensed to a single node in the graph), so it ends up being a Directed Acyclic Graph (DAG). The
graph is sorted topologically, and the analysis infers success typing for the functions by analyzing
the nodes (the SCCs of the call graph) in a bottom-up fashion.
An example of a type inferred by success typings can be seen in the example in listing 9 below:
1
%% (integer () | list ()) -> integer () | atom ()
2
foo(X) when
is_integer (X) -> X + 1.
3
foo(X) -> list_to_atom (X).
Listing 9: Example of a type inferred by success typings.
The inferred type (integer() | list()) -> integer() | atom() speciﬁes that function foo/1
accepts either integers or lists and produces integers or atoms. There is no mapping that shows
that the function returns integer when called with parameter of type integer, even though this
seems obvious looking at the implementation. This is a design-choice made by Tobias Lindahl and
Konstantinos Sagonas to prioritize simplicity and readability instead of expressiveness. This means
that 5 = foo([4]) is allowed by the success typing, but it will result in a runtime error. The success
typings will however reject all incovations with other parameters than lists or integers, and it will
only allow pattern matches of integers and atoms.
Reﬁning the inferred types. We have now established that success typings is a valuable type
inference system for dynamically typed languages - especially with the restrictions and goals it sets
to meet. The algorithm described above already generates a very general type information. As
stated, the inferred type information captures all possible uses of a function and not necessarily the
intended uses. As an example, we will look at this tail-recursive function that calculates the length
of an arbitrary list in listing 10 below:
17
==PAGE:22==
1
length_1(List) -> length_2(List , 0).
2
3
length_2([], N) -> N;
4
length_2([_|T], N) -> length_2(T, N + 1).
Listing 10: Example of a tail-recursive function.
The call graph dictates that length_2 should be analyzed before length_1. The success typing
for length_2 will take a very general form because of the ﬁrst clause of the function. This results
in an inferred type of (list(), any()) →any(). Secondly length_1 is analyzed, and since it will
utilize the information that length_2 returns any, and accepts list as the ﬁrst parameter, it will
have its type inferred to (list()) →any(). This very general type information inferred for the two
functions is technically not wrong, but it is not very speciﬁc and does not capture the developer’s
intention. The success typings algorithm uses a dataﬂow analysis to reﬁne the success typings. The
dataﬂow analysis analyzes the functions from caller to callee, which is the opposite of the initial
success typings algorithm. Furthermore, the analysis takes its starting point in escaping functions
(i.e., functions that are available outside the module deﬁned in). Considering the length functions
again, we assume that only length_1 is an escaping function, and only length_1 calls the auxiliary
helper function length_2.
The dataﬂow analysis will start by analyzing length_1. When length_1 calls length_2, the analysis
will pause and wait for the analysis of length_2. This time however, the function, length_2, will
be analyzed with the initial type information that is passed from length_1, namely (list(), 0). The
analysis uses the new type information to continue the analysis. Eventually, it will call length_2
again, this time with the type information (list(), 1).
The analysis continues while new type
information is being parsed into the function or until a ﬁxpoint is reached. In this case, a ﬁxpoint
will be the union limit for integers. The example in listing 11 below shows the analysis steps for a
union limit of 3.
1
length_2(list (), 0) : 0 | length_2(list (), 1),
2
length_2(list (), 1) : 0 | 1 | length_2(list (), 2),
3
length_2(list (), 2) : 0 | 1 | 2 | length_2(list (), 3)
4
length_2(list (), 3) : integer ().
Listing 11: Showing how the information is expanded until the union limit of 3 of is reached.
The example shows how the type information of length_2 iteratively expands. When the analysis
reaches the union limit of 3 it will infer the super type integer and stop this part of the analysis.
From here on, length_1 will continue its analysis; now, with the reﬁned success typing of length_2,
it will receive a newly reﬁned success typing of its own, namely (list()) →integer().
The reﬁnement part of the success typings is potent, as it better captures the developer’s intention
by analyzing actual uses of non-escaping functions. It is, however, important to mention that if
we export the function length_2, making it an escaping function, the analysis will not reﬁne the
success typing, and the inferred type will be (list(), any()) →any(). It makes sense, as Dialyzer
cannot determine and analyze all possible uses of length_2 when it is an escaping function.
The formal deﬁnition for reﬁned success typings can be seen below:
18
==PAGE:23==
Deﬁnition 2 Let f be a function with success typing (¯↵) →β. A reﬁned success typing for f is
a typing of the form (ˆ↵′) →β′ such that
1. ˆ↵′ ⊆ˆ↵and β′ ⊆β, and
2. for all ˆp ∈ˆ↵′ for which the application f(ˆp) reduces to a value, f(ˆp) ∈β′ [9]
In other words, a reﬁned success typing is a more speciﬁc version of the general success typing. Since
the reﬁned success typing domain is restricted, the value of the reﬁned success typing might also be
restricted.
Overall, success typings provide useful type information, and it does not rely on any speciﬁcations.
Although the algorithm seems to infer vast types, the reﬁnement of success typings narrows the
usage of all non-escaping functions down, allowing the algorithm to provide helpful information in
most cases.
2.4.3
Persistent Lookup Table
Before we describe how the notion of success typings works in Dialyzer, we will brieﬂy introduce the
Persistent Lookup Table (PLT), which is an essential part of the Dialyzer analysis.
Dialyzer uses and stores inferred type information during the analysis in the PLT. The PLT is a
collection of tables in the Erlang built-in Term Storage (ETS)[11]. The PLT is a key-value dictionary,
where the key is a composite key consisting of the tuple with: Module, Function, and Arity (MFA).
E.g., a function length in the module list, which takes a single argument, will be represented in
an MFA like {list, length, 1}. The value in the table is the type information for the function
represented by the MFA. The value is a tuple on the form {return-type, [input-types]}.
The PLT provides a more speciﬁc type information for the analysis when using modules that are not
available at analysis time. Because of this, when using Dialyzer, the user should parse a reference
to an existing PLT to the tool. It serves as the base PLT and contains information that will be
relevant for the analyzed modules.
In the following section, we will present an example of some information available in the PLT.
2.4.4
Dialyzer in practice
Having introduced some background of Dialyzer, we now want to explain how Dialyzer uses the
algorithm to provide type information. Just as the success typings algorithm, Dialyzer’s analysis is
carried out in a set of similar steps:
1. The call graph is generated based on the modules to be analyzed.
2. Type inference is based on the bottom-up constraint solving algorithm on each SCC of the call
graph (which is referred to as ”Typesig”).
3. The types that have been inferred are reﬁned based on a forward ﬂowing analysis, that uses
a ﬁxpoint computation to determine when to terminate for a function. The analysis is per-
formed on each module, taking its starting point in escaping functions (which is referred to as
”Dataﬂow”).
19
==PAGE:24==
The analysis begins by creating a call graph for all the analyzed functions. The SCCs of the call
graph is then reduced, turning the graph into a DAG. The rest of the analysis carries out in two parts:
Typesig and Dataﬂow. Typesig is the bottom-up analysis described for the general success typings
algorithm and is short for ”Type signature.” We will continuously refer to this term as ”Typesig”
below since this is the term that Dialyzer uses. The Typesig analysis runs on each node of the DAG,
where a node is an SCC. When this analysis is complete, the Dataﬂow analysis begins. The Dataﬂow
analysis is the reﬁnement of the success typings produced in Typesig. The Dataﬂow analysis carries
out modularly, taking a starting point in escaping functions only - just as the reﬁnement of success
typings. The Dataﬂow analysis is a forward-ﬂowing analysis and uses ﬁxed-point computation to
determine when to stop the analysis for a speciﬁc function. Suppose the inferred result contains new
information, the analysis repeats while this is the case. The analysis continues while new information
is being generated and stops once a ﬁxpoint is reached, i.e., the analysis can generate no further
information.
Furthermore, Dialyzer operates optimistically, meaning that if no type information is available for a
function, Dialyzer will use the term any for both input- and return type. This behavior is a design
choice to help ensure soundness in the analysis.
We will revisit the tail recursive functions for computing the length of a list to investigate the exact
behavior of Dialyzer. Recalling the length functions shown in 10 functions, we now add them to the
module moduleA:

Typesig. The ﬁrst part of Dialyzer’s analysis is Typesig. The analysis is done on each SCC in a
bottom-up fashion. This means that the node {moduleA, length_2, 2} will be analyzed before
{moduleA, length_1, 1}. Looking into the PLT during the Typesig analysis, we will see that the
type information stored in the PLT for {moduleA, length_2, 2} is:
{moduleA, length_2, 2} : {any,[{c,list,[any,{c,nil,[],unknown}],unknown},any]}
This means that the function produces the term any and it accepts a list, which can contain types
any, and a second argument, which can be any. length_1 will use the available type information
of length_2. Because length_1 returns what length_2 produces, surely the return-type of the
two functions are equal. Furthermore, as the only argument for length_1 is parsed into length_2,
it must have the same type information as the ﬁrst argument of that function, namely the list.
Therefore the type information available after the Typesig analysis of length_1 is:
{moduleA, length_1, 1} : {any,[{c,list,[any,{c,nil,[],unknown}],unknown}]}
The above result matches the type information we discussed in section 2.4.2, but obviously, it is not
very speciﬁc, and it misses the developer’s intentions.
Dataﬂow. Dataﬂow improves the type information inferred by Typesig. Based on the already
inferred type information, Dataﬂow analyzes the functions in a forward-ﬂowing analysis to reﬁne
the types. It runs modularly, using all escaping functions as starting points. Each function call
pauses the current analysis until the function called has been analyzed. The type information stored
in the PLT is updated according to the reﬁned information. Taking a look at the PLT after the
Dataﬂow-analysis we have:
{moduleA, length_1, 1} : {{c,number,{int_rng,0,pos_inf},integer},
[{c,list,[any,{c,nil,[],unknown}],unknown}]}
First, the type information for length_2 has been used to reﬁne the inferred type for length_1.
After this, because the function is not exported, the information is no longer needed in the analysis.
The information is therefore removed from the PLT. Furthermore, the PLT now contains the reﬁned
type information for length_1. Using this information, the analysis infers that the function returns
a positive integer using this information.
2.5
Gradualizer
While Dialyzer is widely used for both Erlang and Elixir and is integrated into the Erlang/OTP,
other tools provide a di↵erent type of analysis. The most well-known alternative is Gradualizer[12].
Gradualizer uses the principles of gradual typing, where some variables and expressions may be given
types while others may be left untyped. Only variables or expressions that have been given a type are
type checked at compile-time (static type checking). In contrast, type errors in untyped expressions
are only reported at runtime (dynamic type checking). The idea is that only parts of the program
that have been statically type annotated by the developer are type checked at compile-time. With
no annotation, no static type checking is done. Consequently, an unannotated part of the program
can be blamed whenever a runtime error occurs in a gradually typed program. This is known as a
gradual guarantee and means that type errors can never occur at runtime, in parts of the code that
21
==PAGE:26==
has been statically type annotated. However, the analysis is not sound because Gradualizer may
reject a type error in a typed part of the program that will never be reached at runtime.
Like Dialyzer, Gradualizer aims to integrate well into existing codebases in a non-intrusive way.
Dialyzer does so by using the concept of success typings and can provide some static type checking
without annotating any part of the program. While Gradualizer does not require developers to type
annotate their code, only parts of the program that have been annotated are statically analyzed.
However, since it still works with only parts of the code annotated, it is becoming a popular way of
introducing static types to otherwise dynamically typed programs.

3.2
Scope of the project
Based on the background and the three discrepancies above, we now want to deﬁne this project’s
scope clearly. From the three discrepancies, it is clear that the gen_server behavior is not currently
being analyzed properly by any of the commonly used static analysis tools for Erlang. Furthermore,
because of the over-approximating nature of success typings that Dialyzer uses, there are some
general errors not related to concurrency that Dialyzer cannot ﬁnd. Gradualizer, on the other hand,
was able to ﬁnd the type clash in discrepancy 3.
As the project mainly aims to improve the static analysis of concurrency-related errors, the main fo-
cus will be on discrepancies 1 and 2. Concurrency in Erlang is a broad topic and can be implemented
in numerous ways. However, as explained, a very frequently used behavior is the gen_server behav-
ior (described in section 2.1.3), which developers use to build client-server applications. Client-server
applications are common in large, real-world Erlang applications mainly because Erlang has built-in
support for concurrency and distribution, and the client-server model follows a distributed appli-
cation structure. Discrepancies 1 and 2 also use this behavior. In the gen_server module we will
focus on the server callbacks handle_call and handle_cast and correspondingly the gen_server
module functions gen_server:call and gen_server:cast. We will focus on these because they
make up most of the use cases when using the gen_server behavior.
It will be helpful for most Erlang developers to improve the detection of discrepancies between client
APIs and server implementations when using the gen_server behavior. It could potentially help
26
==PAGE:31==
detect and ﬁx many code bugs before deploying critical applications to production.
Since Motorola Solutions already use Dialyzer to provide type checking, and Dialyzer is integrated
into the Erlang/OTP and is widely used, we wanted the tool produced in this project to follow
the notable characteristics of Dialyzer. Namely, the analysis should provide sound warnings, the
analysis should be fast, the analysis should require no modiﬁcations to existing code to work, and
the analysis should require little to no annotations or guidance from the user. We can now clearly
state the main objective of the project and the requirements that the produced tool must satisfy:
Main objective:
• Improve the detection of discrepancies between client APIs and server implementations in
Erlang and Elixir, when using the gen_server behavior.
Requirements:
1. The tool should remain sound, i.e., no false positives.
2. The tool should remain fast.
3. The tool should require minimal, preferably no, annotations or modiﬁcations from the user.
3.2.1
The requirement of soundness
Dialyzer is a sound type checker.
Soundness was especially important since Dialyzer was ﬁrst
introduced long after the creation of Erlang. Developers could easily be discouraged from using
Dialyzer if it reported errors in parts of the code, that they had seen run successfully in production
for a long time. If large Erlang-based projects were to adopt Dialyzer, it would need to be without
the complication of modifying existing working code because of false positive warnings. As we want
the Erlang community to adopt our tool in the future, we believe that the same requirement of no
false positives must be satisﬁed by our tool.
3.2.2
The requirement of the analysis being fast
In computer science, one of the most important factors is the time it takes to complete a task. The
longer a task takes, the more of an inconvenience it becomes. The same is especially true with
regards to analyzing large codebases. Dialyzer should be used as an active part of the development
process of Erlang programs, and the same applies to our tool. If the time to complete the analysis
is too large, developers will likely not use the tool.
The requirement of the analysis being fast is also essential since most Erlang projects in the telecom
industry are huge codebases with thousands of lines of code. Even small ineﬃciencies in our tool
could make the runtime explode when analyzing large projects.
3.2.3
The requirement of little to no annotations or modiﬁcations from the user
Using a tool to aid in the development process, often becomes a trade-o↵between the ease of use
of the tool and the help it provides. When Lindahl and Sagonas introduced Dialyzer, it required
no code modiﬁcation for it to work. Later, Dialyzer was able to consider speciﬁcations during its
analysis, but it has never been a requirement for the tool to work. Because of this, it requires no
27
==PAGE:32==
e↵ort to start using Dialyzer in a project. Furthermore, if our tool forced the users to type annotate
their Erlang and Elixir programs, it would signiﬁcantly increase the e↵ort required, making them
less likely to use it. Also, they would be limited in the freedom of expression that dynamically typed
languages such as Erlang and Elixir provide.
The requirement of no annotations or modiﬁcations is therefore very important to increase the
chance that more developers will use it actively.
The three requirements for the tool listed above will inﬂuence the choices we make when choosing
an approach to solve the overall objective. Later, we will provide measurements to justify that we
meet the objective while satisfying the three requirements of the tool (see section 7).

3.4
The Chosen Approach
The approach we chose to improve the detection of discrepancies with gen_server was to extend
the already well-established tool Dialyzer. Dialyzer builds on top of the Erlang/OTP, which is an
open-source project. The Erlang/OTP has plenty of documentation on building local versions of the
tool, allowing developers to extend or modify the tool themselves.
Even though Dialyzer cannot detect certain discrepancies, the analysis is extremely powerful overall.
During our analysis of Dialyzer, we discovered that Dialyzer already infers type information for the
server implementation, which we can utilize in our extension. Furthermore, this approach would
make much sense since Motorola Solutions and many large projects already use Dialyzer as part
of their development. Therefore, it would be simple for users of Dialyzer to use our tool, as no
modiﬁcation to their code should be required, and running a separate analysis tool would not be
necessary after running Dialyzer.
Another strength of extending Dialyzer is the fact that Dialyzer works on BEAM ﬁles, which both
Erlang and Elixir compile to (see section 2.1.2). Making our tool work on Erlang and Elixir would
therefore require minimal modiﬁcations.
3.5
Outlining Alternative Approaches

thoroughly researched and considered other possible solutions.
Below we list some of the alternatives to the approach chosen and justify why we did not choose
these:

3.5.2
Building a new tool
Building a new tool would allow us to work from a clean slate, where we could shape the tool to ﬁt
our requirements. Extending already well-established tools can be complicated, as the overhead of
understanding the individual system components can be pretty signiﬁcant.
However, building a tool from the ground up would not allow us to use the available information that
these well-established tools already provide. An approach could either be to build a tool that fully
replaces Dialyzer, which realistically would not be possible within the given time frame, given the

magnitude and complexity of Dialyzer. Instead, we could build a tool that compliments Dialyzer,
but this would decrease the ease of use, as it would require developers to use multiple tools to get a
complete analysis.
3.5.3
Extending Gradualizer
As mentioned, an alternative to Dialyzer for static type checking in Erlang programs is Gradualizer.
It works by using gradual typing, which requires the user to annotate their code, and only type
annotated code is type checked. Gradualizer violates requirement 3 for the project: the tool should
require little to no annotations from the user. Furthermore, Gradualizer provides complete warnings
and may give false positives, violating requirement 1 for our tool: that the warnings should be sound.
Finally, in contrast to Dialyzer, Gradualizer is not widely used, and a stable version of Gradualizer
is yet to be released. According to the author, it is only close to a beta release[12].
Since Gradualizer violates two of three requirements for the project, this approach would not be a
viable solution.

4
Design
This section will outline how we designed our tool. Speciﬁcally, we will outline how we plan to
extend Dialyzer, making sure we do not break any of the existing functionality while still providing
better type information during the analysis. In section 4.2 we will describe a speciﬁc gen_server
pattern that we require for our analysis to work.
4.1
Extending Dialyzer
Having looked into Dialyzer, how it structures its analysis and how type information is saved and
fetched during the analysis, we wanted to sketch an overall implementation idea.
The discrepancies not caught by Dialyzer seemed to be caused by a missing link between client
APIs and server implementations, when using the gen_server behavior. If we could link the client
function with the server implementation, we should be able to let Dialyzer detect type mismatches
between client and server. We should be able to do this by using the MFA that contains the correct
information rather than the generic MFA used when looking up type information for gen_server
functions.
We must make sure that the analysis executes in the correct order. That is, ensuring that Dialyzer
analyzes the server implementations before the client APIs. It should be possible as Dialyzer creates
a directed graph before analyzing the di↵erent components. Here we should be able to add edges
from the client functions and to the server implementations to ensure a correct execution order.
4.2
Supported gen server pattern
When using the gen_server behavior, the client APIs and the server implementations are usually
deﬁned in the same Erlang module.
However, gen_server:call and gen_server:cast take a
server reference (e.g., a pid) as an argument to specify the server that is requested, and this server
might be deﬁned in a di↵erent module. Creating requests to servers in modules other than where
the client API is deﬁned greatly complicates our planned approach to linking client and server
implementations.
We know that Dialyzer looks for the type information in the PLT using the
form {Module, Function, Arity} (MFA), and often we cannot determine the module where the
requested server implementation is deﬁned. A server reference can be supplied in many di↵erent
ways, and in most cases, there is currently no way of knowing the module that is referenced2. Section
8.1.4 of Future Work will discuss a possible approach to allow the analysis to handle sending requests
to server implementations in di↵erent modules.
The typical pattern for the gen_server behavior is to have both client and server implementation
within the same module. This pattern is also typically seen in many open source Erlang and Elixir
projects. Because of this, we determined that we could limit our extension to Dialyzer to only work
on gen_server patterns, where client APIs and server implementations are deﬁned in the same
module. This approach will hopefully prove practical and valuable for many Erlang projects.

4.3
Fault tolerance
As we are extending a widely used, well-established production-grade tool, it is very important that
we do not cause breaking changes in any of the analysis steps.
Because of this, we will design
the tool to always default to the traditional behavior of Dialyzer when the extended analysis is
missing information. What we mean by this, is, that if anything unexpected happens throughout
our extended part of the analysis, instead of crashing or reporting warnings, we aim to fall back
to the default behavior of Dialyzer, letting it carry out that part of the analysis. E.g., suppose
Dialyzer cannot ﬁnd the correct type information in our extension. In that case, Dialyzer will fall
back to the default implementation, namely that it accepts any and returns any. By doing this, we
believe that we will, worst case, have an analysis that produces the same results as Dialyzer, and
best case, we will detect more discrepancies. This approach adheres to the general spirit of success
typings. Success typings optimistically type checks the code and will only report warnings that are
guaranteed to exist.

5
Implementation
This section will describe how we implemented the extension to Dialyzer to improve the overall de-
tection of discrepancies when using the gen_server behavior. The ﬁrst improvement we introduced
is explained in sections 5.2.1 and 5.2.2. Then, we will introduce the concept of MFAAA in section
5.2.3, which enabled our tool to provide much more precise type information.
During the implementation of Dialyzer, we continuously iterated on the process and the changes.
We discovered new uncaught discrepancies throughout the implementation, which helped shape the
product. The implementation of the extension to Dialyzer consisted of two main phases.
The ﬁrst phase of the implementation creates a connection between the client- and server implemen-
tation in the gen_server behavior. This connection allows us to ﬁnd the uncaught discrepancies in
the examples provided by Motorola Solutions, introduced in section 3.1.1 and 3.1.2. This approach
includes using another MFA to get the correct type information when analyzing the gen_server
client functions.
The next phase of the implementation tries to make our analysis more speciﬁc without sacriﬁcing the
soundness of our tool. We came up with a way to tell di↵erent server implementations apart. This
part of the implementation extends the well-known MFA to the MFAAA, providing more precise
type information for gen_server client APIs and server implementations.
Speciﬁcally, this will
improve the analysis to properly handle the return type information when analyzing the gen_server
callbacks. The idea, heavily inspired by Erlangs’ general way of grouping functions, identiﬁes the
functions by their name, arity, and the Request argument.
Besides these two phases, we will review some of the changes made to our extension to ensure the
best user experience and ease of use.
5.1
Locally building Erlang/OTP
To introduce the extension of the analysis of Dialyzer, we chose to modify the current implementation
of the tool.
However, the ability to modify Dialyzer directly was not the most straightforward
approach.
Dialyzer is written in plain Erlang but is built directly into the source code of the
Erlang/OTP runtime. Therefore in order to modify Dialyzer, we needed to have the ability to build,
compile and run the whole Erlang/OTP locally.
The Erlang/OTP source code is edited and tracked through git and hosted on GitHub as an open-
source project[13].
Therefore, creating our copy of the Erlang/OTP project involved creating a
forked version of the git repository. The forked version of the Erlang/OTP source code enabled us
as a team to make any modiﬁcations required to improve the analysis. The general source code of
Erlang/OTP is open source and relies on the modiﬁcation by any interested individual that has the
time and interest in improving the project. The project includes multiple guides and tools to locally
build and run the modiﬁed source code on di↵erent architectures of machines. These guides and
tools enabled us with few issues to get the modiﬁcations to the Dialyzer tool to run locally with the
ability to debug and analyze how our new modiﬁcation performed.

5.2
Extending Dialyzer with gen server
Dialyzer works upon the concept of success typings (as described in section 2.4.2) and relies on the
input and return types of the di↵erent functions and modules saved in the PLT. Dialyzer bases its
analysis on an existing PLT. Building the PLT is done by running Dialyzer on Erlang applications
that the PLT should include. Often the default PLT contains type information for Erlang standard
libraries, but it can also be helpful to include personal projects that are not frequently changed.
The PLT will provide type information for modules that are not necessarily part of the analysis.
The standard libraries include the gen_server module, which a project can use to implement the
generic part of the server.
5.2.1
Mapping the gen server functions
During our project analysis, we established that Dialyzer already analyzes the handle_call and
handle_cast implementations.
These functions are analyzed because the gen_server behavior
dictates that exporting the server implementation is required, and all exported functions are by
default analyzed by Dialyzer.
Therefore, our ﬁrst goal in extending Dialyzer was to map gen_server:call and gen_server:cast
to the requested server implementations. By default, when Dialyzer needs information for a function
invocation, the MFA-tuple of that function is used to search in the PLT. For gen_server:call that
would be either {gen_server, call, 2} or {gen_server, call, 3}, depending on an optional
Timeout parameter that can be parsed to the function. For handle_cast the MFA-tuple would be
{gen_server, cast, 2}. As gen_server is a generic module meant for server implementations,
no speciﬁc type information can be fetched from these MFA’s as the type information currently
available for the gen_server methods, is too broad and has no connection to the actual server
implementations.
Speciﬁcally, the type information available for gen_server:call will be that
input and output can be of any type.
For gen_server:cast, the input type can be any type.
However, the return type is limited to the atom ok.
Instead, we want to fetch the available information for the server implementations. We will do this
by intercepting all functions that invoke gen_server:call or gen_server:cast and modifying how
it looks for information in the PLT. Using pattern-matching in Erlang, we can overload the existing
functionality with a speciﬁc case for gen_server-speciﬁc function invocations. We will search the
PLT with an MFA speciﬁc to the module we are currently analyzing for these speciﬁc function
invocations. Returning to the module introduced in example 2 of the uncaught discrepancies in
section 3.1, this will mean that instead of searching for the key {gen_server, call, 2}, we will
look for the key {discrepancy2, handle_call, 3}.
Because we have introduced a pattern requiring the location of the client and server implementation
to be within the same module, and because the arity of handle_call is consistent when using
the gen_server module, we know that the PLT lookup {discrepancy2, handle_call, 3} always
should contain meaningful information. Looking at the PLT information available with the new
PLT lookup, we can see the following available type information:

First the atom reply, which is speciﬁc for handle_call in the gen_server behavior to inform that

disregard the type information for State and From and the atom reply, as it is not relevant for the

argument, which should be the server reference, can be of any type (for simplicity). In contrast, the
second argument uses the type information available by the success typings, which is that it should
contain a tuple with the ﬁrst element being the atom my_api_server and the second argument
should be a number.
5.2.2
Modifying the analysis call graph
Having altered the way that Dialyzer looks for type information in the PLT whenever gen_server
behavior functions are invoked, we have introduced new function dependencies. Recalling section
2.4.2, Dialyzer initially creates a call graph that helps dictate the order of the analysis. Speciﬁcally,
there is an edge in the call graph e(f,g) if function f calls function g, or in other words, when
function f depends on function g. Since functions with gen_server:call/cast statements now
explicitly need access to the information of the server implementation, we need to introduce an edge
from a function containing the statement to the actual server implementation. Without this edge,
38
==PAGE:43==
there is no guarantee that handle_call is analyzed before the information is required in a function
invoking gen_server:call.
Using discrepancy 2 again, we want to introduce an edge, e(my_api/2, handle_call/3), stating that
there is a dependency between the client implementation and the server implementation. To conﬁrm
this missing link, we used the built-in ﬂag exposed by Dialyzer, --dump_callgraph, to investigate
the call graph used by Dialyzer when analyzing discrepancy2. Unsurprisingly, the generated call
graph is empty - or rather, it consists of several nodes with no edges connecting them. Dialyzer,
by default, does not register any dependencies between functions within this module. The lack of
dependencies is due to the usage of the gen_server pattern.
We solved this by adding an edge in the call graph for a function, f, whenever it contained either
the statement gen_server:call, to the function handle_call, and correspondingly we created an
edge to handle_cast if f contains the statement gen_server:cast. After this introduction, the call
graph now has an edge representing the added dependency. This is seen in ﬁgure 3, showing how
an edge from function my_api/2 to handle_call/3 is now present, because my_api/2 contains the
statement gen_server:call.

However, Dialyzer infers the return type of the
request to be the union of all handle_call clauses. In the example, the return type is therefore
a | a_5 | number() | b. The speciﬁcation stating that the function returns b is therefore valid
in the current implementation.
Splitting the server implementations. To improve upon this, we want Dialyzer to analyze and
save the type information of the handle_call clauses as separate functions when their Request
argument is di↵erent. We, therefore, want to extend upon the MFA concept to include the atom
and arity of the pattern matching argument of the handle_call. We call this construct MFAAA
and the two added A’s correspond to the Atom and Arity of the Request argument. The following
cases are possible for the Request argument:
40
==PAGE:45==
• The Request argument is a single term: E.g., my_atom.
We deﬁne the arity of this argument to be 1.
• The Request argument is a tuple with an atom as the ﬁrst element, followed by any number
of arguments: E.g., {my_atom, Arg1, ..., ArgN}.
We deﬁne the arity of this argument to be the length of the tuple, i.e., N + 1.
If the Request argument does match any of the cases above, the MFAAA information is discarded.
Only the MFA for the handle_call is then saved in the PLT.
The motivation for the MFAAA construct comes from how Dialyzer analyzes functions. Multiple
function clauses with the same name and arity are treated as the same function. The arity of a
handle_call clause is always 3, and all handle_call implementations are therefore treated as the
same function. Function clauses with the same name but di↵erent arities are treated as separate
functions. The MFAAA for a handle_call clause allows the server implementations to be seen as
separate functions. This concept of using MFAAA to separate the functions can be shown as an
example by using normal Erlang functions. Looking at the listing 23 below, we have four implemen-
tations of the handle_call function, with di↵erent Request arguments. These implementations

5.3
Supporting Elixir
After introducing the concept of MFAAA in Dialyzer, we saw that we could detect signiﬁcantly more
discrepancies in the gen_server behavior in Erlang programs. We then wanted to ensure that the
improved analysis worked on Elixir programs as well.
The extension to Dialyzer improves the analysis of the gen_server pattern, a built-in behavior in
Erlang/OTP. However, in Elixir, the corresponding gen_server behavior is called GenServer. This
di↵erence results in di↵erent MFAs, speciﬁcally the module, M, when analyzing Erlang and Elixir
programs, as the implementation of the gen_server behavior is slightly di↵erent. For example, the
MFA for a request to the server implementations is {’Elixir.GenServer’, call, 3} in Elixir, and
{gen_server, call, 3} in Erlang.
When Dialyzer performs the improved analysis, the input and return types of the server implemen-
tations are inferred and stored in the PLT, using MFAAA and MFA. Then, whenever a client API
creates a request to the server, we intercept it to determine which server implementation the client
API is calling. We then use either the MFAAA or MFA to determine the input type the server
implementation accepts and the type it returns.
Supporting Elixir was therefore done by also intercepting requests to the server implementations with
the ’Elixir.GenServer’ module name. This simple modiﬁcation allowed the analysis to retrieve
the correct type information in Elixir programs, thus providing the same improved analysis as in
Erlang.

5.4
Toggleable gen server Analysis
As discussed in section 4.2, our analysis requires that the client APIs and their corresponding server
implementations are deﬁned in the same module. We consider deﬁning them in di↵erent modules
as an anti-pattern. However, some Erlang and Elixir programmers may disagree with this idea.
Analyzing codebases that use this anti-pattern might produce false positive warnings. Requirement
1 for our tool states that it should never produce false positives.
Therefore we decided to add an option to enable and disable our extension to Dialyzer easily. With
this option, we believe there is a higher possibility of the tool being accepted in the Erlang community
and integrated into the Erlang/OTP.
To enable and disable the additional analysis of the gen_server behavior, we added a warning ﬂag
that can be speciﬁed in the command to start Dialyzer’s analysis. When the ﬂag is not included,
Dialyzer works precisely the way it did before introducing any changes. When it is included, all
the components of the improved analysis are performed by Dialyzer. The MFAAA constructs for
the server implementations are generated and stored in the PLT. The call graph is modiﬁed, so the
analysis occurs in the correct order. All requests to the server are intercepted and type checked.
The ﬂag is implemented similarly to the existing ﬂags that can be speciﬁed when running Dialyzer.
Developers should therefore easily be able to enable and disable our extended analysis, given that
they are already familiar with Dialyzer.
5.5
Distributing our tool
Having implemented the improvements to Dialyzer, we looked into the options of how we could enable
easy distribution of our tool to allow projects to get started using the modiﬁcation e↵ortlessly. Our
modiﬁcations are directly written into the source code of Erlang/OTP and can thereby easily be
distributed by building and compiling Erlang directly from the source code. However, this process is
not straightforward and does not allow us to easily enable projects to run our modiﬁcations without
running a list of commands to prepare and build Erlang directly from the source and using the built
binaries.
To better distribute the tool and allow for, e.g., the possibility of running the tool as part of a
project’s automatic pipelines to validate code before human review, we implemented and built a
Docker[14] image. Using this Docker image, we enabled the possibility of distributing and running
our modiﬁcations to Dialyzer on all types of machines without having to conﬁgure and recompile
Erlang directly from the source code on each machine. The usage of Docker allowed us to create a
platform-independent runtime for our modiﬁcation to Dialyzer to easily run our tool on any project
without the hassle of recompiling from scratch on each independent machine.

6.2.1
Overshadowing function clause
During testing, we discovered a bug where our extension to the tool would discard some of the type
information generated during the analysis.
We realized that Dialyzer discarded the information
because of the way we ordered the grouped type information. The following listing shows a simple
example of where this would go wrong:

Listing 25: Showing an example of a handle call discrepancy due to an overshadowing bug.
There are no discrepancies in this module, however Dialyzer would report the following warning:
handle_call_catch_all.erl:5:2: Invalid type specification for function
handle_call_catch_all:my_api/2. The success typing is (_, _) -> ’b’
Even though the warning relates to the function my_api, Dialyzer emits the warning because it
believes that handle_call can only return the atom, ’b’. We can see in the implementation that
when handle_call is invoked with {my_api_server, a}, it returns the atom ’a’.
The actual cause of this false positive discrepancy is that when we introduced the MFAAA concept to
Dialyzer, we forgot to adhere to the order of the handle_call-clauses in the code. In this example,
Dialyzer would analyze the handle_call-clause with the more generic ”catch-all” variable _Atom
as if it was the ﬁrst clause. As Erlang, and therefore Dialyzer, evaluates function clauses from the
top, developers should list the most speciﬁc clauses at the top and then list the less speciﬁc clauses
afterward. Suppose one places the more general clause above a less general clause. In that case,
it will overshadow the less general clause. During our analysis, we would reorder the clauses by
accident, resulting in Dialyzer believing that handle_call would only be able to return the atom
’b’ because the clause returning ’b’ would overshadow the other clause.
This misbehavior of our extension to Dialyzer could have gone under the radar for a long time if
we had not tested the tool on large real-world applications. The number of di↵erent scenarios and
cases that apply to gen_server is signiﬁcant, and it would almost be impossible to come up with a
representation of them all. However, with these stress tests on large real-world applications, we can
detect these scenarios in most cases, ensuring a bug-free implementation.

tions where handle_cast functions are involved in a circular dependency, which would explain the
high amount of failed lookups.
7.2
The requirement of soundness
Requirement 1 for our tool states that it should provide sound warnings. We ensured that our tool
satisﬁed this by running tests extensively on both open-source and Motorola Solutions projects. We
ran tests both before, during, and after all implementation changes. We thoroughly analyzed every
discrepancy warned by Dialyzer during these tests to ensure that we did not introduce any false
positives in our tool. We included the code developers for the Motorola Solutions projects to ensure
that the found discrepancies were factual.
It happened once that our tool would report a false
positive discrepancy; however, after having analyzed the related functions thoroughly, it turned out
that the false positive warning was unrelated to our extensions to the tool. However, we discovered
the discrepancy because we have broadened the analysis of Dialyzer.
This discrepancy was the
example described in detail in section 6.2.2.
7.3
The requirement of the analysis being fast
To ensure that our tool satisﬁed requirement 2 of being fast, we compared the runtime to the runtime
of the original version of Dialyzer.
As mentioned, we allowed developers to enable or disable the gen_server-detection using a ﬂag,
which developers can include in the Dialyzer command. We wanted to test our tool both with and
without the gen_server-detection. Therefore, we wanted to compare three di↵erent versions of the
Dialyzer tool:
• Default Dialyzer.
• Modiﬁed Dialyzer: gen_server-detection disabled.
• Modiﬁed Dialyzer: gen_server-detection enabled.
When ﬁnding open source projects for the analysis, we generally found that Elixir projects were the
most relevant. First of all, most of the open-source projects we found were Elixir projects, which
makes sense since it is often large industrial applications such as telecommunications that use Erlang.
Those projects are typically not open-source. Second, most of the Erlang projects we found were
not actively using Dialyzer or had at least not run Dialyzer on the project for a long time. In some
cases running the default version of Dialyzer on the project resulted in several hundred (occasionally
in thousands) emitted warnings, making it somewhat unmanageable to use the results. The many
warnings were also present for some Elixir projects, but we did discover some projects that used
Dialyzer actively. Furthermore, when running Dialyzer (through Dialyxir) in Elixir, a total count of
warnings produced by Dialyzer is emitted, making it much easier to identify new errors. Lastly, it

7.4
The requirement of minimal annotations or modiﬁcations
Requirement 3 was essential to make it as easy as possible for developers to adopt our tool. If using
our tool requires too many changes or e↵ort to make it work, many users might be discouraged from
using it.
Because we introduced a required pattern to use our tool, we can not conclude that we adhere
entirely to this requirement. However, we believe the enforced pattern is a common design choice,
and most projects already comply with the pattern. Furthermore, because we hide the analysis
behind a ﬂag, we believe that our extension still complies with Dialyzer’s behavior. Integrating our
tool into Dialyzer would not break it and its principles.
Besides the required pattern, we did not encounter any projects where it would require a code
rewrite to use our tool.
Our tool extends the capabilities of Dialyzer, and the success typings
algorithm automatically infers all the information used throughout the analysis. The information
used to analyze the gen_server modules properly does not rely on speciﬁcations but is a product
of modifying the already available type information in Dialyzer.

8
Conclusion & Future Work
In this project, we have extended the already well-established, static analysis tool Dialyzer, to
improve the detection of concurrency-related discrepancies in Erlang and Elixir, when using the
gen_server behavior. We decided on three requirements that our produced tool should satisfy: 1)
It should be sound, i.e., report no false positives. 2) It should be fast. 3) It should require minimal,
preferably no, annotations or modiﬁcations from the user. These are the exact requirements that
Dialyzer satisﬁes, making it e↵ortless for the Erlang and Elixir communities to adopt our extended
tool.
We improved the analysis in two ways: First, we used the type information about the client APIs
and server implementations of gen_server, which was already available in Dialyzer using the MFA.
This change already signiﬁcantly improved the number of discrepancies that our tool was able to
detect. Second, we introduced an extension to the MFA, namely the MFAAA (Module, Function,
Arity, Atom, Arity), to improve the analysis further. The MFAAA construct provided signiﬁcantly
more precision in the type information of the return types of the server implementations, to detect
even more discrepancies.
To measure the e↵ectiveness of our tool, Motorola Solutions allowed us to analyze their actual
production codebase. From this, we can conclude that all of the synchronous gen_server requests,
and the majority of the asynchronous requests in their codebase, were analyzed with more type
information than was previously available in Dialyzer.
Furthermore, we were able to catch 17
discrepancies in their codebase.
The discrepancies found during analysis and testing were thoroughly inspected and veriﬁed as actual
code errors, ensuring that we have not introduced any false positives. We described what we deﬁne
as an anti-pattern, namely deﬁning the gen_server client APIs and server implementations in
di↵erent modules. As long as this anti-pattern is not used, the produced tool satisﬁes requirement
1 of soundness.
Applying our tool to several open-source projects and measuring the runtime, we could conclude
that the analysis was still fast, thereby satisfying requirement 2.
Lastly, since we require developers not to use the gen_server anti-pattern, we made it easy to enable
and disable our extended analysis. This allows all projects to adopt and use our tool, without having
to modify existing code. The tool thereby satisﬁes requirement 3.
The fact that our tool could ﬁnd discrepancies in Motorola Solutions’ seemingly bug-free, high-
quality codebase that runs in production, proves how e↵ective it is. This is especially true since the
tool requires no modiﬁcations or guidance from the user, while still providing a fast analysis with
sound warnings.

8.1
Future Work
The following section will discuss several ways that the work we presented in this thesis, can be
improved with future work.
Sections 8.1.2, 8.1.3 and 8.1.4 describe ways that our speciﬁc extension to Dialyzer could be improved,
to improve detection of concurrency-related errors with the gen_server behavior and in general.
Section 8.1.5 describes some more general improvements that could be incorporated into Dialyzer,
to catch more discrepancies not necessarily related to concurrency or gen_server.
8.1.1
Integrating the modiﬁcations into Erlang/OTP
In order to correctly distribute our tool as part of the source of Erlang/OTP, the changes made
to incorporate our improvements would need to be accepted and merged into the main repository
of Erlang/OTP. In this project, we did not create a pull request to get our tool integrated into
Erlang/OTP. However, it could be completed in the future to enable all Erlang and Dialyzer users
to get these improvements natively with their Erlang installation.
However, to introduce new functionality to the Erlang/OTP, it is required that the subject of the
functionality is discussed in the community.
We believe this project can be used as a proof of
concept to start the discussion on concurrency-related discrepancies in Dialyzer. We already ran
the Erlang/OTP’s GitHub Actions, ensuring that we did not introduce any breaking changes to
Dialyzer. Furthermore, the GitHub Actions ensures that we adhere to the development rules of the
Erlang/OTP project.
8.1.2
Enabling more generic behaviors
The current implementation of our extension to Dialyzer only supports the gen_server behavior
in Erlang and Elixir. However, multiple di↵erent behavior modules exist for generic concurrency
concepts in Erlang and Elixir, such as the gen_event- and the gen_statem behavior.
These two patterns follow the same concept as the gen_server pattern, that client APIs call server
implementations using generic functions.
Therefore, as we saw with gen_server, Dialyzer does
not use the correct type information when analyzing the client APIs and server implementations.
Along with this, the gen_statem- and gen_event behaviors further extend the complexity of the
analysis by creating conditional and dynamic e↵ects based on the program’s current state.
For
example, when utilizing the gen_event behavior, the individual handlers are dynamically added
to the process. Therefore, Dialyzer would have to track and detect which handlers are added to
individual processes to analyze the modules correctly. Enabling Dialyzer to analyze more of these
generic concepts would further increase its e↵ectiveness.
8.1.3
General concurrency type analysis
The extension we added to Dialyzer focused on improving the overall analysis of programs utiliz-
ing the gen_server behavior. The extension, however, is built around a supported pattern that
developers must adhere to when implementing gen_server behavior modules. We introduced and
described the pattern in section 4.2. The pattern was introduced because Dialyzer cannot assign
57
==PAGE:62==
type information to the speciﬁc process identiﬁer used when invoking the functions. This inability
for Dialyzer to store and detect information about process identiﬁers is a general shortcoming of
Dialyzer, which will disregard all information when a program is using message passing in their
implementation.
In order to further improve upon Dialyzer and potentially increase the amount of information avail-
able in Dialyzer, the ability to analyze and store information related to process identiﬁers would
add a more comprehensive analysis of message passing errors. Maria Christakis and Konstantinos
Sagonas already did some exciting work on this subject, which could serve as great inspiration when
addressing this issue. They describe it in their paper Detection of Asynchronous Message Passing
Errors Using Static Analysis[16].
Generally, when talking about the concept of message passing, we use the notation of !, receive,
and spawn to send, receive and spawn processes to send messages from one process to another.
Creating a process is done using the spawn function built into Erlang, the module, function, and
the arguments are speciﬁed to create a new process. This process can be a function that includes
the receive to enable other processes to send messages and, based on pattern matching, perform
di↵erent actions. To improve the general concept of detecting concurrency errors in Dialyzer, we
could potentially use the information passed to the spawn function when initially creating the process.
Detecting discrepancies in message passing should be possible since when initially performing the
spawn operation, we are specifying the module and function along with the original arguments to
which the process should be linked. Therefore, we should be able to give Dialyzer some insights as
to which processes point to which modules and functions to improve the general concurrency type
discrepancy detection.
8.1.4
Cross-module gen server calling
The extension we made to Dialyzer only ensures sound warnings when developers do not violate the
pattern we described in section 4.2. Therefore, some developers might not be inclined to disable the
features we introduced in our extension of Dialyzer.
To improve the extension, the ability to call server implementations in di↵erent modules could remove
the pattern criteria, enabling all projects to utilize the extension we introduced fully. Extending
our analysis to support all gen_server behavior patterns would be an excellent extension to be
designed and implemented alongside the general concurrency type analysis described in section 8.1.3.
Combining these implements could improve the e↵ectiveness because when utilizing the gen_server
pattern, the calling of the server implementation will take a server reference as the ﬁrst argument to
identify which server the generic server is referencing. Having the concept of the general concurrency
type analysis, the ability to directly infer the module could potentially be available to Dialyzer when
performing the analysis, instead of always relying on the design pattern that the client APIs and
the server implementation must live within the same modules.
8.1.5
Improving the overall Dialyzer analysis
During the project, we found multiple shortcomings in Dialyzer that generally could be improved to
give a much better result. We found the concepts described here while developing and running our

improved analysis of Dialyzer upon multiple projects. However, they are generic to Dialyzer and do
not relate to any of the work done in this project. We believe that further investigation into these
concepts will improve the Dialyzer experience by catching more discrepancies.
Anonymous functions
A concept that Dialyzer currently is not optimal at detecting discrepancies around is the concept of
anonymous functions. Invoking a function within an anonymous function (of any kind), then only
one invocation of the function must be valid for Dialyzer to approve every request. Since Dialyzer

In order to further improve the overall experience when using Dialyzer, the improvement of enabling

more discrepancies generally.
Partitioning of types
Currently, Dialyzer stores and looks for information in the PLT based on the MFA, meaning that
each function sharing the same module, name, and arity is stored together. Storing the information
using the MFA means that if a function is created for, e.g., pattern matching, it might split multiple
implementations based on the speciﬁc input type or value. Doing this, Dialyzer cannot correctly
link the input type with the return type of the speciﬁc input type that might generate.
If we take a look at the listing 28 below, we see an example where Dialyzer cannot partition the

T. Lindahl and K. Sagonas describe this concept of not being able to connect the input type to the
return type in their paper on success typings[9]. They speciﬁcally state that ”we currently sacriﬁce
expressiveness for simplicity and readability”. However, we believe that introducing matching input
types to return types will give a much more detailed analysis result.
Discrepancies in function speciﬁcations
Suppose we look back at the provided discrepancy in section 3.1.3, where we have the implementation
of the function safe_div, which takes two arguments and utilizes pattern matching on the second
argument. Here Dialyzer will correctly detect that the function safe_div will take two integers
as arguments. However, it is unable to correctly detect that it should return either the tuple of
{ok, integer()} or the tuple of {error, unicode:chardata()}, the last of which is violated by
the implementation.
This discrepancy happens when Dialyzer tries to connect and report errors when comparing the
success typing of a function with the speciﬁed speciﬁcations for that function. In the example of
the function safe_div, if we ask Dialyzer to output the inferred success typing of the function, it
will correctly generate the expected type of {’error’, 0} | {’ok’, integer()}. However, this
return type and the return type of the function speciﬁcation do not overlap.
Enabling Dialyzer to correctly detect if a function can return multiple di↵erent return types, if the
function speciﬁcation and the success typing have at least a single mismatch, could generally improve
the ability of users to catch more errors when writing Erlang programs.
The inability to detect inconsistencies between speciﬁcations and implementations generally seems
to be an issue, with Dialyzer being unable to handle errors when speciﬁcations and success typings
clash if a function can return multiple di↵erent types.
Dialyzer will report no errors since the
implementation of the case where the argument Y is not zero will return the correctly speciﬁed type
{’ok’, integer()} from the speciﬁcation. However, suppose we changed the speciﬁcation to not
being correct on any of the expected return types. In that case, Dialyzer will correctly report a
discrepancy that the implementation and the speciﬁcation do not match.

A3
The callbacks of gen server
This section contains a list of all gen_server module functions and the callback functions that the
developer can implement. The callback functions are all described after ﬁgure 8.
65
==PAGE:70==
Figure 8: Showing available gen server module functions and their respective callback handlers[3].
In the following, we will brieﬂy touch upon each of the callback functions of the gen_server behavior,
shown in the ﬁgure 8 above.
Module:init/1: Callback function used to initialize the server whenever gen_server:start,
gen_server:start_monitor, or gen_server:start_link is called. This callback can speciﬁcally
be used to initialize the default state for the server.
Module:terminate/2: Callback function called just before the server shuts down by the call
gen_server:stop. The function should be used to handle any cleanup required before termination.
Module:handle_call/3: Callback function that handles synchronous server requests. The function
parameters are the Request, From, and State.
Throughout this paper, we will be looking a
lot at the Request argument, as it will play a crucial role in distinguishing di↵erent handle_call
implementations. The Request is a term8, which typically is an atom or a tuple containing an atom
in the ﬁrst tuple value. This atom can be seen as the identiﬁer for the callback implementation.
Any trailing terms in the tuple are typically used as arguments necessary for the callback function.
From contains information that is required by the gen_server module to return the response to
the callee.
State is the current state of the server.
Both From and State is provided by the
8term can be all possible Erlang terms. It is a synonym for any.
66
==PAGE:71==
gen_server behavior module.
Module:handle_cast/2: Callback function that handles asynchronous server requests. The function
parameters are the Request and State. The Request and State parameter works similarly to
handle_call/3 described above.
Module:handle_info/2: This callback function is called by gen_server if either a timeout occurs
or the server receives an unexpected message. This callback is implemented in the gen_server
module, and an explicit implementation is not required.
Module:handle_continue/2: This callback is called by gen_server whenever a previous callback
returns the tuple {continue, Continue}. It can be used to splitting callback work into multiple
callback functions, thereby updating the process state for each step.
Module:code_change/3: This callback is used to update the structure of the state during release
upgrade/downgrade.
As handle_call and handle_cast are the main callback functions used for client/server communi-
cation, these will be the callbacks in focus throughout this paper.
