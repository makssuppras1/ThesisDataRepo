Abstract
Compression is a widely used process that reduces the space needed to represent infor-
mation. In many applications, it is vital to eﬀiciently store and access similar data, such
as genomic sequences or versioned documents. This thesis aims to compactly represent
highly repetitive data sets while supporting fast random access to any document of the set
by combining well-known compression methods with techniques exploring text similarity.
A basic compression algorithm, Lempel-Ziv 77, has been shown to produce high compres-
sion ratios, though it is impractical for random access. The other approach, using Relative
Lempel-Ziv, supports eﬀicient random access but yields lower compression ratios. A new
approach is introduced, which, together with Relative Lempel-Ziv and locality-sensitive
hashing, has achieved higher compression ratios than using Relative Lempel-Ziv on its
own. This method works by compressing the most similar pairs of documents in the data
set relative to each other. It is left as an open problem to explore whether this solution
can be modeled as a persistent string [1] to support random access queries eﬀiciently.

Introduction
Compression is a process that reduces the amount of space needed to store certain infor-
mation. Compression algorithms may be lossy or lossless. When lossy, some unnecessary
part of the original data may be removed during the compression making the process ir-
reversible. This method is usually applied to image, audio, or video files. When lossless,
all original data is preserved in the compressed format such that the process is reversible.
This method is most commonly used for texts or data that needs to be compressed while
preserving its integrity [2].
This thesis focuses on exploring lossless compression methods for texts of high similarity.
Examples of such texts are genomic sequences of the same species, versioned documents, or
software repositories. Various compression algorithms are compared and evaluated based
on multiple criteria: the ratio of space required to represent the data before and after the
compression (compression ratio), time complexity required for the compression process,
or a possibility to perform random access on the compressed data.
Chapter 2 presents existing solutions for the problem of compressing highly repetitive data
sets. Some of these approaches are used as a basis for developed compression methods.
Chapter 3 introduces a compression method with Lempel-Ziv 77. The proposed imple-
mentation modifies the sliding window to reduce the overall time complexity by using
fingerprints.
Chapter 4 describes a relative compression approach where the whole data set is com-
pressed relative to a single reference document. It compares two techniques for pairwise
compressions and defines two variants of the compression algorithm.
Chapter 5 defines a compression method that uses locality-sensitive hashing to compress
the most similar documents in the data set relative to each other. This chapter explores
different ways to store sparse vector representations of the documents and compute their
dense representations, signatures. In the end, two variants of the compression method are
presented: simple and optimized.
Chapter 6 focuses on adjusting locality-sensitive parameters in order to find truly similar
pairs of documents by reducing the number of false positives and negatives. Final param-
eters are chosen by testing and evaluating the probability-similarity function on the data
set of Escherichia coli bacteria samples.
Chapter 7 presents an overview of the data set used for experimental results and compares
developed compression algorithms by analyzing their compression ratios and support for
random access on the compressed data set.
Chapter 8 provides conclusions about developed compression schemes. Finally, chapter 9
includes ideas and improvements for these methods, some of which could potentially in-
crease compression ratios or add support for eﬀicient random access.

Previous work
Lossless compression algorithms preserve the integrity of the data such that the original
data can be recovered from the compressed form. Two of the widely used approaches of
such compression were suggested by Jacob Ziv and Abraham Lempel in 1977 and 1988
[3, 4] and are referred to as LZ77 and LZ78. The solutions they proposed are dynamic
dictionary-based compression algorithms, i.e., some part of the input string is replaced
by the reference to a substring from a dictionary computed either before or during the
compression process. The referenced substring must be within the defined sliding window.
In LZ77, a part of the string is compressed as a phrase (o, l, c) that corresponds to the
previous occurrence of the same substring, where o is the relative offset to that substring
of length l and c is the character following the phrase. In LZ78, parts of the string are
compressed as phrases (p, c) where p corresponds to the previous occurrence of the same
phrase and c is the character following the phrase. Multiple compression programs such
as gzip, 7zip or rar are based on these methods.
Some proofs have been shown that LZ77 and LZ78 may not compress optimally for families
of low entropy strings. For this case, S. Rao Kosaraju and Giovanni Manzini presented a
compression algorithm that combines LZ78 with run-length encoding and compresses low
entropy strings eﬀiciently [5].
Another approach to compress structured documents with random access was suggested by
Joaquín Adiego et al. [6]. It is based on the Lempel-Ziv method and is suitable for highly
structured data such as XML forms or invoices. The main advantage of this method over
the standard Lempel-Ziv compression is the ability for random access on the compressed
data.
Sebastian Kreft and Gonzalo Navarro presented LZ-End, a Lempel-Ziv family compression
method which achieves compression ratios close to those of LZ77 but allows for much faster
extraction of arbitrary text substrings [7]. They designed a self-index based on LZ-End
particularly effective for representing highly repetitive collections with compression ratios
close to those of LZ77 but with random access up to 4 times faster.
Relative compression for repetitive data sets has been studied by Shanika Kuruppu et
al. [8]. Their approach, RLZ, is based on storing a self-index for a base sequence and
then compressing every other sequence as an LZ77 encoding relative to the base. This
representation uses relatively little memory while supporting random access eﬀiciently.
Jon Louis Bentley and Doug McIlroy described a precompression algorithm that effectively
represents long common strings in files [9].
They proposed a scheme that recognizes
long repetitions of data with Rabin-Karp fingerprints and no compression window. Their
solution achieves promising results when used in conjunction with standard compression
programs such as gzip.

Compression with Lempel-Ziv 77
The first developed baseline compression mechanism was suggested by Jacob Ziv and
Abraham Lempel in 1977 and is therefore called Lempel-Ziv 77 [3]. It is a widely used
dictionary-based compression scheme where a part of the document is replaced by refer-
encing a previously seen substring using a dictionary computed either before or during
the compression. LZ77 iterates over the document and replaces a part of it with a phrase
(o, l, c) where o is the relative offset to the reference substring of length l and c is the
character following the phrase.
3.1
Building the compression
The compression is built by firstly concatenating all documents to produce a single large
string and then iterating through that string to get all consecutive phrases. The size of
such a compression scheme is the total size of all phrases.
Each phrase consists of three values:
■the relative offset to the reference substring, which may be encoded with a 32-bit
integer considering that the maximum offset is less than or equal to 231 −1 or a
64-bit integer otherwise
■the length of the phrase, which may be encoded with a 32-bit integer considering
that the maximum length of a phrase is less than or equal to 231 −1 or a 64-bit
integer otherwise
■the character following the phrase which may be encoded with 8 bits
In total, each phrase may be encoded with 9 bytes when integer values are 32-bits or 17
bytes otherwise.
3.2
Removing the sliding window
When encoding the string, the algorithm finds a reference to a previously seen substring
within a limited part of the string called a sliding window. The longer the sliding window,
the longer it takes to compute the compression as the algorithm would need to search
through all previously seen substrings yielding a total time complexity of O(n2) where n
is the length of the string.
The issue with using the sliding window is that the compression ratio might be a lot smaller
since the most similar phrase at the time of the encoding might be at the beginning of
the string and not within the limited window. On the other side, it is computationally
expensive to iterate through all substrings naively. For this reason, a different solution was
developed which uses Rabin-Karp fingerprints to avoid the sliding window and reduces the
overall time complexity of finding phrases.
3.3
Using Rabin-Karp fingerprints to avoid the sliding
window
Naive searching for a phrase by directly comparing all substrings starting at previous
positions against the current position is highly ineﬀicient. Instead, some substrings may
be filtered out, and only a part of them might be considered candidates for the longest
phrase.

Rabin-Karp algorithm
The algorithm created by Richard M. Karp and Michael O.
Rabin [10] is a string-matching algorithm that uses hashing to find matches of the given
pattern. Its expected running time to find a single match is O(n + m) where n is the
length of the text, and m is the length of the pattern.
In order for the algorithm to work correctly, the chosen hash function should not produce
many false positives, in which case two substrings produce the same fingerprint (hash)
if they are different. Secondly, it should be computationally feasible and more eﬀicient
to calculate fingerprints than naively finding a pattern in the string. The solution is to
choose a family of polynomial rolling hash functions that may be computed eﬀiciently
with additions and multiplications while sliding through the text. This approach allows
calculating all fingerprints of length m for a text of length n in a linear time O(n + m).
A dictionary of fingerprint positions should be built when computing fingerprints for each
substring. Each key of such a dictionary is a fingerprint, and a value corresponds to a list
of starting positions of substrings having the same fingerprint. Next, finding the following
phrase with LZ77 does not require searching and directly comparing all previously seen
substrings but only lookup candidate substrings that occur before the current position
using a dictionary of fingerprint positions in constant time. A candidate substring is a
substring that matches the same fingerprint as a fingerprint of a substring at the current
position. Candidate substrings are then directly compared to find the longest matching
candidate producing a phrase of length l in time O(l + ∑
i=1...k |si|), where si is i-th
candidate substring, and k is the total number of candidates. This approach is significantly
faster than a naive search over all possible substrings.
3.3.1
Using multiple Rabin-Karp fingerprints
Another optimization that can be used when compressing with LZ77 and Rabin-Karp
fingerprints is using additional layers of fingerprints. It is crucial to adjust the lengths of
fingerprints depending on the data set to compress. For genome data sets, fingerprints
of lengths 5 and 10 seem to work well because there are noticeably more candidates for
substrings of length 5 than 10.
The optimization is based on the observation that if the longest candidate substring at
the current position is longer than 9, then both its fingerprints of length 5 and 10 will
match the fingerprint of the substring at the current position. Because of that, candidate
substrings of length 5 should only be considered if there is no candidate substring of length
10. This technique may improve the total running time of the compression if most phrases
are longer than 9.
3.3.2
Skipping consecutive candidates that will not produce a longer
phrase
Using fingerprints is likely to speed up the compression process, but it might not outper-
form a native search in rare cases. Some data sets such as genomes may contain long one-
character words, i.e., words having only one character, e.g., aaaaaaaaaaaaaaab. In that
case, a dictionary of fingerprint positions will include positions of all possible substrings
of length 5 for a fingerprint of aaaaa. Consequently, this results in direct comparison
through all such substrings, corresponding to a naive approach. This issue may be seen in
Figure 3.1 where the number of candidate substrings in the middle of the string rapidly
increases due to this issue.

4
Compression with a star-based
approach
Compressing a highly similar data set could be done by compressing all documents against
a single reference document chosen from the same data set. Such compression is expected
to be eﬀicient if the reference document is highly similar to other documents from the data
set.
4.1
Defining pairwise compression
In order to compress documents against each other, one needs to define a method to
compress a pair of documents. Two solutions were considered in the developed compression
method: the Levenshtein distance and Relative Lempel-Ziv.
4.1.1
Using the Levenshtein distance for pairwise compression
The Levenshtein distance is one way to define a pairwise compression size between doc-
uments. Levenshtein distance, or edit distance, is a measure of similarity between docu-
ments expressed by the smallest number of single-character edits to change one document
into the other. These edits are either insertions, deletions, or substitutions of characters.
Time complexity to find the edit distance between documents a and b is O(|a| × |b|) when
considering all possible transformations between them. This solution is rather infeasible
when documents are very long. The optimized version which performs computation within
the specified threshold k runs in O(min(|a|, |b|) × k) where k is the maximum distance
threshold between documents [11]. This approach should provide acceptable running times
for long and similar documents.
It is expected that highly similar data sets such as genomic sequences of the same species
are suitable for compression with the Levenshtein distance. However, it was found that
chromosome lengths among different strains of the same bacteria may differ by as much
as 1 MB while ranging from 4.5 to 5.5 MB in length [12]. It means that those genome se-
quences differ by approximately 20% of their total length, making the Levenshtein pairwise
compression infeasible due to long execution times.
4.1.2
Using Relative Lempel-Ziv for pairwise compression
The pairwise compression approach that was used in the developed compression method
instead is Relative Lempel-Ziv, in short RLZ [8]. It stores a self-index structure for the
reference document and then compresses target documents with an LZ77 encoding relative
to the reference document. RLZ can eﬀiciently represent genomes as it is suitable for highly
repetitive data. The compression time complexity is O(n + N log n), based on the length
of a single document n and the total length of documents N.
4.2
Building the compression
The compression method is built by randomly choosing a reference document and then
compressing all other documents relative to that document using Relative Lempel-Ziv.

In order to decompress a data set, the compressed file needs to be decompressed with RLZ
relative to the compression reference document and then split into a set of documents.

5
Compression with locality-sensitive
hashing
Compressing a highly similar data set could be done by compressing only the most similar
pairs of documents against each other. Since the most similar documents should share as
much information as possible, the compression is expected to be eﬀicient.
Similar items in the data set may be found naively by evaluating each possible pair.
This approach is problematic as it would be very time-consuming and should be avoided,
especially for larger data sets. Another way to find pairs of similar documents is by using
locality-sensitive hashing [13], in short LSH. This method finds documents similar to the
given document in sub-linear time complexity; this is done by reducing the number of
comparisons between the documents by evaluating only such pairs that are likely to be
similar. It may produce false negatives, in which case truly similar pairs are marked as not
similar or false positives, in which case pairs that are not similar are marked as similar.
LSH parameters may be tuned to reduce false negatives and positives. This is explored in
Chapter 6.
5.1
Similarity of documents
One way to measure similarity between sets is by using Jaccard Similarity. It is defined
as the size of the intersection of sets A and B to their union.
J(A, B) = A ∩B
A ∪B
In order to evaluate the textual similarity between documents, one has to represent doc-
uments as sets. The most common way to do that is to represent a document as a set of
shingles.
5.2
Shingling
A k-shingle is a substring of a document of length k. The basic idea is that documents that
are likely to be similar would have more common shingles than non-similar documents.
Each document may be represented by a set of all possible k-shingles of that document.
5.2.1
Choosing a shingle length
It is vital to choose an appropriate length of a shingle. Choosing a shingle length that is too
short could result in multiple documents sharing too many shingles; therefore, increasing
their Jaccard similarity, where in reality, these documents might not be similar at all. The
shingle length k should be chosen such that the probability of a shingle appearing in any
document of the data set is low. As an example, for a DNA document of length 5 000 000
with four different characters (A, C, T, G), it would be reasonable to choose k equal to
at least 12. For k equal to 14, there is 414 = 268 435 456 possible shingles making this size
appropriate to use for a DNA document that is much smaller in length.
5.3
Sparse vector representations of shingle sets
Measuring Jaccard similarity using shingle sets sounds promising; however, storing and
comparing shingles between each pair of documents in memory is not feasible because a

single set of shingles takes much more space than a single document. For this reason, each
document should be associated with a much smaller representation, a signature. In order
to build a document’s signature, it is first needed to replace its shingle set with much
smaller representations. A common way is to build a one-hot encoding of a set. A one-hot
encoding of a shingle set indicates which shingles occur in the corresponding document.

expensive to compute and store the vocabulary. Therefore, an alternative approach used
in the developed compression method is to approximate one-hot encodings with a random
hash function. Each shingle of the set is hashed into a number between 1 and the maximum
number of shingles. The resulting values are approximate of shingles’ positions in the

5.4
Producing signatures
The next step is to produce signatures, dense vector representations of documents com-
puted from one-hot encodings. The crucial property of signatures is that the expected
similarity of two signatures is close to the Jaccard similarity of shingle sets they represent.
The length of a signature may be a lot shorter than the document; however, the longer
the signature, the more accurate the estimate is to the Jaccard similarity of corresponding
sets.
A process of converting one-hot encodings into signatures is called minhashing. It in-
volves hashing multiple values and finding the minimum for each signature position. Two
approaches were considered when developing this compression scheme.
5.4.1
Using random permutations as minhash functions
A naive way to compute a signature from a one-hot encoding is by building a hash function
from a random permutation of values between 1 and the size of the vocabulary |V |. This
permutation corresponds to a uniformly distributed hash function that maps integers from
1 to |V | to |V | buckets such that there are no collisions at all. A single value of the signature
is equal to the minimum of hashes for all positions of the one-hot encoding. This process

5.5
Finding pairs of similar documents using signatures
Signatures computed in the previous step of minhashing are significantly smaller than
shingle sets and preserve similarity information about the corresponding documents. Once
signatures of all documents are computed, it is finally time to find pairs of similar docu-
ments using locality-sensitive hashing.
5.5.1
Banding the signatures
The main idea of banding is to make similar documents more likely to be treated as
candidate pairs, which are pairs of documents that hash into the same bucket. It may not
be obvious how to build such a hash function that takes signatures as a whole and places
similar pairs in the same buckets. An effective way to do that is by splitting the signature
into multiple parts, b bands, each consisting of r rows of integers, and then hash them
into buckets. If bands of two documents hash into the same bucket, then these documents
are treated as a candidate pair. This technique finds similar documents based on the
similarity between some portion of their signatures.
5.5.2
Optimizing the bands

p, which defines the similarity threshold s at which pairs of documents are treated as
candidates

5.6
Building the final compression
After finding all candidate pairs of similar documents, the final compression may be built.
In contrast to the Star approach, where each document is compressed relative to some
random reference document, the developed compression mechanism aims to utilize simi-
larity of documents by compressing pairs of similar documents and, therefore, increasing
the overall compression ratio.
A graph of documents and compressions
Compression may be viewed as a graph
of documents where each node is a document, and an edge corresponds to the compres-
sion between two documents. Edge weight is equal to the compression size between two
documents, and the total size of such compression is the total weight of all edges. A node
is covered if it is an endpoint of any edge. The compression is complete, meaning that it
compresses all documents of the data set if all nodes of the graph are covered. This can
be seen in Figure 5.5.
(a) A graph corresponding to
an incomplete compression of
documents.
(b) A graph corresponding
to a complete compression of
documents.
Figure 5.5: Figure 5.5a shows a graph with three isolated, uncompressed documents
(marked as red) making the compression incomplete. Figure 5.5b illustrates a complete
compression where all documents are compressed with reference to a single document, a
case of a Star compression.
The compression should give the best results when its corresponding graph has as little
total weight as possible. Pairwise compression between documents from a candidate pair
should produce a small weight considering that these documents are most similar in the
whole data set. For this reason, all candidate pairs should be considered for the final
graph.
5.6.1
Simple compression method
The first compression method starts by building a graph with all candidate pairs as edges.
These edges form multiple connected components or rarely a single component if they
span all nodes. In order to minimize the size of the final graph, some redundant edges
can be removed.
These are the edges that create cycles in a graph and thus do not
expand the number of covered nodes but only increase the total weight of the graph. A
straightforward solution to remove redundant edges while minimizing a graph weight is
finding a minimum spanning forest of the graph, which is a union of the minimum spanning
trees for its connected components.
In order to build a minimum spanning forest, each edge is associated with a weight. Such
a weight is computed from the size of the RLZ compression between two nodes of that
edge. Reference and target nodes of the compression are chosen randomly. This step is
shown in Figure 5.6.

5.6.2
Optimized compression method
It may be observed that the simple compression method might result in relatively low
compression ratios; this is because it requires storing all of the reference nodes apart from
compressions, in the worst case, all nodes of minimum spanning trees. The optimized
approach considers that and builds a more eﬀicient compression mechanism.

of all candidate pairs as edges and finding the minimum spanning forest of such a graph
using Kruskal. Each minimum spanning tree is then traversed using depth-first search,
where an arbitrary node is chosen as a root node of a traversal. The result is a path for
each minimum spanning tree shown in Figure 5.9.

Experimental results
All of the compression methods were implemented and tested on genomic datasets ex-
tracted from the National Library of Medicine hosted by the National Center for Biotech-
nology Information, NCBI1. This chapter presents and compares methods by analyzing
their compression ratios and support for random access on the compressed data set.
The compression ratio is the ratio of total space required to store the data set before
and after the compression process. The value is less than 1 when the uncompressed size
is larger than the original size of the data set, which is undesired. Thus, the larger the
compression ratio, the less space is needed to store the data set after the compression.
7.1
Dataset
The data set used in the following tests are genomes of multiple strains of the Escherichia
coli bacteria, which characterizes in a size of a single genome ranging from 4.5 to 5.5 MB
[12]. Genomes were downloaded from the NCBI online library. Only complete genomes
were extracted, resulting in 1897 assemblies and weighing 9.98 GB in total.
Each genome is a sequence of five nucleotides: A, C, T, G, or N if a nucleotide is unknown.
It was found that there are only 235 unknown nucleotides in the whole data set. As there
are so few unknown nucleotides, it was assumed that only four nucleotides appear in the
data set when choosing the appropriate shingle length in Chapter 6.

sented with a single character corresponding to that nucleotide. Sequences in this format
are divided with headers, each preceded by a “>” character. Each assembly downloaded

7.5
Comparison and evaluation of eﬀiciency
In the below comparison, it is assumed that an average complete genomic sequence is
4 986 085 in length. The value was calculated by summing up the lengths of distinct words
from all performed tests and dividing by the number of these words.
7.5.1
Differences in compression ratios
The first compression method using Lempel-Ziv 77 produces an average of 3 300 148
phrases per test, which is on average around 151 characters per phrase, assuming that
each of 100 documents has a length of the average complete genomic sequence. The av-
erage size of the data set before compression is 498.04 MB, whereas after compression,
29.70 MB yielding an average compression ratio of 16.79. These results are extracted from
Table 7.1.
The second compression method with a star-based approach and Relative Lempel-Ziv
achieves lower compression ratios with an average of 3.28. When compressing each docu-
ment relative to the compression reference document, the total size of these compressions
is on average 148.80 MB. These results are shown in Table 7.2. Alternatively, concate-
nating documents into a single document and compressing that document relative to the
compression reference document yields an average compression ratio of 3.18. In that case,
the average size of the concatenated and compressed document equals 156.21 MB. These
results are extracted from Table 7.3. In both cases, compression ratios are more than five
times smaller than in the compression using Lempel-Ziv 77, which might be related to
the choice of the compression reference document in the star-based approach. Selecting a
random document might often not result in an eﬀicient representation of other documents.
The compression should give the best results if the chosen document is as similar to other
documents as possible.
The simple variant of the compression with locality-sensitive hashing produces an average
compression ratio of 1.50 for a shingle length of 12 and 1.69 for a shingle length of 13.
The average number of candidate pairs is 565 (11.4%) for a shingle length of 12 and 234
(4.73%) for a shingle length of 13. These results are extracted from Tables 7.4 and 7.5. The
optimized variant of the compression with locality-sensitive hashing achieves an average
compression ratio of 9.43 for a shingle length of 12 and 7.38 for a shingle length of 13.
The average number of candidate pairs is 565 (11.4%) for a shingle length of 12 and 267
(5.39%) for a shingle length of 13. Using a shingle length of 13 reduces the number of
candidate pairs by half, making pairwise compressions faster to compute. These results
are extracted from Tables 7.6 and 7.7.
7.5.2
Trade-off between a compression ratio and random access
The best compression ratios are achieved using Lempel-Ziv 77; however, this solution is
impractical when considering random access on the compressed data. In Lempel-Ziv 77,
decompression must always start at the beginning of the compressed string.
In order
to decompress the last part of the compressed string, it is required to decompress the
whole string first. The compression with a star-based approach and Relative Lempel-Ziv
yields lower compression ratios but supports random access to l consecutive symbols of the
compressed data in O(l + log n) time [8], n being the length of a single document before
compression.
Compared to the previous approach, compression with locality-sensitive
hashing produces higher compression ratios and should allow for eﬀicient random access
when represented as a persistent string [1]. This compression model is further described
in Chapter 9 about future work.

Conclusions
This thesis explored various lossless compression algorithms for highly similar data sets.
The first developed compression mechanism was Lempel-Ziv 77, which is the basis of
several commonly used compression schemes. The proposed implementation removes the
sliding window and reduces the overall time complexity of finding phrases using Rabin-
Karp fingerprints. The approach precalculates all fingerprints of length m for a text of
length n in a linear time O(n + m). Then, it finds the longest match for a given position
in time O(l + ∑
i=1...k |si|), where si is i-th substring matching the same fingerprint as the
substring at the current position, and k is the total number of matching substrings. This
technique is remarkably faster than a naive search through all previously seen substrings.
This compression method achieved the best compression ratios; however, it is impractical
for random access.
Another developed solution compresses all documents against a single reference document,
a star-based approach. Two methods for pairwise compression were considered: the Lev-
enshtein distance and Relative Lempel-Ziv.
The former approach was infeasible when
tested on highly repetitive data sets, such as bacteria genomes, due to long running times.
At the same time, the latter was found to be suitable and eﬀicient, having reasonable
time complexity. This approach was implemented in two variants. The first variant com-
presses all documents relative to a random reference document. In the second variant,
documents are concatenated into a single document and then compressed relative to a
random reference document. Both solutions yielded similar compression ratios, about five
times smaller than Lempel-Ziv 77; however, it must be noted that the compression ratio
is highly correlated with how similar the chosen reference document is to other documents
in the data set. This compression scheme also supports fast random access to any part of
the data set.
The last implemented compression method uses locality-sensitive hashing to find pairs of
the most similar documents in the data set. By compressing these documents relative to
each other using Relative Lempel-Ziv, the compression is expected to be eﬀicient. This
method is built using a minimum spanning forest on the edges of similar documents. The
process of measuring similarity with shingles and signatures was thoroughly described,
and appropriate parameters for compression were chosen after some performance testing.
This approach was implemented in a simple and optimized variant and resulted in much
higher compression ratios than star-based compression. It is left as an open problem to
explore whether this solution can be modeled as a persistent string [1] to support random
access queries eﬀiciently.

Future work
The use of Relative Lempel-Ziv and locality-sensitive hashing in the developed methods
offers a competitive implementation for data compression of highly similar strings. This
chapter presents suggestions and ideas that may improve these solutions.
It could be interesting to explore how choosing a reference document affects the compres-
sion ratio of the star-based approach. The chosen reference document could be based on
the similarity information between other documents. Another method could be finding
the most common parts of all documents, merging them into a single document, and using
it as a reference document. While choosing a reference document randomly is straightfor-
ward, there is a potential to implement a method of finding a document that yields higher
compression ratios.
Only two methods for pairwise compression were tested in both the star-based and locality-
sensitive hashing compressions. These are the Levenshtein distance and Relative Lempel-
Ziv. As an improvement, it could be explored how choosing a different method to compress
two documents relative to each other affects the overall compression ratio and support for
random access.
Another area that could be researched is the choice of locality-sensitive hashing parame-
ters. Choosing appropriate parameters, i.e., the shingle length, the signature length, and
the number of bands for a data set, reduces the number of false positives and negatives. In
this thesis, parameters were chosen manually by evaluating the probability-similarity func-
tion of classifying two documents as a candidate pair. This technique could be extended
by implementing a method that finds parameters to maximize the number of distinct
candidates while minimizing the number of candidate pairs, which is especially impor-
tant for reducing the complexity of building a minimum spanning forest in the developed
compression method.
While the current version of the compression with locality-sensitive hashing does not
support random access, it is believed that it may be modeled as a persistent string in-
troduced by Philip Bille and Inge Li Gørtz [1].
In this way, the collection of similar
documents found with locality-sensitive hashing would be represented by a rooted tree
where edges are labeled with an edit operation. This could allow for eﬀicient random
access in O(log n log log n) where n is the number of nodes of such a tree.
Apart from improvements and modifications to current compression methods, it could be
interesting to compare developed solutions to the general-purpose compression algorithm,
gzip 1. Documents would need to be archived into a tarball and then compressed using gzip.
It is expected that this approach would execute faster but produce smaller compression
ratios for documents of high similarity when compared to the compression with a star-
based approach or locality-sensitive hashing.

Code structure
This appendix gives an overview of how the code and data of developed compression
methods are organized.
The script for fetching Escherichia coli data sets is located in the data directory. All
output files from running compression schemes are generated to the output directory.
The README.md file explains how to run each of the compression methods locally.
It is recommended to run C++ source files using the C++20 standard and Python source
files using version 3.9.10 or above. A Visual Studio Code launch configuration is included
that allows running and debugging C++ files from the IDE.
The implementation of the compression with Lempel-Ziv 77 is located in multiple source
files:
■lz77.cpp includes the compression that uses direct comparison to find phrases.
■lz77-fingerprints.cpp includes the compression that uses fingerprints of length
5 and direct comparison to find phrases.
■lz77-multiple-fingerprints.cpp includes the compression that uses fingerprints
of length 5 and 10, and direct comparison to find phrases.
The implementation of the compression with a star-based approach is located in multiple
source files:
■star-full-matrix.cpp includes the compression with the Levenshtein distance for
pairwise compression between documents. The edit distance is computed using an
iterative approach with a full matrix.
■star-two-rows.cpp includes the compression with the Levenshtein distance for pair-
wise compression between documents. The edit distance is computed using an iter-
ative approach with two rows.
■star-two-rows-bounded.cpp includes the compression with the Levenshtein dis-
tance for pairwise compression between documents. The edit distance is computed
using an iterative approach with two rows bounded by the distance threshold.
■star-rlz.py includes the compression with Relative Lempel-Ziv for pairwise com-
pression between documents.
The implementation of the compression with locality-sensitive hashing is located in mul-
tiple source files:
■lsh.cpp includes the compression with the Levenshtein distance for pairwise com-
pression between documents.
■lsh-rlz.cpp and lsh-rlz.py include the compression with Relative Lempel-Ziv for
pairwise compression between documents. The first script computes candidate pairs
using locality-sensitive hashing and outputs the result to a file. The second script
builds the compression tree using Relative Lempel-Ziv based on candidate pairs
generated by the former script. This script depends on the RLZ compression tool,
rlz_build, found on https://github.com/farruggia/rlzap that should be recompiled
by following instructions on the website in case of any compilation issues.
