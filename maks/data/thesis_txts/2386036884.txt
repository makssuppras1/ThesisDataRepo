Abstract (English)
This thesis describes how to solve model predictive control problems and how
they can be eﬃciently implemented on an embedded computer system. In par-
ticular an optimized solver based on an interior point method and an eﬃcient
Riccati recursion formulation is considered. The approach sketches an itera-
tive strategy where the solution can be approximated by a sequence of matrix
operations on small size matrices.
The presented approach is implemented with focus on a NVIDIA Jetson TX2
embedded system. The system is heterogeneous and thus implementations for
both the CPU- and GPU- architectures are considered. A non-traditional data
layout inspired by the small matrix sizes is described and tested.
Timed simulations shows the implementation using purely the CPU architecture
and the non-traditional data layout is preferred for traditionally sized model
predictive control problems. The results chapter should also provide enough
data to estimate the time needed to solve a given model predictive control
problem using the described approach.

Project description
As per start of project start:
Conducting optimal control on systems with fast paced dynamics creates a need
for fast algorithms for solving the arising linearly constrained quadratic prob-
lems which are inherently small in size. The project will focus on the utilization
of GPU-type architectures on embedded chips in order to reduce the time needed
to solve these problems. As research in the area of small-scale solvers is sparse
and usually focused at sequential implementations on CPUs, a key part of the
project will be to investigate the parallelization possibilities of the serial algo-
rithms in order to achieve a balanced workload on modern multi- and many-core
computing units. Another key part will be exploration of implementation tech-
niques for GPUs and hybrid architectures like NVIDIA Jetson to achieve high
load concurrency on the execution of the subroutines.

Chapter 1
Introduction
1.1
Model predictive control
A commonly occurring task in everyday life is the control of a process, it could be
the heater in a room to control the temperature, the throttle of a car to control
the speed or a complex engineering problem. If you do not want to control the
problem manually you will need a unit that controls it for you. Traditionally
these units have been analogue and in the form of a PID-controller, a PID-
controller adjusts the output based of a (Proportional) measured value, the
previously (Integral) measured values and the trend of the recent measured
(Derivative) values. The measured value is often the deviation from the target,
e.g. when controlling the room temperature; the diﬀerence between the current
measured temperature and the desired temperature. Industrial applications of
PID-controllers include pH neutralization in chemistry and pressure control in
physics.
However, if you have auxiliary information you want to take advantage of, then
the PID-controller comes up short, e.g. for the heater example: if you control
two interacting rooms but want to prioritize one of them, if you based on the
weather forecast want to start the heater in advance or if you would rather have
the temperature a little too low than too high when it is not on the target.
One way to handle and incorporate this type of information is by setting up a
==PAGE:14==
2
Introduction
controller based on model predictive control abbreviated ”MPC”.
As the name suggest the controller revolves around a model of the controlled
system, that based on measurements predicts what state the system will be in
based on the all possible control signals from now to a set time in the future.
Based on a control path’s eﬀect on the system, each set of control paths can
be scored, and the best scoring of the feasible control paths is chosen. Exactly
these two traits is what makes model predictive control attractive; the ability to
explicitly enforce constraints to decide feasible control paths and an extended
scoring system that allows nuances to the target.
Industrial applications of
model predictive control include control of substance-ratios in medium-speed
chemical reactions or administering insulin and glucagon to control blood sugar
(artiﬁcial pancreas).
The cost of applying model predictive control is the need of computational
resources in order to ﬁnd the optimal control path. This introduces a delay
on the decision of a control path, making it diﬃcult to apply the method to
processes with fast dynamics as the delay puts a bound on how often you can
sample and send new controls. It is therefore beneﬁcial to strive for faster and
more eﬃcient implementations of the model predictive control algorithms, as
it widens the possible applications. Most strategies to solve model predictive
control problems relies on linear algebra and its therefore attractive to study
eﬃcient implementation of linear algebra routines.
1.2
Modern computing architectures
Traditionally scientiﬁc computing has mainly been implemented for central pro-
cessing units (CPUs), which is the computer’s default processing unit and usu-
ally also runs the operating system. The unit is therefore very versatile and
designed to be an all-round unit capable of all types of operations and priori-
tizes with its layered cache memory system and independent cores to be able
perform complex memory operations well.
Modern CPUs usually contain a
handful of cores that execute independently, each running 2-4 threads with sev-
eral SIMD units on which the core can execute operations on. In order to get
data into the registers on the SIMD units, data is fetched through the cache
memory from the main memory. The cache layers’ purpose is to pre-fetch the
data needed in the registers before it is needed in the execution to lower the
latency of data loads. The lowest cache layers closest to the registers are the
fastest and smallest, while those closer to the main memory is slower and bigger.
A typical processor has one or two cache levels per core, and a shared cache
==PAGE:15==
1.2 Modern computing architectures
3
layer shared between all cores. A ”cache miss” happens if the CPU tried to
look for speciﬁc data at a some level without ﬁnding it, the CPU will start at
the lowest cache level and escalate to the outer levels if a miss occurs. Making
a cache-miss on a lower level cheaper compared to the cache-misses on the
outer levels. If the CPU ends up having to fetch back from the main memory
a translation lookaside buﬀer(TLB) miss might occur, this corresponds to a
very severe cache miss, and occurs when the translation lookaside buﬀer doesn’t
contain the physical address to the requested data in main memory. This forces
the CPU to look-up the location of the data in the very slow page table. This
TLB-miss is severe and the CPU handles in such a way that it is worth making
sure this is not happening.
Data transported to, from and in between cache levels is transferred in blocks
called cache lines, usually of size 64 bytes. This means that even when the CPU
wants to do an operation on data with a size of 4 bytes, the full cache line of
64 bytes containing the 4 bytes of data has to be fetched through all the cache
levels, without necessarily needing the other 60 bytes. In some cases the 4 bytes
could even be split between two cache lines and both lines would then have to
be fetched to access the data.

A core runs in cycles - usually with a frequency of a few GHz (1-4), and for
each cycle a maximum throughput of one instruction can be completed. This
is under the assumption that a full pipeline of instructions can be ﬁlled. This
stems from the fact that it takes multiple cycles to execute a single instruction.
The speciﬁc number of cycles needed is dependent on the architecture, but
Figure 1.1 shows a basic 5-step RISC instruction execution, e.g. ﬁrst row. First
cycle the instruction is loaded, the second cycle decodes it, the third starts
executing the instruction, the forth access the data needed and the ﬁfth writes
the result back to a register. The reason why a throughput of one instruction
per cycle is possible is because the processor can do one of each execution stage

per cycle like an assembly line. This makes it possible to load up the core as in
the ﬁgure. The only condition is that the instructions with overlay can not have
data dependencies, i.e. instruction three can not depend on the ﬁrst instruction.
One can now see a simple way to speed up the throughput of the CPU is simply
increasing the clock rate. But in 2005 when the clock rate on the processors
started to rise above 3 GHz and near 4 GHz, the heat dissipation started to pose
a problem, and the heat issues is now widely recognised as a wall stopping further
increments to the clock speed on traditional types of CPUs. Instead the extra
transistors available from production advancements is spent on implementing
vector versions of the old instructions. Vector instructions allows an instruction
to perform the same operation to an array of operands at the same time. Vector
instructions is also known as SIMD - single instruction multiple data, and the
vector- or SIMD-width or -length is the number of operands the operation can
be applied to at the same time, as with the instruction latency it requires the
vectorized operations to be independent. However if it is possible to do so, then
the use of the SIMD instructions eﬀectively multiply the operation throughput
by the SIMD-length.
In order to fully maximize the throughput of the CPU it is therefore important
to have enough independent similar instructions to ﬁll both the pipeline and the
vectors. This in turn puts a lot of pressure on the cache memory as the increase
in throughput often also increases the demand for new data, only highlighting
the importance of eﬃcient memory management.
1.2.1
Graphical Processing Units
Graphical Processing Units (GPUs) has traditionally only been used for process-
ing of graphics. But due to increased performance enhancements driven by the
commercial market and improved accessibility to programming of GPUs, they
now also play a key role for general purpose computing as an accelerator to as-
sist the traditional CPU. For problem structures which is suitable for the GPU
architecture the gain immense, while for other problem structures the GPU sim-
ply has no value compared to a CPU. This is due to an architectural trade oﬀ.
Whereas the CPU contain a handful of cores, a GPU contains in the range of
hundreds to a few thousand simpler cores that can execute simultaneously. The
cores are however linked as SIMD units and under execution a so-called warp of
threads running on a SIMD unit all perform the same instruction. A usual warp
size is 32, and thus requiring 32 independent instructions of the same type at a
time in order to maximize eﬃciency, several of these warps is executed at the
same time, and the GPU architecture relies on the independence of the warps,
and the fact that well-written algorithms present suﬃciently many of these. The
==PAGE:17==
1.2 Modern computing architectures
5
execution of an instruction is much like described on the CPU, however on the
GPU the scheduler uses the possibility for a cores to swap in between several
diﬀerent warps in order to hide the instruction latency. This is possible as long
as there is enough warps to execute on. To accommodate the large amount of
threads running simultaneously, a GPU has a similar memory structure to that
of a CPU, registers from where operations is conducted, fast memory the warp
shares, a fast data-cache for pre fetching and a top level memory all cores have
access to. Generally there is more and faster memory available in the GPU
layers compared to the CPU, this is needed because of the high level of par-
allelism in the execution model and the out-of-order execution when the cores
swap warp. When it is possible to utilize the GPU the GPUs computational
throughput beats a same generation CPU by more than one order of magni-
tude. One such example is seen for some linear algebra routines, most notably
matrix-matrix multiplication.
1.2.2
The embedded Jetson unit
Jetson TX2 is system-on-chip (SoC) module developed by the chip designer
NVIDIA[2] which oﬀers a powerful embedded system for computational tasks.
The unit has a form factor that makes it viable as mobile solution and thus
oﬀering an opportunity to have computing power where it is needed, but where
a desktop or server is an impossibility. It features two CPU clusters and one
NVIDIA GPU. One of the CPUs being a Dual-core NVIDIA Denver 2 CPU,
optimized for high-performance on single threaded programs oﬀering 128/64 KB
(instrution/data) level 1 cache and 2MB shared level 2 cache. The other CPU
is a traditional quad-core ARM Cortex-A57, more suited for multi-threaded
application featuring 48/32 KB (instruction/data) level 1 cache on each core
and 2 MB shared level 2 cache. Both processors support ARM-v8 executing
both 32- and 64-bit Aarch code, including the older ARM-v7 instruction set.
The two units are connected by a coherent interconnect fabric from NVIDIA
thus enabling both units to work at the same time and move tasks freely between
cores according to performance with no overhead from manual cache ﬂushing.
The NVIDIA GPU is a GP10b chip with compute capability 6.2 and thus the
2 multiprocessors each have 128 FP32 units but only 4 FP64 units. [3]. This
means the expected performance on double-precision type operands is expected
to be just a 32th of the performance seen on single-precision type operands.
Shared between all of the units is 8 GB of LPDDR4 memory on a 128-bit bus.
Popular uses of the Jetson module is autonomous drones relying on the low
weight and form factor alongside the added beneﬁt of low latency for problems
that normally would transferred to a normal size computer, e.g. in the area
of security inspections. Hand-held vision assistance is also a common use case

1.3
Data formats for matrices
The underlying data structure used for storing matrices plays a key role for
performance in linear algebra routines as it constitute the foundation for an
implementations memory management. Most eﬀorts are focused on reaching
eﬃciency for matrix-matrix multiplication;
C ←A · B + C
(1.1)
As all of the more expensive linear algebra subroutines that exhibit cubic com-
plexity in problem size can be written in terms of matrix-matrix multiplication,
which also makes it the fundamental block of more complex routines as fac-
torizations. The traditional formats to store matrices is in a row format (C
style) where the matrix is stored as the rows of the matrix, and column format
(FORTRAN style) where the data is stored as the columns of the matrix. In
terms of matrix-matrix multiplication there is no advantage for using one stor-
age format over the other if the matrix dimensions are arbitrary. As the format
that makes it possible to contiguously access the needed elements of A means
that the access pattern of the elements in B is with high strides, and vice-versa
for the other format. An alternative to the traditional column- or row- major
order structure is pursued in the GotoBLAS project [5][6], which is shown to be
competitive with other state-of-the-art implementations like e.g. Intel MKL.
1.3.1
Panel-major data format
In an eﬀort to reduce TLB-misses they propose an approach where the matrix
operands are packed at run-time into a panel-major data structure.
Unlike
the other formats, this formats have an integer parameter, the panel height or
blocksize bs; the format can be described as storing the sub-matrix containing
the bs ﬁrst rows and all columns ﬁrst in a column-major fashion followed by the
next bs rows in the same fashion. Figure 1.3 sketches the format alongside the
two traditional formats.
The goal is that a ﬁxed set of columns of the panel of the left hand matrix of
the operation(”A”) can be stored contiguously in the order of use in some cache
level (the L2 cache for GotoBLAS), and then stream in a few columns of the
two last operands (”B”, ”C”) at a time. One might note that the packed format
of the three operands can be diﬀerent, for example the second operand ”B” will
be packed transposed so the memory access becomes contiguous. The approach
from GotoBLAS is based on the online packing of the matrices, but in the case
of embedded optimization this is unfeasible as the matrices is generally small

1.4
Thesis approach
The goal of this work is to present an eﬃcient implementation of an MPC-
controller on an embedded system with a GPU accelerator. The primary target
is the NVIDIA Jetson TX2 platform with ARM CPUs and a NVIDIA GPU.
The main targets will be:
• Formulating the MPC-problem in a way that is solver-able so that an
explicit solver algorithm can be written. The focus will be on achieving
a well scaling formulation with regards to problem sizes that is eﬃcient
from a computational perspective. An eﬃcient approach lays the founda-
tion for an eﬃcient implementation in the end and ultimately put up the
boundaries of the achieved performance in the end.
• Implementation for the speciﬁc architecture - making sure the implemen-
tation utilizes and takes advantage of the resources it provides. Asserting

• Asserting the possible gains of using a panel format - like the ones de-

2.1
The model predictive control problem
When you have to control a process and you measure it regularly, you want to
know which actions to take in order to utilize it optimally. One approach to
this problem is model predictive control,(MPC) which is widely applied in the
industry [8]. The strength of the MPC approach lies in its ability to extract
information from the given model of the controlled system, and the feature to
explicitly deal with constraints on the controlled system.
The idea is to, at each sampling instant, consider which input commands un you
want to send for the next n = 0, 1, .., N −1 sampling times in order to optimize
the output of the process, possibly under constraints on the input commands u
or the states x of the plant.
==PAGE:24==
12
Algorithms for Model Predictive Control

2.2.2
Eﬃcient formulation
There are diﬀerent approaches that can be taken to solve the system of equa-
tions 2.10, and the complexity of the approaches vary. The most straight forward
approach to solve the system is by factorization assuming the that the system is
dense. Using a general LU factorization this approach has cubic complexity in
the size of the matrix, O

(N (nx + nu + ns + ng))3
with ns being the number
of soft-constraints, i.e. the sum of rows in Iuu, Iul, Ixu and Ixl while ng is
the number of general constraints and rows of Cu or Cx. But as the matrix
is obviously structured due to the stacked blocks it is possible to do better.
One way to reduce the complexity is to dissolve the stacked vector notation and
group the the variables stage-wise [12], this results in a banded matrix, with a
bandwidth independent of the time horizon N. This reduces the complexity of
LU factorizing the system to linear in horizon length N while still cubic in the
other model variables.
Taking further advantage of the structure is possible, as the block for each time-
window in the horizon is only coupled to the last time window by the value of
the states. This opens for the possibility of rewriting the problem such that
the time-window blocks is symmetric and positive deﬁnite and thus the time-
window blocks can be factorized one at a time by the Cholesky factorization that
is cheaper than e.g. a LU factorization, this approach is based on the so-called
Riccati recursion.

This part introduces BLAS libraries and presents the panel-major format for
matrices. The goal is to argue for the advantages and choices related to the
format from a architectural point of view. Furthermore an implementation of
the, to the model predictive control application, central Cholesky factorization

3.1
BLAS - Basic Linear Algebra Subprograms
The de-facto way to approach implementing higher level algorithms with lin-
ear algebra operations is using a ”Basic Linear Algebra Subprograms” (BLAS)
library. BLAS operations is divided into 3 levels oﬀering progressively more
complex operations; Level 1 oﬀering scalar-vector and vector-vector operations,
Level 2 oﬀering matrix-vector operations and Level 3 oﬀering matrix-matrix
operations.
Multiple implementations of BLAS exists with diﬀerent goals. Netlib [15] main-
tains an open-source reference which is publicly available. The code is not nec-
essarily written with performance in mind but oﬀers a working reference. Other
implementations, both open-source and commercial, focuses only performance.
Some of the currently most notable:
==PAGE:38==
26
Small Matrix Linear Algebra
• The open-source project ”OpenBLAS” that is based on the techniques
from GotoBLAS implements an optimized library for common modern
CPUs.
• ”Intel MKL” a proprietary but freeware library from Intel optimized specif-
ically for Intel processors.
• ”cuBLAS” a proprietary library maintained by NVIDIA optimized for
their GPU architecture.
However, common for all of them is that their primary focus is to provide an
eﬃcient tool for operations with medium to large matrices. This focus is not
aligned with the needs of the described MPC approach, since usually is less than
100 × 100.
3.1.1
Applied subroutines
The subroutines needed in order to implement the outlined algorithms from
section 2.2 are:
Level 1:
• axpy - vector-vector addition(AXPY) - y ←αx+y for equally sized vectors
x and y and scalar α.
Level 2:
• General matrix-vector multiplication(GEMV) - y ←αAx + βy for general
matrix A, vectors x and y, scalars α and β.
• Triangular matrix-vector multiplication(TRMV) - y ←αAx + βy for tri-
angular matrix A, vectors x and y, scalars α and β.
• Triangular vector solve(TRSV) - x ←A−1x for triangular matrix A and
vector x.
Level 3:
• General matrix-matrix multiplication(GEMM) - C ←αAB + βC for gen-
eral matrices A, B and C and scalars α and β.

3.1.2
Merging subroutines
One drawback of using enclosed BLAS routines is that when two routines are
called in a sequence they are not designed in order to ﬁnish the sequence as
fast as possible, instead they are designed to ﬁnish each individual part in the
sequence as fast as possible. One way to take advantage of a sequence BLAS
routines is to construct routines that performs several operations in one. This
allows to further utilize data when it is fresh in memory, as when a block is
completed in the ﬁrst routine, the operation of the second routine can be directly
processed while the data is available in the registers. In the MPC setting the
approach is most notably applicable for a symmetric rank update (SYRK) and
Cholesky factorization (POTRF) that is performed every time a factorization
is performed. The factorization is particularly suited for this construction as it
only has one data input and some parts of the matrix can be processed without
having access to a lot of data.

For the column-major format it is seen that in order to compute one element
the ﬁrst operand is traversed along rows and thus with a high stride. While
the second operand is contiguously in exactly the order of use. However, the
approach with only computing one value at a time exhibits particularly bad reuse
of the loaded data as each element is used only once. If instead multiple values
are calculated side-by-side, e.g. a 2 by 2 sub-matrix as depicted in the ﬁgure, the
memory access pattern becomes more complex. The pattern in the ﬁrst operand
matrix has doubled the data volume to stride ratio and thus making the pattern
more contiguous. On the other side the pattern for second operand suddenly
has to adopt a stride in the memory pattern order to have two elements of the
same row.
The approach with calculating multiple entries at the same time is a common
implementation strategy and is called blocking. It is used for two purposes: in
order to have enough independent work to hide the instruction latency and ﬁll
up the SIMD registers, and to facilitate reuse of the data as the data can be
used multiple times per load-in. The limiting factor of the blocking size is the
volume of available memory, as optimally all matrix elements would only be
loaded once if all matrices could ﬁt in the memory.
The goal of the panel format is to provide a, at compile time, common static
format for all matrices, that provides better support for the blocking behavior
of the algorithms. Instead of storing the matrices in column- or row-major order
it is proposed to store them as sub-matrices, stored in a column-major style [7].
The format, unlike the other formats, has a parameter bs that controls how
many rows will be stored in each sub-matrix. One could note that choosing a
parameter value of 1 results in the traditional row-major format and a param-
==PAGE:41==

eter value corresponding to the number of rows in the matrix corresponds to
storing in the column-major format. The panel-major format therefore oﬀers
compromise between the two formats.
From a GEMM implementation perspective the obvious choice for the parameter
is to set it equal to the number of rows calculated side-by-side in the before
mentioned block. This results in the access pattern of the ﬁrst operand being
fully contiguous, while the second operand is still accessed with strides, it is
however guaranteed to load data in contiguous blocks of at least size bs times
the number of columns in the block. This access is however still with less strides
than for the case with the column-major format.
A particular favorable instance of the GEMM operation, especially using the
panel-major format, is when the second operand of the matrix-matrix multipli-

3.2.1
Choosing the panel height
As already brieﬂy touched upon, the panel height bs data structure parameter
plays a crucial role in the memory access pattern in cooperation with the block
size. The block size is architecture dependent and eventual a priori knowledge
to the matrix dimensions. Therefore an optimal panel height for the ARM A-57
architecture and the MPC application is discussed below.
The focus in this thesis is on optimizing for the ARM A-57 cores on the Jetson
unit for computing on the small matrices occurring in the MPC algorithms,
where a common case is to receive the second operand in a transposed layout.
To take advantage of this the optimization is targeting the register memory
level, as the full matrix most likely will ﬁt in the level 1 data cache.
3.2.1.1
ARM A-57 processor
We ﬁrst outline the architectural features that have an impact on the choice of
the panel height for the ARM A-57 processor:
• 32 kilobytes L1 data cache, which can hold either a 64 by 64 square matrix
of doubles or 90 by 90 square matrix of ﬂoats, in both cases the cache
should be big enough to ensure no L1 cache misses due to pre-fetching
for the small matrices considered. But it does also outline that there is a
bound to when optimizing for registers is advantageous.
• A A-57 core features 32 128-bit SIMD registers for ﬂoating point opera-
tions, resulting in a SIMD width of 2 for doubles types or 4 for ﬂoat types
types.
• The number of clock cycles needed in order to perform a fused-multiply-
add instruction is 10 cycles.
In order to have 10 independent instructions to hide the instructions latency
the number of SIMD registers used to hold the elements of the block should
be at least 10.
This means a block needs to contain at atleast 20 elements
for double computations and 40 elements for ﬂoat computations. Choosing the
panel height a multiple of the of SIMD width, and the block dimensions as a
==PAGE:43==
3.3 Implementation
31
multiple of the chosen panel height constitute a way of selecting the dimensions
such that a SIMD unit can always be ﬁlled while preserving a simple access
pattern. Choosing a panel height of 4 elements shows to be a good choice for
both double and ﬂoat types, with diﬀerent block sizes.
For 64-bit double precision operations, a panel height of 4 with block size 8 × 4
updates a total of 32 elements, corresponding to 16 SIMD registers at a time. It
leaves 16 registers to hold elements of the two other operands. This correspond-
ing to being able to hold more than a 8 element column of the ﬁrst operand
and 4 element row of the second operand. This will provide data for 16 full
width-SIMD instruction, enough to hide the instruction latency.
For 32-bit ﬂoat operations, a panel height of 4 with a block size of 12 × 4 allows
for updates at 48 elements at a time, occupying 12 SIMD registers, again leaving
20 registers that can hold more than enough data to hide the instruction latency,
e.g. holding a 12 by 4 block of the ﬁrst operand and 4 by 4 block of the second
operand resulting in 48 full width-SIMD instructions.
3.2.1.2
GP10b
The GPU on Jetson TX2 unit is the GP10b chip, the key features being:
• 2 Multiprocessors capable of doing 4 SIMD instructions per clock tick with
a width of 32 for 32 bit ﬂoat types but only 4 normal instructions per clock
tick for double types.
• 256 kilobyte register space per Multiprocessor.
The amount of registers available at this architecture is many times that of
the L1 cache on the A-57 CPU, so for this architecture and the small matrix
sizes the challenge is not to not utilize the register perfectly but more making
sure that the large portion of parallel capacity is utilized. The panel height is
therefore not expected to play a role for the GPU. Instead the panel height is
just set by the CPU architecture to allow heterogeneous computing.

3.3.2
GEMM - General matrix-matrix multiplication
The general matrix-matrix multiplication often appears in the MPC algorithms
where the second operand is transposed. The application in the left looking
Cholesky is another example of this, and it is therefore an implementation of
this variant considered here.
The implementation for the CPU is based on a blocked algorithm with a block
size of 12 × 4 that utilize the available registers, as advocated for in the Panel-
major format section.
One way to implement it is to repeatedly load in 4
by 4 blocks of the second operands while keeping a 12 by 4 block of the ﬁrst
operand.and compute the product. This ensures enough data to ﬁll the SIMD
registers and fully utilize the architecture. The algorithm is outlined in Algo-
rithm 5 without unrolled loops. A real implementation would aggressively unroll
all the loops. Including explicit unrolling of the outer loops that is normally not
unrolled by a compiler.
For the implementation on the GPU the approach is diﬀerent.
Due to the
nature of the architecture with a large register space and a lot of SIMD units
the focus is on exposing enough parallel work on smaller matrices rather than
fully utilizing the register space. One way to expose more independent work than
one thread per element in the resulting matrix is to split the summation over
the contracted index out on several threads. This has a few eﬀects beyond the
increase of SIMD work. It introduces the need of some level of synchronization
between the SIMD units in order to ﬁnalize the summation over the index, in
practice this can be done with atomic operations but it will still cost some time
due to synchronization. Another consequence of the split approach is that it
becomes non-deterministic, as the ordering of summation can change due to the

commutative. It can in extreme cases aﬀect numeric stability, but is normally

4.2.1
SGEMM
Figure 4.1 shows the measured performance of the SGEMM operation on the
Jetson TX2. The panel format approach for the CPU clearly pulls ahead for
the dimensions up to about 80 × 80, which is by far the most relevant MPC
problems. Particularly the panel approach is also superior to the reference CPU
implementation from OpenBLAS. Pulling ahead more than 10% at smaller sizes
with a throughput at up to just under 8 GFlops.
For matrices bigger than
400 × 400 it becomes increasingly important for additional cache management
and the OpenBLAS implementation starts to pull ahead, while the throughput
decreases for the panel approach on the CPU. The throughput of the panel
approach is scaling smoothly as opposed to the OpenBLAS implementation that
experience drops in throughput for certain matrix sizes. This is most likely due
to hardcoded switches between specialized implementations for diﬀerent sizes.
An approach not employed by the panel CPU version.
The GPU panel algorithm is the worst of all the timed algorithms and never
becomes really competitive for matrix sizes below 80 × 80. It provides sudden
jumps at low matrix sizes, due to not having enough work to ﬁll the SIMD units.
Above 80×80 it reaches around 80% of the panel CPU algorithm until 120×120,
where the performance increases abruptly again due to better utilization of
hardware (more active multiprocessors). A way to improve this is by using code
templates so the routines can be written primarily in terms of constants that can
be exploited by the compiler. The other GPU implementation from cuBLAS is
competitive on larger matrices where the highly parallel structure shines and the
throughput scales well with big matrices. Both GPU algorithms provide poor
performance for sizes below 80 × 80 compared to the CPU implementations.

Jetson TX2.
As the implementation for SYRK largely is derived from the
GEMM operation we see the implementations largely rank the same against
each other. For all implementations the throughput is lower compared to the
GEMM implementation. This is primarily due to how much data can be reused.
The number of operations per loaded data ratio is lowered since only half the
matrix has to be computed.
Again, the CPU panel version is fastest for small matrices and the cuBLAS
GPU implementation shines for larger matrices.

Chapter 5
Conclusion
The purpose of this section is to provide an overview of the results presented
throughout the thesis.
The main objective of the thesis was to investigate how practical MPC problems
can be solved on embedded heterogeneous devices such as the NVIDIA Jetson
TX2 that consists of both CPU and GPU cores.
In particular, the applica-
tion and implementation details of BLAS routines tailored for use in embedded
optimization have been considered.
The initial approach in the thesis is implementing the key routines for the MPC
problem in a blocked form that call specialized BLAS subroutines optimized for
small matrices. The BLAS subroutines are hand-coded to utilize the SIMD reg-
isters in an optimal way. To support this structure an alternative data structure
for the matrices is employed that allows for eﬃcient load of data for the SIMD
registers used by modern processing units.
The conclusion is split in three, concluding on the CPU implementations, GPU
implementations and ﬁnally the expected performance for solving MPC prob-
lems on the Jetson TX2 embedded system.
==PAGE:66==
54
Conclusion
5.1
Conclusion for CPU implementations
The panel implementation for the CPU provides high performance for the seg-
ment of matrix sizes relevant for most MPC applications. Figures 4.1, 4.2, 4.3, 4.4
shows a speed up of 10-30% on matrix sizes below 100, compared to other BLAS
implementations. The speed up is attributed to panel data format as it avoids
the need for online packing.
The open-source library BLASFEO [18] that utilizes the panel data format keep
on pushing updates, and shows great performance for smaller matrices [23]. It
consists of implementations speciﬁcally targeted to the SIMD-width and avail-
able register of numerous modern CPU architectures.
5.2
Conclusion for GPU implementations
As initially expected the many-core architecture of the GPU really struggles
for small size matrices, and the sequential nature of factorization algorithms,
like the Cholesky factorization is not an advantage either, as the architecture
heavily relies on the parallelism to hide the instruction latency.
As a consequence of the architectural deﬁcits the GPU implementations is not
competitive with the CPU implementations for the small matrices. This makes
it generally unattractive to use the GPU for traditional MPC problems.
The panel GPU implementations does provide similar performance to the cuBLAS
implementation for the lower sizes, but does not provide the same scaling for
large matrices, and gains of the panel data structure appear absent. It does how-
ever not exclude that it could be an advantage for other panel heights, but due
to the large register space the optimal height may be close to the column-major
data structure due to the very large amount of registers.
The cuBLAS implementation does provide a high computational throughput,
more than one order of magnitude than the CPU implementations, when the
matrix sizes are high.
The GPU therefore brings options for the embedded
system, but not for traditional MPC problems.

5.3 Model predictive control on embedded devices
55
5.3
Model predictive control on embedded de-
vices
When using a system with multiple processors the ideal approach would be to
use all cores at once for computation. However, the traditional interior point
solution method does not oﬀer explicit parallelism to allow taking advantage of
multiple cores at a time. This means the parallelism has to be extracted from
the individual BLAS operations.
In practice we see that the Cholesky implementation for the CPU outperforms
the one for the GPU and thus staying consistent with the BLAS timings. The
point where the GPU implementation starts to gain on the CPU is when the
problem size reaches 400 states.
The Jetson TX2 in the eﬃcient power mode oﬀers a control decision in less than
a millisecond for problems with less than 10 states, less than 10 milliseconds for
30 states and less than a second for 170 states. Figure 4.5 shows the expected
solution time for everything in between and up-to 400 states. All the timings
are for a 10 step horizon but as Figure 4.6 attest the solution times more or less
scale linearly in the horizon length making it possible to scale the shown results
to estimate the solution time for any horizon length.
In general the results shows that at this time its not feasible to use a GPU
architecture for solving MPC problems as the matrix sizes are simply too small.
Instead the GPU architecture could see use in other tasks related to MPC as
processing of measurements or state estimation.
5.4
Future work
Auto-tuning of parameters inﬂuencing the implementation is a common ap-
proach to decide on optimal values of parameters such as block size [16]. Using
auto-tuning to also decide on the optimal value of the panel height parameter is
a natural extension. The auto-tuning has potential for both the CPU and GPU
implementation, but will most likely beneﬁt the GPU implementation more with
a panel height speciﬁcally targeted for the architecture.
Another possibility of improvement comes with the up-coming GPU architec-
ture update from NVIDIA, the Volta architecture, that features a new type of
core called a tensor core [24]. Instead of doing SIMD fused-multiply-addition
instructions the core is capable of doing instructions that perform a full matrix-

matrix multiplication of sizes up-to 16 × 16. Large parts of the MPC algorithm
relies on matrix-matrix multiplication and utilizing tensor cores therefore has
a huge potential to speed up the throughput of the algorithm on a GPU. This
will of course also require a new embedded platform that features a GPU with
the new architecture.
