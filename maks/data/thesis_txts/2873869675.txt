Abstract
This thesis investigates whether open, locally executed vision–language models (VLMs)
can support real-time recognition of simple human–object interactions from video. From
the outset, we impose a reproducibility constraint: the entire pipeline uses only open-
source components and runs offline on everyday consumer machines, avoiding propri-
etary services or cloud dependencies. We evaluate multiple VLMs served via Ollama
under a strict per-frame protocol that emits a small, fixed alphabet of discrete state labels.
To make the stream service-ready, we apply a lightweight post-processing stack: an alias-
controlled object vocabulary, short causal smoothing to reduce frame-to-frame flicker, and
a compact schema-guided validator, drawing on planning/epistemic to enforce legal state
transitions without heavy supervision. The dataset comprises eleven short, single-actor
videos designed to exercise basic hand-object episodes.
Evaluation emphasizes operational behaviour over open-ended caption quality: latency
distributions (percentiles), protocol discipline (strict/loose format adherence), temporal
stability (flip rate, mean segment length), validated transition rate per 100 frames, and
object-label concentration. Beyond offline analysis, we demonstrate a minimal real-time
loop (camera →model →normalization →smoothing →schema) that yields stable state
sequences and coherent primitive events. The contribution is a practical, reproducible
recipe for turning open VLMs into predictable state emitters, together with a compara-
tive evaluation that surfaces latency–stability–discipline trade-offs relevant to on-device
deployment.

Introduction
Motivation.
We investigate whether open, locally run vision–language systems can
drive a small, reliable state stream from video frames in (near) real time. Concretely, each
frame yields one typed token from a tiny alphabet—e.g., HOLDING[obj], PUTDOWN[obj], or
NO_OBJECT—with obj drawn from a short, declared vocabulary. This fixed per-frame out-
put format keeps the machine–human interface predictable and enables lightweight post-
processing for stability and validity. We execute everything locally via Ollama to remove
network effects from latency, use only open models, and emphasise service behaviour
(responsiveness and temporal stability) over open-ended caption accuracy [1, 2, 3]. Lo-
cal, typed outputs with guardrails form a pragmatic path toward deployable perception for
assistive/interactive AI.
Context: MLLMs and VLMs (mini overview).
Multimodal large language models (MLLMs)
couple an LLM with one or more modality encoders (e.g., a vision encoder) so the model
can condition generation on images; recent surveys cover taxonomies and training pat-
terns [4, 5]. Vision–language models (VLMs) are the image→text specialisation: a vision
tower (e.g., ViT/CLIP) feeds a language decoder (or an LLM via a lightweight connector),
trained either generatively (captioning) or contrastively (alignment) [5, 6, 7, 8, 9]. Classic
captioners maximise

text [10]. Our contribution is a usage pattern: restrict outputs to a small token set and add
thin symbolic guardrails so raw, per-frame emissions become consistent action events.
Problem (general).
Given a live video stream of human–object activity, produce a com-
pact, machine-parseable, temporally stable state sequence in (near) real time that sum-
marises what is happening (e.g., which object is manipulated and whether an atomic
action such as pickup or putdown occurs). We require local execution on commodity
hardware using open models, and we emphasise service behaviour so the output can
drive downstream logging, UI, or control.
Implementation note (wire format).
In the live demo we adopt an isomorphic JSON
wire format enforcing a single structured reply per frame, {"action":"HOLDING"| "PUTDOWN"|
,"NO_OBJECT","object":"<v>"|null,"confidence":x}, which we map to the same in-
ternal alphabet as HOLDING[v], PUTDOWN[v], and NO_OBJECT before normalisation and
validation. This choice simplifies robust parsing while leaving the formal specification in
token form.
Specialisation used in this thesis (narrow instance).
To make the problem measur-
able and auditable, we constrain the interface as follows: (i) one typed token per frame
drawn from a small, fixed alphabet; (ii) a short, declared object vocabulary with surface-
form aliasing (deployment-specific exceptions are stated explicitly later); (iii) a thin sym-
bolic layer that validates legal transitions (a finite-state view over “no object held” vs.
“holding o” for o ∈V ). This specialisation is a controlled setting chosen to test whether
open, locally run models can reliably power a real-time state stream.

Micro-examples (terms used throughout).
• Off-format reply (free text instead of a token): e.g., “the person is grabbing a mug
now” rather than HOLDING[mug]. In our specification this is strictly non-compliant; in
this submission we use such cases qualitatively and do not report format-discipline
rates.
• Surface drift (name variants for the same object): HOLDING[mug] →HOLDING[coffee_cup]
→HOLDING[cup]. A surface-form alias map collapses these to a canonical label
(e.g., cup). Aliases do not merge distinct concepts unless declared as a deploy-
ment exception (e.g., glass→cup).
• Flicker (adjacent-frame alternation near a boundary): HOLDING[cup], NO_OBJECT,
HOLDING[cup]. A short causal mode/majority over a k-frame window (k=3 in the
live demo) smooths labels while preserving genuine transitions.
Design sketch (what sits on top of the model).
We combine three thin layers: (i) a
fixed per-frame output format enforced by the prompt; (ii) a normaliser (whitelist V , lower-
casing, and an alias map for surface variants); and (iii) a finite-state validator over states
{⊥} ∪V that only permits legal transitions (pickup, putdown, switch) and guards against
single-frame glitches via minimum hold and cooldown (in seconds). This is equivalent to
a tiny action theory with STRIPS-style preconditions/effects specialised to hand–object
states [11, 12, 13].
Why local execution (practical reliability).
We run all models on device via Ollama
to keep latency dominated by compute rather than transport and to avoid WAN-induced
variance (DNS/TLS handshakes, congestion, rate-limit backoff, cold caches). Tail-latency
sensitivity is a first-order concern for interactive loops; SRE practice therefore tracks la-
tency percentiles (p50/p95/p99), not just averages [3, 2]. In this submission we report
percentiles; SLO hit rates and strict/loose format rates are defined in the specification but
not reported quantitatively.
This thesis: scope, artefacts, and claims.
Our focus is service-centric behaviour
rather than dataset accuracy. We contribute:
1. A fixed-format per-frame interface and a finite-state validator that transform
VLM emissions into a consistent event stream with auditable invariants.
2. A local benchmarking harness that compares open VLMs (via Ollama) across
commodity GPUs using latency statistics (mean/median, p90/p95/p99) and a lightweight
action trace (counts and unique action types) derived from raw replies.
3. A qualitative real-time demo that instantiates the same interface and validator
in a live camera loop (used for behaviour inspection; not used for the quantitative
results).
We define additional sequence- and format-discipline measures in the Canonical Specifi-
cation (Chapter 3) as design goals, but we do not report them quantitatively in this submis-
sion because the present aggregation scripts do not compute them yet. All experiments
use only open models and local execution; no cloud calls are made. The remainder of
the thesis gives background (Chapter 2), specifies the canonical interface/validator (Chap-
ter 3), details the benchmarking harness and the qualitative real-time pipeline (Chapter 4),
the metrics and environment (Chapter 5), and presents results (Chapter 6).

To ensure legal, auditable transitions we maintain a minimal latent store Ht ∈S with
S = {⊥} ∪V,

Evidence alphabet: JSON replies mapped to {HOLDING[v], NO_OBJECT}, and optional in-
stantaneous markers PUTDOWN[v]. With small seconds-based guards, we implement the
following deterministic transitions:
pickup(o) : Ht−1 = ⊥, st = HOLDING, o ∈V, confidence ≥τpick =⇒Ht := o, emit pickup(o).
holding(o) : Ht−1 = o, st = HOLDING =⇒Ht := o. (optionally mark span)
putdown(o) : Ht−1 = o, st = NO_OBJECT or marker PUTDOWN[o] =⇒Ht := ⊥, emit putdown(o) if held ≥hm
switch(o→o′) : Ht−1 = o, st = HOLDING, o′ ∈V, o′ ̸= o, confidence ≥τpick =⇒Ht := o′, emit switch(o→o′).
Guards and preconditions. We use a minimum holding time hmin (for putdown), a per-
action cooldown to avoid chatter, and a minimum confidence τpick for pickup/switch deci-
sions. Actions that require a concrete object disallow the catch-all label other. A short
persistence tolerance δ is optional in the specification but not used in the live demo.
Interpretation.
The validator can be viewed as (i) a small finite-state machine over S
with guarded edges, or (ii) a set of STRIPS-style action schemas with preconditions/ef-
fects on a 1-variable store; both are standard in planning/goal recognition [11, 12, 13].
Invariants (e.g., “every putdown has a preceding pickup of the same object”) follow from
the transition structure.
2.6
Why local execution (latency tails and reproducibility)
All models are served locally via Ollama to remove wide-area network variance and sim-
plify attribution of latency to compute rather than transport. Tail behaviour—not just averages—
matters for interactive systems; therefore we track and report latency percentiles (p50/p95/p99)
in this submission [3, 2]. Definitions for additional measures (e.g., SLO hit rates, format
discipline) are included for completeness but are not reported quantitatively here. Local
serving also makes runs reproducible by versioning model tags, decoding parameters,
and prompt templates together [1].
Where metrics live. Formal definitions of latency-related measures and (design-goal) se-
quence/discipline measures appear once in Chapter 5. Only the latency statistics are
reported in our Results chapter for this submission.

Canonical System Specification
This chapter specifies the interface and post-processing that turn per-frame model replies
into a consistent, auditable event stream. It is implementation-agnostic: any VLM/MLLM
that can be prompted to emit one symbol per frame (Chapter 2) can be wrapped by this
specification. We also state the concrete demo instantiation used in our live system when

3.1
Per-frame output format

5
Benchmarking Methodology
This chapter specifies how runs are executed, logged, converted to tables, and sum-
marised by service-centric metrics. Our focus in this submission is the responsiveness
of locally-served vision–language models (VLMs) measured via per-frame latency, plus
a lightweight action trace for basic activity statistics. Sequence- and format-discipline
measures (e.g., strict/loose compliance, SLO hit rate, flip rate, mean holding span, val-
idator event yield, label concentration) are defined in the specification (Chapter 3) but not
computed or reported here.
5.1
Scope and assumptions
We evaluate on eleven short, single-actor videos featuring simple hand–object episodes
(pick up, hold, put down, occasional switches). Frames are sampled at a fixed cadence;
each frame yields one model call via Ollama and a raw reply string (or JSON in some runs).
For the benchmark path we do not apply smoothing or the finite-state validator when
computing the quantitative numbers; these are used in the realtime demo (Chapter 4)
and retained in the canonical design for future quantitative work.
Why not accuracy against frame-level ground truth?
Our goal is to test whether
open, locally served models can sustain a predictable, low-latency loop under a fixed
interface. Producing dense labels at frame cadence is costly and not required to mea-
sure service behaviour. We therefore report operational indicators (latency distributions)
and simple activity counts; accuracy-style metrics are left for future work once a labelling
protocol is fixed [3, 2].
5.2
Local serving with Ollama
All models are executed on-device via Ollama [1].
Runs specify explicit model tags
and decoding parameters (temperature, max-tokens). The client records enqueue/re-
ply timestamps around each call. Local serving removes WAN variance (no DNS/TLS
handshakes or congestion), making tail-latency attribution primarily a function of compute
and scheduling [3, 2]. For reproducibility, we log CPU/RAM/GPU strings per run.
5.3
Execution and instrumentation
For each sampled frame t of video v under model m:
1. capture enqueue time et,
2. issue one image+prompt call to the local model,
3. capture reply time rt and compute latency ℓt = rt −et (seconds),
4. store the raw reply rraw

Some runs also emit a lightweight action list (e.g., pickup/putdown) inferred by a small
state machine directly from raw replies; we use it only for descriptive counts (no validator

tionable for interactive/assistive settings where latency tails dominate user experience.
6.6
Qualitative demo observations
The live demo runs the full stack online (JSON replies, k=3 pairwise mode, seconds-
based guards). Typical failure modes observed:
• Look-alikes. Visually similar items (e.g., cup vs. glass) can swap; our declared
alias exception reduces surface drift but does not merge concepts by policy.
• Occlusion/motion. Fast motion yields transient NO_OBJECT emissions; the short
window helps without erasing true transitions.
• Off-format protection. The JSON wire format largely eliminates off-format text;
rare deviations are parsed and logged for audit.

7
Discussion
This chapter interprets the results (Chapter 6) in light of the specification (Chapter 3) and
the implementations (Chapter 4). We focus on design choices that paid off, observed
trade–offs, and next steps.
7.1
What the results say (at a glance)
Across three local machines we consistently observe: (i) clear latency tiers by model
(fast/compact vs. slow); (ii) model–dependent tails (some steady but slow, others faster
on average with rare long spikes); and (iii) modest warm-up cost for most models. A
descriptive scatter of event frequency vs. latency shows no strong relationship, confirming
that activity level in the videos does not explain the latency distributions we measure.
These are service findings: they speak to responsiveness and tail behaviour rather than
recognition accuracy.
7.2
What worked and why
Fixed per-frame interface.
Confining replies to a tiny alphabet (§3.1) decouples pars-
ing from recognition and gives a predictable wire format. In the realtime path, using a
JSON schema further reduced off-format drift. While we define strict/loose discipline in
the canonical spec, we did not compute those rates in this submission; the benefit we
exploited here was operational: predictable parsing and easy logging.
Short-window smoothing.
A causal mode over k ∈{2, 3} frames (§3.3) stabilises the
state projection (holding vs. no-object) with minimal delay. Keeping PUTDOWN[o] as in-
stantaneous evidence preserves boundary sharpness for the validator. In the live demo
this was sufficient to suppress flicker without a heavier temporal model.
Finite-state validator.
A single-variable store with guarded edges (§3.4) enforces sim-
ple invariants (e.g., every putdown is preceded by a matching pickup), yielding an au-
ditable event stream. The equivalence to STRIPS-style schemas makes preconditions/-
effects explicit [11, 12, 13]. Even when not used for the benchmark numbers, the validator
is crucial for producing interpretable timelines in the demo.
Local serving.
Running everything on-device via Ollama simplified attribution: per-
centiles and ECDFs reflect compute/scheduling rather than WAN variance [3, 2, 1]. This
made per-model tails comparable across machines.
7.3
Design trade-offs
Alphabet size vs. expressivity.
A small Σ makes systems predictable and metrics
comparable, at the cost of expressivity (e.g., two-handed interactions, partial grasps, in-
tent phrases). This was deliberate: our question is whether open models can sustain a
reliable state stream under tight constraints. Extending Σ is straightforward, but it stresses
prompt discipline and validator complexity.
Surface-form aliasing vs. semantic collapse.
The alias policy is surface-form only
(§3.2). We do not merge base concepts (e.g., cup vs. glass), avoiding artificially inflated
stability. In deployment we used a single exception (glass7→cup) to reduce display churn;
we make this explicit so it can be audited.

Smoothing vs. timing precision.
Short windows reduce jitter but can shift state by up to
⌊k/2⌋frames. Using PUTDOWN[o] as boundary evidence compensates, keeping validator
timing sharp while stabilising the state display.
Guard choices.
Seconds-based guards (minimum hold, cooldown, pickup confidence)
in the demo kept oscillations in check. We did not run a quantitative ablation in this sub-
mission; parameter values are reported transparently so future runs can compare.
7.4
Alternatives considered
Detector+embedder pipelines.
A modular pipeline (e.g., YOLO for boxes + CLIP for
naming) [16, 6] provides strong object evidence. We used VLMs (e.g., BLIP/BLIP-2/LLaVA)
[7, 8, 9] because a single image→text call simplifies per-frame instrumentation and en-
forces “one token per frame”. A hybrid (detector output fed to a constrained VLM) is
attractive but adds latency and engineering complexity.
Heavier temporal models.
HMMs or sequence decoders could model persistence ex-
plicitly. We prioritised transparency and auditability: a short mode filter plus a guarded
FSM is cheap, deterministic, and easy to inspect. Nothing in the spec precludes replacing
the smoothing step with a learned temporal component later.
7.5
Lessons for practitioners
1. Treat the interface as a contract. Constrain outputs and log every stage: raw reply,
canonical token, smoothed state, validator store/event.
2. Optimise for tails, not means. Report per-sample ECDFs and p50/p95/p99; these
determine perceived responsiveness in interactive loops [3, 2].
3. Keep post-processing deterministic first. Thin, auditable guardrails (mode filter +
FSM) buy most of the stability; add learned temporal models only if needed.
4. Version everything. Model tags, decode settings, prompt template hashes, V /A
versions, and capture settings belong in the run manifest.
7.6
Threats to validity and limitations
• No dense ground truth. We intentionally measure service behaviour without frame-
level labels, so we cannot report accuracy-style metrics.
• Heterogeneous capture/decoding across machines. The three systems were
not normalised for resolution/FPS/decoding; cross-hardware comparisons are de-
scriptive only.
• Limited video diversity. Eleven short, single-actor clips may not represent harder
settings (multi-actor, clutter, fast motion, egocentric viewpoints).
• Partial instrumentation in the bench path. The benchmark results cover latency
and a lightweight action list; sequence/format metrics (strict/loose, SLO hit rate, flip
rate, mean holding span, validator yield, label concentration) were defined but not
computed here.
• Alias choices. The deployment exception (glass7→cup) reduces churn but could
mask systematic confusions; we document it explicitly.

8
Conclusion and Future Work
8.1
Conclusion
This thesis asked whether open, locally run vision–language models (VLMs) can power
a small, reliable state stream from video frames in (near) real time. We deliberately nar-
rowed the interface to a fixed per-frame token, added a deterministic normaliser and short-
window smoothing, and enforced legal transitions with a tiny finite-state validator. We built
two implementations that share this specification: an offline benchmarking engine and a
real-time demo.
What we found. Across three local machines, we measured per-frame latency distribu-
tions and warm-up vs. steady behaviour for a range of open models. We consistently
observed: (i) clear latency tiers by model, with tight “fast” clusters and distinct “slow”
tiers; (ii) model-dependent tails (some steady but slow, others faster on average with rare
long spikes); and (iii) modest warm-up costs for most models. On the RTX 5070, many
models achieved sub-second steady averages; on the RTX 4070 and GTX 1080, steady
averages typically fell in the seconds range. A descriptive scatter of event frequency
(pickup+putdown per 100 frames) showed no strong relationship with latency. These are
service findings about responsiveness and tails; we did not evaluate semantic accuracy.
What worked. A small, typed interface plus thin symbolic guardrails made the system
auditable end-to-end: raw reply →canonical token →smoothed state →validated event.
Local serving via Ollama simplified tail-latency analysis and reproducibility by removing
WAN variance [1, 3, 2]. The result is not a new model, but a usage pattern and measure-
ment recipe aimed at deployable behaviour with open, on-device tools.
8.2
Future work
Compute the full metric set.
Extend the converters to produce the sequence/format
metrics defined in the spec (strict/loose compliance, SLO hit rate, flip rate, mean hold-
ing span, validator event yield, label concentration) from existing logs; then add them to
Results.
Add weak labels and accuracy metrics.
Pair service indicators with lightweight labels
(e.g., sparse event timestamps) to report precision/recall and timing error for pickup/putdown/switch.
Normalised cross-hardware runs.
Fix resolution/FPS/decoding and repeat runs across
GPUs to study scaling cleanly; optionally add throughput (fps) and energy per frame as
secondary service metrics.
Richer alphabets and multi-object states.
Extend Σ to bi-manual actions, intermedi-
ate grasps, and object identities; generalise the validator from a single variable to a small
relational state.
Hybrid perception.
Combine region proposals (e.g., YOLO) with a constrained VLM
head to trade a small latency increase for stronger object grounding [16, 6, 7, 9].
Temporal models.
Replace or augment the k-mode filter with an explicit temporal model
(HMM/CRF or a small causal Transformer), keeping the validator as the final guard.

You are given a single video frame. Respond with exactly one of: NO_OBJECT or
