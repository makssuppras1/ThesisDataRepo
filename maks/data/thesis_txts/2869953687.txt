Abstract
In this thesis, we tackle graph­based problems, specifically for directed acyclic graphs
(DAGs). We aim to solve the problem of finding Minimum Chain Covers in DAGs. To
better understand the problem, we design and implement a greedy algorithm, giving an
approximate solution to the problem. Then, Based on the paper ”Minimum Chain Cover in
Almost Linear Time” by Manuel Cáceres[1], we implement a state­of­the­art algorithm and,
through analysis and testing, attempt to improve upon it with our alternative algorithms.
In order to compare the algorithms, we construct a system for generating a set of graph
types. We analyze the algorithm’s strengths and weaknesses across the different graphs
and conclude that there is no significant difference between the algorithms across our
different types of graphs. We then compare the algorithms’ performance directly against
each other in our most generalized type of graph, Random Graph. Here, we conclude
that there is no meaningful difference between the state­of­the­art algorithm and our al­
ternative algorithms.

Introduction
A graph is a collection of vertices and edges. A vertex is most often represented by a
number or a name, and an edge is simply just the indication of a connection between
two vertices. An edge can be directed or undirected, depending on whether the relation
between the vertices is a one­way street or not.
A multitude of scenarios can be modeled using a graph structure, be it a city map of
road connections, a social network, or a factory of machines and their processes. It is an
extremely powerful tool for solving problems in complex systems.
In this thesis, we tackle problems based on directed acyclic graphs. In these graphs, all
edges are directed, and a topological ordering exists. For one such graph G, the primary
problem is to find a Minimum Chain Cover in which all vertices are part of exactly one
chain, and the number of chains used is minimum. For a set of vertices to be a chain, it
requires that there exists a path between each adjacent vertex in the chain.
Minimum Chain Covers can be used to help answer the question of reachability in Directed
Acyclic Graphs. It can tell in O(1) time if a path exists between two vertices, using only
O(k · n) space.
The aim of the thesis is to implement state­of­the­art Algorithms for calculating Minimum
Chain Covers and, through analysis and testing, expand upon these through our own
designs. Finally, we test and compare the algorithms against each other in different en­
vironments.
To achieve this, our paper builds upon the information provided in the paper ”Minimum
Chain Cover in Almost Linear Time” by Manuel Cáceres[1]. He proposes the pseudocode
for two algorithms that can calculate Minimum Chain Covers for Directed Acyclic Graphs,
named Algorithm 1 and Algorithm 2. We have built a framework for graphs in Java 21
and implemented these algorithms. As a part of that process, it required us to design
and implement our own minimum flow solver. In addition, we designed and implemented
a greedy algorithm. Through analysis and experimentation with our implementation of
Algorithm 2, we proposed two slightly altered algorithms that may improve performance.
These are our Algorithms 3 and 4, and through testing on our own generated graphs, we
compare each of these algorithms across various types of graphs we have generated.

So if P = {v1, v2, v3, v4, v5}, any subset chosen from P would be a chain, such as C =

An anti­chain is a set of vertices for which no vertices are pair­wise reachable. This means

other words, none of these vertices would be able to coincide in the same path or chain.
2.1.3
Path Cover and Chain Cover
The definitions of a path cover and chain cover are very similar;
• A path cover P = {P1, P2, ...Pk} of the graph G is a set of k paths, such that every
vertex appears in at least one path in the cover. The cover is minimum when k is
minimum.
• A chain cover C = {C1, C2, ...Ck} of the graph G is a set of k chains, such that every
vertex appears in exactly one chain in the cover. The cover is minimum when k is
minimum.
The notable difference between the two is that in a path cover, a vertex may be covered
by multiple paths, whereas in a chain cover, each vertex is covered by exactly one chain.
Additionally, by Dilworth’s theorem, k is minimum when it is equal to the length of the
longest anti­chain in the graph.
If a path cover is minimum, it is called a Minimum Path Cover, often denoted MPC. Like­
wise, if a chain cover is minimum, it is called a Minimum Chain Cover, often denoted
MCC.
A path cover can easily be converted into a chain cover. For any one particular path
P = {v1, v2 . . . , vl}, choosing any subset of elements from P would be a valid chain, as
any element vj ∈P is reachable by all elements vi ∈P for i < j. So ∀vi ∈V , simply
remove vi from all but 1 of the paths it is included in. The remaining subsets of each path
will be a chain in the chain cover. This is illustrated for a small example in Figure 2.1.

2.3
Tree Structure
For usage in our MCC algorithms, we construct a tree data structure. For all intents and
purposes, we treat this as a DAG, with a set of vertices and edges as defined previously.
To better distinguish between the vertices in the graph and the tree, the tree’s vertices are
instead called nodes. Every node has exactly one parent and a set of children. Each tree
has a root node, being the only node with no parent. The rules and properties of the tree
are defined below;
2.3.1
Rules
The core of the structure is based on 2­3­4 trees with the following rules;
1. element limit: Each node may store up to 3 elements.
2. child limit: Each node may have up to 4 children.
3. ratio rule: Each node must have one more child than it has elements or be a leaf.
Note that in usual 2­3­4 trees, it is also a requirement that the tree remains balanced and
sorted at all times. This restriction is not needed for our purposes, so disregarding this
allows us to avoid complexities in our functions.

1. root: A pointer to the root node of the tree.

2.3.4
merge(A,B)
The merge function merges two trees into one tree. Without loss of generality, we assume
tree A is at least as tall as tree B. We traverse from the root of tree A down from child to
child until we find a node of equivalent height to the root of tree B. In this node, we insert
all elements and children from the root of tree B. This has the risk of breaking the element
and child limit rules, but this is fixed later in the fixTree() function, which we run after
we are done merging.
When traversing tree A to find a suitable node to merge tree B into, we traverse the
children greedily, meaning we select the first child that is not below the desired height
and traverse into that. We are still guaranteed to find a node of the desired height as for
every node, its height is one higher than the highest child, and since we never traverse
down nodes with lesser height than the desired amount, we know we must eventually hit
a node with the right height.
The running time of this function is O(log(k)), as it takes time proportional to the height of
the tree to find the location to do the merge in, and the merge itself takes constant time.
2.3.5
fixTree()
At the end of the merge() call, multiple tree structures may be appended to nodes on
the path from the root to the mergeNode. This may disrupt the tree structure, namely the
element­limit, child­limit, height, and treeSize1. The fixTree() function restores the rules
and properties of the tree. It is initialized at the mergeNode location, as this is where the
last merge() call ended. By recursively fixing the current node and moving to its parent,
we ensure that all nodes affected by the Merge() call are fixed.
Before we changed the tree, all the rules and properties held, and because we only made
changes down to mergeNode, we know that both the rules and properties still hold in
mergeNode’s sub­tree. As mergeNode’s sub­tree’s properties still hold, we can easily cal­
culate the height and treeSize of the mergeNode from equation (2.12) and (2.13). As
we traverse the tree, we know that we have already fixed the sub­tree, making it trivial to
calculate the properties at each node continuously.
To fix the element­ and child­limit rules of a given node v, we use a bubbleUp() function.
If there is an overflow of elements and children in v, we split it into two nodes. The extra
node will have the same parent as v, and like every other time we add a child to a node,
we need to add an element to the parent to still uphold the ratio­rule. This, fortunately,
works out great because if the original v had one less element than it has children, and
both the new v and the additional node need to have one less element than children, then
there will be one element to spare.
Our bubbleUp() function can handle nodes with any number of elements and children as
long as the node still upholds the ratio­rule. For nodes with 8 children or less, the node is
split into 2 parts, and for nodes with larger than 8 children, additional splits are necessary.
In this case, we extract 4 children and 3 elements into a new node and add one element
to the parent as well. We repeat this until the original node has 8 or fewer children left so
that we can split it in two.

Split-size(A,k)
Splits tree A into two trees where one is of size k. The Split-size() algorithm consists
of 3 parts. The main function and the runtime of each step are:
1. Removing sub­trees: Splits k elements from the original tree. Takes O(log(k))
time.
2. Merging sub­trees: Merges the elements split from the original tree into a new tree.
Takes O(log(log(k)) · log(k)) time.
3. Fixing trees: Restores tree properties to both trees. Takes O(log(k)) time.
An in­depth explanation of the steps and their running times can be seen below.
Removing Sub­trees
When removing sub­trees, we want to end up having removed exactly q elements, so we
keep a counter c = q for how much is left to be removed. The overall steps for removing
sub­trees are then to start at the root of the tree, loop over the children, and remove them
if their treeSize ≤c. If a sub­tree is removed, update c. Then, traverse to the first child
and repeat the process until c = 0. Whenever we remove a sub­tree, we remove one
element from the parent as well. This is done in order to preserve the ratio rule, both in
the old tree where a child was extracted and in the new tree where a child is inserted. An
example of this can be seen in figure 2.4. This extra element is removed from the parent
we call a loose parent.
When traversing to the first child, we guarantee that treeSize > c, as otherwise, the child
would have been removed in the previous iteration.
As actions are performed on at most one node per layer, the running time of this step is
O(log(k)).

Algorithms and Strategies
Here, we introduce a range of different algorithms, each capable of finding chain covers.
One is a greedy algorithm, providing a solution quickly, albeit not optimal. The rest are
various flow­based algorithms, each providing an optimal MCC solution.
We describe how each algorithm works in detail, analyze the theoretical worst­case run­

3.1
Greedy Algorithm
A simple approach to finding a chain cover is a greedy algorithm. A side effect of the
greedy algorithm’s simplicity is that it might not find the optimal solution. We denote the
number of chains used in the greedy algorithm as k′. The size of k′ is bound to O(k·log(n)),
which will be proved in section 3.1.5.
Our greedy algorithm finds the longest chain possible, after which it extracts the chain
and tries again. The process for finding each chain can be split into three stages, all of
which are executed k′ times:
• Stage 1 ­ Initialization: give all vertices, not in a chain, a value of 1, and give the
rest a value of 0.
• Stage 2 ­ Evaluate vertices: Calculate value for all vertices.
• Stage 3 ­ Extract chains: constructing the chain Ci with the highest value.

more in­depth description of the steps can be found in the following sections.

Argument for Correctness
The algorithm continues to run as long as there are vertices that are not included in a
chain.
The algorithm constructs the chain containing the most vertices which are not in a chain
already. Therefore, it must include at least one vertex that is not already in a chain in
each iteration. From this, we can conclude that the algorithm must terminate, and upon
termination, each vertex is placed within a chain, producing a chain cover C.

Algorithm 1
The premise is a directed acyclic graph G = (V, E). The goal is to construct a minimum
chain cover C = {C1, ..., Ck} of G. The algorithm is divided into three main stages.
1. Transform the graph in preparation for running a minimum flow algorithm on it.
2. Run a minimum flow algorithm on the graph and return the flow fmin.
3. From the transformed graph and the flow solution fmin, construct the chains on G.
3.2.1
Stage 1: Constructing Gflow
For the first stage, we construct Gflow from G (described in section 2.2.4), on which
the flow algorithms will be executed. In addition, we construct the constraints for Gflow,

(vin, vout) ∈Vflow, we then set cmin(vin, vout) = 1.
These steps each take O(n) time, as the construction of both Gflow and cmin is solely

fmin(u, v) = ffeasible(u, v) −fmax(u, v)
∀(u, v) ∈Eflow

Our maximum flow algorithm runs BFS from s, where only edges with available flow may
be traversed. Edges may be traversed in the backward direction without limitation, ef­
fectively adding negative flow to the edge. Once a path to t has been found, a maximal
amount of flow is sent along the path. This process is then repeated until no path from s
to t exists.
To optimize performance, two optimizations are applied:
Algorithms for Minimum Chain Covers

1. Scaling: Starting at some large integer value d, only accept edges (u, v) for which
f(u, v) ≥d. If no path from s to t can be found this way, set d = d/2 and try again.
This allows paths of higher value to be prioritized, which is helpful as the running
time of the algorithm is linearly affected by the number of paths required. In the
worst case, scaling adds an additional log2(d) iterations, which is irrelevant for a
well­chosen d.
2. Prioritized Edges: When choosing edges to expand upon, for each level of scaling,
we initially only allow edges to be traversed in the forward direction. If no path from
s to t can be found for these restrictions, we then retry for the same value of d, but
this time allowing edges to be traversed in the backward direction as well.
This allows for the BFS tree to reach t quicker, as well as only adding positive flow
to edges, generally filling up their requirements quicker. This optimization generally
had much more impact than scaling for our graphs.
For running time, construction of cmax requires reading through both ffeasible and cmin
which has O(m) and O(n) elements respectively. As there may be up to k paths of flow
1 for our max flow algorithm, it may take up to O(k · (n + m)) time. Calculating fmin takes
O(m) time.

Proof of flow optimality
We have shown that if our flow algorithm provides a minimum flow, Algorithm 1 is able to
extract a Minimum Chain Cover. In order to prove that our minimum flow algorithm yields
a minimum flow, we must first derive some basic conclusions about flow;
3.3.1
Flow Is Always an st­path
Let us start with a single unit of flow. Naturally, the flow must start in the source s and
go to some child of s, say p1 ∈Vflow. As flow conservation must be upheld for any
pi ∈Vflow\{s, t}, the flow must continue to some child of p1. The flow would then follow a
path akin to
{s, p1, ..., pl}
We know the path must conclude, as the graph is a DAG, and we can not have cycles.
The path then ends in some vertex pl, where there no longer is a child to go to. As t by
definition of D−
E(t) receives an edge from all vertices without children; we know that the
path must conclude in t, and thus pl = t. Flow conservation is then satisfied, and we can
conclude that adding 1 unit of flow to the graph must always result in an st­path.
3.3.2
The Sub­tree of s Spans the Whole Graph
Let us look at a particular vertex v ∈Vflow and assume no path exists from s to v. This
must mean there exists a way to traverse an edge backward to a parent of v (if one exists)
and continue to do so as long as a parent exists without ending in s. Thus, there should
exist some vertex vorigin ∈Vflow where v is reachable by vorigin, where D−
E(vorigin) = ∅,
and vorigin ̸= s.
But by definition of D+
E(s), s has an edge exactly to every vertex v ∈Vflow\{s, t} which
has no parent, so therefore vorigin = s for any sequence of parents chosen. So, by proof
of contradiction, any vertex chosen can be reached by s. By extension, the sub­tree of s
must span the whole graph Gflow.

Any Feasible Flow Solution Is a Path Cover

It’s worth noting that both splitSize(), merge(), and fixTree() have the potential of
being constant time. When a vertex sends all its flow to a child, the running time of
splitSize() becomes constant time. Likewise, when a vertex only receives flow from one
parent, there’s no need to merge multiple trees together, and the merge() call becomes
constant time. And lastly, when no trees have been split or merged, there are no changes
to any trees, and thus fixTree() is not needed either, making it constant time.
Tree Height Argument
When performing a sequence of splitSize() and merge() calls, one may wonder how
unbalanced the tree may become. From the definitions of splitSize() and merge (),
they each go to the first child (from the left) suitable for their needs. This means if we
initially had a perfectly balanced tree, from which a splitSize() call has removed one
side of the tree, any subsequent merge () calls will try to fill in the same empty children
first, before inserting into the full side of the tree.
This, however, can be circumvented by inserting trees higher than the ”empty sub­tree”.
In this case, the inserted tree will be inserted into the full side of the tree. If the full
side eventually overflows, the bubbleUp sequence will move nodes up towards the root,
rebalancing the rest of the tree, excluding the single empty child. Hence, it is impossible to
further unbalance the tree through a sequence of insertions and deletions (splitSize()
and merge() calls).
As the tree can not be continually unbalanced, the height of a tree containing k elements
must be bounded by O(log(k)).
3.5
Algorithm 2 Extensions
After implementing, analyzing, and testing Algorithm 2, we considered ideas for improving
it. In this section, we describe two such alterations of Algorithm 2.
3.5.1
Algorithm 3 ­ Prioritized Merges
To test if our tree height argument is true, we introduce a different version of Algorithm
2. Here, when merging trees, the smaller tree is merged into the sub­tree of the smallest
height possible. This ensures that insertions are prioritized in the most unbalanced part
of the tree.
This adds an extra cost to the algorithm’s merge function. The difference occurs when
traversing from the root towards the correct tree height:
• Algorithm 2: Traverse into the first child possible.
• Algorithm 3: Traverse into the child with the lowest height possible.
In Algorithm 2, we simply check if the height of the subtree still is not smaller than what
we are looking for. If it is still suitable, we traverse it and proceed to the next iteration.
For Algorithm 3, we evaluate and store the height of each child’s sub­tree. Once we have
looked at all the children, we go to the one with the smallest height while still being at least
as high as what we are looking for.
In terms of theoretical running time, this change is insignificant. As the maximum amount
of children in a node is 4, the running time of merge becomes 4 · O(log(k)) = O(log(k)).
It is a small extra cost that should help ensure that the tree height is kept as small as
possible, which in turn can reduce running time.

3.5.2
Algorithm 4 ­ Delayed Merges
Another idea for merging trees we tested was to delay the merging of the smaller sub­trees
from the parent until all the parents had delivered their sub­trees.
The case for this is some vertex v where it receives flow from multiple parents. In the base
Algorithm 2, we simply split the desired amount of elements from one parent in a set of
smaller sub­trees and merge them together to gain the desired tree. Then, a bubbleUp()
sequence is performed to fix the properties of the tree along the path in which the merges
were performed. This is repeated for every parent from which the vertex receives flow.
Instead, we put the smaller sub­trees into a list and hold them until all the parents with flow
have sent their contributions. Only then are they all merged together in one sequence,
after which bubbleUp() is performed only once.
There are two improvements to gain from this algorithm. One is that the sequence of
merge() calls only traverses the tree once per vertex instead of once per parent with flow.
The other is that the bubbleUp() sequence is only performed once per vertex instead of
once per parent with flow.
There are two downsides, however. First, the list of all the parent’s sub­trees becomes
much larger than the list of a single parent’s sub­trees we would have in Algorithm 2.
In extreme conditions, this might be very costly in space usage compared to the base
Algorithm 2 approach. Second, it takes more time to sort one big list than multiple smaller
lists, adding up to the same size.
WholeTrees
A wholeTree is a tree that does not have a loose parent, and that is not a leaf. As was
briefly mentioned in section 2.3.6, to preserve the ratio rule when merging trees, we need
the small tree that is merged into the bigger tree to have a loose parent.
Whenever a sub­tree is extracted using splitSize, it is guaranteed to either have a loose
parent or to be a leaf. If splitSize removes an entire tree, it will not have a loose parent,
making it a whole tree. This means that when we merge all the sub­trees in Algorithm 2,
there are no wholeTrees, but in Algorithm 4, when we get sub­trees from multiple parent
vertices, some parents might send an entire whole tree to be merged with the other sub­
trees. With no additional redundancy to handle this case, the ratio rule would be violated.
The way we handle this is to have a counter that is incremented every time a parent
vertex sends a wholeTree. Once all the sub­trees are extracted from the parent vertices,
we extract elements equivalent to the number of wholeTrees from the smallest sub­trees.
Now, whenever a wholeTree is merged into another tree, and it violates the ratio rule, it
will use one of the removed elements to uphold the ratio rule.
The time required to extract enough singular elements to be used as loose parents for
WholeTrees is proportionate to the number of parents. This is because SplitSize() is
called once per parent and can at most return 1 WholeTree. So, therefore, the extraction
of loose parents is free of charge.

Test Descriptions
We have created three different tests that can be run in our code:
• Specific Test
• Compare Algorithms Test

Specific Test
The Specific Test runs a chosen algorithm on a chosen graph. The graph will then be
drawn with the calculated flow displayed on the edges and the contents of the chains
printed in the console. The primary use is to test small­scale examples which can be con­
firmed by hand. Initially, we used this to test the correctness of the graphs and algorithms
during implementation.
Compare Algorithms Test
The Compare Algorithms Test runs a small performance test on a selected amount of
algorithms using a selected graph. This should not be used to compare the running time
of greedy to the other algorithms, as this only times the 3rd step of Algorithm 1 through 4.
Instead, it can be used to compare Algorithm 1­4. The test also prints k, so the test can
also be used to test the precision of the greedy algorithm.
When measuring the running time of the algorithms, we run step 3 multiple times to get
a median average. There are two loops repeating the 3rd step of the selected algorithm:
the inner loop calculates an average, and the outer calculates the median of multiple
averages. The number of loop repetitions can be customized by changing input variables
for the function. This repetition is done to eliminate outliers.
Runtime Test
The Runtime Test is an expansion of the Compare Algorithms test. Firstly, it now gathers
timestamps from every step of Algorithm 1­4, so greedy can be compared to the rest
of the algorithms. Secondly, it will do multiple iterations of the Compare Algorithms step,
increasing the graphSize on each iteration. The idea behind this test is to start on a smaller
graph, then increase the graphSize and rerun the test until the graph becomes so big it
would run out of memory or just take too long to finish. Both the starting size of the graph
and the scaling of the number of vertices and edges can be individually customized by
changing the input of the function. Another change to this test compared to the Compare
Algorithms test is that it will always test every algorithm, and it is possible to select multiple
graphs to test at the same time. The results will then be printed into multiple files, each
containing a latex table. The README in the code contains a more detailed description
of the structure of the output.

Graph Connectivity
Let us first define what it means for a graph to be connected;
For some DAG G, let all edges be undirected. Then, if a path exists from every vertex to
every other vertex, G is connected.
In our non­tree graphs, Bottleneck­ and Random Graph, there is no requirement for the
graph to be connected. The simplest way to ensure the graph is connected is to start
with a tree structure, such as Simple Graph, and then add random edges to that graph
until the desired amount is reached. This would work for both the Random Graph and the
Bottleneck Graph individually in each of its smaller sections.
The issue we saw with a scheme like this is that we would inherently force the properties
of our Simple Graph upon these graphs, which we did not want. The algorithms we run
are unaffected by sections of the graph being unable to reach other parts.
In general, this means that k will be larger for these graphs than if we forced connectivity.
This will, in turn, increase running times across all algorithms, as k is a direct factor.
Arguably, the only downside to this is the removal of choice. If we have singular vertices

in the graph without any ingoing or outgoing edges, the vertex is trivially in its own chain
without any alternatives. So, it becomes the user’s choice to select appropriate values for
the input parameters to make the graph sufficiently versatile.

5.1
Performance of CombineTrivialVertices()
Here, we test and analyze the performance of the combineTrivialVertices() function
upon our different graph types and algorithms. The running time and space usage are
guaranteed to be reduced for all graphs and algorithms, but the question becomes if the
time saved on the algorithms makes up for the time spent simplifying the graph.
All tests conducted in this section are made on graphs with 500 vertices, doubling after
each datapoint. The non­tree graph types start with 2000 edges and double after each
datapoint, while the tree graphs simply have n −1 edges.
Note that we here for Algorithm 1 through 4 only include the running time of stage 3,
the part which extracts an MCC from the flow. The Greedy Algorithm here displays the
entirety of the process, so it will be significantly slower than the flow­based counterparts.
Their direct comparison is not the aim here, and therefore, the selective comparison is
not important. Here, we analyze how the use of combineTrivialVertices() affects each
algorithm. So, they are, in effect, measured against themselves.

Upon the Notgate Graph, the simplification means that all the vertices on the shared ”tail”

Figure 5.1: Time saved by each algorithm by utilizing combineTrivialVertices(). Time spent

Here, both Greedy Algorithm and Algorithm 1 have significant running time improvements,

This makes sense for greedy, as the Greedy Algorithm continues to find the longest path of
vertices not already in a path, which means it will, on every iteration, traverse the entirety
of the tail of the graph. It will find n/2 paths, each of which is of length n/2 + 1, so the
amount of operations is n/2 · (n/2 + 1) = O(n2). Unlike greedy, Algorithm 1 only moves
through the tail once, but for every new vertex it traverses, it will have to update the entire
chainHolder­list, which is of size k (which is n/2). The number of operations required for
Algorithm 1 to traverse the tail is then k · |tail| = n/2 · n/2 = O(n2).
Like Algorithm 1, Algorithm 2 through 4 moves all the flow across the tail in one go, mean­
ing it is only traversed once, but unlike Algorithm 1, The amount of operations performed
on each vertex in the tail is constant time, making the total amount of operations for the
tail being n/2 = O(n). The time cost of performing the simplification is shown as the
dotted line in the figure above, which is significantly less than the time saved by Greedy
and Algorithm 1 while being larger than Algorithm 2, 3, and 4. This also correlates with
the fact that the time saved by Greedy and Algorithm 1 is O(n2), which is more than the
running time of CombineTrivialVertices() at O(n + m), and with the fact that the time
saved for Algorithm 2­4 is O(n) which is less than O(n + m).

5.1.5
Closing Thoughts on combineTrivialVertices()
For the graph types that we have chosen to focus on, the performance of our function,
combineTrivialVertices(), is consistent across all 5. In general, the time saved by
Greedy and Algorithm 1 is greater than the time used by combineTrivialVertices(),
while for Algorithm 2 through 4, the time saved is smaller than the time spent.
For the Greedy Algorithm, using this graph simplification is somewhat cheating, as it uti­
lizes information gained by the minimum flow calculation, which is not done for the Greedy
Algorithm. So naturally, by applying the information of the minimum flow to the algorithm,
it greatly reduced the size of the graph in which the algorithm must traverse, as can be
seen in Table 5.1.
For Algorithm 1, utilizing the information from the minimum flow comes naturally, as the
minimum flow is already calculated through Algorithm 1 in a prior step. The fact that the
time saved is higher than the time spent comes from the fact that the extraction of the chain
cover from the minimum flow is inefficient ­ up to O(k · (n + m) as seen previously. The
time save makes sense, as the running time of combineTrivialVertices() is O(n + m).
Likewise, for the more efficient algorithms 2, 3, and 4, the same step takes up to O(log(k)·
(n + m). Theoretically, there is still potential for a sufficient time save such that running
combineTrivialVertices() would be worthwhile for these algorithms as well. In practice,
however, combineTrivialVertices() guarantees to check every vertex and every edge,
while Algorithm 2 may finish without having scoured every element of the graph. For the
types and sizes of our graphs, the time save proves to be too small for these algorithms.

trees from each parent in one list and only merge them together once flow from all parents

the properties along the merge path.

several calls proportionate to the number of parents. However, as we see, this is not the
case. We suspect this happens for larger graphs, where the list of trees to be merged can
not be stored in fast access memory, adding great costs to access these elements from
a slower memory source instead.

6
Conclusion
In this thesis, we have studied and implemented the algorithms presented in Minimum
Chain Cover in Almost Linear Time by Manuel Cáceres [1]. We introduced modifications
to these algorithms, resulting in a total of five algorithms for testing and comparison.
To further optimize performance, we designed and implemented combineTrivialVertices()
to remove any redundant vertices and edges after the flow calculation. This proved very
effective for Algorithm 1 and significantly reduced space usage across all graph types.
However, the time saved was not worthwhile for Algorithms 2 through 4. It was extremely
effective for the Greedy Algorithm. However, it was disqualified, as it made use of infor­
mation from the flow solution that is not inherently available for the Greedy Algorithm.
Through our comparisons, we found that each algorithm’s relative performance is unaf­
fected by the type of graph, with one exception: Algorithm 4 performed significantly worse
on the Notgate Graph. Based on this conclusion, we move on to compare the algorithms
directly against each other on a Random Graph, in which Algorithm 2, 3, and 4 perform
equally well. This leads us to conclude that the changes made in Algorithm 3 and 4 are
not an improvement to Algorithm 2 and are, therefore, redundant. The fact that Algorithm
2 and 3 perform similarly suggests that the tree heights within Algorithm 2 are reasonably
small already, as Algorithm 3 aims to reduce them even further.
Additionally, in the comparison of Greedy Algorithm versus Algorithm 2, we see that the
greedy solution uses 25% more chains but only 1/200 of the running time. This suggests
that the Greedy Algorithm is a suitable solution if time is a more precious commodity than
precision.
6.1
Future Work
Given the natural time­restraint of a thesis, we have had to exclude ideas of things we
wanted to do from the scope. In this section, we write about and briefly describe some of
these ideas.
6.1.1
Algorithm 1 Extension
We have focused a lot on making improvements on Algorithm 2, so making improvements
for Algorithm 1 was not in the scope of the thesis. However, we did come up with one
optimization for Algorithm 1 that would be something similar to Algorithm 4. In Algorithm
1, moving the flow to the current vertex is done parent by parent. This process requires
the algorithm to linearly traverse the chainholders list, CH, for each parent. This gave a
worst­case running time for Algorithm 1 of O(k · (n + m)).
If we instead gathered a list of the parents and how much flow they are sending each in
a hashtable, the traversal of CH could be done in one go for all the parents. This would
give a theoretical worst­case running time of O(k · n + m).
6.1.2
Improvements to Algorithm 3
Currently, our Algorithm 3 only evaluates the height of every child when traversing down
the tree in the merge function. Had we had more time, it could have been interesting
to test and compare if other ways of testing tree­balance would be better. One such
evaluation could be to look at the element size of all the child sub­trees instead. This
would also combat the possible sparseness of our trees, which might also reduce the
practical running time.

6.1.3
Make Even More Types of graphs
We have made and used five different graphs for our tests, which we are satisfied with.
However, if we were given more time, it would always be interesting to try and find new
graphs that might cause interesting outliers like our Notgate Graph.
We had it in our pipeline to use the Bottleneck Graph to generate a graph in which the
number of bottlenecks was a function of vertices, such that there may be one bottleneck
for every 1000 vertices rather than a fixed amount for all graph sizes like in our current
setup.
We could also expand upon the bottleneck principle and use other graph types as inter­
mediates instead of the Random Graph between the bottlenecks. This could really be any
type of graph.
6.1.4
Use Other Data Structures Besides a Modified 2­3­4 Tree
In the paper we based our thesis on[1], there are multiple proposed data structures with
the same theoretical performance. This means that, given more time, we could have
implemented Algorithm 2 using different data structures(e.g., a mergeable dictionary). As
our parts of experimental improvements are based on the implementation of the data
structure, it would be fair to assume we could have found more ideas for optimization
using the different data structures.
6.1.5
Compare Reachability With Other Methods
The main use case for our algorithms is to check for reachability. This would, however,
require some additional work setting up the algorithms using the calculated chains for
such a purpose. If given more time, it could also be interesting to test if other ways of
finding reachability are easier or more effective than ours.
6.1.6
What Would Be Needed For High Computing Test?
Another change we could have made was to focus more on the limits of how fast we could
get the algorithms to run. This would, first of all, have required us to write our code in a
more time­efficient language than Java. As the minimum flow calculation took up over
99% of the total running time, it has the largest potential for improvements. Investigating
it further could have been interesting, as it would have reduced the overall running time
and made it more comparable with the greedy solution. This would also be required to
press the upper limits of the algorithm’s efficiency. Lastly, our tests were not performed
on a computer suitable for performing highly critical runtime tests.
