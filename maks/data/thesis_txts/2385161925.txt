Abstract
This thesis deals with the objective of rebuilding the old FindZebra.com - which
is a search engine for rare diseases - by applying modern practices. This project
investigate how logic part and presentation part of the website can be decoupled
in order to have a source code which is easy to navigate and expand. This paper
also shows the actual product attached to this thesis is deployed and launched
successfully on the cloud. Lastly, the thesis explains important aspects about
user experience of the product.

Chapter 1
Introduction
This chapter will introduce the reader to the motivation behind the project,
as well as give an insight into how the project is deﬁned. Furthermore, it will
include related work from the industry and ﬁnally give an outline of the thesis.
1.1
Motivation
The Growth rate of the Internet is roughly estimated at 100 percent annually
[CO98], hence the desire to learn how to use this growth of data is increasing.
On the other hand, there is an urgency to have a dedicated tool for health
care professionals in order to help them look up for patient’s symptoms more
eﬃciently and quickly. That’s why FindZebra was born.
FindZebra is a search engine for helping physicians and patients generate hy-
potheses about diagnosis for rare diseases. Unfortunately, development of the
website itself has stagnated somewhat because the code is old, poorly docu-
mented, and hard to work with. The purpose of the project is to recode the
backend for the ﬁndzebra.com website that follows modern best practices so the
website be easier to maintain and improve in the future.
==PAGE:12==
2
Introduction
1.2
Problem deﬁnition
The problem deﬁnition for this project has been developed by the company
FindZebra. Main and the most important problem in this project is that logic
part and representation part of the old FindZebra have been extremely coupled
together in a way that any further development would take huge eﬀort and
energy to understand the very complex structure of the code, hence further
development was stagnated. Designer had a really hard time ﬁguring out how to
proceed with any new ideas due to having an old undocumented code, in other
word, frontend and backend of the website was merged together and became
diﬃcult to distinguish them, which made it impossible to develop further and
test new ideas.
As the title of this thesis describes, this project deals with
best practices for search engine API and web design, following goals have been
deﬁned in that regards.
• Creating static templates that can recreate pages (front page, search page,
etc.) on the current website. (This requires understanding how to separate
the presentation of each page from its logic).
• Working on ﬁlling out the dummy pages with real content by implementing
the needed logic and calls to internal APIs one-by-one.
(This involves
integration between diﬀerent systems with diﬀerent responsibilities)
• Deploying the new website on a server in the cloud. (Working with servers
and cloud services like Amazon Web Service or Google Cloud)
• Improving the web design, either with small changes or a major redesign
(Design, user experience, markup languages, style sheets, mobile friendli-
ness)
1.3
Related work
An interesting article has been published by FindZebra where they compared an
earlier version of their own engine against PubMed, Google and OMIM [SJW15].
The result of this comparison showed an astonishing improvement in ﬁnding a
correct diagnosis compare to other engines. For speciﬁed set of queries with
recognized diagnosis at recall @10 (fraction of cases where the correct result
appears in top 10) FindZebra’s engine provided 59 percent accuracy and at
recall @20 showed 64 percent accuracy while the other engines provided only 33
percent of accuracy in the best case.

Chapter 2
Architecture
2.1
Overview
The most important objective of this project is to separate the logic part from
the representation part in a way that future developer/designer be able to easily
navigate through the source code and ﬁnd the ﬁles he needs to work with, both
frontend and backend ﬁles have to be decoupled in a way that designer does not
have to touch the backend logic to make some changes in frontend and vice versa.
This is considered as the most important architectural design of this project,
since the problem statement clearly indicate this as the signiﬁcant issue.
In the old website this was not the case in a sense that applying any minor
changes in each side would take so much time and eﬀort and could lead to
introducing more issues.
2.2
Model View Controller
Model View Controller is a software architectural pattern for implementing user
interface and relating it to underlying data models successfully and eﬃciently.
==PAGE:15==
2.2 Model View Controller
5
MVC divides an application into three interconnected components in order to
separate internal representation of data from the way that is presented to the
user [Rou]. A simple illustration of MVC is shown in the following diagram
[mvc].
Figure 2.1: Model View Controller
As you can see, three major components have been decoupled for eﬃcient code
reuse and parallel development by applying MVC pattern. Parallel development
is somehow a main concern in this project, even so that I worked on this project
individually, but that is not always the case.
In the future, whenever new
developers and designer join the project, they will not block each others work,
in a sense that backend developer can implement whatever they want without
interfering with the designer who is taking care of the frontend at the same time.
With all that being said, we can fairly support the claim that MVC is a good
idea for such a project, but certainly achieving a pure MVC pattern might not
be completely possible and I will explain why it might not be fully met in such
project.
Model View Controller known as MVC has always been a great technique to
form and design an architecture of a software project. I tried to respect the
MVC as much as possible and as I mentioned in the previous section separation
between view and controller has been fully achieved, but the model was a bit
tricky to handle separately in this project. In the following section, I will explain
some details about the requirements we have to meet and then I will explain
some of the complication and challenges that I have encountered brieﬂy.

2.3
Requirements
In this section I will discuss the general requirements that had to be fulﬁlled in
order to have the basic functionality for diﬀerent part of the website. In order
to draw the big picture I will explain the process brieﬂy.
For the disease section of the website, user would have to insert a search query
which is a symptom(s) of a disease, then he/she would see a list of diseases and
articles that are related to that symptom(s) sorted by the relevancy. There is
also a gene section that allows the user to check the list of the genes that are
contributing to that symptom(s) sorted by their relevancy score. This is the
main functionality we are trying to achieve. There might be one question in the
reader’s mind. What is that relevancy score and how do we calculate that ? I
will explain that in detail when I discuss the gene section.
2.3.1
Overview
The website is supposed to have two diﬀerent sections named ’Diseases’ sec-
tion and ’Gene’ section with diﬀerent requirements, I broke down the task into
smaller pieces and made three helper functions for easier code navigation and
reuse. Brief notes about the methods are mentioned below:
• provideQuery()
This method clean up all the special characters in the query we get from
the user and return a cleaned up query as string
• provideResponse(query, start)
This method takes a query and start argument which is the starting point
for Solr - will be discussed in Solr section - and return a response to show
for the disease section
• provideGene(response)
This method takes a response and return a sorted gene list based on their
scores
The logic behind these functions are explained in the following subsections.

micro framework because it does not require any particular tools or libraries to
function. There is no built in database layer, admin system or any other third
party libraries. In case of any of the libraries were needed, they come as Flask

3.2.3
Alternative option - Advantages over Django
In this section I will review the alternative framework - Django - for Flask and
discuss why I did chose Flask over Django.
The biggest diﬀerence between Flask and Django is that Flask implements a bare
minimum and leaves the details to developer so he can add whatever function-
ality he wants by adding extensions, but Django implements lot of functionality
and follow a "battery included" philosophy. While Flask provides simplicity,
easy control over everything and ﬂexibility by letting the developer decide how
he want to implement the application, Django provides an all-inclusive experi-
ence with functionality such as admin panel, database interface and etc.
In the scope of my thesis having an all-inclusive experience is not the best choice
since I would not use many of those functionality. It seems unreasonable to have
an admin panel and database interface if I am not going to use any of them.
Furthermore, I decided to choose a framework which gives me more control over
components and have the very minimum functionality. Mentioned reasons along
with the fact that Flask has a much lighter footprint consisting of under 10000
lines of code compare to 240000 lines for Django, would lead to choosing Flask
as the optimum framework for this project [Dwy16].

3.3.2
Alternative option - Advantages over Bootstrap
There are plenty of frontend frameworks out there to choose, but two of the most
common are Bootstrap and Foundation. They both have some similarities such
as being open source, responsive(mobile ﬁrst), having 12 column grid system,
stylized CSS components and optional JavaScript extensions.
Despite having these similarities, numerous advantages of Foundation over Boot-
strap had led me to choose Foundation. I will discuss some of these advantages
brieﬂy. [Bra]
==PAGE:25==
3.3 Frontend framework - Foundation 6
15
• Better control over design
Customizing a website is much easier and faster with Foundation, but do-
ing it with Bootstrap would take huge eﬀort to make the site look diﬀerent
than other Bootstrap websites.
• Novelty
The popularity of Bootstrap is like a double-edged knife, as many websites
are using it, recognizing a Bootstrap website is fairly easy, and having a
product similar to others has never been good. Achieving uniqueness is
much easier and cheaper with Foundation.
• Classes don’t have to be added to achieve responsiveness
With Bootstrap, the correct CSS classes have to be added otherwise the
eﬀects will not appear. However with Foundation, base CSS appearances
are built in.
• Better grid system capabilities
Foundation grid oﬀers more ﬂexibility, such as center column, collapse
column and block grid with minimal markup.
• Additional built-in widgets
There are many extra built-in widgets in Foundation which Bootstrap
does not provide, but I will only mention one of them since it is the
most important one for further development which is right-to-left support.
FindZebra will be extended to support more languages and right to left
languages might be added to the project in the future, in that case the
built-in widget is ready to use without any extra eﬀort.

4.1
Cloud Computing
Cloud computing is a well known term describing the delivery of on-demand
computing resources [IBM]. Cloud computing can be described as a collection
of computers that can be utilized for various tasks, such as hosting applications,
hosting websites, data storage and etc. A network connects this collection of
computers, and parts of this collection can be used to achieve a certain goal.
Virtualization is the main technology which makes cloud computing possible.
Virtualization, means a physical computer can be divided up into several virtual
computers and serve diﬀerent goals. By combining these two principles, we have
internet access to compute resources instead of relying on a local machine.
Several third party vendors are oﬀering variety of products, but one of the most
successful one is Amazon Web Services (AWS). One of the important advantage
of cloud computing is scalability, which makes it possible to scale up/down the
amount of resources to meet the ongoing demand of an application. Scalability
provides great ﬂexibility and sooth the urgency to have physical hardware on a
location.
I chose to use cloud computing so I can create and launch an application which is

Deployment
Deploying a ﬂask application to AWS Elastic Beanstalk is fairly easy and straight

The main issue I had faced during the deployment process was an error called
"WSGI path was not found" which caused an unsuccessful deployment process.
WSGI stands for Web Server Gateway Interface which is a speciﬁcation for basic
interface between web servers and web application for Python.
After doing
some research I found out Elastic Beanstalk conﬁgures the WSGI environment
(Apache + WSGI) to launch a WSGI object called application inside a source
ﬁle named ’application.py’ [Cro12]. It’s really unfortunate that all the tutorials
out there are using ’app’ instead of application in their naming and you would
never think such a small detail can lead to a failure. With all that being said,
in order to take advantage of the default conﬁguration the name the ﬁle has to
be ’application.py’ and nothing else.
This little tweaks has solved the problem and I could launch the app successfully.

of the searches to the user which is not really convincing, hence the company
has rolled out the login feature at this stage.

6.1
Progressive Web App
Progressive Web App (PWA) concept is all about web apps that are using lat-
est web technologies and can be installed on mobile devices such as phones or
tablets. Progressive web apps also known as installable web apps are basically
ordinary web pages that can appear like a native mobile app to the user. In-
stallabe web apps might be playing a big role in the progression of apps, and
they can serve as an important piece of a large web ecosystem.
6.1.1
Installation proccess
Installation process is pretty similar to the famous bookmarking process. Once
the user has opened the website, he/she can easily add it to home screen and
install it on the device. This will lead to having an app which is full screen and
stays in open apps separate from the browser. Main advantage and the most
exciting point about PWA, is allowing the developers to create and update apps
across all the platforms by a single code change. Additionally, web apps can
also beneﬁt from device-speciﬁc functions, and with the help of HTML5 web
apps can beneﬁt from local storage, vibrations, sensors and etc.[Fir13]

8.2
Feedback from the user
In this section I am going to discuss some of the most important comments I
received from the user. It should be noted whatever responses he made I took it

without arguing or trying to convince the user. Due to having limited time and
resource I just managed to interview one user. Most of the comments I received
will not be possible to implement in this project, because mostly they are out

to improve the product in the future.
His ﬁrst comments was to criticize about listing Wikipedia as a source. He was
not amused to see Wikipedia as a source for a scientiﬁc product, even though
each Wiki page might have links to a scientiﬁc journal/paper, but his immediate

the suggested words in the ﬁlters did not make much of a sense to the user.

provided a suggestion to reform the ﬁlter section completely.
He wanted to
have the sorted result based on the following categories, gender(male/female),
age(children, middle age, elderly), race(Caucasians, Asians and etc).
The reasoning behind his comment was really interesting to me, since some
of the points were out of the scope of this research I will just mention most
relevant ones. He claimed treating patients from these diﬀerent categories can
be very diﬀerent due to having diﬀerent metabolic rate and other physiological
diﬀerences, for example there are some diseases mostly relevant to woman and
by using the ﬁlters he would like to see those results at the top of the page,
same goes for other ﬁlters, if a disease is more relevant to elderlies then those
results needs to be sorted at top.
The last suggestion he had for this project was adding links to the communities
for certain diseases, so if a patient refer to the website can easily ﬁnd out which
community can get help from. For example by searching "22q11" which is a gene
responsible for Progeria genetic disorder, the link to the supporting community
should be listed as well.
Finally, he had rated his experience 5 out of 10, he had also found the website
very easy to use, used PC to access the website, and would recommend the
website to colleagues.
In my opinion these points worth being implemented for further development in
a side project, but as I mentioned earlier they are considered to be out of the
scope of my project.
