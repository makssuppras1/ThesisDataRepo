Abstract
In the context of automating end-to-end test cases for web applications, testers
need to ﬁnd a set of locator expressions for the automation framework to de-
terministically select the correct interaction points. Locator expressions have
shown to be fragile in the face of web application evolution, even minor changes
to the web application under test can lead test cases to break undermining the
eﬀectiveness and fault ﬁnding capabilities of end-to-end test case suites. To this
aim, researchers have devised methods to compute robust locator expressions,
that are resilient to such minor changes of the application. In this research, we
propose a robustness metric on locator expressions and show how the most ro-
bust locator expression by this metric can be derived through forward-chaining
state space search. The algorithm is implemented in a tool we called Locator-
Factory. Our preliminary empirical evaluation shows that LocatorFactory has
a robustness performance increase of 17% w.r.t. to state-of-the-art algorithm
Robula+ and 174% w.r.t. absolute XPath locator expressions.

Chapter 1
Introduction
For many years web applications have been the primary interface for consumers
to manage online services such as hotel reservations, ﬂight booking, bank ac-
count management, shopping, entertainment services, etc. For some companies,
web applications have indeed completely replaced retail stores, as they provide a
faster and more convenient way for customers to review, compare and purchase
items. It is, however, crucial that the provided services of a web application are
functionally correct. Breakage of service functionality has potential economic
consequences for the service providers caused by the extremely competitive na-
ture of electronic commerce.
A discipline practiced by quality assurance (QA) teams to maintain the func-
tional correctness of a web application is end-to-end testing. For web appli-
cations, an end-to-end test case is a black-box test that speciﬁes the expected
outcome of a sequence of graphical user interface (GUI) interactions. The set of
end-to-end test cases are often derived from system acceptance tests such that
executions of the end-to-end test case suite verify the system acceptance crite-
ria. Traditionally, end-to-end test cases are executed manually and the actual
outcome of the test cases are compared to their expected values. As such, the
set of derived end-to-end test cases can be used to verify the system acceptance
criteria when the web application under test evolves.
For web applications there exists programmable web automation frameworks

such as Selenium1.
Selenium is used to create WebDriver instances for any
popular modern web browser such as ChromeDriver, FirefoxDriver, InternetEx-
plorerDriver, etc. A WebDriver instance provides a set of application program-
ming interfaces (APIs), providing the user with the ability to automate web
browsing. An automation QA team can, therefore, use Selenium WebDriver to
implement end-to-end test cases. Automatic end-to-end test cases written in the
language of the Selenium WebDriver API tends to describe the internal struc-
ture of the web application, as a consequence when the automatic regression
test suite grows, it has a risk of becoming hard to maintain. The page object
model (POM) design pattern [2] is an eﬀort to overcome this risk. For every
webpage of a web application, a page object can be created. A page object is a
class that models the elements of a certain webpage such as the contained but-
tons, hyperlinks, input-ﬁelds, etc. by e.g. the Selenium WebDriver API. A page
object is also comprised of a set of methods that interacts with the modeled
elements in a way that a user would interact with them, such methods could
be named corresponding to the service provided by the webpage. An example
method of a page object could be LoginUser which interacts with the modelled
username and password input ﬁelds and a submit button. As such, page ob-
jects are APIs designed in the language of the application services instead of
the internal application structure.
The research described in this thesis has been carried out at Testware ApS, that
specialises in web test automation. The framework implemented by Testware
automatically explores a web application through web crawling to obtain a nav-
igational model of the web application under test. Page objects [2] are derived
from the navigational model [14, 15] and used as Application Programming In-
terfaces for developing functional end-to-end test cases. In cooperation with
their customers, Testware deﬁnes a set of system-level acceptance test scenarios
to ensure that the application under test preserves its set of core services; such
scenarios are implemented as low-level tests using the interfaces provided by the
derived page objects. Testware has experienced that the most prevalent cause
of failing tests is fragile web element locator expressions.

1.1 Problem Statement
3
1.1
Problem Statement
To make automated tests on web pages, one needs to locate a number of web
elements for interaction. These locators must be unique; a locator must access
one, and only one, element on the web page. Additionally, it needs robustness; if
the element is moved in the same graphical user interface the locator has to keep
selecting this element. In recent years web development has gravitated towards
dynamically loaded, reactive web pages, where content is loaded on runtime
based on the actions of the user. Frameworks like AngularJS and React provide
web developers with the tools to create dynamic web pages, with relative ease.
For these web systems, the web elements are forged by a backend and injected
into the HTML page, often with auto-generated properties. Locators based on
nondeterministic auto-generated properties may break on consecutive runs.
1.2
Research Contributions
This research examines state-of-the-art on web element locators. Based on this
research, an algorithm for deriving robust locator expressions automatically
called LocatorFactory is presented.
1.3
Requirements
In order for the algorithm to be viable in an industrial setting, we outline some
algorithm requirements in this section. These requirements will also serve as
performance metrics for evaluation of the algorithm.
Requirement 1 (REQ1): Completeness
Does the algorithm output unique
locator expressions for any target web element?
In order for the algorithm to be useful in an industrial environment such as the
framework implemented by Testware ApS, the algorithm must have determinis-
tic behaviour. Particularly, the algorithm should always output a unique locator
for any target web element.
Requirement 2 (REQ2): Robustness
How does the novel algorithm com-
pare with state-of-the-art solutions in terms of robustness?
==PAGE:14==
4
Introduction
The algorithm must be able to compete with state-of-the-art solutions for deriv-
ing robust locator expressions. Robust locators diminish page object breakage
and hence prevents false positives when evaluating end-to-end test case suites.
Finally, the algorithm should take precautions to avoid nondeterministic auto-
generated attributes, as locators based on these are indeed very likely to break.
Requirement 3 (REQ3):
Performance
What is the running time and
overhead of the algorithm?
The proposed algorithm must terminate within a reasonable time for the algo-
rithm to be useful in an industrial setting. Furthermore, the measures taken
to ensure the reasonable runtime must not be at the cost of an unreasonable
overhead.
Requirement 4 (REQ4): Readability
Are the locator outputs of the algo-
rithm readable?
In order for test automation developers to correct or be able to debug the broken
test cases, the derived locators should be readable. Readable locator expressions
makes it possible for testers to quickly identify the web element and correct the
broken locator expression manually, or alternatively re-run the locator discovery
algorithm.
1.4
Background
The Selenium WebDriver automates web browsing by navigating the document
object models of a web application. A document object model (DOM) is an
API for HTML and XML documents [10], a DOM tree is a representation of the
structural properties of a HTML/XML document. A web application is often
comprised of a set of DOMs and for every DOM a page object can be derived
and used for automation quality assurance.
In Figure (1.1) an example DOM tree of a HTML webpage is presented.

Nodes of a DOM tree as visualised in Figure 1.1 are denoted web elements. A
subset of web elements are often interactive i.e. clickable or modiﬁable in some
way, this subset is denoted web controls. A set of properties is associated with
every web element of a DOM tree such as a set of attributes: an attribute is an
element modiﬁer that either adds or modiﬁes the functionality of the element.
A standard web element attribute is the id attribute. The id attribute speciﬁes
the web element’s unique identiﬁer, the id attribute value must according to
standards be unique amongst all id’s in the DOM tree that it resides in. The
id attribute of a web element can hence be used for scripting languages like
JavaScript or web automation frameworks like Selenium to select the element
for interaction or modiﬁcation. Using the id attribute hence makes it trivial
to uniquely select elements in a DOM tree. These standards are however inter-
preted diﬀerently by some JavaScript frameworks that forges nondeterministic
auto-generated id attribute values for custom web elements and inject them into
the DOM tree at runtime. These JavaScript frameworks can as such guarantee
that the id values comply to the uniqueness criteria of the standards. They do
however completely neglect the intention of unique identiﬁers i.e. to use them in
scrips for locating web elements. Scrips relying on the forged id values may fail
to select the web element as the id value is updated on every page reload. For
these frameworks, the id value should hence be avoided for scripts to reliably
locate the web element. Furthermore, it is at the web developers discretion to
decide whether to enforce the standards at all. Web applications in which the id
attribute value is not unique amongst the set of id’s is hence not uncommon.
As a consequence the task of ﬁnding the correct attributes for deterministic
and reliable selection of web elements by scripts or automation frameworks is
non-trivial.

Another set of DOM tree properties is the structural properties of a web element.
As visualised in Figure 1.1 web elements are related to each other, they can have

and positioned according to the HTML-tag and/or attribute properties as well
as optional styling from cascading style sheets (CSS). In the context of this
research, we denote the universal set of all web element properties P.

Chapter 2
Related Work
This chapter presents state-of-the-art web automation technologies and algo-
rithms that deal with the problem of discovering unique locators for HTML
web elements. In general, the solutions fall within two classes.
1. Proactive solutions: Discovering robust locators to endure the web ap-
plication under test evolution. An example research belonging to this class
is Robula+ [8]
2. Reactive solutions: Creating an algorithm for repairing broken locators.
An example recent research belonging to this class is COLOR [5].
The research presented in this chapter falls within the proactive solution class
and as such only proactive state-of-the-art solutions are presented and compared
within this chapter.

Katalon Recorder is a record and replay browser extension for creating web

For every web control interaction, Katalon Recorder derives a pool of locator

a locator for the target element from the derived pool of locators.
Katalon

right Selenium locator is studying (a) the web technologies used by the web

2.2
Browser Developer Tools
The web browser applications Google Chrome and Mozilla Firefox have de-
veloper tools allowing the user to inspect and modify the HTML source code
rendered. The user can navigate and highlight source code and output locators
for web elements. In particular, Mozilla Firefox allows a user to right-click a web
element in the source code editor and copy a CSS-Selector as well as an XPath
locator expression for the element. The algorithms implemented by Firefox and
Chrome for deriving locator expressions are however proprietary and not avail-
able for this research. The tools are nevertheless of value as their outputs can
be compared and evaluated against the outputs of the algorithm presented in
this research.
2.3
Ranorex Selocity
Ranorex is another proprietor of the test automation tool providers. Ranorex
Studio is like Katalon Studio a desktop application for managing and automat-
ing end-to-end test case suites.
Ranorex Studio is proprietary software and
the implementation details are hence not available for this research. The team
did, however, release a free browser extension tool, namely Ranorex Selocity.
Ranorex Selocity can be used to derive unique locator expressions, such as CSS-
Selectors or XPath locators. Additionally, the team behind Ranorex Selocity
created a new language RanoreXPath [4] (rxpath) derived from the W3C XPath
language speciﬁcation and a Selenium SDK library to support this language.
The team behind Ranorex presents a solution for dealing with nondeterminis-

Robula+
Robula+ (Robust Locator Algorithm) is a state-of-the-art algorithm for deriving

2.5
Remarks
The study of Katalon Recorder revealed that there does not exist a general
notion of good locator expressions. In order to derive good locator expressions,
the user must have expert domain knowledge regarding (a) the technologies used
and (b) the conventions of the development team. The problem of automatically
deriving unique locator expressions is hence a domain-dependent problem.
In the study of Ranorex Selocity, the development team expressed how the id at-
tribute is, in general, the most stable attribute when creating robust locator ex-
pressions. In recent years, web application development has gravitated towards
frameworks that forge web elements with auto-generated attributes such as ids.
Keeping track and identifying the framework used by the web application under
test to identify whether the id attributes are manually or automatically created
becomes diﬃcult as the number of these technologies is increasing. Ranorex
used machine learning to create and train a model to distinguish the reliable
id values of these technologies. ids are however not the only auto-generated
attribute of these frameworks and the classiﬁer may hence not be suﬃcient.
Robula+ provides a detailed description of how relative XPath locator expres-
sions can be derived from absolute XPath locator expressions to reduce the
locator fragility. The Robula+ algorithm provides a solution belonging to the
proactive solution class. The algorithm is, however, based on a ﬁxed heuristic,
which may be hard to generalise in practice, and in contrast to the proposals of
the team behind Katalon.
The related work research additionally provided the research with the following
ensemble of locator algorithms to use for evaluation and comparison: Testware’s
proprietary locator algorithm, Katalon Recorder, Ranorex Selocity, Robula+,
Absolute XPath, Chrome DevTools and Firefox DevTools.

The feasible regions of the state space search problem instances Ms are com-
prised of all states s ∈S where R(s) = true. For these instances, R(s) = true if
and only if s = le
d(d) i.e. s is a unique locator for e. As discussed, computing the
feasible region is infeasible and is hence not available to the state space search
problem instances. There does however exist the absolute locator i.e. le
a(d) that
resides in the feasible region for any problem instance. Given this information,
we can immediately discard the unique locators of size |le
d(d)| ≥|le
a(d)| as these
goal states are found by longer sequences of atomic XPath transformations and
hence less robust compared to the input absolute locators, according to the pro-
==PAGE:39==
4.4 LocatorFactory Example
29
posed robustness measure. The heuristic function is presented in Equation 4.3:
h(s) = |sa| −|s|
(4.3)
where s, sa ∈S and sa the state representing the absolute locator le
a(d). The
cardinality of a state s denoted |s| is deﬁned in terms of the locator l(d) that s
represents, it is given by:
|s| = |xp| + n −k
where |xp| denotes the amount of XPathNodes contained in the XPath xp, k
is the count of Kleene stars in the XPath xp and n the number of numeric
predicates in xp. As such the diﬀerence |sa| −|s| is a value that represents the
necessary amount of transformations to transform an XPath xp of state s to the
absolute XPath contained in sa. The proposed heuristic function is hence only
admissible in a problem instance where the absolute locator le
a(d) is the only
goal state. The heuristic function hence serves as an attempt to avoid locator
expressions l(d) in the state space where |l(d)| ≥|le
a(d)| and may, as a result,
reduce the number of explored nodes by the algorithm.

4.5
Sequence Caching
The runtime bottleneck of the LocatorFactory is the state space search. The
runtime is remarkably slow for state space problem instances where the closest
goal state is nowhere in the near proximity of the initial state s0. The algorithm
can, however, be conﬁgured to use a sequence caching strategy that may reduce
the algorithm runtime for consecutive runs. When a problem instance (d, le
a(d))
is solved it is possible to cache the sequence of XPath transformations that
derived a unique locator for e. When a similar problem instance (d, le′
a (d) is
initiated where the structural properties of e′ resembles those of e, then the
cached sequence that derived a unique locator for e could be tested for the
problem instance (d, le′
a (d). There is however a concern regarding this approach,
as the robustness of a locator derived by a cached sequence is not guaranteed
to be the best. The LocatorFactory algorithm deals with this issue by timing
the search, and after a set time limit pauses the search to test whether a cached
solution for the target exists. By this approach, the algorithm is given a chance
to derive the most robust locator before using the sequence caching approach.
If no solution was found in time, the algorithm attempts to derive and output a
unique locator by the cached sequence of XPath transformations. The sequence
caching strategy should hence (a) be optional and (b) have a conﬁgurable time
limit.

5.5
Implementation
In this section, a few important implementation details of the LocatorFactory
algorithm are presented.
Conﬁgurations
The algorithm implements a set of standard settings that can be customised
through the Settings interface presented in the previous section. The Settings
interface is comprised of a Strategy interface as well as a Blacklist interface.
The blacklist interface allows the algorithm user to deﬁne a set of policies that
the transition model will enforce, such as blacklisting attributes or transition
functions. The intended primary use of the blacklist interface is to blacklist
auto-generated attributes. As of the current version the blacklist must be con-
ﬁgured manually. The strategy interface allows for further customisation such
as deﬁning a diﬀerent ordering on the stability of the DOM tree properties.
Furthermore, the strategy interface exposes the following set of conﬁgurable
properties:
CachePlans A property that speciﬁes whether LocatorFactory should cache
sequences or not.
CacheTimeout A property that determines when a cached sequence should
be used, the resolution is in milliseconds.
HardTimeout A property that sets a timer for the algorithm to derive a lo-
cator, if the timer expires the absolute XPath is returned, the resolution
is in milliseconds.
LocatorApproach A policy that can be set to either Unique or Robust, if the
property is set to Unique the LocatorFactory algorithm will not produce
locators containing the Kleene star.
MaximumXPathLevels A restriction policy for the AddLevel transition func-
tion.
==PAGE:59==
5.5 Implementation
49
NumberOfAllowedAttributeCombinations A restriction policy for the Ad-
dAttributeCombinations and AddAllAttributes transition functions, such
that the StringPredicate of the resulting XPath retains a reasonable length.
TargetMustContainStringPredicate A property that adds an additional
goal condition to the state space search in which the target web element
must have a StringPredicate in the generated XPath to be accepted as a
goal state.
UseCaseInsensitiveTransformations A property that modiﬁes the AddText
transition to use a case insensitive XPath function.

6.1
LocatorFactory Completeness
The transition model of the state space search solution is ﬁrst and foremost
implemented such that any transformed XPath conforms to the XPath 1.0 lan-
guage speciﬁcation. Furthermore, the transition model is designed such that all
transformed XPath expressions are guaranteed to retain their locator property.
The implemented SSLGoalCondition ensures that the goal states of the state
space search must be unique locator expressions. As the absolute locator le
a(d)
resides in any problem instance of type Ms it is always reachable through the
implemented transition model ∆. The state space search will hence always be
able to derive a unique locator for any valid problem instance. In the vast major-
ity of the cases, the algorithm will however output a locator that is more robust
than the absolute locator consider the following example absolute locator:
le
d(d) = /html[1]/body[1]/div[2]/div[1]/a[2]
Before arriving at this state, the following state will for most of the state space
search instances be explored:
==PAGE:62==

le
d(d) = //body[1]/div[2]/div[1]/a[2]
There can only exist one html tag in a valid HTML document, which means that
the latter presented locator expression is just as unique as the former, and by
the robustness metric more robust. Furthermore, in a case where the algorithm
is unable to derive a unique locator e.g. if it times out, then the algorithm will
output the absolute locator that was given as an input.
6.2
LocatorFactory Robustness
In order to evaluate the robustness of the derived locators by LocatorFac-
tory, an experiment was conducted.
By the use of the Wayback Machine
web.archive.org the evolution of a target web application can be obtained.
For this experiment, 9-10 versions of ﬁve distinct web applications have been
obtained. The 9-10 versions span a ﬁve-year development era from the end of
2014 to the start of 2019. The tested domains are Wikipedia, IMDb, Amazon,
StackOverﬂow, and Google. From each domain, a set of 8-10 web elements such
as hyperlinks, buttons, input ﬁelds, and combo-boxes have been selected and
identiﬁed on the 9-10 versions. An absolute XPath has been derived for each
web element on each version, these will serve as ground-truth locators. A set
of algorithms are run in parallel with the implemented algorithm to see how
the proposed algorithm compares to state-of-the-art solutions. The algorithms
are Absolute XPath, LocatorFactory, LocatorFactory*(conﬁgured version), Ra-
norex Selocity, Katalon Studio, Chrome Dev-tools, Firefox Dev-tools, Testware’s
proprietary algorithm and ﬁnally my implementation of ROBULA+.
From the ﬁrst version, each of the competing algorithms has derived a unique
locator for the target web elements.
The derived locators are tested on the
remaining 8-9 versions and the identiﬁed elements (if any) are compared to
the ground-truth identiﬁed elements, if the tested locator uniquely identiﬁes
the same element as the ground truth locator, it has endured the test. This
evaluation hence reveals (a) the robustness of a derived locator across 9-10
diﬀerent versions of 5 distinct web applications and (b) a direct comparison
between the implemented algorithm and state-of-the-art algorithms in terms of
robustness.

6.4
LocatorFactory Readability
As the LocatorFactory algorithm optimises robustness by ﬁnding the shortest
XPath the outputs of the algorithm are in general short XPaths. Short XPaths
are in general easy to comprehend and read but let us consider the case with
a DOM tree consisting of a single hyperlink. The algorithm could output the
XPath //a which selects all hyperlinks of the DOM which is a single hyperlink
and the locator hence unique. The XPath is indeed readable but there is no
additional information describing the element that the locator was created for,
which is a problem if the locator breaks. Should this locator break, the user is
left with no information regarding the web element it was created for and hence
it becomes hard to repair. With the LocatorFactory algorithm, it is however,
possible to conﬁgure it to reject locator expressions of this kind. The user can set
the TargetMustContainStringPredicate property to true which modiﬁes the goal
condition instances for the state space search. This property will hence ensure
that the algorithm outputs locator expressions with additional information with
the goal of increasing readability. Furthermore, the LocatorFactory algorithm is
==PAGE:80==
70
Empirical Study
in some cases able to translate the derived XPath locator to a Selenium locator
expression which for these cases eliminates the need to understand XPath.

Chapter 7
Discussion
This chapter discusses locator completeness, robustness, performance and read-
ability in the light of the presented results. Furthermore, a section describing
relevant future work to improve or further automate the LocatorFactory ap-
proach is presented.
7.1
Evaluation of Requirements
In this section, the research requirements are evaluated in light of the presented
results.
Requirement 1 (REQ1): Completeness
Does the algorithm output unique
locator expressions for any target web element?
The algorithm has a fallback mechanism to output the absolute locator that
it receives as input, which is required to be a unique locator.
Furthermore,
the state space search in combination with the presented transition model will
eventually always derive the absolute locator. There is however the case where
the user sets the TargetMustContainStringPredicate to true, which makes it
==PAGE:82==
72
Discussion
impossible for the state space search to derive the absolute locator if the state
space search is unsuccessful however the fallback mechanism still guarantees this
requirement.
Requirement 2 (REQ2):
Robustness
How does the algorithm compare
with state-of-the-art solutions in terms of robustness?
The conﬁgured algorithm: LocatorFactory* has a relative robustness perfor-
mance increase of 17% w.r.t. Robula+ and 174% w.r.t. the absolute locators.
The algorithm: LocatorFactory has almost equal performance to ROBULA+
with a 0.8% relative robustness performance increase and a 136% increase w.r.t.
the absolute locators.
Requirement 3 (REQ3):
Performance
What is the running time and
overhead of the algorithm?
The LocatorFactory algorithm has an exponential runtime and space complexity
but terminates in a reasonable time in practice. From the empirical study, 3944
locators were derived in total from ﬁve distinct web applications. The average
time to discover a locator was for the slowest conﬁguration e.g. greedy best-
ﬁrst search 72.151 ms and 16.351 ms for the fastest i.e. depth-ﬁrst search. In
practice, however, it is not recommended to use the greedy best-ﬁrst search or
the depth-ﬁrst search instances as they do not optimise for robustness. For the
recommended algorithms: Dijkstra’s had an average runtime of 22.200 ms and
A* 22.077 ms.
Regarding the overhead, for the recommended algorithms: Dijkstra’s explored
on average 3.219 states and generated a total of 19.872 states on average. The
A* instance explored on average 3.000 states and generated a total of 17.968
states on average. From this study, it seems that our assumption regarding the
implemented heuristic function is true; i.e. that it may help in reducing the
average amount of explored states.

7.2
Future Work
7.2.1
Auto-generated HTML Attribute Value Detection
The way that the LocatorFactory algorithm adresses auto-generated HTML
attributes is by enforcing a blacklist policy conﬁgured by the user. The user
is hence required to have knowledge of the web application under test to be
able to conﬁgure the algorithm properly.
The problem with auto-generated
attributes is that it is a hard problem to determine whether a string is auto-
generated, and as such automating this process was deemed out of the scope
for this research. Using a machine learning model to distinguish auto-generated
from manually created string values deserves further investigation and could be
a solution for reducing the manual conﬁguration of LocatorFactory. Another
possible solution would be to implement an automatic page analyser, the page
analyser should save the state of the DOM trees of a web application, reload the
web application several times and in diﬀerent browsers to compare the DOM
trees with the original and discover diﬀerences. If the diﬀerences occur in the
attribute values, these attributes can be blacklisted.
7.2.2
Heuristic Function Improvements
The presented heuristic function is not admissible and as a consequence the
informed algorithm instances are not guaranteed to discover optimal paths in
the state space. The implemented heuristic function serves as a performance
optimisation currently which is also visible in the preliminary tests, it would
however be interesting to ﬁnd an admissible heuristic function to further reduce
the size of the generated state spaces. One method that could be investigated is
to create a machine learning model with the objective of estimating the optimal
heuristic function h∗(s). The machine learning problem could be modelled as a
regression problem, where the model could take as input a state s and a target
web element and output an estimate on how many transformations s is from a
goal state. The machine learning model could be trained on a large set of web
applications with high variance and possibly be able to generalise an estimate
on h∗(s). Once trained, the LocatorFactory state space search could input the
target web element and a state to the trained model and use the output of the
model as a heuristic.

Discussion
==PAGE:85==
Chapter 8
Conclusion
In this research, the problem of discovering robust locator expressions for web
elements has been examined. A set of formal deﬁnitions and terms were initially
introduced regarding locators and XPaths as well as a measure of the robustness
of locator expressions. The problem was modeled as a combinatorial optimisa-
tion problem with the objective of discovering an optimal solution in the feasible
region, the problem instances are tuples of DOM trees and absolute locator ex-
pressions, and the performance measure of the problem was set to maximise
for the introduced robustness measure. An analysis of the feasible region did,
however, show that it is infeasible to compute and iterate. The feasible region of
the optimisation problem was hence redeﬁned to consist of sequences of XPath
transformations, and the objective changed to ﬁnd a minimum sequence which
by an assumption implicitly solved the same problem as the original optimisa-
tion problem. A solution for exploring the large space of the feasible region is
state space search.
A generic state space search problem formulation and algorithm has been pre-
sented and implemented in a tool that we named LocatorFactory. The runtime
and space complexities of the LocatorFactory tool are exponential in the depth
of the solution, to overcome the inherent concerns of exponential complexity,
concepts from the ﬁeld of dynamic programming were used to derive a sequence
caching approach. The LocatorFactory tool was extended with interfaces for
conﬁgurations such as modiﬁcation of the transition models weight distribution,

blacklisting auto-generated attributes, modiﬁcation of the ordering on attribute
robustness, state space search algorithm selection and timeouts.
The LocatorFacatory tool was evaluated through four requirements namely: ro-
bustness, performance, completeness, and readability. An empirical study on
the robustness showed that the conﬁgured LocatorFactory tool has a relative
robustness performance increase of 17% w.r.t. Robula+, 21% w.r.t. Katalon and
174% w.r.t. absolute locators. The unconﬁgured version of the LocatorFactory
tool has a relative robustness performance increase of 0.8% w.r.t to Robula+
and 136% w.r.t.
absolute locators.
An empirical study on the performance
of the LocatorFactory tool was conducted in which 3944 robust locators were
derived for each state space search algorithm, the results showed that for the
recommended algorithm Dijkstra, the average time to derive a robust locator
expression is 22.200 ms and for the A* instance 22.077 ms. The completeness
requirement forced the algorithm to output unique locator expressions for any
given problem instance, as the problem instances contain absolute locators as
input, the completeness requirement was trivially satisﬁed by returning the ab-
solute locator as a fallback mechanism. It was however shown that in the vast
majority of the cases the absolute locator was never returned; in the performance
experiment where 3944 locator expressions were derived, none were observed to
be equal to the input absolute locator expressions.
In the discussion chapter, we discussed some improvement proposals for further
automation of the LocatorFactory algorithm and end-to-end test generation in
general.
The LocatorFactory algorithm does not automatically detect auto-
generated attribute values but instead relies on a manually conﬁgured blacklist
to avoid these. The team behind Ranorex presented their solution in which
they trained a machine learning model to distinguish auto-generated attribute
values from manually created. This approach could ﬁt nicely as a policy for
the transition model of the introduced state space search. Furthermore, the
implemented heuristic function used by the informed instances of the state space
search is not admissible and as a consequence does not guarantee to ﬁnd the most
robust locator expressions.
The state space search algorithms: breadth-ﬁrst
search and Dijkstra’s are however guaranteed to ﬁnd the most robust locators.
Finally, let us review the problem statement. The objective of this research was
to ﬁnd an algorithm that (a) derives unique locator expressions and (b) derives
robust locator expressions. The implemented LocatorFactory tool ensures by
its completeness criteria to satisfy (a), and has been shown in an empirical eval-
uation to be competitive with state-of-the-art algorithms regarding robustness
and is hence deemed to satisfy also (b).
