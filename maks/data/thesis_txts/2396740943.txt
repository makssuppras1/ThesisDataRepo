Abstract
Central Oﬃce Re-architectured as Datacenters (CORD) is a new paradigm emerging
in mobile networks to handle the increasing load on the Evolved Packet Core (EPC)
in a more scalable way. It employs a data centre approach to provide connectivity
services for mobile users and will eﬀectively replace the traditional EPC. The use of a
Software-Deﬁned Networking (SDN) controller allows to employ automated behaviour
in a mobile network. Through an Application Programming Interface (API), control
applications in ONOS and the CORD deﬁned services can communicate and hence
leverage direct access to state and conﬁguration data.
This paper suggests a veriﬁed proof-of-concept where in the Radio Access Network
(RAN) is managed by an SDN controller. Especially the the Remote Radio Head (RRH)
is interesting to manage since the conﬁguration and monitoring can be automated,
and hence introduce Self-Organising Network techniques. The design suggests using
the Network Conﬁguration protocol (NETCONF) to communicate between the SDN
controller and the devices, which requires a data model of the accessible data. Thus,
an approach on how to model a digital hardware component in YANG is given. The
design covers aspects from hardware registers to APIs

Chapter 1
Introduction
Citizens of the world have embraced the mobility gained from the development of
wireless communication. Enhanced production methods in consumer electronics and
globalisation lowers prices on equipment and continuously bring mobile communication
to broader audiences. Concurrently, new applications and services emerge which utilises
cutting edge technologies. Hence, telecommunication companies (telcos) are foreseeing
new bottlenecks in their existing networks and the technologies they use. According to
statistics from Cisco [6] and Ericsson [1], the traﬃc load from ﬁrst quarter of 2016 and
ﬁrst quarter of 2017 the amount of data sent on mobile networks increased by 70% and
the forecast predicts it to keep increasing. By 2021, the total data transferred in mobile
networks will reach 49 Exabytes. This is a seven-fold increase of traﬃc compared to
the 7 Exabytes of traﬃc in 2016.
How to support the traﬃc growth in the RAN and mobile front and backhaul is
currently being addressed in collaboration between equipment vendors and telcos. One
topic is the transitioning between the currently deployed 4G network to include 5G tech-
nology. Through this expansion, telcos seek to improve the performance and capacity
of their networks.
5G aims to fulﬁl two challenging requirements: Lower latency on the end-to-end
connection and gigabit throughput. The radio properties deﬁned for 5G includes the
use of spectra above the six-gigahertz range, since most of the long ranging frequencies
already are reserved for other technologies. High frequency spectra have a limitation
in line-of-sight due to the poor propagation properties they have, which means they
hardly can penetrate a human body or nature phenomena like precipitation. Thus,
antennas must be placed strategically and relatively close to each other in order to
provide continuous coverage. The latter forces the number of RRH to increase, as 5G
RRHs are not able to replace the need for the long-ranging LTE RRHs. The physical
design of 5G base stations contains up to hundreds of small antennas that individually
requires a set of parameters, which per base station sums up to a substantial amount
of conﬁguration data that potentially must be updated. [10]
1
==PAGE:13==
Traditionally, a dependency on special purpose hardware has been commonplace
for telcos to operate their networks. The available hardware is not sold in huge quan-
tities since they have limited use cases, hence it has a signiﬁcantly higher price than
oﬀ-the-shelf hardware. Furthermore, the hardware components do not necessarily scale
well with their related components, making it an expensive and ineﬃcient setup for
telcos. To address this issue, telcos have joined forces to design a data centre platform
to replace the special purpose hardware with commodity alternatives. This project is
called CORD and is an open-source project that brings together NFV and SDN and
deﬁnes a deployable installation to use by telcos for multiple purposes within the in-
dustry. CORD also solves the utilisation problem, as it can be deployed in data centres
and thus all resource investments goes into one pool that can be allocated to individual
sources as their resource consumption grows. Since it is designed for a data centre,
the capital expenses are lowered as data centre hardware is an oﬀ-the-shelf product.
The scope of this thesis is limited to the version of CORD targeted the mobile indus-
try named M-CORD. The requirement for additional RRHs to support 5G reveals a
necessity to maximise the gain from investments, leaving only the least possible re-
sources unused. An approach to do so, is simplifying the hardware most possible and
relocate heavy computational operations to a centralised location to reduce the total
investment in hardware and increase the overall utilisation. To address the massive
conﬁguration issue originating in 5G, there exists multiple network conﬁguration pro-
tocols which allows telcos to automate network conﬁguration. E.g. through concepts
like Self-Organising Networks (SON), which can be included as a part of CORD.
1.1
Problem statement
This thesis investigates the components necessary to employ automated conﬁguration
and monitoring of components in a mobile operator’s network. The goal is to implement
a proof-of-concept with an RRH that can be conﬁgured through an SDN controller. The
SDN controller will allow proprietary applications to deﬁne the behaviour of the network
and hence give the opportunity to have an automated network. The goal can be split
into two subsidiary parts.
First part is a software implementation which can communicate with a provided
hardware component. The hardware component is implemented on an FPGA board, so
conﬁgurations should be written directly to the hardware, preferably from an on-board
processor. It is not the intention that the outcome of this part of the project is limited
2
==PAGE:14==
to be applicable for an RRH only. Ideally the software package should be conﬁgurable
for other hardware components with a similar architecture and where remote controlling
is desired.
Second part is an SDN controller which should be capable of both conﬁguring a
device automatically when the connection between hardware and controller is estab-
lished, and push conﬁgurations to the device that originates from human interaction or
a control application.
1.2
Thesis structure
The thesis is structured as follows:
• Chapter 2 describes a set of concepts within mobile networks and software-deﬁned
networks which will assist the reader’s comprehension of related technologies
• Chapter 3 covers technologies that are leveraged throughout the following chap-
ters.
• Chapter 4 describes the architecture of a setup where software-deﬁned networking
is introduced into conﬁguration of the mobile network.
• Chapter 5 veriﬁes the implementation of the previously described architecture
and tests scenarios which could lead to undesired behaviour.
• Chapter 6 describes use cases where the functionality of a managed remote radio
head would be needed.
• Chapter 7 concludes the thesis and examines whether the goals are achieved and
suggests future work.

Chapter 2
Concepts within next generation mo-
bile networks
This chapter is to give an overview of the concepts that are helpful to comprehend while
reading the following chapters. Initially, a comprehension of the elements existing in
a mobile communication network will be necessary to understand how virtualization
can be incorporated. The description will focus on a 4G network, as it is the most
recent mobile technology that is in service intended for commercial use. The concept
Software-Deﬁned Networking has potential to be the future way of managing the mo-
bile network together with the concept of Network Function Virtualization which is a
way of instantiating network functions, all in a virtualised environment. This chapter
furthermore explains the basics of network conﬁguration.
2.1
Mobile Communication Networks
Long Term Evolution (LTE) consists of two main parts: The Radio Access Network
(RAN) and the Core Network (CN). The RAN has the purpose of transmitting control
and data plane traﬃc to the User Equipment (UE) via the radio interface existing
between the UE and the Remote Radio Head (see ﬁgure 1). The main device in the
RAN is the Evolved NodeB (eNodeB) which has the properties to receive control and
data plane traﬃc from the Core Network. The Baseband Unit (BBU) schedules the
traﬃc onto the radio interface and converts it to radio waves and forwards it to the
Remote Radio Head (RRH) via Common Public Radio Interface (CPRI). CPRI is a
protocol which samples radio waves and packages it to be send on either an electrical
or optical link. In later releases of CPRI, other splits deﬁning which tasks the BBU
pool and RRH respectively handles have been deﬁned.
The CN consists of several entities which each is a part of connecting a UE to
the internet. It has the Mobility Management Entity (MME) which is responsible for
signalling which do not regard the radio interface. For example, when authenticating a
4
==PAGE:16==
UE, the MME uses the Home Subscriber Server (HSS) which is a database of subscribers
of the operator, to see if a device attempting to access the network is allowed. The
Serving Gateway (S-GW) is responsible for managing all traﬃc on the data plane, thus
the data tunnels which exists between UEs and the Packet Data Network Gateway
(PDN-GW) or other devices within the operator’s network. The Packet Data Network
Gateway is responsible for the connection to external networks, so for example the
internet or other mobile network operators. Hence, it is also responsible for assigning
IP addresses to mobile devices as a part of the authentication process. Remind that
LTE is a designed in a packet-based approach rather than circuit switched and thus
have no entities for this type of connection as seen in previous generations of mobile
networks. These entities are also referred to as Evolved Packet Core (EPC). [28]

2.1.1
Fronthaul network and Cloud RAN
In the initial description of LTE, the intention was to install a BBU at each physical
location to work for multiple RRHs, but the baseband processing is one of the more
power consuming tasks in the eNodeB. Thus, initiatives were taken to reduce the costs
were to create a design that relocates multiple BBUs to one physical location which
5

Figure 2: LTE network overview with Cloud Radio Access Network.
then serves RRHs in multiple physical locations. The extraction of the BBU arises
some new challenges in the backhaul, as sampled radio data generated by the BBU
pool requires many times more link capacity than the IP packets which it is generated
from. Moreover, it also has timing requirements for when it should be aired by the
destination RRH. A solution to this is Radio-over-Ethernet (RoE) that also introduces
a new notation of the network to transfer the data on: the fronthaul network, as
depicted in ﬁgure 2.
The concept is called Cloud Radio Access Network (C-RAN)
The BBU pool introduce several other improvements regarding network capacity and
architecture which can be found in [19] & [21].
The existing RRHs does not support the RoE standard. Furthermore, it is lowers
installation and maintenance expenses to only have installation of the RoE hardware.
To enable the communication, a gateway with RoE support is used. It can connect to
the BBU pool and inform about which RRHs it has contact to. Hence its name Remote
Radio Head Gateway (RRH-GW).
2.2
Software Deﬁned Network
SDN evolved as a set of already set ideas about network routing featuring "plane sepa-
ration, a simpliﬁed device, centralized control, network automation and virtualization,
and openness". [13] These ideas were concretised by the standardisation of the Open-
6
==PAGE:18==
Flow (OF) protocol which describes two traﬃc planes: the control plane and the data
plane. The data plane is where all host packets are sent, e.g. one host sending a packet
another host. In SDN, when a host sends a packet into the network, the packet is
received by an OF-switch which has a prioritised list of rules. Each rule has a pattern
to compare a packet’s header with and an associated action for the matching rule. A
rule is called a ﬂow, referring to the ﬂow of data since one packet is rarely alone, and
a list of rules is called a ﬂow table (see ﬁgure 3). For simplicity, a rule can only be a
forwarding rule like in a router. If a rule matches, the action is applied to the packet
by the switch and the packet reaches its destination. If there is no match, a copy of
the packet is encapsulated in an OF packet and forwarded via the control plane to an
SDN controller. The controller is a centralised entity which has knowledge about every
host that is connected to the network. If the destination host exists, the controller can
report back to the switch what port it should forward the packet and all coming packets
to and if not, it can us discovery techniques, or eventually drop it if the host is never
found. Looking beyond the example above, a ﬂow can match on patterns on multiple
layers in the OSI model and has several other actions than block and forward. [13]
Figure 3: Simple SDN network.
The controller is the brain of the SDN network. It holds the knowledge about the
topology of the network, e.g. which switch port each host is connected to, about which
ﬂows that currently are installed in the switches and about each device which exists in
the network as seen in ﬁgure 4. The northbound Application Programming Interface
(API) exposes the network properties to applications which can be used to control
how the network should be acting to each packet, meaning that proprietary routing
7
==PAGE:19==
protocols or new discovery protocols can be implemented. The exposed information is
communicated to and from the switches through the southbound API.
Figure 4: SDN controller anatomy. (modiﬁed version of original from [13]).
2.3
Network Function Virtualization
While SDN has focus on packet switching, e.g. determining routes between hosts in a
network and enabling some functionalities which are typically maintained in the edge of
a network, like the ﬁrewall, boxes containing special purpose hardware still exists in the
network. Network Function Virtualization (NFV) refers to the implementation of these
boxes in a virtualized environment running on commodity hardware and each service
that a box represents, is referred to as a Virtual Network Function (VNF). Regarding
the network topology, the change will not have any impact to move functions from
boxes special purpose hardware to VNFs. Furthermore, it is a change which according
to [17, 20], is anticipated to lower the capital and operational expenses within the mo-
8
==PAGE:20==
bile communication industry among others. Moreover, it lowers the deployment time
and improves the scaling options for VNFs. A key feature in NFV is the automatic
provisioning of resources for each VNF and to achieve this, is using a "model driven
architecture": "Model driven templates describe the entire network service, with conﬁg-
urations for each VNF, its topology and connectivity, various dependencies and policies
for triggering actions speciﬁed in a human-readable ﬁle." [17] NFV also promotes con-
tinuous development and update delivery as time-to-market for software products are
signiﬁcantly smaller than hardware. The virtualised environment enables operators to
roll-out updates to few services initially and isolate each update to monitor for oper-
ational disturbances caused by the update. The ultimate goal for NFV is to achieve

3.1
Network management protocols
Over time networks has increased in size and it became necessary for network ad-
ministrators to have a tool to assist with conﬁguration of network devices. Network
management protocols deﬁnes a standard way to read and modify conﬁguration data of
a managed device. In the following, an insight to typical network management protocols
is given.
3.1.1
Simple Network Management Protocol & MIB
The Simple Network Management Protocol (SNMP) is an open standard for managing
devices over a network and was standardised by the Internet Engineering Task Force
(IETF). It provides a collection of basic set and get operations for an SNMP manager
to read and alter conﬁguration data through an SNMP agent located on a managed
device. The device decides which manager that has access to what data. There is also
the operation ’Trap’ which is used to subscribe to a change in a variable and thereby
monitor a device. SNMP is based on UDP, which is a "ﬁre-and-forget" protocol. Thus,
admins could not know whether a message reached its destination without checking. In
later versions came the ’Inform Request’ message which introduced acknowledging
messages that was received which increased the reliability of the protocol. SNMP does
not feature a message type to execute commands on the agent. Instead, either ﬂags
are used to trigger events, like a reboot ﬂag, or a countdown value for when the reboot
11
==PAGE:23==
should be performed. In the ﬁrst versions of the SNMP protocol no noticeable security
existed. Through events the past decades people using SNMP started to realise that it
had become necessary. Proposals on how to overcome this challenge started to appear
and are described in various RFCs where the widely used Secure SHell (SSH) and
Transport Layer Security (TLS) are suggested as optional transport channel (see RFCs
5590-5592, 5608, 6353).
The structure of information in SNMP is a Management Information Base (MIB).
It uses a subset of the Abstract Structure Notation One (ASN.1) syntax. Standardised
MIBs have been deﬁned for several types and purposes: Printers, PTP devices, servers,
backbone hardware, etc. For SNMP it is called Structure of Management Information
(SMI) and is standardised in cooperation between IETF and Institute of Electrical and
Electronics Engineers (IEEE). It is designed for use with multiple encoding formats
(XML, JSON, TLV, etc.) and uses a tree data structure. A structure which is possible
to model in all the formats. The top levels of the tree are general categories which
represent organisation or technologies, and for each step down in the tree the nodes
get more speciﬁc. Each level in the tree is represented by an integer, hence every node
has its own Object Identiﬁer (OID) that can be used to locate the desired information.
OIDs are standardised by IETF which ensure interoperability between SNMP enabled
devices. A string of the IDs is per standard long because of legacy design with all the
diﬀerent classiﬁcations. Also, each company can apply for their own OID sub-tree to
deﬁne proprietary device information. The tight data structure allows fast classiﬁcation
for hardware, but hard to read for humans. SNMP is in general good to extract speciﬁc
data points from a device. Each data node in an MIB can be assigned with access
rights to tell whether the value is read-only or read-write.
3.1.2
JSON Remote Procedure Call
JSON RPC is a HTTP(S) based protocol which is an extension of the data modelling
language JSON that enables easy data transfer between two hosts. Its popularity in
web-programming has had its inﬂuence on portability to other programming languages
as there are parsing libraries between JSON and a wide range of programming languages
including: Java, C++, C#/.NET, Python, etc. [16] JSON RPC is not an extensively
described protocol like standards originating from IETF, but rather a tool to commu-
nicate between a host and a server. This allows a degree of freedom to the designer to
use custom commands is only applicable for the application. This means that JSON
12
==PAGE:24==
basically supports all features which a developer can think of. Though, lack of stan-
dardisation requires the developer to write documentation for each custom feature and
possibly write a JSON Scheme to deﬁne the behaviour of RPCs. For example, if invalid
input to an RPC is given.
3.1.3
NETCONF - NETwork CONFiguration protocol & YANG
YANG is a data modelling language used to describe data models of object-oriented
(OO) management approaches as known from Java, C++, etc. It has a tree structure
and has leaf nodes and lists of leaves, and an object speciﬁer called choice that can
contain multiple object types. Data in a leaf is deﬁned using type speciﬁes like uint16
known from programming languages. It can contain both standard data types or objects
which is deﬁned by multiple data types.
Like SMI/MIB, YANG can also describe
value constraints and defaults for leaf nodes. YANG is a collaboration between IETF
and IEEE, and each of the organisations have deﬁned modules and data types for
representing data related to their portfolio. These are continuously being developed
and improved as standards emerge. [22] Modules, standardised or proprietary, can be
imported when designing new modules.
As many features found in YANG has equivalents in programming languages and
as they share the OO style, it is possible to make use of it with most programming
languages used today. The motivation to develop YANG was the creation of Network
Conﬁguration protocol (NETCONF) - a protocol for intended for programmatic con-
ﬁguration of devices, which required a data model which was more modular. Thus, are
all functionalities found in NETCONF deﬁned in YANG.
NETCONFs design excel at conﬁguring devices eﬃciently. It is based on a subset
of XML which provides high readability for humans and simplicity when designing and
reading conﬁguration ﬁles. This also had impact on the development of YANG which
made it easy to read. Readability has a distinct preference as most link speeds are well
above what is required by NETCONF. For common networks it is believed that the
load hereon caused by NETCONF is insigniﬁcant. For example, retrieving 227 data
nodes is limited to <10 KB including authentication on a gigabit link will not have
signiﬁcant impact on other traﬃc, if any. [32] The use of XPath furthermore allows
administrators to conﬁgure or retrieve arbitrary parts of the conﬁguration in a single
transaction.
NETCONF uses SSH which is a widely used protocol as communication channel. It
13
==PAGE:25==
is a TCP based application layer protocol and is thus secured re-transmission in case of
packet loss. A key feature of SSH is the is the cryptographically secured host-to-host
communication channel which e.g. hinders others to eavesdrop or other types of attacks.
There is also a version of NETCONF called RESTCONF describes an implementation
with reduced functionalities leveraging HTTPS. A common feature for the two versions
are the ability to authenticate using certiﬁcates or keys, which imposes a high level of
encryption. NETCONF also bring Access Control Management (ACM) so e.g. only
a subset of actions can be executed by ﬁrst-line support and administrators have full
access.
It also brings locking which allows e.g.
a debugger to prohibit automated
services to change settings while the debugger is connected to the device. [9]
NETCONF do not support discovery of enabled devices. As an alternative, it has
"Call Home".
If the feature is conﬁgured, the device will connect to the speciﬁed
manager right after booting has been completed. The purpose is to register itself as
online and available to the manager. The authentication can use either SSH keys or
username and password credentials, but the server must provide its key to the manager.
NETCONF has two types of notiﬁcations: Base and Event. The Base version is
designed to inform a manager for example if the conﬁguration is changed or a new
SSH session is established to the server. The Event version is a more versatile type of
notiﬁcation, which allows a manager to subscribe to data nodes or subscriptions deﬁned
in YANG. Much like when a <get> operation is performed. Filters can be applied to
the subscription and hence be customised as needed.
NETCONF is designed to use a Remote Procedure Call (RPC) model to communi-
cate where a set of commands is speciﬁed. The combination of NETCONF and YANG
allows developers to implement custom commands. In YANG the RPC is described
with a name of the RPC, the input parameters and the output using leaf and list
elements like for the containers. See for example Conf-D (see section 3.2.2). This is
possible as each party informs the other about its capabilities via a list of namespaces

3.2.1
Proprietary solution
A proprietary solution would require several central functions described in the RFCs
which collectively deﬁne NETCONF. In the following the necessary entities for a com-
plete system will be outlined. For reference see ﬁgure 8.
The software should ﬁrst consist of a NETCONF client, which is a software part
existing on both the manager and the server, which can communicate with another
NETCONF enabled device. This means validating the syntax of incoming packets,
parse the XML and execute behaviour deﬁned in the NETCONF standard. To allow
incoming packets, the client should be able to establish an SSH tunnel - possibly util-
ising the SSH client that is available on the operating system, or otherwise a separate
installation.
As a storage of all the conﬁguration data which is accessible for the client, a database
architecture is needed which must support the features described in the NETCONF
standard: Rollback, which means that it should track changes in the conﬁguration
part and be able to revert to an earlier state; Data validation, which means that it
can validate according to the underlying data model; Multiple datastores, which means
several copies of the database that can be edited independently for speciﬁc purposes;
Among others. The database should be able to parse a YANG model into its own

which each have diﬀerent data models. Moreover, it should have an API to allow other
programs to interact with it. Through this, the NETCONF client on the server side

As the last part, which would be an individual implementation for each device, a
software part which can interact with other software or hardware components available

3.3
Open Network Operating System
ONOS is an SDN controller which emphasises code modularity, conﬁgurability, sepa-
ration of concern and protocol agnosticism. Through these properties, it follows the
described idea of an SDN controller. The disaggregated design of ONOS allows modules
to be added or removed from the source code, without impacting the functionality of
other modules. ONOS also has properties to be dispatched over multiple hosts giving
a higher availability and the opportunity to expand with additional processing power
when required.
The ONOS core has the means to maintain a snapshot of the current state of
the network. It has subsystems which maintains the information and functionalities
for diﬀerent domains related to both the management and data plane, e.g. network
devices, hosts or data ﬂows. The example in ﬁgure 10 shows the path of accessing
a device through a driver that has implemented a behaviour.
Each domain has a
manager that takes care of internal procedures and a service that exposes information
and functionalities to applications via the northbound API. The information is obtained
through the southbound API which permits a provider to register itself to the core using
callback methods and at the same time generating an event in the core to notify e.g.
applications about the new device. Through the callback methods, a provider oﬀers the
means for the core to communicate with hardware. It has support of network protocol
libraries that are necessary for ONOS to communicate with a device a driver which
leverages one or more of the network protocol libraries. A driver represents support for
a speciﬁc device or a family of devices. Through implementation of behaviours, a driver
21
==PAGE:33==
can selectively expose the capabilities of the represented device to ONOS. A behaviour
is a template of talking about a device or interacting with it and are implemented in
the ONOS core.
ONOS as an entirety has two northbound APIs: Java and REST. REST is a concept
based on stateless operations which is in ONOS using the HTTP protocol which has
the necessary operations to transfer JSON formatted information between client and
server. The Java API is implemented in the ONOS core whereas the REST API is
implemented as an application utilising the Java API.

The technologies described up till this point, all comes together because of a wish
from telecommunication companies: To avoid more special purpose hardware for each
new technology and be able to leverage economies of scale in their networks.
It is
realised through the open-source project Central Oﬃce Re-architectured as Datacenters
(CORD) which targets to rethink central oﬃces into data centres. In order to deﬁne
this, it was necessary to change the conception of what is provided to the costumers.
Larry Peterson from ON.Lab described it as seeing CORD like everything as a service
(XaaS). [11] So, one thing that CORD is providing, is connectivity as a service to their
customers and the service is described via a service graph. Hence, multiple diﬀerent
designs based on CORD is available for similar use cases. Examples on this are R-CORD
and M-CORD which stands for residential and mobile. CORD achieves so by refactoring
the control part of legacy hardware components in to new software components. So, for
example Optical Line Termination (OLT) devices are redesigned to be an "I/O blade"
in a server rack and the control and management is performed by a service located
somewhere in the data centre.
The architecture of CORD is a combination of NFV and ONOS and is comprised
of components that are termed diﬀerent:
• Controllers: XOS, ONOS, OpenStack and Docker.
• Services: Virtualised versions of e.g.: EPC, routers, BBU, etc.
• Hardware: E.g. OLT, BBU, etc.
The controllers are responsible for the "cattle" where the cattle are for example network
components and VNFs. They are managing the life cycle of their respective domains.
Services are predeﬁned components which are designed to manage the control plane
of a network.
They are chained in order to achieve end-to-end connectivity.
The
hardware are components which perform tasks which is impossible or ineﬃcient to
perform in general purpose hardware.
Cattle is also a change of the perception of
central components which traditionally has been treated like delicate entities and often
referred to as "kittens". Instead, it must be expected that the components fail eventually
and thus the focus must be how to restore a customer’s connectivity fastest possible.

before a stable release is available. A diﬀerent approach is to use the same structure
as how e.g. an SDN network port is represented in ONOS which is the general way to
implement new features in ONOS.
4.2.1
Extending the ONOS core to support an extra device type
Extending ONOS’ capabilities to support a device like an RRH which only is to be
conﬁgured, requires extension of three elements. First is the classiﬁcation of devices.
It must be possible for applications to distinguish an RRH from other devices not only
by protocol or driver, but also which type of device it is and for this ONOS has an
enum to classify a device. Next step is to describe the device. The device is described
by creating a Java interface, a device description interface, which determines which
methods and constructors the implementing class should have in order to follow the
design. Additionally, this interface must have an implementation in the core that works
as a default implementation. Last step is to add relevant behaviours to ONOS. It is
for example used to gather information about the RRH and specify which information
that must be provided via the mentioned device description interface. But behaviours
are also used to read or write conﬁguration data to a device.
4.2.2
Adding a device driver in ONOS for a proprietary device
While the in-built NETCONF driver is able to establish a connection to any NETCONF
server and has the behaviour to retrieve all conﬁguration data, in order to modify
the conﬁguration on the device dynamically, it is necessary to create a proprietary
driver for ONOS. Dynamically as in the device driver sending conﬁguration changes
either on its own or on the behalf of an application. Furthermore, if device or state
information is to be extracted or a ﬁlter applied to a <get> or <get-config> request.
This must be implemented as behaviours as there is no direct access to the driver from
the northbound API. Device information could be driver-, hardware- and ﬁrmware
version and state information could be the operational status of the device and which
errors that has occurred. Furthermore, if it is desired to control the device via the
CLI during development, in order to communicate with the device, it is possible to
implement behaviours and create CLI bindings to ONOS’ southbound as well as its
27
==PAGE:39==
northbound API. Behaviours implemented in the device driver are ones not relevant for
applications to interfere with.
The device driver is interacting between the provider and the protocol which is
speciﬁed (see ﬁgure 10).
More speciﬁcally, the provider receives requests generated
through the ONOS core. It provides a context in which a behaviour implementation
is executed. Thus, when the device driver is attempting to acquire the NETCONF
session belonging to the context, it has the device id which enables it to locate the
device instance using the device map. The behaviour implementation can then execute
its protocol commands and return information speciﬁed by the behaviour interface.
The device driver uses an XML ﬁle to describe which behaviours it supports and which
classes that implements them.
4.2.3
Device discovery
There is no automatic device discovery available in the NETCONF standard, because
a NETCONF server is not broadcasting details about its presence on the network.
Instead as described earlier, the Netopeer2 server supports the "Call Home" feature
in NETCONF which enables the administrator to instruct the server to register itself
with a managing device and hence providing some type of device discovery. This will
not be implemented in this setup, as support for the feature is yet to be developed
for ONOS. Thus, it is only possible to manually add NETCONF devices to ONOS,
namely the DeviceManager. The DeviceManager is monitoring the availability of the
NETCONF device as long as it is in the registry by checking if it is possible to open a
TCP connection on the port speciﬁed. This is done periodically where it waits a speciﬁc
time interval before timing out. These timeout values can be speciﬁed or be omitted
to use the default values. If the device is attempted added to the device registry again,
ONOS changes details of the information provided related to the device. That be for
example which driver that should be used for communicating with the device. Beware
that a device once added in ONOS is immutable, thus information like version and
serial cannot be altered. A NETCONF device is added to the registry by posting a
JSON scheme through ONOS’ REST interface using a provided Python script. The
JSON ﬁle could look like seen in listing 2.

4.4
Remote Radio Head implementation
The implementation of the RRH comprises of three parts.
One is an RRH YANG
module, which is used to describe the RRH-GW and which RRHs that are connected
to it. As described earlier, there exists no standardised YANG model for RRHs yet, this
model is therefore only an example on elements it can comprise of and thus there is no
integration with any hardware or software components. The second is a YANG model
for the hardware component, L1Oﬄoad. It is a part of the Comcores design of an RRH
as seen in ﬁgure 13. L1Oﬄoad is included as an example on the process of modelling
the conﬁguration registers of a hardware component into a YANG model. Hence the
remaining hardware and software components that are seen in the ﬁgure are out of
scope for this project and it is just repeating the process. The last part is integrating
a hardware component with the on-device database through software, which in turn
utilised the Operating System (OS) to interface with the hardware. More speciﬁcally
this means how Sysrepo communicates with hardware as well as software components.
The platform used at Comcores is a Xilinx FPGA connected to a Linux OS from
which data can be read. During the project, it was attempted to run a Linux distribu-
tion on an Multi Processor System-on-Chip (MPSoC). For this, Xilinx has their own
software called Petalinux to generate all the ﬁles to create a bootable SD card contain-
ing FPGA conﬁguration ﬁles and the OS ﬁles. The most essential part of the ﬁles for
this project is the root ﬁle system of the Linux installation. It was successful to follow
the instructions to generate the ﬁles and boot the OS and thereby modify and verify
the changes in the memory, which is the equivalent process of reading and writing to
a register. Furthermore, it was also possible to access the LAN as well as the internet.
Petalinux has a wide range of libraries which can be installed along with the OS but
unfortunately not all the required libraries for Sysrepo, Netopeer2 and their dependen-

cies are included. Instead it is possible to cross-compile these libraries and import them
to Petalinux, but this is out of the scope of this project. For the implementation, an
approach similar to the on-board OS is to connect a Field Programmable Gate Array
(FPGA) to Linux using a Peripheral Component Interconnect express (PCIe) connec-
tion. This allows the software regarding the implementation to access the memory in
the same manner and hence map the memory into userspace. A hardware intercon-

Setup and conﬁguration of Netopeer2
The Netopeer2 repository contains three software components. A server, a CLI and a
keystore. Onward when mentioning "Netopeer2", it refers to the server implementation
and when not, it will be explicitly stated. Like all the other software repositories used,
Netopeer2 has an installation instruction and furthermore also a separate conﬁguration

the local machine user information. This should not pose a risk to the security of the

on SSH one can also authenticate using SSH public keys. To do so, it requires that the
public key of the remote machine is added to the server as described in the conﬁguration.

authenticating Call Home sessions
which Netopeer2 supports. Furthermore, the YANG data model (ietf-keystore) for
key storage, that the private key under no circumstances can be communicated over
a communication channel while following the YANG implementation. [30] If an SSH
key was to be inserted manually, it would require terminal access to the server and
knowledge about the internals of the setup. As standard, Netopeer2 imports a default
RSA key during installation which should be changed for production environments.
Although keystore is a part of Netopeer2, it is installed separately from the other parts
found in the repository. In the installation instructions provided for the server, it is
described that the keystore must be installed prior to the installation of the server.

4.4.5
Integration Sysrepo with hardware
The integration software is developed in C++ since other similar code projects within
Comcores is written in it and the code will either have to integrate with or can work as
a template for future Sysrepo integration. Although Sysrepo is written in C and thus
it has an API available for C, it also has APIs for other commonly used programming
languages including C++, Java and Python. Access via C++ is achieved by providing
one or more callback classes per YANG module to Sysrepo. The callback class is a
subclass of a Sysrepo class containing a set of virtual methods, which are methods that
can be overridden by the inheriting subclass. Like this, pointers to the code that should
be executed are provided to Sysrepo and it can call the methods when required by the
NETCONF server. Vice versa does it work for the NETCONF notiﬁcations. To trigger
a notiﬁcation from the hardware, a physical button must be pressed. This generates
a PCIe interrupt which the OS registers and the software is notiﬁed. The integration
software is written to ﬁt speciﬁcally with the hardware provided from Comcores and will
therefore need adjustment if changes are made to the PCIe or AXI memory conﬁguration
or if it was to be used with diﬀerent hardware. It should be noted that Sysrepo requires
on of these subscriptions per model to regard it as a part of the running conﬁguration.

and conﬁgures itself using the "start-up" conﬁguration that is installed. The previous
conﬁguration and state data will be lost, if the device has not been conﬁgured with an
additional conﬁguration as a backup. Results are seen in ﬁgure 28 and shows that once
the RRH-GW is rebooted, the information is changed into the state as the "start-up"
conﬁguration deﬁnes (see appendix C).
Figure 28: Results from scenario 3.

The fourth scenario happens when an administrator wants to clear all information
about a device and thus removes it from ONOS. Hereafter the administrator is adding
the device using same device information. When a device is removed, it should no
longer be visible in the devices list in ONOS. When adding it again the device should
reappear in the devices list. When removing the device, it disappears from the devices
list, though when adding the device through the REST API like initially, it does not
reappear.
5.7
Summary
The YANG models were veriﬁed, and the errors reported when adding the –ietf does
not have any inﬂuence on the functionality of the model. They cover missing descrip-
tions on e.g. leaf nodes and namespaces not following the IETF convention. Therefore,
no action was taken upon this result as the models are not to be released. The func-
tionality of the RRH and controller was veriﬁed separately. The full-stack connectivity,
or integration test, between the hardware component and ONOS was realised as seen
in section 5.5. The fourth scenario is not due to details regarding the implemented de-
vice driver, but rather some internal ONOS processes. The remaining listed scenarios
followed the expected behaviour.

Chapter 6
Use cases for fronthaul management
The proof-of-concept delivers management of devices in the fronthaul where especially
the RRH is interesting. In CORD context, the access to data from the hardware opens
up for new applications within CORD. To achieve this, it is necessary to have APIs
between the S-GW and the control application in ONOS (see ﬁgure 29). This would
allow either of the two as the decision-making entity to have detailed information about
both the UE’s and RRH’s radio properties. Hence, enhancing the quality of the decision.
In ﬁgure 29 there is also an antenna type called "small cell". It is an antenna with a
physical coverage which is good for stationary UEs.
In the following subsections, three potential use cases for fronthaul management are
described.
6.1
Energy saving in areas with rush hour traﬃc
A hot topic within the mobile telecommunication industry is energy saving as this is
one of the biggest operational expenses. A scenario could be an area that in rush hour
is heavily crowded and thus a combination of wide ranging and small cells is installed.
Outside of rush hour, only few people remain in the area and thus all the capacity
that is available in the area, is no longer needed. Access to an RRH’s conﬁguration
data allows entities in CORD to change the operational state of it when it is no longer
needed. The RRHs could furthermore be shut down gradually as the traﬃc subsides.
This scenario could also be relevant when looking at a football stadium in a metropo-
lis, where several small cells are placed around and inside of the stadium. When the
match is over, it could be relevant to turn up the emission power of some small cells to
handle the people exiting. Thus, preventing them from overloading the RRHs.
49
==PAGE:61==
Figure 29: Depiction of a scenario where CORD is managing multiple devices in fron-
thaul.
50
==PAGE:62==
6.2
Resource prediction
A topic all mobile providers are concerned about is their customers’ quality of expe-
rience. With measurement data from both the RRH and the UE side, an entity in
CORD could create coverage "pictures" of the network and use them to predict the
in-near future state of the network as the UEs move. In this way, the network can
allocate the resource expected to be utilised and thus be one step ahead of the users.
This could example be relevant in case the cars in ﬁgure 29 follows the road over to
the other RRH. Then, in case the mobile user is served by the cell where the cars are
going, it could be handed over to the small cell while the cars are passing by.
6.3
Automatic error reporting
Maintenance and monitoring of networks are the most important task when operating
a network. NETCONF provides the opportunity for operators to receive errors when a
device is registering certain errors indicating needs for a reboot, errors in the fronthaul
network, etc. It could for example be a switch in the fronthaul that registers a link
which is no longer active, a NETCONF notiﬁcation would be sent to entities which
subscribed to this notiﬁcation. The controller can then try some predeﬁned steps to
resolve the error or notify the relevant personnel.

Chapter 7
Conclusion
The current interest in data centres has evolved ideas to utilise them for divers ap-
plications.
For telcos, they are employed instead of hardware in the core network.
The thought of large savings, easy deployment and easy management on an extensible
platform is appealing.
Today a mobile network requires specialised hardware which requires expertise and
extensive manuals to operate. Equipment is expensive to buy, and personnel is either
expensive to train or hard to get.
Therefore, an additional level of abstraction is
relevant. Telcos have CORD which is still under development and thus dependent on
the existing solution for now. Since CORD is an open-source project, the source code
is available to the public and in the git repositories there is a high level of activity at
the time of writing. Although the objective of CORD is to have a deployable solution
available to all telcos to use for ﬁeld trials, the development of M-CORD services is still
in the making. [29] E.g. one part of CORD is ONOS which is currently being hardened
in cooperation with Ciena’s Blue Planet division.
The currently used, but declining network management protocol SNMP is shown
to no longer be satisfactory. It is gradually being replaced with the more recent devel-
oped protocol NETCONF which is more contemporary. It promotes extensibility and
readability before simplicity and is being adopted by the big vendors.
To incorporate NETCONF in this project, it was chosen to use open-source soft-
ware. It showed good opportunities for integrating with other components and oﬀers
transparency in the implemented solution.
The controller component was implemented as a control application, a device driver
and a minor extension of ONOS’ core functionality. Work performed in relevant univer-
sity courses and existing components in ONOS served as templates and inspiration to
the design of the controller related software. The implemented controller was veriﬁed
through interfacing with an example VM which had the suggested software package
installed and support of the developed YANG models.
The RRH component was implemented as a package of several open-source or pro-
52
==PAGE:64==
prietary software subcomponents containing a database to store conﬁguration, a server
for NETCONF support and a subcomponent for integration with a hardware com-
ponent provided by Comcores. M-CORD does not yet deﬁne an approach to YANG
models and thus it is not certain how RRHs will be represented. Instead, a simple
model for an RRH was modelled along with one for the provided hardware component.
The implemented RRH component was veriﬁed by connecting the Netopeer2 CLI to a
host computer which was connected to an FPGA board through PCIe.
As the last step, an integration test was conducted. The RRH was connected to the
controller and managed sending commands through the ONOS REST API extended
by the control application. The described tests all showed satisfactory results which
followed the expectations and fulﬁlled the speciﬁed goals.
7.1
Future Work
The implementation of the SDN controller is only a proof-of-concept, thus several in-
teresting enhancement opportunities exists. However, before developing the presented
implementations further, an important consideration regarding the YANG and NET-
CONF support in ONOS should be investigated ﬁrst. ONOS’ core is currently being
expanded with a module called Dynamic Conﬁguration which combines YANG and
NETCONF. This enables ONOS to automatically extract data from devices if the
YANG models exist in ONOS’ list of models. This brings up the question whether
to pursue an implementation of Dynamic Conﬁguration into the design or continue re-
thinking ONOS into a mobile network context leveraging most possible from the current
design. Both choices would inﬂuence the continuing implementation approach. Below
is described a list of possible enhancements to the proof-of-concept:
• Enhance RRH YANG model and develop YANG models for the remaining hard-
ware and software components depicted on ﬁgure 13.
• If the OS is to run on the hardware, work must be done in developing an image
that includes the dependencies for running Sysrepo and Netopeer2.
• Providing assistance to ON.Lab to implement NETCONF Call Home support in
ONOS to introduce an additional level of automation in the mobile network.
• Packaging of software components for easy installation, removal and update.
• Develop a decision-making entity to control RRHs as mentioned in chapter 6.
