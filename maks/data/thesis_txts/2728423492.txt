Abstract
An important parameter used for investigating changes to the sea level globally is the
Sea Level Anomaly (SLA). Several public providers supply gridded SLA data prod-
ucts, interpolated using satellite-based altimeter data. However, in order to cover
data gaps and mitigate data errors, these products are created using large temporal
windows during interpolation, as well as spatial- and temporal filtering. This project
has multiple purposes. First, to investigate the feasibility of reducing the size of the
temporal window during interpolation of the gridded SLA data product, as well as
not applying any filters throughout the process. Furthermore, additional variables
Sea Surface Temperature (SST), Significant Wave Height (SWH) and Wind Speed
are included in the gridded product. More high-frequency features, as well as some
noise and satellite tracks, are found in the data product. Overall, the products need
additional work to remove unwanted signals, however, results are promising for an
SLA grid product containing more well-resolved features in time. Next, an autoen-
coder is applied to the data after gridding, in an attempt to create a smart filter for
removing satellite tracks and non-physical noise. Preliminary results are propitious,
as satellite tracks are successfully removed from the data without influencing the
underlying features. Finally, the data are used to forecast near-future values of the
SLA grid using an Encoder-Decoder Convolutional Long Short-Term Memory with
Attention (EDAConvLSTM) architecture. The prediction power of the model is very
dependent on the smoothness of the data. Forecasts using the Copernicus Marine En-
vironment Monitoring Service (CMEMS) SLA data set are reliable for about 30-40
days, while the reliability is reduced to 7-10 days when using the SLA data product
produced in this project. Thus, for CMEMS the product could be extended to almost
real-time using the forecast. For the SLA grid product produced in this report, how-
ever, the forecast may be used to extend the grid product slightly beyond real-time,
such that a few future days are included.

CHAPTER 1
Introduction
An important tool for monitoring changes in both global and regional climate is
measurements of the sea level. The introduction of satellite altimetry has massively
improved the amount of data available. Public entities, such as National Aeronautics
and Space Administration (NASA) and European Space Agency (ESA) provide grid
interpolated SLA data products [2, 3] based on satellite altimetry data. These prod-
ucts are provided with both daily and monthly resolutions. However, the temporal
window in the daily products is very extensive.
The Making Earth System Data
Records for Use in Research Environments (MEaSUREs) data product contains un-
wanted signals, and the way to remove these in the CMEMS product is by overly
smoothing the data. The purpose of the first part of this project is to provide a grid
interpolated SLA data product with a daily resolution using a smaller temporal win-
dow. In the second part of the project, efforts are made to remove unwanted signals
from this product, using an alternative method to smoothing, after which it is used for
forecasting near-future changes in the SLA using Artificial Neural Network (ANN)s.
As implied, this project is made in two parts. In the first part, the grid interpolated
SLA data product is produced and presented. The spatial resolution of the daily
product presented in Part I is 1‚ó¶√ó1‚ó¶, which spans the years 1993-2023. Additionally,
the SST, SWH and Wind Speed are grid interpolated in the same period and reso-
lution. The data originate from Radar Altimeter Database System (RADS), but are
processed further before being grid interpolated and presented as a global, daily prod-
uct. Moreover, data over the polar regions which have been processed further, are
utilized in the interpolation for increased information in these areas. The potential
issue with similar products is that in order to cover data gaps and ensure suÔ¨Äicient
coverage at all times, a large temporal window is used for interpolating each grid.
ForCMEMS and MEaSUREs, this window is 6 weeks and 30 days respectively. Thus,
the data product is always delayed by at least the size of the temporal window when
published. In Part I, a temporal window of just 5 days is used in an attempt to
reduce the influence of temporally distant data in the product, as well as improve the
recency of data when published. The current product from CMEMS utilizes heavy
filtering throughout the pipeline, in order to create both spatially and temporally
smooth data. In this project no filtering is applied, and the product solely relies
on data corrections. However, a novel method of removing unwanted signals from a
gridded SLA data product is introduced in the second part of this project.
==PAGE:16==
2
Introduction
The second part of the project is dedicated to two main purposes. The first purpose is
to investigate the use of an autoencoder for removing unwanted signals from the grid
interpolated SLA data product created in Part I. The second purpose is to forecast
the SLA grid on shorter timescales of up to 90 days.
For the first purpose, the global grid from Part I is used. The pipeline allows for
creating grids of arbitrarily high resolution, so an SLA grid product in a local Area
Of Interest (AOI) near South Africa has been created in a 1/6‚ó¶√ó 1/6‚ó¶resolution for
testing the autoencoder in a higher resolution as well.
For the second purpose, the global SLA and South African data products have been
used. Data from CMEMS are used to test the impact of smoothness in the data
product on forecasting. The CMEMS data are restricted to the same AOI near South
Africa, as well as another less volatile area in the South China Sea. Thus, the influ-
ence of volatility on the prediction power of the models may be investigated. Using
both the data products created using the pipeline from Part I and CMEMS data
allows for extended scrutiny and comparison of data quality. The forecasting ANN
used in Part II has an EDAConvLSTM architecture. The results are compared to a
baseline as well as results from similar work.

CHAPTER 2
Data origin
The following is a description of the sources of data for the grid interpolation, as well
as a brief explanation of the SLA.
2.1
Sea Level Anomaly
The SLA is the height of the sea surface relative to a long-term average. In RADS,
the DTU15 Mean Sea Surface (MSS) is used as the long-term mean of the sea surface
[4]. For this project, all SLA data have been corrected to use the DTU21 MSS as the
reference sea level [5]. The DTU21 MSS is computed over the 1993-2012 period using
multiple satellites.
2.2
Radar Altimeter Database System
The main source of data for the grid interpolation is RADS [6], made available through
the Maxwell and Kepler clusters at the National Space Institute at the Technical
University of Denmark (DTU).
2.2.1
Satellite coverage
This database contains satellite altimetry data from 1985 until today, although be-
tween the end of Geosat in September 1989 and the start of ERS-1 in August 1991
there is a gap in the data. Therefore, all data before the initialization of ERS-1 are
excluded. Table A.2 in appendix A.2 shows the altimetry satellites in RADS. This
table is also colour-coded to indicate which satellites are included in the grid inter-
polation. Light blue indicates the satellites which are included, while pink indicates
those which are not. The satellites included in the grid interpolation provide a con-
tinuous flow of data from August 1991 until today for the grid interpolation, mostly
with multiple satellites at the same time. Some days, however, do not have coverage
of a single satellite and are therefore missing from the data set.

2.2.2
Derivative products in RADS
In RADS, data with different levels of pre-processing may be extracted. Anything
from the raw altimeter readings to heavily processed products, such as the SLA,
may be found in the database. All parameters extracted for grid interpolation in this
project are products derived from the raw altimeter measurements, using models such
as atmospheric and global mean sea level models. The main derivative product used
in the gridding, as well as the forecasting in the second part of this project, is the SLA.
Other derivative products included in the grid interpolation are the SST, SWH and
Wind Speed, where the SWH is the mean wave height of the highest one-third of the
ocean waves. Details of the derivation of these parameters for the RADS database
may be found in the documentation [6].
2.3
Processed polar data
Usually, satellite altimetry data do not reach the polar regions, and the sparse data
that do exist are often prone to error. A few satellites do venture into the very high
and very low latitudes however. In order to increase the available data in the polar
regions, a researcher at DTU Space has processed some of these data in the Arctic
and Antarctic regions using Synthetic Aperture Radar (SAR) data and SAR Inter-
ferometry [7, 8]. The result is a data set several orders of magnitude larger than the
raw data alone. Unfortunately, only the SLA parameter has been included in this
computation, meaning the extent of the remaining parameters in the grid interpola-
tion cannot be extended reliably into the polar regions. Figure A.3 in section A.1
shows an example month of CryoSat data coverage from the Arctic and Antarctic,
both including and excluding the polar data. The light blue, purple and green colours
indicate which processing method/instrument has been used, while the pink data near
the boundaries indicate data from RADS. As the figure shows, the data coverage is
increased significantly in the polar regions. CryoSat has by far the best coverage of
the three polar satellites used.
The polar data include data from the satellites ERS-2, Envisat and CryoSat in chrono-
logical order. The data have been processed individually by satellite, and are thus
separated into 3 individual data sets. ERS-2 was launched in 1995, and thus no addi-
tional polar data are included before this point. The polar data sets from ERS-2 and
Envisat have been produced using a different reference frame than CryoSat, which
has used the same reference frame used in RADS. Therefore, ERS-2 and Envisat
polar data have been corrected to fit the reference frame used in the other data.

3.4
Ball tree
Ball trees are a space-partitioning data structures, which makes it faster to find
multidimensional data than using brute force [11]. A ball tree is a binary tree where
each node has an associated ball. The ball is constructed such that it is as small as
possible, while still containing the child nodes in the binary tree. The binary tree is
constructed such that the data set starts by being inside one ball. Then, the ball/data
set is split into two. These two balls are then the balls of the two child nodes. In the
na√Įve approach, this process keeps on going until only one data point is within each
ball. This means when performing a query for finding the closest point to a query
point the tree just needs to be traversed.
One implementation of such a tree is the ball tree implementation done by SciKit-
learn [12]. This implementation has been used during this project to find all points
within some distance from a query point. The distance metric used by the ball tree is
the Haversine distance metric (see Equation 3.1). The ball tree was used extensively
to find the N points used for the RBF interpolation method.

4.1.1
Computational framework
Processing of the RADS data has been executed on CPU‚Äôs in the Maxwell and Kepler
clusters at DTU Space.
These are old and not suited for heavy computing, but
suÔ¨Äiced for the relatively light processing steps necessary for preparing the data for

Initial testing and implementation of early versions of the grid interpolation were

has been utilized [13]. Specifically, several Intel Xeon Gold 6226R (16 core, 2.90GHz)
CPU‚Äôs have been used in parallel for a tremendous speed-up in processing time.
4.1.2
Programming framework

The first step is extracting the data described in section 2.2 from the database in

Since the polar data has a much higher sampling frequency it has been downsampled
to 1Hz, which is approximately the same sampling frequency the RADS data have.
The polar data use a different reference system, and thus a correction has to be
applied. The correction consists of first computing the mean of a designated area
from the original satellite data and a mean, in the same area, of the polar data for
each satellite. Then, the difference has been subtracted from the polar data, resulting
in a common reference frame. After applying the reference frame correction on the
polar data, these are inserted to directly continue the RADS data for satellites ERS-2,
Envisat and Cryosat.
As the data are slightly offset between the satellites, another correction is necessary.
For this, the TOPEX and Jason series satellites are assumed stable and used as
reference satellites. The correction is then made by finding the points where satellites
paths cross each other, calculating the offset between the start and end points for each
satellite, and realigning the other satellites to match the references. The crossover
correction has only been applied to the SLA feature, as the other features do not
require correction. In overlapping years, later satellites are corrected to match the
earlier ones. For example, Jason-1 data are corrected to match TOPEX data. After

5.2.2
Interpolation method
Here, the gridding pipeline described in section 4.3 is reviewed, discussed and com-
pared to methods used in CMEMS and MEaSUREs.
5.2.2.1
Block averaging
The reasoning behind block averaging of the data is to increase stability and reduce
processing time. Tracks may contain very different values in close proximity both
spatially as well as temporally. As the interpolation provides an exact fit in data
points, this may lead to high gradients in the output grid should the original points
be used without block averaging. The grid resolution of 1/6‚ó¶√ó1/6‚ó¶√ó3 h in the block
averaging has been chosen, as this resolution should at least be higher than that of
the grid interpolation, but not large enough that averaging becomes irrelevant due
to a too-small number of data points within each block. When the grid resolution
approaches zero, no averaging is performed. If the resolution of the grid interpolation
is increased, so should the resolution of the block average.
For the CMEMS data, along-track subsampling to half the original number of data
points is used [3]. Before gridding the MEaSUREs data, it has been filtered and fun-
damental computing steps have been reevaluated after extraction from the database
[19]. These steps contribute in some regard to achieving the same stability and speed
optimizations of the block average implemented in this project.
5.2.2.2
RBF interpolator
Initially, several interpolation methods were considered.
The obvious choice for
geospatial interpolation is Kriging, but the method proved too slow and time-consuming
to be feasible in this project. Using RBF interpolation is much faster than Kriging

5.2.3.1
Grid resolution
The processing time increases quadratically with increased grid resolution. Due to
time constraints, the interpolation grid resolution has therefore been set to 1‚ó¶√ó 1‚ó¶
for the global product. Large features are resolved fairly well at this resolution, but
smaller features are not necessarily.
Ideally, the resolution of the grid should be
higher, i.e. 1/4‚ó¶√ó 1/4‚ó¶like CMEMS or 1/6‚ó¶√ó 1/6‚ó¶like MEaSUREs, which have
been downsampled for comparison with the data product produced in this project.
5.2.3.2
Temporal interpolation window
The NoD used for the temporal interpolation window in this project was determined
based on when the data had global coverage with only 2 active satellites. The reason
for choosing that 2 satellites should have global coverage, is that in the 1990s only
TOPEX and ERS-1/ERS-2 conveyed measurements, which is the minimal number of
active satellites in the entire period the data product spans. Figure A.4 shows the
data after the RADS data processing using 4, 5 and 6 NoD (9, 11 and 13 days in
total). This figure shows global coverage is obtained when using 5 NoD. Furthermore,
since the reference satellites have cycles of 10 days (See Table A.1 in section A.1),
better global coverage can not be obtained than when using more than 5 NoD, as the
passes are repeated after each cycle.
Comparing the NoD chosen in this project with CMEMS and MEaSUREs, far fewer
have been used in this project.
The CMEMS data product uses 42 days while
MEaSUREs uses 30 days. In this project, the much smaller window is used, as a
large NoD in the temporal window contributes to over-interpolation and therefore
smoothing, especially if the time-to-distance weight is low. Moreover, a small NoD
in the window will result in a shorter delay between observing a data point and pub-
lication of the grid for that day, as the latency is equal to the NoD plus one. Thus,
the data product of this project has a publication delay of only 6 days.
5.2.3.3
Number of data points
As explained in subsection 3.3.2, the memory required increases quadratically when
the number of data points included in the interpolation of each grid point is increased.
However, a lower number of points could decrease the overall quality of the final
data product. Thus, a higher number of points is desired. This number should not

necessarily become too high at all locations, however, as some geographical areas
may affect others in an unexpected or unwanted way. Deciding on a set number of
nearby points to use is, therefore, a complex matter. An alternative, which has been
explored briefly, is to use all points within a set radius around the current point. This
proved infeasible, however, as the number explodes near the polar regions where the
point density is much higher. Random sampling has also been attempted when the
number of points in a set radius rose above some set number. This yielded unexpected
behaviour, however, with non-physical noise appearing in areas near the polar regions.
The final decision was to use the nearest 1000 points within a radius of 1000 km, as
the behaviour was as expected and the processing time was within the bounds of the
project.

The time-to-distance weighting constant used to produce the data product, presented
in section 5.1, were chosen based on experimentation of the weighting value and
observations based on the data set. This resulted in two different values of time-
to-distance weighting.
One for the SLA and one for the other features.
For the
SLA a time-to-distance weighting of 0.028125 were chosen. This was due to the SLA
changing slowly over time and thus a small weighting can be used to diminish the
importance of the time feature. For the SWH and the wind speed, a time-to-distance
weighting of 11.25 were chosen. The high value was chosen, as these features change
much faster in time. Therefore, the importance of time should be greater than that
of spatial distance. For the temperature feature, the choice of the time-to-distance
weighting parameter does not change the resulting grid significantly, at least in the
span 0.01-15. Thus, for optimization purposes, the temperature is interpolated with
the same parameters as the SWH and the wind speed. The interpolation can then
be computed simultaneously for these three features.

CHAPTER 6
Motivation
In Part I, a daily grid product containing the global SLA, SST, SWH and Wind Speed
in a 1‚ó¶√ó 1‚ó¶spatial resolution spanning the years 1993-2023 has been presented. In
the second part of the project, the SLA from this product is used in an attempt
to forecast near-future values. Furthermore, efforts are made to remove unwanted
signals from the data product.
Investigations of the grids produced in Part I show visible satellite tracks in the grids
as well as non-physical noise. In Part II, methods for removing satellite tracks and
other noise sources are investigated to produce a less noisy data product, which still
includes physical high-frequency changes in the SLA. Noise in other data products,
such as the CMEMS and MEaSUREs, is mitigated by the use of heavy filtering both
before and after the gridding process. These filters have a tendency to over-smooth
the physical features when noise such as satellite tracks are removed.
Thus, the
techniques explored in Part II aim to only remove the noise and not smooth the
actual data. For this purpose, autoencoders are tested as a means to remove the
non-physical noise from the SLA data products created using the pipeline in Part I.
Several regional forecasts exist, i.e. the northern and equatorial Pacific [22], the South
China Sea [23] or the Caspian sea [24]. Likewise, forecasts have been produced both
globally [25] and locally using tide gauge data, such as in the North Sea [26] or the
Grand Isle, Louisiana [27]. Some services exist to provide SLA forecasts, however,
these either only show the trend locally at tide gauge positions [28], or only provide
monthly forecasts for the Pacific [29]. The last model utilizes an ensemble method
of different models using tide gauge data. The resulting forecast is, however, very
smooth and only shows the general movements, not the high-frequency changes.
To the authors‚Äô knowledge, forecasting SLA globally using satellite altimeter data
has not been attempted successfully. Even the local predictions are prone to error
at relatively short time scales, when satellite altimeter data are used. The forecasts
made using tide gauges are usually more accurate, however spatially restricted to
the position of the tide gauges as interpolation is rarely used for these. The second
part of this project is therefore dedicated to utilizing the SLA data product from
Part I for a daily forecast on both local and global grids up to 90 days into the future.
For comparing the forecasting power of the methods used in the second part of this

The EDAConvLSTM architecture is an expansion of the Encoder-Decoder ConvLSTM
architecture, since this architecture adds the Attention block described in section 9.3
to the previous architecture and uses BiConvLSTM blocks as an encoder instead of
ConvLSTM blocks.
The attention block is used to keep the information from the input images through-
out the decoder. The input to the attention block (source tokens) are the output
images at each time step produced by the BiConvLSTM blocks. The query used in
the attention block is the hidden state at each time step in the decoder. The output
==PAGE:65==
10.3 Autoencoder
51
of the attention block is then used together with the output of the input network in
the decoder as an input to each ConvLSTM block. Furthermore, the output of the
attention, the output of the input network, and the output of the ConvLSTM block
are combined and used by the output network. The output network then makes a
prediction, which is used as the next input in the input network.

(a) Encoder
(b) Decoder
Figure 10.3. Encoder and decoder architectures of the autoencoder.
The autoencoder is a network which maps the input to a latent space Rn ‚ąí‚ÜíRp(encoder)
and reconstructs the input based on the latent space Rp ‚ąí‚ÜíRn(decoder) [41]. In this
project, the autoencoder has two main purposes. The first purpose is to reconstruct
the images with less noise and the second is to speed up other forecasting models,
which can then operate in the latent space of the autoencoder. To fulfill the first
purpose, the latent space has to be small enough that the noise will not be possible
for the decoder to reconstruct, while still being large enough that the desired features
will be reconstructed. To speed up the forecasting models using the autoencoder, the
autoencoder has to reconstruct the input as well as possible, while still having as
==PAGE:66==
52
10 Neural network architectures
small latent space as possible. Otherwise, the forecasting model could try to fit the
flaws in the decoder, since the loss is computed after the decoder is used. When using
an autoencoder, the network should be trained separately and then be inserted into
another architecture.
The autoencoder used in this project has been implemented as shown in Figure 10.3.
The encoder in Figure 10.3a takes the input data, which yields an encoded result.
The result is then passed to the decoder, resulting in the autoencoded data. The
autoencoder shown in Figure 10.3 uses the basic blocks defined in section 9.4. The
first BaseEncoder block is used to change the number of channels from the initial
number of channels to 64. N-Downsamples in Figure 10.3 is the number of times
the input should be downsampled. For each downsampling the (x, y) size is halved
and the number of channels is doubled. For the decoder, the opposite is true, i.e.
the (x, y) size is doubled and the number of channels is halved. The last Conv2d is
used for decreasing the 64 channels down to the initial amount of channels. Since
the autoencoder uses tanh as the activation function (see Table 8.1), the output is
restricted to the interval [‚ąí1, 1], which means the input is also required to be in this
range. Thus, in order to use this autoencoder either all the data have to be within
these limits or the data must be feature scaled (see section 8.2).

E2D2AConvLSTM architecture consists of an autoencoder, which is wrapped around
the EDAConvLSTM architecture. The encoder and decoder in E2D2AConvLSTM
consists of the autoencoder described in section 10.3. In the E2D2AConvLSTM ar-
chitecture, the autoencoder is used to make the image size smaller in the latent space.
Furthermore, the autoencoder increases the image channels in the latent space to
make sure the reconstruction has as few errors as possible. Since the image size is
smaller, the number of weights needed in the ConvLSTM with Attention decreases,
which results in a faster training time and lower memory usage. The decoder then
takes the output of the EDAConvLSTM and reconstructs it into the predictions.
In the optimal case, since the autoencoder should be able to reproduce the input
(x = Decoder(Encoder(x))), no information is lost in the encoding and decoding.
Thus, the information available as input for the EDAConvLSTM should be the same
as if the autoencoder was not used. In the case where the autoencoder is too small,
however, information may be lost and the prediction power of the E2D2AConvLSTM
model may diminish.

CHAPTER 12
Discussion
The following chapter contains a complete review and discussion of the methodology
and results throughout this project, including both the grid product produced in
Part I, and the autoencoder and forecasting results in Part II.
12.1
Smoothness of the grid
During initial testing of the time-to-distance weighting in Part I, only the 1‚ó¶√ó 1‚ó¶
global grid was used.
Naturally, the spatial grid resolution determines how well
physical features and noise are represented in the grid. For the global grid, the noise
levels from satellite tracks were not visible at the selected time-to-distance weight of
0.028125 for the SLA data product. When the 1/6‚ó¶√ó 1/6‚ó¶South Africa product
was introduced in Part II for investigation of the impact on the forecasting power,
however, the importance of carefully selecting the time-to-distance weight was made
clear. Using a low weight, such as the one in the global grid of 0.028125 would result in
temporally continuous features. However, as data further away in time influences the
gridding process more, satellite tracks become more visible and the spatial features
therefore become less smooth. On the contrary, a higher value of the weight results in
the gridding process being influenced less by data further away in time. This results
in spatially smooth features with less visible tracks. Here, however, the temporal
continuity of the grid is poor and the variation between each day is much larger,
such that the general underlying motion is less visible. Therefore, without further
filtering the time-to-distance weighting is effectively choosing between smooth spatial
features, with a less visible underlying motion between days, and temporal continuity
with more visible satellite tracks. The time-to-distance weight in the high resolution
AOI near South Africa of 0.28125 was deemed an appropriate value at the time of
re-gridding. Further experimentation of using a lower weight to increase temporal
continutity, and then removing unwanted signals with the autoencoder could prove
beneficial.
12.1.1
Number of satellites in orbit
The trade-off between spatial and temporal continuity may be amended by increasing
the number of altimetry satellites orbiting the Earth at all times, such that the
==PAGE:100==
86
Discussion
density of the data available to produce a grid increases. The impact of the number
of satellites in orbit at one time, and the resulting grid product at that time has
proven significant. Especially before February 2002, when the maximum number of
satellites orbiting simultaneously was predominantly 2 and at most 3 (see Table A.2),
and the grid quality was very poor.
With the exception of 2014 and 2015, there
have been at least 4 satellites orbiting simultaneously since 2008.
The fact that
there were only three satellites operating simultaneously in 2014 and 2015 had an
impact on the train-validation-test split for the forecasting model. In early versions
of the forecasting pipelines, the validation data split started in 2014, rather than
2016. Including these in the validation set, rather than extending the training set
until 2016 decreased the forecasting power of the model during testing beyond what
was expected from just a decrease in the temporal length of the training data. Thus,
the training split was extended in an attempt to feed the models more information
and learn more general patterns. In general, the number of satellites in orbit has
increased in later years, meaning that the quality of future data and the resulting
grid products should improve. This is especially true if the trend continues, such
that the amount of simultaneously orbiting altimetry satellites remains high or even
increases.
12.1.2
Noise reduction in the grid
For other products such as CMEMS and MEaSUREs, several methods have been
used to reduce noise in the data. Most prominently, the unwanted signals consist
of data errors resulting in e.g. visible satellite tracks or non-physical high-frequency
noise. For removing noise, a large temporal window for gridding, along-track noise
removal, Gaussian smoothing and low pass filtering in the grid etc. were applied.
For the data product produced in Part I, none of these methods were utilized as the
goal has been to create a SLA grid data product as close to real-time as possible
while maintaining not only the low-frequency temporal movement but also physical
phenomena with higher frequencies. Although challenging, with high quantities of
noise in the SLA grid product, a method for improving the data product may have
been found in Part II. Initial results show that it may be possible to remove satellite
tracks, as well as possibly other high-frequency noise sources from the grid product by
using an autoencoder. This may function as an alternative or complementary method
to the other filters used on the grid. Further experimentation with the architecture of
the autoencoder is necessary, however, as the investigations performed in this project
have only explored a limited amount of configurations.

12.1.4
Pre-gridding processing and corrections
The SLA data extracted from the RADS database is a derivative product made using
several models, as briefly described in subsection 2.2.2. Some of these models are
sensitive and prone to errors ranging from the order of centimetres to meters, e.g.
the ionospheric, dry and wet tropospheric corrections. Optimizing these models, or
applying corrections post-creation to the SLA would improve the quality of the grid
product tremendously. Likely, thoroughly examining the data from RADS could be
advantageous in enhancing the grid product‚Äôs quality. Many errors seen in the grid are
a direct result of faulty data, resulting in very visible satellite tracks of completely
different values than those surrounding them.
Further scrutiny of the data from
RADS may assist in alleviating this issue.
In the TOPEX-Jason series used as reference, the satellites continue to function until
the end of a mission, rather than being switched at the start of a new mission. For
example, the Jason-1 satellite is used as a reference from the end of the TOPEX
mission on 2005-10-04 until the end of the Jason-1 mission on 2013-06-18. The old
satellite is moved to a different track when a new mission is launched, and the new
mission continues on the track of the old mission. Thus, for a continuous reference,
the reference satellite should be changed at the start of each new mission in the series
instead of at the end. Especially in the period of transition between TOPEX and
Jason-1, this change to the processing chain may alleviate some errors, due to the
especially high noise level in the TOPEX period. This may even extend the years of

12.2
Autoencoder
For the autoencoder, the main topic of discussion is the impact of parameter choices
made throughout section 11.1.
12.2.1
Impact of parameter choices
The amount of information retained by the model depends directly on the number of
channels and thereby the size of the latent space. For the global autoencoder models,
it is evident that 32 channels do not constitute a suÔ¨Äiciently large model space in
the current version of the autoencoder. This is most likely due to a combination
of complex data and the choice of model architecture. The architecture is mainly
discussed in subsection 12.4.1.
The 128-channel model generally reconstructs the
data well, apart from the areas with Eddy currents. This suggests that the model is
capable of properly learning the intricacies of the data, with a large enough latent
space.
For the South African AOI, it is apparent that far fewer channels are necessary in
order to reconstruct the data. Due to the limited size of the area, the autoencoder
is required to learn fewer underlying processes, which in turn are resolved better due
to the high spatial resolution. With the increased resolution, data errors also become
visible as the model is unable to recreate the apparently random features. This is
especially true for the satellite tracks, which are visible in all 3 models.
At this
resolution, the autoencoder may therefore be used as a tool for finding and removing
noise such as tracks from the grid. It is apparent in subsection 11.1.2, that the ability
to filter noise depends on the size of the latent space. However, the model is not
capable of separating physical features from data errors. Thus, just as when using
other types of filters, such as Gaussian filters, there is a trade-off between how much
noise is kept and how many desired features are removed using the autoencoder.

12.3.1
Assumptions
Before the SLA data was used in any of the forecasting models, the linear and first-
order Fourier fits were subtracted. For this purpose, a linear system of equations
had to be solved. The assumptions used in this project for solving the linear system
of equations are Gaussian distributed residuals and i.i.d data. Further analysis has
shown that this assumption was violated for some of the time series, as the residuals
were not Gaussian distributed. This is however not an issue, as the fit is subtracted
from the data to make forecasting easier and then added again to achieve the correct
SLA forecast.
In chapter 7, data before 2006 were discarded based on a weighted global mean
times series of the global data set produced in the chapter 4. This assumes that
the distribution for each region is the same as the weighted global mean times series
distribution. The assumption has not been investigated further, as the amount of
data in this period was deemed suÔ¨Äicient for training a forecasting model.
12.3.2
Data set dependency
Based on the forecasting results, one could get the impression the forecasting model
produced on the South African CMEMS data vastly outperforms the model produced
using this projects 1/6‚ó¶√ó 1/6‚ó¶resolution data set in South Africa. However, the
RMSE‚Äôs for the two data sets are not comparable since the magnitude of the SLA in
CMEMS is lower compared to the data product produced in this project (the mean
difference is 2 cm). Thus, the RMSE will be lower for the CMEMS data set compared
to this project‚Äôs data set. Furthermore, the volatility levels of the two data sets are
very different, as discussed in subsection 12.1.3. Therefore, the overall performance
of the models is also different. The CMEMS model has to fit much less variation in
the data, while the model using this project‚Äôs data is more prone to overfitting to the
volatility. This can also be seen in the example forecasts in section 11.2 where the
CMEMS data set forecast changes over time, while for this project‚Äôs data set its almost
constant. This happens because the model has not learned movement in SLA before
overfitting occurs. Thus, the apparent model performance is highly correlated with
the smoothness of the data. The phenomenon is also seen clearly when comparing
the forecasting model between the South China Sea and the South African area. The
South China Sea CMEMS model has more prediction power compared to the South
African CMEMS model, which is likely due to the lower variability in the South China
Sea and the lower magnitude of the SLA.

Based on the forecasting results in section 11.2, the prediction power is mostly seen
in the RMSE as a function of predicted days figures. The investigated models dis-
play that lower variability regions have a high er prediction power, while the regions

with higher variability have lower prediction power. The South China Sea model

before and after the gridded day is used. Thus, a forecast of 42 days means whenever
a new grid is published a forecast from the last gridded day until the published day
can be made. For this project‚Äôs data set in the South African area, the model can
only forecast dependably 7-10 days into the future. In this case, it is likely that the
forecast is only valid for the first week because the model overfits much earlier than
for the CMEMS data set, due to the higher variability. The 7-10 days forecast is also
the estimate for this project‚Äôs global data model forecast. Here, however, the model
forecasts in the higher volatile areas are only reliable 3-5 days into the future. This
either indicates that the model needs more information or that a different approach
is needed. One way of adding additional information is by using global grids in higher
resolution. For all grids produces in this project a moving window of 5 days on either
side of the gridded day is used to produce each grid. This means a reliable forecast
longer than 5 days is interesting, since then the forecast goes beyond the publishing
day of the product.
In [23], the forecasts in the South China Sea were only conducted for the next 21
days, compared to the 90 days in this project. Therefore, the comparison can only
be made for these. The CMEMS data product has been used for all models in their
work, allowing a direct comparison of RMSE. The average RMSE at predicted day 1,
for the best model in their work is approximately 3 mm. For the forecasting model
presented in subsection 11.2.2, the RMSE for the test set on the first day is approx-
imately 5 mm. Thus, their model outperforms the one in this project for the very
early predictions. On the 21st day, the RMSE for their best model is 5 cm, compared
to 4.5 cm in this project. The forecast in this project outperforms their model from
the 8th predicted day and onwards, with a very small discrepancy in the order of mm
before this.

the models struggle with predicting rapid change in the SLA. Furthermore, when
comparing the global spatial RMSE to the global current system, it is observed that
not all the current systems are visible in the spatial RMSE figure. However, those

12.3.6
Feasibility of forecasting on a global scale
Using the EDAConvLSTM model to forecast the 1‚ó¶√ó 1‚ó¶global SLA data set has

The 1/6‚ó¶√ó 1/6‚ó¶resolution SLA grid from the AOI

data set may improve the model by adding information about physical processes
surrounding the highly volatile areas. However, the model may be overwhelmed by
the amount of additional information in the data.

Architecture
Here, the architecture choices are discussed for both the autoencoder and the fore-
casting models.
12.4.1
Autoencoder
The autoencoder architecture introduced in section 10.3 produces reasonable recon-
structions of the data sets made in this project as shown in section 11.1. Although
==PAGE:106==
92
Discussion
the outcome is satisfactory, a common flaw in all the autoencoders‚Äô reconstructions
is the poor reconstruction of the 2-3 pixel border. This is due to the design of the
architecture itself. To be precise, it is because the last 2D convolution layer has to
map 64 channels down to 1 channel and the last padding layer uses a padding of 3.
For the 2D convolution, the problem is this layer acts like a bottleneck of information
since this layer has to up the density of information by a factor of 64, while also
moving the information out to the edges of the image. This may be amended by
reducing the number of channels in the outermost layers of the autoencoder, as well
as possibly adding more layers with a gradual increase/decrease in channels.
The example forecast and target time series in an example position shown in sec-
tion 11.2, show the volatility of the data set produced in this project. The volatility
in the time series is thought to both include actual high-frequency changes in SLA,
but also noise. Thus, the autoencoder could potentially also be used to remove the
noise in time instead of only removing the noise which comes from the satellite tracks.
This will, however, require either a change in the architecture of the autoencoder or
inserting multiple days of data as channels into the autoencoder with modifications
to the loss function.
12.4.2
ConvLSTM based architectures
One inherent flaw in the encoder-decoder ConvLSTM based architectures, is that the
information has to propagate through multiple ConvLSTM blocks before being used
in the decoder. This means after each timestep the initial information has a tendency
to get weighted less and is thus lost over time. One approach used in this project to
fix this inherent issue, is to use attention to better propagate information over time.
Another flaw was found during preliminary testing with many different ConvLSTM
based architectures. The number of channels for the hidden state and cell state is
highly correlated with how fast the models overfit. The biggest problem with over-
fitting, especially at a low amount of epochs, is that only the general features are
forecasted. Then, the model has not had enough time to learn how features move
and change over time.
The autoencoder-based ConvLSTM architecture has shown a large speed improve-
ment over the ConvLSTM based architectures since the ConvLSTM layers operate in
the latent space of the autoencoder. Thus, the size of the hidden state and cell state is
decreased by 2(N-Downsamples). Furthermore, the results show the model understands
movement in the SLA spatial features much faster than other architectures. However,
this type of model is even more prone to overfitting, as the latent space distribution
and magnitude are very different between the train, validation and test splits. During
testing of the model architecture, a large number of different parameters, channels
and settings were used. However, aside from the single model producing the results
shown in subsection 11.2.4, all the attempts resulted in very early overfitting. This
indicates the architecture has potential, but further investigation of regularization
of the latent space is needed. Adding regularization to the latent space would force

CHAPTER 13
Future work
Below, suggestions are made for a continuation of the project, as well as potential
improvements to the methodology in both Part I and Part II.
13.1
Gridding
One of the main parameters to be investigated further in regards to gridding, is the
time-to-distance weighting. Either efforts to find an optimal weight should be pur-
sued, or a different approach entirely on how to weigh the data temporally compared
to spatially should be found. Alternatively, different filters could be used to achieve
the best possible grid product after choosing a time-to-distance weighting, i.e. to at-
tain both spatial continuity and temporal continuity. One approach briefly explored
in this project is using autoencoders instead of using simple Gaussian filters. Fur-
ther investigation into removing non-physical noise from the data should be explored.
This could be removing satellite tracks and some of the underlying high-frequency
noise which is present between gridded days. Different changes to the loss function of
the autoencoder, which will provoke the model to not fit specific kinds of underlying
data noise distributions also have to be investigated. This could be weighting the
loss produced by the satellite tracks much lower. Other types of ‚ÄĚsmart‚ÄĚ filters can
also be utilized to remove the noise, while still retaining the physical higher frequency
features. Already before the data is gridded, more outlier detection should be per-
formed to further remove data errors from the grid. Another approach is to remove
the along-track noise using e.g. an autoencoder instead of using a low pass filter as
done for CMEMS. One of the biggest causes of errors in the gridding product is de-
viations in the RADS processing chain. Thus, more analysis of the processing chain
should be performed to improve the processing steps in RADS. Another approach is
to investigate the uncertainty of the data and models used in RADS, propagate the
uncertainty to the gridding process and weigh the data according to the inverse of
the uncertainty. Furthermore, the produced grids tend to have features stretched in
the latitude direction, which is likely due to the track following this direction, which
results in a bias. Thus, it should be investigated if a weighting in latitude is needed to
remove the bias in this direction. Adding information from tide gauges could increase
stability in the data product, especially near the coasts where most tide gauges are
positioned.
==PAGE:110==
96
Future work
The most important change to make in the next iteration of the gridding product
produced in this project, is to change when a satellite is replaced as the reference
satellite in the TOPEX-Jason series.
Currently, the reference is changed when a
satellite mission ends. However, the reference should change when a new satellite
takes over the orbit of the last satellite. This change will likely improve the crossover
correction made in the processing pipeline and thus improve the grids slightly.
13.2
Forecasting
Based on the forecasting results produced on the CMEMS data sets, the architecture
used in this project seems reasonable. Rather, it appears that data volatility is the
problem for this architecture. One approach to consider to amend this, is to grid
the data in 1/4‚ó¶√ó 1/4‚ó¶resolution and train the model on the CMEMS data sets,
and then do transfer learning to make forecasts on the data set used in this project.
Alternatively, a more smooth version of the grid produced in this project could be
made to train on first, before making the switch.
To improve the forecasting power of the investigated model, including additional
features in the model should be tested. These features include, but are not limited
to the SST, SWH, wind speed, the number of satellites used to produce the grid,
an ENSO parameter, and the grid uncertainty, either a value per day or per grid
cell. The grid uncertainty can likely only be obtained by either propagating the data
uncertainty from RADS, or by using a different algorithm for the gridding process,
i.e.
kriging (Gaussian processes), which provide an error estimate.
Adding these
features could help the model to better estimate the SLA where the Eddy currents are
present. If multiple features are used, changes to the Encoder-Decoder ConvLSTM
based architecture are necessary. Either all the features have to be forecasted along
the SLA, which will likely make the model more uncertain about its forecast, or one
layer has to be added between the encoder and decoder, which translates the cell and
hidden state. The translation is necessary because the encoder will use the additional
features, while the decoder will only use the SLA feature. A combination of these
two approaches can also be used, or other models which forecast one of the features
very well can be integrated into the decoder‚Äôs input.
An alternative architecture to investigate in regards to forecasting is the Transformer.
In text prediction, the results from models of this type are very good compared to
LSTM based architectures, when tested on the same data sets. Other slightly less
relevant architectures include but are not limited to: spherical convolutions, PINN
and Graph Neural Networks.

CHAPTER 14
Conclusion
This project was made in two parts. In Part I, a pipeline using data from RADS to
construct a global gridded product containing SLA, SST, SWH and wind speed was
created. The purpose has been to investigate the feasibility of using a smaller tempo-
ral window for interpolation than similar products, such as CMEMS and MEaSUREs.
In this project, a window of ¬Ī5 days has been used, compared to 6 weeks in CMEMS
and 30 days in MEaSUREs. Furthermore, the other products use heavy filtering in
order to remove noise such as track residuals in the grid products. For this product,
some processing has been applied to the data after extraction from RADS, but no
further processing has been applied to the grids after interpolation.
The resulting product of the grid interpolation using the pipeline in chapter 4 shows
the same general features as seen in CMEMS and MEaSUREs. However, more high-
frequency variations are seen in the product from this project compared to the other
products. These are assumed to be a combination of actual physical features, which
are filtered out of CMEMS and MEaSUREs, and data errors propagated from the
data in RADS or created during the gridding process. Comparing the SLA grid pro-
duced in this project to MEaSUREs based on summary statistics, reveals a very small
deviation between the two data sets, especially in the non-polar regions. For CMEMS,
the discrepancies are slightly larger, but still within a reasonable margin. Thus, the
pipeline used in Part I is deemed a valid method for creating a grid interpolated SLA
data product.
In Part II, two major results have been produced. First, an autoencoder was imple-
mented followed by a forecasting model for the SLA data product. For this part of
the project, the global SLA grid product produced in Part I has been used. Further-
more, an AOI near South Africa was regridded using the pipeline in chapter 4 and
used in the autoencoder. Initially, the autoencoder was implemented in an effort to
use the trained latent space in the forecasting model for a speedup. However, the
autoencoder may also function as a filter for removing unwanted signals and noise in
the data. Using the autoencoder for speedup in the forecasting model was successful.
However, due to inconsistencies in the model architectures and a lack of restriction on
the magnitude of the autoencoder latent space, the models using the autoencoder for
speedup were very prone to overfitting. Preliminary testing of using the autoencoder
for removing unwanted signals has proved very promising. Thus, the autoencoder
may be used as a smart filter for the data errors in the data product from Part I,
rather than resorting to smoothing methods such as Gaussian filters.
==PAGE:112==
98
Conclusion
The forecasting model produced in Part II consists of an EDAConvLSTM architecture.
For the forecasting, CMEMS data from the AOI near South Africa and an additional
AOI in the South China Sea were both used to train models in order to test the data
dependency. In general, the model performance was similar for the two data sets in
South Africa, although the RMSE is lower for the model trained on CMEMS data
due to the lower magnitude and variation of the data. The model performance is
significantly worse in the South African area than in the South China Sea, due to the
complexity of predicting ocean currents and Eddy currents. Comparing the results
from the South China Sea to those in the same area in [23] reveals that the forecasting
model produced in this project performs better overall. The RMSE is lower on all pre-
dicted days, especially for longer forecasts. For the CMEMS SLA data product, the
forecasting power is reliable for approximately 30-40 days. Thus, extending the prod-
uct by the size of the temporal window in the gridding process of 6 weeks is almost
possible. Furthermore, for the SLA data products from this project, the prediction
power is reliable for about 7-10 days. This allows for a continuation of the data prod-
uct beyond the temporal window used of 5 days, thus extending the product a few
days beyond the current date if continually updated. Wrapping the EDAConvLSTM
model in the autoencoder, creating the E2D2AConvLSTM architecture, made the
model very prone to overfitting.
Results from the one model successfully trained
without immediate overfitting have shown that using this architecture is feasible, as
they were comparable in error to the results of the EDAConvLSTM without the au-
toencoded data.
In conclusion, creating a data product containing SLA, SST, SWH and wind speed
using a small temporal window in the interpolation is feasible. The current implemen-
tation is not optimal, however with the suggestions from section 13.1 it may be an
improvement from currently existing ways of creating products, in regards to includ-
ing higher frequency features. Using the autoencoder for speeding up the processing
of the forecasting model has been proven possible. The issues of overfitting may be re-
solved if the architecture is altered and the latent space is constrained. Furthermore,
the autoencoder shows great promise as a smart filter for removing unwanted signals.
Forecasting using the EDAConvLSTM allows for a decently reliable extension of the
SLA data products to the current date, such that they are no longer behind due to
the temporal windows of interpolation. The model may be improved but outperforms
similar models from other work. Using the model for global forecasting with a high-
resolution version of the SLA grid product is deemed feasible, although issues may
arise with the added information.

A.3
Algorithms
Algorithm 1: Block-averaging algorithm where memory usage has been optimized.
N is the size of the positions set X. M is the size of the dimension of each position (xi),
while F is the size of the dimension of each target (ti). ‚ÄĚconstructVectors‚ÄĚ constructs
the orthogonal vectors that define the block average space. ‚ÄĚwhere‚ÄĚ finds the index
of where x lies in the block average space, i.e. an integer. ‚ÄĚassignBlock‚ÄĚ assigns or
adds t to a block.
1: procedure blockaverage(X, T, blockSize, resolution)
2:
orthogonalVectors ‚ÜźconstructVectors(blockSize, resolution)
3:
blocks ‚ÜźAllocate(N, M + F + 1)
4:
counter ‚Üź0
5:
lookup ‚ÜźHashmap[int, int]
6:
for x, t in X, T do
7:
gridIndex, block ‚Üźwhere(orthogonalVectors, x)
8:
assignBlock(blocks, gridIndex, counter, loopkup, block, t)
9:
end for
10:
Del blocks[:counter]
11:
blocks[:, M + 1:] = blocks[:, M + 1:]/blocks[:, 0]
12:
return blocks
13: end procedure
