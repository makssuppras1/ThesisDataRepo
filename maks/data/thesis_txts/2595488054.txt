Abstract
In this Master’s Thesis we investigate whether it is possible to use deep learning neural
networks to predict segmentations of various Sea Ice Concentrations (SIC). This is currently
being done manually by several operators at the Danish Meteorological Institute (DMI),
where the SIC maps (ice charts) are drawn by hand in a Geographical Information Systems
(GIS) tool. This is a very time consuming process with varying results from operator to
operator.
From the Sentinel-1 satellite we get HH and HV data in the form of larger
geographical areas (scenes), from which we extract smaller patches (cut-outs) of equal size.
These cut-outs serve as training, validation and test data for our Convolutional Neural
Networks (CNN). The project is split into two parts, where we ﬁrst produce a binary
classiﬁer splitting water and ice at an SIC of 10%. We test several encoder-decoder network
architectures and ultimately pick the one which performs best. As this classiﬁer produce
binary segmentations with noise, we use Markov Random Fields (MRF) with Grapth-cut
optimization to produce a more uniform segmentation. In the second part we produce
multilabel classiﬁers. We test two diﬀerent multilabel classiﬁcation methods; Root Mean
Square Error (RMSE) regression and Soft-max classiﬁcation. Each of these methods are
tested with four diﬀerent networks for a total of eight multilabel classiﬁers. One model
will do multilabel segmentation using the HH and HV satellite data as with the binary
predictions. The other three will try involving the binary predictions as a third network
input. This is tested on three diﬀerent scales, where water and ice are given as 0/1, −1/1
and 0/100 respectively. We conclude that the RMSE model with binary segmentation given
on the 0/1 scale overall performs best, as it has a relatively low RMSE value. Additionally,
when the model predictions are binarized, it either performs the best or is among the best
performing models. Since training of both binary and multilabel classiﬁers indicates overﬁt
of training data, we also conclude that more data and further regularization is needed in
future experiments.

1
Introduction
In the colder regions of the world such as Canada and Greenland, the waters are inhabited
with large amounts of ice in many diﬀerent concentrations. It is dangerous for ships to
fare in these areas, and therefore they use Sea Ice Concentration (SIC) maps provided by
the Danish Meteorological Institute (DMI) when they are to pick the best route to steer
around the large regions of ice. Today these maps are produced manually, where they are
hand drawn into a Geographical Information Systems (GIS) tool by an operator. This
process can be really time consuming, which means that the information might no longer
be relevant by the time maps are drawn and handed oﬀto the ships. Another problem is
the poor reproducibility of the maps, as the maps would vary from operator to operator.
We would also see varying results over time from the same operator. Thus we experience
both inter and intra operator variability. It is therefore diﬃcult to produce consistent SIC
maps, that can keep up with the demand, resulting in an unnecessary strain put on the
economy of the involved companies as well as the environment due to longer travel.
With the project Automated downstream Sea Ice Products (ASIP) the goal is to au-
tomate the process of producing the ice charts, in hope of achieving standardized maps at
a higher rate and better coverage. In this Master’s Thesis we will build on the Synthesis
Project ”Predicting Sea Ice Concentrations with Deep Learning” [3], where Convolutional
Neural Networks (CNN) were were used to estimate the mean SIC of smaller areas called
cut-outs, which were extracted from larger scenes. The results of the predictions would
then be put back in the position from which the cut-out was extracted. In this project we
will build upon that idea and design new networks with inspiration in preexisting segmen-
tation network architectures. This means that instead of predicting one mean SIC value
to represent one cut-out, we will have a feature map of the same size as the cut-out, such
that every input pixel has a network output counterpart. In the Synthesis project we used
two data sets, the ﬁrst of them being Synthetic Aperture Radar (SAR) recordings from
the Sentinel-1 (S1) mission and the second data set coming from the Advanced Microwave
Scanning Radiometer 2 (AMSR2). We want our predictions to be available as soon as
the data is retrieved from the satellites, and since AMSR2 data comes with quite a delay
compared to S1, we only keep the S1 data for this project. From the S1 satellite we get
HH and HV amplitude measurements, which serve as input data in our neural networks,
where they are paired with the ice charts produced by DMI. The network will train on this
data and learn to recognize patterns that are associated with certain levels of SIC. This
way our CNNs will predict the SIC when given new data, and thus produce larger sea ice
maps in the fraction of the time it would take for a human to draw them in hand.
This project is part of the already ongoing ASIP project, where most of the data
used is already processed and ready for use. We will then take this data and ﬁrst ﬁnd
a proper network structure for doing binary predictions, which will separate water from
ice. Afterwards we will take this information and use to do multilabel segmentations using
diﬀerent methods, which we compare to each other.

The Sentinel Mission and Data
The data used for estimating SIC in this project comes from the S1 mission.
For this
project we use data recorded between the 1st of December 2014 and 16th of May 2018.
Newer data is available, but has not yet been processed for further work (at the time of
this project). There are currently 6 Sentinel missions, where Sentinel-1 (S1) is the ﬁrst
launched back in April 2014. Since then Sentinel-2 (S2) and Sentinel-3 (S3) have been
sent into orbit along with Sentinel-5P (S5P), which works as a precursor for the upcoming
Sentinel-5 (S5) mission. Other than that, Sentinel-4 (S4) and Sentinel-6 (S6) have been
conﬁrmed for future launch. The Sentinel Mission is developed by the European Space
Agency (ESA) to serve as a next-generation of satellites to replace older missions nearing
their operational life span. Each mission has diﬀerent Earth observation goals within in the
ﬁelds of atmospheric, oceanic and land monitoring, and will contain two twin satellites to
provide better coverage. The mission is part of the Copernicus programme, which is a joint
initiative between ESA and the European Commission (EC) [4]. S1 has been continuing
the C-band SAR Earth observation heritage of ERS-1 (July 1991 - March 2000), ERS-
2 (April 1995 - 2011) and ENVISAT (March 2002 - May 2012), all by ESA, as well as
Canada’s RADARSAT-1 (November 1995 - 2013) and RADARSAT-2 (December 2007 -
still operating). With the new satellite generation we will see improvements in reliability,
revisit time, geographical coverage and rapid data dissemination [5].

and S1 is no exception. We have both S1A and S1B, which are polar-orbiting satellites
with SAR instruments launched approximately two years apart on 3rd of April 2014 and
25th of April 2016 respectively. The S1 instruments includes C-band (around 5.405 GHz)
imaging and operates in four acquisition modes.
• Stripmap (SM)
• Interferometric Wide Swath (IW)
• Extra-Wide Swath (EW)
• Wave Mode (WM)
The principles of each acquisition method is illustrated on Figure 1. Our data is col-
lected using the EW mode meant for oceanic, ice and polar areas, where wide coverage
and short revisit times are essential. In EW mode the data is collected much like with
the IW mode, where larger subswaths are combined to the full image. For IW we collect
three subswaths spanning a width of 250 km. It is the default mode for data acquisition
over land areas. In comparison the EW mode collects ﬁve subswaths spanning 400 km and
has a lower resolution [1]. The radar collects data using electromagnetic (EM) radiation in
the microwave range of the electromagnetic spectrum. In this range the radiation will pass
through clouds, meaning that weather conditions won’t aﬀect the acquisition of data. The
SAR system has four polarization conﬁgurations. The polarization refers to the orientation
of the EM waves emitted or received by the radar antenna. The four types are:
• HH: Signal is sent and received as horizontal EM waves.
• HV: Signal is sent as horizontal EM wave and received as vertical EM wave.
• VV: Signal is sent and received as vertical EM waves.
• VH: Signal is sent as vertical EM wave and received as horizontal EM wave.
We use the ﬁrst two of these channels, i.e. the HH and HV bands [6][7].
2.2
Data and Preprocessing
Our data consists of large scenes with an approximate size of 400 km × 400 km. Each
of the scenes will typically be around 10,000 pixels × 10,000 pixels, with a resolution of
approximately 90 m and a pixel spacing of 40 m × 40 m. Each pixel intensity is represented
by the backscatter coeﬃcient and given in decibel [dB]. The data has been scaled down
from a [−30; 10] range to a [−1; 1] range, such that −1 corresponds to -30 dB and 1 to
10 dB. This will also make the data easier to work with, when we start using it in the
neural networks. If we collect all the pixel intensities from each scenes, we will see that
the HV channel mainly has intensities, which lies below the range of [−1; 1]. If we were
to depict our data in this range, the whole HV scene would not show us a whole lot of

Scenes
A scene is a single local area around the coast of Greenland. These are as already stated
approximately 10,000 pixels × 10,000 pixels each. The S1 scenes has been recorded from
both S1A and S1B, and are given as netCDF4 ﬁles with the name format:

NNNNNN NNNNNN NNNN.SAFE.nc

polarization and sar secondary is HV. It also contains the SIC maps made by DMI, which
are given in the variable icechart. The S1 ﬁle also contains latitude, longitude, heights
etc., which for example can be used for making GeoTiff ﬁles. We can see several examples
of both the HH and HV channel along with the corresponding ice chart in Figure 2 on

characteristics.
They are good examples for test scenes, given that the examples vary
in diﬃculty when it comes to predicting. We have mixes of scenes with many diﬀerent
concentrations of ice and some with fewer. In some scenes we clearly see the subswaths
from the EW data acquisition mode, and in others this isn’t apparent at all. In all our
scenes we see a lot of white areas, which are registered as invalid values (NaN). They are
known land areas that have been masked out since computing SIC over land would make
little sense. The ice concentrations are given in 22 classes between 0% and 100%, which
are distributed with 5% intervals as 0%, 5%, 10%, . . ., 90%, 95% and 100%. This would of
course only give us 21 classes, but an additional class of 1% sea ice have also been added.
This is for areas where we can’t be absolutely certain there is no ice, simply because the
concentration might be so low, but we also can’t conﬁdently say there is any ice. This 1%
class occurs so rarely, that we will omit it by rounding it to 0% and have it included in
that class, thus forward work will be with 21 classes.

Prior Work with Neural Networks for SIC Prediction
This project itself builds on the ASIP Synthesis Project [3], where neural networks were
used to predict mean ice concentrations of entire cut-outs. Here the HH and HV channel
from the S1 satellite also served as input data, but data from the AMSR2 mission men-
tioned was also used. The goal of the mission was to combine the high resolution images of
the S1 satellite with the lower resolution AMSR2 images, which were better at distinguish-
ing water from ice. The models used had both elements of CNNs as well as Multi-Layer
Perceptrons (MLP), where diﬀerent sizes of input data were tested. To estimate the mean
ice concentration of each cut-out, regression were used with a Root Mean Square Error
(RMSE) loss function. We don’t use the AMSR2 data for this Master’s Project, since the
the S1 data is more frequently updated in comparison, and thus we want a model which
only depends on the S1 data.
The Synthesis Project was originally inspired by [8], which is therefore the foundation
of this project. Here HH and HV channels too were used from the RADARSAT-2 mission.
This data was collected around the Gulf of Saint Lawrence between 16 January 2014 and
10 February 2014 during the freeze-up. Data is collected from larger scenes in the form
of 45 × 45 patches. The networks used contains both CNNs as well as Fully Connected
(FC) layers. An L2 loss is used for the regression problem and is trained both with and
without incidence angle as input data along with the satellite images. The results hereof
are compared to a MLP network trained on ten Gray-Level Co-occurance Matrix (GLCM)
features. Finally they use AMSR2 data in the ARTIST Sea Ice (ASI) algorithm to also
produce predictions of the entire scenes. The best results (lowest values of RMSE) were
achieved with the CNN model without incidence angle. The article is further outlined in
Ph.d. thesis [9] from author Lei Wang.
In [10] they built upon [8], by using a slightly diﬀerent network structure, but still
combining CNN and FC. The focus hereof is on the marginal ice zone (MIZ), which is the
middle sea ice concentrations at 15% to 80% [11]. These are diﬃcult for the AMSR2 sensor
to recognize, since it is very sensitive to cloud liquid water and water vapour, which are
found in higher quantities in the MIZ. They train the network using same data set from
the Gulf of Saint Lawrence as in [8] during freeze-up. They do several tests including data
from other areas of Canada, only high and low (HL) concentrations of ice (outside the MIZ)
and sampling more data from the MIZ. The tests indicate that only using the HL data set,
has a higher error than using all the data. Hence inclusion of data samples in the MIZ is
important, and the results from sampling even more of these also shows improvement.

Neural Networks
The idea of neural networks goes way back, as it was originally inspired by the study of
the human brain. Back in 1943 a paper was published by neurophysiologist Warren Mc-
Culloch and mathematician Walter Pitts on how brain neurons supposedly works, which
is where the neural networks got their name. In the paper McCulloch and Pitts had tried
to model a simple neural network using electrical circuits, which would lay the foundation
of the neural networks as we know them today. Over the years computers became more
advanced allowing for the modelling of these networks, yet the ﬁrst attempts were rather
unsuccessful. As neural networks got better over the years, there was also a rising con-
cern with the capabilities of ”thinking machines” heavily inﬂuenced by popular ﬁlms and
literature, which often depicted horror scenarios of robots working against mankind. This
would result in a halt of funding into the research of neural networks. In the beginning of
the 1980’s, the Americans started to fear that they might fall behind in this ﬁeld, as the
Japanese had invested a lot of money and put a lot of eﬀort in the neural networks ﬁeld,
and the funds once more started ﬂowing [12][13].
Today neural networks have become quite the hot topic, as computers have gotten
so powerful that even personal computers can start to keep up with the computational
power needed. As problems get more complex, more neurons are needed to ﬁnd a proper
solution, and thus more calculations need to be done. Here a Graphics Processing Unit
(GPU) is an essential tool for doing heavy parallel computations. Since powerful GPUs
can be quite expensive, many companies like Amazon and NVIDIA oﬀer cloud solutions,
where users rent GPUs. For this project we use the GPU Cluster oﬀered at DTU Compute.
By now there exists quite a few diﬀerent packages for working with deep learning in
diﬀerent coding languages. In Python alone there is a wide range of libraries that can be
used for building and training neural networks. Some of these include TensorFlow and
Keras, but for our project we will be using the PyTorch library [14].
4.1
What is a Neural Network?
Neural networks come in many shapes and sizes. They are designed such that they can
solve a speciﬁc problem, for example classiﬁcation or segmentation. If one is familiar with
the term neural networks, one might very likely also have heard about machine learning
and deep learning, which are not exactly the same.
In machine learning we would for
example extract features, such as edges or blobs from images, and then train our network
to recognize these. In deep learning we would feed the images directly into the network,
and have the network itself ﬁgure out which features would belong to a certain class (if we
are doing classiﬁcation). This also means that a lot more data is necessary for our network
to properly identify these features. We would also need our deep learning network to have
a lot more neurons, since the network does not know what it is looking for. It will just
recognize patterns as it learns.

layers, what really happens inside these is hard to account for, which is why the neural

the best solution it can ﬁnd is.

4.4
Gradient Descent
At this point our network has processed the image once, provided us with its qualiﬁed
guess of what number the image depicts, and we have a measure of how well the network
did this; the loss L. Since the weights are randomly initialized, the loss will be very high
during the ﬁrst round of training. We will therefore have to update the weights such that
next time we process the same image, we get a result that better resembles the targets.
The way we do this is using Gradient Descent, where we move closer to a solution of lower
loss in model space.

When we train our network, we input a lot of training data, compute the loss and use
Gradient Descent to update our weights. We call it an epoch, when we have trained on all
our training data once. We repeat the process over several epochs until the weights have
been adjusted to satisfactorily solve the problem. Since deep learning requires a lot of data,
we would rarely ever be able to train the network on all the data we have at once, given
that it would be too computationally heavy. We therefore split up our data into smaller
chunks of data called mini-batches. The data in a mini-batch should be drawn at random
from the complete data set, in order to avoid the neurons from recognizing a pattern in the
order it receives the images, as the total loss of a mini-batch aﬀects the directions Gradient
Descent will have us move in hyper space. Therefore when choosing a mini-batch size, it
should include enough data to generally represent the complete data set, but not so many,
that the memory of the computer or GPU can’t contain it. The weights will update with
every mini-batch, and since they are drawn at random, the direction towards the minimum
won’t be as direct as if we were able to process all data at once. On the contrary we would
see a little wiggle back and forth in the hyper space, much like a random walk towards the
minimum. This is called Stochastic Gradient Descent (SGD) due to the noisy nature of
the descent.
4.5
Back-propagation
With forward propagation we computed the neurons of a new layer using the neurons of
a former layer with a bias node and the weights. With back-propagation we investigate
which adjustments to our weights is going to cause the highest decrease of the loss. Thus
we start at the last layer, where we compute the loss at the output layer y. We will then
use the information from the current layer to update the weights between this layer and the
preceding layer, moving backwards in the network. We give our layers an index, to denote
how they are tied to each other. Last layer has the index (L) and the output layer will
thus be called ˆy(L). (L) is not an exponent or the loss function, but the layer index, and
the reason we write as such will make more sense with the subscripts introduced shortly.
The preceding activated layer will be called h(L−1) and the weights between the two layers
W(L). The non-activated current and preceding layers are then written as z(L) and z(L−1)
respectively just like in the forward propagation.

In the example just presented, we saw a fully connected neural network, where the input
data had 784 neurons, which were all connected to a hidden layer of 20 neurons. Including
the bias node, it would take 15700 weights to compute the ﬁrst hidden layer of the neural
network. This is a lot of parameters, which require quite a lot of computational power to
work with. As images get larger and larger, the number of weights needed would blow up
if we were to stay fully connected. Not only do the images have more pixels, but we would
also see images, like RGB, which would have several channels.
We therefore introduce
CNNs, which is the idea of using convolutions to compute a new layer of neurons. This
is done by taking a convolution kernel and apply it to an image and all of its channels,
which would result in one neuron in a new layer. The kernel will then contain our weights,
which will be shared throughout the entire image instead of applying a unique weight to
each neuron. We then update the weights in the convolution kernels using SGD. In Figure
7 we see how the kernel is applied to produce a single neuron.
Figure 7: Convolution kernel used across three channels
Applying a ﬁlter to the image and all of its channels will produce a feature map, and by
using more than one convolution ﬁlter with diﬀerent weightings, we will get a new layer in
the network containing several feature maps/channels, one for each ﬁlter kernel. The size
of the feature map will depend on kernel size, stride (step size of kernel) and whether or
not we use padding. Let us consider an example where a 3 × 3 kernel is applied to a 4 × 7
image as seen in Figure 8.

==PAGE:27==

If we use a stride of one in each direction (we only move our kernel one pixel at the
time), the kernel will produce a new 2 × 5 feature map. Due to the size of the kernel, we
lose the outer rim of the image, resulting in two less pixels in each direction. If we want to
avoid this, we could use padding where we add an extra edge of some values, and there are
a few padding methods to choose from. We could use zero-padding, where the extra edge
is made up of only zeros, but this could negatively aﬀect the outskirts of the image. We
could instead pad the layer with the same value as the neighboring pixels. This way the
local information is better preserved.
4.6.1
Downsampling
There are diﬀerent types of CNNs used for solving diﬀerent problems. If we consider the
MNIST problem, which could advantageously be solved using a CNN, we would decrease
the size of the feature maps, such that we ultimately end up with the 10 neurons, one for
each class. This would be done by either downsampling the layers until we had a 1×1×10
layer (one single pixel in 10 channels). We would then use this layer as the ˆy mentioned
in the section on forward propagation. Alternatively we wouldn’t have to downsample the
layer to 1×1×10, but merely to a point, where there are signiﬁcantly fewer neurons. Then
we can squeeze all the neurons from all channels into an input vector, which we can use
for a fully connected network. Either way we need to downsample the image, which can
be done by not using padding when we perform convolutions (as seen in Figure 8). This
would however only eliminate a few pixels at a time, so instead we could use pooling, where
we highlight some of the most dominant local features. If we consider an area of 4 × 4 like
seen in Figure 9, we can use a method called max-pooling, where we take the largest local
values and make a new layer out of these. We would thus have a new hidden layer with as
many channels as the prior, where each channel consists of the highest local values of each
respective preceding feature map. Other methods include taking the mean of these local
areas or merely pick one of the local values at random. The max-pooling shown in Figure
9 would downsample the feature map just like if we had used a convolution with a ﬁlter
kernel of 2 × 2, a stride of 2 and padding of 0.

Upsampling
As we will see in Section 4.8 on network architectures, downsampling is only one part of the
structure. We will have to upsample the feature maps again to get a segmentation similar
to the target ice charts. We do this by unpooling the feature map, which happens as an
extension of the max-pooling. In Figure 9 we see four neighbourhoods of diﬀerent colors,

in PyTorch is nearest neighbour [18].

The principles of the fully connected and the convolutional deep learning neural network
are very much the same. We start out with an input with a matched output (target), set
up our network such that it has enough hidden layers and neurons to recognize certain
features in the input data, and then we train the network on the data and update the
weight parameters such that the loss is minimized until we reach a satisfying result, when
we present the network with new test data [19].

4.7.2
Mini-batches and Batch Normalization
We have already discussed mini-batches as being a helpful tool for dealing with limited
GPU memory. We discussed how the a mini-batch should be large enough to properly
represent all the data, but not so large that we see this ”lack of memory” problem. But a
small batch size could actually have a regularizing eﬀect, due to the noise added during the
learning process. With very small batch sizes we would need a much lower learning rate,
resulting in a much longer running time, since the network learns at a slower rate and it
takes more steps to observe all the data [22].
The data we load into our networks must have some common scale, which can be
achieved by normalization or scaling like we saw with the input data, where the range of
[−30dB; 10dB] was scaled down to [−1; 1]. We can do the same thing for the batches of
data we load. After we computed and activated a new hidden layer, we normalize the
whole batch.
Of course this won’t give us an overall normalization of all the available
data, but instead only the single batch, so depending on the randomly chosen input data,
the normalizations can vary, which could alter the intended inﬂuence of the weights in the
already trained network, which in terms could also add robustness to the network.
4.7.3
Dropouts
Dropouts work by ”eliminating” neurons of the network, which is done by setting their
output weights to 0. The method called Bagging allows us to combine several models and
then have them vote for the output. Since it would be very computationally heavy to train
several fully connected models, dropouts work as a sort of psuedo-bagging, where shutting
down neurons would work as diﬀerent models. The network architect would pick a certain
percentage (typically 10% −20%) of the nodes to be dropped, thus each subnetwork would
resemble the original network, but with closed oﬀpaths. Each network shares the weights
by inheriting a subset from the original weights. Just like bagging, the diﬀerent models
then vote on a solution, such that we get an average of each solution.

4.8.1
SegNet Architecture
Figure 13: Example of a simple SegNet, which reaches a bottleneck of the size 75 × 75
The SegNet architecture, which we can see an example of in Figure 13, has both an en-
coder and a decoder. The encoder part works very much like described in Section 4.6, where
feature maps are downsampled using either convolutions or max-pooling. Eventually the
feature maps will be very small, and we would decide to stop downsampling anymore. We
call this point of the neural network the bottleneck, due to the narrowing shape hereof. In
the example depicted in Figure 13 we have downsampled till we reached a bottleneck size
of 75 × 75. When we start doing experiments, we will see diﬀerent bottleneck sizes, and
thus go deeper in the encoder. With the example of the MNIST data set, we would keep
downsampling until we only had 10 neurons, one for each class, but for segmentation exer-
cises we typically want a ﬁnal feature map of the same size as the input data. Therefore we
will start upsampling the feature maps from the bottleneck in the part we call the encoder.
We keep upsampling until we once again reach the same feature map dimensions in width
and height as the true ice charts. At the end of the encoder we will thus have achieved our ˆY.
Figure 13 shows a layer with N number of channels as the light blue rectangles, which
are the results of the preceding operation (convolution, max-pooling or upsampling as used
in the example). The sharp reader would already have noticed that the numbers written
vertically indicates the width and height of the respective layers. The numbers written on
top of each layer denote the number N of channels/feature maps in a layer, and thus how
many convolution ﬁlters used to compute a layer (in the case of convolution). The very ﬁrst
layer is the input layer containing the HH and HV channels, i.e. two channels. Hereafter
16 convolutions are used for computing the next layer, one for each channel. Following is
the max-pooling used to halve both width and height. We continue like this until the last

layer, which has X channels given, as this varies based on classiﬁcation and loss method.
The SegNet structure has a hierarchy of decoder blocks for every encoder block, such that
every time we downsample in the encoder, we upsample accordingly in the decoder. As we
go deeper, we double the number of channels in each layer when performing convolutions.
We reverse this process such that the number of channels is halved in the decoder after
every upsampling [27].

4.9
Receptive Field
Though it can be diﬃcult to interpret what we see in the feature maps of our neural net-
works, we have other ways of accessing the impact of nodes further down in the neural
network. We could for example look at the receptive ﬁeld of the feature maps. As we go
deeper into the network, a node (or pixel) from the original network input will have a larger
and larger eﬀect further down the layers. Our input cut-outs have dimensions of 300×300,
and the way we move from layers to layers could be by using convolutions, max-pooling or
upsampling. In the neural networks we will be working with (see Section 5), ﬁlters will have
the size 3 × 3 when performing these convolutions. In order not to downsample the feature
map during these convolutions we use padding of 1. We will also be using a stride of 1,
which denotes how many pixels we move the kernel every time we perform a convolution.
With stride 1 we end up performing a convolution on each pixel in the feature map. Thus
when performing the convolutions on a 300 × 300 feature map, we get a new feature map
with the same dimensions. Each pixel in the new feature map has been computed using
a ﬁeld containing 3 × 3 pixels from the previous feature map. We say the receptive ﬁeld
of the network thus far is 3 × 3. Were we to just stop here, we would have a very poorly
performing network, so we will continue to go deeper, by having more convolutions as well
as max-poolings.
In ﬁgure 15 we see an example of how the receptive ﬁeld grows larger as we go deeper.
For the the simplicity of the examples we are using square kernels and feature maps (as
we will for our networks). Initially we have the blue input data seen at the bottom. This
contains n = 5 input features in each dimension (width and height). The convolutions

and padding p = 1, thus preventing shrinkage of the layers. After the ﬁrst convolution, we

further we perform another convolution with the same values in order to produce another

now expanded to be r = 5. This means that when computing the center pixel of the red
top layer, we actually use all of the pixels in the 5 × 5 blue bottom layer.

5.1
Texture Analysis as Data Preprocessing
The data used in our neural networks is as already mentioned the 300 × 300 cut-outs

new input data that could be used for improved training of our models. The basic idea of

every pixel with an SIC value below 10% would be classiﬁed as water, and every pixel with

to ﬁt this problem. We could follow the approach of the MNIST classiﬁcation problem,
and use a channel for every class (two channels for binary problems) in the output layer
and thereby activate this ﬁnal layer with the soft-max activation function given in Eq. (4).

instead consider our water class as a negative class with the label 0, and our ice class will
be a positive class with the label 1. When we try to classify, we can then ask the question;
”Is the pixel ice?” or ”What is the probability of it being ice?”. Our goal will then be to
predict the positive class by having the network tell us the probability of a pixel being ice,
rather than giving us a probability of each class. To do this we will use a sigmoid activation
function for the output layer of our neural network. The sigmoid function is given in Eq.

5.3
Performance Measures for Binary Classiﬁcations
When we have designed our neural networks and trained them, we can test them on the
cut-outs in our test data set, to assess how well the model is performing when presented
with new data. New data in this case means data that hasn’t been used for training (or
validation (see Section 5.6.1)). Whether we want to know how good a job our models are
doing at predicting an entire scene or an entire test set, we will need some measurements
of the performance of each individual neural network, which will come in handy when we
try to compare them.
5.3.1
Confusion Matrix
For the binary segmentation there is a variety of useful performance measures, which can
all easily be computed if we ﬁrst extract the confusion matrix. The confusion matrix is not
strictly limited for use in binary problems, but when we only have two classes, there is a
variety of rates we can compute based hereof. We have the two classes; water and ice. In
section 5.2 we asked ”What is the probability of a pixel being ice?”. This meant that when
we sigmoid activated our network output, we would get a probability (number between 0
and 1) of how likely it is that a certain pixel represents ice. Afterwards we discretize the
output probability feature map such that we would get either zeros or ones. We do this
by assigning the pixel to the closest upper or lower limit, corresponding to thresholding at
0.5. In this binary segmentation problem, we say that we want to predict ice, such that
ice as an outcome is considered positive (1) and water is the negative (0) outcome. When
our model makes a prediction of the positive class, there are bound to be small deviations,
which is illustrated in Figure 24. Here we see an example of a true segmentation (yellow),
which our model tries to predict (orange).
Figure 24: Example of comparison between true segmentation and predictions from binary
classiﬁer.
The mask created by the overlap of the true segmentation and the prediction, shows
us the points where our model has predicted the location of ice correctly. We therefore say
that all pixels in this overlap are True Positives (TP). Outside both the predicted and true
segmentation we see the water, which due to the nature of the problem, is denoted as a

5.7
Post-processing of Binary Predictions using Markov Random Fields
5.7.1
Introduction to Markov Random Field
Markov Random Fields (MRF) can be used for segmentation of noisy images, where each
pixel in the image is initially assigned a discrete class based on the pixel intensity, and
hereafter we remove the noise in the segmentations. To be an MRF it has to follow the
Markov Property telling us that the probability of classiﬁcation is dependent only on the
local properties (also called the Markovianity). For images this means that the surroundings
of a pixel will determine the probability of being in some class. If the majority of this pixel
neighbourhood belongs to a certain class, chances are that so does the respective pixel. This
way we smooth out out the initial segmentation, which for noisy images are prone to be
noisy themselves. Our goal using MRF is to maximize the a posteriori probability, in order
to get the most probable segmentation from our criteria. This is the same as minimizing
the posterior energy, which is a cost of applying a class to a pixel. The posterior energy is
a combination of a likelihood energy, derived from the data itself, and a prior energy used
for smoothing the segmentation (give it the uniform appearance we seek). We can write
the posterior energy U(f|d) as
E(f) = U(f|d) = U(d|f) + U(f)
(44)
where U(d|f) is the likelihood energy and U(f) is the prior energy. f is then our classes
and d the data. This might look familiar as it is related to Bayes Theorem
P(f|d) = P(d|f) · P(f)

5.7.3
Optimization of MRF
We have now modelled our MRF, which is still noisy, and thus we need a way of optimizing
the MRF. There is a variety of algorithms for MRF optimization, and we will be using a
method called graph-cuts. It can be rather complex to set up these graphs, but luckily the
PyMaxflow-package for Python simpliﬁes this optimization process as soon as the problem
is set up like we did in former section. The package is available online [45]. It is produced
by one of the authors of the 2004 article on Min-Cut/Max-Flow algorithms and based on
the graph-cut algorithm outlined here as well [46]. The binary problem can we solved by
ﬁnding a minimum s-t cut of a graph constructed from the energy functions. Here s denote
a source (class 0) and t a sink (class 1) of the graphs. We then set up graphs between
every data point, and the source and sink called the terminal nodes.
If we cut one of
these graphs, the cost is determined by the one-clique potentials between the data point
and the respective class. Between each neighbouring data point (as deﬁned by two-clique
potentials) we draw a graph both back and forth with the cost of β to cut. These are called
the called the internal nodes. As an example we set this up for two data points.
Figure 37: Example of graph-cuts for two diﬀerent values of β.
Figure 37 shows the principle of graph-cuts. We cut the least expensive graphs (connec-
tions between points) thus assigning each data point to either source or sink. For β = 10 it
is cheaper to cross over at the graphs with β costs, thereby giving the two data points dif-
ferent labels, but at β = 100 it is too expensive to make the same cut, and thus both points
end up with the same class. This is of course a very simple example, but the principles

7
Discussion
7.1
Binary Classiﬁcation
We initially did binary classiﬁcation, where we split up our data at 10% SIC, such that
concentrations below would be classiﬁed as water and above as ice. We tested oﬀa variety
of network architectures and found the best to be a SegNet structure with a bottleneck of
25 × 25. The loss curves of this model and a similar less deep version with a bottleneck of
50 × 50 can be seen in Figure 32. It is normal to see the model performing better on the
training data compared to validation and test set, since the model parameters is adjusted
according to the training set. But in this case we see a rather large deviation, which would
indicate that our models are overﬁtting the training data. This means that our models
are adapting so much to the speciﬁc training data, that it is not able to recognize patterns
that are shared across all data (training, validation and test). A potential way to ﬁx this
could be through more regularization or more training data. In our SegNet25 network
we have a total 195,025 trainable parameters compared to the approximately 50,000 data
points, and it would therefore be preferred if we included more data.
However we are
limited both on new data and space to store it. Alternatively we could resample the saved
data several times to reach a total of data points above our number of parameters. The
downside hereof is that we would be recycling the same data several times (even more for
the minority classes we oversampled). Therefore more regularization is perhaps a better
solution for our problem. For training both SegNet50 and SegNet25 we used a batch size
of 128 during training, but as presented in Section 4.7.2, using a smaller batch size could
help regularize the model. Figure 50 actually shows the loss curves from models trained
with a batch size of 16 only, but the results indicate that the best model is found very
early during training, and then doesn’t improve substantially for validation or test. Then
we could have gone further in depth with the total variation, and tested diﬀerent values of
λ to add more regularization. In second generation of networks we tested λ = 0.1, which
didn’t have any signiﬁcant eﬀect on the results. At last we could change the percentage
of dropouts we use per block, as 20% might not be enough in a network with this many
parameters compared to the training data. These are just the regularization methods we
have tried out, but there are of course many other ways to regularize. More data could for
example be introduced by augmentation of preexisting data such as rotation, or data could
be split up into smaller cut-outs.
In the part about binary segmentation we tried many diﬀerent network structures,
which were omitted at later generations. These could potentially have produced better
results had we just saved the best model from each training session, trained for longer or
used the improved sampled data seen later in the process. As a deep learning problem can
be complex, it is likely that the deeper networks seen in second generation would have per-
formed a lot better than what we ultimately saw under the aforementioned circumstances.
But as we have just discussed, with a deeper network comes more parameters, and we
would therefore need more regularization or more data (or both) if we were to make use of
the networks with two or three convolutions per block.
70
==PAGE:81==
A recurring problem in both the binary and multilabel classiﬁcation is the prediction of
subwaths. If we use our binary predictions as a third input in the multilabel classiﬁcation,
it is essential that our model knows to classify these areas over water as water. To solve
this problem we could use soft-max classiﬁcation like we did in the multilabel section, as
this proved to do a good job of eliminating the subswaths (see Section 6.2). Unfortunately
we also saw that the soft-max models had a tendency to predict water more often in areas
of ice, but with a binary training set, we might see better results.
The sigmoid activated network output from SegNet25 still had some noise, which we
tried to ﬁx by using MRF. We generally saw improvement in all parameters when doing
this, but often we would end up with large squares in the segmentations. This is likely the
result of a high β value, where it would be least costly to have an entire cut-out belong to
one class. Though this overall helps improve the F1 score and balanced accuracy, we would
also see larger areas with a lot of noise being misclassiﬁed. This eﬀect is most clearly seen
when our model predicts the subswath over water as being ice. To meet this problem we
could use a lower value of β, as all tested values improved the overall balanced accuracy.
When visualize larger scenes we extract a cut-out, feed it through a model, denoise it using
MRF with grap-cuts and then put it back where we extracted it. If we ﬁrst applied the
MRF denoising after patching the predicted scene together, we would avoid seeing these
squares in our scene visualizations. However, this takes a very long time for low β values,
and might take hours for a single scene if we use β = 40. Alternatively, we could ﬁrst
distribute each pixel to its closest class mean like with the one-clique potentials. Hereafter
we could apply a morphological ﬁlter, e.g. a majority vote ﬁlter, to ”patch up” where the
noise isn’t too dominant.
7.2
Multilabel Classiﬁcation
With the binary prediction from the SegNet25 network, we tried out two diﬀerent methods
for multilabel classiﬁcation. For each method we had a model, where we didn’t include the
binary predictions as a third input and then three models where we tested the third input
on diﬀerent scales (0/1, −1/1 and 0/100). When we look at the visual representations
given by the scenes in Section 6.2, we clearly see that the inclusion of the third input has a
visual impact. This is especially clear in Scene 6 and 7, where the squares from the binary
segmentations are visible in both the RMSE and Soft-max models. For both methods this
has improved each respective models ability to distinguish values below and above 10%
SIC. However, when we investigated each models performance on the individual classes
(see Figure 51 and 52), we found that not a single pixel in the test set had a true SIC of
5% or 25%, and since the 5% is half of the water class when binarized, this could give a
false impression of the performance of each model.
In Figure 51 and 52 we also see that the soft-max models has a larger error than the
RMSE models, which was apparent already in Table 9. The Soft-max models misclassify
71
==PAGE:82==
with up to two classes more than the RMSE models, which is very likely contributed to
the way we predict. With the RMSE models we have a continuous function able to pre-
dict in between the ﬁxed classes, after which it is discretized by rounding to nearest class,
allowing for more freedom of uncertainty for the model. If it couldn’t decide between two
neighbouring classes, then the RMSE models have the luxury of predicting a value between
the two classes, giving a lower error. The soft-max model predict one of 21 classes, which
are separated by an interval of 5. Therefore if the Soft-max model is uncertain of a class,
it would instead misclassify with a starting point of 5% error.
With both the RMSE and soft-max models we predict one of 21 classes per pixel, which
are values from 0% to 100% separated with an interval of 5%. We can thus describe the
pixels as being variables on an interval scale as there is an ordering of the data, meaning
that the class 0% is lower than 5%, which again is lower than 10%, and we could keep
going all the way up to 100% [47]. However, when our soft-max models predict the SIC
of a pixel, we’re given the probability of it belonging to a certain class. We then pick the
class deemed most probable, which would ignore the principle of the interval scale, as each
class is instead treated as a category.
As a model structure we simply copied the same architecture found in the SegNet25
model we used for binary classiﬁcation. Though it overall seems to work ﬁne, multilabel
classiﬁcation is a more complex tasks than binary classiﬁcation. It would therefore seem
valid to suggest that a more extensive model would produce better results, as having more
neurons would mean more parameters to train. Our model might then be better at rec-
ognizing some of the more complex patterns in the input data needed for a multilabel
classiﬁcation. The SegNet25 model was chosen as it was already established to perform
better at binary segmentation, and we wanted the multilabel segmentation to focus more
on testing whether involving a third input in the models would improve the results seen.
In Table 9 we saw that the RMSE regression models didn’t beneﬁt from the having the
third input, which all had an overall worse performance, except for when the predictions
were binarized. This is likely due to the extra input helping to lower the eﬀect of the
subswaths over water. For the Soft-max model we only see improvements after including a
third input. This goes for both ERMSE and the binary classiﬁcation metrics. In Section 6
on Results, we could establish, that models with a lower recall would predict ice as water
more often. This is seen very clearly for the soft-max model with only two inputs in both
Figure 45 and 47, where the information in the HH and HV channels were rather sparse,
and as a result we see large areas of ice predicted as water. This problem is dampened
when we include the third input, as the binary segmentation seemingly serves as a guide
to the model for where we see ice and water. Therefore we also see these square holes in
the ice areas originating in the predicted binary segmentation. The RMSE models handles
the sparse information from the input data better, but when we introduce the third input
with large square holes, it forces our models to wrongfully predict lower SIC within the
ice, thus increasing the ERMSE. The eﬀects of the diﬀerent scales can be rather diﬃcult
72
==PAGE:83==
to determine when just looking the RMSE models.
We see a lot of the patterns from
the original HH and HV channels in the predicted scenes, and the results come oﬀrather
arbitrary. On the soft-max models we can more easily distinguish the eﬀects. For the
0/1 and −1/1 Soft-max models we see some of the original noisy patterns from the input
data, whereas for the 0/100 model we usually have very uniform segmentations. We also
notice that the soft-max models rarely predicts low SIC near the ice edges. This goes for
both the models with two and three inputs. This is yet another point where our RMSE
models proves better than than the soft-max models. In the soft-max models, where we
use the binary segmentation, we generally only see high ice concentrations. So if our binary
prediction input forces the model to predict high SIC values we have another source of the
high ERMSE values seen in Table 9.
8
Conclusion
In this project we have been using CNN’s with diﬀerent architectures to see if we could
predict the location of water and ice around the shores of Greenland, as well as the concen-
trations of the found ice. We had data from the Sentinel-1 mission at our disposal for this
project, where we used the HH and HV channels. This was a project split into two major
parts. First we investigated what network structures would be the best for working with
the given satellite data. We designed several CNN’s for binary classiﬁcation using a sigmoid
activation of the network output and the BCE loss function. The results showed us that
simpler networks produced the best binary segmentations of our satellite data. We also saw
that increasing the receptive ﬁeld of our models would help improve the performance of our
models, as long as these ﬁelds didn’t get too large. We ultimately ended up choosing the
SegNet25 model to perform our ﬁnal binary segmentations to be used in second part of the
project. These predictions were improved using MRF with graph-cut optimization, which
decreased the noise of the segmentations. As an early alternative to this post-processing
of data, we tried implementing total variation in our loss function, but the value of the
regularization parameter made this method impactless on the segmentations.
In the second part we checked whether using the binary segmentations, which we pre-
dicted in the ﬁrst part, would improve the performance of our CNN’s. This was done by
training both a model with this third input and without. Thus we would have a model,
which only took in the HH and HV channels as input data, and a model which also took
in the binary segmentation along with the other two inputs. To see if the results achieved
from doing this would just be arbitrarily generated or if there was actually some connection
between the scale used on the binary predictions, we tested three diﬀerent scales. 0/1 was
the original output produced, −1/1 corresponded to the scaling of the satellite data and
0/100 to see what eﬀects drastically extending the scaling would have. This was tested for
both RMSE regression models and soft-max regression. For the RMSE models we saw that
including the third input would worsen the overall ERMSE of all three cases, thus the best
performing model only used the satellite data as input. The soft-max models generally
performed worse, as the ERMSE for the entire data set as well as the individual classes
73
==PAGE:84==
was signiﬁcantly higher than the RMSE models. However, when the predicted multilabel
segmentations was binarized we saw that including the third input helped both methods
distinguish between water (< 10%) and ice (≥10%). The RMSE models had the highest
F1 scores, telling us that they were generally better at predicting the ice class, whereas
the soft-max models had higher balanced accuracies, meaning they where better at bal-
ancing the prediction of both water and ice. If we wanted to produce a proper multilabel
segmentation of our scenes, we would get the best results from one of the RMSE models.
However, if we wanted to produce a proper distinction between water and ice, one of the
soft-max models with a third input, would give the best results. The RMSE model with
the 0/1 scaled binary segmentation input would give us the best balance between both of
these criteria.
Though we saw some rather promising results for both the binary and multilabel clas-
siﬁcations, there is still room for improvement.
It would be worth to return and look
into some of the earlier networks used. Here we’re thinking about both the CNN’s with
more convolutions per block from the second generation, but also the abandoned U-Net
architecture brieﬂy used in the ﬁrst generation. In both the binary and multilabel we saw
indications of training data overﬁt. To ﬁx this we should therefore include more data and
regularization in future experiments. More data could be achieved by ﬁrst of all sampling
more cut-outs from each chunk. We could also use data augmentation such as rotation of
cut-outs or we could include data from other areas of the world, for example the Gulf of
Saint Lawrence, which was used in the article that inspired this project.

ASIP 000 asip functions
Contains all the functions that are used repeatedly throughout the code in other scripts.
These functions include:
• mrf graph cut: With the probability map produced by our binary classiﬁers, class
means from our training data and a chosen β values, we use Vladimir Kolmogorov’s
Graph-cut algorithm to produce uniform segmentations.
• conf mat bi & conf mat multi: Produces the confusion matrix for binary and
multilabel segmentations respectively using the binary predictions and the binarized
ice charts.
• accuracy bi, balanced accuracy bi, IoU bi, dice bi & f1 bi: Computes accu-
racy, balanced accuracy, intersection over union, dice coeﬃcient and F1 score respec-
tively using the confusion matrix obtained using conf mat bi.
• sizeof fmt: Used to display the size of all variables in memory. Code supplied by
Fred Cirera in a stackoverflow.com thread. Link found in code.
• print time: With a time span given in seconds this function will print time given
in hours, minutes and second. Used to keep track of run time during training.
• stats: Prints type, maximum and minimum value, and shape (dimensions) of either
numpy array or PyTorch tensor.
• color bar: Produces nice color bars for our plots.
• ﬁnd closest: Rounds array to nearest value which is a multiple of a given value. This
is used when we binarize data or when we discretize our RMSE model predictions by
rounding to nearest 5.
• compute receptive ﬁeld Takes in operational parameters for e.g. convolutions or
max-pooling as well as the input data dimensions and compute the development of
receptive ﬁeld, jump and feature map for each operation. For more information see
section 4.9 on receptive ﬁelds. Given the shown parameters the function will print
the following output when prints=True.

• norm shape & sliding window: Functions provided by David Malmgren-Hansen
