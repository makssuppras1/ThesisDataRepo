Abstract
Central Oﬃce Re-architected as a Datacenter (CORD) is a platform that provides a virtual-
ized Central Oﬃce (CO) built upon existing work on Software-Deﬁned Networking (SDN),
Network Function Virtualization (NFV) and Cloud technologies such as ONOS, OpenStack
and XOS. However, despite its ﬂexibility when compared to legacy (i.e., non-virtualized) de-
ployments, a CORD-based telco provider is still limited to the existing CORD software (e.g.,
OpenStack and ONOS) and cannot swap to other Virtual Infrastructure Managers (VIMs).
This thesis proposes a way to accommodate multiple VIMs in CORD. To this end, we ﬁrst
identify and analyze the existing interactions from XOS towards OpenStack and ONOS both
during the installation of CORD and during its runtime processes. Then, we deﬁne a VIM
adaptation layer for CORD that translates proposed VIM-generic calls to the identiﬁed VIM-
speciﬁc calls. Finally, we also propose implementation details on how to accommodate this
layer into CORD.

Introduction
1.1
Problem Statement
Central Ofﬁces (COs) are the edge infrastructure of telco providers. They consist of dedicated
and vendor-speciﬁc hardware that provide customers with Internet access as well as other ser-
vices. Central Ofﬁce Re-architected as a Datacenter (CORD) is a platform that provides a
virtualized CO, built upon Software-Deﬁned Networking (SDN), Network Function Virtualiza-
tion (NFV) and Cloud principles, to achieve better scalability, cost-effectiveness ratio and time
to market.
However, despite its ﬂexibility when compared with legacy (i.e., non-virtualized) deploy-
ments, CORD still has a relatively rigid architecture. The three main building blocks of CORD
(XOS [1], OpenStack [2] and ONOS [3]) are logically separated but their interactions are inter-
woven in the CORD processes. This implies that despite the plethora of other available products
such as Kubernetes [4] instead of OpenStack or OpenDayLight [5] instead of ONOS, the ad-
ministrator of a CORD deployment cannot swap amongst these products. Therefore, developers
are forced to reprogram their applications and services to make them work on top of OpenStack
or ONOS in order to accommodate them to CORD, which could be a problem for the success
of CORD in the near future.
1.2
Objectives
The objective of this thesis is twofold: to identify the existing interactions from XOS towards
OpenStack and ONOS both during the installation of CORD and during its runtime processes,
and to deﬁne a virtual infrastructure manager (VIM) adaptation layer that translates generic
calls to the identiﬁed technology-speciﬁc calls. First, these interactions should be documented
not only in terms of message exchanges but also with regards to the scripts/classes/methods that
generate them. This should yield an understanding of how OpenStack and ONOS are integrated
into CORD, as well as how they could be decoupled from it. After that, the identiﬁed interac-
tions should be deﬁned as generic functions, with generic names and parameters independent
of the VIM that will call them, to compose the VIM adaptation layer. This layer should allow
CORD to replace OpenStack/ONOS with a similar technology without major changes in the
CORD system. Finally, even though the actual implementation of the VIM adaptation layer is
out of the scope of this thesis, some guidelines on how this layer could be implemented are also
provided.

The remaining parts of this report are organized as follows:
• Chapter 2 provides a broad description of what CORD is. It starts with an overview that

ONOS. The purpose of this section is to identify all the processes that OpenStack and
ONOS go through in order to teach an operator how to deploy its speciﬁc software in
CORD.
• Chapter 4 analyzes the interactions between XOS and OpenStack/ONOS during the run-
time processes. To this end, we ﬁrst provide an overview of the internal XOS processes
and then we describe each of the calls between XOS and OpenStack/ONOS, analyzing for
each call what it does and how it does it.
• Chapter 5 deﬁnes the VIM adaptation layer and discusses implementation details of this
interface.
• Chapter 6 presents how the initial planning of this thesis has changed, a summary of the
outcome of this thesis, and future work.

2.1
Overview
Traditional Internet service providers (ISPs) give their customers access to the Internet through
the Telco Central Ofﬁce (CO). Each ISP usually has thousands of COs placed at the edge of their
network, making them a signiﬁcant source of capital expenditure (CAPEX) and operational
expenditure (OPEX). A CO consists of a set of speciﬁc-purpose proprietary hardware assembled
over the last decades without a uniﬁed architecture. This legacy equipment limits innovation
due to a lack of programmability, hardening the creation of new services and the management
of existing ones.
Central Ofﬁce Re-architected as a Datacenter (CORD), a collaborative effort between Open
Networking Lab and other industry partners, proposes a new architecture for the CO that deals
with the aforementioned problems by replacing the proprietary hardware with software running
on a few commodity servers and white-box switches. This allows the ISPs to beneﬁt from
economies of scale, which signiﬁcantly reduces CAPEX/OPEX. However, CORD’s ﬁnal goal
is not only to replace the current hardware devices with their virtualized counterparts, but also
to make the COs a central element in the ISPs’ cloud strategy. To this end, CORD builds upon
existing work on SDN, NFV and Cloud technologies to leverage the programmability at the
edge of the network, increasing the ability to rapidly deploy and elastically scale services.
As mentioned in the introduction, the main building blocks of CORD are XOS, OpenStack
and ONOS. XOS is a Cloud operating system (OS) that deﬁnes a coherent framework for com-
bining SDN, NFV, and Cloud services, using Everything-as-a-Service (XaaS) as its organizing
principle. OpenStack manages large pools of compute, storage, and networking resources on a
cluster of commodity servers, and is responsible for creating and provisioning virtual machines
(VMs) and virtual networks (VNs). Finally, ONOS is the network operating system of CORD
and its functionality is twofold: to manage the underlying switching fabric and to host ﬁrst-class
CORD services.
CORD leverages these open-source technologies, enabling the creation of (1) control plane
services, such as content-centric networking or virtual networks on demand, (2) data plane
services, such as parental control or network address translation (NAT), and (3) global cloud
services, such as content delivery network (CDN) or Internet-of-Things (IoT). CORD aims
to offer this kind of services to residential, enterprise and mobile customers, hence without
restrictions on the type of access network.
In short, CORD aims to outpace traditional COs by following a two-step process: to disag-
gregate and virtualize today’s speciﬁc-purpose hardware, and to provide a framework where the

The CPE corresponds to the home router, and the optical network unit (ONU) is the re-
sponsible for converting the optical signals into electrical signals, and vice versa. For the users
that have ﬁber-to-the-home (FTTH), the ONU is inside the home router; otherwise, the ONU is
shared with other users and is located out of the user’s premises (e.g., in ﬁber-to-the-building
(FTTB) architectures). A part from the essential functions of a router, a CPE includes other
functions such as a Dynamic Host Conﬁguration Protocol (DHCP) server, NAT, and optional
services such as a ﬁrewall and parental control. However, the capabilities of the CPE are con-
strained by its hardware limitations. To overcome this issue, CORD proposes to virtualize and
move the CPE to the cloud, allowing for new value-added services and customer-care capabil-
ities. The virtualized version of the CPE is called virtual subscriber gateway (vSG) and runs
inside an OpenStack VM, created for each subscriber, on a commodity server. There is still a
device in the homes, but it just keeps the routing functionalities.
The set of optical links after the ONU, which give connectivity to tens of thousands of sub-
scribers, are aggregated in the optical line terminal (OLT), located at the CO. An OLT consists
of a set of racks of propietary hardware and, thus, implies a large capital investment. CORD
proposes a virtualized version of OLT, called virtual OLT (vOLT), which runs on top of ONOS
and implements the same functionalities contained in a legacy OLT. For instance, vOLT imple-
ments authentication on a per-subscriber basis, manages a virtual local area network (VLAN)
that connects the subscriber’s devices to the CO switching fabric, and manages other control
plane functions of the OLT.

ferent components of the CO, do not have a speciﬁc virtualized counterpart running on top
of ONOS or OpenStack. Instead, they are just replaced with white-box switches managed by
ONOS that will connect the virtualized components of CORD.
A BNG provides the means through which subscribers connect to the public Internet, as-
signing an IP address per subscriber and managing features such as quality of service (QoS),
policing, virtual private networks (VPNs), etc. CORD proposes a virtualized version of the
BNG called virtual router (vRouter), which is implemented as an ONOS application. vRouter
does not necessarily implement all the functionalities of a BNG, since some of them are imple-
mented either by the vOLT or the vSG, but it can be seen as a private virtual router.
Summing up, the subscriber’s packet ﬂow through CORD would work as follows. First,
when the subscriber powers up the home router, it sends an authentication message, which
is passed through ONOS to the vOLT application. The vOLT application authenticates the
subscriber, assigns VLAN tags to it, and installs the appropriate ﬂow rules in the switching
fabric. Then, it ﬁres up a VM and binds it to the VLAN. Finally, vSG requests an IP address
from vRouter, which installs the necessary ﬂow rules to route packets from and to the VM. Once
everything is set up, packets ﬂow over a VLAN from the home router to the subscriber’s VM,
where packets are processed according to the subscriber’s contracted bundle, and are eventually
forwarded to the Internet using the assigned IP address.

Docker
Docker is a platform that allows developers to develop, deploy and run applications with con-
tainers. While a VM runs a full operating system with virtual access to host resources through a
hypervisor, a container shares the kernel of the host machine with other containers. In this way,
a container is more lightweight than a VM, and still offers enough isolation so that applications
installed in different containers can have their software dependencies installed independently.
In CORD, Docker is responsible for deploying the different components of XOS, as well as
provisioning the containerized compute instances.

2.3.3
ONOS
ONOS stands for Open Network Operating System and plays two roles in CORD: to manage
the physical and virtual connectivity and to host ﬁrst-class CORD services. The functionality
of ONOS in CORD is split into two ONOS instances, which are not mapped to the previous
two roles:
• onos-cord: it is responsible for the virtual networking, service composition and access in-
frastructure, i.e., the overlay infrastructure. It hosts the Virtual Tenant Network (VTN) and
vOLT services, which provide service composition and Access-as-a-Service (ACCaaS) ca-
pabilities, respectively.
• onos-fabric: it is responsible for conﬁguring the fabric switches, i.e., the underlay in-
frastructure. It hosts the vRouter service, which provides Internet-as-a-Service (INTaaS)
capabilities.
2.3.4
XOS
XOS is designed as an operating system for the cloud. While everything is a ﬁle in a computer
OS such as Linux, everything is a service in XOS. XOS provides a framework for assembling
and composing these services. The core of XOS consists of a set of models that deﬁne the
desired state of the underlying system. In order to add a new service to XOS, a model or set of
models that characterize this service based on a list of parameters have to be deﬁned. A service
is created in XOS when the models that deﬁne it are instantiated, that is, speciﬁc values are
given to the parameters of these models. XOS provides two ways of giving these values from
the point of view of a CORD administrator: through a graphical user interface (GUI) or through
a TOSCA [7] recipe. These model instances are just an abstraction of what the service is, so
the aforementioned parameters have to be translated into some real conﬁguration in the actual
system. This is the responsibility of what XOS calls synchronizers, which communicate with
OpenStack, ONOS and other services, sending them conﬁguration values generated from the
instances’ parameters of the service models.
XOS is divided into the set of Docker containers depicted in Figure 2.4. Each of them have a
speciﬁc role in the process explained above. The three containers on the top represent the XOS
northbound interface used to create, update or delete model instances:
• xos-tosca: contains the TOSCA engine, which takes as input a TOSCA recipe and sends
the parameters deﬁned in it to xos_core over gRPC [8].
• xos-gui: provides a graphical interface so that a system administrator can easily manage
the services. When a change is made in the models, a REST API message is sent to
xos-chameleon.
• xos-chameleon: translates REST API message received from the GUI to gRPC and for-
wards the message to the xos-core.
The four containers in the middle layer manage the data model.
• xos-db: PostgreSQL [9] database that stores the model instances.

Deployment
CORD can be deployed in two ways, using either a physical or a virtual deployment. A physical
deployment is usually referred as a POD, which consists of three servers and four switches.
There is a similar deployment, called Half POD, which requires less hardware but is insufﬁcient
to test all the features of CORD. The virtual deployment, often referred as Virtual POD or

CORD in a Box (CiaB), is the equivalent of a POD, where the servers are VMs and the switches
are virtualized in a single physical host.
The CORD deployment used for the development of this thesis is a CiaB based on CORD
4.0, which comprises the following Vagrant [14] VM nodes:
• Head node: it hosts the XOS Docker containers, including the synchronizers, the Open-
Stack LXD containers and the two instances of ONOS. It also hosts Metal as a Service
(MaaS) [15] components.
• Compute node: it hosts the service instances on behalf of each subscriber. These instances
run on VMs, containers or containers nested inside VMs. In a CORD deployment there
can be serveral compute nodes; however, in our deployment there is just one compute

• Development node: it is used during the installation process to reconﬁgure and recompile
the source tree of CORD in the head node. In a virtual deployment (i.e., CiaB) it is
unnecessary since the physical host that hosts the virtualized nodes could be used for this
purpose. However, it is kept in the virtualized deployment in order to have the equivalent
nodes of a physical deployment.
2.5
Service composition
This section describes the three parts that compose a service in CORD. Two of them have al-
ready been introduced and are the models and the synchronizers, which are used to characterize
a service and to conﬁgure the underlying system, respectively. The third component consists of
the security policies, which manage the access to the models.
2.5.1
Models
The data model represents the abstract state of the system. It consists of a set of models, which
are composed by a set of data ﬁelds and a set of model relations ﬁelds. The former are typically
string, integer or boolean variables that describe the model itself, while the latter describe how
different models are linked to each other. These ﬁelds can be further classiﬁed into two types
depending on the state they hold:
• Declarative state: data or model relations ﬁelds that are provided through the GUI or a
TOSCA recipe to conﬁgure and control the underlying component. This state is enough
to recreate the operational state of the system, i.e. the actual conﬁguration of the system,
with the help of synchronizers.
• Feedback state: data ﬁelds that hold the operational state of the underlying component,
recording the result of the actions of a synchronizer. They are typically used to hold the
ids of the underlying components given by the underlying software.
The data model is deﬁned as an xproto ﬁle. xproto is a language for deﬁning models that
extends the functionality of Google protobufs [16]. The data model is stored as a relational
database (PostgreSQL) in the xos-db container. This database contains a table for each model,
where each row of a table is an instance of the corresponding model and may link to tables
of related models. XOS converts this database into a set of Python classes using the Django
framework. These classes are loaded to the xos-core container, and synchronizers use them to
access and modify the model instances.

CORD is based on a set of core models deﬁned in the core.xproto ﬁle. The core models
related to the deployment of a new service and the key bindings among them are explained
hereafter.
• Service: represents an elastically scalable, multi-tenant service and supports both server-
based services (e.g., a VNF running in VMs) and switch-based services (e.g., an SDN
application).
• ServiceInstance: represents an instance of a service1 on behalf of a particular tenant
(hereafter tenant instance)2. It is linked to a Service, and it is used to represent the speciﬁc
behavior of this service for a particular tenant.
• TenantWithContainer: this model extends the ServiceInstance model so that it bounds a
tenant instance of a service to an Instance.
• Slice: it is linked to a Service and it represents the compute and network resources used
by this Service. It is bound to a set of Instances and Networks, and it deﬁnes a default
Flavor and Image for its Instances.
• Instance: represents a compute resource associated with a Slice. It can deﬁne a speciﬁc
Flavor and Image. Moreover, it can be of three types depending on its isolation, namely a
Kernel-based Virtual Machine (KVM), a Docker container or a Docker container inside a
KVM. These resources correspond to the compute node.
• Flavor: represents the physical resources, such as the disk space, the memory or the
number of cores allocated to an Instance.
• Image: represents the content that boots in an Instance.
• Network: represents a virtual network that interconnects Instances with each other. The
speciﬁc conﬁguration of a Network is deﬁned using the NetworkTemplate model, which
allows to deﬁne parameters such as the visibility (public or private) and the translation
(none or using a NAT), among others.
To sum up, in order to add a new service to CORD, a new model that extends the Service
model has to be deﬁned. This new model will provide the general behavior of the service for
all the tenants. Therefore, CORD will only hold one instance of this model. In case that part
of the behavior of the new service is speciﬁc for each tenant, a new model that extends either
ServiceInstance or TenantWithContainer is needed. In this case, CORD will hold an instance
of this new model for each tenant that uses this service. However, only if this instance extends
TenantWithContainer, the tenant service will be hosted in an Instance, i.e., as a VM or container
in the compute node.
CORD also supports the creation of cloud applications based on microservices, which con-
sists of dividing an application into independent service blocks. These blocks are interconnected
forming the so-called Service Graph. This allows different subscribers to traverse this Service
Graph based on their customer proﬁle, i.e., choosing speciﬁc services. To handle the intercon-
nection of Services and tenant-speciﬁc ServiceInstances, CORD uses the following models:
• ServiceDependency: represents a relation between two Services, the provider and the
subscriber. The provider performs a task and provides its output to the subscriber, which
performs his task using the given input. A subscriber can be a provider to another service,
and vice-versa, forming a chain of services. The ServiceDependency model represents the
edges of the Service Graph, whereas the Services represent the vertices.

2.5.2
Security Policies
A Security Policy is a boolean expression that is typically associated with a model. In XOS,
these policies are deﬁned in the same xproto ﬁle as the data model and their purpose is to
determine who can do what. The who refers to the user or context that tries to access something.
This ’something’ (i.e., the what) refers to the requested information, i.e., a model, an object of
this model, or a ﬁeld within this object. Finally, the do is the purpose of the requested access
(e.g., read or write).
2.5.3
Synchronizers
Every service in CORD has a synchronizer, which is responsible for mapping the declarative
state of this service into the actual operational state. In other words, it monitors the instances
of the XOS models that deﬁne the corresponding service and it applies the changes to the
underlying system (OpenStack, ONOS...) if needed.
A synchronizer drives the system towards the state deﬁned by the model instances regardless
of the changes that led to that state. This property requires the synchronization functions to be
idempotent, which means that the resulting system state is the same no matter how many times
the actions are applied as long as they are based on the same parameters.
A synchronizer is divided into a set of steps and model policies, each of them focusing on
a model or a set of models used by the service. Depending on how the step interacts with the
data model, it can be classiﬁed in two types: actuator and watcher. These two step types along
with the model policies represent the three actors in a synchronizer and are explained below:
• Actuator: it reads the parameters of the instances of a particular model and idempotently
applies them in the actual service. Moreover, it can also apply feedback state to the model
instance that it is actuating on. A model can only be actuated by one step. If the actuated
model has dependencies with other models, the changes to these models are applied in a
consistent order.
• Watcher: it subscribes to a data model to gather information of this model when there
is a change on it but it does not actuate its state in the system substrate. A watcher for a
given model is an actuator for another model, and the actuator part is the one that applies
the changes to the actual service. The watcher cannot apply feedback state to the watched
model either.
• Model policy: it manages relationships between data models so that for a given model in-
stance it can create other model instances that complement it. For example, the OpenStack
policy that manages the Network model makes sure that every instance of this model has
an associated interface (in this case, a ControllerNetwork instance), and if it does not exist,
it is created. Therefore, a model policy can write declarative state in a model instance as it
can manually be done by a CORD administrator from a TOSCA recipe or from the XOS
GUI.

Installation
In this chapter, the installation of CORD (CiaB 4.0) is analyzed, with emphasis on the XOS,
OpenStack and ONOS deployments. Not all the installation and conﬁguration ﬁles and pro-
cesses are described in detail, but only the main installation blocks and their connection with
OpenStack and ONOS. This is because the objective is not to understand all the internal instal-
lation processes but to understand how OpenStack and ONOS are deployed in CORD, in order
to be able to replicate this deployment for other software in the future, using this chapter as a
guide.
In order to install CORD, the ﬁrst step is to bootstrap its source tree, which consists of
the necessary ﬁles to deploy any use case of CORD. Then, two make commands have to be
executed, one for deﬁning the use case (e.g., R-CORD) and generating the corresponding con-
ﬁguration ﬁles, and the other for building the different components of CORD with the given
setup.
Make is a build automation tool that executes targets from a so-called Makeﬁle. In
CORD, the Makefile (located at cord/build) contains targets that mainly execute Ansible
[17] playbooks. Ansible is a framework that can deploy and conﬁgure IT systems. In CORD, it
drives most of the installation processes by running shell commands and Ansible modules in a
predeﬁned manner.
As mentioned before, the ﬁrst step of the installation process is the conﬁguration of the use
case, which is deﬁned using the following make command:
make PODCONFIG=<POD_config_file> config
In our case, since we have used a virtual R-CORD deployment, the POD conﬁguration ﬁle is
rcord-virtual.yml, located at cord/build/podconfig. This make command runs the Ansi-
ble playbook genconfig.yml located at cord/build/ansible, which generates conﬁguration
ﬁles in the folder cord/build/genconfig for the chosen use case. These ﬁles are described
below1:
• config.yml: YAML ﬁle that contains a set of Ansible variables to be used by the Ansible
playbooks.
• config.mk: Makeﬁle that contains a set of scenario-speciﬁc conﬁguration variables and
speciﬁes the prerequisites for some blocks (a.k.a. targets) of the Makefile.
• inventory.ini: ﬁle that deﬁnes groups of hosts where the Ansible playbooks are run.
After these conﬁguration ﬁles are created, the Makefile is able to build CORD with the chosen
use case. It is done through the following make command:
make -j4 build

Based on the obtained steps and the conﬁguration of the synchronizer, up to three threads
can be created: an observer thread, a watcher thread and a model policies thread. Each of these
threads instantiate a particular Python class, located at the new_base directory, and they are
explained in the following subsections.
4.1.1
Observer
This thread is created if steps exist in the steps directory. It instantiates the XOSObserver class,
and runs its run method. This thread is responsible for creating, updating or deleting the actual
conﬁguration of a service to match the state of the service’s model instances present in XOS as
well as updating the feedback state ﬁelds of these model instances.
The code of this thread is run every 5 seconds and is known as the event loop. Every
execution consists of two iterations, one for object synchronization and one for object deletion.
The ﬁrst step of the code is to call its fetch_pending function, which starts analyzing the
synchronizer’s steps received from the Backend class. If a step implements the function call,
this function is called and the event loop continues with the next iteration. This is the case of
legacy steps and they are almost deprecated in the current OpenStack synchronizer. For the
other steps, the logic is fairly more complex. First, for each step, its fetch_pending is called,
which will collect the objects that have to be synchronized (a.k.a. DIRTY objects) or deleted
(a.k.a. DELETED objects), depending on the iteration, for a particular model (the one being
observed). If a step does not overwrite this function, the one deﬁned in SyncStep is used. The
description of this function is provided later in this subsection.
The objects returned by fetch_pending are grouped into dependent cohorts. A cohort is
a sorted list of objects whose models are linked in the model dependency graph. In order to
generate this sorted list, the model dependency graph is ﬁrst topologically sorted. In a directed
graph, such as the model dependency graph, a topological sort consists of listing the nodes
of a graph in a way that for every node u connected to node v (u →v), u comes before v in
the list. Finally, the cohort is created from the returned topological sort in reversed order, and
the cohort objects are synchronized in the cohort order. This order is necessary because when
new instances of a model are created in XOS, the changes in the actual conﬁguration of the
existent services have to be done in a speciﬁc manner. For example, if the synchronizer realizes
it has to synchronize an Image and an Instance that links to this Image, it will synchronize
the Image before. This is because the Image is part of the declarative state of the Instance;
therefore, the Instance cannot be created without specifying an Image. The deletion process is
performed in the opposite direction for similar reasons. For example, if an Image associated to
an Instance was deleted ﬁrst, the Instance would be incomplete and could raise errors. If the
28
==PAGE:47==

Instance is deleted ﬁrst, the Image is not affected. In order to synchronize the cohort objects,
the event loop calls its sync_record or delete_record method, depending on the iteration,
in the speciﬁed order.
On the one hand, the sync_record function aims to apply the state of the objects returned
in the fetch_pending to the underlying system. The sync_record function of the Observer
thread basically calls the sync_record function implemented in each step. If a step does not
overwrite this function, the one deﬁned in SyncStep or SyncStepUsingAnsible, depending
on the step’s superclass, is called. The description of this function is provided for both su-
perclasses later in this subsection. On the other hand, the delete_record function aims to
remove the corresponding component of the underlying system for each object returned in the
fetch_pending. In the same way than sync_record, it basically calls the delete_record
function implemented in each step. Also, if a step does not overwrite this function, the one de-
ﬁned in SyncStep or SyncStepUsingAnsible is called. The implementation of the functions
of these two classes is described below. The description of these functions is also provided for
the steps that overwrite them, both for OpenStack and ONOS, in sections 4.2 to 4.4.
SyncStep
• fetch_pending: as mentioned before, it returns the objects of a particular model to be
synchronized or deleted, depending on whether the passed parameter deleted is set to
False or True, respectively. On the one hand, the objects to be synchronized are the ones
in which the value of the enacted ﬁeld is smaller than the updated ﬁeld or set to None.
Note that when a new object is created, enacted = None and updated = NOW (current
timestamp), so it will be returned for synchronization. On the other hand, the objects to
be deleted are the ones whose deleted ﬁeld is set to True. This parameter is by default
set to False, but is modiﬁed either through the GUI or TOSCA recipe when a system
administrator marks an object for deletion. This logic is explained in detail in subsection
4.1.4.
• sync_record: this function assumes that the step is going to use an Ansible play-
book to apply the changes to the underlying substrate.
To this end, it ﬁrst calls the
map_sync_inputs function of the step, which has to be provided by the user. Its pur-
pose is to obtain the values of the variables used in the Ansible playbook. Then, it calls
the function run_template, which runs, in the head node, the playbook deﬁned in the
self.playbook attribute of the step class by passing ﬁrst the previously obtained vari-
ables as a parameter. Finally, if the step implements the function map_sync_outputs, it
is called and the output of the execution of the Ansible playbook is passed as a parame-
ter. This function is used to write the feedback state of the given object in the XOS Data
Model, if necessary.
If a step needs a behavior different than the default one (e.g., it cannot be implemented

4.1.2
Watcher
This thread is created if steps exist in the steps directory and enable_watchers is set to True
in the conﬁguration ﬁle, since it is set to False in the defaults ﬁle. Neither the OpenStack
synchronizer nor the ONOS synchronizers overwrite this parameter in their conﬁguration ﬁles;
therefore, this thread is not executed in these synchronizers. Anyway, we provide a basic de-
scription of what this thread does if started.
First of all, the class XOSWatcher is instantiated, generating a dictionary called watch_map
that contains the information of the watched models of the synchronizer, deﬁned in the watches
attribute of the synchronizer steps. Then, a Redis pubsub object is created and subscribed to a
set of channels. These channels are the models that the synchronizer watches, which are stated
30
==PAGE:49==
CHAPTER 4. RUNTIME
in the keys of the watch_map dictionary. After instantiating XOSWatcher, the Watcher thread is
created, which runs its run method on start.
The run method listens to messages in the subscribed channels. The way this works with
Redis is the following: the senders do not send messages to particular receivers. Instead, they
send them to speciﬁc subchannels. Then, a client can subscribe to a set of subchannels and if any
message is sent to these subchannels, it will be pushed by Redis to all the subscribed clients. In
this way, when a subscriber of a channel receives a new message, it retrieves the corresponding
step, through the watch_map dictionary, and calls the function handle_watched_object of
this step. This function contains the particular behavior of the synchronizer for the watched
models.
4.1.3
Model policies
This thread is created if a model policies directory is speciﬁed in the conﬁguration ﬁle. It
instantiates the XOSPolicyEngine class, and runs its run method. The code of this thread is run
every 5 seconds and is known as the model policy loop. This loop is very similar to the event
loop, with minor differences. First of all, it calls the function fetch_policies two times,
one to obtain the policy objects that have to be created or updated, and the other to obtain the
policy objects that have to be deleted. For each retrieved object for synchronization, the loop
will call the function execute_model_policy passing as parameter "create" or "update",
and for each object obtained for deletion, the same function will be called passing as parameter
"delete". Then, this function will eventually call the policy step’s functions handle_create,
handle_update and handle_delete accordingly, which will create, update or delete an object
as speciﬁed in the model policy step.
The details of what these functions do in the speciﬁc steps of OpenStack is not provided
since they do not involve communication between XOS and OpenStack and, therefore, are out
of the scope of this thesis.

• New object: when an XOS object is created either from the TOSCA API, GUI or policy

represents the current timestamp of the system. Since, in this case, the save method is

this object is created and its attributes are set; however, it is not synchronized yet. As it
is already explained, the fetch_pending method is executed iteratively in the backend.
Let’s recreate what will happen to this object in the next iteration of the fetch_pending4.
In the next iteration, the fetch_pending will send a request to the XOS core in order to
get the objects of the model that it synchronizes. However, only the objects that fulﬁlls
speciﬁc conditions are returned. These conditions depend on if deletion = False or
deletion = True. Both passes (synchronization and deletion) are executed every time.
– deletion = False: in this case, only the objects whose enacted timestamp is lower
than the updated timestamp, OR the objects whose enacted timestamp is None are
returned. The enacted attribute of the object that has just been created is equal to
None. Therefore, this object is returned and sent to the sync_record method. The
sync_record method will synchronize this object, i.e., apply the necessary conﬁgu-
ration in the underlying substrate (e.g., OpenStack). Moreover, it will set the enacted
attribute of the object to NOW. Again, this NOW represents the current timestamp of the
system. Finally, the save method of the object is executed in order to save possi-
ble feedback state received from the underlying substrate. This time, though, the
caller_kind is "synchronizer" and, therefore, the updated timestamp of the ob-
ject is not updated.
– deletion = True: in this case, an object is only returned if its deleted attribute
is set to True. Since this attribute is False by default, and nobody has modiﬁed
it, it will still be False. Therefore, the object is neither returned nor passed to the
delete_record method.
• Old object: we assume that an old object is an object that has already been created and
synchronized, and that has not been recently updated nor deleted. Therefore, we can as-
sume that the object described above will be an "old object" after the next iteration of the
fetch_pending and all the following ones, as long as it is not updated or deleted. Let’s re-
produce what will happen to this object in the following iterations of the fetch_pending.
– deletion = False: again, the same condition has to be fulﬁlled in order to return
this object and pass it to the sync_record method. Remember that the last modiﬁ-
cation of the updated attribute was when the object was created from the TOSCA
API, GUI or policies thread. On the other hand, the last modiﬁcation of the enacted
attribute was in the next iteration of the fetch_pending method after creating the
object. In this way, enacted > updated and enacted != None. This means that
none of the conditions are fulﬁlled and so the object is neither returned nor passed
to the sync_record method. In the next iterations of the fetch_pending the same
will happen, since neither enacted nor updated will be modiﬁed.
– deletion = True: since nobody has modiﬁed the deleted attribute, deleted will
remain as False. Therefore, the delete_record method is not executed for this
object.
• Updated object: when an object is updated from the TOSCA API, GUI or Policies thread,
the save method of this object is called. Since the caller_kind is not "synchronizer",
the updated attribute is set to NOW. Let’s reproduce what will happen to this object in the
following iterations of the fetch_pending.

(using the sync_record method) only when it has just been created or updated, and not every
5 seconds. Similarly, an object will be deleted (using the delete_record method) just after
deleting it from the GUI, TOSCA or model policies thread.

OpenStack synchronizer
The OpenStack synchronizer runs as a Docker container like all the other synchronizers in
CORD. When this container is started, the run-from-api.sh ﬁle is executed, which sim-
ply executes the openstack-synchronizer.py Pyhton script5. This Pyhton script loads the
openstack_config.yaml ﬁle, which deﬁnes (1) the absolute paths of some important ﬁles and
folders, (2) the models that this synchronizer actuates on and (3) some parameters and creden-
tials for connectivity matters. All this conﬁguration comes in YAML syntax, and is translated into
a Python dictionary using the Python yaml library.
After loading this conﬁguration, openstack-synchronizer.py conﬁgures the global vari-
able DRIVER by calling the method set_driver from event_loop.py located at new_base
and passing an OpenStackDriver object as a parameter. Then, the XOSObserver class deﬁnes
its attribute self.driver as the global DRIVER variable. The steps will then use self.driver
to directly access the OpenStackDriver class, which is deﬁned in the driver.py ﬁle. This
class uses the OpenStackClient class from client.py in order to create an OpenStack driver,
which is used as an interface for interacting with OpenStack using a Python OpenStack API. In
the OpenStackClient class, some certiﬁcates for authentication purposes are used, which give

If the call method is executed with deletion = False, a sync_ports method is called. The

already exists, it means that this port already exists in OpenStack.

this case, the driver is created in the get_driver method of the step, instead of as an
admin driver. According to the documentation: They need to use a client driver that

instances if the port’s tenant does not match the instance’s tenant. Then, the following
call (create_port) to the Neutron API is issued through the driver:
neutron_port = driver.shell.neutron.create_port({"port": <args>})["port"],
where <args> is a dictionary with necessary information of the port that is to be created,
obtained from the Port object.

feedback state from the XOS Port object as follows:

a) The port is associated to an instance.
b) The port is associated to a network. In this case, the Neutron networks are ﬁrst ac-

actuator nor a watcher) cannot write declarative state into an XOS model object. There-

VIM Adaptation Layer
In this chapter, we provide the deﬁnition of the VIM adaptation layer, as well as a few imple-
mentation details based on the knowledge acquired during this work. In fact, we propose two
layers: a VIM adaptation layer and an SDNC adaptation layer. Both the title of the thesis and
this chapter refers only to VIM, since the SDNCs are also often considered as a VIM. Note that
VIM stands for Virtual Infrastructure Manager and a SDNC can also be a manager of virtual
infrastructure (e.g., it can manage virtual switches). However, we divide the layer in two parts
since the functionality of OpenStack or Kubernetes (VIMs hereafter) is very different than the
functionality of ONOS or OpenDayLight (SDNCs hereafter).
5.1
Deﬁnition
The ﬁnal objective of this thesis is to deﬁne a generic VIM layer interface that allows XOS to
support multiple VIMs and SDNCs. The speciﬁc interactions between XOS and OpenStack-
/ONOS analyzed in the previous chapter are now converted into generic calls. In this way, we
deﬁne two interfaces where the methods have generic and meaningful names corresponding to
the identiﬁed technology-speciﬁc calls. Moreover, we deﬁne what each method exactly does,
which parameters have to be passed from XOS, as well as what they should return to XOS.
5.1.1
VIM layer interface
The deﬁnition of the VIM layer interface can be found in Appendix A1. In general, the purple
boxes of the ﬁgures in Section 4.2 map to a generic call in the VIM layer interface; however,
there are some exceptions2. For example, in Figure 4.2, two purple boxes are used for the dele-
tion of a site: tenants.findall and tenants.delete. These two calls, though, happen inside
the delete_tenant call from the OpenStackDriver class. Therefore, it is sufﬁcient for the
VIM layer interface to call the delete_tenant method of the driver if the object to be synchro-
nized is from OpenStack, instead of calling both tenants.findall and tenants.delete in
the OpenStack API. In this way, we consider that the methods of the driver can be reused by the
VIM layer. Moreover, it is not interesting from the point of view of the CORD/XOS developer
to call two methods in order to delete a site, but it is more interesting to just call delete_site
and have the layer take care of how this site is to be deleted in the corresponding VIM. In case
the driver of another VIM have no inner method that takes care of deleting the corresponding
site, the layer can still be able to do so inside the delete_site method by doing the necessary
calls in the underlying VIM API. This, though, are implementation issues that will have to be
addressed in the future when implementing the layer based on our deﬁned interface.
Note that most of the methods of the VIM interface follow a generic pattern. The "create"
methods generally receive as parameters the XOS object to be mapped (created/updated) in the

underlying VIM, as well as the Controller object where this object is to be created repre-
senting the VIM. This is because, in general, a model contains all the necessary information in
its attributes so that it can be created in the underlying system. However, there are also some
exceptions such as in create_instance. In this method, there are other parameters passed to
the function that have to be passed from the synchronizer. Assuming that the XOS models are
generic enough, these parameters (e.g., nics) can be obtained in the same way no matter the
underlying VIM that takes care of this object. In this way, this necessary common processing to
obtain these parameters has to be done in the synchronizer, which sometimes uses other models
(i.e., not only the model that it synchronizes) to obtain them. Then, the VIM layer will receive
all the necessary parameters and will only be in charge of communicating with the correspond-
ing VIM to create whatever is to be created (e.g., an instance) with the received parameters.
In this way, the synchronizer is responsible for accessing the required models and parameters
from the XOS core while the VIM layer is responsible for interacting with the underlying VIM.
On the other hand, the "get" methods only need to receive a Controller object representing
the VIM from which the required objects are requested. For example, get_networks receives
a Controller object and this method knows that it has to retrieve all the networks of the given
Controller. If a speciﬁc object is to be retrieved based on its id or name, it can also be done by
providing these parameters to the "get" methods. However, if a speciﬁc port is to be retrieved
using the get_ports function, it can only be done through its id, since a port does not specify
a name in XOS. There is another exception, which is the get_admin_role function. This
function can only receive a Controller since it already knows that it has to return the admin
role. Therefore, it is clear that it does not need any other parameter.
Finally, the "delete" methods generally receive the id or name of the object to be deleted
and the Controller where this object is to be deleted from. Only the id/name, and not the
entire object, is passed for two reasons: ﬁrst of all, because only the id or name of the object
are needed to delete an object in the underlying system, and not the rest of parameters of the
object. And second, because some XOS model objects (e.g., User) do not contain their id but it
is their controller model (in this case, ControllerUser) the one that keeps their id. Therefore,
deleting a User from the underlying VIM will not be possible by passing as parameter the User
but only by passing as parameter the ControllerUser object or directly the id. For the sake of
clarity, we have decided to directly pass as parameter the id/name. There is one exception that
not only receives one id/name and the controller, but several ids/names. This is the case for the
delete_user_role method. In this method, a role is not deleted in general but only the role
of a speciﬁed user from a speciﬁed site/slice. Therefore, the name of the role, the id of the user
and the ids/names of site/slice are needed, as well as the controller. Note that a role can only be
speciﬁed by its name and not its id, since there is no id attributed saved in XOS for a speciﬁc
role. On the contrary, a user can only be speciﬁed by its id and not its name since two users
may have the same name but never the same id.
There are three other methods that are not a "create", "get" or "delete" method: add_image,
add_user_role and quotas_update. The ﬁrst one, behaves as a create method; however, it is
not called create_image since no image is created in the VIM. Instead, the image is already
created and contained in a ﬁle and it is just added to the VIM. The second one is responsible
for adding a role into a user on a site or slice. Therefore, it does not belong to any of the
three general categories since no object is being created, retrieved nor deleted. In this way, this
method receives all the XOS model objects necessary to add the role into the user. Finally, the
last one is used to update the speciﬁed quotas; therefore, it does not belong to any of the three
general categories either. In this case, it receives the quotas, as well as the slice id/name and the
controller where the speciﬁed quotas have to be updated.

SDNC layer interface
The deﬁnition of the SDNC layer interface can be found in Appendix B3. All the purple boxes

deploy_sdnc and only receives an SDNCService object, which represents the SDNC to be de-

and receives the data that has to be posted to the SDNC as well as a FabricService object,
representing the SDN controller that manages the switching fabric in CORD.
5.2
Implementation details
In this section, we propose some implementation details that could allow the proposed layer
interfaces to be implemented and work in CORD. First, we describe how the current OpenStack
and ONOS models should be modiﬁed so that they can be handled by our deﬁned layers. Then,
we propose new synchronizers that use the proposed generic models and VIM/SDNC layers to
support multiple VIMs and SDNCs.

Note that the names of these models are generic, since they belong to XOS terms. In fact,
all these models belong to the XOS core models, and not to OpenStack-speciﬁc models. As an
example, let’s consider the model ControllerSlice. The step that synchronizes this model is
responsible for creating/deleting a slice into OpenStack. However, in the OpenStack nomen-
clature, there is no element called "slice". Instead, when creating a "slice", the synchronizer is
actually creating a "project" in OpenStack, and there is no model in XOS named "Project". For
this reason, we can assume that, at least the names of the models that OpenStack synchronizes
are generic enough, and that they can be reused for another VIM.
It is true, though, that some attributes of the models should be renamed. For example, when
creating a User in OpenStack, we obtain the id of this user as feedback, and this feedback
is saved under the kuser_id attribute of a ControllerUser object. The "k" of kuser_id
belongs to Keystone, which is a component of OpenStack. Another example is the attribute
glance_image_id from the ControllerImages model. Again, the name "glance" belongs to
the Glance component of OpenStack. Therefore, it would be interesting to change this kind of
attribute names into more generic ones. The same happens for some help texts of the models
attributes. For example, in the aforementioned attributes, the help texts are "Keystone user
id" and "Glance image id". Although it does not affect the performance of the synchro-
nizer, both some attribute names and some attribute help texts should be changed in the current
models.
Given that we are not experts in other VIMs, we cannot claim that the current attributes
of the models synchronized by OpenStack are sufﬁcient for the other VIMs. For example,
ControllerNetwork contains attributes such as gateway or router_id. Although we can
assume that these attributes are VIM-generic, we do not know if other attributes should be
added into this model in order to support another VIM in CORD such as Kubernetes. This
discussion, though, is out of the scope of this thesis.
ONOS synchronizer models
The ONOS synchronizer synchronizes the following models:
• ONOSService
• ONOSApp
In this case, it is clear that the models synchronized by the ONOS synchronizer are ONOS-
speciﬁc. Therefore, they should be replaced for generic ones. We propose SDNCService and
SDNCApp instead of ONOSService and ONOSApp, respectively.
We think that the attributes of the ONOS models could be reused for the new proposed mod-
els as long as the new SDNC supports REST API calls. This is because most of the attributes
of the ONOSService model are speciﬁc for the REST API: rest_hostname, rest_port, etc.
Without being experts in other SDNCs such as OpenDayLight, we also assume that the at-
tributes of the current ONOSApp model could also be reused, since it just deﬁnes how to install
and conﬁgure an application.
Nevertheless, we propose for the SDNCService model to add an attribute indicating which
is the SDN controller corresponding to this SDNCService model object. This attribute could
be called backend_type and be deﬁned as a String object whose value could be "onos" or
"opendaylight", for example.

Conclusions
6.1
Planning update
The initial planning of this thesis can be found in Appendix C, as presented in the document
"Project Plan" on 07/03/2018. In general, we have followed the initial plan; however, there
have been some modiﬁcations. First of all, we decided with our supervisors to focus only on
the "layer approach" to integrate another technology to CORD. In this way, we discarded the
"native integration" since the objective of this thesis was from the beginning to deﬁne a VIM
layer interface. On the other hand, regarding the different planned tasks, we ﬁnally discarded
to spend 10 days analyzing communication between the nodes with Wireshark. Although we
tried to do so at the beginning, we ﬁnally decided that it was more useful to dedicate this time
to better understand the synchronization process of the synchronizers, rather than analyzing
packets in Wireshark. The rest of the plan was strictly followed since most of the tasks took the
expected time.
6.2
Outcome summary
The objective of this thesis was twofold: (1) to analyze interactions between XOS and Open-
Stack/ONOS, and (2) to deﬁne a generic VIM layer interface that allows XOS to support mul-
tiple VIMs and SDNCs. A summary of each process is described below.
Regarding the interactions part, we analyzed both installation and runtime processes. The
installation process description in Chapter 3 describes how OpenStack, ONOS and XOS are
deployed in CORD. We consider that none of the installation processes have to be handled by
the proposed layers, since the installation process for different technologies can be very differ-
ent. Speciﬁcally, the different OpenStack components are deployed by Juju in Linux containers,
whereas ONOS and XOS are deployed by Docker in different Docker containers. The XOS de-
ployment includes the deployment of the service synchronizers (including the VIM and SDNC
services). In this way, the objective of this chapter is to show how these technologies/ser-
vices are deployed in CORD and, therefore, how another technology could also be integrated
in CORD.
The runtime processes are characterized by the synchronizer processes since the synchro-
nizer is the one that actually interacts with the underlying technologies/services of XOS (e.g.,
OpenStack or ONOS). To this end, the synchronizer deﬁnes a set of steps, being each of them
in charge of synchronizing a speciﬁc XOS core or service model. This synchronizing pro-
cess is responsible for taking the corresponding parameters of the XOS model object to be
sychronized, and then use these parameters to issue a call to the underlying system. In order to
understand how and when the synchronizer decides to interact with the corresponding system,
we ﬁrst described the backend of a synchronizer, i.e., the default behavior of a synchronizer.
Then, we analyzed each of the steps in each of the synchronizers of interest, i.e., the OpenStack,
67
==PAGE:86==
CHAPTER 6. CONCLUSIONS
ONOS and Fabric synchronizers. In each step, we illustrated a diagram showing the speciﬁc
processes and inner calls that take place before issuing the corresponding calls to OpenStack-
/ONOS. Then, we analyzed each of these calls by describing what it does, as well as the main
parameters that it uses. After having analyzed all the calls, we can see that the XOS/OpenStack
interactions happen in two different ways (either through OpenStack-speciﬁc Ansible modules
or through an OpenStack-speciﬁc Python API), whereas the XOS/ONOS interactions happen
only through the ONOS REST API.
Finally, regarding the proposed layer interface, it has been split into two: a VIM layer in-
terface and a SDNC layer interface. For each layer, we described the general behavior of the
proposed generic methods, as well as the speciﬁc behavior of some outliers. We also proposed
some modiﬁcations to the XOS models, so that the VIM and SDNC synchronization processes
can be generic and the proposed layers can be used. Moreover, we also proposed which meth-
ods of the deﬁned layer interfaces would be called from each of the steps of the corresponding
synchronizers.
6.3
Future work
The future work for this thesis is to implement the proposed layer interfaces and to test them
in CORD. A proof of concept with the same OpenStack and ONOS could be initially devel-
oped. The idea is that CORD should be able to work as it currently does but making XOS
use the proposed layer to interact with OpenStack/ONOS instead of doing so directly. Then,
a new software (Kubernetes as a VIM, or OpenDayLight as an SDNC) will also be able to be
integrated into CORD in a way that the generic synchronizer performs the same processes no
matter the underlying software, and that it is the layer who takes care of which software the
object to be synchronized belongs to.

Appendix C: Project Plan
07/03/2018
To this date, the tasks have been to install and build the system that we will use during the
project, and to have an overview of CORD both theoretically and practically. By theoretically,
we refer to understand the context in which we will be working on (i.e., the idea and purpose of
CORD), the different elements that comprise the system (e.g., containers and virtual machines),
and the Ansible automation tool, which drives most of the processes in CORD through the so-
called Ansible playbooks. By practically, we refer to understand and work on the software
system itself, going step-by-step through its online guide [6], as well to have an overview of
the installation and runtime processes in CORD. This provides us a basic hands-on experience
with the system that, along with the acquired theoretical knowledge, lets us identify and plan
the following steps and tasks that are required in order to fulﬁll the proposed objectives.
In this way, the next steps entail analyzing both the installation and runtime processes in
detail. For the installation process, we ﬁrst plan to list all Ansible playbooks related to Open-
Stack/ONOS that are executed during the CORD build process, as well as all the ﬁles that these
playbooks use. After that, we will identify and characterize all the shell commands, Ansible
modules and scripts executed in these ﬁles.
Regarding the runtime process, we will analyze OpenStack and ONOS synchronizers, which
are characterized as Docker containers. We will analyze ﬁrst the dependency among all the ﬁles
within these containers, i.e., how and when the different ﬁles are invoked, as well as the purpose
of each ﬁle. We will then analyze the so-called synchronizer actuators, which consist of Ansible
and Python scripts that execute the required actions in each synchronizer. Moreover, we will
probably need to analyze some of the XOS Docker containers.
At this point, we will have mainly analyzed scripts and ﬁles in order to ﬁnd interactions
between the different components. We also plan to investigate interactions by snifﬁng trafﬁc
with a snifﬁng tool such as Wireshark. We will install Wireshark in the physical host, as well as
in the head and compute nodes. In the physical host, we aim to see trafﬁc between the different
Vagrant VMs (i.e., head node and compute node) through the virtual switches (Open vSwitch).
In the head node, we aim to see trafﬁc between Docker containers (i.e., XOS, OpenStack and
ONOS containers). Finally, in the compute node we aim to see communication related to the
OpenStack VM instances.
After that, we expect to have a proper understanding of the system that allows us to decide
and explain how the integration of another technology into CORD might take place. In case we
decide that the best approach is to implement a VIM adaptation layer that translates messages
between XOS and the underlying technologies, we will deﬁne how this layer should be imple-
mented. In case we decide that the best approach is to directly create the whole integration
process for the other technology, we will deﬁne again how this should be done.
