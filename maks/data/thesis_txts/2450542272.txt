Abstract
Increasing applications in maritime technology, as off-shore wind farms or ship inspection,
have aroused the need for autonomous ROVs as the use of traditional teleoperated ROVs
is too expensive for many of these applications. This thesis is a step forward in enabling
the autonomy for these robots, seeking in the near future the full autonomy and thus
removing the human component in its operation.
Towards solving these complex missions, the ROV has to be able to follow complex
trajectories smoothly in order to simplify the mission objectives and improve sensor
readings. To do so the ROV has to firstly be capable of reaching a certain orientation
effortlessly. Hence the main objective in this thesis is to propose a control strategy for
the attitude of the ROV.
In order for this to happen several objectives have to be performed. Firstly a new
eight thruster prototype is assembled and tested for operation. Sequentially, using Fos-
senвҖҷs framework [6], a dynamic model for the ROVs attitude in Pitch and Roll is proposed.
Therefore to extract the model parameters an experimental campaign and methodology
is engineered. Lastly, a control strategy is proposed and using the existing model an
optimal controller is designed. This controller is tested in simulation and is implemented
as a ROS package.

CHAPTER 1
Introduction
This is the master thesis Roll and Pitch Control of a Remotely Operated Vehicle (ROV).
This thesis was performed at the REconfigurable MOdular Robotic system for Aquatic
environment (REMORA) Lab at DTU Elektro.
1.1
Background and Motivation
The popularity and usage of ROV in the last decade has risen exponentially. Cheaper
manufacturing, developing battery technologies and specially the growth of open source
software, like the Robotic Operating System (ROS) framework, have created the perfect
environment for companies to sell proprietary hardware like the BlueROV from Blue
RoboticsВ®[2] to fulfill the demand of having a mobile platform capable of driving different
sensors and actuators in an underwater remote environment.
This can have multiple applications as surveying and inspecting structures or other
marine vehicles, mapping areas or even doing some simple maintenance tasks.
The
motion control for most of this solutions was done by remote operation, this means
there was a human in the loop generating movement commands to fulfill the mission
tasks. In order to do so the mechanical design was done to have a stable, self restoring
body, this means the buoyancy center is displaced in the vertical positive direction with
respect to the gravity center. This means that the all vehicle maneuvers are parallel to
the seabed and instead of using the 6 possible degrees of freedom, the ROV was fixed in
two(Roll and Pitch) while only 4 were controllable(Surge,Sway,Heave and Yaw).
This constraint, although it simplifies navigation and control for the pilot, limits
the possible in trajectories the ROV can describe. As the navigation will be done au-
tonomously, the limitation of a human in the loop can be removed, allowing for very
specific trajectories that can simplify and speed up the tasks. As 6 degrees of freedom
are available the fixed sensors can get multiple readings from different positions and
orientations.

1.2
Goals
The main goal of this project is to develop a dynamic model and a control strategy for
controlling the roll and pitch axis. Being able to navigate as it did previously,but with
a specific roll and pitch orientation.
In order to do so the following sequential secondary goals will be pursued:
вҖў Build and test the ROV electronics
вҖў Prepare, test and fine tune the ROV for water tests
вҖў Design and implement an experimental campaign for the model identification
вҖў Identify the mathematical model that describes the dynamics of the ROV
вҖў Design an active control system for roll and pitch validating it using Matlab
вҖў Implement the controller in a ROS node
вҖў Verify the control system with real experiments and compare to simulation.
After the success of these previous steps the ROV should be able to follow any pair
of orientation angles and navigate keeping such orientation. With further development
it should be feasible to follow different way-points with a specific orientation in order to
describe complex trajectories and so fulfill any task in a more efficient way.

2.1.1
Frame and Enclosure
The frame was custom made at the REMORA Lab from a previous student project. It
was designed in AutoCADВ® and it was cut from thick Polyvinyl chloride (PVC) slab
using a Computer Numerical Control (CNC) mill. The shape is thought to be able to
let water flow easily and reduce the drag, additionally the thrusters flow is unrestricted
so there is no loss of thrust. The resulting structure is a very rigid, resilient and light
weight frame to box all the components.
The electronic enclosure is made out of an acrylic cast transparent cylinder and is
6 inches in diameter. The lids are made out of aluminum and contain several ridges
for O-rings to ensure water tightness.
All the connections and attachments have to
be previously applied with silicone grease so the O-ring does not wear from each slide
motion while opening and closing. The closing mechanism is non existent, it purely
relies on friction and when it is submerged, on water pressure to keep it tightly closed.
In order to get cables in and out penetrators are used, see Figures 2.2 and 2.3. They can
be bought from Blue RoboticsВ® and they allow a water tight reusable connection. The
cable is passed through the penetrator hole which is filled with epoxy resin as it can be
seen in Figure 2.3a.
The front lid of the main enclosure is transparent so a camera can be installed. There
are two more additional enclosures for the batteries, same as the electronics one but these
are 4 inches in diameter.

2.1.2
Thrusters
The T200 thrusters used are proprietary from Blue RoboticsВ®. It is basically a brushless
motor like any other drone or Radio Control (RC) airplane except this one has been
designed for marine applications using plastic bearings instead of steel ones to avoid
rust and salt degradation.
Figure 2.4: T200 Thruster from Blue RoboticsВ®
When the thrusters are powered with a 4S battery, 16 Volts, they can have a maxi-
mum current of 25 Amps and power of 350 Watts delivering 50 N of forward thrust and
40 N of reverse thrust with a rotational speed range of 300 - 3800 rpm.

The thruster configuration proposed for the ROV follows a popular approach for
marine vessels. There are four thrusters working on the same plane (Thrusters 1 to 4)
that induce force in forward, lateral movement (Surge, Sway) and Yaw. Another set of
thrusters (Thrusters 5 to 8) are located on the same plane but the orientation is normal
to this plane, these are responsible of the vertical displacement (Heave) and Roll and
Pitch rotations. In Figure 2.5 it can be seen such configuration.
Figure 2.5: Vectored 6 DOF Configuration
2.1.3
Electronic Speed Controllers
The electronic speed controller controls the speed of motors by generating a variable
three-phase Alternating Current (AC) power signal. The Pixhawk generates a Pulse-
Width Modulation (PWM) signal with the desired speed encoded on the duty cycle.
Figure 2.6: Afro 30A Race Mini ESC

In order to use these ESC they have to be flashed with a specific firmware that allows
reverse spin. As they are mainly used for drones, these only need to spin in one direction,
but in the water the ROV can make use of reverse throttle to help other motors and
correct the attitude.
This firmware is provided by Blue RoboticsВ® and can easily be flashed using an open
source flashing tool and a dongle that allows serial communication through the signal
cable.
The ESCs can be powered from 2-cell up to 4-cell batteries as they can regulate the
supplied voltage to each thruster.
2.1.4
Up Squared
The on-board computer is a single board computer from the brand Up. It is their second
iteration and its a reliable low consumption computer. Its core is an Atom x7-E3950 that
is a 64-bit quad-core x86 system. It comes with 8GB of RAM and a 128GB eMMC for
storage. It has three Universal Serial Bus (USB) 3.0 and two Ethernet RJ-45 connectors
[12].
Figure 2.7: Up SquaredВ®Atom Quad-core 8GB RAM 128GB eMMC
It requires a filtered 5 Volts input with a maximum current draw of 6 Amps, thus a
maximum power of 30 Watts.
This Single Board Computer (SBC) was chosen differently from the one proposed
from Blue RoboticsВ® as it has a full x86 architecture and not Advanced RISC Machine
(ARM). ARM is part of the Reduced Instruction Set Computing (RISC) family of pro-

cessors needing a different compiler for building the programs. As the ROS development
is done in x86 architecture it is preferable to run the program on the same architecture

This SBC had a stock passive voluminous heat-sink, to dissipate heat from the Cen-

The Pixhawk 4[10] is an autopilot micro-controller designed by HolybroВ®. It contains
two micro-controllers one for the main flight control algorithm, a 32 Bit ARMВ®CortexВ®-
M7 running at 216MHz with 2MB of memory and 512KB of RAM and another to
manage the inputs and outputs, another 32 Bit ARMВ®CortexВ®-M3 running at 24MHz
with 8KB of SRAM.

an open-source tool called QGround Control, that also allows configuring the Pixhawk
parameters and visualizing measurements using a Graphical User Interface (GUI).

measurements. Additionally the heat flow was also considered for the placement of the
components. During operation the ROV is submerged and the enclosureвҖҷs heat can be
easily extracted by its whole surface.
2.1.7
Power Distribution
The power supply for the ROV was quite a challenge as there are two separate battery
enclosures to supply a third enclosure with all the electronics. In order to reduce the
overall effective resistance and lower the energy loss due to Joule effect the thickest wire
that would fit through the penetrators was used. Additionally every solder joint was
done very carefully with sufficient amount of material to ensure good conductivity.
For the PDB a screw terminal was used on both of the sides of the electronics tray. In
order to connect each terminal with one another, and thus make a positive and negative
bus, jumpers are used. The configuration of these was thought to minimize losses and
resistance using the maximum number available.
In Figure 2.10 the whole connection schematics can be seen. The parallel connection
of the batteries is done on one of the enclosures and then only two cables go to the
electronics enclosure. This was done as the available number holes would not allow for
this connection to be made in the main enclosure.
Figure 2.10: Power distribution diagram
The batteries have the following characteristics, 4 cell(16 Volts) with 18 Amps hour
of capacity and a maximum continuous discharge of 90 Amps. As they are connected in
parallel the maximum current draw from the electronics can be up to 180 Amps.

2.1.8
Assembly Details
The weight distribution and buoyancy center are very important parameters in the
hydrodynamic stability and controllability of the ROV. For such reasons during all the
assembly process every part and piece was placed in a way it would keep the symmetry
of the weight distribution. Also every thruster position and orientation where tightly
screwed to ensure decoupling of motions and reduce imbalances.
Additionally for the buoyancy compensation floating rigid foam was used. A rough
overestimated calculation of the total volume of foam needed was performed to counter-
act the weight of the ROV underwater. After this, a finer tune on the pool was done by
removing foam material and making sure there were not strong restoring forces as this
would affect on the attitude control.
Moreover before doing any tests with the electronics the water-tightness of the en-
closure and epoxy fillings were tested on a full 24h submersion, checking for any water
drops or signs of failure in any of the O-rings. In Figure 2.11 it can be seen such water
test.

After all this process the ROV was ready for the first water tests and fine tune of
the low-level parameters to enhance its performance.

2.2.1
Architecture Overview
The whole system architecture relies on ROS[11] framework to manage sensor readings
and actuator commands, as well as, the controllers, drivers and other necessary processes.
In Figure 2.12 it can be seen the connection diagram for all the components needed to

Figure 2.12: Signal flow diagram
The off-board computer is connected using a long Ethernet Cat 6 tether directly

This computer interfaces with the Pixhawk using serial
communication by a USB cable. Then the Pixhawk micro-controller outputs eight PWM
signals to each of the ESCs. After this each ESC generates a three phase signal to drive
the brushless thrusters.
2.2.2
Off-board Computer
All the high-level control and interfacing is done from the off-board computer.
The
necessary processes are launched from here. It connects via ssh to the on-board computer.
In order to do so both computers need to be set up with an static IP address. To use
just one ROV the connection can be direct but also a switch or router can be used to
manage several ROVs at the same time.
==PAGE:29==

In this computer the heavier more complex processes should be run. For example
any neural network, Simultaneous Localization and Mapping (SLAM) or any other com-
putational heavy algorithms. Also if the ROV wants to be controlled with a remote, the
driver reading the joysticks has to be run in here, as the device is connected to a local
serial port.
Additionally, the GUI rviz from ROS can be used to visualize the state of the robot,
for instance, the position and attitude corresponding to different reference frames. Ad-
ditionally it is also possible plot and monitor topics as a way to debug any unexpected
behavior. Moreover, more importantly, it is viable to record rosbags, which are the data
stream of messages in the ROS network. This is useful for reproducing the robot behav-
ior and measurements without it being running. This translates that in the developing
phase these files can be played and new functionalities may be tested.
2.2.3
Up Squared
The on-board computer main task is to run the low-level ROS processes to interface
with the Pixhawk.
The main ROS packages needed are the following:
joint_state_publisher : this package establishes a universal description for the
moving and fixed joints of the robot. It extracts the definition from a Unified
Robot Description Format (URDF) file and sets a known structure for the robot
in the ROS environment. Also sets up different frames according to the file which
can then be used for visualization tools.
mavros : this package manages the serial communication with the autopilot using
the universal protocol for mobile robots MAVlink. It is able to read information
generated in the autopilot like the EKF estimation, publishing it on the ROS
network and also to send any command, reading from the available ROS topics,
robot_localization[9] : this package is a set of nonlinear state estimators for 3D
space movement. In this case the ekf_localization_node is used. It is capable of
computing estimates for linear and angular positions, linear and angular velocities
and linear accelerations. With a comprehensive setup of the parameters, specially
the initial covariances for the sensors, the filter will continue to estimate even if
there is a long period in which no data is received.
2.2.4
Pixhawk
The Pixhawk manages all the lowest-level control for the ROV. It has a double micro-
controller to manage the Input-Output (IO) and the controller algorithm. This is done
using an open source framework fork called ArduSub. This code has to be cross-compiled
for the specific target board in order to flash it using Qground Control.

The tasks carried by the Pixhawk are the following. On the one hand there are
running two EKFs which are then fused again for more robustness and accuracy to
estimate the position and attitude of the ROVs, being possible to fuse it with other
measurements. This allows to have a reliable estimation for the orientation angle and
filtered measurements for the gyroscope and the accelerometer. On the other hand there
are several control loops to actuate the motors.
All these control loops are based on a ProportionalвҖ“IntegralвҖ“Derivative (PID) struc-
ture, the controller gains are customizable to fine tune the desired behavior of the ROV.
Additionally there are many more parameters that are vital for the proper operation as
for example, the IMU offsets in position and orientation and the motor spin directions.
This spin direction can be changed by switching two cables of the three-phase signal but
it is more convenient to do it from the software side. Also it is good practice, every time
the electronics enclosure is closed, to re-level the IMU as the electronics tray can rotate
inside the enclosure and offset measurements in roll specially.
Unfortunately it is not possible to command PWM thruster values directly from
the high-level (Upboard or off-board). This signal needs to be updated at a high rate
they have to be generated from the control loop as any failure from the computer may
result in a catastrophic event for the ROV. The possible commands can be in position
or velocity of the body, depending on what flight mode is active. The algorithm, which
has previously been properly configured, has the thruster configuration into account to
render the individual duty cycle values for each PWM signal.
Lastly the Inter-Integrated Circuit (iВІc) pressure sensor is managed by the autopilot
in order to have an estimate of the depth and enabling depth control.

CHAPTER 4
System Identification
In this chapter the System Identification methodology is illustrated. The techniques
used are based on the following literature [1], [4] and [8].
The purpose of this model identification is to achieve a mathematical model thats
describes the orientation behavior of the ROV with a certain angular speed command
from the high level. Due to the nature of the software architecture it is not possible to
directly command a specific PWM value to the thrusters and then measure the output.
All the commands have to be done in a high-level body command, this means a target
orientation or angular speed. This command is then computed by the low-level (Pix-
hawk) and then, using the embedded controllers, the input signal is generated and sent
to the thrusters.
In order to remove these controllers dynamic, instead of using the high-level command
as an input in our system identification, the PWM thruster command signal is also
captured, and it is used as the input for the identification. Using these controllers is
very helpful during the experimental phase, as the ROV is able to maintain a certain
orientation and depth and just actuate in the desired motions that want to be captured
to extract their dynamic.
For clarification purposes, the proposed procedure is the following: a target angle
is commanded from the high level then depending on the current state the pixhawk
generates a PWM command every control loop until the desired target orientation is
achieved. In this process the PWM command is captured for every loop as well as the
angle and angular velocity. This way we can create two differential thrust signals using
these PWM values as our input to the dynamic system and our outputs will be the angle
and angular velocities. This way it is possible to bypass the controller dynamics on the
response and it is possible to model the movement dynamics of the ROV.

The experiments where carried out at a DTU Mechanical Engineering facilities in build-

was decided to do two different sets of experiments, as it can be seen in Figure 4.2. All
experiments start at the equilibrium position of [0В°, 0В°].
On one set of the experiments there is a step command in one of the axis angle
position and back to the starting position. This means sending a command to control
the ROV to [0В°, 0В°], start recording, then [xВ°, 0В°], let it settle for 5s and then commanding
[0В°, 0В°], let the ROV reach steady state and then stop the recording. The choice for this
settling time was decided after some visual inspection from the first experiments, it had
been seen it is more than enough time to reach the desired orientation.
Similarly on the other set there was a second step from the first to the symetrical
value in the other direction, as it can be seen in Figure 4.3. Schematically the sequence
of commands is the following: [0В°, 0В°] [+xВ°, 0В°] [-xВ°, 0В°] [0В°, 0В°]
In order to capture the data the ROS tool called rosbag is used. This tool is able
to save the stream of different timestamped topics into a bag file. This can be played
again in the ROS environment or it can be read, using the appropiate tools, in Matlab
or other programs. It is important to note the mavros node is running at a frequency of
10Hz and so the readings are done at such frequency.

CHAPTER 5
Controller Architecture and
Design
In this chapter the controller placement and strategy will be introduced as well as the
proposed control objectives and the LinearвҖ“Quadratic Regulator (LQR) design.
5.1
Control Strategy
The proposed control strategy is to control the ROV in velocity. This means instead of
commanding orientation angles to the low-level controller (Pixhawk), angular velocities
will be commanded to achieve a desired angle. In further projects a Model Predictive
Control (MPC) trajectory generator will be developed. This trajectory generator will
have a set of way-points the ROV has to describe to fulfill the desired path, with the
corresponding system constrains and limitations.
Having a velocity control allows for a smooth following of the orientation set-points,
as the controller can alter the commands from feedback measurements, accounting for
disturbances, and also smoothening the transition between way-points. Additionally the
control algorithm does not have to wait for the angle command to settle in the position
control.
The model previously described in Chapter 4 allows for prior knowledge on how the
ROV is going to behave with certain inputs. As all the states are observable from the
IMU measurements and with the prior insight on the body dynamics the desired angular
velocity command can be computed with a specific control objective, in this case a quick
settling time and small steady state error.

5.3
ROS Implementation
This controller has to be implemented as a ROS package for it to work on this environ-
ment. In this part the approach and minimal functionality this package needs will be
described.
In these packages there are two main parts of the code, the setup part in which
the environment is created and initial calls to other processes are made, and its only
executed when the package is firstly launched. The loop part, which is executing at a
more or less constant rate (see Figure 4.4) computing the same code in every iteration,
this loop can be timed or be part of a callback function. In this case the loop will be
part of the IMU reading callback, that translates, every time the ROS environment gets
a reading from the IMU a new control action signal will be generated and sent to the
Pixhawk. This description will be done in a readable, pseudo-code approach.
The setup part includes the following:
вҖў Read launching parameters or configuration file
вҖў Declare subscribers for IMU reading and set-point reading
вҖў Declare publisher for Pixhawk
вҖў Arm Pixhawk
вҖў Set PixhawkвҖҷs flight mode to Acro (angular rate)
The following instructions will be executed every IMU reading:
вҖў Check set-point
вҖў Filter IMU reading
вҖў Compute new control signal using both
вҖў Scale signal to PWM values and saturate for safety
вҖў Publish to pixhawk
On top of this routine there can be further functionalities added as some error check-
ing like check if the Pixhawk is alive, or other processes alive, add more controllers which
can be set through topics or the launch parameters. Also it is good practice having a
shutdown procedure for when the process is killed.

CHAPTER 6
Conclusion and Future
Work
In this thesis, a new prototype of and eight thruster full six DOF ROV has been assem-
bled and tested. An attitude dynamic model for the ROV has been successfully estimated
and a control strategy has been proposed. In this chapter it will be discussed and sum-
marized the obtained results and challenges encountered during the work. Lastly, some
future development and steps will be introduced.
6.1
General Conclusion
First of all the assembly and test phases for the ROV were a total success. During this
stage the most challenging aspects came from the power distribution and ensuring there
where no major losses or shortcuts. There had to be a lot of attention in making the
enclosures water-tight, with special focus in applying silicone grease to all joints, as any
minimal leak would turnout in a fatal event rendering the ROV unusable. It is also
important to bring to attention that the Pixhawk parameters play a vital role on the
ROV performance and behavior as these parameters govern the PWM values that are
finally sent to the thrusters.
Secondly a linear dynamic model has been estimated for the attitude of the ROV. Es-
timating this model was quite challenging at the beginning as the PWM signals cannot
be enforced from the high-level. An strategy to overcome this controllability issue was
proposed by recording the control output form the low-level controller and then using
this as the input to the estimated model.
This linear model has its limitations as it describes the motion at a specific operation
point. In principle this is not a big issue due to two reasons. On the one hand it will be a
closed loop control architecture, this means every loop (around 10Hz) the measurements
will be taken and the system will be able to correct the control action. On the other
hand this control action will be saturated to this operating point to ensure the ROV is
always being commanded in the design region and thus the expected behavior closely
matches the real one. Taking these two facts into account having small inaccuracies in
the model should not affect significantly on the performance.
==PAGE:64==
52
6 Conclusion and Future Work
Lastly a control strategy, taking into account the software architecture limitations,
was proposed and using the estimated model an optimal controller was designed. With
this proposed control composition the approach to control the ROV changes from how
it used to be done using a velocity control allowing the ROV to describe complex tra-
jectories facilitating the completion of the mission objectives. Due to lack of time and
some issues with the testing facilities during the duration of this thesis this controller
was not able to be tested in the real case scenario. It is implemented as a ROS package
and ready for these tests to be performed.
6.2
Future Work
In this section future steps in the development of the line of research will be proposed.
First of all, by the end of this thesis this robot can be added to the REMORA fleet
for future projects and development using the old architecture as ROS packages are
interchangeable.
Secondly, the first step would be to test and validate on the real platform the designed
and implemented controller for the two DOF. If this proposed control architecture yields
to perform satisfactorily the work could be extended to the six DOF enabling a full
velocity control.
After completing this the next step would be to develop the MPC trajectory generator
so that the mission way-points fulfill a comprehensive reachable trajectory for the ROV.
As another possible approach to solve the problem would be to try and remove the
Pixhawk from the software architecture, directly publishing PWM duty cycles for the
thrusters in the ROS environment. This would facilitate greatly the possible controllers
governing the body dynamics. On the other hand many associated issues would arise
as any fault or halt on the operating system could make the robot unmanageable, this
problem is greatly reduced as underwater the body dynamics are slow and some halts
on the control would not render the robot useless.
