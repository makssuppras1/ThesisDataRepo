Abstract
Process mining is a data­driven discipline that extracts insights from process execution
data stored in event logs, combined with process models or business rules. Numerous
tools, including open­source options like ProM and commercial products, offer diverse
algorithms for process analysts. Choosing specific tools involves navigating various fac­
tors, such as context­specific requirements, data constraints, and algorithm sharing pref­
erences. In practical applications of process mining, researchers typically require special­
ized technical knowledge, posing a complex challenge for those with only basic program­
ming skills. Institutions like universities or companies often need to use data or algorithms
from other organizations, raising privacy concerns. This paper introduces the concept of a
process mining pipeline and proposes a method for user identity and permission authen­
tication within this pipeline. This approach aims to address privacy issues in the process
mining pipeline, enabling organizations to grant access permissions while ensuring the
security of their data and algorithms. Consequently, this method enhances the accessi­
bility and security of process mining tools for non­technical audiences and simplifies the
traditional approval processes for privacy concerns, contributing significantly to the field
of process mining.

Introduction
Process mining is a discipline rooted in data analysis that aims to derive insights from the
data captured in event logs, often in conjunction with process models or business rules.
A wide variety of tools are available for this purpose, ranging from open­source platforms
like ProM to commercial software, each offering a range of algorithms tailored for process
analysts’ needs [1]. The selection of appropriate tools requires careful consideration of
various factors, including the specific context, data limitations, and preferences for sharing
algorithms.
Given its maturity, process mining often involves complex operations that may encompass
multiple datasets and employ a variety of algorithms across different organizations[2].
Conducting these activities without a well­structured process mining pipeline or workflow
can make research cumbersome and repetitive. For example, processing two event logs
individually using different alpha miners and then using another algorithm to perform a
comparative analysis and save the results highlights the need for an efficient process
mining pipeline or workflow system[3].
Building a process mining pipeline often requires the use of resources from multiple orga­
nizations, which involves the emergence of cross­organizational process mining pipelines.
Building cross­organizational process mining pipelines poses some significant challenges,
especially in terms of privacy concerns and the necessity to perform manual data tuning or
programming [4]. Many of the datasets used in process mining involve sensitive and con­
fidential information, so extra care must be taken, especially in cross­organizational envi­
ronments such as hospitals or educational institutions. The greater need for security has
made privacy­preserving process mining an increasingly important area of research[5].
This area involves the use of a variety of privacy­preserving techniques to ensure that
process mining is conducted securely, although these methods often require advanced
technical skills . This contrasts with the goal of making processes simpler to use for users
without specialized knowledge.
The purpose of this research is to make it easier for researchers who are not experts
in the field to build process mining pipelines. To this end, we propose the Distributed
Architecture for Process Mining Pipelines (DAPM). This platform will allow researchers to
build and share process mining pipelines across different organizations without requiring
extensive programming skills. The use of a distributed architecture enables resource
sharing across organizational or network boundaries while still respecting organizational
boundaries and protecting sensitive data[6]. There are now three simultaneous groups
working on the DAPM project at the same time. The conceptual group is responsible
for creating a conceptual model and developing a front­end for the pipeline, the platform
group is responsible for developing the back­end of the platform so that the pipeline can
be built and deployed, and the security group is responsible for building an access control
framework for DAPM to ensure the security of resources.
The DAPM is designed to offer a user­friendly environment where individuals without pro­
gramming expertise can create process mining pipelines using modular, plug­and­play
components. It is essential that DAPM clearly delineates organizational boundaries to
prevent the unauthorized transfer of sensitive data. Additionally, the system will guide
users by outlining the necessary inputs and expected outputs for each algorithm. DAPM
will function as a workflow management system, enabling users to design pipelines in the

2
Background
In this chapter, we research the evolution and application of process mining, introduce the
concept of data science pipelines, and review the current state of the art. We conclude
with a discussion of related work and the significance of our research.
2.1
Context and Foundations
This section provides an overview of the historical development of process mining and
the use of pipelines in computer science, setting the stage for a deeper understanding of
these topics.
2.1.1
Evolution of Process Mining
Process mining is an advanced analytical discipline that focuses on deriving actionable
insights from event logs to enhance process efficiency and compliance. The field, pio­
neered by Wil van der Aalst, traces its origins to the use of Petri nets for representing
workflows [9]. This foundational work laid the groundwork for the development of auto­
mated methods aimed at discovering, monitoring, and optimizing process models using
real­world data [2].
Discovery Techniques in Process Mining
Discovery in process mining involves generating process models from event logs, with the
goal of accurately reflecting recorded activities. These models are evaluated based on
fitness, simplicity, precision, and generalization. Several algorithms are employed in this
context, including the alpha algorithm, heuristic miner, and inductive miner, each offering
distinct benefits and facing specific challenges[1].
Alpha Algorithm
The alpha algorithm is known for its straightforward and formal ap­
proach to converting event logs into Petri nets. Despite its educational value, the algorithm
is sensitive to noise and struggles with complex process structures, limiting its practical
applicability [10].
Heuristic Miner
Heuristic miner is widely used in industry due to its robustness against
noisy and incomplete data, as well as its ability to capture frequent behaviors effec­
tively[11]. However, it can be challenging to manage due to the complexity of its pa­
rameters and a lingering sensitivity to noise [12].
Inductive Miner
The inductive miner is valued for its soundness guarantees and scal­
ability, making it flexible and customizable[13]. However, it can be computationally inten­
sive and may generate overly complex models that are highly sensitive to the quality of
the event logs [4].
Conformance Checking
Conformance checking assesses how well a process model matches actual event logs,
identifying both commonalities and discrepancies. This technique is essential for ensur­
ing business alignment and for auditing purposes, as it highlights deviations that may
indicate inefficiencies or potential fraud. It also serves as a benchmark for evaluating the
performance of process mining algorithms [5].

Process Enhancement
Process enhancement uses the insights gained from discovered models and conformance
analysis to refine processes. By identifying deviations and incorporating additional data
such as timestamps, these models can be optimized to highlight bottlenecks and stream­
line commonly used pathways [1].
2.1.2
Ingredients for Process Mining
Event logs, Petri nets and BPMN models are some of the standard resource types in
process mining. We will shortly introduce these.
Event Logs
Event logs serve as the fundamental data source in process mining, cap­
turing records of events that reflect the execution of activities within a business process.
Typically, each event log includes essential components such as a case ID, an activity
label, and a timestamp, with additional details sometimes provided to enrich the context.
These logs can be stored in various formats like CSV, XML, and XES, with XES being a
widely adopted standard in process mining to ensure compatibility across different tools
[10].
Petri Nets
Petri nets are versatile tools for modeling concurrent systems and have be­
come integral in applications such as workflow management and distributed systems.
Introduced by Carl Adam Petri, these models are now a crucial resource for researchers
and professionals working in the field of process mining [14].
BPMN
Business Process Model and Notation (BPMN) is a graphical standard devel­
oped by the Object Management Group (OMG) for defining business processes within
workflows. BPMN provides a comprehensive set of symbols and elements that clearly
and intuitively represent complex business workflows, bridging the communication gap
between business analysts and technical developers [15].
2.1.3
Challenges in Process Mining
Process mining faces several key challenges as it matures. Technically, issues like scala­
bility with large datasets and maintaining data privacy during analysis are significant hur­
dles [1]. Additionally, applying process mining in real­world scenarios presents significant
security challenges, such as ensuring the confidentiality and integrity of sensitive data
during analysis, and implementing robust access control mechanisms to prevent unau­
thorized access to process insights.Adapting process mining methods to handle diverse
and dynamic data environments is another ongoing challenge [16].
2.2
Access Control
This section introduces various commonly used access control methods and analyzes
their respective strengths and weaknesses. By examining these methods, the paper iden­
tifies the access control approach best suited to the project’s needs.
2.2.1
Role­Based Access Control
Role­Based Access Control (RBAC) assigns permissions to roles rather than individual
users. This approach simplifies management by associating permissions with roles, which
makes permission management more intuitive and easier to maintain. Administrators can
manage roles and their assignment to users without needing to handle permissions for
each user individually[17]. RBAC also supports the definition of various roles with specific
permission sets, catering to different organizational structures and business needs. From
a security standpoint, strict role definitions and permission assignments ensure that users

have only the permissions necessary to perform their tasks, reducing the risk of misuse.
Additionally, RBAC provides clear records of permission assignments, aiding in audits and
compliance checks [18]. However, setting up RBAC in large organizations can be time­
consuming and complex. The system’s flexibility is also limited; it may not be suitable
when access decisions need to be based on contextual information like time or location.
In complex organizations, a large number of roles may be needed to cover all scenarios,
leading to a ”role explosion” that complicates management [19].
2.2.2
Attribute­Based Access Control
Attribute­Based Access Control (ABAC) makes dynamic access decisions based on user
attributes, resource attributes, and environmental conditions. ABAC offers high flexibil­
ity, allowing dynamic decisions based on various attributes such as user roles, resource
types, and access times[20]. This makes ABAC well­suited for complex and evolving
access control needs. Compared to RBAC, ABAC offers finer­grained control, enabling
precise definitions of various access scenarios and conditions. Its attribute­based rules
are also easy to extend and modify, adapting to changes in organizational structures and
business processes[21]. However, defining and managing ABAC policies is more com­
plex than RBAC, requiring deeper expertise and higher initial setup and maintenance
costs. ABAC can also impact system performance, particularly in environments with high
concurrency, as it requires evaluating multiple attributes and conditions during access
decision­making [22].
2.2.3
Rule­Based Access Control
Rule­Based Access Control manages access to resources through predefined rules based
on specific conditions, such as IP addresses or request types. This method allows or de­
nies access requests according to the defined rules. Its advantages include the simplicity
of rule definition and ease of understanding, making it suitable for straightforward ac­
cess control needs[23]. However, Rule­Based Access Control struggles with complex
and dynamic access control requirements. As the number of rules increases, manage­
ment and maintenance become challenging, potentially leading to rule conflicts and re­
dundancies[24].
2.2.4
Comparison and Discussion
In practical applications, RBAC is best suited for environments with stable organizational
structures and clearly defined roles, such as internal enterprise systems and government
agencies. ABAC, on the other hand, is ideal for environments with complex and dynamic
access needs, such as cloud platforms and large distributed systems. Rule­Based Ac­
cess Control is effective for simple and clear access control scenarios, like network firewall
rules or basic API access control. While RBAC is widely used for its simplicity and intuitive­
ness, it lacks the flexibility to handle complex and dynamic access requirements. ABAC,
with its high flexibility and fine­grained control, is a powerful tool for meeting these needs,
though its complexity and potential impact on performance must be carefully managed.
Rule­Based Access Control, despite its simplicity, falls short in handling complex access
scenarios. Therefore, it is often necessary to combine multiple access control strategies,
selecting the most appropriate method based on specific needs and environments. The
development of artificial intelligence and machine learning is leading to new access con­
trol approaches, such as behavior­based access control, which is gaining attention as a
potential future strategy [25].

shown that using Extended Access Control Markup Language (XACML) in IoT environ­

2.3
Related Work
With the growing interest in process mining, numerous tools have been developed, in­
cluding those from private companies and the open­source community. Examples include
ProM, Disco, the PM4PY library, and the web application Smyrida, each offering different
access control mechanisms. For instance, Smyrida, which shares similarities with our
project, lacks built­in access control. Some tools have begun to address access control
issues, such as the RapidProM extension and organization­sponsored projects like DTU’s
DAPM project.
2.3.1
ProM/RapidProM
The ProM development team collaborated with RapidMiner to create the RapidProM ex­
tension, which simplifies the use of process mining tools for non­expert users and enables
them to build pipelines containing process mining components [27]. In terms of user rights
management, RapidMiner offers advanced features such as role and permission control,
enabling the assignment of access rights to specific user roles and restricting access to
sensitive data for different user groups [17].
2.3.2
Disco
Disco is a desktop application that implements process mining algorithms, consistency
checking, filtering, and statistical visualization. However, Disco lacks a robust user rights
management mechanism, meaning that users must rely on external methods to control
access and ensure security during collaboration. The absence of built­in resource­sharing
capabilities further complicates privilege management, requiring users to manually man­
age these processes [28].
2.3.3
PM4PY
PM4PY is a popular open­source process mining tool available as a Python library. While
it supports various process mining operations, it does not include detailed user rights man­
agement and authentication mechanisms. However, PM4PY can be integrated into larger

allowing it to leverage the security features of these systems [29].
2.3.4
Smyrida
Smyrida is a recently introduced web application that uses a shared open­source modular
software system as its backend. It provides an open API interface for easy integration into
information systems and offers basic process mining functions like consistency checking.
In terms of user rights management, Smyrida is designed to be integrated into larger
systems that already have user management and security controls, meaning its user rights
management features may be basic but rely on the security measures of the integrated
system to provide reliable user authentication and rights management [30].

Process Dry Lab
Prior to our project, another group produced a project capable of deploying a simple pro­
cess mining pipeline called Process Dry Lab, which was the prototype for DAPM [7]. Since
this project focused mainly on process mining there was no access control involved.
2.3.6
A Distributed Architecture for Process Mining: Controlled Sharing
of Data and Algorithms
This article (DAPM CAiSE) explores the challenges and requirements faced in building
complex process mining pipelines in cross organizational environments. A distributed
architecture (DAPM) has been proposed to simplify the creation, deployment, and exe­
cution of process mining pipelines. This architecture not only supports analysts to share
data and algorithms, but also effectively protects the privacy of these resources. The ar­
ticle elaborates on the background of this issue and clarifies the access control system
required for DAPM. In this system, users can manage and maintain their credentials, and
assign them to algorithms or repositories in specific scenarios. By verifying the effective­
ness and necessary credentials of these scenarios, DAPM can ensure that only users
who meet the requirements can successfully deploy the pipeline[6].
2.4
Summary
In this chapter we explore the history and development of process mining and access
control. We have emphasized the challenges faced in process mining and pointed out
the importance of our research in facilitating secure sharing of process mining workflows,
data, and algorithms. We need to choose appropriate access control models for pro­
cess mining and data pipelines. This foundation sets the stage for the explorations and
contributions we make in the next chapters.

3
Problem
The goal of the DAPM project is to realize a platform where process mining pipelines can
be easily created, deployed and executed. This also includes helping analysts share data
and algorithms while ensuring the privacy of these resources. In this paper, we design
an access control framework on top of a prototype DAPM platform designed by two other
groups based on research from THE DAPM paper and PM DryLab. We need to identify
the appropriate type of access control and analyze what needs to be access controlled.
3.1
Problem Statement
Process mining researchers in different organizations will face the problem of access
rights, not only in cross­organizational projects in the organization will also face the situ­
ation that different departments have different rights. Also, when constructing a process
mining pipeline, the researcher may need to perform many different operations on the
components in the pipeline, and distributing and analyzing the permissions required for
these operations is also a major focus of the research.
3.2
Research Questions
With these considerations, we formulate the following research questions. These ques­
tions cover several key areas, including process mining pipelines across organizations,
fine­grained access control, the role of different access control models (RBAC and ABAC),
and the specific access control needs for various components such as data, algorithms,
and their implementation.
• How to achieve fine­grained access control in distributed process mining pipelines?
– How can user access to different data and process mining components be
effectively managed and controlled to ensure data security and privacy in dis­
tributed systems?
• What is the effectiveness of Role­Based Access Control (RBAC) in process
mining pipelines?
– How does RBAC impact system security and management efficiency?
– What are the advantages and limitations of RBAC compared to other access
control models?
• How can attribute­based access control (ABAC) models be utilized to improve
the security of process mining pipelines?
– How can ABAC be applied to process mining pipelines to improve security and
flexibility?
• How to achieve cross­organizational access control and data sharing in pro­
cess mining pipelines?
– How can effective access control mechanisms be designed and implemented
to ensure secure and compliant data sharing between different organizations
in a cross­organizational process mining environment?
• What are the applications and challenges of dynamic access control in real­
time process mining pipelines?

3.3
Scope

5.2
Access Control Strategies
Based on our user stories, it is evident that researchers require appropriate permissions
to build process mining pipelines. For security reasons, these permissions should be
managed by an administrator, who holds a different identity than the regular users. This
necessity leads us to implement Role­Based Access Control (RBAC), which simplifies
permission management by assigning permissions to specific roles. However, RBAC
alone is insufficient for managing diverse file permissions for users, hence the need for
an additional access control mechanism.
We propose the integration of Attribute­Based Access Control (ABAC) with RBAC. ABAC
makes access decisions by evaluating user attributes, resource attributes, and environ­
mental conditions, providing a more flexible and fine­grained access control. For instance,
researchers in an organization who wish to build a process mining pipeline using depart­
mental resources would find it cumbersome and complex to gain access if permissions
are granted on a per­resource basis.
To streamline this process, we propose granting users access to repositories where re­
sources are stored within their department. This method reduces the complexity of au­
thentication by minimizing the number of judgments needed. Additionally, the results
generated during the buildion of the process mining pipeline must be stored in a reposi­
tory. Therefore, researchers also require corresponding permissions to store these results
securely in the repository.
By combining RBAC with ABAC, we can ensure a more efficient and secure access control
system that meets the diverse needs of researchers while maintaining robust security
protocols.
5.3
Permission Type
With the user stories provided, we can categorize the operations into three types, corre­
sponding to the three levels in DAPM: organizational level, repository level, and resource
level. We will analyze here what users can do when they have the corresponding permis­
sions.
5.3.1
Organizational Level
The only type of operation is the creation of repositories at the organizational level.Users
can create a new repository in their organization. This operation should not be widely
granted. If all users have the right to create repositories, it will lead to an excessive
number of repositories, which can burden the server and increase the risk of attacks by
malicious users. Therefore, this functionality should be restricted to individual users with
specific permissions.
5.3.2
Repository Level
The second type of operation is at the repository level, which involves uploading resources
to the repository and operating on all the resources in the repository.
Uploading Resources
Users can upload resources in the repository. Uploading resources is a common operation
in the process of building mining pipelines. While this can be granted broadly, it should
still be subject to access control to prevent abuse. Malicious users could cause harm by
mass uploading files to the repository.
operate all resources
The user can operate all resources stored in the repository to allow easy access to multiple
resources. The corresponding permission types should be the same as those on the
resource level below.
18
Distributed Architecture for Process Mining
==PAGE:27==
5.3.3
Resource Level
The third type of operation is at the resource level. Accessing resources to build process
mining pipelines is the most common operation. This can be implemented at both the
resource level and the repository level.
Accessing Resources
Users can access specific resources. Both data and algorithms are used as a type of
resource that users need to use to build a pipeline. As a basic permission this should be
granted to all users who need it.
Deleting Files
Users can delete specific resources. Deleting files is a critical operation that should not be
widely granted. The appropriateness of this feature being broadly available depends on
the policies of different organizations. This paper focuses on access control mechanisms
and does not delve into the specific application of these policies. In conclusion, deleting
files is a potentially dangerous operation and should only be granted to individual users
with specific permissions.
Rewriting Resources
Users can rewrite specific resources. Rewriting resources is another operation encoun­
tered in process mining pipelines. This operation should only be granted at the repository
level. While rewriting a resource can be as threatening as deleting it, there are scenarios
in which this operation is necessary for pipeline buildion. For this reason, it can be granted
to general users, but only for individual resources.
Downloading Resources
Users can download specific resources. Downloading resources should be allowed for
researchers within an organization to access resources under their department. This
action can be granted at both the resource level and the repository level. For users outside
the organization, downloading all resources from a repository should be restricted for
security reasons. The specific policies regarding this may vary by organization and are
not discussed further in this paper.
Grant Permissions
Users can grant others permission to operate on resource.When building a pipeline users
need to be granted permission to do their work. This requires a corresponding user with
the ability to grant permission. Granting permissions as a special type of operation should
only be used by users with specific permissions.
Deploy/Execute Pipeline
Users can deploy/execute pipelines. Pipelines, as a special resource, are also stored in
the repository. If a user who is granted access to a repository can access other user’s
pipelines, this brings up some security issues. Therefore, permissions to deploy/execute
pipelines can only be granted at the resource level, which means that users cannot de­
ploy/execute pipelines even if the pipeline is stored in a repository to which the user has
permissions.
In summary, a balanced and carefully considered approach to access control is essential
to ensure the security and efficiency of the Distributed Architecture for Process Mining
(DAPM). The proposed access control strategies at the organizational, repository, and
resource levels aim to mitigate risks while enabling necessary operations for process
mining pipelines.

Evaluation
In this chapter, we evaluate the implementation presented in Chapter 6, focusing on the
design outlined in Chapter 5. The evaluation covers three main components: the user
microservice, the gateway, and the front­end interface. Additionally, we discuss the sys­
tem’s overall performance, scalability, and potential areas for future improvement.
7.1
User Microservice
In the user microservice, we effectively combined Role­Based Access Control (RBAC)
and Attribute­Based Access Control (ABAC) to implement our design. The user database
consists of six key columns that are essential for permission validation. RBAC was applied
using the Rolename, categorizing users into administrators and regular users, each with
distinct permissions. Additionally, five other IDs were used to implement ABAC, enabling
fine­grained permission checks based on these attributes.
For user authentication, we successfully implemented token issuance during login and
registration. This token works in conjunction with the gateway, forming a crucial part of
the user authentication mechanism. The microservice also includes a permission man­
agement system that allows administrators to assign permissions and modify user roles.
By interfacing with the user permissions system, we can retrieve the IDs of authorized
resources and repositories, which are used as conditions for front­end access control.
Although role models are typically static, in our design, users can dynamically switch
between regular user and administrator roles. This design allows users to switch roles
based on different tasks or contexts, enhancing the system’s flexibility and adaptability.
7.2
Gateway
The gateway plays a critical role in evaluating client requests. Upon user login, the gate­
way uses the issued token to authenticate the user’s identity. Once identity verification
is successful, the gateway utilizes the user’s ID and role to determine their permissions.
However, the current implementation has not yet fully integrated pipeline validation on
the front end. Currently, pipelines are only stored in the repository post­deployment, with
the front end directly invoking deploy and execute functions. This limitation means that
the pipeline ID cannot be pre­stored in user data, resulting in the ID changing with each
deployment.
Nevertheless, tests have shown that when the pipeline ID is unauthorized, the gateway
correctly identifies and rejects operations. For other interfaces, testing confirmed that
the gateway accurately determines permissions and calls the correct ports to execute the
required functions. This area will need further development.
Because user roles can be dynamically switched, the gateway re­evaluates permissions
based on the current role with each request. This dynamic role management proves
advantageous in handling complex and frequently changing scenarios, although some
challenges remain when dealing with certain resources. Overall, the functionality has
been well implemented.
7.3
Front­end
The front­end implementation showcases the completion of basic functionalities, including
login, registration, and permission management. Because user roles can be dynamically

that permissions are accurately enforced following each role switch. The current version

Regarding front­end permission validation, due to the project timeline of the Conceptual

Discussion
We have now proposed an access control design and completed its development. Our
product has gone through multiple versions of changes now addressing most of the re­
search questions, and some of these versions are also what we believe to be alternative
ways of thinking. We will discuss the results in this chapter and analyze them from three
perspectives. The first one is about solving our research problem, the second one is about
the design of the access control, and the last one is about the connection between the
Conceptual and Platform groups.
8.1
Solving Research Questions
We have designed and developed access control within our project, solving most of our
research questions through multiple iterations. Our current implementation of DAPM pro­
vides robust access control within an organization. We have implemented Role­Based
Access Control (RBAC) by categorizing users into two roles, and Attribute­Based Access
Control (ABAC) by linking permission­related IDs in the user database. Our gateway
system effectively evaluates operation requests and performs permission authentication,
ensuring internal access control.
However, for cross­organizational access control, there are limitations. Overall, our de­
sign meets the basic access control needs of the project, but future work is needed to
enhance cross­organizational functionalities. Our framework has been tested and proven
to be feasible and scalable, laying a strong foundation for subsequent work.
8.2
Access Control Design
Initially, our access control design was less detailed.
As back­end functionalities ex­
panded, we updated our control logic to accommodate additional features. Our design’s
extensibility allows us to implement new operation types by analyzing the operation and
associated role permissions, then configuring the appropriate interfaces in the gateway.
Initially, we categorized users into three groups: administrator, employee (for internal
users), and visitor (for external users), as illustrated below:

Future Work
9.1
Cross­Organizational Access Control
Currently, cross­organizational access control is not functioning as intended. User infor­
mation and permissions are manually input into the databases of other organizations, a
process that should be automated. Based on the handshake mechanism designed by the
Platform group, we devised a solution but did not have time to implement it.
9.1.1
Solution
When users connect to other organizations, their information should be automatically
stored in the respective databases, and permissions granted in other organizations should
be communicated back to their original organization’s database. This would allow users
to view all their permissions and enable seamless front­end access control.
9.2
Front­End Access Control
At present, front­end access control is not operational. Users should only see resources
for which they have permissions; resources without permissions should be invisible to
them. Based on the front­end pages designed by the Conceptual group, we devised a
solution but did not have time to implement it.
9.2.1
Solution
We need to add a mechanism to check the resources displayed. By using the user per­
missions interface designed in this paper, we can obtain the IDs of resources for which
users have permissions. Using the same method as the gateway, we can implement
front­end access control.
9.3
Pipeline Validation
The pipeline validation during construction is still at a basic level, according to the Concep­
tual group’s design. If users bypass the front­end to construct a pipeline, access control
does not take effect. Therefore, additional pipeline validation is needed. Through dis­
cussions with the Conceptual group, we confirmed a solution but did not have time to
implement it.
9.3.1
Solution
Currently, our pipelines are stored in JSON format, which includes information about the
resources and repositories used in the pipeline. This information is not transmitted through
the Platform group’s designed interfaces. The operator microservice will be responsible
for identifying the pipeline JSON and actually deploying and running the pipeline. By ob­
taining these information IDs, we can compare them with the contents of the user database
to achieve access control in pipeline validation.
9.4
Pipeline Access Control
In the current project, we can only obtain the pipeline ID during actual deployment. When
deploying and executing a pipeline from the front­end, the first step is to call the interface
to upload the pipeline to the repository, followed by deployment and execution. We can
only obtain the pipeline ID after uploading the pipeline, and multiple deployments of the
same pipeline will use different pipeline IDs. Through discussions with the two groups,
we devised a solution but did not have time to implement it.

Conclusion
This thesis presented a Distributed Architecture for Process Mining (DAPM), focusing pri­
marily on designing an appropriate access control method to ensure user management
and security. The main objective was to create an access control framework that facili­
tates user management, ensuring that the construction and execution of process mining
pipelines can be carried out securely.
Our solution is based on microservices, which manage user permissions through a vi­
sual interface. In the access control component, we designed a gateway to intercept
requests and perform permission checks based on the request type. Users can simply
add permissions in the microservice section to access the corresponding interfaces. Our
solution implements an access control framework for the DAPM platform. We analyze the
resources involved in constructing a process mining pipeline and identify the permissions
required to perform different operations on resources. This makes it simple for process
mining researchers to gain access to build pipelines, providing security for the platform.
Key areas of this thesis include:
Architecture for Distributed Process Mining:
We designed the types of permissions and criteria for judgment involved in process min­
ing, based on the visual front­end and back­end platforms designed by two other groups.
Our design enables access control at both the front­end and back­end levels to ensure
security.
Access Control Components:
Our access control is implemented through three main components: the User Microser­
vice, the Authentication Gateway, and the Visual User Management Interface. The User
Microservice handles user login, registration, and permission granting. The Authentica­
tion Gateway authenticates user identity and permissions, while the visual interface al­
lows users to manage permissions more intuitively. Both the user database in the User
Microservice and the Authentication Gateway are extensible, requiring only minimal at­
tribute additions to accommodate new access control needs.
Future Work and Improvements:
Although our access control design meets most requirements, some pre­designed func­
tionalities were not implemented due to time constraints and collaboration issues between
different groups. We have provided design ideas and solutions for these issues, which
can be addressed in future work.
In summary, this thesis designed an access control architecture for the DAPM, a pipeline
construction platform in the field of process mining, and demonstrated the feasibility of
access control in the DAPM project. Through access control, users are assigned roles
and varying permissions based on their operations, enhancing the security of pipeline
construction. Future research can build on the access control framework proposed in this
thesis to further refine access control mechanisms.

interactions between the client, user microservice, and gateway middleware.
