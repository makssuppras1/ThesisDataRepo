ABSTRACT
LoRaWAN is increasing in popularity more and more devices are coming online. Better
and easier to setup LoRaWAN gateways are needed. This thesis proposes a self-sustainable,
low-power, low-cost, wireless, compact, modular and open-source LoRaWAN Gateway.
The gateway prototype will works 24/7/365 all day every day and especially in the danish
winter weather. The thesis show what the specification of such gateway prototype should
be, the design and implementation of the gateway. The results show that even though
the tests has been conducted en favourable condition, which is in the summer it is still
possible to assume that it will work in the winter. This is because it is assumed that cloudy
days in the summer are very similar to cloudy days in the winter. The gateway prototype
consumes the less than the amount of power it produces in a cloudy day, this means in
any other sunny winter day, the gateway will produce surplus power. With this in mind
the prototype is design modular and easy to replicate, such that everyone else can build it
freely.

INTRODUCTION
1.1
Background and author’s motivation
Throughout my studies at the university I have gained a lot of experience solving practical
problems and solving problems that are usually not within my field of study. This is due
to the many extracurricular activities I have done. I co-founded DanSTAR where the
ambitious goal was to build and launch the first European bi-liquid rocket. I was both
a board member and the head of the telemetry team for 5 years. The telemetry team
was responsible for building the radio communication system between the rocket and the
ground area. In fact, this radio communication system is my bachelor thesis designed and
build by me from scratch. Beside DanSTAR, I am also the chairman of a HAM Radio
club called "Polyteknisk Radio Gruppe" at DTU. In PRG we spend time building our
own antennas, Yagi, helical and dipole antennas to communicate with other HAM radio
operators around the world. We also build and design our own PCB’s for any personal
electronics project and finally we maintain the open-sourced LoRaWAN Gateways at DTU
to allow students to use this technology for free - more on this further down.
Therefore, my motivations for this master thesis is to make something in which I have to
use as much as possible of the stuff I have learned from my extracurricular activities and
from the general DTU courses I have had prior to this project.
The idea of this master thesis started initially in 2019 when I was doing a course with Martin
Nordal Petersen and Sarah Renée Ruepp, called IoT Prototyping. I built a LoRaWAN
node tracker using WiFi for geolocation instead of GPS. The WiFi module of the tracker
would listen for at least two nearby different WiFi nodes to get their MAC addresses
and their respectively RSSI (received signal strength indicator) values. This data was
then transmitted back using LoRaWAN to a homemade server. The homemade server
would then send the data to google’s geolocation API, and the API would then return
the latitude and longitude of the tracker. Because the tracker was only listening for WiFi
nodes and used LoRaWAN (A low-power long-rang protocol) to transmit the data back to
the homemade server, it could be build very tiny and last practically years. I was proud of
this project and I wanted to share it with the world, but it turned out that there was a
big problem.
The problem was that, in order to relay the data back to my homemade server, I had to use
a private LoRaWAN network maintained by a company called Cibicom. This means that in
order for other students at DTU wanting to use my tracker or to build their own personal
projects that uses the LoRaWAN protocol, they would have to go through multiple tedious
steps. Firstly, they would need to know what Cibicom is, then know that my Professor
Martin Nordal Petersen exists, then know that he has contacts within Cibicom, then
convince my professor to convince Cibicom to give the them access to Cibicom’s private
LoRaWAN network. These steps could of course have worked out for the students, but in
my mind I believe most students would give up before trying.
So, I naturally thought there must be an easier way for students to start using LoRaWAN.
I did my research for 2 years, and found out that LoRaWAN has an open-source network
called The Things Network. This network is naturally the first thing the students would

find on google and then try to use, since it only requires you to make a login and that is it
- or at least that is what I thought. In fact, like the private networks, TTN also requires
that there are LoRaWAN gateways nearby in order for the student’s projects to work. So
in summer 2021, PRG bought one open-source gateway and deployed it at DTU Lyngby
Campus then pointing it to TTN Network. This gateway was powered and connected to
the internet via a POE (Power Over Ethernet) cable connection. I knew very well that
one gateway was not enough to cover the whole of DTU Lyngby Campus.
That is when I contacted Bo Carlsen Head of DTU Lyngby Campus Service, responsible
for everything about building and maintaining the buildings at DTU Lyngby Campus. He
told me that it was DTU who bought the already existing 4 LoRaWAN gateways and they
were also responsible of deploying them at DTU Lyngby Campus, but it is Cibicom who is
responsible for the backhaul to function. I told Bo that the current setup that the students
have to go through are too tedious to do, therefore we should deploy more LoRaWAN
gateways pointing to the open-source network TTN. He and I agreed that we needed at
least 3 gateways to cover the whole of DTU Lyngby Campus. He also stated, that it was
under the condition that PRG will buy the gateways and maintain the network and DTU
Campus Service will be responsible for mounting the gateways on the roof DTU buildings.
About 2 month later, we deployed another LoRaWAN Gateway, on the roof of a DTU
building. This gateway was deployed with minor mounting costs since the server room was
practically just below the gateway. The third and last one was tricky. I convinced Bo that

2.1.1
LoRaWAN (LoRa)
LoRa (Long-Range) is a patented wireless data communication protocol, developed by
Cycleo in France, which later was acquired by Samtech in 2012. LoRa uses the Chirp
Spread Spectrum (CSS) modulation. The LoRa advantages is in the technology’s long-rang
low-power capabilities. A single gateway or baste station can cover entire cites or multiple
kilometers in radius and a node if designed correctly can last for years with a relatively
small battery, compared to using cellular technology.
LoRa uses the license free ISM band, 863-870MHz in Europe and 902-928MHz in USA,
Canada and South America [The Things Network(2022)]. The technology is presented in
two parts - LoRa, the physical layer; the communication protocol built upon the underlying
LoRa physical layer. LoRaWAN, the data-link layer; where LoRaWAN is using Media
Access Control (MAC) protocol, which is built on top of the physical layer; the LoRa
modulation. It is a software layer that defines how devices use the LoRa hardware, for
example when transmitting data or what the format of the payload to be. LoRaWAN is an
open-source communication protocol defined by the LoRa Alliance consortium [LoR(2022)].

2.1.1.4
What is a LoRaWAN gateway?
Since this project is mainly about prototyping a LoRaWAN gateway, it is important to
explain what it is and what what goes inside one of them.
LoRaWAN gateways usually come in two categories; they can be bought for commercial
use or for DIY projects. The main difference between commercial and DIY gateways,
is that commercial gateways are way more expensive, better at handling harsh weather
conditions and can be bought having 16-channels for full duplex communication. This
project is about building a DIY gateway that has to work on TTN. TTN only allows for
8-channel gateways to be connected to the network, thus it does not make sense to use
a 16-channel gateway in this project. Years has past since the release of LoRaWAN to
the public in 2015 and with it the LoRaWAN gateways. Nowadays, a DIY gateway can,
with proper enclosure, be used as a commercial gateway - in other words the line between
commercial and DIY LoRaWAN gateways has been greatly blurred.
Both LoRaWAN commercial and DIY gateways are used for deploying a LoRaWAN
Network. The gateways are responsible of communicating with the end-nodes and to
relay the data back to the Network Server. Inside a commercial or a DIY gateway, one
can at least find 4 important components; A microprocessor, a LoRaWAN concentrator,
the necessary hardware to relay the data back, this could be hardware for WiFi, LTE or
Ethernet and finally sometimes a GPS module. Antennas are also needed; one for the
LoRaWAN concentrator and one for LTE module and one for GPS module, if GPS is used.
The microprocessor is usually any processing unit that can run the software, called Packet
Forwarder. Most DIY LoRaWAN gateway uses a clone version of Linux called Raspbian
as operating system to execute the Packet Forwarder. It is hard to say anything about
commercial gateways, but they either use Linux or run pure C++ code on a microcontroller
to execute the Packet Forwarder code. Figure 2.9 show two different open-sourced Linux
based microprocessor from the same family; Raspberry Pi [Raspberrypi(2022)], they are
usually used for other DIY project and DIY LoRaWAN Gateways.
The LoRaWAN concentrator is the actual component that makes a LoRaWAN gateway.
It is usually connected to the Raspberry Pi or any other Linux based processors and
communicate through Serial Peripheral Interface (SPI) ["Serial Peripheral Interface"(2022)].
The key job of the LoRaWAN concentrator is to modulate and demodulate the LoRa
messages then send the data back to the linux based microprocessor (Raspberry Pi) where
the Packet Forwarder will further send the data back to the Network Server. Figure 2.10
shows a LoRaWAN concentrator from RAK, that uses the first LoRaWAN concentrator
chip SX1301 from Samtech, it fits purposely well on top of a Raspberry Pi. More about
LoRaWAN concentrator further down.

WiFi
WiFi is a family of wireless networking technologies based on the IEEE 802.11 family
of standards. The IEEE 802.11 is a part of the IEEE 802 set of LAN protocols, which
specifies the set of media access control (MAC) and physical layer (PHY) protocols for
implementing Wireless Local Area Network (WLAN). WiFi communication spans over
different frequencies, including but not limited to 2.4 Ghz, 5 Ghz and 60 Ghz frequency
bands, these are all license-free bands (ISM). WiFi uses multiple parts of the IEEE 802
protocol family and is designed to work with Ethernet. Compatible devices can network
through wireless Access Points to each other, as well as to wired devices and the internet.
WiFi is promoted and developed by the WiFi Alliance non-profit organisation, they are
also responsible for certifying WiFi products ["Wi-Fi Alliance"()].

NB-IoT and LTE CAT M1
Narrowband Internet of Things (NB-IoT) and LTM CAT M1 is a standard introduced in
the Release 13, by the organization, Third Generation Partnership Project, 3GPP. Both
NB-IoT and LTE CAT M1 is a low-power, wide area network (LPWAN) technology that
operates on the licensed spectrum and is designed for IoT devices with long range, low
power and low bandwidths.
The standard on NB-IoT was completed in 2016 and was part of the telecom industry
answer to the rise of LPWANs such as LoRa and Sigfox. It only took nine months from
study to standardization which indicates that the IoT industry is being taken seriously by

the telecom industry. NB-IoT operates in a very narrow bandwidth - 180kHz and ccan be
deployed in the guard band portion of an LTE network, which is in-between channels in
an unused portion of the spectrum, thus allowing for the technology to co-exists with the
already deployed LTE network. NB-IoT features uplink speeds of 66Kbps and download
speeds of 26Kbps and it has a latency of 1.6 to 10 seconds. As a results of this it is best
suited for simple IoT devices that requires small, intermittent data transmission where
latency doesn’t matter.
Just like NB-IoT, LTE CAT M1 has the similar functionalities, but LTE CAT M1 is faster
up to 1Mbps upload and download speeds and has a much lower latency of 10 to 15ms.
CAT M1 supports Cellular tower handoffs, so it works with mobile applications such as
asset tracking and fleet management. This standard, which uses 1.4MHz of bandwidth,
also has enough throughput to transfer firmware, software and other security updated to
IoT devices, including a full-blown Linux operating system - something that NB-IoT can
not do. LTE CAT M1 supports full and half-duplexing, meaning companies can lower
power consumption by choosing half duplex. Figure 2.21 shows the different parameters
between LTE CAT M1 and NB-IoT.

PRODUCT REQUIREMENT SPECIFICATION
In order to fulfill the problem statement it is important to specify the requirement for the
product, otherwise it becomes to hard to conclude why the product is working. Furthermore,
some important assumptions and constrains are needed in order to prevent the project to

But firstly, the meaning of the title of this project is explained below:
3.1
The meaning of the project title
There are multiple important words in the title of this master thesis and they are all
thoroughly thought of. It is important to clarify the meaning of these words in the context
of this project and its prototype. The important words are the following:
Wireless - The idea here is that the LoRaWAN gateway prototype can not be wired to
anything, either due to the need of power or the need for data connection to send the data
back to the Network Server. One could call this gateway a "mobile" LoRaWAN gateway,
since it will be possible and easy to move the gateway around. Even if it is possible to
move the gateway around, the extra possibilities of that - what ever they might - are not a
part of this project.
Modular - The meaning of "modularity" is that, it should be fairly simple to replace a
part of the gateway without having to do greater changes to the rest of the gateway. This
makes it easier to iterate and upgrade the product. For example, if it turns out that the
solar panel is too small, they can easily be replaced by a bigger one, because the solar
panel mounting equipment are designed to handle different size of solar panels. Another
example could be replacing the old SX1301 LoRaWAN concentrator with the new SX1302
LoRaWAN concentrator, without having to change anything else other than maybe some
software. The same goes with using different LTE modems, just replacing one with another
should be easy.
Compact - If the are no requirement on how small a wireless LoRaWAN gateway prototype
should be then any other commercially available LoRaWAN gateway could be made wireless
no matter how big it might end up being. Commercial gateways consumes a lot of power,
because they where initially not designed to be wireless. This means that, multiple square
meters of solar panels and huge batteries are needed just to convert them to wireless and
make them work in the danish winter weather - and similar other weathers. Technically,
there is no problem in having multiple square meters of solar panels and huge batteries, as
long as one can pay for it and have the space for it. Practically, only a few people would
have the space and money for it, thus a compact LoraWAN gateway is required.
Low-cost - People should be able to afford to build one, test it and use it. The more
expensive it is the less people would want to buy the product and a wired LoRaWAN
gateway would be more attractive.
Open-source - The open-source philosophy is derived mostly from the author of this
project. Most of the equipment that are going to be used to build this LoRaWAN gateway

are themselves open-sourced. Using open-source electronics for the prototype and then
later open-sourcing the final prototype, will make the it quickly popular in the LoRaWAN

Low-power - In general the lower power the gateway consumes the more compact and

3.2
Requirements
There are many requirements for the prototype in order for it to work, they are the
following:
3.2.1
Power management
The optimal goal is to produce enough power to compensate for the power consumption,
such that power production and consumption are in an equilibrium through out the whole
day. When it comes to power management there are three ways that the product can
end up working, in order to explain the reason for this one must assume a worst case
scenario. The worst case scenario is the shortest day of the year and cloudy at the same
time, therefore it is also a requirement to optimize the product for the worst case scenario.
Below are the following three ways the product can end up working:
Average power production is greater than the average power consumption -
The solar panels are producing more power than the gateway is consuming. Because it is
the shortest day of the year and cloudy, then that means that the solar panel and battery
are sized too big. This will result in a higher product price than what actually is needed.
Page 23 of 121
==PAGE:36==
3. Product requirement specification
Average power production is equal to the average power consumption - This
is the scenario that one should aim for, because in this case the solar panel and battery
are sized correctly. The gateway will theoretically work throughout the winter without a
problem.
Average power production is less than the average power consumption - This is
a huge problem, because that means the gateway will last until the battery is fully drained.
Then it will shutdown and only restart when the battery is sufficient full. This scenario
must never happen otherwise the problem statement is not fulfilled successfully.
Since this product is optimized for the worst case scenario - meaning when it is the shortest
day of the year and cloudy at the same time - then that means in the summer the average
power production will be much greater than the average power consumption, but sadly
there is nothing that can be done to change that.
So the requirement for the power management, is that the average power production should
be equal to the average power consumption on the shortest cloudy day of the year. That
means the solar panels are pointing towards south at an angle where it is optimized for
the worst case scenario.
3.2.2
A compact product - size and weight
An other requirement is to make sure that the product is as compact as possible. It should
be one pole where one solar panel and one battery box is mounted on. This is needed to
reduce size and complexity. If the product requires two independent solar panels it will be
very hard to mount the solar panels on the pole and they might shadow each other. The
battery box can consist of smaller batteries inside of it, but there should be no more then
just one battery box. The individual parts of the prototype can not weigh more than 50kg
otherwise it is going to be very hard to move the gateway to the roof of a building.
3.2.3
Modularity
The requirement for this, is that the prototype must be build with modularity in mind
- to the point where everyone can change anything on the gateway without having to
drastically change how the product works. An example could be, changing LoRaWAN
concentrator from the power efficient SX1302 concentrator to the more power hungry
SX1301 concentrator. This could simple be because a person has the SX1301 already from
another project and wanting to save money by not buying the newer SX1301 concentrator.
The downside, is that the more power hungry concentrator will cause a higher power
consumption, therefore bigger solar panel is required. This is not a problem, since the
prototype is build with modularity in mind, the solar panel is quite easy to upgrade.
3.2.4
The cost of the product
The requirement here is that the wireless LoRaWAN gateway prototype should not cost
more than twice the price of a DIY wired LoRaWAN gateway, otherwise it might be a
challenge to compete against wired LoRaWAN gateways. But more importantly, if the
cost must be increased to fulfill the power management requirements, then that is okay,
otherwise the product will never work in the winter.

3.2.5
A wireless LoRaWAN gateway
It is required that the LoRaWAN gateway prototype is wireless, otherwise one could just
buy a wired gateway driven buy a PoE connection. The reason why the prototype has to
be a wireless solution is to save the installation cost of a PoE wire that goes from the server
room to the gateway that is on the roof. Furthermore, if there is a wired connected between
a LoRaWAN gateway and the server room, then money needs to be spend on protecting
the server room and the building from lightning strike. With a wireless LoRaWAN gateway,
this is not needed. If the wireless LoRaWAN gateway is also compact enough then it can
be moved around quite easily, but the pros of this - whatever they might be - are not
included in the project.
3.2.6
A open-source LoRaWAN gateway
This requirement is set by the author of this project. In order to get as much as possible
feedback on how well the LoRaWAN gateway prototype is doing, others must be able to
replicate the prototype, such that it can be tested and made better. This also means that
it is important to buy parts that are cheap and easy for people to find and also to try
to avoid building home-made electronics - this is sadly really hard but most home-made
electronics requires easy equipment to build that everyone can get hold on cheaply.
3.2.7
A self-sustainable prototype
The LoRaWAN gateway prototype is required to function by itself without the need of
an external wired connection. Self-sustainable means that it is both wireless and does
not need an external power source, it also means that it is a contained entity that must
be able to last for years. Ideally, it should last until the next version of the LoRaWAN
concentrator chip is released, this usually happens every 3. year. That is when one has to
go up to the gateway and replace the chip manually, but it is also when the battery can be
changed if needed. So the battery, solar panels and solar charge controller must last for at
least 3 years.
3.2.8
Maintenance and stability
Since this prototype is possible going to be mounted on the roof of a tall building, it is
important to make sure that stability is not going to be an issue. It is required that all
the individual modules have to be securely mounted on a 2m aluminium pole, that also
is securely mounted on the roof of a building. Since stability is important, it is also a
requirement to use components that are rigid and can be tidily secured on the pole. For
example, a flexible solar panel is a very bad idea, since the wind might catch it underneath
and rip it off the mounting structure. Furthermore, it is a requirement, for the maintenance
sake, that there is no use of mechanical moving parts, like a wind turbine - the prototype
must only use passive non-moving components.

which is essentially a breakout board, that contains all that is needed and some more. It
has an ESP32 microcontroller, a regulator/charger (marked as "REG" white white color) a
LoRa module (marked as "LoRa" with white color) a GPS module that is not going to be

battery can fit. The battery slot holder can not be seen on the picture, because it is on the
back side. Once the batter is inserted in the T-beam, and PWR bottom has been pressed

battery and the USB to feed all the other modules the correct voltage. It will also charge
the battery if the USB power is inserted. The ESP32 inside the T-beam is connected
to to the LoRa module via SPI, but this is not seen on the figure is it is a part of the
T-beam. The LoRa module has a small LoRa Antenna to transmit the data back to TTN.
Furthermore, there are two more components connected to the T-beam’s ESP32, it is the
AHT10 temperature and humidity (humidity is not used in this project) sensor and the
INA3221 power measurement sensor. Both components are connected to the same I2C line
which is usually power (red), ground (black), SCL (yellow) and SDA (green). The INA3221
has 3 channels and for each channel there is a shunt resistor that is used to measure the

Assuming that Raspbian is installed on the Raspberry Pi Zero W’s microSD card, the

the file, save the file and boot up the Raspberry Pi. The above steps allows us to disable
the HDMI module to reduce power consumption and to turn on UART.
Once the Raspberry pi has booted up, it is then possible to login into it using SSH. Once
in, run the following command sudo apt update && sudo apt upgrade, then install PPP and
minicom by running the command "sudo apt install PPP apt install minicom". This will
update the Raspberry Pi to the newest version. Inside the "/home/pi" folder add 3 folders
called; ""lora_1302", "lora_1308" and "run" from the GitHub repository, here: https://
github.com/FadiBunni/masterthesis/tree/main/LoRaWANGatewayScripts. Now run
the command "sudo crontab -e" and add the following in the bottom: "@reboot sh
/home/pi/run/powersave.sh" and add "@reboot py /home/pi/run/startLTEModule.py"
and also "*/30 * * * * sh /home/pi/run/wifioff_lteon.sh" then save and exit.
This
execute the files "powersave.sh" and "startLTEModule.py" on every boot up and the
"wifioff_lteon.sh" file will execute every 30 min. The "powersave.sh" file will turn off
the HDMI and a LED in the Raspberry Pi Zero to reduce power consumption. and the

How do we know how much power the Raspberry Pi zero W is consuming while the
concentrator is working?
By comparing test 10 with 9, we can get a pretty strong picture. Test 10 the Heltec

isolate it from the Raspberry Pi Zero W power consumption. Test 10 shows that the power
consumption of Heltec concentrator is 1.7W constant. Meaning if we take the results from
test 10 and subtract it from test 9 the answer is 2.3 −1.7 = 0.6W This means that the
Raspberry Pi Zero W is consuming 0.6W when in use with the raspberry pi this is 0.1W
more than when it is in idle mode and no concentrator is connected to it. This prove that
the bulk of the power consumption comes from the concentrator. This is not a surprise
since RF consumes a lot of power.

capacity otherwise the battery life cycle will be degraded heavily. In the winter it is known
that the battery capacity decreases with lower temperature, in fact for AGM batter it
is around 15% at -15 degrees Celsius. This means that the actual battery is only worth
33.22days · 0.85 = 28days. 28days is still close to a month. This is approximately 1/3 of

DISCUSSION
The biggest challenge with this project is that the test where conducted in the summer.
Which means that the assumption about the light intensity in a cloudy day in the summer
is the same as in the winter days. Obviously, this is not exactly correct. Since the sun will
rise much higher up in the summer than the winter, meaning less atmosphere to penetrate
through in summer than winter. But his assumption had to be done otherwise, the author
would have to wait for the winter to do the tests. To be exactly sure, the winter test will
be conduct!
The LoRaWAN gateway prototype did fulfil the overall requirements that are set. The
only downside is the cost, which is around 2-3 times higher than a DIY gateway. But the
setup is where one would save a lot of money. In fact it is not needed at install a POE
cable neither lightning strike equipment’s since the gateway is just an extension of the
building and not grounded to earth. One could chose to use a glass-fiber pipe instead of
aluminium rod to mount the gateway and reduce the chance for any lightning strike.
Once could argue that maybe the solar panels and the battery are over-dimensioned for
the power consumption. To reduce cost one could try to test with smaller solar panels and
batteries, and see the results in the winter.
Finally, with some more testing and documentation a step-by-step guide on how to build
this gateway prototype, it will be possible to open-source the gateway for the TTN
community.

CONCLUSION
It can be concluded that the project is quite successful and the problem statement has
been fulfilled. It was possible to design and build a wireless compact LoRaWAN gateway
prototype that supposedly will work in the worst period in Denmark, which is the cloudy
winter weather days. With the correct setup the gateway will consume around 1W, which
is 4-6 times less than a commercial gateway, depending on the type of gateway. The 80W
solar panel with combined with a MPPT solar charge controller can produce close to twice
as much power than the gateway consumes. With this setup and a big battery 110Ah, the
gateway should be up and running 24/7/365 days every day all day.

FUTURE WORK
There are multiple things that needs to be done to improve the prototype. The goal is to
be ready to do tests in the winter, this is where this product either succeed or fails - the
author based on the data that exists now is pretty convinced that it will work just fine.
If the prototype survives the winter without turning off a single time, then it is a huge
success. Looking at the data now it is possible to assume that it will work, but one can
never be sure until the winter tests has been made. Hardware and software wise, there are
plenty to do, below are some of the following task and possible tests to be done:
8.1
UART multiplexing
It is proven that UART consumes less power than USB cable does, when powering the LTE
module. So, it makes sense just to use hardware UART. The problem is that if UART is
used there is only one port, and if one wants to turn the prototype to a commercial product
3 UART ports are required; one for PPP connection, one for AT commands control and
the last for GPS, if ever used. It is possible to use something called AT+CMUX that the
SIM7080 LTE modem has, this commands turns one hardware UART to multiple virtual
UART connection. It does require some Linux kernal updtates to the Raspberry Pi Zero
and a lot of testing to confirm that it is stable enough for commercial use. But if this
works out then one can retain the low power consumption from using UART and also keep
multiple virtual ports at disposal just like the USB connection has. This will improve the
product hugely and allow for Over-the-Air updates.
8.2
Comparing MPPT vs PWM solar charge controller
All the tests conducted are done with the best setup witch is using the MPPT controller.
The MPPT controller is 3-4 times more expensive as the PWM controller. Even though it
is known that MPPT controller are much better in cloudy environment, maybe this is not
entirely true for the projects. Therefore, one should tests the difference between the two.
Two gateway prototypes needs to be build and placed close to each other one with MPPT
and with PWM then compare the data to make a cost-benefit analysis.
8.3
Light intensity of cloudy weather
In this project all the tests where conducted in the summer. But what is really needed is
to do the test in the winter. Obviously the author can not speed up time, so the tests had
to be done in the summer and assume that any cloudy day in the summer is the same in
winter when it comes to light intensity - not daytime. This assumption might turn out to
be a huge mistake to make, so the prototype needs to be prepared for a 2-3 month length
tests in the winter to confirm if it works through out the winter, if this is true then it will
definitely work 24/7/365 all day every day.
8.4
Latency of NB-IoT vs LTE-CAT M1 vs Ethernet
The latency when transmitting the data back to the Network server, might have big
influence on how the LoRaWAN stack (gateway, network server and application server)
