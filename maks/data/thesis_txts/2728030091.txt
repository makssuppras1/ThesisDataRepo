1
Introduction
Blockchain technology has disrupted various industries by enabling secure, transparent,
and decentralized transactions. Ethereum, the most popular blockchain platform, extends
the capabilities of blockchain by supporting the creation and execution of smart contracts.
Smart contracts are self­executing contracts that allow parties to transact without inter­
mediaries, ensuring transparency, and immutability. However, despite their advantages,
smart contracts are not immune to vulnerabilities.
The consequences of smart contract vulnerabilities can be severe. Over the years, nu­
merous smart contract hacks have resulted in millions of dollars in losses. For instance,
in 2016, the DAO hack resulted in a loss of approximately $150 million [1]. Similarly, in
2021, the Poly Network hack led to a loss of $600 million [2]. The industry is experienc­
ing more and more losses every year because of the vulnerabilities in smart contracts.
Last year, in 2022, there were 33 smart contract exploits which amounted to $1.25 billion
loss.[3]
Figure 1.1: Smart contracts exploits in 2022
To address the problem of smart contract vulnerabilities, researchers and practitioners
have developed various techniques for analyzing and testing smart contracts. One such
technique is fuzzing, which involves generating inputs automatically to expose vulnerabili­
ties. Fuzzing has been used successfully in software engineering to detect vulnerabilities
in applications. However, fuzzing smart contracts is a challenging task because of the
complexity of the Ethereum Virtual Machine (EVM) bytecode and the complexity of smart
contract states.
1.1
Objectives
The main aim of this project is to develop a black box Ethereum smart contract fuzzer that
will not require access to the source code of the smart contract for detecting vulnerabilities.

All other objectives of the project can be,
• Identifying common vulnerabilities in smart contracts and learning how to detect

• Improving the security of smart contracts deployed on the Ethereum blockchain.
1.2
Motivations and contributions
Considering the significant financial losses every year because of smart contracts vulner­
abilities and the increased adoption of smart contracts in various applications an effec­
tive tool for detecting the vulnerabilities automatically is in high demand. All the existing
Ethereum smart contract fuzzers seem to require access to the source code of the smart
contract, so those fuzzers are not useful in the situation when someone dont have access
to the source code but still needs to check for vulnerabilities on the deployed contract. In
this situation, a black box smart contract fuzzer can be very useful although it is not the
most effective solution for detecting smart contracts vulnerabilities.
All the contributions of the thesis are listed below,
• Design and implementation of black­box Ethereum smart contracts fuzzer that will
not require access to the source code of the contracts, instead it will leverage the
deployed contracts ABI and the EVM bytecode for generating fuzzing inputs and for
detecting vulnerabilities.
• Comprehensive evaluation of the black box fuzzer’s effectiveness against real­world
smart contracts.
• Enhancing the overall security and reliability of Ethereum smart contracts.
• Contribution to the growth, security, and reliability of blockchain technology.

2.1
Ethereum basics
In the past decade, our world has seen an unprecedented rise in the popularity of cryp­
tocurrencies. Among the various cryptocurrencies that have been released, Ethereum
has emerged as a leading player, revolutionizing the blockchain landscape [4]. While Bit­
coin paved the way for cryptocurrencies, Ethereum took the technology to new heights
by introducing smart contracts and decentralized applications (dApps) [4]. According to
a statistic from CoinGecko, the total market capitalization of cryptocurrencies reached
$1.88 trillion at the beginning of 2023 [5], with Ethereum being one of the top contenders.
This innovative platform has not only attracted investors but has also opened up endless
possibilities for businesses and developers.
The robust capabilities provided by Ethereum to blockchain technology apart from others
and contribute to its popularity [6]. Ethereum offers the creation and execution of smart
contracts, which are self­executing agreements with predefined conditions [4]. These
contracts eliminate the need for intermediaries, ensuring transparency, security, and ef­
ficiency in various sectors such as finance, supply chain management, and real estate
[4]. The decentralized nature of Ethereum ensures that no single entity has control over
the network, resulting in being resistant to censorship and tampering [7].
Moreover,
the Ethereum platform also supports the development of decentralized applications or
dApps. These applications run on the Ethereum network and provide numerous bene­
fits, including enhanced privacy, data integrity, and increased user control [7]. By utiliz­
ing Ethereum’s infrastructure, developers can create innovative solutions, ranging from
decentralized finance (DeFi) platforms to non­fungible tokens (NFTs) for digital art and
collectibles. The flexibility and versatility of Ethereum have attracted a vibrant community
of developers and entrepreneurs, contributing to its continuous growth and adoption.
2.1.1
Ether
Ether (ETH) is the native cryptocurrency of the Ethereum blockchain and is listed as ”ETH”
on cryptocurrency exchanges. It serves as a value token and is utilized for multiple pur­
poses within the Ethereum network. One primary use of ether is to pay for computational
resources and transaction fees associated with executing transactions on the Ethereum
network. Similar to Bitcoin, ether operates as a peer­to­peer currency. Additionally, ether
is used to purchase gas, which covers the computational costs of processing transactions
on the Ethereum network [8].

2.1.2
Smart Contract
Smart contracts in Ethereum are self­executing agreements written in code that oper­
ate on the Ethereum blockchain. They enable the creation of decentralized applications
(dApps) by automating processes and removing the need for intermediaries. These con­
tracts define the conditions and rules that govern their execution, such as triggering ac­
tions when specific criteria are met [7]. Developed using the Solidity programming lan­
guage, smart contracts are compiled into bytecode and deployed on the Ethereum net­
work [4]. When a transaction interacts with a smart contract, the Ethereum network vali­
dates and executes the contract’s code across multiple nodes, ensuring the integrity and
security of the execution. Once the predefined conditions are met, the smart contract au­
tomatically triggers the associated actions, such as transferring funds or updating data,
with the results recorded on the blockchain.
Figure 2.2: Smart contract on Ethereum
The decentralized nature of smart contracts in Ethereum brings numerous advantages.
It eliminates the need for intermediaries, reducing costs and enhancing efficiency. Smart
contracts also enhance transparency and trust as the code and execution are visible
and immutable on the blockchain, eliminating the risk of tampering or fraud [7]. With
Ethereum’s smart contracts, various industries can benefit from streamlined processes,
secure transactions, and the potential for innovative decentralized applications. Whether
it’s decentralized finance, supply chain management, or voting systems, smart contracts
in Ethereum offer a powerful and transformative solution for automating agreements and
revolutionizing digital interactions.
However, it’s important to recognize that smart contracts in Ethereum come with certain
risks that need to be addressed. Software bugs pose a significant threat, as coding errors
or vulnerabilities can lead to financial losses or security breaches. Protocol changes can
also impact the functionality of existing contracts. Furthermore, the unclear regulatory and
tax frameworks surrounding smart contracts add additional complexity and potential legal
risks (Swan, 2015). To mitigate these risks, thorough code audits, staying informed about
protocol updates, and seeking legal guidance are essential steps to ensure the secure
and compliant use of smart contracts in Ethereum.
2.1.3
EVM
The Ethereum Virtual Machine (EVM) is a key component of the Ethereum blockchain
network that allows the execution of smart contracts. It serves as a runtime environment
for executing code written in Ethereum’s native programming language, Solidity. The EVM
is a Turing­complete virtual machine, meaning it can perform any computation that can
be expressed algorithmically [10].

Figure 2.3: Overview of Ethereum virtual machine [10]
The EVM consists of three main components: the machine state, EVM code, and accoun­
t/storage (Ethereum Foundation, n.d.). The machine state represents the current state of
the EVM, including information such as account balances, contract code, and storage.
The EVM code is the bytecode representation of the smart contract that is deployed on
the Ethereum network. It is a low­level, stack­based language that the EVM can under­
stand and execute. The EVM processes the instructions in the EVM code sequentially,
manipulating the machine state and performing the desired operations [10].
2.1.4
Transaction and Gas
A transaction is a fundamental operation in Ethereum that involves the transfer of value
or the execution of a smart contract on the Ethereum network. It represents a request
or an action initiated by an account to modify the state of the blockchain. Each transac­
tion consists of essential elements, including the sender address, recipient address, the
amount of Ether (cryptocurrency) being sent, and optional data for executing smart con­
tracts. As the transaction changes the state of the EVM it needs to be broadcast to the
whole network. Anyone on the network can request a transaction to the EVM and if it gets
validated the state change gets visible to everyone [11].
Gas is a unit of measurement in Ethereum that determines the computational effort re­
quired to execute a transaction or perform a smart contract operation. It serves as a
mechanism for allocating resources and preventing abuse on the network. Every trans­
action in Ethereum consumes a certain amount of gas, depending on its complexity and
the computational operations involved. Gas is priced in Ether, and the transaction sender
needs to specify the gas price they are willing to pay for each unit of gas consumed. The
total cost of a transaction is calculated as the gas used multiplied by the gas price [12].

2.2.1
What is Fuzzing?
Fuzzing, also known as fuzz testing or fuzzing, is an automated software testing tech­
nique that involves feeding malformed or random inputs (known as ”fuzz”) into a target
program to discover vulnerabilities and crashes. Fuzzing aims to uncover unknown bugs
and security weaknesses by testing the program’s response to unexpected or invalid in­
puts [13].

==PAGE:16==
A general software fuzzing process begins by starting the target program, ensuring its
successful initiation. Tests are then supplied to the program, which can be done internally
or externally by sending tests to it. While a test is being executed, constant monitor­
ing of the program is essential to gauge its responsiveness and detect any crashes. If
the program becomes unresponsive for a certain period, it is assumed to be hanging,
prompting the termination of the program and a return to the initial step. In the event of
a crash, a thorough triage is conducted to determine if the issue is a known problem or
not. Depending on the outcome of the triage, the tests executed thus far are saved for
future reference, enabling the reproduction of the problem at a later time if needed. This
systematic approach ensures efficient testing, detection of issues, and preservation of
valuable information for troubleshooting and further analysis [14].

2.2.2
What are types of fuzzing
Primarily based on the level of knowledge and access to the target system, fuzzing can
be divided into three types, black box fuzzing, grey box fuzzing, and white box fuzzing.
An overview of the types can be seen in figure 2.5.
Figure 2.5: Different fuzzing approaches
Black­box Fuzzing is the type of fuzzing where the tester has no knowledge of the in­
ternal structure or implementation details of the target program. Fuzzing is performed
by generating random or mutated inputs and observing the program’s behavior. This ap­
proach is useful for testing closed­source or proprietary smart contracts where the internal
workings are not accessible [15].

White­box Fuzzing, is a fuzzing approach that involves having access to the source
code and leveraging this knowledge to generate more intelligent and targeted test cases.
Test inputs are generated based on the program’s control flow, data flow, and specific
conditions. White­box fuzzing is highly effective in identifying deep vulnerabilities within
the smart contract’s code [16].
Grey­box fuzzing combines elements of black­box and white­box fuzzing. Testers have
partial knowledge about the target program, such as the API or data structures. This
information is used to guide the fuzzing process and improve the quality of generated
inputs. Grey­box fuzzing strikes a balance between the two approaches, offering a good
trade­off between efficiency and test coverage [17].
2.2.3
How to fuzz smart contracts and why?
Fuzzing smart contracts involves generating and providing inputs to the contract in order
to discover potential vulnerabilities or unexpected behaviors. There are a few exciting
fuzzing tools available in the market, and the fuzzers also follow different approaches. In
general, fuzzers for detecting smart contracts can include the following steps. A general
overview can also be seen in fiqure 2.6.
• Determining the specific objectives of the fuzzing process. It could be discovering
vulnerabilities, testing specific contract functionalities, or evaluating contract robust­
ness.
• Choosing an appropriate fuzzing technique based on the determined goals and
available resources.
• Generating a set of valid and invalid inputs that cover a wide range of scenarios.
This can include crafting malicious inputs, boundary values, edge cases, or random
inputs.
• Running the fuzzer by providing the generated test inputs to the smart contract.
Monitor the contract’s behavior during fuzzing, observing for crashes, exceptions,
unexpected outputs, or other indicators of vulnerabilities.
• Analyzing the results of the fuzzing campaign and identifying any detected vulnera­
bilities or unusual contract behaviors.
• Addressing the identified vulnerabilities by fixing the smart contract code and con­
ducting further testing and verification to ensure the fixes are effective and do not
introduce new issues.

2.3
Smart contract vulnerabilities
There are several smart contracts vulnerabilities have already been identified which is just
not only making significant fund losses but also damaging the reputation of the blockchain
technology. Here in this section, ten common vulnerabilities are explained.
Reentrancy vulnerability occurs when a contract allows an external contract to call back
into its own code before the previous execution is complete. An attacker can exploit this
vulnerability to manipulate the contract’s state and perform unauthorized actions repeat­
edly, leading to financial losses or unintended behavior [19]. A very famous example of
reentrancy will be The DAO attack. An anonymous hacker managed to drain most of the
$150M worth of ETH from the DAO’s smart contract within just a few weeks [20].
Integer Overflow and Underflow, these vulnerabilities arise when mathematical opera­
tions result in values that exceed the maximum or minimum limits of the data type. At­
tackers can exploit these vulnerabilities to manipulate calculations, potentially leading to
unexpected behaviors, data corruption, or financial losses [19].
Unchecked External Call vulnerability occurs when a smart contract makes an external
call without properly validating or handling the return values. An attacker can exploit this
vulnerability by deploying a malicious contract that reverts the transaction or performs
unintended operations, leading to financial losses or other undesirable outcomes [19]
[21].
Front­Running, it refers to the practice of a miner or an attacker intercepting and prior­
itizing their own transactions to exploit price fluctuations or gain an unfair advantage. In
the context of Ethereum smart contracts, front­running attacks can occur when sensitive
operations, such as trades or auctions, are susceptible to manipulation due to transaction
ordering [22].
Denial of Service (DoS)vulnerability in a smart contract allows an attacker to consume
excessive gas or computational resources, leading to a degradation of service or render­
ing the contract unusable. This can be achieved through infinite loops, recursive calls, or
resource­intensive operations that exhaust the contract’s resources [19].
Timestamp Dependence Vulnerabilities can arise when contracts rely on the timestamp
for critical decisions or calculations. Manipulating the timestamp can lead to undesired
outcomes, such as exploiting time­based functionalities or front­running attacks [19].
Unhandled Exceptions and Errors, failure to properly handle exceptions and errors can
result in unexpected contract behavior. This can include scenarios where the contract
fails to handle out­of­bounds array access, division by zero, or invalid input, leading to
unintended consequences or even contract failure [19].
Access Control Issues, inadequate access control mechanisms can allow unauthorized
users to perform privileged operations or modify sensitive contract data. This can lead to
unauthorized transfers of assets, unauthorized modifications, or unauthorized access to
restricted functionalities [19].

Incorrect Access Control Lists, misconfiguration or improper management of access
control lists can grant unintended permissions to unauthorized users. This can result in

Logic Errors occurs when the intended behavior of a smart contract does not match its
actual implementation. These errors can lead to vulnerabilities such as unintended con­
ditions, incorrect calculations, or flawed decision­making processes, potentially resulting
in financial losses or undesired outcomes [19].

3.1
Echidna
Echidna a Haskell[23] program has been developed by Trail of Bits, a renowned and
leading cybersecurity firm. It stands out with its innovative approach known as ’property­
based fuzzing,’ which focuses on disproving user­defined invariants or properties, rather
than solely seeking crashes like conventional fuzzers. This distinctive methodology sets
Echidna apart, as it enables comprehensive testing of complex contracts. To further en­
hance its capabilities, Echidna provides a wide range of tools for crafting custom analy­
ses specifically designed for intricate contracts. With a specific focus on Ethereum Virtual
Machine (EVM) bytecode, Echidna utilizes the powerful cryptic compile to effectively test
contracts and uncover potential vulnerabilities [24][25][26].
Echidna’s architecture is divided into two steps, preprocessing, and the fuzzing campaign.
it begins with a set of given contracts, and the plus properties are integrated into one of
the contracts. In the first step, it leverages Slither[27], the smart contract static analy­
sis framework, for compiling and analyzing the contracts to identify useful constants and
functions that handle Ether (ETH) directly. Next in the following step, the fuzzing cam­
paign starts. The process uses the application binary interface (ABI) to generate random
transactions provided by the contract, crucial constants defined in the contract, and also
any previously collected sets of transactions from the corpus. When a property violation is
detected, a counterexample is automatically minimized to report the smallest and simplest
sequence of transactions that trigger the failure.[24]
Figure 3.1: The Echidna architecture [24]
Overall, Echidna is an effective, and fast fuzzer for Ethereum blockchain smart contracts.
it offers an out­of­the­box fuzzing experience with little setup or preparation but allows for
considerable customization. Along with custom property­checking, it supports assertion
checking, and estimation of maximum gas usage—a core feature set based on experience
with security audits of contracts. However, it requires access to the smart contract code
as an input in the preprocessing to generate random values for fuzzing the contracts.

3.2
ContractFuzzer
ContractFuzzer is an open­source Go[28] program that has been developed by a team of
researchers and developers. It tests Ethereum smart contracts for security vulnerabilities
by generating fuzzing inputs based on the ABI specifications of smart contracts, defining
test oracles to detect security vulnerabilities, instrumenting the EVM to log smart contract’s
runtime behaviors, and analyzing these logs to report security vulnerabilities [29] [30].
ContractFuzzer is a smart contract fuzzing tool that comprises an offline EVM instrumen­
tation tool and an online fuzzing tool. The offline EVM instrumentation process involves
instrumenting the EVM to enable monitoring of smart contract execution for vulnerabil­
ity analysis. Additionally, ContractFuzzer includes a web crawler that extracts deployed
smart contracts, their creation code, ABI interfaces, and constructor arguments from the
Ethereum platform. The online fuzzing process analyzes the ABI interface and bytecode
of the target smart contract, generates valid and mutated fuzzing inputs, and performs
fuzzing by invoking random functions. During fuzzing, ContractFuzzer analyzes execution
logs to detect security vulnerabilities, continuing the campaign until the available testing
time is exhausted.
Figure 3.2: Overview of the ContractFuzzer tool [30]
Overall, ContractFuzzer combines offline EVM instrumentation, web crawling for smart
contract extraction, and online fuzzing with analysis to comprehensively test and analyze
smart contracts. Its systematic approach helps identify vulnerabilities, enhance security,
and improve the reliability of smart contracts in the Ethereum ecosystem.

3.3
ChainFuzz
ChainFuzz has been developed by ChainSecurity, a prominent security company special­
izing in blockchain technology. It is an open­source tool available on GitHub [31]. It has
been designed specifically for Ethereum­based smart contracts.
In terms of functionality, ChainFuzz employs a mutation­based fuzzing approach to sys­
tematically test smart contracts for vulnerabilities. By generating a variety of inputs, includ­
ing invalid, unexpected, and mutated inputs, ChainFuzz explores different execution paths
within the contract’s code. This process aims to uncover potential vulnerabilities that could
lead to exploits or unintended behavior. Additionally, ChainFuzz incorporates coverage­
guided fuzzing, leveraging the coverage information obtained during the fuzzing process
to guide subsequent input generation. By prioritizing unexplored code paths, ChainFuzz
maximizes code coverage and increases the likelihood of discovering vulnerabilities.
The integration of ChainFuzz with the Slither[27] static analysis framework further en­
hances its effectiveness. By combining dynamic analysis techniques with static analysis
results, ChainFuzz leverages valuable insights extracted from the contract’s source code
before initiating the fuzzing campaign. This integration improves targeting and enables a
more comprehensive approach to vulnerability identification. Overall, ChainFuzz stands
as a powerful tool for smart contract security, offering a combination of mutation­based
fuzzing, coverage­guided testing, and integration with static analysis, all aimed at proac­
tively identifying vulnerabilities and bolstering the security of Ethereum­based decentral­
ized applications.
3.4
Foundry
Foundry is an open­source Ethereum smart contract fuzzer written in Rust[32] by Rodrigo
Herrera and it can be found on GitHub as a repository.[33] Foundry generates random
input call data for fuzzing transactions but does not generate the sequence of transactions.
Moreover, it also requires access to the source code in the fuzzing process.
In addition to the fuzzing, Foundry includes property­based testing which is a method for
testing software by defining properties that the contract should possess and using random
inputs to check if the contract adheres to those properties. Fuzzing is used to generate a
large number of inputs, while property­based testing is used to specify the general behav­
iors that the contract is expected to satisfy. By combining these two methods, it makes
sure that the contract is thoroughly tested for identifying vulnerabilities quickly [34].
3.5
Harvey
Harvey is a closed­source Ethereum smart contract industrial grey box fuzzer. Although
grey box fuzzing can detect bugs and security vulnerabilities effectively, however, it also
faces several challenges in covering code protected by narrow checks and exploring dif­
ferent states of smart contracts. For addressing these challenges, Harvey employs two
key fuzzing techniques one is extending grey box fuzzing by predicting new inputs which
are likely to uncover new paths or bugs in smart contracts. And next, it fuzzes transaction
sequences in a targeted and demand­driven manner. And this approach has been tested
on 27 real­world contracts, illustrating significantly improved effectiveness in achieving
high coverage and detecting vulnerabilities at a much faster rate. The experience gained
from using Harvey has also provided valuable insights into contract code. Overall, the
fuzzer is capable of performing bug detection very well, but as it is not open source and
so the access is limited [35].
