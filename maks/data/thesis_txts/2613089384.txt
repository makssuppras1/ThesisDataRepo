Summary
The increasing complexity of modern software systems and the constraints under
which these systems are developed pose limitations in the amount of testing and
quality assurance practices implemented. Being able to predict the quality of
a release, before the software is made available to customers, can be of great
value and can help in deliver higher quality software. Bug prediction is a ﬁeld
that aims at exploiting software metrics in order to predict faults early in the
development process.
On the other hand, Continuous Integration is an increasingly adopted practice
which requires developers to often integrate their changes on a shared repository,
and automates the execution of several steps which ensures the code meets
certain standards during development.
Our project has investigated a new type of metrics derived from the Continuous
Integration pipeline and their performance in the bug prediction task. In prac-
tice, we utilized data from a software project’s continuous integration pipeline
to predict the number of post-release bugs for software releases.

Chapter 1
Introduction
The goal of this chapter is twofold: to introduce the reader to our work and
to function as a short, but representative, summary of the key steps followed,
results obtained and limitations. In depth explanations, and details, of each of
the here presented elements can be found in the relative chapters.
We start by initially describing the problem statement and the relative thesis
objective. Subsequently we outline the methodology followed to research the
topic and the contributions of our work. We then present the main limitations
and ﬁnally the thesis structure where, for each chapter, a short summary follows.
1.1
Problem statement
The popularity and adoption of software products have increased drastically in
the last decades. Software applications are employed in many diﬀerent ﬁelds,
ranging from entertainment to critical applications. Software systems have also
experienced an increase in complexity, which allowed them to become more
powerful and keep up with the market requirements. These products are usually
developed under various constraints, such as time, resource or budget [ÖT18],
and this aspect represents a challenge when it comes to assess the quality of the
product and deliver fault free software. Indeed, requirements and constraints
==PAGE:14==
2
Introduction
may vary from one ﬁeld to another, but one common ambition is to develop
software products with no faults. Faults, or bugs, are intended as deviations
from the expected behavior or the desired functionality, and software companies
are interested in detecting faults in the code as soon as possible, possibly before
the code is released to customers. The early detection of a bug helps solve it
faster, before it is deployed in production and before other software components
rely on that functionality, allowing to limit its consequences.
Depending on
the ﬁeld, a bug reaching the end-user may simply aﬀect the user-experience,
e.g. in case of entertainment, or cause serious damage, in case of safety critical
scenarios.
For these reasons it is important to develop solutions that can support the
decision of whether or not the software is faulty or ready to be released, by
exploiting data produced during its development.
1.2
Thesis objective
Continuous Integration (CI) is a development practice where developers inte-
grate their changes daily into a shared repository. Upon integration, a pipeline
of steps is triggered to ensure the quality of the code. This operation is per-
formed in an automated fashion by a CI server. The most common steps are:
build and test the code. The historical results of these executions are usually
stored on the server, representing a possibly valuable source of information on
the software development. For what it concerns the adoption of this practice,
in a 2017 survey from DigitalOcean [Qui20] 42% of the developers stated to
be using CI in their development workﬂow, while in a more recent survey from
mabl1 [Hec20] the reported number was 53%. This shows that CI is a popular
and increasingly adopted practice.
This thesis aims at exploring the data generated by the continuous integration
pipeline as a possible source of metrics, helpful in predicting bugs in the code.
Speciﬁcally we used a case study centered around an open-source project and
its CI data to investigate which metrics are informative and to what extent they
can help in predicting the number of post-release bugs for a software release.

1.3
Methodology
Literature research
The methodology used to investigate the thesis objec-
tive started with the literature research. We began by researching the existing
work performed in the bug prediction ﬁeld. This was done to collect knowledge
on the existing approaches and solutions. In particular, since the novelty of
our approach is based on a new type of metrics, we were mostly interested in
discovering what metrics had already been researched, to conﬁrm that no prior
work had used this type of data for the same purpose.
Secondly, we performed a literature research to identify the speciﬁc metrics
from Continuous Integration data that have been previously deﬁned, and for
what purposes they have been used. This was done to learn more about the
best practices and established standards when working with this type of data.
Moreover, some existing metrics, datasets or tools could possibly be reused in
our research. Unfortunately, none of the existing datasets was deemed complete
and up-to-date to be used in our research.
At this point, having assessed the novelty of our approach, and the lack of a
suitable dataset, we set out to create one which could serve as case study to test
the predictive power of CI metrics.
Dataset construction
We started by deﬁning the requirements to be satisﬁed
by the case study project, and then proceeded to ﬁlter out projects which did not
comply. We then compared the remaining ones according to the requirements,
and selected the one with the most suitable characteristics.
We performed data collection from the three data sources: version control sys-
tem (VCS), continuous integration server (CIS) and issue tracking system (ITS).
At each step of the data collection, we validated the retrieved data to ensure its
completeness, and analysed behaviors and patterns to extract useful insights to
exploit during dataset construction. Indeed, to guarantee high quality data in
our ﬁnal dataset, we investigated what software releases were eligible to be in-
cluded and what issues were relevant for our use case. Moreover, we downloaded
over 130.000 execution logs from the CI pipeline, and designed an ad-hoc log
parser to extract metrics from them. We also deﬁned several approaches to iden-
tify the relevant commits for each software release, and from these approaches
we selected two to compare and evaluate. This led us to obtain two diﬀerent
ﬁnal datasets, with the same set of metrics, but constructed using diﬀerent sets
of commits for each release.

Analysis
We used the constructed datasets to measure correlations between
the number of post-release bugs and the CI metrics, and trained machine learn-
ing models to estimate their predictive performance. During the preliminary
analysis performed on the raw data, we observed two distinct development pe-
riods with quite diﬀerent CI behaviors.
We therefore decided to divide the
datasets to represent each of the two homogeneous CI periods (similar number
of monthly builds and same build tools). We then repeated the analysis and
compared the results with the ones obtained from the complete datasets.
1.4
Contributions
The contributions of our work can be divided into two macro categories:
• The datasets and their construction methodology
• The analysis of the datasets
Datasets and construction methodology
The datasets construction is a
fundamental part of the contribution of this work. The methodology followed
to retrieve, manipulate and combine the raw data from the three data sources
is outlined in Chapter 4, but explained in details in Chapter 6 and Chapter
7. We believe this to be a methodology that can be applied to other projects,
and can guide other researchers, or practitioners, to exploit the historical CI
data in the ﬁeld of bug prediction.
Speciﬁcally, we have deﬁned what data
sources are needed (VCS, CI and ITS) and for what purpose. Considerations
and strategies to extract and combine the raw data from the data sources are
extensively presented and discussed.
We contribute by deﬁning the set of metrics to be extracted and evaluated
from the CI historical execution data. Our suggestion contains both previously
researched metrics and new ones based on both our knowledge and the literature
research on the continuous integration guidelines.
Moreover we divide these
metrics into two categories: execution metrics, obtained by aggregating data
provided by the CI server, and logs metrics, to be extracted from build logs.
In absence of a standard way to assign commits (and therefore CI builds) to a
software release, we suggested and discussed diﬀerent ways to solve this task.
Finally we selected two diﬀerent methodologies and used them to construct two
diﬀerent datasets. These datasets represent the result of the implementation of

the presented methodology and are made publicly available (see A.1) to be used

Analysis of the datasets
Our analysis of the aforementioned datasets is com-
posed by several tools and experiments. As tools we used correlation analysis
and machine learning models to evaluate the relationships and predictive power
of the suggested CI metrics.
As experiments, we compared results between
regression and classiﬁcation machine learning tasks, between the two datasets
(constructed using the two diﬀerent commit sets), and between the complete
datasets and the ones obtained by limiting observations to the diﬀerent homo-
geneous CI development periods.
The ﬁrst experiment showed that in all datasets the performance and stability
of the machine learning models improved when the task was simpliﬁed from
regression to binary classiﬁcation. Indeed, in some cases our regression models
could not stably outperform the baselines.
We therefore conclude that our
datasets struggle with a complex task such as regression, possibly due to the
small number of observations, but can handle binary classiﬁcation acceptably.
The second experiment did not show signiﬁcant diﬀerences between the results
obtained using the two datasets. The set of metrics identiﬁed as correlated was
similar, except for small diﬀerences, and machine learning models achieved sim-
ilar results, sometimes using the same metrics from both datasets. We conclude
that no approach provides better results in our scenario.
The third experiment compared correlation and performances between the com-
plete datasets and the ones limited to homogeneous CI periods. We observed
stronger correlations and signiﬁcantly better performance in the latter, an indi-
cation that even within the same project it is important to limit data to stable
periods in order to obtain meaningful metrics.
Finally summarizing what metrics were mostly correlated and used by diﬀerent
models we ﬁnd that diﬀerent scenarios has used diﬀerent metrics. However, the
most useful ones have been: bursts of failed metrics, duration of builds, the
number of skipped tasks, the number of builds per pull request and the number
of failed tests.

1.5
Limitations
Among the most signiﬁcant limitations our work presents, we have the project
speciﬁcity. Our datasets are constructed from data collected from one single
open-source project, and the validity of the consequent results is limited to
that project only. This work can however serve as case study, and practical
showcase, of what steps are needed, and what results may be obtained, by using
Continuous Integration historical data.
The second main limitation is that the bug predictions are performed at release
level. This level of precision limits the range of usability of the predictions. The
more speciﬁc the prediction is, the easier it is for developers to investigate it.
However, we believe to have performed the prediction at the lowest level allowed
by the available data, and that it is still a valuable information when it comes
to form an opinion on the quality of the software to be released.

Chapter 2
Literature research
The objective of this thesis is to investigate the feasibility of exploiting the
continuous integration-generated data to perform bug prediction on a software
product.
Researching the existing work in both ﬁelds to assess the novelty
of our approach has been the ﬁrst step performed. This has also allowed us
to gain knowledge, which we then applied during our own investigation and
implementation.
In the following sections we are going to present the literature review method-
ology and ﬁndings for each of the topics involved in our research. We therefore
divide the chapter in sections, the ﬁrst is about bug prediction and the second
is about continuous integration and related data analysis.
2.1
Software bug prediction
The goal of bug prediction is to detect faults and defects early in the develop-
ment process of software projects. This is used by test and management teams
to form an opinion on the quality of the software, or by developers in order
to more eﬀectively utilize testing and quality assurance resources on the more
fault-prone components. Bug prediction models are built using software metrics
collected from diﬀerent aspects of the source code and at diﬀerent stages of the
==PAGE:22==
10
Literature research
development. This topic has been studied for over two decades and therefore
much research has been produced investigating it.
The focus was on what metrics had been used for bug prediction, in order to
assess that the category we suggested had not been researched previously. We
therefore tried to answer the following question:
• What metrics have been used for bug prediction?
We also outlined the prediction techniques and the evaluation metrics generally
used. This served as inspiration for the technique and evaluation metrics used
later on in our research.
[PC13] explains that the words bug, fault and defect are used as synonyms in
literature. We will therefore, from here on, use these terms interchangeably.
2.1.1
Literature review methodology
We started our literature review by performing a generic query on Google
Scholar, which is a platform that can help to avoid publisher biases, as sug-
gested by [Woh14]. Our goal was to retrieve all the works related to bug pre-
diction. The query was constructed in order to be as generic as possible, using
the synonyms for the word bug:
"(defect OR fault OR bug) prediction" AND "software"
The query returned 25.500 results.
Given that the ﬁeld is mature and well
researched, we decided to select systematic review and surveys on the topic
with the following query:
allintitle:
prediction software
(defect OR fault OR bug)
("systematic review" OR survey OR "systematic literature review")
Patents and citations have been excluded from the search as they have been
deemed not relevant.
The query returned 29 results, of which only 27 were
unique. The results have then been manually examined in terms of their com-
pliance with the following criteria:

In the following sections we ﬁrst present the diﬀerent prediction categories,

employed in bug prediction. Finally we brieﬂy summarize what techniques and

2.1.1.1
Prediction categories
We here present the diﬀerent aspects which characterize the types of defect
predictions performed in literature.
Granualrity of fault predicition
Defect prediction can operate at diﬀerent
software component levels. Ideally, the more granular the level of prediction, the
more useful the prediction is in helping developers testing and ﬁxing the possibly
faulty component. The possible levels are: class, ﬁle, modules, package, entire
release.
Types of predictions
According to [RK19] the existing literature has built
models to predict various types of faultiness. The three main ones are:
• Binary Classiﬁcation: whether the software component is faulty or not
• Number of Faults: the number of faults in the software component
• Severity of Faults: the severity of the faults in the component. Little
research has been done on this type of task. The main reason is the little
availability of data about the severity of the faults
Development and release stage
The existing literature (e.g [PC13]) makes
a distinction between defects which are detected at diﬀerent points of the soft-
ware development and release process:

• Post-release defects which have been detected after releasing the software

2.1.2
Software faults datasets
The literature presents studies performed in diﬀerent contexts and types of
projects.
The following sections categorize the datasets used based on their
level of accessibility for other researchers.
2.1.2.1
Private/Proprietary datasets
These datasets are based on software metrics collected in private companies,
or industrial projects, and are therefore not made publicly available to other
researchers. Results from these studies can therefore not be replicated. Exam-
ples of this are: fault prediction performed by Microsoft Research on data from
Windows 8 releases in [Her14] and at Eriksson [AR19].
2.1.2.2
Partially available datasets
Datasets in this category are obtained by mining open source projects’ reposi-
tories. Popular examples of such projects are Eclipse and Mozilla. In these con-
texts only the source code and the faults information (usually available through
an Issue Tracking System such as the one on Github, Bugzilla or Jira) are avail-
able. Software metrics need to be extracted from the source code, and issues
need to be ﬁltered, as they may contain non-faults related issues as well. In-
deed, [RK19] stresses that in these cases extra attention needs to be paid to the
mining process as it can lead to biased results.
[RHTŽ13] introduces the term "partially available datasets" to deﬁne these
types of datasets. In their systematic review, they estimated that before 2013,
30% of the analyzed work on bug prediction used partially available datasets.

2.1.2.3
Publicly available software datasets
We are going to list and brieﬂy introduce the publicly available datasets which
have been used in literature for defect prediction evaluation and analysis. These
datasets have drawn attention in the defect prediction research since their re-
lease. Results and prediction performance obtained from these datasets should
be easier to compare with each other. Moreover, researchers can focus on build-
ing models and trying new techniques, instead of having to go through the time
consuming task of mining software metrics and faults information.
NASA datasets
The NASA datasets are collections of diﬀerent software met-
rics at both module and class level [Mal15], along with the number of faults in
the modules. There are 5 of these datasets: CM1, JM1, KC1, PC1 and KC2.
The diﬀerences between these datasets are the diﬀerent metrics contained in
each of them: McCabe, Halstead and LOC metrics are present in the diﬀerent
datasets. These data are the product of a NASA program called Metric Data
Program. They have been widely used within the ﬁeld of bug prediction, such
as in [MMQLCHJ16], [AS14] and [VG12].
Many critiques have been though raised to the quality of both the data and
the results obtained from them.
Both [GBD+11] and [GBD+12] discovered
the presence of duplicates in the dataset, which may lead training and test
data to contain common datapoints and therefore bias the results obtained. In
[SSSM13] the authors compared the results of studies using the NASA datasets
and found signiﬁcantly diﬀerent results: this highlighted the need for more
precise description of what pre-processing steps have been undertaken when
working with the datasets.
Promise repository
This online dataset repository maintained by the Uni-
versity of Ottawa contains several datasets, along with the NASA ones, which
have been used for defect prediction.
Metrics from diﬀerent releases of the
PROP project have been used in [RG12] in order to assess single and combined
contribution of Object Oriented metrics in predictions, whereas PROP along-
side with other 9 Java projects (e.g tomcat, lucene and xalan) have been used
to predict faults in [ZLXS18]. These datasets contain diﬀerent metrics, rang-
ing from module wise object-oriented metrics to LOC metrics along with the
faultness of the module.

2.1.3
Software metrics
In order to build models to predict defects in software, researchers have extracted
and quantiﬁed qualities and characteristics of the software to use as training data
for their models. These quantities are called metrics. Diﬀerent types of metrics
have been proposed. We combine metrics categories from [FN99], [RHTŽ13],
[PM20] and [RK19]:
• Traditional metrics
• Object-oriented (OO) metrics
• Code smells metrics
• Dynamic metrics
• Process metrics
• Test metrics
2.1.3.1
Traditional metrics
These metrics mainly focus on quantifying the size and complexity of methods
and modules.
In this section we are going to introduce the existing metrics
within the traditional metrics category.
Lines of code (LOC)
This metric counts the lines in a software module. It
measures the length of the code. As explained in [YZ10], this metrics is based
on the idea that the higher the number of lines code, the higher density of bugs.
Several experiments showed a U relationship between bug density and LOC: the
density tends to decrease up until a certain number of LOC (estimated between
200 and 400) but increases after that value.
LOC is a simple, programming language agnostic and in some cases eﬀective
metric. Its simplicity leads to some disadvantages: it does not take the line
complexity into account, or the logic that is expressed in those lines. When
using LOC to measure the size of a project, this metric fails in capturing all the
non-coding activities such as user documentation, requirement collection and
design, as explained in [Jon13].
==PAGE:28==
16
Literature research
Function Points (FP)
have been suggested as a metric for software product
size, by Alberecht from IBM in 1979 [Alb79]. FP overcomes some of the limita-
tions in measuring the non-coding activities of LOC. Function Points are a way
to estimate the size of a project very early in the process, even before any line of
code has been written. This is possible since FPs are derived based on the design
and user requirements: it deﬁnes speciﬁc types functions for each requirement.
In this way the metric is more oriented towards the business functionalities of
the product, making FPs very suited for business-type projects.
Cyclomatic complexity (CC)
Proposed by McCabe in [McC76], the goal
of this metric was to measure the complexity of the program and its testability.
In order to achieve this, the control paths (formed by conditional statements in
the code) were analyzed using concepts from graph theory. A program control
graph is built on the analyzed program, with an entry node and an exit node. A
node is a block of code with no conditional or loop statements. Each node can
be reached from the entry node and any node can reach the exit node. From this
graph, the maximum number of linearly independent circuits is calculated. This
is the minimum number of circuits needed to construct all the possible circuits
in the graph. The paper suggests therefore the cyclomatic complexity (the
number of linear independent circuits) as complexity measure.
Halstad complexity metric (HCM)
Proposed by Halstead in 1977, this
metric is based on the counting of operators and operands in the code, which
are the basic logic units. Operands are the object which operators operate on.
Operands can be variables whereas operators can be the symbols specifying the
type of operation to perform.
This metric does therefore measure the logic
volume. These numbers are then used to calculate other metrics such as
• Software vocabulary: by summing the number of distinct operands and
operators
• Expected software length: an estimation of the ideal length for the soft-
ware, given the operators and operands
along with others such as: Volume, Diﬃculty and Programming Eﬀort. The
HCM is language agnostic and easy to compute. It does suﬀer from not taking
any jump or branch in the code into account, which are important factors in
code complexity.
Traditional metrics resulted the most commonly used ones among all types of
software metrics in [Mal15], being used in 51% of the analyzed studies.

2.1.3.2
Object-oriented (OO) metrics
These metrics focus on capturing the speciﬁc properties of object-oriented (oo)
software. Diﬀerent metrics have been proposed within this category. This sec-
tion introduces and explains the most popular ones.
Chidamber and Kemerer’s metrics
These oo-metrics have ﬁrst been pro-
posed by the authors in [CK94] in 1994, as a way to measure speciﬁc properties
of an object-oriented software and support design and development decisions.
The inheritance tree formed by classes and their relationships in a oo-software
can be seen as a directed acyclic graph. The authors ﬁrst deﬁne some basic
concepts to measure properties of this polytree, upon which they build the sug-
gested metrics. They deﬁne the following concepts:
• Coupling: two classes are coupled if at least one of them can act upon
the other. In practice, if class X can invoke or operate on a variable which
belongs to class Y, it is said that X can act upon Y.
• Cohesion: is deﬁned as the cardinality of the intersection of the instance
variable sets of two methods. The fewer common instance variables, the
lower the cohesion between the two methods.
The proposed metric are:
• Depth of inheritance (DIT): the depth level of the class in the software
inheritance tree. This metric measures the degree to which the class is
inﬂuenced by properties of its ancestor classes in the tree.
• Number of children (NOC): the number of children of a class in the
inheritance tree. This metric measures the possible inﬂuence that the class
has on its descendants.
• Weighted methods per class (WMC): is deﬁned as the sum of the
complexity metric ci (deliberately not deﬁned in the paper in order to
maintain generality) per each of the n methods in the class: Pn
i=1 ci
• Coupling between object classes (CBO): the number of non-inheritance
related classes to which the class is coupled to.
• Response For a Class (RFC): counts the number of methods which
can be invoked in order to respond to an external method call (or message
received). This measures how diﬃcult it is to test a class, since it captures

Another popular set of metrics for OO software is the MOOD
metrics set. In [HCN98] the set is explained and evaluated.
• Encapsulation: two metrics are proposed in order to measure the en-

what methods each class can call, it calculates the percentage of hidden
methods in a system. Attribute Hiding Factor (AHF) does the same
for attributes in the system.
• Inheritance: two metrics are proposed with focus on measurng the inher-

percentage of methods of the system which are inherited and Attribute
Inheritance Factor (AHF) does the same for attributes.
• Coupling: All pairwise sets of classes are considered and their coupling
(either through message passing or reference to an attribute or method
of the class) is measured (0 or 1). The Coupling Factor (CF) is then
calculated for the system.
• Polymorphism Factor (PF): calcualted as the percentage of redeﬁned
inherited methods over all the inherited methods. It captures how many
of all the polimorphic situations have been exploited in the system.
In [LH93] proposed Coupling through message passing and Coupling through
abstract data typed, which are a version of CBO but only focused either on the
message passing relationships or declaration of classes within classes (classes are
seen as an implementation of the concept of abstract data type).
CK metrics resulted the most commonly used ones among OO metrics in [Mal15],

2.1.3.3
Code smells
Code smells represent signs or indicators which can be found in code and that
may hint at bigger and more serious problems. Usually badly structured code,
anti-patterns, not following best-practice represent code smell. This type of met-
rics and its possible use in defect prediction has been extensively analyzed in the
systematic review [PM20]. God Class and God method are the ones which
turned out to be most helpful in prediction. They represent classes and meth-
ods which have too many responsibilities and logic. In this sense they remind
of size and complexity metrics, which are extensively used in literature. Along
with these, Average number of anti-patterns and Number of comments
resulted good predictors as well. This conﬁrms the idea of self-explanatory code
being usually of quality (no need for long comments) and the presence of anti-
patterns a possible source of complexity and diﬃcult management which may
lead to bugs.
2.1.3.4
Dynamic metrics
The previously discussed traditional and object-oriented metrics are all
based on static analysis of the code.
No runtime behavior is analyzed and
measured. In the OO context, there is indeed a very clear distinction between
classes (the static deﬁnition) and objects (the runtime instantiated classes). A
static analysis can identify relationships between classes, but it cannot estimate
how often each of these relationships will be used at execution time, and hence
measure which are stronger than others. The relationship between static cou-
pling (CBO) and the dynamic counterparts, here discussed, has been evaluated
in [MP06] concluding that the two types of metrics do present independence,
conﬁrming the need also for dynamic metrics.
In [YAR99] this problem is addressed by introducing dynamic metrics such as:
calls (import) or is called by (export)
• Export Object Coupling EOCx(oi, oj): given the execution of an ap-
plication scenario x, the EOC represents the percentage of messages sent
from object oi to oj over all the massages exchanged. This metric mea-
sures the mutual coupling of objects, which can help in identifying tightly
coupled ones and better testing their interactions.
• Import Object Coupling; IOCx(oi, oj): similarly to EOC, the IOC
measures the other direction of the coupling, namely the messages received
by oi and sent by oj over all the messages exchanged in the execution of
the scenario x.

Authors in [ABF04] expand on the dynamic metrics EOC and IOC looking at

2.1.3.5
Process metrics
The development process and life cycle has also been used to extract metrics for
defect prediction and software quality evaluation. These metrics do not focus on
the code itself but more on the process which lead the software to be developed
over time. Data for these metrics is not always available: some metrics require
access to whole process of development, which can be diﬃcult if the subject is
a private or commercial product. On the other hand, other metrics are easily
obtainable from OSS repositories which give access to the history of changes.
We follow the categorization proposed in [RK19] for process metrics:
• Basic change metrics: such as Deleted LOC, LOC added, Max LOC
added, Loc deleted, Max LOC deleted from the softare’s changes history
have been suggested and used as basic process metric.
• Code bursts metrics In [NZZ+10], authors use the change burst:
a sequential series of changes in the commit history on the same mod-
ule/component/class. The minimum size of the burst and the burst gap
(the largest allowed time window with no changes, which does not break
the burst) are parameterized to allow the approach to be more ﬂexi-
ble.
Authors use this as concept to derive other process metrics such
as: TimeMaxBurst, TimeLastBurst, TotalBurstSize and so on. The as-
sumption behind these metrics is that a module which has been changed
often, is more likely to be faulted.
• Code churn metrics In [NB05] the concept of code churn, which is the
amount of changes that aﬀected a software unit over time, is used to derive
some relative metrics. The amount of changes are normalized after various
other metrics such as the LOC. Some of the metrics are: Churned LOC
/ Total LOC, Deleted LOC / Total LOC, Churn count / Files churned.
The assumption is that the bigger the change, the more likely defected the
software will be.
• Developer based metrics In [MKM+10] authors have explored metrics
extracted from single developers’ activities in the history of a project. The

[PCseq(d, Appv)] which is the list of commits performed by a developer d

2.1.3.6
Test metrics
In [FN99] approaches using testing metrics as predictors for residual defects are
discussed.
The ﬁrst approach, used both in the Japanese software factory approach and
by the IBM NASA Space Shuttle team, is to collect defect rates at the diﬀer-
ent testing phases (the last one being e.g integration testing) and predict the
number of residual defects for the ﬁrst x months from the product release, by
using statistical and extrapolation techniques and comparing the behavior of
the current software with similar behaviors of past products.
Test coverage
Another approach is to use test coverage related metrics. One
metric deﬁned as Test Eﬀective Ratio (TER), measures the percentage of blocks
of code tested over all the existing ones in the software. Three diﬀerent types of
TER have been introduced in [Bro72]. LCSAJ stands for Linear Code Sequence
and Jump and represents a block of code followed by a jump to another part
of the code. TER1 measures the eﬀective ratio for statements, TER2 for code
branches and TER3 for LCSAJs.
In [Vee90], authors have correlated TERs measured on random testing with
reliability of the software, showing that TER3 is the strongest correlated with
reliability.
Testing eﬀort
extensive research has been performed in the ﬁeld of software
reliability. Software reliability is deﬁned as the probability of not encountering
failures when executing software for a speciﬁc period of time. Stochastic models
known as Software Reliability Growth Models (SRGM) have been de-
veloped in order to predict either the probability of failures in periods of time
or the expected number of residual defects, given diﬀerent metrics such as: the
number of faults detected by testing up until that moment and the amount of
time spent testing (Testing eﬀort). These models can help developers decide
when to stop testing and assess whether the software is reliable enough to be

released. Diﬀerent function shapes have been proposed to describe these curves
such as Non Homogeneous Poisson Process (NHPP) [PG13] and exponential

Test execution metrics
as explained in [Her14], little exploration has been
performed on correlating execution metrics from testing to defects. In [Her14]
the author has mined the test results from integration tests running for the
Windows 8 software, with the goal of exploring the correlation of both pre and
post-release faults and the new derived text execution metrics. At Windows,
new features are implemented on feature branches which need to be merged into
the main branch. When merging, the code has to pass through several gates,
which are integration test suites that, among other things, test the code on
diﬀerent architectures. The author diﬀerentiates test results as either passed,
truly failed (True Positive, TP) and falsely failed (False Positive, TP) (e.g failed
not due to errors in the code but due to issues on the architecture running the
tests). The proposed metrics are grouped in 3 main categories:
• Counting test failures: in general, several diﬀerent metrics are pre-
sented in order to count diﬀerent types of failed tests (TP, FP) and at
diﬀerent levels of granularity (counting the number of diﬀerent architec-
tures and gates at which test failed).
• Test Failure Burst the concept of change burst (previously explained
for code changes) is here applied to test failures. Consecutive gate failures
are counted, using a burst size and a burst gap (from 1 to 3) as parameters.
The rationale is that the more times a code component causes tests to fail
the more likely it is that this component is defected.
• Code review metrics the number of manually performed code reviews of
the merged changes is also used as metric. Bugs which did pass a manual
code review, but were detected by tests, may either signify a low quality
of the code review, or hint at a complex code change which may hide more
undetected bugs.
These metrics have been used for building prediction models in order to predict
the number of pre and post-release defects of the three development milestones of
Windows 8. Results show a promising accuracy and suggest that test execution
data may be a good predictor for post-release defects.

2.1.3.7
Metrics performance
There seems to be a general consensus on the fact that a combination of diﬀer-
ent types of software metrics (e.g traditional static-code metrics together with
process metrics) leads to better performance, as concluded by both [HBB+11]
and [RK19]. [HBB+11] also reports that static-code metrics and object-oriented
metrics are outperformed by simple LOC metric. This is contrast with ﬁndings
from other papers which are mentioned in their work, and with what [RK19]
and [Mal15] report on object-oriented and traditional metrics usefulness. From
our analysis we conclude that there is no wide consensus on the best perform-
ing metrics. Researchers often argue the lack of reproducibility in studies which
claim to have obtained good results using speciﬁc metrics. For example [LSG20]
found that 25 of the 49 selected studies did not report if cross-validation had
been applied or not. Often not all the steps which have been conducted during
research are clearly mentioned and this can lead to diﬀerent results once other
researchers try to replicate the same study. It seems also that metrics and meth-
ods performance are very dataset dependent, as conﬁrmed by [LSG20] in their
systematic comparison of unsupervised and supervised prediction techniques.
2.1.4
Prediction techniques
In the following section we are going to summarize the methods which have
been used in literature for predicting bugs using software metrics. The main
two categories are statistical methods and machine learning models [CD09].
2.1.4.1
Statistical methods
The purpose of these models is to infer relationships between variables. They
make assumptions on the distribution of the sample data and express a mathe-
matical relationship between two or more variables. They are not optimized for
prediction, which is the main diﬀerence with machine learning. Some statistical
methods such as Logistic and Linear Regression, Naive Bayes and K-Nearest
Neighbors are classiﬁed as statistical in some papers [RK19] and as machine
learning methods in others [Mal15]. Discriminant analysis and correlation anal-
ysis are the mentioned statistical methods used in defect prediction along with
regression based models ([SAG15], [RK19], [ÖT18]).

2.1.4.2
Machine learning
There are two main categories in machine learning models: supervised and un-
supervised. This categorization is also used to classify techniques employed in
defect prediction ([AS14], [PP14]). As [AS14] explains, both supervised and un-
supervised methods have been used. Supervised methods are the most popular
ones as it is often possible to use previous releases’ data in order to build model
for future predictions.
Supervised models
In supervised machine learning, the training data con-
tains the correct prediction/output for each example, which the model should
learn to predict/output. In this way, the model is supervised in its learning, by
having access to the labels for each example. The model uses the examples and
the labels in order to build a model which aims at accurately output the right
labels given new observations.
In [AS14] authors tried diﬀerent techniques and concluded that Random Forest
is the model which performs best on both small and big datasets (4 NASA
datasets are used in this survey).
In [PP14] authors conducted a survey on
machine learning techniques applied to defect prediction by analyzing the related
literature. The comparison explains that Decision Trees provide better results
than other methods, even though the creation of the tree is an expensive and
complex task. Support Vector Machine gains from the use of multiple kernels
but is not suitable for large number of software metrics.
A systematic reviews of papers using machine learning techniques up to 2013
has been performed in [Mal15]. The most popular model was Decision Trees,
along with Bayesian Learners. Authors compared the performance of diﬀerent
models on 15 datasets. Random Forest outperformed the other models in 13
cases, resulting the best performing model. The study underlines however, that
in defect prediction there is no silver bullet.
The results showed that some
models may perform well with speciﬁc datasets and hence a tailored selection
needs to be performed in each case.
According to [HBB+11] Naive Bayes and Logistic Regression are among the
best performing techniques. This is probably also due to their simplicity and
well understood functioning. Other techniques such as SVM do not perform as
well: this can be due to the required complex tuning and optimization which not
all researchers are conﬁdent with. Decision trees seem to underperform when
dealing with imbalanced data. In general, authors notice how each dataset may
perform better with speciﬁc models, and how important is the tuning process
over the dataset instead of using default parameters.

Unsupervised models
Unsupervised training consists in not providing the
model with labels for the data. In this type of training, the model has to learn
to either group similar instances of data together (Clustering) or discover rules
and associations among features (Association Rule Mining).
This type of training is not popular among defect prediction techniques as it is
usually possible to access labelled data from previous releases and the history
of bugs of a software.
It is however handy when only software metrics are
available and faults data are not. In these cases the models will automatically
detect clusters of data points which share common aspects.
[LSG20] conducted a systematic literature review on unsupervised methods
applied to defect prediction.
The study focuses mainly on clustering tech-
niques such as: K-means, Hierarchical clustering, Neural Network, Fuzzy Logic,
Spectral Clustering, Expectation Maximization clustering. Authors have com-
pared the best performance of unsupervised and supervised models on diﬀerent
datasets and discovered that performance were similar and that the two types
of models outperformed each other roughly the same number of times. This
hints at unsupervised techniques being valid alternatives to supervised ones.
The comparison showed that it was mostly the datasets which drove the per-
formance, since both approaches had similar results (low and high) in all the
datasets.
Evaluation metrics
The selection of the right evaluation metric for machine
learning models is an important task as it allows to both assess how well the
model performs in the defect prediction task and to compare diﬀerent models
in order to select the best suited one.
Not all evaluation metrics capture the same aspects of the model performance.
Some capture the rate of correct predictions, others more the incorrect ones. It
is therefore important to identify the key elements of the context in which the
model performs in order to make sure to select an evaluation metric which is
able to capture them.
The evaluation metrics used depend on the type of machine learning task under
consideration: regression or classiﬁcation. The deﬁnitions of these evaluation
metrics can be found in the Chapter 3. The studies list a wide variety of metrics
used, depending on the task and context. For instance, in regression tasks the
most used evaluation metrics, according to [ÖT18], are the error based ones
(MAE and RMAE). On the other hand, for classiﬁcation [ÖT18] reports that
accuracy, AUC, precision, recall and F-measure are the most used evaluation
metrics in the defect prediction studies analyzed. This is conﬁrmed by [RK19]

2.2
Continuous Integration
The concept of Continuous Integration (CI) was ﬁrst introduced as part of the
principles of the agile software development methodology called Extreme Pro-
gramming. This methodology advocates for small and frequent development
cycles and releases.
It was only after the article from Martin Fowler from
ThoughtWorks [Fow20], that it gained popularity as a principle which could
be applied in any context, even outside the Extreme Programming methodol-
ogy [HTH+16].
The goal of Continuous Integration is to avoid the usual problems that arise
when developers integrate their changes after a long period of development on
their own version of the code. Possible conﬂicts and integration bugs may ap-
pear, and it is often time consuming to identify the root cause and resolve the
integration. The main source of problems is the fact that the two (or more)
versions of the code to merge have accumulated many changes overtime, and
they have possibly changed the same parts of the code or developed features
that do not work together.
The solution is to have one single source code repository and integrate changes
continuously. The source code repository is usually a version control system
(VCS, such as git) where all ﬁles related to the development of the software are
stored and versioned. There should be a mainline (or master branch) where the
latest working version of the software can be found, and developers should always
start working from a local copy of the mainline. At least once a day, ideally
even multiple times, developers should retrieve the latest version of the mainline
(checkout in git) and integrate their changes. This way if other developers have
updated the mainline, the integration is performed periodically on small sets of
changes. Once the integration is performed, developers should check the quality
of the code before updating (committing and pushing) the mainline.
These checks include the compilation and linkage of the code ﬁles of the software
(e.g. classes in Java), the packaging of the code into an executable and the exe-
cution of suites of tests, which ensure that the software works as expected. This
process should be automated, according to Fowler. Nowadays several services,
both hosted and locally deployed, exist which enable developers to conﬁgure the
automatic execution of these steps every time a new commit is pushed to the
mainline (master or branches). The execution of these steps is called build, even
though it may comprise of more than just building the code. A build is success-
==PAGE:39==
2.2 Continuous Integration
27
ful when all checks passed, however the success of a build is only as informative
as the quality of the checks executed. It is therefore important to maintain a
vast and thorough set of unit and integration tests to ensure the code executes
correctly.
Of vital importance is the quick feedback developers receive when they integrate
their changes. Running the entire suite of tests locally may not be practical.
They may therefore limit themselves to check that the code compiles and leave
the execution of all the checks to the CI server. If the execution of the build takes
long time or is delayed, developers may stop waiting and continue working on
a faulty version of the code in the meantime. Extreme programming suggested
10 minutes as a reasonable build time.
The execution of builds produces data in the form of the outcome, build logs,
duration and other metrics. Overtime, as the software is being developed, these
data are produced and stored on the CI server. This historical data of build
executions can represent a valuable source of data which can be exploited in
diﬀerent ways.
The literature research on the topic of CI data analysis is presented in the
following sections. We start by presenting the methodology followed and the
papers included. We then present the results of our review divided in main
topics: the existing datasets available, what the research has investigated with
the data and ﬁnally the type of metrics and the methods used to extract them.

an intrinsical limitation as it means that builds may be missing from the ﬁnal
dataset and bias the relative results obtained from it. The second is the fact that
the last available version has been collected in August 2017. In the time elapsed
from that date, more data has been produced by the involved data sources and,
depending on the requirements of the research conducted, it may be desirable
to work on a more extended and up-to-date dataset.

Research goals
Anti-pattern detection in CI pipelines has been the focus of [GM18] and [VPGDP19].
Authors have ﬁrst analysed general trends and practices in CI conﬁgurations
and execution. They then developed tools to detect deviations from normal
behaviors or anti-patterns (similarly to code smells in software) in the CI con-

aﬀects the speed at which developers obtain feedback and is at the core of the
CI principles.
Another topic of research is the correlation of several metric types with the
error in failing builds. [RHLS17] analyzed the data from CI builds for 14 OSS
projects in order to investigate the most common types of errors in failing builds
and analyze their correlation with diﬀerent development metrics. [DAS17] used
TravisTorrent to investigate what metrics from the same dataset contribute
to the failure of builds. Authors were able to ﬁnd only moderate correlation
between some of these metrics and the status of the builds. In [PFDM17] au-
thors analysed the relationship between non-functional requirements expressed
in commit messages and the relative statuses of builds. It showed that some re-
quirements tend to cause more broken builds and that the time elapsed between
a broken build and a sequent successful one presents a bimodal distribution
along all requirements.
The prediction of build failure before their execution has been the focus of
several papers.
[HW17] used TravisTorrent dataset to investigate statistical
distribution of build duration and argue that CI servers may experience overload,
e.g due to frequent commits, and feedback to developers may be delayed causing
them to continue working on a possibly broken version of the code. Authors
have ﬁrst extracted error messages from the build logs to identify clusters of
diﬀerent failure types, then used data from the dataset to predict which of
the clusters a build would belong to, before executing it.
This way, in case
of stall of the CI server, developers would obtain a quick prediction of the
feedback.
[XLW17] expanded on this idea by measuring the performance of
failure prediction cross-project with promising results. [CCZP20] developed a
predictor for build outcome based on both the features of the previous build
from TravisTorrent, but also on the changes introduced in the commit to build.
Moreover, authors have used historical data from the previous builds, such as
the ratio of failed builds and the number of consecutive failed builds: max,
average and sum were used as aggregators.

2.2.4
Metrics and extraction methods
Metrics
There seems to be a wide consensus on the CI metrics used for anal-
ysis. This is mostly imposed by the structure and functioning of the continuous
integration process. Builds are at the core of the execution and many metrics
are therefore calculated at this level.
The duration of the builds, their out-
come (passed, failed or other statuses), the tests executed, passed, skipped and
failed and error messages in the log ﬁles are common ones ([BGZ17], [PFDM17],
[MRDH20]). Some articles suggested to use the ratio of builds with the diﬀer-
ent outcomes and the consecutive number of builds in a state, where often the
failed state is the one of interest ([CCZP20], [Her14]). Other articles included
development metrics in their analysis, e.g. the number of lines of code modiﬁed
in the commit which triggered the build, but we believe these metrics to be
outside the strict scope of continuous integration.
Methods
The most common way to retrieve continuous integration data is
by querying the public API of the CI services used by the projects under
analysis.
As mentioned earlier, Travis CI seems to be the standard-de-facto
service used in research thanks to its popularity and publicly available API.
API usually provide diﬀerent endpoints and therefore access to diﬀerent kinds
of information. Endpoints returning build metrics are the most useful ones.
Not all metrics are directly provided via API calls. In the analysed papers it
is clear how build logs contain useful information that needs to be extracted.
The process is called log parsing. Logs need ﬁrst to be retrieved, which is
usually achieved via API calls, and then parsed to extract the metrics of interest.
Among the metrics mentioned earlier, the ones relative to the number of tests
executed and error messages required log parsing.
In [BGZ17] authors have
developed a build log analyzer which can interpret logs from diﬀerent build
management tools (e.g. gradle and maven) and extract test metrics. Authors
have made the tool publicly available on a Github repository2. In [RHLS17]
authors have implemented a log parser called LogCat, which parsed the log ﬁles
from failing builds. It extracted message patterns and relative error labels from
selected builds and subsequently labelled the other builds from the same project
matching their log ﬁles with the created patterns-labels mapping. Unfortunately
the parser is not open-source.

2.3
Conclusions
From our literature research we concluded that, in the context of bug prediction,
there is no such thing as a silver bullet. Several metrics have been proposed and
analysed, often with positive results. However, systematic reviews claim the lack
of reproducibility of many studies, for example due to unspeciﬁed steps in the
preprocessing of the dataset. Contradictory results have been obtained when
comparing the eﬀectiveness of metrics on diﬀerent datasets and diﬀerent context.
Indeed, as concluded by [LSG20], the performance of methods and metrics seems
to be dataset dependent. This suggests that new metrics are welcome in the
plethora of software metrics for bug prediction and can contribute to ﬁnd the
best performing solution in each particular case.
Continuous integration is a development practice that has gained popularity in
the last decade. Research has shown that 40% of the OSS-projects has included
CI in their workﬂow [HTH+16] and, as explained in [CDA+19], the execution of
builds overtime constitutes a valuable source of data which is often overlooked
and unused within projects. Our literature review has identiﬁed one main exist-
ing dataset, dated 2017, and one main data source: Travis CI. We have identiﬁed
the most common metrics used and the methods to extract them (API and log
parsing). Continuous integration data has been used in research mainly to ex-
plore the reason for build failures, predict build outcome or investigate common
practices. To our knowledge, no existing work has tried to predict bugs from
the execution data of a CI pipeline.
We therefore concluded that the suggested approach may contribute to expand
the existing plethora of metrics used to perform bug prediction and exploit
execution data which is inevitably produced by software projects adopting con-
tinuous integration.

3.1
Git
A Version Control System (VCS) is a tool that allows to record and track the
changes to the ﬁles of a software over time. This allows to recover and compare
past versions of the code, and create new ones as the development continues. Git
was originally developed by Linus Torvalds as a tool to version the changes of
the Linux kernel code. According to OpenHub, in 2020 git was the most popular
Version Control System in the open source software world [Ope20], being used
by the 72% of the open-source repositories. Given we are going to work with
==PAGE:46==
34
Relevant theory
OSS, we focus only on this VCS and explore its functioning and mechanisms.
Git is a distributed VCS, which means that every client working on the reposi-
tory will clone the whole history locally: this way every client has the complete
mirror of the remote repository and can serve as the oﬃcial (elected as contain-
ing only reviewed code) repository. However, this is usually hosted on a server
or a cloud service such as the popular Github.
Commits and tags
Changes are recorded in the form of commits. A commit
is a set of changes applied to ﬁles and is uniquely identiﬁed by a commit hash
(or SHA, from the SHA-1 hashing algorithm used to produce the ID). The hash
is based on the (hash) parent of the commit, the author/committer, the date
of the commit, the commit message, some meta data and the current version
of the ﬁles (with the changes). Changing even one of these items would result
in a diﬀerent commit hash.
Commits are linked together via a parent-child
relationship: every commit has a parent, and in special cases even two (merge
commits), except for the initial (root) commit of the repository. Developers can
create tags to name commits in a human friendly way, and are often used to
identify commits which represent a release of the software. In other words, tags
create aliases for the commit hashes.
Branches and check out
Developers can check out a commit, which means
that the working directory is set to the state represented by that commit. One
important ﬁle in git is the HEAD. This ﬁle contains the reference currently
checked out, e.g.
a branch, or a commit (a case known as detached HEAD
state). Branches are human friendly pointers to commits. One branch points to
one commit. Unlike a tag, however, when checking out a branch and creating
a new commit, the branch is updated to point to the newly created commit.
The main diﬀerence with tags is that tags do not change what they point to,
while branches do. Branches are used to allow developers to create separate
development threads, e.g for bug ﬁxing or implementing new features.
The
main branch is usually called the master branch.
Merging branches
Branches are created to allow developers to work in iso-
lation from the main branch (the master branch or another branch). Once the
work is done, the changes need to be merged back into the original branch. The
original branch is called base branch, whereas the branched one is called head
branch. In git there are 3 ways to merge the work done on the head branch
back into the base branch:
==PAGE:47==
3.2 Correlation analysis
35
• Merge: the merge operation creates a brand new commit with 2 parents:
the tip of the head branch and the tip of the base branch.
The code
versions in the two commits are merged together and conﬂicts may arise
if the same ﬁles have been modiﬁed in both branches.
• Rebase: the commits created in the head branch, after the branching
from the base branch, are replayed on top of the current tip of the base
branch. Conﬂicts may arise and need to be solved. Once this is done,
the merge can be done in a fast-forward manner: the pointer of the base
branch is simply moved forward to the current tip of the head branch.
When rebasing, commits can be squashed into fewer commits: this is
usually done to keep the commit history clean.
• Cherry pick: in case only speciﬁc commits should be applied on another
branch, git allows to use the cherry-pick strategy.
This way commits
can be individually picked from a branch and replayed on top of another
branch.
Pull requests (PRs)
A popular mechanism to notify the other team mem-
bers that a task developed on a branch is ﬁnished, or requires feedback, is to
open a pull request. Pull requests are deﬁned by the base and head branch (and
sometimes also the repositories, in the context of external contributors). This
mechanism is natively supported by git, but ﬁnds a more interactive and con-
structive implementation in services such as Github1 and Bitbucket2, where it
is possible to discuss, comment and review the proposed changes. Pull requests
can then be approved by other members (they can be conﬁgured to require sev-
eral checks to be approved) and closed. An approved pull request can then be
merged (using one of the mechanism presented earlier) or not. Thanks to their
collaborative nature, pull requests are used to obtain feedback, discuss imple-
mentation details and are often the only way to merge branches in companies or
OSS-projects. If new commits are added to either the base or head branch, the
pull request is automatically updated to display the current comparison between
the tip of the branches.

3.3.3
Cross-validation
In machine learning it is of great importance estimating the performance of
models. When estimating how precise the model is, we usually calculate the
error the model introduces in the predictions compared to the real observations.
Ideally we would use the whole dataset to train the model, but if we also tested
the model with the same observations it was trained with, we would obtain
a very biased evaluation, and we would end up selecting the most overﬁtting
model. Overﬁtting happens when the model has learned very complex patterns
and has become extremely precise in predicting the observations it was trained
with. However this means that it has possibly also learned to model the noise
in the data, and would therefore poorly generalize to new observations it has
not seen during training.
Since we do not have access to other observations other than the ones in the
dataset, in order to evaluate objectively the model performance, it is crucial to
split the dataset into:
• Training set: the set of observations used to train the model
• Test set (or validation set): remaining observations used to evaluate the
model’s performance.
This way we can compute the test error by letting the model predict the values
for the observations in the test set and calculate the error function between the
expected value and the predicted one.
However, this test error depends on the observations contained in the test set.
A more accurate generalization of the true error of the model can be obtained
==PAGE:56==
44
Relevant theory
by using the K-Fold approach. The dataset is split into K equally sized sets of
observations. For each of the K sets, we use the other K −1 sets to train the
model and test it on the remaining set. The model error can then be computed
as the average of the errors in each split.
This is a better approximation of the generalization error which is the true error
of the model which we would be able to calculate only if we had access to an
inﬁnite test set.
Cross-validation can therefore be used to evaluate more objectively the perfor-
mance of a model and consequently compare diﬀerent models in order to select
the best one.
Hyperparameters selection
We have seen that machine learning models
may be conﬁgured to work in diﬀerent ways.
The parameters which cannot
be learned by the data during training but are instead provided to the model
at initialization are called hyperparameters. An example of this is the number
of neighbors that KNN should use when predicting, the distance measure it
should use, or the maximum depth of a decision tree. These hyperparameters
should be determined according to which ones work best for the speciﬁc prob-
lem we are addressing. Cross-validation can be used to objectively select the
best hyperparameters for a model, according to how it performs in the cross-
validation. Similarly to how we would select the best type of model among the
available ones (KNN, Decision Tree,..) we can use cross-validation to select the
best model initialized with diﬀerent hyperparameters. As an example we take
KNN and we simply create a list of possible models with diﬀerent numbers of

Build burst metrics
We have previously introduced the concept of code
bursts metrics, which were proposed by Nappagan in [NZZ+10], and consisted
in counting the sequential series of changes to a module/component/class in the
commit history. Two parameters were deﬁned: the minimum size of the burst
and the burst gap. The gap deﬁnes how many elements are needed to break
the burst: if gap size is 2, then the sequence True-True-False-True is considered
a burst, whereas True-False-False-True is not, as there is a gap of size 2. The
minimum size of the burst deﬁnes the smallest number of sequential changes
(including gaps) needed to be considered a burst. This concept was later applied
to test failures on integration gates at Microsoft in [Her14]. In this context test
failure burst metrics were introduced, and the idea of counting sequential
failures was applied to single tests (see section 2.1.3.6 for more details). We
propose an adaptation of these metrics to the continuous integration scenario,
where we can look at builds and their states to deﬁne bursts. Builds associated
with a release can be ordered by their starting time, but this does not guarantee
a relationship between consecutive builds. Indeed, they may build commits from
diﬀerent branches,and this does not suit the idea of sequential failures needed in
bursts. Instead we will look at builds sharing the same reference (pull request
or branch) and ordered by starting time. This way we can apply the concept
of bursts to represent consecutive builds within the same branch/pull request
(allowing gaps) in particular states. In particular, we will calculate bursts for
builds in the following states:
• Not passed builds: builds in any state other than passed
• Failed builds: builds in the failed state
• Canceled builds: builds in the canceled state
Depending on the states available in the CIS we may add more metrics. We will
use the same set of values as used in [Her14] for the parameters:
• Gap size: 1, 2, 3
• Minimum burst size: 1, 2, 3
For each of the combination above (build state, gap size, minimum burst size)
we store the actual size of the bursts found and the number of bursts.
For each of the build state deﬁned, we have 9 combination of gap size and mini-
mum burst size. We have deﬁned 3 build states which gives us 27 combinations.
Each combination we store the actual size of the bursts found and the number

they can be when combined together. Their predictive power can increase but
can also lead to either overﬁtting or a worsening in performance (curse of di-
mensionality) if not truly informative features are fed to the model. Given the
high number of features our dataset comprises of, we believe this step is fun-
damental. For this reason we implemented a feature selection algorithm called
forward selection.
Ideally we would evaluate all combinations of features and select the best per-
forming one using cross-validation.
This is impractical since the number of
combination quickly explodes and it is therefore not computationally feasible to
try out all combinations. A possible solution to this problem is forward feature
selection. Forward feature selection starts with an empty set of features s, and
for each feature fi in the pool of available ones sf, it selects the features in s+fi
from the input data X creating X′ and uses cross-validation to evaluate the per-
formance of the model with X′. The best performing feature is then added to s
and removed from sf. The process is then repeated adding each feature from sf
to s and training and evaluating the model on that subset of features. The best
performing one is added to s and removed from sf. The algorithm ends when
no features added provides better performance than the one from the previous
round or no features are left in sf. A variation of this algorithm reverses the
procedure and starts by including all features and removes one at a time. This
goes under the name of backward feature selection.
These feature selection techniques are not exhaustive as they do not test all the
combinations of features, but they represent a good compromise between the
computational complexity of trying diﬀerent combinations and the performance
of the models. We will apply this technique to all the models to identify the
best set of feature, and for the subsequent steps we will limit the dataset only
to that subset when training the model.
The evaluation procedure
We are interested in evaluating how well each
model can generalize the patterns in the data and predict new observations. As
explained in earlier sections cross-validation is a valid procedure for estimating
the generalization error and hyperparameters of the models.
We will structure our validation procedure as a two-layer cross-validation. The
inner layer will determine the best hyperparameters for each model and the
outer layer will estimate the generalization error of the model with those hyper-
parameters. The number of folds for the inner and outer layer will be deﬁned
once we know the size of the ﬁnal dataset in order to strike a good balance
between training and test sets.
==PAGE:71==
4.4 Analysis
59
The performance metrics for regression
The type of problem helps us
narrow down the large plethora of performance metrics which can be found
in machine learning. Being our task a regression problem, and therefore deal-
ing with continuous output variables, we know from literature that the most
commonly used metrics are error metrics calculated as diﬀerence between the
predicted value and the real value.
We select the Mean Absolute Error
(MAE) as our ﬁrst performance metric. The reason is that this provides an
easy-to-understand measurement of how much on average is the error in pre-
dictions from the real value. The baseline is the performance of simple models,
such as one that always predicts the mean or the median value of the data. We
will compare the performance of our models with the baselines.
The second metric we will use to evaluate our models is R-squared (R2). This
metric represent how much of the variance in the data is explained by the model.
The baseline is a model which always predicts the mean value and therefore does
not explain any variance leading to an R2 of 0.
The performance metrics for classiﬁcation
In case we deal with an im-
balanced scenario, we know from literature that it is important to use metrics
which can capture the performance in predicting all classes and therefore preci-
sion and recall would be our choice. In case the classes are balanced, we would
use accuracy as it is both intuitive and eﬀective when both classes are equally
represented.

From our methodology, we knew that some metrics needed to be extracted from

log ﬁle associated. As for the terminology, we will use the terms "build log" and

are associated with jobs and not builds. Travis CI provides access to the logs
of a job via the endpoint /job/{job.id}/log. Given the large number of jobs,
we were faced with some challenges. In this section we brieﬂy present them and
explain how we decided to tackle them.
Local vs remote
One option was to download all the build logs locally and
run the parsing script on the log ﬁles.
This approach had the advantage of
querying logs only once from TravisCI servers, and not having to hit them again
in case a new parsing was needed. Unfortunately this method was not trivial
to achieve: we had roughly 136k jobs to download logs for, and an initial batch
of 11k build logs resulted in 3.47Gb of storage occupied. A quick projection
showed that downloading all the job logs would have resulted in 42Gb. The
laptop available for computation had an SSD with 192Gb of which only 15Gb
were available. Compressing the 3.47Gb of logs led to a zip ﬁle of 370Mb in size.
Downloading logs and compressing them would therefore occupy 4.5Gb. Given
that 42Gb was not a viable option given our resources, we opted for downloading
build logs and compressing them in batches of 10k ﬁles.

Concurrent fetching of logs
TravisCI APIs do not support batch requests
for logs retrieval, therefore each job resulted in one HTTP request to TravisCI
servers: in our case we generated 136k requests. This type of task is said to be
I/O bound and network intensive: when all the requests are performed by one
process sequentially, network delays and latency keep the process waiting for
results for a speciﬁc request and prevent it from doing useful work in the mean-
time. This scenario is well suited for performing HTTP requests concurrently:
one main process creates multiple threads and each one of them is assigned
one HTTP request to execute. While some are blocked on I/O waiting for the
response from the server, the CPU can schedule other threads which can per-
form their request to the server and wait idle for the response. Only once the
response from the I/O is available, the sleeping threads will be awaken and will
be eligible to be schedule on the CPU and complete their task. The concurrent
scenario is more time and resources eﬃcient. We implemented and tested two
diﬀerent functions: one fetching logs sequentially and one concurrently (using
the python package concurrent.futures and ThreadPoolExecutor). We measured
their execution time on the same set of job ids to fetch, using the %timeit func-
tion of Jupyter notebook. The results conﬁrmed the performance increase, as
shwon in Figure 6.5, and we therefore opted for a concurrent logs retrieval.
Figure 6.5: Performance of concurrent and sequential logs fetching. The com-
parison shows that concurrent execution took 1/9th of the time it
took for the sequential execution
Failure handling
We designed the script to place the logs ﬁles in a speciﬁc
folder and log all the ids already retrieved in a ﬁle. Once the number of ﬁles
in the folder reached a certain number (set to 10k) the script would zip the
folder and create a new one where to place the new ﬁles. Logging the ids for the
jobs retrieved made the script fault tolerant in case of unexpected interruptions:
when restarted the script would read the ids already retrieved and remove them
from the list of jobs to fetch. Fetching logs for 20 jobs took on average about 4s,
which means that fetching 136k would take 8 hours. In time consuming tasks as
this one, it is important to implement such basic mechanisms to avoid starting
from scratch every time.

Regression
We have estimated monotonic correlations between the single CI metrics and
the target variable. These results can indicate relationships between variables
but do not quantify how well a combination of these variables can predict the
number of post-release bugs. In this section we will evaluate how well machine
learning models trained on the collected metrics can predict the target variable.
Feature selection
As explained in the methodology chapter, we ﬁrst apply
forward feature selection to all the models: decision tree, KNN and Linear
Regression. We used a 5 fold cross-validation for evaluating the performance
of the model for each feature combination and we used r-squared as evaluation
metric. We considered a set of feature superior to another one if the minimum of
the 5 r-squared results from the cross-validation was greater than the minimum
of the other set.
Hyperparameter tuning
We have previously introduced the concept of hy-
perparameters, which the model itself cannot learn from data, but instead are
provided to it. The tuning of these parameters needs to happen outside of the
training phase. The tuning can be based on domain knowledge or by testing dif-
ferent combinations. In our case we do not have knowledge which can guide our
choice, and we are left with the only option of performing cross-validation of dif-
ferent parameters combinations. Among the models we are going to use, KNN
and Decision Trees need hyperparameter tuning. The ordinary least squares
linear regression does not require any hyperparameters.

8.1.3
Multiclass classiﬁcation
In order to evaluate the predictive power of our features, and in the attempt of
overcoming the limitations imposed by the small size of the dataset, we simplify
the problem by formulating it as a classiﬁcation problem. The values of the
target variable are therefore divided into ranges, and each of these ranges will
be considered a class. This way our target variable becomes a discrete variable,
and this is what classiﬁcation models try to predict.
The choice of the range values can either come from domain knowledge, if speciﬁc
ranges are of particular interest, or deﬁned empirically from the data. In our case
one could be interested in predicting if a release will present bugs or not, deﬁning
the problem as a binary problem with separation value 0: releases with 0 bugs
as a class and those with more than 0 as the other. However, we do not see this
scenario suitable for our case: we know that only 2 releases did not present any
bugs which would lead to an extremely imbalanced dataset with 50 observations
belonging to one class and only 2 to the other. This imbalance would lead the
minority class not to be present in many cross-validation folds, and models
would have hard times deﬁning what characterises that class, leading to models
extremely good at predicting the majority class. There are ways to overcome
imbalanced datasets (over-sampling or down-sampling) but they would not help
in this scenario where too few observations belong to the minority class. Given
these reasons, we decided to aim for a balanced dataset. We used quantiles
values from the target variable to deﬁne range values and assign observations
to their classes.
We tried dividing the dataset into 4 classes but did not obtain performance
stably above the baselines (similarly to the regression case, again possibly due
to the small number of observations in each class). We therefore reduced the
number of classes to 3.
We report the division of observations according to
the quantiles in Table 8.3, where we can see that the dataset is balanced with
regards to the number of observations in each class.

9.1
Dataset construction
As we could not ﬁnd any existing dataset which met our requirements, we
set out to construct a new dataset from scratch. We started by deﬁning the
methodology to follow, initially deﬁning the criteria which the case study project
had to satisfy to allow us to research our task. In machine learning and data
analysis, the dataset construction is an important step. The quality of the data is
reﬂected in the predictive performance of the machine learning models, and the
relevance of the results obtained from the analysis. It is therefore crucial to only
collect and include as complete and relevant data in the dataset as possible. For
this reason, at each step of the data collection and dataset construction step, we
performed data analysis to ensure that completeness and quality were satisﬁed.
Successful methodology
We were able to successfully implement our method-
ology plan. From the data sources deﬁned initially, we were able to extract all
the necessary information to combine, and aggregate the metrics into our ﬁnal
datasets. Indeed, our selection process for the project to use as case study, re-
sulted in valid data sources from which it has been possible to retrieve data via
API or by cloning the repository locally. This proved our methodology to be
eﬀective in allowing to create the dataset needed for analysis of CI metrics and
prediction of post-release bugs.
Missing values
Among the limitations we encountered and discovered by
ensuring completeness and quality at each step, we found some missing values
in the CI data collected.
Canceled builds were the main reason for missing
values. The date at which builds were started, build duration, and log ﬁles were
the most notably aﬀected data. We overcame these limitations by excluding the
aﬀected builds by our analysis when the missing values were to be used. We
believe that this has only marginally aﬀected the metrics collected: indeed we
know from analysis and logs parsing, that canceled build represent the 25% of
all builds, and only 25% of these were aﬀected by missing values. Moreover,
30% of canceled builds were intentionally canceled due to identical builds being
triggered by open pull requests. It is diﬃcult to quantify the eﬀect that this
exclusion has had on our results. We can however notice that the most correlated
and informative features are mostly based on failed and passed builds which are
only marginally aﬀected by missing values. It is not clear if metrics based on
canceled builds could have had a more central role, if they had been complete.
It is to be noticed that they intrinsically represent mistaken builds, and most of
the metrics we deﬁned, based solely on them, were only measuring their number
or percentage.

Issues
During our analysis of the issues retrieved from Jira, we noticed that
not all issues speciﬁed what releases they aﬀected. More precisely, only 55% of all
the bugs reported had the aﬀected version of the software stated. Our analysis
is therefore biased towards these types of post-release bugs, whether they are a
type of bug which is easily backtraced to a speciﬁc version of the code, or being
reported by diligent developers or users which speciﬁed it. This is an aspect to
take into consideration when discussing the validity of our results. Nonetheless
we could not identify any systematic pattern which explains what type of issues
were reporting the aﬀected version. It is unlikely that this phenomenon is purely
random, and therefore our results do apply to the type of bugs with the version
reported, even though we do not know what characterizes them.
Releases
Another limitations we had to face, was the number of releases which
were eligible to be included in our datasets. The ﬁrst factor reducing the number
of suitable releases was the date the project started using Travis CI. We could
not include any of the releases developed prior to the migration to the publicly
accessible CI server, and even a couple of those released immediately after, given
that most of their development occurred on the previously used CI platform.
Another factor was the type of releases. Part of the ones tagged on Github
represented preliminary releases, which we could not ﬁnd any issues for on the
ITS. Having assessed that the lack of issues was not due to releases being faults-
free, but more likely to be a convention not using preliminary releases as labels
for issues, we had to exclude these as well. Although given the structure of our
dataset this reduction directly aﬀects the number of observations, and having
many observations is desired, we could not ignore these aspects and, aiming for
high quality data, we could do nothing but only retain the eligible releases.
Small size datasets
When it comes to learning patterns from the data, the
limited set of observations in the ﬁnal datasets poses several challenges. The
inevitable presence of noise in the data is emphasized by the small number
of observations, and this challenges machine learning models in capturing the
underlying patterns and correctly predict new data. Several approaches have
been implemented to overcome this problem. The ﬁrst has been limiting the
complexity of the models to avoid overﬁtting. Indeed, one problem with small
datasets is the fact that models can learn to ﬁt well the training data, learning
to model the noise as well, and poorly predict unseen data. For example, in case
of the decision trees, we chose the hyperparameters which limited the depth of
the trees. Moreover, our datasets present high dimensionality and we therefore
performed feature selection, which has reduced their dimensionality of a factor of
10 and boosted the performance of our learners. Indeed, including uninformative
features can lead to worse performance or overﬁtting. Finally when evaluating

the performance of the models, we used again cross-validation and reported
the results for each fold. This is crucial, especially on a small dataset, where
one single split into training and testing sets may not be representative of the

consistently learn the underlying patterns. Using cross-validation however, we
could evaluate the stability of the performance: indeed we observed unstable

9.2.2
Informative features
In this section we are going to merge the ﬁndings regarding what features, among
the ones deﬁned and extracted, was identiﬁed as informative with the goal of
predicting the behavior of the number post-release bugs across the diﬀerent
experiments.
From previous section we have concluded that the CI metrics were more informa-
tive when limited to the restricted datasets. Same goes for feature correlations,
where we observed stronger correlations in maven and gradle datasets.
The
highly correlated features in the complete dataset are also present in the ones
from maven. This is possibly due to maven representing the 61% of observations
of the complete dataset. We therefore believe to be more relevant to focus only
on the ones found in the restricted datasets.
Maven
Burst metrics for failed builds showed the strongest correlation along
with the maximum and mean number of failed tests. The maximum number of
builds per pull request also presented a high correlation. A general tendency is
that the aggregation preferred is the maximum and is associated with positive
correlations, while the minimum is associated with negative ones. There are
some exceptions, e.g. maximum duration of failed builds showing a mild negative
correlation, but the trend is for the extremes (max and min) to be the most
informative.
These categories of features are also the ones selected by the best performing
models. The average number of burst is the attribute with highest coeﬃcient in
the linear regression model, along with the number of error classes thrown and
the builds per pull requests. For the binary classiﬁcation, again a burst metric,
burst size, was used for eﬀective classiﬁcation in dataset1 and the number of
failed tests and builds per pull request for dataset2.
From these results we conclude that the number of failed tests, bursts metrics
and the number of builds per pull request are the most informative ones for the
maven dataset.

The question is then whether these metrics are a proxy for other well-known
metrics in literature for bug prediction. We know that process metrics have
been investigated and used metric for bug prediction. Development processes
do directly reﬂect on CI metrics, and it is therefore important to investigate the
degree to which the two measure the same aspects. We select the basic process
metrics based on changes: the number of commits and number of line of code
changed in all the commits associated with a release. For the former, we simply
count the number of commits per release, while the latter requires to query the
local repository to retrieve the list of changes for each commits. We calculated
the correlation between the most informative metrics and the process metrics.
Indeed, the maximum number of burst for failed builds was highly correlated
with both process metrics (0.91 and 0.86) but the rest of the features presented
weak to mild correlations indicating that they do not just work as a proxy for
basic process metrics.
Gradle
A diﬀerent set of correlated features was observed for the gradle
datasets.
Builds’ duration scored high in correlation coeﬃcients mostly pre-
senting a negative sign. The only exception was the duration for failed builds
which resulted in a positive mild correlation. Another interesting observation
was that burst metrics were less predominant. Among the logs metrics we found
the number of skipped and failures occurrences to score highest, right after du-
ration metrics.
This was indeed reﬂected in the features selected for linear
regression, including these along with the number of passed tests. In binary
classiﬁcation only the number of skipped words was selected and allowed the
model to outperform the baselines.
Similarly to what we did with maven features, we calculated the correlation
between the process metrics deﬁned earlier and the selected features for the
best performing models. The strongest correlation was between the maximum
duration for failed builds and the maximum number of deleted lines (0.65). All
the others were below that value.

In this section we present and discuss some of the limitations aﬀecting our work.
Project speciﬁcity
The data we have collected and used in our analysis
comes from one single code project: sonarqube.
Our results have therefore
validity only within that scope. This does not mean that the ﬁndings from our
==PAGE:151==

work cannot be applied to other code projects, but it is not guaranteed they will
lead to the same results, and that the metrics we found to be most informative,
will be in other contexts too. Our experiments on two CI periods indeed show
that metrics are very much tool and conﬁguration sensitive, and even within
the same project the migration to another build tool has impacted the most
useful metrics. The reason for selecting only one project was to allow us to
explore and extract insights on the CI processes and practices. This has been
key in polishing and preprocessing data to only include high quality observations
and metrics in our ﬁnal datasets. For instance, we removed issues not related
to the community edition of the project, and accurately selected the eligible
releases.
This process could not have been performed on multiple projects,
given our limited time window available. In this way we tried to trade quantity
and generality with quality. However, we believe to have deﬁned a methodology
which is general enough to be adapted easily to other projects.
Limited usability
In terms of practical usability of the predictions, we worked
at release level which is the least precise level of prediction among the ones within
the bug prediction ﬁeld. Although these predictions do not provide a precise
indication of what component will likely present faults, they can still be used as
an additional factor to take into consideration when deciding whether to release
the software. We realize that a more precise prediction is desired, but we had
to face the fact that rarely the available data on faults is labeled at a compo-
nent level. Previous work has used existing datasets which have tracked faults
back to various software component levels. Unfortunately, we had to cope with
the requirement of working with projects using CI and for which CI data was
accessible. This has restricted our range of possibilities, as presented in the sec-
tion discussing the project selection, and we could only perform predictions at
release level. This is however the ﬁrst step in working with CI data and future
research could try to predict bugs at component levels if the necessary data is
available.

Chapter 10
Conclusions
This research aimed to evaluate whether Continuous Integration execution data
could represent a valid source of information to help predict the quality of a soft-
ware release. Based on the results collected from the analysis of the constructed
datasets, it can be conclude that this type of metrics does provide informative
contributions with regards to the prediction of the number of post-release bugs
in software releases. Although we observed unstable or biased predictions in the
regression tasks, in binary classiﬁcation models performed acceptably, reaching
up to a median of 87% accuracy.
Our approach provides several beneﬁts. Firstly, it allows to exploit often unused
data produced and stored on the CI server. Secondly, it adds a new possible
type of data to be used in the ﬁeld of bug prediction, which is a complex task
and can beneﬁt from a large plethora of metrics to allow to ﬁnd the best suited
combination for each project. Lastly, using these CI metrics allows to identify
common CI related traits of releases with similar number of bugs, and therefore
provide guidance in what practices to improve.
The experiments carried out suggest that the proposed metrics are more in-
formative when collected from a relative homogeneous (conﬁguration and build
tools wise) Continuous Integration period. This result can be of great value to
those interested in utilizing CI metrics for release quality prediction, as they
can limit data collection and extraction only to releases developed using similar
==PAGE:154==
142
Conclusions
Continuous Integration settings. On the other hand, this poses a limitation to
the quantity of valid data available: given that the CI process may evolve over
time, it may be necessary to release a certain amount of versions before having
enough training data.
Future work
Due to the limited time frame available for the development of
this project, possible further investigations or experiments have not been carried
out. Future work could focus on applying the proposed methodology to other
open or closed-source projects and test the validity of our results outside the
scope of the project used as case study. Another interesting experiment would
be to combine the proposed metrics with other well-known software metrics
used in bug prediction, and evaluate the contribution and possible increase
in performance provided by the inclusion of the CI metrics. Furthermore, as
previously mentioned, if the necessary data is available, CI metrics could be
tested in the prediction of bugs at a more ﬁne-grained level.

A.3.2
Cherry pick
An example of cherry picking, is the commit ed1a72601 which is a commit
diﬀerence between 8.4.2 and 8.4.1. This commit was cherry picked by master
(1644704e6) and was built from master but not with the sha ed1a72601. Cherry
picked commit can be seen using the git cherry branch1 branch2 command.
This will show what commits do not need to be kept in branch1 when rebasing
on top of branch2 7. Unfortunately this won’t provide the original commit that
it was cherry picked from . Cherry picked commit do not maintain the same
commit sha. The best way to identify the original commit is by comparing the
commit message. We restrict the search to only commit messages starting with
SONAR- as other commit messages may be too generic. First, for each commit
in the comparison dataset we retrieved the ﬁles aﬀected by changes with the
following command:
git log --pretty=oneline --name-status -n 1 {sha}
For each release we proceeded into detecting the original cherry picked commits.
For each commit we fetched all the builds which were not already associated
with the release and with the same commit message and whose commit was
created before the current commit. We then compared the list of ﬁles aﬀected
by the commits. If the match is positive, the two commits are deemed to be
cherry picked.
We identiﬁed procedures which were using the same commit messages and af-
fecting the same ﬁles without being cherry picked. We therefore removed from
the commit messages the following:
• maven-release-plugin prepare for next development iteration
• Prepare for next development iteration

A.3.5
Maven and gradle keywords
Maven has been the most commonly used tool for building and running tests
(unit and integration) throughout the sonarqube CI history up until its migra-
tion to gradle. Unfortunately it turns out that some trivial searches such as for
the words "maven" or "gradle" in the logs is not eﬀective: in several builds we
could ﬁnd logs for the installation of both of these tools along with their wel-
come banner even if only one of the two was actively used (our guess is that they
may come installed by default in the java environment provided by TravisCI).
We therefore opted for searching for the string "Reactor summary" or "Reactor
build order" as this are only present in maven builds: maven reactor plugin is
responsible for deciding the dependency order of multi modules projects (such
as sonarqube) and its banner is present in all maven builds. If this string is not
present but "gradle" string is, the job is labeled as gradle build.
