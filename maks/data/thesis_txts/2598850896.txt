Abstract
The drone technology is evolving rapidly and makes great advancement. Drones provide new pos-
sibilities for automation, due to their fast moving and ﬂying capability. The future of drones can
help improve the working conditions and safety for many people if used the right way.
This project combines the ﬁeld of autonomous robots to create an autonomous drone that is able to
do a search for an object. This object is called the mission object. A localization system based on
markers has been made. This system provides the drone with knowledge of its position in the ﬁeld,
which is necessary for the drone to navigate through the search path. A ﬁlter has been created to
search for the mission object located on the ground. This ﬁlter compares the object in the image
with a characterization of the mission object. A vision based controller has been made, in order
to load the mission object on to the drone. A way to plan a path to a given drop oﬀpoint has
also been made. This drop oﬀpoint is the location of where the mission object will have to be
transported to.
There has been made a demonstration video to showcase the drone.

1.2
Mission of this Thesis
The mission of this project it to make a drone do several tasks totally autonomous. The drone will
have to ﬁnd a mission object, which is a given object the user would like to ﬁnd. This could be a
speciﬁc item in a warehouse, or a speciﬁc location for a delivery.
The tasks making up the mission will be including: searching and ﬁnding the mission object in a
given ﬁeld, use control action to get the drone navigate to the mission object safely and get a hold
of it, then ﬁnally to deliver this mission object to a given drop-oﬀpoint.
First step in the overall operation is for the drone to follow a search path in order to do a systematic
search of the entire ﬁeld. Next step will be for the drone to position itself directly above the mission
object when it has been found and land on it. The drone will now have to load the mission object
on board and take oﬀagain. Finally it will go to a drop oﬀpoint where the mission object should
be placed.
1.3
Thesis Contribution and Outline
The goal for this thesis is to combine the diﬀerent ﬁelds of autonomous robots, in order to create a
well rounded project that showcase the possibilities of drones. A lot of subjects will be taking part
of the development, the most relevant contributions for this thesis are therefore listed below.
- Development of a localization system that is able to give the drone a clear understanding of its
position in the ﬁeld.
- Creating a map and a navigation system that enables the drone to navigate through the search
path.
- Making a computer vision ﬁlter that can decide based on information given about the mission
object, rather or not this object has been found.
- Designing a controller that can make the drone able to land on the mission object.
- A way to make the drone get a hold of the object and transport it to the drop oﬀpoint.
The next chapter, Chapter 2 will present the preparation for this project. Chapter 3 will be going
through the making of the localization and navigation system and the controllers designed for this.
Chapter 4 presents the work done for the computer vision making up the ﬁlter ﬁnding the mission
object. Chapter 5 describes the challenges of loading the mission object and elaborates on the
performance of the drone. Chapter 6 will be going into a reﬂection of the work done and what
future work can be implemented. Lastly will Chapter 7 contribute a conclusion.

PC or EC means if the sensor is a proprioceptive or a exteroceptive sensor respectively. A propri-
oceptive sensor only measures the inside of the drone. This is why a gyroscope is a proprioceptive
sensor, since it measures the drone’s angles.
A exteroceptive does measure the environment of the drone, this could be temperature or the image
from a camera.
A or P tells if the sensor is an active or passive sensor. An active sensor interact with the environment
by sending energy into it, like an ultrasonic sensor does. While a passive sensor do ambient energy
in order to measure.

2.4
Quadrotor Dynamics
A good understanding of a drone’s dynamics and movement will lead to a better chance of succeed-
ing in controlling it. A drone is a very complex system with many inputs and outputs, but many
drones do come with some build-in controllers to maintain a stationary position while ﬂying. This
project will be modeling controllers as a SISO system, but a good knowledge of the entirety of the
drone will be very beneﬁcial.
A quadcopter drone can move in many ways, but the four most basic movements is all SISO. Some
of the motors gets an identical input speed, while others stay at normal speed. This result in a
change of a angle or the altitude. These four movements consist of moving the drone up and down,
left and right, forward and backward and ﬁnally spinning clock- and counter clockwise. Figure 2.3
shows how these movements are achieved.
Figure 2.3: Overview of the four basic movements for a quadcopter.
The reason for these four control actions being SISO, is because there is only one input, which is
the motor speed. Some motors will move faster, this is the input, and some will move at normal
speed. There is also only one output, this output is either an angle, pitch-, roll- and yaw control,
or the output can be the altitude for throttle control.

Chapter 3
Localization and Navigation
The drone will have to ask itself a very important question in order to complete the mission. This
question being ”Where am I?”. To answer this question some sort of localization must be made for
the drone to navigate through the ﬁeld successfully. The implementation of a localization system
is crucial and needs to be made in order for several tasks to work. As an example, a very precise
knowledge of the exact position of the drone will be needed for, when the drone will do the system-
atic search for the mission object.
The drone will have to know its precise (or close to) position relative to the map of the ﬁeld, at
almost all time. This is a necessity in order for the drone to navigate save and precise. A little
oﬀset can result in the drone crashing in to a wall or an object, but it can also cause the drone to
miss a spot on the ground in the search for the mission-object. This can then of course result in
the drone failing the mission of ﬁnding the mission-object and by that not being able to deliver the
mission-object to the given drop-oﬀpoint.
The risk of misplacing the mission-object after it has been found and picked up, also increases as
the quality of a localization system decrease.
But what type of localization system can be made and how will it work? This chapter tries to
answer this and go through the steps of making the localization and navigation system.
This will include the idea behind making a map out of checkpoints (nodes) by using markers to
guide the drone, what type of controller will be used to control the drone’s position, the design of
this controller and the diﬀerent methods to implement this controller.
3.1
Localization with markers
Localization is often done with a combination of odometry, lasers and/or cameras, but for this
project the localization is done with cameras only. Odometry is often not even a possibility when
talking about drones, this is because it is hard to make precise internal measurements of the drones
movements, due to a high level of disturbance, especially with the Parrot AR 2, which is not the
most stable drone one can buy for money.
But even if there was some way to measure the rpm of each copter with an extreme high precision,
it would still not be possible simply because of uncertainty in the environment. The reason behind
odometry for mobile vehicle failing, is often because of a slippery ﬂoor that makes the wheels slide,
but when talking about drones many more things can go wrong. An unexpected wind blowing to
the drone, will make any odometry on the copters useless.
8

The Parrot AR 2 does not have any lasers on board and implementing and installing lasers on to it
will be way too complicated due to calibrating the drone in respect to the added weight and change
in the center of mass, let alone the amount of workload that would go into it.
The best sensor for localization is therefore the two already mounted cameras. The localization will
only be done with cameras as a result of this.
A smart way of utilizing the cameras is to look after markers, since markers tell a lot about the
given location. The idea is to put markers on the walls around the ﬁeld and by this making the
map as a topological map, with checkpoints (notes) relative to the markers.
The horizontal camera will be doing the localization by looking for markers, since the vertical cam-
era will be looking for the mission-object, but also since it is a lot harder to always keep a clear
ﬂoor, than keeping a clear wall. Therefore the markers will be placed on the walls, for the drone
to search for. The company behind the Parrot AR 2 drone, has made a maker that can be seen in
Figure 3.1. This is the same marker that has been used for this project.
Figure 3.1: Image of the marker used for navigation. The name of the marker is OBWRoundel
(Oriented black and white Roundel)
A Python script from the API ”PS-drone” has a function that will ﬁnd the marker and store its
pixel values for the X and Y position. It does also estimate the distance from the drone to the
marker and is able to get the orientation of the marker, but this last feature has not been used.
This function for ﬁnding markers is very useful since the information about the placement of the
marker is able to tell the position of the drone relative to that marker, which resolve in the drone
knowing its position in the ﬁeld, as long as the drone know each of the markers positions inside of
this ﬁeld.
The idea is to make a systematic search of the mission object, by moving through the ﬁeld with the
help of markers. Figure 3.2 below shows how this can be done:

Figure 3.2: Example of how a ﬁeld could look like. The map is made up by checkpoint connected
to each other. The drone will move from checkpoint to checkpoint in search of the mission object.
Every checkpoint is unique and has a speciﬁc distance to a speciﬁc marker. By every check point,
the drone will calibrate its position by the help of the marker in front of it. The calibration is done
in the x-axis (horizontal) and in the z-axis (depth). The drone will proceed to take a picture on the
ground when a stable correct position relative to the marker is obtained. Object recognition is then
used to check the ground for any objects that match the mission-object features. The drone will
then initialize next task if the mission object is found, but if not, then the drone will move forward
to next checkpoint and do the same procedure.
If the checkpoint is the last one in the marker lane, it will be called a turn-point, the drone will do a
180 degree turn and face a new marker on the opposite wall. A function diagram for this procedure
can be seen in Figure 3.3

3.1.1
Marker Detection - Contrast Issue
The idea of using markers to navigate is great and very eﬃcient, but it does come with some prob-
lems. The contrast issue can occur when setting up the markers if one not being careful. This is
due to how lighting aﬀect the quality of the marker detection. The detection will often fail if the
frame with the marker has a large contrast between the darkest pixels and the lightest pixels.

3.3.1
Finding The System Time Constant
τθ is a time constant for the system and is representing the time it takes the drone from it getting
an input and by that, starting to turn to the speciﬁc input-angle, to the point which 63 percentage
of that angle has been achieved.
τθ is by that basically a representation of how fast the system is able to respond to an input. This
will of course mean a short time constant is ideal, since a system with fast respond can stabilize
faster and with less overshoot.
A short time constant is especially great when working with drones, due to drones being very fast
and maneuverable. A too large time constant could mean an increased risk of the drone making a too
big overshot and by that getting the marker out of frame, which of course is unacceptable, because
of how the localization works. The drone will get lost and lose its way, if the marker gets out of sight.
On the other hand, having a short time constant comes with a condition- the sample time of the
system always needs to be faster. It is great that the system has a short time constant, but it can
only work if the system samples with an even shorter sample time. This is due to the need of the
system being able to react on the fast input response.
Finding the time constant τθ is essential, to make a good model of the drone system, which is once
again essential for making the right controller and by that stabilizing the drone while ﬁnding its
path.
A way to determine the time constant, is to log the drone’s data while ﬂying and having a look at
how it responds to an input. Multiple test have been made where the drone was asked to get a
speciﬁc angle, at a speciﬁc time.
Data of the drone’s pitch angle changing through time, can then help determine this time constant.
The drone is able to deliver data 200 times per second which presumably is enough.
The test data in Figure 3.6 shows how the pitch angle change through time. The x axis shows
the number of samples and the y axis shows the roll angle in degrees. At sample 1000, the drone

The design of the controller is one thing, but the way of implementing it into the code running the
system is another. For example if the system in question is a simple system that needs a controller
to be implemented fast and quick, the Ziegler-Nichols method is a good way of doing it. While a
very complex system with many inputs and outputs can be made with a full state space feedback.
A lot of systems can also be made by a combination of modeling, measuring and hand tuning. Let’s
take a PD or PI controller for example. KP could be calculated from a system model and then KD
or KI could be hand tuned by trial and error.
Simulating the step response of a system with a controller will often result in a very little steady
state error, like the step plot for this PD controller as seen in Figure 3.17. But the steady state
error will often be a lot diﬀerent, when the system is tested in the real world.
Many systems will never get a stable steady state error due to noise. This noise could be from the
environment, the wind as an example, or it could be noise from the internal of the system, this could
be measurement noise resulting in the wrong pulse for the pulse wide modulation of the copters.
Even when the Parrot AR 2 is asked to not move at all, it still hover around a bit and this makes it
impossible for the controller to position the drone with a very little steady state error, when trying
to clear a checkpoint. It would simply not be able to do it.
Stable zones has therefore been established, in order to get the drone to stabilize, while still being
able to clear the checkpoint. A stable zone is a reasonable interval of an acceptable error on the
system. This could be ±10% of the reference value. As an example this will mean that the drone
will be inside of the stable zone as long as the marker is in between 288 and 352 pixels in the x axis
(left/right), since the reference is 320 pixels.
The stable zones are of course not based on pixels, but a length in meters.

Chapter 4
Computer Vision
This next chapter will be about the construction of the necessary ﬁlter features and algorithms
needed for recognition and determination of objects. The chapter will be looking at the ﬁlter re-
quirements and how the ﬁlter can be composed, center of mass calculations for the mission object
that the drone will have to carry, ways to optimize the process time and a evaluation of the ﬁlter
performance.
The purpose of this chapter is to make a vision based method that can determine whether or not
the mission object has been located. This will be done by a sequence of feature evaluations. The
drone do in fact have a characterization of the mission object, even before take oﬀ. This include a
description of the color and size of the object. The chapter will also be looking at some potential
additional features like shape and multiple objects, that could be implemented.
The drone will study the ground picture at every checkpoint, by classify objects based on these
features. Finding an object that fulﬁll the requirements and by that matches the characterization
of the mission object, will initialise the ”Loading Control” which will be discussed in the ﬁfth
chapter.
4.1
Filter Requirements
Increasing quality of the classiﬁcation is a byproduct of an increment of the detail level of the
characterization. A deﬁned set of features will boost the chance of success for the mission object
search, which amplify how important these ﬁlters are for the mission and its likeliness to succeed.
This project will be focusing on two very important user input features, which is color and size, but
others could be explored, such as height-wide ratio, temperature, absorption and emission levels
and many more.
4.1.1
Color
The ﬁrst ﬁlter feature to be made is a color threshold. The reason for this feature to be made as the
ﬁrst on, is due to how the ﬁlter is build up. The ground image will be segmented by a threshold,
in order to extract all pixels without the desired color matching the mission object. This threshold
method is also the ﬁrst step in BLOB extraction, which is the foundation of possessing multiple
objects in one image.

This will result in a binary image, containing only pixels ﬁtting the desired color. This way of
ﬁltering the image can however be very problematic, due to several occurring issues.
The ﬁrst issues being loss of information. A given pixel contain information of how red, green and
blue it appears to be and by making the image into a binary image, this information gets lost.
Let’s say an object is red, but one part of the object is in direct sunlight, while the other is covered
by a shadow. This will make one part of the object appear more light red, and the other part
appearing dark red. The knowledge of which part and how much of the object that is in direct
sunlight would get lost, if the algorithm in Equation 4.1 would be used on this particular image
containing the object.
One way to solve this is by letting the pixel values be, by not changing the RGB-values of them,
and only using them for calculating the center of mass. This will also lower the process time due
to the elimination of overwriting the image.
This can only be eﬀective if there is only one object in the frame is assumed, since this will not
make the object pixels stand out from the rest.
Another problem is the consistency of the scene. When talking about computer vision, lighting
is alpha omega. One day the sun is shining bright, but the next day is a cloudy day with low
luminosity. Shadows and the environment can change the pixel value in a scene meter by meter.
Rmin and Rmax is supposed to deﬁne the interval for the red contribution of the object, but this
interval can be useless if the object is lighted up with the wrong brightness level. The exact same
object, in the exact same position, with the exact same unnatural light setting, could appear very
diﬀerent depending on the time of the day the images has being taken, if any sunlight has an eﬀect
on the object.
One way to ﬁx this problem is by not evaluating a pixel by its exact values, but rather by its values
relative to each other. This way will limit the drone in specifying the shade of a given color, but will
in return make the drone able to ﬁnd the color regardless of the lighting level, with the exception
of extreme cases with supersaturation.
The idea is to compare the color contributions of the pixel(RGB). The object color could be red
as an example. Red appears if the red contribution of the pixel has a higher contribution than the
green and blue, while green and blue is close to each other in value. Equation 4.2 is showcasing the
code thresholding the color of the image by this method.

BLOB Detection
A BLOB is an object in a digital image consisting of similar pixels connected by 4-connectivity or
8-connectivity. BLOB detection will often be done on a binary image, since all white pixels (value
256) will be considered object pixels, but evaluating a group of pixels by an acceptable interval of
the pixel’s values, can also be a way of detecting BLOBs without modifying the original image.
This could be by considering all pixels with a value between 100-170 to be objects if done with gray
scaling or all pixels considered red (or any other color of choosing) if done with a RGB image.
A third way to do BLOB detection is to ﬁrst create an image like the one in Figure 4.2 left, and
then accept all pixels that are not black.
An object detected by 4-connectivity require all pixels to be connected directly to the left of right
or connected by the pixel directly above or below.
Pixels from an object detected by 8-connectivity can on the other hand be connected by any given

4.3
Optimizing - Shortening Sample Time
The ﬁlter has of now accomplished its purpose of identifying objects and trait match every time the
drone is facing a new object, in order to ﬁnd out if that one, can classify as the mission object.
The drone do also calculate the position of the object, for the loading control to use.
The only problem now is the process time. The time it takes at this point in development, for
processing one single image, is an average time of 6.283 seconds. This is for good reasons totally
unacceptably and not suited to work with. The controller would never be able to stabilize a fast
system like a drone, if it was only updated every 6.3 second. The only system able to work with
such a slow sampling time, would be temperature control of a room. The process time will therefore
has to be shortened - by a lot.
A number of smart solutions can be used to minimize the sampling time. The ﬁrst one involves
only doing the absolute most necessary, when processing the image. It is not necessary to overwrite
background pixels black, like in Figure 4.2, since it can be assumed the there will only be one object
in the same frame. The drone will therefore not be required to use time on modifying the image,
but instead only calculate center of mass when an object pixel has been found and do nothing the
rest of the time.
46
==PAGE:55==
CHAPTER 4. COMPUTER VISION
Some method has also been developed to make the process go faster. The name of them is ”skip
row” and ”skip pixel”. If the drone has looked throughout an entire row, without ﬁnding any object
pixels, it will be asked to skip the next 14 rows.
If the pixel evaluated is not an object pixel, it will skip the next 25 pixels and start evaluating pixel
number 26. If however an object pixel has been found, it is assumed that the next 3 pixels to the
left in the same row, is in fact also object pixels.
Keep in mind that these numbers are based on the size of the objects used and the altitude of the
drone while ﬂying missions. If the objects was smaller and/or the drone was operating in a higher
altitude, these skip values would also have to be smaller.
Another great advantage of assuming only one object, is the ability to stop the image processing
when the entirety of one object has been found. This can sometimes result in only about 15 per-
centage of the image being processed, if the object is located in the top of the frame.
When the ﬁrst object pixel has been found, a false variable will change to true. This indicate that
the object has now been found and is being processed. It can then be assumed that whenever a
full row without any object pixel gets found, the entity of the object has been processed, and the
image processing is stopped.
After the implementation of all these time saving additions to the ﬁlter, the average process time
gets shortened down to only 0.153 seconds. This is most deﬁnitely a realistic sampling time for the
loading control.

This chapter will be focusing on the control action required for the drone to get a hold of the mission
object. The drone needs to center itself above the mission object in order to land on it.
The possibility for upgrading the already existing PD-position-controller, to a PID-controller will
also be discussed in this chapter. This will be followed up by the new facing compensation problem,
coming as a result of controlling position with the ground camera and a way to deal with this
problem.
The design opportunities for a grabbing device that can get a hold of the mission object are many.
The drone could lift it with a magnet, a suction cups, some sort of robot hand, and many more
alternatives. Many of these solutions have a common setback, that being the obligation of installing
new hardware on board of the drone. The ﬁrst problem coming from this being the workload and
the recalibration of the drone, due to both a new weight and center of mass. An even bigger prob-
lem is the fact that the Parrot AR 2 can simply not lift that much weight, 100 or 150 grams are
deﬁnitely close to its limit.
This lead to the idea of using velcro mounted on the drone, in order to left the mission object,
which also ended up being velcro. This concept ended up being very beneﬁcial and eﬀective.
The idea is to have a red object acting like the mission object, but having the drone lifting some
white velcro bands underneath this red object. This is done because the red object is simply too
heavy for the drone to lift, while the velcro bands are to light on its own, to not blow away when
placed on the ground.

left, the object will appear to move even more to the left. This will mean that the control output

A compensation for the additional error caused by the angle of the drone, will have to be imple-
mented for the drone to have any change of stabilize its position. This error oﬀset is fortunately a
geometric subject, meaning the drones build in sensors are able to solve this problem. Figure 5.4

Both the angle ”v” and the distance ”c” can be found from the drone’s sensors. The gyroscope can
tell the tilt of the drone, while the ultrasonic sensor will inform the system of the distance ”c”. The
compensation ”b” can then be calculated according to Equation 5.5.
b = sin(v) · c
(5.5)
The drone will once again need to know the relation between pixels and meters. This can be done
the same way as in Section 3.3.2, with the diﬀerence being that the distance to the ground will now
also be calculated with the ultrasonic sensor. This is very similar to Equation 5.5, except it is ”a”
that is being calculated.
a = cos(v) · c
(5.6)

5.5
A* (A star)
The drone has so far followed the search path from its map of the ﬁeld. This path will be assumed
to be clear of any obstacles and therefore save to follow, but this does not mean that there can not
be any obstacles inside of the ﬁeld, these just have been taking into a count when generating the
search path. This does however change when the mission object has been found, since the path
from the location of the mission object, to the drop oﬀpoint can not be generated before hand.
This is of course due to the fact that the location of the mission object is not known before the
search start, this path will therefore need to be generated in real-time.
This is where the A* path planning algorithm comes into play. The map is made up by checkpoint
(notes) which are perfect for this method. The goal is to ﬁnd the fastest path from the location
of the mission object, to the location of where this object is set to be delivered. The checkpoints
gets two values each, one called ”g” and being the cost from start to the given checkpoint, and
another called ”h” which is a heuristic function, giving the cost from the given checkpoint to the
goal. The algorithm then evaluate all checkpoints around the stating position and chose the one
with the lowest total cost. Here f(n) is that total cost and n is the given checkpoint.
f(n) = g(n) + h(n)
(5.7)
This checkpoint will then be considered the current checkpoint and all its surrounding checkpoints
will be evaluated the same way. A list of checkpoints will be generated by following this formula.
This list will consist of the checkpoints the drone will have to follow, in order to get to the goal, by
the fastest path.
This project has not been focusing on ﬁelds with any obstacles, but instead ﬁelds with open space
and the A* algorithm will therefore be a bit of a overdo to implement. If the drone however was
not this free to move around, it would be obvious to use this algorithm.
An easy way to make a path for the drone to follow, is to look at the number of lanes between the
checkpoint with the mission object and the checkpoint with the drop oﬀpoint. This combined with
the diﬀerence in reference distance to the markers, will tell the drone how much to move.
An example of this could be if the mission object gets found in lane 2, with a reference distance
of 3.2 meters. The drop oﬀpoint is located in lane 4 with a reference distance of 1.4 meters. The
drone will then have to move two lanes to the side and 1.8 meters towards the marker. This will be
possible if the drone do not have to worry about any obstacles.

Chapter 6
Discussion and Future Work
As Table 5.1 from Chapter 5 shows, the drone does succeed in many of the goals for this project.
Some tasks did get solved as expected, while others got solved in an alternative way. The lifting of
the mission object is a good example of this. The red object in the videos was originally thought
to be lifted, but turned out to be too heavy. This was ﬁxed by lifting white velcro bands instead,
while the drone was still navigating after the red object.
The problems in the project have all roots in the choice of drone. The Parrot AR 2 is a good drone,
but does have some setbacks that makes it hard to work with from time to time. That is why many
of the elements from this project can be used a lot more eﬃcient, if implemented on a high quality
drone.
The navigation and localization system did perform very well in the ﬁnal tests, with 10 out of 10
test succeeding. The PD controller does in general work very well, with an acceptable overshoot
and fast stability. The problem is when the lightning is bad, the marker detection will often then
fail to detect the marker. This will rarely cause the drone to get totally of track though.
The yaw control is made to correct any noise causing the drone to spin around itself. The controller
itself do work very well, fast and with very little to none steady state error. The only downside is
the lack of precise information given to it, since the controller use the gyroscope to estimate the
error. The gyroscope do unfortunately give some odd numbers from time to time and the whole
build in coordinate system seems to drift a bit while time goes on. The yaw control do, because of
this, only get used in the code, at a turn point.
This lead into the last part of the localization: the turn points. The drone do manage to turn
180 degrees after clearing the last checkpoint in a lane and by that facing the next marker. The
command from the API to start spinning is not working that well and a ﬁrmware error causes the
drone to sometimes keep spinning. The only way to get it to stop spinning, is to land it and get
it back up in the air again. This does make the whole ﬂow of the navigation a little bit rough. It
does however manage to clear all checkpoints in the three lane map.
The computer vision made for this project has made good results as well. The features implemented
works great and fulﬁll the goal of ﬁnding the mission object. More features could be added, like

shape matching and multiple object analysing. This will once again require a faster process unit,
since it is simply not possible to allow the sampling time to be any longer.
The loading control and the goal of getting the drone to get a hold of the object do work, but it
does also have some problems. The drone is not always able to land on the mission object, even if
it ﬁnds the object with the ﬁlter. This is due to the control action making the drone lose sight of
the object. The controller code has a backup plan for whenever the object gets out of the frame.
It will make the drone follow the last control action given, before losing sight of the object. It will
continue doing this for 15 samples and give up if the object do not appear, but if it does appear,
even if it is only for one frame, it will restart the backup and make it run again for 15 samples. It
does however manage to get a success rate of 60 % from the last tests.
The ground camera has a very narrow ﬁeld of view, resulting in a very close up look at the ground.
This makes it easy for the drone to lose sight of the object. A faster sampling time or a wider ﬁeld
of view could ﬁx this problem. A third solution would be to make the drone ﬂy at a higher altitude,
this could however make the ﬁlter less eﬀective if the added height makes the detail level of the
object on the ﬂoor decrease.
The grabbing device ended out being velcro and this turned out to work great. The object that
was lifted was also velcro, which means the weight was very light. If the drone should lift a more
heavy object, it would require another grabbing device, since velcro can only lift so much.
This grabbing device could be a electromagnet for lifting heavy metal objects. This implementation
would be great if as an example the drone was operating on a construction ﬁeld or was installing
hardware on an industrial wind turbine. Places that require the drone to lift heavy electrical com-
ponents or metal tools.
Another idea is to use an electric vacuum cup. This would work like a suction cup and could be
able to lift a lot of weight. The items that should be lifted must have a ﬂat surface for this to work.
If the drone should be able to pick up more the one object, maybe if three or four objects was
scattered around the ﬁeld, either the grabbing device or the camera should be moved. As of now
the camera will be blocked after the loading of an object. This could be ﬁxed, but needs to be
decided before the development of the hardware.
The way the drone will deliver the object to the drop oﬀpoint, is by estimating the distance based
on the reference values of both the actual location of the loading and the location of the drop oﬀ
point. This works for the ﬁeld used in this project. The A* algorithm could be used for a more
complicated map with obstacles.

Conclusion
The overall goal for this project was to promote the idea of autonomous drones. To develop an
autonomous system that would show the great potential of drones and how much work they can do
for us. The idea was to combine the diﬀerent ﬁelds of autonomous robots, to create a well rounded

This goal should be accomplished by the development of a drone that can navigate a map on its

The localization system reﬁned by this project, is very consistent and accessible, since the only
sensor used is a camera. It do perform very well, with only few ﬂaws. These includes the slight
drifting and the problems with turn around. This way of localization can be used for many other
robots. Markers could be placed on farming ﬁelds for farming robots to use. This could give the
robots a more precise idea of their position, in an easy and cheap way. GPS is a great sensor, but
can be oﬀby multiple meters, this system can add another level of precision.

time will make room for a more detailed search.

highest precision, it is also the part that suﬀers the most due to the hardware used. The overall

least important relative to the setup for the project. The drone did not operate in a ﬁeld with any
