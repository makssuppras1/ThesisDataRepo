CHAPTER 1. INTRODUCTION
1
CHAPTER 1
Introduction
In the ﬁeld of mobile robotics, the robots are developed to perform a broad range of tasks for humans. Such
as in the industrial manufacturing, transportation of items or humans, and in the military purposes. In this
project the area of interest is on hospitals where they are complex and have dynamic objects. Considering
hospitals with a hug amount of transportation tasks are comparable to a factories.
The common for all of them is that they rely on some kind of representation or belief of their state in the real
world. Without the perception features of world the robot will be deemed to make "guess" of what it is doing
or where it is in an environment. Thus in order to have a robot that is functional it must be able to estimate
its location.
Taking ideal world in to consideration it will know its absolute location and thus it is able to act on that
information. But unfortunately real world is not ideal and this is not always at hand. For the outdoor
environment the robot can use GPS which is able to give an estimated absolute position. Unfortunately this
approach is not feasible for indoor robot localization where it demand higher accuracy than GPS can provide,
in order to avoid it tortes its self or humans.
According to (A.G. Ozkil 2009) there is growing interest of utilizing automated transportation system that
can care items in hospitals. One of the motivations behind this is to reduce operational cost.
Thus the work of this thesis is to enable an automated mobile robot performing the transportation tasks,
that staff can gain beneﬁt of and relieve the workload. Taking as an example, one of the tasks could be
transportation of medicine and blood bags from clinic or lab to the desired room.
Similar to the ﬁeld of the interest has it been seen, systems such as HelpMate Robotics (Long n.d.) that was
designed for transporting items between departments, autonomous navigated and avoided obstacles. One of
the latest system that does the same with more features is TUG robot. (Aethon n.d.)

CHAPTER 2. PROBLEM FORMULATION

CHAPTER 2
Problem Formulation
In many places of work, such as companies institutions and hospitals, humans are still responsible for
transporting items. This type of work typically entails a physical workload, which cause work-related
injuries. By looking at industrial companies, how work is divided between humans and machines, it can be
seen that machines take care of the dull, and heaviest work, in other hand humans undertake the control and
testing aspects of work. This distribution of work is common in the industries where robots and machines
are undertaking much of the hard transportation work.
In this context there is interest in implementation of robots to help staff in hospitals instead of humans
transports items. According to (A.G. Ozkil 2009) the number of hospital patients is increasing in western
countries such as Denmark. Thus the load of transportation is substantially increased.
The aim of this project is to build a system for a mobile robot that is able to help humans by undertaking the
transportation of dull, and heavy items in hospitals.

CHAPTER 3. REQUIREMENT SPECIFICATIONS
3
CHAPTER 3
Requirement speciﬁcations
The mobile robot must be able to fulﬁl the following requirements:
• Robot should have a base station where it can charge and receive new orders.
• It should be able to localize its self.
• It should be able to select the closest path to a goal.
• It should be able to avoid the collisions, and keep distance from obstacles.
• It should be equipped with sensor device in order to precept the environment.
• It should both move manually and be able to navigate autonomous.
• In case of emergency it must be able to stop immediately.
• A proposal for how staff can use the robot should be presented.

4.3
ROS
Robot Operative System (ROS) is a ﬂexible framework for robotics software design. ROS is a meta-operating
system that is not only takes care of hardware abstraction, or process management, but also provides high
level functions such as synchronous, asynchronous calls and robot conﬁguration system. (Garage n.d.)
The robot’s OS consists of several packages, which is one of the main ideas for organizing the software. It
provides tools and libraries to build, modify and run codes across multiple computers, which is an advantage
for this project.
In ROS, nodes are processes that perform the computation from its content. Information sharing between
nodes appears via topic. Node A, shown in the following ﬁgure, sends a message by publishing to given
topic. This topic is used to identify the content of a message. When Node B is interested in that information,
it will subscribe to the topic and get the content of the message. ROS Master takes care of node declaration
and the registration service, which make it possible for nodes to ﬁnd each other and share messages.
Figure 4.3: A diagram of ROS master and two nodes.

4.4
Nav2d Simulator
The simulator is a 2D navigation and localization system for an indoor mobile robot in ROS that is called
Nav2d. It allows users to either autonomously or manually control a robot in a 2D plan. The simulator
contains packages that allow the robot to be capable of reacting on obstacle avoidance, and path planning for
a desired goal position. Nav2d packages are organised in the below ﬁgure based on the concept from ﬁgure:
4.2.
Figure 4.4: Block diagram of Nav2d simulator.
The simulator primarily consists of the following packages:
• Stage: The simulated robot that provides laser scan and wheel encoder information, while it movies
in its environment.
• Self-Localizer: In the "perception" module the Self-Localizer node is constructed to track the robot’s
position from particle ﬁlter along with other models. The following items are the contents of the
Self-Localizer node:
– A model for Odometer
– A model for Beam range measurement
– A model for Likelihood ﬁeld range measurement
– A model for Particle ﬁlter
• Navigator: This is the "Decision making" module of the mobile robot that handles the plan for
navigation. It requires a map, and the robot’s position within the map, in order to create a plan for
navigation. This node implements the Dijkstra algorithm, that allows the robot to ﬁnd the shortest
path to a goal in its environment. (Dijkstra n.d.)
• Operator: This is the "Action" module that handles the robot’s movement, and obstacle avoidance by
changing the robot’s position when it has to follow the path or needs to avoid obstacles.
• Remote control: This is for manually controlling the robot with a joystick. This is a good feature that
can stop the robot if it is about to hit a wall while it is in autonomous mode.

5.1
Probabilistic Robotics
The ﬁeld of probabilistic robotics is applied in this project where it permits us to work on the perception and
action ﬁelds of the mobile robotics system. The idea of probabilistic robotics is to make a representation of
location uncertainty from probabilistic theory calculations in order to localize the robot. The probability
estimation comes with different guesses of where the robot is located instead of only one guess. While the
robot moves around in the given map, different guess are eliminated when the robot observe the obstacles in
its environment. This approach is because the state of the robot cannot be measured from the sensor directly
and used for knowing its pose, because poses are not measurable directly, therefore robot most calculate
its pose from data. Robot localization is based on probabilistic algorithms that processes information from
its environment by laser scan and make move in order to ﬁnd itself in map. This type of localization is
called global localization where the robot can start anywhere in the environment and it has to localize itself
from scratch. The estimation of the robot’s location uses the Bayes ﬁlter algorithm to calculate posterior,
the Probabilistic Belief (bel) from probability density function of all space locations. In general, the Bayes
ﬁlter algorithm has two parts; the ﬁrst is prediction, which is probability distribution function bel(x), and the
second is correction, which is measurement update.
bel(xt) = p(xt | z1:t−1, u1:t)
(5.1)
Where xt is distribution over state conditioned on zt−1 all past measurements at time, and u1:t is all past
controls data. The update of function is to gain the interested sensor measurement and avoid information
lose through processes that increase the robot’s uncertainty. The update measurement, also called correction,
in equation 5.2
bel(xt) = p(zt | xt)bel(xt)
(5.2)
The measurement update bel(xt) is the Bayes ﬁlter algorithm that is multiplied with probability measurement
zt that may have been observed, and it applies for each hypothetical posterior state xt. (S. Thrun 2000).

5.2
Particle ﬁlter
Particle ﬁlter is a Baysian based ﬁlter that is one of nonparametric ﬁlter, which means it does not rely on a
ﬁxed function, like Gaussian ﬁlter. Instead, it makes approximation of previous state (posterior) from a set of
particles, Xt which corresponds to regions in state space. The idea of this method is to make a representation
of posterior belief bel(xt) from set of a random samples with weight function. The construction of bel(xt)
is continuously dependent on belief bel(xt−1) from one time step earlier. The algorithm is divided into 4
following steps:

5.3
Motion model odometry
The motion model is the robot’s motor revolution detection, and processing module where the odometer
algorithm is applied. The odometry motion model uses state transition probability:
p(xt | ut, xt−1)
(5.6)
That describes the posterior distribution over kinematic states where:
xt: Kinematic output state
ut: Motion model (a pair of poses: xt−1 xt )
xt−1: Initial pose (input)
The robot’s conﬁguration is deﬁned by location and orientation, which can be described by 6 variables. The
ﬁrst three are Cartesian coordinates (x, y, z), where z is not used in the two dimensional plan. The remaining
three are Euler angels (Roll, Pitch, and Yaw), where we only use Yaw. Thus, the robot’s pose is denoted of 3
parameters (x, y, θ), where θ is angular orientation, which is also called bearing or heading direction.
Figure 5.1: The circle indicates robot with pose in a coordinate system. (S. Thrun 2000)
The ﬁgure above shows the robot (black circle) in the coordinate system with location (x,y) and orientation
θ. Thus the robot’s pose is the kinematic state xt in probability conditional density in equation (5.6).
For the motion model ut there are two methods to move the robot; the ﬁrst one is the velocity model
and the second is the odometry motion model.
The velocity motion model is based on velocity commands that are given to the robot’s motors. This
model makes the assumption of controlling the robot by rotational velocity and translational velocity. The
second method is the odometry model, which uses kinematic information.
For both models, the motor’s drift and slippage has inﬂuence on accuracy but the velocity model has
an additional drawback where there is a mismatch between actual motion and its mathematical modelling.
In practical cases, the odometry model tends to be more accurate than the velocity model, therefore the
odometry model is applied in Nav2d.

5.4
Range ﬁnder models
Since we are working with Probabilistic robotics and sensor measurements are from the real world, they
are included with noise, which increase the uncertainty. Range ﬁnder models are processing the sensor
measurements that are generated from the world in order to perceive the environment. The uncertainty in the
system can be eliminated with either the Beam Model (BM), or Likelihood Filed Model (LFM) algorithms
which both of them uses "error" models to construct a model for range ﬁnder measurement . The range
ﬁnder model is deﬁned as conditional probability distribution:
p(zt | xt, m)
(5.8)
Where xt is robots pose, zt is measurement, and m is map of environment. In the next three sections
functionality of these two models and the limitations will be described.

5.4.1
Beam model
The error models for Beam range ﬁnder measurement are the approximation models of range ﬁnders. This
model has 4 internal measurements error types and all of them are essential to make this model to work:
1. Small measurement noise
Figure 5.3: Distribution of phit . (S. Thrun 2000)
These types of errors come from the limited resolution of the range sensor, and atmospheric effect on
the measurement signal. The measurement noise is modelled by implementing Gaussian distribution.
2. Errors due to unexpected objects
Figure 5.4: Distribution of pshort . (S. Thrun 2000)
The unexpected objects are eliminated by calculating the range of the closest moving objects. For
example, if there are two moving objects (R1 and R2) with different ranges, and if R1 < R2, then
we want to calculate the range of object R1 then R2. This type of error is described by exponential
distribution.
15

3. Errors due to failures of object detection
Figure 5.5: Distribution of pmax . (S. Thrun 2000)
These types of errors are from sensor failure when the laser range ﬁnder senses black objects, and good
light absorbing materials. These errors can be eliminated when the laser sensor returns its maximum
allowable values.
4. Random unexpected noise measurements
Figure 5.6: Distribution of prand . (S. Thrun 2000)
Range ﬁnders have a tendency to produce unexpected measurements. To avoid these measurement errors, a
model of uniform distribution is spread over the entire sensor measurement.

5.4.2
Limitation of beam model
In practical cases, BM measurements lack smoothness in situations where the robot is in a cluttered
environment with small obstacles. For example, the distribution of small obstacles such as chair and
table legs will give big changes in the robot’s pose, which causes that heading direction is highly affected.
Therefore, another approach is taking care of this problem.
5.4.3
Likelihood ﬁeld model
An alternative model is the likelihood ﬁeld model that overcomes the limitations of the BM.
1. Measurement noise: (Phit)
Noise that arise from the measurement process is modelled using Gaussian. In xy-space this means
ﬁnding the nearest obstacle in the map, by calculating distance (dist) between measurement coordinates
(xz, yz) and the nearest object in the map. The probability of sensor measurement is given by a zero-
centred Gaussian that detects sensor noise:
phit(zk
t | xt, m) = εσhit(dist)
(5.9)
2. Failure: (Pmax)
This is the same as the previous section, "Errors due to failures of object detection" in BM, where
max-range is taken into algorithms assumption.
3. Unexplained random measurements: (Prand)
A uniform distribution (Prand) is used to model random noise.
From these three types of noise and uncertainty, the desired probability distributes:
zhit · phit + zrand · prand + zmax · pmax
(5.10)
A ﬁgure from book: Probabilistic Robotics (S. Thrun 2000) shows an example of resulting distributions of
all three models.

5.5
Occupancy grid mapping
A map in an environment is a list of objects m = {m1, m2, ...mN} with their locations. Occupancy grid
mapping is location-based where it assigns each x, y coordinate in a binary occupancy value that speciﬁes
whether or not a location is occupied with an object.
Figure 5.9: Map m is distributing information for poses (xt) and measurements (zt).
The ﬁgure above shows graphical model of mapping when both poses and measurements are known; the
task is to make a representation of the map.
Mapping presents challenges, due to the the noise and uncertain measurement data that are generated from
the odometry and sensor model. If the sensors were noise free, the challenge would be less.
The occupancy grid mapping algorithm is based on the estimation of the posterior over maps from the given
data set.
p(m | z1:t, x1:t)
(5.11)
m : map (grid cells)
z1:t : Set of measurements
x1:t : Set of poses through a path.
Since the path is known, control u1:t is not used in this section. The parameter m is a set of grid cells where
each cell is a binary occupancy value, whether a cell is occupied or is free (1 or 0).

7.4
Particle distribution
Nav2d simulator has a provided map to work with, where it has simple walls and rooms. The robot can
easily move in this environment. But in order to examine the robot’s performance as a service robot in
hospital, it has to be exposed to challenges which are similar to those presented by objects found in the real
world. Therefor, the simulated robot has been tested in different maps to observe its behavior.
The experiments showed that the robot’s task to a desired goal failed in situations when the robot met many
obstacles, and it was almost impossible to localize it self because the distributed particles were all over the
map. The problems with localization was when the robot begin to move from its initial state where it has to
localize it self. This was due to the particle distribution area being too large, which caused great uncertainty,
and failed. Figure 7.6a shows a large amount of particles assuming that the robot is in the middle of the map,
but the robot is actually still its initial position and has simply moved few centimeters forward. This was due
to particles being distributed over all places in the map.
If the particles are distributed only around the robot at the initial state, then the robot has a better opportunity
to localize it self. In this way, the uncertainty of where the robot is located will decrease. This is shown in
ﬁgure 7.6b, where particles are only around the robot at the beginning. After this modiﬁcation it showed that
the robot was able to avoid the situations where it had to make turns around itself in initial state, in order to
localize it self. This was also more time consuming at beginning for the robot to reach a goal, which is now
reduced.
As a result of this change, the robot was able to localize it self in a short time and was able to reach the goal.
This modiﬁcation was also applied on other test maps with the same results.
The next challenge came when the robot needed to run in between two walls, but was not able to do that
since the distance for avoiding obstacles was set to a high value. This is one of the scaling problem that will
be described in next section.

CHAPTER 9
Conclusion
Through project period the theory of self-localizer models has been analyzed which was implemented
in Nav2d simulator. The performance of the movement in simulator for cluttered environment has been
improved where particles are around the robot’s area. The simulated robot is able to avoid collisions and
keep a distance by change of parameters from conﬁguration setups.
The SMR robot moves with different velocities, depending on how far away obstacles are e.g., it has a higher
velocity in the corridor than it does in rooms while it is manually navigated with the joystick. This should
be optimized in a such way that the robot only moves with one velocity to avoid slippage of wheels while
running in corridors.
The SMR has been able to reach to goal in a few cases on a laboratory map without any obstacles. But it was
not able to do the same performance in iterations. Several examinations has been done in operator node but
without any useful result. Since the operator node is controlling the robot’s direction and publish the velocity
commands then this node should be analyzed more in detail.
