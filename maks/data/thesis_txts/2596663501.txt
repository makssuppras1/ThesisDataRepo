Concatenated FEC for High-throughput Optical Communication
Abstract
Forward error correction (FEC) has become one of the key enabling technologies
for today’s and future optical communication systems. A concatenated staircase and
Hamming code (CSHC) coding scheme has been proposed, which can be applied to
data center interconnects and core optical networks.
In this project, we focus on fast FPGA implementation and veriﬁcation of FEC
for high-throughput optical communication. We investigate a low-complexity maxi-
mum a posteriori probability (MAP) decoding algorithm for soft-decision Hamming
codes, which is based on an eﬃcient implementation of dual code decoding with the
help of fast Hadamard transforms. The fast MAP decoding is applied to Hamming
(128,120) code. Decoding performance and VHDL design for the MAP decoder are
both presented. The goal is single FPGA implementations at data throughput of
400 Gb/s and beyond. The decoding performance and hardware complexity of fast
MAP decoding is further compared with Chase decoding of Hamming (128,119)
code. Synthesis results show that MAP decoding of Hamming (128,120) code fea-
tures lower hardware complexity and provides a higher data throughput. Further-
more, performance of the fast MAP decoding is investigated in a multilevel coding
(MLC) scheme, when 16 and 64 quadrature amplitude modulation (QAM) is used.
Finally a hardware Gaussian noise generator based on the Box-Muller method is
implemented for real-time FEC performance validations on the FPGA.

1
Introduction
1.1
FEC for optical communication
The ever-growing demand for a connected world contributes to the explosive
growth of data being transported across networks [1]. This trend further brings new
challenges and accelerates innovations in the network infrastructure, in particular
data center interconnects (DCI). Data centers are vital for today’s internet based
applications [2].
Distributed data centers need to communicate with each other
to share data, balance workloads and scale capacity when needed [3]. 100 Gigabit
Ethernet (100GE) speeds that are commonly used in data centers today are no longer
fast enough to support the performance demand of new applications and services.
Accordingly there is an urgent need for telecommunication operators to increase the
capacity of data center interconnects to cope with the growth in network traﬃc load.
Inter-DCI links up to 100 km at 400 Gb/s and 800 Gb/s is being developed. Looking
forward, the data rates will be further scaled to 1 Tb/s and beyond.
In order to deal with optical impairments and keep the bit error rate (BER) suf-
ﬁciently low (e.g., 10−15) in transmissions, forward error correction (FEC) is applied
(Figure 1) such that a reliable high-speed data center connection can be achieved. It
becomes evident that FEC represents one of the key enabling technologies for future
optical communication systems.
Figure 1: FEC in optical communication systems
FEC for optical communications has undergone evolution over the years [4]: from
single code to concatenated coding construction, from hard-decision (HD) to soft-
decision (SD), from independent FEC to joint FEC, modulation and probabilistic
shaping design [5]. Figure 2 brieﬂy shows the key characteristics of error-correcting
codes and performance metrics to evaluate FEC decoder implementations.
1
==PAGE:14==

Figure 2: Key characteristics of FEC coding schemes
The SD-FEC provides a large coding gain advantage over the HD-FEC, at the
cost of higher decoding complexity and large amount of data ﬂow to be processed,
which typically increase the chip area and power consumption of the decoder. The
error correction capability of a FEC coding scheme can be characterized by the post-
FEC BER. For HD-FEC, the pre-FEC BER threshold can also be used to predict
the post-FEC BER performance for a given FEC code.
Furthermore, the error
ﬂoor/ﬂare behavior of a FEC coding scheme is investigated and diﬀerent techniques
are proposed to mitigate this problem, e.g., code concatenation [6].
The construction of FEC codes and the corresponding decoding technique are
highly related to diﬀerent application scenarios and their speciﬁc requirements. For
example, ultra long-haul transmission, notably submarine links, where reach is the
most important target, codes with larger overheads and iterative soft-decision decod-
ing can be applied to guarantee high performance of the system. When it comes to
short-reach applications, e.g., inter-DCI links, power consumption and cost should
also be considered. Preferably the power consumption and implementation com-
plexity is maintained or even decreasing when the system throughput increases.
1.2
Concatenated coding
A concatenated staircase and Hamming code (CSHC) coding scheme is speciﬁed
by the Optical Internetworking Forum (OIF) for the 400G ZR standard [7, 8], which
can be applied to short-haul optical transmissions such as data center interconnects
and core optical networks. An output BER down to 10−15 is speciﬁed for the de-
coding FEC performance. A block diagram of the CSHC coding scheme is shown in
2
==PAGE:15==
Concatenated FEC for High-throughput Optical Communication
Figure 3. The inner code uses a double-extended soft-decision Hamming (128,119)
code for error reduction. The outer code uses hard-decision staircase code (255,239)
constructed by extended Bose–Chaudhuri–Hocquenghem (BCH) component codes,
“cleaning up” the residual errors out of the inner decoder. The CSHC scheme re-
sults in a net coding gain (NCG) of ∼10.8 dB (16-QAM) with ∼14.8% overhead [7].
The outer staircase code is concatenated with the inner Hamming code through an
interleaver, denoted by π1, in order to reduce the correlation between bit errors.
Figure 3: Block diagram of a concatenated staircase and Hamming code coding scheme
[10]. π1 and π2 denotes the interleaving and φ denotes the modulation scheme.
In this project, we study the concatenated staircase and Hamming code coding
scheme for high-throughput optical communications, with particular focus on the
inner soft-decision Hamming code, referred to as SD-Hamming.
A low-complexity maximum a posteriori probability (MAP) decoding algorithm
of the SD-Hamming (128,120) code [22] is investigated and applied. We present
VHDL design for the MAP decoder, achieving the data throughput at 400Gb/s and
beyond on a single FPGA. Diﬀerent techniques for eﬃcient FPGA implementations
of ﬁxed-point computations are presented and applied. For fast FEC performance
validations, a hardware Gaussian noise generator is implemented and an emulation
system is ﬁnally set up on the FPGA.

FEC decoding techniques and implementation
In this project, we have conducted a survey of the latest researching works in the
literature. Hardware design and implementation of FEC is investigated in Section
2.1.
Related works on diﬀerent approaches to decoding SD-Hamming codes are
presented in Section 2.2, with particular focus on fast MAP decoding.
2.1
Hardware design and implementation of FEC
Performance of diﬀerent FEC coding schemes and decoding techniques can be
veriﬁed using Monte Carlo simulations, providing theoretical guidance for the de-
sign and optimization of FEC solutions. However, the simulated performance results
are not always feasible for practical implementations. Today’s high speed optical
communication systems have raised diﬀerent requirements for the FEC design. Par-
allelization of decoding implementations is deployed to achieve high throughput of
the system. Wen it comes to SD-FEC codes, quantization bit-width of the soft in-
formation should also be taken into consideration. Speciﬁc system applications have
more stringent requirements on energy consumption and chip footprint.
Considering the aforementioned metrics, hardware design and implementation
of FEC solutions have received great interest among researchers. Comparing with
Application-speciﬁc integrated circuits (ASIC), Field Programmable Gate Array
(FPGA) is usually preferred for prototyping and validating the design due to its
reconﬁgurability. Fast and hardware-friendly FEC solutions are investigated and
VDHL design is synthesized for the target FPGA device or ASIC fabrication technol-
ogy. A summary of related works on FPGA/ASIC implementation of FEC solutions
is presented in Table I.
Among these works, some of them focus on single FPGA implementation of
the proposed FEC solution, such as [17] for HD product codes. Synthesis results
are reported, including the estimation of data throughput, hardware resource re-
quirements (chip area) and power consumption. Low circuit complexity and energy
eﬃcient FEC implementation on ASICs is well investigated in [14] and [18]. Rate-
adaptive designs are studied in [13] and [16].
On the other hand, [11, 15, 16, 18] considered real-time FEC performance em-
ulation on FPGAs. In comparison with computer software simulations, hardware
emulations by FPGA prototyping enables us to shorten the veriﬁcation time [6], espe-
cially when enough statistics should be collected to investigate the FEC performance
at ultra-low BERs such as 10−15. In these works a complete veriﬁcation system is
set up on the FPGA, including a random data source generator, FEC encoder &
decoder, modulator & de-modulator and the transmission channel is modeled. Cai

SD-Hamming decoding approaches

techniques, but performance results for the fast MAP are not reported in [23]. I.

MAP is applied to decode extended BCH (128,120) code, which is the same as ex-
tended Hamming (128,120) code. The decoding performance outperforms its TPC
counterpart, using Chase and BCJR for decoding, respectively. In [25] General-
ized LDPC (GLDPC) code with Reed-Muller (RM) and BCH codes as component

3
Hamming codes and fast SD decoding
In this section, Hamming codes as error-reducing codes in the concatenated cod-
ing scheme is ﬁrst introduced. Chase decoding and MAP decoding for fast soft-
decision decoding of Hamming codes are presented. We provide relevant theoretical
background for fast MAP decoding and details of the decoding algorithm. Finally
the computational complexity of fast MAP decoding is analyzed.
3.1
Hamming codes
Hamming codes belong to the class of linear block codes. For a positive integer
m ≥3, the (n, k) Hamming code can be deﬁned, where the codeword length n =
2m −1, information bit length k = 2m −m −1 and the minimum distance dmin = 3.
Thus it is capable of correcting a single error. The associated m × (2m −1) parity-
check matrix H has all non-zero m-tuples as its columns exactly once. The order
may be chosen to ease implementation or to provide some speciﬁc properties.
There are diﬀerent techniques to modify the basic version of a error-correcting
code. For a linear block code, each code is characterized by the three parameters:
codeword length n, dimension k and its redundancy r = n −k. Given that one
of the three parameters is ﬁxed, the other two parameters can increase or decrease
accordingly. In this way we may ﬁnd new codes with diﬀerent properties. By adding
an overall parity check bit to a binary Hamming code (n, k), an extended Hamming
code can be obtained, with codeword length n = 2m and the minimum distance
dmin is increased to 4. The extended Hamming code is known as a single-error-
correcting and double-error-detecting code. By expurgating some codewords away
(e.g., codewords of odd weight), the minimum distance can be increased as well.
In this project, we focus on the (128,120) extended Hamming code. The (128,119)
double-extended Hamming code speciﬁed by OIF 400G ZR [8] is studied as well.
In the context of our concatenated coding scheme, the inner SD-Hamming code
is used for error reduction. A fraction of the corrupted bits in a received word is
corrected, which brings the output BER down to the pre-FEC BER threshold of the
outer decoder [30]. A more powerful outer code decoder cleans up the remaining
errors and take the BER further down, e.g. 10−15.
3.2
Chase decoding
The Chase decoding [21] provides a class of reliability-based algorithms achieving
bounded-distance (BD) decoding.
It can be applied to a wide variety of error-
correcting codes that are designed for hard-decision decoding, extending them to
8
==PAGE:21==
Concatenated FEC for High-throughput Optical Communication
soft-decision decoding.
The Chase decoding can be illustrated by Figure 4. The soft values (“Soft in”)
retrieved from received symbols are ﬁrst sorted in an increasing order. Instead of a
complete sorting of all the values, we only need to ﬁnd the smallest K positions which
indicate the least reliable K bits in each block, and they are selected as candidates
for bit-ﬂipping. A set of test patterns are generated from diﬀerent combinations
of the K least reliable position (LRP)s. Based on each pattern, the received hard
bits (“Hard in”) are ﬂipped. The resulting words are then fed into parallel hard-
decision Hamming decoders, where an actual decoding is performed according to
the calculated syndromes. Finally, the codeword that has the shortest distance from
the received values is selected as the decoding result.
Figure 4: Chase decoding of Hamming codes
In [21] three algorithms have been proposed, referred to as “Chase-I”, “Chase-
II” and “Chase-III”. and distinguished by diﬀerent set of testing patterns for bit
ﬂipping in the decoder. Among the three algorithms “Chase-II” has a good tradeoﬀ
between decoding complexity and performance, and it has been applied to several
works, see e.g. [15]. In this project, we consider Chase decoding for SD-Hamming
(128,119) code. The performance and hardware complexity is compared with that
of MAP decoding.
3.3
MAP decoding
3.3.1
Dual code decoding
A MAP decoder of a linear code computes a posteriori probabilities for all values
of each transmitted symbol and choose the maximum one. One option for imple-
menting the MAP is the BCJR (after Bahl, Cocke, Jelinek and Raviv) algorithm
[20] based on trellis representation of a code. It is a computationally complex algo-
rithm. For short length codes, the BCJR decoding could be applied and feasible for
hardware implementations. For SD-Hamming code with length 128, the complexity
of BCJR is much higher, compared with Chase decoding or fast MAP decoding.

5.1
Fixed-point computations
Although the simulated results in Figure 5 demonstrate the MAP decoding per-
formance with quantization on the input soft values, the remaining steps for com-
puting the intermediate values within the MAP decoder still uses the full precision
of ﬂoating-point numbers, e.g., the IEEE double-precision (64-bit) ﬂoating-point
format in MATLAB simulations. When it comes to VHDL design, the ﬁxed-point
arithmetic is performed instead, since they are as straight-forward and eﬃcient as
integers arithmetic.
In two’s complement, each value is quantized to a ﬁnite precision, represented
by one sign bit, L integer bits and N fraction bits. The quantization error is accu-
mulated and propagated in the computation chain, and the actual eﬀective number
of soft information bits becomes smaller, which degrades the MAP decoding perfor-
mance. Accordingly, ﬁnding the minimum requirement of data widths to store the
primary variables within the MAP decoder is a critical part in the VHDL design.
It should also be noted that, since the inner code serves as an error-reducing code,
a slight loss in performance due to the ﬁxed-point computations can be tolerated if
fewer fractional bits are allocated to the internal signals.

5.3
Fast Hadamard transform
Eﬃcient architectures suitable for FPGA implementations of FHT have been
proposed in the literature. However, to the best of our knowledge, most of the pre-
vious works aim for much smaller size FHTs (up to size of 16×16, see e.g. [35, 37])
and shorter input/output word-lengths. Although their approaches and fast algo-
rithms can be easily extended to larger size FHTs due to the properties of Hadamard
matrix (2), implementation results for size 128×128 FHT have not been reported.
Synthesis results in [35, 37] target for earlier generations of FPGA platforms, which
are less powerful than the Xilinx UltraScale+ platform considered in this project.
In these works, an assumption has been made that the available FPGA resources
are used only by the FHT, i.e., the FHT design block is not considered in a more
complex design when combining with other components on the FPGA.
Among the computation steps in the fast MAP algorithm, the twofold use of FHT
requires the largest number of addition and subtraction operations (see Table II).
Accordingly, an area-eﬃcient and high-speed implementation of the fast Hadamard
transform is our main focus in the VHDL design.
Due to the recursive property of the Hadamard transform (2), the FHT can be
implemented in a fully pipelined way. The size 4×4 transform is considered as a
kernel module and computed in two stages. Each stage consists of 2 adders and 2
subtractors (Fig.12a). Then the size 8×8 transform is further implemented in three
25
==PAGE:38==
Concatenated FEC for High-throughput Optical Communication
stages, and the result can be obtained from two size 4×4 transform modules and
an additional stage of addition/subtraction operations (Fig.12b). Similarly, the size
128×128 transform in the MAP decoder can be computed from these short length
modules, and cascaded in a 7-stage pipeline. This design can execute FHT in a
minimum number of clock cycles, at the expense of requiring more FPGA resources.

8.1
Gaussian noise generation approaches
As a good model for ampliﬁed spontaneous emission (ASE) noise and thermal
noise dominated scenarios in optical ﬁber systems, an AWGN (Additive white Gaus-
sian noise) channel is usually used to evaluate the performance of a communication
system, such as a BER vs. SNR relationship.
To model the AWGN channel, one can generate the noise samples according to
the cumulative density function (CDF) of a Gaussian distribution, which can be
interpreted by the Q-function for a given SNR value and modulation format. The
probabilities can be easily computed oﬀ-line and stored in look-up-tables. However,
separate tables are required for a series of SNR values, making the design less ﬂexible.
Other popular transformation based methods include the Box-Muller method, which
can be deployed independently [49] or in conjunction with the Central Limit Theorem
(CLT) [47].
In this project, we choose the Box-Muller method for VHDL implementation of
the AWGN channel. It has been applied in many previous works on FPGA emulation
of the FEC performance see, e.g., [15], [16]. However, implementation details for the
Box-Muller GNG are not given in [16] and FPGA resource requirement for emulating
the channel is not reported in [15, 16].
[49] and references therein presented detailed analytical error analysis and bit-
width optimization strategies for evaluation of the elementary functions involved in
the Box-Muller method. Their work targets for an earlier generation of Xilinx FPGA
platform. Here we follow [49] to implement the Box-Muller Gaussian noise generator
with minor modiﬁcations. A reference MATLAB model is set up to analyze the
impact of ﬁxed-point computations on the quality of the generated noise samples.

To initialize the register value in the Tausworthe URNG, one should be careful
about the seed selection. The random number generator could be ﬂawed if a wrong
combination of seeds is used. In our implementation of the Tausworthe URNG, 0
should not be used since any logical operation will keep the output value as 0 for the
next state of the shift register. 1 is also a bad seed for the generator here, leading to
a ﬁxed output value 0. This problem results from the output from the logical AND
operation in the feedback circuit, which gets stuck at 0 if the initial state of the
shift register is set to 1. To avoid this, an initialization algorithm is provided in [48]
to bring the generator in a valid initial state. When parallel multiple Box-Muller
GNGs are paralleled for high-throughput, we should assign diﬀerent seeds and make
sure that no overlapping data patterns occur.

Conclusion
10.1
Summary
In this project, a concatenated staircase and Hamming code coding scheme for
400 Gb/s and beyond optical communications is studied. We investigate the fast
decoding of inner soft-decision Hamming code, with particular focus on the low-
complexity MAP decoding based on an eﬃcient implementation of dual code decod-
ing with the help of fast Hadamard transforms.
Fast MAP decoding of SD-Hamming (128,120) code and Chase decoding of SD-
Hamming (128,119) code are both investigated, both in terms of their decoding
performance and hardware complexity. VHDL design for the MAP decoder is in-
vestigated and further optimized, achieving the data throughput at 400Gb/s and
800Gb/s on the Xilinx UltraScale+ FPGA. Synthesis results indicate that, both the
Chase and the MAP decoder leave suﬃcient resources available to also accommo-
date a powerful outer hard-decision code, on a single FPGA. Compared with Chase
decoding of SD-Hamming (128,119) code, MAP decoding of SD-Hamming (128,120)
code features lower hardware complexity and provides a higher data throughput.
Furthermore, we investigate the performance of MAP decoding in a multi-level
coding scheme when using 16 and 64 QAM. For fast FEC performance validations, a
hardware Gaussian noise generator based on the Box-Muller method is implemented
and an emulation system is set up on the FPGA.

Abstract—In
this
paper,
we
consider
fast
decoding
of
soft-decision (SD) Hamming codes as inner codes in concatenated
forward error-correction (FEC) schemes for high-speed optical
communication. The goal is single FPGA implementations at
speeds of 400 Gb/s and beyond. A low complexity maximum a
posteriori (MAP) probability decoding is applied to a (128,120)
Hamming code. Chase decoding of a (128,119) Hamming code is
also implemented. The VHDL designs for both decoding schemes
are presented. The FEC performance and FPGA resource uti-
lization are investigated and compared. Synthesis results indicate
that, both the Chase and the MAP decoder leave sufﬁcient
resources available to also accommodate a powerful outer hard
decision code, on a single FPGA. Furthermore, MAP decoding
of (128,120) Hamming code features lower hardware complexity
and provides a higher data throughput.
Index Terms—Soft-decision, Hamming codes, Chase decoding,
concatenated coding, MAP decoding, VHDL, FPGA.
I. INTRODUCTION
For fast and reliable high-speed connections between data
centers, forward error correction (FEC) is applied. In optical
communication, FEC provides the required bit error rate
(BER), which is often speciﬁed at 10−15 [1], [2]. Highly
efﬁcient hard-decision (HD) FECs are deployed in optical
communication using product codes [3], [4] or staircase codes
[5], [6].
To improve the performance further, soft-decision (SD)
FECs are studied and proposed, at the expense of increased
complexity. One approach is concatenated coding using a SD
inner code and a HD outer code [7], as in the 400G ZR
implementation, recently speciﬁed by the Optical Interconnect
Forum (OIF) [8], [9]. This system combines an inner SD-
Hamming code with a HD outer staircase code in a 16 Quadra-
ture Amplitude Modulation (QAM) optical communication
system targeted at Data Center Interconnects (DCIs).
In this paper, we consider fast decoding of soft-decision
Hamming codes. We shall refer to this setting as SD-
Hamming. We consider the case where the SD-Hamming code
is intended as the inner code in a concatenated coding scheme
(Fig. 1), as in OIF 400G ZR [8], [9], where the outer code is
a HD staircase code.
This work was supported by the IFD INCOM project and DNRF Research
CoE SPOC, ref. DNRF123.
Fig. 1.
Block diagram of concatenated FEC coding and modulation over
AWGN channel with an inner SD-Hamming code and the outer staircase
code. π denotes interleaving and φ the mapping of the modulation format.
We set the 4.5×10−3 output BER of the inner code as our
target to match the 10−15 output BER of an outer staircase
code, as is speciﬁed in [1] at the overhead ∼6.7%. Higher
BER can also be relevant, if relaxing the BER requirement
out of the outer code [1] or allowing a higher overhead
[10]. An implementation of 400G ZR based on 50 FPGAs
was presented in [9] for veriﬁcation and investigation. This
implementation had a throughput of 200 Gb/s.
The goal here is fast decoding of the FEC enabling a
400 Gb/s concatenated decoding implementation on a single
FPGA. We also investigate the feasibility of having SD-
Hamming decoding at 800 Gb/s on the FPGA. We focus
on a fast maximum a posteriori (MAP) probability decoding
scheme for SD-Hamming. For comparison we implement and
compare with fast decoding based on a Chase algorithm.
The FEC solution presented in this work can be deployed
to potential ZR+ applications, with extended capabilities of
400G ZR.
Sec. II presents related works on different approaches for
SD-Hamming decoding, with a particular focus on fast MAP.
In Sec. III, the Hamming codes considered are presented
together with fast soft-decision decoding using both fast MAP
decoding and Chase decoding. In Sec. IV, a VHDL design
is presented and in Sec. V, VHDL synthesis results towards
400G/s and 800G/s are presented. Sec. VI discusses the
concatenation of SD-Hamming with the HD outer code.
II. RELATED WORKS
Chase decoding [11] and fast MAP decoding [12] have been
considered and deployed in different coding schemes. A low-
complexity serial code concatenation for high-speed optical
==PAGE:74==
communication systems was proposed in [13], where the inner
code is a Turbo Product Code (TPC) based on two extended
Hamming code, with parameters (128,120) and (64,57). The
fast MAP algorithm is used among other techniques, but no
performance results were reported for the fast MAP in [13].
I. B. Djordjevic et al. proposed a reverse concatenated BCH-
LDPC code [14]. Fast MAP is applied to decode extended
BCH (128,120) code, which is the same as extended Hamming
(128,120) code. The decoding performance outperforms its
TPC counterpart, using Chase and BCJR for decoding, re-
spectively. In [15] Generalized LDPC code with Reed-Muller
(RM) and BCH codes as component code was considered.
After decomposing the RM(4,6) to several ﬁrst-order RM
codes, 64 parallel low-complexity MAP decoders are used for
decoding. It has been shown that the RM(4,6)-based GLDPC
code features a much lower decoding complexity, compared
with the TPC based on BCH(128,113) × BCH(256,239),
which requires 239 parallel Chase decoders. Chase decoding
for TPC constructed by extended Hamming code (64,57) was
considered in [16], and the synthesis results on a Xilinx Artix-
7 FPGA were presented. In [9], Chase decoding of the 400G
ZR (128,119) Hamming code was implemented. We design
and implement a Chase decoder for the same code.
It is noted that MAP decoding for SD-Hamming (128,120)
is only seen in [13] and [14] as part of different coding
schemes. [13]–[15] deployed fast MAP in simulations for ex-
tended Hamming/BCH and RM code decoding due to its low-
complexity, whose structure is also suitable for FPGA/ASIC
implementation. However, to the best of our knowledge,
hardware implementation results for fast MAP decoding have
not been reported in the literature.
III. HAMMING CODES AND SD DECODING
We consider Hamming codes as the inner code in a concate-
nated coding scheme as in OIF 400G ZR [8], where the outer
code is an HD staircase Code. Thus, the Hamming code serve
as an error-reducing code utilizing the soft information in the
received signal. The outer code serves the purpose of bringing
the BER to the low values required, e.g. 10−15. This means a
combined design, where the BER output of the SD-Hamming
decoding is required to be better than, e.g. 4.5 × 10−3 [1],
depending on the channel and the parameters and performance
of decoding the outer code.
A. Hamming codes
Hamming codes were the ﬁrst error-correcting code to be
deﬁned. The (n, k) = (2m −1, 2m −m −1) code is speciﬁed
from the (2m −1) × m parity check matrix having all non-
zero m-tuples as rows in some order. The order may be
chosen to ease implementation or to provide some speciﬁc
properties. Hamming codes have a minimum distance of 3,
and thus are always able to correct one error. The codes may
be extended by an overall parity check to have length 2m
and minimum distance 4. We shall consider the (128,120)
extended Hamming code below. Error performance could be
improved by expurgating some codewords away by converting
an information symbol into an extra parity check which is
speciﬁed by an extra column in the parity check matrix. We
shall consider such a (128,119) code as speciﬁed by OIF in
[8]. The extra column in the parity check matrix excludes
codewords where a selection of positions (64) has odd weight.
For SD decoding of the Hamming codes, we assume that
the demodulator provides the Log-Likelihood-Ratio (LLR) for
each received value yj given transmitted binary cj at time j

too complex, compared to the fast Chase and MAP decoding
considered.

C. Chase decoding of (128,119) Hamming code
The basic idea of Chase decoding [11] is simply to ﬁnd
multiple possible error patterns for the received word by a
combination of bit-ﬂipping and hard decision decoding. For
each error pattern, the distance to the actual received word
is calculated based on the soft input values. Finally, the error
pattern with the best distance (maximum likelihood) is chosen
as the most likely error pattern. Chase decoding can and has
been applied to a wide variety of codes designed for HD

IV. SD-HAMMING VHDL DESIGN
The VHDL design of the SD-Hamming decoders studied
differ signiﬁcantly. The transforms are applied in the MAP
decoder, whereas many parallel decoding instances are per-
formed based on potential error patterns in the Chase decoder.
A. MAP decoding of (128,120) Hamming code
1) Fixed-point computations: Although the simulated re-
sults in Fig. 2 demonstrate the MAP decoder performance
with quantization on the soft values, the computation of soft
decisions, in principle, here still uses full precision of the
ﬂoating point numbers in all steps. When it comes to VHDL
design, the ﬁxed-point arithmetic is performed for computing
all the temporary values. To further reduce the hardware
complexity and allow for fast implementation, exponentiation
and logarithm operations in the MAP decoding algorithm have
been substituted by pre-computed values in look-up tables,
based on the knowledge that input values have been quantized
to a given precision.
In two’s complement, each value is quantized to a ﬁ-
nite precision, represented by one sign bit, L integer bits
and N fraction bits. The quantization error is accumulated
and propagated in each computational step, and the actual
effective number of soft information bits becomes smaller,
which degrades the decoding performance. Hence, ﬁnding
the minimum requirement of bit-width to store the primary
variables within the MAP decoder is a critical part in the
VHDL design.
A rough estimate can be ﬁrstly achieved by simulations,
gathering enough statistics for all internal values at different
SNRs. This involves ﬁnding the maximum absolute value to
determine the number of bits to allocate, and ﬁnding the mini-
mum absolute value and minimum fraction for each parameter
to determine the number of fractional bits to allocate. Since
the inner code serves as an error-reducing code, a slight loss
in performance can be tolerated using fewer fractional bits.
From the detailed description presented in [12], the value of
ρ is only used in the last step for computing the soft decision.
By comparison, FHT is performed on τ (3) to compute the
sum of logarithms of probabilities in (5). Hence quantization
errors in τ are more likely to accumulate and the precision of
τ plays a dominant role in the MAP decoder. By comparing
different bit width settings, the desired decoding performance
can be achieved, when allocating 5 and 8 fractional bits to
ρ (2) and τ (3), respectively. For all other temporary values,
they are set to 16-bit width, which appears to be enough by
conducting behavioral simulations for the VHDL design.
2) Fast Hadamard transform: Among all the computations
of temporary values in the fast MAP algorithm, the twofold
use of FHT (6) requires the largest number of add/subtract
operations. Accordingly, an area-efﬁcient and high-speed im-
plementation of FHT is our main focus in the VHDL design.
Due to the recursive property of the Hadamard transform,
the FHT can be implemented in a fully pipelined way. The
size 4×4 transform is considered as a kernel module and
computed in two stages. Each stage consists of 2 adders
and 2 subtractors. Then the size 8×8 transform is further
implemented in three stages, and the result is obtained from
two 4-point transform modules and an additional stage of
add/subtract operations. Similarly, the 128×128 transform in
the MAP decoder can be easily computed from these short
length modules, and cascaded in a 7-stage pipeline. This
design can execute FHT in a minimum number of clock
cycles, at the expense of requiring more FPGA resources.
Alternatively, to allow for efﬁcient hardware resource reuse,
an iterative FHT architecture is considered as the explanation
in (6). The 128×128 FHT design block in FPGA is constructed
by one stage consisting of 64 adders and 64 substractors. The
termination of computation is controlled by a counter and
extra logic units, which switch the register values out of the
FHT back to the FHT input multiplexer, and ﬁnally outputs
the FHT result after 7 iterations. Both of these two ways have
been implemented and performance results are given in Sec.
V-C1.
B. Chase decoding of (128,119) Hamming code
First, our Chase decoder does a partial sorting of the soft-
decision input values to identify unreliable bits. Then, parallel
hard-decision decoders will decode the received word with
selected combinations of bits ﬂipped, and ﬁnally the best
one is selected. To optimize resources, an initial syndrome
is calculated and thereafter modiﬁed according to the ﬂip-
patterns.
1) Sorting the soft-values: Each soft input value is received
as one sign bit and 4 soft decision bits. A partial sorting of the
128 values is ﬁrst performed to ﬁnd the 6 minimum absolute
LLR values. All the 128 values of one code word arrive in
each clock cycle and 128 instances of the basic circuit are
required. The solution we chose, is based on the idea of
merging two sorted lists of 6 elements into one sorted list
==PAGE:77==
of the 6 minimum elements. First the 128 values are inserted
into 22 sorted lists of length 6. Then these lists are merged
in a tree structure to one ﬁnal list of the 6 minimum values.
2) Syndrome calculation: The syndrome is calculated by
multiplication of the received 128 bit sign vector and the
128×9 parity check matrix - modulo 2 [8]. Each syndrome
bit is an XOR of the sign vector bits at the positions in the
column where the parity matrix is 1.
Some syndromes indicate that 1 error is the most likely
while others indicate more than 1 error. We only make a
correction when the syndrome indicates 1 error. Since the
last syndrome bit is just an XOR of all 128 positions, all
syndromes indicating 1 error have this bit as a 1. This means
that some of the ﬂipping patterns can be excluded based on
this syndrome bit. If it is already 1, we will ﬂip an even
number of positions (0 and 2) otherwise we will ﬂip an
odd number of bits (1 and 3). Based on this parity bit, the
maximum number of Hamming decoders required is reduced
from 42 to 26.
The basic syndrome calculated based on the received bits
is modiﬁed to match the various ﬂip-patterns. This is found
by a table look-up and the modiﬁcations are XOR’ed to the
basic syndrome for each ﬂip-pattern.
3) Hamming decoder: After the syndrome calculation, the
error position is found by a look-up table with 256 entries. The
fast implementation uses 26 Hamming decoders in parallel.
V. FPGA SYNTHESIS RESULTS
A. FPGA device and synthesis tool
The SD-Hamming decoder VHDL design has been synthe-
sized for a Xilinx Virtex UltraScale+ family device: XCVU9P-
L2FLGA2104. The synthesis, place and route use all the
default strategies in Vivado v2019.1.3. In the target UltraScale
architecture, each Conﬁgurable Logic Block (CLB) provides
8 × 6-input look-up table (LUT)s and 16 ﬂip ﬂops [18]. 2160
on-chip block RAM blocks are available in this VU9P device.
Each block stores up to 36 Kb data, which can be conﬁgured
as either two independent 18 Kb RAMs (RAMB18E2), or one
36 Kb RAM (RAMB36E2).
B. Performance and complexity metrics
After the place and route process, the maximum operating
frequency (Fmax) is achieved, based on which the information
data rate is calculated. The hardware complexity is depicted
by the logic utilization (CLB, CLB LUTs, CLB Registers)
and the number of occupied block RAMs [19]. Dedicated
multipliers on the FPGA are not used. Power consumption has
not been considered in our work. The numbers for resource
consumption and their percentage of maximum available
amount are both presented in Tables I-II.
C. Analysis of results
1) MAP decoder: The MAP decoder is synthesized under
different settings of input quantization levels (Fig. 2). We have
investigated 8, 6 and 4 bits, respectively. The logic utilization
is slightly lower when using fewer bits. This is attributed to
the reduced size of look-up tables to store ρ (2) and τ (3).
In this work, we focus on the 8 bits (6 fraction bits) setting,
which gives a better decoding performance.
As is discussed in Sec. IV-A, we’ve considered two different
architectures to implement the FHT block. Both of these
VHDL designs enable a data rate at 400 Gb/s and 800
Gb/s, using 9 and 20 parallel MAP decoders, respectively.
MAP decoding based on the fully pipelined FHT architecture
requires 33.4% and 70.1% of CLBs for 400 Gb/s and 800
Gb/s, respectively. This is two times higher than that of the
other version using the iterative reuse FHT architecture, as is
shown in Table I. Speciﬁcally, the two FHT blocks occupy
approximately 60% of the CLB LUTs used in the MAP
decoder (9250 out of 14716). Accordingly, the iterative reuse
FHT architecture greatly lowers the requirement of FPGA
resource utilization and reduces the overall complexity.
It is worth mentioning that, such an iterative architecture
could possibly lead to inferior timing closure performance
when the combined design gets larger. The increased routing
delay lies in the path between the FHT output and FHT input
multiplexer, i.e., the path to perform each iteration. According
to our experiments, there is still a good timing margin in
this path when targeting at 400G and 800G implementations,
since parallelizing 9 and 20 MAP decoders also relaxes the
minimum speed requirement of each decoder.
2) Chase decoder: Table II shows the synthesis results
of a single Chase decoder, which is conﬁgured to ﬁnd 6
candidates for ﬂipping and correcting up to 4 errors, as
is illustrated in Sec. III-C and performance given by the
green curve shown in Fig. 3. By using 8 and 20 parallel
Chase decoders, the achieved information rates out of the
SD-Hamming decoder are 428.4 and 854.4 Gb/s, respectively.
This allows for redundancy required by an outer code for 400
and 800 Gb/s information rate out of a concatenated coding
scheme. A lower hardware complexity of the Chase decoder
can be achieved by considering fewer ﬂipping candidates.
When ﬁnding 5 candidates for ﬂipping (see “ﬂip 5 reduced”
in Fig. 4), synthesis results show that the CLB utilization
is reduced to 1.7%, at the cost of 0.1 dB in performance
compared with the Chase decoder above.
VI. CONCATENATED CODING
The target application of the SD-Hamming code is, as
mentioned, to concatenate it with a HD outer code for a
400G (and beyond) FEC solution. We have evaluated the
performance of (128,119) Chase decoding and (128,120)
MAP decoding respectively in a concatenated staircase and
Hamming code (CSHC) setting, as in 400G ZR [8]. In a
CSHC set-up with 16QAM modulation, the MAP decoder
achieved a 0.1 dB improvement over the Chase in terms of
gap to constrained capacity for 16QAM modulation (without
constellation shaping) at an overall output BER of 10−9
[20]. As shown in Sec. V-C1, the MAP decoder requires
15.5% and 33.6% of the CLBs on the UltraScale+ FPGA,
for 400Gb/s and 800Gb/s respectively. This leaves sufﬁcient
resources for an efﬁcient outer HD code. As an example,

decodings, the product code is comparable to a staircase code
with window size 4 and 1 iteration in terms of complexity.

on the more powerful UltraScale+ FPGA, we conjecture that
there will also be space for a staircase code implementation
at 400Gb/s, also with window size more than 4 and/or more
iterations. Thus, if we aim for a fast low complexity set-up
of a staircase code in terms of window size and number of
iterations, it should also be feasible to implement along with
especially the MAP but also the Chase SD-Hamming on a
single FPGA.
VII. CONCLUSION
In this paper, two approaches to decoding SD-Hamming
codes were presented, based on the fast MAP algorithm and
Chase algorithm, respectively. In both cases, VHDL design
and synthesis results towards 400 and 800 Gb/s on a single
Xilinx UltraScale+ FPGA were presented. Compared with
the Chase decoder for SD-Hamming (128,119), the MAP
decoder for SD-Hamming (128,120) features a lower hardware
complexity for FPGA implementation and achieves a higher
information data throughput. By paralleling 9 and 20 MAP
decoders, 15.5% and 33.6% of CLBs are utilized to achieve a
maximum net data rate of 452 Gb/s and 890 Gb/s, respectively.
Furthermore, a slightly higher error correction performance
is obtained using MAP decoding of the (128,120) Hamming
code. It is further argued that, concatenating the MAP or
Chase based decoding of SD-Hamming with a HD product
code or a low complexity HD staircase code, both based on
BCH component codes, decoding of a combined concatenated
code at 400 and 800 Gb/s on a single Xilinx UltraScale+
FPGA is feasible.
