Summary (English)
The goal of the thesis is to explore and formalize the model named Finite State
Machine with Datapath(FSMD). Based on the ﬁndings a high level hardware
description language is deﬁned, with clearly deﬁned semantics and a syntax that
is closely related to the theoretical model proposed. Both static and dynamic
semantics are deﬁned, and both are expressed using f# as a meta language.
The programs deﬁning semantics doubles as a type checker and interpreter for
concrete hardware descriptions expressed in the designed language.
Finally
the language is explored by implementing a description of an elevator, and this
process gives rise to areas of future work. The suggested areas of suggested work
includes extending the language with a loop construct named macros as the
elevator example revealed it to be tedious to implement hardware descriptions
with high repetitiveness. Additionally it is proposed to the extend the language
to handle interaction between multiple FSMDs, in order to help modularization
of hardware designs.

Chapter 1
Introduction
Motivation
While it is easily taken for granted, embedded systems are found in a plethora of
devices that we interact with daily. These are the systems that control our cars,
regulate the temperature of our fridges and ensure that the traﬃc lights work as
intended. Additionally as the complexity of digital electronic circuits are con-
stantly increasing, design of Hardware Description Languages(HDLs) that can
help overcome this complexity is of great importance. These languages can be
speciﬁcally designed to assist in handling the concurrency of hardware design,
that is often complex to model with general-purpose programming languages.
The area of HDLs is well established, and some of the most popular HDLs in-
clude Verilog[Tho99] and VHDL[VHD09].
Background
With Verilog and VHDL being the most widely spread HDLs, both allow hard-
ware designers to synthesize a hardware description created in these languages,
if the description constrains itself to a subset of the language. The two diﬀer
in their type systems, with VHDL being strongly typed while Verilog is not.
==PAGE:14==

Furthermore VHDL allow for incorporation of low-level details, speciﬁc to a
particular hardware technology[MH06], which allows for more optimization at
the cost of tying a hardware description to a speciﬁc technology.
Another HDL named gezel[Sch], can be considered to be at a higher abstraction
level than VHDL and Verilog, and as such is independent of any implementa-
tion technology. However, Gezel lacks formal semantics, and it is therefore hard
to get a deep understanding of some of the language constructs[MH06]. Gezel
is based on the Finite State Machine with Datapath(FSMD) model, that has
proven to be useful when designing hardware[Sch13].
This model separates control ﬂow and data processing by handling the former
in a ﬁnite state machine(fsm), and the latter in what is known as a datapath.
To demonstrate FSMD, an elevator is suitable to model, as the complexity of
such a model can range from trivial to substantial. A trivial elevator is illus-
trated on ﬁgure 1.1. The elevator has two ﬂoors, and buttons (B) on each ﬂoor
that users can press to create a request for the elevator to move to that ﬂoor.

can open and close at command. When modelling an elevator as an FSMD,
the datapath can be seen as an entity exposing a set of actuators and sensors,
to be controlled or observed by the fsm.

Introduction
Goal
The goal of this thesis is to establish a theoretical foundation of FSMD from
which a high level Hardware Description Language can be designed with clearly
deﬁned semantics and a syntax that reﬂects the theoretical model in a direct
manner. The language should be strongly typed, in order to increase possibili-
ties of veriﬁcation. Using F# the syntax of the language is to be captured by a
lexer and parser in order to produce an abstract syntax on which analysis can
be performed. The goal of analysis is both to verify static as well as dynamic
semantics of a concrete speciﬁcation. To this end a type checker shall be imple-
mented, which can simultaneously function as a formal deﬁnition of the static
semantics of the language. Similarly an interpreter is to be constructed veri-
fying dynamic semantics of a description, while in itself deﬁning the dynamic
semantics of the language.
Overview
The model of FSMD is formally deﬁned in chapter 2, and a GCD algorithm is
modelled with this formal deﬁnition. Based on the ﬁndings of chapter 2, the
language syntax is speciﬁed chapter 3 and the semantics are deﬁned in chapter
4, using F# as a meta language. Finally in chapter 5, the language is used in
practice to highlight limitations of the language in its current state, in addition
to some natural areas for future work to focus on.

Chapter 2
Introduction to FSMD
Finite State Machine with Datapath or FSMD, is a model that has been found
to be useful for eﬃcient design of custom hardware[Sch13]. It consists of two
components: The controller and the datapath.
The controller implements a
ﬁnite state machine whereas the datapath processes data, as dictated by the
controller.
An FSMD is deﬁned by a tuple:
< S, Cin, DPin, Cout, DPout, SS, CS, V, δc, δdp >
An illustration of the model can be seen on ﬁgure 2.1.
The elements of an FSMD are deﬁned as follows:
• S is the set of states in the controller.
• Cin and DPin are sets of inputs. Cin is the set of control input, and DPin
is the set of datapath input.
• Cout and DPout are sets of outputs. Cout is the set of control output, and
DPout is the set of datapath output.
• SS is the set of status signals, sent to the controller by the datapath.

Language Design
In this chapter, a language for modelling FSMD’s are presented. The language
introduces three important constructs. These are modules, datapath and con-
troller. The datapath and controller is of course used to express their theoretical
counterpart of the same name. A module is the keyword used for an FSMD,
that is a construct containing a datapath and controller. Beyond these con-
structs, other concepts such as types, statements and expressions need also be
deﬁned.
Some semantic properties are informally introduced, to provide in-
tuition about the language. The formal semantics are however deﬁned in the
following chapter.
On ﬁgure 3.1, is presented a program in the language, that ﬁnds the largest
number of a sequence given to the module as input. It should serve as a prac-
tical example while reading the formal description of the syntax in this chapter
and will hopefully be understandable after reading the chapter to end. To help
readability, keywords has been colored blue, action names are colored orange,
and states are colored teal. The same program is illustrated visually on ﬁg-
ure 3.2, however in an attempt to avoid clutter, the ﬁgure omits details about
control- and status signals.

3.1
Types and declarations
Types in the language speciﬁes whether a variable is a register variable, in-
put/output variable or a status signal.
The type is speciﬁed by one of the
following keywords: reg, in, out and status. Furthermore, a declaration spec-
iﬁes the primitive type associated with an identiﬁer, which can either be integer
(int) or boolean(bool).
A declaration has the following form
identifier : primType
type
As an example a status signal of type bool is declared in the following manner:
newLargestFound: bool
status
Note that there is no need to distinguish controller input/output and datapath
input/output, since this can be determined based on where the declaration is
found as seen in section 3.5. Variables of the same name may or may not give
rise to problems, as will be seen in the next sections.
3.2
Assignment Statements and Expressions
An assignment statement binds the value of an expression to a variable. This
has the syntax
identifier
:= expr
Expressions can be classiﬁed by their primary type, yielding two kinds of ex-
pressions, arithmetical and boolean. Arithmetical expressions are formed by
applying integer operators such as -, + or *, to integer constants or integer
variables. Boolean expressions are formed by applying integer values or boolean
values to boolean operators, such as && (logical and operator) or integer opera-
tors such as >, <, >= or <=.
The following line of code assigns a boolean value to a variable called newLargestFound.
newLargestFound := dpIn > largest;

3.4
Controller
The ﬁnite state machine of an FSMD is expressed with the controller construct.
As such it deﬁnes the possible transitions of the state machine. The structure
of this construct is as follows:
controller {
transition1,
transition2,
...
transitionm
}
That is, a list of m transitions.
On listing 3.4 and excerpt of the controller from the FindLargest module is
found. The excerpt contains two transitions. The ﬁrst is taken if cIn && newLargestFound
is true, and the state of the controller is ReadIn. If this is fulﬁlled, two actions
are requested to be executed, namely ReadInput2 and SetNewLargest. The
other transition is taken if cIn and newLargestFound are both false, and the
current state is ReadIn. In this case, Return is executed by the datapath, and
the control output variable cOut is set to true.

3.5
Module
A module is the construct used for modelling an FSMD. As such, it consists of a
datapath, a controller and the input/output of these. FindLargest is an example
of a module and can be seen on 3.1. In general a module has the following form:

Chapter 4
Semantics
The semantics of the language are described using F# as a meta language. The
semantics deﬁne well-formedness of the constructs found in the language, in
addition to how well-formed constructs are to be evaluated, given input-stimuli.
The full source code can be found in appendix D
4.1
Environments
During run time and static analysis, an environment maintains the information
needed to execute or analyse a program. An environment maintained during
run time is referred to as a variable environment and contains the state of all
variables of the FSMD at a given time during interpretation. It is a map from
variable names, to Const, where Const has the following abstract type:
type Const =
| N of int
| B of bool
and thus has the signature Map<string, Const>. As some variables in a mod-
ule might legally have the same names, it is necessary to maintain two variable

necessarily the same. Taking the FindLargest module as an example:

Semantics of Expressions
The semantics of an expression is deﬁned recursively in terms of a function. The
abstract type of an expression is shown below:
type Exp =
| C of Const
| Access of Access (* Variable access *)

| BExp of Exp * BinOp * Exp (* Binary Operators *)

The function can then be declared with the following signature:
intpExp: Exp -> VarEnv -> Const
That is, a function that takes as input an expression and a context, and returns
a constant.
Well-formedness of expressions is deﬁned by a function tcExp with the following
signature:
tcExp: Exp * TypeEnv -> bool
4.2.1
Constant Expressions
Evaluation of a constant expression, exp is done trivially as so:
let intpExp (exp: Exp) ctx: Const =
match exp with
| C (B value) -> B value
| C (N value) -> N value
...
As one would expect, the ctx parameter does not impact the result of evaluating
constants.
Constant expressions are trivially well-formed, and given a constant expression
such as C (B true), tcExp simply returns true.

Unary Expressions

4.3
Semantics of Statements
Evaluation of a statement is deﬁned by a function with the following signature:
intpStm: Stm * VarEnv -> string * Const
As the only statement in the language is variable assignments, the function
returns the name of the assigned variable, along with the value:
let intpStm (stm: Stm, ctx: VarEnv) =
let (Ass (lval, exp)) = stm
let value = intpExp exp ctx
(lval, value)
In order for a statement to be well-formed, a number of conditions must be
fullﬁlled, which are captured by the function tcStm:
let tcStm (stm: Stm, decls: TypeEnv): bool =
let (Ass (lHand, e)) = stm
let lHandDeclared =

let (Dec (name, _, _)) = dec
name = lHand
) decls

false
else

of the datapath, all actions are computed using the input variable environment.

Macros
If the state space cannot be reduced, macros could be added to the syntax
to help implement repeated blocks of transitions.
This could have a syntax
resembling the listing below, much like loop constructs in other general purpose
languages, with the twist of using the keyword create to make it clear that the
macro creates transitions:
controller {
for i in [1,2,3] create {
s..i [a == b] (action..i)> s..(i+1)
}
}
With .. deﬁning string concatenation, this should then generate the following
controller:
controller {
s1 [a == b] (action1)> s2,
s2 [a == b] (action2)> s3,
s3 [a == b] (action3)> s4
}
The usage of this might however be limited as the guard expression deﬁned in
the macro might be hard to adapt to all the generated transitions. For example,
in order to create diﬀerent expressions, the variables would have to be named
a1, a2, a3 etc, to be able to refer to the them in the macro body. To increase
usability of this, arrays might be a useful tool to have at hand, and this could
be added to the language.

Chapter 6
Conclusion
In this thesis the Finite State Machine with Datapath model is formalized,
describing the behaviour of an FSMD in terms of a transition system that clearly
deﬁnes the relation between controller and datapath. In particular the controller
function δc : (S × Cin × SS) →(S × CS × Cout) depends on the status signal,
SS, previously output by the datapath function δdp : (CS × DPin × V A) →
(V A × SS × DPout). And δdp depends on the control signal CS output by the
controller.
Based on this model a high level language is designed to capture a description
of an FSMD. The language is strongly typed, with all variables being declared
with a type and a primitive type. The primitive type deﬁnes whether a variable
is boolean or integer, while the type deﬁnes whether a variable is a register,
status signal or input/output port.
Static semantics are deﬁned by a type checker written in F#, which can si-
multaneously be used to ensure wellformedness of a program, for example by
verifying that no input ports are assigned a value, and that all variables used
in statements are declared. Similarly, dynamic semantics are deﬁned by an F#
program, that can simulate stimulating an FSMD deﬁned in the language, with
input.
To explore the language, a theoretical model of an elevator is constructed and an
attempt to concretise the model using the implemented language is performed.
It is clear from this, that an eﬀort must be made to minimize the size of the
==PAGE:58==
46
Conclusion
controllers state space, in order for a specifcation to be feasible to implement.
Furthermore, this process gives rise to ideas for further work on the language.
This includes macros, a loop construct, that can help with generating multiple
transitions that only vary slightly in form. This feature can be implemented
without modifying the abstract syntax, exclusively by changing the lexer and
parser.
Furthermore the idea of module composition is introduced, to help modularize
a speciﬁcation. Implementation of this would require extension of both syntax
and semantics of the language. Further work in this area is likely to give rise to
investigating time aspects of the presented model. Currently the model assumes
all actions to be executable within one clock cycle. The beneﬁt of this is that
a speciﬁcation can maintain a high level of abstraction, but as the work on a
speciﬁcation progresses, it is probably desirable to have the option of including
such details.

Appendix A
Transition Systems
Transition systems are used to describe dynamic processes by deﬁning a set of
conﬁgurations, and a set of transitions to describe how conﬁgurations are related
to one another[MN04]. Formally, a transition systems S is a pair of the form
S = (C, T)
where C is a set of conﬁgurations and T ⊆C × C.
As an example, one can describe an elevator in terms of a transition system. Let
Se = (Ce, Te) denote this transition system. A simple model could have each
conﬁguration describe the ﬂoor that the elevator is currently on. If the elevator
can move between three ﬂoors then Ce = {1, 2, 3}. Assuming the elevator can
move freely between any of the three ﬂoors, the set of transitions T is deﬁned
as
Te = {(i, j)| i, j ∈C}
To describe the processes of a transition system, we deﬁne the set of sequences
S(S), of a transition system S = (C, T) in the following manner
1. The inﬁnite sequence c0, c1, ...cn, ... belongs to S(S) if
(a) c0 ∈C
==PAGE:60==
48
Transition Systems
(b) for all i ≥1: (ci−1, ci) ∈T
2. The ﬁnite sequence c0, c1, ...cn for n ≥0 belongs to S(S) if
(a) c0 ∈C
(b) for all 1 ≤i ≤n: (ci−1, ci) ∈T
As the modelled elevator transition system has no restrictions on the movement
between ﬂoors, S(Se) contains all sequences of conﬁgurations.
With this, we can deﬁne the processes generated by S, denoted P(S). P(S) is
the subset of S(S) containing
1. All inﬁnite sequences of S(S)
2. All ﬁnite sequences c0, c1, ...cn of S(S) for which it holds that there is no
c ∈C with (cn, n) ∈T.
Rule two speciﬁes that there is not transition leaving the last conﬁguration of
a ﬁnite sequence. This means that for the elevator example, P(Se) contains all
inﬁnite sequences of S(S), as we allow the elevator to move from any two ﬂoors
instantaneously, and the elevator never enters a ﬁnal ﬂoor.

Modelling an elevator
As can be seen, the current model of the datapath abstracts away the actual
actions of the elevator, and signals that any requested action was completed
instantly.
While modelling an abstract elevator like above is beneﬁcial for exploring the
controller part of the FSMD model, it is hard to see the actual purpose of the
datapath.
Furthermore, as the state space of the model consists of a triple
S = {BasicState × Floor × ReqQ} the size of the state space |S| grows unman-
ageable quickly as the number of ﬂoors and the size of request queue increases.
As such, it does not seem plausible to synthesize the above model.
