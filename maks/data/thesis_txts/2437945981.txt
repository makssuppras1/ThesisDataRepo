Abstract
This thesis aims at deﬁning a general method for constructing program ﬂow analyses
on concurrent programs modelled as disconnected program graphs. The analyses are
modelled on an example language based on guarded commands. The language has been
extended to include channels, local and global variables and arrays.
The ﬁrst part of the thesis describes the extended language and how program graphs
of this language are deﬁned. Afterwards, the general methods are introduced and then
extended for better use on integer analysis. Lastly, a security analysis shows how the
methods can be used in practice.
The correctness of some methods will be proved or argued to be correct. In addition,
some benchmarks are made to compare the runtime of the diﬀerent methods.
The thesis show that by using the concurrent inﬂuence of disconnected graphs, an
analysis can be run on multiple components without having the same state explosion
problems as methods where programs are modelled by interleaving models. It will also
be shown how the methods can be used to increase the precision of the concurrent
inﬂuence, by adding to the complexity.
The thesis will also explain how these methods are implemented in an analysis tool
written in F#.

Introduction
Concurrent programming is the practice of writing programs utilizing concurrency. Pro-
gram analysis is the practice of obtaining information about the program through ap-
proximations.
Flow analysis is a method within program analysis, where programs
are analysed over a graph describing the ﬂow of the program. The analysis bases the
result on how the ﬂow changes states between actions within the program. The ﬂow
of a concurrent program can be represented in many ways. One way is to construct
a ﬂow of all interleaving paths possible within execution of the concurrent processes.
Concurrent programs can however often have many interleavings. This is due to an
exponential increase for each new process. Instead, we can model concurrent programs
as separate ﬂow graphs, where each graph describes the ﬂow of a single process. This
thesis will examine program analysis for concurrent programs written in a simple pro-
gramming language based on Dijkstras guarded commands [6], and deﬁne a method for
constructing classical ﬂow analysis over disconnected program graphs.
This thesis will go through most of the background for the theory introduced, it will
however be assumed that the reader has basic knowledge in both concurrent program-
ming and program analysis similar to what can be gained by reading Principles of
program analysis [13] and Foundation of Multithreaded, Parallel, and Distributed Pro-
gramming [3].
In particular, it will be assumed that the reader knows about basic
ﬂow analyses such as reaching deﬁnition and sign analysis on ﬂow graphs. It will be
assumed the reader is familiar with basic set theoretical methods, such as lattice struc-
tures and ﬂow graphs. In concurrent programming, we will assume the reader to have
a basic idea of the diﬀerent terms typically used in concurrent programming, such as
concurrency, parallelism, race conditions and deadlocks. It will also be assumed that
the reader understands functional programming, preferably F# as can be obtained by
reading Functional Programming Using F# [7].
1.1
Concurrent programming
Originally introduced in the 1960s to run applications simultaneously with I/O devices,
concurrent programming has gone from mainly an operating system concern to being an
essential part of all larger modern software. A concurrent program is a program where
the software is split up in two or more processes. Each process is run either concurrently
by interleaving the actions or parallel where actions can be run simultaneously. Con-
currency is essential for applications with a graphical interface, where in its simplest
form, one process runs the graphical representation, and another runs the logical code,
although there are many other uses. Running programs concurrently incurs risks not

1.2
Properties of program analysis
Program ﬂow analysis is the practice of statically analysing programs based on the ﬂow
modelled as a graph. The general method is deﬁned by Kildall [10] and later proven by
Kam and Ullman [9]. The method was deﬁned using ﬂow graphs, where each node in
the graph is an action of the program, and the edges are the ﬂow between these actions.

The result of ﬂow analyses is used in compiler optimizations, where redundant behaviour
can be removed. Another use is in security, where analyses can be used to ﬁnd unwanted
behaviour in the program.
An example of a well known ﬂow analyses is reaching
deﬁnitions concerned with where each variable may have been deﬁned at any given
point in the program, Another is interval analysis concerned with which interval of
integer values a variable may have in any given point.
If all actions in a program are seen as atomic actions, each point in between two actions
can be seen as a program point. In all program points the program will be in a state of
the current execution sequence. We will call this state s. This state can be everything
ranging from the concrete memory of the program, expressions used at this point or

Analysis environment
Before deﬁning the methods, we will ﬁrst deﬁne an environment in which we will work.
Instead of an actual programming language, we will use a small example language
based on Dijkstras guarded commands language. The alternative would be to base the
analyses on some existing language, but using a modiﬁed custom language have some
advantages. Firstly, we can keep the syntax small and simple, only introducing what
we need to show the behaviour of concurrency. Surely more advanced features could
be added, but for the sake of showing how concurrency behaves, it would only increase
clutter. Secondly, keeping a simple example language, makes it easier to transfer over
to other languages.
The programs of the example language will be modelled as a program graph.
The
more classical approach being ﬂow graphs, but program graphs has some interesting
properties, especially when combined with languages such as guarded commands.
2.1
Guarded command language
Originally deﬁned by Dijkstra [6], guarded commands is a simple nondetermanistic
language based on the idea of guards. A guard is a small boolean expression before
a code block.
If the guard is satisﬁed, the code block can be executed, otherwise
another code block will be chosen. The language has the property that it can deﬁne
nondetermanistic behaviour, and naturally translates into program graphs. A guarded
command can be seen as a boolean expression protecting a section of the program
<guarded command> ::= <boolean expression> -> <code block>
A guard is used when there are multiple choices in a program. This occurs normally
when we have a branch in a program. In guarded commands, branches are either an
alternative or a repeating construct.
The alternative construct, often written as if
<command set> ﬁtakes a list of guarded commands, and nondeterministically chooses
one of them to execute. If no guard is satisﬁed, the program execution is undeﬁned.
In concurrent programs, we often assume the program to halt. Loops are constructed
with the repetitive construct, often written as do <command set> od. This construct
does the same as the alternative, but after execution of the code block, repeats the
construct. Unlike the alternative, the repeating construct breaks the loop when no
guard is satisﬁed.

Parallel process:
The biggest diﬀerence between the extended and original guarded
commands is the introduction of the par CP rap construct. This construct is used to
separate a list of processes. To simplify, we do not allow processes to be dynamically
created, instead, processes are created at the beginning of execution. A program (P)
in the extended guarded commands is either a single process, in which case the par
keyword is not needed, or it is a set of processes CP. The set is separated with square
brackets just like do and if. The semantics of par CP rap is that each of the processes
run concurrently with each other. It will be assumed that each actions inside CP are
atomic.
Declaration:
The original guarded commands declare variables when they are ﬁrst
used. In extended guarded commands, we are interested in programs where each process
can have both local and global variables. We will extend the guarded commands to
include var x, which declares a local variable within the process of the declaration.
To simplify the language, all declarations must be at the top of each process. A local
variable only exists in the memory of the process in which it was declared. If a variable
has not been declared at the start of a process, the variable is instead global, and is
declared at ﬁrst use, as normal for guarded commands.
Channel communication:
In concurrent programming, there are diﬀerent ways for
processes to communicate with each other. The most used ways of communication is

either by shared memory or signal passing. Shared memory is naturally implemented
in the extended guarded commands by using global variables. Signal passing however
give rise to two new commands; c!a (send) and c?x (recieve). In extended guarded
commands, send and receive operates by channels where data can be stored in a buﬀer.
Data is sent as an arithmetic expression and receive is applied to a variable.
The
semantics of the send and receive is asynchronous. When sending data on a channel,
the data will be placed in a buﬀer queue, and will be read in a ﬁrst in ﬁrst out manner.
Receive can also be used as a guard. A code block protected by a receive as its guard,
will only be taken when something is received on the channel.
Variables and simple actions:
A variable is either a single integer or an array of
integers. The integer on the index of an array is obtained by A[i] where i is the index.
it is also assumed that arrays are indexed from 0. A variable or index in an array can be
assigned with the assignment action x:=a where x is a variable and a is an arithmetic
expression. There are no maximal or minimal value for the variables. As mentioned,
all assignments are assumed to be atomic.
Lastly, the language has a simple skip action, which does nothing, but is there to create
guards with an empty body, typically for escaping if or do.
2.2
Program Graphs
Let ast be the abstract syntax tree of a program written in extended guarded commands.
The tree will consist of a number of variables, arrays and channel, which will be labelled
with V AR, ARR and CHAN respectively. The union of V AR, ARR and CHAN
should be such that it includes all variable data structures in the program. This also
means that variables, arrays and channels must have unique names, as they share the
same name space.
Deﬁnition 2.2. For any variable x ∈ast it holds that x ∈V AR. Similarly, for
any array a ∈ast it holds that a ∈ARR and for any channel c ∈ast it holds
that c ∈CHAN.
The graphs Kindall used, and have been the most used in program analysis are ﬂow
graphs. An axample of a ﬂow graph is the graph shown in ﬁgure 1.4. In ﬂow graphs,
each node is an action in the program and the edges describe the ﬂow between actions.
Another method depicting graphs, which has for example been deﬁned by Muchnic and
Jones [11], are program graphs. In program graphs, the actions form the edges of the
graph instead of nodes. The ﬂow of the program is then deﬁned by how edges are
connected by nodes. A node can be seen as a point within the program. It is possible
to transform any program graph into a ﬂow graph or vice versa.
Additionally, the
same speciﬁcations for ﬂow graphs deﬁned in [10] and [9] holds when analysing over a
program graph. One way of representing concurrent programs is to tranform them into
a graph over the interleavings of the process executions, as will be shown in chapter 3.
Other methods include dynamically create threads by extending the graphs with forks

and joins [14], much like how ﬂow graphs can be extended to include procedures [13].
In this thesis we will explore the option of creating program graphs, where each process
is described by its own disconnected graph like in formal methods, an Appetizer [12].
The result is a set of program graphs describing the program, compared to having a
single graph over the entire ﬂow.
In single threaded programs, the resulting program graph will always have just a single
start and end node, which can be deﬁned as the point right before and after execution of
the program respectively. When deﬁning the program graph over concurrent processes,
we will deﬁne the program graph with a set of start and end nodes where each start and
end node is deﬁned as the point right before and after execution of one speciﬁc process.
A program graph can be seen as a set of transitions between nodes. Each transition is
deﬁned as a start and end node, an action between the nodes and a process id. The
process id is describing which process the action belongs to. If α is the ﬁnite set of
actions in a program, and pid the set of process ids, then we can create a graph with Q
nodes between the actions α within the graph of pid. The initial nodes will be deﬁned
as a non empty set of nodes, but as we can have graphs without any end nodes, the set
of end nodes can be empty. Putting it all together, a program graph is deﬁned by the

3.1
Program ﬂow analysis
Using product graphs, we have a single connected graph. The advantage of this, is
that we are able to run all analyses as were it a single threaded program.

3.3
Disadvantages and alternatives
We have shown that it is possible to do an analysis on a product graph over the program
graphs of multiple processes. This is a simple way of getting a result using all the
classical deﬁnitions of program analysis. This method however, has some disadvantages.
The ﬁrst and most costly is that we have to produce a product graph. The construction
itself is not that costly, but it is prone to the state explosion problem, as the number
of nodes in the product graph is the product of nodes in each process. For example,
two processes with 5 nodes give 52 = 25 nodes, but three processes with 5 nodes gives
53 = 125. the resulting graph can be huge compared to the original, which leads to
slower run times. Some methods aim at reducing the number of states by using partial
order reductions [4], although there will naturally still be more states than in the original
graphs.
The analysis over the product graph gives the result of every interleaving state, and
not of the individual states in the graph. We could extract this result, and the worst
disadvantage of the product graph is simply the memory consumption and run time.
In this thesis, we will suggest another solution, where graphs are disconnected, instead of
interleaving models or connected graphs. Using disconnected graphs, keeps the models
very close to how concurrent models are described in literature about concurrency like
in [3], and ensures that there is no state explosion, as the model is unchanged.

4.1
Concurrent inﬂuence, C
There are a few diﬀerent options when it comes to change an analysis to a new feature.
We can change the instance of the framework, the transfer functions or the algorithm
creating the result. In this thesis we will extend the framework, and afterwards the
worklist algorithm to match the new framework.
When two processes P1 and P2 are running and P1 is in any state in its execution, P2
can potentially be in any state. As we argued, the inﬂuence of P2 on P1 is potentially
everything that is updated in p2. We will call this the concurrent inﬂuence of P2 and is
deﬁned as a mapping between the pid of P2 mapped to an inﬂuence assignment. The
concurrent inﬂuence assignment can be anything, but will most likely at least consist of
the analysis assignment of the process. In the case of reaching deﬁnition, the concurrent
inﬂuence assignment is the upper bound of all states in P2. We can optimize this slightly
by also removing all local variables of P2, as these are not interesting in P1.

the the state of inﬂuence of that process id.

process. This behaviour can be described as a function returning the change on the
inﬂuence of an action on a given process. We will call this function add. For bit vector
frameworks, where mutual exclusion is not a factor, we can add the concurrent inﬂuence
to a state from other processes without adding any ﬁltering. Later we will show that
some analyses might gain some precision by ﬁltering on the concurrent inﬂuence. How

Correctness of the bit-vector framework
The analysis over a monotone framework for concurrency is shown to be correct if all the
properties of the monotone framework is satisﬁed in addition to the function add adding
all information of the inﬂuence of an action α, and if get extracts all inﬂuence from all
other processes. In case of the bit-vector framework, we add everything gained from gen.
This is clearly everything that an action can inﬂuence other processes. When extracting
with get, we extract all the inﬂuence from other processes, as this is the case, we are
sure to get everything, and do not ﬁlter anything, therefore the get method satisﬁes
the condition too, and the reaching deﬁnition analysis over the monotone framework
for concurrency is correct.

5.3
Advantages and disadvantages
With the path constraint analysis, we can remove variables from the analysis which are
mutually excluded. This naturally increases the precision of the analysis, and can help
identify problems common in concurrent problems. In this case violation of critical
zones. The method is modelled such that minimal changes have to be made to the
original analysis, such that the only diﬀerence between the instance of the monotone
framework and the instance of monotone framework for concurrency is the concurrent
inﬂuence based on a path constraint analysis. Of course, we do sacriﬁce precision for
both speed and memory consumption. The analysis will need to do the entire constraint
analysis before running the actual analysis. As an instance of the monotone framework
is always faster than an instance of the monotone framework for concurrency over the
same analysis, and we know that the constraint analysis is fast, the worst case execution
time is doubled - but often less. Additionally the analysis needs to maintain the path
constraint map χ. Worst case, this map can get slightly large, as the ⊤value of this is
the number of variables times the length of the power set of all basic boolean expressions
in the program. Implementing the map as a map with reusable values gets it down to
variables + basic boolean expressions.
5.4
Correctness of path constraints
We will for now just argue for the correctness of the path constraint analysis. First,
the path constraint analysis will have to be correct, as the analysis is deﬁned on the
monotone framework, the analysis is correct if it satisﬁes all the properties of the frame-
work. The lattice is indeed a complete lattice, as it is the cartesian product between the
lattice of the integer analysis, which must be a complete lattice, and the constraints.
The constraints is a complete lattice, as it is the power set of boolean expressions in the
program, it also satisﬁes the ascending chain condition, as there is a speciﬁc amount of
expressions in the program, which gives the lattice ﬁnite height. As argued earlier, the
transfer function adds a constant value to the nodes. As the set always increases the
transfer function it can be said to be monotone, and the path constraint is an instance
of the monotone framework.
By adding path constraints to the concurrent inﬂuence functions, we ﬁlter oﬀsome of
the data gained by the basic concurrent inﬂuence functions. We will now argue that
the new concurrent inﬂuence functions do not remove anything that should be in the
result.
add still adds the same amount of information as the basic function, but additionally
adds the constraints χ.
As the information added to C is a super set of the basic
function, the new add does not remove anything it should not.
get does ﬁlter oﬀsome parts of C. It will only accept values where the contraint is
satisﬁed in the abstract memory of its analysis assignment. As the abstract memory
of this analysis assignment will be a super set of the analysis assignment for the cor-
responding state in the interleaving model, we can conclude that the value ﬁltered oﬀ

6.1
Policies on programs
The correctness of a concurrent program is often proven based on the notion of invari-
ance. An invariance is a logical rule within the program, that is known to be true in
all states of the program. Usually, an invariance is proven either mathematically, by
==PAGE:56==
6.1. Policies on programs
47
induction or using model checkers like SPIN [8] or more graphical tools like Uppaal [2].
When proving the invariance of an application, we assume the invariance to be true at
the beginning of the program, and show that no matter which transition is taken, the
invariance still holds. An invariance can be based on diﬀerent systems. Some based on
simple expressions, propositional logic, temporal logic or computational tree logic. We
will deﬁne policies of a program as a form for invariance. A policy will be deﬁned as
a set of rules, the rules will be deﬁned over a single variable, and a condition over this
variable.
Let a condition Con be the set of some conditions that can be applied to a program
on a variable. A condition should be resolved as either true or false for any variable
in the program. For example > n, = n or parities like Even and Odd can be true or
false for any given variable. The set of available conditions can vary depending on what
analysis is conducted. Let a rule r (deﬁnition 6.1) be a condition c ∈Con bound to
some speciﬁc variable v ∈V ar. A rule can be modelled as a tuple, where the ﬁrst value
is a variable in the program, and the second is the restriction on that variable.
Deﬁnition 6.1. A rule r is a the relation between a variable in V AR and a
condition in Con. A rule is described by a tuple

Detection of signs
We will let the complete lattice of detection of signs be P(V AR × SIGN)
where sign is either +, −or 0. The signs are added as types in the tool to make them
easier to recognize. The least upper bound operator for detection of signs is the union,
and ⊥is the empty set.
Parity analysis
Parity analysis is very similar to the detection of signs analysis in
that it has a complete lattice consisting of variables pointing to either the parity odd
or even. The complete lattice is P(V AR × PARITY ), and the ⊥is the empty set.
Detection of interval
A new type is deﬁned for the interval analysis to simplify
operations on intervals. An interval in the analysis tool is either a tuple of integers,
undeﬁned or empty.
There are two deﬁned inﬁx operators on intervals.
+ deﬁnes
the union of two intervals. Given two intervals (min1, max1) and (min2, max2), the
union will be deﬁned as (max(max1, max2), min(min1, min2)). −deﬁnes the intersec-
tion of the intervals, which returns undeﬁned when the two intervals are disjoint and
returns (min(max1, max2), max(min1, min2)) otherwise.
undeﬁned simply indicates
that the interval has reached an undeﬁned state, empty on the other hands indicates an
empty interval, closely representing what null represents in programs. The complete
lattice of the interval analysis used in the tool is the powerset of variables and intervals
P(V AR × INTERV AL). The bottom of an interval is the empty set.

Conclusion
Analysis methods for concurrent programs modelled as disconnected program graphs
have been constructed at diﬀerent levels of precision. The ﬁrst model argues for the
use of a product graph of the disconnected graphs, giving a precise result in partial
states. The setback of this method is that it needs to run on a new graph with far
more states and transitions than the disconnected graphs. Therefore, concurrent in-
ﬂuences were introduced, and an extended version of the monotone framework called
a monotone framework for concurrency was introduced. This method, now running
on disconnected graphs, add the inﬂuence of other processes to each state in a graph
based on a analysis speciﬁc function. This new version being far faster than the prod-
uct graph gave precise results for analyses in the bit vector framework, and with the
extension of a pre-executed path constraint analysis, integer analysis could give a better
approximation where mutual excluded variables were ﬁltered of.
Lastly, policies on analyses were introduced to greater increase the precision by giving
a set of rules that must be satisﬁed at the beginning of the program and in all states.
The result of this method also gives a guess on the satisﬁability and can be used to
test some properties without having to rely on computation heavy model checkers or
mathematical proofs.
A tool for trying out the diﬀerent methods described were implemented in F# using a
small test language based on Dijkstras guarded commands. The language was purposely
kept small in order to keep the focus on the properties of concurrency.
The proposed method using concurrent inﬂuence adds a new level of complexity to the
model, and will naturally run slower per transition then it would on ﬂow analyses on
singe threaded programs. The method does however provide a much faster version than
the one running on product graphs, and with the addition of path constraints, loses
very little in terms of precision. Path constraints is the most time consuming addition,
as we need to add a whole new analysis on top of the concurrent one. This can in worst
case double the time of execution.
The method is kept rather simple, trying not to change the existing analyses and only
extend them. The eﬀect is that it should be possible to easily extend the language
without too many changes to the methods, or extend other instances of monotone
frameworks over other languages with the monotone framework for concurrency.
With some testing, it has been shown that the proposed method within this thesis gives a
general way of implementing program ﬂow analyses on disconnected ﬂow graphs. Other
methods, such as interleaving models with partial order reduction are alternatives to
this model, but have not been compared. What have been compared, is the complete

removes state explosion that comes from the product of multiple components.
