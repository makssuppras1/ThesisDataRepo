Introduction
Since the advent of computers in the modern sense, we have been able to calcu-
late expressions that would previously have taken pages to reduce in an instant.
We can input an expression into the computer, and it will output the final re-
sult with generally little to no problem. However, if something were to happen
inside the computer (in a most physical sense) while it is processing an input,
the output may be off from the expected result, or in worse cases be complete
nonsense.
This was already noticed in 1947 by Hamming when a bit had been flipped in
his program overnight. He dedicated a significant part of his studies to come up
with the first error correcting code, that is a code where even if a bit flipped,
the computer would still know to interpret the data properly. This was done by
encoding each sequence of bits to a slightly longer sequence of bits that could
carry enough additional information. Specifically, if a bit was lost, we would be
able to reconstruct the encoded sequence of bits with the remaining bits, and
thus decode back to the original bit sequence.
Since then, this kind of encodings came to be called error correcting codes, and
have been innovated upon to allow correction of more than one error. Examples
of such are Reed-Solomon codes and Goppa codes named after those who first
constructed them. Where Hamming had thought of it in terms of bits and matrix
operations, Reed-Solomon codes introduced the idea of encoding sequences of
bits as polynomial evaluations. And Goppa codes further built upon the idea
by expanding the evaluations to be at the points of an algebraic curve.
Both came to meet or break new bounds for how “good” an error correcting
code could be, but in the pursuit of correcting as many errors as possible they
came with a drawback. Recovering an encoded message took up to all available
==PAGE:6==
iv

data points, even if it were to only correct a single lost or changed bit.
As computers and the ways of storing data have evolved, so have our needs
in terms of error correction. A prominent example of modern data storage is
cloud storage, one of the primary ways of storing data with products such as
Amazon’s Dynamo, Microsoft Azure, and Google’s Bigtable, which specifically
use distributed storage (multiple databases to retrieve the data faster). Then,
if the data one seeks to access is in an inaccessible database at the time, we can
consider this the erasure of a particular data node.
This thesis examines a relatively new way of looking at error correcting codes
presented in [TB14] and [BTV15]. These codes are designed to retrieve a single
instance of erased data (inaccessible data), while reducing the amount of data
needed to perform the recovery.

Chapter 1
Block Codes
To build a grounds-up understanding of Locally recoverable codes (LRC code),
we need to cover the basics of encoding data.
Specifically the chapter will
cover going from a message of raw data (a tuple of symbols) to an encoded
codeword, and the properties of the latter. We begin by reviewing the concepts
as introduced in [JH04] and [JP] Before moving on, we need to choose an alphabet
for our data.
Definition 1.1. The alphabet is the finite field Fq of definition of messages
and codewords, where q = pn is a prime power. Given the structure of a field,
we can use its related operations.
Example 1.1. {0,1} is a natural alphabet for considering a computer’s op-
erations, and naturally inherits the structure of Z/2Z. This again translates
directly to a computer’s XOR operation.
1.1
Encoding a message
The messages are encoded in a space of higher dimension than the message
itself in an effort to add more information to the message.
Using the extra

Remark. Orthogonality and linear independence
Vector spaces over finite fields are not inner product space despite letting us
define the inner product. This is a natural consequence of finite fields not
being totally ordered: there is thus no way to define positive-definiteness.
To build on the notion of linear operations on vectors, it is natural to represent
encoding as a matrix operation. The natural coordinate basis over a field further
motivates the following general construction.
Definition 1.4. A generator matrix G of an (n, k) code C is a k ˆ n matrix
whose rows are linearly independent codewords.
Encoding with such a matrix is defined from the left. In other words let m be
a message of length k over Fq, then the encoded codeword c by the (n, k) code
C with generator matrix G is given as
c = mG
From the definition of a generator matrix, any k linearly independent codewords
can be used, but likewise it is also clear another set of k linearly independent
codewords will be row space equivalent. Since they span the same space, it
is worth considering if any particular generator matrices would have special
properties.
We call a generator matrix systematic if G P Fkˆn
q
is given as
G = [Ik | A], where Ik is the k ˆ k identity matrix. The obvious benefit of this
matrix is that the message is directly preserved as the k first coordinates in any
codeword.

1.1.2.1
Parity checks
To detail the n ´ k dimensions we don’t use with a code, it is useful to think
of them in terms of the orthogonal complement of the codewords. Specifically
a vector h of length n satisfying
GhT = 0
is called a parity check, and clearly belongs to the orthogonal complement of
G’s row space.
Definition 1.5. A parity check matrix H for an (n, k) code C is an (n−k)ˆn
matrix whose rows are linearly independent parity checks.
A parity check matrix is defined similarly to how a generator matrix is defined.
In fact this is because parity checks matrices are also codes.
Definition 1.6. The code spanned by rows of a parity check matrix H of a
code C is called CK, the dual code. Namely,
CK = tx P Fn

Definition 1.7. The rate of an (n, k) code is defined R := k
n, thus the rate

1.1.2.2
Hamming weight and minimum distance
For error recovery, it is particularly important to have the codewords be as
distinct from one another as possible. But how does one measure how distinct
codewords are?
Definition 1.12. The Hamming weight of a vector x P Fn
q , denoted wH(x),
is its number of nonzero coordinates.
Definition 1.13. Let x, y be vectors in Fn
q , then we define their Hamming
distance as dH(x, y) = wH(x ´ y).
We note the Hamming distance is a distance, a metric that is.
It respects
positivity, symmetry, and the triangle inequality.
Thus our codewords now
exist in a metric space.
Definition 1.14. The minimum distance of a code C is the minimum Ham-
ming distance between any two distinct codewords.
It is commonplace to
denote C as an [n, k, d] code over Fq.
Beyond having the metric, error correction specifically relies on a largest mini-
mum positive distance, such that one can assume all codewords are at least that
distinct from one another.
Lemma 1.15. In an (n, k) code, the minimum distance is equal to the mini-
mum weight of a nonzero codeword.
Proof. Let c be a codeword of minimum weight (wH(c) = wmin). Then we
know the distance can at least get as small as dH(c, 0), which lets us conclude
dmin ď wmin.
Now, let c1 and c2 be codewords of minimum distance (dH(c1, c2) = dmin).
Then we know dH(c1, c2) = wH(c1 ´ c2), where c1 ´ c2 is also a codeword.

We have thus found at least one codeword of weight equal to the minimum
distance and we can conclude wmin ď wH(c1 ´ c2) = dmin.
We can combine the two inequalities to get the desired result.
Furthermore, there is also a direct link between parity check matrices and the
minimum distance.
Lemma 1.16. Let C be an (n, k) code and H a parity check matrix. Then, if
j columns of H are linearly independent, C contains a codeword with nonzero
elements in some of the corresponding positions. If C contains a codeword of
weight j then there exists j linearly dependent columns of H.
Proof.
Since HcT = 0 holds for a nonzero c P C and H has j linearly
dependent columns, we know the only way to get the 0 vector is a linear
combination of at most those j columns.
Conversely, if we know there is a codeword of weight j, we know it is equivalent
to a linear combination of exactly j columns of H equal to 0.
Lemma 1.17. Let C be an (n, k) code with parity check matrix H.
The
minimum distance of C equals the minimum number of linearly dependent
columns of H.
Proof. By Lemma 1.15 we know the minimum distance is the same as the
minimum weight of nonzero codewords in C. Thus by Lemma 1.16 there are
at least as many linearly dependent columns in H. Additionally, by the same
lemma, if there were a set of fewer linearly dependent columns in H, there
would exist a codeword in C with at most that weight. Thus we can conclude
that the minimum number of linearly dependent columns in H is equal to the
minimum weight in C.
Finally, most importantly for this section and chapter, we need to introduce
the singleton bound. While the generalised version of the theorem covers codes
regardless of cardinality of the codes, we restrict our attention to the linear case.
Theorem 1.18. For any [n, k, d] linear code, it holds
d ď n ´ k + 1.

Chapter 3
LRC Codes
We have seen how encoding a message as a polynomial to be evaluated in n places
generates a code where we can recover the full codeword (and subsequently
message) when there were fewer errors than half the minimum distance. These
computations can be resource intensive relative to how many errors need to be
corrected. We note that for a (n, k) RS codeword a single erased symbol can be
recovered using k other symbols we can use to interpolate the (k ´ 1)-th degree
encoding polynomial and evaluate in the missing coordinate. If the codeword
instead contains duplicates, that is the code is a repetition code, recovering a
single erasure is a matter of looking up duplicate of the missing value. We now
seek to define the number of symbols we need to recover any single erasure and
understand theoretically what it takes for an error correcting code to correct it
while minimising the number of computations.
We call an (n, k, r) locally recoverable code (LRC code), a code that maps an
information vector of k symbols to an n-symbol codeword, such that any of the n
symbols can be recovered by using at most r other symbols from the codeword.
Definition 3.1. Given an (n, k, r) LRC code, any coordinate i can be re-
trieved as a function of at most r other coordinates. We call the set of those
coordinates Ii Ă [n]zi, the recovering set of i.
Formally an LRC code is as defined in the following a (n, k) with locality r,

3.4
Modifying LRC codes, LRC(t) codes
So far, we have only considered how to recover a single erasure, but what hap-
pens if there are two instead? If the second erasure is in the same recovery set
as the first, our previous constructions would simply fail, we can’t interpolate a
polynomial of high enough degree with the remaining symbols.
Therefore we seek a construction that for any location has more than one re-
covery set- in case another erasure would invalidate the recovery proces.
Definition 3.20. Let F be a finite field. A code C Ă Fn is locally recoverable
with t recovering sets (an LRC(t) code) if for every i P [n] there exists disjoint
subsets Ai,j Ă [n]zi for j = 1, . . . , t of size r1, . . . , rt respectively. Thus the
value of a symbol xi is a function of each subset txl P Ai,ju for j = 1, . . . , t.
An LRC(t) code of dimension k, length n, and t disjoint subsets of size ri,
i = 1, . . . , t is denoted as an (n, k, tr1, . . . , rtu) LRC code.
We note the disjointness is part of the definition as otherwise we would not
avoid the problems too many erasures would cause.

Algebraic Curves
We have seen how we can construct LRC codes inspired by error correcting
codes using polynomial evaluations in select locations. But just like RS codes
are not the only ways of correcting errors, we will once again borrow some ideas-
this time, using algebraic curves. As for their error correcting counterparts, the
benefit of using algebraic curves stems from the fact the number of points we
can operate on is much larger than the alphabet itself.
However, while there are ways of simplifying the mechanisms of algebraic curves
for error correction, we will this time need to employ more generalised notions
from algebraic geometry. Notably, we will need to briefly visit the aﬀine and
projective spaces, identify places and points, and define the Riemann-Roch space
associated to a divisor.
Algebraic curves are closely related to function fields, in fact one can say that
function fields are the algebraic description of algebraic curves. This allows us
to employ methods from both algebra and geometry, and the switching back and
forth between curves and function fields. This comes at the cost of a potentially
confusion notation, so the following is an attempt to clarify the terminology
ahead of time.

Chapter 6
Conclusion
In this thesis we have shown how error correcting codes may be further charac-
terised by their locality 1 ď r ď k, for messages of length k. We have demon-
strated two general ways to construct error correcting codes built for the express
purpose of reducing the locality.
The first kind was built as a direct extension of Reed-Solomon codes, inheriting
some properties regarding their rate and minimum distance. Most notably, there
exists an equivalent to the Singleton bound for LRC codes which is met by these
codes. We have shown how the codes can be generalised in terms of arbitrary
maps. Concretely, we have shown how the parameters can be chosen arbitrarily
at a small cost in the minimum distance.
With other alterations, we have considered how the codes could be expanded to
always be able to correct more than one erasure, also at the price of a smaller
minimum distance.
Further work with such LRC can be directly related to finding ways of increasing
the minimum number of possible errors corrected, or more tangentially related,
how we can find good polynomials.
In the latter case, significant advances
have been achieved using Galois theory and function fields such as [CM22] and
[GY24].
The other family of LRC codes constructed translated the work in [BTV15] to
==PAGE:100==
function fields in an effort to illustrate the principles behind them. Though these
codes, which are built on algebraic curves are not optimal LRC codes, much like
the Goppa codes have done before, they provide more evaluation points without
increasing the alphabet size linearly.
Furthermore, an explicit family of LRC codes on algebraic curves was provided,
in addition to alteration of the general construction, thus allowing for more
control over both the locality and the number of recovery sets. Using the natural
language of field extensions allowed for the further construction of such codes
by considering towers of function fields.
The idea of studying LRC codes on function fields directly is not new.
In
particular, the papers [JMX20] and [CGM24] use this approach to directly study
LRC codes, thus providing an avenue for further work on this kind of code.
For a closing remark on this family of codes, future work could consider the
apllication of the codes on other well known families of curves such as elliptic
curves, Giulietti-Korchmaros curves [GK09] and their generalisations [BM18;
GGS10], or Suzuki curves [Suz60]. For the purposes of greater error correction,
curves whose automorphism groups have many subgroups (as seen in [BMQ20])
are of particular interest so as to allow more recovery sets.

Appendix B
Algebraic geometry
While the codes introduced in chapter 5 are particularly well suited for study
through the eyes of function fields and towers of such, they were first introduced
as operating on curves in [BTV15]. In parts, this can be attributed to the fact
evaluating a polynomial in a point with coordinates is arguably more intuitive
than evaluating a function in a place.
This appendix is therefore dedicated to geometry behind the function fields, by
highlighting some key results presented in [Har77]. The appendix will not go in
detail with all the prerequisites for this, as a full study of the Zariski topology
and rational maps fell outside the scope of the thesis.
The primary result from Hartshorne that invites the study of curves through
function fields is
Theorem B.1. For any two varieties X an Y over a field k, there exists a
bijection between
• the set of dominant rational maps from X to Y , and
• the set of (k) algebra homomorphisms from K(Y ) to K(X).
Where K(X) and K(Y ) are the rational function fields of respectively X and
Y over k.
==PAGE:105==
Algebraic geometry
Then in particular, when X is non-singular,
Proposition B.2. Let X be a projective smooth curve over a field ¯K, let Y
be any curve over K, and let f : X Ñ Y be a rational map defined over all
points in X. Then either:
1. f(X) is constant,
2. f(X) is surjective.
In the latter case, K(X) = Frac(K[X1, . . . , Xn]/xIXy) is a finite extension
field of K(Y ), f is a finite morphism (of the same degree as the extension),
and Y is also projective.
The last result makes intuitively sense, for f surjective, then it would naturally
hold g(f(P)) = 0 for all P if and only if g P ker(K(Y )), thus naturally defining
the embedding K(Y ) ãÑ K(X).
To complete the study of the LRC codes
by purely using algebraic geometry, one would however need to replicate the
results of [PST98] to characterise the Riemann-Roch space of the unique totally
ramified point projecting to the point at infinity on the line.
