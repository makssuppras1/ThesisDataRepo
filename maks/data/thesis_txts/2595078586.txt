Abstract
While the amount of data keeps increasing, CPU clock speeds have stagnated over the last decade. Yet,
a number of features on modern CPUs are under-utilised. Single instruction, multiple data (SIMD) is
one such feature, in which the CPU operates on vectors of values instead of individual elements. One
area that has not yet seen much research on the possibilities of SIMD is the indexing problem, which
consists of maintaining a dynamic set of keys while supporting updates, predecessor- and range queries.
In this report we introduce several new SIMD enhanced data structures, as well as a simpliﬁed interface
to express SIMD computations and a model to analyse programs utilising them. We introduce what we
believe to be the ﬁrst data structure that uses SIMD to directly improve insertions. The data structures
are optimised for response times, and several of the new data structures are more than a factor two
better than their un-enhanced counterparts for both queries and updates. The best data structure is a
Log-Structured Merge-Tree (LSM) which uses SIMD to merge levels together. It is more than 10 times
faster for insertions than a SIMD enhanced standard B+-Tree, while being slightly faster for searches.
As a performance baseline for searches the most eﬃcient, static, cache optimised, SIMD implementation
of a search tree that we could devise is used. The LSM stays within a factor four of this data structure.

Introduction
In recent years, hardware designers have been adding new powerful features to processors. One of these
features is single instruction, multiple data (SIMD). SIMD is a form of vector processing paradigm in
which a single instruction can be applied simultaneously to multiple data elements. While introduced
in the 60s [9], it has recently become commonplace in modern processors. For instance, Intel’s newest
CPUs ship with AVX512 which provides SIMD operations on words that are 512 bits wide [17].
While the beneﬁt of doing more work per unit of time is obvious, SIMD poses certain challenges. To
fully beneﬁt from SIMD, algorithms must be restructured such that it is useful apply the same operation
to multiple data at the same time. More importantly, data for SIMD registers must be aligned properly
and located consecutively in memory to avoid penalties when loading data into a SIMD register.
Due to these constraints, it is not always obvious how to improve the many algorithms and data structures
that were developed for the standard computational paradigm. Despite this, SIMD has been used to
great eﬀect in areas such as compression [19], set intersection [11], sorting [10, 12] and searching [18].
These papers indicate that there is research potential in exporting well-known ideas into the domain of
SIMD to achieve signiﬁcant speedups.
In this report we will tackle the indexing problem which consists of maintaining a dynamic set of keys
while supporting predecessor and range queries.
We introduce a Simpliﬁed SIMD Interface along with a model of computation to more easily communicate
and analyse SIMD based data structures and algorithms.
Using these tools we introduce new SIMD-enhanced variants of well-known data structures that perform
signiﬁcantly better than their un-enhanced counterparts.
To our knowledge we introduce the ﬁrst data structure which uses SIMD to directly improve insertions
and show that our best structure is a factor of 3 and 20 better at searching and insertion respectively, than
a balanced binary search tree, while remaining within a factor four of the best static search procedure
we could devise.
The layout of the report is as follows. Section 2.1 introduces basic notions about SIMD and how to
analyse SIMD programs theoretically. Section 4 introduces related work on data structures that use
SIMD to solve the indexing problem. A high level introduction of our own data structures can be found
in Section 5, while Section 6 covers all of our data structures in more depth. A simpliﬁed SIMD interface
is introduced in Section 6.1. The big picture of the data structures in this report can be found in Section
6.2. It contains a helpful summary over all the data structures in Section 6, including their running
times and main characteristics. The experiments and results are introduced and analysed in Section 7.
Section 8 contains further discussions as well as future work.
2
Preliminaries
Throughout this report, N denotes the number of elements in a data structure. It is not to be confused
with the lowercase n, which will be used on an ad-hoc basis to analyse operations that do not directly
relate to a data structure. Given two sequences (arrays, SIMD vectors, etc) x and y, x ◦y is the their
concatenation. It is assumed that the largest signed 32 bit integer never will be used as a key. Instead,
it is used as a sentinel in several data structures and is commonly referred to as ∞.
2.1
SIMD
SIMD (single instruction, multiple data) is a computation paradigm where operations are performed
on vectors that contain a given number of elements each. Algorithms utilising SIMD are said to be
vectorised, as opposed to unvectorised algorithms which are scalar.
For this project, Intel’s Advanced Vector Extensions 2 (AVX2) which supports 256 bit vectors were
used. Many operations are supported, including variants of most of the normal arithmetic and boolean
operations, an example of which can be seen in Figure 1. More powerful operations include reordering

While convenient for implementations, this makes describing and reasoning about vectorised algorithms
diﬃcult. To remedy this, programs using SIMD will be analysed using an idealised1 extension of the
Word RAM model where SIMD words contain B elements and all SIMD operations run in O (1) time.
An important implication of this is that data can be moved in chunks of Θ(B) elements in constant time.
Initially, the focus in this report will be on the core ideas of the data structures. The speciﬁcation of
SIMD operations is deferred for Section 6.1.
The most important design principle for SIMD algorithms is that data that is moved from memory to
a SIMD register or vice versa, should be located consecutively in memory. Otherwise it is necessary to
read or write information from or to diﬀerent locations in memory. These operations are called gather
and scatter, respectively, and are prohibitively expensive. In some cases they can entirely negate the
increased computation power from using SIMD. Accessing data that is not properly aligned in memory
also incurs a penalty. In most cases this is easy to avoid, however, and alignment issues are disregarded
as implementation details in this report.
3
Problem Statement
In this report, we will study how to apply SIMD to the fundamental and well-studied data structuring
problem of indexing. In particular, we will focus on data structures that maintain a set while supporting
incremental updates and predecessor queries. Note that the predecessor queries are all implemented such
that extending them to range queries is trivial and does not induce signiﬁcant overhead. The goal is to
explore diﬀerent trade-oﬀs which minimise the worst case running time for the diﬀerent operations.
Due to restrictions on the available SIMD hardware, we will restrict ourselves to working with 32 bit
integer keys. Furthermore, we will restrict ourselves to a single thread, to using only internal mem-
ory (RAM), and to the setting where insertions and queries can occur in any order. In other words,
optimisations that rely on concurrency or on bulking updates or queries are excluded.
The data structures will be compared using empirical results as well as theoretical analyses using the
model of computation presented in Section 2.1. The idea is to capture how the performance of these
data structures will change as hardware continues to improve.

Previous Work
Indexing is a well researched problem.
Classic in-memory data structures for the indexing problem
include balanced binary search trees and van Embde Boas trees [8]. Solutions for the external memory
version of the problem include B-Trees [3],B+-Trees [3], Bϵ-Trees [5], Log-Structured Merge-Trees (LSM)
[15], and the Cache-Oblivious Lookahead-Array (COLA) [4]. In particular B-Trees and its derivatives
have seen much use both in practise and in research.

The literature on using SIMD for indexing is sparser. Table 1 shows an overview over following paragraphs
on the previous work.
Schlegel et al. showed how to use SIMD to implement B-ary search 2 in a static collection of keys. B-ary
search is a generalisation of binary search, where instead of recursively eliminating half of the remaining
elements you divide the remaining elements into B + 1 equally sized groups and recurse only on one.
This leads to a lookup time of O (logB N). Note that the technique requires a static data structure and
that updates run in linear time [18].
Huber et al. implemented an in-memory B-Tree where the keys in the B-Tree nodes are searched using
B-ary search introduced by Schlegel et al. [23]. They apply the same idea to a trie (technically a preﬁx
B-tree) where the keys stored in a trie node are searched using B-ary search. Note that trie nodes contain
only segments of keys rather than the entire keys themselves, and that more segments than keys can be
ﬁt into a single SIMD register. This speeds up the searching process. For instance, if the segments are
a quarter of the length of the actual keys, each SIMD segment can ﬁt 4B segments, leading to the more
eﬃcient 4B-ary search in each trie node [23]. Since the B-ary search relies on a static data structure,
both of the data structures suggested by Huber et al. are static in the sense that at least one entire
B-Tree or trie node needs to be rebuilt on any update.
VAST (A Vector-Advanced and Compressed Structure for Massive Data Tree Traversal) by Yamamuro
et al. is another eﬃcient vectorised indexing data structure. The authors apply cache-conscious design
on the level of a page block, cache line and SIMD registers, as well as a lossy compression technique
with error correction in order to decrease the size of the data structure and decrease bandwidth use.
As far as SIMD-speciﬁc optimisations go, they apply the same technique as mentioned in the previous
works of using a SIMD register in order to compare multiple keys simultaneously, eﬀectively increasing
the ”branching factor” of the search [22].
The Adaptive Radix Tree (ART) introduced by Leis et al. in [14] is a trie in which the implementation
of a node is adapted to the number of keys stored in the node. While they use SIMD to search in the
same way as explained above in one particular type of node, in certain other nodes they use a direct
addressing table which is not suitable for supporting predecessor and range queries.
Graphical Processing Units (GPUs) are eﬃcient and highly concurrent processors. At a low level, multiple
processes form a group that is controlled in a fashion similar to SIMD, where the same instruction is
executed on each processor in the group simultaneously. Consequently, we will brieﬂy summarise some
recent work on index structures for the GPU. Common for all of the data structures mentioned here
is that they use the concurrency of the GPU in order to run multiple operations simultaneously, thus
greatly increasing throughput. While immensely useful, this is not relevant in our context.
Zhang et al. [24] introduce the Harmonia, which is a B+-Tree where the keys and child pointers are
stored separately and where the keys are stored in a breath-ﬁrst order. They use SIMD to perform
multiple simultaneous queries, by batching them up, and sorting them before they are run. This ensures
that queries that are headed down the same path of the tree are performed in the same search, which
gives a good cache performance. This gives them a high throughput on both the CPU and the GPU.
Shahvarani and Jacobsen implement a B+-Tree that is resident both in CPU and GPU [20]. By storing all
nodes but the leaves of the B+-Tree in both the CPU and GPU RAM, the bandwdith of both processors
can be used and the throughput is increased. They demonstrate that bandwidth, not processing power,
is the performance bottleneck. Notably, they use SIMD in order to traverse trees similarly to Schlegel
et al.
Ashkiani et al.
implement a Log-Structured Merge-Tree (LSM) for the GPU for the context where
updates and queries arrive in batches of a given size [1]. Each of the levels in the LSM is a sorted array.
When one level is full, it is merged with the next, until an empty level is found. While they do not use
the SIMD-like capabilities of the GPU for anything other than running concurrent queries, it is worth
noting that one of our data structures is inspired by this LSM.

Data Structure Overview
This section will present a high-level, intuitive overview over our data structures. Broadly speaking, the
data structures are organised into two classes: the B-Tree and the Log-Structured Merge-Tree (LSM).
Each will be covered in turn.
The SIMD B+-Tree
The SIMD B-Tree draws inspiration both from B-ary search, as presented by Schlegel, et al. in [18], and
the B-Tree, as presented by Bayer and McCreight in [3].
B-ary search is a generalisation of binary search for speeding up searches in a static collection of N
unique keys that are stored in a sorted array A.
While binary search would eliminate half of the
remaining candidates by choosing a central pivot and recursing either left or right, B-ary search chooses
B pivots which divides A into B+1 evenly sized partitions. The B pivots are loaded into a SIMD register,
and the correct partition to recurse into is identiﬁed in O (1) time. Hence, the running time of B-ary
6
==PAGE:7==
search is O (logB N), which is a factor Θ(log2 B) faster than binary search for B > 2.
However, naively implementing this procedure using SIMD would be slow because a gather load is
necessary to get the B pivots into a SIMD register. Therefore, Schlegel et al. suggest B-ary search tree
which, in essence3, is constructed in the following way. Initially, extract B pivots from A as described
above. Let those pivots be the keys in the root node of the tree. Recursively construct B-ary search trees
from the B + 1 partitions and let the resulting trees be the children of the root in the order they appear
in A from left to right. The recursion terminates for partitions of size Θ(B). Each node in the B-ary
search tree can now be eﬃciently loaded since its keys are located consecutively in memory, removing
the need for gather operations.
The B-ary search tree is similar to a B-Tree: the branching factor of every node is Θ(B), and under
certain constraints on N, every root-to-leaf path is of equal length. A natural idea following from this
observation is that the B-ary search tree can be made dynamic by implementing a B-Tree where all nodes
except the root has a branching factor of Θ(B).
The SIMD B+-Tree is such a tree, where the number of keys in all internal nodes is kept in the range
[ B
2 , . . . , B] by splitting nodes when they get too full. B+-Trees diﬀer from B-Trees by the fact that all the
keys are stored in the leaves of the tree, while keys in internal nodes are merely routing elements used
to ﬁnd the correct leaf. Consequently, range queries can be supported by maintaining left-to-right links
between adjacent leaves, performing a single predecessor search and traversing rightwards through the
leaves until all elements in the range have been reported. As will be justiﬁed later, searching, insertion
and splitting spends O (1) time in each node, leading to both update and query time of O (logB N).
The SIMD LSM
Log-structured merge-tree was ﬁrst introduced by O’Neill, Cheng, Gawlick and O’Neill in [15]. The
core idea is to organise data in data structures in multiple levels, where the size of each level increases
exponentially. When one level is full, it is merged into the next level. If the merge results in the next
level also being full, the merge cascades further down the hierarchy. If necessary, a new level is added. To
perform a predecessor query, each level must be searched individually. The LSM is eﬃcient for updates,
intuitively because it gathers the updates together into groups of increasing sizes and processes them in
bulk. However, the search performance suﬀers as a result of maintaining multiple data structures.
The SIMD LSM is inspired by the GPU LSM [1]. The ﬁrst level is a sorted array of length B, and every
subsequent level is a sorted array of twice the size. Insertions are buﬀered until there are B of them
before the cascading merge is initiated. The buﬀer exactly ﬁts into level 0. If level 0 is full, the result of
merging the buﬀer and level 0 exactly ﬁts into level 1, and so on. Hence, at all times each level is either
entirely full or completely empty.
An eﬃcient SIMD merging procedure introduced by Chhugani et al. in [7] is used for the cascading
merge. The algorithm uses a SIMD implementation of a sorting network which can merge two vectors of
size B in O (log2 B) time. Longer arrays are merged by repeatedly extracting the B smallest values using
the sorting network until both arrays are depleted. Hence the amortised time spent for one particular
element in the merge is O

Implementation Overview

6.3.1
SIMD B+-Tree
The SIMD B+-Tree is an implementation of the standard B+-Tree [3], which we assume the reader is
familiar with.
Each node contains at most B routing elements.
The routing elements are stored in
a SIMD vector which, if necessary, is padded on the right with the sentinel value ∞to ﬁll the vector
entirely. Furthermore, internal nodes contain B+1 pointers to their children. How many of these pointers
are in use is obviously determined by the number of routing elements in use in the node. The ith routing
element is strictly smaller than every key occurring in the (i + 1)th child.
Suppose you want to determine which child of some internal node v you should go to in order to ﬁnd the
predecessor of some key x. The number of routing elements of v that x is strictly greater than exactly
corresponds to the index of the desired child. This can be determined in constant time using SIMD. First,
create the key vector which has B occurrences of x. Comparing the key vector to the routing elements
of v using the greater-than comparison yields a bitvector where the number of ones corresponds to the
number of routing elements x is greater than. The number of ones can be extracted using the popcount
function. The same procedure can be used to ﬁnd x’s predecessor in a leaf, with the only diﬀerence
being that the greater-than-or-equal-to operator is used instead, in case x itself is present. Since only a
constant number of SIMD operations are used, this runs in constant time.
The search procedure for the SIMD B+-Tree can be seen in Algorithm 1 and works as follows. First, the
key vector is created (line 2), which happens only once during the search. Until a leaf is reached, use
the procedure speciﬁed in the previous paragraph to identify the next node to visit and move there (line
7). Finally, ﬁnd the key’s predecessor (line 12-13) and return it.

As usual, nodes in the B-Tree must be split if they become full during insertions. The insertion procedure
for the SIMD B+-Tree uses preemptive splitting; whenever a full node is encountered during an insertion
procedure, it is split. Splitting the node consists of copying the B/2 −1 largest routing elements and the

pointer to the new node along with a routing element which diﬀerentiates the contents of the two nodes
to the parent node. This can easily be done in constant time in the SIMD Model of Computation since
it assumes that Θ(B) data can be moved in O (1) time. An illustration of the splitting procedure can be
seen in Figure 2.

This too can be done in O (1) time, as at most Θ(B) elements must be shifted rightwards to make room
for the new key.

Since it is maintained that all internal nodes (except the root) have a branching factor of at least B/2+1

Improving Insertion Performance
As will be seen in Section 7, the SIMD B+-Tree performs quite well on searches, while not so well on
insertions. This section introduces an optimisation which trades some of the search performance for
better insertion performance. It is based on the following observation: if it is possible to eﬃciently
determine a key’s predecessor in an unsorted vector of keys, the sorting of the keys can be deferred until
the leaf node needs to be split.
Instead of inserting keys into the leaves so that they always remain sorted, new keys are simply added
at the end of the already present keys. When a node is to be split, the keys can be sorted in O (B log B)
time before the node is split in O (1) time. While this is theoretically worse than simply keeping the
leaves sorted, empirical tests proved it to be slightly faster.
What remains is to ﬁnd the predecessor of a key in an unsorted vector during a search. Intel’s AVX2
has a function which returns the index of the minimum element in a vector which can be used for this
purpose. In essence, it works by ﬁrst extracting all the numbers smaller than the key, which are all
candidates for being the key’s predecessor. Then they can be negated so that the key’s predecessor
becomes the smallest number, which can be found by extracting its index. Unfortunately, the function
for extracting the index of the minimum element works only for 16 bit unsigned integers and the actual
implementation is quite complex. However, it still runs in O (1) time. Refer to Appendix A for a detailed
description of the implementation.

An insertion of a single key into a node requires Θ(B2) elements to move in the worst case. Since elements
can be moved in chunks of Θ(B) in constant time, this takes Θ(B) time. The same argument holds for
splitting a node x, where Θ(B2) elements must be copied and a new routing element must be inserted
into the parent of x.
At ﬁrst glance, this seems to indicate an insertion time of O (B logB(N)). However, splits do not occur
often, and the insertion procedure can be shown to run in O (logB(N) + B) amortised.
To do this
using the accounting method, leave Θ(1) credits on any internal node encountered during an insertion
procedure, and Θ(B) credits on the leaf the insertion procedure ends on. The Θ(B) credits are always
enough to pay for work done in the leaf since both inserting an element and splitting the node takes Θ(B)
time. Let x be an internal node. When x is entirely new, or when it has newly been split, it is no more
than half full and has room for at least B2/2 keys. x gains at most one new key whenever an element is
inserted in a descendant of x (this happens only when one of x’s children split, and the cost of inserting
this new key in x is charged to that child), which means that it receives Θ(B2) insertions before it needs
to be split. Each of these insertions leave Θ(1) credits at x, which means that by the time x should split
it has Θ(B2) available credits. Splitting x and inserting a new routing element in x’s parent both cost
Θ(B), for which there is more than enough credits. Since there are O (logB(N)) internal nodes and only
one leaf on every insertion path, the amortised running time of an insertion is O (logB (N) + B).
Note that even though there are Θ(B2) credits available whenever an internal node should split, we only
actually use Θ(B) of them. In fact, the analysis would still work if Θ(1/B) rather than Θ(1) credits were
left on each internal node. There are two reasons for including this slack. Firstly, constant time still
needs to be spent at each node to identify the next node to visit, so the tighter analysis would not make
a diﬀerence. Secondly, the slack will be used to pay for some extra work that will be introduced in the
next section.
6.3.3
Buﬀered SIMD B+-Tree
The Buﬀered SIMD B+-Tree is inspired by the Bϵ-Tree [5]. The Bϵ-Tree is an external memory indexing
data structure which consists of a B+-Tree where a certain amount of space per node is used as buﬀers.
When inserting an element, it is added to the buﬀer of the root node. When the buﬀer becomes full,
it is ﬂushed by removing a bulk of elements headed to the same child and adding them to the buﬀer
of that child instead. If the buﬀer of the child is now full, the ﬂush cascades further down in the tree.
Intuitively, this is beneﬁcial in the external memory model because moving the elements in bulk rather
than one by one reduces the total number of I/Os required.

a new routing element is passed to the parent node as usual.
In case an internal node splits, half of the buﬀers must also be moved. Since each node contains a buﬀer
of size Θ(B) for each of its Θ(B2) children, a total of Θ(B3) elements must be moved. Since elements
can be moved in bulks of B in constant time, that requires Θ(B2) time. However, remember that in the
amortised analysis in the previous section Θ(B2) credits were available whenever a node splits (although
that analysis only utilised Θ(B) of them). All the arguments made there still hold true for the Buﬀered
SIMD B+-Tree. Hence, charging each insertion a constant number of credits is suﬃcient to pay for all
splits that might occur. Whenever a bulk of size B reaches a leaf, they can all be inserted one by one.
Each insertion requires Θ(B) time. Hence, the amortised insertion time is still Θ(logB(N) + B).
The search routine now additionally needs to check a buﬀer in addition to the work it does in the Two-
Level SIMD B+-Tree. However, that can be done in constant time by using the procedure for ﬁnding
a predecessor in an unsorted SIMD vector that was mentioned in Section 6.3.1. Hence there is only a
constant amount of extra work and the search time remains O (logB(N)).
If the running time remains the same even with the buﬀers, what is the purpose of this data structure?
While the Bϵ-Tree is theoretically better than a B-Tree in the external memory model, the Buﬀered
SIMD B+-Tree does not demonstrate a theoretical beneﬁt in terms of the model introduced in Section
2.1. However, the intention and intuition is that using the buﬀers leads to better cache performance.
One can assume that the root node of the Buﬀered SIMD B+-Tree is always in cache because it is used
by all operations. Further, during insertions a node is only loaded into cache if B elements will be moved
into it, in eﬀect reducing the total amount of time spent loading nodes into cache.
6.3.4
Implicit SIMD B+-Tree
In all the previously mentioned implementations of B+-Trees, nodes are individually allocated on the
heap and pointers are used to organise the nodes in relation to each other. This can exhibit poor cache
performance due to nodes being scattered around memory, and the Implicit SIMD B+-Tree is an attempt
at mitigating the downsides of this method.

out the entirety of T2, and so on. An example of a complete tree laid out in this manner can be seen in

are padded with ∞as required if they have less than B keys. The only additional information required

rather than in the individual nodes.

Issues with Memory Utilisation
Even though the implicit layout eliminates the need for storing anything other than the actual keys of a
node, it is ineﬃcient in terms of memory usage. Table 3 displays an estimate of the amount of allocated
space which is wasted as a function of the height of the tree when B = 8. The estimate is based on the
assumption that each node is ﬁlled with keys to the average of its minimum and maximum capacity. For
instance, the root must have between 1 and B routing elements, and so is assumed to have B/2 routing
elements. Internal nodes have at least B/2 and at most B routing elements, so they are assumed to have
3B
4 routing elements.
While this sort of capacity usage might not sound that bad, the implicit memory layout magniﬁes the
problem. Consider the representation of a tree of height h. If half of the keys and therefore half of the
children of the root are unused, that means that there are approximately B/2 subtrees of height h −1
that are laid out in memory and not at all in use. After only taking the root into account, the conclusion
is that almost half the allocated memory is wasted.

6.3.5
Static SIMD B+-Tree
The Static SIMD B+-Tree is yet another implicitly represented SIMD B+-Tree which, unlike the Implicit
SIMD B+-Tree, does not support updates.
However, it has no memory consumption issues like the
Implicit SIMD B+-Tree does, and it can be built extremely fast from a sorted array. It will be used both
as a benchmark for search performance, as well as a component of the LSM described in Section 6.4.3.
While stored in a single array, the Static SIMD B+-Tree has a diﬀerent layout than the Implicit SIMD
B+-Tree. It is stored level by level, starting with the leaves and ending with the root. An example of
this layout with B = 2 can be seen in Figure 9. All the vectors in the leaves occur ﬁrst in a left to right
order. They are followed by all the vectors from the nodes on the level above, also in a left to right order,
and so on. The starting point of each level is stored. Levels are padded with ∞as required to ensure
their lengths are divisible by B. Also, there is a minor deviation from the SIMD B+-Tree: In the Static
SIMD B+-Tree each node has a maximum branching factor of B as opposed to B + 1. The Bth routing
elements is greater than or equal to anything in the Bth subtree, and there is no (B + 1)th subtree.
Searching the Static SIMD B+-Tree relies on the observation that all the nodes on a level are tightly
packed: all but the rightmost node on a level must contain B actual keys (as opposed to the padding
element ∞). Suppose a node v has i nodes to the left of it on its own level, and that you want to ﬁnd the
kth child of v. All the i nodes have B actual keys and therefore B children each, for a total of iB children.
Therefore, on level below v, there will be exactly iB nodes before the ﬁrst child of v is encountered. If
level the below v starts at index x, the kth child of v is at index x + iB + k, which can be computed in
constant time since the start of each level is stored.
This gives rise to the search procedure which can be seen in Algorithm 3. With the exception of the new
way to calculate the location of a child, the search procedure is identical to that of the Implicit SIMD
B+-Tree. The running time remains O (logB(N)).

The LSMs
6.4.1
Scalar LSM
Our LSM is inspired by the GPU LSM [1] and the COLA[4]. This section describes a Scalar LSM which
is the basis for the work in both of these papers. In subsequent sections, the LSM will be extended with
SIMD to improve both insertion and search performance.
The Scalar LSM consists of the levels 0, 1, . . . k. Even though an earlier section stated that each level is
twice as large as the one preceding it, this is not entirely true. All the LSMs are parameterised with a
growth factor, commonly referred to as GF. For all intents and purposes each level is a factor of GF
larger than the preceding level. There is additionally an insertion buﬀer (or simply buﬀer) of size B.
To insert a new element, it is added to the insertion buﬀer. If the buﬀer happens to be full, we ﬁrst ﬂush
the LSM in the following way: First, the buﬀer is sorted. If there is room in level 0, the buﬀer is merged
into that level. Otherwise the level is emptied by merging it with the buﬀer and attempting to merge
the new sequence into level 1. This process continues until a level where there is enough space is found.
If there is no level with enough space, another level is allocated.
To be a little more precise about the growth factor, we follow the work in [4]: Level 0 has the size B and
every other level i has size 2B(GF −1)GF i−1. Under these constraints, every level i will receive GF −1
merges before it is completely full. The next attempt at merging into the level will cause the merge to
cascade further down the tree and leave level i entirely empty.
Amortised constant time is spent per element involved in a merge. Each element is involved in at most
GF merges on each level in the LSM. Since there are O (logGF (N)) levels in the LSM, the amortised
insertion time for an element is O (GF logGF (N)).
For any constant value of GF the running time
becomes logarithmic. Note that unless speciﬁed otherwise, it is assumed that GF = 2.
To search for an element, perform binary search in each level which is full. In a worst case situation,
every single level is searched which leads to a running time of O

SIMD LSM with Fractional Cascading
Fractional Cascading
Fractional cascading was introduced by Chazelle and Guibas in [6] and is a data structuring technique
for speeding up searches in multiple collections. Suppose you want to search for the predecessor of x
in k diﬀerent sorted arrays, A1, A2, . . . , Ak, of collective length N.
The straight forward solution is
to binary search each array in O (k log(N)) time. Fractional cascading will allow for a lookup time of
O (log(N) + k) by performing a single binary search in the ﬁrst array, and using the information about
x’s predecessor in A1 to ﬁnd x’s predecessors in the other arrays as well.
Figure 13 displays an example of the fractional cascading data structure, which consists k arrays
S1, S2, . . . , Sk where Sk = Ak. The remaining arrays are inductively deﬁned (in reverse order) such
that Si is the sorted array over all the elements from Ai and every other element from Si+1. For every
element in Si we store its value, a pointer to its predecessor in Ai and a pointer to its predecessor among
the sampled elements from Si+1. Furthermore, every sampled element knows where it came from in
Si+1.
This data structure has the property that once x’s predecessor in Si is found, its predecessor in Si+1
can be found in O(1) time. Assume that p is x’s predecessor in Si. Using the information stored for p
we can in constant time ﬁnd both x’s predecessor in Ai and the element q1 where p was sampled from
in Si+1. Let q2 and q3 be the two elements immediately following q1. x’s predecessor in Si+1 must be
either q1 or q2. Otherwise x is no smaller than q3, which is also sampled from Si+1 into Si, and hence
p would not be x’s predecessor in Si. Therefore, once x’s predecessor in Si is known, its predecessor in
the next level can be found in O (1) time.
This leads to the search procedure of performing a single binary search in S1 and using the pointers as
described above to ﬁnd x’s predecessor in each of the subsequent levels in constant time per level. As we
will argue in the next paragraph, the size of S1 is O (N), which yields the running time O (log(N) + k).
For any level Ai of length ni, ni/2 elements are sampled into Si−1. At most half of those, or ni/4,
are sampled into level Si−2, and so on. Hence the number of extra elements propagated from Ai into
diﬀerent levels is ni/2 + ni/4 + ni/8 + . . . ≤ni. Since this holds for every level, the number of elements
in the data structure at most doubles and therefore stays linear. Consequently, S1 has size O (N).
Note that propagating every second element to the next array is not necessarily the most eﬃcient solution.
In general, if you copy every c’th element to the level below, at most c elements must be checked on a
particular level to ﬁnd x’s predecessor on that level. So long as c is some constant, the running time
remains unchanged, and since c ≥2 the space is still linear.

7.1
Test Setup and Overview
All tests were run on mac os x with 32 GB of RAM on a 8 core i9-9880H CPU(16 hyperthreads)
with a base clock speed of 2.3 GHz and a boost speed of 4.0 GHz. Since none of the data structures are
multi threaded, only a single core was used. The CPU supports Intel’s AVX2 with 256 bit words, which
can contain eight 32 bit integers. Hence, all tests were run with B = 8.
The results of the performance tests are for the most part shown as the response time of operations given
as a function of the number of elements N in the data structure.
The response time of a predecessor search for a given N and data structure is computed by inserting N
elements chosen uniformly at random into the data structure, and searching for all of the elements. The
total time is divided by N to get the response time.
The response time of insertions is computed in the following way. Insert a large number of elements
selected uniformly at random into the data structure while at regular intervals recording the number of
elements ni inserted so far and the total time ti it took to insert these elements. It took ti+1 −ti time to
insert ni+1 −ni elements between the records i and i + 1. The average response time for each of these
insertions is (ti+1 −ti)/(ni+1 −ni), which is used as the response time for the point midway between ni
and ni+1, when N = (ni + ni+1)/2.
Note that the response time for the insertion procedure is not entirely accurate, because the number
of elements changes over time as more insertions are made. However, the average insertion time for

7.3
SIMD B+-Trees
Figure 17 shows the search performance of the SIMD B+-Trees introduced in Section 6. As can be
seen the Static SIMD B+-Tree is by far the fastest, closely followed by the Implicit B+-Tree. This is
to be expected: since both trees are allocated in a single array, only one pointer needs to be accessed.
Since this pointer is used constantly, it will always be in the TLB and TLB misses are avoided entirely.
Furthermore, the fact that the tree is located in the same consecutive stretch of memory likely reduces
the number of cache misses compared to the trees where nodes are individually allocated on the heap.
The diﬀerence in the Implicit and Static SIMD B+-Trees stems from the fact that all nodes in the Static
tree are packed, i.e. ﬁlled entirely with keys. This leads to a higher branching factor and therefore a
shallower tree than in the case of the Implicit tree, where nodes are mostly not fully packed.
Note how the data for the Implicit SIMD B+-Tree suddenly stops at about N = 224 or about 16 million.
At this point, the data structure no longer ﬁt in main memory (32GB!) due to the amount of unused
allocated space in the memory layout. Even though the Implicit SIMD B+-Tree is faster than the rest
of the dynamic trees at searching, it is rendered useless by the excessive memory usage and it will be
disregarded from this point on.
The three remaining trees all exhibit more or less the same performance characteristics. Note that the
x-axis is log-scaled. Since all three data structures have O (logB(N)) bounds for searching, their lines in
Figure 17 should be linear. However, they do for some reason appear to be superlinear. This is probably
because when the number of elements in the data structure increases, more nodes will fall out of cache
and more cache misses will occur. Eventually the root might start to fall out of cache and the root
pointers starts to fall out of the TLB.

7.5
Tuning Parameters for the SIMD LSM
Recall that the SIMD LSM has the growth factor (GF) as a tuning parameter. Furthermore, the SIMD
LSM w/ Fractional Cascading also has the propagation rate (PR) as a tuning parameter. This section
will explore how the choice of these parameters aﬀects the performance of the SIMD LSM.
SIMD LSM w/ Search Trees
Figure 19 displays the performance of the SIMD LSM with Search Trees with four diﬀerent options for
the growth factor. When GF = 2 the search performance is poor. This is because when the GF = 2, a
large number of levels are required to maintain the data set, which means that a large number of searches
in Static SIMD B+-Trees are required. There is no apparent pattern which distinguishes the cases where
GF ∈{8, 16, 22} from each other. However, for growth factors 8, 16 and 22 the number of levels needed
for 250 million elements are 10, 7 and 6 respectively, and so the number of searches performed does not
diﬀer greatly.
For insertions, the LSM performs slightly better when GF = 2 than with the other growth factors.
With a growth factor of k, a level receives k −1 merges before it is full.
That means that many
elements are involved in multiple merges per level. Furthermore, the number of elements in a level keeps
increasing until it becomes full, so merges into that level become progressively more costly until the
LSM is eventually ﬂushed. When the growth factor is 2, every element is involved in at most one merge
per level. Furthermore, sequences being merged are equally long. Hence, GF = 2 performs well for
insertions, and insertions become more expensive the higher the growth factor.
From the above it is obvious that as long as the growth factor is large enough, searching is twice as fast
as when the growth factor is 2. However, a higher growth factor leads to slower insertions. A reasonable
tradeoﬀthat performs well for both insertions and searches is with GF = 16.
SIMD LSM w/ Fractional Cascading
The SIMD LSM w/ Fractional Cascading is parameterised by both a growth factor and a propagation
rate. Searches depend on both parameters because the number of pointers the procedure needs to follow
is determined by the growth factor and the number of pointers that need to be checked on each level is
determined by the pointer density. The insertions are aﬀected by the growth factor in the same way as
mentioned above for the SIMD LSM w/ Search Trees, and it is aﬀected by the propagation rate because
that determines how many pointers must be copied down to lower levels of the LSM. In other words,
there is a complicated interplay between these two parameters and how they aﬀect the performance of
the LSM.

Discussion and Future Work
8.1
Suitability of the Simpliﬁed SIMD Model
The model of computation for analysing programs using SIMD that was introduced in Section 2.1 assumes
that all SIMD operations run in O (1) time. For any particular SIMD hardware implementation, this
assumption is reasonable; the running time of the diﬀerent operations diﬀer, but it is typically bounded
by some low, constant number of cycles.
However, as the width of the SIMD vectors continue to increase in the future, it is not obvious that
the running time of these operations will stay constant, because it depends on how the operations are
implemented in hardware.
Most of the operations mentioned in the Simpliﬁed SIMD Interface in Section 6.1 can scale in constant
time and linear physical space (i.e. the number of transistors used to create the circuit). For instance,
the addition circuit or the minV circuit for 256 bit words can easily be adapted to 512 bit words in this
manner by simply using two of the 256 bit version in parallel.
However, it is not obvious how the reorder function would scale, which we will discuss in this section.
On the Scaling of the ’reorder’ Operation
The reorder operation deﬁned in this report is in reality implemented in two diﬀerent ways in Intel’s
AVX2: Permute, which is slow, but can do everything reorder can do and shuﬄe, which is fast, but
is limited to reordering within 128 bit lanes. In other words, no elements from among the ﬁrst four
elements can move to the spot of the four last and vice versa. Also the shuﬄe within the 128 bit lanes
is limited to shuﬄing all lanes in the same way.
Permute, can not scale in O (1) time and O (B) physical space, since it would need to be able to move B
elements to B other spots in a constant amount of cycles. One of the following solutions would need to
be implemented, non of which scale in O (1) time and O (n) physical space:
• More space: permute can scale and keep time O (1), but that means it would need O

transistor scaling, as it would need B2 paths to make every permutation possible in constant time.
• More time: permute can scale and keep physical space O (B), but that mean it would need O (B)
time complexity, as it would need to shift the output vector over the hardware, and save one word
at a time.
• A trade-oﬀ: Suppose you made a circuit which could move any of the B elements to any location
in a vector of size

Since we have only focused on response time and not throughput of our data structures, only single

eﬀectively removing any response time ﬂuctuations for for inserting into the SIMD LSM.

9
Conclusion
In this report we have developed a number of vectorised data structures that solves the indexing problem.
They are all analysed using a simple model of SIMD computations, and we introduced an interface that
simpliﬁes the communication of SIMD algorithms. To our knowledge, we present the ﬁrst data structure
that uses SIMD to directly improve insertion performance.
The most eﬃcient data structure is a SIMD enhanced Log-Structured Merge-Tree.
It uses a SIMD
merging procedure for insertions and eﬃciently traverse static search trees using SIMD to speed up
searches. It is a factor three faster than a standard balanced binary search tree for searching and a
factor 20 faster at insertions. It achieves these bounds while still being within a factor four of the best
possible static search strategy we could devise. Furthermore, the data structure is easily extensible to
improve both the insertion and search performance using concurrency.

Appendix A: Predecessor Search in Unsorted SIMD Vectors
This can be done, but is not easy, since we only have one instruction to work with, which is mm minpos epu16,
which returns both the value and index of the smallest number among 8 16 bit unsigned numbers.
To use this 16 bit operation, we ﬁrst need to ﬁrst compare the high bytes of the keys and then the low
bytes of the keys. The algorithm can be seen in algorithm 6, and is described below.
• Line 2-3: Here we extract the high and low bytes of the search key, and makes vectors just contain
these keys repeatedly.
• Line 5-6: Here we load a high byte mask and a low byte mask into two vectors.
• Line 8-9: Using bit logic and we extract the high bytes and low bytes from the unsorted vector.
• Line 11-12: Here we rearrange the vector elements, so we have a vector that contains the the high
bytes in the original order ﬁrst, and then the low bytes in the original order of the unsorted vector.
• Line 14-15: Here we subtract the high and low parts of the numbers we search, from the key,
and in that way we make all numbers larger than the key negative numbers, which is seen as the
highest numbers in unsigned numbers.
• Line 17: Find the min number among the high bytes
• Line 18: Create a mask for the low bytes, where all numbers that does not match the lowest high
bytes are masked out.
• Line 20: Mask out all low bytes that did not have the lowest corresponding high bytes, by using
logical or to max them out.
• Line 22: Calculate the minimum low bytes, of those available, after mask out.

Timing
Unsorted leaves: It takes 2.69 ns per predecessor search in an unsorted SIMD vector.
Sorted leaves: It takes 0.68 ns per predecessor search in an sorted SIMD vector.
So even tough it takes 5 times longer to search in unsorted leaves, the time used for the predecessor
search in the SIMD vector is nothing compared with the around 200-1000 ns it takes to ﬁnd the SIMD
vector we need to search in.
So it is worth it not to sort the leaves in the SIMD B+-Tree, since we get a good speedup in insertions
for almost no cost in searching.
