Abstract
Post-Compromise Security (PCS) refers to a security guarantee even after a success-
ful attack where the attacker has revealed some secrets. Hence, the problem is: how
can we assure the security of the next sessions if the attacker may have compro-
mised the previous ones, and thus be able to listen to plain-text messages during
the communications. This thesis provides a new approach to assess state-of-the-art
PCS properties of existing and new protocols. We consider the formal modeling and
implementation of new paradigms to provide an automatic validation for protocol as-
sessment. The implementation is done through the existing security tool Ultramarin
(UM) for the Open-source Fixed-point Model Checker (OFMC), extending their scope
to PCS. Moreover, we evaluate several protocols, using our new validation, in specific
post-compromise scenarios.

CHAPTER 1
Introduction
In a simple Alice and Bob (AnB) communication, the 2 agents use their long-term
secrets to produce reliable messages between them. We model these secrets as private
keys kept under the knowledge of their agent ownership. These long-term keys are
very important since they allow messages digital signature and decryption. Under
these standard parameters, we define an example where Alice and Bob want to use
their asymmetric keys, using the public-key cryptography principle, to exchange and
agree on a Symmetric Key (SK), which will become their secret session key used to
encrypt and decrypt messages.
Figure 1.1: Illustration of a simple SK key exchange.
Figure 1.1 gives a visual representation of a Key Exchange (KE) protocol be-
tween AnB, where ”publicK(A)” is the public key of A, known by everyone, and
”privateK(B)” is the private key of B, only known by B. ”NA” is a nonce generated
by A to ensure the freshness of the messages. For this protocol, B is in charge of
creating the SK and delivers it to A. We can ensure the secrecy of the message M
from A, if SK is kept secret between A and B, as it is used to encrypt the secret
message M. At this stage, we can guarantee direct secrecy of this protocol.
Nevertheless, what will happen if any dishonest agent is listening to this messages
parsing and is able to decrypt the messages using the private keys of the honest
agents?
==PAGE:12==
2
1 Introduction
Let now assume that A’s device is compromised after the exchange of SK, giving
the ability to the intruder to have access to her private key. Can we still achieve full
secrecy after a compromised state? This is an important problematic we will study
in this work.
Modern security systems are made with the idea in mind that an agent’s secrets
can go under temporary adversarial control and return to their owner after a certain
period of time. In this thesis, we study security models that can satisfy these new
compromises situations and explore an automation for their protocol assessment.
1.1
Security tools
The Open-source Fixed-point Model Checker is an on-the-fly model checker tool that
analyses security protocols [BMV15]. OFMC uses Intermediate Format (IF) code as
input for its analysis. The IF is a low-level language that contains initial state of a
protocol, the transitions rules for the protocol to change state and the set of goals
that classifies the state as an attack state or not [Røp21]. By inserting protocols spec-
ification using either the IF or the Alice and Bob (AnB) notation, we can determine
if it is possible to reach an attack state using the transition rules, and as so consider
the protocol safe or not.
The Ultramarin (UM) language is a transaction-style specification language that
translates to the IF. It gives a high-level user-friendly specification that represents
each transaction of a protocol as an atomic unit [Möd22]. Thus, it makes a more
understandable language, as transactions are defined like in standard imperative pro-
grams, with stateful protocols integration. A protocol is stateful when agents can
additionally manipulate a global mutable state, and where updating the global state
during one session might influence other running sessions [HM18].
1.2
Contribution
The main purpose of this thesis is to investigate state-of-the-art Post-Compromise
Security (PCS) approaches in particular for automated verification with existing tools.
Our main contribution is to provide an automatic validation for protocol assessment,
via new paradigms. We will evaluate a number of protocols, known as ”scenarios”, to
apply the PCS verification in specific contexts. As a second contribution, we present
some ideas for an extension of Ultramarin (UM) to allow for easily specifying typical
compromise scenarios.
In order to achieve the goals, this thesis bases the reflection on the PCS paper
from Cohn-Gordon, Cremers and Garratt [CCG19]. This paper gives both formal
and informal definitions, as well as strong security models for their threat models.
They use the concept of ”test session” or ”secure session” that the adversary must
choose to attack in order to breach the security guarantee. They also present several
adversary models that show an abstract representation of the intruder capabilities

during an attack. However, the concept of sessions does not seem convincing to us,
because several sessions are going on at the same time, their start and end-points may
overlap. Moreover, agents of a session may not only disagree on each other’s identity,
but also when the session starts or ends.
Therefore, we have developed different
notions of compromise that work independently of the concept of sessions.
Our version of the PCS models are based on 2 other concepts slightly different
from the PCS paper’s work [CCG19]. We introduce a system of epochs and critical
transactions that are giving more precision on the actual compromises and behaviors
of the intruder. From a taken protocol, its execution is done over time and is composed
of several time periods from the start to the end, that are independent of the protocol’s
structure and the concrete sessions that have started and ended. These time periods
are called ”epochs” and can be compromised by the intruder.
To represent a situation when the intruder only has access to the messages decryp-
tion and signature of an agent’s private key, without knowing their actual long-term
keys, we have the Trusted Platform Module (TPM). Their use by the intruder is
for a limited period of time until the agent cleans their device and so revokes the
intruder’s ability to use their TPM. This is modeled by instantiating a new uncom-
promised epoch.
In a protocol, we can consider some of the transactions as critical, e.g., authen-
ticating a session key, and if the intruder has access to the TPM at this point, the
protocol can not achieve any security goals. Thus, we define games where the compro-
mise can happen before or after the critical transactions have been performed, and
verify if the protocol in question is strong enough to withstand the attack then. These
games exclude cryptographic properties and mainly focus on how to model different
compromise situations we want to describe through this report.

CHAPTER 2
Background
Post-Compromise Security (PCS) is a very important aspect of the Information Tech-
nology (IT) security, specially to ensure privacy-enhancement methods for the end-
to-end communications.
The disclosure about mass surveillance provoked a more
privacy-oriented awareness of the consumers.
Therefore, scientists and developers
proposed new solutions that provide security for the end users even when their ser-
vice providers are not fully trusted [Coh+17].
”A protocol between Alice and Bob provides Post-Compromise Security (PCS) if
Alice has a security guarantee about communication with Bob, even if Bob’s secrets
have already been compromised.” [CCG19]
Nowadays, such compromise can often be caused by malware infecting the agent’s
device, e.g. botnets, that can extract sensitive information such as the long-term keys
or taking adversary control of the machine by remote commands execution throughout
Command and Control channel or Peer to Peer [Tha+21]. Mitigation techniques exist,
and devices can be ”cleaned up” from malware or malicious code execution, throwing
out the intruder and their access.
Ensuring the security goals when devices communicating have been compromised
in the past is a strong challenge, and is defined as Perfect Forward Secrecy (PFS) or as
a more general term forward security. In a more precise way, when Alice talks to Bob,
PFS is achieved if the reveal of the long-term keying material does not compromise the
secrecy of the sessions keys they exchanged or defined before the exposure [Kra21].
To achieve this, the Diﬀie-Hellman (DH) protocol uses mathematics like modular
arithmetic and discrete logarithm for producing a Symmetric Key (SK) for both
sender and receiver they agree on, without exposing the raw key [AKD17].
However, PFS is just a partial section of the PCS notions. The paper [CCG19]
offers 2 security models to represent the challenge of post-compromise, one considered
weak and the other is total. Weak compromise corresponds to temporary adversar-
ial control of long-term key cryptographic operations, i.g. messages decryption and
signature, without any disclosure of the long-term keys. Full or total compromise is
described as being a more severe case where the adversary learns the long-term keys.
This disclosure gives problems in the achievement of PCS in a general spectrum.
==PAGE:15==
2 Background
5
Modern messaging applications such as Signal1 offer a protection for PCS, imple-
menting an asynchronous transmission protocol. The main idea here is to have a Key
Exchange (KE) that outputs a master secret, used to derive 2 symmetric keys, the
”root” and ”sending chain” keys. These keys are often updated using a Key Deriva-
tion Function (KDF) [Coh+17]. A KDF is a function that derives a secret key using
an input. This key refreshing principle is used in the Ratchet Algorithm (RA), where
a session key is exchanged using a KE protocol, e.g. DH, and then changed a certain
amount of times, depending on the protocol [Bel+17]. The RA principle disallows the
ability to look backwards in the keys, and so for the encrypted messages using this
feature. The Signal protocol adds more to this, and establishes the Double-Ratchet
Algorithm (D-RA), giving more advance PCS guarantees.
Other research papers work around with the current state of knowledge, namely
[CDV21] that proposes new constructions that achieve security awareness and a hy-
brid system from a weakly and strongly secure protocols. In addition, [LT18] presents
an adaptable encryption scheme for periodic rotation encryption keys.

CHAPTER 3
Design
This chapter is about the modeling of the games, introduced via their rules and
conception. Their design will be described, before being applied for various protocols
and validated through chapter 5 as part of the evaluation of results.
The first part of this chapter, section 3.1, describes the games paradigms, which
illustrates the idea behind their specification, the rules, and their attack interpretation
for the scope of this report. The second part, section 3.2, is driven by the hierarchy
of the games in a partial-ordering lattice, as well as the games relationships. Section
3.3 explores the extension language specification for the syntax and semantics.
3.1
Games definition
Verification comes from rule sets that define when and how the intruder is able to
succeed in breaching the PCS protocols. We designed 4 games, respectively game
1 (3.1.2.1), game 2 (3.1.2.2), game 3 (3.1.2.3) and game 4 (3.1.2.4), which are yet
elaborated to illustrate multiple situations.
3.1.1
Epochs and critical transaction
In figure 3.1, the protocol has 2 transactions, T1 and T2 that are messages sent from
A to B, with a potential reply from B if illustrated. T1 is the first, and then T2
the second, which means T2 always happens after T1, independently of the on-going
sessions. Let assume that these transactions are coming from a KE agreement that
leads to the elaboration of a secret SK, as a session key. We want to model time
periods that are happening over time, where these 2 transactions are part of a critical
transaction.

changes are for how the intruder can instantiate an new uncompromised epoch, how
to compromise it and then get the attack, while supporting different types of protocols.
To describe how the games work, we represent illustrations of attacks that show
different sections of compromise, for the intruder to be able to get the secrets and
so perform the attack. To help understanding the games from their illustrations of
attacks, we define some assumptions.
For all games, there exists:
• A set Secret that contains all the secrets of the protocol.
• A set of honest agents called Honest.
• A set compromisedEpochs that contains all the epochs that have been in a
compromised state.
• E is the current epoch, and is equalled to compromised when being in a com-
promised state.
• A secret s that has to be discovered by the intruder.
• A function secret(x, y, z) that represents the secret information used during the
protocol. x is a secret, y is the Receiver and z is the epoch when the secret s
is defined.
Hence, for all games, we assume:
• The Intruder I can impersonate whichever honest agent.
• The agents have access to their unique Hardware Security Module (HSM), which
is a secure module that safeguards the long-term secrets, defined by TPM trans-
actions to decrypt and sign any message.
• I can compromise any epoch.
• I only has access to the TPM of the honest agents when the epoch is compro-
mised.
• A new uncompromised epoch can start depending on the rules of the Game.
• The epochs are instantiated from e0 to en where n ∈N by 1-incrementation.
• The transactions are denoted Tn where n ∈N, and are strictly ordered by
1-incrementation. This means T2 always happens after T1, etc.
• s is secret between 2 agents, the one that sent it (Initiator) and received it
(Receiver).
• An attack occurs if the conditions defined for each game are satisfied when
the intruder gets the secret s. This will call the attack transaction that makes
Game Over.

Game 1: Messages pre-composition
Game 1 specifies that the secrets have to be discovered by the intruder during the
same uncompromised epoch as they were defined. The previous epochs can be com-
promised, and as such, TPM access could be used to construct messages before the
critical transaction. The idea behind this specification is to ensure the freshness of the
messages which shall not be guessed or produced for impersonation by the intruder
beforehand.

if the conditions are satisfied:
• s ∈Secret.
• Receiver ∈Honest.
• e1 = E, the epoch, here e1, has to be the current epoch E.
• E ̸= compromised, the current epoch E shall not be compromised.

Game 2: In-middle critical transaction
For game 2, getting secrets can give an attack if happening after the critical trans-

Epochs can be compromised before and after but not during the critical

3.1.2.4
Game 4: Full compromise
The final game 4 defines a full compromise situation where the intruder has full
access to TPM of any honest agent at any time. All the epochs are compromised,
but pre-shared information exchanged between the peers can happen.
This game
specification assumes that the pre-shared secrets happened in previous trustworthy
and uncompromised session(s). However, such assumption makes the specification
incomplete for a real-world perspective and is therefore considered weak. The main
purpose is to illustrate the difference between weak and full compromise, and gives
an approximation on how to guarantee secrecy in a full compromise situation. Since
all epochs are compromised, the attack can happen at any epoch. The Pre section
is a previous session where initial secrets have been exchanged between the peers.

3 Design
All the games start with the same parameter, the state of compromise of the
current epoch ”cuco” is false, and an initial epoch e0 is defined as uncompromised.
Variable ”E” always represents the current epoch, and ”uncompromisedE” the set of
uncompromised epochs.
The TPM transaction gives the ability to the intruder to decrypt and sign if the
current epoch is compromised, so ”cuco” shall be true.
The compromise transaction sets the current epoch to be compromised, so ”cuco”
becomes true. The current epoch is also removed from the set of uncompromised
epochs.
To start a new uncompromised epoch, all the games, except game 3, define the
current epoch compromise state ”cuco” to be false, and generate a new epoch that
will be the current one and add it to the set of uncompromised epochs. Game 3 only
allows a new epoch if the current epoch compromise state is false, so ”if cuco = false”.
The protocol represents the communication between the peers where there exists
a critical transaction. For the first step of the critical transaction, the session epoch
”SessionE” is defined as the current epoch. For each next step, the current epoch shall
be the same as the previous until the end of the critical transaction. ”Transaction
n” is the transaction where the secrets are defined, M as a secret message, B as the
receiver and ”SessionE” as the critical transaction epoch.
For the attack transaction, game 1 specifies that an attack can only happen if
the guessed message from the intruder is the secret M, B is honest and the critical
transaction epoch is the current epoch which is not compromised. Game 2 and 3
specify that the critical transaction epoch shall be in the uncompromised epochs set.
Finally, game 4 ignores the epochs in the attack transaction.
We decide to model the set of uncompromised epochs this way, and not being a
set of ”compromised epochs” as defined in the illustrations of attack because we want
the summary to be consistent with the transactions of the games implementation in
chapter 4.

3.2

We now want to prove this statement by comparison. We know from tables 3.1
and 3.2 that only the new epoch and attack transactions differ between the games.
We have 3 relations we need to prove:
Game 2 −→Game 4, Game 1 −→Game 2 and Game 3 −→Game 2.
We compare each game specification with their differences:
1. Game 2 −→Game 4:
• Attack transaction game 2: Mtmp = M, Ep in uncompromisedE and B in
honest.
• Attack transaction game 4: Mtmp = M and B in honest.
• Solution: When an attack occurs on game 2, all the conditions of game
4 are satisfied for the attack transactions. All the other transactions are
the same.
2. Game 1 −→Game 2:
• Attack transaction game 1: Mtmp = M, Ep = E, B in honest and cuco =
false.
• Attack transaction game 2: Mtmp = M, Ep in uncompromisedE and B in
honest.
• Solution: When an attack occurs on game 1, the current epoch is the
same as the critical transaction epoch and is not compromised, and so is
contained in the set of uncompromised epochs. This satisfies also game 2
which requires the attack transaction epoch to be in the uncompromised
epochs. All the other transactions are the same.
3. Game 3 −→Game 2:
• New epoch transaction game 3: cuco = false.
• New epoch transaction game 2: No condition for a new epoch.
• Solution: For game 3, if the current epoch is not compromised, then
the new epoch transaction can happen. It will also happen for game 2
since there is no extra condition to instantiate a new epoch. All the other
transactions are the same.
Now that these relations have been proved, we can conclude that:
Game 3 −→Game 4 and Game 1 −→Game 4, an attack on game 3 or 1 will lead to
an attack on game 4 as well.
Such hierarchy defined by the lattice is important to estimate the chance of attack
during the games, and furthermore help to define a level of secrecy in the PCS for
each tested protocol, shown in chapter 5.

3.3.2
Semantics
Just like the syntax, the semantics are also similar to UM, with a few modifications
coming from the changes of the syntax. This section provides a specification of the
differences and reflects on how they could be implemented. First, the rules blocks
have specific keywords and operations:
• ”is” assigns a variable to a set, which is defined if not existent before. And so,
”a is honest” will append ”a” to ”honest”. With ”if”, it can be used to define a
condition.
• ”global” keyword makes any variable global for the entire protocol, this is simply
a translation to an one-element set in UM.
• ”new epoch” translates a variable to a kind of struct datatype. An ”epoch” is
composed of 2 global variables symbolised by an one-element set for each. A
set that contains the current epoch and another with the state of compromise
for this epoch. This allows condition or assignment to define if the epoch is
compromised or not, namely as ”x.compromised”.
• ”from” creates 2 local variables for the secret and the receiver agent. If needed,
”in” can specify in which epoch the critical transaction happened.
• ”raise” provokes an attack which has to be specified.
Second, the annotations provided during the tested protocol are defined as:
• ”@critical_transaction(n)” is attached to the critical transaction of the protocol,
where ”n” is the step number defined as an integer. This annotation can be
followed by ”where” which defines the session epoch ”SessionE”, which should
be saved as a global variable using a set.
It is used to make sure that all
the communications happen in the same epoch ”SessionE” during the critical
transaction and will be used in the attack transaction to know if it has been
compromised, for this example.
==PAGE:29==
3.3 Extension language specification
19
• ”@secrets(x,y)” will be used to save the information of secrets as a global vari-
able, defined in the transaction, in a set with 2 elements, respectively the secret
message M and the receiver agent B in the provided example.
This language specification is only a reflection on how the design and implementa-
tion of the language could be done. We use a specific example to illustrate the needs.
This can change for other games, or PCS paradigms that we want to integrate for
future work.

4.1
UM Constraints
During the implementation, we encountered some issues while coding in UM. As it is
still under-development, a few constraints have to be taken into account. For example,
UM can be sometimes really case-sensitive.
When defining the transactions, we need to make sure to indicate the step in
which the transaction shall occur, each transaction in the protocol has a step number
that will increase if the next transaction happens before the previous. To do that, we
check in which step we are in, and then increment with the next positive integer.
1 if Step=0
2 Step:=1
Listing 4.1: (All games) new step.
The problem is that this will only work with the transaction happens once in the
protocol as ”Step” will no longer be ”0”, the condition can not be satisfied again.
This procedure is used to define the example protocol we want to test where A sends
a message to B, for example, and where each transaction shall happen only once.
However, when implementing the game, we want to include transactions that can
happen multiple times or at any time. To accomplish this, we are using a counter-
decrement strategy.
1 MAX := h(h(h(h(0))))

Using an arbitrary function h, which can be illustrated as a hash function, we define a
variable ”MAX” where we apply the function ”h” multiple times on an arbitrary value,
here ”0”. During the transaction, when we want to apply the counter-decrement, we
define a new value for ”MAX” each time. The new value is simply a decrement of
the appliance of the function ”h”. For instance, ”h(h(h(h(0))))” becomes ”h(h(h(0)))”
etc.. The issue here is when there is not enough appliances of ”h”. The only solution
now available for UM is to add more appliances to the definition of ”MAX”, and make
sure the number is not an issue when testing the protocols with the games.
Other issues come from a lack of features like the impossibility of directly know if
a transaction is reached without any errors. Inequations for conditions are not fully
implemented, and thus the keyword ”notin” could not be used. The implementation
goes another way around where instead of looking if the epoch is not in the compro-
mised epochs, we define a set of uncompromised epochs and verify the condition this
way instead.

5.1.5
Diﬀie-Hellman KDF
The Key Derivation Function (KDF) defined in chapter 2 is a very useful concept.
The main idea here is to imagine a protocol that can be strong enough for each game
we test. The problem comes from game 4 where the intruder is able to compromise as
their wish, making the secrecy nearly non-achievable. However, as gathered from the
state-of-the-art knowledge we have for the PCS problematic, having a previous secure
session or channel, that can not be compromised ensures PFS at a more advance level.
Let assume an initial secret exchanged between A and B before the tested session. If
this secret has not been discovered by the intruder, we can simply use it to produce
a new key after a DH KE. When the SK defined by DH is produced and agreed
between the peers, we combine it with an input in order to create new SK that
we use to exchange the secret M. The input here is the initial secret from previous
trust-worthy sessions. This is an approximation on how a Ratchet Algorithm (RA)
is working, where even if you get the actual key, you can not reverse to the initial
created one from the KDF.

Simple KE 2
The second protocol is secure against game 1. But, for game 2 and 3, it reaches a se-
crecy attack. Here all the communications that happen during the critical transaction
will be executed without issues since the epoch can not be compromised, knowing the
rules. However, at the end of the critical transaction, the intruder starts a new epoch
and then compromises it. At this stage, they will use the decrypt operation from the
TPM to decrypt the message from B which is encrypted using the public key of A
and gives SK. ”SK”, ”NA” and ”A” are all signed together using B’s private key, the
intruder can now just use B’s public key to get SK. When SK is guessed, secret M
is discovered even though the critical transaction epoch was not in a compromised
state.

Simple KE 2 Weak
The third protocol has a weakness in the messages pre-composition, and so is consid-
ered vulnerable against game 1. This example also reaches attack traces from game
2 and 3. The attack on game 2 is the same as for game 1, knowing the relation in
the lattice. Attack on game 1 will lead to an attack on game 2, which can be verified
as follows. The intruder simply compromises the first epoch that happened before
the AnB communication. They use the sign operation from the TPM to produce the
message from B that contains ”SK” and ”A”. Then, they define a new epoch to start
the AnB communication. At this stage, they encrypt the message with the public key
of A and send it to A. Since the intruder has defined SK beforehand, they obviously
have knowledge of it and then decrypt the next message from A that contains the
secret M while remaining in an uncompromised state. This is due to the fact that
the protocol does not have the freshness, defined by a nonce that the intruder would
not be able to produce before the start of the communication.

CHAPTER 6
Conclusion
In this thesis, we managed to investigate state-of-the-art Post-Compromise Security
principles, and thus study the level of security needed to achieve a fully secure post-
compromise scenario. Through this work, we were able to propose new paradigms that
satisfy our vision of the problem. We provided an automatic validation for protocol
assessment, made from formal modeling, implementation and protocols evaluation.
The implementation has been done using Ultramarin (UM) language that translates
to the Open-source Fixed-point Model Checker (OFMC), for which we extended their
scope to PCS.
6.1
Related work
This study derived from Cohn-Gordon et al.’s work [CCG19]. However, our vision
presents another aspect to illustrate the PCS problem. Hence, when they introduced
the concept of ”secure sessions”, we had some diﬀiculties to be convinced by this
notion. Since several sessions are often going on at the same time, start and end-
points can overlap. Thus, agents involved in the sessions will disagree on each other’s
identity and on when the session starts and ends.
Our solution is therefore independent of the notion of sessions. We instead intro-
duced the concept of epochs and critical transactions. The epochs are time periods
that separated the protocol, independently of the protocol’s structure and the con-
crete sessions that started and ended. Our representation of the compromise is the
temporary access to the Trusted Platform Module (TPM) of an agent by the in-
truder until they are thrown out of the agent’s device. As such, for a protocol, we
consider some of the transactions as critical, such as for authenticating a session
key for instance. If the intruder compromises the agent’s device during the critical
transaction, the protocol can no longer achieve any security goals. We then designed,
implemented and tested games, where the compromise can happen before or after the
critical transaction and make sure that protocols are secure against attacks then.
For the games’ implementation, it has been done using the Ultramarin (UM)
language and covers various scenarios. Game 1 is designed to expose problems in
protocols where messages can be pre-composed.
Game 3 checks Perfect Forward
Secrecy (PFS), where game 2 permits before and after-critical compromises. Lastly,
game 4 allows compromise for every epoch in the protocol, even during the critical
transaction. It seems hard to compare these notions now that they are based on so

different concepts, i.e. secure sessions vs. epochs, and a formal comparison must be
left for future work. We do believe however that our game 2 and 4 are in spirit similar
to the notions of weak and full compromise of Cohn-Gordon et al.’s paper [CCG19].
Finally, we explored the specification for a language extension of UM. Røpke’s
work [Røp21] really inspired us to work on it. Even though they implemented a multi-
factor authentication improvement for OFMC’s Intermediate Format (IF), they gave
well-explained instructions on compiler constructions, e.g. syntax and semantics, and
how the IF shall be apprehended. Hence, future work can consist on pushing the
extension language specification of UM to a fully implemented and tested version for
PCS protocol assessment.
6.2
Further work
As stated before, several works must be taken into consideration for the future. Re-
garding the implementation itself, many constraints can be avoided by working di-
rectly on Ultramarin and correct some of the limitations we had. It should also be
possible to optimise the code to be faster, yet an OFMC check can take a while,
because of all the combinations that have to be tested.
For the games, the authentication goal must be implemented aside from the se-
crecy goal, providing new properties for the implementation and protocol assessment.
Formal comparisons with the related work are also needed, as well as the implemen-
tation of state-of-the-art PCS-secure protocols, such as Signal’s protocol for instance.
The addition of equivalent or approximations to those protocols could help to estab-
lish a larger spectrum for the PCS paradigms.
Finally, during this project, we reflected on a language specification for the new
paradigms. Further work will be to implement an extension language for Ultramarin
or the Intermediate Format. Implementing directly upon OFMC’s Intermediate For-
mat can give more flexibility, but be harder to implement, since it is low-level lan-
guage.
