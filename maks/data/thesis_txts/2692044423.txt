Summary
Model learning for model checking is a ﬁeld that has seen a lot of research
in recent year. It is focused on automatically constructing system models by
observing system executions. Recent work in model learning strives to improve
the estimates of transition probabilities for stochastic/probabilistic models.
The goal of the thesis is to create a workﬂow for learning stochastic models from
automatically generated implementations of non-deterministic models. The gen-
erated implementations includes a component that takes care of making the
non-deterministic choices. This workﬂow utilizes work previously done in the
realm of model learning in order to learn the stochastic properties of a model.
The thesis includes a Java implementation of the workﬂow and furthermore an
evaluation of the performance and precision of the output of the workﬂow.

Chapter 1
Introduction
Model checking is a formal veriﬁcation technique used to analyse a system model
against properties that can be speciﬁed by a logical expression [MCJ+12]. There
are diﬀerent types of models that can be used in model checking, such as Tran-
sition Systems (TS), which are non-deterministic, and Discrete-Time Markov
Chains (DTMCs), which are used to model stochastic systems. The PRISM
tool is a probabilistic model checker, where a model can be described using the
PRISM language and properties of the models can be automatically analysed
using temporal logic.
Although model checking has many advantages to other methods utilised in the
industry, such as unit testing, it is not widely used. One reason for this is that
the construction of accurate models for real-world systems is usually diﬃcult and
time consuming. Therefore, researchers have in recent years been studying model
learning techniques, that can automatically construct a model from executions
of a real-world system [MCJ+12].
The learned model can then be used for
further analysis and veriﬁcation of properties of the system. Researchers have
in recent years adapted machine learning techniques more and more for model
learning in order to improve the performance of the learning process and the
precision of the learned models (e.g., [WSYP17, WCSQ17, WSQJ18, MCJ+11]).
One of the hardest part about learning a stochastic model is estimating the
transition probabilities of the model.
In [BGJS20], the authors presented a
==PAGE:12==
2

learning algorithm that gives global bounds on the error made by the learning
process. The assumption that the authors make is that the set of states and
the transitions between the states of the model are known, but the transition
probabilities are not known.
Although a lot of research has been done in the realm of model learning, there
is not much work to be found in the other direction. That is generating im-
plementations, or executable code, from models of a system. This might be
especially useful in order for the industry to utilize model-driven development
(MDD) techniques more for their development. No prior work has been found
on generating implementations of models speciﬁed in the PRISM language. The
PRISM model checker does however oﬀer to simulate the execution through a
given PRISM model [KNP11].
In this work, we are inspired by all the recent work that has been done in
model learning, especially [BGJS20], with the goal of building a new workﬂow
for learning stochastic systems.
It is also motivated by the fact that there
is no functionality available in the PRISM model checker that can generate
an executable implementation based on a model. In particular, we would like
to create a workﬂow where given a non-deterministic model, one can obtain a
version of it with probabilities, that reﬂects some kind of an external component
that makes the non-deterministic choices. Included in that workﬂow will be a
process that generates executable implementations from models.
To show an example of how the workﬂow would work, imagine a non-deterministic
coin. The only action that can be taken is to toss the coin, but the outcome of
the toss is controlled by some external component. That is the probabilities of
the coin landing on either side are not known. A fair coin on the other hand
is stochastic and has a 50/50 chance of landing on either side. The workﬂow
would allow us to move from the non-deterministic coin to the fair coin.
In this work, our main contribution is a tool implemented in Java that per-
forms the workﬂow described above. Given a non-deterministic model, the tool
generates an executable Java implementation from the model. Included in the
generated implementation is a component that makes non-deterministic choices
uniformly at random. The generated implementation is then used by an im-
plementation of the learning algorithm presented in [BGJS20] which learns the
stochastic properties of the model. The output of the learning algorithm is then
used to construct a stochastic model. The models will be described using the
PRISM language.
In chapter 2 we will introduce notations and the theory behind models, model
checking and model learning, along with a brief background on the PRISM
model checker. In chapter 5 we will point out related work. Our main contri-

2.2
Model Checking
Model checking is a formal automated system analysis technique. Given a model
and a property, a model checker can be used to automatically check whether the
property holds in the model, or provide a counterexample to falsify it [WSYP17].
This is especially useful for checking for bugs in systems, or to check if a failure
property can be reached.
Properties can be speciﬁed in temporal logic, e.g.
Linear Time Temporal Logic (LTL) or Computation tree logic (CTL).

Models must be speciﬁed in the PRISM language in order to able to use them
for analysis. An example of a PRISM model can be seen in listing 2.1. The ﬁrst

model is a DTMC. PRISM models can be composed of many modules, where
each module represents a diﬀerent process of the system. A module is speciﬁed
as module name ... endmodule.
Modules contain two things, variables and commands. Variables describe the
possible states of the module while the commands specify the possible behaviour
of the module. The variable speciﬁed in the example is setting :[0..2]init
0;, which represents an integer with range [0..2] initially set to 0. Variables in
PRISM can either be integers or Booleans.
The ﬁrst command in the example is []setting=0-> 0.5:(setting’=1)+ 0.5:(
setting’=2);. Commands are split into two parts, everything left of the arrow
is the guard and on the right side are one or more updates. The guard setting=0
represents that this command can be executed when the variable setting is set

Once a model has been speciﬁed in the PRISM language, properties in temporal
logic can be speciﬁed in PRISM in order to analyze the model and perform model
checking. PRISM oﬀers more functionality that can be read about in the PRISM
manual [KNP11].

Chapter 3
From Non-deterministic to
Probabilistic Models
This chapter will give an overview of the workﬂow where each part of the work-
ﬂow is broken down. The main goal of this thesis is to develop a model-based
approach to learn stochastic models from implementations of non-deterministic
models. The workﬂow can be split into three main parts.
First, a methodology will be designed to automatically generate implementa-
tions from non-deterministic models, so that the generated implementations are
amenable to log-based learning algorithms. The generated implementations will
include an external component that takes care of making the non-deterministic
choices. Then, the learning algorithm mention in chapter 2.3.1 will be imple-
mented and applied to infer the stochastic properties of the models. It will utilize
the generated implementation to obtain traces for the learning process. Lastly,
the result from the learning algorithm will be used to construct a stochastic
model that can be utilized for further analysis.
This approach will be instantiated in a speciﬁc case based on PRISM, to learn
stochastic models (DTMCs) from generated Java implementations of non-deterministic
models (TSs). The workﬂow was implemented in Java as a console application
and is available at GitHub.

3.1
Non-deterministic Models
Non-deterministic PRISM models can vary in complexity, size and format. For
the sake of this thesis, the types of models that the project accepts as an input
will be restricted. The model must fulﬁll these requirements:
• It only has one module. The model is of a monolithic system and it is not
split into many components.
• The actions are sequential. The model is modeling a sequential process
where there is no concurrency. That means that the action set is either
empty, or all commands have the same action label.
• PRISM type bool is not used. All variables of the model should be of
PRISM type int. PRISM type bool can be converted to type int by
making it a variable with range [0..1] with 0 representing false and
1 true.
• It does not include formulas or constants. Models using PRISM formulas
or constants are not supported in this project at this time.
• It must be a PRISM model type mdp. The model must be a non-probabilistic
mdp, which represents a TS.

3.2
Code generation
The ﬁrst part of the project is the code generation. It receives a non-deterministic
PRISM model, a TS, as an input and outputs a Java implementation of the
model. The implementation is generated in such a way that it can be easily
simulated and the state of the system is recorded after each action is performed.
The implementation is therefore ready to be used for the learning algorithm.

The code generator will ﬁrst read through the PRISM model ﬁle that is given
as an input and gather information that is needed about the model. That is,
the model commands, their guards and updates, and the model variables, their

3.3
Executing the implementation
In order to be able to use the implementation in the learning algorithm, the
implementation must be able to be executed. The implementation can be exe-
cuted in two diﬀerent ways. The ﬁrst way is to execute the implementation with
predetermined number of steps, that is the implementation takes as an input
how many transitions/actions should be performed. The second way to execute
the implementation is to initialize the system and then perform one transition
at a time.
When the system is initialized the variables of the system are set to reﬂect the
initial state of the model. The system then retrieves all possible update/tran-
sitions that can be performed given the state of the system. It was decided to
utilize the Java ScriptEngine, in order to do this since it provides scripting
functionality. The ScriptEngine enables the guards to be evaluated to ei-
ther true or false, given the values of the variables in the system. If a guard is
==PAGE:27==

evaluated to be true, then all their update identiﬁers are added to a list.
Once all possible transitions are known, the system needs to choose which tran-
sition to perform next. At this point in time the transitions are chosen uniformly
at random, which reﬂect a component that chooses between outgoing transitions
of a state where the outgoing transitions all have the same probability of being
chosen. This is done by choosing a update identiﬁer at random from the list of
all possible updates. The update to the system is then performed as speciﬁed
in chapter 3.2.

3.4.2
Backward Reachability
The set R∗(s) is one of the things required in order to be able to compute the
stopping condition for the algorithm. There was no functionality available in
PRISM that could easily ﬁnd R∗(s), so we implemented an algorithm that would
ﬁnd it. This algorithm uses backward reachability to ﬁnd for each state s, the
set of states i ∈R∗(s) such that there exists a path from i to s. We exclude s
from R∗(s). Algorithm 2 is based on other model checking algorithms presented
in [BK08].
Algorithm 2: Backward reachability
Result: For each state s, the set of states R∗(s)
S;
forall s ∈S do
W := ∅;
R := ∅;
Add s to W and R;
while W ̸= ∅do
Remove some state t from W;
forall t′ in Pre(t) do
if t′ /∈R then
R := R ∪t′;
W := W ∪t′;
end
end
end
Return R∗(s) := R\s;
end
Here W is the working set that maintains the set of states to be explored and R
is the set of states that have a path to the state s, including the state s. Pre(t)
is the set of states that have a transition to t. The algorithm then returns the
set R∗(s) which is later on used to compute one of the stopping conditions for
the algorithm.

Chapter 4
Evaluation
In this chapter, we will evaluate the workﬂow described in chapter 3 on 5 dif-
ferent models.
The objective is to gain understanding on how the workﬂow
performs and how precise it is. The performance will be evaluated in regards to
how much time it takes, from start to ﬁnish, and how well it scales with models
of increasing size and complexity. The evaluation of the precision will test if
the implemented learning algorithm works as expected, that the output of the
implemented algorithm follows the guarantee stated in chapter 2.3.1.
The results for the evaluation were obtained on a 2.4 GHz Intel Core i5 laptop
running macOS Big Sur with 8 GB memory.
4.1
Models for evaluation
The 5 models used for evaluation were chosen to provide some idea on how well
the workﬂow scales in regards to the size of the models. That is, how well it
works with models with few states and transitions, to how it works with models
with many states and transitions.
The models were also chosen to have an
increasing number of non-deterministic transitions in order to test the learning
algorithm.
==PAGE:34==
24
Evaluation
The 5 models are as follows: Model 1 is the example presented in ﬁgure 2.1.
Model 2 describes a client that can create a new task, where the length of the
task is chosen non-deterministically from 1 to 5, decrease the length of the task
until it is 0 and then repeat. Model 3 is based on a model provided in the
PRISM case studies [KNP11]. It describes a six-sided dice where a fair coin is
tossed repeatedly to simulate throwing a dice and ﬁnding the value that it lands
on. Model 4 includes three variables, all initially set to 0. At at each point,
the value of one variable can be increased by 1 until the value of any variable
reaches 3. Model 5 is based on one of the examples evaluated in [BGJS20]. It
is a 100-state clique, where every state has a transition to every other state in
the model, including itself.
We assume that the expected transition probabilities of the learned DTMC
should be uniformly distributed, depending on how many outgoing transitions a
state has. That is for a transition from state s to state t, the probability of that
transition should be 1/sout, where sout is the number of outgoing transitions
from state s. This assumption is set since the workﬂow that was implemented
chooses which transition to perform at each time uniformly at random.

4.3
Precision
In order to evaluate the precision of the workﬂow we will compare the learned
DTMCs to the expected DTMCs, where transition probabilities of the non-
deterministic transitions are uniformly distributed. We test our workﬂow 100
times for models 1 and 2 with ϵ = 0.1 and δ = 0.05. This means that at least
95 of the learned models should have probabilities that diﬀer by less than 10%
from the expected model.
For model 1, the probabilities diﬀer by less than ϵ 99 out of 100 times. For model
2 they diﬀer by less than ϵ 100 out 100 times. This indicates that the guarantee
provided by the authors of [BGJS20] holds in the implemented learning algo-
rithm. That is, the stochastic models output by the workﬂow have estimated
transition probabilities that are close to the expected probabilities.

Chapter 5
Related Work
This paper draws inspiration from recent work that has been done in the ﬁeld
of model learning. Mainly from work where machine learning techniques have
been adopted in order to learn probabilistic models. In [WSYP17] the authors
presented a genetic algorithm (GA) in order to learn a DTMC. The GA proved
to give more accurate result than previous learning algorithms and provided a
more intuitive approach to control the degree of generalization of the learned
model.
In [WSQJ18] the authors were motivated to verify a cyber-physical
system (CPS) and created a method that combined model learning, abstraction
and reﬁnement. This method takes a property of the system to be veriﬁed as an
input to the algorithm and abstracts away all information that is not needed in
order to verify it. This method proved to be more eﬃcient than the GA method
at learning models for speciﬁc properties.
The work mentioned above focuses on automatically learning models from sys-
tem executions, where information about the set of states is not known before-
hand. In this thesis, the set of states along with the transitions is known since
that information is obtained from the non-deterministic model. Therefore, work
on estimating probabilities of transitions of a model with a know set of states is
more applicable. In [WCSQ17] the authors assume that the states of a model
is known and apply an approach built on the idea of active learning to sample
system executions that are more likely to include rare transitions. The work
presented in [BGJS20] provided global bounds on the error made by the learn-
==PAGE:40==
30
Related Work
ing process, where the aim was the estimate transition probabilities with known
states and transitions of the model.
The method presented in [BGJS20] provided a foundation for the project pre-
sented in this thesis.
However, in this thesis a full workﬂow was created to
obtain probabilistic models from non-deterministic models instead of only pro-
viding the algorithm with a set of states and a transition matrix. That workﬂow
included automatically generating executable Java code from non-deterministic
PRISM models. There is not much related work to be found on generating im-
plementations from PRISM models. However, the PRISM model checker does
oﬀer to simulate the execution of a given model [KNP11]. There has been some
prior work done on code generation from other high level descriptions of a sys-
tem, such as [AT98] and [NS14], but they do not work with the PRISM model
checker.

Chapter 6
Conclusion
In this paper, we presented a workﬂow to learn a stochastic version of a non-
deterministic model.
It works by generating an implementation of the non-
deterministic model, then using it as an input for the implementation of the
learning algorithm from [BGJS20] and outputting a learned stochastic version
of the model. While it turns out that the workﬂow does not scale well with
models of increasing size, it is still a good starting point at establishing a new
way of performing model learning. The results from the evaluation showed that
the workﬂow works well with models of smaller size and that the estimated prob-
abilities of the stochastic model satisfy the guarantee provided by the authors
of [BGJS20].
There is potential for a lot of future work that could be done for this project.
Obvious limitation of the workﬂow that could be improved upon is the ability
to work eﬃciently with larger models.
In order to do that, a more eﬃcient
way to compute B( ˆA) must be found, but changing the learning algorithm to
check the stopping condition less frequently could also help the performance.
Parallelizing the generation of traces for the learning algorithm could also help
improve performance.
Another limitation that was exposed during the evaluation is that a implemen-
tation can not be generated if there are too many transitions in the model.
Therefore, a better way to store the PRISM commands in the generated imple-
==PAGE:42==
32

mentation must be established. One possible solution to this problem might be
to store the commands in a database that the generated implementation can
read from instead of storing the commands in the generated class.
The largest limitation set by the workﬂow are the requirements set on the non-
deterministic models it can take as an input.
Future work should focus on
enabling the models to contain more than one modules and enabling the models
to contain concurrency.
This would make the workﬂow more useful since it
would allow more complex models to be used.
To make the workﬂow more
useful for practical use, one could also enable sampling the non-deterministic
choices on other distributions then uniform distribution.
