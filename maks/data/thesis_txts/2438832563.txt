Summary
Normal user interfaces are often awkward in virtual reality because they generally
require a 2D surface. In contrast everyday objects are typically easy to use because
we know their affordances and well-designed objects have signifiers that clearly indi-
cate precisely how we actually make use of those affordances. The aim of this project
is to greatly limit the need for a traditional user interface in a VR based 3D mod-
eling system by imbuing the work piece with clear signifiers that assist the user in
understanding and making use of its affordances which in this case are the possible
modeling operations.

CHAPTER 1
Introduction
Virtual Reality is a fairly young, ever-expanding field with a lot of room for improve-
ment on nearly every aspect. One of such aspects is User Interface. Traditionally
programs’ UI require a flat 2D surface on which the elements of said UI are placed.
Such an approach does not work well with VR, because the way we interact with
objects in VR is very different. In a nutshell VR interactions are very similar to our
everyday interactions in real life. To simulate that lots of developers create a UI by
making specific tools that the user holds in his hands and uses them to interact with
the objects in the scene.
While a tool-based UI works well in a lot of cases, there is still room for improvement.
The idea of this project is to try out a new way of making UI and find out how viable
it is. The new approach consists of looking at the way we interact with the real world.
It includes studying affordances and signifiers (for more details on that concept, refer
to chapter 2.2) of different objects in real life to create a UI embedded into the object
to give it characteristics that will enable the user to understand how he is supposed
to interact with that object. Such an approach can already be seen in some of games
created for VR, where it mainly applies to grabbing different objects, or pushing some
buttons. However, it have not yet been used to imbue objects in VR with affordances
that they do not possess naturally.
For this project it was decided to look into 3D modelling systems and create one using
a new approach to UI. The reason for choosing to create a 3D modelling system is
that it poses a lot of challenges. First of all, every modelling tool is a very compli-
cated system with a lot of functions. If we look at some of popular programs, like
Autodesk Maya[Aut] or Blender[Fou], we can not help but notice how complicated
and cluttered their UIs are. Obviously, such UI is not reproducible in VR, thus it
must use a different approach than the traditional one. One of programs that already
sucessfully employs the idea of a tool-based UI is Google Blocks[Goo], which we will
describe and compare a little later in chapter 4. Moreover, making a 3D modelling
tool poses a problem of somehow integrating the UI into the object being modelled.
It will include thinking of possible signifiers to provide affordances to an abstract
object, that does not possess any of those in the real world. That is exactly the type
of challenge that is sought after for this project.

CHAPTER 2
Theory
The project relies heavily on two core concepts, therefore it is necessary to explain
both of them before diving deeper into the work done. One of those concepts is Half-
edge Mesh Data Structure, which is essential for creating and manipulating meshes
in an efficient way. The other one is the principle based on Affordances and Signifiers,
where two terms come very closely intertwined, therefore it is logical to consider them
as a whole unit rather than separately, the reasoning behind that is explained more in
details below. Affordances and Signifiers would be exploited heavily further on when
the time comes to design user interface.
2.1
Half-edge Mesh Data Structure
To easily manipulate a complex multi-faced mesh, we require a proper data struc-
ture. There are multiple ways of representing a mesh in Computer Graphics. Most
common and simple is an Indexed Face Set (further referred to as IDFS), where a
manifold M is represented by n vertices V = [v1, v2, ..., vn], where vi = [x, y, z] and m
faces F = [f1, f2, ..., fm], where fs = [vi, vj, vk, ...]; i, j, k, ... ∈[1..n]; s ∈[1..m]. This
representation is very straightforward and is used in many rendering engines. How-
ever, the simplicity of IDFS is also it’s downfall when it comes to manipulating the
manifold it represents. For example, if you would want to find all the faces adjacent
to face fi, you would have to iterate through the whole list F and check if fj; j ̸= i
has at least two vertices in common with the face fi.
This is where Half-edge Mesh Data Structure (further referred to as HMesh) comes
in place. The key feature of HMesh is that each edge is split into two directional
opposite edges, each half belonging to one of the two adjacent faces. As seen on
figure 2.1, edges hold references to:
• the vertex it is pointing to
• the face it is ”inside” of
• the next half-edge belonging to the same face
• the previous half-edge of the same face

• the opposite half-edge
Figure 2.1: Half-edge Mesh Structure Example.
With such a setup, every face needs only to reference one random half-edge belonging
to it, which automatically grants access to all of the vertices belonging to this face,
as well as to all the adjacent faces. Furthermore, every vertex has a reference to one
of the half-edges coming into it, which easily lets you find out which faces this vertex
belongs to just by going to the NEXT half-edge and taking the OPP half-edge’s
face, until we reach the initial half-edge.
Finally, it is important to note, that all the edges inside the face are oriented in a
counterclockwise fashion to uphold the right thumb rule, which gives us a very easy
control over the direction of the face’s normal.
To summarise the Half-edge Mesh Data Structure:
• A Manifold consists of an array of vertices, faces and half-edges.
• Each HalfEdge is a directional edge, pointing at a specific vertex.
It has a
reference to the next HalfEdge from the same face, to the previous HalfEdge,
and to the HalfEdge opposite to it.
It also holds a reference to the face it
belongs to and to the vertex it points at.
• Each Face holds a reference to one of its HalfEdges.

2.2
Affordances and Signifiers
The term affordance in application to human-computer interaction was introduced
by Don Norman in his book ”The Design of Everyday Things”. Norman defines affor-
dance as: The term affordance refers to the relationship between a physical object and
a person (or for that matter, any interacting agent, whether animal or human, or even
machines and robots). An affordance is a relationship between the properties of an
object and the capabilities of the agent that determine just how the object could possibly
be used[Nor13]. This way a ball has an affordance of being picked up, a cup has an
affordance of pouring liquids into it, a table has an affordance of putting objects on it.
However, together with affordances, Norman introduces anti-affordances, i.e. what
an object forbids us from doing. As an example, he states that a window glass has an
affordance of seeing through it, but at the same time it has an anti-affordance of not
letting things pass through it. It is easy to notice, that although a glass window or a
glass door posses such anti-affordance, a lot of people actually try to pass through a
closed glass door at full speed, which often results in injuries and spoiled mood. The
explanation of that phenomena lies in the understanding of signifiers.
Once again, Norman chooses this definition of signifiers: For me, the term signifier
refers to any mark or sound, any perceivable indicator that communicates appropri-
ate behavior to a person[Nor13]. So in other words, a signifier is something that is
there to explain to us how we should communicate with an object. Going back to the
previous example of glass doors, just adding a handle or a steel plate (which would
indicate pushing) can save us the trouble of getting a headache from bumping into
the transparent glass, because either a handle or a steel plate will clearly communi-
cate both the anti-affordance of not letting things pass through a solid object and
the affordance of an object that can be opened by performing some action. However
beautiful the concept of signifiers is, there exists a danger of false signifiers. As an
example, it is easy to remember so many doors that had a handle, but also had a very
==PAGE:16==
6
2 Theory
strong signifier accompanying them - a huge sign saying that you have to push, not
pull. Why those signs had to be placed? It is simple, the handle, as we already un-
derstood conveys to us the action of pulling, but the door was designed to be opened
outwards, rather than inwards, thus making the handle turn from a good signifier
into a false one. It is worth mentioning here that when one designs a signifier it is
good to abide by the principle of least astonishment (POLA). It is a relatively new
and undocumented concept in software development. From Colishaw’s ”The Design
of REXX language” [Cow84] it reads: If a feature is accidentally misapplied by the
user and causes what appears to him to be an unpredictable result, that feature ... is
therefore undesirable. More information can be picked from IBM article ”The cranky
user: The Principle of Least Astonishment”[See01], but in the end, it all comes down
to one simple rule: ”Do not design something that might confuse the user”. If we
apply that rule to our topic, it will mean that you should never create a signifier
that could be misinterpreted, it must not confuse the user leading to an unpredicted
outcome (as in our example with the handles above).
Finally, Norman emphasises another type of false signifiers: misleading signifiers.
It’s a type of signifiers that are designed to convey something particular to a non-
knowledgeable person (usually an anti-affordance), while allowing people who know
the trick to not be confused by this signifier. As an example, Norman brings up a
special row of pipes, working as a fence, installed on a service road leading to a park.
This fence clearly signifies that cars can not pass through here, however, what is hid-
den is that the fence is made out of rubber, so whenever the maintenance personnel
needs to go into the park, they can easily drive over those pipes.
Let’s summarise everything said.
• Affordances of an object show us what actions with the object are possible (a
ball has a round form so that it can be easily picked up and thrown; a cup has
an empty space inside where you can pour liquids).
• Affordances might be easily perceived, thus acting as signifiers by themselves
(a handle has a curved form that is easy to grab and pull), and might be hidden
(glass lets you see through it, but doesn’t allow you to pass through it).
• Signifiers convey to you how you should interact with an object. They must
be easily discoverable and understandable (a clear sign that says whether you
should push or pull).
• Obey to POLA when designing a signifier (don’t make a handle on a door if
you want the user to push it; don’t put a floppy disk icon on a button if that
button is not going to save user’s work).

CHAPTER 3
Implementation
As described in chapter 1, the goal of the project is to create a minimal functioning 3D
modelling system in VR, that will not rely on classic external interfaces, but embrace
the concept of affordances and signifiers (see chapter 2.2) to create a UI embedded
into the object being modelled. The system is built around the extrusion of different
faces. That being a pretty difficult and costly operation, it is wise to perform it with
the help of a C++ library for both speed and light-weightness. One more important
thing to keep in mind is that we are going to be working with quad faces in order to
not impose additional complications for the extrusion algorithm, where now we can
assume that we are always going to have 4 vertices on each face, and each vertex will
have 3 half-edges coming in and out of it.
The development of the prototype for this project has been a long process, consisting
of multiple steps and iterations. Unity 3D was chosen to create the system for its sim-
plicity, means of connecting DLL libraries, API availability for Oculus Rift headgear
and general familiarity. As shown on figure 3.1, the system consists of the following
components:
• Oculus Rift headgear with Oculus Touch controllers which are connected to
Unity through Oculus Utilities for Unity and OVR Avatar SDK.
• Unity scene for generating user interface and managing user input.
• A C# script layer that is, besides it’s usual usages inside Unity, responsible for
communication between Unity and a C++ DLL.
• GEL Library compiled into a DLL that is responsible for applying actions, such
as extrusion and movement, to the manifold and sending the result back to the
C# layer.

3.1
GEL Library
GEL is a C++ library, created and managed by Jakob Andreas Bærentzen[Bær14].
It is light-weight, yet provides a powerful set of tools for 2, 3 and 4D vectors, some
linear system solving and Mesh representation of surfaces. The latter is of the ut-
most importance to us. GEL provides a convenient way of creating manifolds and
performing different complex actions on them. A manifold in it is represented by a
half-edge mesh data structure (see chapter 2.1), which enables us to easily implement
the operations needed, i.e. extrusion and movement of the face.
The extrusion algorithm is a crucial but fairly understandable part of extending the
GEL library. As mentioned above, we are assuming that we work exclusively with
quad faces. Shown with pseudo-code 1, the algorithm starts by accepting the manifold
and the list of face ids that are to be extracted. Next we circulate each face counter-
clockwise and collect all half-edges involved. Then we combine all the half-edges into
pairs of themselves and their opposite half-edges, that will be needed later for creating
new faces. After that we need to iterate through all the half-edges to find half-edges
going into vertices and going out of them. When such a pair of half-edges is found
we slit the vertex, creating a new one and re-assigning all the half-edges that used
to point to the old vertex to the newly created one. After we have our new vertices
created and half-edges pointing to them, we can now iterate through the previously
gathered half-edge pairs and create new faces by taking the vertices that the first
half-edge in pair points to, it’s opposite counter-part (NB: the opposite half-edge now
is not the same as the second half-edge in the pair, because of slitting), the second
half-edge in the pair, and its opposite half-edge. It is important to note that we
should assign a specific cluster id to the vertices mentioned above, which we will later
assign to the vertices created from adding a new face. After all the new faces are
added, we stitch the mesh with the cluster ids we assigned to them. For the actual

3.2.1
Pitfalls of C++ compiler
During the step of integrating GEL with Unity, a strange bug was uncovered, which
was extremely difficult to debug. The symptom appeared when trying to extrude a
face and move it out. Visually it reflected in one of the newly created faces ”missing”
a triangle. Due to a layered structure of the system, the debugging process included
a lot of testing and logging to files. It turned out that under the hood the mesh
was not being stitched properly, which resulted in the wrong vertex being collapsed.
Further investigation uncovered that the cluster ids in the extrusion algorithm 1 were
not getting proper values assigned to them. Interestingly enough, it was not due to
the error in the code, when we tried to use the same algorithm on a machine running
MacOS it worked flawlessly, which allowed us to uncover the source of the bug.
The main clue was that the cluster ids were getting strange values, that looked like a
random memory access (see appendix C for an extract from logs). So, it was important
to look into the cluster ids vector class. It is an extension over the standard vector
class from the C++ STL.

• First time such an item is accessed, it performs the re-size, gives back a proper
reference, but does not save the value that was tried to be put in there.
• Second time this same item is accessed, any operation can be successfully per-
formed.
• This same behaviour does not appear when compiling for another system, i.e.
MacOS (no Linux distribution was tested in that scope), assignment can freely
be used from the first access.

3.3.1
User Interface for the prototype
The UI for the prototype had to be developed in a way that the object being modelled
would receive a clear affordance which the user could easily understand. If we look
at the process of a face extrusion it is very similar in nature to pulling a piece out of
plasticine without breaking the whole thing. In order for a simple cube to get such
an affordance it can be as easy as placing a handle in the center of a face. In this
case, a handle works well, providing the affordance of something that can be pulled,
as well as, it is a clear signifier to the user the he can interact with the model just by
grabbing onto the handle.
In order to test if different shapes would work better or worse as signifiers, two differ-
ent options of a handle were created. One would be a simple ball, sunk half-way into
the face (see figure 3.2 for a reference) and the other one would be a torus shaped
handle (figure 3.3). The reason for choosing a ball as another candidate for a signifier
is that much like a torus shaped handle it has an easily recognisable affordance of
being grabbed. Thus, potentially it could be as good as a handle.

3.3.2
Architecture
As we have mentioned in the beginning of the chapter, the system is a complicated
structure consisting of many components (figure 3.1), which are also broken down
to smaller components. An overview of the architecture can be seen on figure 3.4.
We have already covered what the top branch Manifold →IManifold →GEL DLL
does in the previous sections, shortly, it is the one responsible for connecting C++
library with C# layer to manage the Manifold, and it connects into the central com-
ponent ExtrudableMesh.
ExtrudableMesh is responsible for:
• Holding the reference to Manifold.
• Passing actions received from controllers to the said Manifold.
• Calling triangulation, translation to IDFS and drawing the Manifold.
• Generating the controls placed on the Mesh.
The FaceHandleController is responsible for how the placed handles react to virtual
hands provided by Oculus Avatar SDK. The Controller implements the IInteractableObject

abstract class, which holds three methods StartInteraction, Interact and StopInteraction.
The StartInteraction method is called by the OVRAvatarHand script. The Interact
is called from StartInteraction and is the actual method that will be doing all the
work and communicate with the ExtrudableMesh. Whenever the hand releases the
object (in case of FaceHandleController and VertexController, which are dragged
alongside with the hand) it calls StopInteraction in case something should be done
in the Controller. Such a setup allows us to abstract ourselves from the objects we are
interacting with in the OVRAvatarHand and delegate all the work to the Controllers.
Lastly, OVRAvatarHand is working on top of a simple 3-step state machine. It can
be in EMPTY, TOUCHING or HOLDING state, and it can go from EMPTY to TOUCHING and
back, and from TOUCHING to HOLDING and back. Whenever the virtual hand crosses
an interactable object’s collider, if it is in an EMPTY or TOUCHING state it switches to
the TOUCHING state and remembers the object it came in contact with. If the user
then proceeds to squeezing the grab button (in the appendix D what we call the grab
button is Axis1D.PrimaryHandTrigger) on the touch controller, the hand goes into
the HOLDING state and sends the signal to the object it started the interaction with.
When the user releases the grab button, the script goes into the TOUCHING state, and
when it leaves the collider it returns into EMPTY state.

3.4
Expanded System
After the first prototype was made and tested, it became clear that we are lacking
some important functionality to have a modelling system with which we could ac-
tually create something, simple as it might be, but something that could be used
nevertheless. After some consideration and weighing what will be achievable within
the little amount of time left, several functions were decided to be added. They are:
• Extrusion of multiple faces.
• Movement of vertices.
• Scaling of faces along the normal when extruding multiple, instead of just mov-
ing.
3.4.1
User Interface
For the new functionality that we want to introduce, it is obvious that new UI ele-
ments had to be developed, or the old ones re-adjusted. Starting off with the extrusion
of multiple faces, we already have a well working mechanism for extruding a single
face. The trick would be to find a way to bind two or more faces together to utilise
the same handle already created for the prototype.
As further described in chapter 4, it was decided to go with torus shaped handles,
since they performed better than simple balls. Now we needed a mechanism that
would allow us to link multiple faces together so they could be extruded simulta-
neously. An idea emerged from remembering latches that could be used to lock to
objects together, like on the front of a suitcase. So each pair of faces gets a latch on the
edge between them, as seen on figure 3.5. Upon activation, the latch turns signalling
that it is closed, and for easier recognition by the user, it also turns red as can be seen
on figure 3.6. Now whenever the user grabs a handle and pulls it out, if the face is
locked with any other face, all of the locked faces will be extruded and moved together.
For the controls to move a vertex around, it was useful to remember a of a way to
connect things on a cork board. We use pins and strings when we want to connect
two points on the board and in our case we could imagine the edges of the mesh to

two faces together and pins for moving vertices.
be our strings and the vertices to be the dots that we pin down.
Finally, the scaling of faces along their normals. That one was fairly tricky to make
because our UI is already getting pretty cluttered when it comes to the manipulation
of faces. We already have the extrusion, movement and both those actions but on
multiple faces with the help of latches. So creating another set of controls would
overcrowd the UI to the point when it is not usable.
Furthermore, having some
external way of switching the modes (for example, by a pressing a button, either a
one on the Oculus Touch controllers or one placed inside the scene) goes against the
general idea of embedded UI we are sticking to. The turning point here was relating to
a movement we make when we try to stretch a piece of fabric, or a piece of plasticine,
or playing an accordion, i.e. we grab onto two edges of the object and start pulling
on both ends. Thus, we only needed to modify the behaviour of the face handles.
Now, when the user is grabbing a handle with one of his hands and he tries to grab
onto another handle with his other hand, if this handle’s associated face is linked to
the face user is already grabbing onto, the movement mode changes and instead of
following the user’s hand, the faces start scaling along their normals according to the
distance to the user’s hands. So if a user moves his hands away from the faces, the
faces expand, and if he moves them inside the object, the faces start to shrink.

3.4.2
Architecture
Sticking to the architecture chosen for the first prototype, we expand it to include
two new Controllers: LatchController and VertexController.
Thanks to the
IInteractableObject interface, in order to create a new interactable object, cre-
ating a new Controller script implementing IInteractableObject is all that it takes.
The hands have no idea what they are interacting with, thus, everything that the new
control should do goes inside the Controller. As can be seen in figure 3.7, besides the
two new Controllers we introduced a ControlsManager.
The reason for it is that generating and updating all the controls on the Mesh was ex-
panding too rapidly, thus making it a pain to maintain it within the ExtrudableMesh,
which already has a lot of responsibilities of its own. Thus, it was decided to del-
egate generating, setting up and updating the controls to a separate component
ControlsManager. It holds the reference to ExtrudableMesh which it passes through
onto Controllers when creating them, which allows the Controllers to interact
with the ExtrudableMesh directly, as they were designed to.

CHAPTER 4
Tests and Results
Testing of the developed system is a fairly difficult task. There are many things that
could be looked into, from performance to how user-friendly it is.
Unfortunately,
there are some serious limitations to what we could test, imposed by the accessibility
of equipment (it is not possible just to send a build to somebody, since Oculus Rift
is still not really wide-spread, so it must be arranged at a pre-designed place) and
the amount of people who could be collected into a focus group within the time limit.
With that in mind, both the prototype and the expanded system were tested to a
certain extent.
4.1
Comparison of affordances-based interface with
tool-based interface
The prototype (for it’s capabilities, see chapter 3.3.3) was developed to see if the idea
of creating an embedded UI, that will provide the object with affordances that the
user could easily percieve and use, was feasible. For that reason, we decided to test it
against an already existing system, that uses a tool-based approach to user interface,
i.e. Google Blocks [Goo]. It is a pretty powerful modelling tool and even though it
is designed for entertainment more than professional use, it’s capabilities are wide
enough. More importantly, they include face extrusion, which is the corner stone of
our prototype and the function that we will be testing against.
4.1.1
Test setup
For the testing a group of three people was gathered, who had never done any 3D
modelling before. The reason we are satisfied with such a small focus group is that
at this moment we are not so much trying to figure out whether the UI created is
optimal and most efficient (for that we would have to gather a significantly larger
group and present them with a wide range of different shapes to find the one that
works best as a signifier), but rather see if it is competitive enough with the existing
UI solution under a blind test. Since all the people it was tested on, had no experience
in 3D modelling, their performance was not going to be influenced by past experience
of working with some 3D modelling program.
==PAGE:32==
22
4 Tests and Results
As briefly mentioned above, we are going to put the subjects through a blind test,
meaning the subject has never engaged neither system before. That would allow us
to measure how long does it take for the user to find the function (which accounts
to intuitiveness of the system) and how fast is he at using this function to model
something (that would account to convenience of the system). The test would be
following this protocol:
1. A single subject is picked from the focus group.
2. The subject is explained what the extrusion operation is.
3. The subject is asked to find the way to extrude a face in the system he is
presented and after he found how to extrude faces - make a letter N just by
extruding faces of a cube.
4. The subject is then presented with three different modelling systems(Google
Blocks, Prototype with sphere shaped handles, Prototype with torus shaped
handles) chosen randomly, one after another.
5. The subject is recorded throughout all of the three systems for further analysis.
6. After the test, the subject is asked to share his opinions on two questions: Which
way of extruding they prefer? Which handle type in the Prototype worked better
for them, i.e. more natural and easier to use?
4.1.2
Test results
After the tests have been carried out, the time has come to review and analyse the
outcomes. As presented in figure 4.1, it is clear that the extrusion function is sig-
nificantly easier discovered in our developed prototype compared to Google Blocks’
tools-based UI. It is obvious that unlike the prototype created, Google Blocks has
many more functions in it and extrusion is not the main focus of it. Nevertheless,
this shows that in the prototype we created it is very natural for the user to realise
what he is supposed to do.
What is also very interesting to see (from figure 4.2) is that as soon as the user learns
how to use the function, extruding comes very natural to him, no matter if it is by
using a tool attached to his hand, as in Google Blocks, or by using his hands as is like
in our prototype. That effectively means, that both UI types are comparably effective.
Which in turn means that our embedded UI based on affordances and signifiers is
certainly a viable way of creating the UI for VR.
On the other hand, when the testers were asked to choose which method of interac-
tion they preferred, no definite answer could be drawn. They were uncertain whether
one was better than the other. Though it was not the purpose of the tests, it is still

CHAPTER 5
Discussion and
Conclusion
5.1
Conclusions
This project came a long way from the concept to the final system. We effectively
managed to show that the idea of creating UI using affordances and signifiers is
achievable and viable for the VR world, which was our main goal. It is clear that the
embedded UI has an advantage over both tool-based and traditional UIs in conveying
the available actions to the user. That comes from the fact that our way of making
the UI draws from the everyday experience by using real world familiar signifiers to
communicate with the user.
The final expanded system, even with its limited functionality, with a little bit of
imagination from the user proved to be sufficient for creating simple models. Fur-
thermore, with the architecture chosen, the system is easily expandable, whether it
comes to adding a new UI element, or replacing some parts of the system making it
compatible with other input devices like HTC Vive, or swapping the underlying DLL
library with another processing engine.
5.2
Future work
However the developed system is usable and proves the viability of embedded UI,
there is still a lot of room for improvement. There are multiple directions in which
the future work could be conducted, some of them are:
• Improve the performance and looks of existing system. The system can
be optimised to perform faster and smoother in many ways, from tweaking the
GEL library to finding a more efficient way of rendering the mesh in real time,
or fitting more accurate colliders for hands and interactable objects. Moreover,
some work can be done on the overall looks of the system, from setting up a
better scene to improving the models for controls.
==PAGE:38==
28
5 Discussion and Conclusion
• Expand the functionality of the system. There is a plethora of functions
that were discarded for this project, due to either their complicated nature, or
them being non vital for the minimal prototype. Some examples of such func-
tions could be: ways of creating new objects, a possibility of merging together
parts of the object, ways to import/export the models, etc. A challenge here
would be to stay true to the idea of making UI based upon affordances and
signifiers, however, some deviation might be required, for example, creating a
control panel of sorts for external functions like import/export.
• Analyse the performance of different signifiers. The signifiers created for
this project are just few of many. There are a lot of other ways the affordances
could be conveyed to the user. One of such ways have been shown in chapter 4
with different objects used as handles. It will be definitely interesting to find out
the correlation between different signifiers and user performance or intuitiveness
of the UI.
• Exhaustive comparison analysis.
As interesting as testing different sig-
nifiers, it is interesting to perform a comprehensive analysis comparing the
developed system with embedded UI to several other systems with different
approaches to UI which could reveal advantages and drawbacks of the new
approach.
