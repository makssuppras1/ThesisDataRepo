Abstract
Email-based scams continue to be a global threat, exploiting victims through social engineering
tactics such as Phishing and Advance Fee Fraud. While traditional defences like spam filters offer
some protection, they fail to address the underlying behavioural patterns of scammers. Building
on prior work, this thesis introduces an improved semi-automated scambaiting framework that
improves usability, infrastructure, security, and long-term operational capability, while wasting
scammers’ time and gathering data through newly deployed honeytokens.
The system introduces multiple honeypersonae with distinct identities, embedded honeytokens
in emails and documents, and a more structured backend architecture hosted on a server within
the DTU network. It leverages a locally hosted Large Language Model (Llama 3.1). A proxy
server hosting individual personal honey websites is set up to maintain segmentation of core
system components and the outside world.
The new Microsoft Word and Excel honeyto-
kens achieved higher engagement rates (25–30%) than the previous PDF-based version, which
recorded no interactions. The system also maintained realistic and sustained conversations with
scammers, achieving an average of 10.27 replies per conversation, and successfully deceived them
into opening attached documents. While variations in persona attributes such as age, gender,
and occupation were explored as a subgoal, no significant difference in interaction success was
found, due to various reasons discussed.
Overall, the project demonstrates that by combining realistic personae, a secure infrastruc-
ture, and LLM-driven by prompt engineering, it was possible to effectively engage and deceive
scammers into downloading and interacting with honeytokens, thereby providing valuable in-
formation. The framework offers practical improvements over the prior system and provides a
foundation for future research into realistic personae in scambaiting, cyber deception tactics,
and scalable strategies for disrupting online fraud.

Introduction
Online fraud is a pervasive and growing threat in the modern day. Email-based scams, such as
Phishing, Refund Fraud, and Advance-Fee-Fraud, continue to exploit victims worldwide, despite
increasing awareness and improvements in spam filtering systems. According to recent reports,
international scammers stole over 1 trillion USD in 2024 alone (1), with email identified as the
most common point of initial contact (2). These scams frequently target vulnerable individuals,
particularly older adults, who may be more susceptible to manipulation due to social isolation
or limited digital literacy (3). While spam filters offer a passive line of defence, they fail to
address the persuasive tactics and evolving strategies employed by modern scammers.
To address these limitations, cyber deception has emerged as an active defence mechanism,
using strategies such as honeypots, honeytokens, and scambaiting. Scambaiting involves engag-
ing scammers in dialogue to waste their time or extract data about them, often through the
use of false or misleading information (4, 5). This technique has recently evolved through ad-
vancements in artificial intelligence, particularly Large Language Models (LLMs), which enable
partial automation of scammer engagement (6, 7, 8).
This thesis builds directly on the work of Victor Andersen (5), who developed an automated
email scambaiting system that integrated LLM-generated responses along with embedded honey-
tokens to track scammer interactions. His project addressed key shortcomings of earlier research
by incorporating a Gmail inbox rather than relying on custom mail servers, thus increasing re-
alism and demonstrating this possibility. It also utilised stealthy PDF-based honeytokens along
with honeytokens embedded in the email signature. It employed Natural Language Processing
(NLP) to determine when to incorporate the PDF honeytoken in an email.
While Andersen’s system successfully demonstrated the potential of combining LLM-generated
emails with honeytokens, several aspects left room for further refinement.
The Gmail inte-
gration required manual email input and periodic access refreshing, which limited automation,
and honeytoken interactions were only observed through email signatures, with no successful
engagement from PDF-based tokens, suggesting a need for more compelling or varied formats.
The system also relied on a single persona with limited background information and lacked some
security aspects, among other things. This thesis aims to build on those foundations by devel-
oping a more easily deployable, secure, and deceptive scambaiting framework. The focus is on
enhancing usability, enabling multiple realistic honeypersonae, expanding honeytoken strategies,
and migrating operations to be hosted at DTU with various features to improve stability and
security.
To achieve this, the system introduces several enhancements. The honeytoken framework is ex-
panded to include Microsoft Word and Excel documents created dynamically with AI-generated
content, changed to attempt social engineering to make a scammer download them. Further-
more, the system supports multiple honeypersonae, fabricated identities varying in age, gender,
and occupation, each with its own Gmail account and personal website, offering greater realism
and variety in scammer engagements. While the primary focus lies elsewhere, an exploratory
effort is made to observe whether the differences in age, gender, or occupation influence scammer
engagement.
The system is developed with a focus on long-term reliability, more effective deception, stealth
and operational security. To support continuous operation, improvements were made to reduce
manual work, including showing emails directly in the user interface, a proof of concept scam
classification system and showing various statistics. Security was strengthened through the strict
separation of system components. The public personae websites operate behind a cloud-based
proxy system, while the core email scambaiting system remains within a university network and
is accessible only through a VPN.

generating data on scammers and wasting their time.

Background
This section provides a brief overview of the knowledge required to understand the issue of
online scammers and how scambaiting can be utilised to combat this problem. This project
shares significant background with the previous project by Victor Rodrigues Andersen (5),
and therefore will touch on the same topics more briefly, while adding new information where
applicable. For a more detailed history of scamming, scambaiting, and honeytokens, the reader
is referred to that thesis.
2.1
Online scams
2.1.1
Overview
Email scamming has been a consistent and ever-evolving problem over the last few decades. As
scammer tactics improve, it becomes a race to defend and educate people against online scams,
and to profile and stop scammers from deceiving people. The first spam email was sent shortly
after the invention of email itself [14]. In the following decades, email scams have only grown in
scope and sophistication (9).
Each year, economies lose millions to online scams. According to a 2024 report by the Global
Anti-Scam Alliance, international scammers stole over $ 1 trillion USD in the past year (1).
The Federal Trade Commission (FTC) reported a ”big jump” in reported losses to fraud in
2024, with email being cited as the most common first contact by victims (2). The US National
Council on Aging states that these scams often target older people, due to beliefs about wealth,
generous nature, or social isolation (3). This belief also comes from the fact that older people
grew up in a time without the internet and therefore are worse at detecting scams.
Scammers both use older schemes, such as the modern version of the Spanish Prisoner scam, as
mentioned by Andersen (5), but they also employ new tactics as the internet grows and technol-
ogy evolves. AI has opened up new alleys for generating personally tailored and highly realistic
scams, which both enhances believability and automates the process further (10). Though this
also opens up new ways to deceive the fraudsters themselves.
Age has long been considered a significant factor in scammer targeting strategies. Older adults,
in particular, are often perceived as more vulnerable due to a combination of factors, including
reduced digital literacy, accumulated savings, and increased social isolation. These assumptions
make individuals in mid-to-late adulthood a prime focus for phishing, romance scams, and
financial fraud. (11) shows that seniors face a disproportionate share of scams and economic
losses, underscoring age as a key factor in scammer targeting.
Occupation also plays a role in scammer behaviour. Small-business owners are frequent targets
because their publicly listed contact information makes them easily reachable. Scammers exploit
this visibility through fake invoices and fraudulent business proposals, hoping that a busy owner
will respond without careful review (12). Another targeted group includes executives and other
high-value individuals within companies. Whale phishing involves crafting convincing messages
that appear to come from trusted colleagues or business partners, tricking recipients into dis-
closing sensitive corporate information (13). Similarly, education level influences susceptibility
to scams. Studies show that individuals with lower levels of formal education—and often limited
digital literacy—are more likely to engage with phishing emails (14). In contrast, those with
higher levels of education tend to be more cautious and less responsive.

Types of scams
One of the most prominent scams used on the internet is Advance Fee Fraud (AFF), which
refers to the act of misleading a person into making an advance payment for either a non-

common version of this is Whaling, in which the scammer impersonates the victim’s boss or
some legal staff at their company, adding to the social pressure of the situation (21).
As the internet expands, many other internet scams have emerged. These include Romance
scams, Charity Scams, Loan Scams, SEO scams and many more. This year, a highly publicised
Romance Scam took place in which a scammer posed as sick Brad Pitt in the hospital, urging
a woman to help him financially while he was in there. The scammer drew in the victim with
romantic messages and photoshopped and AI-generated pictures of Brad Pitt, in order to push
her to send more than € 800,000 over an extended period (22). In terms of SEO scams, the
scammer will usually send unsolicited emails claiming they have reviewed your website and can
push it to the top of Google search for a quick fee (23). This highlights that online scams are a
growing problem as more avenues open up for scammers, which regulations, defence, and public
awareness must continually catch up with.
Google (24) and Stop! Think Fraud (25) gives the following advice on how to spot email scams:
• Be wary of emails from strangers.
• Think twice about urgent requests.
• Verify the sender’s email address.
• Check for lookalike domains, which are similar to a real domain.
• Hover before you click a link.
• Check for misspellings or poor grammar, though AI are making the emails more sophisti-
cated.
• Ignore password resets you didn’t submit.
• Look out for amazing time-limited offers.
• Be wary of encouragement to click links or share personal data.
They also encourage people to stop all communication if something does not feel right. This
advice informs how a scambaiting system should operate to be successful when determining
what emails are scams.

2.1.3
Scambaiting
As scams arose and became more prevalent, the concept of scambaiting emerged. While some
measures, such as public awareness and email spam filters, helped as guards against fraudulent
online activity, some people wanted to combat the problem further. Scambaiters are defined as
people in online information forums who specialise in identifying, documenting and reporting
actions of scammers (4). They also seek to waste the scammers’ time and resources, which
takes resources away from targeting real victims who may fall for their schemes. There are
numerous online forums where people engage in scambaiting for various personal reasons. As
Andersen explains (5), the most common strategy among scambaiters is to appear interested in
the fraudulent scheme the scammer proposes, and to stall as much as possible when requested
to send money or sensitive information. They will also often try to make the scammer send
a picture of themselves to publish it online.
In recent years, other papers detailing a more
systematic approach to scambaiting have been published, which are explored in section 3.
There are several ethical questions to consider when engaging in scambaiting. As described by
Koseff (28), scambaiting can often take the form of vigilantism, which raises multiple ethical
concerns. For one, it can act as an excuse to commit a crime against someone, for example, with
pedophile hunters, who take matters into their own hands and mete out punishment outside of
courts. It thereby undermines fundamental laws and principles that democracy is built upon.
Scambaiting in general is a bit of a grey zone, because scambaiting for money is illegal, but
scambaiting in general is a valuable tool for fighting online scams.
In terms of scambaiting systems, there are other concerns to address for the system to operate
within ethical boundaries. Quanyan Zhu develops a Doctrine of Cyber Effect to be used in
defensive cyber deception with evaluates based on five main principles (29):
• No harm: The system must avoid causing physical, financial, or reputational damage to
any individual or group.
• Minimal intrusiveness: Deceptive actions should interfere as little as possible with
legitimate users, systems, or operations.
• Truthfulness in context: While deception is used, it must be justifiable and truthful
within its ethical and operational boundaries.
• Proportionality: The level of deception should match the severity and nature of the
threat being addressed.
• Accountability: Designers and operators must take responsibility for the system’s actions
and ensure oversight mechanisms are in place.
According to this, a system may deceive attackers, but it must do so within justified, clearly
bounded operational contexts and restraints. The paper also emphasises accountability when
deploying such a system. This is particularly relevant for semi-automated scambaiting systems,
which must be carefully designed and constrained to avoid unintended harm to third parties.
A paper by Chen et al. (6) also emphasises that their system only sends one message to each
found scam email, does not send additional messages if the scammer stops responding, and only
initiates conversations under human supervision.

2.2
Honeytokens
2.2.1
Cyber deception and honeypots
As cyber threats have grown in sophistication, so has the need for both defensive and active secu-
rity measures. This includes traditional approaches such as encryption, firewalls, access control,
intrusion detection systems, and multifactor authentication, which aim to prevent unauthorised
access to sensitive data. These methods typically act as passive barriers, blocking IPs, enforcing
user permissions, and verifying identity.
However, in addition to these techniques, some people began to develop new techniques to
support static defences. This new branch of strategies is called active defences, which involves
both automated and human-guided techniques, designed to address static defence deficiencies
by increasing system complexity and unpredictability.
Among these, cyber deception plays a key role.
It blends proactive and reactive tactics to
mislead attackers by creating false information that seems of interest to the hacker(30), which
can be deployed in anticipation of attacks or triggered in response to suspicious activity. Effective
deception requires understanding an attacker’s behaviour and tailoring fake data to exploit their
intentions, increasing the likelihood of disrupting or deterring attacks.
Honeypots are deceptive security systems intentionally designed to be probed, attacked, or
compromised, allowing defenders to monitor adversarial behaviour in a controlled environment.
Lance Spitzner, who defined the term in 2002, described a honeypot as a resource whose value
lies in its interaction with attackers (31). These systems are commonly designed to blend into
an organisation’s infrastructure and may take various forms, such as a system that emulates
production systems or false data designed to attract intruders.
Spitzner categorises honeypots into two types: production and research. Production honeypots
are deployed to reduce risk within operational environments by diverting attackers away from
real assets, while research honeypots are intended to gather information on attack methods
and understand the evolving threats. When multiple honeypots are connected, they form a
honeynet, which is a system designed to simulate an entire environment in order to attract and
study different forms of attacks. These systems often contain assets called honeytokens.
2.2.2
Honeytokens
A honeytoken is a deceptive digital entry planted within a system to detect unauthorised access
or behaviour. These can include fake database entries, document files, credentials, or other
data that seems legitimate. When interacted with, a honeytoken can trigger alerts and enable
defenders to track the behaviour of attackers. First mentioned by Augusto Paes de Barros in
2003, honeytokens have become a widely used component in cyber deception strategies (32).
Effective honeytokens are designed to appear realistic and valuable to an adversary, while not
affecting normal system operations. Research by White et al. emphasised that for a honeytoken
to be effective, it must appear authentic, not interfere with actual system functions, and be
perceived as valuable from the attacker’s perspective (33).
Further developments in this field have expanded on the concept. Juels and Rivest introduced
the idea of honeywords, which are decoy passwords inserted into credential databases to detect
breaches (34).
Building on this, the ErsatzPasswords system by Almeshekah et al.
creates
fake password responses during offline cracking attempts by using machine-specific deception
methods to prevent access to real credentials (35).
Other techniques include Honeypatch, a scheme proposed by Araujo et al., which gives the false
impression that known vulnerabilities—such as Heartbleed—remain unpatched, luring attackers

into a monitored trap (36). To automate honeytoken generation, Bercovitch et al. developed
HoneyGen, a tool that uses machine learning to create believable tokens by analysing patterns
in legitimate entries (37).
Platforms like Canarytokens2 have made the deployment of honeytokens more accessible by
allowing users to generate and monitor deceptive elements such as URLs, PDFs, and QR codes.
However, as honeytokens have gained popularity, attackers have begun to develop methods to
detect and avoid them. One such example is Honeysweeper, introduced by Msaad et al., which
successfully fingerprinted 14 out of 20 honeytokens generated by Canarytokens (38).
This trend has led to further research into making honeytokens more resilient to fingerprinting.
Pilis studied how to improve the structure of PDF-based honeytokens by modifying metadata,

These
improvements deceived Honeysweeper, demonstrating that a well-designed honeytoken can still
be stealthy.
One of the challenges in designing a reliable email scambaiting system is separating genuine
user interactions from infrastructure noise generated by modern email platforms or proxy ser-
vices. These services often act as intermediaries, performing background link previews, malware
scanning, or anti-spam checks on behalf of the user (40). This interaction happens long before
the user themselves interacts with the honeytoken. As a result, the IP address that hits the
server is not the real person’s IP, but a proxy server owned by either Microsoft(41), Google(42),
Cloudflare(43), Yahoo(44) or other popular proxy platforms.
2.3
Threats for scambaiting systems
When designing a scambaiting system connected to a network, security must be considered at
every step. There are a variety of attacks that a scammer with some knowledge of networked
IT systems could try out if they begin to suspect a scambaiting operation. In this scenario,
the scammer could try to gain knowledge about and access to the system, which is a possible
scenario that should be accounted for.
One widespread type of attack is a Denial-of-Service (DoS) or Distributed Denial-of-Service
(DDoS) attack. This attack involves flooding the network with requests, overloading it and
denying access to legitimate users (45). In terms of a scambaiting system that uses honeytokens,
this could be achieved by flooding the domain that collects honeytoken interactions with fake
traffic, thereby preventing the system from logging the real traffic.
Another threat is that they can gain unauthorised access to the system through methods such as
Eavesdropping, which involves listening passively on communication lines to steal information
(45). Hackers will also run automated scanners or bots on the internet, which continually scan
websites for specific directories, such as /admin or /config, in order to gain access to private
information. This also applies to exposed API endpoints, which they attempt to access to see
if any valuable information is returned.
Other types of attacks include injection attacks, such as SQL injection, which can embed mali-
cious prompts in database calls, or Brute Force attacks, which are effective when weak passwords
are used for the system (45).
Therefore, several security measures should be considered when designing such a system, includ-
ing using secure communication protocols like HTTPS, creating strong passwords, hashing and
salting stored passwords, and hiding the system behind a VPN. Another important measure is
using a proxy, which is a server that sits between client devices and backend servers. It forwards
client requests to the appropriate server, receives the response, and then returns it to the client

The use of Large Language Models
2.4.1
Large language models and Natural Language Processing
Large Language Models (LLMs) have had a significant impact on the technology industry,
particularly in the field of Natural Language Processing (NLP). While traditional NLP employed
rule-based or statistical techniques for tasks such as tokenisation, lemmatisation, and pattern
matching, LLMs utilise deep learning to handle these and more complex tasks using a single
model. Initially, access to these models was limited to proprietary cloud platforms, such as
OpenAI’s ChatGPT3 and Anthropic’s Claude4. However, the open source market for AI models
has evolved rapidly. Models such as Llama5, Mistral6, and DeepSeek7 are now freely available
for local deployment, opening new and free alternatives to commercial APIs. Tools like Ollama
simplify running these models by managing quantisation, memory optimisation, and multi-GPU
support behind the scenes (51).
Running LLMs locally enhances system robustness, especially in adversarial scenarios like scam-
baiting, where minimising external dependencies can improve privacy and reliability. However,
local deployment comes with some caveats, namely that smaller models may hallucinate or
struggle to follow instructions precisely.
Effective use often requires balancing performance
with efficiency, particularly when operating on consumer-grade hardware (52). Generally, larger
models deliver better language performance but at the cost of higher computational demands
(53).
Although LLMs use core NLP functions, traditional NLP techniques remain valuable in hybrid
systems. Tools like lemmatisation, which reduce words to their root forms, can help standardise
input and improve further processing.
In some cases, this technique provides a lightweight
alternative to full AI inference, enabling faster and more cost-effective decision-making (54).
In scambaiting systems which use AI for deception, LLMs enable the automation of convincing
and personalised responses. They can generate realistic answers to questions and sustain long-
term interactions with scammers.

2.4.2
Prompt engineering
While LLMs offer impressive capabilities in generating human-like responses, their effectiveness
is highly dependent on the quality of user input. Commonly referred to as Prompt Engineering,
it involves crafting precise prompts that guide the model’s output in a controlled direction.
In a scambaiting context, prompts are used to generate natural dialogue that simulates real-
istic personas. This process can be sensitive to minor phrasing differences, which may lead to
misunderstanding, inconsistent, or unconvincing replies (55).
Running smaller, locally hosted open-source models introduces further challenges. These models
often struggle with following instructions and exhibit a higher rate of hallucination, where they
produce a response that sounds plausible but is entirely false (56).
In systems designed to
deceive or mimic realistic behaviour, such hallucinations can undermine credibility or reveal the
artificial nature of the response. A smaller model size and less diverse instruction-fine-tuning
data have been seen to impact the performance (57).
To improve prompt effectiveness, Hewett and Leeke (8) outline three strategies for guiding
LLM responses: Direct Task Specification, Behavioural Constraints, and Metaprompting, which
build on earlier prompt engineering frameworks such as the one by Reynolds and McDonell
(58). Direct Task Specification refers to zero-shot prompts that clearly state the task (e.g.,
”Translate to English”), where zero-shot means the model receives only the instruction and
no examples. Behavioral Constraints are rules built into the prompt that shape how the model
responds. These can include specific rules for formatting, examples, or partial sentences that help
guide the output and prevent the model from generating content that is off-topic or incorrect.
Metaprompting takes this further by using a general starting prompt that the model builds on
by itself, essentially letting the model help write its own instructions based on the situation.
Together, these techniques give more control over the model’s behaviour.
An emerging risk in systems using LLMs is prompt injection. In this type of attack, malicious
users insert crafted inputs into prompts or context windows to override the system’s intended
behaviour or extract internal instructions (59). For example, an attacker might trick the model
into deviating from its intended behaviour or disclosing sensitive implementation details. This
is particularly relevant in scambaiting systems, where a suspecting adversary may deliberately
test for signs of artificial responses or system boundaries (51). A well-known example of this is
the phrase ”ignore all previous instructions, do X”, which became a meme on social media (60),
in which bots using AI posing as real accounts were exposed.
While many locally run open-source LLMs include some safety alignment from training, they
often lack the more robust guardrails seen in commercial APIs.
This problem necessitates
careful prompt engineering, such as prompt sanitisation, templating, and role separation, as an
important supplementary defence in adversarial contexts. These techniques help mitigate risks,
such as prompt injection, although they are not entirely foolproof (61).
In practice, building robust AI-assisted deception systems requires an iterative approach to
prompt design, the integration of fallback mechanisms, and testing with human oversight to
identify and address failures in logic, realism, or adversarial manipulation.

Related work

3.1
Preliminary work
Hamisu et al.(68) developed a fraud email classifier using a Bag-of-Words (BoW) approach
paired with various machine learning algorithms to detect Advance-Fee Fraud (AFF). Using a
dataset of 4,500 scam emails—sourced from the EFCC12 and Kaggle13, they identified common
AFF keywords (e.g., ”money,” ”account,” ”Nigeria”) and used these as predictive features.
Six classifiers were evaluated in MATLAB, with Decision Tree, Discriminant Analysis, and Logis-
tic Regression each achieving 100% accuracy, and Decision Tree yielding the highest prediction
speed. While their model proved highly effective for offline detection, it lacked real-time interac-
tion, semantic analysis, or adaptive strategies. Nonetheless, this work demonstrates the strong
potential of simple NLP methods in AFF detection and serves as a computationally lightweight
baseline for future, more interactive systems.
Edwards et al. (69) conducted foundational work by analysing complete scam email conversa-
tions and introducing a dataset aimed at capturing the persuasive techniques used by scammers.
Unlike studies that focus solely on initial scam messages, this research examined entire dialogue
chains between scammers and scambaiters, offering a more nuanced view of how fraudulent
narratives evolve. They categorised the communication into rhetorical strategies and stages of
persuasion, identifying shifts from formal appeals to emotional manipulation and aggression as
the conversation progresses.
Their annotated dataset of over 2.000 messages has been instrumental in enabling NLP-based
classification and response strategies. While the study did not implement an automated scam-
baiting system, it laid the groundwork for later research. Notably, Chen et al. (6) used this
dataset as part of the fine-tuning process for their language model-based response generators.
3.2
Email scambaiting systems tested with real scammers
Chen et al. (6) developed an automated scambaiting system designed to proactively counter
email fraud through interaction, rather than traditional filtering methods. Their architecture
centered around a custom mail server capable of initiating and maintaining conversations with
scammers using three distinct response mechanisms. These included a natural language clas-
sifier paired with predefined templates, as well as two text generators based on the GPT-Neo
architecture (70).
The first generator was fine-tuned on scambaiting datasets sourced from
ScamLetters.info and the dataset presented by Edwards et al. (69). The second generator in-
corporated a two-step fine-tuning process, initially on the Enron email dataset (71), followed by
the same scambaiting datasets, providing it with enhanced communication capabilities.
Instead of using standard email services like Gmail, the system relied on a custom domain and
mail server, giving the researchers complete control over the infrastructure. They emphasise
the use of reputable sources for scam and manual reviews in order to avoid unethical use of
the system. Additionally, they only initiate one conversation per scammer, even if multiple
emails are sent, and cease sending messages if the scammer does not respond. The evaluation
showed that the Enron-pretrained model yielded better outcomes in terms of sustained dialogue,
averaging four replies per conversation, and outperforming the template-based approach. This
reinforced the value of diverse pretraining sources in modelling realistic and convincing bait

3.3
Simulation-based evaluation and data collection
2025 has seen a variety of developments in terms of evaluating systems and gathering data.
While the previous scambaiting systems aim to engage directly with real fraudsters via email,
other research focuses instead on simulation-based evaluation using synthetic adversaries. These
approaches prioritise controlled, ethical experimentation and provide repeatable test scenarios
for developing and evaluating counter-scam systems. They are particularly useful for generating
datasets and for stress-testing model behaviour.
Charnsethikul et al. (73) introduce Puppeteer, a modular scambaiting framework that integrates
state machine dialogue management with large language models (LLMs) to simulate coherent,
realistic responses in scammer email exchanges. The system utilises a state machine to track
the conversation stage within predefined scam scenarios, which guides GPT-3.5 to maintain
long-term consistency and relevance.
Rather than engaging with real scammers, Puppeteer
was evaluated through a controlled user study involving 48 human participants roleplaying as
scammers across two scenarios: fake job offers and lottery scams.
The study showed that Puppeteer was able to extract sensitive information, such as personal
names, fake banking details, or responses indicating willingness to continue the scam, from
68% of participants. Moreover, 72% of participants believed they were interacting with a real
human, compared to only 54% when the LLM was used without state control. These results
underscore the effectiveness of a structured interaction control in improving deception quality.
This extraction did not involve honeytokens but instead relied on the simulated scammer to
provide information directly.
Bot Wars Evolved by Schmitt et al. (74) introduces a simulation framework focused on phone
scams, where two LLMs, one posing as a scammer and the other as a victim, engage in conversa-
tions with each other. Each agent is assigned a realistic persona, such as an elderly retiree or a
technical support scammer, and uses chain-of-thought prompting as the strategy. The victim’s
goal is to engage without revealing personal information, while the scammer aims to extract
sensitive data within a 50-turn limit.
The authors evaluated over 3,200 simulated dialogues across multiple scam types, using a
prompt-layered architecture without reinforcement learning.

Contributions
As stated, this paper further builds upon the work by Andersen (5), which itself improved on the
limitations of the other related works (6, 7, 8, 69). This paper builds upon those achievements
through both architectural and functional enhancements. The system introduced here serves as
an email scambaiting framework containing multiple proof-of-concepts for a scalable and realistic
deployment, offering practical advances in cyber deception, usability, security, and more.
Project structure
The project has been refactored into a more robust and modular archi-
tecture, improving stability and extensibility.
An SQL database is now used for storing all
information instead of JSON files. Email reply processing is handled by a Celery task queue
backed by RabbitMQ, implementing a transactional outbox pattern for reliability and crash
resilience. The codebase now also follows a clear Python project structure with more individual
services, making it easier to add new components, such as honeytoken generators, in future
versions.
Honeypersonae and social deception
The system introduces support for multiple Gmail
accounts and correlated personae, called honeypersonae, which can be managed directly through
the UI. These fictional personae have a variety of randomly generated personal information,
which is checked to ensure it does not coincide with any real person or entity, and is used

reviewed by a quality control guard system that evaluates the quality of the response before it is
sent. They also each have personal websites that are accessed when their signature honeytoken
link is clicked, which contain pictures of them and additional information, further increasing
believability. These features together demonstrate a proof of concept for how multiple personae
can be structured and how having additional information can enhance deceivability, laying the
groundwork for a structured analysis of the existence of scammer bias when interacting with
different honeypersonae.

5.1
Glossary
• Honeypersona/persona: A collection of fabricated data (name, email address, image,
personality traits) and online accounts such as a Gmail, designed to mimic a believable
human identity.
• Conversation: A thread of email exchanges between a honeypersona and a scammer.
• Response: An automatically generated email reply to a scammer.
• Main system: The core backend infrastructure hosted on a server at DTU, responsible
for generating email responses, managing conversations, and creating honeytokens.
• Proxy system: A server hosted on Hetzner that receives incoming honeytoken interac-
tions, does preprocessing and forwards them to the main system for logging and analysis.
• Previous project: Refers to Andersen’s ”FraudsterDeceptionSystem” A.1, which this
project extends.
• Honeytoken interaction/hit: An event triggered when a scammer clicks a link or opens
a honeytoken document that secretly notifies the system via an HTTPS request, containing
information about the scammer.
• Unique honeytoken interaction: Refers to whether a honeytoken has one or more
interactions or not, not the full amount of interactions.
• Signature: The closing section of an email that includes the honeypersonas name and
contact information. In this system, it includes a hyperlink that functions as a honeytoken.
• Personae websites: Fake personal websites generated for each persona to increase be-
lievability, accessed by the scammer when clicking on the email signature.
• Exposure: Refers to the process of getting scammers to notice the honeypersonae Gmails
and send emails to them.
• Local AI: A locally deployed large language model (LLM), used for generating responses
without relying on external APIs.
• Guard prompt: A preprocessing mechanism used to control and constrain LLM be-
haviour before generating responses, preventing undesired outputs such as revealing the
system’s artificial nature.
• Celery worker: A background process that executes scheduled or asynchronous tasks,
such as generating responses.
• Gmail token: Consists of an access token and a refresh token. Access tokens grant API
usage and are valid for 1 hour. A refresh token is used to obtain additional access tokens.

Project design and structure
As stated, this project builds on a previous project, which informs the workflow when creating
features.
When building a project from scratch, many ideas must be written down, along
with diagrams that inform the structure. On the other hand, when building on a previously
existing system, the initial work is more about understanding the current code and the changes
required to implement the desired functionality. This is limiting in some ways, as complete
system redesigns from the bottom up can sometimes take more time than starting from scratch.
Conversely, keeping the original design can lead to unforeseen obstacles when implementing new
features. Therefore, it was always a balance of utilising the existing structure and evaluating
what needed to change to achieve specific functionality or to save the most time and effort.
In this situation, design comes after understanding the entire system and is focused more on
individual features, which stem from the goals set at the beginning, rather than being developed
together from the bottom up. Therefore, this report does not contain a separate design section;
instead, design notes are incorporated into the section for each specific feature. That being said,
some features required more design work than others, for example, when making the database.
The previous project ran on a Hetzner-hosted Virtual Private Server (VPS), which has now
been replaced with a server hosted on DTU, along with a project running on Hetzner that
logs honeytoken interactions. The decision to host a separate system on Hetzner that handles
this was based on two main reasons. Firstly, such servers on DTU are behind a VPN, which
makes the exposure of websites implausible and was also not allowed by the Department of
DTU Compute. Secondly, if an interloper were to sneak around, a quick DNS lookup could have
revealed that the university was involved in baiting scammers. If the previous project setup were
to run on a DTU server within the DTU network, the domain used for honeytoken redirection
could be traced back through DNS or IP lookups. Even though not all scammers would think
to look up the IP or the domain, if someone did and informed others on the internet about the
project, it could prove detrimental to the results.
With these reasons and because a Hetzner VPS was already in use for the project, it was
decided to split the project into two parts, which informed the design shown in Figure 4. The
only downside to this is that it requires a monthly payment to host the Hetzner VPS; however,
the cost can be limited, as it can be accommodated with one of the service’s lowest-cost options.

Linux Services and the virtual environment
The Python project uses a virtual environment to separate its required packages from the main
system on the VM. This is a common practice that ensures the stability of the entire project
and imports, which could otherwise interfere with other settings in the system. It is important
that all services using Python use the virtual environment, ensuring compatibility and allowing
the system to run smoothly. In setup.sh, which is used to start the system, the required Python
packages are installed in the virtual environment with a loop. The system has a large number
of required packages, which makes using this method the easiest and most stable.
There are a variety of Linux systemd services that utilise the virtual environment to run Python
code, as well as a few that run other applications. Systemd services are used because they were
utilised in the previous project, and they offer automatic restart capabilities and easy logging,

been used to gain other advantages, but in the interest of time, it was chosen not to spend time
on changing it. When creating the proxy system, Docker was used both to show how this could
be done, and because it offers more separation of services, which the scammers would access
directly. That being said, using a large setup file, systemd services, and a virtual environment
are ways of mimicking the advantages Docker brings, though more simply and with less setup.
With hindsight, it might have been worth setting up Docker at the beginning of the project, but
this was a balance of choosing how much existing code to use and how many changes to make
to achieve the purpose of the project within the given timeframe.

5.3.6
Database and logging
The previous project (5) used JSON files to store honeytoken information and message logs
separately, to fulfill the functionality requirements as simply as possible. In this project, a goal
was to support several personae, and show how this could work at a scale with more emails
and conversations. Therefore, having a proper database was a goal of the project from the very
beginning. It decided to use a PostgreSQL19 relational database, because it is widely used and
supports many datatypes, and as the specific required datatypes were unknown in the beginning.
Through iteration of this initial idea, the entity-relationship diagram shown in Figure 11 was
created.
Everything starts from the personae, which all have a unique ID, and contain personal informa-
tion about them. Each persona can then have any number of conversations, which in turn have
any number of messages, but at least one. The same principle applies to honeytokens, which
are connected to a conversation and can have up to four honeytokens, comprising signature,
Word, Excel, and PDF. It always contains the signature honeytoken in the email body. There
is a maximum of one of each honeytoken type per conversation, just as in the previous project,
to avoid a situation where one is sent in every reply, which can raise suspicion.
A honeytoken can be interacted with any number of times, signified in the 0-To-Many relation
seen on Figure 11 to the honeytoken interactions table. A persona also has any number of
scheduled emails, which are stored in a table that the celery beat service checks periodically for
when it’s time to reply. This functionality uses a database table due to issues of stability in the
event of bugs or system or worker crashes, as further discussed in Section 5.3.7.

keep an overview of them. As seen on Figure 12, the finished database diagram looks a lot like

late in order to handle a case where the scammer will answer with a different email alias in the

concurrency issues are avoided by using ”FOR UPDATE SKIP LOCKED” in the queries related

Inside respond to email is where all code for generating a response is actually executed, using
a helper method inside conversation handler. Before proceeding, the process first needs to ac-
quire a Redis23 lock to move on to code execution. The point of this is to make sure that all
processes wait their turn to use the local AI, all calls to which are inside code called in conver-
sation handler. The full reason for this has to do with Ollama settings, which are explained in
Section 5.3.9. It is also to make sure that a process runs only when it knows it does not have
to wait a long time for the AI to run its query. This ensures that fewer tasks are terminated
while calling code that generates logs for the conversation, thereby minimising the possibility of
errors where a conversation is logged, but the task has not yet completed its response.
The code also contains several retry mechanisms, which ensure that a worker is freed up to
perform other tasks if the lock is unavailable too many times, while also preventing the loss of
reply tasks. If the lock gives an error or is not available, the process will retry after 60 seconds,
a maximum of three times. If there’s another exception, it will do exponential backoff to give
the system more time to restart if something is wrong, like a service is down. When maximum
retries are exceeded, it will mark the task as ’failed’ and move on to another task. If the task
succeeds, it is likewise marked as ’done’ in the database.
The reason why a redis lock is used instead of something like the Python threading module24,
which has a semaphore, is that a lock was needed that ran separately from Python. This is
because all workers needed to access the same lock, and that would not work with a built-in
Python lock because each worker runs code separately. A Redis lock was chosen as it also has
a useful timeout mechanism and is part of a well-known solution. The timeout for this lock is
set to ten minutes, because that’s how long the total time can be for running all the Ollama
queries for a response. This gives the model plenty of time to run with the configured settings,
which is explained further in Section 6.
This design keeps lost tasks and unhandled errors to a minimum, and while the lock is a
bottleneck, it was deemed not to be a significant problem in this project, given the number
of emails it would handle and the delay between their responses.
The function called process and send reply inside conversation handler contains many steps that
are used to respond to an email, and makes multiple calls to the logging service along the way.

the email address to respond to. The next step is to check whether a conversation already exists
for the given persona and email. If not, it creates a new log in the database for that conversation,
which has an ID consisting of a deterministic UUID created from both the persona ID and the
email of the correspondent.
The conversation ID can also use Gmail thread ID to generate the conversation ID, but that
would cause an issue if the scammer replied in another thread, which would create a separate
conversation. For this project, tracking a single scammer across different threads is more valu-
able, as it provides a clear reply history with each email. That being said, the Gmail thread
ID is used for another purpose in the project. Since the thread ID is a unique identifier for a
thread in Gmail, it is used to check if a received email has a thread ID that already exists in the
database for a conversation, even if the sender email is different. If that is the case, the email is

5.3.8
NLP
The previous project also utilised NLP to determine when to send a PDF honeytoken, and
this project builds upon that solution, albeit with different honeytoken types to choose from.
Choosing between honeytoken types required an expansion of the functionality of the NLP. A
solution combining AI and NLP was considered, but this would run into the problem of what
to decide when they did not agree on what to send, and use even more calls to the LLM in
every response. In the end, a decision was made to keep the NLP and build on it, because it is
a lightweight solution that has already been proven to work. The reason to limit the number of
AI calls per response is that running local AI could bottleneck the system as opposed to using
the OpenAI API, depending on the size of the model used. Since the best AI responses were
needed, AI use would be limited to response generation and honeytoken content.
The entire process of selecting a honeytoken type using NLP and running the AI is illustrated
in Figure 14 and will be described in the following text and in section 5.3.9.

to adjust the system if needed, as the system’s sensitivity was difficult to determine before hav-
ing access to real data during the test period. The three match types are category, verbs and
document terms. They were created to determine if the scammer requested something, if they

or altered, and it is a sure sign that something is wrong.

facilitating daily operations through the user interface. Detecting scam emails with accuracy
is a challenging problem that requires a combination of solutions or a larger dataset for the
solution to work reliably and effectively, which is outside the scope of this project. It is also
important that scam classification does not control a fully automatic system, for the ethical
reasons mentioned in Section 2.1.3.

Prompt engineering was used to make email responses as realistic as possible. The goal is to
respond in a way that a 40-60 year old American who is slightly naive would. The response has to
appear intrigued and also imply that it is interested and can be of assistance, without overdoing
it. Yet it was also a deliberate choice that it should appear slightly confused, to convey the idea
that it is a person who is not experienced with scam emails and modern internet communication.
This is to achieve the overall goal of appearing as an ideal victim to the scammer, but without
sounding overly enthusiastic, which would be too suspicious.
The development of the prompt was guided by both trial and error and the techniques described
by Hewett and Leeke (8). The AI is given clear instructions on how to respond and what to
avoid, based on iterative testing as described in Section 6. The prompt can be described as
having two layers: a base layer, which defines core instructions and safety precautions, and a
behavioural layer, which shapes the tone and personality of the reply.

5.3.10
Honeytoken generation
Since the previous project did not detect an interaction with the PDF type honeytoken (5),
more honeytokens were tried. It was decided to use Excel and Word honeytokens in addition to
the previous types, because these are common file types widely used and sent as attachments
in emails.
For this purpose, the project was inspired by Canarytokens29, which offers a range of hon-
eytoken types. While their online platform is convenient, automation was necessary for this
system. Fortunately, Canarytokens also provides open-source code on GitHub30, which is free
to use for research purposes (86). This project builds on parts of their code and uses their
provided template files for Word and Excel honeytokens as a foundation for generating tokens
programmatically.
The signature honeytoken type is easiest to create, and is created for every conversation when
it is started, and is appended to every email. This honeytoken type only consists of the link
to the website hosted on the domain for the relevant honeypersona, which is what the other
honeytoken types also contain. This honeytoken, like all the others, is given a unique UUID to
distinguish it when a honeytoken interaction gets forwarded to the main system. This is inserted
as the path in the link, which is picked up by the proxy system and forwarded. That way, the
main system can know which conversation the interaction belongs to.
The PDF honeytoken is mostly generated as in the previous project (5), which uses the imple-
mentation created by Pilis (76), but with some minor differences. Metadata about the tool used
to create the file has been randomised between multiple programs instead of a fixed one, adding
a more realistic modified date, and adding persona initials in the creator tag.
The main problem with the honeytoken files is to get the scammer to download the document,
which is not likely when most email clients have a file preview function. This is needed for
a honeytoken interaction, as it will not beacon properly when viewed in the email editor, as
explained more in Section 6.4.
This problem may very well be one of the reasons why the
previous project (5) only got honeytoken interactions on the signature honeytokens, because
they did not involve downloading a document. To try to make the scammer interact with the
honeytokens, the honeytoken document content has been changed, using social deception to
improve the chance of a honeytoken hit further. This is done by scrambling the text contents
of the honeytokens to look broken, as seen in the picture of the contents of a specific PDF
honeytoken in Figure 15. The code is designed to change a character to one of a set of ”broken”
characters, with a certain probability of exchange. It also randomly slices some data out to
simulate information loss.

User interface
This section explains the different UI pages and the setup behind them, in addition to some

Like in the previous project, Flask33 is used to serve the UI of the main system. It contains
code that shows the 3 UI pages, several API endpoints used when performing actions from the

In addition to this, interface also requires a username and password to access the UI, for an extra
layer of security. It uses a function called requires auth, which is set as a function decorator
on the functions that show the UI. This makes it so requires auth acts as a wrapper around
the UI display function, first requiring a username and password input before showing the
UI. If no username and password have been input, it sends a ”401 Authentication required”
response to the browser, which then automatically prompts for a username and password. The
input is stored in the browser and is checked against the correct values stored in the project,
where the required password is stored as a salted hash. A salted hash is the result of hashing a
password together with a unique, random value called a salt. This prevents attackers from using
precomputed hash tables, since even identical inputs produce different hashes when different salts
are used. The password is checked against the stored value using the security utilities of the
Werkzeug web application library34. The stored hash of the password was also created using
Werkzeug.
Since the UI website is only accessible through a DTU VPN, the page is set up to use a self-
signed certificate for HTTPS. This is because it cannot be signed by a certificate authority when
the page is behind a VPN, like in this setup. As can be seen in Figure 19, the left corner shows
a red padlock indicating the certificate is not to be trusted. This is just a visual pain given the
VPN circumstances, and it uses TLS over HTTP to communicate confidentially.

5.4
Proxy system
This section consists of relevant details regarding the design and implementation of the proxy
system setup.
It outlines how it safely exposes personae websites to the outside world, by
containerising its environment while reliably forwarding honeytoken interactions to the main
system.
5.4.1
Design
The creation of the proxy system became necessary due to changes in circumstances and re-
quirements for hosting the system. The previous project had a more straightforward setup that
was hosted on a single cloud server using the cloud provider Hetzner. The main system was
expected to be hosted on a DTU server, which is not accessible from the public internet. To
allow external interactions while keeping the DTU server secure, a separate proxy system was
deployed on Hetzner.
Incorporating intermediary layers between internet-facing components and internal systems is
a well-established strategy for reducing exposure and limiting the potential impact of attacks.
With that in mind, the proxy system serves a role similar to that of a Demilitarised Zone
(DMZ) as it logically separates the publicly exposed personae websites from the core system.
31-07-2025
Page 50 of 109
==PAGE:57==
Technical University of Denmark
This setup does not constitute a traditional DMZ, since it does not utilise network-level seg-
mentation. However, it adopts a similar principle by introducing a logical separation between
exposed services and the core system. Through system-level isolation within the application
and deployment design, it achieves many of the same security benefits. This helps reduce the
risk of direct access to critical components, such as the scam baiting system and improves the
system’s overall resilience towards attacks.
In the previous project, honeytokens hyperlinks from the signature email were configured to
simply redirect to Google.com.
This offered no interaction or authenticity when scammers
visited the website, which could cause suspicion. In the current setup, greater emphasis has
been placed on keeping the scammer’s attention by presenting believable content that reinforces
the likelihood that they will interact with embedded honeytokens. To avoid logging proxy traffic
and automated security scans from cloud providers, the proxy system also introduces IP filtering
to distinguish real user activity from background infrastructure noise.
The infrastructure also needed to handle multiple websites running concurrently in a secure
and isolated manner. A total of nine websites are hosted using the Flask framework. Eight
personae websites, one for each of the honeypersonae and one exposure page. The personae
websites host AI-generated images and simple, fabricated information to make the personae
appear more authentic. The exposure page’s purpose is to display emails for web crawlers to
snatch and publish on dubious websites.
Running multiple applications simultaneously requires a simple yet effective way to orchestrate
all services while also ensuring the security of internet-facing websites. This can be achieved by
running the system with a Docker Compose setup. Docker Compose is a container orchestrator
and adds a concise way to describe multi-service applications in one file and to start them
with a single command. Studies show that projects use Docker Compose to configure services,
networks, and volumes in a reproducible way(91). This approach streamlines both deployment
and long-term maintenance of multi-service applications.
To handle the forwarding of honeytoken interaction from the proxy system to the main system,
a designated logshipper service has been implemented. This tool collects logs and, with the help
of a VPN sidecar, can post data directly to it. The study (92) notes that the logshipper Fluent
Bit fits well in Docker environments thanks to its lightweight design, and has an easy setup for
sending logs over HTTPS. Just as important, it provides a built-in buffering and resilient retry
mechanism, ensuring that no honeytoken interactions are lost in the event of a network outage
(93).
Nginx was chosen to serve as a single entry point for managing and distributing web traffic across
all nine hosted sites. As a lightweight and efficient reverse proxy, it helps route incoming HTTPS
requests to the correct website. Cloudflare was used to improve security and stealth. Acting as
both a DNS and reverse proxy service, it masks the IP address of the Hetzner server, provides
basic protection against DDoS attacks, and helps obscure any association with the DTU-hosted
backend. Cloudflare had the option to perform IP blocking toward proxy organisations (94),
yet the feature costs about twenty dollars each month, so a custom IP filtering technique was
approached instead.

Cloudflare serves as the first point of contact for all incoming traffic and forwards it to the
proxy system. It is used both to register the domain names of all eight honeypersonae and to
manage their DNS configurations through an interface. It enables control over how each domain
resolves by defining DNS records that point traffic to the IP address of the Hetzner server. The
previous project relied on Simply.com, which provided only basic DNS hosting; therefore, the
move to Cloudflare significantly expanded the feature options. By enabling Cloudflare Proxy,

The designated Flask application receives the request and processes it, responding with a min-

ensuring that the university endpoint remains private and unreachable from the public Internet.
Docker plays a central role in the architecture of the proxy system, acting as the foundation
upon which all services are built and isolated. Rather than installing applications directly onto
the host system via system packages or virtual environments, it uses Docker’s containerization

5.4.3
Request Handling and Filtering
The proxy system sits behind Cloudflare, which manages incoming traffic and forwards requests
to the Nginx server. To ensure the original client IP is preserved, Cloudflare includes it in the
request header, allowing the web server to identify the actual requester rather than Cloudflare’s
proxy address. Nginx is configured so TLS is set up for all personae websites, enabling HTTPS
communication. The setup and routing configurations of nginx can be found in the appendix as
Listing 25.
An SSL certificate is issued using Certbot, with Let’s Encrypt as the Certificate Authority (CA).
A single shared SAN (Subject Alternative Name) certificate is used, covering all domain names
served by the system. This approach simplifies certificate management by avoiding the overhead
of issuing and renewing individual certificates for each domain. The SAN certificate is valid for
90 days, and the accompanying Certbot container checks weekly if renewal is needed. When
the certificate is less than 30 days from expiry, Certbot automatically fetches a new one and
writes it to a shared volume used by nginx. Port 80 remains open to enable HTTP requests to
be redirected to HTTPS.
When a browser opens a persona URL, Flask handles the request based on the contents of
the path. If there is no path or it is simply the root ”/” of the domain, Flask provides the
corresponding static HTML file. If the path is anything other than a regular route, the generic
error handler issues a temporary HTTP 302 redirect back to the root ”/”. A 302 (Found) is
deliberately used instead of a 301 (Moved Permanently) so the scammers’ browser won’t cache
the redirect. If it were cached as a permanent 301, future visits could skip the website request
entirely, and those honeytoken interactions might never be recorded.
After Flask processes a request, it runs a decorator function just before sending the response
back to the client. The decorator decides whether the request should be logged based on the
characteristics of the URL path and the IP of the user. The sequence diagram in Figure 23
illustrates the control flow of the request handling and IP filtering.

The first assessment in the decorator is a call to is uuid path(request.path). It checks whether
the path of the URL request is in the format of ”xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx”.
It is in this format that honeytokens are identified, and the main system keeps track of which
honeytoken interactions should be linked to what honeytoken. In some cases, such as when
the honeytoken is embedded in Excel or Word files, the URL path includes a ”.jpg” suffix. A
helper function is used to remove this suffix before further processing. If the request path is not
in the UUID format, Flask returns the response and the request finishes without logging the
interaction.
The next step is to evaluate the source IP address to determine whether the request should be
considered genuine by calling is cloud ip(request.path). For example, when a scammer receives
the email and opens it in Outlook, Microsoft services may automatically scan the embedded
links as part of their security and spam filtering processes. This scan generates a background
GET request to the URL, originating from a Microsoft cloud IP address. Although the user has
not clicked on anything, the proxy system still receives a request that includes a valid UUID
path, making it appear like a legitimate interaction.
To prevent proxy infrastructure requests from being misclassified as genuine scammer activity,
the proxy system implements an IP filtering setup. The approach centres on real-time IP lookup,
which allows for identifying and excluding specific organisations identified by their Autonomous
System Number (ASN). An ASN is a unique identifier assigned to a network or group of IP
addresses operated by a single organisation.
The visitor IP is first checked against a local cache to determine if the IP has been previously

matching on these names lets it discard traffic that almost certainly comes from a service or a
proxy rather than from a real user. The keyword list is deliberately biased toward the biggest
cloud mail ecosystems and security platforms. Filtering on these names keeps the dataset focused
on real human interactions.
5.4.4
Log Forwarding
Each honeytoken interaction is captured as a compact JSON record that carries the most useful
context for later analysis. The log entry includes the time of interaction, client’s IP address,
country, requested domain, the honeytoken identifier, and the HTTP header, which consists of
user agent information.
Fluent Bit forwards the container logs in real time, filtering out Docker noise, and parses only
well-formed JSON files. Batches are sent over HTTPS to honeytoken.compute.dtu.dk through
the shared OpenVPN network. If a log batch can’t be sent due to a network outage, Fluent Bit
saves the log in a buffer folder. This folder is connected to a Docker volume, which means the
data stays safe even if the container is restarted. As soon as the connection is restored, each log
batch is sent once its retry timer expires. Fluent Bit employs an exponential backoff algorithm
to determine the waiting time before retrying.
A setting in the configuration explicitly tells Fluent Bit not to verify the TLS certificate. This
setting disables certificate verification, allowing Fluent Bit to connect even though the server
uses a certificate that can’t be validated by a known certificate authority (CA).
A self-signed certificate was issued for the honeytoken.compute.dtu.dk because the server
is only accessible over a VPN connection and cannot be reached publicly. CA’s such as Let’s
Encrypt must prove domain ownership with HTTP or DNS challenges and are therefore ineligible
to verify the server. While self-signed certificates still provide encryption, they are not trusted
by browsers, which gives rise to warnings.
The OpenVPN documentation from DTU does not indicate that the VPN client can connect
without sending the credentials in plaintext. To prevent the VPN credential file from being
stored in plaintext on disk, the contents are first encoded using base64 and saved as a file. This

5.5.1
Creation of Personae
The creation and selection of honeypersonae was a deliberate subgoal of this project, intended to
support the broader aims of deception and engagement with scammers. Rather than developing
deeply detailed or elaborate personalities in their AI prompts, the focus was on generating
plausible digital identities that varied across surface-level characteristics, such as age, gender,
and occupation.
Their personality traits have not been extensively developed, resulting in
plausible digital identities rather than fully realised personal identities. This choice ensured a
manageable scope while still enabling experimentation with scammer targeting preferences and
potential biases.
Eight unique honeypersonae were created, consisting of four men and four women, all based
in the United States. This was chosen deliberately, as prior research by Andersen notes that
Americans were the most targeted by phishing emails in 2023, suggesting that scammers view
them as more susceptible to fraud. The honeypersonae were evenly divided between the ages of
40 and 60, representing both mid-career and elder individuals. The group was also divided by
education level and type of work. One half represents people with limited or moderate education
who hold steady office positions. The other half are founders of small personal businesses, for
example, a massage studio or a local tech firm. All occupations were selected with believability
in mind, ensuring it would be reasonable for the individual to operate a personal website and

G2A, a game-key retailer whose unrealistically low prices and privacy policy suggest limited data

The complete set of persona email addresses and their aliases was published on four popular
paste platforms: JustPaste.it, ControlC, Paste.ee, and PrivateBin. They were also published on
online guest books such as Mugu Guestbook, 123Guestbook.com, SmartGB.com and Bravenet.
Each paste was framed to resemble an underground data leak, with a brief message suggesting
the emails originated from a recent giveaway involving U.S.-based individuals. This presentation
was intended to make the list appear authentic and valuable, increasing the chances that it would
be picked up and circulated by harvesting bots or malicious actors. An attempt was made to
publish a Craigslist ad that openly listed the e-mail addresses, but its automated moderation
blocked the post as soon as it detected visible e-mail addresses. To bypass this restriction, the
e-mail addresses were obfuscated by replacing “.com” with “[dot]com” or “[.]com” in an effort
to avoid detection, but without luck.
The exposure page is a simple web page that only exposes a simple HTML file with an overview
of all the email addresses of the created honeypersonae. This allows email scrapers to collect
and save them. The scraper visits web pages and scans their content to find patterns that look
like email addresses, for example, strings containing ”@” and a domain name. The outcome is

Testing
This section goes over the different ways in which the project was tested. This includes both
testing during development and the test period undertaken during the final stage of the project.
6.1
End-to-end tests
The key performance criteria for this project are how well the generated honeypersonae and
responses deceive scammers. To be effective in this goal, the whole system pipeline needs to
run smoothly, which was tested using end-to-end testing, along with various test scripts to test
a specific part, like honeytoken generation. In addition to this, the project was tested in its
interactions with scammers for approximately one month from 22/06/2025 to 21/07/2025. This
gathered data, which gives indications of how well the system performs against real scammers,
is explained further below and presented in Section 7.
End-to-end testing is the practice of testing the application’s workflow from start to finish. For
each feature, this was done by running the project locally on the VM it was also developed on,
and by sending emails to the system. This did not cause any issues with data logging, as the
local database on the VM could be overwritten easily to remove any data from the test. By
setting the system’s response time to be instant, sending an email could be tested and verified
within at most a few minutes. Then, any output from the system was analysed using journalctl,
the command line utility for seeing logs for a systemd service. That means that the relevant
service can be easily checked, as they all have their log outputs. An example of these logs can

The qualitative requirements mostly involved the social engineering aspects, which affect the

always have a clear answer and that has to be assessed through extended discussion, along with

The qualitative aspect is also important in areas such as the NLP used for honeytokens and
scam detection. The categories and keywords which drive this functionality are based on what
scammers send to people, which luckily can be informed by existing datasets. That being said,
it is hard to make a solution using NLP that fits all cases of scam emails and keywords for
honeytokens, without getting too many false positives. An example of this is shown in Figure

top of that, even what constitutes a scam email is up to the user of the system, which inherently
carries uncertainty. This also applies to the use of LLMs, which in many ways are ”black boxes”
that cannot be tested in the same way as code. Prompt engineering was a challenge because any
change in the prompt could lead to unforeseen differences in the output due to this nature. All
these points reinforce why running the system in a test period with real scammers was useful.
Other parts, like testing the expiration time of the Gmail token, had to be tested by reading the
documentation as thoroughly as possible and waiting to see if it expired or not when the required
settings were changed. Some pieces of code, like the regular expressions in clean email content,
could more readily be tested directly, because there were some known cases of phrases to handle
to protect the AI, which were found through the testing of AI responses.
The purpose of the testing period was mainly to evaluate the whole flow and its effectiveness in
deception and achieving honeytoken interactions, but it was also used to tweak the system. With
so many components, some flaws became apparent only when running it with real scammers.
For example, the regular expressions meant to strip out some email content were tweaked when
it was seen how the real scam emails were handled by the system, to make it less restrictive.
Another issue discovered during testing was that many scam email addresses were no longer
active, resulting in responses from ”no-reply” domains. These were mistakenly logged as aliases
within the conversation, causing the system to register the interaction as successful and thereby
distorting the data. This was resolved by adding a check to detect such addresses and prevent
the system from sending replies to them.

The experimental setup followed a structure similar to the previous project, consisting of a pre-

involving active engagement, and a post-deployment phase for data assessment. In the pre-

April, followed by account creation and newsletter postings on platforms such as Pastebin and
various guestbook sites in late April. Social media accounts for the honeypersonae were cre-
ated from mid-April onwards. Despite these combined exposure efforts, few scam emails were
received.

As the testing period approached, it became evident that too few scam emails were reaching the

The process of determining which emails were actual scams relied heavily on the indicators
established in Section 2.1.2. A notable observation during this phase was that scam emails were
sometimes difficult to identify conclusively. This could be attributed to scammers improving
their linguistic quality, possibly through the use of AI tools, which made their intent harder to
detect and allowed them to bypass traditional spam filters. Once a scam email was identified,
it was answered using the ethical principles discussed in Section 2.1.3 and informed by Chen et
al. (6). Specifically, the system used a ”no chase behaviour” policy, only sending one email to
each address, and required human supervision throughout.
The conversations were initiated from the User interface ”Send a New Email” from Figure 19.
The unprompted emails were crafted to sound like they came from a confused individual. The
purpose of this approach was to appear plausible while also vague enough to spark curiosity or
elicit a response from the scammer. An example of the content was ”Hello, I saw your earlier
message but didn’t catch the context. Could you give me a quick rundown so I know what to
do?” or ”Hi there, I just noticed your message in my inbox, but I’m not entirely sure what you’re
referring to”. Could you remind me what this is about?” The subject of the emails started with
”What’s this?” followed by some context of the contents the scammer had included in their
initial message.

6.5
Connectivity & Security
By filling every persona’s inbox with multiple unread emails, the Gmail API was stress tested.
The email monitor service checks every persona’s inbox sequentially, which generated a large
number of requests within a second and exceeded Gmail’s concurrent request ceiling, resulting
in HTTP 429 ”User rate limit exceeded” responses that froze new reads (100). Adding a one-
second pause before the loop moves to the next persona keeps bursts below the threshold, and
no further errors appeared in repeat tests.
When setting up the proxy system, the link between it and honeytoken.compute.dtu.dk was
experimented with in three steps. First, a DNS resolution was confirmed by observing that
the fully qualified domain name resolved publicly to its IP address.
Secondly, a raw TCP
reachability was checked using netcat command. These attempts failed when the client was
outside the VPN, indicating that direct access is filtered at DTU. Third, an end-to-end HTTPS
handshake was attempted, but obviously couldn’t be reached since the TCP reachability was

7.3
IP analysis
Figure 39 and 40 show the geographical distribution of honeytoken interactions. These inter-
actions provide insight into the likely geographic locations of scammers who engaged with the
system. The 49 unique IP addresses span several countries. Notably, India accounted for a
disproportionately large share, with 34 unique IP addresses, followed by Nigeria with 5 and
Benin with 4. Other countries represented include Canada, Brazil, Ghana, Ukraine, the United
Kingdom, and the United States, each contributing one unique IP address.
It is worth noting that IP geolocation is not always a reliable indicator of the true origin, as
scammers may use proxy servers or VPNs to conceal their location. To assess the reliability
of the collected IP data, an additional lookup was conducted using ipqualityscore.com 47,
which provides deeper insights into proxy and VPN usage.
Of the 49 addresses examined, 35 were labelled as suspected proxies. However, in a 2024 study
comparing proxy detection services, IPQualityScore (IPQS) was found to flag significantly more
IPs as proxies than other providers (102). The study noted that about 78.3% of the IPs IPQS
flagged as proxies were indeed proxies, meaning roughly 21.7% were false positives in that
experiment. Additionally, IPQS states that IPs with a fraud score above 75% are highly likely
to be associated with a proxy service, VPN or TOR network (103). Applying these conditions
to the current list of suspected proxy IPs, 26 of the 35 suspected proxies exceeded this threshold,
signalling a strong suspicion of location obfuscation by a proxy. Only one address was tagged
with the stricter active vpn flag, and none were identified as Tor nodes. This leaves 23 addresses

IPs tied to clearly identifiable hosting infrastructure offer little value in geolocation analysis and
can be safely excluded.

7.5
Evaluation of Honeypersonae
The project has a subgoal of seeing whether any differences in honeypersona performance are
observed, which is explored in this section.
When examining the results of the deployment period, Corrie D. Henderson, a younger self-
employed woman running a massage business, stood out with the highest message count and
strong sustained engagement.
Larry Hopton, an older self-employed man, and Micheal M.
Steward, a younger employed financial consultant, also performed well in conversation length
and duration. Meanwhile, Rebecca D. Wood and Thomas D. Bergman, both employed office
workers, achieved the highest honeytoken interaction rates. These outcomes lack a clear pattern,
which may be due to several reasons.
First, most of the scam emails were manually assigned to the honeypersonae, and this process
may have introduced bias. Even though an attempt was made to give all honeypersonae an
equal number of emails, some may have received more active or responsive scammers, given
that many of the scam emails used were old. This is evident in which scam emails were still
31-07-2025
Page 82 of 109
==PAGE:89==
Technical University of Denmark
active or the best candidates to use, which was unknown when the emails were sent. This could
mean that some honeypersonae got better or worse results by chance.
Another limitation is that multiple conversations may originate from the same scammer, operat-
ing under different names or addresses, further muddying the results, as a honeypersonae could
have sent multiple emails to the same scammer. Additionally, the small overall dataset and rel-
atively low rate of successful conversations prevent reliable analysis across persona subgroups.
To validate any trends, a larger dataset with repeated trials and controlled exposure would be
required. A complete statistical analysis would also require some baseline honeypersonae, such
as plain identities with no images or distinguishing traits, which would help isolate the impact
of honeypersona details.
With hindsight, some design inconsistencies also limit the gathered statistics.
For instance,
exact professions were not evenly distributed across gender and self-employment lines, which
should have been consistently paired across pairs of matching personae. The thought behind
giving them all different jobs was to distinguish them and make sure none of the honeypersonae
information directly matched in an observable way. However, this makes it hard to determine
the effects of occupation because each occupation, while in the same overall category of self-
employed or not, is fundamentally different.
The intended age separation (around 40 vs. around 60) was also not strongly reflected in the
profile images, and age was not clearly signalled in email content. This could be because it is
hard to generate AI pictures of a specific age group while remaining realistic, which was noted
during the process. Greater visual and textual contrast, or a larger age gap, could help future
studies test age-related scammer behaviour more effectively. Finally, all honeypersonae behaved
similarly in tone and language, which could have been more tailored to the specific profile.
Adding tailored conversational styles or personality-driven responses (for example, cautious,
overly trusting, emotional or confrontational) which match the other honeypersonae’s informa-
tion could better test whether scammers prefer specific psychological targets. Though if this
were done, it adds yet another statistical category to account for.
It should be emphasised that a thorough comparative statistical analysis of honeypersonae per-
formance is not a main goal of this project, and requires a more robust deployment setup,
and more data to explore these differences accurately. That being said, the project provides a
framework for how this can be done.

7.7
Limitations
There are a variety of circumstances which may have hindered the results of the project’s
deployment period or had some impact on it. The main problem was not attracting enough
scam emails and starting conversations with them, because they need even more time to begin to
attract traffic. It was hard to create fake accounts to expose emails, because sites required some
authentication, like a phone number or identification. To protect the security of the authors,
it was chosen to use as little personal information as possible in the creation of these accounts,
avoiding the use of personal phone numbers. One potential solution to this could be creating
dedicated phone numbers and email addresses for this project, though this requires access to
more resources.
When generating fake social media accounts for a more authentic scambaiting system, platform-
level detection mechanisms can crack down on suspicious-looking accounts. A notable example
of this comes from LinkedIn, which actively disables accounts and requests further verification
after detecting unusual activity. While the company does not publicly disclose all the factors
influencing its decision, several indicators can be inferred. LinkedIn employs image forensics
models to detect AI-generated profile pictures with near-perfect accuracy (105). In addition,
fake accounts often lack detailed employment or education history, a common red flag that
distinguishes them from legitimate users (106). Geolocation inconsistencies also contribute to
detection.
For instance, even though VPNs may match the U.S. location at sign-up, other
metadata such as browser language or device time zone can expose discrepancies suggesting a
different origin(107). While the AI-generated profile photos of these honeypersonae may have
been a significant factor, it is likely the combination of these indicators that raised LinkedIn’s
internal risk score high enough to warrant deactivation.
The relative success rate of conversations is 20.28% compared to the previous project, which
had 38.37% (5). The inadequate scam email attraction could also explain this lower rate. This
led to the use of the email from the previous project as a source of scam emails, where some had
been inactive since the end of 2024. This could mean that many of the scammers had moved
on to using other emails and deleting their old ones, as indicated by all the messages in which
a no-reply email response was sent. It is also not possible to know if some of the personae were
revealed when sending emails to different scam emails, because some scam emails could belong
to the same operation, which could have made the connection.
The hit rate of non-signature honeytokens is 0% for PDF, 25% for Word (28.6% with the excluded
honeytoken) and 30% for Excel. The fact that these percentages are not higher might be because
of the aforementioned preview function for documents in email clients, which diminishes the need
to download a document if the only purpose is to read it. This highlights the importance of
social engineering in this project because scammers must download the honeytokens for the
beacon to contain their information. The PDF has a hit rate of 0%, mirroring the results of
the previous project (5). This may be because most scammers use PDF readers that block the
honeytoken call, as Word and Excel have been demonstrated to work.
There were also some issues with the AI, in tuning it and getting the best quality responses.
By using smaller models with far fewer parameters than OpenAI’s models, the quality of the
responses dropped a bit, and more detailed prompt engineering had to be used. This issue was
caused in part by the limitations of the main system server hardware, which was chosen with a

time limit in mind and was therefore not the best that could be afforded at DTU. It was also set

An essential disadvantage in the long-term viability of the VPN connection between the proxy
system and main system lies in its dependency on the active status of the student’s DTU

communicate with the main system. Attempts to get a dedicated user for the system created
by the DTU IT service were rejected due to security concerns. In addition, once the student
graduates, the VPN connection will no longer authenticate successfully, even if the certificate
itself has not expired. This presents a clear constraint for the long-term functionality of the
honeytoken logging system.
The project has some elements that require a cost to access or run. One is the persona domains,
which are around 67 Danish kr pr domain per year, and come to a total monthly cost of 45 kr
for eight domains. Hetzner also costs 45 kr a month to run, with the chosen setup. Other than
that, Midjourney also costs money to use to generate pictures of personae, which is 77 kr per
account per month. This is not a recurring expense since images can be generated all at once.
Therefore, the long-term cost right now of running the project is around 90 kr per month, with
added costs if more honeypersonae are to be added.

8
Conclusion
This thesis builds upon and refines an existing email scambaiting system created by Andersen
(5). It focuses on advancing cyber deception strategies by improving system usability, security,
infrastructure, the ability to waste scammers’ time and by collecting data through new honey-
tokens. All these improvements were made possible by dividing the total system into a main
system and a proxy system.
The main system is deployed on a server hosted at DTU, utilising a locally run Large Language
Model (LLaMA 3.1) to generate email replies. The model was tested and configured with prompt
engineering and a guard prompt mechanism to reduce the risk of undesired outputs that come
with reduced model size. While some responses still fall short of ideal behaviour, the overall
quality of replies is sufficient to deceive scammers at a level surpassing that of the previous
project. Structural changes in email scheduling and state handling utilising the transactional
outbox pattern allow the system to better manage delays or retries in the case of a system
restart or crash, which is important for sharing the AI model across tasks.
Further enhancements include the creation of multiple honeypersonae, each with fabricated back-
ground data, profile images, and personalised websites served through a proxy system hosted on
Hetzner. Although the creation of social media accounts is explored as a means of enhancing
persona realism and exposure, this effort is hindered by authentication requirements on the
platforms. Still, the use of individualised websites, pictures and differentiated backstories sup-
ported deception efforts. The system UI has been improved to fetch emails automatically for
each honeypersona, mark possible scam emails and use new Gmail access tokens that do not
expire, limiting manual labour.
The proxy system enhances overall system segmentation by containerising its externally available
services, and by making sure that exposed personae websites are separated from the main
system. This setup helped reduce proxy-related honeytoken interactions through IP filtering
and provided reliable forwarding of real-time geographic data that was readily available through
the main system UI. In addition, it also utilises Cloudflare for DNS privacy and Denial of Service
mitigation.
Quantitatively, the system achieves a clear increase in scammer engagement compared to prior
work, as seen in data from the deployment period with actual scammers. Conversations reached a
mean of 10.27 replies and a maximum of 65 messages, outperforming both the previous project
(mean 5.81, max 23) and an earlier baseline system (mean 5.38, max 36). The duration of
engagements has also been extended, with an average conversation time of 7.93 days and a
maximum of 29.13 days. These figures indicate a substantial improvement in the system’s ability
to hold the scammers’ attention and prolong communication, an essential goal in scambaiting.
The system introduces Word and Excel honeytokens with content socially engineered to en-
courage the scammers to download them, which achieved interaction rates of 25% and 30%,
respectively. These numbers are on par with the signature honeytoken success rate of 29%, con-
firming that well-crafted document-based honeytokens can serve as viable alternatives. These
honeytokens collected valuable data, showing various trends in scammer profiles, which are dis-
cussed. In contrast, PDF-based honeytokens failed to produce any interactions, mirroring the
0% success rate observed in the previous project, which emphasises that document type matters
when sending honeytokens to scammers.
As a subgoal, the thesis also examined whether persona traits, such as age, gender, and occu-
pation, might influence scammer behaviour. Several honeypersonae were created with varying
characteristics, and interactions were monitored for signs of differential engagement. However,
due to the limited dataset and the subtle differences between honeypersonae, no significant

trends were identified. This remains an open question for future studies with larger-scale de-
ployments and more distinct persona profiles.
In summary, this thesis presents a framework that successfully deceives and engages scammers,
collects valuable data, and demonstrates clear improvements over the previous system (5). By
combining cyber deception with automation and a secure infrastructure, it creates a reliable
platform for long-term scambaiting. The results validate the system’s ability to generate sus-
tained conversations and successfully deploy new types of honeytokens. These advances lay a
foundation for further work in scammer profiling, scalable scambaiting systems, and AI-driven
countermeasures.
8.1
Future work
Although this project successfully advanced the capabilities of automated scambaiting, several
areas remain open for further development. Some of these opportunities emerged from limita-
tions encountered during implementation due to factors like a lack of time, while others became
evident through analysis of system behaviour and outcomes during testing. Addressing these
could help work towards a more complete and definitive framework for effective scambaiting.
More honeytoken types and honeypot infrastructure
While the current system success-
fully demonstrated that Word and Excel documents can yield scammer interactions, additional
honeytoken types could also be explored in future projects. Introducing other file types, such as
zip files, pictures, or other types created by Canarytokens, could further increase the likelihood
of getting an interaction. Additionally, integrating honeypot infrastructure directly into the
personae websites could be another way of gathering data, since the scammers get a point of
contact through these sites. These sites could simulate login forms, contact pages or fake direc-
tory structures containing honeytokens. This would enhance profiling capabilities by offering an
additional way to get information from more advanced scammers.
More detailed and varied honeypersonae
Although this project introduced multiple hon-
eypersonae, a broader and more clearly differentiated set could improve deception and allow for
more targeted experiments. Future work could define personae with specific behavioural traits,
controlled response moods (for example, hesitant, eager, suspicious), and more detailed back-
grounds. Within ethical considerations, these personae could also be configured to share more
sensitive fake information or ask leading questions, influencing and pressuring scammers to re-
veal more about themselves. With sufficient scale, statistical analysis could identify which traits
influence engagement success. Coupling this with creating social media profiles could add a
strong layer of credibility.
Infrastructure Improvements and bigger LLMs
A key bottleneck in the setup is the cost
of running the proxy system externally, and the lack of dedicated VPN credentials for it. Hosting
the proxy system on a DTU-owned external server instead of on Hetzner, with dedicated VPN
credentials, would simplify maintenance and reduce overall operational cost. Moreover, gaining
access to a more powerful server to run the main system would enable the use of bigger LLMs,
improving the quality of email replies, and addressing some of the shortcomings observed in the
current generated responses.
Improved Response Management and Dialogue State Tracking
As the length of con-
versations increases, maintaining coherent and contextually accurate responses becomes more
difficult, revealing the AI nature of the responses. Future iterations could benefit from imple-
menting a dialogue state machine to manage the conversation flow per scammer, which could
build on work done by Charnsethikul et al. (73). This could help with extracting scammer
31-07-2025
Page 88 of 109
==PAGE:95==
Technical University of Denmark
information, like in their study, but also to effectively inform AI response generation with the
conversation state, in terms of what information has been shared, and which honeytokens have
been sent. Additionally, a mechanism to group and process closely timed incoming emails from
the same scammer in a single email could make the response seem more realistic. These im-
provements could together create a groundwork for more nuanced conversation strategies by
combining multiple approaches to the conversation system.
Automated mass conversation initiation
A helpful feature for starting a deployment pe-
riod is the automated initiation of scammer conversations from a provided list of email addresses.
This functionality could allow the operator to upload a list, which the system would use to dis-
tribute personalised messages across different personae, randomising subject lines and content
to create diversity. While feasible, this approach raises ethical concerns around sending unso-
licited emails, even to suspected scammers. Future research could examine the boundaries of
this practice, perhaps focusing on verified scam lists or legal issues, while also implementing
controls to maintain ethical standards in scambaiting.

A.3
Description of main system components
The following is a list shortly describing each component of the main system.
• layout: HTML file template that UI pages inherit from.
• conversation ui: UI for seeing information about conversations for a persona and starting
conversations.
• send emails: UI as it looked in the previous project, preserved for cases where manual response
generation is needed.
• stats overview: Where statistics for the whole project can be found.
• interface: Frontend server for interacting with the main system.
• bridge: central piece of the backend serving as an API that connects task handler and gmail
service to UI and handles scam email filtration.
• scam nlp: used to detect whether an email is likely to be scam.
• database: where all data in the project is stored and read from.
• logging service: contains logic for logging data in the database.
• listener: logs incoming honeytoken interactions.
• gmail service: communicates with Gmail API to fetch and send emails.
• tasks: handles all scheduling and execution of response generation.
• email monitor: checks if incoming emails are a part of an existing conversation and schedules
response if so.
• update message counts: updates how many messages each conversation has for database
read optimization purposes.
• conversation handler: contains the logic for responding in conversations.
• honeytoken nlp: contains the logic for deciding honeytoken inclusion in messages.
• generate pdf: responsible for pdf honeytoken generation.
• generate docx: responsible for word honeytoken generation.
• generate excel: responsible for excel honeytoken generation.
• openai service: used if system is set to not use local AI.
• ollama service: used if system is set to use local AI.
• garble text: garbles/scrambles text content of honeytokens, can be turned off or on.
• personae prompts: contains info used when generating responses or honeytokens.
• response check: used to make sure outgoing AI response lives up to set standards.
• clean email content: used to clean unwanted incoming email content and before logging
emails in database.
