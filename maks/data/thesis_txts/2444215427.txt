Abstract
Sensor technology plays a key part in measuring physiological signals from a user,
where these signals provide a way of assessing, e.g. a cognitive load of a user with a
further analysis of the collected signals
This thesis aims to develop an architectural blueprint and implement the architec-
ture in a prototype, which is capable of collecting data from different physiological
measurement devices, simultaneously, in a synchronised way. Moreover, provide a
collection of data processing services such as pre-processing and feature extraction
that gives a premise for further analysis of the collected data to assess, e.g. cognitive
load. Furthermore, provide storage for the collected and processed data. The pro-
totype is implemented in microservices structure to improve the modularity of the
system, furthermore, containerising these microservices to improve the sharability.
The implemented prototype is capable of collecting measurements from an eye
tracker (gaze, pupillometry) and a wearable skin sensor (GSR, PPG), in addition,
the prototype has the capabilities of adding additional devices and processing services
with minimal changes. In conclusion, the final prototype can provide a system for
users which is suitable to collect, process and store these physiological measurements.

CHAPTER 1
Introduction
1.1
Motivation
Information technology jobs are among the fastest growing jobs according to [29];
these jobs require much more mental focus, rather than jobs that require physical
activities, as most of their tasks are in front of a computer. The mental focus can
be crucial when working on specific tasks; therefore the necessity for evaluating the
employee psychological state is increasing, not only for the benefit of the employer
but also for the employee. This measurement of a psychological state is not only
bound to the public or private sector but also in the educational sector by collecting
physiological data to analyse the workload and be able to detect the mental state of
the subject, e.g. cognitive load, which can be used for productivity and motivation.
Cognitive load describes the requirements of tasks imposed on the limited information
processing capability of the brain.
Furthermore, describes an individual measure
depending on available resources interacting with given tasks on an individual level[8].
A mental state can be measured in various ways, such as from an eye tracking
device, where researchers have found a correlation between pupillometry and cogni-
tive load [23], moreover, assessing cognitive load using Photoplethysmography (PPG)
signal [24] as well as a Galvanic Skin Response (GSR) signal [36].
The main inspiration of the system presented in this thesis is the technology and
computing power that is available today that can be used to detect an online mental
state of a subject. Therefore, the inspiration is to implement a system that can collect
multi-model event time series data, from different types of measurement devices, for
further analysis to be used as a classifier for future online mental state detections.
At the moment of the publication of this thesis, there is, unfortunately, no acces-
sible system that deals with a similar approach as described earlier. By implement
this prototype and have it open source for anyone interested in, e.g. mental state
detection could benefit from this prototype in the way of reducing resource, time and
money implementing their own system.

1 Introduction
1.2
Problem Statement
The problem that this thesis address is the problem of collecting multi-model event
time series data that is a collection of psychological parameters. This system needs
to be able to collect time series data from multiple devices that will furthermore be
synchronised to store the data in an orderly way for future analysis. This system needs
to have the possibility to add new measurement devices without complex configuration
to the overall system.
1.3
Goals
The main goal of this thesis is to develop an architectural blueprint and implement the
architecture in a prototype following a microservices structure to improve modularity,
that can be furthermore be used as a neuro-adaptive system, i.e. a software system
that can adapt to the mental state of its users. The system has to be able to collect
multi-model event data stemming from different types of sensors or devices such as
an eye tracker or a skin sensor to measure GSR signal and be able to synchronise
the collected data via a middleware layer. This architecture should also be able to
add on other modular services such as cleaning, pre-processing and feature extraction.
Moreover, the collected raw data and processed data must be stored in a database in
an orderly way for further analysis. To accomplish this, the system must be able to:
• Collect eye tracking data from a subject.
• Collect physiological data, e.g. GSR and PPG data from a subject.
• Capable of add new measurement devices.
• Synchronise the measured data.
• Process the data, e.g. preprocess, cleaning and feature extraction
• Implemented in a microservices structure.
• Store collected data for further analysis
Therefore, this thesis will present a system design and an implementation of an ar-
chitectural blueprint in a prototype. Furthermore, the system will be evaluated with
previously recorded research data as well as other performance testing the prototype
with various demonstrations as an evaluation of the system to prove the concepts that
are presented in this thesis.

CHAPTER 2
Background
2.1
Neuro-Adaptive System
Neuroadaptive system (NAs) can be described as a system that is a combination of
computer and sensor technologies, among other technologies, that form into a system
that responds to the neural signal of the subject [16]. Moreover, selection of features
extracted and classified from these neural signal can be used to adapt the task that is
in front of the subject depending on the measured neuro-signal and its classifier, where
a classifier is a learning algorithm to predict the adaptation for the task. Furthermore,
using machine learning approaches gives promising results of having a NAs online and
adapt to the tasks of the user, according to this study [4].
2.2
Physiological Measurements
Physiological measurements can be from a simple body temperature measurement
with a thermometer or can be more complex measurements such as measuring an
ECG, a recording of electrical activity of the heart over some time. Many physiologi-
cal measurements can be used as an index for a cognitive load, by studying different
types of measured data related to specific tasks [21]. To name a few physiological
measurements, an eye tracker which can track gaze position, pupil size, blink rate
and more, a biosensor that can track galvanic skin response (GSR) and Photoplethys-
mography (PPG), an oximeter that can track pulse and blood oxygen. Therefore, a
physiological measurement is an excellent source of data to measure conditions such
as cognitive load [21]
2.2.1
GSR and PPG
A PPG signal is often obtained by using a sensor, typically placed on a finger, that

2.3
Synchronisation
All collected data that are mentioned in Section 2.2 are collected through different de-
vices. If a device provides a timestamp, then that timestamp might vary from another
device even though the data were collected simultaneously. Therefore, synchronisa-
tion is needed to synchronise incoming data from different devices. Lab Streaming
Layer (LSL), is a solution that provides synchronisation of time series that are coming
from multiple sources. LSL is based on the principle of Network Time Protocol (NTP)
[20], which is a protocol used over the internet, where a client requests the current
time from a server. From the response, the client can derive reference time at the
server as well as the elapsed time, from this exchange the client can adjust the local
clock to be synchronised with the server. LSL has a built-in time synchronisation
based on the NTP protocol that handles the synchronisation of the data stream.

Microservices
Microservice architecture is a development technique where the overall system is split
into a collection of loosely linked services.
The benefit of having an application
split into smaller services is that it improves modularity. The microservice architec-
ture makes the application easier to understand, develop and test [10]. Furthermore,
smaller teams can work on different services at the same time. As each service can be
written in different programming language, therefore, the communication protocols
between these microservices is an important choice, it could be a Representational
State Transfer (REST), Java Message Service (JMS) or any other messaging protocol
that are compatible between services. These microservices can then be encapsulated
in a Docker container, with communication protocol established between them.

2.6
Communication protocols
Communication between microservices is an important choice to make, as briefly
mentioned in 2.5. There are several types of communication protocols. The following
list is a short list of commonly used protocols:
• MQTT is one of the oldest machines to machine (M2M) communication proto-
cols. It is a basic messaging protocol which an MQTT client publishes messages
to a topic; meanwhile, a client subscribes to these topics to consume messages.
The payload can vary from being a small to a maximum size of 256 MB. For
security, it uses TLS/SSL, and the transport is over TCP
• CoAP is an M2M protocol, lightweight. It uses Universal Resource Identifier
(URI) instead of topics like in MQTT. When a Producer sends a new message
to the URI, then all subscriber to that particular URI will be notified of the
new message. The payload can vary from being a small message to undefined
maximum, which depends on the server capabilities. For security, it uses DTLS
and UDP for transport
• AMQP is also M2M protocol, lightweight, and it supports both request/response
as well as publish/subscribe architecture. The payload can vary from being a
small message to undefined maximum, which depends on the server capabilities.
For security, it uses TLS/SSL, and SASL and the transport is over TCP.
• RabbitMQ is a message broker that implements the AMQP standard, an essen-
tial feature about RabbitMQ is that it accepts connections between different
==PAGE:24==
10
2 Background
platforms, e.g. python and Java could communicate using RabbitMQ. As it im-
plements the AMQP standard, then the payload also varies from small message
to undefined maximum, which depends on the server capabilities. For security,
it uses TLS/SSL, and the transport is over TCP.
• ActiveMQ is written in Java and implements the JMS standard. The payload
can vary from being small message to large message, up to 8 GiB messages
have been tested in sending and consuming according to [3]. For security, it
uses TLS/SSL, and the transport is over TCP.
• SOAP has a built-in error handling and can be used on any transport not limited
to HTTP only. SOAP relies on XML to provide messaging services. Works well
in distributed enterprise environments
• REST relies on HTTP URL instead of using XML to make a request like SOAP.
There is a smaller learning curve than SOAP and frameworks like swagger can
be used to help with designing, building and documenting the REST API.
This list is a summary of comparison from different sources of comparison research,
such as [26] compared MQTT, CoAP and AMQP messaging protocol for IoT systems
while [22] compared the performance between RabbitMQ and ActiveMQ and [25]
compared REST and SOAP in a microservice architecture. This list can, therefore, be
a basis of communication choice between the microservices for the proposed solution.

Behavioural driven development (BDD) is a combination of practices from Test Driven
Development (TDD) with ideas from Domain Driven Design (DDD)[5]. In BDD, the
focus is set answering following questions; Where to start in the process, What to
test and what not to test, How much to test in one go, What to call the tests, How
to understand why a test fails [13]. Typically these questions are answered through
a user story; these user stories are written out as a scenario that includes a brief
background and a title that summarise the intended use of the application. In [14]
they describe scenarios as properties in natural languages using Given - When - Then
to formulate the structure of the scenario and each sentence represents a sentence
called a step and is grouped as a step definition. The implementation is developed
iteratively from the step definition where the step definition contains fragments of
test code that describes the behaviour of a single step in a scenario.

Gossip is a peer-to-peer (P2P) communication protocol that nodes periodically ex-

runs each second with information message about states of up to three other nodes
that they have gossiped about; the purpose of this process is that all nodes quickly
learn about all other nodes in the cluster20.

REST

JMS
JMS messages are sent using the ActiveMQ message broker, which is the default

messages are also used for inner messages in the Wildfly Application Server on the
Processing server which is used to add messages to queue as soon as a REST request
arrives from the PC client, therefore the incoming request is processed when the
previous request has been processed, a first in first out queue. This design ensures
that the data is on a queue waiting to be processed in the correct order as it arrives.

5.3
Data Processing
From the output in Figure 5.8, there are information regarding several types of data
even though only raw data was sent to Processing server this is due to the configura-
tion set in apiUrl values in the JSON string of the request that is sent from the PC
client. These values are the routes to the processing procedures in the Python Web
Framework.
"apiUrl":"/eyetracker/substitution,
/eyetracker/avgPupil,
/eyetracker/avgPupil/perTask,
/eyetracker/interpolate"
"apiUrl":"/shimmer/normalize"
Figure 5.9: apiUrl values for Eye tracker and Shimmer.
The apiUrl list in Figure 5.9 is iterated in the Processing web application and raw
data is sent to the Python Web Framework for processing, when a response arrives
with the processed data, then the processed data is sent, via JMS message, to the
Database server for storage. These processing procedures are described as followed.
/eyetracker/substitution route procedure scans data for missing values, if
there is a missing value and if opposite value is not missing then the missing value
will be replaced by the opposite value, i.e. if left X value is missing and not right X,
then the left X will be replaced with the right X value.
==PAGE:76==
62
5 Demonstration
/eyetracker/avgPupil route computes the overall average pupil diameter from
the raw data, for the left eye pupil and the right eye pupil.
/eyetracker/avgPupil/perTask route computes the overall average pupil diam-
eter from the raw data for each task, if there is more than one task in the data, for
the left eye pupil and the right eye pupil. If there is only one task, then this will
compute the same value as in the previous route /eyetracker/avgPupil.
/eyetracker/interpolate route computes linear interpolation with new value(s)
on missing value(s) using known value(s) before and after the missing value(s).
/shimmer/normalize route normalises the data according to the formula (2.1)
explained in Section 4.3.3.2. Where the raw data is normalised by dividing each data
value in each task by the average value of all tasks.
As mentioned at the beginning of this chapter, these processing procedures would
be beneficial for an application scenario such as the DCR model experiment, where eye
tracking data and GSR are collected, processed and then stored for further analysis,
e.g. obtaining a measure of cognitive load.

CHAPTER 7
Conclusion
In this thesis, an architectural blueprint was developed and implemented in a proto-
type, that is capable of collecting physiological measurements from multiple devices
in a synchronised way, with the possibility of adding additional devices to the proto-
type with minimal changes. Moreover, a data processing framework was implemented
that contained collection of processing procedures such as pre-processing and feature
extraction to use for further analysis, e.g. assessing cognitive load, the framework was
implemented in a way that minimal changes are needed to add additional processing
procedure to it. Furthermore, the prototype provides storage for all types of collected
data in a database. The architecture was implemented in a microservice structure,
to improve the modularity of the system, as well as to containerise the microservices
to improve shareability.
This thesis lays a blueprint of a system as well as practical information about
adding additional devices and processing services to expand the prototype further.
Moreover, demonstrations and guides were presented to provide practical information
on how to run the implemented prototype.
While there are many useful features in the system and achieving promising per-
formance result, there are limitations. The first limitation is the lack of graphic user
interface (GUI) in the PC client with more possibilities of configuring the PC client
for data collection in a user-friendly way. Another limitation of the system is the
lack of online capability of data collection, which was part of the requirement ad-
dressed earlier on in this thesis. However, promising results in the evaluation shows
the system has potential in being used in an online setup.
In conclusion, this thesis created and presented a suitable system that can col-
lect simultaneously physiological measurements in a synchronised way, process the
collected measurements and provide storage in a database, with all defined require-
ments of the system fulfilled except the capability of collecting data in an online
setup.
Future Work
There are few directions that the proposed solution can be expended to. As mentioned
the lack of capability of an online setup, then one of the possibility is to have the
system in an online setup by changing the PC client in a way that it can track online
by sending data periodically to the Processing Server instead of buffered data after
==PAGE:94==
80

collection has finished. By implementing this online setup, then the system is open
for being used in an online application scenario where the goal is to measure, e.g. a
cognitive load of the subject, a direction towards the neuro-adaptive system.
Necessary processing procedures are needed to be added to the system to assess
the cognitive load. Therefore the Python Web Framework can be expanded to add
services to it such as classifiers and machine learning algorithms to assess and predict,
e.g. a cognitive load of the subject in an online setup.

Docker
Here below is the Dockerfile that is in the GitHub repository. For this image, ubuntu

# Ubuntu 18.04 is the base image
FROM ubuntu:18.04
RUN apt-get update -y && \
apt-get install -y python-pip python-dev
# Copy the requirements.txt first to the app directory
COPY ./requirements.txt /app/requirements.txt
# Create a directory called /app
WORKDIR /app
# Install all the libraries that are in requirements.txt
RUN pip install -r requirements.txt
# "." is the current directory, and then copy everything to the /app directory
COPY . /app
# python is the entry point when the docker image is ran
ENTRYPOINT [ "python" ]
# app.py is the python script that will run when docker image will run
CMD [ "app.py" ]
