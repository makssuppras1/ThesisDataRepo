Abstract
Formal verification is essential to establish confidence in the security of cryptographic
protocols. However, bridging the gap between high-level protocol specifications, such as
Alice and Bob notation, and verification tools remains a challenge due to the level of
abstraction these notations introduce.
This thesis presents a translation framework that converts security protocols defined as
Choreographies, which is an extension for Alice and Bob notation, into models suitable
for automated analysis with Tamarin Prover and ProVerif.
The framework translates
the choreography language into a corresponding syntax based on process calculi, namely
SAPIC+, and in doing so introduces some general assumptions regarding key derivation,
substitution, as well as process initialization. The framework supports constructor/de-
structor theories, as well as reasoning modulo equational theories, such as the multi-
plication and exponentiation operations, thus allowing for Diﬀie–Hellman key exchange
verification.
The framework builds on, and extends, the translation semantics to Local Behaviors de-
fined in [1]. Finally, security properties for secrecy, weak and strong authentication are
implemented and automatically translated to Tamarin lemmas and ProVerif queries.

1
Introduction
1.1
Motivation
The Alice and Bob notation for security protocols has a succinct and expressive syntax,
which is easy to understand at a glance and is therefore also frequently used for expressing
security protocols. The simplicity of this syntax comes also with the downside of extensive
abstractions: What/Who are the different participating entities? How many of them are
there? How do they decrypt encrypted messages? The syntax leaves out many details
of what is actually going on in a protocol run, as these are not essential for the protocol
description, and we simply assume on a conceptual level that the communication between
the protocol roles is abstracted away. It then becomes obvious that reasoning about such
specifications leaves a great deal of room for interpretation.
When it comes to automatic protocol verification, Tamarin and ProVerif are two widely-
used formal verification tools for analyzing cryptographic protocols in the symbolic model,
with the goal of proving security properties and discovering possible attacks in security
protocols.
Both Tamarin and ProVerif are extensively referenced in academic works,
as well as practically used in verifying real-world protocols. For example, ProVerif and
Tamarin have been used to successfully analyze complex protocols such as TLS 1.3 (used
for HTTPS) [2], 5G EAP-TLS authentication [3], the WPA2 protocol [4] among others.
This broad adoption makes these tools highly relevant to automatic protocol verification.
Tamarin is an open-source tool for the symbolic analysis of security protocols, where
it represents protocols as multiset rewrite rules and supports unbounded verification –
reasoning about an arbitrary number of protocol sessions and messages. One of Tamarin’s
strengths is its support for rich cryptographic theories (e.g. Diﬀie–Hellman exponentiation
and algebraic equations) and even observational equivalence for privacy properties [5].
ProVerif also operates on a symbolic abstraction of cryptography, but instead uses a
representation of protocols in an applied pi-calculus and translates them into Horn clauses
for resolution-based analysis [6].
As both of these tools are based on a syntax similar to process calculi, where a protocol
modeler can define participant roles as processes that synchronize over channels, every
action a protocol participant executes must be explicitly defined. Furthermore, as previ-
ously mentioned, the tools allow reasoning about an arbitrary number of protocol sessions
and messages. All of these properties of the model cannot be expressed in the simple AnB
syntax, and therefore there is a need for bridging the gap between these types of security
protocol declarations.
In this thesis, we focus on the syntax of choreographies [1], which is an extension of
the plain AnB notation, aiming at introducing non-determinism regarding the possible
messages a participant can emit.
We propose to design and implement a translation
framework that can take specifications of choreographies and output an automatically
verifiable specification in a process calculi syntax, that can be interpreted by SAPIC+ - a
translation framework in itself that targets Tamarin, ProVerif and DeepSec as output [7].
1.2
Related work
This thesis builds upon existing work and research on the topic of choreographies, where
a formal definition can be found in [1], as well as much of the translation semantics that

noname has been implemented in [8], where we now build upon this translation tool to

1.3
Contributions
This thesis brings the following contributions:
1. We provide a translation procedure from choreographies to the process calculi syn-
tax of SAPIC+, which in turn allows automatic verification in Tamarin, or further
translation to ProVerif.
2. We enhance the choreography syntax with goal definition in an OFMC fashion, and
implement goal translations to Tamarin and Proverif lemmas/queries.
3. We define and implement a generic model that applies the defined roles from a given
choreography specification. This model bridges the gap between the abstractions
that AnB operates with and the low-level definition of agents and processes.
4. We add support for algebraic properties for the multiplication and exponentiation
operators, which are necessary for modeling protocols that use Diﬀie-Hellman key
agreement.
5. We enhance the tool developed in [8] by integrating our translation framework into
it.
We name this combined tool Choreographer, which now allows to choose the
target syntax for the translation, which can either be noname or SAPIC+ (and by
extension Tamarin and ProVerif). The implementation can be found at [11].
The remainder of this paper is structured as follows. Chapter 2 introduces the theory
background on which the translation framework is based, reviews the relevant concepts
and definitions, and presents the adaptations required to support our approach. Chapter 3
describes the implementation details of the various components of the translation frame-
work. Chapter 4 discusses the key considerations involved in bridging the gap between
the high-level Choreography syntax and the more lower-level process syntax. Chapter 5
offers a reflection on the overall journey, as well as on the results of this thesis. Here, we
also outline directions for future work. Finally, Chapter 6 concludes the paper.

finite sets of function symbols and variables.

2.3
Local Behaviors
Alongside the formal definition of Choreographies, the authors in [1] provide the definition
of Local Behaviors which summarize the local communication patterns of a single role in
a Choreography specification. Furthermore, a translation semantics is presented, which
has as previously mentioned been implemented in [8]. Despite our translation framework
closely following the referenced specification and implementation, we introduce modifica-
tions and adaptations in order to directly translate to SAPIC+ processes, rather than to
Local Behaviors. We do so because operationally, processes capture the same concept as
Local Behaviors, which is the strand of actions seen from a role’s perspective. This simi-
larity allows us to easily translate directly to the target syntax. The translation semantics
is presented in section 2.5.3.
2.4
SAPIC+ overview
SAPIC (Symbolic Applied Pi Calculus) was originally introduced as a Tamarin plugin with
the purpose of providing a convenient way of handling global state, as handling it directly
in Tamarin’s multiset rewrite rules is tedious and error prone [14]. Further research has
built upon the original SAPIC plugin, with several notable improvements, which have cul-
minated in the introduction of SAPIC+, which is a security protocol verification platform,
enabling the use of multiple verification tools as possible backends: Tamarin, ProVerif and
DeepSec [7]. SAPIC+ is also oﬀicially integrated into Tamarin, which currently allows to
specify protocols in either the original multiset rewrite rules syntax or using processes. In-
ternally, Tamarin will translate processes to multiset rewrite rules before the verification
is started. A multiset rewrite rule has the following form:
1
rule example_rule:
2
[ Fr(~k) ]
3
--[ ActionFact(k) ]->
4
[ Out(k) ]
Listing 2.7: Example of a multiset rewrite rule
A rule has a premise and a conclusion, which are separated by an arrow. The arrow can
optionally contain an action fact. These rules operate on the system’s state, which is
expressed as a multiset (i.e., a bag) of facts. Facts can be seen as predicates storing state
information [5].
This thesis focuses on implementing a translator from Choreographies to SAPIC+ speci-
fications because:
1. These specifications are directly executable in Tamarin.
2. Can be automatically translated to ProVerif and DeepSec by the SAPIC+ toolchain.
3. The process calculi syntax of SAPIC+ is conceptually similar to Local Behaviors,
where any given role process is structured as a chain of actions seen from that specific
role’s point of view. This allows us to build upon the existing work from [8].

P | Q

Messages in SAPIC+ are also terms, but in contrast to the definition of Choreography
terms from section 2.1, SAPIC+ terms are constructed over not only variables and function
symbols, but also constants (in SAPIC+, names). This is a subtle difference, but it opens
up to supporting a substitution of the type:
σ : X →T (F, N, X)
where X is the set of all variables, N the set of names, F the set of function symbols
and T (F, N, X) the set of all constructible terms. What the definition of this substi-
tution implies, is that variables can be seen as black boxes containing terms, which is
an accurate abstraction of what received messages are. Further checks must be applied
(i.e. substitutions) to establish the validity of a given message in the specific context of a
protocol.
Special attention must be paid to the specifics of let bindings, as there are 2 relevant
aspects to take note of. let bindings allow pattern matching:
1
let pair(a1,=a2) = x in P else Q
Listing 2.8: Pattern matching
Here, SAPIC+ will try to decide if the variable x matches the provided pattern, in this
example, a pair with 2 arguments. It will furthermore compare that the second argument
equals an already known variable a2. If both checks succeed, then the first argument of
the pair will be bound to a new variable a1.
Deconstructing a term is only allowed in let bindings:
1
let message = sdec(ciphertext , key) in P else Q
Listing 2.9: Deconstructors in let bindings

For example a deconstruction operation of a term can be a decryption of a ciphertext.
SAPIC+ will apply the rewrite rules it knows in order to attempt a deconstruction, and if
it fails, it is to be handled in a separate sub-process. In the case of success, a new variable
is bounded to the result of the operation. Finally, note that in the case of both in and
out, if only one argument is supplied then the public channel is assumed by default.

In order to declare a function private, the [private] tag can be appended.
• Equations. This section is meant for user defined rewrite rules, similarly to corre-
sponding section of the Choreography specification.
• Builtins. This section allows to include predefined equational theories, alongside

revealing-singing, diﬀie-hellman. For the full description of each theory, refer to [5].
In the middle part of the specification, there are the agent processes as well as the main
process definitions, which are in accordance with the syntax illustrated in section 2.4.1.
Additionally, the last part of the specification are the security goals in the form of lemmas,
written in first order logic. Here, the facts that are defined in the processes can be used,
as well as built-in predicates such as K(x), which signifies attacker knowledge.

The translation framework requires a general restriction, where we allow rewrite rule

Intuitively, this means that constructor terms under a destructor may contain at most one

a well-defined way. Because destructors may only appear at the top of a rule, and because
constructor nesting inside rules is limited to a fixed depth, every rule application derives
terms that are structurally less complex than the subject of the rule application, which
ensures that rewriting always makes progress and eventually terminates when extensively
applied on a finite set of terms. In addition, each destructor symbol is defined by exactly
one rule, so different rules can never overlap or interfere with each other. In later sections,
we describe how our unification algorithm matches rules with Choreography terms; without
the above restrictions, this problem would be undecidable in general [16].

2.5.2
Introduction of foundational algorithms
The translation semantics from Choreographies to SAPIC+ processes uses two important
algorithms which are introduced in [9], and which were partially implemented in the related
work of [8]. These algorithms were originally created for translating AnB specifications to
strands, which in turn are closely related to Local Behaviors. The two algorithms are: (1)
the composition of recipes from terms and an analysis algorithm in two phases where (2.1)
message decomposition takes place and (2.2) the generation of a complete set of checks is
performed to validate newly derived terms from received messages.
In this thesis, we have improved algorithm (1) to support equivalence classes, re-implemented
(2.1) the first phase of the second algorithm, and fully implemented the whole description
for the generation of checks, while keeping the improvements proposed in [8].

2.6
Verification of privacy properties
As we adapt and extend some elements of the tool developed in [8] (such as the parser/lexer,
some parts of recipe composition and check generation), we provide a short summary of
the work, where a translation framework is designed and implemented for privacy proto-
cols written in the Choreography specification, targeting noname - a verification tool for
(Alpha, Beta)-privacy in security protocol (see [18]).
From [8] and [19], (Alpha, Beta)-privacy specifies 2 formulas in Herbrand Logic, where
α summarizes the knowledge that is explicitly released within a protocol and β is all the
knowledge that the intruder has derived. The main idea behind (Alpha, Beta)-privacy is
then that there is no information in β that is not already present in α, meaning that the
intruder has not derived something that was not explicitly made public.
The translation is realized in 2 steps, first from Choreographies to Local Behaviors based on
[1] and then further to noname+, which is an enhanced syntax to support the translation.
A series of extensions to the Choreography syntax are added as well, in order to express
privacy properties, which is not possible by default.

A Choreography specification only defines the message flow between two or more distinct
participants and completely leaves out any circumstantial details. Since our translation
framework can construct a valid role process for each participant in the Choreography,
for a SAPIC+ specification to be valid, we need to also decide how the translated pro-

Choreography specifications:
Meaning of Choreography roles.
Roles can be interpreted as sort of templates,
because they easily can be associated with a fixed flow of messages that can be reproduced
by one or more protocol participants.
This concept maps perfectly to the SAPIC+
processes, which have the possibility of instantiation through parameters.
Honest agents. From the context of Choreographies it is not directly obvious what
the nature of protocol participants is.
Process calculi on the other hand allows the
reasoning in terms of agents that perform a specific set of actions, despite that the precise
characteristics that define an agent can vary.
It is however tightly coupled with the
parameters of a process. We designate honest agents as potential protocol participants
and introduce a fresh name (constant) for each such agent, which is then used as a
parameter to the corresponding processes.
Intruder and dishonest agents. The context of Choreographies does not explicitly
touch upon the concept of intruders or agents that act dishonestly. There is a subtle
difference between the two, where an intruder is an external entity to the protocol that
actively tries to compromise the security goals. A dishonest agent is a valid protocol
participant that is either compromised or secretly works to compromise the security
goals. In verification tools as Tamarin and ProVerif, the intruder is represented internally
in the tool itself, where under the process of verification any provided security goals
are attempted to be disproved. Dishonest agents can be modeled very specifically, for
example as an agent that behaves exactly like an honest agent, but with small alterations,
such as sending extra messages or omitting some of them, using compromised encryption
keys, etc. We would therefore need to define the exact behavior of a dishonest agent in the
context of the specific situation/protocol. This can of course be done by having a separate
process that behaves in some malicious way relevant to the given protocol. However,
it is not possible to express this in a generic way that is applicable to any protocol
specification. Therefore, we choose to instead treat dishonest agents as compromised
agents, whose communication with other participants is visible to the intruder. This is
achieved by leaking the cryptographic secrets of a valid protocol participant right from
the start of the protocol verification process.

4.4
Security goal specifications
The specification and handling of security goals is inspired by the automatic verification
tool OFMC (Open-Fixed Model Checker) which is described in depth in [10]. The reason
for this choice is that OFMC works on AnB specifications, which are very similar to the
Choreography specifications.
The supported security goals are secrecy and authentication. In [13], Lowe formalizes
several definitions of authentication with different degrees of weakness: aliveness, weak
agreement, non-injective agreement and injective agreement and then lifts these definitions
to ensure recentness. The authentication goals used here are the 2 latter, which we also
refer to as weak and strong authentication.
The implementation is realized in two parts: (1) ensuring correct traces by emitting events
and (2) constructing standard Tamarin lemmas and ProVerif queries that operate on the
emitted events for each of the specified security goals.

This is due to ProVerif being less flexible in the expressiveness of its queries, specifically
because of certain limitations imposed by the underlying Horn clauses that ProVerif uses.
[7] describes in detail the diﬀiculties of translating Tamarin lemmas to ProVerif queries.
For example, ProVerif permits only one quantifier alternation in reachability queries. More
specifically, a query must be of the form F1@i1 ∧· · ·∧Fn@in ⇒φ, where φ is a quantifier-
free trace formula and its disjunctive normal form (DNF) does not contain negations of
facts F@i. Unfortunately, neither does the attacker fact from ProVerif correspond to the
KU fact from Tamarin, specifically when it is existentially quantified. These queries are
therefore not equivalent:
query i, j : time; event(A)@i ⇒attacker(a)@j.
∀i. A@i ⇒∃j. KU(a)@j
This is due to the different semantics between Tamarin and ProVerif, where KU(t)@j
holds when the attacker did deduce t at timestamp j (e.g., to execute an input on a public
channel); whereas attacker(t)@j holds if t is deducible at the timestamp j. This then
means that two KU facts cannot hold at the same timestamp but two attacker facts can.
In our case, only the weak authentication goal lemma can be directly translated to a
query, and the reader might have noticed that the respective lemma is annotated with

furthermore parametrized with a second parameter for the identity of the agent acting as

duced. In order to add support for automatic verification of more traditional classes of

as preventing regressions during ongoing development, where we adopted an incremental
approach, extending our translator step by step. Each new feature was developed together
with test cases and validated on relevant protocol examples. The tool is meant to be
extensible, so that other future translation frameworks can be added to the tool, and
make use of existing logic if needed.
5.5
Future work
The translation framework presented thus far can be regarded as a proof-of-concept of
sorts, as it supports mostly essential features needed for the translation of Choreographies
to work. There are therefore a great deal of improvements that can be covered in eventually
further work on the Choreographer tool. We mention some of them here:
Implementing mutual exclusion based on the lock/unlock syntax. Choreogra-
phies already support locking a memory cell for exclusive access. This is supported by
SAPIC+, and therefore it is an obvious improvement that is relatively straightforward to
implement.
Support for annotating destructor function symbols. We do not currently anno-
tate functions in the target specification with [destructor]. This is relevant for custom
defined rewrite rules, so the destructor function symbols are correctly interpreted by
SAPIC+.
Different security goals for different branches. Currently, the security goal spec-
ification works well for protocols without branching. It is a proposed improvement to
support different security goals for different branches, as these can contain different in-
structions. This would offer granularity over the different scenarios that can unfold in
the Choreography.
Improvement of read and write operations. We have but briefly touched upon the
use cases of reading from and writing to cells during the present work. There is therefore
an opportunity to explore more in depth the capabilities of SAPIC+ in this regard, as
there might be missing nuances that can be integrated with the translation framework.
Optional optimizations. The optimizations presented in section 3.6 are not manda-
tory, and it can be useful in some situations to disable them with the use of a parameter
in the command line.
Trusted roles. Currently, there is no way of defining a trusted third party, that is
constant and not replicated. There are many protocols that use this, and support for it
should be added.
Support for strings. The Choreography syntax does not allow the use of text in the
form of strings directly in the specification. This is, however, something that SAPIC+
A Choreography Translator for Automatic Protocol Verification
51
==PAGE:60==
supports, and it would be a great feature for helping the expressiveness of the syntax.
Advanced features. SAPIC+ has several advanced features such as isolated execution
environments, modeling timeouts, natural numbers, etc. Extensions for the Choreogra-
phy specification can be done to add support for some of these more advanced features
from SAPIC+.
DeepSec. We have focused only on protocol verification using Tamarin and ProVerif,
however, SAPIC+ supports a direct translation to DeepSec, which has been left unex-
plored.
Alternative process initializations. Additional ways of initializing SAPIC+ specifi-
cations are possible, which represents an opportunity for exploration. Particularly, there
might be better ways of initializing the process for specific scenarios. Adding support
for choosing a suitable initialization type would be a great addition to the translation
framework.
Privacy properties. Further work can be done for the verification of privacy properties
as suggested in [8], and integrating them with the present Choreographer tool.

6
Conclusion
The work presented in this thesis lays a foundation for automatic protocol verification
based on Choreographies, targeting two of the most widely used verification tools: Tamarin
and ProVerif. A formalization of the translation procedure has been presented, with an
in-depth discussion of the theoretical concepts. This is an important contribution, as it
aims at making formal verification more accessible to protocol designers by providing new
tooling and enabling the modeling of protocols in terms of Choreographies. We further-
more enhance Choreography specifications with support for security goal definitions, which
are fundamental to the verification process. Last but not least, we add support for some
algebraic properties for the multiplication and exponentiation operations. This paper thus
contributes with a formal definition and an implementation description for enabling the
said support, which in addition allows further extensions with other equational theories
in the future.
We have also looked at implementation details for the tool and discussed relevant design
choices in order to illustrate the main thoughts behind the development process. Here,
we have shown how the theoretical aspects and definitions are translated into concrete
procedures that are executable.
The resulting framework is successfully evaluated by verifying the Needham-Schroeder
public-key and Needham-Schroeder-Lowe protocols. An attempt is made to verify the
envelope protocol, which is partly successful. Despite the achievements, there is still room
for improving the performance of the verification process, especially when verifying more
complex protocols. All in all, we confirm the reliability of the developed tool and validate
its correctness.
We continue by covering the main challenges presented by bridging the gap between a high-
level specification like Choreographies and the more low-level syntax of process calculi,
tackling the abstractions inherent in Choreographies in a generic and correct manner,
which is an important contribution in itself. Next, we identify the main avenues for future
work on the translation framework.
Overall, this thesis demonstrates the feasibility of generating verifiable SAPIC+ speci-
fications directly from Choreography specifications. We conclude that this lays a good
foundation for further research and improvements on the topic.
