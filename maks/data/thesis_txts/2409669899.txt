Abstract
This project describes and documents the phases in order to develop an Internet
of Things (IoT) system for harbours to ﬁnd a free spot. It contains a description
of IoT solutions (SIGFOX, Wi-Fi Halow, LoRaWan, NB-IoT) and presents use
cases for the given project. The design, implementation, ﬁeld test and simulations
are made for Long Range Wide area network (LoRaWan) and NarrowBand IoT
(NB-IoT) and underlines which of these technologies are relevant to use in a
Danish harbour environment. The work evaluates two types of sensor technologies
(IR, Ultrasonic) and compares LoRaWan and NB-IoT in terms of coverage, cost,
capacity and network management. In order to see how the system works when
many devices are connected to a base station for LoRaWan and in relation to
investigate the coverage of NB-IoT, simulation of these conditions has been made
in Matlab. The ﬁnal system consists of an end-device, containing a IR-sensor
or an Ultrasonic sensor, and either a LoRaWan module or a NB-IoT module.
In addition, two servers have also been used to handle the data as well as a
mobile application to display the data in a sensible manner. The system design is
evaluated in terms of analysis, ﬁeld tests and simulations, which is implemented
by using Arduino, python and MQTT. The results show that, both LoRaWan and
NB-IoT are useful solutions for the Danish harbour environment, and the choice
depends on the cost, when NB-IoT is commercially available in Denmark. It is
concluded, if cost and coverage fulﬁl the requirements for the IoT harbour system,
NB-IoT is preferred based on the network management. However, based on the
technological possibilities and regarding this project, LoRaWan is the solution at
the given time.

Introduction
The monitoring system for harbours in Denmark is based on manual power, and this
project addresses the challenge that sailors face when trying to ﬁnd a free spot in
harbours, by using a IoT technology. The project is focusing on how it is possible
to implement a sensor based monitoring system in harbours. By implementing such
a system both the sailors and harbour masters, will have a better overview of what
happens in the harbour, without having to scan the harbour physically from one end
to the other. From the sailors’ point of view, the system makes the "ﬁnd a free spot"
process easier, because by using a mobile application the sailor can check the status of
the harbour without entering. From the harbour masters point of view the everyday
life becomes easier, because instead of controlling each gangway manually, the harbour
master gets real time data from each device, which is controlled from an application.
In this project, relevant Low Power Wide Area Networks (LPWAN) technologies is
highlighted, in order to ﬁnd the technology that suits this environment best. This is
done by highlighting pros and cons for each technology. Four diﬀerent technologies are
covered in the theory section, which are: NB-IoT, Wi-Fi/HaLow, Sigfox and LoRaWan,
which all are relevant technologies developed for IoT systems. Other technologies like
Zigbee and Bluetooth meet some of the IoT requirements such as low energy consump-
tion and low cost, but in comparison to wide coverage, this requirement is still not met
with these technologies.
A prototype is built with an Arduino board as a base, with the two fundamentally dif-
ferent technologies (LoRaWan and NB-IoT), to compare unlicensed and licensed band.
The device controls the sensor to detect whether there is a boat or not at a spot in the
harbour. This means that there is also a need of relevant sensor technology in order to
detect the boat. For developing a reliable device, the project has also tested two kinds
of sensor technologies, which are used to detect a boat.
The prototype has been tested in a harbour in Denmark (Svanemøllehavn), in order to
gather information in relation to the communication aspect and stability. This is done
because it is important to see how water and the harbour environment (e.g. masts,
boats, reﬂections) aﬀects the signal, and thereby how to optimise.
In addition to get a complete system, a mobile application has been implemented too,
to get the full impression on the idea and solution. The application has been imple-
mented in another course and is not described in detail in this report. It has not been
the main focus for the project, the application is only used for illustrating the system.

The application has been slightly adapted in the implementation for this project.
In order to scale the system, a simulation has been made. Due to limitations regarding
the number of devices, a complete test in the harbour has not been possible. It has
only been possible to test few devices, to see how they act and communicate. For the
ﬁnal solution, the devices have to be implemented at each spot in the harbour, and
therefore it will approach several hundred devices, that should be able to communicate
without loss. Therefore, it has been necessary to make a simulation of the complete
system for one harbour.
Overall the main focus of this project has been to ﬁnd the best solution regarding
wireless communicating in terms of coverage in harbours, where a lot of interfering
objects are present, and at the same time a lot of devices should be connected to the
system. NB-IoT and LoRaWan has been tested and compared. These two technologies
have been chosen because they both are focusing on IoT solutions, and it is then a
licensed (NB-IoT) and an unlicensed (LoRaWan) band which have been tested. They
are also diﬀerent in the setup and how to control the system, because there is a need
of a gateway for LoRaWan, where the NB-IoT uses the existing Long Term Evolution
(LTE) networks with modiﬁcations, which gives the network operators the coverage
responsibility.
In relation to the requirements and expectations of the project, stability and reliability
are highly prioritised, because the data from each sensor may not be lost. Since the
people around the harbour want to trust blindly on the system, it is necessary that
all devices are online and give the right status of each spot. This also applies to those
guests who are interested in ﬁnding a place in the harbour. If a boat leaves the spot,
it is not necessary to update the status for the spot within milliseconds, so the delay is
not highly prioritised as long as the update takes place within seconds.
For scaling, the goal has been to show how it will aﬀect the technologies when there are
more sensors. The goal is to cover harbours with 100 to 1500 spots in Denmark, which
is the relevant size of a harbour for the system. The simulation for LoRaWan focuses
on the amount of traﬃc and collision rate, and the simulation for NB-IoT focuses on
coverage. Based on the simulations, the theory and the ﬁeld tests, the pros and cons
for each technology will be highlighted. In ﬁgure 1 is a rough sketch of how the system
is connected, and what it contains. The mobile application is not fully implemented in
this project but is connected to visualise the project and complete system.
It should be mentioned that there is already a company called Compusoft [2] dealing
with intelligent harbour systems, but the solution they use is complex, expensive and

Theory For IoT Solutions
In recent years, the concept of Internet of Things has become a major focus area for
many diﬀerent things. In this way, the interest in Machine-2-Machine (M2M) commu-
nication has also increased. M2M communication connects devices directly and forms a
network that can exchange information and perform actions without the use of human
instructions. This network is an essential part of the IoT concept, which can use the
data in various applications. In order to meet the requirements of the IoT system,
there are some diﬀerent conditions to be met, such as wide coverage, low energy con-
sumption, low cost and relatively low throughput. Among these, especially low energy
consumption and wide coverage are the major requirements, as the IoT devices can be
located in areas and environments without main power and operate only on battery
power. In addition, these devices will be located in areas that are diﬃcult to reach
because it is either underground or deep inside buildings or because they are remote.
These requirements cannot be solved by the existing technologies designed for human
consumption, such as cellular mobile networks.
Due to the diﬀerent limitations of the mentioned technologies, LPWAN have been de-
signed to meet the IoT requirements for low energy consumption and wide coverage.
One important thing about these LPWAN technologies is that they all provide a trade-
oﬀbetween the battery life, data rate and costs. These trade-oﬀs are acceptable for
most M2M applications that do not require high data speed and low latency. This is
the signiﬁcant diﬀerence between LPWAN networks and traditional wireless commu-
nication. There are diﬀerent providers that handle M2M communication and some of
them will be highlighted below. [3]

2.2
LoRa
Longer Range (LoRa), is the physical layer or the wireless modulation utilized to cre-
ate the long range communication link. LoRa is a modulation technique that oﬀers
signiﬁcantly longer range than the traditional telecommunication provider. The LoRa
technology is based on a variety of Chirp Spread Spectrum (CSS) and spread spectrum
techniques with an integrated Forward Error Correction (FEC). [4] CSS is a signal which
either increase or decrease with frequency in time. This technology is very common in
radar and sonar technology but can also be used in spread spectrum. CSS signals have
a constant amplitude and uses the whole bandwidth in a non-linear or linear way from
one end to another end in a given time.
In LoRa modulation the spreading of the spectrum is done by generating a chirp signal
that continuously varies in frequency. By using this method, the frequency oﬀsets and
timing between transmitter and receiver are equivalent. [5]
Each LoRa symbol is built by 2SF chirps, where SF is the corresponding Spreading
Factor (SF). By using six diﬀerent orthogonal SFs in the range of 7 to 12, it is possible
==PAGE:19==
2
THEORY FOR IOT SOLUTIONS
7
to provide diﬀerent Data Rates (DRs), which will result in eﬃciency and increased
network capacity. As mentioned earlier in the section, LoRa modems also uses FEC,
which adds a small overhead to the message, and thereby providing recovery features
against bit corruption. Diﬀerent Coding Rates (CRs), from 4/5 to 4/8 is used in order
to implement FEC. [6]
The LoRa modulation is the Physical Layer (PHY), which only contains the link layer
protocol, meaning it is perfect to be used in Peer-To-Peer (P2P) communications be-
tween nodes.
2.3
LoRaWan
Build on LoRa modulation is LoRaWan, which is operating on data link layer(MAC).
The LoRaWan also includes the network layer so it is possible to send information to
any base station that already is connected to a Cloud platform.
LoRaWan deﬁnes the system architecture for the network and the communication pro-
tocol, while LoRa on the physical link enables long range communication. In order
to determine the network capacity, Quality of Service (QoS), the security and battery
lifetime of a device the protocol and network architecture have the most inﬂuence.
In the LoRaWan network devices is not connected to a speciﬁc gateway. This means
that when data is transmitted by a device, the data is typically received by multiple
gateways. Each gateway will forward this data from the device to the cloud service
through the backhaul network. By having these cloud-based network servers all the
intelligence and complexity is pushed to the network server. These servers will manage
the data, the network and perform security checks, ﬁlter redundant received packets,
perform adaptive data rate and much more. If a device is moving or mobile, there
will not be a handover from gateway to gateway, as know from the traditional cellular
networks. [7]
2.3.1
Battery Lifetime
The devices in a LoRaWan network are asynchronous, which means that they com-
municate when they have data ready to send. This type of communication protocol is
similar to the Aloha protocol [8], where the device is not interested in external data
but only focuses on the data the device should send. In synchronous network which
can be a mesh network, the device frequently has to ‘wake up’ to synchronise with the
network and check for messages. By waking up over and over again, the device will use

2.3.2
Network Capacity
The gateway in a LoRaWan network must have a very high capability to receive mes-
sages from many diﬀerent devices. By using adaptive data rate and by using multi-
channel multi-modem transceiver in the gateway, so that all messages on diﬀerent chan-
nels can be received, it is possible to achieve this high network capacity for LoRaWan.
In relation to capacity, there are diﬀerent factors that can aﬀect it: The number of
current channels, how often a device transmits, data rate and payload length. As men-
tioned earlier LoRa is a spread spectrum based modulation, and thereby the signals
are orthogonal to each other when diﬀerent spreading factors are utilised. Since the
spreading factor can change, the eﬀective data rate also can change. By being able to
receive multiple diﬀerent data rates on the same channel at the same time, the gateway
takes advantage of this property. If a device is close to and has a good link to the gate-
way, there is no reason to use the lowest date rate and ﬁll up available spectrum longer
than it needs to. By using a higher data rate, the time on air will be shorter, opening
up for more potential space which can be used by other devices. The battery life is
also optimised by using an adaptive data rate. By using these features, the LoRaWan
network can reach a high capacity and by introducing Transfer Packet With Conﬁrmed
(TPWC) make the network stable. This network can be deployed in environments with
minimal amount of infrastructure, and when more capacity is needed more gateways can
be deployed shifting up the data rate and thereby reducing the amount of overhearing
to other gateways. [9]

2.3.4
Devices
Regarding devices in LoRaWan network, there are some diﬀerent end-devices serve
diﬀerent applications and have diﬀerent requirements. In order to meet a variety of end
applications, a LoRaWan network has diﬀerent device classes. These classes depend on
network communication versus battery life. [13]
• Class A (Bi-directional end-devices): Class A devices use bi-directional com-
munications, which means that each end-devices’ uplink transmission is followed
by two short downlink receive windows. Class A devices have the lowest power
consumption and is used for applications that only require downlink communica-
tion from the server after the device has sent an uplink transmission. Downlink
communication at any other time will have to wait for next uplink transmission.
• Class B (Bi-directional end-devices with scheduled receive slots): Class
B devices open extra receive windows at scheduled times. The devices receive
a time-synchronised beacon from the gateway, with information about open its
receive window at the scheduled time. By doing so it is possible for the server to
send messages to the devices when it is listening.
• Class C (Bi-directional end-devices with maximal receive slots): Class
C devices can receive messages almost any time, only closed when transmitting.

harder to guarantee QoS, since the number of diﬀerent networks can create collisions.
Having a closer look at the 868 MHz band, the band is split into 6 diﬀerent sub carriers
where diﬀerent rules applies.
There are two limitations in relation to this band: [14]
• Transmission power - which is the maximum power a device can use on the channel
when it is communicating. The usual power of a device is 23 mW (14dB).
• Duty cycle - which is the maximum ratio of time on the air set to 1% per hour.
This limitation means that the device only can speak for 36 seconds per hour.

Security
Devices in LoRaWan have a 64-bit unique identiﬁer Device Unique Hardware ID
(DevEUI), which is assigned to the device by the manufacturer. All the communication
with the device is done with a dynamic 32-bit Device Address (DevAddr), where

devices. This procedure is called Activation and following two methods:
• Over-The-Air-Activation (OTAA): Is the most secure way to connect with
LoRaWan. When a device wants to connect to the network, it performs a join
procedure with the network, where a dynamic DevAddr is assigned and security
keys are traded with the device.
• Activation-By-Personalization (ABP) In this method the DevAddr
and the security’s keys needs to be hardcoded into the device. In this method

Looking at the complete system, three diﬀerent distinct 128-bit keys is known. The

When a device connects to the network, a Network Session Key (NwkSKey)
and an Application Session Key (AppSKey) is generated. The AppSKey is

The physical NB-IoT channels are normally multiplexed in time, and each NB-IoT

Each channel has a speciﬁc function, for how to use the network and connect the NB-

occupies 6 PRBs, which does not ﬁt with the NB-IoT structure, which only spans over
1 PRB . NPSS is transmitted in every 10 ms frame in subframe number 5, which then
will use the last 11 OFDM symbols in the subframe. The NSSS is transmitted every
20 ms in subframe 9 and are also using the last 11 OFDM symbols in the subframe.

and is transmitted in every frame. This block is used for providing the most important
information about the network. It carries information about the bandwidth, structure
and frame number, which is just like for LTE, where the broadcast channel also is used
for only the MIB. [27]

and then the network replies with a contention message to solve problems with multiple
devices using the same preamble from the ﬁrst step in the uplink setup.
The NB-IoT is supposed to be as simple as possible, but also to gather reliability, which
is achieved by using HARQ. Because of NB-IoT is so simple and limited in bandwidth,
the system only allows one HARQ process in both uplink and downlink. The HARQ is
also used in LTE, where the receiver is able to buﬀer information if some of a packet

Sigfox
Another provider for the IoT segment is SIGFOX, which provides an end-to-end solution
for a communication chain. SIGFOX resembles LoRa in many ways, but there are still
some things that diﬀerentiate them. SIGFOX is a narrowband technology, which uses
the method Binary Phase-Shift Keying (BPSK) as radio transmission. SIGFOX takes
narrow chunks of spectrum and encode the data by changes the phase of the carrier
radio wave. In this way, it is possible for the receiver to only listen in a tiny slice of
spectrum, and thereby bring down the eﬀect of noise. In order to do so it requires
an inexpensive endpoint device and a more sophisticated base station to manage the
network. Looking at the big picture of SIGFOX, the company owns all its technology
regarding the gateway, software and backend for the cloud service, versus LoRa which
is open source. SIGFOX looks at smaller amount of spectrum than LoRa, and thereby
have less interference, but on the other hand SIGFOX is strictly related to the duty
cycle in EU compared to LoRa. [32]. Since SIGFOX is monitoring the network like
the operators in NB-IoT, and uses the unlicensed band, which is the same band as
LoRaWan, the SIGFOX technology is not included in the project, even though IoT
Denmark claims SIGFOX provides a coverage of 99,7% of Denmark. [33]

Wi-Fi HaLo(w)
Wi-Fi Halow or Wi-Fi Halo, which is actually the right way to pronounce it, but because
of the big focus of low use of data, energy and so on, it has become Wi-Fi Halow. This
is also a technology which has been developed especially for the IoT devices based on
existing Wi-Fi standard. It is an extension to the IEEE standard 802.11ah, which are
using the unlicensed 900MHz band, which can cover a lot more compared to standard
Wi-Fi, which uses the 2.4GHz and 5GHz bands. The focus for this technology is the
same as all other IoT technologies, which is to minimise the power and cover a lot of
devices in a simple way, which is essential for IoT devices. Normally the devices are
simple, which sends simple and small amount of data. The Wi-Fi Halo is based on Wi-
Fi and it will be possible to connect devices easy through a router, just like standard
Wi-Fi. It will also be able to use the same security as already known from the current
Wi-Fi, which also gives an advantage when connecting the devices.

2.7.1
Ultrasonic distance sensor
In order to check if there is a boat on the spot or not, an ultrasonic sensor is tested. In
this project, the HRXL MaxSonar WRS [36] is used. The way this sensor operates is by
ejecting sound, and from the time the sound is reaching an object and being reﬂected
back, the distance is calculated. The sensor consists of a speaker that generates an
audio source and a microphone that detects the audio source. Furthermore, the sensor
is waterproof and can operate from -40 C to + 65 C, which means the sensor can handle
the weather conditions in Danish harbour, regarding temperature and storm surge.
An ultrasonic sensor is based on measuring the sound waves at a frequency above
human audible range, often at 40 kHz. They are typically operating by generating a
high-frequency sound and receiving and evaluating the incoming echo pulse.
The Ultrasonic sensor is based on three physical principles:
• The Doppler eﬀect
• The attenuation of sound waves
• Time of ﬂight
In this project, the time of ﬂight mode is used and therefore only this principle will be
highlighted.
In this mode, an ultrasonic transmitter outputs a short burst of sound in a certain
direction. The pulse reﬂects oﬀa target and returns to the receiver after a certain time.
The receiver takes the length of this time interval, and calculates the distance r based
on the speed of sound c: [37]
r = c ∗t
Depending on the distance, the calculation of the distance is relatively quick. However,
two factors aﬀect the ﬁnal result.
• Weather: Humidity and temperature aﬀects the speed of sound in air. In relation
to meeting this problem, the HRXL MaxSonar WRS has auto calibration. This
means each time the sensor takes a range reading, it calibrates itself. [38]
==PAGE:34==
2

22
• Angle: In order to receive the echo pulse, the target surface must be placed
within a certain angle so that the sound reﬂects back to the sensor. This may
cause some problems as the front of a boat is relatively tapered, and therefore the
sound is split rather than reﬂected, which is highlighted in section 4.3.3 too.
2.7.2
Infrared distance sensor
There are a lot of diﬀerent Infrared (IR) sensors, and for this project the
SHARP GP2Y0A710K0F [39] is used, which can measure a distance from 100 to 550
cm. IR distance sensors use infrared wavelength to detect the distance to an object. It
returns an analogue voltage, which can be used to determine the distance to the object.
The SHARP GP2Y0A710K0F sensor has an IR LED with a lens, which emits narrow
light beam.
This light beam is reﬂected from the object, and then the light beam
is directed through the second lens in the sensor, which is a Position-sensible Photo
Detector (PSD). The conductivity of the PSD is converted to voltage, and when the
voltage is digitalized by using analogue-digital converter on a logic board, the distance
is calculated. At ﬁgure 12 an illustration of how the beam is send and reﬂected can be
seen. [39]

The output voltage of the distance is inversely proportional, meaning that when the
distance is increased the output voltage is decreased. This gives a linear relation, which
can be seen in ﬁgure 13 [40]. This is used in section 4.3 to convert the digital signal to
a distance, when the sensor is connected to an Arduino.

This section is focusing on the second server and the connection to TTN and how the
data is passed from TTN to the database. To get the information from TTN to the
private server, the MQTT protocol has been used, which TTN supports [63].
The
private server has been setup to subscribe to the given topic at the TTN, and thereby
gather information every time there is activity at the subscribed topic. Each device at
TTN has its own topic, and thereby it is possible to get data from each speciﬁc device.
At the private server, the python script application.py is implemented, which
uses the paho MQTT library to use MQTT in python. The Application ID and the
Access key for the application created at TTN, are set manually in the python script
by using client.username_pw_set("Application ID", password="Access key application
TTN"). The client connects to the application at TTN by using
client.connect("eu.thethings.network", 1883, 60). The ﬁrst input is the Handler for
the application at TTN, which is chosen at the creation of the application. 1883 is the
network port of the server, which is used [63], and last input is the time to live input,
which sets the maximum length for the communication with the broker at TTN [64].
The private server subscribes to the topic, reads the payload message and prints the
payload and a timestamp in a .txt ﬁle [64]. This setup is used for the sensor and ﬁeld
tests, and for the complete system, the server makes an API call based on the payload
message. If the payload message contains "UPDATEX101X101X1", (which means a
boat has arrived at the spot for the given device) the script updates the status in the
database, by calling the PHP script boatUpdate.php?Spot_ID="SPOT_ID"&Harbour
_ID="HARBOUR_ID"&Status="STATUS" with the values for the given sensor. All
inputs for the API are given by numbers and the PHP script updates the database,
with the given status.

DISCUSSION

8
Discussion
When a communication system is designed and implemented, there will always be pri-
orities to be made. Such priorities will always be discussed, as there will always be pros
and cons. The pros and cons for this project is discussed below.
Looking at the capacity aspect for LoRaWan, it is shown from the simulation that the
system could support 1500 devices that sends 20 messages per day at a size of 10 bytes,
with a collision rate of 120 messages. This setup shows a worst case scenario, as the
simulation requires that there is 20 boats sailing out and in at each spot every day,
however this case will rarely be a scenario in real life. In order to get a better view of
how the everyday collision rate would be, a simulation for this case has also been made.
This simulation shows that if 300 devices are active during the day and the need for
messages is only 6 per device, the total collision rate is maximum 12.
The result from the simulations shows that even the number of collisions in the worst
case scenario is accepted. Additionally, by introducing Transfer Packet With Conﬁrmed,
the lost messages are retransmitted. A possible challenge for LoRaWan may be the use
of the unlicensed radio spectrum, and thus speciﬁc duty-cycles on the devices. In order
to investigate how the duty cycle aﬀects the communication aspect of this system, a
calculation of a ﬁxed size of the packages needed in this system has been performed.
The result shows that using the highest SF 12, the total number of messages for each
device is 3164 messages per day. This number is far higher than the worst case scenario
in the harbour, and therefore the duty cycle is not a problem in relation to this system
Regarding the coverage for LoRaWan, it can be seen from the theory that the expected
coverage in urban environment is approximately 6 km. However, the ﬁeld test that
was carried out for the environment around Svanemøllehavn shows that the coverage
could reach up to 1 km. This result is not nearly close to the theoretical distance, but
it should also be mentioned that the height of the base station in the ﬁeld test was
around 2 meters, which is far from the 30 meters used in theory. Obviously, the height
of the base station aﬀects the coverage as the signal risk having to travel through far
more objects, compared to a higher base station, where the signal will be lifted up over
many of these objects.
The simulations of NB-IoT shows diﬀerent coverage, depending on if it is for the setup
or theoretically values, and if the Okumura-Hata or Extended Hata are used. This
leads to the question; Why is there so much diﬀerence between the results? Looking
at the setup and theoretical scenario, where Okumura-Hata is used, the diﬀerence is
==PAGE:102==
8
DISCUSSION
90
the transmit power and receiver sensitivity. These two values are the key factors for
the coverage. The high receiver sensitivity for the module and the high transmit power
for the TDC base station in Guard-Band, gives a long coverage for the setup compared
with theory, in relation to downlink. The big diﬀerence in uplink and downlink is based
on the receiver sensitivity, which is higher for the module than the base station.
Theoretically there are a lot of diﬀerent values for the end-devices and base stations,
regarding NB-IoT. The values in this project are collected and linked as reliable as
possible, based on source and values.
The simulation for the Extended Hata and Okumura-Hata leads to a discussion, whether
the Extended Hata is better than Okumura-Hata? This is very diﬃcult to say, but based
on the results of the simulations, it seems that the Extended Hata is more reliable for
a NB-IoT setup than Okumura-Hata. For downlink the Okumura-Hata shows a cover-
age of around 30 km and the Extended Hata shows around 15 km. The 15 km in the
Extended Hata is equal to the theory, and 30 km seems unrealistic, based on both ﬁeld
test and theory.
The simulation of NB-IoT also shows a coverage around 10 km for the setup in uplink,
but in the ﬁeld test it has only been possible to achieve a coverage around 2,5 km, which
leads to the question, why is there so big diﬀerence? The setup with the Sara-N210
module, which has been used for the ﬁeld test has some possible interference factors,
which could be why the coverage in the ﬁeld test is very low. The wires that are con-
nected to the module, have no resistance and are not well isolated, which could lead
to a lot of interference. Due to lack of capacitors, it has not been possible to connect
capacitors, which should give a more reliable signal, as described in the system inte-
gration manual. The antenna at the module could also be a reason for low coverage.
Due to a lot of challenges to set up the module, with limited time, the antenna had
just been created as simple as possible. Since it is placed close to the other wires and
the sim card, it could create interference too, and interrupt the signals. Based on the
results from the ﬁeld test, the module is not able to send or receive understandable
messages, when the signal power gets lower than -100 dBm, probably because of too
much interference.
Two diﬀerent sensors have been tested, in order to deﬁne which type is best suited in a
harbour environment. The results show that both the ultrasonic sensor and the IR sen-
sor can be used in this system. However, the ultrasonic sensor is worth selecting as this
device is water resistant and generally more stable compared to the IR sensor. Light
from the sun, and reﬂections from the water, aﬀects the IR sensor measured distance
==PAGE:103==
8
DISCUSSION
91
and makes it more unstable. In addition, the results also show that when a boat leaves
a spot, the distance is set to the last measured value by the IR sensor, and this means
that it is not that reliable when the boat leaves. On the other hand, the IR sensor is
cheaper compared to the ultrasonic sensor.
LoRaWan and NB-IoT work diﬀerently regarding network structure and bands, which
gives them advantages and disadvantages. Looking at LoRaWan, there is a need of a
gateway, probably in each harbour, since the total number of LoRaWan gateways may
still be few in Denmark, leading to an eﬀort implementing gateways by the project
owners. The gateways are public, and can be used by others, this means if a potential
user of LoRaWan equipment is near an already existing gateway, the user may use this
gateway and hereby there will be more traﬃc for the gateway. The gateway needs
access to the Internet, and after connecting to the LoRaWan platform the gateway will
be ready to use. For NB-IoT there is no need of investing in gateways for the project
owners, because the base stations, which operators already have deployed, are used for
NB-IoT. Instead there is a need of a sim card in each device to authenticate at the
network, and hereby making the operators responsible for network management. On
the other hand, if the base stations do not cover an area, it will not be possible to use
NB-IoT. In this case LoRaWan is more ﬂexible, because it is possible for everyone to
set up new gateways and get coverage if there is access to the Internet for the gateway
nearby.
When it comes to choosing the right IoT-solution in relation to this project, it would
probably be more reliable using NB-IoT, since the network will be controlled and mon-
itored by an operator. For LoRaWan the system in each harbour is depending on the
gateway, which the project owners are responsible for. At the given time the equipment
for a solution with LoRaWan is cheaper than NB-IoT equipment, and there is no price
on sim card and data usage for NB-IoT in Denmark yet. The price is for a NB-IoT
solution at T-Mobile 6$ for 12 MB per year [81]. Even though the price is higher for
NB-IoT it will give a more reliable system, which the project owners do not need to care
about compared with LoRaWan. The discussion will lead to priorities of cost, security,
reliability and coverage, when NB-IoT is deployed by operators.

9
Conclusion and Outlook
Throughout this project, the goal has been to build an IoT system for harbours. Many
diﬀerent tools (Matlab, Arduino, python, MQTT, TTN, servers) have been used to
implement the system, to test LoRaWan and NB-IoT, and to simulate how the system
will operate in real life. Throughout the project, knowledge about IoT technologies,
sensors, servers, and simulation tools has been increased. The main challenges dur-
ing the project have been insuﬃcient information in terms of conﬁguring the NB-IoT
module (e.g. ﬁrmware, sim card, internal connections), since the technology is in a
test stage in Denmark at the given time. The sensor, which has the best results, is
the ultrasonic sensor, and based on the weather conditions in a harbour, the ultrasonic
sensor is preferred.
Based on test and simulation, it can be concluded that, the technologies do not meet the
expected requirement in relation to the theory. However, it has been possible to build a
complete system with LoRaWan, which could clearly be a candidate for a complete IoT
system in harbours. The conviction is also great for NB-IoT to be a major candidate,
but due to a lack of authentication of the private server and limited base stations, it
has not possible to forward the data to the mobile application through NB-IoT and
make a ﬁeld test in a harbour. Generally, both solutions are a clear improvement,
compared to using LTE and Wi-Fi regarding cost, coverage and complexity, and it is
believed that the complexity of authentication for NB-IoT, will be less complex when it
is commercially available in Denmark. The test results show that not only functionality
is important in an IoT implementation, but also careful considerations and priorities,
regarding system design are necessary.
The project also shows that the choice between the diﬀerent technologies depends on
many diﬀerent factors (e.g. cost, coverage, network management). In the end it all
comes down to how sensible the data is and how important it is with a strong and
robust network. Each technology has its price whether the user want a low cost and
a sensible network, compared to a higher price with better resilience, and this issue
applies in any environment when it comes to IoT systems.
Considering the situation for NB-IoT at this moment, it is diﬃcult to judge which tech-
nology is best suited for this project, because NB-IoT is not ready for use in Denmark,
and there is not any indication of the cost for a system with NB-IoT.
For this project it can be concluded, that when NB-IoT is commercially available in
Denmark, and if the requirements for cost and coverage in harbours are met for NB-
==PAGE:106==
9
CONCLUSION AND OUTLOOK
94
IoT, the choice will be NB-IoT, since network management is highly prioritised.
The project has shown that the IoT technologies LoRaWan and NB-IoT are possi-
ble solutions to optimise harbour operations, regarding monitoring the harbour. In
relation to implement the system in all Danish harbours, the IoT technology will be
the same for all harbours, the adjustment in relation to adapting the system to other
harbours will be minimal. The adjustments may be changes in width and depth at
certain spots. The changes may occur after the system has been implemented, and by
sending diﬀerent variables (e.g. measurement length), it will be possible to adjust the
end-devices.
Throughout the project it has become clear that the system can be used for other use
cases. The implementation used in this project, can be implemented in other envi-
ronments (e.g. parking, warehouse, goods and container registration) without major
changes.
