Abstract
The use of well­known protocols for achieving security in multiple digital applications is
a major deciding factor that engineers use nowadays. While multiple protocols are de­
signed for general purposes, specific areas such as Internet­of­Things (IoT) and more
generally resource­constrained devices, flounder in using these technologies due to their
resource demands which can’t comply with. What motivates researchers in designing
protocols that need fewer resources, while at the same time provide solid security. These
are the pragmatic considerations that rTLS protocol was designed. So that one could use
a variant of the TLS 1.3, a well­known used mostly in securing web applications protocol,
with limited resources that would comply to constrained devices and at the same time
provide security properties that would defend any potential attack. In this thesis project,
we are developing the first practical implementation of the theoretical design of rTLS so
that we could start measuring metrics which will prove if that protocol could achieve a
lower bandwidth overhead. One of our objectives is also to identify any potential differ­
ences between the theoretical design and the PoC implementation while also evaluate
the expected functioning of the protocol. We base on a TLS library where we modify their
TLS 1.3 implementation, by adding rTLS’s security requirements and protocol specifics.
After multiple runs of the Proof­of­Concept implementation, we were able to measure the
bandwidth overhead of rTLS, and therefore compare it with the TLS 1.3 and the estimated
numbers. The results were promising, as we prove that rTLS could be “cheaper” than
TLS 1.3, although we could also observe that in a real application, unexpected data can
appear that add in the overhead of rTLS. Up to this point, we can see that rTLS is promis­
ing, while additional research should be done in terms of security testing, storage, power
consumption so we could discover if rTLS is suitable for low constrained applications.

Introduction
1.1
SSL/TLS
The Secure Sockets Layer (SSL) protocol is one of the primary protocols that the internet
technology was introduced with having as goal to achieve security in the e­commerce,
Voice over IP (VoIP), etc. SSL or by extension TLS are dependent on digital certificates
as they utilize X.509 certificates for both authentication and public key exchange [1]. By
saying providing “security” into communication with SSL/TLS, basically, the communica­
tion is encrypted using an algorithm that the protocol supports, and at least one of the
entities of the communication is authenticated using a digital certificate. That protects the
communication from an eavesdropper or an adversary who wants to get in the middle
of the communication to alternate the exchanged information. In theory, security can be
achieved using various algorithms and techniques, thus when it comes to the application
of these techniques we cant say the same. From the protocol perspective until the library
used vulnerabilities or bugs can cause a security breach with bad consequences. In SS­
L/TLS various vulnerabilities were found through the years to certain versions, even to
the latest one the TLS 1.3.
1.2
Evolution of SSL/TLS
TLS is a successor of the initial SSL 1.0 protocol which was initially designed by Netscape
[2] back in 1994, thus it was never published due to security flaws that were found inter­
nally by the firm. The next year Netscape published SSL v2.0 in which once again, serious
flaws were identified in a short amount of time which led to the publication of SSL v3.0 in
1996 which made SSL ubiquitous and laid the foundation for what followed next. The In­
ternet Engineering Task Force (IETF) standardized the SSL v3.0 and gave it a new name
as the TLS v1.0 (published as SSL v3.1) while there were no major differences just simple
incremental fixes from the TLS over a series of updates and that’s also the reason why in
the literature is found as “SSL/TLS”. By 2006 TLS v1.1/SSL v3.2 was published with once
again minor and new capabilities until 2008 where TLS v1.2/SSL v3.4 was published and
it is recommended until now. Therefore, in 2014 the first draft of TLS 1.3/SSL v3.4 was
created and finally, the 23rd and the final draft was released in August 2018 [3]. SSL/TLS
in general, is used in various applications over the internet and it is most frequently used
in the Hypertext Transfer Protocol (HTTP) making it HTTP Secure by providing authenti­
cation, to the one side of the communication, and encryption using various cipher suites
and therefore we have (HTTPS). Another common application of TLS, is at the File Trans­
fer Protocol (FTP) and moreover, functionalities of TLS were applied by many protocols
such as Wireless Transport Layer Security (WTLS) where we can see that TLS adapts to
new environments like a wireless mobile phone network and various IoT applications [4].
1.3
TLS usage and IoT applications
TLS nowadays can be found mostly in the web surfing environment in order to provide
secure access to various websites and web services. The procedure for accessing se­
curely websites using any TLS version requires a valid certificate where website hosts
need to purchase in order to use it and authenticate their selves to the visitors. Therefore,
the number of certificates can potentially indicate the approximate number of websites
that support TLS which in 2021 is about 156,979,428 as found by BuiltWith [5], without
even considering the number of visitors of the websites. Unfortunately, a number of these

be that responsible, patching is not reliable and power consumption is crucial [9].
To suit these heterogeneous needs that present in the IoT field, new extensions of the TLS
protocol have been proposed, in order to enable extremely lightweight devices to partake
in TLS connections as well. To encounter these problems and keep the benefits of the
TLS, researchers had proposed multiple variations of TLS that are designed specifically to
serve these needs and bring the benefits of the TLS suite. Such examples are described
briefly in Chapter 3.
1.4
What is next ?
Considering all of the above, the authors of [10] and therefore supervisors of this Thesis
have designed a new variant of the TLS. This variant is called rTLS, and its main purpose
is to solve some issues of the latest TLS 1.3 by combining it with an algorithm used in the
Signal messaging application while taking advantage of TLS’s 1.3 supported cipher­suites
and provide low bandwidth overhead 1. Signal uses the Double Ratchet algorithm [11],
and hence the name rTLS. Hardening the security of TLS1.3 with the features of the
double ratchet algorithm, and decreasing the lower bandwidth overhead, rTLS would be
an ideal solution for IoT applications.
rTLS has been initially published in [10] where the authors provide a brief description
of the design and functioning. Moreover, the bandwidth overhead is presented with the
estimated transmission and storage overheads while also comparing it with the original
TLS 1.3. In this thesis, we are taking this to the next step by practically developing a
Proof­of­Concept version of the rTLS protocol while we also bench­marking it to prove
the performance factors that were calculated in the aforementioned publication. Similar
to the theoretical approach we compare the performance with an original TLS 1.3 imple­
mentation of the WolfSSL library, while we also identify some differences that were not
expected from a theoretical view but appeared in the implementation.

2.1
TLS 1.2
TLS 1.3 as mentioned in 1.2 is an evolved and improved version of TLS 1.2. The func­
tionality of TLS 1.2 targeted to provide the users with various encryption algorithms and
eventually establish secure communication. Initially, for every communication, a hand­
shake takes place where the Client and Server agree on some terms such as the TLS
version, the cipher suite, certificate exchange, sequence numbers, and other various im­
portant characteristics of the TLS structure. The handshakes consist of 10 different steps
as illustrated in Figure 2.1:

suites, a list of compression methods, and a list of extensions.

TLS 1.2 supports multiple cipher­suites which makes it “expensive” where some of them
are in addition vulnerable to cyber attacks. Besides that, the handshake procedure con­
sists of 10 different messages which make it time and power­consuming. Taking all of the
above into consideration, the RFC 1 created the new version TLS 1.3. Which was created
with fewer handshake steps and improved security [6].
2.2
TLS 1.3
TLS 1.3 was published as first draft in April of 2014 and its final version was published
in August 2018. It consists of three sub­protocols, the handshake protocol, the record
protocol, and the alert protocol. While the latter two are not as important as the handshake.
The handshake procedure is the core process of the communication and it is the case
where most modifications were made from previous versions.
TLS 1.3 Handshake
The handshake process of 1.3 involves only one round­trip, in contrast with TLS 1.2 which
has three, making a significant change to the amount of exchanged messages. Reducing
the number of cipher suites supported by the protocol decreases the bandwidth overhead
and increases the security since older cipher suites have been obsoleted and some of
them are indeed vulnerable. These ciphersuites are the RC4 Stream Cipher, RSA Key
Exchange, SHA­1 Hash Function, CBC (Block) Mode Ciphers, MD5 Algorithm, some non­
ephemeral Diffie­Hellman groups, EXPORT­strength ciphers like DES and 3DES [12].
Moreover, giving to the client the option to send his key share with the expectation of
being accepted ( the 0­RTT resumption), eliminates many steps in the case where the
server accepts it. Once the handshake is complete, the peers use the established keys
to protect the application­layer traffic [13]. The handshake is depicted in Figure 2.2:

The three phases of the handshake are:
1. Key Exchange:
The client generates the public/private key for the key exchange that the Diffie­
Hellman requires (i.e., x25519 elliptic curve). The client then sends the client random
data, the list of cipher suites that he supports, a list of public keys that the server
could use for key exchange, the Pre Shared Key (PSK), and the protocol versions
that the client can support.
2. Server Parameters:
The server sends the server random data, the selected cipher suite, a public key for
the key exchange, and the negotiated protocol version.
3. Authentication:
The client is able to authenticate the server using the certificate, optionally vice
versa, provides the generated shared keys and indicates to the server the end of
the handshake.
More briefly, at phase 1, the client sends to the server the ClientHello message which
contains:
• A random nonce ClientHello.random
• The protocol versions that the Client supports (i.e. TLS v1.0, v1.2, v1.3)
• A list of symmetric ciphers or Hash Key Derivation Function (HKDF) pairs
• A set of Diffie­Hellman key shares or a set of pre­shared key labels or both
• Any potetnial additional extensions
• And maybe additional fields or messages could be added for middlebox compatibil­
ity, making the TLSv1.3 handshake look similar to a TLSv1.2 session resumption
which misbehaving middleboxes understand and generally allow through [14]
At phase 2, the Server receives the ClientHello message and determines the corre­

2.3
Double Ratchet Algorithm
The double ratchet algorithm was designed in 2013 aiming to provide secure key man­
agement and forward secrecy to the TextSecure messaging application [29]. Nowadays
though, it is most popular in its application to the Signal messaging service that was cre­
ated after the merge of TextSecure with an encrypted voice calling application. Signals
security combines various cryptographic mechanisms along with the double ratchet algo­
rithm, i.e. Elliptic Curves, SHA256, AES [30]. In this thesis though, we are interested in
the double ratchet algorithms and the corresponding mechanisms and not the protocol
specifications.
The purpose of the algorithm is for two entities to be able to exchange messages that
would be encrypted with a shared key that they will both agree on, using a specific key
agreement protocol such as Diffie­Hellman. The difference here is that the two entities
will derive new keys after every message using a Key Derivation Function (KDF) and
the double ratchet. The message will be encrypted with the fresh keys ensuring forward
privacy, while the two entities will have synchronized ratchets, which key they have to use
for every message. The algorithm consists of KDF chains and two ratchets, the Diffie­

==PAGE:26==
Hellman and the Symmetric which both create the Double Ratchet.
2.3.1
Basic Mechanisms of the Algorithm
KDF Chains
A KDF is a function that gets as input a key material, a length, a salt, and a context variable
and gives as output an indistinguishable string of the given length. Even if the input is
not random, the output should still satisfy the properties of a secure hash, while HKDF
is based on the Hash­based Message Authentication Code (HMAC). There are various
formal definitions of a KDF in [31] which mathematically and briefly define KDFs while
showing their security properties. The KDF chain definition was introduced in Signals’
documentation for the Double Ratchet algorithm in [11]. This “chain” is created when
the output of a KDF goes to the input of another KDF, one could say similarly of how
block ciphers work. KDF chains have properties such as resilience, forward secrecy, and
break­in recovery.
While two entities are in a Double Ratchet session, each one of them has a KDF key for
three chains. These chains are the root chain, the sending chain, and the receiving chain.
Symmetric­Key Ratchet
The Symmetric­Key ratchet is the result when the two chains, sending and receiving, it­
erate after a message transmission. The output of the chains is used for encryption and
decryption. The output of the chains is also referred to as chain keys. The other KDF in­
puts are constants, ensuring that the receiving and sending chains encrypt the messages
with a unique key. The reason we call it a ratchet, is because the next calculation of the
key and message key is since it can be likened as a ratchet step which is a one­way step.

Diffie­Hellman Ratchet
The Diffie­Hellman ratchet gets created when two entities exchange also new Diffie­
Hellman public keys when they exchange messages. The DH outputs are the inputs
to the root chains, while the outputs of the root chains become the new KDF keys for the
sending and receiving chains. This ratchet protects the protocol from an attacker who can
obtain the chain tickets and use them to decrypt the messages. Each entity has a DH pair
which is the ratchet key pair, and in every message’s header is the sender’s public DH
key. When the receiver obtains the message, then he/she is able to create his private key
and create the DH output, and advertise his public DH key as depicted in 2.8. Figure 2.8

2.4
rTLS Protocol
Taking into account the properties that the double ratchet algorithm has, along with the
vulnerabilities and specifically the replay attack vulnerability of TLS 1.3, the authors of [10]
designed a new protocol named as rTLS. rTLS is a TLS based protocol aiming to harden
the security of TLS 1.3 while at the same time decrease the bandwidth overhead, making
the protocol a suitable solution for low constrained IoT devices. The motivation of that
comes after suggestions from various communities in IoT where in order to increase the
security, developers should use TLS/SSL mechanisms and certificates [33]. Therefore
TLS/SSL and DTLS are being studied regarding their application due to their well stan­
dardized security [34]. While on the other hand we have issues with cost over bandwidth,
power consumption due to encryption computations or storage and therefore many rest
that TLS over IoT is difficult [35]. Making engineers look over other solutions that could
avoid TLS due to its cost and use alternative solutions (i.e., SSH[36]).
rTLS is designed over TLS and therefore minimal changes should be done to the original
TLS 1.3 protocol. We use the benefits of TLS 1.3 as described in 2.2.1 while also the
security properties described in 2.3. The number of handshake messages of TLS 1.3
had been decreased significantly from its previous version, while has also the 0­RTT
resumption, a great improvement in terms of latency. With rTLS, the 0­RTT resumption
hardens its security from any potential replay attack, while at the same time achieves a
lower bandwidth overhead due to the double ratchet application.

2.4.3
Security of rTLS
The NewSessionTicket is transmitted by the server and a potential eavesdropper can
only see it as encrypted application data, as it is authenticated, verified and confiden­
tial. Moreover, an assumption is made that both entities store their states securely and
by extension the resumption_master_secret from the standard TLS, so that none can
derive the decryption keys. Regarding a potential replay attack now, when an adversary
replays a session resumption handshake with no changes, the server is able to check
the ratchet indices and here takes place the replay condition is < ic. Where is and ic
are the server respectively received client ratchet indices. Moreover the session ticket
(the 4­bytes Session ID) is protected with an HMAC from the TLS standard. Even when
an adversary records different resumption messages and is certain that at least one DH
handshake has performed and tries to exploit the replay condition, he will find another
problem. Since the resumption master keys would be different due to the ratcheting, and
the HMAC protects the identity binders of the session ticket, the exploit will fail. Similarly
to the double ratchet algorithm, the 0­RTT resumption has also the property of forward
secrecy as the entities store only the last resumption key. An attacker can’t compute past
keys based any compromised resumption key, since after each attempt a key is derived
with a KDF. Brek­in protection is also a property rTLS inherits from the double ratchet al­
gorithm, as the shared secret from a DH exchanged is fed to the KDF adding fresh entropy
to it. In a scenario when an adversary compromises one of the two entities and obtains
resumption keys, he would not be able to decrypt messages after the next DH exchange.

Related Work
As we have already mentioned, SSL/TLS is a standard that is used widely in the industry
due to its trust from the communities, its security, and the multiple suites it comes with.
There have been created various alternatives based on SSL/TLS so that one could benefit
from its advantages and discard any other unnecessary extensions while also add some
specific modifications for the specific application. Similarly to rTLS that emphasizes in
constrained IoT devices, multiple IoT protocols use for their security TLS/SSL [39]. Such
protocols are explained below, chosen from the literature as most related ones when
compared with rTLS’s security, uses and functionality.
3.1
DTLS and DTLS 1.3
Datagram Transport Layer Security (DTLS), is the corresponding of TLS but this time not
in connection­oriented protocols but connectionless protocols such as User Datagram
Protocol (UDP). It provides security for various applications based on datagram commu­
nication and stream­oriented TLS for the prevention of any malicious activity [40]. Like
in TLS there have been multiple versions such as DTLS 1.0, DTLS 1.2, and the latest
published version on April 2021 DTLS 1.3 [41]. It is designed by the same team, and
it provides security in protocols like Constrained Application Protocol (CoAP) and Data
Distribution Service (DDS) [39] while it can also be backwards compatible with DTLS 1.2.
For datagram protocols, TLS can’t be applied for multiple reasons. That includes that
TLS doesn’t allow independent decryption of individual records, while DTLS adds explicit
sequence numbers to solve that. The order of messages in TLS matters in contrast with
the DTLS, and reordering needs to happen. There are not acknowledgment messages in
datagrams wherein cases such as the NewSessionTicket of TLS 1.3 are necessary, and
last but not least UDP needs a return­routability check with the assistance of cookies to
encounter abusive behavior of packets and Denial­of­Service (DoS) attacks.
DTLS consists of two layers the DTLS Record Protocol and the DTLS Handshake Proto­
col. The Record Protocol diverges from TLS 1.3 and DTLS 1.2 with differences such as the
exclusion of version number and type fields, the addition of epoch and sequence number
to the TLS record layer, and an additional variable length header in the DTLSCiphertext
structure. In the Handshake Protocol now, the TLS 1.3 message flow is used with dif­
ferences such as reordering and fragmentation modifications on the header, addition of
re­transmission timers which are for the purpose of handling message loss. Moreover, a
new acknowledgment type is introduced for reliable message delivery. The handshake
in Message Sequence Chart 3.1, illustrates the handshake of the DTLS 1.3 along with its
0­RTT resumption procedure. We can observe that the initial 0­RTT is almost the same as
the 0­RTT of TLS 1.3 with differences a reply of early_data in the TLS 1.3 case from the
server indicating the receival and the ACK message after the resumption in the DTLS 1.3
case. The release of DTLS 1.3 was promising while various suggestions were done (i.e.
[42]) for improvement for IoT applications, and therefore IETF published profiles for both
TLS/DTLS 1.3 that aim in constrained IoT device application [43]. At the time this thesis is
written, there is no library that supports DTLS 1.3 to the fullest so that we could compare
it with real measurements, although the latest release provides a GitHub repository with
the source code of the draft by IETF [44].

3.2
CoAP
Constrained Application Protocol (CoAP) is a specialized web transfer protocol, designed
for constrained resource devices and constrained networks1. Similar to TLS, it is designed
by the IETF and it is specified as Standards­Track RFC7252 as part of the IETF Con­
strained RESTful Environments (CoRE). It is mostly used in IoT and machine­to­machine
(M2M) communication applications due to its low power for operation, and simple com­
putation needs [46]. It is based on the REST model that involves resources from a server
where clients interact with them using methods similar to the HTTP such as GET, PUT,
POST and DELETE methods while it relies on UDP over IP and not on TCP. A CoAP packet
has a 4­byte header and a compact encoding of options which enables small messages
that result in little or no fragmentation on the link layer.
CoAP has four security modes to establish security, namely NoSec where DTLS is dis­
abled, PreSharedKey where it uses the preshared key mechanism of DTLS, RawPublicKey
with DTLS and asymmetric encryption and Certificate where the nodes use X.509 cer­
tificates for validation [47]. Multiple suggestions have been made for CoAPs security
where most of them consist either of DTLS or IPSec [48], where the three aforemen­
tioned modes (NoSec, PresharedKey and Certificate) are supported by DTLS and like
in HTTP­HTTPS, CoAP becomes CoAPs. Although DTLS is providing security to CoAP,
attacks that could be leverage in DTLS, could also threaten it too. Such attacks could
be parsing where nodes could get crashed with arbitrary code execution, caching from
a malicious proxy, amplification of packet attacks, spoofing and Cross­Protocol due to
TCP­to­UDP translation [49]. An example of CoAP over DTLS is shown in Figure 3.1 with
4 round trips.

3.3
OSCORE vs TinyDTLS
As proposed in the previous Section, multiple issues could be addressed in CoAP espe­
cially when it comes to the proxy and its responsibilities. Motivated to overcome these
concerns in a methodical way, OSCORE has been standardized by the IETF [52] in 2019.
OSCORE protocol contributes to end­to­end security for the communication of two CoAP
nodes. It uses object security which is a mechanism that protects data objects and by
extension does not rely on layer 3 security while provides message protection and verifi­
cation from the application layer. Security properties such as integrity, confidentiality, and
reordering come along the protocol, while it also protects the weak freshness weakness
of CoAP messages and commits firmly in their security. It uses security mechanisms
such as HKDF, Authenticated Encryption with Associated Data (AEAD) and has its own
security context [53].
The authors of [53], implemented OSCORE to evaluate its efficiency in regards to the
resource consumption on a constrained device in regards to the security features that
the protocol needs to function securely. Moreover, they experimentally evaluated the ef­
ficiency of their implementation on resource­constrained devices that run IoT specific OS
while also compared them with the TinyDTLS3 implementation. They have created three
test cases an insecure plain CoAP scenario, an alternative secure scenario using CoAP
over DTLS, and a third with CoAP with OSCORE. In these test cases, they performed
measurements such as responsiveness between client and server, and for the server’s
side CPU, memory and radio usages. At the responsiveness measure, the authors evalu­
ated the RTT when performing a full request­response exchange and the measurements
wherein time ms metrics similar to the time difference we present in 2.3. For the radio
usage a Contiki NG utility was used that interacts directly with the radio interface and pro­
vides the energy consumption. We refer to these metrics as they are the most related to
what we expected to accomplish in this project. With respect to having a fair overhead
comparison, the authors ensured of using the same AEAD cipher for both DTLS and OB­
SCORE, since DTLS has always a fixed payload overhead, in contrast with OBSCORE’s
payload overhead which varies due to two fields that change size.
In regards to responsiveness the authors illustrated the three different tests in payload size
against RTT, along with their standard deviations, and for radio usage the radio occupancy
(receiving and transmitting ) vs the % CPU utilization. After the whole experimental eval­
uation, the results showed that OSCORE had lower radio transmission overhead, RTT,
memory usage, and energy efficiency for the server.

3.4
QUIC ­ MQTT
The Quick UDP Internet Connections (QUIC) protocol, is a general­purpose protocol de­
signed by Google and IETF. It runs over UDP and it is supported by most internet browsers
while its goal is to improve the performance of TLS­oriented web applications that are cur­
rently using TCP [55]. QUIC uses its own Record Layer, instead of TLS’s, with its own
framing format and at the same time keeps the same TLS handshake carrying handshake
data in CRYPTO frames [56]. Additionally, it supports 0­RTT resumption as TLS 1.3 which
is depicted in Figure 3.2. As it is a protocol designed for web applications, its application
to IoT is still under consideration; therefore multiple research is conducted on the specific
protocol for application in the IoT field.
In [57] the authors integrated QUIC with the Message Queuing Telemetry Trans­
port (MQTT) protocol and ran various tests over IoT scenarios. MQTT4 is suggested for
IoT applications due to its lightness in terms of bandwidth and power consumption. The
authors ran two scenarios, where they benchmarked the combination of MQTT/QUIC and
also MQTT/TLS/TCP. In the first scenario, the authors concluded that QUIC outperforms
TCP notably when in small­numbered RTT cases, while in the second scenario although
the results were not clear enough they were promising indeed. Since the benefits from
the 0­RTT in QUIC showed that it is more stable than TCP, the authors suggested that
more research should be conducted on it as it is a promising protocol for IoT applications.

3.5
CoAP, MQTT, Websocket overhead comparison
In [60] the author studied and compared the data protocol overhead of MQTT, CoAP,
and WebSocket5 protocol. Initially, a mathematical model was developed which could be
used to estimate the overhead of the protocols in comparison. For simplification, multi­
ple assumptions and observations took place like paying attention only to the overhead,
disregard the latency due to the physical channel, compare overhead differences starting
from the transport layer. After the calculated overhead estimations, an experimental test­
bed was created where experiments were conducted with the aforementioned protocols.
The results proved that the estimations were precise and therefore CoAP when function­
ing with non­confirmable messages had the least overhead, wherein second place comes
MQTT when in QoS 0, and last comes WebSocket with a very small difference. Although
the estimations met the experimental results, the author highlights that not considering
TLS, or DTLS security at the tested protocols is a huge drawback as they could com­
pletely change the results.

Surveys on IoT protocol comparison
The authors of [62], evaluate the performance and compare protocols that aim in IoT con­

They compare MQTT and CoAP, while concluding that both have similar characteristics
and further work should be done in a lab environment to achieve real­world and a closer
approach to the implemented protocols under multiple scenarios.
In [63] we have a more statistical performance comparison of the IoT protocols Advanced
Message Queuing Protocol (AMQP) [64], MQTT and CoAP. Two scenarios were con­

the second one; fault injections into the network routes took place. Some sensors and a
Raspberry Pi were deployed for sampling and experiment data collection. The authors
presented their results with ANOVA6 analysis for throughput, message size, and packet
loss while to explain them they used the outcome that they obtained from Turkey’s range
procedure7. The results presented that CoAP was able to outperform the other protocols
in the three chosen metrics, while it is also referred from the paper that in the future more
tests with more metrics should be performed.
[39] presents a brief analysis and evaluation of protocols used in IoT applications. Sim­

consumption etc. We could observe that all mentioned protocols are used for their secu­
rity rather TLS or DTLS, proof of TLSs and DTLSs frequent use. An analytical hierarchy
process was conducted to find the greatest weight between Security, Energy, Bandwidth,
and Latency from which security was highlighted as the most important. The paper con­

applications in reference to needs, efficiency, and suitability.

rTLS Implementation
For the first Proof­of­Concept (PoC) implementation of the rTLS protocol, besides the
literature review of various implementations, we had to look into various TLS libraries and
take advantage of their TLS 1.3 implementations. Based on multiple criteria we chose the
most suitable and therefore proceeded in developing rTLS over the source code of the
library.
4.1
Library Research
For searching the most suitable TLS library, we looked for the most known and used ones
in the industry, so that we could find one with well documentation and possible support.
We also looked into various scientific resources and journals such as IEEEXplore, ACM,
and Google Scholar to observe also scientific approach on them. The first libraries we
decided to look into were OpenSSL, GnuTLS, wolfSSL, mbedTLS, and matrixssl. Then
we realized that mbedTLS at that time had not TLS 1.3 implementation and thus led us
to discard it. Next, we have seen that there was no activity in the matrixssl repository [66]
since last year, while the official domains and other resources from the library were not
available. In [67] a comparison has been done between various TLS libraries, where the
author ended in the conclusion that the “best” library depends on the criteria of the de­
veloper’s needs, while OpenSSL was characterized with “Higher Throughput and simple”
and GnuTLS “Portable and lightweight”. Although portability and lightness would be an
advantage for rTLS as it is designed for low­constrained IoT devices, we wanted to choose
a well­documented library with ease of implementation and readable code. Therefore we
ended up with OpenSSL and wolfSSL, where we had to choose between the two. We
could observe that both libraries had support, active communities, and well­written and
updated documentation, while they had both TLS 1.3 implementations since 2018 from
the official standard release. Both libraries had their own crypto package for multiple ci­
pher suites, while wolfSSL has also an OpenSSL Compatibility Layer. Choosing between
the two was difficult since we had to look into reviews, comparisons, and as libraries
created by industrial firms, we had to be aware also of non­biased reviews and ratings.
Some points we often saw were that wolfSSL had smaller build size, hardware acceler­
ation, portability, and ease of use [68], while on the other hand due to the legacy code
OpenSSL had sometimes, it was characterized as “spaghetti code” [69]. WolfSSL indeed
had some portability as we were able to see that it can support multiple devices and OSs
in contrast with OpenSSL while it also advertises itself as “targetted at IoT, embedded,
and RTOS environments primarily because of its size, speed, and feature set”. The next
thing we had to do was to look into the code directly and decide ourselves. By looking
into both libraries’ code and after discussing with the other team that was working with a
TLS 1.3 thesis project, we ended up that the best choice would be to use wolfSSL as it
looked more straightforward and had well­documented code.

4.2
WolfSSL and WolfSSL library
WolfSSL library, previously named CyaSSL, is an open­source SSL/TLS library that ad­
dresses the needs of embedded systems developers by being a small and portable library.
It consists of the implementation of all TLS versions with the only exception of DTLS 1.3
which is not fully supported, but up to a specific draft release. It is written in C pro­
gramming language while it also supports various APIs. Additionally, wolfSSL supports
other multiple­purpose libraries such as crypto engines, Trusted Platform Module (TPM)
libraries, secure bootloaders, and others. Some examples are wolfTPM, wolfMQTT, wolf­
SSH, wolfCrypt FIPS etc. [70].
Therefore the ones that are important to us for this project are:
• wolfSSL, the standar SSL/TLS library
• wolfCrypt, a lightweight cryptography library for resource constrained devices that
supports multiple popular algorithms and ciphers (i.e., Curve25519, ChaCha20,
Blake2b, NTRU etc.)
The official source code of the wolfSSL library can be found in the repository [71]. This
repository includes also some of the additional libraries and APIs, like the wolfCrypt for
cryptographic operations, sslSniffer that provides sniffing support from wolfSSL for traffic
analysis, a C# wrapper, a test suite for testing the wolfCrypt library, and multiple examples.
Some specific APIs have to be defined when building the library so that the built project
files use them (i.e. /usr/bin/cmake -DWOLFSSL_SNIFFER=yes, which enables the sniffer).
The given examples can be run with arguments described in the official documentation
(i.e., .\client -v 4 -r -s -0) meaning that the client executable will run with version 4
(which maps to TLS 1.3), with a resumption session using PSK and it will send early data
in the 0­RTT resumption. The core functionality of the library is developed into the C files
in the src directory. The library can be built with both make and CMake automation tools,
and it can run on Windows using Visual Studio and therefore has the analogous .vcproj
supported files. It can also function in MAC and Linux environments with any appropriate
compiler i.e. the GNU Compiler Collections (GCC).
The main files that we used for the rTLS development are:
• examples\client.c, examples\server.c: Which are the source codes that imple­
ment a client and a server example that can be run using arguments to use specific
sessions, encryption algorithms, TLS versions, include certificate or key files etc.
• src\ssl.c : Includes the basic functions for the extensions of the SSL versions
while also creates the main ssl struct that the SSL/TSL library uses. This C struct is
defined in the ssl.h file and contains all the extensions and flags which map to the
arguments that the program is build and executed with.

In this first rTLS implementation, we did not emphasize in memory consumption and com­

so that we could decrypt the protocol to analyze it, or just debug the protocol to verify its

For the resumption session, we had to modify the Client Hello message.
The
psk_key_exchange_modes extension was already modified from the initialization hand­
shake so no further work had to be done. For the other two extensions key_share and
pre_shared_key a slightly different modification was done from the proposed one of rTLS.
Instead of using the key_share extension to advertise Clients public DH key, we used
the pre_shared_key. That would be an easier modification and therefore would use less
bandwidth. In the case of sending the DH key through the key_share extension, we would
use 32­bytes for the X25519 key, and 10­bytes for additional extension data such as size,
type etc. which would use 42­bytes in total. On the other hand, by sending the X25519
concatenated in the pre_shared_key extension with a structure as the one described in
Table 4.2 that would use just 32­bytes. By default the pre_shared_key extension, when in
a resumption session (executed with the -s -r arguments), will send a second pre­shared
key and therefore we have Client_Identity and the NewSessionTicket ticket, which the
server receives and using the hashed binders authenticates the client. The modification
we had to do here, was to replace the DHsPublic from the ticket with the clients’ DHsPublic

Benchmarking rTLS
In order to benchmark the developed rTLS code, we initially looked into wolfSSL’s bench­
marking feature. We observed that the benchmarks were focusing mostly on wolfCrypt’s
performance on encryption algorithms and were giving information regarding the size and
time of encryption. Therefore we contacted two other students from our TLS team that
in their thesis they added some checkpoints in wolfSSL and could get data from TLS 1.3
such as overhead and maximum memory used. Although that could be used, we decided
not to proceed with this solution due to the limited amount of time we had to embed that
in the rTLS code but also because that was still in development at some points. At last,
we concluded to use the Scapy packet manipulator in a Python script to capture, filter
the packets from the network and measure the bandwidth. Moreover, we used Wireshark
to verify the number of packets captured and make sure that there wasn’t any packet
loss. We decided that using well­known and commonly used tools by network engineers
we could ensure better reliability on our captures. The packet captures are attached in
Appendix A.
We have created two scripts, one that gives the average bandwidth of total resumptions
and one for average bandwidth per packet. The first one, filters the packets and keeps the
Client Hello, Server Hello, Encrypted Extensions, End of Early Data and the two Finished

they could change at the Client’s convenience regarding the message he wants to send.

the early data twice. The answer we got from the wolfSSL team was “sending two Early­

those, and we did not proceed with modifying that. Moreover, the selected packets were

7
Discussion
In this thesis project, we have developed the first rTLS draft which functions over a com­
monly used TLS library, wolfSSL. We were able to run it in multiple sessions so that we
could get measurements regarding the average bandwidth overhead and compare it with
both the estimated, and TLS 1.3. Despite that we had some deviation from the estimated
overhead, this does not make rTLS “expensive” as it is still cheaper from TLS 1.3. The
results we got prove that, and therefore we could confidently say that rTLS could be a
promising and cheaper. The main differences that made rTLS “cheaper” are the use of a
4-byte Connection ID + 1­byte Ns which took the role of the session ticket that TLS 1.3
normally uses that has a size of 146 bytes. Moreover, a modification that was done in this
thesis which was not proposed in the theoretical description, was to use the Identity field
to send the X25519 32­byte DH key, instead of the key_share extension which saved us
10 more bytes as the whole extension with its properties (i.e., Type, Length, etc.) has a
size of 42­bytes.
One could say that could be “dangerous” since the DH key is sent plainly and there
is not any response from the server as in the key_share case.
We argue here that
also key_share is sent plainly in the same message, while due to ratcheting, in a fur­
ther implementation where messages get also exchanged and not just handshakes and
resumptions, there would be a new key to feed the HKDF and that would still pro­
tect the protocol. The only additional extension that adds extra bytes into rTLS is the
psk_key_exchange_modes which in the full rTLS implementation would trigger and sig­
nal the communicating entities that rTLS is used and therefore respond correspondingly.
Currently, a formal verification of rTLS is being under research where it would be good to
consider also this change and ensure its security. Regarding the security of the protocol,
we were not able to test it and perform a Replay Attack so that we could ensure that it is
indeed safe. While one could say that since the protocol is formally verified with regard to
security there is no need to proceed with practical testing, many times implementations
and especially in “low level” languages such as C, present vulnerabilities that wouldn’t be
even thought of, in the theoretical approach. In terms of security though, rTLS could be
considered also as a solution to protect the aforementioned study of Chapter 3 QUIC ­
MQTT [57].
For future work on this subject, we would suggest an rTLS implementation that would
proceed in exchanging Application Data messages so that its full TLS­based implemen­
tation could be security tested. With the current status of implementation, rTLS could
be continued over wolfSSL as it is a well­structured library with support, documentation
and we made sure to develop it with provisions for further development. Moreover, in
this implementation, we were not aiming for in­memory cost, which is also a constrain in
IoT devices that rTLS targets. Meaning that, multiple things in the source code could be
changed and use dynamic memory allocation for most data structures rather than just a
few, to make it also memory efficient. Additionally, multiple changes and polishing can be
done to the current rTLS draft so that the code complexity becomes more efficient. It is
important to mention also here that besides that it was a PoC implementation, we had in
mind best practises in secure programming (i.e. not using strcpy but strncpy or memcpy
instead etc.)

Conclusion
rTLS was proven and evaluated as a promising protocol to replace TLS 1.3 in low­
constrained applications. The results we got are indeed promising, and we could use
them to strongly argue about its low bandwidth overhead especially when compared with
TLS 1.3 and moreover comparable with the aforementioned alternatives mentioned in the
related work of Chapter 3 in a practical evaluation. We studied from various specifications
such as TLS 1.3 standard [13], wolfSSL’s documentation [70], double ratchet documen­
tation [11] and we were able to construct the first PoC implementation of rTLS. Many
difficulties appeared such as choosing the environment, the library but the most challeng­
ing part of this thesis, was to be able to get into the wolfSSL’s huge source code and find
the specific spots where each modification/code insertion had to be done. That took a
lot of time rather than the modifications themselves and the benchmarking. Therefore,
the development code can be the basis to build on it and perfect the whole protocol until
an official release. Through that work, we were able to gain valuable knowledge regard­
ing navigating through huge codebases, learn to use debugging and network tools such
as gdb, Wireshark, Scapy to inspect code from the memory and analyze network traf­
fic. Moreover, we gained valuable knowledge regarding TLS­based protocols and other
IoT protocols that are currently used in the industry but also under research when we
performed our literature research.
