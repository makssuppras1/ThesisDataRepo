Abstract
The goal of the thesis is to attempt to solve the speciﬁc Semantic Parsing prob-
lem of going from natural language to Structured Query Language (SQL) queries
- known as the text-to-SQL problem. We base most work on the paper ’Towards
Complex Text-to-SQL in Cross-Domain Database with Intermediate Represen-
tation’ (Guo et al., 2019), which introduces a state of the art model for the
text-to-SQL problem (as deﬁned by the Spider dataset (Yu et al., 2019)) called
Intermediate Representation Network (IRNet). The contributions made in this
thesis are listed below,
• We present a detailed overview of the IRNet introduced in the original
paper (Guo et al., 2019).
• We successfully extend the introduced SemQL grammar to account for
values as well as DISTINCT tokens.
• We construct a model that obtains an exact matching accuracy of 33.0%
on the development set of the Spider Dataset.
• We manage to successfully augment the original IRNet model with an
adaption of a pointer network to allow for the prediction of values in SQL
queries. Exact match accuracy only falls by 5% in this setting.
• We manage to successfully develop a multilingual model based on IRNet,
capable of taking questions formulated in 14 diﬀerent languages as input.
The performance is equivalent/slightly higher than our implementation of
IRNet.
• We experiment with the use of transformer architectures in IRNet.

Chapter 1
Introduction
This thesis deals with a speciﬁc problem within the ﬁeld of Semantic Parsing,
namely that of converting natural language questions into Structured Query
Language (SQL) queries - more commonly referred to as the text-to-SQL prob-
lem. A speciﬁc manifestation of this problem in a real world application setting
could be to create better search functionality within the domain of emails. Con-
sider the following request,
”I remember getting a mail last week from Nordea, show me the ones with
attachments.”
Without a text-to-SQL model, the user would have to specify the ﬁlters in a very
precise manner to obtain the mails, however by using a model the statement
can be converted to the following SQL query,
SELECT * FROM mails WHERE start_date = (NOW-1w) and end_date = NOW
and domain LIKE ’%Nordea%’ and has_attachment = 1.
This query can then deterministically be transformed to an Outlook API call,
retrieving all the desired mails that fulﬁll the requirements of the statement
without the user needing to specify anything. Hence the problem is not only
of academic interest, but also has the potential to become very inﬂuential in a
commercial setting.

Introduction
The scope of this thesis is two part and has changed signiﬁcantly throughout
the 5 months spent. Initially, the scope of the thesis was to build upon the work
done in the paper ’Towards Complex Text-to-SQL in Cross-Domain Database
with Intermediate Representation’ (Guo et al., 2019), under the assumption that
a respective code base would be shared by the authors (as was promised in the
paper). The paper managed to achieve State of the Art (SOTA) results on the
text-to-SQL problem. However, when this never happened, the scope increased
signiﬁcantly as the model would have to be built from scratch. Naturally this
took time away from the work related to extending upon the proposed frame-
work. That said, we still manage to extend the model to deal with values in
the prediction of database queries - bridging the gap between pure academic
performance and real world application - as well as manage to make the model
language agnostic (taking questions written in 14 languages as input).

Background
2.1
Semantic Parsing
Important for the research area of Natural Language Processing (NLP), as well
as Artiﬁcial Intelligence (AI) in general, is the development of a true understand-
ing of human language, known as Natural Language Understanding (NLU). This
is important since for a machine to be truly intelligent, it must in some shape or
form posess the knowledge of human beings, and the way in which knowledge
is passed around in our human world is through the means of human language
(Chollet, 2019).
A central part of NLU is that of Semantic Parsing. Semantic Parsing refers to
the task of mapping Natural Language (NL) utterances into Meaning Represen-
tations (MR). These MR’s, often interchanged with semantic representations,
logical forms and programs, are then typically executed against an environment
or context to yield a desired output. The ﬁeld is not only important for the rea-
sons stressed above, but in a world where a large proportion of information is
stored in structured Knowledge Bases (KB) (the largest of which is the internet
as we know it), it is becoming more and more important to have eﬃcient and
simple methods to query them.
Before delving deeper into such methods, it is important to note that there
==PAGE:20==
8
Background
is a clear distinction between Semantic Parsing and other sequential learning
tasks such as Machine Translation and NL generation.
The reason for this
lies in the fact that a model for Semantic Parsing has to predict structured
objects which are inherently tree-like. This is very diﬀerent from the freedom
generated by the inherent ambiguity within languages, thus the ﬁeld is subject
to strong constraints that introduce unique challenges.
Semantic parsing is
also inherently more complicated in comparison, as it requires understanding
concepts from diﬀerent word phrases.
This means that semantic parsing is
more about capturing the meaning of the sentence rather than plain rule-based
pattern matching.
The main component of a semantic parsing framework is the language of the
meaning representation. Meaning representations is often interchanged with log-
ical forms or programs in general (such as a python snippet). There are three
overall types - logic based formalisms, graph based formalisms and programming
languages (Kamath & Das, 2018). The most straight forward language repre-
sentation is one that uses First Order Logic (FOL) - consisting of quantiﬁed
variables and functions that can only take objects as arguments. An example
of such a representation could be taking the utterance ’All gardeners who like
the sun’ and converting it to the FOL (∀x)gardener(x) →likes(x, sun). This
was explored extensively in the paper ’Learning Executable Semantic Parsers
for Natural Language Understanding’ (Liang, 2016).
However, FOL quickly
falls short when it comes to the manipulation of sets - which a major issue
regarding more general queries such as ’Count all gardeners who like the sun’.
This is solved through the introduction of lambda calculus (Carpenter, 1998),
which allows for the a set, x, that satisﬁes a given condition to be represented
as λx. The statement ’Count all gardeners who like the sun’ can, using lambda
calculus, be represented as count((λx)gardener(x) →likes(x, sun)).
One problem with using logic meaning representations directly is the lack of easy
comprehension by humans. One way of alleviating this is through the means of
graph based representations. In this setting the semantic form of an utterance
is represented by a labeled graph where the nodes denote entities and the edges
represent semantic relations between the nodes. The reasons why graph-based
formalisms have become increasingly popular within newer literature on the
subject of semantic parsing lies in the fact that there exists a rich literature on
graph algorithms that can be utilized when learning a parser to produce these
structures. Another pro with graph-based formalisms lies in the fact that the
MR is broken down into abstract high level segments (parent nodes in the tree)
allowing for a potential model to get some context dependent on the current
location in the tree.
The ﬁnal type of language representation is that of programming languages
(PL). By converting utterances directly into PL’s such as Python, Java, SQL,

Bash etc. one avoids the inevitable domain speciﬁcity of standard logical forms.
This problem arises due to the relatively simple schemas and syntaxes available
in logical as well as tree based formalisms. Another pro with converting NL
directly to PL lies in the fact that PL’s are widely used within the developer
community and act as the primary access point to KB’s - avoiding the extra
step of converting logic based formalisms into executable code againist the KB
in question. In this thesis, as shall become apparent, we make use of a clever
integration of all three methods of representing MR’s.
Another essential component within the realm of semantic parsing is that of the
grammar upon which the MR’s are created. Within the grammar there exists
a trade-oﬀbetween the expressivity of the grammar and it’s computationial
complexity. On one hand, given that the language utterances are structured and
manage to encapsulate all necessary information, having lots of rules provide a
computational advantage in the sense that the possible derivations are greatly
limited. However in a setting where the NL can be ill posed, such as data from
crowd sourced utterances, it may help to generate higher level abstract rules
that can capture most ill formed sentences at the cost of generating a lot of
possible derivations.
2.1.1
A brief history
In the following section, a brief history of the developments within semantic
parsing from the 70’s to present day is given. The reason for such an overview
lies in the fact that it helps shed light on some of the main challenges of the
overall task as well as reasons for currently employed methodologies. Some of
the earliest attempts at creating semantic parsers relied solely on rule based
systems. Examples include pattern matching such as SAVVY (Johnson, 1984),
which is simple in nature, however extremely brittle in practise due to the shal-
lowness inherent in pattern matching (no true context is learned).
Another
example is LUNAR (Woods, 1973), in which a syntactic parser is developed
with the intentions of creating a parse tree, which then is mapped to an un-
derlying database query language through the use of rules. In an attempt to
generate stricter grammars and thereby less ambiguous potential results, parse
trees were generated taking not only the syntactic information into account, but
also some semantic. This was speciﬁcally done by having non-terminal nodes
in the parse tree represent semantic categories, making it easier for the tree to
be transformed to the query language. However, doing so implies that a lot of
semantic information be hard-coded in the parser - thereby creating a very do-
main speciﬁc solution. Using syntax alone, gives the ability to transfer learning
across domains and is thereby extremely versatile.
==PAGE:22==

Background
In the mid 90’s the ﬁeld saw a rise of statistical learning techniques, a path
paved by the paper ’Learning to Parse Database Queries using Inductive Logic
Programming’ (Zelle & Mooney, 1996). In this paper, a model capable of train-
ing on a corpus of sentences paired with database queries was introduced, by
means of inductive logic programming. However, the main contribution lies in
the fact that by producing database queries directly it was easy to evaluate
models (a problem for prior rule-based methods) - given that one could execute
the query and determine if the returned answer was satisfactory - but it also
paved the way for developing models that could learn given examples of desired
input output pairs.
One of the problems with the model developed was the fact that it assumed the
existence of a lexicon mapping phrases to meaning representations. These lexica
were generated by hand and thus very time-consuming and domain speciﬁc. In
the paper ’Acquiring Word-Meaning Mappings for Natural Language Interfaces’
(Thompson & Mooney, 2003), the idea of generating a such lexicon from the data
directly was explored. Experiments showed that a learned lexicon performed
better than any man-made version and thus paved the way for data-driven
learning.
In the past couple of years, there has been a rise in end to end approaches
using encoder-decoder frameworks based on recurrent neural networks. There
are pros and cons with this method as compared to earlier attempts. Learning
end to end models alleviates the need to use lexicons and hand-engineered fea-
tures making it an extremely versatile method to be used across domains and
speciﬁc MR languages. The drawback, however, lies in the fact that traditional
approaches are able to better model and leverage the in-built knowledge of logic
compositionality - as they are directly embedded into the model by use of lexica.
In order to alleviate some of the issues regarding the true hierarical structure
of logical forms, the paper ’Language to Logical Form with Neural Attention’
(Dong & Lapata, 2016) force the model to take this structure into account by
recursively calling their model on sub-trees of the MR. Speciﬁcally this is done
by keeping track of parentheses in the MR - denoting sub-trees. The paper also
introduces the notion of parent feeding, creating extra connections to the start
of the sub-tree when synthesising as well as uses soft attention models of the
form introduced in the paper ’Neural Machine Translation by Jointly Learning
to Align and Translate’ (Bahdanau, Cho, & Bengio, 2014).
There still exists a major setback in these kinds of encoder-decoder architectures
in the sense that they are incapable of learning decent parameters for words
that appear very rarely in the dataset. One way of dealing with this issue is
by anonymizing the words and later ﬁlling them in during post-processing. An
example being to take the name "Bruce Dickinson" and replacing it with the
==PAGE:23==

token person. Another approach is to use Pointer Networks (Vinyals, Fortunato,
& Jaitly, 2015) which directly copies given words from the input. A combination
of the two can be achieved using attention-based copying (Jia & Liang, 2016) in
which the decoder can choose to either copy or select from a certain vocabulary
at any point in time.
The current frontier of the ﬁeld explores the use of intermediate representations
as a means of disentangling higher level syntactic information from lower level
syntactic information. The reason behind which lies in the fact that the higher
level representation will be more abstract in nature allowing it to better general-
ize across training examples (speciﬁcally useful if the dataset at hand is limited).
This sharing between data points also allows for faster training of models. An
example of this was introduced in the paper ’Coarse-To-Fine Decoding for Neu-
ral Semantic Parsing’ (Dong & Lapata, 2018) in which a generated sketch (the
intermediate representation) of the ﬁnal output is used for easier decoding.
A ﬁnal interesting point, of which is used extensively throughout the thesis,
is the idea of constraining the output decoding through use of deterministic
grammars.
Speciﬁcally, instead of generating the ﬁnal query to be executed
against the knowledge base in question, the NL utterances are transformed into
Abstract Syntax Trees (AST) for the PL in question and then deterministically
transformed into the PL itself (Yin & Neubig, 2017). The model is then tasked
with determining the ’action’ sequence necessary to generate the AST implying
that each step can be signiﬁcantly constrained making it simpler for the decoder.
2.2
Structured Query Language
In the following section we give a formal introduction to Structured Query Lan-
guage (SQL), a standard Database language which is used extensively to create
and maintain relational databases as well as retrieve records from them. The re-
lational model, upon which SQL was born, was introduced in 1970 by Edgar F.
Codd who published the inﬂuential paper ’A Relational Model of Data for Large
Shared Data Banks’ (Baxendale & Codd, 1970). Prior to the paper, databases
were stored using either the hierarchical model or the network model both of
which were types of navigational models. They maintained large tree or graph
structures in disk using pointers to preserve the links between the data. Al-
though fast (due to the vast amount of well established graph algorithms), they
suﬀered from the fact that inorder to retrieve records stored at the bottom, one
would need to navigating through all of its ancestor records. The need for data
independence, i.e. being able to uniformly query data regardless of the method
by which the database was stored, is what motivated the relational idea.
==PAGE:24==
12
Background
The relational idea is as follows, and is greatly inspired by the mathematical
sense of the word. Given n sets, S1, S2, . . . , Sn, we can call R a relation on such
sets if it is a set of n-tuples with the ﬁrst element from S1, the second element
from S2 etc. In his work (Baxendale & Codd, 1970), Codd referred to Sj as the
jth domain of R. In modern lingo, we now refer to a relation as a table in a
relational database, and the respective domains as it’s columns. In the paper,
Codd also deﬁnes several more terms including primary and foreign keys as well
as the notion of normalization. These concepts were later reﬁned to form SQL
as we know it.
Having given a brief history of SQL, we now delve into the details needed for
the purpose of this thesis. As mentioned, a relational database is composed
of several tables, each of which are composed of columns. Each table, when
normalized, will have a primary key and possibly a number of foreign keys. A
primary key is a column composed of distinct atomic values which can uniquely
identify a record in the given table.
An example could be a column called
’Customer_ID’ in a customer table, since each customer (or record) in the table
will have it’s own unique id. A foreign key is a column which acts as primary
key in another table within the database - thereby enabling two tables to be
joined. We introduce the speciﬁc syntax of SQL through an example.
Assume we are in a database consisting of the two tables 2.1 and 2.2, contain-
ing demographic and ﬁnancial information of customers respectively (referred
to as table A and table B in the following). If we wish to retrieve or query
any information from this database through the use SQL, we make use of the
SELECT and FROM commands which form the foundation of any SQL query. For
instance if we wish to get all the countries from table A we execute the following
query: SELECT Country FROM A, returning the list DK, NO, DK, FI. If we want
the distinct countries, i.e. DK, NO, FI we simply add the DISTINCT command
before the column name in the query. In order to deal with ﬂoat records, SQL
provides a number of aggregators including sum, avg, max, etc. Hence if we
wish to ﬁnd the total income, we can write: SELECT sum(Income) FROM B.
Till now we have only dealt with queries on single tables, but what if we want
to know what the total income is grouped by countries, it is necessary to make
use of the JOIN command, which allows for multiple tables to be joined on
their primary keys.

key ID and thus we can answer the before-mentioned question with the fol-

as T2 ON T1.ID = T2.ID GROUP BY T1.Country. As evident, we make use of
aliases for the tables T1 and T2 respectively to ensure that the columns come
from the correct tables, it should also be noted that whenever an aggregation

GROUP BY command to ensure that the aggregation be grouped by the column

2.3
Text-to-SQL Problem
The scope of this thesis is to attempt to solve a speciﬁc subproblem within
the ﬁeld of semantic parsing, namely that of synthesising an SQL query from a
question written in natural language (NL). This problem is often referred to as
the text-to-SQL problem and has received renewed interest over recent years.
The reason for this renewed interest is in part rooted in the fact that most
companies have their internal data stored in SQL databases. Thus creating a
solution allowing everybody within the company to access to this information
without the need for expertise within formal languages is of great value.
There are many public datasets available for the task at hand including ATIS
==PAGE:26==
14
Background
(Dahl et al., 1994) , GeoQuery (Zelle & Mooney, 1996) and WikiSQL (Zhong,
Xiong, & Socher, 2017), for which recent state of the art (SOTA) aproaches
have managed to score impressively (within the range of 80% accuracy). One
may therefore ask the simple question whether or not the text-to-SQL prob-
lem has been fundamentally solved? The answer to which is a resounding no.
In February 2019 the Department of Computer Science at Yale University re-
leased a new large-scale, complex and most importantly cross-domain text-to-
SQL dataset called Spider (Yu et al., 2019). It happened that Spider brought a
lot of new challenges that proved to be hard for existing approaches. Not only
are the SQL queries more complex in nature, including various nested queries
as well complex uses of HAVING and GROUP BY commands, but the fact that it
was cross-domain, in the sense that the development set consisted of completely
new schemas not present in the training set, implied that the model in question
had to be database agnostic. In the paper introducing the Spider dataset, it
was shown that best SOTA model (SQLNet (Xu, Liu, & Song, 2017)) at time
of writing achieves a mere 12.4% exact matching accuracy in the database split
setting. More details regarding the Spider dataset can be seen in Chapter 4.
Thus, the months following the release of Spider saw a huge inﬂux of combining
various methodologies within the ﬁeld of Semantic Parsing, many of which have
been mentioned in section 2.1, to improve on the low benchmark. At time of
writing the ﬁrst position was held by IRNet (Guo et al., 2019), which managed
to outperform prior models with an incredible 19.5% absolute improvement.
This fact, together with the fact that IRNet makes use of a combination of
most new advances within the ﬁeld of Semantic Parsing - including Abstract
Syntax Trees, Parent Feeding Connections, Intermediate Representations etc. -
it seemed like a viable model to reproduce and build upon. This thus forms the
scope of the thesis and the remaining work.

3.3
GloVe Embeddings
In the ﬁeld of NLP, each token in a given input or target sequence is represented
by a vector of certain dimensionality d. A token can be a single word, subword
or a character. The most intuitive way of achieving such a vector representation
is through the means of one-hot encoding. Given a vocabulary consisting of V
tokens, a one-hot encoding would generate an orthogonal V dimensional vector
representation of each token in V . This leads to a lot of issues. Firstly, the
vectors become increasingly large and sparse with increasing vocabulary size -
making it diﬃcult to deal with in terms of further modelling. Secondly, due to
the orthogonality of the vectors, they don’t capture any semantic nor syntactic
similarity between tokens (the vectors are equidistant from one another). This
motivates the idea of generating distributed representations, of which GloVe
(Pennington, Socher, & Manning, 2014) poses a great example.
However, before formally introducing GloVe, we point towards a slightly less
naive method of generating word embeddings which captures dependence be-
tween words, namely the Word2Vec method (Mikolov, Chen, Corrado, & Dean,
2013). In general there are two methods that can generate Word2Vec embed-
dings, namely Skip Gram and Continuous Bag of Words (CBOW) both of which
rely on neural networks of the form detailed in the preceding sections. The basic
idea being to either predict surrounding words based on the current word (Skip
Gram) or predict the current word based solely on surrounding words (CBOW).
This is done by creating a neural network with a single hidden layer (no ac-
tivation function) that takes either the context or current word as input and
predicts the current word or context respectively (done via a softmax operation
(see section 3.7) across the entire vocabulary). The hidden state then serves as
word embeddings to be used as input to further modelling.
The main drawback with methods like Word2Vec is the fact that the model
only takes local context into account (typically within 3-5 neighbouring words).
A solution would be to somehow incorporate global statistics into the vector
representations, which is exactly what Global Vectors (GloVe) (Pennington et
al., 2014) do. The main idea upon which GloVe is built, is that semantic re-
lationships between words can be found from a co-occurance matrix. Assume
a corpus consisting of V distinct words, then the co-occurance matrix X is a
V × V matrix whose i, jth element is the denotes how many times word i has
co-occurred with word j. A word is said to co-occur with another word if it is

Sequence to Sequence Task
In the following section we formally introduce the sequence to sequence task. In

denoted by yi ∈NTy,i ∀i ∈{1, 2, . . . , n} where Ty,i denotes the length of the
output sequence and each element again corresponds to a one-hot encoding of
the vocabulary. An example of a sequence to sequence task could be that of
taking a question as input and producing an answer as output (a well known

3.5
Recurrent Neural Networks
One of the main drawbacks with standard feed forward neural networks lies
in the fact that they have a hard time dealing with temporal data. Temporal
data in this context refers to any input that can be written as a sequence (or a
series). Enter Recurrent Neural Networks (RNN), an architecture very similar
in nature to a standard FFNN with the exception that it possesses the capability
of remembering things learnt from prior inputs while generating outputs. There
are many diﬀerent types and adaptations of RNN’s, in the following we start by
introducing the standard architecture.
3.5.1
Vanilla RNN
A vanilla RNN works by feeding items from the input sequence through a single
set of model parameters, as well as maintaining a hidden state vector repre-
senting the context based on prior inputs and outputs. The output of an RNN
thus depends not only on the weights an biases applied on inputs like a nor-
mal FFNN, but also on the hidden state vector - implying that the same input
placed at diﬀerent positions in the input sequence could produce diﬀerent out-
puts. It is important to emphasize that the model parameters are shared across
diﬀerent items of the input sequence, which is the reason as to why RNN’s in
==PAGE:41==
3.5 Recurrent Neural Networks
29
theory can deal with input sequences without a predetermined size and thus why
RNN’s have been ﬁrmly established as state of the art approaches to sequence
modelling.
Formalising RNN’s in the sequence to sequence setting, we have that the tth
hidden state of the RNN (with hidden dimension h), ht ∈Rh, is given by,
ht = g1(wh · ht−1 + wx · xt−1 + bh),
(3.26)
where g1 is an activation function, wh ∈Rh×h is a set of weights for the hidden
states, wx ∈Rh×m is a set of weights for the inputs and bh ∈Rh is a bias term.

yt = g2(wy · ht + by),
(3.27)
where g2 is an activation function, wh ∈Ry×h is another set of weights for the
hidden states, and by ∈Ry is a bias term. Some of the pros and cons of a simple
RNN structure are summarised in the following.
The pros of a standard RNN include the fact that they can process input of
any length, the model size does not increase when increasing the size of the
inputs (this would be the case if one were to model sequences using a standard
FFNN), computations take historical information into account and model pa-
rameters (weights and biases) are shared across time. There are however also
cons associated with the simple RNN. These include computation time being
slow due to the very sequential nature of the architecture (the next output
depends on the previous output), it being diﬃcult to access information from
a long time ago making long-term dependencies near impossible to ﬁnd, and
ﬁnally that the model cannot consider any future input for it’s current state.
Before attempting to deal with the cons of the vanilla RNN, we start by consid-
ering how a vanilla RNN would be trained - a learning algorithm very similar
to that of standard back propagation known as back propagation through time.

As explained in section 3.5.2, the problem with exploding/vanishing gradients

3.6
Encoder-Decoder Architectures
Encoder-Decoder architectures are a type of neural networks that are used ex-
tensively within sequence to sequence modelling. The network is composed of
two modules, an encoder and a decoder that are coupled through a hidden state
representation of the encoder.
The encoders job is to encapsulate all of the
information important for the decoder to make accurate predictions given the
task at hand. This information, is typically held in the ﬁnal hidden state of the
encoder. It should however be noted that the exact connection between encoder
and decoder is an art-form and can be done in a variety of ways as shall be
evident in the models developed for the problem at hand.

3.8
LASER
It is becoming more and more important for models to be able to deal with the
problems at hand in a multilingual setting. In the paper ’Massively Multilin-
gual Sentence Embeddings for Zero-Shot Cross-Lingual Transfer and Beyond’
(Artetxe & Schwenk, 2018), an architecture to learn joint multilingual sentence
representations for 93 languages is introduced. The architecture is based on
a single bi-directional LSTM encoder coupled to a decoder LSTM in a stan-
dard encoder-decoder architecture. The coupling between encoder and decoder
is achieved by a max-pooling operation over the output of the bi-directional
LSTM encoder the result of which is denoted as the sentence embedding of the
input. This sentence embedding is then sent through a linear transformation
and used as the initial state of the decoder LSTM.
With respect to input, given that the model has to be language agnostic, a
joint byte-pair encoding (BPE) vocabulary with 50,000 operations was built by
concatenating all training corpera across languages. By doing this, the encoder
has no information about the language it is processing at any given time and
is thus forced to learn language independent representations of the input. The
task that is considered in order to learn high quality sentence embeddings is
that of neural machine translation. Thus, the decoder takes not only a BPE
of the previously predicted token as input but also a language id embedding as
well as the sentence embedding concatenated at every time step.
The ﬁnal pre-trained sentence embeddings were built using a 5 layered encoder
with hidden dimension 512 - implying that the ﬁnal sentence embeddings are of
dimension 1024 (concatenating the forward and backward hidden states) - and
a single layer decoder with hidden dimension 2048. The BPE were set to 320
dimensional and the learned language id embedding had 32 dimensions.

an encoder block and a decoder block and is thereby a encoder-decoder model,
see section 3.6. The encoder block is composed of N encoders stacked on top of
each other, and the decoder block is similarly composed of N stacked decoders.
In the following, we describe the encoders and decoders respectively in detail.
The complete architecture can be seen in Figure 3.2.

3.10.2
Decoder
A single decoder within the decoder block of a transformer is very similar in
nature to an encoder layer. It takes the input, x, and sends it through a self at-
tention layer followed by a encoder-decoder attention layer and ﬁnally through a
feed forward layer. The encoder-decoder attention layer utilizes the multiheaded
attention mechanism introduced in section 3.10.1, where the query is composed
of a linear transformation of the input x whereas the key and value vectors are
calculated by taking a linear transformation of the ﬁnal output of the encoder
block. Thus this sublayer allows for the decoder to attend to diﬀerent inputs
encodings.
3.10.3
Masking
The inputs for both the encoder and decoder blocks are masked. In both the
encoder and decoder, masking the inputs at padding indices allows for the atten-
tion mechanism to only take relevant information into account (see section 3.9).

3.10.4
Residual Connections and Layer Normalization
A detail regarding both the encoder and decoder layers is the addition of residual
connections and layer normalization after every sublayer - sublayers being the
self attention layers, the feed forward layer, and the encoder-decoder attention
layers. The residual connections allow for the inputs to bypass the sublayers,
thereby allowing for the gradients to ﬂow more eﬃciently through the archi-
tecture (one of the solutions to the vanishing gradient problem in very deep
neural networks). Layer normalization ensures that the range of values in the
layers don’t change too much, meaning the model trains faster and has better
ability to generalise. Formally we have that the output from a given sublayer, z,

3.11
BERT
Bi-directional Encoder Representations from Transformers (BERT) (Devlin,
Chang, Lee, & Toutanova, 2018) is a speciﬁc use-case of the transformer ar-
chitecture explained in section 3.10. Essentially BERT is at it’s core a language
model and thus only utilizes the encoder block of the transformer architecture.
Language models compute the probability of occurrence of a number of words
in a particular sequence and form the foundation of most NLP models. Before
BERT, many language models focussed on directly predicting the next word,
however this gives rise to a bias that limits the learning of true context. Thus,
to overcome this challenge, BERT uses two training strategies - namely Masked
Language Model (MLM) and Next Sentence Prediction (NSP).
MLM refers to the task of guessing the correct words in a masked input sentence.
Speciﬁcally the original paper masks 15% of words in each sequence with a
[MASK] token. Doing this requires the addition of a classiﬁcation layer ontop of
the encoder block that takes the ﬁnal outputs of the transformer encoder and
predicts a word/token for each input. Speciﬁcally the loss function applied only
takes the masked tokens into account, such that the encoder isn’t completely
biased towards guessing the original input.
In the training process of BERT, the inputs are composed of a pair of sentences
and NSP refers to the task of indicating whether or not the second sentence
is a subsequent sentence in the document that the ﬁrst sentence is a part of.
Speciﬁcally, during training, 50% of inputs are a pair such that the second
sentence is the subsequent sentence in the document while in the other 50% the
second sentence is simply a random sentence from the corpus. The assumption
being that the random sentence statistically should be disconnected context-wise
from the original sentence.
Combining the two tasks into a single task requires the following modiﬁcations
to the transformer encoder architecture. The input is now composed of two
sentences (either a pair or not based on a coin-ﬂip), to allow for NSP. Each
sentence is tokenized and seperated by a [SEP] token. We also place a [CLS] as
the ﬁrst token of the input, which shall later be used as a means of predicting
whether the second sentence is connected to the ﬁrst in NSP. As with a standard
transformer encoder, a positional encoding is added to the token embeddings,
however we also learn a sentence type embedding which too is added.
The
sentence type embedding tells the model whether or not the input token is part
of sentence one or two. Finally, to accomodate MLM, 15% of the input tokens
are replaced by the previously introduced [MASK] token. The total loss function
is formed as a combination of the NSP loss - calculated by transforming the
output of the [CLS] token to a 2 by 1 vector using a classiﬁcation layer and
==PAGE:55==

applying a softmax to get a binary cross entropy loss - and the MLM loss which
is calculated using standard negative log likelihood on the masked outputs.

The dataset used to train and evaluate the models developed throughout the

The dataset consists of

are predicting based on semantic matching rather than semantic parsing. This
is evidenced in the fact that SOTA approaches to the WikiSQL dataset (Zhong
et al., 2017), SQLNet (Xu et al., 2017) and TypeSQL (Yu, Li, Zhang, Zhang,

database split setting.

Evaluation Metrics
The original Spider paper (Yu et al., 2019) proposes three types of evaluation
metrics for the dataset at hand, namely component matching, exact matching
and execution accuracy. Component matching refers to a detailed analysis of
a given models performance on the dataset in the sense that an average exact
match is evaluated between the prediction and the ground truth on diﬀerent SQL
components. Doing this yields direct insight into the shortcomings of the model
at hand. The components considered are SELECT, WHERE, GROUP BY, ORDER BY
and KEYWORDS. In this context KEYWORDS refers to protected SQL tokens, i.e.
tokens without column names nor operators. As evident, the collection of com-
ponents cover all possible SQL statements (note that the WHERE component also

metric.
Each component is then decomposed into bags of several sub-components which
are then checked for exact match. Speciﬁcally for the SELECT component con-
sider the following predicted query "SELECT avg(col1), max(col2), min(col1)
FROM tab", in this case the SQL is parsed and decomposed into the following
set {(avg, min, col1), (max, col2)} which is then matched against a similar
set from the true SQL statement. By doing this, we ensure that the order in
which the columns are selected is disregarded in the evaluation metric - this
makes sense to do as the same output is returned regardless of order when
querying the database.
In other words the following two queries would re-

"SELECT avg(col1), max(col2), min(col1) FROM tab".

on exact set matching.

5.1.1
SQL to SemQL
To generate a SemQL query from a SQL query, we start by initializing a Z
node. We then pass through the entire SQL statement looking for the tokens
UNION, INTERSECT and EXCEPT. For the sake of simplicity we assume that only
a single of the tokens can be present in any SQL statement, thus if a token
is found we note the ﬁrst action as Z →intersect R R, Z →union R R
or Z →except R R depending on the value of the token. This is the same
assumption made by the authors in the original Spider paper (Yu et al., 2019).
We then split the SQL statements into two separate statements, using the two
SELECT tokens. In the case none of the tokens are found, we know the ﬁrst
action is Z →R.
In the following we explain how to process a SQL statement containing a single
SELECT token. We start by scanning through the statement looking for values,
and save the values with corresponding position in a dictionary.
There are
two types of values contained within the Spider dataset, strings and numerical.
For strings, we denote tokens surrounded by quotations as values. As for the
numerical values, we look for SQL operations - including >, ≥, <, ≤, =, ̸=,
between, like and not like - and register the following token as a value. In
the case that the operation is between we store the two values separated by an
and token. Finally we look for the token limit and save the following number
as a value.

This is done by

dictionary with key corresponding to the following token. In the case where
there is no as token (when there is only a single table) we simply insert the token
following the from token with default key T. In order to determine the next
action for the generation of the SemQL tree, we parse the SQL query looking for
where, having, order and limit tokens. Depending on which tokens appear, we
are able to determine the next action, for instance if a having token and a order
token appears, we know that the next action is R →select order filter.
Next the parser deals with the columns present between the select and from
tokens - i.e. expands the select node of the tree. This is done by counting
the number of ’,’ tokens present, indicating the number of columns and thereby
the next action in the sequence. If there are 3 commas present we know the
next action to be select →A A A A. The parser then ﬁlls in the columns and
table names for each A node, checking for aggregations as well as table preﬁxes,
which can be mapped back to a table name using the table dictionary. It should
also be noted that we make use of the extra action →distinct C in the case
of distinct columns. This procedure of generating columns and tables from A
nodes is saved as a function to be used later in the parser.

the original SQL string from either the having or where tokens until a group or
order token is reached. We start by looking for and and or tokens. In the case
either of these tokens are found the corresponding action is saved to the action

One of the child filter nodes is then resolved as follows. First the operation is
identiﬁed determining the correct following action, where-after the child A node
is resolved in an identical manner to the way described for the select node.

filter node have an additional V node. This is dealt with by use of the value
dictionary initially generated by the parser. Speciﬁcally the position in which
the column arises is used as key to ﬁnd the corresponding value.

similar fashion to what has been described.

5.1.2
SemQL to SQL
In the following section we describe the process of going from a SemQL tree to
an executable SQL query. However, before going into details, it is important to
note that the method by which a given SQL query is transformed into a SemQL
==PAGE:68==
56
Intermediate Representation Network
tree ensures that the tree is generated in a depth-ﬁrst, left-to-right order (more
commonly known as a preorder traversal).
Thus, when regenerating a SQL
query from a SemQL tree, we start by performing a preorder traversal of the
tree, generating a list of nodes.
We then initialize an SQL statement as an empty list and iterate through the
list of nodes, appending SQL tokens to the list depending on the type of node.
The ﬁrst type of node considered is that of a select node. For the select
node we append the token SELECT to the SQL list and process all the following
nodes that have a depth in the tree greater than the select node. Speciﬁcally
we resolve A nodes by checking for aggregations and distinct tokens - formu-
lating the corresponding SQL components. For instance if it is the second A
node encountered, we append a ’,’ token to the SQL list, and if the node has the
children max, col, T, and the col node has children distinct and C, the corre-
sponding SQL statement produced would be ’, max(distinct t1.c1)’. This is
under the assumption that the column name provided by the C node be c1. As
for the provided table name, we maintain a dictionary of tables that appear as
we process the list of nodes, assigning each distinct table the key tx where x is
a counter that increases with the number of tables. Hence, we can ﬁnd the key
to pre-append to the column name by a simple lookup in the dictionary. We
also maintain a column dictionary as well as an aggregation list, such that we
can infer which columns to group by later in the SQL generating process.
Once all child A nodes have been decoded, we append a FROM token to the SQL
list and continue iterating through the node list. If a filter node is met, we
start by checking for and and or children, if any is found a respective ﬂag is set
true and we move onto the next filter node. We then decode this node by ﬁrst
checking whether or not the child A node has an aggregation. In the case that
it has an aggregation we know that the ﬁlter is linked to a having clause (in
which case we update the global aggregation list), and otherwise just a simple
where clause.
The child A node is then decoded by the same method as for the select node,
i.e. by the use of the table dictionary, followed by the appending of the respective
operation and the value in question or a VAL token depending on whether the
SemQL tree contains values or not. The resulting SQL statement is then stored
for later use. In the case of the filter node having a between operation, we
add two VAL tokens separated by an and token in the case where no values are
present and simply the two values if they are.
We then continue iterating through the node list until all filter nodes have
been dealt with - remembering to separate the resulting where and having
clauses with and and or tokens respectively. In the case where one of the ﬁlter
nodes contains an R child, i.e. a sub-query, we simply make use of a recursive
==PAGE:69==
5.2 IRNet
57
function call appending the resulting SQL to either the having or where list
depending on the corresponding A node. We use the depth of the tree to ensure
that we only process the nodes that belong to the sub-query.
Once all filter nodes have been decoded, we append WHERE and HAVING fol-
lowed by the respective sub SQL statements generated in the decoding process
(assuming that the lists are non empty). We then continue iterating through
the node list. In the case that we ﬁnd an order node, we append the tokens
ORDER BY to the SQL list and decode the child A node using the same methods
as previously detailed, appending either an ASC or a DESC token depending on
the node type.
The ﬁnal node type considered is the superlative node. This node is decoded
in a similar fashion to the order node, with the exception that, depending on
whether the tree contains values or not, the ﬁnal SQL will contain a LIMIT token
followed by the true value or a VAL token respectively.
Having processed all the nodes in the SemQL tree, we simply need to include
a group by clause (in the case that there are aggregated and non-aggregated
columns present) as well as the tables and any potential joins. The group by
clause is formed using the maintained column and aggregation list. Speciﬁcally
we iterate through the column list appending the column if it does not contain
an aggregation.
The from clause is generated from the table dictionary by
identifying the shortest path that connects these tables in the schema. In general
database schemas can be formulated as undirected graphs, where vertices are
tables and edges are relations among tables - which can either be primary or
foreign keys.
Speciﬁcally we choose the primary table to be the ﬁrst table
encountered in the SemQL tree. For any additional tables we use a depth-ﬁrst
search to ﬁnd the shortest path between primary keys and foreign keys to the
original table. In the case where there is no direct path, we add a join to the
intermediate table.

5.2.1
Schema Linking
The goal of schema linking is to provide the network with the ability to recognize
columns and tables mentioned in the NL question, and assign diﬀerent types to
those columns and tables depending on how they are mentioned. The mechanism
works by generating a list of all n-grams of length 1 to 6 in the original question,
and then rearranging them in order of descending length. We then loop through
the n-grams and stem the words within using the Porter Stemming algorithm
(Porter, 1980), we similarly stem the words that appear in the natural language
version of the column and table names respectively. The reason for stemming
comes from the fact that when posing questions in NL, column names and table
names typically appear in their inﬂected versions, and by stemming we generate
the roots of the words which tend to preserve the true meaning of the word. To
our knowledge, this was not done in the original IRNet paper. We then iterate
through the stemmed n-grams checking whether they exactly match or are a
subset of a stemmed column or table name. If any n-gram matches we recognize
it as either a Column or Table respectively, and remove all overlapping n-
grams from the remaining list.
In the case that an n-gram matches both a
column and a table we prioritize column. In the case where an n-gram starts
and ends with quotations, we recognize the n-gram as a Value and similarily
remove all overlapping n-grams. Once we ﬁnish iterating through the list of n-
grams, we develop a non-overlapping n-gram sequence by joining all recognized
n-grams with the remaining 1-grams. We refer to recognized n-grams as spans.
Having mapped types to the NL question, we turn to the schema. For each
span recognized as Column, we assign the type ExactMatch to the column in
the schema if the span exactly matched the stemmed natural language version
of the column, and assign the type PartialMatch otherwise.
In order to
link the cell value with a it’s corresponding column in the schema, we utilize a
open, large-scale knowledge graph known as ConceptNet (Speer & Havasi, 2012).
Speciﬁcally we pass all Value type spans through ConceptNet creating a list
of words that appear in the IsA (with the value as head) and RelatedTo
edges.
This list of words is then compared to the natural language column
names in the schema, if an exact match occurs that column is given the type
ValueExactMatch and if a partial match occurs (the result belonging to a
subset of the column name) the column is given the type ValuePartialMatch.

BERT
In order to further boost performance IRNet integrates a pre-trained BERT
model to replace to question and schema embeddings. This idea originally stems
from the paper ’A Comprehensive Exploration on WikiSQL with Table-Aware
Word Contextualization’ (Hwang, Yim, Park, & Seo, 2019), and is motivated
by the fact that since the release of BERT (Devlin et al., 2018) it’s integration
into NLP models of various kinds have led to impressive performance boosts.
Speciﬁcally in IRNet BERT is used to encode questions, database schemas and
the schema linking results.

question concatenated with all the distinct table names in the schema.

6.3.4
Memory Augmented Pointer Network
The next addition made within the ablation study is that of the memory aug-
mented pointer network used in the prediction of the action SelectColumn.
Speciﬁcally we initialize a learned parameter wm ∈Rda as well as a memory unit
such that the action probability for the SelectColumn action is calculated us-
ing Equations (5.13), (5.14) and (5.15). The memory unit contains previously
selected columns, of which are removed from the original schema once chosen.
This implies the rise of another type of frontier node, namely that of having
selected a column from memory. We thus include an extra type of node in our
embeddings of nf such that the model can distinguish between having chosen
from memory or not when predicting following actions wrt. parent feeding.
6.3.5
Skeleton Decoder
We now alter the architecture to make use of coarse-to-ﬁne decoding (Dong &
Lapata, 2018). This requires a signiﬁcant change to the current decoder archi-
tecture as well as the input data - the encoder however remains unchanged. We
start by generating a sequence of "skeleton" SemQL sequences to be fed to the
coarse decoder, which shall be referred to as the skeleton decoder. Speciﬁcally
we take the original sequence of actions and strip all actions containing a head
of C or T. Doing this gives rise to a sequence of actions preserving the funda-
mental query structure without details - being column and table names. The
skeleton action sequence is then embedded using purely a learned embedding as
the sequence is composed of SQL type actions only.

We then apply methods similar to the ones described above generate a skele-
ton sequence given the question embeddings from the encoder. The only real
diﬀerence being that there is no need for SelectColumn or SelectTable ac-
tions and thus we remove the type of node embedding nf from the input to the
skeleton decoder (as there is only one type of node) and strip away the memory
augmented pointer network as the decoder doesn’t predict columns.
Given the output of the skeleton decoder, a skeleton encoder is constructed. The
skeleton encoder is composed of a bi directional LSTM very similar in nature to
the former developed encoder. The only diﬀerence being that instead of taking
question tokens as input, it takes the predicted skeleton as input and returns
ﬁnal hidden layer of the LSTM.
The ﬁnal addition consists of a detail decoder. The detail decoder is similar
in nature to the original decoder developed in the previous sections (that is
contains both memory, schema linking, parent feeding etc.). The only diﬀerence
being that the input is given by Equation (5.19), in which case the vectors vk
refer to the outputs of the skeleton encoder. It should also be said that whenever
an action of the type ApplyRule is encountered, the output is overwritten by
the greedy decoding of the skeleton sequence formed by the skeleton decoder.

Results Discussion
The results of the ablation study can be seen in table 6.3. We ﬁrst notice the
signiﬁcant increase in performance for both the Basic and the full dataset - go-
ing from 12.5% and 6.8% (Seq2Seq Attention) to 25.7% and 18.1% respectively
(Seq2SemQL Attention). This increase seems to indicate that the core idea of
IRNet, being the prediction of an action sequence based on the SemQL gram-
mar, is well suited to the problem at hand. Secondly, we notice that the exact
matching accuracy indeed increases with model complexity. Speciﬁcally we see
two large increases for both datasets with the addition of the schema linking
mechanism (SL) as well as the coarse to ﬁne decoding (C2F) - the biggest of
the two being the schema linking. However, we see a large discrepancy between
the results found in table 6.3 and the ones reported in the original paper (Guo
et al., 2019). The original paper achieves an exact matching accuracy of 40.5%
on a base model without the use of schema linking, memory augmented pointer
networks or coarse to ﬁne framework, whereas our model (thought to be similar
in nature) only achieves a mere 18.1%. It should be noted that this discrepancy
seems to indicate a lack of details mentioned in the original paper as the models
built in this thesis are based completely on the explanations provided by the
paper. It would have helped the process if the source code had been uploaded by
the authors, however (although promised in the original paper) this was never

as a value can be composed of multiple words/values and without access to the
values within the schema in question it is not possible to reduce this to a simple
selection problem. We deal with values by letting the GenToken(v) append
terminal nodes to the current frontier node, v. Furthermore we introduce the
special token <\n> to denote when the node should be closed and the program
move onto the next child. These so-called terminal nodes can be generated from
a pre-deﬁned vocabulary or copied directly from the input question.
The terminal nodes are dealt with by introducing the following marginal prob-
ability,
p(at = GenToken(v)|x, a<t) =p(Gen|x, a<t)p(v|Gen, x, a<t)
+ p(Copy|x, a<t)p(v|Copy, x, a<t),
(6.15)
where the selection probabilities p(Gen|·) and p(Copy|·) = 1 −p(Gen|·) are
given by multiplying the current decoder hidden state st with a learned param-
eter ws followed by a softmax operation, i.e. p(Gen|·) = softmax(ws · st). The
probability of generating token v from a predeﬁned vocabulary is then given
by a similar equation to the one used to generate a SQL speciﬁc action (using
the ApplyRule(r) action) given in Equation (5.12). However, instead of using
the matrix w we multiply by the wv matrix, containing value token embed-
dings from the predeﬁned vocabulary. In the following results we generate these
embeddings using GloVe.
In order to model the copy probability, we use a pointer network (Vinyals et
al., 2015) to compute the probability of copying the ith word from the input
by attending to the output of the encoder layer. Speciﬁcally the probability is
modelled as follows,
p(wi|Copy, x, a<t) = softmax(ω(hi, st, ct)),
(6.16)
where ω denotes a single hidden layer FFNN with hidden dimension 300. We
also use the sigmoid activation function in the hidden layer. It should also be
noted that if the word to be copied is an out of vocabulary word (OOV) we
simply copy it directly from the input.

6.6
Multilingual Model
In the following section we use LASER embeddings (see section 3.8) in an at-
tempt to make the base model multilingual. Speciﬁcally we translated the ques-
tions that appeared in the Basic dataset into 13 diﬀerent languages using the
Google Cloud translation API. The languages chosen were, Czech, Danish, Ger-
man, Spanish, Farsi, French, Icelandic, Italian, Kurdish, Dutch, Norwegian,
Portugese and Swedish. Doing this inceases the training set to 50,563 training
points as opposed to 3,945 and a total of 7,005 development points compared to
544. An example of how some of these translations might look is given below,
EN: How many singers do we have?
FR: Combien de chanteurs avons-nous?
CS: Jaký je celkový počet zpěváků?
The only diﬀerence occurs in the input used for the NL encoder as well as
the representation of the columns and tables present in the schema. Instead of

elements of IRNet are left unaltered.

Results Discussion
Due to the fact that the questions can now occur in 14 languages, it is impossible
with the methods described in the original IRNet paper to perform any sort of
Schema Linking. Thus the base model used to build a multilingual model upon
is that of the Seq2Semql Attention PF. The exact matching accuracies across
diﬀerent SQL diﬃculties can be seen in table 6.6. As evident, when comparing to
the base architecture (see table 6.3) the multilingual model obtains an increase
in performance of roughly 4%. This may at ﬁrst glance seem counter intuitive
given that we have made the problem more diﬃcult by forcing the model to
become language agnostic. However, by translating the questions using Google
Translate, we have inherently introduced a lot of noise to the questions (due to
diﬀerences in word position across languages as well as speciﬁc wordings), as well
as provided a lot more training data (roughly 13 times more). This extra noise
helps reduce the potential for overﬁtting and hence actually increases model
performance.
Given more time we would have extended upon the schema linking methods
detailed in the IRNet paper to be able to deal with multiple languages in order
to leverage upon the extra performance boosting components. The results also
seem to indicate that translations provide a good means of data augmentation
for the speciﬁc task at hand.

Transformers
In the following section we present some of the experimental architectures con-
sidered in an attempt to improve on IRNet. Given the recent surge of interest in
transformer models (BERT being the prime example), and due to their training
eﬃciency, we thought it interesting to pursue architectures similar in core to
IRNet but using transformer architectures inplace of RNN’s.
6.7.1
Baseline Transformers
To start with, we make a naive transformer model. This transformer model
consists of a full transformer architecture of the type detailed in section 3.10
with 6 multi-headded attention heads and 6 encoder and decoder stacks. As with
the naive baseline models outlined in section 6.2, we use GloVe embeddings to
initialize both questions and columns and tables present in the schema. In order
to make ﬁnal predictions we use the same equation as the one used in the orginal
baseline models (see Equation (6.5)). However, in this case yd represents the
ﬁnal output of the decoder stack as opposed to the ﬁnal hidden layers of the
decoder LSTM.
The exact matching accuracies on the Basic dataset and the full dataset were
9.4% and 5.5% respectively.
This places the naive transformer between the
Seq2Seq and the Seq2Seq Attention baseline models (see table 6.2). This seem-
ingly indicates that the LSTM decoder seems like a method well-suited for the
speciﬁc task at hand and can’t necessarily be improved by the introduction of
transformers.

Chapter 7
Discussion and Conclusion
In conclusion, we managed to build a model that performs with development
exact matching accuracy of 33.0% taking distinct tokens into account.
This
result beats the SyntaxSQLNet + augment model on the oﬃcial Spider leader
board and would have been in the top 3 ranking at the time of initializing
work on the thesis. The result is however far from the reported results in the
original IRNet paper (Guo et al., 2019). There can be many reasons for this,
some of which are related to the fact that the SemQL grammar used throughout
the thesis diﬀers from the originally speciﬁed one. The reason for this was to
allow the model to predict on distinct tokens (thought very useful in real world
applications), which were ignored in the original paper. This addition of depth
to the SemQL grammar may have impacted performance. Other reasons include
a diﬀerence in overall implementation. Due to the lack of implementation details
speciﬁed in the original paper a lot of design choices were made throughout the
model building exercise, some of which may have been wrong. An important
note however, is the fact that most deltas in the ablation study are of the same
magnitude as the ones reported - indicating successful implementation of the
extra components introduced in IRNet.
We also successfully managed to implement an extension to the proposed ar-
chitecture in the original paper allowing for the model to predict on executable
SQL queries rather than queries with special tokens instead of actual values.
Speciﬁcally we show that the performance only drops with approximately 5% in
==PAGE:108==

exact matching accuracy compared with the oﬃcial Spider evaluation methods.
This means that the academic proceeds within the realm of text-to-SQL can be
translated into real world applications.
We also show that LASER embeddings (Artetxe & Schwenk, 2018), can be
applied with great success in an attempt to make text-to-SQL models language
agnostic.
Again this is a very desirable result with respect to possible real
world applications. Actually we show that augmenting the original dataset with
translations of the original questions provides better results and can thus be
seen as an eﬀective method of data augmentation going forward.
7.1
Outlook
Given more time, further investigation into transformer based methods would
have been of great interest. From current work, it would seem that naive im-
plementations of transformers are ill-suited for the very constrained grammar
decoding utilized in IRNet. Results seem to indicate that the models are overpa-
rameterized, a problem that could be solved by either providing the models with
more data (through data augmentation) or by making simpler architectures.
Another issue that is inherent with constrained decoding is the fact that if any
given action along the sequence is wrongly guessed, the remaining sequence can
be branched in a completely wrong direction by the grammar. A usual method
to overcome this is through the use of beam search during the synthesis of
queries - however in the current architecture this is impossible to implement.
An interesting paper to overcome some of these issues is ’Sequence-to-Sequence
Learning as Beam-Search Optimization’ (Wiseman & Rush, 2016). The core
idea is to embed the ideas of beam search into the training process of sequence
to sequence models. This means that instead of being a point-wise loss function
(as is the case with Negative Log-Liklihood) the loss is on sequence level -
implying that issues regarding the model never experiencing the impact of it’s
own errors during training time due to teacher forcing are alleviated. Given
more time, an implementation of Beam-Search Optimization would have been
of great interest.
