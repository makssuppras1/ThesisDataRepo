Abstract
This thesis manages to extend the model­based testing tool SeTTs such that it is appli­
cable to complex RESTful applications. Additionally this extension enhances the test­
generation of SeTTs and implements a coverage­based, search­based test­generation in
opposition to the inherent random test­generation.
GitLab­CE (Community Edition), an open­source RESTful application for version­control
was used as an example of a complex RESTful application. Challenges encountered in
this extension of SeTTs to be able to test GitLab included: The gathering and several
refactorings of the swagger definition (the OpenApi.yml) for GitLab. Providing utility tools
for the system under test for storing information while testing. Implementing tools for the
encapsulation of HTTP request methods which become increasingly complex as the sys­
tem under test grows in complexity. This encapsulation managed to wrap the desired
complexity for the tester, so that test­cases remain simple and readable.
The implementation of code­coverage as a metric for testing and a defined, directed
structure for the test­generation, managed to enhance the performance of the test­driver
generated by SeTTs remarkably.
The enhanced search­based, coverage­based test­
generation results is recorded in logs generated while testing the system under test.
These results suggests a substantial reduction in the API calls required to achieve a high
code­coverage, as much as a 33.3% less. Lastly the thesis addresses limitations of the
implementations made and suggests a direction for future work and research relating to
SeTTs.

Introduction
Within the domain of Software testing, there is multiple challenges that are unsolved, and
it remains challenging and expensive to develop tests for software. In 2020 Vahid et. al.
released a paper exploring the challenges facing the software industry. Through an empir­
ical survey asking software engineers, they discovered that there is an unfulfilled demand
for options in regards to improving both the efficiency and the effectivity of software test­
ing. The practitioners of the survey proposed a wide variety of desired research topics,
including: ”Need for better test­case data generator tools”, ”Need for automated unit­test
code­generation based on source­code” and ”Help is needed for writing mock­driven unit
tests (which are tedious [to write])” etc. Vahid et. al. invites the research community to
undertake these research topics in order to provide suggestions for the challenges they
pose [1]. Besides from that, it is no secret that software development is expensive and
there by as extension, software testing remains expensive too. In the classic textbook
”Software Engineering”, Sommerville claims that a staggering 40% of the budget of soft­
ware development should be reserved for testing [2]. It therefore seems obvious that
improvements in either the development or the streamlining of the testing process would
be more than appreciated in software development.
This thesis embarks in the pursuit of further development of a model­based testing tool for
RESTful applications ­ SeTTs (Session Type­based Test Synthesis)[3]. The SeTTS tool,
developed in Scala, by Christian Bartolo Burlò, Alceste Scalas, Emilio Tuosto and Adrian
Francalanza proposes an alternative model­based way of developing and executing tests.
This tool has the potential to drastically reduce the lines of codes, increase the amount
of code­coverage and transparency of the test developed for a RESTful application. Fur­
thermore, this tool enables testing of RESTful applications, where the test developer has
little to no information about the system under test, besides from the documentation(in
this case the swagger definition). The tool works by parsing a domain specific language,
based on Session­Types and thereby generating a test­driver. This test­driver then makes
API calls to the system under test, based on these Session­Type files and receives HTTP
responses, which the driver then can compare against expected HTTP responses.
Essentially, The test developer only needs to develop the Domain specific language file
(with the extension .st) for the tool, to generate a test­driver and the tester is thus able
to develop test suites to applications only knowing the use­cases and expected behavior
of the System under test.
This tool has great potential, but has yet to be applied on large scale systems. So far it
remains only applied on example RESTful applications like spring­petclinic [4] and alike
with great results. This bids the questions whether it is possible to apply SeTTs to a
more complex RESTful application, and in which manner the tool has to be modified in
order to achieve such an appliance. For test­generation, SeTTs uses a random approach
which also invites to further research. Could this test­generation benefit from alternative
approaches, and if so how would such an approach work in practice?

Contributions
This thesis manages to extend the model­based testing tool SeTTs which enables it to be
applicable to complex RESTful applications (GitLab in this thesis). This is achieved by
implementing utility methods for storing of data while testing and introducing tools for the
encapsulation of complex HTTP requests. Additionally a brief guidance on how to apply
the tool to any giving RESTful application is provided.
Additionally this thesis achieves a more effective test­generation by implementing a coverage­
based search­based algorithm rather than the default random test­generation that is inher­
ent to SeTTs. This is achieved through a proof of concept where the generated test­driver
is modified to adhere to a tree­like structure which enables a conceptual coverage­based
and search­based algorithm design. Additionally this novel design displays a remarkably
increased performance compared to the inherent random test­generation.

2.1
Model­based Testing
Model­based testing is a branch of testing that relies on an explicit model that encodes
expected behavior of a System under test (SUT) and perhaps its environment. Test­cases
are then generated from one of these models, in order to fit a specific SUT. This process
is aiming to make the testing more structured, easier to reproduce, easier to document
and with a much more clear rational for the testing that is conducted[5].
2.1.1
Taxonomy & Approach
In 2012 Utting et. al. published the article ”A taxonomy over Model based Testing ap­
proaches”, which developed a taxonomy over model­based testing (MBT), in order to
achieve further transparency when model­based test suites was being developed. This
thesis will try to adopt this taxonomy and further develop on the model­based test tool
SeTTS.
Utting et. al. describes six essential dimensions of MBT which characterizes the different
approaches to developing a MBT tool [5]. Utting et. al. furthermore describes a process
of five steps to follow in order to achieve a generic development process of a MBT­tool.
The general five steps to develop a MBT­tool is as follows:
1. Building a model: A model of SUT is build or generated from existing specification
or requirements.
2. Test Selection: In order to achieve a well rounded test suite, test criteria have to be
selected which aligns with the test policy defined by the SUT. Test selection can re­
late to either functional (requirements­based), structure (state, transition or dataflow
coverage), data coverage(pairwise, boundary or value), pure randomness or a set
of faults [5].
3. Test case specification: The above mentioned test selection criteria is transformed
into test case specifications. These are listings of desired reachable states from the
SUT’s performance under the tests.
4. Test generation: Once the model and test cases are developed, test cases that
satisfy these are generated.
5. Test execution: After everything have been generated and align properly, the test­
cases are run and it will then be clear to what extend the SUT complies with the test
case specification.
[5] Utting et. al. describes the additional six dimensions as the product of an investigation
of how the process of developing a MBT­tool is conducted. These dimensions are mostly
independent of each­other but still contain some overlaps. The dimension directly reflects
the process described above, where each step has one or more dimensions attached to

2.5
Session Types
The MBT­tool that this thesis is based upon uses a concept know as session types. SeTTs
[3] uses Session Types to the structure the Domain specific language files, known by the
file extension .st. These structures perform and describes the test case specification of
a given test suite. Session Types proposes a way to interpret distributed systems in an
elegant way with the help of clearly defined types. Hüttel et. al. describes: ”well­typed
systems are well behaved” [13]. Hüttel et. al. describes in their survey on Sessions Types
and Behavioral Contracts, session types as a way to view communication between two
participants in a system as a message exchange[13]. The survey uses the example of
!nat.?bool.end, which denotes that one part is sending a natural number, nat with the !
notation (output), and then expects a bool answer with the ? notation (input) [13]. Given
that T1 and T2 are different types, then !T1.?T2 denotes the output of type T1 and an
expected followed input of type T2. The end of the interaction and protocol is denoted by
end. A very simple example would be an interaction of two parts of a system. This would
then be denoted as !T1.?!T2.end. It then also follows that the opposite participant of the
protocols interaction could be described as ?T1.!T2.end [13].
in the paper ”Lightweight Session Programming in Scala”, Scalas et. al describes the
grammar of session types as the following:
S ::= &iϵI?Li(Ti).Si| ⊕iϵI?Li(Ti).Si| µx.S| X| end T ::= B| S(Closed) [14]
Where S is a session type, T is a payload type, B is the set of basic types, end denotes
the termination of a session, µx.S denotes a recursive session type where µ binds X
and X is a recursive variable. The two longer pieces of the syntax is denoted by the
& and the ⊕, which are the external choice and the internal choice of the session type.
&iϵI?Li(Ti).Si denotes that the system is required to receive input in the form of Li(Ti) for
any iϵI chosen at the sending endpoint of the session. Following this, the continued use
of the channel must follow Si [14]. In layman terms, this notation denotes that an external
source can choose one and only one, in an array of possible sessions shown in figure 2.2.
In opposition to this, the ⊕iϵI!Li(Ti).Si denotes an internal choice that the system has to
make. The system has to send one session of i choices followed by Si, also shown in
figure 2.2.
The example at 2.3 describes a very simple ATM that patiently awaits an input in the form
of either a withdrawal, a deposit or an exit. This session type is recursive. This means that
if either the withdrawal of the deposit is chosen, the sequence will pick up at the beginning

2.6.2
Model­based Taxonomy
SeTTs is a model­specific tool that according to the previously described taxonomy in sec­
tion 2.1.1 and according to the authors of the paper ”Model­based Testing for Message­
passing, RESTful Applications” can be described with Utting et. al.’s taxonomy[3]. The
following is the authors understanding of how SeTTs fit into the taxonomy based on Utting
et. al. and the report describing SeTTs [3] [5].
• Scope: The scope both surrounds the input and output of the SUT, meaning that
SeTTs is both able to capture and verify the outputs of the SUT. Simultaneously
SeTTs can model the SUT’s environment and thereby provide API calls [5]. Both
of these features are provided by SeTTs DSL that captures both input and output
through the syntax of session­types [3].
• Characteristics SeTTs includes discrete events, meaning that it is a discrete MBT­
model. The DSL that SeTTs uses describes API calls as separate and discrete
events, while still maintaining a hierarchical order. Some events cant occur before
others and vica versa. An example of a discrete event created by SeTTs could be a
POST request creating an account or user on a certain SUT at 2.6. This also entails
that the order of API calls that SeTTs can invoke, can be understood as a tree or a
graph of these events (see 2.8).
• Paradigm SeTTs is a MBT with an operational paradigm because it simulates and
tests the interaction between two elements: The SUT and the pseudo client that
SeTTs generates from a swagger definition.
In addition there is stochastic ele­
ments embedded in SeTTs, every time a choice(internal or external) has to be made.
Where SeTTs will choose one at a probability of 1/choices, in 2.8 this probability is
1/4 = 0, 25 i.e. 25% [5].
• Test Selection Criteria: SeTTs differentiate between structural, random, stochastic
and ad hoc test specifications [3]. The DSL support all of these specifications.
When the test­driver is generated, the test cases that are structured according to
the DSL and thus inherits what ever structure that the DSL file models. This can
be more or less structural in the sense, that there may or may not be a rigorous
structure defined by the DSL, with no randomness present in the testing. SeTTs
could potentially also support random test selection, if methods supporting this was
implementing in the util class. Additionally ad hoc also applies to SeTTs, since the
tester has total control over which test (in the case of SeTTs: API calls) should be
performed. The tester using SeTTs, can make however large or small of a DSL file
they want, and thus generate test­driver that only performs a single API call to test
a very niche case, if desired.
• Test generating technology Is mainly random because the DSL has the ability
to produce random chosen tests, which can have random input generated by the
util.scala class. Currently, whenever an internal choice is to be made by the test­
driver, the choice is random. This means that the introductions of internal choice
also entail the introductions of a random generation of tests. Utting et. al. describes
this as a ”Random walk” [5]. Probabilities can however be customized by the user
of the SeTTs tool [3].
• Test execution: On/Offline SeTTs test execution is online [3]. SeTTs models a
client to the SUT and interacts with the SUT in real time, invoking API calls that can
be used as assertions. When a test fails, SeTTs provides an offline analysis of why
14

==PAGE:20==
and how this test failed [3].

2.7.1
RestCT
Huayao, Xintao, Lixin and Changhai published a paper ”Combinatorial Testing of Restful
APIS’s” in 2022 [16]. This paper describes a testing tool ”RestCT” developed for au­
tomated testing of RESTful API’s. This tool uses what Huayao et. al. calls a cover­
ing array (CA). the RestCT tool assumes that the SUT is influenced by n parameters:
P = {p1, p2, ..., pn}, and that such parameters has value domains: V = {v1, v2, ..., vn}
thereby the covering t­way array becomes the array of the parameters and the values
that influence those parameters, which thereby is a quasi array of test­cases for the
RestCT tool [16].
Additionally the system can often be influenced by distinct events
E = {o1, o2, ..., on} which in combination with the covering array, becomes a sequence
covering array [16].
Figure 2.9: The covering, and sequence covering array from RestCT [16]
The t­way combination in the arrays at 2.9 thus covers the entirety of the SUT functionality.
Using the swagger definition for the SUT, RestCT then progresses in a workflow of two
phases, an Operation Sequence generation and constructing covering arrays as a set of
the operations sequences of the SUT. The key point here is to make sure that the opera­
tions sequence does not violate CRUD semantics ie. POST X should be conducted before
DELETE X, GET X should not be conducted after DELETE X etc. A t­way sequence is
constraints­satisfying if s = {ok1, ok2, ..., okn} if k obeys the CRUD­semantics as described
with X above. RestCT achieves this by implementing a greedy algorithm [16]. Once the
operation sequence is created, RestCT generates concrete HTTP requests to these se­
quences by using the swagger definition’s parameters to create the input­parameters.
These would either by dynamic (created by information gathered at run time), specific
(generated directly from values in the swagger definition), successful (values that have
been successful previously) or random (within a defined value domain)[16]. The test suite
that RestCT creates is thus a combination of the operation sequences and the input pa­
rameter sequence.
Through testing on 11 REST API, restCT manages to rest 56% of the described oper­
ations within a swagger definition while being substantially cheaper (in regards to time
spend running the tool) to run than RESTler (another automated testing tool) [16]

2.7.2
RESTful API Automated Test Case Generation
In 2017, Arcuri proposes a novel way to automatically generate test for RESTful API. This
approach was to test RESTful API in isolation, i.e. not for testing SUT’s that were com­
posed of several microservices. Besides from that, the availability of the source code
from the SUT was assumed to be accessible. The goal of the project was to generate test
cases that resulted in high code coverage, and that could detect faults in the implemen­
tation of the SUT [17]. Before Acuri proposes an approach, four elements are defined:
”The test case”, ”The Oracle”, ”The Code Instrumentation” and ”The search algorithm”.
Acuri described a test case as follows: One or more HTTP requests sent towards a REST­
ful service. To make sure that the correct API methods was provided for this projects,
Acuri, just like SeTTs, uses the swagger definition of the SUT to provide the correct road­
map over the API methods available for the SUT [17].
Furthermore Acuri tries to define an oracle for the automated testing. An Oracle is, in
this context, a functionality that determines whether or not the test­cases are asserted
correctly, i.e. if the response to a test­case (in this project an HTTP request) is correctly
asserted. Acuri refers to the response­codes from the SUT to be a potential oracle due to
the nature on the 5xx response codes. These usually represents an internal server error
or bug, but also refrains a little from this oracle, due to the concerns that the SUT does not
correctly perform input validation. This would thereby misleadingly returns a 5xx code,
when in reality, a 4xx code should have been returned [17].
In order to collect code coverage efficiently, Acuri invokes a JAVA agent (the paper focus
primarily on JAVA code coverage) that probes the bytecode generated from set classes.
An interesting point is made regarding code coverage and its efficiency as a metric. Acuri
points out that certain code can be gate kept by if­blocks guarded by conditions that ran­
dom inputs is unlikely to solve. Acuri proposes to use heuristics to measure how far the
random generation is from these potential gated block and thus use this heuristic to ac­
cess them [17]. As the search algorithm, Acuri uses a genetic algorithm to generate tests.
The program will then produce a test­suite that tests against code coverage and HTTP re­
sponse codes. The genetic algorithm will mutate this test­suite [17].
The implementation of these approaches was called EvoMaster. This tool was able to
identify 38 new bugs (5xx response codes) in the web­services: FeaturesService, Indus­
trial and ScoutAPI [17].

Model­based Testing in Practice
In 1999 Dalal et. al. published an article that proposed a model­based testing technology
for software systems, that would focus on generation of infrastructure and data models,
instead of individually crafted tests [18].The motivation behind this was, among academic
aspiration, to ease the required expertise from commercial testers ­ which according to
Dalal et. al. needed to be 1/3 developer, 1/3 system engineer and 1/3 tester [18]. Dalal
et. al. Describes that there are three technologies that lay the foundation for MBT: the
notation used for generating the data model, the test­generation algorithm and the tool
that generates supporting infrastructure for the test­system [18].
Dalal et. al. used the AETG software system to generate combinations of input values to
the SUT [18]. AETG uses data­models that could potentially consist of a range of values,
constraints among these values and a list of required values that must appear in the test
generated. When such a data­model is created, the data­model is checked and then the
AETG software generated the test­cases from this data­model. Dalal et. al. is somewhat
dated (1999), and therefore the exact technology that is described in this paper seem
out of scope. But Dalal et. al. still touches the heart of the issue. Testing seems to
require an excessive amount of resources. Through multiple cases using this approach
of developing data­models and then generating test­cases with the AETG tool, dalal et.

2.7.4
Morest: Model­based RESTful API Testing with Execution Feedback
Another MBT­tool like SeTTs is the tool morest. morest is developed by Liu et. al. and
described in the article ”morest: Model­based RESTful API Testing with Execution Feed­
back” [19]. Liu et. al. utilizes swagger definitions [7] and OpenAPI generator[8] to gen­
erate test­suites. morest uses OpenApi generator to build what they call an ”RPG”. The
RPG is used to encode CRUD relations, schema relations and dataflow from the swag­
ger definition(a schema would be an API endpoint). The design of the RPG is based on a
property graph, which is a directed, edge­labeled, attributed multi­graph [19]. morest tra­
verses the RPG and creates concrete inputs for the API of the SUT. The traversal is done
with an algorithm that visits every schema in the generated RPG. The operations related
to these schemas are then collected to build up a call sequence, additionally, morest re­
cursively applies this process to other schema and their related operations[19]. The call
sequences that morestcreates requires parameters, and therefore random generated pa­
rameters (based on ranges from the swagger definition) are used as input for these call
sequences. If a call sequences is successful in a giving run, Morest is more inclined to
use that parameter again. Liu et. al. notes that these parameters are not generated all
at once, but rather generated on the fly because a parameter of a call can be determined
by the previous call [19].
morest dynamically updates the RPG generated from the swagger definition, which en­
ables the tool to fix errors in the graphs generated. The edges in the RPG can be wrong­
fully depicted, mistakenly indicating that certain operations contain rather than consume
certain objects and versa versa. morest is able to add edges in the RPG where schemes
align but are not documented as such. morest is also able to perform edge deletion where
operations fails to execute in a giving call sequence after a certain amount of tries.
With the MBT­tool morest, Liu et. al. manages to return some astounding results. Liu
used morest over 6 open source SUT’s and managed to detect 44 bugs, whereas 13 of
them could not be detected by existing approaches [19].

3.4
Future testing of alternative REST API
This appliance of SeTTs to Gitlab does not only provide a MBT suite for GitLab, it also
entails that this process of creating a testing­suite for a REST API could be done for any
REST API (that the tester has access rights to). The process of applying GitLab and
SeTTs can now generally be described with the following steps:
• Create the example folder that refers to the desired REST API and populate it
with the required classes and directories (util, directories for the auto­generated
classes, DSL files etc.) and make sure that the generation of the test­driver and the
OpenAPi client is included in the .sbt build file.
• Acquire the swagger definition (or create one from the documentation of the REST
API).
• Populate the util class with methods according the desired test­cases.
• Create a wrapper class containing wrapper methods for the desired REST API in
the package wrapperMethods, and populate it with methods covering the desired
HTTP requests.
• Create a DSL file that describes both the API calls and the hierarchy of the API calls
and the response code assertions,
• Rebuild the project (thereby generating the test­driver and the client with sbt clean compile)
and run the Wrapper (Not located in the package wrapperMethods) class that initi­
ates the driver.
In this case a challenges included: Finding the swagger definition, determining bugs in
relation to OpenAPI generator and the development of the wrapperMethods class as well
as the test­drivers access to this file. Giving that future testing already has a clearly
defined swagger definition and that the tester is using this new version of SeTTs with the
support of the wrapperMethods, setup of a new testing suite should be remarkably easier.

4.1
Status­quo traversal
A potential optimization of SeTTs is the traversal of the test­cases. Currently, SeTTs tra­
verses every internal choice at random (making the test generation random), which begs
the question: could the traversal be directed in a meaningful matter?
Looking at figure 2.8, currently the HTTP request­responses of a test­driver that covers the
REST API’s in an acceptable manner, is only guaranteed in the number of repetitions of
the test­driver which does essentially only guarantee desired code coverage in terms of
chance, and is is no guarantee at all.
As an example, for a test driver that is generated from 2.4, and run over 10 repetitions,
there is a possibility (while being very unlikely) that the internal choice made between the
API call, would execute the path faillingRequest path every time. Concluding a test­run
which has covered very little code and is therefor not of much worth. Thus it would be
meaningful to investigate the possibility of an alternative traversal through these API calls
and thereby change the test­generation. The following configuration of the test­driver will
be a conceptual prof­of­concept design, that is implemented in a specific instance of a
generated test­driver. This proof­of­concept is provided to inspire a future iteration of the
SeTTs parser, that would implement this configuration in every test­driver generated.

evenly distributed between the API calls that are available from an internal choice. The

Drawing inspiration from software like ClusterFuzz (and software like Morest and EvoMaster
mentioned at 2.7), code coverage was chosen as the metric for progress when altering the
test­drivers traversal of API calls. ClusterFuzz is a fuzzing infrastructure which is used by
Google to investigate security and stability issues within Google software [22]. Fuzzing,
is in short the process of injecting semi­random input into a program to find faults [23].
ClusterFuzz is able to use code­coverage as a metric while fuzzing for bugs in the soft­
ware. ClusterFuzz mutates the input injected into the program, to grow the code coverage
­ making sure that the fuzzing is not futile in regards to what has already been explored
[23].
In regards to code coverage being used a metric for the test­drivers traversal of API calls,
two types of code coverage is to be defined:
• Coverage ­ noted as %. This is the total code coverage from the entire run of the
test­driver, meaning the summation of the code coverage from every repetition that
the test­driver performs during testing.
• Delta Coverage ­ noted as ∆%. This is the coverage gained from each repetition
performed by the test­driver. ∆% should optimally grow in every repetition in order
to achieve a meaningful traversal of the API calls.

4.4
From petclinic to GitLab
Unfortunately the GitLab­CE server already required a lot of computing power, caused
freezes and halting the development. Switching to the GDK version of the server caused
development to be virtually impossible. The time and computing power required to both
make meaningful API calls on the GDK server already proved to be challenging, while the
required repeated initialization and shutdown of the server would be impossible with the
computing resources available to this project. A virtual VM was requested from DTU, to
smooth out the development process adding additional computing power, but the process
of gaining access to such a VM proved too long for the project at the stage of development.
Therefore a change in the SUT was made. Since SeTTs is a MBT­tool, which automatically
generates a test­driver to any giving SUT if provided with a DSL file, it would be just as
meaningful to proceed with an alternate SUT that requires much less computing power.
Therefore the spring­petclinic (mentioned earlier in 2.5, 2.4, etc.) [4] was chosen. Given
that improved performance is achieved with the spring­petclinic REST API, it follows that
the same alteration would be just as beneficial to every other REST API tested by SeTTs,
since the tool is based on general concepts and not SUT specific functionality.

4.6
Traversal Tools
After the structure of the API calls in the test­driver has been altered, it is possible to ad­
just the way the the test­driver performs the API calls. To do this is a generic and possibly
accessible way for future different SUT’s, a class called TraversalTools is created. The
class consists of methods acting as tools to configure the probabilities of the nodes in
the test­driver in order to manipulate the way that the test­driver traverses them. Since
the choices of which API call to choose in an internal choice, is based solely on a ran­
dom generated number and each nodes matching probability, all that has to be done to
configure the way of traversal is to adjust the probabilities in a meaningful manner.
4.6.1
Traversal algorithm design
Currently there is a tree­like­structure consisting of nodes representing the API calls de­
fined in the DSL file and there is two metrics that can be used to navigate and configure
32
Model­based testing of REST­full applications
==PAGE:38==
the traversal of this tree: % and ∆%. When testing the SUT with a purely random ap­
proach for every internal choice, it would naturally follow that the % gained from a total
run and the ∆% gained from each repetition would be totally random as well. A run re­
sulting in a satisfying % would be the product of randomly generated numbers. But why
should the test­driver traverse a path of nodes that has already yielded no ∆% in a pre­
vious repetition? It must follow that if no additional code coverage was previously gained
from traversing a path, then future traversal of the path, would neither gain any additional
coverage.
The underlying thought for the following algorithms can be summarized as follows:
Did the previous path yield ∆%?
• If yes, then encourage that path in future repetitions.
• If no, then discourage that path in future repetitions.
The algorithm is thus designed to be a greedy algorithm, that evaluates on the previously
traversed path, in which it afterwards is decided how to traverse the same tree again. To
know whether to terminate or not, the algorithm simply has to evaluate if there is any paths
left that is yet to be discouraged. If all possible paths has yielded no ∆%, then there is no
point in further testing since we have covered all the possible % there is to gain from the
current DSL file.
4.6.2
Implementation
The implementation of this algorithm consists in two phases, encouragement and dis­
couragement. Two additional ArrayBuffers are added to the test­driver of type Node:
currentPath[] and pathHistory[]. Then every time an API call is made, the Node in­
stance that makes this call is added to both the currentPath and the pathHistory. Each
time a new repetition starts, currentPath is cleared while a separator node is added to
the pathHistory. This is a special node, with a prob = 999 that makes it easy to separate
each path when logging the traversal at 4.7.1.
The encouragement of the algorithm is made with the method.
1
configure_nodes(path: ArrayBuffer[Node], gainedCov: Double)
This method takes the currentPath and the ∆%. First configure_nodes() checks if the
∆% is greater than 0. If it is not, the method discourages the path taken. If it is, the
method knows to encourage the last traversed path.

concept the test generation technology effectively changes SeTTs from having a Random
generation of test, to having a search­based algorithm for generation the tests [5]. Since
this new traversal relies on a tree­like structure and the coverage of different nodes in this
tree.

5.1
Research Question 1 ­ Application to a complex SUT
The first research question is answered through the process of extending and then ap­
plying SeTTs to larger RESTful application, namely GitLab as a specific example, written
out in chapter 3.1. It is shown in this thesis that it was possible to extend SeTTs to be
applied to a more complex REST API than the example servers that the base version of
SeTTs tested. Multiple modifications were made to SeTTs in order to fully achieve this,
most importantly the introduction of the
GitLabWrapperMethods class and the test­drivers ability to reference this as well as other
classes that can wrap the complex HTTP request methods. This enables the user of SeTTs
to tackle the increasing complexity that can follow applying SeTTs to a SUT of more com­
plex matter, where each API call has multiple modifiers that can be enabled or changed.
The
GitLabWrapperMethods enables the user of SeTTs to correctly encapsulate the desired
functionality of a more complex SUT, without being overwhelmed by the the vast mul­
titude of options that an API call may require. It is noteworthy that this is not only an
extension that applies to GitLab. Classes made within the wrapperMethods package, is
able to achieve the same results with another SUT.
This extension entails that a multitude of different RESTful applications could be tested
by SeTTs. Since the steps to create this extension only in small regards was specific to
the SUT, and could be replicated only with few moderations given that SUT was another
complex open source RESTfull application. A guide for how to construct a test­suite for
an alternative RESTfull application is noted at section 3.4.

Research Question 2 ­ Enhancing traversal of the
test­driver
The second research question is addressed in the proof of concept alteration of the test­
driver, the introduction of the
TraversalTools class and the extraction of code coverage after each iteration of the test­
driver. It is shown throughout section 4, that it is possible to increase the performance of
SeTTs, specifically by reducing the amount of API calls that the test­driver has to perform
throughout a run, using code coverage as a metric for improved traversal of API calls.
This new way of traversal ensures the highest possible amount of code coverage, while
minimizing the API calls required to reach this code coverage. This is very clear from the
logs files shown at section 4.7.1.
Additionally, this proof of concept of alternating the traversal of the API calls, is very gen­
eral.And thus It should be not a bigger challenge to apply these changes to a test­driver
that performs API calls to an alternate SUT. Additionally the alteration of SeTTs changes
the way that the adheres to the MBT taxonomy [5]: namely going from a random test­
generation, to a search­based algorithm test­generation. The SUT chosen for this exact

5.3
Limitations
5.3.1
Recursive DSL files
As hinted in section 2.5, Session types support recursive definitions and so does SeTTs
DSL files. This would mean that when a sequence of API calls end, instead of starting a
new repetition of the test­driver, a new sequence of API is made in concatenation with the
previous. The Altered traversal of the test­driver does not currently support these kind of
DSL files. The choice not to work with recursively defined DSL files was taken in order to
prioritize the immediate improvement of the test­driver. It is assumed in this thesis, that a
sequence of API calls run alike regardless of additional recursive calls. If a tester desires
to test a longer and more specific sequence of API calls (that a recursive definition will
create), these can be formulated in the DSL file. Besides from that, the introduction of
recursive definitions in the DSL files heavily increases the complexity of a search­based
algorithm needed for the same results as those already achieved in this thesis.
5.3.2
Rigidity of the P.O.C. traversal
Currently the proof of concept (P.O.C.) is very rigid receding in only one example related
to the DSL files shown multiple times in figure 2.4, 2.8, 4.3 and 4.2. Since this alteration of
the test­driver is merely a proof of concept, it serves to show a conceptual design on how to
effectively increase the performance of the test­driver. To apply this traversal to alternate
SUT’s, only a few structural changes has to be made to the generation of the test­driver.
The methods in the TraversalTool class are almost purely general and can be applied to
any test­driver using the node trait defined in section 4.4.1. The only non­general methods
defined in the TraversalTool class is the the methods that are related to the extracting
of code coverage, which is somewhat SUT specific. In this specific case regarding the
spring­petclinic, extraction of the code coverage is specific to the code coverage tool
JaCoCo [26]. Besides from that, the P.O.C. is a prototype limited to only one specific
case of a DSL file. This could perhaps result in undiscovered problems being introduced
as the DSL file that the P.O.C. is applied to grows in complexity.
5.4
Future Work
5.4.1
Random generated parameters
With the implementation of the Wrapper packages and the alternate traversal introduced,
an obvious next step would be to add randomness to the methods that invoke the HTTP
request of the test­driver. Articles like the ones concerning the development of EvoMaster
[17] and morest [19], suggest that there is a lot to gain by modifying the parameters of
such HTTP request methods from repetition to repetition. This would exhaust more options
and potentially additional ∆% could be yielded from the same path multiple times. In con­
junction with the code coverage based search traversal introduced, this would suggest
that the prob fields in the nodes traversed would not be modified as discrete as they cur­
rently are. With the introduction of changing inputs to the HTTP requests, it could possibly
follow that even though a path yielded no ∆%, another traversal of the path in question
could be meaningful. Perhaps a ∆% < 0 would result in a prob being halved, rather than
completely set to 0. This would perhaps enable the test­driver to generate test­cases that
gain unexpected amount of additional code coverage.

5.4.2
Alternate path­finding algorithms
The tree­like structure enforced on the test­driver could potentially suggest alternate algo­
rithms for the traversal of the API calls. The P.O.C. implementation is merely one of many
different algorithms that could be applied to a test­driver with this structure. Currently the
alternate traversal could be interpreted as a greedy depth first search algorithm that uses
code coverage as encouragement for taken a specific path, while there is a wide array
of alternate tree­traversal algorithms. A note for further development of tree­traversal al­
gorithms is that they should support the inherent hierarchy that the DSL files suggests.
Breath­first­search and alike might experience challenges since API calls would have to
be invoked in parallel.

Conclusion
6.1
Conclusion
This thesis has successfully achieved the research questions defined in section 1, and
thus the MBT­tool SeTTs was extended to be able to handle more complex RESTful ap­
plications than the example servers that the base version of SeTTs supported. This thesis
successfully achieved this with the GitLab­CE RESTful application, being able to invoke
HTTP requests that SeTTs could use as assertions for testing. The expansion covers the
implementation of the GitLabWrapperMethods class, which provides methods for wrap­
ping complex HTTP request methods and thus only capturing the desired functionality for
testing. By including the import of the package wrapperMethods in the SynthDriver class,
additional wrapper classes are able to be made for alternate SUT’s. Besides from that,
a guide on how to apply this MBT­tool to a general RESTful application after this im­
plementation was produced. Additionally this extension to SeTTs alters the way that the
test­driver traverses the test cases, completely redefining its approach for test­generation.
This is achieved by enforcing a tree­structure in the test­driver using the trait node. The
implementation of the class TraversolTools provides methods for a search­based test­
generation as an alternative to the random test­generation that SeTTs inherently used.
This search­based algorithm uses code coverage of the SUT to determine how to traverse
the API calls of the test­driver. This new way of traversing the API calls of the test­driver
proved to achieve a substantial reduction in overall API calls, while maintaining a high
code coverage. The results indicated a reduction of the API calls made to achieve high
code coverage of 33%. Even though this reduction is achieved on spring­petclinic it is
clear that the approach to achieve this was general and therefore likewise results could
be expected on more complex RESTful applications like GitLab.
