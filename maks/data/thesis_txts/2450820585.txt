Abstract
Relative compression is a widely used technique for data compression.
It is
particularly eﬀective when dealing with highly repetitive data, because it avoids
saving a lot of duplicate information. We present a structure that, combining
relative compression and interpolation search for predecessor queries, achieves
remarkable random access time and uses little space. Our theoretical contribution
is an extension of an existing structure for static interpolation search predecessor
to the dynamic case, allowing editing operations. This structure still maintains
the same asymptotic bounds as its static counterpart, except for amortization.

Introduction
Compression is the process of reducing the amount of data needed for the storage
or transmission of a given piece of information.
Data compression can be lossy or lossless. While lossless compression allows us
to recover completely the original data, lossy compression will discard redundant
information or decrease detail. Tipically, lossless compression is related to text,
while lossy compression methods are used for images or sounds, like JPEG or
MP3 formats. [8]
A compression technique should be chosen taking into account various factors:
the degree of compression achieved, the computational resources (both time and
space) required to compress and decompress the data, and also how easy it is to
access or modify the data while compressed.
In this thesis we focus on lossless string compression, in particular on relative
compression of highly repetitive data. One of the main ﬁelds of study that would
beneﬁt from improvements on this technique is genomics, as we will see in Chap-
ter 3.
Regarding the compression process itself, we present a method which requires
O(n) time and space. Yet, we put our eﬀort in designing a compression that
will allow fast random access to the compressed data without the need of decom-
pressing it.As it is presented on Chapter 4, we achieve a compression around 16
times smaller than the original source, still allowing very fast random access.
In chapter 2 we give a deep theoretical description of a useful method to achieve
our goal, that is extended to allow also dynamic changes on the compression.

1.1
Compression of Strings
Given a source string S, a compression C of this string is a structure that needs
less space than S to be stored but still contains all its information. That means
that one could decompress C and obtain a string that is exactly equal to S.
The main disadvantage of working with compressed objects is that accessing the
information inside is slower than in the original string. That is why there are
many diﬀerent approaches to string compression, and they mostly vary around
this space-speed trade-oﬀ. The smaller the compression, the more time it might
take do queries on it.
The most common and straight-forward approach is to compress a string based
on its own redundancy as an individual. For repetitive collections, however, it
seems to be more eﬀective to use Relative Compression.
1.1.1 Deﬁnition (Relative Compression) Given a static reference string R
and a source string S, a relative compression of S with respect to R is an encoding
of S as a sequence of references to substrings of R (called phrases).
Within relative compression, there are still diﬀerent methods on how to store
the compressed strings. One particularly interesting way of doing it is due to
Lempel-Ziv [12]. In the case of Lempel-Ziv, the references to substrings of R are
tuples (si, ci), where si is the absolute index in R and ci is the cumulative length
of the matching substrings; such that, if there are n phrases, cn = |S|. Table 1.1
shows an example of this compression technique.

Having deﬁned how to compress a string, we now explain how to perform queries
on them.
In this thesis, we particularly focus on random access queries.
A
random access query of a string basically asks for the character the string S
contains in a certain index i. This can obviously be done in constant time if the
whole string is stored, but it is not that trivial to do when only the compression
of S is stored. The following section describes how to perform a random access
query to a string compressed using Relative Lempel-Ziv (RLZ).

1.2
Random Access Query for RLZ
Given a source string S, a reference string R and the compression C of S with
respect of R, the aim is to answer queries that ask for a particular character S[i]
using only C, R and i. There are three main steps into obtaining S[i]. First,
ﬁnd which phrase of the compression contains i. Second, compute the index in R
that contains the same character as S[i]. Finally, access and return the character.
Suppose C = (si, c1), (s2, c2), . . . , (sn, cn).
The phrase that contains i can be
accessed performing a so called predecessor query (explained in the following sec-
tion) in the array of cumulative lengths Σ = (c1, c2, . . . , cn). This query returns
the basically the largest number among Σ which is not larger than i. Say this
number is cj, it means that the jth phrase covers some indices of S up to the
cjth index (it covers exactly cj −cj−1 indices). Therefore, phrase j + 1 is going
to be the one covering i and the relative index of i in this sentence is r := i −cj.
The phrase (sj+1, cj+1) is the matching substring of R that starts in sj+1 and
has length cj+1; so the character to return is R[sj+1+r], which is the same as S[i].
In all the process, only a ﬁxed number of constant operations are performed
except for the predecessor query. Namely, the performance of a random access
query will be determined by the speed at which the predecessor query can be
executed. As mentioned above, this is the topic we address in the next section.
1.3
The Predecessor Problem
The predecessor problem consists of maintaining a set of integers U = {u1, u2, . . . , un}
in a data structure that allows the following operations:
predecessorU(x) = maxi{ui ∈U | ui ≤x}. It returns the largest number
in U which is smaller than or equal to x.
successorU(x) = mini{ui ∈U | ui ≥x}. It returns the smallest number in
U which is greater than or equal to x.
In addition, to solve the dynamic version of the predecessor problem, the structure
has to support:
insertU(x): Add the number x to the set U.
deleteU(x): Delete the number x from the set U.

Introduction
The predecessor problem has been studied in depth in computer science and al-
gorithm theory, because of its numerous applications. In the particular case of
this thesis, the predecessor is used in the random access queries of compressed
sequences as mentioned in chapter 3.
It is used in the Relative Lempel-Ziv
compression to know in which compressed sentence does a speciﬁc index of the
original string lie in. Therefore, from now on we focus on the predecessor query
rather than the successor. However, in all the presented cases and structures the
successor query works analogously to the predecessor.
1.3.1
Notorious Methods
The predecessor problem can easily be solved using O(n) space by just using
binary search, taking O(log n) time query. This time bound has been improved
by more complicated structures. Van Emde Boas [18] proposed a tree structure
that can solve the problem in O(log log N) time and O(N) space (where N is
the maximum value among the n values, so it represents the universe that the
numbers are in). Later on, Dan Willard proposed a structure, Y-fast Tries, tak-
ing O(n) space and using O(log log N) time for the predecessor problem [6]. The
main problem with these approaches is that although the theoretical bounds are
promising, its implementation is extremely complex.
A fairly simple structure that achieves good time bounds for well-behaved data
is due to M. Patraşcu et al. [7]. Their method is relevant for this thesis since
our aim is to combine it with relative compression. This structure is explained
thoroughly in next Chapter.
1.4

Kuruppu et al. [22] and Ferrada et al. [11] use structures that support the pre-
decessor query for the purpose that is relevant for us in this thesis: accessing
indices in relative compression. The ﬁrst [22] reaches a time bound of O(log N
n )

2.2.2
Predecessor Query Solutions
When a query asks for the predecessor of a number x, there are two diﬀerent cases
that can happen depending on the position of x with respect to its predecessor.
1) x and its predecessor (let us call it p) lie in the same bin. This happens
when the bin that covers x contains at least one number smaller or equal
to x.
2) x and p lie in diﬀerent bins. This can happen if x is covered by an empty
bin, which means that the predecessor will be in a previous bin. It can
also happen if x is smaller than all the numbers contained in the bin that
covers it. Note the diﬀerence between contain and cover; we mean that x
is smaller than all the numbers contained in the bin, which does not mean
that it is smaller than all the numbers covered by the bin.
The ﬁrst obvious attempt to solve this problem dynamically is to replicate the
way predecessor queries were dealt with in the static structure.
That is, by
adding the nearest neighbor from below in the tree of each bin. The main prob-
lem with this is that an edit operation could cause many bins to change their
nearest neighbor. Take the case where there are several consecutive empty bins;
they all have the same nearest neighbor corresponding to the maximum value in
the nearest non-empty bin to the left. If, for example, the maximum value of
this non-empty bin was deleted; then, all these consecutive empty bins should be
updated. That could potentially lead to large time bounds.
We bound the maximum number of consecutive empty bins between the maxi-
mum and minimum integers in the structure (we call it E) and give the bounds
with respect to it. We have the following result.
2.2.1 Proposition The maximum number of consecutive empty bins E cannot
be larger than the maximum gap ratio ∆: E ≤∆.

2.3
Maintaining ∆Dynamically
Patraşcu et al. explained in their paper [7] that the whole dynamic structure
needs to be rebuilt when ∆> ˆ∆. However, there is no mention on how to keep
track of the current ∆after each editing operation. In this section we try to
come up with structures that support the calculation of ∆dynamically without
altering the structure bounds.
Recall that we are maintaining an ordered set of integers in which we deﬁne ∆as
the maximum gap ratio (Deﬁnition 2.1.1). The structure uses space linear with
the amount of numbers: O(n); and time O(log ˆ∆) for searches and predecessor
(or successor) queries. The time bounds for insertions and deletions are the same
but amortized. This might seem like an easy problem to solve but it is deﬁnitely
not a trivial one.
The reason why ∆cannot be maintained in constant time using only O(n) space
is that when inserting or deleting a number from the set, one might be removing
the maximum or minimum gap; and the new maximum or minimum can be any-
where in the array, not only in the gaps created by the ongoing editing operation.
Naively, one could compute the ∆of a set of integers by walking and storing the
maximum and minimum gaps along the way; in the end, the maximum gap ratio
is obtained by dividing these two values. This process takes time O(n) and does
not fulﬁll the requirements.
In this situation we decided to explore more complex structures to deal with the
problem. In the following we present the evolution of our ideas that led to a
working solution.

3.1
Relative Lempel-Ziv
We want to do a Relative Lempel-Ziv implementation that supports compres-
sion, decompression and random access. The decompression of a source string S
with |S| = N is easily done in O(N) time by checking each encoded sentence in
order and looking into the reference string. So we now focus on explaining the
compression and the random access.
To compress a source string S we need an eﬃcient way of ﬁnding matching
substrings in R, this can be done with a suﬃx tree.
3.1.1 Deﬁnition (Suﬃx Tree) A suﬃx tree of a string R is a tree where all
leaves are associated to suﬃxes of the string R and the path from the root to the
leaves gives the exact characters in the suﬃx (see Figure 3.1). In a suﬃx tree, the
edges contain the starting and ending indices of the substrings they represent. It
uses only linear space on the length of the string.
With the suﬃx tree on top of the reference string R, a source string S of size N
can be parsed and compressed with reference to R in O(N) time. The steps for
the compression are as follows.
==PAGE:44==
32
3.
Implementation
First we deﬁne the index for the sentences j (start at 1) and the index in S while
we parse it, i (start at 1). For simplicity and convenience, we add a ﬁrst sentence
(0, 0), which corresponds to j = 0. Then for each j:
1) Look into the ith character of S, Si.
2) From the root of the suﬃx tree (built on top of R), select the edge which
starts with Si,
3) Go down on the suﬃx tree by selecting the edges that match the maximum
substring of S starting at Si.
4) When step 3 stops, look at the length l of the matching substring, deﬁne
cj = cj−1 + l and add the sentence (i, cj) to the compression. cj is the
cumulative length up to the current sentence.
5) Then set i ←i + l and j ←j + 1.
6) Go back to step 1 unless the total length of S has already been covered
(i = N).
For the implementation of the suﬃx tree, we are using Ukkonen’s algorithm [25].
This algorithm builds the suﬃx tree in linear time for the reference string (in-
stead of cubic time that would take to build it greedily). Ukkonen’s algorithm
uses smart techniques to speed up the construction and avoid performing dupli-
cate steps. For example, the ending indices of the edges that touch a leaf are
updated all in just one step using a pointer; because all of these edges will always
have the same ending index. It also uses pointers named suﬃx links to avoid un-
necessarily long walks up the tree. More details on this algorithm can be found
in the implementation or in the detailed explanation by Dan Gusﬁeld [10].

All the data used throughout this study are DNA genome sequences. The DNA is
the molecule that carries the genetic instructions of all organisms. This molecule
is always composed by a sequence of nucleotides, and it is the occurrence and
order of these nucleotides what determines the information carried on the DNA.
The chromosomes are the high-level structures in which the DNA is organized
inside a cell. The majority of the living beings have almost all their genetic infor-
mation divided in diﬀerent chromosomes inside their cells (along with some piece
of DNA contained in smaller organelles like mitochondrion). We call genome to
the totality of the genetic material of a species or organism in particular, that
is, to the set of chromosomes in which the DNA is divided inside a cell. Not all

vary as well (from 200 genes in some bacteria, to around 20000 in humans or

Genomics is the ﬁeld of study that tries to understand the structure and func-
tion of the genome, and so its main source of knowledge is DNA sequences. The
process of determining that occurrence of order of nucleotides in a DNA chain is
called DNA sequencing; we could say that is the process of translating the bio-
logical information into a simple string. As the nucleotides present in the DNA
are cytosine [C], guanine [G], adenine [A] or thymine [T], we can represent
all the information just by a chain of consecutive letters A, C, G or T.
Sometimes when a genome is codiﬁed, some part of it cannot be fully sequenced,
and it is common to use an N letter to represent a piece of genome that has
not been deciphered.
It is possible to ﬁnd other characters in the data that
give more information than just an "unkwown" nucleotide, but for the sake of
our study we will assume that in all of these cases, the character used in the

which is necessary in the construction of suﬃx trees.
The high-repetitiveness of our data seems evident, given that even a simple yeast
genome, such as Saccharomyces Cerevisiae contains 12 million nucleotides, that
is, a string of length 12·106 formed with only 4 diﬀerent characters. Nevertheless,
two random pieces of DNA can have little in common. For example, when trying
to compress one chromosome of S. Cerevisiae versus a diﬀerent one, the longest
phrases in the compression have length 6 or 7, which is surprising given the fact
that each chromosome contains several thousand characters. Why is that?
Let us analyze the problem from a combinatorics point of view. Given that we
have a set of 4 diﬀerent elements (A, C, G, T), we can build 46 = 4096 diﬀerent
chains of length 6. It is then plausible that in a chromosome of 12 million nu-
cleotides, we can ﬁnd at least once all possible combinations, so we could codify
any genome against it in phrases of length 6. But following the same argument,
we can build 412 = 16, 777, 216 diﬀerent chains of length 12 from a set of 4 char-
acters. It is clear now that a chromosome with 12 million nucleotides will never
contain all of them, not even close. And it is worth remember that a Lempel-Ziv
encoding is parsing left to right, not ﬁnding the longest common substring in
reference and source, but the longest common substring from a particular index.

3.5.1
Tightness of the ∆Bound
As mentioned above, the time used in a random access query is spent mostly in
doing a predecessor search. Our implementation performs predecessor queries in

Once the structure with bins explained in Section 3.2 is built, the worst case
takes place when a query asks for an index covered by the fullest bin.
This
means that the bin which contains more numbers determines the worst running
time of the access query. By [7] we know the the amount of numbers contained
in a bin cannot be larger than ∆(which leads to the bound O(log ∆) for the
query time). However, we have seen that in the case of our data the largest
number of elements in a bin is rarely ever close to ∆: the bound is not tight.
This is mainly due to the fact that diﬀerent genomes have unknown bases in
diﬀerent places, often represented with several consecutive Ns; this leads to short
sentences in the compression which at the same time lead to too large values of ∆.
In Figures 3.5 and 3.6 one can see how the value of ∆aﬀects the average query
time and the worst-case query time.

The object containing the bin structure on top of the cumulative lengths.

length b and we need to save an extra integer to store the number of bins (hence
the "+8"). For instance, reducing the number of bins to the half, the space usage
would become 13n+8 which is a signiﬁcant change. And the amount of numbers
in the bins would not increase enough to have a relevant eﬀect on the query time.

have only one number (because they would also be theoretically empty), not af-

Note that this parameter could also be used to increase the number of bins.
However, the query time would most likely not decrease much in this case, and
deﬁnitely not enough to compensate the extra space.
3.6
Further Optimization
At this point we revised our own implementation to see if we could do any ad-
ditional optimization. We focused on improving the space usage rather than the
query time because that is what results in Chapter 4 suggested.
We found out that the bin structure suggested in Figure 3.3 could be compressed
more by not saving the number of elements inside a bin (the bin length). Since
the bins are ordered and consecutive, the length of a bin can be obtained using
the starting positions of the same bin and the next one. Take for example two
consecutive bins (si, li) and (si+i, li+1) with their respective start and length
values; the length value li of bin i can be obtained by:
li = si+1 −si + 1.
We then changed all the implementation to use this calculation instead of saving
the lengths of the bins. This adds only one extra addition and subtraction to the
query and it saves a space of 4 bytes per bin, which is very signiﬁcant.
On top of that, the fact that each bin uses the following bin to compute the length
might be a problem when dealing with the last bin. In this case, the length of
the bin would be the total number of elements n minus the start of the last bin.
However, the length of the last bin is never used. That is because the predeces-
sor is applied to the vector of cumulative lengths of phrases in the compression;
and the last phrase’s length is the total length of the source. Therefore, since
==PAGE:60==
48
3.
Implementation
indexes for random access will always be smaller than the length of the source,
the predecessor call will never return the last phrase; it will at most return the
second to last.
With this improvement, the space usage of our implementation is reduced to
9n + 4b + 8 bytes, where n is the number of phrases and b is the number of bins.
The second and last version of our implementation is the one that includes the
variable number of bins and does not save the bin lengths.

4.1
The Bitvector Implementation
The idea of the bitvector implementation is to save a bitvector with length equal
to the source length that supports constant time rank and select queries. This
way, the authors claim to achieve constaint time random access. The detailed
theory on how to achieve this can be found in Constant-Time Random Access[11].
We would expect our implementation to take less space for compression and use
more time in the random access queries. However, the authors did not implement
their own bitvector structures and instead used an external library (SDSL - Suc-
cinct Data Structure Library) [21]. This library uses a structure for the bitvectors
that takes only logarithmic space on the length of the bitvector. And they defend
that while the theoretical bounds for rank and select using logarithmic space are
not constant, they are extremely close on practice. Hence, this solution will most
likely use less space than the Interpolation Search RLZ and we therefore expect
our random access query time to be faster.

Conclusions
We have extended the static solution in [7] to be a dynamic structure that al-
lows for predecessor and successor queries. The structure is a simple and natural
extension of the static case and it needs an extra structure that we propose to
maintain ∆dynamically. This structure is a counter of the gap sizes grouping
them in buckets of exponentially increasing width.
The proposed solution is able to maintain an approximation of ∆dynamically.
We proof that this is enough to obtain the same time and space bounds as the
static version, save for amortization. We leave as an open problem if a dynamic
structure can maintain the exact value of ∆with the same asymptotic bounds.
We provide an implementation of relative Lempel-Ziv with interpolation search
for predecessor that is very simple and achieves remarkably good results. The
random access query time is more than 5 times faster than the benchmark used
for comparison, and the code structure is extremely simpler.
The space usage of our algorithm is around twice as large as the one used by the
benchmark. However, except for cases that are particularly space sensitive, our
solution provides a great improvement. A better implementation of a suﬃx tree
(which has been out of the scope of this project) would also improve our cur-
rent solution greatly; mainly by increasing the size of the data it is able to handle.
We have proved that our implementation is not dependent on the original size of
the source or the number of compression phrases. Rather, it is mainly dependent
on the number of elements in the bins, which has been shown to be quite small
thanks to the notably uniform distribution of the compression phrases. It is also
relevant to mention that the worst-case query time is almost only aﬀected by the
number of elements in the fullest bin. While not being theoretically constant,
the access queries of our solution remain virtually unchanged across very diﬀerent
sizes of sources and numbers of phrases.

5.
Conclusions
==PAGE:79==
6
Future Work
We are convinced that the use of relative Lempel-Ziv for compression, and in-
terpolation search for the predecessor queries oﬀer a competitive implementation
for data compression. Still, there is always room for improvement.
Regarding the random access query time, an interesting suggestion for future
work would be to add a new parameter max_num_bin to the implementation.
This parameter would limit the number of elements in the bins such that they
never have more than max_num_bin. However, there is a main thing to con-
sider here. All the bins have to be of the same size (cover an interval of the
same length) for the interpolation search to work. Therefore, this modiﬁcation
would potentially lead to very large number of bins. Because a very close group
of number could make the bins in the whole interval be unnecessarily small. It is
hard to say if query time reduction would be worth the extra space consumption,
so it is not clear if the change would suppose and improvement. Still, it would
be relevant to investigate it further.
The current compression size also has potential for improvement. One immedi-
ate change that can be done is a change in the way of storing char variables on
the compression (the mismatches characters). Previous implementations, as the
RLZ implementation referenced through our study, are using the fact that the
DNA alphabet is composed only by the characters (A,C,G,T,N,$). That is an
alphabet of cardinal 6 instead of the 256 possible characters of the ASCII code.
Then, while a char is typically stored in 1 byte of memory, the DNA alphabet
characters could be stored in 3-bit elements. Some implementations even manage
to store them in only 2 bits [4], by using only A,C,T,G, changing all the N for A
and storing them in a look-up table. An equivalent trick could be used in the way
integers are stored in the compression. If the maximum number is signiﬁcantly
smaller than the INT_MAX, then the arrays of integers could also be stored in
a lower number of bits.
==PAGE:80==
68
6.
Future Work
Another area that could be tackled, although is has not been a focus of this the-
sis, is the suﬃx tree implementation. We have seen that the Ukkonen’s suﬃx tree
that we have used needs a larger amount of memory during construction than
we expected. Given that the time to build the suﬃx tree is not crucial for our
project, because our focus is on compression size and access time (none of which
depend on the suﬃx tree construction time), a future version could implement
a better version of the suﬃx tree. Better in the sense that the memory used
is less even though the construction time might increase. This would allow our
implementation to be used for larger data.
Apart from modiﬁcations in the implementation, a relevant path of future work
would be to test and benchmark our implementation further. Not only with other
computers where we have more control of their current status and memory usage.
But also using more and larger data and comparing it to other implementations.
We deeply thank the authors [4] for providing us with their implementation and
it would be very remarkable to compare our algorithm to other state-of-the-art
implementations.
From the theoretical component of this project, we believe that the most inter-
esting continuation of our work is to ﬁnd a structure to maintain the exact value
of ∆dynamically. We have provided a structure that is capable of calculating a
close approximation, and that is enough to achieve the desired bounds. However,
a structure able to maintain ∆dynamically while maintaining the bounds would
be more understandable and was what we were trying to come up with in the
ﬁrst place.

A.1

A.1.1
Problem Description
The data used in many real-life problems is too big to deal with in real time.
To tackle this, some solutions use compressed versions of the data which oﬀer
the possibility to access (query) it without the need to decompress. In particu-
lar, compression seems to be very useful when dealing with highly repetitive data.
A widely used compression technique that supports fast random access from the
compressed data is Relative Compression. Given a static reference string R and a
source string S, a relative compression of S with respect to R is an encoding of S
as a sequence of references to substrings of R. The algorithm to access a random
index from S consists on ﬁnding the substring of R that contains it and looking
it up on R. This process involves a predecessor query and a ﬁxed amount of
constant time look-ups. Therefore, solving fast random access queries is reduced
to solving the predecessor problem eﬃciently.
The predecessor problem is an extensively studied problem in algorithm theory.
Given a set of integers U, the predecessor of a given value x is the largest number
in U smaller or equal than x; i.e. predecessorU(x) = maxi{ui ∈U|ui ≤x}.
Besides solving the predecessor problem, one can obtain advantages in practice by
implementing modiﬁcations that use properties of the particular dataset at hand.
Usually, the impact of this modiﬁcations cannot be reﬂected in the theoretical
bounds, but they show their worth in practical tests.
==PAGE:86==
74
A.
Evaluation
A.1.2
Previous Work
The relative compression of highly repetitive data has already been researched
in several publications. In this section we call S the source string with |S| = N,
R the reference string with |R| = r and we let n be the amount of substrings in
the relative compression of S with respect to R.
Theoretically, one could obtain a structure taking O(n) space and using O(log log N)
time, using Y-Fast Tries for the predecessor problem [6]. However its implemen-
tation is extremely complex. S. Kuruppu, S. Puglisi and J. Zobel presented the
"RLZ" structure, a combination of self-index and LZ77 which allows a really ef-
ﬁcient compression, and fast-random access using little memory [22].
A result of particular interest for our thesis regarding predecessor problem is
given by E. Demaine, T. Mihai and M. Pătraşcu, who give a data structure which
provides a bound of O(log ∆) time per operation (insertion, deletion, search and
predecessor/successor) for the static set of integers, where ∆is the ratio between
the maximum and minimum distance between elements in the integer set. Also a
variation of the data structure is provided that supports searches, insertions and
deletions with the same space and time bounds.
A.1.3
Objective
In this thesis we aim to use the predecessor problem solution of [7] to implement
a relative compression algorithm for highly repetitive data. [7] obtains a bound
of O(log ∆) for predecessor queries where ∆is the ratio between the maximum
and minimum distance between elements in the integer set.
This thesis attempts to (i) theoretically extend the solution in [7] to the dynamic
case while allowing predecessor/successor queries in the same bounds as the static
case, and (ii) implement a relative compression algorithm using the O(log ∆)
bound. In the latter, we will try to come up with smart techniques that take
advantage of the current bound, including, for example, clever choices of the
reference string R in order to control the size of ∆.
