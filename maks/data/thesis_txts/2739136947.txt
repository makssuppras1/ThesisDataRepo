Abstract
Process mining is a growing field that sits at the intersection of data science and process
analysis, enabling organizations to discover, monitor, and improve their processes. Execut-
ing process mining projects may require using multiple data resources and/or algorithms
that may be owned by different organizations and that should be executed sequentially
in what we call process mining pipelines. Currently, there is a lack of tools that allow to
share resources and to design and execute such pipelines.
This thesis proposes a distributed architecture titled DAPM (Distributed Architecture
for Process Mining) to address these challenges and enable collaboration between process
mining researchers and process analysts across different organizations. DAPM provides
users with capabilities to share data and algorithms and to design and execute process
mining pipelines in a decentralized manner using shared resources.
The proposed architecture is based on a peer-to-peer architecture where each organization
becomes a peer. Peers collaborate and share resources based on established agreements.
Pipeline executions are orchestrated internally or across peers in an asynchronous manner,
with individual algorithm executions isolated in Docker containers. The proposed archi-
tecture is implemented to demonstrate the feasibility of the solutions capability to enable
the design and execution of cross-organizational process mining pipelines.

Introduction
Process mining is a research discipline that sits between computational intelligence and
data mining on one hand, and process modeling and analysis on the other hand. Process
mining is used to discover, monitor, and improve real processes, meaning processes that
are not assumed but observed [1]. This is done by extracting process models, and insights
from event logs readily available in the organization’s systems. Event logs are records of
events that have occurred during numerous executions of a business process. These events
can refer to activities performed by people or systems [1].
The analysis and understanding of process models is critical as it allows for process opti-
mization and the resolution of potential bottlenecks in the business process. This is why,
in a world where eﬀiciency and productivity are the primary goals for organizations, this
field is experiencing significant growth [2].
As in many other disciplines, getting the best results in process mining requires the usage
of different tools, algorithms, or data sources. These may not all be available in the same
organization, but scattered between various organizations. Additionally, in some scenar-
ios, the available process mining tools lack certain functionality needed [3]. This translates
into the design and execution of process mining pipelines. In software engineering, we can
define a pipeline as a chain of processing elements arranged so that the output of each
element is the input of the next.
In process mining, these pipelines are formed using
different kinds of operators such as miners or conformance-checking algorithms, each one
providing new insights into the process model.
Even though pipelines are widely used in many other branches of data science, it has not
caught on in the process mining sphere, where it still requires a lot of tedious and manual
programming work [4]. For process analysts, this means that designing and executing pro-
cess mining pipelines is a tiresome and ineﬀicient task due to the lack of automation. On
top of that, many elements used to build pipelines, such as data sources or algorithms, are
often located outside of the boundaries of the analyst’s organizations. This adds an extra
level of complexity, as process mining analysts have to deal with privacy and copyright
requirements for sharing data sources and algorithms.
Having discussed all of these problems, it is clear that a solution is needed to help process
mining analysts design, deploy, and execute process mining pipelines across organizations.
In this report, we investigate the state of the art, looking at existing proposals for solving
the issues and problems that arise when setting up pipelines for process mining across
different organizations. Based on this investigation, we propose an implementation of a
distributed architecture that enables sharing of resources such as event logs or process
mining algorithms, and defining, deploying, and executing process mining pipelines across
different organizations.
This project is part of a larger collaboration that consists of three master’s projects where
each has a specific focus area. Our project is focused on the design and implementation
of the architecture of the software solution, while another project, run by Mads Dyrved

In this chapter, we take a deeper look into the problem we are addressing to better
understand the requirements it imposes. To do this, we define a case study of a real-world
scenario that helps us to understand the problem and its related challenges but also helps
us through the development and understanding of the solution. To validate the problem
and evaluate its relevance, we make use of the work of A.Burattin et al. in their paper

Algorithms [5]. In their paper, they also write about the problematics and the lack of a
solution for building and executing distributed process mining pipelines. Moreover, they
define a set of scenarios and validate them with a set of process mining researchers and
analysts.
2.1
Case Study and Associated Challenges
A case study can be used to illustrate and better understand the problem and the proposed
solution. A case study is a detailed examination of a real-world case within a subject. We
use the following case study not only in this chapter but throughout the report to help
the reader better understand the solution. The case study is formulated as the following:
A professor named Bob at Danmarks Tekniske Universitet (DTU) is conducting
process mining research and is aware that a professor at Southern Denmark Univer-
sity (SDU) named Alice has developed a revolutionary implementation for a process
mining algorithm that Bob wants to use. Bob has contacted Alice and Alice has
agreed to let Bob use the algorithm but without allowing him access the source code.
Besides that, Bob has also been made aware that Aalborg University has a
very rich dataset that Bob finds interesting for his research.
The univer-
sity has also agreed to let Bob use the data only for the algorithm execution but
does not want to hand it to Bob as it would compromise some privacy requirements.
Finally, Bob wants to combine all the resources provided by Alice and Aalborg
University with more algorithms of his own to create a process mining pipeline, but
he does not have a tool to easily design and execute such pipelines.
The case study defines a situation in which multiple parties belonging to different orga-
nizations which are geographically distributed, want to collaborate by sharing algorithm
implementations and data resources. Moreover, the case study also indicates that a party
(Bob) wants to use the shared algorithms in combination with some of his own to create
a chain of data processing elements, which we call a process mining pipeline.
This case study highlights two main challenges, which can be decomposed:
1. Sharing Data Resources and Algorithm Implementations
(a) How and where to store these objects while providing a high degree of avail-
ability and respecting the defined privacy requirements.
(b) How to make these objects available for sharing with other parties upon a
previous agreement.

In this chapter, we investigate the state-of-the-art of process mining pipelines to better
understand the problem domain. This includes a study on the process mining field and a
literature review, that can shed light on the current state and problems. Following this,
we take a look at PDL, a prototype for sharing and validating process mining algorithm

3.1
Study on Process Mining Challenges
We begin our investigation of the state-of-the-art by looking into a study on the chal-
lenges process mining analysts face, conducted by Zimmermann et al. [3]. This study is
relevant for us since its objective is to reveal the challenges perceived by process analysts
in their practice. The study is based on interviews conducted with 41 analysts and a
questionnaire survey with 24 respondents used to validate the findings. Getting insight
into the challenges perceived by process analysts is relevant for this project, as they are
key end-users and important stakeholders for any process mining solution developed. The
study identified 23 distinct challenges across various phases of process mining projects.
One of the challenges identified is the accessibility and usability of process mining tools.
Some analysts reported that in certain cases the required tools were not implemented in
the organization. The reason for the suitable tools being unavailable is not explained;
however, we can speculate that this may be due to either financial reasons or that the
tools were not taken into consideration during project planning. Other analysts also re-
port that they have experienced not knowing how to work with a specific tool, and others
claimed that the tools have poor usability. This challenge highlights the importance of
the availability of suitable tools in process mining projects.
Another challenge uncovered by the study is the diﬀiculty of extracting the data required
for the project. Analysts reported that many source systems do not offer out-of-the-box
options to extract data and that some systems impose a daily limit for extracting data,
slowing down the progress of the project. This challenge indicates that there is a need
for a system with the functionality of defining policies for management and sharing of data.
The study also uncovered other data-related challenges, such as data access and source
system knowledge. The analysts reported that in some scenarios they experienced a lot
of diﬀiculty with gaining access to event data due to either the General Data Protection
Regulation (GDPR) or the absence of application programming interfaces (APIs). This
prevents direct access to the data, forcing the analysts to manually access the source sys-
tem to obtain the data. This leads us to the next challenge, which is analysts missing
knowledge about the source system. Analysts reported that they often experience that
the source system they need to work with has no data catalog or documentation. This
means that instead of immediately starting working with the data, analysts must learn
to navigate the source system, understand its functionalities, and how to use them before
they can obtain the data and begin their analysis.

3.2
Systematic Literature Review
We continue our investigation of the state-of-the-art by examining a systematic literature
review conducted by Mads Dyrved Møller, an MSc student at DTU Compute [6]. Møller
searched for relevant literature and examined it to explore terminology, definitions, and
approaches for sharing data and algorithms across organizations in the context of process
mining.
The literature review reveals that process mining pipelines are not as advanced or widely
adopted as in other data science domains. This lag can be partly attributed to the complex-
ity of process mining and the fact that the domain is still relatively young. Nevertheless,
there is a growing demand for pipeline solutions due to the extensive manual work and
tedious programming involved in process mining projects. At the current state, pipelines
still require hands-on programming and manual work, which makes it harder to ensure
repeatability and provenance. While some tools have been developed with the purpose of
facilitating the build and execution of pipelines for process mining, like RapidProm which
is an extension to RapidMiner [7], these tools lack the functionality of setting up pipelines
across organizations, which is crucial for many process mining projects. RapidProm, for
instance, only works locally on a computer, limiting its ability to facilitate collaboration
between organizations [6].
The review reveals that the research conducted in this field is primarily focused on task
scheduling, data streaming, and interaction between different organizations. A significant
challenge highlighted is the sharing of sensitive data across organizational boundaries [6].
This is relevant since there is increasing attention on personal data regulations, and more
often, data recorded and stored contains sensitive information and therefore has security
constraints. This, as mentioned in the previous section, makes it diﬀicult for process an-
alysts to conduct their analysis and delays the process mining projects. This underlines
the need for a solution that facilitates organizational collaboration while addressing the
challenge of sensitive data crossing organizational boundaries.
Overall, the review highlights that while some progress is being made, there still ex-
ist significant challenges in developing eﬀicient, collaborative, and secure process mining
pipelines across organizations.
3.3
Process Dry Lab
Process Dry Lab is a distributed software platform used for sharing and validating process
mining algorithm implementations. It was built during a master’s thesis project conducted
by Sebastian Vestrup and Nicholas Dyhre at Danmarks Tekniske Universitet [8].
The objective of this platform is to provide researchers and organizations a tool to share
and validate newly implemented process mining algorithms, without sharing their source
code. Because designing and deploying process mining pipelines was not a part of PDL
requirements, it is not supported. PDL only supports the sharing and execution of min-
ers from external organizations for the purpose of validation. Even if it does not allow
for pipeline execution, it is still an important prototype for us as it proves that building
a distributed software architecture for sharing algorithms respecting source code privacy
requirements is possible.
PDL follows a peer-to-peer (P2P) architectural style, in which organizations work as peers

to provide each other with storage capacity and computing power. The P2P architecture
makes PDL a distributed system. We can define a distributed system as a system in which
its components are located on different computers, these computers then communicate and
coordinate their actions to provide the intended full functionality of the software. The
P2P architectural style also provides the advantage of not needing a central authority to
take care of providing the service, as each peer is responsible for collaborating and con-
tributing with its computing power. This is very important as the PDL is intended to be
decentralized and to be used freely between any interested party.
Each peer deploys software that is structured following the microservices architectural
style in contrast with the traditional monolithic architectural style. Thus, as seen in Fig-
ure 3.1, each peer can deploy three types of services:
• Miner services, which host and execute process mining algorithms.
• Repository services, which store and provide data for process mining.
• Service registry services, which stores references to the available peers and ser-
vices in the network.

Even though the PDL platform is a good starting point, it still has some limitations. We
now take a look at them:
• Execution of Process Mining Pipelines. As explained before, executing process
mining pipelines using miners and data from different organizations was not part of
PDL requirements, as its main use is to share and validate process mining algorithms.
• Inter-Process Communication. In PDL, all communication between processes
is done by using REST APIs. We ought to explore other alternatives that might
be more suitable for this problem in particular, such as event-based communication
using message queues.
• Data Storage. In PDL the data storage strategy is very basic, consisting of local
file storage. We should define a more complex data storage strategy consisting of
defined data schemes and the usage of database technologies and languages such as
SQL.
• Handling of Large Amounts of Data. In process mining, data files can be heavy
(gigabytes or even terabytes). PDL does not take into account cases in which the
input data files might be too large to handle or to transport.
• Scalability and Availability. PDL does not have any strategy to tackle server
overloading and the need for scaling. This is admitted in the thesis’ ”Future Work”
section [8], in which a reverse proxy using load balancing is proposed.
• User Management. Even though user management and security are mostly out
of the scope of this project, it is worth mentioning that PDL does not provide any
user management capability.

Problem Statement and Research Questions
Process analysts working on process mining projects often face challenges when it comes
to the availability and functionality of process mining tools. In some cases, privacy and
security concerns prevent data from leaving the organization, thereby limiting the use
of remote process mining tools. Furthermore, building, deploying, and executing process
mining pipelines is a complex task that requires analysts to perform a significant amount
of manual programming. This includes adjusting interfaces between algorithms, setting
up a framework to execute the algorithms in a sequential and eﬀicient way, and sharing
data and algorithms in a secure and usable way.
These challenges can lead to ineﬀicient and ineffective process mining projects, resulting
in missed opportunities for process improvement and cost savings for the organizations.
There is therefore a need for a solution that enables process analysts to deploy and execute
automated process mining pipelines while addressing data constraints and reducing the
amount of manual programming work required.

pipelines to discover more challenges that were not addressed in Chapter 2. Moreover, we

A pipeline is composed of several sub-concepts, each of which is carefully designed and

At the top of the model, are the two main sub-concepts of the pipeline: Edge and Node.
The pipeline has a compositional relationship with both the edge and node concepts. This
relation implies that neither edges nor nodes can exist without the pipeline. Furthermore,
the model specifies that a pipeline must contain at least two edges and three nodes, with no
limitation on how many edges or nodes a pipeline maximum can contain. This definition
provides flexibility to the design of pipelines while maintaining the essential structure of
a process mining pipeline.
6.8.1
Edge
Having established the basic structure of a pipeline, we can explore the rest of the model,
starting with the edge concept.
The edge is a crucial component of a pipeline and represents the connection and flow of
data between two nodes. Each edge is associated with an edge configuration, containing
operational parameters. Additionally, each edge is associated with one input handle and
one output handle, which both are of the type handle and represent endpoints on flow
nodes. This definition enables management of the direction in which data flows between
components. Each handle is associated with one type, which represents the type of data
the handle should either receive or send, depending on whether it is an input or output
handle.
6.8.2
Node
Nodes are classified into two categories: Flow Node or Organization Node.

The Organization Node represents the organization and contains all flow nodes be-
longing to the specific organization. The organization node and the flow node have an
aggregation relationship, meaning the organization node has zero or many flow nodes.
The flow node is associated with exactly one organization node. This means that no flow
node can exist outside the boundaries of an organization and that a flow node cannot be
associated with more than one organization.
The Flow Node has, other than the relationships with the handle and the organization
node, a generalization relationship with three concepts, which are: Data Source Node,
Operator Node, and Data Sink Node.
In this relationship, the flow node is con-
sidered the parent. The other three concepts are children, which indicates that they all
inherit the relationships and attributes of the parent. The child concepts each have their
responsibilities and representations, which are the following:
• The Data Source Node represents the source of the input data to be used in the
pipeline. The data source node is associated with a data source configuration,
which in turn is associated with a Data Resource (see Section 6.6).
• The Data Sink node, on the other hand, represents the destination repository in
which the output should be stored. The data sink configuration contains the
instantiating parameters defining the repository where the output should be stored
and the name under which the output should be stored.
• The Operator Node is responsible for executing the specified algorithm with the ob-
jective of processing and analyzing the input data. It is associated with an operator
configuration. The operator configuration is associated with an algorithm resource
(see Section 6.6) which specifies the algorithm that should be executed on the input
data to generate an output.

7.1
DAPM as a Peer-to-Peer Network
We can define a distributed system as a network of geographically dispersed and indepen-
dent computers that are linked together to share resources and achieve a common goal
[12]. This is exactly what the Distributed Architecture for Process Mining (DAPM) is,
a system running in multiple locations to achieve the sharing of process mining data and
algorithms to later design and execute process mining pipelines also in a distributed way.
The choice of making DAPM a distributed system is quite obvious, as it is inherently a
distributed system, meaning that a system whose main use is to collaborate and share
resources, needs to be distributed by nature. But, there are other motivations why to
develop a distributed system [12]:
• Performance and cost. Distributed systems provide distributed computing (which
often becomes parallel), reducing bottlenecks and improving all-around performance.
• Scalability. Distributed systems can easily be scaled to include more resources,
normally by using more machines.
• Fault tolerance. Distributed systems can adapt and overcome better failures in
the system thanks to the multiplicity of different machines.
Distributed systems is a broad topic and encompasses a large number of different archi-
tectures including client-server architectures, which, even if they are distributed, are still
dependant on a central server. DAPM seeks not only to be distributed but also decen-
tralized to avoid dependence on any central authority.
A peer-to-peer (from now on P2P) architecture or network is a distributed system ar-
chitecture that distributes tasks and workload amongst peers [13]. A peer is a participant
of this network and it constitutes an individual and independent computer or process. All
peers are equal, meaning they are equally privileged and have the same rights and duties.
A P2P network is not only distributed but also decentralized. Decentralized networks do
not have a central authority or server in contrast with centralized architectures like the
26
A Distributed Architecture for Process Mining
==PAGE:35==
client-server architecture, where the client is normally lightweight and the server provides
the service.
P2P architectures have several advantages which are relevant to this project, in contrast
to centralized architectures (see Figure 7.1):
• No Single Point of Failure. Due to the lack of a central authority server, there
is also a lack of a single point of failure. So, if a peer fails, the service can still be
provided with the help of the rest of the peers.
• Enhanced Scalability. Because all peers provide storage and computing capabil-
ities, the P2P networks are very easily scalable.
• Enhanced Privacy. Because there is no central authority storing private data,
privacy is enhanced as peers keep control of their data and resources.
• More Democratic Participation.
As there is no central authority, peers can
participate more freely.

7.1.1
Resource Storage, Discovery, and Sharing Among Peers
One of the requirements of the DAPM platform is that it must allow a user to upload
and share its data resources and algorithm implementations. In a traditional client-server
architecture with a centralized database, this would be mostly straightforward. But when
talking about peer-to-peer networks, this becomes more complicated.
In this section,
we describe how we manage resource sharing in the DAPM network while respecting
information privacy requirements.
Based on our requirements and as explained previously in the report, data storage is
straightforward. Each organization or peer should store the data resources and algorithm
implementations of its users. In traditional peer-to-peer file-sharing systems (like uTor-
rent) the data would be scattered around the different peers, and then it would be put
together in the destination peer. This way, the storage capabilities that each peer has to
provide would be more or less equal. In DAPM, due to the data privacy and ownership
requirements, this approach is not possible, meaning that data should stay within the
systems of the owner when that is required.
28
A Distributed Architecture for Process Mining
==PAGE:37==
Resource discovery in a peer-to-peer network refers to the process of looking for and
retrieving a specific resource in a network. In DAPM, resource discovery works by using
resource references. A resource reference is a data object that contains a value that refers to
a resource, which serves to know where to find a resource for its retrieval. This means that
if a peer stores a set of resources, the peers that have a collaboration with it hold references
to those resources without actually storing or copying them. When a peer needs a resource
from another peer, it uses the reference to locate it and to ask the owner peer for it. This
approach respects the privacy and ownership requirements of the DAPM platform and
is computationally eﬀicient compared to other alternatives like resource polling, in which
peers regularly ask other peers which resources they have to offer.
As a summary, we establish that:
• Peers store the resources they own.
• Peers share resources only with the peers that have a collaboration agreement with
them.
• Peers store resource references to the resources of other peers (if they are connected
between them).
7.1.2
Initiating a Connection Between Peers
Having understood the role of each peer when storing and sharing resources and the con-
cept of collaboration between peers, we can now explain the concept and implementation
of the collaboration handshake.
A collaboration handshake is the enactment of a collaboration agreement between two
organizations or peers. During the handshake, the DAPM platform establishes that two
peers are connected and thus, they become aware of each other and their resources are
shared between them. The handshake is performed by two peers, one that initiates the
handshake who is the requester, and the other one who receives the handshake request
who is the requestee. The handshake consists of the following steps:
1. Handshake Request: the requester peer sends a handshake request to the reques-
tee with its Identity (see Section 6.3) and some other information. At this point, no
resource data has been sent yet.
2. Handshake Response: the requestee evaluates the request and decides if to accept
it or not. The requestee then sends a request-response also with its Identity to the
requester. If the response is positive, it also sends a resource registry update with it.
A resource registry update is a list of resource references that refer to the resources
stored by a peer. This concept is discussed further in the report.
3. Response Evaluation: if the response is positive, the requester peer stores the
registry update in its system and sends back a registry update of its own to the
requestee.
4. Acknowledgements: once all registry updates are received and processed, both
the requester and requestee send acknowledgments, and the process is closed.
By the end of this handshake, both peers know about each other and hold resource ref-
erences to the resources that the other peer has to offer. In Figure 7.3, an illustration of
the collaboration handshake can be seen.

7.1.3
Communication Between Peers
When designing a distributed platform, having an eﬀicient and coherent communication
strategy between processes is fundamental to the functioning of the system. In this section,
we take a look at the two main strategies when handling inter-process communication:
synchronous and asynchronous communication. We also discuss the strategy that better
adapts to the communication between peers in the DAPM Platform.
Synchronous communication is a model of communication in which the sender trans-
mits a message and waits for a response before continuing its execution [14]. The main
characteristics of this model are the following:
• It is blocking, meaning that the sender can’t continue its execution while it has not
received a response.
• It involves a very high time dependency between parties, meaning that both
parties have to be available and running so the communication can succeed.
• Provides better and more reliable error control, because of the awaiting of the
response, tracking of errors or failures is easier.
On the other hand, asynchronous communication is a communication model in which
the sender does not wait for a response and thus, does not block its execution [14]. In
contrast with synchronous communication, its main characteristics are:
• It is non-blocking, meaning that the sender does continue its execution and oper-
ations as normal and does not need a response to keep functioning.
• In some cases, it does not cause time dependency. This is because some forms of
asynchronous communication provide a middleware that handles undelivered mes-
sages, making the availability of both systems at the same time not needed.
• Error and exception handling are harder to implement.
When choosing a model for a system or platform, the requirements of the system have
to also be evaluated to understand which model fits it the best. In software engineering,
decoupling refers to the practice of designing the components in a system in such a way
that they are the least dependent on each other. This is a practice that is desirable in
most software applications, but in DAPM it gains special relevance due to its decentral-
ized and distributed nature.
Due to the lack of a central authority that manages the
30
A Distributed Architecture for Process Mining
==PAGE:39==
availability, performance, and status of the different peers, the decoupling between peers
is very important as it allows for:
• Higher degree of availability, as failures and errors do not propagate as easily.
• Higher degree of extensibility, flexibility, and reusability.
• Higher degree of fault tolerance.
Taking all of this into account, the chosen method for handling communication between
peers in the DAPM platform is the use of an asynchronous web API, titled Peer API.
An Application Programming Interface (API) is the tool used by two or more computer
programs to communicate between them [15]. An API exposes a set of functions to allow
interaction with the computer program it belongs to. The main purpose of an API is to
hide and abstract the underlying implementation of a computer program to simplify the
interaction with external components. The Peer API is a web API, which is an API used
to interact with a web service. Even though web APIs are seen as generally synchronous,
asynchronous behavior can also be implemented. In the case of inter-peer communication
in DAPM, asynchronous communication is beneficial to increase decoupling between peers
and decrease blocking times, as some operations in DAPM take a long time to complete.
These two benefits also improve the scalability of the platform.
An asynchronous web API is implemented by processing requests independently of the
client’s control flow. This means that when a request is received, an immediate response
with status information is sent to the client as an acknowledgment that the request has
been received and that the result will be served in an undefined amount of time. This way,
the client is not blocked and can continue with other tasks. Once the request is completed,
the server can notify the client and provide it with the request result. In Figure 7.4 an
example of asynchronous communication between peers using the Peer API can be seen.
In the example, Peer 1 sends a request and immediately gets a response indicating that
the request has been accepted and is being processed. Peer 2 then processes the request
asynchronously by sending it to other components in the architecture for its resolution
and sends the result back to Peer 1 using a POST method when finished.

7.2.2
Microservices Architecture and Overview of a Peer
Even though there is no oﬀicial definition for a microservice architecture, the one we agree
the most on, which was given by Lewis and Fowler, is the following one [17]:
“An approach to developing a single application as a suite of small services, each running in
its own process and communicating with lightweight mechanisms, often an HTTP resource
API”
This definition is tied to a set of principles and benefits in contrast with monolithic ap-
plications, which are single-unified software applications that due to their low decoupled
nature, generally lack extensibility, fault tolerance, and scalability[18].
Independence and loose coupling of the different services is crucial in a microservice
architecture [17]. It helps mitigate the dependencies between different parts of the appli-
cation. It also allows for better employment of the single responsibility principle, which
states that each software component should have one function. In contrast to monolithic
applications, the decoupled services help with having a decoupled application.
Decomposition into feature-driven services is another pillar of microservice archi-
tectures [17]. Instead of dividing services by technology layers, services are divided based
on the features they provide. This allows developers to better understand certain features
or areas of the product, making it easier to work on specific services.
Polyglot programming is a major advantage in contrast with monolithic applications
[17].
Due to the separation of the application into independent services, there is no
strict imposition on the programming language of use for the whole application.
The
programming language can be changed and adapted depending on the requirements of
each service.
Independent data persistence strategies is another of the principles of microservice
architectures [17].
Similarly to the polyglot programming benefit, each service has an
independent database where it persists the data that is relevant to itself. This principle
brings major benefits like higher scalability and more resistance to failure due to the
decentralization of the database. However, the most important one is the ability to choose
the best persistence strategies and technologies based on the requirements of the service.
Continuous delivery and easier deployments are also a major advantage of mi-
croservice architectures [17]. As with the rest of the principles, this main benefit is due
to the decentralization of the application. With independent services, specific deployment
strategies can be designed for each service, which also means that independent and lighter
deployments and fixes can be delivered without affecting the rest of the application.
36
A Distributed Architecture for Process Mining
==PAGE:45==
Increased fault tolerance is another benefit [17] due to the lack of a single point of
failure. This means that if a service fails, the rest of the services could in theory continue
their normal execution unless they have a high dependency on the failed service.
All of these principles, help microservice-based applications be more scalable and main-
tainable due to their facility to allow for a distributed development [17], that is why,
nowadays more and more applications choose this architectural style against monolithic-
based software. The DAPM Peer is one of them.

7.2.3
Repository Microservice
The first service we will describe is the Repository Microservice. The purpose of this ser-
vice is to store and manage the various resources that are both uploaded to the system and
produced by it. The resources that the Repository Microservice is responsible for storing
include algorithm implementations, pipelines, and data resources such as event logs.
To achieve an optimal solution, the Repository Microservice has to fulfill a couple of
requirements. First, the storage of resources should be organized to ensure eﬀicient re-
trieval. Second, the service should be able to store files that are several gigabytes in size.
This is because, in process mining projects, it is not improbable that an event log contains
several gigabytes of events.
To meet both requirements, we have designed the Repository Microservice to employ
a hybrid approach, utilizing both MongoDB and PostgreSQL. MongoDB is a document
database that we use to store files, such as event logs and algorithm implementations [19].
The decision to use MongoDB is based on the fact that it provides GridFS, a specification
for storing and retrieving large files. Instead of storing a file in a single document, GridFS
divides the files into smaller chunks and stores them as separate documents. With GridFS
A Distributed Architecture for Process Mining
37
==PAGE:46==
the database is structured into two collections, one for storing the file chunks and the other
for storing the file’s metadata. The metadata includes a list of all file chunks belonging
to the corresponding file. With this, the system can store and retrieve large files, even if
they are several gigabytes [20].
PostgreSQL is an object-relational database management system (ORDBMS), meaning
that it organizes data into tables with rows and columns, and relationships between the ta-
bles [21]. The Repository Microservices uses PostgreSQL to arrange data such as pipelines,
in JSON format, the different repositories, and references to the files stored in MongoDB.
We have in figure 7.10 provided an entity relationship diagram to illustrate the structure
of the PostgreSQL database.
Figure 7.10: Entity Relationship diagram(ERD) for PostgreSQL Database
The PostgreSQL database contains five tables: Files, Operator, Resource, Repository, and
Pipeline. The Repository table contains the IDs and names of available repositories. This
table has a One-To-Many relationship with the Operator, Resource, and Pipeline tables,
meaning repositories may contain zero or many operators, pipelines, and resources. All
three tables also contain a foreign key which is the repository ID that referees to the
repository it belongs to.
The Files entity has a One-to-One relationship with the Resource entity and two One-
to-One relationships with the Operator entity. This is because the Resource entity only
refers to one file and vice versa, while the Operator refers to two files, the source code
file and the Dockerfile. We will later in Section 7.2.5 explain the purpose of the Dockerfile.
With this structure, the Repository Microservice can maintain an overview of the reposi-
tories of the hosting peer and the pipelines, operators, and resources stored in each repos-
itory. Furthermore, it can store files in MongoDB and store references to these files in the
PostgreSQL database. This approach allows the Repository Microservice to easily retrieve
the exact MongoDB ID from the PostgreSQL database for a resource or an operator, and
then query MongoDB for the file.

Operator Microservice
The Operator microservice is the component in the system architecture that is responsible
for executing the algorithms on input data, both of which are provided to it, and handing
back the output data. The Operator microservice is only concerned with the algorithm
execution and does not contain information about either the progress of the pipeline exe-
cution or the subsequent steps.
For the solution architecture to fulfill its intended purpose, the Operator microservice
has to meet certain requirements. First, it must be able to execute multiple algorithms
A Distributed Architecture for Process Mining
39
==PAGE:48==
simultaneously, since the system is designed to be used by multiple users. It is, therefore,
likely that numerous users will request a pipeline execution at the same time, and some
might even request more than one pipeline execution. Being able to execute multiple al-
gorithms is therefore essential to avoid a bottleneck.
The second requirement is that the Operator microservice must be agnostic concerning the
programming language, meaning that it must be able to execute the algorithm regardless
of which programming language it is written in. Process mining algorithms are written in
several programming languages, such as Python and R. The Operator microservice must
therefore be inclusive. Additionally, the Operator microservice is also required to be de-
pendency/library agnostic. Depending on the language used, multiple dependencies are
available to be utilized in the context of process mining. Therefore, to be inclusive, the
Operator microservice must be able to execute the algorithm regardless of the dependen-
cies utilized.
To meet these requirements, the Operator microservice uses Docker to containerize (con-
tainerization is discussed in more detail in Section 7.3) each algorithm execution. We call
these algorithm execution containers Operator Instances.
By containerizing each algo-
rithm execution, the Operator microservice can:
1. Manage and execute multiple algorithms, as Docker containers are easily scalable.
2. Execute algorithms written in any programming language since Docker containers
are language-agnostic.
3. Ensure that the required dependencies are readily available within the container, as
Docker uses Dockerfiles, which are essentially instructions to build the image used
to create the container.
Furthermore, Docker containers ensure that the algorithm can be executed consistently
on any machine, thereby eliminating any compatibility issues that may arise due to differ-
ences in host systems. Docker containers are also isolated and independent, resulting in
a fault-tolerance solution, where the failure of one container does not affect the execution
of any other container.
By utilizing Docker containers, the Operator microservice works as follows: the Operator
microservice is provided with the input data, the process mining algorithm, the Dockerfile,
and other necessary information. Equipped with this, the Operator microservice can cre-
ate and start an Operator Instance containing the process mining algorithm and provide it
with the corresponding input data. While the Operator Instance is running, the Operator
microservice remains available for new algorithm execution requests. When an Opera-
tor Instance has completed the execution, the Operator microservice retrieves the output
file, sends it to the requester, and terminates the completed Operator Instance. This is
achieved by using the Docker.DotNet package in our implementation to interact with the
Docker Daemon. This package handles the communication asynchronously, allowing the
Operator microservice to proceed with other operations while waiting for a response [22].
To transfer data between the Operator microservice and the Operator Instances, Docker’s
volume mechanism is employed. A Docker volume is a mechanism for storing data created
and used by Docker containers [23]. Docker volumes are managed by Docker, meaning
that Docker will create a volume and mount it to both the Operator microservice and
each created Operator Instance. The Operator microservice uses this volume by creating

7.2.7
Communication Between Microservices
One of the main challenges when designing a microservice architecture is the design and
implementation of the communication between them. To implement the communication,
a set of different patterns already discussed in the literature can be used [14], but we must
choose the correct one for our unique use case. In this subsection, we discuss the most
relevant communication patterns between microservices for the DAPM Peer and describe
the design we developed for our solution.
When describing the communication between DAPM Peers in Subsection 7.1.3, we al-
ready introduce the differences between synchronous and asynchronous communication.
The choice of communication between microservices is also very influenced by these two
different communication methods. It is worth noting that no solution is best for all cases.
We are fully aware that to design the best communication strategy, most of the time a
hybrid solution must be designed. For example, mixing synchronous and asynchronous
communication based on the specific microservices that communicate with each other.
Despite being aware of that, a unique method of communication was designed for all cases
due to the limited time and scope of this project.
Synchronous communication between microservices is discarded. One of the reasons is the
high coupling between services that it provides. As already discussed, one of the main char-
acteristics of the microservices architecture is its ability to be decoupled and the services
be more independent. Introducing synchronous communication would limit the benefits
that the architecture provides.
Synchronous communication is mostly implemented as
point-to-point communication [14]. This communication pattern consists in communicat-
ing directly with each service probably using a RESTful API. This is also undesirable as
it would increase complexity and coupling due to the lack of a common communication
abstraction layer (meaning that every service should be aware of the specifics of the APIs
of all the other services). Finally, in our concrete case, synchronous communication is
not desired because some tasks in the DAPM platform take a lot to execute, which would
result in high thread-blocking times.

On the other hand, asynchronous communication allows for a more decoupled architecture.
It also does not block the client service while waiting for a response, which in our case is
very important because DAPM has processes that take a big amount of time. Because of
this, we have chosen asynchronous communication between services in the DAPM Peer.
One of the most common techniques and the one we use is the usage of asynchronous
message brokers. A message broker is an asynchronous communication media in which
a message producer sends messages to a queue and the consumers receive messages from
the queue asynchronously [14].
This type of communication includes an extra service
(the queue provider) between the message generators and the message consumers. The
point of this intermediate service is to store messages until the consumer service can
consume them, effectively implementing asynchronous communication. A great benefit of
this approach is that services are unaware of each other or their interfaces, as they only
have to communicate with one interface, the queue provider. This means that message
producers do not know what service will consume the messages they produce, improving
the decoupling between services. This communication pattern can be seen in Figure 7.9

Collaboration Between Microservices
In Subsection 7.2.2, we described a microservice architecture as a set of small and inde-
pendent services in which each service has a limited set of responsibilities. Due to the
separation of the application into services, these services often need to collaborate in order
44
A Distributed Architecture for Process Mining
==PAGE:53==
to provide the full functionality of the application. This is why defining the correct strategy
of collaboration between microservices is very important. In this subsection, we describe
the two main strategies or techniques when designing the collaboration between different
services, their advantages and disadvantages, and how we designed and implemented this
collaboration in the DAPM Peer.
The choreography pattern or technique of collaboration between microservices refers to
the capability of the services to coordinate and organize the interactions without the need
for any central authority [24]. Each service produces a set of events that are subscribed
and processed by other services. In choreography, services process events without being
aware of the full picture. When a service processes an event, it does not know what it is
for or who will process the resulting event.
Figure 7.17: Example of an Event-Driven Choreographed Microservice Architecture [24]
The main benefits of choreography are that it provides a loosely coupled architecture,
making it more scalable and resistant to failure, due to the lack of a main authority.
Choreography also provides ease of maintenance, as each service has its responsibilities
and thus can be developed independently. The downside is, that due to the lack of a
central coordinator that has a view of all the system and its interactions, choreography
produces a lot of complexity when coordinating the different services. Besides that, it
also produces a lot of diﬀiculty when troubleshooting, due to the distributed nature of the
program execution.
On the other hand, orchestration provides a central authority that is responsible for the
coordination and management of the different services to achieve the common goal [24].
In contrast with choreography, orchestration provides simplicity in the implementation of
the services, as the services only need to execute their tasks and go back to the orches-
trator, who understands the bigger picture. Due to the existence of a central authority,
orchestration also provides greater visibility, which means the system and the developer
can understand easier what is happening in the system and the different interactions be-
tween services. Finally, orchestrated systems are also easier to troubleshoot. Despite the
major benefits, orchestration provides a severe downside, which is the existence of a single
point of failure. If the orchestrator fails or contains an error, it can affect greatly the rest
of the system.

45

To design the collaboration between microservices in the DAPM Peer, we decided to use
the orchestration pattern. During the initial phases of the development, the choreography
approach was used, but it was discarded due to the following reasons:
• Increased complexity in the individual services. As soon as the complexity of the
implemented features increased, the complexity of implementing the workflows in
the individual services also did until a point in which it was unsustainable.
• Lack of visibility of the flow. Services in a DAPM Peer depend highly on each
other, because of this, a general vision and visibility of the flow is very important
for the system and the developer, as it makes development simpler.
Orchestration in DAPM serves mostly a practical purpose and provides an easier develop-
ment experience, as having a central authority provides easier troubleshooting and greater
visibility of the collaboration between services.
As seen in Figure 7.9, orchestration in the DAPM Peer is implemented with the usage of the
Orchestrator Component. This component acts as the brain of the peer and coordinates
the services to provide the full functionality of the peer. Features in a DAPM Peer can
be decomposed into a set of independent actions each one carried out by a service. The
Orchestrator Component models and implements these chains of actions, and asks the
different services to execute them in a coordinated manner. This allows the services to
be only dependent on the orchestrator, not worry about the general flow, and execute the
actions in a decoupled and independent way.

Deployment Using Containerization
One of the requirements of the DAPM Peer is that it should be easily deployable. We
could say that the ease of deployment is a requirement beneficial to any kind of software,
as it allows for easier development and provides more agility and ease of management.
But, in the case of the DAPM Peer, it gains special relevance. This is because due to
the decentralized nature of the platform, it needs to be installed, set up, and deployed
independently with no help from a central authority.
We can deploy a DAPM Peer by using containerization technology. Containerization is
the packaging of software code with just the operating system libraries and dependencies
required to run the application to create a lightweight executable that runs on any in-
frastructure. This executable is what we call a container [25]. Even though containers
are isolated processes, they share the host’s machine operating system. This makes them
lightweight as they do not require the overhead of associating a whole operating system
to the running of the application. The main benefits of containerization are the following
[25]:
• Portability. With traditional methods, developers write code on a specific platform
and use a set of libraries compatible with that platform.
When deploying that
software to another platform, bugs, and errors would arise due to incompatibilities.
Because containerization bundles any needed file for the application to run, the
application is portable to any platform and operating system.
• Speed and Eﬀiciency. Due to the sharing of the host’s operating system, the
containers are light and thus, faster than other alternatives like virtual machines.
• Fault Isolation. Because each container is isolated, its failure does not affect the
running of the rest of containers.
• Ease of Management. Even though this technology is not used in DAPM, con-
tainerization can be paired with container orchestration software, which allows for
easily manage and automate container deployment.
As already hinted, one alternative to containerization is virtualization.
Virtualization
allows multiple operating systems to be run on the same computer by using a component
caller hypervisor, which separates the hardware from the operating system. The problem
with virtualization is that it is much heavier, as it makes a copy of the whole operating
system [25]. Also, the fact that virtual machines do not share the same operating system,
makes it harder to implement communication between the different applications running
on the same computer.
In DAPM, we implement containerization by using Docker. Docker is an open platform
for developing, shipping, and running containerized applications [26]. In Figure 7.19, we
can see an overview of the Docker architecture and its most important concepts.

Figure 7.19: Docker Architecture Overview [26]
The Docker Daemon is responsible for managing the different Docker objects, these
being images, containers, networks, or volumes. It uses the Docker API to communicate
with the client. A Docker Image is a read-only file file with instructions for instantiating
a Docker Container. Docker images can be uploaded to a Docker Registry for others to use,
effectively sharing software. A Docker Container is a runnable instance of an image.

8.3
Decomposing the Pipeline Into Steps
Once the pipeline model is received at the DAPM Peer, it is the responsibility of the
pipeline orchestrator microservice to orchestrate its execution. The first step for doing
this is to transform the pipeline model, in which the pipeline is represented using a set of
nodes and edges, into an executable sequence of pipeline steps.
A pipeline step is an action within the context of pipeline execution that is executed by
a peer and can be executed independently, that is, without having any relation to other
steps in the same pipeline. The fact that steps are executed independently and without
any relationship with external events, helps us ensure the correctness of the software, as
edge cases are more diﬀicult to appear.
52

A process mining pipeline execution can be decomposed into two main kinds of actions:
transferring data from one place to another and executing algorithms. Because of this, we
have two kinds of steps: data transfer and operator execution steps.
Data transfer steps mainly move data from one place to another. The responsible peer
to execute this kind of step is the peer that contains the data that needs to be transferred.
In this context, we have different possibilities:
• Transferring input data from a repository (data source) into an operator (algorithm)
to have it ready for its execution.
• Transferring output data from an operator so it becomes input data for another
operator.
• Transferring output data from an operator into a repository (data sink) to have it
stored as a result.
Operator execution steps trigger the execution of an algorithm, consuming input data
and producing output data. To execute an algorithm, the Operator microservice has to
be ready for it. This means that the Operator microservice has received all input data
needed to execute the algorithm. The peer responsible for executing this kind of step is
the peer that owns and stores the algorithm implementation to be executed.

8.5
Executing Steps
Until now we have seen how a pipeline can be decomposed into a set of steps and how

8.5.1
Data Transfer Step
As we explain in Section 8.3, the objective of a data transfer step is to move data from
one place to another. This data can be an event log, a process mining model, or any kind
of result from an algorithm. The step is executed by the peer that holds the data that
needs to be sent. To execute this step, we can divide it into two parts:
• Getting the target data.
Depending on the case, the data is retrieved from
the operator microservice or the repository microservice. When we retrieve data
from the operator microservice, it is because the data is the result of an algorithm
execution. When we retrieve data from the repository microservice, it is because the
data is stored in a repository.
• Sending the target data.
Once the data is retrieved, the data is sent to the
recipient. Again, the data can be sent to two microservices, the operator microservice
and the respository microservice. To the repository microservice for storage in a
repository, and to the operator microservice to set it up as input data for an algorithm
execution. If the data has to be sent to another peer, the data is sent to the Peer

• Store the received source code and Dockerfile in the dedicated directory in Docker

• Modifies the Dockerfile, replacing the argument placeholders with the actual paths

• Sends a request to Docker Daemon with the Dockerfile and source code to build an

container, the Operator Microservice sends a new message to the Docker Daemon, re-

When the Operator instance container starts, it pulls the input data from the provided

With this message, the Pipeline Orchestrator knows that the step with the provided ID is
complete and the following step of the pipeline can be executed.

9.1
Solution Overview
The resulting solution of the entire project, including the work done by the conceptual
and security teams, is a peer-to-peer platform that enables its users to design pipelines,
initiate them with actual resources, and execute them. The organizations (peers) are able
to establish collaborations with other organizations, which enables them to utilize each
other’s resources and algorithms. This allows users to design and execute pipelines that
cross organizational boundaries.
We, the platform team, have contributed to the project by developing a software solu-
tion that allows organizations to form a decentralized network of peers that enables the
sharing of resources and their use for designing and executing process mining pipelines.
For this solution, we have developed a conceptual model and a software design, and have
implemented a prototype to validate it. Additionally, we have contributed to conceptual
work other than our own during the weekly discussions with the other teams. The most
notorious contribution is the development of the process mining pipeline model in Figure

The architecture we have designed for a peer in the platform is based on microservices,
where each service has its own specified purpose such that the services collectively ful-
fill the overall purpose of the system. With this architecture implemented, the system
provides the users with the following features:
1. Design process mining pipelines: Users are able to design process mining pipelines,
utilizing the resources stored in the system’s repositories.
2. Instantiate and execute pipelines: Users are able to instantiate previously de-
signed pipelines with resources and algorithms available on the platform and execute
them.
3. Upload and store resources: Users are able to upload and store resources such
as event logs, algorithms, and pipelines in repositories.
4. Establish collaborations with other organizations: Users of one organization
are able to establish collaborations with other organizations.
It is worth noting that our solution is a decentralized platform that does not provide any
resources, but provides the users with the ability to upload their event logs and algorithm
implementations, to be stored and organized in repositories for later use. Additionally, the
users can design process mining pipelines by using the front end and instantiate them with
the uploaded resources and algorithms. The pipelines can then be stored and executed
in the platform, producing an output file. Furthermore, the solution provides a feature
for organizations to establish collaborations with other organizations. This feature allows
authorized users to create connections between their organization and external partners,
enabling cross-organizational cooperation and resource sharing within the platform.

Discussion and Future Work
In this chapter, we discuss our project in relation to the research questions stated in
Section 3.4. We start by providing a general discussion on the aims of the project, the
outcome, and its fulfillment in regard to the general project research question. Afterward,
we go into detail by discussing each sub-question, providing a discussion on whether the
objectives for a specific question have been achieved and the possible improvements for
future developers.
This project is aimed at resolving the issues many process mining analysts face when trying
to design and execute process mining pipelines in the process of collaborating with other
process mining analysts. This aim is formulated as a research question in the following
way: How can we design an architecture to assist researchers and data analysts in deploy-
ing and executing automated process mining pipelines? The results to this question have
been achieved through the introduction of a Distributed Application for Process Mining
(DAPM). The DAPM Platform provides a distributed and decentralized architecture to
which organizations can adhere, providing their members (users) the capability to col-
laborate with organizations and their members by designing and executing joint process
mining pipelines. In Chapter 9, we describe the functionalities achieved by the DAPM
prototype in relation to the scenarios presented in Chapter 2. This provides a general vi-
sion of the extent to which the research question is answered by the prototype. To answer
the main research question, we divided it into four sub-questions, each one focusing on
one area of the project. The following sections provide a detailed discussion of each one
of them.
10.1
Sub-Question 1
How can we design, deploy, and execute robust pipelines capable of eﬀiciently
handling large volumes of event logs?
To design pipelines, a user must be able to access the different elements that compose them.
DAPM does this by providing the capabilities of uploading data resources and algorithm
implementations and accessing them through a well-defined and comprehensible interface.
DAPM defines a comprehensible data model, based on organizations, repositories, and
resources. The access to this model is then implemented by using well-defined RESTful
APIs or by using the DAPM Client frontend application. Despite this, more work on
the description and validation of the content of the resources is needed. To allow for a
better user experience, a solution to describe a specific resource and determine its concrete
structure should be put in place.
This would allow the validation of a concrete data
resource as an input or output of an operator, which would also translate into an easier
usage of the input data within the operator source code. In the current prototype, DAPM
allows any resource to be uploaded, without any validation.
The same applies when
designing a pipeline, any resource can be used. This means that DAPM still expects much
manual work and understanding of the resources from the process mining analysts.
To deploy and execute process mining pipelines, DAPM provides the capability of trans-
forming a given pipeline into a set of actionable steps and the capability of orchestrating
and executing these steps in a decentralized manner to achieve the final objective of the
pipeline. One of the main requisites of the DAPM platform is to respect privacy and
data usage policies across organizations. This means that in some cases, data resources
A Distributed Architecture for Process Mining
65
==PAGE:74==
can’t leave the owner organization’s systems. This feature is not currently implemented
in DAPM, as the orchestration algorithm always sends the data where the source code
algorithm is. More work should be done to allow for different privacy and data usage
settings. Due to the time limitation of the project, this was not done, but the extensibility
of the current software allows it to be implemented.
DAPM is designed and implemented to have the capability of executing multiple process
mining pipelines simultaneously, which also means that it can execute multiple Operators
(algorithms) concurrently. As mentioned in Section 7.2.5, this is achieved by containerizing
each process mining algorithm using the Docker Daemon. We believe this is a good solution
for a few reasons:
• Docker containers are language- and dependency-agnostic, meaning they can run
and execute the algorithms on any host machine. This gives users the freedom to
use any programming language and dependencies they see fit, without having to
worry about compatibility.
• Docker containers are lightweight, making them a better option for running multiple
instances concurrently.
• Docker containers provide process isolation. The isolation provided by containers
helps prevent processes in one container from directly affecting those in other con-
tainers or on the host machine, enhancing overall system stability and security.
However, even though we believe Docker containers are a good solution, they are still not
perfect. For instance, while they are more eﬀicient than traditional virtual machines, it’s
important to note that resource management is still necessary, especially when running
multiple containers. Proper allocation and monitoring of CPU, memory, and storage re-
sources are crucial for optimal performance. Additionally, even though Docker containers
are isolated, they still cannot be considered completely secure, especially in our scenario
where the Operator containers all share access to the same Docker volume with the Oper-
ator Microservice. Since the source code is provided by the users, it is reasonable to argue
that a validation of the code should take place before execution.
Also due to the time limitation of the project, a concrete solution for handling large
volumes of event logs was not put in place. This poses one of the main limitations of the
current prototype, which only works with light files. This is due to the ineﬀicient transfer
of large volumes of data between services, which slows down the prototype.
10.2
Sub-Question 2
How can we design inter-process communication to implement collaboration
between services in the architecture?
The communication between peers in the DAPM P2P network is designed to use an asyn-
chronous communication approach. This is important as asynchronous communication
allows the network to be more decoupled and reduces block times, as some operations in
DAPM take a long time to complete. The chosen implementation method is to use an
asynchronous web API. Even though we are sure of the asynchronous approach, we are
not that sure about the chosen implementation method because it produced high cou-
pling between peers, because the server peer has to be aware of the client peer in order
to return the request results. More investigation could be used to determine the best
implementation for the communication between peers while respecting the design decision
of asynchronous communication.
66
A Distributed Architecture for Process Mining

The communication between microservices within a DAPM Peer is also based on asyn-
chronous communication. Again, this helps with keeping the decoupling within the system,
improving failure tolerance, and reducing blocking times. The implementation method is
asynchronous message-based communication using message queues. This implementation
method adds another benefit, which is the existence of a middleman between services and
thus another layer of abstraction. The main problem with this approach is that message
queues are not the best option for sending files because messages are supposed to be light,
while files are a lot of times heavy. This is even worse in the context of DAPM, as event
logs might be gigabytes large. Even though the message-based approach can be kept, a
new approach has to be designed for when sending files, so that the platform can be fast
and reliable.
To design the collaboration between microservices, two options were considered: the chore-
ography and orchestration patterns. At first, we chose to implement a choreography-based
system, but soon we realized that this approach made the implementation too complex
and thereby neither extendable or maintainable. This is because in DAPM, even if the
services are decoupled and independent, they still rely highly on each other to provide
the full intended features. Even if we made the choreography-based approach work, the
amount of complexity would have made the software very hard to work with and hardly
extensible. We implemented the orchestrator pattern by introducing a new service called
Orchestrator. The function of this service would be to manage all the microservices, so
they would only communicate with the orchestrator instead of between each other. This
approach was also questioned because it creates a very obvious single point of failure,
which is contrary to the microservices architectural principles. Even though it is true,
this single point of failure does not pose a high threat to the system’s availability be-
cause the orchestrator service only persists temporal data in memory and can be easily
re-instantiated without affecting the functionality of the system. Another objection to this
approach was the fact that it does not match the asynchronous method of communication
because it implements a highly synchronous execution flow, meaning that even if we do
not block the thread, we are always waiting before executing the next step in the flow.
Even though this is true because of the coupled nature of DAPM execution flows, the
justification for asynchronous communication using message queues relies on the decou-
pling of the components (not calling directly the other services) and the insertion of an
abstraction layer between microservices, which does not make the microservices be aware
of other services and their interfaces.
10.3
Sub-Question 3
How can we design an effective data handling strategy to manage the different
types of data that the platform deals with?
Data handling strategy refers mainly to the storage and transferring of data. Data storage
in DAPM is implemented in the Repository microservice within each DAPM Peer. The
chosen data storage strategy was to store all data as files in a hybrid database system
comprising an SQL database and a file-oriented NoSQL database. During the inception
phase of the project, many alternatives were discussed. One of them was storing files but
using the built-in file system within the operating system. This was rapidly discarded
due to the complexity of managing a file system, the slowness of it, and the lack of a
usable API to work with it. Another alternative was the storage of individual events as
rows in a database. This would give much more control over individual events and could
ease the implementation of streaming-based algorithm execution. But this alternative was
also discarded due to the complexity of creating a database model that could fit all event
A Distributed Architecture for Process Mining
67
==PAGE:76==
definitions and also because there was an assumption that events would never be accessed
individually, they were always going to be accessed jointly with the rest of events in the
event log.
In the current implementation, data storage is very general, meaning that it is not tailored
to specific use cases. Both event logs or input data and operators and their source code
are stored within the same database system. We would have liked to design more specific
and need-based data storage strategies, but due to time limitations, this was not possible.
Despite this, the microservice architectural style we implement allows for the creation of
new independent microservices that implement specific storage strategies with independent
databases based on specific needs.
In decentralized peer-to-peer systems, the database is also decentralized, meaning that
the data is scattered around the different peers. In DAPM, we established that each peer
would store the data owned by its users. Several strategies were proposed to allow data
discovery between peers, in other words, allow peers to be aware of the distributed database
and where to find the different pieces of data across the network. It was established that
instead of copying data around, which would be very ineﬀicient, we would use object or
data references. These references are information about where to find a specific piece of
data or file. After deciding this, there was another debate on what references to store and
how to retrieve them. It was agreed that each peer would only store references of the peers
that have a collaboration agreement with it. The initial solution was designed to store
references to all data and resources in the network, making the DAPM Network a unique
and fully connected one.
This first approach was deemed unnecessary and ineﬀicient
because each peer did not have to know about the data held by all the other peers, only
the data of the ones that had a collaboration with it. A debate on how to obtain the
references is still open. The current approach is that when a peer has a new data object
to offer, it broadcasts it to the peers it is connected to. But, another approach would be
to access these references on demand, asking for them each time a peer would like to use
them. At first, this last approach was deemed computationally ineﬀicient, but it could be
better for maintaining the privacy requirements of the data.

We

Conclusion
Process mining is a field that in the last years has gained special relevance due to the
advancements in data and process analysis. In this field, the usage of multiple algorithms
and different data sources, often developed and owned by different analysts across orga-
nizations, in the form of a process mining pipeline is crucial to get the best results. The
lack of a tool to design and execute such pipelines poses a challenge for process mining
experts.
This thesis provides an architecture and implementation for a platform titled Distributed
Architecture for Process Mining. This platform aims to provide the means for sharing data
resources and algorithms between data analysts and across organizations. Moreover, this
platform also provides the capabilities of designing and executing process mining pipelines
using the aforementioned resources.
DAPM is distributed, decentralized, and based on a peer-to-peer network. In DAPM,
organizations take the role of DAPM Peers and form collaborations with each other to
provide their users access to the data resources and algorithms of the other organizations.
Each organization (or peer) stores the data and algorithms belonging to their users, ef-
fectively complying with any data privacy requirements. Each DAPM Peer also provides
their users with a frontend application, which can be used to design and instruct the ex-
ecution of process mining pipelines.
A DAPM Peer is designed using the microservices architectural style.
This approach
helps the DAPM Platform be more scalable, extensible, and maintainable. The commu-
nication between peers and between microservices within a peer is performed using an
asynchronous approach, providing a great degree of decoupling between processes, and
allowing the platform to be more resilient to failure.
Pipeline execution is also performed in a decentralized and asynchronous way. Pipeline
designs are transmitted using a data structure which is later transformed into a set of
executable steps, each one executed by a specific peer, depending on the needs. Algorithm
execution is performed by using containerization implemented with Docker. This tech-
nology allows the algorithm executions to be language and dependency-agnostic, meaning
that it does not have to explicitly know what technologies are used to implement the algo-
rithm, as they are defined by a Dockerfile and can be run independently of the operating
system. Containerization also provides lightweight process instances, making them faster
and more agile than traditional virtual machines.
The main limitation of the current solution is the handling of large amounts of data. In
process mining, data sets can be large, getting to gigabytes of data for one single file. As
discussed previously, the current solution has limitations for this use case, mainly because
the transmission of files between services is done using lightweight messages, which is a
temporary solution due to time and resource limitations. The current solution for storing
files has not been tested properly with large files. This could also slow down the system
when using large amounts of data. Another possible improvement for the current solution

==PAGE:78==
is the enhancement of the pipeline execution logic. Currently, the input data is always
transmitted where the algorithm’s source code is (often moving it from peer to peer). This
can pose a challenge to the privacy requirements, which are of great importance in the
DAPM context. A more complex pipeline execution flow should be put in place to allow
for different logic that could comply with any data privacy constraints.
The current prototype of DAPM is set to be the result of the first iteration of this project.
For each research question we proposed at the beginning of this report, we have provided an
answer which can also be considered a first iteration per question. Even though we would
have liked to develop the project further, we understand the time and resource limitations
of the project. Precisely because we were aware of these limitations, the architectural
principles of the platform we have designed now allow for a big degree of extensibility and
scalability, allowing future developers to keep iterating on the platform and the research
questions of the project.
This project has been one of the first steps to develop a platform to allow collaboration
between process mining analysts. We are sure that this prototype will be further developed
and will be used by process mining experts to get the best results out of their research.
Finally, we would also like to deeply encourage the reader to visit the Git repository in
which the implementation of this project has been developed, which can be found on the
first pages of this report, in order to have an even better understanding of the structure,
design, and execution of the solution of this thesis.
