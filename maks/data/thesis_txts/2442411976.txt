Summary
For students studying control in university, being able to get hands-on experience with
designing controllers for physical systems is a huge benefit. For this reason, the company
ŌĆØEducational Control ProductsŌĆØ (ECP) have created a line of products which are specif-
ically meant for use in control course exercises. The Automation and Control Group of
DTU have recently acquired their ŌĆØModel 205a Torsional PlantŌĆØ - A system comprising
of three disks in a stacked configuration connected to each other via flexible metal shafts,
with a drive motor connected to the bottom disk for control.
The system is simple in its setup, yet exhibits a slew of interesting dynamic properties.
Combining this with the fact that it is highly customizable for a range of purposes, it is
seemingly ideal for use in control courses. However, it was discovered fairly early on in its
use that it exhibited some undesirable traits, which made controller design unnecessarily
difficult. A problem when used in introductory control courses for students. It was
established early on that friction phenomena, which was neglected in the original model
of the system, was the likely cause for this.
This report documents the process of analyzing the system to explain this phenomena,
modeling the friction to improve the model, and designing a new controller for the
updated system to show the improvements made.
During the analysis phase, a number of experiments were set up and used to get
important information regarding the system behavior, specifically getting hard data
regarding the undesired phenomena. It was discovered that not only is there a significant
amount of un-modeled friction in the system, but this friction exhibited erratic and
unpredictable behavior. A systematic analysis was carried out to identify the source of
this phenomena, which culminated in adjustment of the hardware itself.
Following this, the system was re-analyzed and it was seen to exhibit much more
predictable behavior. The friction in the system was then quantified and a model was
developed to describe it. Finally, the new controller designed for the updated system and
model, now including the previously un-modeled friction, was shown to have improved
performance over the original design.
Whether or not this updated setup will help students who were previously struggling
with the original setup remains to be seen, however the preliminary results are looking
positive.

CHAPTER 1
Introduction
As a student of a control engineering course, it can oftentimes be hard to visualize how
different controllers affect different systems, and thus the whole process of designing
controllers can seem rather vague when, for example, designing controllers for a simulated
system. Although the theory is all there, when thereŌĆÖs no direct connection to the real
world it can be difficult to gain much more than a shallow understanding of the subject.
Because of this, it is often desired for there to be a practical module to the course,
where students can get hands-on with designing controllers for physical systems. In
this vein, the company ŌĆØEducational Control ProductsŌĆØ (ECP) have created a line of
products tailored specifically towards use in control course exercises and experiments.
In this project, the focus is directed towards their ŌĆØModel 205a Torsional PlantŌĆØ - A
system, on its most basic level, consisting of three disks connected to each other via
flexible shafts, and driven via a brushless DC motor connected to the bottom of the
three disks. A picture of the full setup can be seen in Figure 1.1.
The disks are horizontal to the drive axis and set up in a stacked configuration,
with a bottom disk (disk 1), middle disk (disk 2), and top disk (disk 3). These can

the interplay between the disks through the flexible metal shafts.
Due to the flexibility of the shafts, a torque input to the bottom disk will see a
delayed reaction from the top disk.
An interesting challenge may be to control the
bottom disk using only the measurements from the top disk.
Likewise, the flexible
connections between the disks give rise to noticeable vibration in the system when doing
rapid velocity or position changes, due to the interplay between the disks and their
inertia. So, moving into slightly more advanced territory, vibration control could also
be implemented as part of a control exercise or project.

whichever disk one chooses, to change its inertia value, and thus alter the dynamics of
the system. A second motor is included and can be connected to whichever disk one

The 3-Disk System
As mentioned before, the ECP is a system comprised of 3 disks connected to each other

ECP with numbered guides showing the location of each element of the system (with the
disks being numbered ŌĆÖ1ŌĆÖ, and shafts numbered ŌĆÖ2ŌĆÖ). A brushless DC-motor (3) drives
the bottom disk. A second motor (4) can be connected to either of the three disks to

the position, an encoder (5) is connected to each of the three disks, which each have
a resolution of 16,000 ŌĆØticksŌĆØ per revolution, as described in the manual[Par99, p. 65,
section 4.5]. Figure 2.2 shows a close-up of the disk to encoder connection.
Going into more detail, the motor driving the bottom disk is connected via a belt

shaft. Each encoder is also connected to their corresponding disk with belts, but with a

ensures that no slipping occurs under normal operation of the system. Figure 2.3 shows
an image of the bottom assembly of the system, including the belt connecting the drive
motor to the system.
There are a total of seven ball-bearings. One for each disk, one for each encoder, and
finally one at the bottom of the assembly.
As can be seen in Figure 2.1, each disk (1) contains four slits, along with rings
expanding from the center, which mark the distance to the center in 1cm jumps. Weights
can be attached to these slits at various distances to the center, to alter the dynamics
of the system.

2.2.1
Behavior Of The Ideal System
It should be reiterated here that this model is based on the equations provided in the
manual for the system. These equations (and corresponding model parameters) make
assumptions regarding the use of the system, and as such, the user working with this
model is limited by these assumptions.
To repeat, although they are not explicitly
stated, these assumptions seem to be that the system is run at a reasonably high velocity,
which means the friction effects most prominent at low speeds can be simplified to the
stated pseudo-friction biŽēi terms. The effect of this is that for the simulation model
of the system, henceforth referred to simply as the ŌĆØIdeal systemŌĆØ, there is no static
friction. Any torque input to the system, no matter how small in magnitude, will make
the simulated disks react, which is contrary to what preliminary tests on the system
revealed.
To study the behavior of this ideal system, the model is implemented in Simulink.
Here, the response of the system to various open loop inputs can be easily simulated.
The simplest input one can apply to the system is the open-loop step, which would
correspond to the system getting affected by a constant 1Nm torque. What one would
expect to see, based on the equations, is a sudden increase in speed in the beginning,
which will then level out as the acceleration goes to zero.
The results of one such
simulation can be seen in Figure 2.7.
It behaves as expected, based on the equations, with a constant input eventually
leading to a constant velocity. This means the disks spin at a constant speed, therefore
leading to a constant increase in disk position over time, as shown in the plots. It should
be noted that the disk speeds in this simulation reach about 111 radians per second, or
about 17.5 full revolutions per second, which is perhaps a tad fast, so based on this
simulation one can also see that for experiments on the physical system, one should stay
well below 1Nm for torque inputs.

2.3
Software For Controlling The Rig
Going over to the physical system, there are two main ways of controlling it. One is
through the dedicated software provided by Educational Control Products, called ŌĆØECP
Executive ControlŌĆØ, and the other is through Matlab/Simulink, which also works
through drivers provided by Educational Control Products.
2.3.1
ECP Control Executive
The ECP Control Executive software is tailored towards the ECPs, and as such it is
pretty easy to set it up and get the rig running quickly with this. It allows the user
to chose between various control algorithms (PID, State Feedback etc.), and also the
trajectory the user wishes it to take (impulse, step, sine etc.).
ItŌĆÖs also possible to
generate ones own custom reference paths, by writing a file using the specific syntax
mentioned in the manual[Par99, pp. 20-21]. There are a lot of limitations for custom
trajectories though, so itŌĆÖs not ideal. It is also possible to program a disturbance, using
the second motor.
When executing the trajectory with the selected controller (or in open loop), one can
choose to log data from the machine, such as the encoder positions, the control effort,
reference signal and so forth. The user can also specify the units, though the default
selection is ŌĆØcountsŌĆØ, which is the raw input/output to/from the control box.
After this, the compiled data can be plotted and analyzed, although itŌĆÖs quite tedious
compared to Matlab/Simulink. Raw data can also be saved to a .txt file, and be
exported to Matlab or similar.
One final piece of functionality, which is actually quite important even when using
Simulink for control, is the ability to reset the rig, using the Reset Controller option
in the utility menu. This zeros all the encoder positions and disables any control
inputs to the system. This also works even when the control box is switched off. It is
recommended to do a controller reset before operation, so one doesnŌĆÖt risk a controller
trying to drag the disk positions back to zero, after for some reason having spun the
disks a million radians during a previous session.

5.1
Models For Friction
Many models for friction exist, which vary in accuracy and complexity. The simplest of
these is the Coulomb friction model, which only takes the sign of the velocity into account,
so that there is a constant friction torque present regardless of velocity. Combining
this with the viscous friction gives a slightly more accurate model, while still retaining
simplicity. This model is included in the class of friction ŌĆØstaticŌĆØ models. These models
do a decent job of describing the system in steady state (i.e. at some constant velocity),
however they do not take into account the complex dynamics which are present in the
surfaces interacting with each other.
As a response to this shortcoming, the ŌĆØDahlŌĆØ model was developed, which looks
much deeper into the physics of the interacting surfaces, modeling the point interactions
and how they deform under stress before breaking away and sliding. This model belongs
to the ŌĆØdynamicŌĆØ class of models, which much more precisely model the friction, at the
cost of increased complexity. A lot more models have been developed, improving on the
models in both classes of models, including the static ŌĆØStribeckŌĆØ friction model which
takes stiction into consideration as well, and the dynamic equivalent, the ŌĆØLuGreŌĆØ model.
A comprehensive review of friction models is given in [Mar+16]. A more to this project
relevant source is [ADW94], which describes in great detail the models and methods
with which friction is compensated for in machines, and also [LYT13], which relates to
friction compensation for a brushless DC motor.
Although many friction models exist, in this project the simple Coulomb+Viscous
model is chosen, as more advanced models, particularly dynamic models, add a layer of

Chapter 5 Summary
In this chapter, a calculated velocity-friction plot of the system was first shown. From
this plot, the characteristics of the friction in the system could be clearly seen, and
it was decided to model it as a Coulomb+Viscous friction. The viscous friction was
already included in the model, but the Coulomb friction had to be modeled seperately,
with its own parameters. This meant that the friction parameters would need to be
determined, including the b values, which had otherwise previously been identified in
chapter 4. Motivation for this was the fact that the ŌĆØviscousŌĆØ friction value didnŌĆÖt truly
represent the effects of the viscous friction on the system, but a hybrid of both the
viscous and Coulomb friction.
The parameters were estimated by using a Kalman filter to extract the values of
the friction effects from the system dynamics, and implementing a gradient descent
algorithm in order to identify the specific system parameters from this. Having estimated
the parameters for the updated model of the system, this model was then subjected to
a series of tests, comparing measurements from the physical system to the simulated
counterparts, along with simulations from the original model. These comparisons showed
a clear improvement to the model, although it still faltered slightly at low speeds.

within it as well. This will impact the performance slightly, as the Coulomb friction is
then no longer compensated for within the specified velocity window, so thereŌĆÖs a slight
trade-off.

range of ŌłÆ0.5 to 0.5 gave adequate results. That is, once the velocity of the disks fell

Coulomb friction estimation, to avoid ŌĆØhuntingŌĆØ phenomena, is as follows:

7.1
Conclusions
When looking back to the problem formulation in the beginning of the report, it is quite
clear from the results that the main question can be answered with a definite ŌĆØyesŌĆØ.
The 3-disk system was first thoroughly analyzed, though this part of the project
came off to a bumpy start when a handful of unexpected phenomena were encountered.
Systematically analyzing these phenomena and ruling out possible causes, the source
was eventually found, with a bit of outside help, to be overly tight connecting belts in
the system.
Having altered the system, the rest of the project revolved around analyzing the un-
modeled phenomena in the system and modeling this so as to achieve a better simulation
model, and thus, better control. Due to the relative simplicity compared to other friction
model types, it was decided to implement a Coulomb friction model, in addition to the
already modeled viscous friction.
After having updated the model with this friction, the model parameters were identi-
fied. A Kalman filter was used to estimate the otherwise non-measurable friction torques,
and an implementation of a gradient descent algorithm was used to estimate the specific
parameters that were of interest.
With this, the new simulation model was shown to follow the real system quite
a bit better than the original. With basis in this, a new controller was designed for
the system which was shown to have noticeably better performance than an equivalent
controller designed for the original system setup, and with this, the project goals had
been achieved.
In the context of the ECPŌĆÖs original purpose, being used for course exercises by
students, only time will tell whether the results from this project will help them, though
it seems quite likely.

7.2
Future Work
Although a lot of work has been put into this project, there is still room for improvement
and expansion. It would of course be interesting to test other kinds of control with the
system, like vibration control, or fault-tolerant control etc., however this is what the
machine was made for anyways, so this will come naturally as it gets used in coursework.
In the context of project work like this, it would make more sense to dive deeper into
the simulation model. As mentioned in the introduction, the ECP allows for a great
deal of customization, while the simulation model used throughout this project has only
taken one specific setup into account. Given more time, it would be nice to modify the
simulation model such that one can select which configuration of the rig one is using,
and then have the model adapt accordingly. This would at the same time open up for
more possibilities with regards to how the ECP can be utilized for exercises.
Looking elsewhere in the model, the friction model used for this project, while def-
initely useful, was still the most simple model implementable. As briefly gone over in
the report, more accurate, yet complex models exist which could be implemented. This
could perhaps be combined with real-time parameter estimation to further increase the
performance of the system.
