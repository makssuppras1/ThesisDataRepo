1 Abstract 
This report deals with large instances of the University Timetabling Problem by using Mixed Integer 
Programming, and more particularly finding why this problem is so hard while having a rather low 
amount of variables and constraints. 
After the problem is presented and the mathematical formulation written, a small literature review is 
done in order to see how this problem can also be solved. Then, a presentation of the instances of the 
problem will be presented: the instances from the University Timetabling Competition of 2007, and 
some instances created using an instance generator. The method that will be used in this paper 
consists in splitting the problem into two subproblems in order to isolate the problematic constraints. 
Using this method, we can try to generate good lower bounds for the problem. As the first subproblem 
is easy to solve, the focus is put on the second subproblem which includes two of the four soft 
constraints of the problem. This subproblem is solved using some methods including using specific 
cuts, by relaxing the problem, and by using a column generation method. 
All of these methods do not give good lower bounds, but the curriculum compactness constraints are 
found out to be the constraints that really slow down the solving of the problem. Therefore, in order 
to improve the way the University Timetabling Problem is solved, a particular focus has to be set to 
the curriculum compactness constraints. 
Keywords: operations research, mixed integer programming, university timetabling problem, 
international timetabling competition 2007, lower bound, C#, Gurobi.

2 Introduction 
Timetables are an essential part in school life, and has a huge impact on a lot of people. With a good 
timetable, students are able to follow courses they want, and have compact days. Teachers can 
manage their days easily and work in good conditions. Finally, for universities, a good timetable will 
permit a good allocation of resources.  
However, creating a good timetable is not that easy because all the stakeholders have their word to 
say and some of their demands might be incompatible with others’ one. Moreover, some demands 
might be unreasonable and not realistic. Therefore, a good timetable has to find a balance between all 
the stakeholders’ demands. 
2.1 Making a timetable by hand 
But first, let us try to make a timetable by respecting some constraints. In order to make it feasible, 
the problem will be greatly reduced compared to real ones. A time horizon of 3 days will be chosen, as 
well as 4 periods per day, 2 rooms, 5 courses and 2 curricula. Here is the data needed for the problem:

2.2 Automating the process 
As done before, it is possible to make a timetable by hand. One idea is to start with a feasible solution, 
and then try to move or swap courses in order to reduce the objective function. This can be done quite 
easily if there is not a lot of courses, rooms, or curricula. However, this can become a bit trickier if more 
==PAGE:8==
 
 
EXAMINATION OF HARDNESS OF UNIVERSITY TIMETABLE MIP’S 
 
 
8 
 
constraints are added. For example, some universities may want to integrate the concept of buildings 
in the timetable, by adding a penalty if students have to move a lot from one building to another, or if 
two adjacent courses are in two different buildings. When the number of constraints grows, it becomes 
complicated to reduce the objective function, because if we try to reduce it by moving some courses, 
it can also involve some other penalties due to other constraints, making the move not worth it. 
Moreover, even if it is possible to move or swap courses in order to reduce the objective function, it 
also becomes more and more difficult because there are more and more things to take into account, 
and it becomes nearly impossible to try to make a visual representation of it. 
Finally, even if it were possible to find easily some moves or swaps to reduce the penalties, in real 
instances there are a lot of courses to schedule, and this may take a lot of time to make the timetable. 
Moreover, as soon as one variable is changed, the whole timetable has to be made again. For example, 
if a room has its capacity reduced, then some courses have to be moved, and might create conflicts 
with other courses, other teachers or other curricula. If another course is created, it has to be 
scheduled without creating conflicts, so making moves or swaps is unavoidable. 
Therefore, if for some small instances it might not be worth it trying to solve it using a computer, it 
becomes much more practical when the instances are getting bigger, or the number of constraints 
higher. Indeed, by doing so, it is assured that the solution found is optimal and cannot be improved. 
Moreover, it makes it very flexible insofar as even if some variables are modified, the problem can be 
run again quite easily. 
2.3 Why using Mixed Integer Programming (MIP)? 
There are several ways to solve this problem, and one is using heuristics. This has already be done, as 
it can be seen further in the report. These methods provide good results, but the objective function is 
still high.  
As we would like to have an optimal solution for the problem, a Mixed Integer Programming can be a 
good solution. Moreover, if the optimality is not reached, we can still have a lower bound of the 
problem, which can be very useful. 
Finally, by using the MIP approach, we can have more insight about how some constraints or variables 
are making the problem hard, in terms of solving time as well as in terms of bounds. 
Therefore, automating the process with the use of Mixed Integer Programming may help a lot solving 
this problem. However, solving that using the standard way appears to be very time consuming the 
bigger instances grow. It may even not be possible to solve them due to the huge memory demand. 
This leads to the following formulation of the problem: 
Why is the University Timetabling Problem so hard to solve, and is there any way to make it easier? 
In order to answer this problem, we will focus on the MIP approach, and more particularly we will try 
to get good lower bounds. Indeed, the MIP approach is very hard to solve to optimality, and impossible 
for big instances. Moreover, good upper bounds can already be achieved using heuristics. Therefore, 
to reduce the gap between the upper bound and the lower bound, we will try to get the highest lower 
bound as possible. 
In order to do so, we will first present the mathematical formulation of the problem in the part 3. Then, 
a literature review containing some methods to solve the problem will be studied, which will be done

3 Problem definition 
3.1 Constraints 
There are several formulations of this problem, with different hard and soft constraints. The more 
constraints there are, the harder the problem is to solve. There are five formulations, from UD1 to UD5 
(UD stands for Udine, which is the university system the problem is based on). The two most studied 
formulations are the formulations UD1 and UD2. Formulation UD1 is called the Basic Formulation and 
was introduced by Di Gaspero and Schaerf (2003). The formulation UD2 was introduced by Di Gaspero 
et al. (2007). These two formulations were used for the International Timetabling Competition of 2002 
and 2007 (ITC2002 and ITC2007). The last three formulations, UD3, UD4 and UD5 were introduced by 
Bonutti et al. (2012).

constraint, if there is a number it represents the penalty associated to the violation of that constraint 
(making this constraint soft constraint). Finally, if there is a dash, the constraint is not taken into

Here is the meaning of the different constraints: 
 
Lectures: every lecture of every course has to be scheduled, two lectures cannot be hold at 
the same time. 
 
Conflicts: Each lecture has to be taught by one lecturer, and two courses of the same curricula 
cannot be hold at the same time. 
 
Room Occupancy: There cannot be more than one lecture at a time in each room. 
 
Availability: some timeslots are unavailable for some courses. Then, the course must not be 
scheduled at that timeslot. 
==PAGE:11==
 
 
EXAMINATION OF HARDNESS OF UNIVERSITY TIMETABLE MIP’S 
 
 
11 
 
 
Room Capacity: The room capacity should be ensured: if possible, a course cannot be put in a 
room with not enough space. A penalty is added in the objective function for each student 
without a seat. 
 
Min Working Day: Each course has a minimum amount of days in which lectures have to be 
told in order to help students and professors to prepare for lectures before. For example, it is 
not very realistic to put one single course the whole Monday only. A penalty is added for each 
day below the minimum. 
 
Curriculum Compactness (Isolated lectures/Windows): Courses which belong to a same 
curricula should be placed next to each other, in order to prevent gaps in the timetable for 
student who are following a curriculum. A penalty is added each time there is this kind of gap. 
There are two ways of adding the penalty. The “Isolated Lectures” way consists in adding a 
penalty each time a course is isolated, whereas the “Windows” way adds a penalty each time 
there is a gap between two courses of the same curriculum. 
 
Room Stability: Every lectures of a course should be taught in the same room in order to make 
it simpler, both for lecturers and for students. A penalty is added for each room, more than 
one, a lecture is scheduled in. 
 
Student Min Max Load: for each curriculum, the number of lectures should be within a range, 
so the students are not too busy. A penalty is added for each unit away from that range. 
 
Travel distance: Students should have time to move between the different buildings. 
Therefore, if two courses of the same curriculum are taught in adjacent periods and are in 
different buildings, a penalty is added.  
 
Room Suitability: some courses need a special room to be in. For example, a course might 
need a projector, or other equipment. Therefore, a penalty is added each time a course is not 
scheduled in a suitable room. 
 
Double Lectures: Some courses require to have double courses: if there are two lectures of 
the same course one day, these lectures should be adjacent. A penalty is added if it is not the 
case. 
The problem can be very different depending on which formulation is used, and some formulations 
can be much harder to solve than others. The choice of the formulation also depends on where the 
university wants to focus on. 
Moreover, the instances need to be more detailed for some formulations: formulations UD3, UD4 and 
UD5 need more data than formulations UD1 and UD2. 
The formulation UD2 will be used, with a minor modification: the Availability constraint will not be 
used. This formulation has been chosen because it is the one used in the article that will be mainly 
used in this report. 
The goal is to manage to have a feasible solution respecting the hard constraints, and with the less 
violation possible for the soft constraints.

4 Literature review: how can this problem be solved? 
At lot of research has already be done on the University Timetabling Problem, and some Operational 
Research techniques has been tried in order to make it possible to solve the problem in a reasonable 
amount of time.  
4.1 Graph coloring 
De Werra (1997) considers the University Timetabling Problem as a graph coloring problem, with some 
additional requirements. Indeed, if a conflict graph is created, then coloring this graph creates a 
feasible timetable. Each color of the graph corresponds to a specific day and time period, and the 
nodes are the lectures to be scheduled. Then, if two lectures cannot happen at the same time, then 
the corresponding nodes are linked. This can happen if the teacher has to teach the two lectures (and 
cannot be at two different places at the same tine), or if these two courses belong to the same 
curriculum. Then for special cases, results can be used to find a way to solve the problem in a 
polynomial time. However, for most of the real life cases, this approach does not give satisfactory 
results. 
4.2 Heuristics 
Heuristics have been by far the most used technique used in order to solve the University Timetabling 
Problem. For example, Costa (1994) uses the Tabu search in order to solve the problem. Starting from 
a feasible solution, but with a potentially large objective function, the algorithm modifies a bit the 
current solution, and sees if the objective value is lowered. If it is the case, then the modified solution 
is kept, and becomes the solution. However, if the objective function is not improved, then this 
modified solution is put into a Tabu list in order for the algorithm not to choose this solution again for 
a predefined number of attempts. In our case, the solver will choose a modified solution in the 
neighborhood composed by changing the period of exactly one lecture. 
Another heuristic used is a hyper-heuristic simulated annealing, by Bai et al. (2006). The idea is to have 
a set of heuristics, and choose them during the local search. By doing so, it can adapt to a lot of 
situations, as some heuristics may be more powerful in case of high rooms occupancy, and others 
might be more efficient when there are a lot of curricula. This method is similar to the Monte Carlo 
hyper heuristics, by Burke et al. (2012a). 
Some other heuristics can also been applied, such as the genetic algorithm heuristics introduced by 
Goldberg (1989) and used by Adewumi et al. (2009). 
4.3  Branch and Cut 
Branch and Cut is another solution used to solve the University Timetabling problem. Burke et al. 
(2012b) used two types of cuts. The first type contains cuts that are from event or free period patterns, 
whereas the second type regroups all the problem-specific cuts. 
4.4 Non-Operational Research solutions 
Some approaches which are not usually used in Operational research can also be tried. This is the case 
of Lee et al. (2000) who used a software engineering approach to solve the problem. This approach is 
called Task-Based Conceptual Graphs (TBCG). It permits, among others things, a high personalization 
of the problem, by making it easy to add or remove some constraints.

5 Italian approach 
5.1 Why using this approach? 
As said before, because the MIP is very hard to solve, we will focus on the lower bound of the problem. 
The problem is that if we are using standard relaxation methods in order to get lower bounds, they do 
not give good results, as most of the time the lower bound obtained is 0. Therefore, two options are 
possible: either a new relaxation that can provide a good lower bound can be found, or a new 
formulation can first be found, before doing any relaxation. That is the option Cacchiani et al. (2013) 
chose to use. At the end, results are good, as most of the lower bounds of the instances from the 
ITC2007 are improved. 
5.2 The idea, and mathematical formulation 
The main idea that will be used in this report in order to solve the problem is to split the problem into 
two subproblems. The first part of the problem consists in assigning courses in rooms, and therefore 
the two first terms of the objective function are taken into account. The second part of the problem 
takes into account the curriculum compactness and the minimum working days constraints.  
The formulation of the problem now differs slightly with what it was before. Indeed, we need now two 
sets of binary variables. The first set of variables is for the subproblem taking into account the room 
capacity and stability, the second set takes into account the minimum working day and the curriculum 
compactness.

6.2 Instance generator 
Instances provided by the ITC2007 are slightly big, and not very convenient for several reasons. First, 
most of them are impossible to solve in a reasonable time, or even not possible at all due to a lack of 
memory of the computer, making the benchmarking of ideas difficult. Moreover, even if for some 
instances it is possible to find a solution, it will not be easy to interpret the results due to the high 
number of variables. 
Therefore, some very tiny instances have to be created. At first, this was done by creating a C# function 
that takes the number of courses desired as a parameter, and returns a corresponding instance. This 
has been done by: 
1. Generating the number of curricula, lectures, day spreading… These are chosen randomly 
between bounds provided by the instances of the ITC2007. For example, the ratio between 
the number of courses and the number of lectures is between 2.48 (comp12) and 5.4 
(comp11). Therefore, for an instance of 𝑛 courses, between 2.48𝑛 and 5.4𝑛 lectures will be 
created. 
2. Assigning lecturers to lectures. The general idea is to assign one lecturer for each course, and 
for the remaining course assign randomly the lecturers. 
==PAGE:22==
 
 
EXAMINATION OF HARDNESS OF UNIVERSITY TIMETABLE MIP’S 
 
 
22 
 
3. Assigning courses to curricula: this is done completely randomly, with at least one course in 
each curriculum (the minimum found on the `real` instances). 
4. As we do not take into account the rooms, the other variables are not generated. 
At the end, with all these numbers and affectations generated, we have smaller instances that can 
include 10 courses or even less. 
However, due to the high randomness of the affectations, these instances are most of the time not 
representative of the reality of timetables. Moreover, a Python code (that can be found at 
http://code.google.com/p/udinettgen/) generating a more realistic instance has been written by Lopes 
and Smith-Miles (2010). Some functions have to be coded in order to use the data in C# and GAMS, 
but nevertheless, this instance generator will be used. 
The instance generator needs two inputs. The first input is an integer representing the number of 
events, that is to say the total number of lectures. The second one is also an integer, and is the room 
occupation (the number of lectures divided by the number of pairs room/period). The output is a text 
file, in a .ctt format that represents the instance generated.

7 Solving the second subproblem 
7.1 The choice of the programming language 
7.1.1 Using two different languages 
In order to solve the subproblem, we have the choice between two solutions: 
 
Using the modeling software GAMS in pair with the CPLEX solver 
 
Write a C# program, and use the Gurobi solver. 
Both solutions have their advantages and their drawbacks. If we use GAMS, it is quite simple to 
implement the model, and as we will see later, results are better. However, it is more difficult to 
implement some techniques such as column generation using GAMS. Moreover, the data handling has 
to be done outside the software, by using another programming language. For example, it is not 
possible to use the data provided by the ICT2007 because they are not in the right format. It is the 
same for the instance generator. 
On the other side, using C# allows us to have more flexibility regarding the format of our data. 
Moreover, it is possible to add some basic coding, making it more flexible. Therefore, we are able to 
implement a column generation method easily. However, it requires more adaptation than GAMS, it 
is slightly slower, and the debugging can take some time. 
7.1.2 Benchmark of the ITC2007 instances using CPLEX/GAMS or Gurobi/C# 
In order to choose which solution we are going to use more wisely, it might be a good idea to compare 
the performance of both of them. This performance can be measured either by the solving time of 
several instances, or, if it not possible to solve it, by the gap between the lower bound and upper 
bound after a fixed time. 
The instances provided by the ITC2007 are good to benchmark these two solutions because of their 
diversity. Indeed, some of them are very quick to solve, some take some time, and the last part of it 
are not possible to solve in a reasonable amount of time and memory.

Furthermore, a lot of tests can be automated, so the computer can run overnight without having to 
launch a test every hour. Therefore, the duo C#/Gurobi has been chosen to try to solve the University 
Timetabling problem.

disable some types of cuts.

objective value. Therefore, the objective function has to be compared to min𝑢𝑑. 
Nevertheless, problems occur when it comes to the implementation of the column generation into C#. 
Now, the master problem has to be rebuilt every time a column is added because adding a column also 
modifies the existing constraints. Therefore, the whole model has to be rebuilt. If it does not take a lot 
of time at the beginning of the run, it is becoming more and more time consuming as new columns are 
added, making the problem very long to build and to solve. Therefore, this solution has not been 
implemented.

8 Discussion 
8.1 Why is the problem so hard? 
If just the numbers of variables and constraints are taken into account, the University Timetabling 
Problem does not look like a hard problem. However, as soon as the instances are becoming bigger, it 
becomes harder and harder, longer and longer, to solve the problem. When a certain size of the 
problem is reached, it becomes even non-solvable. 
If we split the problem into two using the formulation of Cacchiani et al. (2013), the first subproblem, 
which includes the room capacity and room stability constraints, is not very difficult to solve. On the 
other hand, the y-subproblem, which takes into account the minimum working days and the 
curriculum compactness constraint, takes a lot of time to solve when the instances grow bigger.  
More precisely, as soon as we relax the curriculum compactness constraint, the problem becomes 
much easier to solve. Moreover, as it was already pointed out, the number of curricula affects hugely 
the solving time, and can even make the problem impossible to solve. 
We can also try to modify the penalty associated to the violation of one constraint or another. 
However, for the four instances tested, making one penalty much higher than another is making the 
results worse, and in most cases making the problem not solvable. The reason of this comportment 
has not been found.

Data has been sorted by the solvability of the instance first, and then the percentage of curriculum 
compactness constraints. As it can been seen, the percentage of curriculum compactness constraints 
is low: at most a third of the total number of constraints. This is the same for the percentage of

compactness constraints grows, instances are less likely to be solved. Finally, the percentage of 
variables used seems to have no impact on the solvability of the instance. 
Therefore, the answer to this question, which is the first part of the problematic, is that the curriculum 
compactness constraint makes the problem very difficult to solve. 
8.2 Is there anything that can be done? 
In order to make the problem solvable in a reasonable time, even with bigger instances, the first 
solution that can be applied is also the easiest: remove the curriculum compactness constraints. This 
might seem to be an overreaction, but if we think about it, this constraint might be one of the less 
important one. Even if the penalty associated to this constraint is higher than the one involved with 
the room capacity one, there are more students than curricula. Therefore, the total penalty of the 
curriculum compactness constraint is likely to be lower than the penalty of the violation of the room 
capacity constraint.

Another idea might be to try to find another formulation for adding a penalty when there is a ‘hole’ in 
the timetable. A good formulation would include less variables, and would need fewer constraints than 
the actual one. Another formulation has already be found, counting the ‘holes’ instead of the isolated 
courses (the “Window” constraint that is used in formulations UD3, UD4 and UD5), but the results 
stays the same: too many variables and constraints are used. However, with less variables, or less 
constraints, the impact of this penalty can be greatly lowered, and therefore the solving time can be 
shorter (or if the problem is still not solvable, the gap can be smaller). 
As we have already seen, the number of curricula affects heavily the solving time of an instance. 
Indeed, the number of curriculum compactness constraints is directly related to the number of 
curricula. Therefore, using less curricula will make the problem solve faster. Therefore, universities 
have to find a balance between a good timetable and a fast solving time. 
8.3 What other improvements can been made? 
Now, several constraints are taken into account in the problem: room capacity, curriculum 
compactness… The model can always be extended in order to include new constraints. For example, 
one constraint that can be added is a room constraint: some courses can be taught only in a specific 
room. This can happen if there are for example some experiments to do in a lab, and that cannot be 
made in common rooms. This constraint is part of the Extended Curriculum-based Course Timetabling 
formulation introduced by Bonutti et al. (2012). 
Other constraints can always been added. The problem is that these constraints are making the 
problem more and more difficult, and there can be a kind of constraint that affects the problem in the 
same way than the curriculum compactness. Therefore, each university has to find the right balance 
between their needs, and the computational time spent to solve the University Timetabling Problem.

9 Conclusion 
In this report, we tried to improve the way to solve the University Timetabling Problem, either by 
reducing the solving time, or by reducing the gap between the lower bound and the upper bound. In 
order to do so, the method brought by Cacchiani et al. (2013) has been implemented: splitting the 
problem into two subproblems in order to isolate the problematic constraints. With this method, the 
lower bound of some instances can be improved, and therefore the gap between the upper bound and 
the lower bound can be reduced.  
As the first subproblem is easy to solve, the focus had been made on the second subproblem, including 
the curriculum compactness and the minimum working days soft constraints. Some techniques had 
been used in order to solve this second subproblem such as using some cuts, relax the problem or 
using a column generation method. However, the curriculum compactness constraints are making the 
subproblem very hard to solve, and even finding a good lower bound can be difficult. 
As a conclusion, even if the gap between the bounds or the solving time of the problem have not been 
improved, we know exactly what makes the problem hard, and where to focus the research in order 
to improve the solving of the University Timetabling Problem.
