Abstract (English)
As the amount of products available gets higher and higher, the need for finding a way to navigate
through them all, is getting higher as well. This thesis will address this problem, by looking into
different models, which can be used for a sparse dataset, given by the software company Nordija.
For comparison, the same models will be used on a more dense dataset, to see how important the
amount of data is, when making recommendations. Both of the datasets consists of whether or not
a set of users have seen a set of movies. In this thesis, the products being recommended is therefore
movies, but the models investigated, can also be used for other products as well.
The models addressed in this thesis are two Baseline models; a Random recommender and a Popular-
ity recommender, two Neighbourhood models; using either user-user CF or item-item CF, two Latent
Factor models; Single Value Decomposition and Non Negative Matrix Factorization, and different
Association Rule Mining models.
All the models are evaluated theoretically by using four different offline evaluation methods and by
using a direct evaluation method. The offline evaluation methods are the Mean Absolute Error, the
Root Mean Squared Error, the Hit Rate and the Mean Average Precision at N. The direct evaluation
is made by creating a small test group, and see what they thought about the recommendations
from each of the models. The purpose of this, is to see if the theoretically best model is also the
model performing the best when shown to the users. Additionally, if a recommendation system can
not make recommendations which the users actually want to use, it does not matter how good the
recommendation system is in theory.
Comparing the performance of the models using the sparse dataset, with the performance using
the more dense dataset, all models did significantly better in the more dense dataset. This therefore
showed how important it is to have enough data when computing recommendations.
Additionally, it was seen, that theoretically the Latent Factor models outperformed all of the other
models. When showing recommendations computed by the different models to the test group, instead
the Popularity recommender and the Association Rule Mining model were performing the best. Of
course a lot of potential errors and limitations have to be considered, when making these statements.
But overall this shows the difference between using offline evaluation and direct evaluation, and that
sometimes, what theoretically seems like the best models, are not necessarily the ones performing
the best in practice.

Abstract (Danish)
I takt med at mængden af produkter stiger, stiger behovet for at finde en måde at kunne navigere
blandt alle disse produkter også. Denne afhandling vil tage fat i dette problem ved at kigge på
forskellige anbefalings modeller, som kan bruges til et sparsomt dataset stillet til rådighed af soft-
ware firmaet Nordija. Til sammenligning vil de samme modeller blive brugt på et mere fyldt dataset,
for at se hvor meget mængden af data har at sige, når man laver anbefalinger. Begge datasæt
består af hvorvidt en mængde brugere har set en mængde film eller ej. I denne afhandling er de
anbefalede produkter film, men modellerne dækket vil også kunne blive brugt til andre slags produkter.
Modellerne som vil blive dækket i denne afhandling er to Baseline modeller; en tilfældig anbefaler og
en popularitets anbefaler, to nabolags modeller; som enten bruger en bruger-bruger model eller en
produkt-produkt model, to skjulte faktorer modeller; som enten bruger Single Value Decomposition
metoden eller Non Negative Matrix Factorization metoden og forskellige modeller der benytter sig af
Association Rule Mining metoden.
Alle modellerne er blevet testet teoretisk ved hjælp af fire forskellige offline evaluering metoder og
ved hjælp af direkte evaluering. De offline evaluering metoder brugt er Mean Absolute Error, Root
Mean Squared Error, Hit Rate og Mean Average Precision at N. Den direkte evaluaring er blevet lavet
ved at samle en lille test gruppe, og se hvad de synes om anbefalinger der er blevet lavet til dem ved
hjælp af de forskellige modeller. Formålet ved dette er at se om de teoretisk bedste modeller også er
dem der fungerer bedst i praksis. Derudover, hvis et anbefalings system ikke kan give anbefalinger,
som brugerne rent faktisk vil benytte, er det lige meget hvor god den teoretisk set er.
Når man sammenligner præstationen af modellerne for det sparsomme datasæt, imod præstatio-
nen for det mere fyldte datasæt, kan det tydeligt ses at alle modellerne præsterer langt bedre for det
mere fyldte datasæt. Dette viser hvor vigtigt det er at have nok data, når man skal lave anbefalinger
baseret på dette.
Yderligere kunne det ses at teoretisk præsterede modellerne med skjulte faktorer signifikant bedre
end alle de andre modeller. Når modellerne i stedet blev brugt til at lave anbefalinger til test gruppen,
kunne det ses at popularitets anbefaleren og Association Rule Mining modellen præsterede bedst.
Potentielle fejl og begrænsninger skal selvfølgelig tages i betragtning, når disse erklæringer bliver
lavet.
Men overordnet set viser dette, hvor stor forskel der kan være på at evaluere modellerne
teoretisk og praktisk, og at det ikke altid er den model der teoretisk virker bedst, der også virker
bedst i praksis.

Chapter 1
Introduction
Recommendations come in various forms and shapes. It can be everything from a friend recommend-
ing a book they have read, to web pages recommending items to buy in their shop etc. Recommen-
dations have always been a part of our lives, and with the huge amount of items and content that
has been made available the last couple of years and decades, recommendations are more important
than ever. There is so much content out there, that it is hard for the users to navigate through it
all by themselves. Getting help from friends are also no longer sufficient, since they do not have the
full overview of all the content available either.
One way this problem has been solved is by search engines. The search engines help the user navigate
through all the items by making the user search for something and returning the items that could be
relevant based on the search. But what if the user does not know what he is looking for? Or if he
still does not know how to choose between the shown products? This is where the recommendation
engines come into the picture. The goal of the recommendation engines is to make a personalized
recommendation based on what the user otherwise likes or has looked at before. This should help
the user make a good and fast decision from all the possibilities available.
Besides helping the user, recommendations can contribute to an increase in sales, by making sure
the user is having a great experience. This could potentially make the user buy more, or make
them come back another time. Amazon is an example of this, since a lot of their sales comes from
recommendations. Netflix is highly reliant on recommendations as well, since if a user does not find
anything they want to watch quickly, they will stop using their service [30].
This thesis is created in partial collaboration with the software company Nordija. Nordija is dealing
with TV providers as customers, and their goal is therefore to provide TV viewers with the content
from the TV providers platform in the most user friendly way. Therefore, recommendations is an
interesting topic for them, since giving recommendations to the TV viewers will make it easier for
them to find something to watch from the platform.
This thesis will explore some of the various recommendation models out there, to see which will work
on a sparse dataset provided by Nordija. It will also try to cover some of the advantages and dis-
advantages of the different recommendation models and see how a more dense dataset impacts the
recommendations. To see how well the recommendation models are performing on the two datasets,
different ways of evaluating the recommendation models will be investigated as well. The focus of
this thesis will be recommendations for movies, but many of the models covered will work for general
items or products as well.

Chapter 2
Literature Review
Before going in depth with the theory and the models implemented, an overview of already existing
recommendation systems and their history will be given in this chapter. This is to get a better
understanding of the area of recommendations and why recommendations have become so important.
Additionally, it will give an overview of which models could be relevant in this thesis, and what is
the state of the art regarding recommendation systems.
2.1
General recommendations
Recommendations can be given for everything; books, movies, clothes, restaurants etc. Therefore,
first an overview of the history of general recommendations will be given in this section, since this
gives an idea of when and what created the need for recommendations.
2.1.1
Tapestry
One of the first recommendation systems invented was Tapestry created by the Xerox Palo Alto
Research Center in 1992. Back then, emails were becoming a greater part of the communication and
distribution of information within organizations. This led to employees getting a huge amount of
emails, which could be hard to search through to find the relevant ones for the individual employee.
One solution was that the employees should subscribe to mailing lists relevant for them, to reduce
the amount of emails. This reduction was still not enough, since many irrelevant emails still got
through to the different mailing lists, and interesting emails from other mailing lists could still be
missed [11],[14].
Tapestry was therefore invented to solve this problem and help the employees with the overload of
emails sent to them. A filtering system was created, where the users could specify which emails
are relevant for them. This could for example be that the user specifies all emails that a particular
employee likes, is relevant for the user. By doing this, they created a Collaborative Filtering environ-
ment, where the users help each other filter through which emails are interesting and which are not.
Another filter the user is allowed to apply, is a keyword based filter, where the user specifies that
emails containing one or more specific keywords are of interest to them. This filter is looking more
into the content of the emails, creating a Content Based environment. Lastly, these two filters can
be joined as well, specifying emails with a certain keyword liked by a specific employee are of interest
to them [11].
One of their key takeaways was that the filter got more effective by having the users be a part
of the process of creating the filters, instead of only having predefined keywords or relevant users.
Thereby, they created one of the first recommendation systems using a mix of keyword based rec-
ommendation and the interest in emails from other employees, creating the base for Content Based
Filtering and Collaborative Filtering respectively [11],[14].

2.1.2
GroupLens
A little later in 1994, the GroupLens research team developed a recommendation system for rec-
ommending articles and news in newsgroups. The idea is reminiscent of what Tapestry was doing,
and a lot of inspiration came from here as well, but with one major change; instead of just using
the information and opinions inside one given site, GroupLens shared the information across sites to
make the accuracy of the recommendations better [14]. GroupLens is a research group located at
the University of Minnesota. They have worked, and are still working, on many different projects.
To read more about GroupLens and their projects see [28].
To recommend articles to the users, GroupLens use the users ratings or opinions for previously read
articles. They use the ratings from the user, to find other users to whom they are similar. When the
similar users are found, they predict articles based on what the similar users think of these articles.
This approach is called user-user Collaborative Filtering and will be explained in more detail in Sec-
tion 4.3 [14],[24].
GroupLens is considered the first modern recommendation system, since the approach they used,
is still used in a similar manner in many recommendation systems today [14].
2.1.3
Amazon
A more recent and very successful recommendation system, is the one used by Amazon. Amazon.com
is an online shop, where users can find a huge amount of different products. Since Amazon has so
many products, it was an important step for them to implement their recommendation engine. In
fact, 35% of their revenue today is estimated to come from recommendations [30]. Since they have
a huge amount of products and users, scalability of the recommendation system is very important.
Scalability of the recommendation system defines the ability to take in a huge amount of users and
products, and still produce recommendations relatively fast [26],[30].
To overcome the scalability problem, they created their own system using item-item Collaborative
Filtering. This is similar to the recommendation system used by GroupLens, but instead of making
similarities between users, item-item Collaborative Filtering makes similarities between items. This
approach will be explained further in Section 4.3. By using the ratings for finding similarities between
the products, Amazon are able to recommend different types of products, even though the user are
not looking at or have bought that type of product before [26],[30].
With their recommendation engine, Amazon shows recommendations almost everywhere on their
web page. The different types of shown recommendations includes the following [30]:
• Your recommendations; This is an overview of products that Amazon thinks the user is likely
to click on and perhaps buy. The list is created based on the user’s previous history.
• Frequently bought together; Here, products often bought together with the product the user
is currently looking at, are listed. This is meant to increase the amount of products bought by
the user.
• There is a newer version of this item; This is showing the user a newer version of a product
previously bought, since many users are prone to wanting the newest products available.
• Best selling item in this category; Here a user will be presented to products that the user might
not have looked at otherwise. This is showing the most popular products in a category, telling
the user that other users buy this, so maybe you should too.
Only the first recommendation is based on the item-item Collaborative Filtering, whereas the others
are looking into the popularity and patterns of the products. In this way, Amazon is making sure
that the users have recommendations available all the time, and therefore have a great experience
browsing Amazon. It shows how important recommendations, and how these are shown, can be for

2.2.1
Demographic Filtering
Demographic Filtering gives a general recommendation to all users based on some kind of demo-
graphic. This is therefore a simple way of giving recommendations, that needs only little information.
One recommendation, computed by a Demographic Filtering, could be recommending the most pop-
ular items in general. Even though this is a very simple way of recommending items, to some extent
it works very well, since a popular item will have a higher probability of being liked by many. Another
way is by combining the popularity of the item, by some other demographic feature. This could be
recommending the most popular items based on a keyword, an age group, a genre etc. The demo-
graphic feature can be almost anything that is available, and of course also depends on the kind of
item recommended. [3],[4].
Even though Demographic Filtering can be effective, especially when considering how simple it is,
more advanced recommendation systems are needed, for making more personal recommendations.
Additionally, something else have to be recommended as soon as the user buys or use all the popular
recommendations, such that the recommended list does not end up being empty.
2.2.2
Content Based Filtering
In Content Based Filtering (CBF) the content of the items are used for making recommendations.
This could for example be the content of a book, document, movie, or anything else with some kind
of textual information. The Content Based Filtering is using this information to create a keyword
based vector for each item, with a weight for how important the different keywords are to the item.
In this way, a profile for all the items can be created. How important each keyword is for each item,
can be computed by using the TF-IDF, which will be described in Section 2.3.2. The TF-IDF is the
term frequency times inverse document frequency, and computes how often a keyword occurs for the

==PAGE:14==
item out of all the keywords, taking into account the number of items with the keyword in it [16].
A user will then be recommended the items, where the item profile is most similar to the item profile
of previously liked items. Another way to recommend movies, is by creating a user profile. This
user profile is a keyword based vector as well, with weights for how much the user likes the different
keywords. This user profile can then be matched with the movie profiles, to find the ones that are
most similar to the user’s preferences. The similarities between two movie profiles, or between the
user profile and the movie profile, can be found by using different similarity measures. Such measures,
will be described in Section 4.3.1 [14],[16].
Content Based Filtering often makes good recommendations for the users, since it is recommending
items which are very similar to previously liked items. Unfortunately, this also makes the recommen-
dations quite predictable, since no surprising recommendations are made based on similar content.
Another problem is that CBF requires some kind of textual information about the items. If no infor-
mation exists, it has no base for creating the user or item profiles. Additionally, if a new user enters
the system, it can not build a user profile, since the user has no information about any previously
liked items. Though, as soon as the user likes one item, the similar items can then be recommended,
based on the movie profiles created. These problems will be further discussed in Section 2.2.6 [16].
Another way to make recommendations, where the taste of the user is taken into account is therefore
needed, which can be used if no textual information is available [16].
2.2.3
Collaborative Filtering
Collaborative Filtering (CF) is giving a recommendation based on a user’s behavior and previously
rated movies, and does not create a profile for the user or the movie in same manner as CBF does.
Collaborative Filtering is very popular since it can find unexpected recommendations for the users,
by only using the rating patterns, and therefore does not need any textual information about the
items. Collaborative Filtering can be divided into two methods; Memory Based CF and Model Based
CF [16].
Memory Based Collaborative Filtering
Memory Based Collaborative Filtering is giving recommendations based on which users are similar
to you, or which items are similar to the items you like, based on the ratings. This is user-user
Collaborative Filtering and item-item Collaborative Filtering respectively. Different models can be
created based on Collaborative Filtering.
One of the most popular models is referred to as the
Neighbourhood model. The idea behind the Neighbourhood model, is to create a neighbourhood
consisting of the most similar users or items. The ratings from the neighbourhood is then used for
making a recommendation. In this way, the recommendation for a given user is either based on the
users with the most similar taste to this user, or the items which is most similar to items the user has
previously liked. More details about this approach is given in Section 4.3. By using Memory Based
CF, the taste of the user is taken into account, since the user’s rating pattern for different types of
items, with different kind of content, is found [16].
The Memory Based CF has problems with sparsity, which means it needs many ratings in order
to make good recommendations, which is not always available. Additionally, it does not scale very
well, since computing e.g similarities for users or items can be very time consuming, if the number
of users or items is large. Using Model Based CF instead, can help solve this problem [4],[16].
Model Based Collaborative Filtering
Model Based Collaborative Filtering is finding patterns based on ratings as well. But this time,
instead of using these patterns to find similar users or movies, the rating patterns are used for finding

latent factors explaining these patterns. These models are therefore often referred to as Latent Factor
models. The patterns can be learned by e.g using machine learning or by using Matrix Factorization.
The latter is highly applied, and is especially used for solving the scalability problem. Generally, the
Matrix Factorization model works by using the rating matrix R for finding the latent factors, and in
this way create two new and smaller matrices M, describing the item characteristics by these factors,
and U, describing the user preference for these factors. The latent factors represent some kind of
information about the items. Therefore, in a way, this is similar to the Content Based Filtering, but
instead of making user or item profiles based on textual information, the profiles are based on the
latent factors. The Latent Factor model will be explained in more detail in Section 4.4 [16],[38].
Even though Model Based CF is partially solving the scalability and sparsity problems, this model has
its own problems as well. Both of the Collaborative Filtering methods are both limited by the cold
start problem. The cold start problem is defined as the problem that occurs when a new user or item
enters the system. When this happens the recommendation system can not recommend anything,
since the user or item does not have any ratings, and no similarities or patterns in the ratings can
therefore be found. The cold start problem is further discussed in Section 2.2.6. Therefore, a way
to get around the cold start problem is needed [16].
2.2.4
Social recommendation systems
Besides looking at the user or item relations based on the ratings, a Social recommendation system
can be created by taking the social relations between users into consideration as well. One way to
define a Social recommendation system, is by any system taking the social relations as an input. The
idea behind a Social recommendation system, is that people who have a social relation will likely
have correlated preferences. Often Social recommendation systems will therefore be combined with
Collaborative Filtering. The social aspect can be added to the CF system, by only using the ratings
from users which the given user has a relation to [16].
By using the relations between users, the cold start problem can to a certain extent be avoided.
When the social relations for a new user is known, the ratings for this user is not needed. The similar
users will therefore be the users with a relation to the given users. With the increase in the use of
social medias, a lot of data is available. Using social recommendations system could therefore both
overcome the cold start problem, and additionally add data to the system, when there is a lack of
ratings [16].
Using social recommendations therefore seem to be solving a lot of problems, while also poten-
tially being an effective recommender. One though have to keep in mind the personal data rights
for the users, discussed in Section 2.2.6. Using social relations are highly sensitive data, and can not
be used without the permission from the users, and therefore computing social recommendations is
not always possible [15].
2.2.5
Hybrid recommendation systems
Another way to get around all the problems each recommendation system can have, is by making a
Hybrid of the models. The idea behind a Hybrid, is to use two or more models in one approach, to
make a better recommendation system than each of the systems individually. There are different ways
of creating a Hybrid. One common Hybrid is between the Content Based Filtering and Collaborative
Filtering. These can be combined by using CBF in some cases and CF in others, where they will
each perform the best. This could for example be when a user do not have any ratings yet. To
recommend items for this user, CBF can be used to recommend based on the textual information.
When the user then starts rating items, CF can instead be used. Another approach is getting results
for the models separately, and ensemble these results for one final prediction. An example for this,
will be shown in Section 2.4 [16],[14].
No matter the approach, and which models are used together, the idea of the Hybrid recommendation

2.2.6
Typical challenges
The mentioned types of recommendation systems, all suffer from different challenges and limita-
tions. Some are already mentioned, but the following will sum up some of the common challenges
recommendation systems can have, and what one have to consider when using the different types of
recommendation systems.
Predictability
As mentioned in Section 2.2.2, the recommendations computed by Content Based Filtering can be
quite predictable. This happens because the CBF model is recommending items similar to previously
liked items. Additionally, CBF is based on the textual information available. This therefore means,
when recommending e.g books, the same author as previously liked books could be recommended or
books with the same type of plot. This is applicable for all sorts of items, and the recommendations
are therefore rarely surprising when using CBF or other text based recommendation systems. Some
recommendation systems need to be applicable for different types of items. In this case CBF can not
be used, since the keywords for a book and a restaurant are not similar at all. In some cases when
e.g a movie is based on a book, this book could be recommended based on the movie [14],[16].
Another problem by using textual information can be the subjectivity of this text, explained next.
Subjectivity
In Section 2.2.2, it could be seen that making recommendations based on Content Based Filtering,
need some kind of textual information. This could be a description, keywords etc. of the item. One
therefore have to keep in mind the subjectivity of the text, since different problems can occur when
using textual information. First of all, many item descriptions are created for the purpose of selling.
This means the descriptions are biased into making the product sound in a specific way, to make
you buy the product. Another problem can be misleading descriptions or descriptions written in a
different time period. [25] is a great example of this. Here different movie descriptions are given,
which are correct, but misleading the plot of the movie. The misleading description will then be
used to compare different descriptions, leading to mistakes when finding the similarity between the
descriptions [7].
Recommendation systems based on text, suffers from the text describing the item, both in the
sense of subjectivity and predictability, and it is therefore important to be critical about the textual
information used.
Scalability
The Neighbourhood model mentioned in Section 2.2.3 is an example of a recommendation model
that suffers from the scalability problem. A scalable recommendation system, can take in a growing
number of users and items, without it sacrificing the computational cost. Many companies and plat-
forms using recommendation systems, have a huge amount of users and items available. Therefore,
the scalability of the recommendation system is very important. This was e.g seen for Amazon in
Section 2.1.3 [4],[16].
Recommendation models with a lot of heavy computations, such as the Neighbourhood model, will
suffer from scalability problems, and this has to be taken into consideration when choosing these
models. One way to get around this problem, is by making sure the computations does not have to
be computed every time a new rating gets into the system [4].

Sparsity
A problem most recommendation systems suffers, is the sparsity problem.
All recommendation
systems need some kind of information, whether it is ratings, text or information about the users or
items. Getting the information needed, is not always easy. First problem which is applicable to all
recommendation systems, is the GDPR rules, covered shortly. Because of the GDPR rules, the use
of some kind of data, needs the permission of the users. If the users does therefore not allow the use
of their data, the recommendation systems will suffer from lack of data [15].
Another problem is that many users only buys or use a very limited amount of the products out of all
products available. Even if a user is very active, it does not necessarily mean that the user is rating
the products bought or used. This is a problem for recommendation systems requiring some kind of
rating, since the more ratings are available, the better the recommendations. Nor is it easy always
getting the textual information needed for the text based recommendation systems. Some products
do not already have a description or keywords, and if they do, these descriptions and keywords are
not necessarily adequate or could be misleading as described before [4],[16].
Cold start problem
Even if all the above mentioned problems are dealt with, what happens when a new user or item enters
the recommendation system? This problem is called the cold start problem. All the recommendation
models need some kind of user preference to be able to recommend items for the user. Even text
based recommendation systems need the user’s preferences to use the textual information from these
items, for making recommendations. Collaborative Filtering needs, besides ratings given by all users,
ratings given to all items, to be able to recommend all items [14],[16].
One way to get around this problem, is asking the new user to rate some items, or simply recommend
the new items because of its novelty. This is not always possible, and not every user wants to rate
items before being able to use the platform [14].
GDPR
In may 2018, the General Data Protection Regulation (GDPR) law became applicable. Overall the
law is meant for uniforming the data protection laws throughout Europe, and make sure that people
have more control over their personal data and how it is used. Personal data can be all kind of
information as names, emails, IP addresses etc. Because of the GDPR law, many companies had
to revise how they collect and save personal data, and using recommendation systems for gathering
data is no exception. For more information about GDPR see [9].
Recommendation systems have always been limited of the laws regarding personal data, but after the
GDPR law was established, this limitation has only been greater. When making a recommendation
system, one therefore have to think about what data is allowed to use, and if the users will allow
their data being used for recommendation purposes. As mentioned above, the more data used for
the recommendation systems, the better the recommendations. If the users will not share the data,
it does therefore not matter how good the recommendation system is, since it will then have no data
available. What can be done is e.g anonymize the data, such that the identity of the users can not
be found. At the end of the day, the users have to allow the use of their data, and if they refuse, no
recommendations can be computed for them [9],[15].

2.3.1
Demographic Filtering
As described in Section 2.2.1, Demographic Filtering is making general recommendations based on
some kind of demographic trend. In [3] and [5] the overall popularity of the movies is used as the
trend. One way to generate the popularity score, is by computing the simple average of the ratings
for a movie. A more fair way of getting this score, is by also weighting the score with how many
have seen the movie. In [3], the IMDB’s weighted rating is used, defined in Equation (2.2).

The first term weights the average rating of the movie, while the second term weights the overall
average. Both is taking into account how many have seen the movie, such that if many more have
seen the movie than the minimum number of ratings required, the movie’s own average will be

minimum number of ratings required, the overall average will be weighted the highest. In [3] m is

movies in the dataset. Now that the score for a movie is defined, this score is simply calculated for
all movies with more ratings than m. Afterwards they are sorted in decreasing order to see which
movies are the most popular and can be recommended to the users [3].
[5] describes the same Demographic Filtering system, but also suggests a model where the most
popular movies are divided into genres. In this way, if a specific genre is wanted, the most popular
movies in this genre is recommended, and can show movies that would not have been shown for the
general most popular movies.
As mentioned in the beginning, Demographic Filtering makes a generalized recommendation to
all users, and does not take the user’s taste into consideration. Therefore, [3] and [5] also looks at
Content Based Filtering.

2.3.4
Hybrid recommender
The last approach [5] is using is a Hybrid of Content Based Filtering and Collaborative Filtering. By
using CBF for the movies, the most similar movies based on the content are found. These movies are
then ranked using CF, by finding the predicted ratings for the user for these movies. If two users have
liked the same movies, by only using CBF, the same movies would be recommended for both users.
But by using CF for the movies computed by CBF, the two users get different recommendations,
based on their ratings.
Making a hybrid of two or more recommendation systems is common and often gives the best
recommendation systems, since the best from each model can be used, and the limitations for the
different models can be overcome. As described in Section 2.2.5, there are many ways to combine
the different models into one. Another example for this, is looking at the winning solution for the
Netflix Prize, explained in the next section [16].
2.4
The Netflix Prize
Netflix is a streaming service that connects movies and series with its users.
Since the amount
of available content at Netflix is huge, recommendations are an important part of their platform.
Their recommendation system is therefore more advanced than the basis recommendation systems
just described. To make their recommendation system even better, in 2006 Netflix announced a
competition to beat their current movie recommendation system CinematchSM. Netflix shared their
dataset with the public and brought participants together from all over the world with a common
goal [21].
2.4.1
The data
The training data given to the participants consisted of data collected from October 1998 until
December 2005, with 100,480,507 ratings on a scale from 1 to 5, given by 480,189 randomly picked
and anonymous costumers for 17,770 movies. Besides the training set, a probe set was given with
1,408,395 ratings. Additionally, a qualifying dataset consisting of 2.8 million pairs of costumers,
movies and rating dates, split randomly into a 50% test set and a 50% quiz set, was provided. The
quiz set gave a feedback on the leaderboard, whereas the score on the test set was hidden. The
actual ratings were not given in this set, and the challenge was therefore to predict these ratings.
In [1] they have visualized this division of the data, which can be seen in Figure 2.1. Here it can
visually be seen how the ratio of data is in the training set versus in the qualifying set [21].
The Root Mean Squared Error (RMSE), which is the difference between the true rating and the
predicted rating seen in Equation (4.22), were computed for the quiz and test set to see how well
it predicted. The goal was to beat the CinematchSM RMSE at 0.9525 with at least 10%, which
corresponds to a RMSE at 0.8572, to win the grand prize of $1M. Besides the grand prize, a $50,000
progress prize was offered each year to the team with the currently best RMSE, to make sure people
stayed motivated [21].

The winner
The winner of the competition was BellKor’s Pragmatic Chaos, which was a team combined of the

The solution
In the papers [1], [17] and [22] the team BigChaos, BellKor and Pragmatic Theory respectively
describe their contribution to the combined solution that won the grand prize. An overview of their
combined solution will be given, but for a more detailed and deeper understanding of the models,
their papers [1], [17] and [22] are referred to.
Their overall solution was to find many different models that solve the problem in different ways, and
combine all these models by blending to get a final prediction. The word blending was introduced
by the winning team and is normally called ensemble methods. First an overview of the models used
will be presented, and afterwards their blending method will be described.
Models
The overall sense of the models used for the solution will be described. It is important to state that
this is only an overview for each type of model, and different versions was used for the final solution.
• Baseline predictors: The baseline predictors look at user or item rating habits, and can be
described as the biases of a user or an item. This could be users’ habits of rating movies either
higher or lower than others, or tendencies for some movies to either be more or less popular
than other movies. The basic baseline prediction can be seen in Equation (2.5) [17].
Rui = R + bu + bi
(2.5)
R is the overall average of the ratings, bu and bi is the deviations from the average rating for
the users and items respectively. The baseline prediction is computed by first initializing the
prediction as the overall average of the ratings. The user u and movie i biases are then added
to this initial prediction. The biases indicates how the user rates and how the movie is rated,
compared to the overall average. In this way a baseline prediction can be computed, and can
also be extended [17].

• Neighbourhood models: The next model used is the Neighborhood model, introduced in
Section 2.3.3, which is one of the most popular approaches in recommendations. The dataset
consists of many users, but the users do not have that many movies in common. Therefore,
they used item-item Collaborative Filtering, since this model computes the similarity based on
the movies, which will be more exact because the movies have more ratings. More about this
model and how it works can be seen in Section 4.3 [1],[22].
• Matrix Factorization: In Section 2.3.3 Matrix Factorization was introduced. As an extension
to the SVD model already explained, they also used the SVD++ model. The standard SVD
model creates, a matrix M consisting of the latent features for the movies, and a matrix U
with weights for how much the users like the latent features. The SVD++ model incorporates
this, but also includes the information given by the implicit ratings. An implicit rating, is a
rating which is not directly given by the user, but by the behaviour of the user, since this gives
implicit information about what the user thinks of the movies. The idea is, that when a user
has rated a movie, the user will likely like this movie more, than a movie not rated. By using
the implicit ratings as well, more ratings are available when making the recommendations [17].
• Restricted Boltzmann Machines: Another way of computing the latent factors, is by using
Restricted Boltzmann Machines (RBM). RBM use a neural network to discover the latent
features. The visible layer of the neural network consists of the ratings, and the hidden layers
are then the latent features the network is trying to learn [17],[22].
• Temporal effects: In many of the above models, temporal effects have been added to improve
the quality of the models. The reason for this is that ratings often get less relevant over time,
or that some movies get more attention in some periods. Therefore, in some of the models
a temporal effect have been added, to weigh more recent ratings higher than old ratings, and
improve the performance of the models [17],[22].
Another temporal effect used is what they call the frequency measure. This measure describes
how many ratings a user has given on a day. The idea behind why this measure is so important,
is that if a user has only given one rating on a given day, the movie rated is probably recently
watched. On the other hand, if a user has rated many movies on a given day, many of the
movies are probably watched a while back, and are therefore less relevant [17],[22].
Tuning parameters
During all these models, adjustments of parameters was also needed. The goal of parameter tuning
is to minimize the RMSE of the model. One way is to randomly change a parameter at a time,
and see if this changes the RMSE for the better. Therefore, the probe set was removed from the
training data, the model trained on this data and the parameters changed, until the RMSE on the
probe set was minimized. The found parameters was then used for another training with the probe
set included in the training data, to get the predictions for the quiz set [1],[2].
Blending
As the team said, the key for performing well and get competitive results, was to have an efficient
way of blending the models to get one solution. Some of the models described may not give the
best RMSE score individually, but contributes to a lower RMSE in a blend. At first they used Linear
Regression to combine all their models, but in their final solution they found that Gradient Boosted
Decision Trees (GBDT) performed better. GBDT works by, in a forward step wise manner, using an
ensemble of decision trees on the current residuals. Some of the advantages of GBDT are that they
are computationally robust and have a high scalability, and therefore works great on large data [17].
In this way BellKor’s Pragmatic Chaos combined all their models into one prediction, which could

4.1
Notation
As described in Section 2.2, when making recommendations, both users and items are needed. In
this thesis items are movies and the matrix R is therefore a rating matrix consisting of the ratings
given by users for each movie. Ru is the average rating for a user u and Ri is the average of the
ratings for an item i. A neighbourhood consists of the users, also called the neighbours, that are
most similar to a given user, or the items that are most similar to a given item. kNN is the k nearest
neighbours, consisting of the k most similar users or items for a given user or item.
One of the models in this thesis, uses a set of seen movies by a user, as a transaction or a pur-
chase. One transaction defines all the movies seen by a user, and the number of transactions T is
therefore equal to the number of users. Interesting rules can be computed from the dataset, describ-
ing some kind of relationship between movies, based on information from the transactions. A set of
such movies are called an itemset S, where the number of movies in each itemset is the cardinality.
The cardinality, when considering interesting rules of an itemset, should at least be 2, such that one
of the movies is seen by a user and the other movie can be recommended for the user.
Generally, the recommendation models can be used for all kinds of items, but in this case, since
this thesis is about movie recommendations, an item is a movie.

will be examined in this thesis; the Popularity recommender and the Random recommender. The
Baseline models, are simple methods for recommending movies. The purpose of these recommenders
is to create a baseline for what a more advanced recommender should as a minimum be performing
better than.

4.2.1
Popularity recommender
The first simple recommendation system implemented, is the Popularity recommender. A version of
a Popularity recommender is described in Section 2.3.1 by using the Demographic recommendation
system. The purpose of the Popularity recommender is to recommend the most popular movies to
the user, which the user has not seen. This therefore does not give a personal recommendation to
the user, but will often perform very well since people are very influenced by the opinion of others.
This can both be in the sense of popular movies being highly advertised giving people a feeling of
familiarity when looking for a movie to watch, or simply telling people that this movie is very popular,
making them believe they should watch it as well. Besides this, popular movies are popular for a
reason, and if many people have seen the movie, there is a good chance that you will want to as well.
This is also the reason why Amazon is showing the best selling items, and why Netflix is showing
the most popular movies, making people want to buy the item or watch the movie, simply because
others are doing it [20].
A Popularity recommender can therefore be a powerful recommendation system, and it is used in
this thesis as a baseline for what the more advanced recommender systems should be able to beat.
In Section 2.3.1 the data consists of ratings, and therefore it makes sense to use both the average
of the movie itself and the average of all the movies. Since the dataset used in this thesis consists
of whether or not the users have watched the movies, this average does not make sense. Therefore,
the movies are simply ranked based on how many users have watched the movies. The movie which
most users in the dataset have seen, is therefore the most popular movie.
4.2.2
Random recommender
The next baseline model which was implemented, is the Random recommender. The Random rec-
ommender picks out a list of random movies as the recommendation for the user, which the user has
not already seen. Doing this, means that no prior knowledge about either the movies or the users
are used and it is therefore not giving personal recommendations, like the Popularity recommender.
A more advanced recommendation system should therefore always outperform the Random recom-
mender, since picking out a random movie among all possible movies, should perform very badly.
The Random recommender is therefore implemented as the very lowest bar for the performance of a
recommendation system.
4.3
Neighbourhood model
The first more advanced model implemented is the Neighbourhood model. In the everyday life, people
will often talk about the movie they watched last night, or a series they are currently watching. This
is a low level recommendation given by another person, where you have a feeling about how similar
their taste is to yours, and thereby how good the recommendation is. As the amount of movies
available increases, it is not enough to get input from a few people anymore, since they can not
keep track of the amount of movies either. Due to this, another way to get recommendations is
necessary. The Neighbourhood model solves this problem, while also keeping the idea of the low
level recommendation in mind [26].
The Neighbourhood model is, as said in Section 2.2.3, a Collaborative Filtering model and therefore
uses the behaviour and previously rated movies for the users. More specifically the Neighbourhood
model is based on Definition 1 [8],[26].
Definition 1 (The Neighbourhood model)
The Neighbourhood model is based on the following two assumptions [8],[26]:
Assumption 1: You will like items that users similar to you like.
Assumption 2: You will like items that are similar to previously liked items.

The first assumption in Definition 1 is the basis for user-user Collaborative Filtering, while the second
assumption is the basis for item-item Collaborative Filtering. In both cases, the matrix R, consisting
of ratings, is needed. Using this matrix, the Neighbourhood model will be able to predict what a
user will think of the movies, the specific user has not rated. This process is divided into three
sub processes; finding similar users or movies, finding the neighbourhood and in the end making a
recommendation [26].
4.3.1
Finding similar users or movies
For user-user and item-item Collaborative Filtering the goal is to find the users or the movies, that
are most similar. This is done by finding the correlation between a user or a movie and all other users
or movies respectively, which gives a user or movie correlation matrix. The correlation is a measure
of how correlated two users or two movies are, and therefore how similar they are. There are many
different ways of calculating the correlation. Three correlation measures widely known and used, will
be described here [26].
When computing the similarities between users or movies, it is important to consider how many
movies or users respectively they have in common. Normally, If two users have only one movie which
both have seen, but they have rated it the same, it will look like the users are very similar. This is
because normally only the ratings for the movies seen by the users is included. In this thesis, since
all movies which the users have not seen is included, this is not a problem. For each user or movie a
rating vector can therefore be constructed, consisting of the ratings, which are either 0 or 1 [4],[26].

4.3.2
Finding the neighbourhood
No matter which similarity measure is used, the next step is to find the neighbourhood for the user
or movie. The neighbourhood consists of the users or movies most similar to the one at hand. As
for the correlation measure, there are many ways to compute this, but the following two approaches
are highly used.
The first approach is the k nearest neighbours. This approach sorts the correlations in descend-
ing order and then picks the k highest correlations. This will give k neighbours to the user or movie.
The problem with this approach is that it is uncertain how similar the users or movies in the neigh-
bourhood is going to be. This can result in the recommendation being poor if the neighbourhood
consists of lots of neighbours with low correlations.
Another approach is to make a threshold for the value of the correlations used in the neighbourhood.
In this way, the similarity of the neighbours are forced to be above a certain value, and makes sure
all neighbours will be similar. The problem with this approach is that there is a risk of getting no
or few neighbours in the neighbourhood, if the threshold is chosen too high, resulting in the recom-
mendation being based on only few, or even no, other neighbours [4],[26].
In this thesis, the first approach will be used to define the neighbourhood. This is to make sure
that there will be enough neighbours, since the data is very sparse, and it will be likely that not many
users or movies will be very similar. Instead, different number of neighbours will be used, to see how
many neighbours is the optimal amount for making the best recommendation.

4.5
Association Rule Mining
A model not previously mentioned is the Association Rule Mining Model, which is more of a clustering
recommender. Association Rule Mining is concerned with finding rules of interest in a dataset. One
form of Association Rule Mining is called Market Basket Analysis, and it is the one that will be
addressed in this thesis.
Market Basket Analysis is looking to find rules in datasets concerning
purchases of an item by a customer, also called a transaction. This can be made into the rating
matrix, where 1 and 0 shows whether or not the customer has bought the item. Using Association
Rule Mining for making recommendations is divided into two processes; finding interesting rules from
the dataset, and using these rules to make relevant recommendations [6].
4.5.1

All the measures are somewhat similar to the Popularity recommender. The support, confidence

By using these three measures, interesting rules from the dataset can be generated. There are many
different ways of generating these rules by help of the three measures. One of the simplest ways of
doing this, is by doing the following two steps [6]:
1. Generate all itemsets S1 ∪S2 where the support for the itemset is higher than a defined
threshold, and the cardinality is at least two.
2. For each of these itemsets, generate all possible association rules, and keep the ones where the
confidence or lift is higher than a defined threshold.
This approach is the same as clustering, and therefore cluster movies together, which have something
in common. In this case, the thing in common is which movies often appears together in a transaction.
Especially the first step is critical, and is very dependant of the total number of items n. Generating
all possible itemsets given n items, gives 2n itemsets in total.
To get only the itemsets with a
minimum cardinality of 2, will give 2n −n itemsets. Lastly, removing the empty set gives 2n −n −1
itemsets. Therefore, if n is large, so will the number of itemsets generated be. Instead, the Apriori
algorithm can be used, which is a more efficient way to find all interesting rules in a large dataset
[6],[33].
4.5.2
Apriori Algorithm
The Apriori algorithm finds interesting rules in a realistic timescale, for datasets that are not too
huge. The algorithm was created by Agrawal and Srikant. Afterwards a lot of investigation into how
to make this algorithm even more efficient, and therefore more appropriate for huge datasets, has
been conducted. The algorithm is based on two theorems, which can be seen in Theorem 4.5.1 and
Theorem 4.5.2 [6],[33].
Theorem 4.5.1. For all non-empty itemsets, if an itemset itself is supported, all of its subitemsets
will also be supported [6],[33].
From Theorem 4.5.1 follows the next theorem:
Theorem 4.5.2. If an itemset with cardinality k is empty, all itemsets based on this itemset but
with higher cardinality will also be empty [6],[33].
By using Theorem 4.5.1 and Theorem 4.5.2, a more efficient way of creating all the supported itemsets
is available and called the Apriori algorithm. The algorithm works by first generating all itemsets S
with cardinality 1 which has a support higher than a threshold. If an itemset with cardinality 1 does
not have a support that is high enough, creating a new itemset based on the old itemset but with

higher cardinality, will also not have a support that is high enough due to Theorem 4.5.2. In this
way all itemsets are not generated, and hence saves a lot of time [6],[33].
4.5.3
Making a recommendation
When all the interesting itemsets are found, a recommendation can be made. In the supermarket
this would mean putting the items that are in the same itemset in the same place. Then the items
frequently bought together are close to each other, such that the customer does not have to go
through the whole supermarket to find the items needed, giving the customer a good experience. Or
perhaps this could show the customer what he is missing besides the item he came to buy, leading
to additional sales [6].
The same approach can be applied to movies. If a user has seen one or more movies which are in
some of the itemsets, the one movie in the itemset unseen by the user can be recommended, since
the user will likely like that movie as well.
4.5.4
Advantages and disadvantages
If a user or an item does not have any transactions, it is impossible to recommend anything just as
for the Neighbourhood model and the Latent Factor model. However, there is a chance that the
ARM model will perform better than these, for users who have only seen a limited amount of movies.
If the user has only seen one movie, this movie is enough to find other movies often seen together.
But if user-user CF was used it would be almost impossible to find similar users based on just one
movie [16].
As stated, the Association Rule Mining model is reminiscent of a Popularity recommender, in the
sense that it is looking at which movies are often bought together, and which are bought more often
together with other movies than alone. Therefore, the Association Rule Mining model shares the
advantage of the Popularity recommender in the sense that the most popular movies are often a
good choice for a recommendation, as described in Section 4.2.1 [6].
4.6
Evaluation
All the different recommendation systems just described, are worth nothing if they can not be
compared to each other or evaluated.
This section will therefore present how recommendation
systems can be evaluated. The best way to evaluate a recommendation system is by getting direct
feedback from the users using the system. This will directly tell you, whether or not the users will
actually watch what is recommended to them. The problem with this way of evaluating is that it is
very time consuming and hard in practice. Instead offline evaluation can come in handy, especially
since it can give some feedback while developing the recommendation system. Four different offline
evaluations will be examined; Mean Absolute Error, Root Mean Squared Error, Hit Rate and Mean
Average Precision at N. As an attempt at getting direct feedback, a small test group was assembled,
which will in the end be described.
4.6.1
Offline evaluation
Even though offline evaluation is not the best measure of performance, it is definitely the easiest
to perform.
It also gives a good idea of how well the recommendation system performs when
developing it. Therefore, different ways of measuring this performance will be described, together
with the advantages and disadvantages of these methods. Before describing the evaluation methods,
how to use the data for offline evaluation will be presented.
Training and test set
When using the dataset for computing offline evaluation, the same overall idea is often used. The
idea is to split the dataset into two sets; a training set and a test set. The training set is used for
training or computing recommendations for the missing values in the training set. This is done, since
Movie Recommendation Given Sparse Data
32 of 92
==PAGE:42==
generally, one should never test on the same data, on which the models are fitted or trained. The
recommendations computed from the training set, is then held up against the true ratings in the
test set. The vast majority of the data will be a part of the training set and the rest will be in the
test set. Since the training set is the vast majority of the full dataset, the predictions made should
therefore be representative for the full dataset. This simple way is called the Hold-out method. If
the predicted ratings are then similar to the true ratings from the test set, it must mean that the
model is doing a good job, since it can predict the movies the users have seen in the test set, based
in the training set [34].
Another way to split the data is by using k-fold cross-validation. Here, instead of just splitting the
dataset once, the dataset is split into k training sets with corresponding test sets. This is to make
sure, that all ratings have been a part of the test set. All k training sets are then used to make
predictions and compared to their respective test set, and the average of the evaluation for all the
k sets is computed. An extension to this method, is the leave-one-out cross-validation. For this
method k is equal to the number of ratings in the dataset. In this way, for every training set, one
rating remains in the test set. By doing this, one makes sure that the training set is as representative
as possible of the original dataset [34].
Especially leave-one-out cross-validation is very time consuming, and therefore the k-fold cross-
validation is preferable.
Because many of the models took a while to get predictions from, the
Hold-out method was used in this thesis. To make sure, that one split of the data was not com-
pletely unfair for one or more of the models, two different splits were made, and the average for
these two are computed and used as the evaluation result. Many of the models in this thesis does
not learn from the training data, and therefore will give the same results every time the model is
used for the same dataset. Thus, it makes no big difference whether or not the dataset are split into
many different training sets, as long as the training set is representative for the full dataset, which
should be the case when using the Hold-out method.
Now that a way of using the dataset for evaluation has been presented, the offline evaluation process
itself will be examined.
Mean Absolute Error
The first evaluation measure examined is the Mean Absolute Error (MAE). The MAE is a measure

4.6.2
Direct evaluation
As said, offline evaluation is a great and fast way of getting an idea of, how the recommendation
system is doing while building it. But at the end of the day, direct evaluation is the only measure
saying whether or not the users will actually watch the recommendations. As for the offline evalu-
ation there are many ways of making direct evaluations, also called online evaluations. One way is
by using A/B or A/A testing. In A/B testing, recommendation system A is given to one group of
users, while recommendation system B is given to another group of users. In A/A testing the same
recommendation system is given to the two groups. A/A testing is often used before A/B testing,
as a kind of control, to see if there is any performance difference between the two groups.
No matter which of the two testing methods is used, the next step is to calculate the performance
of the recommender systems. This can for example be done by looking at the number of clicks for
the recommended movies, or how many of the recommended movies were watched in each group [12].
This kind of testing was unfortunately not possible in this thesis. First of all, A/B testing is very time
consuming, and to get a sense of whether or not the recommendation system is working, the users
need to be presented with the recommendations in a certain number of days. Secondly, because of
GDPR explained in Section 2.2.6, it is not possible to use the data directly and get in touch with
the customers in the dataset, to show them recommendations. To be able to do this, a lot of rules
must be respected, and a lot of permissions must be given.
Instead, with inspiration of A/B and A/A testing, another way of directly evaluating the recommen-
dation systems in this thesis, was created. First of all a small test group was gathered. All participants
got the same questionnaire with 100 different movies. The movies was chosen at random, in order to
include a wide variety of movies from different genres. For each movie the participants should answer
whether or not they had seen the movie by 1 or 0 respectively. The top of the questionnaire can
be seen in Appendix D.1. All the answers were then collected and put into the full original dataset,
given by Nordija. This dataset, was then used to give the top 10 recommendations based on all the
movies not seen by the participants. The recommendations were created based on the models doing
well in the offline evaluation, to see if they would do just as well in the direct evaluation. Lastly, the
personal recommendations created were shown to each participant, consisting of the movie title and
a cover picture of the movie. By giving them different recommendation lists, the participants could
evaluate what they thought of the different models.

The first evaluation measure is the MAE. The library sklearn has a method mean_absolute_error,

is the MAE value. The implementation for this can be seen below.

The two arrays were created, such that the true ratings array only consisted of the ones, where
the users have actually seen the movies from the test set, and therefore the predicted ratings array
consisted of the predictions for these movies. This choice was made since the test set consists of
many 0’s, and so does the predictions. Therefore, if the full test set was used, the MAE would
become low, since many of the predictions would become correct. Also, the focus is that the models
can predict the movies the user has seen in the test set, and not that it can predict the ones they
have not seen. Additionally, this removes the users which has no seen movies in the test set. For

Discussion
In this chapter, some of the problems that occurred during this thesis will be discussed. First some of
overall problems and assumptions will be discussed, including a discussion about the running time of
the models. Next the results from the Neighbourhood models will be discussed since the results here
did not correlate with results from other articles. Lastly, the evaluation methods will be discussed.
7.1
Overall discussion
First of all, when inserting a 1 or a 0 whenever the movie is seen or not respectively, and recom-
mending based on this data, gives a hypothesis that a user has liked the movie, when the user has
seen the movie. Even though it is ensured that a movie has at least been seen 75% and therefore
should have been watched to the end, there is still a risk, that the user did not really like the movie.
This hypothesis can therefore lead to mistakes, when recommending other movies based on all seen
movies, instead of all movies which has been rated positively by the user.
Normally, if the dataset had consisted of ratings in some interval, the recommendations, which
would lie in the same interval, with a value below a threshold would have been classified as bad
recommendations. The ones above, on the other hand, would have been categorized as good rec-
ommendations and used as the final recommendations. Since the given dataset consisted of 0’s and
1’s, it was decided to use all recommendations higher than 0. This also means, when evaluating
the results, there is a chance that many movies are recommended for each user, which will make at
least the RMSE values higher, than if some of the recommendations were left out. It does not mean
that much for the Hit Rate and the MAP at N value, since these are both dependent on the shown
recommendations, which will still be the ones with the highest values.
If the dataset had consisted of ratings, the performance for many of the models would possibly be
higher, since whether or not a user likes a movie would be more varied.
7.1.1
Running time
When recommending movies for users, it is important that the recommendations are shown imme-
diately, to make sure the attention of the users are kept. Therefore, the recommendation system
has to be able to either quickly compute the recommendations, or have a way of storing recom-
mendations for the users, and then update these recommendations when more data gets into the
system. Therefore, the running time of the models is an important factor as well, when defining the
best model. If the model is making great recommendations, but takes several hours to compute the
recommendations, it is of no use. On the other hand, if the model is making fine, but not great,
recommendations, but only takes a few seconds to compute the recommendations, it is of greater
value.

Some of the models took several hours, and the most computationally heavy models could take up
to 10 hours, before all the predictions were computed. Therefore, many of the predictions were not
computed in one go, and the running time for each method was therefore not computed as well. To
get a representative value of the running time, when the recommendations were computed for all
the models, the conditions should be the same. Therefore, it is not optimal to start and stop the
computations when the running time is measured. Additionally, it would be best to measure the
running time several times and take the average, since the running time can vary from time to time.
These factors prevented realistic results of the running time to be computed, and therefore it was
chosen not to invest time in this aspect of computing recommendations. Even though a precise run-
ning time was not computed, it was clear to see which models were computationally harder than the
others. The Popularity recommender and the Random recommender was by far the fastest models.
It only took a few seconds to get recommendations from these models.
The running time for the Neighbourhood models were very dependant on the number of neighbours
used, and if the user-user or item-item CF was used. Overall, the item-item approach was much
faster, since there are far less movies compared to users in the dataset, and as a consequence creat-
ing and using the correlations was much faster. Looking at the number of neighbours, for every 10
nearest neighbours added to the total number of neighbours, the running time was close to being
doubled every time. This resulted in the running time for the user-user CF for 50 neighbours being
approximately 10 hours. For the item-item CF, this running time was around seven hours. The
number of neighbours in the Neighbourhood models was therefore significant for the running time,
at least without further optimization.
Looking at the running time for the the Latent Factor models both of them would take around 30
minutes to two hours. Using 5 to 100 factors did not seem to have a big influence in the running
time. But as soon as more than 100 factors were included, the running time would significantly in-
crease. The running time using these models for the MovieLens dataset, would be more than a day.
Therefore, unfortunately the models was not tested on this dataset. This indicates that the running
time of the SVD and NMF models are very affected by the size of the data used. As mentioned in
the results, one have to keep in mind, that the models normally only uses the ratings for the movies
the users has seen. In this thesis, both the seen movies and the movies not seen had to be used for
the models. Otherwise it would predict everything as 1’s, since this would then be the only input.
Therefore, the dataset inputted would normally not be as huge as the one used in this thesis, and
the running time would be smaller.
Comparing the models using the Association Rule Mining models, using support as the measure
was definitely the fastest model, and would take around one hour. Using lift would for the highest
amount of rules, take around 10 hours and around five hours for the lowest amount of rules. Using
confidence would also take around 10 hours for the highest amount of rules, but only approximately
one hour for the lowest amount of rules. Using the MovieLens dataset increased the running time a
bit, but the greatest impact was the number of rules used.
To sum up, the Popularity and Random recommenders was definitely the fastest models, which
makes sense, since none of them is using prior knowledge or the behaviour of the users. The rest of
the models were very dependent on which parameters were used, but overall the SVD and the NMF
models were the fastest of the more advanced models, where the Association Rule Mining models
in general seemed to be the slowest. Considering that SVD was also performing the best on the
training sets and decent in the direct evaluation, this indicates that this model in general seems to
be a good choice when making recommendations, since they also seems to be able to relatively fast
giving recommendations.

7.2
Discussion of the Neighbourhood model
Looking at the comparison of the Neighbourhood models in Figure 6.3 and Figure 6.4, using the
Nordija dataset and the MovieLens dataset respectively, it was seen that the user-user CF was the
best model for both datasets. This is a surprising result, and opposite of what was found in e.g.
[4], where the item-item CF model was performing the best. In Table 3.3 and Table 3.5 it can be
seen that the movies on average have significantly more ratings than the users. Because of this, one
could imagine the items having a more precise neighbourhood, since the similarities are then based
on much more ratings.
In [4] the data consisted of ratings from 1 to 5, which is not the case in this thesis where the ratings
are between 0 and 1. This could potentially be the reason for the difference of the results in this
thesis and of [4]. The similarities in this thesis are based on whether or not the users have seen the
movies. The user-user similarities are therefore defined by how many movies two users both have
seen, where the similarities between the movies are defined by how many of the users have seen both
of the two movies. One could imagine that if two users have seen many of the same movies, they
must be quite similar and a good basis for recommending other movies the similar user has seen.
Even though a user does not have to have liked all the movies they watched, the probability that the
user liked the majority of the seen movies is high. On the other hand, if two movies have both been
seen by the same users, since there is no rating saying whether or not they liked the movie, the basis
for saying the two movies are similar might be slightly worse. If many of the users who watched the
two movies disagreed, the similarity is calculated on a wrong basis.
It is hard to tell if this is the exact reason for why the results in this thesis is differing from e.g
the results in [4]. One thing for sure, is that it is interesting to see, how the results will change when
the datasets looks different from each other. Additionally, the similarities are based on which movies
the users have not seen, which can be a bad basis for being similar. This says that two users can
be similar, if they both have a lot of movies in common they have not seen, or that two movies are
similar if many users have not seen either of them.
7.3
Discussion of the evaluation methods
The results in this thesis are very dependent on the evaluation methods used. Some of the problems
with these are therefore discussed.
7.3.1
Discussion of offline evaluation
When comparing the models with different values of the parameters, in most cases the evaluation
measures are consistent in which model performs the best. However, this is not always the case.
Looking into Table 6.4 and Table 6.6 e.g, for the user-user CF and the item-item CF respectively
using the MovieLens dataset, the best model using the HR and MAP at N values was the one with
50 neighbours. When looking at the RMSE values, the opposite was the case, where the best value
was for 10 neighbours, and the highest RMSE value was obtained for 50 neighbours. When using
50 neighbours versus using 10 neighbours, the opinion of more users are taken into account. This
can mean that the prediction for the movie does not get as high, since the opinion of the movies
will then vary more, compared to looking at 10 users opinion. Therefore, when taking the RMSE
values the difference between the true ratings for the seen movies will be greater, since the predicted
ratings are closer to 0.
Another case where this is happening, is when looking at Table 6.11 for the Association Rule Mining
model, using support as the measure for the MovieLens dataset. Using the HR and MAP at N value
the highest threshold with support 0.09 gives the highest values. Looking at the RMSE value, this
is the worst model, and the lowest value is for support 0.06 where the HR and MAP at N values are
lowest. This could be due to when the threshold for the support gets higher, less interesting rules
are computed leading to fewer recommendations. Since the recommendations for the Association

Rule Mining models are 1 for recommended and 0 otherwise, less recommended movies will lead to
a higher RMSE.
It only happens using the MovieLens dataset, that the thresholds making fewer recommendations
still gives the best 10 shown recommendations. It therefore shows that when more data is available
it does not have to make as many recommendations to get many of the top recommendations correct.
The basis for calculating the RMSE value also differs from model to model, since all the models
are creating a different amount of ratings, and the prediction value is either a value between 0 and
1, in the case of the Neighbourhood models and the Latent Factor models, or exactly 0 or 1 for the
Popularity recommender, Random recommender and Association Rule Mining models. Therefore to
get a RMSE value that more precisely can be compared between the different models, these condi-
tions should be more similar. It was tried to calculate the RMSE value for the models when maximum
50 recommendations was created, but this only changed the RMSE value slightly. The HR and the
MAP at N values are more precise measures when comparing the models, since the condition for
calculating these are more similar. Additionally, as described earlier the HR and MAP at N values
are in general a better measure, since they tell whether or not the users will like the movies shown
to them.
7.3.2
Discussion of direct evaluation
When making direct evaluations, there are many biases to keep in mind. First of all, the test group
only consisted of 15 participants, which is not close to being representative to the actual user size of
the platform where the data is from. To make the test group representative, it is also important that
people from different age groups, which could be potential users of the platform, are represented.
Secondly, only 100 movies out of 5,358 total movies in the dataset was shown to the participants.
This meant, that the base for making recommendations for the participants was similar, leading to
the recommendations being quite similar for the participants, when looking at the recommendations
computed by the same recommendation models. More critically, it also means that the recommen-
dations are biased towards the movies shown to the participants. Even though different genres and
types of movies was chosen for the 100 movies, it will never be representative for the full set of
movies. This could mean that some movies, which the participants have seen, will not have an
impact on their recommendation, which could otherwise have improved the recommendation. Since
the users were not shown all the movies from the full dataset, it also implies that the participants
can get recommendations they have already seen. When the participants are then asked to rate a
recommendation list, with one or more already known movies, they will already have an opinion of
the movie, good or bad, making them biased towards the full recommendation list.
Because of all these biases, it is important to look at the results from the direct evaluation crit-
ically. In [12] they are looking into evaluation of recommendation systems by using A/A testing.
One of their results show, how challenging it can be evaluating recommendation systems online, since
their results are significantly different, which is not wanted in an A/A test.
Lastly, the SVD model was not performing very well in the direct evaluation. This could be due
to a mistake made when the recommendations was created for the participants. Here, only the
ratings where the participants had seen the movies was used in the dataset, and therefore only 1’s
occurred for the participants. This did not give the model a varied reflection of the users, and instead
the ratings for all 100 movies should have been inserted. This fact was realised too late, but it would
assumable have given better results for the SVD model using direct evaluations.

result from this is how powerful a recommendation system such as the Popularity recommender can

Conclusion
Different models have been investigated in this thesis, in the process of finding the ones performing
the best on a given sparse dataset. By this investigation, it was clear how many different types of
recommendation systems are out there, for many different purposes. Additionally, many different
ways of evaluating these systems was found.
Overall, it was clear to see that all the models were performing better, and most of them sig-

if there is a lot of data available. With more data also comes more computations, which for most
recommendation systems increases the running time.
More specifically, by using offline evaluation, the Latent Factor models significantly outperformed
the other models. They were able to get almost a 50% Hit Rate for the test set, whereas the other
models maximum got a Hit Rate at less than 2%. Looking at the direct evaluation, saying whether
or not the users will actually watch what is recommended, the Popularity recommender and the As-
sociation Rule Mining model were performing the best. This shows how different results can appear,
depending on the chosen method of evaluating the models.
Thus, it is shown that an acceptable recommendation model was obtained by the Latent Factor
model using Single Value Decomposition which had a Hit Rate at approximately 50%, even though
the dataset given was very sparse. If more data with more rating variation was available, it might
result in even better performance.

Future Work
The data provided by Nordija did not only consists of data where at least 75% of the movies was
seen. It therefore could be interesting to find the movies interrupted by the users, and give these
movies a −1 in the rating matrix. This would give three different ratings in the dataset; 1 for liked,
0 for not seen and -1 for disliked. In this way the data would be more diverse and could perhaps give
better performance for the different recommendation models.
The original MovieLens dataset, consisted of ratings between 1 and 5. The original dataset could
have been used for computing recommendations if more time had been available, to see if the more
varied rated data would improve the recommendations.
In the same way as people watch more movies during the weekends and holidays, the weather
can have an impact in how many and which movies people watch. People might have a tendency to
watch some movies when the weather is good, and others when it is raining or freezing. To really
find a pattern in this, data from more of the year is needed. The data provided by Nordija consisted
of data from three winter months, and therefore it could perhaps be difficult to find movies watched
when the weather is good and warm.
Some of the models took several hours to compute predictions.
This was especially the Neigh-
bourhood models with many neighbours, and the Association Rule Mining models when increasing
the number of interesting rules. In general, it would therefore have been interesting to experiment
more with the parameters for each model to get a more precise result on the best parameter for
the models. For the Neighbourhood model an adjusted way of using the similarities for making
predictions, were presented in Section 4.3.3. If more time was available, it would be interesting to
see if this prediction measure is better than the weighted measure, used in this thesis.
Besides being able to test more scenarios for each model, optimizing the models would also give the
possibility for running experiments for the running times for each model. By doing this, the running
time for the models could be compared, to see if the best model is also the fastest at computing
recommendations, since as stated in Section 7.1, it is important to be able to show recommendations
quickly. Additionally, the running time for different parameters for each model could be evaluated
as well, to see how big an impact the parameters have on the running time for the model.
In the beginning of the thesis, Content Based Filtering is introduced.
Unfortunately, no textual
information was available for the dataset, and therefore this model could not be used. If more time
had been available, it would have been interesting to try and retrieve some kind of textual informa-
tion for all the movies, and use this for a Content Based Filtering model. This could show how the
Content Based Filtering model would perform compared to the Collaborative Filtering models.
