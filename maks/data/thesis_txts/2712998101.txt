Abstract
More and more companies migrate from developing monolithic software applications to
developing cloud-native applications. The complex and dynamic infrastructure of cloud-
native applications makes them challenging to monitor. There are many cloud monitoring
tools available. However, it requires time and technical skills to select, configure, and use
these tools, especially for small companies with limited resources.
This thesis addresses the challenges by providing a method that enables companies to
find the appropriate tools and set them up with minimal investments.
The method
consists of three parts.
One is a decision guidance model that can convert company
requirements to concrete monitoring approaches. Another part is a survey that classifies
the available tools in the field of cloud-native monitoring. The tools are labelled based
on the approaches identified in the decision guidance model. The last part is a selection
algorithm that automatically selects tools from monitoring approaches.
The method has been implemented in an interactive web application. The application
allows stakeholders to use the method in a business environment.
In addition, it is
extendable, which makes it easy to improve and update, making the application relevant
and usable in the industry. However, further evaluation is needed to validate the method
and the application.

CHAPTER1
Introduction
In the last couple of years, it has become appealing for small companies to use cloud-
computing in the software industry [1]. The main concept of cloud computing is that
companies can subscribe to storage and computing resources that a cloud provider hosts
instead of buying computer hardware. The cloud provider will invest in the necessary
hardware and maintenance, making it easier for the company to scale.
The surge of cloud computing usage has given rise to new software architectures that
utilise new possibilities. One architectural style that has gained a lot of attention is cloud-
native architecture.
According to the Cloud Native Computing Foundation (CNCF),
there was 4.7 mil. cloud-native developers worldwide in May 2020 [2]. In August 2020,
this number had increased to 6.5 mil. [2]. A cloud-native application uses cloud com-
puting and modern technologies to achieve high scalability and resilience, among other
things.
While cloud-native applications have many advantages that make them attractive to
developers, they also come with challenges that require extra attention, as opposed to
monolithic applications. One of these challenges is to make the system observable. An
observable cloud-native application can help the company improve the user experience,
incident handling and resource management. However, the distributed nature of cloud-
native applications makes it hard to achieve the desired observability because services
are running in independent processes spread over different hosts [3].
There are many tools available that improve the observability of cloud-native applications.
But most of the tools have different features and configurations. In addition, many of
them have a steep learning curve, and a company can quickly end up combining multiple
tools to meet their requirements.
Multiple tools result in more complicated systems
and higher maintenance. These problems are amplified for small companies. They have
limited resources, and the high competitiveness will typically make them push for new
releases [1]. This makes it challenging to invest in monitoring.
In this project, we create a method to help small companies select, configure, and use
monitoring tools for cloud-native applications. The method is developed by using a deci-
sion guidance model. The model aims to help stakeholders identify monitoring objectives
and possible approaches for achieving those objectives. The selected approaches are in-
put to a selection algorithm that determines which tools the stakeholders should apply
to their system. The possible tools the algorithm can recommend have been selected by

1.1
Research Methodology
The goal of this project has been to design a software artefact that can assist in solving
the identified problem. To ensure that the artefact is developed scientifically, following
all the best practices, design science methodology has been applied.
Design science
is based on three cycles: the relevance cycle, the rigour circle, and the design cycle [4].
Each cycle represents activities that should be performed iterative throughout the project
duration. The relevance cycle describes the activity of evaluating the business needs in the
environment where the artefact is to be applied. Relevance can be regarding the people
using it, the organisations that might need it, or the technologies available in the industry.
The rigour cycle describes the flow of information back and forth between the knowledge
base and the artefact. Development of the artefact should be based on existing knowledge,
which it will extend and append to the knowledge base. The acquired knowledge can be
research and methodologies that are applicable to the artefact. It should comply with
scientific standards and methods that are used in research. The design cycle describes
the activity of developing the artefact. Designing should continuously be evaluated and
modified based on the input from the two other cycles.
In the following list, we will describe how the different cycles have been applied throughout
the project:
• Relevance cycle: The artefact has continuously been compared to business needs
through recent surveys and industrial studies. CNCF regularly publish data col-
lected from their communities, highlighting challenges and new insights in the field.
The contribution of this project to the industry is an artefact that addresses these
challenges. It is extendable and publicly available.
• Rigour cycle: The project required us to understand the field well, the prob-
lem and existing solutions in the early stages. This knowledge has been based on

1.2
Problem Statement
The problem statement of this project is based on initial research. According to surveys
by CNCF [5] and research in the field [6, 7, 8], observability is one of the significant
challenges in cloud-native development.
Especially small companies under-emphasise
observability and quality assurance to focus on developing new features unless it requires
minimal setup efforts [1].
The problem statement that has been the basis of this project is as follows:
[PS]: The benefits of monitoring are rarely realised for small companies work-
ing in cloud-native environments because they lack the resources or knowledge
to select, configure, and use the proper monitoring tools.
Monitoring can improve the product’s overall quality and stability by making aware of
deviations that the application is behaving as expected. There are many monitoring tools
available that can assist the developers. But the tools have different focus areas, which
makes it challenging to navigate the features they provide. Additionally, the complex
nature of cloud-native applications makes it complicated to configure and use the tools.
Our research process will focus on two areas to investigate which solutions might exist to
address the problem. The first area is the advantages and challenges that companies face
when monitoring cloud-native applications. The other area is research that surveys and
classifies the monitoring tools available in the field. A set of research questions has been
formulated to guide our research process. The questions should assist in finding solutions
to our overall problem.

• [RQ1]: Why would businesses using cloud-native applications want to use moni-

• [RQ2]: What are the challenges of selecting, configuring and using monitoring in

• [RQ3]: Which limitations do the available tools have? Are there cases where they

by using monitoring in cloud-native environment. This involves looking at the effects of
the company size. RQ3 and RQ4 focus on the technical aspects of different methods used
by the monitoring tools. These questions should illuminate the considerations that small
companies make when introducing monitoring. The first four research questions lead up
to RQ5, which aims to identify the best possible solution for the problem statement.
1.3
Related Work and Contributions
Much research has been conducted on monitoring in cloud-based systems. Fatema et al.
[9], and Aceto et al. [10] have created surveys that analyse existing monitoring tools for
cloud computing and provide a taxonomy for them. However, these surveys are from
2014 and 2013, respectively, and things have changed in the way monitoring tools are
used for cloud-native applications over the last ten years. Kufel [11] and Birje et al. [12]
are more recent sources that analyse different monitoring tools and their capabilities.
In this project, we build on some of the abstractions and concepts they introduce, like
the concept of agent-based/agentless monitoring (section 5.2). Kufel provides companies
with guidelines that can help them select proper tools, but he does not look into how the
tools complement each other and how they should be configured. Also, neither Kufel nor
Birje specifies which criteria they used to select tools to analyse.
Tamburri et al. [1], and Waseem et al. [6] each takes a more empirical approach to the
issue. They perform surveys of the practitioners of microservice monitoring. They map
out the common challenges faced when monitoring microservices through interviews and

They provide decision guides for each of these solutions and map them to

CHAPTER2
Motivations for Monitoring
To monitor an application is to generate, process, disseminate and present information
about the system’s state. What is meant by these activities will be described in detail
in section 3.2. But why do we need to know what is going on inside the system in the
first place? Clearly, we cannot hope to collect data about everything going on inside a
complex application, so how can we determine what is important to monitor and what is
not? In this chapter, we will try to identify objectives companies might want to achieve
using monitoring. Implementing monitoring is an investment. Depending on the moni-
toring goal, it can require changing the source code and adding additional infrastructure
layers to the system. The additional monitoring configurations in a system increase the
maintainability and complexity. Especially, small companies and startups are reluctant
to invest in monitoring because they prioritise speed of development and delivery [1]. In
this project, we have identified three objectives where monitoring can bring value to the
company, which can justify spending resources on monitoring. These are user experi-
ence, incident management and resource management, as illustrated in figure 2.1.
Security is also a very relevant objective for monitoring, but that is outside this project’s
scope.
Figure 2.1: The three identified objectives where monitoring can bring value to the company.
A good user experience can help increase the customer conversion rate. The user expe-
rience is affected by many different factors, but some of them could be the performance
and availability of the service the user needs. Gan et al. describe the impact of these
factors in the following way:
Cloud computing services are governed by strict quality of service (QoS) con-
straints in terms of throughput, and more critically tail latency. Violating
6
==PAGE:14==
Master Thesis
July 6, 2022
these requirements worsens the end user experience, leads to loss of availabil-
ity and reliability, and has severe revenue implications ([16], p. 1).
Research shows that a page’s load-time significantly impacts the number of users leaving
the page before making a purchase [17, 18]. Another factor that can affect sales is the
availability of the service.
If the service is often unavailable, then customers cannot
make any purchases. The first step towards mitigating these user experience issues is
monitoring them.
The next objective is Incident management. In ”ITIL Service Operation: Best Man-
agement Practice” [19] the author defines incidents as
An unplanned interruption to an IT service or reduction in the quality of an
IT service ([19], p.327).
Many things can cause incidents. It can be errors in the system that reduce functionality.
It can also be an external event, like a security breach or cloud provider downtime.
Incident management is about detecting the cause of an incident so that appropriate
action can be taken. Incident management can also be a means to prevent incidents from
happening in the first place.
Incident management includes detecting bugs and mitigating incidents that cause services
to be unavailable. Any software will have bugs in it when it is shipped, but the sooner
they are discovered, the cheaper they are to solve. In a report from The National Institute
of Standards and Technology (NIST) in 2002 [20] it is estimated that bugs cost as much
as 30 times more to fix after the code is put into production compared to during the
design phase. Monitoring can be used to detect errors as early as possible, making it
easier and cheaper to fix them. Sometimes issues can make the entire system, or parts
of it, unavailable.
Incident management ensures that guidelines exist for what to do
when an incident happens, and incident response time is the time it takes from when the
incident is reported until it is solved. Tamburri et al. [1] have conducted a survey where
141 people from different companies were asked about cloud monitoring within their
company. The survey showed that the respondents who did not consider monitoring a
profitable investment had significantly longer average incident response times and longer
unavailability times. This is an example of how monitoring can help companies limit
downtime and streamline incident handling.
The third objective is resource management. Subscribing to storage and computa-
tional resources from a cloud provider can be very expensive, and small companies need
to spend their resources leanly, ensuring that resources are not spent on something that
could have been avoided. Monitoring can help analyse the application’s resource usage,
which can help developers detect unnecessary use of cloud services.

The monitoring objectives are loosely based on literature. Aceto et al. [10] identify a list
of ”needs for monitoring”, including things like troubleshooting, Performance monitoring
and capacity and resource management. But some of the other needs focus only on the
cloud provider, which is irrelevant for this project. Chakraborty et al. [13] have a different
model when describing monitoring domains. However, their domains are more focused
on different technical responsibilities within an organisation, e.g.
services, containers
and cloud infrastructure. Both Aceto and Chakraborty have categorised the need for
monitoring in a concrete and technical way. By combining the two approaches, we have
chosen a set of objectives that capture how the user would benefit from a monitoring
solution. As opposed to Aceto and Chakraborty, the goal of this project is to make it as
easy as possible for a user to select a monitoring solution for their company. The step
of picking objectives should be possible even for people without a technical background.
For this reason, we aim to incorporate the business benefits in our solution by making
our objectives business-oriented. Each objective still contains technical concerns that
technical staff can address after an objective has been picked.
There are overlaps between the three objectives. For example, better incident manage-
ment can also improve the user experience. But it is a good starting point for selecting
what kind of monitoring solution would fit a given company best. Later in the project,
we will get into more detail about how to approach each monitoring objective and which
tools provide the required functionality.

CHAPTER3
Observability and
Monitoring
Before it makes sense to look into specific monitoring tools and approaches, it must
specify what monitoring means.
Otherwise, it can be hard to know what would be
perceived as monitoring and what would not. In the context of cloud-native applications,
the term ”observability” must also be explained. Monitoring and observability are often
used interchangeably in the literature and the industry [14, 21, 13]. In this section, we
will try to create an overview of the two concepts, how they are related and how they
will be used in the rest of the project.
3.1
Observability
The term observability stems from Control theory. It was introduced as a mathematical
concept by Kalm´an in 1960 [22]. In mechanical engineering, it means the ability to deter-
mine the state of a system based only on the output of that system [23]. An observable
system is inherently transparent, where it is easy to understand what is happening inside
at run-time. The term has recently gained more attention in Software Engineering [14,
15]. In this context, the observability of a software system is the degree to which an
analyst can determine how the system is working, what kind of issues the system might
be facing and why based on the system’s outputs. An analyst should be able to explore
an observable system and debug it on the spot without knowing beforehand what to look
for. Figure 3.1 the interaction between the system and the analyst. The analyst can
query a database of run-time information about the system. These queries are visualised
in dashboards with tables, graphs, or other visual components. By looking at the dash-
boards, the analyst can gain knowledge about the system. This process can be repeated
until the analyst has obtained the knowledge they sought, and then they can draw the
relevant conclusions and take action accordingly.
The system information that the analyst queries from will often consist of some combi-
nation of logs, metrics and traces. These data formats are often referred to as the three
9
==PAGE:17==

July 6, 2022
Figure 3.1: An illustration of the interaction between the analyst and an observable system.
The analyst queries a body of monitoring information to get insights in the system.
pillars of observability [14, 13]. The data formats have different levels of complexity. A
more thorough description can be found in section 5.1.
The history of observability and its three pillars is a relatively short one, and it includes
some amount of disagreement. At the 2017 Distributed Tracing Summit, Peter Bourgon
wrote a blog post on logs, metrics and traces [24]. According to a blog post on the his-
tory of observability by Humio [25], Peters’s blog post led to the observability community
accepting logs, metrics and traces as the three pillars of observability. However, in ”Ob-
servability Engineering” by Majors et al. [15], the authors claim that the definition of
the three pillars of observability as logs, metrics and traces is misguided:
It is hard to decide which is more objectionable about this definition: its
redundancy (why exactly do we need another synonym for telemetry?) or its
epistemic confusion (why assemble a list of one data type, one anti-data type

3.2
Monitoring
Monitoring is a more classical term in computer science. Generally, it means looking at
something and keeping track of its process. In computer science, monitoring a system
means continuously collecting and investigating specific information about the system’s
state. Where observability is about exploring a system, comparing data and drawing
conclusions, monitoring is focused on how to collect concrete data and the path the
data takes from the bowels of the system to somewhere an analyst can access it. Figure
3.2 shows a running system where data is continuously flowing through the application.
Monitoring is used to extract meaningful information from the system, process it and
store it. The stored data will often be in the form of logs, metrics or traces. This is
where there is an overlap between monitoring and observability.
In their thesis from 1993, Mansouri et al. [26] divide monitoring into four activities:
• Generation of monitoring information: Detection of events or status reporting,
trace generation.
• Processing of monitoring information: Filtering, merging and aggregating the
data.

==PAGE:19==
Master Thesis
July 6, 2022
Figure 3.3: The four monitoring activities that are performed throughout the monitoring life
cycle.
• Dissemination of monitoring information: Information is forwarded to users
or other processing entities. This could be a list of subscribing users or a central
monitoring service.
• Presentation of monitoring information: UI elements, dashboards, animations
or textual representations.
They describe the goal of the model in the following way:
In this thesis we shall present a generic model of monitoring [...] which extends
the existing models to encompass all facets of the monitoring life cycle. It
is used to characterise the monitoring process and identify the functionalities
performed by a generalised monitoring service. ([26], p. 15)
The four activities each represent a part of the monitoring life cycle. The monitoring
life cycle entails everything that happens from the data is recorded, while it is stored
and consumed, and until it is later discarded.
Figure 3.3 shows the activities.
The
monitoring life cycle will always start with generation of information, but apart from that,
the activities have no specific ordering. The same activities can be performed multiple
times throughout the monitoring life cycle. We will use this model as a guideline for what
we consider ”monitoring activities”. A tool or a service will be regarded as monitoring if
the main focus is to perform at least one of these four activities. Consequently, tools that
only focus on visualising and not on the other activities are also classified as monitoring
tools, even if they must be combined with other tools to be useful.
We define a monitoring solution as a set of monitoring tools that, when combined, take
care of each of the four monitoring activities that make up the monitoring life cycle. A

monitoring solution should also include configuration instructions that specify how the
tools are set up and tied together. A central goal of this project is to create a mapping
from monitoring requirements and system configurations to monitoring solutions.
Mansouri et al. use the term ”monitoring information”, which they consider to be events
or status reports. To use the same terminology throughout the project, we will specify
monitoring information as logs, metrics, or traces since these are the specific data models
we will focus on.
There are many different definitions of what monitoring is. A lot of them resemble the one
given by Mansouri et al. One such definition comes from Rob Ewaschuk ”Site Reliability

Monitoring means collecting, processing, aggregating and displaying real-time

Continual checking, supervising, critically observing or determining the status
in order to identify change from the performance level required or expected.
[28]
Here monitoring is a means to detect changes in the performance of the system. But this
definition mentions ”observing, ” which creates unnecessary confusion between observ-

context than just detecting changes in performance, which is generally associated with
latency and throughput.

Architecture Center, they use the following distinction:

3.4
The Probe Effect
Before moving on, it should be mentioned that monitoring can cause what is known
as the probe effect. This effect describes the change that monitoring or measurement
instrumentation can introduce to a system [30]. This also applies to software systems. For
example, if monitoring is introduced to a system to collect processing time measurements,
then these measurements can change the system’s processing time. Implementing new
monitoring solutions should always be justified, or it might just introduce additional
overhead. A similar concept is the observer effect. Mythkowics et al. [31] describe the
observer effect as when data collection alters the system’s performance in a way that can
cause the consumer of the data to draw invalid conclusions. As such, observer effects can
be seen as a specific type of probe effects that distort monitoring results. They point
to two methods of detecting observer effects in the system, causal analysis and variant
generation. But both of these methods require thorough monitoring instrumentation data
analysis. In this project, we will not go further into detecting observer effects. However,
15

Master Thesis
July 6, 2022
it is critical that the monitoring solution that is to be implemented does not include
unnecessary measurements.

Summary
This chapter has discussed the concepts of monitoring and observability. They are often
used interchangeably, but an interpretation has been established based on the literature.
In this thesis, the concept of monitoring is based on four monitoring activities: generation,
processing, dissemination, and presentation of monitoring information.
On the other
hand, observability is the ability to investigate the state of a system based on monitoring.
Therefore, to achieve observability, one must have monitoring, which is the focus of this
project.

Cloud computing has transformed our understanding of how computer resources are per-

report from Flexera [32] shows that SMEs have 69% of their workload in a public cloud
environment, which is expected to increase by 5% within the next 12 months. Cloud

4.1
Cloud Computing
The central idea of cloud computing is explained through the definition provided by the
National Institute of Standards and Technology (NIST). The essence of cloud computing
is based on the fact that instead of buying a computer, one now rents computer power.
Cloud computing is a model for enabling ubiquitous, convenient, on-demand
network access to a shared pool of configurable computing resources (e.g.,
networks, servers, storage, applications, and services) that can be rapidly
provisioned and released with minimal management effort or service provider
interaction. ([33], p. 2)
The cloud computing paradigm can be compared to how we are supplied with electricity
and water, where we are billed based on consumption. Instead of buying physical com-
ponents installed in a physical location and manually configured, storage and computing
power are now offered as a service. This paradigm shift eliminates the need for investing
in physical machines that are challenging to scale and quickly become obsolete. The
17
==PAGE:25==

July 6, 2022
definition above is loose and very compact. The following is a more detailed description
of concepts that characterise cloud computing which NIST has defined [33]:
• On-demand self-service: A consumer should be able to access computer re-
sources, such as server time and storage, provided by a cloud provider whenever
required. On-demand self-service allows users to request resources at run-time with-
out any direct communication with the cloud providers.
• Broad network access: Resources provided by a cloud provider must be ac-
cessible from multiple client platforms over the network using standard internet
protocols such as workstations, mobile phones, and tablets.
• Resource pooling: The computing resources offered by cloud providers, e.g.,
storage, memory, and network bandwidth, are served using a multi-tenant model,
with different physical and virtual resources dynamically adjusted to the consumer
demand.
• Rapid elasticity: The cloud provider must be able to dynamically scale the
resources they provide to be commensurate with the consumer’s demand.
• Measured service: Cloud systems must be able to control and optimise resources
by monitoring usage automatically. The goal is to provide transparency for the
provider and consumer of the utilised services.
The above characteristics indicate the advantages of using cloud computing. However,
one must use a cloud deployment model to use cloud computing. NIST describes four
different deployment models [33]. These deployment models are defined based on where
the deployment’s infrastructure exists and who controls the infrastructure. The different
deployment models satisfy different organisational needs, and it is therefore important
that every company choose the deployment that fits its strategy and system. Figure 4.1
illustrates the different deployment strategies.

Containerisation
An essential part of a cloud-native system is the ability to deploy parts of the system
in isolated environments, where they are independent of other parts of the system and
the underlying operating system (OS). This can be achieved through containerisation.
A container is a lightweight virtual machine (VM). It is a software package with all the
necessary binaries and libraries executed on virtualisation of the underlying OS [41]. The
binaries and libraries are called the container image.
The container can use storage,
memory and computing resources as if it was directly connected to the hardware. But in
fact, the container is communicating with the host OS through a container runtime. This
runtime is a framework that is shared by all containers on the host machine. Figure 4.3
from Docker’s webpage [42] shows the layers between the containers and the hardware
infrastructure. It is seen how multiple containers can run on the machine with different
images. They communicate with the same container runtime, which in turn communicates
with the host OS.
Figure 4.3: An illustration of the difference between the containers and virtual machines. In
the figure the container runtime is Docker, but it could be any container runtime like Podman
or LXD [42].
The difference between containers and VMs is that VMs have a guest OS that runs on top
of the host OS. This gives the VM direct access to the hardware infrastructure through
a layer called the hypervisor, but it also increases the size and complexity. Additionally,
multiple VMs on the same machine will all need a guest OS, where containers share the
24

Master Thesis
July 6, 2022
same container runtime. The result is that containers are much easier to create and deploy
on the run. Cloud-native applications use containers to create a dynamic environment
where software components can easily be deployed and destroyed independently from the
rest of the system.

Microservices
For the past 15 years, most traditional applications have been monolithic [44]. Dragoni et
al. [45] define a monolithic application as a software application in which all functionality
must be deployed as a single unit. Monolithic applications run on a single processor,
had a single code base, and are often deployed in a three-tier architecture with UI,
logic, and database. The simplicity of using a monolithic architecture also gives some
limitations and challenges. The following are some of the essential challenges when using
the monolithic architecture [45]
• Scalability As mentioned in section 4.2, scalability describes the system’s ability to
handle an increase or decrease in workload. In a monolithic application, the usual
strategy for handling increased workload is by creating a new instance of the same
application and splitting the workload among the instances. However, this is not
optimal regarding resource usage since it is often only a subset of the application
that is stressed by the increased workload. The newly allocated resources for the
rest system become inconvenient.
• Deployment To make a new feature, change a previous feature, or fix a bug in
the application, the entire code-base had to be redeployed into production. This
will result in delays and subsequent loss of revenue for the business. Furthermore,
businesses today that use agile methodologies aim at faster release cycles, which
involve developing new product features to meet the high demand and competition
between companies.
• Technology Lock-In A monolith application’s tightly coupled components result
in developers’ technology lock-in.
Figure 4.5: Illustrates the monolithic architecture of the reference application. The application
contains all of its domain logic and the modules (Identity, Catalog, etc.) directly communicate
with each other within a single server process. All of the modules share one large relational
database [44]

Using a monolithic application can be time-consuming and may limit the agility and speed
of software development teams. The microservice architecture addresses the challenges

monolith application was split into serval independent deployable services. Microservice’s
support for independent deployability makes it possible to change the functionality of

To achieve the loosely coupled components, one must consider the communication ap-
proach between the microservices, also known as interservice communication. There are
two main approaches, synchronous and asynchronous communication. Synchronous com-
munication is done by calling an API that another service exposes by using a protocol
such as HTTP. This approach is synchronous since the service that makes a request must
wait for a response from the receiver. In contrast to synchronous communication, a ser-
vice making a request using asynchronous communication does not need to wait for a

to be called cloud-native, it is very common because it helps keep the code deployed in
the containers independent and encapsulated. In short, it can be said that each service is
simpler than a monolith application, but the entire microservice system is more complex.
The complexity of using microservice also goes for monitoring which will be discussed in

5.1
Data Models
A data model is an abstract model of how data is represented in a system. In a monitoring
system, the data usually refer to a specific event, but events can be represented in many
different ways. The tools we have looked at in this project usually focus on logs, metrics,
or traces. If a tool can handle more than one type of data, those functionalities are often
split into separate tools. For example, the Grafana Project have devoted tools for each
of the three mentioned data models: Grafana Loki for logs, Grafana Mimir for Metrics,
and Grafana Tempo for traces [53]. This section will explain the three data models and
how they relate to one another in more detail.
5.1.1
Logs
A log is the most straightforward data model type. It represents a collection of informa-
tion about events with timestamps indicating when the events were recorded and in what
order. Each event in the log is represented by a log line containing the information about
that event. The simple structure of logs makes them easy to generate. It is possible
to apply simple instrumentation for logging. An example of instrumenting a PowerShell
program with logging could be to add

provided in a log can help operators investigate an issue or incident that might have
occurred at the host, service or application [54]. However, the logs would often need
to be more specific than the one above to extract useful information from the data in
the log. Logs generally have three forms: plaintext, structured, and binary. However,
fundamentally they contain the same information in all their forms: a timestamp and
a payload of some context. The plaintext logs come in free-form text like the example
above, in contrast to the structured form where logs are represented in JSON format as
in Figure 5.2. Binary logs are, as the name implies, stored as binaries and are often used
as replications and backup recovery.
Figure 5.2: Examples of log lines from structured logs
Figure 5.2 is an example of two log entries from a log formatted in a structured way
using JSON. Using structured logs makes extracting useful information from the data
more manageable, faster, and cheaper. For example, the field name ”path” in the first log
line implies a certain meaning and structure for the corresponding value. Furthermore,
searching, filtering, and aggregating activities become more manageable when placing
key elements into properties. Tools like LogStash or Grafana Loki focus on monitoring
through logs in cloud-native applications. Logs can be useful for providing particular
knowledge about a certain event. But there are also limitations to monitoring tools that
focus on logging [14]:
• Logs can have different formats, as we have seen above. Some logs can have specific
fields, while others can be generic. It can be hard to extract information from a
collection of logs if applications write logs in different formats.
• Logs have a poor signal-to-noise ratio, meaning it is challenging to decide the num-
ber of log lines one should write. Too many make it difficult and time-consuming
to go through logs, and too little does not give enough insight into the application.
• Logs are hard to scale. An event generates each log line, and if instrumented events
happen frequently, this can generate a substantial amount of data. Log lines can be
aggregated based on common field values to reduce storage requirements. However,
the aggregation of multiple logs can result in being a bottleneck for the system
because of the amount of bandwidth it requires to send them to the aggregator.

Often, it is unnecessary to know the details about every event monitored but rather to
have a general idea of what is going on in the application. One way of achieving this is
by using the metric data model that aggregates data by counting occurred events over
time.
5.1.2
Metrics
A metric is a numerical value of collected data measured over time, e.g., the number of
requests handled per minute. There are multiple ways of collecting measurements for
computing metrics, but the most applied approaches are using code instrumentation or
running an agent. Labelling the metrics makes it possible to present them in graphs, take
statistics or use them for alerting on predefined thresholds. The information obtained
from a sequence of metrics can be used to derive knowledge of the behaviour of the
system state. In contrast to logs, metrics can be used to tell whether a problem is about
to occur or why the system behaves as it does at a given time. For example, a high
latency from the application could potentially be explained by looking at the metric for
CPU usage or network traffic for a specific service. This information can provide the
developers with the knowledge that can assist them in identifying where a problem might
be, thereby decreasing the time for incident management. As mentioned in Section 4.6,
one of the challenges of monitoring in a cloud-native environment is the dynamic nature
of the services. Therefore, the traditional way of handling metrics is unsuitable since
metrics in a cloud-native need additional context on where the metric is generated in
the system. For handling this, modern monitoring tools represent every metric with a
name and optional key-value pairs called labels [55]. Applying tags and labels makes
identifying and filtering metrics straightforward. Figure 5.3, illustrates the components
of a metric.

operational costs when using metrics, such as disk utilization or processing, in case of
increased traffic in the application. Since the client-side handles data aggregation, met-
ric storage does not increase proportionally with user traffic. Adding a new service or
instrumenting existing services, as described in section 5.2.2, only increases the metrics
storage with more permutations of label values.

5.1.3
Traces
A trace is a collection of related distributed events that make up the end-to-end flow of
a request in a distributed system. In distributed tracing, traces can be seen as a more
complex way of logging that provides visibility of the path and structure of a request
[56]. A trace represents the entire transaction of an end-user making a request. The
request will traverse through multiple and distinct services that affect the request to
satisfy the client’s request. Looking into the path of a request in a distributed system
can help developers understand the behaviour of a specific service in the context of the
whole system. This, combined with the structure that distributed tracing uses, allows
the developers to understand the exact time and effects of a request. Being able to do
38
==PAGE:46==
Master Thesis
July 6, 2022
this is essential in a cloud-native system where a request is routed through many services
with many different dependencies [23]. The primary building block that a trace consists
of is spans. Each span represents a unit of work done in a distributed system. A Span
generally contains references to other Spans, making it possible for multiple spans to
assemble into a trace. The following list shows the states that a span encapsulates [58]:
• Tags are a set of key:value pairs that allow user-defined annotation of spans, e.g.,
db.instance:”customers”. This makes it possible to query, filter and comprehend
trace data. Spans should be applied to the whole span.
• Logs are also a set of key:value pairs used for capturing span-specific logging mes-
sages from the application. In contrast to tags, logs are only used for informing
about a specific event.
• SpanContext contains data that is immutable and visible across process bound-
aries. The data within a spanContext would always contain trace- and span ID
that refers to the distinct span within a trace.

5.2
Instrumentation
Instrumentation, in the context of monitoring, is everything that can help us collect data
that informs us of the state of a system [23]. There are primarily two different approaches
for instrumenting an application, black- and white-box, that must be considered when
monitoring a system. The two types of instrumentation indicate the degree to which a
system is measurable, with the system being the box.
Black-box instrumentation is based on testing externally visible behaviour that re-
flects how an end consumer would access the system. By applying black-box instrumenta-
tion, it is not possible to access any technical details or build configurations of the system.
The Ops team often uses the data collected from applying black-box instrumentation to
gain an understanding of the hardware resource usage such as disk space, memory us-
age, and CPU usage [60]. This collected data can be used to get a detailed picture of
the system’s state and investigate problems. An example of black-box instrumentation
would be to send a request to the system and check the response. This can be used to
understand whether or not the system is up and running, but not as to ‘why’ this is the
case.
In contrast, white-box instrumentation provides details about the system’s internals,
which can help us understand the reason for the data produced by black-box monitoring
[60]. By changing the inside of the system, we can instrument the source code to gain
important performance insight, e.g., average request latency. It is possible to drill down
in the system using white-box instrumentation and see what internal method is causing
the latency.
It is important to state that both types of instrumentation are equally critical for gaining
useful insight into the system.

5.2.1
Agent-Based Instrumentation
One way of achieving black-box instrumentation is by using agents to collect and process
data. Agent instrumentation belongs to the concept of black-box instrumentation since
an agent is an external process that does not have any internal knowledge of the system
in which it is used.
One primary purpose of agents is to make monitoring easier by
generalising all applications that can use the agent without rewriting code [23]. The
Kubernetes documentation describes two methods for collecting data with agents in a
cloud-native environment [61].
Figure 5.5 is a visual illustration of running an agent on pod-level, which, as the name

directly by querying the service and the infrastructures in which it operates. The collected
data would then be sent to the monitoring device, e.g., SaaS application [23]. A container
inside a pod can access other containers in the same pod, making it easy for an agent to
collect data from other containers. However, this comes with performance issues such as
high resource consumption when too many pods are on a single machine.

5.3
Data Dissemination Models
One of the monitoring activities described in section 3.2 is dissemination of monitoring
information. This activity is concerned with how the data is transferred from the pro-
ducers to the consumers. A producer could be an agent or a service with source code
instrumentation, as described in the sections above. However, in order for the monitoring
information to be useful, it should be transferred to a consumer. This could be a central
storage server, a dashboard or another place where an analyst can access the monitoring
data. There are two fundamental ways this data can be transferred from producers to
consumers: The push model and the pull model [62]. They have different pros and cons,
and monitoring tools will usually be devoted to one of them.
5.3.1
The Push Model
The dissemination model that is most common in monitoring tools is the push model.
In this model, the producers are responsible for emitting their data to subscribers. In
43
==PAGE:51==
Master Thesis
July 6, 2022
this context producers are the services of the system. Figure 5.7 shows a system with
three services and a monitoring server. The services produce monitoring information,
which they immediately send to the server. In this model, the service must be aware
of the central service.
The data is usually transmitted as UDP packages to improve
performance. The push model is useful in a dynamic environment where services are
created and destroyed with short notice. The monitoring server does not have to be
aware of new services, the only requirement is that the services know where to send the
data. This also means that dissemination is easy to set up. Once the server is up and
running, the services just need to know where the server is, and then the data flow can
be initiated.
Figure 5.7: The push model for data dissemination.
5.3.2
The Pull Model
The counterpart of the push model is the pull model. In the pull model, the consumers are
responsible for collecting data from all producers. For example, a service could contain
an agent that collects data and exposes the data in an API. A central server can access
this API to scrape all data from the agent at regular intervals. Here the services do not
need to know anything about the server, they only need to make their data available.
But the server needs to know all the targets that should be scraped at any time. This can
be complicated to implement if there are many services. It is not feasible to hard code
the addresses of the targets since these addresses can change, and targets can be created
or destroyed at any time. To solve this, the server must implement service discovery.
This is an automated process that detects available targets. Service discovery can be
implemented in many ways. Many applications use Kubernetes for orchestration, which
supports service discovery integration with many monitoring tools.

5.3.3
How to Select a Model
While both push and pull models can fulfil most requirements for a monitoring solution,
they still have strengths and weaknesses. It is good for developers to be aware of this
when selecting tools because some features can be more challenging to implement with
specific data dissemination models. As it will be apparent in the tool survey in section
6.2, the push model is popular, but some tools also use the pull model. Table 5.3 shows
some concerns regarding the data collection and which of the two models provides the
best solution. The concerns are a combination of issues described by Haselb¨ock et al. [3]
and by Statkeviˇcius’ blog post [63].
The first concern is preventing data loss. For some companies, it can be very important
that monitoring data is never lost, e.g., if the data consists of logs with transaction
information.
In this case, a push model might be the best choice.
In a pull model,
the data is stored locally until the server scrapes the service. If the service terminates
unexpectedly before being scraped the data is lost. Some services might also have very
short lifetimes by nature. If the scraping interval exceeds the service lifetime, the data
can be lost because the service is destroyed before being scraped. By contrast, a service
with a push model sends its data immediately as it is recorded. There is often no local
storage, and the risk of data loss is much smaller.
Reducing network traffic can be important if network bandwidth is a bottleneck for
the company. This is best achieved with a pull model. Here it is only the server that
produces network traffic by pulling data. The server can control the pulling intervals.
The data is often aggregated locally on the services, which generally reduces the number
of packages sent. Overall, the pull model provides more control over when and what
data is transferred. The server can be instructed only to pull the data when needed, and
some tools might support rejecting services with too much data that might overload the
45
==PAGE:53==
Master Thesis
July 6, 2022
system. There are no such mechanisms in the push models, where all services send their
data right away to the server in separate packages.
Supporting multiple ingestion points can be a problem in a pull model.
If two
servers pull from the same service, they might not get the same data because they will
pull at different times. So multiple servers are not a good way to achieve data redundancy
for backups in a system with a pull model. This is much easier to accomplish in a push
model, where the services can choose to send identical data to multiple ingestion points
at the same time.
Availability monitoring is often a build-in part of a pull model.
The server must
include service discovery, so it already knows which targets are supposed to be sending
data. It is harder to monitor availability in a push model because the server by design is
not required to know where the data comes from. The tool will need to estimate which
services are expected to send a status and which are not.

of scrape targets. But if different firewalls or other safety measures protect some services,
then a pull server might need to perform extra steps to access those services.
On the other hand, push models do not require service discovery, and all services will
need to perform the same steps to access the monitoring server. But it can become less
manageable because all services need code identifying where the server is. If the address
of the server changes, all services will need updates.
One might also wonder which of the two models has the best scalability. But for now,
there is no consensus on this in the industry. As Chakraborty et al. [13] points out, a
pull model puts the entire data collection workload on a central server, making it act
as a monolithic monitoring silo. Push models usually use UDP packages to send data
efficiently, as opposed to the more costly TCP connections often used in pull models.

However, the popular monitoring tool Prometheus is widely used in the industry, even
though it uses a pull model. A recent survey from CNCF shows that 83% of respondents
from their cloud-native community use Prometheus [48]. The authors of Prometheus
argue that the cost of TCP connections is negligible compared to the cost of processing
the metrics and that Prometheus can still scale to millions of machines without issues
[64]. This is probably much more than any small companies targeted by this project
will encounter. Pull models can solve this with horizontal scaling if additional scaling is
needed. Other servers can scrape Prometheus servers, so they achieve horizontal scaling
by having a hierarchy of servers.
In the end, pull and push models should not decide which monitoring tools to use. But
both models have advantages and disadvantages, and choosing a model that fits the
application can reduce the system’s complexity. In a podcast called ”The New Stack”,
creator of Prometheus Brian Brazil says:
In reality, the question of push versus pull largely does not matter, because in
either case there is advantages and disadvantages, and with some engineering
effort you can work around them in both cases [65].
Based on the findings of this section and the quote above, we have decided not to in-
clude data dissemination models in the decision guidance model in section 6.1. While it
might be worthwhile for small companies to avoid what Brian Bazil refers to as ”some
engineering effort”, it should not be a deciding factor.
5.4
Summary
In this chapter, three different components of a monitoring tool have been identified
and analysed. The first one is the data model. Logs, metrics, and traces are identi-
fied as the most common ones. The type of instrumentation is the next component of a
monitoring tool. Tools can utilise agent-based instrumentation or source code instrumen-
tation. Agent-based instrumentation is black-box monitoring where the tool is not aware
of the internals of the application. In contrast, source-code instrumentation is white-box
monitoring. The last component is data dissemination models. Tools can perform data
collection with push or pull models, which each has its pros and cons.
The goal of the analysis has been to categorise tools and investigate the building blocks
they consist of. This categorisation will be used in the following chapter to decide which
tools are most suitable for different monitoring approaches.

6.1.1
First Objective: User Experience
We have identified two main concerns that are directly related to user experience. There
can be many other concerns that affect the user experience. Some of them overlap with
the other objectives, and some may be specific to an industry or a customer segment.
• Availability: It will always give a bad user experience if a system, or parts of it,
is unavailable. The easiest way to monitor the availability is to use health checks.
Health checks are periodical pings that ensure that a service is in a healthy state.
This can be done by having an external agent ping all endpoints in the application
50
==PAGE:58==
Master Thesis
July 6, 2022
that are publicly available.
The status of the endpoints can be displayed in a
dashboard, for example, styled like an andon board with red and green colours.
The health checks can also include service endpoints that are not publicly available
if the agent is authenticated to access them. In some cases, the agent may need to
be deployed as a node-agent. That way, the agent runs in the same environment as
the services, so it can access them even if they are unavailable from outside. Health
checks are often easier to incorporate with monitoring tools that use a pull model,
as it was argued in section 5.3. These tools have a server equipped with some sort
of service discovery, so the server will know which services are expected to work and
where they are located. In a push model, the services can still push their status to
the server. But it is harder for the server to determine which services are expected
to send statuses and whether missing messages should be interpreted as the service
being unavailable.
• Performance: One thing is the availability of the service, but the user experience
is also affected by the performance of that service. The performance of a service
can be assessed in different ways. One way is to monitor the API performance.
This entails monitoring metrics like average latency or throughput on API end-
points. Usually, the metrics are calculated with source-code instrumentation of the
API, and then the data is collected to a central storage. Another approach that
addresses performance is to monitor interservice communication. This can be done
using distributed tracing by tracking requests sent back and forth between services.
This approach can show performance bottlenecks and give a good overview of the
data flow in the system. The tracing tools require source code implementation, as
mentioned in section 5.1.3.
Availability will often be the easiest to address of the two concerns above. Some tools, like
Prometheus, even provide this functionality out of the box with minimal configuration.
Performance can be monitored in more or less detail, depending on what is needed. Both
approaches described here most likely require source code instrumentation.
6.1.2
Second Objective: Incident Management
Monitoring can help find the cause of an incident and take appropriate action. As de-
scribed in Chapter 2, incident management focuses on detecting and mitigating incidents.
We have identified two concerns that are particular to improving incident handling in
cloud-native environments through monitoring:
• Error Management: Proper error management allows developers to track errors
and find the root cause of an incident. The nature of cloud-native applications
makes it very difficult to detect errors since services are distributed.
Errors in

Master Thesis
July 6, 2022
another service might cause errors in one service, and different services can be
responsive even though they are in a faulty state. A way to address some of these
difficulties is to use error tracing. If there is an error, traces can be used to reproduce
the sequence of events that led to that error, making it easier to detect a root cause.
A more simplistic approach is to use error logs to identify problematic events in
the system. However, one should be aware that this makes it hard to track errors
across services. Additionally, error management can be improved with an alerting
system. An alerting system will produce alerts based on metrics or specific events
that notify stakeholders of something is wrong in the application. Alerts can be
used for many monitoring aspects, but the critical nature of incidents makes them
very applicable for this objective. Error tracing, logs, and alerts can be combined
to improve error management.
• Network Security: As mentioned in chapter 2, security is not in the scope of
the project. However, we will briefly mention here that network security is another
monitoring concern that can improve incident handling in case of a security breach.
Suspicious behaviour can be detected by monitoring the network traffic. This can
also help determine whether an external or internal actor caused the incident.
Incident management can also be improved by having workflows and pipelines that enforce
code reviews, versioning, and automatic testing when new code is introduced to the
system [19]. Such constructs would make detecting bugs and errors in the development
process more manageable. But they are not related to run-time monitoring and will not
be investigated further in the project.
6.1.3
Third Objective: Resource Management
A core part of cloud-native systems is how they use resources. The resources can be
distributed across many different services. The system should be able to deal with varying
demand levels through vertical and/or horizontal scaling (see section 4.2). A monitoring
concern within resource management is how well the services scale. Another concern is
capacity planning.
• Scalability: At its core, scalability is whether or not the system can do X% more
work when deployed on X% more resources compared to a base configuration [68].
If the system is not very scalable, the design should probably be refined. A cloud-
native architecture that does not scale well is much less appealing. Scalability can
be monitored by looking at different OS metrics for a service, like CPU, memory,
and storage usage. In practice, this could be achieved by deploying a node-agent
that monitors all pods on that node through the OS. The agent can collect data on
the resource usage of the service.

July 6, 2022
• Capacity planning: Both Aceto et al. [10] and Fatema et al. [9] highlight ca-
pacity planning as an important monitoring feature.
The demand on different
parts of a system can change over time. The fluctuation in demand can be use-
ful for companies to know if they have sufficient resources available to maintain a
high-performance level. Cloud providers will ensure that resources scale with the
demand, but depending on what subscription model the cloud provider uses, the

6.3.1
The Configuration Model
To quantify the complexity of a configuration of a set of tools, we will refer to Keller

Even though the article is from 2007, it operates with a
notion of configuration that fits very well with the process of setting up monitoring tools:
[configuration is] a process whereby individual components are assembled and

Furthermore, the complexity of a configuration is defined as:

The configurations used in this project are expected to be carried out by a human system
manager or a similar role. A goal of the project is to make the configuration process as
easy as possible. With the definitions above, the presented complexity measures should be
applicable to the project. Keller et al. present thirteen measures across three categories:
execution, parameters, and memory. A subset of these measures has been selected for use
in the project.
Before the selected measures can be explained, the underlying system model needs to
be established. Keller et al. use a system model representing an IT-system by nested
containers. Presumably, containers had a different meaning in 2007 than they do now,
so to avoid confusion, we will refer to them simply as contexts. Keller et al. describe
them as either a hosting environment or representations of system resources that have
configuration controls. A configuration control can be an adjustable property or an in-
vokable method. For example, a database might have configuration controls like changing
the access policy or the request timeout. Here we will add a small extension to Keller’s
definition. We will also consider source code instrumentation as a configuration control
in cases where the source code is fully available to the developer. Figure 6.3 shows an ex-
ample of a system with Kubernetes. The system has a service, a monitoring agent and a
monitoring service. This corresponds to the sidecar pattern illustrated in Figure 5.5. But

the figure, Kubernetes, pods, containers, and applications are defined as contexts nested

Keller et al. break down a configuration into a sequence of configuration actions to be
performed by the operator. The level of abstraction can vary, but an action should rep-

action to look up data. All actions are assumed to be known in advance when assessing
the complexity. The same goes for the order of the actions. There will sometimes be
optional actions in the configurations we want to assess in this project. The operator
decides whether or not to carry out an optional action. For example, the operator may
want to change a default option to something different. To ensure that the configura-
tion is deterministic, we assume that optional actions are always chosen. This way, the
complexity of the configuration is never underestimated.

6.3.3
Computing Complexity Ratings
In the section above, four different measures were described. In order to be able to rank
configurations against each other, these four measures should be combined to a complexity

6.4
The Tool Selection Algorithm
Each configuration can now be associated with a complexity rating with the procedure
described above. This complexity rating gives an idea of how hard it is for a tool to im-
plement a monitoring approach. Now an algorithm is described that uses the complexity
rating to select the optimal set of tools. The algorithm should produce a monitoring
solution that addresses all the selected monitoring approaches. As mentioned in chapter
3.2, we define a monitoring solution as a set of tools and configurations. The solution
will describe which tools should be used and how they should be configured.
Given a set of approaches A and a set of tools T, let C be the set of configurations where
ca,t ∈C if tool t ∈T can implement approach a ∈A with configuration ca,t. A monitoring
solution S is a subset of C, where ca,t ∈S if approach a should be implemented by tool
t using configuration ca,t. This way, a solution contains information about all the tools,
their configurations, and how they address the approaches. The tool selection can be
defined as a function f that takes in a subset of A to produce a solution S. Thus, the
domain of f is the power set of A, i.e. the set of all possible subsets of A. The formal
notation for this is P(A). The range of f is all possible solutions, i.e. P(C). This can be
written as:
f : P(A) →P(C).
(6.2)

8.1
Addressing the Research Questions
This section will answer the research questions defined in the project’s initial phase in
chapter 1.2. However, the research questions have been considered and covered through-
out the project. Therefore, the answers in this section should only be seen as a summary
of the relevant chapters for in-depth explanations.
[RQ1]: Why would businesses using cloud-native applications want to use monitoring?
In chapter 2 we present the motivations for using monitoring in software applications.
These motivations include better user experience, incident management, and resource
management, which ultimately leads to increased quality and reliability of the appli-
cation. The motivations are identified based on scientific studies and surveys of cloud
applications. The research does not mention the term ”cloud-native”, and we were not
able to identify articles that specifically target that domain. However, since cloud-native
applications are a category of cloud applications, it is assumed that the motivations
transfer well.
[RQ2]: What are the challenges of selecting, configuring and using monitoring in cloud-
native applications for small companies?
In section 4.6 the challenges of monitoring in cloud-native applications have been pre-
sented. The findings were that a lack of standards, an abundance of tools and a lack of
appropriate skills and time make it very difficult for companies to obtain a good monitor-
ing solution for their system. The section also explores some more technical challenges:
the variety of data, the interaction of microservices, the dynamic environment, and het-
erogeneity.
The challenges from section 4.6 are not directly related to the size of the company be-
cause there is limited research that explores this relationship. However, the challenges of
82
==PAGE:90==
Master Thesis
July 6, 2022
handling monitoring that comes with using cloud-native are assumed to be reflected, or
even amplified, for small companies.
[RQ3]: Which limitations do the available monitoring tools have? Are there cases where
they cannot be used?
Monitoring tools are often specialised to work in a certain way, which can limit how they
can be used. In chapter 5 the different components of a monitoring tool are explored.
One of the components is the data model. The tools often use one of the three data
models: logs, metrics, or traces. Tools that use one data model cannot be applied to
problems that require another data model. For example, Logstash solely focuses on logs
and does not have the functionality of accumulating errors and representing them in a
metric. But the limitation can be overcome by adding a different monitoring tool to the
system, such as ElasticSearch. The consequence of too specific monitoring tools results
in companies using many tools.
The instrumentation is another component limiting a monitoring tool, as described in
section 5.2. Some tools target white-box instrumentation by changing the source code,
which allows the tool to record a service’s internal state. However, source code instru-
mentation requires more work to set up, and it increases maintenance. Other tools only
rely on external agents, which allows monitoring without changing the source code, but
it also limits what can be monitored.
[RQ4]: Which criteria do small companies apply to decide what monitoring tools to use?
Which criteria should they use?
Little research exists on the different criteria that small companies consider for selecting a
monitoring solution. However, the survey described in 4.6 indicates the most important
criteria when selecting monitoring tools.
The top-ranking criteria CNCF community
members have voted for is that the tool should be open source. But the survey’s sample
size is relatively small, and the possible answers to the question are predefined.
The predefined questions in the survey do not concern the components that make up the
monitoring tool. In Chapter 5 we describe the different components and their associ-
ated challenges. The components, data model, instrumentation, and dissemination, are
important criteria that a company should consider for using the best approach to solve
business needs.
[RQ5]: How can the process of selecting, configuring and using monitoring tools be eased
for small companies?
In chapter 6, the first challenge we address is the abundance of tools to select from.
The selection process can be eased through the categorisation of the available tools.
Categorisation makes it easier to navigate in a field where the tools lack standards and
to get an overview. Hence, the decision process is also simplified since it is easier and
less time-consuming to find tools that fit the company’s needs.
Another challenge is

that selecting and setting up monitoring tools requires specific knowledge about the tools
and cloud-native technologies. This challenge can be reduced by collecting and sharing
knowledge about the tools. Additionally, automatic tool selection and setup will minimise
the learning curve of using monitoring in cloud-native environments.
The above challenges are addressed by the method developed in chapter 6.
Here, a
method is introduced for how to select, configure, and use monitoring tools. The method
uses a decision guidance model that categorises monitoring tools to make the selection
process easier. It allows companies first to formulate objectives for monitoring and their
associated concerns. The method will find concrete approaches and recommend tools
based on those approaches. It has been necessary to analyse the components of monitoring
tools to determine which approaches a tool can handle. This is done in chapter 5.
The method has been implemented in an interactive tool with a user interface that stake-

companies to select a set of objectives they want to achieve with monitoring and get back
specific instructions on which monitoring tools they should use and how to set them up.
The system also allows extensions, like introducing new monitoring tools, approaches, or
selection algorithms into the backend.

in more detail in section 6.2.

8.4
Validity
In this section, we will examine the results achieved during the project and their validity.
The results of this project are the method composed of a decision guidance model, a
survey, and an algorithm, all described in chapter 6, as well as the artefact presented in
chapter 7.
The decision guidance model was based on the model developed by Haselb¨ock et al.
[3]. We have selected elements to use in the model from literature within the field of
cloud monitoring. The objectives, concerns, and approaches are picked from different
sources and grouped based on our assessment. The model’s validity is challenged by
the fact that the literature focuses on cloud monitoring, not specifically monitoring in
cloud-native environments. Also, the literature does not mention the size of the company.
It is an assumption made in the project that the needs and challenges of companies of
generic size transfer to small companies. One way to ensure the assumptions are valid is
to perform a questionnaire and evaluate whether needs and challenges are aligned with
small companies. To some extent, the method has been validated through evaluation with
Developyn. However, a more thorough evaluation is needed to strengthen the validity.
This evaluation should be based on more small companies.
GitHub stars have been used to select the tools for the survey. A threat to validity is
that GitHub stars might not precisely represent which tools are used in the industry.
The GitHub stars are biased towards older tools, and the stars do imply that the tool
is being used. A star simply indicates interest in an open source project. In addition, if
one were to use commercial tools, it would be impossible to rank their popularity using
GitHub stars. An alternative approach would be to use one of the existing monitoring
87
==PAGE:95==
Master Thesis
July 6, 2022
tool surveys. Other surveys have been conducted in the field of cloud monitoring tools.
But the surveys are either outdated, not applicable to cloud-native environments or do
not have a rigid justification of how they have selected tools to include in the survey.
The selection algorithm in the method uses the complexity of configurations for the
monitoring tools. The configuration complexity requires human interpretation. It must
be calculated manually by defining the abstraction level of the actions, the context they
are applied to, and what parameters are included. Such ambiguity is a threat to validity.
Optimally, the complexity computation should be completely deterministic and unbiased.
Another concern is that the algorithm only considers configuration complexity during
selection. One could argue that the complexity is insufficient to identify the most suitable
tool.
The evaluation strategy used for the artefact has been to conduct an interview with
a company. The company fits with the target of this project because it is small, uses
cloud technologies and needs monitoring. However, since the evaluation only includes one
interview, it is hard to make conclusions based on it. To properly evaluate the artefact,
more interviews would need to be conducted, or different evaluation strategies should be
used. The assumptions made in the project could be supported by conducting a controlled
experiment and a field study with the artefact.
8.5
Future Work
In this section, we will describe the next steps that should be considered to improve the
developed method and the artefact.
The artefact developed in this project is a proof-of-concept. Not every configuration is
implemented and analysed. Before using the artefact in a business context, the configu-
ration data for all scenarios should be included in the system.
The current method does not consider information about the company’s system infras-
tructure. The quality of the recommendation could be improved if the artefact included
user input. For example, knowing the monitoring tools the company already uses makes
it possible to ensure that the recommended tools integrate well with existing tools. Ad-
ditionally, the artefact assumes that companies use Kubernetes and Docker. However,
this may not be a valid assumption, and it would be optimal to have alternative cloud
technologies as user input. Programming languages are another possible user input that
could make the recommendation more specific toward the user’s system. For instance,
steps regarding source code instrumentation can include language-specific libraries. Fi-
nally, configuration complexities could be computed without making assumptions about
the number of services if this was user input. Consequently, the estimated configuration
complexity would be more accurate.
88
==PAGE:96==
Master Thesis
July 6, 2022
Another concern about the configuration complexities is that the algorithm uses the sum
of complexities for each tool to make a recommendation. Instead, it should use the com-
plexity of configuring all recommended tools together. In general, more research should
be put into the way configuration complexities are calculated. The current method leaves
room for interpretation because it is unclear what the contexts are for that configuration.
Computing complexities should be deterministic and preferably automatic.
Furthermore, the algorithm used in the project can be improved. Right now, it only
considers configuration difficulty and the number of tools. However, the findings in section
4.6 suggest that stakeholders consider other parameters when selecting monitoring tools.
For example, the algorithm could include measures of the maturity and feature breadth
of the tools. The algorithm could also be extended to consider commercial tools. This
would enable it to have recommendations on cloud platforms like Azure or Amazon Web
Services.
Lastly, the decision guidance model would be improved with further company evaluation.
Company feedback could confirm that the elements in the decision guidance are applicable
in the industry. The evaluation should target small companies that utilise cloud-native
technologies and concepts.

CHAPTER9
Conclusion
In this thesis, we have developed a system to guide stakeholders of small companies
through every step of selecting, configuring, and using monitoring tools. The system
explicitly targets applications that make use of cloud-native technologies and concepts.
Using this tool, stakeholders can choose a set of objectives they want to achieve with
monitoring and get back a recommendation of a monitoring solution.
A monitoring
solution consists of tools with descriptions, setup guides, and configuration files.
We have used a decision guidance model to convert business needs to technical monitoring
approaches. This is made possible by identifying monitoring objectives and categorising
concerns and approaches into these objectives. The categorisation allows all company
stakeholders, technical and non-technical, to influence the decision process of selecting a
monitoring solution. In this thesis, components of a monitoring tool have been defined
that describe implementation and functionality on a conceptual level. These components
have been included in the decision guidance model to select tools that address the identi-
fied approaches. The tools in the model have been selected based on a survey of available
tools in the field.
One of the system’s central aspects is automated monitoring tool selection. The system
achieves this through a selection algorithm. The algorithm computes the set of tools
that addresses all the selected approaches with minimal configuration. A collection of
configuration complexity measures has been implemented in the algorithm to estimate
the difficulty of configuring each tool.
We expect the system to reduce the knowledge and time needed for a small company to
achieve an appropriate monitoring solution. The system aims to minimise the learning
curve for monitoring in cloud-native environments. A challenge independent of company
size is the lack of standards and abundance of tools. The system addresses the need for
standards and seeks to filter the available tools based on company needs.
Additional evaluation should be performed to prove that the system achieves these goals.
Following the design science principles, the system has been evaluated using an informed
argument based on relevant research and an interview in collaboration with Developyn.
They had reservations about using the tool in its current state, but believed that it could
give value if more work was put into it. Given more time, a controlled experiment and a
field study would support the assumptions made in the project.
