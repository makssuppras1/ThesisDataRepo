Summary
The main focus of this project is to propose an obstacle detection and avoidance solu-
tion for indoor mobile robots. The solution should not only provide means to avoid
collisions with obstacles present in the environment but also acquire information about
the localization, the class and the shape of encountered objects.
A deep learning approach is used to provide means for the object classification task.
The state-of-the-art image processing network is combined with a SLAM algorithm to
generate semantic pointclouds of the environment. The last part of the project is fo-
cused on pointcloud processing - providing methods for instance extraction and instance
processing.
The proposed approach proves to meet all of the project’s goals and through experi-
ments the performance is evaluated.

CHAPTER 1
Introduction
Obstacle detection and obstacle avoidance are one of the most crucial tasks for a safe
operation of autonomous robots in an unknown environment.
The obstacle detection process allows the robot to localize and map obstacles and
objects that are encountered during the robot’s operation.
To accomplish this task
multiple types of sensors might be used, including RGB cameras, RGB-D cameras and
laser range finders. The most common and the most basic approach to the obstacle
detection problem is to use sensors to detect occupied areas and mark them as obstacles.
In such case, the robot is aware of the placement of obstacles but lacks any information
about the type of the detected objects. This approach is sufficient when the objective
is to maneuver around obstacles but doesn’t provide any additional information, that
might be useful when interacting with encountered objects.
The obstacle avoidance is a task during which the robot fulfills an objective (for
instance, navigating to a given position) while avoiding collision with obstacles present
in the environment. Therefore, it is clear that that the obstacle avoidance process is
highly dependent on the obstacle detection process.
The main focus of this project is to enhance the basic obstacle detection process
by expanding it with an obstacle classification solution.
In the result, not only the
occupancy information will be acquired but also the semantic classification for each
detected object will be determined. Such approach allows to not only navigate around
objects but also interact with them depending on the obstacle class (for instance a chair
might be moved, whereas a table should be maneuvered around). The information about
the class of detected obstacles can be passed to the obstacle avoidance algorithm and
expand it’s capabilities of the obstacle interaction.

CHAPTER 2
Project Overview
The main goal of the project is to provide the robot with an obstacle detection system
that not only acquires the information about occupied areas but also equips the robot
with information about the class, the position and the shape of encountered objects. To
achieve that goal it is necessary to combine different sub-systems together and in this
chapter the analysis of the problem and the overview of the proposed solution will be
presented.
2.1
Guidebot
To be able to perform experiments in the real-life scenario and verify the performance
of the system it was necessary to use a mobile robot. For this purpose a DTU robot -
Guidebot was used. It is a wheeled mobile robot equipped with a range of sensors. The
on-board sensors that might be used for the obstacle detection task include a Kinect
camera (XBOX 360) and a laser scanner (SICK LMS200-30106). Additionally, to localize
the robot in the environment the sensory information from wheel encoders might be used.
2.2
Related work
As the obstacle detection and obstacle avoidance are crucial tasks in the autonomous
robot’s operation, there is a lot of research done to improve these processes. However,
there are not many projects focusing on a full solution that would fulfill the goals of this
project.
There’s been several attempts to provide solutions capable of generating semantic
maps - maps in which each area is semantically classified. SemanticFusion [McC+16]
combines an adaptation of, at the time, state-of-the-art Deconvolutional Semantic Seg-
mentation network [NHH15] with an ElasticFusion SLAM [Whe+16] algorithm and a
Bayesian update scheme. This approach allows authors to generate accurate semantic
3D maps of the environment. It has to be noticed that this solution does not provide
any information about instances of objects.
PointNet [McC+16] is a deep neural network that directly consumes pointclouds. The
network provides an end-to-end pointcloud processing capabilities, outputting a variety
of results including: object classification, part segmentation and semantic segmentation.

Similarly to SemanticFustion, this approach does not provide instance detection so the
solution would need to be extended to fulfill the project’s requirements. Moreover, to
meet the project’s objectives, PointNet would need to be combined with a localization
and mapping algorithm to allow to build a global semantic map of the environment.

2.3
Analysis
An analysis of the problem was conducted to propose a solution that will fulfill all of
the the project’s objectives. The designed solution should work with the DTU robot -
Guidebot. This condition limits possible approaches to only these that can work with
available sensors. Moreover, it was decided that the Kinect camera should be the only
sensor used for object detection.
The choice to ignore other sensors (such as laser
rangefinder) was made to minimize the cost of the solution and make it as simple as
possible.
The Kienct camera acquires RGB-D images, which can be simply transformed into
corresponding pointclouds. A pointcloud provides sufficient amount of information to
localize occupied areas - obstacles.
However, the crucial part of the project - the classification of detected obstacles -
still needs to be fulfilled. As the only available sensory input is the camera image, the
solution has to be based on a computer vision algorithm. The computer vision module
should be able to divide the input image into segments containing different object classes.
This task can be achieved by an image segmentation algorithm. Once a segmented image
and a corresponding depth image are available it is easy to generate a pointcloud that
contains information about classes of obstacles. It was initially considered, to use a
==PAGE:17==
2.4 Modules
5
solution similar to PointNet and provide an end-to-end pointcloud segmentation. This
idea was however dropped, due to the initial lack of a training dataset and uncertainty
about the performance of such solution.
So far the solution can generate a single pointcloud for each of the captured images.
To fulfill the project’s objectives it is necessary to combine pointclouds taken at different
positions of the robot into a global pointcloud. Such operation will allow the robot to
remember the localization of obstacles and gather detailed information about the shape
of the encountered objects. To compile a full map of the environment it is necessary
to localize the robot and then transform the acquired pointcloud into global coordinate
system. The approach used here is similar to the SemanticFusion algorithm - combining
an image segmentation neural network with a SLAM algorithm.
Once the global semantic pointcloud with class annotations is generated, it is neces-
sary to add an sub-system that can process the global pointcloud. This modules should
be able to extract information about single object instances (localization, model, class)
and process the instance models to allow interaction with objects.
Such system fulfills all of the requirements.
2.4
Modules
The proposed system consists of four main modules:
• Image acquisition - this module is responsible for the acquisition of RGB and
depth images from the Kinect camera.
• Image segmentation - this module is responsible for processing of the acquired
images. The performance of this module has a great impact on the whole system
as this module adds the semantic meaning to the acquired images. The output
images include annotations about detected objects, their classes and localization
in the image.
• Robot localization and mapping module - this module is responsible for keep-
ing track of the robot position and simultaneously creating a semantic pointcloud
of the environment. The semantic pointcloud is a pointcloud in which each of
the points has a value that represents the class that it belongs to. The semantic
pointcloud is created by expanding the segmented RGB images into the depth
dimension using depth images.
• Instance detection module - this module is responsible for splitting the created
semantic pointcloud into separate object instances. The output of this module is
the localization (in the global coordinate system), class and the shape (represented
by a pointcloud) of each of the detected objects.

CHAPTER 3
Image segmentation
Image segmentation is a computer vision process in which the input image is divided into
segments that make understanding and further processing of the image easier. One of
the examples of the image segmentation is the process of semantic image segmentation
during which a semantic meaning is assigned to each of pixels of the input image.
The result of the semantic image segmentation can be seen on the Figure 3.1. On
the left there is the input image and on the right the result of the semantic image
segmentation. The output pixels have values corresponding to assigned classes.
Figure 3.1: The semantic image segmentation [NYU18].
The process of semantic image segmentation is an essential part to this project as it
allows to detect, localize and determine the shape of objects in the robot’s environment.
3.1
Deep learning architecture selection
In recent years, deep learning networks have been gaining a lot of attention, especially
in the computer vision field. The rapid development of the field can be attributed to
harnessing powerful Graphical Processing Units for computation. It allowed to train
and test networks with multiple layers and millions of parameters. These multi-layered
networks have achieved state-of-the-art results in different tasks of the computer vision
field such as object recognition, image segmentation and even image generation. The
==PAGE:20==
8
3 Image segmentation
unmatched performance of these networks is the main reason why a deep learning so-
lution was chosen to be used in the project. Several different state-of-the-art solutions
were considered to achieve the segmentation of the input image and advantages and
disadvantages of each of them were considered.
The first of the considered architectures is SegNet[BKC15] which performs a semantic
pixelwise segmentation of the input image. The semantic pixelwise segmentation is a
process in which the input is a regular RGB image and the output is a corresponding
image in which each of the pixels has a semantic class label assigned to it. This approach
provides a lot of information about the environment in the input image as each of the
pixels is assigned a semantic meaning. On the other hand, pixelwise segmentation is
computationally expensive and for the purpose of this project it is not necessary as it is
sufficient to detect only the objects of interest.
Figure 3.2: The input and the output of the SegNet [BKC15].
Another architecture that was initially considered is the You Only Look Once (YOLO)
network[RF18]. This network is used for object detection, meaning that the input to the
network is a RGB image and the output are the locations of classified objects. One of the
initially considered approaches was to use a neural network only to identify bounding
boxes for the objects and then perform the pointcloud segmentation as post-processing
task [MRA17]. It was decided that such approach would require significant customiza-
tion for each of the classes. Moreover, it seems that the performance of the whole system
would be inferior as the most complex process - the segmentation - would not be done
by the state-of-the-art neural network anymore.

The last of the considered solutions is a deep neural network called Mask-RCNN[He+17].
The Mask-RCNN is a neural network architecture created in Facebook AI Research. This
network processes the input RGB image and provides locations of the classified objects
together with segmentation masks. This approach is in a way a fusion of the SegNet
and YOLO networks. After in-depth tests and verification of feasibility of the solution
based on Mask-RCNN, that network was selected to be used in the project.
3.2
Mask-RCNN description
To understand the architecture of the Mask-RCNN network, it is beneficial to initially
introduce basic layers used in neural networks, followed by predecessors of the Mask-
RCNN - the Fast-RCNN[Gir15] and the Faster-RCNN[Ren+15]. These architectures
are used as building blocks of the Mask-RCNN so their understanding is essential for
understanding of the Mask-RCNN.

3.2.1.2
Layers: Convolutional and maxpooling layers2
Convolutional layers are commonly used in computer-vision tasks as they allow the
network to learn banks of filters that can be convoluted with input images to extract
image features. The network is able to learn these feature extractors based on training
examples (each training example consists of an input image and a corresponding labelled
output).
The main objective of the convolutional layer is to extract features from the data.
Usually the layers are stacked in series with the initial layers finding low-level features
and proceeding layers extracting high-level features.
One of the big advantages of convolutional layers is that they re-use parameters by
learning just a small filter window and then shifting it through the input data. This
requires considerably less computational power and memory than the densely connected
layer. Moreover, the solution is shift-invariant meaning that the model will find the
feature no matter where in the image image it’s located.

A commonly used architecture combines convolutional layers with maxpooling layers.
The maxpooling operation is a simple process that allows to downsample the layer’s
input. Maxpooling is done by shifting a filter-window through the input data and for
each step only the maximum value is kept while the others are discarded.There are two
parameters that describe the maxpooling operation: window size and a stride. The
window size describes the shape and the size of the window that is being used, whereas
the stride is the number of samples by which the window is moved at each step. The
maxpooling procedure is illustrated on Figure 3.5.
An architecture based on convolutional-maxpooling layers allows to represent the
features of the input image in latent space that has smaller dimensionality than the
input but preserves the input information.
Figure 3.5: The illustration of the maxpooling procedure with a 2x2 window and a
stride of 2.
3.2.2
Fast-RCNN overview
The Fast-RCNN network is an object detection architecture. The task performed by
object detection networks is to find, localize and classify objects of different classes in
the input image.
Therefore, the input of the Fast-RCNN are RGB images and the
produced outputs are bounding boxes and classification labels.
The Fast-RCNN consists of several steps that the input image goes through before
the output is produced. The initial operation of the Fast-RCNN is to process the input
image through a fully convolutional network (convolutional and maxpooling layers) to
extract the full feature map of the input image. Different feature extraction networks
can be used for this step including VGGNet[SZ14], ResNet[He+15] and Feature Pyramid
Networks (FPN)[Lin+16] which tend to be popular choices. The result of this step is a
full feature map - a reduced representation of the input image. The Fast-RCNN archi-
tecture introduced the idea of re-using the extracted feature map for different regions of
interest.
==PAGE:24==
12
3 Image segmentation
The next step is acquiring regions of interests - areas of the input image that are likely
to contain object. To achieve this task, the selective search algorithm is applied. As this
algorithm is replaced in the Mask-RCNN network, the details about that operation will
be omitted. The proposed regions of interest are used in the layer called RoI pooling to
extract the corresponding parts of the full feature map. The output at this point is a
fixed-size feature map that corresponds to the currently processed region of interest.
During the final phase, the Fast-RCNN processes the generated region of interest
feature map to produce network’s outputs.
The first output, class probability, is acquired by feeding the feature map through
series of fully connected layers ended with a softmax layer.
The softmax layer is a
common choice for the final layer in the classification task as it produces the output
that can be treated as probability (sum of the all output values is 1).
As the initial region of interest rarely provides a precise bounding box for the detected
object, it is required to refine the initial prediction. To achieve that goal there is another
output of the Fast-RCNN - the per-class bounding box offset, which provides information
necessary to adjust the initially selected region of interest.
Figure 3.6: The structure of the Fast-RCNN network.
3.2.3
Faster-RCNN overview
One of the bottlenecks of the Fast-RCNN architecture is the selective search algorithm
mentioned in chapter 3.2.2. The selective search algorithm is an order of magnitude
slower than solutions based on neural networks[Ren+15]. To solve that problem the
Faster R-CNN architecture introduces one significant change - the Region Proposal
Network - a neural network used to generate region of interest proposals.
==PAGE:25==
3.2 Mask-RCNN description
13
The architecture of the Faster R-CNN is based on the Fast-RCNN architecture. All
of the steps present in the Fast-RCNN are also present in the Faster-RCNN. Starting
with feature map extraction, followed by region of interest proposal and RoI pooling
with the final output generation stage. The only change is the algorithm used for the
region of interest proposal.
The Region Proposal Network creates initial proposals by sliding an anchor point
throughout the image.
At each anchor point multiple rectangular region of interest
propositions are generated, these propositions vary in aspect ratios and scale. For each
proposition the Region Proposal Network outputs bounding box coordinates and likeli-
hood of the object being present inside of the region. Once the propositions are filtered
to accept only the ones with high likelihood of containing an object, the propositions
are processed in a same manner as in the Fast-RCNN network.
Figure 3.7: The structure of the Faster-RCNN network.
3.2.4
Mask-RCNN overview
Both of the previously described networks lack the ability to conduct the operation that
is essential for this project - namely the semantic image segmentation. This feature is
the main change that was introduced in the Mask-RCNN. The Mask-RCNN is a network
that expands the capabilities of the Faster-RCNN by introducing an additional output
branch.
==PAGE:26==
14
3 Image segmentation
In parallel to the classification and the bounding box refinement outputs, the seg-
mentation mask output is provided. The segmentation mask is generated by adding
a fully convolutional layer that processes the fixed-size RoI feature map. Through the
training process, this branch learns to upscale and predict the segmentation mask of the
portion of the feature map fed into it.
It has to be mentioned that the authors of the network also modify the original RoI
pooling procedure and introduce the RoI Align operation. This is due to the fact that
for high-quality image segmentation, the original RoI pooling operation is not precise
enough. Both RoI poling and RoI Align procedures extract a region of interest from
the full feature map and then downscale it to a fixed-size feature map that corresponds
to the currently processed region of interest. The difference between this approaches is
in the downscaling method. The RoI pooling operation rounds the results, whereas the
RoI Align procedure interpolates the results - what results in a more precise estimation
of values.
These modifications greatly expand the capabilities of the architecture and make it
perfectly suitable for the project goals.

The proposed modification is fairly simple, the network is expanded by adding an

Dataset
The training of the Mask-RCNN is a supervised learning process. This means that during
the training procedure the model requires an annotated dataset. As a deep learning
neural network is a highly complex model it requires significant amount of examples
to be able to generalize well. Therefore, the acquisition of a representative dataset is
oftentimes a highly time-consuming and mundane task. Not only it would require taking
thousands of pictures but, more importantly, hand labeling different objects on these
pictures to provide supervision for the model.
3.4.1
Dataset requirements
To avoid creating a new dataset from scratch, a variety of available datasetes was an-

different possibilities. To systematize the search for acceptable dataset a set of require-
ments was determined to filter results.
• The dataset has to consist of indoor images as this is the environment that the
solution should operate in.
• The dataset should provide all or at least majority of the labels required for the
supervised learning of the selected model. In case of the Mask-RCNN the required
labels are: bounding box, class label and segmentation mask for each object in-
stance.

3.5
Performance metrics
It is important to define a way of measuring the performance of the proposed solu-
tion. For the classification task a very common choice is the mean average precision.
To calculate mean average precision two additional performance measurements have to
introduced: precision and recall.
3.5.1
Intersection over Union
Firstly, it is important to introduce an evaluation metrics that is used to determine if a
prediction was matched with a ground-truth instance or not. For this task an Intersection
over Union (IoU) is used. It is based on two simple geometrical operations - intersection
and union. To calculate an IoU value for a pair of prediction and ground-truth masks, it
is necessary to calculate the intersection and the union areas for these two masks. Then
the IoU value is simply given by the following ratio:
IoU = intersection
union
(3.2)
3.5.2
Precision
The precision value is defined as the ratio of correct predictions to all predictions made
by the model - meaning that it provides information about how accurate the predictions
made by the model are. The precision is defined by the following formula where tp
stands for true positives (predictions that have a corresponding ground truth instance
with the IoU value over 0.5) and fp stands for false positives (predictions that do not
have a corresponding ground truth instance with the IoU value over 0.5):
precision =
tp
tp + fp
(3.3)
3.5.3
Recall
The recall value is defined as the ratio of correct predictions to all all ground truth
objects - meaning that it provides information about how good the model is in detecting
the ground truth objects. The precision is defined by the following formula where tp

no corresponding prediction):
recall =

3.5.4
Average Precision and Mean Average Precision
Having defined the precision and recall values it is possible to define the average precision
(AP). To calculate the average precision all of the predictions have to be sorted by the
confidence level, in case of the Mask-RCNN the confidence of the prediction is provided
as an output of the network.
Once the predictions are sorted, the precision and recall values are calculated for
each subset of top n predictions (starting with the subset consisting of the top prediction,
then top two predictions etc.). That operations provides values necessary to plot the
precision-recall curve. It has to be noted that the operation might results in the recall
range different than <0; 1>, in such case an additional point is added with precision
equal to 0 and recall equal to 1.
The next step is the interpolation of the precision-recall curve. This operation is
done by adjusting precision values at each recall step (r) to the maximum value for any
recall step r′ > r. Once the interpolated precision-recall curve is obtained, the average
precision is calculated simply as an area below the curve.
The Mean Average Precision value is a mean of average precision values for different
testing examples. In this project the Mean Average Precision is calculated as the mean
over the whole validation dataset.

Mask-RCNN parameters and training procedure
One of the essential tasks during the neural network training is the process of hyperpa-
rameters tuning. Hyperparameters are parameters describing the neural network that
are not optimized during the training and therefore the user has to select their values
before the training starts. Due to the fact that the impact of the hyperparameters is
oftentimes highly unpredictable, hyperparameter tuning task might be challenging. It
has to be taken into consideration that each tuning iteration (the training of the whole
network) takes tens of hours so the process is extremely slow. The time limitation is
so significant that usually the tuning process is stopped after achieving acceptable per-
formance. In this section hyperparameters that are crucial to reproduce the experiment
will be listed.

Learning rate
The learning rate is a basic parameter that determines how strongly the weights of the
network are being changed during each iteration. All of the training procedures used the
value of the learning rate that was suggested by the authors of the used Mask-RCNN
implementation. The used learning rate value was 0.001.
3.6.1.3
Training procedure
Instead of training the whole neural network at once it is possible to train parts of it
separately - by freezing parts of the networking during different training phases. Such
training was suggested by the authors of the Mask-RCNN implementation that was used
in this project. Initial training sessions were done with the suggested training scheme in
which at first only the heads (classifier, segmentation branch) of the network are trained
and later on the rest of the network is added to training procedure.
For other training sessions, a more basic approach was tried. In this training scheme
the whole network was trained together in a single phase.
3.6.1.4
Transfer learning
Transfer learning is a process in which pre-trained weights from one network are trans-
ferred and re-used by another neural network. This process is very commonly used in
image processing networks as image features are often shared between different image
classes and the learnt weights can be easily re-used for different tasks. The transfer
learning allows to make the training process faster and allows to limit the number of
required examples to train the network.
The pre-trained model used for transfer learning was a Mask-RCNN model trained
on the the COCO dataset [Mat18b]. For the training sessions in which the Mask-RCNN
enhanced with depth was used, the weights of the first convolutional layer of the network
could net be transferred as the shape of the convolution was modified. In such a case
this layer was ignored during the transfer learning process.

CHAPTER 4
Robot localization
4.1
Introduction
In Chapter 3 a solution for detection and segmentation of a RGB image was introduced.
However, the output of the segmentation algorithm is not enough to fulfill all of the
project’s goals.
Segmented RGB image does not provide any information about 3D
shape of objects and by itself does not provide information about the localization of
objects in the robot’s environment.
To be able to meet the project’s goals it is necessary to keep track of robot’s position
as it moves. Moreover, the depth images have to be used to build a 3D map of the
environment. These actions are achieved by applying a simultaneous localization and
mapping (SLAM) algorithm.
Few different SLAM algorithms were considered during the initial phase of the design
of this module.
The most important factor when selecting a SLAM algorithm was
it’s availability as a ROS (Robot Operating System) package. Due to selected system
architecture (see Chapter 6) ROS packages are compatible with the rest of the solution.
A comparison by Ilmir Z. Ibragimov and Ilya M. Afanasyev [SBA17] was used to get an
overview of solutions available for ROS. The authors compare five different algorithms
but only three of them are suitable for use with RGB-D sensors: ORB-SLAM [MT17],
DPPTAM [CC15] and RTAB-Map [Lab18]. However, it has to taken into account, that
the authors compared the RGB-D version of RTAB-Map with monoclular versions of
DPPTAM and ORB-SLAM. The performance comparison shows that the ORB-SLAM
performs slightly better than RTAB-Map and DPPTAM is significantly worse than other
solutions. After a detailed analysis of the documentation of corresponding packages, it
was observed that only the RTAB-Map package provides an exhausting documentation
and a complete mapping solution - including 2d obstacle map generation. Therefore
RTAB-Map was selected to be the SLAM solution used in the project.
4.2
RTAB-Map
4.2.1
Overview
The selected SLAM algorithm is called Real-Time Appearance-Based Mapping (RTAB-
Map).
==PAGE:44==
32
4 Robot localization
The RTAB-Map algorithm is based on a RGB-D Graph-Based approach. As the
robot moves, the robot’s encoder odometry is used to localize the robot in the environ-
ment. The RTAB-Map provides also possibility to use a visual odometry - meaning that
robot’s position would be estimated based on the features extracted from consecutive
RGB-D images. The key-frames (RGB image and depth image) from the robot’s envi-
ronment are memorized by adding them as graph nodes together with the robot’s pose.
By expanding the graph the robot acquires information about the environment. A point-
cloud map can be generated by processing the generated graph. The RGB image and
the depth image provide enough information the create a pointcloud for each key-frame
and the pose information allows to place the pointcloud in the global map.
There are multiple features that make the RTAB-Map algorithm suitable for the
project goals:
• The RTAB-Map solution is available as a ROS package what makes it compatible
with the rest of the project.
• The algorithm provides real-time results. This is a huge benefit when dealing with
a quickly moving robot.
• It can be based purely on RGB-D input without the need of a laser scanner sensor.
• The algorithm provides a loop-closure mechanism. The loop-closure process allows
the robot to recognize already visited locations and based on multiple visits opti-
mize the generated map of the environment. As the project’s performance relies on
accurately generated map of the environment, this feature is extremely valuable.
All things considered, the parameters of the RTAB-Map algorithm are well suited for
the purpose of the project and it can be easily incorporated into the proposed system
architecture.
4.2.2
Modifications
Several modifications to the original RTAB-Map implementation had to be introduced
to adjust it to the project’s goals. Most importantly, the original RTAB-Map algorithm
generates a pointcloud of the environment using the RGB image acquired from the
camera.
This RGB representation of the environment had to be replaced with the
semantic representation of the environment (obtained as shown in Chapter 3). To achieve
this a separate graph is generated in which the RGB images are replaced with outputs
of the Mask-RCNN network for each of the frames. A separate graph had to be used
as the RTAB-Map algorithm needs the original RGB frames for correct operation. The
parallel graph with semantic images allows to generate a semantic global pointcloud.
During experiments it was noticed that segmentation masks generated by the neural
network oftentimes are slightly larger that ground truth boarders of detected objects.

but this is a marginal problem for two reasons. First of all, slightly reduced size of the

One final modification had to be introduced in the function generating the pointcloud

points are removed from the global semantic pointcloud.
4.2.3
Critical parameters
It is worth mentioning one of the RTAB-Map parameters, the tuning of which has signif-
icantly improved the performance of the solution. A large amount of miss-classifications
tends to happen for points that are far away from the robot. Two reasons for that
observation were identified: firstly, the segmentation masks for the distant objects seem
to be less precise. Additionally, all of the miss-classification on the object’s borders are
casted into distance if there are no other physical objects around. It was decided that to
reduce that problem, the depth of the generated pointcloud should be limited by setting
the RTAB-Map algorithm’s parameter. The value was selected experimentally as 2.5[m].
This operation has allowed to achieve better quality of the generated pointclouds.
4.2.4
Limitations
Although, the RTAB-Map has provided results of great quality, it has one crucial lim-
itation. The algorithm works well with static environment but can’t handle moving
objects. Although this limitation greatly reduces the ability of the solution to perform
in the dynamic environment, it was still used due to it’s advantages.
In the future it might be beneficial to either replace RTAB-Map with a SLAM al-
gorithm suitable for dynamic environments or propose a modification to the original
implementation to address this problem. For now, an assumption was introduced that
the robot operates in a static environment.

CHAPTER 5
Instance detection and
processing
The generated semantic pointcloud provides substantial amount of information but lacks
the division of the cloud into single instances of objects, a task that is necessary to
accomplish the project’s goals.
To achieve this task, the generated semantic pointcloud is processed by an instance
detection algorithm. Once the instances are detected, the robot is provided with a list
of locations of objects of different classes together with pointclouds representing the
shape of each of the detected objects. In this chapter the cluster extraction algorithm
is described in detail and an use case is presented to demonstrate how the instance
information can be used.
5.1
Cluster extraction
5.1.1
Euclidean cluster extraction
The euclidean cluster extraction algorithm is used to process the semantic pointcloud
and extract from it single instances (clusters) of objects. The algorithm is fairly simple
- an initial point is added to the cluster and then processed by finding all of the points
contained in a search sphere of a given radius. The points that were found are added
to the cluster and then processed in the same manner until there are no more points to
process. This way an instance cluster is extracted.
The big advantage of this algorithm is it’s simplicity and the fact that it is general,
meaning that it can be easily applied to any object class. It has to be noted that besides
these advantages there are risks related to the euclidean cluster extraction algorithm.
There are two main scenarios in which the algorithm performs poorly:
• In the first case, there are two objects that are closer to each other than the search
radius. In that case a single cluster will be extracted for two ground truth object
instances (see Figure 5.4).
==PAGE:50==
38
5 Instance detection and processing
• The other problem appears when a pointcloud for a single instance is not dense
enough. The reason for the poincloud being too sparse is usually the fact that the
robot has not seen the object from a certain angle or it has seen the object from
distance. The result in such case is oftentimes that the algorithm extracts multiple
clusters for one ground truth object instance (see Figure 5.5).
Both of these problems are a result of either too large or too small search radius. By
tuning that parameter it is possible to select a value that will balance out these errors
but to eliminate it completely it would be necessary to use a more complex algorithm.
An additional operation is run at the end of the cluster extraction process. The
minimum size (number of points) of each extracted cluster is defined. All of the clusters
that are smaller than a preset threshold are discarded. This allows to filter out all of
the unwanted segmentation artifacts.
5.1.2
Euclidean cluster extraction with class separation
To minimize the clustering errors, present while using the euclidean cluster extraction
algorithm, a slightly different approach is introduced. This solution is focused on en-
hancing the performance of the algorithm when objects are close to each other, allowing
to detect correct instances without reducing the search radius.
Instead of applying the euclidean cluster extraction algorithm to the full semantic
pointcloud with all of the classes, the algorithm is applied separately to the semantic
pointcloud of each of the classes. The reasoning for that approach comes from the fact
that it is desired that each of the instance clusters consists only of the points belonging
to a single class.
The increase in the performance was noticeable especially for the chair and table
classes due to the fact that usually these objects are within a small distance from each
other and were often clustered together by the full cloud approach.
5.2
Finding optimal approach pose
One of the tasks required from the robot is to approach the detected objects. To achieve
that the goal position and direction have to be determined based on the robot’s knowl-
edge about it’s environment. Most importantly the robot has to take into consideration
the general obstacle map that provides information about areas inaccessible by the robot.
Moreover the robot should face the target object and possibly approach it at the mini-
mum distance.
As stated before there are two main variables that have to be taken into account
when determining the desired goal pose of the robot. The first one is the distance of
the robot from the object and the other one is the direction at which the object is
==PAGE:51==
5.2 Finding optimal approach pose
39
approached. Based on these parameters, two different concepts were investigated while
designing the algorithm.
Both of the proposed approaches are based on the same principle. The space around
the object is sampled by checking in the general obstacle map if the location is accessible.
The difference between methods lies in the order of selection of sampling points.
5.2.1
Direction priority approach
In this approach, the robot’s final orientation is prioritized over the robot’s distance
from the object. This method is useful in cases in which it is crucial to approach the
object from the correct angle. One practical example is approaching a chair from the
front-side.
The desired direction is provided (as shown in Section 5.3.1) and the algorithm starts
sampling the general obstacle map, starting at the center of the object and moving away
in a ray-like fashion on a line determined by the initial desired direction. If no accessible
points are found the direction is slightly altered and probing continues on a differently
angled ray.
The procedure is visualized on Figure 5.1. It should be observed that this approach
prioritizes the correct direction over the distance from the obstacle.
Figure 5.1: Direction priority approach.
5.2.2
Distance priority approach
In this approach, the distance from the object is prioritized over the robot’s orientation.
This method is useful in general cases where the direction is unknown and the goal is
to simply approach the object. For instance, it might be used for objects classified as a
table or a box.
==PAGE:52==
40
5 Instance detection and processing
In this algorithm, again, the desired direction has to be provided to initialize sam-
pling of the general obstacle map. Unlike in the previous algorithm (Section 5.2.1), the
sampling is conducted on arcs that get further away from the object with each iteration.
A boarder case of this approach can be used, in which the probing arc forms a full circle
around the object. In such case the impact of initial direction becomes marginal and the
algorithm looks for the goal position around the object. This method is very useful as it
can be used for any object class without the necessity to provide the desired direction.
The procedure is visualized on Figure 5.2. It’s shown that the distance from the
obstacle has higher priority than the direction.
Figure 5.2: Distance priority approach.
5.3
Use case: instance pointcloud processing
Once the object instances were extracted from the generated semantic pointcloud and
the robot has acquired significant amount of information about it’s environment, various
post-processing applications can be implemented. In this section, a use case is presented
in which the robot determines the orientation of detected chairs.
5.3.1
Chair orientation detection
This part of the project is less general than the previous sections as it applies only to
the objects belonging to the chair class. Depending on object class and the expected
robot behaviour different algorithms would need to be used.
For a robot, to interact with a chair, it might be beneficial to approach the chair
either from the front or the back. The proposed solution provides all of the necessary
==PAGE:53==
5.3 Use case: instance pointcloud processing
41
information for the robot to firstly detect chair instances in the environment and secondly
determine the chair orientation and propose a robot pose to approach the selected object.
The algorithm for detection of the front of the chair is divided into two parts. Firstly,
a plane parallel to the z axis is fitted into the instance pointcloud. To find the plane the
RANSAC algorithm is used. The detected plane corresponds to the backseat of a chair.
Obviously this part works only with the assumption that the chair has a distinguishable
backseat. The plane provides two normal directions that represent the front and the
back of the chair.
Once the backseat plane is detected it is necessary to determine which direction
represent the front and which represents the back of the chair. This part of the algorithm
works on the assumption that that distance-wise the front of the chair is longer than
the back. To determine the orientation of the chair a simple algorithm was proposed.
Points with maximum distance from the plane are found on the both sides on the plane.
The one with larger distance is determined to be on the front side of the plane.
The detected front direction is then represented with an unit vector placed in the
mean point of the instance pointcloud. To allow operations on the 2D obstacle map, the
found vector is casted onto the ground plane. The results of this operation is presented
on Figure 5.8.

To implement the solution, a suitable system architecture had to be proposed. It was
evident that different modules, implemented in different technologies, would need to be
synchronized and the flow of information between them had to be provided. To fulfill that
objective, the Robot Operating System (ROS) was used. The Robot Operating system
is a software platform used for robot system development. It allows to create nodes
(executable applications) that can communicate with each other. The communication
between nodes can be set up easily, even when nodes are running on different platforms
(robot and PC). Moreover, ROS provides multiple tools for nodes synchronization. In
section 6.1.1, the environment specification is provided.
From the beginning, it was clear that running the neural network will not be feasible
on the robot’s side due to limited computational power.
A separate PC with GPU
was used to compute the forward passes through the neural network. The technical
specification of the PC will be provided in section 6.1.2. The PC and the robot were
connected through Ethernet. To allow more mobility to the robot, an attempt was made
to establish a WiFi connection but the connection baudrate was insufficient.

Future work
Although the results of the project are satisfying, there are possibilities to further im-
prove the performance of the solution. As the presented project consisted of wide variety
of tasks, it was not possible to optimize the performance of each of modules. The em-
phasis was put on proving the feasibility of the concept and designing a reliably working
base solution that might be improved later on. In this chapter, a list of possible im-
provements will be provided. To maintain the logical division, the improvements will be
divided into groups for corresponding sub-systems of the solutions
8.1
Image segmentation
Even though the results achieved by the neural network are acceptable, there are multiple
means to improve the performance of this sub-system:
• The process of hyperparameters tuning might be expanded with additional param-
eters. There are many parameters that were not tuned due to time limitations,
including: learning rate, anchor size, optimization algorithm.
• A more insightful experiment should be conducted to provide a definite answer
about the performance of the enhanced Mask-RCNN. Multiple training sessions
should be run for original and modified versions of the network to remove the
impact of the stochastic nature of neural networks. The training procedure deter-
mined in this project might be used (whole network training, 30 epochs, sampling
every 2 epochs).
• The performance of a neural network depends strongly on the number of available
training examples. Throughout experiments it was proven that many of the classes
available in the used dataset were underrepresented and hence the performance for
these classes was inferior. A different dataset might be used or the existing dataset
might be expanded to improve the quality of the image segmentation process.
8.2
Robot localization
During the experiments, there were no problems observed with the quality of robot
localization or mapping. However, no in-depth analysis of algorithm’s parameters and
==PAGE:66==

their impact on the solution was conducted. It might be beneficial to experiment with
different values of RTAB-Map parameters.
Most importantly, the used SLAM algorithm (RTAB-Map) adds a significant lim-
itation to the solution - the environment has to be static. In the future, it would be
reasonable to replace the current SLAM algorithm with one that is suitable for operation
in a dynamic environment. This would greatly expand capabilities of the solution.
8.3
Instance detection and processing
During the final experiment (see Chapter 7), a problem with instance detection was

even further.
In the project, only a single scenario for instance processing was presented. It is

algorithms - allowing the robot to approach and interact with different classes of objects.
In the future, a set of behaviours should be implemented for each of the detectable object
classes.

Others

obstacles but also to interact with them.

Conclusion
The main goal of this project was to provide an obstacle detection and avoidance solution
for indoor mobile robots. The objectives of the project required that the solution should
expand the basic approach by semantically classifying encountered obstacles. Moreover,
the proposed system should be able to map the environment and memorize the location,
shape and class of detected objects.
The initial objectives of the project were fulfilled. A proof of concept has been pre-
sented by designing, implementing and verifying the solution. The final experiment has
shown that the robot equipped with the proposed system was able to map it’s environ-
ment and assign semantic meaning to sections of the map. Moreover, the robot was able
to divide the environment into single instances of objects belonging to various classes
- keeping track of their localization and shape. Additionally, it has been proven that
the acquired information about the shape of objects can be reliably used to determine
orientation of objects. Finally, it was possible to use the acquired information to au-
tonomously navigate the environment and approach objects. Therefore all of the initial
objectives were fulfilled by the proposed solution.
To improve the results of the system, a detailed analysis of the performance of the
image segmentation module has been conducted. A modification to the original Mask-
RCNN implementation has been introduced by adding a depth channel to the input.
No improvement of the performance was observed for the network with RGB-D inputs.
However, to draw a definite conclusion, additional training sessions would be required
to take into account the stochastic nature of a deep learning solution.
