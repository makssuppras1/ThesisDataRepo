Abstract
In this master thesis project a solution for the detection of a pedestrian crossing
by means of synthetic images and machine learning is proposed.
This thesis is a
continuation of the author’s bachelor’s thesis where the same problem was solved
but by means of only computer vision hard coded algorithms. The project could
be divided into two main sections: (1) generation of the data set and (2) machine
learning algorithm.
Using an open source platform as Blender eight data sets are generated based on
the idea of ”domain randomisation” [1]. These data sets are different renders of the
same scene where three binary variables (stripe’s colour, stripe’s noise and distractors)
determine the final render. In addition to these eight sets, two more sets with the
segmentation map of the images are also generated. After analysing the segmentation
map with simple computer vision algorithms, a COCO data set file is generated for
each of the data sets.
In terms of machine learning, the chosen algorithm has been Mask R-CNN [2] a
state of the art framework for object segmentation. The performances of eight data
sets are compared between each other in order to see which combination works better.
In addition to it, a ninth data set made of a mix of the previous eight is also tested.

CHAPTER 1
Introduction
This chapter serves as an introduction to the project and an overview of the report.
1.1
Motivation
In a world of constant technology development we are starting to be surrounded by
autonomous systems 24 hours a day, 365 days a year. From the vacuum cleaner at
home to the metro we take every morning, it is hard to look around us and not
see autonomous systems. Then, it may not come as a surprise that automation and
artificial intelligence are gaining more and more presence in something as used in
our society as the car.
According to a study in the United Kingdom [3], 83% of
women and 86% of men used either public or private means of transport to go to
work. One of the implications of these numbers is that chances of traﬀic accidents
are also high, some of them with fatal ending. In a study carried out by CESVIMAP
[4] they publish that in only 10% of traﬀic accidents the source is a technical failure.
Thus, in nine out of ten the source is human error. Therefore, it is of great interest
to everyone to develop and deploy progressively autonomous and intelligent systems
in the transportation field. A research from Fundación MAPFRE found that 40%
of run overs in Spain between 2005 and 2010 happened in pedestrian crossings [5].
Being the only place where cars and pedestrians share the same space, it is key that
future autonomous cars develop a strong detection system of these points to reduce
to the minimum the risks.
Machine learning in computer vision seems the way to go. Big car companies such
as Tesla or VolksWagen, who are trying to achieve full autonomous driving, put a lot
effort in making these two fields work together. The problem comes with the nature
of machine learning: it needs a lot of data. This data can be obtained from previous
images, from new real images or it can be synthetically generated. Developing an
object detection system via computer vision and machine learning implies that the
data has to be labelled. If we use real images we have to manually label it (or at least
for some part). This takes a lot of time, which at the end of the day is money spent
for these companies. In addition, we may need to prepare the car for a situation from
which we may not have images yet. As well, gathering big amounts of data comes with
ethical and legal problems regarding privacy. This is where synthetic generated data

can help. Apart from having control over what situations we want to generate, we
can easily obtain the annotations without the intervention of a human on each image.
This is something companies have already been using for some time. As an example,

and NVIDIA has an extremely advanced and realistic product (Omniverse) which
multiple companies use to create simulations of all kind. However, these products

In this project, using open source tools and average computer hardware, a model to
generate synthetic images of pedestrian crossings is developed and presented.
1.2
Objectives
During the development of the project the objectives have slightly changed over the
original ones. The original objectives were to develop a machine learning algorithm
to detect a pedestrian crossing, comparing different techniques and algorithms and
choosing the one that provides the highest performance overall. However, after start-
ing the development the objectives have shifted towards investigating the generation
of synthetic images and their application in instance segmentation learning algorithms.
The project's objectives could be presented as follows:
• Development of an automatic tool to generate thousands of synthetic images.
• Creation of multiple data sets to compare performance between them.
• Development of a tool to obtain a COCO data set from the aforementioned
images.
• Training of a machine learning model for instance segmentation of a pedestrian
crossing.
• Performance comparison between this project and state of the art works in the
field.
• Test models on a video feed.
Other objectives, more related to the academic field are:
• Research methods in engineering.
• Past and current works in synthetic images generation for machine learning.
• 3D modelling of assets for machine learning algorithms.
• Presentation and defence of all the knowledge and materials used in the devel-
opment of the project.

CHAPTER 2
State of the art
This chapter contains a recapitulation of the relevance of synthetic data over differ-
ent industries, the work done in previous years regarding synthetic data in machine
learning, previous works in domain randomisation and some theory overview of Mask
R-CNN and other quick concepts relevant to the project.
Over the last decade the skyrocketed development of artificial intelligence has
made data one of the most valuable assets to have. Some even refer to it as the new
oil. However, and as it happens with oil, not everyone can have access to the vast
amount of data required to train machine learning models. Some because the data is
property of someone else, and other because a proper data set has not been developed
yet. That is why many are making their own synthetic data. In general, the more
diverse training data the better AI models perform. But there is a problem, which
is obtaining and labelling huge data sets containing thousands or even millions of
elements. Not only it is time consuming to perform this process, but as well extremely
expensive. According to co-founder of AI.Reverie Paul Walborsky, labelling a single
image could cost even 6$ from a specialised company, whereas generating an image
artificially can cost six cents. This company was bought by Facebook earlier this
year [6], in a clear example of how much big tech companies bet for this type of
technology. Another concern real data sets rise is privacy. It is well known the legal
and ethical problems these type of companies have faced during the last years due to
this reason, being actually Facebook one of the most known cases. There have been
some works where this problem is tackled, such as in [7]. They propose a method
to release personal data while preserving individual privacy. They called the method
K-anonymity. There are two common ways to obtain K-anonymity: suppression and
generalisation. As an example, let us think of medical data with six columns (name,
age, gender, country, religion and disease). Without going in depth into the methods,
the data set will have 2-anonymity whenever for any combination of the attributes
(column's values) there are at least two rows with the same exact values. 3-anonymity
would be for at least 3 rows etc. Even if this method can work for multiple cases it
also typically yields worse results in terms of accuracy. Synthetic data is information
that has been generated by a computer as an alternative to real world data. Even
if this information is artificial, its intention is to reflect real world data. This data
is not just limited to images (as is the case of this project) but it can be applied to
almost any field: stock prices, financial fraud, medical data, text recognition etc. In

2.1
Previous work
The idea behind domain randomisation is pretty simple though still one of the most
rising techniques in synthetic images generation for machine learning. In fact, this
concept has been known since the beginning of the 90s in [14] where, without explicitly
mentioning the term, they propose the use of noise in robotic simulations to improve
performance. As far as I know, the first time this term was explicitly used on a paper
was in 2017 [1]. Before getting in depth about this paper let us see the idea behind
domain randomisation.
Let us consider a model supposed to be trained on a synthetic domain Dsyn with
data distribution psyn (Dsyn ∼psyn) to then be applied on Dreal ∼preal. We would
like to make psyn so close to reality that the trained model on psyn performs well on
preal. In the best case psyn will cover the whole spectrum of preal. However, this is
hardly possible when attempting to generate solely realistic images. Thus, synthetic
data can be randomised so that psyn covers preal, even if those images do not seem
realistic to the human eye. In different words, we want to generate a wide enough
spectrum of images so that a real image is just another variation inside this spectrum.
There are multiple ways this randomisation can happen but, generalising, we could
divide it in two separate processes: scene construction and rendering.
1. In the first step we can randomise the scene itself. This is, randomising the
number of objects in the scene, their poses, the backgrounds, textures, the use
and number of distractor objects etc.
2. In the second step we are talking about light sources, image resolution, rendering
type, noise, orientation etc.

In this project both steps have been performed, though with more emphasis on the
first one. As it will be better explained in Chapter 3, the textures, number of dis-
tractors, poses, camera positions, backgrounds and noise are randomised. However,
rendering type and image resolution have not. This will be very briefly explained in
the next section 2.2 why.
Coming back to [1], they proved that these concepts worked well in their proposal.
They present a simple scenario where they tried to estimate the position of geomet-
ric shapes such as cubes, cylinders, trapezoids and other polyhedra by training ”a
deep neural network...only on simulated RGB images (without pre-training on real
images)”. They mention the diﬀiculty of applying realistic simulated images on real
cases as they often fail to reproduce the natural noise that would appear in a real
scenario. This difference between the simulation and reality is what is commonly
referred to as ”reality gap”. What they propose in their approach is to address this
reality gap not by trying to simulate those errors in the real scenarios, but by using
domain randomisation. Their hypothesis is the following: ”if the variability in simu-
lation is significant enough, models trained in simulation will generalise to the real
world with no additional training”. The aspects they randomised could be mapped

As it can be seen, the last two fields have not been applied in this project. Again,

in the final result, or added an extra step of diﬀiculty hard to assume.
Another similarity between both projects is the renderer used. In their paper they
use MuJoCo Physics Engine's built-in renderer [15] which is not intended to be photo-
realistic, like the rendering mode used for this thesis. In terms of machine learning
model used, it differs as their objective is to estimate the position of the cubes and
not segmentation. Finally, their results are evaluated comparing the performance
focusing on some of the different parameters seen on table 2.1. They compare how
it performs varying the number of images, number of unique textures, use of ran-

in the detection model. These are also tested here, thus the reason of having the
eight different data-sets. Summing up their results, the models trained with all the
randomisation possibilities outperformed slightly those where the camera was not
randomised. However, it significantly improved those where no distractors were used.
Another important factor that affected performance was the size of the training set.
When it was smaller than 1000 images there was a significant drop in performance.
Another paper that took one step further the idea of domain randomisation is [16].
Their approach was very similar to the previous one but in a relatively more ambitious
scenario. Instead of detecting the position of polyhedra they attempt to detect the
bounding boxes of cars. In general the steps followed to randomise the domain are
the same: positioning the objects of interest randomly on the scene, choosing random
backgrounds, adding what they call ”flying distractors” which fulfil the task of serving
as negative objects, and finally randomising textures, camera location and lighting.
In their case they compare their approach with Virtual KITTI dataset [17] and real-
world KITTI datset [18]. One interesting factor they highlight is the distribution of
the number of cars and their position in the frame and how they compare to VKITTI
and KITTI, as it can be seen in figure 2.2. It clearly shows there is a greater spread
on their data set, as in the first two cases the cars are positioned in expected positions
(either on the road or parked on the sides), whereas in the third case they can be at
any location as they do not try to recreate reality. They claim this fact is better as
it forces the deep neural network to focus more on the structure of the cars.

Later work on domain randomisation has been published in multiple papers. As

2.2
Theoretical Concepts
2.2.1
Mask R-CNN
Mask R-CNN [2] is currently the state of the art machine learning model for object
segmentation. Published only three years ago (2018) it has already been cited over
15.100 times. Despite its great performance its architecture is pretty simple, according
to its own creators. Mask R-CNN can be described as an extension of Faster R-CNN
[19] which at the same time is a continuation of R-CNN [28]. In Faster R-CNN there
are two outputs for every candidate object: a bounding-box and a class label. What
Mask R-CNN does is to add another ”branch” which is the object's mask.
For starters, let us quickly explain how Faster R-CNN works. It consists of two
phases. The first one is called Region Proposal Network (RPN) and outputs candi-
date object bounding boxes. In the second phase the features from each candidate
bounding box are extracted by means of Region of Interest Pooling (RoI Pool) and
classification and bounding-box regression is performed. In Mask R-CNN the first
phase (RPN) is the same. However, things change a bit in the second phase, where
Mask R-CNN outputs a binary mask for every RoI in addition to the class and bound-
ing box prediction. This is performed in parallel, therefore the classification does not
depend on the binary mask prediction. Another difference is that Mask R-CNN uses
RoI Align instead of RoI Pool. The reason is that in RoI Pool the stride is quantized,
which means that the stride will be rounded to the nearest integer. To understand
this better let us see an example. If we have a RoI of size 17x17 pixels and we want
to map it to a 7x7 the stride will be 17/7 = 2.42 which in the case of RoI Pool will be
rounded down to 2. This, as can be seen in figure 2.4, leads to a loss in data, as not
the whole image is being considered. Then this is once again subdivided into spatial
bins, which are also quantized.

2.2.2
Blender rendering and data augmentation
It has been mentioned in the previous section that some randomisation such as render-
ing type, resolution and lighting among others have not been applied in this project.
In this section it will be explained why.
As explained in section 2.1 synthetic images for machine learning is not something
new. However, there is not any specific tool that is a ”suit-it-all” for this purpose.
Therefore it was needed to develop a tool from zero. There are a bunch of 3D mod-
elling tools that could be used for solving this task: SolidWorks, SolidEdge, Fusion360,
SketchUp, OpenGL, Unity, Blender etc. There is not one software which is better
for every case than the others, but rather each of them will suit better depending on
the purpose. For example, to generate images of complex, highly detailed 3D models,
such as machinery tools or mechanical components, the first three mentioned will
most likely suit better. In case of big scenarios like natural environments, cities or
planets, both Blender and Unity can carry out the task with good performance. Both
are open source, have a big community which makes it easy to find information about
it, and bring the possibility of writing scripts in Python to manage them (apart from
a graphical user interface). For this project both Unity and Blender were considered
and studied. At the end of the day, the final decision came to a personal preference.
Even if both could have been used for this project there are some differences which
made Blender the best option for the project. Unity is a game engine, which means
that its purpose is to create video-games. It does not give you the possibility of mod-
elling 3D objects but rather you need to import them from other software, such as
Blender for example. Being a game engine, it is designed to create scenarios rather
than renders. Blender, on the other hand, even if its main purpose is not rendering
images, does have a pretty good system of rendering multiple images. Plus its scala-
bility in this task is rather simple with the help of some coding.
Once Blender was chosen it is important to understand the different types of ren-
dering it provides. There are four, but we will only be seeing two: cycles and eevee.
Cycles is a more realistic rendering mode, which uses ray tracing and light bounces to
calculate the final render. As one can imagine, this process requires of a lot of comput-
ing power. It is possible to modify the amount of light bounces, hence increasing or
reducing the rendering time. However, unless this amount is extremely high, images
will turn out a bit ”grainy” as if a light layer of noise was applied on top. Eevvee
is a faster rendering mode. As one can expect, this implies that the images will not
==PAGE:31==
2.2 Theoretical Concepts
17
look as realistic. To the human eye they do not look realistic as there are little or no
shadows, transparent materials behave as opaque, and reflections do not happen. In
figure 2.7 a couple of examples of both rendering modes can be seen. The strong point
of eevee is that the rendering times are drastically reduced in comparison to cycles
mode. It is possible to find a configuration that make an eevee render look pretty
realistic, but it has to be tweaked individually for every scene, which is impossible in
this project.
Figure 2.7: Cycles (left) vs Eevee (right) renders
Considering that in this project realism is not important and that rendering time
is, the decision between eevee and cycles has a clear winner. On average for this
project, every scene rendered in eevee took 5.5 seconds (this is for ten images: eight
rgb + two segmentation). For cycles the average was 4 minutes and 27 seconds. Con-
sidering the amount of images needed for a data set and these rendering times, it was
discarded the use of cycles. One initial idea was to compare the final performance be-
tween a data set of eevee images and one of cycles images. However, the time needed
to generate a cycles data set just did not add up. As lighting is not so relevant in
eevee, random light sources were not included, as they did not provide any change in
the final render, but did affect rendering time.
In terms of applying data augmentation, it was as well considered. The most feasible
solution was to randomise resolution of the render. This, even if possible, increased
the Blender script complexity in a way that there was a relatively high chance of re-
sulting in bugs. As we are rendering not one but ten images each time, the resolution
would need to be changed in all of them. This should not be a problem at first sight.
==PAGE:32==
18
2 State of the art
However, this script failed every once in a while generating bad segmentations, which
implies that every once in a while the computer fails. Solving the problem of bad
segmentations is simple: you just delete the images. On the other hand, solving the
problem of having multiple resolutions for the same render is not, and thus the risk
was not worth taking.
Regarding other data augmentation techniques such as cropping, brightness changes,
rotation etc. there were two reasons to not apply it. Firstly, as seen from previous
work as in [16] data augmentation did not have a significant improvement in per-
formance. Secondly, this process takes a lot of time in the current set-up. In the
best case, were each image takes one second to modify and save, we would be talking
about 24.800 seconds which is almost seven hours (assuming nothing goes wrong). In
addition, these new images would need to be uploaded to Google Drive, from where
Mask R-CNN script is taking them from. Considering the size of one data set of 3.100
images is almost 8GB uncompressed, they need to be compressed in order to upload
more than one data set to Google Drive at a time. This requires a significant amount
of time. Summing up, to apply data augmentation would require an amount of time
close to two weeks adding it all up (data augmentation, compression, uploading and
training). Seeing that the increase in performance obtained in previous papers is not
significant, it was discarded.

Layers play an important role in this project as they facilitate the rendering of different
images from the same scene. The idea is simple: every layer can show or hide every
element independently from the other layers. Therefore, we can take advantage of

we are looking for. It is not as simple as creating one layer for each image though.
In fact, in this project there are eight layers but some are used as auxiliary to create
compositions. The existing layers and their role are explained in the list down below.
The combination of some of these layers and the use of some python code make it

3.1.7
Code workflow
Once all the pieces of the rendering generation tool have been presented, it is time
to explain and show the workflow. The python code can be found at the Bitbucket
repository (link to file). The flow itself is pretty simple. For every new scene it follows
a linear workflow as it can be observed in figure 3.12. It is the different functions that
can be a bit tricky to understand. In this sub-section they will be briefly explained,
but for a more detailed information, please refer to the python file were annotations
can provide a more detailed understanding.
Once the code starts running it needs to do a few operations to ensure that the images
will be rendered as expected. This is represented in the workflow as First time set-
up. In summary, it checks that the rendering mode is the correct one, creates the list
==PAGE:46==
32
3 Proposed Approach
with the 3D models that can be used, hides or shows specific elements based on the
rendering mode and sets the correct file output path.
After this initial set-up the rendering loop starts. This loop basically generates a new
scene and renders the ten images. Once the ten images are rendered and saved, it
repeats the process for a new scene as long as we still need to generate more images.
The loop repeats always the following steps:
1. The first step in the loop is to choose a random background for the scene. This
is represented in the workflow as Change background. Basically, it chooses
randomly one background from the 132 available and applies it to the scene.
2. Afterwards it generates a new pedestrian crossing, represented as Generate
pedestrian crossing. Here, the number of stripes, width and height are ran-
domised between the ranges shown in table 3.4. As well the segmentation plane
is set to match the same size.
3. Once the pedestrian crossing is created its materials are also randomised (Randomise
crossing's material). Here there are three materials randomised: white-noise,
colour-solid and colour-noise. The noise can be adjusted by tweaking five differ-

3.2.2
Segmentation to COCO
As seen in the previous section, it is needed that the information regarding the data
sets is stored in a JSON file with COCO format. There are multiple tools online
that allow you to do this process. They are useful tools when the only way to obtain
the segmentation is to draw it by hand. They fill in all the extra metadata such as
directory, date etc. However, for this project it is nowhere near to be the best solution.
For starters it would take an extremely long time to draw the segmentation for over
24.000 images. But secondly, and most important, we can take advantage OpenCV
[36] and from the segmentation data sets generated in Blender which provide a pixel-
perfect segmentation. With very simple OpenCV commands it is easier, faster and
more reliable than doing it by hand, one by one. On average, the time it takes for
the algorithm to generate the COCO file for 3.100 images is 87 seconds, whereas by
hand the time estimation is even diﬀicult to imagine.
Before proceeding to generate the COCO files it is needed to filter some of the images.
Even if the Blender script is pretty reliable some of the images had to be discarded due
to two main reasons: bad segmentation and pedestrian crossing too occluded. In the
first case, every once in a while, there was a problem with the segmentation images.
As explained in section 3.1.4 the segmentation images were obtained by comparing
two layers. For some reason, some segmentation was not done correctly and, apart
from the pedestrian crossing, other 3D models were included. It is not clear what the
exact reason is, but one hypothesis is that the computer was overloaded at the time
of rendering those images and failed to do it correctly. This hypothesis is based on

As of the OpenCV code, it is a pretty simple one. A simplified workflow can be
seen in figure 3.13. In the initial set-up, one of the eight data sets is chosen, the
corresponding directories selected and the ”info”, ”licenses” and ”categories” blocks
of the COCO file are defined. Once this is done, the first image of the directory
is taken. The image that is taken is the one from the segmentation, not the RGB
image.
In fact, the only difference between the final COCO file of the data sets
Clear-colour-noise, Clear-colour-solid, Clear-white-noise and Clear-white-solid is in
the ”images/filename”. The rest of the file is exactly the same as the segmentation is
the same for all these four. The same happens for the data sets with distractors.
Once we have converted the image into OpenCV format the contours are found. In
the case of no distractors it will most likely yield only one contour. In the case of
the distractors in the middle it can range from anywhere between one to tens of
them.
As the image is just black and white this is a very simple operation that,
in addition, yields very good results. Then, for every contour we will perform the
following operations:
1. Simplify the contours. The contours are stored as a list of X and Y coordinates.
In general, OpenCV tends to return a very long list of coordinates for every
contour, some of them forming almost straight lines. As this list of coordinates
will be saved into the COCO file, the shorter this list is, the better.
The
command cv2.approxPolyDP() carries out this simplification given a parameter
called ”epsilon” which is defined. The smaller the value the closer to the original
==PAGE:53==
3.2 COCO data set
39
shape. Even with a very small value, the polygon is significantly reduced on its
number of points. This may not be relevant for a few images, but when dealing
with 3.100 images per set it does make a difference with no loss in definition at
all.
2. Then, and as one of the fields in the COCO file is the contour's area we need
to find it. OpenCV already contains this information for every contour, so it is
just a matter of adding it up for every contour in the image.
3. As well, it is needed to provide the bounding box. OpenCV does not provide
this information. With just a simple operation of finding the most extreme
points of all the contours the solution is obtained.
Once all these steps have been done for every contour in the image, the remaining
”images” and ”annotations” blocks are filled in. After all the images are processed
the file is ready to be saved as .JSON. As expected, the COCO files of those images
without distractors in the middle are much smaller than those with. In general, with
no distractors the contours can be stored even with just four points, whereas with
distractors in the middle this list can go up to thousands per image. An example of
these files can be found in the Bitbucket repository (COCO no distractors, COCO
with distractors). The second file only contains 300 images as it was impossible to
upload a file with 3.100 images.

4.2
Discussion
From these results we can see that the project has not worked as expected but that
domain randomisation does have an impact on performance. It is clear that when the
model has been trained on images without distractors it has performed poorly when
faced to real images. When trained with distractors, even if it did not yield great
results, we can see an improvement. Therefore, we can conclude that the inclusion
of distractors is relevant. We can not say the same about the use of noise. Even if
there are more cases where its use has yielded better results than worse, there is not
a clear evidence that it improves performance. Regarding the use of different colours
vs just white, there is a slight evidence that it works.
Taking into account that domain randomisation has proven to work in previous cases
([1], [16]) one hypothesis as to why it has not worked in this case is that the data sets
are not covering enough spectrum to make reality fall inside it. The problem does
not seem to come from Mask R-CNN, as both the validation loss and the AP@0.5 on
synthetic images yield very good results, even with low number of epochs. Looking
into the differences between the cited projects where it worked and this project, we
can get a hint of other possible problems. One of the most obvious differences is
the data set size. Whereas in this project we are dealing with 3.100 images per set
they worked with sets of over 100k training images. In [1] they noticed a significant
drop in performance when the data sets were smaller than 1000 images. In this case
we are dealing with almost three times that amount. However, their objective was
not segmentation but detection, and the machine learning algorithm was different.
In general in machine learning the more data you have, the better performance. In
addition, this also implies that if the randomisation script is well designed the model
will be trained in more scenarios. As well, the number of epochs was significantly
greater in their cases, which also (usually) improves performance.
Another problem may come from the noise applied to the textures. The initial in-
tention of applying noise to the textures was to try to recreate the imperfections a
real pedestrian crossing has. It is common to see in the real images that the edges
of the white stripes are not perfect straight lines. This makes the stripes not perfect
rectangles, which is really hard to recreate in a simulation. It may have happened
that the model has learned to detect perfect rectangles, which would then make it
fail when being faced to a not perfect one.
One more difference between this project and the cited ones is the object's shape. In
this project we are dealing with a flat object, whereas they were attempting to detect
objects with volume. It is possible that their models managed to learn they shape
characteristics of the objects, which is one extra variable to identify them. Here, be-
ing flat, it relies on learning the 2D shape of the stripes.
Lastly, one modification that could most likely help is the inclusion of more than
one pedestrian crossing per render. This was studied during the development but
discarded due to the complexity of its implementation in Blender.

5.1
Conclusions
This master thesis project has consisted in the development of a tool to create syn-
thetic images of pedestrian crossings for machine learning purposes. It has been based
on the technique of domain randomisation [1]. The tool has successfully created data
sets from an open source platform as Blender and in an average hardware set up.
These data sets have been used to train Mask R-CNN models, prior conversion to
the COCO data set format [35]. For the nine data sets an identical configuration
model has been run for 50 epochs and their performances have been compared. In
the results section it has been exposed how the results are not as good as someone
could desire. However, looking at the objectives (section 1.2) of this project we can
see that most of them have been accomplished.
• The automatic tool to generate synthetic images has been developed and is open
source and available for anyone to try it.
• Eight data sets have been created from this tool and in section 4.1 the compar-
ison between their performance is shown.
• A script that automatically creates a COCO format data set from the synthetic
images has also been developed.
• Mask R-CNN, the current state of the art algorithm in image segmentation, has
been used to train models on these synthetic images
• And finally a performance comparison against previous work has also been
carried out

there are a few ideas.
For starters it would be good to see if making a hybrid data set improves results.
As seen that with these synthetic data sets the trained model has not managed to
perceive reality as just another scenario, it is highly probable that including a small
set of real images will expand the model's spectrum.
Another possible addition to the project could be to add into the domain randomi-
sation parameters that have been used in previous papers such as [1], [16] that were
not included here. These are for example random light sources, different image sizes
and resolutions and more than one pedestrian crossing per image. As well, data aug-
mentation could increase the size of the data sets without the need of rendering more
images.
It would also be interesting to see whether rendering scenes in Cycles mode, which
provides more realistic renders, would have any impact on the performance. One
hypothesis is that a small amount of ”realistic” renders could improve performance
in the same way as adding real images.
One last extra work to do could be playing with the different configurations of Mask
R-CNN, as not much tweaking has been done in the project. In general a model
is trained for hundreds of epochs, reducing the learning rate after 50-100 epochs.
As well, multiple people change during training the configuration from ”head” layer
training only during the first epochs to ”full” layers training.
