Summary
The use of unmanned mobile robots in the industry, as well as private homes, has
grown over the past few years.
With the increased demand on the market, several
autonomous solutions has been developed for various applications such as self driving
cars, lawnmowers, vacuum cleaners, guide bots, cargo transport, etc. Historically, many
solutions has required the environment to be modified by installing guide marks, signal
beacons or other systems to help the robot find its way. Installing such systems is a time
consuming as well as costly process which only helps to solve highly specific problems.
But, with the growth in available computing power and increased sensor precision, new
methods for robot navigation are becoming increasingly feasible for realtime use. By
enabling a mobile robotic system to navigate safely and autonomously in an unknown
environment, one can greatly reduce the time and money needed for the realization and
thereby facilitate the transfer to an autonomous robot based solution.
The aim of this project is to develop such an autonomous solution by enabling a
mobile robot to navigate safely, precisely and autonomously in an unpredictable and
partially unknown environment, avoiding obstacles as they are encountered, while ful-
filling its main objective. To achieve this, a Model Predictive Path Following Controller
is used to maneuver the robot along its path, defined as a series of waypoints. By using
MPC, the robot is able to anticipate upcoming waypoints and embark on the turnings
before the corners are reached and thereby steer the robot smoothly through sharp inter-
sections. In order to ensure the safety and stability of the robot, and to avoid excessive
overshooting, the vehicles velocities and accelerations are constrained such that the mo-
tion of the robot remains feasible and smooth. In order to damp the dynamic behavior of
the robot an actuator model is introduced to the MPC problem, such that the controller
can predict the relevant dynamics and compensate accordingly. An optimal Kalman
observer is used to estimate the unknown state introduced by the actuator model. The
path-following MPC was tuned using a simulation of the closed loop system. Results
from the real system showed that the simulation study was successful as no additional
tuning was necessary.
To avoid obstacles along the path, the robot must know its location and what is
around it. To achieve this, a grid-based map representing the environment is interactively
updated from sensor data, obtained from the laser scanner and from the wheel odometry
sensors. The map representation is used to extrapolate useful information about the
surroundings, such as the distance to known obstacles and what is known and what
is still unknown territory. An A∗path planning algorithm capable of computing an

(optimal) obstacle free path, from the robot’s current map location to some goal, while
taking obstacles into account by using the obstacle distance and an obstacle repulsive
potential, is used to continuously update the (optimal) path as the robot learns more
about its environment. The computed A∗path is formulated as the minimum set of
way-points needed for the path-following MPC.
The proposed strategies are tested in both simulated and real environments. The
implementations are carried out in C++ and fused into DTU’s Mobotware platform [1],
such that a Small Mobile Robot (SMR) is enabled to solve different challenging scenarios,
safely and autonomously. The results demonstrate the appropriateness of the proposed
solutions.

CHAPTER 1
Project Description
Navigation is in English Oxford Dictionary defined as “The process or activity of ac-
curately ascertaining one’s position and planning and following a route.” Mobile robot
navigation is thus the robot’s ability to determine its own position in its frame of refer-
ence, planning a route towards some goal location and following that route with good
precision while avoiding dangerous situations such as collisions and unsafe conditions. In
order to navigate in its environment, the robot must have a map or other representation
of the environment, and the ability to interpret that representation and act accordingly.
Navigation can thus be achieved by the combination of three fundamental competences:
sense, plan and act. In the sense stage, the robot uses its sensors to obtain meaning-
ful data about its surroundings. This data is then processed and combined with prior
knowledge to obtain highest degree of useful information. In the planning stage, all the
collected information is combined with the objectives, or mission commands, and based
on all of this, the robot decides what is the most appropriate course of action. Finally,
in the act stage, the decisions taken during the planning stage are simply carried out.

2
1 Project Description
This thesis deals with all three aspects of robot navigation, in both known and
unknown environments. The robot considered throughout this thesis is a differential
drive unicycle-type robot, but most of the methods and also the developed code can be
fairly easily adapted for robots with other configurations. The actual implementations
are tested on a ”Small Mobile Robot” (SMR) which is available at DTU’s department
for Automation and Control.
1.1
Motivation
The recent growth of the use of mobile robots in the industry, as well as private homes,
has led to the development of autonomous solutions for numerous applications, such as
self driving cars, lawnmowers, vacuum cleaners, guide bots, cargo transport and so on.
Historically, many solutions required the environment to be modified before a solution
could be installed, usually by installing guide marks, signal beacons or other systems at
the location to help the robot find its way. Installing such systems is a time consuming
as well as costly process which only helps to solve highly specific problems. But, with
the growth in available computing power and increased sensor precision, new methods
for robot navigation are becoming increasingly feasible for realtime use. By enabling
a mobile robotic system to navigate safely and autonomously in an unknown environ-
ment, one can greatly reduce the time and money needed for the realization and thereby
facilitate the transfer to an autonomous robot based solution. Although the above men-
tioned aspects of robot navigation are among the most researched disciplines inside the
robotics community, they remain relevant studies and new methods and technologies are
continuously investigated in order to achieve higher performance and better accuracy.
Another strong motivator is that the field of mobile robot navigation is among the
most interdisciplinary fields in modern science [2], thus making it a very good study
for strengthening my personal skill in engineering over a broad spectrum of scientific
disciplines.
1.2
Project goals
The main goal of this project is to enable a mobile robot to navigate safely, precisely
and autonomously in an unpredictable and partially unknown environment, avoiding
obstacles as they are encountered, while fulfilling its main objective. To achieve this,
the overall goal is split into a number of sub-goals:
Motion Control
The first sub-goal is to enable the robot to follow a path, given as a
series of way-points connected by straight line segments, possibly with perpendicular or
even sharper intersections. The motion control should be such that the robot moves in
a way that seems natural to a human inspector, i.e. the motion of the robot should be
==PAGE:19==
1.2 Project goals
3
retained and smooth which suggests the use of a predictive controller that will embark on
a turning before the way-point is reached. Furthermore, in order to avoid situations that
will jeopardize the stability and safety of the robot or cause saturation in the motors, the
vehicle heading- and angular -velocities and accelerations must be properly constrained.
Map
The second sub-goal is to design and implement a map for representing the envi-
ronment, and to enable that map to be interactively updated from sensor data obtained
from the laser scanner and from the wheel odometry sensor. The map representation will
be processed in various ways allowing the robot to extrapolate useful information about
the distance to obstacles at all points and also differentiate between what is known and
what is still unknown territory.
Path Planning
The third sub-goal is to design and implement a path planning al-
gorithm capable of computing an (optimal) obstacle free path from the robot’s current
map location to some goal, and to formulate that path as the minimum set of way-points
needed for the path following controller. The path will be optimal only with respect to
the cost function which is minimized.
Autonomy
The last sub-goal is to make the robot capable of autonomously solving
various problems by connecting the methods above with some problem specific cognition.
The autonomy level for the motion controller, in itself, is simply that the robot will
drive the route sent to it by the operator without further human intervention. The
autonomy lies in how the algorithm controls the motion of the robot based on the control
parameters. For the obstacle avoidance, a couple of scenarios will be implemented:
1. Go to point: The robot should be able to drive in an unknown environment with
obstacles from its start location to some goal location while avoiding obstacles
along its way.
2. The Labyrinth: The robot should be able to solve a narrow ”one-way” labyrinth
populated with many obstacles. Navigating a robot through such a labyrinth using
direct commands is a challenging task, therefore it is a good idea to implement a
general strategy.
1.2.1
Approach
For each of the sub-goals listed above, many possible technologies has been investigated.
The work in this phase mainly involved reading papers, articles and books, trying to
get an understanding of the state-of-the-art, so that an appropriate method could be
selected for further investigation, development and implementation. Next, a solution to
each sub-problem was implemented in MatLab and tested in different scenarios using
set of functions simulating the behavior of a robot with laser scanner.
During the
simulations, the solutions were further developed and extended with new capabilities.
==PAGE:20==

1 Project Description
When this process completed with promising simulation results, the developed methods
were implemented in C++ and optimized in terms of speed and memory. Finally, the C++
libraries were wrapped in functions/plugins that serves as a bridge between my general
purpose libraries and DTU’s Mobotware platform which is running on the SMR (Small
Mobile Robot) that has served as the test robot for this project. SMR-CL1 commands
has been implemented such that most of the above can be achieved on the robot with
intuitive and simple script commands. After implementing the solutions on the SMR,
additional testing and tuning was performed, and some of the methods were further
improved upon.
1.3
Overview
The thesis is structured as follows. Chapter 2 covers an analysis of the theory needed for
the fulfillment of the project. The analysis covers; A description of the coordinates sys-
tems and the transformations between them. A kinematic study related to the unicycle
(SMR) robot. An analysis of each of the blocks; Motion control, Perception, Map Build-
ing, and Path Planning. The analysis of each block results in an appropriate method
being selected for implementation. Chapter 3 presents the design and implementation
of those selected methods and also the design and implementation of the autonomous
solutions to the scenarios; ”Go to point” and ”Labyrinth”. Also described in chapter 3
is how to use the implementations and how they were integrated with Mobotware. In
Chapter 4, the implemented solutions are tested in simulated as well as real environ-
ments, using a real (SMR) robot. The performance is evaluated and many figures with
results are presented to demonstrate the usefulness of the proposed strategies. Chapter

2.3
Motion Control
Motion control for nonholonomic robots is generally not an easy task. However, it has
been studied intensely for many years and many good control strategies have emerged
from the research. Some of the most popular control types are:
• Open-loop control: The wanted trajectory is split into a number of simple
motions such as straight-line segments, circular arcs and in-place turns, which are
carried out in open loop, see [2].
• Path following: The path following problem describes the task to follow a given
path while the forward velocity v not being part of the control problem. Thus, the
controller seeks to minimize the distance, and heading error, to the given path by
controlling only the angular velocity ω, see [4]–[6].
• Trajectory tracking: A virtual robot having the same kinematic is tracked. Both
heading and angular velocities are controlled. The controller seeks to minimize the
posture difference between the robot and the virtual reference, see [4], [5].
• Posture stabilization: A feedback control law that will drive the robot to some
reference posture from an arbitrary starting posture, see [2], [5].
The motion controller selected for implementation in this project is of the path-following
type, and the rest of this section will explain the requirements to the controller along
with the theory needed to realize a suitable solution.
Path Following
The path-following problem at hand is a subset of the general path-
following problem related to under-actuated vehicles with non-holonomic constraints
(such as the unicycle robot). The paths can in general be arbitrarily defined, but here
it is assumed that the robot has to follow a path, given by straight-line segments be-
tween a series of waypoints. Especially in indoor environments, such as factories, storage
facilities and office environments, where rooms often are rectangular, possibly with per-
pendicular intersections, this kind of paths are often encountered. A good path-following
controller should therefore be capable of maneuvering the mobile robot safely through
such intersections. Its motion should be retained and smooth, and it seems natural
to anticipate upcoming intersections and to embark on a turning before the actual way-
point is reached. This suggests the use of a predictive control strategy, where the control
signals are based on predictions of the robot’s posture. Furthermore, in order to avoid
situations that will result in unsafe conditions or cause saturation in the motors, the ve-
hicle heading- and angular -velocities and accelerations must be properly constrained. In
path-following control where only the angular velocity ω is controlled, the extra degree
of freedom (the ability to control v) allows for the constraints to be satisfied by using
velocity scaling. A good such MPC based path-following controller is presented in [6],
and the controller implemented in this project has used [6] as a starting point. However,
numerous improvements has been made, including a punishment on the change-rate of

velocity trajectory generator which is only responsible for delivering references to the
motor controllers, only constraints on the velocities and accelerations of the robot are
considered. The constraints can be physical limitations or set by the operator as tuning

generator. This is unwanted since a hierarchical control structure seems preferable, and because
motor torque generators are already implemented and installed on the SMR’s.

2.3.3
Kalman Filter
The Kalman filter is an efficient recursive filter that computes the optimal reconstruction
ˆxk|k and prediction ˆxk+j|k of the internal state xk of an LTI state space model, given a
series ”noisy” measurement y0:k, the previous inputs u0:k−1, and a model of the system.
The resulting reconstruction estimate ˆxk|k ∼N(ˆ¯xk|k, ˆP k|k) is a weighted average between
two normal distributions, one for the measurement (filter) update ˆxf,k ∼N(ˆ¯xf,k|k, ˆP f,k|k)
and one for the time-update, or prediction ˆxp,k+1|k ∼N(ˆ¯xp,k+1|k, ˆP p,k+1|k). The weight-
ing between these distributions are determined by the Kalman gain Kfx,k and, if there is
cross-correlation between the process noise and the measurement noise, Kfw,k. However,
in this assignment no cross correlation between process noise and measurement noise is
assumed, i.e. Kfw,k = 0. The Kalman filter was co-invented and developed by Rudolf
E. Kálmán and presented in his 1960 paper: [8]. The Kalman-filter can compactly be
stated as calculating
ˆxk+n = E[xk+n | y0:k, u0:k−1]
(2.38)
ˆP k+n|k = Cov[xk+n | y0:k, u0:k−1]
(2.39)

2.3.4
Model predictive control
Model predictive control (MPC), or receding horizon control (RHC), is an advanced
control algorithm that has the ability to anticipate future events and therefore start
reacting before an event occurs. In MPC control strategy, at each sampling instant,
the current control law is obtained by solving a finite horizon open-loop optimal control
problem. An optimal control sequence is achieved and only the first control action in
the sequence is implemented as control input. With the current state as the initial state,
this on-line optimal control problem will be solved at each sample instant, repeatedly.
MPC control can be achieved easily for linear as well as non-linear systems. The basic
working principle of Model Predictive Control can be seen in Figure 2.6. It is worth
noting that MPC can deal with input or state constraints in a very natural way, since
they are formulated directly into the optimization problem. A good introduction to
Linear MPC can be found in [9] and [10].
In most cases, MPC controllers makes use of an optimizer for computing the optimal
control sequence, but in some cases the constraints can be satisfied using other methods,
which, as mentioned previously, is the case for the path-following controller developed
and implemented in this project. In this case the constraints can be satisfied by scaling
the heading velocity of the robot. Therefore, it is only needed to solve the unconstrained
MPC problem and then apply velocity scaling afterwards. This approach results in a very
fast MPC controller, since the MPC computation at each sampling instance turns out to
be just a few matrix multiplications, as opposed to online minimization of a quadratic
program subject to inequality constraints which is the ”normal” way of computing a
constrained Linear MPC.

Perception
Obtaining situational awareness is one of most important tasks of any autonomous
system. This is done by taking measurements using various sensors and then processing
the measurement data to extrapolate useful information about the environment.
A
multitude of sensors are available, but in this thesis only odometry and active ranging
(laser scanner) are covered.
2.4.1
Odometry
The idea behind Odometry is to use data from the robot’s motion sensors to estimate
change in position over time. In robotics, it is used to estimate position relative to
a starting location. Odometry is highly sensitive to errors due to the integration of
velocity measurements over time to give position estimates. Therefore, to use odometry
effectively, rapid and accurate data collection, processing, and instrument calibration,
are required in most cases. In mobile robots the most common form of odometry is
wheel-odometry, in which the position and velocity estimates are obtained from a rotary
encoder attached to each wheel.

a suitable way arises. An efficient, and highly suitable, way of storing such data is by
building maps to represent the environment and to navigate it, just as humans have done
throughout history. Maps can be represented in many ways, but in robotics two main
paradigms exists: topological maps and metric maps. A topological map describes the
connectivity between feature points, such as hallway junctions, in the environment, by
graphs, see Figure 2.8. The 2005 paper by Huang and Beeyers [12] describes various as-
pects of topological maps for robots with limited sensing capabilities. Topological maps
offers a mapping strategy with a very low memory consumption, which can be highly
attractive in many cases, but on the other hand, it lacks the ability to model free space
in a suitable way. In metric maps, the geometry of the world is explicitly represented,
either in a continuous space representation using feature extraction techniques, or in
discrete space using grid-based maps where the probability of a cell being occupied is
kept in memory. The problem with the continuous representation, is that it relies on
feature extraction, which might not be able to extract all that is needed or might be
slow due to all the necessary processing used for extraction. Grid-maps on the other
hand relies on simple processing but can have a big memory consumption.For instance;
a 9x9 meter area with a cell size of 3x3cm has 90000 cells. However, many ways of de-
composing metric maps to save memory exists in literature, see [2] for an overview. For
this project, a grid-based map is selected, partly because the implementation is simple

2.5.1
Grid-based Maps
A grid-based map, often called an occupancy grid map, is a simple way to represent
data obtained from the robot’s sensors, where the world is modeled by a grid with a
given cell size. Each cell can be in one of the following three states; unknown, free,
or occupied. A simple approach to grid-map building is to update the map by casting
rays from the center of the laser scanner to each of the scanpoints. This can be done
using Bresenham’s line algorithm [13] to iterate over the xy-grid, marking all cells along
each scan line towards the scan point as free, while marking all the scanpoints that lies
within some threshold as occupied. In Figure 2.9 the result of the mapping process in
a simulated environment is shown and it is easily verified that the map matches very
well with the ”real” world representation. In order to facilitate the planning process, the
map can, as shown on the right plots, be dilated such that the robot can be considered
as a single point.

2.6
Path Planning
In robotics, path planning is a term used for the process of finding an obstacle free
path from the robot’s initial position to the goal location, while satisfying movement
constraints and possibly optimizing specific parts of the movement, such as minimizing
the total length of the path or maximizing the distance to obstacles in the environment,
etc.
Configuration Space
Pathplanning for robots, manipulators or mobile, is formally
done in a representation called configuration space.
A single configuration describes
the pose of the robot, and the configuration space C is the set of all possible robot
configurations. The dimensionality of the configuration is determined by how the robot
is represented:
==PAGE:46==
30
2 Theoretical Work
• If the robot is considered as a particle moving in the two-dimensional plane, then
C is a plane, and the configuration can be represented by two parameters: (x, y).
• If the robot is considered, more accurately, as a rigid body moving on a 2D plane,
the workspace is still two dimensional. However, C is the special Euclidean group
SE(2) = R2×SO(2) (where SO(2) is the special orthogonal group of 2D rotations),
and the configuration space can be represented by three parameters: (x, y, θ).
Completeness
A classic term in the discussion of robot architecture. In [2] the term
is defined as: The robot system is complete if and only if, for all possible problems (i.e.,
initial belief states, maps and goals), when there exists a trajectory to the goal belief state,
the system will achieve the goal belief state. Thus, when a system is incomplete, then
there exists a problem for which, although there exists a feasible solution, the system
fails to generate it. Analytically, it is important to understand how completeness is
compromised by particular parts of the system [2].
Many path planning algorithms exists, for solving the navigation problem. Among
the most popular are:
• Artificial potential fields (APF)
• Rapidly-Exploring Random Trees (RRT)
• Dijkstra’s Algorithm
• A∗search algorithm
all of which will be briefly covered in the following.
2.6.1
Artificial potential fields (APF)
Artificial Potential Field [14] is a popular approach to path planning (see for example
[2] or [15]). A common approach is to treat the robot’s configuration as a single point
(such as an electron or a particle in fluid) in a potential field that combines an attractive
potential at the goal location, and repulsive potentials at the location of known obstacles.
A big advantage of the method is that the trajectory can be generated with only little
computational effort. But, on the down side, the method is not complete, since the
particle can get stuck in local minima. Artificial potential fields can be achieved by using
the equations related to electrostatic potential fields, fluid field dynamics or many other.
The main importance is that the attractive potential generates a strong force towards the
goal location that spans the entire configuration space and that the repulsive potential
generates a very strong force near the obstacles that effectively repels the particle and
no force at the ”safe” distance from obstacles, such that the trajectory is not affected
when unnecessary.

2.6.4
A∗search algorithm
A∗is an informed search algorithm, or a best-first search, meaning that it solves problems
by searching among possible paths from the starting node towards the solution, or goal
node. The algorithm searches for the solution that incurs the smallest cost according to
the cost function
f(n) = g(n) + h(n)
(2.90)
where f(n) is the total cost of the path starting from the start node to the goal with
g(n) being the cost of the path from the start node to n and h(n) being a heuristic that
estimates the cost of the cheapest path from n to the goal.
The algorithm was first described in [19] in 1968 by Peter Hart, Nils Nilsson and Bertram
Raphael of Stanford Research Institute (now SRI International). It is a natural extension
of Edsger Dijkstra’s 1959 algorithm mentioned above, achieving better performance by
using heuristics to guide its search. Figure 2.12 shows the same paths computed by the
A∗search algorithm using 8-point connectivity. Notice the performance improvement
over Dijkstra’s algorithm (Figure 2.11) by how much less nodes were expanded during
its search.

to vertices that are in close proximity of obstacles, or terrains that are not meant for
driving, but where driving is possible. This will be clarified further below where an
obstacle repulsive potential is added to the cost function.
The Heuristic function
The Heuristic function can also be used to control the be-
havior of the A∗algorithm:
• For the algorithm to find the actual shortest path, the heuristic function h(n)
must be admissible, which means that it must never overestimate the actual cost
of going from node n to the goal node.
• If h(n) is exactly equal to the cost of moving from n to the goal, then A∗will only
follow the best path and never expand anything else, making it highly efficient. In
many cases, the computation of an exact heuristic is not possible, but it is nice
to know that given perfect information, A∗behaves perfectly. Furthermore, the
better the heuristic, the better the performance of the A∗algorithm.

• If the heuristic is set to zero, A∗turns into Dijkstra’s Algorithm, which is guar-
anteed to find a shortest path but will examine far more nodes than necessary,
thereby slowing down the process.
• If h(n) is sometimes greater than the cost of moving from n to the goal, then A∗
might run faster, but is not guaranteed to find a shortest path.
• If h(n) is much to high, relative to g(n), then only the heuristic plays a role, turning
the algorithm into a greedy Greedy Best-First-Search, which might perform very
poorly in the presence of obstacles.
For grid-maps without diagonal movement; the Manhattan distance (L1-norm) and with
diagonal movement; the diagonal distance (L∞-norm), are exact heuristics in obstacle
free maps. Moreover, they are admissible in any case. The Manhattan distance is

behavior of A∗and the computed optimal cost path can be manipulated to match the
wanted behavior in most cases, making the algorithm highly versatile.

Breaking Ties
One thing that can lead to poor performance is ties in the heuristic.
When several paths have the same f value, they are all explored, even though it is only
needed to explore one of them. Therefore an important part of an efficient A∗algorithm
is to break such ties and only examine one of the equal cost nodes. One way to do this is
to carefully construct A∗’s priority queue so that new insertions with a specific f value
are always ranked better (lower) than old insertions with the same f value. A different
way is to use a tie-breaker that favors some specific behavior, such as preferring paths
that are along the straight line from the starting point to the goal. The effect of such a
tie-breaker is shown in Figure 2.14 and it should be noted that there is no problem of
having both tie-breaker and repulsive potential active simultaneously.
Due to the ease of expanding the method and the optimality of the solution with
respect to the cost function, the A∗path planner expanded with repulsive potential and
tie-breaker is chosen as the path planning algorithm to be implemented in this project.

Implementing an MPC sets the demand for a good Linear Algebra package, since it in-
volves quite some matrix and vector manipulations (as clearly shown in Section 2.3.4). A
few packages were considered (GSL, LAPACK, OPENCV) before settling on the Eigen
library: Eigen is a C++ template library for linear algebra: matrices, vectors, numerical
solvers, and related algorithms [20]. Eigen is partly chosen because it offers aligned mem-
ory, which speeds up the math operations a good deal, partly because of the beautiful
interface that it offers, and finally because it is a header only template library, which
facilitates the build process. Eigen is the only non-standard functionality that has been
used for the path following MPC implementation. Important design criteria have been
good real-time performance, high degree of customization via tuning capabilities, easy
integration onto new platforms and core system stability. To ensure a good performance;
memory is re-allocated and variables recomputed, only when necessary. Similarly, when
possible, values are pre-computed and simply retrieved from memory at runtime. A high
degree of customization is supported by the many runtime adjustable tuning parameters.
The pathfollowing MPC is implemented in a class named:
RobotPathFollowMPC. This class uses two classes internally, namely: LTIsys and LMPC:
• LTIsys: Represents a Linear Time-Invariant state space system, such as (2.31) or
(2.36).
• LMPC: Implements a generic Linear Model Predictive Controller capable of solving
the unconstrained MPC problem (2.53) for any given state space system, given as
an LTIsys pointer (i.e. LTIsys*).
The helper classes LTIsys and LMPC will not be explained further here. They are only
mentioned because the implementations are highly general purpose with dependency

3.5
Map Implementation
In this section, the implementation of the occupancy grid map, and thus the solution
to the second sub-goal stated in Section 1.2, is covered. Important design criteria for
the grid map implementation have been good real-time performance, good visual rep-
resentation, ability for both ”local” and ”global” mapping, full integration with the
implemented A∗path planner, easy integration onto new platforms and core system sta-
bility. To ensure a good performance; memory is re-allocated and variables recomputed,
only when necessary. To facilitate a good visual representation, OpenCV (Open Source
Computer Vision Library) [21] is used for the implementation, making the OpenCV Mat
class the main data-structure used for the map representations. Besides good options
for map visualization, OpenCV offers the following desirable functionalities:
• Image dilation: A morphological operations that dilates an image using a specific
structuring element. In this case a circle with radius equal to half the robot’s width
is used to produce the dilated map such that the robot can be considered as a single
point.
• Distance transform: The OpenCV distance transform operator takes a binary
image as input and generates an output image such that the gray level intensities
of the points inside the foreground regions are changed to their respective distances
from the closest 0 value (boundary). In this project the distance transform is used
to compute the distance to obstacles in the environment, such that A∗can use the
information with the obstacle repulsive potential. Since the obstacles are 1 values,
the dilated map must be inverted before applying the distance transform, which
can easily be done using the bitwise_not operator.
The occupancy grid-map is implemented in a class named: GridMap. This class uses
two classes internally, namely: Astar and LaserScanner:
• LaserScanner: Represents a laser scanner. It contains data structures for main-
taining the laser scan data Sk and the scanner pose in the inertial frame ζi
k. More-
over, it implements several functions for returning the scan data in various coordi-
nate system and finally, it implements a simulated laser scanner, such that it can
==PAGE:62==

be used to assist the development process by real-world decoupling and when no
real laser scanner is available.
• Astar: Implements the A∗search algorithm with obstacle repulsive potential in
the cost function and the ability to work with wrapped maps.
The A∗search
implementation is explained in the next section.
The helper class LaserScanner will not be explained further here, as it is only used
internally for computing simple transforms, but the simulator has received a great deal
of attention to avoid numerical problems and might be useful in other projects. It is a
port of a MatLab implementation which was written by Christian Overvad og Kasper
Strange in 2006 and re-written by Nils Axel Andersen in 2011.
In order to make the map capable of functioning as a ”local” as well as ”global” map,
all reading and writing operations has been implemented as two-dimensional circular
buffers. In order to make the two-dimensional circular buffers system work seamless
with the A∗path finding algorithm, allowing its search to also wrap around the edges of
the map, it was instrumental to let the GridMap have full control over an Astar instance
such that the two map representations could be tightly synchronized with minimal com-
putational overhead.

This section covers the design and implementation of the A∗pathfinding algorithm
and thus, the solution to the third sub-goal stated in Section 1.2. The A∗algorithm,
implemented in this project, have used an implementation by Makito Sumi as a starting
point, see [22]. However, the code has changed significantly and many extensions and
improvements have been added. The fixed problems with the scaling of the heuristic
function led to great performance improvements, so they were suggested to Makito Sumi

50

who modified his original to mach the suggestions. Besides the corrections, the algorithm
has been expanded with a tie-breaker and the possibility of adding an obstacle repulsive
potential into the cost function as described in Section 2.6.4. Also, since the OpenCV
version running on the SMR’s does not support the distance transform, the possibility
for the Astar algorithm to calculate the obstacle distance as nodes are expanded, has also
been implemented. Luckily, in most cases, computing the distance using this method is
way faster than using OpenCV’s distance transform, because the distances to obstacles
are only computed for the nodes expanded by the A∗search.
Important design criteria for the A∗implementation have been good real-time perfor-
mance, good visual representation possibilities, ability to work with a wrapped grid map,
protection from getting stuck if no path exists, and core system stability. To ensure a
good performance; the search process uses pointer arithmetic such that only pointers
needs to be passed between various functions. Furthermore, memory is re-allocated and
variables recomputed, only when necessary. To facilitate a good visual representation,
some functions for visualizing the map and the path using OpenCV are available if the
following the macro ASTAR_USE_OPENCV is defined. If not, the class can be compiled
without non-standard dependencies. The A∗search algorithm is implemented in a class
named: Astar. This class uses two classes internally, namely: MapSize and MapNode:
• MapSize: Data Structure that maintains the size of a grid-map.
• MapNode: Represents a Node in the Graph.

Implementation
The implementation of the ”go to point” solution is quite simple
when the motion controller, the grid mapping and the path-finding algorithms are im-

the implemented solution: When the robot receives the goal point from the operator, the
main control loop is started: First, it is checked whether the goal is reached or not. If
not, the robot posture is determined by localization. When new laser scan data becomes
available, the map (and dilated map) are updated. After updating the map, the path is
re-planned using the implemented A∗algorithm. The path is simplified, converted to a
vector of world coordinate waypoints and send to the MPC, such that the path execu-
tion can begin. After executing a small part of the path, the whole process is repeated
until the goal is reached. One important detail about the implementation is that the
A∗search must be allowed to search across unknown areas within the map, since the
goal point can be placed far from the robot in unknown territory. Therefore, the Astar
implementation includes the flag ”unknownAsObstacle” that should be set to false, and
also a user definable terrain cost ”unknownCost” for crossing over unknown territory.
The penalty for crossing unknown areas is necessary, especially when the obstacle repul-
sive potential cost is enabled. Otherwise, the path planner will be to likely to assume
that it is cheapest to take the long route around all obstacles, but in unknown areas, the
probability of encountering an obstacle is not zero. Obviously, that must be reflected in
the cost function in order for the A∗algorithm to come up with an appropriate solution.
If the unknown cost is high, the algorithm will prefer to travel across known areas and
thus aim for directions where much is known.

CHAPTER 5
Conclusion
The primary objective of this project was to enable a mobile robot to navigate safely,
precisely and autonomously in an unpredictable and partially unknown environment,
avoiding obstacles as they are encountered while fulfilling its main objective. Based on
the results shown in the previous section, it can be concluded that the main objective has
successfully been achieved. The proposed solution has enabled the unicycle-type (SMR)
robot to perform fully autonomous missions in known and unknown environments of
high complexity. In the remaining of this chapter, it is discussed how the individual
sub-goals of the problem statement have been solved. This is followed by some ideas for
future improvements.
Motion Control
The first sub-goal was to enable the robot to follow a path, given as
a series of way-points connected by straight line segments. This was achieved by imple-
menting a path following model predictive controller, capable of anticipating upcoming
intersection and react before reaching the turning point. By using velocity scaling to sat-
isfy the velocity and acceleration constraints, the MPC could be realized without using
online quadratic programming optimization. This results in a fast and reliable controller
highly suitable for use in a hard real time system such as MRC. The implemented MPC
performs as intended in both simulated and real (SMR) experiments.
Map
The second sub-goal was to design and implement a map for representing the
environment, and to enable that map to be interactively updated from sensor data. This
has been achieved by the implementation of a grid based map. The map is implemented
as a two-dimensional circular buffer such that the map can be used in both as a ”local”
as well as ”global” map, by simply turning on/off the map wrapping. Using OpenCV’s
Mat class for representing the map provides good options for visualization. Furthermore,
several OpenCV functions are useful when working with grid maps. In this project the
dilation and distance transform has been utilized.
Path Planning
The third sub-goal was to design and implement a path planning al-
gorithm capable of computing an (optimal) obstacle free path from the robot’s current
map location to some goal. To achieve this, an A∗algorithm, augmented with obstacle
repulsive potential and obstacle distance calculation during node expansion, has been de-
veloped and implemented. The implementation uses pointer arithmetic for maintaining
and expanding the graph, as well as constructing the path, making the implementation
quite efficient.
==PAGE:90==
74
5 Conclusion
Autonomy
The last sub-goal was to make the robot capable of autonomously solving
various problems by connecting the methods above with some problem specific cognition.
Two cases were set up to verify the appropriateness of the suggested solutions:
1. ”Go to point”: The robot was successfully enabled to drive in an unknown
environment with obstacles from its start location to some goal location while
avoiding obstacles along its way, autonomously.
2. ”The labyrinth”: The robot has been shown capable of solving a narrow ”one-
way” labyrinth populated with many obstacles, autonomously.
All the methods has been implemented as self contained, easy to use, libraries. The
libraries was successfully linked into DTU’s Mobotware platform, such that testing could
be performed with the SMR robots available at the institute. Finally, included as part
of the implementation are functions for simulation the behavior of a unicycle robot with
laser scanner, such that further development and parameter tuning can be facilitated by
the possibility to directly simulate the behavior of the relevant closed-loop systems.
5.1
Future work
The implemented solutions presented in this thesis, provides a good foundation for future
work with autonomous applications for mobile robots. The next steps could be to expand
and improve the system by implementing the following suggestions:
• Currently, the robot localizes itself using plain wheel odometry. As mentioned,
odometry is highly sensitive to errors due to the integration of velocity measure-
ments over time to give position estimates. The odometry can be improved by
including the laser scan data in the instantaneous velocity estimation, know as Li-
dar Odometry, see e.g. [23]. If scan matching is used for Lidar Odometry one can
also ”close the loop” when the robot sees a part of the map that it already knows.
Lidar Odometry with loop-closing is known as a SLAM (Simultaneous Localization
and Mapping) system, see e.g. FastSLAM [24], GMapping [25] or tinySLAM [26],
for some popular approaches.
• The A∗search algorithm can possibly be optimized a by using online graph pruning
for pathfinding on grid maps, such as the Jump Point Search discussed in [27]. The
Jump Point Search method presented in [27] would have to be expanded in order to
work with maps with uneven terrain costs as the one implemented in this project.
However, in maps that contains uniform cost regions (when the obstacle distance
is greater that the threshold), we simply treat any neighbor which is of a different
terrain type to the current node, as forced. This will allow JPS to quickly search
across a uniform-cost region, stop to expand a node when crossing into a different
region, and continue jumping on the other side. Speed improvements by factors of
10-20 has been reported for uniform cost grids.
==PAGE:91==
5.1 Future work
75
• The path definition is currently implemented directly in the RobotPathFollowMPC
class. It would be suitable to implement a PathManeger class such that the ma-
nipulations to the path could be performed there.
Furthermore, the proposed
PathManeger class should be usable with the GridMap and path finder.
This
would allow for the paths to be fully compatible with the mapping and obstacle
avoidance system. Effectively this would expand the system from being able to
”Go to point” to ”Follow path while avoiding obstacles” which is far more generic
and usable. Implementing this would be quite straight forward, but unfortunately
it was deemed unfeasible for implementation withing the duration of the project.

Week
Activity
Description
4-6
Research Phase
Investigating different approaches to robotic
control, MPC strategies in particular.
6-8
Matlab Implementation
Implementation of various controllers in MatLab to
asses the quality of different control strategies.
9-12
C Implementation
Implementation of selected controllers as a C Library.
13
Mobotware Wrapper
Writing C-wrapper functions to match the interface in
AUT’s Mobotware platform and compile for Mobotware
14
Path-Following SMR
Test Path Following with an SMR using the
position estimate provided by the localizer
15-16
Thesis Writing
Prepare Latex Template and document the implemented
controller and evaluate its performance both in simulation
and in reality.
17-18
Autonomous
Navigation
Strategy
Extend the implementation to not rely on the Localizer.
Instead the path should be generated, based purely on the
robot’s sensory perception and som set of navigation rules.
19-20
MatLab Implementation
Autonomous Navigation Strategy
20-22
C Implementation
Autonomous Navigation Strategy
23-25
Thesis Writing
Finishing the Thesis, cleaning the code, etc. Thesis
writing in obviously an ongoing process that runs
simultaneously with the project goals.
