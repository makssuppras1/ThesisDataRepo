Abstract
The Heterogeneous Compute Library is a proof-of-concept framework developed to
create a streamlined, heterogeneous compute pipeline from OpenGL, allowing the
user to focus on the CPU/GPU code simultaneously without sacrificing clarity. We
have implemented two different graphics techniques, Fast Approximate Anti-Aliasing
and Screen Space Ambient Occlusion, as well as a data structure for future work with
ray-tracing, in HCL. The first two techniques have been implemented in both front-
end and back-end of HCL, while the data structure has been implemented in back-end
only. Through these implementations we examine the functionality and flexibility of
HCL as a heterogeneous computing framework.

CHAPTER 1
Introduction
In this section, we will first provide the motivation for this project, and the area of
research it falls within. We will then outline the project itself. Finally, we will provide
an overview of the remaining report sections.
1.1
Motivation
Traditional graphics programming APIs draw a very clear line between what runs
on the CPU and what runs on the GPU, with most GPU processes being managed
through various bookkeeping methods on the CPU. Very little programming is done
directly on the GPU. With the progress and developments within CPU and GPU
compatibility, and the advent of heterogeneous system architectures, it would make
sense for graphical APIs to also move in this direction. While some APIs have been
developed in response to this - such as Metal and Vulkan - they generally still require
high degree of bookkeeping and memory management alongside the GPU program-
ming. Moving this memory allocation, shader management, draw calls and so on
more to the back of an API would let the developer focus more directly on the spe-
cific executions that the CPU and GPU should handle. This would essentially give
the developer the feeling of writing code to and allocating memory on the GPU as if
it was another CPU, which is an interesting idea of a heterogeneous system.
1.2
The Project
This project builds upon a proof-of-concept framework, HCL: Heterogeneous Com-
puting Library, developed by Alessandro Dal Corso at DTU. HCL aims to provide
a more user friendly framework through which programmers inexperienced with, or
fed up with, traditional graphics pipelines can render 2D or 3D graphics, or run cal-
culations through compute shaders, without directly handling the necessary memory
management. While still using a traditional graphics pipeline in the form of OpenGL,
this gives the end user a feeling of operating in a heterogeneous system, writing code
to and managing memory on CPU and GPU simultaneously.

CHAPTER 2
Framework
In this chapter we present the general workings of HCL: Heterogeneous Computing
Library, upon which this project is built, including a comparison of HCL and tra-
ditional graphics programming using OpenGL. After this, we look at similar APIs,
specifically Metal and Vulkan.
2.1
HCL: Heterogeneous Computing Library
HCL is a proof-of-concept library developed in an effort to address the ”problems of
expressing a performance-oriented GPU without sacrificing clarity” [Cor16]. The li-
brary was made possible with the movement of GPUs from being specifically dedicated
to specific tasks like vertex and fragment processing, to also having programmable
shaders and performing computations on specialized streaming multiprocessing units.
These units allow for a shift in the graphics pipeline, such that compute shaders can
be used to handle more general computations. However, most of the graphics pipeline
is still implemented in hardware. As such, HCL employs the idea of using compute
shaders before and after sending the vertices to the rasterization pipeline, and receiv-
ing the fragments, to act as easily customizable vertex- and fragment shaders.
HCL is particularly handy for developers who are either inexperienced with graph-
ics APIs, or wish to avoid the hassle of managing memory and synchronization be-
tween the CPU and GPU. Included in Appendix A is a simplified code example of
the rendering of a white ”Hello Triangle” program in OpenGL. This implementation
highlights the amount of resource management necessary for producing even simple
graphical images with OpenGL. In contrast follows an implementation in HCL which
will render the same triangle:

To display the results of rasterization and/or compute shading, the user must cre-
ate an hclTexture object with the hcl_get_display_texture method. This object
will act as a Framebuffer Object. The user can then clear the Framebuffer and pass
the manipulated fragments from the various rasterization- and compute shader-calls
to display on screen.
HCL allows a user to effortlessly create a window, load a number of vertices into
a stream buffer, manipulate the vertices in a compute shader, pass the manipulated
vertices to a rasterization stage, perform specialized fragment shading on the resulting
fragments, and finally pass these manipulated fragments to a framebuffer. As seen
earlier in this section, the code for such a program can be executed in as little as 18
lines (which, if cutting out the fragment shading and simply passing the interpolated
fragment vec4s to the fragment buffer, could be cut down to 13 lines).
However, while HCL is great for handling simple graphics rendering in a minimal
number of lines of code, it is lacking in more advanced control of memory or buffer
types, and has no way of initializing texture objects or atomic counters. An analysis
of the advantages and limitations of HCL is conducted in Chapter 7, after outlining
the implementation processes of the algorithms and data structures in this project.

2.2
Comparisons To Other APIs
HCL aims to provide its user with only the bare minimum of ”bookkeeping”, while
simultaneously letting the user perform computations seamlessly on both CPU and
GPU. HCL is not the only framework with the aim of letting the user program
heterogeneously however. We here give a very brief overview of Vulkan and Metal as
APIs.
2.2.1
Vulkan
Vulkan is derived from the Mantle API, developed by AMD in cooperation with
DICE, the swedish video game developer owned by Electronic Arts. It was originally
intended to be a successor to OpenGL, but was made its own individual API due to
the differences in approach to graphics programming. Where OpenGL is a high-level
programming API with a high degree of driver abstraction, Vulkan was developed to
allow the user greater control over dispatches of compute shader and multi-threading
and has a thin driver level. At also allows greater control over multi-threading on the
CPU.
Compared to HCL, Vulkan allows for a much greater degree of control of pretty
much everything, from memory management to work group dispatches, while still
presenting the user with the ability to run GPU and CPU code in the same files,
seamlessly. However, like OpenGL, Vulkan cannot hide or disregard this degree of
management, and requires a large amount of code for even simple tasks. For example,
the ”HelloVulkan” template available at the ”GPUOpen-LibrariesAndSDKs” reposi-
tory on github1 which sets up a window and a Vulkan context and renders a triangle,
has a total of 412 lines of code in its main rendering file.
2.2.2
Metal
Like Vulkan, Metal is a low-level low-overhead API intended for graphics and compute
implementations, developed for iOS, macOS and tvOS. It was essentially intended
to offer these platforms the benefits of APIs like Vulkan. Unlike OpenGL, Metal is
specifically designed to work with Apple’s hardware architecture.
Compared to Vulkan, Metal requires much less management and can be initialized
fairly simply. However, metal still requires a larger degree of setup before being able
to run simple computations, and is somewhat limited with regard to platform, as it
is only available on Apple products.

Project
This section covers a previous project to extend the HCL rasterization pipeline. First
we present the motivation for said extension, followed by a description of the project
itself. This is then followed by a walkthrough of the implementation. Lastly, we
present a practical implementation made possible by the extension.
3.1
Motivation
The general motivation for this previous project was similar to the motivation for the
project which this report details: implementing extensions and algorithms in HCL in
order to examine to what extend HCL can be used as a general library and framework
for graphics programming and general purpose computing.
3.2
The Project
The aim of this previous project was to implement an extension to the HCL rasterizer.
The rasterizer in HCL before this extension was very simplistic: it took the given data
(such as vertex-coordinates, normals, colors, etc.), and rasterized these into fragments.
However, it was effectively only able to output the fragments ”closest” to the ”screen”
- the fragments with the lowest depth. This limits the possibilities of HCL, as the
front-end user cannot alter the fragment shader that is used in the rasterization pro-
cess.
==PAGE:22==
12
3 HCL Rasterization Pipeline Extension Project
In this previous project, an extension was built into the framework to allow the
rasterization process to capture all fragments rasterized within the frame, regardless
of depth, and output them in the form of a linked list in a streambuffer object. This
previous project was conducted by the author of this report, but never completed.
3.3
Implementation
The HCL framework had three custom shader objects:
• ComputeShaderAssembler
• VertexShaderAssembler
• FragmentShaderAssembler
These shader objects contain the general functionality of the respective shaders in
the form of a stringstream. Depending on the given input and output hclStreamBuffers,
as well as custom uniforms set by the user, these stringstreams would then be com-
piled to have the correct input and output values, as well as the correct custom structs
and uniforms.
The original rasterization setup in HCL consisted of two glDrawElements(...)
calls with two different sets of assembled shaders objects, which are the HCL equiv-
alent of a GLSL Program. Both would have the same vertex shader - taking the
various values given in the input streambuffer and passing them individually through
rasterization to the fragment shader - but had different fragment shaders. The first
shader object would rasterize with writing to the depth buffer enabled, and do no
computations on the fragments themselves, nor output them to the given output
streambuffer. The second would rasterize without writing to the depth buffer, but
would only output values for pixels found by the previous shader to have correspond-
ing fragments, and only the foremost fragments.
Additionally, the framework has an object implemented called a FragmentAddressBlock,
henceforth a fab. Through these fabs, the pipeline can attach different types of
OpenGL texture attachments to the hclTexture acting as the front-end HCL ver-
sion of a framebuffer. Due to the way the pipeline and various objects interact, any
output streambuffer of a rasterization call must have fab attached. The framework
has two types of fabs implemented: a version attaching a depth buffer to the frame-
buffer, and a version with nothing attached, simply acting as an empty version to
allow a streambuffer to have no texture attachments for the framebuffer to read. In
the original HCL version, the rasterizer attached a fab of the first type to the output
streambuffer.

The FragmentShaderAssembler had an enumerator in place to shape the assem-
bled shader depending on whether the rasterization process needed to output raster-
ized fragments, write to the final framebuffer texture, or simply write to the depth
buffer. The first step of the extension was to add a new type to the enumerator,
namely the FRAGMENT_LIST. Furthermore, the original rasterization call was extended
with a new parameter in the form of another type of enumerator, namely the one indi-
cating fab-type. This was done because the front-end developer does not have access
to types of FragmentShaderAssembler, but does have access to fab-type through the
rasterizer. Therefore, the fab-enumerator was extended with a mode for producing
the linked list of fragments.
When initializing a new rasterization stage, the rasterizer then reads the input
fab-type and assembles the correct fragment shader, as well as initializes the new,
desired buffers and textures in OpenGL.
The rasterizer extension builds upon the idea of using a single linked list to link
every fragment of the same pixel together, regardless of depth. Each index in the list
will have the data from the rasterization, along with an unsigned integer designating
the next fragment in the list rendered to the same pixel position. To do this, a 2D
texture object is generated in the dimensions of the rendered window and of data
type GLuint, and initialized to FFFFFFFF in hexadecimals. Along with this texture
object, an atomic counter buffer is implemented to ensure that each fragment gets
a unique linked-list index. The FragmentShaderAssembler is extended with layouts
for the new texture and atomic counter, provided that the desired rasterization type

Every field of the input from the vertex shader input type is stored in its corre-
sponding position in the output buffer type, along with the index of the next fragment

Additionally, the shader program writing to the depth buffer is disabled when
producing this linked-list, so that all fragments in the scene are processed and output.
This way, the rasterizer runs once and writes all fragments to the desired streambuffer,
and can be accessed in the compute shader call by using the compute shader thread
x and y coordinates as index coordinates in the texture for each pixel and walking
the list as desired, depending on the type of shading wanted.
However, the implementation has limitations. Due to the nature of HCL stream-
buffers, initialized buffers cannot be resized, if the new size is smaller than the pre-
vious. This means that the output buffer size cannot be adapted depending on the
number of output fragments, if that number can be smaller in one frame than in
the previous. This of course creates a memory overhead. Additionally, this imple-
mentation requires the data type of the output streambuffer to be of a type with
the same parameters as the data type of the input streambuffer, with one additional
parameter in the form of an unsigned int to hold indexes. Solutions to both problems
were explored during the course of this previous project, but none were implemented
in the course of the project. As the project that is the subject of this report seeks
to continue to build upon HCL in a different way, solutions to these problems were
explored but not implemented, and still exist within the version of the framework
that is presented here.

CHAPTER 4
Fast Approximate
Anti-Aliasing
In this chapter we present our implementation of Fast Approximate Anti-Aliasing
(FXAA) into the HCL framework.
FXAA was chosen as an algorithm for imple-
mentation suitable for heterogeneous computing in HCL because it does not rely on
rendering to a high resolution and then downsampling, which fits poorly with our
extended rasterization linked-list implementation.
First we outline a few of the more common techniques for anti-aliasing in com-
puter graphics. Then we go over the design of the chosen anti-aliasing technique.
We then present the implementation of the design. Finally, we showcase how this
implementation is used by a front-end user of the framework, and how this compares
to a similar implementation in a traditional OpenGL pipeline.
4.1
Techniques
The outlined anti-aliasing techniques are Supersampling Anti-Aliasing (SSAA), Mul-
tisampled Anti-Aliasing (MSAA), and Fast Approximate Anti-Aliasing (FXAA).
4.1.1
Supersampling Anti-Aliasing
One of the first techniques for anti-aliasing to appear on early graphics cards, SSAA
revolves around the principle of taking multiple samples of color per pixel rendered.
In order to do this, the screen resolution is increased during the rasterization stage,
effectively producing more fragments per final pixel each frame. The most common
magnitudes of resolution increase is two times and four times the original screen res-
olution.
==PAGE:30==
20
4 Fast Approximate Anti-Aliasing
Figure 4.1: Examples of different supersampling patterns.
After rasterization produces the additional fragments, different algorithms can be
used to sample fragments around each individual fragments, and are then used to
determine the color of the corresponding pixel in the larger resolution image. When
finished, the image is then downscaled with the new pixels based on the sampling.
While this technique produces excellent results, it is very expensive, as it requires
more GPU memory to store the additional fragments, it requires greater memory
bandwith to perform the many additional reads and write operations, and it requires
the rasterization process to produce and manipulate a greater number of fragments.
Additionally, it performs sampling for all fragments in the enlarged picture, effectively
performing anti-aliasing multiple times for the same pixel.
Strategies to improve
performance include only sampling pixels at the edges of objects, which eventually
led to MSAA.
4.1.2
Multisampled Anti-Aliasing
MSAA was born from the observation that aliasing only occurs at the edges of ras-
terized triangles, since texture fetches do not suffer from aliasing due to mipmaps.
MSAA works in a similar manner to SSAA, in that the scene to be rendered to the
screen is processed at a higher resolution than that of the final screen rendition. How-
ever, while MSAA performs sampling similar to that of SSAA, only the pixel locations
that will be rendered to the final image are processed and written. Furthermore, in
the case of all the sample locations within a pixel being covered by the same pixel,
MSAA only performs a single computation for that pixel, effectively resulting in an
only slightly more expensive pixels than normal rasterization for those cases. MSAA
performance can be further improved through edge detection, to ensure that MSAA
is only performed for pixels with sampling locations covered by multiple triangles.
==PAGE:31==
4.2 Design
21
While MSAA is a serious performance improvement over SSAA, in the case of HCL
it would still require the additional storage, as HCL seeks to move fragment shading
from the rasterization pipeline to compute shaders programmed by the front-end user
by manipulating fragments stored in shader storage buffers. This would effectively
require the user to perform their desired fragment shading over all fragments in the
higher resolution image, before MSAA would downscale that resolution for the final
image.
4.1.3
Fast Approximate Anti-Aliasing
In contrast to the two aforementioned anti-aliasing techniques, FXAA only samples
the final image before it is rendered, as a post-process. Where SSAA and MSAA
depend on rendering the target scene to a higher resolution and sampling from the
resulting image in different ways, FXAA and similar post-processing anti-aliasing
techniques use edge detection-based solutions to find and blur edges on the final im-
age. In the case of FXAA, it samples from the pixels found to its the upper-left,
upper-right, lower-left, and lower-right, and compares the sampled colors over the x-
and y-axis’ to determine the final color. The exact process is discussed in the next
section.
While this technique generally produce images of lower quality than SSAA and
MSAA, it does not require further modification of the rasterization pipeline, it does
not require extensions to the output streambuffers in HCL, and it can be performed
at any time the front-end user wishes. Furthermore, as illustrated in section 4 of
this chapter, it can be implemented with relative ease in both the framework itself as
a post-process extension, and directly into a compute shader by the front-end users
themselves, should they wish to try.
4.2
Design
The version of FXAA designed for this project builds upon FXAA II, as proposed by
Timothy Lottes, the author of the original version presented in [Lot09].
FXAA works by sampling the color values of the pixels to the North-East, North-
West, South-East, and South-West of the working pixels, as well as the working pixel,
and converting them to greyscale. Using the greyscale values as contrast, the algo-
rithm determines which direction to sample in and how far to sample, and returns
a blurred color of the sampled values. This blur is performed over the horizontal or
vertical axis, depending on the direction of the blur. This direction is more prominent

over diagonal lines than over horizontal or vertical lines, as horizontal and vertical
lines do not produce aliasing to the same degree.

CHAPTER 5

Ambient Occlusion
In this chapter we present our implementation of Screen Space Ambient Occlusion
(SSAO) into the HCL framework. SSAO was chosen for implementation because, like
with FXAA, it can be implemented as both an extension to the framework, as well
as directly through front-end by a user of the framework.
First, we outline the design of our SSAO implementation. We then present the
implementation itself. Lastly, we showcase how this implementation can be done by
a user of the front-end of HCL, or implemented as an extension to the framework.
5.1
Design
Our version of SSAO is a straight forward version of the original version developed
by Crytek as presented in [Mit07], with one notable exception: where the original
version version samples around the working pixel in a sphere, our version samples in
a hemisphere oriented along the fragment normal.
The original Crytek method resulted in flat areas being colored darker than they
were supposed to, because the method sampled in a sphere, resulting in half the sam-
ples being behind the processed fragment, thereby contributing to the occlusion factor.
By only sampling in a hemisphere oriented along the processed fragment’s normal, we
can avoid this pitfall. The requirement is that when performing our SSAO process, we
need to have access to the fragment’s normal data. HCL provides this functionality
in two ways: by rasterizing our vertices with the original, built-in rasterizer which
writes to a depth buffer, or by rasterizing our vertices as part of structs, which also
contains vertex normals, and outputting the resulting fragments to a streambuffer
similar structs.

In order to sample the depth around processed fragment, we design two additional
streambuffers and initialize them through the framework, so that their data can be
accessed by the compute shader. The first streambuffer will contain a predetermined
number of random points within the hemisphere. These points will act as our sam-

oriented so that only the direction of its normal changes and not its rotation, it will
result in what is called ’banding’, an effect that produces images which appear like
layers of various gray shades of the same picture, shifted slightly in random directions.
To avoid this, our second streambuffer will contain randomized ’noise’ values, which
will be used to semi-randomly rotate the sample hemisphere for each fragment.

CHAPTER 6
Data Structure
Extension
In this chapter we present the design of our data structure, and the present the
implementation into the HCL framework. In the last section, we outline an algorithm
for traversing our data structure in real-time. Due to time constraints, this algorithm
was not implemented, and thus remains theoretical. As such, while the data structure
has been implemented according to the design, as we have no way of walking through
it in this project, we cannot ensure that it works perfectly, and this data structure
should be taken as a concept implementation of extending the HCL framework with
different data structures.
6.1
Design
The main problems with the two aforementioned data structures, is that it is often
necessary to pre-compute and assemble them prior to actual rendering, since they
need to be sorted to be effectively traversed. While this would also be possible to
implement in HCL, and indeed a voxel grid (not an SVO) was partially implemented
during this project, the two previous implementations focused on real-time applicable
techniques. To stay within that area, a new data structure needed to be designed.
This alternative data structure builds upon the same principles as Voxel Octrees
and Bounding Volume Hierarchies, that is splitting the scene geometry up into dif-
ferent blocks. However, instead of dividing the geometry into smaller and smaller
volumes and storing them in a tree for traversel, our data construction consists of
only two layers of depth.
The idea of our data structure is to use a low resolution 3D texture to register
larger cubic areas in the scene intersected by triangles, and allocate space in a linked
list for another 3D grid of cubic areas within the larger area. This is in principle the
same idea as with a voxel octree, with the main difference being that we only have
==PAGE:46==
36
6 Data Structure Extension
two levels, and that the upper level is already a subdivision of the scene.
Figure 6.1: Example of registered geometry and storage, here in 2D with an 8 by 8
map and 4 by 4 sub-linked-list.
Initiating the ”low res” 3D texture, hencefort a ”map”, to dimensions of 64 by 64
by 64 blocks, and initializing each linked-list, from now on ”bricks”, for each inter-
sected map-area with dimensions 4 by 4 by 4, we can effectively gain a voxel resolution
of 2563 voxels. If we first detect which of the larger voxels are intersected by trian-
gles and mark those using an atomic counter, we can ensure that we only allocate
”brick”-memory for ”map”-voxels intersected by a triangle. We then check for each
smaller ”brick” in each intersected ”map”-index whether that brick is intersected as
well.
==PAGE:47==
6.2 Implementation
37
Since it is possible for each ”brick” to also be intersected by multiple triangles,
we create another linked list with indexes for all primitives processed by our shader.
Whenever a primitive is processed, for every ”brick” that is intersected by that trian-
gle, we add a new index to this linked list and link the ”brick” to this primitive index.
If the ”brick” is intersected by a new triangle, we read the index of the last triangle to
intersect, swap this index with the new one, and store a pointer in the linked list to
the previous index. This way, we have one linked list with primitive indexes, where
each sub-linked list is a list of pointers to all triangles intersecting a given ”brick”.
In order to determine which ”map”-indexes are covered by triangles, we rasterize
the scene using an orthogonal projection and with identical dimensions in all three
major directions. We use the approach described in [Voo92]. For every voxel that we
suspect a triangle intersects, check the following and reject triangle according to test:
• Check if one or more vertices are inside cube, or if all outside the same cube
face
• Check if all vertices are outside same edge face for all 12 edge faces
• Check if all vertices are outside same edge face for all 8 corner faces
• For each triangle edge, if the vertices cannot be disregarded according to previ-
ous tests, check whether the edge they form intersects cube
• Determine whether triangle plane intersects cube diagonals
• If not, disregard triangle, else determine whether plane/diagonal intersection is
within triangle
If the triangle is not rejected, it intersects the cube.
With this approach, we can build the data structure in two steps - building the
”map” and list of primitives, then allocating and populating the ”bricks” - while
keeping memory consumption relatively low.
6.2
Implementation
This design has been implemented as another mode of rasterization - next to the stan-
dard, depth writing mode, and the extension with a fragment list - but integrates the
linked fragment list. This was done to allow for more possibilities when shading and
tracing, and to ensure that no primitives were discarded when building the structure.

The geometry shader for the ”map” reads the vertex coordinates for the three
vertices of the primitive, and determines a bounding volume in ”map” voxels. This
is done by finding the minimum and maximum values of all three dimensions of all
three vertices, and snapping them to the 643 ”map”. As we work on vertices which
have been projected using orthogonal projection, our vertex coordinates are in clip-
space. We therefore scale all vertex coordinates by 32, effectively resizing clipspace
coordinates from [−1; 1] to [−32; 31].
Before we check whether the working ”map”-index has been intersected by the
current primitive, we check whether it has already been intersected by another. If
it has, we skip this voxel, as we already have an index for the ”bricks” linked-list,
and we do not initialize the ”bricks” until later. If not, we translate the vertices
depending on the current voxel’s coordinates within clip-space, so that the voxel is
an axis-aligned unit cube for ease of intersect detection.
If the triangle/cube intersection returns true, the triangle intersects the cube,
and we check whether the current ”map” index has been marked as intersected. We
perform this check with an imageAtomicCompSwap, attempting to substitute the place-
holder value with some different placeholder (1 in the case of the code above). If the
check is successful, the index has not been marked. We increment our atomic counter,
swap in the resulting number, and continue checking triangle/voxel intersections, if

6.2.2
Intersection test
The triangle/cube intersection itself is implemented according to the approach pre-
sented in [Voo92] and on their website with the official code1. The exact, individ-

ual calculations and checks can be seen in Appendix B. The functions pass a long
outcode between them which represents a list of bits indicating the results of the
individual tests. Every time a test is run, the resulting long outcode represents, in
bits, which test-part passed and which failed. Depending on these bits we can then
determine whether to disregard the voxel for intersection, or keep testing.
The approach takes advantage of the fact, that when working on a unit cube, the
most trivial tests - whether a vertex is outside a given surface plane - can be boiled
down to simple calculations and comparisons. For example, if we want to determine
whether a vertex is outside the edge plane that is formed over the edge above the
x-axis on a 45 degree angle with the x-y-plane (from corner (0.5, 0.5, 0.5) to corner
(0.5, 0.5, -0.5), we can simply check for X + Y > 1, where X and Y are the corre-
sponding coordinates for the input vertex.
After performing the trivial tests for planes, we can move on to the two inter-
esting tests: whether the triangle edges intersect the cube, and whether the triangle
intersects a cube diagonal. In the first test we use the results from the previous tests
to ensure that we only check edges where the connected vertices are not outside the
same faces. If no triangle edges intersect the cube, we perform the final test, as the
cube may be inside the triangle. This is done by finding the triangle plane, testing for
plane/line-intersection with the cube’s four diagonals, and finally checking whether
any intersection lies on the same side of all triangle lines.
6.2.3
”Bricks”
After performing the ”map”/triangle intersection tests, we recover the final atomic
counter value on the CPU side and initialize an uimageBuffer of size 4 · 4 · 4· atomic
value. The buffer has format GL_R32UI, as every index only needs to hold a pointer
to the start of the given ”brick’s” linked-list of intersecting triangle pointers. Along
with this buffer we reset a pre-initialized buffer for holding the linked list of triangle
pointers. As we cannot know how many triangle/brick intersections we will have, this
buffer needs to be initialized to some default value that we assume is large enough to
hold all intersections.
Additionally, we initialize another buffer with format GL_RGUI to hold a pointer
pair, one for the next index in that ”brick’s” linked list, and one for the index of
the intersecting primitive. Finally, we initialize a buffer to hold the primitive vertex
indices, so that for each primitive we can load the corresponding normals, world space

The ”brick” geometry shader works on the same principles as the ”map” geometry
shader, it simply also checks for individual ”brick”-voxel/triangle intersections, and
store the different values in the new buffers.
6.3
Traversal
This data structure can be traversed through a rasterization/ray tracing hybrid algo-
rithm, based on the voxel traversal algorithm presented in [Joh87]. For every fragment
closest in the scene, a ray is calculated as the direction from camera position to the
fragment, with origin in the camera position. One can then perform whichever direc-
tion change they wish (reflection, refraction, shadowray, etc.), and walk the ”map”
in that direction with the fragment position as origin.
The walk itself consists of finding the starting voxel from the fragment origin.
This ”map” index is then checked for triangle intersections. If any are found, we find
the entry point into the 43 ”brick” grid and walk this grid in the same fashion as
with the ”map”. For each ”brick” we encounter we perform intersection tests with all
triangles registered as intersecting that ”brick”. If intersections are found, we ensure
that the distance to the intersection point is within the ”brick” coordinates in space,
so as to not incorrectly mark an intersection as closest if it happens further along the
ray than where we currently are. When we exit a ”brick” group, we continue walking
through the ”map” and reiterate the process for each ”map” index we intersect, until
we choose to stop or exit the scene.
Since we store pointers to the triangles and their respective vertex data, we can
interpolate the values we need from the intersection points. This allows us to inter-
polate normals for further ray tracing if we intersect something, or to calculate color
attribution.

Advantages Of HCL
After implementing our algorithms in both the back-end and front-end of HCL, we

what data is on which processor, and how to work with that data. The freedom

Regarding back-end, the HCL framework is set up for further extensions, as proven

7.4
Limitations Of HCL
As of this project, the back-end of HCL is stronger than the front-end. Except for
the rasterization implementation, the framework is quite flexible, as mentioned. The
rasterization pipeline, while having certain template classes which invite further ex-
tensions, is rather hard coded, and the final code of this project is unnecessarily
cluttered due to the original rasterizer setup. Furthermore, the rasterization shaders
had to be somewhat hardcoded to pass the necessary values for the extended fragment
linked-list to work. This was due to the framework not properly handling pointers to
the output streambuffer, which meant that the shaders could not load the data types
and name fields of the output buffer.
Additionally, while the framework is flexible regarding further development, it
does produce a small amount of overhead when performing calls, as the calls have
to be handled by the pipeline and hclContext class, before performing the standard
OpenGL operations. At the moment, this overhead is negligible when only running
rasterization and fragment shading. However, this overhead naturally increases with
every consecutive operation, and must be kept in mind when expanding the frame-
work.
Finally, from a front-end perspective, the user is limited to streambuffers and uni-
forms when performing operations, and as mentioned, while the rasterization pipeline
seemingly invites the user to give different types of input and output streambuffers,
the pipeline is only able to handle the input and expects the output buffer to be in

shader calls directly alongside standard CPU processes, this effectively means that the
user must switch constantly between C++/OpenGL and GLSL context, and know
the different paradigms. As a final note, the user is unable to divide the compute
dispatch work groups into smaller groups of threads should they wish to: they can
only operate with one local invocation per work groups.

CHAPTER 8
Discussion
In this section we discuss our implementations of FXAA and SSAO. We then present
possible uses for our implemented data-structure. We lastly mention some workarounds
necessary due to the nature of HCL.
8.1
Algorithms
Our implemented algorithms are fairly straight forward and by-the-book, except for
using streambuffers instead of textures. The fact that they operate on streambuffers
means that more conversions between 1D indexes and 2D indexes were necessary than
normal, and it is very possible that versions of FXAA and SSAO exist which work
better on buffers. Likewise, it is highly likely that our versions could be improved
upon, particularly SSAO, which runs at approximately 48 ms. per call.
The extension of the framework with a post-processing stage was done primarily
to showcase that it could be done, and is also in need of improvements. Most im-
portantly, the implementations in this project make assumptions regarding the data
types of the input and output buffers. Ideally, the post processing stage should be
capable of taking certain parameters indicating how the desired process should be
used, such as what data types the inputs and outputs are in, and how to load the
from the buffers. As the shader programs of HCL currently work with stringstreams,
these parameters could be passed as strings or char pointers.
Other types of anti-aliasing and screen space ambient occlusion based techniques
could also have been implemented to show the different possibilities for the front-end
developer. For example, Horizon Based Ambient Occlusion could be implemented
using the linked fragment list as an indicator of the depth of fragments in a pixel
from different angles.
It could also be interesting to see how well HCL handles
the downscaling happening in Supersampled Anti-Aliasing and Multi-sampled Anti-
Aliasing against our implementation of FXAA.
==PAGE:60==
50
8 Discussion
8.2
Extension With Ray-Tracing
As mentioned in Chapter 6, our implemented data structure is more a concept im-
plementation than a workable implementation. Initially, the idea was to implement
a rasterization/ray-tracing hybrid technique to expand the framework with simple
ray-tracing. However, due to time constraints, this feature was dropped.
While data from atomic counters and buffers show that the data structure reg-
isters and stores primitives and intersections within realistic boundaries (it registers
the correct total amount of primitives for primitive storage, as well as registered an
expected area of the map as intersected), a visualization of this data was not possible
within the time frame of this project.
Ideally, this data structure can give an idea of the possibilities regarding extending
the HCL framework with new functionalities.
8.3
Workarounds
Most notably, the built-in object loader functionality, which takes an .obj file as input
and stores the relevant data in lists, does not appear to function correctly. It only ap-
pears able to handle files where normals and texture coordinates are in the exact same
order as the vertices, as it simply stores the input normals and other coordinates in
the same order as it receives the vertex data. As such, a custom .obj-loader function
was implemented in the sample code, which sorts normals depending on the order
given in the file, so that they are read in the correct order by the HCL rasterizer.
Additionally, since the framework has trouble reading certain values from its cus-
tom data types - most importantly, from the output buffer of the rasterization process
- some functionalities of the fragment list generation had to be hardcoded in in order
to be included in the output buffer. This is a general area of possible future work
with HCL: allowing the user a greater degree of control over what gets passed to the
different processes, so as to better tailor the processes to their specific needs.

CHAPTER 9
Conclusion
We have designed and implemented adapted versions of Fast Approximate Anti-
Aliasing and Screen Space Ambient Occlusion within the HCL framework, both as a
front-end user of the framework and as a back-end extension. The implementations
produce acceptable results graphically, and while FXAA runs fast, the SSAO imple-
mentation leaves something to be desired regarding performance.
A concept implementation of a data structure for future extension with ray-tracing
was designed and implemented, but not thoroughly tested. Ideally this data structure
can be used for a ray-tracing/rasterization hybrid, or regular ray-tracing, with some
adjustments.
The implementations in the front-end of HCL show the flexibility of HCL as a
heterogeneous computing library with a focus on hiding traditional memory manage-
ment and allocation, letting the user focus on pure development on CPU and GPU
seamlessly. The implementations were possible without needing to extend the back-
end functionality of HCL, but were integrated as well to examine the possibilities
regarding back-end development.
While the front-end of HCL presents a very fo-
cused framework which lets it’s user focus on handling shading without the hassle of
OpenGL, it also has limits. Specifically, it is limited to working with Shader Storage
Buffer Objects (streambuffers) for data inputs, which eliminates possibilities for tex-
ture based shading and operations.
The back-end of HCL shows much promise, having been developed to generally
be very flexible regarding extensions. A post-processing stage and additional shaders
(geometry shaders and a post-processing compute shader) was added with relative
ease and integrated into the main hclContext to run alongside existing stages. The
rasterization stage was extended to be able to handle rasterizing for a specialized
data structure, as well as initialize different geometry shaders and fragment shaders
depending on in input enumerator value.

ways in which HCL can be extended as a framework with additional stages and data
structures.
