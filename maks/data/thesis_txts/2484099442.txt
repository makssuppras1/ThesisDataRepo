CHAPTER 1
Introduction
First of what is a mobile robot:
A mobile robot is a machine with the capability to move around in their environment
and are not fixed to one physical location. Mobile robots can be autonomous which
means they are capable of navigating in an uncontrolled environment without the need
for a operator input. They are generally controlled by software and uses sensors and
navigation devices to identify their location and surroundings.
Research in the field of mobile robots has steady advanced the last decade, studies cov-
ering problems such as autonomous navigation, obstacle detection/avoidance, path plan-
ning and mapping have been deeply researched and integrated in mobile robts. Together
with advances in electronics areas such as sensor miniaturisation and computational
power, this means robots have become smaller and is able to run more computation-
intensive tasks thereby bringing the cost down in development for researchers and the
industry. Autonomous mobile robots are emerging as work force in the industry and
with it a need to use a variety of sensors to accomplish more tasks with added difficulty.
This can be seen in the farming and car industry, where tool carriers for the fields1 and
transportation vehicles 2 complete driving operations autonomously/ semi-autonomously
meaning without a driver/operator input, thus they have to use more sensors and com-
putational power to accommodate the missing input from an operator. Consumer robots
have also been emerging in the form of lawn mower, vacuum cleaner and personal robot
kits have been introduced in the form of turtlebot3 and LEGO mindstorm. 4
The rest of this chapter describes the context and motivation for this study, its objective
and outline.
1.1
Context and motivation
Developing robust and general-purpose software for mobile robots can be a tedious and
hard task to accomplish, because mobile robot systems are often complex distributed
systems that interact between various different components such as hardware, operating
systems, communications protocols and programming languages.

2.4
Mobotware
Mobotware is a plug-in based software platform and the software has been used for edu-
cation and research in mobile robotics in the last decade. Important design criteria have
been real-time performance of the control level, easy integration of sensors, fast porting
to new robots and core system stability and maintainability in an undisciplined pro-
gramming environment. Real-time performance is assured by using RTAI-Linux kernel
1, core stability is obtained by using plug-ins for user developed modules. The plug-in
based module structure combined with inter module communication based on TCP/IP
sockets and human readable XML-protocol makes it easy to use the system on a wide
range of hardware platforms and configurations.2
Figure 2.3: Overview of MobotWare framework.

2.6
MRC
The MRC uses RHD to interface with hardware and provides basic low level real-time
control for the SMR and have build in core features such as: Odometry, Small Mobile
Robot Control Language(SMR-CL) interpreter, XML-based socket interface to sensor
servers, Calibration support for odometry, light-sensors and distance sensors.
2.6.1
SMR-CL
SMR-CL is a interpreter language that is developed at DTU to do simple control of
mobile robots such as SMR. The language supports two types of input execution, a
mission script file or direct script though a socket interface prompt. The interpreter
takes actions commands with multiple stop criterion in a line by line execution structure
and supports standard mathematical expressions. The basic format for the interpreter
is composed of five elements:
command
parameters
[@v velocity]
[@a acceleration]
[:(condition)]
The [ ] means optional for the action command. A more detailed description of the
elements:
• command: is the desired command within the interpreter categories motion control,
flow control, variables, logging of date, miscellaneous and socket interface.
• parameters: can be numerals given in SI prefixes, variable or a string required by
the command.
• velocity: is the desired max velocity in m/s with specifier @v before numeral, if
not entered the last used velocity will be used, in case of first time use and the
default is 0.3 m/s.
• acceleration: is the desired max acceleration in m/s2 with specifier @a before
numeral if not entered the last used acceleration will be used in case of first time
use and the default is 0 m/s2 .
• condition: is a list of stop condition expressions that will continue current com-
mand until a expression is true or the command has finished its action.
Uses
specifier : before expression and for multiple expressions using a | between each
expression. :(expression)|(expression)
The motion controller(MRC) provides the basic path control based on odometry as well
as sensor based input for movements, SMR-CL commands is as following:

2.8
ROS
ROS as mentioned earlier provides the services seen in ordinary robotic operating sys-
tem this includes hardware abstraction, low-level device control, implementation of
commonly-used functionality, message-passing between processes, plug and play for mul-
tiple devices and package management.
In general ROS’ architecture is graph based and consist of nodes, topics, Messages
and ROS Master, which will be investigated further below.
2.8.1
Nodes
A node can be thought of as a independent software module that preforms an action
or computation. Nodes are combined together in a graph architecture as seen in Figure
2.4 and communicates with another nodes though topics. Nodes that needs data can
subscribe to the relevant topic, likewise if a node have data to share it can publish this
on a topic. Note that a topic has no limits for the amount of subscribers it can have.
2.8.2
Topics
Topics are message carriers for nodes, each topic is defined by the ROS message type used
to publish it, nodes can only receive messages if the subscribing node have a matching
message type. There can be multiple publishers and subscribers to a topic.
==PAGE:21==

2.8.3
Messages
Messages are a simple data structure comprised of standard types such as integer, float,
string, arrays and variation of before mentioned types. An example can be seen in Figure
2.5, where an IR sensor is publishing its measured distance to the topic IR_distance and
message type is a float32 called range. Another node Distance_to_object is subscribing
to the topic to find out how close the nearest object is, thus having the same message
type.
Figure 2.5: IR sensor publishing its message to topic IR_distance and another node
subscribing to receive the message.
2.8.4
ROS Master
The ROS master is a subroutine of roscore, which is a collection of nodes and programs
that are pre-requisites for the ROS system to work, without roscore ROS cannot run
anything. The ROS Master establishes communication between the nodes and facilitates
naming and registration services to the rest of the nodes in the ROS system. The role
of the Master is to enable individual ROS nodes to locate one another in a distributed
system. Once these nodes have located each other they communicate with each other
peer-to-peer.
2.8.5
Gazebo simulator
Why is a simulator needed? Having a real robot often implies the need to have laboratory
space, renew batteries or tires and other hardware quirks.
In a real case scenarios,
even the best robots break periodically due to various combinations of operator errors,

environmental conditions, defects in manufacturing or design. These type of problems
can be mitigated using simulated robots that move in a simulated environment.
Simulated robots can be quit useful, in simulation we can model reality as high
or low as it is desired.
Sensors and actuators can be modelled as ideal devices or
have various levels of distortion or\and errors.
The features a simulated robot and
environment creates is the perfect opportunity for low-cost test platforms.
Gazebo
is a 3D simulator that provides standard robots, sensors, modules such car or chairs,
environment models for 3D simulation required to robot development and offers a more
or less realistic simulation with its physics engine. Gazebo is a popular simulators for
open source robotics and has been widely used in the field of robotics because of its
high performance and reliability. Gazebo uses OGRE (Open-source Graphics Rendering
Engines) for rendering the 3D Graphics used in its GUI, sensor libraries, gaming and
for light, that can be realistically drawn on the screen.
A great deal of sensors are
already supported such as: Laser scanner, 2D camera, depth camera, IMU and force-
torque sensor. Sensor noise can also be added to the sensor data like in real environment
scenarios. A range of robot models comes preloaded with gazebo: PR2, Pioneer2 DX,
iRobot Create, and TurtleBot are supported in the form of SDF file, a Gazebo model
file type, and users can add their own robots with an SDF file.

2.8.7
URDF
ROS uses the Universal Robotic Description Format (URDF), to describe all elements
of a robot in XML format. Simply put the urdf describes all the things about the robot
that don’t change, such as the size of the base footprint frame, wheels frames, robot
arms and sensors. They can be used in both real and simulated robots, but to use a
URDF file in Gazebo requires additional simulation-specific tags which must be added
to work properly with Gazebo such as controllers.
2.9
Drawbacks of using ROS
One of the drawback of using ROS is that real-time control is not supported, it is possible
to get performance that seems like real-time, but it cannot not be guaranteed for critical
operation.

2.10
Navigation
Navigation have always been a challenging task for mobile robot and can be loosely
described as the ability for a mobile robot to plan a route and execute it without operator
input. This description indicated two different goal strategies, the robot knowing how
to move towards a target and having the ability to avoid any obstacles along its way.
Achieving these goals are not easy and for success in navigation four building blocks
must be in place:
• Perception - the robot must have sensors to extrapolate meaningful information
• Localization - positioning in the environment must be determined
• Cognition - the robot must decide how to act to achieve its goals
• Motion control - A desired trajectory must be achieve from its motor output
In map-based navigation, the robot localises by collecting sensor data, then updating
its estimation about its position with respect to the given map of the environment. A
map can either be created by an operator or the robot itself, a useful method for auto-
matic map building is SLAM (Simultaneous Localization and Mapping) that explores
the unknown space and detects its surroundings, estimating its current location as well
as creating a map. The benefits of this method is high accuracy in localization and
position of the landmarks(Doors, walls) and the map can be read by humans and robots.
The goal of path planning is to find a path in the physical configuration space from the
initial position of the robot to the goal position, thus avoiding all known collisions with
the obstacles. Path planning that’s mentioning here can also be thought as a global
planner that turns the environmental model into a discrete map suitable for the path-
planning algorithm. These planning algorithms are usually based on a graph search or
Potential field planning algorithm. Lastly motion control is determined from the robots
local planning strategies(obstacle avoidance algorithm) where its focuses on changing
the robot’s planned trajectory as informed by its sensors during robot motion.

2.10.1
Local Motion Control
The planning strategy mentioned earlier(global path planning) is designed to calculate a
collision-free trajectory for the robot in the case the environment in partially or totally
know. When the robot finds it self in changed area and does not have information about
the surrounding area, these algorithms fail and do not produce any usesable solution.
For this kind of situations, the local motion planning is needed.
2.10.2
Obstacle Avoidance
Local obstacle avoidance is the task of satisfying some control objective subject to non-
intersection or non-collision position constraints and thereby changing the robot’s tra-
jectory as informed by its sensors during robot motion. Normally obstacle avoidance
is considered to be distinct from path planning in that one is usually implemented as
a reactive control law, while the other involves the pre-computation of an obstacle-free
path.
2.10.2.1
Dynamic Window Approach
In the local dynamic window approach, the robots kinematics is taken into account by
searching within a window of velocity space. This velocity space is all possible sets of
elements( v , ω ) where v is the velocity and ω is the angular velocity.
Given the current robot speed and each sample element, the algorithm first selects
a dynamic window of all elements that can be applied within some short period of
time. Each trajectory is evaluated to reduce the dynamic window by keeping only those
element that ensure that the vehicle can come to a stop before hitting an obstacle. The
remaining subset of velocities are called admissible velocities. Direction of the motion
controller is then chosen by applying an objective function to all the subset velocity’s in
the dynamic window, the highest scoring trajectory will be chosen together associated
velocity and send to motion controller. The objective function weights high towards fast
forward motion, maintenance of large distances to obstacles and alignment to the goal
heading. The objective function F has the form:
F = A1 · Goal(v,ω) + A2 · velocity(v,ω) + A3 · Obst(v,ω)
(2.1)
Goal = Distance to local goal position.
Velocity = Reachable forward velocity on that point by the robot.
Obst = Distance to the closest obstacle in the trajectory.
The parameters A1, A2 and A3 tunes the behaviour of the robot.

Sensor calibration
For autonomous mobile robots an important aspect is to acquire knowledge about its
environment and internal state. Taking measurements from various sensors and extrap-
olate meaningful information about the in environment is key, since this enables the
mobile robot to move around according to unforeseen obstacles and solve task without
damaging it self or the environment.
Sensors used in a robotic system can typically be classified by two parameters: pro-
prioceptive/exteroceptive and passive/active. Proprioceptive sensors measure values in-
ternal to the system, some few exampels: wheel encoders, temperature of components,
robot arm joint angles and toque, and battery voltage. Exteroceptive sensors acquire
information from the robot’s environment, some few examples: distance measurements,
Laser rangefinder, and reflectively sensors. These measurements will usually be the onces
to be interpreted by the robot to give meaningful environment information. Passive sen-
sors measures energy coming from the environment to the sensor and measurements
are therefore heavily influenced by the environment. Active sensors emits their proper
energy and measures the reaction. This gives a more controlled interactions with the
environment and often leads to better measurement performance.

Sensors are typically not perfect, meaning that their actual measurement is not always
the same as ideal measurement thus leading to inaccuracy measurement. The common
cases for inaccuracy: Variation in two sensors produced from the same manufacturer
may have a slightly different response. Difference in sensor design between two sensors
can cause a difference in response as well. Other environmental factors such as temper-
ature(heat and cold), vibrations/shock and humidity together with age may lead the
sensor to drift over time.

4.1
Odometry
The nominal values for wheel base and wheel radius are often inaccurate. This leads the
odometry to drift over time and therefore increasingly become unreliable without any
correction. The calibration aims is to estimate the deviations from the nominal values
for these parameters r1,2 and wb, effectively calculating suitable estimates,thus getting
higher accuracy.
The standard method for odometry calibration comes from the J. Borenstein and
L. Feng [FBE94] and is also the method implemented on MobotWare and is grate for
short-term accuracy. The method revolves around correction of systematic errors in
Bi-directional Square Path experiment, where two error types is investigated. Type A
is defined as error that reduces or increases the total amount of rotation in the square
path experiment for both cases of clock wise(cw) and counter clock wise(ccw), this error
mostly is caused by wheel base (Eb). Type B is defined as error that reduces the total
amount of rotation in one direction but increases in the other direction of the square
path experiment and is mostly caused by ratio error between the wheel diameters(Ed).
The procedure for the square path experiment is rather simple, the robot has to go
cw and ccw in a square pattern with a side length Lside and rotation of either ± 90
degrees, where the endpoint is measured in (Xcw,Ycw) and (Xccw,Yccw) coordinates with
respect to the starting points coordinate system. From this point the error correction

SMR-CL.
Small  Mobile  Robot  Control  Language, SMR-CL is  a  simple  interpreted  language 
intended for control of mobile robots. The language supports sequences of basic robot 
actions with multiple stop criterion as well as standard mathematical expressions. The 
system has two kinds of variables system variables that reflects the state of the vehicle 
and user variables that are created the first time they are used in an assignment clause. 
The language is intended to be used in the tactical layer just over the control layer i.e. it is 
fast enough to shift between control strategies and react to state changes in real time. The 
hope is that it will be able to bridge the gab between the hard real-time demands of the 
control layer and the soft realtime demands of the planning (strategic) layer.  The 
language may be used in two ways either as scripts run directly from text files or  as a 
command language through a socket interface. Most commands are available in both 
modes but a few are only available in one of the modes.
Basic format.
The language is line oriented i.e. it is interpreted line by line. The basic format is
command   parameters  [@v velocity] [@a acceleration] [: controlconditions] 
where [ ] means optional.
command
is the desired command
parameters 
is a number of parameters required by the command. The parameters can 
be numbers, variables or string litterals (characters within "  " )
velocity
desired vehicle velocity in m/s (variable or number)
acceleration
desired acceleration in m/s2  (variable or number)
controlconditions
( expression ) [ | (expression)]*
                       a list of ORed stopconditions, the interpreter will execute the next 
command  if one or more of the conditions are true.
                        The system variable $condition will hold the number of the first true 
condition (from left)
Examples:
fwd  2.0 @v0.5 @a0.05                          
move forward 2 m with acceleration 0.05 m/s2 max velocity 0.5 m/s
drive   @v0.5   :($drivendist > 4)| ($cmdtime > 12)
drive at velocity 0.5 m/s until 4 m is driven or time used is 12 seconds
turn 90
      
turn 90 degress around center
