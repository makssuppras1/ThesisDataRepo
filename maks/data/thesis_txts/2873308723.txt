Prologue
1.1
Abstract
In recent decades, compulsive use of screens have been on a rise. This project aims to examine
the problem, innovate a solution, and make considerable progress towards its development. We
explore the problem of compulsive and problematic technology use and its effects on individuals
through existing literature and research on the topic. We map some of the current solutions to
problematic technology use, in particular application-based solutions. Blue Ocean Strategy is
used to analyze some of the companies in the current market and propose a novel alternative to
existing solutions, which we call the product vision.
We describe how a company was founded with the goal of realizing the product vision. We
specify the requirements needed to build a Minimum Viable Product, and formulate a detailed
plan for the execution of the project using agile methodology. We describe the choices and
considerations we have with regards to commercialization of the solution. Through a product
oriented methodology, we go over the actions taken towards building the product vision, and
explain the technical structure in detail. We document the architecture of the current imple-
mentation. The solution we made, which exceeds the requirements we originally defined for the
Minimum Viable Product, is presented at its current stage. Then a discussion is provided. It
details the choices made during the project and the future perspectives. Finally, we conclude
on the project.

1.3
Introduction
This thesis is the result of a growing frustration with how difficult it can be to stay focused
while working or studying.
As students, we have experienced firsthand how much time is
unintentionally spent on mobile devices, even when we know it is unproductive. The project
began from that personal experience and the idea that we might be able to build something
that could help not just ourselves, but others facing the same challenges.
The project takes place within the domain of problematic technology habits, with a focus on
digital habit changes. It aims to explore whether software can support healthier usage patterns,
without requiring users to rely purely on willpower. This exploration is mainly through the
development of a proposed solution. The goal is to not just build something that technically
works, but to explore what it takes to make such a system useful in practice.
The thesis follows a practical, product oriented methodology, with the main objective being
to build a working solution and gather early usage insights. While the primary focus is on
development, considerations about the business model supporting the product is also presented.
The business sections follows a more theoretical approach, aiming to evaluate feasibility and
potential.
Throughout the thesis, we will be referring to both sections in the appendix, external sources
and supplementary files that has been uploaded alongside the thesis. We have collected and
explained each of the supplementary files in the appendix 10.16 Supplementary Files.
The following chapter outlines the methodology, which defines the structure of the thesis and
how each part is connected.

Methodology
This project concerns problems arising from excessive technology use. The project aims to de-
velop an adequate solution.
In Chapter 3, we examine the problem of excessive technology use on a societal and individual
level. We argue for the existence, prevalence and severity of this problem, referencing existing
academic knowledge and using deductive reasoning.
In Chapter 4, we consider possible solutions to the problem presented in Chapter 3. We argue
that the current widespread solutions are insufficient by referring to existing literature on the
subject. A software-based solution is proposed. Using Blue Ocean Strategy [1], we analyze the
existing value offers in the related market. The Remove-Reduce-Raise-Create matrix from Blue
Ocean Strategy is used to create a new, currently non-existing value offer as a solution to the
problem from Chapter 3, serving as a long term product vision.
In Chapter 5, we scope the project using the product vision found in Chapter 4. We define the
MVP. We provide specific system requirements and non-functional requirements.
In Chapter 6, we explain how the project was structured and executed according to the require-
ments defined in Chapter 5. This process was supported by a company which we founded to
facilitate the completion of the product vision as defined in Chapter 4.
In Chapter 7, the product development process was documented according to computer sci-
ence best practices and software industry standards. It was guided by the requirements defined
in Chapter 5 and assisted by the process planning and workflow structure described in Chapter 6.
In Chapter 8, we show the state of the latest iteration of the product. A strategy for continuing
the progression towards completion of the product vision is presented.
A discussion and conclusion is provided at the end of the report.

The Problem
3.1
Introduction
Whether or not compulsory behaviors should be treated as addiction is disputed [2]. Some sci-
entists prefer not using the word ”addiction” about technology use as there exists some debate
about the definition of addiction. Some authors argue that the term addiction ”pathologizes
everyday human behavior” [3].
One could argue against that view by stating that what is
considered everyday human behavior changes over time, and that a particular behavior is not
necessarily healthy by virtue of it being normal. Addiction is a powerful word, and technology
habits that might not pass the threshold for addiction can still be problematic and deserving of
a solution. ”Problematic technology habits” will be used to describe any type of involuntary,
irrational or compulsive use of technology in this report, to include nuances of damaging and
irrational behavior happening below a pathologized threshold.
In this chapter, various arguments will be presented to support that problematic technology
use is a serious public health problem. An argument for why addiction as a business model is
prevalent in today’s world will be presented. We will then introduce the concept of superstimuli
as a possible way to explain problematic technology use. In section 3.4, data on how widespread
the problem is and its consequences for individuals is covered.
Finally, in section 3.5, arguments will be presented for why the current popular discourse and
view on these problems is insufficient to solve them. The usage of the product built later will
depend on the public perspective, so the solution should take that into account.
3.2
Addictive Products as a Business Model
For a business, slowing the loss of customers is an effective strategy. If a product has addictive
qualities, those alone may prompt customers to buy it repeatedly.
The potential success of addictive products solely by virtue of them being addictive, could
provide a plausible explanation for why several significant industries uses them [4] [5].
Classic liberal economic theory suggests that competition benefits the customer in a multitude
of ways [6]. The absolute nature of this claim has been disputed in recent years [7]. In the
paper ”Competition and Consumer Protection: A Behavioral Economics Account” [8], it is
suggested that ”the benefit (to consumers) of competition does not necessarily extend to a world
with imperfectly rational consumers who systematically misperceive the benefits and costs of
products”. In the book ”The Law and Economics of Irrational Behavior” [9] in chapter 5.2, it
is discussed whether or not addiction is a rational choice, and it is concluded that addiction
is at least partly irrational. If addiction is at least part irrational, and that the prevalence of
imperfectly rational consumers in a market suggests that the competition in said market is not
necessarily beneficial to consumers, it suggests that addictive business models presents a way
for companies to operate successfully without necessarily being of benefit to customers. It seems
plausible, that said companies may be competing on the addictiveness of their products, rather

How Technology Companies Engineer Addiction
In a series of famous experiments, the Dutch ecologist, Niko Tinbergen in his book ’The Herring
Gull’s World: A Study of the Social Behavior of Birds’ [10], found that across many species of
animals, greatly exagerrated sources of natural stimuli was preferred. The most famous example
is with the species herring gull, which focus on hatching the largest and greenest eggs in its
nest, as these are perceived the most genetically promising. Tinbergen showed that artificially
exaggerating the size and color to a degree never found in nature would make the birds leave

phenomenon would never occur in the wild, the evolution of the herring gull did not have to
select for any form of resistance against such clever manipulation.
In ’Vanedyr’ by Nicklas Brendborg [11], Brendborg explains how companies use what he calls
superstimuli to engineer addiction. He argues that technology companies do this by appealing
to our needs for information, feeling of progress or sexual needs, depending on the industry
in question. Sensationalist news can be said to be particularly stimulating [12]. It is hard for
humans to resist temptations like these, as they are specifically engineered to exploit irrational
biases and psychological weaknesses we naturally posses [11].
3.4
The Prevalence and Consequences of the Problem
According to an article from Simon Kemp from DataReportal [13], global adoption of the inter-
net, social media, gaming and adult content has been on the rise for two decades. Average time
of all internet users globally on social media (including those who do not use it at all) is by the
end of 2024 2 hours and 21 minutes, with females spending significantly longer than males [13].
Smartphone usage is habit forming and people consistently underestimates their screen time [14].
Screen time may have different consequences dependent on the type.
A study [15] on the
relationship between self-reported screen time, actual screen time, self-reported stress, and actual
stress, found several interesting insights, the most relevant summarized below:
• Screentime for entertainment was statistically significantly correlated with stress.
• A possible link between self-reported addiction and actual stress was found.
• Individuals using screens for entertainment were higher in perceptual stress, while individ-
uals using screens mostly for work had lower perceptual stress and higher life-satisfaction.
• Self-reported addiction was correlated with higher screen time scores, suggesting that
individuals’ assessment of their screen time was quite accurate.
• Dependence on social media and gaming was in particular correlated with self-reported
screen addiction.
While this was a study of correlations and therefore not conclusive to show that screen time for
entertainment is the direct cause of higher levels of stress, it does highlight interesting nuances
of screen usage. There is an opportunity cost associated with spending many hours on a screen
each day for entertainment - the time must come from other activities, that one would have
done instead. This opportunity cost could explain why social media use causes a reduction of
in-person activities, as well as subjective well being [16].
Screen entertainment has potentially negative effects on the brain. Research finds that smart-
phone use in children causes a higher risk of developing Attention-Deficit Hyperactive Disor-
der [17] [18]. Results from an experiment done in the study [19] indicates that nomophobia, a
specific way to measure smartphone addiction, has a persistent negative correlation with fluid
intelligence whether or not the phone is present. This suggests that it is not merely phone
presence, but long-term effects on the brain, which causes the lowered fluid intelligence [19].
With the modern knowledge that the brain is plastic, the idea that consuming several hours of
fast-paced and highly stimulating content daily could reduce the ability to stay focused seems
plausible.

4.3
Blue Ocean Strategy - Theory
For the development of our idea, we have used the methods in the book ”Blue Ocean Strategy” [1]
aiming to make direct and expensive competition more or less irrelevant by targeting new market
segments with a new product.
The core idea of the book is to use a tool called the ”Strategy Canvas” to map out the value-
offer of either an industry or a single business. A value-offer, as denoted by the author, is the
collection of important aspects of a given product or service. Frequently, the authors argue,
businesses in an industry end up benchmarking themselves against one another, using the same
Key Performance Indicators. This leads to harsh and expensive competition and falling profits
as competition for the same customers intensifies.
The goal is to diversify from existing companies to access a different market without competition.
The process of doing that requires four modifications to the existing value-offer:
• Remove
• Reduce
• Raise
• Create
By removing aspects of a value-offer which a subset of customers does not appreciate or directly
dislike, the product is simplified and its cost is brought down. By reducing aspects that the
industry consider benchmarks, but customers are not especially attached to, further simplicity
and cost reduction can be achieved. By raising existing aspects which customers appreciate, the
product is improved. Finally, after creating entirely new aspects, a new customer base can be
attracted, and a new value-offer is created.
Several strategies for choosing the right aspects of a value-offer to scale up, down, remove or
add, is proposed. One way is by combining the offer of two or more existing companies or
industries. Another is by looking at why different groups of non-customers have chosen not to
be customers. The latter is particularly useful as industries frequently attempt to compete on
certain metrics which the customers are then ”trained” to care about - thus, non-customers are
frequently more precise at suggesting improvements to products than existing customers.

6.7
Key Performance Indicators
It is important for a business to know and measure its Key Performance Indicators (KPI’s). We
have taken inspiration from other subscription B2C businesses to understand which metrics are
important.
• Churn - How many customers in % are lost on a monthly basis. The most important
KPI for our business, but also one that is hard to measure since it requires acquiring a
significant number of customers and then measure the rate at which they stop using the
product.
• CLV - Customer Lifetime Value - Calculated by multiplying the subscription payment of
a customer with the amount of time they stay in the business. It is necessary to know the
churn to accurately estimate CLV.
• CAC - Customer Acquisition Cost - This KPI is usually tied to a specific marketing
channel. It describes how much money the business must spend on the channel to gain
one customer. This is difficult for us to measure directly as most of the marketing strategies
we mentioned are organic, and it is less clear how many customers come from each channel
and how expensive the channels are to run. A way to do the calculation instead, could be
to use time spent instead of money, to measure efficiency of organic marketing strategies.
• Conversion Rate - How many users go from being users to being customers in %. Maxi-
mizing the conversion rate is important to monetize the user base more efficiently.
• Free User Churn - How many free users are lost per month. Less important than the other
metrics, but the chance of a user converting to a customer increases if they do not stop
using the product all together.
These are the most important KPI’s. A lot of data is needed to estimate them correctly, and
for some of them, like churn, some time is needed as well. Early tracking of the changes in
the KPI’s before the long-term data is obtained speeds up the process of iteratively building,
tracking, learning, ideating and building again. For example, higher engagement with premium-
only features could suggest the customer is gaining value from the premium subscription, and
may be less likely to leave.

6.11
Development Guidelines & Principles
To ensure code quality and maintainability, the development process focused on several principles
and best practices.
The main branch was kept deployable at all times, with all changes developed on feature branches
and merged via pull requests after testing, see section 7.10 Version Control & CI/CD for further
details.
We followed the principle of separation of concerns, structuring the codebase into well defined
modules. This is also referred to as Hexagonal Architecture [30]. It was done to simplify testing
and extensibility.
Security best practices were followed.
Secrets like passwords and tokens were managed via
environment variables or Azure Key Vault, see section 7.5.1.6 Key Vault - axion-cloud-vault for
further details. Sensitive data was never stored in code.
The overall project structure followed established software engineering principles, particularly
SOLID and DRY. SOLID is a set of five guidelines that improve the maintainability and flex-
ibility of codebases. While these principles are especially valuable in larger projects, research
shows that they should be applied thoughtfully and can be adapted when needed to balance
performance and complexity [31]. The following outlines each principle based on this research.
S – Single Responsibility
Each class or module should have one clearly defined responsi-
bility. This is the sane as the principle of separation of concerns.
O – Open/Closed Principle
Software components should be open for extension but closed
for modification. Existing code remains stable while new functionality is added via extensions,
rather than modifications.
L – Liskov Substitution
This principle states that, when building classes in a derived man-
ner, the child class should behave the same as the parent class, without modifications. The child
class should always extend the functionality, to minimize confusion and keep the integrity of the
functionality, for any classes that has been derived from the parent class.
I – Interface Segregation
Interfaces should be specific to individual needs. It is better to
use multiple small, but more focused interfaces than a single, general purpose one.
D – Dependency Inversion
High-level modules should depend on abstractions, not funda-
mental implementations. This decouples the system and enables easier swapping of components.
This is especially useful, when using third-party APIs or libraries.
Using the SOLID principles supports the building of a flexible and maintainable system design.
The DRY (Don’t Repeat Yourself) principle was applied to avoid code duplication and instead
reuse logic. These combined efforts contributed to a scalable and clean codebase.

7.3.1
MVVM and Jetpack Compose
The UI toolkit we used is called Jetpack Compose. It is a declarative framework [34], and works
by ”composing” the individual UI functions, called ”composables”. Jetpack Compose only needs
to update the elements which has been changed - unlike older toolkits, it does not recreate the
entire screen every time data changes.
It is good practice to store the application data properly, so that each variable only exists in one
place. A repository is a class that functions as the true source of data. Letting the UI elements
call the repository directly is not ideal, as it does not allow for temporary changes in data. For
example, if the user wants to edit an object and they make changes, they cannot click cancel
and delete their changes, as the data in the repository has already been changed.
To handle temporary data, we use an MVVM, Model-View-Viewmodel, architecture. In the
MVVM architecture, the viewmodel is a class that communicates with the UI elements and the
repository. If the user clicks a button to change data, the button composable calls the viewmodel
to change its data.
If the change needs to be saved, the viewmodel calls the repository to transfer the changes. This
separates concerns, and provides a flexible architecture with which updating the application
will be unlikely to conflict with existing data structures. On figure 4, the manner in which data
flows down and events flow up through the data hierarchy can be seen.

7.3.5
Advanced Blocking
In the 2nd and 3rd iterations of the application, upon turning on a block, the user was faced
with a pop-up box with three choices; cancel, lock without picking a time and pick a time.
If the user was to pick a time, they could not disable the block before the timer expired. To
ensure that the user did not block an application or website for an extensive amount of time by
accident, the maximum duration was limited to 14 days.
This timer-based approach aimed to introduce commitment, a core feature of the application.
One of the major problems with the operating system native solutions for managing screen time
(for example Apple Screen Time) is they do not allow the user to set an unbreakable rule.
However, time-locking is just one method. Other applications have had success with allowing
the users to set a short delay before opening an application. Freedom and Opal offer weekly
scheduling of blocks, such that a user could block certain applications while at work or studying.
In comparison, our initial time-lock implementation seems too simplistic. Users should be able
to tailor their blocking strategy based on their needs.
We had a brainstorm, where we mapped all the ways of blocking we could imagine. Examples
would be by time, with password, with a delay to open, with a small challenge, with restrictions
on the amount of opens per day, etc. We then tried to group these ideas into categories, as not
being able to edit a block for a certain time is not the same as the block being enforced.
The result of the innovation exercise was a three-variable system: Locking, Blocking and Access.
Blocks have a variable for each of the three variables, and the user must select one value for
each variable:
• Locking: When can the block be edited?
• Blocking: When is the block enforced?
• Access: How is the block enforced?

• None - The block may be edited or turned off anytime

• When Active - The block is locked only when it is active according to the blocking schedule.

the library Jsoup extracts the category based on keywords like “social” or “games.” If still
unidentified, a keyword match on the application label with the domain lists is attempted.

7.4
System Architecture - Backend
The backend operates as the core layer between the application and the database and other ex-
ternal systems. It is responsible for serving HTTP requests, managing authentication, handling
business logic and ensuring that data consistency is preserved across devices. While currently
implemented as a monolithic system, the architecture was intentionally designed with depen-
dency inversion to allow future decomposition into micro-services as the system scales.
The system exposes a RESTful web API as the entry point for all application interactions. A
REST (Representational State Transfer) API is a standardized architecture style for building
web APIs. It uses stateless HTTP requests and maps operations to clearly defined resources,
as defined by the original creator Roy Thomas Fielding in his dissertation [44]. These resources
could be presented with /accounts, /users or /blocklists, with predictable behavior based
on the HTTP methods (GET, POST, PUT, DELETE). The REST structure is widely adopted
and understood among developers. Being easy to use with strong conventions and wide support
in different frameworks and languages, the REST API structure decreases onboarding time of
new developers.
The backend itself is structured using a layered architecture of controller →service →repos-
itory. Controllers expose API endpoints and parse requests. Services contains the business logic
and repositories are responsible for direct communication with the database layer. This separa-
tion of concerns makes for better testing, maintainability and easier debugging.
7.4.1
Monolithic Structure versus Micro-services
The entire backend currently has a single entry point called ApiService. This is due to it
being build as a monolithic system. The ApiService exclusively defines the REST API and
leaves request handling to internal services. For example, a POST request to /accounts would
trigger the postAccount() function within the ApiService, which then calls the corresponding
function in the AccountService namespace. That service in turn uses the appropriate repository
to save the data. This sets up a direct flow for later replacing internal function calls with HTTP
requests to the different services, when changing from monolithic to micro-service architecture.
Each namespace can be split into its own deployable micro-service. Figure 17 illustrates the
current monolithic design, where the API Controller routes all logic internally. As shown in
figure 18, these internal calls can be swapped for HTTP requests.

While micro-service architecture offers better error handling, isolation and horizontal scaling,

7.4.3
Account Management
When implementing account management, several security concerns must be addressed, from
safely storing credentials to ensuring that only authenticated users gain access. The following
provides an overview of the complexities and outlines potential approaches to address them.
Password Storing – Confidentiality
Securely storing passwords is necessary to keep online users safe. Rather than storing passwords
in plain text, they can be hashed with a one-way algorithm implemented with salt and pepper
(values added to prevent certain security attacks). This ensures the original passwords remain
encrypted if the database is compromised.
Authentication – Authenticity
Verifying that each request comes from the claimed user is also critical. Authentication es-
tablishes identity, typically through bearer tokens issued after login transmitted over HTTPS.
Proper token management with principles like token families and token rotation ensures that
users can only access their own data and that stolen tokens have limited impact.

7.6
System Architecture - Database
The database serves as the central storage component of the system, handling all persistent
data related to users and other essential entities, that need to be accessed across devices and
user sessions.
A relational database model was chosen due to its built-in functionality for
enforcing data integrity. Integrity is upheld by specifying what should happen to data when
references change or get deleted. This ensures consistent and reliable data management across
the database.
Minimizing data redundancy and maintaining integrity are key goals in database design. Re-
dundancy can lead to inconsistencies and increased storage costs, while integrity ensures that
the data remains true. To address these concerns, the database design follows the principles of
normalization, reducing duplications and ensuring data is organized logically.
This section outlines the conceptual design of the database, details the normalization process
applied to its schema and describes the implementation using Entity Framework within .NET.

After defining the database schemas, we verify that each table conforms to the first, second
and third normal forms (1NF, 2NF, 3NF). Normalization minimizes redundancy and issues with
updates by decomposing tables into smaller tables linked via foreign keys. Normalization ensures
any insert, update or delete only affects one reference and maintains overall consistency [48].

To monitor both the application and backend effectively, Sentry was integrated for real-time
error monitoring. It captures both handled and unhandled errors, providing valuable context,
such as logs and showing user actions leading up to the issue. This made it significantly easier
to reproduce and debug issues.
Sentry automatically assigns severity levels to errors based on both their impact and frequency.
For example, a minor bug occurring for every user daily is marked as severe due to its frequency,
even though the bug might not disturb the user. A critical crash experienced by a single user
however, might also be severe due to its impact, even if only a single user experienced it. This
prioritization helped us determine what we should focus on.
Each error log includes device information, Android version and the application version, allowing
us to determine whether an issue was fixed in a later release. If an error shows up again in a
version where it was marked as resolved, Sentry flags it with the highest severity to indicate a
recurrent error.
To keep the application and backend errors separate, the Sentry project was split in two. This
separation made it easier to identify whether an error originated from the application or the
backend, for better isolation.

a version of the project is saved. This allows developers to revert to earlier states of the systems
if errors occur. In this project, Git was used for version control, with GitHub serving as the
third-party repository platform. GitHub provides additional features such as pull requests and
visual tools for managing branches and commits.
A branching model was adopted to ensure that the main branch remains stable, tested and
deployable at all times, as described in section 6.11 Development Guidelines & Principles. New
features, bug fixes and improvements were developed on separate branches derived from the
main branch. Once completed, tested, and validated, these branches were merged back into the

With version control, CI/CD pipelines were integrated to automate testing, validation and

7.11.1
Internet Slowdown
The Concept
The Internet Slowdown feature was intended as our first premium feature.
Instead of fully
blocking applications, it would allow slowing down the network connection of individual appli-
cations by a set amount when opened. This would allow access to the application, but make
teh experience of scrolling less stimulating.
What Went Wrong?
While the idea was good, its implementation proved not to be feasible due to security issues and
a lack of network expertise. We explored two implementation strategies: a VPN service and a
local proxy server.
Using the built-in Android VpnService, we would establish a TUN (tunnel interface) to intercept
raw IP packets, delay them, and then forward them. Figure 7.11.1 shows the flow.
HTTP Request (Host)
VPN Service
Tunnel
DNS Lookup & Internet
In practice, forwarding each packet requires parsing IP/TCP/UDP headers, resolving DNS for-
warding and lookups, managing sockets, reconstructing packets and writing responses back into
the TUN interface. Existing libraries (tun2socks-android [50], sockstun [51], VPNclient-engine
[52]) were outdated, UDP packets only, or required external servers, making them all unusable
for our particular case.
As a fallback solution, we considered implementing a local HTTP proxy server to introduce
delays without having to do packet level handling.
This also introduced complexity due to
HTTPS traffic. This is because HTTPS is encrypted with TLS, masking the packet content
and complete destination. It would be necessary to decrypt and re-encrypt packets every time
they were forwarded. This would introduce security issues and greater technical complexity, as
exposing the packet content and complete destination, can easily introduce confidentiality and
integrity risks [53], through the use of DNS tunneling attacks.
These things considered, we determined the feature was not feasible within the scope of the
project. It requires the ability of an individual who specialize in network protocols and config-
uration building. It was therefore not implemented.

Category Blocking With DNS
The Concept
The Category Blocking With DNS was the initial idea for the category blocking, before it was

the category of the website from the content that it has. The idea was to use either of the two
DNS setups in our application to attempt to correctly identify what category a certain webpage
or application belongs to for improved category blocking.
What Went Wrong?
The same technical and security challenges described in the section 7.11.1 Internet Slowdown
applies here as well.

For a visual walkthrough, refer to appendix 10.11 Screenshots of the Application and 10.7

The user experience begins with an onboarding tutorial. This introduces core functionalities
and asks the required permissions from the user. The users then proceed to sign in or register
through Azure B2C. If they are creating a new account, the users must accept the terms of
service and the privacy policy.
After the users have signed up/in, they can now navigate to the New Block screen to create a
new block. This could for example be for studying. They can manually select applications or
websites to block, and lists containing the selected items are accessible for review and editing.
Users can also use the Block Categories function to block entire categories.
After creating a block, the user is returned to the home screen, where the new block appears. If

the permission. Once enabled, users can customize locking, blocking and access settings in a
large dialog box. These settings are saved and determine the behavior of the block after turning
it on. Blocks can also be edited or deleted if they are not locked.
At any time, users can access the data reporting page for insights into their phone usage. This
includes category specific usage, trends over time and an overview of the most used apps. Users
can with this information identify areas for improvement to address with the application.

8.2
Future Vision
The product requires more development to reach the completion of the product vision outlined
in Chapter 4. The future timeplan of product development can be seen on the Gantt-Diagram
on figure 26.
Figure 26: Gantt Timeline for Continued Development
The first feature is a custom built database and analytics system. Although this represents a

Windows release will focus only on implementing blocking functionality, excluding data report-
ing and cross-platform synchronization. The fifth feature is the implementation of cross-platform
synchronization. With the application available on iOS, Android and Windows, ensuring syn-
chronization between the platforms becomes a priority. Synchronization of block configurations
and user accounts across platforms will be the primary focus.
The sixth feature is Axion Momentum, a conceptual idea for automating block creation based on
user habit analysis. It involves collecting detailed usage data and utilizing behavioral psychology
to suggest personalized blocking configurations. The seventh feature is a comprehensive overhaul
of the user interface of the application. The goal is to commission a professional designer to
produce concept designs, applying principles from user experience engineering to enhance the
overall usability.
The eight feature is the development of a MacOS version of the application. The development
process resembles the Windows version, and upon completion, cross-platform synchronization
will already be implemented. The ninth feature is improving circumvention prevention. This
involves addressing scenarios, where users can attempt to bypass active blocks, such as disabling
the accessibility service.
Any method that enables the bypass of an active block should be
identified and mitigated.
The tenth and final planned feature for the product vision is a conceptual feature called the
Personal Focus Mode.
Unlike standard blocking functionality, this mode aims to minimize
distractions more comprehensively by disabling notifications, messages, emails and other inter-
ruptive elements. It may include the altering of the user interface, such as enabling grey scale or
relocating distracting applications. On desktop platforms, it is intended to minimize distract-
ing windows, hide the taskbar and silence system notifications. Implementing this feature will
require a re-imagining of the structure of the application, as it diverges significantly from the
current core mechanisms.

Epilogue
9.1
Discussion
While problematic technology use is a multi-faceted problem not easily solved on a society-wide
level, individuals could be empowered to solve it for themselves if they had better tools. The
development of a new solution to be distributed directly to individuals has been started with
this project. Success in further development of the product could help people struggling with
compulsive technology usage.
The project was executed using wide range of disciplines; academic research, business strategy,
computer engineering, startup methodology and agile planning. It has been difficult to tie the
different parts together into a coherent project that serves a single objective. Nonetheless, the
multi-disciplinary nature of the project has proven an interesting and complex challenge.
9.1.1
Problem Exploration
Finding sources to support the claims made in chapter 3 presented several challenges.
As
technologies of digital entertainment has evolved quickly, especially in the last ten years, the
novelty of scientific articles carries importance. Older findings were not used if the subject they
treated had changed significantly since. This was especially relevant with regards to articles
treating the subject of potential plastic changes to the brain stemming from technology use, as
the nature of entertainment commonly found online has become more intense in the last few
years with the advent of short-form video content.
As argued in chapter 3, there is widespread awareness of the issue, but a lack of clear and simple
solutions. Unless a major shift in the regulation of technology companies is eminent, our clear
understanding is that the problem will continue to grow. Technology companies will engineer
better algorithms, more stimulating experiences and new methods of delivering content through
advanced user-interfaces. This will drive an increasing need for new solutions and innovation to
counteract the unregulated progress in digital entertainment technology.
9.1.2
Solution Innovation
The Blue Ocean Strategy [1] was used with the intention of identifying openings in the market.
It is a business strategy method, and thus rely on the ability of the team using the method
to correctly identify which aspects of a product or service to focus on. When the Blue Ocean
Strategy is used by large corporations, it is often accompanied by a considerable amount of
market research, data and customer knowledge. We did not have the resources to do extensive
information gathering work, and thus the conclusions drawn in chapter 4 are entirely based on
our ability to apply the strategy and estimate certain qualities of competitors and customers
accordingly.
While we had the goal of solving the problem presented in chapter 3 since the beginning, the
solution was and will continue to be the result of a continuous process of innovation. The success
of the current solution both technically and commercially depends on assumptions which are
continuously validated or invalidated. This constantly shapes the direction in which we will
steer innovation to achieve a better solution to the problem.
9.1.3
Development
The development strategy was centered around completing the MVP quickly. This was due
to the belief that launching quickly would allow us to learn faster from user feedback.
We
found some errors in the application, and we incorporated the collected user feedback into the
development; the tutorial, the placement of the accessibility service requests, and the navigation
bar was all shaped by user feedback. While this gave us an advantage in development decision
making, releasing an MVP quickly caused the application to lose the initial user base we got
Page 66 of 118
==PAGE:68==

after launch. As we had an undiscovered crash, as described in section 6.9 First Marketing
Campaign & Feedback, the initial users stopped using the application quickly. Seemingly, the
crash deterred them from using it, and it might have caused the invisible ”ranking” of the
application on Google Play Store to fall. This could explain why we had a low organic user
acquisition when we released iteration five. While general startup advice is to release products
quickly, it is possible that our application would have benefited from more thorough testing.
We did learn from this mistake; as described in section 7.3.8 Global Exception Handler, more
advanced crash monitoring was set up to avoid undetected crashes going forward.
While iterations were made quickly, they were not rushed in terms of code quality. From the
beginning, the idea was to continue the project as a startup. Therefore, we had scalability and
future complexity increases in mind when building the structure of the system. Prioritizing good
architecture has already proven useful - some of the aspects of iteration five was not planned
to begin with, but integrating them into existing code was simple due to a solid fundamental
structure.
One of the major challenges in the project was the lack of user data after the initial surge of
users was lost. While somewhat sophisticated monitoring systems has since been set up, we have
not had the opportunity to collect any meaningful quantity of data. Thus, it is unknown how
well iteration five of the application would perform with users. The lack of data after the initial
surge had the consequence that it was not possible to let user feedback guide the development
towards the end of the project period.
Since we are currently relying on third-party analytics systems like Firebase and Google Console,
the metrics we can measure are delayed and limited in scope. Improving the ability to measure
more specific events and obtain data faster would allow us to extract more data from fewer
users. As described in Chapter 8, the immediate next task after ending this project will be to
develop an in-house database and analytics system, to get improved measurements. Getting
more data from fewer users will be a crucial step towards a better understanding of our KPI’s,
helping guide future growth efforts.
9.1.4
User Acquisition and Marketing
The marketing campaign on DTU, section 6.9, provided the feedback we were looking for -
we got to see how the application was used live. This granted insights into issues with the
onboarding flow, and we made relevant changes to the application. A substantial amount of
development had to be done to accommodate the feedback we got. Therefore, we think it was
the right choice to focus on development after the first user acquisition effort; spending time,
effort or money on acquiring new users when they would likely tell us information we already
knew would have been redundant. We do not yet have any idea how well iteration five will
perform, as it does not suffer from the same shortcomings as previous iterations.
As the financial resources have been limited, due to the project being entirely founder-funded,
paid marketing campaigns has been inaccessible. It is not yet clear what the Customer Lifetime
Value will be, and it is therefore hard to justify paid marketing campaigns. This leads to slower
user-acquisition, which in turn make it harder to estimate the CLV. To get around this, organic
marketing will be used until CLV can be somewhat accurately estimated.
9.1.5
Future Perspectives
The project has provided an excellent start towards developing a novel solution to problematic
technology use for individuals. The plan for continuous development, section 8.2, will be followed
to the best of our ability. Alongside development, more effort will be put into user acquisition
and marketing to validate or disprove assumptions. The completion of the plan will likely be

dependent on our ability to raise pre-seed funding, as such we are still in a vulnerable position.
If the current plan is successful, we hope to build an attention manager with a broad appeal
which could help many change their problematic technology habits.
Further work could explore how different technology habits interplay in more detail. Knowledge
of how the newest entertainment technologies affect human behavior is yet incomplete. In this
report, we argued for one possible solution to problematic technology use. The digital entertain-
ment technologies are advancing at a rapid pace, and researching the impact and spearheading
innovation which could protect customers remain relevant.
9.2
Conclusion
The original objective was to investigate problematic technology use, innovate a solution, for-
mulate a plan and begin an iterative development process. The product quickly reached the
MVP stage, and we reached iteration five within the project period. The agile methodology
was executed well. The cross-disciplinary nature of the project allowed us to experiment with
building an entire system and implement it with current industry infrastructure.
Unlike in
purely academic environments, the launch of the application led to a confrontation with the
requirements of usability and simplicity demanded by real users.
The application in its current form could be used by an individual to monitor and manage their
technology use. Therefore, we consider the project successful in achieving the original objective.
The success of the solution in a broader sense require further efforts in marketing, product
development and fundraising. Future work will focus on refining the product. This includes
developing a behavioral algorithm for automated block creation, expanding the application to
iOS and conducting controlled testing to gain insight into user behavior.
We are personally satisfied with the project. It provided an opportunity to apply our academic
knowledge in a free but demanding setting. We believe that this project is a solid foundation
for the completion of the product vision.

IFNULL(a.active_users, 0) AS active_users

SELECT
event_date,
active_users,
LAG(active_users) OVER (ORDER BY event_date) AS prev_active_users,
CASE
WHEN LAG(active_users) OVER (ORDER BY event_date) IS NULL THEN NULL
WHEN LAG(active_users) OVER (ORDER BY event_date) = 0 AND
active_users > 0 THEN 1.0
WHEN LAG(active_users) OVER (ORDER BY event_date) = 0 AND
active_users = 0 THEN 0.0
ELSE SAFE_DIVIDE(active_users - LAG(active_users) OVER (ORDER BY
event_date), LAG(active_users) OVER (ORDER BY event_date))
END AS user_growth_rate
FROM filled_active
ORDER BY event_date;
This query calculates the daily user growth rate by comparing the number of active users from
one day to the next. It starts by generating all dates. For each of those dates, it then counts
how many unique users triggered the active user event.
To make sure all dates are included in the output, even if no users were active on a given day,
the query fills in missing days with zero active users. This is simply for better readability. It
then calculates the number of users from the previous day using the LAG function and compares
it to the count of the current day to compute the growth rate. The LAG function simply takes

10.9
MVP User Stories
User stories are short, simple descriptions of a feature told from the perspective of the end user.
They help bridge the gap between technical requirements and user needs by focusing on who the
user is, what they want to do and why they want to do it. Using user stories allows us to better
understand the intended functionality and prioritize development based on actual user need.
They are easy to communicate, align well with agile development environments and support
iterative design.
Each user story typically follows the format:
As a [type of user], I want to [do something] so that [goal or benefit].
This format ensures that both the purpose and context of each feature is clear. In the following,
we outline key user stories that guide the development of our system.
Authentication and Account Management:
• As a new user, I want to sign up with an account, so that I can save my blocks and
preferences.
• As a returning user, I want to stay signed in between sessions, so that I can use the
application without logging in every time.
• As a user, I want to securely sign out of my account, so that my data stays protected
when I stop using the application.
Blocking Functionality:
• As a user, I want to create a block for distracting apps and websites, so that I can stay
focused.
• As a user, I want to be redirected to home screen if I try to open a blocked application,
so that I am reminded of my active block.
Analytics and Error Reporting:
• As a developer, I want to track when users sign up, start blocks and other interesting
events, so that I can understand how the application is being used and improve upon it.
• As a developer, I want to collect error reports, so that bugs can be identified and fixed
more efficiently.
Backend Infrastructure:
• As a developer, I want the backend to validate all incoming requests, so that unauthorized
access is prevented.
• As a developer, I want the backend to scale with more users, so that performance remains
stable over time.

10.10
Sprint Planning
Sprint planning and tracking were managed using Asana [56], a project management tool. Tasks
were categorized by domain and visualized using a Kanban board, with states such as To Do,
In Progress, Blocked and Done. At the end of each sprint, tasks that were not completed were
evaluated. They were either moved to the backlog, postponed to a different sprint or resulted
in an extension of the current sprint.
Each sprint either contained or concluded with an event (for example board meetings, supervisor
sessions, user feedback), which helped shape the direction of the next sprint. The initial sprint
plan can be found in appendix 10.4 Initial Project Plan.
However, as is common in agile
development, the plan evolved significantly over time.
This means, that the final planned
sprints, skew a lot from the original plan. Below is an explanation for each of the sprints, that
gives an inside into how things quickly changes and adapt, in an agile environment.
10.10.0.1
Sprint #1 — Application Prototype
Focused on developing the first working prototype and laying the foundation of the application.
It included basic blocking functionality for a single predefined application and website.
10.10.0.2
Sprint #2 — Customization Features
Expanded the prototype to allow users to define blocking rules, like setting which applications
and websites to block, and improved the user interface.
10.10.0.3
Sprint #3 — Account Management & MVP Completion
Introduced account management, creating the backend on Azure, analytics integration and
error-monitoring integration. A meeting with the Board of Advisors led to a slight pivot in
the original release plan. Instead of first creating a closed beta in Sprint #4, the application
would be released via Google Play Stores open testing. This was done to follow the principle
of fail early. We wanted as many users as possible, to gain insight into user behavior and catch
all the issues with the application. Open testing allowed broad testing while preventing public
ratings from affecting the application store listing. Discussions also resulted in dropping iOS
and Windows support at launch, focusing solely on a robust Android release. By building the
Android application well, it it would give a better foundation for the other platforms in the
future. Rather than simply releasing all three, with minimal features and issues, which would
then needs to be fixed on all three.
With finishing the implementation of the account management, this sprint also marked the com-
pletion of the MVP. Reaching the MVP was crucial for an effective launch. This therefore marks
the milestone, for the first complete foundation of the application for further improvements.
Each sprint from here will therefore evaluate new features and improvements, to determine
which to implement. When determined, requirements will be discussed and agreed upon.
10.10.0.4
Sprint #4 — Launch (Europe)
Prepared for and executed the European launch. Tasks included bug fixes, better analytics
setup and creating a public website, to further create and grow the Axion brand. This sprint
also featured the first, primitive implementation of the data reporting feature, which could
simply collect the data, but could not show it.
Feedback from a meeting with “Center for
Ludomani” [35] (the Danish center for gambling addiction) prompted an increase in the need
for the feature for category based blocking, extending this sprint to include its implementation.
The application gained approximately 10 users through this soft launch, also giving a small
insight into user behavior, while also discovering bugs.

10.15
Navigation Bar Comparison
While it might seem obvious that the application should be structured this way, it is actually
a very minimalistic design compared to our competitors. They have entire screens in the navi-
gation bar which has completely different purposes than supporting core functionality. Take a
look at the navigation bar from the freedom app, for example, shown on figure 44.
Figure 44: Freedom’s Navigation Bar
The today tab works a little like our home-screen, but with a lot more confusing features, for
example a blog with hundres of articles with links to the web.
It also shows blocks which
happened in the past, which we believe is of little interest to the users. The ”Start Session” tab
is like our ”New Block tab, but has one major difference. Freedom does not have a ”block” as
a data class, like we do, which then contains content and blocking options. Instead, they have
”block session” which contains ”block lists” and blocking options. The user has to first create
a block session, then create any number of block lists which each contains lists of websites and
applications. Then, they have to select any number of those block lists and attach them to the
block session. Thus, there are extra steps in creating a block in freedom - it is very confusing, and
we tried letting people we knew create a simple 1 hour block of YouTube on freedom. Several of
them could not figure out how to do on their own, suggesting an unnecessarily overcomplicated
data structure.
The next tab is block lists - this tab creates the 2nd element of freedoms blocks. Unneccesary tab
for our application, as we just have one data class: the ”block” class. On their 4th navigation
tab, they made the choice of have a library of what they call ”focus sounds”. It is a collection
of recordings from woods, cafes, or other ambient settings. While some might like this feature,
dedicating 20 % of the applications navigation space in a mobile UI seems like it compromises
the core functionality and simplicity for the user, and thus like an illogical design choice. Their
final tab is settings, just like ours. Freedom chose to prioritize giving the users a sound library,
over for example a screen time rapport.
Opals navigation bar is shown on figure 45.
Figure 45: Opal’s Navigation Bar
Opal’s Home screen contains a acreen time rapport - it is not very detailed, does not show
categories, on does not aid the user significantly in clearly marking which applications generally
takes up the most of their time.
It does have a good-looking design.
Their blocks screen
handles both the creation and management of blocks. While it does make navigation a little
more difficult, it means that creating and managing blocks will only take up one slot in the
navigation bar instead of two. The third tab is ”Leaderboard”, which provides the user with
the option of gaining friends and competing in reducing screen time. It is clearly an attempt
to increase traditional engagement metrics with the application and benefit from some kind of
virality gained by integrating a little social network. Again, the choice of spending 25 % of
the navigation space on features which have nothing to do with the core features is certainly a

network, but it seems like a detour from the functionality and simplicity of the application. The

Hence, compared to the industry, our application has a simplistic, minimalist and easily usable
design. Prioritizing less features, a simple data structure and no non-core features lets the tool
