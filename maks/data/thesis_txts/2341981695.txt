Summary (English)
The goal of the thesis is to create an algorithm capable of gathering traﬃc count
and estimating velocities of vehicles in congested traﬃc. The most widely used
methods for traﬃc data gathering are either to use an induction-loop counter
inserted in the road, or to use manual counting. computer vision oﬀers a cheap
and reliable alternative to the previously mentioned methods.
Some of the
typical methods for gathering traﬃc data using computer vision are median
ﬁlter, Gaussian Mixture Models and otical ﬂow. These methods work very well
in free ﬂowing traﬃc but are in this thesis proven not to work as well in congested
traﬃc due to slow and static vehicles as well as changing light conditions. The
methods are tested under diﬀerent conditions, and a new method is suggested
using edge detection which is more robust towards changes in light and not
sensitive towards static vehicles. A counting algorithm is made on the basis of
this edge detection. The counting algorithm looks at how many edges are in a
region of the video frame. This is used to estimate whether a vehicle is in the
region or not. The counting algorithm works very well and has an accuracy of
98.08% in counting the right number of vehicles. Velocity estimation is done
on the basis of tracking feature points from one frame to another. In order
to translate this distance to a velocity, a homography transformation is made.
This can only be done if the distance between four points on the road are
known. The accuracy of the velocity estimation is measured to be within 10
km/h of the actual value. This is not extremely precise but is accurate enough
to be used for traﬃc models. One of the interesting aspects, of this way of
doing velocity estimation, is that it is possible to estimate the velocity from one
frame to another. This means that it is possible to see if the car is slowing
down, speeding up, stopping or starting. It could potentially be used to make
more complex models of congested traﬃc and give a better understanding of
==PAGE:4==
ii
congestion. Vehicle type estimation is done based on the sum graph created
by the counting algorithm, as well as a area of the vehicle.
Vans create a
problem because they have overlapping sum features and area values with cars
and caravans. To solve this problem 6 groups have been created: motorcycle,
car, car/van, van, truck/trailer, cravan/van. These overlapping classiﬁcations
makes it possible to reach 93% accuracy with the type estimation.

Chapter 1
Introduction
In this chapter a motivation for the thesis is given. The problem formulation
for the project is given which forms the core problem that the thesis addresses.
The initial and revised project plans are included as well as a self evaluation of
the planning and execution of the project.
1.1
Motivation
Traﬃc counting is one of the most important elements in building accurate
traﬃc models. Having a high accuracy traﬃc count is important in developing
appropriate traﬃc models. There exists several ways of collecting traﬃc count
which can aﬀect the accuracy due to their properties. The Danish Road Directorate
uses loop detectors (inductive-loop traﬃc detectors) on some of the main roads
in Denmark. These loop detectors work by having a wire loop installed in the
road.
An alternating current is introduced in the system at 10 KHz to 200
KHz. When a vehicle passes the system it introduces a small current in the
wire which can be measured and registered. Another widely used method is to
use manual counting, however, this can reduce accuracy due to human error.
Using computer vision and image analysis to extract traﬃc data, such as traﬃc
count and vehicle speed, could be an eﬃcient approach to gathering traﬃc data.
Multiple companies, like Picomixer, Mirovision and Puretech Systems, all use
==PAGE:20==
2
Introduction
computer vision and image analysis in their counting and tracking software.
Most of the current methods and research has been done in free ﬂowing traﬃc.
When building traﬃc models to increase traﬃc ﬂow it is also relevant to get
information about congested traﬃc (slow moving and possibly static vehicles).
Since vehicles in congestion are very close and moving slowly or are potentially
static, it requires a diﬀerent approach compared to free ﬂowing traﬃc.
1.2
Problem formulation
Is it possible to create an algorithm to acquire vehicle count and speed in
congested traﬃc? How accurate can this be made, and what will the computational
runtime per frame be?

1.4
Self Evaluation
The initial project plan was based on the fact that one of the typical segmentation
methods (median ﬁlter, Gaussian Mixture Models, or optical ﬂow) would be
useful in this situation. This proved not to be the case, and a new method
needed to be developed for this scenario. In the beginning I recorded all data
with vehicles moving towards the camera. This worked well for counting but
proved problematic for tracking. New data with vehicles driving away from the
camera needed to be recorded, and the algorithms needed to be changed to deal
with this. Writing, getting results, and test data, has proved to take much longer
than initially expected, and sometimes a day of ﬁlming could result in "useless"
data, because the camera was not set up correctly. All this delayed the project
further. Some of this "useless" data has been used to showcase situations where
the algorithm does not work optimally. If I could change something it would
be the naive assumption that some of the typical methods could work without
issues in congestion.

Chapter 2
Related Work
Multiple vision-based methods have been proposed for collecting traﬃc data and
for analysis of the data. The problem is normally divided into segmentation and
tracking. Segmentation is used to separate the vehicles from the background,
where tracking is used to estimate ﬂow, speed, and lane changes etc.
A video is merely a sequence of two-dimensional images taken at a speciﬁc time
interval - this is referred to as frames, where the number of frames per second
is the framerate. Using computer vision on a video is usually done by going
through the sequence one frame at a time.
video = [f1...fn]
(2.1)
where n is the number of frames in the video. fi will from here on describe the
current frame that is being analyzed in the sequence.
In order to get information about movement and time relations, two or more
frames are needed. Vision-based methods for video analysis therefore mostly
use the relation between previous frames and the current frame to get relevant
data.
==PAGE:26==
8
Related Work
In this chapter a short description of the most commonly used methods will be
made.
2.0.1
Segmentation
Background subtraction is the simplest of the segmentation methods. In the
ideal situation the current frame fi would be subtracted by a true background
image of a road with no vehicles.
If the current frame has a vehicle in it,
it would be segmented from the background, as it would be the only diﬀerence
between the true background and the current frame. The main problem with this
approach is that the background scene varies (day/night, weather and seasons).
Since the true background image is rarely available, due to vehicles constantly
driving on the road, an estimate needs to be made. A simple approach proposed
by
[Lipton et al., 1998] suggests that consecutive images are subtracted to
separate the moving objects from the static background - this is called temporal
diﬀerentiation. [Lo and Velastin, 2001] propose to use a pixel-wise median value
of the last n frames as a background estimate. [Nunes et al., 2011] describes a
method where a threshold value is used on the diﬀerence image to segment the
foreground. A histogram describing the diﬀerences between the current frame
and the estimated background is used to compute this threshold value.
Estimating the background by using a previous number of frames is not useful
for situations where the camera is not steady or vehicles are moving slowly and
are static for a longer period of time. Moving shadows will also be segmented as
foreground, which could make it diﬃcult to separate vehicles in some situations.
Another widely used method is "Gaussian Mixture Model" (GMM), which
creates a Gaussian distribution for each speciﬁed class (background and foreground),
suggested by [Stauﬀer and Grimson, 1999]. This is used to classify the probability
of a pixel belonging to background and foreground. The class with the largest
probability will be the classiﬁcation of that pixel.
A similar approach has
been used by [Friedman and Russell, 2009], where an iterative approach called
Estimation Maximization algorithm is used to generate three Gaussian distributions
(background, shadow, foreground).
[Piccardi, 2004] makes a review of the previously mentioned background models,
as well as others. His conclusion is that using a median ﬁlter oﬀers an acceptable
accuracy, but GMM provides a more accurate model.
==PAGE:27==
9
2.0.2
Tracking
Tracking of vehicles can either be based on tracking the segmentation described
in the previous section or tracking image features and movement from one frame
to another.
Optical ﬂow (OF) estimates the movement from one frame to another. This is
done by calculating the partial derivatives in the current and previous frame,
which basically describes the edges in the frames. By comparing the current
and previous frame, it is possible to see how these edges move. Each pixel will
then have a vector describing how much it moves and in which direction, from
the previous to the current frame. One of the problems with this method is
that it is not possible to get the OF of homogenious areas where pixels have the
same intensity (for example the top of a vehicle). This method could prove
problematic in congested traﬃc where vehicles are static and no movement
occurs. An example of using OF for tracking pedestrians has been proposed
by [Patel and Parmar, 2014]. A widely used approach for calculating the OF
is the The Lucas-Kanade algorithm developed by Bruce D. Lucas and Takeo
Kanade ( [Lucas and Kanade, 2009]). This method is used in [Wu et al., 2013]
to do pedestrian counting and crowd density estimation.
[Kalman, 2010] was one of the primary developers of the linear quadratic
estimation algorithm, also known as the Kalman ﬁlter. The ﬁlter can be used to
make tracking more robust and makes it possible to predict motion if the object
is occluded in some part of the image sequence. The algorithm essentially uses a
weighted average of the position in the previous and current frames to calculate
an estimate of the actual position. If the traced object is occluded at some
point it will only have the previous position and an assumption of the motion
model as basis for estimating the new position. The motion model could for
example be constant acceleration or constant speed. This algorithm is used by
[Yu et al., 2003], to improve tracking accuracy.
[Beymer et al., 1997] suggests a feature-based tracking algorithm to track vehicles.
This method ﬁnds image features and tracks them from one frame to another. It
also removes the need for background subtraction, because only moving features
are regarded as valid. This approach can however be computationally expensive
as the entire frame needs to be analysed to ﬁnd the matching feature. Since
they track all matching features from one frame to another, they need to perform
classiﬁcation algorithms in order to match the features to a speciﬁc vehicle which
adds more computations. Algorithms have been made to improve the feature
tracking by suggesting a windowed search ﬁeld ( [Visvanathan, 2009]).
The
algorithm does not need to search the entire next frame to locate the same
feature but only a small area around the place where it previously located a
==PAGE:28==
10
Related Work
feature point.
An adaptive window size is suggested by
[Qian et al., 2007],
where the window changes size depending on movement etc.
The feature-based tracking algorithms have the advantage of not being as dependent
on the environment as segmentation algorithm., However, feature tracking can
be computationally heavy and slow if the entire frame is analyzed.
The median, Mixture of Gaussian, optical ﬂow, and feature tracking methods
will be tested in this thesis to ﬁnd the best choice for congested traﬃc.

Chapter 3
Data Set
This chapter will give a description of the highway video test data, test locations,
camera positions, camera angles, and pre-processing. Choosing these correctly

3.1
Data Description and Location
To create and test the algorithms discussed in this thesis, video sequences of

color and intensity giving a higher variation. Based on this the current history
of each pixel in the frame can be modeled by K Gaussian distributions, where
K in our case is chosen to be 3. An illustration of this is seen in ﬁgure 4.7.

Edge Detection
Both the median ﬁlter and the GMM slowly adapt to the scene, making them
unreliable when dealing with the possibility of vehicles being static. Using edge
detection to segment vehicles could be one way of avoiding adaptive methods.
==PAGE:51==

Many diﬀerent edge detection methods exist. Some of the most widely used are:
• Canny
• Laplacian of Gaussian (LoG)
• Prewitt
• Roberts
• Sobel
In [Maini and Aggarwal, 2009] a study of edge detectors has been made. They
conclude that edge detectors such as Sobel and Prewitt have the advantage
of being simple, but the disadvantage of being sensitive to noise and not as
accurate as some of the other edge detectors. LoG and Canny ﬁnd the correct
places of the edges, and Canny is more robust in noisy situations but is also
more complex and time consuming.
Knowing the exact location of the edge is not that important when segmenting
vehicles from background. The task of this thesis is to make an algorithm that
works in daytime, therefore, sensor noise will not be an issue. If an algorithm
should work at night, the sensor noise would be higher and Canny would be
preferred instead of one of the other methods to increase robustness.

4.4.1
Boundary Box
One of the most widely used ways of tracking is by creating a boundary box
around the segmented blob (see ﬁgure 4.32). This is tested with the Matlab
function "vision.BlobAnalysis()", which calculates the trace of each blob, and
from this the boundary box.
Figure 4.32: tracking using a boundary box and Kalman Filtering
The center of the boundary box is used as the tracked location. By using a
Kalman Filter a new position can be predicted. It essentially works by making
an evaluation between the predicted and the measured locations. Based on this,
an estimate of the actual location is made. Assigning the tracked location to an
existing track is done by minimizing a cost function. The cost function describes
the distance from the current tracked location to all previous tracks. The track
with the lowest cost is chosen unless the cost of creating a new track is lower.
The cost of creating a new track is manually chosen to an appropriate value.
Setting this value too low will result in too many new tracks being created.
Setting it too high will result in diﬀerent vehicles being assigned to the same
track.
The edge segmentation of the vehicles is not very good in sunny scenes. Creating
a boundary box around the entire vehicle is therefore not possible (see ﬁgure
4.33).
The bad boundary box segmentation results in a vehicle often being tracked as
multiple vehicles. Boundary box tracking requires a good segmentation in order
to get a good tracking. This is not possible in a sunny scene, and therefore
tracking using a boundary box is not the appropriate method to use.

4.4.2
Feature Tracking
Feature tracking consists of locating and matching feature points from one frame
to another. In order to do this a description of feature points needs to be made.
4.4.2.1
Feature Points
A feature point describes a speciﬁc point in the image and some also describe the
area around a point. There are many diﬀerent types of feature point detectors:
• Corner detectors: locate corners and save them as a matrix holding the
[x,y] coordinates. These are good for situations with lots of corners. Work
well in man-made scenes.
1. Features from Accelerated Segment Test (FAST)
[Rosten and Drummond, 2005] [Rosten and Drummond, 2006]
FAST works by creating a circle with a 3 pixel radius. This becomes
a circle with a circumference of 16 pixels around the candidate corner,
p. The pixel p will be classiﬁed as a corner if: there exists a contiguous
number of pixels n that have an intensity larger than the candidate
pixel plus a threshold t (Ip + T), or lower than the pixel minus a
threshold (Ip −T). n is chosen to be a value above twelve. This is
visualized in ﬁgure 4.34. The Matlab function "detectFASTFeatures()"
is used.
2. Minimum Eigenvalue
(http://se.mathworks.com/help/vision/ref/cornerdetection.html)
Calculated by computing the horizontal and vertical derivatives of
the image to get the images I2
x, I2
y, IxIy. These are convoluted with
a Gaussian and called A,B,C. This is arranged in a matrix M =
[A, C; C, B] and the minimum eigenvalue describes a corner.
The
Matlab function "detectMinEigenFeatures()" is used.

3. Harris [Harris and Stephens, 1988]
The Harris corner detector works similarly to the minimum eigenvalue
but avoids calculating the eigenvalues of M, thereby making it faster.
This is done by calculating r(x, y) = AB−C2−k(A+B)2. Since a lot
of smoothing has occurred, non-maximum suppression is used to ﬁnd
the speciﬁc place of the corner. the Matlab function "detectHarrisFeatures()"
is used.
• Speeded-up Robust Features (SURF) [Bay et al., 2006]
SURF uses the Hessian matrix to detect blobs and the determinant. A
point is chosen where the determinant of the Hessian is maximal. This
method involves looking at the image at multiple scales of blurring. The
Matlab function "detectSURFFeatures()" is used.

5.2
Improved edge detection
The edge detection, investigated in the Methods chapter, works fairly well
but can be improved to increase the robustness and performance.
The ﬁrst
improvement is to select an appropriate color channel based on vehicle color.
The second improvement is to dynamically change the edge detection threshold
depending on the intensity of the light in the scene.
5.2.1
RGB Color Channel
Edge detection uses gray-scale frames. The usual way of obtaining a gray-scale
image is to calculate the intensity image by using a weighted sum of the R,G,B-channels:
I = 0.2989 · R + 0.5870 · G + 0.1140 · B
(5.2)
The basis of this is that the human eye is better at perceiving green than red
and blue. Based on this formula it is clear that red or blue vehicles will be
relatively dark in the intensity image as seen in ﬁgure 5.8.
By looking at the individual color channels in the rgb image (ﬁgure 5.9), it is
possible to see that a blue vehicle will logically be brighter in the blue channel.
Since the edges are not as pronounced in darker vehicles, the edge detector has
more problems locating them. This is especially problematic in a sunny scene
with a higher edge threshold. By locating the color channel where the vehicle is
brightest and using this for the edge detection, it is possible to make the edge
detector more robust.

The improvement is done by using the intensity image to locate when a vehicle
is inside the sumROI. Once the binary representation rises to 1 (see ﬁgure 5.7),
the sumROI on each color channel is subtracted with a very rough background
estimate.
The background estimate is the frame at t-1, which is the frame
before the sumROI is triggered. This is done because the background intensity
at the diﬀerent color channels are a little diﬀerent, which can eﬀect the ﬁnal
result. Therefore, it is not possible to just look at the maximum of the color
channels without subtracting the background. Subtracting makes it possible to
disregard the background, and only look at the vehicle. By taking the sum of
each individual color channel in the sumROI, it is possible to locate the color
channel where the vehicle has the highest intensity. This procedure is very fast
and takes an average time of 0.000421 seconds per frame (at a resolution of
640 x 360). The process makes it possible to minimize the number of "dark"
vehicles. This approach will of course have no eﬀect on a vehicle that is black
or dark gray.

As previously written it is not possible to do blobtracking very well, because,
the segmentation is not very good under sunny conditions. Feature tracking
does not use segmentation and is therefore much more reliable in this situation.
The main problem is how to get the initial feature points for the tracker. One
approach is to ﬁnd feature points in the entire image, and use clustering to
group these into the diﬀerent vehicles ( [Beymer et al., 1997]). This approach is
a very slow one since there are many feature points that need to be reinitialized
every now and then to get features from new vehicles. The clustering algorithm
also needs to be performed quite often to group features to vehicles.

section on counting, it is possible to determine when a vehicle enters and exits
the sumROI. This information can be used to get an approximate location of
the vehicle. It can be assumed that when the vehicle exits the sumROI it will
be in the area just above the sumROI. Depending on the velocity of the vehicle
it will be further from or closer to the sumROI. This means that it is possible to
create a new ROI above the sumROI, this is called trackROI (see ﬁgure 5.12).

Chapter 7
Discussion
In this chapter the results of the algorithm, and general considerations are
discussed.
7.1
Choosing the Right View
Depending on the speciﬁc information needed, the orientation, FOV, angle, and
resolution can help making it easier to gather the relevant traﬃc data. Choosing
these parameters incorrectly could give poor results from the algorithm. The
algorithm created in this thesis counts the vehicles and estimates their velocity
and vehicle type. For this purpose only a low resolution is needed. As seen in
the result section this also decreases the runtime per frame. In this situation
the FOV does not play a critical role, as long as the vehicles have a high enough
resolution for edges to be detected. When looking at the velocity estimation
the resolution might have a larger impact. A low resolution means less pixels
for tracking movement from one frame to another. This will logically decrease
the accuracy. Since velocity accuracy is not a big concern when creating traﬃc
models, this should not be a problem.
Choosing a view where the vehicles are driving away from the camera, has proved
crucial for the algorithm. This made it much easier to ﬁrst separate the vehicles
==PAGE:126==
108
Discussion
and then track them. This method has eliminated the need for classiﬁcation
and grouping of the feature points.
If the user is interested in reading license plates, the resolution and ﬁeld of
view has a big impact on the results. To be able to read the license plates, the
resolution needs to be as high as possible. This can be achieved by making the
FOV smaller and increasing the resolution of the recorded video sequence.
In this thesis the video has been recorded in either 1920 x 1080 at 23.976 fps,
or 640 x 480 at 50 fps. The 1920 x 1080 video has been downscaled to 640 x
360. This downscaling takes a long time to do on the CPU. It would therefore
make more sense if the user could record directly in a low resolution format.
7.2
Tested Methods
In this thesis, congested, slow moving traﬃc has been the main target of the
algorithm. Due to the slow and sometimes static traﬃc, both median ﬁlter,
GMM, and OF failed in segmenting the vehicles from the background to a
degree where it was useful. These methods work very well in fast moving traﬃc
and to some degree in slow moving traﬃc - however the quickly changing light
conditions made it useless for the purpose of this thesis. Edge detection is more
robust in regards to changing light conditions. In extreme conditions like low
exposure, high exposure, or low shutter speed, it does however not work very
well.
7.3
Counting
Using edge detection to estimate the presence of a vehicle in a ROI is a very
fast, eﬃcient, and robust way of counting the vehicles. The total accuracy of the
algorithm on the tested videos is 98.08%, which is relatively high. One of the
problems is trucks with a very homogeneous surface. Here the edge detector will
have diﬃculty ﬁnding any edges, and the counter might not count the vehicle.
The algorithm counts each lane individually, which could help increase the
complexity and accuracy of traﬃc models. The counting part of this algorithm
is relatively accurate and will create reliable results, if the camera is set up
properly for the situation.
In this algorithm, the situation where a vehicles
enters the frame between the two lanes is not considered and will therefore
trigger a count in both the left and the right lane.
==PAGE:127==
7.4 Velocity
109
It is seen in the results that in a sunny scene, more false positive errors occur.
The reason for this is that the dynamic edge threshold is set higher in a bright
scene than in a low light scene. This means that the edge detector might not
ﬁnd enough edges on the middle part of the vehicle, which results in the vehicle
being counted as two or more vehicles.
7.4
Velocity
Estimating the velocity is based on making a homography transformation of
the road. This creates a warp which stretches the vehicle depending on how
far away from the camera it is.
Since the feature points are located on the
vehicle, this warping will likely result in an error when measuring the distance
of a feature point from one frame to another. This will in the end result in a
velocity estimate that is not as accurate. The accuracy of the velocity estimate
is measured to be within 10 km/h of the actual velocity.
Getting an exact
measurement of the accuracy is diﬃcult - especially at lower velocities, since it
is hard to ﬁnd a test location where it is possible and legal to drive at such low
velocities. The radar gun is not capable of detecting velocities under 15 km/h,
making it harder to test these velocities. The actual accuracy might therefore
be higher at low velocities.
If the vehicles are driving at a constant velocity it is possible to get their
velocities by simply ﬁnding how many frames it takes for the vehicle to cross a
certain known distance on the road. If the vehicle is going at a high velocity
this approach is not very good since few frames are available to estimate the
velocity. At low velocities it is possible to get a higher accuracy. In congestion,
this approach would not be useful, because some vehicles might be static for
a period of time and then move again. The average speed would not describe
what is actually happening in the congested traﬃc. This is where the velocity
estimation suggested in this thesis is very good. It calculates an estimate of the
velocity from one frame to another, making it possible to describe the movement
of the vehicle in the region. This could possibly help in creating even more
reliable and complex traﬃc models for congested traﬃc.
7.5
Type Estimation
Vans have an overlapping area with cars and caravans. The fact that vehicles
do not always have the same number of large peaks in the sum graph, and the
==PAGE:128==
110
Discussion
overlapping area, makes it diﬃcult to classify vans in speciﬁc situations. To
solve this problem, more complex methods are needed, or another parameter,
where vans are more easily segmented from the other vehicle groups. Due to
time limitations of this project, only a simple approach was tested for type
estimation. The method used in this project is able to classify vehicles with two
overlapping groups, with a 93% accuracy, which is quite good.
7.6
Robustness
A general problem of this algorithm is the sensitivity to light conditions. Setting
the exposure of the camera correctly will give good results, where setting it
incorrectly will give poor results. It is important that the shutterspeed is fast
enough, for vehicles not to be blurry. It is also important that the aperture
is small enough to have most of the road in focus. Lastly the ISO sensitivity
should be set so the scene is not too bright and not too dark.
If the scene
changes from bright to dark this could give poor results.
A solution to this problem could be to measure and log the light intensity using
a luxmeter. This would eliminate the need to ﬁx the exposure of the camera.
The camera could then automatically correct the exposure according to the light
conditions. The problem with automatic exposure is that it normally uses the
shutterspeed to adjust the exposure of the frame. This could result in blurry
vehicles in a dark overcast scene. More research is needed in order to verify this
as a valid solution to the problem.
The algorithm does work in diﬀerent weather conditions, as long as the exposure
is set up so the frame is not overexposed or underexposed. It also works, without
any diﬃculties, at other locations than the primary location used for testing. It
handles diﬀerent kinds of FOV, resolutions and angles. This means that the user
does not have to be very careful when setting up the camera. The algorithm
can handle most of the situations, as long as the lane is ﬁlmed from above with
the middle line aligned approximately with the middle of the frame, and the
vehicles driving away from the camera.
7.7
Timing
When the algorithm runs on a video with a resolution of 640 x 360 or lower, it is
very fast. It could potentially run in realtime, which could save a lot of time for
==PAGE:129==
7.7 Timing
111
the user. The algorithm is however not setup to run in realtime. By decreasing
the resolution to 384 x 216 it is possible to record 50 fps and potentially do
realtime processing of the data, which is really impressive. Having such a low
resolution does not seem to aﬀect the counting, but could have an eﬀect on the
velocity estimate. Having a high framerate is good if the vehicles are moving
fast, or if the user wants a more detailed view of the velocity in the region.
Further testing is needed in order to verify potential consequences of reducing
resolution. Due to the limited time of this project, this can not been done.

Chapter 8
Conclusion
Choosing diﬀerent ﬁelds of view, orientations, positions, and resolutions of the
camera, can help solve diﬀerent traﬃc data gathering problems. In this thesis,
traﬃc counting and velocity estimation in congested traﬃc has been the main
focus. For this purpose, a narrow ﬁeld of view has given the best results. A low
resolution video is used to increase the runtime per frame. With a resolution of
640 x 360, it would be possible to run the algorithm in realtime. The average
runtime per frame at this resolution is 0.0322 seconds, which is less than the
times it takes before the next frame arrives when recording at 23.976 fps. If the
video is recorded at 50 fps, the resolution needs to be lowered to 384 x 216 in
order for the algorithm to be able to run fast enough to run realtime. At this
resolution the average algorithm runtime per frame is 0.0197 seconds.
The counting part of the algorithm has an accuracy of 98.08% meaning that
most of the vehicles will be counted correctly.
This is within an accepted
region for making traﬃc models and getting a relatively accurate estimate of
the traﬃc ﬂow. The counting sometimes fails on trucks with a homogeneous
surface. This is due to the edge detection method being used as a basis for
the algorithm. Edge detection proved to be the best solution compared to the
typical segmentation solutions; median ﬁlter, Gaussian Mixture Models and
optical ﬂow. The typical methods all had diﬃculties with static vehicles and
changing lighting conditions. The edge detector is more robust towards changing
light conditions. It is however sensitive towards over- or underexposed frames,
==PAGE:132==
114
Conclusion
which can make it diﬃcult to set the exposure correctly on the camera. In sunny
weather, shadows become very clear and will be detected by the edge detector.
To solve this problem a dynamic edge detector has been created, which changes
the edge threshold (that decides how large edges are used), depending on the
amount of light in the frame. The problem with over- and underexposed frames
could be solved by estimating the light intensity with a luxmeter, and not the
frame - which is done in this thesis. By doing this, it could be possible to use
automatic exposure settings on the camera.
The velocity estimation is estimated to have an accuracy within 10 km/h of
the actual velocities of the vehicles. This is not very accurate, but is accurate
enough to give an idea of the general velocity of the vehicles. This is still useful
for making traﬃc models, where velocity accuracy is not essential. A bonus of
the velocity estimation algorithm created in this thesis, is that it is able to give
a velocity estimate from one frame to another, making it possible to describe
whether a vehicle is slowing down, speeding up, stopping, or starting within the
measurement region. This could make it possible to create more complex traﬃc
models of congested traﬃc. The velocity estimation is based on tracking image
features from one frame to another and doing a homography transformation to
estimate the distance the vehicle moves from one frame to another. A distance
on the road needs to be known in order to do this. In this thesis the length of
the white middle line and width of the road is used to create this homography
transformation.
Type estimation works by looking at the sum graph created by the counting
part. By ﬁnding the largest peaks, and estimating the vehicle area, it is possible
to classify vehicles in most situations.
Vans create a problem because they
can have the same number of peaks and area as most of the vehicles, and
in other situations the same number of peaks and area as caravans.
Due
to time limitations, a simple solution to this problem has been to create 6
groups: motorcycle, car, car/van, van, Caravan/van, truck/trailer. By having
overlapping classiﬁcations it is possible to get a 93% accuracy of this method.
The algorithm is very robust and has been tested at more than one location.
It is not very sensitive towards shadows, changing the position of the camera
slightly, wind shaking the camera etc. This means that it is relatively easy to
setup and use.
==PAGE:133==
Chapter 9
Future Work
The algorithm is currently dependent on the exposure of the camera being ﬁxed.
It is used to estimate the light intensity of the scene. This does however create
a problem when the light conditions change dramatically. This could result in
an over- or underexposed scene, which will give a poor result. By measuring
the light intensity with a luxmeter, and logging the data, it could be possible
to set the camera to automatic exposure. It would then automatically adjust
according to light intensity changes.
This is typically done by changing the
shutterspeed of the video. This could result in blurry vehicles which in return
would decrease performance. Further research in this area needs to be made in
order to ﬁnd a good solution.
One of the problems in the algorithm, is when vehicles enter the frame between
the two lanes. The current version of the algorithm will count this as two vehicles
entering the frame, which decreases the accuracy of the counting algorithm. A
method should be developed to look for this speciﬁc situation and only count it
as one vehicle in one of the lanes.
Further development needs to be done to improve the velocity estimation. The
biggest issue is that a 3D object is being projected to a 2D surface which does
create some inaccuracies, depending on where the feature points are found.
This could be solved by taking height and 3D structure of the vehicle into
consideration.
==PAGE:134==
116
Future Work
Testing of resolutions needs to be done in order to estimate how low a resolution
can be used without aﬀecting the results. Using a low resolution will decrease
the runtime of the algorithm by limiting the number of calculations.
The algorithm could also be adapted to track vehicles for a longer distance. The
current algorithm is only able to track one vehicle in each lane. This would mean
that a diﬀerent track initialization with more trackers needs to be generated.
