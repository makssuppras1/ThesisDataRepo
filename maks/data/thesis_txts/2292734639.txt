Summary
The goal of this thesis is to propose a Software-Deﬁned Network based solution
to the network recovery problem. This report describes the design and prototype
implementation of the service that solves the above stated issue. The service
is extended with a routing optimization functionality and oﬀers compatibility
with ﬁrewall and load balancing rules. The service was based on OpenFlow
protocol and implemented as part of ONOS controller. This report contains the
evaluation results, where Mininet and Open vSwitch were used for emulation.
Tests have shown fair performance with single interface failure.

Chapter 1
Introduction
In the midst of the Information Age, there is the Internet.
It’s exceptional
growth over last decades has changed the way we live and work. Today, our
daily life habits include using multiple devices constantly connected with the
world. This results in a signiﬁcant need for network infrastructure resources
and it’s functions. Ever growing demands increase the complexity in solution
domain, mainly because the underlying network architecture has never been re-
visited. The original principles were designed decades ago, and although proven
successful, they are not free from demerits, for example in security or mobility
[Fel07]. Another drawback of the current networking landscape is the high bar-
rier to enter into the industry, practically blocking rapid innovation [TT05] and
creating vendor lockout [BBL+13].
The concept of Software-Deﬁned Networking (SDN) 2.1 has recently gained a
lot of attention. The paradigm can meet the urge for revising current Inter-
net architecture. Because of its proposal of programming the network from a
higher abstraction level, it can speed up innovation and simplify a lot of prob-
lems present in traditional networks. The most widespread implementation of
SDN, the OpenFlow protocol, 2.1 has remarkably changed the networking land-
scape in the past few years. Thanks to the interest of big vendors, who noticed
the beneﬁts of SDN, it is on the way to become a widely accepted standard.
Although this disruption brings many beneﬁts, it also has some major chal-
lenges that keep network operators reluctant to participate. Moving all control

traditional networks, hence these services have to be developed from scratch.
This thesis deals with the problem of recovery in Software-Deﬁned Networks.

Chapter 2
Background
2.1
Software Deﬁned Networks and OpenFlow
In many cases the traditional network architectures are not well-suited to fulﬁll
the needs of enterprises and end users. Software-Deﬁned Networking (SDN),
[Fou12a], [LKR13], is an approach to design, build and manage networks, and
it is transforming the traditional networking architecture. SDN separates the
network’s control and data planes. The control intelligence is placed in a logi-
cally centralised controller, and the underlying infrastructure is responsible for
simple forwarding decisions only. SDN promises to reduce CAPEX and OPEX,
deliver agility, ﬂexibility and enable innovation, since the software upgrade cy-
cles are decoupled from hardware cycles, [SDN]. SDN architecture, ﬁgure 2.1,
consists of controller, southbound API and northbound API. Controllers oﬀer
a global view of the network, and let operators to instruct the underlying in-
frastructure from a single, centralised point. The southbound APIs specify the
scheme in which the controllers communicate forwarding rules to the forward-
ing element. The northbound APIs are the way controllers interact with the
network administrators.
SDN has already gained a lot of attention commercially both from the business
and the technical perspective. For example VMware has acquired Nicira for over
$1.05 billion, [Nic], in order to expand its business portfolio. A technical example

could be illustrated by Google’s B4, [Hol12], a globally deployed software deﬁned
WAN, which has improved the utilisation of Google’s internal network.
The most deployed SDN protocol, OpenFlow [MAB+08], realises the commu-
nication between the switches and a controller placed in dedicated hardware.
It has been deployed in large scale experimental networks in the US (GENI)
[fNIG], Japan (JGN2plus) [rdtn], and in the EU [iELIO]. However, OpenFlow
starts to appear in operator production environments as well. OpenFlow is not
synonymous with SDN, but is an element of SDNs. Additionally, OpenFlow is
not the only such protocol available for SDN.
An OpenFlow switch is a set of input and output ports and a FlowTable (a set
of rules). The FlowTable is an abstraction of the Forwarding Information Base
(FIB). The protocol provides the rules to program FIBs from a remote controller
by adding or deleting entries. Those entries consist of a header, deﬁning the ﬂow,
action, deﬁning the output port (and possible modiﬁcations of the packet), and
statistics, which keep track of the processed packets. A sequence of packets
sharing certain header ﬁeld values will be treated in the same way (forwarded
to a certain port or dropped). Unlike in the traditional networks, which match
only destination IP address, an OpenFlow switch can match packets against an
ingress port, MAC or IP source and destination address. If the rule is not in the
FlowTable, the switch queries the controller which decides how to handle it and
further instructs the switch, which stores this rule in its FlowTable. OpenFlow
has an on-demand architecture which means that the rules are not proactively
installed in switches and the ﬂow. The above described process is showed in
ﬁgure 2.2.

Chapter 3
Resiliency in an OpenFlow
Network
Networks have become a critical component of business applications and are
expected to be available and reliable at all times. Prolonged service disruptions,
e.g. at a stock exchange, can cause signiﬁcant damage. As stated in [Com02], for
continuous media applications (carrier-grade Voice over IP), VPNs and for other
real-time business applications, it is only tolerable if a network is unavailable for
a maximum of about ﬁve minutes during the course of the entire year. Since the
Internet has become more and more complex and fast-paced, keeping failures in
check is a key issue for network operators. However, research shows that despite
all eﬀorts, failures are frequent events. As observed in [ICM+02], during the
period of ﬁve months from April 2002 until August 2002, failures spread rather
uniformly and frequently. Their duration (time until returning to primary path)
varies from less than one minute (around 50%) to less than ten minutes (around
85%). As per the above, dealing with the availability is a part of every day
operations and recovery mechanisms have to be carefully studied and optimized.
There are various reasons for failures, such as transient router interface faults
or crashes, ﬁber cuts, software bugs or conﬁguration mistakes. Lack or misuse
of fault tolerance mechanisms can obstruct normal traﬃc forwarding, even if
there is an alternate path in the network. Resilience is a key and well-satisﬁed
requirement in IP/MPLS networks [VPD04]. Unfortunately, it is not an inherent
feature of the SDN architecture. Automatic reconﬁgurability of traﬃc paths, in
==PAGE:20==
10
Resiliency in an OpenFlow Network
case of link or node failures, is not easily available. Upon failure, the controller
does not perform any extraordinary action by default.
As stated in [NJBB+09], the desired interval for failure recovery is below 50
milliseconds. This chapter presents an overview of network resiliency techniques
with respect to this requirement. Section 3.1 deals with resilience in traditional
IP networks. Section 3.2 studies recovery and section 3.3 detection mechanisms
in SDNs. Following the introduction of group tables in section 3.4, the chapter
closes with a literature review in section 3.5.
3.1
Resiliency in Traditional IP Networks
Before studying resilience in SDN, it is prudent to review already deployed, well
tested approaches that can be found in the IP Networks and try to transfer
those into OpenFlow.
Eﬀectively dealing with failures in IP networks is a major operational challenge
for Internet Service Providers. Frequently deployed link state routing protocols,
such as the most popular OSPF [Moy98] or ISIS [PSS08], react to link fail-
ures through assessing neighbour reachability information to all other routers.
This information is generated based on network wide link state advertisements.
Upon failure, each node recomputes its routing tables using the updated topol-
ogy information. Default settings of the OSPF parameters result in signiﬁcant
forwarding discontinuity after occurrence of a failure. Studies such as [AC02],
[AJY02] and [NJBB+09] show that this discontinuity typically takes several
seconds. During this period, other nodes that are not aware of the failure may
continue to route packets towards the failed link resulting in a loss. Considering
current high bandwidth networks, even short periods of discontinuity will cause
high packet drop rate.
Fault tolerance consists of four steps ([CCX+15]), as presented on ﬁgure 3.1.
Those steps are: failure detection, failure announcement, route recomputing,
and routing table update. Lowering recovery time essentially means decreasing
the time spent on each step. Reducing the transmit interval of probing packets
[BR01] or combining with other methods such as Bidirectional Forwarding De-
tection (3.3.1) can cut down the failure detection time. With transient failure,
an adaptive timer [BFF07] will decrease announcement delay. For recompila-
tion of routes, routing algorithms can be optimized. In order to further decrease
convergence time, the proactive technology is proposed, which reduces route re-
computation delay to zero.

3.1.1
Reactive and Proactive Routing
Much work has been done to improve failure handling in IP networks.
The
outcomes can be classiﬁed into two categories: reactive and proactive routing,
[LYNZ04]. Generally, reactive approaches recompute a new routing informa-
tion table after two initial steps presented in ﬁgure 3.1. When a failure occurs,
additional signaling is needed to establish the restoration path. Proactive ap-
proaches calculate detours and react locally to link failures as soon as a fail-
ure is detected, based on pre-computed set of rules and pre-booked resources
to forward traﬃc.
Reactive approaches may not provide suﬃcient resilience
due to the time required to complete the convergence. Hence, proactive ap-
proaches have become the main solution to the resiliency issue. Depending on
the scale of malfunction, the proactive mechanism can be divided into link-based
and topology-based ([CCX+15]), aiming at a single or multiple failures respec-
tively. Link-based methods assume each node functions independently and is
composed of both forwarding and control planes. These techniques focus on
protecting certain links or devices e.g. Loop-Free Alternates (LFA) [AZ08] or
U-turn [Atl06]. Topology-based approaches will calculate a few logical backup
topologies, [PSS08], [PMR+07], [WMWZ12], which can help with protecting the
entire network.
Summing up, the proactive technology performs better as far as recovery time
is concerned due to the preliminary work being done before failures happen.
However, these solutions will spend extra network resources which cannot be
neglected, especially in a congested network. Additionally, with multiple points
of failure, proactive approaches may cause problems (for example forwarding
loops).
Hence, a common practice is to combine reactive (restoration) and
proactive (protection) mechanisms.
3.2
Fault Recovery in SDN
Software-Deﬁned Networking, and in particular OpenFlow, simpliﬁes managing
networks and orchestrating a dynamic network conﬁguration. However, its ini-
==PAGE:22==
12
Resiliency in an OpenFlow Network
tial design does not possess the ability of self-healing. Despite existing research
(3.5), recovering from network failures in a timely manner remains complex.
Alike traditional networks, SDN networks must adapt to failures by directing
traﬃc over functioning alternative paths. This functionality must be provided
within the control plane fault tolerance modules as there is almost no intelli-
gence on the data plane. Additionally, the software logic varies across controller
implementations.
Figure 3.2: Fault recovery process in SDN: (1) failure occurs; (2) node de-
tects failure; (3) node communicates failure to the controller with
PORT_DOWN message; (4) information is processed; (5) controller
replies with OFPFC_MODIFY message; (6) node updates ﬂow tables.
In general, SDN’s fault tolerance spreads across both the data and control
planes. In practice, it is achieved in the following manner. Although the docu-
mentation does not specify topology discovery in OpenFlow, most software e.g.
ONOS, commonly use Link-Layer Discovery Protocol, LLDP [LLD09]. When
the information is received by issuing periodical LLDP messages, OpenFlow con-
troller is able to generate current network view and ﬂow tables for each node.
As presented on ﬁgure 3.2, fault recovery in SDN consists of four phases: de-
tecting the failure, communicating it to the controller (3.3), re-computation of
the new shortest path, and conﬁguration of ﬂow tables on OpenFlow forwarder
to restore or maintain end-to-end connectivity. In the re-computation phase,
the controller usually runs Dijkstra’s algorithm on the updated topology infor-
mation. Later, it modiﬁes each aﬀected node’s ﬂow table using OFPFC_MODIFY
messages. In summary, as nodes make only simple forwarding decisions based
on ﬂow table entries, they can only recover from failures if a new, correct ﬂow
entry is added in their memory.
As one may notice, since the paths are not pre-computed, described solution falls
under the deﬁnition of the restoration method. Normally, controllers are not
equipped with protection modules. Some authors have suggested pre-installing
([SGC+13] or [SSC+13b]) ﬂow entries on a switch with diﬀerent priorities or
generating protection paths on the controller ([KPV+13]) and reacting without

Apart from data plane failures (switch or link), there are two other fault domains
in SDN-based networks. First, the control plane, where the connection between
the controller and switch fails. Second, on the controller itself, where the con-
trolling machine fails. This project focuses on the data plane type of failures.
However, the fault tolerance of the control plane should be of concern as well.
Disconnect between switches and the controller can disable the whole forwarding
plane. With multiple controllers, the control plane is more stable. By design, an
ONOS controller is meant to operate in distributed state, which was one of the
reasons for using ONOS for this project 2.2. When distributed controllers are
applied, consistency and synchronisation needs to be assured [LWHH12]. When
an OpenFlow channel between controller and switch is faulty, a diﬀerent path
via neighbours can be used to reach the controller [SSC+13a].
Summing up, by design, SDN is unable to sustain desired availability when
facing failure and dedicated recovery mechanism modules have to be designed.
These modules have to implement protection mechanisms, be able to recover in
maximum 50 milliseconds, and handle multiple failures.

3.3
Fault Detection in SDN
Before a switch can begin path recovery, a failure must be detected. For most
controllers (e.g ONOS) both topology discovery and failure detection relies on
Link Layer Discovery Protocol, LLDP [LLD09]. Detection programme deployed
at the controller sends and receives probe messages from all of the nodes and it
is transferred to the appropriate module. To access this information in ONOS,
one needs to call TopologyService. This information, containing link, interface
and node state, is used to generate network-wide view and the ﬂow table entries
for each OpenFlow node.
LLDP can detect link or node failures within sub-seconds, but it is not able to
monitor the health of links within the required 50 milliseconds recovery time.
The centralised monitoring model has scalability limitations because the con-
troller has to process of all the probe messages. To meet the time requirement,
it needs to run with approximately 100Hz frequency [KBK+09]. This means
that the controller must receive and process around 100 packets per second per
link, i.e. react to millions of packets per second just to monitor the network’s
health. This is not just a burden on the controller; it also puts an unacceptable
load on the network itself.
OpenFlow speciﬁcation [Fou12b] expects the switch to inform the controller
when ports are added, modiﬁed or removed from a data path due to such events
as port being brought down by the user or the link going down.
The node
sends this information in an OFPT_PORT_STATUS message. Hence, the detection
program needs to be deployed at every forwarding node. Mechanisms like LOS,
OSPF [Moy98], STP or Rapid STP [LH05] are commonly used to update port
statuses in switches. These protocols periodically exchange messages to main-
tain the adjacency. If a detection program does not receive a message from its
neighbour within the interval, it assumes the link between itself and the neigh-
bour failed and ﬂoods this message to the network. Once the controller receives
the message about the failed link, it updates the network state database and
trigger the backup path computing. These protocols, however, can be classiﬁed
as slow, as detection windows are into seconds.

end-points. Nodes transmit messages with the current state of the monitored
path. A node that receives such message replies with an echo containing respec-

3.4
Fast Failover
Figure 3.4: Use of Failover Group Table upon packet processing. Green are
used, and red are not matched entries in ﬂow and group tables.
Grey bucket is not active since port 1 is down.
From OpenFlow version 1.1 onwards, a new mechanism of Group Table has
been proposed. It allows advanced forwarding and monitoring on the switch
level. The Fast Failover Group type of Group Table can specify working and
protection path in multiple action buckets.
With the appropriate hardware
support (failover type is optional in OpenFlow switches), these paths can be pre-
installed, allowing nodes to quickly and locally react to failures without the need
of contacting the central controller. This is obtained by creating multiple action
buckets for the same ﬂow entry and selecting them according to the status of
ports and interfaces. The idea behind Fast Failover Group is presented in ﬁgure
3.4. Firstly, the switch performs standard match action with ﬂow entries. Unlike
before, output action can be speciﬁed as a pointer to a speciﬁc group. Group
entry consists of one or more action buckets with a deﬁned order. The ﬁrst
action bucket describes what to do with the packet during standard operation.
If this bucket is declared unavailable, the packets are treated according to the
second action bucket. The process continues until a group runs out of deﬁned
buckets. The status of the buckets can depend on the availability of a physical
port or on status of other group. This makes it easy to associate this monitoring
capability to the local protection in the switches.

3.5
Literature Review
Since the introduction of OpenFlow, extensive research in the area of resilience
has been conducted. Some of the research work, e.g. [BZ12], focuses on the crit-
ical forwarder-to-controller connection. As the authors point out, these links are
the most important part of the SDN network. They propose algorithms to spec-
ify the best location to place the controller. Other work, performed by Sharma
et al. [SSC+11], focused on data path recovery. As showed in the publication,
controller-initiated recovery needs approximately 100 milliseconds to regain net-
work connectivity. Authors have proposed a fast restoration mechanism in which
forwarding rules are statically deﬁned by the network administrator. However,
this raises the risk of problems like forwarding loops or policy inconsistency as
a result of misconﬁguration. This proposition is obviously against SDN’s key
concepts.
Automatic Failure Recovery for OpenFlow (AFRO) [KPV+13] looked at the
problem diﬀerently. As the backup logic ultimately lies in the controller, the
crucial requirement has to be portability. As the list of controller software is
long, the possibility to run recovery software on any controller is desirable. The
proposal decouples failure recovery and controller logic. It is done by emulating
a shadow controller to run the same software as the main controller, which sees
part of the network without the failed element. Hence, it can hold information
about backup paths upfront. As failure occurs, the controller uses this infor-
mation to reply to failure events. Route re-computation time per each failure
is decreased to zero. Additionally, such an approach reduces risk of forwarding
loops in the network [PKV+13].
Another interesting setup is CORONET [KST+12], which tackles the problem
of scalability.
It uses proactive approaches with routing paths computed by
the VLAN growing algorithm creating multiple link-disjoint paths using Dijk-
stra’s shortest path algorithm. However, this work, as many others, address
the problem without taking into account the dynamics of network traﬃc. On
the contrary, [AGPS14] considers variable costs of the links depending on the
current utilisation, estimated in real-time by the controller exploiting OpenFlow
counters. Traﬃc paths are computed accordingly, leading to optimized distri-
bution of ﬂows over the network. It is by using a proactive approach that the
authors successfully meet the 50 milliseconds requirement. This time can be
further decreased using Fast Failover Groups, 3.4.
The authors of [KBK+09] made use of this mechanism.
This way, switches
can react to failure locally. This paper focused on end-to-end tunnel protection
that averaged at 28 milliseconds. However, the proposal included alternative
OpenFlow switch design introducing logical group ports.
==PAGE:27==
3.5 Literature Review

Depending on network topology and the failure, local detours may not be avail-
able or they may be unoptimized from the resource allocation point of view.
[CCNS15] introduces OpenState, which extends data plane abstraction with
the possibility for switches to make states evolve. It enhances the fast-failover
paradigm to act globally. The idea behind OpenState is that control tasks that
require only local knowledge are not handled at the controller. They can be
moved to switches while leaving control for the tasks that require global knowl-
edge.
As per detection of failure, Yang et al. [YSXY11] proposed a solution at the
early stage of OpenFlow which focused on transient link failure. They argued
that the detection of failures will be much faster in such a network, as the
exchange of heart-beat messages will have much higher frequency and the con-
troller will instruct the forwarder what to do much faster. However, although
their packet loss veriﬁcation was much of an improvement, the 300-400ms link
failure detection time was not satisfactory. [vAvAK14] focused on detection as
well and managed to achieve 3.3ms average recovery time. Authors introduced
a failover scheme with per-link BFD (3.3.1) sessions.
At the moment, most papers in this area are focusing on solving a single link
failure. As a result, the solutions for multiple link failures remain a challenge.

Chapter 4
Routing Optimization in an
OpenFlow Network
Traditional networks were designed to deliver packets via the shortest route
available in the network. Such an approach limits the possibility of optimiza-
tion, thus improving overall user experience. In order to adjust to certain use
cases, ISPs had to statically conﬁgure the network. This way the network band-
width and latency requirements of some services were met. This caused both
CAPEX and OPEX increase and often lead to the introduction of conﬁgura-
tion errors, [JPA+13]. SDN features, especially very fast path re-computation
in comparison to manual re-conﬁguration, proved there is plenty of room to
improve the network performance [Hua15] with the use of the existing infras-
tructure instead of increasing the throughput or packet forwarding capacity of
network devices. Besides, [AKL13] showed that even incremental deployment of
SDN enabled switches (where they co-exist with traditional routers) can improve
overall network performance.
Unlike traditional networks, SDN opens up new possibilities for adaptive rout-
ing, instead of static allocation, improving eﬃciency and utilisation of network
resources. The controller sets rules based on a global view and optimizes net-
work resources and time allocation for current services. In general, it is expected
to increase throughput, reduce delay, reduce the number of conﬁguration er-
rors, support Bandwidth on Demand (BoD) adjust resource allocation based on
short-term and long-term predictions, and reduce CAPEX and OPEX, [NTT13],
==PAGE:30==
20
Routing Optimization in an OpenFlow Network
[Bro15], [Tu13]. These expectations have been conﬁrmed by Google’s announce-
ment [Hol12], which states an improvement in Google’s internal network utili-
sation. Also, highly expected is the application aware forwarding, [Vel13]. As
diﬀerent applications have diﬀerent requirements, a controller can make deci-
sions based on the content. For example, a client watching videos needs more
network bandwidth than one who is browsing through Internet pages; online
games need lower network latency than FTP services.

Most common path computation protocols are based on the calculation of the
shortest path, [Rex05], i.e.
the path with the least amount of intermediate
nodes. Such an example is marked in red in ﬁgure 4.1. The easiest yet very
eﬃcient modiﬁcation of this approach is introduction of weights for each link,
i.e. Constrained Shortest Path [Zie10]. Each link can be assigned a value cor-
responding to its throughput, delay, loss, etc. In OpenFlow, situations where
weight can be calculated based on data acquired by available meters open the
possibility of automated adaptive forwarding, [vADK14]. Also, the weights can
be represented by a matrix, and the ﬁnal value depends on the application type.
Traﬃc can be divided by purpose and weights may vary across diﬀerent services.
For example, if the link loss is above acceptable level for critical applications,
it can be assigned a value of inﬁnity, excluding it when calculating optimized
path. On the other hand, for not so demanding applications, one can exclude
links speciﬁcally designed for critical services. In general, the weighted approach
is visible on ﬁgure 4.1. The above illustrated approach was tested in [JHLC14]
and had shown signiﬁcant improvement compared to a non-weighted approach.
It is also necessary to notice the diﬀerence between optimizing a path for one
ﬂow and for the whole network. In most cases, ISPs would like to distribute the
load in a way that the links are equally utilised. Apart from designing network
architecture to deal with this issue, there is room for deploying algorithms. Such
a problem can be referred to as a modiﬁcation of well-studied bottleneck, the
travelling salesmen problem [BDvD+82] (or related Hamiltonian cycle). Com-
==PAGE:31==
21
putational complexity of this problem is NP-hard and is not trivial to implement
in networks, especially when additional rules (ﬁrewall, QoS, BoD, etc.) and re-
quirements are present. However, SDN certainly provides means for automatic
conﬁguration and optimization based on information obtained from OpenFlow
switch meters.
Besides a path computation method, optimization can cover more sophisticated
approaches, such as multicasting [AJM14] or load balancing [YBJ15].

5.2
Failure Recovery
The core of the implemented service is failure recovery mechanisms which consist
of detection, protection and restoration.
For detection, the project is implementing per-link BFD, 3.3.1, session.
Al-
though BFD was designed as a per-tunnel protocol, in order to fully cover the
network, the number of sessions will be too high. The number of sessions be-
tween per-link and per-tunnel will only be equal if the network is a full-mesh
type with only one host per switch. Otherwise, the amount of per-link sessions
will be smaller. Not only is the amount of signaling a problem, but also a delay
added through transient transmission and issue of false positives due to net-
work congestion. A BFD control packet consists of maximum of 24 bytes. With
authentication and encapsulation it results in a maximum of 720 bits. If sent
once every 1 ms, this results in covering only 6.7% in 10 Mbps connection (from
[vAvAK14]).

After a node has detected a failure, the recovery process is split into two com-
plementary steps. To achieve higher availability and scalability of centralised
controller, the ﬁrst step is proactive recovery and it is based on switch-local
decisions. It consists of a switch using pre-conﬁgured backup paths in Open-
Flow’s Fast Failover Group Tables. This guarantees uninterrupted end-to-end
connectivity. Each bucket in Fast Failover Group monitors port that is also its
output port. Therefore, when link functionality of the primary output port fails,
the Group Table will automatically turn to the secondary set of output actions.
Consecutive buckets link to connective, most-optimized path.
As soon as ﬁrst packet arrives, it is captured with PacketProcessor. This mod-
ule invokes FailoverGroupsGenerator to compute backup paths for a various
failure scenarios, including multiple failure. Also, the rules are installed not
on a per-switch basis but all at once through installRule; meaning that the
controller schedules an entire route to the destination rather than only one next
hop. This signiﬁcantly speeds up rule placement since the algorithm has to be
run only once rather than repeatedly for each intermediate switch. Also, it is
solving the rule installation bug, presented in ﬁgure 3.3.

5.3
Sequential Composition
SDN controller modules perform a number of tasks on the same traﬃc. The
problem of composing diﬀerent rules that guarantee not to interfere with one
another is not a trivial task, [Fea]. These hovering modules can cause inconsis-
tencies or performance and space related ineﬃciencies. For example, the same
traﬃc that is monitored can also be load balanced. If the application is not
modularised, but monolithic, it is going to be more diﬃcult to debug, test and
port. In order to combine the rules, applications apply a parallel (simultane-
ous) or sequential (one after another) composition. The most common software
implementing rule composition is Pyretic, [RMF+13], which generates rules for
POX controller. Unfortunately there is no such software for ONOS.
This project uses the OpenFlow pipeline concept and diﬀerent group types. Ev-
ery OpenFlow switch is required to contain at least one ﬂow table with multiple
ﬂow entries. The pipeline deﬁnes how packets are processed. The ﬂow tables on
the switch are numbered sequentially starting from 0. If the processed packet
does not match any ﬁeld in table 0 it is a table miss which determines the
following behaviour. A packet can be processed by the subsequent table. As
presented on ﬁgure 5.3 (based on [Mah] and [Fou12b]) each table can focus on
diﬀerent areas of packet processing.

not specify which one). Although not required, it is suggested that this type of
group is watching an output action port or group and in case of a port or group
going down the switch restricts bucket selection to the remaining set meeting
failover requirements. Thus, for implementing load balancing rules it is enough
to change the type of group from Failover to Select.

6.4
Discussion
Results presented in section 6.1 showed satisfactory results. With a single failure
for simple topology and an average of 4.4 packets were dropped. Since the packet
generation frequency was 10kHz, the recovery time lead to an average 0.44 ms.
Taking in to account pktgen uncertainty, ﬁnal result shall be noted as 0.44 ±
0.005 ms. Similarly, when crankback routing is needed the observed recovery
time was 0.46 ± 0.005 ms. This certainly meets the stated demand of the
maximum 50 milliseconds.
On the contrary, the results for multiple interfaces going down at the same
time is not meeting this requirement. Scenario 3 showed 122.82 ± 0.005 ms
of recovery time. Thankfully, the probability of such occurrences in the real
network is low.
Scenario 4 showed, that even in a congested network, the recovery time is suﬃ-
cient. For link failures close to the source i.e. Stockton-Seattle and Washington-
Stockton is below 50 ms. The result is unacceptable for Atlanta-Washington,
exposing the necessity of optimization of algorithms. Such optimization is shown
in Scenario 7. With the simple use of bandwidth information, algorithm adjusted
to avoid bottlenecks in the topology.

Chapter 7
Conclusions
This thesis dealt with the problem of detection and recovery from failure. Re-
sults for the prototype service proved that it is possible for an OpenFlow network
to recover within the 50 ms time-frame using the proposed design. For detec-
tion, a per-link BFD session was implemented. The protection was realised in
a form of Fast Failover Group rules pre-installed on the switches. The recovery
time is independent of the network size. Despite fulﬁlling the demands for the
recovery time for single interface failure, the application failed to meet those
demands for double interface failure within a single switch.
The thesis extended a proposed algorithm to comply with path optimization
and additional ﬁrewall and load balancing rules.
7.1
Future Work
If one would try to include this thesis service within oﬃcial ONOS sample
application repository there are three main areas of improvement. First, the
memory management shall be of greater concern. The amount of ﬂow and group
rules shall be decreased using wildcards. Also the group management would be
an elegant way to minimise switch memory requirements. Secondly, a revision
of the implemented bucket generation algorithm to remove recursion should be
==PAGE:52==

implemented due to performance enhancement. Finally, the application lacks
test classes making it unacceptable for the oﬃcial channels.
When extending the service to compose with other networking rules, ONOS
controller was clearly missing a way to specify such rules in a simple way, simi-
larly to what Pyretic [RMF+13] does for the POX controller. It is suggested to
further examine possibility of implementing this feature.
Furthermore, as a consequence of scenario 4, one should study the implementa-
tion of Failover Group in Open vSwitch and determine which is the reason for
having a large packet loss when double failure occurs. Also, regarding scenario
6, it would be beneﬁcial to implement alternative Select Group bucket selection
in order to implement load balancing in a desired way.
Finally, targeting switch memory management issues, it is suggested to extend
OpenFlow protocol to support group timeouts. Initially, this project was in-
tended to implement timeouts in order to wipe out inactive entries. However

Appendix B
OpenFlow Extension Draft
B.1
Group Tables in OpenFlow
OpenFlow ﬂow entries main task is to match incoming traﬃc with a speciﬁc ac-
tion. From version 1.1 the action could be either output interface or a group ac-
tion. When controller issues OFPT_FLOW_MOD message with command OFPFC_ADD
a new ﬂow is added to the switch forwarding table. This packet makes it possi-
ble to include two parameters; hard_timeout and idle_timeout, which we later
describe in section B.2, and specifying ﬁeld out_port to directly forward traﬃc
or out_group to use in group action.
The purpose of the group action is to further process those packets. There are
four types of groups: indirect, all, failover and select. The OFPT_GROUP_MOD
packet coordinates group creation and modiﬁcation.
B.2
Problem description
Each ﬂow entry has an hard_timeout and idle_timeout ﬁelds associated with it.
A non-zero hard_timeout ﬁeld makes the ﬂow entry disappear from the switch
==PAGE:56==
46
OpenFlow Extension Draft
memory after the given number of seconds, regardless of how many packets it
has matched. A non-zero idle_timeout ﬁeld speciﬁes the ﬂow entry removal
time when it has matched no packets in the given number of seconds.
Timeouts are very useful in OpenFlow switch memory management since the
ﬂow table will be reduced after those timeouts expire.
OFPFC_REMOVE could
be issued for all the ﬂows by controller, however this creates, in many cases,
unnecessary complexity on controller applications.
When the switch performed ﬂow removal of the entry, with group as an output,
the process does not include group removal. If the group (liked to speciﬁc ﬂow)
is not intended to be permanent two main problems arise:
1. when removal is initiated by OFPFC_REMOVE command, the controller has
to remember group linked to the ﬂows and follow with OFPGC_REMOVE
message that contains speciﬁc group_id information;
2. when removal is initiated by exceeding timeout, it requires the controller
to set its own timers to follow with OFPGC_REMOVE message at speciﬁc time;
When this complexity is not added on controller services, and group actions are
generated in a reactive way, the list can grow rapidly as some groups may be
left as non-controlled orphans. On the other hand if timers on the controller are
not aligned, premature deletion of a group may cause OFPBAC_BAD_OUT_GROUP
error.
B.3
Proposed Extension
Depending on feasibility of the extension, we propose two solutions to the prob-
lem. Both will result in simplifying controller services and leave the possibility
to make a group permanent.
1. implementing hard_timeout and idle_timeout in struct ofp_group_mod,
similar to the ones in struct ofp_flow_mod;
2. assign OFPGC_CLEAN = 4 in enum ofp_group_mod_command that instructs
switch to perform garbage collection; if no ﬂow entries output to the group
in question (speciﬁed in group_id ﬁeld) the group is removed.
When
group_id equals to OFPG_ALL, switch is instructed to follow this procedure
for all the groups installed in the memory;
