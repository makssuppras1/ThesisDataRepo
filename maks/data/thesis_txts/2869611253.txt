Abstract
With developments in automotive systems there is an increasing demand for modern
in-car networking with simpler wiring, better throughput and increased security.
This thesis addresses the challenge of developing a logic-efficient MACsec module for
10BASE-T1S Ethernet. 10BASE-T1S is a multi-drop shared medium Ethernet variant.
The implementation has a focus on logic efficiency while achieving multi-point security
with MACsec. The focus on logic efficiency has led to the development of custom
modules that follow the specifications of AES and GCM to form the default Cipher
Suite required by MACsec.
The implementation of AES achieved a significant logic reduction while maintaining
throughput with plenty of headroom for the requirements of 10BASE-T1S.
The work identifies key strategies for optimizing Galois field multiplication in the
GCM module to improve both logic utilization and maximum clock frequency.
Implementation of modules to support MACsec operation, such as buffering and
efficient memory lookup of Secure Associations were successful in demonstrating the
viability of multiple secure channels. Validating core components, the work establishes
specific strategies for achieving full MACsec operation at a high logic-efficiency.

Chapter 1
Introduction
Modern automobiles are becoming more complex and rely on an increasingly large
number of sensors and connected devices. A traditional bus system such as CAN
(Controller Area Network) requires complex wiring and has low bandwidth.[1]
In addition, CAN was not designed with security in mind. The complexity of the
systems in modern vehicles exposes a large attack surface where vulnerabilites can
lead to theft of cars, tampering leading to catastrophic accidents, and unauthorized
access to private consumer data through unencrypted bus communication. [2]
Automotive Ethernet is the next generation of technology for in-car communication.
Automotive Ethernet brings solutions to the problems that plague traditional bus
communications in the current and next generation of vehicle architecture. [3]
10BASE-T1S is an Ethernet standard that enables 10 Mbps data transmission between
(at least) 8 nodes on a single pair shared medium. This can reduce wiring complexity
significantly. [4]
Since 10BASE-T1S is an Ethernet standard, data transmitted on this interface can be
protected using the MACsec standard. MACsec is a network security standard that
enables both encryption and authentication of Ethernet networking at the Medium
Access Control (MAC) layer. [5]
This thesis explores the development of a logic-efficient MACsec implementation
specifically for 10BASE-T1S based automotive applications. That includes ground-
up development of a cipher suite and a MACsec module as specified in the IEEE
802.1AE-2018 standard. [5]
The intention for the developed module is integration in a larger integrated circuit for
Ethernet MAC for 10BASE-T1S.
The requirements for this MACsec implementation that set it apart from existing
design are focus on an efficient implementation with a low number of logic gates and
supporting multiple secure channels to at least 8 nodes in total at a speed of 10 Mbps,
full duplex.

Chapter 2
Theory
2.1
10BASE-T1S Ethernet
The 10BASE-T1S Ethernet standard supports 10 Mbps over a single twisted pair. It also
supports multi-drop architecture, meaning multiple Ethernet nodes can be connected
to the same single pair wire. The standard specifies that at minimum least eight
transceiver nodes can be connected on a single-pair wire with a length up to 25 meters.
For uses such as next-gen zonal automotive architectures this can enable simplified
wiring that uses less physical material.
2.2
MACsec
MACsec, also known as IEEE 802.1AE, provides network security at the MAC layer. It
can provide encryption and/or authentication of transmitted data.
MACsec is applied to an Ethernet frame.
MAC Security Entities exchange a MACsec Protocol Data Unit (MPDU). In an Eth-
ernet frame an MPDU starts from the Ethertype and ends with the Integrity Check
Value (ICV), which is an authentication tag. An MPDU contains a MAC Security Tag
(SecTAG), secure data and an ICV. (See figure 2.1)
The SecTAG contains the following information: [5]
• MACsec Ethertype, always equal to hexadecimal "88-E5"
• Tag Control Information (TCI), contains information on SecTAG formatting such
as SCI encoding, whether encryption or authentication only is in effect etc.
• Association Number, identifies up to four Secure Associations within a Secure
Channel. This is used for switching secure keys.
• Short Length (SL), indicates if Secure Data field is shorter than 48 bytes. Equal to
zero otherwise
6 of 61

• Packet Number (PN), used both as part of the Initialization Vector for the Cipher
Suite, as well as for replay protection.
• Secure Channel Identifier, optional, but is required to have multiple Secure
Channels.
Secure data is whatever data is being transported securely. This would be the unpro-
tected frame from its Ethertype field until the end of its payload.
Finally the ICV can have a length between 8 and 16 bytes.
A Cipher Suite is a particular specification of cryptographic algorithms and parameters.
The default and only mandatory Cipher Suite for MACsec is GCM-AES-128.

2.3
Galois/Counter Mode (GCM)
Galois/Counter Mode is a block cipher mode of operation that provides encryp-
tion/decryption as well as authentication. It has good performance and is relatively
simple.
In this case AES encryption is used for the block cipher.
Counter mode enables the repeated use of the same encryption key for encrypting
much longer streams of data than the block cipher supports.
Counter mode means initializing a counter block by concatenating an Initialization
Vector and a counter starting from zero. The counter block is then encrypted by the
block cipher resulting in a keystream. The resulting keystream block is XOR-ed with
the data that is to be encrypted.
This also enables support for encryption of data with a length that is shorter than the
block size of the block cipher.
GCM supports both authentication of additional data before an encrypted length of
data, or authentication only with no encryption of data. This is supported without
modification to the GCM algorithm.
For authentication GCM uses Galois Field multiplication on a GF2128 field. This mul-
tiplication is used to update a GHASH value. At the end of an encryption stream a
length block consisting of two 32-bit values concatenated. These values contain the

lengths of the AAD and encrypted data. This ensures that the length of the protected
data can’t be modified.
GCM technically supports an IV of any length, but the recommended length is 96 bits.
An IV should never be reused with a key. The uniqueness requirement is that a key
and IV pair are not used together with a probability higher than 2−32 [6]. This is a
critical requirement for the security of GCM.
As the counter block for keystream encryption comprises a 96-bit IV and a 32 bit
counter, this counter must never overflow. Counter overflow would mean repetition
that could lead to key derivation being possible.
The authentication tag can also be of any length but 128 bits is the default, and most
secure option.
GCM does not need to know the length of the data it processes in advance, it simply
needs the length block at the end of a stream. This is well-suited for MACsec as it can
encrypt data as it is being received.
GCM decryption is very simple. To decrypt a given encrypted stream, the same IV and
key is used. The only difference from encryption is that encrypted data is now XORed
with the keystream, and the output comprises the decrypted plaintext. The same
authentication tag will be derived at the end of the operation as the authentication tag
is based on the encrypted data, no matter if encryption or decryption is performed.
The authentication tag calculated at decryption can be compared to the one delivered
with the encrypted data.
[7]
2.4
Advanced Encryption Standard (AES)
AES is a specification for encryption based on the Rijndael block cipher family. An
AES block consists of 128 bits of data. The input and output of AES is done on a 128-bit
block basis.
In addition to a block of plaintext, AES also requires a an encryption key. This encryp-
tion key can vary in length and typical sizes are 128, 192 or 256 bits. Longer keys can
provide higher levels of cryptographic security.
AES block and state structure
The basic processing unit of AES is a byte, eight bits. Since AES has a block size of 128
bits, this block is divided into 16 bytes in a four-by-four array. This array of 16 bytes is
called the state.
Figure 2.1 shows how the bytes are organized in columns with a single index from
0-15 per byte for input and output blocks. The state itself is organized so each entry
has two indices.

Galois Field arithmetic
For some of the state transformations, AES uses Galois Field arithmetic for addition
and multiplication. The finite field arithmetic used in AES essentially allows addition,
multiplication and matrix multiplication to be expressed as a series of XOR and bit-shift
actions. The XTIMES function describes a way to multiply by 2. XTIMES is combined
with XOR functions to implement matrix multiplication. [8]
Bit-shifting and XOR-ing lend themselves very well to efficient binary hardware
implementations. This is a general theme in AES.
The AES S-box
AES uses a byte substitution table, the S-box.
The S-box was designed to provide resistance against differential and linear cryptanal-
ysis as well as algebraic manipulation attacks. It is possible to use a custom S-box, but
the original is thoroughly designed to avoid statistical analysis. [9]
Rounds and key expansion
AES applies encryption by doing multiple passes of "rounds", where transformations
are applied in the same manner for 10, 12 or 14 rounds for a 128-bit, 192-bit and 256-bit
key.
These encryption rounds are a place where optimizations can happen in particular.
AES also relies on key expansion. Expanding the key proportionally to the number of
rounds allows a fully distinct expanded key to be used per round. [8]

Chapter 3
Design and implementation in
VHDL
For implementation of all the modules, VHDL was chosen as the hardware description
language. Each module is clocked synchronously to the rising edge of the system clock.
All modules share the same clock signal. The reset signal is asynchronous.
Focus is on practical implementation of cryptographic algorithms. Security hardening
is not being considered in this implementation.
3.1
AES
Having a functional AES module is the foundation that everything else in this project
builds on. The U.S. Department of Commerce has published clear details for the AES
algorithm. [8]
Only AES-128 is implemented as this is the default and only mandatory algorithm in
the MACsec Cipher Suites [5]
Since this module will be used as the block cipher for a GCM module, there is no
need to implement the AES decryption mode. This saves significant logic area since
implementing the inverses of all the AES transformations basically doubles the logic
needed.
AES-128 has 10 rounds. One clock cycle spent per round means 10 clock cycles needed
to encrypt 128 bits. However AES key expansion is applied to the key to expand it to
4 ∗(Nr + 1) words, in this case 44. This can be fully parallelized, using a large logic
area or fully linearized, using 44 clock cycles(or 40, since the first four words are equal
to the key . This is a large number of clock cycles, especially considering that 10 linear
rounds of encryption take only 10 clock cycles.
An important factor to consider is that a typical Ethernet frame is much longer than
128 bits. A standard Ethernet frame is between the minimum length of 64 octets and
the maximum (standard) length (MTU) of 1518 octets (includes header and FCS).
10 of 61
==PAGE:14==
3.1. AES
A minimum length Ethernet frame is 64 ∗8 bits, the equivalent of four AES blocks.
Caching the expanded key instead of recalculating it every time a block is encrypted
may take slightly more logic to hold it in a register, but a huge number of clock cycles
is saved since even a minimum size Ethernet frame requires four AES blocks, and a
maximum size Ethernet frame requires 94 AES blocks.
A GCM encryption stream (and thus a single Ethernet frame in MACsec) uses the
same key to encrypt the entire stream. Not performing key expansion every block
represents a significant time saving. It may not be strictly required in terms of available
clock cycles for 10 Mbps, but it sets the design up well for a possible 100 Mbps future
specification.
The weight of clock cycles used in subfunctions is considered compared to the cycles
used for the actual encryption rounds.
3.1.1
Requirements
With the overall project requirements set forth in chapter 1, the AES module should be
designed to use fewer logic gates, rather than focusing on maximal raw throughput.
This means that the increased throughput and logic usage from employing pipelining
and parallel processing is traded for less logic area and using more clock cycles.
The AES module only needs to be able to encrypt a block since the GCM module
will enable decryption. It should be able to cache the expanded key in expectation of
multiple blocks being encrypted with the same key as specified by the GCM algorithm.
The AES module needs to:
• Encrypt a 128-bit block of data using the AES algorithm
• Perform AES key expansion
• Store expanded key while an encryption stream is ongoing
• Handle a continuous data rate of 10 Mbps
• Reduce logic resource usage
3.1.2
Design
The AES algorithm is implemented in VHDL by following the details in FIPS 197
closely, with an eye on minimizing logic gate usage. Sometimes tradeoffs are made
to save engineering time, and minimizing logic usage is done in order of largest
improvement.
Vector representation of the state array
In the design, the AES state is represented as a 1-dimensional 128 bit vector. This vector
is organized in columns-first order. That means that each 32 bits of the state vector
represent a column. The bytes of the vector are organized as in the "input bytes" table
of figure 2.1.
11 of 61
==PAGE:15==
3.1. AES
Transformations to the AES state are applied both row-wise and column-wise. The
column-major representation was chosen because the ShiftRows transformation shifts
the bytes in the state in a very simple manner, whereas the MixColumns step is a
significantly more complicated transformation that really benefits from the column-
major representation of the state vector.
SUBBYTES
The SubBytes function applies a substitution table, the "S-box" to each byte in the state.
[8]
The S-box is a 16 by 16 (a single hexadecimal digit) lookup table. There are 256 values
in total. This means that any possible byte can be looked up and substituted from the
S-box.
The S-box is split by x- and y-axis. For a byte xy, the x, y position is looked up in the
table and the value at that position is the substitute for the given byte.
When applied, the SUBBYTES function performs this substitution on each byte held in
the state.
The implementation of the SUBBYTES function is straightforward. The function is
implemented as a simple for-loop that substitutes each byte in the state with its
corresponding substitution value. A for-loop in VHDL expands to parallel logic,
and this is not necessarily the representation that results in the absolute lowest chip
area. This choice was made because the SUBBYTES function only modifies 128 bits
simultaneously. Compared to the length of the expanded key (1407 bits) it is fairly
small, and the use of 16 clock cycles to split this function up would be a large number
compared to doing the encryption rounds in 10 cycles. The function is also very simple
and readable.
The S-box is implemented as a 1-dimensional array of bytes. Any byte for substitution
can be looked up by setting its value as the index of the sbox_lut array.
SHIFTROWS
The ShiftRows operation is quite simple. Each row of the state is shifted to the right by

KEYEXPANSION
AES uses key expansion to provide a unique key to each round of encryption.
A table of hexadecimal round constants is provided in FIPS 197 [8]. The expanded key
is built on the original key and the table of round constants.
The key expansion algorithm relies on the ROTWORD and SUBWORD transformations.
SUBWORD performs exactly the same transformation as SUBBYTES, except on only
four bytes (one word).
Key expansion happens on 32-bit words, rather than 128-bit blocks.
The ROTWORD transformation is quite simple. It shifts the four bytes of a 32-bit words
as follows: [8]
ROTWORD([a0, a1, a2, a3]) = [a1, a2, a3, a0],
(3.4)
The key expansion algorithm is a place where significant trade-offs can be made on
time/logic resources.
An expanded key consists of 44 words of 32 bits, totaling 1408 bits.

GCM
GCM is integral to MACsec, as the default cipher suite of MACsec is GCM-AES-128.
GCM builds on top of a block cipher, in this case AES-128.
GCM handles both encryption, decryption, and authentication of data.
The GCM module uses the AES module to generate the keystream used for encryption
and decryption.
Counter initialization happens with an IV provided by the higher-level MACsec top
module.
The main consideration for performance and logic efficiency in the GCM module lies
in the algorithm for GF(2128) multiplication.
The rest of the responsibility of the GCM module mostly lies in handling data paths to
manage AAD, encryption, decryption and authentication as requested by an outside
entity, in this case the MACsec top module.
GCM is implemented according to the descriptions in [7] and [6].
3.2.1
Requirements
The GCM module should pass data in and out in 128-bit blocks, as this is the size of
block that AES works on. GCM uses a keystream cipher, and AES provides the block
encryption for the keystream.
The GCM module needs to:

3.2.2
Design
The AES module is initialized only once, and then reused for all the operations such as
calculating H, encrypting counter Yi, and encrypting Y0 to generate authentication tag.
The GCM algorithm as described by McGrew and Viega in [7] assumes that the entire
data for processing is available. Handling the entire stream of data at once is not
feasible for an Ethernet stream, as that would require buffering up to the maximum
length of an Ethernet frame. It would also require processing that amount of data in
parallel.
It is set to accept valid bytes and not bits, because the MACsec module operates on
full bits.
Instead the GCM module needs to handle a continuous stream of data, and to output
data as it is being processed. The following is a description of how the algorithm is
adapted for continuously delivering processed blocks:

For encryption with AAD:
• Initialize Y0 by concatenating IV with a 32-bit counter starting at one.

– Output data block

– If data block is for decryption:
* Increment counter Yi

MACsec for 10BASE-T1S Ethernet
The MACsec module is the entity that ties everything together. The primary function
of authentication, encryption and decryption is mostly handled by the GCM module.
MACsec however requires management of different Secure Associations, and handling
of Ethernet streams in both RX (receive) and TX (transmit) directions from the module.
Furthermore, there are specific requirements on data widths and rates. All of these
factors combined necessitate a multitude of supporting modules such as buffers,
memory, and classifiers to parse Ethernet frames and handle them appropriately.
3.3.1
Requirements
This implementation needs to live up to the specific needs of 10BASE-T1S. 10BASE-T1S
is a standard specifically intended to reduce wiring complexity, saving space and
weight. This implementation needs to follow those intentions as well.

3.3. MACSEC FOR 10BASE-T1S ETHERNET
As the name indicates, 10BASE-T1S supports a transfer speed of 10 Mbps. It is config-
urable but supports up to full duplex [4].
It is a shared medium protocol. The specification requires support for at least 8 devices
on one shared medium. In this case that means 7 other devices besides this module
itself. Being a shared medium, there is no need for the MACsec module to handle
simultaneous Ethernet frames on both RX and TX inputs. 10BASE-T1S uses Physical
Layer Collision Avoidance (PLCA) at the PHY layer. That means only one frame should
arrive at a time.
This module must handle MACsec over the shared medium. This means that there is a
requirement of handling multiple Secure Associations, and applying the relevant one
to Ethernet frame in both Receive and Transmit directions.
Ethernet data will be transferred on 8-bit wide ports. Along with the Ethernet bus, there
will be signals indicating if a given byte on the bus is the start or the end of an Ethernet
frame. There will also be signals indicating if an error has happened. This error signal
can indicate a malformed frame, authentication error, no Secure Association found,
etc.
MACsec supports multiple cipher suites. Only the GCM-AES-128 Cipher Suite is
mandatory to support [5]
• Handle MACsec Authentication, Encryption and Decryption
• Support default GCM-AES-128 Cipher Suite
• Support 10 Mbps Full Duplex speeds
• Support Secure Associations to 7 other devices (8 in total including itself)
• Take Ethernet input and output on an 8-bit wide bus
• Be optimized for efficient use of logic area
Early discussions led to a specification on the external ports that the final implementa-
tion should have. This specification can be seen in figure 3.4. One deviation in ports
from this original specification to the design is the removal of the port that signals
amount of frame padding. This was removed because the minimum data unit of Ether-

directions is one byte wide.
In addition, ports would also have to be added to indicate which particular Secure
Association a given Ethernet frame belongs to on the TX-input, as well as an indication
of whether only authentication or encryption should be applied. This would mean
input ports on the TX side for SA, AN and encrypt/decrypt.

3.3.2
Design considerations
There are many considerations to make when designing the MACsec module. Ethernet
frame need to move in both Receive (RX) and Transmit (TX) directions. In either
direction there are specific requirements for how Ethernet frames should be handled.
In the RX-direction, Ethernet frames arrive with MACsec already applied. That means
that those frame already contain a complete MPDU and must be handled accordingly.
Their SECtag needs to be parsed to determine whether the frame contains only au-
thenticated data, or if encryption has been applied. Then according to whichever is
true, the Ethernet frame is authenticated or decrypted (once the Ethernet header is
authenticated). An ICV is then calculated and compared to the ICV that the Ethernet
frame arrives with.
In the TX-direction, an unprotected Ethernet frame needs to be protected with MACsec.
That means that a SECtag must be built and added to the beginning of the ethernet
frame. Information must be given to the MACsec module about which Secure Associ-
ation the given Ethernet frame belongs to and whether or not encryption should be
applied. Then according to this information a SECtag must be built and injected be-
tween the Ethernet frame header and the payload. Once the frame has been processed,
an ICV must be calculated and added to the end of the Ethernet frame.
Modules should be designed to handle both parsing an incoming Ethernet frame
with MACsec applied on the RX side, as well as building a frame with MACsec for
24 of 61
==PAGE:28==
3.3. MACSEC FOR 10BASE-T1S ETHERNET
transmission on the TX side.
In either case, the nature of 10BASE-T1S Ethernet being multi-drop, means that a
memory of Secure Associations must be maintained. This memory must be able to
perform a lookup of a given Secure Association and return the information necessary
to perform encryption/decryption and to verify the Packet Number against replay
attacks.
Since the design should be considerate of logic utilization, performing memory lookups
in the fashion of a linear search seems like an appropriate choice. The number of
memory lines should not be so big that a linear search will dominate in time usage.
Since GCM generally works on blocks of 128-bits (with exceptions for partial final
blocks) and the bus for Ethernet frames is 8 bits (one byte) wide, there is a need for
buffering on all external inputs and outputs.
On the RX side there is the additional need for recovering the ICV on an incoming
frame. The ICV must be recovered before any of it is passed to the GCM module so it
can be compared to the calculated ICV for the frame.
Because Ethernet frames can have a length in bytes that is not divisible by 16, the
buffer for the RX side was extended to 32 bytes. This means that when the frame_end_i
signal is asserted on the RX-in side, the ICV will be available in the buffer. The buffer
then signals how many bytes are available, and the MACsec module will know that
the final 16 bytes contain the ICV.
On the TX-in side the buffer only needs to hold 16 bytes, as the ICV is to be generated
by the MACsec module.
Either buffer must support frame_end_i, and inform the MACsec module how many
bytes are available upon frame end.
For output on both RX and TX, a 16-byte buffer should be adequate, as GCM outputs

Secure Association Memory Module
To store multiple Secure Associations, some kind of memory is needed. For the design
implemented here, the memory module is intended as a simulation stand-in for what
would be a real memory module that can be accessed by both the MACsec module, but
also updated by an upstream (external) module that handles MACsec Key Agreement.
The memory module described here is therefore a "dummy" module that is easy to set
for simulation purposes, and uses a realistic amount of time to search for entries.
Since the point of all designs in this project is to minimize logic usage, the memory

maximum search time in clock cycles is equal to the number of memory lines.
Searching the memory can’t happen simultaneously with GCM, as GCM needs the
output of the memory in order to even start.

stream.

MACsec RX classifier
When a new Ethernet frame begins on the RX input, this module is passed the SECtag
from that frame, parses it for all the necessary information to find and verify the Secure
Association belonging to that Ethernet frame.
The module supports a single VLAN tag and can handle the offset of data that comes
with a VLAN tag. If such a tag is present it will then search further for the MACsec
Ethertype identifier. It will then extract the necessary information from that point.
This module extracts: SCI, AN, PN, and parses TCI to determine if frame has secure
data or needs authentication only. It also signals the presence of a vlan tag so that AAD
can correctly be separated from potential secure data.
Using the module:
The module is passed the first 16 bytes of the Ethernet frame. If those first 16 bytes
contain a VLAN tag, the module will move on to the next 16 bytes. Those next 16 bytes
contain the SECtag, and the necessary information can be derived.
If the first 16 bytes contain the MACsec Ethertype, then those first 16 bytes also contain
the TCI, AN, and SL fields. The next block will then contain TCI, AN, SL, PN and SCI
fields.
The module works through a very simple state machine that parses the first block

==PAGE:32==

already in the s_idle state. The module then switches to the s_block_2 state to parse
the second block. The presence of a VLAN tag in the first block shifts the indices where
the module will look for the relevant data.
If a MACsec Ethertype is not present, the module raises an error. It also checks validity
of the TCI according to the MACsec standard [5], and the specifications of the top-level
module.
The TCI field of the SecTAG is parsed as follows[5] [10]:
• V = 0, mandatory
• ES = 0, indicates presence of SCI
• SC = 1, indicates presence of SCI
• SCB = 0, otherwise SCI omitted
• E = 1/0. If 1, encryption enabled. If 0, authentication only

Input Buffer
The 16 byte buffer is not completed to a functional degree, as it belongs on the TX
input side. The TX path has not been implemented. It would be very similar to the 32
byte buffer, however.
On the TX-side input, the buffering can be quite simple. GCM needs blocks of 16 bytes,
and the Ethernet frames are delivered one byte at a time. This buffer takes in a byte at
a time and shifts it into a 16 byte vector. It raises block_rdy_o when it has collected a
full block.
Input Buffer (32 bytes)
On the RX-side input for the MACsec top-level module there are some special consid-
erations.
Even though GCM only requires 16 bytes input per block, each Ethernet frame comes
with an ICV. It is crucial that this ICV is caught in time and not added to the GCM
stream and GHASH. Since the frame_end_i flag is only asserted on the very last byte
of the stream, it is necessary to buffer an additional 16 bytes.
Buffering these 16 bytes allows the ICV to be kept out of the GCM stream. Once the
frame_end_i flag is raised, there will be minimum 16 bytes left in the buffer. Even if
there are more, the final 16 bytes in the buffer will contain the ICV.
The buffer will indicate exactly how many bytes are left by the end of the frame so the
top level module can finish the final, potentially partial, block of GCM.
When the buffer receives the frame_end_i signal, it goes into a special state. Since
the buffer shifts data in, it will shift the data all the way to the front, and then raise
frame_end_o, block_rdy_o and frame_bytes_o will indicate how many bytes of data
are left including the ICV.

12 cycles ≈10.67 bits/cycle

4.1.3
Logic area utilization
To compare logic area utilization, the entity is synthesized in Intel Quartus. Since the
point is to compare relative logic area utilization, the specific FPGA board selected for
synthesis is not important as long as it is consistently used across entities.
For consistency in synthesis for all these tests the Cyclone V: 5CGXFC9E6F35C7 board
is selected. It has 113,560 Adaptive Logic Modules (ALMs). ALMs are groupings of
logic. The number of ALMs will be used to compare the relative logic utilization of the
various modules.
Multiple versions of the AES entity were tested as they were designed and improved.
• Version 1: The entire AES and key expansion algorithm were implemented
to be executed fully sequentially in a single clock cycle. That meant simple
implementation for the sake of simulation to verify math. It also meant a vary
large logic footprint, and a very long critical path.
• Version 2: AES rounds now implemented in a more logic efficient manner, where
each AES round is repeated through the same circuit, across a number of clock
cycles equivalent to the number of AES rounds. Key expansion still previous
design. This version already shows significant improvement.

• Version 3: Both AES and key expansion both serialized in a logic-efficient manner.

• Version 5: Further use of a shift register to perform the ADDROUNDKEY trans-

Logic area utilization
Two versions of the GCM module were tested. The first version featured multiple
redundant uses of the gf_mult function that needlessly increased logic utilization.
The second version eliminates about half the redundancies. Further optimizations
would be possible to reduce logic even more, but there is also a very long critical path
in the gf_mult function. gf_mult would need to see design optimizations as well. The
current implementation is fully serialized, and thus has an extremely long critical path
as it uses a loop for 128 iterations, all in a single clock cycle.
An optimization across multiple clock cycles would likely yield a decrease in logic
utilization roughly equal to the number of cycles that the function is spread over.
Halving the redundancy lead to an approx. 5000 ALM reduction, indicating about
2500 ALMs per gf_mult. Reducing to only a single gf_mult, and then splitting that by
a factor of 8 (2500/8 = 312.5) might bring enough reduction to take the total number
of ALMs for the GCM module down about 10000 −312.5 ≈9700. This would bring
the logic utilization around the same order of magnitude as the AES module.
Improving the critical path should be a relatively simple operation. Executing this
improvement would only be a matter of having more time to implement it.
This improvement should also bring a large increase to Fmax. The specific number
of clock cycles that the gf_mult function is spread over is a compromise between low
logic area and high Fmax vs. clock cycles used. There should be further testing to
decide where to draw the line.

Testing
5.1
AES testing
For AES the test vectors are sourced from FIPS 197 [8]. The document includes an
example of expansion of a 128-bit key, with all intermediate calculations in appendix
A.1 of FIPS 197 [8].
Appendix B of FIPS 197 [8] is a cipher example with intermediate values.
The AES module was tested against both these examples and with the given inputs,
all intermediate and final values were realized by the design.
The AES module is only specified for AES-128, so only 128-bit keys are tested.
Testing the AES module is quite simple as it only involves passing the key and plaintext
input to the module, and then verifying final values.
Since the AES module is exclusively for the purpose of generating the keystream for
GCM, only the encryption mode is needed. Decryption is not tested as it is not required
of the module.
5.2
GCM testing
Testing of the GCM module is a little more involved than testing the AES module.
The GCM module has to handle a lot more different data. It handles both authentica-
tion, encryption and decryption. The GCM uses the AES module as a block cipher to
generate a keystream. Only the AES-128 mode is supported.
For input, the GCM module take a key, an IV, AAD, plaintext/ciphertext for decryption
and a length vector.
It outputs ciphertext/plaintext, and an authentication tag. This particular implementa-
tion of GCM also outputs authenticated text for ease of passing an encrypted stream
with AAD as an Ethernet frame.

Chapter 6
Discussion
With the overall goal of reducing logic usage by efficient engineering, there are many
ways to achieve the required specifications.
Overall many requirements were met, but there are also shortcomings and issues.
Changes, solutions and future possibilities for improvement are discussed in this
chapter.
6.1
AES
The AES module is designed with the purpose of showing adequate performance with
minimal logic area. Beyond just the tradeoff between performance and logic area is the
consideration of engineering effort.
The AES module attempts to strike a good balance of performance and logic area. It
has also been implemented in a manner that transparently follows the specification of
the algorithm [8] and has a simple design.
The AES module could be designed to use even less logic area for a super efficient
design. This would require far more engineering resources and would be unlikely to
show an order of magnitude in improvement.
Simple optimizations like reusing logic for key expansion rounds and encryption
rounds showed an improvement in logic utilization of about 10 times. This gain was
realized with very little additional engineering time.
Even cutting the logic utilization in half from 870 ALMs to 435 ALMs represents little
absolute gain compared to the rest of the MACsec implementation. To some degree it
will be hard to reduce the usage of registers without a heavy performance penalty.
In the final synthesis of the whole MACsec module, AES logic utilization did increase
to 2278 ALMs. This is likely due to the synthesis tool testing additional data paths.
Even with this result, however, the AES module has a reasonable logic utilization.
This kind of thinking has also lead to the decision of implementing the AES S-box as a
51 of 61
==PAGE:55==
6.1. AES
constant array. Implementing a combinational circuit is certainly possible but would
represent additional engineering effort that would likely be better spent improving
some of the other modules in the MACsec implementation. [11]
6.1.1
Performance
The AES module overall shows excellent performance. With the complete MACsec
implementation requiring 10 Mbps in each direction for 10BASE-T1S Ethernet, the
AES module shows plenty of headroom for various modes of operation. Throughput
starts at 242 Mbps in the slowest case of performing key expansion to encrypt a single
block. Once key expansion is done, however, the throughput of subsequent blocks is
1067 Mbps. Naturally key expansion must be performed each time the encryption key
is changed, but over several blocks of encryption the throughput converges on 1067
Mbps.
For full duplex operation of 10BASE-T1S a throughput of 20 Mbps for the full MACsec
implementation is required. Comparing the slow case of AES encryption gives a factor
of 242
20 = 12.1 times the needed throughput. This should leave plenty of headroom for
additional processing required by both GCM and MACsec modules.
6.1.2
Possible improvements
In case of a future scenario requiring much higher performance the AES module
could be modified to use a pipeline for processing. The current design is optimized
for the least logic utilization possible. Even moderate pipelining could increase the
throughput multiple times, for a proportionally larger logic utilization.
This implementation of the AES algorithm follows FIPS 197 [8] very closely. In that
description of the algorithm there is little mention of hardening against side-channel
attacks.
Side-channel attacks include timing analysis and power analysis, and these techniques
can be used to compromise the security of an AES implementation that is not hardened
against them. [12]
It is also not impossible that bits could be flipped randomly. This be be due to back-
ground radiation, a targeted radiation attack or just faults in the fabricated circuit.
Some kind of error-checking or correction of stored secure and expanded keys may be
needed to really ensure security.
Depending on the final application of the AES module, it should probably be hardened
against some of these attacks. This particular implementation is not focused on hard-
ening against these attacks. Useful hardening techniques could include even timing
across all lookups and operations and avoiding asymmetric operations. [12]
The AES module relies heavily on variables for calculations. This is quite appropriate
for the purpose of sequential updates to the AES state during each round. For storing
information between clock cycles and the states of the finite state machine that operates
the AES module variables should be refactored to signals. Making this change should

make the design more predictable to synthesize. This would also be more in line with
traditional VHDL techniques.
6.2
GCM
The GCM lives up to the majority of its requirements, with the exception of the logic
efficiency requirement.
The GCM module is fine in terms of throughput performance and functionality. The
main issue with the module lies in its large logic footprint. Optimizations need to be
applied to the implementation of Galois field multiplication.
6.2.1
Performance
The GCM module shows excellent performance in terms of throughput. It handles
all the operations necessary for GCM. Authentication, encryption, decryption and
generation of an authentication tag is fully functional.
Throughput is many multiples of the required performance at 175 Mbps for the slowest
block and 415 Mbps on average for 65 bytes.
The logic utilization of the GCM module dominates the total logic area of the entire
MACsec module. Out of 13109 ALMs the GCM module uses 9931. This is due to an
inefficient implementation of GF(2128) multiplication. The gf_mult function also has a
very long critical path, leading to an unacceptable maximum clock frequency of Fmax
= 22.73 MHz.
6.2.2
Possible improvements
Masking out invalid bits on partial block For the purpose of correctly masking out
bits when encrypting partial blocks, a quick fix was added. This fix uses a for-loop that
zeroes out irrelevant bits. a more efficient solution with fewer levels of logic should be
added. This fix was made in order to verify correct operation of the GCM module.
GF multiplication implementation (goes under improvements?)
The gf_mult function used in the GCM module is a direct implementation of algorithm
1 from NIST 800-38D [6].
Reducing the redundancy of the gf_mult function lead to a significant reduction in
logic utilization. Going from four copies of gf_mult to two, showed a reduction from
14648 ALMs to 10127 ALMs. That’s ∼4500 ALMs. Extrapolating would bring the
number of ALMs used for the GCM circuit to ∼7600. This is still about an order of
magnitude more of logic compared to the AES module.
This implementation also comes with an extremely long critical path. The straight im-
plementation of gf_mult from NIST 800-38D [6] means that 128 sequential operations
need to be performed in a single clock cycle. This long critical path is also evident in
the result showing Fmax = 22.73 MHz. 100 MHz is generally used as a comparable
53 of 61
==PAGE:57==
6.2. GCM
benchmark for these designs, but it is possible that a final design may be ran at a higher
frequency. This maximum frequency is unacceptable.
Improving logic utilization and Fmax is likely to go hand in hand. In the original GCM
specification by McGrew and Viega [7] there is a table of implementation methods. Of
those methods, the digit serial method is an optimization similar to the one performed
in the AES module where the processing is repeated over multiple clock cycles. This
should reduce logic area approximately by the number of cycles that the processing is
spread over. gf_mult repeats a similar processing step many times, making it suitable
for this optimization.
Estimating logic area by synthesizing for an FPGA board is an opaque operation and
it can be hard to know exactly which optimizations are applied. Therefore an estimate
can’t be made for logic utilization with these optimizations. Since gf_mult should be
the dominant logic, it is likely a significant reduction. Apart from passing data to the
AES module and performing gf_mult the GCM module mostly just manages data flow,
which shouldn’t require large amounts of logic beyond some registers.
GF(2128) is performed on a 128-bit vector. Suitable divisions would be 8 or 16. Spread-
ing the calculation over 8 cycles, would require processing of 16 bits at a time. That
would also mean 16 levels of logic per cycle.
Alternatively processing 8 bits per cycle for 16 cycles would mean a shorter critical
path and less logic usage.
Both 8 and 16 cycles are on the same order of magnitude of the time that encryption
of a single AES block takes. Making this change would roughly more or less double
the processing time of a block. Given that the throughput, even in the worst case, is
several times higher than required leaves plenty of room for this change.

6.3.1
Possible improvements
Higher data rate support While there is currently little literature on official standards,
it is not unlikely that future standards for Ethernet may expand the principles of
10BASE-T1S to higher data rates of 100 or 1000 Mbps. While the MACsec module
is currently not near the throughput needed for 1000 Mbps operation, it would be
capable of handing 100 Mbps full-duplex with the current throughput. Adding simple
pipelining to the AES module and possibly a parallelized memory lookup should
make the throughput significantly higher. Increasing the clock frequency also provides
a direct boost to throughput, as long as timing requirements are met by the critical
paths in the circuit.
Buffer look-ahead On the RX side, the SECtag is currently parsed 16 bytes at a time.
The input buffer holds 32 bytes and allows look-ahead of the entire buffer. Parsing the
beginning of an incoming MACsec frame could be done in a single cycle as the first 32
bytes contain the entire SECtag, even if a VLAN tag is present. This would save logic
and at least a single clock cycle.
State machine improvements The states that the MACsec module currently steps
through suffer from being hastily engineered. Adding more states to separately handle
AAD and decryption states may simplify the design. Adding a function to control the
GCM module may also improve readability of the VHDL design significantly.
Replay protection The current implementation has very simple replay protection.
Incoming packets are checked against the stored packet number. If the incoming
packet number is larger than the one stored in memory, then it is accepted. The current
memory is a read-only test unit but in a final implementation it would update the
packet number.

6.4. TESTING
The MACsec standard specifies that sliding window replay protection may be used to
allow frames within a certain window to arrive out of order. This could be implemented
in a future iteration. The MACsec standard includes support for a replay window [5].
Error signals The error handling is currently very rudimentary. Adding multiple error
codes and error counters would be more in line with the MACsec specification.
Management engine The MACsec module would be part of a larger MAC entity. It
should therefore be configurable according to the MACsec standard. The overall MAC
entity should easily be able to configure the MACsec module to any operation required
by the specification.
More cipher suites While not essential, supporting the optional cipher suites of MAC-
sec such as GCM-AES-256, or the extended packet numbering cipher suites could
be a consideration as the security landscape evolves in the coming years. Adding
support for these modes would mean adding 256-bit key support to the AES module,
extending this support to the GCM module and supporting extended packet numbers
in the MACsec implementation.
6.4
Testing
For now the only testing of the modules has been official test vectors. Official test
vectors do test the functionality of algorithms and modules, but there are many corner
cases that should be tested as well.
To that end, developing a comprehensive testing framework would be necessary to
test a great variety of cases. This would include building a system, likely with software
libraries, to generate randomized Ethernet frames. Both valid and invalid frames
should be generated, with care taken to test corner cases and overflows of packet
numbers and the counter in GCM.
As well as generating these test frames, the correct result should be generated. This
would enable validation of outputs as they are processed by the MACsec module.
Generating both inputs and expected correct outputs will enable fully automated
testing where billions of test cases can be processed and checked against an expected
correct output.
These tests should first be validated in a software simulation. Should potential issues
arise, they can quickly be fixed.
Once the module performs correctly in software simulations the testing should be
expanded to testing the module on an FPGA board. This means that the physical
timing and implementation can be tested. A thorough validation in this manner would
prove the module ready to eventually be integrated in a larger context of an Ethernet
module.

Chapter 7
Conclusion
This project explored a clean-slate, ground-up implementation of MACsec for 10BASE-
T1S Ethernet. This included ground-up RTL designs for AES, GCM and MACsec
modules, including submodules.
Serialization and logic reuse in the AES module reduced logic usage by a factor of
11.4x, going from 9885 ALMs to 870 ALMs in individual synthesis tests but ending up
at 2278 ALMs in final synthesis of the entire MACsec system. This comes out to 17.4%
of all logic. Shortening of the critical path increased maximum clock frequency by 8.9x
going from 21.86 MHz to 195 MHz. With a throughput of minimum 242 Mbps at 100
MHz, the logic-efficient implementation shows plenty of performance headroom.
For the GCM module full functionality was achieved, but optimizations on the level of
the AES modules were not realized. The GCM module dominates the total logic usage
at 76 % of the total logic area with 9931 ALMs. The GCM module has problems with
the critical path at a maximum frequency of 22.73 MHz. The GCM module does show
adequate throughput for a 65 byte encryption case at 408 Mbps at 100 MHz.
The MACsec module did not reach full functionality. The Ethernet receive part of
the MACsec module came close to proof-of-concept. The submodules on the RX side
attained full functionality. This includes an input buffer, an output buffer, a SECtag
classifier, memory module with search. The MACsec module also relies on the GCM
module for its cipher suite. The partial MACsec module with most of the TX logic
missing has a logic usage of 900 ALMs, about 7% of total logic.
The techniques employed to reduce the logic usage of the AES module were effective,
and should have a similar impact on the logic efficiency of the GCM module. The
Galois field multiplication function in the GCM module should be the specific target
of these optimizations.
Utilizing these optimizations should yield the logic efficiency required to develop this
into a complete module that can eventually be realized as part of a complete MAC
implementation. Even with the additional time overhead from a more logic efficient
there is enough throughput to still achieve the speeds necessary. Adding the most
pessimistic estimate of time overhead still results in a throughput of 263 Mbps for a 65
58 of 61
==PAGE:62==
byte Ethernet frame + 24 bytes of overhead. With full-duplex 10BASE-T1S requiring 20
Mbps, 10 each for receive and transmit, the MACsec module provides a factor of more
than 10 times the requirement. At this performance, the MACsec module may be able
to support 100 Mbps operation with little or no modification.
As noted, the current design in the Galois field multiplication is the main limitation
of the current design. It is responsible for the bulk of the logic area utilized, and it
has a very long critical path that limits the maximum clock frequency of the circuit to
significantly below acceptable levels.
Estimates based on the results from design of the AES module indicate that serialization
and logic reuse in the GCM module will result in a much lower logic utilization and
increase in maximum clock frequency roughly proportional to the decrease in logic.
The additional processing time brought on by these optimizations should not impact
throughput enough to give any issues. Including overhead, MACsec throughput goes
from 375 Mbps to 263 Mbps in the most optimized (slowest) case. A generous margin,
even for full-duplex operation at 20 Mbps.
The primary other limitation is the non-functional aspect of the MACsec module. In its
current state the MACsec module is not able to deliver a correct Ethernet frame. The
underlying modules are functional, but all of the data paths need more work on their
overall implementation.
The most functional is the RX authentication only path. While it has a minor buffer
issue it was useful enough to estimate MACsec overhead from memory search and
SECtag classification.
No particular technical challenges prevent a functional module to be developed other
than the requirement of more engineering effort and time.
Next development steps should prioritize the optimization of the Galois field multi-
plication function. This should significantly decrease the logic area used by the GCM
module. It would also raise the maximum clock frequency to a range that may be
required, depending on integration into a larger module.
While there are issues with large logic utilization and functionality that need to be
addressed for this MACsec implementation for 10BASE-T1S to be ready for real-world
use, the path forward is well-defined. The work discussed here lays out a clear path
for the enhancements needed, and evidence from improvements shown on the AES
module show the viability of applying these improvements where they are necessary
on the GCM module. With the performance that is shown in the current iteration, the
trade-offs needed to make the necessary optimizations will not impact performance
enough to cause any concern.
With pending improvements, the performance and logic usage of the implementations
suggest that it is very possible to design a logic efficient MACsec module with adequate
performance for 10BASE-T1S Ethernet.
