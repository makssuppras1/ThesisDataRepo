Summary
In this report, we have investigated the literature surrounding TLS 1.3 performance and extensions, and
found there to be a gap for performance evaluation of the extensions of TLS 1.3. To add to the area,
we have designed and implemented a test bench to perform a performance evaluation of extensions and
documented the use of it. During the paper we document and discuss results as well as the lessons
learned with a hope that further investigation in the area can have a starting point. To help further
the research we also provided a list of suggested contributions as part of the future works to guide
the research. We conclude on the the results from the analysis, giving recommendations based on the
results, and finally summarize on the lessons learned in this project.

1.1
The big picture
A lot of work is being put into making the digital lives of everyone secure in such a way, that it appears
invisible. Many people will go about their day, not knowing what complex systems reside right under
their noses. Most will be acquainted with the lock-screen on their phone, or the password to their bank,
but most security is abstracted away from the users, yet it is ever-present.
One of these ”hidden” layers of security found whenever a browser makes a request using the HTTPS pro-
tocol. The Hyper Text Transfer Protocol Secure encrypts traﬀic between a client and a server, making
it unreadable to any third-party that observes it. Often indicated by a little lock in the address bar of
any browser, that show the connections made by HTTPS are secure. This security comes from a proto-
col known as TLS (Transport Layer Security). TLS provides end-to-end encryption, meaning only the
server and the client can read the traﬀic, all other intermediary hops are unable to. As attackers become
more advanced and employ more powerful attacks, so must the requirements for the protocol follow
suit, which means that TLS is currently at version 1.3.(Table 1.1 shows how SSL 2.0 became TLS 1.3) 1

When developing new products HTTPS is an attractive choice of protocol because it allows compat-
ibility with a plethora of already available products and software packages.
By basing a products
communication on HTTPS, it is plug-and-play with web-servers, desktops, phones and laptops. It also
enjoys the security provided by TLS 1.3.
IoT-devices and wireless sensor networks can employ TLS to secure their communications. The other
end of the communication line is also important; large websites getting well above 10000s of connections
every second need to provide a secure channel of communication to every connection. The problem is,
that security is computationally expensive. No matter the scale, having security that is both eﬀicient
and secure is something many engineers work hard on achieving and TLS 1.3 is an attempt at making
a flexible protocol that can make the internet more secure.
At the highest level, TLS is a protocol that allows a client to authenticate a server and establish a

1.2
The CIA triad
In the world of information security, a commonly used description of security is the CIA triad:
• Confidentiality
The data should only be readable by those for whom it is intended
• Integrity
The data must not be modified, or if it has been, it must be detectable
• Availability
The data must be available and use-able.
The triad is a very effective way to visualise the objective of many, but not all, security implementations.
When any part of the triad is no longer met, we end up with un-usable systems:
• Confidentiality & Integrity
No availability means no-one can use the system
• Confidentiality & Availability
No integrity means we don’t know if the data has been tampered.
• Integrity & Availability
No confidentiality means everyone can read the messages
Systems such a payment platforms have additional requirements, such as non-repudiation, the inability
to deny that a transaction took place.
Now that it is established that fulfilling the CIA triad gives a good baseline for a secure implemen-
tation, how does one go about doing such a thing?
1.2.1
Confidentiality
Confidentiality is achieved through the use of encryption, using a secret key to allow for messages and
data to be scrambled leaving it only readable to those that hold the key.
Often the most studied,
sometimes encryption and security are used interchangeably, but this is often not the case, especially
not for protocols . Encryption is most commonly found in two forms:
• Symmetric
The same key is used for encryption and decryption.
• Asymmetric
Encryption and decryption is done with two different keys.

1.2.1.1
Symmetric keys
Symmetric key cryptography is based on having a single key that is used for both encryption and
decryption.
It is a fast and eﬀicient way to encrypt, especially since many symmetric ciphers are
designed to use operations a computer can eﬀiciently perform. Encrypting files locally is often done
using symmetric ciphers, but for network traﬀic, we run in to an issue: We can easily encrypt our traﬀic,
but how do we make sure our recipient can read it? Perhaps we could share the key out of band, but
for a fully functional system, that is not feasible. Another is that of the number of keys: If we have n
devices that need to be able to talk to each other, they would need
n · (n −1)
2
(1.1)
keys. To solve the key distribution problem, another class of encryption can be used:
1.2.1.2
Asymmetric encryption
Instead of having one key, asymmetric employs two linked keys. One that can encrypt data, and one
that can decrypt data. Normally, the key that has the ability to encrypt is called the public-key and
the key that can decrypt is known as the private-key. As their names suggest, the public-key is public
and anyone can encrypt a message using it, but only the owner of the private key can read it. Two
people, each with a key-pair, can simply use the partners public-key to encrypt and so both parties can
enjoy encrypted messages. It also significantly reduces the number of keys needed for n devices that
need to communicate:
2 · n
(1.2)
The catch then, is that asymmetric cryptography is much slower than symmetric. They also run in to
an issue of trust: How do you know if a public-key really is the one you believe it is? But that is a
question to be answered later.
1.2.2
Integrity
Integrity is analogous to a tamper seal or shipping manifest: Does the contents of the crate match our
expectations? Integrity is important, without an mechanism to verify if the the data received is what
was expected, attackers can inject further instruction, change messages or change the supposed sender
of it. integrity is often proven by the inclusion and proof of inclusion of something an attacker couldn’t
have known or made.
• HMAC (Hashed Message Authentication Code)
A code appended at the end of the data, that can be calculated again by the receiver and compared
to what was sent
• Signatures
Using a private key to encrypt a message that can be decrypted by all that hold the public key
1.2.2.1
HMAC
HMACs are primarily used when no encryption or symmetric encryption is used. Using a secret key
as part of the data the hash is being calculated from, it makes it impossible for a third-party forge
messages. If it is attempted, the receiver can easily see that the checksum sent and the one calculated
are different, and the message can be dismissed.

1.2.2.2
Signatures
In the world of asymmetric encryption, if you use your private-key to ”decrypt” a plain-text, the
resulting scrambled data can be restored if ”encrypted” with the matching public-key. Thus, a signed
message could only have originated from the owner of a private key. By signing some data unknown to
an attacker and including it in a message, the attacker is unable to forge a convincing message. The
only requirement is that you trust that the public key in question actually belongs to the person. In
other words, if you trust the signature to be the correct one.
1.2.3
Availability
Availability describes what is left, and perhaps what is most noticeable: That the system you design
is alive and responsive. It locks out edge cases as: A system with no users or ways to interact with it,
is the best kind of security. Availability also describes the diﬀicult ask of combating Denial-of-Service
(DoS) attacks.
1.2.4
Authentication
Sometimes the A in the CIA triad is taken to mean authentication, which is to describe that one actor
is convinced that it is indeed talking to the correct partner. Usually authentication comes from one of
the following
• Something you are
Fingerprints, voice recognition, Iris scanning etc.
• Something you have
Key-card, Phone, Security Token, Nem-ID card, etc
• Something you know
Password, Pattern, security question etc
1.3
Protocol security
When moving from protecting data to protecting communications, another set of challenges become
apparent. Even if all the edges of the CIA triad appears to have been met, it is not enough to simply
encrypt and use an integrity check when securing a protocol. When talking about protocol security,
often times the adversary is a man-in-the-middle, an actor with control over the communication medium.
Being in control of the network, allows this adversary to perform a number of attacks:
• Eavesdropping
• Fabrication of messages
• Preventing messages from arriving
• Intercepting and modifying messages
• Re-transmitting messages
An illustration of these are in figure 1.1.
These challenges are the reason why protocols are needed to establish secure communications. Sim-
ply encrypting a string of data containing a command to open a lock could easily be intercepted and
replayed. This is why protocols need to be designed with security in mind, from the ground up.

Designing a protocol
Designing a good protocol is a diﬀicult challenge, and a lot of things needs to be taken into consideration.
Questions such as: Where will the protocol be used? Are there requirements for timing? What is the
network constellation? Which devices do we consider secure and trust-worthy, which ones do we not
trust? Are just a few of the many things a good protocol will incorporate in its design.

Once we need to connect to something that is not directly in our control, things become tricky: We

1.4 TLS 1.3
8
All this discussion of security and protocols leads us then to TLS 1.3
1.4
TLS 1.3
TLS 1.3 is specified in RFC 8446 [Res18]. It is stated that the primary goal if TLS is to
”provide a secure channel between two communicating peers; the only requirement from the underly-
ing transport is a reliable, in-order data stream.”
This means, that all TLS needs for it to function, is a TCP connection. The high-level security goals
of TLS can be expressed as a CIA triad with authentication:
• Confidentiality
Data sent by TLS after establishment will be visible to the endpoints only.
• Integrity
Data sent by TLS after establishment with cannot be modified without detection.
• Authentication
TLS always authenticates the server and optionally authenticates the client.
These properties are to remain true, even if if an adversary controls the network, as described in section
1.3. An informal description of the security properties can be found in Appexdix E. of [Res18]
TLS 1.3 is logically split into two components:
• Handshake protocol
The protocol that establishes a new connection between two devices
• Record protocol
The protocol that facilitates communication between two devices
An informal explanation of the TLS 1.3 compnents. For a full specification, we refer to the full standard
in its chapters 4 and 5 [Res18].
1.4.1
TLS 1.3 Handshake
The basic TLS 1.3 handshake consists of the following steps
• Client Hello
The first message sent by a client; contains data such as supported ciphers, compression methods
and extensions supported.
• Server Hello
If the server can complete the requested handshake, the server will do so. Alternatively, it will
send a HelloRetryRequest.
• HelloRetryRequest
If the server is able to fulfill the client hello, but is missing data, this message can be send to
prompt the client to send the missing data as part of another Client Hello.
• Finished
Necessary to prove that a handshake is complete and correct, as well as the computed keys resulting
from the agreement process.
==PAGE:16==
1.4 TLS 1.3
9
1.4.1.1
An example handshake
The basic full TLS 1.3 handshake is illustrated in figure 1.4. The RFC abstracts the protocol in to 3
phases:
• Key Exchange (Key Exch)
Establishing and selecting key-material as well as cryptographic parameters. After this, following
fields are encrypted
• Server Parameters (Server Params)
This phase establishes handshake parameters, such as whether or not the client is authenticated,
which application layer protocols are supported, etc.
• Authentication (Auth)
This phase authenticates the server, and optionally the client. It also provides key confirmation
and handshake integrity
At the end of a handshake, the client should have been able to authenticate the server and if needed
the server can authenticate the client as well. Key material has been shared and a session key derived.
1.4.1.2
Extensions
Extensions allow for more options and wishes to be communicated by the client. Extensions are the
building block of TLS 1.3, with some extensions being mandatory, while others can range from optional
to recommended. They also allow the protocol to grow with future revisions and supported extensions.
We cover the functionality of extensions in chapter 3.
For the sake of completion we briefly acknowledge the record protocol:
1.4.2
The record protocol
With the handshake completed, the protocol proceeds to the record stage. Where application data is
transmitted. This stage has been configured by the handshake, and the signature of it is dependent on
how the handshake was carried out. It contains a variety of security measures to protect against the
attacks described earlier. For the full specifications on how the record layer operates, we one again refer
to the standard [Res18].

1.5
Motivation
TLS 1.3 is a very powerful protocol, it is extremely flexible and solves a lot for problems that arise
when connection devices over the internet where trust might not be a guarantee. It then makes sense
that many different applications could be found for such a protocol, even if it was originally designed
to run in a classic client/server configuration. The question is then, how flexible is this protocol actually?
If TLS 1.3 is used both on constrained devices and on large server networks, does it matter if they
are configured the same? What is the cost / benefit of using extensions other than the mandatory ones.
Could using more extensions give better performance? Finding out what the performance impact of
==PAGE:18==
1.6 Rest of the report
11
TLS 1.3 extensions are could give an insight into whether or not it is feasible to tailor specific TLS 1.3
extension configurations to various classes of devices. Things such a throughput, memory usage, pro-
cessing time or round trip time as well as the impact on security the extensions have, could help software
architects and developers make more informed decisions. By only looking at extensions, any improve-
ments or concerns found could be mitigated and/or implemented without the need to re-implement the
entire platform.
1.5.1
Aim of the report
With this report we hope to contribute in the following ways
• Carrying out a performance analysis on TLS 1.3 extensions
• Providing a foundation for further research into the topic of TLS 1.3 extensions
• Providing configuration recommendations of TLS 1.3 extensions based on the results of the anal-
ysis.

Other performance studies

• ECDHE-ECDSA, AES-128-CCM
• ECDHE-ECDSA, AES-256-CCM
It finds that for TLS 1.3, we see a slight increase in bytes-over-air but a energy reduction, a slightly

Finally the authors note the huge difference in energy consumption the choice of authentication method
provides. Instead of looking at low-power devices; [Sy+20] looks at the internet as a whole. The paper
presents the idea that since the TLS 1.3 handshake is that most demanding part of establishing a secure
connection, doing less of it whenever possible, would reduce loads on servers.
They provide the notion of resuming sessions on shared SNI’s,(eg. api.google.com and google.com).
They provide an overview of how many TLS connections are prequired to load a page on average, look-
ing at the top 1000 Alexa pages. To improve the performance, they propose an extension that allows
for TLS resumption accross hostnames. The authors show that using their proposed methods could
reduce the number of TLS handshakes and in the scenario of sequential handshakes, lead to a 30.6 %
performance gain in terms of RTT and 44.6% less CPU time. All by reducing the total number of
handshakes required.
Similarly we also find [Hil+19] featuring an empirical study that promises to increase TLS perfor-
mance for energy and latency. They do this by reducing the overhead from the protocol by means
similar to the ideas presented in [Sy+20] where sessions are shared, thus avoiding multiple handshakes.

Issues with the PSK extension
PSK is a very common approach to getting more performant TLS. [DG21] shows that if a key is shared
out of band, a reflection attack known as the selfie attack becomes viable. We refer to the paper for a
description and discussion of the attack. The authors propose two mitigation paths:
1. External PSKs MUST be used together with server certificates.
2. A PSK MUST NOT be shared between more than one client and one server.
The authors acknowledge that option (1) is not viable since PSK is used exactly to avoid certificates.
The paper also features a lengthy discussion on provable security, which is out of the scope for this
project. The discussion of external PSKs is furthered in [Akh+20], it reaches the same conclusions
as [DG21] but provides an example of groupchats and how this flaw ties in to the Post-Handshake
Authentication extension.

Discussion
Looking at table 2.1 of the performance studies found, extensions were either newly proposed or a part
of a requirement for another aspect of the study. Others propose not only extensions but redesigns of
TLS 1.3 to support. No studies of the performance impact of the extensions themselves were found.
Table 2.2 focuses on the part of the literature that identified the reflection attack on the out-of-band
PSK, apart from the security recommendations, it also shows that extensions can have serious security
implications, and as such should be treated with some care when proposing or advocating for their use.
The metrics tested in the papers covers what would be reasonable for the kinds of studies that were
carried out
In table 2.3 we also see little discussion of extensions, apart from the arguments against the use of
a non-standard extension. In terms of performance metrics and simulation of real-life conditions these
papers had metrics such as https load times and packet loss as part of the measurements.
Table 2.4 shows that in terms of quantum security, no papers considered how extensions would im-
pact the performance of the PQ TLS Handshake. The key metrics for these papers are not surprisingly

related security and execution time of cryptographic implementation. Papers here also find unconven-
tional ways of reducing slowdown by also configuring the TCP layer. A highlight from this seciton was

3.1
Server Name
3.1.1
Description
If multiple servers are running (virtually) on the same network address, it might be desirable for the
client to specify which server to connect to. This extension allows the client to include a host name as
part of the extended handshake, so the correct server can handle the request and respond.
3.1.2
Testing motivation
If a server a client want to contact a specific server on a network, the use of the server name extension is
a requirement that the client can not get around. Being forced to use the extension means that finding
out how much overhead it introduces in the usage of the extension is quiet crucial, and if it’s to much,
finding a alternative method to the same problem might be needed.
3.2
Max Fragment Length
3.2.1
Description
The Max Fragment Length extension specifies the maximum size of a plain text fragment. This allows
constrained clients to negotiate a smaller fragment size for less memory use per message, or to overcome
bandwidth limitations. The default fragment size is 214 bytes, and the enum provided scales as follows
29(1), 210 (2),211 (3) 212(4). Furthermore, wolfssl implements it’s own fragment length of the sizes
213(5) and 28(6). After a non default fragment length has been negotiated, the client and server must
immediately begin fragmenting their messages.
3.2.2
Testing motivation
By testing this extension we hope to see how the performance is affected by fragmenting packets. We
expect to see the time increase as fragments get smaller, but how the time increase takes shape will be
interesting.
3.3
Trusted CA Keys
3.3.1
Description
If a client has limited memory, it might only have a short list of CA root keys, and as such indicate
that it only trusts a limited number of CAs. Sending this extension may reduce the number of failed
handshakes.
This extension is NOT part of TLS 1.3, but can appear in TLS 1.3 handshakes for
compatibility reasons.
3.3.2
Testing motivation
This extension can appear in TLS 1.3 handshakes for compatibility reasons. If for any reason a system is
required to be backwards compatible with TLS 1.2, it might be interesting to know how much overhead
on the TLS 1.3 communication this brings.

3.11.1
Description
The main point of TLS is to establish a secure connection. This is done through a exchange of keys

we should use, depending on the clients enviorment.

CHAPTER 4
Design
This chapter documents the design process of the bench-marking tool. The design process was broken
down into 4 phases
1. Choice of library
2. Selection of desired metrics
3. High-Level description
4. Low-level description
4.1
Choice of library
To make the project feasible, an already existing implementation of TLS 1.3 would be chosen and used
as platform on which the tests were to be conducted. Since a few implementations exist, the choice
comes down to features and preference. An important feature is the number of extensions the library
supports, since that is what we are evaluation. The library should be well documented and maintained,
such that any questions we might have could be answered by an expert. Based on the the literature, we
also wanted the library to have IoT in mind such that the library was not only optimized for execution

4.3.1
Solving the issues with the chosen approach
During the overview of approaches we identified some key issues and challenges that a given design will

us a list of further design challenges.
• Limiting the scope.
• Limit side-effects.
• Accounting for overhead.
• Deciding on which data to save.
• Saving benchmark data.
4.3.1.1
Limiting the scope
To avoid exceeding the scope of the project, we decided that certain things would be done manually,
such as static data structures and configurations. We also decided that rather than trying to provide
one ”super” configuration for the server, the server and client would be restarted multiple times to
simplify work.
==PAGE:37==
4.3 High-level design
30
4.3.1.2
Limit side-effects
WolfSSL is thread-safe and avoids global objects as a rule. However, when it came to making the choice
between being less invasive or avoiding global object, we decided that our design try to not interact
with the low level code. And where it does, it should only read values, never write. Following this rule
we can be sure we add no side-effects to the wolfSSL code, but we might ourselves experience issues
with global data. This trade-off was made on the basis that slowing issues in our limited benchmarking
code, would be much easier than in the full scope of wolfSSL.
4.3.1.3
Accounting for overhead
Whenever we add new code, we must also have a good idea of how the added code affects the results
we observe. Since we are primarily going to deal with program size and execution time, it make sense
that we should keep track of how much memory our bench-marking additions use and the extra time
spent. Because we decided to use statically-sized data structures we can subtract that from the results,
similarly we timestamp starting and ends of the bench-marking to get an estimate of timespent taking
measurements.
4.3.1.4
Deciding on which data to save
To compliment the basic metrics, we want to save the following extra data points:
• Timestamp
A timestamp in µs that marks the system time when the measurement was taken
• Name
To identify where a sample was taken, the benchmark data needs a way to identify itself
• Max Memory
A way to see the maximum RAM usage so far
• Output Buffer Size
The number of bytes in the wolfSSL output buffer at the time of the sample
• Time Overhead
The time in µs it took to produce the bench-marking data
4.3.1.5
Saving benchmark data
Because we need to transfer data between two different program, we need to find a good way to do this,
the choices were writing and reading to the same file, or reading directly from stdout. We decided
on reading from stdout because we already have a time-overhead calculation, meaning the extra time
spent printing wont affect the results in a way we can not account for.

To better explain how wolfSSL is used, we provide a short example of how to turn a normal application

is for an arbitrary client application, similar steps need to be taken for a server, the only differences
being loading server methods as opposed to client methods.
1. include the wolfSSL header
#include <wolfssl/ssl.h>
2. Change all calls from read() or recv() to wolfSSL_read()
result = read(fd, buffer, bytes); →result = wolfSSL_read(ssl, buffer, bytes);1
3. Similarly change all calls from write() or send() to wolfSSL_write()
4. Before any calls to the wolfSSL library, add a call to wolfSSL_Init()
5. Initialise the wolfSSL context, this must be done after the call to wolfSSL_Init()
WOLFSSL_CTX* ctx;
ctx = wolfSSL_CTX_new(wolfTLSv13_client_method();
6. Create wolfSSL object and set the file descriptor after TCP connect.
WOLFSSL* ssl;
ssl = wolfSSL_new(ctx);
wolfSSL_set_fd(ssl, fd);
7. Add cleanup after conenctions
wolfSSL_free(ssl);
wolfSSL_CTX_free(ctx);
wolfSSL_Cleanup();

5.2.6
Race conditions
The implementation poses a risk for side effects, but it is mitigated by running a control run before
the many sequential runs. The two risks are: Reading the output buffer, and printing the timestamps.
because we do not change extensions during a sequence, we do not consider it likely that any output
buffer will be incorrectly read, and since all timestamps are printed as one operation with a timestamp,
the order in which they are output can be resolved by simply sorting afterwards. We do not believe
there to be a risk of these race-conditions affecting our data, but we acknowledge the potential flaw in
our implementation, and proceed to the data processing with this in mind. More in Chapter 6

CHAPTER 6
Methodology
This section outlines how our experiments were carried out and how we processed our data.
6.1
Caveats regarding the experiments
6.1.1
WSL2 and subprocesses
Our experiments are were carried out by running the super-visor program on a desktop computer, with
a AMD 7 3700X 3600 Mhz 8 Cores cpu, a geforce GTX1070 graphics card and 16 GB of ram. This
means that both server and client is running on the same networks, as such we expect to see very low
RTTs. Super-visor is running in Windows Subsystem for Linux (wsl2) and as a result the spawned
processes for the client and server, are also running in that environment. Furthermore, because the
implementation spawns a sub-process, the memory measurements will be slightly affected, since some
overhead is introduced. It also forces us to use the RUSAGE_CHILDREN instead of RUSAGE_SELF, since
this would include the parent process of the super-visor program, and leave the total memory used
indistinguishable.
6.1.2
Using the pre-made wolfSSL benchmarking utility
The wolfSSL avg_connect time only measures in milliseconds, and takes the average on the fly, as a
result of this, we don’t have access to the samples it took. This is why we can only provide a single
value per extension. The data is output at the end of a all N sessions, which means that theoretically,
a small memory overhead of a few variables will be present, but we consider this negligible.
6.2
Procedure
6.2.1
Setup
To prepare wolfSSL to run in WSL2 we ran the program dos2linux to convert all line-endings to the
correct format. We also made sure wolfSSL was correctly installed with correct permissions. For some
tests manual compilation arguments were made in order to generate the data. An overview of this is in
table 6.1.
6.2.2
Running
• Run the main.py file. This generates the data and process it.
• Run wireshark.py. This handles the wireshark data
• If the wolfssl data need to be processed differently, but still use the same data, make the changes
and run the dataProccess.py.

6.3.1
Timestamps
The data for the timestamps is found by taking the difference of the timestamps between the begging of
communication (In the Clients case ”Client Start” to ”Client Done”, in the servers case ”Server Accept”
to ”Success”)
However in the case of the early data extension, this is the only case where multiple messages where
send at the same time meaning that multiple Accepts and (and therefore) Successes where present on
the server side at the same time, but due to race conditions, these do not always appear in the order
as they come in. Thus one could mix up the order resulting in negative timestamps. Since there was
no way to identify a message from another, we simply remove the cases where the result would have
been negative. This does mean that the timestamps for Early data on the server is not 100% correct
and prone to errors, but this is something we just have to accept.

8.5
Extensions
In this part we will go through each of the extensions and discuss whether the data we see from wolfssl
and wireshark is as expected, and why we might see the results we do
8.5.1
Server name
The server name extension adds a name of the server the client is looking for. This is a simple field
that the server has check matches with it’s own name.
Therefore there is only expected to be a slight increase in bytes sent, from the inital server name,
but nothing else should change much.
Looking at the figures this matches what we would expect, where it is equal to or slightly above
the standard case. The only figure with which differs significantly is figure 8.3b, however this is also
the case for most of the other extensions, and on figure 8.3a we can see that server name has a lot of
outliers which fit into the value of standard case, so this could very likely just be caused by general
randomness.
8.5.2
Max Fragment Length
Max Fragment Length (MFL) is used to change the max length of a package if the network or a client
doesn’t support the normal max length 21̂4 for TLS. To allow this both the client and the server needs
to use resources to split the messages, and combining them again, as well as send more packages.
This means that more time would be used processing the packages before sending them for both
the server and the client. Sending a message has a overhead in order to make sure the message arrives
to the correct person, and the order of the messages that are send. This should be seen in the form of
more bytes being sent over the network.
Timing
On figure 7.1 we see that this is indeed the case, where the max fragment length 3 is closer to the other
extensions times which was to be expected since the max length is larger, and therefore closer to the
normal Max Fragment Length. The same can be seen (Though to a much smaller degree) on figure 7.3.
Bytes
On figure 7.10a we see that MFL 1 has more bytes as expected, but MFL has no discernable difference
from the other extensions. This could be explained by the client not sending any message exceeding
the max length, thus not needing to send any extra messages.
From the server however, on 7.10b there is a bigger difference on MFL 1, and a actual difference
on MFL 3, which indicates that the server is sending bigger messages, and thus needing to send more
messages.
RTT
For the unecrypted data RTT, which can be seen on figure 8.6c, there is no difference between them
and the standard case since the MFL has not been exchanged yet, and has therefore not taken effect.
However, on the RTT for the whole communication, the smller MFL can be seen to have a longer
RTT, likely because of the proceessing time to split and rebuild messages. This can be seen on figure
8.6d. The bigger MFL does not seem to take more time. This is probably because the amount of
messages needed to be proceed are so small they don’t make a overall difference

8.5 Extensions
65
Throughput
On figure 8.6a we see that MFL 1 has no changes, even though the RTT of the communication has
increased. This implies that the increase in RTT is from the server communication, which corresponds
to assumption that the server is sending the largest messages, and is therefore more affected by the
smaller max length
MFL 6
Max fragment length 6 is a wolfssl specific setting that is even smaller than MFL 1. On the figures it
can be seen that the difference between MFL 1 and 3 is even more amplified with MFL 6
8.5.3
Trusted CA Keys
For trusted CA keys, the client sends a list of CA root keys that it supports, and the server chooses
one they use. This is mainly used to prevent multiple handshake failures. However, since we are using
a client and a server designed to be compatible, this will have little to no effect on the handshake, and
might at most increase the amount of bytes sent to enable the extension.
Looking at the data, we see that this is the case. This is a positive for the since it shows that the
inclusion on this extension on restricted clients has little, to no effect.
8.5.4
Truncated HMAC
Truncated HMAC is used to save bandwidth by only using 80 bits of the hashed MAC.
One would expect to see less data being sent since it doesn’t need to send the whole MAC.
However looking at the data, there is nowhere that the extension does anything out of the ordinary.
The lack of less bytes could in theory be because that the extension itself uses as many bytes as it saves,
this however seems unlikely.
8.5.5
Status Request v1 & v2
The Status Request extension is used to check a certificate, using a status protocol such as OCSP, so it
can verify a certificate without sending Certificate revocation list, saving bandwidth.
The version two (With the value 17 rather than 5) does the same thing, but supports multiple
status request being send at the same time, with a prioritized list. Thus, when making the inital Status
Request, v2 is expected to be slower than v1, but after that they should be equal.
RTT
The point of the handshake is to save bandwidth, by sending the request in the handshake, but making
up for it by not having to send the information later on. Thus we expect the initial RTT to be slightly
slower, but the over all RTT to be slightly faster. Looking at figures 8.6c and 8.6d we see that the
status unencrypted data is indeed slower, v2 being even slower, but on the full communication, they
are both within the standard case duration, v2 having a slightly higher average.
Throughput
While the Throughput of the client is mostly unchanged, the server Throughput on figure 8.6b for v2
it’s significantly lower on average, implying that using the Status Request takes more time for the server
to handle.

8.5 Extensions

8.5.6
Supported Groups
All of the extensions uses supported groups as the standard. This ”extension” here is wolfssl’s internal
test, that increases the number of supported groups, as well as changes the priority of the supported
groups. It switches from secp256r1 to secp384r1. These are both NIST/SECG curves, using 256 and
384 bits.
The increase in bit length of the key generation would mean increased time in generation of the key,
the amount of data sent, and potentially the time it takes using the key.
Client/Server bytes
Looking at the bytes sent on figure 7.10a and 7.10b, we see that there are virtually no difference between
secp384r1 and the normal extensions. This is not what we expected, since we expected more data to
be sent to get the bigger key.
RTT
The 384 bits has a longer RTT on both the unecrypted, and the full part. See figures 8.6d and 8.6c.
This fits with the idea that the longer key requires more data to send and time to encrypt/decrypt.
Timing
secp384r1 uses more bits than the usual, so more time is expected to be used on this. This is the case
for both internal and wolfssl timestamps seen on figures 7.3 and 8.1
Throughput
The Throughput for supported group on both client and server, is largely unchanged. The data should
fill the same amount once encrypted, so this makes sense.
8.5.7
Pre Shared Key
Pre shared key (PSK) uses already established keys instead of creating new ones. This means that we
can cut out key share, as well as the certificate verifying.
This should result in lot less messages sent a lot less bytes being sent, since a big part of the normal
TLS handshake is the key exchange.
Client/Server bytes
Looking at the amount of bytes being sent from the client and the server on figure 7.10a and 7.10b it
is very obvious that PSK uses a lot less bytes than non-PSK extensions, just as we expected.
RTT
For the handshake on figure 8.6c we see that the unencrypted RTT of PSK is higher than the standard
extension. This happens because wolfssl sends the key share extension even though the PSK extension
is used. This means that even though there could be huge saves in data since we wouldn’t need to send
FFDHE or ECC (This is assuming that they have a shared PSK which is not always the case). But
since wolfssl sends both the PSK and the key share, we get a longer unencrypted RTT as seen.
On the otherhand, looking at the RTT of the full communication on figure 7.10f it is very clear that
PSK is much faster than any of the other extensions. This is what is expected, since a lot of the data
being sent by the other extensions to exchange the key is not needed for this extension, saving a lot of
time.
==PAGE:74==
8.5 Extensions
67
Timing
We expect the extension to take a lot less time from start to finish, since it is a lot shorter than the
other extensions. this is confirmed on the figures 7.2a, 7.2b and 7.3, where on the client the timestamps
are almost 1/4’th of the time, and on the server about 1/8’th of time
Throughput
Where the RTT, timing and bytes were exacly as expected, the throughput looks very interesting. The
throughput of the client, seen on figure 7.10c is about 3 times larger than the other extensions. But on
the server side, seen on figure 7.10d, the througput is almost 5 times less than the other extensions.
This would imply that the throughput of the client extensions are highely reduced during the
exhchange of the key, probably due to the need for generating a key before sending it to the server,
but when only encryption is needed, it can go up in speed, while the exact opposite is the case for the
server.
8.5.8
Early data
Early data is a extension on top of the Pre Shared Key Extensions, where data is send earlier in the
process, encrypted with the pre shared key (PSK).
One would expect it to behave very much like PSK except it should send fewer messages, which
should result in fewer bytes sent due to overhead and a shorter duration due to fewer messages.
In almost all cases early data matches that of PSK very closely.
However, in the unencrypted
communication, seen on 8.6c, Early data is noteably slower, which would come from the fact that it is
already sending data here.
On the client Throughput, figure 7.10c, the lower bound of early data is higher than that of PSK,
which could very much be because of the early data.
Looking at ther server time there is a clear difference between PSK and early data. (See figure 7.2a).
However, this is also where the race conditions take effect, so the data here is not necessarily 100%
correct
8.5.9
Post Handshake Authentication
Post Handshake Authentication allows for the client to request a authentication at any time after the
handshake. However the client example does not perform a Post Handshake Authentication, so the only
change from the standard case is a very slight amount of more bytes being sent.
Looking at the different data for Post Handshake Authentication, we see nothing of interest.
8.5.10
Key share
Since the point of TLS is to exchange a key, if we are not using a pre shared key, we must use this
extension. The normal case for a wolfssl client is to send both the key share extension with ECC, as
well as a FFDHE, and let the server decide which key sharing method to use. The normal case for a
wolfssl server is to to use ECC. This being the case means that there is little meaning in comparing the
key shares with anything else than themselves and the standard case, since all places where there is a
difference between ECC and FFDHE, they are expected to be closer to ECC.
Client/Server bytes
Looking at the figure 7.10a, it’s clear that ECC is uses about 10% less bytes than FFDHE and the other
extensions. You can also see that FFDH uses less bytes than the other extensions. This difference in
bytes as explained, because the client dosen’t send both the ECC key and the FFDH key, and ECC is
smaller because it is design to use smaller keys than non-EC keys.[Eff09]
==PAGE:75==
8.6 Summary
68
Looking at figure 7.10b we see a change. FFDHE now instead uses way more bytes than ECC. This
again occurs because ECC is designed to use less bytes, and the server discards the FFDHE, and instead
uses purely ECC.
This would mean enforcing the use of ECC on systems with low bandwidth could be beneficial.
RTT
Looking at figure 7.10e we see that the time for the FFDHE unencrypted communication is significantly
lower than ECC and the other extensions, cutting the time almost in half. However, differently from
the bytes, there seems to be very little difference between between ECC and the other extensions. This
would seem to imply that the RTT delay does not happen on the clients part, but the servers part,
when making the response to the client. It could be that the handling of the ECC key takes more time
than FFDHE on the server.
To get a better look at the difference between ECC and the other extensions we can look at figure
8.6c. Here we see that ECC is slightly lower on average, but only very slightly. This time save could
be from the fact that the client does not have to handle the FFDHE data.
The difference on figure 8.6d is the same as figure 7.10e, but with a slightly larger spread, since have
been seen more messages which can have variations. It does not seem like there is any difference here.
Throughput
As we just discussed, FFDHE sents more data than ECC, and does it in less time according to the
figures. This should mean that the throughput of the FFDHE should be higher than that of ECC, and
the other extensions. Looking at 7.10d, we that this is very much the case. Figure 8.6b shows that
ECC is very close to the standard case, which corrosponds with sending less data, slightly quicker.
Timing
Having a higher throughput one could expect the FFDHE to have a better timing, and looking at figure
7.3 it does seem like FFDH is about 10% faster on average than normal extensions, with ECC being
about 5% faster than the standard case.
But looking at the internal timestamps on figures 7.1 we see that FDHE is actually as fast or slower
than ECC.
8.6
Summary
Here we talk about whether the motivations for testing the different extenions, as disccused in 3, matches
the what we just discussed about the results.
8.6.1
Server Name
The server name extension showed very little to none overhead, and there seems to be no reason to try
and find a better alternative.
The overhead on this is so small that if you know the name of the server, using the extension makes
sense, since you don’t run the risk of contacting the wrong server, thus incurring more overhead from
re-conditioning. If you only know the address of the server, there is of course no reason to use this,
since there is no name to be sent.
We find that for the functionality this extension provides for cost of the overhead, the extension can
be freely used on large-scale or constrained devices without adverse effects.
==PAGE:76==
8.6 Summary
69
8.6.2
Max Fragment Length
The data clearly showed that there is a correlation between total number of bytes sent and how low the
Max Fragment Length is much as expected.
Max Fragment Length as a extension should be avoided if possible, since it only slows down the
communication, and only be used if the network has a bandwidth restriction. However in the case of
bandwidth restriction, it is recommended to use it, with as large a MFL as possible.
8.6.3
Trusted CA Keys
The inclusion of the TLS 1.2 extension showed to have almost no overhead. The use of this extension
comes down the use case: For systems where both client and server is under the control of the same
organisation the use of this extension is not recommended, since it will likely only add overhead. For
systems where multiple handshake failures is a concern, the overhead is minuscule enough to be worth
it to reduce the number of failed handshakes.
8.6.4
Truncated HMAC
The fact that there was no difference of note between this and the standard case, while reducing the
security of the handshake, means that even if this extension is implemented, it should be avoided
completely. Especially to those implementing wolfSSL. Because wolfSSL includes this as part of the
--enable-tlsx command, we can not reccomend compiling the library using this, without also explicitly
disabling the Truncated HMAC extension
8.6.5
Status Request v1
There was no obvious save in bandwidth for using this extension, but there was no big overhead either.
The data was inconclusive on whether Status Request v1 had a positive impact on the communi-
cation, but there was also negative impact. Since IANA recommends using it, so will we, but further
research should be done on this extension.
8.6.6
Supported Groups
As expected it took more time to use secp384r1 rather than secp256r1, but what was not expected was
the fact that the same amount of bytes where send through the whole communication.
Since secp384r1 is slower, it’s recommended to just use secp256r1. However if you are working with
very sensitive data, such as data qualified as TOP SECRET or equivalent [15], using the added security
of a the higher keylength of secp384r1 is recommended.
8.6.7
Status Request v2
This is the same story as v1. No big bandwidth save, but no big overhead either. The overhead is
however bigger than that of v1.
Status request v2 seemed to give no improvements, and only increased the overall time. This needs
to be researched further before we can make any conclusions on whether to use it.
8.6.8
Pre Shared Key
As expected PSK was much faster than the key sharing methods.
Due to the huge decrease in time used, if the client has a PSK with the server, it is ALWAYS
recommended to use this where possible rather than do a new key exchange.
==PAGE:77==
8.6 Summary
70
8.6.9
Early Data
The effects Early Data was a lot less noticeable than expected, which could come from the fact that
PSK already is so effective.
The data was inconclusive on whether Early Data had a positive impact on the communication,
but there was also negative impact. Since IANA recommends using it, so will we, but further research
should be done on this extension.
8.6.10
Post Handshake Authentication
The overhead of PHA is very small as excepted.
The opportunity cost of using RHA is very low, and if you are handling any kind of sensitive data,
making a authentication call before sending this data would seem advisable, so the extension should be
used. If you are handling absolutely no sensitive data, you could decide to skip on this extension.
8.6.11
Key Share
We could conclude that ECC uses less bytes than FFDHE, but FFDHE makes up for it by being quicker.
On most lowend networks, where bandwidth is a issue the sheer amount of bytes saved from only
sending ECC can be a big factor. However if bandwidth is not a problem, there is only 10% bytes saved
by doing either or, and having to redo a handshake because the server does not support the key share,
takes way longer, so on highend we recommend that you send both, with ECC as the prioritization.

CHAPTER 9
Future Work and suggested
contributions
Because we found a gap in the literature when it comes to performance evaluation of TLS 1.3 extensions,
we found that throughout our work on the project, a lot of aspects would be interesting to investigate.
We have compiled these as a combined future works and contribution suggestion, to help further research
into this topic.
9.1
Testing in real network conditions
The data of this report was gathered on a local machine running both instances of client and server.
This leads to some very narrow tolerances when it comes to timings, and very minor things can disturb
the results to a larger degree. Performing these tests on a real network with real traﬀic would be a
natural next step of this kind of study. This could be done similarly to [Tib+17] and [SKD20]
It would be interesting to investigate how servers not configured in tandem with the client would
react to the suite of extensions. In our tests the server and client were guaranteed to be configured
the same. Further investigation the prevalence of various configurations to provide an additional di-
mension of which configurations are most likely to succeed with the fewest attempts possible. Finally,
investigating how failed attempts affect the time to connect and if any differences can be found within
the extensions, to provide some kind of ”fast failing” configuration, which might be useful in highly
concurrent large scale applications.
9.2
Testing on constrained devices
A lot of the existing research on TLS 1.3 deals with constrained devices (see chapter 2), this also provided
a lense through which this report was guided. Continuing the research done in this report on constrained
devices is another natural step forward. Especially testing various constellations, where both server and
client runs on constrained hardware, or any mix between constrained and non-constrained hardware.
As part of this, adding power consumption to the list of metrics would be valuable, and the hard-
ware specific implementations could be tested in a similar way to the extensions have been. Analysis of
implements implementations can be made to try distinguish them from each others based on extension
timing.
9.3
Testing of non-implemented extensions
As shown in table 3.2 there are a large list of extensions not tested by this report. Testing of any subset
of these would yield a better insight into the options available to those that implement TLS 1.3. This
future work further increases in scope if combined with other future works
==PAGE:79==
9.4 Testing of interleaving extensions
72
9.4
Testing of interleaving extensions
Some extensions tested depend on another, as an example, in wolfSSL Early Data requires Pre Shared
Key. However, a lot of extensions can work independently of each other, and investigating how combin-
ing such extensions across various configurations would affect performance is future work, as we aim to
give a basic overview here. Generally the trade-off is time versus memory, but with more extensions
and more capabilities a more in-depth survey into the use cases of highly custom TLS 1.3 configurations
could supplement the performance analysis.
It would then also be interesting to investigate if the current standard configuration is the best general
purpose configuration of TLS 1.3 or if an interleaving of extension could prove to be better over all.
9.4.1
Recommended interleaving for future testing
As part of our own investigation, we have identified two configurations that might be interesting to
investigate further.
• Max fragment length & Early data: MFL tries to minimize package sizes, but at the cost of
more packages being sent. Early data tries to minimze the amount of packages sent by using all
packages to send data, but requires a PSK. Looking at how effectivness Early data is depending
on the MFL size could be ver interesting.
• Supported Groups & Key share (FFDHE): This is mostly for checking the proper usage of the
Supported Groups, since only one of the send Supported Groups works with FFDHE
9.5
Testing of large-scale networks
If more work is done while simulating the network traﬀic, simulating a high traﬀic system with many
servers would be an interesting direction for research. Investigating how well the operations scale and
if certain extensions scale better than others.
Situations such as low bandwidth per connection could be tested and the cost/benefit of using fewer
extensions is better or worse than using more. This could also be extended to investigate the extensions
that works by requesting checks from Certificate Authorities, such as Status Request.
9.6
Testing of non-homogeneous hardware
Because all tests were run on one machine, we might lose insight in how certain extensions run on
certain classes of devices. Testing this on other hardware such as IoT, Laptops, Phones, Dedicated
server boxes and even sensor networks and comparing the impacts of the extensions.
One could also propose using any of the above mentioned devices as a TLS 1.3 server, and investi-
gating how each might perform and if using the correct extensions could facilitate a mesh-network
based on TLS 1.3.
9.7
Testing of proposed extensions
Proposed extensions for TLS 1.3 are found in the literature. It would be interesting to survey them
and find comparable extensions that can be evaluated to each other, since most papers already compare
to the extensions described in the standard. However, often these studies only covers the standard
configurations of TLS 1.3 and could be extended with more test cases.
==PAGE:80==
9.8 Refinement of the internal benchmark tool
73
This also opens up for the possibility to investigate TLS 1.3 alternatives and if there exists a set
of extensions that could approach the proposed alternatives from literature.
9.8
Refinement of the internal benchmark tool
As seen in the figures like figure 3 in the appendix, the checkpoints from the show key timings, but more
time could be spent finding checkpoints to get more gradual timing data, such as identifying time spent
building network packets, performing cryptographic operations, waiting for packets etc. Optimizing the
program to support after-the-fact decryption and extracting the key-data used could also add a layer
of robustness to the data.
As described in Chapter 4 one could also take a manual debugger approach to the data collection,
but another future work option to explore would be automated debugging scripts extracting data by
running in non-real time. This would open up for a different kind of metrics and focus more on the
on-board performance as opposed to the network impact.
9.9
Testing of other implementations
This project was based on the wolfSSL implementation, because it provides many useful features and
has many developer friendly perks, such as many ready to go device specific implementations. However,
comparing how one implementation handles extensions may vary. It may also help identifying if results
are a result of implementation quirks or if the are a result of specification.
It might even be worth exploring the idea of an TLS 1.3 implementation built with analysis in mind, a
compromise between speed and being able to easily monitor the internal state.

CHAPTER 10
Conclusion
We conducted a study of the performance of TLS 1.3 extensions, we have done this because there
is a gap in the literature for this kind of investigation into TLS 1.3.
To do so, we first surveyed
the literature for other performance studies to consolidate methodology and metrics. Then a design
consisting of a bench marking system was made, the design described an automated system to configure
a client/server pair, run them and capture data. The data captured comes both from program output
and from wireshark. We detail the implementation of such as system and provide a methodology section
to document our use of the tool. We present the results from the analysis and discuss the data. We find
that most of the extensions behave in a way expected of them. Max fragment Length and a greater key
length in supported groups had a negative impact. Pre shared key had a huge positive impact. Post
Handshake Authentication, Trusted CA keys and Server name had a negligible impact. Key share saw
some positives for using either ECC or FFDHE, where ECC sent fewer bytes, but FFDHE had a better
throughput.
The once that did not behave as expected was Status Request v1, v2, Truncated Mac and Early
data. These are supposed to have a positive impact on the data sent, but seem to have none or a
negative impact.
Based on these findings we produce a number of recommendations from the extensions: Use Server
name if you have one, and Post Handshake Authentication, unless you are certain you won’t need it.
Do not use Max Fragment Length unless absolutely necessary, and always use as a large a length as
possible. Only use trusted CA if necessary. Avoid Truncated HMAC at all cost. Only use larger keys
in supported groups when handling sensitive data. Always use pre shared key when possible. Unless
necessary send both FFDHE and ECC, but always prioritize ECC.
Finally we recognize that the topic of TLS 1.3 extensions is vast, and that a lot of studying is still
to be carried out. To aid with this we produce a number of suggestions for further research and topics
related to the evaluation of TLS 1.3 extensions. The lessons learned during the project were: Not all
implementations of the TLS 1.3 standard follow the standard completely, as such working with a library
that is well documented, is crucial for the success of the study. It also is apparent that any evaluation
study will only be representative of the implementation, a fully comprehensive study needs to include
multiple implementations. As an extension to this, it is diﬀicult with a single library to conclude of
any notable data is the result of implementation, or specification. Finally, we believe that there is still
much to gain from investigate the performance of TLS 1.3 extensions.
