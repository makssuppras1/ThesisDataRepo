Summary (English)
This thesis investigates how, the use of a model-based debugging tool can help
developers locate modeling errors for models describing a domain model with-
out interacting with any code. The thesis also deals with the development of
such model-based debugging tool for the model-driven development notation,
"Event Coordination Notation".
Several studies have shown that the key feature of a debugging tool is to clarify
what the system is actually doing, and collect information of the internal values
in the system and present the information to the user. Since developers use that
information when they have to locate the root cause of the bug in the system.
The thesis extends the existing framework, The ECNO Tool, developed to
realize The Event Coordination Notation. The ECNO tool does not support
debugging on an abstraction level equal to the one used the ECNO tool
, where it is only possible to debug at the code level.
To develop the model-based debugging tool, it was essential to explore what in-
formation was important to illustrate, when developers had to locate modeling
errors.
In light of the above, modeling errors have been analyzed and the debugging
tool for the Event Coordination Notation has been designed. The design has
then been partially developed, so the debugging tool can visualize many of the
modeling errors that can occur when trying to model systems with the Event
Coordination Notation.
==PAGE:4==
ii
Some special features like: inheritance and "counting events" from the Event
Coordination Notation, has not been prioritized, but the possibility for extend-
ing the debugging tool to handle these features are discussed in the thesis.
The developed debugging tool was tested subsequently by modifying models
that already described a domain. The debugging tool was then used to locate
errors that had occurred from the modiﬁcations. Based on the tests, it is clear
to see that, the developed debugging tool is able to use a high abstraction level
similar to that used by the Event Coordination Notation.
The developed debugging tool was then compared with other debugging tools
according to three criteria:
• How well the debugging tool would show why computations of interac-
tions are valid or invalid?
• How much time would be used to follow the computation ﬂow of an
interaction?
• What abstraction level is used by the debugging tool to visualize the
information?
The comparison showed that the developed debugging tool in general would
perform better at displaying the information according to the listed criteria.
Based on authors own evaluation of the criteria, it is concluded that the advan-
tage of using a debugging tool, that has been specialized to a speciﬁc notation
is signiﬁcant better than that of using a more general debugging tool.

Summary (Danish)
Denne afhandling undersøger hvordan brugen af et modelbaseret debugging-
værktøj kan bidrage til at lokalisere “bugs” i systemer, beskrevet ved hjælp
af modeller. Derudover beskæftiger afhandlingen sig med en udvikling af et
modelbaseret ”debugging”- værktøj for modelleringsnotation kaldet, ”Event
Coordination Notation”.
Afhandlingen udvider det eksisterende framework ("The ECNO tool") udviklet
til at realisere “Event Coordination Notation”. Frameworket har dog ikke et
debugging-værktøj, der opretholder det abstraktionsniveau, som benyttes af
“Event Coordination Notation”. Det var derfor kun muligt at ”debugge” på
kodeniveau.
For at kunne udvikle det modelbaserede debugging-værktøj, var det væsentligt
at undersøge hvilke informationer der var vigtige at skildre for at vise fejlene,
der kunne føre til opklaringen af modelleringsfejlene.
Flere undersøgelser har vist, at det vigtige for et debugging-værktøj er at klar-
lægge, hvad systemet foretager sig, samt indsamle informationer om de interne
værdier i systemet. Det er nemlig disse informationer, der benyttes når udvik-
lere skal ﬁnde årsagen til de fejl, som opleves med systemet.
På baggrundafovenståendeermodelleringsfejlblevetanalyseretogdebugging-
værktøjet for “Event Coordination Notation” er blevet designet. Designet er ef-
terfølgende blevet delvist udviklet, så debugging-værktøjet kan imødekomme
mange af de modelleringsfejl, som kan opstå når man forsøger sig med “Event
Coordination Notation”.
==PAGE:6==
iv
Der er dog nogle specielle muligheder såsom: nedarvning og ”counting events”
som ikke er blevet prioriteret, men muligheden for udvidelsen bliver diskuteret
i afhandlingen.
Det udviklede debugging-værktøj er efterfølgende blevet testet ved at modiﬁ-
cere modellerne, som allerede beskrev et eksisterende system, nemlig ”Worker
example”. Herefter blev debugging-værktøjet benyttet til at lokalisere fejlene,
der var opstået. Ud fra testsne er det tydeligt at se, at det udviklede debugging-
værktøj formår at benytte et højt abstraktionsniveau svarende til det, der er
benyttet af “Event Coordination Notation”.
Debugging-værktøjeterefterfølgendeblevetsammenlignetmedandredebugging-
værktøjer i forhold til tre kriterier:
• Hvor godt skildrer “debugging”-værktøjet, hvorvidt udregningerne af
interaktioner er gyldige eller ugyldige?
• Hvor meget tid skal benyttes for at forstå udregningen af en interaktion?
• Hvilket abstraktions niveau benytter “debugging”-værktøjet til at skildre
informationerne?
Sammenligningen mellem debugging-værktøjerne gav et resultat, der viste at
det udviklede debugging-værktøj generelt ville klare sig bedre i overensstem-
melse med kriterierne end de sammenlignede debugging-værktøjer ville have.
Afslutningsvist er det af forfatterens evaluring af testsne blevet konkluderet, at
det er en fordel at benytte sig af et debugging-værktøj, der er specialiseret til
en bestemt notation fremfor at benytte sig af et generelt debugging-værktøj.

Preface
This thesis was prepared at DTU Compute in fulﬁlment of the requirements
for acquiring an M.Sc. in Engineering. The thesis describes the work done in
the period from February 2015 to July 2015 and is written under guidance of
Ekkart Kindler. The thesis accounts for a total of 30 ETCS points.
The thesis explores how using a model based debugging tools, speciﬁcally
developed for the Event Coordination Notation(ECNO), can help debug mod-
elling errors occurred using the Event Coordination Notation.
The thesis consists of the following chapters:
Chapter 1: Introduction to the context and motivation of the study.
Chapter 2: Research questions and hypothesis of the thesis.
Chapter 3: Deﬁnition of the ECNO and its elements.
Chapter 4: The computation methods for interactions.
Chapter 5: Analysis of modeling errors that can occur using the ECNO.
Chapter 6: Debugging as a skill - What are bugs and how do developers locate
them)?
Chapter 7: Design of the ECNO debugging tool.
Chapter 8: Exploration of interaction computations in the ECNO tool.
Chapter 9: Implementation and limitation of the ECNO debugging tool.
Chapter 10: Tests and use examples of the ECNO debugging tool.
Chapter 11: Comparing the ECNO debugging tool to other debugging tools.
Chapter 12: Recommendations for future work.
Chapter 13: Conclusion of the research questions

Chapter 1
Introduction
Bugs in software systems are irritating, diﬃcult to locate, and are estimated to
cost nearly $60 Billion dollars in USA [17].Therefore, it is necessary to under-
stand why the system enters an undesired state that results in a bug. This can
often be achieved with the use of a debugger.
In model-driven development (MDD), the developers use models to deﬁne the
systems. "This gives the devloper a high abstraction level that is closer to the
domain than the code realizing it" [13, page iii]. The event coordination nota-
tion (ECNO) is such a MDD notation, like UML, Meta Object Facility (MOF),
XML Metadata Interchange (XMI), and the Common Warehouse Metamodel
(CWM) [18]. Common for these MDDs are that they all use class diagrams to
deﬁne the classes in the system. The ECNO diﬀerentiates itself from the others
by modeling the local as well as the global behavior of the system. Other MDD
notations does not model the global behavior but leaves it up to the developer
to code the global behavior. This additional modeling of the global behavior
allows, the ECNO to generate programs only using models to deﬁne the system.
Like in normal software development where systems are designed through
lines of code, bugs might occur. The ECNO is no diﬀerent since bugs can occur
due to human mistakes while modeling the system. It is therefore important
==PAGE:22==
2
Introduction
to be able to debug the system generated from the ECNO models. The infor-
mation acquired during such a debugging session should make it possible to
locate the bugs in the models and repair them. The problem is however that
there currently does not exist a debugging tool, which keeps the abstraction
level for the ECNO. The only way to debug systems modeled by ECNO is to
use a debugger on the code generated from the source models. This is not
beneﬁcial since the design is done with models and it is not always possible to
directly map an error found in some code to the source model.
This thesis contributes with a basic debugger for the ECNO Tool that realizes
ECNO, the so called ECNO Tool[13]. The developed debugging tool allows in-
vestigation of design errors in the source models. The debugger uses graphs to
illustrate how computations of valid and invalid interactions were performed
in the system. This helps the developer to determine whether or not the models
are behaving as intended, or some deign error was made. The thesis prioritized
to show how a model-based debugging tool, can be used to locate modeling in
the core parts of the ECNO.
This thesis explores how humans use debuggers to locate errors in software
systems as well as how ECNO can be used to model domains.

Chapter 3
ECNO
The Event Coordination Notation (ECNO) is as mentioned a MDD notation,
meaning that the ECNO uses models to described the system. The design of the
system in the ECNO is done through the use of class diagrams, local behavior
diagrams, and global behavior diagrams.
Since the ECNO uses models it
allows the developer to "model the behavior of an application on an abstraction
level that is higher than the code realizing it" [13, page iii]. The beneﬁt from
modeling system is that the abstraction level is closer to the domain model.
Using models minimizes the time required to produce the system describing
the domain since, it is possible to generate code automatically from the source
models. Keeping the abstraction level as close to the domain as possible also
makes the software more reliable. This is because "the purpose of a domain
model is to concisely capture the concepts of an applications domain, and
their relation among other." [13, page iii]. Therefore with the abstraction level
closer to the domain model, than that of the code realizing the domain, the
mistakes regarding the system will be minimize. The ECNO extends the notion
of objects and classes from object-oriented notations. To make sure that it is
possible to distinguish between objects in the ECNO and objest in the oject-
oriented notations this thesis will use Elements for objects and Element types
for Classes. This should help to distinguish between situations where object-
oriented perspective and the ECNO perspective are being described. Before
going further into any details about ECNO, an example taken from section 1.1
in the technical report [13] is given.

3.2
Events
An event is an instance of some speciﬁc event type, just like an object is an
instance of some class in the object-oriented notation. Instances of event types
is therefore be called events. Events are as described in [13, page 42]: "used to
coordinate and synchronize diﬀerent elements in an interaction". Later in section 3.6
further details about what an interaction is, will be examined.
Events has
parameters just like an object can have parameters, which can be set when
the object is created or with some setter method. Events are diﬀerent from
this. Events are not of the typical caller and callee type, where the caller can
passe arguments to the callee and the calle can return information when it
is done.
Events only has participants.
The participants are as seen in the
example from section 3.1 deﬁned through the use of a coordination diagram,
which will be further described in section 3.5. Any information that is available
for the event is also available to all the participants in the interaction. The
participants therefore has the possibilities to deﬁne the event, by assigning
values to the events parameters, which is then known to all of the participants.
How the information speciﬁcally is transferred to the events parameters from
the participants is described further in section 3.4.
3.3
Elements
Elements are just like events deﬁned by types and instances. The instances
of the element types are referred to as elements. Element type are the same
for elements as classes are for object in object-oriented development. Further
elements actually are objects, with the small diﬀerence that they extends objects
by two speciﬁc things.
1. Element types are deﬁned by a life cycle, which describes the local behav-
ior of an element.
2. The life cycle is not deﬁned through methods, but rather through choices
which takes the element to diﬀerent states in its life cycle. Just like a
ﬁnite state machine, where objects can be in diﬀerent states, and actions
brings them to new once. The diﬀerence is though that the actions are
substituted with choice deﬁning event, which will be described further in
section 3.4.
Examples of such life cycles can be seen in: Figure 3.7, Figure 3.6, and Figure 3.5.
The whole idea of life cycles is further explained in section 3.4. For now it is

elements of that speciﬁc type. Another characteristic of elements are that they

3.4
Local behavior diagram
The local behavior diagrams of the ECNO model are called ECNO nets. ECNO
nets deﬁnes the life cycle of a speciﬁc element type. The life cycles can, as it
was illustrated in section 3.1, be ﬁnite as for the job element type in Figure 3.6
and inﬁnite as it was seen for the worker and car element types in Figure 3.5
and Figure 3.7. The ECNO nets uses the idea of transitions and places from
Petri nets [19] and [20]. Since it is not required to know Petri nets to understand
ECNO nets, they are not discussed further but instead ECNO nets are explored.
In the life cycles from the local behavior diagrams in the worker example from
section 3.1. The circles are the diﬀerent states elements of the given type can
be in. The rectangles or boxes are called transitions and represents the diﬀer-
ent choices the element can perform. Each choice is associated to some speciﬁc
event type, which will bring the element to the state pointed at by the transition.
An example of such a scenario is when a worker element is at the home state.
He has the choice to perform a transition using the choice deﬁning an arrive
event, which then brings the state of the worker element to the work state. This
is only the case if the remaining elements can participate in the interaction.
A choice is also associated to a set of event bindings, conditions and actions.
Event bindings will be described a bit later in this section.
Conditions are
logical expressions that deﬁnes some conditions for the element to perform the
given choice, where the actions are additional information that will be executed
when then element performs the choice besides moving to the state pointed at
by the transition.
The object diagram and the state of the local diagram for all the elements in the
object diagram deﬁnes a situation. The object diagram can be updated allowing
new possible situations, but this will be explained in section 3.6.
The initial state of an element is deﬁned by the ECNO net of the elements type.
The initial situation is deﬁned by all the ECNO nets for the elements from the
object diagram.

3.5
Global behavior diagram
The global behavior diagram of the ECNO is the core concept of the ECNO.
Other MDD notations extends the idea of creating life cycles for objects, but not
the global behavior.
The global behavior diagram deﬁnes what other elements has to participate in
an interaction given another participating elements choice deﬁning an event
together the references from the object diagram. The global behavior diagram
is also called the coordination diagram in the ECNO. The name has been given
for a reason, and it is because the global behavior diagram coordinates all the
local behaviors to each other. This coordination is done through an interaction,
where the elements choices deﬁnes together with the links in the global behav-
ior diagram, who has to participate given the events deﬁned by the elements
choices.
An example of a coordination diagram can be seen from the worker example in
section 3.1 in Figure 3.4. As mentioned in the example there was two diﬀerent
coordination quantiﬁers: ONE and ALL.
The ONE coordination quantiﬁer deﬁnes that only one of the elements of that
speciﬁc type, which the current element has a reference to in the object diagram
needs to participate.
==PAGE:35==
3.5#1
15
Just like in the worker example. If worker Ali from the object diagram want to
participate in a choice deﬁning a doJob event. Then one of the elements: "ab",
"abcd", or "acd" of the Job type would have to participate. Since the coordina-
tion quantiﬁer for the link associated to the doJob event type going from the
worker element type to the job element type is of type ONE.
Hence the ONE coordination quantiﬁer can be thought of as a logic OR, when
determining if a worker can participate with a choice deﬁning a doJob event.
The ALL coordination quantiﬁer on the other hand deﬁnes that all references
the current element has to elements of the other type in the object diagram has

This means that for the example before that if the quantiﬁer was an ALL in-
stead of a ONE then all of the three job elements had to participate for Ali to
participate in an interaction with a choice deﬁning the doJob event type.
Hence the ALL coordination quantiﬁer can be thought of as a logic AND, when
determining if the current element type can participate in an interaction, where
the type is associated to a link with the ALL coordination quantiﬁer.
Since a worker element can be assigned multiple jobs, there will be multiple
possible coordinations for the doJob event type for a worker element. Each co-
ordination are therefore stored in a coordination set. A coordination set keeps
information of the elements that are part of the current coordination.
A more formal deﬁnition of the coordination is given on page 47 in [13], which
says: "The meaning of this is as follows: If an element el of some type t partici-
pates in an event ev of some type et, then one of the coordination sets c of the
element type t for event type et is selected. For this selected coordination set
all the coordination annotations associated with the coordination set c must be
met for the element el and event ev. By allowing more than one coordination
set for the same event type for an element, we can model a choice of diﬀerent
coordination annotations that need to be followed up together for that event
type. This allows us to formulate disjunctions of conjunctions of coordination
requirements for each event type.
We still need to deﬁne what it means that a coordination annotation is met for
an element el and event ev. Let us assume that the coordination annotation is
associated with some reference r and L is the set of all the links of element el
with respect to r in the current situation. If the coordination annotation has
quantiﬁer ONE, one link l ∈L must be associated with event ev, too; if the
coordination annotation has quantiﬁer ALL, all links l ∈L must be associated
with event ev, too.
When a link l is associated with an event ev, this in turn requires that the ele-
ment to which the link points must participate in the event ev, too."
Given this formal deﬁnition it can be seen that the elements have to participate
according to the links. So if multiple links points to an element, the element
would have to participate this speciﬁc number of times. So as mentioned earlier

the element would only have to participate once in the coordination.
3.6
Interactions
An interaction in ECNO is a volatile actions, meaning that it is only present
in the short amount of time from when the interaction is computed to when
the interaction is executed. Interactions are a collection of some events and
participating elements choices given the links from the coordination diagram.
Therefore the local and global behavior models of the overall ECNO model
deﬁne the possible interactions in the system, for the speciﬁc situation. The
situation in turn is deﬁned by the state of each elements life cycle and the object
diagram.
For an interaction to be valid, all the choices associated to the participating
elements given some event must fulﬁll the requirements of the global and local
behavior models for the speciﬁc ECNO model. This has been further broken
down, to four speciﬁc points that needs to be fulﬁlled [13, page 49]:
1. "The choices associated with an element are enabled in the given situation in
parallel - as deﬁned by the local behavior."
2. "The values assigned to the parameters are compatible with the parameter assign-
ment of the event binding of every choice."
3. "The conditions of each choice evaluate to true (in the given situation and the
values of the event parameters)."
4. "For each element, the coordination condition as deﬁned by the coordination
set and the coordination annotations for each event that is associated with the
element are met."
If an interaction is valid it is possible to execute the interaction, meaning that
all the choice in the ECNO nets for the participating elements are performed
according to the given event type from the interaction. The execution might
update the object diagram and local behavior state of participating elements.
Hence this might generate a new situation of the system with new possible
interactions, which will be put in the set of possible interactions. The set of

Bug is an ambiguous word used by developers as a broad deﬁnition of the
collection of diﬀerent kinds of faults and errors in software system. According

error that has occurred. Several people have classiﬁed the diﬀerent types of
bugs, with case studies on novice programmers in Java [11] and C++ [21]. A

thesis will not go further into the diﬀerent kind of bugs, but just describe the
super sets of bugs:

Chapter 6
Modeling errors
When creating ECNO models, the developer might choose some references
and coordination relations which results in some unintended behavior of the
system generated from the source models. In the ECNO this would typically
result in interactions being valid or invalid when they were supposed to be the
opposite.
Several reasons can result in the interactions not behaving as expected. This
chapter explores some of the most likely reasons.
Since interactions are computed according to the situation and the global be-
havior diagram, it is most likely that errors in the three diagram types: Local
behavior diagram, Object diagram, and Global behavior diagram can cause
undesired results. Therefore possible modeling errors is explored from these
three diagram types.
The type of errors discussed in this chapter falls under the logic error deﬁnition
from chapter 5, since they are due to misunderstandings of how system should
be modeled rather than using the wrong model type.

6.1
Local behavior diagram
For the local behavior diagram errors will most likely occur from the diagram
deﬁning the life cycle of the element, or by the choices that deﬁnes the actions
taken in the life cycle.
6.1.1
Choices
Errors in the computation of valid or invalid interactions, can among others
occur from issues with the choices. The errors can be due to the conditions not
being properly described or it can be due to the assignment of parameters for
the event deﬁned by the choice.
These situations have impact on the ﬁnal stages of a verifying if an interaction
is valid or invalid.
6.1.2
Life cycle
The other mistakes that is likely to happen when designing the local behavior
diagram, is that the life cycle is not properly modeled. If choices are made
possible in the wrong states, it can lead to elements being able to participate in
interaction, when they should not be. It could also be the other way around
where the element cannot participate, when it is supposed to.
With the choices and current state of an element having impact on the compu-
tation graph, the mistakes made in the local behavior diagram will be reﬂected
in the computation graph.
6.2
Object diagram
If the wrong references were set in the object diagram, then during the compu-
tation of an interaction, some elements might be left out who were supposed to
participate or some might be required to participate and should not have. This
will result in computation graphs that are not as the user would have expected
it to be.
Reference errors could be discovered by looking at the computation graph for
the interaction. The computation graph shows as illustrated in chapter 4 which
elements could participate for the given interaction computation.

The last place where some modeling errors might occur is in the coordination
diagram. The problems in the coordination diagram can be separated in two
parts. Those errors that are due to the wrong coordination quantiﬁer being
used, and those errors that are due to having a bad link reference between
element types. Both of these error types will interfere with the computation
graph generated to ﬁnd a valid interaction.
6.3.1
Wrong coordination quantiﬁer
When it comes to the coordination quantiﬁer, there are two possible types to
choose: ONE or ALL. However they both makes signiﬁcant changes to the
computation graph. Given the computation example in chapter 4 it was seen
that the ONE coordination quantiﬁer would require that only one of the refer-
enced elements had to participate.
For the ALL coordination quantiﬁer all of the referenced elements would have
to participate.
Given the interaction computation where a worker initiated a choice deﬁning
a doJob event, then it would only require one Job to participate, which in turn
would not violate the assignment of the job parameter of the doJob event. In
turn the job would require all of its referenced workers to participate in com-
pleting that job.
If the coordination quantiﬁer for the link going from the worker to the job el-
ement type for the doJob event type, was modiﬁed from a ONE to an ALL as
it is in Figure 6.1. Then ALL references from the given worker element to job
elements would have to participate when computing the computation graph.
The validation would however even if the computation graph could be suc-
cessfully computed, be false. Unless only one job was part of the computation
graph. Since the condition of the job parameter of the doJob event type only
allow one job element to be assigned to the parameter.

Now also assume that a reference is set between the cars and jobs in the ob-
ject diagram. Then when trying to compute the computation graph it would
require that the car participate in the doJob events. The only problem is that
cars never have any valid choices deﬁning the doJob event. Since there exist no
choices deﬁning a doJob event in the cars life cycle.
So referencing two elements of diﬀerent types with one another and creating a
link reference between them. Does not necessary mean that they can commu-
nicate. As it turned out both element types has to have a choice deﬁning that
event.

The main objective for the ECNO debugging tool is to help the developer un-
derstand why some interactions are valid while others are not. However this
has to be done without loosing the high abstraction level gained by using the
ECNO to develop systems. It will then be the developers decision to decide
whether or not the observations of the interaction computations are correct.
Given the information of what developers use when they debug system. The
debugging tool provides them with the knowledge of what the actual program
does, when it computes interactions.
7.1
Features
As in many other software developments the features of a debugging tool can
be divided into two parts: Nice to have and need to have.
==PAGE:62==

7.1.1
Need to have
The features falling under the need to have section for the ECNO debugging
tool is:
A way to describe why the interaction is valid or invalid.
To see why an
interaction is valid or invalid. It is required to be able to see, the computation
graph computed for the interaction as well as whether or not all assignments
could be done and the conditions could be evaluated to true.
Given these information it is possible for the developer to see where it went
wrong in the computation and he will then be able to pin point it to some
location in the source models.
Another important feature is also setting up the scenario. It should therefore be
possible for the user of the debugging tool to execute valid interaction to bring
the system into the situation, which he wants to investigate some interactions
for.
7.1.2
Nice to have
There are also additional information besides the computation graph, param-
eter assignments and the evaluation of the condition which would be nice to
know.
Two of these features are:
• The actually value for events as well as elements references to each other.
• Current state of an element given the elements local behavior.
The parameter feature, will tell us what bindings have currently been set for
the event given the event bindings for the choices chosen by the successful
elements in the computation graph.
Having this feature will reduce the time to map the elements local behavior
diagram to the computation graph.
The current state feature will help the user by reminding him where his pre-
viously executed interactions has led the diﬀerent elements to in their local
behavior.
However these informations are not vital since it is possible to acquire these
information without a tool, showing them to the user.

7.1.3
Narrowing available interactions
Showing a list of all computation graphs computed for a system would be
confusing. Therefore it is important that the user of the debugging tool, can
narrow down the search of interesting interaction computations.
To narrow down the list of possible interaction computations. It should be
possible for the user of the debugging tool to select the initiating element and
the event which has to be deﬁned by a choice. Doing so will help zoom in on,
some speciﬁc interaction computations the developer want to investigate. The
list can then be further limited by having options for the user to select whether
or not he wants to see valid and invalid interaction computations, in the list
of possible interaction computation. It should then be possible to execute the
interaction from the interaction computation or investigate it.
These search features should help work as to decide when to investigate like
inserting break points does in the classical debugging tools working on the
code level. But since the abstraction level here is higher than that of the code,
it is not possible to mark some speciﬁc execution. Therefore this marking of
some speciﬁc situation where it would be interesting to debug, would have to
be deﬁned in another way. One way would be to let the user limit the list of
possible interesting interactions.

8.1
Classes used for interaction computations
In chapter 4 an example of a complete computation was given. It consisted of
elements, choices, events, links, and coordination sets.
The elements state deﬁned what choices they had given the local behavior
diagram. The available choices then deﬁned what events the elements could
participate interactions with. The interaction was then coordinated through
links between the elements of a speciﬁc coordination set.
This section will discuss the diﬀerent classes used to generate the computation
graph, rather than the speciﬁc element of the computation graph, like the Choice
class or Event class.
The ﬂow of the computation of elements are described by a an abstract class
named Continuation. The continuation is then implemented by the following
ﬁve diﬀerent classes.
ElementEventContinuation: The ElementEventContinuation describes a sin-
gle element, which is part of the current Coordination set.
ElementChoiceContinuation: The ElementChoiceContinuation deﬁnes one of
the possible choice for the current element, which is being computed as
part of the interaction computation.
ElementCoordinationSet: TheElementCoordinationSetContinuationdescribes
whether or not the link is an ALL or ONE coordination quantiﬁer. This
classes therefore deﬁnes the appropriated action to take depending on
the coordination quantiﬁer, which will be to either generate a new Ele-
mentEventContinuation or an ElementALLLinksContinuation.
ElementALLLinksContinuation: TheElementALLLinksContuniationcomputes
a list of ElementEventContinuation objects, which needs to be computed
to verify if the element containing the reference with an ALL coordination
quantiﬁer can be added as a successful computed element.
Failure: The Failure class deﬁnes a stop in the computation. It is used to tell the
ECNO Tool to stop the calculation of the speciﬁc coordination set, since a
failure of an element participation was found.
Using these a set of the ﬁve diﬀerent classes it is possible for the ECNO Tool to
generate the computation graphs.

8.2
Information available from the computation
Starting a computation of an interaction in the ECNO tool is often initiated by
a user using an application. To compute the possible interactions an Interaca-
tionIterator class has been developed. The InteractionIterator class works as
any other typical iterator with the two characteristics of having a next() and
hasNext() method. The next() method gives the caller of the method the next
possible interaction, which was computed, where as the hasNext() will only
inform if there are any other valid interactions.
The computation of the interactions are started inside the InteractionIterator
which is constructed for a speciﬁc Element and an Event. The element and
event two are then be passed on to the computation as described above. Since
the InteractionIterator can only give us the possible interactions, lets have a
look at what information a speciﬁc Interaction has of its calculation.
The Interaction class can give information of the participating elements and
their choice for the interaction, as well as information of whether or not the
conditions are fulﬁlled. However these are always fulﬁlled when they are re-
ceived from the InteractionIterator class, since the InteractionIterator class only
gives successful interactions, so if a condition cannot be evaluated to true, that
interaction is then discarded.
Using an object of the ExecutionEngine class it is possible to retrieve the
diﬀerent choices of the associated elements of the interaction. The Executio-
nEngine tells the application when something happens to the object diagram.
So if an element is added or removed the ExecutionEngine will now. The Exe-
cutionEngine also knows the underlying models for the elements, and can give
the information of their choices.

Chapter 11
Discussion of related work
With illustration of how the ECNO debugging tool displays the relevant debug-
ging information for computing interactions. It is time to compare the ECNO
debugging tool to other debugging tools, both in terms of what information
they provide, but also how the information is displayed.
Two types of debugging tools will be explored and compared to the ECNO
debugging tool. Namely the classic debugging tool and graphical debugging
tools.
11.1
Criteria
Three criteria has been used during the comparison of the other debugging
tools, to the developed ECNO debugging tool. Each of which are important for
the overall evaluation of the tools. The criteria are:
1. How good would the debugging tool be at informing the user why inter-
action computations are valid or invalid?
2. How much time would be used to follow the computation ﬂow of an
interaction?
==PAGE:102==
82
Discussion of related work
3. At what abstraction level would the information be visualized?
To evaluate the debugging tools to each of the criteria diﬀerent scales will be
used.
For the ﬁrst criterion a scale from 1-5 will be used where 1 is bad and 5 is good.
The second criterion will be evaluated on a scale from 1-5 with 1 being a lot of
time and 5 being quick.
The third criterion will be evaluated in therms of the abstraction level they are
providing, like code, models, combination of the two and so on.
11.2
Classic debugging tool
The classic debugging tool, is a sequential tool that most developers are fa-
miliar with. This is due to how widespread. The classic debugging tool is in
almost any integrated development environment (IDE). The classic debugging
tool works on the code level and allows users to insert breakpoint to inform
the debugging tool, where they want to investigate from. In most of the IDEs
the information provided by the classic debugging tool is: Stack trace, value
of variables, execution view and a list of current breakpoints. From these in-
formation the user can gain information about what the actual program does
during execution.
To explore the ﬂow of a system execution the classical debugger allows features
for stepping forward in the execution process, by skipping over subroutines or
going into them for further exploration.
To evaluate the classical debugging tool according to the criteria mentioned
earlier.
The ﬁrst criterion is about how well the debugging tool would be at informing
the user why a speciﬁc interaction computation was valid or invalid. For this
criterion the classical debugging tool scores a 3. The reason for the score is
that given the right placement of the breakpoint, the classical debugging tool
will allow the user to follow every single execution performed by the ECNO
tool. The reason why it only received 3 is that a lot of additional information
is also acquired. The extra informations are informations are more focused
on how the ECNO tool realizes the ECNO, rather than how interactions are
being computed. This extra information has to be ﬁltered from the relevant
information and hence it might confuse the user.
For the second criterion about time required for debugging interactions, the
classical debugger scores a 2. The reason is that every execution has to be
==PAGE:103==
11.3#1
83
followed step by step. With the ECNO tool containing 1000s of line of code. It
can take a long time to explore even when a decent break point is placed. Placing
a decent breakpoint however requires that the user has a good overview and
understanding of how the ECNO tool works. If a decent breakpoint is placed
several subroutines in the ECNO tools computations of interactions would still
have to be investigated and then mapped to the model. Hence it would take
quite some time to locate the information wanted.
The classical debugging tool only works on the code level. This removes the
great abstraction level provided by the ECNO tool. Since the system is deﬁned
using models, which then are used to generate code from the source models, so
it can be executed. The classical debugger would therefore work on the code
abstraction level and the model abstraction level would be lost.
11.3
Graphic debugging tool
Other debugging tools using graphical models to help display the data exist.
Two of them are "Data Display Debugger" (DDD) [24] and "Java Interactive
Debugging Environment" (JIVE) [7]. They each focus on speciﬁc parts of the
system being executed.
11.3.1
JIVE
JIVE is an extension of the classical debugging tool from the Eclipse IDE, so it
can be used in any Java application in Eclipse. JIVE allows the user to see the
stack trace and execution view, just like the classical debugger. However the
main attribute of JIVE is that it allows the user to explore the currently executed
commands through a sequence diagram, as well as the objects with an object
diagram. If JIVE were to be used for ECNO and evaluated according to the
criteria this is what it would score:
For the ﬁrst criterion JIVE would score a 4. The reason why it scores slightly
better than the classical debugging tool is the sequence diagram provided by
JIVE further helps the user to understand the ﬂow of the calculation. The object
diagram gives information just like the regular variable view in the classical
debugging tool. The reason for why it did not score 5 is that the information
still has to be ﬁltered between what is relevant for the ECNO tool and what is
relevant for the interaction computation due to the models.
For the second criterion JIVE would score 3. The reason for that is that JIVE
allows dynamic slicing making it faster to skip speciﬁc parts of the executions

12.1
Debugging counting events
One of the features of the ECNO is that in speciﬁc scenarios an element can
have choices which deﬁnes counting events. Counting events are when an
element should have choices that can participate multiple times in the same
event instead of only once as seen so far. A more precise deﬁnition of counting
events can be found in the technical report [13, section 2.2].
12.2
Debugging inheritance objects
Another feature of the ECNO which was not added to the develop debugging
tool is checking for inheritance bugs. Inheritance is as in traditional object-
==PAGE:108==
88
Recommendations
oriented development, where objects inherit functions and variables from other
classes. In the ECNO it could be events or elements that inherited from a super
event or element. A more detailed description can be found in [13, chapter 4]
12.3
Displaying current state of an element
As mentioned earlier it is currently not possible to see the speciﬁc state of an
element. The user would have to remember what interactions were performed
as well as what the diﬀerent local behavior diagrams deﬁne. It would therefore
be a good idea, if the debugging tool could be extended in such a way that
examining the current state would be possible. That could be done by making
it possible to select a given element and then see its local behavior diagram,
where the current state would be colored or marked in some way.
This would also help the user to remember the current state and also why some
choices are not valid.
12.4
Improving the visualization
As it could be seen from the results of the debugging view in section 10.1, some
of the view can get really large. In some of the cases that is due to the hidden
connection node, which manipulate the image to let an edge be connected to
an edge. By removing the requirement of the hidden node, it could help the
layout manager to create better layouts.
The features of the visualization could also be improved by allowing features
like zoom. By allowing the user to zoom in and out will make it easier for the
user to get an overview of the computation graph as well as exploring speciﬁc
parts of it.

Chapter 13
Conclusion
This study consisted of the exploration of what information was important
when debugging ECNO models. It resulted in a debugging tool which illus-
trated the information that was required to investigate interaction computations
for the ECNO Tool.
The developed debugging tool was tested given several unintended modiﬁca-
tions to a working ECNO model, and the interaction computation were dis-
played. Through these interaction computation is was clear to see, that some
of the proposed valid interaction would violate the description of the domain.
Hence interactions that should not be valid were and other interactions was not
valid when they were supposed to be. This was due to errors in the modiﬁed
models.
Finally the ECNO debugging tool was compared to other debugging tool, and
it was overall found better for the three criteria which was used for the evalu-
ation. Performing the evaluation made it possible to answer the the research
question of the eﬀectiveness of the developed ECNO debugging tool in terms
of investigating interactions in the ECNO.
==PAGE:112==
92
Conclusion
Using the same criteria for the classical debugging tool. It was clear that the
hypothesis proposed in this thesis was proven. Since the result was that the
developed debugging tool performed better in all three categories compared
to the classical debugging tool. By better deﬁning how the computation of the
speciﬁc interaction was done as well as reducing the time required to debug
and map the information observed to the source models.
Given the answers in section 3.7, section 6.4 , and to the sub research questions
in section 2.1. The main research question:
Will the use of a completely model based debugging tool help maintain the high
abstraction level provided by the Event Coordination Notation. While also reducing
the time and eﬀort required to debug systems generated from the Event Coordinaiton
Notation source models?
Based on the results seen in chapter 10 and chapter 11 it is clear to see that the
proposed debugging tool designed through analysis of the answers from the
sub research question, would indeed fulﬁll the requirements of a high abstrac-
tion level, as well as reduce the time required to debug systems generated from
the ECNO.
This proves the hypothesis stated in section 2.2:
Using a model based debugger for ECNO reduces the time required to debug
interaction computations from the Event Coordination Notation, compared
to classical debuggers.
Therefore the developed debugging tool should be used while debugging
source models in the ECNO tool, since it will help reduce the time required
to debug the scenarios.

type. If an interaction is executed then all actions for the participating
elements choices will be performed. The actions might result in changes
of the object diagram.
Choice: A choice deﬁnes events, by assigning value to its parameters. Possible
choices are given from the current state of an element in its local behavior
diagram. In the local behavior diagram the conditions and actions of the
choices are deﬁned.
Computation graph: A computation graph is the computation for elements
participation in an interaction. To determine if the elements can partici-
pate, they have to be in a state where they have an active choice deﬁning
the event required to participate, given their link from another participatin
element.
Condition: Conditions are restriction for element types to participate in in-
teractions with a speciﬁc choice.
The conditions of a choice given an
element has to be evaluated to true for the element to fully participate in
the interaction.

Terms
Coordination diagram: The coordination diagram describes the global behav-
ior of the system by linking elements to each other with an associated
event type and coordination quantiﬁer.
Coordination quantiﬁer: Coordination quantiﬁer deﬁnes whether one or all
referenced elements of the given element type has to participate for the
current element, to be successful part of an interaction computation. The
given element type is deﬁned by the links endpoint.
Counting event type: A counting event type, is a special event type, where it
is possible for elements to have multiple choices deﬁning the same event.
ECNO Net: ECNO net is the diagram deﬁning the local behavior of an element
type.
Element: Instance of an Element Type. Just like an object is an instance of a
Class. The element is also aware of its current state in the local behavior di-
agram which deﬁnes its associated element type. The diﬀerence between
elements and objects are that elements are deﬁned by a local behavior
diagram involving choices which described the elements interaction with
other elements.
Element Type: The type deﬁning an element. The element type is for an ele-
ment what a class is for an object. The main diﬀerence between an Element
Type and a Class is that the element type have choice deﬁning events. The
diﬀerent choices for an element type is deﬁned by its associated ECNO
net deﬁning its local behavior.
Event: An event is an instance of an event type. The event can have speciﬁc
values for the parameters deﬁned by the event type. The values are set
by the choices deﬁning the event.
Event Type: Event Types deﬁnes the name of its instances, as well as the pa-
rameters and the properties of the parameters.
Global behavior: The global behavior of the system is deﬁned through the use
of a coordination diagram, to coordinate the elements interaction with
each other.
Interaction: An interaction is a collection of elements participating with choices
deﬁning some conditions, events, and actions. The required elements are
those deﬁned by the computation graph from the interaction computa-
tion.
Interaction computation: The interaction computation, computes the diﬀerent
computation graph. Then if a successful computation graph is found
the assignment of parameters and test of the conditions deﬁned by the
participating choices are performed.
==PAGE:115==
A.2#1
95
Link: Links are the references between element types in the global behavior
diagram, deﬁned by a coordination quantiﬁer and an associated event
type.
Local behavior: The local behavior deﬁnes the life cycle of the Element type.
Life cycle: The life cycle deﬁnes the diﬀerent states that an Element type has,
as well as what choice can bring it to other states.
Situation: A situation is deﬁned by an object diagram, and all of the elements
state in their local behavior diagram.
State: A state is the current location of an element in its life cycle.
A.2
Technologies
Classic debugging tool: The classical debugging tool refers to the debugging
view most IDEs implement, where it is possible to explore the stack trace,
variables and execution view.
DDD: Data display debugger is front-end tool for Unix systems. DDD uses the
language speciﬁc debugging tool to generate a model based debugging
tool, which allows the user to examine the objects through a graphical
front-end.
ECNO: The Event Coordination Notation is a model driven development no-
tation where it is possible to model an entire system. The ECNO uses
coordination diagrams to coordinate the diﬀerent local actions.
ECNO debugging tool: The developed debugging tool for this thesis that fo-
cuses on debugging interactions in the ECNO tool.
ECNO Tool: The ECNO tool is a Java based framework that implements the
features of the ECNO. It relies on JET to generate the code from the models
described using EMF.
EMF: Eclipse Modeling Framework. Tool containing features to among others
model some of the features for the ECNO.
GraphViz: Collection of graph visualization tools. Often used in this thesis to
refer to the speciﬁc graph visualization tool called DOT.
JET: Jave Emitter Template are template which allows auto generation of ﬁles
with or without some dynamic information.
