Supersingular Isogeny Diffie Hellman Key Exchange
Abstract
This Master Thesis has three main goals. The first goal is to explain how the cryptographic
protocol Supersingular Isogeny Diffie Hellman key exchange (SIDH) works. To do this, some
theory about elliptic curves is introduced. The second goal is to get an understanding of
the security-properties of SIDH. This includes (but is not limited to) state-of-the-art attacks
against SIDH. The third and final goal is to make a security comparison of the original Diffie
Hellman key exchange protocol and SIDH. Aside from these main goals, a small effort is
spent looking into the elliptic curve Diffie Hellman key exchange.
1
Introduction
Cryptology, as defined on Britannica [12], is the "science concerned with data communi-
cation and storage in secure and usually secret form. It encompasses both cryptography
and cryptanalysis". It is a science that has been around for a long time. In fact, it can be
tracked at least 4000 years back to the Egyptians who used hieroglyphic codes for inscription
on tombs. Since then, there have been many developments in the field, and it is crucial to
keep this trend going given the continous progress in computer technology.
The breakthrough in cryptology that laid the first foundation for this thesis was made in 1976
by two American scientists Whitfeld Diffie and Martin Hellman. They introduced public-key
cryptography and more specfically the Diffie Hellman key exchange protocol. The protocol
enables two parties to establish a shared secret key after exchanging messages over an open
channel, and it relies on a Discrete Logarithm Problem, which for the time being is believed
to be hard for ’good’ choices of the paramters involved [1].
A subsequent breakthrough is the elliptic curve Diffie Hellman key exchange protocol, and as
the name suggests, this protocol builds upon the the original Diffie Hellman protocol using
elliptic curves. Elliptic curve Diffie Hellman key exchange is in wide-spread use today, and
like its predecessor it relies on a Discrete Logarithm Problem [3].
However, with the possible arrival of quantum computers, the Discrete Logarithm Problem
will no longer be hard, and the two mentioned versions of Diffie Hellman key exchange will
then along with other crypto-schemes be insecure. It is therefore of utmost importance to
invent new crypto-schemes that are resilient against quantum computers, and this thesis is
about one such scheme: Supersingular Isogeny Diffie Hellman key exchange (SIDH) [7].
SIDH is analogous to the original Diffie Hellman key exchange protocol from 1976. In short,
it enables two parties to establish a shared secret key after exchanging messages over an
open channel. The new thing in SIDH compared to earlier versions of Diffie Hellman key
exchange is the isogenies. Isogeny based cryptology seems more resilient against quantum
computers, and this is a key motivation for SIDH.

Isogeny based cryptography is a new field from the 2000s. It has roots in Elliptic Curve
Cryptography (ECC), which can be tracked back to the 1980s. However, it was not before
the 1990s that ECC gained popularity, and it happened after Schoof’s algorithm made
it possible to find elliptic curves of large prime order easily.
ECC today is widely used
in public-key cryptography with the mentioned elliptic curve Diffie Hellman key exchange
protocol being an example. One of the inventions in ECC in the 2000s is the mentioned
isogeny based cryptography, but the field has been mostly off the radar until the second half
of the 2010s. A key reason for its current popularity is that isogeny based cryptography as
mentioned seems resilient against quantum computers [4].

Diffie Hellman key exchange
This section is based on [1], and here we will briefly go through the original Diffie Hellman
key exchange protocol from 1976.
2.1
How it works
The protocol takes place in the group (Z∗
p,·) for a prime p. There are two public parameters.
• The prime p
• A primitive element of Z∗
p denoted g
Alice now chooses a random a ∈Zp−1 that is kept secret, and she calculates ga (with the
group operation that is multiplication mod p). This value ga is sent to Bob, that similarly
has chosen a random b ∈Zp−1 that is kept secret and calculated gb. Bob sends gb to Alice,
and they are now both able to arrive at a key K that is shared only between them.
Alice calculates the key K as
K = (gb)a = gab
while Bob calculates it as
K = (ga)b = gab .
2.2
Security of the Diffie Hellman key exchange
One can distinguish between passive and active adversaries. A passive adversary simply
observes information sent between Alice and Bob without interfering. An active adversary
makes changes in the information sent between Alice and Bob.
2.2.1
Passive adversary
A passive adversary, Eve, knows the public parameters p and g and the values ga and gb.
Her task now is to calculate the key K = gab = (gb)a = (ga)b, and this problem is called the
2
==PAGE:6==
Supersingular Isogeny Diffie Hellman Key Exchange
Diffie Hellman problem.
Recall that the Discrete Logarithm Problem in Zp is as follows. Given the odd prime p and
α,β ∈Z∗
p, find a ∈Zp−1 such that β ≡αa mod p. Setting β = ga (or gb) and α = g, then we
have a DLP problem that solves the Diffie Hellman problem. It has not been proven that
the reverse implication is true, but it is believed to be. Hence, the two problems are believed
to be equivalent. The Diffie Hellman key exchange protocol can therefore be considered as
secure against passive adversaries as the hardness of the DLP in Zp.
2.2.2
Active adversary
An active adversary, Eve, using a man-in-the-middle attack does not have to solve the DLP
in order to break the system. The idea is for Eve to intercept Alice’s value ga and replace
it with ge before Bob receives it, where e ∈Zp−1 is a value chosen by Eve. Similarly, Eve
intercepts Bob’s gb and replaces it with gf before Alice receives it. What has happened now
is that Alice obtains the key (gf)a while Bob obtains (ge)b. Eve knows ga and gb, and she
can therefore compute both keys easily, and then she is able to decrypt any message between
Alice and Bob.
However, Alice and Bob can secure themselves against this attack by adding a digital signa-
ture system. The idea is to do what they did before, but now also send a signature of their
value ga or gb to the other person. That is, Alice now sends ga and sigA(ga) to Bob, and he
sends gb and sigB(gb) in return. They both verify the digital signature on receipt, and then
they calculate the shared key K as before. If Eve would attempt a man-in-the-middle attack
in this setting, she would need to be able to make signatures on behalf of Alice and Bob.
However, this requires Eve to know their private keys used in the digital signature system
- keys that Eve does not know. The man-in-the-middle attack therefore fails assuming the
digital signature system is secure.

Elliptic curves form abelian groups as this theorem tells us.
Theorem 3.1 (Elliptic curves form abelian groups) Let E be an elliptic curve together
with the point addition operation +. The following properties are true:
1.
∀P,Q ∈E : P + Q ∈E
2.
∃e ∈E ∀P ∈E : e + P = P + e = P
3.
∀P ∈E ∃Q ∈E : P + Q = Q + P = e
4.
∀P,Q,R ∈E : (P + Q) + R = P + (Q + R)
5.
∀P,Q ∈E : P + Q = Q + P
Therefore, (E,+) forms an abelian group.

4.1
How it works
Now we are not working with elliptic curves over R. Instead we are working with elliptic
curves modulo a prime p, and the curve equation on Weierstrass form therefore now looks
as follows
y2 = x3 + ax + b mod p ,
and the condition on a and b is now 4a3 + 27b2 ̸≡0 mod p. As usual, we also have the point
at infinity O. The definition for point addition is valid for elliptic curves in general and
therefore also for elliptic curves modulo a prime p. However, the calculations are of course
to be done modulo p in this case.
In the protocol there are five public paramters:
• The prime p
• The parameters a,b ∈Zp

• An element G on the elliptic curve E(p) with a high order, n. Ideally G is a primitive
element in which case n = |E(p)|.
Alice chooses a secret integer α where 1 ≤α ≤n −1, and then calculates αG and sends it
to Bob. Bob has similarly chosen β where 1 ≤β ≤n −1, and calculated βG and sent it to
Alice. They can now arrive at their shared key K = αβG.
Alice calculates it as
K = α(βG) = αβG
while Bob calculates it as
K = β(αG) = αβG .
4.2
Security of the elliptic curve Diffie Hellman key exchange
4.2.1
Passive adversary
A passive adversary, Eve, knows αG and βG aside from the public parameters. Her task
is to find the shared key K = αβG, and this problem is the Diffie Hellman problem on
elliptic curves. As in the original Diffie Hellman protocol, this is closely tied to the Discrete
Logarithm Problem.
Definition 4.1 (Discrete Logarithm Problem on elliptic curves) The Discrete Log-
arithm Problem on an elliptic curve E is as follows:
Given P,Q ∈E, find a scalar α such that αP = Q.
Exactly as we had in the original Diffie Hellman case, solving the Discrete Logarithm Problem
implies solving the Diffie Hellman problem. Eve would set P = G and Q = αG (or Q = βG).
The reverse implication has not been proven for elliptic curves either, but it is again believed
to be true in which case the two problems are equivalent.
The Diffie Hellman problem
on elliptic curves can therefore be considered as secure against passive adversaries as the
hardness of the Discrete Logarithm Problem on elliptic curves.
4.2.2
Active adversary
An active adversary, Eve, can use a man-in-the-middle attack analogous to what we described
regarding the original Diffie Hellman protocol. In short, Eve, intercepts Alice’s αG and
replaces it γG before Bob receives it, where γ is chosen by Eve with the condition 1 ≤γ ≤
n −1. Similarly, Eve, intercepts Bob’s βG and replaces it with δG before Alice receives it.
Alice then arrives at the key αδG while Bob arrives at the key βγG. Both can be easily
computed by Eve, and then she is able to decrypt any message between Alice and Bob.
A solution against this attack is again adding a digital signature system so that Alice sends
sigA(αG) to Bob in addition to αG. Similarly Bob sends sigB(βG) to Alice in addition to
βG. They both verify the signatures at receipt and calculate the shared key K as before.

Eve cannot do the man-in-the-middle attack now because it requires their private keys used
in the digital signature system.
4.3
Elliptic curve Diffie Hellman versus original Diffie Hellman
Elliptic curve Diffie Hellman is in wide-spread use today. For Zp there are subexponential
running time algorithms for solving the corresponding DLP, but no equivalent algorithms
are known for elliptic curve groups. The best known approaches to solve the DLP on elliptic
curve groups have complexity of around the square root of the group order. In conclusion,
elliptic curve groups can be chosen smaller than in the original Diffie Hellman protocol while
obtaining similar levels of security. The recommendation in [2] is to choose p ≥21024 when
working in Zp and |E(p)| ≥2160.
One should of course keep in mind, that the fact that no known subexponential running
time algorithm is known to solve the DLP on elliptic curves does not mean that no such
algorithm exists. However, elliptic curve groups have been around in cryptography since
1985, and that gives some confidence that the DLP on elliptic curves is indeed hard.
With quantum computers things change. With quantum computers Shor’s algorithm can
be used to efficiently solve the DLP in both Zp and in elliptic curve groups. Consequently,
both the original Diffie Hellman and elliptic curve Diffie Hellman protocol are insecure given
quantum computers. This is the motivation for SIDH, which we will turn our attention to
now [7].

6.1
First glance
The two previous versions of Diffie Hellman key exchange we have considered before SIDH
(original Diffie Hellman and elliptic curve Diffie Hellman) both rely on a Discrete Logarithm
Problem. Quantum computers can as mentioned solve this type of problem efficiently with
Shor’s algorithm. This means these two versions of Diffie Hellman are not secure given a
quantum computer. The good news is that SIDH is unrelated to any DLP. However, this
obviously does not on its own imply that SIDH is secure, so let us take a closer look at the
security of SIDH.
There are two main computational problems related to SIDH. Given all the information a
passive adversary would see, then the attacker is asked to either find the underlying secret
isogeny or decide whether or not the information actually is from a well-formed instance
of SIDH. These two problems are called the search problem and the decisional problem
respectively.
One thing to be aware of is the public keys. As we saw, Alice’s public key is of the form
(and Bob’s is similar)
PKA = (ϕA(E),ϕA(PB),ϕA(QB)) .
The first part is the elliptic curve she arrived at, and the two last parts are for Bob to
use when arriving at their final shared secret. Obviously these two last parts provide extra
information to the attacker, and the question is if an attacker actually gets an advantage
from this. In fact, it has not been proven that no advantage is given, and it is a main as-
sumption for SIDH that knowledge of the public basis points PA,QA,PB,QB and their images
ϕA(PB),ϕA(QB),ϕB(PA),ϕB(QA) from the public keys does not help a passive adversary in
solving the problem of computing the degree ℓe isogeny
ϕ : E →E′
that connects E and E′ in the supersingular isogeny graph via their j-invariants.
The
assumption has remained valid to date with no known passive attacks that make use of the
mentioned basis points and image points for the types of SIDH parameters used in practice.
These basis points and image points do bring a drawback when considering active adversaries
and static keys however. In short, if Alice or Bob is reusing a secret key for many instances
of the protocol, an attacker can actively learn their entire secret by performing as many
interactions as the bitlength of their key. This result is due to Galbraith, Petit, Shani and
Ti [11]. We will consider this attack later.
There are tests that can be used to gain confidence in that a public key is well-formed and
non-malicious [11]. However, there is currently no such test that with full certainty can
declare a public key well-formed and non-malicious. As a consequence, to avoid attacks
each party in SIDH should only use a secret key once or alternatively use something called
32
==PAGE:36==
Supersingular Isogeny Diffie Hellman Key Exchange
a generic passive-to-active transformation to allow one of the parties to reuse a long-term
secret. This is where SIKE comes into the picture.
SIKE stands for Supersingular Isogeny Key Encapsulation, and here such a generic transfor-
mation is applied to SIDH with the purpose of allowing Alice to safely use a long-term static
secret. Essentially, Bob uses Alice’s fixed public key and his own secret key to compute the
shared secret jAB before he sends any information to Alice. Further, Bob’s secret key, kB,
is chosen as the output of a hash function with the input being Alice’s public key and a
randomly chosen value m. Mathematically put:
kB = H(PKA,m) .
Bob sends PKB as before but also something more. He takes another hash function, calcu-
lates the hash code of the shared secret jAB and XOR’s it with the random value m. What
he sends to Alice is therefore
(PKB,H′(jAB) ⊕m) .
Alice can now use PKB and her secret key as she would in SIDH to compute the shared
secret jAB. Then she computes H′(jAB), and from H′(jAB) ⊕m she learns m. She will now
recompute Bob’s secret key kB = H(PKA,m) and afterwards recompute PKB. Now she
can check if her result for PKB is the same as the PKB she received from Bob. If it is,
Alice accepts what she got from Bob and carries on. If it is not, then she can assume Bob
or someone on behalf of Bob is acting maliciously in which case she can output nonsense.
Bob or anyone on behalf of Bob will therefore not achieve anything from Alice by acting
maliciously.
6.2
Expander graphs
An important part of understanding the security of SIDH is to understand the supersingular
isogeny graphs. Supersingular ℓ-isogeny graphs are connected meaning there is a path be-
tween any two vertices. Further, they are (ℓ+ 1) regular meaning every vertex has exactly
ℓ+ 1 neighbors. One technicality here is that this is counting with multiplicity, which solves
a few anomalies. Importantly, supersingular ℓ-isogeny graphs are expander graphs, and this
is a type of graph that has rapid mixing [5]. This essentially means that when starting at
any vertex in the graph, only a small number of steps is needed in order for a random walk
to converge to a uniform distribution over the whole graph. The word ’small’ here means
logarithmic in the number of vertices. We can therefore pick any vertex we like and be
assured that just a small number of steps is needed to reach any other vertex. To put it
more formally, the maximum distance between any two nodes in a graph is called the graph’s
diameter, and for an expander graph with N nodes the diameter is O(log N).
One thing to note, however, is that it is far from the whole supersingular isogeny graph that
is used. The size of the supersingular isogeny graph in SIDH is O(p) while the size of Alice’s
33
==PAGE:37==
Supersingular Isogeny Diffie Hellman Key Exchange
and Bob’s public key space is only O(√p). This means Alice and Bob take random walks in
a notably smaller space than the whole supersingular isogeny graph. It is not known at the
moment exactly how this affects the security.
6.3
State-of-the-art cryptanalysis against SIDH
This discussion of the state-of-the-art cryptanalysis against SIDH is based on [5]. We will
assume that Alice and Bob are using one-time keys or that SIKE is used to protect a static key
as explained earlier. The best known attacks is then concerned with the problem: Given two
ℓe-isogenous curves E and E′ in the supersingular isogeny graph, compute the corresponding
isogeny.
The smallest set of SIKE parameters has p = 22163137 −1, and in this case there are more
than 2429 nodes in the supersingular isogeny graphs. We have the property 2216 ≈3137 which
good parameter choices should have as mentioned. It is then equally hard to solve for Alice
and for Bob, so let us consider the problem of finding Alice’s secret isogeny
ϕA : E →EA .
This isogeny is of degree 2216 for this choice of parameters. The degree is known publicly, so
the problem is much more specific than the general problem of finding an isogeny connecting
any two nodes in the isogeny graph without any knowledge of degree. To get an appreciation
of how much simpler the specific problem is compared to the general problem, the number of
possibilities for Alice’s EA is the number of distinct 2216-order subgroups on E. This number
is 2216 + 2216−1, and this is obviously much less than the total number of nodes 2429. In fact,
as p grows large the isogeny graphs have O(p) nodes and only O(√p) possible destinations.
This is the reason why random walk algorithms are not the best algorithms against SIDH.
Given the two nodes E and EA, it is publicly known that they can be connected in 216 steps,
and this is much less than the average number of steps between any two nodes in the graph.
It is therefore likely, that Alice’s walk from E to EA is the shortest walk between these two
nodes, and this makes the meet-in-the-middle algorithms best suited for solving the problem
of finding ϕA. In such an algorithm, for this choice of parameters, an adversary would take
108 steps from each of E and EA until they meet in a middle vertex. The adversary would
then have a very likely candidate for Alice’s walk.

(there are 2108 + 2108−1 such vertices). Then each 2108 isogeny from E is computed until

storing all vertices of the types we are interested in. Let S be the set of vertices that are
either 2108-isogenous to E or 2108-isogenous to EA. We then want to take a pseudo-random
walk on S and send certain elements to memory. What elements are sent to memory is
decided by a distinguishing-property on S. There are 2108 + 2108−1 vertices that are 2108
isogenous to E and the same number for EA. Therefore there are 2109 + 2108 elements in
S, but let us for simplicity assume there are 2110 elements. Given the storage limitation
of 280, our distinguishing property then has to distinguish 1 in every 230 elements in S.
The distinguishing property can be anything as long as it distinguishes the right amount
of vertices. Hence, one obvious distinguishing property for our example can be to hash the
vertex and check for 30 leading zeros.
The pseudo-random walk is made using a pseudo-random function f : S →S. It takes
in a vertex from S and puts it into a hash function to produce a string. One bit in the
string decides whether we get an isogeny from E or EA, and the rest of the string chooses
an isogeny of the specified type. This isogeny leads to another vertex in S, which is then the
output of f.
We start with a random initial element x0 ∈S and we iterate through f until we arrive at an
element xn that is distinguished. Then x0 and xn are sent to memory. Many of such walks are
made, and these walks are independent, which explains why the algorithm is parallelizable.
Let x ∈S be such that when x gets mapped via f, then the chosen isogeny is from E.
Similarly, let y ∈S be such that when y gets mapped via f, then the chosen isogeny is from
EA. Further, let x and y be a collision in f i.e. x ̸= y and f(x) = f(y) = z. Now we have
a middle node z, that solves our problem but it is distinguished with a probability of only
2−30. However, f is deterministic so as long as we have a walk that finds x and another that
finds y, then the two walks eventually will end in the same distinguished element. When

Security comparison of original DH and SIDH
We will now compare the security properties of the original Diffie Hellman protocol and
SIDH.
7.1
Two major differences
Two major security differences are the following:
1. The original Diffie Hellman protocol is not quantum secure whereas SIDH is believed
to be.
2. SIDH has a vulnerability for malicious public keys that the original Diffie Hellman
protocol does not.
We have already touched upon the first point. To summarize, the original Diffie Hellman
protocol relies on a Discrete Logarithm Problem being hard, but quantum computers can
use Shor’s algorithm to efficiently solve this problem. The original Diffie Hellman protocol is
therefore not quantum secure. On the other hand, we have SIDH that does not rely on any
Discrete Logarithm Problem. SIDH has shown to be resilient against quantum computers
and is believed to be quantum secure although it has not strictly been proven to be [1][4][7].
A drawback of SIDH is the vulnerability for malicious public keys, which brings us to the
second difference. We have also touched upon this. As we know, Alice’s public key (similar
for Bob) is of the form
PKA = (ϕA(E),ϕA(PB),ϕA(QB)) .

As we also know, not all combinations of values correspond to well-formed keys.
This
gives rise to the mentioned decisional problem for SIDH where all the information a passive
adversary would see in an instance of SIDH is given. Then the attacker is asked to decide
whether or not the information actually is from a well-formed instance of SIDH. As also
mentioned, there is currently no known way for Alice and Bob with full certainty to test
whether the other person’s public key is well-formed or not. This has the consequence that
using the same long-term secret (without a generic passive-to-active transformation as in

rotated.
The other dhi secrets contain EKA, which is only an ephemeral key, and it is provably honest
as we shall see later. Hence, there is no problem with dh2,dh3,dh4. We therefore just need to
find a replacement for dh1 without changing its purpose of safely authenticating Alice using
IKA. We therefore have the two following candidates for the new dh1:
dh1 = SIDH(IKA,EKB)
and
dh1 = SIDH(IKA,IKB) .
The first candidate has the same issue as SIDH(IKA,SKB) in addition to the potential
issue of EKB being out of supply. We are therefore forced to set
dh1 = SIDH(IKA,IKB) .
This is fine as long as the identity keys are proven to be honestly generated. Identity keys
are fixed, so this is something that just needs to be done once, and efficiency is therefore not
an issue in case an expensive proof is used. This is done prior to the protocol.
When it comes to Bob’s signature on SKB, a SIDH signature is not efficient. Any efficient
quantum secure signature is fine and in [17] a hash-based signature is suggested. The verifi-
cation key Bob uses needs to be registered and verified prior to the protocol similar to the
identity keys.
One thing in SIDH that does not exist in the original Diffie Hellman protocol is the asym-
metry between the two parties. As we know, one user usually works with 2-isogenies while
the other works with 3-isogenies. For this reason users of the SI-X3DH protocol are required
to have two identity keys - one of each type. It has been chosen that the 3-isogeny identity
key is for initiating contact, and that the 2-isogeny key is for receiving contact. Of course
the semi-static keys and ephemeral keys of a user in a contact should be of the same type
as the identity key.
We are now ready to illustrate the SI-X3DH protocol which is done below.

Both parties have registered their two public identity keys and their signature verification
key to the server, and as mentioned these keys should be verified before the protocol. We
again assume that Alice is the initiator and Bob is the receiver, so Bob has uploaded SKB
a signature of SKB and a number of EKB keys to the server. Alice makes a request to
the server and gets IKR
B, SKB, SigB(SKB) and an EKB key if supply allows it. She also
has access to V KB, which she uses to validate Bob’s signature on SKB. If the validation
fails, then she aborts the protocol, and if it succeeds, then she carries on. She generates a
uniformly random s = {0,1}n where n is the length of the outputs of two hash functions
H1 and H2. She uses H1 to get a value kA = H1(s). This value is her secret key in the
generation of a public key EKA. As we know, she also needs basis points to generate her
public key, but these are public SIDH parameters known by Alice.
Now Alice calculates the three dh1,dh2,dh3 secrets and dh4 is she received an EKB key.
Further, she calculates a pre-shared key PSK and a FO-proof π. She does it as follows
dh1 = SIDH(IKA,IKB)
dh2 = SIDH(EKA,IKB)
dh3 = SIDH(EKA,SKB)
dh4 = SIDH(EKA,EKB)
PSK = KDF(dh1||dh2||dh3||dh4)
π = s ⊕H2(dh1) ⊕H2(dh2) ⊕H2(dh3) ⊕H2(dh4) .

Of course dh4 is only included if it is defined.
Alice now sends IKA, EKA and π to Bob, and in case she received an EKB key, then she also
sends him a fingerprint identifying this key. Bob can now for himself calculate the secrets
dhi and PSK. The expression for π can be rewritten to
s = π ⊕H2(dh1) ⊕H2(dh2) ⊕H2(dh3) ⊕H2(dh4) ,
and using the π he received from Alice, he can calculate the right-hand side to obtain Alice’s
random s. Now he can recompute kA = H1(s), and from this recompute EKA and check if
it matches the EKA Alice sent. If the verification succeeds both Alice and Bob can compute
their shared ’master’ secret as
K = KDF(s||EKA||PSK) .
If the verification fails, then Bob instead calculates K = KDF(r||EKA||PSK) with a uni-
formly random r = {0,1}n. His K will then almost certainly be different from Alice’s which
will cause the exchange to fail without Alice learning about Bob’s secret keys.
In [17] they specify a security model that captures the security of X3DH, and they prove
that SI-X3DH is also secure in that security model. This is under the assumption that the
underlying SIDH problem of finding the shared secret given the information available to a
passive adversary is hard. As long as this underlying SIDH problem is hard, SI-X3DH is a
quantum secure replacement for X3DH that even closely resembles X3DH.
8
Conclusion
We have explained the SIDH protocol after having introduced some theory about elliptic
curves.
We looked into the security properties of SIDH and found several things. Not all public keys
are well-formed and although there are some tests that can be made, there is no test that
with full certainty can declare a public key well-formed. We saw how this can be exploited
in an attack, and the lesson to be learned is that each party in SIDH should only use a
secret key once. However, a generic passive-to-active transformation as used in SIKE allows
a party to safely use a long-term static secret. We also talked about how isogeny graphs
from SIDH has the important property of being expander graphs. This essentially makes
sure that from each vertex in the graph, just a small number of steps are needed to reach any
other vertex in the graph. Further, we found that generic meet-in-the-middle algorithms can
be a good way to break SIDH. However, they come the disadvantage of having large storage
requirements when applied to SIDH, and this ultimately makes them infeasible in practice.
Instead, a state-of-the-art attack against SIDH is the van Oorschot-Wiener parallel collision
finding algorithm that we covered the essentials of.

We compared SIDH to the original Diffie Hellman key exchange protocol with regards to
security and found two main differences. The first difference is in favor of SIDH and it is
that the original Diffie Hellman protocol is not quantum secure whereas SIDH is believed to
be. The second difference is in favor of the original Diffie Hellman protocol. It is that SIDH
has a vulnerability for malicious public keys that the original Diffie Hellman protocol does
not. However, we saw how a generic passive-to-active transformation (as used in SIKE) can
enable a party in SIDH to safely reuse a secret key. We also saw how SIDH can be used in
the SI-X3DH protocol. This protocol is secure in a security model capturing the security
properties of X3DH under the assumption that the underlying SIDH problem is hard. X3DH
is a protocol using the original Diffie Hellman protocol (or alternatively elliptic curve Diffie
Hellman), and it is in wide-spread use today in multiple messaging apps. SIDH being a
quantum secure replacement via SI-X3DH is evidence of the great potential of SIDH.
The three main goals of the thesis are now achieved. Further, we have obtained insights about
elliptic curve Diffie Hellman offering similar levels of security as original Diffie Hellman but
with smaller parameters. However, elliptic curve Diffie Hellman still has the same problem
as the original Diffie Hellman protocol in relying on a Discrete Logarithm Problem being
hard. If quantum computers arrive, neither of the original Diffie Hellman and elliptic curve
Diffie Hellman protocol is secure.
