Abstract
This thesis presents an approach to quantitative security modeling, based on graphs and
probabilistic attacker behaviours, to deﬁne highly customizable attack scenarios. The
graph describes an attack goal by deﬁning required sub-attacks and includes defensive
measures. The behaviour provides a dynamic aspect by modeling speciﬁc attackers as
a rate-based labeled transition systems where the transitions represent attacker actions.
More precisely, we deﬁne the domain speciﬁc language BADgraph, which separates the
attack description from attack executions. To develop a tool based on BADgraph we
specify a formal deﬁnition and semantics. The tool interacts with the statistical analyzer
MultiVeStA to provide quantitative analyses of BADgraph models. The properties of
interest range from average cost of attacks to the eﬀectiveness of certain defenses. To
show the capabilities of the tool we provide a few case studies.

Chapter1
Introduction
The topic of computer security gets increasingly more attention.
As the complexity
of IT–systems increases it gets diﬃcult to comprehend the diﬀerent ways they can be
breached. Graph–based security modeling provides a useful way of representing and an-
alyzing security scenarios by combining intuitive, visual features with formal semantics.
Several such approaches exist [11], providing a method for qualitative and quantitative
analyses using a directed acyclic graph or an attack tree [3]. The First step in security
modeling, using attack trees, is creating a scalable model for quantifying security with
key attributes such as cost. This allows for probabilistic risk analysis which helps deter-
mine where defensive resources are best spent to better protect a system. In this thesis
we present our approach to graphical security modeling which, in conjunction with a
graph for an attack description, also includes a procedural part for specifying attacker
behaviours as transition systems. The transition systems describe how diﬀerent attackers
would approach the attack.
We have constructed a language speciﬁcation for Behavioural Attacker Defender graph
(BADgraph), inspired by features from [6, 8] such as defense and countermeasure nodes,
and deﬁned formal semantics for it. The speciﬁcation is used to deﬁne a domain speciﬁc
language (DSL). In all further writings, BADgraph refers to both the speciﬁcation and
the DSL. A tool is developed using Xtext [18] to provide a GUI similarly to the Eclipse
IDE [16]. Figure 1.1 showcases the GUI of the tool, consisting of an overview of the
diﬀerent projects (1), a fully featured editor for the DSL (2), an outline of the model
(3), and a view to display analysis results including plots (4).
The tool is based on QFLan [14]. Furthermore, it has been integrated with MultiVeStA
[10], a distributed statistical model checker, performing Monte Carlo simulations on the
speciﬁed model. The simulations provide average values of model properties such as the
eﬀectiveness of defenses, likelihood of attacks, and costs of attacks.
We have created a case study, in collaboration with Saxo Bank, for a denial of ser-
vice attack scenario on a generalized network infrastructure for a web service. The tool
is used to analyze the probability of successful attacks and the eﬀect of defensive mech-
anisms. Furthermore, we include a toy case study. Throughout the thesis a running
example is used to show how model components are speciﬁed in the tool. The example
describes a scenario where the attack goal is opening a safe. The graphical representa-
tion of the example is depicted in Figure 1.2. To reach the attack goal, a path, satisfying

Chapter2
Analysis
This chapter contains an analysis of existing graph based formalisms to motivate the
speciﬁcation of the BADgraph components. Furthermore, QFLan is examined for inspi-
ration and to determine the task at hand.
2.1
QFLan
QFLan [14] is a tool for quantitative modeling and analysis of software product lines. It
is possible to model a high variability of features in a product family. A QFLan speciﬁca-
tion consists of a clear separation between system conﬁguration and system behaviour.
The conﬁguration of a system is described using a feature diagram. A feature diagram
consists of optional and mandatory features. The diagram is used to deﬁne how all
products of a family can be conﬁgured. The behaviour of a system is speciﬁed using
a rate–based labeled transition system. Furthermore, the behaviour is used to describe
how the product, speciﬁed by the feature diagram, behaves and changes. The system
behaviour needs to conform to a series of speciﬁed constraints. Using the procedural
part of a QFLan speciﬁcation it is possible to dynamically install and remove features.
Analyses of QFLan models are performed using statistical model checking with Multi-
VeStA. This has the advantage of avoiding state space explosion in the case of large
models. It is possible to determine the likelihood of speciﬁc events and the expected
values of variables such as the cost, weight, and load.
In [14] ter Beek et al.
introduce an example of how QFLan can be used to model
attacks on a physical safe by using the feature diagram as an attack tree. The system
behaviour is used to model the behaviour of the attacker. The example is based on the
attack tree presented by Schneier [2]. This initial example is what sparked the idea of
developing a tool, with a language better suited for security modeling, using the same
approach for quantitative analysis as in QFLan. The available analyses in QFLan are
interesting because they can be used to model the cost of attacks, probability of attacks,
how likely diﬀerent parts of a system are to be breached, etc.

2.1.2
MultiVeStA & QFLan Analysis
In QFLan [14] it is possible to evaluate a model in two ways. Either by performing a
simulation, which is mostly for debugging purposes, or by performing an analysis. The
simulation is performed by simulating transitions starting from the initial state. From
the current state the allowed transitions are calculated. An allowed transition does not
violate any constraints. Next, we complete the simulation step by picking an allowed
transition and updating the current state. This continuous for the speciﬁed number of
steps.

When performing an analysis we are instead invoking MultiVeStA and using it to per-
form a high number of simulations to generate statistics. An analysis is speciﬁed using
a query, a default delta, and an alpha. The query is translated into MultiQuaTEx, the
language used to specify a MultiVeStA analysis. A query can be speciﬁed in two ways.
The ﬁrst being to check properties when a certain conditions evaluates to true such
as having successfully performed the attack goal. The other type of analysis evaluates
certain properties for each step of the simulation. The speciﬁcation of both types of
analyses and a simulation will be further elaborated in Section 6.
2.2
Attack Diagram
The tool needs an attack diagram to describe the possible ways of accomplishing an
attack goal. An attack tree describes the diﬀerent steps of an attack against a system.
The notion of attack trees was ﬁrst introduced by Schneier [2] and has since been ex-
tended to ﬁt diﬀerent security modeling purposes such as risk analysis.
An attack tree provides a clear graphical representation of an attack using nodes and
relations in a tree structure. It provides a description of an attack easily comprehensible
for both technical and non technical individuals. The root node describes the attack
goal. Each child node represents a reﬁnement of the corresponding parent node, i.e. the
sub-attacks necessary to perform the attack described by the parent. The leaf nodes
of the tree corresponds to ”basic actions” i.e. attacks that cannot be decomposed into
smaller attacks. If the attack tree is decorated with the correct attributes, it can be
used to assess the cost, the diﬃculty, and the probability of an attack. In the literature
survey ”DAG–based attack and defense modeling: Don’t miss the forest for the attack
trees“ [11] Barbara Kordy et al. provide an overview of the many extensions to the
original deﬁnition of attack trees. In this section we will describe our design choices for
an attack diagram, for the tool, with regards to nodes and the relations between the
nodes. Furthermore, we will describe how existing tools for security modeling, which use
attack trees, have inﬂuenced the design.
2.2.1
Nodes
The attack diagram of the BADgraph speciﬁcation requires attack nodes in order to
specify how to achieve the attack goal. Operations on attack nodes are performed in
the attacker behaviour. In ADTool [9], attack nodes are added automatically when the
relation to their children is satisﬁed. However, we deviate from this approach by adding
all attack nodes using the behaviour. Instead, the attack graph is used to specify hier-
archical constraints.
The survey [11] generally groups the diﬀerent formalisms by two aspects: Whether
it is used for attack and/or defense modelling and whether it has a static or sequential
approach. An argument for adding defensive modelling to the attack trees is the fact that
it enables some interesting analyses. Analyses, such as the value of adding a particular
defense to a system, how to best utilize defensive resources, and the return of attack
compared to the return of defense, i.e. if it is worth attacking the system. One type
of attack tree to incorporate defenses are Attack–Defense Trees [6]. An attack–defense
tree models defenses against attacks, such as adding 2nd factor authentication to a

Attack–defense trees have also been used when developing ADTool [9]. In ADTool a
bottom–up evaluation algorithm is used to evaluate measures such as attributes based
on real values (e.g. cost), skill level of the attacker, reachability within a given time
frame, and boolean properties (e.g. satisﬁability of a given scenario). All measures can
be evaluated both from the point of view of the attacker and the defender. Defense
nodes are included in BADgraph to model static system defenses.
However, some defenses require an interaction to be activated such as burglary alarms.
This implies the need for a dynamic defense, or countermeasure, as a response to an
ongoing attack. This feature is captured by the Attack Countermeasure Trees formalism
[8] which includes detection and mitigation events. In BADgraph countermeasures will
be included as a special node type, which is activated when the attacker is detected
attempting certain attacks. This type of node enriches the model with the possibility of
adding a limited defender behaviour. The only diﬀerence between defense and counter-
measure nodes is that the countermeasure nodes are activated by attack events. Thus,
they do not have an eﬀect before such an event is detected.
2.2.2
Relations
The attack diagram uses relations to describe attack and defense reﬁnements.
The
relations use logical operators to describe attack requirements, or requirements for a de-
fense or countermeasure to be active. In other words, the relations specify constraints.
It should be noted that even if the logical relation for a countermeasure node evaluates
to true, it still needs a trigger event to become active.
The diﬀerent formalisms, presented in the survey [11], include AND and OR relations.
As part of Attack Countermeasure Trees [8] they propose an additional relation, namely
the k-out-of-n relation. This expresses that only a certain number of the sub-attacks
are necessary. The relation is syntactic sugar and can be modeled exclusively using AND
and OR relations. However, this can prove cumbersome for the designer for large values
of n. The k-out-of-n relation is useful for modelling scenarios related to voting or
sensors. This relation is included in the BADgraph speciﬁcation. In [11], formalisms
are classiﬁed by a sequential or static modeling approach. For now, we have only intro-
duced static relations, i.e. relations that do not enforce a speciﬁc ordering of events.
In Enhanced Attack Trees [5] they use an ordered–and relation and in Improved Attack
Trees [7] they use a similar sequence–and relation. This type of relation allows for the
speciﬁcation of an ordering on sub-attacks thus, adding a sequential aspect. This can be
interpreted in two ways; either that the sub-attacks only can be done in a certain order,
or that the sub-attacks can be done in any order but only the speciﬁed order satisﬁes
the relation. The latter is used for the ordered-and relation, OAND, in the BADgraph
speciﬁcation. It should be noted that speciﬁc ordering of attacks can also be modeled in
the attacker behaviour. However, the tool can include several attacker behaviours and,
in some cases, it can be interesting to model an attacker who does not know the order
of attacks.
In Attack–Defense Trees [6], a special role–changing–relation is used when changing
from an attack reﬁnement to a defense reﬁnement and vise versa. This relation de-

2.3
Properties
Attack trees can be decorated with properties such as boolean or real values. These
properties are placed on nodes and can be used to deﬁne constraints or for analysis
purposes. In BADgraph attributes are used to evaluate real values such as cost. Each
node can be decorated with its corresponding attribute value. In [2], values are evaluated
bottom–up as introduced on attack trees. However, e.g. the total cost of an attack is

not always equivalent to the sum of successful sub-attacks, as it does not consider failed
attempts. Thus, the total attribute value in BADgraph is the sum of both successful and
failed attack attempts. We do not consider decorating the attack graph with boolean
values as it can be modeled in the behaviour or by constraints. E.g. if an attacker does
not have the required skill level the attack is not included in the behaviour. Attributes
are also used to model the defensive aspect.
In SecurITree [4], they introduce a property called ”noticeability”. The noticeability
of an attack node speciﬁes how likely the defender is to notice that the attack was
attempted. This property is particularly interesting as it can be used to determine the
activation of a countermeasure. Thus, the property is included in BADgraph to assign
the probability of being detected to an attack node. SecurITree also uses the property
to deﬁne constraints and thus, vary the stealthiness of the attacker. This can be done
using the attacker behaviour of BADgraph.
In attack–defense trees [6] and ADTool [9], a node is not accessible if it has a child
node of opposite role. It is a common conception in system security that nothing is ever
100% secure. Thus, the last property of BADgraph is Defense Eﬀectiveness, which is
used to measure a defensive node’s eﬀectiveness against attacks using probability. How-
ever, each node, in a defensive reﬁnement, can have a certain eﬀectiveness against the
attack. Thus, all these values must be applied when calculating the defense eﬀective-
ness against the attack. Furthermore, defenses can have diﬀerent eﬀectiveness against
diﬀerent attackers and defensive nodes can be applied to multiple attack nodes. Thus,
the defense eﬀectiveness of a defensive node depends on both the attack node and the
attacker.
The combination of countermeasure nodes, detection rates, and defense eﬀectiveness
provides a ”limited” defender behaviour. Defense eﬀectiveness models the probability of
successfully defending. The combination of detection rates and countermeasure nodes
gives the defender an ability to react to attacks The defender behaviour is purely reac-
tive.
2.4
Attacker Behaviour
To specify the behaviour of an attacker a rate-based labeled transition system is used.
Modeling the attacker behaviours requires a more intimate knowledge of the potential
attackers. It provides the ability of making speciﬁc analyses on speciﬁc attacker types.
Furthermore, it allows for more complex types of analyses with multiple successive at-
tempts of an attack instead of a simple bottom–up evaluation as in ADTool [9]. It
enables modeling of speciﬁc attack paths instead of a simple best or worst case evalua-
tion of the attack graph. Thus, the behaviour is an addition worth having as it allows
for a more detailed modeling of the attacker.
It is important that it is possible to specify all types of attackers such as script kid-
dies, insiders, hackers, and government agencies. The main advantage of being able
to declare multiple attackers is that it might not be desirable for a company to secure

its systems against the very strongest of attackers. In most cases adequate security is
desired to not waste resources. The attacker should be able to perform the attacks
speciﬁed in the graph. This is modeled by the addition of attacker actions as part of
transitions in the behaviour. Two basic actions model succeeding or failing attacks using
the two actions add and fail on attack nodes. It should be noted that we do not allow
the add and fail actions on already successfully performed attacks.
Within protocol security the Dolev-Yao attacker model [1] is often used, which states
that the attacker controls the communication medium i.e. he can read, send, and inter-
cept messages. The attacker can, however, not break cryptographic algorithms i.e. the
attacker can only decrypt messages that are meant for him. The fact that the attacker
controls the communication medium implies that he has knowledge about all parts of
a system, or, at the very least, is able to obtain this knowledge. To model the act of
being able to obtain information about a system an action called query is included in a
BADgraph speciﬁcation.
The last included attacker action is remove. This action allows for the removal of an
already performed attack. This action is for instance useful for children in ordered–and
relations as they can be added in any order. In cases where a child has multiple parents
the child can be added to satisfy the relation to the other parent, which can result in a
violation of the order in the ordered–and relation. To satisfy the ordered–and relation,
later on, the child has to be removed and added in the correct order. Furthermore, user
deﬁned actions are included. The user deﬁned actions are used for transitions without
attacker actions, i.e. unrelated to the attack graph.
Variables are included in BADgraph in order to model context information and con-
straints. They are updated by side eﬀects. A side eﬀect is a declaration of a function,
which can be added to a transition. The beneﬁt of having side eﬀects in conjunction
with variables is that it allows for modeling of properties where the eﬀect on the variable
is not tied to a speciﬁc node in the graph or the performed action. This is useful as
variables can be used in MultiVeStA analysis.
The behaviour of an attacker is bound by a set of constraints. These are hierarchi-
cal constraints, quantitative constraints, and action constraints. The hierarchical con-
straints are imposed by the attack graph and speciﬁes when an attack is allowed. The
second type of constraints, namely the quantitative constraints, allows the designer to
impose restrictions on the attacker behaviour by using the current value of attributes
and variables. A quantitative constraint can be used to model that the total cost of an
attack must not exceed a certain limit or that an attack needs to succeed within a given
number of attack attempts. This is particularly interesting due to the inclusion of the
behaviour, which allows for modeling of failed attacks or multiple attempts of an attack.
A failed attack also requires a resource investment from the attacker. To impose further
restrictions on the attacker behaviour, the notion of action constraints is added. Action
constraints imposes constraints based on previously performed actions and quantitative
measures. Furthermore, we include optional action guards when deﬁning transitions. An
action guard works like an action constraint but is applied to a single transition. This is
for instance useful for user deﬁned actions.

In BADgraph we distinguish between two diﬀerent types of actions. We include user-
deﬁned actions and attacker actions.
Deﬁnition 2 (Actions) The set of actions α in a BADgraph speciﬁcation S is deﬁned
as the union of
• αu is the set of user deﬁned actions.
• αa is the set of attacker actions. These are the composition of add(na), remove(na),
fail(na), and query(n), where na ∈Na and n ∈N.
The attacker actions are speciﬁed on the transitions in the attacker behaviour. Here
add(na) represents having successfully performed the attack na, fail(na) describes
the attacker failing the attack na, remove(na) removes an existing attack na, and
query(n) is a request, checking the existence of a node n. The attacker actions relate
to the behaviour of the attacker. Thus, they mainly rely on attack nodes. However,
the query action can be performed on all nodes of any type n ∈N. Thus, query(n)
represents an attacker inspecting the system and gaining the knowledge described by
n. This can cause the attacker to take alternative attack paths, eﬀectively changing
the attacker state, as the behaviour of an attacker depends on the realization of certain
defensive measures.

3.5
Constraints
The BADgraph speciﬁcation contains a ﬁnite set of constraints denoted C. The con-
straints impose restrictions on the order of attacks with hierarchical constraints CH.
Constraints on attribute or variable values are deﬁned by quantitative constraints CQ.
Lastly, constraints on the actions of an attacker is speciﬁed using action constraints CA.
Thus, the constraints of a BADgraph model is C = CH ∪CQ∪CA. All types of constraints
are deﬁned in the model using the language constructs from Table 3.1.
In Deﬁnition 1 all constraints are denoted C.
The set of constraints is used to de-
ﬁne a constraint store C for resolving constraints in regards to model execution, fur-
ther elaborated in Deﬁnition 12. The union of two constraint stores C1, C2 is denoted
C1 ⊗C2. Next, we need to deﬁne the notion of entailment. Entailment of a constraint
C′ in C is denoted C ⊢C′ and states that C′ can be derived from the constraint store C.
The constraints, in a BADgraph speciﬁcation, use two boolean functions has(n) and
allowed(n). The has(n) function returns true if the attack n ∈Na has been added
or if the defense n ∈Ndef is active. The boolean function allowed(n) expresses that
the node n ∈Na is allowed for certain actions and can be added without violating
hierarchical constraints. However, situations can occur where allowed(n) returns true
while add(n) is blocked by a non-hierarchical constraint.

In the DSL, hierarchical constraints are speciﬁed as relations between nodes. Relations
are speciﬁed in blocks of the form n ⊕N, as speciﬁed in Table 2.1. In the relation N is

cannot include an ancestor of n, as this causes a cyclic graph. As -OR-> is the default
relation it is in all further writings equivalent to writing ->, which also is the case in the
developed tool.

Deﬁnition 10 (Attacker behaviour deﬁnition) Given a BADgraph speciﬁcation S,
the behaviors B is given by tuples ⟨a, z⟩where a ∈A is an attacker and z is the
corresponding behaviour. An attacker behaviour z is speciﬁed by ⟨S, T⟩, where:
• S denotes a ﬁnite set of states.
• T denotes the ﬁnite set of transitions between the states. A transition is described
as ⟨s, (act, cg, r, u), s′⟩, where s, ∈S is the current state, s′ ∈S is the resulting
state, and act ∈α is an action, i.e. a user deﬁned action, αu, or an attacker
action, αa. The rate is speciﬁed by r and u describes memory updates of form
x = e where x ∈V ∪AT and e is an expression. Lastly, cg is a action guard
speciﬁed by a boolean expression as in Deﬁnition 9.
A memory update allows the designer to specify changes to the value of variables. Fur-
thermore, it updates attribute values att ∈AT , if the action is an attacker action of
form act ∈{add(na), fail(na)} where na ∈Na, or if a countermeasure node is acti-
vated. It is important to note the the memory updates, described by u, are done in
parallel using the initial variable values. I.e. ﬁrst all variables values are read, then the
expressions, e, are evaluated, and, lastly, the variables, x, are updated. The action guards
are applied locally on transitions where action constraints are applied to all transitions
with the speciﬁed action.

Evaluation & Future Work
This section contains an evaluation of the BADgraph tool with regards to multiple as-
pects such as functionality, maintainability, and usability. As part of the evaluation, we
consider possible improvements and future work. It includes feedback from meetings
with Mads Syska Hasling, Saxo Bank, and Zaruhi Aslanyan, Alexandra Institute, along
with our own observations.
The BADgraph tool can perform security analyses on detailed models. The attacker
behavior and attack graph speciﬁcations allow the modeling of custom attack scenarios.
Section 6.2 contains a case study from industry. The result of an analysis depends on
the roughness of the estimated property values. The results prove useful when determin-
ing how to best use the available resources on defenses. As seen from the results, the
probability for reaching the attack goal converges to 1, over the 1000 steps. However,
depending on the attack and attacker, it is unlikely to see that many attempts. An
attacker would, most likely, run out of resources or give up. In the tool, this can be
modeled with constraints. However, with unlimited resources and attempts the attacker
will always succeed.
The possible attacks on a system are limited to what is deﬁned in the model, i.e to
the imagination of the model designer. However, when performing risk assessment it
is important to consider unexpected attacks i.e. attacks not included in the model.
Furthermore, the assumption that system parts are compromised should be taken into
consideration. These considerations are useful when determining the most vulnerable
system components. One way of modeling an unexpected attack, in the BADgraph tool,
is to specify the case of an unexpected attack explicitly with an attack node. A compro-
mised system component can be modeled as an initial attack, as these represent attacks
existing prior to the analysis. Diﬀerent versions of the analysis can help determine where
a system requires additional or fewer defenses or where it is vulnerable to insider attackers.
Generally, slower execution times are caused by the amount of properties included in
a query. To improve execution times, the simulations can be distributed on multiple
MultiVeStA servers. Furthermore, the algorithms used in the core, can be optimized. In
general, there is room for improvement as this is the ﬁrst iteration of the tool. How-
ever, the tool is tested, in Section 5.1, and works as expected for the diﬀerent test cases.
A downside of the tool is the fact that a model can be costly to maintain. New threats
44
==PAGE:57==
Chapter 7. Evaluation & Future Work
and attacks on systems are constantly emerging. This changes the threat landscape
and requires maintaining and updating a model. The attack graph, properties, and the
attacker behaviour would need to be updated, which can be cumbersome and expensive.
A solution is to add sub attack graph libraries for common attack scenarios that can be
imported into models. The libraries can contain speciﬁcations for common behaviours.
Thus, only the libraries need to be updated when new attacks are discovered, implicitly
updating all models using the library. A model’s maintainability depends largely on the
amount of detail in it, i.e. the number of attackers, properties, etc.
In general, the main advantage of attack trees or DAGs is that they provide a clear
overview of attack scenarios. However, the BADgraph tool does not have functionality
to visualize models which the ADTool [9] does. Implementing such a view would increase
usability by providing an overview, which makes it easier to keep track of models. A
view could include both the attack graph and the behaviours. Furthermore, the Xtext
components proposal provider, used to provide improved auto-completion of code, and
quick ﬁx provider, used to suggest ﬁxes to errors and warnings, can be implemented.
Combined with a more detailed validator, this would increase usability by providing a
more user friendly IDE. Usability can also be increased by implementing a feature for
import and export of models. This could work with models from other tools such as
ADTool SecurITree [4], providing an easier transition to the BADgraph tool.
Currently, the biggest improvements to the tool, is those that increase the usability.
However, there are also other changes worth considering. The restrictions when deﬁning
relations in the DSL should be revisited. Furthermore, the value of attributes are stored
both for the attacker and defender. However, the value is only used for the attacker. By
using the attribute for the defender role, a new type of analysis can be performed which
compares the cost of the attack with the cost of defending against the attack. The cur-
rent version of the query action does not work as intended. Currently, it is semantically
identical to a user deﬁned action. The purpose of the action was to investigate a system
by specifying a question. The question is formalized as a constraint and is evaluated
using the constraint store, thus inferring the state of the system. The transition is valid
if the constraint evaluates to true. However, this is covered by action guards. Thus,
both versions of the query action are redundant.
As of now the Attack Detection Rates are only used to trigger countermeasures. It
would be interesting to expand on this feature. One way would be to allow for the speci-
ﬁcation of constraints where an attacker would avoid attacks beyond a certain detection
rate threshold. This type of constraint is included in SecurITree. In BADgraph this is
currently done by not including the attacks with a high detection rate in the behaviour.
45
==PAGE:58==
Chapter8
Conclusion
We have presented the language BADgraph, including a formal deﬁnition and semantics.
It provides a quantitative modeling environment for highly customizable attack scenarios
supported by an Eclipse-based tool. BADgraph provides a high-level DSL to specify an
attack scenario with probabilistic attacker behaviours and statistical analysis of prop-
erties. The GUI of the tool oﬀers modern editing support, including error highlighting
and auto-completion. Furthermore, it provides a view to clearly present the result of an
analysis.
The tool has been validated using black box tests which verify that the tool is im-
plemented accordingly to the deﬁned semantics. Furthermore, we have provided a case
study, in collaboration with Saxo Bank, on a general network architecture for a web
service which prioritizes availability. The tool is also applied on a toy case study, of a
bank robbery, which showcases the diﬀerent features of BADgraph.
Finally, we have evaluated the project and provided suggestions for future work. The
addition of the provided suggestions increases functionality by extending constraint and
analysis capabilities. The usability of the tool will be increased by providing a higher
degree of editing support, a visualization of the speciﬁed model, a collection of attack
libraries, and features that support import and export of models.
