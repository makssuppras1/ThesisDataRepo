Abstract
This project focuses on implementing the three cryptographic authenticated en-
cryption (AE) algorithms, ACORN, ASCON, and AES-GCM, while applying them to
various IoT protocols and systems. The AE algorithms require 16-bytes each for
an IV, Tag, and a variable size of ciphertext, thereby a minimum payload size of
32-bytes. This is with the purpose of providing end-to-end security of the payload
transmitted by an IoT device to another device or a high-end server. The two
long-range protocols, NB-IoT and Sigfox, were implemented in IoT systems, com-
municating via a device to a custom-built IoT back-end. This custom back-end was
implemented with functionality to both receive UDP packets from the TDC NB-IoT
infrastructure as well as the Sigfox back-end’s HTTP Callbacks. The payload of
each protocol was encrypted via an AE algorithm thereby providing confidentiality,
integrity, and authenticity of the payload. Concerning Sigfox, a fragmentation
process was necessary as Sigfox only allows an uplink payload size of 12-bytes.
The two short-range protocols, WiFi and BLE, were implemented in Device-to-
Device setups. In these setups, the payload transmitted between the two devices
was also encrypted via AE, thereby providing end-to-end security. This showed
complete confidentiality as no sniffer, was able to make sense of the data. It was
proven possible to receive both IV, Tag, and Ciphertext to perform Tag validation
and decryption, verifying it was possible to securely communicate. It should be
noted that similar to Sigfox, BLE requires fragmentation, as a payload size of
maximum 20-bytes is supported. Finally, the application layer protocol MQTT
was also implemented in a setup combining Device-to-Device and Device-to-Server.
Here it was also concluded that it was very advantageous to employ an AE algorithm
for protection of the payload, as MQTT allows large payload sizes, amongst other
things.
The three AE algorithms were tested for performance in relation to the processing
power (CPU) of four microcontrollers, Arduino UNO, SODAQ SFF SARA R410,
ESP8266, and ESP32, all with different MHz specifications. It was concluded, that
the ACORN algorithm achieves the best results of the three, with good performance
on all devices. In the case of the UNO, with the lowest MHz CPU, it was shown that
the choice of the algorithm was even more important, signifying the importance of
using lightweight cryptographic ciphers when dealing with IoT devices.

Resumé
I dette projekt har fokus været implementering og anvendelsen af de tre authenti-
cated encryption (AE) algoritmer, ACORN, ASCON og AES-GCM, indenfor diverse
IoT protokoller og systemer. Ved brug af AE opnås en minimum pakke størrelse på
32-bytes, idet både IV og Tag kræver 16-bytes hver og hvor ciphertext derudover
udgør en variable størrelse. Formålet med følgende er at sikre kommunikationen
mellem to enheder i systemer bestående af to ressourcesvage IoT enheder eller til
en ressourcestærk server.
De to protokoller, NB-IoT og Sigfox, designet til lang distance transmissioner,
er blevet implementeret som en del af et samlet system, hvoraf kommunikatio-
nen er rettet mod en IoT back-end server.
Denne IoT back-end er en løsning
implementeret med henblik på at kunne fremvise transmitterede data. Derudover
besidder serveren funktionalitet til at kunne modtage og håndtere UDP pakker
fra TDCs NB-IoT infrastruktur, samt HTTP POST requests sendt fra Sigfox.
I
forhold til datasikkerhed og ved anvendelse af kommunikationsprotokollerne er
AE algoritmerne benyttet for at opnå fortrolighed, integritet og ægthed. Under
benyttelse af Sigfox, har det været nødvendigt at foretage en fragmentering af
den samlede besked, da denne protokol kun tillader en brugerdefineret besked
størrelse på 12-bytes. Heraf sikres det, at hvert fragment ikke overstiger 12-bytes,
og beskeden kan blive sendt i mindre dele. Protokollerne WiFi og BLE, designet
til kort distance transmissioner, er også blevet implementeret. Følgende er med
henblik på integrationer relateret til udelukkende at imitere kommunikation fra
enhed til enhed. I forhold til datasikkerhed er samme AE-krypterings algoritmer
benyttet, som beskrevet for NB-IoT og Sigfox, hvilket sikrer transmissionen mod en
tredjepart, der potentielt vil have interesse i at fremskaffe sig adgang til data’en.
Derudover blev det også bevist, at det var muligt at modtage både IV, Tag og Cipher-
text, således at det modtagede Tag kunne valideres og Ciphertext’en dekrypteres.
På tilsvarende måde som Sigfox fragmentering er blevet beskrevet, er BLE im-
plementeret, da denne protokol har en maximum payload størrelse på 20-bytes.
Ydermere blev application layer protokollen MQTT også implementeret med hen-
blik på både at skabe enhed-til-enhed kommunikation, men også enhed-til-server
kommunikation. På lige vis med de andre kommunikations implementeringer, blev
en AE algoritme også benyttet for at beskytte data.
De tre AE algoritmers ydeevne blev testet i forhold til processorkraft (CPU) på
de følgende fire enheder, Arduino UNO, SODAQ SFF SARA R410, ESP8266 og
ESP32.ACORN opnåede de bedste resultater, blandt de tre udvalgte, på samtlige
enheder. Ved analyse af resultaterne kunne det ses, at ved brugen af en Arduino
UNO havde valget af krypterings algoritme stor betydning, da resultaterne i
høj grad korrelerede med den tilsvarende processeringskraft.
Ved brugen af
kryptografiske algoritmer i forbindelse med IoT systemer, fremgår det herved, at

valget har stor indflydelse på ydeevnen i et aktuelt system. Derudover kan det
også konkluderes, at lightweight kryptografi er det mest optimale valg i disse
systemopsætninger.

Chapter 1
Introduction
Internet of Things (IoT) is defined as any form of a system of some size that is
interconnected while being able to communicate with the internet.
Although
a specific IoT device is somewhat difficult to precisely define, it is commonly
designated as any type of device which operates in a constrained environment,
in terms of physical size, processing power, available memory as well as being
potentially powered via a compact battery source. Typical IoT applications are built
with microcontroller units (MCU)s with differing capabilities, naturally depending
on the intended use case. Since Kevin Ashton officially defined the term Internet
of Things in 1999, its widespread usage has caused an explosion in development
targeted towards use cases where embedded devices and modern IoT solutions
would be applicable.[1]
The true purpose of IoT is rather straightforward in its original inception, as it
seeks to interconnect devices, thereby enabling them to communicate and exchange
information. From here, multiple sub-purposes are imaginable such as improving
the overall quality of life for individuals, creating smarter applications, improving
existing industrial solutions, and many more. One can view IoT devices as regular
appliances and gadgets, which have now been enhanced with various embedded
devices in order to enable them to communicate and interface with both the
physical world and the cyber one. There is no doubt, IoT is meant to essentially
touch every aspect of our lives in a not-so-far future.
In 2015, it was estimated that approximately 15 billion IoT devices were in op-
eration, while steadily increasing throughout the years, it is now predicted that
upwards of 75 billion devices will be up and running.[2] Another statistic revealed,
that end-users single-handedly spent approximately 110 billion USD in 2017, fore-
casted to increase to 1567 billion in 2025, showcasing just how immense the IoT
market truly is.[3] However, as the demand for IoT devices has increased at an
incredible rate, it has become obvious, that various new potential security flaws
have been exposed and exploited, as a consequence of the IoT revolution.

2.1
OSI model & IoT stack
In this section, a brief description of the Open Systems Interconnection (OSI)
conceptual model is made, along with a comparison of the corresponding IoT
model. The OSI model consists of seven layers describing the entire flow of a
networking system. Each layer operates different from the other layers, either
adding or removing information via a Protocol Data Unit (PDU). The two models
can be seen compared side-by-side in Figure 2.1. Noticeably, only five layers are
present in the IoT model, where the application layer now comprises the session
layer, presentation layer, and application layer. [7]
In the physical layer, data is handled as raw bits, meaning the PDU is a bit. The
objective of the physical layer is to facilitate the bit transmission of data from the
sender to the receiver of the data-link layer, such as transmitting the data through
an actual fiber, cable, electromagnetic signals, or similar. Communication protocols,
dependent on the physical layer, have different requirements and specifications,
e.g. the type of physical medium used for data transfer. [8] In regards to the
related layer on the IoT stack, there are no major changes between these.
In the data-link layer, the data is encapsulated in frames. The size of each frame
varies depending on the protocol. Furthermore, defining and controlling com-
plex functionality, determining whether the communication is connectionless or
connection-oriented is also done in this layer. Noteworthy is the control header
included in each frame, which identifies the receiver and/or sender, meaning it
is used to direct the data to another entity. In the case of a connection-oriented
communication flow, it is possible to also set the data rate dynamically, perform
error correction, and much more. [7, 8]

The third layer is the network layer where the PDU is a packet. The main responsi-
bility of this layer is to perform routing of packets from one network to another. As
an example, in Figure 2.1 at the network layer, a logical address is an IP-address.
Internet Protocol (IP) addresses are used as reference points of devices connected
to a network, or a set of networks. Likewise, it can be used to determine the
routing of a packet going from one network to another, through the internet. This
naturally assumes the usage of various other protocols to facilitate the overall
transfer and translation. In addition, the network layer handles functionalities such
as segmentation, blocking, error detection, sequencing, and flow control and more.
[8]
In the Transport Layer, the PDU is a segment, where the two most common proto-
cols used are the User Datagram Protocol (UDP) and Transmission Control Protocol
(TCP). As these two protocols are completely different in regards to offering connec-
tionless versus connection oriented communication, the layer provides a different
set of functionality dependent on the utilized protocol. In terms of a connection ori-
ented protocol, the transport layer initiates the connection between the end-point
and initial-point and also drops the link between these whenever the transfer is
considered finished. Moreover, in relation to the chosen protocol, if needed and

The application layer of the IoT stack is a compressed version of the three layers
in the OSI model, session layer, presentation layer and application layer. This
layer is the very beginning and defines the communication link between various
applications, where protocols such as HTTP, MQTT or CoAP are used. Moreover,
the application layer defines various service properties of the system, such as
security mechanisms, data transfer methods, Quality of Service, etc. In regards
to the security properties, defining authentication and potentially access control
schemes is also part of this layer.[8] It should be noted that the stack naturally
changes for each specific IoT communication protocol, and the models are merely
used as general mappings.

Protocols
The type of communication technology is especially relevant in IoT systems, and
is typically classified in two groups, either Long-range or Short-range. Short-
range based technologies are regularly making use of the well-known Industrial,
Scientific, and Medical (ISM) band, examples including WiFi (IEEE 802.11) and

transmission ranges compared to the actual date rate of transfer. [7] The protocol
utilized in IoT systems is typically very dependent on the specific use case, however,
it is often selected in terms of needed payload size, duty cycle, financial cost, power
consumption costs and various other metrics.
2.2.1

2.2.1.1
Sigfox
Sigfox is a Low Power Wide Area Network (LPWAN) technology suitable for a low
amount of transferable data in infrequent bursts. It is a long-range communication
technology optimized for devices with low power and resources; a typical IoT
device that needs to transmit a small message once in a while. Compared to other
LPWAN technologies, the Sigfox organization itself is a network operator meaning
that it provides the entire network infrastructure. Naturally, this means that Sigfox
is a non-cellular IoT technology compared to e.g. NB-IoT. Sigfox enables quick
configuration of new devices, as a device is required a Sigfox certification if one
wishes to join the network. In common with other technologies, Sigfox makes use
of the unlicensed ISM band around 868MHz in Europe.1 [9] By utilizing 192KHz
of the band in order to exchange messages through Ultra NarrowBand (UNB)
modulation, Sigfox is able to transmit packets of 100Hz wide with a data rate of 100
or 600-bits.2[10] The payload size of a Sigfox message is capped at 12-bytes in the
uplink, while merely 8-bytes in the case of downlink.[11] It should be noted, that
throughout the report, the terms packet and message will be used vice-versa to
indicate a Sigfox message.
Compared to alternatives, Sigfox has no form of attachment procedure. This was
opted for, as in a lot of IoT use cases downlink communication is unnecessary.
If it is the case that a downlink is needed, the Sigfox network chooses the most
fitting base station on various parameters for the transmission.[11] This form of
"no attachment" procedure requires less of an IoT device in terms of saving power
and complexity. However, as a consequence, this means it is not possible to verify
if a "connection" is established for an IoT device using Sigfox. Furthermore, in

The Sigfox network infrastructure is built upon advanced Software Defined Radios

Sigfox operates with secret keys in order to provide various security properties.
These are defined during the initial manufacturing process where each device is al-
located with a 32-bit globally unique device-ID, a unique symmetrical authentication
key (for authentication and integrity), and a Network Authentication Key (NAK).
Furthermore, in order to provide authentication and integrity of the Sigfox packet,
a Message Integrity Code (MIC) is used which is able to identify if the packet was
truly sent by a device. It should be noticed that MIC is commonly exchanged with
Message Authentication Code (MAC). The packet and NAK are used to generate
a token to provide security against adversaries. Additionally, Sigfox also protects
the packets from replay attacks by providing a sequence number to each packet,
which the core network can then check to ensure no duplication. [12–14] All of the
major security properties of Sigfox are listed in Table 2.2.
Sigfox avoids the problematic scheme of key exchange by directly settling on
specific keys for each device during manufacturing. However, as these are likely
never to be changed, it means that if the secret key were to be broken (brute-
forced) or in any way leaked, any communication done via the respective device
could potentially be intercepted and read in plaintext; future, or past messages.
Sigfox does not per default provide encryption of its payload, but merely offers
it as an option. It uses AES-128 in Counter mode, performing as a stream cipher
for the data, encrypting one bit (or byte) at a time. As stated by Sigfox, a client
must enable it, to achieve payload encryption or design it themselves, depending
on their individual scenarios.

MQTT is a widely used application layer protocol in IoT. In 1999 the first version

subscribers can distinguish the material they wish to receive by a topic filter,
meaning a client can subscribe to a set of topic filters such that the broker knows
what data the different clients wish to receive. In addition, this means that a
publisher chooses to which topic filter the data should be related to [31]. It should
be mentioned that a client can make a wildcard-filter subscription, targeting a

2.2.4.1
User Datagram Protocol
User Datagram Protocol (UDP) is defined as an unreliable, connectionless transport
layer protocol. It is defined as unreliable, as the protocol itself provides no form
of guarantee of delivery, it merely transmits a packet to a chosen destination and
continues on. Neither acknowledgements, time outs, re-transmission or ordering
of packets are available compared to TCP. Moreover, UDP has no allocation or
confirmation of connection between parties, meaning it has no way of knowing if
the intended recipient is able to receive a packet.
As a consequence, UDP is not able to provide any form of sequence of packets as
well as having high plausibility of losing packets unknowingly.[33] In regards to
error control, only a checksum field is included to provide a value to check if the
content of the packet was corrupted during transmission.
Fig. 2.8 UDP segment format.
In figure 2.8 the UDP format is shown, with a header consisting of a Source Port,
Destination Port, Total length, and Checksum. The header is 8-bytes, equally split
between the four parts, represented in 16-bits each. The total message length,
including both header and data, varies from 0 to 65535-bytes, meaning the data
(payload) itself can be between 0 and 65527.
With the defined properties, the protocol has many use cases, and is especially
suited for real-time/live data transmissions, as no error correction is needed when
the end user is only interested in the very latest data. Additionally, in terms of
IoT, the simplicity of UDP is an advantage due to factors such as significantly
lower overhead (storage, power consumption, etc.), no handshake, delays, re-
transmissions, and more.[34]

The Transmission Control Protocol (TCP) is defined as a connection-oriented,
process-to-process protocol. This means a connection is established whenever data
needs to be transmitted between entities using TCP, and when the communication
is done the connection is terminated. In other words, a connection is established
before the entities are allowed to transmit packets to each other, this is done by
the three-way handshake methodology and the use of sequence numbers.
Furthermore, TCP also supports flow control and error control, opposite UDP, which
is a significant functionality difference compared to UDP. Flow control is achieved
by six different flags, that can be used in different scenarios such as activating the

handling Urgent Pointer is URG, and the flag for Acknowledgment Number is ACK.

also accessible.[34] To give a better overview, the six different flags can be seen in

2.3
Cryptography
The objectives conventionally associated with the field of cryptography are many-
fold, however, a fundamental one is to ensure that the communication link between
trusted entities is kept confidential. This is of course without the possibility of an
adversary intercepting and extracting sensitive data. Encryption is the operation
of concealing the plaintext, which is the unencrypted data, in such a way that it
is computationally impractical for an attacker to reveal the true content of the
message.4 Contrariwise, decryption is the operation of reversing the encryption,
thereby retrieving the previously encrypted data and revealing the plaintext.
Fig. 2.11 General encryption & decryption model.
The general model of encryption and decryption is represented in Figure 2.11. The
plaintext, denoted as PT, is virtually any form of data that is possible to be inter-
preted as binary. As noted, the ciphertext (CT) is produced by CT = Enc(PT), while
it is possible to retrieve the PT by reversing it, i.e. PT = Dec(CT). Consequently,
this naturally shows the identity Dec(Enc(PT)) = PT. In many cryptographic algo-
rithms, a key or set of keys is used to perform the encryption/decryption. Whether
these keys should be kept secret or be publicly available depends on the type of
cryptography used, namely symmetrical versus asymmetrical.
2.3.1
Symmetrical
Algorithms based on the concept of symmetric cryptography are where a single
secret key is used in both the encryption and decryption process. Thus, if two
(or more) parties wish to communicate securely (encrypted), all the respective
parties must hold an identical key, either by agreeing on one or using a previously
distributed key. This is to avoid a potential eavesdropper intercepting the key if
it was sent accidentally over an insecure medium. Once all parties have the key,
messages can be encrypted by a sender and later decrypted by a receiver. The
symmetrical approach just described is depicted in Figure 2.12. The key used to
encrypt/decrypt the messages must remain secret indefinitely, considering that
if the key was retrieved in one way or another, an adversary would be able to

2.3.1.1
Stream Ciphers
A specific type of cipher, within the domain of symmetric cryptographic algorithms,
is a stream cipher which customarily operates on a singular bit or byte in a stream.
This means that each bit that is to be ’processed’ is encrypted (or decrypted)
one-by-one, segregated from the rest of the possible data in the stream. A stream
cipher performs its encryption/decryption in a series of operations.
Firstly, a
pseudorandom bit generator (keystream) is used with the purpose of producing
8-bit numbers. These produced bits are then added (XOR’ed) with the respective
bit from the plaintext. One must keep in mind that the generated keystream is not
the actual key itself. The secret private key used in symmetrical ciphers, is instead
used as a seed for the pseudorandom generator in a stream cipher.[37] The general
principle of stream ciphers is illustrated in Figure 2.13.
By using a PSK as a seed, it enables the stream cipher to start at a specific state in
the generation of bits, ensuring consistency on the other side when the decryption
is to be done. Besides the secret key, modern stream ciphers (and generally all
kinds of ciphers) implement an IV whose purpose is randomization. Normally,
a unique IV must be generated for each encryption session to ensure security.
The IV itself is perfectly safe for transmittance in cleartext, as the secret key is
still needed. If one does not use some form of unique IV, the stream cipher will
become incredibly deterministic, meaning it is possible to easily be attacked and
compromised.

In 2001, the U.S. National Institute of Standards and Technology (NIST) picked the
cryptographic algorithm ’Rijndael’ to be used for all encryption of data. Accordingly,
the algorithm’s name and minor details were changed, officially declaring it as the
Advanced Encryption Standard (AES). The algorithm is designed as a symmetric
block cipher, meaning it handles data in blocks of 16-bytes at a time. It requires
two inputs, a plaintext message, and a single secret key, as was illustrated in
Figure 2.12. AES defines three versions, dependent on the bit size of the utilized
key, AES-128, 192, and 256. This bit size also dictates the number of rounds used,
e.g. AES with a 128-bit key executes 10 rounds, AES-192 uses 12, and finally AES-
256 employs 14 rounds. Each round contains a set of functions that are performed
in a specific order, to achieve no connection between the original plaintext and the
resulting encrypted ciphertext.

to the original key, the functionality can be reversed, using the inverse methods,

Galois/Counter Mode (GCM)
In block ciphers, just like AES, it is possible to apply certain modes which extends
(affects) the algorithm’s functionality, and as a result, its produced security proper-
ties. For example, with AES, it is possible to apply a mode known as Galois/Counter
Mode (GCM), which transforms the AES block cipher into an authenticated encryp-
tion stream cipher. Thereby providing confidentiality, integrity and authenticity in
a single cryptographic scheme. The mode is defined by NIST, which is intended to
provide the popular AES as a stream cipher variant with extreme throughput, low
latency and low cost. GCM operates with two main functions, GHASH and GCTR.
Firstly, GHASH is a hash producing function which uses a key in its calculation.
Secondly, GCTR implies the usage of "Counter" mode where a counter is incre-
mented, i.e. "blocks" are numbered in sequence. In GCM, a block is then processed
by first using the count value, combine it with a unique IV, and then encrypt with
AES. Afterwards, the intermediate result is XOR’ed with the respective plaintext
resulting in the final ciphertext. The GHASH function produces the necessary
authentication Tag, which is used for validation.

2.3.2
Asymmetrical
Compared to symmetrical based algorithms, those constructed in an asymmetric
order make use of two distinct keys; a private and a public key. Theoretically, it
does not matter which key is used for encryption, as it provides different security
properties depending on the order. If the public key is used for encryption, only
the holder of the private key is able to decrypt the message. Meanwhile, if the
private key is used for encryption, anyone with the public key is able to decrypt the
message and read its content, while implicitly verifying that it originated from the
holder of the private key. This also solves the problem that was previously described
in symmetric algorithms with a PSK. As the public key can be publicly shared,
anyone can use it to encrypt a message, and the receiver (with the private key) is
able to decrypt it. Alas, asymmetric algorithms have enormous efficiency issues in
regards to encryption speed and memory consumption, rendering it impractical
on its own in an immeasurable number of use cases. This is a fundamental reason
as to why asymmetric algorithms are not used exclusively; they are remarkably
slow and exceedingly costly in comparison to symmetric algorithms, which are far
lighter, faster, and more efficient. [39]

2.3.3
Authenticated Encryption
Authenticated encryption (AE) is a scheme which is able to provide more than
the traditional encryption schemes, as it is able to provide both confidentiality
(encryption), integrity and authenticity simultaneously. This means that the data is
both protected from adversaries, as well as a receiver being able to verify that the
data has not been tampered with/corrupted in any way. AE uses three components
to perform its operations, a secret symmetric key, an IV, and plaintext. First the
plaintext is encrypted returning the ciphertext, which is then combined (XOR’ed)
with the IV. These two values, now a single data structure, is sent through a hashing
algorithm (HMAC-SHA-256) to compute a Tag.9
Now, the IV, Tag and Ciphertext is sent to a receiver, which means that if an
adversary should tamper/corrupt with either the IV or ciphertext, it would require
the PSK to calculate a new Tag to ensure consistency. On the other hand, if the
adversary was to corrupt the Tag, then the decryption function of the AE scheme
would simply discard the decrypted plaintext as the IV and ciphertext could not
be authenticated (as the Tags are not consistent). This is possible as the receiver
calculates its own Tag with the received components and finally compares it to the
original Tag to provide integrity of the data (Figure 2.18).
An expansion introduced to AE schemes is "Associated Data" (AEAD), which is data
that is not required to be encrypted (confidential), however it must be verified.
One scenario is typical network packets where header information10 generally
does not require confidentiality such as in a case of routing purposes. However,
the header information does require integrity as one can imagine if the sender or
destination address were to be manipulated, it could have grave consequences.[40]
As shown in Figure 2.18, the authenticated encryption with associated data uses
the same symmetric secret key with encryption and decryption. In the decryption,
the decrypted plaintext is only ’released’ if the integrity Tag can be verified, in
essence concluding that the hash of the data is identical. [41]
The sender transmits both the IV, Associated Data, Ciphertext and Tag to the
receiver. The IV is allowed to be sent in cleartext as it is merely a starting point of
the algorithm; it is the secret key that is needed to make sense of it all. Once the
receiver has these necessary components, it is possible to ’start’ the decryption
at the correct place, with the IV and secret key.
One can relate this to any
statistical/mathematical operation that includes seeds, as a ’starting point’. The IV
however, must be unique for each message it encrypts, this does not mean random,
but could simply be an ever increasing-counter.

It is clear, that authenticated encryption is far superior to traditional encryption
algorithms that only provide confidentiality, as AE can additionally guarantee that
the receiver can verify that a message was truly created by the sender and the
data is not manipulated during its time in transit. AE is generally very well suited
for resource constrained environments such as those present in IoT solutions, for
multiple reasons. The most prominent one being that AE is able to combine several
cryptographic primitives providing confidentiality, integrity and authenticity in
one. Comparably, in order to achieve these properties without AE, one would need
to perform them in specific order and with individual implementations causing
heavier resource usage and many possible rooms for error.

Lightweight Cryptography
Lightweight Cryptography (LWC) is centered on the complicated task of developing
cryptographic algorithms that are able to provide as strong security as heavy
resource algorithms but instead operate with far less overhead in terms of memory
footprint, general resource consumption, power and many more relevant metrics.
This is targeted towards embedded devices that are considerably poorer in all mea-
sures, while requiring the capability of transmitting encrypted data and ensuring
that it is not possible for a malicious third party to read the data in plaintext. [42]
The recently finished Competition for Authenticated Encryption: Security, Appli-
cability, and Robustness (CAESAR) was created with the objective of selecting
appropriate cryptographic algorithms with desirable features within contrasting
scenarios. Use cases were selected with differing focus:
1. Lightweight applications
2. High-performance applications
3. Defense in depth
In the lightweight application use case, the main focus was turned towards per-
formance in resource-constrained devices, while also taking special note of side-
channel attacks.[43] The two ciphers selected as finalists in the lightweight cate-
gory were ACORN and ASCON.

ASCON
ASCON-128 is an authenticated encryption stream cipher that was selected as
one of the main choices in the CAESAR competition for lightweight applications.
The algorithm provides 128-bits of security while internally utilizing a 320-bit
permutation state. The 320-bit permutation is a design choice made to increase
the security and robustness level within the cipher. One extremely interesting
note about ASCON is the operating functionality of obfuscating the data, which is
done solely with bitwise AND, NOT, and XORs. These types of calculations are very
effectively implemented on hardware devices, but also execute exceedingly fast in
software. Furthermore, the operations are quite light in relation to computational
power, meaning operating on IoT devices such function is considered optimal. As

Prior to performing the encryption, decryption, and Tag computations, ASCON
requires its main 320-bit state to be initialized. This is done by employing the
secret key and the initial IV supplied. In ASCON encryption, a state is continuously
transformed by internal permutations in order to ensure the confidentiality of the

ASCON, the IV used in each encryption stage must be unique, i.e. an IV must never
be re-used.
Encryption Procedure
In ASCON the encryption and decryption process consists of several rounds which
are specified in relation to the corresponding setup. Furthermore, it should be
noted that the 320-bit permutation state (S) is split up into five pieces, to be able
to use each one individually, more specifically for round transformations. In some
scenarios through out the encryption/decryption process the state S needs to be
interpreted as a byte-array, where this segmentation come in handy.
State : S = s0||s1||s2||s3||s4
The encryption consists of four stages, a strong initialization function of the key,
the functionality of integrating the associated data, the integration of plaintext
with an output of ciphertext, and to finish the encryption a finalization function
which is eventually outputting a 128-bit Tag.

3.2
Cryptographic Algorithms
The cryptographic algorithms selected to provide end-to-end security of the payload
in terms of confidentiality, integrity, and authenticity are the two finalists of the
LWC competition, ACORN and ASCON, and the current modern standard AES-
GCM. These three ciphers are all authenticated encryption algorithms as previously
explained.
ACORN and ASCON were both chosen due to their favorable design specifications
in relation to constrained environments. They are by definition designed to operate
with IoT devices as they are both considered/defined as lightweight encryption
algorithms.
Specifically, ACORN in comparison to AES-GCM, is less resource
demanding and more efficient in terms of power consumption and processing
power utilization. Another advantage of using ACORN is the supported possibility
of performing parallel computing, resulting in both high-speed hardware and
software, as multiple steps can be computed at the same time.[47]
In regards to ASCON, this authenticated encryption algorithm is explicitly designed
for great security robustness without being heavily resource demanding, and
without leaving a remarkable areal footprint in hardware. ASCON is also defined as
a lightweight cryptographic algorithm, implying good performance on embedded
devices and great for scenarios where communication is directed to servers with
unlimited resources (High-end servers).[52] Both ACORN and ASCON competed
in the CAESAR Competition with an end result of taken first and second place.
Both algorithms are outstanding choices where cryptographic operations must be
performed by IoT devices themselves.
AES-GCM was chosen due to its major popularity throughout almost all branches
of industry, where authenticated encryption is required. This means, it is extremely
well tested as it is used practically everywhere, while also being guaranteed to
operate properly. As was shown in the theoretical section, most of the protocols
employ AES-GCM which is the industry standard, while many alternatives are
far better suited for resource-constrained environments present in IoT solutions.
This means AES-GCM is implemented to compare the two previously mentioned
algorithms, ACORN and ASCON, which are allegedly far better than AES-GCM in
almost all characteristics.

4.1
Cryptographic Algorithms
In this section, the three AE algorithm implementations will be explained. They are
combined in two generic encryption/decryption functions, meaning that they are all
designed in the code in such a way that the specific algorithm’s internal operations
are abstracted away. The algorithms are all implemented with the help of the Rhys
Weatherley’s arduinolibs Crypto library. [57] The library provides various forms of
encryption algorithms applicable for 8-bit platforms written in C/C++ meaning it is
able to be used for practically any framework.
It must be explicitly stated that in all of the forthcoming implementations, it is
assumed that the necessary symmetric keys are present on "both" sides of two
communicators. Particularly the element of a conceivable key exchange is not
considered, as it is not a scope of this project. A PSK is necessary on the receiver’s
and sender’s side as the three symmetric AE algorithms require an identical key
applied during encryption and decryption. Moreover, the implemented ciphers
are treated as authenticated encryption algorithms, therefore not operating with
Associated Data. The associated data is not considered in these implementations,
as the focus is strictly on the payload.
4.1.1
Preparation for encryption
Before actually performing the encryption, a few imperative steps must be taken,
such as generating the IV. As described previously, at first a new IV must be used
for each message (that is to be encrypted) to ensure that is not possible to break
the encryption. For this reason, the following function has been defined:
GenerateInitializationVector(uint8_t *IV, int size)

variable size. However, as all of the AE algorithms use 128-bits/16-bytes IVs, 16-
bytes have been statically declared as the size. The library TransistorNoiseSource
is used which implements a large set of the necessary functions to ensure ’true’
randomness from noise sources such as analog pins on MCUs. As observed in
Figure 4.2, the function creates a transistor noise source with a selected analog
input pin.
This pin is attached to an RNG object which handles the random
generation of bits. Next, a while loop is entered on the condition of a boolean

4.2.1
NB-IoT
The NB-IoT implementation is realized with a SODAQ SFF board, mounted with a
U-blox SARA-R410M module enabling the usage of NB-IoT, amongst a selection of
technologies. The SODAQ board holds a CPU clock speed (frequency) of 48MHz,
compared to the UNO which only holds 16MHz. In consistency with many other
modules, the SARA is communicated with and instructed via a serial connection
with traditional AT commands. However, the implemented code adopts the use of
two external libraries, ’Sodaq_nbIOT’ and ’Sodaq_wdt’. [58, 59]. The main purpose
of the two libraries is to abstract away the low-level AT commands and provide
high-level functions. This makes it far less prone to serial communication errors,
as one does not need to handle the direct serial communications of sending an AT
command, waiting for the response, interpreting it, and so on. This is all handled
by the libraries as they perform the required error handling, thereby only returning
error codes and booleans in case of faults.
The main logic of the designed NB-IoT solution operating on the SARA can be
inspected in the flowchart of Figure 4.6. First, the mentioned NB-IoT libraries are
imported and declared for usage, and the IP, Port of the IoT back-end, AP name,
and IP of the TDC NB-IoT infrastructure is defined. Next, the setup() is entered
where the NB-IoT modem on the SARA module is initialized and begins to acquire
a lock on a successful establishment of a connection to the TDC network. It will
remain ’indefinitely’ in this state until it receives an acknowledgement from the
network that it should be registered and connected with.
After the connection has been established in the setup(), the main loop() is entered.
Here, it is once again checked if the NB-IoT connection is still alive which is mainly
done as a redundant check in case the connection should have failed. Following
this, the message that is intended for transmittance is encrypted producing the
IV, Tag, and Ciphertext which are all collected in a single packet, as previously
explained. NB-IoT allows a maximum payload size of 1600 bytes, thus making
it irrelevant to stage any form of fragmentation of packets or similar. When the
assembling operation has finished, the resulting payload is then transmitted as a
UDP packet via the TDC infrastructure. From here, the device returns to the loop()
state and repeats the process.

computation of a Tag. The calculated Tag is compared to the received Tag, which
leads to two scenarios. If the tag is valid, the decrypted data is released and it is
displayed. If the tags are invalid, the packets are discarded.

With the aim of conducting a fair assessment of the performance by the three
implemented AE algorithms, as many elements as possible must be held static.
Accordingly, a fixed byte is used for all the encryption/decryption operations, a
fixed key, and a fixed IV. Moreover, the timing measurements are initiated only
once the respective operation begins and stopped once done. No superfluous code/-
operation/etc will affect the measurement. This also means that operations such as
computing the Tag and checking it for integrity are not included in the calculation,
it is exclusively the encryption/decryption operation of the AE algorithms.

testing. These can be viewed in Table 5.2.

The values are fixed as it would produce incomparable results if e.g. dissimilarities
would occur between keys, IVs, and plaintexts. In addition, to perform accurate
timing measurements, multiple rounds of the particular algorithm’s encryption/de-
cryption must be done. A single iteration is incredibly fast, averaging at around a
few microseconds (µs) meaning it is difficult to correctly measure the time-span
with micros() and other typically available timing functions on MCUs, this is es-
pecially evident on ARM/ESP/etc, based ones. An example of this is the micros()
function which is only able to measure in 1MHz increments, making it exceed-
ingly problematic to get an accurate result in terms of the duration of operations
executing in less than a microsecond (<1 µs).
Another consideration to take into account is unexpected disruptions caused due to
underlying interrupts. These interrupts are naturally performed by the MCU’s CPU
to perform its tasks. If it is the case that an interrupt should occur unknowingly
(which is likely) during the timing measurements of the cryptographic operation,
then the result will be corrupted as it will possibly appear to be relatively slower.
Hence, depending on the specific frequency of the interrupts during the test, each
’run’ may produce fluctuating results, which are consequently inoperable.
All these aforementioned complications are dealt with by executing hundreds
of iterations and averaging them out. The anticipated overhead in the results,
stemming from the interrupts, is now inconceivably minor noise which therefore
is implicitly ’ignored’. As a result of doing this, it is now possible to accurately

6.1
Cryptographic Algorithms
In the Cryptographic Algorithms result section (section 5.1), it was seen that the
implemented authenticated encryption algorithms passed all unit tests as well as
verification of the serial consoles. In terms of the performance of the separate AE
algorithms (ACORN, ASCON, AES-GCM) several aspects can be discussed. Firstly,
it was explained how difficult it is to measure accurate sub-microsecond operations,
which is the case with all of the implemented AE algorithms. It was selected that
500 iterations were enough to average out the noise, however, this claim requires
additional testing in order to truly conclude if it affects the performance of the
ciphers. Additionally, it would have been beneficial to investigate the importance of
the noise in the generated results. One could argue that several thousand iterations
should be executed to even further minimize the unintended noise generated by
the described underlying interrupts by the CPU. However, it is not possible to
conclude, without further research, if an increase in iterations would not cause
other unforeseen alterations to the measurements.
In regards to the actual speed performance of the authenticated encryption algo-
rithms, it was observed, regardless of the CPU frequency of the respective MCUs,
how the ACORN cipher performed far superior in all cases. Although, it could also
be seen from the plots that a lesser difference was observed with a higher CPU
frequency (e.g. in the ESP32 case). It is self-evident that a higher CPU frequency
would indicate faster performance, regardless of the algorithm, which was also
shown. Especially in the case of the UNO, the MCU with the lowest CPU frequency
(of the ones tested), it can be seen how big of an importance the actual algorithm
utilized has on the performance. This is fairly interesting, as one could imagine
that a MCU with an even inferior CPU frequency (<16MHz), would benefit greatly
with either the ACORN or ASCON cipher for encryption and decryption purposes.
In addition, it could also be observed how the UNO approximately halved the
==PAGE:140==
124
Discussion
necessary time during encryption with ACORN compared to ASCON. The same
observations are remarked for the decryption results.
The SODAQ board has a clock speed of 48MHz, i.e. three times the speed of an UNO,
however, comparing the confidence intervals during the encryption measurements,
it performed remarkably quicker and with less fluctuation. The SODAQ performed
10.45, 6.275 and 4.645 times faster with ACORN, ASCON and AES-GCM, respectively.
The ACORN algorithm is the one with the best performance speed in all cases, then
comes ASCON and finally AES-GCM. This is as expected since both ACORN and
ASCON were designed to operate with extremely limited MCUs and still perform
as fast as AES-GCM.
In conclusion, in these tested scenarios, it is deemed that ACORN is the best
possible choice in terms of processing power, as it is able to perform the necessary
cryptographic operations extremely quickly with minimal provided processing
power. Finally, in the results extracted above, it is noted that it is exclusively
the encryption & decryption performance that is considered. It would have been
interesting to investigate how the CPUs handled the Tag computation and validation.
However, it is likely that the exact same findings would have occurred, as they are
both very much dependent on the CPU frequency.

NB-IoT
The NB-IoT result section, (subsection 5.2.1), showed and verified that the network
communication and encryption that was implemented, worked correctly. In this
section, the extracted results and implementation for the NB-IoT system will be
discussed.
In the implementation of the NB-IoT system, it was decided to collect the IV, Tag,
and Ciphertext as a singular packet and transmit it. This implementation decision
was taken, as the protocol supports a large payload, therefore requiring no form of
packet fragmentation. As was observed, this functioned flawlessly as the device and
the IoT back-end had no trouble sending and receiving the content. Regarding the
security of the payload, it was seen that it is kept confidential from any "outsider",
even the network operator facilitating the packet flow, as they have no way of
decrypting the content. Any intermediate entity in the TDC NB-IoT network will
not be able to extract any intelligible information, due to the AE algorithm applied.
This showed that it was possible to provide end-to-end security of the payload,
beginning at the SODAQ device and ending at the custom-built IoT back-end and
its web application.
==PAGE:141==

It is deemed very beneficial to apply a cryptographic AE scheme, as the one
implemented, as it provides a NB-IoT system with end-to-end security. Depending
on the actual requirements of an IoT system employing NB-IoT device(s), minor
bottlenecks can be imagined. One conceivable downside is the larger packet size
(due to now containing IV, Tag, and ciphertext). This is a very minor inconvenience,
compared to the benefits, as the protocol is intended to support larger payload

This is also in regards to the cost of sending packets, imposed by the
network operator. Another minor bottleneck imaginable is the IoT device, however,
as shown with ASCON and ACORN, both algorithms are extremely lightweight,
thereby making the cryptographic operations highly effortless, quick, and non-
costly to perform.
6.2.2
Sigfox
Through the result section (subsection 5.2.2) of the Sigfox system, multiple steps
were verified to assure its functionality. In Sigfox, a maximum of a 12-bytes payload
is available to a device transmitting a message. This warrants a fragmentation
process as was implemented, where it was shown that it required a total of five
packets to contain the IV, Tag, and a single Ciphertext payload of up to 12-bytes. As
observed, this performed very well as the Sigfox back-end was able to receive all
of the messages and execute the respective HTTP Callbacks to the IoT back-end.
Through the usage of an AE algorithm, any third party listening in on the trans-
missions is unable to understand what is being sent by the device. Additionally,
Sigfox has no way of making sense of the messages as they are all either IV, Tags,
or encrypted data, which cannot be decrypted. This also means, that all the data
stored, by the Sigfox back-end, is unusable to anyone except the holder of the PSK.
This implies that if the Sigfox back-end is compromised at some point, the data
transmitted by the device(s) would be incapable of being decrypted, providing far
better security. Additionally, the applied AE algorithm on the payload data assures
persistence in payload confidentiality. As a result, once the payload leaves the
Sigfox back-end, via an HTTP request, it is guaranteed to remain secure, thereby
providing end-to-end security.
Considerations regarding Sigfox’s own security mechanism have also been made,
stated in section 3.1. The payload encryption offered by the protocol, will eventually
be decrypted on the Sigfox back-end, meaning further distribution of the data will
be insecure. Enabling Sigfox’s security will not be sufficient in use cases where
the data is used after being received and decrypted by the Sigfox back-end, as it
will only secure the radio layer link. In addition, imaging a case where the Sigfox
PSK being compromised between the device and the Sigfox back-end, a malicious
party will be able to act as a MITM, whereas if ACORN or ASCON is applied, it is
==PAGE:142==
126
Discussion
not a necessity to trust the security of Sigfox. That being said, enabling this as an
additional security level, would obfuscate the data even further than just applying
your own solution. I.e. having two types of encryption enabled in the transmission
to the Sigfox back-end and only one applied when distributing data onward.
It was shown that the implemented logic was functioning error-free, however, the
fragmentation process can become costly in a Sigfox solution. If an IoT system,
and its designers, have taken the definitive choice of utilizing Sigfox, as well as
needing complete assurance in terms of the security of the payload, it is an option
to use such a scheme. Nevertheless, this approach has multiple downsides in
the domain of Sigfox and IoT. Requiring an average IoT device to send multiple
packets is costly in terms of power consumption, the lifetime of the device, and
other relevant parameters. Additionally, as Sigfox is a subscription-based model, it
is also possible that a multiple fragmentation process is financially undesirable. It
might impose a higher financial cost since more packets have to be sent (the IV,
Tag, and Ciphertext).
Sigfox messages are susceptible to interference of various levels, meaning packet
loss occurs over longer distance of transmissions. This can be another underlying
complication of applying such a scheme that requires multiple packets to be
sent. Imagine a message flow like the one described in subsection 5.2.2, where
five messages are transmitted with the final purpose of having the IoT back-end
display the encrypted content. If just one of those five packets were lost during
transmission, the entire flow of the five packets would be considered in vain.
Without all of the five packets, the ciphertext cannot be decrypted or checked for
integrity, thereby making it impossible for the receiver to evaluate the message.

both IV, Tag, and Ciphertext, which is transmitted and then disassembled at the

BLE
BLE is a widely used technology within the field of IoT, which is one of the main rea-
sons for choosing it for this project. In subsection 5.3.2, the results are presented,
gathered by executing and analyzing the implementation made for BLE. Similar
to Sigfox, BLE has a substantially lower payload size allowed than many other
technologies, while only allowing to transmit 20-bytes of payload, at a time. In other
words, while encrypting a defined message, usually the IV, Tag, and Ciphertext
are assembled into one single packet and then transmitted, whereas for BLE it
has to be split up into three packets due to the payload size limitations. This is
only applied in the case where encryption is enabled, as solely transmitting the
plaintext, the payload will not exceed 20-bytes, in relation to the implementation.
For the communication between the Master and Slave, a custom-made exchange
was implemented, which can be seen in Figure 4.24, where the IV, Tag, and
Ciphertext were transmitted via their own write command.
Potentially, if the
ciphertext exceeds 20-bytes, this would be split up and transmitted in several
packets and finally be concatenated on the receiving end.
The custom-made
exchange implementation is deemed successful, as the content on both sides,
Master and Slave, are identical after the Master performed decryption on the
ciphertext.
Noteworthy is that the newest version of BLE actually supports, upon an agreement
between all entities, a larger payload size than the default 20-bytes. Utilizing
the concept of packet length extension, introduced in BLE 4.2., it is possible
to have attribute data with a payload size of maximum 242-bytes.
Optimally,
the communicating entities could have experimented with adjusting the allowed
payload, fitting the size of a message containing the IV, Tag, and Ciphertext in one
singular packet, such that the complex fragmentation could have been avoided
together with the custom message exchange. [24] This would have been an ideal
situation, as this would produce the same results as in the protocols where no
fragmentation was required, merely transmitting a single packet containing all the
necessary information to decrypt the ciphertext.
Furthermore, as an additional verification layer, besides verifying the content
sent and received, an Ubertooth One sniffer was used to capture/intercept the
communication. In cases where a malicious party, whether it is used for verification
purposes or not, tries to obtain the communication, an AE algorithm is a perfect
choice, as it offers properties ensuring confidentiality, integrity, and authenticity.
The intruder was under no circumstances able to see the actual content transmitted
while one of the three AE algorithms was applied. Disabling the encryption while
performing sniffing with Ubertooth One, it was observed how the content could be

As a direct consequence of this, the default security metrics of BLE are considered.
In BLE setups the security metrics are designated by the implementer of the

Per default in many BLE implementations, encryption is not applied,
though different GATT authentication types are available e.g. allowing one to
choose an authenticated encryption mode. However, insisting on solely using BLE’s

hopping where this is further elaborated on in Appendix E. Several retries of the
Ubertooth would eventually enable one to acquire a successful lock, allowing BLE
packets to continuously be captured.

Chapter 7
Conclusion
In this ending chapter of the report, a conclusion will be given to each aspect
investigated and implemented in this project.
In conclusion, the cryptographic authenticated encryption algorithms, ACORN,
ASCON, and AES-GCM were implemented with the purpose of providing end-to-end
security of the payload in different IoT protocols and implemented systems. The
security entailed in supplying the IoT systems with confidentiality, integrity, and
authenticity of the payload. The respective algorithms’ performance was evaluated
which concluded that ACORN performed significantly better than the two others in
terms of the processing power (CPU frequency) of the respective MCUs. This also
showed that the poorer MHz available to an MCU, the more significant became the
choice of algorithm.
The protocols, NB-IoT and Sigfox, were implemented to be used with the AE
ciphers in the scenario of having a D2S setup. In the case of NB-IoT, it was possible
to assemble the necessary AE components in a single packet, providing complete
end-to-end security from the device to the custom-built IoT back-end. This is due to
NB-IoT providing a very large payload size, thereby making it a perfect candidate
for a ’user’ applying payload security.
Concerning Sigfox, a fragmentation process was required, due to Sigfox scarcely
providing 12-bytes of payload in its uplink transmissions. However, it was shown
that the Sigfox back-end was able to receive all the packets, thereby making it
possible to forward the IV, Tag, and Ciphertext in a proper manner. The Sigfox
back-end allows HTTP Callbacks to be used to forward the data to applications and
services. This content was sent to the IoT back-end thereby providing end-to-end
security.
The communication protocol MQTT was implemented as both D2D communication,
but also as D2S. To realize the communication from D2S, AWS was set up to be
operating as a Broker, where the data was further distributed to the IoT back-end
for data handling. In regards to the communication from the Publisher to the
Subscriber, the Subscriber subscribes to a specific topic, such that the Broker
knows what data should be forwarded. The implementation in terms of commu-
nication links was successful as no flaws or issues were experienced. However,
without an AE algorithm applied, the content of the data packets would be sent as
==PAGE:149==
133
cleartext, as MQTT offers no default security properties, as these are designated
by the implementer. The MQTT communication was secured by applying one of the
AE algorithms, where only the Subscriber with the PSK can successfully decrypt
the content. No matter if the data was encrypted or sent as plaintext, it was
collected in a single packet and transmitted. The payload size of the MQTT packet,
when using the AE algorithm, was a bit larger, as the IV, Tag, and Ciphertext were
assembled into one packet and transmitted. Overall MQTT worked impeccably,
thereby providing end-to-end security.
An IoT back-end was implemented on a cloud-hosted VM to facilitate NB-IoT, Sigfox,
and MQTT which made it possible to receive and display the payload content, veri-
fying the systems operated correctly with the algorithms, providing confidentiality,
integrity, and authenticity. This was done through the implementation of a UDP
Python Server to receive NB-IoT UDP packets, and a web application to receive
HTTP POST requests.
WiFi and BLE were implemented in the scenario of Device-to-Device communica-
tion with the same objective. It was concluded that it was possible for the WiFi
implementation to securely communicate with the use of the implemented AE algo-
rithms. By performing WiFi frame sniffing to capture the communication, it was
verified that the content of the frames was completely secure. Furthermore, the
IV, Tag, and Ciphertext, was collected in a single frame, as WiFi (and UDP) allows
large payload sizes. This made it possible for the receiver to perform validation of
the received Tag.
The BLE implementation proved that even if available security mechanisms pro-
vided by BLE were disabled, the usage of the implemented AE algorithms guaran-
teed the payload content was secure. Similar to WiFi, the BLE communication was
also sniffed and captured, though for this communication an Ubertooth One was
used. Through analysis, the packets were deemed to be secure, as no content was
readable whatsoever. Additionally, the BLE communication used a custom-made ex-
change of the IV, Tag, and Ciphertext, since the maximum payload size is 20-bytes,
and therefore fragmentation of the content was required, which was also verified
through sniffing.
In summary, it is concluded that all problem-statements were fully achieved, as AE
algorithms were implemented, which can be applied in order to provide end-to-end
security of the payload in all of the analyzed protocols. Besides the encryption, a
wide spectrum of different communication protocols were set up, covering both long
and short range, inspecting various security properties in each technology. Lastly,
a performance analysis was made showcasing that the best choice of algorithm
was ACORN, operating in resource constrained environments/devices.

Chapter 8
Future Works
In this final chapter, an exploration of possible extensions to the project will be
made. These should be seen as new paths for the project, and not necessarily
specific improvements for the different systems. Multitudes of extensions are
always available for research and further investigation, which is why only those
deemed the most significant and interesting are explored.
In the matter of the authenticated encryption algorithms, it was decided not to
focus on associated data in regards to the communication exchanges. It could
have been interesting to design and implement functionality to support this in
regards to the D2S and D2D scenarios. Moreover, it was decided to focus on the
software side of applying encryption to protect the payload. However, hardware-
based encryption would have been a great aspect to include, as it serves as a very
good comparison. This would have been a major undertaking, due to needing
to develop several specific hardware components able to perform the respective
cryptographic algorithms. Moreover, these custom-built hardware components
should then further be able to interface with various MCUs and transfer the
encrypted content and necessary AE components.
As for the performance of the ciphers, the processing power of the IoT devices
was the main focus, however, it would have been especially suitable to investigate
other metrics such as power consumption, memory usage, and more. The power
consumption was at one point considered during the project. It was deemed out of
scope, as it is exceptionally difficult to measure specific power consumption levels
during incredibly fast operations on MCUs. This would have required expensive
equipment as well as detailed expertise within this domain. With respect to memory
usage, it could have been beneficial to look at the memory footprint of the various
algorithms. However, this is incredibly implementation specific, as it depends on
the usage of libraries, operations and more.
In regards to the results produced, it was seen that sniffing served as a great way
to verify the implementations operated correctly in the cases of WiFi and BLE.
In the case of NB-IoT and Sigfox, it would have been fascinating to research and
investigate how to perform sniffing of those two protocols and their respective
spectrum. Sniffing those two protocols would likely also have produced even more
results to investigate in terms of the encrypted payload.
==PAGE:151==
135
Another context that would have been interesting to delve into, would have been
downlink communication in the context of the D2S scenarios. In such a case, it
could have been interesting to see how much complexity would arise by having the
devices (in NB-IoT and Sigfox) receive packet(s), and disassemble its content into
IV, Tag, and Ciphertext, and finally perform the decryption. This would likely have
proven even more difficult, especially with Sigfox as it has an even more limited
downlink payload size of only 8-bytes.
In connection with the web application, further, development could have been made
to provide user authentication, more complex display of the packets, advanced
graph interactions, inspection layers, and much more. With relevance to providing
user-based interaction, e.g. through a web application, it could have been poten-
tially rewarding to interface with a custom-built mobile application. This could
have performed similar tasks as the web application, although with a different
perspective in regards to functionality and objective.
In a completely different level and scope, the form of encryption known as homo-
morphic encryption could be extremely interesting to dive into, both in regards
to operations on embedded devices and the complexity of modifying data without
decrypting the content. The essence of homomorphic encryption is allowing com-
putational permutations without decrypting the content, where the entity initially
applying the encryption can decrypt with the applied changes. This is considered
a very efficient method of handling secured data on embedded devices, since all
cryptographic operations in relation to encryption can be done by the IoT device.
Once it transmits the encrypted data, all data handling may be done by a high-end
server. This means that the server does not need to know the secret key used to
encrypt the data, only the device needs to know it. As such, this would mean that a
key exchange is unnecessary, solving a very big problem in relation to IoT security.
The main objective of homomorphic encryption is to provide encrypted data from a
transmitter to a receiver with far broader resources without prior communication
of a key or similar. [66]

professional development. Other more sophisticated IDEs such as the CLion IDE

tool-set. This makes it possible to use the incredibly powerful CLion IDE’s tools

With the aim of providing a clear and understandable overview of how the develop-

the concept of defined development environments, which specifies multiple nec-
essary configurations, both regarding the software as well as the hardware specs
for the used board. These environments are defined within the file ’PlatformIO.ini’,
which makes it possible for CMake to build the C/C++ code. An example of such an
environment can be seen in Figure A.1, where it is the one used for the development
of the MQTT Publisher with an ESP32 board.
It is clear from the configuration that a large number of configurations are available,
however, the ones deemed most important are the ones displayed here. Noticeably,
it is possible to include a src_filter that specifies exactly which files are to be
flashed to the microcontroller. Further, lib_deps makes it possible to use external
libraries without actually manually downloading them and installing them. They
simply need to be referenced and they will be available for use in the implemented
source code.
Naturally, each specific development environment will have its own source filter and
library dependencies depending on the actual code that is meant to be implemented.
A simple example could be that the NB-IoT environment will include the libraries
necessary to operate the SODAQ board with the U-blox module.

Appendix E
Ubertooth One
In this appendix, information related to the Ubertooth One will be presented, as
well as a comprehensive overview of how one intercepts Bluetooth (and BLE)
packets.
The Ubertooth One is a fully enabled Bluetooth classic and BLE sniffing hardware/-
software tool which is able to interpret Bluetooth packets in the 2.4GHz spectrum
even while the Bluetooth protocols employ frequency hopping. This enables a user
to read all packets sent across the air while being within range of the devices
communicating. The Ubertooth One superseded the original Ubertooth Zero which
was created by Michael Ossmann and others, which later evolved into the current
Ubertooth One in 2011.[67] The Ubertooth is not only designed to receive, but
also be able to act as a transceiver, making it possible for the user to both be a
passive sniffer as well as crafting Bluetooth packets. The source code operating on
the microcontroller of the Ubertooth is developed as open source, meaning that
all source code can be found publicly available on Github.[68] This also makes it
possible for essentially anyone to create custom firmware, apply new features or
anything else within their creativity/needs. On a physical aspect, the Ubertooth
is made up of several hardware components, which are listed and described in
Table E.1.

Appendix F
WiFi Frame Sniffing
In this appendix, the necessary steps in order to perform WiFi frame sniffing, will
be explained. In theory, any WiFi card/dongle/etc can be used to perform frame
sniffing of 802.11 versions as long as they fulfill the requirements of the system
and support Radio Frequency Monitor (RFMON) mode1. What is extremely vital
is making sure that the chip-set is compatible with the used OS (e.g. Kali Linux,
Ubuntu, etc.) and that the 802.11 version of the system targeted is supported by
the WiFi dongle. If some form of virtualization software is used for the OS (e.g.
VMWare, Virtualbox) it must also be ensured that the USB is passed through to the
VM OS, and not the host itself.
One must make sure of several additional parameters such as:
• Card is able to enter "monitor" and/or "promiscuous" mode.
• Specific drivers for the dongle is installed.
• Determining if channel switching is used.
• Identify the channel of the selected WiFi (e.g. channel 1)
• Identify frequency (2.4GHz or 5GHz)
This is of course not an exhausting list, merely the most important ones. In this
specific case, the dongle ’AWUS036ACH’ is used. The following steps described
are all required to get the dongle to operate in monitor mode, thereby enabling it
to receive all the frames within its range. All the following commands also assumes
that they are executed as root (sudo). First, the OS itself is updated and a reboot is
initiated.
$apt update -y && apt upgrade -y && reboot
Afterwards, the appropriate driver must be installed.
$apt install realtek-rtl88xxau-dkms -y
Once this is done, the interface should be detectable by executing iwconfig.
As marked, the interface is recognized with the identifier ’wlan0’, however, it
automatically starts in the "Managed" mode. As noted, the interface needs to be
1RFMON mode is commonly simplified to the term Monitor mode
