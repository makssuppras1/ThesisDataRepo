Abstract
In this thesis, we integrate an hardware accelerator to a processor, to perform computationally
expensive algorithms in a performing way.
In particular, we work on a processor based on RISC-V architecture called Rocket Chip,
going through the typical design phases of a processor such as implementation, synthesis and
veriﬁcation, and evaluating its performance in terms of area, timing and energy consumed to
perform certain functions.
These types of processors offer different options for interfacing with external devices, but
since these processors are designed through the Chisel hardware description language, the
interface mode can be challenging for a hardware designer who has always worked with
VHDL or Verilog. For this reason, we design a wrapper that allows future users to connect
an external device without having to know the Chisel and the processor communication
protocols with the memory and external devices.
With the wrapper, we connect an FIR ﬁlter designed in VHDL to the processor to analyze the
beneﬁts in terms of performance and energy savings.
In particular, we compare the execution speed of an algorithm for bi-dimensional image
ﬁltering performed directly on the processor via software with the same algorithm performed
via hardware on the external FIR ﬁlter.
Based on this comparison, we observe that the integration of the external circuit in the
processor brings enormous advantages in terms of speed and energy consumed to perform
the function mentioned above.

CHAPTER 1
Introduction
The rapid development of technology and integration in recent years is allowing to run
increasingly computationally expensive algorithms on microprocessors.
What once seemed a utopia, today is reality, and so, with actual hardware, it is possible
to perform complex calculations, such as that of a machine learning algorithm, relatively
quickly.
What is even more amazing is that the technology is allowing these sophisticated algorithms
to run on mobile devices, which therefore it cannot beneﬁt from a continuous supply of
current by a cable. For this reason, the main challenge for modern hardware designers has
become the reduction of devices power consumption.
Many times a solution can be to couple a processor designed and optimized to perform a
speciﬁc function to the central processing unit. In this way, it is possible to have very high
computational capabilities computing in parallel many similar calculations, and to have a
lower total energy consumption.
This type of processor is often referred to as co-processor (or accelerator) in the IT world.
Co-processors are designed to accelerate tasks which are too slow to be performed by soft-
ware routines running on a general purpose processor, and besides, they are often optimized
to perform that function by consuming as little power as possible. Last but not least, the
co-processor integration allows the processor to perform other tasks while it is dealing with a
speciﬁc computationally expensive task.
Over the years, with the term co-processor, we referred to a great variety of arithmetic logic
units integrated into the processor and part of the pipeline, or to external units used to expand
the system capabilities, such as graphics processors. In the last few years, it has instead gone
towards the direction of integrating these computational units within the die of processors.
For instance, FPUs (Floating Point Units), which were once considered external units to CPU
(Central Processing Unit), are now recognized as an integral part of processors.
The integration of a co-processor, in particular, can be based on direct control through co-
processor instructions, which are integrated into the processor ﬂow of instructions. Instead,
others are independent asynchronous processors, which are therefore not optimized for
general purpose code.

1 −Introduction
In this thesis, we evaluate a particular type of processor called Rocket-Chip, belonging to
the family of RISC-V processors.
RISC-V®is a free and open ISA (Instruction Set Architecture) based on reduced instruction
set computer (RISC) principles.
The goal of the RISC-V project is to make a practical ISA which is open-source, usable in
any hardware or software design without royalties. This is because of developing a CPU,
usually, requires design expertise in several specialities: electronic digital logic, compilers,
and operating systems, and since hiring experts in all these sectors has very high costs,
commercial vendors of computer designs, such as ARM Holdings and MIPS Technologies,
charge royalties for the use of their designs, patents and copyrights.
The Rocket processor easily allows the integration of accelerators without extending the
RISC-V ISA.
In fact, it contains an interface structure for accelerators, called RoCC (Rocket Custom Copro-
cessor), for which a set of custom instructions for calling co-processors is deﬁned.
In this work, we analyze the speciﬁcs of the area but above all the timing and power of the
Rocket processor in the presence or absence of a co-processor, when complex computational
algorithms must be executed.
In particular, the main goal is to design an interface that can decouple the Chisel design ﬂow
from VHDL or Verilog design ﬂow. In this way, the interface can be used by a future user to
connect any accelerator designed in VHDL or Verilog, without having to know the Chisel
and put a hand on the classes written by us.
In this thesis, we focus on a speciﬁc image processing function, which is implemented through
the use of an FIR ﬁlter (software or hardware in the case of accelerator integration).
One of the main challenges we face in this thesis, it is the implementation of an interface
environment that allows the communication of a co-processor with the core and directly with
the memory. The interface is implemented so that it can be "general purpose" and thus allows
the integration of a co-processor that is different from the one described in this thesis.
In detail, in Chapter 2, we describe the RISC-V ISA and the Rocket Architecture, paying par-
ticular attention to the pipeline structure. Moreover, we illustrate the operation, the structure
and the data transfer protocol of the RoCC interface.
In Chapter 3, we describe the RTL design ﬂow and the use of the simulation and veriﬁcation
toolchain. Then, we illustrate the design ﬂow for post synthesis simulation.
In Chapter 4, we describe how the structure of the interface is designed, also showing what
the core design parameters were and, in detail, the design of a wrapper, used to connect
the accelerator to the processor. Then, we describe the particular RTL design used as an ac-
celerator (Direct FIR ﬁlter), ﬁrst illustrating the algorithm and then the actual implementation.
Finally, in Chapter 5, we describe in detail the synthesis and validation process, analyzing
the results obtained and thus verifying the convenience or otherwise of the integration of the
accelerator chosen.

CHAPTER 2
RISC-V®Architecture
The RISC-V is an open source Hardware Instruction Set Architecture developed by the
University of California, Berkeley and many other contributors and volunteers starting in
2010. This project started with educational and training purposes, but over the years has got
the attention of industries and academics, becoming a standard for industrial implementation
[8]. RISC-V architectures follow a similar instruction set to that of MIPS because both are
based on "reduced instruction set computer" (RISC), which allows having a lower CPI (Cycles
Per Instruction) than a "Complex Instruction Set Computer" (CISC). In the next sections, we
are going to describe the elementary assembly instruction formats of this kind of architecture.
2.1

The RISC-V ISA is deﬁned as a base integer ISA, plus optional extensions to the base ISA.
The main difference compared to the previous RISC ISA is the support for optional variable-
length instruction encodings. The base is restricted to a minimal set of instructions which
satisﬁes the target for compilers, assemblers, linkers, and operating systems, providing a
software toolchain "skeleton" that helps to build more customized processor ISAs[11].

The main features of each base integer instruction are the width of the integer registers and
the corresponding size of the user address space. RV32I and RV64I are the two primary base
integer variants.
These primaries provide 32-bit or 64-bit user-level address spaces. Considering the RV32I base
integer instruction set, the register ﬁle organization provides 31 general-purpose registers
x1-x31, which hold integer values.
Register x0 is hardwired to the constant 0 and register x1 is used to save a return address on
a call.
Moreover, exists another variant for 128-bit registers, called RV128I, which supports a ﬂat
128-bit user address space. As already mentioned, the base integer ISA can be extended with
more optional instruction set extensions.
The extensions are divided in standard extensions, which do not conﬂict with other stan-
dard extensions and non-standard extensions, which can conﬂict with other standard or non
standard extensions.

2.2
Rocket Architecture
The particular RISC-V Architecture adopted in this work is the "Rocket Chip". The strong
point of this architecture is the ease of conﬁguration of the core and the blocks that make up
the system, as well as a toolchain ecosystem that allows it to be tested in a comfortable way.
Rocket Chip is Berkeley’s RISC-V based SoC (System on Chip) generator.
It is BAR1’s parameterizable chip generator which can generate an RTL RISC-V imple-
mentation that has virtual memory, a coherent multi-level cache hierarchy, IEEE-compliant
ﬂoating-point units, and all the relevant infrastructure to talk to a running system. The
generator works on the basis of the Chisel2. It leverages the Chisel hardware construction
language to compose a library of sophisticated generators for cores, caches, and interconnects
into an integrated SoC[1]. The power of this generator lies in being able to build general
purpose processor cores based around the RISC-V ISA.
There are two types of generators:
• Rocket : in-order core generator.
• BOOM : out-of-order core generator
Moreover, Rocket Chip supports the integration of custom accelerators in the form of instruc-
tion set extensions, coprocessors, or fully independent novel cores.
The basic structure of the Rocket consists of a single or multiple Tiles that contains a Rocket
Core and L1 instruction and data caches. The generator provides a Memory Management Unit

Rocket is a 6-stage single-issue in-order scalar core generator. The design minimizes the
impact of long clock-to-output delays of compiler-generated RAMs.
The RISC-V instruction set is designed for pipelined execution.
RISC-V instructions are fetched in the ﬁrst pipeline stage and decoded in the second stage
easily, as the instructions are all the same length.
The source and destination register ﬁelds are located in the same place in each instruction,
while the memory operand only appears in loads or stores. Therefore, we can calculate the
memory address in the execute stage and then access memory in the following stage.
The pipeline stages are :
1. PC : Program counter generator
2. IF: Instruction fetch
3. ID: Instruction decode and register ﬁle read
4. EX: Execution or address calculation
5. MEM: Data memory access
6. WB: Write back
Figure 2.9: Pipeline stages. Source [1]
The accelerator integration can be done in two ways, depending on how much coupled the
accelerator is to the core.
One way is attaching the accelerator directly to the core pipeline, but there is no maximum
decoupling. Moreover, the accelerator can act as a coprocessor and receive commands and
data from the processor via the RoCC interface. In this way, there is a maximum decoupling
and the accelerator can instantiate his tiles and connect coherently to the memory system
using TileLink.

The main challenge is interfacing the Rocket Chip, which is written in Chisel language, with
any accelerator written in VHDL, without having to go through the design ﬂow of the RoCC
interface each time.
Explaining better, we want, starting from a Chisel class, to generate a Verilog ﬁle that allows
a user who does not know the Chisel, connecting an accelerator written in another hardware
description language.
The main problem is that with the Rocket-Chip generator we have a translation from Chisel
into a Verilog that is not easily readable by a human being.
In fact, usually the size of this Verilog ﬁle exceeds 16 mb and 250000 lines of code.
Therefore, what we need is a module that manages the typical protocols of the Rocket-Chip
and translates them appropriately for the accelerator.
This module is commonly called Wrapper.
The wrapper must ﬁrst and foremost decouple the accelerator from the way the RoCC picks
up data and instructions from memory and processor. Since usually the time in which the
RoCC communicates with memory is different from the time it takes for the accelerator to
request and process data, we need a synchronization element between accelerator and RoCC.
For this purpose, we have chosen to insert a couple of FIFO between RoCC interface and
accelerator, so that the accelerator requires data when it is needed, regardless of how the
RoCC takes them from memory.
In this way, the RoCC interface will take care of completely managing the communication
with the processor and the memories, and will also control the start and stop of the accelerator.
In Figure 4.3, we show the various signals which connect the modules.
As can be seen, we need to deﬁne a Controller which manages the communication with the L1
Cache and the Rocket Core.
Core communication signals are those that have been shown previously.
In particular :
• RED : RoCC request (CMD) signals, including the two data buses rs1 and rs2 with
which the core directly supplies data or a memory address from which to take data.
• BLUE : RoCC response (RESP) signals, which include the RESP DATA bus, with which
the RoCC returns data to the processor.
• GREEN : Memory Request (MEM REQ) signals, which have been explained previously.
In particular, if the application requires that the data processed by the accelerator be
saved in memory, the output of the FIFO OUT must be connected to the request data
bus (MEM REQ DATA) in the event of a store. This link is represented by the dotted
line that connects FIFO OUT DEQ DATA with MEM REQ DATA.
• BLACK : Memory Response (MEM RESP) signals, including the MEM RESP DATA,
which is the bus with which the memory returns data to the RoCC. Usually, data from
the memory must be loaded into the accelerator. For this reason, the memory output
must be connected with the input of the FIFO IN. This link is represented by the dotted
line that connects MEM RESP DATA with FIFO IN ENQ DATA.

To manage all these operations, we need to deﬁne a Finite State Machine for the controller.
All the controller and FIFOs were implemented in Chisel language in order to exploit the
protocols of the RoCC interface.
The state machine is shown in the Figure 4.11.
In particular :
• IDLE : When the accelerator is not called, the controller is in the IDLE state.
In this state all the memory requests and communication signals with the FIFOs and
the accelerator are forced to ’0’.
When the processor calls the accelerator with the signal CMD VALID, the state changes
to the value DECISION.
• DECISION : The controller decides whether to load the FIFO in with the data taken
from the memory or if to save the data from the FIFO out to the memory. In particular,
the decision is made by verifying that the FIFO out is or is not full. Because if the FIFO
out is not full, it means that it is still possible to save results on it, and therefore it is
possible to take other data to be processed from memory.
Note well! The FIFO size can be parameterized based on the particular application. You
have to make sure that the size of the FIFO is always smaller than the number of results
that have to come out. Otherwise you would never reach the "full FIFO" situation to
move from the load to the store.
Therefore, if the FIFO out is not full, the state after becomes LOAD FIFO IN.
On the other hand, if the FIFO out is full, the state changes to the value STORE FIFO
OUT.
In the state STORE FIFO OUT there is a counter which increases for each request
made. The counter is used to increase the address in which to save the data correctly.

4.2.2
FIR Filter for 2D-Convolution
For image processing, we need to perform a bi-dimensional convolution in the time-domain [2].
With the bi-dimensional convolution, one can perform averaging, smoothing and sharpening of
a digital image, which is represented by a pixels matrix B(n × n).
The convolution is performed between this matrix with a speciﬁc mask W(m × m). Consider-
ing m=3, a speciﬁc pixel B
′[x, y] can be computed as :
B

i=−1 ∑1
j=−1 B[x + i, y + j] · W[i, j]
In the beginning, coefﬁcients are loaded in the shift register A. Pixels are fed line by line, from
top to bottom. To perform the ﬁrst result, the algorithm needs to store two complete lines
and three pixels of a third line in the shift register B. From this moment on, each new pixel
inserted in the shift register moves the convolution window one position.
Because B can store up to 64 pixels, we have to cut the image in vertical bands 30 pixel-wide
and process a vertical band at the time.

If we want to execute this algorithm on the ﬁlter described in the previous subsection, we
must take into account some things.
The image is represented in grayscale. Consequently, the inputs and outputs of the ﬁlter are
values from 0 to 256 ( 8 bits).
Therefore, to have the appropriate result it is necessary to make a truncation on the output.
In particular, only the 8 most signiﬁcant bits of the 16 bits of the result must be considered.
As we can see in Appendix B, when the controller saves the data from the FIFO out to memory,
it takes just the 8 most signiﬁcant bits of the result.
As we can see in Figure 4.16, the accelerator instead has an 18-bit output.
In this way, in fact, the calculation is not wrong in the case of saturation.
Saturation, in particular, occurs when the result coming out of the tree is greater than 256.
In this case, the result, if we consider the 16 bit output, would be wrong. And so instead of
having white, we would have gray or black.
The two extra bits therefore serve to indicate when there is overﬂow and therefore we ﬁnd
the saturation condition.

creases the n index and the SEL mux terminal.
So, the X shift register shifts one position by iteration.
Since the accelerator has a latency of 2 clock cycles from the moment it receives the
sample to the moment in which it brings out the corresponding result, when the SEL
mux terminal is 2, can start saving results on FIFO out.
Therefore the state changes to WRITE RESULT.
• WRITE RESULT : Here, the FIFO OUT ENQ VALID signal is asserted.
Then, the state changes to the LOAD LAST SAMPLE.
• LOAD LAST SAMPLE : Here, the ﬁnite state machine iterates the data loading process,
and the saving of results on the FIFO out until the SEL mux terminal is "7".
==PAGE:63==
§4.3 −FIR Accelerator Integration
55
At this point, the state changes to the LOAD LAST RESULT value.
• LOAD LAST RESULT : Here, the FSM blocks loading data on the X shift register,
forcing to "0" the LOAD SAMP signal.
However, since there are still results to be saved (due to the latency), the FIFO OUT
ENQ VALID signal remains asserted.
Also in this case, when the SEL mux terminal has reached the value "7", the FSM needs
to load another 64 bits word, so the state return to SETUP.
As long as there are samples to load, the cycle just illustrated repeats. Otherwise, when the n
index has reached the value "n samples + 64" (+64 is the number of coefﬁcients), the state
returns to the ST0 value, where it waits for another call.

Moreover, it is to be considered that, with the particular interface realized, we can directly

take 64 bits from memory every two clock cycles.
Before integrating an accelerator with this interface, an analysis must be made of what is the
bottleneck in the execution of a given algorithm. In fact, if the bottleneck turns out to be data
processing, as in the case under consideration, this interface brings great advantages. How-
ever, if the bottleneck is communication with the memory, and for a particular application,
it needs much more than 64 bits every two clock cycles, this interfacing could be no longer
advantageous.
At the beginning of the work, we had to analyze the various existing RISC-V architectures,
and we had to choose one to work with.
We have chosen the Rocket-Chip architecture as it is the one of most signiﬁcant relevance and
allows extensive freedom in conﬁguring the processor and other parts that make up the chip.
The main challenge was to be able to interface the accelerator with the Rocket, in a way that
we can respect all communication protocols with the processor itself and the memories.
The ways to follow were different, as the Rocket-Chip provides two ways to interface acceler-
ators with the processor, and these are :
• MMIO Peripheral (a.k.a TileLink-Attached Accelerator)
• Tightly-Coupled RoCC Accelerator
In this thesis, we decided to follow the second way, as the RoCC interface allows a greater
decoupling between external accelerators and processor, and allows easy extension of the
ISA to include custom instructions.
The use of the RoCC interface requires that a designer knows the Chisel language, because,
according to the method proposed by the Rocket Chip manuals, if we want to interface an
accelerator to the Rocket chip through RoCC, we have to design the accelerator in Chisel,
since the Verilog generated from Chisel is not easily readable.
So we set ourselves the goal of designing an interface that could be used by a future user to
connect any accelerator designed in VHDL or Verilog, without having to know the Chisel
and put a hand on the classes written by us.
For this reason, we have developed a wrapper that allows to decouple the design ﬂow of the
interface in Chisel with that of an accelerator in VHDL or Verilog.
As we have seen, it is enough to take the generated Verilog from our Chisel classes, and
connect the accelerators through a simple mapping in the module that deﬁnes the wrapper.
With the help of two FIFOs, the wrapper also allows to have two different rates between the
requests of the data, and the processing by the accelerator.
In this way, it is possible to load all the data to be processed much earlier than the accelerator
requests. On the other hand, the wrapper can stop the accelerator if we can’t provide enough
data to start processing.
In our case, while the accelerator is processing old data, the wrapper has already loaded
many new data on the FIFO input.
This is possible since the FIR accelerator requires an 8-bit data at each clock cycle, while the
RoCC interface allows 64 bits to be taken from the memory every two clock cycles.
Moreover, the Rocket-Chip repository gives us all the information needed to generate and
test the chip on a behavioral level. However, our goal was to synthesize the whole system
==PAGE:79==
71
according to a particular standard cell library and then simulate it. So, we followed another
design ﬂow, synthesizing and simulating the whole system according to a particular 28 nm
standard cell library.
With the synthesized system, we could therefore have more realistic results and analyze the
physical characteristics of the device.
In particular, we have chosen to run the chip and the accelerator at a frequency of 500 Mhz.
As we have seen, however, you can run the Rocket Chip at a maximum clock frequency of
1.3 Ghz.

6 −Conclusions and Future Developments
6.1
Future Developments
Possible future developments for this work may be different. For example, it has been said
that we have chosen a Rocket-Chip type architecture. However, there are many other types
of architecture on which one could do the same job. Just consult page 1.
Continuing to work instead on the Rocket-Chip, you could decide to use the other way
of interfacing with the accelerators, the previously mentioned MMIO Peripheral (a.k.a
TileLink-Attached Accelerator).
With this, you could try to see if you can establish a communication with the memory that
can provide more than 64 bits every two clock cycles, and that, therefore, leads to even more
signiﬁcant advantages in terms of speed and power savings.
Furthermore, you could try to design a DMA port for direct communication of peripherals
and accelerators with memory. This surely would bring signiﬁcant advantages in terms of
speed.
