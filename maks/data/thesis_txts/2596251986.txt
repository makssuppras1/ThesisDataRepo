Abstract
The use of modular control architectures are broadly used in robotics, and by all appear-
ances also in nature, in that the brain utilizes some sort of modular architecture when
sending signal to muscles. In the attempt to replicate the working mechanism of the
human brain in regards of control, much research has been made in the field of neural
networks. This thesis’ deals with an attempt to advance the research performed not only
within neural networks, but neural networks in the combination with a modular control ar-
chitecture, specifically when learning the inverse model of a system. A modular neural
feed forward controller is designed, implemented in a simulation environment and tested
against other implementations in the same or similar fields.

1
Introduction
The human brain is capable of fast and adaptive control of the human body. Picking up
and moving a cup of unknown contents, catching an object flying through the air or doing
repetitive operations in an ever changing environment, all seems like effortless tasks for
a human. Though it is effortless, it does not come out of nowhere. In fact the skills are
developed through childhood by rewarding or punished interactions with the environment.
But not all situations can ever be experienced by one person, yet when exposed to a new
situation, adaptation can happen often quite fast. Theories exist, that humans or animals
in general, does not learn to handle or control a task specifically, rather we generate
a set of fundamental methods or primitives, which can be combined at will to adapt and
complete a new task. This can be modelled as a modular control architecture, where each
module is in charge of a fundamental method or primitive. Work has been done to mimic
this behavior, not only in regards to the modular architecture, but also how the brain is
constructed of neurons communicating with each other. Specifically much work has been
done using the concepts to learn the inverse models of various systems and use them as
feed forward controllers. The motivation of this thesis is to add to the research done in
this field of neural networks combined with modular control architecture for inverse model
learning, to see if any improvements can be made.
1.1
State of the art
One of the works mentioned, is [1], in which experiments are perform on frogs to investi-
gate what kin of signals the frogs brain uses for control. It is found that the signals in fact
consist of a super position of multiple individual signals, which refers to the mentioned
modular control architecture. It stands to reason, to suggest that the human brain oper-
ates in a similar fashion. In [2] the concept of modular architecture with Artificial Neural
Network (ANN) is presented. It uses neural networks to learn the inverse model, which
acts an adaptive local expert, or a module in the architecture. As suggested what exist
in a brain, they also use neural networks as decision functions, to decide what expert is
responsible at what time. [3] is another example of modular control architecture, which
learns several inverse models. For each inverse model a forward model of the system
is also learned, and this forward model precision is used as an indicator for the perfor-
mance of the corresponding inverse model. The work done with Modular Selection and
Identification for Control (MOSAIC), was a part of the inspiration of work done in [4]. Mod-
ular Architecture with control primitives (Macop) is uniquely different in its way of deciding
which controller should control. The principle is that at all time, a controller should not be
selected, but weighed. It is achieved by training the controller selection to make the ar-
chitecture achieve certain properties. Macop proposed in their future work, that it should
be investigated whether the principles of Macop could be useful in a inverse dynamical
model learning task, as opposed to the learning of the inverse kinematic model. This
investigation is what started this thesis.
1.2
Purpose
The purpose of this thesis was initially to use the principles of Macop, to design and
implement a feed-forward controller in the shape of an inverse dynamical model. For the
purpose of getting familiar with the concepts of Macop, the thought was to first replicate
the work already done with Macop, which deals with the learning of the inverse kinematics
of a system, and then move on to learning a inverse dynamical model. The main problem

with Macop, is that it is very time consuming and complicated to implement, and at some
point the remaining time of this project became an issue, and the purpose was changed.
Instead of working with inverse dynamical model learning, the purpose was changed to
designing and implementing a new concept for learning an inverse kinematic model. The
new concept should contain as much of the working concepts and advantages of Macop,
while gaining a simpler implementation cost with regards to time and complexity. This
leads to the main objectives of the thesis.
• To design and implement a modular neural feed-forward controller using inverse
kinematic model learning.
• To design the architecture to be simpler and less time-consuming than the design
of Macop.
• To design the controllers to be able to control a system to follow a trajectory with a
fixed control rate.
• To design the controllers to be able to compensate for any dynamical behavior
caused by the fixed control rate.
• To retain the advantage of using multiple controllers compared to a single controller,
which was found to be obtained with Macop.
• To obtain a broadly applicable solution, for any system to be controlled where a
inverse kinematic model is suitable.

Methods
2.1
Macop
Macop is as mentioned designed for finding the inverse kinematics of a robot manipulator,
which is the mapping from the position of an end-effector of a robot manipulator in carte-
sian space, to the joint angles. Besides the inverse kinematics, Macop is also designed
for trajectory control with a fixed control rate. This project has mainly been about utilizing
the concepts of Macop and combining them into a simpler implementation. In this section
the theory about these concepts will be explained.
2.1.1
General control architecture
In figure 2.1, the control architecture of Macop is shown. The structure consist of Nc con-
trollers, all receiving the same inputs, and a decision or responsibility function, which mix
the output of each controller into a total joint command signal which is send to the robot.
The main concepts of this architecture is the modularity of the architecture, what a single
controller consist of, and the uniqueness of the responsibility function compared to others
in the field of modular control architectures. Though these are the most important fea-
tures of the Macop, there are a few details of the figure which can’t go unexplained. As
mentioned, Macop is designed for learning the inverse kinematics of robot manipulator,
but it does in fact not purely learn inverse kinematics nor does it fully utilize the inverse
kinematics.
Figure 2.1: A figure showing the overall modular control architecture of Macop
As seen in figure 2.1, the current end-effector position and the desired future end-effector
position is fed into the controllers. The reason for feeding the current end-effector po-
sition to the controllers, is that the desire of the implementation is to be able to control
the trajectory of the end-effector by a fixed control rate. This means that after giving a
end-effector position command, the robot is always given the same time to reach the the
desired position, no matter what position is given. Let’s say the robot manipulator is in
a given point in a world frame, P1 = (x1, y1, z1), and we have two different possibilities
for the future desired position, P2 = (x2, y2, z2) and P3 = (x3, y3, z3). Let’s assume that
Implementation of a neural feed-forward controller based on inverse model learning

==PAGE:14==
the two desired future positions are different, and let’s assume that the distance between
P1 and P2 is different from the distance between P1 and P3. This means that the internal
joint position controllers of the robot manipulator has to be able to handle two different
distances in the same amount of time. This dynamical behavior that needs to be handled
is why the implementation is not purely in the inverse kinematics setting. The solution is,
as described in figure 2.1, by feeding the current end-effector position together with the
desired future end-effector position. This in turn results in the fact that the joint angles
produced by the controllers, and also the total joint angle command, is in fact not the joint
angles of the inverse kinematics which corresponds to the position desired, but the joint
angles commands that will effectively make the robot end up with the correct joint angles
at the right time, which then are the joint angles which corresponds to the inverse kine-
matics of the robot.
The inverse kinematics of a robot, is as mentioned the mapping from end-effector position
to the joint angles, but when working with a robot manipulator, the position is not always
the only thing that matters, you may also want to know what direction the end-effector is
pointing in, meaning the orientation of the end-effector. In the original Macop concept,
the orientation is not controlled, but for the new design and implementation explained in
section 2.2, the orientation will be included.

An ESN is a special case of a Recurrent Neural Network (RNN), which fall in the category
reservoir computing. A RNN differs from the other neural networks like a general feed-
forward neural network, which consist a number of hidden layers, by how the neurons are
connected. If a feed-forward neural network have 2 hidden layers, then the neurons in
the second hidden layer will depend on the values of the neurons in the first hidden layer,
yet not the other way around. In the hidden layer of a RNN, all neurons are connected
to each other, and therefore depend on each other. This recurrent nature makes the net-
work capable of retaining memory of the input, which makes a RNN proper when trying
to estimate time series or time dependant functionality in general. Learning the inverse
kinematics of a robot manipulator is not inherently a time dependant task, but as men-
tioned in section 2.1.1, Macop expands the inverse kinematics into a fixed rate trajectory
control task which includes some time dependant dynamics.
As mentioned, a RNN consist of neurons and connections between them as all other neu-
ral networks. The neurons are defined by their activation or what value they represent.
The collection of the neurons in a network are kept in a vector, that is one vector for each
layer. In the used ESN method, only one hidden layer is used, meaning only 3 vectors
is used, the input, output and the network. The connections between the neurons repre-
sent the influence one neuron has on the other, and the collective influence one vector of
neurons has on another, like the input neurons’ effect or weighted effect on the network
neurons, can be represented by a weight matrix. The weight matrices are represented by
W p
k , being the influence or weight of the neurons layer k on the neurons in layer p. In this
formulation, the inputs and outputs are considered layers. Besides the inputs, outputs
and the recurrent layer, there is also a bias influence, which is represented by a constant
1, and influences the recurrent layer through a bias weight. In figure 2.3, the neurons are
represented by circles and the weights between neurons are represented by the arrows.
What makes ESN different from the general use of RNN, is the way that the weights are
initialized and treated over time. All weights in the ESN are initialized random, by drawing
the matrix elements from the normal distribution, and then scaling them by a factor to en-
hance or adjust how they each perform. Also all the weights, except the output weights,
are kept constant at all times. The idea is, that by randomly initializing the input, bias and
network weights, a sufficient reservoir of functionality is created for the task at hand, to
be linearly combined by the trained output weights to effectively achieve the goal of the
task, which in this project is to generate the desired angles for a desired position.

Proposed design
As mentioned earlier, the idea of replicating the Macop concept and utilizing it for inverse
dynamics model learning had to be given up. Even though the replication of the combined
working elements wasn’t achieved, the knowledge of the individual parts was gained.
Therefore the project wasn’t lost, just changed. Instead of aiming for inverse dynamic
model learning using the Macop concepts, a simpler version of the of the inverse kinematic
model learning has been implemented. Also, as mentioned earlier, even though Macop
learns the inverse model, it does not utilize the orientation of the robot manipulator. This
has also been a focus in the new design, though less of a concern compared to position
control precision. This section will be about the changes from the original Macop, which
is described in 2.1, to the proposed solution.

introduced by [4]. A optimal tuning should exist for any system to be controlled, but be-

regards just that initialization. A random seed, could be keep constant and then all tuning
could be done optimally. But that would maybe not results in the best possible solution,

are tuned to achieve.

Webots simulation environment
Throughout this project, the Webots simulation environment has only been investigated to

you like either through a predefined model of the object or something specific you create

is where, for an example a robots dimensions, actuators and sensors are given. Webots
offers a lot of parameters to adjust the simulation in any way you want, but throughout this
project, only one parameter has been important, which is the simulation environment time
step, called basicTimeStep. This poses a few constraints on the controller design pro-
cess. The basicTimeStep, decides how often the numerical model, that is the whole world
of objects, is being calculated, that is the positions, velocities, forces etc. The simulation
world, is interacted with through a controller scripts. Through the controller script, the
simulation is started, sensor values are read, commands are send to actuator etc. This
is done by execution a Webots controller library command: robot.step(TIME_STEP).
This will run the simulation for the amount specified by the parameter called TIME_STEP.
TIME_STEP and basicTimeStep has to be in sync, else the amounts become unreliable.
If, for example, basicTimeStep is set to 20 ms, then TIME_STEP can be set to a multiple of
20 ms, like 40 ms, 80 ms etc. In all experiments with the simulation model, the parameter
TIME_STEP is used as the control period and sample period, and attempts has been made
during this project, to determine a proper value for this parameter, which will be discussed

3.4
Selection of sample time/control rate
Usually in control theory a sampling frequency and control rate is decided based on the
system to be control. A general rule, is to follows the Nyquist theorem. It is generally
applied for digital signal processing, and simply states; to be able to recreate a signal,
which contains a maximum frequency of f, that has been sampled using a frequency of
fs, the sampling frequency has to be at least twice as fast as the fastest frequency in the
signal.
fs ≥2 · f
(3.9)
Generally when controlling a system, the sampling frequency will be selection somewhat
higher than just 2 times higher than the biggest sampled frequency. When working in a
kinematic setting, the sampling frequency becomes less important. All disturbances to

kinematics don’t have to handle any. The sampling frequency becomes important in the

This of course depends on the PID controllers which are internal. Another dependency
on the internalPID controllers, is if they have enough time to react at each command. This
puts a lower boundary on the sampling frequency, or a higher boundary on the amount of
change in joint angles that is required.

achieved by utilizing increasingly more controllers. In this thesis, the best performance
was achieved with Nc = 2 and in [4] the best performance was achieved with Nc = 6,
both with a fixed number of neurons of 250. They do however say that in some of the
experiments done with Macop, it ended in failure, due to the robot getting stuck. One
of the main properties of the proposed solution, compared to Macop, is that the inverse
kinematics is constraints to a single solution. This is achieved by the use of the forward
kinematics which is fed angles which are known to be within the allowed limits. In Macop,
the solution is not constrained to a single solution, because the angles aren’t provided
by the forward kinematics, but by using the single controller two models setup. Though
using the forward kinematics prevents a inverse kinematics solution which could depend
on angles not available, it does heavily punish the solution compared to Macop, in that
the angles does in fact have to be provided. In reality, the more convenient solution would
be to only need the trajectory given in end-effector position and orientation. An argument
could be that if the angles are already known, what is the need for the inverse kinematics,
especially if the solution only works properly on and tightly around the trained trajectory.
An argument is of course that knowing the joint configuration, won’t in it self compensate
for the fixed control rate.
In [6], the results are even better, but it is also without the concept of fixed control rate,
and also with the use of ANN and not RNN networks, and the used networks are trained
offline. The reason for comparing to [6], is that their proposed solution also utilize the
current measured joint angles as input to the inverse kinematics. In the proposed solution
the measured joint angles as input to the controllers, was mainly implemented for a new
suggestion for the compensation for the fixed control rate, while in [6], it is done purely for
improved performance.

6
Conclusion
In conclusion, the objectives of the implementation were completed. The goal of design-
ing and implementing a modular feed-forward controller which utilizes inverse kinematic
model learning was achieved. But by comparing to other solutions in the same field, it per-
forms somewhat worse. The control architecture is able to follow a trajectory with a fixed
control rate. The performance decreases significantly when deviating from the trained tra-
jectory. This means that the inverse kinematic model is only learned in the space defined
by the joint trajectory and the joint space closely related that of the training. Compared to
the original implementation of Macop, the proposed implementation was significantly sim-
pler to implement. Though the control architecture is simple to implement, it is a complex
task to tune the controller for a change in trajectory. In theory, the implementation should
be able to be used in any system where an inverse kinematic model could be useful,
though the change of system would only add to the complexity of tuning the controller for
the learning task. The advantage of using multiple controllers compared to using a single
controller, was partly achieved. A comparison was made with Macop, which achieved
increasingly better performance when using an increasing number of controllers up to 6
controllers, while in this implementation, an increase in performance was only achieved
when increasing the number of controllers from 1 to 2. All in all the objectives of this thesis
were to some degree fulfilled, though many of the underlying expectations were not met.
6.1
Future work
A suggestion to future work is to try variations for the input of the controller. In this imple-
mentation the current measured joint angles, the desired future position in x, y, z, and the
desired future orientation is fed into the controller as inputs.
The idea of using the desired orientation of the end-effector as an input to the controllers,
was that the controllers would have more relevant information to base the calculations of
the outputs on. But if the model can be learned without the use of desired orientation as
input, which it is shown to be able to in [4], maybe the orientation is redundant and con-
fusing for the recurrent networks. An idea could be to rid of the orientation, and changing
the the information used as input from the current joint angles, to the current position.
Basically, to revert further back to the principle of Macop. It could be interesting to see if
the change would improve the performance to the same level of Macop.
Although the implementation seems to be able, at least to some extend, to compensate for
the fixed control rate, it would be interesting to remove this aspect of the design and turn it
into a purely inverse kinematic learning task. Without the fixed control rate, a different type
of neural network might work better. A suggestion for future work would be to investigate
alternative online learning algorithms, and if any other would be better suited for this type
of implementation. A big part of the proposed solution is the use of the responsibility
function presented in [4], to see if the advantages gained by using it could be retained in
a simpler implementation. In Macop, the responsibility function has to be trained online
together with the controllers. The believe is that this is done because it depends on the
joint angle configuration of the robot, which can’t be known beforehand. In the proposed
solution, both the desired position and joint angles are known due to the use of the forward
kinematics. Another suggestion to future work, could be to train the responsibility function
before starting the training of the controllers and moving the robot. This could ensure that
the properties wanted from the responsibility function are achieved in all experiments
