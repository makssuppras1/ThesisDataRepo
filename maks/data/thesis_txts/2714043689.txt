Abstract
The status quo of software applications is in a constant evolution due to emerg­
ing of new technologies, performance improvements, and new business require­
ments. The classical way of developing applications is by implementing a mono­
lithic architecture. However, this type of architecture has some drawbacks as the
software is built as a single indivisible unit where the application’s components
are coupled.
In the past years a new architectural style named microservice architecture emerged
which takes a different approach to develop application characterised by a suite of
small services (microservices) each running in its own process being decoupled
from the other application’s components.
Nevertheless, implementing a microservice architecture is not trivial and it also
comes with several downsides such as a higher complexity of configuring and de­
veloping the services, proper componentisation for supporting a single business
capability, and implementing the appropriate related software design patterns.
Additionally, it requires writing of boilerplate code templates to configure the com­
munication with the services and their deployment.
State of the art research tries to address these issues by providing domain­specific
languages than enable users to specify and generate microservice applications
without having to concern about an appropriate split of the microservices and the
need of manually configuring them. However, these solutions do not provide a
tool for specifying and generating a microservice­based application similar to any
other application available on the internet, but rather only the microservice suite
(server­side) and not a client­side. In addition to this, the services are not de­
ployed to any containerisation engine which helps isolating services further.
This dissertation proposes MaGiC, a DSL framework for implementing language
agnostic microservice­based web applications. The framework can be used to
specify a software application which consists of a microservice suite, client­side,
and a gateway layer that acts a communication facilitator between the microser­
vice suite and client­side. Furthermore, the framework aims on easing the imple­
mentation, building and deployment process of microservice­based applications
by eliminating the need of manually configuring the microservice suite and the
containerisation engine.
After having implemented the framework a validation process was conducted which
was characterised by implementing and deploying a fully­functional microservice­
based web application. Moreover, in order to determine whether or not the MaGiC
framework simplifies the process of developing a microservice­based application
a couple of software development metrics (lines of code, cycle time) were ob­
served.

Introduction
Software applications are constantly evolving due to business requirements, per­
formance improvements, and emerging of new technologies. The traditional way
of building applications is by implementing a monolithic architecture which is char­
acterised by a single and indivisible unit, which usually consists of a client­side
user interface, a server­side application, and a database, where everything is uni­
fied and served in one place. The downsides of such an architecture is scalability
as components of the application cannot be scaled individually, rather the applica­
tion as a whole needs to be scaled. In addition to this, as the application evolves
its complexity increases on a non­linear basis and becomes hard to manage by
the engineering teams [1].
Due to these downsides a new architectural style started to be adopted by busi­
nesses in the past years called microservice architecture which is part of the
broader area of Service Oriented Architecture (SOA). Lewis and Fowler [1] de­
fine the microservice architecture paradigm as an approach for developing an
”application as a suite of small services, each running in its own process and com­
municating with lightweight mechanisms” [1]. Division of the application in a suite
of services allows for implementing loosely coupled components which can be
scaled and deployed independently. Furthermore, it allows for development flex­
ibility as each service can be owned and maintained by different teams without
any particular dependencies, and the services are technology agnostic, meaning
that each team can implement each service using different technologies and pro­
gramming languages [2] [3].
Nevertheless, implementing a microservice architecture properly is not a trivial
task as the componentisation of the software application is highly dependent on
how well the software can be split in components and in the same time it is difficult
to define their context boundaries. Additionally, in a complex microservice­based
application with a big number of services where they communicate with one an­
other the dependencies between them might become an overhead and potentially
a source of errors.
Furthermore, the team skill factor has to be taken into account since it is direct
proportional with how well the microservice architecture is implemented and if
not implemented properly the architecture might create more dependencies com­
pared to a monolith architecture. With other words, according to Martin Fowler, ”a
poor team will always create a poor system” [1].
Lastly, configuring the microservice architecture it is a complex, cumbersome, and
time consuming task requiring writing of boilerplate code [1] [4].
State of the art research, MicroBuilder [4] and AjiL [5], try to address the above
highlighted issues by providing domain­specific languages (DSLs) that enables
users to specify and generate microservice based applications with ease. How­
ever, the DSLs allow for describing only the microservice suite rather than a com­
plete application which consists of a server side / microservice suite, a user inter­

With other words, the generated application is not an application which can be

Concepts and Background
The following chapter is dedicated to exploring the main tools and concepts that
are used throughout this project for achieving the initially set goal.
2.1
Domain­Specific Language
A domain­specific language (DSL) is a programming language or a specification
language which offers through proper notations and abstractions an expressive
capacity focused on, and typically restricted to, a certain problem domain. Many
computer programming languages are domain­specific rather than general pur­
pose, and the main difference between a DSL and a general­purpose program­
ming language (GPL) is that DSLs, as their name asserts, trade generality for
expressiveness in a limited, specific domain.
As the DSLs are tailored for a particular application domain, through notations and
constructs they offer substantial gains for expressiveness and ease of use com­
pared to GPLs. Additionally, the productivity and operation costs are reduced as
well as the amount of domain and programming expertise needed, making it ac­
cessible for a larger software developer audience [6].
Domain­specific languages are not a new concept as they date back as far as
1957 when APT, a DSL for programming numerically controlled machine tools
was developed.
The main reason DSLs were designed to begin with, is simply due to the fact they
offer a more robust domain­specificity. Corresponding domain­specific notations
are beyond what GPLs can usually offer, and they are of high importance as they
are directly related to the productivity improvements the language offers. Appro­
priate domain­specific constructs are by design in the language, thus allowing for
implementing several notions of the particular domain in a more expressive way
compared to general­purpose programming languages. In addition to this, DSLs
enable reuse of software documents such as source code, software designs, and
lastly domain abstractions, as the application generators use the DSL as the input
language thus allowing programmers to reuse the semantic notions implemented
in the DSL without needing to do their own in­depth domain analysis [6].
DSL development is not trivial and it is not always obvious that a DSL will be
helpful or that establishing a new one would be profitable. The literature [6] pro­
vides some general guidelines about when and how to design a DSL, and splits
the development phases into several stages: decision, analysis, design, and im­
plementation.
Deciding to develop a new DSL is not easy and evident of its necessity, and the
investment in the development of the language must eventually compensate for
itself through more cost­effective application development and/or maintenance.
Nevertheless, interrelated concerns about whether or not to design a DSL are:

==PAGE:15==
cost­effective development and maintenance of the software; allowing individuals
with less domain and programming knowledge or even end­users with some do­
main but little programming competence to build an application.
The analysis phase is characterised by identifying the problem domain and do­
main knowledge which is accumulated from a variety of sources such as technical
artefacts, domain experts, existing implementation in a GPL, and customer sur­
veys. The domain analysis produces a variety of results, but the most common
ones are a domain definition that specifies domain’s scope, domain­specific termi­
nology and semantics in a more or less abstract form which are to be incorporated
in the lexicon of the DSL.
The approach for designing a DSL is defined along two dimensions, the formal
character of the design description, and respectively the link between the DSL
and other existing languages. According to the literature [6], the most accessible
way to design a DSL is to base it on an already existing language as the imple­
mentation might be easier, and provides familiarity but only for the users of the
already existing language. Another option is to extend an already existing lan­
guage with new elements to support particular domain concepts, and the benefit
of this approach is that the features of the initial language remain available, how­
ever it is hard to seamlessly integrate domain­specific features with the rest of the
language. Lastly, language designers are able to choose to design a DSL which
bears no relationship with any existing language, nonetheless it comes at great
efforts.
When implementing a DSL there are several approaches from which the most
suitable one should be chosen and these are: (1) interpreter, a fetch­decode­
execute cycle is used to detect and interpret the DSL constructs; (2) compiler /
application generator, the DSL constructs are translated to base language con­
structs and library calls by the DSL compiler also called application generator;
(3) preprocessor, the DSL constructs are transformed into constructs of an ex­
isting language; (4) embedding, the DSL constructs are integrated in an existing
general­purpose programming language by specifying new abstract data types
and operations ; (5) extensible compiler / interpreter, the compiler of a GPL is ex­
tended with domain­specific code generation; (6) commercial off­the­shelf, using
tools and notations and applying them to a specific domain; (7) hybrid, a combi­
nation of the previously mentioned approaches [6].
Generally, as illustrated in the section, developing a DSL is hard as it requires
both domain knowledge and expertise in developing languages. Nevertheless,
there are several software tools providing a language development system which
can be used for the implementation of the language. The types of these tools
vary from a consistency checker and interpreter to a complete application in the
form of an integrated development environment (IDE), sometimes called meta
programming system (MPS), consisting of a syntax­based editor, analysis tools,
application generator, and even a debugger. The main features of these IDEs
are the metalanguages they support and the meta­environment provided to help
in constructing the desired DSL with ease. Therefore, domain experts without
knowledge of developing GPLs can use a metaprogramming system to design
and implement a DSL with ease [6].

2.2
Metaprogramming System
The following section describes the concept of metaprogramming and metapro­
gramming systems. It important to note that there are several metraprogramming
systems available however in the scope of this project only Jetbrains MPS 1 will
be described.
Metaprogramming is a programming method that allows computer programs to
treat other programs as data. With other words, a metaprogram reads another
program, manipulates it, and returns a modified program. One example of such a
program is Jetbrains’ MPS (Meta Programming System) and it is widely used for
developing domain­specific languages. It provides an IDE with a complete set of
tools which allow for the implementation of a DSL with ease as developers do not
have to concern with elements such as parsers or compilers, but only about the
actual language design.
Jerbrains’ MPS is a language workbench and supports the design and implemen­
tation of DSLs. Languages written in MPS are projected rather than textually rep­
resented, known as projectional editing, implying that instead of entering plain text
to program a language, the user of the language workbench selects from a list of
concepts that are valid in their respective contexts. As a result, the source code
is not textual, but rather comprises of graphical features such as graphs, tables,
and formulas, which might be basic text or more sophisticated aspects.
This type of programming might appear limiting initially and the learning curve is
steep, however there are some great advantages such as it is virtually impossible
to write syntactically incorrect code (e.g. there are no errors caused by missing a
parenthesis or a semicolon). Furthermore, the code structure is strictly tree­like,
particularly an abstract syntax tree (AST). An AST is tree representation of the
abstract syntactic structure of text written in a formal language and the benefits of
the AST structure are fast data­flow analysis, inherent AST transformation, and
easy language embedding. [7] [8] [9].
A DSL developed in MPS is formed of a multitude of entities that represent dif­
ferent aspects of the language. The entities can be split in two categories, the
semantical category and structural.
The semantical category consists of the project itself which is split in: languages,
representing the concepts of the program; solutions, used for writing the actual
program based on the language concepts; and devKits which are groups of lan­
guages that can be used to refer to a number of different entities.
When it comes to the structural category, projects in MPS are called repositories,
which in turn can have multiple semantical modules, namely solutions, languages,
or devKits [8].
It is worth mentioning that domain­specific languages developed in MPS are usu­
ally formed of multiple language modules. In turn, some languages depend on
other concepts of other languages or extend them. Therefore, the DSL is split

in several modules making it easier to maintain the language and have a better
overall overview of its different aspects. To conclude, MPS focuses on enabling

tools which make the implementation phase of the language easier [8].
In MPS, a DSL is formed out of concepts and each concept is further divided
into aspects such as structure, editor, behaviour, generator, and others which are
out of the scope of this project.
• The structure aspect is used to model the abstract syntax of the concept. A
concept is represented by: properties, which are a primitive type (e.g. string,
boolean); children, representing instances of other concepts from the same
language or different one; references, which, as their name asserts, are
used to refer to nodes of the AST [8].
• The behaviour aspect is used for encapsulating functionality related to the
concept using object­oriented programming (OOP) principles. The methods
defined in the behaviour of a concept can be called on an already initialised
node of the AST and take different decisions, such as imposing constraints
on whether or not the user DSL is allowed to use different concept refer­
ences. It can be stated that it allows for implementing custom behaviour
depending on current state of the program implemented using the DSL [8].
• The editor aspect is used for defining the syntax of the concept. It repre­
sents the format of the concept that the user of the DSL will use to write
a given program, therefore one could say that designing the editor aspect
properly is crucial for the user experience (UX) of the language. An impor­
tant feature of the MPS editor is that it allows for inserting static text as part
of the concept, which contributes to the overall expressiveness of the lan­
guage so that the designer of the DSL can use natural language to express
its lexicon [7] [8].
• The generator aspect provide the generation of the DSL language in plain
text which can either be text or a GPL. The generator aspect is important
for the language as it is strongly related to the structure aspect as its model
is transform into another model then to plain text, this generation process
is called model­to­model and model­to­text. It is worth mentioning that the
generator language in MPS is not an easy tool to use, however there are
several open­source frameworks which aid in constructing a generator with
ease [8].
The concepts in MPS organize the DSL into three main pillars: the abstract syntax
tree, the user interface, and the execution engine.
The structure aspect defines the AST which is responsible for the semantics of
the language, the UI is responsible for the usability of the language, can also
be defined as the UX, and it is controlled by the editor and behaviour aspect of
the DSL, respectively the execution engine generates, using the generator, the
program using its AST representation which was modelled using the DSL [8].

2.3
Microservice Architecture
Microservices have been around for a while now, nevertheless they are still con­
sidered to be a new trend despite their popularity and that the software design
principles on which they are based on have been developed before this trend
emerged. Lewis and Fowler [1] define the microservice architecture style as an
approach for developing an ”application as a suite of small services, each run­
ning in its own process and communicating with lightweight mechanisms, often
an HTTP resource API” [1].
When describing microservices they are often compared to a different architec­
ture style in particular the monolith architecture paradigm, which is on the opposite
spectrum, as the microservice architecture is characterised by small independent
services, while the monolith architecture is defined as a single indivisible unit. Gen­
erally, in the monolithic style systems are built on three main pillars that is: (1) the
client side, representing the UI of the application; (2) a server­side application; (3)
the database.
In a monolithic system all the logic for request handling lies under a single process
allowing for basic implementation of the separation of concerns principle, that is
the different modules of the application can be split only as much as the program­
ming language can offer such as classes, functions, and packages. Despite the
fact that this architectural style has been the norm for most of the software de­
velopment lifetime, there are clearly several limitations as for any small or big
change the whole system needs to be rebuilt and deployed, scalabilty has to be
implemented in the system as as a whole rather than to its smaller units, and its
complexity and the effort for maintaining the different parts of the application sep­
arate increase on a non­linear basis.
Due to these downsides the new microservice architecture paradigm emerged
which focuses on building systems as a suite of services rather than a single indi­
visible unit. Furthermore, as the application is split in multiple autonomous mod­
ules they can be deployed and scaled independently, maintained by other teams,
and developed in different technologies [1]. An illustration showing the difference
between the two architectures can be seen below in Figure 2.1.

2.3.1
Aspects of Microservice Architecture
A microservice architecture enables componentization of an application via ser­
vices by design, meaning that an application is divided in a suite of components,
often called services, that are different than the traditional components (e.g. li­
braries) in an object­oriented (OO) program as they run in different processes and
communicate through a mechanism such as a web service request (e.g. HTTP
request) or remote procedure call, rather than using in­memory function calls.
By splitting the system in multiple services rather than libraries brings benefits to
the overall application as each single service is an autonomous unit which can be
deployed and scaled independently. In addition to this, the boundaries between
the components, in this case services, are better defined leading to loosely cou­
pled components and with more explicit interfaces preventing misuse of services
and possible production outages. Nevertheless, even tough the componentiza­
tion of an application brings benefits there are several downsides as well. For
example, remote procedure calls (RPC) are more expensive in terms of perfor­
mance and are more prone to failures due to network issues, compared to local
procedure calls (LPC) [1].
Moreover, the microservice architecture provides decentralisation governance,
meaning that the software does not have to be built on a single standard technol­
ogy stack which is used across allover the company but rather on multiple stacks.
Though it might not be always a good idea to split the technology stack, but the
approach can be quite beneficial as it enables teams to use the right technologies
for providing a solution for a given problem. For example, a team can opt to use
Node.js for implementing the payment service and Python for developing the cart
service.

meaning that each service focuses on the implementation of software for a spe­

As the underlying technologies used to build the suite of microservices can be

2.4.1
API Gateway
The API Gateway pattern is characterised, as its name asserts, by a gateway
though which a number of various services are accessible. The gateway acts
as a single entry point for all client requests and based on the request type the
message is forwarded to the appropriate microservice [14]. An illustration of the
pattern can be seen below in Figure 2.3.
Implementing the API Gateway in the MSA has many benefits such as [14] [15]:
• The division of the application in microservices is sealed from the client.
• The client is not concerned about communicating with each microservice at
the right endpoint as there is only the gateway endpoint, thus simplifying the
client logic.
• By using a variation of this pattern it can provided specifically tailored APIs
for each particular client.
• Performance increasing as the number of requests and round­trips are sig­
nificantly reduced in a scenario where data is aggregated from multiple ser­
vices.
Nevertheless, despite its many benefits the API Gateway pattern has a couple
of disadvantages as well, such as: (1) increased development complexity, the
actual gateway has to be developed, maintained, and deployed; (2) increases
response time, on a client request an additional network hop is performed through
the gateway [15].

to another changes. The BFF is tightly coupled with the specific client platform
and it is generally developed and maintained by the same team which develops
the client’s UI, therefore making it more simple to properly define and adapt the

Publish / Subscribe
The publish / subscribe pattern is defined by the usage of an asynchronous mes­
saging system. In standard microservice architectures the communication be­
tween the units is provided through a REST mechanism which makes use of the
synchronous HTTP protocol. However, synchronicity is not ideal in some scenar­
ios as it maintains the clients in a blocked state until a given request fails or is ful­
filled. Therefore, in applications where synchronicity is problematic event­driven
communications (publish / subscribe) and asynchronous messaging systems are
used to facilitate communication between the microservices [14].
The publish / subscribe model involves two type of actors, the publisher who sends
the message, and the subscriber who receives the message via a message sys­
tem. In addition to this, the core component of the functionality is the topic, which
is used by the publisher to send messages with a certain topic and the subscriber
listens for messages which are related to a given topic [17] [14]. An illustration of
how the publish / subscribe pattern works can be seen below in Figure 2.5.
There are several advantages for using this pattern as the publishers and sub­
scribers are isolated and decoupled of each other thus allowing for implementa­

2.4.4
Circuit Breaker
The Circuit Breaker pattern is inspired from how an electrical circuit breaker works
and it is used for handling failure of services. Microservices collaborate to han­
dle the client requests and when a service synchronously tries to communicate
with another service it is always a chance that the service is unavailable or expe­
riences high latency thus making it inoperative. Due to the unavailability of the
service resources are consumed up to a point where they are exhausted thus the
calling service would become unable to handle other requests as well. Therefore,
in a complex system it is highly probable that the failure of one service leads to a
snowballing effect where multiple services become unavailable.
The pattern in question addresses the issue by preventing the client to directly
communicate with the suite of microservices, but rather communicate through a
remote service via a proxy which works the same as an electrical circuit breaker.
Particularly, when a certain threshold of consecutive failures is met the breaker
trips and for a cool­down time invocations of the remote service will fail immedi­
ately. Consequently, when the cool­down time expires the circuit breaker starts
allowing a finite number of requests to propagate. If the requests are successful
the break resumes to normal, otherwise the cool­down time is started again [18].
An illustration of the design pattern can be seen below in Figure 2.6.
The Circuit Breaker pattern is beneficial in terms that it helps handling potential
failures of services, however choosing the right value for a cool­down time is tricky
as it can create false­positives thus introducing latency [18].

Proxy
The main characteristic of the Proxy pattern is that the client request is made to
an edge service which invokes different microservices without the client knowing
which microservice was actually invoked to fulfil the request. There are two dif­
ferent implementation types of the Proxy pattern: (1) the dumb Proxy, where the
Proxy simply acts as a delegator of requests between the client and the service;
(2) the smart Proxy, where the proxy performs logical operations such as aggre­
gation, rejection, or filtering, before forwarding the request to the service.
Typically, the Proxy pattern is beneficial in a context where there is a need for im­
plementing a unified API interface for the microservice suite in the form of a proxy,
while the functionality is handled by independent service units. In addition to this,
the proxy can be used for monitoring, logging, and enforcing security aspects [19].

and moreover, it enables fast application horizontal scalability due to the ease of
spawning new container instances when required.

Microservice DSL (MDSL)
Microservice DSL represents a domain­specific language to describe microser­
vice contracts along with their data representation and API endpoints, and was
constructed based on abstractions from domain­specific interface definition lan­
guages such as OpenAPI [27], WSDL [28], and ProtocolBuffers [29].
MDSL supports the API description pattern provied by Microservice API Patterns
(MAP) [30], drawing from its concepts such as endpoint, operation, client, and
provider. Additionally, the data contracts are related, yet abstracted to a general
form, to JSON [31] and Jolie [32] type system message exchange formats.
The design goals of the MDSL are to facilitate agile modelling practices, that is
to focus on the readability of the DSL over the parsing efficiency of the abstract
syntax of the language, as well as to be platform decentralised meaning that the
the language features which aid in the specification of a given microservice suite
should not be limited to a single communication protocol (i.e. HTTP). In addition
to this, it should facilitate a meet­in­the­middle service design specification, top­
down from requirements outlined in user stories related to API integration, and
bottom­up from an already existing system.
The abstract syntax of the DSL is based on the domain model and concepts of
MAP consisting of several features such as endpoints, operations, and elements
for representing data. The concrete syntax of API endpoints is elaborate consist­
ing of a set lexicon elements to enable a detailed API specification, and in contrast
the data representation lexicon is compact and simple abstracting from data for­
mats such as XML [33] and JSON, but also from service­oriented languages such
as Ballerina and Jolie.

Based on the previously covered material presented in Chapter 2, and Chapter 3,
some considerations will be drawn which will identify the set of non­functional and
functional requirements, and design guidelines for developing the MaGiC frame­
work.
The research projects MicroBuilder [4] and AjiL [5] are trying to address the several
issues encountered when implementing a microservice­based application. Their
solution is to provide a DSL which can be used to specify a suite of microservices
and generate them to a lower level programming language.
Nevertheless, best to the authors knowledge there is no state of the art research
which focuses on providing a DSL framework for specifying and generating a
microservice­based web application which consists of a client­side, a communi­
cation layer / gateway, and a microservice suite.
Drawing from the work of MicroBuilder [4] and AjiL [5], the focus of this project is
to develop a DSL framework for specifying and generating all three parts of the
MSA, which means that for each of the three parts of the application a DSL needs
to be provided.
Therefore, the goal of this project is to provide the MaGiC framework for devel­
oping MSAs which consists of three different domain­specific languages: (1) a
DSL that allows for specifying the microservice suite; (2) a client side DSL used
for constructing the UI of the application; (3) a gateway DSL for specifying an
efficient communication design pattern between the client­side and microservice
suite.
In Chapter 2 a research into domain­specific languages was done and it was found
out that when it comes to designing such a language there are several phases
each bringing a different challenge, and that there are some important key factors
to take into consideration such as DSL’s readability and the most accessible way
to design a language is to base its syntax on an already existing language.
Furthermore, implementing the DSL and generating it to a lower level program­
ming language is a highly complex and daunting task, however metaprogramming
systems make the implementation process easier allowing the user to focus more
on designing the language rather the generation part as tools provide a generator
for the DSL transformation to lower level programming language.
Therefore, based on the identified guidelines and the findings presented in Chap­
ter 3, MDSL [34] represents a great source of inspiration for designing the domain­
specific language this project is concerned with, as one of the MDSL’s design
goals is to promote readability over parsing efficiency and it is focused on the
microservice domain. Similarly, for the client­side DSL, the Wasp DSL [39] is an­
other beneficial source of inspiration, particularly the way the language is used to
specify the structure of the web application in terms of pages and routes could be
implemented in the client­side DSL.

5.1
Microservice DSL
The Microservice DSL represents the domain­specific language used to specify
a REST MSA, particularly users can express a microservice suite where each mi­
croservice exposes CRUD operations which are used to interact with a particular
set of data. As it was previously mentioned in Chapter 4, the syntax of the DSL
is inspired from a subset of features of MDSL [34] due to its high readability and
microservice domain­specific focus, to which several alterations were added.
5.1.1
Syntax
To begin with, the DSL consists of several properties which are used to spec­
ify metadata about the microservice such as its name, version, description, and
maintainer but also technical specific configuration such as its running port, gen­
erated language target, and data reference. In addition to this, a microservice
can be configured to expose several operation methods each operation support­
ing CRUD operation types interacting with a given DataType which represents
a JSON array data structure [43]. Furthermore, the operation’s API endpoint is
automatically generated, however the operation’s query parameters, expected /
delivered payload, and respectively its success / error message, need to be spec­
ified.

The Microservice DSL design choices are strongly related to the concepts of lan­

of constructing a verbose enough language and simple in the same time to allow

server­side [48]. Respectively, Express is a lightweight and flexible framework
for Node.js which provides a robust set of tools for developing web and mobile
applications with ease [49].
Python is a popular high­level general purpose programming language which has
a strong focus on code readability, and it is used for several computer science
areas such as machine learning, data analysis and visualisation, task automation,
and for developing web applications [50]. Flask is a small and lightweight frame­
work providing a set of features for developing Python based web applications

5.2
Gateway DSL
The Gateway DSL is used to specify and generate the communication layer be­
tween the server­side / suite of microservices and the client­side / application’s
user interface.
In Chapter 2 the most recurring software design patterns for implementing MSA
were researched and it was decided that the most suitable design pattern for im­
plementing the MSA using the MaGiC framework is the Backend for Frontend
(BFF) pattern. BFF is characterised by an approach where each client application
has a dedicated server side component called a gateway which acts as a commu­
nication layer between the client­side and the microservice suite (see Figure 2.4).
Additionally, as each client has a specific gateway the overall UX of the application
can be improved due to the fact that different clients support and have different
features. For example, a developer of an application is able to choose to display
information in a particular style on a desktop client, compared to a mobile one.
The BFF design pattern was chosen due to the fact that other surveyed design
patterns such as Circuit Braker, Load Balancer, are a good fit for a complex sys­
tem where the suite of microservices is large and horizontal scaling techniques
are employed, while in the context of the MaGiC framework this is not the case.
In addition to this, patterns such as Publish/Subscribe were not implemented due
to the scope of this project as the focus was not on facilitating asynchronous com­
munication with the services.
Nevertheless, the generated MSA can be deployed on both mobile and desktop
clients, and the user of the DSL might choose to serve a different UX depending
on the client. Therefore, BFF design pattern fits perfectly for this scenario. It is
important to note that when designing the Gateway DSL for implementing BFF a
strong focus was placed on the performance of the application as the user is able
to specify the exact information to be sent from the microservice to the client.
According to World Wide Web Consortium (W3C) best practices for mobile web
application include conservative use of resources as ”the most effective way to
ensure that applications run smoothly and with low latency is to minimize use of
device memory, processor power, and network bandwidth which are more limited
on mobile devices than on the desktop” [55]. Therefore, as mentioned previously,
the Gateway DSL was designed so that when specifying the communication layer
DSL users are able to specify what exact properties of the EntityType are expected
to be sent to the client­side, rather than sending all the properties, thus reducing
the network bandwidth.

Lines of Code
To validate further the MaGiC framework and compare the results with state of the
art research it was decided to compare the lines of code (LOC) needed to specify
the MSA using the three DSLs of the MaGiC framework and the actual generated
lower level programming language code.
Lines of code is a software metric used for measuring the size of a software appli­
cation by counting the number of lines of text of the program’s source code. It is
usually used for determining the effort amount required to develop an application,
as well as it represents an estimation of the programming productivity and main­
tainability once the software product is developed.
Nevertheless, counting lines of code is not always an accurate metric as imple­
menting an piece of software differs in terms of code lines from one programming
language to another due to their syntax and structure. Additionally, the number of
lines of code may differ from one engineer to another depending on their experi­
ence, but also on how verbose the program is written and its redundancy [67].

6.2.3
Cycle Time Tracking
To further validate the MaGiC framework, in particular the ease of the implemen­
tation of a microservice based web application, when implementing the web shop
MSA the cycle time was tracked which is a software development key performance
indicator (KPI) metric.
Cycle time, also referred as lead time, refers to span of time required for an engi­
neer to deploy the implemented code to production from the moment he started
working on it. With other words, cycle time indicates the time required to complete
a particular development task.
The cycle time KPI is used for determining the development velocity of teams as
well as their efficiency and ability to deliver a working piece of software within a
confined time frame [68].
In the context of validating the MaGiC framework the cycle time was tracked for
implementing the microservice based web shop application using the three main
DSL pillars. Particularly, the time was tracked using a stopwatch from creation
of the sandbox solution in MPS to the point in time when the application was de­
ployed to Docker containers and ready to be used.
It was found out that the cycle time required to implement the MSA web shop was
approximately 3h 27m.
The cycle time can be considered low for developing the whole application com­
pared with the manual process, nevertheless the result is influenced by several
variables and therefore is biased. This is due to the fact that the skill of person us­
ing the MaGiC framework is highly correlated with the cycle time result, meaning
that a more skilful person would require less time to implement the microservice
based web application.
However, implementing the MSA manually, without using the DSL, would require
several skills such as knowledge of microservice architecture and its related soft­
ware design, Node.js and Python programming languages, React SPA develop­
ment, and lastly Docker deployment configuration. Thus, it can be stated that as
the user would have to have experience with the MaGiC framework to implement
a microservice based web application, which requires time, making the learning
curve steeper and perhaps increasing the cycle time.
To conclude, it can be suggested that the ease of implementing and deploying
a MSA using the MaGiC framework is increased compared to the manual imple­
mentation process which requires knowledge of different technologies and writing
of boilerplate code templates for microservice suite and Docker configurations. In
addition to this, it could be argued that the cycle time of manually implementing
the application would be higher compared to the framework, but this could only
be determined if an actual experiment was done where both variables would be
measured.

Conclusion
To conclude, the dissertation focuses on microservice architecture, particularly on
developing a framework for specifying and deploying microservice­based web ap­
plications.
The research was inspired by the MicroBuilder [4] and AjiL [5] projects which en­
able developers to specify and generate a microsevice application using DSLs.
Nevertheless, the DSLs allow only for the generation of the microservice suite
and not a complete MSA.
Therefore, after having done an in­depth analysis of different concepts related
to microservice architecture and DSL development, the author of the thesis con­
cluded that the goal of the study is to develop a DSL framework, MaGiC, which
allows for specifying a microservice­based web application that consists of a mi­
croservice suite (server­side), a communication layer / gateway, and a client­side.
Additionally, it was decided that the framework must be language agnostic, elimi­
nate the need of writing boilerplate code templates for configuration the microser­
vice suite, as well as to provide a tool for building and deploying the application to
Docker containers.
Furthermore, it was determined that the generated MSA must be developed em­
ploying the best practices and software design patterns when it comes to imple­
menting a microservice architecture, and the MaGiC framework should ease the
overall development process of a MSA
In order to accomplish the formulated goals, the MaGiC framework was designed
and implemented and consists of three main DSLs:
Microservice DSL , which is used to model and generate the microservice suite.
Gateway DSL , used for specifying the communication layer between the client­
side / UI and the microservice suite.
Client DSL , representing the DSL for specifying the client­side of the web appli­
cation.
Having implemented the MaGiC framework, it was decided that in order to deter­
mine whether or not the framework meets the outlined objectives of the paper a
validation of MaGiC ought to be done.
For that a microservice based application was specified and deployed. The re­
alised application can be executed both on mobile and desktop platforms (see
Figure 6.1). Once the application was generated and deployed to Docker contain­
ers, an End­To­End testing scenario was designed and performed to determine
that the generated MSA is fully­functional.
It is important to note that the different components of the MSA were generated
to different programming languages so that the language agnostic aspect of the
framework was validated as well.
64
MaGiC: A DSL framework for implementing language agnostic microservice based web
applications

Moreover, to further determine if the set objectives of the paper were met it was de­
cided to perform software measurements which can be correlated with the ease of
developing an application. One of the observed software metrics were the lines
of code required to specify the different components of the MSA using MaGiC,
and the LOC of the actual generated MSA application in one of the two supported
programming languages. Another observed software metric was the cycle time
which refers to the span of time required for a developer to deploy to production
the application from the moment he started working on it.
The outcome of the validation was positive as the end­to­end testing was com­
pleted without issues, and based on the fact that the different components of the
MSA were generated to different programming languages the language agnostic
aspect of the MaGiC framework was validated as well.
Additionally, the results of the LOC comparison presented in Table 6.10 and the
required cycle time to implement the MSA, which was approximately 3h 27 m,
suggest the framework does ease the development process of MSA.
To stress this further, it can be stated that the objective of developing a frame­
work which eases the process when it comes to develop a MSA was met due to
the fact that implementing a MSA manually would require knowledge of microser­
vice architecture, suitable software design patterns, SPA development, as well as
Docker deployment and configuration.
To conclude, based on the process outlined throughout the whole paper and the
findings from the evaluation process of the MaGiC framework it can be stated that
the objectives of the thesis were met. Nevertheless, it is important to mention that
the current scope of the MaGiC framework is somehow limited as it supports spec­
ifying and generating MSAs with a low degree of complexity, and further develop­
ment of the framework and validation would be required to determine whether or
not MaGiC can be used for generating complex microservice­based applications.

Future work
To begin with, the thesis focused on developing the MaGiC framework which pro­
vides three different DSLs for specifying and generating a microservice­based
application formed of three different parts, a server­side / microservice suite, a
communication layer / gateway, and a client­side UI.
In the initial chapter of the paper several objectives were made, nevertheless due
to the scope of the project few limitations, though important when it comes to
implementing a MSA, were outlined. Particularly, aspects such as security, avail­
ability, asynchronous message queuing, and monitoring were not a focal point.
Therefore, for the further development of the MaGiC framework these aspects of
MSA development ought to be considered and incorporated in the framework.
Secondly, the current implementation of the microservice suite facilitates only a
North / South communication, that is the services are able to communicate only
with external components, in this case the BFF gateway. However, in a more
complex MSA the services communicate with other services as well, East/West
communication [69]. Thus, in order further develop the MaGiC framework and
allow generating of more complex MSA, East/West communication could be de­
signed and implemented in the Microservice DSL.
Figure 8.1 illustrates the concept East / West and North / South communication in
an MSA generated using the MaGiC framework.

and North/South microservice communication.
Moreover, the Client DSL provides a limited amount of UI components therefore
the complexity of the UI in terms of its design its also limited. Thus, in order to
further add value to the MaGiC framework more UI components could be imple­
mented in the language.
Another important aspect of the Client DSL is that due to the scope of this the­
sis, only generation to React SPA was implemented. Nevertheless, in order to
66
MaGiC: A DSL framework for implementing language agnostic microservice based web
applications

promote technology decentralisation generation to other UI frameworks, such as
Angular [58] and Vue.js [59], for building SPAs ought to be implemented as well.
Additionally, the current implementation of the MaGiC framework does not sup­
port the use of database management systems. Particularly, the data with which
the microservices interact is stored in a locally stored JSON file which is some­
what limited. For the scope for this project storing the data in a JSON file sufficed
as the data is minimal and the implemented MSA has a low complexity, however
it would be highly beneficial for the MaGiC framework to allow storing of data in
a database system and expose the necessary operations for interacting with the
data.
Lastly, in the Concepts and Background chapter several software design patterns
related to implementing a microservice architecture were presented. From those,
the Backend For Frontend pattern was implemented in the MaGiC framework,
nevertheless multiple design patterns could be implemented as well. For exam­
ple, if the MaGiC framework was to be developer further and support generation
of complex MSAs with a high traffic, the Load Balancer pattern could be beneficial
as the application would have to employ horizontal scaling techniques.
Furthermore, the MaGiC framework could benefit of DevOps principles especially
CI / CD as it would improve the deployment rate of the system, and in the same
time maintain a higher quality of the software.

A.1
MaGiC Framework: MSA Implementation
The following section presents the implementation and deployment process of the
microservice­based web shop application used for validating the MaGiC frame­
work.
Note that the section does not provide the DSL documentation, and a detailed
explanation of the MaGiC framework in terms of its DSLs syntax and concepts is
presented in Chapter 5.
A.1.1
Software Requirements
In order to reproduce the implementation of the microservice­based web shop
application the following software tools are required:
MPS , version 2021.2 .
Node.js , latest version.
Docker , latest version.
GIT , latest version.
MaGiC framework , the framework can be cloned from the following GitHub repos­
itory https://github.com/claudiuciumedean/language-agnostic-microservice-dsl .
A.1.2
Implementation
The implementation of the microservice­based application required a bottom­up
approach (see Figure. 6.2), thus initially the microservice­suite was generated,
then the BFF gateways, and lastly the client­side.
A full overview of the application’s components can be seen in Figure 6.1
Microservice Suite
The microservice suite of the application consists of three different services Cus­
tomersService, OrdersService, and ItemsService.
To start implementing the services a MPS sandbox was created where the Mi­
croserviceDSL was imported as a dependency. Once the dependencies were
imported, for each microservice three different files representing MPS structures
were created, namely a Microservice structure, a DataType, and a Deployment­
Config. Figure A.1 illustrates the microservice structure in MPS in terms of its
files.

BFF Gateways
The application has two BFF gateways for supporting the two different client plat­
forms, mobile and desktop (see Figure 6.1).
To specify the BFFs the BFF DSL was imported as a dependency in the same
sandbox as the one for the microservice suite. Different from the Microservice
DSL, for configuring a BFF only a BFF type file had to be created which is a MPS
structure that represents the abstract model of a BFF gateway.
Similar as in the case of declaring a microservice, when the BFFs were declared
metainformation was specified and their respective REST CRUD operations.
Note that the exposed operations of both BFFs can be seen in Table 6.4 for the

Deployment
Deploying the application to Docker containers is trivial as the MaGiC framework
comes with a tool which automatically deploys and builds the application.
Figure A.12 illustrates the deployment script written in Node.js .
Figure A.12: Illustration of the build script for deploying the application to Docker

The steps for deploying the application were the following:
1. Downloaded Node.js and installed it.
2. Opened the terminal tab in MPS and navigated to the root folder of the
MaGiC framework.
3. Run the following command: npm install.
4. After NPM packages are installed, run the following command: node build.js
­l <application generation location>, in the web shop app case was ./build/ar­
tifacts/SandboxBuild/Application. It is important to make sure Docker is run­
ning in the background.
5. Open a browser window and navigate to localhost:<client port number>,
in this case it was 3000.
