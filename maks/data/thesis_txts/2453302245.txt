Summary
This project is the preliminary work towards winning the MBZIRC 2020 challenge 2
with focus on the ground vehicle. The ground vehicle is a robot built by DTU named
iRobot. The iRobot have a laser scanner, a Universal Robots arm, a camera and two
electromagnets.
The camera and magnets are attached to the arm which allow the
iRobot to detect and lift blocks. The iRobot is controlled using MRC and it drives
using skid steering.
Challenge 2 can be divided into four main challenges for the project: the detection of
the blocks, the lifting of the blocks, the navigation and the wall building. The detection
of the blocks is achieved with deep learning using five different neural networks, which
all serve different purposes. These neural networks are created with Tensorflow Object
Detection API.
The iRobot uses the results from the neural network to locate the blocks.
The
location of the blocks are translated to the iRobots arm’s coordinate system in order to
lift a block.
The navigation is achieved by the iRobot saving the location of points of interest.
These points include the stacks of blocks and the wall. The iRobot can use the location
of the points to drive between them.
To place the block on the wall, the iRobot uses point of reference to improve the
precision of the placements. The point of reference is the corner of the wall, because
this is the first block placed. This means that, the iRobot uses the first block to place
the remaining part of the wall.
The solutions are made scalable because the carry capacity of the iRobot will increase
in the future. Using these solutions, the iRobot can build a wall autonomously consisting
of two different colored blocks. The iRobot does this consistently. The blocks have to
be located in two randomly placed stacks in front of the iRobot’s start position in order
for the iRobot to locate them.

CHAPTER 1
Introduction
It is common that workers become worn out during construction due to heavy lifting.
Therefore, developing a robot that does the heavy lifting could help alleviate the work-
load of workers and allow the workers to focus on intricate work tasks. This would
reduce the likelihood of work related injuries and stress, and it would reduce the cost of
construction. This is the motivation behind the second challenge of MBZIRC.
MBZIRC is a large international robotics competition held biennially in Abu Dhabi.
This project aims to do the preliminary work for the MBZIRC 2020 challenge 2 solu-
tion, where a ground robot and three drones will have to collaborate to build a wall
autonomously. This project is focused on the ground robot.
The project can be divided into four main challenges, which are, the object detection
of the blocks, the lifting of the blocks, the navigation and the building of the wall. The
object detection of the blocks is made with deep learning by using Tensorflow Object
Detection API, and the solutions are achieved using Python.
This report describes the solutions to the four challenges and their implementation,
as well as, an evaluation of these implementations.
In Figure 1.1 the robot is shown building a wall.
Figure 1.1: An image of the robot while it is building a wall.

1 Introduction
1.1
Problem formulation
The aim of this project is to program the robot provided by DTU to autonomously
build a wall within the regulations of the MBZIRC 2020 challenge 2. This project is the
preliminary work towards winning the challenge.
To win the challenge, the iRobot and the drones need to autonomously build a pre-
defined wall as large as possible within 30 minutes. In order to do this, methods for
detecting the blocks, lifting and placing them need to be developed. Control of the
iRobot and drones which ensures that they can avoid collisions, while moving, lifting
and placing blocks, also have to be designed. Lastly, a dynamic navigation system, which
allows the iRobot and drones to move between the stacks and wall, has to be designed
and implemented. This project focuses on the iRobot, because the project is only the
preliminary work towards winning the challenge. Methods for completing the following
tasks must be designed, in order for the iRobot to autonomously build a wall:
• Object detection
• Lifting and carrying the blocks
• Navigating the arena
• Consistently placing the blocks
The solutions to these tasks must be designed in a manner that allows the iRobot to
collaborate with the drones or it must be possible for the solutions to be expanded to
allow the iRobot to collaborate with the drones.
1.2
Overview of the report
Chapter 2 illustrates the different components of the iRobot and analyses the potential
solution to the tasks
Chapter 3 describes the implementation of the object detection, how the data-sets for
the detections are made, and how the training and verification functions
Chapter 4 outlines how the lifting of the blocks was implemented, including how the
angle of the blocks was estimated using both laser and camera
Chapter 5 describes how the robot locates the blocks in the arena, as well as how it
manages to drive back and forth between the blocks and the wall
Chapter 6 describes how the robot keeps track of which block to place and where to
place it, and how the robot ensures that the blocks are placed as precisely as possible
Chapter 7 evaluates how precise the implemented methods where as well as discussing
potential reasons for imprecisions and possible solutions to them

2.1
The iRobot
The robot used for this project is a robot built by DTU named iRobot, which has been
used for various earlier projects. The iRobot is built with skid steering for its wheels and
a gyro to account for its heading. The iRobot has had various attachments in relation
to each project’s task. For this project the iRobots attachments consist of a SICK laser
scanner, a Universal Robots arm, a camera and two electromagnets. The iRobot can be
seen with the attachments on in Figure 2.1.
Figure 2.1: The iRobot with attachments on.

2 The analysis of the iRobot and the tasks
2.1.1
The laser scanner
The laser scanner attached to the iRobot is a SICK 2D-LiDAR-sensor of type LMS100-
10000, which can be seen in Figure 2.2.
Figure 2.2: SICK 2D-LiDAR-sensor of type LMS100-10000 [AS19].
The laser scanner is attached under the iRobot’s bumper with an aim to sense the
front environment. The opening angle of the laser scanner is set to 180° with an angular
resolution of 0.5°, which gives 361 data points of distance measures in front of the iRobot.
The working area for the distance measurements is between 0.5-20m [AS19].
2.1.2
The Universal Robots arm
The Universal Robots arm attached to the iRobot is a UR5, which can be seen in Figure
2.3.
==PAGE:21==
2.1 The iRobot
7
Figure 2.3: Universal Robots UR5 arm [Rob19b].
The UR5 is attached on top and in the front of the iRobot. The weight of the arm is
18.4 kg and it’s able to carry a payload of 5.0 kg. The reach of the arm is a radius of
850 mm from the base with a repeat ability of +/- 0.1 mm. The arm has 6 degrees of
freedom with 6 rotating joints [Rob19a].
Two different commands were used to move the arm during the project. The first,
moves the arm in Cartesian coordinates and the tool of the robot will move in a direct
line from its current position to the given position. The second, moves the arm in joint
coordinates where it moves each joint simultaneously to the given position

2.1.7
Skid steering
The steering of the IRobot is skid steering and the disadvantages and advantages of
skid steering can be found in the book [RS11]. The disadvantages, according to the
book, are that the exact center of rotation of the robot can be hard to predict and the
exact change in position and orientation can vary due to skidding during a turn and is
dependent on the friction of the surface: Therefore, the dead reckoning on the IRobot
is highly inaccurate.
The advantages for the trade-off, is that the IRobot has good
maneuverability and traction over rough and loose terrain.
2.1.8
Mobile Robot Control
The iRobot is controlled using a control language written by DTU. This language is
called Small Mobile Robot Control Language (SMR-CL) and the full documentation of
it can be found in [Ele19]. The following commands from the documentation was used
in this project:
Turn: Turn is used as the basic turning command. It takes an angle in degrees and
turns the iRobot by that amount.
Drive: Drive is used when driving over longer distances, since this command regulates
the driving of the iRobot. The command takes a line and tries to make the iRobot drive
along said line.
Fwd: Fwd is used when driving straight for small distances, and in the end of drive
commands. Fwd regulates the distance driven, and gives smoother stops than Drive.
The command takes a distance and makes the iRobot drive that distance
Wait: Wait is used to give the iRobot time to regulate between commands. The com-
mand simply sleeps the control language.
Stop: Stop is used to stop the iRobot and keep its current position and direction.
Eval: Eval returns parameters from the iRobot, and is used to get the odometry of the
iRobot.
Set: Set is used to reset the odometry of the iRobot.
Targethere: Sets the targetpose of the iRobot to current pose.
Log: Log is used to save data from the iRobot for later debugging.
Flushcmds: Flushcmds clears all current commands and is used to clear the drive
commands when searching for the stacks and wall.
Four internal parameters from the Mobile Robot Control (MRC) are used. They are
drivendist, odox, odoy, and odoth. Odox, odoy, and odoth is the odometry parameters of
the iRobot. These were used to get the position of the iRobot in the iRobot’s coordinate
system.
Drivendist is a parameter which is used as an exit condition on the drive
command. It is the distance driven from when the drive command started.

2.1.10
Sleeps in the program
The python program which is the master and the MRC which is the slave, run interde-
pendently. This means that the python program will sometimes have to wait for the
MRC before continuing, because it might queue a lot of commands in the MRC or give
commands to the arm, camera or laser at the wrong time.
When the MRC runs a command, it will not proceed to the next command in its
code until the first command is completed. This means that the program sometimes
has to wait the same amount of time before sending the next command to verify that
the robot is in the right location before sending the command. This has been done by
using sleeps inside the python program, where the program will wait a specific amount
of time before it proceeds with the code. Most of the times have been calculated to
reduce waiting time, but some of them are longer to ensure that the robot is in the right
position before proceeding.
It is possible to get feedback from the MRC to know when it is finished with each
line, because the MRC can indicate when it is done with a command. The MRC does
it by saying it is done with a command number and then the program has to remember
what line of code the number belonged to.

A comparison between consistency and speed
To build a pre-defined structure in 30 minutes is difficult: The robot will probably use
the full extent of the time period. This gives a lot of time where the robot will need
to work without failing and ruining the run. Which means that if speed is chosen over
robustness the robot might fail after 5 minutes or it might fail after 25 minutes, because
each task the robot has to do in the challenge is repeatable and there is a risk for the
robot to fail it each time. It is then a statistical calculation of how long the robot needs
to run with a risky higher speed to get the same amount of points as if the robot ran
with a lower more robust speed for the entire 30 minutes. The statistical calculation will
then be how often the robot manage to get the same amount of points or higher with
the higher speed to see if it is worth taking the risk.

2.4
Object detection
For this project the detection of the blocks has been achieved with deep learning. An
easy and fast approach to object detection with deep learning is using Tensorflow Object
Detection API. This is an open source framework built on top of Tensorflow, which
helps with the construction, training and deployment of new object detection models.
The Object Detection API can localize and identify multiple objects in a single image
[jch+19].
By localizing and identifying multiple blocks in a single image it becomes possible to
choose the best block and to lift multiple blocks without having to take a new image.
2.4.1
Pre-trained models
There are already numerous pre-trained models for the Tensorflow Object Detection
API. A pre-trained model is used because it provides weights in the neural network that
fit together. The pre-trained model is able to recognize features it can use to detect the
objects in the image.
Pre-trained models can be divided into two categories, the Single Shot Detector
(SSD) models and the Faster Region Convolutional Neural Network (Faster R-CNN)
models. The SSD models focus on running quickly with many images per-second. The
Faster R-CNN is slower, but it is more accurate in its predictions [ptn19].
This project needs a balance between accuracy and speed, however it will not work if
the accuracy is too low. The pre-trained model chosen is a Faster R-CNN model named
”faster_rcnn_inception_v2_coco”: It is slower compared to the SSD models, but faster
then the rest of the Faster R-CNN models. It has a high accuracy compared to the SSD
models, but a low accuracy compared to the rest of the Faster R-CNN models.
2.4.2
Faster R-CNN neural network
The Faster R-CNN models are state-of-the-art object detection networks, which move
towards real-time object detection with region proposal networks [Ren+16]. The Faster
R-CNN network can be seen in Figure 2.10.
==PAGE:32==
18
2 The analysis of the iRobot and the tasks
Figure 2.10: The information flow of the Faster R-CNN network [Ren+16].
The Faster R-CNN network is a unified network for object detection. The difference
between other networks and Faster R-CNN is that it is a unification between standard
object detection and the region proposal network (RPN). Figure 2.10 shows how an image
given to the network creates feature maps of the image using multiple convolutional
layers. The feature maps are then passed on to the RPN and to the classifier. The RPN
will give proposals to where in the image objects are depending on the features in the
image. By combining the proposals with the feature map it is possible to classify what
kind of object the RPN has found. Classification of the object and where it is in the
image will then be sent via the network.
The proposals are the test-time computational bottlenecks in the detection systems
[Ren+16]. It takes 1.06 seconds to analyse a single image on the robot which equals an
fps of 0.94 fps.

2.5
Lifting the blocks
In order to lift the blocks, the iRobot has to recognize where the blocks are placed
relative to it. This can be achieved by driving close to a stack and taking an image of
the stack with the camera in a high position in front of the iRobot. The neural network
identifies where the blocks are in the image and marks the blocks. The blocks will be
detected in bounding boxes where it will be possible to lift each block in the middle of
its individual bounding box. The metal plate is attached to the middle of the block. The
pixel coordinates of the bounding box translate into metric coordinates for the iRobot’s
arm to lift the block.
2.5.1
Detecting the angle of the blocks
To place the blocks correctly in the wall, the iRobot needs to detect the angle of the
blocks: Consequently, each block will be placed with the same orientation. There are
two sensors on the iRobot which can be used to detect the angle of the block: The first
is the laser scanner and the second is the camera.
2.5.1.1
Using the laser scanner
The laser scanner can only detect one block’s angle at a time. Using the laser scanner
the iRobot will have to lift the block and drive backwards to avoid hitting other blocks
in the stack. The laser scanner’s working area is in the same height as the blocks. After
reversing the iRobot can move the block down in front of the laser scanner and take
the block’s measurement. From the laser’s measurement it is possible to calculate the
orientation of the block and with trigonometry it is possible to calculate the angle the
block needs to be turned to align it with the iRobot: Turning its wrist joint with the
calculated angle will align the block with the iRobot. The iRobot will turn the block
until the requirements for the angle threshold are met.
2.5.1.2
Using the camera
The camera can detect the angle of multiple blocks at the same time. The image used to
lift the blocks is also used to calculate the angles. Using the bounding box and its label
the iRobot can calculate the angle of the desired block. By calculating triangles in the
bounding box it is possible to detect how much the block needs to be turned. The label
of the block indicates if the block needs to be turned to the left or to the right. When
lifting the block it will be possible to lift the block free from the other blocks and turn
it immediately with the calculated angle: Which makes it faster than the laser scanner.

2.5.2
Choosing between multiple blocks
After having detected all the blocks in the image the iRobot chooses which block it lifts

is selected. The iRobot has a level of uncertainty linked with the distance of the block
from the center of the image when lifting the blocks. The iRobot changes the order of
the list to lift the blocks closest to the middle.
2.5.3
A partial block inside the image
If any blocks are not completely inside their bounding box the iRobot should not lift
them. This is because the bounding boxes are limited by the edges of the image, the
middle of the blocks are therefore not the middle of the bounding boxes.
The iRobot should move according to the positions of the blocks detected in the
image. If a block is partly detected to the left or to the right of the image the iRobot
will have to turn to the left or to the right. If a block is too close or too far away the
iRobot will have to reverse or drive forward.
After the iRobot has moved according to the first image the iRobot will take a new
image. It will keep moving and take new images until it manages to get a block fully
inside the image.
2.5.3.1
The size of the image
The iRobot has physical limitations, height and maximum arm extension, which limits
the size of the image. The blue and the orange blocks will not be fully inside the image

Finding the stacks

the image with measurements obtained from the laser scanner.

The pulsing of the laser scanner should start when the iRobot is close to the point

The proximity to the point of interest depends on the distance between the stacks.
It will be possible to locate the wall with the laser scanner after the first block has
been placed. This can be done by turning toward the wall, which makes the distance
and angle measurements of the wall more accurate.

wall. If the odometry drifts too much the iRobot may pass a point of interest without
detecting it.

according to color and the second dimension is the height of the wall.
It will then be possible for the iRobot to calculate the next block it needs by looking
at the array and it can calculate where the block should be placed.
This block array would allow the robot to skip certain blocks, because the array can
be used to calculate the position of any block. The calculations will take any potential
holes and the block space into account and place the next block further down the wall

After the first block has been placed the blocks can be placed in the wall using various
methods with some being faster and others more consistent: The run can only take 30
minutes in total.
The first method to place the blocks in the wall is to drive to the corner of the wall
where the first block is placed. Then the iRobot should do a laser measurement and
calculate how much it needs to turn until the iRobot is aligned with the wall to be in
the threshold. The iRobot should then do another laser measurement to calculate how
much the iRobot needs to drive forward to get to the right distance to the wall. When
both the angle alignment and distance to the wall have been accepted the iRobot should
turn parallel to the wall and start driving along the wall. The distance it has to drive
should be calculated relative to how far the iRobot stands from the corner of the wall,
which is possible with a laser measurement before it turns. When the iRobot has driven
along the wall the iRobot will turn to point the front of itself toward the wall and do a
laser measurement to align itself with the wall to remain within the threshold. When the
threshold for the alignment has been reached the iRobot should do a laser measurement
and calculate how far from the wall it is standing relative to the block next to the block
it is about to place. It should then take that calculation into account when it places the
block with the arm in front of itself.
The second method to place the blocks in the wall is in the beginning much like
the first method. The iRobot should drive to the corner of the wall where the first
block is placed, where it should do a laser measurement to align itself with the wall to
be within the threshold and then do a laser measurement to drive closer to the wall
to be within the threshold. When the iRobot has driven closer to the wall it should
do a laser measurement to see if it is still aligned with the wall and to see where the
==PAGE:39==
2.7 Wall building
25
iRobot stand relative to the first block placed. It should then take that measurement
into account while turning parallel to the wall. Parallel to the wall the iRobot should
drive backwards relative to how far it is from the corner of the wall and finish with its
laser scanner near the middle of the first block. The iRobot will stop for a moment
in this spot and then start pulsing laser measurements looking to the side on the wall.
While pulsing the iRobot will start driving backwards until the first laser measurements
to the side are past the wall, where it will trigger the iRobot to stop. The iRobot has
then found the edge of the wall and can drive forward to where it wants to place the
block. While driving forward the iRobot should pulse laser measurements off the wall
to use them if it drives awry. When arriving at the right spot the iRobot should take
the last laser measurements into account before placing the block with the arm to the
side of the iRobot.
The third method to place the blocks in the wall is to look at the wall from a longer
distance and then calculate where the iRobot has to drive to place the block.
The
iRobot should do the calculation relative to the first block placed in the corner of the
wall. While driving toward the right spot on the wall the iRobot should align itself with
the wall with its drive commands. When the iRobot arrives to the right spot it would
do a laser measurement of the block next to the block it is about to place to calculate
the distance to the wall relative to that block. It should then take that calculation into
account when it places the block with the arm in front of the iRobot.
The first method takes a long time with all its turns and it could be problematic
given the time limit. The second method also takes a long time, but it is faster than
the first method. It is the best scalable method if it is possible for the iRobot to carry
more than one block at a time. Which would improve its time calculations. The third
method was faster than the other two, but it was less consisent and it is not a scalable
method.
2.7.4
The precision of the block placement
The precision of the blocks placed in the wall is affected by a combination of the ability
to lift the blocks in the middle of the blocks, the ability to turn the blocks to a consistent
angle and the ability to drive to the exact spot on the wall. With these uncertainties
it is difficult to place the blocks exactly next to each other. A way to cater for the
uncertainties is to make a gap between each block, because even if a block is skewed
to one side it will then still be possible to place it in the wall without interfering with
the other blocks. Ideally the gap size should alleviate the uncertainties, because if the
first block is skewed to the right and the next to the left then the blocks will still not
interfere with each other.

CHAPTER 3
Object detection
To be able to use the deep learning object detection it requires the neural networks, from
here on known as networks, to be trained using the right images to be able to create the
correct feature maps for the object detection. Which means that the images will have
to be taken in the right scenarios for each various network. The images for the networks
will have to be generated and gathered into data-sets before being used.
After generating the data-sets the images in the data-sets will have to be labelled
and given the right classes to provide to the networks which objects in the images the
networks have to detect and what the detected objects are called. It is then possible
to train the networks on the data-sets and determine if it is possible for the object
detection to find the right features in the images to be able to give a correct prediction
of the objects in the images.
3.1
Generating the data-sets
The right images had to be linked with the right xml-files so that the network could
know what to identify.
The xml-files are telling the network which kind of objects
are interesting and where they are placed in the images. This was done by generating
data-sets for each scenario trained in this project.
Each time a part of the code is tested on the iRobot and it takes an image the image
will be saved for the given data-set. The data-sets will keep growing with time and
provide different images for the data-sets with various backgrounds. This will improve
the consistency and correctness of the networks when they are trained with various
images.
3.1.1

To improve the network, the images taken for the data-set are taken in the same position
as the images would be when the network is used.
The images for the image position were taken with the arm raised as high as possible
to make the images contain as much of the blocks as possible. The camera was facing
directly down to be able to see the top of the blocks.

3.2
Training and validation
The same images should not be used for training and validating, because it could result
in overfitting. To avoid this 20% of each data-set is separated into a validation set and
the 80% rest of the data-set are used for training the network.
The network trains by trying to detect the objects in each image in the data-set
where it tries to adjust its weights in the layers to find the best possible fit. After having
been through each image the network will be given a score of how well it did. The
network will then try to change the weights in the same direction as it did during the
last run and run through each image again. It will keep doing this until the weights have
settled around a point where the network believes it has found the right combination
to find the desired features. To keep the network from being stuck in local minimums
the network will change the weights with a random parameter or with momentum if the
network has been given the same score a couple of times. Which also means that if the
network had found the best fitting weights it might be pushed away from them again
for a short time before the weights get adjusted back.
The network saves different versions of the weights during the training, so it is possi-
ble to take an earlier version of the network if it fits the data better. It saves the weights
of the network every 10 minutes to a maximum of five different versions when it saves
a new version the oldest will be deleted.
It is possible to see how well the network is doing during its training. If the network
stops improving, by not maintaining the level of object detection, this indicates that the
network is finished training and it should be stopped early. The network can then be
tested to see if it found the right weights to detect the desired objects or if the training
parameters need to be changed before new training is started.
In this project the training of the networks has taken approximately 2-3 hours, before
the networks scores started to level out and repeat.

3.2.2
Training loss
The way the loss scoring of the network works is by trying to adjust the weights in the
direction of that which makes the total loss of the network as small as possible. The
total loss of the network is calculated by adding the individual losses together. There
are four losses in this network. The first two losses are in the RPN part of the network,
where there is a objectness loss and a localization loss. The other two losses are in the
box classifier part of the network which is the second part, where there is a classification
loss and a localization loss.
When the network is trying to make the total loss as small as possible it means that
if one of the individual losses are higher than the others it will be prioritized higher and
the weights will be adjusted primarily in the direction of that individual loss. A way
to control which individual loss should be prioritized is by adjusting the weights of that
loss. The weights of the losses are a scaling factor that scales how much the losses should
be multiplied when calculating them.
The adjustments of the weights can be required if it is important for the network to

the classification loss, will it be prioritized higher and the weights in the network will

3.2.3
Overfitting
When training a network there is a risk that the network overfits its weights. Overfitting
is when the network is really good at predicting in the training data-set and gets the
predictions right every time, but when it is taken out and tested in the same environment
on new images it can not predict them correctly. Which indicates that the weights had
identified certain features which were only present in the training images and not in the
task it is trying to solve.
The smaller the data-set the bigger the risk to the network to start overfitting. A
way to handle this is to increase the number of images in the data-set to the network
and to change the background of the images if the background is not task specific. The
network will learn that it should not identify anything from the ground if there is a
special mark on it.
==PAGE:57==
3.2 Training and validation
43
If there is two similar objects then the network can learn to identify only the correct
object. This is done by showing the network images of the wrong object and telling it
that it is not an object, therefore any object which is similar to the wrong object will not
have a label. That way the network will have to find more features of the two objects
to differentiate them and it will start identifying only the right object.
In this project the images are taken with a lot of various backgrounds and lighting
conditions for each data-set. This is to avoid the risk of overfitting the network and
training it to be more consistent. In this project there have been no signs of overfitting
of the different networks and they all seem to work in different environments.
3.2.4
Data augmentation options
In the configuration file with the training parameters it is possible to set data augmen-
tation options. This improves the data-sets without having to add more images to the
data-set. This can especially be advantageous if the initial data-set is small. It will
improve the consistency of the network.
The data augmentation options can be found in the preprocessor.proto file which
came with the installation of the Tensorflow Object Detection API. For this project only
some of the options have been chosen to increase the data-sets. The five options chosen
for this project are:
random_horizontal_flip: Randomly flips the image and the detections horizontally
with a probability of 50%.
random_vertical_flip: Randomly flips the image and the detections vertically with
a probability of 50%.
random_rotation90: Randomly rotates the image and the detections 90 degrees
counter-clockwise with a probability of 50%.
random_adjust_brightness: Randomly adjusts the brightness of the image to be
between 0 and 1.
random_adjust_saturation: Randomly adjusts the saturation of the image to be
between min_delta = 1 and max_delta = 2.
The top three data augmentation options are used to expand the data-set, while the
bottom two both expands the data-set and improves the networks resistance to changes
in illumination.
In this project it is only the newest trained networks that have all of the options
implemented. The first networks only had ”random_horizontal_flip” as a data augmen-
tation option. The newest trained network for finding white lines from above is trained
with all five data augmentation options implemented because it would make more sce-
narios for the white lines to be in: It will not interfere with the object detection of the
network.

CHAPTER 4
Lifting the blocks
This chapter describes how the different challenges related to lifting the blocks were
solved. This chapter will explain the calculations behind the solutions and the code
which implements them.
The two main challenges related to lifting the block were: How to lift the blocks and
how to estimate the angle of the blocks.
4.1
Calculating the lifting position
The neural network only returns the position of the bounding box surrounding the object
and the color of the object. The position of the bounding box is given in four numbers,
bottom, top, left and right. Top and bottom are the lower and upper bound of the
bounding box in the y direction of the image, and left and right are the lower and upper
bound in the x direction.
If the bounding box is placed perfectly, the exact middle of the bounding box will also
be the exact middle of the block. This is because the bounding box exactly surrounds
the block and nothing more. This only holds true if the entire block is within the image.
Section 4.2 will explain how the iRobot ensures that any block that it tries to lift is
entirely within the image.
The middle of a bounding box, meaning also the middle of a block, can be found
using the top, bottom, left and right coordinates given from the neural network. The
y coordinate of the middle point is the average of bottom and top plus top, since the
y coordinate of the image goes from the top to the bottom. The x coordinate is found
by taking the average of left and right plus left, since the x coordinate goes from left to
right.
Knowing the location of the block cannot alone be used to lift the blocks, because
the position of the block is only known in the image’s coordinate system. The position
of the block in the image’s coordinate system needs to be translated and rotated to fit
the arm’s coordinate system. The units of the image coordinate system is in pixels, and
the units in the arm coordinate system is meters. It is therefore necessary to translate
from pixels to meters.

Arm movements during lifting
The iRobot needs to ensure that the blocks does not bump into each other when it lifts
them, because this might cause the block to fall off the magnet. The iRobot therefore
cannot move the arm in a straight line from where it lifted the block to the front of itself.
The iRobot makes sure to lift the block above the ground far enough that the block that
it is carrying is entirely above the other blocks.
It is assumed that there will be no block which is on top of each other in the stacks,
and the height of which the arm needs to be in is therefore the height of two blocks.

4.2
A partial block inside the image
There is no guarantee that the blocks will be completely within an image that the iRobot
takes, and the method of lifting the blocks assumes this. This means that the image must
be analysed to see if at least one block which is completely within the image before the
iRobot attempts to lift that block. If there are no blocks completely within the image,
then the iRobot must change position such that a block can be completely within the
image.
The bounding boxes of the blocks was used to determine if a block was completely
within the image and what direction the iRobot had to move to get the blocks completely
within the image. A block that is partly within the image will still be detected and receive
a bounding box, and the bounding box of said block will always end up in the edge of
the image, since that is where the block is located. This mean that if all four edges of
a block’s bounding box is away from the edge of the image, then the block itself must
be completely within the image. This method does have false positives, but all a false
positive costs is time, since the iRobot simply readjusts its position to get the blocks
more within the image,
The way this was implemented was with a for-loop. The for-loop runs through all
the bounding boxes in the image and checks if any of the edges, of each bounding box,
are within 10 p of the sides of the image. The bottom of the image had to be 20 p away
from any bounding box edges, since the bumper of the iRobot was sticking out into the
image. If no edges are close to the sides of the image, then the index of that bounding
box is saved. This results in a list of indexes of all the blocks which are completely inside
the image.
If this list of indexes ends up empty, it means that there are no blocks completely
within the image, and the iRobot needs to adjust its position.
The location of the
edges of the bounding boxes can also be used to figure out what direction the iRobot
needs to move. The iRobot moves towards the block closets to the iRobot, in order to
avoid driving into any blocks. The order, of which the edges were checked, also followed
this concept: checking the bottom edge first, then the sides and at last the top, using
whatever result which came first.

4.3
Choosing between multiple blocks
When there are multiple blocks all completely within the image, the iRobot needs to
decide which one it lifts. The closer the block is to the middle of the image the more
precisely the iRobot can lift it. The iRobot therefore calculates the distance from the
middle of the bounding box to the middle of the image for each bounding box, which
is completely within the image. The iRobot compares these distances and chooses the
smallest one to lift. This way the iRobot always lifts the block which it can lift with the
highest precision.
Before lifting the block the iRobot checks if the block is within its reach, if it is not,
the iRobot will drive forward to get within range of the block and take a new image.
4.4
Finding the rotation of the blocks
The iRobot have two methods of getting the rotation of the blocks. The first method is
to calculate the rotation of the block from the bounding box data. The second method
is to use the laser scanner to measure the angle of the block by holding the block in
front of the laser scanner. Both methods are used in this project. The first method is
used when lifting the block. This method is not as precise as the second one.
The first method reduces the rotation of the blocks, which improves the second
method, since the biggest errors appear when the rotation is large. A combination of
the two methods therefore give the most precise results.

5.1
find_cluster
A cluster in this report is defined as a group of points, where it is possible to go from
point to point by not moving more than a certain distance at a time, and still reach all
points in that group.
The find cluster function takes five inputs. The first two inputs are the x and y
coordinates of all the points from the laser scan. The third input is indexes of all points
that are within a desired distance. This desired distance defines the space of which the
function will find the clusters. This input will be called close data in the rest of the
description of the function. The fourth input is the minimum size that a cluster must
have and the last input is the max distance that is allowed to be between points in a
cluster.
The function works by having 3 for-loops. The first for-loop runs through all the data
points in close data. If a point in close data is in a list of failed clusters or established
clusters, then that point is skipped. If the point is not skipped then it is used as the
first point in a new cluster.
The next two for-loops then checks if the new cluster actually is a cluster.
The
function first checks if any of the points in close data are within the max distance from
the chosen point. If some other points are within the max distance and not already in a
cluster, they are added to the cluster. If the cluster have been expanded, the new points
are compared in the same way. This process is repeated until new points are no longer
added to the cluster.
The new cluster is then checked to see if it has a minimum size. If the cluster is large
enough, it is saved in a list of clusters. Otherwise, it is saved in a list of not clusters.
These lists are then used by the first for-loop which continues until all points of close
data have been checked.

5.4
Drive commands
Two functions were made in order to more easily use the MRC commands in the Python
framework. The first function is called evaluate_pos and it returns the current x, y and
theta values of the iRobot. The second function is called move_to_pos and it moves the
iRobot in a direct line to a target position by a line created from the iRobot’s current
position to the targeted position.
==PAGE:82==
68
5 Navigation
5.4.1
evaluate_pos
The MRC commands do have their own functions which can return x, y and theta.
The problem is that these commands are called via a socket, which also sends other
information. The evaluate_pos command is therefore a command which sorts the x, y
and theta information from the other information.
The way this function does this is by first asking the iRobot to evaluate its x, y and
theta coordinates via the MRC commands. The function then receives all the data from
the MRC socket and splits it into the individual messages. Each message is checked if
it can be converted into a float, since the other messages are text messages. This means
that only the answer to the evaluation of x, y and theta can be converted to floats.
The answers are also received in the order that the evaluations are asked, which is
how the function tells the different floats apart. The function will always ask for x, y
and theta in that order, and it will therefore receive them in the same order.
If all three floats have not been received the first time the function checks the socket
the function will receive everything from the socket again. It repeats this process until
all three values have been received. Once all three values have been received the function
returns the x, y and theta of the iRobot.
5.4.2
move_to_pos
The move_to_pos function takes a target point as its input. The target point must be
in the iRobot’s coordinate system. The function then uses the evaluate_pos function
mentioned above to get the iRobot’s current position.
The slope of a line from a point to a second point can be calculated using the following
formula:
a = ytarget −ycurrent
xtarget −xcurrent
This formula is used to calculate the slope of the line from the point that the iRobot is
standing in to the target point. The slope can then be converted to an angle by taking
the arc tangent of the slope of the line.
The angle is then converted from radians to degrees, and the target and current
position is used to check what direction the line needs to have. If the target position is
behind the current position, the angle is changed by 180° in the direction of the target
position. The current angle of the iRobot is subtracted from the angel from the line,
such that the iRobot only turns the amount it is missing to turn towards the target
position.
The angle is then checked if it is above 180° or below -180°, and 360 is either added
or subtracted if that is the case. This is to ensure that the iRobot always turns the
==PAGE:83==
5.5 Decision making of the iRobot
69
shortest possible distance towards the target position.
The turn angle is then divided into multiple smaller turn angles, which are all below
50° but adds up to the same total turn angle. This is because the iRobot hits a resonance
frequency when turning, but splitting the turns up in smaller turns causes the iRobot to
slightly slow down and speed up, breaking the resonance frequency. The function then
sends a turn command to the iRobot for each turn angle. This causes the iRobot to
align with the line from its current position to the targeted position.
The distance from the iRobot’s current position to the targeted position is then
calculated and a drive command is sent to the iRobot with the line drawn. The stop
condition is set such that the MRC will continue when the iRobot have driven the
distance from the iRobot’s current position to the targeted position minus 0.5 m. The
last 0.5 m is then driven with a fwd command. This is because the fwd command have
a distance controller, which causes the iRobot to more precisely hit the desired distance,
whereas the drive command has tendency to overshoot the targeted position.
5.5
Decision making of the iRobot
The iRobot uses a state machine to decide what actions to take during the runs. The
path planning of the iRobot ensures that it does not drive into anything.
5.5.1
Path planning
The path planning of the iRobot is simple due to the assumption of the arena being
empty. The iRobot does not need to avoid anything else than the wall and the stacks.
The iRobot avoids the stacks by driving backwards before turning and driving towards
the wall. The same cannot be done when driving away from the wall, since the iRobot
is parallel to the wall after placing a block. The iRobot instead turns 10° right, drives
20 cm forward, turns another 10° right, drives 40 cm forward, and turns 35° right. This
ensures that the iRobot gets away from the wall without touching it.
The only decision the iRobot makes in its path planning is whether to drive to the
green stack or the red stack after placing a block. To determine this the iRobot looks
at its block array and checks what block should be placed in the earliest gap.

CHAPTER 6
Wall building
This chapter describes how the iRobot keeps track of which block to place and where
to place it, as well as, how the iRobot ensures that the blocks are placed as precise as
possible.
6.1
The first block
The first block is the most important block to place, since the next blocks are placed as
an extension to the current wall. If the first block is placed at an angle the entire wall
will be placed at an angle, since the placement of the next blocks all build on each other.
The first block is unfortunately also the only block where there is no block to align
with. This means that the placement of the first block is based entirely on the odometry
of the iRobot. The iRobot simply drives back to coordinate 0,0 and turns the opposite
way of what it started. The iRobot then places the block down directly in front of itself.
6.2
Block array
The iRobot uses two 2D arrays to keep track of the placements of the blocks. The first
array consists of ones and zeros, where a one represents a placed block. This array is
called wall array.
The second array consists of one letter strings: It is called placed array. Placed array
is filled from the beginning and it is placed array which determines what order the iRobot
is going to build the wall in. A ”g” in the placed array symbolizes a green block, and
an ”r” a red block. For example if the first element of the first row in placed array is ”g”
the iRobot will place a green block as the corner of the wall.
The iRobot can compare these two arrays to determine what blocks to lift, and where
to place them.
6.2.1
Calculation of the placement of the block
The calculation of the placement of a block is done in two stages. First a function is
used, which calculates at which index the next block should be placed. This function
==PAGE:88==
74
6 Wall building
can also calculate at which index a block of a specific color should be placed.
In the next stage a function is used to calculate how far down and high up the block
should be placed in the wall. This function uses the indexes from the first function. The
second function counts how many red and green blocks that are in placed array between
the start of the wall and the block to be placed. It then adds the length of these blocks
and the gaps between them together. This number is how far the iRobot needs to drive
along the wall, in order to place it’s current block. The second function also returns the
height of which the block is to be placed in. The calculation of this is the row number
that the block has times the height of a block.
6.2.2
Validation of the block spot
The two functions described in section 6.2.1 only returns the position of where a certain
block should be placed. It does not check if it actually is possible to place a block that
position. If multiple blocks of the same color are placed in a row then there would be
holes in the wall, which could result in the iRobot placing a block on top of a not yet
placed block. This block would therefore be dropped, and that would ruin the run when
the iRobot would try to place the missing block in the hole.
The iRobot therefore needs to check if it can actually place a block before placing
it. It does this by checking if the blocks below the block that it is trying to place have
been placed. The way this is calculated is by calculating intervals of placed blocks, and
comparing it to the interval of the block to be placed. An interval is in this context two
points where all the blocks between the points have been placed. For example, if an
entire layer have been placed then the interval will be the 0 and the length of the wall.
The function then checks if the lowest number in the interval of the block to be
placed is higher than the lowest number of an interval in the layer, and if the highest
number in the interval of the block to be placed is lower than the highest number of that
same interval. If both are true at the same time then it means that the entire interval of
the block to be placed is placed within another interval. This guarantees that the block
to be placed will be placed on top of other blocks.

6.3.1
Resetting the odometry
The odometry of the iRobot drifts a little all the time when the iRobot drives around.
It will therefore get more and more wrong the longer the iRobot drives. This can be
solved by resetting it to a point of reference.
The only point of reference in the arena is the wall, since the stacks change size when
the iRobot lift blocks from them. The iRobot therefore uses the wall to as reference
point, and more precisely the corner of the wall.
The iRobot uses the distance to the left corner of the wall to calculate where it
actually are. The iRobot then changes its odometry such that it matches the actual
position instead of old one. This means that the iRobot resets its odometry each time
it returns to the wall.
6.3.2

The angle of the wall is measured using the same method as when aligning the blocks
described in Section 4.4.2. The iRobot saves the angle value that it obtains from the
method. The iRobot adds this value to its 90° turn when it later turns to be parallel
to the wall. This way the iRobot adjusts its angle such that it is parallel to the wall in
case it did not get back exactly perpendicular to the wall.

there are also no turns.

The iRobot uses the y distance to determine how far to move the arm to place the
blocks. The iRobot uses the laser to obtain the y distance and the more recent these
measurements are the more precise the blocks can be placed.
The second distance is the y direction. The y distance can is obtained by looking
at the five first measurements of the laser scan. This however only work when the first
layer have been placed. The iRobot therefore uses a different method to place the first
layer. The iRobot therefore drives to the start of the block, instead of driving to the
middle of the block that is to be placed.
While doing this the iRobot constantly takes measurements of the wall, and saves any
where the five first does not reach further away than 0.8m. This limit is to ensure that
all of the five first measurements actually hits the wall. This way all the measurements,
which would hit the gaps in the wall or any measurements after the last placed block,
are discarded. The latest non discarded measurement is used to place the block in the
y direction.

Test and Results
This chapter presents the testing of three main tasks of the projects, the object detection,
the block alignment and the block placement. The fourth task, navigation, was not
tested because any imprecision in this method would be rectified by the other methods.
The range of the uncertainties in the main tasks was tested and analysed to be able to
determine the size of the combined uncertainties.
All the tests have an degree of measurement uncertainty: This uncertainty is ignored
in this chapter, because the uncertainty was not possible to quantify. The iRobot’s
ability to the place the blocks are in reality better than what is stated in the chapter,
because the measurement uncertainty was not subtracted from the results.
7.1
Precision of the object detection
The variables that affect the precision of the object detection are its ability to detect the
objects, if it classifies the detected objects correctly and how well it places the bounding
boxes on the objects.
In this project the test and results will only be based on three neural networks for
object detection, because only three out of the six are currently used. The first is the

7.4
Gap size
The uncertainty of the block placement in the wall is outlined in section 7.3.4 where the
gap size must be at least 2 cm to ensure that two blocks would not overlap. There is
additionally, the uncertainty of the lifting of the block, which was calculated in section
7.1.1, where the gap needed to be at least 4.6 cm in order to ensure that two blocks
would not overlap. When combining these two uncertainties it results in a hypothetical
gap size of at least 6.6 cm. The uncertainty of the angle was found in section 7.2.1
where the gap size needed to be at least 0.76 cm in order to ensure that two blocks
would not overlap. When combining this uncertainty with the two others it results it
the hypothetical gap needing to be at least 7.36 cm in order to ensure that two blocks
will not overlap and to ensure that the run will not fail because of this.
The uncertainty in the y direction is small enough that the iRobot could build the
wall as high as it can reach without risk of the wall collapsing due to unevenness. Since
the wall is only one block deep in the y direction the uncertainty in the y direction is
less important since there will not occur a block overlap in that direction.
It is possible to make customizable gap distances for each block color, since the
uncertainties for the green blocks are lower than for the red blocks. This would have to
be recalculated three times if the gap was between two red blocks, a green block and a
red block or two red blocks.
The entire concept of gap size could be avoided if there was some kind of feedback
when placing the blocks, because the gap size is only there to circumnavigate the un-
certainties. An example of this type of feedback could be a force meter attached to the
arm, such that the iRobot could sense when it touches the former placed block.

and it will be improved when the linear error is taken into account and the hypothetical

CHAPTER 8
Future work
This chapter contains suggestions about which parts could be improved and which parts
are missing in order to win the challenge 2 in MBZIRC in 2020.
8.1
Object detection
Figure 7.4 shows that the new neural network for lifting blocks does not always find
all the blocks in the image. This is due to the different training parameters between
the two sets and due to the old neural network having a larger data-set than the new
neural network. The new neural network can be improved by finding the right training
parameters and increasing the size of its data-set.
Figure 7.5 indicates that the neural network for finding stacks has some errors where
it classifies the stacks wrongly or does not detect a stack. This can also be improved
by finding the right training parameters for this network and increasing the size of the
data-set. This is one of the difficult neural networks to train since the working area of
the stacks can vary and the objects can change in shape depending on how the blocks are
placed in the stacks. It is important to have a bigger data-set for this neural network.
The other neural networks have not been tested to the same extent, because they
have not been implemented yet. Further tests will have to be made to make sure it is
consistent enough for the challenge. These neural networks need a larger data-set to
ensure their consistency.
8.2
Aligning blocks using the camera
Rotating the blocks, using the camera during the lift, to align the blocks with the iRobot,
has errors which are pointed out in section 7.2.1. The camera turning method needs to
be improved and tested further to make it possible for it to stand alone and be used
without the laser scanner. The linear tendency of the errors will have to be investigated
to identify what causes the linear tendency.
==PAGE:122==

8 Future work
8.3
Drones
For this project the focus was on the UGV of the challenge and making it possible for
the UGV to solve the challenge by itself. This was done by simplifying the tasks and
there were no drones. The methods for collaboration with the drones have not been
implemented yet. The current methods have been designed in a way that the drones
can be incorporated at a later stage.
8.3.1
Avoiding drones
The methods described in sections 2.8.2 and 2.8.3, for avoiding the fallen drones and
avoiding collisions with drones, still need to be implemented. The iRobot needs to be
able to avoid drones that have fallen and are obstacles in the working area. In addition,
the iRobot needs to be able to avoid drones actively engaged in completing their assigned
tasks.
8.3.2
Updating the wall
The neural network for the iRobot to update the known wall has been made ready for
this project although this has not been implemented. The iRobot has to be able to
detect changes to the wall made by the drones, in order to collaborate with the drones.
8.4
Block trunk
The block trunk has not been added to the iRobot and this limits the iRobot to carry
one block at a time.
However, considerations and choices toward designing scalable
methods for lifting and placing blocks were made.
8.5
Additional types of blocks
The blue and the orange blocks mentioned in section 2.2.2 have not been manufactured
and therefore are not part of the solution. The blocks will have to be manufactured and
included into the data-sets of the neural networks in order for the neural networks to be
able to predict them. A solution of how to lift the blocks is needed, because the blocks
are too big to fit inside the image, see section 2.5.3.1.
8.6
Lines
Even though the description of the challenge does not mention anything about a line
there is still an assumption that the wall will be built on an L shaped line, refer to

section 2.2.1 for more information. The methods for building the wall on the line have
not been implemented, but the neural networks to detect the line are made and are
ready for implementation, see 3.1.3.3.
8.7
Optimising the run
The speed of the iRobot needs to be increased. This is not currently possible due to the
safety regulations on the wheels’ regulator. The safety regulations have to be changed
to allow the iRobot to drive faster. The sleeps of the program need to be refined to
reduce excessive waiting time. A way to refine the sleeps could be to receive feedback
from the MRC communicating with the python program. A procedure to decipher the
information from the MRC would have to be implemented if this approach is chosen.

CHAPTER 9
Conclusion
The iRobot can build a wall of 8 blocks successfully and with the possibility of building
a larger wall because the iRobot’s inputs are adjustable. The iRobot needs to be within
the starting area facing the stacks to successfully initiate the construction of the wall,
once that criteria is fulfilled the iRobot can find the stacks, lift the blocks and build the
wall autonomously.
The iRobot is capable of detecting objects in its immediate environment, and change
its behavior to changes in its surroundings. Deep learning through the neural networks
allows the iRobot to dynamically change to its environment.
The iRobot can do the individual tasks consistently even if the gap size is too small,
because several errors cancel each other. The errors have been diagnosed and can be
solved in the future. A solution to reduce the errors has been proposed, in order to make
the run more consistent. The solution suggests that the gap size needs to be at least
7.36 cm to ensure that two blocks will not overlap and ruin the run.
The solutions of the individual tasks are made scalable, in the context of lifting
and placing multiple blocks at a time. These solutions are not necessarily the fastest
solutions for placing a single block, but they are faster overall if the iRobot is able to
carry multiple blocks.
Collaboration with the drones will be easy to incorporate because the iRobot already
has the required neural networks in place. It would be easy to update the iRobot’s code
so it can build the wall anywhere in the arena. The neural networks for identifying the
line exists such that the wall can be built on a line.
