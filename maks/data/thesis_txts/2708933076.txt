Summary
The goal of this thesis is to help software developers create scalable, robust and secure microservice
architecture based software systems. Developers working in the industry are often pressured to deliver
new features faster due to lack of resources or the need of a shorter time to market, resulting in a
neglection of good software practices.
This neglection leads to technical debt and in the end, the
company will have to pay for that. This problem can be mitigated by choosing the right software
architecture, following specific development policies and using state of the art software technologies.
This thesis introduces a literature study revolving existing trends and challenges in microservice
based software systems, which results in a framework called Exemplar. Exemplar creates software
projects with a foundation that allows for a scalable, robust and secure microservice architecture based
software system. Exemplar is realized as a web application that developers can use to generate projects
linked to their own Github account, which includes a frontend, a backend consisting of multiple mi-
croservices, an authentication mechanism, message queue and deployment files to make the application
ready for deployment upon generation. All this functionality provided alongside a guide on how to
further extend the generated project, enhances the development process of scalable, robust and secure
microservice architecture based software systems.

CHAPTER 1
Introduction
For a software developer, setting up a new project from scratch can be a diÔ¨Äicult task. The developer
has to come up with a good overall design and architecture and choose which technology stacks to
use. This could be what programming languages to use, which cloud providers to pick and what design
patterns to follow. When given complete freedom, it is often really exciting for a developer to have the
flexibility to choose the preferred technologies. However, for real-world projects, this freedom is often
not given to the developer, as there is most likely a budget for the project as well as a deadline for
when the software needs to be production ready. The developer has to do trade-offs between how much
resources to spend on developing scalable, robust and secure code vs the time and money it takes to
get the software ready for production. In the paper, Exploration of Technical Debt in Start-ups [9, page
83], Eriks Klotins et al. research how prevalent technical debt is in startups by applying a case survey
method to identify aspects of technical debt in start-ups. One of their conclusions were that:
[...] even though start-up engineers realize the importance of good engineering practices,
they cut corners in product engineering, mostly due to resource pressure and a need for faster
time to market.
The choice of technologies, frameworks and initial setup of a software project help mitigate these
challenges. Mitigating these challenges are important as E. Klotins et al. state: ‚ÄùAs a result of these
challenges, most start-ups do not survive the first few years of operation and cease to exist before
delivering any value‚Äù [9, page 75]. This decision is especially diÔ¨Äicult for early stage startups where the
budget and the amount of experience in the team is limited. For a software application reaching end
customers1, there are a lot of different designs, architectures and technologies available. To properly
assess how many resources to spend on building the software application, it is important to define what
the characteristics of a high quality application should be. This is a very opinionated discussion and
is influenced by individual preference and experience of the software developer as well as the context
the software application lives in. However, generally speaking, companies want their applications to be
robust and properly scale as their business grows. I.e., when the amount of traÔ¨Äic the application is
exposed to increases, the software should continue to work and be easy to maintain and extend with
new features by different teams. Thus, the term scalability for software architectures can refer both to
the degree of which a software application can scale to endure increasing amount of traÔ¨Äic [12] but it
can also be referred to in a project context for how well teams or developers are able to maintain and
extend the project as it increases in size [16].
Different design patterns, frameworks and software architectures help with some of the issues ad-
dressed here. For example, a microservice architecture (MSA) helps projects being scalable by allowing
multiple teams working in parallel on the specific software project. Containerization, auto-scaling and
load balancing help solutions becoming more scalable by adding resources dynamically when an appli-
cation experience a high amount of user traÔ¨Äic. DevOps techniques such as automatic deployment and
testing pipelines reduce the time it takes for a developer to push their code to a production environment
and to do so in a safe and consistent manner. However, these different principles and techniques take
time to setup and to get familiar with. It is often these principles that are ignored or compromised

when the budget is limited and the deadline for time to market is near. This can become an issue for
software projects, since when the projects become larger, not following these principles, might result in
significant costs for the project in the form of errors, down-time and customer dissatisfaction. When the
project is already large, implementing these designs and principles becomes increasingly more diÔ¨Äicult.
With our solution, a framework called Exemplar, we want to help developers mitigate these diÔ¨Äiculties.
We want to help developers implement good design principles and software architectures from the very

Specifically, Exemplar helps developers by generating a complete, working web application with
a dedicated backend consisting of multiple microservices2. The use cases for developers in different
settings vary a lot and therefore we provide working but limited amount of code. The generated code
is enough to get the developer started with the project, which can then be extended with features
for the specific software project the developer wants to create. The generated project has a modern
architecture with a separated frontend and backend. Furthermore, the backend consists of a number
of microservices communicating with each other through a message queue, each in its own dedicated
git repository. The code generated for the user is pushed to the user‚Äôs own Github account3. Not only
is the code working locally on the user‚Äôs machine but deployment pipelines is setup across the entire
generated software project, such that when utilizing Github for version control, the web application is
automatically deployed and accessible in the browser with a domain. In order for this to work, we have
setup the necessary integration with one or more cloud providers. To further ensure that the software
project remains scalable and robust, we have provided examples of how to write unit, integration and
end-to-end tests across the entire software project. These tests are executed through testing pipelines
that we have created, such that before the project can be deployed, these tests are run.
As mentioned, the user is able to run the generated project from the very beginning. In order to
showcase the flow through the entire application, a login feature is generated where the user can sign
in to their generated application. This means that the generated backend also have the necessary code
to handle user authentication and authorization.
The generated project serves as a good foundation as it allows different developers to work in parallel
with different parts of the project, and it helps ensure that the project is scalable, robust and secure.
One of the challenges for the user is to properly grasp and comprehend the project generated for them,
such that they can maintain the code and extend it with the desired features for their project. To
mitigate this challenge, we have provided a dedicated documentation page, that explains all the details
of the generated project and guides the user on how the project can be extended with new features.

RQ1: What are the existing trends in building, deploying and operating microservice based

RQ2: What are the biggest challenges software developers have in terms of developing and

RQ3: Is it feasible to create a framework that eases the development of scalable, robust and
secure microservice based software systems?

based software systems and how does the solution compare to existing methods?

1.2
Methodology
This thesis followed the research methodology of design science research [8]. The methodology specifies
how scientific research projects should be conducted and what the outcome of the project should be.
Thus, a study of and the creation of an artifact was conducted throughout the duration of this thesis
with the intention of solving a practical problem. In the case of this thesis, the artifact is a framework
that helps developers build scalable, robust and secure software applications. Throughout the project,
the different phases of design science project were carried out and the results are presented in this thesis.
The first phase is to explicate the problem. Here, we investigate and analyze the problem itself. First,
we design research questions relevant to the identified problem, which serve as an overall guidance of
the project. This phase also includes a justification of the relevance of problem, where a literature
study of related work and existing knowledge about the problem is conducted. From this knowledge,
we bridge the gap between problem and solution by translating the relevant problems into specific
requirements for the artifact. This results in a feature set for the artifact which outlines its behavior
and functions. Then, we design and develop the artifact. This process follows an agile and iterative
approach and good practices from agile software development are followed4. This will be elaborated
a bit later in this chapter. After the design and development phase, we demonstrate the artifact by
showing and explaining its design and functionality. This includes an explanation of how to use the
developed artifact. The last phase of the design science methodology is an evaluation of the developed
artifact. Here, we assess the success of the project both in terms of the developed artifact and the
project as a whole. This includes relating and comparing the developed artifact with existing solutions.
Furthermore, we revisit the research questions and summarize the answers found in this thesis. We
also evaluate how the artifact itself and the knowledge gained throughout the project contributes to the
existing scientific knowledge base on the topic of this thesis. Finally, we discuss which potential next
steps that could be carried out to further enhance the solution, which concludes all of the steps in the
design science methodology.
For the third phase of design science, designing and developing the artifact, an agile methodology has
been followed. Specifically, the Scrum framework has been used. In the book chapter Agile Methodology
[11, page 301], Gerard O‚ÄôRegan describes how a product should be built following the Scrum framework:
The main deliverable produced using the Scrum framework is the product itself, and Scrum
expects to build a properly tested product increment (in a shippable state) at the end of each
sprint
In the context of Scrum the deliverable corresponds to the developed product and in the context
of design science it corresponds to the developed artifact. The two terms are used interchangeably in
this thesis. As stated by Gerard O‚ÄôRegan in his book [11], Scrum enforces an iterative development
process by having sprints of 2-4 weeks, daily stand ups, the use of Kanban boards5 and more. Scrum
as a framework can be used alone, but in this thesis we used it along with an overall project plan with
milestones and a risk register. It might be counter-intuitive to both follow the Scrum methodology
and having a project plan, but in this context the project plan with milestones was used as a guide for
which direction the thesis should be heading. Using Scrum to iteratively develop the product in sprints
of 2-4 weeks meant that throughout the entire development phase, our product was close to a state
where it could be released. It also meant that we could continuously get feedback on the product even
during the development phase. The milestone plan we created to accompany the Scrum framework
helped guide the project in the right direction and each milestone corresponded to an end date of a
Scrum sprint. As milestone deadlines came closer, the tasks of the milestones were defined in more
detail. Milestones further out in the future, were more broadly defined to ensure that the project was

1.3
Report Structure
In order to guide the reader and to understand in which direction the thesis is headed, a report structure
is presented.
In Chapter 2, we present the technologies used to develop Exemplar, the framework described in
RQ3 and RQ4, and we introduce relevant tools, technologies and frameworks in relation to RQ1. The
intention is to provide a context such that the reader is familiar with the terms used in the rest of this
thesis.
In Chapter 3, we argue for the relevance of the problem by introducing several papers on the topic.
This includes papers on some of the issues companies have when building software applications and
specifically the challenges of MSAs.
In Chapter 4, we outline tools and technologies that are similar to Exemplar and make a summary
of their pros and cons and how they compare to Exemplar.

2.2
Architectures
In this section we explain the two fundamental architectures for designing software applications. First,
we outline the traditional way of designing applications, namely with a monolithic architecture. Sec-
ondly, we introduce MSAs, which we see has both benefits and drawbacks compared to traditional
methods. In the paper The Comparison of Microservice and Monolithic Architecture[7, page 150], K.
Gos et al. explains the importance of software architectures with the following statement:
Software architecture is the fundamental structure of a software, that defines technical and
operational requirements. It is responsible for optimization of each attribute of an application,
like eÔ¨Äiciency, manageability, scalability, reliability, modifiability, deployability and more
other aspects.
That is why the choice of appropriate architecture is so important in the
primary phase of software development.
In other words, picking the correct software architecture for an application is crucial, as it can have long-
term effects on an application and companies as a whole. Essentially, choosing the wrong architecture
from the beginning can end up costing companies a lot of money, since it takes a significant amount of
time refactoring from one architecture to another.
2.2.1
Monolithic Architecture
A monolithic architecture is an architecture of a software application in which its components and
modules cannot be deployed independently. Each component in a monolithic architecture is tightly
coupled which makes extending the application more diÔ¨Äicult. If someone would change a piece of the
application, to accommodate a feature request, or if parts of the application is under heavy load, one
would have to deploy a whole new instance of the application[5].
A simple analogy would be to change the tyres of a car. For a monolithic application this would
mean replacing the entire car. From a technical point of view, it seems like an easy way to change the
tyres, because the tyres are not even changed, one just gets a new car with new tyres. But from an
eÔ¨Äiciency point of view, having to change the car every time it needs a modification, is suboptimal. In
a monolithic application, if one would want to extend a part of the application or increase its resources,
it is required to redeploy the whole application. If a lot of people are using a specific service of the
application, and that part becomes slow, it is wanted to add resources to ease the pressure on that
specific service, but the only way to do that is to create a new instance of the whole application, and
split the load between the old and new instance of the application.
From a developers point of view, a monolithic application is easy to setup and usually easy to
manage, but once the application grows in size and many dependencies are introduced, tracking down

bugs and understanding the functionality of the various parts of the application becomes diÔ¨Äicult. This
is also called dependency hell[5]. The dependency hell can be categorized as a certain state of a software
application where the application relies on many different software packages in order for it to run, and
these software packages can also rely on various versions of other packages. This often results in the
developer losing control and especially in a big application with many dependencies, it is hard to track
where exactly an error originates from. One of the benefits of using MSAs is to avoid this issue.
2.2.2
Microservice Architecture
An MSA is an architecture related to software applications in which its components and modules,
called services from now on, can be executed, developed and deployed independent of each other. An
MSA is composed of a set of microservices, which are independent components running their own
processes, that communicate with each other through messages. The MSA shares similarities to service-
oriented architecture (SOA), because both architectures try to break large applications into small,
flexible components[4]. Microservice architecture distinguishes itself because it focuses on decoupling
components, not using shared databases and this results in an architecture that is more scalable[3].
The communication between the services should be technologically agnostic, e.g. if service A relies
on an authentication service that is developed in a specific programming language, and later on said
authentication service is developed in a different programming language, service A should not have
to change anything in order for it to send and receive messages to the authentication service. The
services should be decoupled from each other, such that service A has no idea how the authentication
service is developed and whether it changed or not. This can be obtained with message queues such as
RabbitMQ3 or Redis4 as examples, or even omitting message queues and using HTTP as a means of
communication. A key feature of the message queue as a protocol is, that it is asynchrounous (whereas
HTTP is syncronous). This means that applications acting as producers can publish messages to the
queues, while application acting as consumers, can listen to the queue can receive messages.5
From a developer‚Äôs point of view, developing a functionality tied to a specific service that is small
is desirable. That is because it limits the scope of the development process and maintaining that piece
of code, as well as debugging it is easier, because there is less code to maintain.
It also gives the
developer the freedom to chose the framework and technology of choice, since the internal composition
of a microservice does not matter, since its interaction with other services are through messages or an
API. Looking at Figure 2.3, each of the services such as FDWDORJ, FDUW, RUGHU and SD\PHQW are developed
and deployed independently. They could be written in four different programming languages using
four different database technologies etc. Besides that, an MSA also allows for features such that if the
payment service is down, the rest of the application still works and thereby, if implemented correctly, it
means that you can develop fault-tolerant software systems [14]. In a monolithic application, a scenario
where the payment service is down, could mean that the whole application stops working. Using the
EA in Figure 2.1, the IURQWHQG repository is equivalent to GHVNWRS EURZVHUV in Figure 2.3, while PV and
DSL could be one of FDWDORJ and FDUW for example.

2.3
Development Process of Microservices
As mentioned, microservices can be developed and deployed independent of each other. This means
that in software projects using microservices, there are typically more deployments than other types
of software projects, which introduces a new way of developing software applications. The operations
aspect of a software project using microservices is also different, because even though the microservices
are deployed independent of each other, they still need to be able to either communicate with each
other, or at least communicate with a common message queue, that distributes the communication. A
suitable development process is to use an agile approach, meaning developing features in sprints and
releasing them often.
2.3.1
DevOps
By adhering to an agile approach, setting up a development process to secure high quality and continuous
delivery of software is important, which leads us to the concept of DevOps. DevOps is the practice
of combining development (Dev) and IT operations (Ops). The core idea is to setup pipelines, which
are a set of automated processes and tools that allows both developers and operations professionals to
work cohesively to build and deploy code to a production environment. Typical pipelines could build
your application and run a set of automated tests, but in general DevOps touches upon the following
categories6:
‚Ä¢ Planning, e.g. use agile practices to improve the development cycle and ensure frequent release
and a quality product
‚Ä¢ Building, e.g. optimizing the code for a production build
‚Ä¢ Testing, e.g. testing the application
‚Ä¢ CI/CD, e.g. releasing the products frequently and predictably. This should be automated includ-
ing tests that ensure the released product is robust.
‚Ä¢ Operating, e.g. configure, deploy, maintenance of all IT infrastructure.
‚Ä¢ Monitoring and alerts, e.g. monitoring the production environment and making sure to alert if
the application is not behaving as expected
All of this contributes to having an application which can be continuously tested, integrated and de-
ployed.

Automated Testing
With automated testing we ensure that applications function exactly as intended and continue to do
so when the application experience higher loads of traÔ¨Äic. This is important for developing scalable
applications and thus is an important topic to fully understand. In this section, we go through the
different types of automated tests as well as some important technologies and principles for developing
automated tests.
2.5.1
Types of Automated Tests
Typically, automated tests are categorized into three different types of tests; End-to-end (e2e), integra-
tion and unit tests. The different types of tests works on different levels of the code.
E2e tests aim to test the entire application across its different components, classes and methods.
Usually, only few e2e tests are written, since they take longer to both write and execute. For applications
with a GUI, e2e tests are typically UI tests, that through an automated manner interact with the
application through the UI and thereafter verifies that certain actions occurred or elements appeared.
These tests could be written with UI testing tools like Selenium, which is explained in the following
section. For API services without a GUI, e2e can also just be automated tests that test the entire API
service across its different components. This could be written with Postman Automation, which is also
explained later in this section.
Integration tests aim to test smaller parts of the code than e2e tests, but still across different
methods or classes. Generally, integration tests is a loosely defined term, since it can have different
meanings depending on the size of the application, the amount of components in the application etc.
For a monolithic application, integration tests can refer to testing across methods. For an application
with multiple microservices, integration tests can verify that an entire microservice works properly. In
this report, the latter definition of an integration test is used.
I.e. tests that e.g. validates that a
microservice, given certain inputs, responds to those inputs properly or that its state is updated as
expected. Typically, more integration tests are written than e2e tests but it is still a limited amount,
since they also take longer to execute.
Unit tests aim to test small pieces of code typically limited to one or two methods. Unit tests usually
take up the vast majority of automated tests in a project. They take less time to write and typically
execute very fast.

2.5.4
Dependency Injection
The vast majority of automated tests written should ideally be unit tests that tests small parts of the
code because they are executed quickly. This also means that external services should not be included
in the unit tests, since waiting for these services would cause the tests to execute slowly. How do we
avoid testing certain external services? Code is often deeply integrated and dependent on the services
that it communicates with. This could be a database, an external library etc. This is where the concept
of dependency injection becomes important. With dependency injection we aim to decouple the classes
we write from the external services by injecting them into the classes instead of having them as a default
part of the class. Concretely, instead of creating the external service inside of the class, it is instead
injected into the class either through the constructor or a method as an argument. That way, when we
want to test a specific method in the class, we can substitute the injected service with a so called mock
object. The mock object is just an object with the same type as the external service, but where the
implementation is mocked i.e. customized for the specific test.
With NestJS, the code code is structured to allow for dependency injection which makes testing a
lot easier. As explained in the section on NestJS, code is structured into separate components which
usually represents features or resources. For each component there is a module file which specify all of
the resources the component needs. In these module files we also specify external services. This could
be a module reference to a separate component, an HTTP service allowing the class to perform API
requests or a database service. When initializing the classes of the particular component, all of the
external services defined in the module will be injected into these classes through the constructor of
that class. This makes testing a lot easier as we see in the following example.

As explained, one of the key advantages to the MSA over a monolithic architecture is, that the software
is split into different smaller components that can live in isolation and therefore be worked on without

operating system is compatible, doing network configuration that allows the services to communicate

2.6.1
Docker
Containerization is the packaging of software code with the operating system, libraries and dependencies
into a single executable called a container.
This container can run consistently on any operating
system15.
The most popular tool for creating and running containers is Docker.
In a report from
Flexera[6] from 2021 it was found that 74% of companies either use or plan to use Docker.
With
docker, you write the configuration for the container in a Dockerfile. The Dockerfile is then used to
build a Docker Image, which is a blueprint for creating containers with the configurations specified in
the Dockerfile. You can then create containers from the built image and if you want the container to
communicate with external components, you can open ports for that container. As seen in Figure 2.16,

issues experienced when wanting to move code from one location to another, for example from a local
environment to a virtual machine. This technology works especially well with MSAs where a software
project can consist of many small components each being completely isolated from the rest.

2.7
Kubernetes
With docker-compose, developers are able to easily launch the entire application, that consists of mul-
tiple different containers locally on the developer‚Äôs machine. However, in a production environment,
you typically want microservices distributed over multiple different virtual or physical machines. With
docker-compose this is not possible. For production applications, other container orchestration tools
are therefore used, such as Kubernetes, which is arguably the most popular tool for handling and or-
chestrating docker containers. Tools like Kubernetes become relevant especially for larger applications
that consists of many docker containers and have requirements for up-time, performance etc. As appli-
cations grow in size and more docker containers are added, it quickly becomes very complex to manage
each individual container and the communication between them. Kubernetes helps developers manage
systems consisting of a large number of containers and in this section we explain the core concepts and
functionalities of Kubernetes.
With Kubernetes, we define a desired state for how we wish the containers of our application should
be configured and maintained. This desired state is written in \DPO files as seen in Figure 2.20. In
the figure we see the folder kubernetes with 3 yaml files located. The files DSL\DPO and DXWKBPV\DPO
define the configuration for the the two microservices DSL and DXWKBPV which were introduced earlier
and the file UHGLV\DPO define the configuration for the Redis message queue, that the two microservices
communicate through.
These configuration files are given to a component in Kubernetes, called the control pane. This is
one of the main components of Kubernetes, and is responsible for translating the configuration files
with the desired state into the actual containers and infrastructure that makes up the application - the
so called Kubernetes cluster.
When the control pane translates the yaml files it creates one or more kubernetes nodes. These nodes
are simple servers which can be either physical or virtual machines where containers can be deployed.
Each node has an important component called a kubelet, which is responsible for managing the state of
the node including managing the health of each container. If, for example a container fails, the kubelet
component of that node is responsible for restarting the container.
In the nodes (the simple servers), there can be multiple containers. These containers live inside
another Kubernetes component, called a pod.
A pod is the smallest unit of a Kubernetes cluster
and consists of one or more containers. Each pod is assigned a unique ip address, which allows the
different components of the application to communicate with each other. You specify the declarative
configuration details about the pod in the yaml files as mentioned. These files are called Kubernetes
deployments where the desired state of the pod is defined.

Figure 3.9: Concerns in the taxonomy for gains of MSA
As seen in Figure 3.9, it is interesting to note, that one of the biggest gains when it comes to MSA
is also the architecture itself (it was also one of the biggest pains). Specifically, what the developers
gain when it comes to developing software systems with a MSA are its decentralized governance, fault
tolerance and flexibility among others. Fault tolerance was also outlined earlier in the description of
Figure 2.3, because if one of the services is down, the application can still work - whereas flexibility for
developers, means that they can develop each service with their preferred technology stack.
Another gain is developing the microservices. As mentioned earlier, microservices should be techno-
logically agnostic, meaning the developer of each microservice has the freedom to choose the program-
ming language of choice. If developers for a project are specialized in different programming languages,
they could develop the microservices independently in their respective preferred language, which is also
a huge advantage to the company. Each microservice will also cover a smaller code base, making each
service easier to debug. This also means avoiding the dependency hell5 that can be present in larger
software systems, because each service has its own dependencies, and thereby the scope of where a bug
can arise is narrowed down to the dependencies of each service instead.
Regarding operations, the management and deployment of the system are one the biggest gains
when it comes to MSAs. In relation to the car analogy, if you wanted to change a tyre in a monolithic
context, you would have to change the whole car. In an MSA context, to change the tyre of a car, you
could do that, without affecting the rest of the car. When it comes to software systems, if you want
to change the authentication mechanism to e.g. use a different hashing algorithm, the authentication
service alone could be changed and redeployed, while the rest of the application lives on.

7.3
Authentication
As explained in Chapter 5, one of the features of the generated app is a generic authentication mechanism
initially available in the generated app. In this section we describe the design of this functionality. We
begin by outlining the different steps that occur when the user uses the generated app to login through
the frontend. This also give a better understanding of the architecture of the generated app as all of
the generated components are involved in the login feature.
1. User clicks login in the frontend
2. Frontend sends a 3267 request to ORJLQ with a username and password in the body
3. API receives the request and puts a message on the message queue with the message id ORJLQ
4. AuthMS listens for the message with the id ORJLQ, receives it and gets the data
5. AuthMS verifies if the username and password is correct and creates a JWT that is returned all
the way back to the frontend (through the message queue and API
The login flow showcases the events and data flow that are usually present when users interact
with the generated app through the frontend. As can be seen, the login flow is relatively simple. The
API receives a POST request from the React app, with the payload being the username and password.
The API puts a message on the message queue, with the id ORJLQ which the AuthMS consumes. The
AuthMS verifies whether the user is authenticated or not and returns the result.
Thus, the responsibility of the authentication microservice AuthMS is to handle all authentication
of the users.
Authentication is implemented with support for JWT and showcased with the login
functionality.
The initial authentication mechanism provided is simple in the sense that the token
generated has no expiration date and is generated based on only the username of the user. But as
explained in Chapter 5, the JWT authentication mechanism was also chosen because it scales well and
can be used for more advanced access control methods e.g. role based access control (RBAC). For the
concrete design of the application, this would result in the authentication microservice being extended
with functionality to support RBAC. This means generating the JWT based on a specific role of the
user and for each future request sent to the backend, the JWT is validated and the specific role is
evaluated in order to ensure that the user is allowed to access the requested resource.

8.3.1
Stateless architecture
Adding to the architectural simplicity of the backend, it has no persistent data storage and is thus
completely stateless.
This is achieved by utilizing the authentication methods of Github, which is
needed anyway in order to upload the generated code the user‚Äôs Github account. In the frontend, the
user is redirected to Github where they are prompted to approve that our backend can create and modify
public repositories in the user‚Äôs Github account. When this is approved, the user is redirected back to
the frontend with a Github code. From thereon, the frontend sends this code to the backend and the
backend makes a request to Github and receives an authentication token in exchange of the code. The
authentication token is then sent back to the frontend and is used in all future requests to backend and
is thus never saved in the backend. Encrypted in this token is the Github credentials of the user, and
the permissions that the backend needs for the user‚Äôs Github account. Storing the token or even having

8.3.2
Deployment Technologies
In this section we describe the deployment technologies used for the backend of Exemplar. The backend
has been containerized using Docker which has multiple different purposes. First of all, it enables easy
local development of both the frontend and backend with docker-compose - just like with the generated
app. Secondly, it allows for a simple e2e test setup with Github pipelines, which is explained in greater
detail later in this chapter. Finally, the containerization enables deployment of the backend using an
AWS service called AWS App Runner, as indicated in Figure 8.1. AWS App Runner is an AWS service
that integrates well with containerized services. App Runner links the service to a specified container
that is stored in the AWS Container Registry. This makes it easy to deploy new versions of the backend
to production, since it only requires that a new image is deployed to the AWS registry. This can even be
automated with a deployment pipeline as we explain in section 8.4. App Runner also enables blue-green
deployment, which means that when a new container image is deployed, and App Runner is deploying
the new image, the service is still functioning using the old version. When the deployment is completely
done, all traÔ¨Äic is switched to the new version, and the old version is discarded. Thus, downtime of the
service is avoided. Furthermore an important functionality of App Runner is that it allows for vertical
auto-scaling, meaning that the service monitors traÔ¨Äic, and automatically add or remove computation
resources depending of the amount of traÔ¨Äic. This feature has been enabled for Exemplar.

application to a cloud provider has been generated. These files were then accompanied with at guide
on how to manually deploy the application on a cloud provider.

How does this framework help developers build scalable, robust and secure microservice based

The approach to answer this question is two-tiered; first, we will identify some of the features that

discussion. Secondly, we compare Exemplar to existing, similar solutions and hereby argue for how
our solution differentiate itself from existing solutions. This leads to a discussion of how we thereby
contribute to the existing knowledge base in relation to the topic of this thesis.
First of all, it is necessary to derive what challenges developers face when it comes to building
scalable, robust and secure software applications. This was answered in RQ2. In RQ3, we concluded
that creating a framework that eases the development of secure, robust and elastic software application
is feasible. Some of the biggest challenges were the fact that there is a lack of internal resources with
expertise in this domain, and with Exemplar, we tighten the gap, by providing a framework that helps
setting up a system that mitigates what developers have identified as challenging, i.e. directly helps
them build scalable, robust and and secure software applications. The inclusion of guidelines amplifies
the help, by making sure the developers understand what happens under the hood and thereby ensure
they can maintain the project on a longer-term basis.
Based on this logic, one can also conclude that the more experience you have, the less Exemplar
can help you and vice versa. Exemplar generates an application using very opinionated architectures,
designs and technologies and therefore it is helpful for developers that are less experienced and therefore
thrives on the opinionated nature that the framework provides. This is also aligned with one of the users
from the forum posts. The user is a moderator of the NestJS subreddit and experienced in software
architecture and some of his feedback was, that ‚Äù[...] companies that are new and fresh and want to
get started somewhere, this might be useful‚Äù. Based on all this, we argue that the framework helps
developers, especially less experienced developers, or projects where the tech stack has not been chosen
yet. It does so, by providing a project with a lot of the technologies and tools that helps ensuring
scalable, robust and secure software application, already set up and ready to use.

9.3.1

First and foremost, a qualitative solution interview asking potential users about Exemplar would gain

The structured interview is where the interviewer asks interviewees a series of pre-established
questions, allowing only a limited number of response categories. Organizing and quantifying
the findings is thus generally straightforward. Structured interviews are therefore rigid as
the interviewer reads from a script and deviates from it as little as possible.
If an approach where a structured interview would have been conducted instead of a survey, there

As the quantitative analysis have been conducted and evaluated, it makes sense to focus on a more

9.3.2
Make Exemplar More Flexible
As it was deemed in the product evaluation phase, some users found Exemplar very opinionated. As
argued earlier in this chapter, this has been somewhat intentional since with the framework we wish to
guide the developer to build scalable and robust software solutions, while still allowing the solution to be
technologically agnostic. We do not wish to compromise on the framework guidelines itself4 but rather
allow for more technical flexibility by e.g. supporting multiple backends, frontends and message queue
frameworks. Furthermore we would like to provide deployment guidelines for deploying the generated
app on different cloud providers, resulting in a generated app that is cloud provider agnostic. Concretely,
in the project generation phase, there should be options to choose the frameworks in the Exemplar web
app. Some examples could be to add support for Angular for the frontend, Golang for the backend
and RabbitMQ instead of Redis as the message queue. This leads to a more generalized solution which
aligns with design science methodology followed in this thesis.
9.3.3
Autoscaling
Currently, the architecture of the generated app lays the foundation for scalabilty. Kubernetes deploy-
ment files are provided and one can manually control the resources provided for the different services,
as well as the number of replicas of each service5. Ideally, the developer should not need to worry
about manually scaling the system - the system should scale itself based on the load. I.e. if the traÔ¨Äic
increases, the system should allocate more resources. Therefore, providing a setup that defines how to
autoscale the system, is one of the features that should be added. Specifically, this could be defining
that the response time should never exceed a certain threshold and if it does, the system automatically
deploys more replicas of the services such that this threshold is respected.
9.3.4
Pipeline Optimization
DevOps pipelines are a part of the generated app, but there is still room for further optimization. For
each testing scenario, the whole system is built. First and foremost, one could optimize the pipelines
by building the system on one machine and pushing the images to a container registry. Each testing
pipeline would pull the images and launch the containers, resulting in less resources spent, since the
images are not built for each testing scenario but only once. Secondly, to optimize the testing pipelines,
we could enable that only the component that was updated would be rebuilt instead of building the
entire system in each testing scenario [2]. Concretely, when a developer works on a new feature for a
specific microservice and they want to deploy the updated microservice, only that microservice should
be deployed. The rest of the services are the same, hence building the whole system again is a waste.

CHAPTER 10
Conclusion
This thesis has shown that creating a framework that eases the development of scalable, robust and
secure microservice applications is feasible. The framework, Exemplar, was developed by taking different
well-known technologies and tools, and using these together to create an application that generates a
software project for the user. Exemplar is also used to guide them through the further development
process, ensuring their application allows for scalability on a long-term basis. Specifically, it provides
the user with an microservice architecture based software application with a dedicated frontend. The
microservices internal communication between each other is facilitated by a message queue protocol.
Exemplar helps ensuring that the user‚Äôs application is robust, scalable and secure by design, as long
as the user follows the provided development policies. A project is generated with multiple example tests
provided; unit tests, integration tests and end-to-end tests. These tests are executed using pipelines
with various triggers, i.e. upon opening a pull request to the main branch. Forcing the users to follow
the defined development policies, helps the user to keep their application robust. A large volume of
requests can stress even a robust application and therefore deployment files are provided, to deploy
the application in a Kubernetes cluster. This includes a guide specifically to deploy it to AWS, taking
the user from not having a project, to generate a robust and scalable project, ready to be deployed
straight away. This includes integration with Github Pages for the frontend, completing the primary
functionality that Exemplar provides.
Building a framework as Exemplar is only relevant, if there is a problem to solve. Through literature
research and a focus on what challenges developers face when it comes to develop cloud applications and
microservice architecture based software systems, the feature set and relevance of Exemplar was justified.
Both when it comes to cloud and microservice architectures, the lack of internal resources with expertise
is a big challenge. Other challenges worth noting are the testing of microservice apps, specifically across
multiple microservices. The operations, i.e. managing and deploying microservice apps were also one
of the biggest challenges. Thus, the feature set of Exemplar has been created in order to mitigate
these challenges for the developer. With Exemplar, we tried to reach a large user group by making
the solution as general as possible. This is also why AWS, Kubernetes and more were deemed as the
proper technologies to mitigate these challenges, because they are also trendy and in-demand.There are
still ways to go in terms of generalization as the framework currently only supports limited options of
design architectures and technologies. This was also indicated by the feedback received in the conducted
surveys and is one of the most important features to include in the future work of this thesis.
Thus, in conclusion, Exemplar helps software developers build and choose the correct foundation to
start a software project, focusing specifically on the criteria; robustness, scalability and security. By
design and through the development policies provided, these criteria should be kept on a long-term
basis.
