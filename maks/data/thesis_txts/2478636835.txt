Abstract
Some software implementations of cryptography face attacks that are diﬀerent from
traditional attack models in cryptography. Most notably, DRMs and mobile payments
can beneﬁt from the use of White-box cryptography (WBC), where the attacker has ex-
tended access to the source or binary code. This study aims to describe the state of the
art attacks in WBC and how to use them against implementations with secret designs.
The report looks at some of the most recent attack types. DCA, DFA, and LDA are all
recent developments that can attack WBC implementations without any prior knowl-
edge of the implementations. It is shown that these attacks can be automated. Some
implementation has protection from these automated attacks, but they can still be at-
tacked by disabling the protections with reverse engineering. The report details working
attacks on secretly designed white-box implementations from the WhibOx competition.

CHAPTER 1
Introduction
When Conan Doyle wrote about Sherlock Holmes, he fascinated his readers with crimi-
nals who sent codes that consisted of only a series of page numbers and word numbers
associated with a particular book. Without knowing the system and the book, it would
be very diﬃcult to understand the message. The security is based on the fact that the
book’s title is unknown. Sherlock Holmes could only guess and try his way through.
In cryptography, two parties want to communicate, typically called Alice and Bob. Alice
is the sending party that wants to transmit a message to Bob. In symmetric encryption
it is assumed that Alice and Bob already have some shared secret called a key. If Alice
only has access to an insecure channel, she wants to hide the message. To do this, she
uses a cipher to encrypt the plaintext message into a ciphertext using the secret key. This
ciphertext can then be sent over the insecure channel to Bob. Bob can then decrypt the
ciphertext using the key and end up with the original plaintext. This is a cryptosystem.
There are three models for attacks on cryptosystems. They diﬀer in the amount of infor-
mation the attacker has access to. In the black-box model the attacker only has access to
the input and the output of the function. In the gray-box model the attacker has access
to additional information, namely information that the hardware implementation might
yield. In the white-box model the attacker has access to the software implementation
while it is running and the attacker therefore has access to anything they want.
Imagine that Sherlock Holmes had been able to watch everything the criminals were
doing while preparing the code: Sherlock could see which book was used, see precisely
how the criminals scrolled through the book and counted. It would have been almost
impossible to hide the message from him. This is the situation in modern White-Box
Cryptography (WBC): Everything is visible, and yet the cipher must be able to hide the
key. Is it even possible to construct such a system? To hide something where everything
is visible?
The white-box attacker model is closely related to the software implementation of the
cryptography. When more than one piece of software are running on a device the oper-
ating system will be responsible for isolating the processes. No process should be able
to gain any information about any other process. But what if a security ﬂaw exists in
the operating system and one process is able to read memory from another process? In
that case the software is running under the white-box attacker model.

1 Introduction
There are two types of attacks against white-box implementations. There are those that
are speciﬁc to a given implementation and an exact structure, and there are generic at-
tacks which do not require knowledge of the white-box design. Generic attacks have been
shown to be able to break a wide range of white-box implementations [Bos+16][Gou+18].
Generic attacks are generally preferable because they do not require any prior knowledge
of the WBC design. Three such successful generic attacks are: diﬀerential computation
analysis (DCA) and diﬀerential fault analysis (DFA) both [Bos+16], and linear decoding
analysis (LDA) [Gou+18]. In the genre of generic attacks and at the time of writing
these attacks must all be considered state-of-the-art attacks. All three will be further
explained in the report.
WhibOx is a competition to test white-box implementation and attacks. The competi-
tion enables researchers and practitioners to confront their white-box implementations
to state-of-the-art attackers.
1.1
Project Description
The purpose of the project is to try the latest attack methods, and to show how to break
real White-Box implementations, and hereby gaining experience of real world problems
and tools.
The report will look at general attack techniques and also test how these techniques can
be used in combination with more specialized approaches. Speciﬁcally it will build on
the research in DCA, DFA and LDA by showing how these attacks can be applied to
new implementations.
The report will illustrate the central role of reverse engineering in WBC attacks. Generic
attacks are supposed to work against many WBC implementations, but often some
understanding of the code structure is necessary for an attack to be successful.
As a test site, we have chosen the WhibOx competition. In the WhibOx competition,
the goal is to hide a key in an AES implementation. Therefore the methods and attacks
used are often closely related to how AES works, so to help the reader the report includes
a section that explains some basic concepts about AES encryption.
Finally the report will address a gap in the literature. The gap exists because most
designs and attacks in WBC are done in private and never published. By publishing
the code and the results from a successful WhibOx attack the author hopes to make a
contribution towards closing this gap.

CHAPTER 2
Concepts in Cryptography
Cryptography is always used in a context where an attacker tries to interfere. Therefore
two things need to be considered: What is the capability of the attackers and what is
their goal? The focus here will be on white-box cryptology, but we will ﬁrst introduce the
classic attacker types and goals and then see how these relate to white-box cryptography.
2.1
Attack Types
Cryptoanalysis is the art of breaking cryptosystems. Historically, cryptoanalysis of block
ciphers has focused on key-recovery. However, it turns out that most of the time, more
protection is needed. Imagine a cipher that is protected from a key-recovery attack, but
an attacker can decrypt the ciphertext without knowing the key. In most cases, this is
not desired, and therefore the following attack goals are deﬁned [KR11]:
• Key Recovery: The adversary can ﬁnd the full key.
• Global Deduction: Given the encryption/decryption function Ek/Dk the adver-
sary can construct an equivalent function A without knowledge of the key.
• Local Deduction: The adversary can generate the message to a previously unseen
ciphertext. That is, he can invert the cipher.
• Distinguishability: The attacker is given access to two functions: one is the ci-
pher with a random key the other function is a random permutation. The attacker
only has black-box access to the functions, which means they can only see the
input and the corresponding output of the functions but not how the functions are
computed. The goal of the attacker is then to decide which of the two is the cipher
function. If the attacker is unable to do this, then the cipher leaks no information.
The four attacks types form a hierarchy: If one attack goal is achieved, then all the
following attack goals are achieved. Achieving security against distinguishability, implies
achieving security against all the others. So consequently we call distinguishability the
strongest security goal.

2.2
Attacker Models for Ciphers
It is essential to consider the attacker model when determining the security of a cipher.
The attacker model describes which information the attacker can access. The diﬃculty
of an attack depends signiﬁcantly on the information the attacker has access to.
Figure 2.1 shows an overview of the three overall categories of attacks, with the attacker
getting access to more and more information. The next three sections will describe these
three categories.
Figure 2.1: The three diﬀerent categories of attacks. The security threat increases as
the attacker get access to more information [De 14].
2.2.1
Black-Box
We will ﬁrst consider the black-box model where the attacker only has black-box access
to the cipher [Knu18]. There are four diﬀerent attacker models:
• Ciphertext-Only Attack (COA). The attacker only has access to some set of
s ciphertexts: c1, c2, · · · , cs.
• Known-Plaintext Attack (KPA). In addition to the ciphertexts, the attacker
also has knowledge of the plaintext corresponding to each ciphertext. The attacker
has access to: m1, m2, · · · , ms and c1, c2, · · · , cs.
• Chosen-Plaintext Attack (CPA). As the name suggests the attacker can choose
some plaintexts m1, m2, · · · , ms and then receive the corresponding ciphertexts
c1, c2, · · · , cs.
• Chosen-Ciphertext Attack (CCA). Here the attacker gets to choose cipher-
texts c1, c2, · · · , cs. and then receive the corresponding plaintexts m1, m2, · · · , ms

2.2.2
Gray-Box
For some practical applications, the black-box model is not strong enough. The attacker
might have access to the hardware doing the encryption. This allows the attacker to
make side-channel attacks where they extract extra information from the hardware. To
capture these kinds of attacks, we use the gray-box model. This threat model no longer
considers an abstract cipher but instead thinks of a concrete implementation running
on some hardware. Implementation attacks can be split into two kinds [De 14]:
• Passive Attacks. The encryption proceeds as usual, but the attacker collects
some information while the implementation is running. The information could be
the power the hardware is consuming during the encryption, or it could be the
measured timing diﬀerences from one message to another.
• Active Attacks. The attacker disturbs the implementation while it is running.
The attacker can, for example, expose an embedded processor to high temperature
or a higher supply voltage than what is supported. The attacker then hopes that
the interference will change some internal state of the cipher, which then leaks
information, for example, giving a diﬀerent output. This process can be destructive,
but it does not have to be.
2.2.3
White-Box Cryptography (WBC)
As we move toward the attacker getting more information, we arrive at the white-box
model. The white-box model describes threats to software implementations. In the
white-box world the attacker will have access to everything. [Cho+02], therefore intro-
duced the white-box model:
Deﬁnition 2.2.1 White-Box Attack Context assumes that:
1. Fully-privileged attack software shares a host with cryptographic software, having
complete access to the implementation of algorithms.
2. Dynamic execution (with instantiated cryptographic keys) can be observed.
3. Internal algorithm details are completely visible and alterable at will.
The white-box model is a strong model for the attacker. Here are a couple of the things
an attacker can do:
• Static analysis. The attacker has access to either the source code or the binary
code and can do reverse engineering to ﬁgure out how the implementation works.
==PAGE:14==
6
2 Concepts in Cryptography
They will also be able to alter the implementation as they want. They can also
use the traditional static analysis tool to create a ﬂowgraph to ﬁnd the structure
and important parts of the cipher.
• Dynamic analysis. The attacker can access the implementation while it is run-
ning. They can run the implementation in a debugger and dynamically change
it while it is running. By pausing the debugger at any moment, the attacker can
read the content of memory and log every value written to memory.

CHAPTER 3
Methods in WBC
I White-Box Cryptography (WBC) Alice still want to sent a message to Bob. But Bob is
unable to do secure computations without someone observing everything he does. In this
scenario an adversary would easily be able to see the key that they are using, when they
use standard black-box techniques. Instead Alice can send a white-box implementation
with the key hard-coded in the code. If the white-box implementation is secure the
attacker will not be able to recover the key. This is also useful in a situation where Alice
does not trust Bob, if she sends a white-box implementation even Bob will not be able
to ﬁnd the key while still being able to decrypt any message that Alice sends using the
key.
But no one has been able to ﬁnd a cipher implementation that is secure in the white-box
model and it is therefore only a dream. But Alice might be able to make it diﬃcult
for Bob to recover the key from an implementation, by using the current white-box
implementations.
The goal of WBC is to hide secret information in a software implementation of a cryp-
tographic primitive. Figure 3.1 shows a high-level overview of how the process works.
A cryptographic primitive that needs a key is transformed into an implementation in
which the key is hard-coded. To hide the key, it is combined with random data and an
obscure code ﬂow.
3.1
White-box and Security
White-box cryptography is part of a ﬁeld called software security. When WBC is used in
the real world, it will be part of a binary program as seen on Figure 3.2. It is essential to
consider this context because without the software security protecting the binary, some
trivial attacks are possible. Consider the global deduction attack goal. In the global
deduction attack goal the attacker has access to an implementation with a hardcoded
key and can, therefore, copy the entire implementation instead of just extracting a key.
The copy will be functional equivalent to the cipher and can encrypt messages with the
secret key. This attack is called code lifting and can be protected against by binding
software to a given device.
There are many challenges in software security, but we will focus on the WBC.

3.2
Use Cases of White-Box Cryptography
So far, we have only given a vague argument for why WBC is essential. In this section,
we will explore some of the real-world use cases for WBC. First, consider the following
quote from [Wie16]:
If your smartphone only contains trusted software and is only ever under the
physical control of trustworthy people, then you don’t need WBC.
In reality, software implementations are executed in an untrusted environment.
We,
therefore, need to consider how to protect them. WBC is not perfect, as there still is
no construction that has not been broken. It is still an open problem, if it is possible to
construct an unbreakable white-box implementation [Mui12]. It has been conjectured
that all white-box resistant implementations get broken at some point [Cho+02]. It is
like a cat and mouse game that goes on forever. The goal for many real-world imple-
mentations is, therefore, to slow down an attacker or prevent a less technical user from
obtaining the secrets.
3.2.1
Digital Rights Management
WBC was ﬁrst introduced in the context of Digital rights management (DRM) [Cho+02].
DRM aims to protect digital content by limiting where and when it can be used. DRM
is found in many online stores that sell digital multimedia such as music, videos, ebook,
and applications. DRM schemes work in a white-box model where the attacker is a user
that wants to share the content without a license to do so. The content provider wants
to limit the user only to be able to play the content in an authorized manner. In this
section, we will see how a DRM scheme called FairPlay works and how it can utilize
WBC.
Apple’s online digital media store iTunes uses the FairPlay DRM technology [Ven07].
Most DRM schemes are kept secret as an added layer of protection. However, FairPlay
was broken not long after its introduction in 2004 [Ven07]. It has therefore been docu-
mented how this old version worked. One reason for failing in the early version was that
the old version of the scheme was only secure in a black-box attacker model. However, an
attacker can reverse engineer the media player application and obtain white-box access.
Figure 3.3 is a simpliﬁed model of how the old FairPlay works based on the description
in [Ven07]. It uses multiple barriers of defense by encrypting keys with other keys. The
following three keys are used:
1. kc is the content key which is used to encrypt/decrypt the content.

3.2.2
Host Card Emulation (HCE)
Host Card Emulation (HCE) is a software architecture that can emulate contactless
smart card cards using only software. Contactless smart cards can, for example, be used
for access, transit, or banking. In HCE all cryptographic operations run in software
along with potentially malicious software. HCE solutions, therefore, needs to consider
the white-box attacker model. HCE was introduced in 2013 to Android version 4.4 and
is endorsed by Visa and MasterCard [All14].
Contactless smart cards use Near Field Communication (NFC) to communicate wire-
lessly with a reader terminal. Most mobile devises have an NFC chip; this allows them
to be used in card emulation mode where the reading terminal sees the mobile device
as a contactless smart card. When using card emulation, credentials need to be stored
inside a hardware-based secure element (SE) on the mobile device as seen on Figure 3.4.
A SE is a tamper-resistant chip capable of securely hosting applications and their conﬁ-
dential and cryptographic data. The SE can be in either the sim card or on an SD card
and is issued by the mobile network operators.
Figure 3.4: NFC card emulation with a secure element [Dev17].
HCE provides an alternative to card emulation with an SE. In HCE, the credentials are

3.3
Table-Based WBC
In this section we will introduce how to make a table based white-box implementation
with a ﬁxed key. This implementation was ﬁrst introduced in [Cho+02]. Let us ﬁrst
look at a cryptographic primitive into a white-box implementation.
We could implement it as one big lookup table. If we have a 128-bit block cipher then
we need a table with 2128 entries all of size 16 bytes. This would take up 5.445 ∗1039
bytes which is way too big. The idea is instead to replace smaller parts of the cipher
with lookup tables. Imagine a part of the cipher that takes 8-bit input i and a key k to
return some output o. C(i, k) = o. To hide the key we can turn this operation into a
lookup table of size 28 bytes. This size is much more manageable. We can use this same
principle to change all operations in the cipher into tables.

3.4 Dynamic-Key

3.4
Dynamic-Key
When we do as described above, we call it a ﬁxed key implementation. But this is not
always what we want. In some cases we need to update the cryptographic keys. We
call this a key-update message. With a ﬁxed key implementation we can not actually do
key-updates, we have to generate a new implementation and send it to where the update
is needed. In this way the whole implementation will be replaced. A more optimized
approach is to just send the part of the implementation that is key dependent. It might
be that only the lookup tables are key dependent and the rest does not need to be
changed.
For the implementation presented in [Cho+02] this would mean sending 752 kB if the
whole thing is sent and 208 kB if only the tables are sent. This is still a big update
and for some applications it would not be practical. There are therefore dynamic-key
white-box schemes that try to minimize the key-update size. There is even an optimal
key-update size. Since the key-expansion is never a part of the white-box implementation
the optimal size for AES-128 is the size of the fully expanded key 16 ∗11 = 176. These
schemes generally work by sending the diﬀerence to the new key, ∆= k ⊕k′, where k
is the old key, k′ is the new key and ∆is the diﬀerence that is sent as the update. This
diﬀerence can then be used to update the tables.
3.5
The WhibOx Competition
Most of the time WBC is designed in secrecy. But there has been some competitions
where white-box implementations have been shared. The Conference on Cryptographic
Hardware and Embedded Systems (CHES) has organised 2 capture the ﬂag contest
on white-box cryptography, called WhibOx ﬁrst edition[ECR17] and WhibOx second
edition[Com19]. Both competitions have almost the same rules. We will here look at
the rules for the second edition. There are two ways to participate:
1. Coders post challenges that are white-box implementations of AES-128 with a
secret key.
2. Attackers try to break the challenges by inverting a ciphertext or by guessing the
key.
A challenge must be a single C ﬁle with no assembly instructions and no use of external
libraries. A submission is uploaded with its corresponding key. The judging system then
tests that the implementation is indeed AES-128 with the given key and then deletes
the key. The judging system has to be trusted to delete the key, but this is the only time
the judging server has to be trusted. The judging server code is open source, and the
server hosting is overseen by an organizing committee. Later, when an attacker submits

Challenges accumulate points as long as they stay unbroken. In the end, the challenge

3.6
Advanced Encryption Standard (AES)
AES is a block cipher standardized by NIST in 2001[RD01]. AES has a block size of
128 and supports three diﬀerent key sizes: 128, 192, and 256 bits. Here the focus will
be on AES-128, which has 10 rounds. The 192 and 256 variants are similar except they
respectively have 12 and 14 rounds. AES maintains a 128-bit state, which consists of 16
bytes. The bytes can be arranged in a 4x4 matrix as seen on Figure 3.6. The operations
all operates on this matrix. Unless otherwise noted we will use zero indexing starting in
the top left corner then going down one column at a time. Therefore the top left bytes
is byte 0, and the top right byte is byte 12. Every round consists of 4 steps:
• AddRoundKey(AK). Every byte in the state is combined with the round key using
bitwise XOR. AES has a key schedule to produce a new 128-bit key for each round,
using the original key. The details of the key schedule are omitted here, except for
the fact that the key schedule is invertible.
• SubBytes(SB). Every byte in the state is substituted with the result of the S-box.
The AES S-box is a function that maps 8-bit inputs to 8-bit outputs. The S-box
can be represented as a lookup table. This operation provides the non-linearity in
the cipher.
• ShiftRows(SR). The bytes of every row is shifted a diﬀerent amount to the left.
The ﬁrst row is not shifted, the second is shifted by one and the third by 2 and
the fourth by 3.
• MixColumns(MC). Every column is multiplied by a matrix in GF(28). This oper-
ation, together with ShiftRows provides diﬀusion in the cipher.

CHAPTER 4
Generic Attacks on WBC
When attacking WBC we can either have generic attacks or attacks for speciﬁc imple-
mentations. But since we do not know which speciﬁc protections are used it is hard
to launch a speciﬁc attack in practice. We will therefore here present 3 generic attacks.
They are all widely applicable and variations have been shown to break many publicly
available white-box implementation. We will focus on attacks on AES, but most of them
can be generalized to a white-box implementation of any cipher. We will show Diﬀer-
ential Computation Analysis (DCA) and Diﬀerential Fault Analysis (DFA) which both
are attacks that originally were developed as gray-box attacks, but they have later been
adopted to work on white-box implementations. Linear Decoding Analysis (LDA) is an
attack that was made speciﬁcally to attack masked white-box implementations. In the
end we show a 5 Step Attack Methodology for attacking white-box implementations.
4.1
Differential Computation Analysis (DCA)
Diﬀerential Computation Analysis (DCA) is an attack that utilizes information that an
implementation might leak about a predictable intermediate value within the state of
the cipher. A predictable value is some intermediate value within the cipher that we
can predict by knowing the plaintext and assume some part of the key to be known.
We want the assumed part of the key to be small enough to be able to calculate the
predictable value for all possible key guesses. A distinguisher can then tell which key
guess is the most likely. [Bos+16] show that DCA can be used to extract keys from
white-box implementations. If we only guess a single byte of a 16 bytes AES key, there
are only 256 possibilities. We can then recover the key one byte at a time using a divide
and conquer attack.
The idea behind DCA comes from the gray box model where diﬀerential power analysis
(DPA) is a side channel attack that can be used if an implementation leaks power in-
formation. Instead of being limited to just information on the power used, in WBC we
can get any information from the running code that we want. Some examples of inter-
esting information includes memory locations written to, register values, and timing of
functions.
Overall the DCA attack has three steps:
==PAGE:28==
20
4 Generic Attacks on WBC
1. Acquire traces for randomly chosen plaintexts.
2. Calculate predicted values of the plaintext.
3. Use a distinguisher to ﬁnd the best key guesses.
4.1.1
Acquiring Traces
The ﬁrst step is to acquire the software execution traces. This is done by generating
random plaintext and then save some information while the implementation is running.
In principle anything can be traced. If we have some insight into the implementation
then we might know approximately which part corresponds to the place in the cipher
we want. In other cases we know little of how the implementation works. In these cases
we have to attack more broadly and trace everything without any ﬁlter. This could for
example be memory or stack accesses. Both the values written and the addresses.
If the two traces are not of the same length then values in one trace will not line up
with the values in the other trace. We would therefore have to do some kind of post
processing on the traces to align them. Since this is a complicated process we generally
want to ensure that the traces have the same length.
4.1.1.1
Techniques for Acquiring Traces
We will consider two ways to do this: instrumentation and emulator recording.
Instrumentation is a technique that inserts code which can collect information at run
time. For example, it could log the address every time something is written to memory.
In source code instrumentation we need access to the source code. We can then insert
code into the source ﬁle. For example a line of code can be added to the top of every
function or just before every memory access.
Binary instrumentation uses the same principle as source code instrumentation, but
here we insert new instructions into the machine code. The advantage is that we do not
require access to the source code, but at the cost that it will be harder to accomplish.
If the instructions are inserted at run time it is called dynamic binary instrumentation
(DBI). DBI adds a layer between a running program and the operating system that
allows us to inspect and log the program while it is running. These tools are often very
complex but oﬀer a simple API, that makes things easier.
Emulator recording works be running the implementation in an emulator. The emulator
can then be instructed to trace the desired values. The disadvantage is that you need
an emulator that can accurately run the binary of the implementation. The advantage
is that tracing values will be simpler since we do not need to modify the binary.

Sometimes we will have a speciﬁc insight into how the cipher is implemented and we

4.1.3
Distinguisher
We have now acquired the traces from N plaintexts. And for each plaintext we also
have the candidates for each key-guess. Now we can try to identify which is the correct
key.
We expect the candidates for the correct key to have some connection to the
traces. In some cases we will have an exact match between the candidates for a key and
somewhere in the trace, but other times the connection is weaker. To determine which
key is the most likely we use a distinguisher. A distinguisher is a function which maps
the computational traces and corresponding candidates to a score vector. The score can
be anything, but when an attack is successful some of the key guesses have high scores.
Even when more than one guess have high scores then it might still be good because the
attack has narrowed the key space down and we might be able to brute force our way
forward. If for example we are able to reduce each key byte to the top 3 guesses, then
we will have to brute force 16 bytes: 316 = 43.046.721 combinations which is feasible.
We will here present two diﬀerent distinguishers but it is by no means the full list.
The exact match distinguisher looks for every candidate at every position in the traces.
If a candidate is found somewhere in the trace it will return 1 otherwise it will return 0.

The Absolute Pearson Correlation works like exact match except instead of looking for
an exact match, it calculates the correlation between the candidate and every position
in the trace. The score is then the highest correlation between all the positions.

4.2
Linear Decoding Analysis (LDA)
Linear Decoding Analysis is an attack that works with traces from the implementation
just like DCA. Usually this will be traces of memory values, we will later see why this
makes sense. LDA is made speciﬁcally for the purpose of attacking implementations
that use masking to hide the state. We will therefore ﬁrst present what masking is and
why you might want to use it. [Gou+18] introduced the LDA attack in order to break
the winning challenge "Adoring Poitras" from whibOx edition 1. [BU18b] introduces a
similar attack but calls it Linear Algebra attack.
4.2.1
Masking
Masking is an implementation technique used to hide the values in the state. It is a
technique used to protect against DCA. Instead of storing the actual intermediate state,
the state is spilt into multiple values called shares. A separate decoding function will
then do convertions from the shares to the correct state. The implementation is modiﬁed
to work on the shares instead of the actual state. This means that the state is never
exposed and it will therefore be impossible to do DCA.
A concrete example of masking of a white-box is implementations based on a boolean
circuit. We want to hide the state w. Instead of computing and saving w directly in
memory we split it into s shares v1, · · · , vs such that v1 ⊕· · · ⊕vs = w. We can do this
by setting v1, · · · , vs−1 to random values and then setting vs = w ⊕v1 ⊕· · · ⊕vs−1. In
this case the decoding function is XOR. This will mask one variable in the state, and
we need to do this for every variable. All operations also need to be updated to work
with the masked shares instead of the state directly.

narrow down the possible keys by taking the intersection of the two possible key sets.
The intersection works because the key was the same when doing both faults. [PQ03]

single key.
We now have a strategy for ﬁnding 4 key bytes after introducing a fault in the ﬁrst
column. The same strategy can be applied to the three other columns to recover the
full key. Since we need 2 faults per column and there are 4 columns this attack need a
total of 8 faults. Python code to perform the attack can be found in Appendix A.
This procedure only ﬁnds the last round key and not the initial key used for AES. If
we are attacking AES-128 we can run the invertible key schedule backward to get the
initial key. If we are attacking the 192- or 256-bit variants we have to recover the keys
from the last two rounds. We can do this by ﬁrst applying the attack to get the last
round key. Then apply the attack again but targeting a round earlier and rolling back
the result a round using the just obtained last round key. We will then have the key for
the two last rounds and we can again run the key schedule backwards to get the initial
key.
We can improve on the attack by targeting the 8th round of AES. If we get a single byte
faulted in round 8 it will propagate to one error in each column in round 9. All of these
errors will propagate independently, we can, therefore, apply the round 9 attack on each
of these. This technique requires 1/4 the faults, but it will be harder to identify whether
the fault was correctly injected since all output bytes will change instead of just 4 bytes
in a speciﬁc pattern. In WBC generating more faults is usually not the problem, but
identifying whether we got a good fault is more important. Therefore this 8th round
attack is rarely useful.

4.3.2
Generating the Faults
Now that we know how to recover the key from a faulty output, we will look at how
to generate the faults in a white-box setting. Overall there are two ways faults can be
generated:
• Statically: When we generate a fault statically it means generating it before the
program is run. This could be done by changing the source code or the machine
code. It could also be done by changing data tables in an external ﬁle that the
program uses.
• Dynamically: Here we generate the error at run time. This can be achieved with
a debugger, instrumentation framework or an emulator. The idea is to pause the
execution at a speciﬁc point and then change the content of memory.
Sometimes only one of the methods is available. The implementation might be running
on a platform where attaching a debugger to the process is impossible thus making the
dynamic route harder. Generating faults statically might be hard if there is no access
to the source code, because then the fault can only be inserted into the machine code.
Some implementations might also feature an integrity check that checks whether the
code has been modiﬁed. Statically inserting faults may still be possible but the integrity
check will need to be patched.
4.3.2.1
Statically Generating Faults
Let us ﬁrst assume that an implementation does not hide its structure in any signiﬁcant
way. Inserting an error will then be as simple as inserting this one line of code:
if (round == 9) state[0] ^= 1;
Here we insert a fault in the state only in round 9.
Inserting faults can also be done statically in the binary machine code. We simply make
a copy of the original binary and then change one or more bytes. This might be necessary
because the source code is not available. But it gives the advantage that it will be easier
to change the program automatically since we do not have to worry about generating
valid program code. It also has the advantage that you do not have to wait for the
program to recompile every time a change is made.
[TH16] presents a tool for statically generating faults. The tool is called Deadpool. It
takes a copy the original implementation and changes some bytes. We will later look
at how we can narrow the area where we change bytes. If the implementation includes
lookup tables in a separate ﬁle those can be changed as well.

4.3.2.2
Dynamically Generating Faults
Dynamically generating faults can be done in a simple way as well:
1. Run the program in a debugger like GDB.
2. Pause the debugger at the desired step.
3. Modify a memory location. In GDB it would look like this: “set int0x83040 = 4”.
4. Continue running the program and record the output.
This is a simple example but often you need to automatically search for the correct loca-
tion and breakpoints. It is therefore necessary to script the debugger to automatically try
diﬀerent combinations. [SMH15] analyses diﬀerent white-box implementations. Faults
where injected dynamically by running the implementations in the Unicorn emulator. A
script was used to inject faults.
4.3.2.3
Finding the Correct Location
How do we know that we introduced the fault in a correct location? We do this by looking
at the output of the implementation. If we compare the output of the implementations
with and without a fault we can observe which bytes are diﬀerent. We want 4 bytes
to be changed, but the locations of the bytes is important.
We want the positions
to correspond to the positions of a single column after ShiftRows. This means we are
looking for output diﬀerences in speciﬁc bytes. We want one of these four patterns which
each corresponds to a column: [0, 7, 10, 13], [1, 4, 11, 14], [2, 5, 8, 15] or [3, 6, 9, 12].

• 1 byte changed: The error was introduced at the end of the cipher. If the error

• All bytes changed: This means that we introduced the error too early or that

• Program crashes: The error we introduced caused the program to crash.
• Program loops: The change caused the program to enter an inﬁnite loop. This

run the program.
Even if we can get one of the correct patterns, we might not have a usable output. A
couple of things can go wrong:
• We have introduced errors in multiple bytes in the same column. In this case, we
would see a correct pattern. However, it would not work since our analysis assumes
just one byte will change. If two bytes changed, we could still do the analysis, but
it would result in too many unknowns in the equations.
• The implementation might have a masking scheme that breaks our assumption of
how the AES was implemented.
• The implementation might have a counter measure that detected the error and
then introduced the error in the correct positions. This could be done to confuse
an attacker, who is trying to ﬁnd the correct locations automatically.
4.3.3
Learning the Structure of an Implementation
Most white-box implementations will try to hide its structure. This is a hindrance to
all the described attacks. In DFA we try to insert the fault at very speciﬁc locations.
When the structure is hidden there are two approaches when inserting the errors.
1. Insert errors at random place.
2. Try to learn the structure of the implementation.
Both DCA and LDA works with traces from the computation performed by an imple-
mentation. While both attacks work with long traces, the complexity of the attack will
increase with the length of the traces. It is therefore advantageous to to know the struc-
ture of the implementation and use the knowledge to only trace speciﬁc parts of the
cipher.

4.4
Automated Attacks
One advantage of both DFA and DCA is that the attacks can be applied without any
knowledge of how an implementation works. The authors of [Bos+16] released their au-
tomated attack tool on GitHub: https://github.com/SideChannelMarvels/
Deadpool.
They released two tools called deadpool_dca and deadpool_dfa to do
these automated attacks.
deadpool_dca will automatically trace an implementation and apply a variety of distin-
guishers to automatically ﬁnd the key. By default it will trace every instruction in the
implementation. This results in some long traces. On large implementations a single
trace can be over a GB. This is potentially a problem.
deadpool_dfa tried automatically to ﬁnd good faults for DFA analysis. Since implemen-
tations are longer brute forcing by searching for 1 bytes changes will take too long. To
improve on this it uses a divide-and-conquer strategy. It starts by faulting large areas
at once. It does this by faulting all consecutive bytes of the area. Then depending on
the result it will improve the search area:
• If the output is not aﬀected
• If one single output byte is aﬀected it will ignore the area.
• If exactly four bytes are aﬀected, then we have a good fault that we can save.
• Otherwise the area is divided in two.
After collecting the speciﬁed number of faults the automated search will stop and a DFA
analysing program can be called to check whether a solution was found.
4.5
5 Step Attack Methodology
In the ﬁrst edition of the WhibOx contest the implementation "Adoring Poitras" won by
a large margin. After the contest the winner in the attacker category published how the
attack was done [Gou+18]. The paper also contained a generalized attack methodology
for WBC that consist of 5 steps. This is a good attack method when the automated
attacks do not work. The 5 steps are:
1. Initial reverse engineering. This step consists of removing the obfuscations.
The goal is to understand where the code is and what it does.
One common
obfuscation layer is virtualization where most of the program is in some other
bytecode language that is often made for the occasion.
In the paper most of
==PAGE:43==
4.5 5 Step Attack Methodology
35
the calculations were done using 32-bit numbers but by studying the code and
removing the obfuscation layers they where able to deduce that it was all hiding
some binary operations. The output is a arithmetic circuit. The node can be any
calculation. This could be binary operations, ordinary math operations or any
generic operation deﬁned in look-up tables.
2. SSA transformation. Here the arithmetic circuit is converted to Static Single
Assignment form (SSA). This form requires that each variable is assigned exactly
once, and that every variable is deﬁned before it is used. This can be achieved by
replacing the target of each assignment with a new variable, and replacing each
use of a variable with the previous deﬁnition of that variable.
3. Circuit minimization. In this step the circuit is minimized. The value of nodes
from the SSA is traced using a large number of randomly selected plaintexts. Using
this information we can then remove some nodes. This is actually done in several
sub steps:
a) Constant removal. If a node takes the same value in all the traces we call
it a constant. All nodes that the constant depends on can then be removed.
b) Pseudo random variable removal. If a variable can be changed in all
traces without aﬀecting the ﬁnal result we call it a pseudo random variable
and we can replace it by the constant 0.
c) Propagate constants Using mathematical rules we can propagate through
the circuit. For example a multiplication node where one of the inputs is the
constant 0 we know the output will be 0.
d) Remove unused nodes. The initial circuit may contain some unused nodes,
but they in turn might be the result of another sub step removing a node that
make some other nodes useless.
Since the minimization in one step can inﬂuence the others, all sub steps are
repeated multiple times.
4. Data dependency analysis. Hopefully one can now ﬁnd some structure. This
can be found by using manual inspection or by automated analysis. At this state
the authors plotted the data dependency graph of the ﬁrst 5% of the boolean
circuit. Here reproduced in Figure 4.6. It can be seen that this revealed some of
the structure of the AES implementation.
5. Algebraic analysis. With the data dependency analysis hopefully pointing to
some information about where the encoded state might be, we can then perform
LDA. The authors also suggested that DFA or DCA could be used.

In WBC, there is currently no tool that makes encryption secure.

encryption is, therefore, very diﬀerent from method to method.

5.2
Reverse Engineering
In practice reverse engineering is a huge part of WBC, we will therefore elaborate on
strategies used in reverse engineering. The goals of reverse engineering are to simplify
the program, ﬁnd areas of interest, and to understand it. When doing this, we form
a hypothesis about how the program works; most of the time, this hypothesis can be
checked by modifying the program. The basic principle is to modify some code and then
check that the program calculates the same function. Checking whether the program
makes the same calculation can be a diﬃcult task since the input space is vast. But we do
not need to verify it 100% we can check that it behaves correctly on a few selected inputs.
Using the unmodiﬁed implementation, we can generate several plaintext ciphertext pairs.
We can then run the modiﬁed implementation on the same plaintexts and check that we
get the same ciphertexts. This procedure does not verify 100% that the two programs
do the same thing, but it gives a high conﬁdence.
When we use this procedure, we can even test a modiﬁcation that is only a weak guess.
An idea can be validated quickly. In case the modiﬁcation breaks the program, we learn
something, namely that the assumption we made was wrong.
As an example imagine we have an implementation that calls a function in regular
intervals:

From the automated DFA, we found that the array here named program was interesting.
By following the use of this array, it is clear that it stores the instructions for some virtual

Every virtual machine instruction writes a byte to the program array and then increases
the pointer.
But it will always write to a position before the position in which the
instruction pointer is located. Therefore we can think of this array as a separate array
which is initially empty. Since the write counter is always increased by one it must
function like a stack. The write pointer is always modulo 65535 and it is therefore used
as a circular buﬀer.
The program contains 7 functions:
1. readpt reads a bit from the plaintext. Every bit is read 46 times.
2. writect reads a speciﬁc byte from memory that is ready to be output as the
ciphertext.
3.–5. read_bits, read_10_bits, GET_BITS are all for reading a speciﬁc amount
of bits from a bytelocation and bit oﬀset.
6. AES_128_encrypt is the required AES function.
It runs a virtual machine.
Instructions are read bitwise.
7. update_pc is used to update the program counter by a given number of bits.

Most of the complexity of these functions comes from this being a bitwise implementation
where everything is not aligned to bytes.
The virtual machine contains the following instructions:
• noop this instruction do nothing but increase the pc. This is useful because the
memory is a circular buﬀer and sometimes we want to save old values.
• binary_op this instruction performs one of four binary operations. One of the
inputs selects which table to use corresponding to an operation. The operation is
on two 10-bit values resulting in a 10-bit value.
• write ct uses the function writect to output a ciphertext byte. This happens
exactly 16 times.
• load plaintext uses the read_pt function to load a single plain text byte. Since
every bit is read 46 times this instruction is called 46 ∗128 = 5888 times.
• copy copies a value from one place to another.
• intermediate operation is the same as binary op except it reads one value from
memory, and the other is an intermediate value.
In addition to the above mentioned actions, some of the functions also deal with encoding.
The binary_op instruction has additional ﬂags that controls how to encode each of
the two values.

CHAPTER 6
Conclusion
The purpose of the report is to describe state-of-the-art attacks on white-box cryptog-
raphy and show how they can be practically applied against implementations.
The report initially described several recent generic attack types:
• DCA attacks which work by collecting traces of information from an implementa-
tion. And the comparing to a predictable value within the cipher.
• DFA attacks that change the implementation while it is running and then compares
the modiﬁed output to the original.
• LDA attack which solves a linear system of equations to defeat masked implemen-
tations.
The report went on to show that DCA and DFA can be made into automated attacks
that can run against any implementation without prior knowledge.
Implementations submitted to the WhibOx competition was used to show how the at-
tacks works against real implementations. The automated attacks were successfully used
against two diﬀerent secrete implementations submitted to the WhibOx competition.
The automated DFA was able to insert enough faults to get the key.
It was shown that one implementation was immune to an automated DCA attack. It
was also shown that with some reverse engineering it was possible to use static instru-
mentation to trace speciﬁc information. Using the static traces DCA could successfully
attack the implementation. A weakness was exploited which made it possible to trace
the order functions were called in. Some functions transitioned to two other functions
with a 50/50 chance. The key was leaked by tracing the order these functions were called
in.
A 5 step attack methodology was used to attack another implementation but the attack
was unsuccessful. By considerable manual eﬀorts the obfuscations and the virtual ma-
chine was removed. But after circuit minimization no structure revealed it self and it
was not possible to identify a place to either inject faults or trace information.
==PAGE:62==
54
6 Conclusion
The investigations found that it is important to know not only how the attacks work
in theory, but also how they work in practice. Sometimes practical attacks requires
more knowledge and restructuring than the theoretical attack can provide. One of the
conclusions therefore is that the general attacks like DCA, DFA and LDA might work
on some implementations but most attacks will also require some reverse engineering.
This is part of the cat and mouse game where some people are trying to implement
protections while others are attacking.
There are still many implementations from the WhibOx competition that were broken
during the competition, but no public information is available on how this was done.
Most of the implementations can be attacked with DCA, DFA or LDA, but the diﬃcult
step is to reverse engineer and disable protections. There are still work to do and attacks
to make public.
There is also work to be done in designing white-box implementations. Every implemen-
tation from either the ﬁrst or second edition of WhibOx was defeated within 51 days.
This shows that there is still no public WBC that has not been broken. It is an open
question whether it is possible to create a secure WBC design. There is still much work
to be done in designing protections. Is it possible to create a protection that can not be
disabled?
We publish an attack on the "Stoic Thompson" implementation from WhibOx second
edition. There was no previous public know attack.
