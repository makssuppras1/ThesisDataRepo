Chapter 1
Introduction
In the world of enterprise software, it is a common requirement that the product is secure. However,
users of the software also want the product tailored to their needs. The enterprise product from Microsoft
called Dynamics 365(D365) solves this problem by allowing users to extend the product with their business
logic. This is done in a way that resembles workﬂow languages, where operations from the product can
be extended with workﬂows that are run before or after the operation.
The goal of this report is to apply techniques from the academic world of formal methods to the domain
of D365. This is done by deﬁning a workﬂow language called Flow and applying the techniques to that
language. Flow seeks to capture the functionality of plugins in D365, which is one of the ways to add
business logic to D365. Plugins are chosen as a reference, since they are one of the more complex workﬂow
languages that exist in the product, since their implementation is written in .NET code, usually C#.
The Flow language will be deﬁned formally with a type checked grammar, as well as a deﬁnition for how
the grammar is translated into a program graph. Semantics, program analysis’, and a security analysis
will be deﬁned for the Flow language, thus covering a broad spectrum of techniques from formal methods.
The attached code project will contain an implementation of these techniques in order to argue that they
are correct.
The reader of this report is expected to have understood the concepts used in program analysis and
language-based security, such as those taught in 02242 - Program analysis and 02244 - Language Based
Security. It is also useful to have a basic understanding of databases. Most of the deﬁnitions in the report
will be based upon the techniques from the book that is used in those courses, Formal Methods[7].
The report is structured as follows. Section 2 will deﬁne the language of Flow and describe how D365
works and relates to Flow. Section 3 will describe the semantics of Flow and show that the semantics
ensure the type restrictions deﬁned in section 2.
Section 4 will describe a Reaching Deﬁnitions and
Available Expressions analysis for Flow. Section 5 will describe how security is handled in D365, and
how it can be improved using information ﬂow.

Chapter 2
The Flow language
This section will describe a language called Flow. This language is intended to capture a subset of the
functionality of D365. First the programming language will be demonstrated, next the grammar is shown,
and ﬁnally the program graph generation will be described.
2.1
What is Dynamics 365?
Dynamics 365(D365) is the latest name for what was once Microsoft’s Customer Relation Management
solution. For simplicity, this report will not describe the entirety of D365. Instead it will describe an
abstraction of the system. The correctness of this abstraction will be based upon my extensive personal
experience with the platform.
The system can be understood as a SQL database with an API layer. This database can only be managed
by Microsoft internally, the only way to access it for users is through the API layer. The API supports over
200 diﬀerent requests [3]. For simplicity, this report will only concern itself with the CRUD operations,
Create, Retrieve, Update, and Delete.
What makes this system complex, and the reason for this analysis, is the ability to add code that is run
before or after an API call. This code is executed in the D365 transaction pipeline.
D365 transaction pipeline
A normal database system has triggers and stored procedures, which allows developers to add business
logic to operations on the database. To abstract D365 from the underlying database, they have added the
plugin structure. Plugins are pieces of code that are registered on the D365 instance. They are registered
on speciﬁc entities, for speciﬁc API calls. All these plugins are executed in a single transaction, from the
API call is made to the result is returned to the user.
Figure 2.1 shows an abstraction of the transaction pipeline in D365. The user calls a speciﬁc API on a
D365 instance. This call is forwarded through all plugins that are registered pre-operation. These plugins
can modify the parameters of the API call, such as adding additional ﬁelds to a Create call or removing
columns from a Retrieve call.
3
==PAGE:5==
Figure 2.1: An abstraction of the transaction pipeline in Dynamics 365.
The call is then sent to the operation. This will do the action of the API call, e.g. a Create call will
create a new record in the database.
The API call is then sent to the post-operation plugins. Note that this is done when the operation is
ﬁnished. These plugins do operations that require the operation of the API call is ﬁnished, e.g. adding
related data to a record after a Create call.
Finally, a response is returned, but not all calls will have a return value. Create and Retrieve will return
a value, but Update and Delete will not.
This abstraction shows a simple request-response system. The complexity comes from the reality that
plugins can make API calls. This means that the system is vulnerable to potential inﬁnite recursion and
makes it hard to trace what the system has done during a transaction. This is one area where program
analysis can prove very valuable.
Registration of a plugin
If all plugins ran on every API call the system would be terribly slow. Instead, a plugin registers on a
combination of the API call, the entity type of the record, and the attributes on the record.
As an example, assume that we have an Account entity with two attributes, Name and Description. If
we require that the Description is always based upon Name when an Account is created, it would be a
good idea to add a plugin that enforces this.
Figure 2.2 shows an example of how such a plugin is registered. The plugin is registered on Account
records. It will run on Create calls and will run pre-operation. Once the plugin is triggered, it will run
the Execute function, more on that later. Additional restrictions are placed with AddFilteredAttributes,
such that the plugin will only be triggered if the record that is being created has a value for Name.

There are several other aspects of plugin registration in D365, but these will not be described in this
report.
Executing a plugin
The plugin registration deﬁnes a function that is called once the plugin is triggered. In the above example,
that function was Execute. This function can be written in any .NET language, but is usually written in
C#. This function has a single parameter of a context. This context contains a lot of information related
to the transaction. This report will only focus on a few aspects of that context.
The ﬁrst aspect is the ability to get the record that is going to the operation. This is done by accessing
the InputParameters of the PluginContext. The third line in ﬁg. 2.3 shows how this is done.
Figure 2.3: An example of how to get the record and the depth from the context in a plugin.
This returns a generic Entity object. For our registration, we know that the record will always be of
type Account, thus the object can be cast to that directly. Figure 2.4 shows the Execute function of our
registration. It fetches the account and sets the description. Since the plugin is registered on pre-operation
the API call is modiﬁed directly and the description will be set on the Account once it is created in the
Operation step of the transaction.
Figure 2.4: An example of an execute function in a plugin registration
The other information shown in ﬁg. 2.3 is the Depth of the PluginContext. Since plugins can trigger
each other, it is useful to record how deep the current execution is. The depth starts at 1 and can go no
deeper than 7. If the depth goes deeper than 7, then D365 will throw an exception and the transaction
is rolled back. The eﬀects of this will be shown, when it is shown how plugins make API calls.

Deﬁning an Entity
As in regular database terms, entities are tables in the database and records are rows in that table. An
entity can have several types of attributes, where each attribute corresponds to a column of the table.
Figure 2.5 shows the diﬀerent types an attribute can have. This report will only deal with Whole Number,
Single Line of Text, and Lookup. A Whole Number is an integer, a Single Line of Text is a string of
maximum 4000 characters, and a Lookup is to be understood as foreign key relation to the referenced
entity. D365 also supports foreign key constrains, but these will be ignored in this report.
Figure 2.5: The diﬀerent types of attributes that can be deﬁned on an entity.
Calling the API
The API is called by invoking methods on an IOrganizationService. Figure 2.6 shows the diﬀerent methods
that are available. The more than 200 requests can be called using the execute method. However, some
operations, like the CRUD operations, are readily available.
Figure 2.6: The methods that are available on an IOrganizationService.
The methods also require diﬀerent parameters. A Create call needs a record, but a Retrieve only needs
the name and id of the record. This report will simplify this, such that all CRUD calls require the record.
The same goes for the response to the API call. The Create request only returns the id of the record
it created, whereas Retrieve will return the entire record. However, Update and Delete return nothing.
Therefore, the calls will be grouped into two, ones that return a record and ones that return nothing.

2.2
Deﬁning Flow
A program written in ﬂow consists of three parts.
• Deﬁnition of entities in the database.
• Deﬁnitions of each Flow in the system.
• Actions that are done to the system.
Each of these parts will be described in detail in this section, as well as their relation to D365.
Deﬁning entities in Flow
When you deﬁne an entity in D365 you give it a name and deﬁne the attributes. Each entity deﬁnition
in Flow starts with the name of the entity, followed by the attributes deﬁned on the entity. It looks like
this
ent Account,[string name, int accountNumber, Account parentAccount]
As described previously, this report will only deal with Whole Number, Single Line of Text, and Lookup
attributes. Whole Number is a 32-bit integer in D365, so Flow models this as the attribute type int, which
is the type of the accountNumber attribute in the example. Single Line of Text is a string of a maximum
of 4000 characters. This is modelled as a string in Flow, which resembles a sequence of characters, which
is the type of the name attribute in the example.
A Lookup attribute refers to a speciﬁc relationship between two entities. The value of a Lookup is an
EntityReference that references a speciﬁc record. It contains a LogicalName, that is the name of the entity
type the record has, and an Id that is the Id of the record. Flow models this as the type name, where
name is the name of an entity. In the above example, the parentAccount attribute has type Account, thus
it is a lookup to the Account entity, e.g. a self-reference in this case.
Deﬁning plugins in Flow
A plugin in D365 is represented as a ﬂow in Flow. A deﬁnition of a ﬂow is seen in the example below.
flow SetAccountDescription(Account,Create,Pre,[string name]) {
string prefix;
prefix := "Name is ";
reqEnt[description] := prefix + reqEnt[name];
}
The parameters of a ﬂow match the registration of a plugin. In this is example, the ﬂow is registered on
the Account entity. It triggers for the Create request, pre-operation, and only if the record has a value
for name. The body of the ﬂow, limited by the brackets, contains the deﬁnition of the function that is
called when the plugin is triggered.
To reference the record that is tied to the request, a special variable, reqEnt, is deﬁned by the system
in each ﬂow. The above example uses this variable to assign a new value to the description attribute.

Calling ﬂows
A system of plugins is completely reactive. Thus, the ﬁnal part of a Flow program exists to simulate a
user acting upon the system. This part contains declarations followed by statements and looks something
like this.
Entity<Account> acc;
acc[name] := "Some name";
acc := execute(Create,acc);
Here the variable acc is declared. The name of this empty entity is set to Some name and is ﬁnally
created. The speciﬁc API calls are replaced with a single construct called execute. This method takes
a request type, in this case Create, and a record, in this case account. This will trigger the valid ﬂows
in the same way a create call in D365 would trigger the valid plugins. As speciﬁed earlier, in order to
simplify the language, execute either returns a value or it does not. If it returns a value, it will return the
entire record. Thus, Create will return the record it created, which makes it assignable to the variable
account again.
The grammar of Flow
The grammar of Flow is loose based upon a project from the course 02242 - Program Analysis, where
the task was to design analyzes for a C like language called MicroC. The grammar is deﬁned as follows.
Programs : P
::=
ENT F D S
Statements : S
::=
x := e; | x[attr] := e; | S1 S2 | {D S}
|
if (b) {S} | if (b) {S1} else {S2}
|
return; | skip; | execute(req, x) | x1 := execute(req, x2)
Expressions : e
::=
n | ”s” | context.Depth | x | x.Id | x[attr] | e1 op e2
|
Entity(name, e) | EntityRef(name, e)
Boolean expressions : b
::=
true | false | !b | e1 op e2 | b1 op b2
Declarations : D
::=
int x; | string x; | Entity<name> x;
|
EntityRef<name> x; | D1 D2
Entities : ENT
::=
ent name, [AT] | E1 E2
Flows : F
::=
flow flowname(name, req, ES, [AT]){D S} | F1 F2
Attributes : AT
::=
int attr | string attr | name attr | AT1, AT2
Requests : req
::=
Create | Retrieve | Update | Delete
Execution Stages : ES
::=
Pre | Post
Operators : op
::=
+ | −| ∗| / | < | > | <= | >= | == | ! = | and | or
V ariable names : x
Flow names : flowname
Entity names : name
Attribute names : attr
Integer contants : n
String contants : s
The reader will familiarize themselves with the grammar through the deﬁnitions of the type system and
the program graph. Notice that it is not possible to retrieve multiple records. This is because it would
require support for arrays in the grammar, and that would be beyond the scope of this report.
10
==PAGE:12==
The implementation of the grammar is done using FsLex and FsYacc, which create a lexer and a parser
respectively. In there, the ambiguity of the grammar is handled by splitting up the rule. For example,
expressions are split up into 3 pieces to capture the order of operators in math.
This grammar has been implemented in reference to an Object-Oriented language. The style of Flow is
similar to the world of Aspect-Oriented programming. There are several ways of modelling this. The
authors Wang-Oliveira made a small functionally inspired Aspect-Oriented programming language [11].
They approached the grammar the same way this report has, in that the grammar does not connect the
diﬀerent ﬂows, but it is done indirectly. In their example, it is done through the deﬁnitions that evaluate
the program, in this report it will be done by deﬁning a program graph with semantics on top.
Before this can be done, it needs to be ensured that the program of the grammar is well-typed. This will
by done through the use of a type system, which will be deﬁned in the next section.

2.3
Deﬁning a type system
The grammar of a program is used to structure a string into an AST. Unless the grammar is very
descriptive, it does not prevent undesired programs that will fail at runtime. For example, it does not
prevent addition of diﬀerent types, such as 2 + ”cheese”. It is possible that this is allowed, but it is nice
when programs are well-typed, since errors can be caught during the compilation of the program. These
errors are found by checking the AST against a type system.
A type system creates an abstraction of the values in the program, that should be type checked. The
type system deﬁned in this report will work on the types that can be declared, which are integers, strings,
entity references and entities. The purpose of deﬁning these types is to prevent assignment and use of
values that does not match the deﬁned type. This means integer types can only be assigned numbers and
string types can only be assigned strings. The entity type is more complex. If a general Entity type is
used, then it will result in an incorrect analysis. To see this, assume the following entity deﬁnitions from
the grammar:
ent Account,[string name, string description]
ent Contact,[string name, int postalcode]
If the general Entity deﬁnition is used, then it will be possible to make attribute assignments of the type
x[description] := ”Kalle is a discount supermarket”; regardless of whether x is an Account or Contact,
since the type system can not distinguish the two deﬁnitions. If the Entity type instead refers to the
name of an entity deﬁnition, then the problem is solved. In the previous scenario, if x had the type
EntityContact then the type system would be able to check against the entity deﬁnition of Contact and
see that description is an invalid attribute. For similar reasons, an entity reference type should include
the name of the entity it references. This results in the following type deﬁnition.
τattr = int
| string
| Ref name
τ = τattr
| Entity name
Notice how the types are split into types that are valid for attributes τattr and all types τ.
The next thing a type system needs are a series of inference rules that deﬁne the allowed subset of deﬁni-
tions of the grammar. Each rule will deﬁne a single step in the unpacking of the AST. The deﬁnition of
each inference rule will depend on the structures from the grammar that they work with. For example,
the result of evaluating an entity deﬁnition should be a structure that knows which attributes are asso-
ciated with the entity. Additionally, evaluating a declaration should result in a structure that maps the
variable to the declared type. Using this thought process, the following types are deﬁned.
Γ : Names →γ
γ : Attributes →τattr
∆: V ar →τ
Where the entity environment, Γ, maps entity names to an attribute environment, γ. γ maps attributes
to their type, τ, and the variable environment, ∆, maps variables to their type, τ.
Using these structures, it is possible to think about the deﬁnition of the diﬀerent types of inference rules.
Consider an incomplete inference rule for P from the grammar:

? ⊢ENT :?
? ⊢F :?
? ⊢D :?
? ⊢S :?
⊢ENT F D S
This rule is a skeleton for how the groups of inference rules are connected. As previously discussed,
inference rules for entities should know about entity deﬁnitions and should return their addition to the
entity deﬁnitions. Thus, inference rules that work on ENT will have the following form:
Γ ⊢ENT : Γ
Since inference rules should only take a single step, the type system also needs inference rules for AT. To
motivate this, look at the ENT structure ent name, [AT]. It is not possible for the inference rule for this
structure to evaluate AT in a single step.
The inference rule working on AT should return an attribute environment. To do that it needs the deﬁned
entities, as discussed previously. This result in the following deﬁnition of inference rules on AT:
Γ ⊢AT : γ
Declarations should return the deﬁned variables, but since it is possible for variables to be entities it
should know which entities can be declared. Thus, inference rules that work on declarations will have
the following form:
Γ, ∆⊢D : ∆
Using these deﬁnitions, the incomplete inference rule can be updated.
[] ⊢ENT : Γ
? ⊢F :?
Γ, [] ⊢D : ∆
? ⊢S :?
⊢ENT F D S
There is no initial deﬁnition of entities, so an empty mapping is given instead, marked as [].
Flows and statements can not declare new variables or entities. Therefore, these inference rules will not
return any structures. Flows are run in isolation; therefore they only need the entity deﬁnitions, such
that the ﬂow can be executed and the registration of the ﬂow can be checked. Statements need both
entity deﬁnitions and variable deﬁnitions, for example in attribute assignment, x[attr] := e;, where the
variable type needs to be an entity and the attribute needs to be assignable to that entity. This results
in the following deﬁnitions for F and S.
Γ ⊢F
Γ, ∆⊢S
It is not trivial to evaluate statements. Looking at assignment, x := e;, it is clear that it will be impossible
to evaluate this inference in a single step, since there exists expressions like e1ope2. Therefore, the type
system needs inference rules for expressions. Looking at the expression that fetches an attribute from
an entity, it is seen that expressions need information about both Γ and ∆. The result of evaluating an
expression is a τ, since expressions return values and τ can describe each type of value that should be
type checked. Thus, the deﬁnition for e is as follows.
Γ, ∆⊢e : τ
There is a ﬁnal structure in statements, that can not be evaluated in a single step. Looking at the if..then
structure, if (b) {S0}, it is seen that it would be impossible to evaluate b in a single step, since it contains
structures like b1opb2 and notb. Boolean expressions compare expressions, which means they need Γ and
∆to be evaluated. The Flow language does not allow declaration of boolean values, which is why they
are not part of τ. Therefore, the inference rules do not need to return anything. This results in the
following deﬁnition for inference rules on boolean expressions.
Γ, ∆⊢b

Using these ﬁnal deﬁnitions, the inference rule for P can be completed.
[] ⊢ENT : Γ
Γ ⊢F
Γ, [] ⊢D : ∆
Γ, ∆⊢S
⊢ENT F D S
The rest of this section will describe the deﬁnitions in each group of inference rules.
Inference rules for ENT
ENT has two diﬀerent structures. The inference rules for ent name, [AT] is deﬁned as follows:
Γ ⊢AT : γ
Γ ⊢ent name, [AT] : Γ[name 7→γ]
This rule evaluates the AT structure and updates Γ such that the name of the entity maps to the result
of evaluating AT. This means that any previous mappings for name in Γ are erased.
The inference rule for E1, E2 is deﬁned as follows:
Γ ⊢E1 : Γ1
Γ1 ⊢E2 : Γ2
Γ ⊢E1, E2 : Γ2
This inference rule should combine the mappings of E1 and E2. This is done by evaluating E1 and pass
that mapping onto the evaluation of E2. This also means that circular references between entities are
impossible, since E1 needs to evaluate fully before E2 is evaluated.
Inference rules for AT
AT has four diﬀerent structures. The inference rules for int attr and string attr are very similar, in that
they can directly be assigned a type. Their deﬁnitions are as follows:
Γ ⊢int attr : γ[attr 7→int]
Γ ⊢string attr : γ[attr 7→string]
The mapping of the attr is simply updated with the new type. The inference rule for name attr is
Γ(name) is defined
Γ ⊢name attr : γ[attr 7→Ref name]
The rule maps the attribute to an Entity of name but can only do so if there exists an entity deﬁnition
for name. The inference rule for A1, A2 is as follows:
Γ ⊢AT1 : γ1
Γ ⊢AT2 : γ2
dom(γ1) ∩dom(γ2) = ∅
Γ ⊢AT1, AT2 : γ1[γ2]
This diﬀers from the deﬁnition for E1, E2, since the inference rule can not pass the result of A1 to the
evaluation of A2. Therefore, the rule evaluates the two types separately, that gives the results γ1 and
γ2. These two mappings are then combined using the notation γ1[γ2]. Problems can arise if the two
mappings contain overlapping keys and are combined, thus the ﬁnal check ensures that the domains of
the two mappings are distinct from each other.
Inference rules for F
The inference rule for flow f(name, req, ES, [AT]){D S} is as follows:
Γ ⊢AT : γ
∀attr ∈dom(γ) : γ(attr) = Γ(name)(attr)
Γ, [reqEnt 7→Entity name] ⊢{D S}
Γ ⊢flow f(name, req, ES, [AT]){D S}
14
==PAGE:16==
The ﬁrst two conditions seem complicated, but combined they describe that all attributes in AT are
attributes on the entity deﬁned by name. The ﬁrst condition gets the attributes deﬁned by AT. The
second condition checks that all attributes on AT have the same type as those on the entity deﬁnition of
name.
The last condition makes sure that the code inside the ﬂow is well-typed. Since the context entity is
passed along, the ﬂow adds that entity to the variables that are available before the check is done.
The inference rule for F1, F2 is as follows:
Γ ⊢F1
Γ ⊢F2
Γ ⊢F1, F2
This rule simply ensures that both sub structures are well-typed.
Inference rules for D
These inference rules are very similar to the rules for AT. Again, the inference rules for int x; and string x;

Implementation
The implementation follows the rules deﬁned in this section. It is located in TypeChecker.fs and starts
with the function checkIfWellTyped. This function matches the inference rule for a program.
The implementation uses the expressive nature of F# to handle each case explicitly. The inference rules
only describe the positive cases, but a well-made program should return meaningful errors. Therefore,
each type is matched explicitly instead of matching by a wild card at the end.

2.4
Deﬁning the program graph
This section will describe how an AST from the grammar is turned into a program graph. This is done
by deﬁning a function called edges. This function takes two states and a program written in Flow. It
returns a list of edges, as well as type information for each variable in each state in the program. What
this means will become clear throughout the section.
To do this, some additional types have to be deﬁned. A program graph consists of states, Q, and directed
edges that are labeled with actions. The actions are listed below.
Actions =
int x
string x
Entity<name>x
EntityRef<name>x
x := e
x[attr] := e
return
skip
execute(req, x)
x1 := execute(req, x2)
b
enter
exit
ExecuteFlows(q◦, q•, req, x)
ReturnFlows(q◦, q•, x)
Most of these are directly derived from the grammar of Flow. The purpose of these actions will become
clearer as the program graph generation is deﬁned.
These actions are used in an edge, which consists of a pair of states as well as an action.
Edge = Q × Actions × Q
This is to be understood as the start state, the action that is performed, and the state that is achieved
as a result of applying that action.
For further analysis, it is desired to relate each state in the program graph to the type information that
is available at that point in the program. This is to be captured in a state type environment, Λ.
Λ = Q →∆
The program graph is generated from an AST of a Flow program, just like the type checker. Therefore,
it is possible to relate the two. The values in the Λ are to be deﬁned to be exactly as they are in the type
checker. When this is clear from the deﬁnition, the argument for this will be omitted. If it is not trivial,
why the types are the same, then an argument will be provided why they match.
The ∆relates variables to τ. The Λ will use the same types, but one addition is needed. There are points
in the program where it is known that a variable is an entity, but not which type of entity. Therefore, a
polymorphic version of the entity type is added. This means the new deﬁnition for τ is as follows.
τ = τattr
| Entity name
| Entity α

This leads to the deﬁnition of the edges function.
edges(Q ⇝Q)[[ENT F D S]] ∈P(Q × Actions × Q) × Λ
The rest of this section will describe how the program graph is generated. In order to build up to the
whole graph, it will be shown how diﬀerent parts of the program is turned into a program graph, which
is then used in the ﬁnal deﬁnition for edges.
Declarations
Declarations allow the programmer to deﬁne variables and will therefore modify the Λ. Declarations are
turned into a program graph using the following edges function.
edges(Q ⇝Q)[[D]] ∈P(Q × Actions × Q) × Λ
There are 4 basic declarations int x;, string x;, Entity<name> x;, and EntityRef<name> x;. These

Chapter 3
Semantics of Flow
This chapter will present the semantics of the actions in the program graph. First an abstraction of the
system will be presented. This is followed by the deﬁnition of the semantics, Finally, a partial proof will
attempt to show that the deﬁnition does not break the type constraints made by the type system.
3.1
Abstracting the system
Semantics are deﬁned in relation to a real system. In the case of Flow it is a program and a database.
To deﬁne the semantics, it is important to choose an appropriate abstraction of the actual system. This
abstraction should be representative of the actual system and be as simple as possible.
An abstraction of integer values could be a label, int. If this label was used, then the program would
know it would be any integer value that can be described by 32 bits. The values that should be abstracted
are known based upon the deﬁnition of the type system, τ. Following this thinking, values in the program
can be abstracted to one of the following four labels.
V alattr = int
| string
| Ref(Names × string)
V al = V alattr
| Names × (attr ,→V alattr)
Three of the labels are valid attributes, int, string, and Ref(Names × string).
Integer values are
abstracted as int, as described. Any string value is abstracted as string. All EntityReference objects are
abstracted as Ref(Names × string), where an EntityReference with logicalname name and id entityId
is represented as Ref(name, entityId).
The ﬁnal value is an abstraction for all Entity objects, which are abstracted as Names × (attr ,→V alattr).
An Entity object with logicalname name and a single attribute value description with the string value
some desc is abstracted as (name, [description 7→”some desc”]).
These values are used in both the local memory and the database. The local memory can be abstracted
as a mapping from variables to values.
Mem = V ar ,→V al
Where Var is a collection of all variables deﬁned in the program, including reqEnt. This works if all
variables are global. However, ﬂows are executed without the knowledge of other ﬂows. Therefore, the

abstraction needs to handle scope. Following the deﬁnitions of procedures from Formal Methods [7], it is
seen that this can be handled by representing the memory as a stack of memory frames, where a frame
is a mapping from variables to values.
Mem = Frame∗
Frame = V ar ,→V al
This allows the semantics to add a new frame on enter and pop it again on a exit, thus handling scoping
of variables.
However, there is still an aspect of the program that needs to be handled. The execution of ﬂows can
be seen as series of function calls. In the program graph, these calls are made using ExecuteFlows(..)
and returned to the correct state using ReturnFlows(..). These two actions need information about the
program graph in order to store where they come from and should return to. The ﬁrst two arguments of
these actions are two states, which serve this purpose, but it is impossible to store this information in a
frame, given the current abstraction.
Thus, a unique variable, ∗, is used in order to reference this information. This unique variable will map
to a triple (Q × Q × Requests). The ﬁrst two values are the states in the program graph, that will be
used as a reference. The third value is the request that was made to start the execution of ﬂows. The
information of the entity that was passed as an argument will be stored in a variable called reqEnt.

3.4
Well-typed programs can not go wrong
A man called Robin Milner proved in 1978 in his paper A Theory of Type Polymorphism that well-typed
programs can not go wrong [8]. He did this by deﬁning a type system and semantics for his language
called Exp. He proved that by following the semantics on a well-typed program, the program would
remain well-typed, thus not introducing any runtime type errors.
This section will prove a similar, but weaker, theory for the type-system and semantics deﬁned in this
report. This theory will work by comparing the two structures Val and τ. An English wording of the
proof is, whenever an action in the program graph is evaluated using the semantics, where the system
was well-typed at the start state, the system will also be well-typed at the end state.
There are several things to unpack in that sentence. The ﬁrst thing is to deﬁne, when a state in the
program graph is well-typed. The ﬁrst step is to establish a relationship between Val and τ. The attribute
values and types are easily related. All integer values have the type integer, all string values have the
type string, and all references with the entity name name have the type Ref name.
This relationship will be deﬁned using the following form of logical entailment.
Γ ⊢V al : τ

Chapter 4
Analysis of Flow
This chapter will deﬁne improved versions of the two analysis’ Reaching Deﬁnitions and Available Ex-
pressions. This report is not the ﬁrst to deﬁne analysis’ on programs that look like Flow. The domain
of Flow is similar to the domain of Aspect-Oriented programming. There are several papers that have
used program analysis techniques on that domain. The article from Bouterra-Bounour studied slicing,
which one of the things this report seeks to accomplish as well [5]. They tried several ways of slicing
a program but ended up deriving the result from an analysis similar to a Reaching Deﬁnitions, thus
this report will use the same approach. The same idea of using dependencies between states is used
by Syarbaini-Abdul-Fazlida, where they seek to create valid Control Flow Graphs for Aspect-Oriented
programs [4].
The problem with both articles is that they both use Control Flow Graphs as a representation. It is easier
to represent graphs with function calls, thus also aspect-oriented programs, in this paradigm. However,
the Formal Methods book has shown that analyzes become much simpler and clearer if the analyzes are
deﬁned on Program Graphs instead [7]. Thus, this report will use the techniques from there instead.
First it will be shown that the standard analysis is too imprecise to show any relevant results. Then
an appropriate analysis framework will be shown, followed by deﬁnitions of Reaching Deﬁnitions and
Available Expressions in that framework. Finally, a discussion will show how to improve the accuracy of
the analysis’ even further.

Reaching deﬁnitions
A Reaching Deﬁnitions analysis discovers which assignments may have been made and not overwritten,

whenever plugins are called, and the assignment to that is located either outside of ﬂows or inside a

• E is the set of extremal states
• i ∈L is an extremal value for the extremal states
• f is a mapping from edges to transfer functions in F

better performance dramatically.

Chapter 5
Security properties of D365
D365 is a large enterprise product from Microsoft. Enterprise products are expected to be secure and

The ﬁrst section will deﬁne how the security properties can be expressed in the Flow language. The

5.1
Abstracting security in D365
D365 handles security in many ways, but the core the security model is security roles. The explanation of
security in the report will be largely simpliﬁed, to make it simpler to analyze. It will still be representative
of the security model.
Security roles are added to users. Users can have multiple security roles. Security roles can be seen as a
collection of privileges. A privilege is deﬁned for a operation and an entity, and has a depth. Figure 5.1
shows how a security role is visualized in D365. Each row is an entity, each column is an operation and
the circle deﬁnes the depth of that privilege.

5.3
Data level security
The previous sections have been showing possibly undesirable situations that are not caught by the
security model in D365. However, D365 is still considered secure, so what is the discrepancy?
D365 only tries to enforce security on a schema level. The "errors" shown in the previous sections are
not considered errors with regards to the schema. In reality, the security roles do not say whether data
from John should be conﬁdential with regards to Hans and whether records that John owns lose their
integrity if they have data from Hans.
Therefore, D365 would need another level of security - hierarchical security roles.
This hierarchical
structure is not about extending security roles or inheriting privileges. It is about deﬁning a partial order
that can be used in an information ﬂow analysis. If one only seeks to enforce the standard policy, then all
security roles are in the same node. However, if John and Hans have diﬀerent security roles, and John’s
security role has a higher order, then the information ﬂow could be able to prevent the previously shown
issues. This is similar to how information ﬂow is handled in the Formal Methods book [7]. In that book,
ﬂows are assigned by asserting roles, this is similar to how the caller variable is used. The diﬀerence is
that assertions are direct, where the ExecuteFlows action sets the caller indirectly.
In order to do this, attribute values need to be tagged with a reference to the caller that deﬁned them.
Therefore, the V alattr is changed to the following.
V alattr =

By updating the semantics as shown in this section, it is possible to verify security properties using
information ﬂow. This report does not contain a deﬁnition for a declassify operator. Such an operator

will then allow the higher-tiered user to make a subset of the information he/she knows available to the
lower-tiered user. Such an operator should be deﬁned in a similar way as deﬁned in the Formal Methods
book [7].
The reference-monitor will add additional overhead performance wise compared to the original semantics.

5.4
Static security analysis
The previous section deﬁned a reference-monitor that ensures several security properties. The purpose
of a static security analysis is to ensure some of those security properties before the program is run, such
that the overhead of adding the reference-monitor becomes smaller.
The fastest static analysis is one that works purely on the AST of the program. Such an analysis would
look similar to the type system, where the analysis succeeds if the inference rules work for the program.
The approach of creating a type system for capturing information ﬂow has been tried and done by both
Volpano-Smith-Irvine [10] and Myers-Liskov [9]. Thus, it makes sense to extend the inference rules of the
type system.
For the same reasons as with the reference-monitor, it will not be described how the security roles,
business units and so on are stored. Assume that they are stored in a new environment, Ξ, just like
variables are stored in the ∆. This new environment is sent to all inference rules for statements, since
only statements can make execute calls.
The caller is a declaration, so it is handled in the same way that declarations are handled by the type
system.
Γ, ∆⊢caller = username; : ∆[caller 7→username]
This ensures that the ∆contains the deﬁnition of the caller, if it is deﬁned by a declaration. This will
allow the inference rules for execute(req, x) and x1 := execute(req, x2) to reuse the HasRight functions
that is used by the reference-monitor. This results in the following new deﬁnitions.
req ∈{Update, Delete}
∆(x) = Entity name
Ξ ⊢HasRight(∆[caller], name, req)
Γ, ∆, Ξ ⊢execute(req, x)
req ∈{Create, Retrieve}
∆(x1) = Entity name
∆(x2) = Entity name
Ξ ⊢HasRight(∆[caller], name, req)
Γ, ∆, Ξ ⊢x1 := execute(req, x2)
The HasRight function has been modiﬁed to take a request instead of an operation. The point is to
group the multiple HasRight calls for each request together, such that the inference rule is simpler. It
will not be possible to statically analyze the ﬂow of values, since the ﬂows are not connected to each
other through the program graph, not through the grammar.
However, it is also not possible to remove the HasRight calls using these additions to the type system. The
problem with this setup is that the caller can also be set from the ExecuteFlows action. This is not part
of the grammar, and thus a static security analysis on the AST will not be able to remove all HasRight
calls, but it will be able to remove some of them. If the caller variable is set through the ExecuteFlows
action, then the ∆will not have an entry for caller. Thus, it is possible to diﬀerentiate between the two,
such that the performance overhead is reduced. This report will not deﬁne the modiﬁcations needed to
diﬀerentiate between them, but it could be done by having the declaration and ExecuteFlows use diﬀerent
caller variables, and have the declared variable take precedence over the ExecuteFlows one.
One could imagine an analysis that functions on the program graph that could reduce the amount of
HasRight calls, but since such an analysis would never be able to evaluate boolean expressions precisely
I can not imagine that it would be able to remove all HasRight calls, which would likely render it
unnecessary.
This chapter has shown how the security properties of D365 can be expressed in Flow, and how they can
be improved using information ﬂow. It has been shown how a reference-monitor can prevent illegal ﬂow
of values. Finally, this section has shown that a subset of the reference-monitor checks can be replaced
by a static security analysis.

Chapter 6
Conclusion
The goal of this report was to apply techniques from the academic world of formal methods to the domain
of D365. To accomplish that, a workﬂow language called Flow was deﬁned. This language was deﬁned
to be as close as possible to the plugin model in D365.
The language was deﬁned using a grammar, to which a type system has been deﬁned. The type system
resembles the restrictions one experiences by developing plugins in D365. It has also been shown how
to turn an abstract syntax tree from the grammar into a program graph, that includes a mapping that
maps variables deﬁned in states to the types of those variables.
A semantic deﬁnition has been deﬁned in relation to the program graph deﬁnition. It has been shown
that it works by running example programs written in Flow on an interpreter that uses the semantic
deﬁnitions. In addition to that, a partial proof is used to argue that the semantic deﬁnitions preserve the
type restrictions made by the type system.
Two analysis’ have been deﬁned on top of the semantics, a Reaching Deﬁnitions and Available Expressions
analysis. It has been shown how the Reaching Deﬁnitions analysis can be used to ﬁgure out which ﬂows
call each other, which in turn can be used to ﬁgure out which plugins in D365 call each other. The
Available Expressions analysis is used to reduce the amount of network calls, by ﬁguring out whether the
records that are used in those network calls have been modiﬁed. The correctness of these analysis’ has
been shown by running examples on an implementation of each of them.
The security properties that are guaranteed by D365 have been shown to be insuﬃcient when they are
compared to the security properties that can be enforced through information ﬂow. It has been shown
how to modify the semantics such that the security properties of both D365, and the ones guaranteed by
information ﬂow, are ensured. This was done by adding static checks for D365 and a reference-monitor
for information ﬂow. The correctness of the information ﬂow has been argued by running examples of an
interpreter that is based on the modiﬁed semantics.
There are several aspects of D365 that were not included in this report, and only a subset of the ways
of adding business logic have been analyzed. If some critical aspects that invalidate the ﬁndings in this
report were omitted, then this entire report would be useless. That is why the plugin model was chosen
as a baseline for Flow, because it is the most feature rich way of adding business logic. All other methods
can be expressed in the plugin model.
Some attributes were left out, but since attributes are handled generically, it does not matter what speciﬁc
type they have. The most interesting type, the lookup type, is included. The reason this is special is
because it forms a foreign key in the database. All other attributes types are just columns with diﬀerent
values.
A core aspect that has been omitted is the management of multiple records at the same time. This is
a common use of the system, and therefore it can be considered critical. It is possible that some of the
work done is this report becomes invalidated if multiple values are considered, but I believe it is unlikely.
The handling of multiple values can be considered as handling arrays. Analysis’ either become more
91
==PAGE:93==
expensive or more imprecise, but at the core it is the same. It is known based upon several studies of
formal methods in relation to arrays that it can be handled with great precision.
One aspect that I would like to put more research into is the validation of unknown code. It is common
to use third-party solutions in D365. A big assumption that is made in Flow is that all code is known.
Solutions with unknown code is common in D365. Today, code is executed blindly with the assumption
that it was written by competent people. These people would probably lose their business if their solutions
were proven to be incorrect or insecure, but when the future of large corporations are on the line, it might
matter to avoid unknown code.
This report has shown several formal method techniques that can improve the development of plugins
in D365. In order to use them they will have to be ported from Flow to the entire .NET language.
This is a big task, one which is most likely only realistic for Microsoft to do. Such a task might also
reveal problems that were not discovered by this report. Flow is an abstraction of what is possible in
C#. Aspects such as concurrency, network communication etc. are not part of Flow and could cause
problems. However, Object-Oriented languages have been greatly studied in formal methods, so I believe
that there exist known solutions to those potential problems.
Although I would love to use the techniques shown in this report, they are not useful for everyone.
When large software projects are developed there is a trade-oﬀbetween cost and safety. Applying formal
methods techniques are, in my experience, more expensive than other techniques, such as testing, code-
reviewing, etc. Thus, the results of this report are useful if security guarantees are signiﬁcantly important
to a company that uses D365. These types of companies include very large corporations or the military.
The trade-oﬀwould be to consider whether the system should be built from scratch, or whether it makes
sense to extend D365 with the features presented in this report. Before these companies did not have
a choice, I hope at some point Microsoft will implement the ﬁndings of this report such that they can
decide for themselves.

A program analysis is only useful whenever its speciﬁcation captures the formal semantics of the language
of interest, we call such an analysis is semantically sound. In this section we show a common approach
for proving analysis correctness with the use of abstraction functions.
Whenever specifying a lattice L that describes the properties of interest on concrete values from a set M,
it is important to have a clear deﬁnition of which property l ∈L is best for describing a value m ∈M.
For this reason it is useful to deﬁne an abstraction function
β : L 7→M
to show the relation between the two domains of interest.

1.5
Private remarks
During the entire three week period we were mostly focused in developing a context-sensitive analysis for
the analysis of integers. Concequently our choice of abstract memories was motivated by that, since we
had to be as close to the semantics as we could. We then moved to the bit-vector frameworks and being
motivated by an example we thought that our context-sensitive framework could also be adapted to it.
However, there are several things that we are concerned about and in this section we discuss about them.
1.5.1
Bit vector framework on other than reaching deﬁnition
Our "Bit-vector" frameworks has really been tested only for the reaching deﬁnition analysis, and we
do not exactly know how it would work on a backward or must analysis. By the design of our analysis
framework one can see that we are interested in properties that are related to the assigned variables (e.g as
in the reaching deﬁnitions analysis). This can be understood by the way we have design our speciﬁcation
for the new construct ⌊x⌋:=a where we kill and gen something at the frame below rather than the topmost
one. Therefore for an analysis that talks about properties about the arithmetic expressions our feeling is
that we shouldn’t be looking at the frame below.
