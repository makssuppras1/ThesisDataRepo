Summary
AIF[4] is an approach for verifying stateful distributed systems such as web servers
and crypto APIs. It is based on an abstract interpretation approach that represents
an over-approximation of all facts that can ever become true in a system. The problem
is first encoded into a problem of evaluating Horn clauses, and different tools can be
employed to check whether an attack predicate can be derived in the Horn clauses,
namely ProVerif, SPASS and a direct fixed-point computation. Although analysis
results indicate whether the protocol is secure or not. If an attack is found, it is
not immediate for a user to understand how the attack state is reached, because the
result is with respect to the Horn clause encoding and not with respect to the original
specification. Moreover, since the Horn clauses represent an over-approximation, the
abstraction may have an attack while the original is secure, i.e., a false positive. In
the positive case, the user doesn’t get any feedback, i.e. why the protocol is secure.
This thesis is about presenting the result of the abstract verification to the user
in both negative and positive cases. In the negative case, We use LaTeX command
to display the proof tree of the reconstructed attack trace and use the attack trace
from the abstract level as a heuristics trying to search the attack state in the original
model. In most cases, if an attack is found in the abstract level, our tool is also able
to find the attack in the original model except the false positive case, i.e. an attack
is found in the abstract level but the original model is secure. In the positive case,
we making proof understandable to the human. To do that we construct the term
implication graph with ordering and reduced fixed-point by using term implication
graph. For each rule, we also give the most general substitutions under which the
rule can be applied. This allows the user to check and understand the proof.

CHAPTER 1
Introduction
Tools based on over-approximation like ProVerif have been very successful in the ver-
ification of stateful security protocols such as web servers [4]. It is an abstraction
approach that over-approximates every possible protocol behavior by a set of first-
order Horn clauses, rather than considering the set of reachable states[5]. This avoids
the state explosion problem that the number of the reachable state grows exponen-
tially with the number of sessions. The approach assumes that all information that
is dealt with is either local to single sessions of the protocol or immutable like shared
among a number of sessions like agent names and long.term keys.
AIF[4] has overcome the limitations by allowing one to declare a fixed number
N of sets. Then one can specify a transition system with an unbounded number
of the constants. These constants can be added to or removing from each of sets
upon transitions, and transitions can be conditioned by set memberships [5]. The
idea is that one can abstract these constants by their set membership. The AIF tool
generates a set of Horn clause using this abstraction, and can use ProVerif, SPASS,
or a direct fixed-point computation to check whether a distinguished symbol attack
can be derived from the corresponding Horn clause model. Although analysis results
indicate whether the protocol is secure or not. If an attack is found, it is not immediate
for a user to understand how the attack state is reached, because the result is with
respect to the Horn clause encoding and not with respect to the original specification.
Moreover, since the Horn clauses represent an over-approximation, in the negative
case, the abstract level may have an attack while the original model is secure. i.e, a
false positive. In the positive case, the user doesn’t get any feedback, i.e. why the
protocol is secure.
The contribution of this thesis is to present the result of the abstract verification
to the user. We offer new ways of presenting the results to the user in a clear way
in both negative and positive cases. To do that, we first defined an output format
for the fixed-point collaborated with Zhenwen Sun[7], This output format is able for
us to reconstruct the attack trace if an attack fact is found in the fixed-point. In
the negative case, we make an attack understandable for the user. We used LaTeX
command to display the proof tree of the reconstructed attack trace from the abstract
level. Then by using heuristics from the abstract level to search for the attack state in
the original model. If without such heuristics but by directly searching for the attack
state in the original model may result in the number of the reachable state grows
exponentially. Based on the heuristics derived from the abstract level to guide the

In this way, it may miss some attacks in the original model since we use the heuristics

rule. In this way, the user is able to check the proof and understand the proof. In
chapter 4 we described the testing results of our tool.

CHAPTER 2
Preliminaries
AIF is an abstract interpretation approach that represents an over-approximation
of every possible protocol behavior by transfer to a set of first-order Horn clauses,
rather than considering the the set of reachable states. The AIF framework consists
of AIF specification language and a translator from AIF to first-order Horn clause
that incorporates the set-abstraction technique[6]. It is the first tool allows one to
declare a fixed number of sets in a protocol. Each set stores one type of value and
values are considered equal if they belong to the same set.
In this chapter, we give a run example of an AIF specification and introduce the
set-based abstraction of AIF as well as the state transition principles of AIF rules. We
also introduce the most general unification algorithm. It is the basis of our unification
algorithm(symbolic fact[7]) we will describe in the later chapter.

We use the keyserver example form [4].
In this example we consider three agent
types: honest users (Honest), dishonest users (Dishon), and server (Server). Two
union types: the type User is the union of Honest and Server and the type Agent is
the union of User and Server. The operator ”...” is to declare that the type contains
a countable number of elements.
Honest = {...}
Dishon = {...}
User = Honest ∪Dishon
Server = {...}
Agent = User ∪Server
Each agent has its own database which contains all information that this agent has
to maintain over a longer time. For instances, in keyserver example, the user keeps a
database of all its valid public and private key pairs that are currently registered in
the with the Server. The private key is denoted as inv(k) that belongs to public key
k. However, it’s sufficient to represent the database entries only by the public key k
since for every public key, the corresponding private key is unique. We thus denote a
key membership of a user’s keyring as k ∈ring(A), where A is represent one of the
user type.
The state transition system is defined by a set of rules.
The first rule in our
example allows a user to register a public key with a server out of band:

2.2
Set-Based Abstraction
We abstract the fresh data according to its membership in the used set. For instance,
if we have three sets s1, s2, and s3, we can abstract all elements that are contained
in s1 but not in s2 and s3 into one equivalence class denoted as val(1, 0, 0)[4]. In
the keyserver example, we have three sets s1 = ring(A), s1 = db_valid(S, A), and
s1 = db_revoked(S, A), where the variables S and A are the user-defined types.e.g.,
the variables S and A are types of Server and User. The val(ring(A), 0, 0) represent
all the public keys that has been created by user A but not register yet in server
database as valid or revoked keys. Therefore, the abstraction model does not distin-
guish between different keys that have the same status in terms of set-membership.
In particular, when the set membership of a constant a changes from the abstract
value a to the abstract value a′, then for every derivable fact f[a] that contains a
also f[a′] is derivable[4] e.g.,2.11. Therefore, we need an extension with a new kind
of rule that can exactly express f[a] ⇒f[a′] for every context f[.] and formalize in
paper [4]. This kind of rules is different from an algebraic equation such as a ≈a′,
because f[a′] does not necessarily imply f[a]. It also different from a rewrite rule,
because f[a] is not replaced by f[a] but are both hold.

positive set conditions. For instance, the state 2.12 have two ground facts and two

iknows(inv(PK)) · PK ∈db_revoked(s, a) · NPK ∈db_valid(s, a)

A transition rules r has the form[6]:
LF · S+ · S−= [F] ⇒RF · RS
where LF and RF are the left-hand and right-hand side facts, R+ and RS are sets
of positive conditions, R−is a set of negative conditions. F is a set of variables which
are type of V alue and they are not occur in any facts, positive conditions, or negative
conditions before. The left-hand side of a rule describes whether a rule can be applied
in the a state. The right-hand side of the rule describes the changes to the current
state after the rule applied.
To apply a rule r in a state S, we first unify the left-hand side facts of the rule
with all facts in the state S to get substitution σs. e.g., applying the rule 2.14 to the
state 2.12, we can get a grounding substitution σ = [PK 7→pk, NPK 7→npk]. The
state S can reach to a new state S′ if:
• The left-hand side facts LFσ and positive condition R+σ are present in the
state S.
• None of the negative conditions R−σ hold in the state S.

• The fresh constants Fσ do not occur in state S or the AIF-ω specification.
In this example all of these conditions hold, therefore, the state S can reach to

r(X1 : T1..., Xn : Tn) = LF · S+ · S−= [F] ⇒RF · RS
where X1, ...Xn are the variables occurs in the rule r and T1, ...Tn are their types.

CHAPTER 3
Problem & Approach
AIF is based on an abstract interpretation approach that presents an over approxi-
mation of all facts that can ever become true in a system. The AIF tool generates a
set of Horn clauses using this abstraction, then the different tools can be employed
to check whether an attack predicate can be derived in the Horn clauses, namely
ProVerif, SPASS, and a direct fixed-point computation. Although analysis results in-
dicate whether the attack predicate can be derived in the Horn clauses, the problem
is that it is not easily for a user to understand how an attack predicate is derived,
because the result is with respect to the Horn clause encoding and not with respect
to the original specification.
Moreover, since the Horn clauses represent an over-
approximation, the abstraction may have an attack while the original specification is
secure. In the positive case, if an attack cannot be derived from the abstract level
then it proofs that the original model is secure.
Our approach is that we first define an output format for the fixed-point so that
we are able to reconstruct the attack trace from the abstract level if an attack fact
is found in the fixed-point.
In the negative case, i.e.
an attack fact is found in
the fixed-point, then we reconstruct the attack trace in abstract level and generate
LaTeX command to display the attack tree in LaTeX. We use the attack trace from
the abstract level as a heuristics to search for the attack state in the original model.
Because of the abstract level is over-approximation of the original model, therefore if
we would found an attack in the original model, then we proof that the attack is indeed
occurs in the original model.In the positive case, we sort the term implications with
ordering, remove the redundant facts in the fixed-point, and classified them according
to the form of the fact so that the facts have the same form grouped together and
display to the user. Finally, in order to convenes the user that the attack fact is not
derivable, we implement an analyzer to find most general substitutions for every rule
under which the rule can be applied. This allows the user to check and understand
the proof better.

3.1
An output format for fixed-points
AIF has only one built-in fact symbol attack [6]. The user use the attack to specify
an attack rule in a protocol. Therefore, to determine whether a protocol is secure, we
just need to determine whether an attack fact can be derived from the specification.
The fixed-point contains all the facts that can be generated by the rules defined in
the specification. Thus we can search every fact in the fixed-point to see whether
the attack fact is contained in the fixed-point. If it is, then attack fact is derivable.
Otherwise, the protocol is soundness, i.e. the attack cannot be derived.

3.3.3.1
Filter out redundant facts
In the previous section, we have described how to filter out the redundant facts that
are directly generated by the term implication and introduced the algorithm 18 which
is used to check whether the one fact can be derived by the another. In this section,
we introduce an algorithm 19 which is used to further reduce the redundant facts in
a list. The idea is that we add a flag to the end of the list to indicate that is the
last fact in the list. Then we take out the first element of the list compared to all
rest of the elements in the list if any of the facts in the list that can be derived by
the first element by using the term implication then it will be removed from the list.
After the first element of the list has been compared with all the facts in the list,
then it added to the end of the list. This will guaranty that every two facts have
been compared in both directions if they cannot be derived by each other by using
term implication. The flag that we add to the end of the list cannot be derived by
any other facts in the list. Therefore, when the algorithm takes out the first element
from the list which is the flag, it means all facts in the list have been compared and
the algorithm terminates.

3.4.2
Finding the most general substitutions
In the previous section, we have introduced how to convert a concrete rule to an
abstract rule. In this section, we describe how to find the most general substitutions
in the reduced fixed-point for every rule under which the rule can be applied. This
step is done by the algorithm 24.
The algorithm 24 takes a concrete rule, an abstract rule, the reduced fixed-point,
and a map of variables type as input. It tries to find the most general substitutions
in the reduced fixed-point under the rule can be applied, then returns a set of new
facts that can be generated by applying the substitutions to the rule.
The algorithm starts with finding substitutions by unifying each left-hand side fact
of the abstract rule with every fact in reduced fixed-point. If the there are any facts
that can be unified with the LF facts, then we get a set of substitutions, otherwise,
it returns ∅to indicate that there is no substitution can be found in the reduced
fixed-point which can be applied to the rule. The LF facts needs to be substituted
before unify with each fact in the reduced fixed-point. For the first fact of LF, it
just substituted with an empty σ, the rest of LF facts, they need to be substituted
with the σ which is the substitutions found from the previous facts. After all facts
in the LF ′ have been unified, we either get a set of substitutions or a empty set. If
we get a empty set, it means there does not exist any substitution that is applicable
for the rule. In this case, the algorithm just return a empty set indicate no new fact
can be generate by the rule. Otherwise, it iterates through each substitution in the
set and apply the substitution to the RF ′ and add the substituted RF ′ to the set F
and return it.
The middle part of the algorithm (from point 3 to 5) is only used for display
the substitutions we have found in the fixed-point. In order to better display the
substitutions which we found in the reduced fixed-point to the user. We substituted
the RF ′ with the substitutions we just found. Then we unify the facts in RF with
substituted facts RF ′ to get the substitutions from the concrete facts to the abstract
facts.
e.g., we have a concrete fact iknows(PK) and a substituted abstract fact
iknows(val(ring(A), 0, 0), if we unify these two facts then we will get a substitution
{[PK 7→val(ring(A), 0, 0)]}.
This gives a better view for the user of the result
generated by our analyzer.
So far, we are able to find the most general substitution for every abstract rule
under the rule can be applied. The new facts generate by each abstract rule are in
abstract form. In order to make it easier understand to the user, we convert each
new generated facts to the corresponding concrete fact. The algorithm 25 and 26 are
designed to convert new generate facts back to the corresponding concrete facts.

4.1
Key Separation
Key-separation[8] protocol model a system such that a key can have several attributes.
In the protocol, it defines 4 key sets and 5 rules in AIF. The protocol specification is
in appendix A.2
Four key sets are: sensitive(TOKEN), extract(TOKEN), wrap(TOKEN),
decrypt(TOKEN), each set reflect the one kind attribute in the system. It uses two
rules to define initial knowledge of intruder as follow:
iik1(K1 : value, Token1 : TOKEN)
= [K1] ⇒K1 ∈sensitive(Token1).K1 ∈extract(Token1).iknows(h(K1))
iik2(K2 : value, Token1 : TOKEN)
= [K2] ⇒K2inwrap(Token1).K2indecrypt(Token1).iknows(h(K2))
(4.1)
Rule 4.2 express that is intruder gained two keys’ handles which one key has
extract attribute and the another one has wrap attribute then he can wrap the keys.
wrap(K1 : value, K2 : value, Token1 : TOKEN)
iknows(h(K1)).K1 ∈extract(Token1).K2 ∈wrap(Token1).iknows(h(K2))
⇒iknows(senc(K1, K2)).K2 ∈wrap(Token1).K1 ∈extract(Token1)
(4.2)
Rule 4.3 express that is the intruders a handle of a key which has decrypt attribute
and the cipher which encrypted by the key, then he can decrypt the message to get
the plaintext of the message.

CHAPTER 5
Conclusion
In this thesis, we have explored a new way to present attacks and security proofs
for the stateful protocols.
We implement a fixed-point analyzer which is able to
reconstruct an attack in the abstract level if an attack fact is found in the fixed-point
and trying to use the corresponding attack trace found from the abstract level to
investigate whether the attack state can also be reached in the original model. In
the positive case, our fixed-point analyzer is able to construct the term implication
graph, to remove redundant facts (64% of the facts in the key server example) from
the fixed-point, and to classify by form of the facts in the reduced fixed-point. Our
analyzer is also computes for every rule the most general substitutions under which
the rule can be applied. In addition, we have also implemented a symbolic fixed-point
computation algorithm in Java which is different to the one described in [7].
During the writing thesis, we found that most of the algorithms we have imple-
mented can be conceptually simplified. Therefore, some of the algorithms we wrote
in the thesis are an improved version compared to our implementation, thus making
the algorithms more concise and easy to understand.
In the measurement results, we demonstrate that the analysis results output from
our tool is understandable for the humans in both negative case and positive case.
In the negative case, we display the abstract attack trace in LaTeX proof tree and
displayed how the attack state is reached in the original model. While in the positive
case, our tool found the most general substitutions for every rule under which the rule
it can be applied and the analysis result convinced the user that the attack cannot
be reached. It couldn’t only convince the humans of the proofs, maybe also help in
machine checker proofs, i.e., the machines understand the proofs better. That is the
step for getting things integrated into the tool like Isabelle. In the future work, we
could actually fit this into a proof checker, so that not only the human can check the
analysis result but also for the proof checker could check it.
