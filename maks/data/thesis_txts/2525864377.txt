Abstract
This thesis presents a new programming language - Stateful AnB - designed to model
the security aspects of stateful protocols - and a translator to IF to facilitate testing with
OFMC in a ﬁnite number of sessions. No such language exists at the moment, and it is
analysed how this completes the existing languages and tools for stateless protocols - AnB,
OFMC and IF.
There’s an introduction to the symmetrics for the language and explanations on how the
translation is performed.
A list of experiments is showing the use of the language and the translation to the IF
language is tested.

Introduction
Security protocols has been in use for over 40 years, and a number of tools has been designed
to help solve problems like ﬁnding security ﬂaws in the protocol or prove the security of the
protocol.
A type of protocols called stateful protocols will be the main focus of this theses.
These
protocols are deﬁned by the participants keeping their state for more than one session, by
example a server that functions as a keyserver. This server will need to have some state where
it knows if keys are valid or revoked. There will also need to be a stateful protocol to update
keys and others actions.
It is challenging to verify the security of these protocols with many of the existing veriﬁcation
methods. Some tools has been developed to perform veriﬁcation for inﬁnite sessions, like AIF-
omega, StatVerif and nuFMC. For ﬁnite sessions the tool OFMC[3] can be used. With these
tools only low-level input languages are available. OFMC has a Alice-and-Bob style language,
but this will only support stateless protocols. To improve on this, it will be required to create
an easy to use high-level language that can be translated to a state that the already existing
veriﬁcation tools support. Other valuable features would be description of the composition of
the protocols and checks for the requirements of the composition are satisﬁed.
1.1
Prior works
The need for creating tools for protocol veriﬁcation and security ﬂaws arouse from the fact,
that ﬂaws was found in old protocols, that was believed to be safe. Since it can be hard to ﬁnd
the errors in the protocols, the need for creation of computing tools that can ﬁnd these error’s
emerged.
1.2
Tools
OFMC is one of these tools. It is designed to ﬁnd security ﬂaws from a ﬁnite number of sessions.
This can be usefull when creating a protocol to ﬁnd where there are errors in a protocol. It
has a High-level language Alice-and-Bob that can handle stateless protocols. Another essential
function of OFMC is to test if a protocol will perform. This is useful when creating and changing
protocols for diﬀerent purposes.
Other tools like AIF-omega[5] and StatVerif[1], can be used to verify a protocol for an inﬁnite
number of sessions. This means they will check if any attack is possible but they will not show
how the attack happens.
1.3
Problem statement
The existing tools used for verifying and ﬁnding security ﬂaws, in stateful protocols all use
a low-level input language. The goal is to create a high-level language where it is easy to
create stateful protocols. This language should be based on Alice-Bob-style languages, in order
to make it is easy to see what actions the protocol does and how the actions of the stateful
protocol are connected.
Having a high-level language to describe stateful protocols, is not that useful if it is not possible
to ﬁnd and verify security ﬂaws. To create a new high level language to ﬁnd and verify security
ﬂaws would be a major task. It would be more simple to translate the high-level language into

When translating from the high-level language into the low-level input the translation must

1.4
Two main goals
One is to create a high-level language, to write stateful protocols in an easy-to-use manner. The
style of the language should be close to Alice-Bob-style languages, with easily observed protocol
actions. The Alice-Bob-style language should be able to create the most known protocols like
keyserver protocols and others, in order to support new protocols.
The other is to create a translate tool that can translate the high-level language into the low-
level input language the existing veriﬁcation and security tools are using. First of all the nuFMC
format. In addition it needs to be ensured that the high-level language and the translate tool
works as one entity and creates veriﬁable outputs. To that end a comparison between the
output of the translate tool and known stateful protocols will be done.
If it is possible to create an output that can be veriﬁed correctly for the most common stateful
protocols the goal is considered to be met.

Impact innovation and application
Since no High-level languages for ﬁnding and verifying security ﬂaws in statefull protocols exists,
such a languages can be a stepping stone in a eﬀort to diminish complexity in creating stateful
protocols.

2.1
Protocols
In order to communicate between diﬀerent components in the computer (cpu, cache, disk, RAM
...) or to communicate between computers protocols are deﬁned as a set of standardised, pre-
determined rules and regulations for this communication.
These protocols deﬁnes how the connection is established between the computers, how to ad-
dress it, and how it should transfer the data from sender to recipient. They divide into stateful
and stateless protocols. For a stateless protocol, the information about a transaction is not
maintained after it is processed. In a stateful protocol, the state information is kept after the
transaction is processed.
TCP/IP (Transmission Control Protocol/Internet protocol) are the fundamental networking
protocols for the Internet. TCP is stateful and IP is stateless and they are often supplemented
with additional layers in a stack of both stateful and stateless protocols.
In a stateless protocol, no information about a transaction is maintained after a transaction
is processed. In a stateful protocol the opposite situation occurs and the state information is
Page 2
==PAGE:6==

kept even after a transaction has been processed.
In a stateless protocol, there is no record of the state saved at the server end. The client
sends requests to the server and server responds according to the current state. The depen-
dency between client and server is loose and the server does not keep state between connections.
When a request is send to a stateless server, it does not create any objects that track infor-
mation regarding the requests. Stateless protocols are light and fast, and examples are UDP,
HTTP etc.
Servers using stateful protocols maintain information about the status of the connection, the
processes running and the status of the processes running. If the client sends a request to the
server, it expects some kind of response. If it does not get any response the request will be
resend. There’s a tight dependency between client and server, the state of connections are kept,
and the stateful protocols are heavy and handle transactions slower. Examples are FTP and
Telnet.
2.2
Cryptographic Protocols
A cryptographic protocol is an abstract or concrete protocol, stateless or stateful, that performs
a security-related function and applies cryptographic methods, often as sequences of crypto-
graphic primitives, and the use of algorithms. They are frequently used for secure application-
level data transport and usually incorporates some of these aspects:
1. Key agreement or establishment
2. Entity authentication
3. Symmetric encryption and message authentication material construction
4. Secured application-level data transport
5. Non-repudiation methods
6. Secret sharing methods
7. Secure multi-party computation
By example, Transport Layer Security (TLS) is a cryptographic protocol that is used to secure
web (HTTPS) connections. It has an entity authentication mechanism, a key setup phase,
where a symmetric encryption key is formed by employing public-key cryptography, and an
application-level data transport function. [6]

These components has mainly been developed within the AVISPA project [2].

are the Actors in the protocol, and there are 3 Actors in the example: A,B and s. When an

are dishonest and s is honest). When an actor is dishonest it means that an intruder can act as

Numbers in uppercase are considered variables and lowercase are constants. Each time the
protocol runs, every variable is assigned a value at the start. This means that variables might
vary in each run of the protocol, while constant are the same.
Functions are in most cases constant. A function can use one or more variables and constants
to create a new value. There are no limitations to the number of inputs assigned (variables
and/or constants) to the same function. An example is the private key function "pk(Agent A)"
which generates a private key for the Agent A.

The value of the function pk applied on A. (This is a representation of Agent A’s public key).

inv(pk(A)) is a representation of Agent A’s private key. inv is a special function that represent
the inverse of what it is applied to.

3
Stateful AnB - a new language
To expand on the OFMC frame work this theses will create the new Stateful AnB language
based on the existing AnB language. It will follow the same style, but include ways to write
stateful protocols.
In addition, a translator has been created to translate protocols written in the Stateful AnB
language into the IF language where it can be veriﬁed with OFMC.
Another goal is to support sub protocols. These will be smaller protocols that can be called
from a main protocol.
It is essential that the new language has both the clarity of the AnB language and the ability
to model stateful protocols.

t ::= V

– NotEqualLogic Is deﬁned as "!="

Subprotocols and Assignment
From cryptographic protocols it is common to call a Trusted Platform Module (tpm) or a
trusted third party (ttp) to perform a given service. These calls might change the state of the
tpm/ttp by extending it with a given value.
Such calls are by example performed from the Mark Ryan Envelope protocol, but the tp-
m/ttp are not concerned about who is asking, and don’t need to store that information after
the service is completed. This makes the emulation of subprotocols more complex.
Before a protocol can be translated to IF and checked by OFMC. All subprotocols needs to be
inserted into the main action list and the variables needs to be assigned.

5
Translation
After the insertion of subprotocols and renaming done by assignment actions, the protocol is
ready to be translated into IF. These lists handles the translation:
• Type list : List functions numbers and Agents
• Knowledge list : List of Agents and what initialize state they have

5.1
Initialization
Before the action list is transformed into steps, the Knowledge of the Agents and intruders
needs to be translated into an initial state. Agents, test and functions needs to be declared and
existing states needs to be added.
• Agents states. These are created by adding session id and step number to each Agent.
The knowledge of the agent is checked for dishonest Agents. The Agents are given an
id number corresponding with the agent number and section number. Example: Agent
A is the ﬁrst Agent, has session id 1 and knows A,B,tmp. Then the state of A will be
(A11,1,0,A11,B11,tpm). Agents which are given a number are called session Agents.
• Agents, test and functions. These are translated directly from the type list. The sessions
Agents are added to agent list.
• Intruder gains the Knowledge of all dishonest Agents with respect to him self, his own
privat key and 0.
• States. These are taken from the state list.

Conclusion
The Stateful AnB language is shown to be an improvement over IF in simplicity when mod-
eling stateful protocols. With respect to functionality, subprotocals has been introduced and
facilitates reuse of code (not supported in Anb).
Modelling attacks are still not supported, and have to be written in IF. Some work on bug
ﬁxing in the translator is outstanding, which causes some protocols to translate incorrectly.
It is possible to model many stateful protocols in Stateful AnB, and with the translator to
IF it is also possible to verify them using OFMC.
6.1
Future Work
6.1.1
Stateful AnB
Modelling of attacks in the same style as AnB, to completely avoid the IF language.
A checker to ensure that protocols in Stateful AnB are executable. This will help to ensure
that OFMC is able to verify the whole protocol. If large protocols are created, false positive
can occur if a part is not checked by OFMC. When writing protocols it is possible to miss an
element that might be needed for a later part.
A library of the most common subprotocols that are used in protocols. This can be added
either by default or from a ﬁle called in the protocol.
6.1.2
Translator
When support for modelling of attacks is introduced in Stateful AnB,the translator needs to
be expanded to support them.
Handling of more than one session, similar to AnB.
More documentation for the translater code.
