ABSTRACT
We study weakest preexpectation calculi for probabilistic program veri-
fication. We contribute with the symbolic verification tool wp4fun that
verifies bounds for probabilistic programs derived by the weakest pre-
expectation and checks for validity using SMT solvers.
We compare the symbolic verification approach to explicit state model
checking to investigate their strengths and weaknesses. Our tool im-
plementation is applicable for loop-free programs, but for programs
with loops it requires user-inputted invariants that can be complex to
find.

Introduction
Software systems are playing a bigger and bigger role in modern society. Today, ad-
vanced software systems can be found almost everywhere. With this propagation of
software systems, they have also taken on more and more crucial tasks, shifting the re-
sponsibility of those tasks to the software systems and the software engineers designing
them.
The more crucial tasks software systems take on, the more important it is that we can
trust the correctness of such systems. But how can we build that trust? Well, one option
is testing. But as Dijkstra said in his Turing Award Lecture [Dij2]:
"Testing can be a very effective way to show the presence of bugs, but is
hopelessly inadequate for showing their absence. The only effective way to
raise the confidence level of a program significantly is to give a convincing
proof of its correctness."
There are several approaches for verifying a the correctness of a program. One well-
known approach, which we investigated during our bachelor‚Äôs project, is explicit-state
model checking, where it is checked whether a finite-state model of a system meets a
given specification. However, even though many model checkers are highly optimized,
they still face an inevitable problem: the state explosion problem, which is one of the
biggest limitations of explicit-state model checking.
During our bachelor‚Äôs thesis we realized that the model checking tool we had developed,
would quickly become very slow for relatively small programs. Therefore, we decided to
look into other ways of verifying programs.
An approach for verifying the correctness of a program is symbolic verification. To avoid
the state space explosion problem, it traverses the program more efficiently considering
a large number of states at a single step.
Probabilistic programs
Many advanced systems that play crucial roles in today‚Äôs society rely on machine learn-
ing and artificial intelligence. One of the core concepts that they are built on is probabil-
ity theory. The systems use probabilities to express uncertainties and make intelligent
decisions.
Reasoning about probabilistic programs is quantitative: It does not make sense to check,
if a probabilistic algorithm is correct or not, but rather how correct the algorithm is. We
need to reason about the probability that the algorithm yields correct results. Deter-
ministic programs start in an initial state and end in a unique final state, if terminating,
whereas probabilistic programs yield a probability distribution over final states.
In order to formally reason about probabilistic programs, we can use the weakest preex-
pectation calculus, developed by McIver & Morgan based on Dijkstra‚Äôs weakest precon-
dition calculus, and further developed by Kaminski. Using the weakest preexpectation
calculus, one can from a program, C, and a postexpectation F, obtain an expected value
of F after termination of C.

Approaches of Probabilistic Model Checking
There are different approaches that can be used to implement a probabilistic model
checking procedure. One of the approaches is explicit-state model checking. In this ap-
proach the model checker generates a finite-state transition system based on the prob-
abilistic model that is to be verified [CHVB].
The transition system is then explored state by state to check for properties such as
reachability, expressed in temporal logic.
The main problem with exploring transition system is that these system can become ex-
tremely large as a result of a combinatorial explosion of the number of states, to then tra-
verse all states explicitly is infeasible. In practice, it is convenient to generate the reach-
able states of this transition system and reduce the problem to a pure state reachability
problem [Wol]. Yet, computing reachable states in state space explosion still remains a
challenge. This challenge is also referred to as the state space explosion problem.
Another approach is symbolic verification. In this project, we shall explore a symbolic
verification approach, see Section 3-5, and later compare it to the explicit-state model
checking approach.

Methodology and Research Questions
We have implemented a tool wp4fun (see Section 7) that performs weakest preexpec-
tation symbolic verification using an SMT-solver. With the presented background and
wp4fun, we aim to answer the following research questions:
6.1
Research Questions
Question 1
How do we handle the least fixed point in the weakest preexpectation of programs with
loops?
The weakest preexpectation for while loops is defined using the least fixed point, but
unfortunately the least fixed point is non-computable. To cope with this problem we
use invariants.
Question 2
How do we ensure valid probabilities in pGCL programs?
In pGCL the probabilistic choice has a probability p which is defined as an arithmetic
expression that represents a probability in the range [0,1]‚à©Q. An invalid probability, i.e.
a probability outside of the range, will ruin the analysis and give an inaccurate answer.
Before we do the weakest preexpectation transformation, we would like to make a pro-
gram analysis determining the validity of the probabilities and give the user a warning
that the input program contains probabilities that can evaluate to an invalid value.
We then need to setup up an algorithm for the probability program analysis.
Question 3
How do we ensure non-negativity in expectations?
In Section 4.1 we see that expectations must be non-negative. Both expectations and
the pGCL grammar have arithmetic expressions which include the minus operator. The
weakest preexpectation transformation can then result in a negative postexpectation
which is a violation against the definition.
Before we do the weakest preexpectation transformation, we would like to make a pro-
gram analysis that reveals if there is a possibility of a negative expectation. A negative
expectation should throw an exception forcing the user to fix the input program before
solving the problem.
We then need to setup an algorithm for the non-negativity program analysis.
Question 4
How can we improve the weakest preexpectation of programs with loops?
When verifying programs with loops, we experienced that the analysis was restrictive
and sometimes rejected programs that we believed were valid, and then provided an
19
==PAGE:25==
unreachable counter example.
In this research question we will optimize the weakest preexpectation for while loops,
making it less restrictive in order to reject fewer valid programs, but still without accept-
ing invalid invariants.
Question 5
Is there a significant difference in performance of the SMT solvers?
wp4fun uses two different SMT solvers, Z3 and CVC5, since we did not know whether
one of them was a better fit for our tool than the other.
For a possible optimization of wp4fun it would be interesting to know if there is one that
always outperforms the other, or a combination of the two in different cases is be the
best solution.
We will have to generate scalable benchmarks of different types of programs and break
down our algorithm into parts, so we can time them separately and compare the solvers.
Question 6
Is there a significant difference in performance when comparing the two approaches for
the weakest preexpectation of programs with loops?
After improving the weakest preexpectation for while loops it is interesting to see the
effect it has had.
We will have to generate scalable benchmarks containing while loops and measure the
runtime of wp4fun using the different approaches for checking the invariant.
Question 7
How does wp4fun compare with similar tools?

The Tool - wp4fun
We have implemented the tool Weakest Preexpectation 4 Fun alias wp4fun [wp4fun],
which is named after its distant cousin Model Checking 4 fun alias mc4fun [DRT]; an
explicit-state model checking tool developed for our bachelor‚Äôs project.
The wp4fun does symbolic verification using weakest preexpectation reasoning, pre-
sented in this report. It takes a pGCL program as input together with a postexpectation,
for which the program shall hold after termination, and a constraint, used for the con-
straint check after computing the weakest preexpectation of the pGCL program with
respect to the postexpectation.
First, it parses the inputs into abstract syntax trees (ASTs), to give the inputs a formal
structure. After parsing the inputs, the pGCL program will be checked for invalid proba-
bilities, such that if a probabilistic choice appears in the pGCL program we only want to
accept valid probabilities. How we handle this check will be described in Section 9.
If the pGCL program does not pass the probability check, wp4fun will give a warning,
making the user aware that the program may contain an invalid probability.
Moreover, the pGCL program with respect to the postexpectation will also be checked
for negative values, such that we only accept non-negative variables in the program and
expectations, this will be furtherly explained in Section 10.
If the pGCL program does not pass the non-negativity check, wp4fun will throw an ex-
ception and it will not continue with the verification.
If the pGCL program passes the probability and non-negativity check, wp4fun will then
perform the weakest preexpectation transformer on the pGCL program with respect to
the postexpectation.
For programs containing loops we will use invariants, explained Section 8, to help han-
dle the least fixed point problem when transforming loops to the weakest preexpecta-
tion. In Section 11 we will present an improved version of the weakest preexpectation
transformer for loops that will use invariant inequalities. The invariants are checked,
and if an invariant does not hold, wp4fun will throw an exception and it will not con-
tinue with the verification.
We have now obtained the weakest preexpectation of the pGCL program with respect
to the postexpectation, meaning we have obtained the expected value of the postexpec-
tation after termination of the program. The weakest preexpectation will be validated
against the constraint, this validation is the constraint check.
Lastly, wp4fun will output whether the constraint check was satisfied or not.
The probability, non-negativity, invariant and constraint checks are checked for validity
using the SMT-solvers Z3 and cvc5. wp4fun converts a formula into to cvc5 and Z3 for-
mulas using their respective APIs.
In wp4fun it is possible to choose whether variables and constants should be converted
into integers or reals in the cvc5 and Z3 APIs. The feature arose because pGCL only
supports integers, but expectations are defined as reals, which entailed some transfor-
mation issues.

Introducing Invariants
Research question: How do we handle the least fixed point in the weakest preexpectation
of programs with loops?
In Table 2, the weakest preexpectation transformer for while loops is defined the follow-

WP ¬Çwhile(œï){C ‚Ä≤}¬É(g) = lfp h‚Ä¢[¬¨œï]¬∑ g +[œï]¬∑WP¬ÇC ‚Ä≤¬É(h)
But what does the least fixed point (lfp) of this function actually mean?
Whenever we meet a while loop, two things can happen: Either the guard (œï) is satisfied
and we enter the loop, or it is unsatisfied (¬¨œï) and we continue without executing the
body of the loop.
When reasoning about the weakest preexpectation for while loops, we need to consider
both cases. If we do not enter the loop, the body of the loop will not be executed, and
the weakest preexpectation with respect to g must therefore be g.
On the other hand, if the guard œï is satisfied, the body of the loop is executed until the
guard is no longer satisfied. Since the body of the loop can be executed multiple times,
the weakest preexpectation transformer must apply for every possible iteration of the
loop.
Since we are looking for the weakest preexpectation of a while loop with respect to g, it
must be the smallest possible expectation, the least fixed point (h), that holds for every
possible iteration of the loop.
However, this leaves us with a problem: the least fixed point is non-computable [Kam].
In order to tackle this problem, we use loop invariants, hereinafter referenced to as in-
variants.

Probability Analysis
Research question: How do we ensure valid probabilities in pGCL programs?
For the probabilistic choice in pGCL, the probability p is an arithmetic expression and
has to be in the interval [0,1]‚à©Q. For arithmetic expressions containing only constants,
we can easily evaluate whether the probability in the probabilistic choice is a valid prob-
ability or not by simply checking if the value is within the interval [0,1]‚à©Q.
Arithmetic expressions can also be expressed with variables, which raises some prob-
lems in ensuring valid probabilities. We cannot always evaluate an arithmetic expres-
sion containing a variable to an exact value, since the value of a variable is not necessar-
ily defined, e.g. when it has not been assigned to a value before reaching a probabilistic
choice in a pGCL program.
For example, consider the following probabilistic program:
{ x:=1 } [1/(y-1)] { x:=2 }
Since the variable y has not been assigned in the program, the variable does not have a
definite value. As the user, we can simply figure out that y should be ‚â•2 for the whole
probability to be a valid one, but we cannot be sure that y is in fact ‚â•2, and whether
1/(y ‚àí1) is a valid probability.
To ensure that a given pGCL program contains valid probabilities we perform a probabil-
ity check (PbC). If the program may contain an invalid probability we will give a warning
to the user, to make them aware about the probabilities in their program.

Non-negativity Analysis
Research question: How do we ensure non-negativity in expectations?
When using expectations we must avoid non-negative values. In Section 4.1 we see that
expectations can be arithmetic expressions and must be non-negative. According to
McIver & Morgan, mixed-sign expectation, meaning expectations that can be negative,
is not assured for well-definedness and that it is a good idea to avoid mixed signs in
general [KK].
With arithmetic expressions we have variables and the subtraction operator, which can
introduce negative values.
For example, consider the program p in Listing 13
pre: <= 10,
post: z,
x:=8;
y:=10;
z:=x-y
Listing 13: The example program of negative variables
There is no problem in the pGCL program itself, since z is assigned to a negative value
and negative values are allowed in pGCL.
The non-negativity problem is raised is when the WP transformer is performed on the
program, which is a mapping from a pGCL program to an expectation.
For this particular example, our post condition regards the variable z, so after perform-
ing the WP transformer on our program example p, we get ‚àí2 as the weakest preexpec-
tation of p with respect to z, which is an illegal expectation.
Due to variables being the cause of the non-negativity problem for expectations, we
have made the decision that all variables must always have a non-negative value at any
program point.
In order to ensure this, we do a static program analysis right after a program is parsed.
The analysis aims to answer the following questions:
‚Ä¢ Is there any point in the program at which a variable might have a negative value?
‚Ä¢ Is there any point in the program at which the weakest preexpectation with re-
gards to the postexpectation computed up to that point could be negative?
10.1

reason about its value, when the loop terminates. Therefore, the invariant must hold for
all potential values of those variables modified in the body.

12.1
Satisfiability Modulo Theories
The Boolean satisfiability problem also called propositional satisfiability (SAT) is the
problem of determining whether a boolean formula is satisfiable.
Satisfiability Modulo Theories (SMT) is the problem of determining whether a mathe-
matical formula of first-order logic is satisfiable. A solution or model is an interpretation
of variable, function and predicate symbols that makes the formula true [BM].
E.g. given the formula: x = 2¬∑ x, a model would be x = 0.
SAT is NP-complete for propositional logic and undecidable for first-order logic. The
high computational complexity makes it is infeasible to solve arbitrary SMT problems.
Thus SMT solvers prioritize solving problems that occur in practice, assuming that most
formulas of verification tools are shallow, and only a fraction of a formula is really critical
for obtaining satisfiability [BM].
In SMT the interpretation of some symbols are constrained by a background theory. For
example, the theory of arithmetic restricts the interpretation of arithmetic and boolean
operators, e.g. +, -, =, ‚â•, 0 and 1.
The theories integrated in the SMT solver depends on the solver, and how it selects what
theories to use solving a given problem also varies. Z3 and cvc5 both extend the SMT-
LIB [SMT], which is a library of SMT theories, most relevant for our project to mention
are: ints, reals, and reals_ints, and besides the SMT-LIB theories worth mentioning are

12.3
Discussion
Based on the findings in this research questions we would choose a combination of cvc5
and Z3 as SMT solvers for a possible optimization.
To summarize the results of our tests, we saw that cvc5 has a better runtime than Z3 with
a few exceptions in the non-negativity check in Figure 2, 19 and 25. Unfortunately we
have also seen that in most cases the non-negativity check constitutes a large part of the
total tool time.
Thus we would choose cvc5 for the main tool, i.e. the probability, invariant and con-
straint check, and for the non-negativity check we would use Z3.
We did not see any discrepancies in the results between Z3 and cvc5.
12.3.1
The cvc5 Rewriter
Z3 outperformed cvc5 on the non-negativity check on the sequential assignment bench-
mark. We tried running the non-negativity check on cvc5 and Z3 ten times in a row on
different assignment programs, see Appendix 19.1 for examples. We saw that cvc5 has a
slow first run, but then it drops in runtime on the second run and then stabilizes at the
second runtime. Z3 has approximately the same runtime on all ten runs. We saw this
tendency on formulas with many arithmetic expressions.
The cvc5 solver has a Rewriter module, which converts terms into semantically equiva-
lent normal forms using rewrite rules. The rewriter caches the normalized term to avoid
processing terms more than once. Rewriting is done during solving [Bar].

We believe that cvc5 is slower than Z3 on the non-negativity check because the rewrite

12.3.2
The universal quantifier
As mentioned, the non-negativity check in general constitutes a large part of the total
tool time. This can especially be seen for while loops, in Figure 23 and 29, where the time
of the non-negativity check seems to grow exponentially in correlation to the number of
while loops. If we look at the non-negativity analysis function in Table 6, we can see, that
the non-negativity check for while loops has a universal quantifier. This may very well
be the reason for its exponentially-looking growth, as SMT solvers are known to have a
hard time solving problems with universal quantifiers.
12.3.3
Non-negativity and probability checks
We added the non-negativity and probability check to discard programs where wp4fun
might return an incorrect result.
We discovered that the two checks take a large fraction of the total runtime. Therefore, it
is possible to disable the checks if the runtime is too slow or you are confident that your
program is correct.
It is also possible use the non-negative minus operator Àô‚àíinstead of the regular minus
operator to avoid negativity in programs.
We have not investigated the runtime of the probability check further, because we have
not had any benchmarks with probabilities that are a function instead of a integer divi-
sion.
In practice, we believe probabilities expressed as advanced arithmetic expressions are
rare. Therefore we did not make it a priority to investigate the runtime of the probability
check any further. Also it is not trivial to make scalable tests for the probability check.
If we had had more time, it would be interesting make benchmarks with different arith-
metic expressions as the probability.
The probability check uses the same universal quantifier for while loops as the non-
negativity check, so we would expect the runtime for the probability check to be the
same as the non-negativity check did for while loops.

12.3.4
Python weaknesses
While testing we split up the runtimes, to see which parts of the program that were the
most time consuming. Though it was not a focus, we discovered that wp4fun had very
slow parse times, especially for the three nested programs: non-deterministic choice,
probabilistic choices and if-else statements, where the parse time was between 25-50%
of the total run time.
Python is not a functional programming language, and tail recursion in Python is a
somewhat inefficient technique. Unfortunately parsing and most of our analyses and
functions are tail recursive.
Also Python has a recursion limit, which by default is set to 1000, which also limited us
in generating bigger benchmarks for while loops.
12.4
Threats to validity of results
If we disregard the non-negativity check, cvc5 outperforms Z3 in all tests. But in some
of the non-negativity checks Z3 performs better than cvc5. This raises the question: Is
cvc5 in general better than Z3, or do our tests favor cvc5?
Knowing about the cvc5 rewriter, we would predict that perhaps for programs with more
arithmetic expressions, Z3 would perform better than cvc5. Since cvc5 invest some time
in reducing the arithmetic expressions before solving.
Besides the rewriter, we have not been able to detect any differences in the solvers that
would be an explanation for their difference in runtimes.
Looking at our benchmarks, we must acknowledge that we only cover a small fraction
of pGCL programs. This leaves us with a gap of uncertainty towards the cvc5 and Z3
strengths and weaknesses between each other.
The thought behind our benchmarks was that we wanted benchmarks that where scal-
able, so we could see if there were any connections between the different types of pro-
grams and the SMT runtime.
We could have auto-generated some more randomized programs with a mix of various
sequential and nested programs. This would perhaps give us a better idea whether cvc5
in general performs better than Z3 for our tool, or if we coincidentally designed bench-
marks that favored cvc5.
The problem with randomized programs is that it is more difficult to control their scal-
ability and detect patterns in their results. Also, we would not be able to generate ran-
domized programs with loops, without a way to generate valid invariants as well.
For the solver comparison we have used integers, and also five out our six types of
benchmarks are linear, with the exception of the sequential assignment that is non-
linear because of the postexpectation x/n. This means that for all test cases, the solvers
will use the linear-arithmetic with integers theory, except for the constraint check on the
sequential assignments in Figure 3 where it is non-linear-arithmetic with integers.

13.2
Discussion
At first, let us look at the two types of benchmarks:
The sequential while loop benchmark in Listing 20 starts with two assignments. The
assignments will not affect the invariant check for the original WP transformer as the
invariant checks are executed immediately when the transformer reaches a while loop,
and substitution is performed after. But when the improved WP transformer reaches
the assignments at the top, it will perform substitution into the invariant inequalities.
The variable y is modified in the loops, so it is only x that will be substituted. And for
example the bottom while loop guard will become (1<10 && y>10) which is a simpler
check than (x<10 && y>10).
The nested while loop benchmark in Listing 21 also starts with an assignment. Though
the variable y is modified inside the while loops, it will not be substituted. The improved
WP transformer will therefore not become smarter and do the same invariant check as

==PAGE:70==
the original WP transformer.
To summarize, this means that the invariant check is different for the sequential while
loop, but the same for the nested while loops. Knowing this lets look at the results.
In Figure 31 we saw that the improved WP transformer was faster than the original for
Z3. This was expected as we assume it is easier for the solver to check a formula with
one variable than two.
Though, we also saw that the improved WP transformer was slower than the original one
for cvc5, which we did not expect. But as mentioned in Section 12.3.1, the cvc5 rewriter
invest some time in reducing arithmetic and boolean expressions, and we believe that
this must be the reason for why the invariant checks using cvc5 is slower even though
the formulas are less complex.
In Figure 32 we saw that the original WP transformer was slightly faster than the im-
proved one. This was expected, because the invariant checks are the same, but the
improved WP transformer has some additional checks, when performing substitution,
which will make it a little slower.
Conclusion
It seems that the improvement of the WP transformer has had an affect on the runtime.
For programs that do not benefit from the improvement, the transformer has become a
little slower than the original. For programs that do benefit from the improvement, the
transformer has become faster if it uses Z3, but slower using cvc5.
13.3
Threats to validity of results
The reason for why we developed the improved WP transformer was to accept more
valid invariants. Therefore we obviously cannot compare the improved WP transformer
against the original one, on programs that will only be accepted be the improved one.
We only show two types of examples: sequential while loops that are affected by the
improvement, and nested while loops that are not affected by the improvement trans-
former. We can not be sure that the results are representative for all programs under
these categories.

Comparison with Absynth
Research question: How does wp4fun compare with similar tools?
There are numerous verification tools and benchmarks available, but in order to make a
comparison to our tool we have some requirements. At first we need to find a verification
tool that aims to solve the same verification problem, i.e. upper bounds on expected
values of probabilistic programs.
Also, the tool has to have an input in pGCL or at least be translatable to pGCL. This
includes invariants, which turned out to immensely limit our options.
14.1
Absynth
The work by Hoffmann et al. [NCH] presents a static analysis for deriving upper and
lower bounds on the expected resource consumption of probabilistic programs. They
describe their analysis to be seen as both, an extension of automatic amortized resource
analysis to probabilistic programs and an automation of manual reasoning for proba-
bilistic programs that is based on weakest preconditions.
Their tool Absynth finds the bounds on the expected resource consumption of proba-
bilistic programs. This differs from wp4fun in the sense that they find the bound whereas
wp4fun checks a given bound. The bound derived by Absynth for a given program can
be used as an invariant in the translated program for wp4fun. This makes it possible
for wp4fun to verify whether or not the bounds hold. See Appendix 19.2 for expected
bounds found by Absynth.
14.1.1

able to pGCL. We were able to manually convert 6 programs to pGCL, see Listing 23 and

and the converted in the folder Input/linearPGCL in the GitHub repository [wp4fun].
We were not able to use the other input programs as we encountered several issues:
‚Ä¢ Their input language contained commands that are not supported by wp4fun, e.g.
hyp(m,n) and bin(m,n,o) which are different numerical distributions.
‚Ä¢ Recursive calls are not supported by wp4fun. An alteration with a while loop is
possible to create the same program, but the expected bound will not be a proper
invariant for it.
‚Ä¢ Some of the input programs failed the non-negativity check. We could ignore the
negativity or alter the program to pass the check, but then comparison would no
longer be valid.
‚Ä¢ Our tool requires that all while loops have an invariant. But many of the input pro-
grams contained sequential or nested while loops, and in those cases the expected
bound of the program didn‚Äôt fulfill as an invariant for all loops.

Symbolic Verification vs. Model Checking
Research Question: How does the fully symbolic verification approach using SMT solvers
compare to explicit state probabilistic model checkers?
After going into depth with the symbolic verification approach in theory and practice,
we are ready to compare this approach with probabilistic model checking. In this sec-
tion we will be comparing PRISM [PRISM] representing explicit-state probabilistic model
checking and wp4fun representing symbolic verification using an SMT solver.
First, let us look at the pros and cons of the two tools.
15.1
PRISM Pros and Cons
Pros
‚Ä¢ Optimized tool:
PRISM is a state-of-the-art model checker, thus we assume it is highly optimized.
‚Ä¢ Explicit engine:
PRISM has four different engines. For the comparison we use the explicit engine
that performs both model construction and model checking using explicit-state
data-structures. This can be beneficial for models with a large state space, but
only a fraction of which is reachable [PRISM].
Cons
‚Ä¢ State space explosion problem:
Model checkers like PRISM suffers from the well-known state space explosion
problem, which affects both memory space and runtime.
‚Ä¢ Finite explicit-state space:
Before checking a model, PRISM has to construct the model. The model is a finite
explicit-state transition system, i.e. each state contains the value of each variable
and the model cannot be infinite. This is why it is required to initialize and set a
bound on all variables in the model.
This limits the verification since the property could have a different result for other
variable initializations. It can find bugs, but only verify up to a bound.
15.2
wp4fun Pros and Cons
Pros
‚Ä¢ No state space explosion:
wp4fun does not suffer from the state space explosion problem, since it does not
have to construct the transition system of the model.
‚Ä¢ Infinite state space:
Opposed to PRISM wp4fun does not initialize variables and can handle model
69
==PAGE:75==
with an infinite state space. The verification hold for all states, though it is con-
strained by the postexpectation.
Cons
‚Ä¢ Difficult to find good invariants:
When trying to find or generate benchmarks, our biggest issue was finding proper
invariants. A proper invariant requires that the invariant is valid, but also that it is
not too big of an over approximation, in order to get good results.
‚Ä¢ Non-negativity and probability check:
We have added the non-negativity and probability check as initial checks before
the verification is performed. We saw in Section 12 that those checks were also
time consuming.
‚Ä¢ Not optimized:
Compared to PRISM, wp4fun has not been implemented with a focus on optimiz-
ing the processes.
Also wp4fun is implemented in Python, so we could use the Z3 and cvc5 APIs. But
Python is a "slow" language, especially parsing using Python and antlr4 in combi-
nation, which we did, is slow.
Based on these knowings we have the following hypothesis:
PRISM will be faster than wp4fun in smaller state spaces, but when reaching a certain
state space threshold wp4fun will "outperform" PRISM.
15.3
PRISM
PRISM is a probabilistic model checker. Development of PRISM is primarily carried out
at the University of Birmingham and the University of Oxford. The tool is open-source
software, released under the GNU General Public License.
PRISM can be be used to analyze several different types of probabilistic models, e.g.
discrete-time Markov chains and Markov decision processes. Properties to be verified
against these models are expressed in probabilistic extensions of temporal logic; among
Probabilistic Computational Tree Logic (PCTL).

In general, the runtime of wp4fun grows linearly in relation to the program size, whereas
the regression of the runtime of PRISM in relation to the program size depends more on
the type of benchmark.

but on 17 nestings it ran out of memory. I.e. between 216+1 = 131.072 and 217+1 =

nested if-else benchmarks, where the explicit engine discards the unreachable states.
This resulted in what looks like exponential growth in runtime of the probabilistic and
non-deterministic choices.

iterations in the while loop. Opposed to wp4fun, for which the runtime is indifferent
from the number of loop iterations.
Under the assumption that wp4fun, representing the symbolic verification, is faster on
probabilistic and non-deterministic choices than PRISM, representing probabilistic model
checking, one could ask: Why is the symbolic verification approach not used more in-
stead of probabilistic model checking?
At first, PRISM is a probabilistic symbolic model checker, and has four main engines. We
use the explicit engine, but the other three are either wholly or partially symbolic, and
the process of constructing a probabilistic model is performed in a symbolic fashion.
Also, we keep returning to the fact that it is difficult to find proper invariants for pro-
grams with one or multiple loops.
At last probabilistic model checking can be used to check multiple properties for mod-
els, and not just verifying bounds like wp4fun. We only use a small fraction of the tem-
poral logic that PRISM supports, for the comparison.

We use the same benchmarks as for the comparison tests of cvc5 and Z3 in Section 12.

It is important to take into account that we translate from pGCL to PRISM. It is possible

Related Work
Weakest Precondition Calculi.
In our work of reasoning about probabilistic programs,
we mainly focus on the Weakest Preexpectation reasoning specifically as an approach of
symbolic verification and how to use the approach to implement it in practice. The rea-
soning is built on the principals of Dijkstra‚Äôs weakest precondition calculus, and McIver
& Morgan‚Äôs weakest preexpectation calculus. Most closely related to our work, is the
work by Kaminski [Kam]. He goes into more details on weakest precondition calculi
for probabilistic programs by giving a comprehensive introduction followed by devel-
oping an advanced weakest‚Äìpreexpectation-style calculi to reason about expected run-
times, conditional expected values and conditional probabilities, and expected values of
mixed‚Äìsign random variables. In both our work and in the work of Kaminski, we reason
with non-deterministic and probabilistic programs.
Slicing of Probabilistic Programs.
Another technique within the weakest precondition
calculi is using the backward propagation of postconditions via the greatest preexpecta-
tion transformer, which is the counterpart of weakest preexpectation transformer. The
technique is used in the work of Navarro & Olmedo [NO], where they present the first
slicing approach for probabilistic programs based on specifications. They show proba-
bilistic programs accompanied by specifications in the form of pre- and postcondition
can exploit semantic information to produce specification-preserving slices.
Analyzing Probabilistic Programs Automatically.
Considering other work on analyz-
ing probabilistic programs, there is the work of Ngo, Carbonneaux & Hoffmann [NCH].
They present a new static analysis for deriving upper bounds on expected resource con-
sumption of probabilistic programs. The analysis is fully automatic and derives sym-
bolic bounds, it can be seen as an automation of reasoning about probabilistic pro-
grams based on weakest preconditions, whereas we use manual reasoning for proba-
bilistic programs.
Semantics for Mixed-Sign Expectations.
In the work of Kaminski & Katoen [KK] they
also present a weakest‚Äìprecondition‚Äìstyle calculus, but reasoning about the expected
values of mixed-sign variables after execution of a probabilistic program, as opposed to
non-negative variables in traditional weakest preexpectation calculus. In our work we
adhere to non-negative variables, but through the process of implementing the weakest
pre-expectation calculus we run into problems of variables potentially becoming neg-
ative. Instead of reasoning with mixed-sign expectations, we handle negative variables
through a program analysis.
Explicit-state Model Checking. In our bachelor project "Model Checking 4 Fun" [DRT]
we explored the approach of implementing explicit-state model checking. The approach
is based on modeling programs as a finite-state transition system and verify such sys-
tems state by state for properties. The problem with this approach is that it may faces
the state space explosion problem Model Checking 4 Fun is a counterpart of the ap-
proach of symbolic verification used in this work.
Binary Decision Diagrams.
The work of van Dijk et al. [DHJL] also mentions the state
82
==PAGE:88==
space explosion problem. Related to explicit-state model checking, they model pro-
grams as transition systems and reduce systems to reachable states to be verified for
PCTL properties. To tackle the state space explosion problem, they present a more com-
pact representation of large state space called Binary Decision Diagrams (BDDs). In
their work, they demonstrate the performances of different BDD packages in the con-
text of probabilistic model checking.
17
Future Work
There are several ways to further extend the work of this thesis.
17.1
Finding invariants
The biggest obstacle for program verification using wp4fun turned out to be finding
good benchmarks, as most interesting programs include loops. And for all loops, wp4fun
requires a valid invariant. To make wp4fun more usable it is necessary to investigate in-
variants further.
In the recently published paper, Probabilistic Program Verification via Inductive Synthe-
sis of Inductive Invariants[BCJKKM], a team of researchers has implemented a tool for
finding loop invariants. The tool "...finds invariants for (in)finite-state programs, can
beat state- of-the-art probabilistic model checkers, and is competitive with modern tools
dedicated to invariant synthesis and expected runtime reasoning." [BCJKKM](p.1).
The tool derives a template, T, from the given program, and passes the template to the
counterexample-guided inductive synthesis (CEGIS) loop (the shaded box in Figure 42),
which upon termination has either found an invariant or can report that T does not
admit an inductive invariant.
Figure 42: Figure 1 from [BCJKKM]: "Our CEGIS framework for synthesizing quantitative inductive
invariants."
In the paper, results of the prototype of the tool, CEGISPRO2, have been compared to
other state-of-the-art verification tools. Figure 43 shows experiments on verifying up-
per bounds on expected values of finite-state programs for CEGISPRO2 and the model-
83
==PAGE:89==
checking tool STORM. The figure shows, that CEGISPRO2 can verify upper bounds for
several finit-state programs in a relatively short amount of time, whereas STORM fails to
provide the upper bounds within the 2 hours, that each test was run.
The conclusion of the paper is, that the approach can outperform state-of-the-art explicit-
and symbolic-state model checking techniques and can scale to huge state spaces.
Figure 43: Figure 4a from [BCJKKM]: "Verifying upper bounds for finite-state programs (TO=2h)."
In order for wp4fun to be useful for programs with loops, it would require a way to find
proper invariants. Using CEGISPRO2 to inductively synthesize invariants would be a
way to fix this problem. CEGISPRO2 and wp4fun both take pGCL programs as input,
so an integration towards CEGISPRO2 or and combination of the two tools, would be
an interesting next step, and we believe it has potential to become a powerful symbolic
verification tool.
17.2
SMT theories
In Section 12 we compared the SMT solvers cvc5 and Z3, but our tests were not very
diverse as we only looked into linear integer arithmetic. It would be interesting to fur-
ther compare the tools on different theories such as linear and non-linear arithmetic for
integers or reals or mixed integer and reals.
17.3
Functions as probabilities
We never got to do any systematic testing of benchmarks with probabilistic choices
where the probability is a function.
E.g. Kaminski uses this program in his PHD [Kam]:
84
==PAGE:90==
{x:=x+1} [1/(|x|+1)] {x:=x-1}
though this exact program would not pass the non-negativity check, it would be inter-
esting to look into the possibilities of probabilities.
17.4
Non-negativity and probability check
After having implemented and tested the non-negativity check and probability check,
we found out, that the two SMT solvers had a very hard time solving problems with uni-
versal quantifiers. When we did the improvement to our invariant check, we managed to
find a way to avoid quantifying over all variables modified in the loop body by introduc-
ing a new data structure. If we had more time, it would be interesting to come up with
data structures, such that we could avoid quantifying over all the variables modified in
the loop body, when doing the probability and non-negativity checks.
17.5
Java API for cvc5 and Z3
During our testing we discovered that our tool had very long parse times. It seems that
the combination of Python3 and antlr4 for parsing generally is not good.
At the beginning of the project, we started the implementation in F#, but struggled with
the integration towards the SMT-solvers. Therefore, we moved over to Python as both
solvers had Python APIs. If we could go back and change that, we would have liked to
do the implementation in Java, but we found out too late, and did not have time to start
over in a third language.

18
Conclusion
We have investigated the symbolic verification approach using SMT solvers for verifying
probabilistic programs as an alternative to probabilistic model checking. The symbolic
verification approach is based on the weakest preexpectation calculi presented in the
PHD Thesis by Kaminski [Kam].
We have implemented the tool wp4fun alias Weakest Preexpectation 4 Fun, which is an
application of the theory of weakest preexpectation calculi in practice. The tool can ver-
ify bounds of the weakest preexpectation of a given pGCL program and postexpectation.
To enforce correct input programs, we perform two program analyses prior to the weak-
est preexpectation analysis. To avoid invalid probabilities we make a probability analy-
sis, and to obtain non-negative expectations we make a non-negativity analysis.
We have implemented the weakest preexpectation transformer presented by Kaminski.
To avoid computing the least fix point we handle the weakest preexpectation of loops
with invariants. We have implemented the general approach for invariants presented by
Kaminski, but have also improved the approach for handling invariants in the weakest
preexpectation transformer. The improvement is less restrictive and rejects less valid
invariants without accepting invalid invariants. At last, we verify whether a bound for
the weakest preexpectation is satisfied or unsatisfied.
We use the two SMT solvers, Z3 and cvc5, when checking for satisfiability in the proba-
bility analysis, non-negativity analysis, weakest preexpectation with invariants, and the
bounded weakest preexpectation. We compare Z3 and cvc5 on scalable benchmarks of
different types of programs, and see that cvc5 in general has faster runtimes than Z3 on
the checks, with the exception of some of the non-negativity checks.
The motivation for investigating symbolic verification is to find an alternative approach
to the explicit-state probabilistic model checking approach, which suffers from the state
space explosion problem. We have focused on randomization, non-determinism, prob-
abilities, conditions and loops which all can cause state space explosion, and investi-
gated if symbolic verification can avoid this problem.
For loop-free programs, symbolic verification with the weakest preexpectation is easy to
apply in practice and is a good alternative to model checking. For programs with loops,
symbolic verification with the weakest preexpectation using invariants is not easy to ap-
ply in practice. At first, invariants are complex and it is difficult to find proper invariants.
An invariant is an overapproximation and the analysis does not provide any indications
of the precision of the approximation. If the invariant is a big overapproximation, the
weakest preexpectation will also be inaccurate.
We did not find any other symbolic verification tools solely based on the weakest preex-
pectation calculus, which made it difficult to compare the correctness of wp4fun.
We tried to compare wp4fun with the verification tool, Absynth, by Hoffmann et al.
[NCH], by letting wp4fun verify the bounds derived by Absynth.
For future work, in order for wp4fun to be more usable it would require that we became
better at finding proper invariants.
