Abstract
The aim of this report was done to examine the overall problem of how a robot obtains
a situational awareness of its environment. An investigation on how a robot was able
to detect doors in different environments was done. In this case, a D435 Intel camera
was used in corridors and rooms. The detected doors would then translate into reality
meter positions, which the robot could use in a navigation map. The test cases have
been split into detecting doors down a corridor and a door in a room. An analysis of
how the door detections down a corridor and in a room have been performed and a
walkthrough of the different computer vision techniques, such as the Hogh line metod,
Canny egde detection, have been explained. An overview of how the algorithms work
has also been shown in the form of flowcharts. Tests were performed for the algorithms
to see how much the algorithms’ calculations deviated from the actual measurement.
From the test, a conclusion has been drawn on whether the code was deemed acceptable
compared with real-life, which is not the case. It was found that the depth detection of
the camera was not optimal. A discussion is made on what methods, such as changing
the probabilistic Hough line method and getting a better depth detection, could be
made in order to improve the code and be deemed acceptable. Finally, a conclusion on
the overall performance of the algorithm was discussed and the current algorithm was
judged to not being suitable for a robot implementation.

CHAPTER 1
Introduction
Robots are used today for a wide range of manual tasks e.g. assembly lines. A great
number of improvements have been made to develop robots to do other tasks for example
the Google car, which has been able to autonomously drive around without a driver.
These big projects require a lot of computer power, complex algorithms, computer vision
techniques, etc.
This led to the question regarding whether simple computer vision
techniques could perform simple tasks such as for a robot to identify a door in different
environments and use this to create a degree of situational awareness. To minimize the
cost of additional sensors only the Intel D435 camera was used because of its many built-
in features. The aim of this project was to research if simple computer vision techniques
could identify doors in a corridor and in a room.
Furthermore, the given camera’s
measurements would be compared to the real measurements. If this was possible with
small errors, the robot could thereby use this information to react accordingly to the
given information about the environment.
1.1
Objectives
For a robot that is performing a mission, it is important that it can adapt to different
situations or obstacles on the mission path. The robot would, in this project identify
opened and closed doors in its environment. The project will therefore investigate:
1. How does the robot identify doors in its environment?
With the sub objectives being:
1. Identify a solution that detects doors with the needed accuracy for an autonomous
indoor robot.
2. Keep processing power low.
3. Implement an algorithm that might be usable on a robot
4. Document the detection quality

2.1
Introduction
To make the door detection algorithm, I chose to split the detection between a door
detection in a room, which usually has one door in its field of view, and a corridor,
which could have multiple doors. Both of these door detections use most of the same
computer vision techniques with some differences.
2.2
Bilateral Filter
Before trying to detect any lines for a door, preprocessing of the images was performed.
The preprocessing step was to downscale the image in the room scenario in order to
remove some details of the image. This was done because the images from the camera
were too detailed when finding vertical lines and could give random false positive vertical
lines in a room. The algorithms removed some details by downscaling but still kept the
vital vertical line details. The algorithm upscaled the image back to the original size
and applied the bilateral filter. The bilateral filter was used in order to enhance certain
features you want to extract. In this case, it was to remove the noise in an image, avoid
blurring between objects, and keep the edges sharp because the edges was what was
wanted[17]. What the filter did was using formula 2.1 [17] on a pixel and its nearby
pixels.

Hough lines
After edges have been detected in the Canny image, they have been used in the Hough
line transformation in order to find all the Hough lines in the image. This was done in
order for the Hough line to only look at the edges. The purpose of using the Hough
line was to find the top and bot frame of detected doors.

was the radius from origin to the line and θ was the angle between the line and x-axis.

Cartesian coordinates. An example can be seen in figure 2.3.

Hough line method has been to find the vertical door frames of the doors.

2.6
Door detection
After finding all the horizontal Hough lines and vertical probabilistic Hough lines, the
intersection point between them was found. This was done to find the corner points of
the door, using formula 2.9.[1]
(a1 · b2 −a2 · b1) · x = c1 · b2 −c2 · b1
(2.9)
Where each line respectively was represented in formula 2.10 and 2.11. [1]
a1 · x + b1 · y = c1
(2.10)
a2 · x + b2 · y = c2
(2.11)
After the intersection points were found it was checked if the door had a certain width
for the ”one door in a room” case. The corridor case was divided into right and left
planes for the projective transformation.
After finding all door corner locations the
algorithms looped through the height of the door and 15 pixels of the width of the door
to find the average depth distance inside the door. This was compared to the average
depth distance of the corner points. If the depth distance inside the door was higher
than the corner average depth distance +0,7 m it was judged that it was an open door.
Otherwise, it was a closed door.

2.9
Summary
All in all, a walkthrough of all the different theories used to detect a door in a room
or corridor has been seen. The algorithms started using a bilateral filter in order to

enhance edges and remove noise. Afterward, the algorithms used Canny to only get the
edges back in an image. This was done in order to use the Hough line to find the top and

Homography was used in the corridor case in order to remove perspective distortion

using the cameras stereo vision of depth. Finally, a ground plane and elevated plane
was made in order to try to get a better depth accuracy for the door corner points.

3.3 Flowcharts

From figure 3.5 The algorithm started by downscaling the image to make it less
detailed and upscaled it again afterward. Then, a bilateral filter was applied to remove
noise and enhance edges. A Canny edge detection was then applied to only have edges
left. Hough line transformation was used to get all horizontal lines and the algorithm
checked if there existed a top Hough line for the top frame. If this was the case, the
algorithm could see if the door was on the left side, right side, or if it was a horizontal
door. If any of these cases were true a probabilistic Hough line transformation was done
to find all the vertical lines, which represent door frames. The most likely top frame
would be found by looking at the line with the most near-vertical line endings. Top
points of the doors would be found by finding the intersection point between the found
horizontal line and the vertical lines. The distance between the points would then be
found and checked if it was within a big door size, normal door size, or small door size.
If that was not the case no door was found and the algorithm looked at the next image.
If a door was found, the points would be stored and the algorithm would check if there
was a bottom Hough line for the bottom door frame. If that was the case, the new
bot points would be stored. The door would then be drawn from the stored points and
converted to meters. the algorithm afterward went to the next image.

the current image was a corridor. Vertical lines were again found from the probabilistic
Hough line transform and the intersection between the top and the bottom frames and
the vertical lines were found. Afterward, the algorithm split the lines into left and right
plane and found the Homography transform to convert to their planes. The algorithm
checked if the Homography was empty since this can occur and if that happened the
algorithm looked at the next image. If it did not occur it converted the door points to
plane points and checked if the distance in pixels between the vertical lines was within
a threshold. Afterward, the algorithm stored the points and converted them back to
the original image coordinate. It stored the new points for the doors, drew the doors,
converted them to meters and afterward went to the next image. This step could have
been improved, but because of time constraints, this was not done.
Lastly a flowchart on how the ground plane and its elevated plane was found can be
seen in figure 3.7. This part was done in both the room and corridor situations but was
later removed in the live script because no improvement was found.

the one from 3D, but had the same x and y coordinate. After this the algorithm looked
at the next image and did the same procedure.

Discussion
As seen from the results of the corridor algorithm tests, I get a true positive rate of
around 75% for three images for both corridor cases and a true positive rate on 65% for
one image for corridor A case and 57% for one image for corridor B case. The detection
rate for three images were acceptable, where the detection rate for one image was not.
The explanation for this is, when trying to decide where the door detection stopped
for real-life comparison it became a bit subjective because it was hard to see where the
algorithm stopped detecting. A decision was made to stop detecting around the middle
of the doors in a corridor. This might affect the confusion matrix to detect a bit better
than what it is in reality, since I disregarded the last doors which were not detected.
Furthermore when the actual measurements were written down, I only wrote down the
five closest doors in the corridor for each side. This was done, thinking that this was the
algorithm’s limitations on predicting doors down a corridor. The probabilistic Hough
line method, also not being robust, further enforced this assumption. The code right
now only utilizes one image, but it should be able to be expanded to take more images.
These reasons made the door detection a bit subjective and thereby leads to the
conclusion that one image true positive rate being 57% and 65% not being enough.
Furthermore, as seen from the depth and door width tables, the depth acquisition was
not precise enough which lead us to use the Homography method instead of the door
width as a restriction of door detection. It was also seen from the ”going down the
corridor” result that no general trend was apparent when seeing where the deviation in
depth occurs for two different doors. Looking at the results, the fault could lie in how
the point cloud and stereo vision method work to find the depth. It could be thought
that there were not enough point cloud points to get an accurate depth detection. Of
course, this could change if I looked at more doors, but because of time constraints, this
was not done. Of the methods used, the 3D median method was the most robust in both
corridor cases and in the ”one door in a room” tests. Looking from the camera’s own
depth perspective, the frames from the 3D median method actually is on the wall and
look somewhat precisely like door locations. Comparing that with the real-life scenario
of the actual length from the camera to the door frames, in the corridor cases, I get that
it is around 46-53% for a deviation within 0,1 m. Within 0,1 m it is assumed to be an
acceptable error for implementing on a robot, but the low percentage deviation within 0,1
m for the algorithm makes it undesirable for detecting doors in a corridor. Better depth
detection equipment is wanted, which could lead to changing the Homography method
back to a restriction on door width instead, and thereby give better door detections.

5 Discussion
When I looked at the results from the door case, I had a true positive rate of 90%
for three images and 75% for one image for a distance up to five m. The results from
the door algorithm are deemed acceptable and are what was envisioned with the width
of the door restriction, when designing the corridor algorithm. But because of the poor
performance of the depth detection of the skewed doors in the corridor case, it was
changed to Homography. From the door test, I got cases with partly detected doors,
which have been counted as true positives. Most of the test cases for when this happened
were for open doors or doors with glass in it which also makes it harder to detect the
door. Other problems like figure 4.21b where there is a gap between the wall and the
door also make the depth detection bad. If I looked at the doors from the perspective
of the camera like in figure 4.24, I saw the doors was where the wall was detected. This
further proved that in the room case, the camera is decent to detect the depth. It has
to be noted that looking at the doors from the side up to a certain angle for one door
case has also been tested. But to look down a corridor and see the skewed look of the
door, has not been tested for the door case.
The open and closed-door part of the algorithm could be better and the major
problem with the detection here is that the doors have glass panels in them.
The
glass makes the algorithm think it is an open door since it can be seen through. This
along with a partly or decent detection of the door, can make it misjudge it as an open
or closed door. Overall the door detection algorithm is acceptable for both door width
and depth detection. The open and closed-door part of the algorithm should be further
improved or rethought to compensate for the glass in the door. The height of the door
could be used in order to reduce the number of wrong detections and partly detected
doors, but because there is not always a guaranteed lower bot frame detection from
the Hough line method, no restriction was made for the height of the door in order to
have more detections. The reason for this is that the navigation map for the robot is not
interested in height, but only the depth z and width x to the door position. Currently, the
height restriction has not been implemented which could cause false-positive detections
for closets or other objects with the same width as a door. Furthermore, windows are
also a problem for the door detection, since they have the same shape as a door.
From the five methods; top 3d, bot 3d, median 3d, plane top, and plane bot, the best
method has been determined to be the median 3D since it is the most robust in both
the corridor cases and better detection rate within 0,1 m than the top 3D method which
the code is currently relying on. The plane methods were originally thought to improve
the depth detection in the corridor case, but this was disproved by the test. The plane
methods had problems with the light reflection from the ground and the vanishing line
for the bot part in corridor B was a bit higher than the ground. Furthermore, the top
plane was also not always 2,1 m, which is about the height of a normal door. These
detections were proved to be fatal for the method which leads to the bad detection rates
for within 0,1 m and within 0.2 m cases. Because of this, it was decided to exclude the
RANSAC plane method from the live code in order to enhance the run time of the code.
From the plane method, I had a run time of about 800 ms and without the plane method,
I had about 50 ms. The values were approximately the same for both the corridor and

being real time.

Analysis
In this section, I compare the algorithm with other state of the art methods. This is
done in order to compare performance with other techniques and give ideas to what
might have been better techniques than the ones currently used. Improvements and
future work that might boost the performance further will also be discussed.
6.1
State of the art
From the test chapter, I have seen that the current algorithms chosen are using only
classical computer vision techniques, which are not ideal. The algorithm relies heavily
on the Hough line and probabilistic Hough line transform in order to determine the
doors and the probabilistic Hough line transform is not always robust in giving the same
number of lines for the same image in a corridor. Furthermore, I also found that the
depth of corridor doors became a problem if I wanted to be within 0,1 m. I compared
the results with other state of the art techniques to see how their algorithm performed.
If I compare the one door detection in a room with others I see that the article
”Computer Vision-Based Door Detection for Accessibility of Unfamiliar Environments
to Blind Persons” [18] which uses canny edge, a corner detector proposed by He and
Yung and some geometric door properties to identify doors. Here they have an accuracy
of 92% with a false positive rate of 2.9% on 203 images. If I compare that with our best
door detection for a door, which is the three images case on 90%, I see that the detection
rate is pretty close. The problem with this algorithm is that the door needs to be fully
viewed in order to identify the door or the bottom or top of the door is occluded a bit.
This means that if one corner of the door has for example a garbage can in front, the
algorithm will have trouble identifying the door. Furthermore, it also has troubles to
differentiate between a bookcase and a door and does not use the depth to find the real
size of the door or its position, which would be useful in our project for a navigation map,
if it worked.[18]. By using real life door sizes from depth detection might also reduce
the detection rate depending on how good the depth detection is. Corner detection has
also been considered in this project, by applying Harris corner detection. This was later
dismissed since fine-tuning on variables did not give the wanted corner points and gave
other false positive points from its detection. This method did have the upside being
that precise doors are detected, which is not always the case for our algorithm. From
their test images, it can be seen that skewed doors can be considered, but skewed doors
==PAGE:62==
56

as looking down a corridor can not be seen in their test. If their algorithm can detect
skewed doors in a corridor an adaptation of their algorithm could be expanded on in the
corridor algorithm case.
Another state of the art technique for the ”one door in a room” case could be using
machine learning to identify doors. In the article ”Real-time Door Detection based on
AdaBoost learning algorithm” [4] they use the machine learning technique Adaboost to
learn the characteristics of a door as a classification problem. These characteristics are
doorknobs, if there is a door gap between the floor and the door, if there is a door frame,
color difference between the wall and the door etc. If it was judged that a door was in
the image, it used a Contour Following algorithm to find the vertical lines of the door
and used the door width as a threshold. The algorithm here has an accuracy of 72% with
0,8% detection error on 70 data tests where the algorithm was trained on 140 training
data. The algorithm here can only detect closed doors. [4]. The upside of this is that
it has been tested on a moving robot and it uses different perspectives of the doors to
increase its door detection rate up to 90%. Furthermore, it should be able to detect a
variety of different door types. Ours on the other hand is 90% looking over three images
at best and does not move its position and is only looking at specific door types. Since
the original intent was to implement on a robot, this solution might be better than ours
since ours have not been tested on a robot.
By comparing our corridor algorithm with the article ”Simple Monocular door detec-
tion and tracking” [10] I see that our two algorithms are actually similar to each other.
They also use vanishing lines to identify the top frame and bot frame of a door and iden-
tifies vertical lines as door frames. Afterward, they somehow get the depth data to find
the width and height of the doors to identify them. Their algorithm got a detection rate
of 82%. The difference between our algorithm and theirs is that they use another vertical
line detection method to identify door frames, which is the Line segment detection. By
comparing their detection rate with ours, which is around 75% for both corridors, I see
that the detection rates are close to each other. The problem with our algorithm is that
it does not utilize the width of the door because of the poor depth performance. As
discussed earlier this gives rise to the wrong positioning of the door. Their algorithm on
the other hand has the width as a restriction, which makes their algorithm much better
than ours. It is however unknown from their article how they get their depth data and
their depth acquisition might be more precise than ours.[10]
6.2
Future work
A lot of improvements can still be done in order to make the detection of doors in real
coordinates better. The first one as discussed before is to implement other sensors or
a laser scanner to get a better depth accuracy of the door frames. By doing this I can
switch the corridor case to be like the 1 door in a room case and use the door width
as a restriction instead of the current Homography method. This will make the door
width and position more accurate especially if the algorithm is to be implemented on

and this method might be more robust than the probabilistic Hough line method and is

CHAPTER 7
Conclusion
The purpose of the project was to understand how the robot identifies its environment
and try to use a camera in order to convert image data into real-life data which the
robot understands. This was done using classical computer vision techniques and math
which afterward was converted into meters where the objective first of all was to identity
doors in different environments. Here the environments chosen were in a corridor and
in a room case. By having the door position in meters the robot would be able to put
the found doors into a navigation map.
Based on the tests it was found that the currently used probabilistic Hough line
method is not robust and can be the reason for why a door was overlooked when looking
at the same images taken with one second delay between them.
Other methods to
replace the probabilistic Hough line method was considered when comparing with other
articles and could be worth investigating, if an improvement was made. Furthermore in
the one door in a room scenario, if no Hough line for the bot frame was detected, the
currently detected probabilistic hough line was used as the door frame. This means that
the height could not be used as a restriction and this further decreased the accuracy for
always detecting a door.
The depth detection was deemed to be not reliable enough in order for the algorithm
to determine a door from its width in the corridor case. This has been deemed to be
a problem and other methods were used to compensate for the depth deviation. As
seen from the detection of the ”one door down the corridor” case, no general trend was
found for where the error usually occurs. A depth accuracy for the positively detected
doors within 0,1 meters was at worst 44.5% for the median 3D method. This lead to
being thought that the problem lied in how the camera detects the depth using a point
cloud and stereo vision. Other errors like the glass panels in the door, if it was an
open door, and light reflection could also be reasons for the bad performance. With all
these errors the median 3D method used was deemed to be the best performing method.
Furthermore by removing the RANSAC plane method the run time of the code is deemed
to be around 50 ms instead of the 800 ms.
Overall it can be concluded that our solution is not optimal enough to be implemented
on a robot. This is due to the poor performance from the depth detection on the camera
in the corridor case and therefore affects the door frame position calculation. The Hough
line probabilistic method was furthermore deemed not robust enough which in turn lead
to missing door detections. On the other hand, future work methods have been proposed
in order to further accomplish the goal of the robot’s situational awareness and fulfill the
