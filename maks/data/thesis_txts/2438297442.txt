Summary (English)
The goal of this thesis is to look into the Fenwick tree and ﬁnd various ways it
can be implemented, with a focus on compression.
We deduced a formula for the minimum space usage of a Fenwick tree containing
elements up to a certain size. We then implemented a number of trees with space
usage close to this formula.
Even the fastest of these implementations was
however using between 50% and 100% extra time to complete our performance
tests compared to the uncompressed Fenwick trees.
We looked into creating Fenwick trees which used less space than a standard
Fenwick tree but was faster than the compact Fenwick trees, which we called
compromise Fenwick trees. Our best solution in this category would at most use
double the space of the compact Fenwick trees and taking between 0% and 50%
extra time in our performance tests compared with the uncompressed Fenwick
tree.
We also experimented with a structure inspired by the Fenwick tree called the
b-ary tree. We made compact and compromise versions of this like with the
Fenwick tree. We got similar performance from them, but for certain parameters
the compromise b-ary tree would use up to 1/3 less space than the compromise
Fenwick tree.
We also looked into concurrent execution of multiple operations. This turned
out to improve the performance of the Fenwick and b-ary trees when the sizes
were in the magnitude of millions of elements. Smaller than this and it resulted
in degraded performance.

Introduction
Given an array A of k-bits integers, with |A| = n. The partial sums problem
is then to create a data structure that can support the following operations:
• sum(i): return the value of Pi
l=1 A[l].
• update(i, ∆): set A[i] ←−A[i] + ∆
• access(i): return A[i].
• search(v): return the smallest index i such that sum(i) ≥v
The Fenwick tree[1] is a solution to the partial sum problem, supporting all the
above operations in O(log n) time while using only Θ(n) words of space. It has
however been shown that the elements from the Fenwick tree can be stored in
a more compact fashion, such that the Fenwick tree is stored in nk + o(n) bits
[2].
The initial application of the Fenwick tree as described in Fenwick[1] was for
cumulative frequency tables in arithmetic coding.
It has however been used
in numerous other applications. Among others, it has been used to improve
the running time of a fast Dijkstra-based all-pair shortest path algorithm used
for airline networks [3], and as a subroutine in reading and writing of genome
sequences[4].
The goal with this thesis is to examine and implement a number of diﬀerent
approaches to storing the Fenwick tree, and compare the space and time usage
of each solution.
First, we will look into uncompressed Fenwick trees which function as described
by Fenwick [1]. Then, we will look into some variants of Fenwick trees that use
==PAGE:9==
CONTENTS

nk + o(n) bits as described by Bille et al.[2]. We will then look into Fenwick
trees using space storage somewhere between n words and nk+o(n) bits. Next,
we will look into the modiﬁed Fenwick tree inspired data structure called b-ary
trees described by Bille et al.[2]. Finally we will explore the performance of
some of the solutions when taking advantage of multiple threads for concurrent
execution.
Implementation details and testing methodology
For this project we decided to use C++ as the development language. This was
for performance reasons, along with developmental, as it gives us a "close to
metal" language, while still having access to classes and objects.
We consider a word to be 64 bit, and use 64-bit integers to implement structures
using no compression.
We are using template classes as a form of inheritance rather than normal in-
heritance because we got signiﬁcant better performance when we did this. The
reason this is faster is that normal inheritance has a small cost for every layer
of inheritance while templates are resolved at compile time. Template classes
gave performance characteristics indistinguishable from writing everything in
the same class.
To achieve a measurable time in our tests, each operation is executed 10 million
times with a random index. To avoid contamination from the random generator,
the indices are pre-generated and stored in an array, which is then read in order.
We also chose to give the random generator the same seed in each test to further
reduce variations.
The PC we ran our tests on had the following relevant speciﬁcations:
• OS: Microsoft Windows 10 Home
• Processor: Intel(R) Core(TM) i5-6600K CPU @ 3.50GHz, 3501 Mhz, 4
Core(s), 4 Logical Processor(s)
• L2CacheSize: 1024kB, L3CacheSize: 6144kB.
• Total Physical Memory: 15.9 GB.

Fenwick Tree
1.1
Introduction
The Fenwick tree is a solution to the partial sums problem. It consists of a tree
structure, containing log n layers on top of the array A of length n, where each
element is the sum of two elements in the previous layer (as shown in ﬁgure
1.1). To get the sum up to a certain index would require reading and summing
together at most one ﬁeld from each layer, instead of summing up each element
individually.
Figure 1.1: An array with a summation structure on top. The numbers are the indexes.
The blue lines connect sums of multiple ﬁelds to sums of a subset of these ﬁelds.
On the bottom we have the original array.

Using the same indexing as before the layer can be calculated examining the
position of the least signiﬁcant bit that is set to 1. To calculate the position in
the layer read the bits to the left of the least signiﬁcant set bit (the black part
of the binary index in ﬁgure 1.4b). Both the layers and the internal index inside

element 10(1010) belongs to layer 1 at index 2(10), and element 8 belongs to
layer 2 at index 0.
This way of storing the structure takes up an additional log n words of space
for the pointers to each layer. The total space usage is n + log n.
1.2
Operations
1.2.1
Sum Operation
Figure 1.5: The tree for the sum operation of the Fenwick Tree. Each number showing the
range the sum of the containing ﬁeld is over. The end of the range corresponds
to its index. The red nodes are the nodes which are visited in our example.
The sum operation up to an index i corresponds to making a traversal from
the node with index i to the root(index 0) in the Sum tree(see ﬁgure 1.5), and
summing up the values of all nodes on the path from node i to 0 (including node
i’s value).
As an example: to get the sum up to index 13, start reading the value at node
13(index 13), then add the value of node 9-12(index 12), and the value of node
1-8(index 8).
When implemented, going up one level in the tree is equivalent to ﬂipping the
lowest set bit in the index. Traversing to the root is then done by ﬂipping each
bit in turn and examining the resulting index. The traversal stops when the
root, index 0, is reached.
Looking at the example from before: Index 13(01101) become index 12(01100),
which become index 8(01000), which become index 0(00000). Notice that the
number of operations needed are the number of set bits in the index.

Update Operation
Figure 1.6: Figure 1.2 without showing the ﬁelds which where removed. Also the tree for

The update operation for an index i and delta ∆corresponds to making a
traversal from the node with index i to the root(index 16 in the example in 1.6)
in the Update tree, and updating the value of all nodes on the path from node
i to the root (including node i) by adding ∆to their value.
As an example: when updating index 5 by adding ∆we have to add ∆to node
5(index 5), to node 5-6(index 6), to node 1-8(index 8), and to node 1-16(index
16).
When implemented the traversal up the tree corresponds to ﬁnding the lowest
set bit in the index, and adding the value of that bit to the current index.
Looking at the example from before: the lowest bit in index 5(00101) is 1(00001)
so the next index is 6. The lowest bit of 6(00110) is 2(00010) so the next index
is 8. The lowest(and only) bit of 8(01000) is 8 so the next index is 16.

1.2.3
Access Operation
Figure 1.7: The tree for the update and access operations of the Fenwick Tree. The red

The access operation on a Fenwick tree for an index i can be done in 2 ways.
1. Do 2 sum operations: access(i) = sum(i) −sum(i −1)
2. Take the value on index i and subtract the value of all the direct children
of i in the update tree (ﬁgure 1.7).
While the ﬁrst approach is very easy to implement the second method will in
most cases access fewer nodes and at most access the same amount of nodes.

LSB(12) = 4.
The distances will be 2 and 1, which means the children to 12 is 10 and 11.
This means access(12) = content(12) −content(10) −content(11).

Table 1.1 shows the running time of the normal Fenwick tree and the Multi-

We have compared the running time with a Naive array and a Summation
array which have constant running time for the Update and Sum operation re-
spectively. We see the naive solutions do perform signiﬁcantly better than the
Fenwick trees in the operations they perform in constant time, being a factor
4 to 10 faster in sum and update. In the smaller data sizes they also perform
access faster by a factor of 5 to 11, though at 5 million they are comparable
with the normal Fenwick tree, and only a factor 2 faster than the multilayered.
Search is done at comparable time at all sizes, with the naive solution achiev-
ing consistent middle performance, which is expected as search is also done in
logarithmic time in the naive solution.
It generally seems like the normal Fenwick tree is faster than the multilayered
Fenwick tree in the Sum, Update, and Access operation, the multilayered Fen-
wick tree is however faster at the Search operation.
The reason that the multilayered Fenwick tree is faster than the normal Fenwick
tree is the way the layers are accessed in the search operation.
The search
operation will always look at exactly one element from each layer, and the way
==PAGE:18==
1.4 Chapter Conclusion
11
the layers are stored in the multilayered Fenwick tree allows it to simply keep
many of the top layers which have few elements in the cache.
The normal Fenwick tree do however have an advantage on the search operation
when we have reached the low layers, because there all the potential elements it
wants to visit will be in the same cache line, which is faster to retrieve compared
to the multilayered Fenwick tree which have to retrieve a cache line for each layer.
These diﬀerences between what they are good at shows in the test that the
normal Fenwick tree have a faster search operation when the Fenwick tree only
has 500 or 50k elements, while the multilayered Fenwick tree is faster when it
has 5 million elements.
Sum and update are generally faster than the search operation since they only
have to visit log n in a worst case scenario. For the normal Fenwick tree there
is a good chance that the ﬁrst couple of elements it has to visit will be in the
same cache line.
For the multilayered Fenwick tree these operations are a little more problematic.
It can still cache the upper layers, but there is a high possibility that some of the
layers it has cached will be skipped entirely in an operation. This is probably
what makes it slower than the normal Fenwick tree.
We also see that the single array implementation has better performance across
the board on the smaller data size, while the multiple array implementation
performs better for the search operation on the large data size.
We also see that the access operation universally takes less time than the sum
operation, thus vindicating the decision to implement access in this fashion, and
not as two sum operations.
1.4
Chapter Conclusion
As we have already discussed in the test section, the normal Fenwick tree seems
to be the best solution for relative small Fenwick tree while the multilayered
Fenwick tree seems to be the best solution when the Fenwick tree reaches a
certain size. This is of course only true if the performance oﬀall the operations
are weighted equally like we do in this report.
If search is a less important
operation, then the normal Fenwick tree would continue to perform better than
the multilayered Fenwick tree as the size of the tree grows.

Chapter 2
Making the Fenwick Tree
Compact
In this chapter we will look into ways to make the Fenwick tree as compact as
possible. The motivation is that using whole words to store numbers can use
several times the space that is actually needed to store them.
We will also look into how we eﬃciently can calculate how much space a Fenwick
tree needs.
It should be mentioned that since we already covered how the operations work
in chapter 1 we will skip that part and simply focus on how the individual
elements in the Fenwick tree is accessed. The sequence they are accessed and
updated in is the same.
2.1
Compact Arrays and Bit Arrays
This is a short introduction to some data structures which are used in our
compact solutions.
==PAGE:20==
2.2 Compact Multilayered Fenwick Tree
13
The compact array is an array which can store n integers using k bits per integer
for some value k. This makes this type of array use n·k bits of space rather than
n words of space. It is however a little slower to retrieve and update elements in
the compact array, because it will make some calculations before it can retrieve
or change a value it contains, and each value in the array will at most be 2k −1.
The bit-array is an array of bits which we have added some extra functions to.
It supports access and update operations, though with modiﬁed input. Rather
than a single index it takes two indices, which is the range the integer you want
to manipulate is over. E.g. accessing the integer stored from index 2 to 5 would
give you the 4 bit integer stored over those 4 bits. For practical reasons the
integer range shouldn’t be more than 64 bits long.
We will not go into further details of how these structures are implemented since
this is less relevant to the subject of this report. All operations on them can be
done in constant time.

Assuming k is the number of bits needed to store an integer from the original

the Fenwick tree. An element of layer 1 would require k + 1 bits to make sure
it can contain the sum of 2 elements of k bits, layer 2 would require k + 2 bits,
and so on. The bits needed for an element on layer L is k + L.
Running time and space usage
The retrieval and writing of elements in a compact array is constant, so the
asymptotic running time of the Fenwick tree is not aﬀected.
The space usage in bits will be: k · n + n + log(n) · w + O(log(n) · w).
k · n + n is the bits needed to store all the numbers in the compact arrays. The
formula will be explained in section 2.3.
The log(n) · w is the bits needed for the pointers to the compact arrays. (w is
the number of bits for a word), with the O(log(n)·w) being for the unused bits
in the end of the compact arrays.
2.3
Lower Storage Limit
Since we in this chapter are working on making the Fenwick tree as compact
as possible, it would be useful to have a formula which can tell us exactly how
much space a Fenwick tree requires.
Lemma 2.1 Let n be the number of elements in the Fenwick tree, k the number
of bits needed for an element in the original array, and bitcount a function
returning the number of set bits in an integer. The number of required bits in a
Fenwick tree is then given as: totalBits(n, k) = n · k + n −bitcount(n).
Proof
We will prove lemma 2.1 in two parts. First an induction proof for cases where
n is a power of 2. Afterwards we will prove it for other cases.
Induction proof
Since we are working with cases where n is a power of 2 then it will always be
the case that bitcount(n) = 1, so for these cases the formula from lemma 2.1
can be rewritten to formula 2.1:
totalBits(n, k) = n · k + n −1
(2.1)

Using totalBits to calculate the bitsize of the upper layers
Before going into the implementation of the concatenated version, we need a
formula to calculate the number of bits a tree would use if a number of bottom

• If the length of F is n, then the length n′ of F ′ can be calculated as
n′ = ⌊n/2⌋.
• While F used k bits for the elements on its bottom layer F ′ uses k′ = k+1
bits on elements in its bottom layer.
Since we know these 2 parameters we can use lemma 2.1 to calculate the bitsize
of F ′ meaning bitsize(F ′) = totalBits(⌊n/2⌋, k + 1).
We want to generalize this to removing the x bottom layers. To do this we
modify the calculations such that:
• n′ = ⌊n/2x⌋= n ≫x, where ≫is a right bitshift operation.
• k′ = k + x
So if F ′ is F with the x bottom layers removed, the bit size of F ′ can be
calculated as totalBits(n ≫x, k + x).

Space and time
Every calculation done to get the element out from the Fenwick tree is done in
constant time, which means that all the operations take the same asymptotic
time as the normal Fenwick tree.
The space usage is: n · k + n + O(w) bits where n is the number of elements in
the Fenwick tree, and k is the number of bits required for an element from the
original array.
n · k + n is the number of bits a bit-array requires to store the elements from
the Fenwick tree.
O(w) is how many bits in the end of the bit-array which is unused.
A third strategy, pointers to where the layers begin
Instead of calculating where the layers begin each time we can simply store
where each layer begin instead. While the calculations are constant time it is
still faster to simply pull out the answer instead. This approach does however
require log(n) more space to store these "pointers" in.
Space and time
The time it takes to retrieve an element in the Fenwick tree is as before constant,
so all the operations require the same time asymptotic as in a normal Fenwick
tree.
Compared to before we use log(n) more space to store these "pointers". This
bring us to a total space usage on: n · k + n + log(n) · w + O(w) bits, given the
number of elements n, number of bits k for an element from the original array,
and how many bits there is in a word w.
Notice that compared to the normal compact multilayered solution from section
2.2 we use O(w) instead of O(log(n) · w) space for the unused bit in the end of
our bit-array.

2.7
Chapter Conclusion
Far the simplest way to get a compact version of the Fenwick tree was to take
the multilayered approach and use compact arrays for each layer rather than
normal arrays. To do anything more compact than that we had to come up with
formula giving us the exact size of a Fenwick tree from the number of elements
it should contain, and their bit-size. Given this formula we were able to create
Fenwick trees which wasted at most a word of space and didn’t need pointers
to multiple layers.
Our most compact structures were the compact Fenwick tree and the concate-
nated Fenwick trees. The compact Fenwick tree stored the elements in the same
way as a normal Fenwick tree, but used the necessary amount of bits per ele-
ment rather than a word, while the concatenated Fenwick tree was a compact
multilayered Fenwick tree where we concatenated all the compact layers into
one bit-array.
We also made a modiﬁed version of the concatenated Fenwick tree which con-
tained pointers to where each layer started in the bit-array. This increased the
space usage, but gave an improvement in performance.
In the end we found the multilayered approaches worked better when the number
of elements exceeded a certain point. Furthermore, the concatenated Fenwick
tree with pointers performed just as well as the compact multilayered Fenwick
tree, but with lower space waste.

Chapter 3
Compromise Solution
In chapter 1 we introduced the uncompressed Fenwick trees which used a lot of
space. In chapter 2 we introduced the compact Fenwick trees which used mini-
mal amount of space, but on the other hand ran slower than the uncompressed
Fenwick trees because it requires extra calculations to access the elements in the
Fenwick tree.
This chapter will be about ﬁnding compromise solutions.
That is solutions
which use less space to store the Fenwick trees than uncompressed Fenwick
trees, but will try to simplify the calculations needed to retrieve the elements of
the Fenwick tree which should make it faster than the compact Fenwick trees
at the cost of using more space than them.

solution. If an element in the tree takes between 5 and 8 bits, then an 8-bit

solution. This holds as well for 17-bit and for 33-bit.

The compromise Fenwick tree does however perform better than the compact
multilayered Fenwick tree, which at least give us one structure that accomplishes
the goal for this chapter.

The compromise Fenwick tree involved only using built-in integer types, but

the numbers for that layer.
The "No Overlap" solution involved modifying

4.1.6
Implementation details
This subsection will contain some details needed to understand our source code,
and also explains where our source code deviates from what we have written in
this section.
b is a power of 2
In our implementation we have limited b such that b ∈{2x | x ∈N} for a
natural number x, and we simply store x rather than b. This makes it possible
to ﬁnd out which layer an index belongs to by looking at its least signiﬁcant set
bit, ﬁnd the bit position and divide the position by x.
For example, b = 8 = 23 meaning we should divide the bit position by 3. Index
640 is binary written as 1010000000, The least signiﬁcant bit is on bit position
7, ⌊7/3⌋= 2 meaning index 640 belongs in layer 2.
Another eﬀect of this is that in operations where we need to divide or multiply
by b we do shift operations with x instead.
Multilayer specialized
The code is written to work with a multilayered storage structure. This means
it will calculate the layer the index belongs to and the position in the layer it
is in. It also means we sometimes in the code can ignore the existence of layers
below the layer we are working in, and the current index doesn’t include all the
layers bellow (this is most aparant in the search method).

these will take. We will use k to refer to the size in bits of the elements in the
bottom layer. For normal B-ary trees this is the bitsize of the elements in the
original array, for the summation B-ary it is that + log(b).

A modiﬁcation of the normal solution where we use compact arrays instead of

An element use k + logb ·L where L is the layer the element is from and k is

every b’th element need ⌈log b⌉extra bits to store sums. Of these, every b’th
element need ⌈log b⌉additional bits to store the sum of these elements, and so

totalBitsb(n, k) ≈n · k + n · ⌈log b⌉

to each layer. So the total space usage become: totalBitsb(n, k) + log(n) · w

b-ary tree more interesting than the summation b-ary tree.

version of the Fenwick tree, the diﬀerence is less than a bit per element1, so it
is of limited interest.

we can see that given certain bit-sizes of the integers of the original array, the
b-ary can cut away nearly 1/3 of the space usage compared to the compromise

Chapter 6
Multi Threading
We have already shown the performance of the diﬀerent Fenwick trees both in
terms of data usage and speed in the previous chapter. In this chapter we will
try to boost the performance of some of the Fenwick trees even more by making
them multi-threaded.
What we will experiment with is not using multiple threads to execute one
operation, but rather how performance is aﬀected when we execute multiple
operations in parallel on the Fenwick tree.
We have two strategies for this. The ﬁrst strategy is that we can run every
operation except update in parallel, since update is the only operation which
modify the data in the structure. We refer to this strategy as semi parallel in
the test data. The other strategy is that we replace the data types we use with
their corresponding atomic version, and then execute update in parallel as well.
We refer to this strategy as atomic parallel in the test data. For this reason we
only examine structures that use a built-in data-type.
In both cases, it is implemented by replacing the for-loop executing the com-
mand with a parallel for loop to run multiple operations concurrently.

It seems like the Fenwick trees are only suitable to be multithreaded if they

atomic, since update is the only operation which gains anything from this.

Conclusion
In chapter 2 we concluded that the number of bits needed to store the elements
from a Fenwick tree is n · k + n −bitcount(n) bits, when it is made an array of
k-bits integer of length n.
We achieved roughly this storage usage with the compact Fenwick tree from
section 2.4, but we achieved overall better performance with the concatenated
Fenwick tree from section 2.5 when we use log(n) extra words for pointers to
where the diﬀerent layers begin in the bit-array. We have also shown that this
extra space usage is negligible as soon as the Fenwick tree contains a couple
thousand elements or more.
In chapter 3 we worked with storage schemes which use less space than the un-
compressed Fenwick trees from chapter 1 but should be faster than the compact
Fenwick trees from chapter 2. We came up with 2 diﬀerent solutions that would
use at most twice the space of the compact multilayered Fenwick trees, while
in theory being faster than the compact Fenwick trees. Our implementation of
"No Overlap" from section 3.2 was however slower than the compact multilay-
ered Fenwick tree. The compromise Fenwick tree from section 3.1 did however
perform better than the compact Fenwick trees. In chapter 5 we can see that the
compromise Fenwick tree is between 20% and 25% faster than the concatenated
Fenwick tree.
In chapter 4 we created a b-ary version of some of the structures in the other
chapters. To be more speciﬁc we created a b-ary version of the compact mul-
tilayered Fenwick tree from chapter 2 and the compromise Fenwick tree from
chapter 3. We conclude that the b-ary structures are a viable alternative, where
performance in either the update or sum operation can be sacriﬁced for better
performance in the other operations. Furthermore the version using standard

Theory
The partial sum problem
Let A be an array of k-bit integers, with |A| = n. The partialsums problem is
to build and maintain a datastructure supporting the following operations on
A.
1. sum(i): return the value of Pı
k=1 A[k].
2. search(j):return the smallest index i such that sum(i) ≥j
3. update(i, ∆: set A[i] ←−A[i]+∆for a given ∆such that 0 ≤A[i]+∆< 2k.
4. access(i): return A[i].
There are two trivial solutions to this problem. The ﬁrst is storing A. This
gives O(1) time for update, but O(n) time for sum, as up to n numbers must
be summed. Search will also end up using O(n) time, as it requires trawling
through A from the beginning.
The second solution is storing the sum in place of the value. This gives O(1) for
sum, but requires O(n) for update, as up to n entries may require modiﬁcation.
Search can here be implemented in O(log n) time, as it can be done by binary
search.
Fenwick Trees
The Fenwick tree data structure supports all these operations in O(log n) time.
The Fenwick tree can achieve this by replacing some of the original values
in the array with sums of more ﬁelds.
A Fenwick tree F of array A can be built by copying A into F. Then build
a binary tree over F which uses all the elements in F as leaves, and where every
node becomes the sum of its children. Then the last step is that every node
should write their own value down into their right child (starting from the root
of the tree, so nodes are written down into by their parents before they write
down into their own children.) The new values in the leaves or rather in F
makes F a Fenwick tree (all the other nodes than the leaves is not considered
part of the Fenwick tree).

and so on. (F and A are 1 indexed here)
Tree representation:
The tree representation of the Fenwick tree can be made by placing all the values
from the array in a tree depending on the binary representation of their index.
The index 0 will be used as the root, with the value 0. A node B is the child
of a Node A if A is has the biggest index which is identical to B’s index except
B has one more bit set (e.g. 1 and 2 could both be the parent of 3, since 3 just
has one more bit set for the both of them, but 2 is bigger than 1 making 2 the
parent). Figure 2 is an example of a tree representation of a Fenwick tree.
If the tree representation is traversed from an index k to the root, then the
sum of all the nodes corresponds to the partial sum of the original array to the
index k.
Figure 2: Tree representation of the Fenwick tree built in Figure 1. In the top
of the nodes the index is shown followed by the binary representation of the
index. Underneath the value of the node is shown.
Get the sum:
To get the sum up to an index i you need to sum as many numbers from the
Fenwick tree together as there is set bits in i. To get partial sum up to an index
i do the following steps:
1. Set the sum to 0.
2. Add F[i] to the sum.
3. Get i′ by copying i and ﬂipping the rightmost set bit.
4. if i′ != 0 set i = i′ and return to step 2. If i′ == 0 then return sum.
Change Value:
To add v to an index i do the following:
1. Check if i is out of bounds, end if it is.

Aim of our project
Create multiple versions of the Fenwick tree, and compare them to each other
on performance for time and space consumption. These versions will include:
• A both naive solutions.
• A Fenwick tree with no steps taken to compress the data.
• A Fenwick tree with multiple entries compressed together into single ma-
chine words.
• A version where the compressed version is built using the given array
memory, with the unused memory being released.
• A version where each level of the tree is put into its own array.
• All versions adapted to run on multiple CPU cores in parallel.
• The partial sum function of the C++ standard library.
