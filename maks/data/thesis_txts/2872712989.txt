Abstract
The Lab Streaming Layer (LSL) is a widely used framework for real­time collection and
synchronization of time series data, particularly in applications involving physiological and
neuroscientific measurements. However, existing LSL implementations require platform­
specific integration efforts, posing challenges for programmers aiming to build cross­
platform applications, including mobile platforms. This thesis presents the design, imple­
mentation, and evaluation of a set of cross­platform Flutter packages for LSL, enabling
programmers to incorporate synchronized data streaming functionality into mobile appli­
cations targeting Android, iOS, macOS, and Windows. The packages’ architecture was
guided by a set of requirements aimed at minimizing platform­specific code, ensuring
maintainability, and aligning closely with the core concepts of LSL. The implementation
leverages Flutter’s plugin system and a Foreign Function Interface (FFI) to interface di­
rectly with native LSL libraries while providing a Dart­idiomatic API to programmers. The
developed system was validated through a series of functional and non­functional evalu­
ations. Functional validation ensured the correct behavior of core streaming operations
and demonstrated the practical applicability of the system in research­oriented data acqui­
sition scenarios. Non­functional validation assessed synchronization performance across
platforms by analyzing timestamp deviations. The results demonstrate that the developed
plugin provides accurate synchronization performance comparable to existing native im­
plementations, while offering a cross­platform integration approach intended to simplify
adoption for programmers.

Introduction
1.1
Background
In today’s mobile age, researchers are relying increasingly on the use of smartphones
to gather and analyze data [1]. Mobile devices can acquire various data longitudinally,
such as location, step count, and heart rate, to name a few. In addition, tech companies
such as Apple have shown a growing interest in physiological monitoring, incorporating
electrocardiogram (ECG) functionality into their smartwatches [2] and exploring electroen­
cephalogram (EEG) recording via in­ear headphones [3]. As commercial­grade sensor
devices such as EEG and ECG monitors become ever better, smartphones can acquire
more fine­grained data.
This presents new research opportunities for smartphones as “pocketable labs” [4]. For
example, traditional experiments involving brainwaves are confined to labs, which, in most
cases, are not a natural environment for the subjects of the experiments. However, us­
ing sensors that are designed for their users, as opposed to their operators, allows for
acquiring similar data in a more naturalistic or familiar environment.
Gathering multimodal recordings of neural, physiological, and behavioral data is an inte­
gral part of research in many fields, such as neurobiology. To model the interrelationships
of these different data streams, they need to be synchronized to a high precision [5].
For this purpose, researchers have in the past needed to design and implement their
own bespoke solutions for their specific setups. In an effort to provide a more robust
multimodal data acquisition solution, the Swartz Center for Computational Neuroscience
(SCCN) started an open­source project called The Lab Streaming Layer in 2012 [6], which
has since gained considerable traction.
LSL’s popularity can be explained by the combination of a number of features. First and
foremost it eradicates the need for complicated hardware synchronization solutions for
sensors, that require specific engineering for a given setup. LSL is on top of that cross­
platform1 in the sense that its source code can be compiled and run on any platform,
and SCCN provides interfaces for C, C++, Python, Java, C# and Matlab, making it widely
accessible.
At the time of LSL’s inception, smartphones were neither as ubiquitous nor as powerful
as today. Hence, the flora of applications made using liblsl, which is the core library of
LSL, consists almost exclusively of desktop applications2. Although LSL can be run on
iOS and Android, programming for mobile devices requires significantly more overhead
than, for example, creating an interactive Python script.
Canonically, to create an app for one of the two major mobile operating systems, one
needs to know either Swift/Objective­C or Kotlin/Java. Mastering both is something few
do, whose main field is not mobile development. However, with the advent of Cross­
Platform Mobile Development (CPMD) Frameworks such as Flutter3, developing for both
operating systems at the same time is possible with a single language (Dart) and a single
UI framework.

1.2
Definition of Terms
The following terms are used throughout this thesis.
• Time series data refers to sequences of measurements where each observation
is associated with a timestamp. The temporal ordering of these measurements is
essential for subsequent analysis.
• Nominal sampling rate denotes the intended or target rate at which data samples
are expected to be collected, typically specified by the configuration of the recording
system or device.
• Effective sampling rate denotes the actual rate at which data samples are col­
lected, calculated as the inverse of the mean sampling interval over a given period.
• Jitter refers to short­term variability in the timing between consecutive data samples.
It is quantified as the standard deviation of the inter­sample intervals, representing
fluctuations around the effective sampling rate.
• Drift refers to a growing offset between two sets of timestamps over time.
• Synchronization, in the context of LSL, refers to the process of estimating and cor­
recting differences between local device clocks and the host clock. This correction
accounts for both systematic clock offset and variability (jitter), allowing recorded
timestamps to be expressed relative to a common reference clock.
• Resolving, in the context of LSL, refers to the discovery and collection of stream
metadata. A stream resolver is similarly an object that performs this resolving.
• Cross­platform, as used in this thesis, refers to the ability of the developed system
to operate on multiple major operating systems, specifically Android, iOS, macOS,
and Windows.
• Platform­specific denotes functionality or configuration that is tied to a particular
platform, and therefore requires separate handling when targeting multiple plat­
forms.
• Flutter package refers to a software package or library made for Flutter, and con­
tains only code written in Dart.
• Flutter plugin refers to a Flutter package that contains both code written in Dart
and in native code, e.g., in Kotlin, Swift, or C++.
• System, in the context of this thesis, denotes the complete set of software artifacts
developed to achieve the design and functional objectives specified herein. This
includes all Flutter packages created as part of the solution.
• Programmers refers to individuals who utilize the developed package, including
both researchers and software developers.

1.3
Research Question
The increasing use of smartphones as acquisition devices for time series data calls for
on­device synchronization capabilities. The technology for this acquisition and synchro­
nization exists in LSL, but integrating it into applications across multiple platforms re­
quires platform­specific knowledge of each targeted platform. Therefore, this thesis will
investigate how the core LSL library can be implemented as a cross­platform system, im­
plemented as a set of packages in Flutter. To this end, the following questions will be
addressed.
1. What do the software architecture and API design of such a system look like?
2. How can the design of the system minimize the platform­specific integration steps
required of a programmer using the system?
3. How does the system compare to existing solutions with respect to supported func­
tionality, cross­platform capability, and synchronization performance?
1.4
Goals and Methods
From the research question and its sub­questions, the following goals are derived.
1. Investigate LSL in the Context of Mobile Devices The LSL framework itself, its
applications, and potential will be investigated through existing literature and arti­
facts, such as software packages or products.
2. Design a Suitable Architecture for the Package The software architecture and
API design are guided by a set of requirements defined in the design phase. The
package should keep platform­specific integration steps needed for programmers

Background and Related Work
The idea to support LSL on a mobile device is not new. However, the creation of a cross­
platform, open­source package enabling programmers to use LSL for their applications
has not been explored in the literature.
2.1
The Lab Streaming Layer
LSL [6] is a software framework for real­time collection, synchronization, and transmission
of time series data from multiple sources. It is widely used in applications involving phys­
iological, behavioral, and neuroscientific recordings, where the ability to combine data
streams from heterogeneous sensors with precise temporal alignment is essential.
At its core, LSL provides a unified API that allows devices and applications to publish and
subscribe to data streams over a local network. Each data stream is published to the
network via an outlet, while receiving applications subscribe to streams via inlets. The
underlying communication between peers is managed by the core LSL library (liblsl),
which handles network discovery, stream resolution, metadata exchange, data transmis­
sion, and synchronization. The specific protocols used for these network operations are
described in more detail in the following sections.
An overview of the core LSL components and their interactions is shown in Figure 2.1.
2.1.1
Discovery Protocol
The discovery protocol is the initial step in establishing communication between inlets
and outlets in the LSL framework. When an outlet is initialized, it creates a set of User
Datagram Protocol (UDP) sockets on the multicast port (16571) to listen for broadcast and
multicast queries. These queries may specify properties such as stream type or channel
count. Each outlet evaluates incoming queries and responds only if the request matches
its own stream parameters. The outlet’s response provides essential metadata required
for subsequent communication phases.
On the client side, a stream resolver is responsible for locating available streams on the
network. This resolver also creates UDP sockets, through which it transmits queries with
the desired properties to active multicast, broadcast, and unicast addresses. Additionally,
it opens a UDP socket to receive responses from matching outlets.
2.1.2
Subscription Protocol
Following successful discovery, a subscriber can connect an inlet to the identified outlet.
The subscription process establishes a TCP connection to the outlet’s network endpoint,
as obtained from the discovery response. This connection is negotiated through a two­
way handshake resembling an HTTP/1.1 GET request–response exchange. During this
phase, critical transmission parameters such as protocol version, byte order, and buffer
size are agreed upon. Moreover, the peers exchange the stream information object and
a predefined test­pattern to ensure protocol compatibility. Upon successful negotiation,
a stable data channel is established, enabling the continuous flow of data from outlet to
inlet.
2.1.3
Stream Transmission Protocol
Time­series data are transmitted as a byte stream segmented into packets by the net­
work layer. For real­time applications, individual samples can be marked for immediate

transmission, signaling the system to prioritize delivery as soon as network conditions al­
low. The byte stream is composed of encoded message frames, each containing a loss­
lessly delta­compressed timestamp and a sequence of data values encoded according
to the previously negotiated format. Transmission can occur on a per­sample basis or in
multi­sample chunks, depending on the desired trade­off between latency and throughput.
While the network operates fundamentally on single­sample packets, high­level functions
in LSL abstract away these details to support both transmission modes.
2.1.4
Metadata Transmission Protocol
In addition to time­series data, LSL must also transmit essential metadata, including
stream name, type, channel count, and nominal sampling rate. This metadata is transmit­
ted once and is treated as out­of­band information. That is, it is requested over the same
port as in­band data, but it follows a separate logical pathway. Metadata is formatted in
plaintext using an attribute­free subset of XML. To ensure interoperability across devices
and applications, it is recommended that metadata adhere to content­type specifications,
such as EEG, PPG, Audio etc.
2.1.5
Time Synchronization Protocol
LSL employs a time synchronization protocol based on the principles of the Network Time
Protocol (NTP) [7] to estimate both the clock offset (θ) and round­trip time (δ) between
peers. To account for clock drift, synchronization exchanges are performed periodically,
typically every 5 seconds when clock synchronization is enabled. The NTP clock filter
algorithm is used to mitigate the influence of random transmission delay spikes.
The synchronization process involves the initiating peer sending a timestamp t0, to which
the receiving peer replies with t1 (time of receipt) and t2(time of resend). Upon receiving

requiring to keep a history of past timestamps. However, the filter may require a short ini­

information.
2.2
Related Work
Several systems and frameworks have been developed to support LSL­based data ac­
quisition across various platforms. This section reviews notable efforts, including official
tools such as LabRecorder, domain­specific frameworks like the Excite­o­meter, mobile
applications developed for Android, and existing cross­platform packages targeting Flut­
ter. Each highlights different trade­offs in terms of platform support, extensibility, and
intended use cases.

2.2.1
LabRecorder
LabRecorder is the official application for recording LSL data streams to disk in the Exten­
sible Data Format (XDF). Although it supports Windows, macOS, and Linux, it is primarily
optimized for Windows, where it is available as a prebuilt executable and generally re­
quires minimal setup. On macOS, particularly newer versions, users may need to build
the application from source, as precompiled binaries are often unavailable or incompatible
without additional configuration.
2.2.2
Excite-o Meter
LSL has seen widespread usage in Augmented Reality / Virtual Reality (AR/VR) research.
In [8] Wang et al. review its usage in the field, where 549 papers have been published
between 2015 and 2023.
They contend that LSL should be used more frequently in
AR/VR applications due to its potential for comprehensive data collection and manage­
ment. In their conclusion they state that in the field of Human Computer Interaction (HCI)
the use of multi­modal data is becoming more popular and even necessary for human be­
havior analysis. As wearable technologies used for AR/VR displays become ever more
portable, AR/VR applications using them are anticipated to become more widely used by
consumers in their daily lives.
One of the technologies mentioned in [8] is the Excite­o­meter, which is an open­source
framework that collects and synchronizes data using LSL. In [9] Quintero et al. show the
integration of cardiac activity signals such as ECG and photoplethysmography (PPG) to
obtain heart rate variability, which are considered valid stress markers. All of this was
done in an interactive VR application, which provided real­time analysis. The framework
in a sense provides what the plugin specified in this thesis aims to provide, with some key
differences. The Excite­o­meter is implemented in Unity. This is prudent, given that it is
aimed at AR/VR applications which in many cases also rely on Unity.
2.2.3
Pocketable Labs
liblsl is written in standard C++ and its dependencies are all cross­platform, making
the library itself cross­platform1, in the sense that its binary can be built to run on most
major operating systems. Support for LSL on Android devices is available through a Java
interface 2, which Blum et al. [10] use to implement open­source Android applications
for synchronized streaming and recording of time­series data. With the growing com­
putational power of smartphones and the array of built­in sensors offered on common
devices, using them as research instruments is gaining favor. They create three apps:
Send­a, which streams sensor data; Record­a, which records data from the network; and
Sine­Wave app, which streams artificial data. The apps were evaluated in four different
scenarios:
1a. Sine­Wave data streamed from Android device, recorded on a PC,
1b. Sensor data streamed from Android devices, recorded on a PC,
2. Data streamed from PC, recorded on an Android device,
3. Sensor data streamed from Android device, recorded on the same device (and on
LabRecorder for validation).
Their results show a high correlation between hardware sensor data from the two Android
devices but a lower correlation between the data from their internal sensors, likely due to
different hardware. Furthermore, in scenario 3, the data recorded on the Android device

2.2.4
The liblsl Flutter Package
A drawback to the aforementioned integration is its exclusion of iOS devices. The appli­
cations are developed for Android natively, and therefore, analogous implementations for
iOS would need to be written from scratch in Swift. Developing exclusively for Android
may be prudent in a global context, given that its global market share is roughly 72% [11].
However, in some parts of the world that market share is more evenly split, e.g., in North
America where the numbers are slightly in favor of iOS, and all Scandinavian countries
(except Finland). Therefore, being able to develop for both platforms can be crucial in
these areas.
Various parties have addressed this code repetition issue through what is now known as
CPMD frameworks. Different strategies are used across frameworks to achieve a unified
approach to app development. There may be performance decreases when using such
frameworks, but it has been shown that they can perform equally or better than a native
benchmark [12].
Oliveira et al. [13] compare the resource usage of mobile frameworks, highlighting Flutter’s
compiled approach, which translates source code to native machine code for Android and
iOS. In contrast, React Native uses an interpreted model with JavaScript and a bridging
layer. They find that Flutter generally outperforms React Native in CPU­ and IO­intensive
benchmarks, likely due to its native compilation.
Flutter’s cross­platform capabilities have made it a compelling choice for applications re­
quiring both responsiveness and wide device support, and another Flutter package for
LSL called liblsl3 has recently been published. Although still considered experimental
and under active development, it is listed in the official LSL documentation as a supported
language wrapper. The rationale behind this package aligns with the motivations of the
system presented in this thesis, enabling real­time physiological data streaming and syn­
chronization across platforms within the Flutter ecosystem.

3
Software Design
The design of this plugin was guided by two primary goals. First, to ensure accessibility
for a broad audience of programmers by offering high­level abstractions and an object­
oriented interface. Second, to expose as much of the underlying functionality of the LSL
framework as possible. Balancing these two objectives within a single package proved
challenging. As a result, the final design separates concerns into two complementary
packages: lsl_bindings, which provides low­level bindings to the LSL C library and
includes the necessary platform­specific binaries, and lsl_flutter, which offers a high­
level, Dart API built on top of those bindings.

3.1
Requirements
The system was designed with the following core requirements in mind, based on the
project’s objectives, and insights from related work. While the distinction between func­
tional and non­functional requirements is traditionally maintained, this line often blurs in
the context of software libraries and APIs, where concerns such as portability, usability,
and performance are tightly coupled [16]. Therefore, the following unified set of require­
ments guided the design process and served as a blueprint for both implementation and
evaluation:
R1 The system must work across major platforms (Android, iOS, macOS, Windows).
R2 The system should provide a Dart interface to the LSL library.
R3 The system should support the creation of LSL outlets and inlets for streaming and
receiving time series data.
R4 The system should support key LSL features such as synchronization, and marker
streams.
R5 The system should expose event­driven or callback mechanisms for handling in­
coming samples in Dart.
R6 The system shall maintain timestamp alignment such that drift between streaming
and recording devices remains within 1 second over a continuous 5­minute record­

3.2
Domain Model
In LSL, all data transmission is conducted via outlet and inlet objects, which serve as end­
points for pushing and pulling samples across the network, respectively. Both inlets and
outlets are instantiated with a corresponding stream information object, which encapsu­
lates metadata describing the stream (e.g., name, type, channel format). Notably, stream
information objects can also exist independently of any inlet or outlet, for instance, during
stream resolution when available streams are queried prior to establishing a connection.
An overview of the core classes and their relationships is illustrated in Figure 3.2.
3.3
Layers
The lsl_flutter package is structured into three layers: a worker layer, a manager
layer, and an adapter layer. The worker and manager layers constitute the public API and
are directly accessible to programmers using the package. These layers are designed
to abstract away all lower­level implementation details, ensuring that the user does not
need to manage artifacts such as native pointers. The worker layer and manager layer
expose a very similar API with the caveat that the worker API is asynchronous, while
the manager layer is not. The worker layer uses isolates to wrap the manager layer
in asynchronous methods, while the manager layer communicates with the lower­level
adapter layer through well­defined interfaces, maintaining a loose coupling that enhances
modularity.
The adapter layer implements the interfaces consumed by the manager layer and encap­
sulates the direct interaction with native code. Conceptually, this layer plays a role similar
to a repository in traditional software architecture. While liblsl is not a database, it can
be analogously considered one, with inlets and outlets acting as network connections.
Extending the analogy, Dart’s FFI, can be seen as the equivalent of an object­relational
mapper (ORM), bridging Dart and native C++ code.
At the lowest level is the liblsl native binary, which contains the core implementation
of the Lab Streaming Layer. This component lies outside the control of this project but
serves as the foundation for all data transmission.

3.4
Sample Strategies
When creating a stream, liblsl allows specifying the format of the data, such as 16­bit
integers or 32­bit floats. While this format is not strictly enforced at runtime, it defines what
downstream consumers expect. To ensure data integrity and compatibility, it is important
that pushed samples match the declared format.
In contrast, Dart provides only two built­in numeric types, int and double, alongside the
String type for textual data [17]. The liblsl API, however, distinguishes between four
integer formats, two floating­point formats, and one string format. Moreover, for each of
these formats, three distinct functions exist for pushing samples to an outlet, resulting in
a total of 21 possible push functions.
Given this complexity, it is desirable for the Flutter package to offer strong static typing,
i.e. type constraints that can be verified at compile­time. For example, if a user creates
a stream outlet specifying an integer channel format, they should not be able to push
samples of type double. Ideally, when a user instantiates a stream outlet service and
provides the channel format, the appropriate strategy for handling the sample type should
be automatically selected. Furthermore, the Dart type expected by the outlet should be
clearly inferred and enforced, minimizing the risk of runtime type errors. This design can
be effectively achieved by employing type generics in combination with the strategy pat­
tern [18], which allows the selection of appropriate runtime behavior based on specified
parameters. Figure 3.4 shows a class diagram of the Outlet manager’s and adapter’s re­
lation. When the OutletManager is instantiated the appropriate strategy is chosen based
on the StreamInfo class passed, which contains the specified channel format.

be resolved.

This method resolves streams matching the given predicate pred (e.g., type='PPG').
The minimum parameter specifies the minimum number of matching streams to re­
solve.

This method resolves streams where the property prop equals the specified value.
The minimum parameter again specifies the minimum number of streams to resolve.
• List<ResolvedStreamHandle> getStreamHandles()

To initialize an OutletManager, a StreamInfo object is required along with an OutletConfig,
which includes the following properties:
• int chunkSize specifies the chunk granularity in samples for transmission.
• int maxBuffered specifies the maximum amount of data to buffer, in seconds.
• OffsetMode mode: the pre­processing strategy to apply to timestamps before sam­
ples are pushed to the stream. The available options are:
– none: No pre­processing is applied.
– record: Records the difference between an externally provided timestamp
(e.g., from a sensor) and the local device time at regular intervals. These val­
ues are stored in the offsets list for later analysis.
– applyFirstToSamples: Computes the offset between the first received sam­
ple’s timestamp and the local clock, and applies this fixed offset to all subse­
quent samples.
• double offsetCalculationInterval: the interval (in seconds) at which offset mea­
surements should be recorded when using record mode.
The OffsetMode pre­processing option addresses the fact that while sensor­provided
timestamps are often preferable, due to transmission delays, their clocks may drift over
time. However, the relative timing (deltas) between samples is typically reliable. In con­
trast, streaming devices such as smartphones and computers maintain more accurate
clocks through regular synchronization. Applying a fixed offset on the device can there­
fore improve alignment when the sensor’s clock is known to be offset.

Implementation
This chapter details the implementation choices underpinning the three modes of inter­
action outlined in chapter 3 (Bindings API, Manager APIs and Worker APIs), including
FFIs and concurrency handling. In addition, platform­specific implementation details and
synchronization mechanisms will be examined.
4.1
LSL Bindings - lsl_bindings
As mentioned in chapter 3, the Dart bindings for the native LSL code were encapsulated
within a separate plugin. This design decision was motivated by several considerations.
First, the lsl_flutter package does not yet support the full feature set of LSL. In cases
where a user requires functionality beyond what is currently exposed, isolating the bind­
ings in a dedicated plugin avoids the need for duplicating or rewriting code; users can
instead directly depend on the lower­level lsl_bindings package to access extended
capabilities. Second, should LSL be ported to Dart in the future, the modular structure
simplifies the integration of such a native Dart implementation, as the bindings would not
be entangled with the higher­level package’s logic.
To interface Dart with the native LSL library, the plugin relies on Dart’s FFI capabilities.
FFIs provide a mechanism for invoking functions written in another language, typically one
that compiles to native machine code such as C or C++. This interoperability is crucial for
projects that require low­level system access, integration with existing libraries, or perfor­
mance optimizations beyond what the host language provides. In doing so, FFIs bridge
differences in calling conventions, memory layouts, and data representations between
languages. For example, a high­level language with automatic memory management
such as Dart must handle raw pointers and manually allocated memory when interacting
with C libraries.
The specifics of these low­level interactions are governed by Application Binary Interfaces
(ABIs), which define how functions receive arguments, return values, and manage mem­
ory at the binary level. Most C libraries strive for ABI stability, meaning that their interface
remains consistent even as internal implementations evolve. This property is particularly
advantageous for LSL, whose C API is ABI stable despite being implemented in C++.
The stability of the LSL ABI has enabled the creation of language wrappers for platforms
such as Java (via JNI), and in the case of Dart, the dart:ffi library provides similar func­
tionality. Complementing this, the dart_ffigen package can automatically generate Dart
bindings from C header files, greatly reducing the manual effort required to maintain the
binding layer (see section A.1).
To generate the necessary bindings, the LSL source code is included within the plugin’s
source tree under src/. Naturally, the generated bindings shaped several aspects of the
lsl_flutter package’s design, in particular, the extensive set of available functions for
pushing samples to outlets and pulling samples from inlets, as well as the differences
in data type representations between Dart and C, which directly influenced the sample
strategy discussed in section 3.4.
Importantly, the plugin requires access to a dynamic library on each target platform, con­
taining the symbols defined by the generated bindings. Ensuring the presence and proper
integration of these libraries required platform­specific solutions. The following subsec­

be found in section A.3.

4.2.1
Concurrency
All Dart code executes within entities known as isolates, which function similarly to threads
but operate with isolated memory spaces [20]. Each isolate maintains its own event loop
responsible for processing events enqueued in its associated event queue. In Flutter
applications, the primary isolate, commonly referred to as the main or UI isolate, handles
the majority of application logic and is responsible for critical operations such as rendering
the user interface. To update the UI, Flutter schedules repaint events on the main isolate’s
event queue at a rate matching the device’s frame rate, typically 60 times per second.
Since the event loop processes events in a first­in, first­out (FIFO) manner, any long­
running tasks that precede these repaint events can delay UI updates, resulting in per­
ceived stutter or unresponsiveness. To mitigate this, asynchronous programming con­
structs such as Futures and Streams are frequently employed. However, native functions
in Dart are executed synchronously and can block the UI thread even when invoked within
asynchronous wrappers. To address this limitation and ensure responsive performance,
the package introduces worker classes that manage the delegation of computational tasks
to background isolates, one set for handling outlets and another for inlets.
For both outlets and inlets, a similar communication pattern was employed, whose ini­
tialization is illustrated in Figure 4.1. Initially, (1) the worker class (running on the main
isolate) creates a receiver port, which serves as the channel for receiving messages from
the worker isolate. (2)­(3) The corresponding send port of this receiver is passed as an
argument when the worker isolate is instantiated. (4) Upon initialization, the worker isolate
likewise establishes its own receiver port and (5) responds by transmitting its associated
send port back to the main isolate via the main isolate’s provided send port.
The outlet worker passes commands and associated data from the main isolate to the
worker isolate. On the other hand, to ensure that the handling of LSL inlets aligns with

4.2.2
UDP Multicast
To enable the Lab Streaming Layer’s discovery mechanism, support for UDP multicast
is essential, as it underpins the protocol used to advertise and locate streams on the
network. However, receiving multicast traffic is often restricted by mobile and desktop
operating systems for security and battery­related reasons.
Android
By default, the Android operating system does not permit the reception of UDP multi­
cast messages. To enable the network protocols of LSL to function correctly, it is nec­
essary to acquire a so­called multicast lock when receiving multicast packets2. At the
time of writing this thesis, several Flutter plugins and packages provided the required
multicast lock functionality. However, most of these included additional, unnecessary
features that were irrelevant to the specific needs of the lsl_flutter package. The only
package offering the precise functionality needed lacked null safety and was therefore
incompatible with the lsl_flutter package. Consequently, a new Android­only plugin,
carp_multicast_lock, was developed with the sole purpose of acquiring and releas­
ing multicast locks. This plugin has been made publicly available through the central
Dart/Flutter repository at https://pub.dev/packages/carp_multicast_lock.
For applications using the plugin to be able to make streams discoverable from Android
devices the following permission deceleration must be made in the AndroidManifest.xml
file.

5.3.1
Procedure
The experimental procedure closely followed that of [15]. In the original study, two individ­
uals (a dyad) were seated side by side and shown a series of video clips intended to elicit
specific emotions. To identify appropriate stimuli, Bizzego et al. conducted a pilot study to
select video clips that reliably evoked the target emotional states. This experiment adopts
the same set of validated clips, with a few minor adjustments:
• Embarrassment: A scene from the film When Harry Met Sally.
• Sadness: A scene from the film Titanic.
• Fear: While Bizzego et al. used a clip from The Walking Dead, this was deemed too
vague (and perhaps gory for some participants) for replication. Instead, the Swedish
short film Blund’s Lullaby was used.
• Calmness: A video of a beach accompanied by ocean sounds.
• Romance: A scene from the film Notting Hill.
Notably, the emotion of pride was omitted in this replication. In the original study, pride
was elicited using a clip of the Italian national football team, which was considered cul­
turally resonant for the original participants, all of whom were Italian. Given the diverse
nationalities of the participants in this experiment, this stimulus was excluded. Video de­
tails can be found in section B.2.
An iPhone 11 and a Google Pixel 4a were used in all sessions. Both devices had the
Sensors app installed, as described in subsection 5.1.1. The iPhone 11 was paired with
a Polar Verity Sense sensor for recording PPG data from the forearm, while the Pixel 4a
was paired with a Muse S headband, which also records PPG signals.
In each session, one participant was fitted with the Muse S headband and handed the
Pixel 4a, while the other wore the Polar Verity Sense and used the iPhone 11. Device
setup, including sensor pairing and the configuration of marker buttons required during
the experiment, was performed by the experimenter (see Figure 5.3). The participants
were seated side by side in front of a screen controlled by the experimenter. Participants
were asked to stay as still as possible, and the participant wearing the Polar Verity sense
was instructed to not use the hand, that the sensor was strapped to, to press the marker
buttons. Prior to each video clip associated with a target emotion, the experimenter in­
structed the participants to press the corresponding marker button to serve as a reference
point in the data. At the conclusion of the final video, participants were again instructed
to press the designated end marker button, signaling the end of the session.
The original study recruited 62 pairs (124 participants). This was not feasible to achieve
within the time constraints of this project, so recruited pairs were only 5 in total (10 partic­
ipants).
In addition to replicating the paired­subject experiment described by Bizzego et al., the
experimenter conducted a self­administered trial, wearing both sensors simultaneously
and manually triggering the buttons on both mobile devices at the designated times. This
additional procedure served as an assessment of the effectiveness of the pre­processing

7
Discussion
The development process was guided by the dual goals of exposing the full capabilities
of the LSL while minimizing platform­specific integration steps for programmers. To sup­
port this, the system was structured into three architectural layers: a low­level FFI­based
bindings plugin (lsl_bindings), a mid­level manager API, and a high­level isolate­based
worker interface (lsl_flutter). This layering ensures a clean separation of concerns.
While lsl_bindings provides access to the complete native LSL API for advanced use
cases, the manager and worker layers abstract away native interoperability and present
a Dart­idiomatic interface for resolving streams, managing timestamps, and pushing or
pulling samples.
Platform­specific complexity, such as dynamic linking, multicast networking, and build
system quirks, was encapsulated within the system. For instance, Android uses CMake
integration, Apple platforms rely on a bundled .xcframework, and Windows builds re­
quired suppression of default install behavior. Multicast lock handling on Android is built
into lsl_flutter, though developers must still request the necessary entitlement on Ap­
ple devices.
Type safety and extensibility were key concerns. The system uses type generics and the
strategy pattern to associate channel formats with the correct data­handling logic, allowing
new formats to be added with minimal friction and reducing the risk of runtime errors.
Concurrency is handled via isolate­based workers, enabling background data streaming
without blocking the main isolate. While this required careful coordination of inter­isolate
messaging, it integrates well with Flutter’s reactive architecture.
The resulting system is modular, ergonomic, and maintainable. At the time of writing, the
lsl_flutter package scores 150 out of 160 on pub.dev, with over 95% of the public API
fully documented, reflecting the emphasis placed on usability and long­term support.
7.1
Validation
As shown in Table 6.1, the estimated 5­minute clock drifts were generally well below one
second. Notably, the drift observed in the Sensors applications was comparable in mag­
nitude to that recorded by the LabRecorder. However, when the macOS version of the
Sensors app acted as the streaming device, the drift was substantially higher across all
recording devices. While this was initially attributed to transient network conditions or
other anomalies, the session was repeated under similar conditions and yielded compa­
rable results. Specifically, the estimated 5­minute drift in both LabRecorder and Sensors
applications was found to be on the order of several seconds. These findings suggest that
the synchronization performance is likely sufficient for applications where sub­second ac­
curacy is acceptable. However, for use cases demanding synchronization precision on
the order of milliseconds, it is advisable to evaluate the specific device setup beforehand.
The results presented in the previous chapter do not conclusively replicate the findings
reported in [15]. This discrepancy is most likely attributable to the limited number of partic­
ipants in the present study, which constrains the generalizability of the findings. However,
it is important to note that the primary objective of the experiment was not to reproduce
the exact results of Bizzego et al., but rather to demonstrate the practical applicability of

the lsl_flutter package as a cross­platform solution for integrating the Lab Streaming

In the study by Golland et al. [21], which Bizzego et al. sought to extend, the authors
observed that “when participants entered the lab (equipped with multiple recording de­

Although the outcome of the physiological data analysis was inconclusive, the experiment

7.2
Summary
The results presented demonstrate that the research goals outlined in section 1.4 were
successfully fulfilled. The LSL framework and related software were investigated through
both literature, focusing on typical use cases, and source code analysis, which informed
integration strategies. This provided the foundation for the design and implementation of
a suitable architecture and API, structured as a three­layered public interface that offers
programmers flexibility in the level of control required over the native library.
A reactive approach to collecting samples from inlets was adopted, distinguishing this
work from the existing experimental Flutter package liblsl, which does not currently
support such functionality. While liblsl is well­structured and shows promise, a key
architectural difference lies in its use of the experimental native_toolchain_c package1,
which automates native C compiler invocation on the host system.
The system developed in this thesis abstracts away nearly all platform­specific integration
steps for the programmer. Native library linking is fully managed by the lsl_bindings
plugin, and multicast lock acquisition is handled internally by lsl_flutter via the
carp_multicast_lock plugin.
The evaluations collectively demonstrated the fulfillment of requirements R1–R5 through
successful operation across all supported platforms. Additionally, the technical validation
indicated that requirement R6 was satisfied in most scenarios. Thanks to the system’s
modular design, the full functionality of LSL is exposed to programmers in lsl_bindings,
and extending this functionality to lsl_flutter involves merely implementing a corre­
sponding high­level abstraction, thereby fulfilling requirement R7.
The three packages, carp_multicast_lock (https://pub.dev/packages/carp_multicast_
lock), lsl_bindings (https://pub.dev/packages/lsl_bindings) and lsl_flutter (https://
pub.dev/packages/lsl_flutter) have been published to pub.dev, ensuring public accessibility
and potential for further development.

7.3.1
Isolates and Performance
In the current design, a single isolate is created to handle all active inlets on a device,
and a separate isolate is responsible for managing all active outlets. While this approach
has proven sufficient for the present use cases, it may not scale optimally in scenarios
involving a large number of concurrent streams or high­frequency data sources.
An alternative design would be to allocate a dedicated isolate for each inlet and outlet. This
could potentially improve parallelism and better distribute workload across CPU cores,
thereby enhancing performance and responsiveness in more demanding recording se­
tups.
7.3.2
Extensible Data Format (XDF)
During the recording sessions for the experiment described in this thesis, a notable differ­
ence was observed in the storage footprint of the generated files: the XDF files produced
by LabRecorder were approximately half the size of the corresponding CSV files gener­
ated by the Sensors application. While the CSV format offers simplicity and broad com­
patibility, its limited compression efficiency may become a drawback in longer or higher­
frequency recordings where storage constraints are more critical.
Integrating support for writing data directly to the XDF file format within the Sensors ap­
plication or the underlying package would offer more efficient storage while preserving
LSL­compatible metadata structures. However, the current lack of a stable C interface
for XDF presents a technical challenge for such an integration, particularly when target­
ing a cross­platform FFI­based architecture. Future work could explore contributing to the
development of a suitable native XDF interface.

8
Conclusion
The objective of this thesis was to investigate how the core LSL library can be imple­
mented as a cross­platform package in Flutter, with the goal of simplifying the integration
of synchronized time series data acquisition into mobile and desktop applications. The
research questions posed at the outset have been addressed through the design, imple­
mentation, and evaluation of a system consisting of two complementary Flutter packages:
lsl_bindings and lsl_flutter.
The architecture and public API were designed as a layered system, separating low­level
and high­level functionality. The lsl_bindings package exposes the native LSL func­
tionality for advanced use cases, while lsl_flutter provides a higher­level API that in­
tegrates naturally with Flutter’s reactive programming model. This design enables both
fine­grained control and ease of use, supporting a wide range of programmer needs.
To reduce platform­specific integration effort, the system includes built­in mechanisms that
handle the configuration and linkage of the native LSL library in a way that is transparent to
the programmer. As a result, programmers can build cross­platform applications without
needing to manually manage native dependencies or platform­specific setup.
The system was evaluated both functionally and technically. A technical evaluation con­
firmed correct operation across all targeted platforms, including core capabilities such as
data streaming. Furthermore, it showed that the synchronization performance is compara­
ble to existing native LSL implementations, achieving sub­second drift in most scenarios.
An applied experiment further demonstrated the system’s practical viability in a real­world
research context.
In summary, this work contributes a cross­platform Flutter­based system for LSL integra­
tion, providing programmers with a reusable and extensible solution for synchronized data
acquisition across mobile and desktop platforms.

Introduction
The purpose of this experiment is to validate the use of a Flutter­based LSL package
for conducting cross­platform physiological data acquisition. In this study, data will be
collected using Android, iOS, and macOS devices. Please begin by reading and signing
the provided consent form.
During the experiment, you will be fitted with sensors that record PPG data along with
timestamps, while you are shown a series of videos designed to elicit specific emotional
responses. You will watch five videos in total, with brief pauses between each. During

D
Practical Solutions
D.1
Audio Service
A significant challenge in utilizing LSL on mobile devices is that the streaming objects
must reside in active memory, they cannot simply be persisted to local storage or an ex­
ternal database. As a result, when a mobile application is backgrounded or suspended,
there is no guarantee that object references, including those tied to active LSL streams,
will persist. On Android, this issue can be mitigated using foreground services, which
are designed to support continuous background tasks. However, achieving similar func­
tionality on iOS is less straightforward. While iOS does permit background processing,
it does so only at periodic intervals, making it unsuitable for continuous tasks such as
pushing samples to an LSL stream. To address this limitation, the Sensor app described
in subsection 5.1.1 employs a creative workaround: it configures the app to behave as a
music player by leveraging the audio_service and just_audio Flutter plugins. Specifi­
cally, when an LSL outlet is created, a one­second audio track is played in a continuous
loop, effectively keeping the app active in the background and maintaining the necessary
LSL object references.
D.2
Muse plugin
Muse offers a proprietary software development kit (SDK) that enables programmers to
interface with their devices. To integrate Muse data into the Flutter­based application, a
preliminary plugin was developed that leverages the artifacts provided by the Muse SDK.
Given that the LSL Flutter package is intended to be open source and publicly accessible,
particular care was taken to ensure that no proprietary software from Muse is exposed
in the repository. All Muse­provided files are excluded from version control and must be
manually supplied by a programmer with authorized access. Moreover, due to its reliance
on proprietary components, the package will not be published on pub.dev.
D.3
App distribution
To distribute release versions of the mobile Sensors and Scripted apps, Firebase App
Distribution was utilized. For Android builds, this simply requires an .apk or .aab file,
while for iOS distribution .ipa files must be built with appropriate provisioning profiles.
For this project, the iOS applications were only run on two iPhone devices, whose device
id’s were attached to the provisioning profile used to build the .ipa files. In hindsight this
distribution may have been excessive due to the limited number of devices.
