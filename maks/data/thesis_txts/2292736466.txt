Summary (English)
As statistical modelling techniques, essential for understanding our world of
data, are spreading into all corners of science and business and the costs of
power and data storage drops, the amounts of data that has to be modelled to
be useful, will increase. This makes systems that allow the rapid and eﬃcient
implementation for diﬀerent models essential. Thus:
Keeping the time between
the mathematical idea of the model and the useful implementation as short as
possible.
We used general implementation and automatic optimization to implement a
Metropolis Hastings sampler that works for any model and automatically opti-
mizes the evaluation of the likelihood at compile time.
The sampler works and the optimizations provide better speedups on runtime
than anything the compiler can do.
The use of domain speciﬁc compiletime optimizations is useful when you are
implementing lots of programs in the same domain because you are able to
implement your understanding of diﬀerent optimizations into code and have
them automatically applied if useful. This results in faster development which
allows you spend your time on more important things, such as evaluation of
your models or interacting with customers.

Chapter 1
Introduction
Statistical Modelling and Machine Learning are becoming more and more preva-
lent as tools for gaining understanding from data. And this while dealing with
the inherent uncertainties of measurement and the errors introduced by sim-
plifying the way the world works with assumptions. As these techniques for
understanding the world are spreading into all corners of science and business
and as the costs of power and data storage drops, the amounts of data that has
to be modelled in some way in order to be useful, will only increase. This will
both increase the demand for eﬃcient inference and provide support for ever
more complicated models. This makes it important to have systems that allow
the rapid implementation for diﬀerent models while still keeping the resulting
inference code eﬃcient enough to be useful.
Thus:
Keeping the time between the mathematical idea of the model and the useful
implementation as short as possible.

1.1
Traditional imperative libraries
The traditional way of implementing tools for statistical inference is in the form
of software libraries. The usual pattern for libraries is the subroutine model,
where each possible task done by the library is done by a separate function
and the only way to combine features of the library is to run the subroutines
sequentially after each other.
This severely limits the ﬂexibility of use, and
makes them hard to extend beyond the speciﬁc cases coded by the creator. This
limits their usefulness to situations where the user needs exactly what the library
provides with no modiﬁcations. The user might add pre- or post-processing to
mitigate this, but this is ultimately a workaround.

1.2
Modeling Languages
A great improvement in ﬂexibility from the library model is the modelling lan-
guages like BUGS[LTBS00] and STAN[Sta14]. They provide very powerful and
ﬂexible modeling tools that use a single or a limited number of inference en-
gines. By making a separate language they gain signiﬁcant ﬂexibility in syntax,
but the upshot of this is that the semantics ie. usable elements of the language,
all have to be deﬁned by the speciﬁc language. This makes it diﬃcult to use
external libraries and makes them hard to extend and it requires users to learn
and use a special language just for this. See ﬁgure 1.1 for an example.

graphical models such as recursive or iterative structures. These simulations
can make use of all the features of traditional general purpose languages such
as: recursion or conditional branching. The main structure of such programs

Introduction
1.4
In this thesis
We implement a modelling framework, leveraging the features of an existing
language and thereby reusing all of it syntax and semantics. The inference is
done with a Metropolis Hastings Sampler implemented in a general way, so
that it is completely independent of the model used. We apply optimizations
to the model using Template Haskell for program transformation. Haskell is
used because of its advanced functional features, such as lazy evaluation and
its syntax that is well suited for manipulating abstract tree structured types,
such as the internal structure of a program, which makes it useful for applying
program transformations as with Template Haskell.
In section 2 we introduce the fundamental objectives of bayesian modelling
and the idea of sampling for inference. We also introduce a Gaussian Mixture
Model as a running example.
In section 3 we explain Markov Chain Monte
Carlo(MCMC) sampling in detail and consider two commonly used algorithms
for realizing MCMC in practice, Metropolis Hastings and its special case Gibbs
sampling.
In section 4 we consider two categories of optimizations applicable to typical
Bayesian modelling with some examples. These categories are Structural opti-
mizations and Sparse updates. We describe these two classes of possible opti-
mizations, but the actual application of optimizations in code is saved until we
have explained the infrastructure for doing so in 5.
In section 5 we cover the features of Haskell used for expressing models in
a way that facilitates very ﬂexible model description and automatic applica-
tion of the optimizations mentioned in section 4. In section 6 we show general
implementations of the algorithms described in 2.
In section 8 we measure the gains from applying the optimizations and validate
that the Metropolis Hastings sampler samples from the right distribution.The
result is proof of concept code, that shows the feasibility of having inference
algorithms implemented independently of the model and have domain speciﬁc
optimizations be applied automatically improving the performance of the in-
ference code signiﬁcantly better than what the general purpose compiler can
manage.
If these concepts were incorporated into the tools used by model developers it
would increase their productivity and increase their ﬂexibility by letting them
focus on developing and testing the consequences of diﬀerent models and their
assumptions and spend less time implementing inference procedures and opti-
mizing them for each model.

Models and Inference
2.1
Models
In statistical modeling the goal of learning is to estimate the value of some latent
variable. This is done using a model of how the world works. e.g. The fruit
in the store has some degree of ripeness. You want to select the fruit that is
the most ripe but, the problem is, you can’t really know how ripe the fruit is
without tasting it, which would usually require you to buy it. To solve this you
use an internal model of the relation of the ripeness of the fruit to something
you can observe, like its color. This model could be that the more ripe a fruit
is the more red it is. This rule of thumb could be more or less correct, but in
any case it would embody your assumptions about how the color of fruit works
By using this model you can estimate the ripeness of the fruits based on their
color and select the one most likely to be the most ripe.
2.2
Inference
In Bayesian statistics we can work with this kind of inference while taking into
account the uncertainties that are introduced by noise in the data as well as

2.2.1
Deterministic Approximations
Two other Deterministic methods are Laplace Approximation and Variational
Bayes, where the former approximates the target distribution with a normal
distribution centered at the mode of the target distribution and the latter min-
imizes the KL-divergence between the target distribution and a parametric one
that is tractable.

If the latent variable space is discrete and small enough the integral becomes
a sum that can be evaluated in a reasonable time-frame. But if the variable
space is large ie. high dimensional, then there are no other options than to use
stochastic methods.
2.2.2
Stochastic Approximations
There are three main categories of sampling methods for inference and they are:
• Importance sampling
• Approximate Bayesian Computation
• Markov Chain Monte Carlo sampling
They all approximate the target distribution in some way by drawing samples
that are used as a proxy for the target distribution itself.
Importance sampling is done by sampling from another distribution that is
similar to the target and has support over the same domain and then weigh-
ing the samples to correct for the mismatch between the sampling and target
distributions.
Approximate Bayesian Computation, draws values for the latent variables
from the prior and generates data from those latent variables. The latent vari-
able values are accepted if the data they generated was "close" to the observed
data according to some criteria1.
In the rest of the thesis we will consider the Markov Chain Monte Carlo(MCMC)
sampling methods that sample from the target distribution by constructing a
Markov chain who’s stationary distribution is equal to the target distribution.
Samples are then generated by iteratively generating this markov chain. We
will discribe this method in more detail in section 3
In our example this would correspond to picking random values of ripeness from
not ripe to very ripe in a sequence in such a way that the values are distributed
according to the target distribution p(Θ|X).

Chapter 3
Markov Chain Monte Carlo
As discussed before, the computational challenge of inference of latent variables
in Bayesian statistics reduces to approximating an integral over all possible val-
ues of the latent variables. In MCMC sampling the integral is approximated
by a histogram of samples drawn from the distribution over possible values of
the latent variables. This is done because, even though the integral is compu-
tationally intractable, drawing samples from the distribution is still possible by
constructing a Markov chain with a stationary distribution equal to the target
distribution over possible values of the latent variables.
A Markov chain is a sequence of values x1, x2, ...xN that are generated by a
Markov process which has the property that the state xn+1 depends only on
the state before it in the chain xn. The Markov chain is completely speciﬁed by
its transition probabilities. The transition probability p(x →x′) determines the
probability of going from one state x to another x′. If the Markov chain has a
stationary distribution π(x), it is the distribution that the distribution of states
approach as the length of the chain goes to inﬁnity. If the transition probability
fulﬁlls the Detailed Balance criterion [Bis06](p. 540), then the Markov chain
has a stationary distribution.
π(x)p(x →x′) = π(x′)p(x′ →x)
(3.1)
==PAGE:24==
12
Markov Chain Monte Carlo
That is for every pair of states x, x′, the probability of drawing x from the sta-
tionary distribution and transitioning to x′ has the same probability as drawing
x′ and transitioning to x ie. the chain is reversible and the sequence is just as
likely to be sampled in one direction or the other.
For the stationary distribution to be unique, the chain must also be ergodic,
which means the expected time for the chain to return to any state is ﬁnite.
Alternatively formulated the chain must be able to go between any two states
in a ﬁnite number of steps.
Because the states are generated in a chain they are highly correlated and this
skews the distribution. This is mediated by doing thinning by a number T where
only every T’th state in the chain is used as an output sample. The chain also
has a heavy dependence on the starting state. This is ﬁxed by discarding the
ﬁrst B number of samples in a technique called Burn-in.

Optimizations
There are some optimization tricks that are especially useful to apply when
doing Metropolis Hastings sampling, especially when evaluating the likelihood
ratio. We are going to describe two categories of tricks that are useful:
1. Structural tricks that exploits rearranging the order of evaluation and
identifying terms that cancel out or other tricks that exploit the structure
of the expressions without worrying about the individual sub-expressions.
2. Low rank update tricks that take advantage of the fact that we are eval-
uating functions on very similar inputs thereby allowing us to cache the
outputs and update them to ﬁt the new inputs.
These tricks use properties of the problem that is being solved and therefore
they will usually trump any enhancements to performance such as parallelism
or fancy implementations because the algorithm deﬁnes the actual scalability
and complexity of the algorithm itself.
The tricks are automatically applied to likelihood functions in section 5.3 and
in section 8 we measure the impact of the tricks and compare them to the
optimizations done by the compiler.

4.2.1
Summary Statistics and Monoids
In this section a general update function is described for all functions f : A →B
that have certain properties:
• The domain A consist of sets of elements like numbers, vectors, matrices
or combinations hereof.
• The codomain(output) B has a cancellative Monoid structure.
• The function preserves the monoid structure in case of disjoint sets.
A Monoid is an algebraic structure that consists of a set S and an operator (⊕)
that satisﬁes the following two axioms.
• Associativity For all a, b and c in S, the equation (a⊕b)⊕c = a⊕(b⊕c)
holds.
• Identity element There exists an element e in S such that for every
element a in S, the equations e ⊕a = a ⊕e = a hold.
As an example, the set of all integers form a monoid when combined with the
+-operator. In this case the number 0 would be the identity element e.
If the monoid has the cancellation property, that is a + b = a + c ⇒b = c,
then is possible to introduce another operator to the monoid that cancels out
the ﬁrst ⊕it could be labeled ⊖and have the following properties.
(a ⊕b) ⊖b = a
(a ⊕b) ⊖a = b
This way an operator ⊖of limited scope is introduced. It is limited in the way
that it can only cancel out elements already added.
The way the function f can fulﬁll the third criteria is by making the following
true for disjoint sets a and b.

Program Transformation
In this section we will introduce the concept of program transformation and
present an overview of how this can be done in Haskell where an extension
called "Template Haskell" allows us to transform Haskell code using Haskell
itself.
Program transformation is the practice of taking code in some language, parsing
it into a representation that can be manipulated and then producing some other
code that provides the same or another functionality. In our case it is optimiza-
tions, where the code is changed to run faster but it could also be correctness
analysis or automatic diﬀerentiation of a cost function.
Program transformation is done programmatically and at compile time so it
takes little development time and as it is done at compile time it does not
introduce run time overhead to the code. This means that any programmer’s
task that can be put into an automatic program transformation can save lots of
time and work. It is something that most compilers do in order to implement
optimizations or advanced syntax and it is sometimes called the preprocessor.
In order to understand the transformation of Haskell code we ﬁrst cover its basic
structure.

Testing
For testing and debugging of the functions, the QuickCheck test framework was
used. It provides the ability to state propositions that are checked when the
command test is run. The propositions are functions that take some inputs
and return a Boolean of whether they hold.
This could be the proposition
that timesTwoIsEven n = even (2*n) where type = n::Int.
When given
an integer n, 2*n this should always be even as tested by the function even.
The QuickCheck framework generates a number of random inputs for the propo-
sitions and if the propositions return False the used random input is presented
to the user.
The random inputs are generated from an instance of the Arbitrary typeclass,
so any type in this class can be generated at random.
If the functions you
need to test have types not in the Arbitrary class the framework provides a
combinator library for combining already implemented types. If your values can
be built from a list of Doubles then the random generating function could be
as simple as: arbitrary = fromList arbitrary, where arbitrary generates
a list of Doubles and fromList uses this list to construct the desired value.

Validation
To test whether the sampler does its job correctly, it is necessary to compare the
achieved frequencies of the values sampled with their probability of occurring
in the target posterior distribution.
If, in case of a discrete latent variable,
there is not an analytical solution of the integral in Bayes theorem(2.1) for this
particular model, then the posterior can be computed for a small dimensionality
by turning the integral into a sum and summing over all possible values of the
latent variable.
In the example a partition of datapoints into k clusters is the only parameter
inferred so this deﬁnes the parameter-space. The number of possible k partitions

Chapter 9
Conclusion
The development of individual implementations of models and their inference is
often redundant since the same algorithm is implemented many times slightly
diﬀerently to optimize for the diﬀerent models. We show that these optimiza-
tions can be done automatically at compile time using Template Haskell. The
optimizations that we use as examples here are not implemented in compilers
since they are too domain speciﬁc to be useful for everybody using a language,
but by using program transformation we eﬀectively extend the compiler with
our own speciﬁc optimizations that are very powerful in a limited domain.
With our proof of concept implementation of automatically applying two dif-
ferent optimizations we have shown speedups of 270% and 230% depending on
which optimization was applicable. These speedups are compared to the code
compiled without the optimizations and outperform the 6% speedup from ag-
gressive optimization by the standard compiler.
All of these results were done with a Metropolis Hastings sampler implemented
completely independently from the model and the rest of the code.
The results are not compared to other code outside the project and might be
slower than those.
The goal of the project was to show the eﬀectiveness of
the metaprogramming techniques in applying domain speciﬁc optimizations,
compared to the compilers standard optimization.
