Summary (English)
The goal of this thesis is to outline how obfuscation can improve whitebox cryp-
tography in a practical way. In order to do so, a brief introduction to AES and
whitebox algorithms is provided. Dierent obfuscation methods are described
including their properties and tools in which they have been implemented. A
few whitebox attacks are described with focus on their practicality. Based on
this, an experimental setup for running a combined DCA attack on the AES
based whitebox algorithm presented in the CHES challenge 2016 is developed.
The combined DCA attack is able to correctly guess the full key only using 130
traces when no obfuscation is used. Experiment results are presented showing
how some obfuscation techniques increase the attack complexity while others do
not. While the static obfuscation techniques applied do not increase the attack
complexity, a dynamic obfuscation technique developed for this thesis is shown
to improve the complexity with a factor 8. Combining this dynamic obfuscation
with program enlarging bogus code, including a number of interleaved encryp-
tions in the compiled program with all but one having a fake key, we are able to
obfuscate the program such that the full key is not recovered for 25,000 traces.
The method is practical and scalable which means the attack time complexity
can be improved such that it is infeasible to use DCA to retrieve the full key.

Chapter 1
Introduction
Cryptography is about keeping secrets between parties, even though the channel
by which they communicate is insecure. In the physical days this was done by
putting the secrets inside a locked box and only distributing the key to the
parties communicating. In the digital world, the same approach is used: One
encrypts the secret, also called the plaintext, using an algorithm and a key,
resulting in a ciphertext that no one can read except other parties knowing
both the algorithm and the key. As with other protection mechanisms, there
are ways of trying to break the protection.
When looking at attacks in cryptography, we normally consider the black box
attacker. The black box attacker is an attacker who wants to know the plaintext
corresponding to a given ciphertext, or obtain the secret key for the encryption.
This attacker has access to an encryption oracle, which means he can obtain the
encryption and decryption of any text, except the given ciphertext. One can say
he have a magic box where he can put in plaintexts and get the corresponding
ciphertext, but he cannot see what is inside, which is where the name blackbox
attacker is comes from.
If the blackbox attacker model is used, the cryptographic algorithm should be
able to withstand a chosen-plaintext attack.
This is an attack were the ad-
versary can choose any number of plaintexts of his choice and observe their
corresponding ciphertexts. The attack can also be adaptive, such that the ad-
==PAGE:14==

versary chooses the next plaintext on basis of the previously observed ciphertext.
Since a ciphertext should not reveal information regarding the equality of the
plaintexts encrypted, randomness is an important part of cryptography, ensur-
ing that the adversary can not predict what his opponent will send and use
this to recover the key. Randomness is also used to generate the secret key,
and a new key for every message is needed if to ensure that a compromised key
does not reveal secret future messages. Following the black box model a cryp-
tographic algorithm is normally implemented using the secret key as an input,
such that the algorithm can compute plaintext/ciphertext pairs using a secret
key dynamically.
Since computers are unwillingly sharing information of what they run, comput-
ers can not be treated as being entirely opaque. Computers share environmental
information in side channels such as power consumption, heat, or electromag-
netic noise. The rst attacks to exploit these side channels were performed in
the late nineties and lead to a new attacker model, which on top of the black
box attacker's capabilities also had access to these side channels [KJJ99], re-
vealing additional information. Thus the box was no longer all black and the
word graybox attacker was seen t to name the new attacker model.
In 2002 Chow et al introduced a new attacker model denoted the whitebox
model [CEJvO03]. The whitebox attacker was dened as a mighty opponent
which in addition to attack vectors of the gray box attacker also has full control
over the executing environment which include capabilities to
• access all memory
• dynamically execute the program
• insert changes to the variables or algorithm during execution
This means that randomness is dicult to obtain because the source that seeds
any random number generator is under the control of the attacker [CEJvO03].
In the whitebox model it no longer makes sense that the goal of the adversary
is to guess a plaintext given a ciphertext, since the adversary has full access to
the algorithm and its execution environment. Instead the goal for the attacker
is only to obtain the secret key. Even though some claim it possible [Arx16b],
it is still to be shown how one can have a dynamic key using a whitebox secure
algorithm. Suggestions on how to make whitebox cryptography with a static key
has been made where the secret key is embedded in the algorithm; whitebox
cryptography is obligated hide the key within the implementation such that
having the binary le and running encryptions or decryptions does not reveal
the secret key [CEJvO03].
==PAGE:15==

Security through obscurity has been criticized by cryptologists since Kerckho
in 1883 published his doctrine [Ker83] stating that sooner or later an adversary
will gure out a cryptographic algorithm, so a cryptographic algorithm should
not rely on obscurity of the algorithm but on a secret key that is changeable.
Kerckho's principle ts the blackbox model well, being directly translatable
into the scenario of a physical box with a key. But it is dicult to apply to the
whitebox model where a binary le for a whitebox algorithm with an embedded
key must hide the key properly within the le. In such cases, obfuscation of the
binary le is essential, and while the methods used for hiding the key should
still be known, thereby following Kerckho's principle, the obscurity of the
binary le and the cryptographic algorithm is an important part of whitebox
cryptography. In short, whitebox algorithms concerns hiding the a secret key
within an algorithm such that it would take an infeasible amount of time to
obtain it.
Obfuscation has the purpose of hiding the meaning of a program such that
looking at the program code one cannot decide what the program is doing. Often
when obfuscation is used, some parts of the program are more sensitive than
others, and the goal of the obfuscation is to hide these sensitive parts. When
obfuscation is used for whitebox cryptography, the key is the part that needs
to be hidden amongst the other parts of the program code. Some have already
looked into the subject of using cryptographicly safe obfuscation, but so far there
is no solution to the problem. It is shown by Barak, Goldreich, Impagliazzo,
Rudich, Sahai, Vadhan, and Yang [BGI+12] that it is not possible to obtain
perfect obfuscation for general programs i.e. making an obfuscation algorithm
that can obfuscate all programs. They show that if such algorithm existed there
must be a deobfuscating algorithm that can reverse the obfuscation. But for a
subgroup of programs, one can still obfuscate such that it is indistinguishable
whether one or another key is used. This is addressed in the paper by Garg,
Gentry, Halevi, Raykova, Sahai, and Waters [GGH+13]. However, the degree
of which general obfuscation methods enhances the time complexity of general
whitebox attacks is not academically investigated.
When using cryptography today the NIST standard [19701] using their Ad-
vanced Encryption Standard, AES, is often used for symmetric encryption.
This algorithm is however only black-box secure so a lot of eort has been
put into making a mode of operation that makes AES whitebox secure. Along
with the denition of the whitebox attacker [CEJvO03], Chow, Eisen, Johnson
and van Oorschot proposed an algorithm for protecting AES against the white-
box attacker. Others followed such as Bringer, Chabanne and Dottax in 2006
[BCD06], Xiao and Lai in 2009 [XL09], and Karroumi in 2010 [Kar10], but all
were breakable using the static whitebox attack, BGE, made by Billet, Gilbert
and Ech-Chatbi [BGEc05], or other whitebox attacks [Mul14]. The static at-
tacks mentioned are slow, needing full access to all lookup tables used by the

implementation, which can be expensive to obtain. This is not the case for dy-
namic whitebox attacks which in contrast to static attacks operates at runtime
meaning that no precomputation is necessary when applying the attack.
This report is an practical investigation into how obfuscation methods can en-
hance the security of whitebox implementations.
1.1
Scope
There are various attacks available in the whitebox model, since the adversary
has full control over the executing environment and all attacks that apply to
the black- and graybox attacker also applies in the whitebox attacker model.
This report focusses on the attacks specic to the whitebox attacker model i.e.
attacks that makes use of the access to the execution environment.
Some whitebox algorithms are based on new principles, but since AES is a valid
cryptographic standard, we focus only on making AES whitebox secure. Of the
whitebox algorithms mentioned in last section, two out of three algorithms were
based on the algorithm of Chow et al. As a means of limiting the scope of the
project, this report therefore consider only the AES based whitebox algorithm
of Chow et al. and more specically how whitebox attacks apply to it.
When cryptographic algorithms are broken, the time needed to attack them is
essential. This makes the dynamic attacks specially important, since no pre-
computation is necessary. Today, generic tools for breaking the algorithm of
Chow et al. in an hour on a laptop using dynamic whitebox attacks are avail-
able [BHMT16].
This report describes the BGE and two dierent dynamic
attacks: Dierential computational analysis (DCA), and dierential fault anal-
ysis (DFA), but we limit the scope to only investigating how obfuscation of
the AES algorithm of Chow et al. protects it against a generic tool applying
the DCA attack [BHMT16]. This allows us to investigate how to prevent the
relatively easy attacks on whitebox implementations.
It is important to note that many of the experiments run throughout the report
are highly time consuming. The results are therefore naturally limited in scope
to what is feasible to run on the available hardware.

Chapter 2
AES
Advanced Encryption Standard is a widely used algorithm for symmetric en-
cryption. It is a standard that has existed since 2001 and is still unbroken when
considering the blackbox attacker model. AES is well known and used, and a
whitebox secure mode of AES would be easier to trust than a new algorithm,
since if an attack on a whitebox AES implementation is found one would have
the fall-back that at least the algorithm is still blackbox secure, which is better
than no guarantees at all.
In order to understand how AES can become whitebox secure, it is important
to understand how AES works. This chapter gives an introduction to AES and
some available implementation possibilities.
2.1
Structure
AES is a block cipher where four small ane and non-ane functions are ap-
plied, the process of which is denoted a round. These rounds are repeated a
xed number of times before the ciphertext is outputted. A single round is not
secure, but security is obtained through repetition. There are 3 versions of AES:
AES-128, AES-192, and AES-256, where the number indicates the bit-length of
the secret key.

• mixColumns mixes the columns of the state by multiplying a matrix with
it.

key that is made for the round with the state.

3.1
WB-AES of CHES Challenge 2016
In the CHES Challenge 2016 (CHES16) a whitebox challenge to attack an imple-
mentation of the algorithm of Chow et al. was proposed. The implementation
is is made of table lookups of AES subrounds with an embedded key encrypted
by a bijection function. The algorithm uses only 42 variables in which half bytes
are stored. The algorithm only operates on half bytes which means that L −II
and L −III are split into 8 tables, one for each outputted half byte. In order
to only use 42 variables, one subround is fully computed before the next begins.
Inside the subround the rst half byte to L −III0 is computed rst. Then
the second half byte input to L −III0 is computed, and so forth. When all 8
half bytes are computed, marked with a red "1" in gure 3.3, the input for the
next round, marked with a red "2" in gure 3.3, is calculated. The variables
are reused in such a way that two subrounds cannot be run in parallel, and
only the result of the full subround is independent with the other subrounds i.e.
the same variables are used for storing the intermediate states for the dierent
subrounds.
3.2
Randomness in whitebox cryptography
Randomness is often critical in cryptography, and if the randomness guessed, the
result is often the secret key being obtained by an attacker. True randomness
is dicult to obtain and therefore random number generators are used which
are seeded with something approximately random input like time or some sort
of noise. In the whitebox attacker model the adversary has full environmental
control. This means that he can control what random source is used, or mea-
sure the noise himself and thereby obtaining the random numbers used. This
is the reason one should be extra careful using randomness in a whitebox at-
tacker model and ensure that the randomness used by the algorithms are not
exploitable for the adversary.

4.1.3
Other obfuscation techniques
There are some techniques that do not fall into the two above categories. We
mention some of these here.
Code diversication is concerned with using randomness in some of the above
mentioned obfuscation techniques such that two compilations of the same
problem will not result in the same binary le [JRWM15]. This is very
powerful with function diverted copies obfuscation, since a random version
of every function will be chosen when a function is called. An example
from Tigress [Col16] on code diversication is the Jit operation described
in section 4.3.5.
VM obfuscation is using virtual machines (VM). The program in need of
obfuscation is compiled to run on a VM and the VM is compiled to run
on the users system, such that the output binary runs the VM that runs
the program. In this way both ow and instructions will be obfuscated
as the entire execution is done in a dierent environment [FWWH11]. In
order to obfuscate further, one can run several layers of VMs on top of
each other. Another way to strengthen this obfuscation is to choose an
unknown VM, such that the environment is dicult for the attacker to
understand.
Code diversication breaks known patterns which could make it dicult identi-
fying the structure of a whitebox implementation. The VM obfuscation can be
seen as a transformation of the program, and it is dicult isolating the actual
program from the program running the VM, which again complicates identifying
the whitebox structure.

Whitebox analysis
Whitebox analysis is concerned with analysing whitebox implementation with
the goal of obtaining secret keys embedded in such. As with obfuscation tech-
niques, whitebox analysis can be grouped into static and dynamic analysis.
In static analysis, discussed in section 5.1, one needs access to all the lookup
tables used in the algorithm. This can be obtained by running the program until
every table lookup has been accessed, mapping the table entries and thereby
reconstructing the tables. But with more that 219 entries as in the algorithm
by Chow et al., the complexity for the encryptions during the precomputation
of the attack is rather large.
But when the precomputations are made, the
computational eort is static, no encryptions are needed, and the correct key is
veriably obtained.
In dynamic analysis, discussed in section 5.2, the attack is run dynamically while
executing the program, meaning no precomputation necessary. But instead of
getting an exact result, a probability for a given key is given. Normally, the more
encryptions/traces used in the analysis the larger probability of correctness is
achieved.
In this chapter we discuss these two types of analysis along with specic methods
of doing such analysis.

5.1
Static analysis
Two years after the algorithm of Chow et al. was published, Billet, Gilbert
and Ech-Chatbi proposed an algebraic attack (BGE attack) applicable to it
[BGEc05]. This attack is very powerful and is proven to retrieve the key, without
using any encryptions, when the precomputed tables are available. The problem
is that since the key is embedded in the tables, ideally there is a new set of
tables for every implementation and a new implementation on every device.
This means that the precomputation cannot be reused unless the same key is
used multiple times.
The BGE attack exploits the structure of AES. It rst identies one round. Then
it removes the ane part of the bijective transformation. It then identies an
inverse transformation to the non-ane part. That leaves an ane part, which
afterwards is retrieved.
This reveals the rest of the encryption in the table
lookup Pi xor'ed with the key byte ki, where i is one of the 16 key bytes. This
can be used to retrieve the key byte by removing the last ane part in another
way, revealing only Pi. When all the key bytes ki|i ∈0..15 are found, we need
to nd the correct order of the bytes, and then we have retrieved one round key,
which easily can lead us to the main key by the AES key schedule. We refer
the curious reader to a more thorough explanation of the attack in the original
paper [BGEc05].
The BGE attack exploits that AES is a commonly known algorithm, and that
all the ane and non-ane structures are known. It is further exploited that
the bijective mappings for every round are reversed for the next round [Mul14].
The complexity of the attack is worst case 230 according to [BGEc05] given that
all tables are retrieved. But in [Mul14] some improvements are made and the
complexity is reduced to 222 (223 if external encoding is used).
5.2
Dynamic analysis
The problem with static attacks is that the tables can be dicult and expensive
to retrieve.
This is omitted in the dynamic analysis, but instead one get a
probability of correctness dependent on the number of encryptions performed.
In the following we describe two types of dynamic analysis that have proven
applicable to whitebox cryptography.

5.2.1
Dierential computation analysis
Dierential computation analysis (DCA) exploits the correlation between the
dierences in data at certain times during program execution. The attack has
much resemblance with the graybox attack dierential power analysis (DPA),
but instead of looking at power consumption, the attacker analyses memory ac-
cesses at certain times, either reads from the stack, memory addresses accessed,
or bytes read from memory [BHMT16].
A stepwise walk through the attack follows:
1. Examine the program and identify the algorithm.
2. Identify the vulnerable part of the algorithm, e.g. in AES it could be after
the sbox of the rst round.
3. Record traces that includes the vulnerable part.
4. Choose a tting selection function.
5. Simulate the selection function for the plaintexts (or ciphertexts) to the
traces, and group them accordingly.
6. Take the dierence in the traces of the two groups.
7. Save the correlation.
If the correlation is high, the key assumption made in the selection function is
more likely to be true.
When choosing a selection function, one wants to choose a sub part of the
algortihm that only depends on a small part of the key and a known state. This
function should split the traces into two groups, where the dierence between
the two groups correlates at a certain point. An example with AES could be
using a selection function as the rst three functions of AES, watching the rst
bit. Step 5 will then simulate this state using a possible key byte and one byte of
plaintext before grouping the traces according to whether a 0 or a 1 is predicted.
Step 6 and 7 are performed afterwards. Step 5, 6, and 7 are repeated for all the
possible key bytes, and the highest correlation is the most likely to be true.
The theory of DCA is the same as for DPA. But since the traces are accurate
without any noise, we can attack on bit level instead of byte level of the key,
and we do not need time alignment, since the traces recorded are accurately
measured without noise. DCA builds on the statistic distribution of bits after

5.2.2
Dierential fault analysis
The idea of dierential fault analysis (DFA) is to inject byte faults into AES at
certain points and see how these faults propagate. A stepwise example of DFA
on AES follows:
1. Record the output of a run of the algorithm.
2. Then in another run inject a change in one byte just before the last
mixColumns (round 9) and then record the changes in the output.
3. Repeat step two until sucient data is collected.
4. Take the dierence between the correct and the faulty output. This reveals
the 4 byte dierences the fault has induced.
5. Solve the equation for the dierences. This gives a subset of possible values
for the 4 bytes of the round key.
6. Repeat from step 4 with other injected faults. Repeating enough times
reveals the true round key for 4 bytes.
7. Repeat step 2-6 when injecting fault in another byte. This reveals 4 other
key bytes, when repeated enough the full round key for round 9 is revealed.
8. Use the key schedule for AES to nd the full key
The trick is that even with the bijective transformations, the equations for the
dierences in the faulty and the correct output can be compute [SMdH] [DLV03].

Bos, Hubain, Michiels, Teuwen, and others have provided a tool, Deadpool,
that rst generates the traces, and then attack the traces using DCA or DFA 1
[BHMT16]. This section describes the details of how their DCA implementation
works.
When breaking AES with DCA, one starts by recording traces of the binary
encrypting with the algorithm. This is done using Valgrind or PIN. The traces
cover the reads to the stack, the data accessed from memory, or part of each
memory address accessed. Then they run AES for the initial round, looking at
one bit, b, of one byte of the plaintext, trying all dierent values for the key
bit, ki (i.e. 2 values), simulating this for the data and addresses. The result is
stored as ti = I(pi, k). In the traces, an intermediate state is found as the result
of a table lookup that is only inuenced by b, and is grouped according to the
simulation ti If the correlation between the two groups of traces is high, the key
guess is more likely to be true. The certainty rises along with the number of
traces.
One can choose dierent intermediate states, but a non-ane calculation is
needed such that dierent values leads to dierent amounts of computation used,
and only one key byte can inuence the output. The places in the algorithm of
Chow et al. where this happens is after and inside the sbox in the initial and
the nal rounds.
To clarify this an example is provided. Consider the CHES16 AES implemen-
tation. We would like to attack the memory data after the sbox in the initial
round, and we want to nd key byte 0 i.e. the key byte for the top left corner
in the block. We provide a set of n plaintexts p0, p1, ...pn = P, where we only
look a rst bit. We also need a set of all possible key bits k0, k1 = K. We trace
the memory accessed by the algorithm on these plaintexts, and afterwards nd
the intermediate state after the rst sbox i.e. the rst two table lookups, which
would be using the tables that only operates on the rst bit of the rst byte P0,0.
The dierent intermediate state values are denoted T = ti,j|i ∈0...n ∧j ∈0...m
where m is a number of intermediate states. We then simulate the data by

the mixColumns step. We now group the traces T into two groups according
to the simulation. By looking at the correlation between these groups, we nd
candidate for the key bit used, which is the one with the highest correlation.
The larger the n, the more certain we get that the correlation with the correct
key is highest. When the attack is done for the rst key bit, the next key bit is

6.4
Enlarging the program
Obfuscation is used for hiding the important parts of a program in less important
parts. CHES16 only contains an implementation of the algorithm of Chow et al.
and nothing more, so in order to illustrate how the algorithm would be included
in a larger program, the challenge is changed such that not only the algorithm is
present, but also some code not related to the algorithm. This is done by making
the program run the challenge two times, but the second time with tables with
random lookups.
This corresponds to running the challenge two times but
with two dierent keys. Only the output from the encryption with the correct
key is used as program output. We denote this bogus implementations, more
specically a single bogus implementation in this case. The dierent algorithm
implementations are executed after eachother.
As seen in table 6.6 this has some impact on the results, and not all keybytes
are found at 150 traces. All key bytes are guessed at 250 traces as can be seen
in table B.4 in the appendix, so the signicance is not that great.

The structure of the AES algorithm of Chow et al. used in CHES16 is well
known. This can be exploited in order to shue the subrounds of AES. The
variable reuse of wb_challenge (see section 3.1) allows every subround of the
AES round to be rearranged due to independency. This is exploited such that
the order in which the four subrounds of the rst round is calculated is chosen
at random.
As discussed in section 3.2 it is crucial how randomness is chosen in whitebox
secure algorithms, since the random source can be monitored. We have designed
the code for this obfuscation so the randomness for choosing subrounds is based
on the plaintext input to the algorithm. While this is indeed not random, and
can not only be monitored but also controlled, it is important for the DCA
attack to trace dierent plaintexts in order to work. So we are ensured that the
plaintexts will dier in order to make the attack possible.

one would make the order depend on the whole plaintext. CHES16 is made such
that all 4 subrounds are independent except for the rst round, where there are

v38, v39, v40, and v41 in a style similar the the rest of the rounds.

Chapter 7
Conclusion
In this report we have investigated whitebox cryptography, more specically
a whitebox implementation of AES, as well as several available obfuscation
methods and tools for applying them in order to test their eect when used
in combination with whitebox cryptographic algorithms. The availability and
quality of the considered tools varied, and obfuscator-llvm was deemed most
suitable for applying static obfuscation, while the code accompanying a yet to
be published paper was used as the basis for the dynamic obfuscation.
An experimental setup was established for running dierential computation
analysis, a dynamic whitebox attack, on the AES based whitebox algorithm
of Chow et al. implemented in the CHES16 whitebox challenge. The attack
works by rst running executing the encryption program a number of times,
recording dierent memory traces, based on which probabilities of key bytes are
computed. This setup allowed for running the attack against dierent versions
of the implementation with various code obfuscation techniques applied.
As a novel approach we combined dierent modes of the attack into a single
attack including both memory address and memory data traces as well as both
the sbox and multiplicative inverse selection functions. The CHES16 challenge
has been broken before in [HA16], but we show that we with the combined
DCA attack introduced one can recover the key with only 130 randomly chosen
traces taking less than 5 minutes on our computer, which to our knowledge the
==PAGE:78==

smallest number of traces yet documented.
We found that the static obfuscation was not eective against the combined
DCA attack. However, when using dynamic obfuscation a signicant improve-
ment was observed. A new dynamic obfuscation method exploiting the inde-
pendence of the AES subrounds was developed, which experiments showed to
improved the security of the whitebox algorithms such that 1050 traces were
needed in order to guess the full key, leading to a factor 8 in attack complexity.
Experimentation with enlarging the encryption program to consist of several
whitebox AES implementations with dierent keys, all but one being fake, was
done, showing an increase in the attack complexity. The complexity of the attack
and tracing time was 2.46 + 4.92x, where x is the number of implementations.
Finally, we showed how combining the dynamic subround obfuscation with the
technique for enlarging the program resulted in obfuscation of the whitebox
algorithm such that even at 25.000 traces the entire key could not be found
by the combined DCA attack. Estimates on this obfuscation method nds it
practically feasible to protect the whitebox algorithm such that the DCA attack
cannot recover all key bytes.
We believe to have contributed to the eld of research, not only by providing
a series of experiment results showing how whitebox cryptography and code
obfuscation can be used in combination, but also through the development of
a novel dynamic obfuscation algorithm shown to work well against the tested
attack. Even so, we believe much future work can be done and hope that others
will build upon the results we have presented.
