Abstract
This thesis presents the development and implementation of a typechecker for a sub­
set of the RAISE Specification Language (RSL). The primary objective is to ensure type
correctness in RSL specifications, thereby enhancing the reliability and maintainability of
formal specifications. The work involves defining the grammar for the subset of RSL, im­
plementing a lexer and parser using FsLexYacc, and designing a robust Abstract Syntax
Tree (AST) structure. The typechecker employs a two­phase approach: the first phase
collects and validates declarations, while the second phase focuses on type resolution
and expression checking. This systematic approach facilitates efficient and thorough val­
idation of RSL programs.
Extensive testing and evaluation were conducted using a comprehensive suite of test
cases, both positive and negative, to ensure the typechecker’s robustness and correct­
ness. The results demonstrate that the typechecker effectively validates correct RSL
programs and identifies errors in incorrect ones. The thesis also explores the benefits of
the two­phase approach, which include improved error detection, efficient type resolution,
and a modular, extensible design.
Future directions for this work include extending the typechecker to support a broader
range of RSL features, optimizing performance, enhancing error reporting, and integrating
with development tools and formal verification systems. This thesis makes significant
contributions to the field of formal methods by providing a reliable typechecking tool for
RSL specifications.

Introduction
The RAISE Specification Language (RSL) is a formal language used for specifying and
designing complex systems. To ensure that these specifications are correct and adhere to
the intended design, a typechecker is implemented. The typechecker checks for correct­
ness by validating that the types of expressions and declarations in RSL are consistent.
1.1
Background
The RAISE Specification Language (RSL) is a formal specification language developed
as part of the RAISE (Rigorous Approach to Industrial Software Engineering) methodol­
ogy. RSL is designed to facilitate the precise specification of software systems, enabling
rigorous analysis and verification of system properties. The language supports a variety
of constructs for defining data types, functions, and modules, making it a versatile tool for
formal methods in software engineering.
1.1.1
The Importance of Typechecking
Typechecking serves several crucial functions in software development. Firstly, it acts as
an early detection mechanism for type errors, allowing developers to catch mistakes at
compile­time rather than at runtime. This early feedback loop is essential for improving
code quality and reducing the likelihood of bugs in production systems.
Secondly, typechecking enhances code readability and maintainability. By explicitly defin­
ing the types of variables and functions, developers can create self­documenting code
that is easier to understand and modify. This is particularly important in large codebases
where multiple developers may be working on the same project.
Thirdly, type systems can enforce invariants and contracts within the code. For example,
a typechecker can ensure that a function designed to operate on a specific type of data
is not mistakenly applied to incompatible data. This helps to uphold the integrity of the
program’s logic and prevents unintended behavior.
Finally, in the context of formal specification languages like RSL, typechecking is indis­
pensable for verifying the correctness of specifications. Formal methods rely on rigorous
mathematical foundations to specify and verify software systems, and type systems pro­
vide a formal mechanism to ensure that specifications are internally consistent and adhere
to the defined rules.
1.2
Motivation
The driving factor behind this thesis is to develop a typechecker for a non­trivial subset of
RSL. While RSL already has advanced tooling, this project focuses on creating a cleaner
and more efficient design for the typechecking phase. This typechecker is a part of a
larger effort to enhance the analysis and validation tools available for RSL specifications.
Moreover, the construction of a typechecker for RSL presents an interesting academic
challenge, combining elements of formal language theory, compiler construction, and soft­
ware engineering. By tackling this challenge, we can contribute to the field by providing
both a practical tool for RSL users and a case study that demonstrates the principles of
typechecker design and implementation.

The primary objective of this thesis is to develop a typechecker for a subset of RSL con­

• Typing Rules: To formalize the static semantics of the chosen RSL subset, pro­

2.1
Type Systems and Typing Rules
A type system is a formal system that classifies expressions in a programming language
according to the kinds of values they compute. It ensures that operations are performed
on compatible types, thereby preventing type errors. Type systems can be static or dy­
namic. Static type systems check types at compile­time, while dynamic type systems
check types at runtime. This thesis focuses on static type systems, which provide the
benefit of catching errors early in the development process.
Typing rules provide a formal framework for defining how types are assigned to expres­
sions and statements in a programming language. These rules are fundamental to type
systems, which are used to ensure that programs behave correctly according to the spec­
ified types. Typing rules are typically written using a notation that includes judgments and
inference rules. Judgments state that an expression has a particular type under a given
environment, while inference rules describe how to derive the type of a complex expres­
sion from its sub­expressions. By applying these rules systematically, one can prove
the correctness of a program with respect to its type annotations, thereby catching many
potential errors at compile time before the program is run.
2.2
Typecheckers
A typechecker is a tool that implements the type system of a programming language to
ensure that the code adheres to the type rules. Polymorphism, a key concept in type
systems, was introduced in ML (Meta Language) and later adopted by other languages
such as C++ and Java [1, 2].
The history of typecheckers is deeply rooted in the evolution of programming languages.
Early programming languages, such as Fortran and COBOL, had very basic or no type
systems. The development of Algol in the 1960s introduced the concept of a more struc­
tured type system, which was further refined in languages like Pascal and Ada. The
introduction of ML in the 1970s marked a significant milestone with the implementation of
polymorphic type inference, a feature that greatly influenced later languages like Haskell
and OCaml [3, 4]. In the 1980s and 1990s, object­oriented languages like C++ and Java
further expanded type systems with features such as inheritance and interfaces, allowing
for more complex and flexible program designs.
Typecheckers ensure that the operations in a program are performed on compatible types,
thereby preventing many runtime errors. They play a crucial role in early error detection,
program optimization, and providing a foundation for advanced tooling such as automated
refactoring, code completion, and more.

2.7
Typechecking
Typechecking is the process of verifying that the types of expressions in a program or
specification adhere to the defined type rules of the language. This involves checking
that operations are applied to compatible types and that type constraints are satisfied.
In the context of RSL, typechecking involves several steps:
• Collecting Declarations: Gathering information about the types of data structures,
functions, and modules defined in the specification.
• Type Resolution: Determining the types of expressions based on the collected
declarations and ensuring that they are consistent.
• Checking Expressions: Verifying that expressions are well­typed according to the
language’s type rules.
Typechecking is essential for ensuring the correctness of RSL specifications, as it catches
type errors that could lead to incorrect behavior or logical inconsistencies.
2.7.1
Type Declarations
Type declarations define new types or aliases for existing types. In RSL, type declarations
can be simple (e.g., defining a new alias for an existing type) or complex (e.g., defining a
new record or variant type). The typechecker must collect and validate these declarations
to ensure that they are well­formed and do not contain conflicts.
2.7.2
Value Declarations
Value declarations define variables and constants, along with their associated types. The
typechecker must ensure that these declarations are consistent with the type rules of the
language and that the values assigned to these variables and constants are of the correct
type.
2.7.3
Type Resolution
Type resolution involves determining the types of expressions based on the collected type
and value declarations. This step is crucial for ensuring that all expressions are well­typed
and that type constraints are satisfied.
2.7.4
Expression Checking
Expression checking involves verifying that all expressions in the specification are well­
typed according to the language’s type rules. This includes checking that operations are
applied to compatible types and that type constraints are satisfied.

2.8
Brief Introduction to Typing Rules
Typing rules provide a structured way to determine the type of expressions in a program­
ming language. These rules help ensure that programs behave correctly by checking that
operations are applied to compatible types.
2.8.1
What Are Typing Rules?
Typing rules consist of premises and conclusions. They state that if certain conditions
(premises) are met, then a conclusion about the type of an expression can be drawn.
For example, a typing rule might say that if an expression e1 has a type T and another
expression e2 has the same type T, then adding these expressions results in a new ex­
pression of type T.
2.8.2
How to Write Typing Rules
Typing rules are often written in a format that looks like a fraction:
Premise1
Premise2
Conclusion
This can be read as: ”If Premise1 and Premise2 are true, then Conclusion is true.”
2.8.3
Interpreting Typing Rules
To understand how typing rules work, let’s look at a simple example.
Example: Addition of Integers
Consider a rule for adding two integers:
Γ ⊢e1 : Int
Γ ⊢e2 : Int
Γ ⊢e1 + e2 : Int
This rule can be read as: ”If e1 has type Int and e2 has type Int, then e1 + e2 also has type
Int.”
Developing a Typechecker for a subset of RSL
8
==PAGE:15==
Explanation:
• The expressions e1 and e2 are being added.
• The symbol Γ represents the environment, which keeps track of the types of vari­
ables.
• The notation Γ ⊢e1 : Int means ”in the environment Γ, e1 has type Int.”
• The conclusion Γ ⊢e1 +e2 : Int means ”in the environment Γ, the expression e1 +e2
has type Int.”
2.8.4
Example Derivation
Let’s go through an example derivation using this rule.
Example: Typing a Simple Addition
Given the expressions 3 and 5: 1. 3 has type Int.
2. 5 has type Int.
Using our addition rule, we can derive the type of 3 + 5:
Γ ⊢3 : Int
Γ ⊢5 : Int
Γ ⊢3 + 5 : Int
This shows that 3 + 5 has type Int.
Typing rules help ensure that expressions are used correctly according to their types. By
following these rules, we can catch type errors early in the development process, leading
to more reliable and maintainable code.
This simplified introduction provides a basic understanding of typing rules and how they
are used to check the types of expressions in programming languages.

3.2
Typing Rules
In this section, we present the formal typing rules for the RSL subset. The rules are
derived from the grammar and ensure type correctness in the specifications. The typing
rules are divided into two phases: Phase 1, which collects and validates declarations,
3.2.3 and Phase 2, which resolves types and checks expressions 3.2.4.
3.2.1
Benefits of the Two­Phase Approach
The decision to implement a two­phase approach for the typechecker was driven by the
need to systematically manage and validate complex type information and expressions in
RSL. The benefits of this approach are manifold:
• Separation of Concerns: By dividing the typechecking process into two distinct
phases, we effectively separate the collection of type and value declarations from
the actual type resolution and expression checking. This separation simplifies the
implementation and debugging processes, as each phase can be developed and
tested independently.
• Improved Error Detection: The first phase focuses on gathering and validating
declarations, ensuring that all types and values are correctly defined before any
type resolution occurs. This helps in catching errors early in the process, such as
duplicate definitions or cross­domain duplications, thereby reducing the complexity
of errors encountered in the second phase.
• Efficient Type Resolution: With all declarations validated and collected in the first
phase, the second phase can focus solely on resolving types and checking expres­
sions. This leads to a more efficient type resolution process, as the environment is
already populated with the necessary type information.
• Modular and Extensible Design: The two­phase approach lends itself to a mod­
ular design, where additional features or checks can be incorporated into either
phase without significantly impacting the other. This makes the typechecker more
extensible and easier to maintain.

3.4.1
Lexer
The lexer converts the source code into a sequence of tokens. F# Lex (FsLex) helps with
tokenizing by providing a way to define lexical analysis rules using regular expressions.
The lexer uses these rules to scan the input source code and generate tokens.
Implementation:
The lexer is implemented using pattern matching to recognize different kinds of tokens.
For example, keywords like ‘scheme‘, ‘class‘, and ‘value‘ are matched and converted into
corresponding tokens.
The lexer processes the input character by character, matching it against known patterns
and generating tokens accordingly. Whitespace and comments are skipped, and recog­
nized tokens are returned to the parser.
3.4.2
Parser
The parser is responsible for transforming the tokenized input from the lexer into an Ab­
stract Syntax Tree (AST). This transformation is guided by the grammar rules, ensuring
that the structure of the input conforms to the syntax of the RSL subset. The parser uses
F#’s FsYacc tool to generate the parsing code based on a set of grammar rules.
Parser Design and Algorithm
The parser operates by applying the grammar rules to the sequence of tokens produced
by the lexer. It constructs the AST by recognizing patterns in the token sequence and
creating corresponding nodes in the tree.
Algorithm for Parsing:
The parsing algorithm follows these steps:
• Initialize the parser with the sequence of tokens from the lexer.
• Begin parsing from the top­level non­terminal (e.g., scheme).
• For each non­terminal, apply the corresponding grammar rule:
– Match the current sequence of tokens against the rule’s production.
– If the rule matches, create the corresponding AST node and advance the token
pointer.
– Recursively parse the subcomponents (if any) of the rule to construct the com­
plete AST subtree.
– If the rule does not match, backtrack and try the next rule (if any).
• Continue this process until the entire token sequence is consumed or an error is
encountered.
• Return the constructed AST or report a syntax error.

Two­Phase Typechecking
The typechecker operates in two phases:
Phase 1: Collecting Declarations
In this phase, the typechecker traverses the AST to collect type and value declarations,
populating the type and value environments. It checks for duplicate declarations within
and across domains (types and values).
Explanation:
• Type Declarations:
– Sort Definitions: Ensure uniqueness and add to the type environment.
– Variant Definitions: Check for duplicates within the variant choices and add
to the type environment.
– Abbreviation Definitions: Ensure uniqueness and add to the type environ­
ment.
• Value Declarations:
– Value Signatures: Ensure no duplicates and add to the value environment.
– Explicit Value Definitions: Ensure no duplicates and add to the value envi­
ronment.
– Explicit Function Definitions: Ensure no duplicates, and add function signa­
tures to the value environment.
• Axiom Declarations:
– Axiom declarations are recorded without further checks in this phase.
Algorithm for Phase 1:
1. Initialize an empty environment.
2. Traverse the AST.
3. For each declaration:
• If it is a type declaration:
– Ensure no duplicate type definitions.
– Add the type to the environment.
• If it is a value declaration:
– Ensure no duplicate value definitions.
– Add the value to the environment.
• If it is an axiom declaration:
– Skip and move on to the next check.
4. Return the populated environment.

Phase 2: Resolving Types and Checking Expressions
In this phase, the typechecker resolves type expressions and checks the types of value
expressions, ensuring type correctness throughout the AST.
Explanation:
• Type Expressions:
– Literal Types: Directly resolve to Bool or Int.
– Named Types: Resolve by looking up in the type environment.
– Subtype Expressions: Ensure validity within the provided constraints.
• Value Expressions:
– Literals: Resolve to their types (Bool, Int).
– Variable Names: Ensure they exist in the value environment and match the
expected type.
– Function Applications: Ensure the function and arguments match in number
and type.
– Infix and Prefix Expressions: Validate the types of operands and ensure they
match the operators.
– Conditional Expressions: Ensure the condition is Bool and then and else
branches are compatible.
– Let Expressions: Resolve and check the initialization and scope expressions.
Algorithm for Phase 2:
1. Traverse the AST.
2. For each node:
• If it is a type expression:
– Resolve the type expression.
• If it is a value expression:
– Check and resolve the value expression type.
• If it is a function application:
– Ensure the function exists and matches the expected signature.
• If it is a conditional expression:
– Ensure the condition is a Bool.
– Ensure the then and else branches have compatible types.
• If it is a let expression:
– Resolve and check the initialization expression.
– Resolve and check the scope expression.
3. Return the typechecking result.

The type and value environments store the types of identifiers. The type environment
maps type names to type definitions, while the value environment maps variable names
to their types.
Explanation:
• Type Environment: Contains mappings from type names to type definitions (SortDef,

AST Traversal and Checks
The typechecker traverses the AST and performs various checks to ensure type correct­

Explanation:
• Value Expressions: The typechecker matches the value expressions against known

accordingly.
• Cyclic Dependency Detection: Detects cyclic dependencies by maintaining a set
of visited types during type resolution.
• Duplicate Declaration Checking: Ensures no duplicate type or value declarations
exist within and across domains.

5.1
Summary of Contributions
The primary contributions of this thesis are multifaceted:
• Grammar Definition: We started by defining a concrete grammar for a subset of
RSL constructs. This involved understanding and documenting the essential ele­
ments of RSL, which served as the foundation for all subsequent implementation
efforts.
• Lexer and Parser Implementation: Using FsLexYacc, we implemented a lexer
and parser that can correctly tokenize and parse RSL code. The lexer and parser
convert the concrete syntax into an abstract syntax tree (AST), which is a critical
step in the typechecking process.
• Abstract Syntax Tree (AST) Design: The design of the AST was carefully crafted
to represent RSL constructs accurately. This design facilitated the efficient traversal
and manipulation of RSL code during the typechecking process.
• Two­Phase Typechecking: The typechecker was implemented in two phases. The
first phase involved collecting and validating declarations, ensuring there were no
duplicate types or values, and checking for cross­domain duplications. The second
phase focused on resolving types and checking the validity of expressions against
the collected declarations. This phased approach ensured a systematic and thor­
ough validation process.
• Typing Rules Formalization: We formalized the typing rules for the RSL subset,
providing a clear and precise set of rules that govern type correctness. These rules
were implemented directly into the typechecker, ensuring alignment between theory
and practice.
• Extensive Testing and Evaluation: A comprehensive suite of test cases was de­
veloped to evaluate the typechecker. This included both positive test cases, which
the typechecker should validate, and negative test cases, designed to test the type­
checker’s ability to detect and report errors.
