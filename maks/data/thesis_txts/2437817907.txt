Abstract
This thesis presents a study of how to securely execute untrusted code.
With web
browsers being used to execute potentially malicious code for millions of users worldwide,
its security is paramount.
Designing and implementing new security techniques are
inherently challenging, partly due to the complexity of modern browsers. Therefore a
set of design principles for building interpreters are designed, using the language Haskell
coupled with the building blocks of monad transformers. The implemented framework,
named Bowser, serves as proof that interpreters can in fact be secure by design, and that
extending it with new security mechanisms is easily achieved. This is demonstrated by
adding dynamic taint analysis to the designed framework, with the goal of mitigating
common attacks extracting sensitive information.

CHAPTER 1
Introduction
Every day, users across the world install apps on their smart phones, run programs
downloaded over the internet or visit web pages of unknown hosts, all without any
assurance of what the relevant code actually does. Untrusted code is constantly executed,
most commonly through web browsers by visiting web pages including dynamic content,
offered by JavaScript. There are very few precautions that are automatically taken by
a web browser before it willingly executes anything the web server gives it. Given a bug
in the browser, an attacker could potentially execute arbitrary code on the users system
or obtain sensitive information.
JavaScript based web applications have become prevalent, with a myriad of different
frameworks to aid programmers in building complex web applications. Instead of hav-
ing a web server handling all user input and constructing markup which is in turn
rendered by the web browser, these actions are off-loaded to the browser itself. Popular
frameworks, like Meteor 1, even have client-side versions of the back-end database for
caching purposes. These purposes provide advantages in web page responsiveness, as
less data has to be transferred back and forth from the web server. Furthermore, it gives
the benefit of putting less load on the web server as more of the work is done by the
client itself.
In light of this shift, going from server to client-based applications, it is natural to ques-
tion the security of the applications executing this untrusted code. Web browsers are
immensely complex pieces of engineering, with millions of highly optimized lines of code.
The language of implementation is one that offers low-level control of memory, typically
C or C++. Writing such programs completely bug free are practically impossible. Sta-
tistically there is a set number of bugs per line, which by extensions mean that when a
developer fixes a bug, there is a possibility of creating a new one. Therefore, the time
it would take to find all bugs and fix them would take an exponential amount of time,
something that is not practically achievable. Take for instance, the GNU Compiler Col-
lection (GCC), a piece of software renowned for its correctness and robustness, still has
thousands of unresolved bugs 2. That being said, the goal of a realistic programmer is
not to create perfect software, but rather usable, scalable and robust software.
This thesis seeks to shed some light on how one can execute potentially malicious code
in a secure manner. There is a need for understanding and scientifically testing the
security implications of adding new features or security mechanisms to software, and

2.1
The JavaScript Language
JavaScript is one of the most ubiquitous programming languages in the world. According
to the popular open source platform, GitHub, JavaScript is more than twice as popular
as any other language 1. This is consistent with the annual StackOverflow Developer
Survey 2, which shows JavaScript being the most active of all topics.
Currently, the most common browsers include their own interpreters, or popularly called
browser- og web engines. There is Google Chrome’s V8, Firefox’s SpiderMonkey (or its
new Servo), Apple Safari’s JavaScriptCore and Internet Explorer’s Chakra, to mention
some.
Due to the myriad of different implementations of JavaScript, the need for specification is
evident, though not all web browser implement the same version of the specification. Due
to the large number of implementations and specifications connected to the JavaScript
language, it has become very flexible with many ways of writing the same program,
including some peculiar behavior of the type system 3.
Internally, JavaScript is a dynamic and weakly typed programming language, imple-
menting the ECMAScript Language Specification [Ecm99]. This specification defines
JavaScript to include the types seen in Table 2.1.

2.2
Monad Transformers
Through pure functional programming and strictly typed type-systems, the programmer
is able to catch more errors at compile time and avoid many common bugs [Wad92;
PW93]. When building complex applications, like interpreters, it is beneficial to leverage
such a paradigm.
Side effects can be difficult to get right, especially in big and complex applications such
as interpreters. Despite of that, the interpreter has to be able to make the necessary
changes to the application’s surrounding environment. As stated by the designer of the
Haskell language, Philip Wadler, interpreters seek a way to “combine the indulgences of
impurity with the blessings of purity” [Wad90, p. 1].
Take for instance the following impure function:
1 f' :: Int -> Int
Listing 2.1: Type signature of function f', taking an integer as argument and returning an
integer.

This function could be made pure by explicitly stating its real world implications, like
so:
1 f :: Int -> RealWorld -> (Int, RealWorld)
Listing 2.2: Type signature of function f with ability to affect the surrounding state.
This new function f, takes an integer, a state of the world and returns an integer, together
with an altered state of the world. In this simple way, side effects are modeled using
pure functional programming.
This technique is very verbose and can be tedious to use. Therefore, the parameterized
type IO is defined, which takes care of the world states:
1 type IO a = RealWorld -> (a, RealWorld)
Listing 2.3: Parameterized type definition of IO, giving support for RealWorld actions to a
given type a.
Now the function f can be simplified by using the type IO:
1 f :: Int -> IO Int
Listing 2.4: Type signature of function f rewritten with the IO type.
This idea can be extended to interpreters, where the state of an interpreter is encapsu-
lated in a simple type abbreviation:
1 type Engine a = InterpState -> (a, InterpState)
Listing 2.5: Type definition of Engine type, handling an interpreter state, Interstate.
Functions manipulating the state of the interpreter could easily be written to use such
a type. To demonstrate, consider the following functions:
1 evalBinExpr :: Int -> (Int -> Int) -> Int -> Engine Int
2 callFunc :: String -> [Int] -> Engine Int
3 declVar
:: String -> Int -> Engine ()
Listing 2.6: Type signatures of interpreter functions handling the Engine type.
The function evalBinExpr evaluates simple binary expressions of integer arguments, and
returns the results, together with the new state of the interpreter. The second function,
callFunc, calls function by taking a string identifier of the function and a list of integers
as arguments. It returns the integer result, together with the new state of the interpreter.
The third, and last function declVar, is a variable declaration statement. It takes both
a string identifier and an integer value, and returns the new state of the interpreter only.

Implementing monads from scratch can be a challenging task for those not used to the
concept or familiar with functional programming. For this purpose, monad transformers
can be used to transform a simple type into a monad. This is done by the transformer
implementing the instances of a set of “typeclasses” for the transformed type. This can
be compared to supporting a set of interfaces, where three central ones are Functor,
Applicative and Monad. By implementing these three typeclasses, the transformed type
implicitly becomes a monad. For a more thorough introduction to monads and type-
classes, please read [Lip11, chap. 3].
2.3
Taint Analysis
When monitoring programs, it is not always sufficient to analyze a program in its static
form, as the behavior the program may first reveal itself when run. Therefore, a solution
is to monitor the given program while running, a technique called dynamic program
analysis. In computer security, dynamic analysis is a key method for analyzing how user
==PAGE:24==
8
2 Background
input propagates through a program. This method of dynamic analysis is named taint
analysis, where a source of taint, for instance user input, is spread through a program
showing what values are affected by the tainting.
The spread of taint can be both explicit and implicit. Explicit taint is when a value
directly “touches” another, for instance where the value of a tainted variable is assigned
to a new variable. Implicit tainting occurs when the programmer uses conditional state-
ments to imply some information about a tainted variable. For instance, if the pro-
grammer checks whether a variable is positive or negative, the programmer learns some
information about the tainted variable. Any data structure that is assigned to hold that
information should also become tainted. By using such conditional statements, and pos-
sibly some preexisting knowledge about the format of a tainted variable, it is possible for
the programmer to deduce the value. Therefore, implicit taint is important to consider.
An attack is when the taint analysis reports an attempted attack, i.e. when a tainted
variable is used in an illegal action, such as to transmit some sensitive information to
a third party. The term overtainting is when such an attack is reported, but no attack
actually occurs. This can be seen as a false positive, and the same terminology can be
used for undertainting and false negatives.
The reader is encouraged to read Schwartz et al. [SAB10] for a more detailed explanation
on dynamic taint analysis and forward symbolic execution.
2.4
Cross-Site Scripting (XSS)
One the most common attacks on modern web browser are using Cross-Site Scripting
(XSS) vulnerabilities, where an attacker can inject untrusted code into a victims browser.
XSS attacks can be divided into two categories; traditional and DOM-based. The former
are vulnerabilities found server-side, most typically by the web server. The latter, on
the other hand, are vulnerabilities found on the client-side, i.e. the code handled by a
browser’s DOM (Document Object Model). In the same fashion, the attack tricks the
web application to insert malicious code into the web page. One of the most common
exploits are to steal the client’s authentication or Cross-Site Request Forgery (CSRF)
token. This allows the attacker to hijack the victim’s session, potentially leaking sensitive
information.

containing JavaScript. An attacker could exploit this by crafting a malicious link and

example of such a link:

change the flow of all looping code patterns. Also, more advanced language features, such
as exception handling using the statements throw and catch, can not be handled by this
naive solution.
Therefore, the interpreter needs a more flexible way of handling such features, without
adding unnecessary complexity to the rest of the interpreter. As suggested in the text-
book ”Theories of Programming Languages” by John C. Reynolds [Rey09, chap. 12],
and shown by Andrew W. Appel [AJ89], continuations solve this issue nicely.
Rewriting the whole interpreter to use continuation passing style (CPS), would take a
bit of a work, even with the example interpreter used in this chapter. It would also
defeat this thesis’s goal of having a modular interpreter where functionality can be
added without having to rewrite everything. Luckily, there is a monad transformer for
it, namely the ContT transformer, found in Control.Monad.Trans.Cont, which saves us
from having to use CPS in the rest of the interpreter, but only utilize continuations
where they are needed.
The ContT transformer is defined below, adding the CPS continuation Cont r to the
monad m. The type of the returned result is r. The callCC function (call-with-current-
continuation) takes a function as argument, applying it to the current continuation.

CHAPTER 5
Adding Taint Analysis
The goal of this chapter is to show how the Bowser Framework can be used for testing
new security mechanisms in JavaScript interpreters. As an example, this thesis will show
how to apply dynamic taint analysis to mitigate leakage of sensitive information. As
mentioned in Section 2.4, one of the most common and easily exploited vulnerabilities
in browsers are cross-site scripting attacks. Such vulnerabilities can be used to steal a
victim’s session through authentication tokens stored in the users cookies. The scope
of this is not to reinvent taint analysis, but show how it can be applied as a security
mechanism to mitigate these types of attacks.
For this, two solutions will be presented.
Firstly, a coarse solution which is simple,
but suffers from overtainting. Secondly, a more fine-grained solution which requires a
larger effort implementing, but does not have as many false positives. Both solutions
will use the Tainted monad and TaintedT monad transformer, available through Hack-
age [Mei15].
5.1

In this section, the simplest solution for applying taint analysis will be discussed, together
with its inherent advantages and disadvantages. When applying dynamic taint analysis
to the Bowser Framework, the simplest and safest option is to taint the state of the
interpreter as a whole. If during runtime, some sensitive information i.e. sources of
taint are touched, the whole state becomes tainted. This way, the attacker can neither
read or write a tainted variable without tainting the state, even though it is for benign
purposes.

advantages and disadvantages.

values of the Value type that can hold data, and wrapping the whole Value type in a

5.3
Summary and Limitations
Throughout this chapter, several solutions for applying dynamic taint analysis to the
Bowser Framework, together with the principles of this thesis, have been presented.
Their advantages and disadvantages have been discussed, and the Bowser Framework
has demonstrated how the solutions are effortlessly implemented.
To review the example XSS attack presented in Section 2.4, it is now mitigated using
all discussed techniques.
The injected JavaScript that touched the document.cookie
property would be tainted, and the web browser could choose to block the request to
the unknown third-party host malicious.com, in the same fashion as Vogt et al. [Vog+07].
This would prevent any similar attack to leak tainted information to a third-party.
Unfortunately, the presented solutions for taint analysis are far from perfect. Even the
last solution presented can be evaded. In the previous section, the context of statement
blocks became tainted if the condition of an if-else statement was tainted. One attack
that is not handled by this solution, is when the attacker applies a similar technique to
the one presented in Listing 5.19. The programmer can use the fact that a block has not
been executed to learn something about a tainted variable. In the following Listing 5.24,
the programmer learns the that the tainted variable is greater than 21. In the program,
the condition of the if statement is evaluated to false and the block is never run, meaning
that the variable small is never set to false. Ergo, the programmer knows the variable
tainted is greater than 21.
1 var taint = 42; // tainted!
2 var small = true;
3 if (taint > 21) {
4
small = false;
5 }
6 // small var is not tainted
Listing 5.24: JavaScript program evading the presented taint analysis solutions.
This attack could be mitigated by tainting all touched variables in all branches of con-
ditional statements, even branches not taken. In this case, the assignment statement of
the small variable, even though it is not run, would taint the variable small.
In summary, this chapter has shown how several solutions to taint analysis can easily be
implemented using the Bowser Framework, though designing the optimal taint policy is
difficult due to the inherent challenges that comes with dynamic taint analysis and the
evasion there of.

CHAPTER 6
Discussion and
Future Work
It is important to remember that the purpose of this thesis is to study how to securely
execute potentially malicious code. The ideas and methods presented in this thesis are
only demonstrated using the JavaScript language, but are in fact general ideas. The
same techniques could just as easily be applied to other languages. There are most
likely language features of other programming languages which would not be supported
by the Bowser Framework, in which case it would need to be adapted to fit said features.
The author has chosen to omit an evaluation of the Bowser Framework’s performance.
This has been done for two reasons; Firstly, there have not been found any other solutions
with the same purpose as the Bowser Framework. Without any comparison, the data
will have little significance to the reader. Secondly, the purpose of this thesis has never
been to create a fully fledged JavaScript interpreter that could compare to the industry
standards found in modern web browsers. Comparing the performance of this thesis’s
implementation to modern web browsers would be irrelevant and possible misleading to
the reader.
Throughout this thesis, the concepts of interpreters and dynamic analysis has been con-
sidered and discussed at length. By contrast, compilers and static analysis are concepts
not touched upon. An interesting extension of this research would be to apply the same
principles of secure interpreter design to compilers, coupled with static code analysis.
Not only would the presented principles provide a framework for secure compiler con-
struction, but also would the technique of using monad transformers create a modular
and robust design.
With regards to future work, in the Bowser Framework the JavaScript Global Ob-
ject [Ecm99, sec. 18] is not implemented. This is a special object found in the root
scope of the interpreter, and includes the functions Object(), Array() and Date(), to
name a few. To implement this feature, one would extent the NativeObj type in the
Bowser Framework, making it fairly simple to implement. The same goes for imple-
menting classes, as they are also a variant of objects. Moreover, only a subset of the
JavaScript AST has been implemented. That being said, the framework is in place, so no
changes has to be done other than adding the implementation for the specific elements
of the AST. Future work could include adding Global Object and class definitions, in
addition to implementing the full AST.

CHAPTER 7
Conclusion
This thesis has presented a study of how untrusted code can be executed in a secure
manner. As the concrete platform to analyze, web browsers was chosen due to how
common they are and the amount of malicious code that they execute.
A set of design principles for building interpreters was designed, using the language
Haskell together with the building blocks of monad transformers. The principles de-
scribes how to build secure by design interpreters, applicable to most programming
languages with similar language features as JavaScript.
As an example implementation of the designed principles, the Bowser Framework was
created. The framework serves as proof that interpreters can in fact be secure by design,
using the language JavaScript as an example. To demonstrate how the implemented
framework is used, it was extended with dynamic taint analysis to mitigate common
attacks leaking sensitive information. The process showed that the implementation was
fairly straight forward due to the modular design of the framework.
The ideas presented are central to general programming practices. With this in mind,
the modular design principles can be extended to all types of software. Applications that
are continuously extended with features that was not taken into consideration during the
design phase, are prone to becoming unstable. Many corporations find value in adding
features to their software products, which in turn can be sold to costumers, but neglect
to maintain the framework supporting them. This creates unstable applications that are
difficult to maintain.
Solving this issue by rewriting the applications are typically rejected by management,
as it is a costly process that does not create enough value, seen from an economic
standpoint. In conclusion, it is important to design software with focus on modularity
and robustness, which should be seen as a small investment making a more stable and
maintainable product down the line.
