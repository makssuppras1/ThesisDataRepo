Abstract
Digitalization of public transport sector is a necessity in the world of today,
with most of the travellers relying on a mobile device or website to buy their
electronic tickets. In such heavily used sectors, new app updates can be highly
expensive and there is no guarantee that the users will update their application
in order to use the new features, which is not an ideal situation, especially when
the update contains new products which the user might benefit from. Ideally
we want a situation where non-developers from the public transport company
would have control over the product catalogue and be able to create and modify
ticket behaviour according to their liking. However, for creating new products
that are further displayed in the app it would not be sufficient to just create a
new ticket type and can depend on new business rules that are not implemented
in the system. Domain Specific Languages have been known for allowing non
developers to interact and change systems according to specific requirements
and domains. DSLs have been used before for product line engineering (PLE),
instead of the classic approach of using feature models, but no apparent research
has been done on digital products. The DSL can be used alongside with a feature
flag design on the mobile application, a way of dynamically changing the source
code behaviour without the need of new app releases, by using meaningful flags
that are interpreted by the system which then renders the right result. This
thesis proposes a software design approach for treating digital products, such as
electronic tickets, as a software product line, by closing the gap between feature
models and DSLs for non-developer domain experts.

Chapter 1
Introduction
1.1
Problem Description
In today’s public transport industry most of the users rely on a mobile device
to buy tickets, which we referred to as travel products. These products come in
various shapes and forms, ranging from single tickets, group tickets, commuter
cards, special tickets, add-ons and many others, with some tickets being very
different to set up compared to the others, e.g. some tickets might require you
to pick a number of hours for its availability, others might require you to pic a
number of zones and have a preset time availability. Due to these differences
in the process of purchasing tickets, the flow of this process is often hardcoded
in the client of the application, meaning that there is usually separate pieces of
code dealing with the UI and flow of purchasing each type of ticket. This can
become a rapid problem when the product catalogue changes, i.e. new tickets
get added, modified or deleted, as the developers need to release a new version
of the application which the users need to download in order to access the new
travel products, meaning that the TPOs (Travel product owners) have little to
no control over their own product catalogue.
This thesis proposes a method to model the structure of digital tickets and gen-
erate application screens that support the described flow, based on the created
models, without the need of hardcoding each screen and ticket type in the source

code of the system.
The mentioned problem was observed at a danish software company called Tri-
fork, when they were writing a tender offer for a well known company in the
Danish public transport sector. A requirement of the customer for this system
was to support a configurable solution for for maintaining the ticket catalogue,
without the need of new app releases or the intervention of other developers.
Further research in this area revealed that there is no conceptual or practical
frameworks to deal with this situation or other similar ones. This arises further
motivation to conduct this project and propose a software design pattern that
fully encapsulates the specific needs of this system or any other similar one.
1.2
Methodology
The main goal of this thesis is to propose a software design modelling approach
that aims at dealing with similar cases of allowing domain experts to maintain
digital products without the intervention of developers and subsequent appli-
cation releases.
We want to argue that this method can potentially reduce
technical debt and that we can close the gap between domain experts using a
DSL and product line engineering based on feature models and improved with
feature flags.
We want to extend the meaning of software product lines by
introducing a way of developing and managing digital products based on fea-
ture models, created using a domain specific language and then interpreted and
further materialized into screens and components based on meaningful feature
flags.
A literature research will first be conducted, exploring the related work of this
problem in relevant domains such as: domain specific languages, product line en-
gineering, software product lines and aspect oriented programming. A synthesis
of these fields should yield the basis of knowledge and understanding required
for solving the problem.
Following the literature research, an in-depth description of the domain at hand
will be provided, explaining the business rules and various terms and concepts
mentioned along this thesis. This section will aid towards building a syntax for
the DSL that is relevant and familiar to the domain experts from the public
transport company.
In order to showcase the utility of a such system design approach, we will de-
velop a DSL, a backend REST API and a mobile application, to serve as our

Chapter 2
Background
2.1
Domain Specific Language
A Domain Specific Language (DSL) is a computer language designed around a
specific domain in order to ease the development processes of some use cases
[KBM16]. These languages offer substantial gains in productivity, as they are
based only on relevant aspects and features of their application domain and
there is no need of understanding the general programming concepts behind
[KMB+08].
Opposed to general purpose programming languages where any
problem can be solved, DSLs focus only on one specific aspect of the system,
usually having a small and simple set of rules [Col14].
The idea of DSL has been around for a long time, firstly under the form of
shell utilities such as awk or sed for Unix systems, query languages like SQL for
databases or for document preparation like LATEX, but also newer languages
such as Cucumber for behaviour driven test development [MHS05] [AR10]. Be-
ing strongly related to model-based engineering, DSLs are getting now more
attention is business-oriented and technical software development, supporting
the communication with the domain experts [Col14].
Although there are many sub genres of DSLs, they mainly revolve around two
major types, external DSLs and internal DSLs [Fow10] [MHS05] [Col14]. An
==PAGE:18==
6
Background
external DSL represents a language different from the main language that is used
to develop the DSL, it usually has its own syntax that is parsed by some external
source code of some system, using language parsing methods. Popular external
DSL are: DOT for graph creation, PlantUML for creating UML diagrams or
Gherkin used to create functional test with a fluent syntax.
With internal
DSLs, the host language is used to create an alternative syntax, that is still
valid code for the main language, but only focused around a group of features
of the language, aimed to handle a specific part of the system. Popular internal
DSLs are: Lisp, Ruby on Rails or the Kotlin type-safe builders.
2.2
Product Line Engineering
Product line engineering (PLE) aims at developing and maintaining a range
of products in an effective way. This is usually achieved by disregarding com-
monalities between products and focusing on the configuration of their variable
aspects and features. This can be acomplished by abstracting away from the
product configuration into a higher level of representation, such as feature mod-
els, that is then mapped automatically to an implementation [VV11]. A PLE
pattern results also in reusable assets for the product line rather starting from
scratch each time and by using a feature-oriented approach assists in managing
variability and exploiting commonality [KLD02].
2.3
Software Product Line
Software product lines (SPLs) is an area of PLE dedicated to software systems.
A common goal for SPL and PLE in general is to have reusable segments from
a product to another in order to ease the development process and to assure
consistency. [AHC+12]. Following this pattern brings advantages such as quick
time-to-market, scaling economy and increased product quality due to reduced
defect rates [Col14]. Features are domain abstractions, which represent program
functionalities and differentiate the products of an SPL, thus each product is
defined by an usually unique combination of these features [TBK09].
SPLs are usually systems configured by static declarations, such as configuration
files, so changing the behaviour or configuration of the system after runtime is
impossible without recompilation or restart.
Dinkelaker et.
al.
[DMFM10]
extended the means of software product lines to dynamic software product lines
by proposing a novel approach based on aspect-oriented programming concepts
and by using dynamic feature models that describe the variability in the system.

2.4
Feature Models
Feature models are organized hierarchies [ACLF09, TBK09] that encapsulate
similarities and differences between system and software system families in PLE.
Feature models suggest a tight formalisation of the features of the product and
possible constraints required for configuration or validation of the models. These
constrains may refer to certain parts or aspects of a system being required in
order to support the selection of some features [AHC+12]. Feature models are
denoted by their feature diagrams, which is a visual representation of the pos-
sible features and constraints of the systems in a software product line. These
diagrams can be interpreted as trees (or DAGs) with the root r being the name
of the product and where the nodes are strings that represent the name of the
features used for that product line. We can see in Fig. 2.1 a feature model
diagram of simple e-shop systems family. The edges between the nodes specify
the type of relation between the features and help further decomposing a fea-
ture in sub-features, with the feature being the parent node of the sub-features
[ACLF09, CHKK06]. Thus the child nodes of a a parent feature can form:
• And-groups: features in this group can either be mandatory or optional,
as we can see in Fig. 2.1, the root node E-Shop forms an And-group with
three mandatory child features and one optional.
• Alternative-groups/Xor-groups: features in this group are mutually ex-
clusive, as presented in 2.1, the optional feature node Security has two
mutually exclusive options, either High or Standard.
• Or-groups: at least one feature in this group is mandatory, like in Fig.
2.1, the feature node Payment forms an Or-group, as at least one of the
features must be selected.
Besides the parental relational types between nodes, there exists also a notation
for specifying constraints between features:
• implies/requires: Represents that the selection of a feature requires an-
other feature to be selected in order to obtain a valid configuration. An
example of this situation can be seen in Fig. 2.1 where CreditCard im-
plies High, meaning that if the configuration of a product uses the credit
card payment option, then it must implement the high level of security.
• excludes: Represents that the selection of a feature excludes the use of
another feature. Similar to previous example, CreditCard implies High
can be interpreted as CreditCard excludes Standard, meaning that if the

cannot use the standard level of security.

2.4.1
Formal Notation
The semantics encapsulated by the feature model diagrams can be represented
as propositional formulas where each feature f is a boolean variable. The formal
representation should conclude with the configurations allowed by the feature
model diagram, thus a valid configuration is represented by a finite set of fea-
tures. If we assume that our diagrams are rooted trees with the root r we can
represent the contents of the diagram as a conjunction of the translation of the
elements from the diagram. These notations can bee seen in Table 2.1.
The original feature model notation was first introduced in Feature-Oriented
Domain Analysis (FODA) [KCH+90] and has since then received significant at-
tention, hence numerous possible extensions have been proposed in order to fur-
ther aid in the modelling process. Cardinality-Based Feature Modelling (CBFM)
was introduced by Czarnecki et. al. [CHE04] as an adjusted combination of the
scarce concepts of feature models existing at that time such as:
• feature cardinalities: annotation of the features in an UML-like fashion
with cardinalities such as [1..∗] or [2..2] came as an extension to manda-

2.5
Feature Models and DSLs
As mentioned in the previous section, feature models are usually an appropri-
ate method to model products from product lines, but in the case of a non-
programmer domain expert, they lack readability and domain concretization.
In such cases we do not care about a general programming approach solution,
in order to not disregard the domain experts from this task, and to not lose dis-
tinction between problem space and solution space, a DSL can serve as common
ground between the configured environment of a feature-model based solution
and the capabilities of a general purpose programming language [VV11]. The
feature model is to be used as a guide throughout the design phase of the system
as well as the design phase of the DSL, meaning that the specified features in
the feature model need to exist in the system, as well as it should be possible
for any feature to be "flagged" (turned on or off) by using the DSL.
2.6

Feature Flags
Feature flags (also known as feature toggle, feature trigger or feature flipper)
represent a design pattern that allows different parts of the code to run based
on some specification/configuration, usually persisted in a database 2.2. This
results in developers being able to remotely enable/disable features or even
change the code behaviour. This technique often used in practice for many ben-
efits, such as: continuous integration, continuous experimentation, continuous
deployment, A/B testing etc. For example if a piece of code or some component
breaks in production, it can be turned off from some remote interface. Even if
feature flags is a design pattern that is heavily used in production practice in
today’s standards, it is not a frequently found topic in academical or science
research, with most information coming from professionals using this technique
in real-world production development from sources as blog posts or interviews

2.7.2
Feature Flags to Feature Models
Based on the provided information about feature models, more specifically the
formal notation using boolean variables, it can be seen how individual compo-

leaf node of a feature model as a feature flag, that is represented by a boolean
variable i.e true or false, on or off, 1 or 0. If the truth value of a feature flag
is true, then the feature represented by that flag will be included in the final
product. Feature flags are considered an implementation detail only, meaning
that the term of feature flag is relevant only in the source code, when the feature
is turned on or off by using the DSL and not be referred in the feature model.

Related Work
Research shows related work has been done in the area of dynamic mobile appli-
cations. Dygimes [CLV+03] is a framework used for dynamically generating user
interfaces for mobile computing devices and embedded systems. The purpose of
this framework is to ease the work of UI designers as well as application devel-
opers. Coninx et. al. showcase the use of this tool by introducing an example
task, which implies the managing of a simple publication database, where a user
needs a login page for performing queries on a database and how the UI for a
such task is generated by their framework. The process starts by modelling the
task using Concur Task Tree [Pat99], a modelling standard that provides a hier-
archical structure and a notation to specify temporal relation between activities.
The Dygimes process takes this task specification and builds task set and an
interaction specification that are then used to identify the required UI elements
that are needed. In order to make their solution available for any type of device
in an embedded system, their framework uses XML descriptors to dynamically
generate the UIs for the given task, based on layout constraints, device profile
and available widgets. This example is strongly related with the subject of our
project since it explains how to generate UIs for a modelled task, but the solu-
tion is oriented for mobile computing devices used in embedded systems rather
than general consumer smartphones.

2.8 Related Work

Research on DSLs shows that this method of using a DSL has been previously
used for managing a line of products. Voelter et. al. have used a DSL based
approach to manage the behaviour of a product line of water fountains. More
specifically, the DSL they created was used to model the fountain itself by
specifying its hardware components such as pumps and basins, as well as the
relationship between them. Moreover the DSL was used to model the water
behaviour of the fountains, which dictates the quantity of water pumped by
each pump from the fountain, nozzle behaviour, time delays, power usage and
how the different pumps should react when certain aspects happen e.g a basin
is full. [VV11]. However their work strongly relates to how different pieces of
hardware such as pumps, basins, nozzles interact between themselves and their
behaviour and do not refer to configuring a software product line using a DSL.
VML4RE (Variability Modelling Language for Requirements) [ASM+09] is a
DSL for specifying requirements in a software product line. The authors start
from the idea that all the products in a software product line share many com-
mon features, as they were derived from the same set of core assets, usually
represented using a feature model. This language allows a domain expert to
describe programatically how the requirements elements from different models
should be composed in order to create a complete model for software product
line. Their use case showacases an example from smart home automation where
different requirements of different devices are stored in separate models and
diagrams of various types: feature models, UML diagrams, use ase diagrams.
The VML4RE language allows reading from the different models and combine
them using provided compositional operators. It allows selecting, inserting or
removing components from the different models. They also illustrate the use of
a such language in application engineering, where the feature model configura-
tion is used as a guide for the process of gathering product-specific requirements
models. The VML4RE language interpreter is able to process the requirements
of a software product line, from different separate models, in order to extract
the product-specific requirements. However their example focuses on diagram
and model composition that eventually aids in the process of creating a product
from a software product line and the DSL does not directly interact with any
live application or system that is bound to change.

Design
This chapter will explain in detail the design choices made for creating the DSL
for configuring tickets. It includes the abstract syntax, design for the grammar
of the DSL, its semantics but also the requirements that the DSL needs to fulfill.
The current chapter will also show examples on how the DSL is to be used, as
well as the result coming from the performed action like the abstract syntax
tree and the mobile application screen resulted.
4.1
Design Choices
The are four main parts in the design phase of the DSL. The first part is the
domain analysis, that was already conducted in 3. The second part is to define
the abstract syntax of the language, which represents the structure of the DSL.
The third part is represented by formulating the concrete syntax under the
form of a context free grammar. The final part is the semantics of the language
which encapsulates the translation of the rules denoted by the abstract syntax
into library calls and base language constructs that perform the required action,
which will be presented in the implementation chapter 5.4.
==PAGE:38==
26
Domain-Specific Language Design
A DSL can be designed based on an already existing language which indeed
speeds up the development process as you can use already existing compilers or
interpreters for structures like JSON or XML. In most cases this is sufficient and
it is the right approach as it can be easier and cheaper to develop. However,
in this case, the users of this DSL will be non-programmer domain experts
from the Danish public transport companies so we can assume they have zero
programming knowledge and would not be comfortable with a programming-
like syntax such as XML or JSON. The chosen approach is to build a complete
custom syntax that can easily be understood by such domain experts.
4.2
Domain-Driven Design
As mentioned in the previous section, the syntax of this DSL should be eas-
ily understood by somebody with zero programming knowledge and the easiest
way to achieve this is to develop the syntax using relevant words from their
domain. The language is designed using a domain-driven approach, meaning
that the terms used in the DSL should relate to their actual meaning in the
business domain and should be terms relevant to the users of the language. The
users of this DSL will be domain experts from the public transport company
with no programming knowledge, if high usability and productivity increase is
expected, the syntax should be as simple as possible. Following the patterns
for evaluating usability of domains specific languages, we need to have a user-
centered language development [BAGB14], by designing our language syntax
tailored around relevant words and concepts from the business model. As un-
derstandability is a key factor for the domain experts, the proposed DSL syntax
will serve as an extra layer of disambiguity over the feature models, allowing a
more natural way of describing a product.
4.3

Our Domain Specific Language needs to be able to describe the different ticket
characteristics mentioned in 3.3. This specification will then be parsed into fea-
ture flags that will represent a sequence of features/functions required to run,
such that the android app can materialize a screen or sequence of screens in or-
der to allow the purchase of the created ticket. Based on the customer require-
ments, the application should already have commuter cards, metro supplement
for commuter cards and single tickets, but in order to explore the limitations
of our proposed solution, we will design our DSL to be able to create any type

4.4.4
Semantics of the DSL
In contrast to abstract and concrete syntax, which define only the well-formedness
and the concrete representation of a language, the semantics of a computer lan-
guage provide the meaning of the different language elements, as well as usage
rules and any possible constraints. There are multiple ways to define the se-
mantics of a computer language:
• Denotational semantics [Sch86]: where the different sentences in a lan-
guage are interpreted as a denotation, like a mathematical object and are
described using mathematical notation.
• Operational semantics[Plo81]: where the language execution is described
directly rather than by translation, since the semantics correspond more
or less with their interpretation.
• Axiomatic semantics[GTWW77]: where the different sentences in a lan-
guage are checked against an axiom and do not make any distinction
between the meaning and logic of the sentence.
To show the semantics of the proposed DSL, denotational semantics will be
used.

4.6
Feature Model Translation and Validation
The feature model is used as a general source of truth for the possible tickets
that can be created and it is based on the grammar of the DSL. A ticket creation
or modification is successful if and only if it is a valid selection of the system
feature model presented in figure 4.4. The feature model will be used to create
a validator for the tickets that are to be created. The features present in the di-
agram, refer to widgets that the user can interact with and are part of the of the
ticket purchasing process, such as SelectableJourney which will allow an user
to select the origin and destination for the ticket he wants to purchase, but also
widgets that cannot be interacted with, such as FixedZones, which will dictate
the zones that the purchased ticket is available. The other features represent
ticket properties and are not to be interpreted as widgets, like PriceRule, which
is a rule that calculates the price of a ticket, or FixedV alidity which represents
a fixed validity of a ticket that does not get incremented by the number of zones.
An exception can be FixedPrice, which is a property of a ticket, but will also
be displayed as a fixed widged that the user cannot interact with. It is to be
noted that the feature model does not contain any features regarding special or
addon tickets, as these are considered tickets too and they are modelled as a
combination o the features in the model. For example if a ticket configuration
has fixed widgets, that would imply the ticket is a special ticket and an addon
would be just a ticket type linked to another ticket type.

• Backend API: this is the REST API that represents the backend services

• Android App: an app user would interact with the transport products

5.2.2
Rejseplanen API
The Rejseplanen API is a Danish service that is used to retrieve public transport
information such as stop locations, journeys departure times etc. Using this API
eases the development process for the proposed system, as it serves as a data
layer that will provide almost all the data needed creating the different ticket
types. The individual services provided by this API are:
• Location service: this service can be used to retrieve a list of possible
journey matches based on some user input. The matches returned by the
service are stops/stations, points of interest and addresses.
• Trip service: this service calculates a trip based on given origin and desti-
nation. The origin and destination can be stop or station IDs, coordinates
based on addresses and points of interests or coordinates sent by the client.

• Multi departure board service: this services is a combination of station-

6.1
DSL with ANTLR
There are various ways to implement a DSL. First of all a DSL can either be
internal, meaning that is build upon the syntax and semantics of a general
programming language and it is usually used for simplifying some programming
aspects of the host language. The other option is for a DSL to be external,
meaning that the DSL syntax is parsed by a parser/interpeter into valid code
of the language used to implement it, which is also the choice made for this
system.
==PAGE:74==
62
Implementation
6.1.1
Interpreter
The most basic approach would be to create everything from scratch, without
making use of any tool that exist out there. This option can be good if the scope
of the DSL is very specific or if size and speed are of really great importance.
In order to simplify the development process our DSL will be implemented using
the ANTLR a top-down parser generator tool, which will generate a parser based
on the provided context free grammar. The structure of the generated code can
be seen in figure 6.1. Usually, the grammar specification is split into parser-
specific and lexer-specific grammars.
Figure 6.1: The folder structure of the ANTLR generated code
6.1.2
Lexer and Parser Grammars
A lexer is the part of the DSL that performs lexical analysis on the input text
and transforms it in meaningful tokens that can be understood then by the
parser. The parser then will build a parse tree from the tokens, that represents
the whole program. ANTLR will allow us to generate Java code from ANTLR
grammar files (very similar to context-free grammars and regular expressions),
which will be used to visit the nodes from the parse tree. The specific ANTLR
implementation of the lexer and parser can be seen in appendix A.1 and ap-
pendix A.2
==PAGE:75==
6.1 DSL with ANTLR
63
The lexer will read the raw data sent from the ticket configurator and convert
it into a token stream which is then passed to the parser that will attempt to
match the stream to an abstract syntax tree.
6.1.3
Domain Mapping
Based on the provided grammar, ANTLR generates a set of methods based on
the visitor pattern, that allows visiting the abstract syntax tree created. These
methods are provided using an interface in order to allow overriding the existing
methods and modify them according to our needs. In order to be able to map
the contents of the DSL to domain models, Kotlin data classes will be used for
serialization 6.2. The data classes represent a copy of the attributes present in
the DSL abstract syntax. The DSL content is mapped to the domain model
using a visitor class provided by ANTLR. This class can be extended and its
methods overridden in order to fit the needs of the DSL. First the method
mapper is used to find out what operation is being executed, create, modify or
delete. This mapper can be seen in figure 6.3. After the method was obtained,
the rest of the DSL content is parsed into a configuration in the domain model,
this method can be seen in 6.4. Since a configuration is composed of multiple
models: a list of widgets, price, validity; this domain mapper wraps the relevant
mappers for the models in a ticket configuration i.e. a widget mapper, seen in
figure 6.7, which extracts a list of widgets from the DSL input, a price mapper,
seen in figure 6.6, which extracts the price and price rules from the DSL input
and a validity mapper, sen in figure 6.5, which extracts the validity and validity

7.1.8
Limitations
This approach has proven to offer certain advantages on the given problem.
Such a system benefits from dynamic configuration of ticket types that then
can materialise into mobile screens, allowing domain experts to manipulate a
product catalogue and the purchase flow of a product without the intervention
of any developer or the need of any deployments or subsequent application or
backend releases. However such a pattern comes with its limitations:
• The widgets that will be rendered on the screen need to be very detailed
in order to handle all the possible cases that a ticket my impose. In our
example the purchase flow is pretty simple but in a real world situation,
the purchase flow of a ticket might span over a series of multiple screens
where specific widgets need to be rendered so a much more complex DSL
structure an syntax is needed.
• Older devices might suffer from screen twitching due to the components
being rendered when the configuration is pulled from database and inter-
preted on the screen. An optimization for this has been providing the
widget map through dependency injection which help with compilation
time.
• For an industry-ready DSL, the language might contain many more to-
kens and rules, thus the domain experts, given the fact that they are
non-developers, might have a rough time getting used to the syntax and
semantics of the language.
• Similar results could have been achieved by using something like JSON
or XML parser, however these notations come with very strict syntax
containing non usual characters and rules for people that do not use this
stuff often, such as curly brackets, chevrons, semicolons, which all add to
the complexity of the language.

7.1.9
Comparing Results
In regards to the related work found when conducting this thesis, a relevant
comparison can be made with the work of Coninx et. al. [CLV+03] on the
Dygimes framework. Their solution is designed to create UI that can be easily
integrated into embedded systems. Although a similar concept, their approach
differs in many aspects. Their approach relies on modelling each UI as a series
of tasks, using a concur task tree specifiation, a visual notation used to specify a
hierarchical structure of tasks and their dependencies. In this type of approach, a
designer with some design and programming knowledge is needed to first design
the UI. Using this approach implies that there is still the need of a designer to
create the UI beforehand and then attach it to the required system or device.
This does not provide the same level of accessibility as the solution proposed in
this project since the domain experts would still have to communicate with a
designer or developer, or learn how to do that task themselves. Moreover, the
concurring task tree model proves to be a much more complex structure than
the feature model used in this system, which also adds to the complexity of the
DSL and reduces its usability by non-developer domain experts.

Chapter 8
Conclusion
During this master thesis project, the main focus was to design and develop a
DSL that supports maintaining tickets for the public transport sector, that also
can materialize into components that can be rendered on the screen of a mobile
smartphone. The primary objective of the DSL is to allow domain experts to
maintain a travel products (create, modify, deactivate) without the intervention
of any developer and without any unnecessary deployments or application up-
dates. The project concluded with a system that showcases the main objective,
a system where a user can interact with the product catalogue and the changes
can be seen live on the mobile application.
8.1
Discussion
This project has proposed a novel approach on building and designing mobile
applications as well as systems in general. The usage of feature models allows to
showcase all the possible combinations of the products in a software product line
as well as being validator for the business rules that allow the different tickets
to be created. There are many more aspects and extensions regarding feature
models that, for simplicity, were not included in this project.
Moreover, this thesis proposes to treat digital products such as electronic tickets
==PAGE:126==
114
Conclusion
or electronic subscriptions as a software product, in order to benefit of the
feature modelling aspects. One of the main ideas that started this project is
the concept of feature flags, a quite new design pattern for mobile and web
applications.
The key factor of the DSL is that it should be able to be used by domain experts
with no prior programming knowledge, which lead to designing a very simple
language. The scope of the language could easily be extended to include more
complex operations and even allow modifying the design of the components,
but implementing these extensions comes with a trade-off in simplicity of the
language but also of the syntax.
8.2
Further Work
As mentioned in the previous section, the scope of this DSL can easily be ex-
tended to allow more complex operations. Eventually, a DSL can be built that
allows full customization of the components rendered on a screen, without any
extra deployments.
Optimization can be an issue here, since the more data
is provided through http requests, the slower the application can render and
perform.
