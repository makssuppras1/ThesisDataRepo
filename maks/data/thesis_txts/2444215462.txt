Abstract
This thesis explores the possible interactions when 3D modeling using AR on a mobile
device. Three versions of a 3D modeling application were developed, where either the
position and orientation of the device, a 3D cursor or gestures were used as input.
The modification tools of translation, rotation and scaling were implemented for each
of these versions. The three interactions for each tool were tested on 15 participants
in order to find advantages and disadvantages of the interactions.
The results showed that foreknowledge about 3D modeling and the use case of the
users had a great impact on the preferred interaction. Specific preferences were influ-
enced by the precision, ease of use, engagement and intuitiveness of the interaction.

CHAPTER 1
Introduction
3D modeling has until the last 5 years primarily been done on computers through a
keyboard and mouse interface. With the advent of consumer virtual reality devices,
such as the oculus rift and the HTC Vive, these established interfaces has been
challenged as users could now use controllers to modify the shape of objects, by
simply dragging the controller through the air. Another technology, which has been
around for a while but only just recently has been improved enough to be practical
for consumer products, is that of augmented reality (AR). It offers the benefits of
being able to see the real physical environment while overlaying virtual content. 3D
Modeling using AR interfaces has mostly been investigated in specialized setups used
for research, but with the advent of consumer devices such as the Hololens and Magic
leap, it is only a matter of time before someone creates a consumer ready AR 3D
modeling application.
One device category - the smartphone and tablet category,
which is by far the biggest, is however still missing. 3D modeling is not regarded
as ideal on a touchscreen device, but new technologies that allow mobile devices
to use practical AR might redefine this assumption. Technologies such as Apple’s
ARKit and Google’s ARCore enables consumer handheld devices such as phones
and tablets to deliver AR. This enables mobile devices to use the benefits of world
tracking comparable to that of virtual reality together with the benefits of using
a touchscreen. Until now, the focus has mainly been on developing hardware and
software solutions that enables practical AR, whereas how we should actually interact
in this environment has been a second priority. User interactions using AR on a mobile
device with a touchscreen interface is therefore still at its infancy and the literature has
only scratched the surface of how we should interact with it. Furthermore, each use
case often requires different user interactions, whether its games, social applications or
in the case of this project 3D modelling. The research investigating user interactions in
3D modeling specifically using this interface is almost non existing. But as millions
of devices now support this technology, it is becoming increasingly important to
investigate whether or not this interface is practical for 3D modeling, and if it possibly
has some benefits over other interfaces. Some established benefits of the touchscreen
are that of the directness[16] of using your fingers to move objects or press a virtual
button. However, the size of the finger and the accuracy of moving it can result in
reduced precision, when having to touch a specific point. In relation to augmented
reality, several benefits have been established, and the ability to see the real world
in context is possibly the biggest advantage. Disadvantages are however also present,
such as increased fatigue, the novelty of the interface and the problem of not having
a completely static viewpoint. The first step to figuring out the potential benefits
and problems of modeling in AR on a mobile device is however to investigate how it

The goal of this project is therefore to investigate the possible interactions when 3D

CHAPTER 2
Background
2.1
Augmented reality on mobile devices
The term augmented reality is widely used to describe cases where virtual objects
are placed in the physical environment, but to put it into perspective, the ”reality
continuum” introduced by Paul Milgram and Fumio Kishino[24] can be used. The
continuum ranges from the real environment, where only real physical objects are
present, to the virtual environment where only virtual objects are present. The latter
is the environment presented in virtual reality, where the user is only able to see the
virtual world. In between lies AR where digital objects are layered on top of the real
and augmented virtuality where real objects are introduced to the virtual world.
Figure 2.1: Representation of the ”virtuality continuum”[24].
In this project AR is used, as only a few virtual objects will be visualized on the
screen. Using a mobile device also restricts how much of the users field of view is
taken up by the screen, so unless a headset is used in combination with the mobile
device, it would never be able to offer a complete virtual environment. The user will
in this project hold the device in front of him, and by displaying the camera feed on
the screen with virtual object layered on top AR will be achieved. To change the
view of the scene, the user can walk to a new position or rotate the device.
2.2
3D Modeling
3D modeling is the process of producing a digital 3D representation of a shape in
a virtual environment. This process and the produced model is often used in the
industries of games, movies, construction and architecture. Some of the most estab-
lished 3D modeling software include Autodesk Maya, Autodesk 3D Studio Max and
==PAGE:12==
4
2 Background
Blender, which are all desktop based and mostly used in the games and movies in-
dustries. In construction and architecture software such as Autodesk Autocad, Revit
and sketchup are used as they offer better tools to create precise geometries. Some
of these are building information modeling software which allow for specification of
additional properties such as material properties, cost, safety and operation of the
model. The proposed interaction solutions in this project can be used in all of the
mentioned industries, but the focus will be specifically on the tools used to modify a
3D model. In this chapter, the entities that define a model and different techniques for
modifying them will be presented followed by a presentation of the specific modeling
tools used in this project.
2.3
Mesh entities
The representation defining a shape can be categorized into different building blocks
that makes up the shape. These building blocks will in this project be referred to as
mesh entities (see figure 2.2). The most simple entity is a point in 3D space defined
by its x,y,z position. These points are referred to as vertices, and can have additional
attributes beyond its position such as color, UV coordinates, bone weights etc. The
second entity is a line connecting two vertices, which is considered an edge. An area
stretched between three or more vertices is considered a face and is the third and
last entity considered in this project. Connecting one or several faces is considered
a mesh. These three entities are defined to ease the modeling process, as they allow
users to modify individual entities. For example when modifying a face, it will keep
the geometric relations intact between the vertices, that defines the face.
Figure 2.2: The mesh entities of a model.

2.4 Modeling Techniques

2.4
Modeling Techniques
There are several different techniques used when modeling. Some of them are listed
here:
• Box Modeling[18] is where you start with a primitive shape such as a cube,
cylinder or sphere and from there modify the shape using various tools until
the desired shape is produced.
• Edge/Contour Modeling[30] is a technique where you create a loop of edges
along the contour of the wanted shape and then continuously add connected
loops until the desired shape is produced.
• NURBS/Spline Modeling[30] creates a mesh that runs along one or more bezier
curves.
• Digital Sculpting[30] creates meshes similar to the process of sculpting with clay.
By using different sculpt tools the user can shape the mesh.
• Image Based Modeling[30] is a process where 3D object are derived from 2D
images.
The focus in this project will be the technique of box modeling. Box modeling is a
polygon based modeling method that is considered a relative quick and easy modeling
technique to learn.
2.5
Chosen modification tools
Some of the most essential modification tools used when box modeling are outlined
below, which are also the ones implemented in this project. Additional tools were
wanted, but because of the scope of the project, they were not implemented.
Entity selection
As vertices can lie close to each other, being able to precisely select a specific entity
is essential in order to start modifying a mesh. This solution should be robust and
not frustrating to use. It should also be possible to select several entities and perform
the same modification on these, while retaining the geometric relations between the
modified entities.
Translation
Translation is the action of moving an entity to a new position and will be possible
to use on all entities in this project.
Extrusion
Through extrusion, the user can modify a mesh by ”dragging” a face from its position,
while new adjacent faces are generated, so the dragged face is still connected to the
==PAGE:14==
6
2 Background
original geometry. This allows the user to extend the geometry, as it generates more
vertices and faces.
Scaling
Scaling can in this project only be used on faces and the scaling has been limited to
being applied on all local axes of the face. This means that all vertices of the face
will be moved towards or away from the center of the face when scaling.
Rotation
Rotation is the action of rotating an entity around an axis. It only makes sense to
rotate edges and faces, but in this project only faces can be rotated.
Undo/Redo
Being able to undo and redo modifications is essential when modeling.

CHAPTER 3
Related Work
AR on a mobile device supports multimodal input in the form of the touchscreen
display and the spatial awareness comparable to that of virtual reality. It also allows
for a 3D cursor, which can be used similarly to that of a desktop mouse. Because
of these properties, it was chosen to look at the interactions used in established 3D
modeling software to modify a mesh, using the input interfaces of a touchscreen,
VR controllers and a keyboard/mouse combination. Several studies has investigated
how to interact with objects in AR on a mobile device, which are also presented.
Finally some general human computer interaction guidelines for augmented reality
will be presented, as these can help avoid any pitfalls in the pursuit of the optimal
interactions.
3.1
Modeling in context
3DC.IO[1] is a mobile application that lets users create models from primitive shapes
such as planes, cubes, spheres, cones, toruses and cylinders. Each shape can be trans-
lated, scaled and rotated along each axis by touching and dragging a modification
widget. You can move around the scene by using one finger to rotate around a fixed
point and by dragging with two fingers it translates the camera along the camera
viewing plane. Pinching with two fingers zooms in or out.
Sketch 3D[25] is another mobile application that offers users to modify individual
entities as well as other advanced features. Entities are selected with direct touch
and then they can be translated, rotated or scaled by using drag. These modifications
are restricted to be in relation to the current view plane, which required the user to
move around the scene every time a new axis of modification is needed.
Google Blocks[13] is an application made by Google for modelling simple 3D shapes
in VR with the use of controllers. You start with a primitive shape such as a cube,
sphere or cone. The user is given the tools to reshape, extrude and subdivide the
chosen model. Reshape lets you translate vertices, edges and faces by grabbing and
then dragging them with the controller. Extrude is done by grabbing a face and then
translate it to a new position. Subdivide makes it possible to divide the face at the
location of the controller into two. The rotation of the subdivision can be controlled
by the orientation of the controller. Entities can be rotated by grabbing them and
rotating the controller to the desired rotation. By holding down the same button on
both controllers it is possible to scale and rotate the entire scene.
==PAGE:16==

3 Related Work
Blender[7] is a desktop 3D modelling software, which makes its interactions built
for a mouse and keyboard interface. You can select faces, edges and vertices with the
mouse and then use the modification widgets(See figure 3.1) to translate, rotate and
scale them. By holding down the mouse button on one of the axes on the widget,
the user is given controls to translate or scale along an axis or rotate around an axis.
These are common widgets used in most desktop modeling software. Users can also
translate entities without a widget by grabbing them, press a shortcut on the key-
board and move the mouse. This will translate the entity along the camera viewing
plane. Similarly an edge or face can be rotated along the camera viewing plane by
rotating the mouse around the center of the entity. Scaling can also be done in this
manner by dragging the mouse closer or further away from the entity after activating
scaling with a keyboard shortcut. Using a touchpad the user can rotate around the
scene by using two fingers and zoom in and out by pinching.
Figure 3.1: Blender modification widgets[31].
The general advantage of desktop modeling software is that they offer an exten-
sive number of advanced tools. This is however doable on a desktop interface as the
comprehensive interface has plenty of space on a large screen. The precision of the
mouse and the number of shortcuts on a keyboard also enables this to be practical.
This however, is also a disadvantage in terms of the learning needed to master them.
One of the most noticeable differences between VR with controllers, mouse and key-
board interface and a touchscreen interface is the precision. 3DC.IO has left out the
option to modify individual faces, edges or vertices - either to make it more simple or
because touch is less precise[20]. This imprecision was also experienced with the use
of Sketch 3D. In terms of modifying a shape, it can be argued that Google Blocks has
the most intuitive method of selecting and modifying entities in one single grab and
move action, whereas Blender typically requires several commands and interactions
in order to achieve the same result. Blender however, offers more precision when
moving, rotating or scaling an entity along one axis at a time.
Other studies have tried to investigate the interactions used to modify an object in
AR. Henrysson[14] developed 6 techniques for translating an object and 7 techniques
for rotating an object. He found isomorphic interaction - moving the object in relation
to the device, best for translation, although not the most precise. For rotation he

3.2
Human computer interaction (HCI)
How to design and use computers software is a vast research field as it encompasses
everything from apps on a smartphone, to software running on desktop and head
worn displays. One distinct difference of using AR is however, that virtual objects
are mixed with the real world, and combining this with the portability of a mobile
device presents some distinct HCI principles to be aware of:
Kourouthanassis et al.[19] made an analysis of the HCI research in mobile AR ap-
plications and from this proposes some guidelines to follow:
• The application should only present relevant information at all times in order to
reduce the cognitive overhead. This is important as the user also have to pay at-
tention to the surrounding environment while using the device. Personalization
and filtering of features can be a useful way to achieve this.
• The application should use familiar icons and interaction metaphors to increase
the learnability of the system.
• The application should inform users regarding its current state or changes in
its state. This reduces frustration and improves the learning curve of using the
application.
Other guidelines such as informing about content privacy is not considered important
for this project.
One problem of finding human-computer interaction guidelines for AR is that a vast
number of different AR systems exists with each their own interface. This includes
head worn displays, touchscreen devices, fixed screen systems or other setups using
specialized controllers. As the domain of AR has not yet settled on a specific inter-
face, some general AR HCI guidelines identified by Dünser et al.[11] will be presented
below. These have been simplified to a bullet point list:
• Using physical objects to manipulate virtual objects can utilize the concept of
affordances to improve the users ability to figure out how to interact with it.
• Reduce cognitive overhead and enable the user to focus on the actual task.
• Reduce physical effort by making it possible to accomplish a task with a mini-
mum of interaction steps.
• Improve learnability by using interactions and design that are similar or akin
to what the user is already used to.
==PAGE:18==
10
3 Related Work
• Focus on user satisfaction and test it.
• Make the application flexible in order to accommodate individual user prefer-
ences.
• Focus on responsiveness and feedback by keeping the users informed about the
current state of the system.
• Keep errors at a minimum in order to reduce user frustration.
The concept of affordances, which are the possible interactions between people and
the environment and signifiers, which signals what actions are possible and how[27]
are also relevant. Thinking about these terms when designing the interactions can
improve the users ability to deduce how the interaction should be.
As a final note, it is relevant to introduce the term of directness [16].
The term
distance describes the factors which contribute to the feeling of directness. There
is a distance between the goals and knowledge of a user and the interface presented
by a system. This distance can be divided into the gulf of execution and the gulf of
evaluation. The gulf of execution is the distance between the goals of the user and
the commands and mechanisms of the system. The gulf of evaluation is the distance
between the systems output and how the user perceives, interpret and evaluate this.
As 3D modeling is inherently direct and visual in its output, the gulf of evaluation is
less important. The distance of the gulf of execution is however important, as a great
distance will mean that the users cannot figure out how to interact with the system.
This is just a brief introduction to some of the AR HCI guidelines, but more will
be introduced in more specific detail in chapter 6, where the chosen user interactions
will be deduced.

CHAPTER 4
Why use augmented
reality?
AR applications have been described as being ”fun” and they achieve a ”wow” effect
from the users[11]. It can be argued that these experiences are simply due to the
novelty of the technology and interaction. This begs the question of whether or not
AR offers any advantages over more established interfaces.
AR has shown to improve learning in educational environments[10] and enable an
improved social experience compared to that of non-AR computer games[17].
In
terms of collaboration, AR enables communication that is more similar to face-to-
face collaboration than that of screen based communication[6]. Having individual AR
devices also allows for each user to have their own individual view of the augmented
environment. Another major advantage of using AR is that the virtual objects are
positioned in the real world. This provides the benefit of seeing objects in the context
of real objects. This is seen with the construction software company Dalux that uses
AR to merge a digital BIM models into the physical construction site[9]. This allows
construction site workers to asses if the current construction matches that of the
BIM model. These are all examples of how AR can be beneficial, but what about 3D
modeling specifically?
Below are some specific use cases for 3D modeling in AR on a mobile device.
4.1
Architectural models
Building physical architectural models is not as commonplace in the architectural
sector as it was before the advent of computers and 3D modeling tools[8].
The
reason for this is that making physical models is time consuming, expensive and
the incremental changes that a design most often goes through are easier and faster
to implement on a virtual model than a physical one[8]. The physical model does
however provide some benefits over their virtual copy when conveying sense of depth
and dimension. This benefit strengthens the ability to visualize the project - especially
in relation to a client, which are often only presented with pictures of various aspects
of the model. This is where modeling in AR can bridge that gap and possibly benefit
the design process as well as the collaboration with clients. It offers the ability to
sketch a new design or create additions to a physical model. The collaborative benefits
of several people modeling a building together in AR is also a use case in itself.
Another specific use case for architects is in the initial phase of designing a building.
==PAGE:20==
12
4 Why use augmented reality?
In this phase they often start by analyzing the site and create simple volumes for
the placement of the building. This process is often iterative, as these volumes might
not fit the site perfectly the first time the site is visited. Being able to shape these
volumes at the site using a mobile device, could be an optimization of the workflow
and possibly reduce the number of iterations necessary.
4.2
Physical design prototypes
Design is a vast field, but a common practice is to first sketch a design either on paper
or in a 3D modeling program. This could be the furniture designer, the kitchen de-
signer or the carpenter who has to create a unique solution. An underlying similarity
for these cases is that they often design something that should fit into an already
established context. This is where 3D modeling using AR can provide the benefit of
the real world context and the ability to create a new design.
4.3
Leisure and fun
The ability to create and position virtual 3D models in the real world also has the
use case of being fun. Imagine creating a small town on your table, or placing a
homemade piece of art in the corner of your living room. It could also be used as a
tool for setting up an AR tabletop game. Two players could start by building a castle
each, and then the goal of the game would be to invade or destroy the other player’s
castle. For simplifying the 3D modeling, a technique such as used in Cubebuilder[33]
could be used.
In Cubebuilder, simple physical blocks were replaced by detailed
models in an augmented view. Similarly, simple shapes created with the modeling
application could be mapped to detailed models.
Another game worth mentioning is Pokémon Go[26]. It was one of the first popular
AR games for mobile devices, but a large part of its success was also that it was
social and required users to move around in the real world. Combining this aspect
of having to move outside with 3D modeling in AR presents an interesting use case.
Having users create 3D models that fit into real world locations could present a new
dimension to exploring the physical world.

CHAPTER 5
Technology Overview
5.1
ARKit
ARKit is Apple’s AR development tool, which can run on devices with an A9 chipset
or newer and IOS 11 and above. It handles the tracking, which keeps track of the
position and orientation of the device in the physical world, using the sensors of the
device. Furthermore, it offers scene understanding, which makes it able to detect
horizontal and vertical planes. One of the major advantages of using ARKit is that
it runs on the Iphone 6S and newer as well as all Ipad pro models and standard Ipad
models released since 2017. This means that applications developed with ARKit will
be accessible on millions of devices. Another advantage is that it support marker-less
AR, which has been the standard for AR on a mobile device until now. Not having
to use a marker enables a less constrained experience.
5.2
Ipad 2017
The hardware used to test and develop the application at, is the Ipad 2017, which has
a A9 chipset, thereby making one of the lowest end IOS device able to run ARKit. It
has a 9.7 inch multi-touch display, thereby making it a tablet device.

Unity
Unity is a game engine developed by Unity Technologies. It was chosen to create the
modeling tool in Unity as it offers the necessary tools to achieve this, and it was well
known by the author. Unity support scripting in the C# and Javascript languages,
where C# was the chosen language to write the application.

interactions
The interactions possible in AR on a mobile device is firstly made possible by the
touchscreen display, which has been the norm for input on mobile devices such as
phones and tablets for the last decade. Secondly, it is made possible by AR, which
enables interactions based on position, orientation and image based inputs. These
two interfaces can be complementary, but they can also be conflicting, which is why
this chapter will start out presenting the possible interactions using these interfaces.
From these interfaces three different interaction structures were persuaded in order
to minimize the cognitive effort[16] needed to use the application. For each of the
structures, the possible interactions, which can be used for each of the chosen mod-
ification tools from section 2.5 will be discussed and one interaction will be chosen.
Some similarities are present in these interactions, which will be outlined. Focus will
not be on the aesthetics or placement of the graphical user interface nor audio feed-
back, but only the specific interactions needed to achieve a mesh modification. First
the possible inputs using AR on mobile devices will be presented.
6.1
Possible interactions in augmented reality on
mobile devices
Tapping
Tapping is the action of touching the screen with a finger and releasing it again. It
can be used for a button press, but a tap can also be confined to a specific area of
the screen, so that specific options appear, depending on where the screen is pressed.
Repeating taps can also be used for specific functions. A double tap is for example
commonly used to zoom in or out of pictures, but it could also be used to trigger a
specific action for a selected entity or perhaps de-trigger it again. The duration of
time the finger is held on the screen can also determine the action, which is often
used to position the cursor in texts or to delete or rearrange apps instead of opening
them.
Drag
==PAGE:24==
16
6 Choosing the interactions
Drag is the action of holding down a finger on the screen, drag it to a new position
and release it. This is often used to drag an object to a new position, change the
camera view of a scene or move around a map or a photo. It can also trigger a specific
action such as opening a menu when dragging from the edge of the screen. Dragging
with two or more fingers can also be used to trigger additional actions.
Pinching
Pinching is the action of touching the screen with two fingers and either move the
fingers closer or further away from each other. It is often used in applications to zoom
in or out. As AR allows the user to zoom in and out by moving closer or further
away, this could instead be used for other operations.
Rotate gesture
A rotation gesture is performed by touching the screen with two fingers and then
create a circular motion with the fingers. This is often used to rotate maps or pictures,
but could also be used to change a value or rotate an object.
Graphical controls
These are controls based on a graphical interface such as buttons, sliders, steppers,
switches and pickers(scrollable lists). Most of these are often used for setting values
or choosing settings.
Interactions in augmented reality
A prerequisite of AR is that the device is able to fairly accurately pinpoint changes
in position and orientation of the device. This allows interactions to be based on
the input from the six degrees of freedom of the device. This enables interactions
such as dragging an object with the device or use the current view of the scene to
determine possible actions. Using the camera, also enables a tangible AR interface,
where physical objects are used to interact with virtual objects. This has proven to be
an intuitive interface[5], but it will however not be used, as a solution only requiring
a mobile device will be pursued.
Additional controls
Audio can be used to control actions, which widely utilized by virtual assistant such
as Alexa, Google Assistant, Siri and Cortana. It will however not be used in this
project, even though it could be useful for complicated modifications or a series of
modifications.

6.2.1
Entity selection
When box modeling, it is common to first select the entities that should be modi-
fied. Yang et al.[34] used direct touch finger input to select an object on a mobile
device, meaning that the fingers should touch the object being manipulated. Apple
recommends using direct manipulation over separate onscreen controls, as it is more
intuitive and immersive[2], but they are also aware that this can be confusing and
difficult while moving around. There are two fundamental drawbacks to direct touch
finger input, which is the occlusion of the object being touched and the size of the
finger being possibly bigger than the object being touched(fat finger problem). These
two problems can result in the user being unable to exactly pinpoint the point of
==PAGE:26==
18
6 Choosing the interactions
touch[29]. Using touch in AR on mobile devices can also cause unwanted camera
motion, which can have a negative influence on the user experience[21]. Bai et al.[4]
tried to use a method where the screen could freeze to omit this problem, which was
found to be useful and natural, but it was ranked lower in terms of fun and engage-
ment. They also tried to use the finger in the camera view to interact[4], but this was
not found as easy or intuitive as using direct touch. A related problem is that the
use of one hand when holding a tablet causes fatigue[28]. Apple also states in their
ARKit guidelines that it is difficult to precisely touch specific points on an object[2]
and therefore recommends that it is best to assume that the closest interactive point
is the one the user tries to select. However, as entities can lie close on a mesh, it might
not always be sufficiently precise to assume what is being touched. The ability to
clearly see the modification of the mesh in real time is also important when modeling.
These studies supports that there are problems when using direct touch in mobile
AR, so in this project interactions that are possible while holding the device with
two hands at all times will be persuaded. It is here important to mention that the
modeling tool was developed and tested on a tablet, which further strengthens this
choice.
So even though many mobile AR studies and AR applications utilize direct touch,
it was chosen not to use this as the precision, user experience and ability to see
the object is important when modeling. This does contradict the advantages of using
the directness[16] of direct touch, but the disadvantages still makes this a valid choice.
The solution was to use a cursor, which in practice works as a ray starting at the
position of the camera with the direction the camera is facing. This means that the
point of interaction will always be in the center of the screen. To indicate where the
cursor is, a graphical element will be visible on virtual surfaces. To select an entity,
the cursor is pointed towards it and when within a certain distance it will be selected,
which also changes the color of the entity. This method was used by Henrysson[14] for
selection in mobile AR, but he did not test it. This selection method has commonly
been used in controller-less VR and AR applications using a head-mounted display
where selection would be triggered by either using a touch interface on the headset or
by gaze-dwelling at a interactable objects[12]. The cursor selection method was used
in all versions.

While being able to select an entity with the cursor, the user needs controls to select
how to modify the entity. As stated in the previous section, it was chosen to focus on
interactions possible while holding the device with two hands, so the controls should
accommodate this. Apple recommends to avoid cluttering the screen with controls, as
it will provide a more immersive experience[2]. This is also reflected in the principles
about only presenting relevant information mentioned by Kourouthanassis et al[19]
and Dünser et al.[11].
However, hiding the controls to access a modification tool
could also become frustrating as these will be frequently used. Because of this, two
different solutions were chosen to pick a modification tool.

6.2.3
Translation
Device version
Henrysson[14] tried several interactions for translating an object. Three of the options
required the user to release one hand from the device and will not be taken into
consideration. The remaining options include physical buttons, the orientation of the
device and movement in relation to the device. He found that translation in relation
to the device was the best overall solution, but that the physical buttons offered
more precise translation. Translating the object by tilting the device was found not
competitive. Hürst et al.[15] also found that translating an object in relation to the
device was faster than moving it with touch or using a finger in front of the camera.
This solution can also be related to the way entities are translated using controllers in
Google Blocks, which on mobile AR is mostly equivalent to translating entities using
the device. This should also give an experience of directness, as the user grabs the
entity with the cursor. However, Some usability aspects of using a six degrees input
device are presented by Zhai[35].
These include speed, accuracy, ease of learning
coordination and fatigue. These can have a negative influence on the interaction and
are important factors to take into consideration.
The device version will therefore translate entities in relation to the position and
orientation of the device, while a button is being held down.
Cursor version
In desktop modeling software a translation modification widget is often used to
move an entity in 3 dimensions, but other interactions are also commonly used. In
Blender[7], the user can also move an entity on the plane defined by the position of
the entity and the view plane of the camera by using the mouse. In Skethcup, the user
can move an entity on the ground plane using the mouse or hold down a combination
of keys to lock the movement to one of the world axes. These methods of locking the
movement to a plane, either requires the user to be very aware of the starting point

dragging left or right on the screen.
6.2.4
Extrusion
Extrusion is often triggered by a single input such as a key or button press as in
Blender[7] and is followed by a translation. This will also be the case in this project
for all versions with some minor differences. Hutchins et al.[16] found that a direct
interface tend to put the responsibility of precision on the user. In terms of moving
and extruding a faces, it is often intended to be along the normal of the face. In VR or
in AR, where controls are often based on the users movement, it is difficult to create
perfect movement along the normal of a face. This is not an issue in the cursor and
gesture versions, where a translation widget is used to contrain the translation to one
axis. In order to alleviate the problem in the device version, the face translation will
initially be locked to be along its normal, but when exceeding a distance threshold
from normal, it will snap out of the lock.

Device version
Henrysson[14] tried several interactions for rotating an object in AR. The proposed
interactions that are possible while holding the device with two hands are using but-
tons, using an arcball, rotation in relation to the rotation of the device and continuous
rotation by tilting the device in the desired direction. He found that buttons and the
arcball were the best solutions. Rotation in relation to the rotation of the device was
tried during this project, but quickly rejected. The reason was that when trying to
perform a complex rotation, it required the user to either move around a lot or rotate
the device such that the face being manipulated went beyond the camera view. As
we are looking for a rotation based on orientation and position, the tilting interaction
was chosen.
The device version will use tilting of the device to control the rotation of a face.
As it is desired to be able to see the manipulated face while rotating it, an orientation
threshold will trigger the rotation and continuously rotate the face. The rotation is
stopped if the device is rotated back to the start rotation or the button is released.
Cursor version
In Blender, the rotation of a face can be controlled by the position of the mouse
by moving it either to the left or right. It is however constrained to rotate around
the viewing axis, so this will require the user to be aware of the point of initiating
rotation and would require the user to move around the object several times in order
to achieve a complex rotation. Another commonly used tool to rotate an object or
entity in most desktop modeling software is using a rotation widget. This is very
similar to the arcball proposed by Henrysson[14], but the difference is that individual
axes can be chosen, which offers greater precision. Using the rotation widget can
reduce the learning needed to use it[11] and utilizes the term of affordances as also
mentioned in translation for the cursor version.
The cursor version will use a rotation widget to rotate a face. To rotate around
an axis, the user hovers over the respective rotation ring, holds a finger down on the
screen and rotates around the center of the face with the cursor.

arcball could work. In order to be able to pick any axis of rotation, the 2D interface
is divided into two controls. A wheel will control the rotation around the normal of
the face, while dragging left/right or up/down inside the wheel will rotate around the
remaining axes. This method could alleviate some of the usability aspects of using

8.1
Probuilder runtime API
This section will present how the probuilder API was used to create some aspects
of the modeling tool. As probuilder is a package for Unity, Unity specific methods,
objects and tools were also extensively used.
The UnityEngine.ProBuilder namespace contains all the necessary data types used
for modifying a mesh such as classes representing vertices, edges and faces. It also
contains the ProBuilderMesh class, which is required on each gameobject that is ma-
nipulated in runtime. This class is responsible for storing all the data necessary for
editing unity mesh objects. The UnityEngine.ProBuilder.MeshOperations namespace
was also used for mesh operations such as extrusion. Some of the specific implemen-
tation is explained below.
Entity selection
To find and highlight the entities hit by the ray starting at the camera position and
in the direction the camera is facing, a lookup is first done in the Unity mesh. The
Unity engine struct ’RaycastHit’ provides information about what the ray has hit,
which makes it easy to get the triangle indexes of the current triangle being hit by
the ray. Using these indexes, the face containing the triangle can be found in the

CHAPTER 9
Method
9.1
Test
The goal of the usability test was to investigate how people experienced the chosen
interactions. This was done by measuring various factors such as subjective experi-
ences of intuitiveness, ease of use and engagement and enjoyment. Additional factors
such as the time to complete a task can also influence these factors and was therefore
taken into account. These measures was used to investigate the advantages and dis-
advantages of each interaction. Inspiration was found in the usability test performed
by Bai et al.[4]. An assessment of the tool selection interface was also conducted, as
it is different in the cursor version compared to the other versions.
9.2
Test setup
A test of three different manipulation tools(translation,rotation and scaling), each
having three different interaction versions was conducted. A special version of the
modeling tool was created for each test, so the participants were only able to use
one modification tool using one of the interaction versions at a time. In the test, the
participant tried all three interaction versions(Device, Cursor, Gesture) for each mod-
ification tool. The task was to manipulate a cube, so it would align with an already
manipulated wireframe cube (see figure 9.1). Each wireframe model had been made
so it could be recreated using two operations with the given tool. The sides of the
initial cube was approximately 7 cm in length, thereby trying to recreate a scenario
where precision is a factor. The setting of the test changed between most tests, as
the mobile device was brought to the test participant.

left to right: Translation, Rotation, Scale..

9.3
Participants
The test was conducted on 25 participants. Each participant tested at least all three
versions of one tool, whereas others tested two or all versions of all tools. This resulted
in data from 15 participants for each tool. 7/15 in each modification tool group had
never tried 3d modeling before. The reason for also using participants without any
prior knowledge of 3D modeling is that it can be argued that the common denominator
is still the best interaction, whether you are operating in a known domain or not. It
will also offer the possibility of investigating whether or not foreknowledge influences
the experience of an interaction.
9.4
Procedure
The Ipad was given to participants running a version with only one kind of manipu-
lation and one version of interaction. For example a version where only translation
was possible with the device interaction. They were then instructed on how to use
the tool to manipulate the mesh and tried it for themselves a couple of times in order
to reduce the interference effect from the previous versions[23].
They were then instructed that they should replicate the wireframe model, with the
tool they were given. They were informed that this task was timed, and that the task
was finished when they were satisfied with their model.
After trying the first version, the participants rated their experiences on likert scales[22].
After this they were introduced to a new interaction version, using the same manipu-
lation tool. This process would repeat until the participant had tried all interaction
versions. After the last likert scales a small interview was conducted, in order to get
them to rate the three versions in terms of which one they would prefer and elaborate
on why they chose as they did.
9.5
Experimental design
A within-subjects design[23] was used in the experiment.
Each participants tried
three different interactions for each manipulation tool. The order of the versions was
changed after each test in order to reduce the learning effect[23]. Some users only
tested one manipulation tool, while others tested all tools.
9.6

Each test was timed in order to pinpoint if there is any difference in productivity.

strongly agree) were used to asses how they experienced the interactions and their
experience of the precision of the model they created. Subjective data was collected
through a semi-structured interview, which was conducted after they had tried all
interaction versions of a particular manipulation tool.
The answers were written
down as notes. Furthermore the participants interaction with the application was

CHAPTER 11
Discussion
In this chapter the chosen and tested interactions will be discussed, but as the imple-
mentation and test method heavily influenced the final assessment of the interactions,
these will also be discussed.
11.1
Implementation
To highlight an entity, the user had to point the cursor within the mesh of the virtual
model. A lot of participants stayed right at the border of the mesh in order to select
a vertex. Because of the inaccuracy of the tracking and the users inability to hold
the tablet completely steady, this resulted in the vertex appearing and disappearing
as they tried to select it. This led to some frustration. The solution is to make it
possible to select edges and vertices even though the cursor is outside the mesh.
When translating in the gesture version, left/right drag was also used for translat-
ing an entity up/down along the y-axis. This issue was caught too late, but should
be fixed, as it can have worsened the experience of this version.
Other improvements, such as increasing the collider size of the rotation widget and
making the translation widget semi-transparent, are also important.
All of these issues have worsened the experience using all or some of the interactions,
which have had an negative influence on the ability to compare them.
11.2
Test
One major point to criticize is having included two different selection methods. The
interactions for modifying a mesh were the independent variables in the test. How-
ever, the difference in selection method between the cursor version and the others
might have had either a negative or positive influence on the experience of the mesh
modification interaction. One fact is that the cursor selection method was slower,
as one additional step was required. Most users also preferred the other versions.
A better test would have separated selection method from the mesh modification in
order to avoid any influence it might have had on the experience.
The interference effect between the versions was noticeable during some of the test.
It was mostly noticed between versions that had fundamentally different interactions,
such as translation in the gesture version, where users had to drag on the screen
==PAGE:58==
50
11 Discussion
instead of moving the device. It was also consistently noticed between the cursor
version and the others, as the method of choosing a modification tool was different.
Either they could not find the button at the side of the screen, or they tried to tap
anywhere on the screen, when they had to press the button on the side. This could
have been avoided by giving the participants more time to try out each version, or a
between-subject design could have been used. Using this design would however leave
out the option of having users comparing the versions.
The setting has also had an influence on the results. The tables on which the test
was conducted was of different sizes, which meant that some participant had the ad-
vantage of quickly being able to walk around the model, while others had to move
a greater distance. This has influenced the time taken to complete the task, and it
might have caused the versions where movement is needed to be rated lower than the
others. The setting also influenced the distance to the model, and as being closer
reduces the amount of precision needed, this might also have had an impact on the
result. For an optimal test, the setting should be the same for each participant.
Not having any statistical tests because of the small sample size made it difficult
to interpret the likert scale results. A larger sample size is desired in order to omit
this problem.
Bias was also introduced by having participants with different knowledge of 3D mod-
eling. Those that had tried it before often preferred the tools they were used to. Some
participants also cared about precision, which often led them to spend more time on
the task and prefer the versions that offered increased precision. Having a test group
from a specific demographic could reduce the bias, but this would in turn only show
which interactions that specific group prefers.
A different understanding of the likert scale questions also introduced some bias.
One user, who was an experienced modeler for example ranked the rotation widget as
not being intuitive. His understanding of intuitive is that it should be something that
anyone would be able to figure out by themselves, without any prior knowledge. Most
others had a subjective understanding of intuitive meaning that tools they had used
before were regarded as intuitive. In order to avoid this problem, words like intuitive
should have been elaborated, such that everyone would have the same understanding
of the word.
Another aspect to criticize about the likert scales, is that it is based on subjective
assessment. This is a valid tool for measures such as fun and engagement and ease of
use, as they are difficult to quantify. Precision could however have been measured,
which could give a more valid foundation for comparing the versions.
Measuring
the exact distance from the wireframe model was however beyond the scope of this
project, but could be a useful tool for future tests.

11.3
Interactions
Testing the chosen interactions provided some valuable insights into how they were
experienced by the users. Advantages and disadvantages were found, which will be
discussed here.
Translation
In terms of the translation tool, it was found that precision was one of the biggest
differentiators between the device and the other versions. It was found that the dif-
ficulty of perceiving depth was the main problem. To alleviate this, the translation
could be constrained, as was done using the translation widget, or visual cues could
be used more extensively. A visual cue could be a semi-transparent grid, so the user
would better be able to pinpoint the position in relation to other objects in the scene.
Other aids could also be implemented such as translation in incremental steps on
each axis or context aware snapping to a position.
The lack of precision in the device version did however not restrain users from prefer-
ring this version. The fact that it was generally ranked higher in all other measures
cements the fact that most users prefer an interaction that is quick and intuitive. So
the device translation did achieve directness[16]. However, because of the usability
issues of using the 6 degrees of the device as input[14], a lack of precision was seen.
Further testing would probably also show usability issues in terms of fatigue and co-
ordination.
There is however another side to the preferred version.
The results also showed
that people with knowledge about 3D modeling actually preferred the other versions.
These versions were preferred because the users knew the translation widget, which
ties directly to the use of known tools and affordances to enhance the learnability
of an interaction[11][27]. So using a known tool will directly impact the experience
of the user. Hutchins[16] also states that frequent use of an interface can result in
a learned directness, but he also states that it is important to distinguish between
learned directness and the directness of the interface itself.
Another reason could be that the users were used to the precision of the widget and
perhaps their use of 3D modeling actually depends on precision and not how fast or
intuitive it is. This would make them prefer the widget, as it is the best tool for
their specific usage. Some users, who all worked in either construction or product
development where precision is essential, preferred the gesture version over the cursor.
This version offered directness from the drag, without the problems of a 6 degrees
input device. It was however less consistent in its interactions, as users would have
to switch from using the cursor to select an entity and then use drag on the screen.
This inconsistency reduced the learnability of this version[11].
To sum up, the preferred translation interaction depends on what you are used to
and what you are using it for. The device version offers a quick and intuitive way
of translation, but with reduced precision. The cursor version offers precision and
an intuitive way of translation, but with more steps, thereby making it slower. The
gesture version offers the best precision, and the advantage of not having to move the
device as much, but it is less intuitive and slow. In an ideal 3D modeling application
==PAGE:60==
52
11 Discussion
all tools should be available.
Rotation
The known widget of the cursor version and the precision, and speed of the gesture
version made these the preferred versions. The gesture version was the most preferred,
because of the directness of the input and the advantage not having to depend on the
imprecision of the device’s 6 degrees of freedom. Another reason was, that the users
not familiar with the rotation widget did not know how to use it. So in a test group
only consisting of users familiar with 3D modeling, the cursor version could be the
most preferred. This shows a tendency similar to that of the translation tool, where
a known tool enhances the learnability[11][27] and results in a learned directness[16].
The cursor and gesture versions both had some implementation issues. These include
the collider size of the rotation widget in the cursor version and the graphical repre-
sentation of the interface in the gesture version. Increasing the collider size of the
rotation widget could possibly improve the cursor version’s ease of use and the use-
fulness to complete the task. Improving the graphical representation in the gesture
version could reduce the confusion surrounding the interaction needed for a specific
rotation. Perhaps showing a representation of the face in the center of the interface,
could provide some cues on how to interact. This project did not focus on aesthetics,
but in relation to this particular interaction, it would have been improved from it.
In terms of the device rotation, it was not found to be competitive, similar to the
results of Henrysson[14]. Some users however regarded it as the most fun and immer-
sive, so it might find its usage as a game related rotation.
To summarize, the gesture version has the potential to be an all-around decent tool,
whether you are familiar with 3D modeling or not. The cursor version is only valued
by those familiar with 3D modeling, which makes it an obvious choice if the applica-
tion is targeting this demographic. The device version is less useful for 3D modeling,
but could find other use cases.
Scaling
The results regarding scaling showed almost unambiguously that the gesture version
was the preferred version. This can again be attributed to the fact that the inter-
action of pinching has become the most commonly known gesture for zooming in or
out. The only disadvantage of using this gesture is that it requires two fingers for
the interaction. This was not an issue in this application, as all interactions were
designed to be possible while holding the device with two hands. If an application
requires one hand free, then the other versions are more suitable. If comparing the
cursor and device version, then the device version scored highest in terms of most
measures, and it was ranked as the second best by most users. It however had one
drawback of requiring slightly more movement than the cursor version.
Tool selection
Two solutions for selecting a tool were presented to the users, from which using but-
tons on the side was preferred. They however each had their own benefits. The main
difference is whether you value to reduce cognitive overhead by keeping the users
focused on the task[11][19] or reduce the physical effort by having less steps[11]. In

this case less steps won, but if the primary focus of the application had been to in-
spect a model, then an argument could be made for the 3D buttons. A third solution
was proposed by several users and also by the supervisor of this project, but it went
beyond the scope of the project. The solution was to have a tools menu, where the
chosen tool would be the one used until another one was chosen. It could function
like a selection wheel on the left side of the screen. This solution would be preferable
if often using the same tool for several modifications, but if switching between tools
is a common action, it would probably be preferable to have them all available at all
time. Using a tools menu could also reduce the graphical interface thereby reducing
the cognitive overhead.
Additional topics
Fatigue has not been a focus of this project and the problem did not arise during the
testing. It is however bound to become an issue for longer periods of use. The device
and cursor versions generally require more movement because movement is required
for the modification itself. So if fatigue is an issue, the gesture version’s interactions
could be preferable. Another solution to decrease fatigue is to make it possible to
rotate the entire model. In AR the user can walk around the model, but at some
point the novelty of this interaction will disappear, and the most practical approach
will be desired.
The solution of using a cursor to select entities did offer an alternative to direct
touch, but without the disadvantages of obtrusion and unwanted camera motion. As
the tracking gets better with new hardware and software solutions its precision is
also poised to improve. The cursor was however not tested against direct touch, so
even though the solution works well, it cannot be confirmed to be the best for 3D
modeling in AR on mobile devices.
A related topic of the cursor is that of using the 6 degrees of freedom of a device as
input. This was utilized by the cursor, thereby making both the device and cursor
versions dependable on the properties of this input. This resulted in some issues, with
precision being a major one. The imprecise nature of these versions can be attributed
to the tracking in ARKit and the user’s ability to hold the device steady. The track-
ing using ARKit can and will be improved, but a shaky hand probably won’t. The
gesture version did not suffer from this, so using the touchscreen will inherently be
better for precision tasks. It however has the disadvantage of having to switch be-
tween input methods from the cursor to the touchscreen.
This brings up the choice of choosing a structure for each version. As the gesture
version used two structures, it is possible it did not obtain the benefits of a single
structure approach. Perhaps it should have used direct touch for selecting entities.
However, no structure was regarded best in terms of all tools. So the question is if
having a structure is more valuable than the experience of the individual interactions.
It might come down to the complexity of the interactions. If they are complex a
structure might help improve learnability. If they are simple, it might be better to
choose the best interaction. Having multimodal input might also be an advantage,
as it can increase user performance[11]. In this project, users did not seem to have
any problems learning a new interaction structure in the matter of seconds. Only the
way the interaction was translated to a modification was a problem. So an argument
==PAGE:62==
54
11 Discussion
is here made for not having to follow a structure. The best tools in this project was
showed to be from different structures and following these results could potentially
be better than forcing through a structure. This can however not be proved without
a test of versions with structure and without, each having all tools available.
In the ideal 3D modeling application using AR on mobile devices interactions can
be changed to fit each user’s preference. It might be a benefit to choose the interac-
tions that give users the best experience as default. Experienced 3D modelers will
probably figure out how to change the interactions, while newcomers might experi-
ence their first time modeling as being a pleasant experience.
==PAGE:63==
CHAPTER 12
Future work
Besides some of the implementation issues mentioned in the discussion, this project
could have benefited from further testing. Future tests could benefit from a different
demographic. An interesting group could be that of architects as also mentioned in
chapter 4. Using a 3D modeling application in AR on a mobile device could prove
beneficial for architects and investigating which interactions they prefer, could help
specify the optimal interactions for an actual use case.
Another test could also investigate issues such as fatigue, in order to find solutions
for this problem. Further testing is in general the best method to pinpoint problems
or benefits and advance the use cases for 3D modeling using AR on mobile devices.
Even though this project has investigated some of the possible interactions for 3D
modeling, there are still several other ways. These interactions should be investigated,
in order to further develop the most optimal way of interacting. This can lead to a
toolbox tailor-suited for any domain of 3D modeling and the preferences of each in-
dividual user. Additional tools should also be investigated. 3D modeling on desktop
pc’s offer an extensive number of tools, and future work could focus on bringing more
of these to a mobile device using AR.
Another important aspect of 3D modeling in AR on mobile devices is to investi-
gate whether or not it offers any benefits over other established interfaces. As this
project focused on how 3D modeling can be done, the far more important question is
whether or not it should be done and why.

CHAPTER 13
Conclusion
The goal of this project was to investigate interactions for 3D modeling in AR on
mobile devices. Three different versions of a 3D modeling application were created
with different interactions for the tools used for translation, rotation and scaling. A
usability test of the interactions provided valuable insights into the advantages and
disadvantages of each interaction.
By testing on users with different knowledge of 3D modeling, it was found that the
preferred interaction is heavily influenced by the use case of the user. These insights
will be able to help determine which interactions fit different needs, and thereby help
avoid any pitfalls when developing a 3D modeling application using AR on a mobile
device for a specific use case. Two tool selection methods were also tested, where it
was shown that users preferred 2D buttons and a minimum number of interaction
steps. The most optimal solution is however to offer several different interactions in
order to satisfy a wider range of preferences.
It can be argued that AR on mobile devices is just a step in the development of
creating cheap and functional head worn AR products. This intermediate step is non
the less important as millions of mobile devices currently supports the technology of
AR. Investigating how to interact with this interface can lead to applications that
are capable of much more than what is currently possible using only the touchscreen
to interact. This project contributes to this research of figuring out how interactions
when 3D modeling should be in AR on mobile devices. In this project, only a few
interactions were implemented and tested from the possible tens or hundreds of pos-
sible interactions. This can however still help solve the next question of whether or
not AR on mobile devices can contribute to the domain of 3D modeling.
