Summary (English)
The indexing problem consists of preprocessing a string S of n characters into
a data structure such that subsequent searches for a pattern string P of length
m can be done in o(n) time. In the packed setting the strings S and P store
α characters per machine word and therefore α characters can be read and
modiﬁed in constant time. In this thesis we provide a new solution to both the
classic indexing problem and the packed indexing problem, with our centroid
suﬃx tree and packed centroid suﬃx tree respectively.
Our solution in the non-packed setting occupies O(n) space, takes O(n log
σ
σ−1 n)
time to construct where σ is the size of the alphabet of S and P. It has a
O(m log
σ
σ−1 n) worst-case query time and O(m+log
σ
σ−1 n) best-case query time.
It is found to be slightly slower than the textbook solution, the suﬃx tree, in
practice.
Our solution in the packed setting takes O(αn) space and has the same construc-
tion complexity as the non-packed solution. It has a O( m
α log
σ
σ−1 n) worst-case
query time and O( m
α + log
σ
σ−1 n) best-case query time. We compare it to the
classic suﬃx tree as well as state-of-the-art solutions and it is found to be faster
in practice, at the cost of more preprocessing time and space.

Preface
This thesis was prepared at DTU Compute in fulﬁllment of the requirements
for acquiring an M.Sc. in Engineering.
The thesis deals with the indexing of packed strings. Indexing a string means to
preprocess it into a data structure such that subsequent searches for substrings
can be done in sub-linear time, with respect to the size of the indexed string. A
packed string is a string where one can access and manipulate multiple characters
at a time because they are stored within a single machine word.
The thesis consists of both a theoretical and practical solution to the problem.
A solution to the problem is developed, which is analyzed theoretically and its
correctness is argued for. The solution is also implemented, benchmarked and
compared to other state-of-the-art solutions.

Chapter 1
Introduction
1.1
Problem deﬁnition
Given a string S consisting of n characters from an alphabet Σ of size σ = |Σ|,
the string indexing problem is to create a data structure that can answer the
following queries for another string P of length m:
• Exists(P): Does P occur in S?
• Count(P): How many times does P occur in S?
• Report(P): Where does P occur in S?
A string is packed if it stores α characters per machine word and allows for
accessing and comparing α characters in constant time. The packed string in-
dexing problem is to create a data structure that can answer the string indexing
queries for packed strings S and P.
If we were to simply compare the two strings S and P, it is easy to envision
that we could perform such a comparison α times as fast in the packed version.
For the more complicated indexing problem it is not as easy, but it should be
possible to speed up the search.
==PAGE:14==
2
Introduction
String indexing is a very fundamental problem and has applications in many ar-
eas of both research and industry. As modern computers are typically equipped
with either 32- or 64-bit processors and text characters often occupy much less
than that, e.g. 8 bit for ASCII or 16 bit for Unicode, solving the packed problem
can often yield signiﬁcant improvements in query time. It is especially interest-
ing in the area of computational biology as DNA can be represented in only 2
bits (with letters "ACGT") or 4 bits (for an extended alphabet) per character.
The aim of this thesis is to provide a solution that can answer queries very fast,
without worrying too much about construction time or space usage, as long as
its still near linear. Furthermore we aim to focus on the practical speed, allowing
for worse complexities if the implementation still performs better.
1.2
Contribution
This project oﬀers a new solution to the packed string indexing problem, the
centroid suﬃx tree. The data structure is a novel combination of techniques from
other, related problem domains and new algorithms and techniques. We begin
by oﬀering a new solution to the non-packed string indexing problem, which
while having a slightly worse time complexity than the standard approach, is
almost as fast in practice. This solution is then brought into the packed domain,
resulting in a ﬁnal structure with query times that scale linearly with α and
perform very well in practice. All theory put forth is implemented and tested.
The downside to the new solution is the time it takes to build the data structure,
as well as the space it consumes. While both scale linearly with input size, the
same as the standard approach, the constants are rather large, bordering on
prohibitive in practice.
1.3
Related work
The simplest way to solve the packed string indexing problem is to treat it
as a classic string indexing problem. Textbook solutions to this problem are
the suﬃx tree and suﬃx array. Suﬃx trees have linear space usage and pre-
processing time and support queries in O(m log σ) time, where m is the length
of P.
Suﬃx arrays, if coupled with arrays storing selected longest common
preﬁxes[MM93], achieve similar space and preprocessing bounds and support
O(m + log n) queries. Allowing for randomization, coupling the suﬃx tree with

perfect hashing can achieve O(n) expected preprocessing time, O(n) space and
O(m) queries.
Recently Bille, Gørtz and Skjoldjensen[BGS16] showed that in the determinis-
tic and packed setting it is possible to achieve linear preprocessing time and
space and O(m/α + log m + log log σ) time queries. This solution has not been
implemented however and is suspected of being too heavy to perform well in
practice.
Takagi et al.[TISA17] proposed a data structure called the packed c-trie that
solves the related packed string dictionary problem, where the goal is to prepro-
cess a set K of string keys to allow queries for the keys. Their stucture takes
O(n log σ + log n) space and allows for queries in O( m
α + f(k, n)) expected time,
where f(k, n) is the query time for an arbitrary dynamic predecessor / succes-
sor structure over k values from the universe [1; n]. E.g. coupling it with y-fast
tries[Wil83] the packed c-trie could achieve O( m
α + log log σ) expected query
time. The packed string dictionary problem is not exactly our domain, but it is
related and illustrates how packing can speed up query time.

1.5
Preliminaries
In this section we lay out the notation and writing conventions of the thesis.
The word "we" is used to describe either the author (singular) or the author
and reader together, depending on the context.
As all our solutions work with strings, we here describe how we denote strings
and common string concepts.
A string S is a sequence of characters S =
c1, c2, . . . , cn and we denote length of the string |S| = n.
We let S[i] = ci
denote the i’th character of S for 1 ≤i ≤n and S[i; j] = ci, . . . , cj denote a sub-
string of S for 1 ≤i ≤j ≤n. A preﬁx of S is any substring S[1; i] for 1 ≤i ≤n.
We say that a string P of length |P| = m preﬁxes S if S[1; m] = P and m ≤n.
A suﬃx of S is any substring S[i; n] for 1 ≤i ≤n. We denote the empty string
ϵ, i.e. |ϵ| = 0. For any two strings A = a1, . . . , an and B = b1, . . . , bm we denote
their concatenation AB = a1, . . . , an, b1, . . . , bm.
We denote arrays exactly like strings. For an array A of length n we let A[i] for
1 ≤i ≤n denote the n’th element and A[i; j] denote a range.
Many of our solutions require manipulation of the bits of words or characters.
We use the standard bit-wise operations: AND (&), OR (|) and exclusive-or /

Chapter 2
Theory
In the following chapter it is described how we solve the packed indexing prob-
lem. We do this in steps — starting with a simple solution to the non-packed
indexing problem and then gradually improving until we reach the ﬁnal solution
for the packed indexing problem. The order of these solutions is the order in
which they where discovered and implemented. The search for the next area of
improvement was at all times primarily guided by proﬁling of the implementa-
tion.
2.1
Solution 1 - Suﬃx Tree
The ﬁrst solution we discuss is one of the textbook solutions to the (unpacked)
string indexing problem, the suﬃx tree[Wei73]. It is important to us, as it is
at the heart of our later solutions. As the structure is both relatively simple
and very well-known, we will suﬃce with a brief explanation of how it works,
which variant we use and the space and time complexities that matter to us. For
the same reason we omit any proof of correctness and refer to literature[Gus97,
Chapter 5].
Let S = c1, c2, . . . , cn be a string over an alphabet Σ, where the last character

cn = $ ̸∈Σ is a special character. The special character is lexicographically less
than all the other characters, i.e. $ < ci, 1 ≤i < n. The suﬃx tree TS of S
is then the compact trie[CLRS09, Section 12-2] over all suﬃxes of S. Compact
means that all non-branching paths in the trie are collapsed to a single edge
and the edges store their labels with indices into S. We call such indices label

With this structure we can query for a pattern P. Common for all the queries is

This structure occupies O(n) space. A suﬃx tree (compact) over a string of
length n has O(n) nodes and at each node we store a constant amount of data.
For every non-root node there is exactly one edge and each edge also stores a
constant amount of data.
The construction takes O(σn2) time, where σ is the size of the alphabet of S.
For every suﬃx we insert we ﬁrst traverse the tree, possibly break up an edge
and then ﬁnally insert a leaf. Each suﬃx is at most n characters and thus can
do at most n comparisons on the way down or visit n nodes. At each node it
must consider at most σ edges to determine the next edge. Thus each traversal
takes O(σn) time. Both the breaking of an edge and the insertion of a leaf takes
constant work, resulting in O(σn) time per suﬃx. As there are n suﬃxes, the
total work is O(σn2). The ﬁnal traversal to update leaf counters takes O(n)
time as every node is visited once. This results in a total of O(σn2) time.
For existential and counting queries we just need to traverse to the end of
P, which takes O(m) time, where m = |P| is the length of the pattern. For
reporting queries we need to do an additional traversal of the subtree, which in
the worst case is O(n2) nodes and time, in the case that m = 0 and thus P ends
at the root. We therefore have O(m) worst-case execution time for existential
and counting queries and O(m + n2) for reporting queries.

2.2.3
Fingerprints
To understand the next improvement, we must introduce the concept of ﬁnger-
prints. A ﬁngerprint of a string is essentially a hash value that is calculated by a
rolling hash function. Let S denote a string of length n. Let h(S, i, j) be a hash
function that calculates a hash value for the substring S[i; j], 1 ≤i ≤j ≤n.
We say that h is rolling hash function if it can calculate the hash h(S, i, j + k)
from only h(S, i, j) and S[j; j + k], where j + k ≤n. That is, it can extend
(or shrink) the "window" of the string it hashes, without re-calculating what is
already hashed.
While there are many rolling hash functions, the one we use was ﬁrst introduced
by Rabin and Karp in 1987[KR87] as part of their now famous string searching
algorithm. We will dub the values they produce "Karp-Rabin ﬁngerprints"1

2.2.4
Centroid suﬃx tree
Now we are ready to introduce our complete solution, which we name the cen-
troid suﬃx tree or simply CST. For a string S we let CSTS denote the centroid
suﬃx tree of S. A CST is a CT that utilizes ﬁngerprints to improve the traversal
time.
For a string S of length n, the CSTS contains the full centroid tree CTT of S
along with a ﬁngerprint array FS. The ﬁngerprint array F stores the ﬁngerprints
of all the preﬁxes of S, such that FS[i] = H(S, 1, i) for 1 ≤i ≤n. Additionally
it stores the preﬁx of the empty string, i.e. F[0] = 0. The structure of FS is
such that FS[i] is the ﬁngerprint of the preﬁx of S of length i. The tree also
augments every node in CTS with the ﬁngerprint of its label. That is, for every
node v ∈CTS with label-indices (iv, jv), it stores fv = H(S, i, j).
The structure is built by ﬁrst calculating FS.
Beginning with FS[0] = 0, it
calculates FS[i] from FS[i −1] and S[i] for 1 ≤i ≤n. This is done in a single
pass of S.
Then it constructs CTS as already described, but at every node
constructed it stores the ﬁngerprint of the label. For every node v ∈CTS with
label-indices (iv, jv), it calculates fv = H(S, i, j) from FS[i] = H(S, 1, i) and
FS[j] = H(S, 1, j), as described in Theorem 2.11.
With this we can perform queries for a pattern P.
This is done much like
in the centroid tree, but we use the ﬁngerprints to speed up the preﬁx check.
Before traversal begins, we calculate the ﬁngerprint array FP of P, exactly like
we calculated FS for S. We also initialize a counter s = 0, that tracks string-
depth of the last node found to preﬁx P (this is similar to the s during CTS
construction). Then the traversal begins, at the root as before. At every node
v ∈CSTS we still check if str(v) is a preﬁx of P, but we do it as follows. Let
kv = |str(v)| denote length of the label of v. Let (iv, jv) denote the label-indices
of v. We then calculate fP = H(P, iv, jv) from FP [iv] and FP [jv]. If fP ̸= fv
then str(v) is not a preﬁx of P and we traverse left. If fP = fv then str(v) might
be a preﬁx of P (with high probability). To make sure we verify with a direct
comparison, i.e. if S[iv + s; jv] = P[1; kv]. If it is, then we set s := s + kv and
traverse right. Otherwise we traverse left. When the traversal terminates there
are again two cases. If we fully matched P to a node then we answer queries like
we did to achieve Theorem 2.8. Otherwise we must check if P ended on an edge
in TS, as we did before. If w = C[(vlast), P[s + 1]] is present in the dictionary,
we check if P is a preﬁx of str(w). First we compare if H(P, 1, m) = FP [m] is
equal to H(S, iw; iw + m). If they are not equal then we report that P does not
occur in S. If they are equal then we directly check P = S[iw + s; iw + m] and
output accordingly.

Let S again be a string of length n over alphabet Σ. The algorithm is built on

a leaf. The implicit suﬃx tree Ti therefore contain all suﬃxes S[1; i], . . . , S[i; i]

a leaf. As we will see, if the string is terminated with a unique character, i.e.

2.5.3
Time analysis
We now argue for the execution time complexity of the algorithm. For a single
phase i the following happens. Steps 1-3 take a constant amount of work. Step
4 can take σ work to locate an edge for the active point. The same goes for
the leaf in step 5. Step 6a takes at most σ work to ﬁnd an edge. In step 6b we
potentially traverse down the tree to update the active point to ﬁt with the new
la. In the worst case we can have la = Θ(r) = Θ(n) in the case that all suﬃxes
up to the last were implicit. However, then this step only happened once (in the
last phase) and it will take O(σn). As i + r only ever increases, that is we only
insert suﬃxes from left to right, then over the course of the entire algorithm
this takes O(σn). While step 7 can provoke repeats of steps within a single
phase, that only happens when an earlier phase added an implicit suﬃx and
immediately ended (step 4). So by the same argument, as we only ever insert a
suﬃx once, the entire algorithm has a O(σn) execution time.

Empty-string canonization
Adding such a character $ to the end of S (if it does not already exist) is what
Gusﬁeld refers to a "canonization". We now show that instead of canonizing
strings with $ we can do it with the empty string, which is not immediately obvi-
ous. This is useful in scenarios where our alphabet Σ only just ﬁt in the allotted
bits per character, e.g. DNA with Σ = {A, C, G, T} that can be represented
with two bits per character. Adding a $ in this case would require additional
bits per character, up to the next power of two, thus eﬀectively halving our α.
What we do is the following. We run an additional and special last phase n + 1
that updates e := n + 1 but instead of checking for a path that continues with
S[n + 1] (which is beyond the end of S) in step 3, it simply treats the check
as false and therefore proceeds to step 5. This will have a similar eﬀect to
if S[n + 1] = $, as it will repeatedly insert explicit suﬃxes until r = 0. Note
that each leaf inserted here will have edge-label (ie, e) = (ie, n + 1). When the
special phase ends we then update e := n. This subtracts one from all leaf edge-
labels. For edge-labels that were only a single character before the subtraction,
i.e. (ie, e) = (n + 1, n + 1), this will result in an impossible string with indices
(n + 1, n) which we by convention interpret as the titular empty string.
The result is a suﬃx tree that explicitly contains all suﬃxes S[1; n], . . . , S[n; n]
(has a leaf for each), but for some leaf edges the label is the empty-string.
This does not aﬀect the correctness of the suﬃx tree and as it only is a single
additional phase, it does not aﬀect the construction time complexity either.

instead of using bit-masks at every word, the constants on the comparison are
considerably smaller.

3.2 Implementation Considerations

3.2
Implementation Considerations
In this section we discuss a few points of how the theory was carried into prac-
tice.
Many decisions were made along the way, that impacted the ultimate
performance of the solutions. As mentioned earlier, the implementation was
done in C++11 to allow for minimal overhead and ﬁne-grained over the per-
formance. We aimed to have minimal dependencies do not use any libraries
outside the standard C++ library STL. The correctness of the implementation
was veriﬁed by testing of the individual parts in isolation, along with testing the
full solutions against known data and queries. It is also veriﬁed with Valgrind2
to have no known memory access bugs.
3.2.1
Packed strings
For non-packed strings we simply used the string structure of the standard
library. The standard string uses a packed layout for characters and could be
used as a packed string implementation for some purposes. We chose however
to implement our own, to have complete control over and insight into how the
structure works and performs. Our implementation uses a vector (STL) to hold
the (32- or 64-bit) words and uses bit-masks to extract individual characters.
3.2.2
Edges in suﬃx trees
When using a suﬃx tree for pattern matching, there is the issue of how edges
are represented in the nodes. Using a list we have to check O(σ) edges in every
node, to ﬁnd the right edge. Theoretically we could have every node contain a
dictionary that with perfect hashing[FKS82b] maps the ﬁrst character of an edge
to the edge object. However, due to the rather heavy structure of a dictionary
this is not practical. We opted instead to use a single hash-map for the whole
tree, which maps a node and a character to an edge object.
This is only relevant when querying the suﬃx tree and is only used in the
implementation we compare our CST solution against. When decomposing a
suﬃx tree with the goal of building a centroid tree, we must check every edge
of a node anyway and a hash-map performs no diﬀerent than a list.

3.2.3
Allocating nodes
Both the suﬃx tree and centroid tree consist of a large number of nodes. One
way to create a new node in memory is to let the standard allocator reserve

are placed sequentially in one large chunk in memory, which is better for cache
behaviour. However, both the suﬃx tree and centroid tree utilize pointers to
their children. It is important that the dynamic array does not reach its capacity
and is re-allocated (to a larger memory region) in the middle of building, as that
invalidates all pointers set so far. When building a centroid tree over a suﬃx
tree of N nodes, we know that it will contain exactly N −1 nodes so we can
reserve just that space. For a suﬃx tree over a string of n characters, we can’t
know the exact number of nodes in advance. However, 2n−1 is an upper bound
so we reserve space for that amount of nodes.
3.2.4
Pre-allocating space for pattern preﬁx ﬁngerprints
When a query is performed for a pattern P, the ﬁrst step is to calculate the
ﬁngerprints of all preﬁxes of P. We found that the time it takes to allocate
and de-allocate the space to hold these ﬁngerprints, could be signiﬁcant and
would vary unpredictably. To avoid this we pre-allocate a memory chunk of ⌈n
α⌉
machine words for a centroid tree built over a string S of n characters. Since no
pattern P can appear in S if |P| > |S|, that is enough space to hold the preﬁxes
of any potentially occurring pattern. This space does not aﬀect the complexity
of the data structure as it takes O(n) space.
3.2.5
Selecting algorithms and BPC at compile-time
Our packed implementation supports c-bit characters where c = 2k, 0 ≤k ≤6.
The bit size must be power of two, to ensure that no characters cross word
boundaries. We inject this number at compile-time as a macro using the gcc
ﬂag -D. If we injected it at run-time instead (e.g. by command-line arguments),
we would probably experience signiﬁcant slow-down due to compiler not being
able to optimize as aggressively around its usage. This is a common thread in
our implementation and many settings are set at compile-time optimize as much
as possible.

4.1
Future Work
Before concluding we discuss some possible future avenues of research that ap-
peared during the development, but that were not explored.
4.1.1
False positives
The performance of our centroid suﬃx trees, both packed and unpacked, is
dependent on the quality of the ﬁngerprint hash function. Any time it produces
a false positive during the traversal of a tree, the structure wastes time checking
characters that must be checked again at the next (possibly false) positive.
In the worst case it checks O(m) characters before ﬁnding that it was a false
positive.
We have not analyzed the quality of any of the ﬁngerprints used in this project,
due to lack of time. All that was conﬁrmed was that at each new hash function
chosen, it resulted in faster queries for the data we tested than its predecessor,
false positives included. Superﬁcial testing of the number of false positives also
==PAGE:82==
70
Conclusion
indicated that it is fairly rare for even the non-packed XOR ﬁngerprints to
collide, for the data we tested.
It is almost certain that the Karp-Rabin function produces fewer false positives
than the XOR function, since it produces diﬀerent results for diﬀerent permu-
tations of the same characters, which the XOR function does not. Remember,
the Karp-Rabin approach gave way to the XOR approach due to the faster cal-
culation speed, which seemed to have a greater impact on query speed than any
false positives did. This diﬀerence in speed could be reduced however, if a way
could be found to calculate and extract Rabin-Karp ﬁngerprints in a packed
manner, similar to how it was done for XOR ﬁngerprints. It is presumed to be
harder, but we did not look into it. If it is possible, it has the additional beneﬁt
that Karp-Rabin can be randomized (by selection of prime p) and thus be more
protected against adversarial queries attempting to generate false positives and
slow the data structure down.
Presumably one could generate data for which the XOR ﬁngerprints always
result in the worst-case running time due to false positives. We brieﬂy attempted
to generate such data, to showcase where our structure would perform worse, but
it was abandoned due to the diﬃculty of the problem. While it is easy to create
substrings for which the XOR ﬁngerprints collide, it is less trivial to ensure that
such substrings appear and are visited not only in a suﬃx tree traversal but also
in a CST traversal. We do suspect that this problem is solvable though and it
might be interesting to study it.
4.1.2
Decomposition of the suﬃx tree
The centroid tree is built by decomposing the suﬃx via edge removal. As we
have shown, this edge-removal technique guarantees that neither of the two
resulting sub-trees are larger than σ−1
σ N, for a tree with N nodes, which results
in a O(log
σ
σ−1 n) time to traverse the centroid suﬃx tree (ignoring time spent
verifying ﬁngerprints).
The destination node of such a removed edge is the
titular "centroid" that yields the centroid tree. In literature however, "centroid"
most often refers to the related concept of a node that can be removed (i.e. node
removal) from a tree of N nodes, such that neither of the resulting sub-trees
are larger than 2
3N. This decomposition guarantee is better and it is suspected
that this concept could be leveraged in the centroid suﬃx tree. It is not trivial
however, as the centroid suﬃx tree must be binary and removing a node from
a suﬃx tree can yield up to σ + 1 sub-trees. It might be an interesting avenue
to explore.
It should be noted however, that in the later solutions we explicitly search for
==PAGE:83==
4.2 Conclusion
71
the edge that most evenly splits the tree. It is therefore suspected that while
the node-removal technique would give a better time complexity, it might not
have a large impact in practice as we already split very evenly in most cases.
4.1.3
Packing a suﬃx tree
Originally the CST approach arose with the idea that we would reduce the
number of nodes visited in the tree and thus have a more cache-friendly and
faster structure. We found out later however, that the CST does not visit less
nodes than the corresponding suﬃx tree in most cases. Assume that we have
indexed a string S of length n and we are querying the resulting data structure
for S. In the worst-case the suﬃx tree visits n nodes during the traversal, if
S is just one repeating character. In the worst-case the CST will only visit
O(log
σ
σ−1 n) nodes during the traversal, which is an improvement. However in
all but the most extreme cases the suﬃx tree is balanced enough that it visits
less nodes than the CST. This is also intuitive since the suﬃx tree has up to σ
children per node, while the CST is binary, often leading to a taller CST than
suﬃx tree. Still, the traversal is actually faster for the CST than the suﬃx
in practice, even with more nodes being visited. This seemingly arises from
the very low overhead of pointers (for the CST) compared to the much heavier
edge-list search or dictionary look-up (for the suﬃx tree).
Most of the speed from the packed CST comes from the packed comparison
and packed ﬁngerprint calculation. It could be interesting to see how well a
suﬃx tree would perform if it utilized the packed comparison techniques that
the PCST utilizes. The CST is slower than the suﬃx tree due to the added
calculation of ﬁngerprints, but in the PCST the ﬁngerprint calculation time is
reduced by a factor α. It is unclear if a "packed" suﬃx tree would be faster
than the PCST.
4.2
Conclusion
We developed a new data structure for solving the string indexing problem in
both the classic and packed setting.
In the non-packed setting we developed the centroid suﬃx tree that indexes a
string of n characters and answers queries for a pattern string of m characters.
It has a O(n) space complexity, a O(σn) expected construction time and the
best-case query-time is O(m + log
σ
σ−1 n) and the worst-case is O(m log
σ
σ−1 n).
==PAGE:84==
72
Conclusion
This is slightly worse compared to the O(m) query time complexities of the
textbook solutions, the suﬃx tree and suﬃx array.
In the packed setting we developed the packed centroid suﬃx tree that indexes
a packed string of n characters and queries for a packed pattern string of m
characters, both of which pack α characters per machine word. The construction
time complexity is identical to the centroid suﬃx tree and the space complexity is
either O(n) or optionally O(nα), where the extra space can be traded for better
constants on the query time. The query time is in the best case O( m
α + log n)
and in the worst case O(m log n).
The solution was developed in iterations, where each step is an improvement to
the previous in terms of practical performance. The ﬁnal product is a highly
conﬁgurable solution where each phase of the data structure oﬀers tried and
tested alternatives that can be combined in many diﬀerent ways.
All iterations were implemented and tested in practice. The four most pivotal
of them were selected for benchmarking and compared to each other and to
the textbook solution. We found that for space and construction time all our
solution were considerably worse than the textbook solution. For query time our
non-packed solution was not an improvement to the textbook solution either,
performing slightly worse. Our packed solution however scales almost linearly
with α for all the data we tested and as a consequence performs much better.
We then compared our ﬁnal solution to solutions from industry and academia.
As expected it trades space and preprocessing time for query speed. It is both
many times larger and slower to build, but it is almost an order of magnitude
faster than the competitors for small alphabets.

A.1.2
Problem Formulation
Problem deﬁnition
Given a string S and a pattern P, both over an alphabet
Σ of size σ = |Σ|, the string indexing problem is to create a data structure that
can answer the following queries:
• Count(P): Return the number of occurrences of P in S.
• Report(P): Return the positions of all occurrences of P in S.
A string is packed if it stores α characters per machine word and allows for
accessing and comparing α characters in constant time. The packed string in-
dexing problem is to create a data structure that can answer the string indexing
queries for packed strings S and P.
Previous work
The simplest way to solve the packed string indexing problem
is to treat is as a classic string indexing problem. Textbook solutions to this
problem are the suﬃx tree and suﬃx array. Suﬃx trees have linear space usage
and preprocessing time and support queries in O(m log σ) time, where m is
the length of P. Suﬃx arrays, if coupled with arrays storing selected longest
common preﬁxes[MM93], achieve similar space and preprocessing bounds and
support O(m + log n) queries. Allowing for randomization, coupling the suﬃx
tree with perfect hashing can achieve O(n) expected preprocessing time, O(n)
space and O(m) queries.
Recently Bille, Gørtz and Skjoldjensen[BGS16] showed that in the deterministic
and packed setting it is possible to achieve linear preprocessing time and space
and O(m/α+log m+log log σ) time queries. This solution might not be practical
however.
==PAGE:87==

In the non-deterministic setting Takagi et al.[TISA17] proposed a data structure
called the packed c-trie that takes O(n log σ+log n) space and allows for queries
in O( m
α +f(k, n)) expected time, where f(k, n) is the query time for an arbitrary
dynamic predecessor / successor structure over k values from the universe [1; n].
E.g. coupling it with y-fast tries[Wil83] could achieve O( m
α +log log σ) expected
time.
Aim of project
The aim of this thesis is to improve on the currently best
results for the packed string indexing problem in practice. This will be achieved
by either improving on existing algorithms to make them faster in practice or
producing new and improved algorithms. Results will be compared to current
state-of-the-art solutions to the problem.
