Abstract
This project presents a practical method for real-time motion planning for robot
manipulators in dynamic environments by means of a modular and ﬂexible architecture.
The implementation is based on two main modules. Motion planning for global
planning and local re-planning when the trajectory is invalidated by an obstacle, and 3D
perception for the dynamic construction of the collision map of the environment. The
collision map includes static and dynamic obstacles. The dynamic obstacles data comes
from the use of an Augmented Reality algorithm and from the use of point clouds. The
framework used for the development of this project has been ROS and more speciﬁcally
the Moveit! library. The implementation has been tested in 4 different scenarios by making
simulation and real world experiments. The results conclude that there is a way to make a
robot behave like a human in a safe way where the environment is unknown and where the
dynamic obstacles are also unknown. However, the overall performance of the real world
applications is drastically affected by the 3D perception step.

1.1
Motivation
Over the past years, a new technology revolution has been happening. This is not only
going on in the industry but also in the private/domestic sector where humans have new
needs that they did not use to have. The way this revolution is emerging is very varied.
From the Google self-driving car to home appliances, tasks that used to be done by humans
are now being replaced by or combined with a new technology.
Some of these new needs are going to be required by a collective instead of a single
person, in fact, as an example the agricultural ﬁeld could be analyzed. Regarding the world
population in 2015, the number reached to 7.3 billion people while the prediction for 2050
is of 9.6 billion people. In addition to these two numbers, the eating habits are going to
change as well as the need for the population to be fed. If the current technologies and
procedures do not evolve to something sustainable and efﬁcient, the demands of the world
population will not be satisﬁed. As a solution for this, there are different companies that
are already making a move. For instance, tractors are becoming autonomous leading to an
improvement in the efﬁciency and productivity.
When talking about either mobile robots or robot manipulators, the ﬁrst requirement to
be imposed is safety. This is critical if human-robot cooperation is hoped to be achieved.
Traditionally, when a robotic arm was programmed to do a systematic task like assembling,
the whole procedure was done with the robot inside of a cage. In this case the robot is not
aware of humans unless it can be remotely controlled. This would satisfy the very ﬁrst
requirement of safety but with a problem of ﬂexibility when humans need to interact with
the robot.
In the last few years, the need for cooperation between humans and robots is becoming
a real need. For instance, in a very near future there will be a domestic robot in every
home. To reach that goal, the robot should be able to coordinate tasks with the human

CHAPTER 2
Related work
2.1
Robotic motion planning approaches
Robotic path planning is the problem of generating a trajectory that connects two
different robot states while avoiding potential collisions with obstacles.
As it is already known, one of the main goals in order to achieve the ﬁnal purpose of
this project is to build a complete motion planner. However, even though this task may be
seen very easy, intuitive and simple from the human perspective, when trying to translate
the human behavior into a machine, the complexity of the problem increases signiﬁcantly.
Chronologically speaking, most of the previous works related to the generation of a
path between two states, mainly focus on deterministic approaches. That means that the
state of the robot, as well as its tasks, are well deﬁned and known. Normally, this type of
way of ﬁnding a feasible plan can be good enough for applications where the system is
fully controlled. Such as robotic manipulators in an assembly line, where the uncertainty
in the setup is nonexistent.
However, even though a feasible solution can be found by means of a deterministic
approach, the ﬂexibility and adaptability of this solution to the same robot but in a
different set up would make the problem more difﬁcult to be solved. That is when the non
deterministic algorithms come into place. It can be said that the goal of a non deterministic
method is to generate a solution within a time frame that does not exceed the constraints of
the problem at hand. It has to be mentioned that the solution could be just an approximation
of to the optimal one, but still interesting regarding the time that it is used to be found.
This type of methods are the base of Artiﬁcial Intelligence where there may be situations
where there are no patterns to follow to ﬁnd a solution. As it can be seen in Figure 2.1, a
deterministic algorithm that has a certain amount of steps, will always ﬁnish at the same
time and return the same result. On the other hand, in the non deterministic case, the
output is not constant and may change depending on the input.
Before going any further, there is a concept that needs to be addressed and understood

• Conﬁguration space: In mechanics, whenever a real system has to be studied, it is
necessary to obtain a model of the system. To do that, one can create a representation
of the system at hand by means of state space. By this description of the system, one
is able to identify the inputs, outputs, internal variables and related them with each
other using ﬁrst order differential equations. In relation to the motion planning tasks,
the state space is a set of feasible transformations that the robot is able to perform.
This is known as the conﬁguration space or C-space, based on Lozano-Perez (1979),
Lozano-Perez (1981) and Lozano-Perez (1983).
In order to clarify this concept, in Figure 2.2 Teller (2010), it can be seen how the
robot is interpreted within the workspace and the C-space. For a robot with n total
motion degrees of freedom, the C-space is represented by a coordinate system with
a single dimension per degree of freedom Teller (2010).
As it can be seen in one of the ﬁrst studies that considers the general motion planning
problem Reif (1985), Hopcroft et al. (1984) as well as in Canny (1987), to ﬁnd a solution
to the motion planning problem by non deterministic means implies the creation of the
conﬁguration space of the robot. This derives into problems whose complexity will
increase drastically when the C-space changes. This increment in the complexity of the
solution will cause the loss of practical efﬁciency, leaving a very weak performance behind.
In order to reduce the complexity of the system it is necessary to ﬁnd a lower-
dimensional conﬁguration space. This is due to the fact that the running time of the
algorithms is exponential in the dimension of the robot’s conﬁguration space Canny (1987).
For that reason, it is not interesting anymore to look for a solution including all existing

broad. However, regarding robot motion planning strategies in dynamic environments,
there are three main groups Kunz et al. (2010): reactive control approaches, path planning
approaches and mixed approaches.

2.1.1
Reactive control approaches
Traditionally, when a robot was used in a dynamic environment, the idea of reactive
control was based on potential ﬁeld methods as it can be see in Khatib (1986). The problem
that arises by using this type of methods is that they rely on local information. This is
something that could cause dramatic results such as the robot getting stuck in a local
minimum, hence the probability of ﬁnding a global optimal solution is very low Kunz et al.
(2010).
However, recent studies derived from the classic reactive control approaches are able
to generate very interesting results in the task of ﬁnding a valid path between two different
states in dynamic environments. As it can be seen in Dumonteil et al. (2015), based on
visual perception coming from a Kinect sensor, a reactive planner can be implemented
by monitoring the workspace of the robot. This information is used to generate real-time
trajectories that will avoid dynamic obstacles.
2.1.2
Path planning approaches
In relation to path planning approaches, they are capable of using the global information
of the environment when trying to ﬁnd a solution. This becomes an improvement compared
to the aforementioned reactive controllers.
Sampling-based motion planning
The main idea behind this method is to avoid the construction of the obstacles region by
instead doing a search that would check the conﬁguration space with a sampling scheme.
This checking is accessible by a collision detection module, which the motion planning
algorithm is not aware of. As it can be seen in Figure 2.3, the collision detection is
decoupled from the motion planning algorithm.
Figure 2.3: The sampling-based motion planning algorithm is characterized for separating
the collision detection, geometric and kinematic models from the planning problem itself
Pointing out some other important properties; this sort of algorithms works better
for high-dimensional conﬁguration spaces since the running time is not exponentially
dependent on the dimension of the robot’s C-space as it happened with the non deterministic

¸Sucan et al. (2012). A graphical representation of this behavior can be seen in Figure
2.4. Planners in this group such as Rapidly-exploring Random Trees (RRT) LaValle
and Kuffner (2001) or Expansive Space Trees (EST) Hsu et al. (1997), plan from
the starting position given in the query. Changing environments can be handled by
these motion planning algorithms, however it implies sampling the conﬁguration
space to afterwards check the different path segments for collisions which leads

to a very expensive procedure computationally speaking. Thus, the usage of these
planners to real-time applications is not feasible without some other extra procedures
Kunz et al. (2010). However, there are some new variants of these algorithms. For
instance, there is an extension of the RRT called dynamic RRT Deng et al. (2016).
This method is based on a global plan and local re-planning. This means that, every
time a new path is generated, the check of its validity is cyclically done. If the path
is invalidated the local re-planner proceeds to calculate the new plan with the same
goal position but changing the initial one to the current valid position.
• Multiple query: In this case, the planners build a road-map of the entire scene. This

be done at the same time. The most popular planner in this group is the Probabilistic

one query to be executed in the same static environment Kavraki and Latombe
(1998). Due to the assumption of the static environment, this sort of planners cannot
be applied to changing scenes. Additionally, since this method contemplates all
possible single queries within the built road-map, some extra work has to be done
to achieve the ﬁnal solution. Due to the high computational cost of combining the
different nodes in order to ﬁnd a valid path, a preprocessing task has to be applied.

DRM (Dynamic Road Map)
Introduced in Leven and Hutchinson (2002) this method uses the approach given
by PRM Kavraki and Latombe (1998) but now it improves the online efﬁciency by
preprocessing a road-map. When this static map is generated and before every planning
attempt, the parts of the road-map that are blocked by dynamic obstacles are invalidated.
The workspace is discretized on a grid that is compared with the road-map to enable a fast
checking and potential invalidation of path segments Leven and Hutchinson (2002). A
variation of this method implies the usage of a raw point clouds instead of assuming the
knowledge of a geometric model of the environment Kunz et al. (2010). But in any case,
it is based on DRM. To sum it up, DRM has a preprocessing stage where a road-map is
generated as well as the mapping of the workspace. Then the planning stage takes part.
The ﬁrst step will be to invalidate blocked parts of the road-map to later connect the start
and goal conﬁgurations to the road-map, where the collision checks can be performed
during the online planning. Once both tasks are achieved, the last step in the planning
process is to search the shortest collision free path.
DMP (Dynamic Motor Primitives)
When facing the problem of real-time tracking of a moving object there are different
approaches to do it, such as the eye-in-hand method and similar E. Malis and Boudet
(1999) and Kulpate et al. (2005). In this study Husain et al. (2014), the depth information
is perceived and used to track the object and generate a 3D position. This will be the
new position of the end effector of the robot manipulator which will be the input of the
inverse kinematics algorithm whenever the error (position and orientation) between the
end effector and the object to grab is below a certain threshold. Once that is done, the new
goal joint positions will be updated online by means of the DMP A. J. Ijspeert and Schaal
(2013), where the desired trajectory is calculated. The DMP method allows to change the
trajectory in real-time and in the meantime the positions and velocities are continuous. By
doing so, there is no need to compute the whole trajectory but only update the goal and
adjust the inverse kinematics algorithm and controllers to the goal movements.
Real-time perception guided motion planning
Even though as it was commented before, the use of 3D perception may be too
expensive in terms of time efﬁciency. However, many methods are based on the online
integration of three dimensional perception. In this study Rusu et al. (2009), the authors
propose a modular and distributed architecture to face the problem of a mobile robot
manipulator. It integrates the perception information for collision checking that will help
to do the real-time motion re-planning task.

Mixed approaches
Trajectory analysis and waiting strategy
This method faces path planning problems in changing environments where several
dynamic obstacles are packed together. It is assumed that the path generated is going to
be blocked normally Yan Li and Ding (2008). A classiﬁcation of the obstacles is done
depending on its capability to avoid collision automatically. Piecewise linear and non
linear obstacles with automatic and non automatic avoidance. The main idea of this method
is the waiting strategy. Sometimes it is not necessary to re-plan a trajectory since the
obstacle may be automatically removed from the scene. It is just a matter of waiting for it
to leave the path. The different steps in this method are similar to the ones in the previously
commented DRM. As in DRM, there is a preprocessing phase where the road-map is built
with a workspace and conﬁguration space mapping, but ﬁrst it is initialized with a set of
nodes provided by uniform random sampling Yan Li and Ding (2008). In the next phase,
the path searching is performed based on a waiting strategy. This follows a variation of
the DRM algorithm that now includes an active and passive waiting step depending on
the type of obstacle. For every single obstacle, a safety factor is assigned. If the path is
collision free, then the path is used. If the obstacles have a high safety factor, the robot
will wait (active wait), otherwise, if the safety factor is low then an improved version of
A∗algorithm Hart et al. (1968) is used to ﬁnd an available path.
Elastic bands
By means of this method, as well as with elastic strips Kunz et al. (2010), there is a
combination of the local information and the global one. First a path is planned given the
environment to afterwards being adapted for a changing environment by means of local
reactive control. This dynamic environment is represented by a cloud of points. It has to
be noted that obtaining the distances from the robot to a point in the cloud can cost time.
The complexity of this method may be too high when comparing its performance to other
ways of planning.
Inverse kinematics methods
Controllers are traditionally based on inverse kinematics. However, when working in
complex environments the need of adaptability increases. That is when artiﬁcial neural
networks (ANN) take action. The main application of ANN is called neurocontrollers
D’Silva and Miikkulainen (2009). Normally, whenever a path is generated, a trajectory is
sent to the kinematics model in order to make the robot move either by means of forward
or inverse kinematics depending on the values available in the trajectory. So the system

the robot is trained in order to generate a kinematic model. This may be useful when the
robot is working in dynamic environments where the obstacles have a pattern of movement
with a low uncertainty. It is true that the training tasks can be long, and the amount of data
necessary to generate a precise model should be considerably big. However, the results
give a great ﬂexibility and adaptability necessary in dynamic environments.

In the next Chapter a detailed description of the implementation is going to be made.
Additionally, in the following lines a description of what the method does and how it is
achieved is going to be explained. The very ﬁrst problem that may come to the reader is,
how can the robot move in a environment where all the elements are static and known?
In a previous research on Moveit! in static environments, Corchado (2017), a simple
task of generating a motion plan from state A to state B, where there is full knowledge
about the environment, was studied. When facing a problem of that kind, the solution is
going to be based on a global plan. That means that the initial and goal states do not need
to be updated in real time. Moveit! offers different possibilities for this planning task.
Essentially there are two main groups: planners provided by the OMPL library ¸Sucan et al.
(2012) and the Cartesian planner.
• Cartesian planner: Moveit! offers the option of planning in the Cartesian space, that
means that given different way-points, the planner is able to compute a path.
• OMPL library: Open Motion Planning Library, is a library that consist of many
different sampling-based motion planning algorithms. Notice that OMPL only focus
on the planning tasks and not on the graphical representation or collision checking.
In Figure 4.1 the main structure of OMPL is represented.
The planners are divided into two different categories: geometric and control based.
Geometric planners only take into consideration the geometric and kinematic con-
straints of the system. These planners are the ones explained in the related work
Chapter under the name of sampling-based motion planning algorithms. On the other
hand, there are control-based planners which are based on differential constraints.
The planners rely on how the states propagate in time instead of on the interpolation
to generate motions.
When comparing the behavior of both options, it can be seen that the trajectory
generated by the Cartesian path is more predictable than the one provided by the OMPL
library when there are no constraints set in the movement of the robot. When that is the
case, the solution of the inverse kinematic solver may change due to the fact that, given an
end effector position and orientation, there are different solutions of set of joint values that
can solve the problem.
This may imply that the ﬁrst planner, the Cartesian one, is the appropriate option for
the case of study but, as it is shown in Corchado (2017), when there are static obstacles the

4.1.3
Motion planning module
There are two assumptions that are going to be taken for the explanation of the motion
planning module. Firstly, the state of the robot is going to be known. Secondly, the
collision map (static and dynamic) is going to be known (provided by the 3D perception
module).
Once a request for planning is received by the motion planning module, a path will
be found in two main steps. The ﬁrst step is to ﬁnd a global path. That means that given
the current state of the robot and the goal state, by means of a planner (RRT for the
development of this project), a path is computed. After this is done, the second step takes
care of checking the validity of the path. That means that if there is an obstacle that makes
the path invalid, then a new planning request will be received by the motion planning
module. This time the computed path will be from the current state (not the initial one
global wise) to the same goal state. These two steps are going to be repeated until the
robot reaches the ﬁnal state.
When checking the validity of the path, it is necessary to keep track of the current
state of the robot as well as the trajectory that is following. Deﬁning a bit better how the
path becomes invalid, there are two possible ways given the available tools. The ﬁrst one
would be using the force sensors of the UR10. This implies that the robot will send a
new planning request when it collides with an obstacle, sending the previous valid state
(without collision) as a new initial state. Due to safety reasons this option, even though it
was tested out as it will be commented later on, is not the most suitable for the purpose of
this project. The other way would be to build a dynamic 3D map of the environment by
means of the Kinect sensor. The construction of this map is explained in the next section.

CHAPTER 5
Details of the implementation
To this point the method designed to ﬁnd solution for the real-time path planning of
a robot manipulator in a dynamic environment has been described. However, there are
different details in relation to the implementation of the motion planner module and the
3D perception module that are considered relevant for the understanding of the solution.
Throughout this Chapter, different milestones on the implementation are going to be
assessed, describing the reasons behind the decisions made during the implementation.
5.1
Implementation of the motion planning module
When thinking about how a human would be able to change the trajectory of its
movement when a dynamic obstacle is obstructing the path, the reaction time of a human,
using visual perception, is of around 180–200 ms regarding the results of this study
Thompson et al. (1992). Moreover, if the human is executing the movement with a velocity
and acceleration within some limits, if a change of trajectory is needed, it will be done
smoothly. This behavior is the one that needs to be implemented for the robotic system.
This is not only due to visualization purposes but also to make the robots user friendly.
When working with the real system there are different factors that need to be taken into
account.
• Moveit! offers the functionality of executing a new trajectory while another one is
being executed, however this feature was not able to be used.
• When a new movement request is sent to the robot, the response time is of around
200 ms. This is the time that the controllers use to receive the signal and execute
it. The planning time needs to be added to this reaction time. Hence, making the
trajectory smooth becomes more complex.
33
==PAGE:52==
34
CHAPTER 5. DETAILS OF THE IMPLEMENTATION
Given these two factors, a different way was used to smoothen the movement of the
robotic arm. This was achieved by means of an asyncronus execution of the movement.
Moveit! offers the option of executing a trajectory and block any other movement signals
sent to the controllers until the goal is reached. But that is not interesting for the purpose
of smoothing the change of trajectory. However, by means of the asynchronus execution
the robot does not need to get to the goal position to receive a new command that could be
either stop or a new trajectory.
Even though this is the only way found to make the movement as close to the human
behavior as possible, when adding the planning time to the reaction time of the controllers,
there is a small pause between two different trajectories. This is measured in the next
Chapter.
As mentioned before, the output of the sampling-based planning algorithm RRT, may
vary depending on the solution found by the inverse kinematic solver. That is why a set of
external and internal constraints are established in order to predict the movement.
• Externally, different static collision objects are included in the scene as it could be
seen in the previous chapter. This will limit the workspace of the robot manipulator
to a smaller area, which will help to analyze the behavior of the movement later on.
• Internally, in order to obtain similar solution from the inverse kinematic solver, the
movement of the joints is restricted to certain limits. For a given current state (set of
joint values), they will be able to move within certain margins. It has to be noticed
that while this helps in the task of making the movement more predictable, the more
restrictive the constraints are the more difﬁcult it will be for the planner to ﬁnd a
solution.
5.2
Implementation of the 3D perception module
Regarding the construction and update of the dynamic collision map, as it was said
before, there will be a static collision map and a dynamic one both merged together by
the planning scene interface. Even though the motion planning module was designed and
implemented to be able to work with real dynamic obstacles, as a ﬁrst step to debug the
performance, a dynamically simulated collision object was used as a dynamic obstacle.
Once this set up was working, the simulated 3D object was substituted by a real one.
Before going through the explanation of the implementation of the 3D perception
module, there is a key factor that needs to be addressed in order to have reliable data of the
generated collision map: the camera calibration. To be more speciﬁc, the robot is going
to be located in a reference frame labeled as world. The positions and orientations of the
==PAGE:53==

end effector are going to be referred to that reference frame. When talking about the static
map, the positions and orientations of the collision objects are going to be represented in
the same frame.
However, when having an external camera it is necessary to have an accurate intrinsic
and extrinsic calibration.
• Intrinsic calibration: Since the device that is going to be used is a Kinect sensor v1,
as it was described in the Hardware analysis, this component is equipped with two
different cameras. A RGB camera and a IR (infrared camera). Hence, two different
intrinsic calibrations need to be applied by means of the camera_calibration package
available for ROS. Notice that the calibration procedure is going to be the same for
both cameras since they are independent.
After the calibration is done, it can be said that calibration was successful if a real-
world straight line in the corrected images keeps being straight. On the other hand, a
failure in the calibration may result in images that do not preserve the straight lines
or the images are simply unreadable.
• Extrinsic calibration: In addition to having the correct images, the location of the
cameras is something that needs to be deﬁned. Not only the location of the cameras
within the device but also with respect to a desired reference frame. Regarding the
internal relative position of the IR camera and RGB camera, this parameters are
provided by the speciﬁcations and can be updated if desired. However, what seems
to be more interesting is the location of the Kinect sensor with respect to, in this
case, the world frame. If this information is not known, it is impossible to make
the robot know where are the objects that the camera sees. To give a solution to
this problem, an application of the method created by Møller (2017), is going to be
accurate enough for the purpose of this project.
Once the extrinsic calibration is done, the position and orientation of the camera
in the world frame is obtained. To check if the results are successful, one simply
needs to locate an object in a certain known position to afterwards check if in the
simulation the location is the same.
Once the calibration step is done, the tools used to implement both, real and simulated
collision objects, are going to be described.
• Simulated dynamic collision object: Once the motion planning algorithm is imple-
mented, as it was shown in the architecture of the system, it needs to have an updated
dynamic collision map of the system. For safety reasons, the ﬁrst implementation of

CHAPTER 7
Conclusion
Throughout this project, a method has been designed, implemented and tested for the
problem of real-time motion planning for a robotic arm in dynamic environments.
As it was presented in Chapter 2, there are many different technologies and methods
to tackle this type of problem. From predicting the movement of the obstacles, to using
neural networks to train the controllers of the robot in order to be able to avoid dynamic
obstacles whose movement is predictable. However, the ﬂexibility of these approaches
was not the most suitable if the robot manipulator is working in a unknown environment.
That is why the methods including some type of real-time perception will have the focus.
Before starting the design of a new method, it is important to know what tools are
going to be used. This is analyzed in Chapter 3. Once the initial idea of using perception
to ﬁnd a solution for the problem was clear, the type of perception was narrowed down
to vision since is the one that will provide better 3D information about the environment.
Given the tools, an overall design of the architecture of the system was presented.
Once the architecture of the system has been design, the implementation of the method
itself has to be described. In Chapter 4, the two main modules, Motion planning and 3D
perception, are explained as well as the global functioning of the system. However, there
were some details related to the implementation that had to be treated in order to have a
better understanding of the method. That was the focus of Chapter 5.
Finally, in Chapter 7, the method was tested. To do that 4 different test cases were
proposed. Two of them simulation based and two of them using the real robot arm. A
number of 40 tests were performed tracking aspects such as the global planning time or
the reaction time. All the data generated was analyzed by means of a statistical approach.
It can be concluded that this method is able to work in dynamic environments where
there dynamic objects and potential obstacles are unknown. However, even though the
global planning time and local planning time were under 200 ms, when combining it with
the image processing and the communication with the robot the performance became

slower. For the cases where the collision map was provided by the AR algorithm, the
reaction time was much faster, staying between 200 ms and 300 ms as a mean value. On
the other hand, when a 3D reconstruction of the map was done, even though the planning
