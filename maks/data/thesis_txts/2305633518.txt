Summary (English)
The goal of the thesis is to design a system, that is capable of creating in-
voices based on the user’s movements and location. Modern smartphones oﬀer
a great opportunity to use personal tracking in our work and spare time, and
have already been utilized in a large amount of applications. In this thesis we
will investigate how personal location tracking, can be used to increase work
productivity, by minimizing the additional work associated with the invoicing
process. The system is an extension of an existing invoicing system used by
a small landscapers company, and we will use this company as inspiration to
deﬁne the functionality of the system.
The thesis shows how we build a location-aware invoicing system. The system is
able to create an invoice with the amount of hours and transportation expenses,
the user have spent working for the customer. By collecting location data using
a smartphone, we are able determine whether the user is driving or not. When
the user is not driving we assume that he is working for a customer and can
then use his current location to determine which customer he is working for.
Combining the knowledge of what the user is doing and where he is during the
day, allows the system to calculate the spent amount of hours and transportation
distance that a given customer should be billed.

2
Introduction
1.1
The problem: Location-aware invoicing
In this thesis we will explore the possibilities of designing and developing a soft-
ware system which is able to generate invoices for a craftsman. By utilising
the location services found in modern smartphones and other mobile device, we
want to build a system which can observe the craftsman’s location and move-
ments during work days and then infer where he has been and which customers
that should be billed to generate invoices to each customer.
To illustrate the system’s purpose we will use the following example of a landsca-
pers company. The company is a small business with mostly regular customers.
The landscaper will agree with a on the work that has to be done and then drive
to the customer, bringing necessary tools and carry out the work agreed with
the customer. The landscaper will then, once a month, write an invoice to all
the customers he has worked for during the month, containing the work hours,
transport expenses and any materials used in the process. This means that the
landscaper who does the work will need to write down the expenses and then at
the end of the month, type in all the information in invoices for each customer.
We believe that a system can be build to automate the billing process, we will
therefore want to prove the following thesis:
It is possible to design and build a location-aware invoicing system that can
automatically generate invoices to customers, based on the user’s movements
and location.
To design such a system there are challenges that we have to consider and inves-
tigate, some of these challenges is how geotracking can be used to determine the
user’s movements and how the user’s locations and movement can be integrated
with an invoicing system.
1.2
Background
To investigate the challenges of geotracking, has the following related work on
location-aware systems been studied.
The related work can be divided into
two categories, ﬁnd location of interest and determine transportation mode.
The ﬁrst category aims to ﬁnd signiﬁcant locations which are often visited, or
visited in a longer period of time. In [1] a signiﬁcant location is found using
the fact that GPS signals can not be received indoor, and a place is then found
when the time interval between two GPS observations becomes large. In [2] a

signiﬁcant location is found as a place where the user stayed longer than a given
time period within a given distance.
The second category of related work aims to determine the user’s transportation
mode, to ﬁnd out whether the user is walking, driving, traveling by bus, etc.
One of the common features used to infer a user’s transportation mode is the
traveling speed. In [3, 4] the transportation mode is mainly determined by the
movements in terms of speed, direction change and stops during the transport,
with this it is possible, in some degree, to infer how the user is traveling. The
articles [5, 6, 7] have a greater focus on using machine learning techniques based
on collected sample data and combining the results with street maps of roads,
bus routes subway system etc. to determine what kind of transportation a single
user is traveling with.
Some of these techniques and methods are not often seen, but the underlying
methods of collecting and processing location data are becoming more widely
used as the amount of devices with built-in GPS receivers grows. Properly the
most common use of personal tracking is satellite navigation systems, it has
become so common that we do not even think of it as an intelligent system.
More recent use of personal tracking is in sports and personal health care, in
these applications tracking is used to help the user understand and improve their
own performance, and used to share personal achievements on social media. The
key functionality in these applications is to transform the user’s movement data
into understandable and useful data to the user.
1.3
Goals
We divide the overall goal of designing a system to automatically generate in-
voices, into four sub goals which describes key parts of the system.
1. Reliable tracking: The system can reliably track the user’s movements
during the day.
2. Categorise trace: The system is able to identify when the user is driving
to a customer, and when the user is working for a customer.
3. Identify customer: The system can identify which customer the user is
working for.
4. Integration with invoice system: The system can generate invoices to
customers in the existing invoice system, based on the tracking data.

Background
2.1
Related work on personal location tracking
The following related work has been studied to ﬁnd inspiration and solution pro-
posals on how activities and signiﬁcant locations can be inferred using personal
location tracking.
2.1.1
Location of interest
In the article Using GPS to Learn Signiﬁcant Locations and Predict Movement
Across Multiple Users [1], the authors creates a predictive model of a user’s
future movements. Based on continuous GPS tracking, the ﬁrst goal is to register
and learn the signiﬁcant locations that the user visits and when he visits them.
The second goal is to use these signiﬁcant locations, to predict where the user
will go to from his current location. A location is found by clustering places,
where a place is the last GPS position before the signal is lost for at least 10
minutes. The method shows that it is possible to ﬁnd locations of interest based
on lost signals, however it only allows indoor locations to be detected.
Another approach is presented in Mining Interesting Locations and Travel Se-
quences from GPS Trajectories [2]. In this article the authors aim to register
interesting locations visited by travelers to improve travel guides and recom-
mendations. Instead of solely relying on lost GPS signal they introduce a stay
point. A stay point is a geographical region where a user stayed longer than a
given time interval as shown in Figure 2.1. The determination of a stay point
depends on two parameters. The time interval and the distance that all GPS
positions associated with the stay point should be within. This method makes
it possible to ﬁnd interesting locations both outside and indoor.
Figure 2.1: Identifying a stay point as a graphical region where the user has
been for a longer period of time. ﬁgure from [2].

2.1.2
Infer transportation mode
Transportation patterns and travel behavior can be valuable information when
it comes to city planning and expanding infrastructure. In [3] the goal is to im-
prove transportation surveys and automatically classify the survey participants
transportation mode Bus, Car, Cycle, Train, Tube and Walk. By equipping
the survey participants with GPS devices and ask them to complete the sur-
vey by classifying their trips as in a regular survey, it is possible to collect test
data. The authors consider the following four variables in their analysis, speed,
acceleration, travel distance and change in direction.
The results show that
speed and acceleration are the most signiﬁcant features to classify the partici-

Tracking technologies
Modern mobile devices provides three localization sensors, GPS, WiFi and GSM.
The GPS receiver uses the satellite based Global Positioning System which de-
livers a high accuracy, but consumes more power and does not work indoor. The
WiFi localization uses the position of known WiFi access points to triangulate
the position of the device. The method delivers a good accuracy at low power
consumption and works indoor, but does not work in open areas away from
WiFi access points. GSM also referred to as Cell-ID uses the location of audible
cell towers, to triangulate the position. Like the WiFi does this method also
works indoor and the power consumption is even less than the WiFi method and
utilizes the range of the GSM signal to improve the availability of the method.
The drawback of the GSM method is the accuracy which is between 100 to 400
meters in optimal conditions as shown in Table 2.1 [9].
To utilize the advantages of each location provider, Google has developed the
Fused location provider API which combines the three providers into a single
service [16].
This means that the location API can hide the complexity of
handling multiple providers simultaneously and deliver a compromise between
fast, accurate and low power consumption, based on the current conditions,
which allows accurate tracking both indoor and outdoor.

Categorize consultant movements
When analysing the consultants movements, it is interesting to know if the

and Stationary, this division of movements slightly diﬀers from the states and
groups presented in [3, 6, 7] which all focus on distinguishing between diﬀerent
transportation modes, but similar for all of them is the use of speed as the main
feature to diﬀerentiate the transportation modes.

normal conditions, possible to infer whether the consultant is driving a car or
being stationary, by testing whether the consultant’s speed Cv is greater or

consultant is driving, and when slower than V the consultant is stationary or
walking. Figure 3.2 shows such a scenario where each observation is captured

3.4
Tracking strategies
In modern smartphones it has become a de-facto standard that they are equipped
with a variety of sensors, including accelerometer, light sensors, GPS receivers
etc. These build-in sensors can provide highly accurate data of where the users
are and clues about what they do right now and over period of time.
This
ability of real-time tracking, computing power and Internet connectivity opens
possibilities to construct applications that can monitor, react to or summarize
the users movements into consumable information.
All three major smartphone operating systems, provides well established API’s
for two types of location awareness, Geofencing and current user location.
• Current user location: Retrieves the current geographical location of
the user. This method is used in various applications, where the user or
application needs to know where the user is right now e.g. Navigation
apps, Find nearby places etc.
• Geofencing: A geofence is a virtual delimiter around a geographical area.
Applications can set circular geofences by deﬁning a center and radius.
When a geofence is registered, the application will receive events when
the user enters and leaves the registered area.
The following sections presents three tracking strategies, using either the current
user location or geofencing.
Each strategy features diﬀerent advantages and
disadvantages.

We will choose a strategy based on the main features and drawbacks of each

The geofence strategy provides a high usability, since the consultant do not have

transportation. The strategy also provides the ability to track time spend at
unregistered locations, which the geofencing strategy does not.

Deciding the state of observations
As described in section 3.3.1 we will categorize the consultant’s movements using
a similar method as the one presented in [6], and divide the movements into the
two states Driving and Stationary. Besides these two we also introduces a

third state, which describes a time period where no observations have been
captured, we call this state Unknown.
Before describing the approach of deciding the state, we have to deﬁne the
following terms: Location trace, trace point and trace segment. The location
trace is a sequence of trace points, pi ∈P, P = [p1, p2 . . . , pn], where a trace
point pi is a single observation containing latitude, longitude and a time stamp
of when the trace point was recorded. A trace point pi describes the state of the
consultant between pi and pi+1 based on the velocity and time span between the
two points. The trace segments are dividing the location trace into consecutive
segments si ∈S, S = [s1, s2, . . . , sn]. A trace segment is the longest sequence
of consecutive trace points of the same state, as shown in Figure 3.6.
using a similar method as the one described in [6] to divide a trip into walk and
non-walk segments
We will now deﬁne how the state of a trace point θ(pi) is decided, using the
following rules:
• θ(pi) is unknown when the time span between pi and pi+1 is greater than
the oﬄine threshold Toff.
• θ(pi) is driving when the velocity of pi is greater or equal to the velocity
threshold V , and θ(pi) is not unknown.
• θ(pi) is stationary when the velocity of pi is less than the velocity thresh-
old V , and θ(pi) is not unknown.
To decide the state of a trace point we use two threshold values, V which is the
maximum velocity of a stationary trace point, and Toff which is the maximum
time span allowed between two trace points pi and pi+1, if the time span is too
long we will not be able to determine a reliable state.

For choosing the velocity threshold V , we will ﬁrst deﬁne the frequency fp the
trace points are captured at. In section 3.3.1 it is discussed how a high frequency
can cause false stationary states when driving in congested traﬃc, and how a
low frequency can hide short driving states. Besides that, will the frequency
also have an impact on the battery time of the mobile device. Taking these
considerations into account we set the trace point frequency to fp = 5min., this
frequency assumes that the consultant will spend at least ten minutes working
for a customer, and when driving in congested traﬃc the consultant will still
move a signiﬁcant distance within ﬁve minutes. By setting the velocity threshold
relatively low, we limit the risk of detecting a stationary segment when the
consultant is driving in congested traﬃc. Setting the threshold too low could
produce false driving segments, in case the consultant walks a longer distance
while working for a customer. An average walking speed has been measured in
[3, 6] to be between 0.5 −4m/s ≈2 −15km/h, which is a rather low speed
compared to normal driving. Setting the velocity threshold V = 2m/s, means
that the consultant can walk a straight line distance up to 600 meters within
the ﬁve minutes before it is considered driving. It is possible to walk longer
within ﬁve minutes but we consider it unlikely to happen while still working for
the same customer..
The oﬄine threshold Toff is used to detect periods where no trace points have
been collected, we will set this threshold to 20 minutes. This allows the system
to skip a few trace points, without notifying the consultant. But in situations
where the device is turned oﬀor tracking is not allowed, the system is able to
notify the consultant that it can not account for the given time period.

5.1.3
Web communication architecture
When distributing the work between diﬀerent devices, we will need to establish
a communication architecture that allows the location collector and the location
manager to utilize the data processor. We will here consider the following two
architectural styles.
• RESTful Web Services
The REST (Representational State Transfer) architectural style was ini-
tially proposed in the PhD dissertation "Architectural Styles and the De-
sign of Network-based Software Architectures" by Roy Fielding [10]. The
architectural style utilizes the features of the HTTP protocol to establish
a well deﬁned interface for retrieving and posting resources over HTTP.
As of today it has been adopted by several larger companies to allow third
party applications to access and consume resources, often referred to as
RESTful services or a REST Web API.
A REST Web API provides an interface for client applications, to access
and add new resources stored on web servers, by providing the standard
Create, Read, Update and Delete (CRUD)-operations to manipulate data
stored on the server. The operations are called using the ﬁxed HTTP verbs
POST, GET, PUT and DELETE to manipulate speciﬁc data entities on the
server. By doing so and following the architectural principles described in
[10] the resulting interface will be a standardized REST API, that behave
and feel like other REST APIs. This attribute has made it widely popular
among developers because of the familiar standardized way of working
with REST APIs from diﬀerent data providers.
• Remote Procedure Call
Another method of communicating with a server is using Remote Proce-
dure Call (RPC). RPC is a method of client/server communication where
the client can invoke procedures to be executed on a remote server, with
information passed from the caller to the callee in the parameters and come
back to the caller in the procedure result. This method was introduced by
Birrell and Nelson in 1984, to abstract the messages passed between the
client and server in distributed systems [11].
==PAGE:57==
5.1 System Architecture
43
RPC has been implemented in several diﬀerent protocols, from platform
speciﬁc implementations like Java RMI and .NET Remoting to platform
independent web service protocols like SOAP and more recent Json-RPC.
These frameworks makes it possible to implement RPC with a high ab-
straction from base network communication and handling sockets, and
instead makes the implementation focus on the procedures and methods
that should be available and able to invoke from a remote system.
Both communication architectures makes it possible to build a web API that
can allow the location collector and manager to communicate with the data
processor.
The main diﬀerence of the two architectures is how they present
concepts of the underlying system. REST presents the entities of the system,
allowing the consumer of the API to get entities from the system alter them
and then post the altered entity to the server. While RPC presents procedures
of the underlying system.
In this system we are going to use a combination of the two architectures. When
the location collector sends the trace points it is expected that the points are
being processed further, which is more a procedure call. While the location
manager is going to use the API to retrieve and update resources of the system,
which leans towards a REST style. We choose to design the web API using the
REST style where the resource names are used as entry points and then allow
the API to execute more complex procedures behind the scenes.

1. Clean trace points: The cleaning step removes duplicated trace points,
a duplicate is a trace point that has the exact same time stamp or has a
time stamp within few seconds of the former trace point.
2. Save trace points to database: The cleaned list of trace points is saved
to the database, if the cleaning and saving is successful, an OK response
is returned to the caller and the execution continues. If the trace points
could not be saved an error code is returned to the caller and the execution
is terminated.
3. Calculate trace point features: Before the segments are computed,
the distance, speed and direction between each trace point are calculated.
4. Compute trace segments: When the features of the trace points are
calculated, the trace segments can be computed, this includes determining
the type of each segment, ﬁnding nearby places and selecting the nearest
place as the actual location.
5. Save trace points and segments: At last the calculated trace point
features and the generated trace segments are saved in the database.

6.2
Location Collector
The location collector is responsible for observing the consultant’s movements
during the day and post the locations to the data processor. To do this the
application will need to execute every ﬁve minutes without user interaction, we
will therefore start by describing how an Android application can be set up to
run periodical at a given frequency. Secondly we will investigate how the fused
location API, is used to get reliable location results.
6.2.1
Execute background operations
Under normal conditions, Android does not allow applications to run while the
screen is turned oﬀ. By suspending running threads Android is able to conserve
the battery of the device while the user is not interacting with it. However there
are two mechanisms that can allow an application to run while the screen is oﬀ.
1. Holding a wake lock: By acquiring a wake lock, the application indi-
cates that the device should not go into sleep mode and keep the CPU on
until the lock is released.
2. Using the Alarm Manager: The alarm manager allows applications to
wake up the device at a speciﬁc time of the day, or repeatedly wake up
the device with a given time interval.
The location collector will need to retrieve a new location every ﬁve minute
and post the locations every half hour, both operations will usually only take a
couple of seconds and it would therefore be a huge waste of battery to acquire a
wake lock and keep the CPU on all the time. The Alarm Manager is therefore
used to wake up our application every ﬁve minutes to retrieve a new location,
and every half hour to send the positions to the data processor. To ensure that
the alarms are registered even after the device is rebooted, we will register the
alarms using a boot complete broadcast receiver which is called when the device
is booted, shown in Appendix C.2.1.
6.2.2
Ensure high location accuracy
Early tests of the system showed results with up to 11.5 kilometers uncertainty
radius, when using the ﬁrst location returned from the fused location API. It is

The location API provides a measure of each location called accuracy, which
describes the uncertainty of the found position in meters. This measure is also
often referred as Estimated Position Error (EPE), which is an estimation of
how close the calculated position is to the true position of the tracking unit [21].
This means that the smaller the accuracy is the more accurate is the location.
Even though Google has not published a strategy to get the lowest EPE using the
fused location API, they have released a strategy guide for using the original
android.location API. This strategy guide suggests the following approach to
acquire the position [22].
1. After application start, begin listen for updates from location providers.
2. Maintain a current best location when receiving location updates.
3. Stop listening for updates.
4. Return the best location update i.e. the most accurate.
This approach is illustrated in Figure 6.1, which shows that the diﬀerent location
providers, receives location updates independent of each other. This means that
the listening time, can have a signiﬁcant impact on the accuracy of the found
location. This is especially important in situations where the network provider,
using Cell-ID and Wi-Fi, can not determine the location with suﬃcient accuracy,
and the result has to rely on the slower GPS receiver to be ready. An early stop
could result in returning a position with poor accuracy even before the GPS
receiver had a chance to receive a location update.

Conclusion
In this thesis we have designed a software architecture for generating invoices
using personal location tracking. The architecture is divided into three main
components, the location collector which is responsible for monitoring the user,
the data processor which is responsible for categorizing the user’s movements
and handle invoices, and the last component is the location manager which
enables the user to verify and alter the categorised movements. The separation
of responsibilities increases the maintainability of the system and allows it to be
distributed between the existing servers and the mobile device. To categorize
the user’s movement, we record the location of the user every ﬁve minutes and
calculates the distance and speed between the observations. The speed is then
used to determine whether the user is driving or not.
We evaluate the system based on how well it meets the goals of the thesis.
The goals have been tested both during development and on the ﬁnal system
and we see that the ﬁnal system is able to deliver reliable results under normal
conditions.
1. Reliable tracking: The system is able to reliably track the user’s loca-
tion, with an accuracy better than 80 meters, both indoor and outdoor.
2. Categorise trace: Under normal conditions the system is able to deter-
mine whether the user is driving or being stationary. Under more unusual
conditions such as a blocked highway, the system is prone to categorize
the slow driving as being stationary.
3. Identify customer: The system is able to identify the customer who
should pay, by ﬁnding the closest customer to the user. This identiﬁcation
is reliable in most cases, but suﬀers when two or more customers live close
to each other.
4. Integration with invoice system: The system is able to generate in-
voices and invoice items, for billing customers in terms of work hours and
transportation distance to the customer.
The achievement of the goals show that we have been able to develop a system
that can assist the user in the invoicing process by automatically tracking the
transport distance and time spent at a customer’s location, and allow the user
to create the invoice directly on his mobile device. This can solve the problem
of spending time on registering invoice items and remove the risk of loosing
handwritten notes. But we have also seen situations where the system is not
able to automatically produce a reliable result. This means that we have not
been able to prove that it is possible to build a system, capable of automatically
generating invoices based on the user’s movements and location.
==PAGE:101==
87
To prove the thesis, we would need to improve the reliability of the categorization
of movements and the identiﬁcation of customers, to ensure that no customers
are wrongly billed. To improve the categorization, the ﬁrst step is to calibrate
the frequency of capturing locations, and adjusting the velocity threshold. By
testing the system more, and analysing how it behaves under unusual conditions
are we able to slightly calibrate the system and expand the window of correct
detection. However there will be situations where the velocity is not enough
for correct detection and to solve this we will have to consider more variables
such as change of direction. The problem of identifying the correct customer
happens when two or more customers live close. In some cases the problem could
be solved by improving the placement of the customer’s location, the location
could potentially be improved, by choosing the location as the average position
of all the times the customer has been visited.
Besides the reliability improvements, the system would also need to support
additional invoice details such as work description and materials that should
be billed. Without this support the system would not be able to replace the
handwritten notes.
Even though we did not prove the thesis, we have shown that the system is able
to assist a landscapers business with the process of writing invoices. Since the
system is able to track the user both indoor and outdoor, it can be used in a large
range of industries such as carpenters, electricians and plumbers. Additionally
it can be used in the consultancy industries, where a person is employed by one
company and provide services to another company at their location. Another
use case could be part time employees on an ﬂexible agreement, the employee
would have to register the hours spend and then send it to the accountant.
Using this software could assist the employee in registering the correct amount
of hours and send the registration directly to the company’s economy system.
