Abstract 
 
An information exchange setup is established between a parametric design software and an 
acoustic ray tracing algorithm. The purpose of the setup is to enable a user to get a fast 
evaluation of selected acoustic properties of a model, while also being intuitive to use. This is 
achieved by setting up a workflow routine within the parametric design software and creating 
scripts that automate the information exchange. An emphasis is placed on making the setup 
dynamic, meaning that a user can simulate the effects of changing the geometry of a room in 
an effective and fast manner. Several aspects of the setup are explored, such as visualization 
and optimization possibilities. The advantages and disadvantages of the current form of the 
setup are identified and some directions for future development are suggested. In general the 
setup can be expected to simulate the selected parameters for complex geometry at a 
reasonable accuracy at mid- high frequencies, compared to other simulations known to be 
accurate. The accuracy at low frequencies is however not good, in large part due to the choice 
of prioritising simulation speed over accuracy.

Introduction 
1 
1. Introduction  
 
Parametric design is the modeling of a geometry based on a set of parameters and the 
algorithmic relationships between them. It allows a designer to design shapes and patterns in 
an iterative manner and thus enables the design of more complex geometry than traditional 
modeling techniques allow. Recent technological developments have made parametric design 
possible on a larger scale than before by linking it with commonly used modeling software. 
This has resulted in increased adoption of the parametric design principles by architects and 
designers.  
The objective of this project is to investigate the inclusion of acoustic simulations into this 
parametric design process. The goal is to create a setup through which a user can get a fast 
evaluation of selected acoustic properties of a model at an early design stage, in order to make 
the need to redesign the model at a later design stage less likely.  
The most commonly used acoustic modeling software today, such as Odeon and CATT-
Acoustics require the user to import a geometrical model in order to preform acoustic 
evaluations. The model therefore loses its link to the software it was modeled in. In addition 
Odeon and Catt are not specifically designed for architects but rather for acousticians. It has 
therefore been difficult for architects to evaluate the acoustics of their design, which is one of 
the key problems this project aims to address.   
The acoustic simulations in this project are done through a ray tracing algorithm made to 
communicate with parametrically designed models. A ray tracing algorithm was inherited 
from FinnurвАЩs Pind master thesis вАЬBroadband Hybrid Room Acoustic Simulations Using 
FDTD and Ray TracingвАЭ [1]. Significant additions were made to the algorithm in order to 
better account for the many geometrical configurations that can be created with parametric 
design. In addition a genetic algorithm was developed in order to aid a user in finding a 
suitable placement of e.g. acoustic panels or absorbers in a room. 
The parametric design of the project is done through a plugin to the commonly used modeling 
software Rhino. The plugin is called Grasshopper and allows the user to implement 
parametric design through the use of visual programming.  
A setup has been created in order to assist the user in taking advantage of the combination of 
parametric design and the acoustic simulations. The workflow of the setup consists of creating 
geometry in Grasshopper, which is then communicated to the ray tracing algorithm, in order 
to estimate its relevant acoustic properties. Some of the results of the ray tracing algorithm are 
then sent back to Grasshopper for visualization purposes.  
 
 
==PAGE:14==
1.1 Research objectives 
2 
The geometry creation process itself is not of special interest, but rather the information that 
needs to be extracted from the geometry. In order to facilitate the information exchange, some 
scripts are created in Grasshopper, which aim to make it efficient. As the design principle 
behind the setup is that a user should be able to interpret it in a quick way, an emphasis was 
placed on making it intuitive to use, while taking advantage of some of the possibilities of the 
parametric design process at the same time. 
 
1.1 Research objectives 
 
 
The projectвАЩs research objectives are listed in the following: 
1. Investigate the inclusion of acoustic simulations into the parametric design process, so 
that an architect can get a fast evaluation of the acoustic properties of parametrically 
designed rooms. 
2. Implement a ray tracing algorithm capable of handling complex geometry. 
3. Implement a genetic algorithm in order to aid an architect in optimizing his design, by 
finding a suitable placement of e.g. acoustic panels in a room.  
4. Set up a communication link between the acoustic simulations and the geometrical 
model that is simple and intuitive to use.

Modeling techniques

2. Modeling techniques 
 
Computer-aided design (CAD) is the use of computer systems in order to help with the 
creation, modification, analysis or optimization of a design [2]. It is a key tool in modern day 
engineering and has made the design process faster and more accurate than before [3]. On a 
basic level a CAD system takes inputs in the form of commands from the user and displays 
that information graphically [2]. There are a number of different CAD software packages on 
the market today and selecting which to use depends on the aim of the user and the scope of 
the design project. 
In this project, parametric design of is of special interest. It can be defined as a process where 
a set of parameters are assigned to a design object, the design process is then the  process of 
assigning values to these parameters and the relationships between them, in order to comply 
with design requirements, constraints or optimization criterion [4]. One of the key benefits of 
parametric design is the ease of change and flexibility it allows. In recent years the 
expectation of CAD systems to contain information on the entire lifecycle of a design or a 
project has increased [5]. As a result a design will go through multiple stages of engineering 
analysis and optimization during its development cycle. This process of development is 
iterative by nature. The ease of which each design stage can be modified is therefore of 
critical importance. Parametric design is an effective tool in this regard as it can reduce the 
time and effort spent on design adaptations [6]. 
This chapter aims to provide an overview of two of the most common methods of combining 
CAD and parametric design in modern day architectural design. These two methods are the 
combination of the modeling software Rhino and the parametric design plugin Grasshopper 
on one hand and the modeling software Revit and the parametric design plugin Dynamo on 
the other. 
2.1 Rhino and Grasshopper 
Rhino is a CAD software that specializes in the creation of geometry. Due to the ease of 
which complex geometry can be created within the software it is used in many design 
industries [7].  
The structure of the geometry created in Rhino consists of three geometry types: Solids, 
NURBS and polygons. The most important of these types is the NURBS type. NURBS stands 
for вАШNon-uniform rational B-splineвАЩ and is the basis for the creation of complex shapes in 
Rhino. It allows for the creation of curves by defining control points, which influence the 
shape of the curve. Each control point is assigned a basis function which defines how strongly 
the control point influences the shape of the curve at a particular location [8].

The combination of the basis functions of all control points of the curve allows for the shape 
of certain parts of a curve to be changed without affecting its other parts. In NURBS 
modeling, the width and height of each basis function can be unique, which is where the non-
uniform part of the NURBS acronym derives from [9]. 
The statement that the curve is rational refers to the fact that each control point is assigned a 
weight in order to determine to which degree the curve is attracted towards that point. B-
spline refers to the desired continuity when two curves are joined together, which is achieved 
by the use of so called knot vectors [9].  
Rhino is not inherently a parametric modeler, due to the fact that when geometry is created 
within Rhino it is not possible to store any additional information or properties outside of the 
geometrical information. When the process of constructing geometry in Rhino is initiated, an 
implicit history of the process is recorded. As an example, when two curves are created and 
connected with a surface, the user gets one chance to influence the surface, after the surface 
has been created it can only be changed by recreating it [10].  
An external plugin by the name of Grasshopper solves that issue by letting the user create and 
control their own history tree, which then becomes explicit [10]. This means that the user can 
use different parts of the history tree in different models, replace or adjust them, use 
mathematical and logical operators, retrieve and store data otherwise not possible and so on. 
The design process therefore becomes parametric. 
The geometry created in Grasshopper is displayed as a preview geometry in Rhino and is 
called a BRep or Boundary Representation [11]. The BRep is not editable in Rhino unless the 
link to Grasshopper is severed and it is baked into Rhino. 
Programming in Grasshopper is done through a visual programming interface. The visual 
programming language implemented in Grasshopper differs in many fundamental ways from 
more traditional programming languages. In its essence it functions by using components, 
which are similar to functions in traditional programming, and connecting them to other 
components via a series of wires. Each component has both input and output parameters. Each 
output parameter can be assigned to multiple other input parameters of other components but 
each input parameter can only take input from one output component.

As the aforementioned qualities of the program structure can limit the functionality of a

2.2 Revit and Dynamo 
 
 
Autodesk Revit is a CAD software aimed at creating, storing and monitoring the data required 
for a building project. As such it is classified as Building Information Modeling (BIM) 
software. A BIM platform is capable of simulating a construction project throughout all the 
project stages, from early design through the project maintenance [13]. A BIM model is 
therefore a data-rich representation of a project which can be used for various purposes. 
A key difference between conventional CAD models and BIM models is that in BIM models 
objects are defined in terms of building elements and systems, such as spaces, walls, beams 
and columns [14]. The BIM model can therefore contain large amount of information 
regarding each object in a model. This is in contrast to CAD software where objects are only 
defined as graphical entities.

Autodesk Revit is a parametric design tool in the sense that objects (called families within the 
Revit infrastructure) can be assigned certain parameters that can then be modified. These 
parameters can range from geometrical parameters such as the height or width of an object to 
engineering based parameters such as sound absorption coefficient or an objectвАЩs bending 
stiffness. However Revit is not a NURBS based geometry modeler and is seldom used for 
modeling complex, curved or free-form geometries [15]. It is therefore limited in terms of 
geometrical modeling possibilities compared to NURBS based software such as Rhino.  
Recently a plugin for Revit by the name of Dynamo has been under development. The plugin 
aims to facilitate the use of parametric design, through a visual programming interface similar 
to GrasshopperвАЩs. It offers capabilities not available in the base Revit application such as 
constraining parameters based on external inputs and can be used to automate otherwise 
laborious and iterative procedures [16].

2.3 Choice of software for the project 
 
The main goal of this project is to integrate acoustic simulations into the parametric design 
process, so that an architect can get an evaluation of selected acoustic properties of a 
geometry at an early design stage. It is therefore of great importance that the software chosen 
to carry out the parametric modeling in this project is widely used in the field, while also 
being flexible and powerful. Due to these reasons Grasshopper is the software of choice to 
create and extract geometrical properties for the acoustic simulations, as well as visualizing 
the outputs of the simulations. It is the most commonly used, most researched and most stable 
parametric modeling software used in modern architecture [17]. Many big architectural firms 
also use Grasshopper and Rhino as their primary sketching program in the early stages of 
design, which further increases the validity of the choice of Grasshopper. 
While Dynamo offers many of the same functionalities as Grasshopper and shares most of the 
same programming principles, the limitations of the geometry creation within Revit and the 
fact that Revit models are often created at a later stage in the design process are both factors 
that make it less suitable for this project. During the project, care was however taken not to 
conduct the analysis in a way which is wholly dependent on Grasshopper. For that reason the 
acoustic simulations are kept external from Grasshopper. For future work it is therefore 
possible to set up a link to any software or modeler that can extract the required geometrical 
information for the acoustic simulations, be it Dynamo or some other modeler.

3. Acoustic simulations 
 
The aim of the acoustic simulations in this project is that a designer gets a fast feedback of 
specific acoustic properties of a room. In order to meet that demand a ray tracing algorithm is 
implemented. A ray tracing algorithm has been chosen due to how computationally fast it is 
compared to other common simulation methods. A ray tracing algorithm can be expected to 
be accurate at high frequencies; the mid and low frequency accuracy depends largely on the 
volume and the amount of absorption of a room. The larger the volume and the more 
absorptive the surfaces in a room, the more reliable the algorithm becomes at lower 
frequencies. This is due to the fact that ray tracing does not account for the wave behavior of 
sound; at low frequencies the modes formed by the sound waves are distinct on the frequency 
spectra. As the frequency increases, the number of modes increases, they begin to overlap and 
the transmission characteristics of the room become uniform [18]. There are different 
simulation methods that produce more accurate simulations at low frequencies, such as FDTD 
and FEM methods, however they require a substantially higher computation time and are 
therefore not feasible for the objective of this project [1]. 
The ray tracer that is implemented is a traditional ray tracer and follows a procedure originally 
suggested by Kulowski [19]. A code originally written in Matlab by Pind [1], reported in his 
master thesis вАЬBroadband Hybrid Room Acoustic Simulations Using FDTD and Ray 
TracingвАЬ, following KulowskiвАЩs methodology is used as a starting point. The code is rewritten 
in Python and some additional features are added to account for more complex geometry. 
Chapter 3.1 is largely based on PindвАЩs work while the following chapters contain details 
regarding new additions to the algorithm.

surface is modeled as an infinite plane, each having their respective inwards facing unit 
normal vector ni. Each surface is also assigned an absorption coefficient ќ±i and a scattering 
coefficient si. The absorption coefficient indicates how large a percentage of the energy in the 
incoming rays the surface will absorb. The scattering coefficient is a measure of how specular 
or diffuse the ray reflection off the surface will be. Both coefficients are in the range of 0 to 1 
corresponding to 0 to 100% absorption or scattering respectively.

In order to model the receiver, the secondary source method originally suggested by Rindel et 
al. is used [21]. The procedure of the method is that at every reflection point a secondary 
source is created. This source has frequency dependent energy content, which depends on 
how much of the rayвАЩs energy has been absorbed or attenuated since leaving the original 
source location. Once the rayвАЩs energy has decreased by a predetermined amount in all 
frequency bands, the ray is stopped and contributions from all valid secondary sources are 
collected at the receiver.

During a ray tracing simulation a predetermined amount of rays is shot and traced based on 
the principles detailed in this chapter. Once all the rays have been traced a histogram of the 
whole reflection time history is created. This is done by grouping all the reflection points into 
bins, based on the time elapsed from when a ray left the source position to when it hit the 
reflection point. As each reflection point corresponds to a secondary source, the energy

selected acoustic parameters of a room.

3.2 Point in polygon test 
 
In many rooms, surfaces often contain some installations or patches where the absorption or 
scattering coefficients differ from those of the base surface. These installations include 
acoustic panels, carpets on floors, curtains, wall tiling and so on. The ray tracing algorithm 
therefore needs to be capable of detecting which part of a surface it hits, so that the correct 
absorption and scattering coefficients can be assigned during the simulation. For that purpose 
a so called point in polygon test has been implemented. The purpose of the test is to check 
whether a reflection point of a ray is within a certain polygonal surface. Each surface in a 
room is split up into multiple separate 2D polygonal surfaces and the point in polygon test can 
determine which polygon on a surface a ray hits. 
There are different ways to perform a point in polygon test, depending on the calculation 
speed desired and whether the polygons are concave or convex [23]. For this project an 
intersection method has been chosen, where a line is drawn between the ray reflection point 
and a point outside the polygon and the number of intersection points of the line with the 
edges of the polygon are counted. If the number of intersections is even the reflection point is 
outside the surface, otherwise it is inside it. This method was chosen as it has been shown to 
be reasonably fast, as well as being able to handle both convex and concave polygons [24].  
To begin with, the part or parts of a surface whose absorption or scattering coefficients differ 
from the base surface can be modeled as any polygonal shape. When a ray hits the surface the 
procedure of the test is then as follows:  
The point in polygon test is run for one side of a polygon at a time. The first step in the test is 
to define a point just outside of the polygon, hereafter referred to as the outer point. A line is 
then drawn from the outer point to the reflection point of the ray. The line is described by the

The main drawbacks of this method are the unlikely events that the line between the outer

reflection point lies on one of the polygonвАЩs sides. Since it is unlikely that these occurrences 
will influence the results given how unlikely they are to occur, the large number of rays used 
for a normal simulation and the fact that Python, in which the test has been coded, uses 
double precision, meaning that each number has 15 significant digits, these cases are both 
treated as a single intersection event.

3.3 Concave geometry 
 
 
A limitation of the ray tracing algorithm adopted for this project is that in its base form it is 
unable to handle concave geometry. This is due to the fact that because the planes that 
represent the geometryвАЩs surfaces are extended to infinity when calculating the reflection 
points of a ray, planes that penetrate the room interior can cause the algorithm to detect a hit 
and set a new reflection point within the interior of the room. There is therefore a need for a 
test to determine whether a reflection point is on a surface or not. The point in polygon test 
described in chapter 3.2 can be used for this purpose. It is however not feasible to run the test 
at every reflection point for computational reasons. To circumvent the problem a procedure 
originally suggested by Kulowski [23] has been implemented.

To begin the procedure it is necessary to divide the planes that comprise the geometry into 
planes that penetrate the room and those who donвАЩt. The first step in computing that 
distinction is to define a front face of each plane. The front face is the face that the sound ray 
reflects of. If a plane penetrates the room interior it can be shown that there is always at least 
one plane which lies on the back side of the penetrating plane [23].  In order to determine 
whether a quadrangular plane j is on the back side of quadrangular plane i, 16 vectors are 
created, from every vertex point of plane i to every vertex point of plane j. The angle between 
each of these vectors and the inwards facing normal vector ni of plane i is then calculated. If 
the angle is a right or obtuse angle for all 16 cases, plane j is behind plane i, and plane i is 
therefore a penetrating plane. This relationship can be defined by the logical variable Ai,j 
where Ai,j=TRUE if the  jth plane lies behind the ith plane [23].

Following this procedure an algorithm has been created that scans the room at the beginning 
of the simulation and checks for mutually invisible planes. The algorithm stores the visibility 
between each individual plane and which planes penetrate the room interior. 
The modified ray tracing procedure can finally be defined as follows: 
1. Find the hit point of a ray with the planes that extend from the surfaces of the room using 
the technique described in chapter 3.1. Only planes that are visible to the reflecting plane, 
i.e do not fulfill any of the criteria I-III, are considered. 
2. Check whether a hit point that passes criterion 1 lies on a room penetrating plane. If the 
hit point does not lie on a penetrating plane it is a valid point. If the hit point lies on a 
penetrating plane, the point in polygon test described in chapter 3.2 is used to check 
whether the hit point is on the surface or inside the volume of the room. If the point in 
polygon test is failed the hit point is inside the volume of the room and it is therefore not 
valid. 
3. The distance from the starting point of the ray to all the points that pass criterion 2 is 
calculated and the hit point that is closest to the source position will be the next reflection 
point. The procedure is then repeated until the energy content in the ray has reached the 
prescribed threshold.

4.1 Definition of parameters

by 60 dB after a continuous sound source has been switched off [26]. As a decrease of 60 dB 
can be difficult to achieve in practice another definition of the reverberation time is to take a 
smaller interval of the decay curve, from -5 to -35 dB below the start value and multiply that 
value by 2. This is the definition of T30.

The comparisons of the simulated parameters between Odeon and the ray tracer show that at 
medium- and high frequencies the two simulation methods are generally in good accordance. 
At lower frequencies there are however quite substantial discrepancies between the two, 
where the ray tracer tends to overestimate the magnitude of the parameters by a significant 
amount. There are a few possible causes for this difference, firstly Odeon uses the image 
source method for the first few reflections, which is inherently a more accurate simulation 
==PAGE:39==
4.2 Odeon comparison 
27 
method than ray tracing due to the higher resolution of the energy decay curve that it achieves 
[1]. Secondly Odeon handles scattering in a more complete manner, even taking diffraction 
into account, which has more pronounced effect at low frequencies compared to mid- high 
frequencies [25]. The S-shape curve of the scattering coefficient over the octave bands which 
Odeon assumes is also not known and therefore the scattering coefficients used for the ray 
tracer are only an estimation of the coefficients used in Odeon. The scattering coefficients 
used for the simulations are also rather low, which means that the surfaces in the room are not 
particularly diffusive. Increasing the scattering can be shown to yield results closer to those 
predicted by Odeon at low frequencies [1]. 
Another possible reason for the differences is the averaging of the scattering coefficient 
shown in equation 3.1.10 which is used to run the ray tracing algorithm in a single pass. As 
the formula aims to take an average of the scattering coefficients in each octave band it might 
create some errors compared to using the scattering coefficient of each octave band 
separately. A thorough research of the error has not been conducted in this project but for 
future development it is an important factor to investigate. 
Despite these differences the ray tracer can be said to give satisfying results for a fast 
estimation of the selected parameters. The accuracy at mid- high frequencies is more relevant 
due to the fact that the ray tracing technique is in its nature not a good predictor for the low 
frequency characteristics of a room. This is due to the fact that at low frequencies the room 
has distinct modes, spread over the frequency range, leading to modal behavior being 
dominant, whereas at higher frequencies there are many modes that are close together and 
tend to produce uniform transmission [18]. The so called SchroederвАЩs frequency is often used 
as a cutoff frequency, above which the effect of distinct modes can be ignored and ray tracing

5. Genetic algorithm 
 
A genetic algorithm is a search heuristic algorithm that imitates the concepts of evolution and 
natural selection. The algorithm encodes a solution to a specific problem in a binary format, 
mimicking chromosomes found in the DNA of most living organisms. Evolutionary concepts 
such as generations, crossover and mutation are then used in order to find a suitable solution 
to a specific problem. 
The fact that a genetic algorithm is heuristic means that it is not designed to find one optimal 
solution but rather to find a вАШgoodвАЩ solution in a reasonable computational time [29]. This is a 
very desirable feature for the scope of this project for two main reasons:  
Firstly it means that a designer can get an idea of e.g. a sensible placement of absorptive 
panels in a room without wasting computational time on finding the one exact solution. The 
nature of the algorithm also means that the designer has great control of how good an idea he 
wants of the optimal placement of panels as he can easily determine how many iterations it is 
desired to go through.  
Secondly genetic algorithms have some key advantages over other common optimization 
methods for the purposes of this project. If the space to be searched for a solution is large or 
non-smooth in nature, genetic algorithms have been shown to outperform many other 
optimization algorithms such as gradient-ascent algorithms [30]. While in this project the 
analysis will be restricted to simple problems, future development of the tool might require it 
to be used for more complex problems and thus an optimization method capable of handling 
those complexities is more suitable. Another important factor is that ray tracing is in itself a 
stochastic method; the results of different simulations of the same configuration can therefore 
vary, with the variation depending e.g. on the number of rays used. This means that 
algorithms that change a single element of a solution at a time (such as hill climbing 
algorithms) might be led astray by the variations whereas genetic algorithms tend not to, due 
to the fact that they accumulate statistics of how well each solution preforms over many 
generations [30].

5.1 Structure of the genetic algorithm 
 
In the following some of the key concepts of genetic algorithms will be explained. A genetic 
algorithm has been coded in Python and its purpose for this project is to help the user to 
place acoustic panels or absorbers in a room, so that a desired value of selected acoustic 
parameters can be achieved. To begin with, the genetic algorithm generates a candidate set of 
random configurations of panels in a room. The ray tracing algorithm described in chapter 3 
then simulates the acoustic parameters for each configuration. The properties of the 
configurations which result in values of the acoustic parameters closest to the desired values 
are then passed on to a new set of candidate configurations until a configuration which gives 
satisfactory values of the parameters has been found.  
Generations and encoding 
Genetic algorithms aim at taking an initial set of candidate solutions to a problem and evolve 
them into a set of more suitable solutions. A set of solutions is called a generation and each 
individual member of a solution set is often referred to as a chromosome. At the start of the 
simulation an initial population of chromosomes is randomly generated. Each chromosome is 
represented by a binary string for encoding purposes.  
For demonstration purposes a simple example is given in order to explain the different steps 
of the algorithm. The setup of the example is shown in Figure 5-1, where a grid made out of 8 
panels is encoded using a 3 bit representation as seen by the black number strings in Figure 5-

Mutation 
After crossover has taken place each of the chromosomes in the new generation has a chance 
to be mutated at each locus depending on the mutation rate pm, a typical value of which is 
around 0.001 [30]. The mutation is done by flipping the bit at a particular locus of the 
chromosome and ensures variation in the solution, by making sure the chromosomes do not 
fixate at a particular locus.  
Elitism 
Elitism is the concept of guaranteeing that a certain percentage of the fittest chromosomes in a 
generation pass on unchanged to the next generation, without any crossover or mutations. 
This has been shown to improve the genetic algorithmвАЩs performance [30].

6. Combination of parametric modeling and acoustic simulations 
 
In this chapter the setup of the information exchange between Grasshopper and the acoustic 
simulations is discussed. The acoustic simulations are coded in Python, relying heavily on 
the Numpy module. The Python code is not scripted internally in Grasshopper and the 
information exchange process is therefore mainly conducted via text files. This is done for 
two main reasons. Firstly because of limitations with regards to the use of Python within 
Grasshopper and secondly because it is not desired to make the process wholly dependent on 
Grasshopper, but rather to show that acoustic simulations can be included in parametric 
design. This approach ensures that the ray tracer does not need to use RhinoвАЩs internal 
scripting language, Rhinoscript, and can therefore be made to work with any geometrical 
modeler, as long as it can provide the required geometrical information.  
The creation of geometry within Grasshopper is not a special focus of this project, but rather 
the information that can be imported to and exported from any geometry, using the concepts 
of parametric design. The geometry can either be created using Grasshopper or imported from 
Rhino into Grasshopper. Thus it will be assumed that any user of the tool will have the 
required familiarity with Grasshopper to create or import geometry into it and extract the 
necessary geometrical information, with the aid of a few custom made scripts that will be 
detailed in this chapter. 
6.1 Basic setup 
In order to facilitate the information exchange between geometry stored within Grasshopper 
and the acoustic simulation code, a few Grasshopper scripts have been prepared. The user 
creates the geometry and can then use the scripts to setup a link to the acoustic simulation. 
The process of using the scripts can be summarized in the following steps: 
1. Create the geometrical model in Grasshopper or import it into Grasshopper. 
2. Mesh the model so that each surface is a single quadrangular mesh face. This allows

planes, surfaces etc.) 
3. If one or more surfaces consist of multiple materials, draw the outline of those 
materials with polygons, using a premade script. 
4. Assign materials to each surface, using a premade script 
5. Choose what acoustic parameters are to be calculated, whether any surface consists of 
multiple materials, whether suspended elements are present in the model and whether 
the geometry is concave or not. This step is optional but can save computation time.

6.6 Curved surfaces 
 
In contrast to planar surfaces, curved surfaces do not have a constant unit normal vector. It is 
therefore necessary to find the normal of the surface at each reflection point. Since the 
number of points on a surface is infinite, the number of unit normal vectors is also infinite. 
This poses a problem since it is not possible to store all the unit normal vectors in the 
computer memory, as is the case for planar surfaces. There are three methods that have been 
identified in order to account for this problem. 
1. Mesh the curved surface, using planar mesh faces and find the unit normal vector of 
each mesh face, the finer the mesh the better the accuracy will be. Store that 
information in memory and send it to the ray tracing algorithm. The algorithm will 
then look up which face the ray hits and the unit normal of that face.  
2. Use Rhino built in scripting language Rhinoscript and code the whole ray tracing 
algorithm within Rhino/Grasshopper. Rhinoscript can be used to find a unit 
normal vector to a tangent plane at the exact reflection point. 
3. Use Rhino built in scripting language Rhinoscript within Grasshopper to trace 
the rays without creating the energy decay curve and stream that information to the 
external ray tracing algorithm, which then creates the decay curve.  
The first method essentially decomposes the sphere into a finite number of planes which 
reduces the problem to one similar to those described in the previous chapters. This means 
that the Python algorithm would not have to be changed to simulate the surface. It does 
however reduce the accuracy of the simulation, as when the curved surface is approximated 
by a set of planes some of the spreading out or focusing behavior of rays reflecting off convex 
or concave surfaces respectively is lost [32]. 
==PAGE:54==

The second method is more accurate as it will find a unit normal vector to a tangent plane of 
each reflection point on the curved surface. The exact shape of the curve is therefore taken 
into account. This method however requires the ray tracing algorithm to be coded within 
Grasshopper using Rhinoscript. The simulation is therefore limited to Grasshopper only. 
Another limitation is that as that it is difficult to get the Numpy module working in the 
Python component within Grasshopper. This means that the use of built in matrices and 
linear algebra operation are not available. For efficient computations it would therefore 
require the coding to be done in C# or Visual basics, which is outside the scope of this 
project.  
The third method is built to circumvent the lack of Numpy functionality within Grasshopper. 
It functions like the second method in the sense that Rhinoscript is used to extract the 
normal of a tangent plane at each reflection point, information regarding each reflection point 
would then be sent to Python to handle computations of the energy decay curve. Thus the 
scriptable component within Grasshopper would only create the rays and not do any further 
calculations.

maximize the ratio of energy in the direct sound versus the energy in the reflected sound, for a 
receiver located in the middle of the room. Due to the fact that the four panels are much more 
absorptive than the room surfaces, a successful run of the algorithm would result in the panels 
being placed in the four mesh faces the rays are directed to when shot from the source 
position. This is due to the fact that when a large portion of the energy in a ray is absorbed at 
the first reflection point, the direct sound energy will dominate the energy coming from the 
reflected sound and the ratio of direct to reflected sound energy will be high. The setup of the 
room can be seen in Figure 6-13.

As previously mentioned one of the strengths of the algorithm is that it accumulates statistics 
of all the configurations over all generations so it is easy to compare the best configuration the 
algorithm can find with others of comparable quality. A designer can therefore get an idea of 
different configurations that satisfy his demands.  
The biggest issue with the use of the algorithm is the fact that a designer is not likely to be 
familiar enough with the underlying concepts of genetic algorithms to set the number of 
generations or chromosomes in each generation for the whole simulation. This number 
depends quite heavily on the complexity of the simulation as a surface divided into 8 mesh 
faces does not require the same amount of generations as the same surface meshed into 128 
mesh faces. The approach taken in this project has therefore been to set the default number of 
generations to a high value (500) and let the user either stop the simulation when he feels it 
has begun to converge to a satisfactory value or change the number of generations himself. 
The number of chromosomes in a generation defaults to 15 for mesh made out of 64 faces 
with 5 chromosomes being added for each power of 2 added beyond that, so that a 128 face 
mesh will for example have 20 chromosomes in a generation, which has been found to be a 
decent compromise between simulation speed and accuracy over a number of test simulations. 
To make the tool more approachable, further research could be done on the possibility of 
making the algorithm adaptive to the geometrical complexity in a model.

Future work and possibilities

7. Future work and possibilities 
 
The field of parametric modeling is vast and there are many different paths that can be 
explored further than are within the scope of this project. This chapter aims at highlighting 
some of the more interesting and relevant directions for future development of the project. 
7.1  Developing with Grasshopper in mind 
As has been highlighted throughout this project, the focus has been to use Grasshopper for the 
parametric modeling of geometry and visualization, while doing the acoustic simulations in an 
external Python script. It is however possible to code the algorithm within Grasshopper in 
either C# or Visual Basics. The main benefit of doing so is that it would make the 
simulation of complex geometry simpler. As explained in chapter 6.5 it would be possible to 
analyze curved surfaces. It would also enable the simulation of surfaces of any shape and not 
only quadrangular. This is due to the fact that the built in Rhinoscript syntax contains 
functions that can handle intersection calculations between shapes and vectors.  
Another important advantage of constructing the algorithm within Grasshopper would be the 
option to take advantage of parametric design for optimization. An example could be a curved 
ceiling where the tangent of the curve at set intervals would be one of the input parameters. 
The genetic algorithm put forth in chapter 4 could be used to explore the effect of changing 
the tangent at one or more points to get the most beneficial shape of the curve, based on the 
desired design goals. This sort of geometrical optimization is unfeasible in the current state of 
the project as the external Python code does not have access to the input parameters of the 
geometry (the tangents in this case) only the outputs (the final geometry). Thus to explore the 
effect of changing the curve shape the tangent parameter would have to be created within 
Python, written to a text file which would then be fed into Grasshopper in order to update the 
geometry. The main problem with this approach is that it requires a specific setup within 
Python for each problem the algorithm is meant to analyze. The problem of changing the 
tangent of a curve would for example require a different setup and scripting than the problem 
of changing the angle of one of the walls of room. A user would therefore need to possess 
knowledge of the Python code and how to modify it in order to do the simulation, which goes 
against the smooth and iterative workflow of parametric design and cannot be expected from 
a user. If however the algorithm were coded within Grasshopper using Rhinoscript that 
need not be the case, as the only setup required could be done within Grasshopper.  
Yet another advantage of keeping the simulation within Grasshopper is that it makes it easier 
to create a custom Grasshopper plugin with its own components for the various tasks the 
algorithm is meant to solve, instead of the scripts shown in chapter 6 that the user has to copy

would be placed in GrasshopperвАЩs toolbar and would be available to input into all worksheets.  
7.2 Different modeling software 
There are many plugins available to facilitate the import and export of different file formats to 
and from Rhino and Grasshopper. One of the more intriguing of these plugins is Flux. It is a 
cloud based plugin that aims at promoting data exchange between different software through a 
cloud based application. Some of the supported software include Grasshopper/Rhino, 
Dynamo/Revit, Google SketchUp and Excel.  With Flux it is possible to connect data and 
models between different software so that changes in the model in one software will result in 
the same changes being made in the other. The plugin also allows multiple users to work on 
the same model in the same or different software and gives each user a choice of if and when 
to synchronize their model to the cloud.  
This plugin is therefore a powerful tool when it comes to cross software use of the simulation 
and could be a key factor in connecting a Grasshopper based simulation to e.g. Revit in an 
efficient manner.  
7.3 Simplified UI 
It is possible to use plugins for Grasshopper that allow a user to create and edit geometry in a 
very simple manner, without the user having any familiarity with visual programing. These 
plugins enable the user to edit selected features in a predefined geometry. They can therefore 
be used as a way to create a simplified setup for users, in order to familiarize them with the 
concepts of parametric design and demonstrate the effectiveness of the acoustic simulations.  
7.4 Different programming language for the ray tracer 
Two methods have been identified in order to increase the computational efficiency of the ray 
tracing algorithm. The first method is to program it in a lower level programming language 
than Python, such as C. The second method is to take advantage of the computational power 
of the graphics processing unit of modern day computers. This could be done by 
implementing parallel processing, where the GPU preforms some of the more intensive 
calculations. These two approaches could greatly increase the speed of the ray tracing 
algorithm and thereby also make computationally intensive options such as using the image 
source method for the first few reflections and accounting for diffraction more feasible.

Discussion 
 
 
50 
8. Discussion 
 
In this project the setup of the information exchange between Grasshopper and a ray tracing 
algorithm has been explored. The goal of the setup is to establish a direct link between the ray 
tracer and geometry created in Grasshopper, so that a user can get a fast evaluation of selected 
acoustic parameters of his model. After a user has setup his model by meshing it in the 
prescribed manner, he can evaluate the effects of changes in the geometry on the acoustic 
parameters by simply changing the geometry as he wishes and run the ray tracing algorithm 
again.  
A ray tracing algorithm capable of handling rectangular geometry has been expanded upon 
and is now capable of handling concave geometry, suspended elements and detect which part 
of a surface a ray hits. The key to this advanced geometry detection is a so called point in 
polygon test that has been implemented. The point in polygon test is flexible as it can detect 
whether a reflection point is inside any polygonal surface, either concave or convex.  It does 
however increase the computation time of the algorithm, depending on the number of planes 
or polygons that have to be tested in each simulation. For future work it is therefore suggested 
to code the algorithm in a lower level language than Python and parallelize it to increase the 
computation speed. Doing so would also result in some more advanced and computationally 
demanding operations such as the image source method or diffraction estimation techniques 
becoming more feasible to implement.   
The algorithm has been shown to give accurate results at mid- high frequencies while being 
inaccurate at low frequencies. These results were as expected due to the nature of the ray 
tracing technique and can be said to be satisfactory. 
In its current form the ray tracing algorithm has been developed in a way that is independent 
of Grasshopper. This has been achieved by extracting the minimum amount of geometrical 
information required from a model and feeding it to the ray tracing algorithm via text files. 
One advantages of this setup is that it is possible to get the algorithm to communicate with 
any modeling software able to extract the necessary geometrical information. An example of 
such software is Dynamo, through which a connection to Revit could be achieved. One of the 
main drawbacks of this approach is however that it requires more work of the user and 
restricts the possibilities of the simulations compared with coding the simulations within 
Grasshopper. This is due to the fact that the number of different geometrical configurations 
possible to design parametrically is vast. For an external algorithm to be able to interpret as 
many geometrical configurations as possible, a certain framework is required so that it can be 
made certain the necessary information gets extracted from a model in the correct way. That 
framework comes in the form of meshing the geometry, which homogenizes the information 
that has to be extracted from a model. It is however not ideal that a user has to mesh the 
model himself after creating it, as it adds an extra step to his workflow. In order to circumvent 
that problem the ray tracing algorithm would have to be coded within Grasshopper to take 
advantage of RhinoвАЩs scripting language.  
==PAGE:63==
 
 
Discussion 
 
 
51 
The genetic algorithm that has been implemented aims at aiding the user in finding the 
optimal placement of acoustic material in a model without the user having to test multiple 
configurations himself. It has been shown to work for a simple test setup. The genetic 
algorithm was built from the ground up within Python, without the use of any existing 
Python optimization modules. The reason for this approach was to be able to customize the 
algorithm according to the demands of the project, while also having thorough control and 
understanding of every part of the algorithm. Due to this fact most of the development time 
went into coding the algorithm, it was therefore not possible to research and implement more 
advanced methods such as adaptive crossover and mutation rate, due to time constraints. 
These methods might however be advantageous to implement as they might increase the 
efficiency of the algorithm, by making it able to adapt to each problem and therefore converge 
to a good solution more efficiently. 
Due to the aforementioned limitations of keeping the ray tracer external from Grasshopper the 
genetic algorithm has been designed to optimize the placement of acoustic material in a 
model. This optimization process was chosen due to how well it can be defined by a mesh of a 
model, therefore making the information exchange process uniform for different types of 
geometries. For future development it would be interesting to implement the algorithm in 
such a way that it can optimize changes in geometry, for selected acoustic parameters. For 
that to be possible, without having to make changes in the source code for different 
geometrical setups, it is again recommended to code the ray tracing algorithm within 
Grasshopper.

Conclusion 
 
 
52 
9. Conclusion 
 
The main goal of the thesis was to integrate acoustic simulations into the parametric design 
process and explore the possibilities of the connection. This was achieved by developing a ray 
tracing algorithm to handle the acoustic simulations and setting up a series of scripts within 
the Grasshopper programming interface for the modeling software Rhino. The purpose of the 
scripts was to automate the information exchange between the ray tracer and parametrically 
created models.  
A ray tracing algorithm originally capable of handling rectangular geometry was expanded 
upon to account for more complex geometry. The results of the simulation of selected 
acoustic parameters were compared to the room acoustic simulation software Odeon. The 
comparison showed that the ray tracing algorithm and Odeon give similar results at mid and 
high frequencies. The ray tracing algorithm is however not accurate at low frequencies, due to 
the nature of the ray tracing technique. Since the goal was to achieve fast computation time at 
the cost of accuracy this inaccuracy was accepted.  
The goal of the scripts created within Grasshopper were to aid a user to establish a link 
between his model and the acoustic simulation in an intuitive manner and take advantage of 
the direct connection between the model and the simulation. A key factor in the setup is that 
the information exchange process is dynamic, meaning that after a user has setup his model in 
a prescribed manner, the user can freely explore the effect of changes in the geometry on the 
selected acoustic parameters without the need for additional setup. Additionally some options 
for visualizing selected acoustic parameters within the model were explored. 
A genetic algorithm was also created and a connection between it and geometrical models 
created within Grasshopper was established. The aim of the algorithm is to give a user an 
intuition into the optimal placement of acoustic material, such as absorptive panels, in a room 
in order to meet certain values of selected acoustic parameters. The genetic algorithm is 
shown to function as intended, although further work is required in order to make the 
algorithm able to adapt depending on the complexity of the geometry in a model. 
At the start of the project, emphasis was placed on keeping the ray tracing algorithm external 
from Grasshopper. This was done to keep the ray tracer flexible and not dependent on 
Grasshopper. In its current configuration the algorithm can therefore be made to communicate 
with any modeling software able to extract the required geometrical information from a 
model. In retrospect it might have been more appropriate to program the ray tracer within 
Grasshopper. That approach would have made it possible to take advantage of the scripting 
language built into Rhino and allowed for an easier evaluation of more complex geometry, 
less work required of a user and a simpler user interface. For any future development of the 
setup it is therefore recommended to code the ray tracing algorithm within Grasshopper in 
either C# or Visual basics.
