Abstract
Clients executing automated jobs, which access resources that require authentication,
need a way to access the authentication credentials. When accessing resources through
the internet, the client and the server as well as their intercommunications are exposed
to cyber threats. At its core, the security of the server’s resources is based on the
security of the authentication credentials. It is not feasible for automated jobs to rely
on a human operator to provide the credentials, whenever they are needed. However,
storing them on a networked computer system exposes the authentication credentials
to cyber threats as well.
The client, which carries out the automated jobs in the considered system, authen-
ticates itself towards the server by use of username and password credentials. This
cannot be changed nor can the security model on the client side. The thesis aims to
propose a solution to the problem, which elevates the level of security, under these
restrictions.
The proposed solution is based on a combination of the existing authentication
mechanism and using tickets on the server side, which grant clients access to its
resources upon successful authentication.
The thesis has proposed a solution designed to enable automated jobs to run
unaffected, while securing the access to the server’s resources in a way that only allows
pre-authorized access by use of tickets. The tickets are generated and submitted to
the server by another system involving mobile phones, which acts as the second factor
of authentication and authorization.

CHAPTER 1
Introduction
The following chapter will describe the motivation of the thesis. Furthermore, it will
round up the vision and outline the chapters of the thesis.
1.1
Project Motivation
Companies and even private citizens have their servers and computers running services
which retrieve data over the internet from different providers. When such providers
make new and updated versions of their data available, their users and recipients
want the new data as quickly as possible. This could be the case in the stock markets,
when brokers need to decide when or whether to buy stocks or not. In companies
which rely on data in their databases when making business decisions, this might
mean adding or updating the data in the databases. Data providers may even have a
schedule telling their users when to expect updates to be made available. Thus users
can request the new data at the moment it is available instead of sending requests out
more often without getting replies. However, data providers may not want to make
all their data accessible to all their users or they may want to know which of their
users have downloaded what data and when it was downloaded. For such purpose a
data provider may require their users to authenticate themselves. Although it can be
done in other ways as well [Fei88] [Way+04], authentication is commonly carried out
by a user providing the authenticator with a username and a password, which are
used to decide whether or not to grant the alleged user access to the resource she is
requesting.
This is fine in home banking and other scenarios when a user is expected to be at
a computer completing her task manually. In such cases users simply type in their
username and password to prove that they are who they claim to be and to get access
to e.g. their own bank account instead of someone else’s. In the case of computer
systems running automated jobs to fetch data, there is no user to type in a set of
credentials, a set of username and password, so how can it authenticate itself? It
can only do so if it has access to the credentials. In terms of security, however, it
is neither advisable nor desirable to have credentials written anywhere in the source
code of a system. If credentials are written in the source code, an attacker will have
access to the credentials and be able to use them, when she has managed to get
past firewalls and other perimeter defenses and thus has access to files on the system
behind such security measures. If credentials are encrypted, an attacker will not be

data is inherently difficult1 [ElG85, Introduction] [Kob87, 4. Cryptosystems] without

A lot of the interaction and communication in the world is performed through the
use of computers over the internet. Gathering information, reading news, banking
and communication with public services like government institutions can be done a lot
faster using the internet than otherwise. Some of this communication, e.g. accessing
free online news or searching information about a topic in freely available white papers,
can be done without the need for anyone to prove who they are. Other communication
like banking is dependending on the bank’s computer knowing which customer it talks
to, even though it is really talking to another computer. The user is also dependent
on knowing that it is in fact the bank’s computer that she is communicating with,
when she is trying to log in to e.g. her bank’s home banking system. If not, a user
could unwillingly be sending her credentials and other valuable information to an
attacker.
There are different ways for a user to assure another party that she is who she
claims to be. Authentication is critical to a wealth of communication types in com-
puter systems and there is more than one way to do authentication. While the best
known is to use a username and a password, other methods include using biometrics
like fingerprint scanning. The different authentication methods have their upsides and
downsides, which make some better suited for certain scenarios than others. As com-
munication is only moving faster towards digitization, the need for new and maybe
differentiated2 ways of authentication and authorization rises. New problems emerge,
when processes are to be handled by computer systems, while a high level of security
is still needed.
Thus there is a need for solutions to the problems concerned with secure au-
thentication and authorization in automated jobs. There are relatively new fields of
cryptography that may be applicable to help solve some of these problems, which are
interesting to look into.
1.2
Vision
The vision of this thesis is to propose a solution, which require minimal human in-
teraction, to the problem of authentication in automated jobs, which strengthens the
security. The proposed solution should aim to balance usability, described in Sec-
tion 1.3.3 Usability, and security, described in Section 1.3.2 Security, in automation.

1.3
Project Description
The thesis aims to propose a system which relies on Two Factor authentication and au-
thorization by adding a transparent layer of authentication and authorization. Trans-
parent in this context means a layer that does not affect the consumer’s system, which
needs to authenticate itself towards an authenticator. Or, in cases where it is possible,
a system which can make credentials available to automated jobs only when needed
by providing clear text credentials to present to the data provider for authentication
and authorization purposes.
Due to the assumption outlining the restriction that no changes can be expected
to be made on the consumer’s system, as described in Section 1.3.1.1 Consumer
Provider, and that the primary scenario for this thesis is exactly that scenario as
stated in Section 1.3.1.6 Primary Scenario For The Thesis, the main focus of the
thesis is on proposing the transparent solution.
1.3.1
Scenarios
Below are the scenarios in which the proposed solution or derivations of it could be
applied to solve the issue of automated authentication and authorization.
1.3.1.1
Consumer Provider
The scenario is shown in Figure 1.1. The Consumer System is an untrusted entity
since it is not possible to enforce rules on the Consumer’s systems or configuration
of such. This includes how to store credentials among other topics. It is assumed
that the Consumer System stores its credentials in an unsecure fashion. An employee
could have written it down on a piece of paper or stored in clear text on a computer
that is connected to the internet.
The Provider System can only control its own
endpoints to where authorized Consumer Systems can make data requests, since it is
not in a position to make demands on the Consumers’ systems and configuration of
these.
This scenario concerns itself with the situation in which the Provider is either obli-
gated to or has keen enough interest in providing data for a Consumer that either can
or will not update or change its system. This type of Consumer is running a legacy
system. In addition, it does not have employees with the skillset to make the neces-
sary changes to be able to support other forms of authentication than credentials of
username and password. The authentication part of the “getData + Authentication”
in Figure 1.1 is indeed this username and password.

1.3.1.2
Sign Code
The scenario is shown in Figure 1.2. The Provider produces hardware, which only
accepts new code for firmware updates if such code has been signed. The Provider has
a private/public keypair for an asymmetric cryptography algorithm, which is used for
this purpose. The Provider keeps the private key stored only on a hardware device
and has rules that explicitly say that this device must not under any circumstances
be connected to a computer which is connected to the internet. When a piece of code
needs to be signed such that it can be uploaded to the Provider’s hardware, a manual
process is currently in place.
The process involves employees who are required to physically move to a special
office to be handed the hardware device storing the private key. This hardware device
is then to be connected to a computer without internet connection to sign the code.
Neither the hardware device nor the computer used to sign the code can ever leave the
special office. Afterwards the signed code can then be uploaded to the distribution
server from another computer than the one which was used to sign the code.

1.3.1.6

The primary forcus will be put on solving the problem of the scenario described in

1.3.2
Security
Authentication can be carried out using different protocols or algorithms. E.g. the
challenge-response protocol which does not necessarily require a password, if the
algorithm is the secret. A challenger in possession of a resource will issue a different
challenge each time a requester asks for access to the resource. If the requester knows
the algorithm, he can compute the correct response to each challenge. In this case
there is therefore no need for a specific secret password.
1.3.2.1
Security Regarding The Primary Scenario
When a job needs to authenticate towards a data provider that requires authentication
by username and password, the job will need a set of clear text credentials to send
==PAGE:21==
1.3 Project Description
9
to the authenticator, e.g. the data provider, to prove its identity. However, since the
restriction limits the solution design from making changes to the consumer’s system,
the consumer’s system will continue to work as it is. This means that it is assumed
that the consumer’s system stores its authentication credentials in a clear text file or
has them hardcoded in the source code of its automated jobs.
This is of course not optimal, but the security of the solution design will therefore
focus on creating a system in which authorization of each data request from automated
jobs have to be present in the data provider’s system at the time of a data request or
be granted on the fly.
1.3.2.2
Security Regarding The Other Scenarios
When considering scenarios that are not bound by that restriction, the security will
rely on storing authentication credentials in cipher text only. Thus even if an attacker
gets access to credentials, they are in cipher text and thus useless until decrypted. If
a system stores credentials in cipher text only, it will need to receive the decryption
key for any given set of credentials, when attempting to authenticate towards an
authenticator, from e.g. a human operator.
1.3.3
Usability
Usability and security are often counterparts. For example, a computer that does not
require a password can be accessed quicker than a computer with password protection.
The downside of having no password protection is that everyone can access and use
the computer which poses a problem in cases where it matters e.g. which user carried
out which commands.
1.3.3.1
Usability Regarding The Primary Scenario
Enabling users to provide the authenticator, e.g. the Provider System in Figure 1.1,
with authorization tickets for each data request while being mobile could solve the

1.3.5
Current Solutions
There are multiple adopted solutions for tackling the problem of authentication in
automated jobs like Password Authentication, Public-Key Authentication, Using an
Agent, Trusted-Host Authentication and others [BS01, Chapter 11 Case Studies]. How-
ever, only Using an Agent, which is also the recommendation of the publication, is
of interest, because the authenticator has no authority to demand or change the way
authentication is performed by the authenticating system. The agent is a program,
which a human operator invokes and afterwards loads with the passwords it should
have. The passwords are stored in clear text, but only within the memory space of
the running agent instead of a file on disk. Every time the agent is invoked, a human
operator has to load the passwords. The agent could be shut down or the system
running the agent could be shut down. When either of those happen, the agent will
have to be invoked again.
It makes it harder for an attacker to get hold of credentials, since it is harder to
make a memory dump than it is to read a file. Though for as long as the agent runs,
an attacker, who has gotten access and who has the means to make a memory dump,
can get hold of the clear text passwords.

1.3.6
Solution Proposal
Location Based Encryption [Sco03] relies on presence at a geographical location to
decrypt cipher texts. An automated system will most likely be run on stationary
computers. Therefore, Location Based Encryption is not the best option or should at
least not stand alone. Restricting decryption or authorization by other factors such
as time, privileges, other attributes or even a combination would be interesting to
this problem. As opposed to an agent, in which the keys are kept in clear text in
memory for the entirety of the agent’s running time, this approach may enable the
ability to keep the secret in cipher text until the time it is to be used.
1.3.7
Technologies To Realize The Solution
It may be interesting to apply a combination of Kerberos [NT94] and a type of cryp-
tography to the problem, which can help limit how, where or when an authorization
token will authorize a request or a key can be used for decryption like with Loca-
tion Based Encryption or Attribute Based Encryption [Goy+06]. Users’ handheld
and portable devices could also be brought into play in solving the problem, while
upholding a high level of security.
1.3.8
Alternative Authentication Methods
The scenario described until now is under the assumption that the authentication
method in the data provider is based on a set of credentials consisting of a user-
name and a password and that this method cannot be changed. If, however, the
authentication method can be chosen or changed without restrictions, there are other
authentication methods to consider. Some of these are described below.
1.3.8.1
Differentiated Authentication
As mentioned previously in ?? ??, the term differentiated authentication is used for
a service, which has two or more intended use cases. Each use case may have its own
authentication method.
A scenario could be that the data provider runs differentiated authentication. One
type of authentication for manual data consumption and another type for automated
data consumption. The concept is somewhat similar to the one described in [BB15]
although the referenced article concerns itself with differentiating authentication by
user instead of differentiating by service.
Manual data consumption would be defined by non-periodic requests for data. As
such it can be assumed that a human operator instantiates the request and therefore
the authenticator will require a username and password. Automated data consump-
tion would be defined by periodic requests for data. As such it can be assumed that
a computer instantiates the requests and therefore authentication could be required

to be evaluated using e.g. challenge-response authentication or one-time passwords
received on the fly from users using their mobile devices.
If this differentiation is based on the assumption that requests using username
and password are carried out manually by a human operator only and the level of
security hinges on this, there is a problem with this authentication setup. The data
provider will have little or no ability to enforce this rule on their consumers. Thus in

1.3.8.2

A third scenario could be authentication exclusively through one-time passwords.
However, if a data provider authenticates through a set of username and password,

required by a system and thus its implementation to authenticate using one-time
passwords scheme. The obvious upside of using one-time passwords scheme is that
even if an attacker manages to get hold of a single one-time password she will not be
able to gain unrestricted access even if the breach is not detected. This assumes of
course that the attacker cannot guess the next password in line even if she sniffs two
or more passwords.

There are a lot of repetitive tasks that can be performed efficiently by computers
running programs that initiate jobs on a schedule, such as a client fetching data from
a server on a regular basis. When such jobs need to access resources from hosts or
systems that require authentication, authorization or both, there are two opposite
extremes of approach, neither of which are ideal.
The first approach is to completely disregard security. This can be done by e.g.
storing credentials for authentication or authorization in files or databases or hard-
coding them into the source code of a job that needs to authenticate.
The other approach is to insist on interaction from a human operator for inputting
the credentials for the authentication or authorization. Whether the job needs to run
several times an hour or once a year, this is cumbersome and impractical.
It is necessary to find middleground that takes ease of use into account along
side security. The following Section 2.1.1 Primary Scenario will review a specific sce-
nario in which the authenticator is dependent on the consumer of whom it requires
authentication. There are lots of scenarios in which it is the consumer who is de-

freely from available authentication schemes. In the primary scenario of this thesis,

denoted Provider System and the system that is required to authenticate itself to-

also have been called the server and the Consumer System could have been called the
client, since it is basically a client-server setup.

2.2.1
Using A Personal Device To Strengthen Password
Authentication From An Untrusted Computer
As stated in [Bra+06, Abstract], authentication of users in a computer system nor-
mally relies on either something you know, something you are, something you have
or a combination of two or all three of them. The concept of the article [MO07, Us-
ing A Personal Device To Strengthen Password Authentication From An Untrusted
Computer] is to keep the password, something you know, out of the computer, which
is used to access personal information, e.g. bank account balance. This is necessary
since the computer is assumed to be untrusted. The argument is that the computer
can be a public computer, which is not that likely at the time of this thesis, since most
people have their own personal computer. However, a personal computer is subject
to viruses, malware, spyware, etc. as e.g. a keylogger. A smartphone is an embedded
system but also a computer. Thus it is also subject to software viruses. However,
since a smartphone runs far less software, it is easier to maintain a smartphone’s
trustworthiness compared to that of a computer, as stated in [MO07, Introduction].
This is due to the smaller attack surface on a device running less software. In case of
a keylogger, an attacker would gain access to any information input via the keyboard.
In that case any username and password combination input through the keyboard
would be disclosed to the attacker.
Through the use of a trusted mobile device, namely a smartphone, the user could
input her password on that device and encrypt the credentials. In this way, even
though the credentials for the authentication are sent through the untrusted computer
to the server, the computer cannot read the credentials. The protocol resulting from
this is called MP-Auth (Mobile Password Authentication).

2.2.2
SSH The Secure Shell
SSH is a protocol for establishing a secure line of communication over an insecure con-
nection, e.g. the internet. It is a great resource for automation, since it can be used
to run commands, batch scripts, cron jobs or even programs on a remote computer.
Batch scripts, cron jobs and other automated tasks can benefit from the security pro-
vided by SSH. However, for unattended jobs, the primary challenge is authentication.
How can a client prove its identity when no human operator is available to type a
password? As stated in [BS01, section 11.1, 2nd paragraph]:
“Note that any kind of unattended authentication presents a security problem and
requires compromise”, SSH is no exception,
and
“Selecting a technique is a matter of understanding the pros and cons of the avail-
able methods, and picking your preferred poison.”
Most people are familiar with using a username and password to login to various
services on the internet, e.g. email, online banking, etc. Passwords can be hard to
remember and thus people are prone to use the same password across services, using
too short passwords, easily guessable passwords like “qwerty”, “password”, “admin”,
“1234” or the like. Social engineering comes into the picture, when passwords are
constructed by names or wordplays on items from peoples daily lives like the name
of their dog concatenated with the number 1. These examples are all problematic,
because they increase the likelyhood of a successful attack on their password, which
can enable an attacker to pose as her victim.
Password Authentication in unattended jobs requires the password to be avail-
able to the code without human interaction. This is only possible if the password is
stored in clear text in the system. It is generally not recommended to use password
authentication in unattended jobs, but there are scenarios in which it is the only
option. [BS01] also mentions that “[...] more methods tend to be available on the
server side to restrict logins with public-key authentication [...]” and for unattended
jobs it is crucial that they are limited to only be able to perform the task they were
intended to. In addition to these parameters, passwords are analogue to the secret

key in symmetric encryption in the way that if either the client or the server is com-
promised, the password is compromised, since the client must reveal its password to
the server in order to authenticate itself.
Public-Key Authentication is based on an asymmetric cryptography algorithm
as e.g. RSA. A client can use a combination of its own private key to sign a message
and then encrypt the message with the server’s public key.

Here the benefit is that the client does not need to reveal its private key to the
server for the server to verify the signature and thus authenticate the client. Still the
question of how to store the private key remains. Either it can be stored in clear text
on the file system or an agent can be used. Storing the private key in clear text makes
it vulnerable to attackers who have gained access to the system. Using an agent is
preferable, but not always viable. The use of an agent has an impact on security as
well as automation compared to storing the private key in clear text.
The security is stronger when using an agent compared to storing the private key
in the file system, since the use of an agent would mean that the key would only be
available in clear text in the memory space of the agent. In the file system it will
be stored under encryption. As stated in [Sil+12, p. 124] “Recall that, normally,
the operating system tries to prevent one process from accessing another process’s
memory.”
Whereas a file on disk is available to any process that lives up to the
Operating System’s permission requirements set on the file or has authorization to
change the file’s permissions for access to it. And since it is harder to gain access to
the memory space of a process, i.e. a running program, than to a file on a file system,
the solution provides better security.
The automation suffers a bit, since a human will be needed to input the passphrase
for the agent after a reboot. Such a system would probably be running with long
uptime and thus this should be a fair tradeoff if it is feasible.
General Precautions for Batch Jobs prescribe using least-privilege accounts
for unattended jobs and to consider one account per job. In addition each key should
only grant restricted access to resources necessary to perform the job.

reveals that something can be done to improve the security of such automated jobs.
An analysis of the attacker model in the scenario is covered, assumptions about all

for an attacker under the current setup is described.
2.3.1
The Problem
If an attacker gets hold of a set of credentials, a username and corresponding password,
she can gain access to the Provider’s system and retrieve data, which is confidential4
due to the “EU General Data Protection Regulation”5, which defines rules on data
protection as well as sanctions and fines if the regulation is violated.
Consumers are not computer or software companies. Thus it cannot be assumed
that they are up to the task of defending themselves against hackers [Cha96]. If an
incident occurs, they may not even know that or when they have been hacked and
much less know what data would have been compromised and possibly stolen from
them in that case. This is of course guesswork, but it serves to illustrate the worst
possible case.
The fact that the Consumers are not computer or software companies also means
that they do not employ people with the skill set to change the configuration of their
IT systems. As described in Section 2.3.2 Restrictions, this imposes further compli-
cations in designing a solution and poses limitations to what attacks and problems
are possible and not possible to mitigate.
2.3.2
Restrictions
In the case of the specific scenario the Provider is obligated to work with and serve
Consumers of various size, skill level and financial capabilities without the right to
deny cooperation with such Consumers. This obligation applies even if Consumers
cannot or will not update their systems to comply with new security standards. Thus
the Provider cannot freely update or change the security model of its services, if it
imposes some of the Consumers to update or change their systems. Effectively, this
implies that the Provider’s current system must support authentication only through
the combination of username and password.

The overall objective is to elevate the level of security. This objective can be

2.4.2
Automation Issue
It is assumed to be crucial that more than one employee of a Consumer can perform
the second factor of the authentication and authorization process, namely the autho-
rization, such that automization can be upheld. The first factor, authentication, is
the system using the username and password. If only a single person is able to per-
form the second factor, authorization, there is a risk that it will be forgotten, pushed
aside by other tasks or will be an annoying task to do such that it will be ignored over
time. That is why it is critical that more than one person can perform the second
factor.
Utilizing mobile devices could make the act of performing the second factor mobile.
This would greatly increase the ease of use, since it would allow Consumer employees
to perform the second factor wherever they are and at a given time, barring lack of
internet connection. Thus releasing employees from having to be at their work station
to do it. Thereby there is a better chance that the authorization is performed in time.
Furthermore, there should be two approaches to the second factor, namely On
Demand and Ahead Of Time.
In the case that the Provider does not have an authorization for a data request
from a Consumer at the time it receives the request, the Provider would request an
authorization from the Consumers employees On Demand.
The Ahead Of Time on the other hand should enable a Consumer’s employees to
provide the Provider with authorization for specific data requests, before the Provider
receives these data requests. Each such predefined authorization should include re-
strictions on e.g. the timeframe in which it is valid. Other restrictions could be on
the type of data it authorizes, etc. This would enable the Consumer to retain the
notion of automation.
2.4.3
Limiting The Timeframe Of A Request
The Provider will enforce a rule towards the Consumer on data requests such that
requests for data no longer will be answered if the Provider does not have an autho-
rization ticket for the data request from the Consumer. The Consumer needs a way
to fulfill this requirement and since it is assumed that Consumers can or will not
change their systems, the Provider will need to make such a way available to their
Consumers. This will be addressed in Section 2.4.6.1 Adding A Transparent Layer
Of Security.
The reason behind limiting the timeframe of a data request is to minimize the risk
that an attacker could successfully request data just by having access to a Consumer’s
credentials and their systems via a cyber attack.
If an attacker is already inside
a Consumer’s system, she might be able to extract the requested data from the
Consumer’s system by sending it elsewhere over the internet.
However, limiting
requests would cause restrictions on such efforts from an attacker, since she would
then have to rely on the current scheduled jobs. At least this limitation would force

2.4.4
Limiting The Origin Of A Request
The reason behind limiting the origin of a request is to make sure that only requests
coming from certain IP addresses, such as the Consumers’, will be accepted and
subsequently evaluated. For this purpose the Provider should store, update and use
an IP address whitelist.
If a request does not come from an IP address already whitelisted by the Provider,
it would simply be discarded like a firewall drops IP packets, which do not meet its
requirements. This way an attacker, who has gotten hold of a Consumer’s credentials,
would not be successful in her attempt to retrieve data from the Provider, if she sends
her request from a source IP address other than one of the whitelisted IP addresses.
That is, barring she spoofs the IP address [Bel89].
This security effort relies on implementation in the Provider’s systems. Since the
Provider is the entity that exposes a web service, from which data can be requested,
it is also the Provider that can address the issue by limiting the origin of requests
for its web services. The best choice would be one that limits the origin in a way
that cannot be spoofed. This can be assured [EL09], but to do so requires specific
spoofing defenses, which may not be practical or possible to implement or enforce.
The obvious choice would be to limit the origin by whitelisting IP addresses or possibly
==PAGE:48==
36

IP ranges. This might proof difficult if Consumers generally do not have static IP
addresses. On the other hand, if they have static IP addresses, it does not present a
problem. Whitelisting IP addresses or ranges, however, does not solve the problem of
spoofing, since IP packets from a whitelisted IP address would be accepted, whether
it is spoofed or not.
As stated in [EL09] spoofing is still a problem: “Although many may feel the
network community solved the spoofing problem through the widespread adoption of
ingress and egress filtering, attackers can still spoof a significant portion of existent
IP addresses, often any IP address”.
Even attackers using botnets can have use
of IP spoofing, since the discovery of a bot in an attack could result in bots being
disconnected from the attacker. So even though an attacker is not directly exposed
when using bots, she would probably prefer to be able to reuse the bots.
IP spoofing can generally be adressed at two levels, namely at the receiving end-
host of ip packets or at routers. The host-based defense solutions can be divided into
active and passive measurements. Passive being hop-count filtering which comes with
inherent problems of false-positives. Among the actives are IPsec, different probing
e.g. TCP and SYN cookies, and IP puzzles. The cryptographic IPsec protocol would
solve the issue of spoofing, but it requires both parties to implement and use it.
It is also not enough for the Provider to only expose its web services over TCP.
Although TCP requires a three-way handshake in order to set up a connection and
begin communication, it is not enough. If an attacker has control over a Consumer’s
systems, she can initiate a connection from a host on that system and spoof the data
request from another host at another IP address. Some of the router solutions that
are based on host-based solutions like IPsec or IP puzzles also require both parties
to use routers that implement them. The other router based solutions require both
parties to use routers that supports the same methods to be effective.
Potential issues to be aware of are:
• IP addresses can be spoofed, which would render the origin limitation indifferent
in case an attacker knows which IP addresses are whitelisted.
• If the Consumers do not have a way of authorizing IP addresses themselves,
they are dependent on the Provider to update the whitelist in case their IP
address changes.
• In case the development of a mechanism, which lets the Consumer embed a
specific IP address for their authorization ticket, is contemplated, it should be
considered how or in what capacity that mechanism can be trusted. In case
of the mobile devices, Section 2.4.6.1 Adding A Transparent Layer Of Security,
being used to generate these authorization tickets, the trustworthiness relies on
the “Trust On First Use” [TV13, 2.1 Trust on first use model] parameter, unless
a design is presented to circumvent this issue.
• In case of IP-range whitelisting, botnets could have hosts in the whitelisted
IP-range.

• Send a request that matches the type of request that is pre-authorized.

gained access to a Consumer’s computer systems and has not been detected. In this
case the proposed solution only limits the attacker in her actions to retrieve data
from the Provider, e.g. by timeframe. The attacker will be able to send requests for
data from the Consumer’s computers and may also be able to exract the data from
the Consumer’s computers to any other computers over the internet.

2.4.8.1
Additional Security Needs To Sign Code Faster
The protocol described in Section 2.1.2.1 Sign Code assures a high level of security
with regards to keeping the private key private. The protocol requires physical in-
volvement to access the private key, which is cumbersome and slow but keeps the
private key off networked devices and thus out of the internet’s reach. The protocol
is easy to follow. However, it is subject to human errors and it takes time to carry
it out, since it requires that a person moves to the office in which the private key
is kept. The possibility for human errors will not be addressed, since it is deemed
unlikely that this part of the protocol would be broken easily.
Changing this protocol to enable access to and use of the private key through a
networked device, opens the private key up to compromise and breaches. Thus it is
a cost-benefit decision whether or not to stick with the current solution or consider
a faster method to sign code. Such a method would involve making the private key
accessible through network by storing it in encrypted form, e.g. in a database. Access
to the key could be based on different approaches and limitations. As previously
described, the origin of the request to access the key could be set to the internal
network of the Provider. This would open up the possibility to access the key from
anywhere on the physical premises or from somewhere else through the internet by
use of a VPN connection.
In case the Provider would consider a method that requires the private key to
be stored on a networked device, the subsequent primary consideration is concerned
with the frequency that the Provider signs code. If code signing is an action that is
infrequently performed, then there is not much of a reason to advocate for a faster
method than the current protocol. Not the least bearing in mind that it drastically
increases the risk of the private key being discovered and stolen. The risk that some-
one could discover the key without being detected also exists. This would jeopardize
the guarantee that the meters cannot be tampered with. If the key was to be stolen
and it was detected, at least all meters could be recalled.
As in the case of the primary scenario, any device should be required to be au-
thenticated by the Provider before being granted access to retrieve the private key.
This could be a mobile device or a computer. A further security measure would be to
overwrite the private key from memory on the device, after it has been used to sign
==PAGE:53==
2.4 Additional Security Needs

the code. This way an attacker’s window of opportunity on the private key would be
further limited.
Currently mobile devices may be the better choice due to a smaller attack surface
and generally more strict operating system in terms of what software is allowed to
run on it. This is at least the case when talking iOS [Mil11].

2.4.8.3

The adoption of continuous delivery in software companies is getting better, whether
they call it “DevOps”, continuous delivery or something else. The idea is to get prod-

as much of the software pipeline as possible. This is great, since it results in faster
updates to software in production and thus correcting issues at a fast rate. However,
if a business is concerned with a high level of quality control, continuous delivery

the test suites, which the software must pass to end up in production.
Even though it opposes the concept of continuous delivery, it might be preferable
to inject an authorization step. This step requires a test manager to authorize code

is important to make a system available to the test managers, which they can use to
authorize code wherever they are. In this way, they will not be more of a bottleneck to
the software pipeline than absolutely necessary. It would probably also be preferable
if the test managers were given notice every time they were requested to issue an
authorization, such that the pipeline is not stalled due to forgetfulness.

3.3
The Tickets
The reason for using Tickets is to authorize each data request coming from the Con-
sumers from another device than the Consumer System, which sends the data requests.
This authorization will either be a pre-authorization or an authorization on demand.
The pre-authorization ensures that a Ticket, which matches an incoming data request,
already exists at the time the Provider System receives a data request.
The Provider System will use the Tickets to authorize data requests from Con-
sumer Systems. Therefore the Provider System needs a way to verify the validity
of each Ticket to make sure that the Ticket has been created by an entity that the
Provider System trusts, e.g. an authorized user. This measure is needed in order
to prevent an attacker from being able to issue Tickets, which the Provider System
accepts as means for authorization of data requests. This will be done by using asym-
metric cryptography to sign each Ticket with a private key, which authenticates the
user, who created the Ticket, towards the Provider System.
The Tickets will contain information about what specific Consumer System it
authorizes to request data, the timeframe which it validates a data request for as well
as information regarding restrictions on the kind of data that the request may access.
The structure of a Ticket is shown in (3.1).

3.4
Ahead Of Time Or On Demand
As described in step 2 in Section 3.1 there are two approaches to Ticket authoriza-
tion in the system, namely On Demand and Ahead Of Time. Figure 3.2 shows the
Ahead Of Time approach. This approach requires Users to have issued Tickets to the
Provider System for data requests before such data requests arrive at the Provider
System. This is the primary approach to minimize the human interaction in the oth-
erwise automated process of requesting and receiving data. However, to assure that
Tickets are available to the Provider System, when it receives data requests from Con-
sumer Systems, the Provider System should keept track of each Consumer Systems’
schedule. In this way, the Provider System will be able to send Ticket requests to
each of the registered Mobile Devices associated with the Consumer System, whose
data request is missing authorization. E.g. if a certain Consumer System requests
data once a day, the Provider System could hold a maximum of 50 Tickets. When
less than half of the Ticket capacity, e.g. 24 Tickets, are left, Ticket requests, asking
for 26 Tickets, could be send out to each mobile device. This way Users would not be
requested to issue Tickets very often and the Consumer System would run seemingly
automatically.
The On Demand approach is based on that the Provider System does not already
have a Ticket for a specific data request, upon receiving it. In this case the Provider
System can send Ticket requests to its registered Mobile Devices, which are associated
with the Consumer System in question, prompting Users to issue the authorization
Ticket for that data request. Only a single User needs to respond for the data request
to be successfully complied with. If the Provider System exposes services for different
types of data requests, an On Demand Ticket could be usefull in case a Consumer
System requests a type of data that it normally does not. Or simply if a Consumer
System sends a data request out of schedule.

• Store Tickets received from the Mobile Devices.
• Check all information from packets when receiving data requests.

Store Tickets: The Provider’s system is responsible for handling data requests,
as is already the case with the existing system. Under the new design the Provider
System will need to store the Tickets in order to validate whether or not authorization
for any given data request exists and act accordingly.
Check all information from packets: When the Provider System receives a data
request, it should timestamp the request and note the source IP address and other
such information to check against authorization Ticket(s). Doing this while also con-
ferring with the relevant Ticket, should reduce the risk of successful IP spoofing or
timeframe tampering.
Store IP addresses: The Provider System should store IP addresses on all its
Consumer Systems. The combination of checking stored IP addresses against the IP
information of a Ticket and checking the IP address of a data request, should reduce
the risk that an attacker could successfully retrieve data from any host. Thus limiting
successful retrivals to be carried out from hosts on Consumer Systems.
Store schedule: The Provider System should store Consumer Systems’ regular jobs’
schedules. This way the Provider System can inform the Users that an On Demand
Ticket request is an abnormality, if it is received out of schedule. And even so if the
data request is of the same type and origin as the regular data requests are.
In this design the Provider has to validate whether or not each data request has
been authorized. This is done by matching each data request with the Tickets in
store. Depending on whether the Tickets are hashed or not, the Provider System has
different approaches to the matching process.
If the Tickets are hashed, the Provider System will have to make up its own
temporary Ticket from the attributes associated with the data request, when it is
received.
E.g.
the time of reception, the origin and what type of data the data
request is requesting. The Provider System will have to hash the temporary Ticket
to compare it with the stored hashes of existing Tickets. Finally it can determine
whether or not the request has been authorized. If the request has not been authorized
by an existing Ticket, the Provider System can send an On Demand Ticket request,
as described in Section 3.4 Ahead Of Time Or On Demand.
In case the Tickets are not hashed, the Provider System has all the attributes
available for individual comparison with the attributes of the data request. Thus
the non-hashing approach will give the Provider System a margin for error. E.g. if a
Ticket has specified a timeframe, which does not match exactly with the timestamp of
the reception of the data request, the Provider System could be configured to accept
it as authorization within a defined margin.

3.9
End To End Encrypted Data Transmission Between
Mobile Device and Provider System
The responsibilities of the data transmission between each Mobile Device and the
Provider System are:
• Confidentiality. The communication between the Provider System and the
Mobile Device must be confidential.
• Integrity. The communication between the Provider System and the Mobile
Device should provide integrity.
Confidentiality: The line of communication between the Provider System and
the Mobile Device needs to provide confidentiality[Mat+06, 1.1.1 Confidentiality] to
protect against evesdropping. This will be guaranteed by scrambling all messages
using cryptography. Thus only the intended recipient will be able to decrypt and
==PAGE:66==
54
3 Design
read the contents of the messages.
Integrity: To guarantee the recipient of a message that the message has not been
tampered with during transmission, integrity[Mat+06, 1.1.2 Integrity] can be assured
by accompanying each message with a cryptographical hash of the message.
Since all data, including the Tickets, sent between the Provider System and the
Mobile Devices, will be sent through WiFi or a cellphone network and thus through
the internet, there is a need for end-to-end encryption on the transmission. This is
due to the fact that the internet cannot be trusted. Even WiFi networks cannot be
fully trusted [BGW01] [TB09]. Therefore, the application will need to establish a
secure line of communication over an insecure connection. This can be done using
the Elliptic Curve Diffie-Hellman Key Exchange Protocol [Mil85], ECDHE, thus pro-
viding confidentiality. Integrity could be provided by accompanying every message
with a cryptographic hash [BCK96] of the message, if needed.
The Mobile Device also needs to be able to trust the Provider System as well
as it needs to be able to trust the Mobile Device. Thus mutual authentication is
needed on this secure line of communication. The ECDHE protocol only establishes
confidentiality between two parties. It does not provide any guarantees to either of
the parties about who either one is. In principle, an attacker could download and
install the application on her own Mobile Device or possibly on a computer with a
simulated operating system.

3.10
Cryptographic System For The Tickets
The ticket part of the design defines a need for mutual authentication. The Provider
System needs assurance that the Tickets, it receives, are sent by a trusted Mobile
Device. And the Mobile Devices need assurance that Ticket requests come from the
Provider System. This secures the design from DNS spoofing attacks [Bel95] [Fel+97].
The application on the Mobile Device will have an URL to contact the web services of
the Provider System through. Suppose an attacker spoofs the DNS records, such that
the URL is translated to the attackers IP address and the application thus receives
data from the attacker. The application would still not accept messages from the
attacker, because she would not be able to sign her messages with the correct private
key.
When confidentiality is not enough and signatures are required, asymmetric cryp-
tography systems provide the necessary tools. Asymmetric cryptography enables two
parties to communicate with each other by sending messages that the other party is
the only one that can decrypt. This is guaranteed by keeping the private keys secret.
This use of an asymmetric cryptography system is based on using the public key to
encrypt and the private key to decrypt. The other way around, one party can assure

the other that a message could only have been sent by party one, by signing the
message with its private key. This enables anyone to verify the message using the
public key for decryption.
Elliptic Curve Cryptography[Kob87], ECC, and RSA[RSA78] are both mature
asymmetric cryptography systems, which are used to solve this problem. There is,
however, great performance differences between the two5. The security of RSA relies
on the difficulty of factoring large integers[ElG85], whilst the security of ECC relies
on the hardness of the elliptic curve discrete logarithm problem, ECDLP[Kob87, 4
Cryptosystems]. As stated in [Gur+04, 2 Public-Key Algorithms ECC and RSA],
there exists sub-exponential algorithms that can solve the integer factorization prob-
lem, while only exponential algorithms are known for the ECDLP. “This allows ECC
to achieve the same level of security with smaller key sizes and higher computational
efficiency; ECC-160 provides comparable security to RSA-1024 and ECC-224 provides
comparable security to RSA-2048.” It was necessary for the authors of the article to
work on optimization due to their focus on microcontrollers. That is not the case
for the design of this thesis. Even the cheapest mobile devices running operating
systems like iOS, Android or Windows 10 Mobile have vastly more computing power
than e.g. an ATmega128 microcontroller, which was one of the chips used by the
authors in the article. These powerfull devices support multithreading and can run
simultaneous processes. Thus it is not critical to adopt all kinds of optimizations.
However, there is still a visible difference in the necessary time and computational
power, when comparing RSA to ECC as shown in the article, see [Gur+04, Table 2].
Thus it is recommended to choose ECC over RSA if the choice stands between well
documented, researched and mature asymmetric algorithms.
There is a newer alternative, Attribute Based Cryptosystems, which provides dif-
ferent possibilities. These can be divided into two categories, namely Attribute Based
Encryption[Goy+06] and Attribute Based Group Signature[Kha07]. They each pro-
vide useful functionality to the Provider System and the Mobile Device, respectively.

With either RSA or ECC, this would only be possible by either creating a key pair

the same private key amongst multiple devices associated with the same Consumer,
only one key pair would be required per Consumer. There is another system that
solves this problem, namely Attribute Based Encryption.
3.10.1.1
Attribute Based Encryption
This technology allows an entity to encrypt confidential data once with its public key
under a set of attributes and distribute the same ciphertext to multiple recipients.
Each recipient has its own private key, which enables it to decrypt the ciphertext,
as long as its private key matches the attributes that the data was encrypted under.
This way data can be made accessible on a fine grained level instead of relying on an
intermediary or encrypting the data once for every recipient.
The attributes are defined as a tree structure [Goy+06, 4 Construction for Access
Trees], in which each node functions as an AND or an OR gate and each leaf is an
attribute.
==PAGE:70==
58
3 Design
Figure 3.5: Access Structure Tree.
In Figure 3.5 an access structure tree is depicted. If a ciphertext has been en-
crypted under the depicted access structure tree with the attributes [Department X
OR CEO], then one has to be an Employee in Department X, or be CEO to be able
to decrypt the ciphertext.
The use of attribute based encryption is based on the following four steps from
[OSW07, 3.1 Definitions] and relies on bilinear maps[Goy+06, 3.3 The Decisional
Bilinear Diffie-Hellman (BDH) Assumption]. This is just a simple overview, though.
• Setup. This is a randomized algorithm that takes no input other than the
implicit security parameter. It outputs the public parameters PK and a master
key MK.
• Encryption. This is a randomized algorithm that takes as input a message m,
a set of attributes γ, and the public parameters PK. It outputs the ciphertext
E.
• Key Generation. This is a randomized algorithm that takes as input an access
structure A, the master key MK and the public parameters PK. It outputs a
decryption key D.
• Decryption.
This algorithm takes as input the ciphertext E that was en-
crypted under the set γ of attributes, the decryption key D for access control
structure A and the public parameters PK. It outputs the message M if γ ∈A.

A short explanations of the parameters from each step follows, as described in
[OSW07]. Consult [OSW07] or [Goy+06] for further details.
The implicit security parameter determines the size of the bilinear groups. The
public parameters, PK, define two publicly computable functions. The master key,
MK, is decided and held by a key authority, who is the only entity who can issue
private keys for decryption. The message, m, is the data to be encrypted. The set
of attributes, γ, would e.g. be Department Y and Manager as one possibility is in
Figure 3.5 depending on the tree used.
The ciphertext, E, is the product of the
encryption process. The access structure, A, would be the whole tree as e.g. the
one depicted in Figure 3.5. The decryption key, D, is a process of a key generation
necessary for the Decryption step. The outputted message, M, is the decrypted data
and should match m.
The authority in charge of the master key, MK, will have to be the Provider
System in this design. That makes the Provider System the central authority in the
system, which complies with its current role in the existing primary scenario. Since
the Provider System is also the entity generating all the private keys, there is an
issue of key distribution. The secure and tedious way to solve the key distribution
problem would be to get all the devices in a room with only local network connection,
no internet, and transmit each key to each device on that closed network. The fast
way requires mutual authentication as well as the Provider System and the Mobile
Devices to trust the confidentiality and integrity of the secure connection between
the two parties. In that case, the private keys for each device can be sent over the
internet to the Mobile Devices.
3.10.2
Mobile Device Application Signing Tickets
The reason for using a cryptographic system to sign each Ticket is to enable the
Mobile Device to prove its authenticity to the Provider System. Thus assuring that
each Ticket is valid and the Provider System can verify the source of each Ticket.
By requiring each Ticket to be signed, an attacker will not be able to inject false
Tickets into the Provider System unless she has gotten access to a private key and
a device that is trusted by the Provider System. Furthermore, assuming that each
Mobile Device has its own unique private key, signing the Tickets will enable the
Provider System to trace, which device issued any given Ticket. This could prove
useful in the case of a Mobile Device being stolen or an employee being dismissed. In
either of those cases, and possibly others, it will be necessary for the Provider System
to be able to discontinue accepting Tickets signed with specific signatures.
Since
the Provider System is a central authority in the design, it controls the generation
of private keys as well as which keys are trusted and which are not. Therefore the
necessity to revoke private keys is not as dire as it may be in other systems, like the

3.10.2.1
Attribute Based Group Signature
As with ABE, Section 3.10.1.1 Attribute Based Encryption, ABGS is also based on an
access structure tree as e.g. the one shown in Figure 3.5. The technologies are similar
and both rely on bilinear maps [Kha07, 2.1 Bilinear Maps]. However, ABE is meant
for encrypting data for distribution to a specific group of recipients by distributing
the exact same ciphertext to all the recipients, whereas ABGS is designed to enable
any member of a group to sign a piece of data on behalf of the whole group.
A simple overview of the five steps from [Kha07, 4 Construction of an ABGS
Scheme] comprising ABGS is showed in the following.
• Setup. Define the bilinear map. Select two hash functions. Select a generator.
Select a random γ from the first hash function and use this to set the ω value
using γ, and finally choose a universe of attributes.
• KeyGen. This step generates a public key for a specific access structure and a
private key for each user. The generation of public and private keys are divided
into each their own algorithms.
• Sign. To sign a message the user takes the public key, its own private key
and the message M. The rest of the values needed are based on picking random
values from the hash functions and then compute the signature, σ.
• Verify. The verification algorithm takes as input a signature σ, the public key,
a message M, a set of attributes ζ and a revocation table R.
• Revoke. Revoke is about building a revocation table R. The algorithm gets an
index of a user, i, and a set of attributes, ϕ, to be revoked.
For details on the symbols and functions mentioned in the steps above, the reader
is referred to [Kha07].

Mobile device. The proposed solution design is based on using the Users’ Mo-

describes how those requirements can be met by employing mutual authentication,

Limiting the timeframe. By introducing the authorization granting Tickets

to authorize her own data requests. This requires more effort on the attacker’s part
than the existing system would, since it will require either physically engagement
from the attacker or attacks on the Users’ Mobile Devices in addition to attacking
the Consumer Systems.
Limiting the origin. The origin limitation is addressed by a combination of
defining either a specific IP address or an IP-range within each Ticket along with
the Provider System holding a whitelist of Consumer System IP addresses or ranges.
When a data request is received, the IP address of the request is compared with the
Provider System’s whitelist. If it matches an IP address or range on the whitelist,
it is subsequently also compared with the IP origin information within the Ticket.
This effort limits an attacker to send requests from whitelisted IP addresses or ranges
in order for her data request to be served by the Provider System. Alternatively
the attacker could use a botnet, thus trying from different IP addresses without the
need to spoof the IP. Otherwise she would have to spoof an IP address. Spoofing IP
addresses, however, would either require the attacker to find out what IP addresses
or ranges are whitelisted or try by brute force.

pin code. However, changing the input passphrase on the Mobile Device to something
other than the pattern, suggested for use in the other scenarios, would require the
system to hold an additional key per employee with access to authorization in this
scenario. Copying the pattern based passphrase solution to the Computer could be
done by asking the operator to use the mouse to input the passphrase instead of the
keyboard.
The installer does not need to be signed in order to be accepted on the Test Meter,
because the Test Meter will be configured without the requirement of a signature on
the installer. Another approach would be to resemble the exact same setup, in which
the Test Meter also requires the code to be signed. The ideal setup for this purpose
would use another key than the one being used in production meters, which are sold to
customers. In this way, the test environment is a closer analogue to the real systems
and tests on whether or not the Test Meters accepts installers with errors in the
signature can be carried out as well.

CHAPTER 6
Conclusion
The thesis has proposed a solution design to the problem of securing authentication in
automated jobs. The proposed solution design solves the problem, while succeeding in
minimizing the necessary human interaction, thus upholding the automation of such
jobs. The design solves this problem under the restrictions that it cannot impose or
require changes on the Consumers IT systems.
The proposed solution design uses mobile devices as a means to realise a trans-
parent system. The system uses the mobile devices to generate authorization tickets.
Any data requests made from the Consumers systems thus need to be authorized
by a ticket generated on a mobile device. The design proposes an Ahead Of Time
approach as well as an On Demand approach.
The Ahead Of Time approach enables the Provider’s system to serve data requests
from Consumers systems. In this way, automated data requests will not have to wait
idle for a user to generate a ticket on their mobile device, since this is done in advance.
To ensure that tickets are always available for scheduled automated data requests, it
is proposed that the Provider requests multiple tickets from its users before it runs
out of tickets.
The On Demand approach enables the Provider’s system to request tickets from
its users, in case it receives a data request, which does not originate from a sched-
uled automated job. Thus Consumers can request data manually as well. Should an
attacker attempt to request data from inside a Consumer’s systems, such a request
will need to be authorized by an On Demand ticket. Thus the design is not a silver
bullet, but relies on the users as do all IT security solutions.
The proposed solution design enables the Provider’s system to enforce limitations
on timeframes, origins and the type of the data requests it receives without requiring
the Consumers to change their systems. As mentioned this is achieved by the use
of authorization tickets. The ticket system allows the Provider to distribute ticket
requests to mobile devices, which are only readable to the devices associated with
the Consumer, for whose data requests the tickets are for. This is achieved by use
of Attribute Based Encryption. The ticket system also allows its users to authorize
data requests on behalf of each other while limiting the number of keys, which the
Provider has to store. This is achieved by use of Attribute Based Group Signature.
The design enables the Provider to provide a complete system to its Consumers
with minimal need for user training and minimal learning curve. The installation is
==PAGE:88==
76
6 Conclusion
the same as for all other mobile device applications, by downloading the application
from the device’s operating system’s app store.
The proposed solution design is not a silver bullet. It does not prohibit an at-
tacker inside a Consumer’s system from retrieving data from the Provider’s system.
However, it limits the attacker’s ability to do so on her own terms, since she cannot
retrieve data at any given time, from a host on any given IP address or retrieve any
type or amount of data she might want. Even in the case that she has access to the
Consumer’s username and password as well as one or more of the Consumer’s hosts.
The thesis highlights how the proposed solution design can be adapted to solve
issues of other scenarios than the primary as well. Thus showcasing that the design
is versatile.
For future work it would be interesting to implement the proposed solution de-
sign with Attribute Based Encryption and Attribute Based Group Signature if it is
possible.
