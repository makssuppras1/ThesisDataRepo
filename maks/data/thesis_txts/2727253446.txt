Abstract
Interactive didactic software has revolutionized the field of education through
the development of robust tools that are designed with user intuition in mind.
Simultaneously, resolution calculus, as it plays a major role in automated theo-
rem proving, is also an important part of university courses on logic in computer
science.
This thesis focuses on the design and implementation of a didactic tool that
facilitates the proof of the validity of formulas using applications of the resolution
rule and the attainment of formally verifiable proofs for first-order logic.
To ensure the correctness of the proofs, Isabelle, a proof assistant, is employed
for formal verification. By integrating a generator for Isabelle, the generation
of machine-checked proofs is enabled, providing the highest level of confidence
in our proofs by eliminating the possibility of errors.
Resolution, a rule of inference in logic, enables the deduction of new clauses
from existing ones. The tool aims to provide a user-friendly experience with
an interface that guides its users in applying the resolution rule effectively and
correctly. By entering any formula in first-order logic, users can interactively
navigate through the proofs steps, visualizing the proof process and understand-
ing the relationships between the derived clauses.

Chapter 1
Introduction
We present ResolutionOnline, a web app that facilitates the operation of the
resolution rule, an important inference rule in the field of AI, specifically in the
area of automated theorem proving, on any formula given in first-order logic.
The web app provides a formally verifiable proof for any given application of
the rule.
Background
We have become dependent on the well-functioning of computer systems, which
are omnipresent in daily life. It is necessary to ensure the correctness of many
critical systems, such that it is certain that systems behave as expected ev-
ery time, in everything from digital alarm clocks to the navigation systems in
aeroplanes, or to ensure the security of assets, whether they be financial or
intellectual.
One way we can raise our confidence in the systems we use everyday is by
testing the hardware or the software rigorously, however the resulting level of
confidence pales in comparison to the confidence provided by machine-checked
proofs, enabled by the inception of automated theorem provers, many of which
are using an extension of resolution as their main proving function.
==PAGE:9==
2
As such, computer science courses at universities dedicate significant time to
the study of proof systems. Notably we have resolution calculus, which is a
proving technique used to prove the validity or satisfiability of propositional
or first order logic formulas. It is comparable to other proving techniques like
sequent calculus and natural deduction.
There is a significant potential for didactic tools to make a difference in the study
of computer science. A plethora of tools online demonstrate the tableau method,
but there is a distinct lack tools that demonstrate resolution. Therefore, this
thesis is focusing on resolution, which is a cornerstone method in the proving
methods used in state-of-the-art automated theorem provers such as E, SPASS,
and Vampire.
Existing tools for resolution
We succintly review existing tools. Logictools is an extensive in-browser toolkit.
It uses Prolog as its logic language, and it involves resolution to some extent
in its part dealing with propositional logic. Furthermore it enables different
algorithms for proving the validity of formulas such as Davis Putnam Logemann
Loveland (DPLL) and the building of truth tables and parse trees. Graphically,
the tools are unintuitive and require some thorough examination to understand
how to use.Logic calculator: Server-side Processing, is a tool that enables the
transformation of formulas in propositional logic and the generation truth tables.
The Tree Proof Generator, is a visually intuitive tool that enables the automatic
generation of proof trees using the tableau method for formulas in a logic that
combines first-order logic with modal logic. A Logic Calculator is a tool that
enables the generation of truth tables and offers automated proving of formulas
for first-order logic together with some modal logic operators. This tool has a
steep learning curve, it requires much effort from the users to get into, as only
the grammar of the language used is given but no significant examples of it being
used. The ’Advances in Modal Logic’[Var20] conference initiative keeps a long
list of tools for logic. Under the ’visualisation’ tab lies ’Logic Animations’ which
is ’A collection of JavaScript programs used for teaching of basic mathematical
logic’. Unfortunately, this tool is completely inaccessible, and this is a trend in
the search for robust tools that enable resolution.
We also review two tools developed at DTU Compute which touch upon other
calculi for first-order logic.NaDeA [VFS19], is a comprehensive tool for using
a natural deduction system. The user experience is streamlined and does not
require the user to ever use their keyboard. It produces proofs for Isabelle for
formal verification. MiniCalc, previously called the SeCaV unshortener [FJV22],
is a tool that facilitates the formal verification of proofs that use a sequent

calculus, and it does so by compiling proofs from a concise syntax into Isabelle.
None of the tools reviewed dealt with resolution in first-order logic and were
intuitive or adequate for the uninitiated. Of course, the range of this research
is limited, but the overall impression is that the tools are inadequate. The fact
that it was difficult to find any suitable tools further motivates this thesis, as
the need for tools in this area is made concrete.

Theory
In this chapter we cover the theoretical background knowledge that is involved

2.1
First-order logic
First-order logic, also called predicate logic, has extensive applications in com-
puter science as well as in philosophy. In computer science, first-order logic is
the basis of logic programming and of automated theorem proving. It extends
propositional logic, which is part of the foundation of computer hardware, by
providing a technique for expressing quantified variables in formulas. Thanks
to this, we can reason about mathematical theories such as arithmetic.
Propositional logic makes use of the following logical connectives: ∧, ∨, ¬, =⇒,
and ⇐⇒. Concretely, the extensions first-order logic brings are composed of
the introduction of the existential and universal quantifiers, denoted ∃and ∀
respectively, and their bound variables. The atoms can now also have arguments,
meaning that their value can depend on the bound variables and on functions,
which in turn can also depend on arguments.
==PAGE:13==
2.1 First-order logic
6
2.1.1
Clausal form
A formula in clausal form is a conjunction of disjunctions of literals[BA12b]. A
literal is an atom or the negation thereof. We can represent a formula in clausal
form in the following template:
(l11 ∨l12 ∨... ∨l1n) ∧(l21 ∨l22 ∨... ∨l2m) ∧... ∧(lk1 ∨lk2 ∨... ∨lkj)
where lij is the jth literal in the ith disjunction.
All variables in a formula in clausal form are universally quantified, and as such
there is no need to order the quantifications as it has no impact on the value of
atoms, and therefore there are no quantifiers at all. Each individual disjunction
is also called a clause.
Every formula in first-order logic has an equivalent formula in clausal form. To
obtain the clausal form equivalent to a formula, one must apply the following
steps:
1. Eliminate the biconditionals and implications by applying these substitu-
tions
(a) p ⇐⇒q ≡(p =⇒q) ∧(q =⇒p)
(b) p =⇒q ≡¬p ∨q
2. Move negations inwards
(a) ¬∀xp ≡∃x¬p
(b) ¬∃xp ≡∀x¬p
(c) ¬(p ∨q) ≡¬p ∧¬q
(d) ¬(p ∧q) ≡¬p ∨¬q
(e) ¬¬p ≡p
3. Standardise variables apart by substituting duplicate variables.
Every
quantifier must use a different variable.
4. Skolemization: In this step, every existential variable is replaced with
a function which represents it and has all of the enclosing universally
quantified variables as arguments.
For example, ∀x∃y p(x, y) becomes ∀x p(x, Y ′(x)) where Y ′ is a new func-
tion.
5. Remove universal quantifiers, after which all variables are implicitly uni-
versally quantified.
==PAGE:14==
2.1 First-order logic
7
6. Distribute conjunction over disjunction, until the formula is a conjunction
of disjunctions, by applying this substitution: (p∧q)∨r ≡(p∨r)∧(q ∨r)
An important notion is the complementary literals, or clashing literals, which
means a pair of two atomic formulas that have the same label but opposite
values, i.e. one of them is negated. Notice that a pair of complementary literals
are not necessarily the negation of each other, by simply have the same label and
opposite values. This means that if we take two clauses p(x, x) and p(y, y) where
x, y are variables, the clauses have the exact same meaning as both variables
are universally quantified. This is the basis of factoring, which is explained in
the section about resolution.
When handling operations on clauses, we use set notation, meaning that for
two clauses C1 and C2, C1 ∪C2 is the union of the clauses, and C1 ∩C2 is the
intersection. When we remove a literal from a clause, we use the set notation
C\{l}, where the literal l is removed from the clause C.
2.1.2
Unifiers
An unifier is a substitution that can be applied to two literals such that they
become identical. Individual substitutions are denoted as a ←−b, where a is
being substituted with b. An unifier can be found for e.g. two literals p(f(x), y)
and p(z, g(w)). If we apply the subsitutions µ = {z ←−f(x), w ←−x, g(x) ←−
y}, both literals become p(f(x), g(x)). Thus, the three substitutions constitute
an unifier.
If an unifier can be found for two literals, we say that they are
’unifiable’, and that the unifier ’unifies’ them.
A most general unifier (MGU) is an unifier that has the additional property of
being the least specific unifier, i.e. it does not assign a function when the literals
can be kept more general by using variables. For example, the two literals p(x)
and p(f(y)) can be unified with the unifier µ = {x ←−f(g(z)), y ←−g(z)},
yielding the literal p(f(g(z)).
The two literals can also by unified by µ =
{x ←−f(y)}, yielding the literal p(f(y)). That is a more general substitution
as another susbtitution can be applied to obtain p(f(g(z)) from p(f(y)), σ =
{y ←−g(z)} that is. Consequentially, y can be replaced with much more flexible
substitutions.
To denote the application of a substitution to a literal or a clause, we simply
add the symbol of the substitution after the name, i.e. Cµ for a clause C and a
substitution µ.

2.2 Resolution
8
2.2
Resolution
Resolution is a rule of inference in logic. It and the resolution calculus was intro-
duced in the 1960’s by Davis and Putnam together with most general unifiers
[DP60]. Concretely, it is a method that allows the derivation of new clauses
based on clashing clauses. It is the basis of logic programming.
2.2.1
In propositional logic
If we have clauses C1 and C2, and a literal l1 ∈C1 and another literal l2 ∈C2
which is complementary to l1, then the resolvent of C1 and C2 is:
C1 ⊗C2 = (C1\{l1}) ∪(C2\{l2})
(2.1)
Essentially, if two clauses contain a pair of clashing literals, we can obtain a new
clause derived by merging the two clauses and removing the clashing literals.
The clashing literals are also called the ’pivot’. It is possible to apply the rule
even if more than one pair of literals clash, but only one pair can be removed.
This can be rewritten as a rule of inference:
a1 ∨a2 ∨... ∨c
b1 ∨b2 ∨... ∨¬c
a1 ∨a2 ∨... ∨b1 ∨b2 ∨...
(2.2)
We can further generalize the rule and mark the positive literal removed in the
process (the pivot), which is necessary for an exact expression in case there are
more than one pair of clashing literals:
C1
C2
C1/{l} ∪C2/{¬l} |l|
(2.3)
where l ∈C1 and ¬l ∈C2.
2.2.2
In first-order logic
Resolution in first order logic extends the resolution in propositional logic by
applying unifiers. With an unifier µ that unifies the literals l1 ∈C1 and l2 ∈C2:

C1 ⊗C2 = (C1µ\{l1µ}) ∪(C2µ\{l2µ})
(2.4)
It is a requirement for resolution in first-order logic that the two clauses used do
not have any variables in common. This is necessary to ensure the soundness
of derived clauses, but there is a method to circumvent this, which consists of
renaming the variables in one clause before applying resolution. This is called
’standardizing apart’, and it does not affect the satisfiability of a clause.
If two literals are unifiable, it possible to use resolution. The inference rule is
similar to the one from resolution in propositional logic, the only difference being
that the susbtitutions in the unifier are applied, giving us the rule of inference:
a1 ∨a2 ∨... ∨c
b1 ∨b2 ∨... ∨¬c
(a1 ∨a2 ∨... ∨b1 ∨b2 ∨...)µ
(2.5)
where µ is a most general unifier of the two clauses. Rewritten schematically:
C1
C2
(C1/{l} ∪C2/{¬l})µ |l|
(2.6)
There is also another type of resolution, where instead removing pairs of clashing
literals, sets of clashing literals are removed. Still, the literals need to clash on
and be unifiable with every other literal in the set, and all literals in one set
need to be positive and all literals in the other set need to be negative. This
gives us another rule of inference:
C1
C2
((C1/L1) ∪(C2/L2))µ |L|
(2.7)
where L = L1 ∪L2 is the pivotal set of literals, and all literals L1 ∈C1 clash on
all literals in L2 ∈C2.
2.2.3
Proving satisfiability and validity of formulas
The validity of any valid formula in first-order logic can be proven using the
resolution calculus, which relies on the soundness and completeness of the res-
olution rule. It is a proof by refutation. To do this, one must first negate the
==PAGE:17==
2.2 Resolution
10
formula, and then convert it to clausal form. At this point we are ready to
employ the resolution rule in our proof, as we apply the rule to these clauses in
search for an empty resolvent clause denoted by {} or by □. It is possible to
apply the rule to the clauses originating from the formula as well as the resol-
vents. If it is at all possible to find the empty clause, then the original formula
is proven valid. It seems peculiar, but the empty clause is unsatisfiable, and
that fact is exploited in proofs by refutation. To explain this, we can choose
to see the clauses simply as disjunctions, and we know that if just one literal
evaluates to true in a disjunction then the entire disjunction evaluates to true.
The empty clause has no literals at all and therefore cannot evaluate to true.
Together with a suitable search algorithm, it is possible to automatically prove
a formulas validity using the proof by refutation.
2.2.4
Factoring
Factoring is a rule which allows for the unification of subsets of clauses, which
e.g. allows the collapsing of identical clauses. In this context, it is crucial to keep
in mind that all variables are implicitly universally quantified when a formula
is in clausal form. In clausal form, variables can freely be renamed within the
scope of individual clauses. Two literals can well be identical in meaning and
satisfiability and have different variables. For example, the literals p(x, x) and
p(y, y) have the exact same meaning. If the two were in a clause p(x, x)∨p(y, y),
it would not change the satisfiability of the clause if either of the literals were
removed. It is, however, different in case those variables are used somewhere
else in the clause also. For example the clause p(x) ∨p(y) ∨q(x), in which, by
factoring, p(y) can be removed but p(x) cannot, as the satisfiability changes.
When ’factoring’ a literal away, it is important to keep track of the variables to
make sure the meaning of a formula is not changed.
If we consider resolution where only pairs of clashing literals are removed, fac-
toring is necessary in resolution to eliminate the redundancy of identical literals
in clauses. Some proofs cannot be completed without factoring, e.g. the clauses
p(x)∨p(y), ¬p(a)∨¬p(b), derived from a valid formula. We can apply resolution
to the clauses in four different ways, targeting different pairs of complementary
literals, but it is not possible to obtain the empty clause. However, if apply fac-
toring to the first clause, we obtain p(x) (or p(y), or any other variable name),
giving us a possible proof in figure 2.1.
If we consider resolution where sets of literals are removed, we do not need
factoring, as the redundancy of identical clauses is dealt with when removing
the sets of clashing literals.

2.3
Proof assistants
Mathematical proofs can be carried out on a blackboard, but there is a signifi-
cant advantage to carrying them out using a program that can verify each step
in the proof. By using a such a program, also called a proof assistant, the risk
of errors going unnoticed is eliminated. Proof assistants have various uses:
• Proving theorems that are infeasible for people to prove by hand, by pro-
ducing so-called non-surveyable proofs, such as the four colour theorem[KA78].
• Verification of software, by proving that some properties about a program
holds, such as termination, type safety, functional correctness according
to a specification e.g. program contracts, by using program verification.
• Verification of cryptographic protocols, which are used to ensure the con-
fidentiality, integrity, and availability of data online, by ensuring that the
protocols enforce a given specification.
• Verification of proof systems, by proving that the axioms are sounds and
complete, as well as verification of individual proofs.
Isabelle is a proof assistant which is available in a interactive form with jEdit.
Isabelle itself uses a higher-order version of resolution as its main proof method
together with higher-order unification[Pau22], but note that this is completely
unrelated to the formalizations and proofs in Isabelle presented in this thesis.
To use Isabelle, one must simply write proofs in the Isar language, which is
Isabelle’s programming language.

2.4.2
Soundness and completeness
For a prover to be sound, means that every correct proof is accepted by the
prover, i.e. it does not produce false positives. For a prover to be complete,
means that it accepts everything that is correct, i.e. it does not produce false
negatives. The prover in the formalization is proven to be sound and complete,
and thanks to this it can be used as an ’oracle’ as we have proven that it always
evaluates a proof correctly. In the paper, the soundness is proven by combining
a susbtitution rule, a simplified resolution rule and a superset rule. All of these
rules are proven sound individually first, and then combined again as one rule
to form the resolution rule, thus soundness has been proven.
Completeness
however is a longer proof, which is usual as completeness is more difficult to
prove. Completeness is proven using a combination of Herbrand’s theorem, the
lifting lemma and some reasoning about semantic trees.

Literals
The first task in a proof is to define the literals. Every literal used in a proof
needs to have a definition for every instantiation that is used, e.g. ¬p(x), p(x),
p(f(x)) and p(f(y)) need to be defined separately, using the definition key-
word in Isabelle. Each of these definitions need to have a name that is unique,
and it is important to be careful with the name as having weak delimiters might
create an overlap. Take for example p(f(x)) and p(fx), if we were to simply
add the label of the literal p, and the function name f or fx and the variable
name x, we would end up with pfx in both cases. Therefore it is important to
find a robust delimiter, and it would be ideal to have a delimiter that implies
the relation between the names, such as parentheses or brackets. Unfortunately,
we are limited to underscores and apostrophes, and it is difficult to find better
delimiters that make more sense because the Isar language is quite restrictive
in this aspect. We therefore add underscores, so it becomes p_f_x and p_fx.
We’re still not clear of all confusion, as the names do not indicate whether
they’re representing functions or variables. The intuition of a knowledgeable
logician states that x must be a variable, but we cannot distinguish function
from variables just from the name alone, which can lead to ambiguity. To make
sure we can tell them apart consistently, we decide that functions must always
start with an uppercase letter. Lastly, positive and negative literals also need
to be distinguished, and we do so by adding the prefix n’ to the front of the
negative literals. The prefix contains an apostrophe to ensure that an overlap
does not occur with any other literals.

What cannot be proven with this method
This method of constructing proofs does not comprehensively make use of the
prover, as the formalization of the prover states that the pivot, i.e. what is
removed in the application of resolution, is a set of literals. This proof construc-
tion handles the pivot only as single literals, meaning that factoring must be
handled a different way. Until a more complete solution is found, the temporary
solution is to simply add the clauses derived by factoring into the beginning of
the derivation, effectively circumventing the formal verification entirely. This
limitation is partly due to how the MGU proofs are constructed, as a proof here
is specific to the two literals provided, and because one substitution that is most
general for one pair of literals is not necessarily the most general for another.
Take for example, the clauses p(x, y) ∨p(A, y) and ¬p(x, B), where x and y are
variables. We need to unify all three to make full use of the formalization, but
we find that the substitution {x ←−A, y ←−B} is an MGU for p(A, y) and
¬p(x, B), but not for p(x, y) and p(A, y), and thus it cannot be proven using
this method. Of course it is possible to use the prover to its full extent, and we
have clear examples of how the MGUs can be proven, by changing the way the
MGU proofs are made such that the MGU encompasses all pivotal literals. An
example proof using this is in the appendix, but more investigation is needed to
make use of it in the automatic generation of proofs, as the automation is the
problematic part.

3.3
User experience
In this section, we will explore the interface design opportunities in this tool,

3.3.1
Design strategy
Through some ideation, by thinking outside-the-box and prioritising quantity
over quality, many approaches for how the tool could work surfaced. Focus was
put on automating as many of the monotonous parts of a proof by refutation,
such as the tedious re-writing of already obtained information e.g. copying the
literals that are unaffected by a step in the proof, or providing information
that can be derived from other information e.g. what the pivot is in a step in
the proof when the clauses involved in the step only have one pair of clashing
literals. With the streamlining of the process of proving, not only would the
productivity of the end user increase, the engagement of the user would also
be encouraged as the uninteresting parts of process are automated away. In
turn, a superficial gamification (which means to change some process such that
it resembles a game) of the proof system would materialize by itself.
This
happens because the design thoroughly puts focus only on the objective of the
proof, to find the empty clause, which in itself arguably is a game. According to
some definitions e.g. the one by Katie Salen and Eric Zimmerman, "A game is
a system in which players engage in an artificial conflict, defined by rules, that
results in a quantifiable outcome.".
3.3.2
Interface designs
In this section we describe different approaches to the concrete UI design. The
process of designing user interfaces culminated in three polished potential de-
signs. These are explained in the next sections.

3.4.4
Automatic distinction between homonym variables
This is the optimisation of the ’standardising apart’ step, used to allow resolution
to be applied to clauses that have common variables, which is otherwise not
sound. To cover this, we take an example unification between the clauses C1
: p(A, x) and C2 : p(x, B). Should these two clauses be unifiable? Using the
strict definition of unification, they should obviously not, as the unifier requires
that we replace the variable x with A and B simultaneously. However, it must
be noted that these two variables are not the same, as they are in two different
clauses, they are both universally bound and they can freely be renamed to
other names. The x in the first clause could be named to y, and then an unifier
can be found: µ = {y ←B, x ←A}. It is possible to design and implement
an algorithm that automatically finds a suitable name to rename variables to
allow such unifications and thus automate the renaming step. It suffices that
the algorithm finds a fresh variable name i.e. one that doesn’t occur anywhere
else in the collection of clauses.
The main presumption of the work in this thesis is that the tools designed must
have a didactic approach in mind. It is not at all intuitive that some automatic
process changes the variables in the clauses and therefore if was decided that
the tool should not support this.

4.2.1
Unification algorithm
This section explains the unification algorithm implemented in ResolutionOn-
line. As previously mentioned, an unification algorithm is implemented in the
Isabelle proof generator, but its sole purpose was finding the right MGU for an
application of the resolution rule. Another, more thorough unification algorithm
is also implemented in the application for the purpose of evaluating resolution
steps, and giving feedback to the front end. As such, there cannot be any room
for unsoundness, as this more thorough unification algorithm in the end decides
whether your proof gets accepted by ResolutionOnline. It is implemented in
Proof.cs.
The unification algorithm aims to find a more concise MGU for the clauses, i.e.
it will try to make the least amount of unnecessary applications of substitutions.
To take a concrete example, the MGUs µ = {x ←−F, y ←−x} and γ = {x ←−
F, y ←−F} are functionally equivalent. Substitutions are applied all at once,
so the order of the substitutions does not matter, and as such substitutions are
not ’applied multiple times’. Therefore, the MGUs are equivalent but γ is an
optimisation of µ. This was done not because of concern with performance, but
because a clearer MGU can make a difference in the end users understanding of
the substitution. The algorithm works as the following:
First we trivially check if the literals that we are trying to unify have the same
labels and that one is negated. Then we traverse the literals and collect the
differences between two in a list of pairs of arguments, which is essentially a list
of yet undirected substitutions. Essentially, these are the ’disagreements’ from
Robinsons unification algorithm [BA12a].
Then we check if these substitutions are consistent. We do so by keeping track
of two dictionaries (a dictionary is key-value map), functions (which stores the
substitutions from a variable to a function) and variables (which stores the
substitutions from a variable to another variable), and we iterate through the
substitutions. We denote the entries in these for a key a and a value b: {a : b}.
Note that functions is directional as the substitutions can only happen in
one specific direction, while the substitutions in variables do not suggest any
direction. For one substitution, depending on the type of the arguments, we
verify various things:
==PAGE:49==
4.2 Data structures and algorithms
42
• In case the substitution {v : f} appertains to one variable v and one
function f:
1. Check that v does not occur somewhere in f, otherwise stop and
return an error. We have now checked for short circular substitutions,
as the variable might directly occur in the function that is meant to
replace it.
2. Check that the functions does not already have a substitution with
v as its key, and if it has an entry, make sure the value is identical to
f, otherwise return an error.
3. Check the same as the previous point but with the variables. In
case there is an entry {v, vi} or {vi, v} in the variables, we can
remove that substitution from and redirect vi to f by adding {vi, f}
to funtions.
4. If nothing returned an error, add {v, f} to funtions.
• In case the substitution {v1 : v2} appertains to two variables v1 and v2.
1. Check that they do not have the same label, otherwise return an error
as different clauses are not supposed to contain the same variables.
2. If functions contains entries for both v1 and v2 as keys, check that
they have identical values, if not return an error.
3. If only one of v1 and v2 have an entry in functions, redirect both
variables to the value in functions, i.e. if there is a entry {v1 : f},
add {v2 : f}.
4. If neither of them have entries in functions, add an entry {v1 : v2}
in variables.
Furthermore we need to ensure that there are no circular substitutions, as those
are not sound. We do so by constructing a dependency graph for every variable
and traverse the graph using depth first search (DFS). In more detail:
• Start by creating a dependency graph, which consists of a dictionary with
the type Dictionary<string, Set<Tuple<bool,string»> which maps a
variable name v to a set S of variable names annotated with a boolean
which represents whether the variable is enclosed in a function as its argu-
ment. This way, a variable v represents a node in the dependency graph,
and the dependencies of v are represented by the set S.
• For every node in the dependency graph, we traverse the dictionary while
keeping track of the nodes we have already visited. If our current node
==PAGE:50==
4.2 Data structures and algorithms
43
doesn’t have an entry in our dependency graph, we immediately know it
cannot be part of a cyclic graph.
• If we at some point encounter a node that has already been visited, and,
importantly, that the node represents a variable that is enclosed in a func-
tion, we have a cyclic graph with a circular substitution.
Finally, after an MGU has passed all of these requirements, we can apply the
substitutions and find the resolvent.
The substitutions are all applied until
it stops making a difference for the clause. As we have already proven that
there are no circular substitutions, there is no need to implement a recursion
limit here, although that could also have been a way of tackling the circular
substitution problem, i.e. by simply trying to apply the substitutions and then
assuming that the MGU is in some aspect circular after e.g. 1000 recursive
substitutions.
The internal prover has one main function, which performs the resolution as
such. It is called Resolve and returns a
Either<Resolve, MultipleClashes,ResolveError> type. When a user tries
to resolve two clauses, the internal prover will first try to unify every combination
of literals possible and check how many combinations successfully unify.
If
exactly one combination unify, we can move on to make the step in the proof with
the resolved clause by applying the substitutions in the MGU, safely assuming
that the combination of literals is the intended pivot of the resolution step.
Based on the insight that the internal prover gains, it return one of three types:
1. a successful resolve step, which is then added to the proof. 2. a flag to
indicate that there is more than one valid pivot for the proof, resulting in a
prompt appearing to the user. 3. an error, describing the fault in case no fitting
pivot or MGU was found.
4.2.2
Clausal form converter
The development of the clausal form converter, crucial in the context of proofs
by refutation, is not a trivial task. It is implemented in the ’Resolution’ project
in ClauseConverter.cs.
The tool uses multiple different internal representations of formulas for different
use cases. On one hand, there is the ’GeneralForm’ used for the purpose of
parsing any formula, and it includes all the operators, and on the other hand
is the ’ClausalForm’, which can by design only be used to represent clausal
formulas. Because of the limitations of the ’ClausalForm’, the basic clausal
form properties are enforced by the type.
==PAGE:51==
4.2 Data structures and algorithms
44
The clausal form converter can be broken down into a few steps which closely
resemble the steps presented in the Theory chapter, but it is preceeded by the
negation of the formula and succeeded by the instantiation of the formula in
the dedicated ’ClausalForm’ representation. Two transformations are worth
commenting on here, which are step 3 and 4 from the clausal form conversion
specified in the Design chapter.
Step 3, where we standardise variables apart, is done using the method
UniquefyVariables, which first collects only the existentially quantified vari-
able objects. It is not necessary to make any changes to the universally quanti-
fied variables given that all the transformations that affect the structure of the
formula are already applied. Then, we generate a dictionary containing unique
names for each variable object, by adding apostrophes at the end of their names,
and then traverse the formula again and identify the variables by their objects
and rename accordingly.
Step 4, where we turn all existentially quantified variables into Skolem functions,
is done by traversing the formula while keeping track of the universally quanti-
fied and existentially quantified variable in two lists, encapsulatingVariables
and instantiatedVariables respectively. When we finally meet a existentially
quantified variable in a predicate, we know what universally quantified variables
it must depend on using the encapsulatingVariables list. We determine the
name of the Skolem function simply by capitalizing the first letter of the exis-
tentially quantified variables name, and add an apostrophe, which we know is
unique in this context as apostrophes are not allowed as input in the tool.

click ’Rename’ on a clause that contains no variables, and you cannot click
’Delete’ if the clause is used somewhere later in the proof. An unintended side
effect of this implementation is that users can optimise their proofs by deleting
only the unused clauses after the proof is finished.
==PAGE:52==
Chapter 5
Analysis
The purpose of the analysis is about the correctness of ResolutionOnlines inter-
nal prover, and the user experience of ResolutionOnline. These two focal points
can be broken up into the soundness and completeness of the tools internal
prover, and the process of making proofs using the tool. Despite the fact that
the tool integrates multiple algorithms for various purposes, the performance of
these is not a focal point in this analysis.
Unfortunately, the extent of this analysis chapter is much shallower than was
originally planned, as the analysis can only lag behind the implementation,
which is not concluded.
Notwithstanding, we explore two analysis methods
that are useful in this project, however without a concluding outcome.
5.1
Testing provers
Program testing can be very effective to show the presence of bugs, but it is
hopelessly inadequate for showing their absence[Dij72]. Therefore, even with
some extensive automated unit testing, taking into account all kinds of different
’outlier’ scenarios, it cannot be ensured that a prover will work correctly in every
case.
==PAGE:53==
5.1 Testing provers
46
The goal of testing the provers is to bring the soundness and completeness
of the prover implemented in ResolutionOnline closer to the formally verified
prover. All-knowing that the same level of confidence as the formally verified
prover can never be achieved, it is still advantageous to minimize the risk of
wrong proofs being accepted (false positives) or that correct proofs are declined
(false negatives) by the tool. This could in turn mislead the users about the
functioning of resolution calculus considering that students, whom are usually
novices, are part of the targeted end user demographic.
Given the scope of proofs that would realistically be carried out using this tool,
meaning mostly short proofs of less than 20 applications, it can be possible
to raise the confidence in the program to such a level, that an incorrect ver-
dict from the internal prover would practically never be encountered. To raise
the confidence in the prover, one can perform some extensive testing, like unit
testing, or supply some convincing correctness arguments, like what is used in
algorithms. Even then, the theory and the implementation are ’proven’ correct
separately, which is not enough as the implementation has many more factors
than can play into its behavior. The method that brings the highest confidence
are machine-checked proofs, which is the use case of the verified prover in Is-
abelle in this context. We can use Isabelle from the command line to check
proofs, and thanks to this we can tie it in to a testing framework with the
tool, which can automatically detect discrepancies in the verdicts of the verified
prover and the internal prover. This method of testing provers is presented in a
paper titled ’Differential Testing of Pushdown Reachability with a Formally Ver-
ified Oracle’[SSST22], where verified algorithms are used to test an non verified
implementation of pushdown automata.
What we can do for this project, is using property based testing, which is a
testing technique that involves checking properties of a system. We can auto-
matically generate many proofs using FsCheck according to specification that
limits the scope of the proofs. This involves limiting the maximum size of the
proof to 30 applications and limiting the literals used, such that time is not
wasted generating and discarding proofs that cannot be used. In order to cor-
rectly set up this analysis, a parser must be made to parse the command line
output from Isabelle, and the individual steps in the proofs should be annotated
with their position such that steps in internal representation of proofs can be
identified with the steps in the Isabelle proof.
==PAGE:54==
5.2 Process mining analysis
47
5.2
Process mining analysis
Process mining is a set of techniques used in the study of processes. The use
case for process mining is to statistically analyse the execution of a process
that is represented as a workflow. The goal is to turn the insights gained from
conducting such an analysis into actions. Actions, that will feasibly optimise
the process itself.
To be able to conduct such analyses, one needs to start with a event log, which
is a list of traces that describe the events in a process. Web tracking technology
is used on websites usually for lucrative purposes, such as the farming of user
information to sell for morally equivocal, targeted advertisement practices, but
it can also be used simply to motivate and influence the further development
of a tool. Through web tracking, we can record the sessions on the tool, and
then though session replay, we can construct event logs that detail the use the
of the tool. Finally through process mining, we can make use of the session
information collected and facilitate the acquisition of meaningful insight into
the general user experience, also of the passive reviewers who do not actively
report their feedback.

Chapter 6
Conclusion
In this chapter we recapitulate the work done in this thesis.
6.1
Evaluation
This thesis set out to make a robust tool that exposes the functionality behind
resolution calculus and that facilitates the construction of proofs. These goals
have not been achieved in an exhaustive manner, but a solid foundation has been
set as the work culminates in a minimum viable product that is extensible. The
aspect where the work in this thesis stands out the most is how the formally
verified prover has been made more approachable to a wider mass of people.
This was not an intended goal at the start of the thesis, but a pleasant surprise.
6.2
Challenges
The work on this thesis was structured in such a way that it prioritised formal
verification first, then the non verified implementation after, e.g. the bidirec-
tional unification had first to be consistently verifiable, using a standard proof
==PAGE:56==
6.3 Future work
49
that is able to be generated to fit any potential proof, before implementing the
functionality in ResolutionOnline.
This means that the overall development
has been heavily dependent on one specific part, which has been a unfortunate
decision in retrospect.
Given the time limitations of and the ambition presented in this thesis, much
of the work that was originally planned for in the scope of this thesis was not
completed in due time. Retrospectively, the project plan was unrealistic and
event then it could only be considered the best case scenario back when it was
first made.
6.3
Future work
The possibilities for extending the tool are rich. The first and most obvious ex-
tension is the full use of the verified prover, using the correct formalization where
the pivot is a set of literals rather than single literals. This is a major change
that affects all parts of this project, but a significant step towards improving
the accesibility of the full verified prover for a wider audience, in particular less
experienced logic practitioners.
Another extension possible is to take the educational aspect to a new level, by
picking up the idea presented in the ’Selection of design’ section. As multiple
different designs for user interfaces were considered, one could examine the idea
of implementing all three and either letting users decide which one they prefer
to use, or even using it as a kind of interactive learning experience, where the
expectations towards the user increases gradually.
A few simpler yet useful extensions come to mind, as the export and import of
proofs through the serialization using JSON for example. One simple extension
that could have a immense impact on the further development of the tool is a
’report’ button, that will allow users to report bugs in the internal prover or in
the generated Isabelle proof. The bug reports would be sent to the main server
for us to inspect when convenient.
Another meaningful extension is the enrichment of the generated Isabelle proofs
using comments. These comments would explain the step the proof is supposed
to represent, in a way that is similar to how it is presented in ResolutionOnline.
Finally, if this tool ever attains a regular user base, an extension allowing the use
of dark mode would eventually be demanded, so this should also be considered.
