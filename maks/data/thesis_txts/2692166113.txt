Abstract
General, autonomous competences are desirable abilities for mobile robots to interact
in man-made environments. The abilities to navigate through openings and manipulate
objects will be investigated in this thesis. The focus lies on general applicability, regard-
less of robot model.
Opening detection was analyzed and led to a general definition relative to the physical
dimensions of the mobile robot. Based on conducted research, an algorithm for opening
detection was implemented using a 2D LiDAR as sensory input. The opening detection
has an accuracy of 97.14% and a sensitivity of 96% in plain indoor environments. Path
planning was used to navigate the mobile robot towards the center of the opening with
a displacement error of 8.5 cm and an orientation error of 6◦. The combined algorithms
performed with high repeatability in a constructed environment.
A larger mobile robot was equipped with a UR5 manipulator to interact with objects.
A 2D camera was mounted on the manipulator to function as visual perception. The
transformation matrix from camera frame to end-effector was obtained through hand-
eye calibration using a calibration pattern. The calibration process was automated for
use as a general robot competence. The precision of the transformation matrix was vali-
dated by modelling the camera and applying BLOB detection. The transformation has
a high precision with an average translation error < 0.4 mm and rotation error < 3.9◦.
The competence was successfully applied to the task of replacing an IV drip bag with a
mobile robot with manipulator.

CHAPTER 1
Introduction
Walking through a doorway or picking up an object are simple tasks for humans. The
underlying functionality of visual processing, logic and body coordination happens in-
stinctively and autonomous.
These simple tasks are desirable abilities for robots to
mimic human behavior. Robots are gaining a larger role in the industry due to the re-
duced cost of manufacturing and eﬀiciency. However, many of the implementations are
bound to repetitive tasks without perception or changeable environments. The demand
for more autonomous robots with human-like abilities is increasing, and general robot
competences are therefore needed.
This thesis will address control and navigation of a mobile robot with a manipulator.
The focus is on developing general and broadly applicable competences independent of
robot model. Specifically, the mobile robot should detect and navigate through openings
as well as detect and manipulate objects with the mounted manipulator.
The ability to detect and navigate through openings must be based on a general
definition acting as the human reasoning. The principle of opening detection will be
analyzed based on sensory input to develop a general autonomous algorithm. An object
will be manipulated by coordination between the robotic end-effector and visual percep-
tion from a 2D camera. The thesis will consider hand-eye calibration and demonstrate
the applicability by replacing IV drip bags with the manipulator of a mobile robot.
1.1
Objectives
The objectives of the project relate to two separate parts; mobile navigation and hand-eye
calibration. The primary objectives for navigation is to define an opening for a general
robot, and to develop a solution which can analyze the environment for such openings.
The solution must additionally be able to plan and navigate the robot through the
detected opening.
An additional objective is to develop an automated and easy method for hand-eye
calibration (transformation from camera to robotic end-effector). The process should be
implemented and assessed for a Universal Robot arm with a camera mounted.

CHAPTER 3
Navigation Analysis
3.1
Introduction
An important attribute for mobile robots is the ability to navigate in human-made envi-
ronments. These environments do often include one or more openings such as doorways.
Detection and navigation through openings can therefore be considered a desired general
mobile robot competence.
This chapter will include an analysis of opening detection and navigation. The anal-
ysis will include an interpretation and definition of what an opening is and focus the
implementation on the available sensors on the mobile robot.
3.2
Analysis Of Opening Detection
When operating in an indoor environment, the detection of openings is crucial for nav-
igation. The openings (usually doorways) connect spaces otherwise separated by walls
or obstacles. For the mobile robot to achieve navigational freedom similar to humans,
the robot must be able to detect and navigate through these openings. Usually, mobile
navigation is based on an existing, known map, where the robot is able to perform path
planning. However, it is not always the case that the indoor environment is known be-
forehand. When deploying mobile robots in unknown or changeable environments, the
robot is more dependent on sensory feedback.
Before any detection and navigation algorithm can be designed and implemented,
the definition of an ”opening” (i.e. the goal which should be detected) must be declared.
At first thought it might seem obvious, but the definition will have a great impact on
how the algorithm can be implemented.
Autonomous navigation in a semi-unknown environment is desired, and the defini-
tion will naturally become somewhat abstract. If the task was to navigate in a single,
known building, the definition might be that the openings are doorways of X width and
Y height as assumed in many papers. In such scenario, a camera could be used with
shape detection for rectangles or certain colors. However, in an unknown environment,
an opening does not have to be a standard doorway - it might as well be double door,
two obstacles or a corridor with varying height and width.
To achieve generally applicable, robust navigation, the definition of an opening must

be as broad as possible. Further, the definition must apply for mobile robots in general,
but still be dependent of the specific robot. As a thought experiment for the reader, one
can observe multiple openings by just looking around the room. An opening can be the
space between the legs of a chair or the door frame. All of these openings will be denoted
as potential openings in this thesis. However, not all of these potential openings ”make
sense” for the robot to be considered an actual opening. It only makes sense, if the robot
is able to move through the opening. Hence, actual openings must be at least as wide
as the width of the robot. Further, an actual opening must be at least a factor of τdepth
times as deep as the depth of the robot. This definition is both dependent of the specific
robot while also being applicable for most scenarios. Depending on the dimensions and
the wheel configuration of the robot, the factor will describe the maneuverability needed.
The definition can be summarized as:
Definition 3.2.1 An opening is defined as a space between two objects, which is at least
as wide as the width of the robot and a factor of τdepth times as deep as the robot in both
directions of the opening.

3.3
Related Work
Based on the on-board sensors of the robots described in Section 2.1, there are two pri-
mary approaches to measuring the surrounding environment; by using the camera or by
using the laser rangefinder. Both approaches rise diﬀiculties and possibilities - and both
are widely used within the area of door detection. Much of the current work within the
area of opening detection tends to focus on classification of doors using laser scanners or
vision.
Regarding vision-based approaches, both 2D and depth analysis is widely used in
studies. Yang and Tian (2010) uses 2D vision-based door detection for unfamiliar, in-
door environments. A robust method is achieved using Canny edge detection for line
extraction combined with corner detection. The proposed algorithm achieves 91.7% true
positive rate. However, the method relies on the geometry of a door i.e., vertical and
horizontal lines of a doorframe, and that it has at least a certain width and height. Sha-
laby et al. 2014 presents a similar approach based on geometry with MATLAB, but only
with closed doors scenarios. Yuan et al. (2016) uses the Microsoft Kinect camera for
door detection using the depth feature combined with corner detection. The algorithm
is based on the Image Acquisition Toolbox in MATLAB and they assume that the cam-
era is perpendicular to the door plane and that the door has typical dimensions. The
method requires high computational power and long distances. Similarly, Zhou et al.
(2014) uses the depth functionality of the Kinect camera, but with feature extraction on
black/white images.
Other studies focus on door detection using laser rangefinders (indoor). Kaleci et al.
2016 uses a scenario related to this thesis. The proposed algorithm uses a rule-based
method using only 2D laser range data in indoor environments. The method relies on
observing that a bottleneck is present in the shape formed by the laser beam readings.
The proposed method was able to detect 90% of the doors with a false positive rate of
less than 6.1%. More relevant towards this thesis is the work by Diaz, Stoytchev, and
Arkin 2001. Only 2D laser measurements are used to detect doors based on least-squares
line fit and a box filter to remove false positive doors. The algorithm achieved a success
rate of 91% based on 32 missions. ElKaissi et al. 2006 detects open doors based on edge
detection on SICK LMS 2D laser measurements.

3.3.1
Interpretation of Related Work
Based on the papers presented in Section 3.3, the following can be derived from the
existing algorithms and research. Many research areas revolve around the detection of
doors and not the elusive definition of an opening as in Definition 3.2.1.
Several methods have been applied in the papers regarding the Microsoft Kinect cam-
era and 2D laser rangefinder - similar to the sensors mounted on the SMR and iRobot.
However, the described methods are highly based on assumptions. The vision-based
approaches tend to lean against geometry and specific aspect ratios of a door. This will
limit the algorithm to certain scenarios and could possibly fail at detecting asymmetric
openings. Both Shalaby et al. 2014 and Yuan et al. 2016 leaned towards the computa-
tional heavy MATLAB software. Furthermore, the depth camera rarely describes how
deep the opening actually is - whereas it is unknown, if the robot is able to drive through
the opening. The laser related approaches tend to be based on edges and linear feature
assumptions.
Due to the many assumptions and computational needs for vision-based approaches,
these will not be considered for robust opening detection. Furthermore, the methods
do not fit Definition 3.2.1. The laser rangefinder is more suitable based on the work
by Diaz, Stoytchev, and Arkin 2001 and ElKaissi et al. 2006, since the rangefinder al-
lows for accurate width and depth measurements with less assumptions. However, the
assumption regarding linear fit does not comply with the definition of an opening.
This thesis will differentiate itself from the related work by applying a more ab-
stract definition of opening detection to potentially achieve greater and more robust
navigational competences for mobile robots.
3.4
Assumptions
When using a 2D laser rangefinder as primary sensor for navigation, a few assumptions
about the environment must be considered.
The assumptions relate to the physical
attributes of laser and the placement of the sensor on the mobile robot.
• No transparent surfaces (e.g. glass).
• No absorbing surfaces (e.g. pitch black) which prevents the laser from being re-
flected.
• No surfaces with specular reflection (e.g. mirrors) (Ikeuchi (2014)).
• All obstacles are visible to the sensor (no obstacles or holes lower than the place-
ment level of the sensor).
3.5
Design Of Opening Detection Algorithm
Based on the analysis from Section 3.2, the overall design is conducted. The develop-
ing platform, Mobotware, sets some limitations regarding the design. Mobotware is a

plug-in based software framework for mobile robots Beck et al. (2010). The framework
features a simple scripting language for robot control called SMR-CL (Small Mobile
Robot Control Language), which includes commands for the most common actions. De-
scription of the language can be found in Automation and Control 2021. The plug-ins
are based on C++ and can interact with the on-board sensors of the mobile robot. Al-
ternate developing options are e.g. Python algorithms, which can communicate through
sockets or text-based files. The process is slower, as it introduces additional data com-
munication/transfer and computation in a less execution-optimized language. For the
opening detection, the timing and execution time is crucial. The system must react
fast to avoid collision in the changeable environment. The C++ plug-ins are therefore
favored because of the fast execution and low level placement relative to the laser server.
The algorithm is then to be implemented as a C++ plug-in combined with control from
SMR-CL.
The four subtasks defined in the Analysis Of Opening Detection Section are converted
to realizable tasks minded towards Mobotware and laser rangefinder sensory input. The
flowchart in Figure 3.2 shows the specific functions needed to implement an opening
detection algorithm. The plug-ins are executed from SMR-CL, where the given script
is ”pushed” - in this case to the laser server handling the laser rangefinder. The design
flowchart shows that computational functions are handled in C++, where the commands
related to robot movement is handled in SMR-CL based on feedback from the C++ plug-
in.
Figure 3.2: Design flow of the functions needed in order to implement an opening
detection algorithm using SMR-CL and C++ plug-in for the laser server.
The design is based on classification of the laser measurements to group data, which are
likely to belong to the same side of an object. This will make the searching for openings
easier. Based on the detected opening, the direction and distance to the opening must
be calculated and returned to the SMR-CL script, to be able to use the commands for
control and navigation. When the robot has reached the opening, a combination of feed-
back from the C++ plug-in and SMR-CL commands should be used to safely navigate
the mobile robot through the opening.
As the design implies, the Sense - Plan - Act architecture is to be implemented.
Two cycles are considered; one for navigation towards opening and one for navigation

through the opening. However, concurrent feedback must keep the planning updated.
The specific functional implementation of each subtask will be presented in the fol-

Several approaches can be used for the laser-based classification task. Three algorithms
were evaluated as potential candidates for implementation; Linear regression, RANSAC
and Euclidean distance. As already discussed, the linear regression1 requires too many as-
sumptions of the opening structure - i.e. straight walls without obstacles. The RANSAC
algorithm2 uses an optimized linear fit, but suffers from the same assumptions. An al-
ternative could be the processing of every segment as polynomials. This would give a
great insight in the shape of the detected objects, but would be a demanding process to
execute. The shape-based information is not necessary, as the robot should only know,
whether there is an opening or not. Therefore, the most suitable approach would be
to treat every measurement as an individual point in relation to a given segment. By
treating the points individually, the distance between them can be compared to the
width of the robot, to determine if a potential opening is present. This method will be
denoted as the ”Euclidean distance” approach.

CHAPTER 4
Euclidean Distance
Algorithm
The objective was to design an algorithm which took the least number of assumptions
regarding opening detection - namely only those restricted by the laser sensor (Section
3.4).
The algorithm must perform several tasks, to determine if an opening is present in
a given laser scan. A block diagram of the tasks can be seen in Figure 4.1. Four tasks

The segmentation is used to group measurement points which are likely to belong to the
same obstacle. Such segmentation has been implemented by combining analysis of the
measurements in both Polar and Cartesian coordinates.
It was decided to segment the points based on lines (not linear, but lines which
describes a side of an object). A line could be a wall or the half-circle of a garbage bin
as seen in Figure 4.4 and 4.5. The reasoning is that segments are easier to process, and
allows the opportunity to remove outliers if they do not consist of enough points.
The primary detection criteria is based on a height threshold on the Polar coordinates.

Considering an object with corners/edges in Polar coordinates, the corners will be shown
as local minima or maxima. By observing if the slope of two points is ascending or de-
scending, it can be evaluated, whether a local minima or maxima has been found. The
fundamental idea is similar to the previous method, as it compares to a threshold. The
algorithm uses the segments already separated by the previous method to search for
local minima and/or maxima. Taking any given segment, the current point, point(i), is
compared to the previous point, point(i −1). If the previous value is larger than the
current value, the slope is descending and vice versa. When a change in de- or ascension
is observed, the segment will be divided between the points.
The two detection algorithms will separate segments based on height and edges. The
algorithms are sensitive to noisy data with varying distance measurements. Such scenario
can potentially divide a real single segment into multiple unwanted segments. The laser
sensor mounted on the SMR is however quite accurate, whereas the potential problem
did not constitute a threat. Outliers will then be segments with one or few points. These
outlying segments must be handled by the opening detection algorithm.
4.3
Finding the Shortest Distance
This section will cover the two blocks after the preprocessing function (i.e., finding the
start and end point of every segment and finding the shortest distance).
The preprocessing function returns an array, describing where the points are seg-
mented. To avoid unnecessary computations, segments with less than four points are
not processed1. The number of points needed can easily be changed when adapting to a
certain environment. However, when removing these small segments, a lot of processing
can be avoided.
To find an opening, an empty space must exist between a segment and any given point.
This empty space must be at least as wide as the robot + an offset (woffset) for naviga-
tional irregularities. The function is therefore dependent on the robot width.
Every segment will have a starting point (left-most point) and an ending point (right-
most point). The left and right point can be used to optimize the function for finding
the shortest distance. Instead of calculating the distance between all points, the left
point should only be compared to points from measurement no. 1 up until the left point.
On the other hand, the right point will only be compared from the corresponding point
until the last measurement (no. 512). For large segments, this approach will reduce the
calculations significantly. An illustration of the concept can be seen in Figure 4.6. The
two dashed lines represent the start and ending point of the segment. All points left of
the orange line will be considered, and all points to the right of the purple line will be
considered when calculating the distance.

Ongoing Scans
The detected opening is returned to the SMR-CL script as described in Section 4.5. Al-
though the bounding box (box filter) ensures that the potential opening is an actual
opening, this is not true in every scenario. One particular case might occur which can
be a threat to the algorithm.
The scenario arises, when the half-circled laser FOV ”cuts” a corner, and mistakenly
interprets it as an opening. The algorithm will be unable to differentiate the missing
view from an opening. The scenario has been visualized in Figure 4.10. Since the open-
ing detection is only based on laser measurements and no camera feedback, the problem
must be solved by ongoing/recurring executions of the detection algorithm. The only
way to verify this is by approaching the opening, until all obstacles are within the FOV.

4.5.2
Inaccuracies Related To The Algorithm
The plug-in covering the Opening Detection Algorithm and the Navigation Towards
Opening contain inaccuracies regarding precision. The Opening Detection Algorithm is
based on a bounding box, where the lower center point is used as the goal point for the
Navigation Towards Opening part.
The lower center point is sensitive to the length of the mobile robot and the factor
τdepth, as this will dictate the distance from the opening. The further away the laser
rangefinder detects the openings, the more likely the bounding box is to be ”tilted” -
placing the desired goal point further from the actual middle of the opening. An example
can be seen in Figure 4.13, where d2 > d1 since the depth is larger. The box can become
skewed due to too few points at long distances or noise objects such as a garbage bin or
other objects near the opening.
Figure 4.13: Illustration of the bounding box with two different robot depths. The
depth will increase the error relative to the center point.
The closer the robot is to the opening; the bounding box is more likely to represent a
non-tilted rectangle.
An additional inaccuracy is introduced by the odometric inclusion in the algorithm.
Although the odometry has been calibrated using the UMBmark test (Appendix A.6:
Odometry Calibration (SMR)) a drift is inevitable and considerable after longer driven
distances. It is therefore important that the opening is evaluated again before driving
through. When using the laser rangefinder for another position estimate, the measure-
ments will be relative to the current position regardless of the odometry. This is an
advocacy for the need of a separate algorithm for navigation through the opening.

4.8
Navigation Through Opening
4.8.1
Design
At this point in the pipeline of the diagram in Figure 3.2 (Section 3.5: Design Of Opening
Detection Algorithm) the last ”box” is considered. The mobile robot is now positioned
approximately in front of the opening.
The orientation and placement are however
not guaranteed due to the inaccuracies related to opening detection from long range.
Therefore, the navigation through the opening will consist of three overall actions;
1. Reposition closer to the opening and closer to the center of the opening
2. Adjust orientation towards opening
3. Navigate through the opening
To realize the three actions, it is quickly observed that the first action is functionally
similar to the plug-in for opening detection, but with a few different requirements due
to decreased distances. From a design point of view, two approaches could be taken re-
garding the implementation. The plug-in for opening detection could either be modified
to also handle the navigation through the opening, or a new plug-in could be designed
to only handle navigation through openings. Both approaches reveals benefits and dis-
advantages as presented in Table 4.7.

The comparison revealed that the disadvantages for a single plug-in, correspond to the
advantages when considering multiple plug-ins. Especially the ability to use the plug-ins
separately is a large bonus. This means that the plug-in for navigation through opening
can be used without the plug-in for navigation towards opening. E.g. if knowledge is
present for a given environment, it might be more robust to use line fit (Diaz, Stoytchev,
and Arkin 2001) or RANSAC algorithm for detection of doors without noisy objects.
The given algorithm could then be followed by the navigation through opening plug-in,
which only assumes that the mobile robot is positioned somewhere in front of the open-
ing. Thereby, the plug-in can be combined with both existing algorithms and ensure
greater applicability for future algorithms or improvements.
Based on the brief analysis, the navigation through opening algorithm will be im-
plemented in an individual plug-in - separated from the navigation towards opening
plug-in.
The plug-in will be using a lot of the same functionality as the previous plug-in,
navigation towards opening, since it still need to detect, whether it is an opening or not.
Afterwards, a suitable solution has to be developed for adjusting the positioning to the
opening, so it will be centered. To find the best approach multiple methods will be
discussed.
Method 1:
The first method suggests that the LiDAR-sensor is used to find the corner indices for
the opening, in the range of 0-5112. Afterwards, it will compare the indices with the
middle values (255,256) and take the differences. The motor speed on each wheel can
then be controlled by a scaled factor of the differences in indices.
This method requires that the robot is already centered regarding the opening. Oth-
erwise, when it has turned to face the middle of the opening, it doesn’t account for
being placed wrong. An example can be seen in Figure 4.22, where it is clearly why this
method isn’t suitable in most cases, but can still have some practical uses.
Figure 4.22: This method use the indices for both corners of the opening. The middle
indices are used to adjust the heading toward the middle of the opening by
applying more speed on one of the wheels (right wheel increases velocity
in this case) .

Method 2:
The second method relies on the drive command from the SMR-CL language. The SMR-
CL drive command works by driving to specific coordinates in the odometry-frame for
the robot. The drive command takes (x,y,theta), where theta is the angle, it should
approach from. This also describes the ending angle of the robot’s position, so theta
can be changed to the desired ending heading towards the opening. The disadvantage
is that if it reaches the position in (x,y), it will stop and not adjust the angle. This
happens because, it is not designed to go to an ending angle, but rather use theta as
an orientation towards (x,y) position. This can give scenarios, where it approaches the
opening, while it is no longer possible to correct the error towards the opening. The
Figure 4.23 shows that it will drive to the lower ”+”.

the input from the LiDAR scan in laser-frame, and then transform the laser-frame to
the odometry-frame + translation. Figure 4.24 shows which parameters is taken inside
the controller, and how the transformation looks from laser-frame to odometry-frame.
The controller takes the distance to each corner in the laser-frame and in the odometry-

The chosen method is a combined implementation of method 3 and method 1. These
seem to be able to correctly reposition the robot, since it recalculates the position com-

will adjust both the heading from the laser and the center of mass. When the robot gets
close to the opening, it should switch to method 1, where the heading is adjusted based
on the laser. After it has been aligned to the center of the opening, it can be navigated
through, and the task has been completed.
4.8.2
Implementation
The navigation through opening plug-in is to be executable independently from the nav-
igation towards opening plug-in. Therefore, many of the same functions are executed
within this plug-in. The laser scan will be segmented and converted to Cartesian coordi-
nates exactly as described in Section 4.1. Additionally, the opening will be found again
by executing the opening detection to verify and receive the coordinates.
The algorithm will use the coordinates of the opening to reposition the mobile robot
closer to the center of the opening. This action is most suitable for smaller openings
and might not be necessary for large openings.
The algorithm will calculate the distance from the corners of the opening to the LiDAR
and the center of mass.
Here it calculates difference in distance, where it finds the
longest for the LiDAR and the center of mass. It then normalizes the distances, which
is used for the controller. The controller features saturation, whereas it will not output
too much to the motor driver. When the robot is under 25 cm from the opening, it will
switch to only rotate around the center of mass, until it adjusted the heading parallel
to the opening. The approach for the code is described in Figure 4.25.

the robot width and depth, and navigates the robot towards the opening. The latter
navigates the robot through the opening.
The opening detection is achieved by measuring Euclidean distance between laser
rangefinder measurements and verifies the opening with a box filter (bounding box).
The navigation through the opening is implemented using a dual controller based on
input from the odometry and laser relative to the center of the opening.
The plug-ins were tested individual and combined. The opening detection performs with
an accuracy of 97.14% while being 1 meter away from the desired opening. The plug-in
navigates the robot to the desired coordinate within a radius of 8.5 cm.
To test the robustness, an SMR-CL script was set to loop a route through an opening
10 times. The test performed successfully with correct opening detection and navigation
without collision.

5.1
Introduction
To allow the mobile robot to interact with artefacts in an environment, a manipulator
has been attached to the top. In order for the manipulator to react on sensory feedback,
it must be calibrated. The sensory feedback is based on a camera mounted to the end-
effector of the manipulator. The end-effector frame must be calibrated relative to the
camera frame in order to respond on the sensory information.
This chapter will describe the calibration of a Universal Robot manipulator and
camera using OpenCV and frame transformations.
The description will lead to the
implementation and test of an automatic calibration script/library in Python as a general
competence independent of camera placement on the manipulator.
To validate the
calibration, a simple BLOB detection algorithm and modelling of the camera was a
necessity.
5.2
Analysis Of Hand-Eye Calibration
Calibration of sensors are an important part of robotics to ensure correct execution of
tasks. A good calibration can take long time and be a result of manual adjustments.
Cameras as sensors are no exception from the need of calibration. Most cameras
introduce some sort of distortion to the images. Although not necessarily visible to the
naked eye, small deviations in the images can have crucial impacts on the manipulator’s
interpretation of the surroundings based on the camera feedback.
Besides the calibration of the camera, the manipulator must be calibrated relative
to the camera frame. This is denoted ”hand-eye” calibration and describe the transfor-
mation from the camera-frame to the end-effector of the manipulator.
Within this scope, it is desired to design and implement an automatic calibration method
for both camera and hand-eye calibration. The process must be easy to execute. The
automated method is intended to be used by students or others, who does not have par-
ticular knowledge of image processing concepts. Hence, the goal is to develop a method,
which can allow ”anyone” to calibrate any given camera placed on the end-effector of
any given UR manipulator.
The camera - and the hand-eye calibration will be analyzed as two separate instances.
==PAGE:60==
50
5 Manipulator
Camera Calibration
Correction of image distortion is widely covered in the literature and many methods
and approaches exist.
Zhang (2000) differentiates between two technique-categories;
Object-based calibration and self-calibration. The object-based approach is performed by
observing a calibration object with known geometry in 3D space. The self-calibration
method requires no objects and is based on the cameras’ internal parameters and dis-
placement in the images taken. It is diﬀicult to obtain reliable results in this method,
while the object-based technique requires an elaborate setup. Zhang (2000) presents a
new object-based calibration method, where the object is a calibration pattern.
Due to the simplicity of the calibration pattern, the method has since evolved and
become one of the most used techniques. Both MATLAB and OpenCV contains func-
tions for camera calibration using calibration patterns. Y. M. Wang, Li, and Zheng
(2010) uses a combination of MATLAB and OpenCV for eﬀicient camera calibration.
Datta and Kim (2009) presents a method for accurate camera calibration using iterative
refinement of control points and tests on patterns with circles and rings. The method
proves to perform better than OpenCV, but with more complexity.
The Mobotware framework does already support OpenCV for image manipulation. Due
to the documentation, flexibility and broad usage of the OpenCV library, this will be
used with the object-based calibration method. While being Python and C++ based,
it will fit under the desire of achieving a user-friendly and easy calibration process of
the camera. Hence, applying the method will not introduce additional changes to the
Mobotware framework and can therefore be used with existing implementations.
Hand-Eye Calibration
When the sensor (camera) has been calibrated for distortion, the sensory information is
assumed to be undistorted and adequate for hand-eye calibration. Just like the camera
calibration, the hand-eye calibration has been widely covered in the literature.
The
basic setup has been illustrated in Figure 5.1, where a camera has been mounted on the
end-effector. The particular configuration is an ”eye in hand” - opposite to the other
hand-eye configuration where the camera is fixed to an object facing towards the manip-
ulator.
The goal of the hand-eye calibration is to obtain the transformation from the camera to
the end-effector (gripper) - i.e. gTc on the figure.
Given a Universal Robot arm, the transformation from base to end-effector (bTg)
is known (obtainable through forward kinematics with Denavit-Hartenberg parameters).
The transformation from base to target can be obtained mathematically based on manual
readings. The transformation from camera to target (cTt) can be obtained through image
processing software by taking images of a checkerboard (as illustrated on the figure).
Hence, the path from camera to end-effector (gTc) can be expressed as the product of
each transformation as presented in Equation 5.1, where ”•” is the dot product.

Many modern approaches has optimized and automated the process of obtaining the

Many

to task frame transformation and can thereby ”skip” a step compared to the manual
approach. Due to the eﬀiciency and popularity of the method, it is already implemented
in the OpenCV library.
Hence, it would be obvious to use the library, as it is already installed - thus making
the automated script easier to execute. A strong benefit of using the checkerboard pro-
cess is that the same images can be used for both the camera calibration and hand-eye
calibration simultaneously.
5.3
Design
The analysis showed that multiple approaches can be used to obtain the transformation
from camera to end-effector. The objective is to design an automatic calibration method,
which can be used by students without particular knowledge of hand-eye calibration.
The process should remove the comprehensive task, which is necessary in every project,

where the camera is mounted on a robot arm.
Experience from the authors of this
thesis shows that the process easily ends up taking long time. For smaller projects this
repetitive task should not consume valuable time.
The primary design requirements has been listed below;
• Ease of use
• High precision (error < 5 mm)
• Python implementation
The requirements are minded towards use with the robot arm series by Universal Robot
(UR). DTU has an extensive Python library with communication setup for UR arms1,
whereas the obvious target platform for the hand-eye calibration would be Python.
The requirements does not specify any particular calibration method as presented in
the section Analysis Of Hand-Eye Calibration. This is based on the desire of comparing
the accuracy of several methods. As the hand-eye calibration process is intended as a
general competence, the precision should ideally be as good as possible. However, the
calibration should be considered an overall applicable method, whereas higher precision
might require a more elaborate method with implementation-based knowledge from the
user. Thereby, the precision requirement of the design is an error of less than 5 mm.

5.4
Camera Calibration
Cameras are usually manufactured with an optical lens, which projects the light onto a
surface, which then stores the image electronically or chemically. Stankiewicz, Lafruit,
and Domański (2018) defines lens distortion as a deviation from the ideal projection
considered in the pinhole camera model. The pinhole camera model is a mathematical
relationship, where the projection hits the image plane through a point (pinhole) with
no lenses used to focus light. Lens distortion occur, as the ideal model cannot be realized
in practice.
The distortion is usually separated into radial and tangential distortion. Tangential
distortion occurs, when the lens and the image plane are not parallel. Radial distortion
is the most visible on the images, and are caused by the deviation from the rectilinear
projection. Examples of the radial distortion can be seen in Figure 5.2. The effect is
worse at the sides of the image compared to the center. The removal of such distortion
is referred to as camera calibration.

6.1
Introduction
The mobile robot navigation and manipulator competences developed throughout this
thesis has been considered as general competences. This means that the modules are
intended to solve a broad range of different real life tasks.
As a proof of concept, the hand-eye calibration was applied on a mobile robot with
a manipulator attached. To utilize the competences in a useful scenario, the Mohamed
Bin Zayed International Robotic Challenge, MBZIRC 2023 competition was considered.
The third challenge considers ”Hospital Applications” with focus on bed side assistance
and sterilization (MBZIRC 2021).
The chosen task was to change an intravenous (IV) drip defined by the following
pipeline;
1. Detect empty IV drip bag on a stand
2. Remove the empty IV drip bag
3. Detect unused IV drip bag on robot tray
4. Pick up the IV drip bag and place it on the stand
The task requires hand-eye coordination and the image detection algorithms already
implemented. Considering the challenge, not much information is known about the IV
drip bag. Therefore, the primary assumption of the implementation is that the bag can
be identified based on a marker. Alternate, more complex approaches could be a neural
network for IV drip bag recognition. For simplicity, this implementation assumes two
red dots are placed on the drip bag, where the right dot is larger than the left. The
difference in size will allow detection of orientation of the bag. However, the orientation-
based approach required an extension to the image analysis script.

6.3
Inverse Kinematics
The robots from Universal Robots can easily end in unwanted positions when moving
in Cartesian coordinates. Physically aligning the joints results in loss of degrees of free-
dom, causing the movement to occur unexpectedly aggressive and often with the result
of triggering the emergency brakes on the joints. The problem can be handled by com-
manding the robot in joint coordinates instead of the linear movement in the Cartesian
coordinates.
The coordinates of the drip bag is returned as end-effector coordinates as described
in the previous section. This means that the goal point is known, and that the cor-
responding joint configuration can be found using the inverse kinematics - and hence
skipping the unreliable Cartesian movements. The inverse kinematics is the process of
determining the joint parameters that provide a desired configuration.
To use the inverse kinematics with the Python implementation, the work by Marie Claire
Capolei (2021) was applied. The work is a Python script, which takes the desired position
[x, y, z] and rotation vector [rx, ry, rz] and returns eight inverse kinematic solutions of
the UR5. The implementation does not account for large, unwanted movements such as
colliding joints or collision with the base to which the robot is attached. To select the
most appropriate solution, a scoring system was implemented to accompany the eight
solutions.
The scoring system is intended to compare the (solution) rotation from the current
pose in each of the eight joint solutions. The difference from current pose to solution
describes the total movement which the joint should rotate. Before comparison, each
solution is normalized by ±360◦, to check if the movement can be accomplished more
effectively. The solution with the smallest rotation from the current pose will be assigned
1 point in a score matrix. This process is performed for each of the six joints. Ultimately,
the solution with the highest score is selected, as this will result in the least rotations.

6.5
Replacing an IV Drip Bag
The process of replacing the IV drip bag introduces two additional assumptions; The
approximate position of the drip bag stand is known, and the stand is within reach of
the manipulator.
The assumptions can easily be ruled out by detecting the drip stand by e.g. the laser
rangefinder and taking advantage of the mobility of the robot to ensure correct position.
However, in this implementation, the stand is assumed positioned in front of the mobile
robot as seen in Figure 6.5. The particular position of the stand is not important, as
long as it is within reach of the manipulator and within the field of view of the camera.
Initially, the manipulator is placed in a 90 degree ”searching” position, where the image
analysis will be deployed to locate the empty drip bag on the stand. The searching
position is shown on the figure. The image analysis uses the Laplacian variance, to
ensure that the camera is focused (i.e. that auto-focus has saturated). The Laplacian
is a kernel, which searches for edges in the image. When the camera is unfocused, the
Laplacian value will be low, where a sharp image will have a high value. The detected
camera coordinate is then transformed with the hand-eye transformation.
The manipulator is commanded to move towards the drip bag by using joint coor-
dinates obtained through the inverse kinematics scoring function. After having moved
the end-effector to the drip bag, the position is stored, and a hard-coded movement is
initiated. The movement is relative to the position, and will grab the bag from the side,
and lift it off the hook on the stand.

CHAPTER 7
Conclusion
The aim of this thesis was to develop general competences for control and navigation
for a mobile robot with manipulator. The competences included opening detection and
hand-eye calibration for the manipulator.
The mobile robot should be able to navigate in unknown, changeable environments
by detecting openings. An analysis led to the definition of a general opening based on
the physical dimensions of a mobile robot.
The ability to detect and navigate through openings was implemented as a C++ plug-in
for Mobotware. The opening detection was based on the Euclidean distance between
segmented measurements from a 2D laser rangefinder. The algorithm has an accuracy of
97.14% and a sensitivity of 96% from a distance of 1 meter in an environment without
obstacles (based on experiments on a SMR). Deploying obstacles to the environment
decreased the accuracy to 90%.
When driving towards the detected opening, the mobile robot reaches the goal within a
8.5 cm radius of the ground truth and an orientation error of ≤6◦.
A dual parameter controller was implemented as a separate plug-in to navigate the
mobile robot through the opening by adjusting the velocity of the wheels.
The combined plug-ins resulted in robust opening detection and navigation. The
SMR was able to successfully repeat the detection and navigation 10 times at a track
course.
The hand-eye calibration process was developed as a general competence independent
of camera placement on the manipulator. A guide for easy calibration was developed.
The resulting hand-eye transformation matrix describes the relation from camera to
end-effector. Image analysis for visual perception was combined with a model of the 2D
camera. The image analysis was used to detect points with the camera, which could
be transformed to end-effector coordinates.
The transformation with image analysis
performed well and revealed an average translation error of < 0.4 mm and a rotation
error of < 3.9◦on a UR5 with a mounted camera. The transformation was used to
successfully and autonomously replace an IV drip bag from a stand.

Operation Mode
The plug-in is configured with four possible modes; all, left, right and free. The mode is
parsed from SMR-CL when calling the laser command;
laser "push t='0.1' cmd='zoneobst mode=x'"
where x represents one of the acceptable parsing integers; 1 (left), 2 (right) and 3 (free).

an opening is expected to be found on either the left or right side of the scan. The

computations occur as less points will be evaluated in the function. The ”free” mode
takes two arguments and requires an initialization of;

A.11
Resetting The Plug-in
The ongoing scan feature from Section 4.5.1 sets certain limitations for the plug-in.
The ongoing scan keeps track of the original opening coordinate, and compares it with
openings found while driving. When the opening has been reached successfully, or the
robustness counter has stopped the robot, an idle state has been achieved by the plug-in.
The plug-in returns that no opening can be found at the given position and orientation
- or that the goal has been reached. The idle state requires a reset of the plug-in, before
it can find new openings - whereas an action must be taken from the SMR-CL script.
==PAGE:104==
94
A Navigation
Similar to the laser Operation Mode from Appendix A.3, the plug-in can be reset by
assigning a bit from SMR-CL by;
laser "push t='0.1' cmd='zoneobst reset=1'"
where a value of 1 will reset the values of the script.
The functionality adds vital usability for more advanced autonomous routes, as more
than one opening can be found when resetting the script after each opening. The call
will work, even though the plug-in is currently being pushed by the laserserver.

B.1
Camera Calibration: Distortion
The radial distortion was described in Section 5.4, Camera Calibration.
The radial
distortion can be solved with the Brown model (Wu, Wei, and X. Wang (2017)), which
have been expressed by the following equations in OpenCV (2021a);
xcorrected = x(1 + k1r2 + k2r4 + k3r6)
ycorrected = y(1 + k1r2 + k2r4 + k3r6)
(B.1)
Where (xcorrected, ycorrected) and (x, y) are the corresponding coordinates of an undistorted
point and a distorted point in the image.
The variable r =
√
(x −x0)2 + (y −y0)2
express the Euclidean distance of the distorted point to the distortion center (x0, y0).
kn = nth describe the radial distortion coeﬀicient.
The tangential distortion part is solved by equations;
xcorrected_tangential = x + [2 · p1 · x · y + p2(r2 + 2x2)]
ycorrected_tangential = y + [p1(r2 + 2y2) + 2 · p2 · x · y]
(B.2)
Where pn = nth describe the tangential distortion coeﬀicient.
Both the radial and
tangential coeﬀicients in the Brown distortion model is expressed as an infinite series.
However, higher order coeﬀicients are not considered in OpenCV.
In OpenCV, the function calibrateCamera returns, among others, the distortion
coeﬀicients;
Distortion coefficients = (k1
k2
p1
p2
k3)
(B.3)

Guide to Hand-Eye Calibration 
 
This guide will help the user obtain the following: 
- 
Camera calibration matrix (focal length and center point in pixel) 
- 
Camera distortion coefficients 
- 
Hand-Eye transformation matrix (rotation and translation of camera relative to end effector) 
The guide is based on Python and requires a camera mounted on a robot arm. The guide uses the DTU UR 
equipment functions. The guide assumes that the UR and camera is connected to the PC. 
 
1. Include/install the following packages in the Python environment. The minimum required Python 
version is 3.8.8 for OpenCV compatibility: 
 
pip             version==21.0.1 
numpy           version==1.20.2 
opencv-python   version==4.5.1.48 
pyserial        version==3.5 
datetime        version==any version 
serial-tool     version==0.0.1 
 
2. Download the folder from GitHub and save it in your working directory: 
https://github.com/chrwave/Hand_Eye_Calibration  
 
3. Print the checkerboard from the file “chessboard_pattern.pdf” or other resource. Measure the width 
of the black squares and count the dimensions of the board as shown below: 
 
The included chessboard has dimensions (9,6) and a width of 25 mm.

4. Edit the script “WorkingHandEyeCalibrationGuide.py” and change the variables directory, 
ImageAmount, FactorPictureScaling, CheckerboardSquareSize and Checkerboard as shown below. 
The directory can be found by typing “pwd” in the terminal. 
############################################## 
#Parameters that can be changed 
directory = 'D:\Scripts\PythonScripts' #Should be changed to your directory 
ImageAmount = 25               # Amount of images 
FactorPictureScaling = 1       # Should be in the power of 2. (1, 2, 4, 8, 16) 
CheckerboardSquareSize = 0.025 # Size of checkerboard square(s) 
Checkerboard = (9,6)           # Checkerboard dimensions 
############################################## 
With the exception of directory, all parameters can be left unchanged if using the included 
chessboard. It is recommended to use 25 images, but the algorithm accepts values ≥3.

use 1, 2, 3 etc. depending on number of potential inputs. The resolution can also be adjusted here. 
#Camera initialization  
############################################## 
cam = cam.DTUCAMERA(1) 
cam.SetCameraResolution(1920,1080) 
cam.GetCameraResolution() 
############################################## 
 
6. Save the changes to the script and execute it. A new window will open with a live feedback of the 
camera. Position the robot such that the entire checkerboard can be seen with the camera. Take an 
image by pressing “Esc”. Move the robot to a new position and take another image. Continue taking 
pictures from different positions and rotations until you have images corresponding to the value in 
the variable ImageAmount. It is important to get as many unique poses as possible. 
Examples of images can be found in the folder “ExamplePictures”. 
 
 
7. The user will be prompted with the following message stating the results from the calibration 
algorithms; Park-Martin, Tsai-Lenz, Horaud and Daniilidis. The results should ideally converge. 
The transformation is from the gripper to camera frame. Rough manual measurements can be used as 
an estimate to verify the transformation.
