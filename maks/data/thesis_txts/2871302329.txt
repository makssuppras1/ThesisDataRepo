ABSTRACT
Accurate motion control of underwater vehicles is crucial for executing complex subsea operations.
This thesis focuses on the problem of rendezvous and docking control for the BlueROV1
platform, addressing both the challenges and practical relevance of these tasks in real-world missions.
Underwater rendezvous and docking control are essential because these maneuvers enable vehicles
to safely connect for collaborative operations such as maintenance, repair, data or payload transfer,
and even battery recharging.
Estimating the position and orientation of an underwater vehicle is particularly challenging due to
the lack of reliable GPS signals, the limited effectiveness of acoustic sensors in dynamic environments,
and the complexity of water dynamics. To overcome these difficulties, this work adopts a vision-
based approach leveraging a pattern of LEDs mounted on the target vehicle. This system enables
the visual estimation of relative position and orientation by extracting key features from
the LED pattern, providing real-time feedback for control. The method was refined and validated
to ensure robust operation under various conditions.
A comprehensive parameter identification of the BlueROV1 was also performed. Various
techniques, including system identification experiments and hydrodynamic analysis, were
used to determine key parameters such as mass, inertia, added mass, and hydrodynamic damping
coefficients. These parameters were crucial for developing accurate dynamic models and improving
control performance.
For both rendezvous and docking, two distinct control strategies were developed to address their
specific challenges. To ensure accurate testing and validation, a dedicated simulation environment
was implemented by adapting and improving a previous BlueROV2-based framework. This
allowed for a more precise representation of the BlueROV1’s dynamics and its interactions with
the environment.
In this framework, two control approaches were explored: a Proportional-Integral-Derivative
(PID) controller, designed to minimize the relative distance to the target, and a Model
Predictive Control (MPC) scheme, which optimizes trajectory tracking in six degrees of
freedom (6-DoF). Both strategies were evaluated in simulation, providing a detailed comparison of
their performance, strengths, and limitations in the context of rendezvous and docking maneuvers.
This work provides a framework for rendezvous and motion control in underwater robotics, laying
the foundation for future research in cooperative control strategies and docking applications for
autonomous underwater vehicles (AUVs).

Project Description
This project builds upon previous developments in vision-based systems for object recognition and
focuses on addressing the challenges of precise identification and alignment between robots. Unlike
traditional approaches that rely on fiducial markers such as AprilTags, this system leverages a
custom-designed docking module that integrates a camera, LEDs, and magnets. This setup enables
vision-based detection while ensuring a secure and reliable physical connection during docking
operations.
The robotic system used for development is the BlueROV, for which limited research and
documentation are currently available. Furthermore, the integration of the custom docking module
significantly alters the vehicle’s dynamic parameters, making existing models from previous studies
less applicable. As a result, advanced control strategies require a comprehensive and accurate
system model capable of representing the primary dynamic behaviors and predicting future outputs
in response to control signals. The process of modeling and system identification is inherently
time-consuming, as it involves extensive experimental work and subsequent analysis.
A challenge in this project is the estimation of the docking station’s pose relative to the AUV
without direct fiducial markers. Instead, the system relies on real-time LED pattern recognition to
extract pose information. This introduces additional challenges such as environmental variability,
lighting conditions, and water turbidity, which can impact the accuracy of vision-based localization.
This project focuses on the identification of the new BlueROV’s model parameters, which are
essential for developing advanced control strategies. In particular, it explores Model Predictive
Control (MPC), which allows for constraint handling and optimization over a prediction horizon,
and PID control, which provides a faster implementation and testing approach.
Moreover, since the docking station itself is subject to movement due to water currents and
disturbances, the control system must consider these external factors when guiding the AUV
towards docking. While the current implementation includes some robustness to disturbances, it
does not explicitly focus on external factor compensation. Enhancing the strategy to more directly
address water currents and co-visibility constraints remains an avenue for future work.

1.3
Problem Analysis
The development of a model-based control system for underwater vehicles requires an accurate and
reliable representation of the vehicle’s dynamics. Therefore, system identification is a fundamental
first step in the process, as it enables the formulation of a model that can predict the vehicle’s
behavior in response to control inputs.
Underwater vehicle modeling can follow different approaches, typically categorized as black-box,
gray-box, and white-box methods [4], depending on the availability of prior knowledge about
the system. In this project, system identification aims to support both the individual BlueROV
configuration and its coupled operation with the docking module. However, due to hardware
limitations, time constraints, and practical consideration, direct system identification experiments,
such as free-diving tests or towing tank experiments, were not performed.
Instead, the vehicle parameters were estimated using alternative methods. These include geometry-
based properties, simulations, and leveraging parameters obtained from existing literature, such
as [5], [6], and [7]. In addition to vehicle dynamics, it is crucial to have an accurate model of the
thrusters, their positions, and the resulting forces and moments they generate in response to engine
commands.
By addressing these challenges, this project aims to develop a more robust and adaptable control
system that enhances the BlueROV’s performance in docking and rendezvous operations, even in
uncertain and dynamic underwater conditions. To achieve this, different control strategies will be
explored, including PID for rapid deployment and testing, as well as Model Predictive Control
(MPC) for advanced constraint handling and optimal performance.
1.4
Problem Formulation
The first phase of this study focuses on developing/understanding a visual system using a pinhole
camera model. This involves refining the perception system to accurately extract position and
orientation information from the observed LED pattern.
Next, the updated vehicle dynamics are estimated by analyzing CAD models and geometries,
incorporating previous parameter identification studies. This step ensures a realistic and calibrated
dynamic model of the ROV, accounting for hydrodynamic effects and vehicle characteristics.
A simulation environment is then implemented in Gazebo, using the existing BlueROV2 model
as a base. In this adaptation, the thruster configuration is modified to accurately represent the
BlueROV1 setup, enabling realistic testing and validation of control strategies under various
conditions before deployment on the real system
Two distinct control strategies are developed: a classical PID controller for baseline performance
assessment and a Model Predictive Control (MPC) approach for optimized trajectory tracking.
A yaw alignment mechanism is incorporated to minimize lateral drift, addressing the inefficiency
caused by high hydrodynamic drag and limited lateral thrust capability. For the rendezvous
operation, where inter-vehicle communication is highly constrained, the most practical strategy is
to navigate toward an intermediate meeting point. This approach is particularly well-suited for
MPC, as it optimally balances control effort and convergence efficiency.
All developments are conducted while taking into account the available hardware and sensor suite,
ensuring seamless integration with the real system. The entire process has been significantly
time-consuming due to challenges such as hardware limitations, integration issues, and fine-tuning
of the perception and control pipelines.

BlueROV1
The platform used is the BlueROV1 produced by BlueRobotics (BR) [8]. The BlueROV, shows

Figure 2.2. BlueROV. Source: GrabCAD.

The BlueROV1 is built with an open-frame configuration, characterized by a box-like frame where
operational sensors, thrusters, and other mechanical components are mounted. The vehicle’s frame
is constructed from lightweight aluminum with plastic side walls, and a water-proof acrylic tube is

The BlueROV1 features six thrusters arranged in a specific configuration to enable movement in all
six degrees of freedom (DOF), as depicted in Figure 2.3. The arrangement of the thrusters is as
follows:
• Thrusters T1 and T2: These thrusters are primarily responsible for forward motion and
yaw control, enabling the vehicle to move forward or backward and to rotate left or right.
This capability is essential for navigating in complex underwater environments, where precise
maneuvering is often required.
• Thrusters T3, T4, and T5: These thrusters are oriented vertically to manage vertical
motion, allowing the vehicle to ascend and descend in the water column. Their symmetrical
arrangement ensures stability and balance during vertical maneuvers, which is crucial for
maintaining the orientation and position of the vehicle while conducting operations.
• Thruster T6: Positioned at the bottom of the vehicle, this thruster is designed for lateral
motion, allowing the vehicle to move sideways.
The BlueROV1 is designed with a slightly negative buoyancy, meaning it tends to sink rather than
float. This characteristic is advantageous for underwater operations as it helps the vehicle maintain
a stable position on the seabed when conducting stationary tasks, such as visual inspections or
sampling.
The BlueROV is powered by a battery, specifically the Turnigy 5.0 battery (4S, 14.8V), which
provides the necessary energy for all onboard systems. However, this battery has certain limitations
in terms of capacity, imposing a constraint on operational time before recharging is required. This
limitation is a critical factor to consider in planning missions, as it affects the duration of underwater
operations and necessitates careful energy management for optimal performance.

Thrusters
The mechanical platform includes six thrusters, all of which are T200 models manufactured by
BlueRobotics, as shown in Figure 2.4. The thrust capability of these thrusters is not identical
in both the forward and reverse directions. According to BlueRobotics’ specifications (based on

Figure 2.4. T200 Thruster
Figure 2.5. Basic AfroESC 30A ESC
The T200 thruster is powered by a three-phase brushless motor, which requires an electronic speed
controller (ESC) to function, as illustrated in Figure 2.5. The ESC is controlled by a PWM signal,
where the pulse width determines the thrust level. A pulse width of 1100 µs corresponds to the
maximum thrust in the reverse direction, while a pulse width of 1900 µs results in maximum thrust
in the forward direction. A pulse width of 1500 µs indicates zero thrust, with a deadband expected
in the range of 1500 µs ± 25 µs.[8]
2.3
Flight Control Unit
The Flight Control Unit (FCU), also known as an autopilot, is a compact hardware module designed
for motion control in robotic platforms. Its main purpose is to manage the most essential low-level
operations such as reading sensors, updating navigation filters and sending PWM signals to the
ESCs. The flight controlled used is a PixHawk as show in Figure 2.7.
The Pixhawk has an embedded ARM Cortex-M4 microcontroller, which executes code in real-time
at a high fixed rate. This capability is crucial for high-performance control, enabling precise thruster
command execution. This device runs the ArduSub software [9], an open-source control system
tailored for subsea vehicles. ArdaSub’s main control loop operates at 400 Hz and performs the
following functions:
1. Reads data from the IMU and a pressure/depth sensor.
2. Updates a navigation filter to estimate the vehicle’s attitude.
3. Runs an attitude controller for roll and pitch stabilization.
4. Sends ESC commands computed by the controllers or commanded by the user.
The IMU sensor, integrated into the Pixhawk, combines an accelerometer, a gyroscope, and a
magnetometer. The accelerometer can be used to estimate the acceleration of the vehicle in the
direction of the surge, sway and heave at the moment. The depth sensor is connected externally.
The specifications of the Pixhawk’s sensors are listed in Table 2.1.
These digital sensors produce quantized readings, making calibration essential to avoid biases and
incorrect scaling. The accelerometer and magnetometer are calibrated using specific procedures
that account for all axes during data logging. The pressure sensor calibration involves measuring
air pressure and adjusting readings to set zero-depth at the water’s surface. Biases and scaling
effects of the gyroscopes are estimated as part of the navigation filter.

Understanding the complex dynamics of Autonomous Underwater Vehicles (AUVs) requires a
robust mathematical framework. This chapter delves into the principles and equations that govern
the motion and control of these vehicles, providing a foundation for advanced control strategies.
The derivations and models presented here draw extensively from seminal works, including Fossen’s
comprehensive treatise on marine craft dynamics [12], the practical insights by Hansen [5], and the
methodologies detailed in related studies such as Makem et al. [13], [14] and also Smogeli [15].
This chapter not only establishes the theoretical underpinnings but also contextualizes the
application of these models in simulating and controlling AUVs under real-world conditions.
The approach integrates coordinate transformations, kinematics, and hydrodynamic forces, offering
a detailed perspective on the challenges and solutions in AUV modeling.

Chapter 4
Parameter Estimation
Estimating the parameters governing the dynamics of Autonomous Underwater Vehicles (AUVs) is
a crucial step in developing accurate and reliable models. This chapter focuses on the methodologies
and techniques used to identify and estimate these parameters. Additionally, the objective is to
implement this model in a simulation environment like Gazebo, enabling comprehensive testing
and validation of AUV dynamics under various operational conditions.
The approaches presented here are inspired by foundational works, including Fossen’s framework for
marine craft dynamics [12] and the practical applications demonstrated by Andreas Hansen [5]. In
the context of parameter estimation, significant focus was placed on the existing works of Andreas
Hansen [5], Stian Sandøy [6], Ole Eidsvik [17], Adam Aili and Erik Ekelund [18], Mikkel Nielsen [7],
Rajini Makam et al. [13], and others, who have also worked with the BlueROV1, employing various
strategies and setups to estimate various parameters. These studies provide valuable insights and
methodologies for tackling the challenges associated with identifying the dynamic properties of
underwater vehicles.
This chapter emphasizes that many of the parameters are derived through practical estimation
methods, often based on rough approximations and heuristic approaches. Some parameters are
extrapolated from experimental plots, while others are obtained from CAD models and small-scale
CFD analyses. These methods, while not always highly precise, provide a practical framework for
defining the dynamic properties of AUVs.
4.1
Thruster Identification
Thruster identification is a critical step in accurately modeling and controlling underwater vehicles,
as it provides the necessary parameters to describe the relationship between control inputs and
generated forces.
To model the dynamics of the Blue Robotics T200 thrusters effectively, the identification results
obtained in [5] and [19] were adopted. Hansen performed extensive testing, including various static
and dynamic experiments, to identify the parameters in Eq.(3.56) such as a+, a−, b+, b−, c+, c−
and the deadband region of the thrusters. These tests provide a detailed characterization of the
thrust generated as a function of the control signal sent to the electronic speed controllers (ESCs).
The experimental setup and methodology are described comprehensively in their work, with data
fitting performed to capture the nonlinearities of the thruster’s behavior.
The thruster parameters were determined using linear least-squares curve fitting on the control
inputs and force outputs. The deadband, d, was calculated as the average of the roots of the second-
order polynomials. Due to the symmetrical configuration of the T200 thrusters, a single curve
fit was used for groups of similar thrusters: steering thrusters (T1/T2) and front heave thrusters
(T3/T4). The performance characteristics of the remaining thrusters (T5 and T6) were fitted
individually. Table 4.1 provides an overview of the identified coefficients for the T200 thrusters.

4.4
Linear and Quadratic Damping Estimation
The calculation or estimation of drag coefficients, both linear (DL) and quadratic (Dn), Eq.(3.29),
can be achieved through a variety of approaches depending on the availability of experimental
facilities, computational resources, or reference data. The most common methods include:
1. Experimental Method (Measurements): This approach involves conducting physical
tests to directly measure the drag force (FD) acting on the vehicle under controlled conditions.
As described in studies such as [7] and [24], the method follows these steps:
• A common experimental setup is the towing test, where the vehicle is moved through a
fluid (e.g., water) at different velocities (u)
• The drag force (FD) is measured for each velocity, and the relationship
F = CdρAu + CqρAu2
(4.5)
is used to derive the drag coefficients, A is the projected area along the axis of the force
• Solving the resulting system of equations provides the values for Cd (linear drag
coefficient) and Cq (quadratic drag coefficient)
2. CFD Simulations (Computational Fluid Dynamics): Computational methods can
accurately compute drag coefficients by modeling the fluid flow around the vehicle. The
process involves:
• Using a 3D CAD model of the vehicle which must include either fully or in a simplified
manner the material properties of its components
• Simulating the flow field at different speeds to capture both linear (Cd) and quadratic
(Cq) drag behaviors
• These simulations provide a detailed understanding of the forces acting on the vehicle
and are particularly useful for complex geometries [7]
3. Literature Analysis: A simpler solution is to use reference values from existing literature,
which provide typical ranges for drag coefficients based on the geometry of the vehicle. These
ranges, extensively documented in studies such as [25] and [26], are as follows:
• For streamlined vehicles (e.g., ellipsoids, torpedoes): Cd ∼0.1 −0.3, Cq ∼0.5 −1.5
• For more complex geometries (e.g., ROVs): Cd ∼0.2 −0.6, Cq ∼1.0 −2.0
These values provide a good starting point for analysis when more precise data is unavailable.
4. Empirical Estimation: Empirical formulas based on geometric shapes can also be employed
to estimate drag coefficients. As highlighted in studies like [17], one common approach is:
Cd = 2Fd
ρAu2
(4.6)
where Fd is the measured or estimated drag force. This method is straightforward but requires
assumptions about the force distribution and flow behavior.
Each of these methods has its own advantages and limitations, and the choice of method depends
on the specific application, available resources, and desired level of accuracy.
CFD simulations were attempted; however, due to the limited precision of the CAD model in terms
of material properties and the high computational complexity, only simplified CFD (sCFD) tests
were performed. Towing tests faced hardware issues, which resulted in limited success. However,
the combined approach, particularly literature data and experimental measurements, allowed a
good estimation of the drag coefficients.

Chapter 5
AUV Control
The term rendezvous refers to the action of two entities meeting at a specific time and location, while
docking refers to the physical connection between two bodies. This chapter focuses on strategies to
achieve a successful rendezvous and docking between two underwater robots, serving as a foundation
for potential docking procedures.
To achieve a rendezvous, two critical aspects must be addressed: target recognition and control
strategy definition. Target recognition is performed using a vision-based system, briefly explained
in Appendix B. This system enables one robot to identify and track the other, providing the
necessary positional and orientation data for control.
A wide range of control strategies has been considered in the literature to address the challenges
of underwater rendezvous, including Proportional-Integral-Derivative (PID) controllers, Linear
Quadratic Regulators (LQR), Sliding Mode Control (SMC), and Model Predictive Control (MPC).
Each of these approaches offers unique advantages and limitations, depending on the specific
application requirements and system constraints.
For the PID controllers, articles such as [32] and [18] were reviewed. Wu [32] implemented a 6-DOF
PID control for the BlueROV2 Heavy, where six uncoupled single-DOF controllers were used, each
designed for a specific degree of freedom. Instead, Aili and Ekelund [18] focused on PID-based
controllers for attitude, angular velocity, and depth control.
In evaluating LQR, studies such as [5], [6], and [33] were analyzed. Hansen and Sandoy demonstrated
the linearization of ROV dynamics through specific simplifications to effectively apply LQR, while
Tchilian’s work focused on gliders, presenting a slightly different approach that could inform the
application for AUVs.
For SMC, articles including [34], [35], [36], and [37] provided key insights. This approach was
particularly considered due to the uncertainty in estimating the dynamic parameters of the ROV
model, a challenge that SMC is well-suited to address. Ren and Hu [37] explored a sliding mode
controller with a saturation function to mitigate chattering, achieving robust motion control for
nonlinear ROV dynamics. Instead, Gonzalez [35] and Ren [37] proposed model-free strategies,
which are particularly advantageous in scenarios where the dynamic parameters are either partially
known or highly uncertain.
Finally, MPC was analyzed extensively through the works of [38], [39], [40], [41], [42], [43], and [44].
Among these, Vivekanandan [38] provided a significant reference for implementation, proposing an
MPC strategy for a BlueROV2 Heavy equipped with a docking system based on ArUco markers.
This approach was adapted in this study for a BlueROV1 using a pinhole camera and a LED
pattern for target identification, aligning the methodology with the specific requirements of the
current project.
For the theoretical foundation of MPC, Mikkel Nielsen [39] provided valuable insights, particularly
in understanding the core principles and methodologies underlying its implementation. Other
35
==PAGE:52==
5. AUV Control
works were consulted to address specific challenges: for example, Hu [40] for implementation in
Gazebo-ROS, Jia [41] for exploring the potential of distributed MPC in cooperative systems, and
Saback [44] for insights into ACADOS and its role in optimizing MPC implementations. Some
articles were also reviewed to gain a broader understanding of MPC’s flexibility and to explore

In this study, all four strategies were initially considered to evaluate their potential effectiveness in
addressing the dynamics of underwater robots. However, certain limitations led to the exclusion
of LQR and SMC from further analysis. While LQR is computationally efficient and provides
optimal control for linearized systems, it is not well-suited to the inherently non-linear dynamics of
the ROV, making it less effective for this application. Similarly, SMC offers strong robustness to
uncertainties and disturbances but was excluded due to challenges such as high-frequency control
inputs (chattering), which can impose stress on the thrusters, and its complexity in practical
implementation.
Given these considerations, the focus of this chapter is on PID and MPC strategies. PID
controllers are valued for their simplicity and ease of implementation, making them a baseline for
comparison in many underwater applications. On the other hand, MPC is well-suited for managing
non-linear dynamics and operational constraints, providing a promising solution for uncertain and
dynamic underwater environments.
5.1
Modeling and Control Considerations
In the study of the AUV model, parameters were calculated for a full six degrees-of-freedom (6
DOF) representation because the vehicle exhibits coupled motions that require a comprehensive
approach for both simulation implementation and improved performance.
However, as discussed in Chapter 3, a simplification to a four degrees-of-freedom (4 DOF) model
was introduced by excluding roll and pitch, as primarily justified in Assumption 1, along with
the other assumptions.
This choice is motivated by the fact that during docking or rendezvous procedures, nonzero roll and
pitch angles provide no significant advantage. Instead, maintaining these angles at zero ensures
stability and predictability, focusing control efforts on surge, sway, heave, and yaw.
Both PID and MPC controllers were implemented in the 6 DOF framework to account for coupled
motions. While simulations in Gazebo-ROS operate in full 6 DOF, in real-world tests, the PID
also maintains this framework, whereas MPC presents challenges in real-time execution. Actively
stabilizing roll and pitch aligns with the 4 DOF assumption, potentially reducing computational
complexity, particularly for MPC, by decreasing optimization variables and improving execution
speed.
Ultimately, while the 6 DOF formulation remains crucial for simulation and theoretical validation,
structured simplification in real-time applications enhances feasibility, balancing model accuracy
with computational efficiency.
The following sections provide a detailed discussion of the implemented control strategies, including
the specific methodologies, challenges encountered, and the approaches adopted to ensure robust
and efficient performance in both simulation and real-world applications.

PID implementation
Based on [32], a PID controller was implemented for each degree of freedom (DoF) of the ROV:
surge, sway, heave, roll, pitch, and yaw. This approach assumes six independent single-DoF
controllers, where each operates separately to regulate motion in its respective direction.
In reality, certain DoFs exhibit coupled dynamics; however, these are mainly pitch and roll, which
are not of primary interest in the docking task. The introduction of the docking module influences
pitch, while the configuration of the thrusters n°6 affects roll Figure 2.3. Nevertheless, by actively
controlling these coupled modes and driving roll and pitch to zero, their influence on the system
can be minimized. As a result, the assumption of six independent PID controllers remains valid for
both the docking and rendezvous processes, which primarily rely on surge, sway, heave, and yaw.
To guide the ROV towards its target, a yaw alignment strategy inspired by the classical Line-
of-Sight (LOS) approach was implemented. Unlike traditional LOS, where the objective is to
follow a trajectory, here the goal is to ensure frontal alignment with the target minimizing sway

5.4
Model Predictive Control
Model predictive control (MPC) is a control scheme where a model is used for predicting the future
behavior of the system over finite time window, the horizon N. At each sampling instant, the control
inputs are calculated by solving an optimization problem that minimizes a predefined cost function
while respecting constraints on the inputs and outputs of the system. Once the optimal control
inputs are computed, only the first control action is implemented. The process is then repeated at
the next time step with updated system measurements or state estimations.
Figure 5.2. Illustration of the Model Predictive Control (MPC) scheme. Image source:Wikipedia.
In Figure 5.2, the fundamental components of the Model Predictive Control (MPC) framework are
visually illustrated. Below is a detailed explanation of the various elements depicted:
• Reference Trajectory (r(t)): This represents the desired behavior or target output of the
system. It serves as the benchmark that the controller aims to track. The trajectory may be
defined as a step function or any continuous signal, depending on the system requirements.
• Predicted Output (ypred): Using a mathematical model of the system, the MPC algorithm
predicts future outputs over a finite time horizon, N. These predictions dynamically adjust
based on the current system state and anticipated control inputs. The predicted output is
initialized using the Measured Output (shown as the current state of the system), which
provides real-time feedback about the system’s actual performance. The goal is to minimize
the tracking error, the difference between the predicted output and the reference trajectory.
• Control Inputs (u(t)): This stair-step signal represents the sequence of control actions
computed by the MPC optimization process. At each time step, the algorithm solves an
optimization problem to find the control inputs that minimize the cost function while satisfying
constraints. Only the first control action of the optimized sequence is applied before repeating
the process at the next iteration.
• Constraints (not explicitly shown, but considered in the predicted control inputs):
Constraints are integral to MPC and are incorporated into the optimization problem. These
include physical limitations on control inputs (e.g., actuator saturation) or safety thresholds
for system outputs (e.g., temperature or pressure limits). While not explicitly represented in
the figure, they are crucial in determining the predicted control inputs (u(t)), ensuring that
all computed actions respect these limitations.
The iterative nature of MPC, also known as the receding horizon principle, allows the controller
to continuously update its predictions and control actions as new measurements or estimations
become available. This capability enables MPC to adapt effectively to changing conditions and
disturbances in the system.

Problem Formulation and Components
The decision variables in the optimization problem are:
• xk ∈R12: The state vector at step k, which represents the dynamic state of the ROV. This
includes position, velocity, orientation, and angular velocities.
• uk ∈R6: The control input at step k, which represents the forces and torques applied by
the ROV’s thrusters.
The cost function is penalized using the following weighting matrices:
• Q ∈R12×12: A diagonal positive-semidefinite matrix that penalizes deviations of the state
vector from the desired trajectory. It ensures that the state of the vehicle is kept close to the
target. Higher values in Q increase the priority of keeping certain state variables close to
their reference values.
• R ∈R6×6: A diagonal positive-definite matrix that penalizes control effort, ensuring that
the applied forces and torques are smooth and not excessive. A higher value of R results in
smoother but potentially slower control responses.
• P ∈R12×12: A terminal cost matrix that enforces convergence to the desired final state
at the end of the horizon. It ensures that the system reaches the goal state efficiently by
giving additional importance to final state errors. Typically, P is chosen as the solution of
the discrete-time algebraic Riccati equation (DARE) in Linear Quadratic Regulation (LQR)
settings or is set equal to Q for simpler tuning.

The constraints are:
• State bounds: The state xk is constrained within physical limits, i.e., xmin ≤xk ≤xmax, for
all k = 0, . . . , N. In the simulation environment (e.g., ROS-Gazebo), these constraints are not
strictly necessary, as the vehicle can theoretically reach any state without physical restrictions.
However, in real-world applications, such constraints become crucial due to environmental
limitations. For example pool operation, tether constraints, or safety constraints in open
water scenarios.
• Control bounds: The thruster forces uk are constrained by hardware limitations, i.e.,
umin ≤uk ≤umax, for all k = 0, . . . , N −1. These bounds depend on the available thrust
capacity along each axis, which has been determined based on the thruster configuration
and the maximum power output of each thruster. The force distribution accounts for the
placement and orientation of the thrusters, ensuring that the control inputs respect the
physical propulsion capabilities of the BlueROV1.
• Yaw alignment constraint: The yaw angle ψk is constrained within a small tolerance ϵ
to ensure that the ROV maintains proper alignment with the target, reducing unnecessary
lateral movement and improving docking precision.
• Initial state: The initial state of the system x0 is fixed to the current position of the ROV. In
a closed-loop configuration, if the model is perfectly accurate and no disturbances are present,
the actual state should ideally coincide with the predicted state. However, in real-world
conditions, modeling inaccuracies and external disturbances can introduce deviations between
the predicted and actual state.
Numerical Implementation
To solve the nonlinear OCP in real-time, the problem is transcribed into a Nonlinear
Programming (NLP) problem, which is then solved using optimization techniques.
The
implementation utilizes the Acados framework, which provides a high-performance solver for
optimal control problems.
The NLP is solved using the following optimization methods:
• Sequential Quadratic Programming - Real-Time Iteration (SQP-RTI): This
approach approximates the nonlinear problem as a sequence of Quadratic Programming
(QP) subproblems, which are solved iteratively to find an optimal solution. The advantage
of SQP-RTI is its ability to provide fast, real-time feasible solutions.
• HPIPM (High-Performance Interior-Point Method): The QP subproblems are solved
using HPIPM, an efficient interior-point solver designed for real-time applications. HPIPM
is optimized for embedded systems and supports structured QP formulations that leverage
sparsity for computational efficiency.
The prediction horizon N and time step ∆t are critical parameters influencing the balance between
computational cost and control performance. A longer horizon improves foresight and stability
but increases computational complexity, which can be challenging for real-time applications. By
leveraging Acados’ efficient QP solvers and optimized code generation, the framework achieves a
balance between performance and feasibility for onboard execution.
As shown in [44], the Acados framework is a powerful alternative for solving NMPC problems
due to its optimized numerical solvers and real-time execution capabilities. Unlike ACADO, Acados
leverages high-performance interior-point methods and efficient integration schemes, ensuring better
scalability and computational speed. Its ability to generate optimized C code makes it particularly
suited for embedded applications, where execution time is critical.

Simulation Results
This chapter presents the experimental results obtained from simulations conducted in a ROS-

A variety of initial conditions were tested to verify the proper functioning of the controllers under

adjustments, and convergence times. For the Rendezvous scenario, the challenge was to achieve a
precise dynamic approach between two moving underwater vehicles, with particular attention to
trajectory tracking, relative positioning, and synchronization accuracy.
Although both controllers were evaluated under these scenarios, it is important to note that this
study is not intended as a direct head-to-head comparison. Instead, the MPC is recognized for its
optimal performance in generating control actions, whereas the PID controller provides a more
conventional approach.
All simulations were performed in a controlled virtual environment that closely replicates real-world
hydrodynamic conditions, as shown previously in Figure 4.6. The results are analyzed in terms of

The tuning of the Model Predictive Control (MPC) parameters was performed through a series
of experiments using different initial conditions and trajectories. The goal was to analyze the
controller’s response to various motion dynamics and adjust the parameters accordingly for optimal
performance.
MPC tuning typically involves adjusting:
• The prediction horizon (N) to balance accuracy and computational load.
• The time step (dt) to ensure stable and responsive behavior.
• The weight matrices (Q, R, and P) to penalize errors and control effort.
• The state bounds (xmin, xmax) and the control bounds (umin, umax).
The tuning process was carried out in three stages:
1. Python Simulation: Initial tuning was performed in a simplified environment to test the
controller’s fundamental behavior, adjusting Q and R to achieve smooth responses without
excessive control effort.
2. Gazebo Simulation: Further tuning was required to accommodate the simulated system’s
dynamics, introducing realistic disturbances and ensuring feasibility under simulated actuation
constraints.
3. Real-World Experiments: Additional adjustments were necessary to account for hardware
limitations and environmental disturbances.

6.7
Final Considerations
The Gazebo simulations for both the docking and rendezvous tasks confirm that PID and
MPC strategies each provide distinct advantages. On one hand, PID control offers a simpler
implementation and has shown robust performance in achieving stable docking and rendezvous
under nominal conditions.
On the other hand, the more advanced MPC approach naturally
accommodates constraints and dynamic variations, resulting in improved accuracy and faster
convergence when dealing with more complex scenarios.
Both methods successfully complete the docking and rendezvous maneuvers, highlighting their
viability for ROV operations. Nevertheless, the choice between PID and MPC often depends on
the specific requirements and constraints of the mission. For simpler or well-defined tasks, PID
remains an attractive solution due to its ease of tuning and implementation. For applications
where predictive handling of constraints or significant environmental uncertainties are critical, MPC
provides a more refined and flexible control framework.

Chapter 7
Experimental Results
The goal of the experimental tests was to evaluate the performance of both control strategies,
PID-based control and Model Predictive Control MPC, in real-world underwater conditions for
both docking and rendezvous operations. While simulations provide valuable insights into the
expected system behavior, physical tests introduce additional challenges, such as hardware issues
(thrusters malfunctioning or broken cables), sensor inaccuracies, and the need for proper buoyancy
and balance adjustments to ensure stable operation in water.
For the PID-based docking strategy, the system relied on three primary sources of data:
• IMU measurements, used to stabilize roll and pitch;
• Depth sensor, providing accurate depth estimation for heave controll;
• Visual-based system, used to determine the relative position of the docking target.
Implementing this strategy required significant effort due to the complexity of preparing the ROV
for each experiment. Each battery replacement slightly altered the system’s center of mass, making
it necessary to adjust the buoyancy and balance by repositioning weights and flotation to ensure
stability. Additionally, since the internal electronics were moved during battery swaps, the IMU
had to be recalibrated to correct orientation errors before each test. These preparatory steps were
essential to maintain consistent and reliable experimental conditions.
The tuning process for the PID controllers followed a structured approach. Initially, controllers for
roll and pitch were tuned independently, ensuring that the vehicle maintained a stable orientation.
Once stability was achieved, additional controllers were introduced for the remaining degrees of
freedom. The initial plan was to leverage the tuning parameters obtained from simulation; however,
due to inevitable physical discrepancies between the simulated and real environments, these values
led to instability in real-world conditions. As a result, a manual tuning approach was adopted,
involving data collection, parameter adjustments, and iterative testing.
Despite these challenges, the PID-based docking strategy was successfully implemented and
tested, demonstrating the feasibility of visual-based docking using a closed-loop control system.
However, the rendezvous strategy could not be fully tested due to time constraints and experimental
limitations.
Instead implementing the MPC strategy in real-world conditions required significant adaptations
to transition from the simulation environment to physical experiments. While simulations provided
an idealized representation of the system dynamics, real-world differences in buoyancy, balance, and
battery positioning affected the vehicle’s behavior, necessitating additional tuning and adjustments.
One of the most critical issues encountered during real-world MPC testing was the computational
delay when integrating the visual-based feedback loop in closed-loop control. In an ideal scenario,
the MPC would compute the optimal control inputs in real-time and apply them continuously.
However, in practice, the computational delay introduced an unavoidable discontinuity in actuation.
59
==PAGE:76==
7. Experimental Results
The issue can be described as follows:
• At step 0, the MPC receives the initial state and target position, computes the optimal control
input u∗, and returns the result in approximately 0.04s.
• The computed control input u∗is applied for a duration of 0.2s.
• At the end of this actuation period, a new state estimate is obtained, and the MPC must
solve a new optimization problem with the updated initial condition.
• The recomputation takes an average of 0.04s, during which no thruster commands are applied,
causing the vehicle to momentarily enter a idle state.
This interruption in control application resulted in a segmented and inconsistent behavior, preventing
the closed-loop MPC from maintaining smooth actuation.

Conclusion and Future Work
8.1
Conclusion
This work has focused on developing and validating a docking and rendezvous system for the
BlueROV1, integrating hardware corrections, dynamic modeling, simulation, and control strategies
to improve autonomy in underwater operations.
The docking module was already available but required modifications to the hardware and wiring
to ensure proper functionality. These adjustments allowed the system to be reliably integrated
with the ROV, facilitating controlled docking maneuvers. Similarly, the visual-based system, which
identifies a predefined LED pattern for relative positioning, was already implemented. However, its
effectiveness was further analyzed in the context of docking and rendezvous.
An important part of this work was the study and estimation of the dynamic parameters of the
BlueROV1, following methodologies outlined by Sandøy and Hansen. Parameters such as mass,
added mass, damping, and buoyancy were carefully identified to build an accurate representation
of the ROV’s behavior. This step was essential for both simulation and control design, ensuring
that the controllers were developed based on realistic dynamics.
A Gazebo-based simulation environment was then created, incorporating the estimated model
parameters. This simulator provided a high-fidelity testing ground for evaluating control strategies
before real-world deployment, reducing risks and refining system performance. The ability to
simulate various environmental conditions and disturbances was crucial for testing the robustness
of the controllers.
In terms of control, both PID and Model Predictive Control (MPC) were implemented for
docking and rendezvous. MPC demonstrated superior performance in trajectory optimization and
disturbance rejection but was computationally demanding, making real-time execution on the ROV
impractical. On the other hand, PID controllers, despite their limitations, were successfully tested
in real-world conditions, proving to be a feasible and reliable solution for practical applications.
This work has provided a comprehensive and structured approach to underwater docking and
rendezvous, integrating hardware adjustments, system identification, simulation, and control design.
The results highlight the trade-offs between computational complexity and real-time feasibility,
emphasizing the importance of model accuracy and simulation in control development.

8. Conclusion and Future Work
8.2
Future Work
Several aspects can be improved and further explored to enhance the performance and applicability
of the proposed methods:
• Improved System Identification for BlueROV1
More advanced system identification techniques should be explored to refine the estimation of
hydrodynamic parameters. One possible approach is the use of Dynamic Neural Network-
Based Adaptive Identification, as proposed in [48], which combines adaptive neural
controllers with nonparametric identification of disturbances and uncertainties.
• Controller Optimization and Real-Time Implementation Challenges
The control parameters should be further optimized, considering not only dynamic
performance but also the physical and electronic limitations of the onboard hardware.
A crucial aspect of this optimization is ensuring real-time feasibility on the embedded
system. Various approaches have been proposed to mitigate these issues [49], including:
– Explicit MPC: Precomputes the control law offline to reduce online computational.
– Distributed MPC (DMPC): Decouples the optimization problem into smaller
subproblems for parallel computation.
– Accelerated Optimization Algorithms: Such as Ohtsuka’s continuation/generalized
minimal residual algorithm, which speeds up convergence.
– Warm-Starting Techniques: Reuses previous solutions to reduce computational effort
in iterative optimization.
Despite these advancements, real-time implementation of MPC for nonlinear AMVs remains
an open challenge. Further research is needed to explore efficient numerical solvers,
embedded computing strategies, and hardware-aware optimizations to enhance
onboard computational performance while maintaining control accuracy.
• Enhancing the Vision-Based System
The current LED-based localization system has significant limitations, particularly in real-
time experiments where processing speed and robustness are critical. Future improvements
should focus on faster image processing algorithms, sensor fusion techniques, and
deep learning-based feature extraction to improve reliability and speed.
• Exploring the Potential of Rendezvous and Docking
Beyond simple docking, potential applications of rendezvous operations in multi-robot
cooperative tasks should be explored. Future research could investigate how connected
robots could collaborate dynamically, for example, by physically coupling for specific missions
that require a larger or more stable structure.

On the right side of the board, the first two connection points are dedicated to the 5V supply, the
next two are for the 12V supply, and the final two (located near the large transistor) are designated
for powering the magnet.
Regarding the circuit itself, there are three pairs of 2-pin headers that supply power to the three
distinct LED groups. Additionally, there is a 2-pin header adjacent to these, which provides power
to the Raspberry Pi.
The four other pin headers are connected to the Raspberry Pi to control the three LED groups and
the magnet. This control is achieved through a circuit consisting of resistors and BS170 transistors
for the LED groups, while the magnet is controlled using a PSMN3R4-30PL transistor. This design
ensures precise and reliable operation of the LEDs and the magnetic latching mechanism.

This system, previously implemented by Bartosz Paryk Gaik [50], improved by Shuo Wen [51] and

The pinhole camera model, shows in Fig-

In the pinhole model, a point in 3D space with coordinates (X, Y, Z) is projected onto the 2D
image plane at coordinates (u, v) using the following relationships:
x = f · X
Z
y = f · Y
Z
where f is the focal length of the camera, Z is the depth (distance from the camera to the object),
and (x, y) are the pixel coordinates on the image plane.
The reverse operation can reconstruct the 3D coordinates if the distance Z is known. For the
vision-based docking system, the distance Z is estimated using the size of known features, such
as the height of an LED marker pattern in the image. The distance D to the docking station is
calculated as:
D = H · f
h
where:
• D: Distance to the docking station.
• H: Real-world height of the LED marker pattern.
• f: Focal length of the camera.
• h: Apparent height of the LED marker pattern in the image (in pixels).
The pinhole camera model assumes an ideal projection system without distortion. This makes
it suitable for the system’s LED-based distance estimation, provided the camera is calibrated to
remove lens distortions.

Image Processing
The system employs OpenCV, a widely-used computer vision library, to process images captured
by the Docking camera. The key steps in the image processing pipeline include:
• HSV Conversion: RGB images are converted to HSV format
• Binarization: Images are thresholded to isolate the brightest regions, which correspond to
the LEDs
• Noise Reduction: Gaussian blurring and morphological operations, such as erosion and
dilation, are applied to reduce noise and enhance marker detection.
• Blob Detection: The blob detection algorithm identifies the LEDs as connected regions of
pixels and extracts their centroid coordinates.
Detection
The initial search in the image should output regions in the image that may indicate the presence
of the station and its approximate location. To achieve this, the algorithm incorporates a custom
iterative thresholding method and conditional checks, which effectively filter and reject regions not
associated with the fiducial markers.
The primary purpose of the conditional checks is to determine whether a binarized image contains
sufficient information for the Pose Estimation algorithm. A secondary objective is to minimize
computational time by rejecting irrelevant regions early in the process.
Fiducial Identification
The identification and numbering of fiducial markers are critical for the docking station recognition
algorithm. The pose estimation process requires the precise location of the pattern LEDs in the
image and information about which diode each centroid corresponds to. Since the AUV is stabilized
along the roll axis, this assumption enables the vision system to identify the markers based on their
relative positions in the image.
Tracking
To improve efficiency and reliability, a fiducial marker tracking method has been implemented
to maintain the identity of multiple objects across frames. Unpredictable disturbances in the
recognition process may arise due to varying real-time conditions, such as: inter-object occlusions,
object occlusions by background obstacles, splitting or merging of objects due to image noise or
diffusion.

The tracking method ensures that each fiducial marker is consistently identified in successive image
frames, improving the accuracy of measurements. Two tracking methods were implemented:
• Mean Shift-Based Approach: In this method, the search window remains constant, with
a size of 50 pixels in both width and height. However, when the target moves too quickly, the
tracking often converges on the wrong object or loses the target altogether.
• Kalman Filter-Based Approach: This method establishes a motion model for the target
object, using its current state to predict its position in the next frame. The Kalman Filter
operates recursively, meaning that only the estimated state from the previous step and the
current measurement are required to compute the new state. This approach enhances tracking
accuracy under dynamic conditions.

To interact with the BlueROV1 system and its docking module, a connection must be established.

roslaunch bluerov bluerov_r1.launch

• Thruster Testing: Verify that each motor responds correctly to commands and functions

• Battery Monitoring: Monitor battery voltage and system status to prevent unexpected
shutdowns.

proper functionality and avoid system failures during a mission.

verification) to streamline setup and minimize errors before an underwater mission.

C.4
Thruster Control Process
Controlling the thrusters for the BlueROV1 through MAVROS requires several steps to ensure
proper operation. The thrusters must be armed, the correct mode set, and appropriate commands
sent using RC override messages.
1. Arming the Thrusters: Before sending any commands, the vehicle must be armed using
the MAVROS service /mavros/cmd/arming.
2. Setting
the
Manual
Mode:
The flight mode must be set to MANUAL using
/mavros/set_mode. This mode ensures that the vehicle operates under a predefined mapping
at the firmware level.
3. Sending RC Override Messages: Commands are sent via /mavros/rc/override, which
directly controls the thrusters.
4. Disarming the Thrusters: After execution, the thrusters should be disarmed for safety.
In MANUAL mode, MAVROS maps the RC commands to control axes:
[Pitch, Roll, Throttle, Yaw, Forward, Lateral]
This mapping allows intuitive vehicle control, particularly via joystick. However, in some cases,
direct thruster control is required, especially for non-standard configurations or different Thruster
Configuration Matrices (TCM). Instead of relying on the default firmware mapping, inputs must
be sent in terms of individual thrusters:
[T1, T2, T3, T4, T5, T6]
Since the firmware applies an internal thruster configuration matrix (TC), sending commands in
direct thruster format without modification leads to an unintended transformation. To correct this,
the input must be pre-multiplied by the inverse of the transposed thruster configuration matrix:
