Summary
In the automotive domain, tasks exist in a distributed system consisting of
CPU’s and network links between them. These tasks sense, process, and actu-
ates to enable ever increasing functions. The safety critical nature means the
tasks and their communication should be deterministic. By scheduling the tasks
and their communication, this determinism can be achieved.
The problem is seperated into the communication and task scheduling, and
two seperate heuristics are used to ﬁnd solutions fast. These heuristics have
parameters which speciﬁes how they schedule the tasks and the communication.
Two metaheuristics are implemented to guide the parameters needed. The ﬁrst
is Simulated Annealing and the second Genetic Algorithm.
The ﬁrst for its
simplicity, and the second to see if it can improve upon results with global
search. They have been tuned using paramILS for a fairer comparison than
what manual tuning can get.
They have been evaluated on synthetic test cases with diﬀerent topologies, and
a mix of periods, several times. The two metaheuristics are compared and it
is found that the Genetic Algorithm is better at ﬁnding feasible solutions. For
ﬁnding solutions as fast as possible, Simulated Annealing is very good in small
test cases.

Chapter 1
Introduction
Autonomous or semi-autonomous vehicles like cars and planes are fundamentally
safety critical systems. Failures in operation often carry large penalties, and
should be avoided if possible. Each element is tested thoroughly and sometimes
even proven to be correct using formal veriﬁcation techniques. Some of these
diﬀerent functions are shown in 1.1, needs to share the same resources, such as
computing, sensor data, and networking.
Adding all software modules together is easy if each module has its own resources
like CPU or sensor and can be put on its own board, but as the number of
features increase, this becomes expensive. To save on cost and decrease the
total number of components, multiple features can coexist on the same compute
hardware, and communicate over the same network. Fault tolerant networking
like the Controller Area Network (CAN)-bus[2] is used to ensure that messages
are sent deterministically.
Networks such as the CAN-bus, ﬂexray and others have limited bandwidth
available, so as the need for more bandwidth, such as video, or lidar information
is needed, these solutions fall short. Solutions such as deterministic ethernet,
and in the case of this report, Time Sensitive Networking (TSN), can increase
the bandwidth in a more scalable way. TSN is a set of additional standards to
ethernet that allows messages to be sent in a way that guarantees the message
latency. This is critical, because too high latency will result in system failure.

It is however diﬀerent in the way of ethernet, since it is not plug-and-play. TSN

Problem formulation
The focus of this project is then the combination of task and network conﬁgu-
ration. It is important to ﬁnd a static schedule that ensures each task will be
able to perform its computation, and it is likewise important that the commu-
nication between tasks, sensors, and actuators are done with strong guarantees
for safety. Because of the need for strong safety guarantees, the solution should
be static, and it should be correct by design.

Chapter 2
Background
An ADAS platform is a collection of computation units, like CPU’s, GPU’s,
DSP’s, and others.
A task often runs on a speciﬁc computation unit either
because of speciﬁc requirements, or because it is convenient.
As shown on
Figure 2.1, some of the computation units are safety related, some are perfor-
mance related, and some a general purpose. Safety critical tasks will often run
on safety micro-controllers(µC) running in lockstep to increase safety. On the
other hand, certain functions like downloading the latest update, can be done
on normal hardware that lacks safety features.
Diﬀerent operating systems supporting diﬀerent tasks can run on these compu-
tation units, and depending on the safety requirements, diﬀerent task scheduling
as well. For the safety critical tasks, multiple diﬀerent options exists, but static
table-driven scheduling is chosen. Static table-driven scheduling is good because
adding tasks does not change the behaviour of other tasks if their schedule does
not change.
Static table-driven scheduling also allows for tasks to preempt each other in an
oﬄine sense, that is, a long non-critical task can start execution, and then be
preempted by a short critical task. Task preemption allows for more eﬃcient
use of the computation units and when used together with static table-driven
scheduling does not change the safety aspects of the schedule.

2.1
Communication
The need for TSN arises from the demand for more bandwidth between nodes,
TSN builds a time-aware system on top of ethernet. Ethernet connects end
systems using cables and switches, takes messages from end systems, splits them
into frames that can be easily sent on the network, and forwards them if needed.
A message from an end system may be of arbitrary length, but the maximum
length of a frame is 1500 Bytes. Additionally, a total of 42 bytes are needed for
preambles, headers, and inter-frame spacing. This allows the end systems and
switches to reliably read, identify, and forward frames as needed. The size of
each physical frame is then between 43-1542 Bytes. If a message is above 1500
Bytes, it gets split into as many frames as necessary.
A speciﬁc feature of ethernet 802.Q used is the ability to assign priorities to
frames. This priority can be between 1 an 8, and can be used to make sure
higher priority frames are forwarded urgently. The 8 priorities map directly to
8 queues for each egress port. As frames enter a switch, they are put into the
queue corresponding to their priority. If two frames are ready to be transmitted
from the switch, the higher priority frame gets right of way. One issue that
exists with this system is starvation, where lower priority traﬃc is starved from
bandwidth. Therefore, care must be taken to not overload the system.
Time Sensitive Networking
To avoid starvation of lower priority traﬃc, and have an ability to give diﬀerent
levels of guarantees, three diﬀerent ﬂow types are speciﬁed: Time-Triggered
(TT), Audio-Video bridging (AVB), and Best Eﬀort (BE). TT is the highest
priority traﬃc, is often small in size, and has a worst case message length. As
the name suggests, it is sent at speciﬁc times so that the switches on the network
can close oﬀother types of traﬃc until the TT traﬃc has passed. AVB is the
second type of traﬃc, and is used for data that ﬂows continuously. It is possible
to ﬁnd worst case latencies of AVB ﬂows, [15] uses network calculus to assess
the worst case delay when factoring in other ﬂows. This means AVB ﬂows can
be used for real time communication. BE is the last type of traﬃc, and as the
name suggests, it is sent without any speciﬁc guarantee of latency.
In order to ensure that the BE traﬃc is not starved, the bandwidth used by TT
and AVB should be less than the amount of bandwidth needed for BE traﬃc.
TSN consists of a collection of standards, which together form the basis of real
time communication. The standards used in this project is given by Table 2.1.

more could help. Another issue could be if the TT traﬃc periodically blocks
other traﬃc, if for example all TT traﬃc was scheduled at the start of their
periods, then some AVB frames might needlessly be delayed.

Chapter 3
System Model
This chapter presents the system model used for the thesis.
It describes a
simpliﬁed version of the real world so that the problem can be solved in an
eﬃcient manner. Two models are created: First, an architecture model that
describes the physical properties of the platform and its network. Second, an
application model that speciﬁes how frames and tasks behave on the system.
The main assumption is that the platform is using TSN for networking and
static table-driven scheduling for the computation. Using static table-driven
scheduling instead of ﬁxed priority or dynamic scheduling allows isolation of
tasks, such that adding a task does not alter the behaviour of previous tasks.
Together with using static TSN schedules, all computation and networking is
static and will always execute the same way, which is highly desirable in safety
critical system.

with the vertices V are the communicating end systems and switches. The edges
E are the links between them. A directed edge is represented by two vertices

the system, where higher complexity in general creates a longer context switch.

A high CPU-macrotick will reduce the context switching, but will not allow the
system to eﬃciently switch between priorities and use idle time. In this project,
it is assumed that the maximum time context switches could use is included in
the Worst Case Execution Time (WCET).
CPU-macroticks can be handled in many diﬀerent ways, but in this project it
is assumed that the CPU-macrotick speciﬁes when context switches are allowed
to occur. This means that tasks can start at any time, but only be preempted
at times divisible in the CPU-macrotick. This is easy to implement and does
not incur more context switches than necessary.

The algorithm relies on feasible regions to ﬁnd out where the frames can be

Since frames can have dif-

scheduled. A feasible region is deﬁned as a set of intervals {[a, b[ , [c, d[} which

Blocking happens at most once for every ﬂow scheduled, while searching can
happen many more times, depending on how hard it is to schedule a frame.
Because of this, a data structure minimizing the time used in searching is
used.

5.3
Computation heuristics
The sender and receiver tasks have already been scheduled, so these are input
into the EDF as tasks that cannot be moved. Other tasks are then preempted
by them.
For the task scheduling, a TaskOffset and TaskDeadlineAdjustment set is
used. The heuristic used is called Earliest Deadline First (EDF), which schedules
tasks according to which task has the earliest deadline. EDF works as follows:
If a task is ready and no other task is currently scheduled, schedule that task.
If a task is already scheduled, preempt if the new task has a smaller adjusted
deadline than the current task. The eﬀective deadline for the EDF algorithm is
given by the task deadline ⌧D and the TaskDeadlineAdjustment.
A useful feature of the EDF is that for a given TaskOffset and TaskDeadlineAdjustment
which is not feasible, a simple repair function exists. The repair function goes
==PAGE:43==

33
as follows: For all unfeasible tasks, set their task oﬀset to 0. With a task being
unfeasible if their deadline is missed, or if not enough all task instances where
scheduled. This repair function does not guarantee a feasible solution, because
deadline adjustments of large tasks can starve smaller tasks, however it increases
the likelihood of feasible solutions.
EDF simulation (TaskOffset, TaskDeadlineAdjustment) Takes a
speciﬁer and schedules CPU tasks on a single core. The hyperperiod
of the core is the lowest common multiplier of the task periods on the
CPU. The queue is a data structure which allows for time/item pairs,
allows for retrieval of the item with the smallest time.
The major parts of this algorithm is as follows:
1. Enqueue all tasks on CPU using their TaskOffset.
2. Remove the earliest task from the queue
3. If the starting time of the task is greater than its hyperperiod,
terminate.
4. Insert it again, but with its period added to its oﬀset.
5. If the adjusted deadline of the new task is lesser than the currently
scheduled task, replace it.
6. Insert the replaced task into the queue such that it is ready when
the other task is ﬁnished.
7. Go to 2.

Simulated annealing
SA is a simple local search algorithm that mimics metal annealing. Starting at
a high temperature, the algorithm is very likely to choose new solutions, good
or bad. The algorithm then cools and progressively is less likely to choose new
bad solutions.

Mutations
There are many diﬀerent ways to implement mutations. There are two main
parts of the representation, orderings and assignments. These have diﬀerent
properties and are mutated in two diﬀerent ways.
An ordering has the property that each element must be unique, so taking a
random element and changing it to another will not produce a valid ordering.
Instead, many diﬀerent approaches exist, for this, we have used the swap oper-
ator. The swap operator chooses two diﬀerent indices in the order, and swaps
them.
The assignment representation represents the task oﬀsets, frame oﬀsets, and
task deadline adjustments. Here, it is not immediately obvious if assigning a
value will make the schedule invalid. Instead, a repair operator is used after
the EDF scheduling. The mutation used for this representation, is to change a
random value a random amount, bounded by its period.

Chapter 7
Conclusion and future work
In this report the combination of network and task-level scheduling in the au-
tomotive domain is modelled. The problem is to schedule both the TT traﬃc
originating from the tasks, and the tasks themselves.
On top of this, com-
plex automotive chains act as precedence constraints where cyclic dependencies
makes it very complex to create feasible solutions.
The problem is deﬁned as a multi-objective problem where minimizing chain
latencies, ﬂow latencies, queue usage, and maximal queue usage. While a feasible
solution only involves ﬁnding a solution with all constraints fulﬁlled, other types
of traﬃc like AVB can perform better if TT traﬃc does not use all queues in
the network.
To ﬁnd combined network and task schedules with complex automotive chains
constraints, two diﬀerent heuristics where used in conjunction with metaheuris-
tics. The ﬁrst is the collection of ASAP/ALAP heuristics which created good
results in the networking part, and then the EDF with the corresponding repair
function scheduled the tasks.
The parameters of both heuristics was guided
using two metaheuristics, SA and GA.
These metaheuristics where tuned using paramILS which gave parameters with
less bias. The parameters obtained from this was then evaluated on multiple test
cases showing that GA performs better on average on huge test cases, whereas
==PAGE:58==
48
Conclusion and future work
SA performs better when a feasible solution needs to be found fast.
7.1
Future work
1. Add AVB/BE to the problem: Adding other traﬃc types to the evaluation
would allow less critical traﬃc to also exist on the network.
2. Add routing to the problem: In certain topologies, using shortest path
routing will render the problem infeasible. Adding k-shortest path rout-
ing and letting the metaheuristic choose which path to use would be an
interesting way to add routing to the problem.
3. Investigate heuristics that does not speciﬁcally care about queue usage
but rather minimal time usage
4. Delta evaluation: While delta evaluation can improve performance a lot,
it can be hard to implement in complex problems. This is because delta
evaluation requires the use of data structures with eﬃcient add/remove
operators. This can sometimes be in direct contrast with requirements for
fast iteration. Nonetheless, it is a known way to increase performance,
and would be interesting to use.
5. Memetic algorithms: Memetic algorithms is an umbrella term for algo-
rithms that combine local search using memes and global search. This
can produce good results, however the ratio of local search to global search
needs to be carefully tuned.
6. Validation of schedules on simulation and hardware: The schedules pro-
duced in this thesis was only validated by the program itself. To further
increase the validity, it would be interesting to ﬁrst see the performance
in a simulation tool, and then a physical installation.
7. Cutthrough forwarding: It is assumed that frames need to be received
fully before forwarding. If cut-through switching is implemented, then the
ﬂow latency can be decreased signiﬁcantly. Furthermore, queue usage can
be decreased as well, since frames forwarded in this way will use less time
in the queue.
