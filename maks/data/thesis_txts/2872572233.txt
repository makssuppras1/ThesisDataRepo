Abstract
As programming languages increasingly adopt automatic code formatters to standardise
style and lower the barrier to entry for contributors, the Lean programming language re­
mains without a robust formatting solution. Formatting Lean code is uniquely challenging
due to its runtime­extensible syntax and context­sensitive constructs, such as do­notation,
which require nuanced handling. In this thesis, we introduce an extensible formatting sys­
tem for Lean built on top of the Pretty Expressive pretty printer. Central to our approach is
the concept of bridges, which enable formatting rules to negotiate layout constraints with
one another. This allows for declarative, composable formatting logic while supporting
new syntax extensions. To ensure performance, we develop a pruning algorithm that dis­
cards invalid formatting combinations early. Our implementation demonstrates a proof of
concept by formatting 95% of the Lean Batteries library, which contains over 33,000 lines
of code, in just 1.7 seconds, showing that our approach is both extensible and reasonably
efficient given the complexity of Lean.

Introduction
Many programming languages like Go, Python and JavaScript have recently adopted
automatic formatters to ensure that projects have a standard code style. These format­
ters lower the barrier to entry for new contributors and make it easier to navigate larger
projects. Unfortunately, there is currently no adequate code formatter for Lean. Lean is an
extensible programming language, primarily used for proving theorems. Formatting Lean
code is a challenging problem due to its vast and runtime­extensible syntax. Therefore,
it must be possible to extend the formatter with new rules, and ideally, formatting rules
should account for new syntax that may be added to the language in the future. This
is complicated by some syntax having special requirements, such as do­notation, which
should be placed on the previous line to avoid double indentation, as shown in Listing 1.1.
This is further complicated by the fact that function applications that end with do­notation
should be formatted the same way. However, other keywords, such as ”let”, should not
be allowed to be placed on the preceding line. This is further complicated by the fact
that ”:=” and ”do” want to increase indentation by two when used individually. However,
when used together on the same line, only the last syntax on the line should increase
indentation.

While naively hard­coding these constraints by examining the next syntax is a possibility, it
would prevent future syntax from being formatted the same way without changing the ex­
isting formatters. At times, it would also require looking more than a single syntax ahead,
for example, whether function applications should stay on the previous line depends on
the content. This approach would grow unsustainable as more syntax is added. There­
fore, we prefer treating these options as patterns that can be used by formatting rules

to communicate with each other. This communication also addresses issues related to
comments, as we can now communicate that a newline must follow the formatting rule.
Furthermore, it can be written in a declarative style, making it easier to reason about.
While there are a plethora of pretty printers available, we are aware of none that currently
support this kind of pattern.
In this thesis, we construct an algorithm that allows documents to influence one another.
We call this concept bridges, since it bridges the gap between two formatting rules and
lets them negotiate the spacing between them. This negotiation can fail, since not all
combinations of documents are valid. Therefore, we have devised an algorithm to limit
our search to valid document combinations by pruning away combinations that cannot
lead to a solution early. We extend the Pretty Expressive formatter[1] due to its highly
flexible and easy­to­extend syntax, while delivering fairly good performance.
To demonstrate that these extensions are efficient and can be utilised to produce an exten­
sible Lean formatter, we show that the Batteries library [2], which contains approximately
33,000 lines of code, can be formatted in 1.7 seconds. Excluding the time it takes to find
the CST.
After a walkthrough of the necessary background, the rest of the thesis follows our con­
tributions.
• In section 3, we extend Pretty Expressive to pass context between documents with
minimal performance overhead. This extension makes it easy to support comments
and enables handling interactions between different formatting rules.
• In section 4, we create an implementation in Lean, build a suite of tools that enable
adding formatting rules for new user­defined syntax, and tools for debugging them.
• In Section 4.5, we show how to make pretty expressive 2­4 times faster using data­
oriented optimisations.
• In Section 5.1, we demonstrate that the formatter can format 95% of the Lean bat­
teries library using 314 formatting rules.
• In Section 5.3, we show that bridges have a negligible impact on performance, and
that data­oriented techniques can be used to make up the difference.

2.1
Formatting
In the following section, we will introduce influential formatters in the wild. Since this
formatter is primarily based on PrettyExpressive [1], we will follow the naming scheme of
PrettyExpressive to facilitate more straightforward comparison of features among different
printers.
2.1.1
Hughes printer
The inspiration for many future pretty printers was introduced by Hughes in 1995[3]. The
core idea was to automatically select the best layout for a document using a greedy algo­
rithm. Documents are constructed using the primitives
text s write string s containing a single line
nest i Increase the indentation by i, which is used in <$>
dl <$> dr Concatenate two documents vertically
dl <+> dr Concatenate two documents horizontally, the entire right document
dl <|> dr provide the choice between 2 alternative layouts.
Documents are constructed using <+> for horizontally aligned concatenating documents
and <$> for vertically aligned concatenation. As an example, consider this document.
text ”Hello ” <+> (text ”aligned” <$>
text ”concatenation”)
Which would be rendered in the following way:
Hello aligned
concatenation
The main way of providing different formatting layouts was through a sep function that
takes a list of possible layouts that would either be combined using <+> or <$>.

2.1.2
Wadler printer
Wadler[4] simplified this design and allowed unaligned concatenation <>. He defined the
pretty printing language using the following constructs
text s write string s containing a single line
nest i Increase the indentation by i, which is used in nl
nl Create a newline and indent according to the current indentation
dl<>dr Concatenate two documents
This allows us to write documents in the following way. Where nest i increases the in­
dentation by i, which means every time we have a new line nl
text ”Hello ” <> (text ”unaliged” <>
nest 2 ( nl <> text ”concatenation”))
They would be rendered in the following way:
Hello unaliged
concatenation
To define alternative layouts, Wadler provides the group function, which considers two
layouts. One where we flatten the content by replacing all newlines with spaces, and one
where the layout is left untouched. This is somewhat limiting because in some languages
like Lean it is possible to define optional ”;” after a let binding, which only has to be there
if we want to write more on the same line. However, it is very safe, since the user cannot
produce invalid layouts.
2.1.3
Derek printer
Derek introduced another influential pretty printer [5]. This printer trades expressiveness
for performance. In fact, the pretty printing step never needs to look at more than a single
line. It can therefore format the text in O(n) time and requires only O(m) space, where n
is the size of the document and m is the maximum length of a line. The printer achieves
this by providing groups of text that ideally should not be broken, followed by either the
option to start a new line or always start a new line. This design assumes that another
process runs in parallel and can accurately determine where newlines should be placed.
This printer is also limited in that it can only output whitespace differences.

2.1.4
Inko Formatter
An alternate formatter loosely inspired by Wadler, was created by Yorick Peterse to format
Inko[6]. His approach builds a tree primarily using the two types a Node and Group. A Node
contains text that must be grouped together, and a group collects an array of Nodes that
must either be placed on the same line or all Nodes will be placed on their own lines.
This system allows nesting groups that will break independently, which means that just
because the outer group is split into multiple lines, it does not necessarily affect the inner
groups. To allow altering the code formatted in response to how the code is wrapped
they introduce an IfWrap node, that takes 3 arguments: A reference to the Group that
may wrap, a layout when the Group has been wrapped and another layout if the group
has not been wrapped. IfWrap allows adding trailing commas to lists or surrounding a
section in parentheses.
2.1.5
Bernardy Printer
Bernardy’s primary goal is to produce visually appealing output at the expense of speed
[7]. It points out that due to the greedy nature of the Hughes printer, there are instances
where it fails to produce optimal results. Bernardy introduces the Pareto frontier to pretty
printing, which is used to compare partial solutions, called measures, to determine if one
of them is strictly better than the other. The goal of a measure is to represent a possible
layout using the minimal amount of work, while still accurately reflecting its characteristics.
Bernardys’ measures consist of maxWidth, height, and lastWidth. If we have two options
and one of them is strictly better than the other, then there is no reason to keep the inferior
one. These optimisations make it fairly fast.
2.2

Pretty expressive[1] is primarily based on the Bernardy printer and aims to address the fol­
lowing concerns. The Bernardy printer fails if the width exceeds the specified limit. Pretty
Expressive will fall back to choosing any option quickly through a property it calls ’tainted’.
Furthermore, Pretty Expressive supports a wider variety of styles, whereas the Bernardy
printer is limited to aligned concatenation. Pretty Expressive defines a document D to
consist of the following elements. This is the definition of a pretty printing language(PPL)
that we base our implementation on.

Pretty Expressive lets the user define a computation width W, after which the printer will
give up and choose any solution. This computation width is separate from the page width,
which is used to determine how wide the page preferably should be. Typically, the page

means that even if the code must overflow slightly, the printer will choose to minimise the

3
Formatting with Bridge Constraints
The problem our extension aims to solve is conditionally concatenating two documents.
The end of the first document must match the start of the second document as shown in
figure 3.1.
Figure 3.1: Concatenate two sets of documents, where the end of the first document must
match the start of the next.
Solving this problem yields several desirable properties when working with composable
formatting rules.
• It allows passing context between formatting rules, which means we can condition­
ally format based on different formatting patterns. This is important for supporting
do, by, structs, and more in Lean.
• It makes dealing with end­of­line comments easier, since a formatting rule can tell
the next formatting rule that it must start on the next line.
• Because the formatting rules can agree on the spacing between them, we do not
end up with doubled whitespace
In this section, we will define bridges and how they can be used to solve the do­notation
problem. Afterwards, we will look at how bridges interact with flatten, and why this must
be treated with care. A bigger problem for bridges was that they break invariant 1, which
is required by tainted to find a solution quickly. Therefore, in Section 3.3, we will define
how to restrict the search space only to valid solutions. Finally, we will update the Pretty
Expressive algorithm to support only examining valid solutions.

3.1
Syntax extentions
We extend the Pretty Expressive syntax defined in section 2.2. Our primary additions to
the syntax are provide and require. Their purpose is to pass context between documents.
provide b Give the next element the option to choose from one of the bridges in b.
require b The previous document must have ended with a bridge that overlaps with b,
otherwise the document fails.
Furthermore, we add bubbleComment and cost. However, we won’t focus on their im­
plementation, as they can be easily added and will be explained separately from the al­
gorithm for simplicity.
bubbleComment c Move a comment after the previous newline. This is used if comments
are in an illegal place or if the comment exceeds the width of the page.
cost c Disincentivize the formatter from doing an action. This cost is weighted
equally to newlines. Amongst other things, it can be used to tell the
formatter that we should avoid newlines in the middle of parentheses.

and do­notation problems 3.2.2

3.2.2
Bridge patterns
Bridges were introduced to solve communication issues between independent syntax el­
ements via patterns. A common pattern when formatting lean is the do edgecase. When
we look at the functions main and mainDo in listing 3.1, we see that they have different
requirements for where the next character is placed after the ”:=” token. We call this pat­
tern immediate value. For the formatter to allow this pattern, it must be provided by the
assignValue syntax (”:=”) and required by the do notation syntax (”do”). In the case of
immediate value, the parent syntax surrenders control of the formatting to the child syn­
tax. This means it will not increase nesting or adjust spacing between parent and child
elements.

We decided to introduce patterns instead of just looking ahead and add special handling
for do/by/structs/... due to the extensible nature of Lean. When implementing look­ahead
or look­behind, we are always restricted when introducing new syntax, because if we were
to add a new syntax that is supposed to behave similarly to these, we would have to modify
an existing formatter. By using formatting patterns that both parties agree to, we can add
new syntax, such as if statements, and allow them to use do/by notation. The downside
to this approach is that it is slower to introduce new patterns. However, we believe that
with time, necessary patterns will emerge.

To ensure our algorithm is compatible with PrettyExpressive, it must be capable of quickly
selecting a valid solution when the computation width is exceeded. However, bridges in­

set of constraints that identify unsolvable paths early, allowing us to prune them before
attempting to solve the problem. It is helpful to think of this problem as a Jigsaw puz­

4.3.1
Nested syntax
The DAG (Directed Acyclic Graph) structure is essential for the performance of the format­
ters, so it must be difficult to accidentally violate. By reusing previously computed results,
we reduce the time required to generate the document from exponential to linear. Without
this reuse, performance could degrade exponentially. If a piece of syntax appears in both
branches of a choice, the formatter would need to process all of its children repeatedly
and re­evaluate any nested choices each time.
Therefore, we repurpose the syntax position of the as a unique ID. The first time the user
formats the syntax, it is added to a cache. The next time we see the same syntax, we can
reuse the cached version. To make this work we temporarily add Lean syntax to the doc­
ument, but this syntax is automatically formatted before formatting begins. This approach
was seen as preferable to relying on users remembering to assign IDs to documents,
which would be required due to the inability to alter global variables, as is the approach
taken in other implementations of Pretty Expressive.

4.3.3
Conventions for formatting rules
Since numerous independent Syntax rules perform formatting, they must adhere to a
common set of conventions to produce a coherent result. These are the rules we are
currently using.
• Parent rules are responsible for the spacing between child rules. By only having
the parent define the spacing between child elements, we reduce the risk of dis­
agreement. Furthermore, children typically can’t reason about what should precede
them. Since even syntax such as let, which one would presume to always start on
a newline, has counterexamples where it can be placed after a semicolon.
• Always use bridges to connect syntax elements. Because otherwise, we can’t rea­
son about whether we are allowed to place comments.
• Never write atoms as strings. Instead, use the original syntax. Otherwise, we lose
the comments attached to the original syntax.
• All formatting rules that increase indentation must check if they are followed by bi.
If they are followed by bi, then the next element will handle indentation.

4.3.7
Comments
As pointed out by Hughes [8], there is a difference between pretty printing internal data
structures and formatting code. Formatting code introduces the additional complication
of supporting comments and preserving the original layout of the code. While comments
have typically been excluded when describing pretty printers in literature, we will include
them here because we want to provide a decent implementation for automatically plac­
ing comments, without each syntax transformer having to handle comments. This choice
is motivated by the expandable nature of lean. As an example, there are around 650
different kinds of syntax used in the batteries library[2]. In our opinion, it would be unrea­
sonable to handle comments for every one of these individually, and ideally, it should be
done automatically.
The approach we will take is to prefer leaving the comment where it is. However, if it must
be moved, then move it before the previous line, maintaining the same indentation. In
theory, this could be done without new syntax by editing the document that came before
the comment. However, imagine the following document.
text ”a” <> ( nl <|>text ” ” ) <> text ”b” <> ( nl <|>text ” ” ) <> text ”–comment”
There are two possible newlines before the comment, and we don’t know which one will
be chosen as the newline. In theory, we could transform the document into something
akin to:
text ”–comment” <> nl <> text ”a” <> text ” ” <> text ”b” <> text ” ”
<|> text ”a” <> nl <> text ”–comment” <> nl <> text ”b” <> text ” ”
<|> text ”a” <> nl <> text ”b” <> nl <> text ”–comment” <> nl
<|> text ”a” <> text ” ” <> text ”b” <> nl <> text ”–comment” <> nl

the operation, we add bubbleComments to the syntax. During the resolve operation ⇓RS,
it will add bubbleComments to the measure. During the render operation ⇓R, it will find
the last newline and place the comment after it, while matching the indentation.

ments by providing a fallback option if it is not possible to keep them where they are, since

4.3.9
Parallel computation and memory leaks
Formatting the codebase in parallel is motivated by 2 reasons. Primarily, we have a mem­

bases this eventually causes segmentation faults due to high memory usage. Secondly,
it enables faster running time since formatting each file is an independent task.

the memory leak issue by moving it to a process that will terminate after the file has been

Performance in Lean is somewhat complicated since the user is not meant to think about

4.5.1
Compiled Lean
When formatting large documents using pretty­expressive, the limiting factor becomes
memory and cache misses. Therefore, we will briefly cover the memory model of lean,
An Extensible Lean Code Formatter
45
==PAGE:52==
highlighting its strengths and weaknesses. While Lean syntax is highly customizable, its
control over memory is somewhat limited. To illustrate this point, we demonstrate that a
C++ implementation can be written that is approximately 2­4 times faster than the Lean
implementation. Therefore, the Lean implementation spends a lot of its time managing
memory.
Immutability
Everything in lean is immutable by default. This is great for safe, reusable and predictable
code. However, this makes caching tricky, as it requires a separate cache object to be
threaded through the program and updated at the appropriate times. If it weren’t possible
to circumvent this immutability, it would be close to impossible to write an efficient cache.
Luckily, there are two options to circumvent immutability. In an IO context, it is possible to
create references that can be updated, which we use to discover which formatting rules
are present in the system3. The second option is linear access.
Linear access allows lean to mutate arrays if there is exactly one reference to the array[10],
which means that for large objects like caches, we can avoid copying the entire array and
instead update the index that has changed. Furthermore, we should prefer arrays, due to
their good cache locality.
Cache locality
As a refresher, Cache locality can be split up into Temporal locality and Spatial locality.
Temporal locality refers to the principle that memory will likely be used again shortly in
the future. Since we are performing a depth­first search over alternative ways to display
branches of the tree, we likely have information about the branches still in the branch.
Spatial locality refers to keeping related items physically close in memory. This is im­
portant because when the CPU fetches information from memory, it typically fetches 64
bytes at a time. Therefore, if data is located next to each other in memory, then we only
have to wait for fetching the first item, and when we access the following item, it is already
there. As an example, an array of numbers has excellent spatial locality as long as you
go through them sequentially.

Object allocation
Lean uses mimalloc to manage its memory allocation. Mimalloc divides the heap into
pages that contain elements of the same size­class [11]. What separates mimalloc from
other allocators is that it defines multiple free lists per size class. This means that each
free list has spatial locality instead of being spread out over the entire heap. When this
change was implemented in Lean 3, performance increased by 25% in some benchmarks
with large data structures.
Therefore, whenever possible, we should allocate related objects simultaneously.
To
demonstrate the effectiveness of this optimisation in isolation, let us consider summing an
array of structs and an array of numbers. Since the array of structs is created in the same
order as they are stored in the list, it is similar to creating two parallel arrays. Where the
first array is referencing the second one, although the second array might be scattered
across multiple pages. When we traverse the first array, we get the advantage of loading
the second array in a reasonable order.
To demonstrate the worst case, if we do not fill the array with values immediately, let us
shuffle the array containing the structs. As shown in Table 4.1, this nearly doubles the
time required to sum the array. All of the extra time taken for this problem is due to cache
misses. We mention this scenario because when memoising, we only add content to the
array when it is needed, leading to an array of pointers that likely point to different places
in memory. Note that we only shuffle once, and that other tests show that bad access
patterns can cause around 16 times longer access times [12].

4.5.3
Performance in c++
The easiest way to achieve better performance is by doing less. It turns out that our C++
implementation executes approximately one­fifth of the instructions and approximately
one­fourth of the cache misses that the Lean implementation does. When performing the
sexp­full­16 benchmark in section 5.3. The primary way this is achieved is by ensuring
memory safety without the use of reference counting. One way to accomplish this is by
using memory arenas that can allocate a large number of objects in a single system call.
When they are no longer needed, the entire content can be wiped out by deallocating the
arena. The second optimisation we do is to reduce the number of pointers by inlining val­
ues in our structures. Other than that, we have the privilege of knowing that our types are
numbers and therefore do not need to check whether numbers are pointers or numbers.
This should be a free check for the CPU. However, when examining the generated lean
code, it appears frequently and likely contributes to the high instruction count.
We should note that instruction count alone is not a problem, as long as the instructions
are independent of each other. This would allow the CPU to perform more instructions pr
cycle. This could also be the reason that, on average, Lean performs 2.41 instructions
per cycle and C++ performs 1.72 instructions per cycle.
Reusing Arrays
Since the implementation uses a lot of arrays to create and store temporary MeasureSets,
we created a pool of arrays. By reusing arrays, we increase the likelihood that the array
is already in the cache, making it faster to use again. Furthermore, we waste less time
creating and cleaning up arrays. The downside is that we must manage when the arrays
are returned ourselves. We always borrow and return them within the same function’s
lifetime. This means that for the resolve function ⇓RS to return a MeasureSet, it must be
given an array by its caller that it can use to return the values.
Array of structs
From table 4.1 we observe that while it can be fast to follow a pointer to find the value, it
is quicker if the array contains the value. Especially if it is likely that we are not traversing
the array sequentially, we therefore structure our data as described in Section 4.5.2.

6
Conclusion
Formatting Lean code is challenging due to its extensible syntax and context­sensitive
constructs. We introduced an extensible formatting algorithm that utilises the concept of
bridges to enable formatting rules to communicate the spacing between them. This solves
problems with do­notation, such as determining where it should be placed on the same line
and avoiding duplicate indentation in a composable manner. This composability enables
new syntax to interact with existing syntax and behave like do­notation, or integrate with
do­notation as needed. Our Lean implementation, based on Pretty Expressive, can format
the Lean batteries in under 2 seconds. When excluding the time taken to find the CST.
Furthermore, it is likely that once we have the option to get the CST for every command
in an open document, then formatting a file can be done in under a second. However, this
does not help use cases such as formatting an entire folder through the command line,
since it would be required to find the CST every time. We managed to format 95% of the
Lean batteries library, which we see as an indication that it is possible to format the entire
Lean language with more time.
6.1
Future work
The biggest performance hindrance is the lean interpreter. When formatting the Batteries
library through the command line, approximately 2% of the time is spent on formatting.
The rest is spent finding the CST and validating that it was formatted correctly. Therefore,
it would be advantageous to optimise it, although that is likely a challenging task given the
complexity of Lean. Due to this complexity, it is also unlikely to create a separate parser,
since it would have to be compatible with Lean itself. This problem might be solved when
and if Jit is implemented in Lean4.
Although the performant architecture introduced in section 4.2 improved performance in
our case, it may not be needed if we can integrate the formatter into the Lean Language
server, since we should be in a compiled environment at that point. However, it may still
be faster to have compiled formatting rules, especially for libraries that introduce a lot of
new syntax. If the formatter cannot become part of the Lean Language server, then the
current architecture is likely the best chance the formatter has to become viable. It is

likely to see considerable performance improvements when all commands are formatted
simultaneously, as this avoids the overhead of starting several executables.
The debugging website should be improved. One feature that would make it easier to
use would be the ability to search for a solution with constraints on a node in the tree.
Currently, the user must manually toggle the choices to configure the document to produce
specific outputs, such as newlines. Adding this would also enable the website to serve as
a learning tool, allowing users to understand how formatting works and why the formatter
prefers a particular style. Furthermore, it should be optimised to be lazy so that it can
handle large functions.
There is also a lot more work to be done regarding formatting rules. After the batteries
