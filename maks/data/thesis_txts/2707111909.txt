Abstract
There is a large demand for storing large amount of data, which is why compression
can help store more data while using less space. However compression often comes
with a compromise on accessing sub-parts of the data. The focus on this thesis is
to improve on the worst case scenario runtime of a random access operation in the
compression algorithm LZ77. For this purpose the algorithm, alongside the other
compression algorithm LZ78, is implented and given an overview of throughout this
thesis. There are two parts to the random access operation in LZ77 compressed
data, searching for next phrases and how many times the search is needed. The
focus is on limiting how many times the search is needed, as opposed to optimizing
the search, and a number of post-processing solutions which can improve on this
are presented. It will be shown how the best solutions can reduce the worst case
scenario of number of searches by a lot while keeping the increase in space small.

Chapter 1
Introduction
The concept of lossless compression entails taking a large data set and reducing the
storage it takes up without losing any information about the data. This data can
then be restored in full by decompressing the compressed data. Seeing as there are
increasingly larger need to store vast amount of data, storing data compressed has
become increasingly popular[5, 2]. However, a big issue with compression is that it
can be diﬀicult to retrieve a sub-part of the data set.
If the sub-part is a single data point at index x this is known as the random access
problem. If a random access operation is made on an uncompressed data set, then it
only takes constant time since the element trying to access can be accessed directly
at the index. However, when compressing data this attribute is likely to get lost,
since a lot of compression algorithm isn’t optimized for random access, but for
optimal compression. In Lempel-Ziv 77 (LZ77) this is especially true since the basic
compressed solution has a worst-case runtime of O(sj + p), with s being the time of
the search method used, j the number of jumps and p the number of phrases. This
will be clarified further in chapter 3.
In the case where the sub-part is a range of data points from index x-y where x < y,
it is known as the substring decompression problem. As mentioned, retrieving an
data from an index in uncompressed data this is fast, because of the fact that it can
be done by a constant random access operation on all the indices. In compressed
data this is significantly more complex.
This thesis will focus on the random access problem in the algorithm LZ77 [8], by
trying to reduce the number of searches needed in worst case scenarios. This will
be done by trying to look for patterns in the test data and apply post-processing
to already compressed data to figure out if any of these patterns can be utilized to
reduce jumps significantly. This thesis starts by testing some of the simple solutions,
which includes limiting the max number of jumps, limiting the number of paths that
runs through a specific index and see if these can be combined in any way. These will
of course not lower the theoretical worst-case runtime, since how low the runtime
gets will depend on the experimental results for every file.

1.1
Related work
The problem with random access in Lempel-Ziv compressed text is has been around
for some time. Several articles have been trying to tackle the issue. Their work
will be briefly introduced in the following section. Whereas this thesis will focus on
decreasing the actual runtime, previous work has mainly focused on lowering the
theoretical worst-case runtime.
LZ-End
LZ-End [5] is an compression algorithm introduced by Sebastian Kreft and Navarro
Gonzalo that takes the original LZ77 algorithm and changes its compression to
support fast random access. This is done by forcing all compressed phrases to end
in another phrase. Furthermore, instead of having a value in the phrase that tells
the algorithm how many steps to go back, like in the original LZ77, it states which
phrase the compressed string ends in. Thus, it removes the need for searching for the
referenced phrase is removed, since it has a link directly to the phrase which allows
the phrase to be found in a constant time by doing a lookup with phrase index.
Because of this, the random access time can then be improved and bounded by
O(j) with j being the number of jumps needed to be made. This comes with a space
trade-off that makes LZ-End have around 5% worse compression than LZ77 most of
the time, but up to 20% on highly repetitive collections, where the compression is
already very good.
LZB
LZB [1] is also a Lempel-Ziv algorithm that uses the same approach as LZ77. LZB
differentiates itself by having a ”gate” telling the algorithm how far back a phrase
can have references. An example could be if the gate is x indices back from the start
of phrase then the phrase has to be able to be decompressed without using any of
the indices before index x. That will obviously reduce worst case scenario number
of jumps needed since it would at the very least not be able to jump past index x.
However it of course also comes at a space trade-off since the compression algorithm
is limited in how far it can look back for better matches.
Relative Lempel-Ziv
RLZ [6] is a variant of LZ77 that is optimized for Genomes. It supports random
access in O(log(r × n
z
) which it can do by utilizing at most O(log α + z log n +
z log r × n
z
+ z) bits. Then it’s also shown [3] that by utilizing sublinear extra space
it’s possible to do random access in constant time.

Chapter 2
Lempel-Ziv compression
In 1977 Abraham Lempel and Jacob Ziv introduced the Lempel-Ziv 77 [8] and
Lempel-Ziv 78 [9] algorithms which are universal sequential lossless data compres-
sion algorithms. They work well on repeating data collections like DNA sequences
and are used for many standard compression algorithms today.
The following chapter will give an overview on how the standard Lz77 and Lz78
algorithms work as well as how the basic random access algorithm works in these.
2.1
LZ77
LZ77 works by compressing data into a number of tuples (here after referred to
as phrases) that with the help of previous constructed phrases describes a specific
sequence of the data being compressed. A phrase for LZ77 looks like the following:
(b, l, c)
where b describes the number of steps back necessary to take in order to be at the
start of the phrase, l is the length of the phrase minus the last character and c is
the last character of the phrase.
These phrases are constructed in the encoding and can be decompressed into the
original string.
2.1.1
Encoding
Encoding data into phrases starts at the beginning of the data meaning it’s being
compressed from left to right. Every time a new phrase is created, the algorithm
searches for the longest prefix from the current index, that already occurred in the
previous encoded part of the string. Then it takes the current index minus the index

10
==PAGE:12==
of the prefix seen earlier and saves that as b in the tuple. The length of the prefix
is saved as l, the next character after the prefix is then saved as c and the phrase is
constructed. The process of making a new phrase begins from the index after the
index of c. This process will continue until all the data is represented in phrases, in
which the file can be referred to as compressed.
Below the 6 steps of the algorithm is shown.
Figure 2.1 shows how the string
abcabcababacabab is compressed as an example.
1. Initialize empty string, length to 0, index to 0.
2. Add the next character to the string
3. If that string has been seen before, increment length and go back to step 2.
Save the start index of where the string was seen before as index.
4. Construct tuple with b = index - start index of string, l = length, c = last
character added to string
5. If not at the end of the string, go back to step 1.
6. Algorithm done.
Figure 2.1: abcabcababacabab compressed into phrases with Lz77
One issue with the encoding part of the LZ77 algorithm is when looking for the
longest prefix in a big data set, there is a lot of substrings to compare with. To
solve this issue the encoding is split into two cases. One where the string to be
encoded is smaller than 8 characters. These cases are handled by simply storing the
starting position of all substrings of size smaller than 8 in a dictionary and then
these can be looked up in constant time if needed. The reason why this can’t be
done for all is simply because of the amount RAM needed to store all possible strings
in a dictionary. Instead for alle substring of size larger or equal to 8 the Rabin-Karp
algorithm is used.
Rabin-Karp algorithm
The Rabin-Karp Algorithm [7] is a string-matching algorithm, which can be used
to filter substrings eﬀiciently, so it isn’t needed to compare all substrings to find
the longest prefix for the encoding. It works by using a rolling hash function that
hashes all patterns of a specific size to a numeric value.
All substrings that has a pattern as a prefix can be found somewhat eﬀiciently with
the help of a hash function that promises very few collisions. The LZ77 algorithm
implemented in this project uses Rabin-Karp whenever the string to be encrypted
is larger or equal to a size of 8 characters. The value was chosen after initial testing
various values, showed that 8 had the fewest collisions, while still being fast. For all

This shows how all the different hashed prefixes of size 8 in the string is put into

Rolling Hash Function
The rolling hash function used here is a basic Polynomial Rolling Hash. This means
whenever a string of length n has to be hashed the following equation is used:
H = a0xn−1 + a1xn−2 + ... + an−1x0
where a0, a1, ..., an−1 is the ASCII values of characters of the string to be hashed, x
is a constant chosen. All operations are done modulo m to avoid integer overflow. If
the string s1 from index a to index b where a < b has been hashed and now the string
s2 from index a + 1 to index b + 1 has to be hashed it can be done in constant time
by reusing the previous hash. It does so by simply subtracting the value at index a,
shifting the position of the other characters in the polynomial by one position and
adding the value at index b + 1. The shifting can be done by multiplying the entire
polynomial once by x. This can be described by the formula:
H(s2) = (H(s1) −s[a]xn−1)x + s[b + 1]
where s[a] is the ASCII value of the first character in string s1 and s[b + 1] is the
ASCII value of the last character in string s2. All operations are still done modulo
m.

If the parameters, the constant x and the modulo value m, is chosen correctly the
algorithm yields few false positives, meaning strings that has the same hash value
despite being different strings.
Analysis
There are multiple angles to look into in regards to the analysis. Firstly, when com-
pressing the algorithm needs to find the longest matching prefix already compressed.
For prefixes less than 8 characters, these can be looked for in constant time because
these values are stored in a dictionary. However, in the case where the prefix is
larger or equal to 8 a different approach is needed. Utilizing the Rabin-Karp al-
gorithm all the strings with the same hashed value from the first 8 characters can
be found in constant time as well. However doing this means that it is needed to
continue checking the next positions on all the matches to see how many of the sub-
sequent characters match. The runtime will therefore depend on how many matches
the Rabin-Karp algorithm finds. There is also the chance of collisions in the hash
function where the strings are not the same, making the number of matches larger
than it needs to be, however Rabin-Karp with a rolling hash function promises low
number of collisions if the right parameters are chosen.
Therefore if the larges number of collisions is denoted c then the runtime can be
bounded by O(nc) since for every character you might need to check up against a
maximum of c other characters. This also shows why choosing parameters to have
few false positive is important, since the size of c will have a large impact on the
runtime. In practice c tends to be quite small for strings of size 8.
2.1.2
Decoding
In order to decode the phrase you go b steps back from the start of the phrase in
the original string, take the next l characters, add c at the end and then the phrase
is decoded. If the start of the phrase n is denoted i, then in order to decode n, the
phrases that contains all the characters i −b ≤m ≤(i −b) + l must be decoded as
well. Using this approach it’s easy to decode an entire LZ77 encoded text, just start
from the beginning and decode the first phrase, then that phrase is already decoded
if the phrase is used again later in the decoding process.
Analysis
Decoding LZ77 phrases is done by iterating through all phrases and since the number
of steps back can be done constantly, the time it takes to decode a phrase is the
length of the phrase. All the lengths combined will of course be the length of the
original encoded text so the final runtime will be O(n)
2.2
LZ78
LZ78 seems a lot like LZ77 on the surface, however in practice they’re very different.
Instead of searching for all previous strings LZ78 only search for strings in previously
encoded phrases. A LZ78 phrase looks like the following:

13
==PAGE:15==
(i, c)
where i is the index of previous phrase and c is the next character. That means that
for all phrases if a phrase j has an index i then the length of the encoded phrase i
is always one less than the phrase j.
2.2.1
Encoding
Similarly to the LZ77, the LZ78 encoding happens from left to right. When a string
is encoded, the algorithm will find out if this specific string has been encoded before.
If the string has been encoded before the next character is added to the string, which
the algorithm then continues to search for. If it has not been encoded before, the
string is encoded with the last character as c as well as the index of the phrase,
where the string minus the last character was encoded. If it’s only a single character
then the character is saved directly.
Figure 2.3: abcabcababacabab compressed into phrases with Lz78
An advantage of the LZ78 encoding is the speed, because it only searches through
phrases that are already encoded, whereas the LZ77 looks through all previous
strings. On the other hand, the encoding is worse, meaning the number of tuples/s-
pace needed is larger so more space is needed to represent the same text compressed
with LZ78 than LZ77. This will be illustrated in section 2.3 where it will be discussed
how well the two compressions work.
Figure 2.3 shows how the string abcabcababacabab would be compressed into LZ78
phrases.
Analysis
According to LZ78, a phrase must start from the beginning of a new phrase. Thus,
it is only necessary to save strings that have been decoded into phrases. Therefore
it’s possible to save all the phrases in a dictionary to be looked up in constant time.
For that reason the algorithm only needs to iterate the initial text once and look in
the dictionary every time. The runtime is therefore O(n).
2.2.2
Decoding
The decoding of LZ78 encoded text is quite simple. Initially, the algorithm looks at
the first phrase, which will be a single character. Then it looks at the next phrase. If
the next phrase contains an index of another phrase, it will take the decoded phrase
and add the extra character. This continues until all phrases are decoded.

Analysis
The decoding of LZ78 essentially works the same way as LZ77 where the algorithm
has to iterate through the phrases from the first to last and decode them one by
one. The time it takes to decode one is, the length of the phrase and all the lengths
combined will be the length of the original text. Thus the runtime is O(n).

Chapter 3
Random Access
The following chapter will give an overview of how basic random access works for
both LZ77 and LZ78 and subsequently discuss a number of possible improvements
for worst case scenario in LZ77. These improvements will mostly consist of small
changes to the LZ77 compression or post-processing the compression. To compare
performances the results will be compared to different post-processing versions as
well as LZ78 random access.
“Number of jumps” is an essential term when discussing the improvement of random
access. It refers to the number of different phrases, the algorithm must go through
in order to find the right character. This will be explained more in 3.1 and 3.2.
When optimizing random access, what will be the focus will be to reduce the worst
case scenario number of jumps.
3.1
LZ77
There are two main parts to random access in Lempel-Ziv compression, which are
necessary to consider; first of all which search method is used to find a specific phrase,
and secondly the number of jumps. In a very basic form of the LZ77 compression
algorithm, the random access works is, if the character at index x needed to be found,
then the algorithm would first need to find out what phrase the index belongs to.
You start from the initial phrase, take the length of that, and if x is larger than the
length, then continue with the second phrase. If the length of the second phrase
plus the length of the first phrase is larger than x, then you know x has to be in
phrase 2. If not, add the length of phrase 1 to length of phrase 2 and use the new
length while continuing with the third phrase.
After finding the phrase containing the index x, if x is not the last character of the
phrase, then if y is the number of steps to go back, defined by the phrase, then
the next index you need to find is x-y. That is 1 jump. These jumps will need
to continue until you reach a phrase where the index you’re looking for is the last
Random Access In LZ77 and LZ78 compressed text
18
==PAGE:20==
character of a phrase and then it’s written directly in the phrase. For every jump
however, it is necessary to find the phrase that contains the new index to be looked
for.
Figure 3.1 shows an example of random access on index 14 on the compressed string
used in figure 2.3. The progress in this specific example is as following:
• The algorithm identifies the phrase containing the index as the last phrase
• The index is not the last character of the phrase so the algorithm finds the
phrase containing index 14 −4 = 10.
• The index is again not the last character, so the algorithm finds the phrase
containing index 10 −2 = 8.
• The index is still not the last and the index 8 −3 = 5 is in the same phrase
so the algorithm finds the phrase containing index 5 −3 = 2.
• The index is the character of the phrase so the result of random access is the
last character in the phrase, which is b.
Figure 3.1: Example of a random access call on index 14
3.1.1
Analysis
The runtime of the search method is marked as s and the number of jumps for a
phrase is marked as j. There must be a search for every jump. Thus, the part of
following the jumps will take O(sj) time worst case scenario. The search method
used for this thesis is a basic binary search s = log(p) where p is the number of
phrases. Then there is the first step, finding the phrase containing the first index.
This will need a iteration of all phrases up until the phrase containing the index,
which could be the last index of text which will be in the last phrase. The iteration
will there for be O(p). The runtime of the random access therefore is O(sj + p) and
having O(log(p)j + p) with the search method used in this thesis.
3.2
LZ78
LZ78 compression has the same approach as the LZ77 when it comes to random
access, however the searching part is simpler. The first step to finding index x in
LZ78 is to identify the phrase that contains the index x. This is done by going
through the phrases from the beginning since the first phrases will always be of size
1. Moreover, we know that all phrases will have a reference to another phrase and
will be 1 larger than the phrase it references. From that we know the sizes of the
Random Access In LZ77 and LZ78 compressed text
19
==PAGE:21==
phrases and can use the same approach as in 3.1 to find the initial phrase index
containing x.
However, the two algorithms varies a lot in some aspects. If the size of the phrase
found in LZ78 is y and the start index of the phrase is z the number of jumps needed
to find the character is (z + y) −(z + x). This is the amount of steps back, that
is needed for the character on x to be the last character of a phrase. Unlike LZ77
however, it is not necessary to find a search method to find the next phrase since
there is already a reference to it in the previous phrase. Thus, all the jumps can be
done in constant time.
3.2.1
Analysis
The runtime of the basic LZ78 is the same as the LZ77 the only difference is that the
search method is constant. Therefore the runtime of the random access is O(j + p)
with j as the number of jumps per phrase and p as the number of phrases.

3.5.2

While the proposed solution in section 3.5.1 showed some great improvements, there
is room for more. Looking at the file dblp.xml.50MB it does not improve a lot with

3.5.8
Choosing first string in encoding
In the baseline, in order to decrease the amount of space used, the last occurrences
of strings are always used if there are more than one in the encoding. This method
reduces space, because the number of steps the algorithm must go back is smaller
and with fewer number of steps, the integer will be smaller and this will require less
space. On the other hand, it is possible that it will increase the required number of
jumps for random access, if the algorithm uses the last match. This is due to the
issue that if the first jump is short, there is room for more jumps afterwards. Thus
the next solution to try is to change the encoding to use the first occurrence instead
of the last. This will not increase the number of tuples. Therefore, the comparison
of space will be done by encoding the compressed text with Huffman coding and
compare it to the baseline set in table 2.3. When compressing with Huffman the
start index is removed from the tuples in order to compare the sizes. The results

Chapter 4
Conclusion
This thesis successfully implemented LZ77 and LZ78 with a competitive compression
ratio. It was shown that the worst case scenario number of jumps during random ac-
cess operation can decrease significantly by doing simple post-processing operations
on the compressed data. Especially the Max Jump Live algorithm can eﬀiciently
limit the number of jumps a random access operation can have, while only increasing
the space a bit. The Paths Live algorithm makes sure all random access operations
that has more references from other random access operations than a specified cutoff
will have 0 jumps. This also improves worst case jumps, although not as well as Max
Jump Live. Importantly, the best improvement was seen, when the two solutions
were combined. When applying the Paths Live post-processing and afterwards the
Max Jump Live, the maximum number of jumps was increased with an average of
87,47% compared to baseline. The best result was improved by 95,54%, whereas
the lowest was 77,03%. In comparison to the LZ78 implementation, the combined
solution the average maximum jumps was 79,00% less while using an average of
31,94% less tuples.
A disadvantage of this approach is that it increases the space which is necessary to
use for the same number of phrases. Nonetheless, the findings show that the number
of maximal jumps decreases even further, when the first string is chosen during
encoding while a combination of the Paths Live and Max Jump Live solutions is
used. The number of max jumps is decreased by an average of 75,63%, ranging from
98,1%to 5,17%. Compared to the LZ78 implementation, this solution improves max
jumps with 81,70% in average and uses an average of 29,59% less tuples.

Chapter 5
Future Work
There are still a number of things not further investigated in this thesis that are
left to be done.
First of all, other possible improvement could be tested.
For
example, the random call operation could improve, if the algorithm could identify
and choose the best string to reference, when finding a matching, rather than simply
choosing the first. The findings show that the number of jumps can improve, if the
compression algorithm chooses the first occurrence of a string instead of the last.
Possibly, it could improve even more, if the algorithm e.g could choose the string
that in average has the lowest number of jumps on all the characters.
Another
possible improvement would be to use the distinct path algorithm in combination
with the length of the paths. Thus the index would not only be saved based on the
amount of distinct paths. Instead, the index would be saved directly, if there was
more than x distinct paths with length of more than l.
Future work could also look into whether there is a better method to find the optimal
parameters.
In this thesis, the data set was tested with a number for different
variables in order to find the best path cuts and max jump cuts. This however, is
demanding and time consuming. It would be beneficial to invent an easier way to
identify the values.
Moreover, it would also be interesting to investigate whether the patterns for the
compression algorithm that are found in this thesis, exists in other compression
algorithms like LZ78, LZEnd, LZB among others to see if the same patterns can be
found in these.
Finally, It would also be interesting to compare the described improvement to other
algorithms like LZEnd and LZB to see whether their theoretical improvements are
superior.

Project Description

Danish Title: Opslag i LZ77-komprimeret tekst
Type: MSc thesis
Student: Frederik Kirkegaard, s165509
Advisors: Inge Li Gørtz and Philip Bille
ETCS: 30
Evaluation: 7 point scale

Project Description
The topic of this project is algorithms for random access in the LZ77 compression.
The goal is survey and implement selected state-of-the art algorithms for the LZ77
and based on the experiences design and develop new solutions.
Teaching Goals
• Survey existing algorithms for random access in LZ77.
• Implement and compare selected state-of-the art algorithms for random access
in LZ77.
• Design new algorithms based on experiences with existing state-of-the-art al-
gorithms.
• Implement a prototype of the new algorithms.
