Abstract
General-purpose graphics processing units (GPGPUs) are the driving force behind
many modern scientific applications. As most of the largest supercomputers are ac-
celerated by large numbers of GPGPUs, applications designed for CPU clusters must
undergo refactoring to exploit the accelerators.
Targeting different supercomput-
ers is challenging, as GPU vendors have developed different programming standards.
OpenMP is now supported by multiple vendors making it a good API for writing
portable GPU-accelerated applications.
This project explores programming patterns exploiting accelerators in multigrid
and multiresolution methods using OpenMP. It will be investigated which program-
ming patterns are performance portable across different hardware and compiler ven-
dors. That is important as researchers try bringing their applications to the new
exascale supercomputers.
Further, many of the largest supercomputers have multiple accelerators per com-
pute node, so the applications now have to target heterogeneous compute nodes. As
part of this thesis, an algebraic multigrid method was ported to multiple GPUs and
tuned to give the fastest time to science.
It was found that some compilers support automatic vectorization in OpenMP
target regions while others do not. That relates to how a compiler maps an OpenMP
thread to the accelerator hardware.
If a compiler maps an OpenMP thread to a
warp (or wavefront), the compute kernels may utilize only a fraction of the available
single instruction, multiple data (SIMD) hardware for each streaming multiproces-
sor (or compute unit) on the accelerator. It was found that using OpenMP SIMD
regions within OpenMP target regions adds a level of parallelism that eases writing
performance portable applications in OpenMP targeting GPUs.

Disclaimer
I would like to clearly state that this Master’s thesis builds upon work that I have
done as part of my employment at DTU Compute, as I do not intend to commit
self-plagiarism. Further, I want to ensure everything is clear about what has been
done ahead of the start of this thesis.
Firstly, this study includes a Poisson solver, a direct continuation of work pre-
sented at the 18th Internation Workshop on Open Multi-Processing (IWOMP22) and
published in the conference proceedings [Rydahl et al. 2022]. The implementation
presented at IWOMP was made in C and was based on the Jacobi method. The
one considered in this thesis is written in C++ and implements a geometric multigrid
method.
Secondly, this thesis is also concerned with solving a linear elastic compliance
minimization problem based on an algebraic multigrid preconditioned conjugate gra-
dient. Erik Albert Träff created the CPU version of the code. We submitted a paper
on a single GPU implementation of Erik’s code to Computer Methods in Applied
Mechanics and Engineering journal. Before starting this thesis, I worked on a single-
node multi-GPU version of the same code. The multi-GPU implementation has been
finalized as part of this thesis.

CHAPTER 1
Introduction
A wide range of problems within physics and engineering are concerned with approx-
imating partial differential equations (PDEs).
The numerical simulations of such
systems can require substantial computing resources. Especially when considering
three spatial dimensions, solving sparse, banded linear systems with direct methods
becomes a bottleneck due to memory and time complexity. Hence the engineering
applications need to be based on iterative methods that enable massively parallel pro-
cessing. Further, they should be designed to exploit all available computing resources
as well as possible.
There has been a recent shift towards increasingly heterogeneous architectures
amongst the world’s largest supercomputers. Nine of the ten largest supercomputers
listed on www.top500.org are GPU accelerated. These systems are typically discrete
GPU systems with no shared address space between the CPU (the host) and the
GPUs (the devices) on the compute nodes. Further, the GPUs require a much higher
level of parallelism in the algorithms exposed to them. Therefore numerical simula-
tion frameworks targeting traditional CPU clusters have to be refactored to target
the new exascale supercomputers where most of the computational power lies in the
accelerators. The developers, therefore, need to handle data movement, or mapping,
between the host and the devices within a single compute node. Further, algorithms
unsuitable for GPUs have to be replaced with alternatives. The fact that supercom-
puters are becoming heterogeneous on the intra-node level may also require further
domain decomposition.
The process of adapting old frameworks, or designing new ones, is further com-
plicated because the different hardware vendors provide different GPU programming
application programming interfaces (APIs). To name a few, NVIDIA has Compute
Unified Device Architecture (CUDA), and Advanced Micro Devices (AMD) has the
Heterogeneous Interface for Portability (HIP).
Suppose developers want their applications to be compatible with different hard-
ware vendors. In that case, they either have to branch the development of their ap-
plication to target different architectures or opt for a portable programming model.
OpenMP is a directive-based programming API that many hardware vendors now
support. In addition, since support for accelerators was added in version 4.0 of the
language standard, it has become an integral tool for accelerating C, C++, and FORTRAN
applications. However, the OpenMP support in different compilers varies, and the
framework provides less hardware control than APIs like CUDA or HIP.

1 Introduction
This thesis explores advanced offloading features in OpenMP, considering the
current compiler support. It will present threeC, C++, and FORTRAN applications, re-
spectively. Further, it will be investigated which programming patterns can be used
for performance portability in OpenMP - both concerning hardware- and compiler
vendors. Finally, it will also be investigated how a selected multigrid application can
benefit from a multi-device implementation.
The first application considered is a particle-in-cell Monte Carlo Collision (PIC-
MCC) code for simulating streamer propagation in air.
The application is under
development at the Astrophysics and Atmospheric Physics section at the Technical
University of Denmark (DTU). The framework relies on a multi-GPU Poisson solver
based on the Jacobi relaxation developed and described in [Rydahl et al. 2022]. The
Poisson solver is used to recover the electric potential in every time step of the simu-
lation.
When the spatial resolution of the simulation increases, it becomes apparent that
the convergence properties of the Jacobi relaxation are unsatisfactory, even if the
multi-GPU Jacobi solver was designed to exploit multiple accelerators. This appli-
cation shows that when targeting massively parallel computing systems, it is crucial
to utilize the available computing resources and to use numerical schemes with good
asymptotic time and space complexities where applicable. This thesis is concerned
with deriving and implementing a Poisson solver with linear time and space com-
plexity. The Poisson solver will be developed as a self-contained project and imple-
mented as object-oriented and modular. It will then be shown that the solver can be
integrated into the PIC-MCC code to find the electric Potential eﬀiciently for high-
resolution 3D domains.
The second application is a C topology optimization code for linear elasticity
compliance minimization in 3D. The code was developed at the Department of Civil
and Mechanical Engineering at DTU by Erik Albert Träff and is referred to as Top3d.
The code contains many steps, but solving the elasticity equations is the most com-
putationally challenging one. The elasticity equations are solved with an algebraic
multigrid preconditioned conjugate gradient solver. The original code was previously
ported to one device and will be ported to multiple devices by introducing domain
decomposition. Tuning a multigrid application for multiple GPUs is a complex task
because the problem size on the different grids decays exponentially for each grid.
This suggests that some levels should be solved on the host since the level of paral-
lelism on coarse grids may not be high enough to benefit from multiple accelerators.
On the other hand, an application with a low parallel fraction cannot be expected to
have good strong scaling properties across multiple devices. This work will attempt
to find an approach that gives the fastest time to science, given the compute resources
available within one heterogeneous compute node.
The third application, DISPATCH, is a framework for numerical simulation of as-
trophysical phenomena specifically developed to target exascale supercomputers. The
framework is developed at the center for Star and Planet Information at the Niels

CHAPTER 2
Background
This chapter will introduce concepts and terminology essential for understanding and
implementing scientific programs targeting massively parallel computer architectures.
First, an introduction to the essential hardware components of a supercomputer is
given. Then, it will give fundamental insight into the interoperability of the central
processing unit (CPU) and the general-purpose graphics processing unit (GPGPU).
This chapter also introduces directive-based programming, especially Open Multi-
Processing (OpenMP). This API can enable many different types of parallelism in C,
C++, and FORTRAN applications. A few examples of shared-memory parallelism avail-
able in the OpenMP standard are multithreading, tasking, data parallelism (single-
instruction multiple-data), and, most notably, in this setting, target offloading.
As different compilers support different subsets of the OpenMP language speci-
fication. As the OpenMP runtime differs from compiler to compiler, one needs to
understand the compilation toolchain to get the most out of OpenMP. Especially,
OpenMP target offloading might require a non-trivial installation of the compiler
where the compiler is set up to compile to two different backends.
Further, the
documentation of the compiler installation process might target people with decent
experience with compilers and offloading.
2.1
Heterogeneous Computing
Heterogeneous computing refers to using systems that contain different types of pro-
cessors or cores. A few common processor types are central processing units (CPUs),
graphics processing units (GPUs), digital signal processors (DSPs), accelerated pro-
cessing units (APUs), and field programmable gate arrays (FPGAs). Some of the
processors mentioned above are heterogenous themselves. For instance, an APU typ-
ically contains a CPU and one or more GPUs per chip. Furthermore, a GPU consists
of compute units that may contain several types of cores per compute unit, for in-
stance, integer, floating-point, and tensor cores.
On a larger scale, a cluster or supercomputer consists of many different compute
nodes, and each compute node may consist of several types of processors.
When targeting an application for a supercomputer, one must keep these different
levels of heterogeneousness in mind. The following sections will introduce the types
of processors and distributed systems most important to this thesis.

2.1.2.1
The Single Instruction, Multiple Thread (SIMT) Execution Model
The execution model of GPUs is called single instruction, multiple threads (SIMT).
SIMT differs from SIMD in the way that instructions are all coming from indepen-
dent instruction streams. In SIMT execution, a warp or wavefront is run on SIMD
hardware in lock-step [Aamodt et al. 2018, page 9]. Lock-step execution means that
the entire warp will be executed in parallel. In other words, all threads in the warp
go from one state to another simultaneously in parallel.
For discrete GPU systems, the execution of a kernel generally starts with memory
allocation on the device. After that comes the memory transfer from host to device,
and finally, a compute kernel can be launched. A kernel typically comprises thousands
of threads [Aamodt et al. 2018, page 10]. The reason that a large number of threads
are offloaded at the same time is that it allows for hiding latency when cache misses
are encountered in the shared memory in the streaming multiprocessor [Aamodt et al.
2018, page 4].
SIMT Execution Masking.
The SIMT execution model gives the abstraction
that individual threads execute independently.
The SIMT stack can handle both
skipped computations and nested control flow. If different threads in the warp or
wavefront follow different control flows, it leads to thread divergence.
The SIMT
stack can handle thread divergence but tends to give poor GPU utilization. Due to
the lock-step, all threads in the warp or wavefront must process all branches in the
control flow.

At a high level, a supercomputer is a computer system that can perform many arith-

2.2
Directive-Based Programming
Compiler directives specify how a compiler should process parts of the given source
code. Directives themselves are not part of the grammar of a programming language.
This way, pragmas can be used to extend the functionality of a programming lan-
guage. However, if the compiler does not know how to use the directives, it may skip
them. Therefore, they are also known as pragmas (abbreviation of pragmatic).
It should be noted that different compilers may accept different directives or in-
terpret the directives in different ways.
Several different directive-based GPU programming APIs exist. The most popu-
lar ones are OpenCL, OpenACC, and OpenMP, to name a few. This thesis is only
concerned with the latter.
A motivation for using OpenMP is that many compiler vendors have started sup-
porting OpenMP offload. For instance, INTEL has developed a tool that translates
OpenACC directives into OpenMP directives to make programs portable. In the re-
port describing the migration tool, they state that ”[...] as OpenACC only supports
limited device vendors, there has been a growing interest in adopting OpenMP for
offloading to accelerators, especially now with an increasing number of accelerator
vendors supporting OpenMP offload” [Servat et al. 2022].

2.2.1
Open Multi-Processing (OpenMP)
Open Multi-Processing (OpenMP) is an application programming interface (API) for
shared-memory multi-processing programming. It enables C, C++, and FORTRAN de-
velopers to parallelize program sections. OpenMP is based on the fork-join model,
where a primary thread forks several sub-threads. The sub-threads then run parts of
the program in parallel before joining the primary thread.
From the OpenMP version 4.0, support for accelerators has gradually been added
and improved [Pas et al. 2017, page 253]. However, as this thesis focuses on accelera-
tors, only target-specific features of the API will be mentioned here.
2.2.2
OpenMP Target Offloading
The target directive is used to define a target region in the source code. When a tar-
get region is encountered, it is run by a new thread on the accelerator. The offloading
directives and clauses in OpenMP are generally concerned with either data mapping
or program execution. The OpenMP programming model supports many different
types of accelerators such as Field-programmable gate arrays (FPGAs), digital signal
processors (DSPs), and GPUs, but this thesis will only cover offloading to GPUs.
2.2.2.1

There are generally two ways of mapping variables from the host to a device. Either
runtime functions such as omp_target_alloc(...), and omp_target_free(...) to allocate
and deallocate memory on the device. Then omp_target_memcpy(...) can be used to
move data between any combination of host and devices. Listing 2.1 shows a simple
example of how the runtime functions can be used in C.
Alternatively, the data mapping can be done by using compiler directives such
unsigned int host = omp_get_initial_device();

// Host pointer

// Updating the result from the device to the host

2.2.3
SIMD Clauses
OpenMP can exploit data parallelism on the instruction level with Single Instruction,
Multiple Data (SIMD) clauses. The advantage of SIMD instructions is that they may
be as fast as the scalar counterparts depending on the processor type. For example,
say the vector length is n, then n data elements can be updated in the same number
of cycles as a scalar operation [Pas et al. 2017, page 221].
The simdlen() clause can specify the preferred number of concurrent loop itera-
tions. This clause is only a hint to the compiler, and the hint may be better or worse
than the default value of the compiler.
The safelen() clause can indicate the maximum number of elements that can be
processed in parallel. For instance, if one uses red-black or eight coloring in loops,
it is crucial to set safelen() to avoid data races caused by violation of loop-carried
dependencies.
The data elements may not be aligned with the memory chunks loaded in each
SIMD instruction. If the starting address of the (one-dimensional) array is not a vec-
tor length multiple, the array’s head and tail have to be treated separately. This is
referred to as the peel and tail of the loop iterations, respectively. One may provide
the compiler with hints of the alignment. For instance, if the aligned(x:16) clause is
specified, and x is a double, the compiler may generate 256-bit load and store instruc-
tions if the target supports AVX2 instructions.

2.3.1
Three-Phase Compiler
In a three-phase compiler, the compilation process consists of a frontend, an optimizer,
and a backend. The advantage of the three-phase design is that components of the
compiler can be reused as opposed to a monolithic design [Freire et al. 2011, page
135]. For example, multiple languages can be brought to the exact same intermediate
representation (IR) and reuse the same optimizers and backends. Thereby supporting
L languages and T target architectures requires L + T components instead of LT
compilers.

OpenMP Target Offloading Execution Modes in Clang and Clang++.
The
LLVM compilers support offloading in two modes, guarded mode and single program,
multiple data (SPMD) mode. SPMD is a programming paradigm in which a program
splits a task into small subtasks and runs them in parallel from a single program. In
a way, SPMD is a broader definition of parallelism than SIMT, but it still maps well
to the SIMT execution model.
The guarded mode will be used when the compiler cannot prove if it is safe to
use SPMD mode. Instead, the primary thread of an OpenMP team will run all the
user code and wake up the idle threads when a distribute region is reached [Doerfert

mode. This can be inspected by setting the environment variable LIBOMPTARGET_INFO=16.
Doing so will make the executable print the number of blocks and threads per block
together with the execution mode used to run a kernel.

CHAPTER 3
Theory
This chapter gives an introduction to the most essential of the numerical methods
implemented in the applications. All applications are based on multigrid methods
or multiresolution such as adaptive mesh refinement (AMR). Essentially, multigrid
methods leverage the fact that it is generally faster to approximate a numerical so-
lution to a partial differential equation (PDE) with low resolution than with high
resolution. By iteratively projecting the solution from finer to coarser resolution and
back, it is possible to significantly speed up the process of computing the solution in
high resolution.
Before multigrid methods can be explained in-depth, it is necessary to explain
and derive the essential building blocks needed in geometric and algebraic multigrid
methods.
Section 3.1 throughout Section 3.6 go through the steps necessary to derive a
geometric multigrid Poisson solver for the PIC-MCC code. Firstly, this includes de-
riving a stencil and appropriate boundary corrections, which is done in Section 3.1
and Section 3.2.
Multigrid methods improve iterative methods for solving linear systems of equa-
tions, and hence Section 3.4 outline the Jacobi relaxation, the Gauss-Seidel relaxation,
and successive over-relaxation (SOR). SOR is also crucial to the DISPATCH frame-
work, as the framework implements a conjugate gradient method based on SOR to
solve the discrete Poisson equation.
After that, Section 3.6 dives into geometrid multigrid methods for scalar elliptic
partial differential equations (PDEs).
The fundamental distinctions between alge-
braic and geometric multigrid methods are explained in Section 3.7.
Top3d relies on an algebraic multigrid preconditioned conjugate gradient solver,
and so does DISPATCH. Therefore Section 3.8 outlines the fundamentals of precon-
ditioned conjugate gradient and the correspondence between solving a constrained
quadratic optimization problem and solving linear systems of equations.
Finally, Section 3.9 throughout 3.11 gives and introduction to the elementary
problems solved in the three applications.

3.5.3
Handling Boundary Conditions
Handling non-eliminated Dirichlet boundary conditions on the coarser grids is pain-
less. Since the Dirichlet conditions give the exact solution in the points, which implies
that the Dirichlet condition on the residual is that the residual must be zero on all
points which lie of a Dirichlet boundary.
For the non-eliminated Neumann boundary conditions, the matter is less trivial.
Various sources on multigrid methods, such as [Briggs et al. 2000] and [Trottenberg
et al. 2001], omit to describe how Neumann boundary conditions are imposed on the

coarser grids (or leave it as an exercise to the reader). Therefore, the solution used in
this project will now be outlined. It may or may not be optimal, but it will be shown

for the test problems outlined in Section 3.3.
On points which are part of a Neumann boundary, the gradient can be approxi-
mated. Then, the difference between the Neumann condition and the approximated
gradient on uh can be used as Neumann condition on the defect e2h, meaning that
the defect must add a contribution to uh that cancels out the difference between the
approximated gradient and the Neumann condition.

3.7
Algebraic Multigrid Methods for Scalar Elliptic PDEs
The majority of the running time in the application Top3d is based on solving a linear
system of equations with an algebraic multigrid (AMG) preconditioned conjugate-
gradient method. As the literature on AMG methods for non-scalar PDEs is limited,
and since a complete understanding of the method is optional for the work done in
this thesis, this section only provides a superficial overview of AMG methods.
Like geometric multigrid methods (GMG), AMG is developed to solve the linear
system Au = f. The main distinction from GMG is that the multigrid hierarchy
does not have to be known before deriving a scheme for algebraic multigrid methods.
Hence they are more general and may also work for a predetermined hierarchy of grids.
That means that instead of basing the scheme on a specific grid Gh and a coarser grid
GH where it is typically satisfied that H = 2h, AMG methods are defined on a
neighborhood around a point. For AMG methods Galerkin operator is typically used
for both restriction and prolongation [Trottenberg et al. 2001, page 416].

3.11.1
Basic Terminology
DISPATCH is written in object-oriented FORTRAN (2003). The most important
classes are tasks, patches, dispatchers, and solvers. A task describes some computa-
tion steps that need to be done. A patch extends a task with a mesh, for instance, a
Cartesian or curvilinear grid, a local timestep, and several other variables and meth-
ods. Solvers are then implemented to update tasks, and dispatchers contain the logic
to execute task updates, fill guard zones, and distribute tasks to other MPI ranks.

3.11.2
Adaptive Mesh Refinement
Figure 3.7: Example of adaptive mesh refinement in 3D based on an octree data

The DISPATCH framework is based on adaptive mesh refinement (AMR). AMR al-
lows the program to have different resolutions in different parts of the domain and
to refine or de-refine subdomains as simulation time progresses. In 2D, each patch is
split into four sub-patches when a refinement criterion is met. In 3D, it is divided
into eight. This results in a tree data structure which is referred to as quadtree and
octree, respectively. It is also possible to split a patch into nine or 27 patches in 2D
and 3D, respectively.
A maximum number of levels is set, typically Lmax = 20, such that the factor
between the grid spacing on the coarsest and finest level is at most 2Lmax ≃1.05·106.
The framework requires that there must be at most one level of difference in refinement
between adjacent patches because this allows for generic ways of interpolating guard
zones.
The AMR imposes a load imbalance issue, as a higher resolution in some parts of
the domain implies more operations in that subdomain (or patch). Patches can be
dispatched to other MPI ranks to avoid load imbalance. In combination, this requires
a complex procedure for AMR. For instance, assume a patch on level L + 1 is refined
to level L+2. This requires all of the new patches on level L+2 to interpolate guard
zones from adjacent patches on level L + 1. Since patches can be exchanged between
MPI ranks, a patch on level L owned by another must be refined. Algorithm 7 states
the procedure used to refine a task (and thereby also patches) in DISPATCH. Note
that there is some delay between when a task is marked for refinement and when it is

3.11.3
Poisson Solvers in DISPATCH
The module poisson_mod.f90 in DISPATCH contains several methods for solving
the Poisson Equation in different coordinate systems. It contains two preconditioned
conjugate gradient (PCG) variants and successive over-relaxation (SOR). The first
version of PCG uses implicit FORTRAN loops, while the other uses explicit loops.
The integral form of Gauss’ Law states that the flux of the electric field out of a
closed surface is proportional to the electric charge enclosed by the surface. Thus the
specific distribution of electric charge inside a volume does not affect the electric field
outside this volume. This allows the solvers in DISPATCH first to solve the potential
on a patch and refine the solution on the child patches. Hence there is a dependency
from parent to child patch but not from child to parent patch.
Essentially, this allows for using relatively simple methods for solving the Poisson
equation on each patch. A patch typically has between 10 to 26 interior cells and 6
ghost cells per dimension. The Poisson equation is typically solved on subdomains
with 163 to 323 cells per patch. These domains are so small that reaching convergence
within a subdomain takes a fairly short time, even with lower-order methods such as
the Jacobi method or SOR.

Implementation
This chapter will describe implementation-specific details and programming patterns
used for the three applications. Section 4.1 describes how a geometric multigrid Pois-

The topology optimization code is described in 4.1.6. The application underwent sev-

4.1
Multigrid Poisson Solver for the PIC-MCC Code
Following the convergence analysis presented in Section 3.4.3, the naïve multi-GPU
Poisson solver based on Jacobi iterations presented in [Rydahl et al. 2022] does not
converge in a reasonable time for large domains. However, Jacobi and other relax-
ation schemes are building blocks in a multigrid solver.
The previous solver was written in C but could be used in the PIC-MCC code by
compiling it separately and including it with extern C {...}. As the PIC-MCC code is
written in C++, the new Poisson solver was also implemented in C++. That allowed for
using abstract classes, inheritance, and templating, which helped develop a modular
framework that can easily be extended with more multigrid components.
The new Poisson solver was designed for the topology of LUMI. Since there are
direct links between the GPUs, and since there is more device than host memory
on the compute nodes on the small-g nodes, OpenMP runtime functions were used
for data allocation and data transfers. This has the advantage that arrays are not
duplicated on host and device and that the device pointer is known.
As shown in Section 5.5, tuning a multi-GPU multigrid solver is challenging be-
cause the domain size decreases as a logarithmic factor for each grid. Hence a multi-
grid single-GPU implementation seemed as a reasonable starting point.
W-cycles and F-cycles have worse parallel eﬀiciency than V-cycles, according to
U. Trottenberg et al. [Trottenberg et al. 2001, page 196]. The short explanation is
that relatively more iterations are performed on the coarse grids where communica-
tion with other ranks is the bottleneck. The same argument holds for SOR, which
==PAGE:74==
58
4 Implementation
has better smoothing properties than the Jacobi method since each MPI rank has
to exchange halo values twice per iteration, doubling the communication overhead
on the coarsest domains [Trottenberg et al. 2001, page 196]. Both V and F-cycles
are implemented as part of this thesis, but it should be emphasized that the simpler
V-cycles have the best parallel complexities.
4.1.1
Matlab Implementation
Before developing the Poisson solver, a Matlab reference implementation was made to
experiment with the different components of the multigrid algorithm. This reference
implementation was convenient for testing whether an exact solver could be avoided
on the coarsest level. However, for the three test problems, using the Jacobi solver on
the coarsest level still gave convergence in a few multigrid iterations for all feasible
test sizes. Therefore, it was decided not to rely on a direct solver, which could have
introduced steps in the algorithm that could not be GPU ported.
4.1.2
Design Choices of the C++ Implementation
Templating, class inheritance, and abstract classes (interfaces) were used to develop
a solver that could easily be modified and extended. Templating made it possible to
make a solver that would work for different floating-point types.
Another advantage of the object-oriented design was that, in combination with
non-eliminated Neumann boundary conditions, there was no need to write separate
kernels for each boundary and separate ones for corner points as we did in the solver
from [Rydahl et al. 2022]. For instance, the Jacobi iteration could be implemented in
only three kernels, whereas 11 kernels had to be implemented in the paper. Thereby
much boilerplate code was avoided, which made it easier to get a correct solver without
dedicating too much time to the debugging phase. However, the kernels could be more
eﬀiciently implemented if 11 had to be made.

4.1.9.1
Strategies for Targeting Multiple Accelerators in Parallel
The experiments which were carried out in feasibility Studies in Multi-GPU Target
Offloading [Rydahl et al. 2022] were mainly conducted to find a good strategy for
overlapping data transfers and kernel execution on multiple devices with the NVIDIA
nvc compiler. It was found that embedding target regions in tasks led to a low level
of idle time on the accelerators. Hence, this programming pattern was adapted for
top3d. In other words, the task-based programming pattern from Listing 2.7 was
used.

the OpenMP device runtime in gcc is synchronous, using the programming pattern
from Listing 2.7 introduced overhead since kernels were launched more times than
necessary. Hence, to improve the performance in gcc, the naïve programming pattern
from Listing 2.6 was used instead. In the end, clang gave better results on LUMI,

GPU Porting the CG_GV Poisson Solver in DISPATCH Using

This section explains how the preconditioned conjugate gradient (PCG) solver from
DISPATCH has been GPU ported.
The solver is named CG_GV and has been
described in pseudocode in Algorithm 8.
The PCG solver supports both Dirichlet and periodic boundary conditions. The
preconditioner is based on a nodal approximation of the inverse of the iteration matrix.
The original CPU version supports both Cartesian and curvilinear grids, but it was
chosen only to GPU port the components for the solver needed for Cartesian grids.
The PCG Poisson solver from poisson_mod.f90 was GPU ported, and the new
GPU solver was named cg_device.
The FORTRAN standard does not allow for a
conditional compilation like C and C++, but in gfortran, the C preprocessor can be
used by compiling with -cpp. In DISPATCH, all the GPU ported applications are
implemented as directive conditionals #ifdef OFFLOAD ... #endif.

4.2.2.1
Bunching of Patches
The good weak scaling properties of DISPATCH stem from the design choice of using
many small patches of work that can easily be exchanged between processors. While
this is a clever way to handle the work imbalance that naturally occurs due to adap-
tive mesh refinement, this design choice only naturally maps well to CPUs. Also, the
OpenMP support for mapping complex data structures to the GPUs is limited.
Because the gfortran GPU runtime is synchronous, and because gfortran is the
only compiler that can compile DISPATCH, it was not possible to use a streaming
implementation where small chunks of data would continuously be sent to and pro-
cessed by the device. Instead, the patches had to be bunched together and processed
by the device in larger chunks.
Other parts of the DISPATCH framework, such as the MHD solver, have already
been GPU ported with bunching. The contribution from this thesis was not integrated
into the bunching mechanism of DISPATCH but instead tested separately in a small
mockup where a fixed number of tasks were executed in parallel on the GPU. The
CG solver was, however, integrated into the Poisson solver module of DISPATCH.
4.2.2.2
Mapping the Patch Data Structure to the Device
It would have been practical to implement a specialized mapper for the patch_t
class.
It was attempted to map the patch sizes to the device by #omp declare
mapper(class(patch_t)::p) map(to:p%mesh%lb,p%mesh%ub,...). However, it was
found that gfortran 13 does not support this feature from the OpenMP 5.2 specifi-
cation at the time of writing.
Instead, scratch arrays were used to store information such as patch size, bound-
ary type, and grid type from the patches needed by the preconditioned conjugate
gradient solver on the device. Scratch arrays were also used to store the potential
and source terms for the patches.
4.2.2.3
Increasing the Level of Parallelism
In the baseline CPU implementation of PCG, many implicit FORTRAN loops existed.
This implementation was designed for one host thread solving the Poisson equation
on a patch. When targeting a GPU architecture, any serial part of the application
will become a bottleneck. The implicit loops were replaced with explicit loops that
could be parallelized with OpenMP directives to increase the level of parallelism.

4.2.2.6
Compilation Issues
For this application, the compilation problems were a big challenge.
First, DIS-
PATCH can only be compiled with gfortran or other GCC compilers. Hence it was
not possible to attempt to use other compilers.
It was found that all optimization levels greater than -O0 resulted in a crashing ex-
ecutable that exited with signal CUDA_EXCEPTION_5, Warp Out-of-range Address.
It was attempted to make a version running only one thread per team, resulting in the
same error. By enabling -g -gdwarf and using cuda-gdb, it was found that the error
occurred in functions that had been optimized with forward propagation of constants.
In gfortran, forward propagation is enabled at -O1, and therefore, optimizations had
to be disabled with -O0.
Further, when adding simd directives in the target region, it resulted in an inter-
nal compiler error.

When first integrating the new solver in the PIC-MCC code, it was found that
the new multigrid solver did not converge. That led to reiterating the convergence
experiments from Figure 4.2 with more test problems. In the first attempt to inte-
grate the solver, a solver based on V-cycles, injection, and trilinear interpolation was
used. However, this solver did not converge to a fine tolerance, and the Poisson solver
framework was extended with full weighting interpolation and red-black Gauss-Seidel,
which could easily be implemented due to the use of abstract classes.
When this still did not converge to a fine tolerance, the PIC-MCC code was de-
bugged, and it turned out that there was a bug in the particle mover that would
place all contributions to the source term in one boundary point where the Neumann
conditions enforced a zero gradient.
From this observation arises the question of fairness of how to measure the speedup
of the new solver. Hence this section will not include a speedup figure for the entire
application and instead focus on the Poisson solver alone.
5.4.1
Hyperparameter Tuning
One of the important hyperparameters to tune is the number of relaxation steps per
iteration. The relaxation is the most time-consuming step in the V-cycle implemen-
tation for Jacobi and Gauss-Seidel operators. Figure 5.5 shows the running time to
solve the electric potential for the first time in the PIC-MCC code. It can be seen
that for this implementation, using Jacobi relaxation is generally faster than red-
black Gauss-Seidel overrelaxation. Depending on the Domain size, the figure shows
that four or five Jacobi relaxations give the fastest convergence to the tolerance. The
spike at 1 for red-black Gauss-Seidel relaxation is caused by the fact that the V-cycle
does not converge in this case. Intuitively, this makes sense that the method does not
converge since the smoothing parameter ω = 1.15 used in the experiment corresponds
to overrelaxation.
Note that the first iteration is, depending on the initial guess, the slowest iteration
as the solution does not change rapidly over time. Hence, the subsequent iterations
require only one or a few V-cycles to converge.
5.4.2
Mixed Precision
Table 5.2.a shows that the NVIDIA Tesla A100 GPGPUs have 6912 FP32 cores and
3456 FP64 cores. Further, arithmetic, load, and store operations take shorter for 32
bit floating points. However, there is a tradeoff between reduced running time and
the accuracy of the solution. As the Poisson library has been templated, one can
easily make two solvers with different floating point precision. Figure 5.6 shows the
relative residual as a function of time for two different V-cycles, one based on 32-bit
precision and one based on 64-bit precision. It can be observed that the 32-bit solver
converges faster than the 64-bit solver, but it converges to a relative residual in the
order of ∼101. In contrast, the 64-bit solver reaches a relative residual of ∼10−9.

CHAPTER 6
Discussion
This discussion will be given in sections as this thesis has dived into three different
applications and searched to compare different programming patterns for OpenMP
target offloading.
6.1
Tuning Multigrid Algortihms for GPUs
The Poisson solver from the PIC-MCC code is a geometric multigrid solver. Top3d’s
elasticity equations are solved with an algebraic multigrid preconditioned conjugate
gradient solver. However, even though it implements an algebraic multigrid method,
the grid is based on an equidistant Cartesian discretization, which makes the two
applications less dissimilar.
Tuning multigrid methods for GPUs is challenging because the kernels need to
process inputs of very different sizes. Ideally, hyperparameters should be tuned for
each domain size, but that is not feasible in practice, as the optimal hyperparameters
vary across compilers and hardware. One way to deal with this is to solve the coarsest
domains on the CPU, which was done in Top3d. However, this does not result in a
large parallel fraction of the code and thereby severely limits the strong scaling of the
application.
Top3d deals with multiple accelerators, making tuning less trivial. Further, some
steps in the AMG method were found unsuitable for GPUs. As the stencil assembly on
matrix levels required sparse matrices in their current form, this step was impossible
to offload to multiple accelerators.
Another thing that is interesting to discuss in terms of Top3d is accuracy. The
original takes conjugate gradient steps until a relative tolerance of ϵ = 10−6 is reached.
However, the output of the simulation will undergo much postprocessing. One step
in postprocessing is a simple threshold on the density. It is worth mentioning the
application runs many times faster if the relative tolerance is decreased. It may be
problem dependent, but from the experiments performed in this thesis, it appears
that the simulation converges towards the same solution if the tolerance is increased
many orders to, for instance, ϵ = 10−4.

6 Discussion
6.2
Comparison of Compilers
Previous publications such as Outcomes of OpenMP Hackathon: OpenMP Application
Experiences with the Offloading Model (Part II) by Chapman et al. [Chapman et al.
2021] found that nvc could not compile their mini-app and that the target code ran
∼21 times slower when compiled with gcc compared to clang. The performance tun-
ing from Section 5.5 points to that it is possible to get gcc to match the performance
of an LLVM-based compiler such as nvc. For Top3d, it was necessary to add SIMD
directives in the target regions and to select a block size that matched the number of
SIMD cores in a compute unit/streaming multiprocessor.
On the other hand, it is clear that even if nvc is to some extent compliant with
the OpenMP 5.0 standard, it is hard to write code with OpenMP target offloading
regions that will compile with nvc. Therefore, if one wants to write OpenMP code
that is compatible with many compilers, one has to decide to use programming pat-
terns that are compatible with nvc. For instance, it is necessary not to nest target
regions in tasks.
On the other hand, nvc/nvc++ can be downloaded and installed in a trivial way.
It also offers good target offloading performance compared to the gcc compiler, where
more tuning is typically required to achieve acceptable performance.
While requiring more knowledge of compilation infrastructure to install an offloading-
enabled compiler, gcc, on the other hand, works reliably across many different plat-
forms. gcc was the only compiler that could compile all three applications in this
thesis, both in FORTRAN, C, and C++.
Comparing clang/clang++ to gcc/g++, it is clear that the GPU kernels are more
eﬀicient when compiled with clang/clang++ due to the automatic SIMD generation
in target regions. A huge performance gap between the two compilers is to be ex-
pected if one does not use SIMD directives in the target region. On the other hand,
this gap can be dramatically reduced by adding SIMD instructions in the target
region, as shown in Section 5.3.1.
6.3
Writing Performance Portable Applications in
OpenMP
Compilers were found to map OpenMP threads to the device architecture in different
ways. Some compilers, such as gcc/g++, map an OpenMP thread to a warp, while
others map it to a hardware thread. That means that the executables generated with
some compilers perform better when an initial thread runs in a warp and fork another
team of threads, for instance, with a nested OpenMP SIMD region in a target region.
However, other compilers such as clang/clng++ do not support embedding a SIMD
region in a target region. Ultimately, this requires a programmer to be able to switch
between the programming patterns from Listing 2.4 and 2.5.

6.4
Data Mapping
In the three applications considered, the data mapping is done differently. In DIS-
PATCH and in Top3d, the mapping directives are used, while the multigrid Poisson
solver from PIC-MCC relies on OpenMP runtime functions.
In FORTRAN, the data mapping clauses are much more user-friendly than in C be-
cause of the object-oriented nature. One does not need to write multi-line pragmas
to map a single array like in Listing 4.6. It cannot be ruled out that the problems
with compiling Top3d in clang are related to the data scoping.
The OpenMP runtime functions for mapping are easier to use and allow the pro-
grammer to allocate memory on the device without duplicating it on the host. This is
a big advantage for some applications. For instance, it would have made sense to use
the mapping runtime functions in Top3d. In Top3d, one bottleneck in achieving fast
convergence for large domains is that the host runs out of memory when computing
the Cholesky factorization of the coarsest domain. Thereby, one has to select enough
multigrid levels so the host does not run out of memory.
For the multi-GPU version of Top3d, it turned out to be a major design flaw
that the directive-based mapping was used.
On the Aldebaran node (from the
small-gpartition on LUMI), only 408 GB of memory is available on the host but
512 GB on the devices in total. Hence, the application was limited by the host mem-
ory because the directive-based data mapping requires the arrays to be duplicated
on the host and device. As a result, the largest feasible domains on the Volta and
Aldebaran nodes were approximately the same size, even though the total amount
of device memory on the Volta node was only 128 GB compared to the 512 GB of
available device memory on the Aldebaran node.
Another disadvantage of directive-based data mapping is that the OpenMP API
does not offer functionality for directive-based device-to-device communication.
6.4.1
Geometric Multigrid Poisson Solver for Particle-in-Cell Monte
Carlo Collissions
The geometric multigrid method developed in this report is a significant contribution
to the PIC-MCC code as it allows the researchers to run simulations on previously
infeasible domain sizes due to the previous solver’s poor convergence properties. On
an NVIDIA Tesla A100 GPU, a simulation domain with more than one billion grid
points can now be solved for the current code version in less than 10 seconds. Solving
the Poisson equation so fast would be impossible with any direct solver due to the
asymptotic complexity of any direct methods listed in Table 3.1. However, the solver
may be insuﬀicient when the code is extended to target multiple compute nodes with,
for instance, MPI. However, implementing the solver in an object-oriented manner,
using abstract classes, ensures that adding another boundary condition should be
possible. For example, adding an interior boundary class that fetches ghost points
from another MPI rank would probably not be too complicated to implement.

6.4.2
Topology Optimization
Ideal strong scaling was not achieved when porting the Top3d code to multiple accel-
erators. That is primarily due to three problems that were encountered. First of all,
the current stencil assembly operation has non-coalesced access patterns, and because
the stencil assembly is run on a sparse matrix, it is not trivial to partition amongst
multiple accelerators. Since this part of the application was not GPU ported, the
parallel fraction of the code remained around only 79 percent. The parallel fraction
could have been increased to 95 percent by using only matrix-free layers, but this
would not have given faster time to science.
It was found that clang provided much better strong and weak scaling than gcc
and nvc.
Using multiple accelerators in parallel did not provide any speedup on
LUMI in gcc. Using eight GPU dies gave a 4.6 times speedup in clang of the GPU
ported code. That was true for a domain of size 1024×512×512 voxels. The speedup
achieved with gcc was, however, not far from the speedup achieved with clang on
the Volta node.
The final multi-GPU version of Top3d suffers from first optimizing the code for
gcc. As the access to LUMI was granted late in the development phase, it was impossi-
ble to know in advance that the application would achieve so much worse performance
when compiled with gcc on the AMD-based system than on the NVIDIA systems. If
it had been known in advance that clang would be the only performance portable
compiler in the case of Top3d, many more kernels could have been overlapped, and
better strong scaling could probably have been achieved. If the application was to
be done over from scratch, it would have made more sense to use the task-based pro-
gramming pattern from Listing 2.7 as that allows for better overlap of data transfers
and compute kernels.
Even if the code did not achieve an ideal speedup, the code is still the fastest
available version of the Top3d code. Further, due to the extended amount of GPU
memory that can be utilized, the code scales to domains way beyond the limits of the

6.4.3
GPU Porting Poisson Solvers in DISPATCH
Six mini-apps for solving the Poisson equation were considered. Two were classical re-
laxation schemes, and four were different variations of the conjugate gradient method.
The experiments presented in Figure 5.19 showed clear potential for leveraging the
compute massively parallel compute resources available on a GPU. The experiment
with the mini-apps showed that large speedups could be expected. For the precondi-
tioned conjugate gradient algorithm from Algorithm 8, the mini-app achieved almost
ten times speedup relative to the CPU baseline on the compute node with NVIDIA
Tesla V100 GPUs. However, none of the mini-apps performed similarly on the AMD
Instinct MI250X GPU. For that architecture, the fastest mini-app was approximately
eight times slower than the fastest on the NVIDIA GPU-based platform. However,
the results could have been different if more time had been spent tuning the mini-
apps.
The mini-apps ignored that different patches in the DISPATCH framework may
impose different boundary conditions and should support different types of coordinate
systems. Further, the experiment did not consider that a full bunch may not always
be marked ready. Hence, the speedups measured for the mini-apps are to be regarded
as the upper bounds of the performance that can be expected when integrating the
mini-apps into DISPATCH.
The only FORTRAN compiler that can compile DISPATCH is gfortran.
Since
gfortran does not support asynchronous operations on the GPU, and because the
computational work needed to solve one patch is small, the patches must be solved
in a bunch.
It was attempted to GPU port the CG_GV solver from DISPATCH. The solver
could run on the GPU and produce correct results for periodic and Dirichlet boundary
conditions. However, the performance was unacceptable, as it ran 17 times slower
than the CPU version of the solver. The primary explanation for the poor perfor-
mance of the solver was that gfortran resulted in an internal compiler error as soon
as SIMD regions were nested in target regions. Further, the executable would crash
if compiled with any optimization level greater than -O0. The solver was believed to
be correctly implemented because it produced correct results and performed similarly
to the CPU baseline version when the solver was used on the host.

Related Work
7.1
Particle-in-Cell Monte Carlo Collision (PIC-MCC)
Many different hybrid codes mixing particle schemes with finite volume or finite dif-
ference to simulate streamer propagation in air exist. However, it is considered out
of the scope of this thesis to compare the framework under development with the ex-
isting ones. As this thesis has implemented a Poisson solver, the discussion of related
work will be limited to how the different streamer codes solve the Poisson equation
for the electric potential.
The framework developed by Chanrion et al. in A PIC-MCC code for simula-
tion of streamer propagation in air Chanrion et al. 2008 simulates streamers in two-
dimensional axisymmetrical coordinates to emulate three spatial dimensions. This
choice allows the code to rely on direct methods for solving linear systems. The Pois-
son equation is solved by solving the linear system with the Cholesky decomposition
routines from the LAPACK library. As demonstrated earlier, the Cholesky decomposi-
tion is unsuitable for solving large linear systems arising from discretization in three
spatial dimensions. Although a GPU-based Cholesky decomposition method has been
implemented as part of the Cholmod library from SuiteSparse, this solver has been
implemented using CUDA and is therefore not compatible with AMD systems such as
LUMI.
Streamers can be simulated in three spatial dimensions with the code developed
by Li et al. in Spatially hybrid computations for streamer discharges: II. Fully 3D
simulations [Li et al. 2011]. This code solves the Poisson equation with a solver based
on cyclic reduction with Fast Fourier Transformation (FFT) that comes as a part of
the FISHPACK FORTRAN library. However, the FISHPACK library has not been GPU
ported and is therefore not taking advantage of the accelerators available on modern
HPC systems. Further, the solver developed as part of this thesis solves a mixed
boundary value problem MBVP). That leads to a non-symmetric discretization, and
therefore, standard FFT cannot directly be applied to this problem, although varia-
tions of FFT meant for MBVPs exist.
Some works solely focusing on solving the Poisson equation in 3D using GPUs
exist. For instance, D. Jacobsen et al. [Jacobsen et al. 2011] use a geometric multi-
grid method based on finite difference.
The components supported are red-black
Gauss-Seidel or Jaobi relaxations, and full weighting and trilinear interpolation for
restriction and interpolation, respectively. So far, the code is similar to the imple-

mentation in this thesis, but it is implemented in MPI + CUDA and is not publicly
available. They reported that using 64 NVIDIA Tesla S1070 GPUs, their solver could
solve a domain of size 10253 in 60 seconds back in 2011. However, since the NVIDIA
Tesla A100 GPU architecture is many generations newer, it is hard to compare the
performance.
A more recent example is by N. Onodera et al. [Onodera et al. 2021], which solves
the Poisson equation on a Cartesian grid using a geometric multigrid preconditioned
conjugate gradient. A similarity between this work and the solver in this project is
that it also is implemented for mixed boundary conditions. Further, it is based on
a second-order accurate centered finite difference scheme. Their implementation is

code is not too bad compared to a CUDA implementation as the PIC-MCC test takes
around 12 to 13 seconds to solve a problem of size 1025×1025×1025 with an F-cycle.
7.2
Linear Elastic Compliance Minimisation in Three
Dimensions
First and foremost, Wu et al. implemented a single-GPU high-resolution solver for
compliance minimization in 3D in A System for High-Resolution Topology Optimiza-
tion [Wu et al. 2016]. This framework demonstrates that they can run 50 iterations
of a 200 × 100 × 100 voxel cantilever example in approximately 145 seconds using
an NVIDIA Tesla C2070, which is an older generation of graphics card compared
to those considered in this thesis. The solver developed by Wu et al. is also based
on SIMP and solving the state equation with a multigrid (V-cycle) preconditioned
conjugate gradient. Also, the stencils are assembled for each node, and the restriction
is based on the Galerkin projection. Some of the key differences are that the V-cycle
preconditioner is a geometric multigrid method and that the kernels are written in
CUDA. Unfortunately, the solver is not publicly available, and since the tolerances
and other hyperparameters are not listed for the cantilever experiment, it is not pos-
sible to directly compare the running time of the two implementations.
The most similar work available compared to the multi-GPU implementation de-
scribed in this thesis is probably the solver developed in Multi-GPU acceleration of
large-scale density-based topology optimization by Herrero-Perez et al. [Herrero-Perez
et al. 2021]. The solver is based on an algebraic multigrid preconditioned conjugate
gradient and SIMP. They target GPUs with OpenCL. The domain decomposition is
more advanced than the one considered in this method. They generate a dual graph
of the mesh and use an optimization criterion to find a decomposition with a minimal
amount of halo exchange. Further, the grid is not restricted to being a Cartesian grid.

CHAPTER 8
Conclusion
Writing portable GPU code is complex because many APIs exist for porting applica-
tions to GPUs. Some popular APIs, such as CUDA, can only be compiled to target
NVIDIA GPUs. Many GPU vendors now support OpenMP target offloading, making
OpenMP a suitable choice of GPU programming API for porting scientific programs.
This project tried to find programming patterns that ensure some level of per-
formance portability. This project found that introducing blocking and embedding
SIMD directives inside target regions improved performance substantially for com-
pilers that do not automatically generate SIMD instructions. These compilers map
OpenMP threads to a warp/wavefront rather than a hardware thread. Adding this
optimization to all kernels in the geometric multigrid Poisson solver gave a ∼11.2
times performance improvement for the V-cycle solver and ∼8.7 times for the F-
cycle solver when using the experimental g++ 13 compiler. It also gave a 27 and 20
percent speedup for V- and F-cycles, respectively, for the experimental Clang 16.0
compiler, which has automatic SIMD generation. After the optimization was added,
g++ and clang++ gave a comparable performance, although clang++ gave a better
performance.
Applying the same optimizations to just a few of the kernels in Top3d gave a ∼7.0
times speedup for the entire application for the gcc compiler giving the code identical
performance in both g++ and nvc. The design mentioned above is an important find-
ing because it allows scientists to design code that can potentially run on different
supercomputers and give good performance after tuning a few hyperparameters.
In conclusion, this project has demonstrated that nontrivial numerical methods
such as algebraic and geometric multigrid methods can achieve significant speedup
from OpenMP target offloading to one or multiple GPUGPUs. However, in some
cases, the compiler maturity is a challenge that limits the potential of OpenMP tar-
get offloading. Even though the two applications leveraged from offloading to devices
with OpenMP, it was impossible to achieve acceptable performance for the precondi-
tioned conjugate gradient method from DISPATCH when compiling the framework
with gfortran 13.
The multi-GPU version of Top3d allows researchers to run topology optimization
code for linear elasticity compliance minimization in 3D on domain sizes that were
previously impossible to solve on a single compute node. However, not all parts of
the algebraic multigrid preconditioner in Top3d were equally suited for the massively
parallel GPU hardware, which limited the strong scaling of the application. With
==PAGE:138==
122
8 Conclusion
clang, the simulation of a domain of size 1024 × 512 × 512 still achieved 3.86 times
speedup using four AMD MI250X (eight GPU dies) on LUMI compared to using one
GPU die. For a domain of size 768 × 384 × 384 voxels, it gave a 2.52 times speedup
using four NVIDIA Tesla V100 compared to one of the accelerators. It was also found
that gcc provided a proper strong scaling on the compute node with four NVIDIA
Tesla V100 GPUs while using more than one accelerator slowed down the application
on LUMI. The reported speedups are not for the combination of matrix-free and ma-
trix levels that would give the largest speedup but for the combination that gave the
shortest time to science.
Altogether, the tuning of the two multigrid applications shows the synergy of
selecting better mathematical models and leveraging hardware acceleration.
