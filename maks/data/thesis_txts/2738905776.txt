Abstract
This thesis focuses on the oFEC standard, an advanced FEC technique designed for
long-reach optical networks, initially targeting 200G-400G ZR+ and OTN transmission
lines, and now being considered for 800G and beyond scenarios. The research aims to
design and implement an oFEC encoder capable of processing 1016 bits efficiently.
The thesis outlines the design and implementation of a Chase-based SISO decoder in
VHDL, achieving correct behavior that is consistent with similar SISO decoders. The
work then moves to the development of the full oFEC decoding system, discussing ways
to properly design the structure of both the interleaving memory and the iterative config-
uration needed to decode, before moving to implement it in VHDL.
Despite substantial work to optimize the design, challenges in routing higher-speed con-
figurations were met. A single-lane decoder operating at approximately 25G was suc-
cessfully implemented and tested. However, behavioral simulations revealed that while
the system performed well under low noise conditions, it struggled with higher noise lev-
els, leading to frequent decoding errors.
This thesis finally concludes that while the current implementation meets the initial objec-
tives, further refinements are essential to fully realize the potential of oFEC in high-speed,
long-reach optical networks. With targeted improvements and comprehensive testing, the
feasibility of deploying oFEC in practical, full-scale testing environments capable of gen-
erating, processing and verifying 1016 bits is within reach.

Introduction
The telecommunications industry is constantly pushing the boundary on data transmission
speed, trying to push more and more data through the use of high-density pluggable
transceiver modules that run multiple lanes in parallel. This results in signals that are
less resistant to noise. Large data centers are key contributor the this need, and their
interconnects come with an additional stipulation of transmission line length [1].
To support these high-speed and long reach interconnects, several standards and agen-
cies have been pivotal. The Optical Internetworking Forum (OIF) introduced the 400ZR
standard, which is designed for data center interconnects over distances up to 120 km
[2]. Later, OIF defined the ZR+ standard, which extends the capabilities of 400ZR to sup-
port even longer distances. The International Telecommunication Union (ITU) has also
established important standards, such as the ITU-T G.709.3, which defines long-reach in-
terfaces for OTN [3]. Additionally, the openROADM initiative aims to standardize interop-
erable and scalable optical networking solutions, promoting multi-vendor interoperability
and network automation.
Most of these standards involve, or are entirely based around, Forward Error Correction
(FEC). When noise increases in a channel, bit errors in the transmitted messages are
bound to occur. If a message is erroneous, and therefore cannot be read, a retransmission
occurs. Retransmissions consume a large amount of bandwidth, and some transmission
channels may even be completely unreliable, affecting the channel performance. FEC
can allow the receiving party to correct some errors in the message, and is therefore
crucial in high-speed communication.
1.1
ITU Recommendations
The oFEC standard is a promising FEC encoding technique [4] meant for long reach
optical networks. It is defined by Open ROADM [5], based on decoding requirements
from [3]. Originally meant for 200G - 400G ZR+ and OTN transmission lines, it has been
determined that it is applicable for 800G scenarios, and talks regarding 1200G and 1600G
scenarios mention it as a candidate [4].
ITU defines several key performance metrics for error correcting capabilities in these high-
speed scenarios. An error correction threshold at 2·10−2 is required, as well as the ability
to attain Bit Error Rate (BER) outputs of as low as 10−15 in specific conditions. This can be
seen in Figure 1.1, where the requirements of a properly implemented code’s performance
is visualized. These requirements apply for both QPSK and 16QAM modulation [3].
1.2
Research Objectives
These metrics must be achieved by oFEC, but they have not been fully verified yet. To
determine the feasibility of achieving the desired error rate, a substantial amount of data
— specifically 1016 bits — must be processed.
The goal of this thesis is therefore to further the investigation into oFEC, and thus the final
goal is to design and implement an oFEC encoder that is able to process this amount of
data within a reasonable time frame.

Error Correcting Codes
Data transmission is typically done over unreliable or noisy channels, but comes with strict
requirements for optimizing channel capacity and efficient use of bandwidth. Because
of this, forward error correction (FEC) is a much-studied field in telecommunication and
information theory.
FEC is a technique for detecting or even correcting errors in received information se-
quences. This is done using error correcting codes, which limit the valid information se-
quences to precise patterns. The transmitting side encodes the information by adding
specific redundant information bits to the sequence, called parity bits, forming the error
correcting code, typically called a codeword. With these, the receiving party is able to
mathematically correct some number of bit errors, provided that there are not too many
in the received word.
An important metric in these codewords is the Hamming distance between two valid code-
words in a code C, say v and w, which describes the minimum number of positions in
which any two valid codewords differ [6]:
dmin(C) = min (d(v, w) : v, w ∈C, v ̸= w)
The greater the Hamming distance is, the more capability for error detection and correc-
tion, but this also requires more parity bits. There is therefore a trade-off between the
amount of redundancy and error correction capabilities.
To illustrate Hamming distance, a simple repetition code can be used. A repetition code
repeats each bit in the transmitted sequence a number of times, i.e. a 0 becomes a
sequence of 0’s and vice versa. If each bit is repeated once, a repetition code of length
2, the code is unable to correct any errors, but it is able to detect a single error. This is
because any error might have come from either a 0 or a 1, as can be seen in Figure 2.1a.
Repetition codes of length greater than two are able to correct errors. A repetition code of
length three can be seen in Figure 2.1b, which is able to correct a single error, by aligning
the decoding with the majority opinion. Figure 2.1c shows a repetition code of length four.
This code is able to correct a single error, just like the one of length three, but it is also to
determine if two errors were made, in much the same way as the repetition code of length
two does it.
There are many other ways of generating an error correcting codeword, such as Hamming
codes, and BCH codes. These, as well as the repetition code, are all what are referred
to as hard-decision codes, i.e. they rely solely on the binary information in the received
sequence, working directly on the 1’s and 0’s from the demodulated channel.

2.1
BCH
Bose–Chaudhuri–Hocquenghem codes (BCH codes) are an extended type of Hamming
code that are cyclic and can correct multiple random errors. In fact, for any code length
n = 2m −1, it is possible to correct t < 2m−1 errors, using n −k ≤mt parity bits. Such
a code is typically denoted as a (n,k) BCH code. A BCH code has the minimum distance
dmin ≥2t + 1 [6].
BCH codes are constructed using Galois fields. One feature of a Galois field, GF(2m), is
that there exists a primitive element, a polynomial α over GF(2), so that every element in
the field, bar zero, can be expressed as a power of α[6][7]. The generator polynomial for
a given BCH code, is then the polynomial g(X) of least degree, with the following roots:
α, α2, . . . , α2t.
2.1.1
Decoding

The oFEC Specification
The oFEC error correcting scheme is designed for use with incredibly high-speed and
long distance communication channels. These types of channels can cause many errors
in the transmissions, both through extreme noise and burst errors, which are a continuous
sequence of erroneous received symbols, over the communication channel. These many
errors can be difficult to deal with, and therefore advanced error correction techniques are
introduced.
The oFEC standard introduces a block-based, product-like encoder and an iterative soft
decision decoder [5], providing robust error correction capabilities. An iterative decoder is
a type of decoder that enhances error correction performance by repeatedly processing
the received data, refining its estimates of the transmitted information with each iteration.
This iterative process is fundamental for correcting many errors, but must be designed in
a way that is correct for the channel and needs of the communications system [12].

3.2
Evolution of Product-Like Codes
Longer codes are better at decoding, since the errors average out on a larger block of
data. However, the complexity of the encoder and decoder increases drastically with the
message length. Therefore, it is often better to cut it into small, predetermined parts where
each of these blocks can be encoded and decoded on their own. For more robustness
in the communications systems, it may be desirable to place these blocks in a way that
allow for constructing longer codes using several shorter codes, as well as ensuring that
each single information bit is part of two codewords, allowing for iterative decoding [7, 8].
3.2.1
Product Codes
One way of constructing these longer codes is known as product coding. These codes
produce a two-dimensional structure, where all rows and all columns are codewords. First,
the information is placed into a square matrix, then each row and each column is encoded
using some block code. This can be seen in Figure 3.3.
Encoding is straightforward. For example, when using the cyclic (n,k) BCH code, the
information is divided into an k × k matrix, whereafter each row is encoded, resulting in a
k × n matrix, where all rows are now codewords.
Now, each column will also be encoded, resulting in a n × n matrix. The bottom n −k
rows are also codewords, since the code is cyclic [7]. Decoding can be done iteratively,
going through all rows, then columns, then potentially continuing the process.
Product codes result in more parity bits ((n −k)2), but if the size of the information blocks
are large enough, the encoding processes are independent and the decoding process
becomes potentially quite efficient [7, 8].

n×n matrices. The first block is an all-zero matrix, but each subsequent block is encoded
by combining either the current block with the previous one, generating a set of information

3.3
oFEC Structure
One notable application of the zipper coding framework is in the development of the
oFEC structure.
By leveraging the principles of zipper coding, oFEC is built using a
BCH(255,239) code with the addition of an even-parity bit, resulting in a BCH(256,239)
code [13]. The BCH code is defined over GF(28), with the generator polynomial:
(z8 + z4 + z3 + z2 + 1)(z8 + z6 + z5 + z4 + z2 + z + 1) =
z16 + z14 + z13 + z11 + z10 + z9 + z8 + z6 + z5 + z + 1
As a zipper code, oFEC has the property that each bit is part of two constituent code-
words. Each component is a binary vector, with a length of 128. To achieve this, oFEC
conceptualizes an infinite matrix with 128 columns and an infinite number of rows. The
infinite matrix is partitioned into square blocks of 16 × 16, arranged in rows and columns.
There are eight blocks in each row, and each block is indexed by a block row number (R)
and a block column number (C). (R) can range from 0 to ∞, while (C) ranges from 0 to
7 [5, 19].
Each block is subdivided into bits. Each bit inside a block can be identified with a bit row
number (r) and a bit column number (c), ranging from 0 to 15. This results in each bit in
the infinite matrix being able to be uniquely identified by the quadruple {R, C, r, c} [5, 19].
Following this logic, a row of bits in the matrix can be indexed by the tuple {R, r}. Each
row is the second half of the constituent BCH(256, 239) codeword. The first half of this
codeword comes from further up the infinite matrix. The kth bit of the constituent codeword

System Design Considerations
With all the components of the oFEC system defined and explained, implementation can
begin. This project seeks to be able to test a large amount of bits, allowing for verifying
the oFEC capabilities at the very low BERout figures expected. These tests are unable to
run in software in a sensible amount of time, and therefore an FPGA will be used.
oFEC is a large system, and therefore a Virtex UltraScale+ FPGA board will be used as
the platform for implementation. The specific device is VCU129, which has 3780 thousand
logic cells and almost 500 megabit of available RAM.
While most of the individual systems of the oFEC codec have already been explained, it
is necessary to provide an overview of how they will all play together. The system will
be designed for ease of implementation, to allow for parallelism and pipelining, and for
scalability in the implementation if needed.

speed than the rest of the decoding, but this added intricacy is most likely not worth it.
It would need meticulous framing, clock-domain crossing signals, and a highly optimized
memory block that can operate at very high clock speeds.

does come with a caveat. As can be seen in Figure 4.5, half of the decoded word is

about the codewords, i.e. iteration n knows the latter half from iteration n −1 and has the
front half from its own memory. This is not guaranteed to be a word, but neither is the
input into a SISO decoder, and therefore this should be fine.

5
RTL Design and Implementation
To place something on an FPGA, it must first be implemented in an RTL language, such
as VHDL or Verilog. RTL design is fundamental in defining the exact behavior and struc-
ture of the hardware components, ensuring that they meet the desired specifications and
performance criteria. This stage involves transforming high-level descriptions of a system
into a detailed and precise hardware model that can be synthesized into physical circuits.
Due to the requirement of speed, as well as the many parallel processes being placed
in the decoder, area and space optimization of some of the blocks must be performed.
Area optimization can be done using smart algorithms or look-up-tables for multiplication
or other, similarly non-trivial math functions. Speed optimization comes in two forms.
First comes latency, the amount of clock cycles, or time, something takes to compute.
Secondly is throughput, which is the speed at which these clock cycles can run, i.e. how
much time is between each output. Generally, by the use of pipelining, it is possible to
run something faster with more clock cycles, but this might come at the cost of increased
latency. Therefore, up to several implementations of each block have been run on some
of the components to ensure an optimal implementation.
The implementation is designed for a clock speed of ≈225 MHz. When building large sys-
tems, however, overspeed is often applied. This means that the clock speed for individual
components is ensured to be a good 20 to 40% higher than what is actually needed, such
that when placing the many blocks on the board, there is some leeway for the synthesis
tool [20]. Therefore, the following implementations are all designed with a clock speed of
≈285 MHz, if not explicitly stated otherwise.

passes them on to the next block. The calculation is done in several steps. In the first

two clock cycles, the block determines which positions are at stake, as well as checks the
parity of the word. The next clock cycle then does the simple addition needed.
5.2.4
LLR Selection
The 42 LLR sums are then passed through another selection network. This time, the
network is of size (2, 42), but this network doesn’t actually want the two lowest values,
and instead it wants the two lowest, but dissimilar values. This is because the two best
codewords must be different, since this gives better decoding [9].
An interesting thing happens when this condition is added. A four input sorting network
actually only has a single comparison more than a (2, 4) selecting network and since
there might be a case where the third or fourth item in the sub-network is needed, it is
simply easier to implement the sub-network as so. The (2, 42) selecting network therefore
outputs the two best, but distinct, codewords from the set of 42. If there is only one valid
codeword found, a flag indicating this follows the output LLR value.
Similarly to the network in Figure 5.5, the (2, 4) version is optimized to use the fewest
amount of comparisons needed, but jumping ahead in the tree with some values. The
details of this are left out.
5.2.5
Soft Decision and Extrinsic Information
The soft decision block calculates the output LLRs of the soft decision decoder. This
is done in several steps, so as not to impair timing. The first and second clock cycles
determines which positions are different in the two codewords, as well as calculates the
parity of the two words.
The LLR calculation is then split into two clock cycles as well. The first clock cycle simply
calculates the values according to Equation (3.2) for the relevant positions, and places
them in a register. The next clock cycle finalizes the calculations of these values by
using a look-up table to apply the α scaling, and also calculates the values according to
Equation (3.3), along with their α scaling, or applies the β constant if needed.
5.3
oFEC Interleaver
The oFEC interleaver consists of three parts, a memory control plane, the interleaving
logic, and the memory itself.
The interleaving logic is based in , and places each bit in the correct position. The inter-
leaving is regular for every even-odd par of block columns, and can be seen in Figure 4.3.
As seen on the figure, the even-odd pair is switched when placed into the infinite matrix.
The implementation of this interleaving is trivial in hardware, and so is the switch to using
LLRs rather than bits.
The memory operates on a binary 2 × 8 block slice in the oFEC structure, meaning that
32 codewords are input at the same time. This is the result of a quirk in Equation (3.1).
As previously mentioned, the interleaver and memory is designed for 32 input and output
codewords each clock cycle, and therefore the memory controller uses a simple state
machine to control a buffer, that when full, is passed to the interleaver and memory block.
The state machine simply counts the number of inputs since it last wrote to the memory.
5.3.1
LLR Memory
After an input frame is interleaved, it must be placed in the memory. The memory is
implemented as six cyclic FIFOs and two bypass register chains to allow for the staircase-
34
FPGA Implementation of High-Speed FEC Codes for Long Reach Optical Communication
==PAGE:49==
like reading. The memory operates on two frames at once, from (R, 0) to (R + 1, 15).
Specifically, these codewords are expected to be an even-odd pair of block rows [19].
When updating this memory to work with the six bit length LLR values, no challenges
were met. The LLR memory takes an input of 24576 bits (32 ∗128 ∗6 = 24576). These are
treated as two frames of eight blocks of LLR values. The first 768 bits are assumed to be
the first row of 128 LLR values (128 ∗6 = 768), and so forth.
Each FIFO corresponds to one of the block columns in the infinite matrix. Each address
at each of the FIFOs holds two 16 × 16 blocks, those being the even-odd pairs mentioned
earlier.
When writing an even-odd block, all of the addresses passed to the memories are the
same. Input data is seen as a single row, built up from the blocks, with the second row
concatenated behind the first row, and so forth. When reading, each address is incre-
mented by one compared to the former. Then, each 128 LLR front half of the constituent
component codewords are joined at the output of the memory.
The block RAMs are all of 32 addresses wide, since this is the smallest power of two that
can hold all the needed addresses when both reading and writing, without any overwriting
when dealing with the constant dataflow. This also allows for easy cycling, and is what
imitates the infinite nature of the oFEC structure.

Hardware Implementation Strategies
Even if all of the sub-components can meet timing, it does not necessitate that a design
combining them will. The oFEC decoder is, by all metrics, a massive implementation, and
therefore several strategies must be put into use to allow the synthesis tool to successfully
build the system.
6.1
Optimizing the Design
When an RTL design grows, the actual, physical placement of the logic on the FPGA chip
becomes more difficult, and designs reaching 80% are deemed difficult to route and time.
This is because routing resources will be congested, causing large net delay. Net delay
is a measure of how long it takes a signal to travel to their destination register. As can
be seen on Figure 6.1, net delay can cause an optimized implementation, with low logic
delay, to fail to reach timing. The synthesis tool determines a design is congested when
a large area of tiles on the FPGA use over 100% of available routing resources [21].
Figure 6.1: Signals on the chip unable to meet timing because
of net delay
6.1.1
Routing Issues
A large contributor to congestion is components with high fanout. Signals that are routed
from one region to multiple components create a zone of high congestion at the output of
the source. Additionally, the synthesis tool wants to place all of the destinations near the
output register, exacerbating the issue [21].
High fanout occurs in multiple parts of the design, because of the 42 duplicate hard de-
coder blocks, as well as the general size. An example of high fanout of the chip can be
seen in Figure 6.2.

ing congestion

In order to reduce the fanout, register duplication was put to use. Register duplication, as
seen in Figure 6.3b, places multiple copies of the same register, such that the synthesis

Therefore, a "dont_touch" flag is added to the signal instantiation of the register chains
where spreading out duplicates would be able to help reach timing.
There are other ways to reduce congestion, such as using input registers. Placing input
registers is a common technique that takes advantage of how the synthesis tool places
blocks. Programming a register at the input of some data processing block allows the tool
to optimize the path towards that block since there is no logic being performed, just the
pathing. Input registers were placed at the beginning of most blocks in the design when
timing was not being met.
6.1.2
Datapath Optimization
The previously mentioned schemes are beneficial, but sometimes they are not enough,
and they cannot fix an implementation that is not optimally designed. When the SISO
decoder was first being implemented, the binary information was routed through most of
the design in parallel with the soft information. This includes the test patterns, which were
then routed through the (2, 42) selection network so they could be used when making the
soft decision. The hard information was also saved in a memory separate to the LLR one.
This was deemed inefficient, and several improvements were made to remedy this. Firstly,
the original hard information is generally not used after the test patterns have been made,
and can therefore be discarded from most of the design. Because the sign of the LLR
values hold the same information, just inverse, this was trivial to change in the test pattern
generator.
But, when the test patterns are made, this information is then passed to the syndrome,
and subsequently the LLR evaluation block needs to know which LLR values to sum.

Additionally, the hard information is needed when making the final decision.
Routing each hard information through the (2, 42) was the primary issue. Each of the 42
words carried not just a nine bit LLR value (it has inflated at this point due to the sum),
but also the 256 corresponding bits. Most of these words are all the same, and any two
test patterns will only differ in six (the minimum weight of the BCH code) to ten positions
(three flips plus two corrections in two different directions) [9].
Since the original LLR values are needed at the soft decision, these are routed past the
selection network without being changed by the logic of the network, and therefore the
original hard information is also available at the soft decision stage. It was determined
that, instead of routing each codeword through the selection network, it would be more
optimal to save the addresses of the flips and route those through.
This uses fewer routing and MUX resources, and even though some extra logic must be
used to generate the test pattern at the soft decision block, this was still a major optimiza-
tion. Each address can be contained in an eight bit field, and there are five of these at
most. To ensure the proper codeword is generated, a flag is added for each address, and
a flag also indicates whether the addresses point to a proper codeword at all. This uses
46 bits rather than the 256 bits for the pure codeword.

Conclusion
This thesis set out to investigate the oFEC decoding scheme introduced in [5]. First, a
Chase-based SISO decoder was implemented in VHDL. Several versions of many of the
blocks were constructed, to ensure an optimal design. The size of the SISO decoder
was decreased by nearly 40%, going from more than 200000 LUTs to just 130000, and
several investigations into proper techniques for succesfully placing logic on an FPGA
were made. This resulted in the SISO being able to reach timing at an overspeed of 25%,
running at a clock speed of 285 MHz.
Several full implementations of the oFEC system were synthesised, placed and routed in
the Vivado synthesis tool. The decoders that ran two lanes in parallel, which results in a
speed of ≈50G were not routed successfully. In the end, a one lane configuration, with a
speed of ≈25G, was able to reach timing, and was fully placed on the board.
The overall goal of this thesis was to implement and test an oFEC decoder that was able
to process and decode 1016 bits within a reasonable time frame. The final product is able
to do this in less than four days.
Finally, behavioral simulations showed that the decoding capabilities of the implemented
system were lackluster. While the system could run with no or few errors for a good while,
with enough noise, the decodings would hit a break point, and begin outputting errors at
an alarming rate. If there were no or few input errors, the system did not break.
8.1
Future Work
Testing and analysing the implementations revealed various areas prime for improve-
ment. Firstly, congestion in the two lane version of the oFEC decoder prevented the de-
sign from being successfully routed by the synthesis tool. There are just a few areas that
have not been fully optimized in the decoding process, however. One area that may help
is the syndrome calculation. The current version calculates the syndrome for each test
pattern individually. This is inefficient, since each input codeword is only slightly different
from the others. The positions where they differ is also known, and it should therefore be
possible to calculate the syndromes of the input word, and then calculate a new syndrome
based on the flips for each of the 42 patterns. This might save some logic, but the current
syndrome implementation is already incredibly small, and it is unknown how much this
would help with the congestion.
Another issue is the problem that occurs when the decoding breaks. The SISO decoders
are fully verified to work correctly, and the issue is therefore in the oFEC implementation.
It is likely that it is a result of choices made when designing either the oFEC memory
or the iteration configuration. One place to look at first is the passing on of LLRs to the
next iteration, where the current design discards part of the decoded word because that
iteration no longer needs it. It is possible that passing this first half of the codeword to
the next iteration, and in some way combining this with the LLRs in the front half from the
following iteration may help. Some considerations must then be made regarding the shift
from soft to hard iterations, as there are no LLRs to combine.
Finally, it would be ideal to know that the synthesis tool does not break the design when
placing it. A full hardware testbench should be developed that allows one to check if the
behavior of the system on the FPGA is as intended.
