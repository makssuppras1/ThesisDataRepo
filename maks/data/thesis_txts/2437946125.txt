Summary UK
This thesis implements Monte Carlo Tree Search (MCTS) in three Sokoban
domains - the original Sokoban from 1982, a version without deadlocks (02285)
and a self made domain with exploding mines. We analyse solution length and
computation time of MCTS in these domains and compare them heuristic search
methods, such as A*.
We show experimentally, that a version of MCTS which combines all search trees
into one tree (Onetree) together with Upper Conﬁdence Bound applied to trees
(UCT) selection, performs great with a heuristic as an evaluation function for
all domains. In the 02285 domain it performs slightly worse than A*. However,
in the Original and Mine domains we see how the introduction of negative
reinforcements makes Onetree outperform A*. We also ﬁnd that using rollouts
with cut-oﬀs does not provide any advantage in static environments such as
02285 and Original, whereas it provides great beneﬁts when used in the Mine
domain that has probability factors.

Introduction
In this thesis we implement and analyse how the Monte Carlo Tree Search
algorithm, introduced in chapter 3, can be used in a Sokoban domain.
We
work with the Sokoban domain deﬁned in the course 02285 [2], the Original
Sokoban and a self made domain with exploding mines.
See section 1.1 for
further explanation of the domains. In the implementation of MCTS we start
with a very simple version and then add more depth and improvements to the
algorithm ending up with diﬀerent versions, where we explain the diﬀerence and
related results. Furthermore, we elaborate on which improvements we chose in
connection to related work and what we believe will have the most eﬀect on our
domains. MCTS is not a domain speciﬁc algorithm but when improvements are
wanted domain speciﬁc solutions are often needed.
1.1
The Sokoban domains
The Original Sokoban is a 2D puzzle game created in 1982 [1], where the player
is located in a room with boxes and goals. The player can move around in a
2D-grid room and interact with boxes by pushing them, however boxes can not
be pulled or pushed around corners. This means that boxes can be places in
situations where it gets stuck or can not be moved to a goal location, this is
called a deadlock. To win the player must push boxes into predetermined goal

Sokoban, has an extended action schemas where the player is allowed to pull

created and called the Mine domain, because it has an added mine mechanic
where mines may explode based on probability.

C h a p t e r 2
Related Work and
Preliminary Thoughts
It has become popular to use deep learning with a neural network to calculate
an evaluation function for the MCTS. This combination of MCTS and a neural
network is very powerful as shown by Google’s AlphaGo and AlphaZero [6],
reaching a level of skill that beats the current human champion in both the
games Go and Chess. MCTS works similarly to the Minimax algorithm but
utilises rollouts, which are very cost eﬃcient in terms of calculation speed. In
the case of AlphaGo and AlphaZero MCTS is used on a multiplayer game and
the algorithm has to take both the player and the opponent current state and
score into consideration.
We will not be using any neural network to calculate our evaluation function and
will instead create our own based on heuristics and domain speciﬁc knowledge.
We will focus on improving the MCTS solution itself without a neural network
calculated evaluation function, and see how well MCTS can be applied to our
Sokoban domains.

2.1
MCTS for real-time games
Several studies [3] have shown that MCTS can provide great results when work-
ing in real-time games. Here the agent must think fast in a constantly changing
environment and ﬁnd a good solution to the games. In these environments it is
crucial to think multiple steps ahead in order to achieve good results and MCTS
has the ability to do this with simulations.
In a real-time game, like Pac Man, there is only a certain amount of time for
calculation before a decision has to be made. In [5] the authors utilise the MCTS
to pilot Pac Man. Because Pac Man is real-time, MCTS constructs a search tree
every frame where it gathers as much information as possible before having to
take a decision and move. They use rollouts to search multiple steps ahead and
to analyse the score based on how often they died and the amount of points
gathered. They also save and reuse some of the last trees generated for more
information. In the end they have some good results using the MCTS in the
Pac Man environment.
In our case we are not restricted on time speciﬁcally, but the goal of Sokoban is
to solve the level with a balance of a short solution length and short calculation
time. So with this in mind we have to analyse and test how we want to implement
our MCTS so that it has a good balance between solution length and speed.
How big should the tree be before making a move and how long should the
simulation be. These factors should change in regards to domains. We also
need to think about our tree building policy and if we should discard or reuse
some of the previously gathered information. Another consideration is that Pac
Man is always changing because of the ghosts moving around, but our Sokoban
game always stays static, and has no outside factors that can change the game.
This means it might be more beneﬁcial for us to reuse gathered information in
previous trees as the information never becomes irrelevant.
In the 02285 Sokoban domain there is no way to lose, since the game action
schema is reversible. This is because of the added actions where we can move
around corners, where in the original Sokoban domain a box would be stuck in
a corner and the current game would be unsolvable. What this means is that
our MCTS is going to get less information back as we do not get feedback about
losing. If our simulation pushes a box into a corner, we do not get immediate
feedback as the game is not over. Therefore, we have to think about how good
the move was and how we evaluate the state. We will use some known evaluation
functions and compare them with their heuristic counter part. We also have to
consider both the original and Mine Sokoban domains and how we can use
the fact that both domains can reach states where the game is unbeatable and
incorporate that into the MCTS, so it learns how to avoid those situations.

C h a p t e r 3
Monte Carlo Tree Search
Monte Carlo Tree Search (MCTS) is a best ﬁrst heuristic search algorithm that
utilises random sampling in a state space. The sampling is based on Monte
Carlo simulation and is also known as rollouts. The algorithm starts in the
initial game state which is selected as the root node for MCTS.
The standard MCTS consist of four steps, starting from the root.
• Selection: Selects the node that is to be expanded.
• Expansion: Expand the selected node using domain speciﬁc action schemas.
• Simulation: Make a rollout for each expanded node, resulting in a score
for each node.
• Backpropagation: Update the score of parent nodes, based on the roll-
outs.

3.1 Selection
9
All four steps are performed iteratively and consist of one cycle. Each cycle
being a reinforcement learning process, providing additional knowledge to the
MCTS tree. When deciding on taking a move, MCTS performs cycles until
reaching a time or computational limit, at which point the best child node from
the root node is chosen as the next move.
One of the advantages of MCTS, is that it can be terminated at any time. Since
results are always backpropagated at the end of each cycle, MCTS can always
stop and pick the best move based on the collective knowledge gathered for each
cycle. We now explore each step of MCTS in further detail.
3.1
Selection
In the selection process the MCTS starts at the root and chooses which child
to be explored.
This is done iteratively until reaching a leaf node, which is
returned as the selected node. When deciding on which node to explore, there
are two key factors to consider.
• Exploration: How often has this node been explored.
• Exploitation: How highly has the node scored solely on rollouts.
To achieve good results it is therefore required to have a tree selection policy
that balances exploitation and exploration.
We have three selection solutions and the ﬁrst is BFS selection where we always
select the least explored node.
We disregard score and create a tree with a
certain cut-oﬀ. The second is a highest score selector which only selects the
node with the highest score and disregards exploration all together.
The third selector is called UCT and is a solution which utilise both exploration
and exploitation.
3.1.1
UCT
Upper Conﬁdence Bound applied to trees (UCT) is derived from the UCB1
function used for maximising rewards in the multi-armed bandit problem [3].
It balances the exploitation of well rewarding nodes by being willing to explore

3.2
Expansion
After a node has been selected, MCTS expands the tree from that given node.
Most commonly MCTS extends the node by exploring all its possible moves,
derived from the action schemas. However, in games like chess with large action
schemas and/or many possible moves, this approach can be very expensive.
In such situations the MCTS will have to do its expansion through sampling,
either by random choice or with the help from domain speciﬁc knowledge to
ignore obvious bad moves.
3.3
Simulation
In Basic MCTS, each node is given a score based on a sample of rollouts. A
rollout is performed by doing random legal actions until the game have reached
a terminal state from where a score can be derived from. These scores can be
very simple as ones and zeroes, in chess a win could give the score 1, a loss 0
and in the cases of draw the scoring would be 0.5.
However, reaching a terminal state through random actions is not always feasi-
ble. As for a puzzle game like Sokoban, trying to ﬁnd a solution is NP-Hard.
Since we can not expect to reach a terminal state in any reasonable computa-
tional time, we need to implement a cut-oﬀmethod during the rollouts.
One approach could be to always return 0, since the rollout did not reach a
terminal state we could assume that it is not important. Another could be to

11
use domain speciﬁc knowledge to predict how well the rollout went and use that
as basis for scoring. If done correctly this approach would lead to much better
results as the rollouts would yield much more information.
3.4
Backpropagation
Once a state has been scored through a simulation and evaluation the backprop-
agation begins updating scores of all its parents ending with the root node. In
most cases for MCTS the backpropagation policy is averaging scores and it is
also what we have chosen in this project.
3.5
Space and Query-Time
For MCTS space is equivalent to the number of nodes explored in the tree,
which can be denoted as ’N’, therefore space is O(N). For Query-time we need
to analyse each step of the MCTS.
Selection, here we start from the root node and traverse down the tree until
reaching a leaf node, for each node we visit we pick the best scoring children
based on our selection method. This we can denote to O(|A| ∗H), were |A| is
the size of our action schema, and H is the height of the tree. If our selection
method was to be based on Breath-ﬁrst search we know the tree will be uniformly
distributed and can describe the height H more precisely to be log|A|(N), N being
the number of nodes in the tree.
Expansion, O(|A|) the number of expanded nodes are equivalent to the size of
our action schema.
Simulation, O(S) rollouts can be done in constant time for each simulation step
S. This is based on each step taking an action from the action schemas and
evaluating the state.
Backpropagation, we start from a leaf node and traverse up to the root, this
takes O(H) time were H is the height of the tree.
In total an MCTS cycle uses O(|A|∗H) time and O(N) space. The computation
time therefore depends heavily on the size of the action schema and the height
of the tree.

3.6
Positive and negative reinforcement
Reinforcement learning is often associated with wanting an AI to learn how to
pursue nodes that gives positive feedback while simultaneously avoiding nodes
that give negative results. In order to understand how we can apply reinforce-
ment learning to our AI it is easier to separate it into two categories that each
contribute to the reinforcement learning, one being positive reinforcement and
the other being negative reinforcement. With this in mind we need to analyse
the diﬀerent domains where positive and negative reinforcement can be achieved.
Furthermore, it is interesting to analyse whether we should prioritise one rein-
forcement strategy over another, in other words should the AI be taught with
the carrot or the stick.
For all Sokoban domains the positive reinforcement would be to reward the agent
to get boxes closer to the goal destinations. However, when we start looking
at both the Sokoban Mine and Sokoban Original domains we will see that the
strategy diﬀers a lot from the Sokoban 02285 domain as we now introduce
negative reinforcement strategies.
If we look at the Sokoban Mine domain the negative reinforcement is present in
teaching the agent to avoid stepping on mines. Since causing a mine to explode
would result in either the agent dying or a box being destroyed, leaving the level
to be unsolvable. This means the strategy should prioritise avoiding these mines
by ﬁnding the safest path towards the goal rather than ﬁnding the shortest one.
What previously was most important (the length of the path) is now a lot less,
with the introduction of mines.
Similarly with the Sokoban original domain the negative reinforcement is trying
to avoid deadlock. Because this domain has a limited action schema compared
to the Sokoban 02285 domain, it is possible to push boxes into location where
they would be stuck, thus making the level unsolvable as the box can not be
moved to a goal destination. Finding and ﬁguring out when a speciﬁc state is in a
deadlock situation is very tricky. In [4] the authors used this domain as a basis to
understand and explore new search methods for the single-agent AI, and one of
their biggest focal point and challenges, was ﬁguring out how to make the agent
avoid these deadlock situations. In our case we use MCTS, which still would
require us to have some knowledge to determine whether a state is in a deadlock.
However, we might not need this knowledge to be as advanced, since MCTS looks
further ahead than a heuristic search method. For the MCTS search method
we can use the reinforcement learning process, to try to understand how likely
a node is going to end up in a deadlock situation through a statistical approach
were as the heuristic simply cannot. When evaluating a node that are not in a
deadlock we combine the positive feedback about how close the node is to the

goal state with the negative feedback about how likely the node is to end up in
a deadlock. This is less extreme than the mine domain strategy as here both
strategies are of importance.
Lastly when looking at the 02285 Sokoban domain, there are no real negative
reinforcement policies to draw from, since every action is reversible we are only
concerned with getting the boxes as close to their goal location. This could be
a reason why heuristics could perform better in this environment, while MCTS
might perform better in environment where negative reinforcements are present.
3.7
Rollout methods and policies
The most common used rollout method for MCTS is taking random moves.
In our MCTS we created a random walk simulator that just takes a random
executable action from the action scheme. We have also experimented a DFS
rollout method to try to make the rollouts search deeper to get better results.
The rollout policies (simulation steps) are highly depended on the domain and
can also vary in levels. In chapter 4, 5 and 6 we will test on diﬀerent domains,
where we will perform experiments with diﬀerent rollout methods and simulation
steps and analyse the results.
3.8
Evaluation methods
After simulations are done an evaluation and scoring is needed before backprop-
agating and updating scores in the tree. The most naive way is to check if we
won the game by checking if all goals are solved. We will not be using this
approach, but will use a simple evaluation Goal Count (CG) and a bit more
complicated one, called All Pair Shortest Path (APSP). For explanation see
section 3.8.1 for GC and 3.8.2 for APSP.
The reason why we are not going to develop an evaluation function better than
the APSP, is that it is almost fully equivalent to a heuristic function when
looking at positive reinforcement. We do not want to spend time improving a
heuristic function and would rather look into improving MCTS in other ways,
such as negative reinforcement, rollout policies, selection policies and so on.
In the terms of positive reinforcement we want an evaluation that is better than
goal count so we are able to solve a decent amount of levels and we achieve this
==PAGE:28==
14
Monte Carlo Tree Search
with the APSP. We also do not wish to dig too deep into evaluation functions
for the MCTS as this could be achieved with neural networks. This is, however,
out of scope for the thesis.
When a simulation is done and we want to backpropagate a score, we want
to ﬁgure out how to evaluate and score the process of the simulation. If one
was only to look at the end result of a simulation, the information might be
lost. This is because the end result could have been a not favoured step but
the process up until could have been preferred.
So when we are looking at
positive reinforcement regarding evaluation of a simulation we want to get the
maximum score of the process, also know as the peak score.
Furthermore,
regarding negative reinforcement we also want to work with the peak score,
but in this scenario we want to give the lowest score. In the original and mine
domain you can explode and deadlock which can make the level unsolvable and
we therefore give the lowest possible score.
We use normalisation as a way of better understanding the score system before
backpropagating so we are normalising between 0 and 1. In this way we have a
system where we know 0 is the lowest possible score and 1 is the highest possible
score. An example where this system is an advantage is when scoring deadlock
or exploding, because we then know that scoring 0 for that is always going to
be considered as the worst possible outcome.

and evaluating upon this. This is better than the naive method but still have
problems with solving goals in orders and does not yield any feedback based on
moving the agent or boxes around in the game until a goal is solved.

We use a distance map to get the distance between one position to another.

Deadlocks
When evaluation the nodes in the original Sokoban domain 1.1 we want to be
able to spot whether the node is in a deadlock situation. Because this domain
has a limited action schema, only allowing the agent to push boxes, it can lead
to situations where boxes are stuck or unable to be moved into a goal location.
We then scan the level for those locations as we want to know the places the
boxes can be moved to without causing deadlocks. We then create a boolean
table containing all the positions in the level, for each position we store true if
placing a box there would cause the level to be in deadlock and false otherwise.
The algorithm iterates through every location in the level, storing the results in
a 2-dimensional array.

3.9
Basic MCTS
Starting from a given root node, the Basic solution iteratively builds and grows
the MCTS search tree by computing MCTS cycles, as can be seen in the ﬁgure
3.1. This continues until the tree reaches a certain size N, that is predetermined,
at which point enough information have been gathered to make a decision on
where the agent should move. By using a highest score selection method the al-
gorithm chooses the best child from the root node, the tree is then discarded and
the selected node becomes the root for next MCTS search tree. This algorithm
continues until a solution has been found in the search tree.
3.10
MultiStep MCTS
We experimented with a diﬀerent expansion policy, where each expansion would
contain a sequence of actions in the same direction, instead of a single action in
the Basic MCTS solution.
This works as a compression of the level and should shorten the computation
time of solving the levels. However, with this expansion policy some states in
the level are never explored, which most of the times turns out to be good since
these states often are irrelevant, but in some levels the expansion policy can skip
an interesting state which can end up being expensive. To mitigate this problem
we introduced polices that would then consider states where the agent was near
a box, next to a corridor or next to a solvable goal to be explored. Although
this helped overall and improved the MCTS, there were still some levels that
caused trouble for this expansion method. We decided not go further into this
idea as we wanted to pursue other ideas in order to improve out MCTS.

3.11
Onetree MCTS
Because we are working in a domain where levels can be solved oﬄine, we are
not obligated to make fast decision and carve out a path early on. Therefore,
discarding our trees as we do in basic MCTS might not make much sense as
each of these trees contains valuable information about the level. By saving
these trees instead we can store all the information gathered by our MCTS in
one large tree that would help us carve out the correct path much quicker for
solving the level.
This is very helpful since the selection process 3.1 is solely based on the in-
formation gathered prior by the MCTS, therefore we ensure that our choice of
selection and exploration is always backed by as much data as possible. How-
ever, with every improvement there comes a cost. Here the cost is computation
time, as shown in section 3.5 the query-time for one MCTS cycle is dependant
on the height of the MCTS tree, as our tree starts growing very large the query
time will begin to suﬀer.
Knowing the pros and cons of building one large MCTS tree, the idea of the
MCTS Onetree was to create an MCTS solution with the best of both worlds.
The beneﬁts of having one large MCTS tree with a ﬁxed and low query-time
like in the Basic solution.
3.11.1
Algorithm
In order to keep the search time ﬁxed we set a threshold of how deep we can
search in the MCTS tree. We denote N to be the maximum search depth that
can be explored from any node. We then initialise our root node R to be the
initial state of the given level. We then construct a maximum heap containing
nodes that are later to be explored as new root nodes, and who’s sub trees are
shorter in height than the threshold N. We use a maximum heap as we are
only interested in ﬁnding the largest element the set of nodes and the maximum
heap therefore perform better than a sorted list.
Onetree Algorithm:
• Step 1: From the root node we continue to run MCTS cycles as shown
in ﬁgure 3.1, expanding and growing the tree.
This process continues until we reach a leaf-node with a search depth equal
to N from the root node in which case we go to step 2.
==PAGE:33==
3.11 Onetree MCTS

If we reach a node containing the solution to the level we simply extract
the path towards the solution.
• Step 2: From the root node, we take each of its children and store it in
a maximum heap.
• Step 3: We then pull out the highest scoring node from our maximum
heap data-structure based on our selection method. We denote that node
to be our new root node and repeat the process from step 1.

3.11.2
Analysis
Compared to the standard tree policy of MCTS where the selection process
starts in the base root of the tree stretching down to the leaves, in our solution
the selected root would be a predecessor to the base root meaning that the
selection process only covers a sub-tree of the whole tree. This means that the
selection of new nodes to be explored is limited to sub-sections of the whole
tree.
However, when deciding on which node to be selected as a new root
node, selection is done by picking the best candidate out of all the predecessor
nodes contained within the maximum heap. Because all the predecessor nodes
together stretches out to cover every leaf of the tree, we can say that the decision
of choosing a node and furthermore the decision to chose a section of the tree
to further investigate, is done using all information gathered in the MCTS tree.
Therefore, we can say that this MCTS solution makes its decision based on all
the knowledge gathered in the tree.
Because the algorithm limits the search depth to the size N, we know that the
query time of one MCTS cycle is bound to O(N ∗|A|), were |A| is the size of
the action schema. For the maximum heap we know that queries are computed
in worst case O(log(n)), where n is the number of element in the heap. This is
a very fast data structure that scales well as the size increases and is commonly
used in heuristic based search methods. To analyse further the algorithm only
does at most O(|A|) queries every time a new root is selected, which is hard to
determine how often happen as this depends on the selection method and the
level layout. The heap would also contain signiﬁcantly less elements than the
number of elements explored in the tree.
Lastly by keeping all explored nodes stored in one tree, the space increases to a
linear size of O(n) where n is the number of nodes explored.
The beneﬁts of using this data structure clearly outweighs its cost, and as shown
in 4.3 it can be seen how much better the Onetree solution performs.

Node and Tree Structure
MCTSNode is the data structure we use to build and construct our MCTS tree.
We construct the tree as a double linked tree, where each MCTSNode have a
reference to its parent and its children contained in a linked list. The children
are not sorted based on score, since the number of children is limited to the size
of the action schema, which in the Original Sokoban is 4 and in 02285 is 13.
Because these are fairly small it is not beneﬁcial to apply a sorting algorithm,
until we work in an environment with a much larger action schema.
We also keep a reference to a Node object which deﬁnes the MCTSNode’s current
game state. The Node object contains data of the current game state, agent
position, box positions etc. We use this information to deﬁne each Node and
MCTSNode’s hashcode, which we use to check if two MCTSNodes are equal.
In the MCTSNode we store data of the total score and number of simulations
accumulated in the node, which are updated during backpropagation. We can
now calculate the average score and the UCT scores for a given MCTSNode.

MCTS
We implement our MCTS by ﬁrst deﬁning its Selection, Expansion, Simula-
tion and Backpropagation methods, by constructing objects of their respective
classes. If we want our MCTS to use a UCT selection, we construct a UCTSelec-

do this for each of the other steps. This makes it very easy to change compo-
nents of the MCTS and therefore easier to build and test new MCTS setups.
Once every method is deﬁned we simply construct and build our MCTS tree
using the algorithm, described in 3.
When we build our MCTS solution we construct a hashset of Nodes, that con-
tains all the current explored states of the level. When we expand the MCTS
tree using our expansion method we check for each of the Nodes whether they
are in the hashset, and discard them if this is the case, as these nodes have
been explored before in the tree. Otherwise the Nodes are added to the hashset
as we construct new MCTSNodes expanding the MCTS tree. We use Java’s
hashset since it stores unique elements based on the objects hashcode. This is
very important as we do not want diﬀerent nodes to be given the same hash
value, which could happen if we used a standard hash map. The data structure
is also very eﬃcient, using O(1) expected time for look-ups and O(n) space.
Lastly to preserve space and improve queries we remove selected MCTSNodes
that cannot be further expanded upon and are not a solution, as these nodes
are no longer relevant for solving the level.

Mine Sokoban domain
This domain is based on the 02285 domain with its action schemas, but has a
new mechanic added to the game world, a mine. The mine is a red tile that
can be walked on like an empty space, however it has a chance to detonate and
explode both boxes and the agent when walked upon. In this domain MCTS
should yield great results as rollouts work as statistic sampling to work out the
probability of a mine explosion. In this section we will show of how negative
reinforcement can highly impact the algorithm.
6.1
Rollout policy
In this Sokoban domain we have increased the number of simulation steps and
have also added more simulations and averaging them. The idea is to focus
on getting information more locally and then simulating more to ensure better
quality of the sampled statistics. It is crucial to avoid bombs because if either
the agent or a box explodes the level is most likely unsolvable. Therefore, the
evaluation function yields the lowest possible score if an agent or a box has died
in a simulation. A possible problem with simulating with too many steps is
getting a low score since it died on one of the last steps. Because even if the
agent was on a good path but died in one of the last simulation step, it would
still be a bad score for that simulation. Imagine if all possible actions of the
==PAGE:52==
38
Mine Sokoban domain
selected node yielded score zero because there was too many simulation steps
so all simulations at some point exploded a mine. However, if the simulation
is more local with many simulations, the probability of walking on the right
path is a lot greater. Also you do not want to only have one step as a rollout
policy since it would not yield enough statistical data. As shown in the test
section 6.3 we can see how increasing from 1 to 5 simulation steps increases
the eﬀectiveness of the agents ability to avoid mines. Furthermore, if you have
too many simulation steps it might walk on the correct path but after a lot of
steps it will properly step on a mine eventually. So even if the initial part was
correct it will not be rewarded for it. However, if we do it more locally we should
have a much greater chance of not getting bad feedback when walking on the
right path. Results regarding this domain and rollout policies is in section 6.3.
Another idea could be to add discount in the rollout policy, such that dying late
in the rollout wouldn’t yield a less bad score than dying early. This might make
rollouts with many simulation steps more precise and allow MCTS to search
deeper into the level. Because of lack time we have decided not to spend time
on this but will consider it as future work.
Now that we have rollout with cut-oﬀthat actually yields good results we want
to try out another rollout method. We will try random walk and DFS rollouts

C h a p t e r 7
Discussion
7.1
Rollout policy
When working in a domain that is static with no outside factors, such as 02285
or original, we can see that doing rollouts with cut-oﬀs does not provide any ad-
ditional information than just evaluating the node itself. If the rollout could be
extended until it reached a terminal state this might provide valuable informa-
tion but in a domain that is NP-hard this is not feasible to achieve. Therefore,
when working with both static and NP-hard domains the best strategy is simply
to search using the evaluation method directly, and gather information through
building the MCTS tree.
Once the domain changes because random outside factors can change the level,
such as in the Sokoban mine domain, we see that doing multiple samples with
rollouts can be crucial when trying to ﬁnd the right solution to a level. Without
random sampling we cannot be sure that the information gathered is correct
which can lead to the agent taking wrong decisions.
==PAGE:60==
46
Discussion
7.2
Selection method
The Onetree MCTS solution is deﬁnitely showing the best results regarding
MCTS solutions in all the three domains. This tree building structure Onetree
provides combined with the selection method UCT is showing to be superior
over any other tested selection methods. We can see how changing the UCT
values eﬀects the outcome of the algorithm to a better or worse solution length
in cost of computational time. Lowering the UCT value increases exploitation
and a higher UCT value increases exploration and we have found that for a
general solution the best balance comes with a UCT value of 0.05. Of course
there will always be exceptions but usually this yields overall great results.
7.3
MCTS performance
The Onetree MCTS solution performance versus the A* heuristic function shows
very diﬀerent results in each domain.
In the 02285 domain the A* is mainly performing better, especially when the
level ﬁts the strategy of the heuristic function. However, there are some cases
where MCTS performs better, as shown in table 4.9, where a blocking problem
appears. The heuristic does not take blocking into account and MCTS seems
better at exploring possible solutions in the blocking problem.
When negative reinforcement is introduced in the original Sokoban domain we
can see that Onetree thrives much better in this environment. Because of its
tree structure and the UCT selection it is able to avoid paths that leads to
deadlocks by looking further ahead in the level than A*. In all the levels tested
in the original the MCTS performs much better than the A* in computation
time and is only slightly worse in solution length.
In the Mine domain we do not even want to test a heuristic function as it does
not work well with probability and uncertainty, but the MCTS is almost build
for these kinds of problems. The rollouts are working as statistical sampling
and is giving a good feedback on what the world looks like by doing random
sampling.

The Basic MCTS solution performs pretty well against the standard search
