Abstract
This project focuses on implementing a novel technique for reducing the overall mem-
ory footprint within a Kubernetes cluster via the analysis of the pressure stall information.
The system was implemented following the microservices architecture where a Swap Con-
troller and a Swap Statistic Informer microservice was implemented. A Kubernetes cluster
was also set up on a cluster of 4x Raspberry Pi’s to serve as a development target for the
solution. This solution allowed for consistent hardware to be used, and as such consistent
results were achieved, this is especially important if the further work is to be done into this
topic.

Introduction
Kubernetes is an extensible, open source, platform for managing containerized workloads,
which is the stable of the modern deployment infrastructure, with 91% of of people us-
ing cloud native workloads using Kubernetes.[1] The rise in the popularity of Kubernetes
corresponds to the shift in computing infrastructure from pets to cattle. Referring to the
decrease in managing individual unique servers, and instead managing large swathes
of identical servers via Infrastructure as Code (IaC) techniques. Container deployment
techniques are preferred within the modern software development process due to various
features such as portability, ease of use, following Dev Ops principles, as well as shifting
focus to the application themselves which are running.
Nearly all cloud service providers today offer some form of managed Kubernetes solution
allowing for developers to easy deploy auto-scaling, fault resistant applications. Due to
the widespread use of Kubernetes, even the smallest of optimizations will result in large
gains worldwide, and for cloud providers such as Amazon and Google, this could lead
to massively decreased computational requirements, in turn leading to reduced power
consumption, making this project contribute towards the sustainable development goals
9 and 13[2]
Kubernetes is also currently trialing the virtual cluster solution, a solution for having Ku-
bernetes clusters within Kubernetes clusters, which could be used to greatly ease the
the management of large scale Kubernetes offerings such as those provided by cloud
providers [3]. With a virtual cluster solution optimizations like the ones defined in this
project would be able to configured for a large amount of clusters easily.
In the recent v1.22 update to the Kubernetes project, alpha support for running nodes
with swap space enabled was introduced.[4] [5] Historically running Kubernetes with swap
space enabled on nodes has been disabled due to the difficulties in providing guarantees
in regards to memory utilization when swap is enabled. However due to an increasing
number of user stories alpha support has been enabled. The project implemented in
this thesis aims to help reduce the performance uncertainties which comes with running
Kubernetes with swap enabled, by dynamically moving workloads between swap and non
swap enabled nodes depending on the determined performance impact of enabling swap
space for workloads.
1.1
Project Scope & Objectives
The aim of this project is to create two microservices which together can dynamically move
workloads between swap and non swap enabled nodes depending on system statistics
and performance of the applications running. This can be split up into a few different key
objectives:
1. Determine impact of swap space on an application.
2. Determine combined impact of swap space on a Kubernetes pod.
3. Dynamically move pods to nodes where swap space is or isn’t enabled.
If all of these objectives are met, the end result should be a reduction in the overall memory
footprint at minimal losses in performance.

Linux Memory Management
While memory management within Linux can be at times quite complex due to the needs
to support systems from micro-controllers to supercomputers a quick primer on the ba-
sics will be given.
1.1
Virtual Memory
Due to the complexities of having to support various kinds of hardware and even having
to support features such as hot pluggable memory, contiguous memory addressing is
complex. Differences in CPU architectures and implementations of said architectures
can also change different how memory addressing works. Because of this the concept
of virtual memory was developed. By abstracting away details of the physical memory,
applications don’t need to keep track of anything complex and only need to use virtual
memory addresses which can easily be contiguous.[6]
When accessing a virtual memory address the memory controller (referred to as the
Memory Management Unit within the von Neumann processor architecture) translates
the virtual memory address to the physical memory address, this is shown in Figure 1.1.
1.2
Memory Pages and Swapping
Virtual memory is split into blocks of pages, where each page is mapped to one or more
corresponding page frames in the physical memory. On most systems the smallest page
size of 4 KiB is used, however this is highly dependent on the processor architecture,
kernel configuration and operating system.
Since virtual memory is split into pages that map onto physical memory, pages can be
updated to map onto storage drives, this process called swapping or paging can be used
to greatly extend the available memory to a system by swapping rarely used memory
pages onto the slower storage, however in doing so leads to increases in latency when
needing to access said pages.
Deciding what pages to swap and when to swap them is a largely complex topic, and is
in Linux handled by the kswapd daemon.
Swapping of memory pages in linux can also be configured using the Swappiness con-
figuration, where how often pages are swapped can be configured.
In memory swap is also possible where when pages are swapped they are compressed
in memory, this comes at the cost of more cpu cycles being spent on the swapping
process. 1.

Containers
Software containers is an approach to deploying software by borrowing from the con-
cepts behind real world shipping containers. By creating a standard ”unit” of software,
the paradigm of how applications are deployed is changed completely, being a large
driver behind the explosive growth of the DevOps development culture present today.
By standardizing how applications are deployed suddenly its possible to manage large
amounts of applications, and allows for software to take a cattle approach compared to
pet approaches.
The Linux foundation currently are in charge of the Open Container Initiative, which is
in charge of maintaining standards for running containers.
Containers themselves however are just an abstraction for a collection of configurations
within the Linux Kernel, making use of the Control Groups and the Linux Namespaces
features.1
2.1
Control Groups
Control Groups (cGroups) is a Linux Kernel feature that provides resource limits and
isolation for processes. The initial version of cGroups, nowadays referred to cGroupsv1,
was first merged into the Linux Kernel in Jan. 2008. However in early 2016 cGroups
v2 was merged into the Linux Kernel. This thesis will only dive into the theory behind
cGroups v2 as the proposed solution is based off cGroupsv2 features.
cGroups provide a singular interface for process control and limitation, from letting sin-
gular processes run with higher or lower CPU priority, to limiting resource usages, to
measurement of a groups resource usage.
cGroups consist of 2 main parts, the cGroup filesystem and the controllers/subsys-
tems.[7]
2.1.1
cGroup Filesystem
Control groups follow the unix philosphy of ”Everything is a file”. Control groups can be
accessed by mounting and accessing the Control Group virtual filesystem, mounted by
default in /sys/fs/cgroup.[8]
Within the virtual control group filesystem control groups are just folders, with all config-
uration being handled by writing plaintext to the various files.
Looking at Figure 2.1 we can see how the virtual filesystem is structured into folders and
files. In the Figure there are 2 different control groups, the low_prio and the high_prio

2.1.2
Subsystems
The Linux Kernel exposes access to different resources via the use of subsystems, also
referred to as controllers or resource controllers. These controllers are what poll the
cgroup filesystem to determine what restrictions to place on the processes. In Cgroups
v2 the following controllers are described in the Linux Kernel documentation.[9]
• cpu: ”The ”cpu” controllers regulates distribution of CPU cycles. This controller
implements weight and absolute bandwidth limit models for normal scheduling
policy”
• cpuset: ”The ”cpuset” controller provides a mechanism for constraining the CPU
and memory node placement of tasks to only the resources specified in the cpuset
interface files in a task’s current cgroup.”
• hugetlb: ”The HugeTLB controller allows to limit the HugeTLB usage per control
group and enforces the controller limit during page fault.”
• io: ”The ”io” controller regulates the distribution of IO resources. This controller
implements both weight based and absolute bandwidth or IOPS limit distribution”
• memory: ”The ”memory” controller regulates distribution of memory. Memory is
stateful and implements both limit and protection models. Due to the intertwining
between memory usage and reclaim pressure and the stateful nature of memory,
the distribution model is relatively complex.”
• pids: ”The process number controller is used to allow a cgroup to stop any new
tasks from being fork()’d or clone()’d after a specified limit is reached. The number
of tasks in a cgroup can be exhausted in ways which other controllers cannot
prevent, thus warranting its own controller. For example, a fork bomb is likely to
exhaust the number of tasks before hitting memory restrictions.”
• rdma: ”The ”rdma” controller regulates the distribution and accounting of RDMA
resources.”
• devices: The v2 devices controller operates differently from the others, ”instead of
exposing interface files within the cgroup filesystem is gated by attaching an eBPF
program to a v2 cgroup”
How the Linux Kernel implements the subsystems is out of scope for this thesis.

2.1.3
Pressure Stall Information metrics
The CPU, Memory and IO subsystems each implement the Pressure Stall Information
(PSI) feature. The PSI feature aggregates productivity losses due to resource scarcity,
providing real time statistics to allow for dynamic management.[10]
PSI Information both resides in, /proc/pressure/{cpu|memory|io} for overall system
statistics, or the corresponding {cpu|memory|io}.pressure files within each cgroup.
The CPU files follow the following format.
1
some avg10=0.00 avg60=0.00 avg300=0.00 total=0
While the memory and io files follow the second format.
1
some avg10=0.00 avg60=0.00 avg300=0.00 total=0
2
full avg10=0.00 avg60=0.00 avg300=0.00 total=0
The some refers to the share of time that at least 1 or more tasks are stalled due to
limitations of a given resource, while the full refers to the share of time in which all
non idle tasks are blocked due to limitations on a given resource, this is also called
thrashing [11]. When thrashing occurs then CPU cycles are being completely wasted
and can have severe performance impacts.
The average values in the PSI output are percentages over the last 10, 60 and 300
seconds respectively where processes have been limited by the resource. The total
value corresponds to the total amount of microseconds in which stalling has occurred.
2.2
Namespaces
While cgroups and their subsystems are used to limit the amount of resources used
by processes and threads, kernel namespaces are used to isolate resources. Just like
cgroups one or more processes can be placed in a namespace, and namespaces can
also be nested. When a process is within a namespace it can only access the resource

System Design
This chapter will look into the structure and the design of the created Kubernetes re-
source optimization solution. The following sections will look into the relevant design
choices and descriptions of the project structure from the perspective of the design.
1.1
Extending or Building new solution
The Kubernetes components are flexible and very extensible, this is supported by Ku-
bernetes having first level support for Custom Resources and Custom Controllers such
as the Operator design pattern [15]. However designing and developing a custom op-
erator can be a complex task, and often developing an entire operator can be overkill
for small tasks. The advantages of developing the solution as a Kubernetes custom
controller would be:
• Easy deployment on new clusters, would be as simple as applying a yaml file.
• Can be interacted with and managed via the kubectl command line interface
• Some other pro
The disadvantages of extending Kubernetes instead of developing a new system how-
ever would be:
1. Complexity and fixed patterns. By creating an operator there is a certain level of
complexity in regards to how the Operator pattern is designed, with APIs needing
to live up to certain specifications.
2. Minimizing Security issues. Due to needing to access the entire cgroup file system
the application needs to be privileged. Building Kubernetes custom resources
would lead to a larger potential attack surface compared to a simple solution.
3. Focus on results. The aim of this project is to the develop an solution which can
reduce the overall memory footprint and is not intended for production usage, fo-
cusing development efforts on the main goal is a better use of time.
While the advantages seem to outweigh the disadvantages of developing the solution
as Custom Kubernetes Resources, one needs to keep in mind that the project is to
be developed for use in large scale Virtual Clusters comprising of many small clusters.
Currently virtual clusters do not support Custom Resource Definitions, and while it is on
the roadmap, the lack of Custom Resource Definitions currently imposes a restriction
and as such a bespoke solution is decided to be developed.
1.2
Project Architecture
The project is split into two different parts which interact with each other. The Swap
Controller and the Swap Statistic Informer.

Swap Controller

3.3
Swap Detection, Analysis and Update process
To test if a service is viable to operate on a swap enabled node first nodes are deployed
to an Swap Enabled Analysis Node with an intentionally low amount of allocated memory.
From here requests are made to the service to simulate traffic while the Swap Statistic
Informer collects pressure data. Once the Swap Statistic Informer informs the Swap
Controller of the pressure values they are analysed by the Swap Controller.
Based off the data received the Swap Controller makes the decision, if the Node is
deemed unable to operate correctly in swap space then the podspec is updated with an
Node Affinity value to make it run on the Non Swap Enabled Nodes.
The swap decision process starts with services being assigned to a ”Tester” node, which
is functionally the same as the swap nodes. Memory pressure is then observed over
a duration of 600 seconds while requests are made to the nodes. From the memory
pressure results a decision is them made to swap the service or not, moving of the
service then involves updating the statefulset for the service.

Go Programming Language
Due to Go being a relatively new language, readers might not know some of the unique
primitives provided by the language for providing parallel operations, as such this chap-
ter will provide a quick overview for the reader.
These features follow the design philosophy of Go, which is a language for develop-
ing lightweight microservices, with high performance multithreading, but by having the
readability and usability of lightweight languages such as Javascript or Python.
2.1
Goroutines
Goroutines are an lightweight alternative to OS threads which are managed by the Go
runtime, where the runtime multiplexes the Goroutines onto a smaller set of OS threads.
When a goroutine is blocked the runtime moves the other goroutines to other OS threads
to prevent unnecessary stalling.
A goroutine is initialized by adding the go keyword to a function call, which then runs the
function in a new goroutine.[17]
2.2
Channels
While the language also supports mutexes, locks and other concurrency controls, the
best practices are to use synchronous message passing in the form of message chan-
nels. The Go language currently does not offer support for generics 1, and as such
Channels are typed, allowing only 1 message type to pass through channels.
Channels have two operations. x <- channel extracts messages from the channel and
stores it in variable x. channel <- x puts the value X into the channel. These operations
are both blocking, however non blocking operations are also possible using the select
statement which functions like a switch operator that the reader might know from other
languages.
2.3
Defer
The defer statement helps avoid memory leakage, the defer keyword, as the name
implies, defers and operation until the function scope of the call returns, this is used to
fx: close channels, and instead of having to make sure to close the channel at every
possible return and error scenario of the function.

Swap Statistic Informer
The Swap Statistic Informer is what works alongside the Kubelet. As described in the
system design the Swap Statistic Informer has the task of receiving commands from the
swap controller, and returning memory pressure information until the the controller says
stop.
3.1
Application Design
The application is designed as a microservice using the ”Master-Worker” pattern, where
for each incoming request a new asynchronous worker is started in a new Goroutine,
allowing for parallel requests to occur.
Due to the low overhead of working with Goroutines, due to the low memory overhead of
a few kilobytes per goroutine there is no need to adopt advanced design patterns such
as thread pools / worker pools and instead just start and stop goroutines as needed,
this helps reduce latency. With multiple operations occurring at the same time one also
needs to take into account concurrency, however as only atomic read operations are
occurring (and with ideally no two goroutines reading the same files) there would be no
concurrency issues present.

libraries.

with the Docker engine.

from the Docker engine.
• github.com/gorilla/websocket: Is used to create and handle websocket com-
munication.

Project Scope Changes
While the initial project had a much larger focus on the integration together with the vir-
tual cluster project, this was dropped from the implementation side of things to make
development easier, however the results gathered are still useful within the context of
virtual clusters. The original scope of the project also had plans for investigating much

ended up producing a useful tool for reducing the overall memory footprint of a Kuber-

applications, however this had to be cut due to the project scope changes.
1.3
Project Limitations
There are several limitations with the project. The tests cases developed are an ide-
alistic scenario and represent a ultimate best case in the memory footprint reductions.
Using this project on real world applications probably will not have the same level of suc-
cess. However I am confident that optimizing the pressure analysis functionality could
definitely lead to significant improvements on real world applications compared to the
current memory analysis functionality.
1.4
Results and Project Objectives
Looking back at the introduction the following project objectives were defined.
1. Determine impact of swap space on an application.
2. Determine combined impact of swap space on a Kubernetes pod.
3. Dynamically move pods to nodes where swap space is or isn’t enabled.
Based on the project objectives the project can be considered an success. An investiga-
tion into how the linux kernel handles swap memory and how swap memory works with
Kubernetes pods was done in the theory section. The system design and implemen-
tation sections covered the implementation of an system which can dynamically move
pods to nodes where swap space is or isn’t enabled based on the impact of swap space
on the pod.

Conclusion
In the final chapter of the report each aspect of the project will be concluded.
In conclusion the implemented solution for determination of the impact of swap space on

3
Future Work
With this project there are many possible avenues for future work. The two main areas
of where future research can be used are as follows.
3.1
Better Pressure Analysis Methods
Future work on the pressure analysis functionality is where the largest gains in perfor-
mance could be had. Developing advanced analysis techniques based off on machine
learning could take into account a large amount of variables for determining if a pod
should be swap enabled or not. Here the speed of the storage mediums, I/O profiles,
memory usage could be useful variables for training a machine learning model. Inves-
tigating this avenue could be an BSc project for another student, where the analyse
results function of the swap controller would be modified.
3.2
Distributed Swap Profiles
Another avenue for future work would be the addition of adding labels to the container
images themselves which define the swap profile of the container, these labels could
then be distributed via the already existing container registries, and could provide spe-
cific swap data for each image. This would involve also extending the project to define
the ratio of memory allocated to swap performance.
3.3
Enhanced Kubernetes Swap Performance
Currently the swap space feature within Kubernetes is in alpha support, only allowing
for swap space to be enabled node wide. However according to the Kubernetes Node
Working group, further developments are on the road map, to allow for pod wide swap
space, and eventually container wide swap space support. Once container wide swap
space is supported this project future research could be done to see if individual con-
tainers in pod, or fx if sidecar containers could benefit from operating in a swap space
enabled environment.
