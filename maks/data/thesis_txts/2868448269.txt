Abstract
In this thesis, we construct a model checker1 that enables queries on properties of AMMs
with a constant product swap rate trading function.
The model checker operates on a transition system where state variables consist of user
wallets, AMM reserves as well as an optional fee level on each AMM. State variables may
be either concrete or symbolically constrained. Transitions between states are enabled by
swap transactions.
The model checker supports both temporal logic formulae, as well as a maximization query
”ϵ-max υ” that returns ϵ-tight bounds around the max value of an expression υ (where
ϵ ∈R]0;1[ is a parameter of our model, which allows tuning the tradeoff between execution
time and precision).
From the initial state space, a bounded search for a witness to the satisfiability of the
formulae is launched. Upon finding a witness, the model checker returns in addition to
the concrete witness a fully concretized initial state from which the transactions contained
in the witness can be executed, to arrive at a state leading to a satisfied formula.
As our tool conducts its search on incrementally longer transaction sequences, any witness
returned by our model checker has the property that it is the shortest possible witness to
the formulae.
The model checker also permits constraining the set of transactions that can be used to
search for a witness. Permitting a potentially empty subset of users to transact freely,
while requiring transactions from other users to be marked as either required or available
for inclusion. This enables more realistic, complex and adversarial scenarios to be checked
on the solver.
The semantics of our AMM model permits the execution of transactions that fail. I.e.
transactions whose premise for execution on the AMM-contract are invalid, and which
hence are reverted by the blockchain state machine - leaving the state unchanged. While
this may seem an odd choice, taking this approach allows modeling scenarios where a
block builder might be forced to include a set of transactions X (e.g. through an inclusion
list), and thus being required to provide an execution ordering for X without necessarily
caring if a subset of these fail, the block builder might still want to optimize the ordering
of X as to maximize some utility.
The results obtained through querying our solver are sound, i.e. if the formulae is unsat-
isfiable it won’t return a witness to the satisfiability of the formula (with the exception of
the ϵ-max operation where Z3 rounding may result in false positives). For completeness,
we say that our solver is relatively k-complete, i.e. if the formula is satisfiable within the
bound of k transactions, it will find a witness - but ”relative” because it is only as complete
as the underlying SMT-solver.

Introduction
Automated market makers (AMMs) are a type of Decentralized Exchange (DEX) deployed
on public blockchain networks that enables non-custodial transfer of digital assets. Trans-
actions are executed atomically and settled on-chain, making them publicly accessible. In
contrast to traditional finance, an AMM does not rely on an order book with bid/ask
entries, rather, asset-pricing is based on a simple algorithmic relationship between the
AMM’s liquidity pool and the input amount of a transaction [1].
Most popular AMMs, such as Uniswap & Curve implement their pricing function, as a
relation between the pre-trade and post-trade values of the reserves, accepting trades only
if the post-trade reserves are in a given relation to the reserves prior to executing the
trade. An example is the constant product function, which requires the product between
AMM reserves be preserved following a trade.
As of January 2025, AMMs such as Uniswap & Curve had a combined liquidity locked of
$6B in on-chain liquidity pools, with daily transaction volume of $1B and $0.4B respec-
tively [2, 3, 4, 5].
Despite the relative simplicity of the economic mechanisms underlying AMMs, a few non-
trivial issues still remain to muddy the picture.
Firstly, is the contentious issue of Maximal Extractable Value (MEV). As swaps are exe-
cuted atomically against a stateful contract - any swap executed in one direction will leave
the state of the AMM contract relatively more advantageous to swaps in the opposite di-
rection. This means that transaction ordering is a primary concern for execution-price.
Actors such as block builders with the ability to determine the transaction order of newly
mined blocks, utilize this to profit through arbitrage - ”sandwiching” user transactions [6].
Secondly, with the widespread adoption of DEXs, there are many different AMMs to choose
from, each supporting swaps between different sets of tokens at varying prices, creating
complexity and fragmentation. Another concern is thus how best to route a transaction
through this network of AMMs to maximize the utility for the sender.
In this project we develop a tool that can be used to model, query and examine properties
pertaining to AMMs.
We will exemplify how the tool can be used, by modeling and
attempting to solve the aforementioned two problems, as well as a handful of others.
1.1
Contributions
Our contributions can be summarized as:
1. The creation of a domain specific model checker or ”solver”1, based on the formal-
ization of AMMs in [7]. This solver allows the user to instantiate an arbitrary initial
state with any combination of concrete and symbolic values (if any variable is left
symbolic, the initial state may be considered as a set of possible initial states, rather
than a single state). From this initial set of state(s), a query can be made to see
if a state with certain (desirable or undesirable) properties is reachable, and if so
the solver will return a fully concrete trace for how to get there. This also implies
concretizing the initial state space to a single starting state from which the sequence
of transactions is fired to arrive at the desired state. This means the solver can be

used to produce actionable transactions to execute, to achieve highly customizable
user-defined goals.
2. We add a maximization-query that constructs a sequence of transactions, to max-

AMMs in example 11.
3. We provide an option to constrain the set of transactions the model can choose from
when searching for a solution, such that e.g. only users A and B are allowed to
transact freely, or a specific transaction from C must be included, while another set
of transactions from D are available but not required. This allows for modeling of
more adversarial scenarios, such as the study of sandwich attacks where an actor
may observe the mempool (the pool of pending unvalidated transactions) and choose
to include and order transactions as he sees fit to maximize his own gain.
4. We include a term in our model to set the fee on each AMM, this allows for analysis
of the cost of transacting on AMMs, such as comparing the cost of slippage vs the
cost of fees in example 6.
5. We show how our tool can be used to do induction-style proofs, by leaving the initial
state entirely symbolic it may represent the universe of valid states. From such an
unconstrained set of initial states, properties can be checked - such as verifying it is
impossible to make a swap to arrive in a state where the reserves on an AMM are
depleted (example 12).
6. We provide an analysis on the required number of steps to be complete based on the
number of users and AMMs in the state. However, we conclude that as we support
infinite precision numbers, there are ways to vary the numbers in the query to make
the threshold for completeness arbitrarily large - and hence bounds can only be given
for a very limited set of combinations of users and AMMs.
7. We provide an explicit formula for the size of the search space, and show that it
grows exponentially in the depth k. However, we provide a set of remedial results,
as there is an element of redundancy in certain transaction sequences that can hence
be removed from consideration.
1.2
Related Work
Many of the sub-problems described above have been studied extensively, and analytical
solutions found to certain classes of problems. Two examples include;
• In [8], the authors consider the routing problem, where a user seeks to maximize a
utility function Ψ. They show that in the absence of fixed-transaction costs, the
routing problem can be formulated as a convex optimization problem where finding
a global solution is computationally feasible.
The way we solve the maximization problem is fundamentally different, as we rely on
an iterative approach, narrowing down on a concrete solution through multiple SMT-
queries, arriving at a close but approximate value. Another point of divergence is that
they study the problem for a network of n AMMs, and 1 user - where we consider a
more general model, permitting more than 1 user. A final point of divergence is that
2

we permit solutions containing more than a single transaction per AMM, whereas
they assume the user has access to suﬀicient capital (e.g. through a flash-loan) to
fund a single transaction on every AMM from the initial state.
• In [9], the authors lay out an analytical solution to the problem of determining the
optimal transaction ordering for extracting maximal value from a set of available
transactions. They construct a ”multi-layered sandwich”, showing exactly how each
layer is constructed. Our tool can be used for similar ends, by using a suitable utility
function that maximizes the net worth of the arbitrageur while marking certain
transactions available for the solver to include as it sees fit. Although our solution is
less eﬀicient (as we rely on an iterative approximate method) it can be used to gain
intuition and construct novel scenarios to check for MEV.
Outline
The remainder of this work is organized as follows:
• In chapter 2 we provide preliminary knowledge on AMMs and a brief formalization
of the transition system our solver works on.
• In chapter 3 we describe the implementation of the solver in detail, as well as what
sort of queries it is capable of answering.
• Next, in chapter 4, we look at more theoretical topics, such as; soundness & com-
pleteness, the size of the search space as well as how to reduce the size of this search
space.
• In chapter 5 we provide examples that showcase how the solver can be used to
check both adversarial scenarios (exemplified by a sandwich attack), collaborative
scenarios (how can users transact to achieve a common goal), an example of how to
best ”route” a transaction across a network of AMMs, and finally an induction proof.
• Finally, in chapter 6 we lay out a road map for future work, to show where develop-
ment on this tool can be taken to further its scope.
• We leave the language-specification used for interacting with the solver in appendix A.

2.1
AMM preliminaries
In this section we provide preliminary knowledge of AMMs, which will be useful to under-
stand the remainder of this work.
Tokens
Tokens are the blockchain-native currency used as a store of wealth and medium of ex-
change, examples include ETH, the native cryptocurrency of Ethereum, but also token
types such as WBTC, which is an ERC20-standard wrapped bitcoin[10]. However, in this
exposition we will simply use τ0 . . . τn to range over the universe of token types T. We use
v : τ to denote an amount v ∈R≥0 of some token τ ∈T.
Users
We assume a universe of users A ranged over by A, B... Associated with each user A is a
map representing a wallet σA : T →R≥0, also denoted A[σ]. If A has no tokens of type τ,
then σA(τ) = 0. We permit increasing/decreasing the balance of A through the operations
+ and −, where A[σ±v : τ] = σA ±v : τ = σA[σA(τ)±v/τ] (where σ[v/x] means updating
the map σ at point x to the value v).
AMMs
We also assume a universe of AMMs, denoted M. An AMM is a an unordered pair {r0 :
τ0, r1 : τ1}, τ0 ̸= τ1, where ri : τi ∈R>0 indicates the AMM reserves of token type τi.
The AMMs we choose to model all have a ”constant product swap rate” function. This
means that following a swap, the product r0r1 must be preserved. The product r0r1 can
be represented as a curve, where trades on the AMM moves the reserves to a new point
along the curve. As an example consider fig. 2.1 where a user swaps 4 : τ0 for 6 : τ1.

3.1
Formalization of supported queries

3.3.1
Numeric precision
A great concern in an implementation like this is the accuracy of the types involved. For
instance, if the high-level solver relied on 32-bit floats, a lot of values received from the
user as well as replies from Z3 would be subject to some form of truncation or rounding
when parsed into the solver as a float.
Luckily, Haskell supports infinite precision rational numbers (the division of two infinite
precision integers). However, Q ⊂R, and by only allowing rational numbers as input we
are slightly limiting the input range, for instance
√
2 is not rational.
The Z3 logic we use supports numbers contained in Q (algebraic reals) [13]. An algebraic
real is a number which can be expressed as the root of a non-zero polynomial with rational
coeﬀicients. This set contains Q, i.e Q ⊂Q, which means that any input from our high-
level solver to Z3 will be well understood - however the reverse is generally not true - as
algebraic numbers includes irrational numbers not contained in Q. This is however not
a great limitation, as we usually print solutions from Z3 directly without parsing them
into Haskell rationals first (with the exception of the iterative maximization described
in section 3.5).
Notably, as Z3 supports algebraic numbers, queries whose solution is expressed in terms
of e.g. a square root (such as the solution to the MEV game of [9]) can be answered
accurately, in its analytical form as the root of a polynomial, as opposed to a truncated
decimal number (although this is also supported).
However, it should be noted that the domain we are modeling itself doesn’t support infinite
precision numbers, nor irrational numbers, for instance the ethereum virtual machine
(EVM) has a word size of 256 bits [14].

3.4
Constraining transactions
The model we have considered so far is one where only the state space has been constrained.
In some sense the model checker has relied on users ”cooperating” when solving constraints,
as it has been free to fire any transaction from any user to arrive at a goal state.
In the real world, users don’t necessarily ”cooperate” in this way, but rather often submit
transactions without knowledge of other user transactions. To enable our solver to reason
about such scenarios, we will need to constrain the transition function. More specifically,
we introduce three types of transaction constraints that lets the solver know how it is
permitted to transact:
1. Available transactions: a transaction can be marked as ”available” for the solver to
use, which means that it is free to include it when looking for a solution - but it
doesn’t have to include it.
2. Required transactions: by marking a transaction as required the solver must include
it in all sequences λs
i ∈Li, i.e. no witness can be found that doesn’t contain this
transaction.
This also entails that if there are e.g.
5 required transactions, no
sequence λs
i for i < 5 is legal.
3. Free, users can be marked as free, which means the solver has freedom to construct
any transaction from this user to arrive at the goal.
When specifying a transaction using the available/required statements, the amounts v0, v1
may be either concrete, or alternatively they can be left unspecified (but user, input and
output token must be specified).
An important (but secondary) consequence of constraining transactions is a reduction in
the search space - as the set of sequences Li that we must consider, is usually more limited
than if we had to consider all combinations. In section 4.3 we look at the size of the search
space in detail.

3.7
Summary
In this chapter we first described the query language supported by our tool, and noted
that as we only support existential quantifiers over paths, we can reduce the satisfiability
problem to the search for a witness πi ∈Pathsi for i ≤k. Secondly, we noted that as
the satisfiability of path formulae φ can be checked incrementally, we are able to find the
shortest possible witness for φ and avoid expanding the search beyond what’s necessary.
We covered how to encode the problem into an SMT-solver, by dividing the search-space
into subproblems contained in the decidable QF_NRA logic, returning the first satisfiable
query. Next, we briefly touched upon the numeric accuracy of our implementation, and
noted that while we only allow rational numbers as inputs, we can still give solutions
contained in the domain Q (with the exception of the ϵ-max query).
We also described a way to constrain the set of transactions available to the solver when
searching for a solution, and saw how by encoding that a transaction is available, the
solver was able to find a better three-transaction sequence to optimize $(σA), by back-
running an available transaction from B. However, we saw that querying iteratively for
progressively higher values of $(σA) was a sub-optimal and time consuming way to find
an optimal transaction sequence. To rectify this issue, we added a maximization operator
that returns ϵ-tight bounds around the true max, and saw how this made it simpler to
maximize $(σA).
Lastly, we added fees to our model, and for various transaction sizes saw how the solver
could be used to find the fee level that would make the cost of transacting on an AMM with
high liquidity but which charges a fee equivalent to a fee-less AMM with low liquidity.

4.1
Theoretical Guarantees
In this section we will look at what guarantees can be made on the results obtained through
the tool.
Completeness
The tool can be said to be complete if there’s no false negatives where the solver reports
unsat, but the formula in reality is satisfiable.
As the search for a witness is bounded by k, we can only be complete to a depth of k, i.e.
if the formula is satisfiable at depth k + 1, we won’t find it.
However, within the bound of k, we believe the solver to be ”relatively complete”. Relative,
because we can only be as complete as the underlying SMT-solver, and although the
QF_NRA logic is decidable, Z3 has not been formally verified, and indeed we found a bug
in the QF_NRA logic where z3 erroneously reports unsat to a satisfiable formula [19].
Nonetheless we believe the ”high level” part of the solver to be complete as the partitioning
of the transition relation into sub-cases spanned by (A, τin, τout) doesn’t leave any sequence
unexamined.
Soundness
For the tool to be sound, we accept no false positives - i.e. no witness should be returned
to an unsatisfiable formula.
As we believe our transformation of the state-space and proposition into an SMT-encoding
in the QF_NRA logic preserves the structure of the problem, we believe our solver is sound
- of course provided that the underlying SMT-solver is sound.
One potentially unsound operation is ϵ-max, (or when using the decimals output mode
generally) as this relies on Z3 rounding algebraic reals to a decimal number. Inside Z3 this
rounding is done by squeezing an interval around the ”accurate” number (some root of a
polynomial), until the interval’s upper and lower bound are
1
16N from each other, before
returning the interval upper bound (where N is the number of decimal places to use).1
This means the ϵ-max operation may return a lower bound that is too-large by
1
16N and
hence can be deemed a ”false positive”.

4.5
Summary
In this chapter, we first considered the soundness and completeness of the solver - and con-
cluded that while our partition of the problem and transformation into an SMT-encoding
preserves the structure of the problem, a limiting factor is the completeness of the under-
lying SMT-solver. We concluded that the tool was sound, with the exception of the ϵ-max
operator, as this relies on numbers in Q \ Q being rounded to a decimal by Z3, which may
result in false positives if Z3 rounds up.
Secondly, we looked at whether we could establish completeness thresholds for various
combinations of M and A, and saw that while for |A| = |M| = 1 the threshold was
k∗= 1, for |A| ≥2 and |M| ≥3 the bound k∗could approach infinity for certain φ.
Additionally, for |A| = 1 and |M| = n we noted that any sequence λ could be reduced
to λn, containing at most 1 transaction per AMM achieving the same net-effect as λ -
provided that A has (or were given) suﬀicient funds in the initial state Γ0 to execute λn.
Next, we looked at the size of the search space, and noted that when using unconstrained
transactions, the number of combinations to check quickly grows to an untenable size
around k = 8. When limiting the ability of a subset of A to transact freely, we noted a
sizable reduction in the size of the search space. However, the exponential blow-up in size
from having a user free to transact still dominated the expression.
Thus, we finally looked at how we could remove redundant combinations from the uncon-
strained sequences Li, by drawing on some of the results established in the completeness
thresholds section.

Example 12. Induction (Non-depletion)
In this example we will see how leaving the initial state space Γs
0 entirely symbolic can be
used to prove safety properties of AMM swaps, such as the non-depletion of AMM reserves
(which essentially states that the reserves of an AMM can’t be exhausted).
A safety property states that something bad α never happens, i.e. ∀□¬α should hold. As
our solver only supports ∃♢, to reason about global properties over all paths, we must use
the equivalence:
∀□¬α ≡¬∃♢α
and show that ∃♢α doesn’t have a witness - however, we are still limited by the fact that
our solver only checks for witnesses of a length bounded by k. Thus if we don’t find a
witness of length ≤k we can only say that ∀□¬α holds for the first k states on paths in
Paths(Γs
0), but nothing can be said to the effect that it doesn’t hold thereafter. However,
this changes when doing induction on symbolic states.
Initial state, validity assumption
For the case of our transition system, as we don’t support the deposit function which
instantiates an AMM, we will need to assume that the AMMs in the state space of our
model are valid by construction. This means that if we don’t impose any constraints on
the initial state Γs
0, then the initial constraints I(γ0) only contain the constraints that
can be said to hold following a legal sequence of deposit transaction(s). These validity

Future work
In this section we will outline a few ideas for work that could improve the state of our
model checker.
Liquidity provider transactions, Deposit & Redeem
In reality AMMs not only support swap transactions, but also the ”liquidity provider”
transactions of depositing tokens into as well as redeeming tokens from the AMM’s liquid-
ity pool. Supporting these transactions would permit analysis of the returns earned by
liquidity providers, and thus the incentives for depositing resources into the AMM.
New scenarios to be checked might include how large deposits and redeems can be used in
sandwich attacks to affect slippage, or modeling initial states that initially have no AMM at
all, which would allow for stronger induction, since we won’t have to make assumptions
on the base-case that the initial state was validly constructed.
Complete, when?
The results of the completeness thresholds section were somewhat depressing, in the sense
that for most combinations of A and M, we failed to provide a proper bound on k∗(as we
saw due to the infinite precision numbers, values in φ could be scaled to make k∗approach
infinity). A future line of work could be to incorporate φ into the analysis, and say ”given
φ, we know the threshold to be k∗= . . . since . . .”
Alternatively, by changing the domain from infinite precision numbers to bit-vectors (e.g.
of length 256 as is the word size of the EVM), it might be possible to both model the do-
main of on-chain contracts more accurately, as well as get around some of the completeness
threshold issues that we saw arise from using infinite precision numbers.
Further reduction of search space
As the search space grows exponentially in k, it’s infeasible to check depths beyond k ≃10.
To make the solver more scalable for longer sequences, it would be imperative to remove
more sequences λs
i ∈Li from consideration.
It might be possible to remove more redundant sequences purely based on the contents of
the sequence itself (as seen in in theorem 9). However, another line of work could be to
inspect the formula φ, and use this to impose constraints on sequences in Li.
For instance, say that a subset B ⊆A of users must have a wallet different from their
wallet in the initial state to satisfy φ. Then sequences λs
i ∈Li lacking some sender A ∈B
may be discarded, as such a sequence will never satisfy φ. Along the same lines, if φ
implies A is converting some τ0 to τn, then only sequences that contain a ”route” from τ0
to τn should be considered.
Another alternative way to speed up the search could be a parallelization of the SMT-
invocation step - as each sequence in Li, represents a disjoint transaction path, checking
each sub-problem could be done independently.
Temporary balance overdrafts
Another line of future work could be to add a setting for ”temporary” overdrafts, e.g.
along the lines of taking a flash-loan. As we saw in section 4.2, if |A| = 1, then by allowing
A to take a (cost-free) loan in Γ0 the completeness threshold would be |M|, making it easy
to conclude when to terminate the search. Additionally, if we make the solver return the
amount ∆σA that was added to σA in Γ0 - it would enable users to identify how much to
A domain specific model checker for AMMs
49
==PAGE:58==
borrow, to enable their transactions - perhaps using the utility function to minimize the
loan amount ∆σA.
In [21] the authors introduce an AMM that allows for temporary overdrafts in user bal-
ances, swaps are then submitted to a queue until the balance of every user has turned
positive, in which case it may be cleared. By allowing overdrafts, the solver could be used
to assist in finding ways of clearing such a queue of swaps to reach a positive state.
Constraints
The constraints used to constrain the transaction relation simply tells the solver which
transactions it can use, but it’s impossible to constrain when it may use them. Thus a
future line of work could be to add an option for imposing an ordering on the available
set of transactions. This can implicitly also be used to shrink the search space, if the user
knows the solution sequence has a certain shape, he may impose it through these ordering
constraints and hence find a solution faster.
The query language also doesn’t support relating states of a different depth to each other.
It should be possible to add a new path formulae φ, which permits imposing constraints on
multiple states along a path π, or even relate them to each other as in υ(π[i]) < υ(π[i+j])
for some depths i and j.
Use domain specific ”sub-solvers”
The tool developed as part of this project is sort of a swiss-army knife that can be used
for many purposes. However, a lot of the sub-problems it may be used for already have
either analytic solutions[9], or eﬀiciently implemented solvers[8]. A phase could be added
to identify the ”class” of problem being solved, and if it falls into one of these categories
where a more eﬀicient solution exist, then delegate it to that solver - otherwise fall back
to the default of checking the formula iteratively through Z3.
Parameterize swap function
While the constant product swap rate seems to be the most popular among AMM-implementations,
some AMM implementations rely on other swap functions. By making the swap function
a parameter of the solver, it would enable checking properties on these as well.
Additionally, this would enable AMM-developers to check that the swap function of new
constant function market maker implementations exhibit certain desirable properties -
such as non-depletion of reserves.

7
Conclusions
In summary, the solver can be used in many settings and solve a variety of problems.
While specific solvers or analytic solutions exist for many of the sub-problems considered
in the examples of this project, no unifying solver capable of encoding every sub-problem
exists to our knowledge. Thus, we believe it to be the first general-purpose model checker
for the domain of AMMs.
However, there are still things left to be wanted - for instance the search space is still
very large for relatively short transaction sequences, and unless the state space is small
it would be infeasible to check formulae beyond k ≃10. However, with more analysis it
would almost surely be possible to shrink the search space further.
Another shortcoming was the lack of a definite completeness threshold k∗for most combi-
nations of A and M, which makes it hard to conclude when to stop searching. To rectify
this, it might be required to either inspect the specfic path formulae φ, to determine a
k∗- or alternatively change the number domain to bit-vectors, to mimic the semantics
of the on-chain AMM contracts, and get around the completeness issues that arise from
supporting infinite-precision numbers.
As discussed in future work, there are many possible avenues to improve or expand the
capabilities of the solver. Nonetheless, the solver is also very usable in its current state,
and lends itself well to gain intuition of a complex domain through active experimentation.

However, as soon as any of the above ⟨command⟩s are listed, the default behaviour that the

For both AVAILABLE and REQUIRED statements, the initial ⟨symbol⟩refers to the user

an amount v1 denoting the least amount required for the transaction to not revert.

amount.
