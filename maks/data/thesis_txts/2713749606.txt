Abstract
Product personalisation is a barrier for many software products in this globalised world.
One way to overcome this problem is to create a modular application that allows users,
or third parties, to add or modify application functionality using a proprietary programming
language. This project aims to develop a highôÙlevel programming language, a compiler
and Language Server Protocol (LSP) based tools to develop management applications
for businesses.
In order to obtain the necessary knowledge to create this solution, a review of the existôÙ
ing literature was carried out. The result of the project is a domainôÙspecific language with
separation between business logic and user interface that can be easily extended with miôÙ
nor modifications, an easily extensible crossôÙplatform compiler and a Visual Studio Code
extension that provides diagnostics during application development.

Programming Languages
Programming languages can be thought of as a tool developers use to design, implement,
document and debug their work. This definition is ambiguous because, as we will see in
this section, the word programming language in itself encompasses quite a large spectrum
of tools.
In order to understand what unites this a priori heterogeneous group of languages, it is
necessary to discuss the link between them: they are all formal languages. A formal
language is defined as a series of tokens (letters, numbers, or symbols) belonging to an
alphabet that, in turn, follow a list of rules. With this definition, one can speak of highôÙ
level languages such as C, Java or Python and lowôÙlevel languages such as assembly
or binary. This framework could also talk about nonôÙTuring complete languages such as
XML, HTML or even graphical languages such as Scratch or Ardublock. Despite this, the
latter does not have direct applications to this thesis, and we will not explore them further.
We will focus on theoretical research into textual TuringôÙcomplete languages.
Programming languages are classified in a variety of ways, and we will use chronological
order wherever possible to enable an understanding of the rationale for their classification.

History
Ada Lovelace is considered the first person to develop a computer language. In 1883,
she developed an algorithm for mathematical computing operations on Charles Babbageãs
mechanical computer, the Analytical Engine. Ada Lovelace and Charles Babbage were
forerunners of modern programming languages and digital computers, respectively.
Today we do not use analogue (mechanical) computers but digital (electrical) computers,
so we will focus on the latter. Due to their operation, the first programming language,
binary code1 emerged in the 1940s around the World War II. In those years, scientists,
mathematicians, and engineers wrote their programs in pseudoôÙcode2 and then transôÙ
lated this code into binary code using opcode tables. Once the translation was complete,
punched cards were used to tell the computer what to do.
Using binary at the hardware level is unintuitive and inflexible, which is why the assembly
programming language was born at the close of that decade. At this point, a fundamental
concept was introduced in programming languages: abstraction. The level of abstracôÙ
tion of a programming language refers to the amount of machine code hidden from the
programmer. Depending on the level of abstraction, we can classify languages into two
main groups: lowôÙlevel and highôÙlevel. If a programming language is closer to human
language, then it is of a higher level, and if it is closer to machine code, then it is of a
lower level. As shown in Table 2.1, lowôÙlevel languages emphasize speed and efficiency,
whereas highôÙlevel languages emphasize simplicity and portability between devices.

2.1.2
Classification
Programming languages can be defined in two ways: informally or formally. The informal
way uses peopleãs written language to describe a language, which makes it easier to
understand but complicates the precision of the language. On the other hand, the formal
way uses mathematical expressions to describe it, making it harder to understand, but
delimiting what can and cannot be done with such language.
Regardless of the format used, each programming language needs to define two compoôÙ
nents: its syntax and semantics. The syntax of a programming language is equivalent to
the syntax of the languages that people use. It describes how a piece of code is structured
to be valid. On the other hand, semantics determines the meaning of the symbols that
the syntax determines. For example, if we want to say that we add 10 and 4, the syntax
would specify that the form must be ã10 + 4ã, while the semantics would specify that this
equals 14.
The way these elements are described leads us to separate programming languages into
two groups: imperative and declarative. Imperative languages describe the steps to be
followed by a computer. In contrast, declarative languages are usually described as nonôÙ
imperative languages, i.e. they specify what is to be achieved, not how to achieve it,
through a sequence of commands. Examples make it easier to distinguish this classificaôÙ
tion. In C, an imperative language, variables and functions are specified for a program,
as well as how they must be executed to run a program. In SQL, a declarative language,
we call the SELECT query with the JOIN operator if we want to see the entries of two
tablesëƒ we do not have to specify the order in which the instructions must be executed.
As of today, some programs offer both paradigms, such as C#, an imperative language
that offers LINQ support for declarative code.

2.3.1
History
Any textual programming language can be developed in Notepad. However, this tool does
not offer many developerôÙrelated features. As highôÙlevel languages were developed, this
problem arose because, as the developer abstracted from hardware, the programs develôÙ

developers to develop code, manage files, compile applications and debug from a single

Other editors followed the same path, such as Borlandãs Turbo Pascal [41] in 1983 and
IBMãs Original Program Model (OPM) [42] in 1988. We did not have the IDE concept
we know today until 1991. This year, Microsoft released Microsoft Visual Basic [43], a
programming editor with multiple functionalities inside a graphical interface.
Over time, IDEs have acquired more capabilities, and today we recognize that a good
IDE must provide syntax highlighting, linting, code completion, refactoring, debugging,
and similar features. Writing an application for every language takes a significant amount
of time, and repeating the process every time the IDE was changed was not costôÙeffective.
It is in this framework that the design of the Language Server Protocol was born. LanôÙ
guage Servers were already available in editors such as Vim or Emacs. Furthermore, in

4.2

4.2.1
LEA Grammar
The initial version of the grammar was defined in a single file. However, to facilitate the
development and ease of use, it was decided to separate the generation of tokens from
the attributed grammar. For this reason, there are two ANTLR4 files to create the gramôÙ
mar: LEALexer.g4 and LEAParser.g4. The content of both documents can be found in
Appendix A, so we will only mention the most relevant aspects below.
During the development, it was indispensable to use the book ãThe Definitive ANTLR 4
Referenceã [49]. With it, we understood the requirements for writing correct grammars for
ANTLR4 using EBNF. However, two problems slowed down the development.
The first one did not appear in the book and is that, when using the ANTLR4 library, a
token cache is generated which, if not deleted, blocks the generation of new documents
with a not very descriptive warning: ãImplicit definition of token X in parserã. The solution
to this problem was to clean up all the autoôÙgenerated content before generating it again
instead of letting the library update it.
The second problem was a bit trickier to find and came from how attributed grammars
work. By default, topôÙdown attributed grammars cannot handle leftôÙrecursion. A leftôÙ
recursive rule is one that directly or indirectly invokes itself. This problem arose while
we were developing the types since a complex type can be made up of simple types. At
first, we decided to create a general type that encompasses simple and complex types.
After that, we allowed complex types to be made up of general types, which generates
a loop. ANTLR4 solves this problem, but only for direct recursion, not indirect recursion.
Until we realised the problem, we could not create structures or arrays, as we were genôÙ
erating an indirect loop. After removing the general classification, the error disappeared.
The same situation happened with expressions, but we solved it faster since we already
knew what was happening.
Another issue to consider during the grammar creation is the order in which the options
appear in a list. Although it was a minor issue, we did not realise it until we tested the
code generated in C#. Generally, the higher a rule is in the list, the greater the priority. It
is not always necessary to specify the correct order, but determining which operator1 to
deal with first is necessary.
It is worth mentioning that, to speed up the development, scripts were created that will
generate the automatic files from the grammars and move them to the correct folder in
the compiler. These files can be reused in the future in Windows.

Once the input is passed through the Lexer and Parser, the program sends the CST to

With this design, based on the visitor pattern, the ASTBuilder does not need to know how

5.1
Objective 6: Construct a compiler
During the analysis of the literature concerning compilers we have realised how crucial
compilers are in the world of computing. They allow developers to create ambitious apôÙ
plications with a high degree of abstraction, which would be unfeasible if they had to work
with machine or assembly code.
Throughout this project, we approached the design of our solution based on the four
main phases of compiler construction. We delved into lexical analysis, syntactic analysis,
semantic analysis and code generation. These phases use the result of the previous
phase to add new content or verify existing content, before sending the result to the next
phase. And as such, we have applied this design to our own solution
5.2
Objective 7: Build a Client & Language Server
IDEs are almost indispensable tools for code developmentëƒ they allow the developer to
understand what mistakes he is making before compiling his program, faster navigation
in large code bases, debugging their code among other functionalities. The Language
Server Protocol establishes a set of specifications when creating a Language Client and
a Language Server so that one IDE can support multiple languages without changing its
implementation.
Thanks to the research carried out in this project, it was possible to implement a functional
Language Server so that it can be reused in other editors in the future.
5.3
Objective 8: Evaluate the usefulness
The language expressiveness of our solution allows the development of the basic needs
for managing a schoolãs teaching staff, student body and finances, as shown in Appendix
C. However, it is somewhat limited in file management formats, only allowing the use of
CSV files.
The design used makes it easy to create new content in the future. However, the aim was
to create a solution allowing people without much technical knowledge to create these
applications.
This means that the solution meets expectations if we measure the usability of the project
by its flexibility, both in interacting with users and in the ease with which it can be extended.
However, we also wanted the process of creating these applications to be simple, and here
we failed. This is mainly because users need to create variables, structures and functions
to implement their business logic, terms that are not trivial for nonôÙdevelopers.

open a terminal inside the folder Solution.Compiler and execute the commands dotnet
build and dotnet run ôÙsource document.lea.
In order to use the LSP functionalities the developers needs to install the .vsix file found

B.2
User Guide
In order to run the compiled application the user must: open a terminal in folder OutputôÙ
Files.Application Name and execute the commands dotnet build and dotnet run.
B.3
Modder Guide
In order to add changes to the language, the modder must modify the .g4 files in the folder
Solution.ParserGenerator.Generator, execute the script full.ps1 and make the relevant
changes in the compiler.
On the other hand, if the modder wants to add these changes to the Language Server, it is
necessary to manually copy the files LEABaseVisitor, LEALexer, LEAParser and LEAVôÙ
isitor to the folder IDEExtension.LEAServerSolution.LEAServer.Tools.Parser.
If these
changes involved changes to the Compiler then the modder will also need to move the
modified files from the compiler to the corresponding folder inside the previously menôÙ
tioned directoy.
