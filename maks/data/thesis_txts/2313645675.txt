Summery 
In this thesis, the subject of generating a level and the sound from the same algorithm, is explored. This is 
done by going through the stages from prototyping the game on paper, to end up with a game, that is 
playable and tested in public, to see if there is a way to make the generation from the same algorithm. The 
generation is explored from three different sides; first a musical algorithm called a Markov Chain, second a 
total random algorithm and third a Curve algorithm, which is normally used for creating levels. This all ends 
up in some interesting findings, when it comes to the final user testing.

The problem to solve 
There are several ways of making levels and music in a game, most common for all of them is the element 
of something being random or pseudo random. Is there an algorithm that can be used to both the level 
creation and the music creation simultaneously? This will help making both music and levels at the same 
time, which might make the music and the levels less predictive.

Introduction 
In the modern gaming industry, there are a lot of talk about procedurally produced content, whether it 
being procedurally created experiences as in Tetris, procedurally created environments as in Minecraft or 
Procedurally generated music as in Spore. Procedurally generated stuff is, and has been for some time, the 
new black in the gaming industry.  But why not combine them, and what happens when doing so? In both 
cases the underlying algorithm must have a set of rules, something that makes it feel random or something 
that makes things feel like they belong together, and keeps the player interested in the game.  
One of the dangerous things about doing things procedurally, is that the experience lacks something, 
thereby making the end player lose interest quickly. Therefore, the algorithm needs to be something that 
feels dynamic, and should make the player feel that things are evolving. This is something that is true for 
procedurally created experiences, environments and music. So Why not combine the music and the level 
making into one thing? And how do we do so? 
In music generation, there are always a lot of talk about Markov Chains to generate a composition. So what 
happens when a production algorithm as the Markov Chain is being use for generating the level and the 
music. Furthermore, can we use the same seeds? Can we use the same chains? These are all questions that 
pops up when looking at combining generation of music and level, seen from the musical side of things. 
Seen from a level generation point of view, the most common methods is to either make chunks or single 
level elements, that is then repeated through different kinds of algorithms. What these algorithms have in 
common is that they try to make things belong together through seeds, difficulty and level elements. Could 
one these algorithms be used to make the music as well? Or be put into one algorithm as with the music?  
It could be interesting to see which of the algorithm types that does the best job of level generation vs 
music generation. Maybe the level generation algorithm is doing a better job of creating music or the other 
way around. Maybe the music generation does a better job at creating the level. It could be interesting 
seen both from a level designer’s perspective but also from a musician’s perspective. This could also mean

that some of the logic that are normally placed in tools like Pure Data or Wwise, actually should be part of 
the normal program logic.

State of the art 
As mentioned in the Introduction, there are different approaches when it comes to generating levels and 
music procedurally, but here there are also some similarities. Both things depend on some sort of seeds, 
and both depends on the things being put together will have a feeling like they belong together. 
With that being said, generally there are different approaches to how the seeds are generated. When 
talking about music, the most common method for creating it procedurally, is by using a Markov Chain. In 
level however, it often comes to feature selection whether it being things in the game like (Medeiros & 
Medeiros, 2014) where a feature is categorized like difficulty, immersion, control and concentration, or it 
being like the "original" platformer algorithm from (Compton & Mateas, 2014) where features are a 
derivative form that boils down to components.  
 
1.1 Music generation in games using Markov Chains 
For generating music to games, (Prechtl, Procedural Audio Now!, 2015) suggest and showcases an 
algorithm that uses a Markov Chain, to evolve music as a player progresses through a simple level, made in 
Unity 3D. The suggested approach is to have a set of states to go between, but doing so according to a set 
of parameters. 
 
==PAGE:16==
6 
 
The parameters that is being explained follows: 
Tempo of the state 
Dissonance 
By using an algorithm to transfer between the two states, the music is no longer becoming two or more 
files that cross fades between each other. This essentially means that it is no longer two or more files 
playing back when transitioning between two states. When transitioning between two or more states using 
sound files, it also means that you are going to that state. 
But by using the suggested approach, there is a more fluent progress between the states. This mean, that 
when going from one state to another, the music is still changing the parameters, therefore the state could 
easily change to another state by adjusting the parameters, where time would have changed it into another 
state.  
Or as to quote Anthony Prechtl, when he addressed this topic in a presentation during a Procedural Audio 
Now event:  
What you will get in the middle is not two pieces of 
music playing but, you know one music source going 
with sort of half the features of each state 
(Prechtl, Procedural Audio Now!, 2015)

1.2 Level generation using Algorithms 
There are a number of ways to make different levels for different game types, but most of them relays on 
some kind of feature selection. These could be a curve generation algorithm, which tries to make use of a 
curve for how difficult the next section/pattern should be, like in ( Bazaz, Kulkarni, Desai, Jaiswal, & 
Chauthe, 2015), or it could be a platformer algorithm that uses a set patterns, cells and cell structures like 
in (Compton & Mateas, 2014). 
1.2.1 
The traditional platformer way of making level 
For both approaches on making levels, some kind of component/feature selection is important. If you take 
a look at the normal approach from (Compton & Mateas, 2014), you will see that the idea is to make a 
pattern from components, and then expand the components further into a cell structure. The cell structure 
is defined as an encapsulation of some of the patterns, where the only thing relevant is the possibility of 
the player coming from one end the another.  
When it comes to pattern building, it all starts with the start and the end of the pattern, and then an 
algorithm is trying out each component, to fill the gap between the start and the end. The most optimal 
component for filling the empty space in between, is chosen. 
The patterns are divided into four different kinds patterns: 
1. Basic patterns – no variation 
2. Complex patterns – same components but changed based on a sequence 
3. Compound patterns – two different types of components 
4. Composite patterns – two components that forces the player into making an action (e.g. the 
famous plants in the green tubes from Mario) 
For the cell structure, it all starts with an initial cell. A new cell is created when a player comes to the end of 
a cell. Then, the new cell is determined by a predefined set of parameters, so each level know which factors 
==PAGE:18==
8 
 
that the level should contain. A new cell is generated and populated with a pattern as described earlier, and 
thereby a level is created. 
1.2.2 
Using a curve algorithm 
Another way of building is by using difficulty curves, to determine which components should be the next to 
pop up, depending on the difficulty of the next point. The way ( Bazaz, Kulkarni, Desai, Jaiswal, & Chauthe, 
2015) suggest to do this, is by having a base point, and then generating the curve from that point. 
The base point of the algorithm is set depending on what patch is currently being used, and the minimum 
difficulty used, when making the level. 
Then the number of high points and low points are defined, and a curve is generated from this, depending 
on how the different characteristics and how much the difficulty will vary from the base point. 
When the general trend of the curve has been made, a number of control points is made on the curve. A 
curve fitting algorithm is used to create these points, and the whole curve is now available for making the 
level. Then, the features or components are fitted to the curve, given their difficulty. These features or 
components can either be procedurally generated or generated by a level designer, to have some human 
touch. 
1.3 Combining both music and level generation with one 
approach 
There is a lot of research based around the two areas of either making music or making level, but what 
happens if an algorithm for making music is used to make both the level and the music? Or the other way 
around, can the same seeds be used for making both the music and level? What are some characteristics 
that can be combined?  
By using the same algorithm to generate both the music and the level, small companies could save time 
and/or money. This will allow them to focus their time and energy on testing and tuning, plus making rapid 
prototypes. It could also be used for opening new possibilities either generation of procedural music or 
procedural level generation in games. 
So the approach here will be to conduct three experiments of different kinds. In a simplistic endless runner 
game, could the same algorithm be used to drive both the level and the music? The three experiments that 
will be conducted will be as follows: 
1. Using Markov Chains 
2. A random algorithm 
3. Using a curve algorithm  
==PAGE:19==
With these three approaches, it will be tested if it may be possible to unite these approaches from the 
opposite world, into a game that is playable, and with music that is bearable to listen to while playing.

2.1 Choosing the software 
There are different tools for making the game development run easy and efficiently. Most of them comes 
with the cost of overhead, either by being inefficient when it comes to performance, or by making the 
application files big. As this project aims to have a small footprint, and to be able to run on mobile 
platforms, specifically the Android, three contenders stood out, when it came to choosing the right 
framework. Among the contenders were the following: 
 
Unity3D 
 
Unreal Engine 4 
 
libgdx 
All the frameworks have their pros and cons, which also leads to the discussion of which tool would be the 
most beneficial for implementing the game. 
==PAGE:21==
When it comes to the platform, development experience in different languages also plays a role in choosing 
the right platform for developing the game.  But performance is also a big factor, as the game is 
procedurally generated on a device with limited resources. 
From experience with Unity3D, it has a tendency to make large APK files for the Android system, as well as 
having big performance issues on some devices. This could maybe be circumvented, but it was chosen not 
to try to do this. 
When it comes down to Unreal Engine 4, it is purely the lack of experience with the game engine, that 
made the choice easy. The learning curve is much greater then when using libgdx or Unity3D, but the 
performance is the among the best when it comes to mobile development of games. The C++ libraries are 
powerful and fast. 
Then there is libgdx, a small and powerful framework, that does not come with a graphical interface as 
Unity3d or Unreal Engine 4. It can be used in any IDE that supports Java as a language, and it is easy to 
begin with. The biggest issue with libgdx is the build system Gradle, but libgdx is lightweight and can handle 
2D and 3D as the others, and it is easy to program with. Therefore, the tool of choice when it comes to 
development, is libgdx. 
 
The choice of development platform also influenced the choice of what tool to use, when it comes to the 
sound design. If another platform than libgdx was chosen, it could have affected the choice of sound tool. 
Therefore, the libpd was chosen, as it has a Java wrapper, and can be use on mobile devices. If another 
development platform was chosen, then there would have been a possibility for using tools as: 
 
WWise 


2.3.1 
Explaining the Markov Chain 
The Markov Chain is seen as an important approach to algorithmic music (Collins, Laney, Willis, & 
Garthwaite, 2011). And was used was as the first algorithm in the game. The Markov Chain, which is a 
production algorithm, where the next stage of the level is dependent on what was produced before. This is 
done through a matrix where each row gives the possibility of what the next step of the algorithm will be.

2.3.2 
Explaining the random algorithm 
The random algorithm is doing exactly what it sounds to do. There are no seeds used to make this 
algorithm. Instead, it uses the random from Java to choose what feature should come up next. The 
implementation of Java's random is returning a pseudorandom value, which is uniformly distributed 
between 0 and a given value. In other words, it returns the following: A number that is bigger or equal to 
zero, but less than the specified value given as a bound.: 
result = on element from the range [0, given value [ 
So, if there are five different features, the result of the random algorithm would be a feature in the range of 
feature = a feature from the range of [1,6[ 
 
2.3.3 
The Curve algorithm explained 
The third and last algorithm used is a curve generation algorithm found by ( Bazaz, Kulkarni, Desai, Jaiswal, 
& Chauthe, 2015). This algorithm is divided into two stages; The first part of the algorithm is used for 
making a basic curve, and then taking some control points. The second part of the algorithm is to use a 
Cubic spline to make a curve, that is more fluent and smooth.  
The first part of the algorithm is defining low points and high points on the curve, this is done by making a 
line of 2000 points, and then modifying points if they have not been touched before.  The rules for 
modifying the points is set based on possibilities of: 
1. How many peak points should there be? 
2. How many break points should there be? 
3. How big should the peaks be? 
4. Should the current peak point be higher? 
5. How low should the break be? 
6. should the current break point be lower? 
To each of these, there are also an associated probability of how often or how high/low a point should be. 
The probabilities of the different characteristics can be set up to show how often and with what limits the 
basic curve is generated from.

The second part of the algorithm is the Cubic spline, which smooths the curve and makes the curve more 
well-rounded from the 21 control points. The 21 control points is taken from doing a control point every

A Cubic spline is a third-order polynomial, which passes through a number of control points, to create a 
smooth curve.

2.5 Summary of the design 
After going through some paper prototypes and rough game play demos, an idea of a simple game to see if 
the three algorithms could produce both sound and level at once, came up. The rules for the game are 
simple but contains the most basic elements for most endless runners; it contains three lanes and elements 
traveling towards the player, who should try to dodge and pickup elements. 
The tools chosen to make the game was libgdx and libpd, even though there are others that can do the 
same. On the surface, libgdx and libpd seems like a good choice because of the lightweight framework of 
libgdx, which comes without a lot of overhead and has the features that a small game engine should have. 
Furthermore, the libpd was chosen as it is also can be built for java, and can be used on an Android 
platform. 
But because of big challenges making libgdx and libpd coming together as a unit, mostly because of the 
build system and making libpd build for the right targets, a lot of time was spent trying to debug on how to 
get the two frameworks to work together seamlessly, which was never fully accomplished. 
Three algorithms were found that have different approaches to building the level, and therefore also the 
music. The algorithms are enlightening the problem from three sides; from a musical point of view, a level 
creation point of view and a pseudo random point of view with no structure.

Implementation 
Throughout the implementation, different kinds of problems were meet when it came to both 
implementing the music and the game itself. The structure was altered a couple of times, trying to figure 
out what the best approach is to implementing the solution. 
3.1 General implementation 
The general structure of the game is divided into different phases with different responsibilities, much like 
the way of (Compton & Mateas, 2014) have their structure build into different layers, using patterns and 
cell structure. But it is also inspired by (Adrian & Luisa, 2013) where the level designer have some control 
over the difficulty curve. 
All the different types of shapes are originating from one single basic shape. The shapes will either pass 
through the screen, or is set as a static element on the screen, with different interactions. The different 
types of shapes are: 
 
Number - destructible number that comes down from the top, if correctly destroyed (if does not 
give ten) the number is added to the score otherwise the player loses a life. If the number and the 
players number added together gives ten the player gets some bonus score, otherwise the player 
loses a life. 
 
Wall - Indestructible shape coming down from the top, player dies if he touches it 
 
Life - extra life on the course (also displayed statically) 
 
End - Has a number on that changes when clicked 
==PAGE:29==
Not all the shapes made it to the final game. 
A feature pattern is a combination of different elements as shown in Figure 4, and is made by a game 
designer. The feature patterns have the shapes that is used for the pattern, the tone to be played when 
spawning and the difficulty that the Curve algorithm uses. This results in easy adjustability, when it comes 
to both tones and difficulty. 
While the algorithm is running, it will select the best feature pattern from all these patterns. To keep the 
algorithms on an even playing field, a total of seven different feature patterns were made. 
The shapes of the feature patterns are made from a three times one row of the different shapes in the 
game, but a place could also be empty. Therefore, if implemented totally random, there would be a total of 
33 = 27 feature patterns. But as stated earlier, only seven feature patterns were implemented, due to one of 
the algorithms being especially hard to make seeds for, but also for making all the algorithms having an 
even playing field to be judge from.  
Then, when the game is running and it is time for spawning a new row of shapes, the level asks the current 
algorithm for the next feature pattern. The level then receives both the shapes that needs to be rendered, 
but also the tone to play. This tone is then sent to the sound engine, which then plays the tone from the 
feature pattern. The shape that the level is getting, is sent into the shape queue and is being rendered with

3.2.2 
Implementation of the Curve algorithm 
When it comes to the curve algorithm, the first thing was to make a basic shape as described in The Curve 
algorithm explained. This was done by following the recipe from ( Bazaz, Kulkarni, Desai, Jaiswal, & 
Chauthe, 2015) where the first thing was to alter all the points to be one of the normal points. Afterwards, 
it was a matter of making the high points, and checking if the high point should be converted into a peek 
point. This is where the probability was used to see if the point should be set, or it should be even higher. 
The same applies for the generation of the low points, but the point can be converted to an even lower 
point. After this, the 21 control points where found and saved as a patch.  
Then, when time comes to find the next step, the function of libgdx for finding the Cubic spline, was used to 
find the next difficulty to use for deciding the feature pattern. Because some of the feature patterns have 
the same difficulty, the feature patterns order where shuffled using a Fisher-Yates shuffle as described in 
(Durstenfeld, 1964) to make the choosing of feature pattern fair. 
Hereafter, the feature pattern was returned as described in General implementation. 
3.3 Performance when it comes to the algorithms 
When it comes to choosing the next step, there were two different ways due to the workings of the 
different algorithms. The first method was straight forward, as the next step was calculated directly when 
the next feature pattern was needed. 
==PAGE:33==
The other way was to calculate a block of next steps to choose from. This was done due to the inner 
workings of the curve algorithm, where the curve would be calculated. The next feature pattern would be 
found by stepping along the curve. When reaching the end of the curve, a new curve would be generated, 
and the new points would be found here. 
Comparing the two different ways of fetching the next step of the algorithms, could have an impact on the 
performance of the device the game is running on. This performance difference can be seen in Figure 5. 
 
 
Figure 5 Performance using Markov vs Curve algorithm 
 
When looking at the performance of the two algorithms, there is not that big of a difference when it comes 
to ram usage. But over time, it can be seen that the game mode 3 using Curve algorithm, is using more 
kernel CPU overall, as it uses 11.65%. In comparison, game mode 1 and the Markov is using 5.66%. It can 
also be seen that game mode 3 has more spikes than game mode 1. This all makes sense because both 
algorithms are utilizing the CPU for generating the levels, but no instance of the game is showing overload 
of that usage. Both ways are acceptable both with the ram usage and the CPU usage.

3.4 How the sound works 
The sound was implemented in two ways, because some of the music was implemented directly in Pure 
Data, and other things of the music was implemented directly in the game itself. As seen in Figure 4, the 
level sends data directly to the sound engine in the game, where the request of the levels is processed. 
From the sound engine, the data is being processed and sent to libpd, where different processes are 
triggered. The processes that can be triggered are as follows: 
 
BPM controlling the speed of the beat 
 
Three frequencies for making the nodes 
 
Three triggers for playing the nodes 
 
Three triggers for stopping the sound 
The BPM was controlled by the speed of which shapes also was spawned. This means that if the speed of 
the next shapes in a feature pattern was set to 100, then the BPM of the music would also be set to a 100. 
This would ensure some consistency between the shapes and the music. Furthermore, as mentioned 
earlier, the tones were played as the feature patterns were spawned. 
For making the tones, the basic idea was to use the keys from the C major scale so the tones had some kind

3.5 Challenges doing the implementation 
The biggest challenge in the implementation phase was to get the instruments working. This is due to some 
of the functions in libpd is using libraries, that is not directly accessible in libpd.  
The way to circumvent this, is to find the library files, and then make a new Make file for these elements. 
Then, the Make files had to be integrated into the Gradle build system, so the files would be built correctly. 
Furthermore, the way that libpd is initialized in the android portion of the code, it does not look for the 
newly created extras by itself. So a static search path has to be set up, which points to the newly created 
files from the build. It took a lot of time figuring out that these elements were missing, because some of the 
elements that were missing, is part of vanilla pd.  
Vanilla pd is the simplest version of Pure Data, which libpd is built from. But libpd needs to be told where 
these files are, otherwise it will not point to it. When debugging libpd with libgdx, no errors are given. So 
the procedure of getting the error is a tedious process of trying to print out the values in Pure Data, and 
sending them to the console of Android Studio.  
Doing the implementation process, it was also found that some DSPs couldn’t handle values bigger than 1 
or smaller than -1. This caused some crackling sounds on some phones, and was circumvented by turning 
down the volume on some of the instruments. This did not solve the problem as a whole, but made it less 
frequent. A better solution was unfortunately not found. 
3.6 Summary of implementation 
When it comes to the implementation of the game, the way of dividing it into different processes by 
making the shapes, feature patterns and the choosing the patterns from an algorithm, made it easy to try a 
lot of things, while it also made a simple structure of the underlying classes. It also provided a good 
transition between the selection of the feature pattern to the sound engine. 
When it comes to the sound engine, the split between responsibilities of libgdx and libpd because of the 
basic beat was made once. The rest of the sound and the game was all done from the game, which meant 
that there were less switching between programs.  
Furthermore, this also enhanced working with the different kinds of tones, as they were shifted a couple of 
times to be placed at a higher or lower  in frequency.

known theories. Therefore, it was necessary to collect data from a questionnaire, to be able to build a 
theory.

4.3 The making of the questionnaire 
Questionnaires are normally quantitatively based, because you receive a set of measurable data based on 
numbers or simple replies, e.g. yes/no. In this case however, it is chosen to use a set of questions which 
also provides some qualitative data, which aren’t measureable in the same way. This proved to be a 
challenge, because qualitative data cannot just be populated into a diagram or used to create calculations 
from. Still, it is chosen to bring them into this thesis, because it brings another point-of-view to the 
assignment, which wouldn’t have been brought to sight otherwise. Having three questions about the view 
of the level brings an aspect, which cannot be achieved by the quantitatively based data. It has to be said 
though, that without the quantitatively based data, there is nothing to support the qualitative based data 
and thus making them more valid in the study. Therefore, this questionnaire is built with both sets of data. 
==PAGE:43==
Quantitatively based data provides the opportunity to see if the test persons says one thing, but thinks 
something different, or if there are more nuances in the reply. They can for example, answer yes to 
whether the music follows the level, but without the qualitative description of what they actually think of 
the level, you cannot analyze your way to the possibility, that it means the music is to boring, because the 
level was to boring. The qualitative data gives you the opportunity to analyze the areas which aren’t 
measureable, and also when it comes to individual perceptions from the test persons, there are limitation 
on how far you can go with numbers and figures alone. It is though, a challenge to make an analyze on 
quantitative data, because of the possibility to misunderstand the collected data, and see associations 
which doesn’t exist. Another challenge by writing qualitatively based questions in a questionnaire is that 
you might only receive keywords. One keyword can be interpreted differently by different people. This is 
where you have to be critical, and find some quantitatively based data to support the analyze based on the 
qualitative data.

Where to take it from here 
As the test results shows, it is possible to make the level and the sound with a single algorithm. But using it 
in other scenarios, besides infinite scroller games, might be hard. This is because the music and the level 
generation is so depended on each other, that if one of the things stops, both them will stop. So, in order to 
make this for other type of games, other techniques have to be developed, or a clever way of setting the 
rules and reusing the algorithms has to be developed. 
But when looking on the isolated environment of the infinite scroller type games, these techniques could 
be used in a commercial matter. Though, they have to be further developed, as the test results also shows 
that music and level scores are around average, tipping towards the better end. But there are two reasons 
that this could be used in commercial products; first, the music in the game is matching the level, and is 
always changing, which would make the music less boring, and therefore making less people turn off the 
music. This would be hard to do if using different algorithms for making the level and the music, as it is 
likely not to be matching, making the music or the level seem odd. Second, the money aspect in developing 
a game. By using an algorithm, money could be saved on hiring a composer, where the risk is that the music 
becomes boring or repetitive quit fast. This would again be solved by using one algorithm, because the 
algorithm is already implemented. However, a developer with experience in making procedural music has 
to be present on the team, otherwise there will be no money saved, and a lot of time would be put into 
figuring out how to use the musical tools. 
Another interesting way to take the research, would be to dive into the psychological aspect of combining 
the music and the level. There seems to be a connection between the difficulty of the level, and how well  
the music is perceived. This could though also be the question when it comes to making games in a non 
procedural way. It could be fun to see if this is only the case, when using the same algorithm for level and 
music generation, or if this is a general perception, and the reason for it. 
 
==PAGE:45==
 
Conclusion 
After examining the problem, it is found that a single algorithm can be used for creating both the music and 
the level. The players enjoyed how the level and music belonged together. But the players also show some 
interesting behavior, when it comes to how difficult the level is. The more the player thinks the level is 
difficult, the better they rate the music. This is interesting because it touches on some psychological things, 
when combining the level and music.  
Another interesting thing is that in general, when generating both the music and the level, the Markov 
algorithm, that is normally used to generate algorithmic music, gets the lowest point across all factors. This 
could be because of the feedback of the play testers, when it comes to them preferring other kinds of 
music and levels. Therefore, it would be interesting to try to use the Markov algorithm in another way, to 
see if this is a general trend that the Markov algorithm does not do as well. 
It is also interesting that the Random algorithm gets an overall better score than the Markov algorithm, as 
both the level and the music is random. It points to the tests persons, in general prefers the unknown over 
the structure that the Markov algorithm comes with. 
All in all, it is possible to make the algorithms to generate both, but the curve algorithm is the preferred 
way.
