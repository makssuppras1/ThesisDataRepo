Summary
Audio processing can be computationally very intensive. For this reason, in many
scenarios, a software oriented approach can not satisfy the required performance.
On the other hand, a hardware approach can be very complex and expensive
because it needs a dedicated hardware for every speciﬁc task. A balance between
software-oriented and hardware-based approaches is thus necessary to address
complex processes like in the audio processing framework.
In this project the Dynamically Loaded Hardware Library (HLL) technology is
used to realize a complete Audio Processing Interface. The aim of the work is
to demonstrate that HLL technology allows the acceleration of complex Digital
Signal Processing (DSP) algorithms on FPGA based platform.

Chapter 1
Introduction
Since the birth of silicon devices, designers frequently face the problem of ﬁnding
the best solution for a determined task against the trade-oﬀbetween costs,
performance and time to market. For years Application Speciﬁc Integrated
Circuits (ASIC) have been the most used integrated circuits. They are designed
to implement a single speciﬁed task that will be the only function performed
for the entire chip’s life. ASICs way allows in general for high performances but
with long development time, very high costs (unsustainable for little/medium
scale production) and, of course, no re-programmability.
When the Programmable Logic (PL) architecture showed up on the market, a
new device was vastly used in embedded systems because of its ﬂexibility. It was
the Field Programmable Logic Array (FPGA). The FPGA architecture is based
on the Logic Elements (LE), they are linked between them by programmable
interconnections and allow designers to directly conﬁgure the integrated circuit
(IC) using Hardware Description Languages (HDL) and reducing the design time.
This huge advantage is paid with speed, area optimization, power consumption
and design for testability. In conclusion, both ASIC and FPGA are technologies
with diﬀerent beneﬁts, however their diﬀerence relies on costs, Non Recurring Ex-
penses (NRE), performance and ﬂexibility. Figure 1.1 shows which one matches
up better to the most signiﬁcant project requirements.
==PAGE:16==
2
Introduction
Figure 1.1: FPGA vs ASIC, project comparison
Today FPGAs are able to perform the functions of entire systems into a single
chip thanks to special additional hardware like Block Rams, DCM modules,
MAC’s memories, High Speed I/O, embedded CPU etc. that are built on the
same chip. Therefore the term FPGA is not exhaustive anymore, these devices
are now called All Programmable System on Chip (AP SoC). Practically, they
are devices that integrate the software programmability of a processor with the
hardware versatility of an FPGA, providing the enormous potential of Hardware
Accelerators [20]: the use of hardware to perform some functions signiﬁcantly
faster with respect to what is possible in software running on general-purpose
CPUs.
Hardware acceleration is not a novelty. Typical examples of dedicated hardware
are graphic cards or sound cards on our Personal Computers, but the idea of
general purpose re-conﬁgurable hardware accelerators is possible only thanks to
reprogrammability of FPGAs. So, by integrating CPU, DSP and mixed signal
functionality on a single device, we could be able to infuse customizable intelli-
gence into today’s embedded systems to suit unique application requirements.
This concept is graphically explained in Figure 1.2
Moreover, an interesting approach involves now the conﬁguration of the FPGAs.
Instead of a full chip conﬁguration, it is possible to re-program speciﬁc spatial
regions of the PL with new functionality while the application is running. This
is the Partial Reconﬁguration (PR) technique. Thanks to this design feature, we
can overcome the mentioned limitation of FPGAs. For example, we can use a
==PAGE:17==
3
Figure 1.2: System on chip, an IC that integrates the major functional elements
of a complete end-product into a single chip
smaller and cheaper FPGA to implement several accelerators that require more
resources, and then space, than are available thanks to the partial reconﬁguration
technique. This would allow us to use only speciﬁc parts in speciﬁc time windows,
saving a lot of the total cost and a large static power consumption. However, it
is needed to ﬁnd the right application ﬁeld in which this approach can be used.
Regarding this, the concept of Dynamically Loaded Hardware Library (HLL) can
be very interesting. It works like the Microsoft Dynamic Link Library (DLL):
anytime an executable calls a function from the library, the operating system
loads the DLL and executes it. Let us imagine now to substitute these software
libraries with hardware libraries and the consequent reduction of the execution
time [18]. This explains the hardware-acceleration concept and paves the way to
the deployment of this methodology in server farms, real-time processing system,
Embedded Systems and Internet of Things. These systems can accelerate speciﬁc
tasks when high performance is requested.
Our idea is to make the partial reconﬁguration not just a design feature, invisible
for the user, but a real product feature. The audio processing ﬁeld could be
perfect for this purpose: today, the digital audio processing world is mostly
software based. Performers utilize a single system (usually a DAW, a Digital
Audio Workstation: a Personal Computer equipped with professional audio
board, speakers and controllers) in which they execute a lot of so called Virtual
Instruments plugins (VST). These are just software that run on the system.
==PAGE:18==
4
Introduction
Moreover, very low latency is essential. If these software become too complex
and heavy for the hardware, a new DAW is needed because the so called "system
requirements" are not satisﬁed anymore.
Consider an hardware based plugin: the partial reconﬁguration allows to over-
come the need of a dedicated hardware for every speciﬁed task and obviously
provides a much lower latency system with respect to a software based plugin.
A system that can be reconﬁgured at hardware level and does not need anymore
of hardware requirements, putting on a nearer level the software update with
the hardware upgrade.
The aim of this work is to create an embedded audio signal processing device
using Dynamically Loaded Hardware Library on a re-conﬁgurable System on
Chip and evaluate performance, beneﬁts and possibilities. The audio processing
ﬁeld is chosen because it needs all the requirements underlined in Figure 1.1
such as: fast time to market, low frequencies and prototyping.
About this dissertation, in Chapter 2 the target SoC platform will be described
with a detailed explanation of the main elements used for our project and the
design tools adopted, while Chapter 3 will describe the making of our HLL library
for audio processing application. In Chapter 4 the hardware design side will be
presented with a focus on partial reconﬁguration elements and work ﬂow. Instead,
Chapter 5 will describe the other side of the project: the software development
for a user application both with and without a Linux-based operating system.
Chapter 6 will be dedicated to the ﬁnal thoughts and future developments.
The results and the skills obtained in this work represent an important step
towards a new concept of designing, based on a software/hardware knowledge and
on how they can be merged to achieve the best results in terms of performance and
power consumption. When targeting FPGA-based SoC computing platforms,
software and hardware designers must face the inevitability of dealing with
HW/SW co-design [23]. New engineers must face this unbreakable bond that
probably will be the base of the future development of electronic systems.

2.1
ZedBoard
The Zynq Evaluation and Development Board is a low cost, community-based
development kit that is ideal for designers want to made a rapid prototype
of an embedded system idea or a proof-of-concept development. The Board
is a joint project between Xilinx (the FPGA owner), Avnet (the distributor),
and Digilent (the board manufacturer). It is based on the Xilinx Zynq-7000
Extensible Processing Platform. Thanks to the combination of a dual Corex-A9
Processing System with 85,000 cells of the Series-7 Programmable Logic, this
board can be used for a wide range of applications. In fact, a lot of on-board
peripherals and expansion capabilities are provided by the board in order to add
versatility to the already mentioned performance features. Figure 2.1 shows our
board and Table 2.1 shows the technical speciﬁcations.

AXI-Interface
The PS and PL are linked through the various interfaces:
• Two 32-bit Master AXI ports (PS master)
• Two 32-bit Slave AXI ports (PL master)
• Four 32/64-bit Slave High Performance Ports (PL master)
• One 64-bit Slave Accelerator Coherency Port (ACP) (PL master)
• Four clocks from the PS to the PL
• PS to PL interrupts
• PL to PS Interrupts
• DMA peripheral request interfaces
Advanced eXtensible Interface (AXI) belongs to the ARM AMBA 3.0 open
standard and makes achievable the interconnection between the most important

Originally developed by ARM for micro-controllers, the last version of AMBA

footprint with a simpliﬁed interface.
• AXI4-Stream: It allows burst of an unlimited amount of data because is a
no address phase transaction that means it is no memory mapped.

2.1.4
PCAP and ICAP
In order to program the FPGA, the Zedboard provides a conﬁguration signal
set composed by: Processor Conﬁguration Access Port (PCAP), Conﬁguration
Status, Single event upset (SEU) and Program/Done/Init. This set is connected
to ﬁxed logic inside the PL conﬁguration block.
In addition to PCAP, a block called Internal Conﬁguration Access Port (ICAP)
can be used to reconﬁgure the PL. The main diﬀerence between PCAP and
ICAP is the physical position on the board. The PCAP can conﬁgure fully o
partially the PL, it is placed outside the FPGA, precisely in the PS. The ICAP re-
conﬁgures the FPGA from the FPGA itself because it is placed inside the FPGA,
clearly it cannot do a fully reconﬁguration but just a partial reconﬁguration. On
the other hand, performances are also diﬀerent: The PCAP interface can send
and receive 32-bit words at 100 MHz rate, resulting in a maximum theoretical
bitstream transfer speed of 400 MB/s for plain PL conﬁguration. Regarding
ICAP, we can build and synthesize our custom high speed ICAP interface,
optimize it and reach a theoretical throughput of 710 MB/s [18].

2.3
Partial Reconﬁguration
As hinted in the introduction, the principal FPGA vendors have introduced a
new and interesting functionality in order to give a greater amount to ﬂexibility
to projects: The Partial Reconﬁguration (PR). This capability allows us to re-
program speciﬁc spatial regions of the FPGA while the entire system continues
to run in the rest of the device. So we can dynamically change part of an active
==PAGE:30==
16
Background
design. In this scenario, the FPGA can be divided in two regions, one, called
static region, in which all the logic that needs to remain always operative are
situated and another, called dynamic region, made of one or more re-conﬁgurable
modules. The reconﬁgurable parts are made with an out-of-context workﬂow
and imported in the main Vivado project. The logical and physical connection
between static logic and reconﬁgurable logic is made by the so called Partition
Pins. They are automatically created for all Reconﬁgurable Partition Ports and
play an important role about the deﬁnition of the generic reconﬁgurable module.
As written before, partial reconﬁguration can be done both through PCAP and
ICAP.
Figure 2.12: Basic Premise of Partial Reconﬁguration [14]
The purpose of Partial Reconﬁguration is to allow the device hardware resources
to be time-multiplexed, similar to the capability of a microprocessor of change
process, but, in this case, the device is switching at the hardware level. This gives
two beneﬁts: the ﬂexibility of a software implementation and the performance
of a hardware implementation [14]. Thanks to this technique, several diﬀerent
applications can be developed demonstrating the power of this technology:
• Networked Multiport Interface. It can be implemented as Reconﬁgurable
Module and changed in a single device as needed.
• Conﬁguration by Means of Standard Bus Interface. Thanks to PR we
can create a dynamically variable port using an interface standard more
compatible with the system architecture
• Dynamically Reconﬁgurable Packet Processor. Partial Reconﬁguration can
use in order to reconﬁgure a co-processor in the FPGA
• Asymmetric Key Encryption. In this case PR is an enabling factor for
some new application and is a very secure method for protecting the FPGA
conﬁguration ﬁle.

Thanks to PR we can obtain a long list of beneﬁts such as FPGA size reduction
(with consequent reduction of cost and power consumption), ﬂexibility in the

computing. Finally, we can assume that Partial Reconﬁguration enables new

• PCAP: The primary conﬁguration mode for Zynq-7000 AP SoC designs. it

inside the FPGA, it allows only Partial Reconﬁguration.

5.2
Linux-based Software Development
Operating systems (OS) are generally used to permit multiple applications to run
sharing the resources of the CPU in the most eﬃcient way. This feature is better
known as multitasking. In addition, OSs provide fundamental abstracted services
such as networking, virtual memory, hardware devices support interrupt handling
and ﬁle system. Furthermore OSs allow designers to save a lot of time dealing
with the low level complexity of hardware thanks to an extremely high-level
abstraction. Well known operating systems are Linux, Android, Windows and
Mac OS X.
Embedded Operating Systems are designed to run on platforms that can not
have huge hardware resources. To increase the portability of this devices, they
require low power, small memory and limited computation. Therefore, the aim
for embedded OS is to oﬀer basic serveries and high level of portability and ab-
straction. Our project matches these requirements. We need a portable platform
that can translate the massive complexity of the partial reconﬁguration technique
to an user level feature and, at the same time, can guarantee portability and
performance. There are many types of embedded OSs for the Zedboard, mostly
Linux-based. They provide diﬀerent solutions for diﬀerent application ﬁelds.
Our choice was done considering the one suiting our needs in the best way.
The ﬁrst operating system analyzed was Xillinux, a Linux distribution for Zed-

hardware based audio processing system that can be dynamically reconﬁgured.

architecture and how to match together hardware and software to make a ﬁnal
embedded platform ready to be used. All this has been explained step by step
to allow the reader to replicate the design and, maybe, ﬁnd new solutions and
improvements.
We have demonstrated that this solution can be very interesting for the audio
processing ﬁeld and in general for hardware accelerators for embedded systems.
The audio eﬀects that we have created in this project have a demonstrative
purpose.

problems. Our main target was to create a platform ready for this technology.
The development of algorithms that needs really to be accelerated in hardware is

6.2
Future Development
The project is obviously open to be upgraded from various points of view. This
version can be described as just the beginning for a new concept of a dynamically
hardware library based audio processing system. Below some possible future
developments are shown in order of priority.
• ICAP DRIVER As described in 5.2.4, the Petalinux system that we had
built uses a ready-to-use driver for the PCAP infrastructure. However, our
hardware architecture provides also a ICAP conﬁguration signal set. It can
not be used until a custom Linux driver for our custom hardware is written.
This solution could be very signiﬁcant for our project, but, even more for
others embedded Linux solution for dynamic hardware accelerators.
• PLATFORM UPGRADE The ZedBoard, described in Chapter 1, pro-
vides a good FPGA for prototype the project, but, the best solution for our

design can be the new UltraScale technology. These new FPGAs provide
more space in terms of CLBs, DSPs and BRAM and we know how the last
two elements are important for our design. This solution open up greater
avenues for new reﬁnements as shown in next points.
• MORE MODULES, MORE EFFECTS The more improvements the
FPGAs vendors provide to us, the more performance we can request from
our design. For example, we could increase the number of reconﬁgurable
modules and perhaps create more than one kind of them for satisfy the needs
of speciﬁc processes. This solution increases parallelism and performances.
Let us image now a real great library of audio processing HLL for this
application.
• PARAMETRIC EFFECTS With more AXI interconnections we could
change dynamically some parameters such as the volume, the delay period,
the oscillation period etc.
• MCAP The UltraScale architecture based FPGAs continue to support
the Internal Conﬁguration Access Port (ICAP). They provide direct access
between FPGA logic and conﬁguration functions. The ICAP interface is
similar to the SelectMAP interface, and allows user logic to initiate active
reconﬁguration. A new Media Conﬁguration Access Port (MCAP) provides
a similar connection to the integrated block for PCI Express. A small
initial bitstream can be loaded quickly at power-up to enable the PCIe
interface, and then the rest of the conﬁguration can be loaded through
the PCIe interface. Alternatively, the second part of the conﬁguration can
be loaded through the standard conﬁguration interfaces this is known as
tandem PROM [13].
