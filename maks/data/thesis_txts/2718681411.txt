Abstract
Linear type systems statically eliminate common errors associated with handling resources
such as files, threads, and memory allocations, and facilitate certain kinds of optimiza-
tions, such as alias analysis, that would otherwise be difficult to implement correctly. In
this project, a compiler for a programming language with linear types is designed and
implemented. The compiler outputs programs for a resource-aware virtual machine, ex-
ploiting some of the benefits of the linear type system. The compiler is implemented and
formalized in the Coq proof management system, and a proof of soundness of the linear
type system and associated type checker is shown. In addition, a well-known framework
for semantic preservation is adapted for use in the compiler, and the first translation step
is shown to preserve the semantics of the input program via this framework. For the re-
maining two translation steps, the proof of semantic preservation is not completed in full,
but substantial progress has been made towards it. Overall, the formalization provides
high confidence that the implementation is correct.

Introduction
Compilers are an integral part of any software stack used to create programs that have
strict requirements for efficiency, safety, or access to low-level system components. By
enabling the use of high-level programming languages, many of the difficulties in writing
low-level programs are abstracted away. High-level programming languages use type
systems to statically prevent common errors in programs, and as type systems evolve,
they prevent broader classes of errors.
High-level programming languages commonly rely on automatic runtime memory man-
agement like garbage collection to achieve memory safety, which imposes some compu-
tational overhead. For performance-critical domains where such overhead is unaccept-
able, programs are usually implemented in C or C++, where memory can be handled
manually. These languages do not prevent memory safety errors by default, so such pro-
grams are prone to memory safety [1] errors. In large, complex software systems, this
results in a vast number of security vulnerabilities [2] [3] [4].
Substructural type systems, which include linear type systems, have gained popularity in
recent years due to their ability to enforce certain desirable properties of programs, such
as resource safety.
A linear type system is a type system that is based on linear logic. In linear logic, assump-
tions must be used exactly once; they cannot go unused, nor can they be used more than
once. Similarly, in a linear type system, a typing assumption must be used exactly once.
The result is that programs that are well-typed in such a type system must use their re-
sources exactly once. This property can prevent common errors associated with the use
of resources, such as threads, files, or memory.
Moreover, linear types facilitate optimizations that are difficult to implement with an unre-
stricted type system. Alias analysis is commonly used in compilers to determine whether
two pointers point to the same memory location. This information can then be used to
optimize a program, e.g. by reordering reads and writes if they are found to be indepen-
dent. This optimization is a common source of errors, even in industry-standard compilers
[5] [6], but with the at-most-once guarantee of linear types, such an optimization could be
applied pervasively. However, in practice, dependencies on the aforementioned industry-
standard compilers prevent such pervasive use [7].
Formal verification is a crucial technique for ensuring the correctness of software systems
that require the highest levels of safety and reliability, such as medical and automotive
applications. Formal verification of programs reason about program behavior via a pro-
gramming language semantics, which formally defines the meaning of the constructs of
the language. In order for such reasoning to be valid, it is critical that the meaning of
the program is preserved through translation to a lower-level representation, such as As-
sembly code; a mistake in such a translation could render any argumentation about the
program invalid. For this reason, if a compiler is used in such a system, it must be be
formally verified.
By implementing linear types in a formally verified compiler, the resulting compiler extends
the benefits of linear types to domains where correctness is paramount.

2.1
Lambda Calculus
Lambda calculus is a formal system of computation, and can be thought of as a prototypi-
cal functional programming language. The core calculus consists of just three syntactical
elements, which comprise terms: variables, abstractions, and application of abstractions.
An abstraction consists of a name and an encapsulated term, and an application consists
of two arbitrary terms. A variable is commonly represented by text, most often just a single
character. The figure 2.1 shows one syntax for lambda calculus terms.
<term> ::= x

Figure 2.1: The syntax of lambda calculus terms
Following this syntax, an example of a lambda term is:
(λx. x) y
In this example, the occurrence of the variable x inside the abstraction on the left is said
the be bound by the abstraction, while the occurrence of y on the right is said to be free
since it is not bound by any abstraction. A term that has no free variables is said to be
closed.
When an abstraction is applied to some term, all occurrences of the bound name in the
abstraction is substituted for the term that it was applied to. In this project, for the appli-
cation of the abstraction (λx. t1) to the arbitrary term t2, the notation used to denote “the
substitution of all occurrences of x in t1 by the term t2” is:
[x := t2] t1.
A semantics ascribes meaning to the constructs of a language. There are different ways
to specify a semantics [11]. In this project, a semantics is defined via a small-step opera-
tional semantics, in which inference rules describe how to carry out some quantum step
of computation. An example of a small-step operational semantics for lambda calculus
terms is shown in figure 2.2.
The rules of a semantics can then be recursively applied in order to reduce a term, until
there are no longer any rules that apply. When there are no more rules to apply for some
term, that term is said to be in normal form.

If a lambda term is consistent with the typing rules, then it is said to be well-typed, or
that it “has type T”, where T is some specific type. A type system is commonly said to
be sound if it guarantees that well-typed terms to not exhibit “undefined” behavior, i.e. if
either there is always an applicable rule in the semantics, or the term has some inherent
value (such as a plain number). Determining if a lambda term is consistent with the typing
rules is called typechecking. A typechecking algorithm is said to be sound if whenever
it determines a term to be well-typed, it is in fact well-typed. It is said to be complete if
whenever a term is well-typed, it determines that the term is well-typed.
A substructural logic is a logic in which one of the three substructural rules of logic are
absent. The structural rules are called Weakening, Exchange, and Contraction. They are
shown in figure 2.5.

When a structural rule is absent, it limits the ways in which assumptions can be used
in order to arrive at a conclusion. For example, if the weakening rule is absent, then
assumptions cannot be removed from the context. Similarly, if the contraction rule is
absent, then assumptions cannot be duplicated. In linear logic, those two rules are absent,
meaning that assumptions have to be used at least once and at most once - that is, exactly
once. A linear type system is one that is based on linear logic, so that typing assumptions
must be used exactly once. A typed lambda calculus equipped with a linear typing relation
is commonly called a linear lambda calculus, or a linearly typed lambda calculus.
2.2
Compilers
Compilers for realistic programming languages tend to be very complex programs, since
the task of translating some program from a representation in a high-level programming
language to a low-level programming language can be challenging. Most often, compilers
are composed of several, smaller steps of translation rather than one translation from
source to target language; each such step going in the direction of the lower level of
abstraction, for this reason sometimes called lowering. When a language at either end of
the lowering is not the source or target language, it is commonly called an intermediate
representation.

2.3
Mechanical Verification in Coq
The Coq home page describes Coq as follows:
“Coq is a formal proof management system. It provides a formal language to write math-
ematical definitions, executable algorithms and theorems together with an environment
for semi-interactive development of machine-checked proofs.”[12]
These facilities make Coq an excellent tool for modelling and working with programming
languages and their type systems.
Coq is based on a constructive foundation, meaning that in order to prove a proposition,
it must be shown how to build the proposition from ground definitions. This is in contrast
to classical logics, where e.g. the law of excluded middle, p ∨¬p, allows propositions to
be shown indirectly, for example by showing that the negation of the proposition leads
to a contradiction. As a result of being constructive, a negation in Coq is defined as the
implication leading to False, which is impossible to construct, thus posing a contradiction
by existence. By the principle of explosion, anything follows from this.
More generally, one can define inductive propositions in Coq, through which e.g. the se-
mantics of a programming language, or the typing rules of a type system can be modelled.
An inductive definition is then automatically supplied with a suitable induction principle, so
that propositions can be proven by induction on instances of such definitions, a procedure
that is exceedingly common in this project.
While Coq allows for the implementation of recursive functions, it imposes some restric-
tions on their definitions. Principally, functions must be “obviously” terminating (for some
vague definition of “obviously”), since otherwise absurd facts can be derived from them,
by virtue of the Curry-Howard correspondence.
Functions can be extracted to OCaml programs, more suitable for practical use. However,
this extraction mechanism - as well as the OCaml compiler - are unverified, meaning that
some confidence in the correctness of an implementation, as established by verification,
is lost upon extraction. This means that such an extracted program intended for use in
practical circumstances should still be accompanied by rigorous automated tests.

Overview of the Architecture
The project is conceptually divided into three parts:
1. The formalisation of the source language, including its type system, as well as the
target and intermediate languages.
2. The implementation of the compiler.
3. The formalisation and verification of the compiler.
All three parts are implemented in Coq. The implementation of the compiler is thus embed-
ded alongside its proof of correctness, guaranteeing that the formalisation of the compiler
matches the implementation. The tradeoff here is that the implementation would have
to be translated to OCaml code via the Coq extraction mechanism in order to be used
in practice. Both the extraction mechanism and the OCaml compiler are unverified, so
this is a potential weakness in the toolchain. In practical terms, the formalisation of the
languages are contained in one compiler-agnostic package (Theory), while the compiler
is contained in a separate package (Compiler) that depends on the former.
The compilation process is factored into four phases:
1. The source program is typechecked.
2. If typechecking succeeds, then the source program is lowered into the first interme-
diate language.
3. The first intermediate representation is lowered into the second intermediate repre-
sentation.
4. The second intermediate representation is lowered to produce the target program.
This split facilitates a reduction in the density of complexity of both the implementation - as
it would in an unverified compiler - but also of the correctness proof [8]. The verification of
the compiler comprises the verification of the typechecker and each of the lowering steps.
In the implementation, the source, intermediate, and target languages are referred to as
LinLambda, DeBruijn, StackVM, and LinVM, respectively. These names will be used in the
following chapters as well.

The Semantic Preservation Framework
In order to verify the semantic preservation of compiled programs, this project employs
the conceptual framework outlined by Leroy et al. [8], as used in the CompCert project,
but with some technical simplifications. Due to the altered definitions, key results are
restated.
The CompCert framework uses the notion of simulations between source and target pro-
grams to reason about the preservation of behavior. Leroy et al. divide them into three
general categories:
1. Bisimulation: The source and target programs have exactly the same set of be-
haviours.
2. Forward simulation: Any behavior of the source program is also a possible behavior
of the target program.
3. Backward simulation: Any behavior of the target program is also a possible behavior
of the source program.
Bisimulation is the strongest of the three, but also the strictest; a compiler might want to
specialize on some subset of behaviors of the source program (such as by selecting an
evaluation order), but bisimulation does not allow this. Forward simulation is not generally
strong enough, as the target program could have other, potentially undesirable behaviors.
Backward simulation is, in this sense, more generally useful, but forward simulation is
posited to be generally easier to show. Leroy et al. capitalize on the insight that if the
target semantics is deterministic, then a forward simulation implies a backward simulation;
if the target program has only one possible behavior, then there cannot be some other,
potentially undesirable behavior. Thus the approach used in the CompCert project is to
show a forward simulation, and that the target language is deterministic.
The assumption that the target language is deterministic is reasonable. In the case of
CompCert, the source language, C, is non-deterministic due to the evaluation order of
expressions being undefined. This non-determinism is then trivially eliminated by select-
ing an evaluation order in compiled code.
In this project, the source language is also deterministic. When this is the case, the com-
bination of forward simulation and backward simulation becomes a bisimulation; the sets
of behaviors are both singleton sets. In addition, it enables the use of a weaker notion
of forward and backward simulation, which is easier to show as a consequence, yet with
determinism of both languages, it is argued here that the definitions become equivalent.
In the following, in order to distinguish between the definitions, this project’s definition will
be referred to as a weak simulation, while that of the CompCert project will be referred to
as a strong simulation.

4.1
Semantics
For the purpose of this project, a semantics is defined by:
1. The type of its states.
2. A relation step between states that describes the possible steps of computation
(written s −→S s′ for a semantics S).
3. A predicate final_state that describes the final states of computation.
4. A proof that final states are normal forms of the step relation.
Here, the definition of normal forms is the usual; an element is a normal form of a relation
R if R does not relate it to any other element.
This definition of semantics differs from the one employed in the CompCert project. Con-
cretely, it has no notion of global environments, traces, or initial states. The first two points
are omitted on account of there being no need for them; the languages employed in this
project do not have global state nor side-effects. In the CompCert project, an initial state
is defined as the opposite of a final state, i.e. a state that has no previous states. How-
ever, for the variants of lambda calculus used in this project, such an initial state cannot
be defined. Instead, part of this burden is moved to the definition of semantic preservation
by requiring that the state (or term, in the case of the lambda calculi) that is produced by
compilation must correspond to the state (or term) that it was compiled from.

4.3
Forward Simulation
A forward simulation is a simulation of a source semantics S by a target semantics T. It
is obtained in two parts. The first part is forward simulation of final states:
Definition 4.3.1 (forward_simulation_final_states). If a pair of states s and t correspond,
and s is a final state, then t must also be a final state.
∀s t, s ∼t =⇒final_state S s =⇒final_state T t.
With this definition, the correspondence is important for the validity of the simulation. For
example, if the correspondence relates any two (final) states that represent numbers,
then the simulation does not guarantee that the source and target programs arrive at the
same result for a given numerical computation. Thus care must be taken when designing
a correspondence.
The second part is forward simulation of stepping states:
Definition 4.3.2 (forward_simulation_step). If a pair of states s and t correspond, and s
steps to a state s′, then t must step to some t′ such that s′ and t′ eventually correspond.
∀s t, s ∼t =⇒∀s′, s −→S s′ =⇒∃t′, t −→T t′ ∧s′ ∼∗t′.
With this definition, given one step in S, it must be possible to choose one step in T such
that those resulting states must eventually correspond, meaning that it must further be
possible to choose some subsequent series of steps in either or both of the semantics such
that the resulting states correspond. This definition is a significant departure from that of
the CompCert project. For reference, the CompCert definition is as follows (excluding the
modelling of traces):
∀s s′, s −→S s′ =⇒∀i t, s ∼i t =⇒∃i′ t′, (t −→+
T t′ ∨(t −→∗
T t′ ∧i′ < i)) ∧s ∼i′ t′.1
This definition can be considered as two separate possibilities, with the first one being:
∀s s′, s −→S s′ =⇒∀i t, s ∼i t =⇒∃i′ t′, t −→+
T t′ ∧s′ ∼i′ t′.
Here, given one step in S, it must be possible to choose one or more steps in T such
that the resulting states correspond. This case cleanly encapsulates the notion that any
behavior in the source program must be a behavior of the target program, with the addi-
tional flexibility that the target program might take more steps in order to exhibit that same
behavior. This is also valid for this project’s definition: s′ ∼∗t′ if ∃t′′, t′ −→∗t′′ ∧s′ ∼t′′.
However, this is not sufficient for cases where it is not the case that a single step in S
translates to one or more steps in T. This is covered by the second case:
∀s s′, s −→S s′ =⇒∀i t, s ∼i t =⇒∃i′ t′, t −→∗
T t′ ∧i′ < i ∧s′ ∼i′ t′.

Here, given one step in S, it must be possible to choose zero or more steps in T such that
the resulting states correspond. Furthermore, the index i' in the post-step correspon-
dence must be strictly less than the index i in the pre-step correspondence. By allowing
zero steps in T, the simulation can apply multiple times in succession, making it possible
for a single state in T to correspond to multiple states in S, so that it is possible to take
more steps in S than in T. This project’s definition also allows for more steps to be taken
in S, with the argument being similar to the inverse situation where more steps are taken
in T.
The requirement that the ordering must be well-founded is put in place in order to avoid
the infinite stuttering problem:
“The original program P1 could perform infinitely many silent transitions S1 −→S2 −→
... −→Sn −→... while the transformed program P2 is stuck in a state S′ such that Si ∼S′
for all i. In this case, P1 diverges while P2 does not, and semantic preservation does not
hold.” [10]
By indexing the correspondence with i and i', it limits the number of times that the zero-
or-more case can apply, which limits the number of extra steps in S to some finite number,
thus avoiding the situation described above. This project’s definition avoids the infinite
stuttering problem by requiring unconditionally that the corresponding state t must step
at least once.
In summary, both definitions allow for either program to take more steps than the other
in order to reach a corresponding state, and both definitions avoid the infinite stuttering
problem. While being similar in these aspects, there are two key differences. First, with
this project’s definition, a single state in T cannot correspond to multiple states in S. One
might imagine that it could be encoded into the correspondence that only the last oc-
currence of such prospective correspondences is a valid correspondence, and if this is
generally possible, then the difference is merely a question of perspective. The second
difference is more significant, however. With the CompCert definition, the series of steps
in S that may be taken in successive instances of the simulation are given, and the steps
in T must be chosen. With this project’s definition, only the first step in S is given, while all
steps in S beyond the first, as well as the steps in T, must be chosen. Thus it is generally
a weaker statement. However, if S is a deterministic semantics, then whatever choice of
steps in S beyond the first is also the only choice, and the statements become equivalent.
The two parts combine into a forward simulation:

correspondence ∼is the combination of a forward simulation of stepping states and a
forward simulation of final states.
forward_simulation S T ∼≡forward_simulation_step S T ∼∧
forward_simulation_final_states S T ∼.

∀s t, s ∼t =⇒∀s′, s −→S s′ =⇒∃t′, t −→T t′ ∧s′ ∼t′.

4.4
Backward Simulation
A backward simulation is a simulation of a target semantics T by a source semantics S.
It is obtained in three parts. The first part is progress of corresponding states:
Definition 4.4.1 (backward_simulation_progress). If s ∼t and s is safe, then t must either
be a final state or there must be some t′ such that t −→t′.
∀s t, s ∼t =⇒safe S s =⇒final_state T t ∨∃t′, t −→T t′.
The second part is backward simulation of final states:
Definition 4.4.2 (backward_simulation_final_states). If s ∼t, and t is a final state, then s
must also be a final state.
∀s t, s ∼t =⇒safe S s =⇒final_state T t =⇒final_state S s.
Contrary to the CompCert definition, backward simulation of final states does not allow
for the source state to step an arbitrary number of times to reach its final state.
The third and last part is backward simulation of stepping states:
Definition 4.4.3 (backward_simulation_step). If s ∼t, and s is safe, and t −→T t′, then
there must be some s′ such that s −→S s′ such that s′ ∼∗t′.
∀s t, s ∼t =⇒safe S s =⇒∀t′, t −→T t′ =⇒∃s′, s −→S s′ ∧s′ ∼∗t′.
The same arguments apply here as for the forward simulation of stepping states, but in
reverse; the semantics T must be deterministic for this to be equivalent to strong backward
simulation.
A backward simulation is the conjunction of the three parts:
Definition 4.4.4 (backward_simulation). A backward simulation of a semantics T by a
semantics S is the conjunction of three proof obligations.
backward_simulation S T ∼≡
backward_simulation_progress S T ∼∧
backward_simulation_step S T ∼∧
backward_simulation_final_states S T ∼.

The source programming language is based on the simply typed lambda calculus [13],
with the unrestricted type system replaced by a linear type system. In addition to variables,
abstractions and applications, the language also has natural numbers with addition. This
section will elaborate first on the semantics of the language, and then the type system.

The Linear Type System
A linear typing relation can be given in declarative terms[14]. One such relation is shown
in figure 6.1. The rules can be briefly summarised as follows:
• T_V ar. The rule for variables requires that the context contains exactly one as-
sumption, and that the assumption pertains to that variable.
• T_Abs. The rule for abstractions requires that the context Γ does not already contain
an assumption for a variable of the same name as the one bound in the abstraction
itself.
• T_App. The rule for application of abstraction requires that the context is split into
two non-overlapping contexts Γ1 and Γ2, one for each of the subterms in the appli-
cation.
The rules of the linear typing relation are inextricably linked together. Considering these
rules together, the declarative linear typing relation enforces linearity by:
1. Prohibiting the shadowing of variables in abstractions, so that variables cannot be
”forgotten”.
2. Prohibiting variables from appearing multiple times by requiring that terms with mul-
tiple subterms, such as applications, must allocate each assumption to exactly one
of the typing derivations of its subterms, not both.
3. Prohibiting assumptions in the context from being ”thrown away” by requiring deriva-
tions to have only the necessary assumptions.
The declarative linear typing relation is mathematically simple, but poses some challenges
for an implementation in a typechecker. The context splitting operator is an integral piece.
The rule T_App provides no hints on its own as to how this operation should be done.
However, an important observation to make from the full set of typing rules is that the
resulting contexts must cover exactly the set of free variables of the terms; thus a type-
checker implementation must be able to determine this set, the verification of which is an
additional burden. Moreover, a naïve implementation would potentially have to determine
this mutiple times for a single term.
An alternative to the declarative linear typing relation is an algorithmic linear typing re-
lation [14]. An algorithmic linear typing relation introduces an output context, describing
the unused portion of the input context in the typing derivation. Rather than splitting the

The rules of the algorithmic linear typing relation used in this project are given in figure
6.2. They can be summarised as follows:
• AT_V ar. The rule for variables requires that the input context contains an assump-
tion about the given variable. The assumption is removed from the input context to
form the output context.
• AT_Abs. The rule for abstraction adds an assumption for the bound variable to the
context, and uses that to type the abstraction body. The output context must not
contain any assumption about the bound variable. Any previous assumption about
a variable of the same name as the bound variable is restored afterwards.
• AT_App. The rule for application requires the output context of the left-hand side
to be provided as input to the right-hand side.
• AT_Nat. The rule for natural numbers imposes no requirements on the context,
and the output context is equal to the input context.
• AT_Add. The rule for addition is similar to that of application.
The algorithmic typing relation is in some sense clearer in its enforcement of linearity.
Firstly, the restriction that variables can appear at most once is enforced by the AT_V ar
rule; they are removed from the context when they appear. Secondly, the restriction that
variables must appear at least once is enforced by the AT_Abs rule directly; the vari-
able must have been used before it would otherwise go out of scope at the end of the
abstraction.
Furthermore, the algorithmic typing relation provides an attractive set of tradeoffs for the
purpose of implementing and verifying a linear type checker. It is no longer necessary
to determine the set of free variables in subterms in order to split the context; this is
done implicitly and incrementally. As a result, it is significantly easier to implement a
typechecker for this typing relation. In addition, it enables some properties of the type

6.0.1
Soundness of the Type System
The type system is proven sound to ensure that it correctly prohibits undesirable behavior,
following the usual syntactic type soundness definition. Specifically, soundness is defined
as the property that well-typed terms are safe, using the definition of safe from section 4.
The proof relies on the usual combination of progress and preservation with the addition
of an output context. However, in order to prove progress and preservation, a number of
supporting lemmas are established first. The high-level overview of the approach is as
follows:
1. Show that the algorithmic typing relation is monotonically decreasing with regard to
the context.
2. Show the relationship between the contexts for assumptions about variables that
appear free in the term, and for variables that do not appear free in the term.
3. Show that free variables appear exactly once in a well-typed term.
4. Show context invariance, in turn providing weakening and strengthening lemmas.
5. Use all of the above to prove progress and preservation.
A fundamental property of the algorithmic typing relation is that it is monotonically de-
creasing with regard to the context; the output context is a subset of the input context.

The DeBruijn Language
The first intermediate language is an untyped lambda calculus with de Bruijn-indices.
The use of de Bruijn-indices simplify binders by making references unique by convention.
When viewing binders as a stack in the order they are introduced, de Brujin-indices index
from the top of the stack. This makes it an especially convenient intermediate represen-
tation before transitioning to a stack-based semantics. The syntax is shown in figure 7.1,
and the semantics is shown in figure 7.2. The substitution function is modified accordingly
to increment the index upon entering an abstraction. Besides its use of de Bruijn-indices,
the syntax and semantics is similar to that of the source language.
Like the source language, the semantics makes use of a value proposition to denote valid
non-reducible terms, and makes use of that proposition to enforce a left-to-right evaluation
order to make the semantics deterministic. Both proofs proceed almost exactly as for the
source language, and thus will not be elaborated here.

Semantic Preservation
In order to prove semantic preservation of the translation from LinLambda to DeBruijn, a
forward simulation is constructed. First, a suitable correspondence between LinLambda
terms and DeBruijn terms must be defined, describing terms that reduce similarly.
In abstract terms, a LinLambda term s and a DeBruijn term t correspond if they are struc-
turally identical and the variables occurring in the LinLambda term correspond to the in-
dices occurring in the same places in the DeBruijn term. Two terms correspond with
regard to a name x and an index i if neither appear free in their respective terms, or if
they appear free in the same positions in their respective terms. The ∼(correspond) and
correspond_wrt relations capture these criteria concretely, and are shown in figure 7.4
and 7.5, respectively.

The rules of the correspond relation can be briefly summarised as follows:
• co_var. Any variable corresponds to any index.
• co_abs. A LinLambda abstraction and a DeBruijn abstraction correspond if the
abstraction bodies correspond, and if the abstraction bodies correspond with regard
to the bound name in the LinLambda abstraction and the index 0 in the DeBruijn
term.
• co_app. A pair of application terms correspond if their components correspond.
• co_const. The constant n in a LinLambda term corresponds to the same constant
n in a DeBruijn term.
• co_add. A pair of addition terms correspond if their components correspond.
Similarly, the correspond_wrt rules can be summarised as follows:
• co_wrt_var. A pair of variables correspond with regard to the name x and the index
i if those variables are exactly x and i.
• co_wrt_abs. A pair of abstractions correspond with regard to x and i if the bound
variable in the LinLambda abstraction is different to x, and the abstraction bodies
correspond with regard to x and i + 1.
• co_wrt_app. A pair of applications correspond with regard to x and i if their com-
ponent terms correspond with regard to x and i.
• co_wrt_add. A pair of additions correspond with regard to x and i if their component
terms correspond with regard to x and i.
• co_wrt_not_free. Any pair of terms correspond with regard to x and i if neither
term has that variable appearing free in it.
The proposition correspond_wrt is designed specifically to be sufficient to ensure that
correspond is preserved over substitution. In doing so, there are two distinct cases to be

This definition of correspondence is very strict. For example, a different definition might
determine that a LinLambda term (λx : Nat. x + 1) and a DeBruijn term
(1 + $0) correspond, since for any arguments to the abstractions that correspond, they
would evaluate to the same thing (though this would not be the case for a call-by-name
evaluation strategy). However, the strictness of the definition given here is desirable; it
both matches the translation of the programs, and its strictness makes semantic preser-
vation more apparent.
In order to arrive at the result of semantic preservation, the high-level procedure of the
remainder of this section is to first show the forward simulation, irrespective of lowering,
and then show that lowering indeed produces terms that fit into the forward simulation.
In order to show the forward simulation, the following properties of corresponding terms
s and t are shown:
1. When the LinLambda term is a value, then so is the DeBruijn term.
2. When the terms correspond with regard to variables x and i, and y and j, respec-
tively, then they continue to correspond with regard to variables x and i after the
substitution of y and j for closed substitutes a and b.
3. When the terms correspond with regard to variables x and i, then they continue to
correspond after the substitution of x and i for closed substitutes a and b.
4. When both terms are closed, and the LinLambda term steps to a term s′, then the
DeBruijn term also steps to a term t′ such that s′ and t′ correspond.

∀s t, s ∼t =⇒value s =⇒value t.

In summary, the following points have been established:
• When typechecking a LinLambda term succeeds in a given typing context, then the
term is well-typed in that context.
• When a LinLambda term is well-typed in the empty context, then it is closed.
• Lowering a LinLambda term produces a corresponding DeBruijn term.
• If the lowered LinLambda term is closed, then so is the DeBruijn term.
• When a closed LinLambda steps, then a corresponding, closed DeBruijn term can
step, and the results correspond.

The StackVM Language
A substitution-based semantics, such as LinLambda and DeBruijn, is very convenient
from a theoretical standpoint. It facilitates a very concise, yet powerful semantics that
elegantly deals with variables and application of abstractions all at once. However, it is
incompatible with usual abstractions of hardware, where values are stored into memory,
and loaded when required for some operation.
The goal of the StackVM intermediate representation is to eliminate substitution in favor
of a load-store evaluation strategy. With this strategy, values are stored into an explicit
environment, which maps variables to values, rather than substituted upon application of
an abstraction. Variables can then later be evaluated by loading the corresponding value
from an environment.
A load-store semantics poses some challenges. In the LinLambda and DeBruijn seman-
tics, a non-capture-avoiding substitution strategy works correctly only when closed values
are substituted into terms (which, as shown previously, is always the case when reduc-
ing an initially-closed term). If such substitutes are open, then variables inside it may be
captured, changing their meaning.
In a naïve load-store semantics, values might simply be stored into an environment, and
when variables are encountered, the corresponding value would then be loaded from this
environment. This is problematic in the same way as applying an abstraction to an open
term in a substitution-based semantics; variables may be captured.
Instead, the StackVM semantics takes the common route of using closures and stack
frames to correctly handle this. A closure is an abstraction that closes over the free vari-
able that occur within it; that is, it bundles the environment in which it occurs together
with the abstraction, so that the closure becomes self-contained, and can be stored and
loaded freely without risk of subsequent capture of those variables. When a closure is
applied to some value, the value is inserted into the closure’s environment, and that en-
vironment - not the enclosing environment - is then used to reduce the abstraction term.
This is modelled with stack frames, which encapsulate a term in an environment, causing
reduction of that term to happen in that environment. The StackVM semantics is shown in
figure 8.2.
The rules ST_App1, ST_App2, ST_AddConsts, ST_Add1, and ST_Add2 are similar to
those of the DeBruijn semantics, with the addition of an environment; composite terms
can step if either one of their subterms can step, and two numbers can be summed to-
gether to form a single number. The difference is in the remaining rules, which can be
summarised as follows:
• ST_Load. A variable i can be loaded from the environment env, producing the value
v, if v occurs at the i’th index of env.
• ST_AppClosure. A closure, bundling some environment env′ with an abstraction
body t1, can be applied to a value t2, producing a stack frame containing the envi-
ronment (t2 :: env) and the abstraction body t1.
• ST_Capture. An abstraction (λ. t) can capture its enclosing environment env, pro-
ducing a closure that bundles env and t.

The LinVM Language
The StackVM language, with its load-store strategy, has no restrictions on its use of vari-
ables. The linearity restrictions enforced by the source language have, in some sense,
been lost in translation. Additionally, closure conversion captures the entire enclosing
environment, possibly producing unnecesary copies of (or references to) values that are
not be used in the abstraction.
The LinVM language is a refinement of the StackVM language, recapturing some of the
guarantees provided by the linear type system. The LinVM semantics makes two key
changes to the StackVM semantics to achieve this:
• Variable access is destructive; loading a variable from the environment renders that
variable inaccessible afterwards.
• Closure conversion splits the environment in two, producing one environment for
the closure and a residual environment that replaces the original environment.
In order to accomodate these changes, the environment of a state is modelled as a list
of options of values, rather than just a list of values. With this representation, a Some v
indicates that the value has not yet been used, while a None indicates that whatever value
was once there has been used. In addition to this, the semantics uses two operators:
take_nth. The take_nth operator takes an index i and an environment env. If there is
something at index i in env, that is if the environment binds something to i, then take_nth
returns a Some containing a pair, consisting of what was at the index, and an updated
list env′ that is the result of setting the i’th index to None. Formally, the specification of
take_nth is:
∀xs xs′ i v, take_nth i env = Some (v, xs′) =⇒
xsi = v ∧xs′

Here, xsi = v means at index i in the list xs, there is a value v.
split. The split operator takes an environment env and a list of indices is. If for all
indices i in is, there is something in the environment env, then split produces a Some
containing a pair of environments env′ and env′′. The environment env′ is like env, except
all indices in is have been set to None. The environment env′′ is like env, except all
indices not in is have been set to None. Formally, the specification of split is:

Since the StackVM and LinVM languages are very similar, it would be a lockstep simulation;

The correspondence between StackVM and LinVM should be strict, since the states are

variable was removed, and since variables appear only once, it cannot appear anywhere
else, so wherever the residual environment is used, it should never be the case that this
particular entry is accessed again. In the ST_Capture case, the challenge is similar.

Evaluation
This work constitutes a proof of concept of a mechanically verified compiler for a functional
programming language with linear types. In the following, it is discussed to what extent
the goals were met, how some technical aspects of the work might be improved, and
lastly, some potential additions to the work are hypothesized.
10.1
Results
The results of this work, in accordance with the goals stated in chapter 1, are enumerated
and discussed here.
A framework for compiler correctness.
The CompCert framework of semantic preser-
vation was adapted for use in this project. In this adaptation, some modifications were
made to the definitions within. Arguments were provided in favor of the validity of these
modifications, giving confidence that they are well-founded. Some key results were reestab-
lished, and the framework was successfully used to establish semantic preservation of the
first lowering step. Overall, this goal was met.
A linear type system.
A linear type system was designed based on an algorithmic typ-
ing relation, inspired by existing texts. A full proof of soundness of the type system was
shown. Overall, this goal was also met.
A typechecker.
A typechecker for the algorithmic typing relation was implemented, and
a full proof of soundness was shown. Overall, this goal was also met.
The programming languages.
Four languages were modelled for use in the compiler.
The semantics of the source language, LinLambda, is a derivative of existing work, and
as such is not to be considered a product of this work, and the DeBruijn is a close deriva-
tive of the LinLambda language. The semantics of the stack-based StackVM language
was modelled based on common notions of closures and stack frames, enabling a shift
from substitution-based evaluation to stack-based load-store evaluation. The LinVM lan-
guage refines the semantics of StackVM, making it resource-aware, exploiting some of
the guarantees provided by the linear type system. Overall, this goal was also met.
The translation steps.
The three translations steps, covering the complete translation
of LinLambda programs into LinVM programs, were designed and implemented, and im-
portant results for these translations were established for use in proofs of semantic preser-
vation. Overall, this goal was also met.
Verification.
The full proof of semantic preservation was split into multiple parts, one for
each of the lowering steps of the compiler. Supported by the framework for establishing
semantic preservation, these individual proofs are divided into subgoals, which includes
showing that the source and target languages are deterministic, as well as showing a for-
ward simulation of the source semantics by the target semantics. Such a simulation was
shown in full for the LinLambda to DeBruijn lowering step. Due to time constraints, for
the DeBruijn to StackVM and StackVM to LinVM lowering steps, analyses and subsequent
sketches of such simulations were given instead. The languages were both shown to

be deterministic, so in conjunction with the framework of semantic preservation, substan-
tial progress towards a full proof of semantic preservation was made. Still, completing
these forward simulations would be a significant improvement to the work, making it more
convincing as a proof of concept. Overall, this goal was partially met.
10.2
Technical Improvements
During the development of this project, some minor technical challenges were encoun-
tered. While solutions were found to avoid or otherwise reduce these challenges, there
are potentially other, more suitable solutions. In the following, three technical challenges
are elaborated, each followed by a discussion on how this might have been solved in a
better way.
Separation of syntactical and runtime structure.
When lowering a StackVM term to
a LinVM term, the lowering function is required to handle terms that include stack frames
and closures, even though such terms are never produced when lowering a DeBruijn
term to a StackVM term - they are concepts specific to evaluation. It would be beneficial to
find a way of separating these concepts in such a way that it is clear which ones pertain to
the structure of programs, and which ones are relevant only for states of programs. The
challenge in moving these concepts to be part of the states, rather than terms, is that a
program is a tree-structure, and any term can contain a closure or a frame and vice versa.
One possible solution is to duplicate the definition of terms in the definition of states, only
then adding the definitions of closures and stack frames. Terms would then have to be
translated to states in order to be evaluated.
Iterative semantics.
The languages StackVM and LinVM are almost entirely syntacti-
cally identical, with the only difference between the languages being their semantics; the
LinVM semantics is a refinement of the StackVM semantics. This refinement exists pri-
marily in order to divide the semantic preservation proof in two. This tactic seems more
generally useful, so it would be interesting to explore if this notion of “iterative” semantics
could be better integrated into the generic framework of semantic preservation in such
a way that the the semantics could be defined for the same language, making such a
lowering step entirely unnecessary in cases where there are no syntactical changes.
Descriptive semantics.
The LinVM semantics makes use of two operators take_nth
and split. In the mechanisation, these operators have an explicit implementation, and
this implementation is shown to satisfy a specification. This makes the semantics prescrip-
tive in the sense that it prescribes how to perform these actions. For the proof of semantic
preservation, however, it is only the specification that is relevant; its implementation and
performance characteristics are irrelevant. From this point of view, the implementation
is purely in the way of establishing those facts. More generally, it might be beneficial to
design semantics to be descriptive rather than prescriptive, so that the focus is on the
logical effects of operations. For example, an implementation of a semantics might prior-
itize memory efficiency over speed, or vice versa, and a different implementation of such
operations might be more appropriate for such priorities.
10.3
Extensions
The compiler is implemented and verified in Coq, but it is not equipped with a practical
interface or a parser. Coq features an extraction mechanism that enables the implemen-
tation to be translated to an OCaml program. This mechanism could be used to provide

a practical way of using the compiler. The parser could be implemented in OCaml, and
integrate with the extracted implementation of the rest of the compiler. In addition to en-
abling the general practical use of the compiler, this OCaml-based parser and extracted
implementation would also allow for automated tests to be implemented, providing even
greater confidence in the correctness of the compiler as a whole.
The source language, LinLambda, features a linear type system, which enforces both “at
most once” and “at least once” usage of variables, or more generally, resources. The
target language, LinVM, is designed to make use of the “at most once” guarantees with
its take_nth and split operators, but does not make use of the “at least once” guar-
antees. One possible extension to the compiler could be to either incorporate this into
LinVM, or add another language that refines the LinVM semantics to incorporate this. For
example, the ST_Return rule of the LinVM semantics could require as a premise that the
environment env′ in the stack frame is empty before being able to remove it.
The target language LinVM is intended to model a low level, stack-based virtual machine.
However, its program and runtime structure is tree-like, which makes for a less obvious
connection to the usual low-level program representations such as Assembly language.
One possible extension would be to add another, lower level representation that uses a
linear program representation. In [16], Leroy et al. show how to structure a proof of seman-
tic preservation for the compilation of programs in the prototypical imperative language
IMP to a linearized format intended for a virtual machine.

Conclusion
A proof of concept of a compiler for a functional programming language with linear types
has been designed and implemented. The target language models a stack-based resource-
aware virtual machine, making use of some of the guarantees established by the linear
type system. The type system and associated type checker have been proven to be
sound, and the first of three translation steps has been shown to preserve the semantics
of the input program, demonstrating the successful use of an adaptation of a well-known
framework for semantic preservation from the CompCert project. Although the remaining
two translation steps were not fully verified, substantial progress was made towards the
proofs of semantic preservation, and together with the successful result of the first step,
this establishes high confidence that it is possible. It has been hypothesized what kind
of improvements might be made to the compiler, including solutions to existing technical
challenges as well as possible general extensions, such as extracting the compiler imple-
mentation to OCaml, and adding a parser to facilitate practical use. Overall, many of the
stated goals were met, and in the cases were goals were not met in full, it was discussed
how they might be met.
