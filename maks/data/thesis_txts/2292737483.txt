Abstract
This project is the design and implementation of a hardware Front End Pro-
cessor (FEP) with an instruction-set that allows for utilization of various ac-
celerators via the connected PC. The FEP will handle data transfer between
PC, on-board FPGA RAM and the connected accelerator. This should make it
easier to design and change accelerators depending on the application.
Hardware accelerators are made for a wide range of applications to increase
computational speed and power eciency compared running the same applica-
tions on modern multi-core PCs. The FEP should handle dierent classes of
accelerators, including the ones with high data-transfer rate between the PC
and accelerator and the the ones with minimal exchange of data with the CPU.
The accelerators use for this project are the Telco benchmark and the Monte
Carlo Simulation. The Telco benchmark require frequent memory access while
the Monte Carlo simulation does not and has low input/output communication.

Introduction
3.1
Project description
In this project a Front-end processor is implemented on a Xilinx Virtex-5 FPGA
to handle data transfer between a Telco Benchmark accelerator and a Monte
Carlo Euro Pricing Option accelerator. Only one accelerator is implemented on
the FPGA with the FEP at one time. The FEP handles needed data transfer
between the accelerator and the CPU though the PCI-X port which connects
the Alpha Data PCI Mezzanine Card. Development for the board was done
using the Alpha Data SDK and the accelerators was based on previous work.

Transfer of data between the CPU and FPGA can be done in several ways.
The basic method is direct slave transfer where the CPU is the master and
the PCI-to-local bus bridge is the slave. When dealing with large amounts of
data direct memory access (DMA) is more ecient and is keeps the CPU as
free as possible for other tasks. This is because the DMA transfer is managed
by separate DMA engines in the bridge between the local bus and the PCI-X
bus so the CPU only have to instantiate the transfer. With DMA transfer the
PCI-to-local bus is the master and the CPU is the slave.

calculation time is balanced when performing binary-coded decimal operations.

application and thus only covers basic decimal operations. That means it's not
suitable for benchmarking modern decimal implementations which needs greater
precision.

Monte Carlo Simulation
The Monte Carlo simulation is a the method where you sample a algorithm
with a random distribution to obtain numerical results. It's mainly used for
optimization, numerical integration and generation of draws from a probability
distribution.
The Monte Carlo Simulation used here is for the European option pricing.
In this case a gaussian random distribution is used to input the possible price
paths the option can take and then calculating the exercise value or payo for
each path. The payos are then averaged which gives the value for the option.

5.1
Applications
Implementation of the hardware needs to be able to handle dierent types of
applications. The two applications chosen for this the Telco Benchmark and
the Monte Carlo European Pricing option. These have been chosen for their
dierences in how they run. The Telco benchmark uses a lot of memory, it
calculates and outputs not only the accumulated sum of all calls but also the
price of each individual call. The Monte Carlo simulation only takes a few input
values and returns a single value for the specic option. The Telco benchmark
species for 1 million calls to be processed, which is ne for a benchmark but in
a actual billing system the amount could easily several millions calls. The input
calls can be store using 24 bits pr. call, while 44 bits is needed for the output.
This gives a RAM utilization as shown in equation 3.
RAM utilization = 1, 000, 000 · (24bits + 44bits) = 68Mb = 8.5MB
(3)
With the two SSRAM modules being 4MB this insucient for the storing
the calls and pricing for the Telco benchmark. Using the DDR2 modules while
slower will also allow for even larger memory utilization with other accelerators.
For the Telco each bank of 256MB hold over 30million calls.
The Monte Carlo simulation on the other hand a 32 bits for the input and
32 bits for the output is enough for every run of the simulation. So the Monte
Carlo simulation can use direct slave transfers to transfer the data between the
CPU and FPGA for quick runs of the accelerator.

Because of the limits of SDRAM is it important to consider the latency.
SDRAM consists of banks divided into rows and columns. To access the data
rst a row must be opened before a column can be accessed, the delay between
the row and column access is tRCD cycles. I will take a few clock cycles before
the accessed data is received because the memory is pipelined but when the
row is accessed it will only take tCAS cycles to access columns. If another row
needs to be accessed there is a precharge delay to close the current row, TRP ,
before a new command can be issued. tRAS is basically tRCD + tCAS + 2 in
other words the minimum time to access a row of data between the request and
the precharge command.
• tCAS is number of clock cycles needed to access a column of data.
• tRCD is delay in clock cycles required between asserting a row and then a
column.
• tRP is the number of clock cycles needed to terminate access to an open
row and open the next.
• tRAS is the minimum number of clock cycles needed to access a certain
row of data in RAM.
These four latency values a normally specied by the manufacture in-order
separated by a dash: tCAS-tRCD-tRP . The typical timing for DDR2-SDRAM
@ 333MHz is 5-5-5. Because of the latency in order to achieve high transfer
rates when using SDRAM, the number of shifts between rows should be as low
as possible. Also any additional shifts between read and write state should be
avoided.

6.1
Front End Processor
The Front End Processor consists of two nite state machines (FSM). The top-
FSM is called control_ fsm and is responsible for reading and writing to the
registers and parsing control signals to the implemented accelerator. The process
is activated by setting the reset high, after which it will go to the rst state,
idle, as seen in Figure 6 that illustrates the state diagram of control_ fsm. In
the idle state the FSM waits for the go signal from the CPU after bank1 have
been written. This triggers the next state instructions which parse a go signal
to the data_ fsm.
It then moves on to the wait_ until_ done state.
This
state waits until the data_ fsm sends a done signal back and then goes on to
wait_ pipe state. This state waits until a counter reaches 0 before it goes to
the retrieve state, the counter is set to the amount of clock cycles it takes for
the accelerators pipeline to be emptied. The retrieve state writes the last data
to a few registers and returns the FSM to the initial idle state.

Results
To quantify the performance of the implementation, the functionality and speed
was tested.
The execution time was conducted on the CPU and FPGA in
order to compare the performance. The logic utilization on the FPGA for both
implementations of accelerators is included in the appendix. In order to avoid
speed stepping the CPU used for to run the tests was set to high performance
mode using the linux command cpufreq-selector. In this mode the CPU, which
is a Core2 Duo E8400, ran at 2.998GHz. The FPGA was set to run with a
LCLK frequency of 66.7MHz and MCLK frequency at 266.7MHz. Execution
time measurements was done with software measurements in the C program
using the clock_ gettime. Timing measurements was performed at the start of
the application and at the end of the calculations.

Discussion
There is several issues with the implementation of not only the FEP but the
accelerators. The FEP have issues with memory transfers. I seems that data fsm
doesn't manage to read and parse the data from the registers correctly. This
seems to have something todo with how much of the registers is utilized. Both
[3] and a previous project using the same platform had this issue when utilizing
large parts of the registers of the SDRAM. This could mean that there is an
issue with either the SDK or the board itself. It has not been conrmed that
the issue is restricted to using memory bank1 or all the banks.
The implementation of the Telco processor had some issues that wasn't re-
solved because of time constraints. The accumulator responsible did not give
correct results as it started calculations with a value present introducing a bias.
The memory error also means that the output from the Telco benchmark output
the calls in the incorrect order.
The Monte Carlo simulation was changed to take dierent inputs but time
constraints meant that there wasn't time to implement a variable counter so the
number of iterations for the simulation could be change as well. While this im-
plementation is does allow for more options to be run without re-implementing
the accelerator from scratch it's also a lot slower than the hardcoded solution.
The FEP should be able to do direct slave transfers with the Monte Carlo
accelerator but still that requires to comment out large parts of the data_ fsm to
disable the DMA. This might improve the speeds of the Monte Carlo simulation
but it also requires rewrites of the C software.
Working with hardware accelerators is most often about saving time in the
long run.
The development time and cost of working with accelerators can
easily oset whatever speed advantages the accelerator provides in comparison
to conventional methods. Designing a FEP to utilize with dierent accelerators
is an attempt to reduce this development time in future projects. Beside this
FEP being to simple to manager a large variety of accelerators, there's still need
to develop the software side programs specic to each accelerator.
A very large part of this project was used for learning and understanding
the Alpha Data Platform. While it did come with a SDK and user-guide, the
documentation is very sparse and the code is poorly commented. Also the SDK
does not support newer versions of ISE and there's some issues when using newer
versions, especially when doing simulations and building new projects.

Conclusion
In this project a FEP and two hardware accelerators was implemented on a Al-
pha Data ADM-XRC-5T2 board. Only one accelerator was used in conjunction
with the FEP at one time. This means that the C program for CPU side ap-
plication and bit-stream led for programming the FPGA are separate entities
for each accelerator implementation. Both accelerators communicates with the
same FEP and uses DMA to transfer data between the CPU and FPGA storing
the data on DDR2 RAM modules on the Alpha Data board. The implemented
FEP was partially functional, with the main problem being it's incapability of
reading data from external SDRAM and parsing it correctly to the Telco proces-
sor, similar issues was present in previous work with the Alpha Data Platform
[3]. Time measurements was taken for each accelerator and compared to CPU
running the same calculations.
One accelerator used was a Telco Benchmark base by work from [3]. It was
chosen for its high memory requirements and large amount of data transfers.
The Telco processor was largely unmodied from the initial design.
The other accelerator was a Monte Carlo Euro Pricing Option simulation
based on work from [2] which was chosen for it's low memory requirement and
low data transfers. The Monte Carlo simulation was 16x ASP with hardcoded
inputs and simulation iterations. The accelerator was successfully modied to
take dierent inputs but there was not time to design a variable counter. The
Monte Carlo simulation however is slower than the original hardcoded solution
indicating that there is plenty room for improvement still.
While there was a speed gain from both the accelerators. The FEP had issues
with DMA transfers which cause both incorrect calculations and the output
coming in incorrect order which pretty much renders the implementation useless
for any real-world applications. This is in addition to the Telco's accumulator
not working correctly. Learning how the the Alpha Data platform work and to
utilize it took a considerable amount of development time. It's recommended
that for future projects using the Alpha Data platform that time before the
project starts is used to learn the platform if the documentation isn't improved.
