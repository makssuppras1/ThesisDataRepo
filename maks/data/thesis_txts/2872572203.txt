Abstract
This thesis presents TGH-RAG, a hybrid retrieval system developed to help users
access precise and contextually relevant information from complex enterprise docu-
mentation covering various topics and areas. By combining taxonomy, knowledge
graphs, structured metadata, and both vector-based and graph-based retrieval, the
system significantly improves answer completeness and relevance compared to stan-
dard RAG methods. Readers can expect insights into the system’s design, imple-
mentation, and evaluation, demonstrating its effectiveness in enhancing enterprise
information retrieval.
Keywords: Knowledge Graph, Retrieval-Augmented Generation, Taxonomy, Ar-
tificial Intelligent, Large Language Model, Hybrid Retrieval-Augmented Generation,

CHAPTER 1
Introduction
In finance enterprises, systems are built around specific domain knowledge. Following
the correct steps is essential, but these systems often involve complex relationships
between different parts of the business, making them challenging to use. For example,
the term domain refers to many specific areas within the financial enterprise such as
compliance, fund operations, risk management and financial reporting. Each domain
has its own rules that should be followed. In order to build an effective system, it is
important to understand how these domains are connected together. Enterprises usu-
ally provides user manuals, policy guides and other procedural guidelines to instruct
the users how to use these complex systems. The users could be compliance oﬀicers,
operational staff, or technical users who need to use the system to perform their roles.
However, the size and complexity of these documents make it diﬀicult to identify
the correct steps, especially when the information is spread across multiple related
areas. Each domain has its own terminology, procedures, and logic, and most tasks
usually require the user to navigate across these boundaries. New approaches such as
Retrieval-Augmented Generation (RAG), have been developed in order to help the
users navigate through the documents which are spread across domains. The goal of
this thesis is to develop a solution that is an advanced version of RAG as there are
still some limitations to using RAG in enterprises.
1.1
Retrieval-Augmented Generation (RAG)
The large language model (LLM) is an artificial intelligence model that usually relies
on the knowledge it is trained on. Therefore, the answer the model generates is usually
based on its training data and not on an external source. The LLM can also be fine-
tuned on specific domain, but this can be costly and still limited to the static data
that was provided at the time of training. Retrieval-Augmented Generation (RAG)
is a technique that is now being used as a better approach to retrieve information
from external knowledge sources, such as document databases and search engines, as it
combines the power of large language model (LLM) and external retrieval mechanism.
The LLM is trained on a vast amount of data, and RAG extends this by not only
using the data LLM is trained on but also dynamically fetching an organization’s
internal knowledge base and using that in its generation process. This is helpful for
==PAGE:14==

general-purpose questions but struggles to answer complex questions that requires
reasoning.
To retrieve information from a document eﬀiciently, the document is split into
pieces of texts, such as a paragraph, section, or a sentence. This is called chunking.
The chunking of documents is usually done using a script within a pre-processing
pipeline. Chunking makes the retrieving process more eﬀicient and can provide more
accurate results. Instead of retrieving the entire document, only the chunk that has
the relevant information can be retrieved instead.
An embedding model is a machine learning model that converts the chunks into
numerical vectors, this is known as embedding. These numerical vectors are stored
in a vector database. RAG uses the vector database as its external knowledge source.
The embedding is done to represent the text in a high-dimensional space and to
capture the meaning of the text. For example, if two words are used in the same
context it assigns them a similar vector representation.
This allows the RAG system to perform vector-based semantic similarity search.
For example, when the user asks a question, the question is embedded into a numerical
vector using an embedding model. The embedded user query vector, is then matched
against other vectors in the vector database, using the cosine similarity measure.
Cosine similarity will be explained in the later chapter. The most similar vector is
retrieved. The original text chunk is associated with the vector. This text chunk is
passed to the LLM, that generates an answer for the user in natural language.
However, RAG still struggles to answer complex user queries within an enterprise.
RAG only matches the user query to a similar vector but fails to understand the user
query context and perform any deep reasoning. This is because it treats text data
as an isolated chunk, and doesn’t consider its context within the entire document.
It may retrieve conceptually similar but procedurally incorrect content, as it cannot
reason over dependencies between chunks.
1.2
Hybrid Retrieval-Augmented Generation (Hybrid
RAG)
To overcome RAG limitations, researchers have proposed an advanced approach called
Hybrid Retrieval-Augmented Generation (Hybrid RAG). As compared to RAG, which
uses only one retrieval method, hybrid RAG is a combination of more than one re-
trieval method, which are vector-based semantic search and retrieval from a Knowl-
edge Graph (KG) [1].A Knowledge Graph represents the data in the form of a graph.
It uses a triplet framework to organize and connect information in the form of a triple.
The triple is also known as a fact.
The triplet is organized as subject->predicate->object and the meaning of
each term:
• Subject represents an entity the triple is about
==PAGE:15==
1.3 SimCorp
3
• Predicate represents the relation between subject and object
• Object is an entity which is the target of the relation
Ontology provides a more structured schema for the Knowledge Graph (KG). It
has three main components:
• Entities, which represents distinct things in the domain
• Relationships, represents how the entities are connected
• Properties, describes an individual entity using attributes
Entities become nodes, relationships become edges and properties are metadata
on the nodes that help understanding what the node represents.
In a traditional hybrid RAG system, when a user asks a question, the system
performs a number of steps to generate the final response.
First, the question is
embedded into a vector and is used to retrieve a text chunk that is semantically
similar to it. This is the same as explained for the RAG system. Then entities are ex-
tracted from the question. The entity extraction is done using a NER (Named Entity
Recognition) model that identifies key information in the text. In newer systems, an
LLM is used to extract the entities. The extracted entity/entities are used to query
the Knowledge Graph to find a triplet/triplets that matches the entity. The graph
database can be queried using a graph query language, such as Gremline or SPARQL.
The result of a graph query is the triplet extracted from the Knowledge Graph. A
simple scoring system can be used to score the extracted triplet, which is usually 1.0.
There are other scoring strategies that can be used as well to score the triplets, such
as assigning an edge weight. The chunks are scored using the similarity score from
the cosine similarity measure. The text chunks from the vector database and triplets
from the Knowledge Graph are combined into a list. The results in the list is ranked
from highest to lowest. The top scored results, usually top-5 are then passed on to
the LLM which generates an answer.
Based on the recent research and development in Hybrid RAG, many new versions
have been introduced for Hybrid RAG. I will introduce my own version of HybridRAG
in this thesis.

Research on Problem
This paper, titled RAG Does Not Work for Enterprises [2] explains why the problem
exists, in terms of using RAG in an enterprise environment and the limitation of using
RAG.
The author made a number of points about why having a RAG solution in enterprise
doesn’t work. Some of the reasons the author mentioned for why the existing RAG
fails to meet enterprise requirements are listed below:
• RAG does not provide clear explanations on why a certain document was re-
trieved and how they can influence the generated response. This creates a lack
of trust on the generated content.
• Enterprise deals with private customer data and current RAG architectures
often lack built-in access controls.
• Enterprise usually has unique data schema, taxonomies, and domain-specific
terminology that current RAG systems have limitation in terms of generating
accurate responses.
• Enterprise have their own IT infrastructures, that a RAG system should be able
to integrate into.
I choose to reference this paper, as it highlights that the problem does exists with
using RAG in an enterprise environment. Another major point, that is mentioned
is regarding the diﬀiculty of handling domain-specific terminologies and taxonomies
within enterprises. This supports the motivation for my proposed artifact (in the
later section). The artifact aims to incorporate domain-specific knowledge sources to
improve relevance and quality of generated outputs.
In this next research paper [3], which is titled Long-Context LLMs Meet RAG:
Overcoming Challenges for Long Inputs in RAG, it discusses how long-context de-
creases the quality of retrieved passages. In this paper, as an insight, it is mentioned
that they also noticed even when relevant passages were retrieved, irrelevant ones
(called ”hard negatives”) confuses the LLM and degrades its performance.
I referenced this paper, as it highlights how RAG faces some challenges with long
retrieved contexts. We have seen from the challenges of HelpSearch (the chatbot),
sometimes it answers the question with irrelevant information. It is relevant in this
research paper, as the artifact will be provided with even more context. This is be-

be able to answer better if it is provided with a long context.

Another research paper raised a question in the title, How much can rag help the

in terms of achieving deeper reasoning, when it is provided with both relevant and
irrelevant context. As a result, if the RAG system can’t perform deeper reasoning,

RAG alone cannot do deep reasoning, another solution is provided in the paper

into a short virtual token, which is a summary. The model is then trained to use

What was common in these papers is that RAG is limited in terms of differen-

answer accuracy.

2.1.5
Existing Research on Challenges
Now that it is clear what the problem is, and that it does exists. The next study was
conducted to examine relevant academic research that solved similar problems like
SimCorp.
In this research paper titled, HopRAG: Multi-Hop Reasoning for Logic-Aware
Retrieval-Augmented Generation [5], the findings they mentioned are:
• Systems based on only lexical or semantic similarity treats text passages as
disconnected units, which means missing intermediate steps and giving incom-
plete workflow. But the passages can be used as starting points to reach truly
relevant ones. For example, in a Knowledge Graph performing multi-traversal
of graph can navigate to find the truly relevant passage.
• They also proposed establishing relations between text chunks. Their proposal
called HopRAG: Multi-Hop Reasoning, addresses this by introducing logical
reasoning across text passages. Multi-hop reasoning is combining information
from multiple relevant nodes, and using all of the nodes to generate an answer.
Their findings showed that HopRAG reasoning retrieval outperforms traditional
RAG up to a large extent, when compared to only RAG solutions.

Their findings proved that retrieving semantically similar chunks without any re-

In another research paper titled, OG-RAG: Ontology-Grounded Retrieval-Augmented
Generation For Large Language Models [6], the following findings were made:
• Standard RAG fails to answer accurately in domain-specific workflow due to

an improvement in answer quality and also reduced hallucination (generation

The results

Overall, the findings proved that domain modeling is important for factual accuracy
in answers, and also for retrieving complete workflow information from the documents.
In this article, which is titled CRP-RAG: A Retrieval-Augmented Generation
Framework for Supporting Complex Logical Reasoning and Knowledge Planning [7],
they used a technique called reasoning graph, which is not a graph but a process of
decomposing a user query into interconnected retrieving steps. These steps are used
to guide the retriever to find relevant information from the context available. This
helps to improve the retrieval process and provide more accurate answers. CRP-RAG
also outperformed baseline RAG solutions. It showed improvements in accuracy and
robustness of the generated content. This also highlights indirectly that some sort
of knowledge construction is effective in LLM reasoning. As the previous research
discussed, RAG is not able to tackle complex queries that require deep reasoning.
This next research article, titled FastRAG: Retrieval Augmented Generation for
Semi- structured Data [8], also concluded that hybrid retrieval significantly improves
answers accuracy and retrieval quality. They also found the structuring content can
optimize the system performance.
Lastly, an article titled Empowering LLMs by hybrid retrieval-augmented gener-
ation for domain-centric Q&A in smart manufacturing [9], also proposed a hybrid
approach, using a hybrid KG-Vector RAG framework.
This integrates structured
Knowledge graph data with unstructured vector retrieval. The evaluation also showed
improvement in answer accuracy as well as in context precision. Another discipline
was utilized in this article called, Prompt Engineering (PE). Prompt Engineering
refers to crafting a prompt for the generative AI models to generate desired answers.

Research Methodology
This project followed the research methodology of Design Science Research (DSR)
[10, 11]. This methodology provides a structured approach to create an artifact by
conducting scientific research for solving real-life world problems. DSR starts with
phase one which is problem formulate phase. In this phase, we investigate and ana-
lyze the problem. This also includes studying the relevant literature to validate the
importance of the problem. Based on this study, we formulate research questions by
analyzing the gap between the existing solutions and the artifact. These questions
are used to guide the project for designing and developing the artifact.
For the design and development phase of the artifact, an agile approach was used
including elements from Scrum. This phase was tracked through the Azure DevOps
5. Sprint boards which helped with structured sprint planning, task tracking, and
monitoring the progress of the project overall. Each sprint lasted 2-4 weeks with daily
stand-ups to provide continuous feedback, monitor progress, resolve any blockers and
re-prioritize tasks. This phase was iterative, new tasks were frequently added as the
project progressed. Weekly catch-up meetings were held separately with both the
university supervisor and the company’s supervisor and every third week included a
joint meeting with both supervisors.
A milestone plan was used to guide the overall direction of the project. Each mile-
stone had sprints that were structured by combining user stories, each of which was
further broken down into tasks. The development of these tasks were feedback driven
and often changed with new requirements, extending its completion time if needed.
A risk plan A was made at the start of the project to assess potential challenges that
could impact the thesis progress. Each risk was evaluated using an impact-probability
matrix, and both the impact and probability of the risk were scored from 1 to 5. The
sum of both was used to identify the severity of the risk which helped to prioritize to
mitigating the highest risk first. This approach helped to address the potential risk
to ensure the project’s success.
After the design and developing phase of the artifact, the last phase of DSR was
performed and that was to evaluate the artifact. In the evaluation phase, quantitative

3.4.2
Think-on-Graph 2.0
The authors of this literature proposed a framework called Think-on-Graph 2.0 (ToG-
2), which tightly integrates structured knowledge graph (KG) retrieval with unstruc-
tured text retrieval (via dense vector search) in an iterative loop.
• Initial Iteration: When the user asks a question, the system begins first by
extracting entities from the question and linking them to other entities in the
knowledge graph by using an entity linking system. Not all entities are useful,
so LLM is used to prune irrelevant entities and only select a subset of relevant
entities, which the authors called topic entities.
From the vector database, the system retrieves top-k document chunks related
to the topic entities using a dense retriever model (DRM). Dense Retrieval
Models [15] use neural networks and embeddings for mapping both queries and
documents into a continuous vector spaces. Relevance is calculated by measur-
ing the distance between these vectors.
This was the initial retrieval process. After retrieving relevant document chunks,
LLM is used as a judge to ask is this information enough or not to answer the
question? If yes, the answer is generated, if not, the system starts the reasoning
loop.
• Loop iteration: The reasoning loop involves both retrievals, vector search and
graph search.
In the loop, for each topic entity, graph search is performed that extracts the rel-
evant relations (nodes and edges). The LLM prunes irrelevant relations further.
The entities in the relations are used further to perform a vector search. The
system performs semantic similarity search and scores the text chunks based on
how relevant they are, given the entity and its relation to the question. Based
on how useful the retrieved chunks are, the system scores the entities that was
used to search them. The top-ranked entities become the new topic entities for
the next round. Before going into the next round, the LLM is used to judge
again with the context available. If that is still not relevant enough to answer
the question, then it goes into the next round.
==PAGE:44==
32
3 Solution Design
After each iteration, the system uses LLM to generate answer from the rela-
tions of the knowledge graph, text chunks from vector database and previously
collected evidence in the iterations.
If the LLM decides that the provided context is enough to answer the question,
the answer is generated and the loop stops.

produce good results. Generative AI models use natural language to produce com-

generate contextually relevant responses in real time conversation.

weight = (hits / total_keywords) + log(hits + 1)
Where:
hits: number of keywords of a concept actually found in a document
total_keywords: Total number of keywords of a concept
log(hits + 1): logarithmic bonus to favor concepts with higher keyword matches
hits / total_keywords is done to have a fair score and not penalize concepts with
fewer or more keywords. For example:
• A concept with 10 keywords where 5 match gives 0.5

• A concept with 4 keywords where 2 matches also gives 0.5
In log(hits+1), logarithmic bonus helps as it grows slowly.
This means if one

is not drastic. The reason why +1 was added is due to the fact, if the concept found
no hits, then log(0) which is undefined. To ensure if no hit is found, the minimum
value is log(1) = 0.

For taxonomy generation, another Function App was implemented for orchestrating

The LLM is instructed to generate hierarchical taxonomy with this prompt:
A hierarchical taxonomy is a structure where taxons (categories or concepts) are
organized in levels. Each taxon may have one parent and multiple children, forming

This process was performed in batches of size 100 concepts at a time, as there were

It uses concepts.json to extract all the concepts, as well as each concept’s domain
information. Glossary.json is used to extract the definitions of the concepts. The
reason for generating definitions separately and then using them in taxonomy, was
to keep both logics separate.
In this way, the prompt is constructed to focus on
performing one main task, since there are many concepts the LLM needs to process.
The main grouping of concepts for each domain is done in a class called Taxono-
myGenerator. I worked with a couple of techniques for grouping the concepts in the
taxonomy. After testing, I narrowed it down to one.
Techniques for Grouping
• In the beginning, I used Jaccard similarity to cluster related concepts based on
their lexical overlap. I tokenized the concepts into lowercase words and then
computed pairwise similarity scores between them, using this formula:
Jaccard(A,B)= ￿A∪B￿/￿A∩B￿
I set the similarity threshold of 0.2, meaning any two concepts sharing more
than 20% of their tokens, were considered similar. Most concepts consisted of
only 2–3 words. This approach had certain limitations, such as it wasn’t able
to capture abbreviation (NAV and ”Net Asset Value”) which are not lexically
similar. The calculation process was also slow. This is because, the process
involved matching all pair of concepts, resulting in O(n²) time complexity. As
==PAGE:65==
4.6 Glossary and Taxonomy
53
the number of concepts increased, the number of comparisons grew quadratically.
This technique can work well for small number of concepts, but not in this case,
as there are hundreds of concepts.
• The next technique I used was semantic similarity. Azure embedding model
was used to make clusters of semantically similar concepts.
At first, I used
both the concept name and its definition. I tested this on a small group of
concepts to observe the results. This also grouped concepts that might shares
same terms but were slightly different. For example, Pre-trade and Post-trade
were grouped in the same category. Pre-trade is executed before the trade and
post-trade is after. Pre-trade checks prevents non-compliant trades from being
sent to the market, while post-trade checks are used to detect issues in already
booked transactions. In SimCorp, to perform pre-trade or post-trade validation,
different steps are required.
I used a semantic similarity measure again to cluster the concepts. But this time,
I used only the concept names, which resulted in more distinct groups. That’s
why I opted out to use only the name, this led to more distinct clustering and
also helped avoid over-grouping conceptually different terms that shared generic
wording in their definitions. The embedding model is used to embed concept’s
name. Then semantically similar concepts are grouped using a technique called
cosine similarity with threshold ≥0.8.

calculated in the same as tokenHitCount, using log(domainConceptCount+1)
but the weight is multiplied by 5.0. These two factors are calculating different
things, tokenHitCount is measured per concept, since tokens match individual
concepts and can therefore result in higher counts than domainConceptCount.
In contrast, domainConceptCount measures the number of concepts matched
within a domain.
• domainSimilarity measures the similarity fit, whether the domain to which
the matched concept belongs has any similarity with the question. This helps
prioritizing concepts from the domain that is most relevant to the question.
This is already in a range of 0.0-1.0, so a logarithm is not needed here. The
weight is set to 10.0 to bring it into the same range as other factors.
• conceptDomainCount is used to penalize generality. If a concept appears in
many different domains, it is likely less specific. That’s why this factor serves
to apply a penalty, but not too heavily, so the weight is set to 5.0.
The score for each concept is computed using the following formula:
==PAGE:76==
64

Score = alpha
* matchStrength +
beta
* log(tokenHitCount + 1) +
gamma
* log(domainConceptCount + 1) +
delta
* domainSimilarity -
epsilon
* log(conceptDomainCount + 1)
This is used to score all the concepts. Only the top 10 concepts are kept for further
steps.
5.3
Top-Related Concepts
For the top 10 concepts, their related concepts are then evaluated. This is done to
understand the contextual relationship between each concept and its related concepts
within the same group. To rank the top related concept, Jaccard similarity and cen-
trality was used. Jaccard was used as it works best for smaller groups. Here the focus
is on group tightness, that is, the specific concept cluster.
The following formula is used to score the related concepts:
Score = 0.7 × JaccardSimilarity(matchedConcept, relatedConcept)
+ 0.3 × (centrality / maxCentrality)
In this formula:
• JaccardSimilarity is checking the lexical overlap between the concept and the
related concept
• centrality measures how important the related concept is within the group.
• maxCentrality is used to normalize centrality to a range between 0 and 1.
JaccardSimilarity is given greater importance with a weight of 0.7, as the focus
remains on the matched concept.
Therefore, identifying which related concept is
the most lexically similar is considered more important. The taxonomy was group
already using an embedding model, so the most semantically similar concepts are
already clustered together. That’s why Jaccard similarity is suﬀicient for filtering the
top related concept. Only one related concept is kept because we already have the
top 10 concepts. This related concept is used solely to expand the search space while
still keeping it within reasonable limits.

5.5
Query Embedding
In the next step, the original user query is enriched with the triplets summary. This
simply involves combining the triplet summary as context with the original user ques-
tion.
In the subsequent search, vector search is performed between the query and the
available chunks of documents. These chunks are stored in vector form. To ensure the
enriched query has the same structure, the embedding model text-embedding-ada-002
from Azure OpenAI is used. This embedding model generates the embedding for the
enriched user query. This enriched query is converted into a vector that can be used
later for vector search.
5.6
Concept-Linked Chunks
The documents IDs of the concepts that were stored during triplets extraction (in
section 5.4.2) are used in this step. As the reader may recall, these document IDs are
used in Knowledge Graph, and are identical to the parent_id field of each chunk in
the Azure Cognitive Search index, which serves as the vector database.
I implemented a method GetChunksByDocumentIdsAsync() in VectorIndexSearcher
class. This method performs vector search only on those chunks that belong to the
document IDs stored earlier. This is accomplished using the search.in filter, on the
parent_id field, which represents the document ID. This type of search improves rel-
evance as these document IDs are directly associated with the concepts.

within the filtered set of chunks. The vector search is powered by the Hierarchical

optimized for high-speed and scalable vector similarity search. It operates by orga-

nizing vector embeddings into a multi-layer graph structure (not a knowledge graph),

comparisons across all stored vectors.
The reason to have only top 5 chunks is that the highest-ranked chunks are usually

Domain Chunks
One more separate vector search is performed between the enriched query and all
the available chunks for documents belonging to two domains: Compliance Manager

mantic vector-based search in Azure Cognitive Search.
Filter search.in is used to restrict the search to only documents within these the
two domains. Aside from this filtering, the vector search is performed the same way
as before. The method returns the top 5 matching content chunks ranked by their
semantic similarity to the input query vector.
The first vector search was restricted to only the list of document IDs. This sec-

the entire index. This broader search provides a wider contextual match within the
two domains.
After this step, the context we now have includes the triplets summary, 5 concept-

For this thesis, GPT-4o was used to evaluate completeness and groundedness. An-

• 5 = Fully complete and grounded answer

6.3.4
Findings
The TGH-RAG system integrates both a graph database and a vector database, which
are used to enhance reasoning and coverage. It leverages triplets from the knowledge
graph (KG) to represent relationships between entities. This enables the system to
draw structured connections that go beyond surface-level text retrieval.
Across all iterations, the prompt for HelpSearch remained unchanged. The prompt
was modified only for the TGH-RAG system developed in this thesis. Even if the
HelpSearch prompt were adjusted, it would not benefit from the changes, as it does
not have access to the knowledge graph data.
Since the TGH-RAG system incorporates both indexed chunks and graph-based
knowledge (triplet summaries), it is able to generate more complete and structured
answers. This is also demonstrated by the results of the last iteration, where the
artifact achieved the highest score in terms of completeness.
However, groundedness is scored by checking whether the answer contains exact
phrases or sentences from the retrieved text. Even if the system provides a correct
answer based on facts from the knowledge graph, it might still receive a low ground-
edness score because those facts are not written word-for-word in the chunks. This
leads to lower groundedness, representing a tradeoff between HelpSearch and the
TGH-RAG system.
HelpSearch provides minimal and general, but grounded, answers. In the case of
HelpSearch, it pulls the actual wording from the documents and presents that as an
answer. In contrast, the TGH-RAG system uses triplets, which often have phrasing
entirely different from the original documents. For example, even if we used a triplet
summary as context, such as:
“Document Align_to_benchmark belongs to module Modelling and Rebalancing
and shares concepts with Simulation_workflows_in_Asset_Manager.”
This statement is factual and semantically accurate, but the exact sentence does
not appear in the chunks. As a result, the grounding score flags this as unverifiable
from the text, leading to a lower score.

6.4
Qualitative Evaluation
The qualitative evaluation was performed through user interviews and blind compar-
ison. The interviews shared some characteristics with A/B testing. It was a blinded
qualitative comparison in which all interviewees reviewed multiple anonymized out-
puts for each question, with all systems equally represented across the evaluation
set.
6.4.1
Methodology
To conduct the interviews, I used the following setup:
• Participants: Six interviewees participated in the evaluation. They were pro-
fessionals working at SimCorp, including product managers, software engineers
with experience in the domain, one business consultant specializing in Compli-
ance Manager. Some participants were also experts who actively contribute to
the user discussion forum by answering questions. All participants were famil-
iar with the domain and the types of information typically retrieved by such
systems.
• Questions: Each interview included between 5 and 10 domain-specific ques-
tions, depending on the available time.
• Time: Each interview lasted approximately 30 minutes, during which partici-
pants assessed the provided answers.
• Evaluation Material: For each question, participants were shown three types
of answers:
– A reference answer from the user forum, publicly available within SimCorp
(the interviewees were only aware that this answer was taken from the user
discussion forum).
– An answer retrieved by HelpSearch
– An answer generated by one of the three iterations of the TGH-RAG sys-
tem developed in this thesis.
• Interview Design: The interviews were designed to minimize bias toward
either system. Participants were aware only of the reference answer’s source
(the user forum). The other answers, from HelpSearch and TGH-RAG were
anonymized and labeled with neutral identifiers (e.g., HNT2, HT2, HT1, HT3),
so participants could not distinguish between systems.
The goals of the interviews were to:
1. To assess the quality of the answers from the user’s perspective.

2. To understand how domain users perceive the usefulness, clarity, and relevance
of answers provided by both retrieval systems.

6.6
Future Work
The future work to achieve even better results in terms of improving TGH-RAG
system performance in both completeness and groundedness includes:
• Further considering filtering techniques that can filter the triplets more precisely
to retrieve only highly relevant content-related triplets.
• Improving the prompt from the third iteration (Section 6.3.3) to provide clearer
instructions on how to use the provided data while answering the user’s question.
• Enhancing context retrieval, such as by limiting the triplets even further, which
could improve groundedness.
• Adding user role awareness so that the answers are more tailored to the user’s
specific needs, while still maintaining multi-domain knowledge.
• Expanding the knowledge graph to include more than two domains to evaluate
how the system’s performance scales and to test its effectiveness in a broader
context.

6.7
Scientific Contributions
Here, we will revisit the scientific contributions mentioned in Chapter 1. We will
go through each contribution individually and assess the extent to which they were
achieved.
• Answer Quality: By integrating both graph-based relations with document
retrieval, we can see from the evaluation results that the answer quality has
indeed improved for complex and multi-relational queries. For example, in the
third iteration (section 6.3.3) within the Compliance Manager domain, the eval-
uated answer quality increased from 15.18% to 52.03% with the TGH-RAG
system. Similarly, in the Asset Manager domain, the answer quality increased
from 9.74% to 41.57% with the TGH-RAG system. This demonstrates a signif-
icant improvement over a simple RAG search.
• Grounding Evidence: Another contribution is the integration of a knowledge
graph with vector-based retrieval through a shared linkage mechanism. (If the
reader recalls) Each triplet in the knowledge graph is linked to the same docu-
ment ID used in the vector database, ensuring that retrieval operations return
chunks from the same source document as the originating triplets.
This alignment enhances contextual consistency and allows the system to re-
trieve not only semantically relevant information extracted separately, but also
document-specific evidence that supports more comprehensive and semantically
grounded answers. However, since triplets often abstract over the exact phras-
ing of the text, this can introduce a trade-off: answers may be factually correct
and contextually justified but not directly traceable to the document source,
leading to lower groundedness scores under strict lexical evaluation criteria.
However, in the third iteration, when the prompt was designed more effectively
to instruct the LLM on how to use triplets and chunks, groundedness increased
significantly for one domain, Asset Manager, where TGH-RAG outperformed
HelpSearch.
For Compliance Manager, the groundedness was almost at the
same level as HelpSearch. Overall, the system improved to a large extent in
terms of groundedness. (section 6.3.3)
• Support for Hierarchically Structured Enterprise Documents: The
TGH-RAG system is designed for enterprise document collections that have a
hierarchical metadata structure. This is evident in how the knowledge graph
(KG) is structured, including the nodes, edges, and the way metadata is con-
nected. For example, the triplets with nodes and relations between metadata
==PAGE:108==
96
6 Evaluation
can be represented as:
Document:aHR0cHM6Ly.. -> belongs_to -> DocType:User Manuals.
In terms of retrieval evidence, the system then pulls the document using its ID
from under DocType: User Manuals.
This design can also be applied to other enterprises if their documents are
structured in the same way. That means each document contains YAML struc-
tured metadata at the top and the textual content under that. This indicates
the document type, the domain it belongs to, the version of the document, and
similar attributes.
This interpretation of the contribution is my own, and I am assuming it could
work if the documents have the same formatting, as many enterprises have doc-
uments from multiple domains. If the documents include information about
which domain they belong to or what type of document they are, it can be help-
ful for creating a domain model for these types of documents without needing
to hire domain experts to build the knowledge manually.
• Simplified Pipeline:
As shown in Chapter 3, Fig, 3.1, the TGH-RAG pipeline follows a significantly
simpler design compared to multi-agent systems such as RAG-KG-IL [17]. Also,
see the Fig, 6.9.
RAG-KG-IL relies on multiple interacting agents, incremental reasoning steps,
and dynamic knowledge graph updates. This requires orchestration and control
logic. In contrast, the TGH-RAG pipeline operates in a single flow. It first
extracts entities, then retrieves concepts and triplets. It then performs vector
search as well as document-linked vector search to obtain relevant chunks, and
completes the process with a single LLM call. This simplicity reduces system
complexity and engineering overhead, making the pipeline more practical for
production deployment.

RQ1 Hybrid RAG System Design
How can a Hybrid RAG system be designed using knowledge graphs, metadata and
taxonomy to support more precise and complete retrieval in a multi-domain context?
With RQ1, we focused on designing the TGH-RAG system developed in this re-
search. In Chapter 3, the emphasis was on how to design a system that uses a knowl-
edge graph, metadata, and taxonomy as precisely as possible in order to achieve
complete retrieval.
In Chapter 4 (Sections 4.5 and 4.6), it is demonstrated how both the knowledge
graph and taxonomy were constructed using metadata.
From Chapter 5 (Section 5.2), you can also see how different strategies, scoring
methods, and unique techniques were applied to the taxonomy to enable the most
complete retrieval possible.
The following techniques were used for taxonomy search:
• Tokenization
• Parallelize Keyword Matching
• Fuzzy and partial matching
• Filtering such as handling prefix, suﬀixes,
• Taxonomy Matching
• Ranking
From the architectural design (Section 3.2), we can see that the retrieval process
is a single flow without steps executed in a loop. In TGH-RAG, the system combines

the use of the knowledge graph, metadata, and taxonomy. The taxonomy was created
and saved in a JSON file, with a hierarchical structure connecting concepts to domain
groups. Furthermore, these concepts were also part of the knowledge graph, creating
a more complete and stronger connection between the two. Metadata was used in
both the vector database and the knowledge graph. The taxonomy and knowledge
graph both encompass multi-domain knowledge. Metadata helped in retrieving infor-
mation specific to individual domains.
As shown in Section 6.3.3, from the statistical analysis across iterations, we ob-

RQ2 Retrieval Precision
How does the proposed system impact retrieval precision compared to standard
RAG?
The proposed system, TGH-RAG, positively impacts retrieval precision compared
to standard RAG, which in this case was HelpSearch.
In the proposed system, integrating concepts, the knowledge graph, and taxon-
omy improves results by cross-checking whether the same concepts exist in both the
taxonomy and the knowledge graph. Grouping the most relevant concepts in the tax-
onomy based on semantic similarity also enhances precision. This approach provides
a stronger focus not only on domain-specific context but also on subgraph coherence,
as the taxonomy organizes domains into groups of semantically similar concepts to
improve retrieval accuracy. Additionally, chunks are cross-checked with the knowl-
edge graph using document IDs, which are consistent across both systems.
In contrast, HelpSearch performs only vector search, which is semantically precise
but often lacks suﬀicient contextual relevance.
This improvement is confirmed in Section 6.3, where results from the third iter-
ation show that TGH-RAG outperforms HelpSearch in terms of both completeness
and groundedness.
However, in iterations 1 and 2, we observed a trade-off between context and
groundedness because the generated answers did not always match the exact wording
found in the documents. Knowledge from the triplets may be accurate but is often
not expressed word-for-word as it appears in the documents. From all the iterations,
we also saw that TGH-RAG is able to answer more questions compared to RAG, even

when the questions are complex.
From Section 6.4, the qualitative evaluation based on interviews also showed that,
even with complex queries, TGH-RAG at least tries to help debug the issue with
some suggestions, while HelpSearch does not attempt to do so.
Nevertheless, TGH-RAG demonstrates a meaningful improvement in retrieval pre-
cision over standard RAG, particularly in complex and multi-domain contexts.
RQ3 Completeness and Groundedness
Does the proposed system improve the completeness and groundedness of answers
compared to RAG?
Yes, the proposed system improves both the completeness and groundedness com-
pared to RAG. From Chapter 6, Section 6.3, across all iterations, we saw that com-
pleteness improved in both domains compared to HelpSearch (RAG), with significant
gains observed in the last iteration.
From the quantitative evaluation done in the evaluation phase, we saw the com-
pleteness increase from 15.18% to 52.03% and from 9.74% to 41.57% in Compliance
Manager and Asset Manager respectively.
This also shows completeness increases in terms of multi-domain questions. The
results in the third iteration increased significantly because we also saw that the
model needs to be instructed well in order to use the data structures like triplets and
chunks together. Good prompt engineering (PE) could potentially further improve
the results.
HelpSearch’s groundedness was higher in iterations 1 and 2, but that is also be-
cause of the strict lexical overlap, as it used only five chunks, while TGH-RAG is
using much more context, such as graph triplets and chunks, and in the first two
iterations it was not provided with enough information on how to use it.
Therefore, prompt design is necessary in order to give clearer instructions, and
thus, in iteration 3, the groundedness also increased for TGH-RAG compared to
HelpSearch.
So yes, it does improve both criterion, but it can be improved further through
better filtering of context and triplets to see even more improvement in the results.

Can knowledge graph help disambiguate domain-specific terms?
Yes, the knowledge graph can help disambiguate domain-specific terms. In this
research, the knowledge graph was constructed using metadata, concepts, and rela-
tionships between the concepts.
During the concept extraction process described in Chapter 4 (Section 4.4), when
concepts were extracted using the LLM, the prompt was designed so that if there were
two concepts belonging to both domains, they were added to both domains. These
concepts were later added as nodes in the knowledge graph. For example, if two
domain documents shared the same concept, their nodes were linked to each other
using a relation label.
This helps reduce ambiguity by showing connections between the two domains
and clarifying in which contexts the concept is used. It also ensures that the system
retrieves relevant answers specific to the correct domain.
From Chapter 6.3.3, the TGH-RAG system demonstrated better precision and
contextual relevance compared to HelpSearch. The knowledge graph plays an impor-
tant role in disambiguating domain-specific terms and improving the overall quality
of retrieval and answer generation.
RQ5 Combining Retrieval Methods
RQ5: What effect does combining different retrieval methods have on the retrieval
answer quality?
Combining different retrieval methods has a positive effect on retrieval answer
quality. In this research, the TGH-RAG system integrates both chunk retrieval (from
vector database) and triplets retrieval from knowledge graph, along with metadata
and taxonomy. Concept retrieval played also a big part in the retrieval process. The
same concepts were used throughout taxonomy and knowledge graph.
By combining both vector and graph based retrieval, allows the system to retrieve
not only semantically similar chunks from documents but also structured subgraph
(in the form of triplets) from the knowledge graph, which provides context and rela-
tionships between concepts.
This hybrid approach helps improve both the completeness and precision of an-
swers.
From Chapter 6.3.3, the evaluation results show that TGH-RAG achieved
higher scores in answer quality compared to HelpSearch, which relies only on vector
search.

a trade-off in groundedness, as some information derived from the knowledge graph
may not directly match the exact wording in the documents. Despite this, the overall

Hypothesis
H1 stated that combining taxonomy, knowledge-level reasoning (using a knowledge
graph), structured metadata (e.g., topicType, domain, docType, module), and a
hybrid retrieval mechanism that uses both vector-based and graph-based retrieval can
improve the contextual relevance, procedural completeness, and domain alignment of
responses when applied to complex and multi-domain documentation.
Based on the results from both the quantitative and qualitative evaluations pre-
sented in Chapters 6.3 and 6.4, this hypothesis is supported to a significant extent.
• Contextual relevance: The integration of the knowledge graph and taxon-
omy helped disambiguate domain-specific terms and provided broader context
for complex queries. This was reflected in improved precision and relevance
scores compared to HelpSearch. By extracting concepts and linking them to
multiple domains, especially when they could coexist in two domains, the sys-
tem also improved contextual relevance because it can answer questions from
both domains and provide users with a complete understanding of the steps
required for their tasks. For this, refer to the example given in Section 6.6 (last
point “multi-domain queries”).
• Procedural completeness: Across all iterations, TGH-RAG showed improve-
ments in completeness scores, but specifically, the system achieved a significant
performance boost in iteration 3 (Section 6.3.3). Answer quality increased from
15.18% to 52.03% in the Compliance Manager domain and from 9.74% to 41.57%
in Asset Manager, showing its ability to provide more complete answers.
However, from Section 6.4, based on the interviews, there was some feedback
regarding the phrasing in the TGH-RAG system. This is possibly because the
system responds using concept wording in the answers rather than the actual
wording from the documents. Additionally, it sometimes provides information
in a very concise manner that could be elaborated further.
• Domain alignment: The system effectively aligned responses to the correct
domain context through the use of metadata and taxonomy. User questions were
matched with concepts in the taxonomy, and scored based on how relevant the

CHAPTER 7
Conclusion
This thesis has successfully demonstrated the benefits of combining taxonomy, knowl-
edge graph, structured metadata, and a hybrid retrieval mechanism that integrates
both vector-based and graph-based search. The TGH-RAG system developed in this
research has shown improvements compared to the standard RAG search (in this
case, HelpSearch), particularly in answering questions that were complex or required
context from multiple domains.
From the quantitative evaluations, we saw significant improvement in answer
quality by evaluating completeness, and in both domains, TGH-RAG outperformed
HelpSearch.
In the Compliance Manager domain, answer quality improved from
15.18% to 52.03% , and in the Asset Manager domain from 9.74% to 41.57% . This
highlights the system’s ability to provide more complete and contextually relevant
answers compared to the HelpSearch approach. This is likely due to the use of the
knowledge graph.
The knowledge graph, concept extraction, and taxonomy were also a big part of
this system. They helped connect and link multiple domains together, which further
helped answer questions that required multi-domain knowledge. The TGH-RAG sys-
tem also used multiple retrieval methods, combining both graph search and semantic
search.
The ranking and filtering mechanisms used also ensured that many perspectives
were considered, such as concept relevance in the domain, user intent behind the
question, as well as checking prefixes and suﬀixes, and handling terms carefully, since
enterprise companies use a lot of short terms, all with important meaning in their
domain context.
The qualitative feedback also provided some insights into the system’s strengths.
Users appreciated the system’s ability to answer questions and to try helping the user
with queries that required debugging or further investigation. Users also accepted the
chatbot suggesting potential solutions that might or might not work, as a way to give
them a head start or useful knowledge.
The simplified architecture of using a single pipeline to avoid iterations or overuse

that can be further enhanced and is easy to deploy in production.
The domain model being general can also inspire or help other scientific commu-

similar in their domain without needing expensive resources.
Overall, this research provides compelling evidence that integrating structured
knowledge representations and hybrid retrieval methods leads to significant improve-
ments in enterprise information retrieval. The TGH-RAG system stands as a promis-
ing step toward more intelligent, precise, and context-aware retrieval solutions, with
strong potential for future improvements and broader applications in the future.
