Abstract
As robots becomes an increasingly large part of everyday life, it is important that they
are capable of ad hoc collaboration with humans. For this collaboration to be possible,
the robots need to know what the human is currently working on, and what their goal is.
Having the human explicitly communicate their intent to the robot is often undesirable, and
ideally, the robot will be able to seamlessly infer this information on its own, while assisting
the human. We design a data structure and algorithm that can allow a robot to identify
subtasks it can work on to assist a human, without being told or determining the goal first.
The algorithm uses observations of the humanãs actions, to exclude subtasks and goals
that it determines the human is uninterested in, and attempts to isolate subtasks that are
beneficial for reaching the remaining possible goals. We discuss the characteristics of the
approach, and restrictions for environments and planning algorithms it can be paired with.
An existing planning algorithm and environment is used to test the proposed algorithm,
and assess its responses to a humanãs actions. Based on these tests, we give a qualitative
assessment of the algorithms ability to help a human achieve their goal, as well as the
algorithms effectiveness in different scenarios.

Introduction
As robots becomes an increasingly larger part of everyday life, it is no longer enough for
them to be able to solve tasks individually. Robots must be able to collaborate both with
humans as well as other robots. This collaboration will introduce additional complexity,
as the robots need a way of knowing what the human or other robots are working on.
With this knowledge, the robot must then figure out whether to assist with the task others
are working on, or work on a separate related task. One way of acquiring the knowledge
of what others are working towards, is for the humans and robots to communicate their
intent to each other. However, communicating intent between robots and humans can be
slow and cumbersome, and it is therefore preferable if the robot can infer the intent of its
collaborators on its own [LRS99].
In many environments, there may be multiple possible goals. In general, algorithms that
plan based on inferring what other agents are currently working on, either assumes that
the goal being worked towards is known beforehand, or makes estimations of what goal
is being pursued (e.g. [Wu+20], [Mui+15] and [Fre+19]). If the goal must be known, there
are existing algorithms for identifying the goal, based on observations of the collaboratôÙ
ing agents (e.g. [ERS15] and [SM20]). As mentioned, explicit communication between
human and robot is not ideal, and therefore it would be favorable to avoid having to tell
the robot the goal. Likewise, waiting for an algorithm to identify the goal is also not ideal,
as the planning algorithm could be forced to wait for a considerable amount of time, beôÙ
fore enough actions are observed to determine a goal. Using an estimation of the goal is
more efficient, as planning can start early, but it can lead to complications as it is difficult
to make accurate estimations until a certain amount of observations are made.
To circumvent these detriments, we explore a concept that does not rely on isolating a
single goal, which is derived from theory of mind and how humans collaborate. When
a human attempts to assist another human with a task, they do not necessarily need to
know what the goal is. Sometimes, humans can identify tasks that will be helpful towards
the goal, even though there might be multiple possible goals. To give an example, imagôÙ
ine Alice observing Bob taking a bag of potatoes out of the refrigerator and putting them
on the kitchen counter. Alice does not know if Bob intends to simply boil the potatoes
or make fries, but regardless of Bobãs goal, Alice can infer that the potatoes have to be
peeled, and can start assisting with this immediately.
Utilizing this concept, robots would be able to start assisting their collaborators earlier
than if they had to identify the goal first, and it does not rely on having a correct estimation
of the goal.
In this thesis, we propose a method for utilizing this concept in collaborative planning. We
develop a graph data structure, that models subtasks of various goals and their environôÙ
mental prerequisites, along with an algorithm for identifying subtasks that will be beneficial
to the human. The algorithm uses observations of the human to find subtasks that, if comôÙ
pleted, will get the human closer to the goals that are still possible, based on the actions
taken so far.
A planning algorithm will be able to use our proposed graph and algorithm to begin planôÙ
ning for subtasks, that benefit their collaborators, without having to isolate a goal. This
approach has the advantage of allowing the planner to start planning earlier. However,
this comes at the cost of losing the ability to plan many steps ahead, since the planner
will only be able to plan towards individual subtasks, as it might not know what subtasks

2.1
The Too Many Cooks Environment
The Too Many Cooks environment used in this project is an extension of the environment
used by [Wu+20], who made their code open source at [WW20]. The environment is
originally inspired by the video game Overcooked [Gam16].
The environment consists of a 2D gridôÙworld kitchen which contains a number of counters.
These counters can contain movable objects in the form of plates and ingredients like
Tomato and Lettuce. The counters can also contain immovable cuttingôÙ and deliveryôÙ
stations. Additionally, the kitchen contains between 1 and 4 cooks, that can move around
on the floor, and interact with objects on the counters.
In addition to the kitchen, the environment also contains recipes. A recipe contains a goal
state, which involves getting a specific combination of ingredients to a delivery station,
as well as a partially ordered series of subtasks that can be executed to reach the goal.
These subtasks each have preôÙconditions that must hold for the task to be executed as
well as postconditions describing the effect of the subtask being completed.

The environment also contains two chopping stations, and a delivery station indicated
with a star. The source code and graphics for the environment is from [WW20].
Inside the kitchen, the cooks can interact with the environment in the following ways: A
cook can pick up movable objects from a counter, and can only hold one object at a time.
The cooks can put the objects down on the counters, but they can not pass objects directly

ãÂ The Merge(x1, x2) subtask involves merging x1 and x2 into a new item x3. The

ãÂ The Deliver(x) subtask is the final subtask in a recipe and represents finishing the
task being worked on, by putting it on the delivery station in the environment. The
deliver subtask has the postcondition Delivered(x). The precondition for Deliver(x)

2.2.2
Planning for Solving Allocated Subtask
Given a subtask allocation taã, agent i uses reinforcement learning to find a suitable
action for solving the subtask Ti it is assigned to. To simplify the reinforcement learning
algorithm, the agent will create a simple levelôÙ0 policy for each of the agents not assigned
to Ti. These levelôÙ0 policies simply assume that the agent is stationary. Using these
levelôÙ0 models the transition function T is converted into a simplified transition function T ãý,
where each of the agents not assigned to Ti, are assumed to follow the levelôÙ0 models.
T ãý is then used in a reinforcement learning algorithm, to find an approximately optimal
policy for agent i to solve subtask Ti.
If agent i is the only agent assigned to Ti, T ãý functions as a single agent transition function,
and the resulting policy can simply be used to find a suitable action for agent i.
If other agents are also assigned to Ti, the agent has to account for them and attempt to
cooperate with them. To do this, the agent emulates a centralised planner, that controls
all the agents assigned to Ti. Using an action space that combines the actions for all the
agents controlled by the planner, as well as a simplified transition function using levelôÙ0
policies for all agents not assigned to Ti, a joint policy for the subtask is found. Agent i
then takes the action it is assigned by this joint policy. Note that since the reinforcement
learning algorithm contains some stochasticity, the different agents might not arrive at the
same joint policy, and they might not even have assigned themselves to the same task.
This means that the cooperation of subtasks might not be perfectly coordinated.
The reinforcement learning algorithm used by BD is Bounded RealôÙTime Dynamic ProôÙ
gramming (BRTDP) [MLG05]. However, any model based reinforcement learning algoôÙ

The Subtask Graph and Algorithm
Here, our main contribution, the subtask graph and related algorithm is introduced.
The subtask graph models the subtasks of an environment, along with their preconditions
and postconditions. A robot then utilizes the graph along with observations of actions
taken by human and robot agents in the environment, to identify subtasks relevant to the
agents.
The intent of the subtask graph is to identify subtasks that a human will want completed,
based on their actions, without knowing the goal the human is working towards. The
advantage of this approach is that it is not necessary for a robot to identify, or be told, the
goal of a human, before it can start assisting them.
This chapter will first give an explanation of the intuition of the actions taken by the alôÙ
gorithm, and then give an introduction to the subtask graph data structure, as well as a
detailed description of the algorithm. Then, an example of the subtasks graph and algoôÙ
rithm used on the Too Many Cooks environment is given. We outline requirements for
environments the subtask graph can be used in, as well as planning algorithms that can
be combined with the graph to solve the identified subtasks. An informal analysis of the
algorithm is conducted, arguing for its correctness and efficiency. Lastly, we outline the
general structure, as well as some of the key design decisions in our implementation of
the algorithm.
3.1
Intuition of the Algorithm
As an example, imagine a robot in the previously described Too Many Cooks environment.
The goal of the robot is to assist a human in completing their desired recipe. The human
wants to create a salad consisting of lettuce and tomato on a plate, but the robot is not
aware of this. The tomato and lettuce are the only two ingredients that are available
together with plates, and the robot knows three possible recipes with these ingredients:
Chopped tomato on a plate, chopped lettuce on a plate and mixed tomato and lettuce
salad on a plate. The robot represents these recipes in a graph, that contains the subtasks
needed to complete each recipe, together with the conditions that must be met to complete
them. The graph the robot uses in this example is shown in Figure 3.1.
The human starts working on the recipe by picking up a tomato and placing it on the
counter.
The robot observes this and assumes that since the human picked up the
tomato, the corresponding subtask Get(Tomato) must be relevant to the humanãs recipe,
and marks it as such in the graph. Now the robot tries to identify additional subtasks that
will also be relevant to the humanãs recipe. In this case it sees that Get(Tomato) satisfies
a condition that is only required by the Chop(Tomato) subtask, and therefore concludes
that this subtask must also be relevant to the human, and marks it. At this point, the robot
is not able to identify further subtasks to mark as relevant, so it instead tries to assist in
chopping the tomato, with the assistance of a planning algorithm.

algorithm decide the best marked subtasks to help with, in order to fulfill the recipe.
Since the robot has now identified the recipe and marked all the subtasks relevant to that
recipe, no more changes will be made in the graph, and the robot will simply continue
assisting in completing the marked subtasks until the recipe is complete.

3.2
The Subtask Graph
The Subtask Graph models all the subtasks that can be completed in order to accomplish
a number of possible goals, which in the case of the Too Many Cooks environment is
a completed recipe. It additionally models the ordering constraints of the subtasks. For
example, it is not possible to put chopped tomato on a plate before the tomato has been
retrieved and subsequently chopped.
The aim of the graph is to assist a human by observing their actions in the environment.
Based on these observations, the graph tries to identify subtasks that will contribute to
the humanãs goal if completed, without the graph knowing the humanãs goal.
Figure 3.2: A segment of the subtask graph shown in Figure 3.1. The graph has been
truncated for simplicity. Nodes in the graph represents a subtask. Edges represents the
preôÙ and postconditions for the nodes, on inôÙ and outgoing edges, respectively. Nodes

The primary difference between marked nodes and traditional landmarks, is that a marked
node is not guaranteed to be necessary for the desired goal to be reached. This comes
from the graphãs partial ordering of subtasks, and the fact that multiple subtasks can have
the same postcondition. If a marked node shares a postcondition with other nodes, the
goal the node leads to can be reached without the marked subtask being completed, but
if the marked node were to be completed, the pursued goal will be closer to completion.
This means, that when multiple subtasks with identical postconditions are marked at the
same time, the planning algorithm would be responsible for identifying which of these
similar paths should be taken. The computation of marked nodes is outlined in Section
3.3.
3.3
The Algorithm
The algorithm uses the subtask graph to identify and mark subtasks that are deemed
desirable for the human agent to achieve their perceived goal.
When marking a node n, the algorithm also marks all the predecessors of n. This is done
to ensure that all ways of satisfying the preconditions of n are marked. This allows a
potential planning algorithm to choose the best way to achieve n.
For example, if the Deliver(LettuceôÙPlateôÙTomato) subtask is marked, the graph should be
agnostic to the order of actions taken to complete this subtask. It might be convenient for
the robot to first merge the Lettuce and Tomato, then plate it. It might instead be favorable
to plate the tomato first and then chop and add the lettuce later. The point is that the graph
has no means of identifying the ideal way to achieve a marked subtask, and as such it
will enable all options for a planning algorithm to choose from.
Later in the algorithm, undesirable nodes gets deactivated in the graph (see Section
3.3.2). When the algorithm searches forward in the graph, to identify new nodes to mark,
it disregards any paths that contains deactivated nodes. However, when marking the preôÙ
decessors of a subtask, deactivated predecessors must be marked as well, to ensure all
options for achieving a subtask is included.
To facilitate the algorithms search for nodes to mark, it maintains a frontier of the most
relevant nodes, (see Definition 2). This frontier is used as a basis to search for new
potential nodes to mark, and to identify nodes that can be deactivated. The frontier is
updated when certain nodes are marked (see Section 3.3.1 for details).
Definition 2: Frontier
For a given graph G = (V, E), a frontier is a set F ãV of marked nodes, with each
node in F having no marked descendants in G.
The algorithm is split into two main parts. The first part is the Activate node function, that
is used to mark observed subtasks in the subtask graph, as well as searching the graph
for any additional nodes to mark. The second part of the algorithm is the Update graph
function, that deactivates nodes in the graph and identifies any newly desirable nodes to
mark that emerges from the deactivation. Update graph is called every timestep, while
Activate node is called on subtasks that are observed to be completed. The two functions
are described in detail in the following sections.

3.3.2
Update Graph
Update graph is called at the start of every timestep. The function is used to check the
graph for any potential updates that can be made without observations, whether it is nodes
that should be marked or nodes that can be deactivated.
When this function is called, it first checks if a goal node has already been previously
marked. In that case, no updates are made since once a goal node has been marked, that
goal is believed to be the one the human agent is working towards. Since all predecessors
of the goal node will also have been marked, all paths for reaching the goal has been
marked, so no further nodes will be relevant.
If no goal nodes have been marked, Update Graph will start by deactivating nodes from
the graph based on the current frontier. Using the sets of reachable nodes for the frontier
nodes, and the assumption that the human does not complete any subtasks not relevant
to their goal (see Section 3.6.1), the function identifies and removes any deactivatable
nodes present in the graph.
Definition 5: Deactivatable Nodes
Given the subtask graph G = (V, E), and a frontier F ãV .
A node n is deactivatable iff it is reachable for some nodes in F, but not all.
Paths containing deactivated nodes are not considered when searching for dominant
nodes. This approach to deactivating nodes is useful since the subtasks are partially
ordered, meaning there might be multiple paths to the same node. The goal of the deôÙ
activation is to remove as many nodes as possible from consideration, without excluding
any goals the human could have. This is to facilitate the algorithmãs aim of isolating and
marking subtasks that will be useful for the humanãs desired goal as early as possible.

Another way to resolve scenarios where humans cannot complete any desired subtasks,

When the specified amount of time passes, the robot will identify a subtask to query. If the
human answers YES, the subtask is activated, and if the human answers NO, the subtask
is deactivated.
When making a query, the robot will choose a subtask based on the nodes in the frontier.

reachable from the highest number of frontier nodes. These nodes will have the highest
likelihood of being needed, and if they are not needed, they eliminate the highest number
of paths. If only one node is in the frontier, the robot will query a random direct successor
of this frontier node. If the frontier is empty, no nodes have been previously marked, and
a random direct successor of the initial node is queried.

3.6
Compatibility Requirements
The subtask graph algorithm is compatible with a range of environments and can be couôÙ
pled together with a variety of planning algorithms. This section covers the limitations of
the algorithm, as well as requirements and restrictions for environments and planners,
that can be used with the algorithm.
3.6.1
Limitations of the Algorithm
The algorithm is designed to only work on Directed Acyclic Graphs (DAG). Given a subtask
graph that contains cycles, many features of the algorithm stops working. For example, it
would no longer make sense to mark predecessors of nodes, as cycles in the graph may
mark nodes that are also descendants. Additionally, the undesired behaviour of a node
being able to dominate itself is introduced. The intuition of the subtask graph breaks down
as well, as a subtask can be a part of the process of satisfying its own precondition.
In order to guarantee that a generated subtask graph is a DAG, the given environment
should contain no subtask that directly or indirectly contributes towards satisfying its own
precondition.
Another limitation for the algorithm is that any subtasks completed by a human must be
relevant for the final goal. For example, if the human picks up a tomato, then a tomato
must be used in the recipe. In essence, this means that the human may not change their
mind while cooking. This is caused by the current lack of a way to revert marking of nodes.
Doing this is a nonôÙtrivial task, as the marking of a node n can cause other nodes to be
marked or deactivated, and it is not immediately apparent what nodes have been affected
by n being marked. This is especially true if other observations have been made between
n being marked and reverted, which could also be affected by the marking of n.
Lastly, the algorithm can only pursue one goal at a time. This is due to the way the algoôÙ
rithm deactivates nodes based on the first observed actions, and the inability to reactivate
deactivated nodes. For this reason, the algorithm will not be able to help any further after
the first goal node has been marked. Furthermore, if the human tries to work on a second
recipe after taking actions towards a first recipe, the algorithm will think all the actions are
part of the same recipe and confuse the graph. This can potentially cause the algorithm
to be unable to identify any recipes correctly.
3.6.2
Environment
An initial restriction on the environment is that it should consist of multiple tasks, each
consisting of a number of subtasks. These tasks should have some overlap, meaning
that some subtasks are part of multiple tasks. Without any overlap, the intended goal will
be obvious after the first subtask is observed. This is trivially identified and does not utilize
the algorithmãs ability to identify relevant subtasks.
The environment must also be observable, meaning that the robot must be able to identify
whenever a subtask is completed by any agent. The algorithm works best when the
environment is fully observable, especially when working with decentralized agents. If the
multiple agents deactivate nodes in their graph based on the same observations, they will
always mark the same nodes. This allows them to work on the same assumptions of what
subtasks can be assigned, which significantly improves the agentsã ability to cooperate
on subtasks.

The environment should ideally not contain any disjoint groups of subtasks that connect
later in the graph. This might lead to problems with deactivating desired subtasks and
goals, as detailed in Section 3.7.4.
Additionally, all agents should ideally have the same set of subtasks in their graph, since
this ensures they are working on the same subtasks, which makes cooperating easier.
It is, however, not necessary for the individual agent to be capable of completing every
subtask, as long as all agents agree on what choices are possible.
Notably, the algorithm cannot adapt to directly adversarial agents, due to the marking of
nodes being irreversible. For example, if the human picks up a tomato, the robot only
considers recipes using tomato. If an adversary removes the tomato, the robot can not
adapt to this change and must be reset to proceed.
Finally, as a general note, the algorithm has been primarily designed and tested for disôÙ
crete time environments. However, continuous environments should work as well, as long
as a suitable way of calling Update Graph sufficiently often is found.
3.6.3
Planning algorithms
For a particular planner to be compatible with this algorithm, it must be able to plan for
individual subtasks, as opposed to planning towards the goal. This will enable the planner
to plan actions as soon as one subtask is marked. Additionally, the planner cannot be fully
offline. As relevant subtasks are identified based on observations, the planner must be
able to plan for subtasks as they are marked.
Since the intention of the algorithm is to identify subtasks relevant to achieve a humans
goal, it is preferable for the planner to be capable of cooperating with a human.
The algorithm works with both centralised and decentralised planners. Centralised planôÙ
ners simply maintain a single subtask graph, where a decentralised planner maintains
a graph for each agent. Decentralised planners benefit greatly from environments with
full observability. This is because the graph operations are fully deterministic, meaning if
each agent has a graph with the same subtasks and makes the same observations, they
will always mark the same subtasks.
Note that in order for the communication between a human and the agents to work, there
needs to be a way for the human and robot to interact. For decentralised systems, the
human should ideally be able to broadcast the message to all agents at once, instead
of having to inform each agent one by one, or risk having the agents develop different
markings.

The algorithm will be analysed separately for each of its two components: Activate node
and Update graph. However, as the two components interact with each other and manipôÙ

Activate Node
The first action of Activate node is to mark the node n along with its predecessors. To
locate and mark all predecessors of some node, a backwards BFS is used, using O(|V |+
|E|) time.
Next, dominator nodes are found for n. When finding dominators, all paths starting from
n and ending in one of the goal nodes are found. The paths are found using a DFS giving
a time complexity of O(|V | + |E|).
In order to bound the amount of paths given by this search, consider a maximally conôÙ
nected graph, with the exception of goal nodes having no outgoing edges. Consider the
paths between the initial node I and one of the goal nodes g. Since the graph is acyclic,
any subset of vertices including I and g and no other goal nodes, constitutes a unique
path between I and g. Given |G| goal nodes, the number of vertices when excluding the
initial node and goal nodes is |V | ã1 ã|G|. Using this we can find the number of possible
subsets containing I and g, which gives the number of paths between I and g bounded
by 2|V |ã1ã|G|. Doing this for all |G| goal nodes gives the total number of paths |P| possible
in a graph:
|P| = O(|G| ôñ 2|V |ã1ã|G|)
(3.4)

and the total intersection is found by repeatedly intersecting two sets, the intersection of

the descendants of n using a BFS, using O(|V | + |E|) time.

Update Graph
The first action of Update graph when called, is to deactivate nodes. When finding deôÙ
activatable nodes, the stored set of reachable nodes for each frontier node is used. The
number of sets of reachable nodes is bound by the size of the frontier |F| = O(|V |). To
find the deactivatable nodes, the intersection of these sets is found. Finding this intersecôÙ

intersection of the reachable node sets take O(|V |2) time.
To find the deactivatable nodes, each frontier nodeãs set of reachable nodes is searched
for nodes not present in the intersection. Checking the O(|V |) nodes of each set will take
at most O(|V |2) time.
Thus, the total time spent deactivating nodes in the graph is bound by O(|V |2 + |V |2) =
O(|V |2).
After the deactivation, the sets of reachable nodes for the frontier nodes are updated.
Updating a set of reachable nodes is done by simply recomputing them using a BFS on
the updated graph, which takes O(|V |+|E|) time. The time needed to do this for all O(|V |)
frontier nodes is bounded by O(|V |2 + |V | ôñ |E|).
Now the dominant nodes for each frontier node is found. Finding the dominant nodes is
done identically to how Activate node does it, and takes O(|V | ôñ |P|) time.
Activate Node is then called on any dominant nodes found through this, with each occurôÙ
rence of Activate node taking O(|V | ôñ |P|) time.
With this, the time Update graph uses per occurrence is bound by O(|V | ôñ |P|), and when
it calls Activate node, this also takes O(|V | ôñ |P|) time per occurrence. Update graph is
repeated every time changes occur, and it can possibly find multiple dominants for each
frontier node, each of which results in a call to Activate node. However, since Activate
node is called at most once per node, and it cannot be called on a deactivated node,
each node can only cause a change once. This results in Update graph and Activate
node maximally occurring a combined O(|V |) number of times.

3.7.3
Does Marked Nodes Always Facilitate the Humanãs Goal?
In order to determine whether a marked node will always be useful towards the humanãs
goal, the three contexts in which a node can be marked are considered:
ãÂ Observation of subtasks completed by a human: The simplest way nodes are
marked, is when the robot calls Activate Node on a subtask that it observes the
human complete, thus marking the node. Since it is assumed that the human only
takes actions that are in some way relevant for their intended goal (see Section
3.6.1), any subtasks the human completes will be relevant for the desired goal and
can safely be marked.
ãÂ Marking of predecessor nodes: Marking predecessor nodes for a node n in a
DAG is equivalent to marking every possible path from the initial node to n. Since
all the nodes marked by this process are on a path to n, they are relevant for n to be
executable. The graph is agnostic to which path ends up being executed to reach
n, but the completion of any subtasks on any of these paths will bring the state of
the environment closer to a point where n can be completed. Since n has been
marked, it is relevant to the goal of the human. As such, any node that furthers the
completion of n, will facilitate the humanãs goal.
ãÂ Marking of dominant nodes: To validate the marking of dominating nodes, conôÙ
sider an already marked node a that is dominated by the node b. By the definition
of a dominant node (Definition 3), any paths starting in a, regardless of what goal
they pursue, will include b. Since completing a will further the humanãs goal, and
every path that includes a also includes b, completing subtask b will also facilitate
the humanãs desired goal.
In conclusion, in every case where marking of nodes happens, the node being marked
will facilitate the humanãs desired goal.
3.7.4
Will Deactivating a Node Ever Hinder the Humanãs Goal?
When deactivating a node from the graph, it should not deactivate the humanãs desired
goal, or deactivate all paths to it. There is just one case where deactivation happens in the
graph, which is when there are multiple nodes in the frontier and their reachable sets differ.
In this case, any deactivatable nodes will be deactivated. Since the deactivated node is
reachable from at least one node deemed relevant to the humanãs goal, it might be part
of some paths leading to the humanãs goal. However, since the human only takes actions
relevant to their goal, all frontier nodeãs reachable sets will at least share the humanãs goal
node. This means the desired goal node will never be deactivated, but it is also important
that at least one path leading from the current frontier to the goal node is preserved, to
allow the algorithm to reach and mark the goal.

==PAGE:33==
If there is at least one common path to the goal that is reachable by all frontier nodes,
this path will not be deactivated. So as long as there is at least one frontier node that is
directly connected to this path, the desired goal cannot be made unreachable. Therefore,
the only scenario in which the desired goal node becomes unreachable, is if the frontier
nodes have no shared paths to the goal node. In this case, all the different paths will have
nodes not in the intersection of the frontier nodesã reachable sets, causing some nodes
from all the paths to be deactivated, and thereby making the desired goal unreachable.
This problem arises when the subtask graph contains subtasks that have multiple ingoing
edges, which are connected to distinct sections of the graph that have no edges between
each other (see Figure 3.13).
When the human starts working on the first of these edges, the frontier will be populated
with nodes from one of the sections in the graph. If the human then starts working on
another of the edges, before the node connecting the two sections has been marked,
a frontier node will appear in the second section of the graph. Since the frontier nodes
cannot share any reachable nodes, except the connecting node, any paths leading to the
connecting node, in both sections, will be deactivated. This will cause the connecting
node to be unreachable from any of the frontier nodes, and the graph will not be able
to mark it unless observed or communicated, even if it is the only node leading to the
humanãs desired goal.
Figure 3.13: A subtask graph representing a scenario that presently would cause the
graph to deactivate its path to the desired goal node (marked in bold). In this graph, the
goal node has multiple ingoing preconditions, and the sections of the graph that lead to
these preconditions have no connecting edges. The issue arises when a frontier node is
established somewhere along the way to satisfying precondition b, as well as a frontier
node along the way to satisfying either of the other preconditions a. In this case, the sets
of reachable nodes for these two frontier nodes would have no overlap, outside of the
goal node. Thus their paths to the goal node would be deactivated, making it impossible
to mark the desired goal node, unless it is observed or communicated.
These different edges coming from distinct unconnected sections in the graph represents
the only scenario in which deactivation can cause a desired goal to become unreachable
in the graph. Thus resulting in the robot being unable to assist in completing the goal.
Such a scenario does not arise in the Too Many Cooks environment, but it is discussed
further in Chapter 6, along with potential changes to the algorithm that can account for
these scenarios.

Implementation of the algorithm
The implementation is based on [Wu+20], who have made their code base open source
[WW20]. The code base is written entirely in Python, using the library PyGame to visualize
the environment. The original code base consists of two components: An environment,
and agents consisting of a navigation planner and a delegation planner. The environôÙ
ment is the Too Many Cooks environment, that contains the kitchen itself, with locations
of counters, cutting and delivery stations as well as the ingredients. It also holds all subôÙ
tasks, as well as the position of the agents, and what objects they might be carrying. The
navigation planner is responsible for finding the best actions for solving specific subtasks.
To do this, it uses the reinforcement learning algorithm BRTDP [MLG05]. The delegation
planner is responsible for identifiyng the most likely allocation of subtasks to agents. It
does this by using the navigation planner to evaluate how well the observed actions for
agents match the different subtasks. A Bayesian model then uses this to find the probaôÙ
bility of each subtask allocation. Further details of the environment and how the planners
work are given in Chapter 2.
Figure 3.14: Visualization of how the different components in the implementation are conôÙ
nected. Blue boxes have been modified, and green boxes are entirely new additions to
the code base.
The subtask graph algorithm presented in this chapter is implemented as its own separate
component. Our addition to the code base as well as the connection between the different
components of the code is outlined in Figure 3.14. The agent contains the two planners,
and it has access to a subtask graph as well as the environment. The agent takes the
subtasks from the environment, and keeps track of which ones have been completed. It
also uses these subtasks to create the graph, ensuring that all the different components
uses comparable versions of the subtasks. This makes it easier for the different compoôÙ
nents to interact with each other regarding the subtasks.
To utilize the graph algorithm, the delegation planner was modified to only assign agents
to subtasks that are marked in the graph. Additionally, Agent was modified to observe
all agentsã actions via the environment, to identify when a subtask is completed and then
activate them in the graph.
In addition to the agents using Bayesian Delegation (BD), there are other agent types in
the code base. The original code base also contained the agent types: Greedy, Uniform
Priors, Fixed Beliefs and Divide & Conquer. These all use the same navigation planner

as BD, but they have different delegation planners of varying complexity. On top of this,
we introduced a human agent type, which does not use any planners, but instead simply

When the graph algorithm is initialised, the graph is constructed using the subtasks passed

deactivate nodes from the graph while still maintaining a complete version of the graph for

nodes in the other. Alternatively, using a single graph requires customised search algoôÙ

deactivating nodes, NetworkX is used to find descendants, which establishes the sets of

the deactivation graph. When searching for dominant nodes for a node n, we go through
all the goal nodes and find all paths between n and the goal node in the deactivation

a simple implementation that covers all the features of the algorithm.

Related Work
In the field of humanôÙrobot collaboration and goal recognition, there is a large amount
of previous work relating to the subtask graph. There are several examples of methods
developed to make the collaboration between humans and robots using goal recognition
more efficient [KA+86] [Che+20] [Fre+19], with several attempting to start planning early
by using estimates of the goal or similar. Additionally, there are various works that use
a graph structure similar to that of the subtask graph, but utilises them differently, like
Graphplan [BF97] and [HS16].
In this chapter, some of the works that particularly influenced the creation of the subtask
graph, will be explored. One such work, is [LW14], that proposes the PIKE algorithm,
facilitating robots to collaborate with humans in heterogeneous teams, while accounting
for uncertainty. Another valuable inspiration is the work of [HS16], which addresses the
problem of collaborative robots needing special representations of tasks in order to learn
them. The paper proposes a solution to this in form of a graph that satisfies this, along
with an algorithm for creating this graph based on a simple representation of the tasks. As
part of this algorithm, the Conjugated Task Graph (CTG) is introduced, which is explored
as an influence on the subtask graph.
Below, the two mentioned papers are described and their influences on the subtask graph
is outlined.
4.1
PIKE
PIKE uses a graph known as a Temporal Planning Network with Uncertainty (TPNU), to
model possible choices in an environment where a collaborative robot and human have
different available actions. An example of such a TPNU, containing choices for making
breakfast is seen in Figure 4.1. In the example, the human is making breakfast, consistôÙ
ing of a drink and a food, and the difference between the humanãs and robotãs choices
is shown. For example, when the human chooses between Get mug and Get glass, the
robot chooses between Get grounds and Get juice. Note that all the actions in the graph
are marked with the time constraints for the task.
In addition to the TPNU, PIKE also uses Causal Links to model constraints in the enviôÙ
ronment. These are directed and connect actions in the TPNU, and represents a preconôÙ
dition. An example of a causal link is the Have Glass condition that connects from the
Get Glass action to the Pour Juice Glass. This means that the Pour Juice Glass action
can only be executed if the Get Glass action was chosen. PIKE shows that they can use
these causal links to exclude actions unwanted by the human, by pruning sets of choices
based on the humanãs choices. Through this, PIKE is able to isolate useful choices for
the robot.

Secondly, the TPNUãs in PIKE are forcibly segmented into human and robot choices, while

greater flexibility in how the human and robot can collaborate, as they are both able to
divide the subtasks between them, similar to what is done in PIKE, and cooperate on the
same subtasks.
Lastly, the TPNU has a strict order the actions can be taken in, where subtask graphs
has only a partial ordering that allows multiple paths to the same goal. For example, in
the breakfast example PIKE must always make the drink first, and then the food, where
subtask graphs will allow the food and drink to be made in any order.
One of the main features from PIKE that inspired the design of the subtask graph is the
causal links. The causal linkãs concept of connecting two actions with a precondition, that
can be used to identify useful actions based on previous choices, is used in the subtask
graph. The edges connecting a subtask that satisfies a condition with a subtask that
requires this condition replicates the concept of the causal links. The subtask graph uses
a concept similar to the causal linksã to deactivate unwanted subtasks and isolate useful
subtask the robot can execute, which is a significant aspect of the subtask graphs utility.
4.2
CTG
[HS16] looks at the problem of training collaborative robots, and more specifically the
fact that most methods in this field requires rich task models. The solution [HS16] proôÙ
poses is a method for converting a simple task graph into a special form of hierarchical
network called a Clique/Chain Hierarchical Task Network (CCôÙHTN), that is then used to
efficiently train robots. The task graph models the possible environmental states in the
nodes, and the actions connecting these states on the edges. In order to convert this to a
CCôÙHTN, it is first converted to a middle state called a Conjugate Task Graph (CTG). In a
CTG, the nodes model the possible actions in the environment, and the edges represents
environmental prerequisites for these actions. These CTGãs have several concepts that
influenced the design of the subtask graph. An example of a CTG is shown in Figure 4.2.

There are a few key differences between CTGãs and subtask graphs, most of them reôÙ
lating to the edges. Even though the edges in both graph types represent conditions or
prerequisites necessary for an action/subtask to be executed, they have some important
distinctions. In CTGãs, the prerequisites on the edges propagate through the graph reôÙ
sulting in the later edges having a long list of prerequisites on them, while subtask graph
edges only contains a single condition.
Furthermore, the edges in CTGãs have prerequisites that are not necessarily satisfied by
the preceding node. This gives a very different way of representing multiple paths to the
same node. In CTGãs, each ingoing edge to a node represents a possible path to that
node, and all the prerequisites on the edge must be satisfied before the edge can be travôÙ
eled, but only one edge needs to be travelled for the action to be executable. In subtask
graphs, the different ingoing edges to a subtask can both represent different paths to it
as well as different preconditions. For a subtask to be executable, one edge for each
different precondition must be travelled.
These differences in the edges are indicative of the different purposes of the graphs.
CTGãs are focused on the different paths through the tasks to create cliques and chains
that represent the relation between the actions well. Meanwhile, the subtask graph is foôÙ
cused on the paths that open up whenever a subtask is completed, to identify new relevant
subtasks.
In contrast to the edges, the nodes are fairly similar between CTGãs and subtask graphs.
Both simply model the possible actions/subtasks, and both graph types include a special
initial/origin node as a root of the graph. The only difference in the graphs is that CTGãs
only have a single termination node acting as the universal goal node of the graph, while
subtask graphs contain multiple goal nodes. This difference is again indicative of the
different purposes of the graphs. CTGãs show the possible paths through the task to
model the relation of the actions, and therefore only need a single end node. The subtask
graph tries to identify desired paths and goals, and therefore need to represent different
goals separately.
Despite the differing purposes of the two graph types, the CTGãs has had a strong influôÙ
ence on the structural design of the subtask graphs.

Discussion
In this thesis, we introduced the subtask graph as a tool to guide agents in a collaborative
environment. The subtask graph allows a robot to identify individual subtasks relevant
to a human, based on observations, without knowing the humanãs goal. By combining
the subtask graph with a planning algorithm, we have shown a robot to be able to assist
a human based on a small amount of observations, in various scenarios in a kitchen
environment. In addition to assisting humans, the graph can also be used to collaborate
with other types of robots.
The tests made in Chapter 5 show promising results for the subtask graph, despite not
having any quantifiable datapoints relevant for comparison with existing methods. Similar
works in goal recognition are often measured on parameters like time until goal is idenôÙ
tified, speed of training some model (or the convergence thereof), and accuracy of the
recognizer. However, these are not particularly relevant to the subtask graph algorithm.
The time it takes for the graph to identify the goal, is not a telling measurement for the
algorithms effectiveness, since the crux of the algorithm is the identification of individual
subtasks. Thus, the metric that will more accurately show the algorithmãs performance is
the time it takes until a relevant subtask is marked. The problem with this metric is that it
is very specific to the situation, and is very dependant on the size of the graph, and what
actions the human takes. This means that it is hard to make any conclusions based on this
metric, as knowing what numbers to expect in different scenarios is difficult. Furthermore,
as there are, to the best of our knowledge, no simple algorithms for selecting subtasks to
work on, without knowing the goal, it is difficult to get a good baseline to compare to, withôÙ
out implementing other similar algorithms, which can be very time consuming. This leads
to a more qualitative test of the algorithmãs effectiveness, involving a subjective evaluation
of how the algorithm marks and deactivates subtasks based on actions, as conducted in
Chapter 5.
Throughout the testing of the algorithm, the time taken for updating the graph was not
focused on. The reason for this is that the running time was for the most part negligible,
compared to the time spent by the planner between time steps. Furthermore, the speed
or realôÙtime execution of the algorithm is not a major focus of this thesis.
However, looking at the analysis in 3.7, we see that the asymptotic time complexity of the
algorithm is exponential. This would suggest that when working with very large graphs, the
time spent by the algorithm may increase enough to become nonôÙnegligible. The primary
contributor to the running time of the algorithm being exponential, is the identification of
dominant nodes. The running time of the dominant node search is based on having the
worst case scenario for every procedure in the search. These scenarios are not very likely
to occur in the first place, but some of them also contradict each other. For example, the
worst case for the size of the frontier, has the graph being minimally connected, while the
number of paths to compare to find dominant nodes, has its worst case in a maximally
connected graph. This results in the graphs that the algorithm is expected to encounter,
often having a better runôÙtime than the asymptotic timeôÙcomplexity indicates.
In regards to the graphs the algorithm may encounter, the analysis also revealed a shortôÙ
coming in the algorithm that makes it unable to cope with graphs containing certain properôÙ
ties. Namely, graphs containing subtasks with preconditions satisfied by multiple different
paths that only share the one connecting subtask. This occurs when an environment conôÙ
tains different groups of subtasks that are internally connected, but only connect between

groups at a few points later in the graph (see Figure 3.13). These types of different groups
do not appear in the kitchen environment we use, as all subtasks are connected to making
some form of salad, and ingredients can be mixed in different orders. Imagine however,
that we introduce dressings to the environment. Dressings would contain different ingreôÙ
dients than the salads, and there would be different orders of mixing them. The salad
subtasks and the dressing subtasks would not be connected until both the dressing and
salad are done individually, at which point there would be connecting subtasks combining
them in different ways.
The problem the algorithm has with this type of environment is that if a human is to pick up
one ingredient for a salad and one ingredient for a dressing, the algorithm will deactivate all
the subtasks those ingredients lead to except the subtasks that combine their completed
salads and dressings. This would mean that all the subtasks necessary for completing
the separate salad and dressing will be deactivated, and the algorithm can therefore not
activate them to allow a planner to help the human. This shortcoming creates a limitaôÙ
tion in the amount of environments the algorithm can consistently work on, since it is only
consistently successful on graphs containing a single group of subtasks, like the salads
in the Too Many Cooks environment, or when the different groups of subtasks are not
connected at all. Due to this limitation, it would be worth addressing this shortcoming in
the future, to improve the generality of the algorithm. We propose a possible solution to
this in section 6.1.
6.1
Future Work
There are three primary points of improvement that can be considered as future work for
the algorithm.
The first is to expand the analysis of the algorithm to include a formal analysis of the main
properties of the algorithm. This thesis gives an informal argument for some of these,
but to properly validate the algorithm going forward, a formal analysis, of especially the
correctness and completeness of the algorithm, is necessary.
Another point of improvement, is to allow the graph to also handle the previously discussed
scenario, involving multiple subtask groups, that it is unable to consistently solve. As
mentioned, the problematic scenario in the graph arises when two wholly separate groups
of subtasks meets later in the graph. Typically, a group of subtasks represents a series of
items that can be combined in multiple orders, resulting in various different paths starting
and ending in the same nodes.
When multiple of these groups contain frontier nodes, the deactivation will remove the
paths to the subtasks that links the groups. A possible solution to this, is to avoid deactiôÙ
vating nodes based on frontier nodes in other groups of subtasks. In practice, this would
be done by preprocessing the graph, establishing these groups of subtasks. When nodes
are deactivated, instead of comparing the sets of reachable nodes for all frontier nodes
together, the frontier nodes in each group are compared separately. Doing this should
avoid deactivating necessary subtasks, and allow the subtask graph to consistently solve
and assist in environments with multiple subtask groups.
The third area to work on, is the improvement of the algorithms running time for larger
graphs. The goal of the algorithm is to use the graph to effectively assist a human, and
the time spent updating the graph should therefore not be inhibitively long. A robot, espeôÙ
cially in a kitchen environment, might know a large amount of recipes. This could result
in a massive graph which may slow down response times for the robot, especially when
looking for dominant nodes. The subtask graph in its current state is by no means optiôÙ
mized for graphs containing thousands of subtasks. Since the identification of dominant

agent, it will create a text prompt in the header of the PyGame window asking for an action
to be inputted for a specified agent. When the prompt is shown, a movement action can
be inputted by pressing an arrow key. If any key other than the arrows is inputted, the
agent will do a noOp that timestep. If an invalid action is inputted (walking into a wall or
into another agent etc.), the prompt will simply remain until a valid action is inputted.
When running the program using the --play flag, the input for the agents changes slightly.
Actions for the agents are still given using the arrow keys, however the program will now
constantly take inputs. When starting the program, inputted actions will always control
agent 1, but the agent being controlled can be changed by using the number keys (press
ã1ã to control agent 1, ã2ã for agent 2 etc.).
Note that the program creates logs for the state of the subtask graph through the run,
saved in the ./gym-cooking/misc/logs folder. However, the logs will only be saved if
the program terminates on its own by completing a recipe, or by closing the PyGame
window, which can not be done while an agent is planning an action. If running with the
--play flag, only closing the PyGame window will save the log file.
A.4
Level Creation
To create your own level for testing, simply create a new .txt file and place it in
./gym-cooking/utils/levels.
A correctly defined level file consists of three parts: The level specification, a recipe name,
and a lists of locations for up to 4 agents.
Each of these three parts are separated with a blank line.
