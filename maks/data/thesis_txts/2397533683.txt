Summary
Refactoring is an important tool for programmers, it provides them with the tools to
handle the different types of errors (code smells) in their code. It is therefore a key
factor that the standard ways of refactoring these problems are easy to understand
and that they really improve the readability of the code.
Previous studies have proven what they best type setting is for reading code, where
programmers pays attention to when reading code, when its best for educational
purpose to read or write code. But none of them have proven that refactoring is
really useful, this gap in the field of study is what this thesis was attempting to
fill. This thesis is using eye tracking to capture the mental load of the participants
during an experiment with them reading 10 pieces of code and afterwards rating their
experience using the self-assessment manikin for valence and arousal.
The results of the 14 eye tracking experiments showed a significant difference in the
personal mental effort ratings and in the manikin valence ratings which confirms
that the participants liked the refactored code more then the unrefactored. Sadly
though thats the only areas where the results were significant, all of the eye tracking
measurements used in the experiment came short on the results. All the results had
the correct trend in the data, but they all seemed to lack more data as in not enough
participants.

CHAPTER 1
Introduction
Does refactoring really improve one’s ability to learn and comprehend source code?
The study in this thesis is aiming to understand the impact refactoring have on source
code readability. Java was chosen as the test language since it a classic and widely
used programming language, and since the experiment is done at the Technical Uni-
versity of Denmark (DTU) where all software students go through at least a few
courses using Java during their Bachelor, it was the obvious choice.
The experiment is using a Tobii TX300 eye tracker to record the participants eye
movements as they are reading the code presented to them on a computer monitor.
The results are based on 6 dependent variables where 3 of them are measured through
the use of eye tracking, fixation count which is the amount of fixations, fixation du-
ration which is the time in seconds spent fixated and visit duration which is the time
spent looking, fixated or not. Also used in the analysis are 3 ratings filled out by each
participant after each of the code examples they read during the experiment, the rat-
ings are mental effort which is a personal rating of difficulty, manikin valence which is
a personal rating of pleasure and manikin arousal which is a personal rating of arousal
For each of these 10 code examples there exist 2 different versions one which is refac-
tored with the type of refactoring called Extract Method, and then the other type
which is the unrefactored.
All the unrefactored code examples are suffering from
code smells related to ”Bloaters” and ”Dispensables”, the specific type of code smells
used in the examples is ”Long Method” and ”Duplicate Code”.
The analysis was created on the seven different hypotheses, where each of them was
based on the different measured dependent variables, each of them was tested and
their results are reported. The results are all there but most of them have failed to
prove significance, and this might be because of the small amount of participants, as
I only managed to recruit 14 participants where one of them had to be discarded.

Related Work
None of the previous studies mentioned here is doing the same experiment as I am
in this thesis, but by using their results as a foundation to my thesis, I have already
confirmed the foundation of my experiment and I can focus on expanding the research
area.
The study conducted by Busjahn and Schulte[1] was able to find the importance
of code reading and comprehension in teaching programming. Their results is based
on interviews of instructors to determine the importance of code reading into five cat-
egories; conceptualisation, occurrences, and effects of successful code reading, chal-
lenges for learners, as well as approaches to facilitate code reading.
The results
show that code reading is connected to comprehending programs and algorithms.
These papers presents a family of studies on the impact of identifier style (i.e.
camelCase or under_score) on source code reading and comprehension using an eye
tracker [2, 3] the results provide a better understanding of the fundamentals of read-
ing and comprehending source code, it proves a difference on reading source code
compared to regular text, it also proves that reading with different text styles yields
different results, and the best style for comprehension is camelCase. Sharif et al.[4]
extended the research with another eye tracking experiment which sat out to deter-
mine if gender impacts the effort, time and ability to recall identifiers, and generally
found no significant results, some of their more narrow hypotheses came with signifi-
cant results but they only had 9 participants which is too few to call a final conclusion.
Turner et al. [5] conducted a study with an experiment to compare source code
between C++ and Python to assess the effect of programming language on student
comprehension. They found no statistical difference between C++ and Python with
respect to accuracy and time, but they were able to prove a significant difference
between C++ and Python for the fixation rate on buggy lines of code in finding bug
tasks.
Jbara et al.[6] uses eye tracking to analyse how programmers read regular code,
the experiment arrives at the conclusion that programmers pay more attention to
the first part of the code when reading, while the latter lines may only be skimmed.
They realise there is multiple reading patterns within each code fragment and finds
a significant variability across their subjects.

3.1
Refactoring
”Refactoring is the art of improving the design of existing code. Refac-
toring provides us with ways to recognise problematic code and gives us
recipes for improving it.”Refactoring Workbook - William C. Wake [7]
Refactoring is improving the design of existing code, which means:
• Refactoring does not include all changes to a system. Changes which are consid-
ered design improvements or add new functionalities are not all considered to
be refactoring. However refactoring can create new code just not new features.
• Refactoring is not to rewrite from scratch. Instead it’s making it possible to
improve code without rewriting it which benefits from preserving the knowledge
already embedded into existing code.
• Refactoring shifts the balance point between up-front design and emergent de-
sign. Up-front design is where the code is designed before the implementation;
emergent design is designed during the implementation. The trade-off between
up-front and emergent design depends on our ability to anticipate problems or
assess them in code, and whether it’s easier to design and then translate to code
or to code and then improve the code. Refactoring lowers the cost and risk of
emergent approach.
• Refactorings can be small or large. Most refactorings are small and they should
be applied anywhere in such cases, meanwhile where the large refactorings are
rarely needed, it’s about refactoring as we go, to keep the program in a run able
state at all times.
The process of refactoring involves the removal of duplicates, the simplification of
complex logic, and the clarification of unclear code[8]. To ensure safety when refac-
toring it’s mandatory to either manually test that the new changes didn’t break the
code or run automated tests. If the automated tests are available programmers seem
more willing to refactor because there is no struggle to prove the refactored code is
running better.

Bloaters is a type of code smell which are, methods and classes which have increased

do not appear right away, rather they accumulate over time as the program grows in
size.
Object-Orientation Abusers is this type of code smell where the code is an incom-
plete or incorrect implementation of an application in an object-oriented programming
language which violates the principles of object-oriented programming.
Change Preventers smells means that if you need to change something in one place
in your code, you have to make many changes in other places too. Program develop-
ment becomes much more complicated and expensive as a result.

Dispensables is a type of code smell which is pointless and unneeded which absence

3.1.2
The refactoring cycle
Refactoring can be seen as an cycle which should be repeated until there are no more
code smells.
1. Choose the worst problem / worst code smell
2. Select the refactoring method which will address the specif problem.
3. Apply the chosen refactoring method
Each cycle selects an area which smells and improves this area while retaining
the same functional behaviours.
Refactoring is often done in small steps to help
prevent the introduction of defects, most refactorings only take seconds or minutes to
complete. Some larger ones can take days, weeks or even months but they are always
done in smaller steps.

3.1 Refactoring
7
Composing methods Refactoring is mostly devoted to correctly composing meth-
ods. In most cases, excessively long methods are the root of all evil. The majority
of the code inside these methods conceal the execution logic and make the method
extremely hard to understand – and even harder to change.
The refactoring techniques in this group streamline methods, remove code dupli-
cation, and pave the way for future improvements.
Moving Features between Objects Even if the code has distributed functionality
among different classes in a less-than-perfect way, there is still hope.
These refactoring techniques show how to safely move functionality between classes,
create new classes, and hide implementation details from public access.
Organising Data These refactoring techniques help with data handling, replacing
primitives with rich class functionality. Another important result is untangling of
class associations, which makes classes more portable and reusable.
Simplifying Conditional Expressions Conditionals tend to get more and more
complicated in their logic over time, and there are yet more techniques to combat
this as well.
Simplifying Method Calls These techniques make method calls simpler and easier
to understand. This, in turn, simplifies the interfaces for interaction between classes.
Dealing with Generalisation Abstraction has its own group of refactoring tech-
niques, primarily associated with moving functionality along the class inheritance
hierarchy, creating new classes and interfaces, and replacing inheritance with delega-
tion and vice versa. [9]
In this thesis the types of code smells dealt with is ”Long Method” and ”Duplicate
Code”, they are from the two groups ”Bloaters” and ”Dispensables”, they have been
selected because bloaters are always impacting the readability of the code, because
they are unnecessarily big, and easy to refactor if you know the purpose of the program.
Dispensables are almost the same, because they fill up the code with either something
you don’t need to read or with something which is almost identical to another part
of the program, with the only changes sometimes being variable names.
The way both of these code smells are dealt with is with the refactoring method
from the type ”Composing methods”, the exact method used is ”Extract Method”.
”Extract Method” is used because this method is easy to use and have a high impact
on source code readability, by completely changing the layout of the code, into a less
confusing and more intuitive code style.

3.2
Eye Tracking and Cognitive Load
In this section the different classifications of eye events and key metrics will be de-
scribed, as well as their correlation to cognitive load.
Eye tracking is a methodology which helps researchers understand visual attention.
With eye tracking it’s possible to detect exactly where users look, how long they look
at specific point and the path their eyes follow from point A to point B. Eye tracking
helps researchers understand complicated user behaviour, even that which the users
cannot describe themselves.
3.2.1
Fixations
Even though the eye is constantly in motion, sometimes there is a pause in the move-
ment on a specific area of the visual field, these pauses are called fixations. A fixation
is a stabilised focus and it can be used as an indicator of where a person is directing
his attention to. Fixations are analysed in two forms:
Fixation count
Is the number describing the amount of times the person paused eye movement and
fixated on a specific point on the screen. The fixation count can be used as an indi-
cator of the cognitive load, more fixations equals higher load[10].
Fixation duration
The fixation duration is the time each fixation lasts fore, it too can be used as an
indicator of the level of cognitive load, the higher the duration indicates a higher
cognitive load[10].
When using eye tracking to analyse cognitive load its different whether a diffi-
culty task will increase fixation count / fixation duration or decrease the fixation
count / fixation duration, it all depends on the components of the task. As stated
by Päivi Majaranta et al. in Gaze Interaction and Applications of Eye Tracking[11]
your cognitive load and eyes behave differently e.x. while driving your fixation du-
ration should decrease but your fixation count should increase, this is because you
need to react fast, another case like a reading task which increases your fixation dura-
tion but not your fixation count, because you can take your time to look at each word.
Later on the fixation count / fixation duration is divided into total and relevant
based on the AOI’s, they are measured to assess the cognitive load on a scale where
lower is a better score.
The reason why fixations are so important as a measurement is because of the eye-
mind assumption which was created in 1980 but still remains relevant which says
”that the eye remains fixated on a word as long as the word is being processed.”[12],

3.2.2
Visit Duration
The variable called visit duration is an output which is available from Tobii Studio, it
is based on the time the participant have spent looking at the area in question. The
total visit duration is an AOI covering the entire code area, it contains the time spent
looking anywhere in the code, this is different than total fixation duration because
visit duration is counting the whole time not just the time where the participant is
fixated.
3.2.3
Areas Of Interest (AOI)
AOI refers to Area Of Interest, such as a rectangular ad region on a web page, which
helps researchers analyse the various components of a visual scene. These AOI’s can
take any geometric shape according to the various components on the screen. For this
experiment the same AOI’s are in use across all participants. Later on the combined
AOI’s of code related to the questions is referenced as the relevant AOI, and an AOI
covering the entire code section is referenced as the total AOI.

Self-assessment

relation to the previous completed tasks.

Mental Effort

referrers to the question which the participant answered just before which was a
question related to the code they just read. This variable is used to compare the
rated difficulty of the refactored versus the unrefactored.

4.1.1.2
Refactoring - Code Smell
At first the experiment was designed to prove refactoring in general was beneficial,
but during the construction of the code samples it was decided that for this kind
of experiment it would require too much time from each participant to test enough
examples for a statistical difference on each type of refactoring style, so only one
type of refactoring was selected it was of the type ”Composing methods” because long
methods are the root of all evil and the most visually changing refactoring style within
this type is ”Extract Method”, therefore it was selected as the refactoring method used
in all examples.
The code smells dealt with in the 10 code snippets are of several different types
from the two groups ”Bloaters” and ”Dispensables”, the specific type of code smell
the examples are dealing with is ”Long Method” and ”Duplicate Code”.
Long Method this type of code problems happen when a programmer keeps adding
more functions to the same method without removing the old. It happens because
it’s often harder to create a new method than it is to add to an existing one. But even
if the new function is just one line its better to write it in it’s own method to avoid
creating a tangled mess of functions within one method. The benefit of doing this
type of refactoring is that the code becomes easier to read, understand and maintain.
Duplicate Code often happens when multiple programmers are working on different
parts of the same program at the same time or smaller duplication’s might happen
when certain parts of the code looks different but actually performs the same task.
It’s always an improvement to merge duplicate code because it simplifies the structure
of the code and makes it shorter, which greatly improves the readability and ease of
use.

Research Question (RQ):
Does refactoring have an impact on source code readability?
Hypothesis (H1):
When comparing the refactored versus unrefactored will the refactored versions have
an overall lower score in perceived effort?
(Based on questionnaires)
H1 is based on a personal rating from each participant it’s not directly used to
confirm the research question, but it can be used to decide whether the participants
finds it easier to read refactored code over unrefactored, the result is expected to be
a lower score for the refactored.
Hypothesis (H2):
Is there a significant difference in the amount of required time (speed) between refac-
tored and unrefactored source code to answer the questions?
(Based on time / visit duration)

Hypothesis (H3):
Is there a significant difference in the average visual effort between refactored and

Experimental procedure
This section is explaining the process in which the experiment was executed.
4.4.1
Procedure
This is the procedure all the experiments followed:
1. Greet welcome.
2. Hand participant the informed consent form, wait for them to read and sign it.
3. Guide the participant through a slide show which explains what they’ll see
during the experiment
4. Calibrate the eye tracker to the participant.
5. Start the experiment. Participants read a piece of code presented on the monitor
while their eye movements are tracked.
6. The participants advance with ”space bar” when finished reading.
7. The participants answer a question related to what the code does.
8. The participants answer the 3 ratings.
9. This process, item 5-7 repeats 10 times, one for each code snippet.
10. Round off the experiment with a post survey gathering information about the
participant
11. If requested by the participant explain anything related to the experiment.

CHAPTER 6
Discussion
The work done prior to the actual experiment is where the majority of the work hours
of this thesis has been spent, the preparation of the materials used in the experiments
and the process of setting up the actual experiment along with the sequence used in
the Tobii Eye tracker, took much longer than anticipated. The first attempt at the
analysis was flawed where each hypothesis extracted and analysed the data needed
to test them individually. The second attempt extracted every piece of data into
one combined table with a row for every model for each participant, the full table
is located in appendix H, having this table made the analysis was much easier and
minimised the chances of errors happening during the data processing, since the data
was now only processed once.
With this new full data table the results came faster but remained the same and
from them it is clear that the first and the sixth hypothesis H1 and H6 indicated that
the refactored version is the preferred style to read at least in terms of difficulty, this
is also confirmed by hypothesis H5 which found a strong correlation between Mental
Effort tested in hypothesis H1 and manikin valence tested in hypothesis H6. But then
again the second H2, third H3, fourth H4 and seventh H7 hypothesis which were all
rejected, couldn’t be used to prove that refactored was better than unrefactored, if
more than the 13 analysed participants had participated the results might have been
able to achieve significance and confirm the 4 rejected hypothesis, which could have
changed the results to confirm that the refactored code is better than the unrefactored.
Even though many of the participants have years of experience with Java program-
ming or other object oriented programming languages, several of them admitted that
they peaked in Java some time ago, so they might have the experience but without
maintains it’s like they never had it.
The AOI’s were created by me, with no prior experience on creating AOI’s there-
fore they might not be giving the full picture and could contribute to the lack of
significant results. All AOI’s were created as rectangles and it might have given an
overall better result if they were all individually shaped to follow the length of each
line within the AOI, to avoid the possible fixations on a blank spot, which the partic-
ipant might make when simply thinking about the code and not fixating on a specific
area.
==PAGE:47==
6.1 Future work
38
6.1
Future work
The work done in this thesis is a big step in confirming the usefulness of refactoring,
however there are still a numerous amount of ways to improve upon the work done in
this thesis. The next step which should already have been taken is to get around 50
participants, which would ensure that every piece of code would have 25 participants
and the results would probably have been able to achieve significance. With a larger
pool of participants it would also have been possible to divide them into groups
based on their experience, like less than 3 years of experience could be novice and
more than 3 years are non-novices, this sort of group creation could have given results
which might indicate that the non-novices were equally fast at reading refactored as
unrefactored but the novices were much faster at reading refactored compared to
unrefactored.
For further expansion it would be easy to change the pieces of code to test several
different types of code smells fixed with different types of refactoring, and thereby
testing whether different types of refactoring is better than other.
The last study referred to in chapter 2 by Jbara et al.[6], which was included as
an additional research possibility, their experiment proved that programmers pays
more attention to the first part of the code they are reading, this could have been
confirmed with some additional AOI’s dividing the code into sections and comparing
the fixation count of each section. This study would then have been able to expand
on the work Jbara et al. did, by applying the same comparison as they used, to the
comparison of refactored code versus unrefactored, which might prove that refactored
code is skipped like proved by Jbara et al. but with unrefactored code it might be
necessary to read every line to make sense of the program.
A possible way to improve the results is to improve the preparation of the statis-
tical data, now the only data processing before the analysis is done is the removal of
the invalid code ”I” and the one invalid participant, this could have been expanded
with the removal of potential outliers in the combined data file, through the many
visual inspections done on each hypothesis in the result section there seems to be at
least a few outliers, which could potentially have shifted some of the hypothesis in
the bad direction.
Further expansion could have been a thorough investigation of the eye tracking
data, because the data is already there it just needs to be analysed and interpreted.
Within the eye tracking data there lies numerous other possibilities like the possibility
to look into the eye tracking measurement called saccade, which is a quick movement
of both eyes from one fixation to another. If saccades was added as an additional
measurement it could be used to expand the investigation of hypothesis H3 focusing
on the visual effort.
==PAGE:48==
CHAPTER 7
Conclusion
The research completed in this thesis is accepting hypothesis H1, H5 and H6 which
can be used to conclude that the participants had an easier time reading the refactored
code compared to the unrefactored, this is the only conclusion because these confirmed
hypotheses are all based on the personal individual ratings and not the measured eye
tracking. Then there are the results from hypothesis H2, H3, H4 and H7 which were
all rejected, which confirms that this thesis is not a clear result saying that refactored
code is better for code comprehension, but neither is it concluding the opposite.
This thesis is simply laying the groundwork for future research by digging into a
research area with room for future exploration, many studies like the ones mentioned
in chapter 2 are studying a very related research area but no other research is actually
using eye tracking to compare refactored versus unrefactored code, so if this research
had arrived at a powerful statement about the research question it could potentially
have filled that gap in the research area.
However with the lack of participants the final result is inconclusive and as men-
tioned in section 6.1 there are still plenty of ways for new research to expand upon
the work completed in this thesis.

Title of project:  
Using eye tracking to investigate the impact of refactoring on source code readability. 
Principal Investigator:

Project Summary: 
This project aims at investigating the impact refactoring has on source code readability when reading Java 
code. The code will be refactored using the refactoring method called “extract method” the two states of 
refactoring (refactored, unrefactored) will be used to ask questions and your eye movements will be 
monitored during the elaboration of proper answers. Data collected will be used to understand when 
refactoring is more suited to the scenario under consideration. 
Procedure: 
You will first be asked to answer few questions in order to understand if you are eligible to take part to the 
test. Then you will be instructed with an example of the provided questions. After that, you will be asked to 
take part to the calibration procedure of the eye-tracker and then the actual questions will be asked. Your eye 
movements will be tracked and recorded during the time you use to read the Java code and when you are 
answering the questions. After that, you will be asked to “reason” aloud regarding the provided answers. This 
last step will be guided. 
Audio/Video Recording

the experiment. Therefore, an audio and/or video recording device may be used to record your answers. Your
