Abstract
In this thesis, Security Protocol Choreography (SPC), a new choreographic security pro­
tocol specification based on Alice and Bob (AnB)­notation which uses Performing Secu­
rity Proofs of Stateful Protocols (PSPSP) to prove security, is introduced. AnB­notation
is a popular way of describing security protocols, and formal languages based on AnB­
notation, such as AnB, allow for connection with protocol verification tools. However, this
notation is limited: it cannot express branching, repetition, or long­term state. Choreogra­
phies (or Multi­Party Session Types) are a way to describe more complex interactions
while maintaining much of the brevity and simplicity of AnB­notation. The task of this the­
sis is to connect choreographies to a verification tool: PSPSP. This tool is embedded into
Isabelle/HOL and thus obtains verification results with very high reliability. It is based on
describing a protocol at an “atomic” level, i.e., transactions are executed as a single tran­
sition and cannot be interrupted by other transactions. PSPSP allows for long­term state
and arbitrary control flows so that the actions of a given choreography can be expressed
in PSPSP. Moreover, PSPSP does not need a bound on the number of sessions required
by other tools like Open­Source Fixed­Point Model­Checker (OFMC). On the other hand,
PSPSP uses an abstract interpretation technique that can lead to false positives (attacks
that only work in the abstraction). Thus, this thesis aims to conceptually design a trans­
lator from choreographies to PSPSP transactions, implement it, and test it on concrete
examples. Besides the practical aspects, the theoretical value of this work is a declarative
semantics to choreographies based on higher­order logic/PSPSP. Lastly, The Envelope
Protocol is modelled in both SPC and PSPSP as a motivation example for the clarity of
SPC.

Introduction
All digital systems use communication protocols; some are for application­level logic, and
others are for securing communication. Foreseeing all types of attacks when designing
new security protocols is challenging, especially since they are often exposed to a mali­
cious third party that, with unlimited resources, can control the communication channel.
Automated verification can be used to overcome the challenge of security protocol verifica­
tion. However, it comes with other challenges, such as how to model intruder deductions,
bounded number of sessions or agents, equivalences and theorem proving, which makes
it an active research area. [1]
AnB­notation is a popular way of describing security protocol, which can be used for for­
mal verification, though it cannot express branching, repetition or long­term state. Chore­
ographies (or Multi­Party Session Types) are a way to describe more complex interactions
while maintaining much of the brevity and simplicity of AnB­notation.
PSPSP [2] is a security protocol verification tool that can verify protocols across an un­
bounded number of sessions with a long­term state allowing for repetition by limiting the
number of agents. This tool is embedded in Isabelle/HOL [3] and thus obtains verification
results with very high reliability. It is based on describing a protocol at an “atomic” level,
i.e., transactions are executed as a single transition and cannot be interrupted by other
transactions. On the other hand, PSPSP uses an abstract interpretation technique that
can lead to false positives (attacks that only work in the abstraction).
In this thesis, SPC, a new choreographic security protocol specification based on AnB­
notation which uses PSPSP to prove security, is introduced. SPC aims to combine the
clarity of AnB­notation with expressiveness of PSPSP.
The work on SPC primarily builds upon a conceptual paper “Security Protocols in Chore­
opgraphies” [4] with formal verification in PSPSP [2]. Furthermore, The Envelope Protocol
is modelled in PSPSP and later shown in SPC to highlight the clarity of SPC.
In chapter 2, the background of choreographies, formal modelling of security protocols in
AnB­notation and PSPSP is introduced alongside the envelope protocol, which is mod­
elled in PSPSP in chapter 3. The protocol flow in PSPSP is highlighted in chapter 4 by
refining a reference Needham–Schroeder Public­Key Protocol (NSPK) model in PSPSP.
SPC is formally introduced in chapter 5 with the translation to PSPSP in chapter 6 where
chapter 7 shows how SPC can be used to describe the envelope protocol and Needham–
Schroeder–Lowe protocol (NSL). This thesis is wrapped up with related work in chapter 8,
future work in chapter 9 and finally, conclusions in chapter 10.

the general concepts of choreographies are put into the context of security protocol anal­

2.2
End-point Projection
Describing a system or a protocol as a choreography gives an overview of the whole
system and how everyone interacts with each other. However, a choreography does not
directly define end­point behaviour. Ideally, one would use the choreography to derive
the end­point behaviour of each peer. This process is called End­Point Projection (EPP).
As choreographies can describe almost anything, making a consistent EPP is not always
possible. WS­CDL address this issue with well­formedness requirements, which in broad
terms can be summarised as “casually must hold”. [5]
A short version of those requirements is that a peer cannot respond to a message they did
not receive (connectedness), all executions must follow a global order (well­threadedness),
and lastly, the protocol flow must be complete and unambiguous within a single session,
i.e., no deadlocks or race conditions (coherence).
Carbone [5] showed that if those conditions are fulfilled, EPP is possible and preserves the
meaning of the original protocol. Moreover, the link between EPP and the choreography
allows for a broad and conceptual description of a whole distributed system/protocol while
keeping a direct link to a concrete implementation for each peer.

2.3.1
General assumptions and abstractions
Like WS­CDL, AnB­notation is a high­level specification that abstracts away from several
implementation­specific details:
• Instead of depending on concrete cryptographic implementations, AnB­notation uses
the Dolev­Yao model for the algebraic properties of symmetric and asymmetric cryp­
tography, which means that in AnB­notation, symmetric and asymmetric encryption
cannot be decrypted without the encryption key. Likewise, signed messages cannot
be forged. [9, 6]
• Hash values (e.g. Message­Digest (MD) or Secure Hash Algorithms (SHA)), includ­
ing Message Authentication Code (MAC), are modelled as private functions and do
not depend on any concrete hashing algorithm.
• Agents can only send messages to each other on a one shared communication
channel that some intruders fully control. Those intruders may intercept, change
or create any message with any sender name using any knowledge they obtain by
listening on the communication channel. However, they can only read the content of
encrypted messages if they know the encryption key. Furthermore, messages are
not subject to corruption or other low­level manipulation, which means they arrive
as is. If they are changed, it is because the intruder could receive that message and
construct a new one based on what they learned.
• →represents this shared insecure communication channel rather than a secure and
authenticated communication channel that WS­CDL assumes. Extensions to AnB­
notation with channel types [10] do exist. For simplicity, only insecure channels are
included in the AnB­notation considered in this thesis.
• Everything written with lowercase letters is constant, e.g. functions or honest agents
such as a trusted server.
• Uppercase letters represent dynamic/fresh values. Uppercase agents are either
Secure Choreographies in PSPSP/Isabelle

honest agents or intruders pretending to be those agents. Intruders pretending to
be honest agents do not know the knowledge of the honest agent, but if they can
learn enough, they may be able to pretend to be them successfully.
• The underlying message container is usually assumed to be type­flaw­resistant,
though sometimes this assumption is excluded to include type­flaw attacks (see
subsection 2.3.3).
In summary, like WS­CDL, the focus of AnB­notation is on the flow of the protocol itself,
the messages used to achieve their goal, rather than verifying concrete protocol imple­
mentation.
2.3.2
Example
An excellent representative example which can clearly describe security protocols such
as NSPK [11] as Listing 2.1 shows in OFMC­style AnB­notation [12].
Note: Advanced features in OFMC­style AnB­notation such as secure channels, and
channel goals [10] are intentionally left out.
Types defines everything that can be used later in the protocol, and those elements are,
in this concrete example, two agents, two numbers, and a single function.
Some security protocols depend entirely on a trusted server (E.g. Kerberos [13]). There­
fore AnB­notation allows specific roles to be fixed as honest/constant by writing them with
a lowercase letter.
Knowledge: A security protocol specification often builds on top of some existing context.
In OFMC­style AnB­notation, knowledge is given in the protocol header, which defines the
initial knowledge for any agent defined in the types above. Any agent may know other
agents and some functions. A function without any argument means that it is known for
any argument, e.g. pk represents knowing the public key for everyone, thus representing
pre­shared public keys which are not authenticated when used with an uppercase letter,
unless the protocol manages to achieve such authentication. pk(s) would represent an
authenticated pre­shred public key for some honest agent s, e.g. a trusted server.
If a function is given with concrete values in the knowledge, then the agent only knows
the function with those values. They can use any fresh value if it is an uppercase letter
different from their name. If any value in the knowledge is the same as their agent name, it
will be that agent. Knowledge could also include pre­shared keys between some agents.
Sometimes a trusted server is known by all.
Actions are the protocol flow itself. Messages are formatted much like in WS­CDL with
the addition of cryptography functions like in the Dolev­Yao model [12, 9]:
• {Msg}(PublicKey) = asymmetric encryption
• {|Msg|}(Key) = symmetric encryption
• {Msg}(inv(PublicKey)) = signing a message, asymmetric “encryption” with private
key
Goals define what the protocol should achieve, with the essential goals being of the type:
• Agent authenticates Agent on Number
• Number secret between Agents

In OFMC­style AnB­notation, it is also possible to weakly authenticate. Weak authenti­
cation is when the authentication is for the right agent but might be in another session.
Strong authentication requires that the authentication cannot be mistaken across ses­
sions. Whether the authentication is weak or strong, an authentication goal involves the
authentication of an honest agent on some Number they created. The intruder must never
know secrets, and additionally, if any agent not mentioned in the secrecy goal knows that
secret, it also counts as a security violation. Secrecy is a weaker goal than authentication.
It turns out that NSPK is insecure. The attack shown by Lowe [14], 18 years after NSPK
was introduced, is a Man­In­The­Middle (MITM) attack violating authentication. The fix
by Lowe is to add the sender at the end of the second message resulting in the protocol
seen in Listing 2.2.
The fixed version of NSPK is called the NSL­protocol.
2.3.3
Type-flaw resistance
A type flaw attack is when an intruder can take a message from one part of a protocol and
make an honest agent accept it as a message from another part of the protocol leading to
a security goal violation. One example of this is the Woo and Lam Protocol π1 from [15]
as seen in Listing 2.3:
In the protocol above, the function sk represent pre­shared symmetric keys, e.g. sk(A,s)
represents a pre­shared key between the trusted server s and A. The intruder can play
both the role of the server and A to trick B into authenticating with them, which is possible
as B cannot decrypt the message on line 3. By adding a tag to the message on line 3
or line 5 such that the messages on lines 3 and 5 can be differentiated, the attack is no
longer possible. [16]
Another example is the Otway­Rees protocol [17], which has the same issue and solution.
Adding a tag to messages is an efficient solution that can easily be applied to any protocol
[18]. Therefore, it is often considered an implementation­specific detail abstracted away
in AnB­notation. In other words, a proper message container implementation is assumed
when implementing a security protocol given in AnB­notation.
2.3.4
AnB-notation as a Choreography
AnB­notation is a bird­eye perspective of the coordination between peers in a security
protocol, which is very similar to, e.g. WS­CDL. In other words: AnB­notation is a chore­
ography. In general, AnB­notation cannot represent a long­term state, choice, parallel
flow, or conditional flow, i.e. if­then­else. However, extensions exist, with a notable ex­
tension being a long­term state and sub­protocols. The focus on a single­session linear

The intention of using AnB­notation is to communicate security goals, which is different
from most other computer science choreographies that focus more on deadlocks or cor­

2.4
Multi-party Session Types
The type­system for WS­CDL [5] uses binary session types, i.e. communication between
two peers. Every time a peer communicates with a new peer, they establish a session2,
thus ensuring that every communication channel session, every binary session type, there
exist is only used by two peers.
Sometimes peers work together in part or whole and depend on each other when interact­
ing with other peers. For example, [20] consider an extended version of the Buyer­Seller
protocol they call the Two­Buyer protocol: Two Buyers coordinate to split the cost of some
item they want to buy. The Seller sends the quote to both buyers, after which they need
to agree on how much each of them pays before sending the quote to accept or reject the
order to the Seller.
The example above highlight the idea behind multi­party session types: They describe
how peers in a system coordinate their efforts, not only as individual actions but in the
context they are in. AnB­notation, as a choreography, describes a multi­party session
type, the coordination of messages sent between multiple peers, in which order they are
supposed to arrive, and how each peer responds to those messages.

Sets can only contain abstract values and are accessible across all sessions. Any trans­
action can access any set. A set has a name with none or some arguments. The ar­
guments can be any enumeration type, except value. The arity for each set must be
set.
Functions work much like functions in the AnB­notation introduced earlier. The difference
between a public and a private function is that the intruder cannot apply a private function.
Functions can have both enumerations and abstract values as arguments.
Analysis may be defined for any defined function. Analysis can be used to make func­
tions transparent for analysis, i.e. what values the intruder can derive or to represent the
algebraic properties of encryption. In Listing 2.4, aenc represents asymmetric encryption
by conditional analysis. Only if the private is known can the content of the message be
known. Similar analysis functions can be defined for symmetric encryption and signatures.
Transactions defines the possible transitions in the protocol specification.
There are types of transactions, normal transactions and attack transactions.
Normal transactions consists of five optional parts:
• Messages received
• Match abstract values in or not in sets
• Creation of new abstract values
• Insertion or deletion of abstract values in a set
• Messages sent
Attack transactions consists of two optional parts and one required part:
• Messages received
• Match abstract values in or not in sets
• The attack keyword.
Attack transactions describe security goals. If messages received in an attack transaction
are bare abstract values (not inside a function), they represent the intruder knowing that
abstract value.
Abstract values may be any abstract values when used across transactions, even if they
have the same name, see subsection 2.6.2.
For all transactions
The first two parts define the conditions for the transaction to run. E.g. certain messages
that have to be sent or values that have to be or not be in certain sets. A transaction
without any conditions can always run.
When the requirements for a transaction are fulfilled, the remaining part of the transaction
is executed. For normal transactions, this includes creating new abstract values, insert­
ing or deleting new or incoming abstract values into sets, and sending messages. For
an attack transaction, the requirements define an attack. Reaching the attack keyword
indicates a security violation.
Any transaction run is atomic. A transaction always ends with a dot.

2.5.1
Compared to AnB-notation
AnB­notation describes a linear flow for a protocol without any way to define a state that
is shared across sessions or branching. Trac­notation avoids these limitations by not
explicitly defining the flow of a protocol. Furthermore, unlike in most AnB­notation trac­
notation store state across all sessions in sets and can thus represent long­term states.
Unlike in AnB­notation, it is possible to define Application Programming Interface (API)­
like behaviour by having some transactions that may have a long­term state that can
respond to messages sent across multiple sessions. On the other hand, it is harder to
read the flow of a protocol in Trac­notation because each transaction has no ordering or
“strong link” to other transactions. Furthermore, the end­point behaviour of every agent
is described as several transactions which are in the same flat scope. Therefore, trac­
notation is not as readable as AnB­notation.
The legitimate flow of a protocol in AnB­notation is generally straightforward, whereas it
could be more apparent in trac­notation.
As trac­notation has no restrictions to the protocol flow, any protocol flow described in
AnB­notation can be defined in trac­notation. AnB­notation implies a state is preserved
and built during execution. In trac­notation, all transactions are stateless without any direct
relation to other transactions. As long as the conditions for the transaction to execute are
fulfilled, it will be able to run, independently of what ordering the author of some trac­
notation might have thought those transactions could run in. Therefore, any transaction
context must be given via the global state from sets and functions.

In AnB­notation, goals can be put in directly as recommended in [6]. However, trac­
notation is a lower­level protocol without support for goals. Instead of goals, the designers
of Trac­notation chose to use transactions with the keyword “attack” to show goals. As a
result, it is not as easy to define goals in trac­notation as in OFMC­style AnB­notation.
2.5.2
Compared to Choreographies in General
Trac­notation has the potential to describe any protocol flow via a state­transition system.
By adding state information to each transaction via sets and functions, it is possible to
represent the flow of a protocol. The nondeterministic way transactions can be activated
is a way to represent any execution order and, by extension, attacks, as the attacks are
defined as a transaction which should never be able to run. Transactions define valid
executions without directly explaining how a transaction affects other transactions.
Choreographies describe the coordination between peers, whereas transactions only de­
scribe individual actions without context. Trac­notation is thus capable of describing a
choreography but is not a choreography.
2.6
PSPSP
Trac­notation is used by PSPSP with the most significant “artificial” restriction being that
sets can only contain single element abstract values, which means that two or more ab­
stract values cannot be linked strongly in the same state using only sets. However, as
shown later in this thesis, linking two abstract values with private functions is possible;
see section 6.6. Trac­notation does not conceptually prevent tuples in sets. However,
as it was made for PSPSP, built on top of set abstraction techniques [22], the notation
inherits all limitations of the underlying platform.
PSPSP [2] is an automated prover for trac­notation. Trac­notation was made for PSPSP
but is more expressive than PSPSP supports. Due to the abstractions in PSPSP, it is
impossible to insert combinations of abstract values into sets. The most direct represen­
tation of a protocol flow would be to insert a combination of abstract values into sets.
PSPSP prove the security of a protocol by computing a fixed point and uses Isabelle [3]
to show this fixed point covers the protocol. During the computation, all transactions are
converted into a set of possible transitions in a state system. In other words: PSPSP
does not “run” a protocol, but instead “explores” all possible states reachable using the
transactions as transitions to new states.
While the finer details of how PSPSP works are beyond the scope of this thesis, the
basics are relevant. In general, PSPSP uses this approach to prove the security of a
trac­definition:
1. Check that the protocol is well­formed.
2. Automated computation of a fixed­point. The fixed point should include all possible
states (checked by Isabelle/HOL).
3. If the fixed­point is free of attack, the protocol may be secure if the fixed point is
correct.
4. Use Isabelle/HOL [3] to prove that the fixed point covers the protocol, i.e. no possible
state is missing in the fixed point.
Assuming the fixed point covers the whole protocol and does not contain attack, the
protocol is proven secure according to the goals set by the attack transaction(s).

2.6.1
Advantages
PSPSP can provide a fully automated proof that is safe and sound in Isabelle/HOL [3] by
default. The underlying trac­notation does not inherit the limitations of OFMC­style AnB­
notation and is thus able to describe protocols with choices, conditional flow and long­term
state, e.g. API’s. Furthermore, the proof is given on an unbounded number of sessions.
2.6.2
Limitations
In this thesis, PSPSP is used to prove the security of a protocol. By using PSPSP, all of
the limitations of PSPSP are inherited.
1. PSPSP uses an over­approximation to be safe when computing the fixed point,
which can give false positives. The attacks found are sometimes only possible due
to the abstractions in PSPSP.
2. In PSPSP “two terms are equal if and only if they are syntactically equal”, which
means that PSPSP does not support the algebraic properties needed, e.g. Diffie–
Hellman (DH). [2]
3. Automated fixed point computation requires finite enumeration constants and sets.
Therefore the number of agents is constant. [2]
4. As previously mentioned, PSPSP requires that all sets only contain abstract values.
trac­notation inherit this limitation from PSPSP. [2]
5. It is impossible to refer directly to a concrete abstract value. As a workaround,
equality of abstract values can often be derived from sets and messages received.
[2]
6. The fixed­point computation can take quite a long time, even though the number
of possible states is finite. Depending on the trac­notation, the number of possible
states may be very high.
7. Waiting for PSPSP to prove a protocol safe can be time­consuming when making
minor adjustments.
8. PSPSP is not always able to prove security automatically but does give the user the
option to prove security themselves.
PSPSP is often slow to show simple attacks. For rapid prototyping of simple protocols,
i.e. linear protocols without a long­term state, using a tool such as OFMC [12] that can
give results much faster would make more sense than using PSPSP. If PSPSP runs fast,
you have a result within a minute or so, and the run­time sometimes goes into a multi­digit
number of minutes, where OFMC often finds attacks in a manner of seconds, even when
showing simple attacks. Depending on what kind of verification/security proof is desired,
it may sometimes be faster to prove it with OFMC.
Writing a specification in trac­notation for PSPSP can be challenging, with one of the
primary reasons being that there is no way to define a protocol from its flow, like AnB­
notation. Every action must be given in the form of a transaction, and the ordering of the

mistakes due to the almost limitless possibilities for transaction behaviour in trac­notation;
see chapter 4.
PSPSP is a powerful tool but requires considerable technical knowledge to use even for
simpler protocols. Making PSPSP more usable is a primary motivation for this work. It
is possible to translate the existing AnB­notation to trac­notation, though no automated
tool is known by the author of this thesis. However, as OFMC­style AnB­notation has no
conditional execution, branching structures or long­term state, it does not capture most
of what trac­notation can represent with trac­notation supporting more types of flow than
OFMC­style AnB­notation. Therefore, AnB­notation would not be a good source language
for a translation.

2.7
The Envelope Protocol
Let us assume there is a teenager who is going to a party. The parents want to know where
the teenager is in case of an emergency, but the teenager wants privacy. In the analogue
world, the teenager could write where they are in a letter and seal it in an envelope. Then,
if something happens, the parents can break the envelope seal and see where they are,
but if they do, the teenager can see that their privacy has been violated.
It is assumed that the envelope is sealed so that it is impossible to forge or seal it again
to its original state. In other words, it is a tamper­evident envelope (conceptually like the
tamper­evident letter in which credit card pin codes arrive, which is almost impossible to
seal again). If the parents open the letter and burn it together with the envelope, if they
burn it without reading the content, or if they reseal the letter in another envelope, they
lose the ability to prove that they did not break the seal of the original envelope.
Ables and Ryan [26] introduced a digital version of this protocol and called it a digital En­
velope, The Envelope Protocol, with one significant difference: In the digital envelope,
the parents have the option for revoking access to the message themselves and will be
able to prove their access is revoked. It would be equivalent to getting a trusted “certifi­
cate” for burning the envelope containing the letter before opening the envelope, which is
not trivial in the analogue world (one would have to completely trust the certificate issuer
can check that the envelope has never been opened). As mentioned in [26], this protocol
cannot be achieved using cryptography alone. The parents would not be able to decrypt
the message without a decryption key (whether it is symmetric or asymmetric does not
matter). They cannot prove whether they have read the message if the parents know the
decryption key, and without the key, they would not be able to open the envelope in the
case of an emergency. It is the chicken and the egg problem.

is a tamper­evident chip that can store secrets. An HSM aims to achieve a higher level
of security than possible using software alone. Communication with an HSM happens
via an API. Suppose an HSM generates a key pair and keeps one part secret. In that
case, the teenager can check from the state of the HSM module whether some message
has been decrypted with that key or not, with the state of the HSM representing both the
state of the envelope with regards to it being the same envelope and whether the seal has
been broken (some message has been decrypted with the key pair the teenager used).

2.7.1
Trusting an intermediary?
Why not just build an app or website that can keep the message and only reveal it in the
case of an emergency? I.e. a website that promises to do what the Trusted Platform
Module (TPM) does?
In the analogue world, giving the message to a website would be equivalent to giving the
message in clear text (i.e. without sealing it in an envelope) to a friend and trusting they
are honest about whom they have shared the letter.
If the teenager cannot trust their friend, then even if the teenager gets the letter back
and can destroy it safely themselves, they would not know if copies of the letter exist or
if anyone has read it. If they leave the proof of “not having read” the message to their
friend, they would have to completely trust their friend to be honest about that as well, for
the same reason as if they get the letter back. The teenager could protect the message
with an envelope, but then, why give it to a friend in the first place?
Most websites would be equivalent to a friend that can only be trusted partially. An HSM
is presumed to be safe on a hardware level and much safer and more robust to unau­
thorised tampering than a website that promises to act honestly. The trust in an HSM
is for the envelope protocol’s modelling purposes equivalent to the cryptography model
of the Dolev­Yao model; assuming an HSM is safe is like assuming encryption is perfect.
Therefore, this thesis only considers envelope protocol models that assume unconditional
trust in an HSM as an intermediary.

3
The Envelope Protocol in PSPSP
Bruni et al. [4] modelled the TPM­based, no software required, digital envelope introduced
in [26] and proved that their model of the protocol is safe in Tamarin [29]. In their version
[4] of the digital envelope, they use choreographies to make a high­level choreographic
specification of the protocol. Aside from the machine­checked proof, all their efforts to
model and validate the protocol are manual, including the EPP of their end­point­oriented
machine­checked proof.
In this chapter, The Envelope Protocol is modelled in trac­notation (see section 2.5) for
PSPSP to prove safe. The motivation is that protocols in trac­notation can be formally
proven with more automation than Tamarin.
3.1
A Simplified HSM Model
Introducing a simplified HSM based on the functionality of a TPM (using the general ideas
from [27] and [4]) that removes all functionality from the TPM aside from generating key
pairs, keeping track of whether those key pairs have been used and decrypting messages.
This simplified HSM model only contains the absolute bare minimum to use the HSM in
the envelope protocol, and those API­calls are:
Request key pair: Sends back a signed message with the public key for the freshly
generated key pair combined with the Token for the request. The Token is to help the API
user detect replay attacks.
Obtain message: If the key pair for the encrypted message sent to the HSM has not
been refused, the HSM will respond with the decrypted message and change its internal
state to mark the key pair as used.
Refuse key pair: If the key pair has not been used to obtain any message, the HSM will
mark the key pair as refused and send a signed proof that the key pair has been refused.
Every API­call can be called repeatedly. However, even if called repeatedly, the HSM
will only respond if the conditions for the API­call are fulfilled. Getting a fresh key pair is
always possible since no conditions restrict that API­call. The decryption of messages
and refusing keys depends on the state of the concrete key pair.
The state of the HSM is defined by two sets:
available_key_pairs and sealed_key_pairs.
• A key not in any set has not been generated by the HSM.
• A key both in available_key_pairs and sealed_key_pairs has neither been used
or refused.
• A key only in available_key_pairs has been used to decrypt a message.
• A key only in sealed_key_pairs has never been used to decrypt any message and
can never be used to do so.

A full­fledged HSM, like a TPM, can be used for many applications with many possible
states. This version removes most of the inner workings of a HSM. Still, it preserves
the mechanism that makes modelling a digital envelope possible using the no­software
solution from [26]. Much like AnB­notation abstracts away from concrete cryptographic
implementations, this simplified HSM model is a high­level representation of the HSM that
abstracts away from the concrete implementation with, e.g. a TPM’s PCR and low­level
commands.
3.2
Enumerations
The HSM is an honest role that cannot be mistaken for anyone else; therefore, it should
get an enumeration with only the HSM in it. The teenager and the parents are regular
agents in the protocol. Neither the teenager nor the parents are to be considered honest
since both might attempt to misuse the HSM API. The protocols should be safe even if
the parent or teenager is dishonest. Therefore the trac­notation enumeration becomes as
seen in Listing 3.1.
While one could argue that the actual adversary/intruder in this protocol is the parents,
the protocol should work just as well if the teenager tried to play the role of the parents.
Nothing prevents the teenager from not giving the envelope or burning it before handing
it over to the parents. Likewise, if the teenager seals an envelope by refusing the key, the
parents should not be able to open that envelope. Therefore, they are on equal terms,
with the only honest participant being the HSM.

3.4
The Key Pairs
The HSM­generated key pairs must be fresh, with both a public and private part that
belong together. In trac­notation, if two abstract values were given, they would not be
strongly linked. Instead of developing a method to link two abstract values strongly, one
could use functions to model different parts of an abstract value. For example, assume a
new value called GenKeyPair, then let the public part be pk(GenKeyPair) and the private
part inv(pk(GenKeyPair))1.
3.5
The Payload
Every message the teenager writes is assumed to be unique and fresh. If the teenager
writes the same message twice, some unique nonce or similar mechanism will ensure
every message is unique. Since a message is always unique and never repeated, an
intuitive way to represent the teenager’s message for their parents is via a new abstract
value, as they are guaranteed to be unique. Thus new Msg represents a new message.
3.6
Communication with the HSM
The secure communication with the HSM is modelled by having everyone know a public
key for the TPM, which is used to encrypt messages sent to the TPM and authenticate
messages from the HSM by checking the signature with the public key for the HSM.
There are two main ways to represent a key pair for an agent in PSPSP: By dynamically
generating keys or with static keys. Dynamic keys use abstract values to represent the
keys, and static keys use the enumeration constants via the enumeration identifiers. If the
first method is used, the known keys must be placed in a “shared” set used by all who need
to know those keys, which means every transaction that receives messages encrypted or
signed with those keys will have an extra check. Messages that use enumerations to form
keys can derive the key relation by the variable to enumeration typing in the transaction
header. See a brief example of both in Figure 3.2
As the type for pk is already defined to be pk(value) in section 3.4, using dynamic
keys seems like the way to go. However, the author of this thesis subjectively prefers
the static keys to avoid “transaction clutter”. A well­formed trac­notation protocol can­
not contain both pk(value) and pk(hsms), as the type value ̸= hsms. Another function
must be used for the trac­notation protocol specification to be well­formed. pk_hsm was
chosen, which means that the key pair for communication with the HSM is represented
as a static pre­shared public­private key pair where pk_hsm(hsm) is the public key, and
inv(pk_hsm(hsm)) is the private key that only the HSM knows.

3.7.1
HSM API
The simplified HSM model is a set­based model, which can be translated directly into trac­
notation. Defining one transaction for each API­call is sufficient to represent the simplified
HSM­model. Every API call starts with a message encrypted with the public key of the
HSM. The content of the encrypted message defines which API­call it is.
hsm_sent(Msg, GenKeyPair) act as an event flag: If a Msg encrypted with another key
pair is obtained and later encrypted with a new key pair, it should not count as an attack,
since the Msg is not secret. The hsm_sent(Msg, GenKeyPair) gives a flag that can later be
used in an attack transaction to differentiate which messages are encrypted with different
keys. The HSM model cannot check whether a Msg was encrypted with the key pair used
when the obtain API­call is made.
hsm_sent(Msg, GenKeyPair) is represented as a private non­transparent function in PSPSP,
which means that only the transactions given in the trac­notation can make create the
event flag.
Example: hsm_api_obtain receives a Msg and pk(GenKeyPair) that should be decrypted.
The HSM checks that the GenKeyPair is in available_key_pairs key pairs. If not, the
transaction will not fire, and the API will not respond to that message. If the condition
holds, then the HSM updates its internal state to save that this GenKeyPair has been
used to obtain some message with delete GenKeyPair sealed_key_pairs and sends
out the message send msg(KeyPair) together with hsm_sent(Msg, GenKeyPair).

3.10.1
No Event Flags
The variant in Appendix D.2 use msg(GenKeyPair) as a private non­transparent function
that acts as the message itself rather than a formatting function. By linking the GenKeyPair
strongly to the message itself, there is no need to use event flags for messages decrypted
or messages that the teenager creates. The HSM cannot decrypt previously revealed
messages, and lastly, msg(GenKeyPair) might seem more intuitive to power users of
PSPSP.
On the other hand, there can only be one message for each GenKeyPair, and GenKeyPair
is the only abstract value that can properly represent the message in a set. There are no
limitations to the number of GenKeyPair that can be obtained. Therefore the “only one
message per key pair” limitation is not an issue in practice. Using GenKeyPair for both
the key and the message preserves the strong connection between the two. By ensuring
only the teenager transaction can create msg(GenKeyPair), it can be certain that when
msg(GenKeyPair) exists, a valid transaction “run” must have created it.
When the teenager uses a GenKeyPair to create a message with msg, that GenKeyPair
becomes a secret, which is properly the least intuitive part about this variant: The ab­
stract value GenKeyPair is used for both the key pair and the message. The message
does not exist as a value, only as a function of GenKeyPair. However, it makes the at­
tack transaction very short and straightforward, as shown in Listing 3.13. In this variant,
defining an attack transaction that checks the refused proof, like in Listing 3.14, from the
HSM gives a false positive. The signed message appears in the fixed point, without the
HSM ever sending it, most likely due to some of the overapproximation of PSPSP.
Consequently, the HSM will only decrypt valid messages, which does not hinder attacks
since any message decrypted with a different encryption key from which it was encrypted
is not relevant for the security goals. If an encrypted message and the corresponding
public key are known, anyone can use the HSM API to refuse or obtain that message.
Furthermore, anyone can still use the API and encrypt their “own” messages by requesting
new key pairs from the HSM, which will then be a function of those new key pair.

Revealing the Private Keys
The effects of the HSM behaviour in the “No Event Flags” variant is that when a mes­

any other message encrypted with that encryption key, but as the key pair is no longer
sealed, encrypting any new message with the public key from that key pair no longer give
any guarantees about the seal of the envelope. It is the same as putting a message into
an envelope with a broken seal. It will not be resealed again.

3.11
Modelling Notes
The envelope protocol in this chapter uses a heavily simplified HSM model. A more com­
prehensive HSM model could prove a result closer to the actual implementation of HSM,
which may be wanted.
In general, some link between the encryption key and the message encrypted is required
to avoid false attacks. The link is required since the HSM will always be able to decrypt
any message, even if they were not encrypted with that key in the first place, representing
how a real HSM would act. However, decryption with a wrong key would not reveal the
messages, which is different from the real world. It is not the task of an HSM to prevent
the decryption of messages previously decrypted with another encryption key; thus, the
model represents the intended behaviour.

3.11.1
Modelling Trust
Using the envelope protocol is a two­way trust exercise in the real world. The parents
would have to trust that the teenager leave a valid message in the envelope. For the
digital counterpart, this means that the ciphertext contains the correct message, and they
give the right public key. The teenager cannot sign the message to prove they have
provided the correct public key without the private key, which is safely stored inside the
HSM. Even then, it would not prove that the message contained within the ciphertext is
usable by the parents. The transactions of the protocol models only describe “correct”
transactions. If the parents try with any other combination, they will not get a response
from the HSM.
Since it is hard to encode trust in any message the teenager writes in the models, it is not
considered an attack if the Msg the teenager sent is garbage/invalid. The content of the
message does not matter for this envelope protocol model. The only thing that matters
is that the abstract value representing the message is kept secret, where “kept secret”
is defined as never decrypted by the HSM. In other words: This model does not model
whether the parents receive a readable message.
3.11.2
Outlook: A Smart Envelope Protocol
A TPM is not the only kind of HSM. A smart card, such as a SIM card, could implement the
envelope protocol with a phone as the interface to the HSM. Since the message would be
on the SIM itself, the SIM card will be able to show whether the message was revealed.
There is still a chain of trusted computing that needs to be in order. However, from a
broad perspective, it is interesting to consider alternatives. An HSM can be implemented
in many different ways, and by modelling the behaviour of a HSM rather than a specific
TPM, the mode has a potentially higher degree of general usability. Like with the Dolev­
Yao model for cryptography [9], the security of the underlying HSM will depend on the
concrete implementation. Implementation is a critical step of any security protocol.

Preserving Protocol Flow in PSPSP
The PSPSP [2] package [21] from the Archive of Formal Proofs (AFP) [30] contains sev­
eral example protocols written in trac­notation, with one of them being NSPK.
In this chapter, the reference model of NSPK in PSPSP is analyzed with regards to the
ordering of actions, which highlights a “limitation” of PSPSP and trac­notation. The refer­
ence model is refined to properly model the protocol flow of NSPK.
4.1
Protocol Flow in PSPSP
When a protocol is checked in PSPSP, the concept of states disappears, and only pos­
sible transitions exist. Evaluating all possible transitions gives the fixed point (in broad
terms, see [2] for details). The fixed point computation works on equality and implica­
tions. Transactions are checked as if they run in any possible order, with no concept
of starting messages. Many protocols build a state, and unless enforced via sets that
differentiate values from each other, PSPSP will not capture this.
In PSPSP, as mentioned in [2], it is impossible to refer to abstract values by name. It is
only possible to refer to an abstract value based on which set it is. If NA and NB are put
into the same set the fixed point is computed assuming they could be the same abstract
value.

5
Security Protocol Choreography
In this chapter, Security Protocol Choreography (SPC), a new high­level choreographic
security protocol specification, is introduced. SPC takes the idea of using choreographers
for security protocols in [4] and is motivated by a desire for more automation in verification.
OFMC­style AnB­notation give the basis for the “look and feel” of SPC.
SPC can describe protocols that contain branching, long­term states and conditional ex­
ecution while allowing for automated verification by translation to trac­notation, which is
the protocol specification language for PSPSP.
5.1
Design Goals
In one sentence, the idea of SPC is to give a choreographic description of security pro­
tocols that cannot be described in classical AnB­notation but could be written in trac­
notation. It could be because those protocols contain choices, conditional flow or a long­
term state across multiple sessions. Transactions in trac­notation describe security proto­
cols by their possible actions, not their choreographic context. SPC is meant to feel similar
to AnB­notation and inherent the intuitive clarity of AnB­notation, which trac­notation lacks.
Aside from the usual flow between agents with choices much like with WS­CDL [5], SPC
aims to allow agents to send and receive messages to and from agents that are not con­
strained by the concrete flow of that protocol. E.g. some API like an HSM. API agents are
defined by the actions of each API­call, which may optionally update their internal state.
Removing API actors from the protocol flow itself makes it clear that the actions of some
API agent do not depend on the state of the overall protocol flow but only on individual
messages sent independently of the concrete session or state of any agent in the main
protocol flow.
Supporting automated verification is a goal for SPC, but it is beyond the scope of this
thesis to build a new protocol verification tool. SPC aims to support verifying security
protocols with choices, conditional flow and a long­term state. PSPSP [2] is made to
verify the security of such protocols, and therefore SPC is built from the ground up for
translation into PSPSP’s trac­notation. Using this notation means that SPC inherit all the
limitations of PSPSP, see subsection 2.6.2.
The most restrictive limitation inherited from PSPSP is how values are “stored”. PSPSP
use sets to represent the current state of the protocol. Sets can only contain values and not
value tuples or functions, which can cause problems, as seen in chapter 3 and 4. Ideally,
SPC would allow anything to be inserted into any set, but if SPC did this, it might not be
possible to translate those protocols into PSPSP later. Therefore SPC preserves these
limitations and builds them into the syntax. SPC hopes to be transparent enough such
that most modelling tricks required to make a protocol specification work in trac­notation
can still be applied to SPC and passed along to PSPSP. As a part of the transparency
goal, SPC inherits the transaction­style security goal specification from PSPSP.
In OFMC­style AnB­notation, cryptographic messages have a special syntax, see sub­
section 2.3.2. Cryptographic functions could be represented with normal functions with­
out any special syntax. However, as this notation is intuitive and differentiates normal
functions from cryptographic functions, SPC aims to use the same syntax. In all other
cases, syntactic sugar is avoided in SPC unless inherited from trac­notation.

5.4
Executability
In AnB­notation, executability is defined with formal mathematically defined semantics in
[6]. This check is a naive informal approach which is only a rough approximation of [6].
It is the goal for SPC to check executability formally as in [6], but it is kept simple to limit
the scope of this thesis. 2
A well­formedness SPC protocol protocol is executable when:
• Every agent can compose the messages they send from the knowledge they could
have.
• Every agent knows the sets they use. See why in subsubsection 5.6.6.
• Every agent could know any variable or constant they use.
• Every agent only creates values or numbers they know.
• All Numbers and Values are initialized with new or received from _ before use.
• Conditional protocol flow does not prevent execution within a single session.
The composability of messages is checked by simulating a protocol run while keeping
track of values that each agent can learn. Every value or message an agent has to con­
struct must be either directly in their knowledge or derivable. In addition to their knowl­
edge, agents may use any public function or formatting function to build messages. How­
ever, they cannot use constants or values they do not know to construct messages.
To compose a signed message, agents must explicitly know the inv of the public key they
use to sign that message.
Every agent implicitly knows themselves and all formatting functions and can apply all
public functions to anything they know.
In the simulated knowledge state, the agents gain knowledge from messages they re­
ceive:
• Messages in formatting functions are opened, and each element in the formatting
function is added to the recipient’s knowledge.
• Encrypted messages, i.e. {Msg}(Key) or {|Msg|}(Key) can be opened if the agent
can compose the decryption key. In the first case, the inv of the key must be in their
knowledge. In the second case, the Key must be composable.
• For signed messages, {Msg}(inv(PublicKey)), the agent learns Msg.
• Encrypted messages that cannot be opened are added as is to the agent’s knowl­
edge.
• Every time an agent receives new messages, the new knowledge state will be used
to attempt to decrypt any encrypted messages they have in their previous knowl­
edge.
When a choice is made, the knowledge is split into a scope for each choice, with the
knowledge state before the choice as a base; a new value in one choice branch does not
mean the value is also defined in the other branch.

5.5
Recommendations
This section covers recommendations that should appear as warnings in any implementa­
tion that check well­formedness. Not following recommendations does not affect whether
a SPC protocol is well­formed but usually indicates something that might lead to unex­
pected results or a less readable specification.
5.5.1
Enumerations
For Enumerations, it is recommended to “repeat” enumeration constants via set union ++
instead of manually repeating the enumeration constant. Figure 5.2 shows an example
where it is recommended that servers are defined as the set union of agents and another
set that contains the server enumeration constant.
The reason for this recommendation is that unless one looks very carefully, it might look
like there are more enumeration constants in a protocol specification than is the case. E.g.
if two letters that look similar are placed in two different enumerations might indicate that
they are two different constants at first, which means that the protocol might be read with
this potentially wrong assumption in mind. Explicitly repeating numbers via set unions
makes it clear that an enumeration might be in might “be in” more than one enumeration
identifier.
Explicitly repeating enumerations constants in the Enumerations header is not wrong.
This recommendation merely expresses the preference of the thesis author
5.5.2
Knowledge
It is recommended to:

5.6
Semantics
A well­formed protocol that is executable ensures that:
• Every set, function, agent, enumeration, number and value can be uniquely and
correctly identified.
• All agents’ roles are clear as they are mapped to an enumeration identifier.
• Causality holds (much like “Well­threadedness” in [5]); the ordering of the actions
represents an executable protocol flow:
– A value cannot be used before it is created.
– Agents can only perform actions after receiving some message unless they are
the initiator of the protocol.
– All agents can compose all messages they are supposed to send and can know
all values they put into sets.
• Sets are never sent as messages.
• Agents only use sets they know. Sets can thus represent the “memory” of agents
with explicit privacy for those sets via the Knowledge header.
A protocol that is not well­formed might have ambiguous names, messages that appear
“out of the blue” or require agents to compose messages that they couldn’t possibly com­
pose.
Note: As transactions represent independent atomic actions, they are not checked for
executability in the global context.
Additionally, if a protocol follows all recommendations, it ensures:
• That the protocol contains at least one attack transaction, indicating at least one
security goal is checked. In other words, it is possible for the verification in PSPSP
to check the protocol for security. A protocol without any attack transaction will
always be proven safe in PSPSP.
• There are no redundant protocol headers.
– With the exception of the Knowledge header as a proper check would require
the methods from [6].
• Clarity of enumeration constants relations to enumeration identifiers in the Enumerations
header.
• That no set is not used in a way prone to modelling errors.
• Some transaction attempts to implement all API­calls.

5.6.2
Numbers and Values
Numbers and values both represent PSPSP abstract values. The difference between
them is that the FlowModel is applied to Numbers and not to Values.
The FlowModel adds flow information to the trac­notation translation of SPC to preserve
the protocol flow. This is implemented by adding the numbers to uniquely named sets that
are only used for those values and include checks for whether those values are in those
sets when the Numbers are received and are thus using the same logic as in chapter 4,
except that the numbers are not deleted from the sets that keep them, since Numbers may
be used more than once in a SPC protocol.
The intention is that all Numbers are “kept in the state” as they would be when learned in
the protocol flow. Sometimes numbers are received and forwarded while a new number
is created. In such cases, it is expected from the semantics of the protocol flow that those
exact values are used together later in the protocol flow. An example of numbers that
should be linked is NSPK (see Listing 2.1). In the second step B receives {NA,A}(pk(B))
and sends {NA,NB}(pk(A)). Notice that NA is forwarded. The FlowModel will add private
non­transparent functions to the trac­notation model so that when B receives NB again, it
will be connected to the context of which NB was created; with the concrete NA received
when creating that NB, and together with the concrete A and B from the context that was
when NB was created in the second step.
Trac­notation does not encode protocol flow by default. SPC allows protocol flow to be
encoded into the trac­notation. Still, as this might not always be wanted, the user can
choose to use or not use the FlowModel by putting abstract values in Values instead of
Numbers.
The FlowModel is a part of the translation process from SPC to PSPSP introduced in
chapter 6, with details of the FlowModel explained in section 6.6.
In summary, the difference between Numbers and Values is that Numbers named this since
the intention is that they should “act” as a “Number” would in AnB­notation. “Values” are
values in PSPSP and are therefore named as such. Both are kept to make SPC modular.
5.6.3
Functions
There are three kinds of functions in SPC: Public, private and formatting functions.
Formatting functions are transparent and known by all. They represent some transparent
message container. Everyone can construct any message with formatting functions, and
anyone can apply formatting functions. Example: A formatting function could describe a
concrete distinct message container such as Extensible Markup Language (XML) where
the format of the function is made so that it cannot collide with any other formatting func­
tion.
Public functions are one­way and known by all. If an agent knows the arguments, they
can check that they match. Example: A public function can represent a cryptographic
hash function.

Private functions are one­way functions that work like public functions but can only be

5.6.4
Ordering of Steps
The semantics behind the enforced ordering of transaction steps and internal steps is that
in and notin act as conditions for the flow of a transaction or an internal step. All steps
within the same block are atomic, and values cannot be redefined. Since new cannot
overlap with any received value, it will always be fresh. Enforcing conditional checks and
freshness before set manipulation ensures that all abstract values that can be inserted or
deleted from sets are defined. Transactions can contain internal steps and are optionally
wrapped by an optional send and optional receive block. Following a logical temporal
causality relation that a transaction cannot send something before the message that is
part of the conditions for sending that message is received, all the elements of the well­
formedness of a transaction should be clear.
Attack transactions are meant to observe the state of the protocol, not manipulate it.
Therefore it only contains the conditional flow elements where attack is a signal to PSPSP
that indicate an attack has been found.
5.6.5
API
A message sent to or from _ represents communication with something external to the
protocol flow, in other words: An API.
Transactions are used to represent API’s. When a message to or from _ is matched by
a transaction, it means that there exists some transaction that “answers” to that API­call.
The check is not strict but a recommendation, see subsection 5.5.3, since multiple trans­
actions may be used to represent a single API­call and because messages could have
another name in a transaction. E.g. if a value is renamed for clarity of the transaction, it
would still match the structure.
5.6.6
Knowledge
A well­formed knowledge block that follows all recommendations means that the knowl­
edge represents everything unique agents need to know to run the protocol; see sec­
tion 5.4 and subsection 5.5.2.
Values and numbers originate from some agent, and only the agents that know those
values can create them unless they received them from an API. Therefore Numbers and
Values must be initialized with new or received from _ before use.
Note: Transactions are not checked for knowledge since it is ambiguous who “owns” a
transaction.
Knowing Functions
Initial knowledge of a function with some arguments means that only the function with
those arguments is known for that agent at the beginning of the execution of a protocol.
Initial knowledge of a function without arguments can mean either knowing a private or
public constant or knowing that function with any arguments:
• If a function is used with some arguments in the protocol, knowing that function with
zero arity in the knowledge header means knowing that function for any arguments,
like knowing a set without any arguments. This could, for example, be a hash func­
tion.
• If a function is used without arguments in the protocol, it represents knowledge of a
constant that is either public or private depending on the type of the function.

Knowing Sets
The semantics of “knowing” a set means that an agent has access to a set that repre­

arguments and the arguments do not contain the agent name since this means that an
agent potentially accesses the memory for another agent, which is allowed (but not recom­
mended in general) since it might be required for some modelling when SPC is translated
into PSPSP.
Knowing a set without arguments can mean the literal zero­arity set or the set with any
arguments, which depends on how the set is used in the protocol. Multiple agents knowing
a set with zero arity might defeat the purpose of making the protocol specification in the
first place, as values could be shared securely via sets, which may not be intentional. If
two agents have a “shared state”, they can just put any encryption keys they need there
and always have safe communication. A shared state could represent an Out of Band
(OOB)­channel (E.g. Two Factor Authentication (2FA)), which may be a valid use­case
depending on the protocol. It is recommended to be extremely careful when sharing sets
between agents.
SPC is translated to PSPSP in chapter 6, and due to that context it is relevant to note that
PSPSP does not have the concept of “set ownership”, so sets may be “shared” between
any transactions, and thereby agents. However, this is usually not how one would model
protocols and could indicate a mistake a user has made. Therefore knowledge of sets
has to be included explicitly in the Knowledge header.
There are no restrictions to what sets can be known, but using e.g. a shared set is against
recommendations, see subsection 5.5.2. Any implementation of SPC well­formedness
should give a warning when the protocol does not follow the recommendations as men­
tioned in section 5.5.

5.6.9
A Comment on Action Lists
What is the logic behind restricting some cases that would otherwise not break the protocol
flow? Restricting the combinations possible in the action list makes it easier to ensure that
all other well­formedness checks hold.
Consecutive InternalStep actions are conceptually acceptable but would be more like
syntax sugar because they would have to be for the same agent to preserve the causal­
ity/flow of the protocol, which would make it harder to check the ordering of the actions
performed within an internal step as the well­formed ordering of internal steps will be
spread across multiple lists that should be correctly merged. The same logic follows with
internal steps before choice; that would be the same as copying the steps to each branch.
Actions after a choice are technically possible but avoided in SPC since it is not as imme­
diately intuitive to describe how two branches join again, and it goes beyond the scope of
the goals for SPC.
Not allowing consecutive internal step actions does not prevent any internal step that is
still well­formed. They must be written as a single internal step instead of two. Not allowing
internal steps right before choice does not prevent the user from adding the same internal
steps at the beginning of both branches. In other words, both cases are syntax sugar that
is avoided as set by the design goals.

possible, as early as possible. The later something is checked, the easier it is to “forget” to
check, e.g. when testing a new module. Checking early means that the code is included
in as many checks as possible, and the parser is one of the essential parts.

5.7.2
Interesting SPC Syntax Choice
By carefully reading the SPC syntax, one might note that the structure for internal steps
and transaction lines are very similar. In the Haskell implementation, they are even parsed
into the same data type. Despite this, there is a minor difference. In the transactions,
each “step” is separated by a whitespace char, whereas each element is separated by
a comma in the internal steps. There are three reasons for this. One is that a comma­
separated list is easier to read if the internal steps are put into one line for compact SPC
specification. Second, comma separation is consistent with the separation of messages,
all other elements in the header, and other parts of actions. Transactions are the exception
since they take strong inspiration from trac­notation. Finally, the default parser generated
by Haskell without comma separators will generate a shift­reduce conflict. Parsers should
be without “shift­reduce” conflicts. For these reasons, a different type of separation in
actions and transactions was chosen.

The overall approach is to perform many “small projections” and merge them all in one

6.5
Why Blocks?
A reader might wonder about the reason for working in blocks instead of Actions directly.
The answer is simple: To divide the task into smaller parts.
The intention is that an unmodified list of blocks is an alternative format of Actions. Blocks
are thus reversible to Actions if not modified after translation. Blocks are ordered to
preserve the ordering of actions in the protocol. Choices are preserved with Split.
Both The FlowModel and the blocks to transactions process (introduced later in this chap­
ter) needed to know “blocks” whether they are a direct data structure or manually derived
60

==PAGE:69==
from the actions. Converting Actions to Transactions and then applying The FlowModel
would not work, as transactions contain less information than blocks; transactions have no
ownership, ordering, no information about the previous sender or next recipient or choice
context. Blocks preserve this information, all of which The FlowModel needs. Converting
to blocks as a separate step keeps the logic of that conversion away from The FlowModel
and the blocks to transactions process.
Moreover, blocks allow sending messages that do not follow a well­formed protocol flow,
e.g., messages sent but not received by the next agent, as they are essentially transac­
tions, which The FlowModel requires.
Blocks are the projection of a single “action” for some agent. All blocks owned by the
same agents conceptually form the EPP for that agent.

6.6
The Flow Model
Translation to trac­notation directly without preserving separation between abstract values
can misrepresent the flow of a protocol, as seen in chapter 4. The FlowModel automates
the process of adding sets and functions to preserve the flow of a SPC protocol when
translated to transactions.
The FlowModel is only applied to Numbers; see subsection 5.6.2.
6.6.1
Workflow
Prerequisites: A list of blocks and corresponding headers from the SPC protocol speci­
fication to differentiate between types of variables and constants.
The general idea is:
• Every time Numbers are sent, put them into a uniquely named set.
• Every time Numbers are received, it is checked if the number is in the previously
generated set.
• If a new number is created in a block where a number is forwarded, they are linked
with a state function.
• When a Split is seen, The FlowModel check that no generated name overlaps in
the branches.
The set names include the agent name and number name (assumed to generate a unique
name for a set that is not defined yet) and use the block owner as the first set argument
and the other agent as the second. The second argument is omitted if the other party is _.
Assuming the protocol is well­formed, the generated set names will not collide but might
overlap with user­defined sets.
Furthermore, a new number with the same name inside two distinct choice branches will
result in the same name for the generated set if used by the same agent. It is only possible
to detect such cases because Split preserves the branching of choices. Such cases are
not supported and require the number to be renamed.
Forwarded numbers generate a private function with the agent names and all numbers
used in the block as arguments, which is added as a sent message to that block. When

6.7
Blocks to Transactions
The last step before export to trac­notation is to convert blocks to transactions. Every
block has an owner, but the concept of an owner does not exist with transactions. This
section explains the process for encoding ownership into transactions while converting
blocks to transactions.
Remember that Transactions in SPC protocol specifications are exported directly to trac­
notation. Those transactions do not need to be converted and are not processed by the
methods in this section.
6.7.1
Workflow
Prerequisites: A list of blocks and corresponding headers from the SPC protocol spec­
ification to differentiate between types of variables and constants. These blocks may
optionally be pre­processed by, e.g. The FlowModel.
The workflow is as follows:
1. Evaluating the enumeration for the owner of each block.
2. Conversion of each block into a transaction.
6.7.2
Transaction "Ownership" Conventions
Even though transactions have no concept of an owner, it is possible to encode who
“owns” a transaction into the transaction header as seen with NSPK, e.g. in Listing 2.4
the owner of a transaction is the agent with the honest enumeration.
Usually, a transaction has at most one agent with the honest enumeration. Attack trans­
actions are different since they observe the protocol state. They are not a part of the
protocol.
All transactions are on a hostile network, and therefore it should not be certain who they
receive messages from or send messages to. Any agent that is not the “owner”, or the
transaction could potentially be the intruder, as included in the agents enumeration.
Transactions may have no honest enumeration. In those cases, the owner of those trans­
actions is either ambiguous or given by some other context, e.g. the transaction name.
In other words: it is assumed that whenever an enumeration can be “resolved” to be
honest, the honest enumeration should be used if it is the enumeration is used for the
“owner” of that transaction.

6.7.3
Block Ownership
The concept of ownership is encoded into transactions using the conventions from the
previous section: The translator will attempt to reach honest for the agent that owns a
block and use it if reachable. Otherwise, the enumeration type given in the Roles header
is used. The translator will only attempt to “reach” honest for transaction owners.
Because of the translation from blocks to transactions, the honest enumeration has a
special meaning in SPC. The special meaning of honest could be made more apparent
and more general as discussed in section 9.4.
6.7.4
Transaction headers
Every transaction must have a transaction header. The header defines the types of vari­
ables used in that transaction, just like the SPC protocol headers do for actions and blocks.
All variables are unique in a well­formed SPC protocol specification. Therefore the types
for all variables can be found directly in the headers. The process is as follows:
For all variables used in a transaction that are not fresh in that transaction (E.g. NA would
be excluded if new NA is seen), lookup the type:
• If the variable is a number or value, use the value enumeration type.
• If the variable is the block’s owner, resolve owner enumeration according to the rules
in subsection 6.7.2.
• Otherwise, use the enumeration given in Roles.

6.7.5
Split
Transactions are flat. Each subpart is translated the same way when a Split block is
seen. Only the transaction name is affected by Split for readability of the transaction
output.
6.7.6
Transaction name
Transactions generated from blocks use the following naming scheme:
transaction_{owner}_c{1,2}_)*(_{counter}_{owner})
The agent that owns the transaction is put into the transaction name along with a counter
that follows the global order of each transaction, assuming the transactions run in the
same order as the Actions in SPC.
Choices are added to the name before the counter and owner, represented as Split in
blocks. The agent that made the choice is saved as a part of the choice. The names c1
and c2 indicate which branch each transaction belongs to.
Transaction names mean nothing to PSPSP. They are merely named in a predictable way
to make the output humanly readable. Any transaction name could be used as long as
the names are unique.
If user­defined transaction names collide with the transaction names generated by the
translator, the translator will halt and require the user to rename their transactions.

6.10
Translation Notes
As mentioned in section 5.4, there is a pretty involved formalisation of how messages are
sent and received by honest agents in the [6] semantics. As this is not the focus of this
thesis, the translation includes several simplifications, namely that whenever an honest
agent receives a message, they can check that the entire structure of that message is
according to protocol. The simplification has the advantage that simple pattern matching
can be used in trac­notation according to the original SPC protocol specification without a
complicated transformation. Therefore, the checks performed to ensure that every agent
has the necessary knowledge to run the protocol are limited.
Not only is the executability check performed in section 5.4 limited, it does not affect
the translation. The executability check is a yes/no based on a simplified knowledge
simulation that imitates some semantics from [6]. In some cases, knowledge is not a
question of yes or no but should be encoded into the protocol translation.
Consider Listing 6.6: Even assuming all agents have the knowledge required to send
their first message B should not know what N is when they receive h(N), which should be
reflected in the trac­notation if this protocol should be translated more correctly. Further­
more, there should be a link between the second message and the N that B receives in
the last message so B can know they are related since transactions can run in any order.
The translator presented in this chapter does not include knowledge and does thus not
translate such cases “correctly”.

The FlowModel strongly links numbers together via private functions that preserve the
state of an agent, which was thought necessary to model NSL and NSPK properly. How­
ever, it was determined late in the project that they were unnecessary for the use case
they were initially designed for. Since the functionality is helpful and does not hide attacks
(NSPK is still unsafe, and NSL is still safe with The FlowModel), the model for forwarding
numbers is kept.

6.10.2
A Comment on Honest Roles
In OFMC­style AnB­notation, servers are written with a lowercase letter. SPC has no
lowercase role, but a role can be constant in practice if the enumeration identifier only
refers to a single enumeration constant. A lowercase letter could have been allowed,
but that would not have been “true to the spirit” of trac­notation. The intention with trac­
notation is to use enumerations such that possibilities can arise. With roles in SPC, the
user may have more than one honest server, making SPC more expressive than AnB­
notation.
Enumeration constants can be used directly in sets and function arguments but not as
roles. Due to the argumentation in the previous paragraph, not supporting lowercase
agent names is not determined to be a “limitation” that prevents any protocol specification.
It is seen as a way to clarify what role each agent has rather than requiring the user to
know the lowercase honest/constant convention.

Related Work
Bruni et al. [4] explore a high­level security protocol specification concept based on chore­
ographies that form the base for this thesis’s work with SPC. Furthermore, they noticed
that choreographic programming has been used extensively in concurrency theory and
programming languages but less with security protocol modelling.
AnB­notation [7, 6, 8] is a well­defined choreographic security protocol specification fo­
cusing on linear security protocols. OFMC [12, 35] is an automated verification tool that
uses a highly optimized search algorithm to efficiently verify protocols in AnB­notation
against some given security goals for a limited number of sessions.
This thesis attempts to mimic some of what OFMC is capable of with AnB­notation, but
on a much smaller scale and with SPC­notation that supports choices. As OFMC is much
more mature with many features [10, 36], it is a superior tool for linear security protocol
verification with built­in automated verification by state exhaustion within some sessions.
Though OFMC always attempts to prove the security of protocols by state exploration,
as there will almost always be one more session to check with an ever­increasing search
scope, it does feature a way to generate a fixed point that can be proven with a proof
assistant.
SPC uses PSPSP [2] for security verification, which unlike OFMC can perform security
verification in Isabelle/HOL [3] without interaction from the user. Furthermore, PSPSP
can be used to prove protocol composition is safe and secure [25], which is an interesting
topic that goes beyond the capabilities of, e.g. OFMC.
The disadvantage to PSPSP, as seen in this thesis, is that trac­notation and the set ab­
straction [22] significantly limit the clarity of the protocol specification in terms of describing
correct protocol flows and the general overview of the protocol specification. Furthermore,
the equality relations required in the fixed point computation in PSPSP prevent the mod­
elling of half­key exchanges such as Diffie–Hellman.
In PSPSP, security goals are set in a raw low­level transaction notation that requires
deep insight into the modelled protocol to capture the security goals adequately. These
goals look like the internal representation of security goals in OFMC’s internal format
[37]. However, unlike PSPSP, the security goals in OFMC are set on high­level, even
supporting channel goals [10], where the concrete “attack signals” that PSPSP would
need to know explicitly are auto­generated by OFMC. Using AnB­notation and supporting
high­level security goal specification makes OFMC more accessible than PSPSP which
requires more profound insight.
OFMC use a different intruder model from PSPSP, the lazy intruder [12], whereas PSPSP
computes a fixed point based on the transitions defined by the trac­notation. The lazy
intruder model makes OFMC’s state search feasible by delaying the instantiation of values
until they are needed. With the lazy intruder model, OFMC can produce more readable
attack descriptions than PSPSP.
The extreme optimization for the concrete use case makes OFMC very good at verifica­
tion of linear AnB­notation security protocol specifications, but those optimizations do not
directly translate to a broader set of protocols, thus where tools such as PSPSP come into
the picture. Many other tools or protocols can be used for security protocol verification
Secure Choreographies in PSPSP/Isabelle
79
==PAGE:88==
[38, 39, 40, 29, 22], to mention a few, but the details of how they work go beyond the
scope of this thesis.
8.1
Outlook: Sound Cryptographic Assumptions?
Formal verification of security protocols, like the methods discussed in this thesis, ab­
stracts from the concrete cryptographic implementations to focus on the core issue of
sending messages. In real life, correct cryptographic techniques are integral to protocol
implementations. A protocol proven safe is only safe if the underlying cryptography is
safe. With the rise of quantum computing, many protocols may not be secure some time
in the future.
A strong argument for modelling the cryptographic properties of encryption is that they can
be easily exchanged for a safer implementation should one method be proven unsafe, e.g.
DES with AES, or RSA with ECC. While not always easy, due to low­power devices often
using hardware acceleration, it is often easier to change the encryption algorithm than to
change a TPM.
TPM is notorious for a significant promise of trust but has repeatedly proven to be less
secure than assumed, and a recent master thesis from DTU [28] shows that it is fea­
sible to interfere even with the latest version of the TPM. Is it as easy to change a TPM
should it prove unsafe? Maybe and maybe not. This outlook quickly goes towards trusted
computing, a significant research area.
The point is: It is essential to remember the abstractions and assumptions made when
working with any verification tool.

Future Work
There are three main areas of future work which are presented in this section: Different
kinds of suggestions for improvements to the structures around PSPSP, extensions to
and soundness of the HSM­model, and lastly, improvements and extensions to SPC.
9.1
PSPSP
Using PSPSP has highlighted some areas with potential for improvements:
• User experience and error messages.
• Interpretation of the fixed point.
• Security goal specification guidelines
All these areas are suggestions that might be subject to wrong assumptions, making the
suggestions below irrelevant.
9.1.1
Error Messages
Formal verification in a proof assistant is prone to be much slower than running an opti­
mized program such as OFMC. However, not all elements of the verification process in
PSPSP require Isabelle. Some checks performed by PSPSP are simple but give less
obvious error messages that are often relatively slow to show up, especially for an itera­
tive workflow. Understandably, correctness has been the focus when designing PSPSP,
which means that the infrastructure around PSPSP has room for improvement that could
improve the user experience. Some suggestions are:
Well-formed Trac-notation
PSPSP will attempt to prove the security of a protocol even if it is not well­formed. Com­
puting proof for a protocol that is not well­formed can take a long time and be misleading
if one does not realize this issue. A long computation time could indicate that whatever
happens, at least the protocol is accepted, and PSPSP attempts to prove it. It does not
seem like an unreasonable assumption that a protocol prover would not try to prove a
protocol known to be nonsense.
As a result, it takes surprisingly long to show a protocol that is not well­formed is unsafe. A
suggestion could be only to prove the security of well­formed protocols by default unless
an override option is set.
Attack in a Fixed Point
Like with well­formedness, PSPSP attempts to prove a fixed point covers a protocol, even
if the fixed­point computation contains an attack. An attack means there is a counterex­
ample. Proof of security is to prove that there is no counterexample. To prove that a
counterexample exists, based on the counterexample itself, seems like a waste of time.
The suggestion here is the same as with well­formedness: Only to attempt to prove the
security of a protocol if the fixed point does not contain an attack unless an override is
set.
Improve Error Messages
In the case of a trac­notation protocol that is not well­formed, PSPSP usually does not say
anything about what part of the protocol is not well­formed, just that it is not. It requires
Secure Choreographies in PSPSP/Isabelle
81
==PAGE:90==
manual checking and insights into how PSPSP works to determine which part of the pro­
tocol is not well­formed. One could argue it should work the other way around; if the error
messages are helpful, a user could learn how PSPSP works by learning from mistakes.
An example of a breach in well­formedness in trac­notation that does not give a clear
error message in PSPSP is if, e.g. pk is used with both a value and enumeration type as
mentioned in section 3.6.
9.1.2
Fixed Point Computation
The fix point computation can take a long time depending on how many possible states
a trac­notation protocol can specify. Of course, the fixed point is not proof of security,
but getting a fixed point without an attack indicates that a protocol might be safe. And
getting a fixed point with an attack signal is a good indication of an attack. The current
computation seems limited to a single CPU thread, with no attack trace shown by default.
Including documentation of print_attack_trace command in the official manual could
increase the usability of PSPSP.
Furthermore, it may be possible to implement the computation in a compiled program­
ming language that supports concurrency1, maybe by direct conversion or export from
the current source code. Improving the performance of the fixed point computation is not
trivial. However, using Isabelle as an interpreted functional language is not optimal.
9.1.3
Goals
Setting goals in PSPSP can be difficult. For example, as a part of this thesis, it was
attempted to prove the security of NSL with a weak authentication goal:
attack_weak_auth(A:honest, B:honest, NA:value)
NA in end(A,B)
NA notin start(A,B)
attack.
Here start is a set where A saves the NA they create and end is a set where B save the
NA they accept and who they accepted it from. If B agrees with a value that they think is
NA from A, but A did not create it, there is an attack.
The attack transaction meant the fixed point computation took a long time and seemed to
crash or get stuck. All attempts to change the attack definition to compute a fixed point
have failed thus far. It is possible that the number of possible transitions becomes too
large, and open­ended authentication goals such as the one described above are not
feasible in PSPSP.
The methods for writing an authentication goal for NSL could likely be applied to other
protocols and is worth investigating. As the weak authentication goal was troublesome to
define, a goal for strong authentication was not attempted.
9.1.4
Summary
It can be overwhelming to start using PSPSP, primarily due to the attack descriptions that
easily lead to false positives and the fact that simple error messages are hidden, as men­
tioned above. More guidelines for attack transactions, improvements to error messages
and the user experience might help.

9.2
Extended HSM Model
The HSM model used in the envelope protocol in this thesis is heavily simplified. A HSM
is used for purposes other than just the encryption of messages, as seen in the envelope
protocol. While it was not relevant to this thesis to expand the HSM model, possible
extensions could be:
• Supporting more HSM states or functions, extending the HSM API
• Modelling boot and reboot of an HSM
• Further work on HSM models where multiple messages can be encrypted with the
same encryption key.
There is no way to safely “delete all” values from a set in PSPSP, which would be the
typical way to “forget all” by rebooting.
An idea for overcoming this limitation in PSPSP could be where the HSM keep track of
an abstract value that can be invalidated instead. There must only be one valid abstract
value. Before the module is booted, the set has no value, and the HSM cannot be used.
Every action and value is bound to this value, e.g. via private functions. When the HSM
is rebooted, a new abstract value is constructed, and the old one is put into a set of invalid
values. This idea is yet to be tested as modelling whether the HSM was booted was not
a core goal of this thesis.
For the third option, a way to view the TPM is like modelling a box that can contain multiple
letters. All letters become available if opened, but the box is tamper­evident, like an
envelope. The concept is like a charity money collection plastic bin or piggy bank. Money
can come in, but the containers must be destroyed (tamper evident) if the money is to be
taken out.
9.3
Soundness of the HSM Model
The intruder model in PSPSP is based on the Dolev­Yao model [9], which cannot univer­
sally represent all one might want to model with regards to cryptography [41, 42]. Despite
the limitations, the model can provide beneficial results.
This thesis implicitly claims that the HSM Model presented in section 3.1 is sound by ar­
guing that it is equivalent to the efforts in [4, 27, 26]. However, argumentation against a
single case cannot tell if a model is generally sound. Further work in this area could be
exciting since disconnecting the concrete implementations of a HSM could make it pos­
sible to conceptually prove protocols that depend on HSM modules, e.g. Secure Boot,
without having to resort to modelling a complex TPM module. This thesis author is un­
aware of any research that conceptually proves the safety of standard protocols that uses
an HSM as a part of the protocol specification instead of focusing on concrete HSM/TPM
implementation­specific details ­ much like Dolev­Yao [9] did with cryptography.

Conclusions
PSPSP can be used to prove protocols with choices, and API­calls are safe across an
unbounded number of sessions for a constant set of agents. As a motivating example, a
simplified envelope protocol has been modelled in trac­notation for PSPSP, which proves
it safe.
With SPC, it is possible to make a choreographic protocol specification that contains
branches and conditional flow. With the built­in translation to PSPSP, Automated veri­
fication of protocols in PSPSP is more accessible as specifying them as choreography is
more straightforward than trac­notation. With automated projection, errors that originate
from manual projection are less likely to happen during modelling.
It is demonstrated that NSL, NSPK, and the envelope protocol can all be described in SPC
and give useful PSPSP outputs. These show the expected results when proving security
with PSPSP in Isabelle/HOL.
As PSPSP is used for the security protocol proofs, SPC inherits a set­based abstraction
that limits how a conditional flow can be described and a low­level transaction style attack
goals. These attack transactions make it challenging to describe high­level goals such as
authentication. Furthermore, the equality classes that PSPSP use prevents the modelling
of protocols with half­key key exchange.
SPC relies on PSPSP for proof of security and the intruder model while keeping the spec­
ification transparent to PSPSP. The reliance on PSPSP means that SPC still requires
in­depth knowledge of PSPSP to use.
This thesis has shown that a high­level AnB­like security protocol specification can be
translated into PSPSP with beneficial results and encourage more work in this area.

Identifiers ⟨Ident⟩are unquoted strings beginning with a letter, followed by any combina­

words with non­letter characters are called symbols and are treated differently from those
similar to identifiers. The lexer follows the rules familiar to languages like Haskell, C, and
