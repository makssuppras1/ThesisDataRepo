Abstract
Migrating from a monolith architecture to a microservice architecture in order to
modernize a system seems to have become popular in the recent years. However,
the benefits of this migration does not appear to have been sufficiently verified. The
goal of this master’s thesis is to evaluate whether or not it is beneficial to migrate to
microservices.
A Systematic Literature Review was conducted to design a methodology for eval-
uating this architecture migration. To further validate the findings of the literature
review a small survey was conducted, and existing approaches was analyzed.
The methodology consisted of designing a reference system and implementing it as
a monolith system and as a microservice system for comparison. These two systems
was then benchmarked and compared using the following metrics: latency, through-
put, scalability, CPU usage, memory usage, and network usage.
Two benchmarking experiments was conducted: one small-scale experiment using
consumer-grade hardware, and one large-scale using cloud services.
These experiments showed that the monolith system performed better overall, with
the exception of the scalability metric where the microservice system performed up
to 6 times better than the monolith system.
These results show that it is better to be using the monolith architecture for small
to medium size systems, particular for smaller systems such as the one built for this
thesis. It also shows that for systems with a need for supporting a large amount
of the concurrent users, the microservice architecture is the better choice due to its
superior scalability.

CHAPTER 1
Introduction
When designing a new system, or upgrading an existing one, one of the first consid-
erations the systems architect have to consider is how to structure the system. This
raises the questions: how important is software architecture for a system? And how
can we determine whether or not the chosen architecture was a good choice?
The articles [Buc+18] [Maz+18] describes the process of the modernization of an
old system, by migrating from a monolithic architecture to a microservice architec-
ture based on the hypothesis that microservices performs better than a monolith. The
migration turned out to be seemingly successful, but one of the major critique-points
of the articles was the lack of verification of the migration being beneficial.
The
system was proprietary and thus the verification was infeasible both due to practical
and legal reasons. To perform such an verification it would be beneficial to have a
self-built and, preferably, open source, system. This means that in order to evaluate
the migration two pieces are needed: a methodology for comparing two systems and
two reference systems for comparison.
The lack of migration evaluation presents a gap in this field which this thesis will
attempt to further investigate. The goal of this thesis is to perform such an eval-
uation in three steps. First a Systematic Literature Review(SLR) is conducted on
how to evaluate software architectures through benchmarking. The second step is
to design a reference system and implement it as a monolith and as microservices to
simulate a migration. The last step is to use the methodology in step one, and reflect
on both the outcome and the methodology itself.
Some of the core concepts used in this thesis are explained below.

In order to compare two systems in a structured way, a methodology must be de-
fined. This thesis will evaluate the two systems in a quantitative manner and thus
benchmarking seem like an obvious choice. Benchmarking is already used today in
computing, particularly for hardware such as CPU1 and GPU2. A benchmark is de-

Software Architecture
Software Architecture is defined by [Sof+00] as
”The fundamental organization of a system embodied in its components,
their relationships to each other, and to the environment, and the princi-
ples guiding its design and evolution.”
Meaning that an architecture defines a system’s structure of components and how
they interact internally and externally.
Many software systems are traditionally built as client-server system with having one
or multiple clients accessing a single server for business logic and data persistence,
called a monolith4. A monolith is typically a single artifact containing all the system
components, which connects to a single database. This makes early development easy
by having all the code and functionality in one place, and the components can freely
interact with each other. It also makes it easier having all the data in one database,
as it enables complex queries such as multiple joins on the database. Deployment also
becomes simple as it is only a single artifact which needs to be deployed to a server.
However, as the system grows the deployment and scalability becomes more difficult
as even a small code change requires the whole monolith to be re-deployed, a process
which becomes cumbersome after reaching a certain size. Scalability also becomes a
problem since creating new server instances to handle higher loads requires the entire
server to be instantiated rather than the components experiencing high load. The
monolith architecture also introduces a technology lock-in to a certain degree, as a
monolith made with Java would have great difficulties adding a feature which would
be made better with e.g. Python, without using some sort of Inter-Process Commu-
nication (IPC)5 or building a new application for that feature.
It was problems like the ones mentioned above which gave rise to the Microservices
[Dra+17] Architecture, which is a decentralized approach to web services, and the

In this chapter, a SLR is conducted to identify benchmarking methodologies and
metrics within the field of software engineering. The results showed that measur-
ing benchmarks and performance type metrics was the most frequently used in the
surveyed papers.
2.1
Motivation
The goal of this SLR is to identify benchmarking methodologies which can be applied
to measure the performance of software architectures. To achieve this a Systematic
Literature Review is conducted following the guidelines of [PVK15]. This will help
identifying methods and metrics to compare monolithic and microservice architec-
tures, to to test the hypothesis of microservices performing better than monoliths as
mentioned in Section 1. To better understand the existing benchmarking practices
the following research questions were made:
Research Question 1: What benchmarks exists within software engineering?
Research Question 2: Which benchmarks metrics are currently used within soft-
ware engineering?
Answering these questions will help determining a good approach to comparing sys-
tem of different software architectures.
2.2
Search Strategy
The PICO [Pet+08] method was used to chose which search strings to query academic
databases based the the research questions mentioned above:
Population: The population for this article is benchmarking which are used within
the field of software engineering.

2.4.2.1
Latency
Latency is the time delay measured between a response and request, this is particular
important for communication between a client and server. This metric is primarily
impacted by network speed and latency + the amount of time it takes for the server
to process the request. This metric was the most popular in the examined articles,
and with good cause as it relatively easy to measure, and it has a big impact on the
end-user experience.
2.4.2.2
CPU
CPU Utilization is another popular metric, and understandably so. CPU usage is one
of the indicators of how well the software system is able to utilize the host machine’s
resources.
It can also be used to indicate code’s efficiency, by comparing similar
functionality different ways and comparing them, a lower CPU usage is better.
2.4.2.3
Throughput
The throughput of a system is how many actions/requests a system can process given
a workload and hardware resources. E.g. a system that can process 100 requests per
second performs worse than a system that can process a 1000 requests per second
given the same resources.
This metric is related closely related to Latency and
Scalability.

2.4.2.4
Network
The Networking metric indicate how many networking resources a system uses. This
becomes particular important when you have a set of Microservices which all commu-
nicate together over HTTP/HTTPS, and thus a high usage of network can impact
the Latency of a system.
2.4.2.5
Scalability
When building a system to serve a few concurrent users, scalability might not be a
concern. However, when a system must handle a plethora of users, it is important the
that the system is built handle the extra work, that it can scale-up. The Scalability
of a system is typically measured by how many requests can be processed while
staying below a set Latency threshold e.g. how many requests can a system process
before the latency rises above 200 ms. When it comes to Scalability in the cloud,
the term Elasticity is often used when talking about how fast a cloud system can
increase/decrease computing resources to allow a system to scale up or down to meet
demand.
2.4.2.6
Memory
Memory is similar to CPU usage as it indicates well much of the systems RAM
is being used, and thus the efficient of the system/source can be determined. This
metric is becoming more important as virtualization such as virtual machines and
containers becomes more prominent as these technologies are generally more memory
demanding.
2.4.2.7
I/O
I/O (Input/Output) describes how much the system utilizes its host’s input/output
devices, e.g. read/write operations to disks or network cards.
2.4.2.8
Fault Tolerance
Fault Tolerance is used an umbrella term for the following metrics: Failure Rate /
Error Rate, Transaction Success, Resilience/Reliability, Mean Time Between Failures,
Uptime. In general these metrics how a system handles something going wrong. This
is usually measured by investigating the amount errors, their frequency, and their
impact on the system e.g. can a web server recover from an exception being thrown
or is a reboot required?
2.4.2.9
Code Quality

tem design. Through the SLR a bunch of different ways to analyze source code has

Interface Quality Interfaces are often used in system design, both on a compo-
nent level and on a source code level, to clearly segregate logic and responsibility.
However, these interface can be difficult to design in the right size. Thus a tool have
been developed to perform static analysis to give a score on how well designed an
interface is.
Coupling is a metric derived from the source code which describes how closely con-
nected code/modules are. It is generally desired to have a low coupling (which results
in a high Cohesion). One way to compare two systems could be to see which system
has the lowest coupling.
Cohesion describes how well data and logic inside a component belongs together.
A high cohesion indicates that a system is divided into some well-bounded modules.
Complexity, also known as Cyclomatic Complexity, is the measure of how many
different paths there is through a piece of code. The higher amount of paths through
code, the more complex the code is to understand is it can result more different out-
comes and state changes. Thus a lower complexity score is preferable.
Changeability is a metric which describes how easy it is to change a system. This
is particular important in a world with agile becoming more popular, and thus it can
be expected to make frequent changes to code. This means that a system with a high
changeability score is often preferable.
Quality Index The quality index is an metric which describes the quality of a
software architecture. In general it examines characteristics, quality indicators and
anti-patters.
For characteristics it checks how the internal communication is a designed, and

2.4.2.11
Consistency
Data consistency was measured by two different ways in the examined articles:
staleness and ordering. In a distributed database there is often multiple replicas of
a piece of data for performance, backups etc. Staleness describes how far behind a
piece of data from one database is compared to its replicas.
Ordering is a metric that describes whether there consistency in the order of events
occurring, e.g. will the order of events happening client-side also reach the server/be
processed by the server in the same order.

Hardware level

miss.

Cost

The microservices are also built with ASP.NET Core Web APIs and Entity Frame-
work, as it was intended to keep performance variables such as different programming
languages out of the way. It was, however, intended that one of the microservices
should be implemented in Java, a language very close to C# both performance and
language-wise, but due to time constraints it was not implemented (see section 6.2).
Each microservice has its own dedicated SQL Server database allowing each service
to scale-up and down their different data stores as need arises.
The microservices are run in Docker6 containers, yielding a Docker image7 per mi-
croservice which are stored in regististries such as Docker Hub8 and Azure Container
Registry (ACR)9. This allows for easy distribution of the microservices services, by
allowing them to be running in the same environment across different machines and
hosts.
Docker containers can be run on their own without any orchestration tools, but it can
quickly become a difficult task to manually keep the containers up and running in
the right condition. To fix this, Kubernetes is used to orchestrate the microservices
allowing for easy horizontal scaling, load balancing, health checks, and much more.
Orchestration
As mentioned above, the microservices of LibraryService is orchestrated through the
usage of Kubernetes. Each service has its own deployment10 which is responsible for
meta data, pulling the Docker image from the registry, internal and external commu-
nication and replication(ensuring multiple pods11 are running).
Each deployment is then exposed to external network through a Kubernetes service12,
which routes external traffic to Kubernetes pods, which can handle the request. Ku-
bernetes offers load balancing13 [ISB86] which attempts to ensure the load is even
spread out among the replicas of the microservices. Each deployment is also set up to
perform a periodic liveliness check. If a replica fails a livenesness check Kubernetes
will shut down the pod, and create a new one.
Locally the load balancing is achieved by using Ingress14 which is an alternative way
of handling load balancing by routing the external requests to inside the Kubernetes
cluster.

The first initial challenge was to get the web API up and running on Docker. This

Kubernetes
Getting the local Kubernetes cluster up and running was by far the biggest develop-

The first challenge was to get Kubernetes it self up and running. Initially it was
attempted to use minikube34, a light-version of Kubernetes, however after multiple

local machine. However it turned out that it was possible to run Kubernetes locally
through Docker Desktop by enabling some settings.
Once Kubernetes was up and running the next challenge appeared, how to get a
Docker image to run in the cluster. After going through tutorials and documentation,
the author was able to successfully created a deployment file which Kubernetes used
to created a pod containing the desired image. However, exposing this pod through
a service turned out to be more challenging as the built-in load balancing worked
for one Kubernetes service, which represents one microservice. But when the 3 other
Kubernetes services got setup the load balancer could not expose those. So a mi-
nor rework was needed to achieve load balancing by installing a third party ingress
controller which could handle the routing and load balancing. This also required a
reconfiguration of the Kubernetes services to be of the type NodePort rather than
load balancer.

CHAPTER 5
Benchmarking
In this chapter, the previously selected metrics will be used to compare the 2 versions
of LibraryService to see if there are any notable differences.
2 benchmarking experiments are performed: one running locally and one in the cloud
which is an environment closer to a more real-life system.
The experiments show that the monolith system in general performs better than the
microservice system on every metric except for scalability.
The data from 2 experiments can be found at GitHub1.
5.1
Metric Measurement
The metrics during these experiments will be measured the following way:
Throughput is measured by the amount of successful requests completed per second
over a period of time. In this paper a successful request is defined as the HTTP
response code 2xx2 and with a latency lower than or equal to 200 ms. The
higher throughput the better, the better the system.
Latency is measured by the difference in time between the client making the HTTP
request and the client receiving the HTTP response. The lower the latency, the
better the system.
Scalability is defined as the ratio of X% more throughput given Y% more resources
[BWT17]. The higher the scalability, the better the system.
CPU, Memory, Network These 3 metrics are all hardware-related metrics, which
is measured by monitoring the host system and its CPU, Memory, and Network
usage. In general the more efficient usage of hardware resources the better.

5 Benchmarking
The benchmark uses 2 synthetic workloads: a simple and a complex workload. A
workload in this benchmark is defined as a series of HTTP requests at a steady rate,
e.g. 100 requests per minute.
The workloads are being generated and executed by Apache JMeter3 (JMeter) on
the client. JMeter uses a Constant Throughput Timer4 to generate the approximate
amount of desired requests over a minute. JMeter logs the sent requests and their
data e.g. http response, latency, etc. which are used to determine the Throughput,
Latency and Scalability metrics. The server uses Window’s built-in monitoring app,
Performance Monitor, to log CPU utilization, Memory usage and Network traffic.
The simple workload consist of 2 actions: get data about a specific book and create
a new user. Getting the book information results in a HTTP GET request (resulting
in a read operation in the database). Creating a new user results in a HTTP Post
request (resulting in a write operation in the database).
The complex workload consist of getting information about a specific loan and
creating an completed order (creating an order and a physical book). The retrieval of
a loan results in a HTTP GET request requiring a join across multiple tables for the
monolithic version, and multiple HTTP GET requests for the microservice version.
The create order results in a HTTP Post request which results in write operations on
multiple tables. For the microservice it requires communication between order service
and book service through a message queue to achieve the creation of the order and
physical book.
These workloads will be applied to the server running the monolithic and microservice
version, separately, meaning that only one version of LibraryService is running at a
time. Each version of LibraryService is benchmarked at multiple amounts of requests
per minute. The samples are in steps of 2500 requests per minute until the average
latency rises above 200 ms indicating the system’s maximum throughput (also ref-
fered to as its break off point). The steps of 2500 requests per minute was chosen as a
compromise between getting enough samples to find patterns, while not making the
experiments too time consuming to perform. For each test iteration the Throughput,
Latency, Memory Usage, CPU Utilization, and Network Usage is measured.
The monolith system is run in release mode through the dotnet run5 command in
PowerShell6.
The microservice version is deployed on a local running instance of Kubernetes through
Docker Desktop. Each microservice is replicated to 3 running instances, using a Repli-
caSet7 configured to run 4 CPU cores and 3GB RAM. The microservices which were
being used during a workload were configured with 9 pods, and unused services were
configured with 3 pods.

6.1
Discussion
The results indicates that the monolith system performs better overall, as it performs
better in all metrics except for scalability. This outcome can possibly be explained
by a number of reasons: the reference systems are not enterprise sized systems, Ku-
bernetes/Docker overhead, and the hardware configurations used for experiments.
The reference systems are relatively small compared to large enterprise systems con-
sisting of millions of lines of code and heavy use of libraries/other dependencies. This
means that spinning up additional instances of a monolithic web server often comes
with a bigger resource costs than the microservice equivalent. However, with both
reference systems being fairly small, there is not a big difference in the resource cost
of scaling up.
This leads to the second point being that Docker and Kubernetes comes with an
extra resource overhead, meaning that potentially the overhead of spinning up an ex-
tra microservice instance might be more costly than spinning up an extra monolithic
server instance.
The increased overhead becomes visible in experiment 1 where the resource usage
is higher to provide a similar, or even lower, throughput for the microservice com-
pared to the monolith. The higher resource usage most likely lead the SQL server to
bottleneck [Wes13] as it did not have access to enough system resources to properly
perform. The underperforming database is also a likely reason for the microservices
in experiment 1 performing worse compared to having a separate machine as a dedi-
cated database server like in experiment 2.
The resource usage of experiment 1 was as expected considering that Kubernetes is
made for a cluster rather than a single computer as mentioned in Section 1. However
the overhead for managing a cluster comes with better scalability as the microservice
scaled 3 times better in experiment 1 and 6 times in experiment 2 compared to the
monolith system.
This indicates that despite the increased resource cost, the mi-
croservice architecture is worth using if the system needs to support a large amount
of concurrent users.
==PAGE:64==
56
6 Conclusion
Experiment 2 was a simpler version of experiment 1, since it used a light version
of the simple workload. This setup gives a less nuanced picture of the overall perfor-
mance of the system, but it allowed to test the scalability better with more and faster
hardware while handling a much higher load.
Based on the results from these experiments it would seem that a smaller to medium
sized monolith generally uses less system resources while performing better than its
microservice counterpart. However the microservice architecture redeems itself with
its great scalability, allowing it to easier scale up. It also showed how a microser-
vice migration can go wrong, as the complex microservice workload performed badly
across the board. In the case of the complex microservice it means that it is impor-
tant to design the microservices with well bounded contexts to avoid communication
between services unless necessary.
The chosen metrics seem to be good choices for evaluation with particular Latency,
Throughput and Scalability being important to evaluate how well a system performs.
CPU, Memory and Network are also good performance indicators, but these serve
just as much as a way to explain a system’s Latency, Throughput and Scalability.
6.2
Future Work
While the results from the two experiments mostly are as expected, it would be in-
teresting to measure the metrics for systems which are closer to the kind of systems
used by companies.
To make the system closer to a real-life system, the reference system would benefit
from a series of improvements. The first improvement is the sheer size of the system,
adding bigger and more components with more dependencies would make it closer an
enterprise system.
This bigger system would also benefit from having a bigger and more complex database
with real, or close to real, data. This would impact query time and thus latency, but
it would make it closer to a real-life system.
To better simulate the possibilities of a polyglot microservice architecture it would
also be beneficial to implement one or more services in another programming lan-
guage than C# such as Java or Python.
If the local experiment is desired to be repeated it would be beneficial to have a
more professional setup with server-grade hardware, wired connections and a sepa-
rate machine for the database server. Ideally for clustering there should be multiple
machines for Kubernetes nodes, or a server machine powerful enough to run multiple
virtual machine nodes.
==PAGE:65==
6.3 Conclusion
57
A final important improvement to make the system more real-life-like is adding se-
curity; currently the system neither uses encryption or authentication. Ideally all
communication would go through TSL1. Additionally the system should require au-
thentication e.g. through a JSON web token2.
To further test the scalability it would be interesting to add more and/or more pow-
erful nodes to AKS in order to see how the microservice system scales at 10, 50 or
even 100 nodes.
Another scalability measure to be measured would be to use auto-scale for App Ser-
vice instances and for Kubernetes rather than the current manual tuning. However
this would also require longer test runs to let the auto-scale scale the system to a
proper size.
6.3
Conclusion
In this thesis different experiments have been used to evaluate whether or not the
migration from a monolith architecture to a microservice architecture is beneficial.
To perform this evaluation, a methodology was constructed based off a Structured
Literature Review, performing a small-scale survey amongst IT professionals, and
looking at existing approaches. This led to a methodology consisting of benchmark-
ing a monolith and microservice system and comparing them based on the following
metrics: Latency, Throughput, Scalability, CPU, Memory and Network.
It was decided to build two new reference systems for the benchmarking. This led to
the creation of the Monolithic- and Microservice version of LibraryService.
Two benchmarking experiments were performed: one using a local desktop PC and
one using the cloud. In the first experiment the monolith came out ahead, while the
microservice struggled, particularly with the complex workload. The microservice
system did however in the best case have a 3 times higher scalabilty ratio compared
to its monolith counterpart.
In the second experiment, using the power of the cloud, the monolith still had better
performance, but the microservice system once again had a favorable scalabiltiy ra-
tio. In this experiment the microservice system had a 6 times higher scalability score
compared to the monolith system.
All in all, the monolithic architecture appear to perform better for small to medium
sized systems as used in this thesis. However, the much higher scalability ratio of
