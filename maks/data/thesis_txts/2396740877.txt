Summary
This thesis presents a domain-speciﬁc language. This language successfully lets
domain experts deﬁne and analyse various pension contracts. This includes con-
tracts of the Danish pension system. These contract are, "Aldersopsparing", "Rate-
pension" and "Livsvarig livrente". The primary focus of the analysis is valuation
of these contracts.
The language presented utilises a robust and scalable architecture. This ar-
chitecture allows for formulation of a wide range of pension contracts. This in-
cludes many possible future pension contract deﬁnitions. This is achieved using
the combinators presented in [Jon+00]. This thesis gives a introduction to this pa-
per describing the syntax and semantics of the language presented.
This thesis also explore many categories of domain-speciﬁc languages. The
language presented in this thesis is in the category of Haskell-embedded languages.
The embedding used is a combination of a shallow and a deep embedding. This so-
lution allows for easier and more maintainable addition of new language features
while allowing for multiple pension contract interpretations. We test this by deﬁn-
ing various pension contract formulations in concrete syntax. We then evaluate
these contract formulations using multiple valuation models. Furthermore, the
language presented also allows for extensions in form of a standalone language.
The language presented also incorporates reﬁnement types. This is achieved
using Liquid Haskell [Liquid Haskell]. This technology serve as a domain-speciﬁc
language. This language allows domain experts of the pension domain to turn
informal pension contract properties into formal ones. Additionally, this also al-
lows the domain experts to abstract away any unnecessary details about run-time
errors. In this thesis we explore in detail how exactly this can be done.
The concepts and technologies presented in this thesis are state-of-the art and
have proven to be very useful. This includes Liquid Haskell and the language pre-
sented in [Jon+00]. These are both widely used in both the industry and the con-

CHAPTER1
Introduction
This thesis introduces a Haskell-embedded domain-speciﬁc language for deﬁning
and analysing various pension contracts of the danish pension system.
This chapter describes the motivation for the thesis as well as its objectives.
Furthermore, we deﬁne the scope of the thesis and its structure.
1.1
Motivation
Planning retirement is extremely important. It is in principle possible for individ-
uals to save up to retirement of earned and taxed funds. However, this raises the
question - can one ensure that these savings will lasts throughout one’s life? Due
to this, many, on the other hand, see an advantage in a pension system. A pension
system is a fundamental part of the welfare state, meant to provide an economic
security in the third age by describing some set of beneﬁts a person receives after
retiring from the labour market. The main principle is that income is set aside
during an employee’s employment years, deferring its taxation to a lower rate. The
set of beneﬁts can cover a wide range but the central purpose of a pension system
is that it should set a minimum living standard for all retired citizens - regardless
of labour history and health conditions. The pension system should in addition
to this also ensure that the living standards of an individual are not drastically de-
crease due to retirement. To ensure this a so called pension plan is made which
can cover a lot of fundamental questions like how do you ensure that a pension
lasts throughout any person’s life and how do you for instance secure your spouse
in case of early death?
The main principles of pension plans are reasonably clear, yet many aspects
have to be taken into account. Typically these aspects are regulations in the tax
system that may cause the taxation of the pension to be high. These regulations
are complex and will additionally be continuously changing, for instance due to
==PAGE:12==

political changes. This makes keeping up with the legislative developments a dif-
ﬁcult and time consuming task. As a consequence, individuals who wish to pick
a good pension plan often seek help from pension experts. Pension experts help
individuals select pension plans and put together different sorts of pension con-
tracts that make up these pension plans. Pension contracts are simply pension
formulations that follow a speciﬁc set of rules. The pension experts then anal-
yse these pension plans using clever calculations to check that they fulﬁl the law
and customer wishes. The clever calculations are done automatically with pen-
sion software, which is being developed by software experts based on the pension
experts interpretation of the legislation. However, due to the incredibly complex
nature of the human language the pension software might not develop into exactly
what the pension experts are trying to express.
Lost in translation is a phrase used when word-for-word translations often are
impossible and this might for instance occur when translating back and forth be-
tween domains where technical terms do not have direct equivalents. As a conse-
quence, it seems likely that some principle of a regulations will get omitted when
the software experts and pension experts in a joint effort develop and extend the
pension software. This will result in incorrect calculations and perhaps it would be
better to make use of a domain-speciﬁc language as an alternative to the pension
software. One could then imagine the domain-speciﬁc language cutting off the
intermediary and have the pension experts do all the work. Additionally, on can
imagine the domain-speciﬁc language becoming some mutual language between
the pension experts and software experts that simpliﬁes their communication and
increase their productivity. This thesis addresses this challenge and we provide
a Haskell-embedded domain-speciﬁc language. The next section describes the
main objectives of the thesis.
1.2
Objectives
The ﬁrst objective of this thesis is to analyse the danish pension system and ex-
plore the possibilities for a domain-speciﬁc languages. This domain-speciﬁc lan-
guage should let domain experts deﬁne and analyse possible danish pension con-
tracts. This domain-speciﬁc language should also be robust and extensible. The
robustness and extensibility is needed as we can only make guesses about future
changes in the danish pension domain. The domain-speciﬁc language will be in
the genre of embedded languages and we will analyse advantages and disadvan-
tages for different kinds of embeddings.
==PAGE:13==
1.3 Scope
3
The second objective of this thesis is to provide a implementation that for-
malises the ideas from the analysis. That is, we want to provide a rich language
with precise notation which lets domain experts describe various pension con-
tracts. This should primarily include current and possible future contracts of the
danish pension domain. This language should be useful for communication be-
tween domain experts and software experts. This language should also allow do-
main experts to formalise pension contracts on their own. In addition to this, the
language should provide means of automatic processing. This includes a pension
contract valuation.
The third objective of this thesis is to provide various pension contract formu-
lations in the implemented solution. This should showcase the extensibility and
robustness of the language.
1.3
Scope
Implementing a new domain-speciﬁc language is a task of a signiﬁcant size. This
can involve many separate sub-tasks. This could include implementation of a
parser and a compiler, and perhaps an interactive editor too. Furthermore, nu-
merous conditions must be taken into account. This ﬁrst and foremost involves
maintenance of the language. Furthermore, domain knowledge is needed to pro-
vide a domain-speciﬁc language. To achieve the described objectives within the
given time frame certain decision are made.
That being said, this thesis implements a Haskell-embedded domain-speciﬁc
language. The focus of the language is to deﬁne a extensible and reliable architec-
ture for deﬁning and analysing various pension contracts. This includes a subset
of pension contracts of the danish pension domain. This subset is "Ratepension",
"Aldersopsparing" and "Livsvarig livrente". This also includes a large set of con-
tracts that can potentially evolve from the danish pension system. These pension
contracts can be analysed in many ways. We provide means of valuating the con-
tracts using several methods.
Secondly, this thesis only cover one central purpose of the danish pension sys-
tem. This is personal savings. Insurance, health-care and multi-party pensions
are not covered.

1.4
Structure
The remainder of this thesis is structured as follows.
Chapter 2 gives an informal speciﬁcation of the different pension contracts of
the danish pension domain. This includes "Ratepension", "Aldersopsparing" and
"Livsvarig livrente".
Chapter 3 describes the general distinctions for domain-speciﬁc languages.
We introduce standalone and embedded domain-speciﬁc languages, as well as
shallow and deep embeddings. We also present several examples implement in
Haskell and showcase how some embeddings can be problematic.
Chapter 4 introduces a deep embedded domain-speciﬁc language for deﬁning
various ﬁnancial contracts. We will go into depths with the syntax and semantics
as this domain-speciﬁc language will serve as a foundation for a shallow embed-
ding. This shallow embedding will allow pension experts to deﬁne and analyse
various pension contracts.
Chapter 5 deﬁnes a scalable architecture by combining shallow and deep em-
bedded languages to a language for the danish pension domain. We test the ar-
chitecture by implementing various concepts of the danish pension domain. This
includes ideas such as annuities, life annuities, lump sums and various methods
for depositing into a pension contract. We also provide various valuation methods
for these contracts.
In Chapter 6 we evaluate the work that has been done. This chapter provides
a well-grounded rationale to why the implementation of the domain-speciﬁc lan-
guage has been a success. In this chapter we also discuss some related works and
look for interesting future works.
Finally, Chapter 7 concludes the thesis.

CHAPTER3
Domain-Specific
Languages
This chapter deﬁnes domain-speciﬁc languages. This chapter also highlights the
main advantages and describe some issues that has to be addressed when intro-
ducing a domain-speciﬁc language to solve a problem. This chapter will ﬁrst give
a historical background to why we might want to introduce a domain-speciﬁc
language and then introduce some general distinctions for domain-speciﬁc lan-
guages and categories of embeddings. Finally we give some concrete examples of
embedded domain-speciﬁc languages implemented in Haskell. We also present
some problems with these concrete examples and introduce the reﬁnement types
of Liquid Haskell [Liquid Haskell]. These reﬁnement types will be used as a solu-
tion to some of the problems.
3.1
Background
Historically when developing software some domain expert will explain the do-
main to a business analyst who then might describe some requirements for the
lead developer. The lead developer might then explain this document to some
other developers, and in the end, it will be these developer’s understanding that
gets released into production [Wla17]. Unfortunately, this approach creates dis-
tance between the person who understands the problem best (the domain expert)
and the people who will implement the solution (the developers). This distance
can be fatal to the success of a project. A much better solution is to eliminate the
intermediaries and encourage the domain expert to be involved with the develop-
ers and regularly test and correct their product. However, this solution also has
the problem that the developer acts as a translator between the domain experts
knowledge and the source code. This can lead to words getting lost in translation.
==PAGE:20==
10
3 Domain-Specific Languages
A third approach is to have the domain experts and the developers and the
actual source code itself share a model. This model should not be a particular di-
agram or drawing on a whiteboard. Instead it should be some abstraction that
should simplify communication between domain expert, developers and code.
That is, we want to have code which is not cluttered with unimportant speciﬁ-
cations that draw attention away from the domain logic. This could for instance
be code concerning communication with a database or error handling, which a
domain expert does not need to know about.
The next section of this chapter introduces domain-speciﬁc languages. Theses
are abstractions that offers syntax and semantics that match the problem domain.
These kind of languages can therefore be a shared model.
3.2
Introduction to Domain-Specific
Languages
Programmatically the problem of deﬁning and analysing a pension contract can
be solved by simply using general purpose languages. This includes C and Java.
However, these languages do not solve the problem of words getting lost in trans-
lation.
Opposed to general purpose languages there are domain speciﬁc languages.
These are languages speciﬁcally tailored to a particular application area, such as
digital electronic hardware design or ﬁnancial contracts [Chr+13]. Ideally the lan-
guage should provide a level of abstraction that describes problem solutions natu-
rally and hides unnecessary detail for the user.
In the book [Fow10], Fowler deﬁnes a domain-speciﬁc language as a computer
programming language of limited expressiveness focused on a particular domain.
We will in this thesis stick with this deﬁnition.
The advantages and disadvantages of using domain-speciﬁc languages [Fow10]:
• Improving development productivity: The easier it is to read a piece of
code, the easier it is to ﬁnd mistakes, and the easier it is to maintain the sys-
tem. So, for the same reason that it is encouraged to do meaningful variable
==PAGE:21==
3.3 Flavours of Domain-Specific Languages
11
names and documentation, one might also choose to invest time in devel-
oping a domain-speciﬁc language. The limited expressiveness of domain-
speciﬁc languages also makes it harder to say wrong things in the ﬁrst place.
• Communication with domain experts: Domain-speciﬁc languages can bring
the computer languages to a level of abstraction that is understandable for
domain experts. By being understandable, domain experts can spot mis-
takes. They can also talk more effectively to software experts or perhaps the
domain experts can even learn to use the language themselves.
• Cost of building: It can be considered difﬁcult to develop a good domain-
speciﬁc language, as it requires both knowledge of the particular domain
and software development. This is a skill which most people seldom have.
Even with this knowledge it can still be difﬁcult to set the proper level of
abstraction.
• Learning curve: Ideally when using domain-speciﬁc languages it should be
simple and easy to learn. However, in reality it can be big time investment
to have to learn new domain-speciﬁc languages instead of simply knowing
one more general purpose language. Also, when a company who has build
its own language has to hire new employees these employees have to learn
this new technology. However, if the company has not build the domain-
speciﬁc language by themselves they might be able to better ﬁnd employees
who already has knowledge of said domain-speciﬁc language. In general
with a homemade domain-speciﬁc language one might also have to make
up code examples or user guides. This is needed as you will not be able to
simply Google answers or get commercially support.
3.3
Flavours of Domain-Specific Languages
Domain-speciﬁc Languages generally come in two ﬂavours. They can be embed-
ded languages or they can be stand-alone languages. Embedded Languages are, as
the name indicates, implemented in some abstraction mechanism in an existing
languages. These languages can then reuse infrastructure from the host language.
Stand-alone languages deﬁne their own infrastructure.
An example of an embedded language is the Lava hardware description lan-
guage which is a Haskell-embedded language for expressing circuits. PostScript is
a example of a stand-alone language. It is possible to implement domain-speciﬁc
==PAGE:22==
12
3 Domain-Specific Languages
languages as both embedded and stand-alone languages. This is the case for, e.g.,
the query-language SQL, which is available both as a set of library functions and
as a stand-alone notation used in interactive front ends to databases.
The two ﬂavours each have advantages and disadvantages [Mog03]:
• Embedded languages: An embedded language inherits the full infrastruc-
ture of the host language, so constructs for conditionals, looping, recursion,
etc. need not be explicitly added to the language. An embedded language
can also take advantage of any other host language features, such as libraries
and development tools.
• Stand-alone languages: The syntax can be chosen so it is natural for the
problem domain without any limitations imposed by a host language. Fur-
thermore, domains-speciﬁc consistency checks can be made at compile-
time, and error-messages can be expressed in terms of the problem domain.
It is clear that the latter is most suitable for the problem presented in this thesis.
This is because the language is intended for non-programmers. However, stand-
alone languages can be a tremendous amount of work, compared to a embedded
language, since any lexer, parser, optimizer, code generator, etc. must all be build
from scratch and afterwards maintained. Alternatively, there is a shortcut. It turns
out that a stand-alone language can share much of the engineering of a embedded
language. This can be achieved by simply parsing from either textual or graphi-
cal terms in the standalone language to terms in the embedded language [Gib15].
This also requires some work.
As a consequence of the limited time frame for this thesis, the language that
will be presented will focus on a embedded language instead. A stand-alone lan-
guage could then, at some point later, be parsed into this language.
3.4
Implementation Strategies for
Embedded Languages
Embedded domain-speciﬁc languages are generally also divided into two cate-
gories. These categories are the deep and shallow embeddings. With deep embed-
dings, terms in a domain-speciﬁc language are implemented simply to construct
an abstract syntactic structure (usually some sort of abstract syntax tree). This ab-
stract syntactic structure is then often subsequently transformed for optimisation
==PAGE:23==
3.4 Implementation Strategies for Embedded Languages
13
and traversed for evaluation. With a shallow embedding, terms in the domain-
speciﬁc language are implemented directly as the values to which they evaluate.
This bypasses the intermediate abstract syntactic structure of the domain-speciﬁc
language and its traversal [Gib15].
Concrete implementations in Haskell of both of these categories will be ex-
plored in the next section.
Advantages and disadvantages of the two categories of embeddings [SA13]:
• Deep embeddings: In deep embeddings adding new language features re-
quires changing the abstract syntactic structure. The consequence of this is
that any transformer and evaluator will also have to be updated. Changing
the abstract syntactic structure and adding new functionality can therefore
be a lot of work. As most domain-speciﬁc languages evolve the abstract syn-
tactic structure can grow quite large, making it difﬁcult to maintain.
• Shallow embeddings: Shallow embeddings work directly on the semantics
of the host language and are usually more concise and elegant compared to
deep embeddings. These languages are therefore often easier to implement
and maintain. However, working directly on the semantics of the underlying
language results in one single interpretation which is not well suited when
we want to do more interpretations. For instance it can be difﬁcult to de-
bug a shallow embedding as we cannot simply print the underlying abstract
syntactic structure.
To overcome the limitations of shallow and deep embeddings a combined strat-
egy is often used. The limitations are commonly referred to as the expression prob-
lem [Wad]. The idea is that we want to be able to do many interpretations but we
do not want to have to update the way we interpret when adding new terms in the
language.
The combined strategy will usually solve this by containing some core lan-
guage features. These will be implemented using a deep embedding and then a
shallow embedding will be implemented on top of this core language. The aim of
the core language is not to act as a convenient user interface, but rather to support
efﬁcient code generation using a shallow embedding. For this reason, the core lan-
guage should be kept as simple as possible. If the deep embedding has reached a
certain level of completeness, new extensions will not require any new changes to
the deep embedding [SA13].

3.5
Implementing Embedded
Domain-Specific Languages in Haskell
It turns out that functional programming languages like Haskell are particularly
well suited for hosting both shallow and deep embedded domain-speciﬁc languages
[Gib15]. This section introduces ways that we can implement both. First we show
a deep embedding using algebraic data types. We will then compare this to a shal-
low embedding. Finally we will combine the two to a more robust language.
Consider a very simple language of arithmetic expressions, involving integer
constants and addition. This can be captured in a deep embedding by the follow-
ing algebraic data type:
data Expr = Val Integer
| Add Expr Expr
An expression in this language will look like the example below:
expr = Add (Add (Val 2) (Val 3)) (Val 1)
This expression can be evaluated using a evaluation function like below:
eval :: Expr -> Integer
eval (Val n) = n
eval (Add x y) = eval x + eval y
eval expr would then result in 6.
We could also have implemented this using a shallow solution. This kind of
implementation does not need any algebraic data type. The implementation will
instead work directly on the semantics of Haskell.
The solution is the following two functions:
val :: Int -> Int
val n = n
add :: Int -> Int -> Int
add p q = p + q

render :: Expr -> String

convenient way using a shallow embedding.

This would present the expression problem. That is, we could extend our data type
for the expressions, but this would require additional cases for the functions eval

the new language extension. For simplicity lets say we want a construct that adds

really easily. This we can do with the function below.

mult x y = Add (mult x (y-1)) x

3.6
Limitations for Embedded
Domain-Specific Languages in Haskell
In the previous section we saw that eval exprAdd8Mult is 42. However, we have to
be careful when we do this. The ﬁrst reason is that if we would render this expres-
sion we would not capture that we had used multiplication. Instead render would
return the string which captures how many time we would have to have added to
do the multiplication. Secondly, the way we interpret the expression is very slow
compared to ordinary multiplication. This is because we use addition but also be-
cause we recursively recompute the expression we multiply by. As a consequence,
it is very important to know the underlying semantics when using a shallow em-
bedding. It is also important to choose a deep embedding which lets a shallow
embedding effectively express the needed functionality.
Another reason that makes this mult function dangerous is that it will only
work for non-negative numbers. We already mentioned that. However, the com-
piler does not know this. As a consequence, we would be in trouble had we tried to
multiply with for instance a negative number. We could of course handle the case
and throw a speciﬁed run-time error. This we wont consider as a good solution.
Luckily, there are many ways to solve such a problem. One method is to simply
use smart constructors [Smart constructors]. Peano numbers are a good example
of this [Peano]. The problem with these are that they are not that ﬂexible. If we
need many different speciﬁcations for types this is not a good solution.
Another solution would be to use type safe constructors. These are simply
types in modules that choose only to expose certain constructors. An example is
given below:
data Natural = Natural { unNat :: Int }
nat1 = Natural 1
nat2 = Natural 2

that "reﬁnes" what possible values are allowed for a type at compile-time. That is,
we turn invalid inputs into compilation errors. This is made possible by a Haskell
extension Template Haskell [Template Haskell]. The goal of this extension is to pro-
vide compile-time meta-programming in Haskell. That is, we want the ability to
modify Haskell code with Haskell code at compile-time. Sadly, this can be rather
tedious to program by yourself. Luckily, the library Liquid Haskell exists [Liquid

used tradable assets are cash and stocks but ﬁnancial contracts themselves may

The two most basic types of ﬁnancial contracts are spot and forward contracts.

called the spot price which should not have changed notably since the contract

futures contract where contract terms are agreed upon now but delivery and pay-

crowns you will earn 10,000 danish crowns. Discounting is the process of deter-
mining the present value of a forward contract.

generating specialise code that, when run, would perform a contract valuation

phase. The employment phase is the individuals employment years and the retire-
ment phase is the individuals retirement years. These phases have the restriction
that the end of employment phase must come before the start of the retirement
phase. We will not at this point make any restrictions on at which concrete times
these phases must end or begin. That is, you can be in the employment phase un-
til you are 100 years old as long as you do not enter the retirement phase before
that. For the remainder of this thesis theses two phases will be used.
5.2
Pension Contract Introduction
Instead of simply setting income aside and using it later, one can draw advantages
from a pension contract. A pension contract is a contract which describes deposits
during the employment phase and withdrawals with some interest during the re-
tirement phase. As a concept both deposits and withdrawals are similar to the con-
cepts of setting assets aside and using them. However, henceforth both deposits
and withdrawals with some interests will be deﬁned in terms of some amount of a
currency.
The concept of a deposit during the employment phase could be formulated
as follows:
• deposit in t1 years the amount of x1 in currency c1
The concept of a withdrawal during the retirement phase could be formulated
as follows:
• withdraw in t1 years the amount of x1 in currency c1
The concept of a deposit during the employment years and a withdrawal with
some interest during the retirement years could be formulated as follows:

5.3
Concrete Syntax for Pension Contracts
A pension contract is a contract which describes deposits during the employment
phase and withdrawals with some interest during the retirement phase. However,
a pension contracts can also be considered as contract between two parties. The
two parties are the pension holder and the counter-party. The pension holder is
the individual who wishes to draw advantages from a pension contract, instead
of simply setting assets aside during the employment phase. The pension holder
does this by depositing assets to the counter-party. The counter-party is some pen-
sion experts who invests the deposited assets to generate interests. These interests
can then subsequently be withdrawn by the pension holder. This leads to deposits
being payment from the pension holder to the counter-party, while withdrawals
can be seen as payment from the counter-party back to the pension holder. That
is, a deposit is conceptually equivalent to the concept of giving a zero-coupon dis-

Danish Pension Contracts
In Denmark the tax legislation of the pension system will usually dictate how a
series of deposits and withdrawals are made. This is described in Chapter 2. How-
ever, the laws changes depending on who you are. The result is that we will need
to deﬁne these pension contracts for any individual group. That is, we could for
instance have a pension contract for people born in the year 1970.
A pension contract like that is described using natural language, like below:
• ratepension for person born in year z1
deposit annually from year t1 until year t2

{-@ ratePensionYear1970
:: { employmentPhase:Flow |
(unTime (start employmentPhase) == 0) }
-> { retirementPhase:Flow |
( unTime (end employmentPhase) ) <
( unTime (start retirementPhase) ) &&
( unTime (end employmentPhase) < 21 ) &&
( unTime (start retirementPhase) >= 20 ) &&
( unTime (start retirementPhase) <= 25 ) &&

This function captures a lot of different constraints. The ﬁrst constraint is that
we must start depositing this year. The reason for this constraint will be explained
soon. The next constraint speciﬁes that the employment phase must end before
the retirement phase can start. Then we make the constrain that the employment
phase must have ended within the next 21 years. More speciﬁcally this would, for
a person born in 1970, be at the age of 67. Similarly the next constraint says that
the retirement phase cannot begin before the individual turns 66 years old. The
next constraint says that the retirement phase must at least last 10 years. That is,
the individual must withdraw some annuity for at least 10 years. Then we make
a constraint that says the retirement phase must have ended within the next 35
years. This also limits the length of the annuity. Finally we also limit the amount
which can be deposited at time t0 (This is why we needed the ﬁrst constraint). We
also limit the annual increasing rate in percentage.
This will also work for "Aldersopsparing" and "Livsvarig livrente". We could
simply say "Aldersopsparing" does not have the constraint that it has to be with-
drawn in an annuity for at least 10 years. The annuity for "Livsvarig livrente" and
how this is calculated is not given in this thesis. However, one simple way that it
could be calculated is given a expected time of death, we just set the end of the
retirement phase to be that year.
The reader might have been wondering why we have been using the time unit
the way we have. The reason is, that we do not want to limit ourselves to a speciﬁc
interpretation for these just yet. This will allow more ﬂexibility later.

language for the danish pension domain. In the next chapter we will discuss the
result.

CHAPTER6
Discussion
This thesis has ﬁrst and foremost shown a technique for embedding domain-speciﬁc
languages in a host language. This technique has proven to be simple and yet very
powerful. Embedded domain-speciﬁc languages provide the full infrastructure of
the underlying language. This has to a great extend simpliﬁed the described im-
plementation task.
The functional programming paradigm and the Haskell language has proven
particularly well suited as a host language. The language features described have
proven very useful. These have all contributed to a implementation of a concrete
syntax that is very natural for the pension domain. The arguments against em-
bedded languages are limitations imposed by the host language. This includes the
concrete syntax which has not been a problem. The primary limitations for the
language presented includes limitations imposed on the compiler error messages.
GHC does have an extension that allows for user-speciﬁed type error messages
[Custom Type Errors]. This feature has due to the limited scope of this thesis not
been explored. Also type errors is only a partial solution. User-speciﬁed syntax
errors seem just as important. Further testing of the domain-speciﬁc language
presented has to be done to justify this extension.
Alternatively this problems can be solved by letting the embedded domain-
speciﬁc language serve as a foundation for a standalone language. This requires
a parsing from either textual or graphical terms in the standalone language. Par-
sec is an industrial strength, monadic parser combinator library for Haskell which
could allow for such a extension [Parsec]. This allows for handling of custom er-
ror messages. The current state of the domain-speciﬁc language presented does
not immediately justify such an extension. Further evaluation of the language pre-
sented is required to determine if these added beneﬁts outweigh the extra main-
tenance needed. This has due to the limited scope of this thesis not been explored.
Haskell has also proven particularly well suited as a host for combinations of

both shallow and deep embedded domain-speciﬁc languages. This is an approach
that has proven capable as a solution to the expression problem. The deep embed-

problems while the shallow embeddings has allowed easy implementation of lan-
guage extensions for the pension domain. In a domain with need for constant

The language presented in [Jon+00] has proven to serve as a very robust core
language even 18 years after the language was originally presented. The ﬁnance
and pension industry has an enormous vocabulary of jargon for typical combi-
nations of contracts. Most of which can be expressed in this core language. This
makes the language still very relevant and only very few additions have been made
since then. The language SPL gives an example of such an extension [AF15]. The
language extension presented solves the problem that programming knowledge
is still needed to deﬁne new ﬁnancial models. This is solved by deﬁning a new
domain-speciﬁc language for deﬁning ﬁnancial models. Furthermore, the lan-
guage presented extends the interpretation of ﬁnancial contract to be computed
on a GP-GPU. This language could serve as a good foundation for an extension to
the language presented in this thesis. However, it is yet to be determined if the
added beneﬁts outweigh the extra maintenance needed.
The focus in this thesis has been on deep and shallow embeddings. However,
there are other techniques for embedding a language. Another popular technique
is the Finally Tagless technique [CKS07]. This technique is about having an inter-
face which abstracts over all interpretations of the language. In Haskell this can
be achieved using typeclasses where each method corresponds to one language
construct. Concrete interpretations are accomplished by creating a data type and
making it an instance of the type class. For example, a string representation would
correspond to one interpretation, integer representation would be another one.
Finally Tagless also allows new interpretations and constructs to be added mod-
ularly (corresponds to adding new interpretation types and new interface classes
respectively), Finally Tagless can be said to be a more exact solution to the expres-
sion problem.
This thesis has also explored reﬁnement types using Liquid Haskell. These are
used to formally specify the parameters for pension contracts using simple math-
ematical notation. This has allowed the language presented to abstract away irrel-

Dependent types are an alternative to the reﬁnement types of Liquid Haskell.

The world of cryptocurrencies seems very relevant these days and a lot of work

CHAPTER7
Conclusion
The problem of developing a domain-speciﬁc language for deﬁning and analysing
various pension contracts has been achieved. We have formulated an abstraction
that match the syntax and semantics of the problem domain. Furthermore we
have provided a concrete implementation in Haskell. This allows pension experts
to formulate various different pension contracts.
The primary objective of this thesis was to develop a domain-speciﬁc language
for deﬁning and analysing pension contracts. The domain-speciﬁc language pre-
sented in this thesis provides a rich language for describing numerous ﬁnancial
contracts. This includes various pension contracts. This also includes the pen-
sion contracts of the danish pension system. These could all efﬁciently be imple-
mented using the language presented in [Jon+00]. The domain-speciﬁc language
presented also provides the possibility for various analysis of such contracts. This
includes various valuations of pension contracts.
The goal of this thesis has been achieved using the language presented in
[Jon+00]. This language indeed provides an effective approach to represent ﬁnan-
cial contracts. This includes pension contracts. The idea of providing a small set
of ﬁxed and well understood primitives that can be combined to a wide range of
contracts has proven very useful. This is shown using the language to create new
combinators for the pension domain. These combinators allows for the deﬁnition
of a broad range of pension contracts. Here amongst pension contracts of the Dan-
ish pension system.
Furthermore, the language presented in this thesis incorporates reﬁnement
types. These reﬁnement types are used to formally specify the parameters for pen-
sion contracts using very simple mathematical notation. This has successfully al-
lowed us to abstract away irrelevant implementation details about run-time er-
rors.
==PAGE:76==
66
7 Conclusion
The contributions of this thesis are not only a proof of concept. It is indeed
possible to create a good embedded domain-speciﬁc language for deﬁning and
analysing various pension contracts. Using a combined embedding of deep and
shallow domain-speciﬁc languages we have provided an easily extensible language
for the Danish pension domain. The combined solution of deep and shallow em-
beddings has proven very advantageous. The deep embedding from [Jon+00] proves
to be a very solid core language. This core language allows for very simple imple-
mentations of shallow embeddings for the Danish pension domain. Not once did
the expression problem occur during the development of the language presented
in this thesis. This is very important as it shows the ﬂexibility of the language and
how it will be able to handle future changes in the pension domain.
