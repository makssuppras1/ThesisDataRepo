Abstract
This thesis describes the implementation of the Join Patterns into Scala 3 as a language
extension, using meta-programming and reflection enabled by the language’s powerful
macro system. Join Patterns are an essential component of the join-calculus which the
implementation has been studied extensively since it was created, yet mainly as a pro-
gramming language library. Aiming to offer both application-specific mechanisms and
native language implementation, we cover the main features and illustrate the achieved
functionality with a few relevant scenarios.

Introduction
1.1
Motivation
Concurrent systems aim to solve problems by decomposing them into smaller parts that
are easier to solve, and assigning those parts to different components, acting as the com-
puting units. As these computing units may run in parallel, the execution is, contrary to
classic sequential program flow, out-of-order; a situation that must not affect the outcome
of the computations. This produces specific problems, like deadlock or starvation, that
are quite different from purely sequential programs, as the program execution flow is not
linear. For this reason, communication between the components plays a significant role,
and concurrent computing is paradoxically essentially cooperative. But as concurrent sys-
tems increase in size and complexity to tackle more advanced problems, recognising a
valid state from an undetermined state becomes much more difficult when the number of
computation paths of the program is exponentially large. Therefore we must change our
perspective from the global (top-down) approach to a more local (bottom-up) approach,
and pay close attention to the logic at the component level. To answer those challenges
has been developed the join calculus, a process calculus that is centred on asynchronous
communication between components, and the join patterns, an important construct for
specifying communication protocols based on message sets at the component level.
The research on join patterns implementation spans over twenty years, the focus on either
join patterns as libraries or language extensions being more or less evenly distributed.
While we do not aim to perform a survey of the field and might suffer from sampling bias
due to the limited number of entries in our literature review (see section 2.5), we have
identified tendencies that help understand the need for our work.
It seems that the early attempts in the early 2000s involving general-purpose impera-
tive languages (C# and Java) brought either libraries or compiler extensions by directly
modifying the compiler, probably due to the lack of metaprogramming features in those
languages at the time. In parallel, functional languages that had those features (namely
Haskell) provided join patterns with advanced aptitudes. The actor model seems to be
taken into consideration from 2008 and onward. Then it seems to be a split in two direc-
tions: on one hand, the research on libraries using a language mixing both imperative and
functional, Scala; on the other hand, language extensions implemented with functional
languages that aim to solve concurrency problems (Erlang/Elixir). It is worth mentioning
that until a recent date, the latter had a powerful metaprogramming system that the former
lacked 1.
The previous approaches to this problem have focused on enabling join patterns through
libraries, which are straightforward and efficient enough for research and cutting-edge
applications. Due to the API that they must provide, libraries might not be intuitive, being
limited to an imperative or functional paradigm (depending on the host language), while
the task to be solved - allowing the programmer to easily write and verify join patterns -
is easier done using a declarative paradigm. In other words, the host language syntax
can be limiting. So as to move towards deployment within more generic, broad-use sys-
tems, we recognise that a better integration using a macro system would provide stronger

Background and Related Work
In this section, we briefly summarize the subjects those understanding is required from
the reader to proceed with the rest of the paper. It starts with a concise description of
distributed systems, and then the formal models relevant to our approach.
2.1
Concurrent systems
Modern-day computing can sometimes involve components on different nodes of a net-
work or platform, for applications such as telecommunications, automation, or distributed
databases. A concurrent system solves problems with logic executed out-of-order, at
the same time and/or at an undefined time. Message passing capabilities are essential
to such systems and can constitute a bottleneck if the communications that make such
systems possible in the first place, do not fit the application and its behaviour. This cre-
ates another dimension orthogonal to the problem to solve that deals with correctness,
to ensure that the results are the same as a non-concurrent system and that no concur-
rency problems arise. To tackle this aspect, we can leverage distributed programming
languages, whose paradigm provides helpful abstractions.
2.2
The CHAM and process calculi
The CHAM [2] describes a peculiar and interesting abstract machine, where logic is ex-
pressed from the perspective of chemical reactions. There are reduction rules to express
computation, and structural rules to express ”the syntactical rearrangements of terms in
solution”. Of course, as an abstract machine, the CHAM is not usable by itself and serves
as a foundation for concurrency formal models, the process calculi. We will consider
the calculus of communicating systems (CCS), then the join-calculus [3], and finally, the
π-calculus [4], that both have the same expressive power.
CSS predates the following other calculi; it only considers two participants and allows to
ensure correctness properties, perform parallel executions, and action choices.
The π-calculus, an extension of CSS, tackles the same class of problems using atomic
non-local interactions (rendezvous), i.e. synchronous communications. However, ren-
dezvous can be difficult to achieve for distributed systems, such that the join-calculus
offers a more realistic and flexible choice.
The join-calculus is a variant of the π-calculus based on a reflexive version of the CHAM,
that is based on asynchronous communication. Distributed systems are specified not in
terms of a sequence of interactions that reflects a protocol, but rather as an asynchronous
set of event-triggered behaviours at the component level. The reactions in the form of
message sets are called join patterns.
2.3
The Actor Model
The actor model [5] centres concurrency on actors, making them stateful primitives that
perform logic upon receiving messages. Actions include sending messages to other ac-
tors or creating more actors. This message-passing is the only way actors have to com-
municate with each other, as they are only aware of other actors’ addresses. This fits
well with the object-oriented paradigm, and as a result, the actor model has often been
directly used for the implementation of concurrent systems. But actors have a significant
drawback: they can only handle a single message at a time; in some way, join patterns

2.4
Metaprogramming and reflection
Metaprogramming reifies programs and allows for their inspection and modification. Of
course, it is a highly desirable feature if the host language enforces the syntactic cor-
rectness and type safety of source and produced programs. It is a feature more present
in interpreted languages, for in a compiled language the pipeline from code to program
is more complex and longer; however, some modern ”hybrid” languages have been ex-
tended with reliable metaprogramming mechanisms. It can be achieved through several
methods, including template metaprogramming, meta-classes, and in our case, macro
systems.
Reflection is a type of metaprogramming where a language is its own metalanguage, i.e.
the mechanism in a host language by which it can inspect and modify itself is a part of the
host language itself, and not a smaller embedded language with its grammar and rules;
the latter approach being often simpler but more limiting and complex to use, and the
former being the one we use in this paper.
2.5
Related Work
2.5.1
Libraries
Cω
Cω 1 started as a language extension in C# 1.x, but evolved into the Joins.NET library 2
[7] when C# 2.0 introduced generics. This choice, in the author’s own words, limits ”the
scope for both optimization and static checking”, in exchange for portability 3 and ease
of modification. It also enabled support for dynamic joins, since the size of underlying
arrays is determined at runtime. The library is implemented using reflection and generic
delegates (equivalent to lambda expressions), the latter ensuring type safety with checked
casts.
The syntax seems a bit heavy and sticks out from normal C# code.
Concurrency and Coordination Runtime (CCR)
CCR [8] aims to address the shortcomings of Cω, where patterns are statements instead
of declarations. They provided a library implementation of join patterns and introduced
the dynamic joins feature. Because this library does not use threads or processes, in
favour of a continuation-passing style code 4, it has a custom scheduler. The authors
also advocate that ”the best way of supporting concurrency constructs may be through
language-level support”.
The syntax seems heavier and more obscure than that of the Joins library. Moreover, the
ports can only accommodate up to 16 different message types. Finally, it requires its own
scheduler and provides a dedicated debugger.
Actors with Multi-Headed Message Receive Patterns
It is unclear how multi-headed message receive patterns [9] in the context of actors are
any different from join patterns. They provide patterns with guards/predicates for Erlang
implemented in Haskell, similarly to [10]. Moreover, the patterns can be non-linear, which

Scala Joins
In [11] is discussed a technique to implement join patterns as a library that offers the
same flexibility as a language extension: extensible pattern matching. It also introduces
the Actor Model as a potential application for join patterns, at the cost of some boilerplate
code, but also works with threads. The message-passing mailboxes are built from a Queue

Our solution draws much from this work regarding how the join patterns are implemented,
but of course, they are solely runtime constructs here.
JEScala
JEScala [12] aims to bring join patterns to Scala 2 and address the concerns on logic frag-
mentation that the authors have identified in the previous approaches (see section 2.5.2,
section 2.5.2, section 2.5.1, section 2.5.2). To that extent they leverage the event sys-
tem of EScala [13], which has interesting properties, including composite events and dis-
junctions. JEScala supports dynamic handlers, that unlike dynamic joins only allow for
modification of the application logic associated with a pattern, and not the pattern itself.
We mostly avoided the issue raised by the author due to the language construct that sup-
ports our implementation: the match clause. However, we also choose to limit guards to
functions/closure applications, that are defined elsewhere. The syntax makes the predi-
cates less visible in the code.
2.5.2
Language Extensions
Jocaml
Jocaml [14] 5 is one of the earliest description of join patterns implementation. It runs on
threads but uses channels for message-passing and not shared memory; however, the
channels are untyped. It introduces the notion of Process (also found in section 2.5.2 and
section 2.5.2, process-oriented languages) with side effects in an expression-based func-
tional language, which is a certain shift from the language philosophy. The components of
the system are called locations and associate an agent to a threaded process (somewhat
similar to the Actor Model). Join patterns are here composed using the pipe/or character
(|) and support message payloads.
Polyphonic C#
Polyphonic C# [15] was developed as a language extension to C# 1.x. It provides asyn-
chronous features for concurrency based on join patterns called ”chords”. Embedding
this feature into the compiler improved optimization and static checking. Polyphonic C#
was then merged into Cω.
Join Java
To our knowledge, join patterns on the JVM started with [16], by directly modifying the
compiler, as in section 2.5.2. The Java standard library already contains several low-
level synchronization techniques (including green threads and synchronized blocks) but
lacks higher-level features. They slightly criticize the syntax of Polyphonic C#, a remark
we share, and aim to offer a more ”straightforward” solution. It is also worth noting that
the ordered property allows specifying pattern priority, an aspect that we implemented as
well. The pattern matcher has a dedicated compiler interface and is rather advanced, as
the authors investigated several matching policies.

Higher Order Combinators
This paper [17] uses Software Transactional Memory (STM) to check that read or write
accesses to shared memory (grouped in transactions) between threads are valid (i.e.,
free of read/write conflicts related to concurrency) before committing the changes. More
specifically, the authors used transactional channels in their implementation. That means
that data is not exchanged through a message-based mechanism; this choice does not
have much impact on the design of the join-patterns that rely on it, but negatively affects
performance [18].
The Combinators in Haskell are Higher Order Functions 6 that are used to transform pro-
grams as functions 7, in a similar fashion to the macros in Scala, but at runtime. Haskell
combinators also enable support for conditional join patterns using predicates, and dy-
namic joins.
The syntax seems quite ”clean”, without much boilerplate code, but it is a bit remote from
real-life applications.
JErlang
JErlang [10] 8 aimed to extend Erlang, a language designed for reliable distributed sys-
tems running on the BEAM virtual machine. The metaprogramming is done using trans-
formations functions that will later the source code before it is handed to the compiler like
it is done with macros. As processes are central in Erlang and inter-process communica-
tion is achieved by sending messages to mailboxes, Actor programming is done directly
through processes. As in our approach, native pattern matching is also reused, which in
Erlang provides non-linear patterns, a desirable feature absent from our solution. More-
over, the real-time paradigm of Erlang allows for easy implementation of timeouts and
time windows. It is also possible to make synchronous calls between processes. That
being said, all those additional features can be emulated in Scala, at the cost of more
work on the software engineer’s part.
The authors have achieved greater performance by directly modifying the virtual machine
Erlang runs on top of, which is a certain drawback in terms of acceptability. They mention
borrowing ideas from the RETE algorithm [19] to improve performance, a concern similar
to ours.
Sparrow
The Sparrow language [20] 9 10 is a Domain-Specific Language (DSL) based on Elixir,
that runs like Erlang on the BEAM virtual machine. Elixir can be seen as a modern and
functional Erlang, as Scala can be for Java, and explicitly follows the Actor Model. It has
non-linear patterns and real-time features as in JErlang (see section 2.5.2 above). Fur-
thermore, the language provides useful logical operators for composing patterns : not,
and, or, and even allows to specify the matching behaviour using options when declar-
ing patterns. A result of the split between pattern and reaction declarations means the
information in the code is in two places instead of one.

claim to be able to extend the Actor Model while maintaining backwards compatibility. Join
patterns can declare an arbitrary number of messages of any type, where the order is not
significant (i.e., our join patterns are unordered), and multiple messages of the same type.
We also allow for wildcard patterns, single message patterns that match any message.
And, since Scala is expression-based, join patterns can also leave a value, whose type
must be provided to the macro.
The message type given as a type parameter of the macro should be a super-type of the
messages of the protocol, or a subset specific to a particular actor. This enables for typed
actors 1, which in turn leverages type checking.

3.4.1
Complexity Analysis
Several operations must be taken into account. Concerning pattern.guard and pattern.rhs,
they are either absent, or supplied by the programmer. As for matchedMessages.isEmpty,
messages.append() and the calls of type Collection.size, they are constant. The op-
erations on the message list messages.--=() and messages.+=() are linear on the size
of the collection supplied as argument, but messages.+=() can be disregarded for it has
no impact on the search, taking place after the search took place. Finally, pattern.match
depends on the size of the join pattern if it matches, or the size of the message list if it
does not (see section 4.3).

• M be the messages list, and |M| its size, i.e. the number of messages
• Pa be the pattern table, |Pa| its size, i.e. the number of join patterns
• Pa∗⊆Pa be the subset of join patterns where |Pai| ≤|M|, and |Pa ∗| ≤|Pa| its
size
• S be the average join pattern size in Pa∗

O(|Pa ∗| · |M|)

4.1
Scala 3
We choose Scala as the implementation language. In particular, our project relies on the
latest iteration of the language, still under active development at the time of writing this
article, which ships with excellent and type-safe metaprogramming and reflection capabili-
ties; unlike languages such as C/C++, whose macro system is more akin to a compile-time
string replacement mechanism, hence very unsafe, prone to undesired effects, and with-
out real scoping rules.
In Scala, the module scala.quoted contains constructs that allow us to extract types and
expressions from the code source (for Scala is an expression-based language, as it is of-
ten the case in functional programming), such that we can transform them, and insert them
back into the code to be then analyzed and compiled. Quoting and splicing are essential
to this process, but the resulting code can sometimes be slightly obscure. However, that
means that the objects that we mostly manipulate are Abstract Syntax Trees (AST), which
is a major change from text-replacement macros and a significant advantage. ASTs are
composed of many different classes that can be found inside the aforementioned module,
for all the language’s possible constructs, terminals or non-terminals. But of course, we
will work with only a subset of those constructs
Moreover, as most functional languages do, Scala has powerful pattern matching features
that we use extensively in implementation, and a safe and extensible type system (despite
running on the JVM). We also were careful not to use null pointers, so as not to lower the
reliability of our macro.

from within, using the same identifier. If the pattern has no guard, the function body is
true.

The field size simply holds the pattern size, that is also the size of the List[M] tuple
member returned by extract when the match is a success.

4.6.1
Dynamic Mailboxes
Since the queue is given as the argument to apply at runtime, we can wander farther
away from the Actor Model. If the programmer was to extend our simple Actor class with
multiple mailboxes, it is perfectly possible to have dynamic mailboxes, as distinct event
sources. However this would face a challenge: to increase performance and reduce ac-
cesses to the queue (see section 3.4), and especially re-inserts of unmatched messages,
Matcher holds a messages field of unmatched messages that are preserved across the
calls to apply. That means that if messages from a queue are left in messages, new
messages from another queue might be mixed with them. For this reason, we encour-
age developers to create a whole new actor altogether and discourage the use of such a
technique for real-life applications.

While the time complexity of our algorithm is O(|Pa∗|·|M|), we can see that it is linear, or
quasi-linear. If it is likely for a pattern within the pattern table to match an arbitrary message
list, the search time can be reduced. Fine-tuning the pattern order/priority, with respect to
the system’s intended behaviour, can help achieve greater performance. Moreover, the
more the message list and the matching pattern are similar, both in message types and
numbers, the faster our algorithm can work. As a counterexample, if the message list
contains a great number of messages irrelevant to the join pattern considered, a certain
amount of time will be taken to recover the matching messages. We must remember that
the matching function at the pattern level is parametric of the pattern size in the best case
and the message list size in the worst case.

5.4
Discussion
We cannot compare benchmark results from other projects directly for they have been run
on very different machines, but we can look at the tendencies and differences between
join patterns and non-join patterns cases.
As we can see, there is a small but certain speed overhead with using join patterns, and
the data is more spread around the median. This discrepancy is reflected in the literature,
and the contrast becomes more significant if the messages carry a payload (a cost partly
related to runtime memory management). This situation is similar to [10], [15] and [8], but
we find the discrepancy in our data to be less significant than the implementations that
use C#. We suspect that our use of LinkedTransferQueue to pass messages, offloading
logic for thread-safe accesses to shared memory, is of importance.
In our case, the passes where the algorithm cannot match any pattern and has to pull new
messages from the mailbox might constitute a large part of the fourth quartile in fig. 5.1
and fig. 5.2.
It also seems that the overhead increases with pattern complexity, as it is shown by a
comparative analysis of the benchmarks: the difference between the control and macro
versions is more significant in the Smart House benchmark than in the Ping Pong bench-
mark. However as we can see from the study in [12] (Figure 10), the drop in performance
we encounter (see fig. 5.4) is smoother than most other implementations.
It is shown in [12] that optimisations, such as a thread pool reusing threads instead of
creating/destroying them, can bring great benefits. When it is possible, compilation to
native code as in [10] instead of JIT compilation by a VM also increases performance, but
it is not available to us in Scala.
The code size is an aspect that should not be neglected, and programming abstractions
should provide expressiveness without code bloat. We only take into account the essential
code ”without futures” (see table 5.1). Indeed, our solution offers little benefit when it
comes to the single-message patterns to the like of the classic actors, but when dealing
with more complex actors the difference becomes more obvious: appendix B makes a
good at showing how difficult and unclear the codebase can become without join patterns.

6
Conclusion
6.1
Contributions
We have written an implementation of join patterns using the new Scala 3 macros that
provide a simple and Scala-like syntax for the programmers to specify their join patterns.
The boilerplate code is very small and language constructs are reused internally, making
room for consistent and transparent compile-time checks by the compiler. Given the re-
duced size of the codebase and the separation of concerns, we hypothesize certain ease
of further extension/modification of our solution, provided the developer is familiarized with
Scala reflection classes and their APIs. The runtime uses a simple and flexible match-
ing algorithm to determine which pattern is eventually fired depending on the received
messages.
Then, we created a simple Actor implementation inspired by Akka, a popular implemen-
tation of the Actor model in Scala 3, to ensure it was feasible to build real-life applications
upon our solution.
Moreover, we harnessed a test framework to ensure the correct behaviour of the proposed
scenarios and created a small benchmark framework to assess the runtime overhead. We
found that the runtime overhead would not be an issue for typical applications and that
the benefits outweigh the disadvantages, and in a broader picture further reinforces the
relevance of actors for concurrent systems by extending their features.
The following objectives were therefore fulfilled: 1) our macro allows for expressing join
patterns in a concise and easily understandable way; 2) well-defined concurrent systems
using our join patterns implementation behave according to their specifications; 3) such
systems can scale in complexity without undue performance penalty; 4) our solution can
fit into the Actor Model, so as to broaden its capabilities.
6.2
Future Work
It seems that the areas of progress are widened to the runtime matching algorithm. Our
ad-hoc solution works well for cases with common complexity but might show its limits
when pushed. We especially could not investigate the RETE algorithm [19]. This can be
enabled through the statefulness of the join pattern runtime, represented by the Matcher
class. Alternatively, it is maybe possible to reuse the new Scala feature ”Match Types” 1
that allows reducing a type according to rules defined in the case clauses. In that regard
especially, there is surely room for improvement regarding the advanced language fea-
tures that our limited knowledge of Scala did not allow us to make use of, but valuable
inspiration and hindsight could be drawn from [1]. Typecase and case classes especially
seem like a better alternative to our current implementation; however, Scala 3 macros
still have limits that, to our knowledge, require Lambda2-crafted AST, somewhat tedious
to write.
As for the join-patterns themselves, we choose to change the more common pattern &
pattern syntax in favour of the (pattern, pattern) for commodity and practicability,
but this is of course a debatable choice and could be modified at the macro level. We
