Summary
The goal of this thesis is to implement and develop Multi-Agent Pathﬁnding
algorithms, analyze and compare them. In the beginning of this thesis Multi-
Agent Pathﬁnding problem is presented and diﬀerent types of conﬂicts are de-
scribed. Then Multi-Agent Pathﬁnding algorithms, naive A* and Conﬂict Based
Search, are presented with examples and implementation. Next there are ex-
periments done with these algorithms, to see how the run time increases when
the graph size, number of agents or length of the tunnel increases. Results are
analysed, summarised and conclusions are formulated.

Chapter 1
Introduction
Multi-Agent Pathﬁnding (MAPF) is the problem of ﬁnding none-colliding paths
for multiple agents in the same environment to reach their goals. Application
is wide including autonomous vehicles, computer games, robotics and environ-
ments like automated warehouses, hospitals or airports, where robots carry out
transportation tasks. This can save signiﬁcantly on human resources and let peo-
ple focus on more complex, creative or intelligent tasks. To reach this, robots
have to be able to move without collisions, preferably using the shortest paths
possible and without spending too much time on calculating paths which is the
goal of MAPF.
At airports there is an endless amount of movement activity for aircrafts, such
as in the departure phase: pushback, navigation to the entrance to the taxiway,
taxiing and takeoﬀ; in the arrival phase: landing, entering the taxiway, entering
the ramp area, and navigating to a gate [MPL+16]. This is a complex logis-
tics problem, involving coordination of potentially huge numbers of humans and
machines. In this MAPF problem agents are the aircrafts and agent goals are
to reach the gate, the taxiway etc. Currently planning movement activities in
the airports is heavily based on humans interpreting surveillance data and com-
municating routing information to pilots. Due to uncertainty, primarily in the
form of delays there is a continuous replanning and rescheduling of movements.
Since safety and eﬃciency are the main criteria for eﬀective airport work, au-
tomation would help to avoid tired or distracted people making mistakes and

Another MAPF problem application is in warehouses and distribution centers

Chapter 2
Background
2.1
Multi-Agent Pathﬁnding
Multi-Agent Pathﬁnding (MAPF) is the fundamental problem of planning paths
for multiple agents in a way that they would be able to follow these paths and
reach their goal locations going around the obstacles and without colliding with
each other [SSF+19]. MAPF problem has a lot of real-life applications and is
also diﬃcult to solve. The number of states and the branching factor grows
exponentially with the number of agents moving simultaneously. [WB08]
Systems with MAPF problem can take centralized or decentralized approaches
[XL02]. In the centralized approach agents behavior are usually considered all
at once, in terms of joint intentions and joint actions [LCN90]. Path calcu-
lations and coordination is often done from the main system. In opposite, in
the decentralized approach each agent is responsible for planning for its own
paths, usually agents have partial view of the system. Agents also have to com-
municate with each other in order to share their partial view of the system
and to coordinate to avoid collisions. In this paper the centralized approach is
considered.
Stern et al. deﬁnes a classical MAPF problem with following assumptions: time
is discretized into time steps, every action takes precisely one time step and in
==PAGE:12==
4
Background
every time step, each agent occupies precisely one vertex [SSF+19]. The input
for such MAPF problem with k agents is a tuple (G, s, t), where G = (V, E) is
an undirected graph, s : [1, ..., k] −→V is mapping an agent to a source vertex,
and t : [1, ..., k] −→V is mapping an agent to a target vertex.
Every time step each agent in the graph can perform an action. An action is
a function c : V −→V where c(v) = v′ represents an agent at vertex v which
performs c and then it is in vertex v′ at the next time step. Each agent has
two action types: move or wait.
A wait action is when an agent stays at
the same vertex for another time step. A move action is when agent moves
from its current vertex v to a neighbour vertex v′ and this action is valid when
(v, v′) ∈E.
For an agent i and a sequence of actions π = (c1, ..., cn), location of the agent is
deﬁned by πi[x] after ﬁrst x actions, where ﬁrst action starts from agent’s source
location s(i). Formally πi[x] = cx(cx−1(...c1(s(i)))). A sequence of actions π is
a solution for an agent if executing this sequence from s(i) ends up in t(i), or
πi[|π|] = t(i). A solution for MAPF is a set of k plans, one for each agent.
Figure 2.1: A small region of a warehouse layout example
A part of a warehouse region is taken as an example, shown in ﬁgure 2.1. Input
for such MAPF problem consists of the source vertexes s(1) = 28, s(2) = 4,
target vertexes t(1) = 15, t(2) = 11 and a graph, where vertex V = (0, ..., 29)
and the edges E = ((0, 1), ..., (28, 29)) are as shown in illustration. Then the
action move left(28) = 27 and can represent agent A movement to the left from
the source vertex 28 to the vertex 27. This is valid action, since the edge exists.
Move down(28) represents illegal action, since in this example, there is no edge
from vertex 28 to go down. The solution for agent A in this example is π =(move
left, move left, move left, move up).
Another way to represent the solution, used in this project, is a set of k paths,
instead of plans. A path pi = (s(i), c1(s(i)), ...) is a sequence of locations for
agent i, starting with the source location s(i) and adding a new location for each
valid action. The path holds a solution if it ends up in t(i). In the warehouse

2.2
Conﬂict types
To ﬁnd a valid MAPF solution, it is important to have a good deﬁnition of
conﬂict. In literature, conﬂicts are deﬁned and categorised diﬀerently. In this
project following types are considered.
Vertex conﬂict
Figure 2.2: Vertex conﬂict
First and most evident conﬂict type is a vertex conﬂict, when more than one
agent tries to be in the same vertex at the same time. This conﬂict type can
be represented by tuple (ai, aj, v, t), where agent ai and agent aj occupy vertex
v at time step t [SSFS15], [LFB+19], [SSF+19]. Vertex conﬂict illustration is
shown in ﬁgure 2.2.
Edge conﬂict
Figure 2.3: Edge conﬂict
Another crucial type of conﬂict is an edge conﬂict when two agents try to move
on the same edge at the same time, but in opposite directions, or in other
words, try to swap their positions. This conﬂict type can be represented by
tuple (ai, aj, v, u, t) where agents ai and aj traverse the same edge (v, u) in
opposite directions from time t to t + 1 [LFB+19]. Edge conﬂict illustration is
shown in ﬁgure 2.3.
==PAGE:15==
2.2 Conﬂict types
7
Stern et al. [SSF+19] call this conﬂict type swapping conﬂict. In their paper
they also have an edge conﬂict, which diﬀers from swapping conﬂict in agents
movement direction. In both conﬂicts agents move on the same edge an the same
time, but in edge conﬂict they move in the same direction, while in swapping
conﬂict agents move in the opposite directions. Moving in the same direction
means that agents move from same vertex to the same vertex, which gives two
vertex conﬂicts so this conﬂict type is not discussed further in this project.
Following conﬂict
Figure 2.4: Following conﬂict
A following conﬂict appears, when one agent tries to occupy a vertex which was
occupied by another agent in previous time step. It can be represented by tuple
(ai, aj, v, t) where agent ai at the time step t+1 is in the same vertex v as agent
aj at time step t [SSF+19]. Following conﬂict illustration is shown in ﬁgure 2.3.
Stern et al. [SSF+19] also deﬁned cyclic conﬂict, which occurs if there is a set
of agents with plan at the same time to move into a location currently taken
by other agent from that set, forming a rotating cycle pattern. Cyclic conﬂict
consists of multiple following conﬂicts. For CBS way of solving conﬂicts it is
not important to diﬀerentiate cyclic conﬂict from following conﬂict so it is not
discussed further in this project.

2.3
Implementation notes
All implementations were written in C# programming language.
Heuristic function
Euclidean distance was selected as a heuristic function h(n) for all A* imple-
mentations. It calculates straight line distance between two vertices from their
coordinates. Algorithms are adapted to work also with graphs without coordi-
nates, but then no heuristic function can be used. This leads, that A* algorithm
works as Breadth-ﬁrst search (BFS), it still ﬁnds an optimal solution, but time
and space cost increases exponentially. All graphs used in experiments (Chapter
4) have coordinates.
Search for conﬂicts
There are two functions, FindConﬂicts and ConﬂictExist, which uses the same
logic, just FindConﬂicts returns a list of conﬂicts and ConﬂictExist returns a
boolean value. Both require paths and maximum length of paths as an input,
FindConﬂicts also needs a list of agents for conﬂict creation.
The functions go through all pairs of two distinct agents’ paths, px and py,
where ax ̸= ay. For each time step t (index in paths) the functions check if
any conﬂict type exists. To then catch a vertex conﬂict, the functions check
if at any one future time step t in these paths, px and py, if the vertexes are
equal px[t] == py[t]. To catch an edge conﬂict, the functions check if the ﬁrst
agent vertex at time step t is equal to the second agent vertex at time step t+1
and the ﬁrst agent vertex at time step t + 1 is equal to the second agent vertex
at time step t, px[t] = py[t + 1] AND px[t + 1] = py[t]. The last conﬂict, the
following conﬂict, is recognized if one agent vertex is the same as another agent
vertex in the next time step, px[t] = py[t + 1] OR px[t + 1] = py[t].
The ConﬂictExist function returns true after the ﬁrst recognized conﬂict, other-
wise if it goes through all iterations without ﬁnding any conﬂict it returns false.
FindConﬂicts adds all recognized conﬂicts to a list, which is returned after all
iterations. If this function returns an empty list, this means that there are no
conﬂicts in the given paths.

Chapter 3
MAPF algorithms
3.1
Baseline, A*
The naive approach to solve the MAPF problem is to run a search for all agents
at once. The issue with this approach is that, it has a high branching factor
which leads to a slow and space costly search. The naive approach checks all
possible move combinations for each time step, so it cannot scale well with a
growing number of agents in a bigger environments. A* was used as a naive
approach.
3.1.1
Example run
A simple MAPF problem is shown in illustration 3.1. Two agents are represented
as mice, goals as cheese. The dark red mouse on vertex 0 has the goal of reaching
the dark red cheese on vertex 1. The light yellow mouse on vertex 2 has the
goal of reaching the light yellow cheese on a vertex 3.
Each agent in a squared grid graph has 5 available actions: u (move up), d
(move down), l (move left), r (move right) and w (wait). When computing a
plan for all agents using A* we use joint actions: a tuple (c1, ..., cn) containing

A* discards joint actions which have any conﬂict, as well as joint action (w,w),
since it introduces an already existing state (time is not considered in this A*
implementation). This leaves A* to create just three new nodes from the joint
actions: (r,r), (r,w), (w,r).
All these new nodes have 2 as a cost of getting
to their vertexes (waiting, as well as moving costs 1 for each agent), but the
estimation regarding how much is left to move diﬀers. The joint action (r,r)
brings both agents into their respective goal positions, so estimation is 0, while
in joint actions (r,w) and (w,r) one of the agents are still left with distance 1 to
its goal. A* continues with a node, which has the smallest cost and estimation
which is the joint action (r,r). Since both agents reached their goals, A* returns
a solution. First agent’s path is (0,1) and the second one’s is (2,3).
As this example shows, A* generates a lot of possible moves combinations. When
each agent has k available actions, and we have n agents, then the branching
factor is kn. In square grid graphs, k = 5. Considering a scenario with 25
agents, branching factor would be 525 = 2, 89 ∗1017. Hence, in each step A*
might have to consider this number of joint actions. The running time of A* to
ﬁnd a plan of length d is O(bd), where b is the branching factor. Hence, in the
worst case, the running time of A* is exponential in the length of the solution
with the branching factor being the base of exponent. This clearly shows that
it will be unrealistic to use A* with many agents.
3.1.2
Implementation notes
A* as an input gets a graph, an array of agent’s start positions and an array of
their goal positions. Output is a solution in the form of an array of paths for
each agent. The most complex part - to get all joint actions, is implemented
as a recursive function. It goes through one agent’s possible actions and for
each possible action it calls the recursive function again, for the next agent. If
the last agent is reached, the recursive function uses ConﬂictExists function to
check if the new joint action has a conﬂict. If yes, the joint action is forgotten.
If not, this combination can be considered further.

3.2
Conﬂict Based Search
Conﬂict Based Search (CBS) is an optimal, complete multi-agent path ﬁnding
algorithm [SSFS15]. This algorithm has a two level design, which allows it to
simplify search without trying to plan for all agents at once. Plans are made
individually for agents and then checked for conﬂicts, which are resolved by
adding constraints at a high level then computing paths, which conform with
those conﬂicts at a low level.
3.2.1
Example run
Looking into the same simple MAPF problem, shown in illustration 3.1, where
the dark red mouse is on vertex 0, whose goal is to reach the dark red cheese
on vertex 1 and the light yellow mouse is on a vertex 2, whose goal is to reach
the light yellow cheese on vertex 3. As was shown in the A* example run, the
naive approach for a such simple MAPF problem can be quite fast to ﬁnd a
solution: the dark red mouse moves from vertex 0 to vertex 1 and the light
yellow mouse moves from vertex 2 to vertex 3 and both mice (agents) reach
their cheese (goals) in one move.
When this simple MAPF problem is given to the CBS solver, it ﬁrst runs the
low level search. The low level ﬁnds a solution for the dark red mouse: 0 −→1,
and then it ﬁnds a solution for the light yellow mouse: 2 −→3. At the high level,
the CBS algorithm checks if there are any conﬂicts between these two paths.
Since there are no conﬂicts in this example, the CBS terminates and returns the
solution.
Another example of an MAPF instance is given in illustration 3.2. The dark
red mouse is on vertex 2, whose goal is to reach the dark red cheese on vertex
3. The light yellow mouse is on vertex 4, whose goal is to reach the light yellow
cheese on vertex 2. The shortest paths to reach corresponding goals for the dark
red mouse is 2 −→3 and for the light yellow mouse is 4 −→3 −→2. This is not a
valid solution, since both mice cannot be at vertex 3 after the ﬁrst move, since
this is a vertex conﬂict. The goal of an MAPF solver is to ﬁnd none-colliding
paths. A solution for this example can be that the dark red mouse moves 2 −→3
and the light yellow mouse is 4 −→1 −→0 −→2.
Giving the MAPF instance shown in ﬁgure 3.2 to the CBS solver, at ﬁrst the
solver runs the low level search to ﬁnd a naive solution. The low level ﬁnds a
path for the dark red mouse: 2 −→3, and then ﬁnds a path for the light yellow
mouse: 4 −→3 −→2. After two moves both mice are at their goals, so the cost of

3.2.2
Algorithm
CBS search pseudo code is given in 1 algorithm deﬁnition. This algorithm starts
with creating a new root node with an empty constraints list, the naive solution
with shortest paths for all agents ignoring others and a solution cost. The root
node is added in the Open list. From the Open list a node with the lowest
cost is selected. Then it is checked if this node’s solution has conﬂicts. If there
are no conﬂicts found, CBS terminates and returns the current node’s solution.
Otherwise, a conﬂict is chosen arbitrarily and nodes are created for each agent
in that conﬂict. Those nodes inherit from the parent list of constraints, solution
and a cost. The constraints list is expanded with information from the current
conﬂict in which place and time the agent cannot appear. Then CBS calls the
low level (in this paper the low level is expanded A* with state time and con-
straints deﬁned in section 3.2.3) to update the solution paths for each conﬂicting
agent in the nodes. The new nodes are added to the Open list and the old one
is removed. CBS continues going through the Open list in these steps until it
ﬁnds a node without any conﬂicts or the Open list becomes empty which means,
that all possible solutions were check and the given problem is unsolvable.

3.2.3
Low level design
For the CBS low level search Sharon et al. [SSFS15] were using A* algorithm
which handled constraints by removing the generated state (v, t), where v is a
vertex and t is a time step, for an agent ai if there exists a constraint (ai, v, t).
This deﬁnition expands the well known path ﬁnding algorithm A* [HNR68] state
with time in addition to place. The algorithm should also include constraints
given as an input from the CBS high level and remove conﬂicting states while
searching for the shortest path.
3.2.4
Implementation notes
Challenge
In some implementations, agents disappear from the graph after reaching their
goals (this can be in a computer game, e.g. when a player reaches some portal).
In others, when one agent reaches its goal, it is still in the way for others, like
robots in warehouses, hospitals, etc. This creates a challenge, when agents’ paths
==PAGE:25==
3.2 Conﬂict Based Search
17
has diﬀerent length. The function for ﬁnding conﬂicts is looking for recurring
vertexes in two diﬀerent paths at the same time step or one time step later.
When one agent stopped at a vertex and another agent after more than one
time step went through that same vertex the conﬂict was not noticed. To start
noticing these conﬂicts, all paths were extended (with the last vertex in them)
to take the same amount of time steps as the longest path takes. After this
update, all conﬂicts were found.
The next step was to consider constraints in the low level A* implementation,
since it was ﬁnishing when agent reaches its goal. Selected way to ﬁx this is
to check if there are any constraints for the goal vertex after goal was reached.
If agent reached the goal vertex, but there is a constraint for this vertex in
the future, A* continues to search for the solution. The low level A* stops the
search, when agent reaches goal vertex and there are no more constraints for
this vertex at current or future time steps.

Such bad CBS performance with the increasing length of the tunnel is biggest
ﬂaw of the design. In this experiment agents meet in the middle of the tunnel.
CBS recognises the conﬂict and tries to solve it by not allowing one and then
other agent to be in the conﬂicting state (time and place). This creates two
nodes, with conﬂicts just one step closer to the end of the tunnel. The process
is repeated and this creates more and more nodes. The nodes with the shortest
path is selected ﬁrst, since the algorithm tries to ﬁnd the shortest solution,
while a valid solution is way longer than initial shortest paths, since waiting is
unavoidable. The CBS solver would beneﬁt from an improved logic, if it meets
another agent moving in the opposite direction in the middle of the tunnel, then
try to solve this conﬂict before one or other agent goes into the tunnel (move
all the way back to the beginning of the tunnel, not just one step closer).
This problem of CBS is not repeatable in scenarios when agents have alternative
paths (second tunnel) or agents move in the same direction. With two 11 ver-
texes length tunnels the CBS run time is around 20ms, while baseline is 80ms.
And when both agents move in the same direction in one 11 vertexes length
tunnel, CBS takes around 30ms to ﬁnd a solution, while baseline takes around

Chapter 5
Conclusions
The MAPF problem is a computationally hard problem, which importance is
shown with a wide range of applications. Since naive solutions with A* do not
scale well, it is essential to design better algorithms. In this thesis we described
and implemented A* as well as Conﬂict Based Search. To see their performance,
various experiments were performed with increasing graph size, number of agents
or tunnel length.
In most of the experiments, the A* algorithm’s run time quickly showed expo-
nential growth, surpassing one minute. Just in the third experiment with in-
creased length of the tunnel A* was faster than CBS in longer tunnels. Whereas
CBS most of the time outperformed A*. When graph size increases the CBS
run time grows exponentially too, but where A* spend more than 17min to ﬁnd
a solution, CBS still needed less than a second. With an increased number of
agents CBS showed linear growth, which is good for scalability, and was way
faster than A* starting with more than 2 agents in the graph. The worst CBS
performance was in the long tunnel, when agents move in diﬀerent directions.
In the tunnel of 11 vertexes length CBS took more than 17min to ﬁnd a so-
lution, due to the large number of conﬂicts in the tunnel, whereas A* found
a solution in 19ms. This is not the case in graphs with more tunnels, or with
agents moving in the same direction. So CBS shows an improvement over the
naive A* approach, and can be used in environments, which do not have single,
narrow, long tunnels to go between two areas.

5.1
Future work
There are several areas that could be improved further. One of them is the
expansion of tasks - improved solvers, so that they would be able to plan for
more than one task for each agent. This could then be extended further with
task dedication - solvers would get a list of unassigned tasks and they would
need to assign those tasks for agents before calculating paths.
Also some features could be added, like agent speed, distance between neigh-
bouring vertexes, types for agents and tasks, for example there can be heavy
lifting tasks which can be only be done with a strong agent or fragile stuﬀlifting
tasks which can only be done with a delicate agent.
More MAPF algorithms could be implemented and experiments expanded. Also
one can always look for ways to optimize an existing solution.
