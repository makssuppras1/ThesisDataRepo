Abstract
In recent years new architectures used for object detection have swiftly been developed and have
gone from being a niche to a commonly implemented method in various industries. This thesis
focuses on testing, implementing, and improving object recognition for autonomous robots
through deep learning models.
In this thesis, a dataset consisting of suitcases viewed from different perspectives similar to the
UAV, was created and used to train several models. The architectures were analyzed, and the
top three models, YOLOv4, YOLOv5, and YOLOv7 were selected based on accuracy, inference
time, scalability and availability. The results showed that YOLOv7 performed the best and the
compressed version had the potential to run in real-time on computationally limited hardware
- such as Nvidia Jetson. Further experimentation was performed with fine-tuning, synthetic
data, batch-size and network input resolution. The final solution was based on the results,
which used a network input resolution of 640x640, a batch-size of 64, and 20% synthetic images
in the training dataset. The final model achieved a mAP@0.5 of 72.22% and mAP[0.5-0.95] of
41.02%.

1.
Introduction
This next section of the report will provide an overview of the general organization of the re-
port. The main focus of the thesis will be outlined, the reasoning behind the research will be
discussed, and the thesis layout will be described in details.
1.1.
Problem statement
The Mohamed Bin Zayed International Robotics Challenge (MBZIRC) is a competition to pro-
mote the development of robotics and push the boundaries of what is possible with autonomous
robots.
Technical University of Denmark (DTU) is participating in the MBZIRC competition in which
a larger problem must be solved. The first part is the Inspection task which is the part where
an unmanned aerial vehicle (UAV) have to identify any vessels in a large ocean area and then
transmit the information to the base station on land. It then has to find a specific vessel and
detect certain objects on it. Once the items are located, multiple robots have to cooperate to
extract the objects. All of this is done by an autonomous system without any interaction from
humans.
The competition is split into multiple smaller sections as shown below.
1. Multi-UAV search and inspection of vessels
2. Intra-system communication and collective decision-making
3. Collective lift and transport of small and large objects to the unmanned surface vehicle
(USV)
4. System coordination for lift and transport of objects by the robotic manipulation arm
5. Proof of equipment feasibility and access. 1
The focus of this thesis will be a smaller part of the Multi-UAV search and inspection of vessels.
In this thesis, the problem that will be solved is object detection from the perspective of a UAV.
The UAV will fly above a ship and must correctly detect a specific object and its position
somewhere on the vessel.
Another aspect of the problem is that the proposed solution should be able to run in real-time.
This is important since the UAV needs to respond to the detection made from an object detec-
tion model while reacting to the surrounding environment.
This thesis will explore advanced object recognition algorithms and advanced computer vision
analysis tools to solve the problem.

1.2
Requirements
1.2.
Requirements
The main requirements for object detection are as follows.
a) The final solution should be able to detect the object and place the bounding box with
reasonable precision. The UAV will navigate toward the object, therefore it needs correct
classification and precise coordinates of the object.
b) The inference time should be appropriate for real-time since the object detection model
will be used by an UAV. Because of that it is important that the inference time is no
longer than the needed response time. If the object detection model takes too long to
reach inference, the UAV will not be able to react to the surroundings, thus, the models
should be optimized to run on hardware, such as raspberryPi or Nvidia Jetson.
c) The model should be able to predict the correct class from varying distances. The UAV
will inspect the vessels from varying distances because it will fly high above the vessel at
first and then move close to the vessel while detecting the object.
All the requirements above have to be considered for the final solution.
1.3.
Main challenges
The main challenges of this thesis are described below.
a) The angle from which the object will be viewed is rarely represented in the available
datasets. The UAV will be flying high above the vessels, looking down while trying to
detect the object somewhere below it.
b) The object’s size in the images will vary since the UAV will fly closer to the object. When
it is far away from the vessel, the object will appear small, and when the UAV is closer
to the object, it will appear larger in the images, which might cause issues for the object
detection model.
c) The object detector has to work in real-time. The UAV will have limited computational
resources, meaning that the chosen model-type has to be fast and lightweight enough to
be used by a UAV.
d) Getting images from the actual environment is not possible. The competition will be held
in Dubai with an object on a ship. Getting images of this environment is not an option,
since there are not enough resources available.

Limitations
The following project is part of a Master’s thesis, and it will be limited to the areas listed
below.

1.5
Assumptions
1. At the time this thesis is being done, some details of the MBZIRC competition are unclear.
This will be elaborated in section 1.5.
2. It will not be possible to validate the final model on images in the correct environment
since it is impossible to get images of the specific object, on the specific vessel and out

3. The project is limited to the period of this Master’s thesis.
4. New algorithms in object detection is rapidly developing, therefore some of the newest
research will not be included in this project. Because the training, implementation or
evaluation of these models can give errors that will be impossible to solve, it will not be
part of this project.
1.5.
Assumptions
Several details about the MBZIRC competition are, at the time of this thesis, still unknown.
Therefore this thesis will have to make assumptions for each of these details. The list below
will briefly describe the assumptions.
1. The object that should be detected will be assumed to be a suitcase.
2. The suitcase will be standing on the deck of a ship. It will be viewed from above by a
UAV.
3. The hardware the UAV is equipped with will be computationally limited, thus making it
a priority that the detection can be run at a decent fps.
These assumptions will be used throughout this thesis as the foundation for the problem.
1.6.
Structure
The structure of this thesis is as follows.
1. The first part of the report will explore the newest research in object detection and
briefly describe all the relevant theories related to machine learning, computer vision,
and hardware.
2. The second part of the report will consist of information and descriptions about the
process of training and implementation of the models and the results of this.
3. Finally, the last part will describe the iterations made to improve the model’s performance.
Additionally, this part will also analyze and evaluate the performance of the chosen model.

2.
Methodology
The following section will describe the main methods and techniques of deep learning, computer
vision, and the concepts related to object recognition.
2.1.
RGB-images
RGB (Red, Green, Blue)-images use a combination of base colors to create a wide range of
colors. Each pixel in an RGB image is represented by three numbers, one for each of the three
colors: red, green, and blue. These three numbers, also known as color channels, are combined
to produce the final color of the pixel. The value of each of these channels can range from 0 to
255, with 0 representing no intensity of that color and 255 representing the maximum intensity
of that color.
Each color is stored in a matrix, and together they form a matrix with dimensions height x
width x 3-channels. An example of such a matrix can be seen in figure 1.
Figure 1: A RGB-matrix with dimensions of 3x3x3channels
In addition to the RGB color representation other types of representation also exist. Among
the more commonly used ones are HSV (Hue, Saturation, Value) and CMYK (Cyan, Magenta,
Yellow, Black). All colors existing in RGB can also be represented in HSV and CMYK.

2.2
Deep learning
2.2.
Deep learning
The following section will describe deep learning in relation to computer vision or more specif-
ically, object recognition2.
If a human had to find a suitcase on a vessel, it would in many cases not be a challenge, but
when translating the same problem to computers, the problem is much more complex. Humans
can easily perceive shapes, colors, structures, and surfaces and can count apples on a tree and
see emotions on a face, but doing the same things through engineering is a complex problem.
These are typical problems that are solved with computer vision tools to perceive vision and
extract information.
In recent years there has been rapid development in computer vision combined with machine
learning, especially deep learning, which has shown promising results and is being used in a
great variety of industries today.
Deep learning is a sub-field of machine learning and artificial intelligence and is often used
when there are massive amounts of data available, because deep learning is great at generaliz-
ing complex data, thus making it a very viable method.
Before exploring deep learning further, some of the more classic machine learning concepts will
be reviewed, since the foundation of deep learning is based on these techniques.
2.2.1.
Unsupervised learning
In unsupervised learning, contrary to supervised learning, the ground truth is not needed. These
types of models are often used to explore and see if there is a pattern or distribution in the
data that might be hard to see.
A commonly used unsupervised algorithm is k-means clustering. The k-means algorithm will
try to cluster the data into different classes and any new data points will then be assigned the

Transfer learning
Transfer learning is a technique in deep learning that involves using the knowledge learned by a
pre-trained model on one task to improve the performance of a model on a different but related
task. This is typically done by taking the pre-trained model and fine-tuning it on the new task,
using a small amount of labeled data.

Transfer learning is commonly used in deep learning because it can significantly improve the
performance of a model on a new task, without requiring a large amount of labeled training
data. This is because the pre-trained model has already learned many of the features that are
useful for the new task, so it only needs to learn the remaining features.
There are several different ways to use transfer learning in deep learning, depending on the
nature of the new task and the availability of labeled data. One common approach is to use
the pre-trained model where the weights of the pre-trained model are ’frozen’ and only the
final fully connected layers are trained on the new task. This allows the model to learn task-
specific features without significantly changing the learned features from the pre-trained model.
Another approach is to fine-tune the entire pre-trained model on the new task. This involves
unfreezing some or all of the weights of the pre-trained model and training them on the new
task. This can be more effective at improving the performance of the model on the new task,
but it also requires a larger amount of labeled training data. Overall, transfer learning is a
powerful technique for improving the performance of deep learning models on new tasks, and
it is widely used in a variety of applications. [1, p. 526-531]

In object detection, datasets are commonly used for evaluation in a number of ways. One

3.1.
Two stage detectors
As the name implies, two-stage object detection algorithms use a two-step process to identify
objects in images. Firstly, they generate a set of potential object regions using a region pro-
posal algorithm. Then, they classify the objects in each area and assign them a bounding box.
These models are generally more accurate than single-stage models, but it comes at the cost
of requiring more computational resources. Two-stage object detectors tend to be slower than
one-stage detectors.
Examples of two-stage object detection models include RCNN and Faster R-CNN, which will
be explained later in the thesis.
3.1.1.
RCNN
RCNN (Region-based Convolutional Neural Network) is an object detection model used to
identify objects in images. It was one of the first breakthroughs in the field of object detection.
Firstly, the model uses a region proposal algorithm. It’s said in the paper that the model is
agnostic to the particular region proposal method, meaning that many different types of region
proposal methods will work. In the original article, they chose to use selective search to find

Afterward, the model uses a separate classifier to classify the objects and assign their bounding
boxes. The RCNN model can be trained end-to-end, meaning that the model can learn feature
extraction and object detection simultaneously. Only a couple of months after the publication,
the author proposed a new model called Fast RCNN, which included smaller architectural im-
provements. The new publication proposed to change the support vector machine (SVM) part

3.1.2.
Faster RCNN
Faster RCNN is an improved version of the original RCNN described previously. This model
also used a two-stage approach but suggested changing the region proposal method from selec-
tive search to a convolutional region proposal network (RPN) instead. The RPN is a neural
network that is trained along with the rest of the Faster RCNN model, allowing it to learn to
identify object regions directly from the training image, thus making the Faster RCNN model
more efficient than the original RCNN model, as it does not need to run a separate region
proposal algorithm.
Another difference is that the Faster RCNN model uses a region of interest pooling layer to
extract features from each region proposal. This allows the model to extract fixed-sized features
from each region, which can then be fed into the classifier. This is in contrast to the original
RCNN model, which uses a sliding window approach to extract features from each region.[44,
p. 304-306]
3.2.
One stage detectors
One-stage detectors, such as YOLO, attempt to predict bounding boxes and class probabilities
for objects in an image in a single pass through the network. This makes them faster and more
efficient than two-stage detectors, but they may not be as accurate. In contrast, two-stage
detectors first generate a set of region proposals and then classify each proposal in a second
stage.
One-stage detectors are generally faster than two-stage detectors since they can make predic-
tions in a single pass. In contrast, the two-stage detectors first have to generate the region
proposals before making the prediction, as was described in section 3.1. Furthermore, because
one-stage detectors tend to be less complicated, they are often easier to implement and com-
prehend compared to the two-stage detectors.
The speed and efficiency gained by the simpler network used in one-stage detectors come at
the cost of accuracy. Two-stage detectors are, therefore, favorable if the inference speed is not

Single Shot Detector
SSD (Single Shot Detector) is a one-stage object detector, meaning it does not use a region
proposal method, contrary to the two-stage object detectors mentioned in the previous section.
At the same time, it is easy to train and deploy, which makes it an excellent choice for already
existing systems.
The SSD uses a set of default bounding boxes with different aspect ratios and scales at each
location on the feature map. These default boxes are used to predict the presence of different
object categories in the image. At prediction time, the network generates scores for each default
bounding box, indicating the likelihood of whether a particular object is located in each of the
boxes.
The network also adjusts the default bounding box to match the detected object’s shape better.
This allows the algorithm to efficiently handle objects of different sizes and aspect ratios using
a single set of parameters.

Alexey continued to improve the YOLO network until he proposed the YOLOv4, which intro-
duced multiple further improvements, as described below.
1. The backbone of the network is still Darknet53, but in the article, it is proposed to use a
different backbone if the model runs on a GPU platform. The other backbones could be
VGG or ResNet, which are more accurate but computationally heavy networks.
2. DIoU NMS is used to remove overlapping bounding boxes based on a threshold on the
IoU between the two overlapping boxes.
This causes some overlapping objects to be
suppressed by each other’s bounding box.
3. path aggregation network (PAN) is an improved version of the previously used feature
pyramid network.
4. Self-adversarial-training (SAT) is a prominent data augmentation in which the input
image first passes through the network, similar to regular training, but then the image
is altered into a form that would be most detrimental for the model. Later, the model
has to learn from this rare combination of image and object, forcing the model to learn
special cases of the object.
5. YOLOv4 uses SPP-Net, which allows the model to process images of different sizes,
improving the model’s ability to handle input images of various resolutions.

dom patches from 4 images and combines them into a single image. These methods are shown

6.4.2.
Network input resolution and batch size
Another approach to optimize the model’s accuracy is to increase the resolution of the images
for training and for detection. Using higher resolution lets each image contain more informa-
tion and thereby provides the model with more features to look for, which often increases the
accuracy of a model.
As the resolution of the input images gets bigger, the time it takes to train the models drasti-
cally increases. Each time the input resolution double the actual information in the images is
4 times larger, e.g., if the input resolution has a height and width of 16*16 then the number of
pixels is 16*16=256, while having a height and width of 32*32 = 1024 pixels, which is 4 times
more information.[3]
The chosen resolutions that are being tested are 320x320, 416x416, and 640x640. Resolutions
above 640x640 take too long to train, and Colab has a threshold for an active session within
24 hours. 640x640 takes approximately 20 hours to train.
Furthermore, batch sizes can influence how well the model generalizes to data it has not seen
before. A larger batch size means that the model will converge faster.
The reason is that a larger batch size makes each epoch faster in terms of reducing the training
time. The downside is that big batch sizes are sometimes known to make the model’s per-
formance degrade[18], however, normally it is recommended to use the largest possible batch
size.[17]
It is important that the batch size is larger than 1 and smaller or equal to the number of images
in the training set.

7.
Discussion
The following section will discuss the results found in section 6 and the process of experimenting
and fine-tuning the model to increase the performance of the model.
Assumptions
Since the competition, MBZIRC, still has a lot of unknown facts about the requirements, hard-
ware and objectives, the foundation of this thesis is based on multiple assumptions.
The object that will be detected is a suitcase, but it is still unclear if this would also be the
object at the time of the competition. However, this thesis should be seen as a proof of concept,
meaning that if the object or setting is different from the ones in this thesis, the dataset can
be replaced by a new dataset, and the methods and concepts still apply.
Dataset
The dataset consists of a total of 101 real images used for training and validation. The im-
ages were captured using several different cameras in an environment that does not resemble
the actual scenario, in which the images will be taken from a UAV. This can potentially be a
limitation of the dataset as the lighting, texture and scale of the objects in the images may not
be representative of the actual scenario.
Despite this limitation, the dataset was chosen for its availability and ease of access. However,
it is important to note that the model performance may not translate well to the environment
in the competition, due to the mismatch in the dataset and the actual scenario.
When training an object detection model, it is standard practice to use images with the same
resolution and aspect ratio as the images that will be used as input for the final model. How-
ever, in this case, the camera model that will be used by the drone is currently unknown, which
complicates the process.
The dataset used for training and validation contains images of varying resolutions. The YOLO-
network automatically resizes the images by downscaling or upscaling the images while keeping
the same aspect ratio. The use of images with varying resolutions can negatively impact the
model’s performance since high-resolution images contain more information than low-resolution
images, meaning the model may have difficulty generalizing the information contained in these
images, leading to poor performance on new, unseen images. Furthermore, some images are
taken at nighttime, meaning that light, reflections, and textures are seen differently and might
decrease the model’s ability to generalize on unseen data.
Glenn Jocher, the author of YOLOv5, has a thorough description of how to create optimal ob-
ject detection on his GitHub. His suggestions to create an optimal model can be seen below.[16]
Page 62 of 71
==PAGE:70==
- Dataset should contain approximately 1500 images per class
- Dataset should have at least 10000 instances
- The images should match the environment in which the model is going to detect in
- Annotations should be tight around the object, i.e., no space between the object and the
bounding box
- The dataset should consist of 10% background images, without any classes in it
However, fulfilling all of the suggestions above is a difficult task if the dataset is unavailable
beforehand or not easily acquired through other datasets, web-scraping or other methods.
For this thesis, the images were not available beforehand, thus making it difficult to create
a dataset that complies with all requirements, due to the limitations mentioned in section
1.4. It is commonly known that the variable that will have the largest impact on the model’s
performance is data, thus making it highly recommended for the final implementation to collect
images for a larger dataset.[42]
Tuning hyperparameters
Furthermore, in section 6.4 it was attempted to improve the performance of the object detection
model by tuning the batch size and the network input resolution.
The model was trained with different batch sizes ranging from 16 to 64, however, it was found
that the default batch size of 64 provided the best performance. Using a larger batch size could
have led to faster convergence during training, but the batch-size can not be larger than the
number of images contained in the training dataset.
Similarly, experiments with different network input resolutions were tried, and it was discov-
ered that changing to a different resolution than the default resolution of 416x416, provided
a minor increase in the model’s performance. The experiment was done with both lower and
higher resolutions, and it was found that a higher resolution increased the model’s mAP@0.5
performance, however, it seemed to decrease the model’s mAP[0.5-0.95], which could indicate
that the model’s ability to place the bounding box precisely was worse.
In this thesis, the focus was testing specific hyperparameters as noted by the authors[3]. How-
ever, tuning additional hyperparameters beyond what was tested in this thesis could further
improve the model’s performance. Finding the optimal set of hyperparameters for a model
can be a challenging task, as it involves repeated experimentation and evaluating the model’s
accuracy through cross-validation, which can be highly time-consuming.
Synthetic images
The option of adding synthetic images as part of the training dataset for our object detection
model was investigated. They can be generated using various methods such as computer-aided
Page 63 of 71
==PAGE:71==
design (CAD) models, simulations, or generative models. In this thesis the focus was mainly
on using generative models, Dall-e 2 and Dreamstudio, to generate synthetic data.
Using synthetic images as part of the training dataset has several advantages. Synthetic images
can easily be generated in large amounts. This can be useful when the number of real-world
images is limited or if it is not an option to access images of the object in a specific domain.
Furthermore, synthetic images can be generated to match domain-specific variables, such as
lighting, poses, view, and backgrounds. These can help increase the model’s generalization
ability and make the model more robust.
Additionally, synthetic images are a very resource-efficient way of getting additional data, com-
pared to the time it takes to acquire images by capturing the object in the real domain. It
could prove very beneficial for a dataset with limited options for gathering images, which is the
case in this thesis.
However, using synthetic images as part of the training dataset also has some limitations. Syn-
thetic images will always have difficulty simulating images from the real world down to every
detail. Therefore, a model trained on synthetic data can have difficulties generalizing images
when deploying it in the real world.
The models used to generate the images also seemed to have difficulties generating images di-
rectly from above the suitcase, as the images always seemed to be from an angle of one of the
sides, which can also be seen in image 38. The figure shows the synthetic images as almost all
have a slightly off-centered view angle. This is probably due to the fact that images of suitcases
seen directly above are not available online, which is the main source of training data for Dall-e
2 and Dreamstudio.[26]
The experiments discovered that using synthetic images as part of the training dataset im-
proved the model’s performance on the test set at a specific ratio of synthetic images. From
the results presented in section 6.4.1, it appears that incorporating synthetic images in a ratio
of 10% and 20% into the training dataset slightly improves the model’s performance. However,
when the ratio is increased to 30%, the model’s performance appears to decrease.
The synthetic images were primarily images of suitcases seen relatively close-up, which could
explain why the performance is not increasing. The dataset could be unbalanced, so the syn-
thetic dataset might need further tuning to balance the dataset, to improve the model’s ability
to generalize images with variations.
Validation
From the results shown in section 6 it can be seen that the results from each model vary a
lot for each training, which makes it hard to estimate the model’s real accuracy. Normally the
Page 64 of 71
==PAGE:72==
estimation of a model’s performance is done through k-fold cross-validation, but as this thesis
had multiple limitations, this was also a limited method.
Often the results are very dependent on the splits between train- and test dataset, since some
of the images has several similarities, such as background, suitcase, and angle from which the
object is viewed.
One workaround could be manually hand-engineering the test and train datasets, but it would
still be difficult to know how the model generalizes real-world data.

Conclusion
This thesis aimed to make object recognition for autonomous robots, in the case of this thesis,
a UAV.
Based on the limitations described in section 1.4, it was decided to create a dataset of suitcases
seen from an angle similar to what the UAV will be viewing the object from.
In section 3, multiple object detection architectures were explored and compared.
Finally,
three models were chosen based on criteria such as inference time, accuracy, scalability, and
availability. The models selected were YOLOv4, YOLOv5, and YOLOv7, which all showed
excellent performance in inference time and accuracy compared to other models. Each model
also provides the option to use compressed models with high fps but lower accuracy.
The results in section 6.1 and 6.3.4 shows the results from each base and compressed model.
The overall results indicated that YOLOv7 had the best performance of the models. Addition-
ally, it was learned from [2] that the models could reach an FPS of 32 on hardware such as
Jetson Nano, which makes the model run sufficiently for real-time object detection.
Based on the results, it was chosen to continue working on improving the performance by fine-
tuning and experimenting with YOLOv7.
In section 6.4, experiments with synthetic data, changes to batch-size, and network input res-
olution were performed. The results showed that using synthetic images improved the model’s
performance compared to a model that was not trained with synthetic images.
Furthermore, the results also revealed that using a network input resolution of 640x640 improved
the mAP@0.5 of the YOLOv7 model, but compared to a model using a resolution of 416x416,
it had a lower mAP[0.5-0.95]. The experiments with different batch-sizes showed that using
higher batch-sizes, such as 64, enhanced the model’s accuracy, even though the author of [18],
says it can cause a decrease in the model’s ability to generalize well.
The final solution is based on the results from section 6.4, which uses a network input resolution
of 640x640, batch-size of 64, and 20% synthetic images in the training dataset.
The results showed that the model performed better by adding synthetic images to the training
dataset and configuration changes. The final model had an estimated mAP@0.5 of 72.22%,
which is the highest of all the models trained. Meanwhile, the model has an estimated mAP[0.5-
0.95] of 41.02%, the second highest among all the models.
However, in section 7, the estimation of the model performance was discussed. Due to the
limitations mentioned in section 1.4, the actual performance of the models could be different
from the estimated results. Furthermore, it was also discussed that the dataset could easily be
unbalanced due to the size of the dataset, which could give the model varying accuracy when
detecting from different heights. Finally, based on the results, the model’s overall performance
could likely be improved by gathering a larger dataset.

Dataset
Based on the results achieved in this thesis and the suggestions from Glenn Jocher in [16], it
would make sense to focus on creating a more diverse dataset of suitcases. Furthermore, getting
images from suitcases in a domain closer to the one from the MBZIRC competition, which is
on a boat, would also help the model’s performance and stability while detecting objects at
varying ranges. It could also be an idea to experiment further with synthetically created images
to increase the model’s performance.

A larger dataset would also help with the evaluation since the final model might be less depen-

It could also be a focus to create a dataset with images captured by the same camera as the
one that the UAV will use in the MBZIRC competition.
Fine tuning and active learning
Below is a list of hyperparameters that could also be tuned for future improvements to the
network. However, the time and resources invested in it may not produce a worthwhile result.
- Try different activation functions
- Try other optimizers, such as Adam, RMSprop or SGD
- Learning rate
- Momentum
- Adding even more data augmentations
- Freezing different layers
All of the hyperparameters above can be changed in the configurations file for YOLOv4 and
YOLOv7, but for YOLOv5 it is defined in the YAML-file.

Implementation
As the model will have different performance based on the hardware it is implemented on, it
makes sense to implement it on the actual hardware that the UAV will use in the MBZIRC
competition. Implementing the model on the hardware will also reveal the model’s actual per-
formance in inference speed and accuracy.
Furthermore, changing the network input size to the same size as the images captured from the
UAV is usually recommended to achieve the best performance from the model. In this thesis,
the model had to re-scale the images at each iteration, which caused training and inference to
be slower than if the images had the same size as the network input size.
