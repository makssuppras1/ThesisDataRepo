Abstract
Deep learning has reached a stage where it can surpass humans in specific tasks, such as
speech recognition and keyword spotting (KWS). Neural networks are moving away from
the cloud and onto smaller devices, however resource­constrained devices have shown
to be a challenge due to their lack of memory availability. Hearing instruments only have
about 100KB of memory available. Network parameters (weights and biases) must be
reduced to fit the memory requirement posed by such devices.
In the following paper, methods such as logarithmic quantization and Huffman encoding
were applied to compress parameters of a network significantly without greatly impacting
accuracy. Converting parameters to the logarithmic domain not only reduced the required
memory, but also changed the arithmetics of the normal linear equation, as the multipli­
cation in the linear domain was replaced by an addition in the logarithmic domain. Two
implementations of the accumulation stage was investigated, linear domain accumulation
and logarithmic domain accumulation. Linear domain accumulation required conversions
between domains, while logarithmic domain accumulation was approximated.
By using logarithmic quantization with linear accumulation, the KWS Deep Neural Net­
work (DNN) chosen was reduced in size by 5.3x compared to FP32, or 1.3x compared to
int8 implementations. The benefit provided by replacing the multiplier reduced the energy
impact of the computations by 65x compared to FP32 or 3x compared to int8 implemen­
tations. All with minimal loss in accuracy at a top 1 accuracy of 76% compared to a 81%
top 1 accuracy of its FP32 implementation.
As this method can be applied to any network, a solution is found in logarithmic do­
main quantization in order to compress neural networks for implementation in resource­
constrained devices.

Introduction
1.1
Deep Learning in Mobile Devices
Deep learning algorithms have reached a stage where they surpass humans for specific
cognitive tasks, such as speech recognition [1],[2] and image classification [3]. These
have seen many uses
One implementation of image classification is self­driving systems such as Tesla’s Autopi­
lot [4]. This particular system involves 48 neural networks at this time, which output 1000
different predictions every ”timestep”. These networks are running on their own hard­
ware located in their cars. This hardware must live up to power constraints defined by its
environment in order to not excessively drain the battery of the car.
The most visible implementations are smart assistants such as Alexa, Google Home and
Siri. These perform keyword spotting on device to detect certain phrases; ”Alexa”, ”Okay
Google”, ”Hey Siri”. After initial detection, these are predominantly run in the cloud by
large data centers to perform speech recognition.
Recently Apple has moved the speech recognition of Siri out of the cloud and on device
[5]. This was done due to privacy concerns and to offer reduced recognition latency.
With these networks running on a mobile device, expansion of these are limited without
compromises. Certain network types must either be reduced in size to fulfill memory
or compute operations constraints, or new network types must be invented to mitigate
these constraints as shown in [6] and [7]. In the latter, showing additionally how different
Keyword Spotting networks can have a profound impact on these constraints.
Taking these implementation constraints even further into resource­constrained devices
such as microcontrollers or hearing instruments, provides a challenge due to their limited
memory availability.
Therefore an emerging issue within the space of deep learning is reducing memory and
computational complexities of neural networks in resource constrained devices.
This project will explore implementation of neural networks in such resource­constrained
devices.

2.1
Neural Networks
To explain neural networks, first machine learning must be introduced. ”Machine learning
is a branch of Artificial Intelligence and computer science, which focuses on the use of data
and algorithms to imitate the way that humans learn, gradually improving its accuracy”[8].
That is its definition according to IBM. A subcategory within Machine Learning is what is
known as Deep Learning. Machine Learning traditionally required human intervention for
most feature extraction. Within Deep learning this extraction is automated.
Neural networks, known as Artificial Neural Networks (ANN) are a subcategory within
Deep Learning. Their name and structure are inspired by the human brain, in an attempt
to recreate certain functions. This is done by linking nodes, called neurons, as in the
human brain. Neural networks can be used in two forms, inference and training.
Training a network is most often done using labeled datasets by large servers, with the
goal of making a network better at particular tasks. When a network is trained, i.e. it has
learned its parameters, known as weights and biases, it can be used to run inference.
Inference does not tweak parameters of the network, it simply uses the learned param­
eters and calculates an output. Many devices can be targeted for inference. This report
investigates inference on resource­constrained devices.

2.1.2
Training
Training a network is done by first completing a forward pass or propagation of the net­
work, and then what is known as a back­propagation or backward pass. Neural networks
train using loss functions and optimizers. Loss functions determine how wrong a pre­
diction is from the true predictions using pre­labeled data and back­propagate this loss
through the network.
The goal for an optimizer is to find the parameter combination at which the loss function
provides the lowest loss i.e. its global minimum. [9]. Many loss functions exists as well
as optimizers. The use case of loss functions can be divided into regression losses and
classification losses. For this project a particular loss function is not important, as training
is not a focus.
Different optimizers all stem from Gradient Descent, which looks at the change in loss,
in order to determine whether to take a parameter up or down in value [10]. This form of
optimizer works on every sample.
A ”learning rate” is given to the optimizer. This acts as a limit of how much a parameter
is allowed to change in value per iteration. Versions of optimizers based on Stochastic
Gradient Descent are applied on a batch­level or on a random amount of samples, in
order to smooth out loss variations over time [11].
A batch is a collection of input samples. The total amount of batches is determined by
the size of a batch divided by the size of the dataset, e.g. a batch size of 16 and dataset
size 64 results in 4 batches. When a batch is given to a neural network during training,
an optimizer based on Stochastic Gradient Descent will evaluate over the average loss
of the predictions made with these batches, thus ensuring better generalization.
One training pass is known as an Epoch. This is when the entire training set has passed
through the network. When training a network, multiple epochs are done.

2.1.3
Inference
When running inference on a device, inputs are given without predisposed knowledge
of the output. The input is sent through the network, and a prediction is created by the
network. This is known as forward­propagation. Inputs are treated individually [12] as
opposed to batches as explained in section 2.1.2.
While training is most often done using labeled data in order to determine a loss, infer­
ence takes an input and provides a prediction. This prediction is calculated using the
parameters found during training.
Running inference on a device is thus the final form of a network. The only goal of which
is to process inputs as fast and efficient as possible.
2.1.4
Types of Networks
The way with which ANNs work can differ between implementation and intended purpose.
Some types of networks are better at certain tasks compared to others. So far what
has been discussed are feedforward or fully connected neural networks. However many
more exist. Popular network types are Convolutional Neural Networks (CNNs) as well
as Recurrent Neural Networks (RNNs). CNNs are similar to feedforward networks, and
are particularly great at pattern recognition, especially in images [3] but have also proven
themselves to be strong in audio recognition [13]. CNNs work by applying a filter or a
kernel over an entire image or input. These filters can vary in size and are trainable.

Challenges with Neural Networks in
Resource­Constrained Devices
In order for a particular network to predict more classes, or achieve high accuracy, the
size of this network must increase as shown by [6] and [7].
As feature set and accuracy of neural networks increase, so does the size of the network,
and with that the amount of parameters needed to be stored. This is true within the same
network implementation, but a different type of network might perform better and have less
parameters. However scaling these to become even more accurate will again increase
the amount of parameters.
Resource­constrained devices present with the two important challenges; a limited mem­
ory footprint, and limited compute resources.
Large memory implementations to accommodate the increase in network size are thus
very costly in resource­constrained devices, such as hearing instruments or microcon­
trollers. This means that not only is computation time and power a limiting factor, but also
more crucial is limitation of how much memory can be made available. This is apparent
with hearing instruments having up to 327KB of memory [16], and typical microcontrollers
only having tens to a few hundred kilobytes of memory.
Due to this limitation, current popular Neural Network implementations such as single­
precision and reduced floating­point formats become difficult to implement in resource­
constrained devices, or they have severely reduced capabilities.
An example is given in [6] which describe MobileNets developed by Google. MobileNets
are special scalable CNNs, which can scale both input resolution as well as size, in terms
of MAC units. Analysis of these networks shows how their precision is directly proportional
to the size of them and thus parameters to be stored on device.
Another example is given by [7] which objective is to demonstrate different types of neural
networks, and their impact on microcontrollers. This paper again shows that more param­
eters, as well as more operations, thus a larger network, yields a higher accuracy. This
paper also shows that choice of network type has a large impact on required memory size
as well as operations. The paper clearly shows the trade­offs that accuracy, memory size
and network size has for each network type. A network type with fewer parameters require
more computations, while a network with more parameters require fewer calculations, to
provide the same accuracy.
Hardware implementations can be optimized to reduce time and power needed for compu­
tations while maintaining accuracy, but removing parameters of the network will diminish
its accuracy significantly, as proven by [6] and [7].
In the end it means that the implementation complexity in terms of size of Neural Networks
in resource­constrained devices are limited by available memory, therefore, despite opti­
mizations to the arithmetic implementation, memory availability will remain a bottleneck.
A method one can apply to reduce model size dependency on memory availability is data
size compression. Which this project will look into.

2.3
Compression
Data size compression is a method with which data is encoded as to reduce the bit­
size of the data. When discussing compression methods, these fall within two categories;
Lossless compression and Lossy compression [17]. Compression algorithms within these
categories can be compared by a so called compression ratio. Determined by:
Compression Ratio = Uncompressed size
Compressed size
(2.4)
One often used compression method employed for specialized Neural Networks are int8
implementations or 8bit quantization. This brings its own limitations such as the need to
scale weights, biases and inputs, in order to minimize loss due to reduced dynamic range.
Thus a way to reduce memory requirements and hardware complexity while maintaining
precision needs to be determined.
2.3.1
Lossless compression
Lossless compression algorithms maintain full reversibility. This can be done by analysing
a dataset statistically and assign encoded values instead of the value itself. These algo­
rithms are often applied to data storage where loss of data would be catastrophic.
One such method is Huffman encoding [18], where a model is generated based on the
statistical occurrence of certain values or symbols, such that values that occur more often
are represented with the least amount of bits.
Huffman devised a way to represent individual symbols according to their frequency, and
thus provide lossless compression. In text compression, this method has been proven to
be the most efficient compression on symbols [19].
This will be explain in further detail in section 4.2.
2.3.2
Lossy compression
Lossy compression algorithms do not maintain full reversibility, however these often pro­
vide a higher compression ratio compared to lossless compression, due to small variations
in values being generalized.
Lossy compression is most often used in imaging and audio applications. Compression in
these spaces are designed around the perception of the data, as an example in terms of
audio and psychoacoustics, human hearing has a non­linear frequency response, mean­
ing human ears are more sensitive to certain frequencies, from this a ”mask” enacting as
a threshold can be created. By filtering out any frequencies that fall below this threshold
data size is compressed.
The most common algorithms exploiting the perception of data for these applications are
MP3, MPEG, and JPEG.
Another common use of lossy compression is quantization, in which a higher bitwidth
value is reduced to a lower bitwidth. This will be explained further in section 4.1.

Previous work
Many works have researched ways to reduce data sizes in a neural network, such that
they take up less space is resource­constrained implementations. Primarily these have
been focused around larger fixed­point implementations. Other methods such as Prun­
ing, which is the targeted removal of weights or neurons below a certain threshold, has
also been investigated with much success. Additionally, different ways of quantizing pa­
rameters to a smaller size have also been investigated, with this the extreme Binary­net,
limiting parameters to either +1 or ­1.
3.1
Fixed­point
One particular paper investigating Deep Learning with limited precision is [20]. In this pa­
per the training and in turn accuracy of networks are compared using different integer/frac­
tional splits in 16 bit fixed­point numbers. Additionally the paper investigated whether
this difference created by the fractional split could be mitigated using different methods
of rounding. The paper investigated Round to nearest rounding as well as Stochastic
rounding.
Round to nearest rounding simply rounds to the nearest possible binary representation of
the number, while Stochastic rounding will round depending on a calculated probability.
This probability is proportional to the proximity of x to ⌊x⌋. Due to this randomness the
expected rounding error is zero.
The paper found that having a fixed­point split with 14 fractional bits, using Round to
nearest provides close to FP32 performance, while for Stochastic rounding, every frac­
tional size, as low as 8 bits, performed very close to FP32 performance, thus completely
negating any rounding error.

Neural Networks on Microcontrollers
The last paper [7] investigates neural networks in resource constrained microcontrollers.

They found that the 8 bit quantized Deep Neural Network implementation required 80KB

the network.

ments using different types of networks rather than standard compression methods.
When considering low powered resource­constrained devices, finding a trade off between
memory and operations required is important.

Quantization
Quantization is a form of lossy compression with
which, given a desired bit size, additional bits are
cut off and the value rounded or truncated to the
nearest representation at this bit size. This creates
a ”Quantization Error” which is the loss of data com­
pared to its original representation.
In this project different Quantization is used in the
linear­domain as well as the logarithmic­domain.

the resulting quantized values will be linearly dis­

4.1.2
Logarithmic­domain Quantization
Quantization in the logarithmic­domain is slightly different comparatively. Values quan­
tized with this scheme will be logarithmically distributed in space of bandwidth 2bw. Due
to this logarithmic distribution, this method of quantization offers a greater amount of pre­
cision around the center, in a normal distributed dataset.
This method of quantization is done in almost the same way as linear­domain quantization,
however it has some differences due to the nature of the logarithmic function.
Firstly, if one were to take the logarithm of a negative number, this would return a complex
value, which is unwanted. Therefore the value to be quantized must be treated such that
its original sign is stored before quantization in order to recreate the value with its proper
sign.
Next, there is no need for a ∆LSB function, because rounding of a logarithmic number
is the same as rounding the exponent of the logarithmic base. This rounded exponent
ensures that values are logarithmically spaced.
With having stored the sign of the value to be quantized, log2(abs(x)) is now taken to
convert the value into the logarithmic domain. This converted value is then rounded. This
rounding could be either to the nearest integer for easier handling or it could be rounded
to contain a fractional part. Rounding to a fractional part will provide higher precision but
still keep bitwidth low while keeping the accuracy relatively high with normal distributed
datasets compared to linear­domain quantization.
As the value conversion is now handled, the range must be established. Again here are
some differences compared to the linear­domain implementation. Since a logarithmic
number rounded by integer will correspond to the position of the leading one in its linear
representation, a range in linear space of [0, 0.5] is equal to [­∞, ­1] in logarithmic space,
as log2(0) = −∞and log2(0.5) = −1. Since ­∞is close to impossible to implement, the
lower boundary is determined by the chosen FSR and bitwidth, bw. With this in mind, the
boundaries become [min, max] = [FSR −2bw, FSR −1].

4.1.4
PyTorch Quantization
PyTorch offers three different types of built­in quantization. Each of these present different
use cases, and can be used to easily convert a FP32 network to int8 [26].
Dynamic Quantization
Dynamic Quantization in Pytorch keeps everything in its regular FP32 format, but then
converts it to int8 at compute time. This method thus provides the benefit in speed of int8
calculations, but the drawbacks of storing everything in FP32.
Post­Training Static Quantization
Once training is done, the parameters of the network can be quantized as they are. This
is called Static Quantization. This is a great quantization methods for pretrained networks
where the user does not have additional data to train the network. It converts the entire
network to int8, ready for CPU or mobile implementations.
Quantization Aware Training
One interesting quantization method provided by leading machine learning tools such as
Pytorch is Quantization Aware Training (QAT). With this quantization method, so called
”Observers” are inserted into the model, which introduce quantization noise, as well as
observe the inputs and outputs during training of the network. When the network is quan­
tized to 8bit, this information gathered by Observers are used to determine scaling of the
quantization range to maintain as high accuracy as possible. This type of scaling can
be compared to the FSR implemented previously in the linear­ and logarithmic­domain
quantization.
As this project is searching for a solution other than int8 Post­Training Static Quantization,
this is ruled out as a test candidate. Dynamic Quantization only changes the network to
int8 but keeps weights and biases in FP32, thus this is not a viable candidate for memory
reduction. Lastly Quantization Aware Training, sounds like an interesting and robust form
of quantization, and is thus a viable test candidate, to investigate its impact on accuracy.

Huffman Encoding
The representation Huffman created is known as a prefix code or prefix­free code, in
which a bit­string representing a unique symbol is never the prefix of another symbols’
representation. Meaning, if ”a” where to be represented by the bit­string ”0”, then no
other symbol representation is allowed to start with a ”0” in their bit­string.
4.2.1
The Algorithm

To achieve this prefix code, Huffman used a binary tree with
2 types of nodes, leaf­nodes and internal­nodes. A leaf node
is an endpoint. This node contains the symbol to be repre­
sented as well as its frequency. An internal node is an inter­
mediary node. This node contains references to its left and
right child, as well as the sum of frequencies for these.
Huffman first organized symbols in a priority queue of leaf
nodes based on their frequency (fig. 4.3a), meaning how
often they were used, using a min­heap. This would ensure
these were organized such that the lowest frequency had the
highest priority.
To create the Huffman tree, the two highest priority leaf nodes
in the queue are taken out or ”popped”. Using these nodes,
an internal node is generated, placing the highest priority leaf
node as the left child and the next highest priority leaf node as
the right child. Lastly, this internal node is placed or ”pushed”
back into the queue, maintaining min­heap order (fig. 4.3b).
This is repeated until only 1 element remains in the queue
(fig. 4.3c & 4.3d). This last element is then popped, and is
the root of the Huffman tree.
4.2.2
Encoding
Using this tree, it can be traversed, in search of all leaf nodes.
From this a lookup table is created, describing the path, and
thus the encoding of each symbol in the tree. The encoding
is such that every left child taken corresponds to a ”0” and
every right child taken corresponds to a ”1”. The internal
nodes ensures that this encoding remains prefix free.

Log­Linear Layer
A variation of the regular linear or fully connected layer as described in section 2.1.1, that
this thesis will investigate is the Log­linear Layer. This layer combines logarithmic­domain
arithmetics in the structure of a linear layer.
Doing this, logarithmic­domain quantization can be applied to weights and biases in order
to compress their size for storage on device. The log­linear layer also yields the benefit of

4.5
Logarithmic­Domain to Linear­Domain Conversion
In order to convert from the logarithmic­domain to the linear­domain, the function α · 2x
is applied, where α is the linear sign of the logarithmic number to be converted. This
conversion is easily done in software, however to execute this function in hardware it
must be further investigated.
If this function were to only deal with integers, this would be fairly easy to implement as it
would result in y = 1 << x. However this function must be able to deal with fractionals,
which is not as straight forward. Using the product rule for exponents, it is known that:
an · am = an+m
(4.15)
Furthermore, for the function 2x, an input range of [0,1) yields and output range of [1,2),
therefore the value x can be split into two parts. The first part is the integer i of the number,
which is easy to put to the power of 2 in hardware. Next is the fractional or decimal part d.
This will be used to create a lookup table, which returns q = 2d. As this table depends on
the fractional as input, the amount of entries it contains will depend on the bitwidth of d.
When q = table(d) has been determined, applying equation 4.15, the final approximation
will be:
y = 2d · 2i = table(d) · 2i = q << i
(4.16)
Which is easily implemented in hardware.

4.6
Linear­Domain to Logarithmic­Domain Conversion
Converting from the linear domain to the logarithmic domain is again easily done in soft­
ware but it is more difficult to do in hardware.
When taking log2(x), this essentially returns the positions of the 1s in relation to the leading
one of x.
As an example, if the linear domain binary fixed point value x = 0100.0000, then taking
log2(x) = 2.0. This is the position of the leading one in the binary representation. Thus
finding the position of the leading one is the same as the integer of the logarithmic conver­
sion. However, x might contain multiple ones, which are harder to estimate the logarithmic
value of in hardware.
If one were to first find the position of the leading one, and then do a table lookup from
this position down, this would mean the input for the table would lie within the range [1,2)
giving an output in the range [0,1). Meaning that the input will always be at least 1 and
always below 2, the lookup table only needs to have the remaining bits as input, thus
excluding the leading one.
The position of the leading one is equal to s, and the remainder, excluding the leading
one, is d. Thus a table can be created for
table(d) = q = log2(1 + d)
(4.17)
The approximation of log2 can now be completed by:
log2(x) ≈s + q
(4.18)
where s is the position of the leading one, and q is the result of the lookup of the remainder
d.
The table depth and thus the size of d required for the estimation of q is user controlled,
but a higher depth will yield a higher accuracy.

As explained in section 4.4, two models are of interest in the logarithmic domain, i.e.

6.2.1
Handling Zero Values in the Logarithmic­Domain
As mentioned in section 4.1.3, it is crucial that zero values in the linear­domain are han­
dled in the logarithmic­domain. One easy way to do this in software is by assigning the
conversion of ”0” to the logarithmic domain, as a NULL. When doing this, every piece of
the arithmetic in the network must handle NULL values appropriately, complicating some
functions. The NULL value in the logarithmic domain, corresponds to a 0 in the linear
domain.
6.2.2
Creating the Model
To create the model, first a logarithmic linear layer must be defined in code using PyTorch.
This is fairly simple, as everything it requires is changing how the forward pass of the layer
functions.
Logarithmic­Domain Arithmetic
As this linear layer consists of matrix multiplication of the input vector, x, and the weights,
w, as well as an addition with the bias, b, this must be replicated in the logarithmic­domain.
Using the arithmetics described in section 4.3, a function logmult(x,y) is created. This
function does multiplication in the logarithmic­domain as described in 4.3.2. Additionally,
to this description a condition if either value is NULL is added such that the product also
will return NULL. In order to turn this into a matrix multiplication, an accumulation step must
be added and this is where the two models differ.
Accumulation in the logarithmic­domain is done using a ∆±(d) function approximated by
bit shifting as shown in 4.11a and 4.11b. From this, the function logadd(sn,pi) is created.
To this function, NULL handling is added such that if both input values are NULL, the output
must be NULL. If only one input value is NULL, the other value is the output.
Accumulation in the linear­domain is in some ways more simple. As the pi value is size
limited, but the accumulation itself isn’t to the same extend, this cutoff must be upheld.
Thus two functions are created. One function simply converts a value from the logarithmic­
domain to the linear­domain, while the other function ensures that the value is of its correct
size before conversion.
The function then becomes logaddfull(log2lin(x),log2lincut(y,size)). The size
parameter will be explained when choosing parameter formats, as it has an impact on the
accuracy of the calculation.
Now two versions of the matrix multiplication has been created, additionally the bias is
handled as the kind of addition each network uses.

Output Size of Conversion
In order to determine the output size of this conversion, it must be estimated what range
the output would regularly lie within. To do this, once again FSR must be considered. This
is primarily due to the fact that if a conversion of ˜pi to pi, based on the size of ˜pi was done.
The resulting size of pi would be b64.8, as the logarithmic size of ˜pi = b6.3 is simply
converted to the linear­domain, thus the integer size would be 26 = 64 and fractional
23 = 8. However, due to the FSR of ˜wi and ˜xi many of these bits in the conversion would
never be filled. This is due to FSRwi being 0, and FSRxi being 5. What this means, is that
no positive iteration of ˜pi can be above 5 in the logarithmic domain. And as the logarithmic­
domain is an encoding of positions from the linear­domain, that means at most the 5th bit
in the linear representation would be filled. Adding the linear sign of the logarithmic value
to be converted, a total of 6 integer bits are needed.
To determine the amount of fractional bits, again FSR plays a role. The minimum value
represented in ˜wi is ­15, and in ˜xi it is ­10. Thus it is possible for a bit to be found on the ­
25th position in a linear­domain fixed­point representation. In reality this value would be so
absurdly minimal that it would bear no influence in terms of accumulation. So arbitrarily a
fractional size of 10 bits was chosen, such that the input for accumulation is 16bits large in
the format b6.10. This size means that the range [­10,5] can be represented in logarithmic
space.

significant error into the calculation. Three sizes were tested, 4 bit, 5 bit and 6 bit. To this

approximation will be worse. This is due to the fact that an approximation can only be
made so close to 0, as the approximation is limited by what is presentable by the output
bitwidth, namely any value between 0 and 0.125.

7.2
Logarithmic multiplication
Figure 7.2: Logarithmic multiplier design
The design of the logarithmic multiplication is fairly simple and one of the benefits of a
logarithmic implementation. It is simply an addition of the input variables, as well as an
XOR of their linear­domain signs. Due to rules for sizing during fixed­point arithmetics, ˜z
will be 1 integer bit larger than the input containing the largest integer size. In the case for
this implementation, ˜x is of size b5.3, and ˜y of size b5.0. Therefore ˜z is of size b6.3. As
only one variable contains a fractional part, these are essentially passed through to the
output. Thus the adder in this design is a 5 bit adder.
7.3
Logarithmic­domain to Linear­domain
Figure 7.3: Logarithmic­domain to linear­domain converter design
In this design, the table found by implementing section 6.3.1 is used as a lookup table
of 8 entries and a data width of 8 bits. The input to this table comes from the fractional
part d of ˜x. The output of the table q and the integer part i of ˜x is sent to a shifter, to shift
q by i (q << i). The resulting value from this shift ys is then sent to a multiplexer. This
multiplexer decides according to the linear­domain sign sz whether y = ys or y = −ys.
In this design, the size of ˜x is b6.3, such that d is 3 bit and i is 6 bits. The output of the
lookup table q is 8 bits. The resulting approximation ys as well as y is in the linear­domain
and of size b6.10 as described in section 6.3.1. Additionally to this design is handling of
a NULL input value as detected by the ”none detector”. If a none value is detected, this is
passed along to the converter, and a ”0” is output into the linear­domain.

7.4
Linear accumulation
Figure 7.4: Linear­domain accumulation design
The design of the linear­domain accumulator is a standard accumulator. Given a single
input, an output is calculated. This output is then fed back into the accumulator to be
added with a new input. The register on the output of the adder saves the intermediary
sum, in order to accumulate. On the accumulation input to the adder is a multiplexer. This
multiplexer is controlled by the FSM. The multiplexer can either send the previous sum
through to the input of the adder, or it can reset the accumulation line to 0, such that a

7.5
ReLU design
The job of the ReLU is to detect negative values and set these to a NULL value. Since
this is not possible in hardware, a value must be arbitrarily selected to represent this.
Essentially this must be chosen such that its value will never naturally happen within the
network.
Setting the sum from the linear accumulation to all 1s, will create a converted value which
is not possible. This is due to its logarithmic­domain conversion would result in a value of
6. This value can never naturally happen due to FSR. Additionally if the output sign in the
linear­domain to logarithmic­domain conversion is set to 1, thus representing a negative
number, this makes the combination easily detectable, as all activations should be positive
as enforced by the ReLU.
As the linear­domain sign is not important for the linear­domain to logarithmic­domain
conversion, due to the fact that valid values always will be positive, the size of the sum is
reduced from b7.10 to b6.10.

7.8
Environment
Figure 7.9: Environment for the circuit
The circuit as described is placed within an environment containing the stored weights,
biases as well as an input vector. In order to always have data for the network these stored
variables are loaded into a FIFO. A controller is connected to each FIFO, which can have
two settings, either it is an input or an output. If set to be an input, the FIFO controller
will make sure the FIFO is always full, and the FSM of the circuit will communicate with it
to load a value from it. If set to be an output, the FIFO controller will make sure that the
FIFO is always empty, and has logic to store the value into a RAM using a write address
and a write enable flag. The FSM will communicate with the controller when a new value
should be stored into the FIFO.
7.8.1
LUT
The LUTs holding the data are simply initialized from a memory file. There are two types
of LUTs per variable, the value LUT and the sign LUT, containing the logarithmic value
and the linear sign respectively.
7.8.2
FIFO
The FIFO is instantiated as a synchronous RAM with a head and a tail. Each FIFO con­
tains the outputs; rd_valid, empty, empty_next, full, full_next. These can be used to de­
termine firstly when the value of a read is valid, especially used for saving the output of
the network. Secondly these are also used as control signals for the FIFO controller, to
determine if it is full if it is an input or empty if it is an output.
7.8.3
FIFO Controller
Lastly, the FIFO controller. It is in charge of filling and emptying FIFOs. This is done in
2 processes; write_to_fifo and read_from_fifo. If the FIFO is placed as an input to the
circuit, write_to_fifo will constantly check if the FIFO is full. If not, it will write to the FIFO
and increment the write address, used to receive an output from the LUT associated with
the FIFO. This address will loop when the max address is reached. This is done such that
a the input vector is simply looped per neuron. While the FIFO is positioned as an input,
read_from_fifo will wait for a signal coming from the FSM within the circuit.
If the FIFO is placed as an output to the circuit, write_to_fifo will wait for a signal from the
FSM. Meanwhile read_from_fifo will check if the FIFO is empty, if not the controller will
read from the FIFO and thus output the value to a sink, such as a RAM waiting for data or
another instantiation of the circuit put in serial with the first layer. However this could also
be handled without a FIFO. The FIFO merely acts in this context as a sink for the value,
giving a structured method to save it.

Discussion
Table 9.4 shows a prediction by the hardware structure implemented in software and the
prediction made by hardware. It can be seen there is a discrepancy between the negative
numbers even though they should be identical. The difference between these two are that
the software can run in full precision, while the hardware is limited. Another difference
is that the hardware only uses rounding in the lin2log calculations, otherwise it simply
truncates. However this do seem like a large difference from rounding errors.
In section 9.1 The hardware structure in software is compared against the Pytorch model.
Although the prediction is the same, the values generated are far from the same. This
is most likely caused by how the Pytorch model is handled, as the Pytorch model does
not understand the logarithmic entity created, essentially a struct. This struct must be
converted to linear domain between calculations and brought back into the logarithmic
domain during calculations. All these conversions not only make the calculation extremely
slow, but also adds an extra large source of error. It is thus assumed that the hardware
emulation is actually more accurate than the pytorch model, however no testing to support
that claim has yet been done. This would require implementation of a function to handle
big datasets for the hardware emulation.
The goal of the project was to investigate a method which could reduce memory require­
ments of a neural network. When looking at table 9.8, this goal can be said to have
been reached. The network implementation succeeded in reaching near similar accuracy
numbers as the original implementation, while compressing the parameters of the int8
and FP32 network by 1.33x and 5.33x respectively. Additionally, converting the multiplier
normally required by such neural networks to an adder achieves a significant drop of 3x
in energy required by the computations for the network compared to int8, or 65x when
compared to FP32.
The latency while running at the latency of the network was only 0.84ms for a full forward
pass. As a new input is only given every 40ms, this is way beyond what is needed, giving
the implemented network a lot of flexibility if features like job scheduling was desired as to
not cause power spikes. As the latency is way below the requirement of 40ms, one could
look to making it more efficient in terms of memory and energy per operation while still
being below the requirement. However as the network is already running serially, thus a
smaller footprint than if it were to run in parallel, this could be tricky. It would have to end
up at the individual compute implementations within the network.
One thing that should have been investigated more is the full logarithmic network. During
initial testing this network presented with an accuracy of about 20%. This was deemed
way to low to continue pursuing. This low accuracy turned out to have been a bug in
the implementation, which when fixed provided very competitive numbers for the Pytorch
model of 69% top 1 accuracy. Such an implementation would completely remove the
need for inter­network conversions, and thus probably result in an even faster, smaller
and more energy efficient implementation.
Another method which should have been investigated further was Huffman encoding.
Once again there was a bug during initial testing, that made it seem undesirable for the
particular implementations. However with this bug found, it showed that it could compress
the values of the parameters used in the logarithmic implementation to an average size

Conclusion
Different methods to reduce the memory impact of a neural network in resource­constrained
devices was explored. Two different logarithmic models were presented, one being a com­
plete logarithmic domain network, and the other being logarithmic in nature with linear ac­
cumulation. These were each implemented in Pytorch, in order to determine accuracies
and to explore how to best implement them in hardware.
Using this knowledge, a logarithmic domain model with linear domain accumulation was
implemented in hardware. In this model the size of weights and biases were determined to
provide the best accuracy at 5 bits in the format b5.0, that is 5 integer bits and 0 fractional
bits. Meanwhile the input was determined to be on the format of b5.3, that is 5 integer bits
and 3 fractional bits. Implementing the conversions of the fractional part proved to be a
challenge as methods had to be developed to do this.
The conversions needed for the model were investigated in terms of table size compres­
sion using the bipartite table method. However for the sizes of the tables in this particular
implementation, this method showed no benefit. Larger conversion table implementations
would however see a significant benefit.
Furthermore the addition of Huffman encoding was investigated. From preliminary testing
it showed that these 5bit weights could be compressed to an average size of 2.6 bits.
However this was not included in the final model.
The implemented model provided a compression ratio of 5.33x compared to a FP32 im­
plementation, as weights and biases could be quantized to 5 bits with an additional 1 bit
for the sign from linear­domain. Due to the nature of the logarithmic domain, the model
also provided a large energy reduction in terms of the arithmetics. This was due to it
consisting of a 5bit and a 16bit adder.
One mistake that was done during the hardware implementation, was to not utilize FSR
as a scale in the calculations. Doing so would reduce the memory required for parameters
by 10KB, that is a 6th of the total size.
The network implemented in Pytorch provided a top 1 accuracy of 76.3%, and a top 3
accuracy of 94.1%. In comparison the FP32 network used as baseline provided a top 1
accuracy of 81.8% and top 3 accuracy of 95.2%.
It was discovered using the hardware model, that the Pytorch model might be less accu­
rate than the hardware model due to an excessive amount of conversions, however this
was not quantified.
11.1
Future work
Future work building on this should include the implementation of Huffman encoding. Fur­
thermore the complete logarithmic network should be investigated further. Initial testing
had a bug which provided bad results, but after this was fixed, it proved quite capable in
testing it in software. Doing so would lower the size and energy impact of the network
even further, as this would do away with various conversions, and thus lowering the size
of the accumulator.
