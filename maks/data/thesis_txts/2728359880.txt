Abstract
Internet┬¡of┬¡Things (IoT) devices have become a prevalent part of everyday life, making
the need for IoT firmware security greater than ever. Firmware dispatch systems play a
crucial role in maintaining the secure operations of IoT devices. Therefore, it is essential
to have robust solutions for such systems.
With the advent of blockchain and Web3 technologies, there are new ways of tackling
security┬¡related challenges. Blockchain provides a tamper┬¡proof, secure, and distributed
protocol, making it an excellent choice for firmware dispatch systems.
This thesis explores the existing blockchain┬¡based solutions for distributing IoT firmware
updates. Furthermore, it proposes a secure firmware dispatch system for IoT devices
using Ethereum. The system employs Web3.Storage, a hosted implementation of the In┬¡
terPlanetary File System, a peer┬¡to┬¡peer hypermedia protocol. The proposed architecture
is a fast, reliable, and scalable system offering a firmware integrity validation protocol. It
can benefit a wide range of manufacturers who intend to increase the security of their IoT
devices.
This work introduces a client application of the aforementioned architecture. It presents
the implementation steps in detail and assesses the related technologies. Finally, an
analysis is provided on the system from a security and performance perspective.

Introduction
In 2019, the Taiwan┬¡based tech giant ASUS was believed to have had its update server
compromised by hackers. Researchers estimated half a million affected devices in the
incident. To worsen the situation, the incident remained undiscovered for over a month.
The infected devices received malicious firmware, which allowed the adversaries to in┬¡
stall a backdoor. [1] The same incident happened in 2021 to Gigaset, a German smart┬¡
phone manufacturer. The attackers were able to install arbitrary software on the devices.
[2] These incidents highlight the importance of firmware security and verifiability when it
comes to Internet┬¡of┬¡Things (IoT) devices.
In recent years, IoT devices have become an essential part of everyday life. While being
connected to the Internet provides many opportunities for users, it also comes with a
greater need for cybersecurity. According to a survey from 2021 made by The Open
Web Application Security Project (OWASP), firmware integrity failures are one of the top
ten vulnerabilities in applications. [3] In 2018, OWASP concluded that insecure update
mechanisms are one of the most common issues with IoT devices. [4]
Keeping the firmware of IoT devices up to date is vital to security, although, as the ASUS
and Gigaset cases showed, this can also be a vulnerability. Therefore, it is essential to find
ways for reliable and secure IoT software updates. To achieve this, the verifiability and
immutability of the updates are crucial. This is where blockchain, an emerging distributed
network technology, comes in, which promises to solve these requirements in an effective
and secure way.
Blockchain technology is becoming increasingly popular, and its use cases are far be┬¡
yond its original application in cryptocurrencies. The technology has seen widespread
adoption, with the most popular applications being financial services, supply┬¡chain man┬¡
agement, intellectual property, governmental and healthcare services. According to Coin┬¡
MarketCap, the current crypto market capitalization is over 1 trillion USD. [5] Blockchain
offers several benefits, including security, transparency, immutability, and decentraliza┬¡
tion, all backed by solid cryptographic fundamentals. On the other hand, the technology is
still new, posing challenges like scalability, operational cost, complexity in development,
and the evolving regulatory landscape.
Decentralized Applications (dApps) are applications built on top of a blockchain network
that offer all the benefits of blockchain while providing some services just like traditional
applications. dApps are prone to censorship and are generally considered more secure
and transparent than conventional applications. However, as previously mentioned, the
technology is still relatively new, and some non┬¡trivial challenges might occur while imple┬¡
menting blockchain┬¡based systems. Thus, this paper focuses on implementing a secure
and verifiable blockchain┬¡based firmware dispatch system and provides an overview of
the relevant technologies and their challenges.
1.1
Motivation and contributions
IoT firmware dispatch systems are designed to distribute and manage firmware updates.
They are essential in maintaining the functionality and security of the ubiquitous IoT de┬¡
vices. Given the increasing number of firmware┬¡related attacks, choosing a secure and
reliable firmware dispatch system is crucial.

Related work
Bettayeb et al. [7] proposed a HyperLedger┬¡based architecture where the manufacturer
stores the firmware update in a centralized manner. The network consists of vendor and

and the gateways communicate with the IoT devices. The paper also involves the im┬¡

systemÔÇÖs capabilities.
Sey et al. [8] demonstrated a solution where the nodes are in a permissioned blockchain
consortium. It employs the Interplanetary File System (IPFS) for decentralized storage
of the firmware updates. It proposes a key┬¡generation service that manages keys with
the possibility of revoking them. These keys are used to encrypt the updates sent by the
server towards the IoT devices, which then can decrypt the update using their private key.
The paper does not cover blockchain┬¡specific details.
Yohan and Lo [9] also proposed a blockchain┬¡based firmware update framework where
the updates are stored in a centralized manner. It provides a way to support both push
and pull update mechanisms. The solution utilizes the Ethereum Mainnet, and it supports
multiple roles: device manufacturers and third┬¡party firmware brokers. The paper also
gives a thorough performance and security analysis of the system.
Projjal [10] outlined an Ethereum┬¡based solution for firmware updating using IPFS. The
paper does not clarify whether the proposed solution uses the Ethereum Mainnet or a pri┬¡
vate network. The solution utilizes IPFS via Infura, which provides an API and a dedicated
gateway to access the IPFS network. Although, the document lacks specific details con┬¡
cerning the integration with Infura and IPFS. An additional layer of security is introduced
through the utilization of MultiSig wallets. The paper describes a local deployment using
Ganache and Truffle for testing purposes. It provides a detailed overview of the hard┬¡
ware implementation aspect of the project, including a crypto co┬¡processor to maintain
the integrity of the system. Finally, it provides a performance test with the implemented
hardware while having a Smart Contract (SC) deployed to the Ropsten Test Network.

Theoretical background
In this chapter, I provide an overview of the theoretical background of the project. Some
knowledge about cybersecurity, blockchain, and web development is assumed from the

3.1
IoT challenges
The increasing use of IoT devices is an undeniable trend. While they provide tremendous
benefits to their users, they also pose particular threats. These threats can be categorized
into two major categories: internal threats arise from the nature of its technology, while
external threats come from how users utilize these devices. [13] I focus on internal threats:
these challenges are mainly security and privacy┬¡related.
Privacy concerns typically stem from centralized solutions. Whoever has control over the
centralized server is also in control of the data collected by the devices. Decentralized
solutions could potentially resolve these issues.
Regarding security┬¡related concerns, a good balance in the following cybersecurity con┬¡
ditions has to be maintained: confidentiality, integrity, and availability.
Confidentiality
means securing the data and incoming and outgoing communications. A typical attack
on confidentiality is the Man┬¡in┬¡the┬¡Middle (MITM) attack, where attackers can read the
data passed towards the server from the client. Integrity means tamper┬¡free data, which
is one of the main goals of this research. Firmware modification is a common type of
attack against integrity. Cui et al. [14] made a report on how destructive and aggressive
these attacks can be. Finally, availability requires the device to be accessible at all times.
Availability concerns most commonly arise from Denial┬¡of┬¡Service (DoS) attacks.
The firmware of an IoT device is the key to fulfilling the aforementioned principles. These
principles also have to be considered when designing the architecture of a firmware dis┬¡
patch system for IoT devices. Therefore, these aspects of my proposed architecture will
be assessed in the evaluation part.
3.2
Current firmware update architectures
This subsection provides a brief overview of the current centralized approach to firmware
updates. The current centralized approach is commonly referred to as Over┬¡the┬¡Air (OTA)
updates, where the update is transmitted typically over the Internet in a client┬¡server ar┬¡
chitecture. IoT devices might directly communicate with the update server═¥ however, a
gateway can be used to handle this part. Using a gateway device can eliminate the bur┬¡
den of downloading the update for IoT devices that sometimes do not have the resources
to perform this operation. Furthermore, gateway nodes can be responsible for controlling
the updates of one or more IoT devices. On the other hand, the gateway introduces a
single point of failure into the system.

3.3
Blockchain technology
In 2008, Satoshi Nakamoto proposed a Peer┬¡to┬¡Peer (P2P) electronic cash system called
Bitcoin, eliminating the need for a central authority. [16] It utilizes digital signatures as
a proof of ownership. The P2P network is responsible for maintaining a public history
of transactions using a Proof┬¡of┬¡Work (PoW) consensus mechanism, meaning voting is
based on oneÔÇÖs computing power. Transactions are grouped into blocks, and blocks are
linked to each other, forming a chain of blocks. Participants in the P2P network are in┬¡
centivized to calculate the next block of transactions correctly. Once a block is added to
the chain, changing its content is computationally very hard. Hence, the concept of mak┬¡
ing a chain of blocks ensures the integrity and immutability of the system. This concept
is called blockchain, and Bitcoin was the first implementation of it. The average block
time is around 10 minutes, meaning the network produces a new block roughly every 10
minutes, creating the ever┬¡growing blockchain representing the latest state.

Ethereum
Ethereum is another open┬¡source blockchain┬¡based technology initially published by Vita┬¡
lik Buterin in 2014 with the intent of creating an alternative protocol for dApps. [17] Since
then, the technology has evolved thanks to its rapidly growing community. The use of de┬¡

==PAGE:15==
centralization, blockchain, and dApps is commonly referred to as the decentralized web
or Web3.
The advent of Ethereum brought a new time to the blockchain community by introducing
a new level of functionality and application. Ethereum is not only a digital currency but
also a Turing┬¡complete programming language that runs on a blockchain. [18] This en┬¡
ables developers to build and deploy dApps on the platform. These applications can be
implemented through SCs, which are executed by the Ethereum Virtual Machine (EVM)
on┬¡chain as part of the Ethereum protocol. SCs act as autonomous agents that always
execute a specific piece of code. Contracts can communicate with each other by means
of messages which are similar to user┬¡initiated transactions.
The platformÔÇÖs main internal cryptocurrency is Ether (ETH), which is used to pay transac┬¡
tion fees. Transaction fees are necessary against DoS attacks and to incentivize transac┬¡
tion validators. EthereumÔÇÖs fee system is based on a computational unit called gas. The
gas fee is proportional to the amount of resources the transaction uses. Each transaction
contains the gas price and the number of computational steps it can make, preventing
infinite loops.
The Ethereum network consists of three types of nodes. [19] A light node only downloads
minimal information, enabling it to verify the transactions. This allows light nodes to effi┬¡
ciently communicate with the network without the need for high network bandwidth and
large storage. Full nodes store the entire blockchain state, and it can verify any transac┬¡
tion, whereas light nodes might need to ask a full node to get more information about a
specific transaction. Archive nodes go one step further, and they store everything, even
entries that have already been verified. This way, the state for each block is preserved,
and one can gain historical insights about the chain.
The technical details of the protocol can be found in the Ethereum Yellow Paper. [20]
It presents the protocol and its underlying technologies. Furthermore, it describes the
system in a formal manner using precise notation and mathematical formulas. The paper
serves as a reference and as a formal specification for the Ethereum community.
Proof┬¡of┬¡Stake and scalability
Ethereum was originally built on a PoW consensus similar to BitcoinÔÇÖs. However, it is
currently in the process of transitioning to a Proof┬¡of┬¡Stake (PoS) approach, which is also
called Ethereum 2.0. PoS is intended to improve the scalability, security, and sustainability
of the network.
The new consensus mechanism is considerably better compared to the now┬¡deprecated
PoW system. [21] Sustainability had a lot of impact on its development, and as a result,
PoS eliminated the need to use lots of energy on computations. This also means reduced
hardware requirements for validators, leading to more participants securing the network.
It involves penalties for validator misbehavior, which makes attacks exponentially more
costly.
Scalability is a commonly mentioned concern regarding Ethereum and blockchain tech┬¡
nologies because every transaction needs to be processed by the nodes. This makes
these technologies inferior to traditional payment processing solutions (like Visa and Mas┬¡
terCard) when it comes to transaction throughput.
The main goal of Ethereum scalability is to increase transaction speed (faster finality)
and throughput while also keeping the decentralized and secure properties of the system.
There are already some notable scaling solutions, like sharding and Layer 2 solutions,
which are expected to be used more frequently in the near future. However, the research

Private and public Ethereum
Public Ethereum refers to the open and decentralized blockchain network that anyone can
join, called the Ethereum Mainnet. While it provides many benefits, there are also some
shortcomings of using this solution, like the aforementioned scalability issues. A private or
permissioned network can be the solution for scalability and privacy┬¡related challenges═¥
however, it can lead to losing the decentralization aspect of the system. Therefore, the
public network is ideal for dApps where open participation is desired. In contrast, private
networks are typically used by businesses where privacy, access control, speed, and
scalability are essential.
A private network consists of multiple Ethereum nodes that can connect to each other
only. An Ethereum node runs an Ethereum client that has multiple implementations, the
most popular being Geth, written in Go. [24] Private networks can be substantially faster
compared to the Mainnet, however, this comes with a price. The operators of the private
network have to invest in high┬¡performance nodes and maintain them. Some analyses
show that this investment is more significant compared to using the Mainnet and paying
gas fees. [25]

Smart contracts
A smart contract (SC) is written in low┬¡level bytecode language, referred to as EVM code.
It has access to a temporary memory only available at runtime and to a long┬¡term stor┬¡
age unit. A SC is always immutable and deterministic. Each contract is identified by its
Ethereum address, which can be used to call the contract. When it comes to the devel┬¡
opment of SCs, they are typically written in a high┬¡level language and then compiled into
EVM code.
In Ethereum, an Application Binary Interface (ABI) is used to define the functions and their
parameters in a SC for the EVM. A SCÔÇÖs ABI is defined in JSON format, and the JSON
contains an array of function descriptors.
The most popular language for Ethereum SCs is Solidity. [18] It is an imperative program┬¡
ming language with a similar syntax to JavaScript and C++. It was explicitly created for
writing SCs within a decentralized environment of Ethereum. The core element of Solidity
is its compiler called solc, which is responsible for converting programs from Solidity to
EVM bytecode. Additionally, solc is used to produce the ABI for the SC.
Given the nature of blockchain, all SCs are public, and any user can call them. If an
attacker finds any exploit in the system, it becomes almost impossible to recover losses.
Therefore, it is essential to keep best practices in mind and adhere to the thoroughly tested
design patterns.
It is important to keep in mind the immutable property of SCs. Immutability means that
once they are deployed, they cannot be changed or updated. There is a strategy for
making contracts updatable called the proxy pattern [26], where the logic is split into two
SCs. The first SC is responsible for delegating calls to the second SC, containing the
implementation part. The address of the implementation contract can be updated, allow┬¡
ing developers to deploy new versions. However, the first SC is still immutable. As a
consequence, it is essential to have it bug┬¡free. On the other hand, the proxy pattern
goes against the immutability principle of blockchain. Hence, it should only be used if it is

3.4
Distributed file storage
A core element of any firmware dispatch system is its file storage. This component is
responsible for storing the firmware in a secure way and for ensuring its availability for
retrieval. Decentralized file storage is becoming a go┬¡to solution for several use cases. It
has excellent protection against data censorship, and its decentralized network is more
resilient against DoS attacks as there is no single point of failure in the system. Addition┬¡
ally, decentralized solutions often utilize cryptography to make tampering with the files
impossible.
3.4.1
IPFS
One of the most popular decentralized file storage solutions is IPFS, an open┬¡source P2P
network for content addressing. Similarly to Ethereum, it is a set of protocols that has
many implementations. [27] These protocols solve challenges like verifiability, centraliza┬¡
tion, and resilience.
Content addressing
On IPFS, data is addressed by its content rather than a location address in centralized so┬¡
lutions. This address is called the Content Identifier (CID), which contains a cryptographic
hash of the addressed file and a codec that holds information about the interpretation of
the data. The newest version is called CID v1, which contains the content hash, the
encoding, the version, and the codec.

==PAGE:18==
For example, there is a file called satoshi.txt with the following content:
Satoshi
Then, the CID v1 identifier for this file is the following:
bafybeihl3eu3so6rfyxz5qixusksqstjshjtqgecz35czhb62uua5wtfuu
This identifier contains the following information: the content hash, the encoding (base32),
the version (cidv1), and the codec (dag-pb).
When adding a file to IPFS, the file is split into chunks. Then, the chunks are structured
into a Merkle Directed acyclic graph (DAG), where the root is the CID for the file. IPFS
uses Content┬¡Addressed Archive (CAR) files, which are serialized Merkle DAGs.
IPFS challenges
IPFS faces some challenges when keeping a file available on the network. Other nodes
in the network will only keep a specific file if it is in high demand. In order to keep files
on the network, one has to run enough nodes to pin the files on IPFS. This comes with
overhead on cost and operations, making it hard to scale up or down on demand. It is also
required to run a node to retrieve files. There are several solutions that help overcome
these challenges.
Filecoin
One of the solutions for making the use of IPFS easier is Filecoin, a blockchain protocol.
It was made by Protocol Labs, the same company that maintains IPFS. The solution
facilitates data storing and retrievals and financially incentivizes Filecoin nodes (storage
providers) to keep data pinned to IPFS. When a user wants to store a file, a storage
deal is created between the user and a storage provider. Once the file is stored, the
client receives verifiable proof of this fact (called Proof of Replication), and the deal gets
added to the Filecoin blockchain. [28] Once the data is stored, the user can get a proof
of storage similar to the Proof of Replication. This is called Proof of Spacetime, which
ensures the file is stored over time. Both of these proofs employ strong cryptography with
zero┬¡knowledge proofs to ensure storage deals are fulfilled. [29]
Web3.Storage
Protocol Labs launched Web3.Storage for a simplified way of using decentralized file
storage. It provides an API for developers to store and retrieve files. Under the hood, it
utilizes Filecoin and IPFS, which means it has all the benefits these technologies provide.
It has a free tier, which is an excellent way to get started and test an application. It only
requires a Web3.Storage account and an API token to start storing and retrieving data
through Filecoin.

with the public IPFS network. The other is w3link, an HTTP Gateway for IPFS developed
by Web3.Storage, which can cache files on the edge while having all the advantages of

Proposed scheme
In this chapter, I present the proposed scheme by first analyzing the requirements. Then,

Before I delve into the details of this chapter, I define the three primary actors of the

4.1
Requirements
I provide an overview of the requirements of the system to show the motivation behind
some of the design decisions I made in the system model. As mentioned in the previ┬¡
ous chapter, the system generally has to meet the following three security conditions:
confidentiality, integrity, and availability (commonly referred to as CIA conditions).
4.1.1
Confidentiality
Confidentiality is about securing the data from unauthorized actors. When it comes to
firmware updates, a vendor might want to protect their update from the public. How┬¡
ever, in some other cases, this is not important as long as the updateÔÇÖs integrity is pre┬¡
served. Since blockchain is a public ledger, everything is accessible to the public by
default. Therefore, it is crucial to address the question of confidentiality for the specific
use case. The framework presented in this thesis is a general┬¡purpose firmware dispatch
system═¥ hence, it must provide a way to ensure the confidentiality of the firmware update.
4.1.2
Integrity
The most crucial aspect of any firmware dispatch system is the integrity of the updates
distributed with it. It is vital to provide a way for integrity checks when firmware is being
distributed. To fulfill this requirement, there is a need for cryptographic proof that the file
has not been tampered with and that the distributed update is identical to what was initially
uploaded. Similarly, the IoT devices need a way to verify the integrity of the downloaded
files.
4.1.3
Availability
While securing the confidentiality and integrity of the data are essential, it is of no use if
the data is unavailable. Therefore, the proposed system model must be highly available.
It must also be resilient against attacks targeting the systemÔÇÖs availability, typically called
DoS attacks, where intruders try to disrupt the service by overwhelming the network with
traffic. DoS attacks might be distributed if the attack is not performed from a single source
but rather from a network of computers.
4.1.4
Scalability and performance
There are some other requirements for a general case firmware dispatch system. One
of these is the scalability of the system. To make a future┬¡proof system, one needs to be
able to scale up (or down) both the storage and the blockchain nodes of it. Furthermore,
it must provide a seamless experience regarding the speed of uploads and downloads.

Some blockchain and storage solutions might take some time to finalize transactions and
distribute the files. Therefore, it is essential to assess this property of the system and to
guarantee the speed and the feasibility of the proposed solution.
Another requirement stemming from the generality of the system is that the performance
metrics of the IoT devices must be taken into consideration. One cannot assume that an
IoT device can act as an active node in the network as it might not have the networking
and computing capabilities to participate in it. This might be caused by a limitation in
bandwidth, computing power, or some other characteristics of the device.

there is a need for an integrity verification protocol.
The integrity of the files is based on CIDs, which were introduced in the previous chapter.

some additional information as a prefix. This process results in a unique CID for each file,

use CIDs for addressing the files, the integrity is preserved by definition at the storage
level. When the server transmits the file to the client, it can locally calculate the CID for
the downloaded file and compare it with the received CID from the SC. Hence, the client

5.4.2
Validating the proposed approach
I validate the proposed approach by first showing that there is no way one can change
an update after publishing a firmware update. Then, I show that a client can always get
proof of integrity when retrieving an update.
As presented in Figure 4.2, when a user initiates a new firmware update publication,
the client locally builds the CAR and saves the fileÔÇÖs CID. Then, it uploads the CAR to
Web3.Storage and retrieves its CID. At this point, the client application has two CIDs:
one calculated locally and one received from the server. The client compares these two
and only continues the execution if they are identical. Then, it finalizes the update by
adding its details to the blockchain, providing an immutable record. The CID of a file is
always based on its contents, and any change in the contents of it will result in a different
CID. Hence, if someone tries to change the firmware later on, that will mean the CID of
the file also changes on IPFS, and it will never match with the value from the blockchain
network.
Subsequently, if a client wants to ensure the integrity of a retrieved firmware update, it can
get proof of it in some simple steps. First, it queries the blockchain for information about
one update containing the CID and the file URL. Then, it retrieves the update through
Secure IoT Firmware Dispatch System with Blockchain
27
==PAGE:36==
Web3.Storage. Finally, it calculates the CID of the retrieved file locally, and if it does not
match what it received from the blockchain, it discards the file.

Discussion

6.1
Key benefits
In conclusion, the architecture provides the following benefits:
ÔÇó Firmware integrity: The uploaded file cannot be altered once published, and the
protocol allows for proof of integrity upon retrieval.
ÔÇó Decentralized architecture: The architecture provides high availability, and the core
of the system does not have a single point of failure.
ÔÇó Fast and reliable file distribution: The files are stored with Web3.Storage, providing
a simple and robust way of interacting with IPFS.
ÔÇó No operational tasks for storing files: The files are stored and kept pinned on IPFS
via Filecoin, requiring no infrastructural operations on the clientÔÇÖs side.
ÔÇó Secure by design: The system lays on solid cryptographic fundamentals, providing
fast access to the updates without compromising security.
The proposed architecture provides a fast and reliable way to manage updates. It uti┬¡
lizes blockchain technology to store records of the updates in an immutable way while
also being highly available. The files are stored in a decentralized storage, achieving
similar advantages to blockchain. Files are uploaded through Web3.Storage, a scalable
cloud┬¡native implementation of IPFS. Both blockchain and IPFS are resilient against DoS
attacks.
The system is recommended to any IoT manufacturer looking for a simple, resilient, and
secure way of distributing firmware updates without significant operational overhead. De┬¡
pending on the specific use case, companies can either choose to use the Ethereum
Mainnet or maintain a private blockchain network. Both solutions are supported by the
proposed architecture.

The Web3 community is evolving, and one can find many resources available. However,

technologies to choose from, however, these are typically not battle┬¡tested. When choos┬¡
ing new technologies for a project, it requires effort and research to assess each solution
to ensure it aligns well with the rest of the project. New technologies often lack proper
documentation, and it can generally take more time to find answers to specific questions.
In some cases, I had to find the answers on my own by searching in the source code of
the package.
In one library, I was getting an unexpected error. To resolve it, I had to debug the package

files that could potentially be shared publicly forever.
6.3
Security and privacy analysis
In this section, I assess the system from a security and privacy point of view. I cover the
following aspects: user authentication, upload encryption, integrity verification, blockchain
deployment type, and network attacks.
ÔÇó User authentication: Keys are pre┬¡distributed for the clients, and access to the client
is managed by the vendor. The security can be enhanced by utilizing MultiSig wal┬¡
lets and key revocation. MultiSig wallets are enhancing security by requiring more
than one signature on each transaction. Consequently, attackers need to obtain all
signer keys to succeed, making the attack a lot more complex. Key revocation can
be helpful when a key is compromised. The two solutions combined can fulfill the
security requirements for high┬¡risk applications.
ÔÇó Upload encryption: Content is addressed with CIDs in the system═¥ therefore, one
has to know the CID of a file to retrieve it. Retrieval becomes hard without know┬¡
ing the CID. This level of obscurity is enough for many use cases. On the other
hand, a company might want to further enhance the privacy of the files. Privacy en┬¡
hancement can be done by encrypting the files before uploading them, which could
be a simple step before creating the CAR on the server. For example, CryptoJS,
a JavaScript library, could be utilized to encrypt and decrypt the files with the AES
standard and a pre┬¡distributed secret without adding much overhead to the system.
Encryption would increase the privacy aspect of the system, making unauthorized
access to the contents of the files practically impossible.
ÔÇó Verifying a file: A protocol for ensuring file integrity was validated previously in Sec┬¡
tion 5.4.2. The process provides a secure and straightforward approach to get cryp┬¡
tographic proof of data authenticity. Hence, participants of the system have a reli┬¡
able and performant way to verify files.
ÔÇó Blockchain deployment type: It is a security, privacy, and performance consider┬¡
ation whether the blockchain network is a private or public deployment. Private
networks provide great scalability options, and they offer strong privacy. Whereas
Secure IoT Firmware Dispatch System with Blockchain
32
==PAGE:41==
public networks are harder to scale, and it is harder to maintain privacy since they
are accessible by anyone. However, they can provide more security as they are
larger networks.
ÔÇó Network attacks: The proposed system assumes the use of secure communication
channels with strong encryption like TLS. TLS ensures data integrity and confiden┬¡
tiality and protects against MITM attacks like spoofing. However, in the case where
a client suffers from DNS pollution when retrieving a file from IPFS, and as a result,
it gets a modified file, the file integrity protocol would still be able to detect the modi┬¡
fication. The hash value of the received file would not match with the one on┬¡chain.
Hence, the client would discard the file.
