Abstract
This project presents the design and implementation of a high-throughput packet
parser for a wide variety of network-related activities. The proposed parser should
eﬀiciently identify the incoming packets irrespective of the header encapsulations in-
volved, making it compatible with headers belonging to Layer 2 (L2), Layer 3 (L3),
and Layer 4 (L4).
A user-configurable field extractor should be followed by the parser, which should
extract the fields required for the user. The extracted fields should be given to the
user as a block of data that can be used for further network operations. The parser
and the field extractor should work at 100 Gigabits per second.
The current project discusses suitable architectures for a packet parser and field
extractor. The RTL design was done using VHDL on the Xilinx Vivado design suite.
Post synthesis and implementation, the design is verified on an FPGA.

CHAPTER 1
Introduction
In modern communication systems, a large number of packets are sent across networks.
Each packet consists of a large number of headers that have valuable information for
different purposes. These packets in the network have to be analyzed at high speed
to understand the header. After understanding the headers present, the necessary
fields from the headers are extracted.
The key fields extracted can be used for any network operations. The following
are the areas where it is necessary:
• A router extracts an IP destination address to forward the incoming packets.
[16]
• For Firewall, to monitor incoming and outgoing traﬀic, identify the source and
destination IP addresses, and recognize ports. [18]
• For a network-based intrusion prevention system to drop packets from certain
addresses. [20]
1.1
What is Parsing
This process of identifying all headers and extracting key fields present in a packet is
called packet parsing. It is impossible to identify all the headers present in a single
step, as there is no fixed sequence of headers. However, each header has a field to
identify the next header. Based on such fields, all the header encapsulations can be
identified to determine the type of packet. Since there is a wide variety of possible
headers, the identification of headers is a sequential process. [16]
1.1.1
Overview
A packet parser is designed, synthesized, and implemented on an FPGA board using
the Xilinx Vivado design suite. The designed parser works at 100 Gigabits per sec-
ond (Gbps). The parser and extractor were kept as separate blocks to make the field
extractor user-configurable, as the field extraction can be done after understanding

Thus, the parser should run at least at 195 MHz, and it is been rounded off and
decided to design the parser to handle the data at 200 MHz.
With the help of the field extractor, the user should be able to extract a chunk of
data. The current design for the extractor helps the user to access up to 8 fields, and
the maximum size of the block that has extracted fields should be 32 bytes.
The maximum size of the extraction block is determined based on the size of the
longest field, which is the address field of IPv6. Each IPv6 address is 16 bytes long,
and by giving the possibility to extract 32 bytes, the user can extract both the source
and the destination addresses of IPv6. Thus, this is how the extracted fields’ size is
determined. [8]

CHAPTER 2
Types of Parsers
Before understanding the type of parser, it is better to have insight into parse graphs.
Parse graphs are acyclic graphs with vertices representing the headers and the
edges specifying the order of headers. [16] In other words, the parse graphs depict
possible header combinations and ordering of headers for a particular packet. [31]
For instance, if we consider the following parse graph 2.1
Figure 2.1: Example of parse graph [32]
Each Node (Ethernet, VLAN, IPv6 etc) denotes the possible headers, whereas the
edges denote the transition or encapsulation of headers.
From the above example, one such combination of headers could be Ethernet ->
VLAN-> IPv4 -> TCP.
Based on the structure of parse graphs, a parser is classified as follows [16]:
• Fixed Parsers
• Programmable Parsers
2.1
Fixed Parsers
Fixed parsers are suitable for a single parse graph, and the structure of the parse
graph is determined during the design phase. In other words, the next possible head-
ers can be pre-determined from the graph and cannot be modified dynamically. A
==PAGE:15==
2.2 Programmable Parsers
5
single parse graph can have a union of smaller parse graphs. [16]
To make a fixed parser suitable for other parse graphs, the design of the parser
has to be modified. Since these parsers are for a particular parse tree, they are faster
as there is a smaller search space for the next header. [16]
2.2
Programmable Parsers
Programmable parsers are suitable where the structure of the graph is not fixed.
These parsers are suitable for a wide variety of parse graphs, and the parse graph
structure is defined during runtime. This makes the parser flexible for different parse
graphs. [16]
However, the problem arises when the parser cannot immediately identify the next
header. When a header does not have any relevant information to predict the next
header, then the parser tries to extract one field after another to find the next header,
which adds overhead to the design. [16]
To overcome this overhead, many techniques are used to reduce the search space for
the next protocol. One such protocol is to identify the next header based on the state
table. [16]
In simple words, the parser notes the latest identified protocol as state information.
Based on this information, it tries to extract the next header identifiers in the pre-
defined order and compares them with the known identifiers (stored in RAM ) [16].
These are the types of parsers available. However, in this project, the parse graph
is known. Thus, it is decided to proceed with a Fixed parser. Following is the parse

CHAPTER 3
Parser Methodology
Once we know the parse graph, the next move is to design a parser. Before designing
the parser, one should have a clear insight into the parser’s working principle.
3.1
Working principle of parser
The parser performs three different operations, and they occur sequentially for each
header [16]. They are:
• Header Identification
• Field Extraction
• Field Buffering
When the parser receives the headers present in a packet, it is sent to the Header
Identification block, as well as to the Field Extraction block.
Header Identification Process: The header identifier uses either the upcoming
header information extracted from the previous header or extracts continuously (if
the next header is unknown). To extract this, either a pointer or an offset is used.
Once the header identifier is extracted, the extracted value is compared with known
header values, which are either stored in RAM or logically represented by the Mul-
tiplexer. Only when a desired identifier is recognized, the Field Extraction begin. [16]
Field Extraction: Field extraction is a comparatively simpler task. The extrac-
tor receives the header types(type of header identified) and location(usually an offset)
from the header identification block. The field extractor already has the information
about the fields to be extracted for each header and their relative offsets. Based on
the information from the header identification block, the field extractor processes and
extracts the necessary fields. However, to increase throughput, the extraction process
is controlled by a state machine [16].

then sent to the output continuously, one part at a time. The Generic model of the

This parser comprises various components [4]:
• Lookup component
• Locate component
• Extract component
• Compute component
• key builder component
Look up component: Look up component receives a header info (mostly header
identifier)from the previous instance or previous cycle. Based on the header informa-
tion received, the lookup component sends a microcode to all other components. This
is stored in flip-flops or RAM, based on the size of the microcode. [4]
Locate component: The Locate component receives the offset from the previ-
ous instance or cycle, which denotes the starting point of the header. Based on the
incoming offset, and microcode received from the lookup component, this component
extracts only the relevant part of the header. This extracted part of the header is
sent to the extract component. [4]
Extract component: The extract component has the information about the
fields and their length that have to be extracted. This information for each field is
associated with a microcode. The extractor component sends the extracted fields to
the compute component by continuously shifting and extracting based on microcode

Compute component: The compute component receives the extracted data

Key builder: The key builder component receives microcode as well as the ex-

this, the user can customize it to have other optional operations. The key is passed
on between instances or cycles. [4]
3.3
Required features of Parser:
Based on this parser, a parser will be modified and designed, and should have the
following requirements:
• The parser should not have an extraction part while identifying the header
included, as it conflicts with the idea of user-configurable field extraction. Hence,
the field extraction should be a separate component.
• Since the field extraction is a separate entity, the parser should uniquely iden-
tify each path from the parse graph. Thus, the field extractor should receive
path numbers.
Thus, it is mandatory to have path numbers and modify
them while identifying different headers(traversing through different nodes of
the parse graph).
• To find different paths in a parse graph, moving from one header identifier to
another is ideal.
• While parsing through the parse graph, some of the headers will be used more
than once. Thus, an extra input called marker to identify those headers in all
different parsing stages.
• marker is also used to denote whether the parser has identified the path number
or not (parsing done or not status).

The required features (necessary signals) of the parser have been decided. The parser
should follow a certain protocol to denote valid packet transmission signals, like a
valid signal is needed. It is also observed that the packet size varies from 48 bytes to
178 bytes (single-cycle to three-cycle packet), obtained from pre-estimation of head-
ers. Hence, it is decided to have signals like the start of the packet and the end of
the packet, too.
Thus, it is decided to have the Avalon Streaming Interface (Avalon-ST packet
interface signals [19]) protocol for transmission.
There are mandatory signals from the protocol [19]. They are:
• valid : A valid signal is asserted whenever the data is transmitted, indicating
that the data is not corrupt(valid).
• startofpacket : The startofpacket signal is used to denote the starting cycle
of packet transfer. The valid signal will be asserted too when the startofpacket
is asserted.
• endofpacket : The endofpacket signal is used to denote the last cycle of
packet transfer. The valid signal is asserted throughout the packet transmis-
sion. Moreover, for single-cycle packets, both startofpacket and endofpacket
will be asserted as part of the valid signal.
Apart from this, an empty signal is used. This signal is used to denote the number
of empty bytes during the last input cycle. Thus, the empty signal value can be seen
only during the last cycle. The following image 4.1 shows the Avalon ST protocol
[19]:

Input block
After breaking the inputs into the required protocol format, the data can be received
by the parser. However, the processing rate of each packet varies. This is mainly due
to the packets spread across multiple input cycles. Thus, an Input block is used to
cluster the packets.
The clustering of data is based on control signals SOP and last(EOP). This works
as follows:
• The incoming packet is stored in a buffer when SOP is asserted. This process of
storing multiple cycles of inputs occurs as long as the last signal is not asserted.
• When the last signal is asserted, the data is sent to the parser.
• Additionally, the SOP and the last signals are checked simultaneously to handle
the single-cycle packets.
• since the data sent to the parser is of fixed size and the packet size varies, the
unused data is filled with zeros.
This is how the clustering of data works. After this stage, control signals, except
for the valid signal, are no longer needed for downstream blocks as the packet is
compressed to a single cycle. Thus, eliminating the necessity of SOP and the last
signals. The data bus is dependent on the maximum number of cycles the input is
spread across. In our case, it is three cycles, each carrying 64 bytes. Thus, the input
bus width for the parser is (64 bytes X 3 )192 Bytes.

4.3
Load-Balancing Architecture
Parser architectures are introduced to increase the throughput of the parser. One
way to improve the throughput is to use a load-balancing architecture.
Load balancing is a process in which the network traﬀic is distributed across
available resources [1]. In our case, to achieve packet-level parallelism, we use a load-
balancing architecture [25].
Thus, multiple instances of parsers are needed. Moreover, logical blocks to dis-
tribute packets to different instances, as well as to combine packets received from
instances in the exact order, eﬀiciently, are necessary. This can be done by the sched-
uler (Distributor) and the rearranger block before and after the parser block [25].
After clustering the packets, there would be empty bytes available in the packet.
These empty bytes can be used by the scheduler to mark the order of the packet
arrival (like a timestamp). However, this timestamp is needed only for certain dis-
tribution algorithms. For instance, if a round-robin scheduler is used, then it is not
necessary.
Furthermore, FIFOs are introduced after each parser to hold the packets to hold
the processed packets since the processing speed varies from packet to packet. These
are additional blocks necessary apart from the parser.
As discussed earlier, the parser needs offset, marker, and the valid signal apart
from the packet data. Thus, the scheduler distributes these, whereas the rearranger
receives the valid signal, packet data, and the path number, which are stored by FI-
FOs before it.

4.4
Pipe-lined Architecture
Another way to improve the throughput is to use a pipe-lined architecture. Pipelining
is a technique where multiple processes are executed simultaneously. To pipeline any
process, it has to be broken into multiple parts. In other words, a single process is
made into multiple parts, and each part of the process is executed in parallel, enabling
higher throughput [2].
In our case, pipelining can be achieved by dividing the parsing of each header
from the packet into different stages. Each of these stages, where a part of the header
is processed, is called a sub-parser. Each sub-parser operates on different headers,
thereby eliminating the need to wait for the parser to completely parse a packet.
Each sub-parser is connected end-to-end such that the output of one sub-parser is
connected to the input of the next sub-parser. Since all the sub-parser performs the
same operation, the Inputs and outputs of all sub-parsers are the same. Each stage
of the sub-parser should receive the following, which are processed by the previous
stages. The signals needed are based on the requirements after going through the
microcode architecture in the previous chapter 3.3:

Comparing architectures:
Both architectures are used to improve the throughput. There are pros and cons for
both architectures. They are:
4.5.1
Load balancing Architecture:
Pros:
• In a fixed parser with variable packet length, the throughput can be higher if
the previous packets are smaller in size.
• Unlike pipelined architecture, if the parser has to handle new protocols, only
the conditions in the offset provider have to be updated.
Cons:
• There is an additional design overhead due to the scheduler block, the rearranger
blocks, and the FIFOs between the parser and rearranger block.
• Furthermore, eﬀicient scheduling algorithms are needed, thereby increasing de-
sign complexity.
• The resource usage would be higher due to overhead.
• latency of the parser will be increased.

4.5.2
Pipe-lined Architecture:
Pros:
• Compared to the load-balancing architecture, there is no additional overhead
as the packets arrive in a particular order.
• By achieving a uniform processing rate of sub-parsers, it is easier to eliminate
the necessity of FIFOs between sub-parser stages.
• No complex algorithms for scheduling are needed.
Cons:
• If the parse graph has to be updated, then apart from the next offset provider
logic, new sub-parsers might have to be added.
• Even though a packet had been processed, the packet should go through all the
sub-parsers, which in turn increases latency.
Thus, it has been decided to proceed with the pipelined parser architecture due
to reduced overhead and complexity, achieving almost the same throughput as that
of the load-balancing architecture.

Architecture
The field extractor block follows the pipelined packet parser. It receives the following
signals from the parser:
• valid
• packet data
• path number
Based on the path number and the valid signal, the field extractor determines and
extracts the fields. However, to extract required fields from the packet, the extractor
needs offset, and the length of the field to be extracted.
Hardcoding this data would lead to the static multiplexer. If such a design is im-
plemented, it is called a fixed field extractor. Moreover, such a design would increase
the logical depth of the design. This opposes our goal to have a user-configurable
field extractor.
The field extraction can be made configurable by modifying the offset and the
length information that will be used by the extractor. Thus, a well-known reconfig-
urable component, RAM, is used to store the information.
Using RAM, the logical depth can be reduced. The content of the RAM should
be accessible based on the path number. Thus, the path number is treated as the
address for content in RAM. Additionally, the user should be able to write to the
RAM to make it user-configurable. Thus, the RAM should be dual-port RAM. In a
dual-port RAM, one port is used to write and the other to read.
However, before the implementation of RAM, the format of the data present in
the RAM has to be determined. Since the extractor will be designed to extract up to
8 fields and each field has an offset and length. The offset and the length information
present or written to RAM for extraction should be 16 bytes. This is determined

then they are converted into a coeﬀicient file.

To optimize further the logic depth, has to be reduced. The current logic depth is

The critical path in the sub-parser can be reduced by introducing a register at the
start of the offset provider block to break the critical path.
It is observed that all the logic in the next offset provider block will not be used
at all stages. Thus, it is decided to have different sub-parsers rather than duplicating
the same instance multiple times. With this, the logic depth of the static multiplexer

provider block can be reduced.
Moreover, it was decided earlier not to use the empty signal to decide whether to
extract the fields or not. This decision was made during the design stage to maintain
the uniform processing rate of the sub-parser despite parsing more than needed. This
decision played a major role in maintaining the logical depth of 5.

extractor instance optimization

10.1
Discussion
The current parser design is for headers of fixed length, which limits the usability of
the parser.
Thus, the parser has to be made more flexible by making it more suitable for
headers of variable size, i.e., the parser design should be suitable for headers with
multiple options, and the following are the options that the parser should be modified
for:
• IPv4 with options
• NVGRE with optional fields
• GENEVE with optional header
• Optional fields of L2TP
• PPPoE with additional header
• TCP with options
• GTP with optional fields
With the optional fields, the length of each header varies. Thus, it is ideal to
identify the length of each header to parse the headers properly. The updated length
of IPv4 can be known with the help of the IHL field.
For NVGRE, it can be
identified with the help of flags such as Key and Sequence. The variance in size for
GENEVE, PPPoE, and L2TP can be found out using the Length field. Similar
to NVGRE, the flags in GTP are verified. The TCP does not have a field to check
its size. However, the size of the TCP is not necessary to check, as it is the last field.
==PAGE:69==
10.2 Future Enhancement
59
10.2
Future Enhancement
The parser block should have 5 more sub-parsers, increasing it to 16 from 11. These
include two for IPv4s, one for NVGRE, GENEVE, and L2TP, and one each for PP-
PoE and GTP.
The width of the bus that sends packets to the parser block and further down-
stream should be increased based on the size of the longest packet. However, there
won’t be any necessary changes required for the marker signal, and it can be main-
tained at the current size of 3 bits.
Furthermore, the future enhancement should be based on converting this design
to be suitable for multiple parse graphs by making it programmable. The fixed parser
design plays a major role in the development of programmable parsers, especially in
debugging the next header identification logic. Since the size of the header and what
the next header is are not explicitly known, the number of sub-parsers needed might
increase.
By a programmable parser, the parser block design should be able to identify
all possible parse graphs, apart from all possible paths for each parse graph. This
information has to be relayed to the field extractor block. The fields to be extracted
should be based on the parse graph number as well as the path number (in that
particular parse graph) fed to the dual-port RAM.
To achieve a higher operational speed, the logical depth can be further reduced
by using TCAM [16] rather than static MUXs in the sub-parser.

Possible header traversals:
In order to identify all possible headers, a graph is first made using the function

Once the graph is formed, all possible paths between the two nodes, i.e., ”port and

source, target)” from the same library (networkx) where
• graph=name of the DiGraph created
• source= starting node of the graph (port)
• target=end node of the graph (payload)
The above function finds all paths without repetition between the source and the
target, and all possible paths are stored in a list (all_path).
Using the len() function in the list, it is found that there are 313 possible header
combinations. Once all possible paths are found the next step is to create packet data

B.2
Packet creation
In packet creation, the first step is to randomize different headers and then different
combinations of these headers are to made.
B.2.1
Creating headers:
In this step, Different headers were created and randomized. The randomized data
function, like Mac addresses, which can be utilized by headers, is defined as a
common function. Moreover, some of the value like next-header etc are automatically
assigned by Scapy. Thus, it is not necessary to mention it. Following are the function
that are defined for headers:
B.2.1.1
Random addresses
Some of the addresses like the MAC address for Ethernet headers, IPv4, and IPv6
extensions are mentioned as separate functions that can be used commonly.
The
MAC is completely random where whereas for IPv4 and IPv6 it is ”192.168.x.x” and
”2001:db8::X” respectively.
B.2.1.2
Ethernet layers
In both the Ethernet layer, only the random source and destination addresses are
mentioned, whereas the Type is auto-assigned based on the next header.
B.2.1.3
IPv4 layers
In IPv4, Only source and destination addresses, internet header length(IHL), and time
of service (TOS). Since, IHL is randomized with just TOS, the header auto assigns
zeros instead of options based on the length mentioned in IHL. The auto-assigned
values are id=1, version=4, and ttl(time to live) =64 where whereas proto(protocol
field) is assigned based on next header.
B.2.1.4
UDP layers
In the UDP header, both source and the destination addresses are randomized. The
sport values are 53 (DNS),67(DHCP),123(NTP),161(SNMP)or it can be any
random value between 1024 to 65535 . For the destination port address (sport), it is
a random value between 1024,65535. However, the data had to be modified for the
next protocol of parse graph.

B.2.1.5
NVGRE
In NVGRE, the presence of key (key_present) and (key) is randomized, and the
protocol value is mentioned as 0x655,8 which mentions the presence of NVGRE are
mentioned.
B.2.1.6
VXLAN
in VXLAN, only vni is randomized where as the Next protocol value is auto as-
signed.
B.2.1.7
GENEVE
Regarding Geneve, the version (0 to 7), oam (0 or 1) and vni(maximum of
0xFFFFFF) are randomized. The proto, which is used to identify next header, is
also auto assigned.
B.2.1.8
L2TP
In L2TP, both tunnel ID and session ID are randomized, which range from 1 to
0xFFFF. The version is auto assigned. Moreover, the flags were tried to randomize
to show the presence/ absence of sequence number and the presence/absence of end
of sequence.
B.2.1.9
PPP
The value of PPP is initialized, and Scapy auto-assigns values. hence, it only shows
the next protocol value (proto)
B.2.1.10
VLAN layers
in VLAN, only VLAN id is randomized where as priorityprio and Drop Eligible
Indicatordei are also randomized. The type is also auto assigned based on the next
header.
B.2.1.11
PPPoE
Similar to PPP, PPPoE is just initialized, and values are auto-assigned
B.2.1.12
IPv6
In IPv6, only the source and the destination addresses are randomized. Moreover,
the flow label is also randomized. The version and hop limit hlim are fixed to 6 and
64, respectively. The next header (nh) is auto-assigned based on the upcoming
header.

ESP

B.2.1.14
TCP
In the TCP header, multiple fields are randomized.
The source port (sport) is
randomized to 80 (HTTP), 443(HTTPS), 22 (SSH), 25(SMTP)
apart from
random values between 1024 to 65535. The destination port has values from 1 to

Moreover Sequence number and ack were also randomized
B.2.1.15
SCTP
In SCTP, both source and destination ports were randomized apart from tag and
checksum
B.2.1.16
GTP
In the GTP header, only the Tunnel endpoint id (teid) is randomized whereas as
the version, protocol type (pt), and gtp_type were auto-assigned.
B.2.1.17
Port and payload
Regarding the port number and the payload, they were not randomized as of now.
B.2.2
combining headers
Here, headers are combined to form a packet, The packets are created for all possible

• All possible traversals in the parse tree are made, and they are stored in an
array of lists
• Then, each element present in each list was called, and the respective headers
were appended.
Once different packets were created, they were stored in a text file called ”path_header.txt”
and they contain the following values:
• path number
• traversed path or a particular row in a list
• header summary
