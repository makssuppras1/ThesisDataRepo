Abstract
As online shopping has increased in popularity, companies have increased focus on delivering good customer
experiences, equal to their performance in their physical shops, on their websites. Recommendation systems
can help deliver such customer experience online. In this thesis, multiple recommendation systems will be
tested on a speciﬁc data set supplied by H&M, under the assumption a DeepFM architecture has the
best performance.
However, it is presented that the best method for creating recommendations is the
Factorization Machines model.
The Factorization Machines model has been deployed in a cloud infrastructure making it accessible 24 hours
a day, from where it can create recommendations. In the cloud setup, adjustments can be made to increase
the number of virtual machines and thus increasing the system’s capacity. This way the capacity can match
the demands from incoming requests for recommendations. The deployment has the purpose of emulating
the recommendation system on a website. Products can be shown to the customer accessing the website,
based on the recommendations made by the model. With an international company like H&M endless
amounts of data is available. It became a constant trade-oﬀbetween model performance and demands of
capacity from the data pipeline created. Multiple decisions had to be made such as excluding data and sub
optimal data preprocessing, which were all necessary to train and deploy the model.
The deployment setup created in this thesis did not perform acceptably given today’s expectations for
the runtime of a real time recommendation system. Instead, it was built to showcase the robustness and
possibilities of deploying a model in a cloud infrastructure.

Introduction
In a time where private companies store personal data from online customers interacting on the websites
and use it to create more sales, the customers may better accept their data is stored, if it means that they
receive a good user experience. If the customer is shown meaningless adverts or recommended a product
that they already bought, the customer gets a bad experience and would probably not use the website
again.
Therefore is it critical to be able to recommend the right products and give customers a good
experience. H&M is a huge international retail company that rely on both physical sales and online sales.
For online sales, they can not give the same personale help to customers as in the physical shops, which is
why meaningful recommendations are critical to them. There are multiple ways for a company like H&M to
achieve meaningful recommendations. They could ask customers what they are looking for when arriving
on a website and tailor the experience to that answer. Another method is to develop a model, which can
make real time personal recommendations for the visiting customers providing them with the next best buy
option. The latter is the focus in this thesis. The model is developed by training on a company’s historical
transaction data and then deployed to create recommendations on the company’s website. In this thesis
data from H&M will be used.
Unfortunately, a large amount of implemented machine learning models tend to be unused because they
are missing a proper pipeline setup. According to VentureBeat 87% of developed machine learning models
never get deployed [2]. Implemented models can perform well when they are being tested locally and then
fade away quickly when deployed to their actual purpose. Proper monitoring of the model can make sure
the performance of the model do not degrade over time. Together a robust pipeline and monitoring can
even make small models perform well when deployed. The motivation for this thesis is to show how to
properly deploy and monitor an implemented state of the art machine learning model.
3.1
Scope
The goal for the thesis is to implement the state of the art DeepFM architecture as a recommendation
system to recommend articles of high quality as the next best buy option for customers, providing a good
customer experience. Historical transactional data from a kaggle challenge hosted by H&M will be used
to train and test the performance. To monitor the implemented model it will be wrapped in a container
and deployed as an app on Google Cloud Platform (GCP). The deployment in the app has the purpose of
simulating the model deployed on a website, from where it could make real time recommendations.
3.2
Method
The data used to build the recommendation system is provided by H&M through a kaggle challenge which
they hosted [3]. The model will be trained on historical purchases all of which H&M has stored. To enhance
the quality of the recommendations, the transaction data will be enriched with metadata from two other
data sets called customers and articles which contain metadata about the customers who made purchases
and the articles in H&M’s catalog.
Our solution is to create a model which solves the binary classiﬁcation problem of how likely the customer
is to buy a given product. The recommended products will be the products that the model predicts to be
the most likely. Therefore the problem becomes a binary classiﬁcation problem for each product in H&M’s

In this thesis, the testing of multiple models will be presented with the assumption that the best is the
adaption of the model developed by H. Guo et. al. [4] called DeepFM. These adaptations and the structure
of the network will also be described in Section 5.4 as will the rest of the models.
As the DeepFM model contains all hyperparameters used for any model tested in this thesis the hyperpa-
rameters will be tuned for this model. For this, the third-party product Weights & Biases (wandb) will be
used to ﬁnalize the model.
In the end, we will also try and measure the performance of the solutions against other solutions to the
best of our capabilities. This can prove diﬃcult for a several reasons. The literature on problems like this is
quite limited, as most academic research has been done on movie recommendations, which has a diﬀerent
type of feedback, and therefore the problem is diﬀerent. Also our hardware setup is limited, as we cannot
make scripts which can run for more than 48 hours.
The entire setup for the DeepFM model will be wrapped into a container using docker and deployed to a
fully scaleable environment using the GCP service. Inside GCP it will possible to monitor the entire setup
for the model. An example of a monitoring technique will be given as a datadrift experiment to monitor
model performance over time.
Finally, an app will be built to simulate user interactions with the recommendation system. These interac-
tions are to emulate the ones that would be made on a website if the recommendation system were to be
deployed on such a website.
It will not be a part of this thesis to deploy the model on a website, as this requires methods for tracking
customers visiting the website and linking them to the data which is previously stored. Instead, to emulate
this an app has been created from where the customer types in one’s customer id. For this customer id, a
set of recommendations will be generated and displayed.
3.3
Litterature Review
In this section related work in the recommendation ﬁeld will be presented. The related work will give
an insight into the literature in the recommendation system area, how other participants in the kaggle
competition handled the problem, and the problem the creators of DeepFM faced.
In the ﬁeld of recommendation systems, content based ﬁltering and collaborative ﬁltering is essential. These
methods can be implemented in multiple ways. However research we found few examples of deep neural
networks being used as an implementation.
Participants from the kaggle challege such as W. Jarred, hallstcd and A. Dembek have published articles
about their solutions. The conclusions made in these publications were in general that it was a diﬃcult
challenge, and their ﬁnal models had low performance [5],[6] and [7]. The winning solution [8] was an
ensemble model of catboost classiﬁer models and lightgbm classiﬁer models [8].
It was mentioned by
Senkin13 that for the winning solution they trained on machines with 300GB of memory [8]. Furthermore,
it was discussed in the kaggle forum that deep neural networks did not show good results.
It seems that there is a large potential to gain if a good implementation of a deep neural network can
be made for this problem. Such a network has been made of predicting Click-Through-Rate (CTR). This

The DeepFM architecture was proposed as a model to learn implicit feature interactions when making

Data exploration
In this section the diﬀerent data sets used in the thesis to build the recommendation system is investigated.
The data used in this project consist of 3 diﬀerent data sets. articles, customers and transactions. The
data sets consists of multiple features which is prepared and investigated before modelling. The data will
be prepared by preprocessing the features and creating a few new features from already existing features.
The data is investigated by exploring which features there is needed for making good recommendations and
which features that can be excluded. Exploring which features that can be excluded is a crucial part of the
exploration, as the goal for the model created and presented in this thesis, is to have the capabilities to
run on a website, and make real time recommendations. In order to make real time recommendations the
model needs a fast response time and by dropping unnecessary features computational powers needed to
calculate the outputs are saved. Especially in the product description data set articles a lot of overlapping
features is identiﬁed.
4.1
Data preprocessing
The three data sets customer, article and transactions are raw data sets, meaning the variables haven’t
been tempered with in any way. The variables needs to be preprocessed in order to use them for analysis
and modelling. The three data sets each contain unique features which will be preprocessed individually.
The data set customers and articles are lookup tables with metadata from H&M about the customers
and products (articles). The transactions data set includes the transactional data H&M have recorded,
and will be used as the basis for our training. In Table 1 three observations from the transactions data
set can be seen. The table shows that for each transaction made at H&M we know which customer who
bought it, the article being bought, the price and which sales channel was used. The price is not the actual
price of the product and it does not come with a currency. The price is a feature which indicate there is a
diﬀerence between the articles and some articles are more expensive than others.

4.1.1
Customers
The customers data set contains personal information about the customers at H&M and the full list of
variables in the data set can be found in Appendix A.1. The data set contains 1,371,980 observations and
has 7 variables. Each observation in the data set is a customer with a unique id and a set of features. The
unique ids are prepossessed by encoding the values so they are transformed from a large string of numbers
and letters into an integer in the interval [0 : 1371980]. The encoding is made with the sklearn python
library Ordinal encoder [9]. In the case the encoder meets a unseen value, it will encode it as "Number of
classes" + 1. The variable postal code is also being encoded this way. For the variable age there are some

Articles

and number.All variables which describe the feature with a number are excluded. The remaining variables
in the data set are encoded with the Ordinal encoder, as the customer data set.

4.3
Analysis of feature overlap
After a preprocessing of the features we can now analyse them. As mentioned in Subsection 4.1, we need
to exclude features which provide little new information in order to save computational powers. In the
preprocessing phase some variables were removed because they described the same thing with two diﬀerent
values a name and an id. However there are still a lot of variables remaining. The article and customer
dataframes contains many nominal variables which describe articles and customers. It would therefore be
interesting to test if there are variables which gives the same information. If there are variables which gives
the same information one of them can be excluded and doing so save computational power. To perform
this test the Cramér’s V value is used which calculate how associated nominal variables are [10]. The
Cramér’s V value lies in the interval [0,1], where 0 indicates the variables are not associated in any way
and 1 indicates a strong association between them. To calculate the signiﬁcance of the Cramér’s value a
p-value is used which is the same used for Pearson Chi squared test [10]. The signiﬁcance level of the the
p-value is 0.05 to see if the result of the Cramér’s value can be trusted. A threshold of 0.9 is used for the
Cramér’s value to determine if the variables are too associated. If so one of the variables will be excluded
from the data set. The formula for calculating Cramér’s V value is given in Appendix A.4.
When calculating Cramér’s V on the customers data set it is found that the variables FN and Active have
values of 0.99 and 0.97 with fashion news frequency. Hence both FN and Active will be excluded from the
data set as they can be explained by fashion news frequency.
When calculating the Cramér’s V on the articles data set multiple features with overlap in information
were found. The variable Prod name have high Cramér’s values with multiple other variables. Prod name

Theory and Methods
In this section, the theoretical methods for building a recommendation system will be explained.
The
basic understanding of a recommendation system and how it works in practice will be discussed. Then the
diﬀerent structure of the models used in this thesis and their learning process will be explained.
5.1
The recommendation system
A recommendation system is a learned algorithm used to suggest relevant products to users. More and
more companies all over the world use this technology to generate more online sales where personnel help
from shop assistants is not available. The recommendation system can be learned in numerous ways and
used to recommend anything from what ad to show a customer or what product to recommend. In this
thesis, the aim is to build a recommendation system for online retail sales using historical transactions from
H&M. As of now the recommendation setup at H&M online webshop is shown in Figure 5.1
The purpose of the recommendation system is to recommend the next relevant product for H&M customers
to buy on their online website. In Subsection 6.8 it will be shown how the recommendation will look for the
customer and this will hereby create more sales for the company as well as creating a better user experience
for the visitor even though they are not physically in the shop.
Fig. 5.1. The recommendation setup when visiting the online shop at H&M [11].
The recommendation system in this thesis is based on ﬁltering a huge set of articles into a list of top choices
for a customer. The top choices for the customer are the most likely articles to buy next. The system has
learned how to output this short list of candidates by training on a huge data set based on previously bought
items. There are many ways to model the ﬁltering process in a recommendation system, but three of the
most popular methods are: content based ﬁltering, collaborative ﬁltering and hybrid ﬁltering (a combination

of two or more ﬁltering methods). For the recommendation system in this thesis a collaborative ﬁltering
method, a hybrid ﬁltering method and a state of the art hybrid method will be used.
5.2
Filtering approaches
The ﬁltering process in a recommendation system can be modeled in many ways and can be anything from
simple ﬁltering of a few variables to complex feature interactions in high and low orders.
A popular ﬁltering approach in a recommendation system is called collaborative ﬁltering. Collaborative
ﬁltering takes users’ action into account and recommends items based on similar taste in items. Users who
have shown similar behaviour in the past are likely to have the same behaviour in the future [12]. It is
important for this ﬁltering method to have user information. As seen in Section 4 the data set contains a lot
of user information given by many user transactions and information about the users in general. The main
advantage of the ﬁltering method is as more users interact with products the better the recommendations
are going to be. For an online retail company like H&M with a large customer base and many historical
transactions, it is ideal to use a collaborative ﬁltering approach to build the recommendation system. On
the other side, collaborative ﬁltering suﬀers like most other machine learning algorithms when there is a
lack of user interactions, this is called "the cold start problem". It is a very common problem when there
are not any past user interactions for a customer, the ﬁltering approach has not learned anything for that
customer and the recommendation system can not recommend any products to the customer. That is why
that customer is called a "cold start". There are many diﬀerent approaches to building a collaborative
ﬁltering based recommendation system and overall these approaches are divided into two subcategories
called memory based and model based methods. The memory based approaches are either item-item or
user-user recorded interactions and are assumed to be without an underlying model. The model based
approaches assume a latent model which can explain interactions between user-item and make predictions
based on what the model has learned [12]. A popular model based approach is matrix factorization which
will be used as a ﬁltering method for the baseline model.
Another popular ﬁltering approach is called content based ﬁltering. The content based ﬁltering is based on
ﬁltering similar items and it is using the respective metadata. When a user engages with an item the items
with similar features will be recommended to the user [12]. Compared to the collaborative ﬁltering content
ﬁltering has the advantage of handling the "cold start problem".

Explicit feedback

up/down. This type of data is convenient to model since the users give feedback data regarding their interest
in products. The explicit feedback is essentially used as a target for the model to learn from through the
ﬁltering process. However, this type of data is not always available. It can either be personal sensitive
information for users or because users need to spend time and give feedback to products. Explicit feedback
is not available in the data sets used in the thesis.

5.3
Matrix Factorization
The idea of MF is the existence of a low dimensional latent space that describe feature interactions between
items (articles) and customers. The latent space is created by transforming the feature values into embed-
dings with a ﬁxed number of dimensions. The feature interactions are learned by computing dot products
of the corresponding dense embedded matrices [12]. In Figure 5.3 the MF method is illustrated.
Fig. 5.3. The MF ﬁltering method illustrated [12].
In Figure 5.3 it can be seen how the feature interactions are learned by computing the dot product between
a dense article and a customer matrix. The result of the dot product will be a score between the article
and the customer. The formula for MF is

which is the ﬁxed number of embedding dimensions. There will also be an error reconstruction matrix as

highest scores will then be recommended.

that share the same input. Because the two components share the same input it is not necessary to perform
feature engineering besides handling raw features, as seen in Section 4. The ﬁrst component is the FM

5.5
Scoring
Scoring is essential for recommendation engines. Scoring can be performed in two ways. The ﬁrst method
is sampling candidates from which the ﬁnal recommendations are generated. The second method is ranking
all the data, where recommendations are made from these scores directly. But none the less scoring is
required, in order to chose which products need to be recommended. Therefore scoring is maybe the most
critical part of any recommendation engine. For all the models in this thesis, a score is calculated for every
article to the customer receiving recommendations. The top k articles with the highest score are chosen as
recommendations to the customer, where k is a parameter for the number of articles to recommend.

The data used in this thesis consist of implicit feedback and has not got an explicit target vector. The only
target response the data has is positive transactions which all are equal to 1. As mentioned in Subsection 3.1
the goal of the model is to solve the binary classiﬁcation problem of whether a combination of a customer c

of 1, it is necessary to generate from which other than positive feedback can be back parsed to the model
when training.
One way to generate feedback is by creating negative samples in the data.
Negative sampling can be
implemented in multiple ways, but for this thesis, a simpler method is used as it proved very eﬀective. The
simple approach consisted of assigning articles that a customer has not bought with the target value "0".
The approach is simple, but crucial as the data now actually have values where something can be learned.
Two diﬀerent approaches were implemented for the negative sampling. Even though a large part of the
data is excluded, there is still an enormous amount of data. Therefore one approach was implemented for
the training and validation set, and another for the test set. For the training and validation n random
negative samples are created for each customer. For the test set negative samples are created for all articles
which is not a purchase for each customer. Ideally, negative samples would be created for all three datasets
using the method of the test set, but using this method the training set would become to big to handle,
within the time constraint of the thesis. In Subsection 8.1.2 the reasoning and consequences of this will be

6.5
The loss function
Throughout this thesis the pytorch implementation of binary cross entropy loss [18] will be used. This
loss function is used as since it is a case of binary classiﬁcation. This means that through training the
dissimilarities between the distribution of the training data and the distribution of the outputs of the
model (Kullback-Leibler Divergence) will be minimized. This is the same as maximizing the Maximum
Likelihood estimators. As the cross entropy (KL divergence) [19] is optimized, the outputted probabilities
of the model can be interpreted as x belongs to the positive class with % predicted probabilities, which is a
nice intuitive way to interpret the results of the model. Therefore both the model outputs and the index of
the topK product will be parsed when recommending products. This makes it possible to inform the user
of the conﬁdence in which the recommendations are made.
This is with the exception of the training of multidimensional MF model, as this is not a binary classiﬁ-
cation problem but instead a multidimensional classiﬁcation problem of all articles. Therefore the pytorch
implementation of cross entropy loss will be used.

6.8.1
Data drifting
The google cloud platform oﬀers a lot of services, and a lot of customization of monitoring is available, it
is not possible to check if the trained model is still relevant, with the built-in features. To make sure that
the recommendations being made are still relevant, and not made from data that no longer applies it is
necessary to monitor if the data changes. Therefore a data drift detection model is implemented.
Data drifting is used for testing model performance over time. An experiment using the Torchdrift methods
and library [26] was set up to test if the model needs to be retrained when given new out-of-distribution
data.
The experiment setup is created with a feature extractor and a drift detector.
Then in and out-of-
distribution datamodules are constructed with a default dataloader. The default dataloader handles the
data and makes sure non-drifted and drifted batches are constructed. In the experiment, the drift detector
KernelMMD detector is used. The KernelMMD detector uses a kernel to deﬁne a metric on the space of
distributions on the latent feature-space. As kernel the Gaussian kernel is default and used in this experi-
ment. An odd ratio of 0.8 and a sample size of 5 were used. The feature extractor is the embeddings from
the model. This means that the data passed to the drift detector is the output of the embedding part of
the recommendation model. The out-of-distribution data is the same data but with noise added.

At ﬁrst, the performance of the models are compared to see how well the models solve the binary classi-

Secondly, the performance of the models’ ability to recommend articles to customers is compared.
To
evaluate how accurate the recommendations are the mean Average Precsion(n) (mAP(n)) metric [27] is
used. This is the same evaluation metric used in the kaggle challenge. Two versions of the mAP(n) metric
will be used to evaluate how accurate the models are. The n in mAP(n) indicates the number of the
recommendations made, which are factored into the calculation when calculating the average precision of
the set of recommendations. This is because diﬀerent business goals can mean that it varies what is the
most important metric. This means that mAP(1) is measuring if one of the recommended articles is an
article the customer has bought in the test set. mAP(12) on the other hand measures the average precision
of all twelve recommendations made and therefore gives more of a general indication of the quality of the
recommendation. If the customer only makes four purchases e.g. and all those four is in the set of the twelve
recommendations, that would give a mAP(12) = 1 = 100%. This means that it is not punished to make
more recommendations than there are actual purchases. Instead, only the amount of recommendations
corresponding to the number of actual purchases is being used in the calculation of the mAP(12) metric.

8.1
Data decisions
In Subsection 4.2 diﬀerent choices of data reduction were introduced, which removed all inactive customers
and all articles which have not been purchased. An argument can be made that it would have made more
sense to remove random transactions instead of inactive customers and customers with few transactions.
The same thing can be said about articles. However, data was not removed randomly, as we believe it
is more interesting and valuable to train a model on the customers and products due to the presence of
relevant data. It will give an unfair advantage if the performance of the recommendations is compared with
the submitted solution of the kaggle challenge. However, this study only had 1/3 of the data given at the
kaggle challenge, making the training disadvantage to this study.
The decision of how the data is reduced means that only make recommendations for customers who have
made at least ﬁve purchases in the training, validation and test set combined. Therefore every customer in
the raw data set is active users and every article is an article which are being purchased. However, when
splitting the raw data set deﬁning the train, validation and test set created cold start problems. The split
resulted in a division leaving customers in the test set, with no prior purchases and therefore no purchase
history on them used in training.
The test set had 12,000 unique customers. Out of those 12,000 customers, 11,230 customers were also in
the training set. This leaves 770 customers which were cold starts, meaning no prior information about
their shopping behavior was available when recommendations were created. Also in the test set we came
across 9,565 distinct articles that had been purchased. Of those only 6,953 had been purchased in the
training set. This results in 2,611 articles that had no previous transaction history. The performance of
the recommendations of these cold start problems has not been investigated, but we can conﬁrm that the
model performed quite well in general even though these cold start problems exist. Also when compared
to the performance of the kaggle challenge winning-solution. Unfortunately, it is not possible to compare
the amount of cold start cases with the testset in the kaggle challenge as it is undisclosed, and therefore is
not known how many cold start problems exist in that data set.
8.1.1
Data enrichment descisions
In Subsection 6.2.1 a decision was made of using diﬀerent methods for enriching negative samples to the
seasonal features.
The reason for including the seasonal features was to let the model learn seasonal
tendencies in the data.
For the training and validation set uniformly distributed negative data points
throughout the year were sampled, and assumed that it would not interfere with the patterns of seasonal
tendencies found for the positive data points (purchases). For the test set random dates were sampled in
the 8 days the test set ranges.
This method of drawing random dates in the speciﬁed date range instead of a random date in the entire
year is a fair solution since this is the same information available to the contenders in the kaggle challenge.
The method can be used when making the training, validation and test set however, when making the
recommendations the method falls short. When the recommendations are provided to a customer the data

8.2
Diﬀerent performance metrics
When choosing which model to use for making recommendations of high quality, it is important to measure
the performance with well-suited metrics.
For this thesis two diﬀerent performance metrics were chosen in regards to evaluating the recommendations:
mAP(1) and mAP(12). They are both important at measuring how accurate the recommendation are, and
they should be prioritized depending on the need.
mAP(1) measures if the model can predict one of the recommended articles that the customer ends up
buying. Therefore this metric is a guiding metric if the business goal is to have the best possible accuracy
at ﬁnding just 1 product that will get the customer’s attention.
mAP(1) do not factor in the quality of every other recommendation than the one which was correct. As
one correct recommendation prediction will give an overall Average Precision (AP) for that customer equal
to 1 = 100%.
mAP(12) on the other hand does take into account every recommendation prediction made for a customer.
This metric measures if the model can predict twelve recommendations correct of twelve possible recom-
mendations. It measures the diﬀerence between the recommendations made for a customer and the actual
48
==PAGE:50==
Master
Construction of a recommendation system for best next buy recommendations
01.01.2023
purchased articles. However, it is not all the customers who made twelve purchases in the test set. Ideally,
all twelve recommendation predictions should be measured for mAP(12), but in the case where a customer
has made less than twelve purchases in the test set it is not possible. This will be further discussed in
Subsection 8.3.
With the metric mAP(12) the quality is quantiﬁed for all the recommendation predictions. The metric gives
an idea of the model’s general capabilities of making good recommendations and not just the capability of
creating 1 good recommendation out of a set of twelve recommendations.
As the main goal of the model is to provide a good user experience all twelve recommendations must be of
high quality and not just one of the recommendations. Customers might get annoyed to get commercials
of articles that have little signiﬁcance for the customer, even though there is one very signiﬁcant article.
This is why mAP(12) was used as the primary metric when comparing model’s accuracy.

8.3.1
Including metadata and comparing model architecture
At ﬁrst the importance of including metadata is discussed.
In Subsection 7.1.6 it was shown that the
baseline model had the worst quality of the recommendations. The diﬀerence between the baseline model
and multidimensional MF model shows the eﬀect of including metadata on the performance. In Table 4

model has an mAP(12) score of 0.006559.
This means that exploiting the metadata when solving the
problem makes the result 23 times more accurate as the baseline model according to mAP(12). However,
two diﬀerent methods for creating feedback when training the two models were used which also could impact

for the multidimensional MF model and therefore necessary for this experiment.
Besides the multidimensional MF model, all the other models use the same method for creating feedback as
the baseline model and they are all performing better than the multidemensional MF model, which indicates
that the feedback method which the multidemensional MF model uses isn’t the reason for the increased
accuracy the multidemensional MF model has compared to the baseline model. With these observations,
it can be seen, that it is possible to conclude that metadata does make an improvement to the model
performance and should be included.
Next, it was decided to use another method of computing prediction scores, with that being the FM model.
This gave an increase in performance, taking the mAP(12) from 0.006559 using the multidimensional MF
model to a mAP(12) of 0.06898. Thus changing the problem setup this way and using the FM model for
predicting scores, the recommendations became 9.5 times more accurate. This new way of setting up the
problem as shown in Figure 6.2 makes it less computationally heavy to create and enrich negative samples
as it would have been with the multidimensional MF model. The results indicate the setup of creating
many binary classiﬁcation problems and then ranking them as allowed by using the FM model is the best
performing method accuracy-wise of the methods presented as of yet.

The next step was changing the method for computing prediction scores by using the deep neural network

they performed when making recommendations for the next best buy option at H&M. As mentioned the
assumption was that the DeepFM model would perform the best, as it should be able to in theory to capture
both low- and high order relations in the data. This wasn’t the case, as the FM model implementation
turned out to perform the best, on this dataset.
We believe the reason for this is because the two components of DeepFM model (FM component and MLP
component), work as noise for each other when the ﬁnal prediction scores are calculated. This is believed
because both components as stand-alone models perform better than the DeepFM model. To conﬁrm this
an experiment is created where recommendations were created for 2,125 unique customers.
From this
experiment, we can make a more thorough analysis of for which customers the respective models can make
one correct prediction. The FM model predicted one correct article for 429 customers. The MLP model
predicted one correct article for 159 customers. The DeepFM model predicted one correct article for 107
customers. Of the 159 customers for whom the MLP model predicted one correct article, 76 were in the set
of the 429 customers for whom the FM model predicted one correct article. Between these two sets, there
is a signiﬁcant overlap, but for the majority of the set, the two models predicted one correct article for
diﬀerent customers. Therefore we believe the prediction scores made of the two components are somewhat
equalizing each other instead of taking the best from the two diﬀerent methods.

8.3.3
The diﬀerence between mAP(1) and mAP(12)
The next observation made in Subsection 7.1.6 is the diﬀerence in the mAP(1) and mAP(12) is not as
big as one might have expected. Depending on the model, the models are somewhere between a bit less
than twice as good to almost four times as good at just getting one correct prediction compared to getting
the average precision at up to twelve purchases. The diﬀerent ratios between mAP(1) and mAP(12) is an
expression of the number of diﬀerent data relations that the model was able to learn. The reason for the
relatively small diﬀerence between the metrics is presumable because of the distribution of the data. In
the test set, there are 12,000 distinct customers and there are 47,784 transactions. This means that on
average a customer made 3.982 transactions. This is quite a bit lower than twelve which is what iw being
recommended in this thesis. In the case where a customer had made twelve purchases it was desired to
factor all twelve recommendations in the accuracy metric. However, as a lot of customers did not make
that many purchases, in most cases all twelve recommendations are not factored into the calculation of the
mAP(12) metric. Instead, it is only the N best where N is the number of purchases made by the given
customer as explained in Section 7. Actually, the median of purchases made in the test set by a customer is
three. Therefore in a large amount of the test cases, an mAP(3) metric would score the same as a mAP(12)
metric. This is unfortunate, and it would have been more informative to have a test set where the accuracy
of every recommendation could have been measured. Although the customer buying the most articles in
the test set purchased 58 articles. This means there are cases of customers buying a lot of articles, for
whom every recommendation in the recommendation set is factored into the metrics.
8.3.4
Runtime of models
From Table 4 it can be seen, there is a relationship between the runtime and the complexity of the models.
The complex models using all features require more computations which is resulting in longer runtime.
The FM model which proved to be the best performing model is not the fastest model and takes almost a
second to predict recommendations for a customer. When making recommendations for an online shopping
service the customer should never have to feel that something is being processed. The customer should
receive recommendations in real time, but at the same time receive meaningful recommendations. The
trade-oﬀbetween these two comparisons is something H&M must decide on which to weigh more. If H&M
decides that the FM model is too slow to make real time recommendations, they could use one of the faster
models but with worse performance for recommendations. Then H&M could instead use the FM model for
something else where performance is crucial. This could be if H&M wanted to send personalized marketing
to their customers. This could be an email with products that they think the customer may like. When
receiving an email runtime of recommendations is not of importance, as the customer is not experiencing
the runtime, only the accuracy of the recommendations.

more engineering work it is clear the runtime could be signiﬁcantly lowered and result in a better user
experience when interacting with the app. However, due to time constraints, this idea was abandoned and
accepted that the cloud pipeline is running slowly. But it is clear to us with a bit more engineering work
that the recommendation time could be lowered substantially.

8.4.2
Datadrift experiment
Datadrift detection is extremely important for machine learning models. The deployed model should be
robust and not degrade in performance over time. The datadrift detection experiment performed in this
thesis showed that as new out-of-distribution data was fed to the recommendation model the datadrift
detection model was able to distinguish between the data the recommendation model was trained on and
the out-of-distribution data.
This indicates that with a change in model input data, which was done
using the implemented corruption function, the performance is degrading for the recommendation model.
Therefore it is important to retrain the recommendation model when it receives new data like the corrupted
out-of-distribution data created in this thesis. It is diﬃcult to set up a precise criteria for when the data
change is enough for retraining the model as it depends on how accurately the model needs to behave.
However when a certain criteria is determined it is possible to monitor when the model has degraded
enough and should be retrained. The monitoring can be performed by deploying the datadrift detection
model on GCP where a certain criteria can be added, which triggers a notiﬁcation to the developer when
breached.
The datadrift experiment in this thesis was meant to be a display of the importance of monitoring models
after deployment and not an actual part of the deployed recommendation system. The drift experiment
was only performed on a small subset of the data and the noise was added to all input features equally
meaning embedding values for sales channel id and customer id received the same noise to the data. The
noise from the corruption function was random uniformly generated as 0 or 1. This was chosen as noise
to the corruption function since most of the data features are encoded string values. It would have been
infeasible to add numerical noise to the words, but an altering of the words could have been carried out.
However, this was decided to be an over-complication of creating noise since changing the encodings with
integers gives the encodings another meaning.
Adding the noise equally to the features was not optimal as the embedding values of the features are
signiﬁcantly diﬀerent. customer id has a huge number of unique values encoded in the embeddings and
sales channel id only has two unique values. When the noise is added to the customer id it does not aﬀect
the overall signiﬁcance for the variable as much as with sales channel id. To make the datadrift experiment
more thorough the corruption function could add noise to the input features individually making the change
in the data distribution more meaningful for the features.

Conclusion
The conclusions made in this thesis regarding models and model performance can be divided into two
parts. At ﬁrst it can be concluded a model was successfully developed using historical transactions from
H&M and deployed in a cloud infrastructure. The deployed model was the FM Model. It created precise
recommendations proved with a mAP(12) score of 0.06898, which is multiple times higher than any of the
other models. Therefore the model can hopefully help create good customer experiences by recommending
the next best buy option.
It was assumed the DeepFM architecture could outperform the traditional methods like FM, since it already
had outperformed FM in a problem of CTR prediction which also deals with binary classiﬁcation by H.
Guo et.
al.
[4].
The assumption that it would do better for the recommendations problem was not
proven true. The DeepFM architecture on the H&M problem proved to be worse in both accuracy of
the recommendations and runtime when compared to the FM method.
Like other participants in the
kaggle challenge [8], [29] and [30] it was concluded that deep neural networks are not good at solving the
recommendation problem for the H&M data set.
Concerning the deployment of the model. It can be concluded that it was possible to construct a container,
in which the entire setup for the FM model and all its dependencies could be contained. This container could
be uploaded and deployed on a cloud infrastructure. This infrastructure hosted the app in the container
in a way so that the capacities of the system infrastructure could be up scaled depending on the needs as
a result of the traﬃc into the app. A concrete experiment was conducted for this where it could seen that
the cloud infrastructure got upscaled when multiple requests were made for recommendations. Monitoring
dashboards were created allowing the hosts of the app to monitor the system and the pressure put on the
system when making recommendations. This system emulates what could be at the H&M website, but
without the customer tracking.
An example was given of how to monitor datadrift in the model and hereby provides information about
when to retrain the model and avoid degradation in performance over time. The datadrift detection model
was able to detect a drift in the data measured on the mapped data in the latent space. It was possible
to detect when the model should be retrained because input data no longer follow the same distribution as
when it was trained.
The deployed pipeline created as a whole in this thesis did not meet the goals of being able to create real
time recommendations. The runtime exceeded the acceptable time for a customer to wait for the page to
load. This is not due to the models’ creation of recommendations as this was done in under a second, but
due to the long time it takes to fetch data and the models from the cloud storage.

Further Works
There a multiple things that could have been done to improve the solution proposed in this thesis. In this
section we will quickly mention the ones we have thought of, but didn’t implement for one of two reasons.
The ﬁrst reason is that it was deﬁned as out of scope in the Subsection 3.1 or because it was decided to
not implement it because of time constraints or hardware limitations.
To improve the FM model it would be interesting to investigate the recommendations made, and see if it
would be possible to ﬁnd a pattern in the errors made by the model. Maybe this would help to ﬁnd a way
to improve the candidate generator. As mentioned in Subsection 8.3 almost all the modelling is done for
the candidate generation. It would be intereseting to build upon the retrival function and go from the topK
towards a neural netowrk to best pick from the output layers. A training of this retrieval layer, could help to
solve the underlying problem which is the recommendation problem, and not just the binary classiﬁcation
problem.
In regards to analyzing the recommendations a deeper analysis of the dissimilarity of the recommendation
could have been made, by placing the articles in segments and see if every recommendation is in the same
segment.
Furthermore it would be interesting to investigate what would happen if the embedding dimension was
individual for the features. There is huge diﬀerences in the number of unique values in the features which
make the embedding representation diﬀerent for each feature. For the sales channel id feature it does not
make much of a diﬀerence if the embedding dimension is changed from 30 to 50. However for customer id
there could be a signiﬁcant diﬀerence.
Another issue relevant to implement was the connection to the image database. H&M also included images
of all the articles in the database, but to limit the project this part was excluded of our analysis and
scope. However it would have been nice to upload all the images to Google Cloud Storage, and create an
infrastrucure which could extract the image corrosponding to the article id which had been recommended.
Then a full implementation of a datadrift detection model on the cloud infrastructure would make it possible
to see when the model receives data, which is too drifted compared to the data which the model was trained
upon.
As written in Section 3, it should be the goal of any website to create a good experience for the customers
of that website. It was argued that good predictive recommendations for a customer can help improve the
experience on the website. The work presented in a case study as this thesis can not achieve this alone. To
run on a website and create real time recommendations the model would need to be fetched data on whom
to create recommendations for. Therefore an extended setup is needed which can ﬁrst of all see who is the
user of the website, either by an login account on the website, or by placing cookies on the user. And then
pass the users customer id to the model. The model will then make predictive recommendation, which can
be parsed back to the system, from which an experience can be tailored on the basis of the models output.
From here the experience can be made even more personalized based on what the users click in the store
and what the users search for. And thus hopefully generating a better experience for everyone visiting the
website where the model have been implemented.
