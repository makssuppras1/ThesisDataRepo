Abstract
Outsourced computation has become a common practice, raising critical concerns about
data privacy and security. These concerns include direct access to client data by the
service provider or malicious entities on the server, which can expose sensitive informa-
tion. Additionally, indirect access through side-channel vulnerabilities, such as attacks
from other tenants on the server, poses a significant threat. The Blinded Memory (BliMe)
architecture addresses these issues by implementing a hardware-enforced taint tracking
policy, ensuring that sensitive data remains confidential and cannot be compromised even
in the presence of malicious software or side-channel attacks.
This thesis addresses the challenge of ensuring confidentiality in a System-on-a-Chip
(SoC) implementing the BliMe architecture. This work models a BliMe system that sup-
ports multiple CPUs, accelerators, and various peripherals, all communicating through a
shared bus. The model introduces a peripheral firewall that enables the integration of
untrusted and I/O peripherals while preserving the system’s confidentiality guarantees.
Using the F* programming language, we formally verify that the modeled BliMe system
maintains confidentiality across all system components.
This thesis’s contributions extend the BliMe architecture and introduce a model for BliMe
systems that formally guarantees confidentiality in outsourced computation. The findings
provide a foundation for future research in hardware security and formal verification.

Introduction
In today’s world, outsourced computation has become a common practice. Clients upload
their data to remote servers for processing, raising critical concerns about the privacy and
security of their data. Typically, a client uploads their data to a remote server, which then
uses a program to execute the desired computation and returns the result to the client.
In this scenario, the server keeps its code confidential, while the client must reveal their
data to the server. This situation presents a dual concern of protecting the confidentiality
of client data against any malicious entity in the server, including protection from direct or
indirect threats such as side-channel attacks [1] while ensuring the server’s code remains
confidential. Addressing this issue necessitates a robust solution for secure outsourced
computation.
The Blinded Memory (BliMe) [2] architecture is a solution to this security challenge. BliMe
is an innovative architecture that enhances the security of outsourced computation. BliMe
defines a set of extensions to Instruction Set Architecture (ISA) that implements a hardware-
enforced taint-tracking policy in Central Processing Unit (CPU) to ensure the confidential-
ity of client data on a remote server. This taint-tracking policy ensures that any client
data processed by the CPU is tainted and cannot be exported from the system. BliMe
employs a secure hardware module to handle data import and export securely. Importing
client data to the server makes the data tainted, preventing data extraction except through
the secure hardware module. This architecture not only guards against side-channel at-
tacks but also defends against malware and run-time attacks, ensuring robust protection
for client data even if the server software, including the OS, is compromised.
Several approaches other than BliMe have been proposed to address this challenge,
though they are less suitable. One approach involves cryptographic methods such as
Fully Homomorphic Encryption (FHE) and Secure Multi-Party Computation (MPC). FHE
[3] allows computation on encrypted data, enabling client data to remain encrypted while
being processed on the server. However, FHE has a significant drawback due to its high
computational costs [4]. MPC involves cryptographic protocols that allow computations to
be performed with each party keeping their sensitive data private. However, MPC is often
slow due to network overhead and the significant number of cryptographic operations
required [5]. Another way to ensure confidentiality in outsourced computation is the use
of Trusted Execution Environments (TEEs). TEEs [6] provide hardware-based isolation
based on a root of trust in the hardware. This isolation ensures that programs outside the
TEE cannot access the data within it. However, TEEs are not suitable when the server’s
code is confidential and unverifiable. Additionally, recent research has shown that the
resource sharing and large code bases of TEEs make them susceptible to bugs and side-
channel vulnerabilities [7, 8, 9].
Because BliMe aims to solve such a critical problem, it is crucial to evaluate the security
of systems based on this architecture using formal methods and prove that it successfully
achieves its security goal.
In this work, we model a System-on-a-chip (SoC) with the BliMe architecture and for-
mally verify that this BliMe SoC ensures the same confidentiality guarantees for the entire
system. Our model incorporates implementation details found in modern systems, in-
cluding multiple CPUs, accelerators, the system bus, and Input/Output (I/O) peripherals,
to ensure confidentiality even in untrusted and I/O peripherals. This modeling provides
an instance of functionally modeling hardware components and verifying their properties
using deductive verification [10] tools, offering a more realistic and applicable solution for
==PAGE:14==
modern systems.
Formal verification of a model can prove that a set of properties holds in the model. If this
model accurately corresponds to a concrete implementation, that implementation inherits
the same properties. Because the model is closer to reality than a more high-level theo-
retical proof, this process provides higher assurance that systems with BliMe architecture
possess the security property. To formally verify a BliMe system, we must first model
the behavior of BliMe systems. Modeling BliMe systems serves two important purposes.
Firstly, it allows us to verify that the system satisfies specific properties. Secondly, it pro-
vides a reference implementation and design blueprint for the system and its components.
In previous work [2], confidentiality guarantees were formally verified for a monolithic
BliMe CPU. The CPU was modeled primarily to construct a sample ISA and define taint
propagation rules within it. This modeling demonstrated that the sample ISA ensures
the confidentiality of client data. However, to extend a formally verified BliMe system
with confidentiality guarantees to the entire system and provide proof for stronger confi-
dentiality guarantees in the presence of malicious clients and untrusted peripherals, it is
necessary to model the entire system with a more accurate representation of hardware
implementation details found in modern systems.
Our main contributions are as follows:
• Provide a BliMe SoC model that realistically represents modern hardware imple-
mentations by modeling the system with multiple processors and peripherals com-
municating over a bus and proving that this model ensures the confidentiality of
client data (Section 4.2).
• Provide a model for all typical system components to prove that the BliMe architec-
ture is completely implementable (Section 5.3).
• Extend the BliMe architecture to provide a solution for attaching untrusted and I/O
peripherals to a BliMe system for increased real-world adoption (Section 4.2.2.4).

Background
2.1
Formal verification
Formal verification is a technique for verifying properties in software and hardware. Un-
like testing, which evaluates a system’s behavior in specific scenarios, formal verification
mathematically proves that a system adheres to its specifications under the assumptions
defined by the model.
Formal verification methods are mainly divided into two categories, which are model
checking and deductive verification [11, 12, 13, 14, 15]. Equivalence checking is another
important verification method, particularly in the hardware electronic design industry [16,
14]. It is a method to prove whether two representations at different levels of abstrac-
tion are functionally equivalent [17]. Static and dynamic analysis techniques, such as
abstract interpretation and syntax and semantics checks, are also noted as verification
methods in some literature [15, 16, 18]. In abstract interpretation, program behavior is
over-approximated by abstracting values, using signs, possible ranges, or relations be-
tween variables instead of actual concrete values [19]. Abstract interpreters and other
analysis techniques have the advantage of using an approximation to address theoretical
undecidability [18].
Deductive verification proves whether logical statements can be derived from axioms,
lemmas, and inference rules. Model checking is based on abstractions of systems into
finite state representations, which are then analyzed using state-space exploration of the
model to check whether the model satisfies the desired properties [20]. Model check-
ing increasingly focuses on enhancing the expressivity of systems and properties without
compromising automation. In contrast, deductive verification is evolving to become more
automated. Typically, model checking offers higher levels of automation, whereas deduc-
tive verification provides greater expressibility of systems and properties [18].
In this work, we use deductive verification to formally verify the confidentiality guarantees
of our model. Deductive verification is mainly performed using theorem provers or tools
that utilize theorem provers or Satisfiability Modulo Theories (SMT) solvers. Theorem
provers can be categorized into two groups, though there are overlaps between the tools
and techniques involved:
• Interactive theorem provers: These tools, also known as proof assistants, require
human guidance. A well-known example is Coq [21]. Proof assistants facilitate
procedural proofs using tactics to automate the verification process by reducing
complex goals into simpler subgoals. They are called interactive because the user
specifies a tactic, which the prover executes to update the proof state. The proof
state comprises the set of goals needing proof and the known facts. This interactive
process continues until all goals are resolved, indicating the theorem is proven.
• Automated theorem provers: These tools aim to automate the verification pro-
cess fully. Propositional logic, for instance, has a decidable theory and can be fully
automated but lacks expressibility. Another example is first-order logic, which offers
greater expressibility and is semidecidable, allowing for full automation. However,
complex proofs in first-order logic might take a significant amount of time to complete
or may never reach a conclusion.
Some theorem provers use SMT solvers to increase the automation of the verification pro-
cess. SMT solvers [22] are tools used to verify the satisfiability of logical formulas based
one or more theories. A famous example of these tools is Z3 [23]. SMT solvers generalize

the Boolean Satisfiability Problem (SAT), which involves determining whether an interpre-
tation exists that satisfies a given Boolean formula, to other domains such as integer and
real arithmetic, bit-vector reasoning, and more [23]. Although SAT is Non-Deterministic
Polynomial-Time complete (NP-complete), many theories lead to a decidable SMT prob-
lem that can be fully automated. SMT solvers check the satisfiability of formulas using
various approaches, such as decision procedures for decidable theories and heuristic
search for semi-decidable or undecidable theories. For instance, the interactive theorem
prover Isabelle/HOL [24] incorporates Sledgehammer [25, 26] to automate the verification
of current proof goals. Sledgehammer utilizes both automated theorem provers and SMT
solvers to find proofs, extending its capabilities beyond those of traditional automated
theorem provers [27].
An important group of deductive verification tools is those that apply deductive verification
techniques specifically to programs. These tools often provide a programming language
with extensive specifications, rich type systems, and functionalities to define axioms, lem-
mas, and assumptions. These tools will transform program specifications, axioms, and
types into formulas to be checked for validity. They may function as theorem provers with
SMT-assisted automation, using SMT solvers alongside tactics and logical inference to
verify the validity of formulas themselves. Alternatively, they may rely entirely on SMT
solvers for this verification. These tools are usually semi-automated, known as semi-
automated deductive program verification tools. Some popular choices in industry and
academia include Dafny [28], Why3 [29], Viper verification infrastructure [30], and FStar
(F*) [31].
2.2
Semi-automated deductive program verification
Program verification means verifying that the program will behave according to a speci-
fication. Then, we use program verification to prove that the program behaves correctly
according to the specification.
Program verification tools and concepts used in this work are based on Floyd-Hoare
logic [32] and the weakest pre-condition calculus [33]. The theoretical foundations of
Floyd-Hoare logic are mostly based on [32] by Hoare. His paper is based on the pa-
per [34] by Floyd which generalizes the idea of Turing in [35]. Floyd-Hoare logic operates
on Hoare triples, which consist of the pre-condition, the program, and the post-condition,
and they are represented as:
{P} C {Q}
(1)
Hoare triples state that any program C satisfying the initial condition P will, after exe-
cuting C, if it terminates, result in a state that satisfies the condition Q. The weakest
pre-condition, denoted wp, is the least restrictive pre-condition that will guarantee a given
post-condition based on a program [33]. We can, therefore, replace the pre-condition in
the Hoare triples with the weakest pre-condition calculated based on the program and its
post-condition.
{P} C {Q} into {wp(C, Q)} C {Q}
(2)
Using Hoare triples and weakest pre-condition, the post-condition Q will hold if:
P ⇒wp(C, Q)
(3)
By calculating the weakest pre-condition with respect to the program and the post-condition,
we can determine that the validity of Equation (3) implies that the program is correct ac-
cording to our specifications. Here, ”correct according to our specifications” means that

program to exhibit, and the pre-condition similarly represents the expected conditions at
the beginning of the program’s execution. Thus, by verifying the validity of Equation (3),

2.3
F*
F* [31] is a proof-oriented programming language. It utilizes an SMT solver to perform
semi-automated deductive program verification. A distinction between F* and other semi-
automated deductive program verification tools is that it can also act as a proof assistant.
Another important feature that F* provides is its ability to compile and extract the resulting
verified code into OCaml for execution. A tool named KaRaMeL [36, 37] can also compile
various subsets of the F* language to F#, WebAssembly (WASM), and C code.
2.3.1
Dependent types and refinement types
F* is a dependently typed programming language. Dependent types and dependent re-
finement types allow the program to express properties of types more specifically in order
to catch more errors during compile time rather than run time [38]. Dependent types are
types whose definitions can depend on values. Refinement types enable constructing
sub-types of other types. That is, given a type T and a predicate P, we can have:
{v : T | P(v)}
An example to demonstrate both of these concepts together would be:

2.3.2
Verification Principle
F* is a powerful deductive program verification tool that leverages dependent types and
refinement types, allowing developers to enrich their programs’ type systems extensively.
By using developer-defined lemmas and these enriched types, specifications can be
seamlessly integrated into programs.
The verification process in F* is facilitated through the use of tactics and SMT solvers.
Verification goals can be proven either through the use of tactics within F* or by passing
them to an SMT solver to check the satisfiability of the generated verification conditions.
Tactics give the programmer more control over breaking down a proof and guiding the
SMT solver to find a proof. This combination enables a high degree of automation in the
verification process while allowing F* to function effectively as a proof assistant.
F* supports the verification of effectful programming, further extending its applicability and
power in developing and verifying complex software systems. However, in this thesis, we
rely only on pure computations, so the explanation of the verification principle provided
here is limited to these types of computations. Each function in F* is annotated with
the type of the return value and the effect type of the function. The default effect of all
functions, if not specified, is Tot, the effect of total computations that always terminate on
all inputs and exhibit no observable effects except for the value they return.
F* provides a mechanism for writing Hoare-style specifications directly. However, the
types of arguments and return value are automatically translated into these Hoare-style

specifications in the F* compiler. So, developers usually annotate their functions’ ar-
guments and return types with dependent refinement types instead of directly writing
Hoare-style specifications. When type-checking a program, F* computes the weakest
pre-condition with respect to the program and its types, which characterizes a necessary
condition for the program to satisfy all its typing constraints.
Lemma is an instance of the Hoare-style refinement of pure computations. With lemmas,
we can annotate a stronger weakest pre-condition specification to help verify certain prop-
erties that are not automatically verified by F*. They are usually helpful for induction-based
proofs.

2.4
Side-channel and other related attacks
Side-channel attacks exploit indirect information leakage to infer sensitive data from a sys-
tem. These attacks do not directly compromise the system’s algorithms or data through
traditional means. Instead, they leverage the implementation-specific physical or behav-
ioral characteristics of the system’s operation [41]. These characteristics include timing
information, electromagnetic emissions, and power consumption.
Some examples of side-channel attacks are:
• Cache timing attacks [42, 43, 44]
• Timing attacks [45, 46]
• Transient execution attacks [47, 48]
• Power analysis attacks [49]
• Electromagnetic (EM) attacks [50]
• Differential fault analysis [51]
• Data remanence [52]
In this work, we primarily target securing the system against cache and timing side-
channels. However the other types of side-channel attacks are also discussed.
2.4.1
Cache side-channel attacks
Cache attacks exploit the differences in access times between data available in the cache
and data that is not. These attacks leverage the fact that memory access times are sig-
nificantly faster for data in the cache compared to data in the main memory. By carefully
monitoring these access times, attackers can infer the victim’s memory usage patterns.
For instance, if an attacker notices that certain cache lines are frequently accessed, they
might infer that specific data is being repeatedly processed, potentially revealing sensitive
information like cryptographic keys.
There are several general techniques used in cache attacks. One common technique
involves filling the cache with the attacker’s own data and then monitoring how this data
Formal Verification of Confidentiality Guarantees in a Blinded Memory SoC
7
==PAGE:20==
is affected when the victim accesses memory. By timing the re-access of the same data,
the attacker can determine whether the data was evicted by the victim’s access, thereby
revealing information about the victim’s memory usage patterns. This technique is called
Prime and Probe [44].
2.4.2
Timing side-channel attacks
Timing attacks exploit variations in the execution time of programs with different inputs to
deduce information about the data involved in those operations. The time it takes for a
cryptographic algorithm to process data can vary based on the input values. Even minor
differences in execution time can leak information about the data being processed. This
type of attack can be particularly effective against algorithms that have data-dependent
execution paths.
Several general techniques are used in timing attacks. One technique involves measuring
the time it takes to perform specific operations repeatedly. By analyzing the variations
in these timings, an attacker can deduce sensitive information, such as cryptographic
keys or other data-dependent values [45]. Additionally, different branches in code can
have different execution times. An attacker can measure the execution time of a branch
and infer the value of the condition that determined the branch. This reveals sensitive
information about the processed data.
2.4.3
Transient execution attacks
Transient execution attacks exploit the traces left by instructions that alter the microar-
chitecture’s state without committing these changes to the CPUs architectural state [47].
These attacks leverage transient execution, which arises from two primary causes, spec-
ulative execution and out-of-order execution. Speculative execution is a hardware opti-
mization technique to increase performance which allows CPUs to execute instructions in
advance based on their prediction of the execution path. Transient traces from speculative
execution occur when the CPU incorrectly predicts the execution branch or the memory
dependency, leading to the execution of instructions that are ultimately rolled back, but
leave detectable traces in the microarchitectural state of the CPU, such as cache line al-
locations. Similarly, out-of-order execution in the CPU pipeline results in transient traces
if a preceding instruction triggers a fault.
Spectre [53] and Meltdown [54] are famous examples of transient execution attacks.
• Spectre Variant 1 (Bounds Check Bypass): This variant leverages speculative
execution to access out-of-bounds memory reads. By misleading the CPU, an at-
tacker can cause it to speculatively read unauthorized memory and subsequently
reveal this data via cache side-channels.
• Spectre Variant 2 (Branch Target Injection): This variant takes advantage of indi-
rect branch prediction. An attacker can manipulate the branch predictor of the CPU
to speculatively execute instructions at a chosen address, potentially exposing sen-
sitive information.
• Meltdown: Meltdown takes advantage of a race condition between checking priv-
ileges and accessing memory during instruction execution. It uses speculative ex-
ecution and cache side-channels to allow unauthorized reading of kernel or other
processes’ memory from user space.
2.4.4
Rowhammer attack
In a Rowhammer attack [55], the attacker can cause bits in memory to flip without directly
accessing those bits. This is achieved by repeatedly writing to rows adjacent to the mem-
ory cell containing the target bit. While these bit flips directly violate the integrity of the
8
Formal Verification of Confidentiality Guarantees in a Blinded Memory SoC
==PAGE:21==
system, they can also compromise confidentiality by providing a means to extract sensi-
tive information or gain unauthorized access using the modified memory. Rowhammer
can also be used as a side-channel attack. An attacker can perform a Rowhammer attack
on accessible parts of memory to infer the values in adjacent memory rows through an
attack known as RAMBleed [56].
2.5
Information flow control
Information flow control is a technique for tracking how information moves through a pro-
gram to ensure the program handles the information securely [57]. It ensures that dan-
gerous inputs do not reach security-critical code and that sensitive data does not leak to
unauthorized parties. By employing information flow control, we can achieve a secure in-
formation flow within a program. Secure information flow is essential for maintaining data
confidentiality and integrity in software systems, particularly in environments with mixed
sensitivity levels.
One approach to implement information flow policy control is with taint tracking. To ensure
data confidentiality, sensitive data is marked (or tainted) and tracked as it propagates
through the system [58]. If sensitive (tainted) data influences other parts of the program,
those parts are also considered tainted. The goal of this type of taint tracking is to ensure
that sensitive data does not leak to untrusted parts of the system.
Typically taint tracking to ensure confidentiality involves:
• Taint sources: Points where sensitive data enters the system.
• Taint sinks: Points where data leaves the system, potentially exposing sensitive
information.
• Propagation rules: Rules that govern how taint spreads from sources to sinks.
Taint tracking has been implemented in a number of famous products. The first scripting
language to add taint tracking mode was Perl [59]. After that other languages such as
Python also provided taint tracking to enhance the security of their programs and to aid
the programmers to avoid mistakes that lead to attacks such as SQL injection and cross-
site scripting [60].
2.6
Non-interference
In [61], Goguen and Meseguer formalize and present the notion of non-interference, which
is a multi-level security policy. Non-interference is a prime example of describing a secure
information flow in a system with the goal of confidentiality. Non-interference requires that
high-level users’ input into a system must not affect observable outputs for low-level users.
In general, a security domain u with sensitive information is non-interfering with domain
v if any action in domain u cannot influence what is observable from domain v. Simply
put, inputs from domain v should always produce the same output from the perspective
of domain v, regardless of inputs from domain u.
2.7
Hyperproperties
Some security policies, such as non-interference, cannot be formalized as properties over
a single execution trace. These security policies can be expressed using hyperproper-
ties [62], which are properties over a set of execution traces. Hyperproperties similar to
properties can also be formulated into safety and liveness categories [63].

2.8
Blinded Memory
BliMe [2] is an architecture designed to enhance the security of outsourced computation.
It introduces extensions to ISA to implement a hardware-enforced taint-tracking policy,
ensuring the confidentiality of client data. The core mechanism of BliMe involves importing
and exporting client data with special hardware and enforcing an information flow policy.
They define confidentiality such that when a client provides sensitive data to the server,
no other entity can infer any information about the data. The only exception is that the
length of the data can be known.
A BliMe system has three main components that differ from an average system. The first
component is an Hardware Security Module (HSM) that performs the handshake with a
client and stores the encryption and decryption keys inside an encryption engine. The
second component is the encryption engine, which provides two atomic operations to the
system. It will decrypt and blind the encrypted data received from the client, and it will
unblind and encrypt the resulting computation and send it to the client. The final compo-
nent in a BliMe system is the extended ISA, which implements taint-tracking to ensure the
data imported into the system does not leak anywhere observable to any adversaries. If
any attempt is made to leak sensitive data, the BliMe CPU is designed to fault, thereby
preventing the leak. BliMe maintains that all observable system states, including those
observable through side-channels, are independent of any sensitive data, thus ensuring
the confidentiality of the data and countering side-channel attacks.
The system operates by performing a cryptographic handshake between the client and
the HSM, storing the shared session key securely within the encryption engine. This al-
lows the client to send its data encrypted, which the service running on the server using
the encryption engine can then decrypt and mark as blinded. After decryption, the service
running on the server performs computations on this blinded data, ensuring that no sen-
sitive information is leaked, and finally re-encrypts the results using the encryption engine
and sends it back to the client.
2.8.1
BliMe protocol
• Remote attestation and key agreement: The client initiates a handshake with the
HSM to agree on a session key. The HSM provides attestation to the client, verifying
that it is part of a remote server equipped with BliMe hardware, which includes both
the HSM and the encryption engine, and that these components are implemented
correctly. The session key is stored securely in the encryption engine, along with a
unique blindedness tag that identifies the client.
• Data import: The client uses the session key to encrypt its data and transmits the
encrypted data to the server. The server then using the encryption engine’s data
import function, decrypts the data and marks it as ‘blinded‘ by setting a blindedness
tag associated with that data in registers and memory. This tag indicates the data’s
origin and ensures it is handled securely.
• Safe computation: The service on the server performs computations on the blinded
data, following a taint-tracking policy to prevent any data leakage. This policy en-
sures that operations on blinded data, which could lead to exfiltration or side-channel
attacks, are unsuccessful and result in a fault in the CPU.
• Data export: After computation, the server using the encryption engine’s data ex-
port function, encrypts the results for communication over the internet and marks
the encrypted result as non-blinded. The client then receives this encrypted data
and decrypts it using the session key.

2.8.2
Taint-tracking policy
BliMe employs a taint-tracking policy to prevent sensitive data from being exposed through
observable outputs. The system state, depending on whether it is permitted to become
blinded, is split into two categories:
• Blindable state: Includes only values and not addresses. These values can be in
the registers (except for the Program Counter (PC) register), main memory, cache,
data buses, and queues. All of these values can be marked with a blindness tag.
• Observable state: Includes information that can be observed from outside the sys-
tem, such as PC, cache line addresses, branch predictor state, and performance
counters, which must never contain sensitive data.
The policy dictates that any instruction with a blinded input produces blinded outputs, in-
heriting the input’s blindedness tag. The policy also dictates that if an instruction depends
on inputs with different blindedness tags, or if an instruction attempts to write blinded data
to a observable state, a fault is raised. This prevents cache side-channel attacks, timing
side-channel attacks, and transient execution attacks by ensuring that sensitive data does
not influence observable states or hardware optimization decisions.
2.8.3
BliMe model
The safe computation performed by the CPU in the BliMe protocol described in Sec-
tion 2.8.1 has been modeled using the F* language. The modeled single-CPU system
has been formally verified to prevent data leakage to observable destinations. However,
it has not been verified to prevent data leakage to other clients.
The model defines a multiblinded type representing blinded values. Using this type, a
non-blinded value is represented by its value, and a blinded value is represented by its
value and its blindedness domain. Instances of this multiblinded type are equivalent
if they are non-blinded and equal or if they are blinded and have the same blindedness
domain. Similarly, two lists are equivalent if they have the same length, and each element
at index i is equivalent to the element at the same index in the other list.
The system is modeled by a type representing the system state having fields including
the PC, a list of cache line addresses, a list of 64-bit blindable registers, and a memory
represented with a list of 64-bit blindable memory elements. Equivalence of two system
states is defined as the PC and the cache line address being equal and the registers and
memories being equivalent.
Execution of the instructions in the system is modeled using a deterministic step function
that represents a single fetch execute cycle. In this step function, the next instruction of the
system is loaded from memory based on the value of PC. If the instruction is blinded, the
CPU faults by changing the value of PC to zero, which is the address for a fault handler.
Otherwise, the system state will be updated based on an execution mapping.
The safety definition of the system states that the step function P is considered secure in
the model if it yields equivalent final states for all equivalent initial states.
Let S be the set of all possible system states. Then, the following property holds:
∀s1, s2 ∈S : s1 ≡s2 ⇒P(s1) ≡P(s2).
Based on the equivalency definition, this safety definition ensures that sensitive data from
clients, which are blinded, cannot result in any observable changes in the system. This
prevents both intentional and unintentional leakage of client data.

Problem statement
In this section, we outline the objectives of this thesis by first establishing the system
model, then detailing the expected requirements from the model, and finally defining the
adversary against which these requirements must hold.
The primary objective of this thesis is to develop a BliMe SoC model that guarantees the
confidentiality of client data.
Client data: Data imported into the system using the import function of the encryption
engine with encryption keys which are stored in the encryption engine by the HSM
through a handshake with the client.
Confidentiality: No entity other than the original client can exfiltrate or infer any
information about the client data stored on the server and tagged as blinded, with the
exception of its length.
3.1
System model
We model a BliMe system with multiple components, including untrusted and I/O peripher-
als, by extending the BliMe architecture and its security assurances to the entire system.
This model is developed using the F* programming language, with its confidentiality guar-
antees formally verified through F*’s deductive program verification infrastructure.
To overcome the limitations of the previous monolithic model [2], we decompose the sys-
tem into distinct components, each representing a peripheral that communicates through
a shared bus, similar to a real CPU.
A key aspect of the model is defining a security property that ensures the confidentiality of
client data for each individual peripheral. By proving that each peripheral independently
adheres to this security property, we demonstrate that each peripheral is safe, establishing
a foundation of trust for the entire system.
To model the system, we connect these safe peripherals and demonstrate that the entire
system satisfies the same security property. The modeled system, composed of safe pe-
ripherals, thereby extends the confidentiality guarantees across the entire multi-peripheral
environment. Formal verification using F* ensures that these confidentiality guarantees
are rigorously upheld, resulting in a secure system architecture.

Design
In this section, we present the design of a model for a generic BliMe system, incorporating
additional features such as I/O, untrusted peripherals, and accelerators. These features
can be applied to a wide variety of modern systems. Our focus is specifically on modeling
a SoC, as the components and challenges it encompasses are common to many other
systems. By doing so, we can generalize a SoC model to other systems, enhancing its
broad applicability and usefulness. The BliMe protocol described in Section 2.8.1 remains
unchanged in our system.
Real SoCs and systems found in cloud or mobile phone hardware have multiple CPUs,
each equipped with multi-layered caching that adheres to a caching policy ensuring cache
coherency. The main memory in these systems is logically separate from the CPU. They
also include one or more GPUs and accelerators to boost performance for specialized pro-
cessing tasks. Additionally, the systems incorporate multiple peripherals, to provide addi-
tional functionalities. Communication among these components primarily occurs through
a shared bus. In addition, communication between components happens via Interrupt
Request Lines (IRQs), Network-on-Chips (NoCs) [64], or other methods. In order for our
model to correspond to reality, we aim to model a system with these characteristics.
The previous model of BliMe [2] described a single CPU that had direct access to the
main memory, allowing the CPU to read from and modify the main memory directly, sim-
ilar to how it interacted with its internal registers. Figure 2 illustrates the design of the
modeled CPU, where it shows that registers, PC, cache lines, and memory are part of the
internal state of the CPU and the step function which mutates this internal state. Figure 3
demonstrates how the registers of the CPU get modified by the step function when an
addition instruction is being executed. Additionally, the model supported a Direct Memory
Access (DMA) operation for an accelerator, which was modeled as a special instruction
executed sequentially with the CPU. This configuration meant that the accelerator would
block the CPU’s operation until it completed its task. Furthermore, the model included
a generic accelerator and cache policy to demonstrate resistance to confidentiality vio-
lations regardless of the specific cache or accelerator architecture. However, it did not
provide any sample implementations for accelerator or cache policies to illustrate safe
designs for these components. Moreover, the original design did not support additional
peripherals. If peripheral support were to be added, it would have required the peripheral
operation to block the CPU until completion, similar to the accelerator.
4.1
Modeling hardware and information flow
To model the hardware’s static aspect, we represent the state of the hardware and all
the data it holds using a type. We define a state type for each component in the system
and a composite state consisting of these state types to represent the state of the entire
system. We also model the hardware’s dynamic nature, specifically how its behavior alters
its state, using a deterministic step function. This step function describes the operations
and transitions that occur, thereby mutating the state to reflect these changes. Together,
the step function and the state, model the hardware and the flow of information within it.
Repeated invocation of the step function represents the hardware’s continuous execution,
modeling its ongoing operation over time.
Let S be the hardware state and B the system bus. The type of the step function P for
the entire system is defined as: P : S →S. The type of the step function P for each
peripheral inside the system is defined as: P : B × S →B × S. The state types and the
step functions will be discussed more in Section 4.2.

To show that our design is safe, we must show that the system’s behavior does not vi-

information flow from the blinded domain to the outside and, additionally, that no blinded
data reaches an observable state in the system. This information flow policy is the same
as the BliMe taint-tracking policy described in Section 2.8.2. To prove this, we define a
leakage function that denotes which portion of the state is observable and introduce a
safety definition. If a system’s step function satisfies this safety definition, we ensure that
no client data will be exposed through the defined observable states of that system.
4.1.1
Leakage function
The leakage function models the adversary’s view of the system, detailing what an ad-
versary can infer from observable states that are directly accessible or indirectly through
side-channel attacks. Our approach follows the method of representing a complete exe-
cution as a series of steps, where each step has an associated leakage. The total leakage
of an execution is then the sum of the leakages of all individual steps.
We base our leakage model on the observable states of the system as described in Rac-
coon [65], Architectural Mimicry [66], BliMe [2], and OISA [67]. These studies define
observable states based on the side-channel attacks that infer information from cache
line addresses, the branch predictor, the PC, and the memory address bus. For instance,
memory addresses on the bus or the addresses accessed by the cache are observable
because information can be inferred by timing analysis or by measuring resource usage
and determining when each memory cell is used.
The leakage of the system is the sum of the leakage of all system components. Leakage
of CPU includes PC and cache line addresses. Additionally, logic related to branch pre-

part of the leakage function of the CPU. Our leakage model also considers all addresses
used for cache lines, memory, and on the bus as observable. Moreover, the model in-

our desired information flow policy as observable states.
4.1.2
Safety definition
In order to prove that the system does not violate the confidentiality of the client data, we
need to define a property for the system that, if this property holds, we can guarantee
there would be no violation of confidentiality. The safety definition is this property. If
the modeled functionality of the system and its components have this property, then their
behavior is safe, and it is guaranteed that they enforce our desired information flow policy,
which is to not allow any data from a blinded domain to go to a non-blinded domain.
The safety definition introduced in the previous model verified that the modeled CPU
based on BliMe architecture was protecting the confidentiality of the client data. This
safety definition stated that equivalent initial states of the CPU should yield equivalent
CPU states. Equivalency in this safety definition meant that clear values needed to be
equal while blinded values were required to have the same blindedness domain to be
considered equivalent. Figure 4 provides an example of confidentiality violation based on
this safety definition.

definition. However, to model a BliMe system and to verify its safety based on our security

Enhancing the safety definition to ensure lack of inter-client data leakage
We discovered a limitation of the previous model’s safety definition. Its safety definition
only prevented information flow from the blinded domain to the non-blinded domain and
did not account for inter-client data leakage. This is crucial for multi-tenant environments
where multiple clients’ data coexist on the same system. So, we modified the safety
definition of the CPU in the previous model to enhance it and use it as the safety definition
of the system in our model.
In the previous model, the safety definition stated that equivalent initial states should yield
equivalent states. Equivalency was defined as that if values are clear, they should be
equal, and if they are blinded, it is enough for them to have the same domain.
Let P be the step function of the system that transitions the system from one state to
another, and S be the set of all possible system states. Then, the safety of the system is
defined by the following property:
∀s1, s2 ∈S : s1≡s2 ⇒P(s1)≡P(s2).

state in the system. However, they can reach a different client and be observed by that
client. This safety definition does not ensure that information does not flow between dif-
ferent blinded domains, which violates confidentiality.
Because of this discovery, we change the definition of safety to include this additional
requirement. The model’s safety definition ensures that information does not flow from
the blinded domain to the non-blinded or clear domain. So, we can define the notion of
equivalency and safety for a specific domain and treat other domains as ’clear’ values from
that domain’s perspective. Using this approach, the blinded values will not be allowed to
flow from this domain to other domains, since the other domains are treated as clear
domain. To ensure that all domains are safe, we perform the same domain-wise safety
check for each domain. This will prevent any domain from allowing information to flow
from its domain to public observers and other domains.

To achieve such a goal, we need to modify the system’s definition of equivalency used in

cannot accept an equivalency definition over this untrusted peripheral as simple as the
previous model’s definition, and we need a more detailed definition that covers this use

is the internal state that is not observable under any circumstance, and the other part,
which is the leakage of the internal state, is the portion of the internal state that can possi-

be observed. Utilizing this leakage function, we can redefine the equivalency of two pe-

the internal states can remain equivalent according to the previous equivalency definition.
We define the leakage of the untrusted peripheral as its entire state, and use this leakage
function in the definition of equivalency.

4.2.1
Modeling communication between peripherals
To create a more realistic system model, we separate the states of the peripherals from
each other. This separation removes the need for any peripheral to have direct knowledge
and access to the state of another peripheral. To communicate and transfer data between
the peripherals they can interact only via a shared communication channel such as the
system bus.
We model the shared communication channel as a bus between system peripherals. The
bus is modeled as a list of pending transactions.
For the safety of the system, the addresses on the bus are not allowed to become blinded,
similar to addresses used in the memory, and cache lines. Furthermore, the equivalency
of the data on the bus is defined the same as the equivalency in the rest of the system.

the described step function to work with a bus rather than having direct access to the entire
system state. Similar to the previous description, the system’s step function receives the
system’s current state and returns the updated system state. However, the peripherals’

safe.

function satisfies the safety definition.

reading and writing of data happens atomically with their blindedness tags. Additionally, in
our model and its safety definition, we have assumed the integrity of the blindedness tags.
So, the implementation of the memory controller must ensure these tags are not accessi-

4.2.2.3
Accelerator peripheral
The previous BliMe model provided support for accelerators as a special instruction of the
CPU. In the previous model, each CPU and accelerator were coupled together, and even
if the model was extended to support multiple CPUs, each CPU could only use a single
specific accelerator because of this coupling.
Moreover, because the accelerator execution was part of a CPU instruction, the acceler-

the CPU until they were done. As described at the beginning of this section, this design
is not close to reality, and any implementation mirroring this design would be severely
limiting.
To solve this problem, we design an accelerator peripheral for the BliMe model. The
accelerator peripheral is a generic accelerator, and similarly, a Graphics Processing Unit
(GPU) could also be modeled as a similar peripheral. Then, we prove that the defined
step function of this accelerator satisfies the safety definition.
4.2.2.4
Untrusted peripherals
To make a BliMe system ready for real-world applications, it needs to be able to interact
with peripherals not designed specifically for a BliMe system. For instance, for the system
to connect to the internet through a Network Interface Controller (NIC) in its current state,
it requires a specifically designed NIC card that ensures it will not allow blinded data to
be sent to the internet. However, this is a very expensive requirement for the system to
be widely used. To solve this problem, we propose a peripheral firewall that enables the
interaction of any peripheral with the BliMe system without compromising the system’s
security.
The peripheral firewall could be a device between the untrusted peripheral and the system
or a device that monitors the untrusted peripheral communication on the bus. The firewall
residing between the bus and the untrusted peripheral makes the design more secure
because, in the other approach, we cannot ensure the firewall can remove the transaction
from the bus before the untrusted peripheral can start processing it.
The untrusted peripheral need not enforce any information flow policy, but instead, the
peripheral firewall will guarantee that the untrusted peripheral will follow the BliMe infor-
mation flow policy. Interposing the firewall between the bus, which is the source of blinded
data, and the untrusted peripheral results in a safe peripheral. The firewall can ensure
no blinded data will enter the untrusted peripheral, and because there is no blinded data
on the untrusted peripheral, no information can flow from a blinded domain to the out-
side of this domain. This firewall can result in the untrusted peripheral not achieving its
functionality when it is possible to break confidentiality, but instead, it will guarantee the
confidentiality property of the system.
The firewall can be designed using multiple approaches to ensure safety. One approach
is to redact any blinded data going to the untrusted peripheral. Another approach is to re-
spond to any request containing blinded data with an error and stop the blinded data from
reaching the untrusted peripheral. A real-world instance of an error response on a bus
from a slave device to a master device is present in the SiFive TileLink Specification [68],
which is a synchronous bus protocol specification.
Each approach has its own benefits and drawbacks. The primary drawback of redacting
sensitive data is that the untrusted peripheral will continue to operate on incorrect data,
leading to incorrect functionality. Additionally, this design lacks a mechanism to alert the
Formal Verification of Confidentiality Guarantees in a Blinded Memory SoC

system when an information flow policy violation occurs. The main advantage of redacting
data over returning an error on the bus is that it prevents the CPU from faulting, thereby
avoiding excessive interruptions in the server’s normal code execution.
Given these considerations, we have chosen the second approach for our design. In this
approach, the firewall is designed to return an error response on the bus when it detects
blinded data being sent to an untrusted peripheral.
The firewall will check whether any blinded data is on the communication channel to the
peripheral. If there is any blinded data, it will not let that data reach the peripheral and will
respond with an error on the communication channel; otherwise, the data will be sent to
the peripheral. This will result in the untrusted peripheral that does not conform to BliMe
architecture not receiving any client data. Therefore, any data leaked from this peripheral
will not break the confidentiality property of our systems.
The leakage function for this untrusted peripheral will be its entire internal state. This
peripheral is untrusted, and that means it does not guarantee that our desired information
flow policy, or, in case this peripheral is an I/O device, it could leak its entire internal
state to the public. We model this untrusted nature of the peripheral by defining the entire
internal state of the peripheral as observable in the leakage function. This means that we
need the entire internal states to be equal when using the notion of equivalency for two
untrusted peripherals and their associated peripheral firewalls.

5.1
Information flow policy
The system modeled in this work enforces an information flow policy that preserves the
confidentiality of client data. This information flow policy ensures that information from
data blinded in one domain never propagates outside of this domain. We ensure that
this information flow policy is being enforced in our implementation by verifying that the
implementation satisfies our safety definition described in Section 4.1.2.
There are different strategies to enforce this information flow policy. The primary strategy
that we utilize in this thesis is tracking the information flow using taint tracking. This taint
tracking is modeling a hardware-enforced taint tracking that associates a taint tag, which
is the blindedness domain of the client data, with the data originating from the client. Using
taint propagation, we track the flow of the tainted data in the system to ensure it never
becomes untainted and reaches an observable state. However, this taint tracking must
be enforced in the hardware, and if the peripheral is not enforcing it, we lose track of these
tainted data. To solve this problem we introduced the peripheral firewall in Section 4.2.2.4.
The peripheral firewall does not let tainted data pass through it. This is the second strategy
we have employed to ensure the BliMe information flow policy is enforced and our safety
definition is satisfied.
Hardware-enforced taint tracking can be implemented at various granularities and ab-
straction levels, as discussed in [69]. For instance, the granularity can vary from a single
bit to a machine word. Moreover, taint propagation rules can be defined at different ab-
straction levels, such as the gate level, the architectural level for each instruction, or an
algorithmic or even system level. The higher the abstraction level of the propagation rule
the more efficient it is. However, the lower the abstraction layer, the fewer propagation
rules are necessary. In this work, we have implemented taint tracking at a machine word
granularity and taint propagation rules at the instruction architecture abstraction level.
These taint propagation rules are a method of approximating information flow. They must
be sound to ensure that, at no point during execution, the system reaches a state where
there exists a non-tainted value originating from a client, which violates confidentiality.
However, due to the choice of granularity and abstraction level, these taint propaga-
tion rules may be over-approximating in some situations. This over-approximation re-
sults in over-tainting of the system’s data. However, our information flow policy allows
non-blinded data to become blinded because this does not result in a violation of confi-
dentiality. Therefore, over-tainting is acceptable and does not pose a security concern.
While designing the propagation rules for the system and its peripherals, we aim to min-
imize over-approximations. It is crucial to note that the enforcement of the information
flow policy is always ensured in this model, as the implementation must satisfy the safety
definition.

5.3.1
Safe peripherals
In order for a peripheral to be considered safe in our model, it must have a step function
that models its behavior and this step function must be proven that it enforces our desired
information flow policy.
To enforce the requirement of providing proof of safety and to provide a layer of abstrac-
tion between the implementation details of each peripheral and the system, we define

We show the definition of the typeclass for a generic safe peripheral in Listing 2. Type-

shared interfaces that define some behavior. Each instance of a peripheral needs to pro-

the step function, which is the model of the behavior of that peripheral; the equivalence
function which checks whether the states of two instances of that peripheral are equiv-

the blinded values in the peripheral state with zero. In our model, redaction is used to
prove safety by showing that a redacted system has the same observable effects as a
non-redacted system. This use of redaction to prove safety is equivalent to the safety
definition defined in Section 4.1.2 and is proven inside the model. To prove safety for
the whole system, each peripheral must prove six properties. Three of those properties
are reflexivity, symmetry, and transitivity; that is to prove that the equivalency function
provided is an actual equivalence relation. Two properties ensure that the redaction func-
tion, when given a peripheral as input, produces a peripheral as the output. and that two
equivalent peripherals will have equal redactions. The last and most important property
is to prove that two peripherals with initially equivalent states will yield equivalent states
after each step. This last property is the safety definition that we need to prove for each
peripheral.
5.3.2
CPU
The type representing the state of the CPU has multiple fields, including PC, cache line
addresses, and registers similar to the previous model described in Section 2.8.3. In
addition to those fields, it also includes registers indicating whether the CPU is waiting for
instruction, data, and which register is waiting for data. These additional fields are mainly
to model the microarchitectural behavior of the system when interacting with the bus.
The step function of the CPU is similar to the step function of the CPU in the previous
model, which we described in more detail in Section 2.8.3. The CPU step function uses a
function called instruction semantics. Instruction semantics is the ISA of the CPU in our
implementation. For the implementation of the ISA we used the initial BliMe model’s ISA.
Most of our implementation work of the CPU peripheral is related to the microarchitec-
tural details of the CPU, which includes the general functionality of the CPU except for
the specific details related to the instruction set of a CPU specified in the ISA. Our CPU
implementation simply executes instructions in the order they come and does not have
features such as out-of-order execution of the instruction, speculative execution, or a
pipelined design.
The CPU will first try to fetch the instruction it is supposed to execute from memory. In
order to do this, it will first issue a request to load the instruction from memory at the
address corresponding to the value in the PC register. Then it will wait until the instruction
becomes available on the bus and reads it. following that the CPU checks if this instruction
is blinded; if it is blinded, the CPU will fault; otherwise, it will continue execution. After this,
it will decode the instruction. Based on the decode instruction, it will check whether the

request; if it has already sent a request, it will simply wait until it is available on the bus
and read it. If the instruction does not depend on any data in memory, it will continue
execution. Next, the CPU fetches the necessary data from the registers and executes the

5.3.3
Memory
An essential peripheral in our system is the main memory of the system. The memory
peripheral implementation is simple. The memory is modeled as an array with blindable
values. The memory peripheral processes the requests in the address range specified
for it. The peripheral translates load and store requests it receives over the bus into
corresponding read and write operations over the memory array. Load requests result
in a response from the memory with the requested data, and store requests result in
updating the target address with the new value provided in the request.
5.3.4
Matrix multiplication accelerator
The initial BliMe model did not provide support for an accelerator. In [70], researchers
provide an implementation for a BliMe compliant accelerator that performs matrix mul-
tiplication and has a specific taint propagation rule specific to the matrix multiplication
algorithm. They also extend the initial BliMe model with support for a theoretical accel-
erator that would return some result based on inputs. This theoretical accelerator was a
generic accelerator that did not have any concrete implementation of an accelerator or
what form the taint propagation rules will take. We, therefore, implement a sample ac-
celerator for the BliMe system to provide an example of a generic taint propagation rule
and prove the safety of the system and the accelerator with a sample implementation.
Furthermore, the design and implementation of the sample accelerator can be used as a
guide and example of how to implement such an accelerator with the proper information
flow policy for the BliMe system on actual hardware.
A simple information flow policy design for the accelerator peripheral is that the accelerator
will make the output blinded if any part of the input is blinded. The output’s blindedness
domain will be the same as the input’s. An important step of this taint propagation rule
is that the accelerator must not generate results that are blinded in a domain different
from the blindedness domain of any part of the input. So, the information flow policy must
trigger a fault or return a default result if different parts of the input are blinded in different
domains.
The sample accelerator modeled in this thesis performs matrix multiplication. It consists of
two main components: a matrix extractor and a matrix computation function. We demon-
strate that any matrix extractor and any matrix computation function that preserves the
confidentiality of its inputs will result in an accelerator with the same security property.
In our implementation, the matrix extractor and computation function will return an empty
result if the accelerator receives invalid input data. The following are considered invalid
inputs:
• Invalid inputs that could cause errors in matrix multiplication, such as incorrect matrix
dimensions or other invalid parameters.
• Blinded input for the matrices’ dimensions.
• Input data containing blinded values from multiple domains.
After the calculations, the final result will have a non-blinded tag only if all the values in
the input are non-blinded. If any value was blinded with domain d, all the results will be
returned with a blinded tag of d.

5.3.5
Peripheral firewall
The peripheral firewall and the peripheral that it shields are also a safe peripheral in our
model.
To model and prove that a peripheral firewall placed in front of an untrusted peripheral
enforces our information flow policy and does not violate our safety definition, we first
model an untrusted peripheral. We did this by defining another typeclass for peripherals
with only a step function. As described in Section 4, the leakage of this untrusted periph-
eral is defined as its entire internal state. In our implementation the peripheral firewall
checks whether there is any data on the bus with the destination address of the untrusted
peripheral. When data with the destination of the untrusted peripheral is received, the
firewall checks whether the data is blinded, and if it is, it returns an error response on the
bus that results in the CPU faulting and the data on the bus not reaching the untrusted
peripheral. However, if the received data is not blinded, the firewall will allow the data to
reach the peripheral, and enable the untrusted peripheral to operate as it normally would.
The implementation of this functionality in the peripheral firewall can be seen in Listing 3.

6.1
Security requirements analysis
We will first perform a security analysis of the model and the safety definition and show

Next, We provide some examples of vulnerable peripherals that our model will not verify

do not compromise the original security requirements of BliMe and how we have improved

6.1.1
Security analysis
A computation on a BliMe system as described in Section 2.8.1 has four stages. The key
agreement, data import, safe computation, and data export. If the key agreement protocol
and the transmission of data in the secure channel established with this key are secure,
then the attacker will not gain information about the session key, client data, or compu-
tation result. To show that the modeled BliMe system satisfies its security requirement,
we only need to show that the safe computation part of the protocol does not violate the
confidentiality of the client data. Based on this assumption the client data available for
the safe computation on the system is blinded with the unique blindedness domain of the
client.
In this thesis, we define the observable portions of the system’s state. By utilizing our
safety definition described in Section 4.1.2, we demonstrate that these observables will
only be influenced by other observables.
When verifying our model, we effectively show that the current system’s design and im-
plementation, as detailed in the model, satisfy our safety definition based on the provided
observable leakage function. This verification confirms that:
• The information flow policy ensures that information from blinded data in a certain
domain will never flow outside of that domain.
• Blinded data on the server, will never reach an observable state.
In the design chapter, we have identified and detailed which portions of the system state
and each peripheral are considered observable. We have provided arguments supported
by research in Section 4.1.1 to show that these are all the possible locations that can be
observed, either directly or indirectly.
These points demonstrate that information from the client’s data, which is blinded within
the system, will never flow outside of the client’s blindedness domain. Furthermore, these
blinded data will never reach an observable state. This ensures that client data will never
be leaked, thereby preserving confidentiality.
By comparing our results with the security requirements defined in Section 3.2.1, we find
that the model satisfies these security requirements by meeting the safety definition.

6.1.3
Side-channel resistance
Side-channel attacks exploit indirect information leakage from implementation-specific
vulnerabilities to infer sensitive data from a system. A BliMe system must protect against
timing and cache side-channel attacks that do not require physical access.
Since our model does not include a CPU with out-of-order speculative execution currently,
we will not discuss transient execution attacks.
36
Formal Verification of Confidentiality Guarantees in a Blinded Memory SoC
==PAGE:49==
We prevent cache side-channel attacks by not allowing blinded data to be used as the
address for retrieving data from memory. The previous model did not allow the address
used for copying a cache line from memory to cache to be blinded. In our model, we also
do not allow the address used on the bus to be blinded. As a result, the timing differences
in data retrieval from memory, whether the data is cached or not, do not leak any sensitive
information, as the addresses used are not related to the client’s data.
Researchers have demonstrated that a system ensuring confidentiality against an adver-
sary, based on the program counter transcript model [71], where the value of the PC is
observable to the adversary, is protected against timing attacks stemming from the pro-
gram’s control flow behavior [71]. These timing attacks exploit differences in execution
time across various execution paths. In our model, we consider the PC as part of the CPU
leakage, making our adversary stronger than the one described in the program counter
transcript model [71]. By verifying our model against this stronger adversary, we demon-
strate that our model is secure against such timing side-channel attacks. Since the PC
cannot be blinded and is part of the CPU leakage, a correct taint propagation rule will
prevent a blinded condition from affecting the PC.
Since our adversary is stronger than the adversary described in [71], our model, there-
fore, also ensures resistance against some forms of power analysis side-channel attacks.
More specifically, our model protects against power analysis side-channel attacks that
infer information based on the difference between the types of instruction. However, our
model does not protect against power analysis attacks that infer information based on the
difference between the values used for an instruction.

The current model supports two new types of concurrency that were not present in the

The second type is the fact that in the previous model, all peripherals other than the
CPUs were modeled as an instruction of the CPU that would block the CPU execution
until they were done. Because the current model allows concurrency of multiple CPUs
and concurrent execution of all peripherals with the CPU, proving the safety property of
the model also demonstrates that any data race bugs resulting from this concurrency will
not violate the confidentiality of the client’s data. An important safety requirement in a
concurrent design is that all data reads and writes and the blinded tag associated with

6.2.2
Extensibility
With the new model’s architecture, the system’s safety is proven separately from its com-
ponents and peripherals. The proof does not directly depend on the implementation de-
tails of any peripheral, even main components such as CPU and main memory. To add
or remove a peripheral from the system, one must implement the interface for a safe pe-

the single line that imports them in the system model to remove a specific peripheral. This
provides a complete decoupling between peripherals and the system as a whole. This
follows the general principle of extendability, which states that to add functionality, one
should only need to add the new functionality to a new file, and it should be possible to
add the feature without modifying any previous codes.
Another important contribution to the engineering requirement of the model is that now we
provide a concrete interface and specific lemma that needs to be proven for a peripheral
for that peripheral to be safe. The previous model did not provide what must be proven
to keep the system safe.

Our focus on this project has mostly been on proving that different features and compo-

• Testing the fetching of new instructions.
• Testing the faulting of the CPU over:
– Not available instruction.
– Blinded instruction.
• Testing the correct execution of an instruction:
– An instruction that is dependent on data from memory.
– An instruction that is not dependent on data from memory.
– An instruction that produces memory requests.
By demonstrating that the CPU’s microarchitectural behavior is correct and that the ISA
is still correct as the previous model, we can argue that the CPU’s functionality is correct.

that we have improved this requirement of the BliMe model; however, this is still far from
a functional correctness guarantee.

7
Discussion
This chapter highlights the issues we encountered and parts of the design that might be
improved. Finally, we will provide a long-term future direction for this research and where
it could lead to benefit the field of system security and formal verification.
7.1
Challenges
This section describes some of the design and implementation challenges we faced. Each
challenge could be explored further, and a more elegant or complete solution could be
offered in the future.
7.1.1
Design challenges
7.1.1.1
Other communication models
In our current model, all communication is handled using a single system bus for interac-
tions between peripherals. However, in a more realistic system, various communication
methods might coexist. For instance, some communication might occur over a bus, while
direct communication and signaling between peripherals could utilize IRQs or other meth-
ods.
A peripheral might not want to wait for a transaction, or it might not be possible to trigger
the behavior with a transaction. For example, a peripheral might need to inform the CPU
that it has data ready to be read. In such cases, the peripheral might use out-of-band
signals on dedicated interrupt lines, such as in IRQs, unlike Message Signaled Interrupts
(MSI) [72] that use an in-band bus for interrupt signaling. Additionally, some transactions
might not occur on the main system bus with typical bus protocols. Instead, communica-
tion might happen directly using an on-chip communication bus protocol like the Advanced
eXtensible Interface (AXI) [73] between the CPU and a tightly coupled accelerator. Other
non-bus-based communication methods, such as NoCs [64], may also be used. A NoC
is an on-chip communication network that utilizes packet switching.
These types of communication are common in modern systems, and our model must
accommodate them. Each communication method can be modeled differently and sepa-
rately to more accurately represent the details of the communication protocol. This diverse
approach allows different peripherals to communicate using the most efficient method for
their specific requirements.
7.1.1.2
Multi-bus Model
Our current model employs a single communication channel for all interactions between
peripherals. In a more realistic system, multiple buses might exist. For instance, a sys-
tem could have two separate buses, peipherals with higher speed would reside on a bus
shared with the CPU while other peripherals share another bus. Additionally, some pe-
ripherals may be restricted to accessing only their designated bus, preventing them from
communicating across different buses.
7.1.2
Implementation challenges
Because we are modeling realistic hardware, we use bounded integers rather than un-
bounded integers. Each word of data in memory or on a bus is limited to a certain number
of bits, which imposes a fixed range on the values they can represent. To accurately
reflect this limitation, we model these data words as bounded integers. This approach
ensures that our model aligns with the actual constraints of hardware, where each data
word has a specific bit-width and thus a defined maximum and minimum value. This will
cause some of the properties that are expected to be inferred automatically by F* not to
happen automatically, and we will have to prove some basic mathematical properties,

This is not a major problem, but in the case of other complex operations, it might become

7.2
Limitations
This section presents some of the limitations of our model, which could be future exten-
sions.
7.2.1
Interposers
In our model, we assume the bus and the communication channel between the bus and
the peripherals are secure, with no interposers actively sniffing the data passing through.
An interposer, which is a device that stands between two components, such as a physi-
cal device that goes into the memory socket between the motherboard and the memory
module, can compromise the confidentiality of the system if confidential client data travels
on the bus.
This is a problem because by installing interposers between the safe and trusted periph-
erals and the bus or because different manufacturers are providing components of the
system that have access to the bus and their peripherals could be sniffing the confidential
data on the bus.
In this work, we present a peripheral firewall that can be used in instances where we
know a peripheral is untrusted; however, future work should explore how this problem
can be better mitigated in more general cases against interposers installed later and in
the presence of hardware trojans in trusted peripherals.
7.2.2
Chiplet interconnects
A modern trend in hardware chip manufacturing is the attraction of chiplet-based man-
ufacturing over monolithic SoCs [74]. SoC designs often incorporate Intellectual Prop-
erty (IP) blocks from different designers, similar to chiplet-based systems. Unlike SoCs,
chiplet-based systems feature IP blocks manufactured in various locations, requiring in-
terconnections for integration [75].
This scenario can lead to the presence of interconnects, which may be either trusted or
untrusted, and are not accounted for in our model. Since, all the data entering and exiting
the chiplet will pass through these interconnects, the assumption that interconnects in
chiplet-based systems are trusted poses a limitation, potentially compromising system
confidentiality upon physical implementation.
Future work should explore presence of interconnects as parts of the system and develop
solutions to ensure system confidentiality in the presence of malicious interconnects, or
establish security guarantees based on trusted interconnect designs.
7.2.3
Speculative out of order execution
In our design, we have modeled a CPU with in-order instruction execution. Better would
be to make the design agnostic to this choice by some abstraction or to implement a CPU
with speculative out-of-order execution. This is an important issue because most CPUs
today employ speculative out-of-order execution of instructions, which is the cause of
some microarchitectural side-channel vulnerabilities that target the speculative execution
of the CPU.
To prove the safety of an out-of-order speculating CPU against transient execution at-
tacks, we first need to extend the model to include an out-of-order speculative CPU with

all the logic and registers related to its queued instructions and branch prediction logic.
Then, we must extend the taint propagation rules to cover these logics and prove that
these CPUs also enforce our information flow policy that prediction logic does not depend
on blinded data and that if a fault happens in speculation, it will not have any feedback to
the prediction logic. This ensures that any timing difference observed due to speculation
will not leak information about the client data.
7.2.4
Pipelined CPUs
Similar to CPU’s execution order of instructions, modern CPUs employ a pipelined design
that could expose additional vulnerabilities and attack surfaces to adversaries. Extending
the model to prove the security requirement of BliMe holds in such CPUs is necessary.
7.2.5
Encryption engine
One essential component of the BliMe architecture is its encryption engine, which han-
dles the atomic decryption and blinding of the imported client data, and encryption and
unblinding of the computation result. Currently, the model only validates the computa-
tion’s safety after the data is processed by the encryption engine and before it is received
by the encryption engine and sent back to the client. However, to ensure the safety of the
full system, we need to model the encryption engine functionality before and after the safe
computation. This way, the full path of data, from importing encrypted data, to generating
the unique blinded tag, and then returning the encrypted result, can be proven to be safe
with a more complete safety definition.
7.2.6
Parallelisation
The implementation of the system’s step function in the model allows for concurrency
but does not offer parallelism. To achieve parallelism, two main functionalities need to
be addressed. First, the call to each peripheral’s step function needs to be modeled in
a parallel fashion. Second, the bus needs to become shared between peripherals to
allow only one transaction on the bus at a time, requiring the peripherals to perform some
method of arbitration.
7.2.7
Making the model executable
F* allows extraction of the executable portions of its code into OCaml. This extracted
OCaml code can be complied into an executable, and the model’s accuracy can be tested,
and simulations can be run using it. For the extraction to work, further software engi-
neering work is needed. The model, excluding the sample ISA, has been tested to be
extractable.
7.2.8
Mitigating side-channel and related attacks
As explained in Section 6.1.3, our model protects against some power analysis side-
channel attacks. However, providing complete protection against all power analysis, dif-
ferential fault, and electromagnetic attacks requires additional defense mechanisms.
Our model protects against timing side-channel attacks based on control flow. However,
there can be measurable timing differences when executing mathematical instructions
with different values. For instance, modern CPUs optimize integer division when the divi-
sor or the dividend is small, resulting in different execution times according to the division
value. The sample ISA in our model does not include complex operations, ensuring it
is safe against these types of vulnerabilities. To prove that our model is secure against
these types of timing vulnerabilities, we must include in the ISA a notion of time, such as
cycles. For instance, each instruction should either return how many cycles it took to exe-
cute or enforce that each step of the step function describing the ISA progresses the state
by only a single cycle. This approach ensures precise tracking of execution time, guar-
anteeing security against timing attacks. The addition of either detail will stop the model
Formal Verification of Confidentiality Guarantees in a Blinded Memory SoC
45
==PAGE:58==
from verifying an unsafe implementation of ISA. One approach for the information flow
policy to protect against such optimizations is by enforcing that the CPU takes the longest
possible cycle time for an operation if the optimization is based on a blinded value. To
ensure mitigation against these timing side-channel attacks, no hardware optimizations
should be performed based on blinded values within the CPU microarchitecture.
Attacks similar to or based on Rowhammer [55], such as RAMBleed [56], which target
confidentiality directly, are not protected against in our model. These types of attacks re-
quire different defense mechanisms. Therefore, information flow tracking techniques are
not helpful in defending against such attacks, and the system needs to employ separate
and additional mechanics to protect against these vulnerabilities.
Our system is currently vulnerable to attacks that exploit memory’s remanence decay
phenomenon [76]. This occurs when data in memory does not immediately disappear
when its power is cut but gradually fades away. An example of such an attack is the
cold boot attack [52], where an attacker can retrieve encryption keys from memory if the
system is abruptly shut down and the attacker gains access to the memory. One approach
to secure against this type of attack is to store the encryption key in a CPU register or
a separate Trusted Platform Module (TPM). Given that our system already includes a
dedicated HSM that allocates a unique key and domain for each client, we can utilize this
existing infrastructure to implement a memory encryption scheme. This scheme would
securely hold the encryption key within the HSM.
7.3
Future directions
We propose two long-term future directions that this research can take that the benefits
are not limited to the design of BliMe architecture.
7.3.1
Other hardware-based information flow research
Our model employs blinded tags to ensure data confidentiality. By redefining these blinded
tags, the model can be adapted to enforce other information flow policies, such as those
targeting data integrity. The Biba [77] and Bell-LaPadula [78] models are prominent ex-
amples of policies focused on integrity and confidentiality, respectively.
In addition to the 8-bit tags used in our model, which serve the domain of blindedness,
we also introduce a simpler 1-bit tag. This tag indicates whether data is blinded, thereby
it can be used as means for distinguishing between sensitive and non-sensitive data in
confidentiality-based policies or between high-integrity and low-integrity data in integrity-
based policies.
Utilizing this single-bit blindedness mechanism, a CPU requires only two additional in-
structions—blinding and unblinding—to support a complete implementation of the Bell-
LaPadula model. In this context, the blinding operation would correspond to tagging data
as sensitive or non-sensitive, enabling the enforcement of the Bell-LaPadula model within
the system.
Similarly, these tags can be applied to enforce the Biba model for data integrity. To do
so, we can adjust the leakage model to focus solely on directly observable outputs, such
as I/O operations, or maintain the original model to ensure that addresses and the PC
are influenced only by high-integrity data. For our safety definition, we change the inter-
pretation of the tags so that blinded data represent low-integrity data, while non-blinded
data represent high-integrity data. This approach ensures that low-integrity data cannot
overwrite high-integrity data and that low-integrity data are never displayed to clients in
our model. This perspective aligns with a system view of a high-integrity process or client.

The final aspect to address is the mechanism for setting the integrity tag for data. Using
our model, we can demonstrate that the CPU and system effectively enforce the spec-
ified policy. Depending on the enforcement target, whether it is client data integrity or
memory protection against vulnerabilities, the mechanism for setting the integrity or sen-
sitivity bit can be adapted. For instance, in HDFI [79], various security applications of
memory tagging are proposed. One application involves adding special instructions to
the ISA to perform a high-integrity store operation, contrasting with the normal store in-
struction which performs the operation with low-integrity. With this mechanism, hardware
can provide memory protection against attacks such as buffer overflow.
Furthermore, others have proposed systems based on tagged memory or, more specifi-
cally, hardware-enforced taint propagation-based designs to improve system integrity and
confidentiality. However, these studies lack formal verification of their proposed design.
This model, with some extensions or modifications, can be used to provide formal proof
for those studies. For instance, researchers in [67] provide an oblivious ISA with microar-
chitecture design for CPU that protects the CPU from side-channel attacks. They provide
a security analysis of a simplified abstract version of their design to show that it is safe
against side-channel attacks. However, by providing machine-checked proofs of their se-
curity in addition to the security analysis, the confidence in their solution increases. This
approach would have allowed them to offer proof on a concrete and complete implemen-
tation of their design. Additionally, it would have enabled them and other researchers to
extend their design and model to a complete system, not just a CPU.
To generalize our model for verifying hardware design for enforcement of any hardware-
based information flow policy, we propose an abstraction over the definitions. This ab-
straction involves various levels of tags ranging from high to low, with the possibility of
having different tags within the same levels. Additionally, we define a list of states that
only accept low-level tags, such as observable states in our model, and a list of states that
only accept high-level tags, such as high-integrity requiring states. By incorporating this
type of abstraction, our model becomes flexible enough to verify different security policies
and various designs with different goals, whether ensuring system-wide properties or fo-
cusing on specific aspects such as speculative portions of the CPU. This flexibility allows
the model to adapt to various security requirements and design objectives. Furthermore,
this approach ensures that our model, without requiring additional modifications, remains
usable for other researchers working in this field. Building on this common model, we can
provide additional features such as performance tests and hardware synthesizable code
extraction for different designs. This extensibility would make our model a valuable tool
for a wide range of applications in hardware security research and development.
7.3.2
Extracting a synthesizable hardware design
The ultimate goal of this research is to extract a synthesizable hardware design from the
formally verified model. This involves translating the verified F* model into a Hardware
Description Language (HDL) like Verilog or VHDL. This enables the implementation of
hardware that inherits the verified security properties, bridging the gap between theoretical
models and practical implementations. Some work has been done in this area but does
not use automated verification tools such as F*. The project ClaSH [80], a subset of
Haskell, can generate Register Transfer Level (RTL) source code. On top of the ClaSH
project, there has been research [81] done to extract ClaSH code from Coq [21], which is
a proof assistant. Another example of an attempt to compile from proof assistant code to
synthesizable Verilog code is the project Kôika [82]. They have achieved this by defining
an embedded Domain-Specific Language (DSL) inside Coq. The hardware that must
be designed would be written in this DSL inside the Coq. Similarly to the work done to

Related work
8.1
Dynamic information flow tracking in hardware
Proposals such as BliMe [2], OISA [67] and Dolma [84] use hardware-based dynamic
information flow tracking techniques to ensure that the CPU is protected against side-
channel attacks, or in case of BliMe [2], against any type of confidentiality violation. How-
ever, these efforts are primarily focused on CPUs. The authors of [70] extend this idea
of information flow tracking to accelerators. Despite this, utilizing information flow track-
ing techniques in GPUs and other system peripherals remains unexplored. We explore
the idea of dynamic information flow tracking to peripherals and how to ensure confiden-
tiality guarantees in the presence of peripherals that do not enforce any information flow
tracking or policy. This approach allows us to extend information flow tracking and policy
enforcement across the entire system.
8.2
Formal verification
8.2.1
Security properties
There are many instances of formal verification of security properties in literature. Far-
rell et al. [85] verify security properties, including spoofing, tampering, and information
disclosure, for an algorithm used in vehicle-to-vehicle communication. They utilize a ver-
ifier called Dafny [28] for this purpose. Schoolderman et al. [86] provide an approach to
verify assembly code by demonstrating the verification of a hand-optimized version of ma-
trix multiplication. They use the deductive program verification tool Why3 [29], which uses
SMT solvers for deductive verification. In addition, they model the instruction set of an
AVR microcontroller to verify the mentioned matrix multiplication on the microcontroller.
Security properties are formally verified by [87] to help generate secure code for a kernel
I/O subsystem, performing this verification using Dafny [28].
In contrast to the previously-mentioned works that verify security properties for a specific
system or program, some research provides an infrastructure for implementing application-
specific libraries that are easily verifiable for functional correctness and the presence of
security properties. The work described in [88], which is based on the Jasmin [89] lan-
guage and compiler, is an example of this type of research. It provides an infrastructure
for developing formally verifiable, efficient cryptographic software. This framework of-
fers automated tools for proving memory safety, termination, and constant-time security,
while also enabling the developers to optimize their code without compromising any formal
guarantees through equivalence checking against the formally verified version of their im-
plementation. Their work’s main limitation compared to ours is its lack of generalizability to
other programs. Their language and infrastructure specifically target the implementation
of only cryptographic libraries, aiming to provide an infrastructure that can be automated
to some extent. Another limitation of software-only work is that even if they prove that
the properties are propagated to machine code, hardware optimization and conversion
from machine code to microcode in lower layers could still break the security properties
these researchers ensure. A broader issue with this approach of software verification and
providing software verification infrastructure arises in outsourced computing scenarios.
In such cases, it becomes challenging to provide a mechanism for clients to verify that
the formally verified software is being used without exposing the service provider’s code.
However, this is not an issue in BliMe due to the presence of the HSM, which enables
remote attestation of the hardware.

8.2.1.1
Side-channel resistance
Cabodi et al. [90] provide a methodology to verify the absence of vulnerabilities related to
speculative execution in modern pipelined out-of-order CPUs using model checking. The
motivation for this work stems from the fact that many side-channel vulnerabilities origi-
nate from issues in speculative execution design. Numerous studies use model checking
to verify side-channel resistance in hardware design because it is highly automated and
requires minimal expert intervention, unlike methods such as deductive verification. How-
ever, the model-checking process encounters a significant challenge known as state ex-
plosion [91], which occurs due to the exponential growth of the state space as the number
of state variables increases. This challenge compels researchers to use model checking
to represent the target system more abstractly, potentially causing certain types of vul-
nerabilities to be overlooked. BliMe [2] provides an instance of deductive verification of a
side-channel resistant CPU using deductive verification.
Other research focuses on ensuring that the code is side-channel resistant rather than
proving the underlying system is side-channel resistant. An example of such a project is
HACL* [92]. This project implements modern cryptographic algorithms that are verified to
be side-channel resistant using deductive verification tools. Constantine [93] is another
approach that, instead of proving that code is side-channel resistant, provides a compiler
extension to automatically transform program code into side-channel resistant versions
of themselves by applying control and data-flow linearization techniques. Control and
data flow linearization means that all possible secret-dependent control flows and memory
accesses are executed no matter the actual value of the secret so that the execution is
not dependent on secret data. However, the problem with this approach is that it is not
sound because its analysis is based on dynamic taint propagation and only considers
the observed execution paths [93, 94]. Because of this, researchers proposed BliMe
Linter [94] that can identify the remaining problematic codes. As mentioned earlier, a key
limitation of projects that ensure side-channel resistant code is the difficulty in verifying
this property for remote clients when the code must remain confidential. Therefore, this
type of research is not an alternative to our work but rather complementary, as it can aid
in the development and transformation of programs into BliMe-compliant programs.
8.2.1.2
Security properties using F*
F* is also utilized to verify large security projects formally. HACL* [92] is a prominent
example of these projects. This project implements modern cryptography algorithms in
F* and then compiles the F* code into C code. This project provides an open-source
verified cryptographic library used in major products such as Mozilla Firefox, the Linux
kernel, and Python. In addition to HACL*, other large projects have benefited from formal
verification using F*. DICE* [95] is an example of verification of protocols related to and
close to the hardware. DICE* is an implementation verified by F* of the Device Identifier
Composition Engine (DICE), a measured boot protocol. A final example of verification of
important security blocks is verifying the record layer of Internet Engineering Task Force
(IETF) standardization of Quick UDP Internet Connections (QUIC) [96].
Much work has also been done to verify protocols. Most of this work has typically been
done using model-checking approaches. However, some major efforts have also been
made in deductive program verification of protocols. DY* is a F* framework to verify the se-
curity properties of the protocol by simulating a Dolev-Yao adversary [97]. To demonstrate
the effectiveness of their framework, they formally verify the forward and post-compromise
security of the Signal protocol. They model the protocol for formal verification with no lim-
itation on the number of protocol rounds, which had not been achieved before this work
and is not possible to do using model checking due to state explosion.

An interesting work on verifying protocols using F* is [98], where authors aim to prove a

8.2.2
Hardware design
We identified three types of approaches that aim to provide deductive verification of prop-
erties for hardware designs in previous research. The first approach is modeling the
hardware using established verification infrastructure, proving the desired properties, and
then, in a few instances, compiling the model to synthesizable HDL. The second identi-
fied approach is to extend established HDLs with deductive verification capabilities. The
third and last approach is to design or use higher-level languages for hardware design
and utilize program verification techniques that are either readily available or easier to
implement in these languages.
The work of researchers in [99] is a unique example of attempting to prove and verify
properties of a realistic system with a multi-core CPU and main memory using software
verification techniques similar to our work. They provide a very complete and generaliz-
able model of a system in the formal language of the proof assistant Coq. This aligns with
the first approach, where hardware is modeled using an established verification infras-
tructure—in this case, Coq—and the desired properties are proven within this framework.
However, they do not extend their work to the extraction of a hardware-synthesizable
code. However, their aim differs from our own in that they try to prove the sequential con-
sistency of the system, which is not a security property, and they implement their model
in Coq, which does not provide verification automation unlike F*. Due to choosing Coq for
their work, their work is not as easily extensible as ours because it requires more man-
ual work for proofs. Another example of this approach to hardware design verification is
the domain-specific language Sail [100], designed specifically for the specification of ISA.
It provides a common specification language for different ISAs and can generate docu-
mentation, executables for testing, and theorem prover code such as Coq [21] to verify
security properties. However, it is limited to modeling and verifying the ISA and is not
designed to extend to other hardware designs.
There has been some work verifying a hardware design’s information flow characteris-
tics. In [101], researchers introduce a hardware design language called SecVerilog that
ensures the design has secure information flow by performing static information flow anal-
ysis over the design. This example represents the second approach, where the authors
aim to enrich existing HDLs with deductive verification capabilities. In [102], authors ex-
tend the SecVerilog language to offer more precise control over the information flow in the
design to reduce the effects of over-tainting to make more efficient designs verifiable in
SecVerilog. However, SecVerilog is not flexible enough to allow innovative designs with
other security requirements such as BliMe. This is an example of the second approach in
hardware deductive verification where the authors aim to enrich the HDL languages with
deductive verification capabilities.
Various attempts have been made to raise the abstraction level of modeling and hardware
implementation. Bluespec [103] is a high-level HDL that offers a rich type system. Another
example is Chisel [104], which embeds the language in Scala, enabling object-oriented
and functional programming concepts for HDLs. These examples illustrate the third ap-
proach, where higher-level languages are used for hardware design, and program verifi-
cation techniques from the software domain are applied. Both Bluespec and Chisel rep-

Conclusion
In this thesis, we addressed the critical issue of ensuring data confidentiality in systems,
by modeling a BliMe SoC and formally verifying its safety. We modeled the system with
implementation details that closely align with modern hardware, including multiple CPUs,
accelerators, and various peripherals communicating over a bus.
A key innovation in this thesis is the introduction of a peripheral firewall. This firewall en-
ables the integration of untrusted peripherals into the BliMe system without compromising
its security. This design broadens the applicability of the BliMe architecture by enabling
the safe use of I/O and other peripherals not designed explicitly for the BliMe architec-
ture. Additionally, we model a matrix multiplication accelerator to demonstrate a practical
implementation of the BliMe architecture for an accelerator. This example shows that the
BliMe system can handle real-world computational tasks while maintaining its confiden-
tiality guarantee.
By leveraging the F* programming language, we apply formal verification methods to en-
sure that our architecture meets its security goal, which is to maintain data confidentiality
even in the presence of adversaries in the system.
Our evaluations include rigorous formal proofs and experiments, demonstrating that the
modeled BliMe system effectively prevents side-channel attacks and ensures data con-
fidentiality while enhancing the extensibility and performance compared to the previous
BliMe model.
However, the CPU modeled in our system does not account for out-of-order speculative
execution or pipelining, common in modern CPUs, which future work should address to
enhance the model’s realism. Additionally, our current focus is primarily on the computa-
tional aspects of the system without integrating the encryption engine, which is crucial for
a complete BliMe system.
The findings of this thesis suggest several future research directions. One avenue is
the design and verification of hardware properties using a deductive verification tool, and
automating the pipeline to compile from Formal Specification Level (FSL) to RTL while
preserving the proven properties in the FSL. Another future direction could be extending
our model to other information flow policies and hardware designs, which could contribute
significantly to the broader field of hardware security.
In conclusion, this thesis extends the BliMe architecture and provides a BliMe SoC model
with formally verified confidentiality guarantees. This work advances the state of the art
in hardware security and offers practical directions for future research and development
in this critical area.

In today’s world, outsourced computation has become a common practice, raising critical concerns
about data privacy and security. This thesis addresses the challenge of ensuring confidentiality in
multi-peripheral systems by extending the Blinded Memory (BliMe) architecture and applying formal
verification techniques. The BliMe architecture is designed to enhance the security of outsourced
computations by implementing hardware-enforced taint tracking and ensuring that sensitive data re-
mains confidential even in the presence of potentially compromised components. This work extends
the BliMe model to support systems with multiple CPUs, accelerators, and various peripherals, each
communicating through a shared bus. The extended model introduces a peripheral wrapper that
enables the integration of unsafe peripherals while preserving the system’s confidentiality guaran-
tees. By using the F* programming language we formally verify that the extended BliMe architecture
maintains confidentiality across all system components. The contributions of this thesis enhance the
practicality and security of the BliMe architecture, making it a viable solution for secure outsourced
computation in complex multi-peripheral environments. The findings provide a foundation for future
research in hardware security and formal verification, with applications in cloud computing.
