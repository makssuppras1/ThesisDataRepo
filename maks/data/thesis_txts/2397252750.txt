Abstract
This thesis looks into the interior design world from a virtual reality perspective.
It investigates the core functions of current 3D interior design software. These
functions will provide the knowledge base for the intent of this thesis to imple-
ment a virtual reality interior design application. The functions of a 3D interior
design software will be used as representation for interactions in the virtual re-
ality environment. These interactions creates technical challenges that will be
analyzed. Technical solutions will be found and implemented in a virtual reality
interior design application prototype. This prototype will be developed in the
Unity game engine. The HTC Vive headset will be used to create the virtual re-
ality environment. The prototype and its functions will be tested and described
thoroughly. Finally this thesis will conclude that it is possible to create virtual
reality interior design applications by using the core functions of a 3D interior
design software.

1.1
Motivation
The technology of virtual reality (VR) is steadily growing and gaining traction
into a whole array of different ﬁelds and interest areas. One of those areas is in
interior design. When designing interior today, 3D software tools is used. There
exists many interior design tools, where one of them is IKEA’s Home Planner,
where the customer can design their new home themselves. The problem with
this is that you can view the design in 3D, but only see it on a 2D monitor. This
degrades the experience signiﬁcantly, because it is difﬁcult to imagine how the
room will look like when realized. This is where VR can make a difference. With
VR it is possible for customers to see their future room and get the feeling of how
the room will look like.
This particular area of VR is new and still in development. The industries in the
interior business has just recently started to use VR as their new designing tool,
which will essentially replace [1][2] the old 3D software tools with VR designing
tools instead.
1
==PAGE:11==
It gives me motivation knowing that the ﬁeld, I am working in, is still in the re-
search, innovative and development phase. Knowing that my work might ﬁnd
some solutions, which has not been found yet, gives great motivation to make
this thesis.
1.2
Project deﬁnition
This thesis is a project research about how virtual reality can be used to create in-
terior design applications. Many VR interior design applications only supports a
static world [1], where the user of the application only can experience an already
ﬁnished design of a house. This means there is no interaction in the application.
This project aims to develop an VR interior design application, where the pur-
pose of it, is to make the customer design their future room themselves, like
IKEA Home Planner [3].
The area for my main focus will be the interaction inside the VR application.
Some of the questions, I will answer, are; how to interact with objects, both from
a close and a long distance; how to rotate and move objects so it makes sense;
how to deal with collision of objects. Another aspect is object alignment, or ob-
ject snapping, where an object will align to e.g. a wall, and only follow the plane
of the wall. This is just some of the technical challenges I will be looking at and
trying to solve.
Furthermore I will look into, how to place interior in a room and I will ﬁnd out
if it is usable to design interior with VR. Some interior design companies design
the room with a 3D software tool, where they import the design into VR [1]. This
way the design will be shown in VR, but it is static. I want to extend this, so the
customer can design the interior in the VR room themselves.

2.1
Plan-driven and agile development
There exists many different strategies and methods how to develop software.
They all have their pros and cons. The important thing is which one I chose to
use.
If we look at the risks that may appear during the project, the most important re-
source I have is time. If I use my time recklessly the whole project can fall apart,
but on the other hand, if I use my time wisely I can put myself in a preferable
situation.
Plan-driven development is all about dividing the whole project into tasks and
then sub-tasks. It is important to plan the whole project in details. A project
plan is made and this contains all the work that needs to be done. This way
there is a clear overview over the whole process. This method is mostly used on
large software products, because large software systems needs to be analyzed
and designed thoroughly. This development method was in the 1980s and 1990s
seen as the best way to develop software [6]. It came from the software engineer-
5
==PAGE:15==
ing community, which was responsible for developing large software, which e.g.
could be for the government. The teams developing the software were large, and
they worked together with different departments and other companies. There-
fore the time was spent on requirements, analyzing and designing the software.
This way it would be easier to delegate the tasks to the different teams and de-
partments.
Though plan-driven development is great for large software, it might be just as
good for small to medium-sized software. The reason for this is instead of fo-
cusing on the code and development, the time could be spent more on how the
system should be, rather than on implementation and testing. This is were in
the 1990s a large number of software developers came up with a new method to
develop software. They called it ’agile methods’. Agile methods or agile develop-
ment is on the other hand focused on the software itself rather than on its design
and documentation.
A manifesto about the philosophy of agile methods was made. This manifesto
was agreed upon by lots of the developers who were working on agile methods.
The manifesto says:
We are uncovering better ways of developing software by doing it and helping oth-
ers do it. Through this work we have come to value:
Individuals and interactions over processes and tools
Working software over comprehensive documentation
Customer collaboration over contract negotiation
Responding to change over following a plan
That is, while there is value in the items on the right, we value the items on the
left more. [7]
Page 6 of 93
==PAGE:16==
The manifesto clearly states that the implementation and responding to soft-
ware changes, is more important than design and documentation. This makes
good sense for small- to medium-sized software, because time is money. And
therefore the time should not be used on analyzing and designing the software,
which would be a waste of time, because it probably would not make a differ-
ence anyway. So if the system is not large and complex, then the focus should
be on implementing the software.
Regarding the aforementioned statements I should use only the agile method,
because my project would be categorized as small- to medium- sized software.
But this is not the case. I chose a combination of plan-driven and agile meth-
ods. I used the plan-driven development to get an overview over the software
I was developing. I started to brainstorm what my application should do, what
functions it needs etc. This way I broke the work down into activities and iden-
tiﬁed the milestones. Next step was to make a project schedule. Here I put in all
the activities and milestones from the previous step. This way I had an overview
over what tasks I needed to do each week. I knew at which time I should reach
my milestones. Of course I did not know the whole application I was develop-
ing, and this is where the agile developing comes into place. I used plan-driven
development to plan my project as detailed as I could, but the rest would be ag-
ile development. Focus on the software and if a change happened in the middle
of the product, I could respond to that instead of following the project plan to a
tee. You could say that the project plan was a guideline, and not a rule book. The
project plan can be viewed in appendix A.1, A.2 and A.3.

3.1
Virtual Reality (VR)
3.1.1
What is Virtual Reality
Virtual Reality (VR) is to replace your reality with a new virtual computer gener-
ated 3D environment. This is done by putting on a VR headset with a screen in-
side, which blocks out the outside world. This VR headset is called Head Mounted
Display (HMD). There are two common approaches to achieve a stereoscopic
3D image, which will trick the brain into believing you are inside a real world.
The ﬁrst is to send two feeds into one display and the second is to show two dis-
plays, one per eye. This will give the experience to see a 3D image. These HMD’s
also have motion control built in, which means, when the user moves and ro-
tates their head, the view in the virtual world will move and rotate accordingly.
This helps to make your brain believe that the virtual world is your new reality.
VR is not only for game entertainment, but is also great to prevent accidents. For
example VR can simulate an airplane in a sky environment, so newly graduated
pilots can train safely without any risks. Or it could be a simulation of surgery,
so newly graduated surgeons can practice a difﬁcult procedure without any risks
9
==PAGE:19==
for the patient involved. This means there is a lot of potential in this technology,
which has really expanded during the last decade.
3.1.2
History
The concepts of Virtual Reality
The concept of VR has existed for quite some time. The idea of being present in
a place somewhere you are not, is hard to trace back to a precise time. Because if
the concept of VR is just believing being present in a ﬁctive world, then many at-
tempts on VR has been made. For example the theatre where you go to see a play,
which have existed for centuries. In a theatre a play is performed and the main
purpose is to make the audience believe the story being told. Or another exam-
ple could be the 360 degrees panoramic paintings in the 20Th century, where
the paintings would make you believe you are in some historical battle or event.
The purpose of these panoramas was to make an illusion for the audience, to
make them visit a place and make them believe they were present. The aim for
the creators was to make the painting as real as possible to give the audience the
best experience [8].
In the essay "The theatre and its double" written by Antonin Artaud in 1938 [9],
Artuad mentions the theatre and its relationship to the actors and the reality. He
explaines how the theatre creates a sense of presence to the audience. This is
done through an emotional and psychological connection to the stage and its
performers. Artaud also believes that the theatre is a place to go to ﬁnd a sense
of humanity. To connect to your inner emotions. This belief can be paralleled to
the virtual world where the user also needs a connection to feel present. Emo-
tional and psychological connections makes the user in VR immerse into the
virtual world. This idea of connections and presence is similar to what Artuad
described. Artaud did of course not talk about VR as we know it, but the concept
behind it is basically the same.
A more described version of VR as we know it and maybe the ﬁrst real model for
VR, is the short story "Pygmalion’s Spectacles", written by Stanley G. Weinbaum
in 1935[10]. The story describes an invention of a magical spectacle which could
Page 10 of 93
==PAGE:20==
make the user enter a "virtual world" which would make the user believe that
the "virtual world" was real. These goggles also included touch and smells for
a more realistic experience. This story was at that time a science ﬁction, which
means that Stanley G. Weinbaum predicted the future pretty accurate. Today’s
HDM’s does not include smells, but the rest is pretty much how he described it.

The father of Virtual Reality
Morton L. Heilig (1926-1997) was a true visionary. He has been acknowledged
as "The Father of Virtual Reality" by many books and articles [11]. Heilig was
a philosopher and a ﬁlmmaker and he traveled in many countries to study. He
took a Bachelor of Philosophy at the University of Chicago, an arts degree at
Academie Julienne in Paris, read Philosophy at the University of Geneva in Switzer-
land, got a diploma in Film Direction in Rome and took a Master of Communi-
cation Arts in Pennsylvania. He was a well taught man with ambitions.
Figure 3.1: Picture of Morton L. Heilig ﬁlming for his Sensorama. Picture found
at [11]

In the 1950s Heilig invented a mechanical device that could make an illusion of
VR and patented it in 1961. He named it Sensorama and it stimulated multiple
senses. In ﬁgure 3.2, we can see how it looks. Only one person at a time could
use the Sensorama and immerse into one of the short ﬁlms used for it. Some
of the ﬁlms the user could experience was a motorcycle ride, a bicycle ride or a
helicopter ride [12]. The senses Sensorama could stimulate was a 3D experience
with a full color ﬁlm. It stimulated the ears with sound, the nose with smells,
it got motion feeling, the seat could vibrate and it got fans to blow wind in the
face or on the body. All these features would result in a virtual experience where
the user could immerse into the movie. Though the Sensorama did not get com-
mercial succes, it was a huge leep forward in terms of creating a VR experience.

The First Head Mounted Display
Many articles credit Ivan Sutherland[13] to be the ﬁrst person to make a HMD,
but Morton Heilig patented a device called Telesphere Mask in 1960 [14]. The
Telesphere Mask can be seen in picture 3.4. The Telesphere Mask shared the
same destiny as the Sensorama, it did not become a commercial success [14].
Maybe that is why it did not get the recognition it deserved. But the majority
believe it is Ivan Sutherland’s creation in 1968 to make the ﬁrst HDM [13]. The
device was called "The Sword of Damocles" because of its appearing. In the pic-
tures 3.3 we can see how the device is hanging from the ceiling. It was a heavy
device which could not be lifted and utilized by a single person, therefore it is
attached to the ceiling. This HDM was also one of the ﬁrst steps into the aug-
mented reality (AR) world.

Unity is a game engine specialized to build games to multiple platforms and de-

4. Playstation VR
5. Gear VR
6. Windows Mixed Reality

Unity supports 2D and 3D game making and uses scripting for the logic. C# is the
most common choice. When programming in Unity and since the new release
of Unity 7.1, they are about to deprecate UnityScript, which is a Javascript-like
language [19].
When developing a software project in e.g. Java, you would probably be using
a design pattern like "Model View Controller (MVC)", to have a better overview
over the software. It is a little different when you are developing with a game
engine.
Unity uses component based design for its architecture and development. This
means that in Unity you create a scene which contains all the components needed
in the application. In that scene the developer can create and add components
(gameObjects). GameObjects can be anything e.g. physical objects, sound and
materials. Components of functionality can be added to the gameObject. There-
fore all components in the scene consists of different gameObjects. For example
Page 17 of 93
==PAGE:27==
if I want to have a cube that should be able to collide with other cubes, then col-
liders will be added to the cube gameObjects. If I want the the cubes to simulate
gravity I will add a rigidBody to them. If I want more functionality I can create
a script and attach it to cube gameObjects. This means in Unity you create the
models, like walls, furniture’s etc. and then create scripts for its behavior. This
way you form the gameObjects to behave like you want. This is why it is differ-
ent than most software projects. But this does not mean you cannot make good
architecture and make it easier for other developers to understand, so they can
further develop the project. Instead of making a class diagram over the structure
of the software, a scenegraph can be made over the Unity scene. This way you

4.0.4
VR sickness
When playing in the VR world VR sickness can occur. The sickness symptoms
is like motion sickness symptoms and can cause nausea, headache and general
discomfort. There is no all-ﬁx around from VR sickness, but there is tools that
Page 25 of 93
==PAGE:35==
can slightly decrease the amount of sickness [22]. This is an important issue, be-
cause if the user cannot stay in the virtual world for more than ﬁve minutes, then
the application is not worth much. Of course every person is unique, meaning
some will get VR sickness faster than other and some will maybe never experi-
ence VR sickness.
VR sickness is coupled to many different factors and one of them is motion.
Moving faster will give higher risk of getting VR sick. Therefore it is an impor-
tant issue if movement is required by the application. And in my case with the
interior design application, the user needs to be able to move in the room. The
question is then, how should the user move around. And for that there are some
different ways. One way would be moving the user/camera so the user get the
feeling that they can walk around. Another way could be teleportation. The
user can point at a point in the room and then instantly move to that position.
First thing to consider would be what makes the most sense. Second would be,
which one would cause the least VR sickness. To answer the ﬁrst consideration,
the ﬁrst thing I thought about would be moving around like we do in the real
world. That would give the best experience, because we cannot teleport in the
real world. But teleport is instant, so we can move really fast to new positions,
instead of walking. So actually that would be a good solution. To answer which
one would give the least VR sickness, motion can cause VR sickness as I men-
tioned above. Teleport is moving from position A to position B in one step. The
brain would then only have to adapt to the new position B. If we walked to the
same position B from A, then the brain would have to adapt to each step over to
the position. And if there were ten steps over to the position B from A, then the
brain would have to adapt ten times, instead of one with teleport. This would
increase the risk of getting VR sickness. Therefore teleport makes good sense
to use, when you want to decrease the amount of VR sickness and it also gives
ﬂexibility to move quickly around.

4.1
Technical challenges
To reach the functionality of the VR application, some technical challenges must
be solved. Therefore this section gives the reader an analysis of the technical
challenges of the functionality and how they were solved.
4.1.1
Interaction with reachable objects
The ﬁrst technical challenge is to interact with reachable objects. This means
the use of the controllers to grab, drop, move and rotate objects. In other words
change the transform values of the object so it gives the desired functionality.
First we look at, how we want the object to be grabbed. We want the grab to be
as real as possible; to give the same feeling as when you take a glass and hold it
in your hand. Then the glass will follow the hand’s movement and rotation. The
same process and feeling should the grab in a VR application have. This means
the object should be attached to the controller and then follow the same move-
ment and rotation as the controller. This can be achieved in different ways and
I will present two technical methods of how to do it.
The ﬁrst method is to combine the controller and the object. Which means
to turn the controller and the object into one object. The controller’s input is
tracked, which means when inside an VR application then the controllers is vis-
ible to the user and follow the user’s hands movements and rotations. Therefore
if we extend the controller object with a new object, then the new object will fol-
low the controller’s movement and rotation. In Unity this can be achieved with
the "Fixed Joints" component, which restricts an object’s movement to be de-
pendent upon another object. Actually the "Fixed Joints" uses force to combine
the objects, which means they behave like a magnet.
The second method to use is the SteamVR package 3.4. This package have pro-
vided examples of how you could use it. They have an example, where you can
pick up objects and throw them. The method they are using is making a trans-
form matrix with the position and rotation. This way we can take the two last
tracked positions and rotations from the controller and put it into the matrix.
The matrix now holds the position and rotation and it can take other objects
position and rotation to calculate what its new position and rotation should be,
compared to the controllers. This way the controller and the object will have the
Page 28 of 93
==PAGE:38==
same movement and rotation because of the transform matrix. This method is
more mathematically challenging compared to the ﬁrst method, because knowl-
edge about matrix is required, and how to make the transform matrix.
The ﬁrst method is more advanced and the only knowledge required is to use the
function right and then the function does all the underlying calculation itself.
This was two methods on how to grab, move and rotate an object. Now we will
look at how to drop the object again. This cannot be denoted as a technical
challenge, or at least not a difﬁcult one. In both the aforementioned methods it
should be rather easy.
The ﬁrst method using "Fixed Joints" we made two objects into one. That was
done by using force to attract the objects. To make them drop one method call
is needed, and it is "Break force". This will - as it says - break the attract force
between the objects, which mean the object attached to the controller will drop.
The second method with the transform matrix a notiﬁcation when to drop is
needed. When a drop should occur, then the notiﬁcation will be sent the method
which will stop calling the transform matrix method. The result is that the object
will no longer get new transform values, which means it will drop.
4.1.2
Interaction with objects at a distance
To interact with an object at a distance new technical challenges occurs. First
of all, the user has to target the object from a distance. And second is how the
control of the object should work at a distance. And third is where the center
of the rotation should be. Should the rotation center be at the controller or the
object itself? It depends on how you want the rotation to be. Should it rotate the
same as when the object is reachable or should it rotate like the user is holding
the object with a stick? If the center of rotation is placed with the object, then
the ﬂexibility of movement is decreased, because you can only move it equal to
the length of your arm. On the other hand, if the center of rotation is placed at
the controller, and it is like you are holding the object with a stick, then you can
move the object equal the length from the object to the controller. Though the
ﬂexibility of the movement is increased, the rotation is decreased. So it is a trade
Page 29 of 93
==PAGE:39==
off for the movement and rotation. What is more important is the movement or
rotation of the object. I think when you are working from a distance, then you
want to move the object freely in the room. This means the rotation of the ob-
ject is low. A good ﬁx to this, is to make the object electable, and then use the
trackpad on the controller to rotate the object around its own center. This way
we get the ﬂexibility of movement and rotation at the same time.
To answer the ﬁrst technical challenge on how to let the VR application know
which object the user wants to interact with, different methods can be used. For
example the user could have some kind of menu, where all the imported objects
can be seen and chosen. This way the user can choose which object to interact
with. This way seems a little time consuming from the users point of view. We
want it to be intuitive and easy for the user to interact with the objects and not
a hassle. Therefore a better solution would be, that the user would point at the
object and then target it. This also make more sense for the user. This can be
achieved by Unity’s "Ray casting". Ray Casting is a way to send out a ray from
a start point and when the ray collide with a collider, then it gives an output of
what it hit. This way we can get the object from a distance. The question now is,
which method should we use when the object should move and rotate? We out-
lined two methods in the above section 4.1.1. We can easily conclude that the
ﬁrst method is not going to work, because it uses force to combine the controller
and object, and therefore have to become one object. Now we want to work with
the controller and the object separately, where the object still follows the hands
movement and rotation.
The second method does not need the objects to be close, because it is using a
transform matrix. With the transform matrix, the object will get the movement
and rotation regardless of where in the room the object is located. Therefore this
method is a good solution, when the object is moving/rotating from a distance.
4.1.3
Object alignment and snapping features
A snapping function were probably the most difﬁcult technical challenge. There
are a whole range of factors that needed to be considered. Also there might be
many ways to implement it. First of all, the VR application have a lot of different
Page 30 of 93
==PAGE:40==
interior with different shapes. Most of that interior needs to be able to snap the
wall. These objects all have unique shapes and differs from each other. There-
fore it would be smart if the snapping function was general, so it would ﬁt any
shape. When the object has snapped to the wall, then it should be able to only
move on the wall plane with no rotation. So one axis need to be frozen plus ro-
tation. It is also important that the object snap directly to the wall, so there is
no gap between the wall and the object. Another point is that no matter what
rotation the object had before it hit the wall to trigger the snapping feature, it
has to align to the wall. Therefore information about the correct rotation to the
wall is needed.
Because the different interior has different shapes, the objects most likely have
different snap sides. In other words an object may only have one or two sides
that should be able to snap to the wall. This factor, makes it difﬁcult to make
a 100% general solution for all objects. Unique information about the objects
might be needed.
A possible solution would be to add a plane or collider on the side of the object
that needs to snap onto the wall. This way the chosen side would function as a
magnet and will seek against the wall.
Another solution which will work with small amounts of interior, will be hard
coding the correct snapping position and rotation for each object. When an ob-
ject triggers the snapping function, the application would know which object
hits the wall, and then search in the register for the transform information and
use it. The object will then transform to the correct position. The problem with
this solution is, that when the list of interior is long, it is very time consuming.
Especially if the room has many walls, because each wall needs to have its own
register for each object. This will require a lot of information that could have
been used on other things. It is also not ﬂexible or scale-able, because if the
walls is just moved a little, then all the information needs to be recalculated.
4.1.4
Object generation and placement
For object generation and placement, Unity has some nice features. To generate
or spawn objects in runtime of the application, the method "Instantiate" is used.
Page 31 of 93
==PAGE:41==
It takes an object and its position and rotation as arguments. The position and
rotation is where the object shall spawn. An empty gameObject can be created
which only contains its transform. The developer can then choose the place the
object shall spawn. The gameObjects transform is then given as input to the In-
stantiate method. Then the method Instantiate is called and the object will be
generated at that position with the given rotation.
4.1.5
Object selection and motion control
The technical challenge in selecting the object is not the most difﬁcult, but to
control the axis values independently is more of a challenge. For the object se-
lected we want the object to freeze at its current position. For this to happen we
just need to change the gravity and "is.kinetic" values of the object.
For the motion control, we need to specify which transform change we want.
Move, rotate or scale are the choices, and a menu seems like a good idea. A menu
that can specify if the object should move, rotate or scale. This menu would also
decide which axis that should change, and the user should just be able to change
the values quickly no matter which location the user has in the room. Therefore
a menu like the interior generator would be a bad idea, because each time the
user would change a value, then the user would move to that location. As I men-
tioned earlier, menus popping up in front of the user inﬂuences the experience
in a negative way. Though in this particular example it seems good. The menu
buttons and text should be as small as possible, so the user can still see the sur-
roundings and it should be possible to toggle on and off. Radio buttons with text
would be a very good solution to this menu. The user just needs to know which
radio buttons that needs to be pressed and then the user can toggle the toggle
menu off and change the values with the trackpad touch. This gives the user
ﬂexibility and can toggle the menu on and off in every location of the room.

Chapter 9
Conclusion
The purpose of this thesis was to look into the interior design world from a VR
perspective. And explore some of the technical challenges that may occur when
working with interaction in a VR environment. I have made a VR application
with a house environment where the user can move around in, with the help
of a teleport function. I have made an interior menu, where the user of the VR
application can choose between two levels. One level with an empty room and
the other one a predesigned room with interior in it. Furthermore the user can
pick up the interior on the table and generate it in the application. The user can
grab the interior with the controllers and drop them. Then the interior are at-
tached to the controllers and it follows the controllers motions. I have made a
laserpointer that can interact with the interior in the room. The user can pick
up interior from a distance and release them. When attached the interior fol-
lows the controllers motion. The user can select interior when it is attached to
the controllers from long range. When the interior is selected then its position
and rotation is frozen. I have made a toggle menu where the user can choose be-
tween the options move, rotate and scale. In each of these options the user has
the ability to choose which axis that needs to be changed. And when the interior
is selected the user can slide the thumb on the trackpad on the controller and
the interior will change accordingly to the option chosen in the toggle menu. I
have made a snapping function, so when an interior touches a wall it will align
itself nicely up against the wall and can be moved along the walls plane. This
function though only works on one wall and one object at this version of the VR
67
==PAGE:77==
application.
I have designed the application to have a proper structure, so further develop-
ments would be easy to read and understand for other developers.
I have used black box testing to test the VR application in means of use cases
and functional tests. Further more I have tested the functionality in every run of
the VR application.
If there was something I would change with my current knowledge throughout
the whole project, it would be to plan my time even more carefully and precisely.
The most important resource is time and therefore it is important to manage it
correctly to reach the ﬁnal goal. Unforeseen scenarios should also be taken into
account, for example when the HTC Vive headset stopped working for unknown
reasons. Small incidents that takes a lot of time in the long run. It is however
difﬁcult to calculate these factors, but it is important to be aware that these in-
cidents occur. It would also be great to have a backup plan for such incidents.
Otherwise I am generally satisﬁed with the project and the results, even though
I wished to have the snapping feature working for all interior.
9.1
Future work
The VR application as it is now, is only an early prototype which contains the
basic interaction. This application still has a long way to go, compared to the
IKEA Home Planner. But with more time and effort it should be possible and
the feeling of creating interior design with VR is great. It is a very nice experi-
ence and feels intuitive and right. The future priorities of this VR application is
for the snapping feature to be working for all walls and objects. Furthermore
implementing a grid would be a very useful tool, because the interior then can
be placed more precisely. Duplicate interior should also be possible and is one
of the things to implement in future versions. This means that there are many
functions and a lot of new challenges ahead, and I think in a few years time VR
will take over the interior design industry. Maybe the 3D interior software will
still be used, but VR is progressing very fast and expands into an array of new
areas. I think it is fair to say that VR is here to stay.
