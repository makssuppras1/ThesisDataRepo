Summary (English)
This master thesis investigated the Size-constrained Weighted Ancestor (SWA) problem, focusing
on the state-of-the-art solutions presented by Bille et al. The primary goals were to reproduce
their 𝑂(𝑛)-space, 𝑂(1)-query algorithm, evaluate their practical performance, and compare them
to simpler baseline approaches. Alongside this, the project sought to explore refinements aimed at
improving the real-world efficiency of the implementation without altering asymptotic complexity.
To achieve this, both versions of the solution from the paper were implemented independently in
C#, together with all required supporting data structures, including Rank/Select bitstrings, Lowest
Common Ancestor (LCA), Fusion Tree, Heavy Path Decomposition (HPD), and ART Decomposition.
Each built from scratch for full control over data layout and ability to fix overhead. For each
queryable component, at least one naive baseline was implemented to serve as a correctness
oracle. Unit tests for targeted edge cases and property-based testing against generated inputs
confirmed correctness across a variety of trees and query node and threshold combinations.
The project extended the original designs by slightly simplifying the query and developing a Data
Oriented Design (DOD) tree representation to improve cache locality and reduce memory usage.
Targeted profiling identified and addressed constant-factor bottlenecks during development,
reducing memory footprint and improving query latency for the final result.
Benchmarking measured preprocessing and query performance for both naive and optimal
approaches on different synthetic tree shapes.
Results confirmed the theoretical complexity bounds under the tested conditions, mimicking
worst case scenarios, and highlighted practical factors influencing performance, such CPU cache
behavior and temporary values. The best solution ended up being the DOD variant of the 𝑂(𝑛log 𝑛)
space solution, displaying linear space growth within the tested data sizes, with the fastest query
of those that didn’t precompute all possible answers.
Overall, the project delivered a complete, validated, and extensible set of SWA implementations
ready for further experimentation. It confirmed the theoretical performance claims of Bille et
al., provided insight into their practical viability, and demonstrated the importance of carefully
considering implementation choices whenever performance is needed.

Resumé (Danish)
Det følgende speciale har undersøgt Size-constrained Weighted Ancestor (SWA) problemet med
fokus på en af de nyeste løsninger præsenteret af Bille et al. De primære mål var at reproducere
deres 𝑂(𝑛) plads, 𝑂(1) tids forespørgelse, evaluere den praktiske ydeevne og sammenligne den
med enklere løsninger. Projektet havde samtidigt til formål, at udforske forbedringer, der kunne
øge effektiviteten af implementeringens i praksis uden at ændre den asymptotiske kompleksitet.
For at opnå dette blev begge versioner af artiklens løsningen implementeret uafhængigt i C#,
sammen med alle nødvendige datastrukturer, herunder Rank/Select bitstrenge, Lowest Common
Ancestor (LCA), Fusion Tree, Heavy Path Decomposition (HPD) og ART Decomposition. Hver blev
bygget fra bunden for fuld kontrol over datalayout og mulighed for at reducere overhead. For
hver løsgning med specific tilhørende søgning, blev der implementeret mindst én naiv baseline,
som fungerede som korrektheds-orakel. Unit test for edge cases samt property tests på genereret
input bekræftede korrektheden på tværs af forskellige træer og kombinationer af forespørgsler.
Projektet udvidede de oprindelige design ved at forenkle bestemmelsen af svarkandidat for SWA
forespørgslen og udviklede en Data Oriented Design (DOD) træ repræsentation for at forbedre
cache-lokalitet og reducere plads forbruget. Profilerings værktøjer blev brugt til at identificere
konstante faktororer under udviklingen, for at sikre at løsningerne blev implementeret med så lidt
overhead som muligt.
Benchmarking målte præprocesserings og forespørgsels tid for både naive og optimale versioner
på forskellige syntetiske træformer. Resultaterne bekræftede de teoretiske kompleksiteteter
under de testede forhold, efterlignede worst case-scenarier, og fremhævede praktiske faktorer,
der påvirker ydeevnen, såsom CPU-cacheadfærd og midlertidige værdier. Den bedste løsning viste
sig at være DOD-varianten af 𝑂(𝑛log 𝑛) plads løsningen, der udviste lineær pladsforbrug inden for
de testede datastørrelser, samt den hurtigste forespørgsel blandt dem, der ikke forudberegnede
alle mulige svar.
Samlet set leverede projektet et komplet, valideret og udvidelsesklart sæt af SWA-imple-
menteringer, klar til yderligere udvikling og eksperimenter. Det gav indsigt i løsningens praktiske
anvendelighed og demonstrerede vigtigheden af nøje at overveje implementeringsvalg, når der
er behov for høj ydeevne.

Preface
This thesis was prepared at DTU Compute, Technical University of Denmark, fulfilling the require-
ments for acquiring a Master of Science degree in Computer Science and Engineering.
The thesis explores the Size-constrained Weighted Ancestor (SWA) Problem, implements and
validates a state of the art solution, which preprocesses the tree in 𝑂(𝑛) time and space to answer
queries in 𝑂(1) time.
Variants and simplifications have been implemented to explore the impact of constant factors, and
compared against three naive baselines, with focus on comparing the theoretical and practical
efficiency.
The thesis was written from 10 February 2025 to 10 August 2025 under the supervision of Inge Li
Gørtz & Philip Bille and is worth 35 ECTS credits.
Kongens Lyngby, 10 August 2025

1 Introduction
1.1 Background
The size-constrained weighted ancestor (SWA) problem is a search problem in the domain of graph
theory, more specifically it is an extension of the predecessor search problem for rooted, node-
weighted trees. At a high level, it involves finding the lowest ancestor of a given node whose weight
meets or exceeds a specified threshold.
The SWA problem is conceptually similar to other tree-based ancestor queries such as the lowest
common ancestor (LCA) problem and the level ancestor problem. Like these, it requires efficient
navigation through a tree structure, but differs by the queries being value dependant. This adds
a layer of complexity, as it isn’t enough to efficiently navigate the tree structure itself, but also do
so for node values. It is for this reason the problem specifies weight restraints of non-decreasing
growth and size-capped weights for nodes of the tree going up towards root.
One of the use case of the SWA query is in string indexing with suffix trees. If each node is weighted
by the number of leaf descendants, corresponding to the frequency of the substring represented
by that node, an SWA query can, for example, find the longest prefix of a substring that occurs at
least 𝑓 times in the text. This is useful in applications such as search engine query refinement and
frequent pattern detection.
Earlier work on the weighted ancestor problem in suffix trees, most notably by Belazzougui et al.
[1], achieved 𝑂(1) query time and 𝑂(𝑛) preprocessing time and space when nodes were weighted
by string depth. However, that approach relied heavily on this specific weighting scheme and could
not be adapted to other weight functions, required for queries like the frequency-constrained
substring search described above.
Bille et al.[2] addressed this limitation by introducing the generalized size-constrained weighted
ancestor formulation, which supports any weight function satisfying the size-bound and max-
heap properties. Presenting a solution that similarly achieves 𝑂(1) query time with 𝑂(𝑛)
preprocessing, improving generality without sacrificing optimal bounds. Their approach combines
several established techniques, including heavy-path decomposition, unary-coded bitstrings with
constant-time rank/select support, and LCA, in a way that ensures both constant-time queries and
strictly linear space usage.
While these complexity bounds are optimal in theory, Big-O notation standardly used for analyzing
algorithms and data structures, only describes how attributes like time and space usage grow
with input size. It does not reflect constant factors introduced by implementing on real hardware,
memory layout effects, cache behaviour, or programming language constraints, all of which
determine whether a solution is competitive in practice. This raises the central question of this
project: Can the optimal SWA solution of Bille et al. be implemented to run efficiently on consumer
hardware, and is it in practice faster or more space-efficient than simpler baseline methods?

1.2 Problem statement
Let T be a rooted tree on a set 𝑉 of 𝑛 nodes, where each node 𝑢∈𝑉 is assigned a positive integer
weight via a size-constrained max-heap function weight: 𝑉→ℕ.
This function satisfies two conditions:
1. For every node 𝑢, weight(u) ≤size(u), where size(u) denotes the number of nodes in the subtree
rooted at 𝑢,
2. and if 𝑢1 is a descendant of 𝑢2 then weight(𝑢1) ≤weight(𝑢2).
A size-constrained weighted ancestor query, SWA(u, k) for a given node 𝑢 and positive integer
𝑘> 0, returns the lowest (i.e. deepest) ancestor 𝑤 of 𝑢 such that weight(𝑤) ≥𝑘. The SWA problem
is to preprocess 𝑇 into an efficient data structure that supports fast queries.
Aim of Project
The goal of this project is to explore and analyze the solution to the SWA problem as presented by
Bille et al. in their paper[2]. We will implement their linear-time, linear-space data structure that
supports SWA queries in O(1) time and compare it to a naive solution to the weighted ancestor
problem. We will compare the theoretical and practical efficiency of the solution implementation.
Methods:
• Implement both steps of the optimal 𝑂(𝑛) space, 𝑂(1) query solution of Bille et al., along with a
simpler baseline methods.
• Build all necessary supporting structures from scratch, including rank/select bitstrings, LCA,
Fusion Tree, and decompositions.
• Evaluate correctness through systematic testing against naive solutions of both final SWA
solution and sub structures.
• Benchmark performance across varied input tree shapes and sizes, including worst-case
scenarios for each solution.
Outcome:
The expected outcome is a working implementation of the state-of-the-art SWA solution, a
comparative performance analysis against naive baselines, and insights into its practical viability
on consumer-grade hardware. This will include identifying conditions under which the optimal
structure’s advantages are most pronounced, and cases where simpler methods may still be
preferable.

2 Methodology
This chapter outlines the method used to understand, implement, and evaluate the size-
constrained weighted ancestor (SWA) data structure, as described by Bille et al. The method was
iterative and exploratory in nature, and can be divided into four distinct continuous steps that will
be the foundation for the discussion in Section 7.
2.1 Overview
Rather than trying to fully define all tasks and challenges in advance, the work progressed in cycles
of research, design, implementation, and testing for each part of the solution to accurately explore
and understand how to best implement, test, and compare against the theoretical expectation.
This approach is considered to be a version of the AGILE workflow.
This agile workflow was chosen to accommodate the complexity of the SWA algorithm, which relies
on multiple advanced data structures working in combination. Each component was treated as an
independent unit of work. For each, the theory was first studied, followed by a design sketch, a
C# implementation, and a test suite using property-based and unit testing. If the tests failed or
behaved unexpectedly, the cycle was repeated, to correct mistakes or revise the design.
The initial focus of the project was to implement the SWA data structure as described in the paper
and establishing benchmarks comparable to the theoretical analysis of the solution. With the aim
of properly understanding each claim of the paper, validate their correctness, implement each
component theoretically correct, and take meaningfully benchmarks.
These goals are complex and difficult to estimate precisely, which was why i had to plan with
flexibility in mind. Instead of working on each component one at a time, only continuing to the
next task when fully done like the waterfall approach, a gantt diagram was used to mark when to
make a focused effort. If a component wasn’t fully explored or implemented, at the very least a
naive version would have been created, which could be used as a placeholder until there was time
available again. Thus ensuring that at least a working prototype would be made, in the event that
some aspects where much more complex than expected.
The gantt chart was left with intentional buffer zones, and the final part of the project was left open
as an exploratory phase. The intention was to investigate whether the design could be improved
in any meaningful way, in terms of performance, structure, or space use.
The method followed in this project is therefore summarized in four steps: (1) research the data
structure, its components, valid input and edge-cases, and the impact of the chosen language.
(2) design the solution based on theoretical understanding, (3) implement the full solution in C#
using a test-driven workflow, and (4) experimentally evaluate the result with profiling, property-
based testing, and benchmarks.

The research step aimed to build a solid understanding of the SWA problem and its solution as
described in[2]. This included not only understanding the high-level solution, but also investigating
how each of its underlying components contributed to the overall performance and correctness
of the data structure.
To begin with, the original paper was read multiple times to form a clear picture of what the
problem entails. The solution builds on a series of well-established data structures, such as Rank/
Select bitstrings[3] and Lowest Common Ancestor  (LCA) structures. Since the paper assumes
familiarity with these components, additional reading was needed to fill in the gaps. Refereed

2.3 Step 2 - Design
This step aimed to define the conceptual design of the data structure, outlining what the

The design focused on describing the data structure as a composition of distinct components, each

The implementation step focused on translating the theoretical design into a working system

3.1 Possible trees
As part of the exploration phase, it is important to consider the types of trees on which the SWA
problem may be applied, and how their characteristics could influence performance. A tree is
defined by two independent aspects: its shape, which determines the structural arrangement of
nodes, and its weighting scheme, which assigns values to those nodes. By examining a variety of
shapes and weightings, it is possible to anticipate how different inputs might affect preprocessing,
query times, and memory usage. This understanding helps in forming expectations about the
practical behavior of each solution before conducting any benchmarks.

To evaluate how different input properties influence the behavior of the SWA solution, a number
of distinct tree shapes were thought of, each designed to emphasize one or more key attributes.
These attributes, most notably the height of the tree and the number of leaves, will affect both
query performance and preprocessing.

Some shapes intentionally push these extremes. The Chain has the maximum possible height,

Other shapes, such as the Caterpillar and Composite variants (e.g., Arrow and Broom), are

rearrangements impact decomposition quality or query behavior, even when high-level metrics

to favor the naive solutions with little difference to the optimal.
On the other hand, shapes like Asymmetrical and Alternating are primarily useful for correctness
testing rather than performance benchmarking. These trees exhibit irregular or mixed structural
features that do not push any single attribute to an extreme. As a result, they are less likely to
expose worst-case scaling behavior, but are still valuable for verifying the implementation handles
arbitrary trees correctly. The Random trees in particular serve as a general test input that can
include a wide variety of patterns, including asymmetries or depth fluctuations.
Because of this no Asymmetrical or Alternating shapes have been defined, and we will instead rely
on exhaustive tests of the Random shape to cover these aspects.
3.1.2 weightings
In addition to tree shape, the weight function assigned to the nodes also plays a role in the
behavior of the SWA solution. The SWA problem, assumes a weight function that satisfies the size-
constrained max-heap property of each node 𝑢 of tree 𝑇.
This constraint still allows for considerable flexibility in how weights are assigned. Depending on
the pattern or distribution of weights across the tree, the internal structure and layout of the SWA
data structure can vary, which in turn affects both preprocessing and query performance. Below
in Table 2 distinct categories of possible weight functions are listed, followed by discussion on how
they might influence the system.

The first three types of weight function share the property that weights increase monotonically

3.2 Data oriented design
Data Oriented Design  (DOD)[5], [6], [7], [8] is an approach to programming that focuses on
organizing data in memory to match how it is accessed at runtime, with the goal of improving
performance. The central idea is to optimize how data flows through the CPU by minimizing cache
misses, reducing memory overhead, and taking advantage of predictable access patterns.
One of the key motivations for DOD is to speed up programs by maximizing the use of fast CPU
cache levels, while minimizing access to slower memory like RAM. Figure 1 below illustrates a
ballpark estimation of the difference in cpu cycles between different operations, including access
cost for registers, CPU caches L1 to L3, and main memory (Ram). This diagram is from 2016 and
CPU’s have improved a lot since then, however these are still the numbers found when discussing
memory reads today, with some even doubling the ram latency to 100-300 cycles.
The actual number isn’t that important and heavily depends on the CPU, just that operations
becomes multiple (∼2-8) times slower when moving up a layer in the memory hierarchy. Data is
moved up a layer when new data is read and there is no space left in the current layer, where the
oldest accessed data is removed. Each layer increases dramatically in size, for example registers
are usually 64-512bit on modern CPU’s, with L1-L3 in the KB-MB range, and main ram 8-128GB.
The objective is therefore to minimize the amount of accessed data, as well as how to store it for
better retrieval, thereby minimizing cache misses, reducing space consumption, and improving
bulk operations.

A typical object-oriented (OOP) representation stores each node as a separate object, with
references (pointers) to its parent and children objects. These references point to arbitrary
locations in memory, meaning that related nodes may be scattered throughout RAM. This
scattering forces the CPU to load multiple, often unrelated, cache lines during traversal, increasing
latency and reducing the benefit of hardware prefetching.
Additionally, OOP objects frequently suffer from natural alignment padding. In memory, struct or
class fields must be aligned to boundaries determined by their type size, and the compiler may
insert unused padding bytes between fields to satisfy alignment rules. For example, a class storing
an int (4 bytes/32 bit) followed by a boolean value (1 byte) and then a long (8 bytes) will introduce 3
bytes of padding after the byte so that the long starts on an 8-byte boundary. In large structures,
this wasted space accumulates significantly, especially in worst case where half the object size is
padding. This can happen because the natural alignment is chosen to equal the size of the largest
field, but if the only other field is for example a bool, which theoretically could be represented in
1 bit, but uses (8 bits/1 byte) in C#, then the size of the large field minus 1 is pure wasted space.

==PAGE:21==
DOD addresses these problems by reorganizing data using techniques such as:
• Using indices instead of pointers: Each node is identified by an integer index into an array,
allowing all node data to be stored in contiguous memory blocks. This eliminates arbitrary
scattering in RAM and ensures predictable memory access patterns.
• Field reordering for natural alignment: Struct fields are ordered by size and alignment
requirements to minimize padding. And in some cases smaller identifying fields like id or type
is refactored out of the object, instead representing this value by having different object or
collections for data that before were of the same type.
• Struct-of-Arrays (SoA): Instead of storing an array of node objects (each containing all fields),
SoA stores each field in its own contiguous array. For example, all parent indices are stored
together in one array, all weights in another, and so on. This means that when iterating
over a single field for many nodes, such as checking all weights, the data is fetched in a
single continuous memory stream, improving cache efficiency and enabling vectorized or SIMD
processing.
These changes reduce the number of cache lines touched during iteration, improve prefetching
efficiency, and make bulk operations on the same field much faster. Another benefit is that the
arrays of SoA forces the user to think of an order for the data, instead of needing dictionaries
for lookups.
DOD also includes broader strategies such as avoiding unnecessary abstraction, minimizing virtual
calls, and replacing polymorphism with flat data representations. These were not applied in this
project.

4.1 Naive solutions
Before exploring the more sophisticated solutions to the SWA problem, lets begin with three
straightforward approaches. These solutions are inspired by well-known methods for the level
ancestor problem[9], a simpler but related query problem in rooted trees where the goal is to find
the ancestor of a given node u, at a fixed distance above it in the tree.
The level ancestor problem admits several classical solutions[10], each with its own trade-offs in
terms of query time and space usage. We adapt these strategies to the SWA setting to establish
meaningful baselines. While none of the approaches presented in this section are asymptotically
optimal for general trees, they are simple to understand, easy to implement, and useful as both
correctness references and benchmark baselines.
The three approaches we discuss are:
• UpTraversal, a brute-force traversal from the given node up to the root.
• DirectShortcuts, a table-based approach using precomputed answers for all threshold values.
• JumpPoints, a binary-lifting scheme that enables logarithmic query time with moderate space
overhead.

4.1.1 Up Traversal
Figure 2: Depiction of Up Traversal on tree T from given node u.
Each parent is compared to the threshold k until ancestor w is located.
The simplest method for answering SWA queries is to traverse upward in the tree from the
query node u until a node is reached whose weight meets or exceeds the threshold k. At each
step, inspect the current node 𝑤𝑖, and if weight(𝑤𝑖) < 𝑘, we move to its parent 𝑤𝑖+1. This process
continues until either a suitable ancestor 𝑤 is found, or the traversal reaches the root without
satisfying the condition, in which case the answer is none.
This method does not require any additional space beyond what is already needed to represent
the tree and the weight function. It performs no preprocessing and operates purely at query time.
As such, it uses O(1) space, but the time complexity is linear in the height of the tree O(depth),
which in the worst case is O(n).
The correctness of this method follows directly from its construction: we examine ancestors in
order of decreasing depth, and return the first whose weight satisfies the constraint. Due to the
monotonicity of the weight function with respect to ancestry, once a node meets the threshold, all
higher ancestors will also meet it. By stopping at the first valid node, we ensure that the result is
the lowest ancestor with the required weight.
Although inefficient for larger trees, UpTraversal is a useful reference point for its simplicity and
as a easy way to test whether any subsequent solution is correct.

4.1.2 Direct Shortcuts
Figure 3: Depiction of shortcut table for node u
The second approach uses the former idea to preprocess a complete lookup table for each node.
For every node u, precompute and store the result of SWA(u, k) for every possible value of k up to
the maximum weight found in the tree. This creates a direct access mapping from threshold values
to answers: for a given node u and threshold k, the query result can be retrieved immediately by
indexing into a table in O(1) time. A bounds check, for whether w is in the table, can be done in
O(1): if 𝑘< 1 return u, if k > weight(root) return none.
Because the number of distinct weight thresholds can be up to O(n), since weight(root) is bound
by n, and we store one answer per threshold for each node, the total space usage is O(𝑛2). The
preprocessing step involves a single traversal of the tree. At each node, an array of size O(n) is
filled based on the node’s own weight and the parent’s precomputed array, which takes O(n) time
per node. With n nodes in total, the overall preprocessing time is 𝑂(𝑛2).
Correctness is immediate from construction: for every node and every relevant threshold, we
precompute exactly the value that a correct SWA query would return, and the algorithm simply
retrieves it.
Nothing much is faster than a single bounds check and table lookup, marking the method
as theoretically optimal in query time, though extremely inefficient in space and therefore the
solution expected to first hit a memory wall.

4.1.3 Jump Points
Figure 4: Depiction of Jump Points query from u to w
The final baseline combines ideas from both tabulation and direct traversal, but instead of
storing answers for every possible threshold, it stores a series of exponentially spaced jump
pointers for each node. For a node 𝑢, this means keeping references to its ancestors at distances
20, 21, 22, …, 2{⌊log 𝑛⌋} steps up the tree.
A query SWA(u, k) is answered using a binary search-like approach. Starting from the largest
possible jump, we check if the ancestor reached by that jump still has a weight less than 𝑘. If it
does, we take that jump and continue from the new node. If it does not, we skip it and try the next
smaller jump. By halving the maximum possible upward distance at each step, we rapidly narrow
down the search range, like a binary search over the path to the root.
This process ends when we have found the deepest node whose weight is still below 𝑘. Returning
its parent gives the first ancestor whose weight meets or exceeds 𝑘, or null if the root was reached
without finding such a node.
The method uses 𝑂(𝑛log 𝑛) space and 𝑂(log 𝑛) query time. Preprocessing also takes 𝑂(𝑛log 𝑛)
time, with jump pointers computed via dynamic programming: for each node and jump length,
the ancestor at distance 2𝑗 is obtained from the ancestor at distance 2𝑗−1.
Correctness follows from the greedy nature of the search: we always try the largest possible jump
that stays below 𝑘, ensuring we never overshoot the correct ancestor and that the final jump yields
exactly the direct child of the size-constrained weighted ancestor.

Before introducing the full 𝑂(𝑛log 𝑛)-space and 𝑂(𝑛)-space data structures for the SWA problem,

4.2.1 Rank/select
A bitvector is a binary string B of length m. Two standard operations on bitvectors are:
• rank𝛼L(𝐵, 𝑖): returns the number of bits equal to 𝛼∈{0, 1} in the prefix 𝐵[1..𝑖].
• select𝛼L(𝐵, 𝑟): returns the smallest index j such that rank𝛼(𝐵, 𝑗) = 𝑟; i.e., the position of the r-th
occurrence of bit α.
A bitvector of length 𝑚≤𝑛 can be preprocessed in 𝑂(1 +

log 𝑛) time into a data structure occupying

4.2.2 LCA
The Lowest Common Ancestor (LCA) problem[11] finds the first ancestor 𝑤 of any given two nodes.
The approach begins by performing an Euler tour of the tree, recording both the node visited and
its depth at each step. During this traversal, the first occurrence of each node in the Euler sequence
is also stored for direct indexing.
Given two nodes 𝑢 and 𝑣, the LCA query reduces to finding the node with the smallest depth
between the first occurrences of 𝑢 and 𝑣 in the Euler tour array. This is equivalent to answering an
RMQ over the depth array in that interval[12].
The RMQ structure is built using:
• The depth array is divided into fixed-size blocks.
• All possible block patterns are precomputed and stored in a lookup table.
• An upper-level sparse table is built over the block minima.
The preprocessing runs in 𝑂(𝑛) time and space, and each LCA query is answered in 𝑂(1) time:
• Look up the positions of 𝑢 and 𝑣 in the Euler tour array.
• Identify the RMQ interval between them.
• Query the precomputed tables to find the position of the minimum depth in that range.
• Return the corresponding node from the Euler tour.
This design is well-suited to the SWA problem as it provides fast, cache-friendly LCA lookups while
keeping space usage linear in the number of nodes.
4.2.3 Fusion tree
A fusion tree[13] is a static data structure for predecessor queries over a small, sorted set of
integers. It achieves constant-time queries while using only linear space in the size of the set.
Fusion trees are designed to take advantage of the word-level parallelism available in the standard
word RAM model.
At a high level, a fusion tree stores a set of integers by identifying the bits that distinguish them,
called significant bits, and combining them into compact representations. These compressed
representations are arranged in a B-tree-like structure with large branching factor. The key idea is
that all branching decisions during a search can be computed in parallel using a small number of
bitwise and arithmetic operations.
Because the number of elements is small, all branching logic fits within a constant number of
machine words. This allows a query to determine its correct predecessor by simulating multiple
comparisons in parallel in constant time.
A set of 𝑚= log𝑂(1) 𝑛 integers can be preprocessed into a fusion tree in 𝑂(𝑚) time and space to
support predecessor queries in 𝑂(1) time.

4.3 𝑂(𝑛log 𝑛) space
This section describes the first of two implemented data structures following the approach

While the core structure closely follows their original algorithm, this thesis presents a detailed
breakdown of its components, supported by minor simplifications and implementation choices.
The design of the data structure is conceptually divided into two main components, each handling
a distinct part of the problem. The first part involves decomposing the tree into a system of heavy
paths, which allows the ancestor relationships within individual paths to be accessed efficiently.
The second is a search layer, which leverages fusion trees to perform efficient predecessor queries
over small sets of integer weights, for locating the heavy path containing the correct answer. These
two ideas are combined to preprocess the tree in such a way that queries can be answered by
navigating a small set of compact structures.
The following subsections, describes each component of the construction in detail, beginning with
the heavy path decomposition.
4.3.1 Heavy path decomposition
Similar to Long Path Decomposition, which is used in the level ancestor problem to uniquely group
each node along its longest downward path, Heavy Path Decomposition (HPD) is a technique for
partitioning a tree into a system of paths that reflect its structural traits. In this decomposition,
each internal node u selects one of its children v, the one with the largest subtree size(𝑣𝑖), as its
heavy child, and marks the edge (u,𝑣𝑖) as heavy. If multiple children share the same size 𝑣𝑗= 𝑣𝑖
one is chosen arbitrarily. All other child edges are marked as light. A heavy path is then defined
as any maximal sequence of nodes connected via heavy edges. Figure 5 is an example of such
decomposition.

red outline. Each node 𝑢∈𝑇 is weighted by weight(u) = size(u), making heavy children visually

This decomposition can be constructed in O(n) time and space using a recursive top-down strategy.
Beginning at the root, the algorithm repeatedly descends through the heavy child to greedily
construct a maximal heavy path. Once a path is formed, all subtrees rooted at light children are

This bitencoding has the two useful properties: The total number of 1s equals the weight of the

4.3.3 Locating heavy path
Having established how to answer SWA queries efficiently within a single heavy path, we now turn
to the problem of identifying the correct heavy path to inspect for a given query SWA(u,k). The
goal is to locate the lowest ancestor w of node u such that weight(w)≥k, but only where w lies at
the top of some heavy path H. Once this top node is located, the answer can be found using the
bitstring B(H) described previously.
To achieve this, we conceptually walk up from node u toward the root, visiting only the top nodes
of the heavy paths traversed along the way. Since a root-to-leaf path intersects at most log 𝑛 heavy
paths, the number of such top nodes is 𝑂(log 𝑛). Importantly, these top nodes are visited in bottom-
up order, so their weights form a strictly increasing sequence which allows us to reduce the search
to a predecessor query over a sorted list of weights.
There are multiple standard approaches to predecessor searching over small sets:
A linear scan over the list of 𝑚= 𝑂(log 𝑛) weights offers 𝑂(𝑚) query time and space.
A binary search improves the query time to 𝑂(log 𝑚), still using O(m) space.
However, for O(1) time queries, Bille et al. employ the Fusion Tree data structure.
Instead of storing a such a predecessor solution for each node in the tree, it suffices to only
construct it for each leaf, with each internal node u having a reference to an arbitrary descendant
leaf leaf(u). This reduction makes it easier to analyze the space complexity, because each leaf-root
path contains at most 𝑂(log 𝑛) heavy paths which corresponds to 𝑂(log 𝑛) top-nodes, and the leaf-
count of a tree is bound by O(n), thus resulting in a total space usage of 𝑂(𝑛log 𝑛) for locating the
correct heavy path in O(1) time.
4.3.4 Data structure
We are now ready to combine the data structure and the preprocessing step therefore consist of:
• Constructing a LCA (±1RMQ) data structure for the tree.
• Using in-order traversal to populate size, depth, and choose an arbitrary leaf for each node if
the given tree doesn’t already support these.
• Perform HPD on the tree, storing each path H as a sequence of nodes with corresponding
bitstring B(H), and construct a lookup for each node to their heavy path.
• Build a rank/Select data structure on each bitstring B(H).
• Lastly, for each leaf node, construct a fusion tree over the weights of all top nodes on the path
to the root.
All components except the fusion trees are constructed in O(n) time and space. Since each of the
O(n) leaves stores 𝑂(log 𝑛) top nodes, the total cost of fusion tree construction is 𝑂(𝑛log 𝑛), which
dominates the preprocessing time and space.

4.3.6 Correctness
The correctness of the data structure follows from the guarantees provided by each component:
• The heavy-path decomposition ensures that any path from a node u to the root intersects at
most log 𝑛 heavy paths, each rooted at a top node with non-decreasing weight.
• The predecessor query guarantees that we in constant time can locate the lowest ancestor(u)
on a heavy path H, containing a top node with weight at least k.
• The bitstring encoding of each heavy path ensures that we can, in constant time, identify the
lowest node on the path with weight at least k.
• The LCA check ensures that the selected candidate is actually an ancestor of u.
Together, these components ensure that the algorithm always returns the lowest ancestor of u
whose weight is at least k, as required by the SWA definition.
4.4 𝑂(𝑛) space
Building on the ideas of the 𝑂(𝑛log 𝑛)-space solution, we will now improve the SWA solution to
linear time and space preprocessing. We continue to use the heavy path decomposition of the
input tree and the unary bit-encoded weight differences for constant-time queries within a single
path.
Remember that the earlier use of fusion trees for fast predecessor search at each leaf is the
primary cause for a total space usage of 𝑂(𝑛log 𝑛). The key challenge therefore lies with identifying
the top nodes of heavy paths in O(1) time while keeping the total space linear.
4.4.1 Contracted tree
The first step toward achieving a linear-space solution is to reduce the complexity of navigating
between heavy paths. This is done by constructing the contracted tree, a simplified representation
of the original input tree where each heavy path is collapsed into a single node.
Formally, given the heavy path decomposition of a tree 𝑇, we define a new tree 𝑇𝐶L, called the
contracted tree, where each node in 𝑇𝐶L corresponds to a top node of some heavy path in 𝑇. An
edge exists between two nodes in 𝑇𝐶L if the corresponding heavy paths in 𝑇 are connected by a light
edge. In other words, the contracted tree preserves the light-edge structure of 𝑇, while ignoring
the internal details of each heavy path.
This abstraction preserves all inter-path relationships and does not affect correctness: any
ancestor relationship between heavy paths in 𝑇 is retained in 𝑇𝐶L. Since each node in 𝑇 belongs to
exactly one heavy path, the size of the contracted tree is at most the number of nodes in 𝑇, and
its construction takes 𝑂(𝑛) time.
What the contracted tree enables—beyond simply navigating to parent paths, is efficient access
to the all light edges including children of each heavy path. By eliminating internal heavy-path
structure and reducing the tree to its light-edge skeleton, we obtain a compact form suited for
further decomposition.

4.4.2 Art decomposition
The ART decomposition[14], is a tree partitioning technique designed to break a rooted tree
into simpler components while maintaining efficient support for ancestor-based queries. Given a
parameter 𝜒> 0, the decomposition partitions a rooted tree 𝑇 into a top tree and a set of bottom
trees as follows:
Each node 𝑣 is examined in top-down order. If 𝑣 is the shallowest node in its subtree with at most
𝜒 leaf descendants, it becomes the root of a bottom tree. This bottom tree consists of 𝑣 and all
its descendants. The process continues recursively, and any node not assigned to a bottom tree
becomes part of the top tree.
The decomposition has the following key property:
• Let 𝑇 be a rooted tree with ℓ leaves. Then the ART decomposition of 𝑇 with parameter 𝜒 produces
a top tree with at most 𝑂( ℓ

4.4.3 Data structure
We continue to operate on the same input tree 𝑇 of 𝑛 nodes, augmented with a heavy path
decomposition and weight encoding as described previously. The goal of this construction is to
reduce the memory footprint of the predecessor lookup mechanism, replacing fusion trees with
a bounded number of compact, navigable components.
To achieve this, we first form the contracted tree 𝑇𝑐, where each node represents a heavy path in
the original tree and edges correspond to the light edges connecting those paths. The predecessor
lookup problem is thus reduced to identifying, for a given query SWA(u, k), the lowest ancestor of
𝑢 in 𝑇𝑐 whose associated top node has weight at least 𝑘.
In order to support such queries in constant time without storing a fusion tree per leaf, we
recursively partition 𝑇𝑐 using the ART decomposition. This is done in two steps. First, we apply the
decomposition with a threshold of 𝜒2, producing a top tree and a set of first-level bottom trees,
each containing at most 𝜒2 leaves. These bottom trees are then decomposed a second time with
parameter 𝜒, forming middle trees and final bottom trees, each now of size at most 𝜒, where 𝜒=
𝜖
log 𝑛
log log 𝑛 and 𝜖 is a small positive constant 0 < 𝜖< 1. This results in a tree divided into tree layers as
shown in Figure 9, of Top, Middle, and Bottom trees.
The key observation is that every path from a node to the root in 𝑇𝑐 intersects only a limited
number of trees in each layer. In particular, a path crosses at most 𝑂(log 𝑛
log 𝜒) first-level bottom trees,

5 Implementation
This chapter elaborates on how the SWA problem and its related components have been realized

5.1 Development Method
As mentioned in Section 2, the implementation was developed using the Test-Driven Development
(TDD) principles. This methodology encourages writing tests before the actual implementation,
which proved especially valuable when dealing with intricate data structures such as rank/select,
LCA, and predecessor search.
The typical workflow followed these steps:
1. Define a property or behavior to test (e.g., correct rank result on a bit string).
2. Implement a minimal naive version as a reference.
3. Write tests that compare the optimized version to the naive across a wide input range.
4. Refine the optimized code to match the correctness expectations.
For most data structures used in the final algorithms (excluding decompositions like HPD and
ART), naive versions were implemented to serve as test oracles. These include linear scan
implementations for rank/select queries, LCA computations, and predecessor search.
The correctness of all implementations was verified using FsCheck, a property-based testing
framework for .NET. It allowed for randomized testing using custom tree generators and
properties that encode invariants and expected query results. Performance, on the other hand,
was evaluated using BenchmarkDotNet, with emphasis on both construction and query timings.

5.3.4 Supporting Data Structures
Because all supporting data structures were implemented from scratch during the project, naive
solutions have been implemented to verify the result of the the more optimal versions. The
following algorithms has been implemented to answer their queries using a linear scan for linear-
time with O(1) space overhead. They prioritize simplicity and correctness over efficiency and are
not intended for use in production code. Unlike the main SWA solutions, these structures do not
conform to a shared interface, although they mostly use the same input as their more complex
counterparts.
LCA
The naive LCA algorithm finds the lowest common ancestor of two nodes by first equalizing their
depth in the tree, then traversing both upwards in parallel until they meet. This is guaranteed to
terminate with a correct result due to the tree’s acyclic structure and the uniqueness of the path
from each node to the root.
The method relies only on Parent pointers and operates in 𝑂(ℎ) time, where ℎ is the height of the
tree. It uses no additional memory beyond local variables.
Rank/Select
The bitstring operations are implemented using the BitArray class, a standard .NET collection that
represents a sequence of bits. BitArray provides convenient indexed access to individual bits as
boolean values, without requiring manual masking and shifting operations, making it suitable for
simple linear traversal of individual bits.
• Rank𝛼 is implemented as a loop that counts the number of bits matching a given value 𝛼∈[0, 1]
in the prefix range [0..j].
• Select𝛼 performs a linear scan and returns the index of the 𝑘-th occurrence of a bit set to 𝛼.
Both operations run in O(n) time and use a single pass over the bitstring. Their correctness
is straightforward to reason about: Rank increments a counter per match, while Select counts
matches until the requested rank is reached.

5.5 LCA
The implementation of the LCA data structure is based on the classical Euler tour reduction
combined with a two-level ±1 range minimum query (RMQ) scheme. It supports O(1)-time queries
after linear-time preprocessing, and is used for both versions of the SWA solution to find the
answer candidate 𝑤2, that is the lowest candidate on the heavy path 𝐻𝑤 that is guaranteed to be
an ancestor of the query node.
The construction begins with a in-order traversal of the input tree, during which four arrays are
populated: E, A, R, and V. The E array records the Euler tour of the tree, expressed as a sequence of
node IDs assigned during traversal. Every time a node is entered or re-entered during traversal,
its integer ID is appended to E. Correspondingly, the A array stores the depth of each visited node
at the time of its appearance in E. Because the traversal is in order and moves strictly between

5.6 Heavy path decomposition
The heavy-path decomposition is implemented using a recursive top-down strategy. Starting from
the root, the algorithm constructs a path by repeatedly selecting the heaviest child, and following
it down toward the leaves. Each such path is built iteratively and added to a collection. After a path
is completed, see Listing 3, the algorithm continues recursively on all non-heavy children of the
nodes within that path.
Since the size of each node’s subtree is precomputed, identifying the heavy child of a node can
be done in linear time over its children. Each node is visited exactly twice: once when its parent
considers it for the heavy path, and once when it is itself the root of a recursive decomposition
call. This ensures the total runtime remains linear in the number of nodes.

The HeavyPath object is implemented as a class which stores both the list of nodes on the path in

The construction of this encoding is done in reverse order, starting from the leaf and proceeding
toward the top of the path. For each node, the difference in weight between the current and
previous node is calculated to determine the number of 1-bits that should be inserted before the
next spacer (0-bit). This produces a unary-encoded representation where the number of 1-bits
between spacer bits reflects the weight difference along the path.
To simplify bit manipulation, the array is first filled entirely with 1s. Spacer bits are then explicitly
cleared by masking out specific bit positions. Any unused bits at the end of the final word are also
flipped off again.

5.7 Contracted tree
The contracted tree is a structural abstraction in which each node represents an entire heavy path
from the original tree. Its purpose is to simplify the tree topology by collapsing heavy paths into
single units, connected by light edges.
Construction begins by assigning each heavy path an index and allocating one CTNode per path.
The weight of each contracted node is set to the weight of the topmost node in its corresponding
heavy path. Parent–child relationships are then re-established by following the light edges of the
original tree.
Each contracted node maintains a list of its child indices, a reference to its parent, and the index
of its heaviest child. The result is an array of CTNodes in the same order the heavy paths where
indexed, and this order and ids will carry over till the ART decomposition and tabulation answers.
In effect, the contracted tree remaps the structure of the original node-level tree into a smaller
tree over heavy paths, preserving all connectivity through light edges. This transformation heavily
reduces the depth and branching complexity of subsequent processing steps without loss of
structural information.

A second traversal of the given tree is used to assign each node to a cluster. Each cluster is rooted
at the node where a transition between tiers occurs (e.g., top to middle, or middle to bottom). The
root-id of each node’s current cluster is stored in a separate ClusterRoot[] array, where a value
of −1 is used for nodes belonging to the top tier. The resulting structure allows each node to be
mapped to its enclosing micro-tree for later tabulation.
To avoid dynamic allocation overhead, the tier information and cluster assignments are stored in
flat arrays indexed by node ID. The tiers are represented as a byte-sized enum, and the cluster
roots as simple integers. This representation causes minimal memory overhead and fast lookups
for locating micro-trees.
5.9 Tabulation
To support constant-time SWA queries in small subtrees, all non-top-tier micro-trees identified by
the ART decomposition are preprocessed into a compact global table. Each table entry encodes all
possible queries for a unique micro-tree shape and weight configuration, allowing constant-time
resolution of any local ancestor query that does not escape upward to a higher tier.

5.11 O(N) Space SWA
The second SWA implementation follows the linear-space solution presented in the paper. It retains
the same fundamental query strategy as the 𝑂(𝑛log 𝑛) version. However, it replaces the fusion-
tree predecessor structures for all leaves with a three-layered ART decomposition with tabulation
of answers for the lower two layers, keeping the predecessor structure for leaves in the top-layer.
The first half of construction mirror the previous solution. The LCA, node depths, and heavy-path
decomposition are built identically, and each node is mapped to its heavy path and preprocessed
with a RankSelect structure over its encoded bitstring.
From there, the tree is contracted, where each node corresponds to a full heavy path. This
contracted tree 𝐶𝑇 is then passed through the ART decomposition, as described in Section 5.8,
which classifies each node into one of three tiers (top, middle, or bottom) based on subtree size.
Given both 𝐶𝑇 and ART decomposition, the ArtMicroTreeTable class tabulates all possible answers
for middle and bottom trees as described in Section 5.9.
For nodes in the top tier, fusion trees are constructed in a similar manner to the previous solution.
The contracted nodes simplifying the light edge traversal slightly, as we simply follow the parent-
id of the current node until the value is −1.
The Query(u, k) is similar to before, with the only difference being how to locate 𝐻𝑤. With the
difference highlighted in Listing 12. The algorithm first determines the path id of the current node
𝑢, which is the same id used for lookups in 𝐶𝑇 and ART. This id is used to search through at most
three ART layers.

ART layer via the parent of the micro-tree root. This is similar to how top-nodes of a heavy path
was traversed earlier, and it continues until either a result is found or the top tier is reached.

The OOP version stores each node as a standalone object, containing fields for its weight,
size, parent reference, and list of children. Relationships between nodes are encoded via direct
references. Which makes recursive operations and tree manipulation intuitive, can be easier
to think about, and was used as the default representation throughout development. This
representation however has some implementation details that can be a drawback for performance
when working with larger data sizes.
Even though TreeNode objects are created right after each other, and we thus expect them to be
created in the same section of memory, this isn’t a guarantee, and it isn’t uncommon for individual
objects to be spread across memory. This can make parent and child traversal take longer, since
these are indirect references to other objects, and can lead to poor cache line usage that often
retrieves more irrelevant data than useful.
The layout also introduces padding inefficiencies, making the individual object take up more space
than it actually uses. For example an object of two fields, a 64-bit reference and a 32-bit field like
Size, will consume two slots of 64-bits due to alignment constraints.
The larger an object is, the faster it will be pushed out of the smaller cache and into the larger ram,
which might have an impact for the kind of micro benchmarks we will be doing. This last issue will
probably only be an issue for the construction since the query doesn’t revisit nodes, however it is
suboptimal that we pay to load values that are unnecessary for the current query. For example
loading Weight or Size, when only looking for the parent reference.
In contrast, the DOD version represents the tree using parallel arrays, where each array
corresponds to a single field such as Weight, Size, Depth, or Parent. A node is represented via an
integer index that is shared across the value arrays. This approach doesn’t completely solve all the
issues just pointed out, however it greatly improves upon most.

Effectively the tree is accessed via 64-bit reference, and each field is itself a 64-bit reference leading

Because the layout for values is fixed and sequential in memory, traversal is better aligned with
CPU cache behavior. The lists are stored in-order, so most nodes are stored in a manner that clumps
related nodes, with narrower trees making it more likely that the parent is just before and children
is just after the current index. This enables the cache-line to return more useful neighboring values
when accessed, especially because we only access one specific value and the alignment issue
gets completely solved. Each array is exactly sizeof(value) · 𝑛, where 𝑛 is the number of values, and
therefore the number of nodes in the tree.
Because the node layout is fixed and sequential, many dictionaries used in the OOP version to
map TreeNode references to auxiliary metadata can be replaced with integer-indexed arrays. This
reduces not only memory overhead, with dictionaries usually needing 2-3 times more space, but
also avoids the computational cost of dictionary lookups.
This common indexing also allows significant reuse across subcomponents. For instance, the same
order of nodes is used when building the Euler tour and corresponding V, A, E, and R arrays for
the LCA structure. So it makes sense to precompute these before the LCA structure, as well as the
depth and leaf mapping, since we need to traverse the tree anyways to translate the OOP tree-
structure to DOD.
This does mean that we have two instances of the tree’s weight size, parent, and children fields for
this optimization, but the latter two along any other reference are half sized, and the reduction in
overhead for dictionaries, more than balance this added complexity.
Fusion tree, Heavy path decomposition, and LCA have all been slightly modified to match this
change, mostly changing any stored array of TreeNodes with int, while contracted tree, ART
decomposition, and tabulation has been impacted to a lesser degree because they already use
another id order.
5.13 Performance-Aware Programming
Throughout the project, a number of general implementation strategies were adopted to improve
efficiency and readability. These choices span data structure usage, memory management,
allocation patterns, and platform constraints specific to C#.
Arrays are used whenever the size of a collection is known ahead of time, as this collection type
simply store each value in continuous memory along a starting pointer, the block size, and the
amount of values, here referred to as the length. Indexing requires no additional work, and an
example of usage are the nodes of a heavy path.
When the number of elements is unknown in advance, the List class is used instead. This is an
abstraction of the classic Array that adds dynamic resizing capabilities to the collection. Because
the underlying structure is still an array, there is little to no query times overhead, however the
space usage is usually larger, and insertions are only O(1) amortized because of the resizing
behavior of doubling the array size and copying each item over. Because of the latter the List
doesn’t have a Length field, but instead differentiates between Capacity and Count.
For mappings where the keys are not in a known order, naturally indexable or sparce, dictionaries
are used. Most of these map object references to other objects like the mapping from TreeNodes
to which Heavy Path they belong to, where multiple nodes map to the same path. To avoid identity-
based mismatch, all such dictionaries use their reference instance for the hashcode instead of
fields. This was done mostly to ensure no object where shallow-copied into a new memory address,
but also because it for this project made more sense to map the actual object and not just a
objects with equal field values. For example node a and b both have equal size, weight, and parent,
however it is very unlikely their children will look identical, and they will always belong to different
heavy paths.

5.15 Benchmarks
Benchmarks where carried out using the BenchmarkDotNet package, a well-established
framework for microbenchmarking in .NET. All implemented benchmarks inherit from a shared
abstract base class decorated with the [MemoryDiagnoser] and [CsvExporter] attributes. This
enables memory usage diagnostics and ensures consistent output logging across all benchmark
results. Each subclass defines specific benchmark targets, such as construction time or query
performance for SWA variants or sub data structures.

6 Experiments
The goal of this chapter is to evaluate the correctness and performance of the implemented
SWA solutions and their supporting data structures. A major goal for the project has focused
on comparing theoretical design with empirical validation through extensive testing and
benchmarking. These experiments aim to document how each part of the solution behaves in
practice and to verify that the theoretical claims hold under realistic input sizes and shapes.
6.1 Background
Correctness will be verified through a combination of naive comparisons and property-based
validation. Each data structure introduced in this project is covered by their own dedicated tests.
These tests check for consistency against simpler implementations, ensure invariants hold across
all operations, and verify the correctness of individual components in isolation. A numbered list of
these tests is included in Section 7.1.
Performance evaluation is carried out using benchmarks, focusing on three key metrics:
• Query Time: Measures the average time taken to answer an SWA(u, k) query.
• Construction Time: Measures how long it takes to preprocess a given input tree.
• Memory Usage: Measured during construction to capture the overhead of auxiliary structures,
since queries do not allocate memory.
These metrics are central to the goals of the project, which is to analyze and compare different
solutions to the SWA problem from both a theoretical and practical perspective. Measuring both
time and space complexity at different stages of execution ensures that the full performance
profile of each approach is understood.

6.2.2 Benchmarking
We are interested in Benchmarking the two common performance dimensions of each SWA
implementation and its supporting data structures, to evaluate the combined and individual
practical efficiency of each implemented data structure:
• Construction time, including memory usage.
• Query time.
The two attributes are benchmarked independently from each other, using BenchmarkDotNet
as described in Section 5.15. Each data structure evaluates each of its implementations one at
a time, on identical query and construction parameters. BenchmarkDotNet run each test fully
separated from one another, so the identical setup is guaranteed by setting a fixed seed value.
The [GlobalSetup], where all non-measured pre-computation is performed, is run once per
unique parameter combination, which is used for the query benchmarks to make the current
implementation an input and thereby save the memory overhead of having each implementation
initialized at the same time. This allows each implementation to grow as large as the test machine
has ram, and isn’t an issue for the construction benchmarks, as only the tree or similar input is
needed.
Most benchmarks take a tree as input, and since the tree is constructed of both size and shape
parameters, these two can have a huge effect on the outcome. The shape of the tree is especially
prone to impacting the result on both query and construction. We will for example expect a huge
difference in query performance for the linear scan SWA solution 4.1.1 on a tree of depth 𝑑= 2,
where all nodes are direct children of root, to the chain 𝑑= 𝑛, where each node have exactly 1 child.
Size’s effect is also measurable, even for the O(1) time queries, which can happen for different
reasons, like cache misses being more common for larger structures, or the new tree having a
better/worse distribution. An example of the latter could be a tree, where the weight of each is 1,
as this would fail almost all SWA queries in the bounds check.
To have a proper point of reference to evaluate for the theoretical efficiency of each data
structure, each benchmark will try and simulate the worst possible case by choosing a set of input
parameters that will have to work over the largest amount of data. For the SWA problem, this will

6.2.3 Scalability and Limits
The last group of tests focuses on testing the limit of each implementation, stability at large scales,
and whether we can see expected space usage trends.
For test of stability, each core data structure was unit-tested individually on a tree with exactly
10 million nodes. These tests were not intended to benchmark performance but to catch silent
failures, such as:
• Overflow in size counters or internal indices.
• Incorrect behavior of structures where smaller fixed-size fields where used, for example ushort
for local tabulation indices.
• Stack overflows caused by recursive implementations or stackalloc usage.
These tests revealed practical limitations around stack usage, particularly for deeply nested trees.
Some instances of Span and stackalloc were revised or removed for this reason.
In theory, all SWA variants can scale to trees of size int.MaxValue (231 ≈2.1 × 109), constrained only
by available memory. This is due to the arrays needing a 32bit integer for indexing, and c# supports
single object up to 2GB[15] by default. These limitations can be bypassed by, using biginteger for
node weights and sizes, using multiple arrays to elongate the collection, and enable the semi-
unsafe gcAllowVeryLargeObjects flag. The latter being potentially unsafe because it breaks an
assumption other people had when writing their libraries.
On the test system, however, construction consistently failed when run on a little more than 10
million nodes, due to RAM exhaustion.
While the former benchmarks tested that all sub data structures are linear in regards to n, some
are more gradual than others. For example the HPD simply store each node differently than the
given tree, scaling 1:1 with the amount of nodes, while others allocate blocks to store the data.
To better understand how sub data structures grow, a set of microbenchmarks was run on small
trees, allowing us to observe their memory footprint and behavior across boundary cases.
• The JumpPointers data structure is built as a binary lifting, with each node storing jump
references to its 2𝑖-th ancestor. As a result, the number of jump levels increases only when the
input tree size exceeds the next power of two. Between these thresholds, the jump table remains
stable in size. Thus, we expect discrete jumps in memory usage at sizes like 16, 32, 64, etc., with
no increases in between.
• The rank/select encoding is based on a bitstring backed by a block-aligned ulong[] array.
Additional auxiliary structures are added per block or superblock. Since each block holds 64 bits,
we expect storage to increase every time the bitstring length crosses a multiple of 64. This leads
to stepwise increases in size and memory footprint as the encoded structure grows.
48

• The LCA structure is built in two layers: small, block-local RMQ lookup tables, and a sparse table
over block minima. The number of RMQ blocks increases with the size of the Euler tour, and
the number of levels in the sparse table grows logarithmically with the number of blocks. While
growth is more gradual than in the previous cases, we still expect non-smooth transitions when
a new block or sparse-table level is introduced.

7 Discussion
This project originally set out to investigate the SWA problem with the goal of understanding,
implementing, and possibly improving on one or more aspects. A large part of this understanding
was implementing the state of the art solution presented in the paper of Bille et al. and comparing
it to simpler approaches.
This led to the development and testing of naive baselines and multiple variants of the two optimal
solutions presented in the paper, as well as all supporting structures. The outcome is a set of
implementations, each with interchangeable variants, ready for further experimentation, as well
as benchmarks that outline their practical efficiency.
The findings from this process provide insights into the technical viability, real-world performance,
and potential of the solution, as well as the methodology used to refine its implementation that
can be used for similar performance critic solutions in the future. This section will evaluate these
outcomes against the original goals, contextualize it, and describe limitations and possible future
directions of the work.
7.1 Evaluation of Aims and Outcomes
The project set out with the aim of understanding, implementing, and possibly improving upon
the SWA solution presented by Bille et al. This began with a direct implementation of the algorithm
in C#, with the goal of remaining faithful to the theoretical description while addressing practical
considerations such as language constraints and memory behavior.
To ensure correctness throughout, all queryable data structures were accompanied by at least
one naive baseline implementation. This included not only the SWA solutions themselves, but also
prerequisite components such as Rank/Select, LCA, and Fusion Trees. These baselines served as
reference points during testing and made it possible to verify the behavior of each optimal variant
in isolation before integrating them into the complete solutions.
The work extended beyond a straightforward reproduction of the paper’s algorithms. Several
variants of both the papers 𝑂(𝑛log 𝑛) space and 𝑂(𝑛) space solutions were implemented, including
two alternative methods for selecting the answer candidate during queries. These refinements
originated from the design stage as attempts to simplify parts of the paper’s approach without
changing its asymptotic properties. Additionally, the tree representation was implemented in both
a reference based object-oriented form and a Data Oriented Design (DOD) layout, with the latter
aimed at improving cache usage and reducing memory overhead in query-heavy workloads.
All supporting structures used in the optimal solutions, were implemented from scratch. This
was in part due to a lack of standalone packages online ready for use, and because it allowed
for full control over their construction, memory layout, and integration. Ensuring that the
implementations were made with the exact data representation in mind, for a fairer performance
comparison to the naive baselines.
Testing combined edge-case unit tests with a property-based approach to validate correctness
under a wide variety of tree shapes, sizes, and weight assignments. Optimized components were
checked against their naive counterparts, and the complete SWA implementations were verified
against all other versions. This process revealed subtle issues early and supported incremental
refinement of the implementations.
Benchmarks measured both preprocessing and query performance under different input
distributions, highlighting the influence of constant factors, memory layout, garbage collection,

and CPU cache behavior. Based on these results, adjustments where made to data layout and
query logic, reducing overhead without altering the practical complexity of the solution.
7.2 limitations and trade-offs
While the implementations successfully reproduce and extend the solutions described by Bille et
al., several limitations and trade-offs emerged during the process. Some arise from language-
specific constraints, while others reflect design choices and the scope of the project.
C#‘s managed memory model imposes certain constraints. Garbage collection can introduce
unpredictable pauses during preprocessing, and some compile time optimizations available in
lower-level languages are missing or unpredictable due to C#’s Just In Time (JIT) compiler. The
default stack size also prevents deep recursion, requiring iterative implementations for tree
traversals. In many cases this just moves the data onto the slower heap in a stack-object. Although
tail recursion is possible in C#, it is less reliable than in the functional dotnet language F#, making
it unreliable for performance-critical paths.
Two distinct tree representations were implemented, an object-oriented (OOP) version and a DOD
version. The OOP representation is conceptually simpler, with nodes holding references to their
direct parents and children, and was therefore used as the baseline design. However, its structure
incurs overhead from reference indirection and object alignment. The DOD layout replaces all
references with integer indices and stores each field in a contiguous array. This improves cache
locality and saves significant memory, as only a single object-of-arrays instance is passed around
rather than many individual node objects.
Some data structures introduced high constant-factor costs. In particular, the Fusion Tree used
in the 𝑂(𝑛log 𝑛) space solution proved expensive in both preprocessing and queries. To address
this, it was replaced with a binary search over the relevant node set, requiring at most log2(31) ≈
5 comparisons. This change reduced overhead considerably for both query and preprocessing,
and the final optimized 𝑂(𝑛log 𝑛) space solution exhibited almost no visible superlinear scaling
in preprocessing benchmarks due to the small memory footprint. With current implementation it
would require much larger data sizes before the non-linear memory increase becomes noticeable.
However this is mostly due to the 𝑂(𝑛) decomposition and sub structures using comparably more
ram. It is possible the implementation still allows for reductions in these areas, as the focus was
on improving obvious profiled hotspots across the full solution.
Benchmarking did not focus on best-case scenarios, as all queries begin with a check to see
if the given node already satisfies the weight constraint. And best-case performance in general
offers no guarantees for other inputs. That said, it was notable that the best-performing SWA
implementation (Paper1_DOD)) remained competitive with naive solutions even on very small
trees of 100 nodes for most shapes. The expectations were that its advantages would only appear
at much larger scales due to the higher overhead of the much more advanced structure. Specific
to the BBT shaped trees with a very low height, the naive solutions had it beat for these lower
sizes. And DirectShortcuts approach was still the fastest, simply performing two lookups for the
answer, but its memory usage grows too quickly to be practical for larger trees.
The scope of benchmarking was limited to a single hardware setup, a single .NET version, and a
single runtime configuration. Results may vary on other architectures or runtime environments.
While a variety of tree shapes were tested, the randomly generated tree-shape with up to three
children per node, were used to simulate more realistic structures. However these random cases
were not suitable for worst-case analysis, as they do not maximize any specific structural property
of the input.
Finally, the implementations support only the two tree representations developed in this project.
No effort were made to support other known graph representations like the adjacency matrix
or list. The use of interfaces and abstraction was kept to a minimum by choice, as these often

though such cases could allow for more specialized and efficient solutions.
7.3 Future work
Several directions remain for extending and refining the work presented here, both in terms of
technical optimization and practical application.
The profiling during this project relied on the allocation and CPU usage tools available in Visual
Studio. While these were effective in identifying major hotspots, it would be valuable to use more
advanced tooling such as the Windows Performance Toolkit[16] to obtain detailed hardware-level
metrics, particularly cache miss rates. This could guide further improvements in both space and
time efficiency of the implementations, especially in the memory-sensitive parts of the solutions.
The DOD integration with the 𝑂(𝑛)-space solution presents another opportunity for improvement.
The current contracted tree, used for both ART decomposition and tabulation, constructs its own
index-based representation independently of the DOD tree. Tighter integration of these could
eliminate redundant transformations and further improve cache locality, which would benefit both
preprocessing and query times.
Exploring alternative tree representations could also be worthwhile. While the object-oriented
(OOP) and DOD layouts have proven effective, other formats—such as adjacency lists, adjacency
matrices, or compressed encodings may offer different trade-offs in terms of memory usage and
traversal speed.
A hybrid approach is another promising direction. Similar to how some standard library
sorting algorithms switch between strategies depending on input size or ordering, an SWA
implementation could analyze the input tree’s structure and select the most appropriate method.
This could make better use of naive, specialized, or fully optimized approaches depending on the
situation.
The DOD layout aligns memory contiguously, making it a good candidate for SIMD acceleration.
Introducing vectorized instructions for common operations, particularly in tight query loops,
could yield meaningful speed-ups without major structural changes. The compiler might already
do these in simple loops that transform a list independently, however this isn’t a guarantee, and
manual implementation might spot applications the compiler never would consider.
An obvious next step is to test the structures in a practical application. For instance in string
applications, as mentioned in the introduction and Bille et al.‘s paper. Comparing these to existing
implementations would give insight into integration costs, bottlenecks, and benefits under
realistic workloads.
Finally, broader platform and runtime testing would show if the results where affected by
the test machine, that could support instructions that prefer some operations over others. All
results presented here were obtained on a single hardware platform and a single .NET runtime
configuration. Extending the benchmarks to other architectures, .NET versions, and runtime
environments would help assess broader real world practicality. As part of this, re-implementing
the solutions in a lower level language such as C++ or Rust, could provide additional insights,
particularly around tighter memory control, zero-cost abstractions, and deterministic deallocation.

8 Conclusion
This project set out to investigate the SWA problem with the aim of understanding, implementing,
and comparing theoretical and practical efficiency of the solution presented by Bille et al. The
work has reproduced both the 𝑂(𝑛log 𝑛) space and 𝑂(𝑛) space solutions described in the paper[2]
independently, validated their theoretical correctness and complexity, and compared them to
simpler baseline approaches.
A complete set of naive and optimal implementations was produced for every queryable
component, including sub structures such as LCA. These served as both functional building blocks
and correctness oracle during development and testing. Unit tests, incorporating targeted edge
cases and specific examples, were applied alongside property-based testing on generated input,
to confirm that query results of the optimal variants matched the naive counterparts across a wide
range of tree shapes, sizes, and weight assignments. Additional expected properties where also
tested, like the weight expectations of the generated trees, the size of bitstrings, or the unique
node representations of tree decompositions.
Benchmarking confirmed that the performance matched the theoretical bounds under the tested
conditions, while also revealing the influence of constant factors, memory layout, and language-
specific behavior. Although no theoretically superior data structure was designed, either lowering
the complexity or allowing for different input, several practical refinements were made.
Two simplified answer candidate selection strategies where designed and tested, the introduction
of a DOD tree representation for improved cache locality and memory efficiency for the
implementation, and targeted profiling to reduce constant-factor overhead to get the best
translation of design to implementation.
The resulting implementations meet the project’s goals, having been thoroughly validated and
structured to support further experimentation. The work has confirmed the theoretical claims of
Bille et al. under the tested conditions and demonstrated their solutions’ behavior in a practical
setting. An unexpected outcome was that the best-performing optimized variant remained
competitive with naive baselines even on relatively small trees, highlighting the difficulty of
guessing the impact of constant factors of the design or real world system, and that careful
implementation can narrow the performance gap in cases where overhead was expected to
dominate.
While the project fulfilled its goals, there remains considerable scope for further exploration,
ranging from improving the implementation with more profiling and alternative tree
representations to integration into practical applications such as suffix trees or tries.

9 Demo
The contents of the accompanying zip archive include a copy of the report, the code as a Visual

The codebase is organized into three projects:
• Benchmarks – performance tests and profiling benchmarks.
• Core – the core SWA implementations and supporting structures.
• Test – correctness tests for the SWA solutions and supporting data structures.
Each folder can be opened and run directly via the command line terminal.
Before running, install the required packages by navigating to the root folder of each project and
restoring dependencies:
dotnet restore
The project uses these packages:
• BenchmarkDotNet
• BenchmarkDotNetVisualizer
• xunit
• FsCheck.Xunit
• Microsoft.NET.Test.Sdk
9.1 Running the projects
Running tests
To run all correctness tests from the Test project:
cd Test
dotnet test -c Release
To run only a specific test class:
dotnet test -c Release --filter "FullyQualifiedName~ClassName"
dotnet test -c Release --filter "Test.Tests.LCA_Tests"
To run only a specific method:
dotnet test -c Release --filter "FullyQualifiedName~ClassName.MethodName"
dotnet test -c Release --filter "Test.Tests.SWA_Tests.IsQueryCorrectAll"
When using Visual Studio or JetBrains Rider, tests can also be run from the Test Explorer pane.
It is recommended to only run one test group at a time. The global random number generator
may cause seed-based tests to fail when multiple groups run concurrently.
Running benchmarks
From within the Benchmarks folder, pass an integer argument from 0 to 8 to choose the
benchmark group:
cd Benchmarks
dotnet run -c Release -- 1
