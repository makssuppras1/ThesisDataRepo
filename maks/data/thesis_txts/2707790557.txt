Abstract (English)
Software development is a growing industry, and the increasing number of software
developers has led to code and programs being shared like never before. Developers
can use code libraries to streamline the development phase, as they alleviate develop-
ers from having to write code already written by others. However, as code libraries
are easily downloaded from the internet, developers often neglect the consideration
of whether a library is benign or malicious.
Code libraries can be linked into a program either statically or dynamically. Dynam-
ically linked libraries become dependencies in a program as they are needed for the
program to run. As a result, programs can both intentionally and unintentionally
depend on a malicious library written by nefarious actors, making it a signiﬁcant
risk to run a program downloaded from an unknown source.
The main goal of this thesis is to create a tool capable of ﬁnding the chain of
dependencies when given an executable program. Such executable programs are
either Windows or Linux compatible and follow the PE or ELF formats. The typical
’.exe’ and ’.dll’ ﬁles on Windows use PE, and executable ﬁles and ’.so’ ﬁles on Linux
primarily use ELF. The tool can aid in deciding whether a program is safe to run or
not.
Before creating such a tool, the ELF and PE formats will undergo a detailed
analysis that helps understand the formats and their parts essential to linking and
dependency identiﬁcation. The analysis phase will also consider attacks known to
exploit dependencies in some way.
Two tools were created during the thesis; one to run on Windows and analyze PE
ﬁles the other to run on Linux and analyze ELF ﬁles. Both tools can get headers
and sections from the executable ﬁles and ﬁnd the chain of dependencies.

Chapter 1
Introduction
Programming and software development have seen a signiﬁcant increase in popularity
and the number of people practicing it. The number of software developers in 2021
was estimated to be 26,9 million people worldwide, a number that likely grows to 45
million when 2030 comes knocking[1].
It has, without a doubt, led to a series of beneﬁts, increasing both eﬃciency and
skill in the ﬁeld. Productivity worldwide is booming; the Google Play and Apple
App Store have a combined 7.3 million apps and games in 2021 as opposed to 190
thousand in 2009[2]. The number of downloads at these two online stores is even
more mind-blowing; 144 billion in 2021 alone[3]!
The example above is mainly related to private people, but the tendency in diﬀerent
industries is the same. More and more software exists that can help improve and
streamline the business process at companies. This increased tendency makes a
supply chain attack an attractive attack vector for malicious actors. In addition,
smaller-scale attacks exist trying to leverage the same logic. For example, 11 malicious
python libraries were found distributed via the popular python package installer,
pip[4]. Another showed 9 million Android phones running malicious apps stealing
user and phone data[5].
The prime example of exploiting supply chain is the SolarWinds attack. Malicious
actors broke into SolarWinds and made malicious updates to a piece of software named
Orion. In total, 18000 customers downloaded the update and thereby malicious code.
The malicious code was found in a dll; "SolarWinds.Orion.Core.BusinessLayer.dll"[6].
The BusinessLayer.dll is a dependency in the Orion software. All software has such
dependencies on Windows it is ’.dll’-ﬁles and, on Linux, it is ’.so’-ﬁles.
The purpose of this paper is to uncover a method for identifying dependencies in
executable software. Ultimately, ﬁnding the chain of dependencies by ﬁnding the
dependencies of dependencies.
Besides uncovering a method, a tool capable of doing so will be developed. Such a
tool could help computer scientists check software before deploying it in production
environments, giving some sense of security about the downloaded software. Devel-
oping such a tool will be the primary goal of the thesis.
The identiﬁcation of dependencies illuminates the inner working of the software and

Chapter 2
Background
In order to fulﬁll the objective of identifying dependencies in executable programs, it
must be clear how executable programs are created and how dependencies are used
within them. The origin is, of course, programming and the process of going from,
e.g., a ".c" source ﬁle to a ﬁle that can run on a machine.
The above raises further questions on what a dependency is and how it can be used
and retrieved. Usually, dependencies are code-libraries, but they must be fetched
from somewhere.
Package managers or the internet, in general, provide such code libraries and ex-
ecutable programs. Blindly trusting a third party, which package managers and
the internet is, undoubtedly creates attack vectors for malicious actors to exploit.
Therefore, the type of attacks seen in the past related to dependencies must also be
covered.
This chapter will touch upon the above questions and try to create common grounds
for the further exploration of dependencies in executable programs and how to exploit
them.
2.1
Compiler Toolchain
The compiler toolchain explains how executable programs are created and loaded
from source code ﬁles into memory. Source code can stem from many diﬀerent
programming languages such as Java, C#, c++, c, Ruby and so on.
The compiler toolchain identiﬁes the tools commonly connected with the process
above. An overview of the toolchain can be found in ﬁg. 2.1. Before diving into the
compiler toolchain, a computer’s information and inner workings must be in place.
The CPU is responsible for executing programs in the machine. The CPU or proces-
sor can only understand bits, which go through speciﬁc circuits. These bits come in
a speciﬁc order understandable by the processor and are known as machine code[7].
For each type of processor, an instruction set exists, which the developers of it create.
An instruction set architecture describes the instructions, data types, and registers
supported by the architecture[8]. An instruction set deﬁnes the translation from
machine code to machine instructions. Machine instructions are more manageable
for humans, as they consist of opcode and operands made of numbers and letters.
The opcode is the instructions to perform, e.g., ’add’ or ’sub,’ for addition and sub-

traction. ’Add’ and ’sub’ is known as mnemonics and make the machine instructions
understandable for humans. The operands are the arguments to the instruction.
Operands can be registers, addresses, or actual values, such as integers, ﬂoats, chars.
The machine instruction can be translated to machine code by the assembler. It can
take machine instructions and create the corresponding machine code. A developer
can create a program in assembly language using the machine instructions of a given
instruction set. Assembly is the lowest level of development a programmer can do.
However, assembly language is awkward to work with, as it has no structure, and one
must remember many mnemonics. The inﬂexibility makes it an undesirable language
for programmers in most cases. This paved the way for higher-level languages to
make it easier for humans to create programs. A programmer would have to learn
the syntax and semantics of the programming language, but this would be easier
to do than learn and remember machine instruction or assembly language because
higher-level languages are more similar to the languages we speak.

When programmers develop a program in languages such as c, c++, c#, and java,
the ﬁles they create are known as source code. A compiler takes a source code ﬁle as
input and generates an object ﬁle. In order to generate object ﬁles, the compiler goes

The deﬁnition of an object ﬁle is:
A computer program which has been translated into machine language
by a compiler and assembler, but not yet linked into an executable pro-

The image ﬁle is ready to be loaded into memory by the loader and, after that, be
executed. Common for object ﬁles of both formats are they are stored on disk[13].
2.1.1
Linker & Loader
The end game of the toolchain is the linker and the loader. Their primary tasks
overlap, but they perform the tasks mentioned below.
As stated previously, high-level programming languages allow programmers to code
in a language closer to what we speak. This abstraction is what the linker and loader
must resolve. They take a symbol and bind it to a speciﬁc address or an oﬀset, which
can be used to get the exact address.
Linkers and loaders perform three main jobs; program loading, relocation, and symbol
resolution. A rule of thumb is that loaders take care of program loading, linkers
take care of symbol resolution, and both can handle relocation. There are numerous
examples where the lines between linkers and loaders are blurry, so it is not worth the
time to discuss their diﬀerences. Instead, the tasks they perform will be listed[14].
Program Loading
It takes the program from secondary storage and loads it into
main memory, ready to run. This task can be straightforward because the program
is just copied from disk to main memory. In other cases, the program loading must
handle storage allocation or virtual memory and virtual addresses[14].
Relocation
When the compiler and assembler create an object ﬁle, the load address
is expected to be 0. Only one program can be loaded at address 0, which is usually
the kernel, so executable ﬁles are rarely loaded at this address. In order to resolve
the problem of using an inaccessible load address, relocation is invoked[14].
Symbol Resolution
Symbol resolution is essential when dealing with libraries
or multiple source-code ﬁles. One ﬁle often uses a variable or function deﬁned in
another ﬁle. This usage happens through symbols. Symbol resolution ﬁnds symbols

in an object ﬁle deﬁned in another and patches the code. First, the deﬁnition must
be found, and the address noted. Then the call to the symbol must be patched, so it
calls the previously noted address[14].
As we will see next, there are two types of libraries that can be used; static and
dynamic. The linking process for each of these is slightly diﬀerent, but it will not be
elaborated further until the next chapter.
The above presentation of the compiler toolchain is far from comprehensive. Many
books have been written on each of the tools presented. Nevertheless, it has become
clear that linking, relocation, and symbol resolution are to be investigated further to
know how to identify dependencies.
2.2
Libraries
As stated above libraries are extensively used when coding. It has already been
mentioned that two types of libraries exist; static and dynamic. They both have
their pros and cons.
When using static libraries the libraries will be a part of the ﬁnal executable. This
makes the executable more portable and adaptable at the cost of the size of the
program. In fact static libraries implies that the same code will be stored multiple
times, since some libraries are often used[15][16]. The prime example is the "printf"
function, which is frequently used when programming in c. If it was part of a static
library the printf code snippet would appear in memory once for each executable
using it along with all other functions and variables from the library.
On the other hand we have dynamic libraries, which are also known as shared
libraries. A shared library will be loaded in memory only once, the ﬁrst time an
executable needs it. All executables using this library would then at load time point
to the address containing that piece of code, either a function or variable and use it
as if it was part of the executable itself[16]. In the next chapter ELF and PE will be
investigated in depth and shine a light on how exactly the dynamic linking takes
place in the two formats.
2.3
Package Managers
In all its essence, package managers are a way of distributing and fetching software.
They come in all shapes, and they exist for many diﬀerent use-cases. Some examples
of package managers used by millions of people worldwide are the App Store, Google
Play, Steam, Amazon Appstore, Xbox Live, and PlayStation Store. These are pri-
marily used to distribute games and applications, and the users do not necessarily
need to have a background in programming or computer science.
Some of the more known package managers are apt, npm, pip, and NuGet in computer
science. Apt is the advanced package tool used on Debian-based Linux systems[17].
Npm is the node package manager and currently the most signiﬁcant software li-

3.1
ELF
ELF is an abbreviation of ’Executable and Linking Format’. In the late ’90s, it
was standardized as the binary ﬁle format across Unix-based platforms. The group
responsible for this was called ’86Open’[39].
ELF was initially described in the ’Portable Formats Speciﬁcation’ published by TIS
(Tool Interface Standard) in 1993.
A few years later, in 1995, TIS published a speciﬁcation for the Executable and Linking
Format[12]. This was the format that ’86open’ later chose to be the standardized
binary ﬁle format for Unix-based systems.
Some of the typical ﬁle extensions that follows the ELF format are; .bin, .elf, .o and
.so.
Files that follow the ELF format can be of three types; a relocatable ﬁle, an executable
ﬁle, or a shared object ﬁle[12].
Relocatable ﬁles can be combined with other ELF ﬁles to create an executable or
shared object ﬁle. The creation happens through the process of linking.
A linker uses shared object ﬁles with relocatable ﬁles or other shared object ﬁles to
create a new object ﬁle. Alternatively, a dynamic linker can use them the ﬁrst time
an executable ﬁle needs that shared object ﬁle during execution[12]. As relocatable
ﬁles are not executable, they will not be considered in this paper. However, the

standard executable ﬁles.
There is a fourth ELF type; core dumps. They will be omitted from the analysis.

Fixing relocation in static linking is pretty straight forward and works as explained
above. When the static linker has patched an entry in the relocation table, the
entry will be removed. The static linker will always know where symbols are located
in diﬀerent object ﬁles, and that will not change after the static linking process is
ﬁnished.
Some relocations depend on shared libraries. The static linker will leave such entries
for the dynamic linker to ﬁx at runtime.
If the dynamic linker worked exactly as the static linker, the patching process would
cause trouble in the future. A shared library does not have a certain space in memory.
It is written as Position Independent Code (PIC). If relocations to such shared
libraries were ﬁxed and the entries in the relocation table removed afterwards, the
executable would work, but the next time it is to run, the shared library might be in
another place in memory. The links to the symbols would then be broken and the
executable would not be able to run.
A level of abstraction is needed to ﬁx it. ELF uses the sections Global Oﬀset Table
(GOT) and the Procedure Linkage Table (PLT).
Relocation needed to be done by the dynamic linker will point into one of these
sections. The address of the symbol in the shared library will therefore go into one
of these sections. The entry in the relocation table will not be removed. In this way,
the dynamic linker must ﬁx the relocations each time the executable is started.
The GOT and PLT are almost the same. The GOT holds the oﬀsets to external
variables, why the PLT holds the oﬀsets to external functions.
3.1.5
Linking in ELF
Linking takes object ﬁles (the ﬁle extension .o) and library ﬁles (ﬁle extension .a)
and links them together into an executable ﬁle. The library ﬁles themselves are also
created from object ﬁles and linked to other libraries. The job of linking such ﬁles
together is static linking. There is also dynamic linking, as already mentioned. In
Linux, the dynamic libraries are known as shared objects, and ﬁles of this type have
the ﬁle extension .so.
All this and how static and dynamic linking is performed on Linux will be discussed
below. The discussion will ﬁnd its base in the GNU C Compiler (GCC), one of the
most used Unix-based compilers for C programs. Along with GCC, some built-in
Linux tools will also be part of the description.
3.1.5.1
Static Linking
If one wants to create a library, some source code ﬁles must ﬁrst be developed. These
can then be compiled with GCC to object ﬁles. The object ﬁles can then be used as

input to the Linux utility ’Archiver’, which can create the library ﬁle, containing the
code from the source code ﬁles. The process can be seen in ﬁg. 3.2[47]:
Figure 3.2: The process of creating static libraries[47]
GCC can link library ﬁles and object ﬁles to create an executable ELF ﬁle.
The executable will contain all the needed code, functions, and variables and thus
makes for better portability across systems. We know that the linking view is based
on sections, and the execution view is based on segments. The linker’s job is to
take the sections and turn them into segments. Sections with similar properties
are usually merged and put in the same segment. An example of this is the ’.text’

which is of type ’LOAD’ with reading and execution rights.
The linker also has to perform relocation and symbol resolution to correct all relative
virtual addresses.
Object ﬁles and libraries that go into the linker are not an executable’s dependencies
because the executable holds all functions and variables from those ﬁles, as already
discovered. Static libraries are interesting but not in the process of dependency
identiﬁcation.
3.1.5.2
Dynamic Linking
On Linux, it is the utility ’ld.so’ or ’ld-linux.so’ that handles the process of dynamic
linking. It has to look through the executable if any shared libraries are used. The
dynamic section is used for that purpose. If so, it must point the parts of the
executable relying on shared libraries to the actual address, where the function or
variable is stored. There are cases where the shared library has not been loaded
into memory. In such cases, the dynamic linker is responsible for ﬁnding the shared
library and loading it into memory[48].
It is essential to say that shared libraries also follow the ELF format. The type of
such shared ﬁles appears in the ELF header as ET_DYN.
The standard linker leaves a clue for the dynamic linker to ease the process of dynamic
linking. It lets the dynamic linker know which shared libraries it must use.[45] The
manual page of the Linux linker ’ld.so’ gives much helpful information on how to
look for shared libraries. The ﬁrst thing it does is look for slashes in the dependency
name. It has already been covered how to retrieve the name of the shared object. If

PE
PE is the Microsoft equivalent to ELF and is an abbreviation of Portable Executable.
The beginning of a PE ﬁle consists of some headers containing information about
the rest of the ﬁle and where certain sections are located. The headers are broadly
referred to as ’File Headers.’ In ﬁg. 3.3 the general layout of a PE ﬁle can be seen.
Figure 3.3: The structure of PE ﬁles[49].
The left layout is for the ﬁle when stored on disk, and the right is after it has been
loaded into memory. The reason the ﬁle is larger when loaded in memory is because
of page alignment.
In the ﬁgure, there is a part of the ﬁle called ’PE Header.’ This is a collective name
for the ’COFF File header’ and the ’Optional Header,’ which will be described later
in this section.
PE describes the structure of two types of ﬁles; executable ﬁles and object ﬁles.
Executable ﬁles are also known as image ﬁles. Object ﬁles are Common Object File
Format (COFF) ﬁles and are given as input to the linker. The linker can use such
an object ﬁle to create an image ﬁle in the PE format. The PE ﬁle can be given to
the loader when the image must be run. It can be a bit confusing, but both image
and object ﬁles follow the PE Format as a rule of thumb, even though object ﬁles
are COFF ﬁles and image ﬁles are PE ﬁles. There are some diﬀerences, which will
be highlighted in the following[13].
Since executable ﬁles are image ﬁles, headers and sections relevant for image ﬁles will
be the main focus of the following analysis. Therefore, the description of sections
only appearing in object ﬁles will be limited or entirely omitted.

3.2.1
MS-DOS Stub & Signature
The ﬁrst part of a PE ﬁle, which is also part of the ﬁle headers, is the MS-DOS Stub.
In today’s architectures, this is not used but is merely there to ensure backward
compatibility. MS-DOS is an old operating system of Microsoft dating back to the
’80s. However, if the ﬁle were to be run in MS-DOS, the stub makes the ﬁle print
the message: "This program cannot be run in DOS mode".
The ﬁrst two bytes of the MS-DOS Stub and any PE ﬁle are the byte representation
of ’MZ.’ MZ is the initial of Mark Zbikowski, who was part of the group developing
MS-DOS[49].
Besides ensuring backward compatibility, the MS-DOS stub also holds the oﬀset
to the ’PE Signature’ in the ﬁeld ’e_lfanew.’ This ﬁeld is often at oﬀset 0x3C and
points to the oﬀset 0x80. The signature immediately follows the DOS stub and is a
4-byte ﬁeld containing the characters ’P’ and ’E’ followed by two null-bytes. The
signature is a way to identify the ﬁle as an image ﬁle using the PE format.
The MS-DOS Stub and the signature are only part of image ﬁles, not object ﬁles.
The remainder of the ﬁle headers are made up of the ’COFF File header’ and the
’Optional Header’[13].

3.2.3
Optional Header
To round the ﬁle headers, we have the optional header. For image and object ﬁles, the
Optional Header is required and follows the COFF File Header. In image and object
ﬁles, the presence and absence of this header are what make the header "optional."
To be clear, this header is not optional for image ﬁles.
The optional header consists of three parts; standard ﬁelds, Windows-speciﬁc ﬁelds,
and data directories[13].
3.2.3.1
Standard Fields
The standard ﬁelds contain seven or eight ﬁelds in total. Eight ﬁelds for 32-bit PE
ﬁles and seven for 32+ PE ﬁles, i.e., 64-bit ﬁles.
The ﬁrst ﬁeld is the magic ﬁeld, not to be confused with the one from the DOS-stub
described earlier. This magic ﬁeld tells what type of ﬁle it is; a normal executable, a
ROM image, or a PE32+ executable. A value of 0x10b represents a normal 32-bit
executable, while 0x20b represents a PE32+, 64-bit, executable.
The following two ﬁelds are the major and minor linker versions. Then there are
three ﬁelds containing the sizes of code, initialized data, and uninitialized data.
Depending on the ﬁle type, the last two or three ﬁelds contain diﬀerent addresses or
relative addresses. These are relevant only after the ﬁle has been loaded to memory.
The ﬁrst ﬁeld is the entry point. The second ﬁeld is the base of the code section,
relative to the entry point. The last ﬁeld is the base of the data section, also relative
to the image base. This ﬁeld is absent in PE32+ ﬁles[13].

such cases, both the virtual address and the size in the data directory will be 0.

3.2.5
Remainder of PE ﬁles
The rest of the PE ﬁle consists of diﬀerent sections, and the placement is given
somewhere in the headers or the section table. So, everything in a PE ﬁle can be
found. For headers by knowing the ﬁxed location, for everything else by reading
oﬀset values. How to ﬁnd sections is important to keep in mind when developing a
tool that analyses a PE ﬁle.
With this in mind, we continue to explain the last part of PE ﬁles.
There will be a walk-through of the COFF Symbol Table and the COFF String Table
before discussing some chosen sections. A description of linking in PE ﬁnalizes the
analysis of PE.
3.2.5.1
COFF Symbol Table
The symbol table in the PE format keeps section and ﬁle names along with code and
data symbols. Each entry in the symbol table is made up of 18 bytes. In this table,
the symbols in the code can be looked up. The oﬀset of the symbol table can be
found in the COFF File Header, along with the total number of symbols in the table.
An entry has the following ﬁelds:
• Name
• Value
• SectionNumber
• Type
• StorageClass
• NumberOfAuxSymbols
The name ﬁeld is eight bytes and behaves almost like the name ﬁelds in the section
header table. If the name has a length of eight bytes or less, then the ﬁeld is just an
array of eight bytes representing the name itself. If the name is longer than eight
bytes, then the ﬁrst four bytes of the ﬁeld will be zero, and the last four bytes will
be an index into the string table. Therefore, the ﬁrst four bytes can be used to check
if the name is given in the symbol table or has to be looked up in the string table.
The value ﬁeld is 4 bytes long and related to the symbol. Its interpretation depends
on the section number and storage class, which comes later in the symbol table
entry. The section number is an index to the section table, representing the symbol’s
section. The index is a 2-byte signed integer.
The storage class occupies one byte as an unsigned integer. Based on this, it is possi-
ble to translate the symbol’s meaning. The storage class value can mean a handful of
things. Some of them are; external, static, member of the struct, enum tag, function,
ﬁle, section, and a lot more, all prepended with "IMAGE_SYM_CLASS." The mean-
ing of storage class makes it possible to interpret the value ﬁeld in the symbol table
entry. For example, if the storage class is "IMAGE_SYM_CLASS_ARGUMENT"
then the value ﬁeld will specify the nth argument.
The type ﬁeld is 2 bytes, where each byte has its meaning. The most signiﬁcant byte
represents the overall type; pointer, function, or array. The least signiﬁcant byte
gives the data types as we know them; void, char, short, int, double, struct, etc.

3.2.5.2
COFF String Table
The string table in the PE format is relatively like that of the ELF format. The
strings in the string table are null-terminated to know when a string stops and a
new one begins. The ﬁrst four bytes represent the total size of the table, including
those four bytes. This makes it possible to calculate the address where the string
table ends.
The headers do not give an oﬀset to where the string table begins. Instead, it is just
known that it must follow the symbol table, making it possible to calculate the oﬀset
manually.

Sections
Following all the above mentioned headers and the symbol and string tables are the
actual sections referred to in the headers and tables. Numerous diﬀerent reserved
sections exist. Following is a short list of the most recognized ones. Just like with
ELF there are section for code, initialized and uninitialized, read only initialized

.bss, .text, .data, .rdata, .edata, .idata, .reloc, .drective, .tls.
It is essential to mention that some sections are relevant to object ﬁles and others to

ignore sections relevant for image ﬁles appearing in object ﬁles. They would only be
relevant to the image loader.
Each entry in the section table gives information about these sections, such as their

ﬁeld is an ASCII string of the name. When linking with a DLL, the linker looks for
the names in this table.
The import address table keeps the addresses of the symbols that must be imported;
before binding, it will be identical to the import lookup table. After binding, which
is typically handled by the loader, the entries contain only the memory addresses of
the symbols, i.e., the addresses that can be found in the export address table of the
DLL[13].
3.2.6.3
Delay-Load Import Tables
The .idata section handles linking that must happen at load-time, when the ex-
ecutable is ﬁrst loaded to main memory. Nevertheless, the linker has the oppor-
tunity to wait until run-time; the ﬁrst time, a symbol of a DLL is called.
In-
formation for run-time linking is placed in delay-load import tables instead of in
the .idata section. This is referred to in the data directories in the entry "IM-
AGE_DIRECTORY_ENTRY_DELAY_IMPORT".
The layout is very much like that of the ’.idata’ section. It has a delay-load directory
table (DLDT), which points to other tables, just as the import directory table. The
DLDT has a ﬁeld that points to the name of the DLL. It has a ’Delay Import Address
Table,’ ’Delay Import Name Table,’ ’Bound Delay Import Table,’ and ’Unload Delay
Import Table.’
These tables also follow the layout of the matching tables in the ’.idata’ section. Only
the ’Unload Delay Import Table’ is diﬀerent and is only used if unloading is explicitly
requested in the code[13].
This section will not be discussed further, but it is vital in the process of dependency
identiﬁcation, as there must be looked for names of DLLs in this section also.
It must be mentioned that even if a ﬁle has an import directory, there is not necessarily
a ’.idata’ section. It can also be put in the ’.rdata’ section by the linker. Finding
the correct section must be handled when directories are read from the ﬁle. That
process will be explained later when creating the algorithm for ﬁnding dependencies.

3.2.7
Linking in PE
Linking was shortly mentioned at the beginning of the section. One or more object
ﬁles goes into the linker, which creates a single image ﬁle.
The linker can also take library ﬁles, which are also created from object ﬁles.
Windows uses dynamic-link libraries (DLL) for dynamic linking, which we have seen
in the ’.idata’ and ’.edata’ sections.
All this will be thoroughly discussed below. The discussion will be based on Mi-
crosoft’s MSVC Compiler to compile C/C++ and C# ﬁles in Visual Studio. However,
multiple other compilers exist, and the concepts described here are assumed to have
their equivalences in those other compilers.
3.2.7.1
Static Linking
A library ﬁle is created from object ﬁles. If one wants to make a library ﬁle, some
source code must be written. The source ﬁle(s) can then be given to the MSVC
Compiler. The MSVC Compiler, eﬀectively cl.exe, will automatically link the ﬁles,
so it must be instructed not to do this, so it creates an object ﬁle and not an image
ﬁle.
When an object ﬁle has been created, the library manager, lib.exe, must be used to
link and create the library ﬁle[52].
To link ﬁles into an executable or a DLL, the linker, link.exe, takes COFF Object
ﬁles and library ﬁles as inputs[53]. The linker will take sections of the same type
in the object ﬁles and merge them into a uniﬁed section. For example, will all text
sections from the object ﬁles become one section in the executable ﬁle, the data
sections are merged into one single data section and so forth[54].
Static linking thus creates what appears to be a single executable ﬁle, where all
functions and variables can be found within the executable. This makes for more
portable executables, as there are no dependencies needed for the code to run. This
comes at the expense of the size of the ﬁnal executable, which will be larger than if
dynamic linking was used instead.
Of course, the static linker or link-time linker still has to perform tasks such as
relocation and symbol resolution when multiple object ﬁles and library ﬁles are
linked together. Relocation is done to ensure that all relative addresses match in the
new ﬁle when sections, tables, and information are replaced to ﬁt into the collective
output ﬁle. If an object ﬁle uses a function or variable from another object ﬁle or a
library, symbol resolution patches the symbol, so it refers to the actual location of
it[14].
Unfortunately, the linker does not save the names of the input ﬁles anywhere, so it
is impossible to ﬁnd these static dependencies. Only the symbol names give hints as
to what libraries have been used, but symbol names in a library are far from unique
and can most likely be found in many library ﬁles.
3.2.7.2
Dynamic Linking
By now, it should be clear that DDL ﬁles are central to dynamic linking. For PE, the
DLL dependencies can be found in the ’.idata’ section and the ’Delay-Load Import
Tables.’ With those sections, it also became clear that dynamic linking can happen
at two times; load-time and run-time.
Load-Time linking happens by taking the information left by the linker, namely the
’.idata’ section, and ﬁnding the DLL’s names used. These DLLs are then searched for.
32

==PAGE:44==
If a DLL can not be found, the process is terminated, and an error is shown. If lucky,
the DLL will be mapped to the process as the executable is loaded into memory[55].
Run-time dynamic linking waits to load the DLL until the ﬁrst time it is needed in
the executable. There is not much diﬀerence between load-time and run-time linking
in regards to loading. Run-time linking searches for DLLs the same way load-time
linking does. The beneﬁt of run-time linking is that the executable can run even
if there is a DLL it cannot ﬁnd. That is not possible for load-time linking as the
executable needs the DLL to start a process. If run-time linking come across a DLL
that can not be found, it can either try to use another DLL or prompt the user to
give the full path of the DLL[56].
The Windows OS comes with standard DLLs such as the ’kernell32.dll’, which is
a ubiquitous dependency in PE ﬁles. In addition, the link.exe program we got
acquainted with in the static linking section can create a DLL from object and library
ﬁles, such that users can create their own DLL ﬁles.
The dependencies that come from using DLLs are the ones we can identify in an
executable. The process of identifying this has already been slightly discovered and
will be thoroughly explained in the next section.
Identifying the dependencies in a single executable ﬁle is relatively simple, but they
can have dependencies themselves. These must also be mapped, as the original
executable depends on those. On Windows, there is a speciﬁc search order when
searching for DLLs for an executable ﬁle. Some of the elements and paths from there
will be used to ﬁnd DLLs, analyze them, and ﬁnd their dependencies.
For standard desktop applications, paths to a DLL can be given by the full path or
DLL Redirection.
DLL redirection is a way to avoid troubles when new versions of a DLL is published.
The old DLL ﬁle must be installed in the directory of the application[57].
If none of those are used, the system uses the following search strategy:
1. Look for DLL’s of the same name loaded in memory. If one exists use that.
2. Use the list of known DLL’s to look for it. The list of known DLL’s can be found
at the registry key: "HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\
Control\Session Manager\KnownDLLs"
The above is not the complete search strategy used by Windows to locate DLLs. The
rest depends on whether SafeDllSearchMode is set or not. This is also a registry key,
which can be found at "HKEY_LOCAL_MACHINE\SYSTEM\CurrentControlSet\
Control\Session Manager\SafeDllSearchMode"[58].
If the SafeDllSearchMode key is set, the following strategy is used, quoted from the
Microsoft Documentation[58]:
1. The directory from which the application loaded. 2. The system
directory. Use the GetSystemDirectory function to get the path of this
directory. 3. The 16-bit system directory. There is no function that
obtains the path of this directory, but it is searched. 4. The Windows
directory. Use the GetWindowsDirectory function to get the path of this
directory. 5. The current directory. 6. The directories that are listed
in the PATH environment variable. Note that this does not include the

The current directory is searched as the second step instead of the ﬁfth if the key is
not set. The remaining steps maintain their order.
There are other paths that the tool must look into to ﬁnd a DLL in the above;
the system directory, the Windows directory, the directory where the application is

Dependency Confusion
One of the ﬁrst times dependency confusion got the public’s attention was when

the tool must deﬁne these structures. An entity with structures for the ELF-header,
entries in the section header table, the program header table, the dynamic section,

All this information is essential for the process of identifying dependencies. Ultimately,
the table in the dynamic section is what must be found. To ﬁnd the dynamic section,

for printing the aforementioned structures also exists in the tool. Nonetheless, the
presentation of information to the user must be handled carefully as it quickly can
become inconvenient and awkward to the user.
To bind it all together, there must be a central logic to the tool that ﬁrst and
foremost retrieves what information the user requested. Afterward, it must fetch the
corresponding structure from the component that handles the ﬁle before sending the
structure to the printing entity, ultimately ﬁnishing the use case invoked by the user.

ﬁrst of all, identify the dependencies in an executable program. Furthermore, to
ﬁnd the chain of dependencies, the path to the dependencies must also be found.

5.4.1
ELF
During the analysis, it was seen that the ’.dynamic’ section contains information

If the ’d_tag’ ﬁeld has a value of ’DT_NEEDED’ or

’DT_SONAME,’ then the entry represents a shared library. In these cases, the

(b) The value is the index to the string table
(c) The name of the shared library is found at the index in the string table.
(d) The correct string table to use, can be found in the sh_link ﬁeld of the
entry in the section header table corresponding to the dynamic section.

5.5.4
Data Structure
The data structure was implemented as a singly linked list and used in the dependency
identiﬁcation of both tools.
The linked list consisted of nodes and was created to keep track of the identiﬁed
dependencies amidst all the recursive function calls. A node in the linked list is made

up of three ﬁelds. The ﬁrst is a char pointer to the name of the dependency. The
second is also a char pointer, but it keeps the path’s name to the dependency. The
last ﬁeld is a pointer to the next node in the list.
Only two operations can be performed on the list; push and search. The push
operation inserts a new node at the head of the list. The search can go through the
list and check if a dependency is already in there. Only the name of a dependency
can be searched for, not a path.
The singly linked list was chosen because of its simplicity and relatively easy im-
plementation. Consideration of an alternative data structure was the general tree
structure, but implementing such a structure would be trickier and disregarded
because of the deadline.
5.5.5

Pathﬁnding is placed in the ’elﬁdentiﬁcation’ ﬁle. It simply follows the steps outlined
in the analysis chapter.
First, it looks up the ’LD_LIBRARY_PATH’ environment variable. Then, the
dependency name will be searched for in each folder deﬁned in the variable until the
path is found. Next, it will be checked if there is a ’DT_RUNPATH’ in the dynamic
section. If so, the value found there will be the path.
The ’ld.so.cache’ is searched if no such entry exists. This ﬁle contains fully qualiﬁed
path names to diﬀerent ’.so’ ﬁles. If a match is found to the dependency name, the
pathname will be returned.
Last, the standard library folders will be searched. For 64-bit ﬁles this is ’/lib64’ and
’/usr/lib64’, and for 32-bit ﬁles it is ’/lib’ and ’/usr/lib’. To the 64-bit pathﬁnding,
additional paths have been added to increase the chance of ﬁnding a path. These
paths are commonly known folders such as the ’x86_64-linux-gnu’.
A big part of the functionality above is to search a folder for the dependency’s name.
A small function was created for this purpose. The function is given the folder to
search in and the name to search for, along with an integer representing the bit
version of the name. The bit is passed to ensure that the bit-version of the ﬁle
searched for is the same as the ﬁle pointed to by the path.

5.5.8
BFS v. DFS
Identifying the chain of dependencies could either be done with a breadth-ﬁrst search
or a depth-ﬁrst search. The breadth-ﬁrst search was originally intended, as the idea
was only to load a speciﬁc dependency once.
The direct dependencies to an executable are deﬁned as level 1 dependencies to make
the discussion easier. Dependencies of level 1 dependencies are deﬁned as level 2
dependencies, and so forth. With a breadth-ﬁrst search, it would be easier to ensure
that a library seen multiple times would be expanded at the lowest level it was ﬁrst
seen.
For example, libc is part of almost every ELF ﬁle, so it would be better only to load
it and ﬁnd its dependencies at level 1, rather than when coming across it at level 3.
However, in the end, the depth-ﬁrst search was chosen. The singly linked list data
structure made implementing a depth-ﬁrst search easier. The depth-ﬁrst search,
unfortunately, caused the same library to be loaded multiple times, but the time it
took to load a library and identify the dependencies were minimal and unnoticeable
and therefore not seen as a problem.
To avoid loops and go through the same dependencies repeatedly, there was still
a check to see if the dependency had been seen earlier in the subchain. If so, the
dependency would not be expanded.

The ﬁrst four created ﬁles were representative of the executable ELF ﬁles and made
a good base for experimenting and testing during the implementation phase. The

grasp the concept of relocation.
When the implementation phase of the elfdepid tool came to an end, other executa-

and a couple of other executables.
For the PE tool, the testing phase was similar. The same ’helloworld.c’ source code

the case of a .exe ﬁle, the ’devcpp.exe’ was used. A program installed on the machine
where development took place.
The kernel32.dll can be found in C:\Windows\System32\ and is in fact a 64-bit dll.
Another kernel32.dll can be found in C:\Windows\SysWOW64\, which is in fact a
32-bit .dll. A bit contradictory, but for the purpose of testing it was perfect.
Of course, SysWOW64 has to be installed in order to recreate the test.
Testing was a big part throughout the implementation phase. When some function-
ality was written it was tested right away. It was easily tested by using one of the
options of the tool and checking if the output was as expected.

Chapter 7
Conclusion
The purpose of this thesis was to identify dependencies in executable programs and
develop a tool able to ﬁnd the chain of dependencies when given an executable ﬁle.
Furthermore, the tool should also show other executable ﬁle information, such as
headers and sections. The two popular executable ﬁle formats, ELF and PE, were
thoroughly described, and examples of attacks related to dependencies were presented.
Algorithms to ﬁnd dependencies were developed for ELF and PE executable pro-
grams. In the end, two tools were created implementing these algorithms fulﬁlling
the purpose of the thesis. Additionally, the tools can retrieve headers and some
sections from executable programs.
If time had allowed it, resources would have been invested in ﬁnding edge-cases not
thought of to make the tools bulletproof and capable of ﬁnding the chain of de-
pendencies in all cases. That being said, the tools are working correctly in most cases.
The original intention was to look up the identiﬁed dependencies in a vulnerability
database and return the ﬁndings to the user. However, research showed that such
databases only contain a few entries related to ’.so’ ﬁles, ’.dll’ ﬁles, and dependencies
in general.
Therefore, prioritizing such functionality would only demand time
for implementation and search for databases and were for those reasons set aside.
Furthermore, it would only automate a process that can quickly be done manually
and probably yield better results that way.

Chapter 8
Future Work
Future works will discuss how the tools can be improved and expanded and how
dependencies can be handled in the future. In addition, the look at the tools will
discuss current functionalities, other possible functionalities, and the code in general.
Then follows a discussion on dependencies and vulnerabilities and how the future
could look. Finally, some ideas evolved during the thesis will be presented.
8.1
Code
Towards the end of the project, there was a wish to improve the code, make it more
elegant, and ensure it complies with various design principles. These improvements
were not prioritized as the code worked correctly, but it is certainly something worth
looking at.
The robustness of the code is also a necessity and must be improved even more. For
example, if the tool cannot retrieve the requested information, it must show an error
message and explain why it could not be given to the user. Additionally, the tool
should be improved, such that it in any given case can ﬁnd the information if it
exists—the case where an ELF ﬁle does not have a section header table resulted in
this consideration.
A change in the data structure used during the dependency identiﬁcation is also
considered. Changing it could make it more ﬂexible and easier to pass between the
diﬀerent components. This could result in removing some responsibility from the
identiﬁcation ﬁles, which have been mentioned to contain too much responsibility.
Especially the general tree data structure can be the primary consideration.
8.2
Functionality
Besides improving the code, additional functionality would also signiﬁcantly improve
the tools. First, additional sections could be added as options, such as relocation
sections and the GOT and PLT sections in the ELF tool. The export address table
and import lookup table could be great additions to the PE tool.
Adding functionality to disassemble the code sections could give valuable insights
into how the analyzed program works. Users of the tools would, in that way, have a
Dependency identiﬁcation in executable programs
65
==PAGE:77==
comprehensive tool with all the functionality needed to assess a program.
Other executable formats would likewise be an excellent addition, so the tools can
also analyze macOS, iOS, and Android applications fx.
As mentioned earlier, the lookup of the identiﬁed dependencies in a vulnerability
database was originally intended. If proper databases could be found, this function-
ality would be a natural next step in the development.
8.3
Dependencies
As attackers keep ﬁnding new ways to obtain unauthorized access, shared libraries will
be guaranteed to be exploited—even more than what was shown in this thesis. There
could therefore be created a centralized database, kind of like the CVE database,
but for shared libraries.
Such a database should allow people to look up the name of a shared library to see
if it has ever been connected with any malware, ransomware, or other malicious
activities. It would be an immense help when people are analyzing programs before
use.
In the world of websites, vulnerability scanners are heavily used. The ultimate goal
of this tool would be to be a vulnerability scanner for executable programs. Find
dependencies, look them up and report them if they are malicious or insecure.
8.4
Static Linking
The thesis showed that the names of statically linked libraries could sometimes be
retrieved, but such libraries are impossible to ﬁnd. This is because the libraries are
built into the executable and become "invisible." It could be interesting to see if static
libraries contain signatures that could be looked up in the executable ﬁle. In that
way, static libraries could be identiﬁed along with their potential risk, expanding the
tool from only ﬁnding dependencies to also ﬁnding used libraries.

1
Project Description
The title of this Master’s thesis is "Dependency Identiﬁcation in Executable Programs" and
it has the following formal project description:
Software systems are increasingly composed as orchestrations of internally and
externally developed components, libraries and services. Some of these may fur-
ther be based on other components and services which makes it hard for software
architects and developers to determine the provenance of all elements that are
included in the system and reason about non-functional requirements, such as ro-
bustness, maintainability and security. This Master’s thesis will investigate how
to identify all dependencies from an executable program. This means identifying
the transitivity closure of all code used in a speciﬁc program. It is expected that
there will be a degree of recursion to this, as libraries included in a speciﬁc pro-
gram likewise will include other libraries. Finding a method for identifying this
chain of libraries will be the main purpose of this project.
1.1
Qualiﬁcation of Project
When a developer is writing software, there is a chance that the developer will include one or
more third-party libraries. This saves the developer the time needed to write code, that has
already been written by others. This way of working gives a huge advantage in increasing
the eﬀectiveness of the development process. It does not matter whether the developer is
creating a web application, java application, shell script etc. the import of libraries will most
likely be used.
The way these third party libraries are used is commonly through packages, which can be
downloaded from diﬀerent public registries. For python pip (Python Package Index)[1] is
used, for package management in Linux distributions it is the "apt" command that is utilized
to fetch deb packages[2] and so there are numerous other examples of package management
tools.
However, even though this is clever, it paves the way for a threat vector; supply chain
attacks. If a downloaded package contains malicious code, it could lead to a remote code
execution. Attackers has diﬀerent ways of doing this. They can infect a legitimate package
or publish a package with a name similar to a legitimate one, then waiting for the developer
to "change" the legitimate name by mistake and thereby specifying the attackers package[3].
The security researcher, Alex Birsan, made a blog post at medium.com on February 9,
2021, describing how he used dependency confusion to hack into Apple, Microsoft, PayPal
and other large corporations. He was able to exploit the private package feeds used by these
companies, by realizing how the package feeds chose the package to install. First it is checked
if a package exists on the intern package index, then if it exists on the public package index,
and then installs the version with the highest version number if a package exists in both
places[4].
This thesis will investigate how one can identify the imported libraries of an executable

2
General Idea
To accommodate the objectives coming to light in the project description there must be a
fundamental understanding of executable programs. The ﬁrst part of the thesis will there-
fore describe the Portable Executable (PE) format and the Executable and Linkable Format
(ELF), which are the most widely used executable ﬁle formats for Windows and Unix sys-
tems respectively.
When the basic concepts of executable ﬁle formats are in order, there will be a detailed
analysis of how to identify dependencies in PE and ELF ﬁles. This will likely include some
reverse engineering techniques, which will be described as well. To extensively describe this a
thorough research will be conducted in order to base it on the leading international research.
Concurrently with the research and preparation of the report, an executable program will be
designed, developed and documented, that is capable of automating the process of depen-
dency identiﬁcation. The research will in this regard also aid in the process of developing
ideas on how to solve the identiﬁed problems in dependency identiﬁcation.
Before concluding the report there will be a discussion on the purpose of being able to iden-
tify dependencies in executables in the ever evolving threat landscape. This discussion will
justify the project and take a more holistic view and see the problems in a broader academic
and societal perspective and reﬂect on how the proposed solution will help with these prob-
lems.
It must be noted that both the research and the program being developed will only consider
code that can be executed directly on the machine, hence the investigation of the ELF and
PE ﬁle formats. The consequence hereof is that diﬀerent byte-code formats and scripting
languages will be excluded from the research. Additionally, the developed program will not
be capable of analyzing and interpreting such ﬁles and formats.

project and a Gantt chart for the program. It is important to say that these Gantt charts

appear in the Gantt chart. However, there will be smaller and more detailed tasks, which
will be controlled through a Kanban board. Ideally, the Scrum framework would be used to
manage the programming objective, but since this is a single person project it is diﬃcult to
adopt all the workings of Scrum. It will instead be implemented as well as possible, meaning
