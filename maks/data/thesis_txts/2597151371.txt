Abstract
Width-based planning algorithms have shown to yield great results within classi-
cal planning, and in recent years it has also been adapted to the Atari 2600 suite
that has been a popular testing grounds for reinforcement learning and other
planning algorithms, since it oﬀers many domains with diﬀerent challenges.
Width-based algorithms plan based of states deﬁned by propositional literals
also called boolean features, and the algorithm then prunes states based of these
features. The most simple form of width-based algorithm expands in order of
breadth-ﬁrst search. When adapting width-based methods to the Atari domain
new algorithm was created called RolloutIW that uses a rollout approach for
exploring that ﬁt better for planning in Atari. The features used for planning
with RolloutIW were B-PROST features, which divide the screen into tiles and
compares pixels between the tiles. The B-PROST features are very big and not
created for width-based algorithms. Therefore in this work, we propose a diﬀerent
method for creating the features from the Atari pixel state.
We propose a variational autoencoders (VAE) with discrete latent variables
to learn a propositional representations based of pixels for the Atari domains.
The learned symbolic representation (features) from the VAE is then used to-
gether with RolloutIW to plan in the Atari domain. We call the combination
VAE-IW, which we compare with human professional game testers and previous
results gained from RolloutIW on the Atari domain with pixel states. The results
shows that we compare with human scores and outperform RolloutIW, showing
that discrete VAE’s create meaningful features for planning with width-based
algorithms.

Abstract
Width-based planning methods have shown to yield state-of-
the-art performance in the Atari 2600 video game playing do-
main using pixel input (raw data). There have been two main
approaches, where one implements an episodic rollout ver-
sion of the Iterated Width (IW) algorithm called RolloutIW.
The other also uses RolloutIW, but as a supervised method for
training a policy to improve how actions are picked. This pa-
per introduces a new approach, where we leverage the power
of variational autoencoders to learn features for the domains
and use RolloutIW for planning based on these features. More
precisely, we train a neural network on images of the domain
so it outputs binarized features. With the trained model we
can generate feature sets from the simulator image and use
these for planning, that is, we plan in the space of generated
features rather than in the original state space of screen im-
ages. The trained model in combination with RolloutIW out-
performs the original RolloutIW and human professional play
on the Atari 2600 domain and reduces the size of the feature
set from 20.5 million to 4500.
Introduction
Width-based search algorithms have in the last few years
become among the state-of-the-art approaches to auto-
mated planning, e.g. the original Iterated Width (IW) algo-
rithm (Lipovetzky and Geffner 2012). As in propositional
STRIPS planning, states are represented by a set of proposi-
tional literals, also called boolean features. The state space
is searched with blind search, BFS, but the state space ex-
plosion problem is handled by pruning states based on their
novelty. First a parameter k is chosen, called the width pa-
rameter of the search. Searching with width parameter k es-
sentially means that we only consider k literals/features at
a time. A state s generated during the search is called novel
if there exists a set of k literals/features not made true in
any earlier generated state. Unless a state is novel, it is im-
mediately pruned from the search. Clearly, we then reduce
the size of the searched state space to be exponential in k.
It has been shown that many classical planning problems,
e.g. problems from the International Planning Competition
(IPC) domains, can be solved efﬁciently using width-based
search with very low values of k.

The essential beneﬁt of using width-based algorithms is
the ability to perform semi-structured (based on feature
structures) exploration of the state space, and reach deep
states that may be important for achieving the planning
goals. In classical planning, width-based search has been
integrated with heuristic search methods, leading to Best-
First Width Search (Lipovetzky and Geffner 2017) that per-
formed well at the 2017 International Planning Competi-
tion. Width-based search has also been adapted to reward-
driven problems where the algorithm uses a simulator for
interacting with the environment (Franc`es et al. 2017). This
has enabled the use of width-based search in reinforcement
learning environments such as the Arcade Learning Envi-
ronment (ALE), where it provides an interface to the Atari
2600 video games. There have been several implementa-
tions of width-based search using directly the RAM states
of the Atari computer as the features (Lipovetzky, Ramirez,
and Geffner 2015; Shleyfman, Tuisov, and Domshlak 2016;
Jinnai and Fukunaga 2017).
Motivated by the fact that humans do not have access to
the internal RAM states of a computer when playing video
games, methods for using the algorithms based on the raw
screen pixels have been developed. Bandres, Bonet, and
Geffner (2018) propose a modiﬁed version of IW, called
RolloutIW, that uses pixel-based features and achieves re-
sults comparable to learning methods in almost real-time on
the Arcade Learning Environment (Bellemare et al. 2012).
The combination of Reinforcement Learning (RL) and Roll-
outIW in the pixel domain is explored by Junyent, Jonsson,
and G´omez (2019), who propose to train a policy to guide
the action selection in rollouts.
Regardless of different efforts, width-based algorithms
are highly exploratory and, not surprisingly, signiﬁcantly de-
pendent on the quality of the features deﬁned. The original
RolloutIW paper (Bandres, Bonet, and Geffner 2018) uses a
set of features called B-PROST (Liang et al. 2015). These
are features extracted from the screen pixels by splitting
the screen into tiles and keeping track of which colors are
present in which tiles. These features have been designed by
humans to achieve good performance in ALE. Integrating
learning of features efﬁcient for width-based search into the
algorithms themselves rather than using hand-crafted fea-
tures is of course an interesting challenge—and a challenge
that will bring the algorithms more on a par with humans try-
==PAGE:10==
ing to learn to play those video games. In Junyent, Jonsson,
and G´omez (2019), the features used were the output fea-
tures of the last neuron layer of the RL network, improving
the performance of IW. With the attempt to learn efﬁcient
features for width-based search, and inspired by the results
of Junyent, Jonsson, and G´omez (2019), this paper investi-
gates the possibilities of a more structured approach to gen-
erate features for width-based search using deep learning.
More speciﬁcally, we use the type of generative models
from deep learning called variational autoencoders (VAE).
VAEs are latent variable models that have been widely used
for representation learning (Kingma and Welling 2019) as
they allow for approximate inference of the latent variables
underlying the data. In this paper, we leverage VAEs with
discrete latent variables to learn propositional symbols di-
rectly from pixels and without any supervision. We then in-
vestigate whether the learned symbolic representation can
be successfully used for planning in the Atari 2600 domain.
We compare the planning performance of RolloutIW us-
ing our learned representations with RolloutIW using the
handcrafted B-PROST features, and report human scores as
baseline. Our results show that our learned representations
lead to better performance than B-PROST on RolloutIW
(and often outperforms human players). This is despite the
fact that the B-PROST features also contain temporal in-
formation (how the screen image dynamically changes),
whereas we only extract static features from individual
frames. Our current approach is however still limited by the
use of B-PROST in training our models (VAEs): We collect
training images using RolloutIW with B-PROST features.
Apart from improving scores in the Atari 2600 domain, we
also signiﬁcantly reduce the number of features (by a factor
of more than 103). We also investigate in more detail (with
ablation studies) which factors seems to lead to good perfor-
mance on games in the Atari domain.
The main contributions of our paper are hence:
• We train discrete generative models to learn propositional
symbols for planning, from pixels and without any super-
vision.
• We run a large-scale evaluation on the Atari 2600 domain
where we compare the performance of RolloutIW with
our learned representations to RolloutIW with the previ-
ously proposed low-level ﬁxed features (B-PROST).
• We show that our learned representations lead to higher
scores than using B-PROST, even though our features
don’t have any temporal information, and our models are
trained from data collected by RolloutIW with B-PROST,
limiting the richness of the dataset.
Below, we ﬁrst provide the required background on IW,
RolloutIW, B-PROST and VAE. We then present the orig-
inal approach of this paper, VAE-IW, combining VAEs with
IW. We then present our experimental results and conclude.
Background
Iterated Width.
Iterated Width (IW) (Lipovetzky and
Geffner 2012) is a blind-search planning algorithm in which
the states are represented by sets of boolean features (sets
of propositional atoms). The set of all features/atoms is the
feature set, denoted F. IW is an algorithm parametrized by a
width parameter k. We use IW(k) to denote IW with width
parameter k. Given an initial state s0, IW(k) is similar to
a standard breadth-ﬁrst search (BFS) from s0 except that,
when a new state s is generated, it is immediately pruned
if it is not novel. A state s generated during search is de-
ﬁned to be novel if there is a k-tuple of features (atoms)
t = (f1, . . . , fk) ∈F k such that s is the ﬁrst generated
state making all of the fi true (s making fi true of course
simply means fi ∈s, and the intuition is that s then “has”
the feature fi). In particular, in IW(1), the only states that
are not pruned are those that contain a feature f ∈F for the
ﬁrst time during the search. The maximum number of gen-
erated states is exponential in the width parameter (IW(k)
generates O(|F|k) states), whereas in BFS it is exponential
in the number of features/atoms.
Rollout IW.
RolloutIW(k) (Bandres, Bonet, and Geffner
2018) is a variant of IW(k) that searches via rollouts in-
stead of doing a breadth-ﬁrst search, hence making it more
akin to a depth-ﬁrst search. A rollout is a state-action tra-
jectory (s0, a0, s1, a1, . . .) from the initial state s0 (a path in
the state space), where actions ai are picked at random. It
continues until reaching a state that is not novel. A state s is
novel if it satisﬁes one of the following: 1) it is distinct from
all previously generated states and there exists a k-tuple of
features (f1, . . . , fk) that are true in s, but not in any other
state of the same depth of the search tree or lower; 2) it is an
already earlier generated state and there exists a k-tuple of
features (f1, . . . , fk) that are true in s, but not in any other
state of lower depth in the search tree. The intuition behind
case 1 is that in this case the new state s comes with a com-
bination of k features occurring at a lower depth than earlier
encountered, which makes it relevant to explore further. In
case 2, s is an existing state already containing the lowest
occurrence of one of the combinations of k features, again
making it relevant to explore further. When a rollout reaches
a state that is not novel, the rollout is terminated, and a new
rollout is started from the initial state. The process continues
until all possible states are explored, or a given time limit
is reached. After the time limit has been reached, an action
with maximal expected reward is chosen (the method was
developed in the context of the Atari 2600 domain, making
use of a simulator to compute action outcomes and rewards).
The chosen action is executed, and the algorithm is repeated
with the new state as initial state.
RolloutIW is an anytime algorithm that will return an ac-
tion independently of the time budget. In principle, IW could
also be used as an anytime algorithm in the context of the
Atari 2600 domain, but it will be severely limited by the
breadth-ﬁrst search strategy that will in practice prevent it
from reaching nodes of sufﬁcient depth in the state space,
and hence prevent it from discovering rewards that only oc-
cur later in the game (Bandres, Bonet, and Geffner 2018).
Planning with Pixels.
The Arcade Learning Environment
(ALE) (Bellemare et al. 2012) provides an interface to Atari
==PAGE:11==
2600 video games, and has been widely used in recent years
as benchmark for reinforcement learning and planning algo-
rithms. In the visual setting, the sensory input consists of a
pixel array of size 210×160, where each pixel can have 128
distinct colors. Although in principle the set of 210×160×128
booleans could be used as features, we will follow Bandres,
Bonet, and Geffner (2018) and focus on features that capture
meaningful structures from the image.
An example of a visual feature set that has proven suc-
cessful is B-PROST (Liang et al. 2015), which consists of
Basic, B-PROS, and B-PROT features. The screen is split
into 14 × 16 disjoint tiles of size 15 × 10. For each tile (i, j)
and color c, the basic feature fi,j,c is 1 iff c is present in
(i, j). A B-PROS feature fi,j,c,c′ is 1 iff color c is present
in tile t, color c′ in tile t′, and the relative offsets between
the tiles are i and j. Similarly, a B-PROT feature fi,j,c,c′ is
1 iff color c is present in tile t in the previous decision point,
color c′ is present in tile t′ in the current one, and the relative
offsets between the tiles are i and j. The number of features
in B-PROST is the sum of the number of features in these 3
sets, in total 20, 598, 848.

VAEs with discrete variables.
Since the categorical dis-
tribution is not reparameterizable, training VAEs with cate-
gorical latent variables is generally impractical. A solution
==PAGE:12==
to this problem is to replace samples from a categorical dis-
tribution with samples from a Gumbel-Softmax distribution
(Jang, Gu, and Poole 2016; Maddison, Mnih, and Teh 2016),
which can be smoothly annealed into a categorical distri-
bution by making the temperature parameter τ tend to 0.
Because Gumbel-Softmax is reparameterizable, the path-
wise gradient estimator can be used to get a low-variance—
although in this case biased—estimate of the gradient. In this
work, we use Bernoulli latent variables (categorical with 2
classes) and their Gumbel-Softmax relaxations.
VAE-IW
Although width-based planning has been shown to be gener-
ally very effective for classical planning domains (Lipovet-
zky and Geffner 2012; 2017; Franc`es et al. 2017), its perfor-
mance in practice signiﬁcantly depends on the quality of the
features used. Features that better capture meaningful struc-
ture in the environment typically translate to better planning
results (Junyent, Jonsson, and G´omez 2019). Here we pro-
pose VAE-IW, in which representations extracted by a VAE
are used as features for RolloutIW. The main advantages are
the following:
• The number of features can be orders of magnitude
smaller than B-PROST, leading to faster planning and a
smaller memory footprint.
• Autoencoders, and in particular VAEs, are a natural ap-
proach for learning meaningful, compact, high-level rep-
resentations from data (Bengio, Courville, and Vincent
2013; Tschannen, Bachem, and Lucic 2018; Kingma and
Welling 2019).
• No additional preprocessing is needed, such as heuris-
tics to mask out the background (Junyent, Jonsson, and
G´omez 2019).
On the planning side, we follow Junyent, Jonsson, and
G´omez (2019) and use RolloutIW(k) with width k = 1 to
keep planning efﬁcient even with a small time budget, since
width 1 is sufﬁcient to get good results, and since it makes
it easier to compare our experimental results with the afore-
mentioned paper. For example, with a planning budget of
0.5s per planning phase, RolloutIW(2) generates only a few
nodes per step, whereas RolloutIW(1) typically generates
hundreds. In the remainder of this section, we detail the two
main components of the proposed method: feature learning
and planning.

2014; Mnih and Rezende 2016; Tucker et al. 2017; Gu et
al. 2015) or with alternative variational objectives (Hinton
et al. 1995; Bornschein and Bengio 2014; Le et al. 2018;
Masrani, Le, and Wood 2019).
Note that we are not necessarily interested in the best gen-
erative model—e.g. in terms of marginal log likelihood of
the data or visual quality of the generated samples—as much
as in representations that are useful for planning. In order
to control the tradeoff between the competing terms in (6),
we use the β-VAE framework, and following Burgess et al.
(2018) we decrease β until good quality reconstructions are
obtained.

Experiments
Experimental Details.
We evaluate the proposed ap-
proach on a suite of 47 Atari 2600 games. We separately
trained a VAE for each domain by maximizing the objec-
tive function (7) using stochastic gradient ascent. As recon-
struction loss we used the binary cross-entropy. We used the
Adam optimizer (Kingma and Ba 2014) with default param-
eters and learning rate 10−4. To train the VAEs, we collected
15,000 frames by running RolloutIW(1) with B-PROST fea-
tures, and split them into a training and validation set of
14,250 and 750 images, respectively. The RGB images of
size 210×160 are grayscaled and downsampled to 128×128.
VAE-IW experiments were performed with 4GB of RAM on
one core of a Xeon Gold 6126 CPU, and a Tesla V100 GPU.
The GPU was used both to train the models and to evaluate
the encoder in the planning phase.
The encoder consists of 3 convolutional layers interleaved
with residual blocks. The single convolutional layers down-
sample the feature map by a factor of 2, and each residual
block includes 2 convolutional layers, resulting in 7 con-
volutional layers in total. The decoder mirrors such archi-
tecture, and performs upsampling with transposed convolu-
tional layers. Further architectural details are provided in the
Appendix. The inference network outputs a feature map of
shape H × W × C which represents the approximate pos-
terior probabilities of the latent variables. Thus, unlike in
traditional VAEs where latent variables do not carry any spa-
tial meaning, in our case they are spatially arranged (Vahdat
and Kautz 2020). As outlined in the previous section, the
Bernoulli probabilities computed by the encoder are thresh-
olded, and the resulting binary features are used as propo-
sitional representations for planning in RolloutIW(1), in-
stead of the B-PROST features. For the planning phase of
VAE-IW, we use RolloutIW(1) with partial caching and risk
aversion (RA) as described by Bandres, Bonet, and Geffner
(2018). With partial caching, after each iteration of Roll-
outIW the branch of the chosen action is kept in memory.
Risk aversion is attained by multiplying all negative rewards
by a factor α ≫1 when they are propagated up the tree. In
the experiments section rollout methods with the RA abbre-
viation uses risk averse rewards.
Because of their diversity, Atari games vary widely in
complexity. We empirically chose a set of hyperparame-
ters that performed reasonably well on a small subset of the
games, assuming this would generalize well enough across
the whole Atari suite. Following previous work (Lipovetzky
and Geffner 2012; Bandres, Bonet, and Geffner 2018), we
used a frame skip of 15, and we allocate a planning budget
of either 0.5s or 32s per time step. We set an additional limit
of 15,000 executed actions for each run, to prevent runs from
lasting too long. Note that this constraint is only applied to
our method. Table 1 summarizes the main hyperparameters
used in our experiments. We expect that better results can be
obtained by performing an extensive hyperparameter search
on the whole Atari suite.

Main Results
In Table 2, we compare the planning performance of our RA
VAE-IW(1) to IW(1) with B-PROST features (Lipovetzky
and Geffner 2012) and RA RolloutIW(1) (Bandres, Bonet,
and Geffner 2018) with B-PROST features. On ﬁgure 1, we
normalize the scores of RA VAE-IW(1) and RolloutIW(1)
with B-PROST using the human scores to give better idea
about how they perform against humans. Following the lit-
erature, we report the average score over 5 runs, with each
run ending when the player dies.
These results show that learning binary features purely
from images, without any supervision, can be beneﬁcial for
width-based planning. In particular, using the learned repre-
sentations in RolloutIW leads to generally higher scores in
the Atari 2600 domain. Note that the performance of VAE-
IW depends on the quality and expressiveness of the fea-
tures extracted by the VAE, which in turn depend on the
images the VAE is trained on. Crucially, since we collect
data by running RolloutIW(1) with B-PROST features, the
performance of VAE-IW is constrained by the effectiveness
of the data collection algorithm. For example, the VAE fea-
tures will not necessarily be meaningful on parts of the game
that are signiﬁcantly different from those in the training set.
Surprisingly, however, our method signiﬁcantly outperforms
the baseline that was used for data collection, providing even
stronger evidence that the compact set of features learned by
a VAE can be successfully utilized for width-based planning
algorithms.
In addition, there appears to be a signiﬁcant negative cor-
relation between the average number of true features and
the average number of expanded nodes per planning phase
(Spearman rank correlation of −0.51, p-value < 0.001). In
other words, in domains where the VAE extracts on average
more true features, the planning algorithm tends to expand
fewer nodes. Thus, it could be potentially fruitful to further
investigate the interplay between the average number of true
features, the meaningfulness and usefulness of the features,
and the efﬁciency of width-based planning algorithm.

Ablation Studies
The performance of VAE-IW depends on several hyperpa-
rameters related to the planning algorithm and to the prob-
abilistic model. Here we attempt to investigate the effect of
some of these parameters.
Modeling choices.
One of the major modeling choices is
the dimensionality of the latent space, and the spatial struc-
ture H ×W ×C of the latent variables. Both of these factors
are tightly coupled with the neural architecture underlying
the inference and generative networks. As there is no clear
heuristic, we explored different neural architectures and la-
tent space sizes up to 10,000 latent variables. Based on the
performance on a few selected domains, we chose two dif-
ferent settings, with latent space size 15 × 15 × 20 and
4×4×200 (see the Appendix for further details). In Table 12
we compare the performance of RA VAE-IW on these two
conﬁgurations, keeping the rest of the hyperparameters ﬁxed
as the ones used in Table 2. While overall the 15 × 15 × 20
conﬁguration leads to a higher score in most domains, the
effect of this modeling choice seems to signiﬁcantly depend
on the domain.
As previously mentioned, we consider the framework of
β-VAEs in which β controls the trade-off between recon-
==PAGE:16==
struction accuracy and amount of information encoded in
the latent space. For our purposes, β has to be small enough
that the model can capture all relevant details in an im-
age. In practice, we decreased β until the model generated
sufﬁciently accurate reconstructions on a selection of Atari
games (Burgess et al. 2018). Table 13 reports the perfor-
mance of VAE-IW when varying β ∈{10−4, 10−3}, and
shows that the effect of varying β depends on the domain.
Intuitively, while a stronger regularization (i.e. higher β)
can be detrimental for the reconstructions and thus also for
the informativeness of the learned features, it may lead to
better representations in less visually challenging domains.
In practice, one could for example train VAEs with dif-
ferent regularization strengths, and do unsupervised model
selection separately for each domain by using the “elbow
method” (Ketchen and Shook 1996) on the reconstruction
loss or other relevant metrics.
Planning parameters.
Regardless of the features used for
planning, the performance of VAE-IW depends on the vari-
ant of RolloutIW being used, and its parameters. In Table 8
we compare the average score of VAE-IW run with and
without risk aversion, and observe that the risk-averse vari-
ant acheives an equal or higher average score in 33/47 do-
mains.
Table 9 shows the results of VAE-IW and RolloutIW with
B-PROST features, similarly to Table 2, except that both
methods are run without risk aversion. With this modiﬁca-
tion, our method still obtains a higher average score in the
majority of domains (32/47).
Another crucial planning parameter is the time budget
for planning at each time step. While the main results are
based on a 0.5s budget, we also consider a 32s budget, fol-
lowing Bandres, Bonet, and Geffner (2018). In Table 10 we
observe that, not surprisingly, the high time budget outper-
forms the low budget in most domains (34/47). However, in
some of them the shorter planning budget yields a signiﬁ-
cantly higher score (e.g. in Asterix, CrazyClimber, and El-
evatorAction). Interestingly, increasing the planning budget
seems to leave the average rollout depth unaffected, while
the average number of expanded nodes in each planning
phase grows signiﬁcantly. This behaviour is consistently ob-
served in all tested domains (see Figures 2 and 3) and points
to the fact that increasing the planning budget improves re-
sults mostly by allowing more rollouts.
Finally, in Table 11 we compare the average scores ob-
tained by VAE-IW and RolloutIW with B-PROST features,
using a 32s planning budget for both methods. Also in this
case, using the compact features learned by a VAE seems to
be beneﬁcial, as VAE-IW obtains the highest average score
in 29/47 domains.
Related Work
Variational Autoencoders (VAEs) have been extensively
used for representation learning (Kingma and Welling 2019)
as their amortized inference network lends itself naturally
to this task. In the context of automated planning, Asai and
Fukunaga (2017) proposed the State Autoencoder (SAE) for
propositional symbolic grounding. An SAE is in fact a VAE
with Bernoulli latent variables. It is trained by maximiz-
ing a modiﬁed ELBO that includes an additional entropy
regularizer, deﬁned as twice the negative KL divergence.
Thus, the objective function being maximized is simply the
ELBO (6) with the sign of the KL ﬂipped. Although uninten-
tional (Asai and Kajino 2019), this proved to be fundamental
for the mitigation of the issue of representation instability.
A variation of SAE, the zero-suppressed state autoencoder
(Asai and Kajino 2019), adds a further regularization term to
the propositional representation (features), leading to more
stable representations.
Zhang et al. (2018) take a supervised approach to repre-
sentation learning for planning, and learn a transition graph
for planning in the representation space with Dijkstra’s algo-
rithm. Konidaris, Kaelbling, and Lozano-Perez (2018) spec-
ify a set of skills for a task, and then automatically extract
state representations from raw observations. Kurutach et al.
(2018) use generative adversarial networks to learn struc-
tured representations of images and a deterministic dynam-
ics model, and plan with graph-search methods.
Junyent, Jonsson, and G´omez (2019) proposed π-IW, a
variant of RolloutIW(1) where a neural network guides the
action selection process in the rollouts, which would other-
wise be random. This is reminiscent of AlphaZero (Silver
et al. 2018), where a policy network guides the rollouts of
Monte Carlo Tree Search (MCTS). Moreover, π-IW plans
using features obtained from the last hidden layer of the pol-
icy network, instead of B-PROST, leading to faster conver-
gence.
Conclusion
We have introduced a new combination of width-based plan-
ning with learning techniques. The learning employs a VAE
to learn relevant features in a video game domain given im-
ages of screen states as training data. The planning is done
with RolloutIW(1) using the features trained by the VAE.
Our approach reduces the size of the feature set from the
20.5 million having been used earlier in connection with
RolloutIW(1) (the B-PROST features) to only 4500. Our al-
gorithm, VAE-IW, has been shown to outperform the previ-
ous methods, proving that the VAE learns meaningful rep-
resentations that RolloutIW can use for planning. Interest-
ingly, in some domains the VAE-IW uses its maximum num-
ber of allowed actions (15000) without dying, indicating that
it could potentially get even higher scores.
A main limitation in our work is that the symbolic rep-
resentations are learned from data collected by RolloutIW
using B-PROST features. Increasing the size and diver-
sity of the training set could lead to better representations,
and thus better planning results. The quality of representa-
tions could also be improved by using more expressive dis-
crete models, for example with a hierarchy of discrete la-
tent variables (Van Den Oord, Vinyals, and others 2017;
Razavi, van den Oord, and Vinyals 2019). Finally, we can
expect further improvements orthogonal to this work, by
learning a rollout policy for more effective action selection
(Junyent, Jonsson, and G´omez 2019).

Chapter 1.
Introduction
Frederik Krieger Drachmann
1 Introduction
Width-based search algorithms have in recent years been accepted as state-of-
the-art planning algorithms, where the original algorithm is called Iterated Width
(IW) [LG12]. In width-based algorithms states described with propositional lit-
erals, also called boolean features. IW uses BFS for exploring the states space
and the increasing state space is handled by pruning states based of their novelty.
The pruning aggressiveness is controlled by k, which means a state is novel if it
has set of k features that have not been generated by any other state.
Th
Width-based algorithms are considered very eﬀective because of their ability
to perform semi-structured (based on feature structure) exploration of a state
space, and reaching deep states that may be important for achieving the planning
goals. In classical planning, these algorithms has been used in combination with
state-of-the-art heuristics search to create Best-First Width Search [LG17] that
performed well at the 2017 International Planning Competition. Other authors
used the same approach [STD16; JF17] and reached similar results.
Within recent years the Atari video games supported by the Arcade Learning
Environment (ALE) [Bel+12] has won favour as a challenging domain for rein-
forcement learning (RL) and planning with the breakthrough of deep RL with
DQN [Mni+15] and DQN was followed by a more standard version using screen
states [Lia+16]. Motivated by these developments, width-based algorithms were
ﬁt to the reward driven Atari domain with great success by Lipovetzky, Ramirez,
and Geﬀner [LRG15] using Iterated Width (IW) algorithm [LG12] with memory

Planning Theory
We assume that knowledge about classical planning problem P = (P, s0, g)
[GNT04], where s0 is the initial state, g is a ﬁnite set of goals and P = (S, A, γ)
[GNT04], where S is a ﬁnite set of enumerable states, A is a ﬁnite set of enumer-
able actions and γ is a state transition function γ : S ×A. A solution to planning
problem is a plan π consisting of a sequence of actions π = ⟨a0, ..., an⟩performed
on the initial state s0. The corresponding state sequence is ⟨s0, s1, ..., sn+1⟩where
sn+1 = g, and the states follows from the state transition function γ(si, ai+1) =
si+1 for i = 0...n. The length of plan n = |π| is the number of actions in the plan.
The cost of a plan is the accumulated cost for every action cost(π) = Pn
i=1 c(ai).
It often makes sense to have diﬀerent costs for actions but in this thesis actions
costs are always 1, which means cost(π) = |π|. This means optimal plans (lowest
cost) are also shortest path.

Goal implication
We consider a goal formula G1 that optimally implies goal formula G2 in a prob-
lem Π. Which means all the optimal plans for G1 are also optimal plans for
G2.
Deﬁnition 2. [LG12] (Optimal implication, |=∗) A goal formula G1 optimally
implies a goal formula G2 in a problem Π, denoted G1 |=∗G2, if all optimal plans
for G1 are also optimal plan for G2.
This is not a standard implication but requires G2 to be true in all reachable
states where G1 is true. The optimal implication G1 |=∗G2 implicitly uses the
information in the initial situation and the actions of the problem Π. If we take
the example of Blocks World problem following the PDDL on ﬁgure 2.2, where
we have an initial situation I and two goals G1 and G2 :
I = {on(b, a), clear(b), ontable(a)}
G1 = {ontable(b)}
G2 = {clear(a), handfree}
Then G1 does not logically imply goal G2, but it holds that G1 |=∗G2 in Π.
From our initial situation there is only one way of optimally achieving G1 and it
is the action sequence φ:
φ1 = ⟨unstack(b, a), putdown(b)⟩
The φ1 action sequence also achieves G2 and there is no cheaper way of achieving
either G1 or G2.
As a note G3 ⊭∗G1 if G3 = {clear(a)} since clear(a) is
achieved with action sequence φ2 = unstack(b, a) and it does not need the action
putdown(b) from φ1 to be achieved.

Width deﬁnition
Given the notion of goal optimal implication we deﬁne the width of a planning
problem Π. We also deﬁne the width for an arbitrary goal formula ϕ relative to
a problem Π.
Deﬁnition 3. [LG12] For a formula ϕ over the ﬂuents in Π that is not true in
the initial situation I, the width w(ϕ) of ϕ relative to Π is the min w such that
Gw contains a tuple t ∈Gw that optimally implies t |=∗ϕ. If ϕ is true in I, its
width is 0.
Deﬁnition 4. [LG12] (Width of problem Π)The width of a planning problem Π,
w(Π), is the width of its goal G relative to Π.
The width of a problem bounds the complexity of solving the problem.
Theorem 1. [Lip14] Given a problem Π, if w(Π) = i, Π can be solved optimally
in time O(ni), where n is the number single atoms in F.
This method creates a bound on planning problem in terms of its goal struc-
ture. Later on we describe an algorithm that achieves the bounded width and
runs in exponential time in the width of the problem.
In the following paragraphs we show examples of simple domains. We also
show the results from [LG12] where the authors run the algorithm on simple
domains at diﬀerent width to test when it can solve the problem, these results
can be seen in the appendix A.1.1. It shows that most domains have small width
(less than 3).

2.3
Iterated width
The previous section has given an introduction to tuple graphs and how they can
be used for solving simple domains. We also introduced the notion of width as
a way of determining the size of tuples needed to solve a given problem. In this
section, we introduce an algorithm that achieves the same complexity and results
expressed in deﬁnition 1. The algorithm is called Iterated Width search (IW) and
it is create by Lipovetzky and Geﬀner [LG12]. It consists of a series of IW calls,
IW(i) for i = 1, 2, ... over a problem Π until the problem is solved. Its complexity
is bounded by O(ni), where n is the number of features (|F|). The time and space
complexity of IW is exponential in the width of the problem w(Π), therefore if
the problem Π is solvable at width w, then IW(i) solve the problem in at most w

iterations of IW with a complexity of O(nw). IW(i) is a plain breath-ﬁrst search
(BFS) with one exception; when a new state s is generated we check if the state
is novel, is the state novel we keep it otherwise it is pruned. Novelty is based on
the i when IW(i).
Deﬁnition 5. [LG12] (novelty) A newly generated state s produces a new tuple
of atoms t iﬀs is the ﬁrst state generated in the search that makes t true. The
size of the smallest new tuple of atoms produced by s is called the novelty of s.
When s does not generate a new tuple, its novelty is set to n + 1 where n is the
number of tuple features with size 1.
To rephrase the ﬁrst state s generated in the search that makes an atom p
true has novelty 1. If s does not generate an atom that has not been true, but it
can create a pair (p, q) that is new the novelty is 2, and so forth. If s does not
generate any new tuple that has not been seen before the novelty is set to n + 1.
The higher the novelty of a state, the less novel the state is. When running IW(i)
the BFS treats states with novelty higher then i as duplicate states:
Deﬁnition 6. [LG12] IW(i) is a breadth-ﬁrst search the prunes newly generated
states when their novelty measure is greater then i.
When IW(i) is used where i = n then the algorithm prunes only true duplicate
states and can therefore be called complete. It is also interesting to observe that
lower values of i means more states are pruned and it is not complete. Speciﬁcally
the number of states not pruned when i = 1 is O(n), and i = 2 is O(n2) and so
forth, O(ni). Since there exists no states with novelty 0, IW(0) prunes all child
states, therefore the only situation where IW(0) solves the problem Π is if the goal
is true in the initial state s0. Since we cannot predict the width a problem the
planning algorithm work by running a series of IW searches IW(i) with increasing
i = 0, 1, 2, .... until the i is equal to the number of problem variables n.
Following the considerations above the algorithm is complete and sound, but
the algorithm is not necessarily optimal when solving a problem Π. It is possible
to create a problem Π of optimal width w (shortest possible solution) but the
problem is solvable at a smaller iteration IW(i) of i < w.
As an example, consider a problem Π on ﬁgure 2.6 where the actual width
of the goal G is 2, but it can be achieved none optimally in width 1. As can be
seen IW(2) achieves the goal cost 5 using action d, but this action is never used
in IW(1). The reason is when applying actions for ai for pj is true for j > 1 are
pruned, and when applying actions bi the states where qj is true for j > 1 are
pruned. Therefore for IW(1) the states where both (p2, q2) and (p2, q3) and more
are pruned and therefore it does not generate the state where the pair (p3, q3) is
true, which is required for achieving G optimally. However IW(1) does actually
solve the problem with a suboptimal path of cost 7 by using action bi to get to
state q7 where it is also possible to reach G.

2.4
RolloutIW
IW(i) is low polynomial time algorithm that does breadth-ﬁrst search (BFS) that
has shown to be eﬀective in classical planning and games. IW’s general structure
means it is not suitable for real-time planning, where actions have to be taken
within a few hundred milliseconds, because IW’s ability to explore deeper nodes
is limited by the BFS when planning within small time frames. Meaning IW
never reach nodes beyond a certain depth.
Bandres, Bonet, and Geﬀner [BBG18] proposes a new version of IW called
Rollout IW that is not limited by the BFS, but uses a diﬀerent exploration
technique that has better anytime behaviour.
In the following section, we introduce Rollout IW and show the algorithm
achieves the same results as IW when run til completion but with the drawback

with a features set F, where the feature set has size O(ni) where i is the width,

rewards and pick actions at random. We start by giving a short explanation

2.4.1
Extended features
As before we are given a problem Π with an initial state I, which is the root of
the search tree, where we run a number of rollouts from the root to create the
tree.
A rollout consist of a trajectory of states si and actions ai leading to a state s.
A node in the tree is a state where the trajectory share a common preﬁx and end
with the same state, and the edges of the tree are all actions. The breadth-ﬁrst
search is replaced by running progressive rollouts, but for the rollout to work we
need to deﬁne a new notion of novelty [BBG18]. To do this we deﬁne new feature
called ”extended features” (e-features).
E-features are discovered in the same way as regular features, therefore if a
feature f is true in a state, the e-feature (f, d) is true/exists, and f is part of the
feature set F. We also consider d, where d is an integer between 0 ≤d ≤|F|
corresponding to the depth of where the node that discovered the feature is in
the tree. This deﬁnitions means there can be several e-features for each state.
The e-features are used to update a table E with the depth of where a feature
was found.
If we ﬁnd an e-feature (f, d) we can assign it to the table with
E[f] = d.
The table is only updated if we are in a new state s with an e-
feature (f, d) at a lower depth than previously seen d < E[f], otherwise the table
is unchanged. In other words E[f] is the length of the shortest trajectory the
makes a feature f true. The features that are true in the root state are given
value zero and the rest are given value ∞.
The second addition is that nodes can be labeled as SOLVED, when they
no longer have e-features that are the same or higher than the value in the E
table. Action leading to SOLVED nodes are no longer considered when running
a rollout. The algorithm terminates when the root node is labeled SOLVED or
a budget (time, action count, etc.) for exploring has been reached. The pseudo
code for RolloutIW(1) can be seen in ﬁgure ?? and it is from [BBG18]. Before
giving a more formal description we give an example of the RolloutIW(1) in the
Blocks World problem, where 1 means we only consider tuples of size 1.

2.4.3
Rollout
A rollout starts from the root node and recursively extends the current trajectory
(n0, a0, n1, a1, ...) with an action a resulting a new node n, where the node holds
the state s and the action is chosen uniformly between possible actions. The
trajectory may already be in to tree or it is a new path. The only condition when
picking an action a is that it does not lead to a node labeled SOLVED. When
deciding whether a node is SOLVED or we continue a rollout, there are 4 cases
to diﬀerentiate between. If we have a rollout trajectory τ and we apply an action
a to the last node in the trajectory and get a new node n, these four possible
cases for evaluating n, which can also be seen in the pseudo code ??:
1. If the node n is new in the tree and it makes at least one e-feature (f, d)
true, where d < E[f] continue rollout and update E table with all true
e-features where d < E[f].
Page 37 of 113
==PAGE:45==
Chapter 2.
Width-Based Planning
Frederik Krieger Drachmann
2. If the node n is new in the tree but d ≥E[f] for all true e-features in n,
terminate rollout, and label node SOLVED.
3. If the node n is already present in the tree, but none of the true e-features
(f, d) have d = E[f], terminate rollout, and label node SOLVED.
4. If the node n is already present in the true, and an e-feature (f, d) is true
where d = E[f], continue rollout.
Cases 1 and 2 involve new branches, which leads to new nodes. The rollout
terminates if the new node does not bring a true e-feature, where the e-feature
(f, d) improves the depth of previously seen features d ≥E[f].
The rollout
continues if it does improve the depth.
In cases 3 and 4, we are in a node
that has previously been explored. It means diﬀerent from IW(i) a node can be
searched several times to explore all of its children. The search beneath such a
node stops when all its true e-features (fi, di) are at higher depth than in the E
table di > E[fi] for all i.
The only case where the table E changes is in case 1, where a new node is
added. The table E is updated with all the true e-features (fi, di) in the state,
that has a lower depth than in the E table, meaning if di < E[fi] for all i, then
we update E[fi] = di. In cases 2 and 3, the rollout terminates and the node is
labeled as SOLVED. The label is propagated up the tree to the parent, where
the parent also becomes SOLVED, if all the parents possible actions have been
applied and the nodes are SOLVED. It means the last SOLVED child will make
the parent SOLVED.
Case 4 handles when nodes are revisited. We have to handle this case since
every time a node is labeled SOLVED the rollout is terminated and restarted
from the root, but the tree and table E is not reset. We can therefore end in
situations where we revisit nodes, and we do not want to label them SOLVED
until all their applicable actions have been explored completely (SOLVED).
Following the cases above we can formulate about rollouts that:
Theorem 2. [BBG18] Each rollout improves the distance d to some feature f,
i.e. it reaches an e-feature (f, d) with d < E[f], or labels the child of a node in
the tree as SOLVED.
2.4.4
Analysis
The length of rollouts are bounded by |F| since worst case scenario we ﬁnd one
new feature per new node, and we can at most ﬁnd a new feature |F| times. It
holds for the e-features because every new feature would improve the E table and
therefore end in case 1 |F| times and end with an E table with numbers from 0
to |F|. The total number of diﬀerent nodes is bounded by how many times we
Page 38 of 113
==PAGE:46==
Chapter 2.
Width-Based Planning
Frederik Krieger Drachmann
can update the E table with e-features (f, d). Theoretically, we have the table
of features E which has size |F|, and we ﬁll the table with depth d, which we
deﬁned can take the values between 0 and |F|. The possible ways of assigning |F|
diﬀerent depths to an |F| big table is |F| × |F| = |F|2, meaning the max number
of diﬀerent nodes is |F|2, or O(n2) where n is |F|. This is an upper bound since
theorem 2 makes reaching |F|2 combinations impossible. Following this, we can
specify how many nodes we at most generate in a run Rollout IW. We just upper
bounded the number of diﬀerent e-feature assignments, which means the number
of times we can fall within case 1 (new node that improves an e-features depth)
in a RolloutIW, and every time we create a new node, we have to explore all its
children (applicable actions). Therefore the upper bound for number of generated
nodes is b×|F|2, where b is the number of applicable actions. Following the article
[BBG18], RolloutIW(1) is ensured to reach all width 1 goals within the b × |F|2
generated nodes:
Theorem 3. [BBG18] Rollout IW(1) is guaranteed to reach every width 1 goal
in time that is polynomial in the number of features O(n2).
From the complexity analysis it easy to see that running IW(1) wins over
RolloutIW(1) when running the algorithm to completion since it only generates
at most b × |F| nodes. What makes Rollout IW an interesting alternative is its
anytime behaviour gained from the rollouts which is suited for exploring. We
are mainly working with RolloutIW(1) since the complexity increases drastically
when working with higher width. This said we try and examine the implications
of increasing the width later on, but only for a few domains because of time
consumption.

2.5.1
Pixels
The input given from the Atari screen is a 160 wide and 210 high with pixels of up
to 128 colors. In principle when planning with IW/RolloutIW it is possible to use
only the pixels as the boolean vector 160×210×128 but is not necessarily smart,
since IW is dependent on quality of features. Speciﬁcally Rollout IW and IW
plans better if it has features that captures meaningful structures in the screen
and not just raw pixels. Therefore in [BBG18], the pixels features are mapped
into the visual set of features called B-PROST designed in [Lia+16] for the Atari
domains.
B-PROST
B-PROST is the combination of 3 diﬀerent features: Basic Features, B-PROS
and B-PROT.
Basic Features
The basic features are attained by dividing the Atari screen
in to 16 × 14 tiles of size 10 × 15 pixels. For each tile (c, r) and color k where
c ∈0, ..., 16, r ∈0, ..., 14 and k ∈0, ..., 128 we check if it contains a color k, if it
is true the ”tile” is 1 otherwise false. This means generating the binary feature
ϕc,r,k. The intuition behind creating these features in plain words is ”there is a
green pixel within this region of the Atari screen”. The total number of basic
features is 16 × 14 × 128 = 28, 672.
B-PROS
Basic Pairwise Relative Oﬀset in Space (B-PROS) captures the the
pairwise relative distance between basic features.
More speciﬁcally B-PROS
checks if a feature pair with colors k1, k2 ∈1, ..., 128 separated by an oﬀset

3.1
Probabilistic models
In machine learning we are often interested in learning a probabilistic model
describing various natural and artiﬁcial phenomena. These models can be used
for understating such phenomena and for predicting unknowns in the future and
autonomous decision making. In this speciﬁc case, we want to learn a probabilistic
model that can create a feature vector of zeros and ones given the screen from
the Atari domain.
These probabilistic models often contain unknowns, and only in rare cases
does the data paint a complete picture of the unknowns, and therefore we have
to assume some degree of uncertainty over the aspects of the model. The degree

and nature of the uncertainty can be described with (conditional) probability
distributions. The models can consist of both continues- or discrete-valued vari-
ables.
Let a bold lowercase letter, e.g., x, denote the observed distribution. Meaning
that x represent the set of observed variables, whose joint distribution we wish
to model.
The observed variable x is a random sample from an unknown underlying pro-
cess, whose true probability distribution is described with p∗(x). We approximate
this underlying process by creating a chosen model pθ(x):
x ∼pθ(x)
(3.1)
In this scenario, learning is the process of searching for a value for θ such the
the probability distribution function of pθ(x) approximates the true distribution
of the data. For any given observed values x denoted from p∗(x) we have:
pθ(x) ≈p∗(x)
(3.2)
When learning, we want pθ to be suﬃciently ﬂexible to adapt the to the data,
and thereby getting a more accurate model.
At the same time we want the
possibility of incorporating more data into the model that is known as priori.
Conditional models
In most cases of classiﬁcation and regression problems, we do not want to learn
an unconditional model pθ(x) but instead learning a model conditioned on some
factor y, pθ(y|x) that tries to approximate the underlying process p∗(y|x). This
means the distribution of values of variable y, is conditioned on the observed
variables x. When working with these models, x will often be the input of the
model. As before we approximate the true distribution for any x and y:
pθ(y|x) ≈p∗(y|x)
(3.3)
A common and simple conditional model is image classiﬁcation, where y is
manually labeled image classes and x is input images. We then wish to predict
the image class y from the images.
Here pθ(y|x) is typically chosen to be a
categorical distribution.

a scalable and ﬂexible type of function approximator. Deep learning is known as
neural networks with multiple hidden layers of neurons. In our case we are specif-

4.2.2
Optimization of planning
Risk averse
The negative rewards in the Atari domain are sparse or almost none existent.
Which means losing a life or dying does not give small enough negative rewards
compared to the positive rewards. The agent therefore prioritize positive rewards
over dying until it has only one life left, where the play style improves because
dying now means no more rewards. The problem was dealt with by multiplying
negative rewards with a constant α and adding −1 + ·α when the agent dies.
When the agent back propagates the rewards the branches with negative rewards
looks very undesirable. This version we call risk averse (RA) [BBG18].
Partial caching
We also added an optimization that is present in all version of VAE-IW, which is
called partial caching. Every time we visit a node and calculate the features we
save the features for the node. After we have exhausted the planning budget and
chosen a new root, some actions from the new root might have been executed in
the previous run and we can therefore reuse the features instead of calculating
them again. It saves computational time and it is also used in Bandres, Bonet,
and Geﬀner [BBG18] for all versions of IW and RolloutIW with B-PROST.

5.3
Detailed experiments
The experiments in the article looks directly at the averaged results, which gives a
good idea on the VAE-IW’s performance, but it is hard to evaluate what problems
it might have. In this section, we analyse the results in more detail to understand
where the model has problems and how we can potentially solve them, speciﬁcally,
we look at the single runs from the previous experiments, loss for the VAE, the
feature distribution of the encoder, the number of expanded nodes within the
0.5 time budget and we discuss how the game play of the Atari domains impact
performance. The conclusion we draw in the following section are based of the
current results, but properly show that they hold in all cases we need to run more
experiments.
5.3.1
Consistency
The results reported in the article (table 5.2) are based on an average of 5 runs, in
the appendix on table A.3 is the lowest, highest and standard deviation over those
5 runs. These results show that most domains vary a lot in scores over those 5
runs, where we would prefer a more consistent score. A very plausible explanation
for this is the inherently random nature of how actions are explored in Rollout
IW, that could result in exploring branches that reap very little rewards. It also
shows that even with the inherently random exploration structure of Rollout IW
the features often leads to rewards and thereby showing the features are successful
in learning meaningful structures in the games.
5.3.2

5.3.3
Game play considerations
Road Runner
The algorithm is purely exploratory and never learns from pre-
vious runs. It is therefore challenged by domains, where rewards are sparse and
planning for 0.5 seconds does not reach rewards, and therefore chooses a random
action in ”real-time”. An example of a game where this is the case is Frostbite
(RA VAE-IW: 254.0, RA B-PROST: 146, Human: 4,335). The player has to
cross a frozen lake by jumping on ice blocks which it gains rewards for, and every
time it jumps on a new block an igloo is build. When the igloo is done the player
has to go into the igloo, and the algorithm never does, because it never explores
the possibility. The game is also all about timing in the sense that it has to
wait for the right moment before jumping between ice blocks. This results in the
”best” trajectory having an overweight of ”No-Op” actions, but since RolloutIW
explores action at random there is only a small probability of it discovering this
path.
Elevator Action
The only domain where VAE-IW never succeeds in ﬁnding
any rewards is Elevator Action. The reason is the domain requires the player
to stand in an ”elevator” at the time it goes down and there are no rewards
going down the ”elevator”. Which means it is up to the VAE-IW exploration
to randomly go down the elevator and ﬁnd a reward to be guided in the right

6 Conclusion
In this work, we present width-based methods and the variational autoencoder
(VAE) and contributed with the combination of discrete VAE and width-based
methods on the Atari domain. Firstly, we showed the theory behind width-based
methods, where we focused on giving an understanding of width, and introduced
the adaptions to the Atari domain that has previously been made. Secondly, we
introduced VAE’s and how they enable us to learn features from a dataset of
images of an Atari domain.
At last we combined the two methods into VAE-IW and compared it to the
previous Rollout IW methods on the Atari domain and humans. We outperform
RolloutIW B-PROST on most domains and on some domains we greatly improve
the results for both plain VAE-IW and with risk averse rewards. We also beat
the professional human tester scores in over half of all of the domains. For some
domains the scores where even comparable to the best reinforcement learning
algroithms on the Atari domain.
These great results are achieved with 4500
features compared to the 20 million of B-PROST, which greatly reduces the space
consumption when planning and B-PROST features uses a temporal information,
where the VAE only gets information from the current frame, and that the VAE is
limited by only training on data collected by the RolloutIW B-PROST algorithm.
In general, it shows that the VAE learns a meaningful representation of the
Atari domains that can be eﬀectively used for planning with RolloutIW(1). The
results also show room for improving the VAE or the exploration of Rollout IW
to perform even better on the Atari domain.
6.1
Future Work
We showed that VAE-IW can plan better than previous methods and human
scores, but we discovered room for improvements in the VAE-IW and other in-
teresting experiments that can be conducted.

6.1.1
Hyperparameter tuning
The parameters used for the experiments here were collected by doing extensive
hyperparameter tuning, but they still have room for improvements. Especially
the network structure still have countless of combination that have not been tried.
6.1.2
Dataset
The models were trained on the data that was collected through the RolloutIW(1)
B-PROST [BBG18] algorithm. RolloutIW B-PROST is a decent algorithm but
it cannot hold its own against some of the state-of-the-art reinforcement learning
algorithms, and it performs especially poor on some domains. If RolloutIW(1)
B-PROST falls through and never reaches certain points in the Atari domain it
will no produce a screen dump of it in the dataset, and the VAE will never be able
to learn that part of an Atari domain. Therefore the performance of RolloutIW
B-PROST could be holding back VAE-IW, and by collecting a bigger and more
diverse training set we could improve performance.
6.1.3
Continuous-valued VAE
In this thesis, we binarized the encoder by using the categorical reparameter-
ization trick Gumbel-Softmax [JGP16] and thereby create a latent space that
resembles a Bernoulli distribution. One of the problems with this method is the
latent space size becomes an important hyperparameter since it controls the size
of the feature vector RolloutIW is planning with. Instead of utilising an encoder
that resembles a discrete distribution, it could be uyseful to model the latent
space with a Gaussian distribution N(0,1), which would support easier training
and proper reconstruction. When we have a models that can properly recon-
struct all our domains, we can chose a method for quantizing1 the output from
our encoder into a discrete vector of zeros and ones. The size of the featurevector
will then be independent on the VAE encoder, but depend on the method of
quantizing.
Utilising a Gaussian distribution would ensure that we can focus on the quan-
tizing process instead of whether our reconstruction is proper. Tuning the method

6.1.4
Ladder-VAE
An improvement could be to swap the discrete VAE with a discrete Ladder-VAE
[Søn+16], which has more expressive power and therefore learns better features.
The main concern with this approach it is really hard to train properly with
discrete values, but if successful it will probably improve performance. Of course,
we could use the approach explained before in section 6.1.3, where we create a
Ladder-VAE with continuous-valued output and quantize that output.
6.1.5
Policy guided
The strength of RolloutIW is its ability to explore deep nodes in a short time
windows, but exploring at random is not eﬃcient if we want to maximize score.
Therefore a method was proposed called π-IW [JJG19], where an action policy
(Neural Network) is trained by running RolloutIW(1) to explore for some budget
(100 nodes, 0.5 seconds etc.), when ﬁnished the maximum discounted reward
is calculated for each action at the root. These rewards are saved to train the
action policy in a supervised manner. The RolloutIW(1) uses the action policy
when exploring and thereby guiding the search when the network is trained.
The features used for planning with π-IW is the output from the last layer of
the action policy network after having applied ReLU. In [JJG19] the output
of the action policy is only 128 neurons, which means it plans based of 128
features, which a very small feature set, but it still gets great scores. This method
greatly resembles that of AlphaZero[Sil+18], which uses Monte Carlo Tree Search
(MCTS) for exploration instead of RolloutIW(1).
Therefore they compare π-
IW with AlphaZero on very simple domains (so a feature for RolloutIW is easy
to deﬁne) and show the exploratory power of RolloutIW is better than MCTS
[JJG19]. We believe adapting the π-IW with the features from the VAE’s could
improve performance even further, and it could be interesting to see if RolloutIW
could compete with MCTS at exploring.

A.2.1
Forward pass
We model the latent space p(z) as Bernoulli distribution because we want the
latent space to have a structure where we can easily obtain binary values that can
be used for planning. In a Bernoulli distribution there are only two probabilities
determine whether it returns 1 or 0, and only one p is needed to calculate the
the other q = p −1. We model our encoder and decoder so they only accept the
p of the Bernoulli distribution since we can then ﬁnd the other. Thereby we can
decrease the size of the network. It also means the total size of the latent space is
the number of features we can plan on afterwards 15×15×20 = 4, 500. We have to
do a trick after having passed the image through the encoder, because the latent
space does not hold the second value of our Bernoulli distribution. Therefore we
append zeros to the latent space 15×15×20 →15×15×20×2. We can then pass it
through the Gumbel-Softmax reparameterization 3.3 as a categorical distribution
of two (Bernoulli) and to approximate a distribution where all values are either
zero or one. After having applied the Gumbel-Softmax we remove the last column
again 15 × 15 × 20 × 2 →15 × 15 × 20 × 1 and pass this to the decoder ensuring
we can keep the encoder and decoder symmetric. The principles here generalizes
to any kernel size model, where the kernel size is M and the z-dimension is N,
M × M × N.
