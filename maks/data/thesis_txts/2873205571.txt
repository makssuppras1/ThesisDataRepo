Abstract
Microservices architectures have gained popularity for developing scalable software sys­
tems due to their modular design and inherent resilience. However, the complexity arising
from design patterns that enhance resilience can obscure inter­service communication,
making it difficult to diagnose issues. There is a pressing need for system­specific re­
silience testing scenarios to proactively address potential inconsistencies and resource
unavailability in microservices.
To address these challenges, this thesis presents a software tool that automates resilience
testing in microservices architectures. By utilizing data from service meshes and dis­
tributed tracing tools, the tool analyzes real­time traffic and traces to identify critical fault
injection points. The tool automates the chaos experiments generation using open­source
tools, employing a cyclical process to refine test cases based on previous faults that oc­
curred in microservice architecture systems.
The resulting contributions of this work include automating the resilience testing pipeline
that relies on using data, improving fault injection by targeting important service inter­
actions, easily working with current, industry­proven DevOps tools, and pushing forward
research in automated resilience testing for microservices.
This thesis demonstrates that open­source, data­driven tools can effectively manage the
complexities of microservices architectures, thereby improving system reliability and main­
tainability when it comes to resilience testing. However, the framework offers limited guid­
ance at the component level, suggesting future work could focus on more detailed support
for decision­making.

Introduction
Many established entities in the industry still employ the architecture known as a mono­
lithic software architecture. Within computer science, while the meaning of ”monolithic”
has evolved, in this context, it refers to an application that is deployed as a single instance,
codebase, or repository. The application is traditionally developed by a single develop­
ment group, and any changes made by developers can potentially affect the entire set of
services because all changes are made within the same codebase [1].
Microservice Architecture (MSA) is a design paradigm that decreases coupling by pur­
posely dividing a system into smaller, loosely coupled services that can operate, be de­
ployed, maintained, and modified independently.
This approach improves flexibility, the ability to expand and ease the deployment pro­
cess, allowing organizations to innovate faster and adjust services as needed [2]. This
architectural design has various benefits, including greater agility, scalability, and main­
tainability.[3]. However, as the number of microservices grows, so does the complexity
of orchestrating these services, particularly when it comes to effectively maintaining their
fault tolerance and scalability [4].
Chaos engineering is a discipline focused on improving system resilience by proactively
testing how systems behave under failure conditions. Chaos engineering extends con­
ventional testing by intentionally injecting service specific faults into live systems to as­
sess their resilience under stress [5]. Although tools in this domain have matured with
automation and structured methodologies, fault injection often remains random, as seen
in fuzz testing, where malformed inputs are indiscriminately introduced into microser­
vices. However, the strategy of injecting faults into all possible services is inefficient.
It is both resource­intensive and time­consuming, particularly in complex microservice ar­
chitectures. Current research lacks a systematic approach to identify the most impactful
fault injection points [6]. A more targeted method would not only streamline testing but
would also enhance the effectiveness of resilience assessments.
1.1
Problem Statement
Distributed nature of MSAs introduces significant challenges in ensuring overall system
resiliency. In particular, when testing how these systems can handle occurred failures
during operation. The key issue lies in the exact selection of fault injection points in re­
silience testing within MSAs.
Expressed by Hui et al., “the employment of fault injection techniques in resilience testing
has yet to fully address the challenges related to the selection of injection points.” [6]
The existing approach of evaluating each service within MSAs as a suitable candidate
for fault injection raises significant concerns about time expenditure and resources. This
fear is particularly exacerbated with large­scale MSAs, where the number of services
is substantial, which can lead to high testing overhead, often overlooking the important
vulnerabilities in business­critical areas. [6]
The inefficiency of traditional fault injection methods is further aggregated by the manual
effort needed in chaos engineering practices. An existing solution, called ChaosEater, ad­
dresses the aforementioned issue of addressing the labor­intensive nature of defining hy­
potheses and planning chaos experiments by proposing a chaos engineering automation

in current resilience testing methodologies by addressing the manual, time­consuming
processes of hypothesis formulation, experiment planning, and system reconfiguration,

To address these challenges, there is an existing need for a systematic framework that
can efficiently identify optimal fault injection points in MSAs.
As emphasized by Hui et al., “An efficient strategy for selecting pivotal injection points
could markedly reduce the extraneous workload, underscoring an area ripe for innovation

Such a framework would not only reduce the resources and time spent with manual labor
associated with resilience testing, but would also enhance the effectiveness of detecting
critical vulnerabilities within MSAs.
The problem statement addressed in this thesis is:
How can we systematically identify and prioritize optimal fault injection points
in microservices architectures to enhance the efficiency and effectiveness of
resilience testing?
Based on the explicated problem definition, the following research questions have been
formulated:
1. What is the effective method for identifying key fault injection points and executing
them in a microservice architecture to thoroughly test system resilience and scala­
bility?
2. How does prioritizing pivotal injection points based on monitoring data affect
the overall resilience of microservices systems?
3. To what extent does the proposed automated tool reduce the time and effort required
for resilience testing in microservices compared to traditional manual methods?
This thesis attempts to address deficiencies and research questions by developing
a conceptual framework for the systematic identification and detection of fault injection
locations, utilizing almost real­time data and service dependencies for prioritized pinpoint­
ing of fault­injection points.
The thesis structure follows with the exploration of identification methods, then assessing
the impact of prioritization on resilience of MSAs, and finally an evaluation of the efficiency
of the proposed tool against current practices. In addressing these research questions,
the thesis aims to achieve a structured resilience testing strategy that minimizes testing
overhead while improving the identification of critical vulnerabilities in MSAs.

Methodology
As for how to approach the problem this thesis is trying to, the principles of design science
from a book by Johannesson and Perjons were used.
Design science is a scientific research method that utilizes so­called artifacts, built specif­
ically within the research context, to demonstrate a solution to practical problems. The
aforementioned artifacts, being a key component of design science, are used as a model
or a demonstration of practical problems a given research is trying to tackle. The devel­
opment of the artifact itself serves as a bridge between the current status of problems
and the desired outcomes. Design science produces not only the artifact, but also the
corresponding science discovery arising from the artifacts themselves. [8]
Design science can be compared with empirical science, which includes the natural and
social sciences. In empirical science, researchers describe, explain, and predict. In de­
sign science, researchers create and construct objects to enhance practices and thus
change the world. [9]
Within the content of the book, a framework for working through practical problems is
proposed through five activities: [10]

2
Background and Related Work
Following the preceding chapter, we stated that that a systematic, automated approach
to resilience testing in microservice architectures (MSAs) is necessary. The dynamic and
distributed nature of MSAs renders manual testing insufficient. This chapter presents the
current state of automated solutions that attempt to facilitate chaos testing for system
resilience.
2.1
Microservice Architectures and Resiliency Mechanisms
Microservices are an architectural style in which a software application is composed of
a collection of small, independent services that communicate over a network. Each ser­
vice encapsulates a specific business capability and can be developed, deployed, and
scaled independently [2]. Because these services are loosely coupled, a failure in one
microservice is less likely to cascade and cause a failure of the entire system. In practice,
organizations often assemble a dedicated team of engineers to be responsible for each
microservice, allowing teams to work autonomously and deploy updates without affecting
other services.
Fallback mechanisms are a set of design patterns and strategies intended to mitigate
faults in MSAs and thereby make the overall system more resilient. In a distributed mi­
croservices environment, independent services communicate across networks and each
typically manages its own data. This distributed nature means failures are not merely pos­
sible but inevitable—network issues or service outages will eventually occur. As a result,
fault­tolerance techniques must be implemented in production to ensure reliable service
delivery, mitigate outages, and maintain system availability for users. These mechanisms
act as safeguards that allow the system to adapt and maintain functionality when individ­
ual components fail. Technically, a fallback mechanism provides an alternative response
or workflow when a service cannot perform its normal action.
Several common fault­tolerance patterns are used in industry to achieve resilience: [11]
• Bulkhead Pattern — This pattern isolates components by allocating separate re­
sources for each, preventing a failure in one part of the system from exhausting
resources in others. Inspired by ship bulkheads, it ensures that if one service faces
high load or crashes, it does not cascade to bring down other services. Proper re­
source allocation is key to avoid under­ or over­provisioning of these isolated pools.
• Circuit Breaker — The circuit breaker pattern prevents cascading failures by moni­
toring the success/failure of service calls and breaking the circuit, like stopping calls
to a particular service, if a threshold of consecutive failures is reached. The circuit
breaker has states like closed (normal operation), open (calls are blocked to the
failing service), and half­open (the system tentatively allows a few calls to test if the
service has recovered). This pattern is effective for handling unreliable networks or
downstream service failures, but it requires careful tuning of the failure thresholds
and reset timings to balance responsiveness with.
• Retry Pattern — In this pattern, a failed operation is automatically retried a certain
number of times, which can help overcome transient issues. Retries are often im­
plemented with exponential backoff delays to avoid overloading the system. It is
important to set an upper limit on retries to prevent infinite loops or retry storms that
could themselves cause additional load or resemble denial­of­service behavior.

• Fallback Pattern — The fallback pattern provides a predefined alternative response
when an operation cannot be completed normally. For instance, if a service call
fails, the system might return cached data or a default value, or route the request to
a backup service. Although the fallback response might be less accurate or slower,
it ensures that the system continues to function in a degraded mode rather than
failing completely. In practice, fallbacks are often used in combination with circuit
breakers and retries to handle failures comprehensively.
• Cache Pattern — Caching involves storing copies of frequently accessed data in a
fast storage layer so that future requests for that data can be served quickly with­
out hitting the primary database. Caches improve performance and also act as a
resilience mechanism: if the primary data source is temporarily unavailable, the sys­
tem can often continue serving requests from cached data. Key considerations for
this pattern include cache invalidation and consistency as well as eviction policies
to manage limited cache space.
2.2
Monitoring and Metrics
In order to maintain the reliability and performance of a microservices­based system, it is
essential to monitor various runtime metrics that reflect the system’s health and behavior.
Modern monitoring systems collect and store time­series data from the services in real
time, allowing engineers to observe trends over both short and long periods. By analyzing
these metrics, one can detect anomalies, identify performance bottlenecks, and receive
alerts when certain indicators breach predefined thresholds, thereby facilitating proactive
incident response.
One common approach is to use a centralized metrics database coupled with a query lan­
guage to evaluate conditions on the collected data. For example, Prometheus 1 provides
a high­dimensional time­series data store and a powerful query language for extracting
and aggregating metrics [12]. Using such queries, engineers can compute statistics over
the collected metrics to assess the system’s state and verify whether it meets reliability
objectives. In this work, PromQL queries are utilized to gather specific indicators relevant
to resilience testing.
Complementary to metrics collection, visualization tools help make sense of the data.
Platforms like Grafana 2 enable the creation of real­time dashboards that display key per­
formance indicators and historical trends from multiple data sources [13]. These dash­
boards make it easier to monitor the system’s behavior at a glance and quickly spot any
abnormal patterns or degradations. They can also incorporate alerting rules to notify op­
erators when metrics exceed acceptable ranges. Notably, such visualization tools are
designed to be intuitive and visually appealing so that operators spend more time observ­
ing the system’s health rather than configuring the monitoring setup [12, 13].
2.3
Distributed Tracing
While metrics provide aggregated insights into system performance, distributed tracing
offers a granular view of how individual requests propagate through a microservices ar­
chitecture. Observability in a broad sense encompasses metrics, logs, and traces—all
of which help engineers infer the internal state of a system and diagnose problems. Dis­
tributed tracing, in particular, tracks the path of a single transaction or a request as it
traverses multiple services, recording timing information and metadata at each step [14].

This yields a trace that shows how long each service took to handle the request and how

In practice, each trace is composed of spans, where each span represents the work done
by one service or one component as part of the overall request. Tracing systems tag each
span with context information, such as a trace ID, span ID, parent relationships, times­
tamps, and operation details, so that the spans can be stitched together to reconstruct the
end­to­end execution flow [15]. By examining a trace, engineers gain a request­centric
perspective on system behavior, which is invaluable for pinpointing sources of latency or
failures in complex workflows [14]. For instance, a trace might reveal that a particular
microservice call is consistently slow or returning errors, allowing the team to target that
service for optimization or fixes.
Tools like Jaeger 3 provide infrastructure for collecting and visualizing traces. A tracing
dashboard displays each request’s journey across services, making it straightforward to
identify which part of the workflow is the bottleneck or where an error occurred [14, 16].
Traces that show abnormally long durations or unusual errors can highlight issues that
may not be evident from aggregate metrics alone. On the other hand, the volume of tracing
data in a large system can be overwhelming, and it remains a challenge to automatically
sift through hundreds of traces to find the most critical anomalies. Engineers often must
rely on a combination of metrics and traces, along with domain knowledge, to focus their
investigation on the most impactful problems, since current tracing tools do not always
direct attention to the rare but significant failure scenarios.
2.4
Service Mesh
A service mesh is an architectural pattern that offloads the service­to­service communica­
tion logic from microservices to a dedicated infrastructure layer. It consists of a network
of lightweight proxies deployed alongside each service that intercept all incoming and
outgoing service traffic. This provides a centralized way to implement traffic manage­
ment, resilience policies, and observability across a microservice cluster without requir­
ing changes to the microservices’ own code. Service meshes operate at the network
communication layer (typically Layer 5–7 of the OSI model), and are generally protocol­
agnostic, making them suitable for polyglot environments where different services use
different communication protocols [17].
In a typical service mesh architecture, there is a data plane comprised of the distributed
proxies and a control plane that manages those proxies. The data plane handles the ac­
tual routing, load balancing, encryption, and monitoring of service calls. The control plane,
on the other hand, is responsible for configuring these proxies with rules and policies that
govern the traffic. For example, in the Istio service mesh, each microservice pod includes
an Envoy sidecar proxy, and Istio’s control plane instructs these proxies on how to route
requests, how to enforce security, and how to collect telemetry data about the service
interactions [17]. By decoupling these cross­cutting concerns from the application code,
a service mesh ensures that features like retries, timeouts, circuit breaking, and traffic
shaping are applied uniformly and consistently across all services. This not only simpli­
fies the microservices themselves but also improves the overall resilience and security
of the system by making behavior consistent. Many organizations adopt service mesh
technology to gain fine­grained control over inter­service communication and to increase
the reliability and visibility of their microservices applications without significant changes
to application logic [17].

2.5
Chaos Engineering for Resilience Testing
Chaos engineering is a methodology of experimenting on distributed systems by intention­
ally introducing failures in order to verify that the system can withstand turbulent conditions
in production. The goal is to build confidence in the system’s resilience by observing how
it behaves under stress and by uncovering weaknesses or hidden dependencies before
they lead to serious issues in real outages [18, 19]. Unlike traditional quality assurance
testing, typically covering predictable or controlled scenarios, chaos engineering involves
creating chaos experiments that simulate unexpected severe events in a production or
production­like environment. The system’s ability to maintain its steady state (normal be­
havior meeting service­level objectives) during and after these experiments indicates its
level of resilience [20].
Netflix famously pioneered this approach with its “Simian Army” tools, particularly Chaos
Monkey, which randomly terminates instances in production to test whether the remain­
ing system can continue to function properly. This practice forced Netflix’s engineers to
design microservices with graceful degradation and automatic recovery in mind. Netflix
later expanded on this with tools like Chaos Kong (which simulates the failure of an en­
tire data center or cloud region) and Failure Injection Testing (FIT) which injects faults
into the communications between services to ensure they handle errors gracefully [19].
These early experiments underscored that deliberately ”breaking things in production” can
reveal systemic weaknesses that are hard to find through conventional testing. Chaos en­
gineering views a distributed system holistically; by injecting real­world failure conditions
and observing the outcome, engineers learn about how faults propagate and whether the
system’s redundancy and fallback mechanisms work as designed [19].
Chaos engineering has since been applied to evaluate self­healing or self­adaptive sys­
tems, which are designed to automatically adjust to failures. In Chapter 3, we will further
explain the core principles of chaos engineering and how they can be used to increase
resiliency in MSAs.
2.5.1
Chaos Engineering Tools for Fault Injection
To execute chaos experiments in practice, engineers often rely on dedicated tools that
can inject faults in a controlled, automated manner. A chaos engineering tool provides
the ability to introduce specific failure scenarios into a system (such as shutting down a
service instance, adding network latency, or exhausting a resource) in order to observe
how the system responds [21]. These tools are designed to carry out failure injections
safely and reproducibly so that experiments can be conducted without causing permanent
harm to the system or undue customer impact.
A recent survey of chaos engineering solutions [21] highlights several common features of
these tools. For instance, many integrate with container orchestration platforms to target
specific microservices or infrastructure components, and they offer declarative interfaces
to define and manage chaos experiments. One widely used example is Chaos Mesh, an
open­source framework that provides a broad range of fault injection capabilities for Ku­
bernetes environments. With Chaos Mesh, engineers can simulate pod crashes, network
delays or partitions, CPU or memory exhaustion, and other failure types. It also supports
scheduling experiments and orchestrating multiple faults as part of more complex sce­
narios. However, like many chaos engineering platforms, Chaos Mesh does not natively
determine whether the system handled the induced faults correctly; it only confirms that
the faults were injected successfully. Assessing the outcome requires external obser­
vation of metrics, logs, or traces generated during the experiment [22]. In other words,
while chaos tools greatly assist in automating the injection of failures, they typically must

The ChaosEater framework [7] leverages large language models (LLMs) to automate the
chaos engineering lifecycle in Kubernetes environments. It is capable of generating hy­

Evaluations across both small and large Kubernetes clusters have demonstrated its ef­

with a high success rate, as verified by human engineers and the LLM itself. This repre­
sents a step toward scalable, hands­off resilience testing, where chaos experiments can
be continually run and interpreted with minimal human intervention.
In contrast, another recent study [23] leverages the Istio service mesh to facilitate re­

introduce failures (such as simulated network latency and service errors) at the network

a tool like Locust to generate traffic) to observe how the system behaves under stress.
Throughout the experiments, monitoring and tracing tools (Grafana dashboards, Jaeger
traces, etc.) are used to gain visibility into service behavior and performance while faults
are occurring. The authors also incorporate common resilience techniques to see how
effectively these measures mitigate the impact of the injected faults. By using a service
mesh, this approach can systematically trigger failures in a controlled manner and exam­
ine their effects on inter­service communications, helping to identify weaknesses such as
bottlenecks or error propagation in the architecture.

Conceptual Framework
This chapter introduces a conceptual framework for resilience testing in microservice ar­
chitectures, with a focus on systematic fault injection and analysis. The framework inte­
grates established chaos engineering principles with an automated, data­driven approach
to identify, inject, and evaluate faults within complex distributed systems. By structuring
resilience testing into clearly defined stages, the framework aims to address the limitations
of manual and random fault injection, as outlined in previous chapters.
The framework presented here extends existing methodologies [7, 23] by introducing an
automated module for generating tailored chaos experiments based on real­time teleme­
try. This approach reduces manual effort, increases coverage of critical system paths,
and provides actionable insights for improving microservice resilience.
3.1
Fault Injection Strategy Design
Fault Injection Strategy Design is a pivotal phase in resilience testing frameworks for mi­
croservices. It involves the formulation of structured, intentional, and hypothesis­driven
plans to inject faults into specific components or communication paths of a distributed
system. The strategy design determines which faults to introduce, where and when to in­
troduce them, the intensity and duration of faults, and what outcomes or system behaviors
are expected during and after fault injection.
This phase serves as a foundation for safe and meaningful chaos experiments. A robust
strategy ensures that failures are not injected arbitrarily but are guided by system knowl­
edge, service­level objectives, and identified resilience goals. The types of faults typically
considered include network latency, packet loss, service crashes, resource saturation,
particularly on CPU and memory, and database connection failures. These faults are
aligned with real­world failure modes observed in production environments.
In the context of microservice­based systems, the significance of strategy design becomes
even more pronounced due to the architectural complexity and fine­grained service inter­
actions. The loosely coupled nature of microservices makes the system highly susceptible
to cascading failures if a single service becomes degraded or unresponsive. Therefore,
designing a fault injection strategy helps isolate fault domains, prioritize test targets based
on service criticality, and validate that fallback or retry mechanisms behave as intended
under failure conditions.
Fault Injection Strategy Design also integrates tightly with fault injection tools like ChaosMesh,
Gremlin [25], and Filibuster [24]. ChaosMesh allows fine­grained fault control within Ku­
bernetes environments, enabling experiments such as killing pods or simulating network
faults [18]. Gremlin provides fault scenario orchestration by manipulating network traffic
between microservices using recipes defined in Python [25]. Filibuster takes this further
by applying Service­Level Fault Injection Testing (SFIT), automatically injecting faults at
service interaction points during the development phase, based on existing test cases
and static analysis [24].
Moreover, resilience testing must consider not only service failures but also data layer
failures. The extension of Filibuster introduced by Assad et al. enables database­specific
fault injection, including Byzantine faults and client­side exceptions, which are often ne­
glected in traditional chaos engineering practices [26].

3.2
Experiment Orchestration
Experiment orchestration refers to the process and infrastructure responsible for coor­
dinating, automating, and managing the execution of resilience testing experiments in a
structured, safe, and reproducible manner. In the context of fault injection and chaos en­
gineering, orchestration ensures that experiments are not only conducted systematically
but also integrated into existing operational workflows, enabling continuous validation of
system robustness without introducing uncontrolled risks.
A core responsibility of the orchestration layer is scheduling and automation. It defines
when and how experiments are triggered, often according to predefined policies or CI/CD
pipelines. This allows chaos experiments to run autonomously as part of nightly builds
or deployment verifications, ensuring resilience checks become a routine part of software
delivery. Tools such as Chaos Mesh and LitmusChaos natively support time­based or
event­driven experiment scheduling, enabling engineers to inject faults like pod failures,
CPU stress, or network delay on a fixed cadence or upon deployment events [18, 7].
Another vital task of orchestration is the coordination of distributed components across
the system. Fault injection scenarios often involve multiple microservices, infrastructure
nodes, or networking layers. Therefore, orchestration must manage the sequence of
actions, such as shutting down pods, isolating services, or injecting latency on specific
communication paths. For example, Gremlin allows operators to define recipes that ex­
press ordered failure scenarios by manipulating inter­service traffic at the network layer,
enabling orchestration of multi­step, interdependent faults [25].
To maintain system safety during experimentation, orchestration tools implement safety
controls such as blast radius limitations, abort conditions, and guardrails. These con­
trols ensure that fault injections are restricted to designated zones or environments and
can be immediately halted if critical thresholds (e.g., error rates, service unavailability)
are exceeded. This is especially important in production­like or shadow environments,
where unbounded experimentation could affect customer­facing functionality. Gremlin
and LitmusChaos both implement such safeguards, allowing developers to define toler­
ance thresholds and trigger automatic rollback when system stability is compromised [7,
25].
Additionally, orchestration supports workflow management for executing complex fault
injection scenarios. These workflows may include multi­stage experiments involving con­
current or sequential faults, as well as comparative analyses using control and experimen­
tal groups. For instance, LitmusChaos supports chaos workflows that compose multiple

3.3
Fault Injection Execution
Fault Injection Execution is the operational phase in which the predefined fault scenarios,
as outlined in the strategy and orchestration stages, are actively introduced into the target
system. This component is responsible for materializing faults in a live or test environ­
ment, enabling the empirical evaluation of system resilience. The fault execution layer
must support precise targeting, safety constraints, reversibility of injected conditions, and
comprehensive observability to monitor the system’s response in real time.
There are several categories of fault injection approaches, each targeting different layers
of the system stack:
1. Service Mesh/Network­Level Fault Injection: One common execution method is
through service meshes such as Istio, which allow faults to be injected directly at
the network layer between services. By configuring traffic rules in the sidecar proxy
(e.g., Envoy), engineers can simulate realistic networking issues such as HTTP 500
responses, gRPC delays, or complete connection failures between microservices.
This method is particularly valuable for testing service­to­service reliability and retry
behaviors. Tools like Chaos Mesh offer integration with service meshes to facilitate
such network­level disruptions in Kubernetes environments, supporting fine­grained
control over traffic manipulation and latency injection [18, 7].
2. System­Level Faults: These faults perturb the underlying system infrastructure,
simulating hardware or operating system­level issues. Typical examples include
killing containers or pods, consuming CPU or memory resources, triggering disk I/O
bottlenecks, or introducing kernel­level network delays. LitmusChaos and Chaos
Mesh both provide rich libraries of system­level chaos experiments that simulate
these failures safely in Kubernetes clusters. For example, an experiment may kill
specific pods to test deployment resilience or stress a node’s memory to trigger
autoscaling behavior [18, 7].
3. Application­Level Faults: Application­level faults involve injecting errors into the
application’s execution logic, allowing the simulation of internal exceptions, timeout
behaviors, or incorrect responses. This form of injection is usually performed with
fault injection libraries embedded within the application codebase. One such tool
is Simmy, a .NET­based fault injection library that allows developers to inject faults
into specific methods during runtime. Similarly, Filibuster facilitates fault injection
at service boundaries by modifying method return values, injecting exceptions, or
altering control flow during integration testing without modifying the core service
logic [24].
4. Environment­Level Faults: Environment­level fault injection focuses on simulating
failures in the cloud or datacenter infrastructure. This includes scenarios such as vir­

3.4
Metrics Analysis and Result Interpretation
Metrics analysis and result interpretation constitute the evaluative phase of a fault injection
experiment. This component is responsible for analyzing telemetry data collected during
chaos experiments to determine the impact of injected faults on the system’s behavior.
The ultimate goal is to assess system resilience, validate or refute hypotheses, and derive
actionable insights to improve system robustness. This phase leverages observability
tools—such as distributed tracing, metrics collection, and logging frameworks—to gather
real­time and post­experiment data for comprehensive analysis.
One core function of this component is anomaly detection. This involves comparing ob­
served system metrics—such as latency, error rates, CPU usage, throughput, or ser­
vice availability—against pre­established baselines or service­level objectives (SLOs).
Anomalies such as unexpected spikes in latency, cascading service failures, or degraded
throughput may signal that the system failed to tolerate the injected fault gracefully. Mod­
ern observability stacks integrated into Kubernetes environments enable automated threshold­
based alerts and visualization of anomalies during test execution [18, 24].
Another essential aspect is correlation and causality analysis. Here, metrics and traces
are time­aligned with the timing of the injected fault to determine causal links. Distributed
tracing, such as that provided by OpenTelemetry or Jaeger, enables engineers to pinpoint
how a fault propagated across services, which dependencies were impacted, and whether
fallback mechanisms or retries were triggered. This capability is central to systems like
Filibuster and Gremlin, which incorporate failure observability into their testing frameworks
to detect downstream consequences of injected faults [24, 25].
In hypothesis assessment, the experiment outcome is evaluated against predefined ex­
pectations. For example, if the hypothesis states that a service should remain available
during a dependent service failure, but observability data shows a crash or timeout, the
hypothesis is considered falsified. Assad et al. emphasized this logic in their database
fault injection tool, which verified whether services degraded gracefully under simulated
Byzantine and network faults at the data layer [26].

3.5
Recovery Verification and Resilience Validation
The recovery verification and resilience validation phase represents the concluding but
critical part of a fault injection experiment. It is responsible for assessing whether the
system has successfully recovered from the injected fault within an acceptable period
and whether its behavior aligns with the resilience objectives defined in the fault injec­
tion strategy. This component validates that the system has not only survived disruption
but has also restored its operational and performance baselines without compromising
functional or non­functional requirements.
One of the fundamental tasks in this phase is steady­state restoration. After a fault is
injected and subsequently terminated, the system is expected to return to its normal op­
erating parameters, as defined by metrics such as response time, throughput, error rate,
or CPU utilization. The system’s return to these baselines is an indicator of its ability to
self­heal or recover autonomously. As emphasized in CHESS—a framework for evalu­
ating self­adaptive and self­healing systems—verifying post­fault stability is necessary to
ensure the system remains within its viability zone [27]. Observability tools integrated into
modern cloud­native stacks support continuous metric tracking, allowing for automated
comparisons between pre­fault and post­fault states [18].
Equally important is the verification of fallback mechanisms, which tests whether resilience
design patterns have activated appropriately during the disruption. These mechanisms
are intended to prevent failure propagation and maintain partial functionality. Tools like
Filibuster are designed to inject faults specifically at the service interaction layer and mon­
itor whether application­level fault tolerance logic—such as fallback method invocation or
retry­on­failure blocks—executes as expected [24]. Failure to activate these mechanisms,
or their malfunction under load, can indicate resilience gaps not visible in normal testing
workflows.

3.6
Design Considerations and Tradeoffs
Designing a fault injection framework for microservices­based resilience testing entails
several key decisions, each involving tradeoffs between realism, control, safety, system
load, and analytical depth. This section outlines and analyzes these tradeoffs, drawing
from current literature and operational tools used in chaos engineering.
Manual vs. Automated Experiments
Manual fault injection enables direct control over the execution of chaos scenarios and
is well­suited for early­stage testing or targeted investigation of specific failure modes.
However, it does not scale and is prone to human error. Automated approaches, by con­
trast, allow for repeatable, scalable, and continuous testing within CI/CD pipelines. The
ChaosEater framework demonstrates the benefits of automation, where chaos experi­
ments are defined and deployed automatically using Infrastructure­as­Code and Kubernetes­
native mechanisms, minimizing human intervention while maximizing reproducibility and
efficiency [7]. The Chaos Engineering Chapter also emphasizes that automation supports
safe experimentation at scale, provided robust safeguards—such as rollback conditions
and monitoring thresholds—are enforced [20].
Static vs. Dynamic Target Selection
Static fault targeting relies on predefined knowledge of critical system components, en­
abling simpler test planning and controlled execution. However, this approach may miss
emergent failure paths in evolving microservice topologies. Dynamic target selection, in­

System­Level vs. Network­Level Fault Injection
System­level fault injection involves perturbing the infrastructure or operating system,
such as killing pods, stressing CPU or memory, or forcing disk I/O contention. These

Running experiments in pre­production environments limits the risk of harming end­user

should only proceed when system health checks, alerting mechanisms, and observability

Single­Fault vs. Multi­Fault Scenarios
Single­fault experiments are simpler to implement and analyze, allowing teams to isolate
the effects of a specific failure—such as a database timeout or service crash—and vali­
date the behavior of resilience mechanisms like retries or fallbacks. However, real­world
outages often involve multiple concurrent or cascading faults. The ChaosEater frame­
work includes examples of multi­fault testing, such as simulating a spike in load during
network degradation to uncover resilience gaps that single­fault scenarios may miss [7].

concolic execution, avoiding full­scale observability instrumentation in early development
stages [24]. Gremlin similarly enables fault detection through low­overhead network mon­

3.7
Chaos Engineering Principles
Chaos engineering is a discipline dedicated to improving system resilience by intentionally
injecting controlled failures into distributed systems. Its primary aim is to ensure that a sys­
tem can maintain its steady­state behavior—its expected performance and functionality—
despite adverse conditions such as network faults, service crashes, or resource exhaus­
tion. This methodology is particularly crucial in microservice architectures, where the in­
terplay of loosely coupled services introduces complex failure modes that are not always
visible in traditional testing environments [27].
The success of chaos engineering hinges on a set of foundational principles. These prin­
ciples guide the design and execution of experiments, ensuring they are effective, repeat­
able, and safe. This section elaborates on these core principles, drawing from established
literature and operational frameworks.
3.7.1
Steady­State Behavior Definition
The first and most fundamental step is to define what constitutes ”normal” or steady­
state system behavior. This is typically expressed in terms of quantifiable metrics, such
as request latency, CPU utilization, throughput, or error rate. These metrics serve as a
baseline against which deviations—caused by injected faults—are measured. The Chaos
Engineering Chapter recommends establishing these metrics as part of a control group,
which remains unaffected by fault injection, allowing for effective comparison against the
experimental group [20]. Similarly, ChaosEater enforces structured steady­state defini­
tions for Kubernetes resources, prioritizing the identification of states most susceptible to
disruption [7].
3.7.2
Real­World Scenario Simulation
To derive actionable insights, chaos experiments must simulate real­world failures that
a system is likely to face. These may include network partitions, disk or memory sat­
uration, application crashes, or service timeouts. The principle is to not limit testing to
high­probability events alone, but also to incorporate low­frequency, high­impact scenar­
ios, such as cloud region outages or cascading service failures [20, 27]. Gremlin and
Chaos Mesh both support these simulations, offering fault injection modules for network,
I/O, and compute failures across services [25].
3.7.3
Conduct Experiments in Production
While it is safer to initiate chaos testing in staging environments, production systems ulti­
mately provide the most accurate evaluation of system behavior under stress. Variability
in traffic, user interactions, and data scale makes production the only environment that can
fully reflect real­world system conditions. According to the Chaos Engineering Chapter,
testing in production enables discovery of latent dependencies and architectural weak­
nesses that would otherwise go unnoticed [20]. However, to mitigate user impact, such
experiments must be designed with strict guardrails—such as limiting scope, enforcing
rollback conditions, and continuously monitoring key performance indicators (KPIs) dur­
ing the experiment.
3.7.4

Chaos engineering is most effective when conducted as a continuous practice, not as a
one­time assessment. Automation enables this by integrating fault injection into CI/CD

Post­experiment analysis is essential to understanding system weaknesses and reinforc­

Jaeger, and spans with the same request ID are linked to form end­to­end distributed

application­level metrics and resource­level metrics.

5.4
Real­Time Feature Extraction
Using the incoming telemetry, the injection­detection service performs real­time feature
extraction to quantify the importance and health of each microservice. These features
form the basis for prioritizing fault injection targets. The key metrics­based and trace­
based features used by the Evaluating Engine include:
• Metrics­Based Features: From Prometheus metrics, the service extracts several
important indicators per service edge:
– Latency (P99): High­percentile response times, the 95th or 99th percentile
latency to be exact, for requests handled by each service. An elevated P99
latency suggests that a service is experiencing slow responses under load or
has performance bottlenecks.
– Error Rate: The fraction or percentage of requests that result in errors (HTTP
5xx or other failure codes) for each service. A high error rate indicates reliability
issues in that service or in calls to its downstream dependencies.
– Request Throughput: The request rate (requests per second) for each ser­
vice, indicating its traffic load. Services with very high throughput are critical to
overall system performance and user experience.
– Resource Utilization: CPU and memory usage levels for each service’s pods.
Services near resource saturation (e.g., consistently high CPU utilization or
memory pressure) might be more prone to performance degradation or crashes
under stress.

• Trace­Based Features: From Jaeger’s distributed tracing data, the service derives
structural and performance insights:
– Service Dependency Graph: By analyzing recent traces, the orchestrator
identifies the topology of service calls, including which services have many
downstream dependencies or act as critical gateways. For example, a ser­
vice that is called by many others (high fan­in) or calls many others might be
infrastructure­critical. A central service that appears in many traces is highly
critical to user workflows.
– Critical Path Latencies: The tracing data can reveal if certain services are
consistently part of the slowest path in end­to­end transactions. For instance,
the traces might show that calls to recommendationservice add significant la­
tency to overall requests.
– Error Propagation in Traces: Traces can be analyzed to see if failures in one
service propagate to others. If a particular service’s failure causes many trace
spans to end in errors, that service is a candidate for fault injection to test failure
handling.
– Fan­in/Fan­out Degree: Numerically, the orchestrator can compute how many
direct upstream and downstream services each service has from the depen­
dency graph. A service with a high fan­in (many callers) is a single point of
failure for many others; a service with high fan­out (calls many others) could
fail due to one of its dependencies failing. Both scenarios are important to test.
Using these features, the Evaluating Engine maintains a dynamic assessment of each ser­
vice’s criticality and health. The combination of high traffic, high latency, elevated errors,
and a central position in the call graph would, for example, yield a high priority score for a
service. Real­time analysis is crucial: as the system conditions change, for instance, a
surge in traffic to a particular service or a new dependency introduced, the feature values
update, and so do the priority scores. This ensures that the chaos experiments remain
adaptive to current conditions, rather than relying on static assumptions. In essence, the
system aligns with the ideals of intelligent chaos engineering – focusing test efforts where
the system is most vulnerable at any given moment. For example, if the recommenda­
tionservice suddenly shows a sharp increase in P99 latency and is part of many critical
user transactions, the orchestrator will recognize it as a prime fault injection target. By
contrast, a service with low load and no errors would be deprioritized. This data­driven
targeting helps concentrate the chaos experiments on components whose failure would
likely have the greatest impact, thereby maximizing the usefulness of each experiment.

• Target Selection (Prioritization): The Evaluating Engine computes a priority score

and resource usage) and trace­based importance (such as dependency centrality).
Services that score above a certain threshold or the top­ranked service(s) are se­

• Chaos Experiment YAML Generation: Once a target service or multiple services

controllers detect the new custom resource and begin the chaos experiment. For
example, if a NetworkChaos resource was created, the Chaos Mesh controller will

specified pod at the scheduled time. The injection­detection service may also log
the action or update an internal state to indicate an ongoing experiment.
• Monitoring and Lifecycle Management: The orchestrator does not simply fire off
an experiment and forget about it—it continues to monitor the system during the
fault injection. It observes how the target service and the overall application re­

example, did the error rate of the target service go to 100% upon killing its pod? Did

plan the next experiment. This approach provides scalable, data­driven resilience testing,
systematically probing the system’s weak points as they are discovered.

Design Rationale for a Service­Based Implementation
An important consideration in this project was how to implement the fault prediction and
injection logic. We chose to build the injection­detection functionality as a standalone
microservice with FastAPI as its interface running inside the Kubernetes cluster, rather

• Modularity and Separation of Concerns: By structuring the solution as a mi­

and maintained independently. In contrast, a monolithic script implementing all logic
would be harder to manage or extend. The service­based approach also makes it
easier to update or replace parts of the logic without impacting the rest of the system.
• Maintainability and Extensibility: A continuously running service in the cluster
can be versioned, containerized, and configured like any other microservice. This
means standard DevOps practices can be applied to the fault orchestrator itself.
The codebase is organized into a service with an API, which naturally enforces a
cleaner architecture than a throwaway script. New features can be added to the
service over time. Additionally, as a Kubernetes­native service, it can easily scale if
needed, for example, if the computation or query load grows, we could run multiple
replicas or a more powerful instance.
• API Access and Integration: Implementing the orchestrator with FastAPI means
it exposes RESTful endpoints. This design allows external tools or users to inter­
act with the orchestrator if desired. For instance, an API endpoint could trigger an
immediate analysis, retrieve the current scoring of services, or manually approve/­
deny a suggested experiment. In a script­based approach, there is no easy way for
external systems to interface or fetch its results in real time. The API also paves the
way for a UI dashboard or integration into pipelines, for example, integrating with
a continuous delivery pipeline to run chaos experiments automatically after deploy­

Implementation Decisions
This chapter clarifies the key implementation choices made for injecting faults and testing
resilience in a microservice­based environment. We describe the rationale behind using
a local Kubernetes cluster, integrating a service mesh for observability, and automating
fault injection through Chaos Mesh. Each design decision is explained in terms of how

6.1
Kind – Local Kubernetes for Testing
During the development of this project, I decided to mock a Kubernetes environment
locally instead of using a managed cloud cluster such as Google Kubernetes Engine
(GKE)1. The reason for this choice is that rather than relying on a proprietary third­party
from the get­go, I wanted to utilize open­source tools as much as possible. There are
many ways to run Kubernetes locally, such as Minikube2, which is a well­established tool
for provisioning a single­node cluster. However, Minikube can be bloated and cumber­
some on low­end machines. That is why I opted for Kind (Kubernetes IN Docker)3.
Kind is a tool for running local Kubernetes clusters using Docker container “nodes.” It
is lightweight and well­suited for simulating a multi­node cluster environment on a local
machine. Using Kind offered a repeatable and isolated test environment without incurring
cloud costs or complexity. It also ensured that all components of the resilience testing
framework could be run locally, which was convenient for iterative development and de­
bugging.
Another advantage of using Kind was the fine­grained control over the Kubernetes ver­
sion and cluster configuration. This allowed installing necessary add­ons in a controlled
setting. In summary, the decision to use Kind provided a cost­effective, open­source, and
easily reproducible environment for implementing and validating resilience testing strate­
gies, without being tied to vendor­specific infrastructure.

tion in algorithm.py creates appropriate fault scenarios based on service metrics. The
logic includes:
• Pod Failure vs. Pod Kill: For services with high error rates (above 1%), a Pod­
Chaos experiment with pod-failure action targets all pods to simulate a complete

outage. For healthy services, a pod-kill action targets one pod to test redundancy.
• Network Delay: For services with high P99 latency (above 500 ms), a NetworkChaos
experiment introduces additional latency to test timeout and fallback mechanisms.
• CPU Stress: For services with CPU usage above 80%, a StressChaos experiment
burns CPU cycles to simulate high computational load.
• Memory Stress: For services with memory usage above 80%, a StressChaos ex­

Each experiment is configured as a dictionary conforming to Chaos Mesh’s CRD defini­

prolonged disruption.
6.3.3
Automated Execution and Orchestration
In execution mode, the /run-mode endpoint applies the generated experiments using the
Kubernetes API, enabling continuous resilience testing. This automation reduces manual
effort and can be integrated into CI/CD pipelines or scheduled tasks. After injecting faults,
the observability stack (Prometheus and Jaeger) detects disruptions, allowing analysis of
whether fallback mechanisms or graceful degradation occur.
6.3.4
Discussion of Design Principles
The implementation decisions reflect several key resilience engineering principles:
• Isolation and Redundancy: Kubernetes and Istio ensure service isolation and re­
dundancy, tested via pod­kill experiments to verify load balancing and failover.
• Monitoring and Observability: Prometheus and Jaeger provide the data founda­
tion for identifying weak points and driving chaos experiments, aligning with AIOps
practices.
• Proactive Failure Testing: Automated chaos experiments uncover weaknesses
before production outages, adapting dynamically to system changes.
• Systematic and Automated Testing: The pipeline is application­agnostic, using
external metrics and infrastructure­based fault injection for broad applicability.
• Graceful Degradation and Fallbacks: Fault injections test whether services de­
grade gracefully, validating mechanisms like timeouts and retries.
In conclusion, the implementation choices attempt to create a self­driving resilience test­
ing system. This approach demonstrates how chaos engineering principles can be oper­
ationalized in a development environment to identify weaknesses proactively.

Evaluation
In this chapter, we evaluate the effectiveness of our automated resilience testing tool by
applying it to two specific scenarios: pod kill and network delay, as described in Chapter

to demonstrate how our tool can identify and execute fault injections more efficiently than
manual methods. Through these evaluations, we aim to answer the research questions:

2. How does prioritizing pivotal injection points based on monitoring data affect the

3. To what extent does the proposed tool reduce the time and effort required for re­

The evaluation compares our tool’s automated approach with a traditional manual ap­

Conclusion
The adoption of microservice architectures has significantly transformed software devel­
opment by enabling modular, independently deployable services that enhance scalability
and flexibility. However, this modularity introduces substantial operational complexity,
particularly in ensuring system resilience against failures. To address these challenges,
tools such as Kubernetes, Istio, and Chaos Mesh have emerged, providing robust mech­
anisms for observability, traffic management, and chaos engineering.
The objective of this thesis was to develop an automated resilience testing framework
that leverages service mesh telemetry capabilities and chaos engineering fault injection
mechanisms to systematically identify and test critical fault injection points in microservice
architectures. Through this work, we evaluated the effectiveness of automated chaos
experiment generation in enhancing system reliability, demonstrating a practical approach
to mitigating the complexities inherent in distributed systems.
8.1
Summary of Contributions
Although the approach presented in this thesis introduces automated chaos test gener­
ation for microservice resilience, it does not yet match the full complexity and scope of
robust frameworks like the aforementioned ChaosEater [7]. In contrast, our implementa­
tion is narrower in scope, focusing on a predefined set of fault injection scenarios identified
through Istio telemetry. The lack of flexibility when it comes to extending fault models or
customizing workloads for specific domains is present. This disparity highlights a limitation
of our work, the absence of the rich extensibility and domain­aware workload generation
that characterizes ChaosEater’s comprehensive chaos engineering platform. As such,
the complexity and breadth of ChaosEater remain beyond what has been achieved here,
marking an area for potential future enhancement.
Nevertheless, the contributions of this thesis represent a significant advancement over
previous Istio­based resilience testing methods. Karn et al. [23] showed that using an
Istio­based approach can effectively perform fault injection and monitoring in a service
mesh, but it doesn’t have ways to expand chaos experiments or manage them beyond
the scenarios that are set up manually. By contrast, our approach leverages the Chaos
Mesh framework to automatically generate and execute chaos test cases at identified fault
injection points, introducing a degree of automation and orchestration. Rather than relying
solely on handwritten fault scenario YAML files, the system developed in this thesis pro­
grammatically creates a suite of chaos experiments to systematically validate microser­
vice resilience. This automation expands the original Istio­based methods into a more
dynamic and data­centric testing process, enabling broader exploration of potential faults
and their impacts on the system. In essence, our work augments the Istio service­mesh
approach with a chaos experimentation pipeline that is both extensible and automatically
orchestrated, bridging part of the gap toward more sophisticated chaos engineering solu­
tions.
While the solution presented in this thesis does not yet reach the advanced feature set of
platforms, it significantly enhances the automation of resilience testing compared to earlier
service­mesh­centric frameworks [23]. The use of Chaos Mesh to drive automatic chaos
experiments represents a clear step forward, reducing the manual effort required to design
chaos tests and increasing the coverage of failure scenarios that can be tested in a given

microservice architecture. This improvement moves the state of practice closer to the
vision of comprehensive, automated chaos engineering and lays important groundwork.
8.2
Future Work
The work presented in this thesis establishes a robust foundation for automated resilience
testing in microservice architectures, but several areas remain for further development
to enhance its capabilities and align it more closely with advanced chaos engineering
frameworks. The following directions outline potential future work to address limitations
and expand the scope of the proposed tool.
Enhancing the fault modeling capabilities may be a reasonable next step. The existing
implementation depends on a predefined set of fault types that Chaos Mesh provides.
To enhance flexibility, future projects might integrate a modular fault injection library, en­
abling users to establish bespoke fault models suited to certain application domains or
failure scenarios. This entails augmenting the injection­detection service to accommodate
dynamic fault model plugins, thus allowing the system to adjust to various microservice
contexts and identify a wider array of resilience challenges.
Integrating our tool into continuous integration and continuous deployment (CI/CD) pipelines
would facilitate continuous resilience testing. The current implementation operates in a
controlled test environment using Kind, but embedding the tool within CI/CD workflows
would enable automated resilience validation as part of regular development cycles. This
could be achieved by developing a pipeline stage that triggers the injection­detection ser­
vice to generate and execute chaos experiments whenever new code is deployed or in­
frastructure changes are made. Such integration would align with the principles of con­
tinuous chaos engineering outlined in [20], ensuring that resilience testing becomes a
routine practice rather than a one­off activity.
Exploring the application of the proposed tool in production environments represents a
significant opportunity. While the current work focuses on a controlled test environment
to avoid risks, even though the case study chosen for this project resembles an almost
production­like setting. Testing the tool in production, as advocated by [20], would provide
a more accurate evaluation of system resilience under real­world conditions.
Future work could investigate strategies for safe production testing, such as limiting the
blast radius of chaos experiments through canary deployments or feature flags. This
would involve enhancing the injection­detection service to incorporate production­specific
observability data and developing strict guardrails to minimize user impact, as suggested
in [27]. Such an approach would enable the tool to uncover latent dependencies and
architectural weaknesses that are only apparent in production settings.
However, real­world systems may involve hundreds of services, increasing the compu­
tational load on the injection­detection service. Future work could optimize the teleme­
try processing pipeline, perhaps by leveraging distributed computing frameworks or ma­
chine learning techniques to prioritize fault injection points more efficiently. Additionally,
exploring integration with advanced observability tools, such as machine­learning­driven
anomaly detection, could enhance the tool’s ability to identify critical services in complex,
dynamic topologies.

A
Technical Details and Supplementary
Material
This appendix provides supplementary material to support the main findings of the the­
sis, including source code, technical specifications, experimental setups, additional re­
sults, and user documentation for the developed resilience testing tool. These materials
are intended to enhance transparency, reproducibility, and usability for researchers and
practitioners interested in automated resilience testing for microservices architectures.
A.1
Source Code of the Developed Tool
The software tool developed in this thesis automates resilience testing by prioritizing ser­
vices and generating chaos experiments. Below is a key excerpt of the service priori­
tization algorithm implemented in Python, which calculates a composite score for each
service based on metrics from Prometheus and Jaeger. The full source code is available
in the project repository at https://github.com/rad0r0man/fault-injection-detector.
A.2
Detailed Technical Specifications
The resilience testing tool is designed as a plug­in that integrates with existing observ­
ability and chaos engineering tools. It consists of three main components:
• Telemetry Processor: Collects and processes real­time data from Prometheus
(metrics) and Jaeger (traces).
• Fault Analyzer: Identifies critical fault injection points using a composite scoring
mechanism.
• Experiment Generator: Creates Chaos Mesh experiment YAML files based on
prioritized services and fault scenarios.
A.3
Experimental Setups and Configurations
The case study was conducted using the Google Cloud Microservices Demo Application
(Online Boutique) on a local Kubernetes cluster created with Kind. The environment was
configured as follows:
• Kubernetes: Version 1.21, deployed using Kind.
• Istio: Version 1.10, for service mesh observability.
• Prometheus: Version 2.28, for metrics collection.
• Jaeger: Version 1.24, for distributed tracing.
• Chaos Mesh: Version 2.1, for fault injection.

A.6
Mathematical Models and Algorithms
The service prioritization algorithm uses a composite score to rank services, calculated
as:
Score = (Request Rate×(Trace Frequency+1))+(0.5×CPU Usage)+(0.5×Memory Usage)
This formula prioritizes services with high request rates and trace frequencies while con­
sidering resource usage. Services without traces are assigned a score of zero to avoid
unnecessary fault injections.
