Abstract
The fourth industrial revolution, along with ongoing developments in the automotive
& aerospace industries, creates a high demand for cyber-physical systems.
Indus-
trial and safety-critical applications increasingly depend on high-bandwidth, network
backbones to operate.
Ethernet networks have long been established as the most successful networks in
the world, due to their flexible and open nature. This flexibility is based on the fact
that Ethernet messaging is conducted using a best-effort strategy which is challeng-
ing to isolate temporally and to guarantee bounded message delivery latency. This
issue has prevented Ethernet from use in safety-critical, real-time applications com-
monly needed in aerospace, automotive or industrial fields. Two notable solutions for
this need are Time-Triggered Ethernet (TTEthernet) and Time Sensitive Networking
(TSN), which provide time-aware Ethernet communication for safety-critical applica-
tions. These aim to ensure bounded end-to-end communication latency with minimal
jitter, utilising a combination of Time-Division Multiple Access (TDMA) strategy,
and rate-constraint strategies. However, TTEthernet is a proprietary solution while
TSN is currently in development as an open-standard, resulting in a lack of open
time-aware switch designs for use in research.
This thesis proposes and implements a system that switches incoming time-aware
traﬀic flows by applying a static schedule, while achieving clock synchronisation using
the IEEE 1588 Precision Time Protocol (PTP). The design provides four Ethernet
ports. Packet switching is handled by a hardware controller that executes a commu-
nication schedule by employing three run-time configurable parameters: activation
time, period, and active window for each port.
The controller’s main function is
to switch incoming Ethernet frames between the ports that are active according to
the timing schedule. To achieve correct timing, the switch is synchronised using an
external hardware PTP unit that ensures the hosts and switch schedules will meet.
The design integrates a real-time soft-core processor Patmos, to process the PTP
synchronisation and control the hardware timestamp unit.
Finally, the design is evaluated in terms of functionality, resources, speed, and
power consumption metrics. An analysis on the switch end-to-end latency and clock
synchronisation is presented.

CHAPTER 1
Introduction
A Cyber-physical system (CPS) is a recent term that evolved from the embedded
systems’ world to express a more holistic approach to classifying systems interacting
with the physical world. We refer to systems as embedded systems when the focus is
on controlling or monitoring software more than the controlled or monitored physical
aspect. A CPS is looking at interfacing the digital world to the physical in a compre-
hensive manner, taking into account both parts and their interaction. As Edward A.
Lee writes in [14]:
“CPS is about the intersection, not the union, of the physical and the cy-
ber. It is not suﬀicient to separately understand the physical components
and the computational components. We must instead understand their
interaction.”
CPS is found in a multitude of applications, from factory floors to operating rooms
and from management systems for city traﬀic lights to fly-by-wire aircraft controllers.
All these applications have in common is their requirement to operate in perfect
synchronisation with their physical environment. We can rephrase and say that they
must operate in “real-time”.
A real-time system’s operation entails that its result’s temporal correctness must
also be guaranteed, along with its computational validity. When designing a real-time
system, a set of timing requirements for each computational task is defined, commonly
referred to as a deadline. There are three main types of real-time systems:[10]
• Hard, when a missing deadline is considered a failure
• Firm, when some missed deadlines can be tolerated
• Soft, when missing deadlines can cause quality of service degradation but no
functional failures.
There have been two core paradigms in real-time system design: event-triggered
(ET) and time-triggered (TT). In ET, the application’s response is dependent on
an external event in which the system must respond, whereas, in TT, the system
responds according to a given schedule. Evidently, the latter method provides high
determinism, while ET is generally more flexible. This results in a real-time design
process that aims to strike a balance between the two, depending on the application

e.g. automotive, aerospace, industrial automation, medical, etc., increase their com-

implement a prototype implementation on a networking hardware development plat-
form. The device must achieve clock synchronisation via IEEE 1588 Precision Time
Protocol (PTP) and operate on a runtime-configurable schedule. Provide an evalua-
tion of the implemented design.

Background
2.1
Ethernet and packet Switching
The vast majority of the communication networks deployed in systems nowadays is
based on a layered protocol stack model. This model was standardised in 1983 as
the Open System Interconnection (OSI) model[8]. It consists of seven protocol layers,
each outlining a specific functionality required for communication:
1. Physical layer: It consists of protocols concerning the physical transmission
of bits across a physical medium.
Timing, encoding, and signal processing
considerations are considered here
2. Datalink layer: It is used to establish point-to-point communication over the
physical link.
Data is organised in frame units, and flow control and error
correction are performed
3. Network layer: Data is being transferred as data units over the links estab-
lished in the previous layers. The main scope of these protocols identifying of
the route the data units will follow to reach their target
4. Transport layer: End-to-end communication between end-systems over the
network defined in the previous layer is established in this stage. Often, trans-
port layer protocols provide reliability, end-to-end error detection, retransmis-
sions, and flow control
5. Session layer: In this level, sessions of data exchange are set up between
network nodes. A session can consist of multiple transport layer connections
6. Presentation layer: It is responsible for presenting exchanged data in formats
that the application layer can be consume.
7. Application layer: It includes protocols that implement or facilitate end-to-
end distributed applications over the network
The first three layers build the low-level based functionality of an Ethernet net-
work, while the higher layers concern application protocols and functions. An Ether-
net network is typically organised from several host devices, Ethernet hub or switch

can have 100% throughput at the expense of resource cost, as the internal fabric must

In real-time system design, buffered designs pose a significant challenge as queues
introduce jitter and lead to missed deadlines, which is not acceptable in hard real-time
systems. As we will see further on, the most successful real-time Ethernet solutions
tend to favour a bufferless design. If messages are passed on a strict schedule, and
only one input/output connection is active, as in TDM, then a buffer is not needed
as the message is not intended to wait at all in the switch. A strictly scheduled com-
munication scheme can also eliminate the possibility of a single input monopolising
the switch and blocking all other traﬀic, either by a technical failure or as a mali-
cious attack. On the other hand, most real-world applications often need to exchange
real-time traﬀic interchangeably with non-time-critical traﬀic. In such a way, there
is a need to find methods and trade-offs that can accommodate each application’s
requirements.

Packets
Data in Ethernet networks are organised in “packets” or frames. These terms are
used interchangably for the rest of this work. These packets are transmitted over the
network, occupying one physical channel until the transmission is over. This routing
and transferring of data using addressed packets is called packet switching, where each
packet consists of a header and payload section. The header is used to determine the
packet’s source and destination, while the payload carries the data needed by the user
==PAGE:23==
2.1 Ethernet and packet Switching
9
application.
Each packet contains the protocol data from all layers encapsulated using the
header-payload design.
Fig. 2.8 illustrates this for the first three layers, that are
relevant to this project. The smallest unit of an Ethernet packet is referred to as a
nibble and is four bits, usually transported in the physical layer in each clock cycle.
Two nibbles form an octet (Byte), and the first stage of encapsulation is called an
Ethernet frame, and it contains the protocol data for layers one and two. An Ethernet
frame begins with seven octets of the preamble and one octet of a frame delimiter
value, both satisfying layer one requirements. The preamble consists of alternating
0 and 1 values that help the devices to synchronise their clock over the transmission
medium.
The delimiter is a byte that breaks the pattern in the second to least
significant bit (10101011), marking a byte and the start of the frame.

2.2
Real-Time Networking
As discussed in the introduction, real-time systems have strict temporal requirements
which the default Ethernet protocol stack cannot meet. As Ethernet development’s
focus in the past decades was to fulfil demands for best speed and quality of service for
the internet, and industrial real-time applications were based on serial communication
schemes. Evidently, this posed severe limitations in scalability and interoperability
within deployed systems. Thus a transition to Ethernet solutions offered substantial
advantages. Apart from the vastly improved performance, Ethernet is also utilised on
a standardised hardware base. With various plug-and-play capabilities, it facilitates
==PAGE:26==
12
2 Background
scalability and seamless integration from the field to the company level. However,
standard Ethernet uses several systems that prevent deterministic operation, and
therefore it has been not readily implemented in applications with real-time require-
ments. Even in modern switched full-duplex Ethernet setups, where data collision on
the bus is not an issue, data are stored in queues, they have not set send deadlines,
and are not even guaranteed not to be dropped due to timeouts.
There have been many attempts to develop real-time IE and provide a solution
to the challenges that hinder Ethernet network from time-critical applications.A real-
time system can be defined as a system whose response does not violate a given time
limit. There are three classes of real-time systems depending on the extent to which
the imposed time conditions are met. Hard real-time conditions are defined by the
punctuality requirement given in Eq. 2.1 [3]:
A ≡r + ∆e ≤d.
(2.1)
In detail, the activation time of a task is noted with the term r, and it executes
over the time ∆e. The task must then be completed by time d. In Eq. 2.2, B shows
that during ∆e from r to d, no other tasks nor technical failures can occur to block
the specific task. Hard real-time systems are characterised by fulfilling this condition.
P(A|B) = 1.
(2.2)
Thus the three classes of real-time capabilities of IE systems can be defined as:
• Class 1: Soft real-time, scalable cycle time 100ms
• Class 2: Hard real-time, cycle time 1 to 10ms
• Class 3: Isochronous real-time, cycle time 250µs to 1ms with jitter less than
1µs
Also, following the categorisation presented in [3], there are three categories to
characterise the IE system that will be presented in this work along with the proposed
design. The categories are shown in Fig. 2.12:
• Category A: Both real and non-real-time services use standard Ethernet stack
and controllers
• Category B: Both services use standard Ethernet controllers but also special
protocols for real-time on top of Ethernet
• Category C: Real-time services require dedicated process data protocols and
they additionally require special controllers and switches

2.2.1
Modbus-TCP
Modbus-TCP[26] is an example of a soft-real time system that operates over standard
Ethernet devices and protocols in the application layer, making it one of the few
category A solutions. This high-layer level placement makes it easily configured and
also has a high degree of reconfiguration. It follows a client-server architecture where
each device can assume these roles, offering greater reliability. The server initiates
a request while the clients can respond either successfully or with an error message.
The number of devices on Modbus-TCP is only limited by the processing power of
the devices.

2.2.2
TCnet
TCnet[2] is an extended IEEE 802.3 LAN method that utilises four new communica-
tion classes.
• Cyclic data with high timing requirements (class 3)
• Cyclic data with medium timing requirements
• Acyclic data
• Cyclic data with low timing requirements
To facilitate the new access method, special Ethernet controllers are required
(category C). Cyclic data transmission takes place inside a sub-network, and the
station participating in real-time communication must keep a copy of shared memory.
Therefore, real-time communication only occurs to process data being reflected in the
256KByte memory defined by Toshiba.
2.2.3
CC-Link IE Field
CC-Link IE Field by Mitsubishi Electric is based on full-duplex Gigabit Ethernet on
arbitrary topologies and can achieve hard real-time requirements (class 2). CC-Link
IE Field also uses a master-slave architecture, where the master initiates data trans-
mission cyclically by issuing a token to slave devices. For the implementation, special
Mitsubishi hardware is required for Seamless Message Protocols implementation (cat-
egory C), making it challenging to implement by third-parties.
2.2.4
SERCOS III
SERCOS III [29] is also using a time-slot scheme by a master passing a Data Telegram
token to slaves. Slaves can communicate directly with each other and the Telegrams
are based on a standard Ethernet frame that transports a limited amount of data.
SERCOS III can be realised in hardware to achieve isochronous (class 3) requirements,
making it category C, or it can be completely implemented in software (category
B). SERCOS III base feature is that it is organised in a double ring topology with
hardware redundancy, preserving communications even on link failures. However, this
feature places an upper limit of 511 devices, inhibiting scalability.

Ethernet Powerlink

PTP
However, real-time systems required higher levels of precision. For this reason, the
IEEE 1588 Precise Time Protocol (PTP) was introduced [17]. PTP allows for sub-

devices, and it uses a master-slave organisation instead of a client-server. This allows
for precise calculation of each slave’s offset relative to the time master by considering
the message’s propagation delay. Each Ethernet interface must be designated as a
PTP master or a PTP slave, and this can be defined explicitly or implicitly by a
master selection algorithm. PTP follows the flow depicted in Fig. 3.2. The master
initiates the syncing process by issuing a sync message that contains the timestamp
of the time it was sent (t1). The master can optionally issue a follow-up sync with
the original timestamp (t1) as not all masters can present an accurate timestamp on
the sync message. The slave timestamps the arrival of the sync message as time t2
and issues a delay request message to the master at time t3. The master responds
with a delay response that includes the received time t4. Then we can find that the
offset is:
θ = (t2 −t1) + (t4 −t3)

The slave then is aware of the offset value needed to be in sync with the master.
An important consideration is to perform this transaction periodically to keep clocks
in sync as the clocks will drift out of sync. Time stamping can also affect the precision
of time synchronisation. The IEEE 1588v2 standard states that timestamping should
occur at the last bit of the Ethernet Start of Frame Delimiter (SFD) of a received
frame. There are three common types of implementing this function:
• Software-based timestamping layer 4: Less accurate of the three, has
been shown to achieve sub-millisecond synchronisation under special conditions.
Frame has to traverse the complete network stack
• MAC-based timestamping (layer 2): More accurate than software-based
as timestamping is handled by dedicated hardware on the data-link layer when
the frame is received from the PHY
• PHY-based timestamping (layer 1): The most accurate timestamp method
as timestamping occurs in silicon where it is shown to achieve nanosecond ac-
curacy
Finally, the standard does not define a specific clock correction algorithm to adjust
the slave clocks, and different approaches have been implemented.

2.5.1
Comparison of FPGA implemented switches for Reliable
Ethernet
The goal of [1] is to compare two Commercial-Off-The-Shelf (COTS) switch IPs imple-
mented on Field Programmable Gater Arrays (FPGA). The switch IPs are following
the International Electrotechnical Commission (IEC), High-availability Seamless Re-
dundancy (HSR) Ethernet protocol, and Parallel Redundancy Protocol (RTP) for
power substations automation (IEC 62439-3). HSR provides redundancy by sending
packets in both directions of a ring topology, with the aid of Doubly Attached bridg-
ing Nodes HSR capable (DANHs). HSR frames are not compatible with standard
Ethernet devices; instead, Parallel Redundancy Protocol (PRP) is used to provide
this capability. These protocols, while very reliable, are not designed to meet real-
time requirements. The authors claim that low-latency hardware implementation can
meet the ”real-time” of power substation control processes requirements. Two COTS
IPs are then compared: a mixed cut-through switch IP that uses cut-through forward-
ing for PRP ports and save and forward for other ports, with a pure save and forward
switch IP. In conclusion, as is evident in Fig. 2.21, the cut-through architecture is
much lower in latency than the save and forward switch. The switch design proposed
in this thesis is using a cut-through approach, not including the Ethernet controller
used, as will be discussed in Section 3.
2.5.2
Enhancing real-time communication over COTS switches
In this paper [16], the authors attempt to enhance the temporal behaviour of COTS
Ethernet switches. They propose an extension of the Flexible Time-Triggered (FTT)
communication paradigm called FTT-Ethernet. The FTT-Ethernet protocols, origi-
nally developed by the University of Aveiro in 1998, essentially define a time-triggered
master-slave Ethernet communication.
They are also characterised by an online-
scheduling mechanism that enables greater scalability as well as plug-and-play capa-
bilities. The suggested extension is an adapted FTT-Ethernet protocol that segments
networks, which are called FTT-SE, by connecting only one station per switch port.
Communication in FTT protocols is conducted in fixed time slots called Elementary
Cycles (EC). One master message termed a Trigger Message (TM), is sent in each
cycle and contains the schedule for that EC. Regural aperiodic traﬀic takes place in
the time left between the scheduled time slots.

A time-triggered Ethernet switch (TTEthernet)
Finally, the design of a TTEthernet switch is presented in [25].
As seen in Sec-
tion 2.2.8, the switch is one of the cornerstones of a TTEthernet network since it
enables the distinction between TT traﬀic and ET traﬀic. One of the big advantages
of this approach is that it offers the possibility to allow competing senders without
affecting the traﬀic’s temporal determinism. ET traﬀic is used for competing senders,
while TT is reserved for cooperative senders with real-time requirements. TT traﬀic
preempts all other ET standard Ethernet traﬀic on the transmission path. In this
manner, it guarantees both compatibility to standard Ethernet devices on the net-
work and deterministic scheduled message passing for all cooperative senders. It is
important to note that this capability is not present in any other designs presented
in this section, where non-cooperative senders can easily flood and overwhelm the
network either maliciously or by failure.
Figure 2.25: Example TTEthernet network end-system components [25].
In Fig. 2.25, a typical TTEthernet setup is shown.
Each node can either be
TTEthernet enabled with a TTEthernet controller participating in TT traﬀic or it can
be a standard Ethernet host where ET traﬀic scheduling must be used for messaging.
To summarise a TT Ethernet switch must:
• Classify packets into TT and ET traﬀic
• Keep constant transmission delay for TT traﬀic
• Have an unbuffered path for TT traﬀic
• Preempt ET packets and re-transmit when possible, adhering to the Ethernet
standard
The information needed by the switch is contained in a standard layer 2 IEEE
802.3 Ethernet frame. The source and destination addresses are used for routing and
the type field is used to distinguish between TT and ET packets. After receiving a
packet and extracting the information from the header, an ET frame will be stored
in a buffer, while a TT frame will be immediately switched to its destination port.
The protocol should protect against two TT frames arriving simultaneously, but if

3.1
Design decisions
To facilitate rapid prototyping and to fulfil the aforementioned requirements a number
of design decisions have been made:
1. Bufferless design
2. Addressing through TDM, MAC addressing is not supported
3. A COTS Ethernet controller is used to remove uncertainty in debugging and
decrease development time
4. Support only for parsing bare 802.3 frames and IP frames, which is the minimum
for providing the required functionality
5. Support PTP synchronisation in one port only
6. No self-created packet send capability, thus disabling two-step PTP protocol
implementation
7. Communication to and from the controller through OCP buses
To summarise, the switch can switch Ethernet packets via TDM using a pre-
configured schedule. The schedule is configured on runtime via the softcore Patmos
processor and is supported by a Real Time Clock (RTC), that is synchronised via

3.2
Ethernet controller
The Ethernet controller chosen for this thesis was the Xilinix Advanced eXtensible In-
terface (AXI) Ethernet Lite MAC IP [28]. It is a minimal implementation of the IEEE
802.3 Media Independent Interface (MII) specification. It supports 10/100Mb/s, has
an internal 2K byte buffer for incoming and outgoing packets, and can be interfaced
as an AXI slave. This core provided a known working Ethernet controller to base the
novel design around as well as save the time to develop a new Ethernet controller,
as this would be out of the scope of this project. However, it also imposed a design
limitation, because of the minimal nature of the core, it does not support promiscu-
ous mode. Promiscuous mode is very useful in switches as in this mode all received
packets are forwarded by the Ethernet controller, disregarding the destination MAC
address. In normal mode, the controller will only forward packets’ addresses to its
MAC controller and drop all others and evidently, this is not suited for a switch that
has to process every incoming packet. This limitation is bypassed for this project by
only using broadcast frames, i.e frames with 255.255.255.255 set as a destination MAC
address. It is also important to note that the controller IP also supports ”ping-pong”
buffering to increase the buffer size to 4K, but this is not used. The communication
to the Ethernet controller is facilitated through configuration registers. The sequence
required to transmit a packet is as follows:
1. Write the packet data to the buffer starting as address 0x0
2. Write the total packet length in the length register at address 0x07F4
3. Write one to the status register at address 0x07FC Bit[0]
4. The controller will reset the status bit at 0x07FC after transmission is complete
The sequence to receive a packet is as follows:
1. Monitor Bit[0] at address 0x17FC
2. When Bit[0] at address 0x17FC is raised by the controller, a frame has been
received
3. Initiate reading the frame from the buffer starting at address 0x1000
4. Reset Bit[0]at address 0x17FC to free the controller for incoming frames

3.3
PTP Unit
The PTP unit uses the PTP hardware as presented in [12], modified to output the
IEEE 1588 clock directly to the switch. Additionally, the timestamp is was modified
to be extracted by the hardware and stored into readable registers for the processor.
This was required as in contrast to [12] the packets are never read by the processor,
thus a method is needed to get the timestamps to perform the clock adjustment. The
components are organised as in Fig. 3.2, Patmos and the switch both are connected to
Patmos as OCP slave devices and the PTP TimesStamp Units (TSU) are connected
only to the PHY of port 1.
In Fig. 3.3, the functionality of the TSU is shown. The unit receives the incoming
nibbles (4-bits) directly from the PHY as it is connected in parallel to the Ethernet
controller. The nibbles are stored in a 64-bit buffer. When the start of frame delimiter
is detected, a timestamp from the internal RTC clock is registered and the internal
FSM begins to parse the packet. If a PTP packet is detected, then the incoming
PTP timestamp is extracted and stored into two 64-bit word registers, depending
on PTP type, one for sync messages and one for follow-ups respectively. The PTP
device is not capable of packet transmission autonomously, in [12] the transmission
is handled by the processor. In this project, the Ethernet controllers are not exposed
to the processor and it would require considerable effort to generate and transmit
Ethernet packets on hardware, therefore as stated in the requirements only one-step
PTP functionality is supported by this switch.

The scheduling provides the opportunity to employ a TDM method to facilitate the
switching. This offers two significant advantages. First, it eases the bufferless design,
as there is no need for a queue if all messages are received and delivered on time.
Secondly, it eliminates the possibility of a port monopolising the connection, thus
creating a blocking condition on the network. To illustrate the port multiplexing we
must first examine the scheduling of the switch. The schedule for each port is defined
by three values: activation time, period, and communication window.
Activation
time defines the starting time and is assigned as the period for initialisation. The
window value defines how long the port link will be active after its starting time. The
period is added to the current time after each activation, to calculate the time for the

The length of an IP packet is located 16 Bytes further on, so the read-write cycle
continues until the word containing the length is read. The length register is then
updated with the new length value. It must also be noted that the address pointer
uses the Ethernet controller memory address where the desired byte is expected and
it is not counting frame bytes. When the read counter reaches the updated length
value, the read-write cycles end after a five-clock cycle delay to clear any bus conflicts.

4.1
Theoretical Latency Analysis
The defining performance characteristic of a switch is of course the end-to-end latency
that can be achieved. The factors that affect latency are the time to store the packet to
the Ethernet controller buffers and the transaction time of the read-write operations,
including the bus overhead:
TLatency = TRX + Trw.
(4.1)
The transmission time of the Ethernet controller is equal to the total packet size
divided by the Ethernet datarate:
TRX = PacketSize(B)/Datarate(Mbps).
(4.2)
The switch needs twelve clock cycles, including the post write delay to complete a
read-write transaction. The word size is 32bits and there is an additional delay of five
clock cycles before entering the read-write phase. Also, there are twelve more clock
cycles to write the length and the status bit in the destination Ethernet controller,
so:

CHAPTER 5
Conclusion
In this thesis, a novel design for a time-aware Ethernet switch was presented and a
functional prototype was implemented and evaluated on a hardware platform. Refer-
encing the goals as stated in the Introduction of this thesis:
• A switch design that switches incoming traﬀic based on a given schedule was
presented. The switch operates using an off-line schedule that controls port-to-
port switching. The internal transmission is conducted sequentially and there
is no internal buffering in the switching controller
• A functioning prototype was synthesised and implemented in hardware
• Although two-way PTP was not incorporated due to time constraints, partial
one-way PTP synchronisation is implemented, making it easy to extend to two-
way. Using one-way PTP and filtering synchronisation of 80ms precision was
achieved
• The schedule is configurable at runtime using the onboard softcore Patmos
processor
• A theoretical analysis was performed to provide a function that estimates the ex-
pected end-to-end latency of the switch, depending on clock speed and datarate
• The implemented prototype is evaluated in a physical experimental setup in
terms of clock synchronisation performance and end-to-end latency. Testing
software and scheduling were developed to take measurements of the second and
nanosecond offset during PTP synchronisation and to measure the roundtrip
delay using two- and four-connected host configurations
• The design is open source and can be found on GitHub

5 Conclusion
5.1
Future works
As future works, the following ideas are presented that will improve the functionality
and usability of the switch device:
• Provide Ethernet frame transmission capability to the PTP unit to enable the
full two-way PTP synchronisation, this could enable up to nanosecond precision
synchronisation that would be a requirement in hard real-time systems
• Investigate the possibility of enabling best-effort or rate-constrained traﬀic, dur-
ing the downtime between the scheduled port-links
• Develop a custom Ethernet controller that operates in a promiscuous mode
that will enable MAC addressing capabilities between the hosts. This is more
important if best-effort traﬀic is incorporated into the design
• Roundtrip latency can be further optimised in hardware by minimising some
intentional delays that were employed for prototyping reasons
• Include more packet parsing capabilities of the switch to extend the supported
Ethernet protocols beyond IP/UDP
• Use the framework presented in [13], which uses a TTEthernet switch, to com-
pare the proposed switch and verify that it provides an end-to-end time-aware
Ethernet switch solution
