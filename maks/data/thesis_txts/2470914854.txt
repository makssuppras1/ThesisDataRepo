Abstract
In this project a complex object in the form of a 3D wooden puzzle, is assembled with a universal
robots UR3 robot arm. The arm is controlled using C++ programming. The puzzle that is
assembled in this project is a 6 piece puzzle, that can be assembled in one known way.
The project is divided into two parts. Thereby dividing the work into easy to consume parts.
First the puzzle is blindly disassembled, to learn how the robot arm works and which movements
are needed to code. Second the puzzle is blindly assembled, using the knowledge of the puzzle
gained when disassembling it.

1
Project description
1.1
Introduction
An automatic production involves using programmed manipulators, such as robot arms to
move objects around a speciﬁed work area. As automatic productions becomes greater, more
complex solutions are required to fulﬁll production demands. This includes the assembly of
complex objects by robots. To do this the robot need great movement precision and ﬂexibility.
This has to some degree been achieved by later models of fair priced robot arms. Moving the
robot manipulator around the object can be a complex coding task. A lot of movement points
is needed to move the arm safely around the work space, without hitting the objects or other
obstacles necessary for the construction of the ﬁnished product.
A robot arms greatest weakness is changes to the predetermined work order. Such as changes
to the predicted position of the object since it is blind, unless a camera has been attached. If
the robot arm is not programmed to take into account small predicted changes in position or
layout of the object, the object could be damaged. Objects manufactured in automatic systems
are often fragile and/or hazardous, which means that even small damages can be extremely
costly.
1.2
Considerations
To avoid small predictable changes in the position of the object, the robot arm should be
programmed to move each piece in a way that take advantage of open areas in the pieces, where
later pieces will be placed at a later time. To increase precision, it is often helpful to move the
robot arm slower, but moving slowly during the entire assembly, would make the assembly take
a very long time. It is important to ﬁnd a route for the arm to avoid hitting the unﬁnished
object, while moving pieces around the working space. When assembling a complex object with
only one arm, it can be necessary to use external helping tools to keep pieces in place while
placing others.
In this project a wooden puzzle is analyzed and assembled, based on above considerations. The
wooden puzzle is chosen, since it represents a complex object adequately, while still been simple
enough that the project won’t become too great to accomplish within the given time.
1.3
Problem statement
Based on above considerations the following problem statements need to be answered, to as-
semble the object with a robotic arm.
• How can a robot arm be programmed to assemble a complex object?
• How is it possible to take advantage of open areas in the pieces, before they are ﬁlled by
other pieces?
• How fast can the robot arm be moved, while still maintaining the necessary precision to
assemble the object?
• How can the robot arm be moved without hitting the object, while moving pieces around
the working space?
• How many helping tools are needed to construct the object?
• What kind of helping tools are needed to construct the object?
• What situations demand helping tools?

1.4
Project limitations
Robot arm
The assembly will be performed with a blind robot arm. Meaning that no camera or sensor will
be used to register the position or layout of the object. There will also only be used one robot
arm, limiting the robots ability to control the object.
Object
The object can only be assembled one way, meaning there are no better or worse way to
assemble the object. In this report, the object pieces are numbered depending on, if the object
is disassembled or assembled. The ﬁrst object piece during the disassemble, is the ﬁrst object
piece that is removed and the next object piece that is removed is number two and so on. For
the assemble it is the opposite, since the ﬁrst object piece is the ﬁrst to be moved into place
and the second is the second to be moved and so on. This means that the ﬁrst object piece for
the disassemble and the last object piece for the assemble is the same object piece. The ﬁrst
object piece for the assemble and the last object piece for the disassemble are also the same
object piece and so on.
2
Analysis
In this section a short description of each part of the project, that needs to be solved will be
given.
2.1
Object
In this project, a wooden 3D puzzle with 6 pieces has been chosen to represent a complex object,
that needs to be assembled. A wooden puzzle was chosen, since it is solid and therefor won’t
break easily during failed tests. The puzzle is also supposed complex enough to exhibit the
robots and students capabilities, without becoming too complex. Becoming too complex would
expose too many variables to ﬁnish the project within the allocated time. An easy accessibility
to the object for repeatablility also aid in the decision, for choosing the wooden 3D puzzle. The
puzzle has a gap of approximately 0.5mm around each block, making the puzzle a little rigid.
This is both useful and inconvenient, since the rigidness allows for some accuracy error, but it
also allows the blocks to move around and therefor not always be a hundred percent predictable
in their positioning. This gap around each block also make some pieces drape, making attaching
other pieces impossible, without helping tools.
2.2
Robot
The UR3 robot arm is the smallest model from universal robotics, but with a reach of 0.5 meters
it is more than big enough to reach around the object used in this project. With 6 degrees of
freedom the robot arm has more than enough ﬂexibility to create safe and necessarily complex
movements. At the end of the robot arm is a clamp used to pick up object pieces. This clamp
has a grasp of 8.5 centimeters, which is both positive and negative, since the object pieces are
only 2x2x6 centimeters. The clamp’s grasp is more than suﬃcient and the extra length could
limit the robots ability to move around without hitting the object or other things. Although the
extra grasp length allow for more ﬂexibility when choosing necessary helping tools, that need
transportation from the arm. Helping tools will be needed since only one robot arm is available.
A tool for holding the object in place while disassembling and assembling it is necessary. Also
thanks to the objects rigidness, a tool is needed to keep pieces in place, while adding other
pieces when assembling the object.
2.3
Code
C++ is used to write the code that controls the robot arm.
To do this, a driver called ur_modern_driver[2] is used, to communicate between the control
5
==PAGE:7==
script and the robot arm controller. This driver works like a library and gives access to useful
commands and classes needed for communication, between the control script written by the
user and the robot arm controller. Using this library for communication, a code based on point
to point movement actions are written. Three functionalities are written, one to disassemble
the object, one to assemble the object and one to do both. Firstly the disassemble function
is written, since there are less object position uncertainties. This will grant the ability, to get
used to the robot and understand the setup of the movements(reverse engineering). Next the
assemble function is written, based on the information gathered from the disassemble.

3
Complex object disassembling
In this section the decision making and results, for the disassembling of the complex object will
be described. The robot arm moves by following a queue of position coordinates, moving from
position to position, thereby creating a build path. By following this build path the robot arm
is able to collect the object pieces and move them into position to disassemble/ assemble the
object, without hitting the other object pieces on the way.
3.1
Helping tools
To begin with the object was disassembled by hand to understand the way it was constructed.
Here it was quickly realized that two hands was necessary for the disassemble, since the friction
between the pieces made it nearly impossible not to move the entire object, while pulling or
pushing on one piece. For this reason a tool for holding the object in place, while the robot arm
move around the pieces had to be created. The result was a piece of wood, indicating the work
area, with a spring clip attached to it. The object will then be placed in the spring clip as shown
on image 3.1.1. Under the work area some adjustable legs were attached to ensure a leveled
surface. A coordinate system, was also attached on top of the work area and below the spring
clip. This is used to calibrate the robot arm to the work area and to use as reference when
choosing what coordinate to move the arm to. The last two attachments, was not necessary for
the completion of the objective, but made the work much easier. Since the spring clip holds the
object slightly above the work area and at the side of the object, a foot for the object is needed,
to balance the object. This foot is designed as a bridge that go over the foot of the spring clip
and under the object, held up by the spring clip.
Image 3.1.1: Work area with robot arm, object and helping tools, front view
3.2
Disassembling code
After the object is kept in place and the working area has been deﬁned, the code to make the
robot arm disassemble the object was developed. The whole code has been developed to work
around one function called gotopose, which was provided by the instructor Nils Axel Andersen
at project start. gotopose takes in the desired coordinates and angles for the robot arm. The
function then moves the robot arm to the desired position, with the desired angle at the end
manipulator. Using this function it is possible to move the robot arm around the object, by
using the function repeatedly with diﬀerent desired position coordinates. It was decided to
make one movement function, that would be able to remove all object pieces, given diﬀerent
position coordinates. A vector containing the position coordinates for each piece was made for
each required position. Doing this, the movement function could be called once for each object
piece and just take in the position coordinates required, for each object piece.

3.2.1
Gotopose function
gotopose takes in the desired x, y, z, x0, y0 and z0, which is the desired coordinates and
angles for the robot arm.
It then uses vector transformation on x, y and z and then calls
the movel function, which given coordinates, a speed and acceleration will make use of the
ur_modern_driver library commands, to move the robot arm to the provided coordinates, with
the given speed and acceleration. movel will also freeze the code, while waiting for the robot arm
to reach the desired position. This is done because the ur_modern_driver library commands
will override each other, if multiple commands are send to the robot before the last command
was executed. This would result in lost commands and the robot arm moving directly from
the ﬁrst given position, to the last given position and forgetting about all the intermediate
commands.
Since it is necessary to do some movements at diﬀerent speeds, three diﬀerent gotopose functions
was made.
One for fast movements, used when operating away from the object.
One for
intermediate speeds, used when operating close to the object, but not touching the object.
Lastly one for very slow movements, used when operating while touching the object.
The
reason why it is necessary to move slower near the object is, because precise movements are
more important then. Sudden unexpected movement might break or simply slightly move the
object from the desired position. If the object is moved from the desired position, the next
commands might be compromised.

There are a total of seven coordinate vectors, with positions for each object piece at desired
positions. One of these are the positions of the object pieces, when they are separated. The
rest of the vectors are the positions used for intermediate movements, between assembled and
separated states of the object.
The object piece that takes the most amount of movement

disassembling process, but changed to the fourth piece when designing the assembling process.
Designing the vectors like this, means that for all other pieces than the fourth, there will be
duplicates of the same coordinates for multiple vectors. This does not change anything in the
movement, since if the coordinates are the same as where the robot arm already is positioned,
the robot arm will not move and skip to the next movement command.

3.2.5
Main function
The main function ﬁrst call a transform function, which transform the work space coordinates
into the robots coordinate space. This makes it, so that the coordinates given to the gotopose
function is automatically transformed into the robots coordinates. This makes designing the
position paths much easier, since it is only necessary to read the work space and not try to
imagine the robots coordinate system, when designing paths.
After transforming the coordinates, connection is enabled to the robot, using the ur_modern_driver

library. After that, a switch case is called, which allow the user to choose between running the
code for disassembling the object, assembling the object or both. To begin with, only the ﬁrst
is operative. If the ﬁrst is chosen the TakeApart function is called. The program is ended and
connection to the robot is disabled after the TakeApart function is ﬁnished.
3.3
Results
Using the above described functions, it is possible to disassemble the 3D wooden puzzle and place
each of the object pieces at predetermined positions, around the object’s initial position. The
hardest part of the disassembly was that object pieces would get attached to other pieces, when
trying to move them and thereby get pulled or pushed into positions, that was not intended.
This was solved by adding an extra position coordinate to the build path, which moves the
object piece that is intended to be moved slightly away from the object piece, that would get
attached and moved unintended. It was possible to make one movement function, that works
for the removal of all object pieces, by giving the function diﬀerent position coordinates for each
object piece.

4.2
Build path design
The UR3 is stated in the data-sheet to have a repeatability of 0.5mm[1]. This has also been
conﬁrmed during this project. It has not been possible to ﬁnd any data on the robot arms
accuracy, but since the repeatability is less than 0.5mm, the accuracy can only be as good or
better. Often repeatabillity is the most important thing for industrial robots like the UR3,
since they have to do the same batch of movements repeatedly. For this project, accuracy is
more important, but since the two are almost equivalent, the repeatability will be accepted as
accuracy for this project.
When choosing the positions for the build path design, it is important to take the robot arms
accuracy into account. Since even if 0.5mm is not enough of an error to make the assembly of
13
==PAGE:15==
two object pieces fail, it can be enough of an error to make the assembly of three object pieces
fail. This position error of 0.5mm, can multiply by each object piece that needs to be attached.
If lucky, the position error of each object piece will equalize and the assembly will not fail, but
for this to happen the error must be close to equal for all object pieces in every direction. This
is highly unlikely and there will therefore most likely be some degree of position error build up,
when assembling an object of more than two object pieces.
For the last two object pieces, it is very important that the other object pieces are at a very
speciﬁc position and that the robot arm is being very precise. This is because, there are not
much room for error with these object pieces and these object pieces are unable to move the
other object pieces, to get in to place. For the ﬁfth object piece, the limiting factor is that it
needs to be placed around two other pieces, that are already placed in a locked position. For
the sixth and last object piece, it needs to go into a hole between all the other pieces. As seen
on image 4.2.1, the other object pieces does not make a perfect square hole. Some pieces are a
little higher or lower than the others, thereby removing the room for error, the rigidness of the
object otherwise provided.
Image 4.2.1: Hole for the last object piece
Since the robot arm is blind and therefore cannot see, if the object pieces have small errors
in their predicted position, the building path has to be designed in a way that limit the position
error build up. Limiting the position error build up, will allow to better predict the position of
the object, even though it is being build blind. In this project three techniques are utilized, to
limit the position error build up. The ﬁrst technique is by fastening the ﬁrst piece, to the work
area with a spring clip. This constrain the x and z coordinates of the ﬁrst piece, thereby limiting
the position error to the y coordinate. The second technique is to move the object pieces slightly
more than needed, thereby hitting and moving other object pieces. This technique will equalize
the position error of the object pieces, by transferring the position error of the second object
piece to the ﬁrst object piece. Thereby giving both object pieces the same position error. The
third technique is to use help blocks that can be moved strategically into places, where they can
hold object pieces in place, while adding other object pieces to the object. The last technique
also provide the same beneﬁts, as the second technique.

4.4
Results
Using the above described functions and techniques for limiting the position error build up, it
is possible to assemble the object. By moving the object pieces from predetermined positions in
the work area and placing them together in the spring clip, the object can be assembled. The
design of the build path, was the most time consuming part of this part of the project. Limiting
the position error build up was very important, to realize and understand. Also designing and
implementing the helping tools was a big task.

==PAGE:19==
5
discussion
It is possible to both disassemble and assemble the complex object that was used in this project,
using the robot arm. Both the disassemble and the assemble is pretty reliable, but there are
still some parts of the process that can fail due to human error. This instability is caused by
misplacement of the object or helping tools, when setting up the work area. The ﬁrst unstable
part is when removing the third object piece, during the disassemble. If the foot under the
object is placed too far to the left, the third object piece will lean and pull the ﬁfth object piece
with it, when being moved.
This misplacement of the foot under the object, will also result in a failure to place the fourth
object piece during the assemble. Since the foot is relied upon to guide the fourth object piece
into place, a misplacement of the foot will result in the fourth object piece not ending up, where
it belong. This problem can sometimes ﬁx it self, when the helping blocks are used to press the
ﬁrst and fourth object pieces together, but this does not always work. Lastly, placing the object
wrong at the start of the disassemble, or placing the object pieces wrong during the assemble,
will also result in a failure. Since the wrong placement of the object or object pieces would
result in the robot arm gripping and thereby placing the object pieces at a wrong position,
creating errors too large to recover from.
During testing the disassemble function was used to place the object pieces at the right po-
sition, before an assemble was started. Since there are less of a chance of placing one object
wrong than placing multiple objects wrong, it resulted in better performance to rely on the
robot arm, for placement of the object pieces. If future work were to be done on the project,
these problems could possibly be solved by mounting the foot permanently on the work area
and by creating some kind of low walled boxes, for the object and object pieces to be placed
in, for optimal positioning. If these boxes where to be made, it would be important to leave
room for some error, otherwise the robot arm could fail to place the object pieces in the boxes.
The boxes could also be design to adjust for error, by having sloped walls. The best way to
eliminate the human error would be to implement a camera, that would allow for adjustment
in the positioning while the program is running.
For testing purposes, the speed of the robot arm was greatly reduced. This was done since
the robot arm at regular speed would move too fast, to easily observe what happens when the
object pieces was placed. Thanks to the work area coordinate system, some knowledge was had
about what coordinates to give the positions, but some adjustments to the coordinates had to
be found through trial and error. To make these trial and error adjustments, the speed of the
robot arm had to be decreased, to better see in what direction, the adjustments had to be done.
The lowered speed also allowed for time to hit the emergency stop button, if something went
very wrong with the movements. The robot arm are moving at 5% of maximum speed, when
operating close to the object and at between 10% and 50% otherwise. It takes the robot six
minutes, to ﬁnish both disassembling and assembling the object.
At the end of the project it was attempted to increase the speed of the robot and thereby
decreasing the amount of time it takes to perform the disassemble and assemble tasks. Sadly
the increase in movement speed resulted in a great instability of the tasks. This is most likely
due to the technique used to limit position error build up, since an increase in speed would
result in an increase in force, when pushing on the object pieces with other object pieces. The
accuracy of the robot arm also seemed to decrease when moving at higher speeds, which result
in the small human errors described earlier being more prevalent. To ﬁx this the instability
caused by the human error has to be ﬁxed and a diﬀerent approach to the position error build
up limitation has to be examined.
6
Conclusion
Using helping tools and designing a build path that make use of position error build up limiting
functionality, it was possible to write a code that allow the robotic assembly of a complex object.
There are still some instability caused by human error, that can only be eliminated by further

development on the project. The assembly is also limited by speed an a full disassemble and
assemble takes six minutes. Increasing the speed courses further instability.
The program has been designed in a way where the same position coordinates are used for the
movement of the robot arm, in both the disassemble function and the assemble function.
