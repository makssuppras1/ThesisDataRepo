Abstract
This thesis focuses on implementing and testing a Multi-State Constraint Kalman Filter (MSCKF) into
Matlab using self generated data to validate the implementation. The project uses the Novatel SPAN Global
Navigation Satellite System (GNSS) reciever and Inertial Measurement Unit (IMU) system placed on a
Unmanned Ground Vehicle (UGV) to create a ground truth state estimation dataset while allowing use of
pure IMU data to be used and tested for the Inertial navigational System (INS) implementation. The UGV
is also equibted with 2 GoPro Hero 8 and a RealSense D455 cameras from which a mono Visual Odometry
(VO) system was implemented for the measurements model of the MSCKF using a single GoPro camera.
The Thesis goes through the implementation of the system in Matlab and how the test were conducted. The
results showed in a simulated system that the implementation was unable to correctly estimate appropriate
update if an error is placed on the state, and problems with the INS attitude estimation or the transformation
matrices in the measurement model is suggested as the cause. The system worked as intended without state
errors and tests of the INS indicate expected position and velocity estimates. The VO system is shown to
perform within ranges of the test capabilities and low sensitivity to state error of triangulation and high
sensitivity of re-projection is expressed.

Introduction
Increasing interest in autonomous vehicles have brought a larger focus on navigation in GNSS denied envi-
ronments. Environments where GNSS struggle or is incapable to provide acceptable navigational solutions is
coined GNSS denied environments, and plenty environments fall under this category such as: indoor, forests
operations, tunnels, urban environments with tall buildings causing line of sight to satellites and multipath
issues also known as the urban canyon phenomenon [Gakne and O'Keefe, 2018], bad satellite geometry, and
an increasing problem GNSS jamming [Olesen et al., 2020].
These environments are not uncommon, and therefore as more automation and especially automation with
UGVs and Unmanned Arial Vehicles (UAVs) starts to become normal robust secondary navigational systems
need to be able to run seperately to GNSS based systems. Such systems have had plenty of research and
many involve the use of Dead Reckoning [Groves, 2015] or VO [Scaramuzza and Fraundorfer, 2011][Lin et al.,
2018][Center Jr and Knuth, 2011], while good results have been shown many solutions require signicant
computational resources. As described in [Clement et al., 2015] combining IMU and camera (VO) solu-
tions have become popular, and two methods seem to have won favour in accuracy versus computational
eciency [Clement et al., 2015]: Sliding Window Filter (SWF) [Sibley et al., 2010] and MSCKF [Mourikis
and Roumeliotis, 2007]. These lters both produce great results however as [Clement et al., 2015] showed
SWF generelly produced better state estimation and is less sensitive to tuning parameters, while MSCKF
showed to be more consistent while requiring less computational power and easier scalable computational
requirements.
These results come from using a stereo camera setup, and in general when using VO for navigation a
stereo setup is common [Sun et al., 2018]Clement et al. [2015][Warren and Upcroft, 2013], however as
shown in [Warren and Upcroft, 2013] as the distance to the tracked object increase the stereo pair baseline
approaches monocular approximations and therefore a signicant reduction in triangulation accuracy occurs.
Uncertainties in the stereo pair also increase with structural deformities which can be caused by signicant
vibration as such found in smaller cheaper UAVs [Warren and Upcroft, 2013].
The objective of this thesis is to implement a navigational solution for GNSS-denied environments using
sensor fusion between a IMU sensor and a camera system.
For this purpose the MSCKF was selected
due to its good performance as shown in [Clement et al., 2015] but more so for its scalable computational
requirement such that the build system could be implemented on many dierent test beds with dierent
computational resources available.
Furthermore this project aims to implement the system as described in [Mourikis and Roumeliotis, 2007]
as a monocular camera system, as the nal intention for the system would be to be used in medium to
high altitude xed-wing drones as a supplement to a separate GNSS navigational system in case of satellite
navigation loss. As signicant vibration and limitations on power and weight is present in such a setup
and with the reduced triangulation due to poor baseline to depth ratio of high elevation ight [Warren and

In navigation we describe the position, orientation and motion of objects. An object could be anything,
a table, a person, a GNSS satellite or IMU sensor, it technically doesn't need to have physical form as it
can also just be a point in space. To dene an object all we need to know is a specic point also called
the origin of the object, and 3 noncoplanar mutually perpendicular axis. The position and orientation of
an object is always as a reference to the position and orientation of something else. For our everyday life

The combination of a origin and its axes for an object, or reference creates a coordinate frame. A coordinate
frame has 6 degrees of freedom, consisting of 3 for the the position of the origin, and 3 for the orientation
of the axes x, y and z. A coordinate frame can be seen in g. 7, where the superscript indicate to which

Reference frames generally follow the right-handed convention, which means that using the thumb, index
and middle nger of your right hand, you can express the orientation of each axis, by extending the ngers
perpendicularly to each other. In this scenario the x-axis will be expressed by your thumb, y-axis by your
index nger, and z-axis by the middle nger. There are many dierent types of coordinate frame, however
we will only describe the 3 types that are used in this report: Earth-Centered Earth-Fixed (ECEF), Local
Navigation- and Body Frame.
4.3.1
Earth-Centered Earth-Fixed Frame
The ECEF frame is explained by its name. The origin is in the center of the Earth's mass and it is xed to
the Earth, meaning that, contrary to Earth-Centered Inertial (ECI) as the Earth rotates, the frame rotates
with it. This means that an object sitting still on the Earth will have constant position coordinates in the
ECEF frame. This is very useful for navigation, as we generally want to know an objects position in reference
to the Earth. The axis of the frame is dened as z pointing along the Earth's rotational axis towards true
north, x points from the center to the reference meridian at 0◦longitude and y is pointing from the center
to 90◦east meridian, as seen in g. 8. The ECEF is normally denoted e when referenced in equations.

Another somewhat less common orientation is the East North Up (ENU) coordinate frame which as the
name suggest has x pointing east, y pointing north and z pointing up. Any combination is possible as long

to relate to. Furthermore if you want the reference according to north, east and down directions as you
might want if you want to know your heading and elevation above sea level, the local reference frame is the
most optimal. When an object nears to poles of the planet an issue starts to occur due to the existence of a
singularity at each pole, where there is no north or east direction. Therefore the local frame can't be used
here, and another reference frame is needed [Groves, 2015, p. 28].
4.3.3
Body Frame
The body frame, also known as the vehicle frame describes the origin and orientation of the object that
the navigation solution is for, as example an airplane, car, rover or just a sensor on a moveable platform.
The body frame is a local frame with the exception that the axis directions are xed to the platform, i.e.
the standard convention is that the x coincide with the forward direction, or the "nose" of the platform, y
points to the right and z points down. Just as the coordinate frame follows the right-handed convention,
the rotation around each axis of the body frame also follows the right hand rule. This rule states that by
aligning your thumb with the positive axis and curl your ngers around the axis, the direction of the curl
is the positive direction of rotation around that axis. With this convention each axis's rotation have its
own name, the rotation about the x axis is called roll, the rotation about the y axis is called pitch, and the
rotation about the z axis is called yaw, as seen in g. 10.

Initialization
As the INS takes direct measurements of acceleration and rotational rate of the vehicle it is mounted on,
the only information that we can get from the system is the change between states (position, orientation,
velocity etc.). This means that to get the actual state of the system, we need to know the previous state to
apply the measured change, and arrive at the current state. This means we require an initial state to start
the process from, and this section will describe how we achieve the initial state for an INS using orientation,
position, velocity and sensor biases to formulate the state.

stationary is enough, if the system is started on steady ground. Another solution could be to use a GNSS
to aid the initial position and velocity, or if the system has run previously, storing the last known position
can be a solution for initialization [Groves, 2015, p. 196].

4.7
Multi-State Constraint Kalman Filter
The INS high frequency and high accuracy at low time intervals is very attractive for navigational purposes
if the accuracy can be extended into a long time frame. A method to achieve such a thing was suggested
by [Mourikis and Roumeliotis, 2007] by creating an Extended Kalman Filter (EKF) that combines the INS
system with a camera system that tracks features as the system moves, and called in MSCKF.
The system is very simply described in g. 21, and it consists of the INS updating the system position at
200 Hz. At 5 Hz the camera takes pictures of the scene, nds features, matches them with previous features
from older frames, and if any feature is no longer in view or the feature has been in maximum number of
frames an update event is triggered.
During an update event the feature that have gone out of views position is triangulated. With the feature
position known the feature is re-projected back onto all frames it has been captured in. Given that there
are errors in the position of the system, there will be a re-projection error just like with the triangulation
using geometric error in g. 18. By minimizing this error for all frames the feature is visible in, the system
corrects the position, and orientation of the system at all the frames the feature is in using an Kalman Filter

Combining

a Kalman Filter with our stationary position as input for the GNSS measurements, and calculate updates

orientation and INS covariance, we can continue in the general script.

6.2
Simulation
We want to test the system without relying on the precision of the ground truth GNSS position and the
accuracy of the camera setup, and to do this we have chosen to simulate all the feature points and the
feature matching system. What this achieves is that we can supply the system with 100% accurate feature
triangulation and re-projection from which we can test if the algorithm works outside any physical camera.
To do this we use the GNSS position we gathered in the VO test section 6.3 to get a ground truth for the
observed features. The features are 1000 randomly generated points in the area placed such that several
features are always visible for the system.
To create the camera observations we use the re-projection method described in eq. (4.7.14) and again
repeated here in eq. (6.2.1). Since we have generated the feature points we have the exact location of the
3D coordinates for each feature, and can therefore get the exact normalized pixel coordinates, which we can
de-normalize using a calibrated intrinsic camera matrix (K ) to get observed pixel coordinates.

8
Discussion
The objective of this project was to create a MSCKF that would be able to provide a navigational solution
for GNSS-denied environments. The intention was to implement and test the system on a UGV and then
nalise on an UAV, however unforeseen challenges occurred that prevented us from being able to implement
a fully working system on the UGV. Determination and insistence to use our own dataset and using the
UGV for testing the vision system, resulted in unusable data 3 out of 4 times the test was performed. The
problems with the tests was that the system did not log data, the GNSS position was too poor for analysis,
or not all the cameras was turned on. All of these problems could have been prevented with proper planning
and more experience with the test equipment, however it did cost valuable time for actually implementing
and nding errors in the very complex MSCKF algorithm.
8.1
The Cameras

8.2
INS
The INS is essential for the MSCKF to work as it is system that provides the actual position estimation.
Using the IMU from then SPAN system resulted in a very poor INS solution, as is clear by g. 48a. We were
fortunate enough that we were able to use the GNSS compensated measurements from the SPAN system
instead, but looking at the dierence between the actual measurements and the compensated measurements
the only real dierence seemed to be from the x axis of the accelerometer as seen from gs. 49 and 50. Given
the area where the test done was very bumpy we would expect that if this caused the errors we would see a
large disturbance on the accelerometer z axis, and in general the disturbance should be across all axis not
just the x axis. This indicate that the error on the IMU is caused by misalignment or general sensor error
as it does seem to measure the same general path but oset from the actual measurements.
What is especially interesting is that such a small error would have a signicant error in the nal position
estimation which is clear and the dierence between the original INS estimate in g. 48a and the compensated
estimate from g. 48b. Looking at the deviation of the INS from the GNSS it seems to follow rather well
until the system starts to turn where the INS turn too slowly making a too large arc, and then clearly the
errors have stacked up too much and a couple of spikes due to uneven ground could cause these aggressive
turns.
Weirdly when looking at the path the INS follows it does not align with the orientation that the axis shown
on g. 48b indicate. This could be because system keeps track of attitude, position and velocity but update
the attitude only on the gyros while the velocity and position is based of the attitude from the gyro and
the accelerometer measurements. The system could also be implemented incorrectly and a rotation matrix
is incorrectly transposed. This is unlikely as while the system drives straight the attitude follows correctly
along the path, it does look like the attitude would be correct if the position was on top of the ground truth,
again indicating that perhaps an error in the IMU accelerometer sensor is present.

marker that moves the least.
Looking at the actual triangulation errors in table 7 we actually get pretty good triangulations for marker 1
and 3 but quite bad for marker 2. In general the RealSense camera seems to perform worse in triangulation
than the GoPro camera, however that could just be down to the camera resolution dierence and therefore

Testing the triangulations sensitivity to constant error in tables 7 and 8 shows that a constant translation
error will just directly oset the triangulation, which means any errors in the initial position will eect all

constant error in the attitude it is clear that the system have dierent sensitivities for roll, pitch and yaw.
For roll even 10◦error doesn't do much to the triangulation while even 1◦of constant oset in yaw can result
in 10's of centimetre error. Pitch seem to have a larger sensitivity than roll but not by much. This means

With logarithmic increasing error such that we would see when using the INS we get interesting results for

described by the fact that this is within the GNSS error range, but for 0.1 m to 1 m error marker 2 and 3
still improves triangulation, with no good explanation. The important thing to note however is that the
triangulation doesn't change too much with such a large error on the system, indicating that the triangulation
isn't too sensitive to translation errors. For the attitude in table 9 it is the same story as with the constant
oset except they are much less sensitive to the errors when they grow over time, where even the yaw seems
to generally improve with the error applied.
Generally it seems the triangulation isn't very sensitive to errors while the system is running, but its impor-
tant that the initialization has low errors especially in the heading determination.

8.4
Simulation
The simulation is the ultimate test of the system, there are no GNSS, triangulation or re-projection errors,
if everything is implemented correctly we should see no update when the system has no error on the position
as the re-projection matches perfect with the observed pixel coordinates, but once position error is applied
updates should correct for the errors.
We see from g. 51 that when there is no error on the system the MSCKF runs perfectly, the state update
that does happen is in the 10's of nano metre and only occur when a spike in features are used for the update
as seen in g. 52. Such a small error could easily be computational and is not important. We do see however
that only using 1000 features meant that few features were used pr. update and as suggested before using
more features would in a non simulated test reduce noise.
We get the complete opposite result when applying error to the state as seen in g. 54. Here the rst update
displaces the position with around 200 m which creates a cascading eect that brings all other updates into
the extreme with the maximum update ending up at 6 Mm. This is of course, wrong and clearly an error
has been made in the implementation of the MSCKF, however due to the complexity of the system the error
have been very hard to nd. The fact that the system does work with no state errors would indicate that the
error could be in the update step where a transformation matrix might have been transposed or multiplied
incorrectly.
The error could also be in the system or feature Jacobians (H x & H f) as these also have
transformation matrices and directly deal with the re-projection if an error was made in the re-projection
implementation. With time this error could surely be found and corrected, but unfortunately as expressed
earlier other issues and the general complexity in understanding and implementing the system meant that
not enough time was allocated for debugging the system.

9
Conclusion
This thesis aimed at implementing a MSCKF as described by [Mourikis and Roumeliotis, 2007], with the
ultimate objective to use the system on a UAV platform after initial tests on a UGV platform. Ultimately
due to unforeseen struggles with camera calibration and ground tests the system was not implemented in a
fully working form on the UGV platform.
The INS system was implemented using [Groves, 2015] equations and functions and tested on the UGV
using the IMU from the SPAN system, however analysis showed the IMU measurements had errors on its
accelerometer x-axis and therefore GNSS compensated IMU measurements were instead used to model the
INS. The resulting INS implementation proved to give promising results of state estimation with the errors
one could expect from a low-cost IMU INS implementation. Unknown issues were found with a disconnect
between the estimated attitude and the estimated velocity and position of the system.
It was shown through several calibrations that the selected GoPro cameras had unknown error that resulted
in uncertainty in the calibration of 20 pixels while calibration of a separate camera conrmed that the
calibration method was sound. Unfortunately this meant that a further source of error existed in the setup
based purely on camera errors and not implementation methods used.
The camera and feature management systems were tested and shown to be capable of nding, removing
outliers and tracking features over several frames, which allows triangulation and re-projection of features
found in the frame. The triangulation implementation were also found to be acceptable given the uncertainty
of the GNSS position and the unfavourable geometry of the camera positions. For the re-projection errors
were found in tests where no re-projection error should be present, but the resulting error could be explained
by the uncertainty of the GNSS position of the cameras.
Tests of the sensitivity of both triangulation and re-projection showed that the triangulation is sensitive to
errors in the initial position and very sensitive to errors in the initial heading, while errors in the initial pitch
and roll were less important. The triangulation were however found to be very robust in regards to errors
while the system is running, which was in contrast to the re-projection which were very sensitive to error in
position and attitude with signicant errors at 10 cm translation error and under 1◦attitude errors. These
results indicate the system performs as wanted to be able to compensate for position errors based on the
re-projection alone.
To properly test the full system as simulation was made to remove all camera and position errors. This test
showed that if the system does not use the INS and have no errors in the position and orientation it performs
perfectly and does not update the position. However if the INS system is used incorrect corrections are made
immediately and the system diverges from the correct position. The error causing this is suspected to either
be the problem with the attitude of the INS or it could be an error in one of the transformation matrices in
the EKF update step or in the feature Jacobians (H x and H f).
Further work would be needed to isolate and correct the error causing incorrect state updates, however
as most systems have been shown to perform as expected there is evidence for a proof of concept in the
UGV being feasible and adaptation for UAV would be possible with minimal change. This system have
been created for use with stored data, signicant optimization changes would have to be made to be able to
implement this system for live navigation.

A.5
Matching features
The simplest form of the description is the intensity patch around the feature, such a description could be
the sum of squared dierences (SSD) or the normalized cross correlation (NCC). Such measures are ne if
no change in orientation, scale or viewpoint occurs as they aren't invariant to these changes. The popular
SIFT descriptor has shown to be robust against illumination, scale, rotation and viewpoint changes of up to
60◦[Fraundorfer and Scaramuzza, 2012].
The SIFT descriptor is created by computing the gradient magnitude and orientation around the feature in a
16×16 grid, which is then weighted by a Gaussian window, and the orientation summarized into histograms
of 4 × 4 cells that show the 8 orientations of the cell. The orientation of the histograms can be concatenated
such that a single vector describing the orientation of all the cells, will be a 4 × 4 × 8 = 128 vector that
describe each feature, as seen in g. 58 [Lowe, 2004] [Fraundorfer and Scaramuzza, 2012].

Once all features have their own descriptor the simplest way match/track the features is to compare every
descriptor from one image, with every descriptor from another image. The best measure for comparison
varies with the descriptor type, however if its local appearance of the feature then SSD & NCC might be
suitable, but for SIFT descriptors the best measure is the Euclidean distance.
As robust as feature descriptors can get, errors will always occur, and incorrect matches will need to be
dealt with. One way is to use RANSAC, which is an outlier removal algorithm. The idea is to create a
model from randomly sampled data, and compare it to the rest of the data, and see how many inliers occur
with the current model. The process is then repeated and the model with the lowest number of outliers is
selected. For computer vision the model that RANSAC estimates is the relative motion of the camera i.e.
the rotation and translation [R, t]. The algorithm can be seen below [Fraundorfer and Scaramuzza, 2012]:
1. Initial: Let A be a set of N features correspondences
2. Repeat
2.1 Randomly select a sample of s points from A
2.2 Fit model to these points
2.3 Compute the distance of all other points to this model
2.4 Construct the inlier set (how many points have a distance of < d)
2.5 Store these inliers
2.6 Until maximum number of iterations reached
3. The set with maximum number of inliers is chosen as a solution to the problem
4. Estimate the model using all the inliers
