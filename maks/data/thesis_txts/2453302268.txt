Abstract
This thesis investigates and implements a parallel and scalable spectral p-multigrid ﬁnite element solver for a
1-dimensional Poisson problem.
The ﬁrst part of the thesis introduces the spectral discretization to partial differential equations. These meth-
ods provide fast and precise solutions to discretized partial differential equations, at the cost of more careful
development of the discretizations. The spectral element method is introduced, enabling the computational
domain to be split into elements, each keeping the high accuracy of the spectral discretization. The elements
allow ﬂexibility to the domain decomposition and generalizes to arbitrary meshes. This solver is developed and
tested in Matlab.
The second part investigates the so-called multigrid methods, a class of numerical iterative solvers for dis-
cretized partial differential equations. Through examples, ﬁgures and theory these methods are developed and
applied to the spectral element discretization from the ﬁrst part. This combination allows fast and accurate solu-
tion to the problems considered in this thesis. Furthermore, the multigrid methods are prone to parallelization,
which is explored in the last part of this thesis. The spectral p-multigrid ﬁnite element method is then develop
in Matlab.
The third part revolves around a high-performance implementation of the aforementioned spectral p-multigrid
ﬁnite element method in the C programming language. C is an old low-level programming language that is
cumbersome to write in, but provides almost unparalleled execution speed. The main motivation for rewriting
the solver in C, except for it being cool, is to enable the application of a message passing protocol for writing a
truly parallel solver. These protocols are only available in C and a few similar languages.
The forth and last part of the thesis is about the parallel implementation of the spectral p-multigrid ﬁnite element
solver. This parallel solver is written with MPI (Message Passing Interface) on top of the C implementation
from the third section. The parallel scaling is tested and considered.

CHAPTER 1
Introduction
Physical phenomena that engineers and scientist work with are described with partial differential equations
(PDEs), rendering efﬁcient and scalable solutions to these extremely important. Looking at an airplane wing,
we can model several interesting things for engineering purposes. The wing can be modelled and tested in a
wind tunnel for creating a wing with optimal lift, or the distribution of material inside the wing can be optimized
for maximal stiffness and lightness together with minimal cost [Aag+17]. To do this, one needs to discretize
the computational domain (the wing in the example) as well as efﬁcient and scalable solvers. These two topics
are the ones we are going to cover in the present thesis – and efﬁcient and scalable solver to partial differential
equations.
At the heart of many engineering problems lie the need of the solution of a Poisson type problem, a particular
interesting partial differential equation. An example of this need is in the area of marine hydrodynamics, where
surface ﬂow is modelled [GEO19]. The surface ﬂow models are complicated, but under some assumptions they
reduce to a Poisson type problem. This need for fast and scalable Poisson type solvers was the initial motivation
for this project.
We explore a parallel p-multigrid ﬁnite element solver for the Poisson problem in this thesis. We start with
the mathematical and numerical aspects of spectral discretizations and convergence, followed by the multigrid
approach. Afterwards the solvers are rewritten in C and lastly made parallel with Message Passing Interface
(MPI) standard. In the end the scalability of the solvers are tested, followed by a brief discussion as to how this
work can be expanded on.

CHAPTER 2
Discretization by the Spectral
hp-Finite Element Method
This chapter introduces the theoretical background for solving Partial Differential Equations (PDEs) using
the spectral hp-Finite Element Method (hp-FEM), together with a discussion about how to implement this in
any given language. The main advantage for employing hp-FEM is the possibility for improving the solution
accuracy by either h- or p-reﬁnement. In h-adaptivity the polynomial order p is kept ﬁxed while the number of
elements is increased, whereas p-adaptivity is achieved with ﬁxing the number of elements and increasing the
polynomial order of which the solution is approximated. Using h-adaptivity achieves polynomial convergence
like a standard FE method [BS94], which is studied intensely in the literature and used widely in commercial
software. When the polynomial expansion is chosen cleverly, p-adaptivity can achieve the so-called spectral
convergence, which is faster than any polynomial order [HGG07].

2.3
Strong and Weak Problem Formulation
Solving PDEs numerically is usually done in one of two ways. The ﬁrst way stems from a weak formulation,
where the differential equation is multiplied with a test-function from the approximation space and integrated
over the domain. Methods from the weak formulation are usually called Galerkin methods. The solution is
then computed as a series expansion (akin to (2.3)) where the unknowns are coefﬁcients in the series expansion.
These expansion coefﬁcients have no physical interpretation and can only be used together with the chosen
basis functions to compute the series solution. The unknown coefﬁcients of the expansion are also called
modes, where the name modal approximation comes from. The well-known Finite Element Method (FEM) is
derived from the weak problem formulation, and the spectral hp-ﬁnite element method we derive later in this
work is based on the same weak formulation. The difference between the two is how spectral methods use
global (for the element) basis function,s while FEM uses local lower order basis polynomials. Actually, in both
the FEM and the hp-FEM formulation, the basis functions are also deﬁned globally, but only with local support.
The other formulation is the strong formulation, where the differential equation is required to be satisﬁed in
certain collocation points and the solution is obtained by solving this requirement in these points. Usually

Galerkin methods, because the element formulation constitutes a way of generalizing to non-standard grids.

3.2.2
The Two Deﬁning Multigrid Properties
The obvious next question is, how fast this convergence is and why this section name indicates this to be slow.
But before we get to this, we need to explore two basic properties for any multigrid method and the reason for
their convergence and widespread usage.
Looking at the Jacobi or Gauss-Seidel update formulae ((3.2) and (3.5)) these take the form of an averaging (or
smoothing) process. This means we average out the error throughout the iteration process, which is encapsu-
lated in the smoothing principle.
DEFINITION 3.2.1 Smoothing Principle [TOS01] Many classical iterative methods (Gauss-Seidel etc.) if
appropriately applied to discrete elliptic problems have a strong smoothing effect on the error of any approxi-
mation.
The next property is that a smooth error that is well approximated on a ﬁne grid can also be approximated
on a (much more) coarse grid without any signiﬁcant loss of information. This amounts to the Coarse Grid
Principle.
DEFINITION 3.2.2 Coarse Grid Principle [TOS01] A smooth error term is well approximated on a coarse
grid. A coarse grid procedure is substantially less expensive (substantially fewer grid points) than a ﬁne grid
procedure.
The principle is used for error correction and is also called the coarse grid correction (CGC) principle.

3.6
Spectral p-Multigrid Methods
The spectral discretization described in Chapter 2 is surprisingly easy to apply to the multigrid strategies of
solving differential equations. Essentially, we need to replace the discretization of the system matrix with the
spectral one based on the Legendre Gauss-Lobatto quadrature points. We need to use the matrix smoothing
described by (3.6) and the interpolation is also changed slightly. However, as described in the theory section,
the Vandermonde like matrices can be utilized to achieve the interpolation. Solving on the most coarse grid is
performed with the spectral discretization, which ensures the coarse grid correction to be very precise, even if
we use a small amount of points.
The method is called p-multgrid where the h from hp-FEM is dropped, because we only implement p-adaptivity
for the multigrid method. Alas, we cannot change the mesh during execution of the multigrid method, but this
is not nearly as interesting as the p-adaptivity.

4.1
Low- and High Level Programming Languages
In a modern setting, C is a low-level programming language. This means that the code is ’close’ to the machine
that runs it and having fewer features and abstractions compared to other programming languages. In turn you
gain the possibility of very fast running code, as the overhead introduced by the language is very small. A
programming language on the other side of the spectrum is the very popular Java language, that is a general-
purpose and object-oriented language. This language includes functionality as classes, where data (members)
and functionality implementation (methods) are fused into one thing [Wik19a]. This provides utility for the
programmer to get more work done with fewer lines of code, but it also moves the programmer further away
from the machine. Java is built for portability meaning that a program written for the Java platform should
run similarly on any combination of hardware with the required run time support [Wik19b]. This is done by
compiling Java into Java bytecode, that is then run in a virtual environment, which is run by the operating
system. These intermediate steps from the code to the machine further slows down code, making Java an
obvious non-choice for the numerical task at hand. However, a general-purpose desktop application would most
deﬁnitely beneﬁt from the high-level features of Java together with the easy portability of the Java framework.

4.2.1
BLAS Levels
BLAS provides three levels categories of operations:
• BLAS level 1 routines implement basic vector-vector operations (scaling, adding, copying, dot product,
etc.)
• BLAS level 2 routines implement matrix-vector operations (multiplication, solving triangular systems,
etc.)
• BLAS level 3 routines implement matrix-matrix operations (multiplication, rank k updates, etc.)
Intel MKL also supports sparse BLAS routines1, where sparse matrices are compressed and only non-zero
entries are saved2.
All BLAS routines have mnemonic names. For example the routines for doing double precision matrix-vector
multiplication is called dgemv that is a mnemonic for remembering double (precision) general matrix-vector
(product). All BLAS routines can be called with different numerical precision, namely single, double, complex
and double complex.
LAPACK provides a number of matrix factorization routines (LU, QR, Cholesky, etc.), routines for computing
eigenvalues and singular values, routines for solving linear systems and routines for solving linear least-squares
problems. LAPACK routines are dependent on BLAS routines hence both libraries are necessary when using
LAPACK. LAPACK has mnemonic names too and the routine for solving a general linear system is called
dgesv for double (precision) general solve.
4.2.2
Matrix Storing Order and Modern Cache-based CPUs
Storing two-dimensional arrays can be done in two ways – column-major order or row-major order. These
orderings refer to how multi-dimensional arrays are saved and accessed in memory. A multi-dimensional array
is always stored as a contiguous piece of memory, such that retrieving any given value in the array can be done
with simple arithmetic by the compiler. In column-major order the matrix is stored column-wise and row-major
ordering stores the matrix row-wise. Thus, the values of a column-major matrix are stored as an array where
consequent values are the columns of the original matrix. A ﬁgure showing how this looks in practice is shown
in Figure 4.1.
One might ask, why we need to bother much about this ordering anyway? How the machine reads values from
memory does not seem to matter for the end-user anyway, as long as the correct value are retrieved. But it
turns out to be of major importance with regards to performance of the program. On modern systems when a
value is requested by a program, the operating locates this value in memory and returns a cache line containing
the value. A cache line is a small block of contiguous memory containing several values (with the amount
depending on the level of cache) including the requested one. This cache line is saved in the memory closest
to the CPU and the fastest memory to read from. Therefore, if our program requests the next value in memory
after using the original one, this can be retrieved basically instantly, if it is present in the previous cache line
and already loaded into the cache.
Reading memory corresponding to the memory layout therefore should result in plenty of cache hits, where
the requested value is already read into the cache. The opposite case, a cache miss, is when the requested
value cannot be fetched in cache and must be found in main memory (typically RAM or, if the array cannot

4.4
Setup, Compilation and Running
To execute a C program on a machine it has to be compiled from text to machine readable code. A C compiler
has four steps: Pre-processing, compilation, assembly and linking. These are the necessary steps to take the
written text (the code) and transform it into machine level instructions. All C ﬁles are compiled individually,
then made into an executable where all functions are linked together. This can be achieved by a long call
to the chosen compiler, but writing this explicitly is both cumbersome and error-prone, while also practically
impossible for larger projects. Instead one can make a Makeﬁle, a collection of build rules for a project. With
a makeﬁle the project can be compiled into an executable by simple running the make command. A Makeﬁle
is included with the project that compiles the project into the executable Driver_MSEM.bin. Commands for
cleaning up the executable and object ﬁles are also included in the Makeﬁle. Speciﬁcally, make clean for
removing object ﬁles and make realclean for removing object ﬁles and the executable.
An important note regarding compilation is that the commercial Intel C/C++ compiler [Int19b] is used for this
project and must be present and working on the target machine to be able to compile this project. This compiler
is chosen because Intel has its own optimized implementation of BLAS and LAPACK with Math Kernel Library
(MKL) and MPI (Message Passing Interface) bindings for crating parallel code with distributed memory. Thus
we can use this compiler and link against MKL with no worries to other third party software. Furthermore and
more importantly, Intel provides the LAPACKE bindings that enable us to call LAPACK functionality natively
with C-style arrays. This will be expanded on later on in this chapter.
The Intel C/C++ compiler is a commercial software package, but is provided free-of-charge for students. Fur-
thermore, DTU has a number of licenses on its high-performance cluster. By cloning the repository for the
project on the cluster, the compiler module can be loaded with the command

which is the most recent 2018 version. The project was initially developed with the Intel 2019 update 3 version,
but this version had problems with running on the high-performance cluster at DTU. After many mails back-

after a rewrite of some parts of the compiler. The HPC support team has ﬁled bug reports to Intel for these, but

4.5.1
General Considerations
The main driver calls the requested solver and all calculations are done inside these. The sequential drivers are
located in src/Drivers.c and parallel in src/Drivers_MPI.c even though this chapter only deals with the
sequential code.
All memory allocation (and de-allocations) are handled within the requested solver. Furthermore, routines used
inside the solvers are generally not expected to do memory allocations inside them. As a rule of thumb, all
memory is allocated in the solver functions and passed to sub-routines to be ﬁlled in. First of all, this allows us
to pass more than one input to the functions to act as outputs (when we ﬁll in vectors/matrices and the like), even
though function usually only have a single purpose and therefore a single ’return’. Some routines does alloca-
tion, but they are appropriately named. An example of this is the function SetupAndAllocateGridAndMesh,
that, as the name suggests, sets up the grid, mesh, connectivity, etc. while also allocates the vectors and matrices
for these. A corresponding free routine (FreeGridAndMeshVectors) is provided that handles deallocation of
all allocated objects from the allocation routine.

to transpose the result.

4.5.3
The Banded Direct Solver
The assembled coefﬁcient matrix solved in the spectral element method (see Chapter 2) is particularly banded
in one spatial dimension. As the polynomial support is only within a given element, the coefﬁcient matrix
has a bandwidth that is two times the polynomial order. That is, for polynomial order P, the matrix has P
sub-diagonals and P super-diagonals and a diagonal. This can be seen in Figure 4.3 for 5 and 50 elements
respectively with P = 8. It is clear that the matrices are banded and that the ratio of non-zeros compared to
zeros deteriorates quickly with increasing number of elements. For the examples in the ﬁgure, the coefﬁcient
matrix with 5 elements has about 23% non-zero entries, whereas 50 elements only has about 2.5% non-zero
entries. The direct C solver described in subsection 4.5.2 sets up the dense system with all the zeros it contains.
This will hurt performance as solving a dense linear system is O(N 3), whereas solving a banded one is only
O(Nab) operations, where a is the number of sub-diagonals and b the number of super-diagonals [LeV07].
For large systems this obviously becomes signiﬁcant, and this section shows how we can use the banded solvers
in BLAS and LAPACK to solve these equations more efﬁciently. It is important to note that Matlab very easily
converts matrices from dense to sparse and that banded matrices are automatically detected4, such that the most
efﬁcient solver is used.
Banded Storage
Like BLAS and LAPACK routines for general (dense) matrices are written as xgeyyy where x is the type
speciﬁer and yyy is the given routine name, banded routines are simply named as xgbyyy. The matrices
supplied to these banded routines have to be saved in a banded format, such that only the diagonals are saved.
This format is described for the Intel version in the documentation [Int19c], where both a row and column major
layout is speciﬁed. However, using this format caused a great deal of frustration for the author of this thesis.
Whether or not this is down to bugs in the Intel code, improper documentation on Intel/Netlibs part or ineptitude
of the author remains an open question. We are going to describe the banded storage speciﬁcation, how we
implemented the solvers using this storage scheme and how we overcame the troubles of the documentation.
In short, the row banded storage packs the matrix A into a banded matrix AB such that the diagonals of A
becomes columns of AB. Similarly, the column banded storage packs A into AB such that the diagonals of A
becomes rows of AB. For banded CBLAS routines everything worked as expected. We used the row banded
layout in conjunction with the CblasRowMajor ﬂag, which is exactly how we expected things to work. For

Implementation in C with MPI
Parallel programming is writing software that can run in parallel, where multiple processes are computing (part
of) the solution simultaneously. All programs have parts that are inherently sequential, which we cannot hope
to run in parallel. Other parts, however, might be prone to parallelization. These are the parts we deal with
in this chapter, where we will describe parallel parts of the spectral multigrid solvers, which we can exploit.
Both to reduce execution time in solving the problem, but also, and maybe more importantly, to increase the
possible resolution of the solution domain. The sequential program is bound to a single CPU and thus only a
single memory bank. A parallel implementation with proper scaling can enable us to solve problem of sizes
that would be impossible on a single CPU or even a single node.
5.1
Shared and Distributed Memory Models for Parallel Computing
There are two different memory models for parallel computing; shared and distributed memory. In the shared
memory model all workers are connected to a single memory pool that they share. Processes will, of course,
have their own cache, but they will share a common memory bank of RAM. Explicit memory management is
not needed as all processes can read and write from the common memory bank. The programmer does need to
ensure no race conditions are encountered, where two processes write to the same memory location simulta-
neously such that the result becomes dependent on the order of execution. OpenMP (Open Multi-Processing)
is the de-facto standard for shared memory parallelization. It describes an API (application programming in-
terface) that enables parallelization within the node. OpenMP has bindings to C, C++ and FORTRAN. Using
OpenMP is relatively straightforward with the use of #pragmas in C/C++. The problem with the shared mem-
ory model is that it does not scale beyond the node.
To go beyond the node the MPI (Message Passing Interface) protocol is widely used in the industry and is the
choice for this thesis. MPI, as the name suggests, enables messages to be passed between processes within and
between nodes. Each process has its own memory space and explicit messages have to be send and received
between processes if they wish to share information between them, as they cannot read from the address space
of other processes. MPI offers bindings for C, C++ and FORTRAN like OpenMP. MPI implements send
and receive operations to facilitate communication between processes as well as collective communication
techniques, that allow the user to make the same call to many or all processes with a single function call. When
starting an MPI program each process (called a rank) is spawned and the program code is run for all ranks. The
programmer has to specify the parts that should only run on certain ranks as well as parts that should run on
all. All memory that has to be shared between ranks is explicitly written by the programmer. This makes MPI
programs powerful, but at the same time difﬁcult to write and prone to hard-to-debug parallel errors and bugs.
Modern CPUs are usually equipped with multiple processing cores per socket. Consumer grade CPUs usually
have 4-8 cores, whereas business grades CPUs can have up to around 64 cores per socket. Sending mes-
sages between cores on the same socket is relatively cheap, as we do not have to pass any interconnection
bands. However, sending messages between two different nodes require a communication layer between them,
which always has both a latency and a bandwidth. The DTU Compute cluster (see speciﬁcation in Appendix
[chap:cpu_information]) used for testing in this project is equipped with InﬁniBand interconnect, speciﬁcally
designed for high throughput and low-latency message passing for high-performance clusters1.

5.3
Parallel Performance
There are several well-known measures for parallel performance, where the most prominent are the speed-up
and the efﬁciency, even if they are very similar to each other. For P processes, the speed-up is deﬁned as
S(P) = T(1)/T(P),
where S is the speed-up and T(P) is wall-clock time for P processes. T(1) is usually deﬁned as the wall-clock
time for the best sequential solver. Thus if two solvers for a given problem exists, one with asymptotic running
time of O(N log(N) that cannot run in parallel, and another with O(N 2) which can run in parallel, the parallel
speed-up of the O(N 2) implementation has to be run against the wall-clock time for the O(N log(N)) algo-
rithm. This obviously makes the quadratic algorithm look bad in comparison, because the gain from running in
parallel is simply dwarfed by the superior sequential algorithm.
A crude illustration of speed-up in practice is shown in Figure 5.1. Ideally, the speed-up would scale linearly
with the number of processes used, but this does not hold in practice. At some point communication cost will
govern the execution time and the speed-up will start to decrease. After hitting the peak of the speed-up graph
(Popt), the execution time is increasing and we do not beneﬁt in any way of adding more resources to solving
the problem.
The efﬁciency E(P) is deﬁned as
E(P) = S(P)/P.
Ideally this would be S(P) = P, so that E(P) = 1, which can be achieved only for embarrassingly parallel
problems.
Another measure for parallel scaling are the concepts of strong and weak scaling. In strong scaling, the problem
size is kept ﬁxed while the number of computing cores is increased. Strong scaling is about how the program

scales, when you keep adding more resources to obtain the solution. Strong scaling is tied closely to Amdahl’s
law, that gives a function for the theoretical speed-up a program can achieve, given the fraction of code than
can be run in parallel. Say you have a program that runs for 10 hours, but 1 hour of the execution is spent in
inherently sequential code, that cannot be parallelized. The rest of the program can run fully in parallel, but the
program can never run in less than one hour, as the sequential part always has to run.
For weak scaling, the problem size is kept at a ﬁxed size per processing unit, meaning that the ratio between
computing cores and overall problem size is ﬁxed. This is tied to Gustafson’s law that provides a measure of
scaled speed-up. This stems from the observation that problem sizes usually scale with the amount of resources
available. It is not beneﬁcial to use large amount of resources for solving a small problem, so the idea behind
weak scaling is to see how the performance scales with ﬁxed problem size per processing unit. Ideally, this
should be linear, as the amount of work per processor is the same however large we make the problem size.
5.4
Implementing the Solvers
Now we describe how the spectral multigrid hp-FEM solver was implemented in parallel. We ﬁrst discuss how
the grid was set up and split between the processes, followed by some parts about how speciﬁc routines were
converted to parallel.
5.4.1
Setting up the Parallel Grid
Exploiting parallelism can be set up in a multitude of ways. For some problems the main hurdle is execution
time, where certain loops or routines can be solved in parallel to lessen the computational burden. For other
numerical programs the underlying problem can, in a natural way, be split into sub-domains that can be solved
completely (or mostly) in parallel. Often times sub-domains are not completely independent and some kind of
information sharing is required. This is usually the case for discretized differential equations, where the local
solution hinges on the surrounding points or elements.
For the problem at hand we use the latter approach to parallelism, where the global domain is split into sub-
domains that are solved on different processes. The idea is to assign the ﬁrst process to the ﬁrst part of the
problem with the corresponding elements, the next process with the next part and so forth. In order to solve the
local (to the process) part of the domain, we need to store neighboring elements, which are outside the local
process domain of interest. The chosen solution to this (very common) problem was to let all processes hold
their local elements together with two neighboring elements. These ’ghost’-elements are not meant to be solved
correctly by the local process, but they are needed for some part of the algorithm to produce correct results.

Matrix Smoothing
The implementation of matrix smoothing was discussed in subsection 4.5.4 and shown in Listing 4.9 for the

part of the parallel solver. As can be seen from the Matlab implementation in Listing 4.10 the matrix smooth-

forward substitution and a vector addition. Matrix-vector products is discussed shortly and can be done rather
elegantly without much parallel overhead. The forward substitution is unfortunately not that straightforward to
implement and required a custom matrix-vector product to be calculated in order for it to work.
The main struggle is that the forward substitution is inherently sequential. You solve the ﬁrst unknown, which
you use to solve for the next and so forth until all unknowns are calculated. The iteration matrix B is (if we
use the Gauss-Seidel iteration) the lower triangular part of the system matrix and can be seen in Figure 5.5 for
the local processes. For the parallel solver and the matrices in Figure 5.5, it is clear that we can solve the local
problem correctly for the ﬁrst process. However, all other processes cannot hope to calculate (any of) their
elements correctly, as they miss the crucial information from the ﬁxed ﬁrst point of the ﬁrst element.
The solution we came up with is the following algorithm. As discussed, we can calculate the ﬁrst process
correctly. Thus all inner ranks are correct, including the shared point between the last inner element and the
ghost element for this rank. This point is the one outlined in red in the ﬁrst plot in Figure 5.5. Having calculated
this point we send it (with an MPI message) to the next rank and insert it as the ﬁrst element (which is also
outlined in red). The local system matrices for A and B are altered such that the ﬁrst ﬁrst value is ﬁxed. The
next process can then solve all inner elements correctly, send the speciﬁc point to the next process and so forth.
This ensures that all process can solve their inner elements correctly, but we still need to update the last element
for all local ranks (the ghost element toward the next process) outlined in green in Figure 5.5. This element
is solved correctly on the next process (also outlined in green) and we need this solution locally for the next
iteration to be correct. Thus all processes except the ﬁrst sends their ﬁrst element to the process before it, where
it is inserted as the last element.
These two steps are sufﬁcient to compute matrix smoothing ’in parallel’. In parallel is in quotation marks,
because the ﬁrst step in the algorithm is not parallel at all. However this is the best solution we came up with.
As we cannot get around this, only sending a single point to the next process was the least computationally ex-
pensive we could get away with. The next part, the backwards update of the last element is done simultaneously
for all ranks and is not a large problem for the parallelism of the program.
The code for the parallel matrix smoothing is found in src/1D_funcs_mpi.c as LocalMatrixSmooth-
ing1DMPI. This function does exactly what is described above and is not super interesting, so we do not
include the code here in the report.

5.4.3
Matrix-Vector Product
Matrix-vector product is a very common operation in numerical mathematics and can be done (almost) fully in
parallel without too much hassle. Looking at the system matrices in Figure 5.3 we see that only the ﬁrst and last
point of inner ranks are missing some contribution. This missing contribution can be calculated on the process
before or after, respectively. The implemented solution to this problem is visualized in Figure 5.6. The rows
shown in green and red represent the points that need to be communicated between processes. It is important
to note that for a given row in the ﬁgure, only the corresponding point in the solution vector is communicated
between processes. The green point is to be received (because it cannot be calculated correctly) and the red
point is to be sent to the previous or next process. The red points are calculated ﬁrst, send to adjacent processes
and then the matrix-vector products are carried out locally. After this, the received green points are inserted at
the correct position in the solution, as the solution for these points were wrong locally.
The code for ParallelElementMatrixVector can be seen in Listing 5.3. After input checks we post non-
blocking receives for the ﬁrst and last point (green points in Figure 5.6). Afterwards we calculate red points
to start non-blocking send operation, before calculating the rest of the points. This way we might save a bit of
waiting time as we can calculate the bulk of the points while transferring the correct points to the surrounding
processes. Even if we only communicate one or two points, some overhead in the communication is to be
expected, which is carried out while calculations are done for the local matrix-vector product. The local points
are calculated with standard dgemv. We ensure all points are received before inserting them into the solution
vector.

5.4.4
Solving the Most Coarse System in the Multigrid Cycle
The multigrid cycle as described in section 3.4 describes how the system is solved directly on the most coarse
grid level. To to this for the parallel version, we need to collect the grid and solve the system on one process.
The global amount of points is only one larger than the number of global elements when P = 1 so we do not
have many points to calculate for.
The solution is obtained by having all processes send their local inner elements (which are only one point each)
to process 0, on which the system is solved and afterwards the global solution is distributed back to all other
processes.
The code for the solving routine can be seen in Listing 5.4. As the solve itself is done on the ﬁrst process (rank
0) most of the routine concerns this. But for this explanation of the code, we start at line 88 where all other
ranks than 0 send information to rank 0. First a single integer is sent denoting how many points the given rank
is about to send. Rank 0 knows the total amount of points, but it does not know how many each individual
process holds and thus we need to send this information ﬁrst. After sending the local number of points, the
local array is sent to rank 0.
The work for rank 0 starts at line 20 and on line 26 an integer array is stack allocated to hold the number of local
points for each rank. The following lines post non-blocking receives for the number of points to be inserted into
the array. After all sizes are received, we can start the transfers of local arrays to rank 0. On lines 53-62 we post
the non-blocking receives for all local contributions. We have ensured all ranks are done with sending their local
sizes (line 50), so we post all receives together as non-blocking, such that they can be received simultaneously.
The local contributions are placed directly into the systemRhs and is, admittedly, not the easiest code to read,
but the job gets done without any intermediate arrays or anything else slowing the computations down. After
ensuring that all local contributions have been received (line 65), we set up the global system and grid on rank
0 and solve the system with the sequential solver routine SolveSEMLaplaceDir1D.
After having solved the global problem, we broadcast this solution to all ranks (line 107). All ranks have saved
their global start index in the MPIInformation structure, so they can easily extract the local solution from the
global one and insert it into their own right hand side vector.

5.4.5
Banded Solver
A banded solver was also implemented for the parallel multigrid solver, as the arguments for why this would be
more efﬁcient are equivalent to the ones in the sequential case. The implementation of this was straightforward
by using the wrappers described in subsection 4.5.3 and the sequential banded multigrid solver, also developed
in the previous chapter. As all communication is done with regards to vectors, we could simply exchange
the matrices (matrix_t) with appropriate banded matrices (band_matrix_t) and use the appropriate banded
solvers with the wrappers.
All matrix-vector products and solves are done using the banded solvers, except the non-square matrix-vector
product for the interpolations, exactly as was the case for the sequential solvers.

multigrid solver will be compared to the sequential banded multigrid solver. The parallel performance will, of
course, be misleading. But as part of the reason for creating the parallel solvers was to expand the domain of
interest over multiple machines, we are still interested in observing this parallel performance.
Speed-up for both the banded and non-banded multigrid solvers are superlinear for some time before plateau-
ing. This might seem surprising, but considering the discussions about dense and banded storage, this can
be explained. When adding processes, the local systems become smaller. If we half the size of a quadratic
operation (such as a matrix-vector product), the new execution time will only be a quarter of the original one.
This is especially apparent for the dense multigrid as all matrix-vector operations become more efﬁcient as the
number of processes increase. The fastest execution time for the dense multigrid solver is at 40 processes. It is
nice to see the execution time decrease even when going beyond a single node and having to do communication
between nodes.
The banded multigrid solver should make all operations scale linearly, which, in theory, would have the speed-
up follow the ideal linear scaling until a plateau was hit. But as the interpolation matrix-vector product was
not rewritten to the banded solver, we still have two (down- and upsampling) quadratic operations to do. This
shows in the slight superlinear scaling for the banded solver for the ﬁrst 10 processes. The execution time
still decreases until 20 processes, where we have to take a second node into account and do message passing
between nodes.

CHAPTER 6
Outlook
6.1
Conclusion
A parallel p-multigrid solver for the Poisson problem in one dimension was successfully implemented in C with
MPI. The parallel p-multigrid solver produced signiﬁcant speed-ups compared to the equivalent sequential p-
multigrid solver. The scalability of the solver was not up to par with the expectation set at the start of the
project. This was mainly due to the global assembly process and the sequential part of the parallel smoother.
The one dimensional spectral element discretization for the Poisson problem is inherently easy to solve se-
quentially, as the system matrix for the discretization is particularly banded. This was exploited in C with the
implemented banded solvers, whereas Matlab automatically exploits the banded nature. For higher dimensional
problems the system matrix is not banded (see an example in Figure C.1b). A problem like this would be a
better candidate for parallelization, as the sequential solver should not have as built-in an advantage as in the
one dimensional case.
We were successful in developing a parallel spectral p-multigrid solver in C with MPI, but for reasons described
above and in the thesis, the performance was not as good as the initial goal.
6.2
Future Work and Discussion
There are many possibilities for future work on this project. Some of the ideas are summarized below.
• The most obvious point of future work would be to ﬁx the sequential message passing in the matrix
smoothing. The author is quite convinced that the reasonably bad scaling is entirely down to this short-
coming of the algorithm. While a solution might exist for the one dimensional case with banded matrices,
it is important to consider how this scales beyond single dimension, such that a new algorithm does not
have to be developed from the ground up, when more dimensions are introduced and the system matrix
stop being banded.
• Direct assembly of system into banded or sparse matrices. As shown the system matrix is not banded in
higher dimensions but rather sparse instead. All system matrices in this project are set up using a dense
matrix, before being converted to a banded one (which could have been sparse instead). This is both
memory and time consuming, thus a direct assembly in banded or sparse systems would be beneﬁcial.
Also, in higher spatial dimensions, this would probably be necessary, as memory demands explode for
dense systems with increasing dimensionality.
• For standard element methods, the stencil implementation is usually done matrix-free to save both mem-
ory and computational effort [GEO19]. This way sparse matrices can be avoided entirely. Whether or
not this is feasible or not in the current setting we are unsure about, but it might be possible to create a
better and more memory efﬁcient solver this way.
• Using graphics processing units (GPUs) instead of CPUs for the parallelization. The recent developments
using GPUs for numerical computations could be interesting for this setting. A lot of work has already
been completed with regards to multigrid ﬁnite elements solvers on GPUs (e.g. [EMG11] for free surface
waves). If some of the issues regarding the parallel performance were ﬁxed, GPUs might add even more
computational power to the solvers.
• Using an already established framework for some computations. The DTU Compute GPULab Library
[Gli13] might be the basis of the multigrid method, for which the spectral solver could be built on top of.
==PAGE:72==
64
Outlook
• Generalizing the solvers to higher dimensions. The theory of spectral element methods in higher spatial
dimensions is readily available and discussed in the literature, see e.g. [Xu+18]. This would have to work
together with the above points in order to achieve good speed-up, but the discretization itself is not new
knowledge. Also the p-multigrid versions have been shown to work in up to three spatial dimensions
[Las18].
