Abstract
With increasing amount of data in the universe, there has never been a beter need to store the
data in a more compact form than is in today’s world. Tere exist a lot of diﬀerent compression
techniques. In this paper, we will be discussing reference compression for compressing strings.
What if the strings that we want to compress are highly similar to each other ? For instance,
DNA or protein sequences. Can we ﬁnd the best reference string that can best represent the
other strings by reducing the overall memory consumed ? And can this compression also
retrieve any random character in adequate time ?
Tis thesis is focused on developing a lossless compression algorithm for highly similar strings
based on reference compression. Given a reference string R and a collection of strings Si where
i = 1 to N (where N is the total number of strings) that are considered very similar to each
other, Si can be represented as a sequence of phrases from R. Tis can be done by mapping
the initial position of each phrase from Si to its corresponding position in R. A character in a
random position from Si can easily be recovered by retrieving the position in R it is mapped
to and fetching the character at that position in R. Te reference string R can further be
compressed using a variant of Lempel-Ziv 78 [1] compression.
A beter compression ratio using this algorithm is provided by using a good reference string,
so as to represent larger phrases in Si with respect to R. Te various techniques used to ﬁnd
a good reference string are discussed in this paper.

Chapter 1
Introduction
Tere are numerous diﬀerent compression techniques available to compress a collection of
strings. Tere are also many instances where the collection contains strings that are highly
similar to each other. We can exploit this property of high similarity, also resulting in an
improved compression ratio. In this thesis, we are going to use the technique called reference
compression which is greatly eﬀective if the collection of strings we are dealing with, are highly
similar to each other.
Reference compression is a technique where a reference string is selected and the collection
of strings to be compressed are divided into long phrases that exist in the reference string.
Tese phrases are then encoded with their position in the reference string. In this paper, the
reference string is a combination of one or more strings from the collection. Since, the strings
are highly similar, we can parse longer phrases reducing the number of overall phrases to be
stored and thus optimising memory.
An example of highly similar strings is genome sequences. Te development of new tech-
nologies has made genome sequencing dramatically cheaper and easier, and the number of
complete genome sequences is growing rapidly. A genome sequence is the complete list of
the nucleotides (A, C, G, and T for DNA genomes) that make up all the chromosomes of an
individual or a species. Within a species, the vast majority of nucleotides are identical between
individuals.
Version control of codes is another example of highly similar strings. Whenever, some code is
added or deleted, a new version is created.
In this thesis, we also support random access on the compressed strings, that is, given a string
index and a position, we can return a character or a substring of length l without decompressing
the entire string. However, it is assumed that the collection of strings to be compressed will
not be modiﬁed. Terefore, we do not support any edits (insert/update/delete) on the string.

Chapter 2
Problem Description
In this report, we study a two-level compression, namely, reference compression to compress
the collection of strings and a variant of Lempel-Ziv 78 [1] compression to compress the ref-
erence string itself. We describe the algorithm used to perform the reference compression
and the variant of LZ78 compression on the reference string. While running the compression
algorithm, we use the concept of ﬁngerprints (described in later sections). Dictionary of ﬁnger-
prints have been extensively used to select and identify newly or previously occurring phrases
and their locations.
We also discuss various techniques to select a good reference string, that have been investigated
as part of this thesis, in order to improve the compression ratio.
Lastly, we describe how to perform random access on a string that has been compressed using
the reference string, without having to perform any decompression.

A sub string or a preﬁx of length limit is called a ﬁngerprint and we denote it by fp. Finger-
prints are great for patern matching as they can be stored as a key in a dictionary and due to

it can be discarded once the process of compressing the strings has been ﬁnished. We don’t
require such a dictionary for random access or even decompressing the strings.

Chapter 4
Previous Work
With the growing amount of data, there is a desire to ﬁnd beter and eﬃcient ways for storage.
Terefore, compression is a highly popular subject. We are going to discuss some of the most
widely used compression techniques in this section.
Jacob Ziv and Abraham Lempel suggested a compression algorithm in 1977, popularly known
as LZ77 [2]. LZ77 is the basis for several popular compression tools such as gzip and 7zip, and is
shown to compress well under certain measures of compressibility [3]. Te LZ77 algorithm is
based on the idea of a sliding window. Te algorithm only looks for matches in a window a ﬁxed
distance back from the current position. Phillip et al. have provided an implementation for
the sliding window Lempel-Ziv (LZ77) problem and the approximate rightmost LZ77 parsing
problem [4] using suﬃx tree [5]. Artur et al. have provided an implementation using suﬃx
arrays [6].
Te Lempel-Ziv 78 or LZ78 algorithm is based on a more conservative approach to adding
strings to the dictionary. Unix compress and the Gif format are based on LZ78. Lempel-Ziv-
Welch or LZW is a variant of LZ78 and it is the one that is most commonly used in practice. Te
algorithm maintains a dictionary of strings (sequence of bytes). Te dictionary is initialized
with one entry for each of the possible byte values, that is, strings of length one. As the
algorithm progresses it will add new strings to the dictionary such that each string is only
added if a preﬁx one byte shorter is already in the dictionary. Te algorithm is described in
detail by Guy E. Blelloch [7]. Dheemanth H N has also discussed the algorithm with examples
in his research paper in 2014 [8].
In 2013, Phillip et al. published a paper on grammar based compression [9], where one replaces
a long string by a small context-free grammar that generates the string. Te paper presents a
novel grammar representation that allows eﬃcient random access to any character or substring
without decompressing the string. It introduces several new techniques and data structures of

5.1
Reference compression
Let R and Si be strings over an alphabet σ of length r and s, respectively. Te string R contains
at least one copy of each character in σ. Te reference parsing of Si with respect to R parses
Si into k phrases p1, ..., pk greedily from lef-to-right. To obtain a phrase pi, we ﬁnd a longest
substring of Si starting at position x that matches a substring of R starting at position y. Te
reference compression consists of the string R and the sequence of phrases p1, ..., pk where
each phrase is encoded as a 2-tuple as follows :
• phrase start position in Si
• phrase start position in R
Tis also means that the former part of the tuple is in an increasing order and can come in
handy when implementing a binary search algorithm to ﬁnd a character in Si at position z.

5.2
Compressing the reference string
In order to further optimise the space needed to store our reference compression, we also com-
press the reference string. For this, we have chosen a variant of Lempel-Ziv 78 [1] algorithm
which is one of the most commonly used in practice. Te data structure for this compression
consists of a sequence of 3-tuple where each tuple stores the following :
• current position of phrase in the string
• respective position of phrase existing earlier in the string
• extra character
Afer compressing the reference string, we are lef with a 3-tuple encoding of reference string
and 2-tuple encoding for each of the strings in S.

Chapter 6
Implementation
Given a ﬁngerprint of size limit, a reference string R of size r and a collection of strings
S = S1, S2, ..., SN that need to be compressed, the implementation consists of a preprocessing
step and the reference compression step. But, ﬁnding a good reference string is a challenge on
its own. And, it needs to be done even before the preprocessing step. We have used various
techniques to ﬁnd a good relative string and they are described in further sections. Given
a random string and a random position, we also describe the technique(s) used to fetch the
character from that position.
6.1
Preprocessing
Te data structure used here are two dictionaries. For the ﬁrst dictionary D1, key is the ﬁnger-
print and value is the list of start positions in the string where the ﬁngerprint exists. For the
second dictionary D2, key is the character and value is the its ﬁrst position in the string. We
read the reference string from lef to right and each new ﬁngerprint is inserted as a key in D1,
the value is a list initialised by the current position. For each existing ﬁngerprint, we update
its value by extending the list with the current position.
At the same time, we also update the second dictionary D2 by inserting each new character as
the key and the current position as its corresponding value. If the character exists in this dic-
tionary, we simply ignore it. Since, the programming language chosen for the implementation
was C++, its inbuilt data structure unordered map is used which acts as a dictionary.

6.2
Reference compression
Each string that needs to be compressed is read from lef to right. Let Si be the string that is
currently being processed and let x be the position we are at. Ten, we select the ﬁngerprint
of size limit as fp = Si[x, x + limit −1] and verify if it exists in D1. If it does, then we
process R at each position fetched from D1 to ﬁnd out which has the longest matching phrase.
If the phrase doesn’t exist, then we check if the character c = Si[x] exists in D2. If so, then
we encode it with its position from R which is found by looking up c in D2. In case, even c
doesn’t exist in D2, then we expand R by appending c at the end simultaneously mapping c in
D2 with this position in R. Algorithms 2 and 3 describe the pseudo code used for compression.

appending the reference string in the previous section with the next longest string is expected
to cover the phrases that were not covered earlier. Tis technique performed beter than the
previous one for almost all test datasets.
6.4.4
String with maximum common ﬁngerprints
Tis technique ﬁnds the string that has the most number of ﬁngerprints such that those ﬁn-
gerprints also exist in any other string except itself. Eventually, this string was used as the
reference string. In case the longest string has many repetitive phrases, this technique could
turn out to be useful. Te data structure used to create such ﬁngerprints is another dictionary.
Let’s call the dictionary D4. Te key for D4 is the ﬁngerprint and value is an array of booleans
and the size of the array is one more than the number of strings. So, each boolean value be-
longs to the corresponding string. It is set to true if the string on the same index contains the
ﬁngerprint. But, the last index stores if the ﬁngerprint exists in more than one string. Te
algorithm 5 describes the method to create D4.

6.5
Processing random access of a character, given the position
Given an index i and a position j, we can fetch the character at position j from string S[i], by
performing a binary search on the ﬁrst parameter of the 2-tuples of vTuplesStrings[i] for j to
ﬁnd its corresponding position in R. Let’s say this position is pos1. Once, pos1 is recovered, we
can again perform a binary search on the ﬁrst parameter of the 3-tuples of vTuplesReference
for pos1. If the second parameter pos2 of the tuple is same as pos1, then the character to
retrieve is the third parameter char. Else, we need to recursively search vTuplesReference
making pos1 ←pos2 until pos1 = pos2 . Te pseudo code for this algorithm is not shown as
this is based on the binary search algorithm and it is assumed that the user is familiar with the
algorithm.
Te time used to fetch a character from ith string is given by O(log |vTuplesStrings[i]) +
log |vTuplesReference|) and is, of course, bounded by the size of the strings, thus giving
worst case running time as O(log |Si|) + O(log r).

Chapter 8
Conclusions & Future Work
In this report, there have been presented various solutions for reference compression by using
diﬀerent techniques to select a good reference string. It can be seen from the experiment results
that by selecting a large reference string, we improve the compression ratio. Te improvement
can be compared with just using a traditional Lempel-Ziv 78 compression.
8.1
Complexities
We can take, for instance, the method that has shown the best compression ratio for our test
datasets, that is, mere concatenating longest and second longest strings. Tis can be done in
O(n) time. In the preprocessing step, making a dictionary with ﬁngerprints as the key with a
very simple technique also runs in O(n) time. Te reference compression algorithm reads each
string from lef to right in linear time. For each new phrase encountered in the string Si, the
corresponding longest phrase from the reference string R can be found in time proportional
to the size of the vector in the dictionary (value corresponding to the ﬁngerprint, let us call the
upper bound |vmax| for simplicity) and size of the longest phrase |pmax|.
Terefore, the total running time for compressing the collection of strings is O(n ∗|vmax| ∗
|pmax|).
In order to compress the reference string, we read it from lef to right and insert and fetch
phrases from a dictionary. Tis gives the running time to compress R as O(r).
Te space occupied by entire compression data structures is given by O(|vTuplesReference|)
+ PN
i=1 O(|vTuplesStrings[i]|).
Finally, performing random access by using binary search takes O(log |vTuplesStrings[i]) +
log |vTuplesReference|) time if we are processing ith string.
