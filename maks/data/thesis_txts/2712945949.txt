Abstract 
 
As gaming as a service spreads rapidly, maintaining an audience is a major challenge. Massive data 
collection makes player modelling shine like never before. This allows developers to understand their 
audience and tailor their games to their players. Each game is specific and the main challenge common 
to many computer science studies is data collection. Building a generic model based on common data 
from different games can help in the early development of video games. In this study, we present an 
approach to estimate players’ skills in a horror game based only on their mouse and keyboard usage. 
Using game sequences of variable duration, we constructed two methods, one based on the 
calculation of features such as the average cursor speed, and the other based on the temporal 
evolution of input actions. Random forest estimators allowed us to predict players’ skills with more 
than 75% accuracy as a binary classification using features extraction. The results showed that mouse 
movements are better at differentiating players than keyboard use. With further studies providing a 
larger amount of data, we are convinced that systems to automatically adapt the difficulty of games 
can be implemented.

Introduction 
 
Player modeling is becoming one of the most important data analysis challenges in the video game 
industry. Its potential applications are very varied and benefit all the actors of video game 
development. Among others, it allows to identify which elements of the game are most appreciated 
by which type of players (1). The level creation can then be adapted to the actual player, and 
procedural content generation can benefit from it (2). Furthermore, one of the aspects that publishers 
are most interested in is to understand player churn. A lot of researches have emerged in recent years 
on modeling players to predict when or why players leave the game (3) (4) (5). 
More generally, player modeling allows the game to be tailored to individual players. The models can 
be taken into account for designing the in-game experience. That can be achieved by adjusting the 
behavior of non-player characters or by adjusting the game environment. One popular form of 
adaptation is adaptive difficulty. Rather than a fixed difficulty, or one that can be adjusted according 
to certain levels, adaptive difficulty adjusts dynamically in real time in order to avoid frustration or 
boredom for example. Some games have implemented it and have been appreciated by players (6), 
and other serious games have used it to improve player motivation (7). 
To create these player models, the algorithms need a lot of data that can be divided into three 
categories. The gameplay data corresponding to the actions taken by the player in the game (shooting, 
opening a door etc.) define the player's behavior. The game context data corresponds to the data 
provided by the game itself (game duration, current level etc.). Finally, the player's physiological data 
can also be considered (speech, gestures, etc.), even if they are less used. It is then necessary to choose 
which data to provide to the algorithm in order to build relevant player models. 
However, these data depend strongly on the game itself. Indeed, each possible action and each 
environment depend on the game. These algorithms must therefore be adapted to each use. Yet, David 
Buckley, Ke Chen and Joshua Knowles (8) suggested another approach by trying to model the players 
based on the keyboard and mouse input data. This approach was more generic, allowing greater 
flexibility of use between different game genres. This study is based on the same observation: raw 
inputs data common across games may be used to create fast implementable player skill models. 
Our goal is to model an aspect of player behavior with generic data that can be used to improve games. 
We aim at: 
- 
Confirming the correlation between mouse and keyboard usage and players’ skills 
- 
Identifying the most important features of mouse and keyboard usage 
- 
Evaluating the viability of a player modeling system based on generic data common across 
games 
We expand the previous work to a different game genre and use two data processing techniques. First 
in Test bed game: Church in The Wild, we present the game developed in collaboration with the Danish 
Filmschool of Denmark used as a testbed for an experiment with players. Then, we present in Data 
collection this experiment by describing how and which data were gathered from the players to make 
player skill predictions based on their mouse and keyboard usage. Next in Data collectionMethodology, 
we explain how the data was processed, with different processing techniques used to analyze raw 
inputs data. We then use a Random Forest classifier and two Recurrent Neural Networks for 
classification and present the results in the Results section. Finally, we put these results in perspective 
in the Discussion section.

I. Related work 
 
A. Player modeling goals 
Player modeling is worth of interest in several areas. First of all, by analyzing the data on the players 
themselves (cultural background, age, gender, etc.), it is possible to draw up player profiles. These can 
for example be useful for marketing purposes. With game related data, there are two major goals: 
analyzing the behavior and analyzing players’ experience. The analysis of in-game behavior allows to 
study the players’ reactions - trying to understand his path through the levels and his other choices in 
game. The analysis of player’s experience aims to understand how the players feel.  Many studies on 
player churn focus on player satisfaction or motivation (9) (10). It is worth noting that player modeling 
is derived from the broader field of user modeling, and a lot of researches extend to the field of 
affective computing or more generally to that of human-computer interaction. This is why some 
techniques presented in this section come from these fields. For instance, studies on modeling player 
churn are often based on sociological theories (5).  
 
B. Data collection sources 
1) Game metrics 
The data needed for player modeling come from multiple sources. The data collected from the game 
itself are the most used. These data can be classified into two categories: player metrics and game 
metrics, the former being a sub-category of the latter. Player metrics are about the state of the player 
(level, hours spent in the game etc.) and the overall actions of the player.  Game metrics add data from 
development such as software performance metrics. The player metrics therefore change depending 
on the game, and the player modeling techniques adapt from it. Rafet Sifa, Anders Drachen et al. 
created a team recommendation system on the MMOFPS Destiny 2 according to player profiles based 
on their in-game characteristics: average ability cooldown time, average kill distance etc. (11). Tobias 
Mahlmann, Anders Drachen et al. tried to predict player motivation to play Tomb Raider: Underworld 
by counting the number of hours played or the causes of in-game deaths (12). We can add to these 
metrics the information from the player's registration such as his age or his geographical location. 
 
1) Surveys 
In order to obtain datasets of values to predict, the research relies heavily on surveys sent to groups 
of users. These surveys provide relatively reliable information. They are often created from theoretical 
models that indicate the most relevant issues in the domain under study. Ahmad Azadvar and 
Alessandro Canossa compared different formats of questionnaires asking about gamer satisfaction 
based on different theories. They designed the Ubisoft Perceived Experience Questionnaire (UPEQ) 
which attests to gamer satisfaction with a higher granularity (13). Christopher Pedersen, Julian Togelius 
and Georgios N. Yannakakis also designed a questionnaire measuring emotional preferences based on 
an alternative 4-choice forced-choice protocol (2). 
 
2) Physical sensors 
In addition to the game metrics and surveys sent to players, some more invasive studies use sensors 
to measure the physiological responses of the players. These data usually measure the emotional state 
of the users (stress, fear etc.). Michal Adam Pikulski measured the fear of players on a horror game via 
==PAGE:10==
10 
 
their heart rate and skin conductance. To do this, he compared the data from the sensors to the 
players' annotations indicating their stress level (14). 
 
3) Device usage 
Finally, raw input data such as mouse movements or keyboard presses have been used in a small extent 
in user modeling. Primarily in affective computing, researches showed an interest in detecting a 
correlation between participants' stress and their use of these devices. David Sun, Pablo Paredes, and 
John Canny aimed to prove the existence of this correlation by using a theoretical mass-spring model 
to model the user's arm with the mouse. As a result, they measured the mouse's displacements and 
reported these measurements in the model calculation in several stress situations. The exercises for 
the participant consisted of classical mouse operations: drag&drop, point&click etc. They then 
demonstrated a correlation between the characteristics of the model and the stress state of the 
participant measured via sensors (ECG, EDA) and questionnaires (15). However, Paul Freihaut and Anja 
S. Göritz obtained contradictory results on similar exercises. They did not use theoretical models and 
measured the stress of the participants only on the basis of questionnaires. Instead, mouse 
movements were interpolated and the movement characteristics were then calculated such as the 
average cursor speed or the total distance covered. These results were then fed into statistical and 
deep learning models to predict the stress level: Logistic Regression, K-Nearest Classifier, Support 
Vector Machine (SVM), Random Forest Classifier, convolution network (CNN) and long short-term 
memory networks (LSTM). The accuracy of these models is about 66%, almost the same for all of them 
(16). 
In player modeling, only a few researches focus on keyboard and mouse inputs data. David Buckley, 
Ke Chen and Joshua Knowles studied the relationship between these inputs and players’ skills in a first-
person-shooter game over game sessions of up to 3 minutes. The input data is processed similarly to 
(16) and several features are extracted: average cursor position, total duration of keystrokes etc. The 
skills of the players were evaluated according to two criteria: the recent number of hours spent playing, 
and the number of different FPS games played (these criteria were more appreciated than the 
kill/death ratio of the players). Three models were tried to predict these two numbers: multi-layer 
perceptron (MLP), SVM and Random-forest classifier. The random forest was the most successful, with 
an accuracy of about 76% on one minute input sequences (8). 
 
C. Player modeling approaches 
 
1) Player profiling vs Player modeling 
A player model can belong to two categories: player profiling and player modeling. Player profiling is 
based solely on "static" player data that does not alter during a game session. Information of static 
nature include geographic location, age, and cultural background of the player. Player modeling, on 
the other hand, is based on data collected during game sessions. While profiling is interested in 
mapping the state of players, modeling is more oriented towards predicting behaviors or interactions 
with the game (9). However, these two categories are not inseparable and it is not uncommon to 
attempt to improve modeling with already established player profiles. For example, David Melhart et 
al. attempted to predict player motivation based on 1) player profiles describing their play style and 2) 
player metrics (hours spent playing, level, etc.). With data from 298 players, they measured a 4% 
increase in model accuracy when using player profiles compared to predictions based on metrics alone 
==PAGE:11==
11 
 
(3). This study focuses on player modeling, as this is an attempt of predicting players’ skills (behavior) 
with raw inputs data (dynamic information). 
 
2) Theoretical and data-driven approaches 
There are two main approaches to player modeling: a "top-down" approach that uses a theoretical 
model to analyze the data, and a "bottom-up" approach that uses the data alone to derive meaning or 
make predictions. Zoheb Borbora, Jaideep Srivastava et al. compared these two methods to predict 
player churn. The theoretical one relies on 4 different types of input data defined with motivation 
theory, while the data-driven one uses 14 different types of arbitrarily chosen data. The theoretical 
data is more complex, such as the ratio of interaction between groups of players, than raw data, such 
as the total number of experience points of a player. With an ensemble classifier, their goal was to 
predict whether the player would stop playing or not. The accuracy of the model was slightly higher 
with the data-driven approach than with the theoretical approach. However, the researchers showed 
that this loss of accuracy can be compensated by a better understanding of the results with the 
theoretical method (5). 
Data-driven methods for player modeling are increasingly used due to the growing collection of game 
data. Early analyses of game metrics used common statistical models such as Support Vector Machines 
(SVMs) and ensemble classifiers (including Random Forests). David Melhart et al. chose to use an RBF-
SVM model to predict player churn because they can provide satisfactory results with limited data and 
features. Their model had an accuracy of 94% (3). 
 
3) Inference using statistical models 
For processing raw input data, both SVMs and Ensemble Classifiers are mainly used. In (8), the authors 
obtained worse results with SVMs than with Random Forest, the latter generally performing better on 
higher dimensional data. Paul Freihaut and Anja S. Göritz also obtained better results with Random 
Forest than with SVM for correlating mouse movements and stress (16). Additionally, random forests 
have the feature importance characteristic, which allows to understand which features are the most 
decisive in the predictions and therefore makes the estimator more interpretable. However, these 
studies lack training data to evaluate deep learning models. 
Unsupervised statistical approaches are used mainly with game metrics. They allow to derive player 
behaviors from their game data. When Rafet Sifa et al. built their recommendation system for Destiny 
players, they created clusters of players from their game data presented above. Their approach 
differed in that a player belongs to multiple clusters. Indeed, each player's data is divided into multiple 
sets, and each set belongs to a cluster. Each set corresponds to a category of data such as the 
character's statistics or performance against in-game enemies. Clusters were created with archetypal 
analysis to identify extreme behaviors. The recommendation system then relied on the membership 
of these clusters for a matchmaking system (11). Alessandro Canossa et al. also categorized Tom 
Clancy's The Division players according to their in-game behavior with a clustering algorithm. However, 
they first performed a sequential pattern extraction. This first approach allows to highlight recurrent 
behaviors among the players. The clustering that follows was then based on a fixed number of distinct 
sequences, and the number of times they had been detected in each player (17). 
==PAGE:12==
12 
 
4) Inference using deep learning models 
Finally, supervised approaches with neural networks are rarely implemented because of the very large 
amount of data needed to train them. A few attempts can be observed with convolution networks 
(CNN) or LSTM (16), but the results obtained do not exceed those of the previous statistical models.  
Deeper understanding of statistical and deep learning models is provided in the Methodology and 
Future work sections. 
 
5) Study contribution 
This study is placed between raw input data analysis and player modeling. Related work has shown 
methods for processing these data and their implementation for distinct purposes. Research on player 
modeling is mainly based on game metrics, which limits the reusability of the model created to the 
game at hand. In addition, few studies focus on modeling players’ skills. Finally, the results obtained 
by analyzing mouse movements and keyboard input are currently only able to correlate these data 
with a targeted behavior of the user (stress detection or evaluation of his skill level on a task), and not 
to predict these behaviors.  
The approach suggested in this paper is generic. Based on the input data, the model obtained can be 
more easily reused on other games using the same devices and the definition of skills is similar. This 
research also attempts to implement two different methods presented above (features extraction, 
sequence analysis) in order to compare the performance of the different models on the evaluation of 
players' skills. The horror game used in this study also allows us to extend the research to a game genre 
that is not yet widespread in this field.

II. Test bed game: Church in The Wild 
 
A. Game description 
Church In the Wild (CITW) is a first-person horror game where the player must survive a psychotherapy 
simulation that went wrong. The world is procedurally generated on each launch, made with long 
corridors and rooms arranged to make the player feel lost. An AiDirector (14) is specifically designed 
to make him frightened at all times by triggering scary events and spawning a monster to kill him. 
Artifacts located on the map grant the ability to make it disappear for a while. The only way out is to 
understand what happened to the player character by helping the other non-player characters (NPCs) 
to get past the dark events in their lives. 
The game was developed in a team of around 15-20 creators, with 3 programmers. The game was 
made with Unreal Engine, a popular game software made by Epic Games (18). The data collection 
system was developed by me and another master student, Michal Pikulski.  
During this study experiment, the game was in its alpha version, allowing to have a full gameplay loop 
despite the lack of story and existence of bugs. The goal for the player was to find two NPCs and talk 
with them twice while killing the monster when it appeared. 
Figure 1 shows the main menu of the game. You can click on the image to access a video of the game. 
  
Figure 1: main menu – link to full playthrough of the game. 
 
B. Gameplay 
The gameplay was divided into three stages: walking around, dialoguing with NPCs and surviving the 
monster hunt. A full playthrough is available to watch with the link given on Figure 1. 
- 
While walking in the map, the player needed to find his way across the maze-like map (figure 
2). Procedurally generated, the map was formed of long corridors, rooms, decorative items, 
==PAGE:14==
14 
 
doors and mirrors. Manually designed rooms contained NPCs that the player could interact 
with (dialoguing). In the other parts of the map, the player would likely hear scary sounds and 
see strange things happening: baby screams, items suddenly destroyed, characters peeking at 
the end of corridors etc. Eventually, if the AiDirector measured a strong enough fear from the 
player, a monster would appear in the map and hunt the player.  
 
- 
When dialoguing with NPCs, the player could select dialogue options to move the story in 
different directions. While in the released version of the game the player will have to carefully 
choose these options, the experimental state of the game in this study allowed only fixed 
options to create a time limited session of gameplay. 
 
- 
The hunt began when the monster spawns on the map (figure 3). Suddenly, the player heard 
a screaming noise and his speed was greatly increased. Either the player managed to find a 
mirror (figure 4) and look at the monster's reflection in it (which kills it), or the player failed to 
lose the monster and it gradually absorbed his life points until he died. In the latter case, the 
player was redirected to the main menu and could then resume the game before the hunt 
begins.

C. Controls 
The game could only be played with mouse and keyboard.  
- 
Movement: WASD keys. A button in the main menu allowed to switch between AZERTY and 
QWERTY configurations. 
- 
Look: mouse movement. The player could look around by dragging the mouse. 
- 
Doors: left mouse button. The player could open and close NPC room doors by clicking when 
looking at them and dragging the mouse in the door pull or push direction (left or right). 
- 
Dialogue: mouse click. The player selected dialogue options by looking and clicking at them.

D. Player’s skills 
Players’ skills are generally subjective and depend strongly on the game. In a platformer game such as 
Mario Bros., the number of deaths or the time completion can both indicate different skills - as a player 
can focus on not dying while another on finishing the game as fast as possible.  
In this test game, while players focused on different aspect on the game, all of them should kill the 
monster in order to survive. Comparing the number of deaths and kills of the player provides a 
relatively stable indicator of player performances. Thus, this study aimed at building models to predict 
if a player mostly killed the monster or got killed by it, based on mouse and keyboard (M&K) inputs. 
Additionally, we evaluated player ability on game controls (WASD keys and mouse movements). As 
multiple participants didn’t play video games usually, they were not accustomed to move a virtual 
character especially with M&K. This evaluation was used to see if there is an observable link between 
“good” or “bad” players and their M&K usage.

III. Data collection 
To gather data to build players’ skills models, two experiments were conducted. A first one in-house 
and another one online in order to reach more participants. 
 
A. Experimental setup 
The first experiment ran over a two-week period and was done in-house to ensure the quality of the 
data. 17 participants played the game described in the previous section once, and finished it after 
playing between 6 and 20 minutes. 
This experiment was conducted with another master student, Michal Pikulski, for a research based on 
the same game (14). Due to that, sensors had to be attached to the participants (2 pads on two fingers, 
2 on the collarbone, one behind the ear and one placed on the frontal lobe). These sensors were 
considered when analyzing the players inputs, as those on the fingers might have make the mouse 
more difficult to move.  
Before playing, the controls of the game were explained to the participants. They were told that the 
game is self-explanatory, giving hints about what to do.  
In this first experiment, the player had to find an item, give it to an NPC and return to the starting point 
to end the game. While walking around, a monster could spawn at random times chasing the player 
who could kill him with multiple mirrors placed in the map. The player could die twice, preventing the 
monster to spawn after that. 
Data was collected silently (without the player noticing) and anonymized through a UUID code 
assigned on each game. 
 
The second experiment last about a month and volunteers could download the game online and play 
by themselves. 23 participants played the game for sessions between 0 and 35 minutes. Compared to 
the first experiment who took place in Denmark, the second one had participants from France as well, 
with a different keyboard configuration. Almost all participants were monitored online while playing 
the game. This second experiment was conducted by one researcher this time. 
Before playing, controls and goals were explained in a file attached to the game folder and also located 
in the main menu.  
During the game, the player had to find two NPCs and talk to them twice to finish the game. With a 
designed AiDirector, a monster spawned when player’s fear reached a certain level while walking 
around, and chased him again. The player could kill the monster with multiple mirrors on the map. In 
this second experiment, the monster could spawn indefinitely. Many players played the game without 
finishing it. 
The data was collected when the game closed and sent to a drive folder after being anonymized 
through the same process.   
 
 
 
==PAGE:18==

B. Captured data 
During each game, all mouse and key events were recorded in a .json file. Alongside, data (telemetry) 
about the game (number of deaths, kills etc.) was recorded as well. 
 
1) Mouse and key events 
Every event coming from the mouse and keyboard devices were recorded during game sessions: 
- 
Mouse event: timestamp, mouse axis (X or Y) and mouse rate. 
- 
Keyboard event: timestamp, keycode and key state (pressed or released) 
These events were recorded through the Unreal Engine game software, which provides an API to 
detect any modifications of inputs coming from devices. 
 
2) Telemetry 
The telemetry corresponded to the data coming from the game itself. It contained logs of events that 
occurred during play such as when a player died or when the game closed. After parsing the logs, the 
features extracted were: 
- 
Total duration of the game 
- 
Number of times the monster spawned 
- 
Hunt durations 
- 
Number of deaths 
- 
Number of kills 
- 
Player velocity over time 
 
3) Data distribution 
In this section, we present a summary of the telemetry data gathered with each experiment. Only the 
data coming from more than 3 minutes and less than 40 minutes of gameplay were kept as the others 
are non-viable. This result in a drastically reduced number of players for the second experiment. 
 
a) Number of participants 
In the first experiment, 17 participants played the first version of the game. In the second (online) 
experiment, 23 participants different from the first played a second version of the game. However, 
due to bugs in the game and bad game usage (early closing), only 9 gameplays were usable. In total, 
the data coming from 26 participants were usable. 
 
b) Gameplay duration 
For the first experiment, the average gameplay duration is about 11 minutes with a standard deviation 
of 4 minutes. The quickest player finished the game in 6 minutes and the slowest took 20 minutes to 
complete. 
For the second experiment, the average gameplay duration is around 13 minutes, with a standard 
deviation of 9 minutes. The players could quit this game without finishing it. The smallest gameplay 
duration is about 6 minutes while the longest is more than 30 minutes.

IV. Methodology 
In this section, we look at the different methods used from formatting the data to predicting a player 
category. 
 
A. Sampling 
We observed that the distribution of sequences in the two categories was not always perfect. 
Depending on the duration, some categories represented 30 to 40% of the total data. These 
imbalances are common in machine learning and a common technique is to randomly copy or delete 
some samples from the smaller or larger category respectively. With this random sampling technique, 
two methods can be distinguished: 
- 
Random Oversampling: randomly duplicate samples in the minority class 
- 
Random Undersampling: randomly delete samples in the majority class 
These techniques were only used on the training set. Indeed, the aim is to limit the weight of each 
class when the model learns. If the test set is modified as well, a bias is introduced that mixes the data 
distribution, and the calculated metrics will not be viable. 
 
1) Random Oversampling 
One way to restore the balance between classes is to generate samples of the least represented class 
in the training set. The naivest method - the one we used - is to randomly select samples of the under-
represented classes and duplicate them in the training set. 
Two other approaches are worth mentioning: the Synthetic Minority Oversampling Technique 
(SMOTE) and the Adaptive Synthetic (ADASYN) sampling method. While random oversampling 
duplicates existing samples, these methods generate new samples by interpolation with the existing 
ones. SMOTE generates a new sample based on its k nearest neighbors, taking one random and 
creating a value by interpolation between the selected sample and the neighbor.  ADASYN is more 
complex and focuses on generating samples next to the original samples which are wrongly classified 
using a k-Nearest Neighbors classifier (19).  
These techniques will not be detailed here as they were not used in obtaining the results. 
 
2) Random Undersampling 
Another way of balancing the data is to select only part of the samples from the most represented 
class. Again, we used a naïve method by randomly selecting a subset of data for the targeted classes. 
More complex methods exist, one of which called the Cluster Centroids method. It works with a K-
Means clustering algorithm to replace data from the majority class with centroid of identified clusters. 
This method has been found useful in some cases but depends strongly on the nature of the dataset 
(20) (21). 
 
B. Scaling 
Data scaling is a common requirement for many machine learning algorithms. Inconstancies in datasets 
are often a source of a higher probability of mispredictions. Different scaling methods have been

developed in the literature to deal with these issues and other outliers in the data. The choice of the 
scaler depends strongly on the model used for making predictions. Studies have shown variable 
performance depending on the scaler-model combination (22). For instance, (23) have shown good 
performance with a min-max normalization technique when using a Support Vector Machine (SVM) 
model with RBF kernel, but better performance with standardization technique when using SVM with 
a linear kernel.  
However, when using decision-trees-based classification methods (Random forests, gradient boosting 
trees, etc.), scaling the data shouldn’t influence the results. Indeed, as the model works with collections 
of partition rules and therefore only with the ranks in the features. We tested both MinMax scaling 
and Standard scaling methods and compared the results without scaling the data to confirm this 
hypothesis.

C. Scoring

compared to the actual classes. The most obvious score is the accuracy score which presents the 
number of correctly labels samples over the total number of tested samples. However, such scores 
don’t bring any information of the class distribution results. For instance, with very imbalanced 
datasets, always predicting the dominant category may give a high accuracy score but poor absolute 
performance of the model.  
Therefore, other scores are available to measure classification performances. We selected a few for 
this study on binary classification. We defined in the test sets:

D. Cross-validation 
Cross-validation is a common statistical method of evaluating and comparing learning algorithms by 
dividing data into two segments: one used to train a model and the other used to validate the model. 
The most used form of is called k-fold cross validation.  
As the data gathered in this study is not very large, we ran a k-fold cross-validation multiple times to 
achieve stability in the results. 
 
==PAGE:27==
27 
 
1) k-fold cross-validation 
k-fold cross-validation consists of distributing the data into k equally (or nearly equally) groups or folds. 
Subsequently k iterations of training and validation are performed such that within each iteration a 
different fold of the data is held out for validation, while the remaining k -1 folds are used for learning 
(25).  
Increasing the k value is desirable as it brings more performance estimations but it reduces the training 
size data. When dealing with large datasets, a common value for k is 5 or 10.  
 
2) Repeated k-fold cross-validation 
k-fold cross-validation allows to get performance estimations from k-trainings. To improve the stability 
of the results, we can run the validation multiple times. This is very convenient especially when dealing 
with small datasets. For instance, k must not be too big as some categories could be missing in the 
validation fold 
 
E. Models 
A multitude of classification models exist and are more or less adapted to the type of data. (24) 
presents a generic path to guide the choice of estimator. The best way to choose a model remains to 
try them and identify the best performing ones. A first analysis and the literature conducted us to look 
in to decision trees and more specifically random forest classifiers when dealing with small amount of 
data. As described in Inference using statistical models, decision trees also have the advantage of 
presenting feature importance and therefore the results are more interpretable. For sequence 
analysis, state-of-the-art models use deep learning models such as GRU, LSTM or Transformers for 
classification. However, these models require a large quantity of data to train. We still wanted to try 
these models at least to confirm the results of the related work with similar dataset sizes. 
In this study, our focus is on decision trees when dealing with inputs features, and recurrent neural 
networks (GRU, LSTM) when dealing with inputs sequences. 
 
1) Random forest classifier 
A random forest classifier is an estimator composed of many decision trees independent from each 
other (26).  
 
a) Decision tree classifiers 
Decision tree classifiers are models that predict the value of a target variable by learning simple 
decision rules inferred from the data features. A tree can be seen as a piecewise constant 
approximation (24). This model classifies samples by posing a series of yes-or-no questions over the 
features. Each question is contained in a node, and each node has two children – one for each answer. 
The hierarchical structure obtained forms a tree where the top-nodes (nodes without children) are 
called the leaves. The questions are created based on the range of value of a specific feature: “is value 
> n?”; or sometimes more complex questions with linear or combinations of multiple features (27). 
The goal of each question is to divide the samples in groups different from each other and samples 
similar in the same group. 
==PAGE:28==
28 
 
To measure the quality of a split after each node, decision trees used a criterion to define the optimum 
split. There are two criterions available: “Gini” and “Entropy”. The “Gini” impurity and the “Entropy” 
are calculated with the following formulas:

𝐸𝐸𝐸𝐸𝐸𝐸𝐸𝐸𝐸𝐸𝐸𝐸𝐸𝐸= − ෍𝑝𝑝𝑗𝑗× log 𝑝𝑝𝑗𝑗
𝑗𝑗
 
Where 𝑝𝑝𝑗𝑗 is the probability of class j. The Gini impurity measures the frequency at which any element 
of the dataset will be mislabelled when it is randomly labeled. The minimum value for Gini is 0 – 
meaning the split is pure and all samples are in the same category - and the max is 0.5. The Entropy 
varies between 0 and 1, and the split is pure when the entropy is null. 
 
b) Random forest classifier 
Random forest in an ensemble method that classifies samples based on the vote of each single decision 
tree. An ensemble method is a technique that combines the predictions from multiple estimators 
(power of the crowd) to make more accurate predictions than any individual estimator. Random 
forests improve the accuracy of single decision tree classifier by using Bootstrap Aggregation (Bagging) 
and features randomness (28).  
Random forests use bagging to create random sub-samples from the original data, and train the trees 
from each sub-sample. The sub-samples are created with replacement – meaning that the same value 
can be present multiple times. The size of the sub-sample remains the same as the original one. 
The other advantage of random forest is the ability to gather uncorrelated (or weakly correlated) 
results from its multiple estimators. Each tree is presented only a subset of features, resulting in each 
tree classifying samples based on different features set. The size of the subset of features is a 
parameter of the model – often set to the square root of the total number of features as default (26). 
Moreover, it is possible to calculate each feature importance within the random forest. This 
importance is defined with the criterion used in the decision trees. It measures how on average each 
feature decreases the impurity of the split. This gives us an idea of the most important features to 
perform the predictions.

F. Finetuning 
Once experimental parameters have been set (sequence duration, data sampling etc.), we search for 
optimizing the model parameters. A search consists of: 
- 
A model, here a Random Forest classifier or RNN 
- 
A parameter space – parameters range to test 
- 
A method for searching candidates 
- 
An evaluation method and a score function 
The parameter space is set manually depending on the dataset and the model being tested. We 
developed the cross-validation and scoring techniques previously. Then, we have two main methods 
==PAGE:31==

to search candidates: exhaustive and randomized search. The exhaustive search will try all 
combinations of parameters possible and report the validation score of each combination. The 
randomized approach searches over parameters, where each setting is sampled from a distribution 
over possible parameter values. The second option allows a budget that can be chosen independently 
of the number of parameters and possible values. 
While an exhaustive search was possible with fast training random forests, we set the hyperparameters 
of the recurrent neural networks manually as their training time was a lot longer.

V. Data preprocessing 
A variety of player modeling techniques exists but a few rely on raw mouse and keyboard inputs data. 
In this study, two techniques were tried to predict players’ skills as a category (good and bad 
performing players). 
The first was based on research on keyboard and mouse usage (16) (8) to predict stress levels. From 
raw inputs data (M&K events), multiple features were calculated on a specific duration. The second 
one was based on some player modeling techniques which take gameplay data as a sequence (17). 
Several types of events were selected, and the M&K input values were interpolated to construct a 
sequence over a short duration of gameplay.  
 
A. Data selection 
This section presents the criterions used to select viable sequences of gameplay. 
 
1) Minimal duration 
The gameplay sessions duration varied between 0 and 35 minutes. To ensure meaningful data, only 
sessions over 3 minutes of gameplay were analyzed. Indeed, shorter gameplays did not bring enough 
information about players’ skills to put him in a category. Sometimes, players just opened and closed 
the game for the second experiment. 
 
2) Recording offset 
The M&K events were recorded over the whole game, meaning the inputs in the main menu were 
recorded as well. As the first seconds spent into the game did not bring too much information about 
the player, an offset was chosen to select the data recorded only after 60 seconds of gameplay. This 
duration has been decided after observing the time players spent reading the rules of the game in the 
main menu. 
 
3) Random sequence offset 
In addition to the recording offset, the random sequence offset was implemented to add variety to the 
data. For instance, with a gameplay session of 640 seconds, it is possible to make 10 sequences of 60 
seconds but 40 seconds remain. Sequences were thus extracted with a randomly chosen offset up to 
the rest of time available (40 seconds). 
 
4) Extracting sequences 
The goal of this study was to be able to evaluate players’ skills based on a short duration of gameplay. 
Thus, multiple sequences of data were extracted from the collected data. A sequence represents all 
the events recorded over a time window of a chosen duration. Each sequence extracted from the same 
playthrough shared the same telemetry data (number of kills, deaths etc.) 
The first sequence durations tested are 60, 120, 180 seconds. The number of sequences available 
obviously decreased when their duration increased, so 180 seconds was the longest we could afford 
to keep enough training data for the models.  
==PAGE:33==
33 
 
 
5) AZERTY / QWERTY keyboard configurations 
The second experiment took place online with participants coming from France. There were two 
keyboards’ configurations used while playing this game: AZERTY and QWERTY and there was no 
automatic mapping when collecting the data. Thus, all the key events recorded had to be transformed 
into one keyboard configuration. We chose the QWERTY keyboard as it is standard.  
To detect which configuration the player was using, we simply evaluated if the player pressed more 
the “W” or “Z” key as they were the keys to move forward in the game. We then switched the events 
of the keys accordingly (if it was an AZERTY configuration, the “Z” events became the “W” events). 
 
B. Features calculation 
To make it more accessible to machine learning algorithms, the sequences of input events were 
summarized with calculated features. Indeed, the sparse data of thousands of events was reduced to 
a set of features selected to best describe the data. Each of the following features was calculated over 
a sequence (time window). 
The selected features were chosen in regards to the literature (8) and subjectively. Rather than 
selecting all the keys on the keyboard, we chose to record the events only related to game controls. 
For mouse events, the features depended generally on the value of the event (mouse rate) and its 
distribution (average, standard deviation, minimum and maximum).  
For keyboard events, there were only two possible values: pressed or released. The features then 
depended mostly on the duration of each press.  
We added two other main features: 
- 
the average frequency of a key being pressed 
- 
the average number of simultaneous key pressed 
After observing that bad players were not accustomed to WASD controls, these last features seemed 
to exacerbate this behavior.

C. Sequence generation 
We focus here on the temporal order of M&K events.  
All the raw information coming from the events was kept. For instance, it could allow to distinguish if 
a player presses one key at a time or a set of keys simultaneously. However, this required a 
considerable amount of data, as the input dimensions of the model are large (number of features times 
length of the sequence). 
First, the data needed to be interpolated. Indeed, as the data is based on M&K events, the frequency 
of them varied constantly. To construct sequences with fixed timestep between rows, we chose to 
interpolate the events values over filling only the rows with a detected event.   
Secondly, the registered events on a sequence were chosen based on the game controls. The mouse 
axis and all the keys having an influence on the game have been selected.  
 
==PAGE:35==
35 
 
1) Events interpolation 
For both the mouse rate and key states (pressed or released), the values were interpolated with the 
same frequency e.g., 100 rows per second for a 100Hz frequency. From each playthrough’ dataset of 
events, empty rows were added at regular intervals corresponding to this frequency. This resulted in 
a dataset with empty rows (NaN values).  
For keyboard inputs, we filled these empty rows with the last state of the key (beginning with a 
released state).  
For mouse inputs, we linearly interpolated with time the value of the mouse rate along each axis 
(beginning with a null rate).  
Finally, once all rows were filled, we kept only the original empty rows filled with interpolated values 
to obtain a dataset with a fixed timestep between rows. 
 
2) Selected events 
Each event corresponded to a specific key or mouse button / rate. In order to have the same entry size 
for the models, we kept only the events related to a control in game. 
Thus, each row of the dataset had the following features: 
- 
Timestamp 
- 
X mouse rate 
- 
Y mouse rate 
- 
Left mouse button state (pressed or released) 
- 
W, A, S, D states 
And each sequence had sequence_duration x interpolation_frequency rows e.g., 6000 rows for a 
sequence of 60 seconds with an interpolation frequency of 100Hz (see Table 11).

D. Training pipeline 
When training the models, we set up a pipeline as described in Methodology to sample, scale and split 
the data. This section presents how we adapted these methods to our use case. 
 
1) Over-Under Sampling 
The data was over- and under-sampled. During validation, both with and without sampling methods 
were evaluated. We duplicated or removed rows from the data when dealing with features and entire 
sequences of inputs when dealing with sequences. 
A custom sampler was developed to deal with sequences and cross validation in order to generate 3-
dimensional data from the training sets. 
 
2) Data scaling 
Predictions were made with and without data scaling. Two standardization algorithms were used to 
compare the results: MinMax and Standard scaling. We expected them to not influence the results. 
These algorithms were fitted on the training sets only, transformations were applied on both training 
and testing sets. This process was respected when performing cross validation. 
These scalers needed to be adjusted to handle time series data. Indeed, the scalers provided by the 
scikit learn package handled natively only 2D arrays (samples, features). To scale 3D arrays (samples, 
timesteps, features), we created a scaler for each feature i, fitted on the training sets as (samples, 
timesteps, feature_i) resulting in n_features scalers created. 
 
3) Cross-Validation 
To evaluate model performances, we used a 3-fold cross validation method adapted to this study. The 
data distribution among categories was not big enough to afford more folds. We then ran the 3-fold 
cross-validation 100 times and averaged the results. 
It is worth noting that the data needed to be split based on players and not sequences. Indeed, the 
training data contained sequences of events from different playthroughs, but the data needed to be 
split based on the players. This was to avoid having sequences from the same playthrough both in the 
training set and the testing set.  
However, each player didn’t hold the same number of sequences as the gameplay duration was not 
the same for each player. For instance, 13% of players holed 25% of all the extracted sequences of 60 
seconds. Therefore, when we split the data based on the number of players, the resulting test set sizes 
were not perfectly accurate.

VII. Discussion 
In this section we discuss player’s skills modeling based on the results of this study.  
 
A. Contribution and comparison with other studies 
Two studies in particular caught our attention in the comparison of results. 
David Buckley, Ke Chen and Joshua Knowles (8) aimed at predicting players’ skills with gameplay input 
as well on another genre of First-Person-Shooter game. Their analysis differs in two main aspects. First 
their testbed game is a First-Person-Shotter multiplayer game with fixed 3 minutes gameplay. Second, 
their considerations on players’ skills are not dependent on game metrics but rely on information 
provided by the participants with questionnaires. They chose to predict the number of FPS games 
played and the number of hours recently spent playing games, both as a four-class value, rather than 
predicting game metrics (e.g., kills/deaths ratio) for consistency, as opponents changed between 
games and could influence the results. Overall, while using only M&K features, they achieved 62% 
accuracy on a four-class prediction task. This result was obtained using a Random Forest classifier with 
500 decision trees and a 5-fold cross validation. The window size was set to 60 seconds. 
Paul Freihaut and Anja S. Göritz used mouse inputs to predict participant stress levels as a binary class 
problem. They used similar calculated mouse features as inputs for machine learning models (among 
which a Random Forest classifier) and also kept the temporal information with mouse movements as 
a time series and predicting stress values with LSTM network. The best accuracy was around 60% (+/- 
1%) with a K-nearest classifier using mouse features, and 50% using mouse movements as a time 
series. 
 
1) Correlation between players’ skills and M&K inputs 
 
a) Model accuracy 
The hypothesis of a link between players’ skills and his mouse and keyboard usage were confirmed in 
this study. Results from Best performing models showed that with a time window of around 3 minutes 
of gameplay, models were able to predict players’ skills as a binary classification with an accuracy 
above 75% with only mouse and keyboard data.  
This study focuses on a binary-class problem compared to other multiple-class problems mentioned 
above. We used a definition of skills based on in-game player performances which differs from player 
background or stress from other studies. Even so, with similar M&K data features preprocessing, our 
results are consistent with those of other studies, with at least similar - if not higher - accuracy on 
players’ skill classification. Therefore, with data from 40 participants (compared to 37 or 53 in the two 
other studies mentioned), some models were able to predict players’ skills with over 70% accuracy. 
These results indicate that this generic player modeling technique could be easily adapted to other 
games. As we focus only on M&K data for predictions, adaptive difficulty systems could be 
implemented without altering the code structure. However, the accuracy is not high enough at the 
time for real-world usage. 
 
==PAGE:55==
55 
 
b) M&K Features selection 
Our best performing models relied only on calculated features from recorded mouse and keyboard 
events. Figure 17 indicates that a small dimensional set of general features gives better predictions 
than a high dimensional set of every calculated features. Our new approach of using a small set of 
important features and average features as shown in the Finetuning section allowed us to further 
increase the accuracy of the models. 
Dimensionality reduction is another possible approach. As a few selected features seem to represent 
a big part of the information, using dimensionality reduction algorithms such as a Principal component 
Analysis (PCA) could bring better performances as the input features would be constructed from 
multiple extracted features and not a subset of them. 
 
c) Time-window length 
As expected, the larger the time windows (sequence duration), the greater the accuracy. Figure 21 
shows the improvement in accuracy with increasing time windows length. Other studies suggest that 
longer time windows are beneficial for accurate predictions as shown here as well (8). 
 
2) Deep learning models 
Recurrent neural networks (GRU and LSTM) didn’t perform very well for the binary classification task. 
The best accuracy reached 60% but the results were unstable between multiple cross validations. The 
training loss and accuracy were improving between epochs but the validation loss was increasing. 
Dropout at 20% and a small learning rate (as default) didn’t prevent this issue. 
Small amount of data is often associated to approximative results in the literature. We didn’t manage 
to confirm if taking the mouse and keyboard usage as a sequence was another viable approach for 
players’ skills modeling. There is little literature on recurrent neural networks using mouse and 
keyboard inputs data (16). The few studies found that tried these networks alongside other statistical 
models obtained the same results - models with random predictions on the test set. 
Further analysis with more data is therefore required to evaluate the viability of these models. 
 
3) Features importance 
Mouse features seemed to matter more than keyboard feature for players’ skills prediction. Figure 14 
indicates that overall, mouse features importances are higher than keyboard features importances. 
However, the number of simultaneous key pressed was also at the top of importance.  
These findings run counter David Buckley et al. results. Although they used a set of 10 features divided 
equally between mouse and keyboard features (compared to our 23), the 3 most important features 
from the random forest were the time spent holding some key. The fourth one was the sum of the 
mouse distance over the time window. The game genre may explain this difference, as FPS games rely 
on dynamic character movements while horror games rely mainly on observation. 
 
4) Gameplay duration influences predictions 
When looking at the whole game sessions from which sequences were extracted, we observed that 
the duration and the position of the sequence matters for the skills prediction.  
==PAGE:56==
56 
 
First, the time at which the sequences were extracted influences the accuracy of the models. Figure 23 
shows that overall, the best sequences to use to make predictions on players’ skills were either at the 
two- or four-5th of the gameplay duration. Further work would be to compare the sequence order with 
the time when the player killed a monster if he did. 
Second, we observe in Figure 24 that the average player accuracy was below the average global 
accuracy. Indeed, the difference is around 3% with a maximum of 5% but the two accuracies evolve in 
the same way with the sequence duration. The maximum accuracy per player is higher than the two 
others but decreases with the sequence duration (from 90% to 80%). This is expected as the number 
of sequences is much higher with 60 seconds duration than 170 seconds duration. Therefore, the 
chances of getting one good prediction from a larger set of sequences are higher. 
Finally, Figure 25 underlines that the predictions were better overall when the gameplay duration is 
high. However, this indication should be taken with caution, as the small amount of data makes it 
impossible to say whether this is due to certain sequences being easier to predict than others. 
 
B. Limitations 
 
1) Data limitation 
This study is based on the data collected from 40 participants with 26 gameplays usable.  
As we increase the sequence durations, the number of samples decreased and models have less 
training data, as shown in the Category distribution section. We observe in Figure 16 that the model 
performances were more variable between each validation fold as the sequence duration increased, 
therefore as the number of samples decreased. With larger datasets, the variations should decrease 
as the models converge more. 
Moreover, the data collected come from two different experiments and the criterion used to 
differentiate good and bad players was not exactly the same, although they are both based on the 
player ratio. The accuracy of the models when using the data coming only from the first experiment 
was better than when using the data from both experiments as shown in Figure 20. As the models 
should perform better with larger datasets, it may indicate that the criterions do not allow to split 
players exactly the same between the two experiments. 
Finally, in this study scope, the data came from a convenient sample of players who wished to 
participate in this experiment and not from a pre-defined exhaustive sample. The participants 
gathered did not represent all types of players. For example, we mainly collected data from players 
who are not used to playing horror games. 
In addition, similar studies on the more general subject of user modeling using mouse and keyboard 
inputs rely on datasets of the same size. The results comparison with the two mentioned studies is 
relevant as their data come from a set of respectively 34 and 53 participants. 
Further studies with larger datasets are needed to confirm the range of accuracy that these models 
can achieve in predicting players' skills based on mouse and keyboard usage data, and provide a 
reliable skills evaluation system.  
Church in the Wild is scheduled to be released on Steam in September 2022. When published, we aim 
at reaching a lot more players that could potentially participate in this study with an in-game option. 
==PAGE:57==
57 
 
 
2) Limited number of calculated features 
In this study, we calculated about twenty different features from mouse and keyboard events. These 
features were chosen regarding previous studies (8) (16), and some of them arbitrarily. There are 
possibly many other features that could be considered and might work better with machine learning 
models.  
In this paper, we showed that, for the random forest classifier with this amount of data, a little set of 
features was better than a big one. The best performing models used the most important features and 
two others based on a selection of other features as shown in Figure 17.  
In the literature, we identified two main approaches when calculating features. One tries to calculate 
all possible features, the other concentrates on a low-dimensional set that best summarizes the data. 
We chose to mix these two approaches, calculating all features first and identifying the most important 
one and summarizing the rest of them afterwards.  
Furthermore, using deep neural networks with more data could solve this issue. The hope with deep 
learning models is for them to understand hidden features from the data. With a greater amount of 
data, convolution neural network or dense neural network could be tried on mouse and keyboard data. 
To the best of our knowledge, there are no studies with big enough dataset that confirm this 
hypothesis. 
 
3) Players’ skills as a binary classification 
In this study, we restrained the prediction tasks to a binary classification with “good” and “bad” 
players.  
Other studies tried both multi-class classification and regression to predict players’ skills characteristics 
such as kill-to-death ratios in other games (8). As discussed in the Player’s skills section, players’ skills 
are highly arbitrary and depend on the game and player modeling goal.  
Future work is necessary to define the possible game metrics or player behaviors predictable with M&K 
inputs data. 
 
C. Future work 
In this section, we discuss how the results could be improved and the next steps to continue player 
modeling with generic data. 
 
1) Dimensionality reduction 
We discussed the influence of the number of selected features for making skills predictions. Although 
random forests are supposed to handle well data with many features, the increase in estimator 
accuracy when the number of features is small has put us on the path to dimension reduction as 
mentioned. Dimensionality reduction aims at transforming a high-dimensional space into a low-
dimensional space so that the low-dimensional data retains the most meaningful original properties. 
Indeed, when dealing with high-dimensional data, the volume of the space increases rapidly and the 
available data become sparse. This empiricist theory is known as the curse of dimensionality (34). 
Dimensionality reduction aims at solving this problem. 
==PAGE:58==
58 
 
There are multiple ways to achieve dimensionality reduction, the main ones being known as feature 
selection and feature projection. In this paper, we have used feature selection with random forest. We 
have also obtained preliminary results using Principal Component Analysis (PCA), a common method 
of feature projection. Briefly, PCA is a linear dimensionality reduction technique relying on Singular 
Value Decomposition of the data to project it to a lower dimensional space (35). 
First results indicated that 84% of the variance is explained with only 5 dimensions. When using this 
method in the training pipeline, we achieved an average accuracy of 67% using 170s sequences. This 
result is encouraging and could be the subject of further studies. 
 
2) Gradient boosting and other classifiers 
In this study, we have focused on the Random Forest classifier and briefly on RNN networks for skills 
prediction. However, many methods exist and are worth trying, one of which is gradient boosting 
classifier. Gradient boosting is an ensemble method similar to random forest but generally 
outperforms it. The main idea of boosting is to add new models to the ensemble sequentially. Each 
newly added model is built based on the error of the whole ensemble learnt so far. Gradient boosting 
adds a gradient-descent method to the ensemble for training. There are several common gradient 
boosting techniques such as XGBoost, CatBoost or LightGBM (36). 
We obtained preliminary results with CatBoost, a gradient-boosting model supporting categorical 
features (37). With sequences of 170s and all the features selected, we have achieved 66% accuracy 
and 61% f1 score. These promising results are not final and as shown in this study, further parameter 
optimization can lead to better performance.  
Furthermore, other classifiers are worth trying. The scikit-learn library provides fast implementation 
of different statistical models (24). 
 
3) Player clustering based on inputs data 
In this paper, we have explored supervised learning for players’ skills prediction. Unsupervised learning 
could be used to gather unexpected information about the player from their mouse and keyboard data 
as well. Clustering is one of the most unsupervised techniques used to construct groups of samples 
based on a distance measure. While some clustering techniques demand a high amount of data, 
hierarchical clustering is well suited to handle small datasets (38). 
Hierarchical clustering relies on a measuring distance, usually Euclidean or Manhattan distance. This 
method builds a hierarchy from merging individual clusters successfully. At each step, the two 
elements to merge are the closest relatively to the defined distance. Other clustering methods such as 
k-means can be worth trying as well.  
Clustering is a powerful technique which allow to extract information from the created clusters after 
interpretation. While some studies use mouse and keyboard data in order to detect user stress, 
clustering may move the research towards new horizons. 
 
4) Predicting skills as a regression task 
Skills can be very subjective depending on the area of research. The different definitions from this 
paper and (8) proves it. Skills can also be handled either as a class or as a numerical value. For instance, 
several multiplayer games implemented matchmaking based on player kill/death ratios to build 
==PAGE:59==
59 
 
balanced teams. (8) also aimed at predicting players ratio with regression, but the results have not 
been concluding.  
The subjective nature of player’s skills broadens the field of research and we hope that this paper will 
motivate further research on the subject. With bigger datasets and in the versatility of game genres, 
we may find ways to explore skills prediction as a regression task.

Conclusion 
Generic player modeling usable across games are a novel and exciting approach. Using gameplay inputs 
common between games, such as mouse and keyboard, can provide information on players’ behavior. 
In this paper, we attempted to link players’ skills as a binary classification with mouse and keyboard 
usage, bringing state-of-the-art results to a rare game genre of the field.  
We have begun by developing a testbed horror game to set up the context of this study. The game 
consisted of a mix of stressful exploration and hunting phases. Two experiments were conducted with 
the game with the aim of measuring players’ skills through the success of the hunt. The first in-house 
provided reliable data with long sessions of gameplay but few hunting phases, while the second was 
developed to reach more participants online but with misusage of the game.  
From the gathered mouse and keyboard data, we have calculated a novel set of features guided by the 
literature. The analysis of each feature importance in a random forest classifier have indicated that 
mouse features such the average cursor speed were more decisive for skills prediction than single key 
features in this game.  
Using calculated features, we managed to classify the players into two categories according to their in-
game performance with 75% accuracy. Moreover, these performances can be achieved after only 3 
minutes of gameplay.  From (8) we achieved a better accuracy with only M&K input data on reliable 
game metrics as output. 
The other deep learning recurrent network models tried have not been conclusive. Basic LSTM and 
GRU architecture perform under 60% accuracy with our dataset. While we believe such models could 
bring interesting results, the data size was not big enough for training. 
We consider this study to be a starting point of player modeling usable across games. While our model 
is not yet suitable for applications, a system capable of predicting players’ skills with higher accuracy 
using information independent of the game has many potentials in the game industry. As anti-cheats 
software that run alongside games, player modeling could become an external API available for 
developers and publishers. More generally, other player patterns are worth of interest than just skills, 
such as stress measurement as (14) suggests.
