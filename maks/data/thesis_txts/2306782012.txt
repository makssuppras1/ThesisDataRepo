Summary (English)
Safety-critical embedded systems are widespread in many application areas, such
as automotive, aerospace, industrial and healthcare. They are typically imple-
mented as distributed systems, composed of processing elements interconnected
using specialized communication protocols.
There is a strong interest in using the Ethernet protocol for communication,
since it is cheap and has high speeds.
However, Ethernet is unsuitable for
safety-critical and real-time applications. In this thesis we address ‘Determinis-
tic Ethernet’ (DE), which are extensions to Ethernet, such as TTEthernet and
Time Sensitive Networking, to support safety-critical and real-time applications.
However, to be safe, a system has to also be secure. Therefore, the focus of this
thesis is on the security of DE. DE integrates multiple traﬃc classes, including
Time-Triggered, Event-Triggered and Best Eﬀort. The Time-Triggered class,
which is intended for the most critical applications, uses schedule tables that
require clock synchronization.
We focus on the security of the clock synchronization algorithm. in TTEthernet,
which uses Protocol Control Frames (PCF) for its implementation. We have
deﬁned two attack scenarios, namely ‘manipulation attack’ and ‘delay attack’,
which can break the clock synchronization, and thus lead to the failure of the
highly-critical Time-Triggered applicatoins.
We propose a monitoring system for clock synchronization that is aimed at
determining if the system is under a security attack. The attack is detected
via an ‘anomaly detection’ method, which in our case is based on ‘entropy’
==PAGE:5==
ii
measures, a concept from information theory.
Our anomaly detection works in three steps: (1) features selection, (2) feature
quantization, and (3) comparison. (1) The experiment selects packet ‘times-
tamp’ and ‘pcf_transparent_clock’ header of a PCF as the features. (2) Prob-
ability Mass Function (PMF) as component to count entropy represents the
feature quantization.
(3) Relative entropy is used to compare the incoming
communication ﬂow to the baseline communication ﬂow (i.e., not under attack).
The technique has been implemented in the OMNeT++ simulator and has been
evaluated on several test cases. We have also implemented a tool for generating
TTEthernet traﬃc under the two attack scenarios, which has been used for the
evaluation. The proposed solution is able to detect anomaly caused by delay
attack to the sensitivity of 3 microseconds and anomaly caused by manipulation
attack by 10 nanoseconds.

Chapter 1
Introduction
Deterministic Ethernet (DE) is a computer network technology that utilizes
time scheduling to deliver deterministic real-time communication to standard
IEEE 802 Ethernet [AG]. DE is based on IEEE 802.3, IEEE 802.1Q AVB and
802.1Qbv TSN (pre-standard) as well as SAE AS6802 standards. The use of DE
has three main beneﬁts: message delivery guarantee for critical traﬃc, fault tol-
erant network for critical traﬃc, and convergance between critical, real-time, and
regular traﬃc within a single network. DE is proposed as layer 2 protocol which
is very similar to Ethernet but has the possibility to bound critical component’s
properties such as arrival time and component’s rate. This reliability feature
serves industrial needs for Ethernet. This feature is used in critical components
in an industry which failures leads to catastrophic consequences therefore it is
called safety-critical Cyber-Physical Systems (CPS). Deterministic Ethernet is
mainly used in industrial automation, defence, automotive, health care, and en-
ergy systems but not limited to these. DE works by scheduling messages coming
from various components.
Fault tolerance is an important feature of DE. Its goal to run reliably requires
the system to able to operate even if fault may occur. While letting fault occur,
it is also important to detect it and if possible ﬁnd the root cause of it. This is
the area where this work will mainly contribute in. Anomaly detection provides
user broader understanding to the system itself. The goal of the anomaly detec-
tor is to be able to provide user with information about communication delay

and changes in the system in real-time. The key diﬀerence of the anomaly detec-
tion with existing fault tolerance mechanism in the system is that the anomaly

1.1
Anomaly Detection
Anomaly detection aims to ﬁnd patterns in data that deviate from its expected
value. Anomaly itself refers to the nonconforming patterns that deviates. Chan-
dola in his work [VCK09] mentioned that anomaly detection is extensively used
in a wide variety of applications such as fraud detection for credit cards, in-
surance, or health care, intrusion detection for cyber-security, fault detection in
safety critical systems, and military surveillance for enemy activities. More ex-
tended examples to show the criticality of these anomaly detections are shown in
other previous works from Aleskrov [AFR97], Spence [SPS01], Kumar [Kum05],
and Fujimaki [Fuj05]. Aleskerov shows the case of anomalies in credit card trans-
action data which could be used to indicate identity theft [AFR97]. Spence in
his work shows the use of anomalous MRI image to indicate the presence of
malignant tumors [SPS01]. Kumar in his work shows that an anomalous traﬃc
pattern in a computer network could show a hacked computer is sending out
sensitive data to an unauthorized destination [Kum05]. Fujimaki shows the use
of anomaly detection even further to the outer space on detecting fault on space
craft sensor to ensure the space craft is operating without fault [Fuj05].
In this work, anomaly detection is used to detect anomalous clock synchroniza-
tion traﬃc in Deterministic Ethernet. One of the use of Deterministic Ethernet
is the in-vehicle network. There was also anomaly detection work in the in-
vehicle network that was done by Müter and Asaj [MA11]. Müter and Asaj
in their work introduced the use of anomaly detection system in a widely used
in-vehicle network, the CAN (Controller Area Network) network. Their work
inspires this work in general. The use of entropy and information theory are
adapted from their work.
This adaptation is considered appropriate as DE
is designed to be implemented in the in-vehicle network and both CAN and
TTEthernet have some consistent values that can be used for anomaly detec-
tion.

1.2
Information Theory
Information theory aims to calculate the ultimate data compression and the
ultimate transmission rate of communication. Even though the theory has been
used for a long time, the most ground breaking theory was written by Shannon
[Sha01]. Information theory itself is normally considered as subset of commu-
nication theory. However, the theory has been used also in other ﬁelds such
as: statistics, computer science, statistical inference, and probability and statis-
tics. In this work, information theory will be used mainly of its contribution
in probability and statistics to quantize the probability of anomalies in a DE
communication.
Information theory is a versatile theory which can be applied in a lot of scenarios.
According to Cover and Thomas [CT12], information theory is used in: data
compression, data transmission, network information theory, Ergodic theory,
hypothesis testing, statistical mechanics, quantum mechanics, inference, gam-
bling and investment, probability theory, and complexity theory. Information
theory in probability is used to categorize a random result into known classes
by calculating distances of relative entropy between known data and new data.
1.3
TTEthernet
TTEthernet expands classical Ethernet with services to meet time-critical, de-
terministic or safety-relevant conditions [AG09]. TTEthernet is compatible with
IEEE 802.3 standards and aims to fulﬁll the compatibility of Deterministic Eth-
ernet in general. TTEthernet is marketed by TTTech Computertechnik AG.
TTEthernet is standardized by SAE International [68011] with standardization
number SAE AS6802.
TTEthernet is used for safety-critical fail-operational applications. This means
that the system remains functional even if a failure occurs [AG09]. Fault-tolerant
mechanism ensures TTEthernet to keep on service even if a switch or a network
branch is faulty on some degrees and to keep the communication going. The
existence of fault-tolerant in TTEthernet is one of the key advantage of TTEth-
ernet.
TTEthernet enables the system to keep more important system to keep on ser-
vice even if other less important services are not working or being stopped on
purpose due to the potential threat that might happen if system is continued to
be in service. This is enabled by diﬀerent traﬃc classes implemented in TTEth-

1.4
Thesis Objective
Focus of this thesis is to ensure the security in safety-critical system, which
speciﬁcally in this thesis is the Deterministic Ethernet. Security to ensure safety
in TTEthernet is chosen as the scenario of the potential attack. We focuses on
detection of anomalies in TTEthernet and have not considered the eﬃciency of
the solution. In real application the solution needs to operate in real time since
deterministic Ethernet is used mainly in real-time environment.
Realizing the beneﬁts of a more connected networks, Deterministic Ethernet
which is commonly used in isolated and safe local network is in its evolution
phase towards being more connected to unsecured external network.
There
are two ways on realizing this goal: implement a fault tolerant network and
ensuring the reliability of the network. TTEthernet as an implementation of
DE has implemented various mechanism to ensure its fault tolerance. While it
is convenient to have a network that is tolerant against fault, it could be better
if a network can be monitored from time to time. We propose a monitoring
system of PCF clock synchronization in TTEthernet. Two features are used
in the experiment to show the possibility of using a ﬂexible feature selection
mechanism based on entropy in information theory by Shannon. The usage of
entropy enables fair, comparable, and ﬂexible feature selection.
We have tested and proposed eﬀective anomalies detection in TTEthernet for
some possible attacks. Anomalies detector can be installed in one of TTEthernet
end system in the network or installed outside and analyze the network based
on input ﬁles. We utilize entropy and other information theoretic measures to
achieve the anomaly detection. Two main beneﬁts of using this technique are:
ﬂexible feature selection and fast on run-time. These two beneﬁts are critical
for Deterministic Ethernet mainly in its usage in in-vehicle network. Flexible
feature means it will be easier for the system to scale up or down.
This is
important in an in-vehicle network because unlike phones or notebooks, the
lifetime of a vehicle is expected to be much longer. Therefore, it is important
to provide a system that can be easily updated without replacing the hardware.

1.6
Literature Study and Related Work
Elena et al. propose calculation of inter-arrival time to detect anomalies and
three mitigation techniques to protect clock synchronization in Deterministic
Ethernet [LGS+16]. Their work uses IEEE 1588 protocol as its clock synchro-
nization method. Apart from the diﬀerent clock synchronization protocol, it is
important to take the considerations of secure clock synchronization protocol.
Their work follow an approach of monitor, extractor, diagnoser, and reconﬁgu-
rator which similar with the method used in our work. The diﬀerence is that in
our work we analyse traﬃc globally while they analyse traﬃc locally on agent
levels. Their work highlights the possibility of delay attack in IEEE 1588. This
work inspired us to analyse similar attack in PCF.

2.1
TTEthernet Traﬃc
TTEthernet divides network traﬃc into three classes to support deterministic
feature. Those are: Time-Triggered (TT) Traﬃc, Rate-Constrained (RC) Traf-
ﬁc, and Best-Eﬀort (BE) Traﬃc. More important than all of these three classes
is the PCF message which contains value used for clock synchronization. PCF
message is given the highest priority of all traﬃc classes.
These traﬃc classes can be distinguished based on the Ethernet Destination
Address. TT, RC, and PCF messages follow the ARINC 664 part 7 standard
[AEE03]. According to ARINC 664 part 7, instead of having 48 bits for ethernet
==PAGE:23==
10
TTEthernet
destination address, destination address is split into 32 bits address and 16 bits
virtual link identiﬁer (VL ID). Additional rule, two least signiﬁcant bits of the
ﬁrst octet in constant ﬁeld must be 1. The other bits may be selected by user
but have to be ﬁxed for all ARINC 664 frames in a given system. Virtual Link
ID is used as actual addressing scheme. One VLID represents a single sender
to and one or more receivers. ARINC 664 switches store VL IDs in a statically
conﬁgured tables. This conﬁguration determines where outgoing ports frame
should be relayed. Another speciﬁcation from ARINC 664 is sequence number in
last octet of the Ethernet payload. This sequence number is used for redundancy
management and integrity checking by the receiver.
2.1.1
TT Traﬃc
Time-Triggered(TT) messages main usage is for applications that require tight
latency, jitter, and determinism [Obe11]. TT messages in TTEthernet has the
highest priority, which means it will be prioritized to be sent on-time. Messages
that are sent in TT messages have predeﬁned times which means the whole
network knows when every TT message is going to be transmit from which
system to which other system. This however, does not mean end systems are
forced to send TT message during the scheduled time. Unused TT message slot
will be freed and will be able for other less prioritize message. TT messages is
suited for communication in distributed real-time systems.
TT traﬃc guarantees predeﬁned bandwidth and latency for each application.
Having such feature, enables TT traﬃc to a critical system such as automotive
or aerospace applications. Other kinds of applications could also beneﬁt from
TT traﬃc mainly those depend on reliable communication and have high quality
requirements.

2.1.2
RC Traﬃc
Rate-Constrained (RC) messages aims is to establish a well-shaped dataﬂows.
Instead of guaranteeing when the packet is sent, RC message guarantees sending
rate. Rate guarantee makes it possible to predict when a RC message is sent
in normal network condition [Obe11].
Rate-Constrained (RC) messages are
generally used for applications with less needs of determinism and real-time
requirements than those required of strictly time-triggered applications.
RC
message guarantees that bandwidth is predeﬁned for each application and delays
and temporal deviations have deﬁned limits.
In contrast to TT messages, RC messages are not sent with respect to a system-
wide synchronised time base. Hence, diﬀerent communication controllers may
==PAGE:25==
12
TTEthernet
send RC messages at the same moment to the same receiver. As a consequence,
RC messages may queue up in the network switches leading to increased trans-
mission jitter. As the transmission rate of RC messages is bound a priori and
controlled in the network switches, an upper bound on the transmission jitter
can be calculated oﬄine and message loss thus prevented.
RC message has lower priority to TT message but higher priority to BE message.
Since RC is in a less critical level, so it can be transmitted only when there are
no TT messages around. As a result, TTEthernet could have full utilization
of its bandwidth by integrating this two message types together, meanwhile,
messages with highest level of criticality like TT messages, could arrive at its
destination in time.

BE frames can be sent and received by any Ethernet (and TTEthernet) com-

BE messages (frames) are the one with lowest critical level and therefore have

2.2
Clock Synchronization with PCF
In this work we will focus on securing the clock synchronization protocol used
in TTEthernet called PCF. The structure of PCF itself is already described in
previous section. In this section we will discuss about how clocks in a TTEth-
ernet network could be synchronized by using PCF. First of all, it is important
to understand the three main roles in PCF: Synchronization Master (SM), Syn-
chronization Client (SC), and Compression Master (CM). Second, which is also
as important as the roles, is the integrity of the CM. There are two kinds of CM
that can be used in PCF communication: High Integrity Compression Master
(HICM) and Standard Integrity Compression Master (SICM). In this section,
ﬁrst these three roles will be described then we will go step by step of the syn-
chronization both with high integrity and standard integrity. At the end of this
section we will also go through three PCF message types which are handled in
a PCF clock synchronization cluster.
2.2.1
Synchronization Master (SM)
Synchronization Master is a role that act as the initiator of the clock synchro-
nization. In every PCF cluster, there should be at least one SM. SM is typically
implemented in a trusted end system even though it is not a mandatory require-
ment.
Synchronization Master may start any session of PCF clock synchronization
message at any time. SM can start coldstart type PCF message and integration
type PCF message. SM should start clock synchronization session when it needs
to synchronize the clock with the system. This is generally done periodically in
a previously deﬁned schedule.
2.2.2
Synchronization Client (SC)
Synchronization Client behaves in a similar way with Synchronization Master
except SC has no privilege to start the clock synchronization mechanism. Both
end system and switch can be conﬁgured as SC. A PCF cluster could still operate
without a SC in the cluster.
Synchronization Client always ready to receive a compressed integration frame
from Compression Master. However, SC should have a spanned window of when
it should consider the compressed integration frame or not. If a compressed
==PAGE:30==
2.2 Clock Synchronization with PCF
17
integration frame arrived out of the spanned window it is called out-schedule.
An out-schedule compressed integration frame is not taken into account.
2.2.3
Compression Master (CM)
Compression Master is a role that act as moderator that calculates the synchro-
nized clock in the clock synchronization. In every PCF cluster, there should be
at least one CM. CM is typically implemented in a trusted switch even though
it is not a mandatory requirement.
There are two variants of CM in PCF clock synchronization mechanism. First
one is called the High Integrity Compression Master (HICM) and the second
one is called the Standard Integrity Compression Master (SICM). The main
diﬀerence between these two CMs is that HICM will always send compressed
integration frame to its end systems regardless the content of the frame while
SICM only send compressed integration frame if the containing clock is synchro-
nized with its local clock.

a minimum of one SM, one CM, and another SM or SC. While not mandatory,

2.2.7
PCF Frame Types
There are three PCF Frame types: Coldstart frame (CS), coldstart acknowledge-
ment frame (CA), and integration frame (IN). Both CS and CA are initiated
==PAGE:33==
20
TTEthernet
by SM whenever SM is not synchronized. It is usually sent at the beginning
of the communication or after a restart. After the clock has been successfully
synchronized, CS and CA frame are not sent anymore. Integration frames which
sent time is conﬁgurable are sent instead.
The existence of integration frame allows node to be synchronized at any point
of communication with less overhead of re-sending acknowledgements. However,
a threat of using integration frame is that a node might be using an outdated
integration frame. To prevent this, PCF has a fault tolerant mechanism which
utilized one of its value: integration_cycle. Value of integration_cycle indicates
a position of an integration frame inside an overall cluster cycle. Maximum
value of integration_cycle in a cluster cycle is conﬁgurable with the parameter
max_integration_cycle. The value of integration_cycle in a frame is also used
by CM to determine which packet should be consider together when running the
compression function. All in-schedule frame with same integration_cycle value
are used by CM to generate compressed integration frame.
Classifying frames into diﬀerent frame types is crucial in this work. The anomaly
detection works with the idea of having something that should be consistent
and monitor the consistency of the value throughout the communication. While
output of frames from a node should be consistent, it is only consistent when
communicating the same frame type. i.e. CS frame type sent by node A is not
necessarily consistent with IN frame type sent by node A. Hence, the anomaly
detection monitor each type of PCF frame individually.

The eﬀorts on maintaining

One of the most common practice on maintaining conﬁdentiality is the use of

receiver can read the encrypted message by decrypting the message using their

of symmetric encryption. In asymmetric encryption the key for encrypting is
diﬀerent with the key for decryption.
In extreme cases where conﬁdentiality is the vital security part, precautions
such as storing only on air gapped computers, disconnected storage devices or,
in hard copy form only might be done. Conﬁdentiality is not as high concern
in clock synchronization. There are no critical concerns when the clock syn-
chronization value is leaked. It is however, can not be denied that knowledge of
the leaked clock synchronization value might lead into another attack. E.g. to
successfully construct a malicious packet attacker needs to sync the malicious
sequence number to the current legitimate sequence number.

Integrity
Integrity in aspect of network security is achieved when a communication can
ensure consistency, accuracy, and trustworthiness of data over its entire life cycle.
It is including making sure that communication is done only by legitimate person
thus enforcing authenticity of the communicators.
The most important fundamental principle on maintaining integrity over a com-
puter network is to ensure that data is not altered by unauthorized party in
transit.
On achieving this goal, there are some measured that can be done
including but not limited to: providing data permissions, user access controls,
checksums, cryptographic checksums, encryption.
Integrity is a main security concern together with availability in clock synchro-
nization.
It is important to ensure that systems only take into account the
packets that are coming from legitimate senders. However, the use of encryp-
tion in clock synchronization is highly debatable as the process of encryption
and decryption require time and resources which might cause delay in the clock
synchronization process.
Other approaches have been introduced to maintain integrity in clock synchro-
nization such as: fault tolerance mechanism and intrusion detection system.
The earlier approach ensures that system works even though communication
integrity is no longer maintained. The later approach tries to deduce an attack
from either the process or the impact of the communication. Both approaches
can be used together to maintain integrity. Anomaly detection system which is
proposed in this work is a sample form of an intrusion detection system.

hardware, performing hardware repairs immediately when needed, maintaining
software versions while ensuring it free of software conﬂicts, keeping system up

high-availability clusters to mitigate serious consequences when hardware issues
occur, fast and adaptive disaster recovery, safeguards against data loss or inter-

Anomaly in this work refers to the change that should not had happened in
the DE network unless there is at least one attack happens. For every kind of
attack, there should be at least one feature that deviates from its expected value.
These features can be very diﬀerent from one to another in case of measurement
units. The diﬀerence in measurement units makes them incomparable from one
to another and makes it impossible to make a generic anomaly detection for all
kind of attacks. This problem is solved by the use of information theory which
was coined by Shannon [Sha01]. Shannon in his work coined the concept of
entropy. Entropy is a measure of uncertainty of one random variable. Entropy is
not bounded to any measurement unit therefore its usage enables us to produce
a generic solution for anomaly detection which result is comparable from one
another.
This chapter explains the concept of information theory which is used in this
work and how is it fused into the framework of the anomaly detection. In later

5.1
Information Theory
Information theory plays an important role in this work. Information theory
makes it possible to compare two features that has diﬀerent measurement unit.

5.1.1
Entropy
Entropy in information theory is a measure of uncertainty of one random vari-
able. Uncertainty in the simplest form could be analogized as ‘How many ques-
tions in average should be asked before a certain answer could be obtained?’.
Let machine A, an abstract machine which is able to produce output a, b, c, d
with diﬀerent probability. The probability of the machine to generate the output
is called the probability mass function (PMF). Probability mass function of a
random variable x is denoted p(x).
Let PMF of machine A as follow:
p(a) = 0.25
p(b) = 0.25
p(c) = 0.25
p(d) = 0.25
Then we can apply the previous question in the respective machine. ‘How many
questions in average should be asked before we know what the answer will be
given by machine A?’. Easiest way to do this is by draw a decision tree and
make it as eﬃcient as possible. Figure 5.1 shows the most eﬃcient way to be

5.2
Anomaly Detection Framework
Anomaly is by deﬁnition a situation where non expected circumstances happen
and it deviates from what is expected. This general deﬁnition applies to net-
work communication as well which in our case is the communication within a
deterministic ethernet or TTEthernet more speciﬁcally. There are several com-
munication ﬂows within TTEthernet but most of them are not uniquely belongs
to TTEthernet. Most of them are due to TTEthernet compatibility with ether-
net. There is however, one unique TTEthernet communication ﬂow which will
be the object of detection in this work. PCF clock synchronization is one of the
communication protocol that can used in clock synchronizing in TTEthernet.
PCF communication ﬂow is unique for TTEthernet but not limited to it.
In this section we will introduce some concepts that is used in the work to do the
entropy based anomaly detection. There are three simple steps that are required:
features selection, quantization, and comparison. In following subsection we will
follow this model to enable the entropy based anomaly detection.

5.2.1
Features Selection
First step on detecting anomalies using entropy is to select the features that
will be compared. Choosing features might seem as a trivial task but it could
be tricky. The most important thing a selected features must have is known
consistency so it could be used as the baseline. Baseline is something that we
believe as standard and the value should be consistent.
Second thing to be
considered is the clarity of the diﬀerence that might occur when an anomaly
happens or in our work. A selected feature needs to be able to be quantized as
probability mass function which is used in further steps.
According to Muter and Asaj [MA11], there are at least three ways to help us
choosing the features in an entropy based anomaly detection system:
• Binary abstraction level: abstracting input data on binary level.
This
means taking the whole input as it is. This is not necessarily classifying a
whole input ﬁle. This abstraction could be taking input in a whole com-
munication ﬂow to classify every ﬂows. The main idea when abstracting
on binary level is there are no diﬀerentiation between headers and ﬁelds.
Practically there are two possible classiﬁers at binary level: bitwise clas-
siﬁer that considers all bits in an input and x-bitwise classiﬁer which bits
are taken from known part of the message.
• Signal abstraction level: abstracting input data on signal level. This means
taking the attribute of a communication ﬂow for each events to classify
them as normal message or anomaly. In most communication protocol,
anomaly will lead to unexpected attribute around the message. By ana-
lyzing each of every sequence within communication ﬂow we can classify
whether an input ﬂow is a normal or anomaly. We can also consider in
the timing of received messages in an end system or a switch as it needs
to be consistent in Deterministic Ethernet.
• Protocol abstraction level: abstracting input data on protocol level. This
means taking the value of at least one protocol ﬁelds into consideration.
Classiﬁer in protocol level will compare the input value of speciﬁc value
of a ﬁeld in protocol with the known value and its label. The similarities
or diﬀerence will then be the knowledge to classify an input.
These three ways of selecting features is independent to each other.
Which
means we can do one without disturbing the other. This makes it possible for
us to utilize more than one level of abstraction.

5.3.1
Feature Selection
Following the framework from previous section, the ﬁrst thing to be considered
when doing an entropy based anomaly detection is to pick the feature that
can be used to diﬀerentiate an anomaly from a normal communication packet.
This thing has to be part of the detected object. In our work, the object is the
communication packet itself which is a PCF packet. Therefore, the feature must
be extractable from the PCF packet.
After features has been selected, we need to consider which packets should be
compared with each other. It seems a trivial thing but it could be tricky. In
common cases, communication packet should be similar even when various de-
vices communicate. However, in the case of clock synchronization every device
has its own delay, hence comparing the output of one device with another is
irrelevant. It is important to diﬀerentiate the way we group the communication
packet depends on the case that we are investigating. Since this work focuses
on detecting anomalies in clock synchronization, the grouping is to group com-
munication packet into the source of the packet. The baseline communication
packet will only be used as a comparison when it is sent by the same device.
Another grouping is grouping based on the type of the packet. In PCF clock
synchronization, there are three types of packet communication: coldstart (CS),
==PAGE:61==
48
Entropy Based Anomaly Detection
coldstart acknowledgement (CA), and integration (IN). Grouping by packet type
is needed since the way a node response depends on the speciﬁc type of the PCF
packet.
These groupings are done in a map processor which process input packets and
output it into a map data type containing the quantized value of each input.
This part of the implementation is called the mapper. The mapper currently
takes input in the form of pcap ﬁle but is extensible to take input directly from
network card interface dump.
There are at least three ways to extract the features based on where we do
the abstraction: data abstraction level, signal abstraction level, and protocol
abstraction level. All each of them has its own beneﬁts and drawbacks which will
be discussed following subsections. All of the abstractions that are considered
and implemented are explained and how it is extracted from the PCF packet.
There are three ways done in this work: timestamp signal abstraction (signal
abstraction level), transparent clock abstraction (protocol abstraction level),
and a combination of timestamp and transparent clock abstraction.
5.3.1.1
Timestamp abstraction
Timestamp abstraction can be classiﬁed as a signal abstraction level. This is
classiﬁed this way since timestamp is not sent within a packet but is a result of a
sent or received packet. Whenever a packet is received or sent, devices will stamp
it with a timestamp for these packets which can be monitored.
Timestamp
abstraction exploits this timestamp obtained when a packet is received or sent.
More accurately is the relative timestamp between when a packet is obtained to
when the continuation of the packet is sent. The timestamp diﬀerence between
these two timestamps should always be consistent in all similar type of packet
communications. The diﬀerence of received and sent timestamp is used as the
single feature in this detection. PMF for this abstraction is a normal distribution
with the timestamp diﬀerence of the baseline as the mean and 50 ns as the
standard deviation of the distribution. Hypothetically, this abstraction is able to
detect delay attack since it is monitoring packets timestamps. This abstraction
is possible to be done in any independent node. This is a beneﬁt since it means
we can have it in selected critical nodes only and installation will also be simpler.
This is possible because both received and sent timestamp are generated in
one device.
However, the drawback of this abstraction would be the harder
implementation when we compare it to the simpler binary abstraction level.

5.3.1.2
Transparent clock abstraction
Transparent clock is one of the payload of PCF. It comprises of accumulated
delay from the ﬁrst node until the last node that transmits the packet. As it is
part of the protocol itself, this abstraction is classiﬁed as a protocol abstraction
level. Every node that forwards a PCF packet should add delay to the trans-
parent clock value based on the delay its making to the packet. Therefore, the
additional delay should be consistent on every time considering the device is on
good condition when it is processing a similar type of packet. Exploiting this
consistency, we can take the relative value of transparent clock when a PCF
packet is received to the value of transparent clock when the same packet is sent
from a device. The diﬀerence of received and sent values of transparent clock is
used as the single feature in this detection. PMF for this abstraction is a nor-
mal distribution with the relative transparent clock value of the baseline as the
mean of the distribution and 50 ns as the standard deviation of the distribution.
Hypothetically, this abstraction is able to detect manipulation attack of trans-
parent clock value. This abstraction could be implemented independently in
every node. This means we can select nodes where maintaining un-manipulated
transparent clock is needed. Since the installation is independent on every de-
vice, then conﬁguration should be simpler as there will be no integration between
nodes. This is possible because both received and sent transparent clock values
could be obtained in one device. However, the drawback of this abstraction is
that the implementation is more complex than the simpler binary abstraction
level.
5.3.1.3
Timestamp and transparent clock abstraction
This abstraction is made to show that it is possible to combine features in
an abstraction. Two previously described features of timestamp diﬀerence and
transparent clock values diﬀerence are used in this abstraction.
To achieve
this, we need to make PMF for both features. PMF for this abstraction is a
bivariate normal distribution with timestamp and its normal distribution and
standard deviation as x axis and transparent clock and its normal distribution
and standard deviation as y axis. Hypothetically, this abstraction is able to
detect both delay and manipulation attack of transparent clock value. There
are a beneﬁt and a drawback of combining features. The beneﬁt is that we can
detect multiple network attacks in a single run which means lower resources
needed. However, the drawback is that we cannot classify what kind of attack
that just happened to the system.

8.1
Discussion
This work shows that it is possible to ﬁnd patterns of some chosen PCF clock
synchronization attacks.
The method used in this work is fairly simple and
straight forward due to the limited feature selection of the PCF itself. It might
be interesting to try to combine some other relevant features into the solution.
Other methods such as Neural Network, Shared Nearest Neighbor, and Kernel
Method could be used for the solution. Information theory is chosen in this
work due to its simplicity and the similarity of PCF clock synchronization with
CAN network.
==PAGE:97==
84
Discussion, Conclusion, and Future Work
One of the most critical obstacle in this work is the availability of libraries to
support the main focus of the solution. While there are many proposed methods
on detecting anomaly, the real obstacle is to generate the test cases. PCF does
not have many open implementations to generate these test cases. Even with
the used simulator, it is tricky to export the simulation result into a format that
is similar to network capture in general. From this work, the writer recommends
improvement for the simulator mainly on the capability of exporting simulation
result into a well-known format such as PCAP or the standardized PCAPNG.
We suggest an implementation of PcapDump instead of work-around solution
used in this work.
Other consideration that might raises doubt in this work is upon the use of
entropy and information theory. Both of these concepts can be seen as non-
beneﬁcial in a glance. The beneﬁts of using these concepts are combination of
features and making the anomaly detection results comparable with one another.

Network monitoring can be done in the concept of all-seeing-eye where the mon-

Work of Elena et. al [LGS+16] implements attack detection and mitigation in
per-device concept. The beneﬁts of having the solution implemented in per-

rather than an all-seeing-eye concept. The drawback of per-device detection and
mitigation concept is the limited amount of information that can be used. Ben-

concept. This work uses ﬁles that are behaving like an all-seeing-eye concept.

of more data. However, an all-seeing-eye monitoring system could be harder to
conﬁgure, hence requires more resources. The solution implementation in this
work is done in a ﬂexible framework by the use of mapper and reader interface.
It is possible with this solution to make network monitoring system in both con-
cept. While the test in this work is done in a per-device concept, all-seeing-eye
concept should not be neglected as it has potential to give more accurate result.

8.2
Conclusion
We conclude the use of entropy to detect anomalies within deterministic ethernet
as this work has tested in TTEthernet as one of DE implementation is capable
on detecting anomalies with some limitations. The experiment proves the hy-
pothesis that timestamp feature is appropriate on detecting delay attack with
limitation to around 1 to 3 microseconds depends on the network architecture.
The experiment also proves the hypothesis that transparent clock feature is ap-
propriate to be used to detect transparent clock manipulation attack. The later
is very precise and able to detect up to 10 nanosecond. It is important to note
that the experiment is done in a limited simulator environment.The experiment
shows the capability of combining features on the detection system. Combining
features however, is not fully implemented. Imporvements are needed for using
the combined features to obtain the beneﬁt of using combined features.
The implementation shows entropy based anomaly detection is capable on mon-
itoring TTEthernet network.
The monitoring system enhances TTEthernet
network fault tolerant system. The solution is given as an independent solution
that treats TTEthernet as a blackbox. This independent nature minimizes the
need of network re-conﬁguration on solution deployment. Utilization of entropy
and other information theory concepts to produce a fair, ﬂexible, and compara-
ble quantization makes it possible for the solution to be used for other protocols
and other system.
8.3
Future Work
There are many possible future works that are possible to be done after this
work. This work itself as stated in the beginning is a stepping stone towards
securing Deterministic Ethernet in an open unsecured network by providing an
anomaly monitoring system. Four of the possible future works are discussed in
this section: implementation of the solution in a real network, extracts more
features for PCF clock synchronization to be used on same or diﬀerent attacks
detection, investigation of other protocols, and implements other possible solu-
tions using similar framework.
First possible future work is to implement the solution in a real TTEthernet
network. This is not done in this work due to some resources constraint. The

also important to analyze the limitation of the solution when it is implemented
in the real solution that might not occur when it is implemented in simulator.
Second possible future work is to extract and use other features of the clock
synchronization protocol. These other features could be used for manipulation
or delay attack or another attacks that have been identiﬁed in this work.
Third possible future work is to investigate the usability of the solution mainly
its framework to be used in detecting anomaly for another protocol. Conceptu-
ally, it is possible by extending the mapper and reader interfaces.
Fourth possible future work is to implement the other anomaly detection meth-
ods in the solution framework. Some of these methods are introduced in chap-

method. Upon implementation of other method, a comparison between methods
could be made to obtain the optimized solution for anomaly detection.
